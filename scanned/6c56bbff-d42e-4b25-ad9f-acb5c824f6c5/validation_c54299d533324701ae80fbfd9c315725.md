### Title
Unguarded `sweepToken` Drains Any ERC-20 Held by the Router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.sweepToken` is `public payable` with no caller restriction and accepts any arbitrary `token` address. Any ERC-20 balance that accumulates on `MetricOmmSimpleRouter` or `MetricOmmPoolLiquidityAdder` — whether from an airdrop, a user mistake, or a two-transaction swap pattern — can be drained by any unprivileged caller to any recipient they choose.

---

### Finding Description

`sweepToken` in `PeripheryPayments.sol` reads the router's full balance of the caller-supplied `token` and transfers it to the caller-supplied `recipient`:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
```

There is no `msg.sender` check, no whitelist of valid tokens, and no transient-context guard. The function is inherited unchanged by both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder`.

The router's `exactInput` multi-hop path explicitly routes intermediate output to `address(this)` between hops:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol  L106
i == last ? params.recipient : address(this),
```

The intended usage pattern — swap output to `address(router)`, then `unwrapWETH9` or `sweepToken` in the same `multicall` — is safe because `multicall` is atomic. However, if a user issues the swap and the sweep as two separate transactions (a realistic mistake given the public API surface), any watcher can front-run the second transaction and call `sweepToken(token, 0, attacker)` to redirect the full balance.

Beyond the two-transaction race, the function also allows draining tokens that arrive on the router via governance airdrops, rebasing distributions, or direct user transfers — none of which are pool-related tokens. This is the exact class of the external bug: an arbitrary `token` parameter with no membership check.

---

### Impact Explanation

Any ERC-20 balance on the router is claimable by any address. A victim who sends swap output to `address(router)` in transaction 1 and sweeps in transaction 2 loses the entire output amount. Airdropped or mistakenly transferred tokens are permanently extractable by the first caller who notices them. Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are affected because both inherit `PeripheryPayments` without overriding `sweepToken`.

---

### Likelihood Explanation

The two-transaction race is a realistic user error: the public API exposes `sweepToken`, `unwrapWETH9`, and `refundETH` as standalone callable functions, and nothing in the interface documentation warns that they must be composed atomically via `multicall`. MEV bots routinely monitor for stranded ERC-20 balances on known router addresses. Airdrops to widely-used router contracts are also common on EVM chains.

---

### Recommendation

Restrict `sweepToken` (and `unwrapWETH9`) so that only `msg.sender` can be the `recipient`, or add a `msg.sender == recipient` assertion. Alternatively, gate the function behind a transient-storage "active multicall" flag so it can only be called from within a `multicall` context. At minimum, document that these helpers must always be composed atomically and never called as standalone transactions.

---

### Proof of Concept

```
// Attacker script (two-block scenario)
// Block N: victim calls exactInputSingle with recipient = address(router)
//   → router receives, say, 50_000 USDC as swap output

// Block N (same or next): attacker front-runs victim's sweep call
router.sweepToken(
    USDC,          // arbitrary token — no membership check
    0,             // amountMinimum = 0 → always passes
    attacker       // attacker-controlled recipient
);
// Result: attacker receives all 50_000 USDC; victim receives nothing.
```

The same call works for any ERC-20 airdropped to the router:

```
// Governance airdrop of TOKEN to router address
router.sweepToken(TOKEN, 0, attacker);
// All airdropped TOKEN transferred to attacker.
