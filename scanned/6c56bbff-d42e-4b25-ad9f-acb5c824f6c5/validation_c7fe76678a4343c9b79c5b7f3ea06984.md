The core of this question is whether `pay()`'s WETH branch consumes unattributed contract ETH, and whether ETH can be stranded on the contract from a prior call. The code confirms both conditions exist.

---

### Title
Unattributed `address(this).balance` in `PeripheryPayments.pay()` WETH branch allows stranded ETH from one user to fund another user's WETH liquidity payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses the contract's total native ETH balance (`address(this).balance`) to partially or fully fund WETH payments without attributing that ETH to the current payer. Because all `addLiquidity*` and `multicall` entry points are `payable`, ETH can be left on the contract between transactions. Any subsequent WETH-pool liquidity call will silently consume that stranded ETH, transferring the loss to whoever deposited it.

---

### Finding Description

In `PeripheryPayments.pay()`, the WETH branch reads the contract's total native balance: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total, unattributed
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

ETH reaches the contract via any of the `payable` entry points: [2](#0-1) [3](#0-2) 

If a user calls `addLiquidityExactShares{value: X}` for a **non-WETH pool**, the X ETH is never consumed (the `pay()` ERC20 branch is taken for both tokens) and sits on the contract until the user separately calls `refundETH()`. [4](#0-3) 

There is no per-caller ETH accounting; `refundETH()` sends the entire contract balance to `msg.sender`, and `pay()` consumes the entire balance for the current WETH payment — whichever happens first wins.

---

### Impact Explanation

**Victim (User A)** sends ETH with a non-WETH liquidity call. ETH is stranded on the contract.

**Beneficiary (User B, possibly the attacker)** calls `addLiquidityExactShares` for a WETH pool needing `V` WETH. Inside `metricOmmModifyLiquidityCallback`: [5](#0-4) 

`pay(WETH, userB, pool, V)` is called. With `nativeBalance = X` (User A's ETH):
- If `X >= V`: User B pays **zero** WETH from their allowance; User A loses `V` ETH.
- If `0 < X < V`: User B pays only `V - X` WETH; User A loses `X` ETH.

This is a **direct, cross-user loss of principal** with no recovery path for User A once the ETH is consumed.

---

### Likelihood Explanation

The pattern of sending ETH with a liquidity call and relying on a separate `refundETH()` step is the standard Uniswap-style router UX. Any user who:
- Sends ETH with `addLiquidityExactShares` for a non-WETH pool, or
- Sends excess ETH with a WETH pool call (more than the pool requests), or
- Uses `multicall` with ETH and omits `refundETH` as the last step

...leaves ETH on the contract. A MEV bot or any observer can immediately follow with a WETH-pool liquidity call to consume it.

---

### Recommendation

Track per-caller ETH deposits in transient storage (alongside the existing pay context) and restrict `pay()`'s WETH branch to only consume ETH attributed to the current `payer`. Alternatively, only consume `msg.value` within the same call frame by passing it explicitly rather than reading `address(this).balance`.

---

### Proof of Concept

1. User A calls `addLiquidityExactShares{value: 1 ether}(nonWethPool, ...)`. Pool does not use WETH; `pay()` calls `safeTransferFrom` for both tokens. 1 ETH remains on the contract.
2. Attacker calls `addLiquidityExactShares(wethPool, ...)` (no ETH sent) for a pool where `token0 = WETH` and the pool requests `amount0Delta = 1 ether`.
3. `metricOmmModifyLiquidityCallback` loads `payer = attacker`, calls `pay(WETH, attacker, wethPool, 1e18)`.
4. `nativeBalance = 1 ether >= 1e18` → `IWETH9.deposit{value: 1 ether}()` + `safeTransfer(wethPool, 1e18)`. Attacker's WETH allowance is **not touched**.
5. User A's 1 ETH is gone. `refundETH()` now returns 0.
