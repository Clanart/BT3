### Title
Missing `address(0)` Recipient Check in Swap Functions Causes Permanent Loss of Output Tokens - (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol)

### Summary
`MetricOmmSimpleRouter` passes the caller-supplied `recipient` parameter directly to the pool's `swap` function without validating it is non-zero. A user who calls any of the four swap entry points with `recipient = address(0)` — by mistake or via a misconfigured integration — permanently burns their entire swap output.

### Finding Description
All four public swap functions in `MetricOmmSimpleRouter` accept a `recipient` address and forward it verbatim to `IMetricOmmPoolActions.swap()`:

- `exactInputSingle` — [1](#0-0) 
- `exactInput` — [2](#0-1) 
- `exactOutputSingle` — [3](#0-2) 
- `exactOutput` — [4](#0-3) 

None of these functions contain a guard of the form `if (params.recipient == address(0)) revert ...`. The pool executes the token transfer to whatever address is supplied. When `address(0)` is the recipient, the ERC-20 `transfer` call succeeds (most standard tokens allow transfers to `address(0)`), and the output tokens are irrecoverably burned.

By contrast, `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` does perform an analogous guard: [5](#0-4)  — demonstrating the pattern is known and intentionally applied elsewhere in the same codebase, but was omitted from the router.

### Impact Explanation
A user who passes `recipient = address(0)` to any swap function loses 100% of their swap output. The input tokens are consumed by the pool (paid via the callback), the pool transfers the output to `address(0)`, and the `amountOutMinimum` check passes as long as the swap itself succeeds — it only checks the *amount*, not the *destination*. The loss is permanent and unrecoverable.

### Likelihood Explanation
This requires a user mistake (passing zero address), which is a realistic scenario for:
- Programmatic integrations that forget to populate the recipient field (defaults to zero in ABI encoding).
- Scripts or bots that construct `ExactInputSingleParams` / `ExactOutputSingleParams` structs with an uninitialized `recipient`.
- Copy-paste errors in integrations.

No privileged access is required; any caller can trigger this against themselves.

### Recommendation
Add a zero-address check at the top of each swap entry point, mirroring the pattern already used in `MetricOmmPoolLiquidityAdder`:

```solidity
// In exactInputSingle, exactInput, exactOutputSingle, exactOutput:
if (params.recipient == address(0)) revert InvalidRecipient();
```

Alternatively, add the check once inside a shared internal helper or at the `ExactInputSingleParams` / `ExactOutputSingleParams` struct validation layer.

### Proof of Concept

1. User calls `exactInputSingle` with:
   - `params.tokenIn = USDC`
   - `params.amountIn = 1000e6`
   - `params.amountOutMinimum = 0` (or any value ≤ actual output)
   - `params.recipient = address(0)` ← mistake / uninitialized field

2. Router sets callback context and calls `pool.swap(address(0), ...)`. [6](#0-5) 

3. Pool executes the swap, computes output, and calls `token1.transfer(address(0), amountOut)` — succeeds on standard ERC-20.

4. Callback fires; router pulls 1000 USDC from `msg.sender` and pays the pool. [7](#0-6) 

5. `amountOut >= params.amountOutMinimum` passes. Function returns successfully.

6. User has spent 1000 USDC and received zero tokens. Output is permanently at `address(0)`.
