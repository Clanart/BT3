The `pay` function in `PeripheryPayments.sol` is the critical code to analyze. Let me trace the exact path:

### Title
Stranded ETH on Router Silently Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay` uses `address(this).balance` — the router's **entire** native ETH balance — when partially covering a WETH payment. Any ETH left on the router by a prior user (e.g., from a `multicall` that omitted `refundETH`) is silently wrapped and forwarded to the pool as part of the next user's payment, causing the prior user to lose their ETH and the next user to have their WETH pull reduced by the stolen amount.

---

### Finding Description

The hybrid WETH branch in `pay` reads the router's global native balance:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {                  // ← partial-ETH branch
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

There is no accounting of *who* deposited the ETH currently sitting on the router. `address(this).balance` aggregates ETH from all sources.

ETH can be stranded on the router because `multicall` is `payable`:

```solidity
function multicall(bytes[] calldata data) public payable override returns (bytes[] memory results) {
``` [2](#0-1) 

The `receive()` guard only blocks bare ETH transfers (no calldata); it does **not** prevent ETH from being deposited via a `payable` function call:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [3](#0-2) 

So ETH sent via `multicall{value: X}(...)` lands on the router and stays there if `refundETH` is not included in the same multicall batch.

The swap callback path that triggers `pay` is:

```solidity

### Impact Explanation

**Direct loss of user principal (HIGH).**

- **User A** loses their stranded ETH — it is wrapped and forwarded to the pool as part of User B's payment. User A receives nothing in return.
- **User B** has their WETH `safeTransferFrom` pull reduced by the amount of User A's ETH consumed. User B effectively pays less than the pool is owed from their own funds, but the pool is made whole by User A's ETH — so the pool is not insolvent, but User A is robbed.

The loss is permanent and proportional to the stranded ETH balance. Any amount of ETH left on the router (from any prior user) is at risk.

---

### Likelihood Explanation

**Medium.** The trigger condition — ETH stranded on the router — is a well-known user mistake with `payable` multicall routers (identical to the Uniswap v3 periphery class of issues). Users routinely:
- Send ETH for a WETH swap but omit `refundETH` from the multicall batch.
- Send excess ETH expecting the router to refund automatically.
- Have a swap partially fill, leaving residual ETH.

Any subsequent WETH swap by any user will consume the stranded ETH. No special permissions or coordination are required by the attacker — they simply call `exactInputSingle` with `tokenIn = WETH` while the router holds ETH.

---

### Recommendation

Track only the ETH that belongs to the current call. The standard fix is to pass `msg.value` through to `pay` and use it as the cap for native ETH consumption, rather than reading `address(this).balance`. Concretely:

1. Thread `msg.value` from `exactInputSingle` / `exactInput` / `exactOutput*` into the transient callback context.
2. In `pay`, replace `address(this).balance` with the stored `msg.value` for the current call.
3. Alternatively, enforce that `refundETH` is always called at the end of any payable multicall by checking `address(this).balance == 0` after each top-level call, or by making the router non-payable for non-ETH swaps.

---

### Proof of Concept

```
1. User A calls multicall{value: 0.5 ether}([exactInputSingle(tokenA→tokenB, ...)])
   — omits refundETH; 0.5 ETH is stranded on the router.

2. User B calls exactInputSingle{value: 0}(WETH→tokenC, amountIn=1e18)
   — User B has approved the router for 1e18 WETH.

3. Pool calls metricOmmSwapCallback → _justPayCallback →
   pay(WETH, userB, pool, 1e18)

4. Inside pay:
   nativeBalance = address(this).balance = 0.5e18  (User A's ETH)
   0 < nativeBalance < value  → partial-ETH branch fires:
     deposit(0.5e18)  ← User A's ETH wrapped
     safeTransfer(pool, 0.5e18)
     safeTransferFrom(userB, pool, 0.5e18)  ← only 0.5 WETH pulled from B

5. Result:
   - User A: lost 0.5 ETH, received nothing.
   - User B: paid 0.5 WETH instead of 1 WETH (discount funded by User A).
   - Pool: received 1e18 WETH-equivalent (correct), unaware of the theft.

assert router.balance == 0          // User A's ETH is gone
assert userA_eth_loss == 0.5 ether  // confirmed theft
assert userB_weth_spent == 0.5e18   // B underpaid by 0.5e18
```
