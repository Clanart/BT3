### Title
`PeripheryPayments.pay()` WETH branch consumes unattributed router ETH balance, enabling cross-user ETH theft via WETH swap - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

---

### Summary

The `pay()` internal function's WETH settlement branch reads `address(this).balance` — the router's **total** native ETH balance — to decide how much ETH to deposit as WETH versus how much WETH to pull from the payer via `transferFrom`. Because this balance is not scoped to the current caller's `msg.value`, any ETH stranded on the router from a prior transaction is silently consumed to settle a later user's WETH obligation, causing the prior user to lose their ETH while the later user receives a free or discounted swap.

---

### Finding Description

In `PeripheryPayments.pay()`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← reads ALL router ETH, not just msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`address(this).balance` is the router's aggregate native ETH balance. It includes:
- ETH sent by the **current** caller via `msg.value` in a payable swap call, and
- ETH **stranded** from any prior caller who sent excess `msg.value` without a subsequent `refundETH()`.

The `receive()` guard only blocks direct ETH transfers from non-WETH senders; it does not prevent ETH from accumulating via `msg.value` on payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). [2](#0-1) 

When User A calls `exactInputSingle{value: X}` with `amountIn < X` and omits `refundETH()`, the surplus `X - amountIn` ETH remains on the router. The next WETH-input swap by any User B will find `nativeBalance >= value` and deposit User A's ETH to settle User B's obligation — without pulling a single token from User B's WETH allowance.

The payer stored in transient context is `msg.sender` of the swap call: [3](#0-2) 

But `pay()` ignores the payer's identity when deciding the ETH/WETH split; it only checks the router's aggregate balance. [4](#0-3) 

---

### Impact Explanation

**Direct loss of user principal — High.**

User A's stranded ETH is consumed to settle User B's pool payment. User A receives nothing in return; their ETH is permanently transferred to the pool as WETH on behalf of User B. User B receives the full swap output without spending any WETH. The loss is 1:1 with the stranded ETH amount and is bounded only by how much ETH a user over-sends.

Additionally, any third party can call the public `refundETH()` helper to drain the same stranded ETH directly to themselves: [5](#0-4) 

Both vectors — the `pay()` silent consumption and the `refundETH()` direct drain — result in the same principal loss for User A.

---

### Likelihood Explanation

**Medium.**

The intended usage pattern (send ETH via `multicall{value}`, include `refundETH()` as the last call) is documented in tests but not enforced on-chain. Users who call `exactInputSingle{value: X}` directly — a natural pattern for a single-hop ETH-input swap — with any `msg.value` excess will strand ETH. The payable entry points accept arbitrary `msg.value` with no on-chain enforcement that it equals `amountIn`: [6](#0-5) 

A front-running bot monitoring the mempool for transactions that leave ETH on the router can atomically follow with a WETH swap or `refundETH()` call to extract the value.

---

### Recommendation

Track the current transaction's `msg.value` contribution separately from the router's aggregate balance. One approach: pass the caller-attributed ETH budget into `pay()` explicitly rather than reading `address(this).balance`:

```solidity
// In exactInputSingle / exactInput, capture msg.value before the swap:
uint256 ethBudget = msg.value;

// Pass it through to pay():
function pay(address token, address payer, address recipient, uint256 value, uint256 ethBudget) internal {
    ...
    } else if (token == WETH) {
        uint256 nativeBalance = ethBudget;  // only the caller's own ETH
        ...
    }
}
```

Alternatively, enforce that `address(this).balance == 0` at the start of every non-multicall entry point (revert if residue exists), or add a `nonReentrant`-style guard that snapshots and restores the ETH balance attribution per call.

### Proof of Concept

```
Setup:
  - Router deployed with WETH and a valid pool (WETH / token1).
  - Pool has liquidity seeded.

Step 1 — User A strands ETH:
  vm.prank(userA);
  // amountIn = 1_000, but msg.value = 2_000 (over-sends by 1_000)
  router.exactInputSingle{value: 2_000}(ExactInputSingleParams({
      pool: pool,
      tokenIn: WETH,
      tokenOut: token1,
      zeroForOne: true,
      amountIn: 1_000,
      amountOutMinimum: 0,
      recipient: userA,
      deadline: ...,
      priceLimitX64: 0,
      extensionData: ""
  }));
  // pay() sees nativeBalance=2_000 >= value=1_000 → deposits 1_000 ETH, 1_000 ETH stranded.
  assert(address(router).balance == 1_000);

Step 2 — User B exploits stranded ETH:
  // User B has 0 WETH approved to router, 0 msg.value
  vm.prank(userB);
  uint256 wethBefore = IERC20(WETH).balanceOf(userB);
  router.exactInputSingle(ExactInputSingleParams({
      pool: pool,
      tokenIn: WETH,
      tokenOut: token1,
      zeroForOne: true,
      amountIn: 1_000,
      amountOutMinimum: 0,
      recipient: userB,
      deadline: ...,
      priceLimitX64: 0,
      extensionData: ""
  }));
  // pay(WETH, userB, pool, 1_000):
  //   nativeBalance = 1_000 (User A's ETH) >= value = 1_000
  //   → deposits User A's 1_000 ETH as WETH, transfers to pool
  //   → User B's WETH never pulled
  assert(IERC20(WETH).balanceOf(userB) == wethBefore);  // User B spent 0 WETH
  assert(token1.balanceOf(userB) > 0);                  // User B received swap output
  assert(address(router).balance == 0);                  // User A's ETH consumed
```

User A loses 1,000 wei of ETH. User B receives the full swap output at zero cost.
