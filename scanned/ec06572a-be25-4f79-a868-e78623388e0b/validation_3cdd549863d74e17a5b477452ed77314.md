### Title
Unbounded `newTimelock` in `proposeOracleStopLossTimelock()` Causes uint32 Overflow, Bypassing All Subsequent Timelocks — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`proposeOracleStopLossTimelock()` accepts any `uint32 newTimelock` with no upper-bound validation. Setting it to `type(uint32).max` causes a silent uint32 truncation in `_afterTimelock()`, making every subsequent parameter proposal (drawdown, decay, watermarks) immediately executable — defeating the timelock protection that LPs rely on to exit before adverse parameter changes take effect.

---

### Finding Description

`proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, and `proposeOracleStopLossHighWatermarks` all compute their `executeAfter` deadline via `_afterTimelock()`:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

The addition is performed in `uint256` space, then truncated to `uint32`. When `timelock = type(uint32).max` (≈ 4.29 × 10⁹):

```
block.timestamp (≈ 1.75e9) + type(uint32).max (4.29e9) = 6.04e9
uint32(6.04e9) = 6.04e9 mod 2^32 ≈ 1.75e9 − 1
```

The resulting `executeAfter` is **one second in the past**, so `_requireElapsed` passes immediately for every subsequent proposal:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

`proposeOracleStopLossTimelock` itself has **no validation** on `newTimelock`, unlike `proposeOracleStopLossDrawdown` (which calls `_validateDrawdown`) and `proposeOracleStopLossDecay` (which calls `_validateDecay`):

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
}
``` [3](#0-2) 

Compare with the validated parameters:

```solidity
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}
function _validateDecay(uint256 decayPerSecondE8) private pure {
    if (decayPerSecondE8 > E8) revert OracleStopLossDecayTooLarge(decayPerSecondE8);
}
``` [4](#0-3) 

The same overflow applies at initialization — `initialize()` also accepts an arbitrary `uint32 timelock` with no upper-bound check:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock: no validation
``` [5](#0-4) 

---

### Impact Explanation

Once `timelock = type(uint32).max` is active, the pool admin can atomically:

1. Call `proposeOracleStopLossDrawdown(pool, 0)` — drawdown floor = 0% (any value drop triggers stop-loss).
2. Immediately call `executeOracleStopLossDrawdown(pool)` — no wait required.
3. The next swap hits `afterSwap`, which calls `_checkAndUpdateWatermarks`, which reverts with `OracleStopLossTriggered`.

All swaps through the pool are permanently blocked. LPs cannot exit via swaps and must call `removeLiquidity` directly. This constitutes **unusable swap flows** — an allowed impact under the contest gate.

The pool admin can also immediately raise watermarks to `type(uint104).max` via `proposeOracleStopLossHighWatermarks` + `executeOracleStopLossHighWatermarks`, guaranteeing every future swap triggers the stop-loss.

---

### Likelihood Explanation

- The pool admin is a semi-trusted role (distinct from the factory owner).
- The timelock is the primary protection mechanism LPs have against sudden parameter changes; the NatSpec explicitly states *"Drawdown and decay changes are timelocked so LPs can react."*
- A pool initialized with `timelock = type(uint32).max` (appearing to offer maximum protection) is immediately exploitable from block 1.
- A pool with a non-zero timelock (e.g., 7 days) is exploitable after the current timelock elapses — LPs have a window to notice, but the overflow is not visible from the emitted event value (`type(uint32).max` looks like a very long delay).

---

### Recommendation

Add an upper-bound check in `proposeOracleStopLossTimelock` and in `initialize`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days; // example

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
    ...
}
```

Apply the same guard in `initialize` before storing the initial timelock. This mirrors the existing pattern for `drawdownE6` and `decayPerSecondE8`.

```solidity
// Pool initialized with timelock = type(uint32).max (appears maximally protective)
extension.initialize(pool, abi.encode(uint32(500_000), uint32(58), type(uint32).max));
