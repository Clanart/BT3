### Title
Unguarded `refundETH()` / `sweepToken()` / `unwrapWETH9()` Allow Any Caller to Steal ETH and Tokens Stranded on the Router Between Transactions — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.refundETH()`, `sweepToken()`, and `unwrapWETH9()` carry no access control and accept caller-chosen `recipient` addresses. The `pay()` helper simultaneously consumes the router's **entire** native ETH balance for any WETH-leg payment, without per-user attribution. Any ETH or ERC-20 residue left on the router after a payable entry-point (e.g., `exactOutputSingle{value: X}`) is immediately claimable by an unrelated caller in the next transaction.

---

### Finding Description

`PeripheryPayments.pay()` handles WETH-leg settlement by inspecting `address(this).balance` and wrapping whatever native ETH the router currently holds before pulling the remainder from the payer:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

There is no per-user accounting. Any ETH sitting on the router from a prior transaction is silently consumed for the next caller's WETH payment.

The three public helpers that drain the router's balance are completely unguarded:

```solidity
// L37-63
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override { ... }
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override { ... }
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) { _transferETH(msg.sender, balance); }
}
``` [2](#0-1) 

Any caller can invoke `refundETH()` to receive all ETH on the router, or `sweepToken(WETH, 0, attacker)` / `unwrapWETH9(0, attacker)` to drain all WETH. Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` inherit this contract. [3](#0-2) [4](#0-3) 

**How ETH becomes stranded:** `exactOutputSingle` and `exactInputSingle` are both `external payable`. A user who wants to swap native ETH for a token calls `exactOutputSingle{value: amountInMaximum}` because the exact input is unknown upfront. The `pay()` callback wraps only the realized `amountIn`; the remainder (`amountInMaximum − amountIn`) stays on the router. The protocol's intended recovery is `refundETH()` inside the same `multicall`, but calling `exactOutputSingle` directly (without multicall) leaves a cross-transaction window. [5](#0-4) 

---

### Impact Explanation

**Direct loss of user ETH principal.** An attacker who observes a stranded-ETH state on the router (via mempool or on-chain balance check) calls `refundETH()` in the next block and receives the victim's excess ETH with no preconditions. The loss equals `msg.value − amountIn` for the victim's `exactOutputSingle` call — potentially the full `amountInMaximum` if the swap reverts after ETH is already held by the router. The same path applies to WETH output stranded by `exactInput(..., recipient = address(router))` without a subsequent `unwrapWETH9` in the same multicall.

---

### Likelihood Explanation

**Medium.** The multicall+`refundETH` pattern is the documented safe path, but:
- `exactOutputSingle` and `exactInputSingle` are independently callable `external payable` functions — nothing prevents direct invocation with excess ETH.
- A user who calls `exactOutputSingle{value: X}` and then issues `refundETH()` in a separate transaction faces a front-running window that any searcher can exploit.
- The `MetricOmmPoolLiquidityAdder` exposes the same surface via its own `addLiquidityExactShares{value: X}` payable entry-point. [6](#0-5) 

---

### Recommendation

1. **Auto-refund excess ETH** at the end of every `external payable` entry-point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `addLiquidityExactShares`, `addLiquidityWeighted`) rather than relying on the caller to include `refundETH` in a multicall.
2. **Restrict `refundETH`** to only be callable via `delegatecall` from `multicall` (e.g., check `address(this) == implementation`), so it cannot be invoked as a standalone cross-transaction drain.
3. **Restrict `sweepToken` and `unwrapWETH9`** similarly, or require `recipient == msg.sender` to prevent arbitrary redirection of stranded balances.

---
