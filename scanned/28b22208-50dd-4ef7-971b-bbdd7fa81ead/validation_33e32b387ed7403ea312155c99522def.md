### Title
Router's `pay()` Consumes Unattributed Native ETH Balance, Enabling Cross-Transaction Theft of Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay()` settles WETH swap obligations by first consuming `address(this).balance` — the router's entire native ETH balance — without verifying that the ETH belongs to the current payer. Any ETH stranded on the router from a prior transaction (e.g., excess `msg.value` sent without a paired `refundETH()`) is silently consumed to pay a subsequent caller's swap, or can be directly stolen via the public `refundETH()` helper.

### Finding Description

In `PeripheryPayments.pay()`, when `token == WETH` and `payer != address(this)`, the function checks `address(this).balance` and uses it to fund the WETH deposit before falling back to `safeTransferFrom(payer, ...)`: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` is the router's **total** native ETH balance — it is not scoped to the current transaction's `msg.value`. If a prior transaction left ETH on the router (because the user sent excess `msg.value` without including `refundETH()` in the multicall), that ETH is indistinguishable from the current caller's ETH and is consumed first.

The `refundETH()` helper compounds this: [2](#0-1) 

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

`refundETH()` is public, takes no arguments, and sends the router's **entire** ETH balance to `msg.sender`. There is no check that the caller deposited any of that ETH. Any ETH stranded from a prior user's transaction is immediately claimable by any subsequent caller.

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses: [3](#0-2) 

It does **not** prevent ETH from accumulating via `msg.value` in payable swap functions (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `multicall`). ETH sent as `msg.value` that is not fully consumed by `pay()` persists on the router across transaction boundaries.

The transient callback context (`TransientCallbackPool`) correctly binds the expected pool and payer per-swap: [4](#0-3) 

But it does not track how much native ETH the current payer contributed, so `pay()` has no way to limit itself to the current caller's ETH.

The test suite acknowledges the pattern but only tests the happy path where the user correctly pairs excess ETH with `refundETH()`: [5](#0-4) 

No test covers the cross-transaction theft scenario.

### Impact Explanation

**Direct loss of user principal.** Alice's stranded ETH is either:

1. Consumed by Bob's swap via `pay()` — Bob receives output tokens without paying any input. Alice's ETH is permanently transferred to the pool as Bob's payment.
2. Directly stolen by Bob via `refundETH()` — Bob receives Alice's ETH with a single public call.

Both paths result in a complete, irreversible loss of Alice's stranded ETH with no protocol recourse.

### Likelihood Explanation

The trigger is realistic and common:

- **Exact-output WETH swaps**: `exactOutputSingle` and `exactOutput` with WETH as input require the user to send `amountInMaximum` as `msg.value` because the actual `amountIn` is determined by the pool at execution time. The difference `amountInMaximum - amountIn` is stranded unless `refundETH()` is included in the same multicall.
- **Integrators and wallets**: Any integration that calls `exactOutputSingle{value: amountInMaximum}(...)` directly (not via multicall) will strand ETH on every successful swap.
- **Attacker cost**: Zero. Calling `refundETH()` costs only gas. The attacker can monitor the mempool or the router's ETH balance and front-run or back-run any stranding transaction.

### Recommendation

Track the native ETH contributed by the current transaction in a transient storage slot at the entry of each payable swap function. In `pay()`, limit the native ETH consumed to `min(nativeBalance, trackedMsgValue)` and decrement the tracker. Alternatively, enforce that `pay()` only uses native ETH equal to the current transaction's `msg.value` by storing it in transient storage at function entry and zeroing it after use. This mirrors the fix pattern for the analogous Uniswap v3 periphery issue.

### Proof of Concept

**Setup**: Pool with WETH/token1. Alice and Bob are separate EOAs.

**Step 1 — Alice strands ETH (realistic exact-output scenario):**
```solidity
// Alice wants exactly 1000 token1 but doesn't know the exact WETH cost.
// She sends amountInMaximum = 2000 as msg.value.
router.exactOutputSingle{value: 2000}(ExactOutputSingleParams({
    pool: pool,
    tokenIn: WETH,
    tokenOut: token1,
    zeroForOne: true,
    amountOut: 1000,
    amountInMaximum: 2000,
    recipient: alice,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Pool determines amountIn = 1500.
// pay() deposits 1500 ETH → WETH → pool.
// 500 ETH remains on the router. Alice did not include refundETH().
```

**Step 2 — Bob steals Alice's stranded ETH (direct theft):**
```solidity
// Bob calls refundETH() in the next block.
router.refundETH(); // Bob receives 500 ETH. Alice's ETH is gone.
```

**Alternative Step 2 — Bob gets a free swap:**
```solidity
// Bob calls exactInputSingle with msg.value = 0.
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 500,
    amountOutMinimum: 0,
    recipient: bob,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// pay() sees address(this).balance = 500 >= 500.
// Deposits Alice's 500 ETH → WETH → pool.
// Bob receives token1 output. Bob paid nothing.
```

**Invariant broken**: `pay(WETH, payer, pool, value)` must only consume ETH that `payer` deposited in the current transaction. Here it consumes ETH deposited by a different user in a prior transaction, violating payer attribution and causing direct loss of Alice's principal.
