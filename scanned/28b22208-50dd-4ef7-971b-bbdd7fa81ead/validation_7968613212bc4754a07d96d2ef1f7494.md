### Title
Payable swap and liquidity-adder entry points silently trap ETH on non-WETH paths, allowing any caller to steal it via `refundETH()` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`, `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

Every user-facing entry point in `MetricOmmSimpleRouter` (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and `MetricOmmPoolLiquidityAdder` (`addLiquidityExactShares`, `addLiquidityWeighted`) carries the `payable` modifier. The internal `pay()` helper in `PeripheryPayments` only consumes native ETH when `token == WETH`; for every other token it falls through to `IERC20.safeTransferFrom` and leaves `address(this).balance` untouched. Any ETH sent with a non-WETH call is silently retained by the contract. Because `refundETH()` sends the entire contract balance to whoever calls it — not to the original depositor — a third party can immediately drain the trapped ETH.

---

### Finding Description

`PeripheryPayments.pay()` branches on the token address:

```solidity
// PeripheryPayments.sol lines 69-88
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
        uint256 nativeBalance = address(this).balance;
        if (nativeBalance >= value) {
            IWETH9(WETH).deposit{value: value}();
            IERC20(WETH).safeTransfer(recipient, value);
        } else if (nativeBalance > 0) {
            IWETH9(WETH).deposit{value: nativeBalance}();
            IERC20(WETH).safeTransfer(recipient, nativeBalance);
            IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
        } else {
            IERC20(WETH).safeTransferFrom(payer, recipient, value);
        }
    } else {
        // ← ETH is completely ignored here
        IERC20(token).safeTransferFrom(payer, recipient, value);
    }
}
```

When `token != WETH`, the final `else` branch executes a plain ERC-20 pull and never touches `address(this).balance`. Any ETH the caller attached to the transaction accumulates in the contract.

`refundETH()` then sends the full balance to whoever calls it:

```solidity
// PeripheryPayments.sol lines 58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← goes to caller, not original depositor
    }
}
```

There is no `require(msg.value == 0)` guard on any non-WETH code path in either contract. The `msg.value` search across the entire codebase confirms it never appears in the router or liquidity-adder production code.

---

### Impact Explanation

A user who calls `exactInputSingle` (or any other `payable` entry point) with a non-WETH `tokenIn` and accidentally attaches ETH loses that ETH permanently from their perspective: it sits in the router until any third party calls `refundETH()` and collects it. The loss is a direct reduction of the victim's principal with no recovery path once the ETH is swept. The same applies to `addLiquidityExactShares` and `addLiquidityWeighted` when neither pool token is WETH.

---

### Likelihood Explanation

The functions are declared `payable` and are the primary user-facing entry points. Wallets, aggregators, and SDK wrappers that batch calls via `multicall` may forward ETH intended for a WETH hop to a subsequent non-WETH call in the same batch. User error (sending ETH to a non-WETH swap) is a well-documented class of mistake, as acknowledged in the original M-05 report. No special privilege or malicious setup is required; any unprivileged caller can trigger the loss and any unprivileged caller can steal the trapped ETH.

---

### Recommendation

Add a zero-value guard on every code path where native ETH is not consumed. The minimal fix mirrors the M-05 mitigation:

```solidity
// In exactInputSingle, exactInput, exactOutputSingle, exactOutput
// when tokenIn != WETH (or when the first token in the path is not WETH):
if (params.tokenIn != WETH) require(msg.value == 0, "ETH not accepted for ERC20 swap");

// In addLiquidityExactShares / addLiquidityWeighted
// when neither pool token is WETH:
require(msg.value == 0, "ETH not accepted for ERC20 liquidity");
```

Alternatively, remove `payable` from functions that have no WETH-wrapping code path, or add an automatic `refundETH()` call at the end of each entry point (analogous to Uniswap v3's recommended `multicall([swap(...), refundETH()])` pattern enforced at the contract level).

---

### Proof of Concept

1. Alice calls `exactInputSingle` with `tokenIn = USDC`, `amountIn = 1000e6`, and accidentally attaches `1 ether` as `msg.value`.
2. Inside the swap callback, `pay(USDC, Alice, pool, 1000e6)` is invoked. Because `USDC != WETH`, the `else` branch fires: `IERC20(USDC).safeTransferFrom(Alice, pool, 1000e6)`. The `1 ether` is never touched.
3. The swap completes successfully. Alice's `1 ether` now sits in `MetricOmmSimpleRouter`.
4. Bob, an unrelated address, calls `refundETH()` in the next block. `address(this).balance == 1 ether`, so `_transferETH(Bob, 1 ether)` executes. Bob receives Alice's ETH.
5. Alice has no recourse; the ETH is gone.

The same sequence applies to `exactInput`, `exactOutputSingle`, `exactOutput`, `addLiquidityExactShares`, and `addLiquidityWeighted` whenever the input token(s) are not WETH. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
