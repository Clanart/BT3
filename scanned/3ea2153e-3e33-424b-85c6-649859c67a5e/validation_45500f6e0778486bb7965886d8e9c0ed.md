### Title
Unguarded `refundETH()` allows any caller to drain excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` has no access control and unconditionally sends the router's entire ETH balance to `msg.sender`. Because `pay()` only wraps the exact amount of ETH needed for a swap, any excess ETH sent by a user via `multicall{value: ...}` remains on the router after the swap completes. A separate transaction from any address can then call `refundETH()` and claim that ETH.

---

### Finding Description

`refundETH()` is implemented as:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // no caller check
    }
}
``` [1](#0-0) 

There is no check that `msg.sender` is the address that originally deposited the ETH. The function is `external` with no `onlyOwner`, no transient-storage payer check, and no reentrancy guard.

ETH accumulates on the router because `pay()` wraps only the exact amount needed:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // wraps exactly `value`, not all ETH
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [2](#0-1) 

`multicall` is `payable`, so users routinely send ETH with it:

```solidity
function multicall(bytes[] calldata data) public payable override returns (bytes[] memory results) {
``` [3](#0-2) 

The `receive()` guard (which rejects ETH from non-WETH senders) does **not** apply here because ETH arrives through the `payable` `multicall` function, not through a bare ETH transfer. [4](#0-3) 

---

### Impact Explanation

Any user who sends more ETH than the swap consumes (the normal pattern for ETH-input swaps) and omits `refundETH()` from their multicall loses the excess ETH permanently to the first external caller of `refundETH()`. This is a direct, irreversible loss of user principal with no protocol-side recovery path.

---

### Likelihood Explanation

- The pattern of sending a round ETH amount and relying on `refundETH()` to recover the remainder is the standard UX for ETH-input swaps on routers of this design.
- A user who calls `exactInputSingle` or `exactInput` in a single-call transaction (not a multicall) has no way to atomically reclaim excess ETH in the same transaction.
- MEV bots routinely monitor router balances; the window between the victim's swap and the attacker's `refundETH()` call is a single block.

---

### Recommendation

Restrict `refundETH()` so it can only be called within a `multicall` context, or record the depositing address in transient storage at `multicall` entry and enforce it inside `refundETH()`:

```solidity
// In multicall entry: tstore(REFUND_RECIPIENT_SLOT, msg.sender)
// In refundETH:
address depositor = address(uint160(tload(REFUND_RECIPIENT_SLOT)));
require(msg.sender == depositor || depositor == address(0), "not depositor");
_transferETH(depositor != address(0) ? depositor : msg.sender, balance);
```

Alternatively, document that `refundETH()` **must** always be the last call in every ETH-input multicall and add an integration test that asserts no ETH remains on the router after a well-formed multicall.

---

### Proof of Concept

1. User calls `router.multicall{value: 1 ether}([abi.encodeCall(router.exactInputSingle, (params))])` where `params.amountIn = 0.5 ether` worth of WETH.
2. `pay()` wraps 0.5 ETH → WETH and pays the pool. 0.5 ETH remains on the router.
3. User's multicall ends; no `refundETH()` was included.
4. Attacker calls `router.refundETH()` in the next transaction.
5. `_transferETH(msg.sender, 0.5 ether)` executes; attacker receives 0.5 ETH.
6. Assert: attacker balance increased by 0.5 ETH; user's 0.5 ETH is gone.
