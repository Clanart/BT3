### Title
Residual ETH in Router Consumed by Subsequent User's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` function's hybrid WETH path unconditionally consumes the router's entire native ETH balance when funding a WETH payment. Because all swap entry-points are `payable` and no automatic ETH refund occurs at the end of a swap, ETH sent by one user can persist in the router across transaction boundaries and be silently consumed to fund a completely different user's WETH payment.

---

### Finding Description

`PeripheryPayments.pay()` contains three branches for `token == WETH`: [1](#0-0) 

When `0 < nativeBalance < value`, the router wraps its **entire** native ETH balance and sends it to the pool, then pulls only `value - nativeBalance` from `payer`'s WETH allowance (line 81). The function does not verify that the native ETH balance belongs to the current caller.

ETH can accumulate in the router across transactions because:

1. Every swap entry-point is `payable`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

2. `receive()` only blocks plain ETH transfers (no calldata); ETH sent alongside a function call is accepted silently: [6](#0-5) 

3. `refundETH()` is a separate, optional call — no swap function automatically returns excess ETH: [7](#0-6) 

**Concrete residual-ETH vectors:**
- User A calls `exactInputSingle{value: 2 ETH}(...)` with WETH as `tokenIn` and `amountIn = 1 ETH`. The `pay()` `nativeBalance >= value` branch wraps exactly 1 ETH, leaving 1 ETH in the router.
- User A calls any swap with a non-WETH token but accidentally attaches ETH. The ETH is entirely unused and stays in the router.
- User A calls `multicall{value: X}(...)` without including a `refundETH()` sub-call.

Once ETH is stranded, the next user whose WETH payment triggers the `nativeBalance > 0 && nativeBalance < value` branch will have User A's ETH wrapped and forwarded to the pool on their behalf, reducing the WETH pulled from their own allowance. User A's ETH is irrecoverably consumed.

---

### Impact Explanation

**Direct loss of user principal.** User A's ETH is transferred to the pool to settle User B's swap. User A receives nothing in return; the ETH cannot be recovered via `refundETH()` because it has already been wrapped and forwarded. The pool receives the correct total, so no pool-side invariant is broken, but the payer-attribution invariant is violated: User B's payment is partially funded by User A's funds.

---

### Likelihood Explanation

Sending ETH with a payable router call without a paired `refundETH()` is a common integration mistake (e.g., frontend over-estimates ETH needed, user wraps a multicall without a cleanup step, or a non-WETH swap is called with ETH attached). The router provides no on-chain protection against this. Any subsequent WETH swap of sufficient size will silently drain the residual balance.

---

### Recommendation

In `pay()`, when `token == WETH` and `payer != address(this)`, only use native ETH that was explicitly sent in the **current transaction** (i.e., `msg.value`). The simplest fix is to replace `address(this).balance` with `msg.value` (passed down from the entry-point), or to zero-out the native-ETH hybrid path entirely and require callers to pre-wrap ETH before calling the router. Alternatively, add an automatic `refundETH()` at the end of every swap entry-point to ensure no ETH persists across transaction boundaries.

function test_staleEthConsumedBySubsequentWethSwap() public {
    // 1. User A calls exactInputSingle for a non-WETH pair, accidentally attaching 1 ETH.
    //    The ETH is unused; 1 ETH remains in the router.
    vm.deal(userA, 1 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
        pool: nonWethPool,
        tokenIn: token1,   // NOT WETH
        ...
        amountIn: 1_000,
        ...
    }));
    assertEq(address(router).balance, 1 ether); // ETH stuck

    // 2. User B swaps WETH -> token2, needing to pay 2 WETH.
    //    pay(WETH, userB, pool, 2e18) sees nativeBalance=1e18 > 0 and < 2e18.
    //    Router wraps 1 ETH (User A's) + pulls 1 WETH from User B.
    uint256 userAEthBefore = userA.balance; // already 0 (spent on step 1)
    uint256 userBWethBefore = weth.balanceOf(userB);

    vm.prank(userB);
    router.exactInputSingle(ExactInputSingleParams({
        pool: wethPool,
        tokenIn: address(weth),
        amountIn: 2 ether,
        ...
    }));

    // User A's 1 ETH is gone — consumed for User B's swap
    assertEq(address(router).balance, 0);
    // User B only had 1 WETH pulled (not 2), benefiting from User A's ETH
    assertEq(userBWethBefore - weth.balanceOf(userB), 1 ether);
}
