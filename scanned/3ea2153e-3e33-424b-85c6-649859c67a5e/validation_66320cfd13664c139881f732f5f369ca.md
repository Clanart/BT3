### Title
`address(this).balance` Used Instead of Per-Transaction Tracked ETH in `pay()`, Allowing Leftover Native ETH to Subsidize Subsequent Users' WETH Obligations — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` helper in `PeripheryPayments` reads the router/adder's full `address(this).balance` to decide how much native ETH to wrap and forward as WETH. Because that balance is not a per-transaction tracked value — it accumulates across calls — any ETH left on the contract by a prior user (e.g., from a `multicall{value: X}` that omitted `refundETH`) is silently consumed to cover a later user's WETH obligation. This is the direct analog of the Yearn bug: `balanceOf(yVault)` (inflatable) was used instead of `totalIdle` (tracked); here `address(this).balance` (inflatable) is used instead of the ETH specifically allocated to the current transaction.

---

### Finding Description

In `PeripheryPayments.pay()`, the WETH branch reads the contract's entire native balance:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← full contract balance, not per-tx
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`address(this).balance` is not a per-transaction counter. It is the aggregate of:
- ETH sent with the current `msg.value`, **plus**
- Any ETH stranded from prior calls (e.g., a user who called `exactInputSingle{value: 1 ETH}` with `amountIn = 0.5 ETH` and omitted `refundETH`).

The `receive()` guard only blocks direct non-WETH ETH pushes; it does not prevent accumulation via `payable` entry points (`exactInputSingle`, `exactOutput`, `addLiquidityExactShares`, `multicall`). [2](#0-1) 

When `nativeBalance` is inflated by a prior user's residue, the current `payer`'s `safeTransferFrom` pull is reduced or eliminated entirely — the prior user's ETH is silently consumed to settle the current user's debt.

---

### Impact Explanation

**Concrete loss scenario:**

1. User A calls `exactInputSingle{value: 1 ETH}` with `tokenIn = WETH`, `amountIn = 0.5 ETH`, and no `refundETH` step.
2. `pay(WETH, userA, pool, 0.5 ETH)` fires: `address(this).balance = 1 ETH ≥ 0.5 ETH`, so 0.5 ETH is wrapped and forwarded. The remaining **0.5 ETH stays on the router**.
3. User B calls `exactInputSingle` (zero `msg.value`) with `tokenIn = WETH`, `amountIn = 0.3 ETH`.
4. `pay(WETH, userB, pool, 0.3 ETH)` fires: `address(this).balance = 0.5 ETH ≥ 0.3 ETH`, so **0.3 ETH of User A's residue** is wrapped and forwarded. User B's `safeTransferFrom` is never called.
5. User A loses 0.3 ETH beyond what they intended to spend. User B receives a fully subsidized swap.

The pool receives the correct WETH amount in both cases, so pool solvency is unaffected. The loss is entirely borne by the user whose ETH was stranded.

The same path applies to `MetricOmmPoolLiquidityAdder`, which inherits `PeripheryPayments` and exposes the same `payable` entry points (`addLiquidityExactShares`, `addLiquidityWeighted`, `multicall`). [3](#0-2) 

---

### Likelihood Explanation

- Users routinely send `msg.value` with WETH-input swaps or liquidity adds to avoid a separate WETH-wrap step, and frequently omit `refundETH` when the exact amount consumed is not known in advance.
- The trigger is fully unprivileged: any address can call `exactInputSingle` or `addLiquidityExactShares` with zero `msg.value` and WETH as the input token to drain whatever ETH is currently stranded on the contract.
- No special timing or oracle manipulation is required; the attacker simply monitors the router's ETH balance and submits a WETH-input swap for up to that amount.

---

### Recommendation

Track the ETH allocated to the current transaction in transient storage (consistent with the router's existing transient callback-context pattern). In `pay()`, read only the per-transaction tracked ETH rather than `address(this).balance`:

```solidity
// pseudocode
uint256 nativeBalance = _tload(T_SLOT_MSG_VALUE_REMAINING);
```

Alternatively, require callers to always pair WETH-input calls with an explicit `refundETH` and document that any ETH left on the contract between transactions is at risk — but this is a weaker mitigation because it relies on user discipline.

---

### Proof of Concept

```solidity
// 1. User A strands 0.5 ETH on the router
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    pool: pool,
    tokenIn: address(weth),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 0.5 ether,   // only 0.5 ETH consumed; 0.5 ETH left on router
    amountOutMinimum: 0,
    recipient: userA,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// router.balance == 0.5 ETH

// 2. User B drains User A's ETH with a zero-value call
// User B has WETH allowance = 0; no ETH sent
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: address(weth),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 0.3 ether,   // paid entirely from User A's stranded ETH
    amountOutMinimum: 0,
    recipient: userB,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// User B's swap succeeds; User A lost an additional 0.3 ETH
```

    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
