### Title
Swap Conservation Violation in `buyToken0InBinSpecifiedIn`: Trader Overcharged When Dual-Refinement Undershoots — (`metric-core/contracts/libraries/SwapMath.sol`)

---

### Summary

In `buyToken0InBinSpecifiedIn`, when both the analytical target and the first proportional-scaling refinement still undershoot (i.e., the recomputed `totalIn1Scaled` remains below `state.amountSpecifiedRemainingScaled` while `targetPos < maxFinalBinPos`), the fallback at lines 620–622 forcibly sets `totalIn1Scaled = state.amountSpecifiedRemainingScaled` **without updating `out0Scaled`**. The pool then consumes the full remaining input but delivers only the token0 output corresponding to the undershot `targetPos`, violating the swap conservation invariant.

---

### Finding Description

The function uses a two-pass refinement strategy to find the correct `targetPos`:

**Pass 1 (analytical)** — lines 581–596: Computes `targetPos` analytically, then derives `out0Scaled` and `totalIn1Scaled`. [1](#0-0) 

**Pass 2 (proportional scaling)** — lines 598–618: If `totalIn1Scaled < remaining && targetPos < maxFinalBinPos`, scales `targetPos` proportionally upward. If `scaledTarget > maxFinalBinPos`, it clamps to `maxFinalBinPos` (line 606), which makes the line-620 condition false. But if `scaledTarget <= maxFinalBinPos`, `targetPos = scaledTarget` and `out0Scaled`/`totalIn1Scaled` are recomputed. [2](#0-1) 

**Fallback (lines 620–622)**: If after Pass 2 the condition still holds, the code sets `totalIn1Scaled = remaining` but **leaves `out0Scaled` unchanged** at the value for the undershot `targetPos`. [3](#0-2) 

After line 621, `totalIn1Scaled == remaining`, so the proportional-scale-down block at line 624 is skipped (its condition `totalIn1Scaled > remaining` is false). Settlement then proceeds: [4](#0-3) 

- `state.amountSpecifiedRemainingScaled -= totalIn1Scaled` → full input consumed
- `state.amountCalculatedScaled += out0Scaled` → only the undershot token0 output credited

The pool receives `remaining` token1 but releases only `out0Scaled` for `targetPos < maxFinalBinPos`. The correct output for paying `remaining` would correspond to a position strictly above `targetPos` (since `remaining > totalIn1Scaled` for `targetPos`). The difference is silently absorbed by the pool.

---

### Impact Explanation

This is a **swap conservation failure** causing **direct loss of trader principal**. The trader pays the full exact-input amount but receives fewer token0 than the bin curve permits for that payment. The pool's token1 balance is inflated by the excess input, which is not owed to LPs or the protocol under the correct curve math — it is simply taken from the trader. The magnitude of the loss scales with the degree of undershoot, which grows with bin non-linearity (large `lengthE6`, extreme price ratios, or high fee settings).

---

### Likelihood Explanation

The condition at line 620 fires when:
1. The analytical solution undershoots (known to occur with non-linear bin curves).
2. The proportional scaling at lines 598–618 also undershoots — specifically when `scaledTarget <= maxFinalBinPos` (line 608–609 path is taken) but the recomputed `totalIn1Scaled` for `scaledTarget` is still below `remaining`.

This is reachable through the normal public swap path (`_swapToken1ForToken0SpecifiedInput` → `buyToken0InBinSpecifiedIn`) with no privileged access required. Any trader submitting an exact-input token1→token0 swap against a bin with sufficient non-linearity can trigger it. [5](#0-4) 

---

### Recommendation

Replace the fallback at lines 620–622 with a proper clamp to `maxFinalBinPos` and a full recomputation of `out0Scaled`:

```solidity
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    // Clamp to maxFinalBinPos and recompute output — do NOT just override totalIn1Scaled
    targetPos = maxFinalBinPos;
    out0Scaled = calculateOutputToken0FromBinPosition(binState.token0BalanceScaled, currBinPos, targetPos);
    finalPriceX64 = calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Ceil);
    avgPriceX64 = calculateArithmeticMean(startingPriceX64, finalPriceX64);
    in1WithoutFeeScaled = calculateRequiredToken(out0Scaled, avgPriceX64);
    totalIn1Scaled = grossInputWithBinFeeCeil(in1WithoutFeeScaled, onePlusBuyFeeX64);
}
```

This mirrors the logic at line 605–606 (the first-pass clamp) and ensures `out0Scaled` is always consistent with the charged `totalIn1Scaled`.

### Proof of Concept

Foundry property test sketch:

```solidity
function testFuzz_ConservationViolation_FallbackOvercharge(
    uint104 currBinPos,
    uint104 availableToken0,
    uint128 remainingIn,
    uint128 lowerPriceX64,
    uint128 upperPriceX64
) public {
    // Craft bin state where analytical + proportional both undershoot
    // (large bin spread, specific price ratio)
    SwapMath.SwapState memory state = SwapMath.SwapState({
        amountSpecifiedRemainingScaled: remainingIn, ...
    });

    (uint256 finalPos, uint256 out0,,,) = SwapMath.buyToken0InBinSpecifiedIn(
        binState, currBinPos, state, feeX64, lowerPriceX64, upperPriceX64, type(uint128).max, 0
    );

    uint256 consumed = remainingIn - state.amountSpecifiedRemainingScaled;

    // Conservation invariant: out0 * avgPrice >= consumed (trader gets fair value)
    // If lines 620-622 fired, out0 corresponds to targetPos < maxFinalBinPos
    // while consumed == remainingIn, violating this invariant
    uint256 fairOut0 = calculateOutputToken0FromBinPosition(
        availableToken0, currBinPos, type(uint104).max
    );
    // Assert: if consumed == remainingIn, out0 must equal fairOut0 (full bin consumed)
    if (consumed == remainingIn && finalPos < type(uint104).max) {
        assertGe(out0 * avgPrice, consumed << 64, "Conservation violated: trader overcharged");
    }
}
```

The assertion fails when the fallback at lines 620–622 fires: `consumed == remainingIn` but `out0` is for an undershot `targetPos`, so `out0 * avgPrice < consumed`.
