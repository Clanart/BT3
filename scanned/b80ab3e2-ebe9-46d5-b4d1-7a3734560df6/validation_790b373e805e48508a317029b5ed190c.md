### Title
Prior User's Stranded ETH Silently Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay` uses `address(this).balance` — the router's **total** native ETH balance — when deciding how much ETH to wrap for a WETH payment. Any ETH left on the router from a prior user (e.g., excess `msg.value` not reclaimed via `refundETH`) is silently consumed as part of a subsequent user's WETH swap, reducing the subsequent user's WETH pull and permanently stealing the prior user's ETH.

### Finding Description

In `PeripheryPayments.pay`, the hybrid WETH branch reads the router's entire native balance: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total router ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

`nativeBalance` is not scoped to the current transaction's `msg.value`. It includes any ETH previously deposited by other callers.

The `multicall` function is `payable` and batches arbitrary calls: [2](#0-1) 

A user who calls `exactInputSingle` (or any payable swap) via `multicall` and omits `refundETH()` leaves excess ETH stranded on the router. The `receive()` guard only blocks direct ETH sends from non-WETH addresses; it does not prevent ETH accumulation via `msg.value` in payable functions: [3](#0-2) 

### Impact Explanation

**Passive loss:** User A sends 1 ETH via `exactInputSingle(WETH→tokenX)` but the pool only consumes 0.7 ETH. Without `refundETH()`, 0.3 ETH is stranded. User B's subsequent `exactInputSingle(WETH, 1e18)` triggers `pay(WETH, userB, pool, 1e18)`. With `nativeBalance = 0.3e18 > 0` and `< 1e18`, the router wraps User A's 0.3 ETH, sends it to the pool, and pulls only 0.7 WETH from User B. User A loses 0.3 ETH permanently.

**Active theft:** An attacker observes the router holds X ETH (via mempool or on-chain balance check), then calls `exactInputSingle(WETH, X)` with `msg.value = 0`. The `nativeBalance >= value` branch fires: the router wraps all X ETH and sends it to the pool; the attacker receives the output tokens and pays zero WETH from their own balance. This is a complete theft of the stranded ETH.

### Likelihood Explanation

- `multicall` is the standard UX pattern for ETH swaps (permit + swap + refundETH). Omitting `refundETH` is a common user mistake.
- `exactOutputSingle` with ETH overpayment is another natural source of stranded ETH.
- Active exploitation requires only a public call with no special privileges.

### Recommendation

Track only the ETH contributed by the **current transaction** rather than the router's total balance. One approach: record `address(this).balance` at the start of each top-level entry point and pass the per-call ETH budget into `pay`, or replace `address(this).balance` with `msg.value` threaded through the call stack. Alternatively, auto-refund excess ETH at the end of every swap entry point.

### Proof of Concept

1. User A calls `multicall([exactInputSingle(WETH→tokenX, amountIn=1e18)])` with `msg.value = 1e18`. Pool demands exactly 1e18; no excess. *(Baseline — no stranded ETH.)*
2. User A calls `exactOutputSingle(WETH→tokenX, amountOut=Y)` with `msg.value = 1e18`; pool demands only 0.5e18. User A omits `refundETH()`. Router now holds 0.5e18 ETH.
3. Attacker calls `exactInputSingle(pool, WETH, amountIn=0.5e18, msg.value=0)`.
4. Callback fires `pay(WETH, attacker, pool, 0.5e18)`. `nativeBalance = 0.5e18 >= value`. Router wraps User A's 0.5 ETH → sends 0.5 WETH to pool. Attacker receives tokenX output.
5. Assert: attacker's WETH balance unchanged; User A's 0.5 ETH is gone; attacker holds tokenX.
