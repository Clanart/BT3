### Title
Excess ETH stranded in router after `exactOutputSingle`/`exactOutput` is consumed by any subsequent WETH-input swap caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
The `pay` function reads `address(this).balance` — the router's **total** ETH balance — to settle WETH-input swaps. Swap entry points (`exactOutputSingle`, `exactOutput`, `exactInputSingle`, `exactInput`) are all `payable` but none automatically refund excess ETH. Any ETH left over from a prior call is silently available to the next WETH-input swap caller, who can obtain a fully-paid swap while contributing zero ETH or WETH.

### Finding Description

`PeripheryPayments.pay` handles WETH payments by checking the router's native balance first: [1](#0-0) 

When `nativeBalance >= value`, it wraps exactly `value` ETH and forwards it to the pool. The remainder — `address(this).balance − value` — stays in the router with no accounting of which transaction deposited it.

The swap entry points are all `payable` and none issue an automatic refund: [2](#0-1) [3](#0-2) 

The only refund path is the standalone `refundETH()`, which must be called explicitly — typically inside a `multicall` bundle: [4](#0-3) 

A user who calls `exactOutputSingle` directly (not via `multicall`) and sends more ETH than the pool requests leaves the surplus in the router. Because `pay` reads the global `address(this).balance` rather than the current transaction's `msg.value`, the next caller whose swap triggers `pay(WETH, ...)` has their payment drawn from the stranded ETH — without contributing any ETH or WETH themselves.

The callback path that invokes `pay` is `_justPayCallback`, which passes `_getPayer()` (the original `msg.sender`) as the payer: [5](#0-4) 

When `nativeBalance >= value`, the `safeTransferFrom` branch is never reached, so the attacker's payer address is irrelevant — no token pull from the attacker occurs.

The same surface exists in `MetricOmmPoolLiquidityAdder`, which inherits the same `pay` function and is also `payable`: [6](#0-5) 

### Impact Explanation
Direct loss of user principal. A victim who sends excess ETH in a WETH-input `exactOutputSingle` call loses the surplus to the next WETH-input swap caller. The attacker receives the full swap output at zero cost. The loss equals `msg.value − actualAmountIn`; in `exactOutput*` flows this can be as large as `amountInMaximum − actualAmountIn`.

### Likelihood Explanation
Medium. Users commonly overprovision `msg.value` in `exactOutput*` calls because the exact input is unknown at call time. The attacker only needs to observe the router's ETH balance (trivially readable on-chain) and submit a WETH-input swap with `msg.value = 0` before the victim calls `refundETH()`. No special permissions or setup are required.

### Recommendation
Add an automatic ETH refund at the end of each swap entry point, after `_clearExpectedCallbackPool()`:

```solidity
uint256 leftover = address(this).balance;
if (leftover > 0) _transferETH(msg.sender, leftover);
```

Alternatively, pass the current transaction's `msg.value` as a budget parameter through the call stack and restrict `pay` to consuming only that amount, falling back to `safeTransferFrom` for any shortfall.

### Proof of Concept

1. Alice calls `exactOutputSingle{value: 2000}(...)` with `tokenIn = WETH`, `amountOut = 500`, `amountInMaximum = 2000`.
2. The pool requests 1000 in the callback; `pay` reads `address(this).balance == 2000`, wraps 1000 ETH, sends WETH to the pool. **1000 ETH remains in the router.**
3. Bob calls `exactOutputSingle{value: 0}(...)` with `tokenIn = WETH`, `amountOut = 500`, `amountInMaximum = 1000`.
4. The pool requests 1000 in the callback; `pay` reads `address(this).balance == 1000`, wraps it, sends WETH to the pool.
5. Bob receives 500 output tokens at zero cost. Alice's 1000 ETH is permanently lost.
