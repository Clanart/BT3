### Title
Router-held native ETH residue is silently consumed by a subsequent user's WETH payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay` function in `PeripheryPayments` uses `address(this).balance` — the router's **entire** native ETH balance — to satisfy a WETH payment obligation. Because the router is `payable` and accepts `msg.value` across multiple public entry points (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `multicall`), any ETH that a prior user sent but did not consume (e.g., by omitting `refundETH()` from a multicall) remains on the router and is silently spent on behalf of the **next** user who swaps with `tokenIn = WETH`. The prior user's ETH is permanently lost; the next user pays nothing for their WETH input.

---

### Finding Description

`PeripheryPayments.pay` handles the WETH leg as follows:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← ALL router ETH, any source
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` is not scoped to the current caller's `msg.value`. It is the aggregate of every wei that has ever arrived on the router and not yet been swept.

ETH accumulates on the router whenever a user sends more `msg.value` than the swap consumes and does not include a `refundETH()` call in the same multicall. This is a realistic and common mistake (e.g., `exactOutputSingle` where the actual input is less than `amountInMaximum`, or a multicall where the user forgets the refund step).

**Attack path:**

1. **Victim** calls `exactInputSingle{value: 2000}(tokenIn=WETH, amountIn=1000, ...)`. The `pay` callback deposits 1000 ETH as WETH and pays the pool. 1000 ETH remains on the router. The victim does not call `refundETH()`.

2. **Attacker** calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=1000, ...)`. Inside the callback, `pay` reads `address(this).balance == 1000`, deposits the victim's ETH as WETH, and pays the pool. The attacker receives the full swap output without spending any ETH or WETH.

The victim's 1000 ETH is permanently transferred to the pool on behalf of the attacker.

The same path applies to `exactOutputSingle`, `exactInput`, `exactOutput`, and any multicall sequence where excess ETH is left on the router.

---

### Impact Explanation

Direct, permanent loss of the victim's native ETH principal. The loss amount equals the stranded ETH balance at the time of the attacker's call, which is unbounded. No privileged access is required; any public swap call with `tokenIn = WETH` triggers the vulnerable `pay` path.

---

### Likelihood Explanation

Stranding ETH on the router is a natural user mistake: `exactOutputSingle` with `msg.value > amountInMaximum`, or a multicall that omits `refundETH()`. The pattern is explicitly documented in the test suite (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) as the *correct* way to recover unused ETH, implying the router is expected to hold ETH transiently. An attacker monitoring the mempool or the router's ETH balance can front-run or back-run any transaction that leaves ETH behind.

---

### Recommendation

Track the per-call ETH budget explicitly. The standard fix is to record `msg.value` at the top of each public entry point (or in `multicall`) and pass it as a parameter to `pay`, so that only the current caller's ETH contribution is eligible for WETH wrapping. Any remaining ETH should be refunded automatically at the end of the call, rather than relying on the user to include a separate `refundETH()` step.

Alternatively, enforce that `address(this).balance` is zero at the start of every non-multicall entry point (revert if residue exists), and auto-refund at the end of `multicall`.

---

### Proof of Concept

```
State before:
  router.balance = 0
  victim.balance = 2000
  attacker.balance = 0
  attacker.weth  = 0

Step 1 – victim swaps 1000 ETH worth of WETH, sends 2000 ETH, forgets refundETH():
  victim calls exactInputSingle{value: 2000}(tokenIn=WETH, amountIn=1000)
  → pay() sees nativeBalance=2000 >= 1000
  → deposits 1000 ETH as WETH, transfers to pool
  → router.balance = 1000  (victim's residue, unrecovered)

Step 2 – attacker steals residue:
  attacker calls exactInputSingle{value: 0}(tokenIn=WETH, amountIn=1000)
  → pay() sees nativeBalance=1000 >= 1000
  → deposits 1000 ETH (victim's!) as WETH, transfers to pool
  → attacker receives full swap output, pays nothing

Final state:
  router.balance = 0
  victim.balance = 0   (lost 2000 ETH: 1000 for own swap + 1000 stolen)
  attacker received swap output worth ~1000 ETH at zero cost
