### Title
Stranded ETH in Router Subsidizes Attacker's WETH Swap, Stealing Prior User's Funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` unconditionally uses the router's native ETH balance before pulling WETH from the designated payer. An attacker who calls `exactInputSingle` with `tokenIn=WETH` while the router holds residual ETH pays less than `amountIn` from their own balance; the deficit is covered by ETH that belongs to a prior user.

### Finding Description
`exactInputSingle` stores `msg.sender` as the payer in transient storage and calls the pool's `swap()`. The pool calls back into `metricOmmSwapCallback`, which dispatches to `_justPayCallback`, which calls `pay(WETH, payer=attacker, pool, X)`. [2](#0-1) 

Inside `pay()`, when `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — the router's total native ETH — and uses it first:

- If `nativeBalance >= X`: the pool is paid entirely from the router's ETH; the attacker pays **zero** WETH from their own balance.
- If `0 < nativeBalance < X`: the router's ETH covers the first `nativeBalance` wei; the attacker pays only `X - nativeBalance`. [3](#0-2) 

ETH accumulates in the router whenever a user sends `msg.value` to a payable swap function (e.g., `exactInputSingle{value: V}()`) and the swap consumes less than `V`, or when a user forgets to append `refundETH()` to a `multicall`. The `receive()` guard only blocks direct ETH transfers from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable entry points. [4](#0-3) 

### Impact Explanation
Direct loss of user funds. The prior user's stranded ETH is irreversibly consumed to pay for the attacker's swap. The attacker receives full swap output while paying less (or nothing) from their own WETH balance. The stranded ETH cannot be recovered via `refundETH()` afterward because it is gone.

### Likelihood Explanation
Any time a user sends excess ETH to a payable router function without bundling `refundETH()` in the same `multicall`, ETH is stranded. This is a common pattern (e.g., sending a round-number ETH amount for a WETH swap). An attacker can monitor the router's ETH balance on-chain and exploit it in the next block.

### Recommendation
Remove the native-ETH subsidy branch from `pay()` when `payer != address(this)`. If ETH-to-WETH wrapping is desired for the caller's own ETH (sent as `msg.value`), require the caller to explicitly wrap it before the swap (e.g., via a `wrapETH` helper in `multicall`), or track how much ETH the current `msg.sender` deposited in the current call and only use that amount. The simplest fix is:

```solidity
} else if (token == WETH) {
    // Always pull from payer; ETH wrapping must be done explicitly by the caller.
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```
