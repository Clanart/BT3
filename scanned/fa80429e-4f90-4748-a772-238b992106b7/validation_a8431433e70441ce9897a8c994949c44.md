### Title
`refundETH` Drains Entire Router ETH Balance to Caller, Enabling Theft of Any Prior User's Overpaid ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` unconditionally transfers `address(this).balance` to `msg.sender` with no per-user accounting. Because `pay()` wraps only the exact WETH amount needed and leaves any ETH overpayment on the router, a subsequent caller — including an MEV bot — can drain that residual ETH by calling `refundETH()` directly.

---

### Finding Description

**Step 1 — ETH accumulates on the router after a WETH-paying swap.**

When a user calls `exactInputSingle{value: X}(...)` with `tokenIn == WETH` and `X > amountIn`, the swap callback triggers `pay()`: [1](#0-0) 

`pay()` wraps exactly `value` (= `amountIn`) ETH into WETH and forwards it to the pool. The surplus `X − amountIn` native ETH remains on the router with no record of who sent it.

**Step 2 — `refundETH()` has no per-user accounting.** [2](#0-1) 

`address(this).balance` is the **total** native ETH balance of the router — not just the caller's contribution. Any caller (attacker, MEV bot) who invokes `refundETH()` after a victim's overpaying swap receives the victim's leftover ETH.

**Step 3 — `receive()` does not prevent ETH from accumulating.** [3](#0-2) 

The `receive()` guard only blocks bare ETH transfers (no calldata). ETH sent as `msg.value` to any `payable` function — `exactInputSingle`, `exactInput`, `multicall` — bypasses it entirely and lands in `address(this).balance`.

**Step 4 — `multicall` is `payable` and uses `delegatecall`.** [4](#0-3) 

An attacker can compose `[refundETH()]` as a single-step multicall, or simply call `refundETH()` directly. No special role or permission is required.

---

### Impact Explanation

Any user who calls `exactInputSingle` / `exactInput` with ETH overpayment in a standalone transaction (not wrapped in a multicall that also calls `refundETH()`) permanently loses the surplus ETH to the first caller of `refundETH()`. This is a direct loss of user principal with no recovery path.

---

### Likelihood Explanation

- Users calling the router directly (e.g., via a contract integration or a UI that does not bundle `refundETH()`) will leave ETH on the router.
- MEV bots routinely monitor for exactly this pattern and can back-run the victim's swap in the same block.
- No privileged access, special token behavior, or malicious pool setup is required.

---

### Recommendation

Track each depositor's ETH contribution in a transient storage slot (EIP-1153) keyed by `msg.sender`, and in `refundETH()` refund only that sender's recorded amount, zeroing the slot before the transfer. Alternatively, enforce that `refundETH()` can only be called as part of a `multicall` initiated in the same transaction (e.g., via a transient reentrancy flag set at `multicall` entry).

---

### Proof of Concept

```solidity
// Foundry test sketch
function test_refundETH_steals_prior_user_overpayment() public {
    // Victim sends 1 ETH but swap only needs 0.9 ETH
    vm.prank(victim);
    router.exactInputSingle{value: 1 ether}(params); // 0.1 ETH left on router

    uint256 attackerBefore = attacker.balance;

    // Attacker calls refundETH() — no ETH sent, no swap performed
    vm.prank(attacker);
    router.refundETH();

    // Attacker receives victim's 0.1 ETH overpayment
    assertEq(attacker.balance, attackerBefore + 0.1 ether);
}
```
