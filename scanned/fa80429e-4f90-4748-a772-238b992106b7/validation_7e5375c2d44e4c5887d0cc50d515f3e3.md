### Title
Router `pay()` consumes any stranded native ETH as WETH settlement, enabling free swaps at victim's expense — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` settles WETH obligations by reading `address(this).balance` — the router's **total** native ETH balance — rather than only the ETH the current caller contributed. Any native ETH left on the router by a prior user (e.g., excess `msg.value` from a multicall that omitted `refundETH`) is silently consumed to cover a subsequent caller's WETH payment, giving the attacker a free or discounted swap while the victim loses their ETH.

---

### Finding Description

`PeripheryPayments.pay()` contains three branches for WETH settlement: [1](#0-0) 

```solidity
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
        uint256 nativeBalance = address(this).balance;          // ← total router ETH, not msg.value
        if (nativeBalance >= value) {
            IWETH9(WETH).deposit{value: value}();               // ← uses ANY stranded ETH
            IERC20(WETH).safeTransfer(recipient, value);
        } else if (nativeBalance > 0) {
            IWETH9(WETH).deposit{value: nativeBalance}();       // ← partial: stranded ETH first
            IERC20(WETH).safeTransfer(recipient, nativeBalance);
            IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
        } else {
            IERC20(WETH).safeTransferFrom(payer, recipient, value);
        }
    } else {
        IERC20(token).safeTransferFrom(payer, recipient, value);
    }
}
```

`pay()` is called inside `_justPayCallback` during `exactInputSingle` and inside `_exactOutputIterateCallback` during `exactOutput`, both of which set `payer = msg.sender` (the external caller): [2](#0-1) 

The callback context stores `tokenToPay = params.tokenIn` at swap initiation: [3](#0-2) 

When `tokenIn == WETH`, `pay()` checks `address(this).balance` — the **entire** router balance — not just what the current caller sent via `msg.value`. Native ETH accumulates on the router whenever a user sends excess ETH in a `multicall` without appending a `refundETH` call: [4](#0-3) 

The `receive()` guard only blocks direct ETH pushes; it does not prevent ETH from accumulating via `msg.value` on payable entry points: [5](#0-4) 

**Full-coverage theft (zero-cost swap):**  
If `nativeBalance >= value`, the function wraps exactly `value` ETH from the router's balance and sends it to the pool. The `payer` is charged nothing — no `safeTransferFrom` is issued. An attacker who observes stranded ETH on the router can call `exactInputSingle(tokenIn=WETH, amountIn ≤ stranded_balance)` with `msg.value = 0` and zero WETH allowance, receiving the full swap output at no cost.

**Partial-coverage theft (discounted swap):**  
If `0 < nativeBalance < value`, the function wraps the stranded ETH first, then pulls only `value - nativeBalance` WETH from the payer. The attacker pays less WETH than the swap requires, with the deficit covered by the victim's ETH.

This is the direct analog of H-19: in H-19, native ETH (WETH) is deposited but a different token (`wBTC`) is recorded for settlement, so the vault's wBTC is drained. Here, another user's native ETH is recorded as the WETH settlement source, so the victim's ETH funds the attacker's swap.

---

### Impact Explanation

**Direct loss of user principal.** Any user who sends excess ETH in a `multicall` without including `refundETH` has their ETH permanently claimable by any subsequent caller who issues a WETH-input swap. In the full-coverage case the attacker pays nothing for the swap output; in the partial-coverage case the attacker's WETH cost is reduced by the stranded amount. The victim receives no output and cannot recover the ETH (it is wrapped and transferred to the pool, not to the attacker directly, but the attacker receives the swap output token of equivalent value).

---

### Likelihood Explanation

The native-ETH multicall pattern is explicitly documented and tested in this repository: [6](#0-5) 

Users following the pattern `multicall{value}(exactInputSingle(...))` without appending `refundETH` will strand ETH. This is a common integration mistake (Uniswap v3 has the same footgun). An attacker can passively monitor the router's ETH balance and exploit any non-zero residue with a single unprivileged call. No special role, no malicious setup, and no flash loan is required.

---

### Recommendation

Track the ETH that belongs to the current multicall context separately from any pre-existing router balance. One approach: record `address(this).balance` at the start of each `multicall` invocation and pass the delta (ETH added by the current call) as the usable native budget into `pay()`. Alternatively, enforce that `pay()` only uses `msg.value` (passed explicitly) rather than `address(this).balance`, and require callers to wrap ETH themselves before calling the router if they want WETH-input swaps without the multicall pattern.

### Proof of Concept

```
1. Victim calls:
     router.multicall{value: 1 ether}([
         exactInputSingle(tokenIn=WETH, amountIn=0.3 ether, ...)
     ])
   → swap consumes 0.3 ETH; 0.7 ETH stranded on router (refundETH omitted).

2. Attacker observes router.balance == 0.7 ETH on-chain.

3. Attacker calls (msg.value = 0, zero WETH allowance):
     router.exactInputSingle(
         pool=<WETH/token1 pool>,
         tokenIn=WETH,
         tokenOut=token1,
         amountIn=0.7 ether,
         amountOutMinimum=0,
         recipient=attacker,
         ...
     )

4. Pool calls metricOmmSwapCallback → _justPayCallback →
     pay(WETH, attacker, pool, 0.7 ether)
       nativeBalance = 0.7 ether >= 0.7 ether
       → WETH.deposit{value: 0.7 ether}()   // victim's ETH wrapped
       → WETH.transfer(pool, 0.7 ether)     // sent to pool
       // no safeTransferFrom on attacker

5. Attacker receives token1 output worth ~0.7 ETH.
   Victim's 0.7 ETH is gone.
```
