### Title
`pay()` uses `address(this).balance` instead of per-call `msg.value`, allowing any WETH-input swap to drain ETH left by prior callers — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's entire native ETH balance — when settling a WETH-input swap. Because this balance persists across transactions, any ETH left in the contract by a previous caller (who sent `msg.value` but omitted `refundETH()`) is silently consumed to pay for a subsequent caller's swap. The subsequent caller pays nothing from their own wallet yet receives the full swap output.

---

### Finding Description

`PeripheryPayments.pay()` contains the following branch for WETH payments:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire contract balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` is not scoped to the current transaction's `msg.value`. It includes ETH deposited by any prior caller that was never refunded. The correct Uniswap v3 pattern tracks only the current call's `msg.value` (passed as a parameter) and falls through to `transferFrom` for any shortfall.

`pay()` is invoked from two production paths:

1. **Router swap callback** — `_justPayCallback` and `_exactOutputIterateCallback` in `MetricOmmSimpleRouter` call `pay(_getTokenToPay(), _getPayer(), msg.sender, amount)`. When `tokenIn == WETH`, the WETH branch fires and uses `address(this).balance`.

2. **Liquidity adder callback** — `metricOmmModifyLiquidityCallback` in `MetricOmmPoolLiquidityAdder` calls `pay(token0/token1, payer, msg.sender, amountDelta)`. Same branch fires when either pool token is WETH.

Both contracts expose a `payable multicall`. A user who sends ETH in a multicall and omits `refundETH()` leaves residual ETH in the contract. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) prevents arbitrary ETH injection but does not prevent ETH from accumulating via `msg.value` in payable entry points.

---

### Impact Explanation

**Direct loss of user principal.** ETH sent by Alice in a multicall that is not refunded is permanently available to any subsequent caller. Bob can execute a WETH-input swap of equal size with zero `msg.value` and zero WETH approval; the router wraps Alice's ETH and delivers it to the pool on Bob's behalf. Alice's ETH is gone; Bob receives the full swap output for free.

The same drain applies to the `MetricOmmPoolLiquidityAdder`: if Alice adds WETH liquidity via multicall and leaves residual ETH, Bob can add WETH liquidity at Alice's expense.

---

### Likelihood Explanation

The `multicall` + `refundETH()` pattern is standard but not enforced. Users who construct multicalls without a trailing `refundETH()` — a common mistake, especially via SDK-generated calldata — leave ETH in the contract. The attack requires no special role, no flash loan, and no front-running beyond standard mempool observation. Any block that contains a multicall with residual ETH is immediately exploitable in the same or next block.

---

### Recommendation

Pass the current call's `msg.value` (or a tracked per-call ETH budget stored in transient storage at entry) into `pay()` instead of reading `address(this).balance`. The corrected branch should be:

```solidity
// Track msg.value in transient storage at the top of each payable entry point,
// then pass it into pay() and decrement it as ETH is consumed.
if (token == WETH) {
    uint256 msgValueRemaining = _getMsgValueRemaining(); // transient slot
    if (msgValueRemaining >= value) {
        _decrementMsgValue(value);
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (msgValueRemaining > 0) {
        _decrementMsgValue(msgValueRemaining);
        IWETH9(WETH).deposit{value: msgValueRemaining}();
        IERC20(WETH).safeTransfer(recipient, msgValueRemaining);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - msgValueRemaining);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

This ensures only ETH explicitly sent in the current transaction is consumed, matching the Uniswap v3 reference implementation.

### Proof of Concept

```
Setup:
  - Router deployed with WETH = <weth>
  - Pool: WETH / TokenX

Step 1 — Alice leaves ETH in the router:
  Alice calls router.multicall{value: 2 ether}([
      exactInputSingle(tokenIn=WETH, amountIn=1 ether, ...)
      // ← no refundETH() call
  ])
  pay(WETH, Alice, pool, 1 ether) fires:
    nativeBalance = 2 ether >= 1 ether → wraps 1 ETH, sends WETH to pool.
  1 ETH remains in router.

Step 2 — Bob steals Alice's ETH:
  Bob calls router.exactInputSingle{value: 0}(
      tokenIn=WETH, amountIn=1 ether, recipient=Bob, ...
  )
  // Bob has no WETH approval and sends no ETH.
  pay(WETH, Bob, pool, 1 ether) fires:
    nativeBalance = 1 ether >= 1 ether → wraps Alice's 1 ETH, sends WETH to pool.
  Bob receives TokenX output; Alice's 1 ETH is gone.

Assert: Bob.tokenXBalance > 0, Alice's 1 ETH is unrecoverable.
```
