### Title
Pre-existing ETH consumed by subsequent callers via `address(this).balance` in `pay()`, enabling theft of stranded native funds — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay` function in `PeripheryPayments.sol` uses `address(this).balance` (total contract ETH) instead of only the current call's `msg.value` when deciding how much native ETH to wrap and forward to the pool. Any ETH left in the router from a prior call (a user who over-sent ETH and omitted `refundETH()`) is silently consumed to settle the next caller's swap, while the next caller's own `msg.value` remains in the contract. A subsequent `refundETH()` call by that next caller drains the stranded ETH, constituting a direct theft of the prior user's principal.

---

### Finding Description

In `PeripheryPayments.sol`, the `pay` helper is invoked during every swap callback to settle the pool's token demand. When the input token is WETH and the payer is an EOA, the function reads the contract's total native balance: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total ETH, not just msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` includes **all** ETH in the contract — both the current call's `msg.value` and any ETH stranded from prior calls. The function wraps exactly `value` ETH and sends WETH to the pool, leaving the remainder in the contract.

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses: [2](#0-1) 

It does **not** prevent ETH from accumulating via `msg.value` in any of the payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `addLiquidityExactShares`, etc.). Users routinely over-send ETH and are expected to append `refundETH()` to their multicall to recover the excess.

`refundETH()` has no access control and sends the entire contract balance to `msg.sender`: [3](#0-2) 

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← sends ALL ETH to caller
    }
}
```

Any caller can drain all ETH in the router at any time.

This is the direct native analog of the LiFi `SwapperV2` bug: both use `address(this).balance` without subtracting the pre-existing balance, causing the contract to treat stranded funds as part of the current call's settlement, and then expose them to the next caller.

---

### Impact Explanation

A user who over-sends ETH (e.g., sends 1 ETH for a 0.5 ETH swap) and omits `refundETH()` from their multicall leaves 0.5 ETH stranded in the router. The next caller who performs any WETH-input swap with a tiny `msg.value` and then calls `refundETH()` receives the stranded 0.5 ETH. This is a **direct loss of user principal** with no protocol-level recovery mechanism. The loss scales with the amount of ETH stranded, which can be arbitrarily large.

---

### Likelihood Explanation

The protocol's intended usage pattern (multicall + refundETH) means users routinely over-send ETH. Any user who forgets to append `refundETH()` to their multicall is immediately vulnerable. An attacker can monitor the router's ETH balance on-chain and follow up with a minimal swap + `refundETH()` to drain it atomically. No special permissions or privileged access are required.

---

### Recommendation

Track `msg.value` separately — store it in transient storage at the start of each payable entry point — and use that tracked value instead of `address(this).balance` in the `pay` function. Only the ETH sent in the current call should be eligible for wrapping. Alternatively, wrap `msg.value` into WETH immediately upon entry and always pull WETH from the payer via `safeTransferFrom`, eliminating the native-balance ambiguity entirely (analogous to the Swapper.sol fix in the LiFi report: `balance - msg.value`).

---

### Proof of Concept

**Step 1 — User A over-sends ETH:**
```
User A calls multicall([
    exactInputSingle({tokenIn: WETH, amountIn: 0.5e18, ...})
    // ← no refundETH() appended
]) with msg.value = 1e18
```
- `pay` sees `address(this).balance = 1e18`, wraps 0.5e18, sends WETH to pool.
- **0.5e18 ETH remains stranded in the router.**

**Step 2 — Attacker drains stranded ETH:**
```
Attacker calls multicall([
    exactInputSingle({tokenIn: WETH, amountIn: 1 wei, ...}),
    refundETH()
]) with msg.value = 1 wei
```
- `pay` sees `address(this).balance = 0.5e18 + 1 wei`, wraps 1 wei, sends WETH to pool.
- 0.5e18 ETH remains in router.
- `refundETH()` sends **0.5e18 ETH to attacker**.
- **Attacker gains 0.5e18 ETH (User A's principal) at a cost of 1 wei + gas.**

The root cause — `address(this).balance` instead of a per-call tracked `msg.value` — is identical to the LiFi `SwapperV2` initial-balance miscalculation.
