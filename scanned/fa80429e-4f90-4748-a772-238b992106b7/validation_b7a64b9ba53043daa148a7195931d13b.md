### Title
Anyone Can Steal ETH Stranded on the Router via `refundETH` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` unconditionally sends `address(this).balance` to `msg.sender` with no per-user accounting. The `pay()` function deliberately leaves excess native ETH on the router when `nativeBalance > amountIn`. Any ETH stranded between transactions is claimable by any caller.

---

### Finding Description

`pay()` handles native ETH input for WETH swaps with a partial-wrap pattern: [1](#0-0) 

When `nativeBalance >= value`, it wraps exactly `value` and transfers it to the pool. The surplus `nativeBalance - value` remains as raw ETH on the router. This is the intended design — users are expected to append `refundETH()` to their multicall.

`refundETH()` has no access control and no per-user accounting: [2](#0-1) 

It sends the **entire** `address(this).balance` to whoever calls it. If any ETH is present on the router — from any prior user's overpayment — the caller receives it all.

`multicall` uses `delegatecall`, so `msg.sender` is preserved across steps within a single transaction: [3](#0-2) 

This means the cross-transaction attack is the real vector: User A sends `msg.value = 2 ether` with `amountIn = 1000` but omits `refundETH` from their multicall. The `pay()` call wraps 1000 wei; `2 ether - 1000 wei` stays on the router. Attacker B then calls `refundETH()` in a separate transaction and receives all of it.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) prevents arbitrary ETH deposits but does not prevent ETH from accumulating via `msg.value` on payable entry points (`multicall`, `exactInputSingle`, `exactOutputSingle`, etc.). [4](#0-3) 

---

### Impact Explanation

Direct loss of user principal. Any user who sends ETH overpayment without appending `refundETH` to their multicall loses the surplus to the first caller of `refundETH()`. The attacker needs no special role — the function is public and permissionless.

---

### Likelihood Explanation

Moderate. The intended pattern (shown in `test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) requires users to manually compose the `refundETH` step. Integrators, aggregators, or users calling `exactInputSingle` directly with `msg.value` (not via multicall) will routinely omit it. A MEV bot monitoring the mempool or router balance can reliably front-run or back-run to claim stranded ETH. [5](#0-4) 

---

### Recommendation

Track per-caller ETH deposits in transient storage at the start of each payable entry point and enforce in `refundETH()` that only the depositor's own balance is returned. Alternatively, auto-refund surplus ETH at the end of each swap function rather than relying on a separate user-initiated call.

---

### Proof of Concept

```solidity
// Step 1: Victim sends ETH overpayment without refundETH
vm.prank(victim);
router.multicall{value: 2 ether}(
    [abi.encodeCall(router.exactInputSingle, (params_amountIn_1000))]
);
// Router now holds 2 ether - 1000 wei as raw ETH

// Step 2: Attacker steals it
uint256 before = attacker.balance;
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance - before, 2 ether - 1000); // attacker receives victim's surplus
```

The invariant that "ETH refunded belongs only to the current transaction's initiator" is not enforced anywhere in the code.
