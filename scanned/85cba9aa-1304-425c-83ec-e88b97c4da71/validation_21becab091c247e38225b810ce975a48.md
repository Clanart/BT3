### Title
Residual ETH Left by Prior Caller Is Silently Consumed by Subsequent WETH-Input Swap, Causing Direct ETH Loss — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` uses the router's entire native ETH balance to subsidize any WETH-input swap. Because all swap entry points are `payable` and no automatic refund is issued, ETH sent by User A (e.g., via `exactInputSingle{value: X}` with a non-WETH `tokenIn`, or by sending more ETH than `amountIn` for a WETH swap) remains in the router. A subsequent User B calling `exactInputSingle` with `tokenIn = WETH` causes the router to consume User A's ETH to settle User B's payment, reducing User B's WETH debit by `nativeBalance` and permanently destroying User A's ETH.

### Finding Description

`pay()` in `PeripheryPayments.sol` (lines 73–84) branches on `token == WETH` and reads `address(this).balance`. If `nativeBalance >= value`, it wraps the full `value` from the router's ETH and transfers WETH to the pool — pulling **zero** WETH from `payer`. If `0 < nativeBalance < value`, it wraps `nativeBalance` from the router's ETH and pulls only `value - nativeBalance` from `payer`.

The router's `receive()` guard (line 32–34) only prevents bare ETH transfers (`msg.sender != WETH` reverts). It does **not** prevent ETH from entering via `msg.value` on any of the `payable` swap functions (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `multicall`). ETH sent this way accumulates in the router's balance and is never automatically returned.

**Concrete attack path:**

1. User A calls `exactInputSingle{value: 1 ether}` with `tokenIn = someERC20` (not WETH). The `pay()` call takes the `else` branch (line 86), pulling ERC20 from User A. The 1 ETH `msg.value` is never consumed and sits in the router. User A does not call `refundETH()`.
2. User B calls `exactInputSingle` (no ETH sent) with `tokenIn = WETH`, `amountIn = 1 ether`. Inside the swap callback, `pay(WETH, userB, pool, 1e18)` is called. `nativeBalance = 1 ether >= value = 1 ether`, so the router wraps User A's 1 ETH and transfers WETH to the pool. User B's WETH allowance is **never touched**.
3. User A's 1 ETH is permanently lost; User B swapped for free.

An equally realistic variant: User A calls `exactInputSingle{value: 2 ether}` with `tokenIn = WETH`, `amountIn = 1 ether`. The `pay()` call uses 1 ETH, leaving 1 ETH residual. User A forgets `refundETH()`. User B then steals the residual 1 ETH via the same mechanism.

### Impact Explanation

Direct, permanent loss of User A's ETH principal. The ETH is not locked — it is actively transferred to a pool as WETH on behalf of User B. User A receives nothing in return. This satisfies the "Critical/High direct loss of user principal" impact gate.

### Likelihood Explanation

Medium. The `receive()` guard creates a false sense of security — developers and users may believe ETH cannot enter the router unexpectedly, but `msg.value` on any payable function bypasses it. The mixed ETH/WETH payment feature (tested in `MetricOmmSimpleRouter.native.t.sol`) is an intended use case that actively encourages users to send ETH with WETH swaps. Any user who sends slightly more ETH than needed, or who batches a non-WETH swap with ETH in a `multicall` without a trailing `refundETH()`, creates a residual that any subsequent WETH swapper can drain.

### Recommendation

Track the ETH balance attributable to the **current transaction** rather than the entire contract balance. The standard fix is to snapshot `address(this).balance - msg.value` at entry and only treat `msg.value` as available ETH for the current caller. Alternatively, enforce that `pay()` only uses ETH up to `msg.value` of the current top-level call (passed down through the call stack), and auto-refund any unused ETH at the end of each swap function rather than relying on the caller to invoke `refundETH()`.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import "forge-std/Test.sol";
import {MetricOmmSimpleRouter} from "metric-periphery/contracts/MetricOmmSimpleRouter.sol";
import {IMetricOmmSimpleRouter} from "metric-periphery/contracts/interfaces/IMetricOmmSimpleRouter.sol";
// ... (standard test setup with pool, WETH, token1)

function test_residualEthStolenByWethSwap() public {
    // User A calls exactInputSingle with tokenIn=token1 but accidentally sends 1 ETH
    vm.deal(userA, 1 ether);
    vm.prank(userA);
    // tokenIn is token1 (not WETH), so msg.value is never consumed
    router.exactInputSingle{value: 1 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(token1WethPool),
            tokenIn: address(token1),
            tokenOut: address(weth),
            zeroForOne: false,
            amountIn: 1000,
            amountOutMinimum: 0,
            recipient: userA,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // userA forgets refundETH(); 1 ETH is now stuck in router
    assertEq(address(router).balance, 1 ether);

    // User B swaps WETH->token1 for 1 ETH worth; router uses userA's ETH
    uint256 userBWethBefore = weth.balanceOf(userB);
    vm.prank(userB);
    router.exactInputSingle(  // no msg.value
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(wethToken1Pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1 ether,
            amountOutMinimum: 0,
            recipient: userB,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // userA's ETH was consumed; userB paid 0 WETH
    assertEq(address(router).balance, 0, "router ETH drained");
    assertEq(weth.balanceOf(userB), userBWethBefore, "userB paid no WETH");
    assertEq(userA.balance, 0, "userA lost 1 ETH");
}
```
