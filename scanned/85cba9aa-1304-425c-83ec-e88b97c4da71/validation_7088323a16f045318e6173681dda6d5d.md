**Tracing the exact execution path:**

**Step 1 — ETH accumulation vector:**
`addLiquidityExactShares` and `multicall` are both `payable`. [1](#0-0) [2](#0-1) 

The `receive()` guard only blocks *plain* ETH transfers (no calldata). It does **not** block ETH sent via a `payable` function call. [3](#0-2) 

So if User A calls `addLiquidityExactShares{value: X}(...)` and the pool only needs `Y < X` WETH, the excess `X - Y` ETH stays in the contract with no automatic refund.

**Step 2 — The `pay` branch that consumes accumulated ETH:** [4](#0-3) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — which includes *any* ETH left over from prior transactions — and wraps it first, reducing the `safeTransferFrom` pull from the actual payer.

**Step 3 — Callback wires `payer` to `msg.sender` of the outer call:** [5](#0-4) 

`payer` is always the user who called `addLiquidityExactShares`, so the WETH branch is always reached for WETH-pool deposits from EOAs.

**Step 4 — No automatic refund after `pay`:**
`refundETH` exists but is never called automatically inside `_addLiquidity` or the callback. Excess ETH silently persists. [6](#0-5)

**Verdict:**

The mechanism described in the question is **real and traceable in the scoped code**. ETH can accumulate in the contract via `payable` entry points, and the `pay` function unconditionally consumes `address(this).balance` before pulling from the payer, meaning any subsequent WETH-pool depositor benefits from prior users' stranded ETH. The original sender loses that ETH with no on-chain recourse.

However, the precondition is that a user must have sent excess ETH without including a `refundETH` call in the same multicall — a user-error pattern that is standard and documented in Uniswap-style periphery designs. The contract does not silently trap ETH in a way that is invisible to the user; the user chose not to call `refundETH`. The attacker cannot *force* ETH into the contract (the `receive()` guard blocks unsolicited deposits), so exploitation is entirely opportunistic and depends on another user's mistake.

Under the contest scope rules, this does not clear the bar for a **High** finding (no direct, unconditional principal loss from a single-actor attack). It is a borderline **Medium** — real fund loss, but contingent on user error and a race condition between two independent transactions.

### Title
ETH stranded in `MetricOmmPoolLiquidityAdder` by a prior `payable` call is silently consumed by the next WETH-pool depositor — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses `address(this).balance` to cover WETH payments before pulling from the actual payer. Because `addLiquidityExactShares` and `multicall` are `payable` and no automatic refund is issued after settlement, any ETH left in the contract from a previous transaction is consumed by the next user who deposits into a WETH pool, causing the original sender to lose that ETH.

### Finding Description
When a user calls `addLiquidityExactShares{value: X}(...)` and the pool requests only `Y < X` WETH, the `pay` function wraps exactly `Y` ETH and leaves `X - Y` in the contract. A subsequent caller depositing into any WETH pool triggers the same `pay` branch, which reads `address(this).balance = X - Y`, wraps it, and reduces its own `safeTransferFrom` pull by that amount. The original sender's ETH is irrecoverably transferred to the pool on behalf of the attacker.

### Impact Explanation
Direct loss of ETH for any user who sends excess ETH to a `payable` entry point without appending a `refundETH` call. The attacker pays less WETH than the pool actually received, with the difference funded by the victim's stranded ETH.

### Likelihood Explanation
Exploitation is opportunistic: the attacker must observe stranded ETH in the contract (e.g., via mempool monitoring or block scanning) and submit a WETH-pool deposit before the victim calls `refundETH`. The precondition (user omitting `refundETH`) is realistic given that `addLiquidityExactShares` is independently callable as a `payable` function.

### Recommendation
Inside `_addLiquidity`, after settlement, automatically refund any remaining `address(this).balance` to the payer. Alternatively, snapshot the ETH balance before the call and revert if it changes unexpectedly, or remove the `payable` modifier from `addLiquidityExactShares` and require ETH to be sent only via `multicall` with an explicit `refundETH` step.
