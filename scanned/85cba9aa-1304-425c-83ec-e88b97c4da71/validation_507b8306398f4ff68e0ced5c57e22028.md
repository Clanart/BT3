### Title
Leftover ETH in Router Consumed by Subsequent WETH Swaps, Stealing Prior User's Funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

The `pay` function in `PeripheryPayments` uses `address(this).balance` (live native ETH balance) to decide whether to wrap ETH instead of pulling WETH from the designated `payer`. When a prior user's `exactOutput` multicall leaves residual ETH in the router (because `refundETH()` was omitted), any subsequent WETH-input swap will silently consume that ETH to settle the new user's pool obligation, causing the prior user to lose their unrefunded ETH.

### Finding Description

`PeripheryPayments.pay` contains a three-branch WETH payment path:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L69-88
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
        uint256 nativeBalance = address(this).balance;
        if (nativeBalance >= value) {
            IWETH9(WETH).deposit{value: value}();
            IERC20(WETH).safeTransfer(recipient, value);          // ← pays from router ETH, NOT from payer
        } else if (nativeBalance > 0) {
            IWETH9(WETH).deposit{value: nativeBalance}();
            IERC20(WETH).safeTransfer(recipient, nativeBalance);
            IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
        } else {
            IERC20(WETH).safeTransferFrom(payer, recipient, value);
        }
    } else { ... }
}
```

When `nativeBalance >= value`, the router wraps its own ETH and sends it to the pool **without touching `payer`'s WETH at all**. The `payer` argument is completely ignored.

ETH enters the router legitimately via `multicall` with `msg.value` (the `receive()` guard only blocks direct sends, not `msg.value` attached to a function call). For `exactOutputSingle`, the user typically sends `msg.value = amountInMaximum`, but the pool callback requests only the actual (lower) input. The difference stays in the router. If the user's multicall omits `refundETH()`, that ETH is permanently stranded until another transaction drains it.

The callback path that triggers `pay` with the victim's address as `payer`:

```solidity
// MetricOmmSimpleRouter.sol L71
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);

// MetricOmmSimpleRouter.sol L192-199
function _justPayCallback(int256 amount0Delta, int256 amount1Delta) private {
    pay(_getTokenToPay(), _getPayer(), msg.sender,
        uint256(MetricOmmSwapResults.extractPositiveAmount(amount0Delta, amount1Delta)));
}
```

`_getPayer()` returns the victim's address, but when `nativeBalance >= value` the branch never calls `safeTransferFrom(payer, ...)`.

### Impact Explanation

A victim who sends `msg.value = amountInMaximum` for an `exactOutputSingle` WETH swap and omits `refundETH()` leaves up to `amountInMaximum − actualInput` ETH in the router. Any subsequent caller whose WETH-input swap amount is ≤ that residual balance receives a fully subsidised swap: the pool is paid from the victim's ETH, the attacker's WETH allowance is never touched, and the victim's ETH is permanently lost. Loss is bounded only by the victim's `amountInMaximum` cap, which can be arbitrarily large.

### Likelihood Explanation

`exactOutput` swaps with ETH/WETH are a standard user flow. Omitting `refundETH()` from a multicall is a well-documented footgun in Uniswap-style routers. An attacker can monitor the router's ETH balance on-chain and front-run or immediately follow any transaction that leaves residual ETH, requiring zero special privileges and no protocol-level setup.

### Recommendation

Remove the dependency on `address(this).balance` when `payer` is an external address. The ETH-wrapping shortcut should only apply when `payer == address(this)` (i.e., the router is paying from its own intermediate balance). For external payers, always use `safeTransferFrom`:

```solidity
} else if (token == WETH) {
    if (payer == address(this)) {
        // Router is paying from its own ETH (e.g. intermediate hop)
        uint256 nativeBalance = address(this).balance;
        if (nativeBalance >= value) {
            IWETH9(WETH).deposit{value: value}();
            IERC20(WETH).safeTransfer(recipient, value);
        } else { ... }
    } else {
        // External payer: always pull WETH; never consume router's ETH on their behalf
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

Alternatively, document and enforce that every `exactOutput` multicall with `msg.value > 0` must include `refundETH()` as the final call, and add a router-level check that reverts if `address(this).balance > 0` at the end of any top-level entry point.

### Proof of Concept

1. **Victim** calls `multicall([exactOutputSingle({tokenIn: WETH, amountOut: X, amountInMaximum: 1 ether, ...})])` with `msg.value = 1 ether`. Actual input required by the pool is `0.9 ether`. The callback fires `pay(WETH, victim, pool, 0.9 ether)` → `nativeBalance = 1 ether ≥ 0.9 ether` → router wraps `0.9 ether` and sends to pool. `0.1 ether` remains in the router. Victim omits `refundETH()`.

2. **Attacker** calls `exactInputSingle({tokenIn: WETH, amountIn: 0.1 ether, ...})` with `msg.value = 0`. The callback fires `pay(WETH, attacker, pool, 0.1 ether)` → `nativeBalance = 0.1 ether ≥ 0.1 ether` → router wraps the victim's `0.1 ether` and sends to pool. Attacker's WETH allowance is never touched.
