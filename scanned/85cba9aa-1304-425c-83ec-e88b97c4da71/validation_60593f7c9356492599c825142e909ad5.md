### Title
`pay()` uses live `address(this).balance` instead of per-call ETH accounting, letting any caller drain leftover native ETH from prior senders — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

---

### Summary

`PeripheryPayments.pay()` decides how much native ETH to wrap into WETH by reading `address(this).balance` — the router's total live ETH balance — rather than the ETH that was actually forwarded with the current call. Any ETH left on the router by a previous sender (e.g., a user who called a payable entry-point but omitted `refundETH()`) is silently consumed to subsidise the next caller's WETH payment, causing the original sender to lose those funds.

---

### Finding Description

`pay()` contains the following branch for WETH payments:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← live balance, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

When `nativeBalance >= value`, the function wraps ETH from the router's total balance and transfers it to the pool **without pulling anything from `payer`**. The `payer` (the original `msg.sender` stored in transient context) is completely bypassed. Any ETH that accumulated on the router from a prior transaction is consumed first, and the current caller pays nothing from their own wallet.

The `receive()` guard prevents arbitrary ETH donations:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

However, ETH legitimately accumulates on the router whenever a user calls any `payable` entry-point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `addLiquidityExactShares`, etc.) with `msg.value` larger than the amount actually consumed, and then omits the `refundETH()` tail call. This is a realistic and common omission — the protocol does not enforce the refund step.

The same `pay()` function is inherited by `MetricOmmPoolLiquidityAdder`, so the same exposure exists in the liquidity-adder callback path. [3](#0-2) 

---

### Impact Explanation

**Direct loss of user principal.** A victim who sends excess ETH and omits `refundETH()` loses that ETH to the next caller who swaps with WETH as `tokenIn`. The attacker pays zero from their own wallet for the WETH leg of their swap; the victim's stranded ETH is wrapped and forwarded to the pool instead. The loss equals the full amount of ETH left on the router, which can be up to the victim's entire `msg.value`.

---

### Likelihood Explanation

- `refundETH()` is a separate, optional call; nothing in the router enforces it.
- Integrators and end-users routinely send a round-number ETH value and rely on a refund step that is easy to omit or drop from a multicall array.
- The exploit requires no special privilege: any EOA can call `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0` immediately after a victim's transaction lands.
- MEV bots monitoring the mempool can front-run the victim's `refundETH()` call or simply watch for unrefunded balances between blocks.

---

### Recommendation

Track the ETH that belongs to the current call in a transient storage slot (the protocol already uses EIP-1153 transient storage for callback context). Record `msg.value` at entry to each payable function and decrement it as ETH is consumed inside `pay()`. Use only that tracked amount — never the raw `address(this).balance` — when deciding how much native ETH to wrap. Alternatively, require callers to pass an explicit `nativeAmountToUse` parameter so the router never silently reaches into its own balance beyond what the current sender authorised.

---

### Proof of Concept

```
1. Alice calls exactInputSingle{value: 1 ether}(
       pool, tokenIn=WETH, amountIn=0.4 ether, ...
   )
   → pay() reads address(this).balance = 1 ether ≥ 0.4 ether
   → wraps 0.4 ether, sends WETH to pool
   → 0.6 ether remains on the router
   → Alice omits refundETH() (or her multicall array is missing it)

2. Bob calls exactInputSingle{value: 0}(
       pool, tokenIn=WETH, amountIn=0.6 ether, ...
   )
   → pay() reads address(this).balance = 0.6 ether ≥ 0.6 ether
   → wraps 0.6 ether (Alice's ETH), sends WETH to pool
   → Bob's safeTransferFrom(Bob, pool, 0) is never called
   → Bob receives swap output for free

Result: Alice loses 0.6 ETH; Bob receives a full swap at zero cost.
```
