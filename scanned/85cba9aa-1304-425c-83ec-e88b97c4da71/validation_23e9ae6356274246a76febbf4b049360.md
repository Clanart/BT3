### Title
`addLiquidityWeighted` Reverts with `MaxAmountExceeded` Due to Per-Bin Ceiling Rounding Accumulation in Pay Phase — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`addLiquidityWeighted` uses a probe-then-pay pattern to scale liquidity shares within user-provided token caps. The scaling arithmetic uses floor division, but the pool's `addLiquidity` applies **ceiling division per bin** when computing token amounts. When multiple bins are used, the sum of per-bin ceilings in the pay phase can exceed `maxAmountToken0` even though the probe phase correctly measured `need0`, causing the function to revert with `MaxAmountExceeded` for valid user inputs.

---

### Finding Description

**Probe phase** (`KIND_PROBE`): The pool computes per-bin token amounts using ceiling division:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol:109-110
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
```

The callback reverts with `LiquidityProbe(need0, need1)` where `need0 = Σᵢ ceil(balance₀ᵢ × w.shares[i] / totalShares[i])`.

**Scale phase** (`_scaleWeightsToShares`): The scale factor and new shares are computed with floor division:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol:231-240
uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);  // floor
uint256 scaleWad  = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;
...
out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);  // floor
```

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol:165-167
if (amount0Delta > max0 || amount1Delta > max1) {
    revert
