The vulnerability is real. Let me trace the exact path.

**ETH entry path:** `receive()` blocks direct ETH transfers from non-WETH addresses, but all swap functions (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and `multicall` are `payable`, so users can send `msg.value` with them.

**Stranding mechanism:** In `pay()`, when `token == WETH` and `nativeBalance >= value`, it wraps exactly `value` ETH and leaves the remainder in the contract: [1](#0-0) 

If a user calls `exactInputSingle{value: 2 ether}(...)` where only `1 ether` is needed, the remaining `1 ether` persists in the router after the transaction ends.

**Theft path:** `refundETH()` has zero access control and sends the entire ETH balance to `msg.sender`: [2](#0-1) 

Any address can call this in a subsequent transaction and receive all stranded ETH.

**The `receive()` guard does not protect against this** — it only blocks plain ETH transfers, not `msg.value` attached to payable function calls. [3](#0-2)

### Title
Unguarded `refundETH` allows any caller to steal ETH stranded in the router by other users — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.refundETH()` sends the entire native ETH balance of the contract to `msg.sender` with no access control. Any caller can invoke it to claim ETH that was deposited by a different user via `msg.value` on a payable swap function.

### Finding Description
The router accepts ETH via `msg.value` on all payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). Inside `pay()`, when the input token is WETH and the contract holds native ETH, it wraps exactly the required amount and leaves any excess in the contract:

```solidity
// PeripheryPayments.sol:75-77
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

The excess ETH persists in the contract after the transaction ends. `refundETH()` then unconditionally transfers the full balance to `msg.sender`:

```solidity
// PeripheryPayments.sol:58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

An attacker monitoring the chain can call `refundETH()` in a separate transaction (or front-run the victim's own refund call) to drain all stranded ETH.

### Impact Explanation
Direct loss of user principal. Any ETH sent as `msg.value` in excess of the swap's actual requirement is permanently claimable by an arbitrary third party. The loss is bounded only by the excess ETH the victim sent.

### Likelihood Explanation
Medium. The intended usage pattern (documented in `IMetricOmmPoolLiquidityAdder` and demonstrated in tests) is to bundle the swap with `refundETH` in a single `multicall`. However, users who call payable swap functions directly with a rounded-up `msg.value` — a natural pattern when the exact WETH cost is unknown — will strand ETH. A bot watching for non-zero ETH balances on the router can reliably exploit this. [4](#0-3) 

### Recommendation
Restrict `refundETH` to only refund the original depositor, or track per-caller ETH deposits in transient storage and only allow each caller to reclaim their own contribution. Alternatively, document that `refundETH` must always be called in the same `multicall` as the deposit and add a check that reverts if called outside a multicall context.
