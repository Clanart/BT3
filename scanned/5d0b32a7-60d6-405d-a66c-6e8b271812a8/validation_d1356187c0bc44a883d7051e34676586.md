The vulnerability is real. The `pay()` function in `PeripheryPayments.sol` uses `address(this).balance` globally — any ETH stranded in the router from a prior user's incomplete multicall is consumed by the next WETH swap caller.

---

### Title
Stranded ETH in Router Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` checks the router's entire native ETH balance when settling a WETH payment. If a prior user's `multicall` left ETH in the router (by omitting `refundETH`), any subsequent caller whose `tokenIn == WETH` will have that ETH wrapped and applied toward their swap, paying less (or nothing) from their own wallet while the prior user's ETH is permanently consumed.

### Finding Description

`PeripheryPayments.pay()` contains the following WETH branch: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

`address(this).balance` is the router's **global** ETH balance — it has no per-user or per-call accounting. ETH sent with a `payable` `multicall` call lands in the router's balance and persists across calls if `refundETH` is not included. [2](#0-1) 

`multicall` is `payable` and iterates via `delegatecall`, so ETH sent with it accumulates in `address(this).balance`. The `receive()` guard only blocks direct plain-ETH transfers; it does not prevent ETH from being deposited via a `payable` function call. [3](#0-2) 

### Impact Explanation

User A's stranded ETH is directly stolen: it is wrapped to WETH and transferred to the pool on behalf of User B, reducing User B's required WETH pull from their wallet by exactly the stranded amount. User A receives nothing in return. This is a direct loss of principal with no recovery path once the subsequent swap executes.

### Likelihood Explanation

- Omitting `refundETH` from a multicall is a realistic user error, especially when the exact swap amount is not known in advance (e.g., exact-output swaps where the input consumed is determined at runtime).
- The attack is passive: User B does not need to do anything special — any WETH `exactInputSingle` or `exactInput` call that arrives while the router holds ETH will silently consume it.
- A sophisticated attacker can monitor the mempool for multicalls that send excess ETH without a trailing `refundETH` and front-run the refund with their own WETH swap.

### Recommendation

Track the ETH that belongs to the current `msg.value` context (e.g., store `msg.value` in a transient slot at the start of each top-level call and decrement it as ETH is consumed in `pay()`). Only allow `pay()` to use native ETH up to the amount attributed to the current call. Alternatively, require callers to pass an explicit `nativeAmountToUse` parameter so the router never silently consumes ambient ETH balance.

### Proof of Concept

```
1. User A: router.multicall{value: 1 ETH}([
       exactInputSingle(tokenIn=WETH, amountIn=0.9 ETH, ...)
       // no refundETH
   ])
   → pay(WETH, userA, pool, 0.9 ETH): nativeBalance=1 ETH >= 0.9 ETH
     → wraps 0.9 ETH, transfers WETH to pool
   → 0.1 ETH remains in router

2. User B: router.exactInputSingle(tokenIn=WETH, amountIn=0.1 ETH, ...)
   → pay(WETH, userB, pool, 0.1 ETH): nativeBalance=0.1 ETH >= 0.1 ETH
     → wraps router's 0.1 ETH (User A's), transfers WETH to pool
     → safeTransferFrom(userB, ...) is NEVER called

assert router.balance == 0          // passes
assert userA lost 0.1 ETH           // passes — stranded ETH consumed
assert userB paid 0 WETH from wallet // passes — free swap
``` [4](#0-3)
