### Title
Router's Accumulated ETH Balance Is Silently Consumed by Any Subsequent WETH Swap, Causing Prior ETH Depositors to Lose Funds - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

### Summary

The `pay` function in `PeripheryPayments.sol` uses the router's entire native ETH balance to subsidize WETH payments for **any caller**, without verifying that the ETH belongs to the current transaction's sender. ETH sent as `msg.value` to any `payable` router function that is not fully consumed and not reclaimed via `refundETH` accumulates in the router. Any subsequent user calling a WETH swap silently drains that ETH, causing the original depositor to lose their principal.

### Finding Description

The `pay` function contains three branches for WETH settlement:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);          // payer's WETH never pulled
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

Branches 1 and 2 consume `address(this).balance` without any check that the ETH belongs to the current caller. The router acts as both the privileged payment intermediary **and** the de-facto payer, consuming ETH that may belong to a prior user — the direct analog to the external report's "operator acting as buyer" scenario.

**How ETH accumulates in the router:**

The `receive()` guard blocks plain ETH transfers:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

However, `msg.value` attached to any `payable` function call bypasses `receive()`. A user who calls `exactInputSingle{value: X}(...)` with WETH as `tokenIn` and `amountIn < X` leaves `X - amountIn` ETH stranded in the router. The intended recovery path is to include `refundETH` in the same `multicall` batch: [3](#0-2) 

But `refundETH` is a separate call. If the user calls `exactInputSingle` directly (not via `multicall`), the excess ETH sits in the router until a separate `refundETH` transaction — a window during which any other user can drain it.

**The `else if (nativeBalance > 0)` branch** (lines 78–81) is a code path absent from the Uniswap v3 reference implementation. Uniswap v3 falls through to `safeTransferFrom` whenever `balance < value`. Metric OMM instead wraps whatever ETH is present and pulls only the shortfall from the payer. This means **even 1 wei of stranded ETH is consumed** by the next WETH swap, making the drain unconditional once any ETH accumulates. [4](#0-3) 

The same `pay` function is inherited by `MetricOmmPoolLiquidityAdder`, so the same drain path exists for WETH liquidity additions: [5](#0-4) 

### Impact Explanation

A user who sends ETH with a WETH swap and does not atomically call `refundETH` loses the unspent ETH to the next WETH swap caller. The loss equals the stranded ETH amount — potentially the full `msg.value` if the user sent ETH with a non-WETH swap by mistake. There is no error, no event, and no revert; the drain is silent and irreversible.

### Likelihood Explanation

The intended usage pattern (shown in tests) is `multicall([exactInputSingle(...), refundETH()])`: [6](#0-5) 

Users who call `exactInputSingle` or `exactOutputSingle` directly with `msg.value` — a natural pattern for ETH-in swaps — will strand ETH. Front-runners or any subsequent WETH swapper in the same block can drain it. The trigger requires no privilege and no special setup.

### Recommendation

1. **Track per-transaction ETH in transient storage**: At the start of each `payable` entry point, record `msg.value` in a transient slot. In `pay`, consume only from that recorded value, not from `address(this).balance`. This prevents cross-transaction ETH leakage.
2. **Remove the partial-ETH branch**: Revert to the Uniswap v3 behavior — if `address(this).balance < value`, pull the full amount from the payer via `safeTransferFrom`. This eliminates the novel drain vector while preserving the intended ETH-wrapping convenience for the `nativeBalance >= value` case.
3. **Document and enforce the `multicall` + `refundETH` pattern**: If the partial-ETH branch is kept, add a guard that reverts if `address(this).balance` exceeds `msg.value` at the time of the call, ensuring only the current transaction's ETH is used.

// Setup: pool with WETH/token1, router deployed

// Step 1: User A swaps WETH→token1, sends 1 ETH, amountIn = 0.5 ETH
// User A does NOT call refundETH (calls exactInputSingle directly, not via multicall)
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 0.5 ether,
    amountOutMinimum: 0,
    recipient: userA,
    deadline: ...,
    priceLimitX64: 0,
    extensionData: ""
}));
// pay() uses 0.5 ETH from msg.value; 0.5 ETH remains in router
// assert(address(router).balance == 0.5 ether)

// Step 2: User B swaps WETH→token1, amountIn = 0.5 ETH, sends 0 ETH
// User B has approved router for WETH but pay() never calls safeTransferFrom
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 0.5 ether,
    amountOutMinimum: 0,
    recipient: userB,
    deadline: ...,
    priceLimitX64: 0,
    extensionData: ""
}));
// pay(): nativeBalance (0.5 ETH) >= value (0.5 ETH) → branch 1
// Router wraps 0.5 ETH and sends WETH to pool; User B's WETH never pulled
// assert(address(router).balance == 0)
// assert(weth.balanceOf(userB) unchanged)  // User B paid nothing
// User A lost 0.5 ETH with no recourse
```
