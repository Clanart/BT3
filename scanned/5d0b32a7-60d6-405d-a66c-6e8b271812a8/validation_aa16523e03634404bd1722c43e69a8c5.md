### Title
`pauseLevel` Enforced on `swap` but Not on `addLiquidity` — Users Can Deposit Into Paused Pools - (File: metric-core/contracts/MetricOmmPool.sol)

---

### Summary

`MetricOmmPool` has a `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused) and a `whenNotPaused` modifier that reverts when `pauseLevel != 0`. The modifier is applied to `swap()` but is **absent** from `addLiquidity()` and `removeLiquidity()`. When a pool is paused — for example, in response to an oracle manipulation or a security incident — users can still deposit new funds into the compromised pool via `addLiquidity()` directly or through `MetricOmmPoolLiquidityAdder`.

---

### Finding Description

`MetricOmmPool` declares `pauseLevel` and the `whenNotPaused` modifier:

```solidity
uint8 internal pauseLevel;
// 0 = active, 1 = paused by admin, 2 = paused by protocol

modifier whenNotPaused() {
    _checkNotPaused();
    _;
}
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [1](#0-0) [2](#0-1) [3](#0-2) 

`swap()` correctly applies the modifier:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
``` [4](#0-3) 

But `addLiquidity()` does **not**:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
``` [5](#0-4) 

The periphery `MetricOmmPoolLiquidityAdder` routes all user deposits directly to `pool.addLiquidity(...)` with no pause-state check of its own:

```solidity
IMetricOmmPoolActions(pool).addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData)
``` [6](#0-5) 

---

### Impact Explanation

When a pool is paused (e.g., because the oracle is returning manipulated prices, or a critical bug is discovered), the pause is intended to halt all pool activity. Because `addLiquidity` bypasses `whenNotPaused`, users — including those interacting through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` or `addLiquidityWeighted` — can still deposit tokens into the paused pool. If the pool was paused due to an active exploit or price oracle failure, newly deposited funds are immediately at risk of being drained or permanently mispriced, resulting in direct loss of user principal.

---

### Likelihood Explanation

The factory can pause any pool at any time via `setPause()`. Pausing is a routine emergency response. Any user who deposits after a pause is triggered — whether through the periphery adder or directly — loses funds. No special permissions or unusual conditions are required beyond the pool being paused.

---

### Recommendation

Add `whenNotPaused` to `addLiquidity()` in `MetricOmmPool.sol`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` may intentionally remain unguarded to allow LPs to exit during a pause, but new deposits must be blocked.

---

### Proof of Concept

1. Factory admin calls `pool.setPause(1)` to pause the pool due to a suspected oracle issue.
2. Attacker (or unsuspecting user) calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, ...)`.
3. `_addLiquidity` calls `IMetricOmmPoolActions(pool).addLiquidity(...)` — no `whenNotPaused` check fires.
4. Tokens are transferred from the user into the paused pool.
5. If the pool was paused due to an active exploit, the deposited tokens are immediately at risk. [7](#0-6) [8](#0-7)
