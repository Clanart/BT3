### Title
Stale Router ETH Balance Consumed by Subsequent User's WETH Payment — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

The `pay()` function in `PeripheryPayments` uses `address(this).balance` unconditionally when settling a WETH payment obligation. It does not distinguish between ETH that belongs to the current transaction's `msg.value` and ETH that was stranded on the router by a previous user. A subsequent user's WETH swap silently consumes the stranded ETH, causing the original depositor to lose their funds.

### Finding Description

`pay()` in `PeripheryPayments.sol` handles WETH settlement as follows: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — the router's **entire** native ETH balance — and uses it to cover the current user's WETH obligation before pulling from the payer's wallet. There is no check that this ETH was deposited by the current caller or the current transaction.

ETH can be stranded on the router legitimately: the intended usage pattern is to send `msg.value` with a payable swap call and then call `refundETH()` in the same multicall to recover the excess. [2](#0-1) 

If a user sends more ETH than the swap consumes and omits `refundETH()` — or calls `exactInputSingle{value: ...}` directly without a multicall — the surplus ETH remains on the router across transaction boundaries. [3](#0-2) 

In a subsequent transaction, any user who calls `exactInputSingle` (or any other WETH-input swap) triggers `_justPayCallback` → `pay(WETH, newUser, pool, amount)`. The `pay()` function sees the stranded ETH, deposits it as WETH, and forwards it to the pool — fully satisfying the new user's payment obligation without touching their wallet. [4](#0-3) 

The transient payer context correctly identifies the new user as the payer, but `pay()` ignores that identity when native ETH is available, consuming the stranded balance instead.

**Analog to the external bug:** In the Perpetual Pools report, stale pool balances from future mint commits are used to calculate fees for earlier burn commits — a later user's deposit silently benefits from an earlier user's state. Here, a later user's WETH swap silently consumes an earlier user's stranded ETH. In both cases, a shared balance that should be attributable to one user is consumed by a different user's operation.

### Impact Explanation

- **Direct loss of user principal**: User A's stranded ETH is permanently consumed by User B's swap. User A receives no compensation and cannot recover the ETH.
- **User B pays zero WETH** for a swap that should have cost them WETH from their own wallet.
- The pool receives the correct WETH amount (the invariant from the pool's perspective holds), so the pool itself is not insolvent — but the router acts as an unattributed ETH sink that redistributes value from one user to another.
- Loss magnitude equals the stranded ETH amount, which can be arbitrarily large (any user who sends excess `msg.value` without `refundETH()`).

### Likelihood Explanation

- The intended usage pattern (documented in `IMetricOmmPoolLiquidityAdder` and tested in `MetricOmmSimpleRouterNativeTest`) explicitly requires users to pair ETH-input swaps with `refundETH()` in a multicall. Users who call payable swap functions directly (not via multicall) or who omit `refundETH()` will strand ETH.
- The `exactInputSingle` function is `payable`, so wallets and integrators can send ETH with it without a multicall.
- Any subsequent WETH-input swap by any user in any later block will silently drain the stranded balance. No special permissions or setup are required.

### Recommendation

Track only the ETH that arrived in the **current transaction** as eligible for WETH conversion. The standard approach is to compare `address(this).balance` before and after the call, or to pass `msg.value` explicitly through the call stack and use only that amount in `pay()`. Alternatively, restrict the native-ETH-to-WETH conversion path to only consume ETH that was explicitly sent with the current top-level call (e.g., by recording `msg.value` at multicall entry and decrementing it as it is consumed).

### Proof of Concept

```
Setup:
  - Router deployed with WETH address.
  - Pool with WETH as token0, token1 as ERC20.
  - User A has 2000 wei ETH.
  - User B has 0 ETH, 0 WETH, but wants to swap 1000 WETH → token1.

Step 1 (User A, Tx 1):
  User A calls exactInputSingle{value: 2000}(amountIn=1000, tokenIn=WETH, ...)
  → pay() sees nativeBalance=2000 >= 1000, deposits 1000 ETH as WETH, pays pool.
  → 1000 ETH remains on router.
  → User A does NOT call refundETH() (omitted from multicall or called directly).

Step 2 (User B, Tx 2):
  User B calls exactInputSingle{value: 0}(amountIn=1000, tokenIn=WETH, ...)
  → pay(WETH, UserB, pool, 1000) is called.
  → nativeBalance = 1000 (User A's stranded ETH).
  → nativeBalance >= value: deposits 1000 ETH as WETH, transfers to pool.
  → User B's WETH balance: unchanged (0 spent).
  → User B receives token1 output.

Result:
  User A: lost 1000 ETH (stranded, consumed by User B).
  User B: received token1 output for free.
  Pool: received correct WETH, unaware of the misattribution.
``` [5](#0-4)

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
