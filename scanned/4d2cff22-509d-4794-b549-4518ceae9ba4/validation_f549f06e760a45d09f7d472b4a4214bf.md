### Title
Missing `bidPriceX64 < askPriceX64` ordering invariant in `_validateHypotheticalPrices` allows inverted-spread hypothetical quotes — (File: `metric-periphery/contracts/lens/MetricOmmSwapQuoter.sol`)

---

### Summary

`_validateHypotheticalPrices` only checks that the `bidPricesX64` and `askPricesX64` arrays have the correct length. It never asserts that `bidPricesX64[i] < askPricesX64[i]` for any hop. A caller can therefore supply an inverted spread (bid ≥ ask) to `quoteHypotheticalExactInput` / `quoteHypotheticalExactOutput`, which forwards those prices verbatim to the pool's `simulateSwapAndRevert`. The resulting quoted amounts are wrong, and any integrator that uses them to set `amountOutMinimum` or `amountInMaximum` for a real swap can be induced into a fund-losing execution.

---

### Finding Description

`MetricOmmSwapQuoter._validateHypotheticalPrices` is the sole guard for the per-pool bid/ask price arrays:

```solidity

    uint256 poolCount,
    uint128[] calldata bidPricesX64,
    uint128[] calldata askPricesX64
) internal pure {
    if (bidPricesX64.length != poolCount || askPricesX64.length != poolCount) {
        revert InvalidPath();
    }
}

The invariant `bid < ask` is the foundational market-maker constraint: the bid is the price at which the pool buys token0 (sells token1) and the ask is the price at which it sells token0 (buys token1). Inverting them (bid ≥ ask) collapses or reverses the spread, causing the pool's bin-math to compute amounts that are either zero, arbitrarily large, or directionally wrong. The quoter surfaces those corrupted amounts as the return value of `quoteHypotheticalExactInput` / `quoteHypotheticalExactOutput` with no indication that the inputs were invalid.

This is the direct analog of the UpDown `startGame` bug: just as `stopPredictAt` and `endTime` have an assumed ordering (`stopPredictAt < endTime`) that is never enforced, `bidPriceX64` and `askPriceX64` have an assumed ordering (`bid < ask`) that `_validateHypotheticalPrices` never enforces.

---

### Impact Explanation

**Bad-price execution induced by inverted-spread quote.** An integrating contract or off-chain system calls `quoteHypotheticalExactInput` with `bidPricesX64[i] >= askPricesX64[i]` for one or more hops. The returned `amountOut` is incorrect (too high or too low depending on the pool's bin-math behavior with an inverted spread). The integrator uses that figure to set `amountOutMinimum` in a subsequent real `exactInput` call on the router. If the corrupted quote is inflated, the real swap reverts (DoS). If it is deflated, the real swap succeeds but the user receives far less output than expected, constituting a direct loss of principal. The same applies symmetrically to `quoteHypotheticalExactOutput` / `amountInMaximum`: a deflated `amountIn` quote causes the real swap to revert; an inflated one causes the user to overpay.

---

### Likelihood Explanation

Any caller of the public `quoteHypotheticalExactInput` / `quoteHypotheticalExactOutput` functions can supply arbitrary `bidPricesX64` and `askPricesX64` arrays. No privileged role is required. The trigger is reachable by any EOA or contract. Likelihood is **medium**: the quoter is a lens contract intended for off-chain use, so the most common path is an integrator bug or a malicious price feed that supplies inverted values, rather than a deliberate on-chain attack.

---

### Recommendation

Add a per-hop ordering check inside `_validateHypotheticalPrices`:

```solidity
function _validateHypotheticalPrices(
    uint256 poolCount,
    uint128[] calldata bidPricesX64,
    uint128[] calldata askPricesX64
) internal pure {
    if (bidPricesX64.length != poolCount || askPricesX64.length != poolCount) {
        revert InvalidPath();
    }
    for (uint256 i = 0; i < poolCount; i++) {
        if (bidPricesX64[i] >= askPricesX64[i]) {
            revert InvalidHypotheticalPrices(i, bidPricesX64[i], askPricesX64[i]);
        }
    }
}
```

This mirrors the recommendation in the UpDown report: add a `require`/`revert` that enforces the assumed ordering before the values are consumed.

---

### Proof of Concept

1. Deploy or reference a live `MetricOmmSwapQuoter` and a valid pool with token0/token1.
2. Call `quoteHypotheticalExactInput` with a single-hop path where `bidPricesX64[0] = askPricesX64[0]` (equal spread, zero width) or `bidPricesX64[0] > askPricesX64[0]` (inverted spread).
3. Observe that `_validateHypotheticalPrices` does not revert; the call proceeds to `simulateSwapAndRevert` with the inverted prices.
4. The returned `(totalIn, totalOut)` values differ from those produced by a valid `(bid < ask)` pair, demonstrating the corrupted quote.
5. Use the corrupted `totalOut` as `amountOutMinimum` in a real `exactInput` call on `MetricOmmSimpleRouter`. Depending on whether the corruption is upward or downward, the real swap either reverts (DoS) or settles with the user receiving less output than the live oracle price would justify (principal loss).
