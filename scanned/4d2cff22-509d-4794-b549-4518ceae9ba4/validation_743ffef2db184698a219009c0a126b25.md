Looking at the actual code to trace the vulnerability path precisely.

The vulnerability is real. Here is the complete analysis:

---

### Title
Trapped ETH in `MetricOmmPoolLiquidityAdder` is silently consumed to subsidize a subsequent user's WETH liquidity deposit — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `receive()` guard in `PeripheryPayments` only blocks direct bare ETH transfers, but all `payable` entry points (`multicall`, `addLiquidityExactShares`, `addLiquidityWeighted`, `refundETH`, `unwrapWETH9`, `sweepToken`) can accumulate ETH in the contract across transactions. The `pay()` function's WETH branch unconditionally consumes `address(this).balance` before pulling from the actual payer, so any ETH left by a prior user is silently spent to subsidize the next user's WETH deposit, causing the prior user to permanently lose their ETH.

---

### Finding Description

**Step 1 — ETH accumulates in the contract.**

The `receive()` guard is:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [1](#0-0) 

This only fires on bare ETH transfers (no calldata). ETH sent as `msg.value` to any `payable` function bypasses it entirely. The following entry points are all `payable`:

- `multicall(bytes[] calldata data) public payable`
- `addLiquidityExactShares(...) external payable`
- `addLiquidityWeighted(...) external payable`
- `refundETH() external payable`
- `unwrapWETH9(...) public payable`
- `sweepToken(...) public payable` [2](#0-1) 

A user who calls `multicall{value: X}([addLiquidityExactShares(...)])` for a non-WETH pool and omits `refundETH()` leaves `X` wei permanently in the contract. This is the standard Uniswap-style multicall pattern and the omission is easy to make.

**Step 2 — The WETH branch in `pay()` consumes the trapped ETH.**

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [3](#0-2) 

`address(this).balance` is the **entire** contract ETH balance, not just ETH the current caller sent. When a second user calls `addLiquidityExactShares` for a WETH pool, the callback invokes `pay(WETH, userB, pool, value)`. If `nativeBalance > 0` from user A's trapped ETH:

- The contract wraps `nativeBalance` of user A's ETH and transfers it to the pool.
- It then pulls only `value - nativeBalance` from user B via `safeTransferFrom`.
- User A's ETH is gone; user B received a discount equal to `nativeBalance`. [4](#0-3) 

**Step 3 — The callback path confirms the payer is always `msg.sender` of the outer call.**

```solidity
(address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
...
pay(token0, payer, msg.sender, amount0Delta);
pay(token1, payer, msg.sender, amount1Delta);
``` [5](#0-4) 

The `payer` is set to `msg.sender` of `addLiquidityExactShares` at `_setPayContext` time: [6](#0-5) 

There is no mechanism that restricts `address(this).balance` to only ETH the current payer sent.

---

### Impact Explanation

Any user who sends ETH to the adder via a payable function and omits `refundETH()` permanently loses that ETH. It is silently consumed by the next user who adds liquidity to a WETH pool. This is a direct, irreversible loss of user principal with no recovery path. The attacker (user B) needs only to observe the contract's ETH balance (publicly readable) and call `addLiquidityExactShares` for any WETH pool.

---

### Likelihood Explanation

The multicall + `refundETH()` omission is a well-documented footgun in Uniswap-style routers. Front-ends, scripts, and integrators routinely forget the refund step. The attacker's action is a normal, permissionless liquidity deposit — no special role or setup is required. The ETH balance is publicly visible on-chain, making the attack trivially detectable and repeatable.

---

### Recommendation

Restrict the WETH branch to only ETH that the **current caller** explicitly sent in the current transaction. The simplest fix is to track `msg.value` at the outer entry point and pass it through to `pay()`, using only that amount as the native contribution — not the full `address(this).balance`. Alternatively, after each `pay()` call, refund any remaining ETH balance to the payer, or require that `address(this).balance == msg.value` at the start of each non-multicall entry point.

---

### Proof of Concept

```solidity
// Foundry test sketch
function test_trappedEthSubsidizesWethDeposit() public {
    // User A sends ETH via multicall for a non-WETH pool, forgets refundETH()
    vm.deal(userA, 1 ether);
    vm.prank(userA);
    adder.multicall{value: 1 ether}(
        _encodeAddLiquidityForNonWethPool(...)
    );
    // 1 ETH is now trapped in the adder

    // User B adds liquidity to a WETH pool requiring 2 WETH
    uint256 wethBefore = weth.balanceOf(userB);
    vm.prank(userB);
    adder.addLiquidityExactShares(wethPool, userB, salt, deltas, 2e18, 0, "");
    uint256 wethAfter = weth.balanceOf(userB);

    // User B only spent 1 WETH (2 - 1 trapped ETH), pool received full 2 WETH
    assertEq(wethBefore - wethAfter, 1 ether); // should be 2 ether — FAIL
    assertEq(address(adder).balance, 0);        // user A's ETH is gone
}
```

The pool receives the full `value` in WETH, user B's `safeTransferFrom` is reduced by `nativeBalance`, and user A's ETH is permanently lost.
