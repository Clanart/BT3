### Title
Stranded ETH in Router Consumed by Subsequent WETH Swap Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **entire** native ETH balance — when funding a WETH wrap, with no check that the ETH was deposited by the current `payer`. Any ETH left in the router from a prior transaction is silently consumed to subsidize a later caller's swap.

---

### Finding Description

The `pay()` function has three branches for `token == WETH`: [1](#0-0) 

```
nativeBalance = address(this).balance   // ← entire router balance, not msg.value
if nativeBalance >= value  → wrap all from router, no transferFrom from payer
if nativeBalance > 0       → wrap partial from router, pull remainder from payer
else                       → pull all from payer via transferFrom
```

The router is `payable` on every swap entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). A user who sends `msg.value > amountIn` and omits a trailing `refundETH()` call leaves the surplus ETH in the router permanently (the `receive()` guard only blocks direct pushes, not `msg.value` on payable functions). [2](#0-1) 

Once stranded, the next caller whose swap triggers `pay(WETH, bob, pool, value)` will have their payment partially or fully funded by the stranded ETH, with no `transferFrom` deducted from them for that portion.

---

### Impact Explanation

**Alice's ETH is stolen.** Concretely:

- Alice calls `exactInputSingle{value: 1 ETH}(amountIn=0.9 ETH, tokenIn=WETH)`. The callback wraps 0.9 ETH; 0.1 ETH remains in the router.
- Bob calls `exactInputSingle{value: 0}(amountIn=0.1 ETH, tokenIn=WETH)`. `nativeBalance = 0.1 ETH >= value = 0.1 ETH` → branch at line 75 fires, wraps Alice's 0.1 ETH, sends WETH to pool. Bob pays nothing.

Alice loses 0.1 ETH; Bob receives a free swap. The attack is **permissionless** — Bob needs no approvals, no special role, and no coordination with Alice. [3](#0-2) 

---

### Likelihood Explanation

The precondition — ETH stranded in the router — is realistic:

1. Users calling `exactInputSingle{value: X}` with `X > amountIn` and no `refundETH()` afterward.
2. `multicall` batches that include a WETH swap but omit the trailing `refundETH()` call. [4](#0-3) 

The router provides no on-chain enforcement that `refundETH()` is called. A griefing bot can monitor the mempool for transactions that leave ETH in the router and front-run the victim's `refundETH()` with a zero-value WETH swap.

---

### Recommendation

Track only the ETH that the **current caller** deposited. The standard pattern is to snapshot `msg.value` at entry and pass it through to `pay()`, then use only that amount for wrapping:

```solidity
// In pay(), replace address(this).balance with a caller-scoped msgValue parameter
function pay(address token, address payer, address recipient, uint256 value, uint256 msgValue) internal {
    ...
    } else if (token == WETH) {
        uint256 nativeBalance = msgValue;   // only current caller's ETH
        ...
    }
}
```

Alternatively, enforce that `address(this).balance == msg.value` at the start of every payable entry point (revert if the router holds pre-existing ETH), or require callers to wrap ETH to WETH themselves before calling the router.
