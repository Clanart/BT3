### Title
Excess ETH Not Refunded After Exact-Output Swaps — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

### Summary

`exactOutputSingle` and `exactOutput` are both `payable` and accept native ETH as the input token (via WETH wrapping in `PeripheryPayments.pay`). The swap callback pays only the actual pool-requested `amountIn`, which is ≤ `amountInMaximum`. Any excess ETH (`msg.value − amountIn`) is never refunded and remains stranded in the router.

### Finding Description

In `exactOutputSingle`, the user sends `msg.value` ≥ `amountInMaximum`. The pool is called with a negative `amountSpecified` (exact output). The pool determines the actual input required and fires `metricOmmSwapCallback`, which routes to `_justPayCallback`: [1](#0-0) 

`extractPositiveAmount` returns only what the pool actually requested — the true `amountIn`, which may be strictly less than `msg.value`. After the callback, `exactOutputSingle` checks the slippage bound and clears transient state, but **never refunds the difference**: [2](#0-1) 

The same pattern applies to `exactOutput` (multi-hop): the recursive `_exactOutputIterateCallback` pays only the pool-requested amount at the innermost hop, and the outer function clears context without refunding: [3](#0-2) 

### Impact Explanation

Any ETH sent above the actual swap cost is permanently stranded in the router (or recoverable only if a `sweep`/`refundETH` helper exists and the user knows to call it in a `multicall` batch). A user who calls `exactOutputSingle` standalone with `msg.value = amountInMaximum` and the pool consumes less loses the difference with no on-chain mechanism to recover it automatically.

### Likelihood Explanation

This is triggered by any user who:
1. Uses ETH (native) as the input token for an exact-output swap, and
2. Calls `exactOutputSingle` or `exactOutput` directly (not via a `multicall` that also calls `refundETH`).

This is a standard user pattern — exact-output swaps with ETH are a primary use case for any DEX router.

### Recommendation

Add an automatic ETH refund at the end of `exactOutputSingle` and `exactOutput`, analogous to the mitigation in the referenced report:

```solidity
// After _clearExpectedCallbackPool():
if (address(this).balance > 0) {
    Address.sendValue(payable(msg.sender), address(this).balance);
}
```

Or, at minimum, document prominently that users **must** batch a `refundETH()` call via `multicall` when using ETH as input for exact-output swaps, and ensure `PeripheryPayments` exposes such a function.

### Proof of Concept

1. User wants exactly 1000 USDC out, paying ETH in.
2. User calls `exactOutputSingle({amountOut: 1000e6, amountInMaximum: 1 ether, tokenIn: WETH, ...})` with `msg.value = 1 ether`.
3. Pool determines it only needs 0.6 ETH to fill 1000 USDC at the current oracle price.
4. `_justPayCallback` wraps and pays 0.6 ETH to the pool.
5. `exactOutputSingle` returns `amountIn = 0.6 ether`, satisfies `≤ amountInMaximum`, clears context.
6. The remaining 0.4 ETH sits in the router with no automatic refund path.
7. User has lost 0.4 ETH unless they separately call a sweep/refund helper (if one exists). [2](#0-1) [1](#0-0)

```text
  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
```

```text
  function exactOutput(ExactOutputParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint8 tradesLeftAfterThis = uint8(params.pools.length - 1);
    address pool = params.pools[tradesLeftAfterThis];
    bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, tradesLeftAfterThis);
    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _initCallbackContextforRecursiveOutput(
      pool, CALLBACK_MODE_EXACT_OUTPUT_ITERATE, tradesLeftAfterThis, msg.sender, params.tokens[0]
    );
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
      .swap(
        params.recipient,
        zeroForOne,
        -expectedAmountOut,
        MetricOmmSwapPath.openLimit(zeroForOne),
        abi.encode(
          ExactOutputIterateCallbackData({
          tokens: params.tokens,
          pools: params.pools,
          extensionDatas: params.extensionDatas,
          zeroForOneBitMap: params.zeroForOneBitMap,
          amountInMax: params.amountInMaximum
        })
        ),
        params.extensionDatas[tradesLeftAfterThis]
      );

    int128 amountOut = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = _getExactOutputAmountIn();
    _clearExpectedCallbackPool();
  }
```
