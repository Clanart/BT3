### Title
`pay()` uses `safeTransfer` instead of `safeTransferFrom` for non-WETH external payers, making all non-WETH swaps and liquidity additions permanently revert — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` internal helper in `PeripheryPayments.sol` has a wrong transfer primitive in its final `else` branch. When the payer is an external user **and** the token is not WETH, the function calls `IERC20(token).safeTransfer(recipient, value)` — pulling from the **contract's own balance** — instead of `IERC20(token).safeTransferFrom(payer, recipient, value)` — pulling from the **user's balance**. Because the router/adder never holds the input token, every non-WETH swap and every non-WETH liquidity addition reverts unconditionally.

---

### Finding Description

`PeripheryPayments.pay()` has three branches: [1](#0-0) 

```
if (payer == address(this))          → safeTransfer  (contract pays intermediate hop)
else if (token == WETH)              → wrap ETH or safeTransferFrom payer
else                                 → safeTransfer  ← BUG
```

The final `else` is reached whenever `payer != address(this)` **and** `token != WETH` — i.e., the very common case of a user swapping or adding liquidity with any plain ERC-20. The code at line 86:

```solidity
} else {
    IERC20(token).safeTransfer(recipient, value);   // transfers from THIS contract
}
```

should be:

```solidity
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value);  // pulls from payer
}
```

Compare with the reference Uniswap V3 `PeripheryPayments` which correctly uses `safeTransferFrom` in the equivalent branch. The Metric OMM version inverted the `payer == address(this)` guard and the WETH guard relative to Uniswap V3, leaving the non-WETH external-payer path with the wrong primitive.

**Affected call sites:**

1. **Router swap callback** — `_justPayCallback` calls `pay(_getTokenToPay(), _getPayer(), msg.sender, amount)`. For the first (or only) hop, `_getPayer()` returns the original `msg.sender` (external user). If `tokenIn` is not WETH, branch 3 fires and the contract tries to `safeTransfer` a token it does not hold. [2](#0-1) 

2. **Exact-output iterate callback** — same `pay()` call at the innermost hop where `_getPayer()` is the original user. [3](#0-2) 

3. **Liquidity adder callback** — `metricOmmModifyLiquidityCallback` calls `pay(token0, payer, pool, amount0Delta)` and `pay(token1, payer, pool, amount1Delta)` where `payer` is the original `msg.sender`. Any pool whose token0 or token1 is not WETH will revert. [4](#0-3) 

---

### Impact Explanation

**Broken core pool functionality — unusable swap and liquidity flows.**

Every call to `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` where `tokenIn` is not WETH reverts inside the pool callback. Every call to `addLiquidityExactShares` or `addLiquidityWeighted` where either pool token is not WETH reverts inside the liquidity callback. Because WETH is only one of many possible pool tokens, the vast majority of pools and routes are completely non-functional. No user funds are at risk of theft, but the entire non-WETH surface of the protocol is permanently DoS'd.

---

### Likelihood Explanation

**High.** Any unprivileged user attempting a standard ERC-20 swap or liquidity addition triggers the bug. No special setup is required. The revert is deterministic: the router never pre-funds itself with input tokens, so `safeTransfer` from the contract's zero balance always fails.

---

### Recommendation

```diff
// metric-periphery/contracts/base/PeripheryPayments.sol

     } else {
-        IERC20(token).safeTransfer(recipient, value);
+        IERC20(token).safeTransferFrom(payer, recipient, value);
     }
``` [5](#0-4) 

---

### Proof of Concept

1. Deploy a pool with `token0 = USDC`, `token1 = DAI`.
2. User approves `MetricOmmSimpleRouter` for 1000 USDC.
3. User calls `exactInputSingle({ pool, tokenIn: USDC, tokenOut: DAI, amountIn: 1000, ... })`.
4. Pool executes the swap and calls `metricOmmSwapCallback`.
5. Router calls `_justPayCallback` → `pay(USDC, user, pool, 1000)`.
6. `payer (user) != address(this)` → skip branch 1.
7. `USDC != WETH` → skip branch 2.
8. Branch 3: `IERC20(USDC).safeTransfer(pool, 1000)` — transfers from the **router's** USDC balance, which is **0**.
9. ERC-20 `transfer` reverts with insufficient balance.
10. The entire `exactInputSingle` call reverts. The user's swap is impossible.

The same revert path applies to `exactInput` (first hop), `exactOutputSingle`, `exactOutput` (innermost hop), and all four `addLiquidity*` overloads in `MetricOmmPoolLiquidityAdder` whenever a non-WETH token must be pulled from the caller.

```text
  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
```
