### Title
Stale ETH in `PeripheryPayments.pay` Is Silently Consumed by Subsequent WETH Swaps, Stealing Prior Users' Funds — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay` helper in `PeripheryPayments` uses `address(this).balance` — the router's **total** native ETH balance — when settling a WETH leg. ETH accumulates in the router whenever a user calls `multicall{value: X}(...)` without appending a `refundETH()` call. Any subsequent WETH swap then silently consumes that stranded ETH, giving the second user a free or discounted swap while the first user permanently loses their ETH.

---

### Finding Description

`PeripheryPayments.pay` is the single settlement path for every WETH payment in both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder`. When `token == WETH` and the caller is an external user (`payer != address(this)`), the function reads the router's full native balance and uses it first:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total router ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value); // payer pays NOTHING
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // payer underpays
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

ETH enters the router through `multicall`, which is `payable`:

```solidity
function multicall(bytes[] calldata data) public payable override returns (bytes[] memory results) {
``` [2](#0-1) 

The `receive()` guard only blocks **plain** ETH transfers; it does not prevent ETH from entering via a `payable` function call:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [3](#0-2) 

So whenever a user calls `multicall{value: X}(...)` and omits `refundETH()`, the surplus ETH is permanently stranded in the router. The next user who executes any WETH swap — even with zero `msg.value` — will have the router silently consume that stranded ETH on their behalf.

The same `pay` function is shared by `MetricOmmPoolLiquidityAdder`, so the same drain path exists for WETH liquidity additions. [4](#0-3) 

---

### Impact Explanation

**Direct loss of user principal.** Two distinct loss magnitudes exist:

| Router ETH vs. swap `value` | Effect on victim (User A) | Effect on attacker (User B) |
|---|---|---|
| `nativeBalance >= value` | Loses up to `value` ETH | Pays **zero** WETH; receives full swap output for free |
| `0 < nativeBalance < value` | Loses `nativeBalance` ETH | Pays only `value − nativeBalance` WETH; receives a discount |

In the worst case User B executes a WETH swap equal to the stranded balance and pays nothing at all, while User A's ETH is irreversibly transferred to the pool.

---

### Likelihood Explanation

Medium. The trigger condition — a user sending ETH via `multicall` without `refundETH()` — is a realistic and documented usage pattern (the test suite explicitly demonstrates it as the expected pattern for native-ETH swaps). Once ETH is stranded, **any** subsequent WETH swap by any address drains it; no special permissions or privileged access are required.

---

### Recommendation

Track only the ETH that belongs to the current call. The standard fix (used by Uniswap v3's `SwapRouter`) is to pass the current `msg.value` through the call chain and cap native consumption to that amount. A minimal targeted fix is to compare `address(this).balance` against `msg.value` inside `pay` and only use native ETH when it was explicitly provided in the current top-level call. Alternatively, enforce that `refundETH()` is always the last call in any `multicall` batch that includes a native-ETH value — but this is a social/documentation control, not a code-level invariant.

---

### Proof of Concept

```
1. User A calls:
       router.multicall{value: 1 ether}([
           exactInputSingle(tokenIn=WETH, amountIn=0.5 ether, ...)
           // ← refundETH() omitted
       ])
   → 0.5 ETH remains in the router.

2. User B (attacker) calls (no ETH sent):
       router.exactInputSingle(
           tokenIn=WETH, amountIn=0.5 ether, recipient=attacker, ...
       )

3. Pool calls router.metricOmmSwapCallback(amount0Delta, amount1Delta, ...)
   → _justPayCallback → pay(WETH, userB, pool, 0.5 ether)

4. Inside pay():
       nativeBalance = address(this).balance = 0.5 ether   // User A's stranded ETH
       nativeBalance >= value  →  branch 1 taken
       IWETH9(WETH).deposit{value: 0.5 ether}()            // wraps User A's ETH
       IERC20(WETH).safeTransfer(pool, 0.5 ether)          // pays pool
       // safeTransferFrom(userB, ...) is NEVER called

5. Result:
   - User B receives full swap output, pays 0 WETH.
   - User A loses 0.5 ETH with no recourse.
```
