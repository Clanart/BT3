### Title
Router `pay()` Unconditionally Consumes Any Native ETH on the Router to Settle WETH Obligations, Enabling Cross-User ETH Theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` helper in `PeripheryPayments.sol` checks `address(this).balance` (all native ETH held by the router) and uses it to satisfy a WETH payment obligation for any caller. Because the router can accumulate native ETH from a prior user's `msg.value` (sent via `multicall`), a subsequent unprivileged caller whose `tokenIn` is WETH can have their entire swap settled using another user's stranded ETH, stealing it.

---

### Finding Description

`PeripheryPayments.pay()` contains the following logic for WETH payments: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — the **total** native ETH held by the router — and uses it to fund the payment before pulling from `payer`. There is no attribution: the function cannot distinguish ETH that belongs to the current caller from ETH left on the router by a prior user's `msg.value`.

The router accumulates native ETH legitimately via `multicall{value: N}(...)` calls. A user who sends `msg.value` for a WETH swap but omits a `refundETH()` step (or whose refund step is front-run) leaves ETH stranded on the router. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in `multicall`. [2](#0-1) 

`multicall` itself is `payable` and uses `delegatecall`, so `msg.value` is accessible across all steps but is not automatically refunded: [3](#0-2) 

---

### Impact Explanation

**Direct loss of user principal.** Alice's stranded ETH is consumed to pay for Bob's WETH swap. Alice calls `refundETH()` later and receives less ETH than she deposited. Bob's swap costs him zero WETH. The loss equals the amount of ETH consumed from the router on Bob's behalf, which can be up to the full stranded balance. This is a direct, unprivileged theft of user funds above Sherlock thresholds.

---

### Likelihood Explanation

**Low-Medium.** The precondition is that native ETH must be stranded on the router. This occurs whenever a user sends `msg.value` in a `multicall` and either (a) omits the `refundETH()` step, (b) the refund step is ordered after a revert, or (c) the swap consumes less ETH than `msg.value`. An attacker can monitor the mempool for such transactions and immediately follow with a WETH-input swap to drain the residue. The router's public `sweepToken` / `refundETH` helpers are also callable by anyone to claim residue, but `pay()` is the path that silently redirects it to a pool on behalf of a different user. [4](#0-3) 

---

### Recommendation

Track the ETH that belongs to the current call by comparing `address(this).balance` before and after, or by only consuming `msg.value` (passed explicitly) rather than the entire contract balance. Concretely, the `pay()` function should receive the caller-authorized native amount as a parameter and cap its native ETH consumption to that amount, rather than reading the unconstrained `address(this).balance`.

---

### Proof of Concept

**Setup:** Pool with `token0 = WETH`, `token1 = TokenA`. Router deployed with that WETH address.

**Step 1 — Alice strands ETH:**
```solidity
// Alice sends 1 ETH but only swaps 0.3 ETH worth of WETH; no refundETH step.
bytes[] memory calls = new bytes[](1);
calls[0] = abi.encodeCall(router.exactInputSingle, (
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        tokenIn: address(weth),
        tokenOut: address(tokenA),
        zeroForOne: true,
        amountIn: 0.3 ether,
        amountOutMinimum: 0,
        recipient: alice,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
));
router.multicall{value: 1 ether}(calls);
// 0.7 ETH now sits on the router, attributable to Alice.
```

**Step 2 — Bob steals Alice's ETH:**
```solidity
// Bob has 0 WETH but calls exactInputSingle for a 0.7 ETH WETH swap.
// pay(WETH, bob, pool, 0.7 ether) sees address(this).balance == 0.7 ETH,
// deposits it as WETH, and transfers it to the pool — Bob pays nothing.
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        tokenIn: address(weth),
        tokenOut: address(tokenA),
        zeroForOne: true,
        amountIn: 0.7 ether,
        amountOutMinimum: 0,
        recipient: bob,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Bob receives TokenA. Alice's 0.7 ETH is gone.
```

**Step 3 — Alice's refund is empty:**
```solidity
router.refundETH(); // Alice receives 0 ETH.
```

The root cause is in `pay()` at lines 74–84 of `PeripheryPayments.sol`: `nativeBalance` is read from the global contract balance with no per-user attribution, so any ETH on the router — regardless of origin — is silently consumed to settle the current caller's WETH obligation. [5](#0-4)

    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {

      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
