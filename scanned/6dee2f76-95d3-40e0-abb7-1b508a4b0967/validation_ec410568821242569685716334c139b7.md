### Title
WETH-native double counting in `PeripheryPayments.pay()` silently drains residual ETH from prior users into a subsequent caller's WETH obligation — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

---

### Summary

`PeripheryPayments.pay()` contains a "partial-ETH + partial-WETH" branch that consumes the router's **entire** native ETH balance (`address(this).balance`) when paying for a WETH swap, regardless of which user deposited that ETH. Any ETH stranded on the router from a prior user's `msg.value` is silently consumed to subsidise the next caller's WETH payment, causing the prior user to lose their ETH with no revert or warning.

---

### Finding Description

`PeripheryPayments.pay()` handles WETH payments with three branches:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {          // ← vulnerable branch
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

The middle branch fires whenever `0 < address(this).balance < value`. It deposits **all** native ETH currently held by the router and transfers it to the pool, then pulls only the remainder from `payer`. `address(this).balance` is the router's **total** ETH balance — it includes ETH left over from any prior transaction that sent `msg.value` without a subsequent `refundETH()` call.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct `transfer`/`send` calls; it does **not** prevent ETH from accumulating via `msg.value` on any of the `payable` entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `unwrapWETH9`, `sweepToken`, `refundETH`, `selfPermit*`). ### Impact Explanation

**Direct loss of user principal (ETH).** A victim user's ETH is silently transferred to a pool on behalf of a different user. The victim receives nothing in return; the beneficiary pays less WETH than they owe. The pool is made whole, so no pool insolvency occurs, but the victim suffers a 1-for-1 ETH loss equal to the stranded balance consumed.

The loss is bounded only by how much ETH the victim left on the router, which can be up to the full `msg.value` of their transaction. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The preconditions are:

1. A user sends `msg.value > amountIn` (or sends ETH with a non-WETH swap) and omits `refundETH()` from their multicall — a realistic mistake, especially for users who copy Uniswap v3 multicall patterns where the router's WETH branch does **not** have this partial-ETH case.
2. A second user calls any WETH swap entry point before the first user's ETH is swept.

Both conditions are reachable by any unprivileged caller with no special setup. The window between transactions is short but non-zero on any chain with a mempool. [4](#0-3) [5](#0-4) 

---

### Recommendation

Remove the partial-ETH branch entirely and align with the Uniswap v3 pattern: use native ETH only when `address(this).balance >= value`; otherwise pull the full amount from `payer` via `safeTransferFrom`. If partial-ETH support is intentional, track only the ETH attributable to the **current** call (e.g., pass `msg.value` through the call stack and cap consumption to that amount).

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        // Do NOT consume residual ETH from prior callers
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0)

### Proof of Concept

**Setup**: Router deployed with WETH address. Alice and Bob are independent EOAs.

**Step 1 — Alice strands ETH on the router:**
```solidity
// Alice calls exactInputSingle with tokenIn=WETH, amountIn=100e18
// but sends msg.value=150e18 (intending to use ETH for the swap)
router.exactInputSingle{value: 150e18}(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 100e18,
    amountOutMinimum: 0,
    recipient: alice,
    deadline: block.timestamp,
    priceLimitX64: 0,
    extensionData: ""
}));
// pay() branch: nativeBalance=150e18 >= value=100e18
// → deposits 100e18 ETH as WETH, transfers to pool
// → 50e18 ETH remains on router
// Alice does NOT call refundETH()
```

**Step 2 — Bob's WETH swap consumes Alice's ETH:**
```solidity
// Bob calls exactInputSingle with tokenIn=WETH, amountIn=100e18, msg.value=0
// Bob has approved 100e18 WETH to the router
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 100e18,
    amountOutMinimum: 0,
    recipient: bob,
    deadline: block.timestamp,
    priceLimitX64: 0,
    extensionData: ""
}));
// pay() branch: nativeBalance=50e18 > 0, nativeBalance < 100e18
// → deposits 50e18 ETH (Alice's) as WETH, transfers to pool
// → pulls only 50e18 WETH from Bob (not 100e18)
// Bob saves 50e18 WETH; Alice loses 50e18 ETH
```

**Result**: Alice loses 50 ETH. Bob's swap succeeds while paying only 50 WETH instead of 100 WETH. The pool receives the correct 100 WETH total. No revert occurs anywhere in the flow. [3](#0-2) [6](#0-5)
