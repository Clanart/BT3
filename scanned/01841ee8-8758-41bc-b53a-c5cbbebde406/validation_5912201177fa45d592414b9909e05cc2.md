### Title
`addLiquidity` and `removeLiquidity` Missing `whenNotPaused` Guard Allows Deposits Into Compromised Paused Pool — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap()` is protected by `whenNotPaused`, but `addLiquidity()` and `removeLiquidity()` are not. When governance pauses a pool at level 1 (admin) or level 2 (protocol security event), users can still deposit principal into the pool via `MetricOmmPoolLiquidityAdder`, and existing LPs can still withdraw. The governance's emergency halt is structurally incomplete: it stops trading but cannot stop new capital from entering a pool that was paused precisely because it is unsafe.

---

### Finding Description

`MetricOmmPool` defines a `whenNotPaused` modifier and a three-level pause system: [1](#0-0) 

The modifier is applied only to `swap()`: [2](#0-1) 

`addLiquidity` and `removeLiquidity` carry only the reentrancy guard — no pause check: [3](#0-2) [4](#0-3) 

`_checkNotPaused()` treats any non-zero pause level as a full halt: [5](#0-4) 

The periphery `MetricOmmPoolLiquidityAdder` adds no compensating pause check before calling `addLiquidity`: [6](#0-5) 

The `addLiquidityWeighted` path compounds this: it reads the live pool cursor via `PoolStateLibrary._slot0` to validate bin bounds, then calls `addLiquidity` twice (probe + pay). Both calls reach the pool without any pause gate: [7](#0-6) 

The protocol documentation explicitly states that level 2 is the "strongest halt (e.g. security event)": [8](#0-7) 

---

### Impact Explanation

**Severity: Medium**

When a pool is paused at level 2 because a vulnerability has been discovered (e.g., incorrect bin accounting, oracle staleness, or an extension bug), the protocol's intent is to freeze all pool activity. However:

1. Any user can call `addLiquidityExactShares` or `addLiquidityWeighted` on `MetricOmmPoolLiquidityAdder`, which calls `pool.addLiquidity()` without obstruction. The user's tokens are transferred into the compromised pool.
2. When the pool is eventually unpaused, the newly deposited principal is exposed to whatever condition caused the pause.
3. The governance has no on-chain mechanism to prevent this: `protocolPausePool` sets level 2, but `addLiquidity` ignores it.

The fund loss is conditional on the pool having a concurrent vulnerability, but the missing guard is what allows users to deposit into the unsafe state — directly analogous to the LensHub finding where pausing swaps did not prevent NFT transfers from continuing during an emergency.

---

### Likelihood Explanation

- Any unprivileged user or bot can call `addLiquidityExactShares` at any time, including while the pool is paused.
- Bots and integrations that monitor pools for liquidity opportunities will not inspect `pauseLevel` before depositing.
- The `addLiquidityWeighted` path is particularly dangerous because it performs a probe call that succeeds (pool state is unchanged by the probe revert), then executes a paying deposit — all while the pool is paused.

---

### Recommendation

Add `whenNotPaused` to both `addLiquidity` and `removeLiquidity` in `MetricOmmPool.sol`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

If the protocol intentionally allows LP exits during a pause (emergency withdrawal), a separate pause level or a dedicated flag (e.g., `pauseLevel >= 2` blocks both directions, `pauseLevel == 1` blocks only swaps and new deposits) should be introduced and documented explicitly. The current code provides no such distinction.

---

### Proof of Concept

```
State: pool.pauseLevel == 2  (protocol security pause)

1. Attacker/user calls:
   MetricOmmPoolLiquidityAdder.addLiquidityExactShares(
       pool, owner, salt, deltas, maxAmount0, maxAmount1, ""
   )

2. LiquidityAdder._addLiquidity() sets transient pay context, then calls:
   pool.addLiquidity(owner, salt, deltas, abi.encode(KIND_PAY), "")

3. pool.addLiquidity() has no whenNotPaused check.
   Execution proceeds normally; pool.metricOmmModifyLiquidityCallback fires,
   LiquidityAdder pays token0/token1 from user into the pool.

4. User's tokens are now locked in a pool paused for a security reason.
   pool.swap() reverts with PoolPaused() for any trader.
   pool.addLiquidity() does NOT revert — deposit succeeds.

5. When pool is unpaused, user's principal is exposed to the
   vulnerability that triggered the level-2 pause.
```

The `addLiquidityWeighted` variant is identical but executes two pool calls (probe + pay), both of which bypass the pause check.
