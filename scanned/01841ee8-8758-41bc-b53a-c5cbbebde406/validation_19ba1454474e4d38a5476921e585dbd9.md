### Title
`addLiquidity` Callable on Paused Pool, Exposing LP Deposits to Compromised State — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap` is protected by the `whenNotPaused` modifier, but `MetricOmmPool.addLiquidity` is not. When a pool is paused (at level 1 or 2) due to an emergency or discovered vulnerability, any caller — including the periphery `MetricOmmPoolLiquidityAdder` — can still deposit LP funds into the paused pool. Those tokens are immediately pulled from the LP via the `metricOmmModifyLiquidityCallback` and locked inside a pool whose swap path is known to be unsafe.

---

### Finding Description

`MetricOmmPool` exposes a three-level pause system (`pauseLevel` 0/1/2) enforced by the `whenNotPaused` modifier and `_checkNotPaused()`:

```solidity
// MetricOmmPool.sol line 174-177
modifier whenNotPaused() {
    _checkNotPaused();
    _;
}

// line 643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [1](#0-0) [2](#0-1) 

`swap` correctly applies this guard:

```solidity
// MetricOmmPool.sol line 224
) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
``` [3](#0-2) 

But `addLiquidity` does not:

```solidity
// MetricOmmPool.sol line 188
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
``` [4](#0-3) 

`addLiquidity` calls `LiquidityLib.addLiquidity`, which triggers `metricOmmModifyLiquidityCallback` on the caller. The periphery `MetricOmmPoolLiquidityAdder` implements this callback and immediately pulls real tokens from the LP's wallet into the pool:

```solidity

if (amount0Delta > 0) {
    pay(token0, payer, msg.sender, amount0Delta);
}
if (amount1Delta > 0) {
    pay(token1, payer, msg.sender, amount1Delta);
}
``` [5](#0-4) 

Neither `addLiquidityExactShares` nor `addLiquidityWeighted` in the periphery checks the pool's pause state before initiating the deposit: [6](#0-5) [7](#0-6) 

The factory's pause transitions confirm that pausing is an emergency action:

```solidity
// MetricOmmPoolFactory.sol line 392-396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
``` [8](#0-7) 

---

### Impact Explanation

When a pool is paused because a vulnerability has been discovered (e.g., oracle manipulation, price feed staleness, bin accounting error), swaps are correctly blocked. However, any LP who calls `addLiquidityExactShares` or `addLiquidityWeighted` through `MetricOmmPoolLiquidityAdder` will have their `token0`/`token1` pulled from their wallet and deposited into the compromised pool. Those funds are then exposed to the same vulnerability that triggered the pause, resulting in direct loss of LP principal. The pool's bin accounting and `binTotals` are updated, so the deposit is fully committed with no rollback path for the LP.

---

### Likelihood Explanation

A pool pause is an emergency event. During the window between the pause transaction and the resolution of the underlying issue, any LP who has pre-approved `MetricOmmPoolLiquidityAdder` and submits a pending `addLiquidity` transaction (e.g., from a bot or queued multicall) will have their funds deposited into the paused pool. This is a realistic scenario: MEV bots and liquidity management scripts often queue deposits independently of pool state. No privileged access is required by the attacker; the LP is the victim of their own valid transaction executing at the wrong time.

---

### Recommendation

Add the `whenNotPaused` modifier to `addLiquidity` in `MetricOmmPool.sol`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` should intentionally remain unguarded so LPs can always exit, mirroring the design intent of the Notional Finance discussion (allow exits, block new entries).

---

### Proof of Concept

1. Pool is deployed with `pauseLevel = 0` (active).
2. A vulnerability is discovered; pool admin calls `factory.pausePool(pool)` → `pauseLevel = 1`.
3. `pool.swap(...)` now reverts with `PoolPaused`.
4. LP calls `liquidityAdder.addLiquidityExactShares(pool, owner, salt, deltas, max0, max1, "")`.
5. `MetricOmmPoolLiquidityAdder._addLiquidity` calls `pool.addLiquidity(...)` — **no pause check** — which succeeds.
6. `metricOmmModifyLiquidityCallback` fires; `pay(token0, payer, pool, amount0Delta)` and `pay(token1, payer, pool, amount1Delta)` pull real tokens from the LP into the paused pool.
7. LP's tokens are now locked in a pool whose swap path is known to be unsafe, with no mechanism to prevent the deposit from having occurred.
