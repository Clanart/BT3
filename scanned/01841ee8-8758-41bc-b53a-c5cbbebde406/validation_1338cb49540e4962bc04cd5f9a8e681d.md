The `pay` function in `PeripheryPayments.sol` is fully visible. Let me verify the `exactInputSingle` signature and whether it's payable, and also check how ETH gets into the router during a swap.

The vulnerability is real and provable. Here is the complete analysis:

---

### Title
Router `pay()` Consumes Residual ETH From Prior Transactions to Subsidize Subsequent WETH Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **total** native ETH balance — when deciding how much ETH to wrap for a WETH payment. It has no mechanism to distinguish ETH sent in the current transaction from ETH stranded by a prior caller. Any user who sends excess ETH to the router without calling `refundETH()` leaves funds that a subsequent WETH-swap caller can silently consume.

---

### Finding Description

The WETH branch of `pay()` reads the router's full native balance: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total router ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

ETH can be stranded in the router because:

1. `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput` are all `external payable`. [2](#0-1) 

2. `multicall` is `payable` and the protocol explicitly documents the ETH-input pattern as `multicall{value}(exactInput*)`. [3](#0-2) 

3. The `receive()` guard only blocks direct ETH pushes; it does **not** prevent ETH from being left over after a `payable` swap call that consumed less than `msg.value`. [4](#0-3) 

**Stranding path (User A):**
- User A calls `exactInputSingle{value: 1 ether}(tokenIn=WETH, amountIn=0.5 ether, ...)`.
- The callback calls `pay(WETH, userA, pool, 0.5 ether)`. `nativeBalance = 1 ETH ≥ 0.5 ETH`, so 0.5 ETH is wrapped and sent to the pool.
- The remaining 0.5 ETH stays in the router. No `refundETH()` was called; it is permanently stranded until someone claims it.

**Theft path (User B):**
- User B calls `exactInputSingle{value: 0.5 ether}(tokenIn=WETH, amountIn=1 ether, ...)`.
- The callback calls `pay(WETH, userB, pool, 1 ether)`.
- `nativeBalance = address(this).balance = 0.5 ETH (A's residual) + 0.5 ETH (B's msg.value) = 1 ETH`.
- `nativeBalance >= value` → the router wraps all 1 ETH and transfers it to the pool.
- User B's swap is fully settled using 0.5 ETH of User A's funds. User B never pays from their WETH allowance. User A's 0.5 ETH is gone.

---

### Impact Explanation

Direct, irreversible loss of native ETH belonging to a prior caller. The attacker receives a full swap output while paying only half the required input. The victim (User A) loses their stranded ETH with no recourse. This meets the High threshold: direct loss of user principal through an unprivileged public call.

---

### Likelihood Explanation

- The ETH-input multicall pattern is the **documented** and **tested** usage pattern for WETH swaps. [5](#0-4) 
- Users who send slightly more ETH than needed (common for slippage headroom) and omit `refundETH()` will strand funds. The test suite itself demonstrates this exact scenario. [6](#0-5) 
- An attacker only needs to watch the router's ETH balance on-chain and front-run or follow any transaction that leaves residual ETH.

---

### Recommendation

Track only the ETH attributable to the **current** transaction. The standard fix (used by Uniswap v3 periphery) is to snapshot `address(this).balance` **before** the swap entry and pass that snapshot into `pay()` as the available native amount, rather than re-reading the live balance inside the callback. Alternatively, enforce that `address(this).balance` equals `msg.value` at the start of every payable entry point (revert if the router already holds ETH), which prevents cross-transaction contamination entirely.

function test_residualEth_stolenBySubsequentWethSwap() public {
    uint128 amountIn = 1 ether;

        tokenIn: address(weth),
        tokenOut: address(token1),

        amountOutMinimum: 0,

        priceLimitX64: 0,
        extensionData: ""
    }));
    // 0.5 ETH is now stranded in the router
    assertEq(address(router).balance, 0.5 ether, "A's ETH stranded");

    // --- User B exploits: sends only 0.5 ETH but requests 1 ETH WETH swap ---
    vm.deal(userB, 0.5 ether);
    uint256 userBWethBefore = weth.balanceOf(userB); // userB has WETH allowance set
    vm.prank(userB);
    router.exactInputSingle{value: 0.5 ether}(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1 ether,   // needs 1 ETH total
        amountOutMinimum: 0,
        recipient: userB,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));

    // Router used A's 0.5 ETH + B's 0.5 ETH = 1 ETH for B's swap
    assertEq(address(router).balance, 0, "router drained");
    // B's WETH allowance was NOT consumed (router used native ETH instead)
    assertEq(weth.balanceOf(userB), userBWethBefore, "B paid nothing from WETH");
    // A's 0.5 ETH is gone
}
