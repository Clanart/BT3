### Title
`addLiquidityWeighted` reverts with `SharesRoundedToZero` when scaled shares truncate to zero, making liquidity addition unusable for small depositors â€” (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

---

### Summary

The `_scaleWeightsToShares` helper inside `MetricOmmPoolLiquidityAdder` unconditionally reverts with `SharesRoundedToZero` whenever a user's `maxAmountToken0`/`maxAmountToken1` cap is small enough relative to the pool's probed token needs that integer division truncates any bin's scaled share count to zero. This is a direct analog of the Napier M-8 class: a non-zero user-supplied amount produces a zero intermediate quantity, and the code reverts instead of gracefully handling the edge case, making `addLiquidityWeighted` permanently unusable for small depositors.

---

### Finding Description

`addLiquidityWeighted` executes a two-phase flow:

**Phase 1 â€“ Probe:** calls `pool.addLiquidity` with `KIND_PROBE`; the callback always reverts with `LiquidityProbe(need0, need1)`, reporting how many raw token units the weight vector would consume at current pool state.

**Phase 2 â€“ Scale + Pay:** `_scaleWeightsToShares` computes a single WAD-scaled multiplier and applies it to every bin's weight to produce integer share counts, then calls `_addLiquidity` with those shares.

The scaling logic is:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol  lines 231-241
uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;
...
for (uint256 i; i < n; i++) {
    out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
    if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
}
``` [1](#0-0) 

When `max0 â‰Ş need0` (or `max1 â‰Ş need1`), `scaleWad` becomes much smaller than `WAD`. For any bin whose weight `w.shares[i]` satisfies `w.shares[i] * scaleWad < WAD`, `Math.mulDiv` truncates to zero and the function reverts â€” even though the user supplied a strictly positive `maxAmount` and a strictly positive weight.

**Concrete numeric example:**

| Variable | Value |
|---|---|
| `maxAmountToken0` | `1e6` (1 USDC, 6-decimal) |
| `need0` (from probe) | `1e12` (1 M USDC) |
| `scaleWad0` | `Math.mulDiv(1e6, 1e18, 1e12)` = `1e12` |
| `scaleWad` | `1e12` (assuming token1 leg is unconstrained) |
| `w.shares[i]` | `1` (minimum non-zero weight) |
| `out.shares[i]` | `Math.mulDiv(1, 1e12, 1e18)` = **0** â†’ `SharesRoundedToZero` revert |

The user provided a non-zero amount and a non-zero weight; the transaction still reverts. There is no fallback path: `addLiquidityExactShares` requires the caller to know the exact share count in advance, which defeats the purpose of the weighted helper.

The revert also fires when only *one* bin in a multi-bin delta truncates to zero, blocking liquidity addition to all other bins that would have produced non-zero shares.

---

### Impact Explanation

Any user whose token budget is below the per-share threshold of the current pool state cannot add liquidity via `addLiquidityWeighted`. Because the threshold is determined dynamically by the probe (it depends on live pool depth and bin cursor position), users cannot predict in advance whether their call will revert. The core weighted-liquidity-addition flow is rendered unusable for small depositors and for any depositor during periods of high pool depth, satisfying the "unusable liquidity flows" impact criterion.

---

### Likelihood Explanation

The condition is triggered whenever `maxAmount / need < 1 / w.shares[i]`. In a pool with significant existing liquidity, `need` values are large, making the threshold high. Any retail depositor providing amounts that are a small fraction of total pool depth will hit this revert. The condition is unprivileged (any caller), requires no malicious setup, and is reachable on every call to `addLiquidityWeighted`.

---

### Recommendation

Mirror the Napier fix: instead of reverting when a bin's scaled share truncates to zero, either (a) skip that bin silently (emit no shares for it), or (b) clamp `out.shares[i]` to `1` when `w.shares[i] != 0` and `scaleWad > 0`. A minimum-threshold guard analogous to the Napier recommendation is also acceptable:

```solidity
// Option A â€“ skip zero-share bins
if (w.shares[i] != 0 && out.shares[i] == 0) {
    out.binIdxs[i] = w.binIdxs[i];
    out.shares[i] = 0; // pool will ignore zero-share entries
    continue;
}

// Option B â€“ clamp to 1
if (w.shares[i] != 0 && out.shares[i] == 0) {
    out.shares[i] = 1;
}
```

If the pool rejects zero-share entries, Option A requires filtering them out before the paying `addLiquidity` call.

### Proof of Concept

1. Deploy a pool with significant token0 depth so that a unit-weight probe returns `need0 = 1e12`.
2. Call `addLiquidityWeighted` with `maxAmountToken0 = 1e6`, `maxAmountToken1 = type(uint256).max`, and a single-bin `weightDeltas` with `shares[0] = 1`.
3. Probe succeeds, returning `need0 = 1e12`, `need1 = 0`.
4. `scaleWad0 = Math.mulDiv(1e6, 1e18, 1e12) = 1e12`; `scaleWad1 = type(uint256).max`; `scaleWad = 1e12`.
5. `out.shares[0] = Math.mulDiv(1, 1e12, 1e18) = 0`.
6. Guard fires: `revert SharesRoundedToZero()`.
7. User's transaction reverts despite supplying a non-zero token budget and a non-zero weight â€” identical failure mode to Napier M-8. [2](#0-1) [3](#0-2)

    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
