### Title
Pre-existing ETH Balance in Router Used to Pay Attacker's WETH Swap for Free — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` function in `PeripheryPayments.sol` uses `address(this).balance` to determine how much native ETH to contribute toward a WETH payment. Because this reads the router's total ETH balance rather than only the current call's `msg.value`, any ETH left in the router from a prior user's transaction can be silently consumed to pay for a subsequent caller's WETH swap — without pulling any WETH from that caller.

---

### Finding Description

`PeripheryPayments.pay()` handles WETH payments as follows:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← reads total contract balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

When `nativeBalance >= value`, the function wraps ETH from the contract's balance and transfers it to the pool — **without pulling any WETH from `payer`**. The `payer` pays nothing.

ETH accumulates in the router whenever a user calls a `payable` entry point (e.g., `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) with `msg.value` exceeding the actual swap amount and does not subsequently call `refundETH()`. The `receive()` guard only blocks arbitrary external ETH deposits:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

But ETH sent as `msg.value` by legitimate users accumulates freely and is never automatically refunded.

---

### Impact Explanation

An attacker observing leftover ETH in the router can call `exactInputSingle` (or any WETH-input swap) with `msg.value = 0` and `tokenIn = WETH`. The `pay()` function will use the router's pre-existing ETH balance to wrap and deliver WETH to the pool on the attacker's behalf. The attacker receives the output tokens without spending any ETH or WETH. The victim (the user who left ETH in the router) permanently loses their ETH — it is consumed to fund the attacker's trade.

This is a **direct loss of user principal** with no recovery path once the attacker's transaction executes.

---

### Likelihood Explanation

- Users who call `exactInputSingle` or `exactInput` with `msg.value > amountIn` and omit a trailing `refundETH()` call leave ETH stranded in the router. This is a common mistake in Uniswap-style routers.
- The attacker requires no special permissions, no malicious setup, and no privileged role — only the ability to observe the router's ETH balance (trivially readable on-chain) and submit a WETH swap.
- The `multicall` function is `payable` and delegates calls, making it easy to batch a WETH swap without a refund step. [3](#0-2) 

---

### Recommendation

Track only the ETH contributed by the current transaction rather than the total contract balance. Replace the `address(this).balance` read in `pay()` with a parameter representing the current call's `msg.value`, passed down from the entry-point function. Alternatively, record `msg.value` at entry and subtract any already-consumed amount before each `pay()` call, so leftover ETH from prior transactions is never silently applied to a new payer's obligation.

---

### Proof of Concept

1. **Victim setup**: User A calls `exactInputSingle(tokenIn=WETH, amountIn=0.5 ether)` with `msg.value = 1 ether`. The swap executes; `pay()` reads `address(this).balance = 1 ether`, wraps 0.5 ETH, and transfers WETH to the pool. User A receives output tokens. User A does **not** call `refundETH()`. The router now holds **0.5 ETH**.

2. **Attacker exploit**: Attacker calls `exactInputSingle(tokenIn=WETH, amountIn=0.5 ether, msg.value=0)`.
   - Pool calls `metricOmmSwapCallback`.
   - `_justPayCallback` calls `pay(WETH, attacker, pool, 0.5 ether)`.
   - `nativeBalance = address(this).balance = 0.5 ether`.
   - Branch `nativeBalance >= value` is taken: wraps 0.5 ETH from the router's balance, transfers WETH to pool.
   - **No `safeTransferFrom` is called on the attacker.**
   - Attacker receives output tokens worth 0.5 ETH at zero cost.

3. **Result**: User A's 0.5 ETH is permanently lost; attacker profits by 0.5 ETH worth of output tokens. [4](#0-3) [5](#0-4) [6](#0-5)
