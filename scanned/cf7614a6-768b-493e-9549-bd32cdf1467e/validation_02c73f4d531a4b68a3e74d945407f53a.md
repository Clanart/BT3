### Title
Router ETH Balance Consumed for Any Caller's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses `address(this).balance` — the router's **entire** native ETH balance — to fund a WETH payment whenever `payer != address(this)`. ETH stranded on the router from a prior `multicall{value: X}` call (where X exceeds the amount consumed) is silently consumed to fund a completely unrelated caller's WETH swap, bypassing that caller's WETH allowance and stealing the original depositor's ETH.

### Finding Description

`exactInputSingle` stores `msg.sender` as the payer in transient context, then the pool calls back into `metricOmmSwapCallback` → `_justPayCallback`: [1](#0-0) 

`_justPayCallback` calls `pay(WETH, msg.sender, pool, value)`. Inside `pay`, the WETH branch reads the router's **global** ETH balance: [2](#0-1) 

The three branches are:
- `nativeBalance >= value` → wraps router ETH, transfers WETH to pool. **Payer's allowance is never touched.**
- `0 < nativeBalance < value` → wraps partial router ETH, pulls only the remainder from payer.
- `nativeBalance == 0` → pulls entirely from payer (correct path).

ETH can be stranded on the router because `multicall` is `payable`: [3](#0-2) 

The `receive()` guard only blocks direct ETH sends from non-WETH addresses: [4](#0-3) 

It does **not** block ETH attached to a `multicall{value: X}` call. If a user calls `multicall{value: 1 ether}([exactInputSingle(WETH, amountIn=0.5 ether, ...)])` without a trailing `refundETH()`, 0.5 ETH is permanently stranded on the router until consumed.

### Impact Explanation

Any subsequent caller who calls `exactInputSingle` (or any WETH-input swap) while the router holds ETH will have their swap funded by the stranded ETH instead of their own WETH allowance. The original depositor loses their ETH; the exploiter receives the swap output without spending any WETH. This is a direct, cross-user principal loss above Sherlock thresholds for any non-trivial stranded amount.

### Likelihood Explanation

The intended usage pattern (send ETH with multicall, swap WETH, call `refundETH`) is not enforced by the contract. Any user who omits `refundETH()` or sends slightly more ETH than consumed strands value. An attacker can monitor the router's ETH balance on-chain and immediately call `exactInputSingle(WETH, amountIn=stranded_amount, ...)` to claim it. No privileged access is required.

### Recommendation

In the WETH branch of `pay`, only use `msg.value` (the ETH attached to the **current** top-level call) rather than `address(this).balance`. Since `pay` is called inside a callback (not a top-level call), the correct approach is to track the ETH that was intentionally deposited for the current swap in transient storage at the `exactInputSingle` / `exactInput` entry point, and consume only that tracked amount in `pay`. Alternatively, require callers to pre-wrap ETH to WETH before calling the router and remove the native-ETH shortcut from `pay` entirely.
