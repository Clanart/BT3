Looking at the Metric OMM periphery codebase, I need to find an analog to the staking rewards theft bug — where a shared balance is used to calculate a user's entitlement, allowing later users to consume earlier users' funds.

The invariant from the external report: **a contract uses a shared total balance (not per-user accounting) to settle a user's claim, so a later actor can drain an earlier actor's deposited value.**

Let me trace the `pay` function in `PeripheryPayments.sol`: [1](#0-0) 

The WETH branch reads `address(this).balance` — the router's **total** native ETH balance, which is shared across all users and all transactions: [2](#0-1) 

This is the root cause. Any ETH left in the router from a prior user's `msg.value` is silently consumed to pay the next WETH swapper's pool debt.

### Title
Shared `address(this).balance` in `pay()` allows any WETH swapper to consume prior users' leftover ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
The `pay` function settles WETH swap debts by first checking `address(this).balance`. Because this balance is global to the router contract and not per-user, any ETH left behind by a previous caller (who sent excess `msg.value` and did not call `refundETH`) is silently spent to cover the next WETH swapper's pool payment, causing a direct loss of the earlier user's ETH principal.

### Finding Description

When a user calls `exactInputSingle` or `exactInput` with `msg.value > 0` and `tokenIn = WETH`, the pool's swap callback fires `_justPayCallback`, which calls:

```solidity
pay(_getTokenToPay(), _getPayer(), msg.sender, uint256(...amountOwed...));
```

Inside `pay`, the WETH branch is:

```solidity
uint256 nativeBalance = address(this).balance;   // ← GLOBAL router balance
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [2](#0-1) 

`nativeBalance` is not scoped to the current caller's `msg.value`. It is the entire router's ETH balance. If Alice sent 1 ETH with her swap but only 0.5 ETH was consumed, 0.5 ETH remains in the router. When Bob subsequently calls any WETH swap costing ≤ 0.5 ETH, the router enters the `nativeBalance >= value` branch, deposits Alice's ETH as WETH, and transfers it to the pool — without pulling a single wei from Bob.

The `receive()` guard:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [3](#0-2) 

only blocks direct ETH pushes; it does **not** prevent ETH accumulation via `msg.value` in `payable` function calls (`exactInputSingle`, `exactOutput`, `multicall`, etc.). There is no per-user ETH accounting and no automatic refund after each swap.

The `refundETH` function exists but is opt-in:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
``` [4](#0-3) 

If Alice does not atomically batch `refundETH` in the same `multicall`, her leftover ETH is exposed.

### Impact Explanation

Direct loss of user ETH principal. Alice's unrefunded ETH is consumed to settle Bob's pool debt. Bob pays zero for his swap; Alice's ETH is permanently transferred to the pool on Bob's behalf. The loss equals the unrefunded ETH balance at the time Bob's callback fires. There is no cap — if Alice left 10 ETH in the router, Bob's single WETH swap can drain all 10 ETH.

### Likelihood Explanation

Medium. The trigger requires:
1. A user sends excess `msg.value` with a WETH swap (common when slippage or price movement means less ETH is consumed than sent).
2. That user does not atomically call `refundETH()` in the same `multicall` — either by omission, by using a single-call path, or by being front-run between their swap and a separate `refundETH` call.

Both conditions are realistic in production. Front-running the `refundETH` call is straightforward for a mempool-watching attacker.

### Recommendation

Automatically refund excess ETH at the end of each top-level swap entry point (after `_clearExpectedCallbackPool()`), or track each caller's contributed `msg.value` in transient storage and only allow that amount to be spent via `pay` within the same transaction. The `multicall` pattern alone is insufficient because it relies on users knowing to append `refundETH`.

### Proof of Concept

```
State: router ETH balance = 0

1. Alice calls exactInputSingle{value: 1 ETH}(params)
   where params.tokenIn = WETH, actual swap cost = 0.5 ETH.

2. Pool fires metricOmmSwapCallback → _justPayCallback →
   pay(WETH, Alice, pool, 0.5 ETH)
     nativeBalance = 1 ETH >= 0.5 ETH
     → deposit 0.5 ETH, transfer 0.5 WETH to pool  ✓
   Router ETH balance = 0.5 ETH (Alice's unspent ETH)

3. Alice does NOT call refundETH() (or is front-run).

4. Bob calls exactInputSingle{value: 0}(params)
   where params.tokenIn = WETH, swap cost = 0.5 ETH.

5. Pool fires metricOmmSwapCallback → _justPayCallback →
   pay(WETH, Bob, pool, 0.5 ETH)
     nativeBalance = 0.5 ETH >= 0.5 ETH          ← Alice's ETH
     → deposit 0.5 ETH, transfer 0.5 WETH to pool
     safeTransferFrom(Bob, ...) is NEVER called

Result:
  Alice lost 0.5 ETH (total loss: 1 ETH sent, 0 ETH returned)
  Bob paid 0 ETH for a 0.5 ETH swap
  Pool received correct WETH in both cases (no pool loss)
```

The structural analog to the staking bug: just as `stakedBalance` records raw deposit amounts while `totalTokenBalance` silently grows with yield — letting Bob's unstake claim Alice's accrued value — here `address(this).balance` silently accumulates Alice's unrefunded ETH, and Bob's `pay` call claims it without any per-user accounting.
