### Title
`PeripheryPayments.pay` Transfers from Router's Own WETH Balance Instead of Pulling from Payer When No ETH Is Sent — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay` helper in `PeripheryPayments` has a wrong-source payment bug in its WETH branch. When the payer is an external user, the token is WETH, and no native ETH is held by the router, the code calls `IERC20(WETH).safeTransfer(recipient, value)` — transferring from the **router's own WETH balance** — instead of `safeTransferFrom(payer, recipient, value)` — pulling from the **user's WETH balance**. This is the direct analog of M-08: the wrong entity is operated on.

---

### Finding Description

In `PeripheryPayments.pay`, the WETH branch handles three sub-cases based on the router's native ETH balance: [1](#0-0) 

```
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);          // wrap ETH → send WETH ✓
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // partial ✓
    } else {
        IERC20(WETH).safeTransfer(recipient, value);          // ← BUG: pulls from router, not payer
    }
}
```

The third branch — reached when `payer != address(this)` (i.e., the user is the payer) and `nativeBalance == 0` — should call `safeTransferFrom(payer, recipient, value)` to pull WETH from the user who approved the router. Instead it calls `safeTransfer(recipient, value)`, which attempts to send WETH **from the router's own balance**.

This function is called from `_justPayCallback` (used by `exactInputSingle` and `exactInput`) and from `_exactOutputIterateCallback` at `tradesLeft == 0` (used by `exactOutput`): [2](#0-1) [3](#0-2) 

The `tokenToPay` is set to the user's input token at swap entry: [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Scenario A — DoS (primary impact):** A user holds WETH, approves the router, and calls `exactInputSingle` / `exactInput` / `exactOutput` with WETH as the input token and `msg.value = 0`. The callback fires, `pay` enters the WETH branch with `nativeBalance == 0`, and calls `safeTransfer(pool, value)`. The router holds no WETH, so the transfer reverts. The swap fails entirely. **All WETH-input swaps without accompanying ETH are permanently broken through this router.**

**Scenario B — Router WETH drain (secondary impact):** If the router holds residual WETH (e.g., from a prior multicall step that did not sweep, or from a partial-ETH wrap that left dust), the `safeTransfer` succeeds and the router's WETH is consumed to pay the pool instead of the user's WETH. The user's WETH is never pulled; the pool is paid from the router's balance. The user receives output tokens without spending their own WETH.

Both outcomes break the core swap invariant: the pool must receive the correct input amount from the correct payer.

---

### Likelihood Explanation

WETH is a primary input token for any ETH-denominated pool. Users who hold WETH and approve the router (the standard ERC-20 flow, not the ETH-wrapping flow) will hit this on every call. No special setup, privileged role, or non-standard token is required. Any unprivileged user calling `exactInputSingle(WETH, …)` with `msg.value = 0` triggers it.

---

### Recommendation

Replace the final `else` branch in `PeripheryPayments.pay` with a `safeTransferFrom` call:

```solidity
} else {
    // No native ETH: pull WETH directly from payer
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [7](#0-6) 

---

### Proof of Concept

1. Deploy the router with a valid WETH address and factory.
2. Mint WETH to Alice; Alice calls `WETH.approve(router, type(uint256).max)`.
3. Alice calls `router.exactInputSingle({tokenIn: WETH, amountIn: 1e18, …})` with `msg.value = 0`.
4. The pool executes the swap and calls `metricOmmSwapCallback`.
5. `_justPayCallback` calls `pay(WETH, Alice, pool, 1e18)`.
6. `payer != address(this)` → enters WETH branch; `address(this).balance == 0` → enters final `else`.
7. `IERC20(WETH).safeTransfer(pool, 1e18)` is called; router has 0 WETH → **reverts**.
8. The entire swap reverts. Alice cannot swap WETH through this router without sending ETH.
