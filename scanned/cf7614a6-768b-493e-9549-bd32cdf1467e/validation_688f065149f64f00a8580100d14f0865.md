### Title
Leftover ETH from Prior User Consumed by Subsequent WETH Swap via Partial-ETH Branch in `PeripheryPayments.pay()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` contains a partial-ETH branch that blindly uses the router's entire native ETH balance — including ETH left by prior users who did not call `refundETH()` — to subsidize a subsequent caller's WETH deposit. This allows any attacker to drain stranded ETH from the router at the expense of the original depositor.

---

### Finding Description

`PeripheryPayments.pay()` handles the `tokenIn == WETH` case with three sub-branches based on `address(this).balance` vs `value`: [1](#0-0) 

The partial-ETH branch (lines 78–81) fires when `0 < nativeBalance < value`:

```solidity
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```

It deposits **all** of `address(this).balance` — with no check that this ETH belongs to the current payer — and reduces the `safeTransferFrom` pull accordingly.

**How ETH becomes stranded:** `multicall` and all swap entry points are `payable`: [2](#0-1) [3](#0-2) 

The `receive()` guard only blocks *direct* ETH transfers from non-WETH addresses: [4](#0-3) 

It does **not** block ETH arriving via `payable` function calls. When User A calls `exactInputSingle` with `tokenIn=WETH`, `amountIn=X`, and sends `msg.value=X+Y`, the full-ETH branch deposits exactly `X`: [5](#0-4) 

The remaining `Y` ETH stays on the router. If User A omits `refundETH()` from their multicall, this `Y` is permanently stranded until the next WETH swap.

**Attack path:**

1. User A calls `exactInputSingle` (`tokenIn=WETH`, `amountIn=100`) with `msg.value=150`. `pay()` deposits 100 ETH, 50 ETH remains on router.
2. Attacker B calls `exactInputSingle` (`tokenIn=WETH`, `amountIn=80`). Now `nativeBalance=50`, `value=80` → partial branch fires.
3. Router deposits User A's 50 ETH, transfers 50 WETH to pool, pulls only 30 WETH from attacker B via `safeTransferFrom`.
4. User A's 50 ETH is permanently consumed; attacker B's cost is reduced by 50 ETH.

The payer stored in transient context is correctly set to `msg.sender` (attacker B): [6](#0-5) 

But `pay()` never validates that `address(this).balance` was contributed by that payer — it uses it unconditionally.

---

### Impact Explanation

Direct loss of User A's ETH principal held on the router. The attacker receives a discount equal to the stranded ETH balance; User A's funds are irrecoverably consumed. This meets the Critical/High threshold of direct loss of user principal.

---

### Likelihood Explanation

Moderate. The `payable` multicall pattern actively encourages users to send ETH for WETH swaps. Forgetting `refundETH()` is a common mistake in Uniswap v3-style routers. An attacker can monitor the mempool or router balance for stranded ETH and immediately exploit it with a crafted `exactInputSingle` call.

---

### Recommendation

In the partial-ETH branch, only use ETH that was explicitly sent by the current caller in the current transaction (i.e., `msg.value` tracked per-call), not the router's entire `address(this).balance`. One approach: track consumed ETH in transient storage and only allow `pay()` to spend up to the current call's `msg.value`. Alternatively, remove the partial-ETH branch entirely and require callers to either send the full amount as ETH or use pure ERC20 `transferFrom`, relying on `refundETH()` for any excess.

### Proof of Concept

```solidity
// Foundry integration test sketch
function test_strandedEthConsumedByAttacker() public {
    // User A: sends 150 ETH, swaps 100 WETH, forgets refundETH()
    vm.deal(userA, 150 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 150 ether}(ExactInputSingleParams({
        tokenIn: WETH, amountIn: 100 ether, ...
    }));
    assertEq(address(router).balance, 50 ether); // 50 ETH stranded

    // Attacker B: swaps 80 WETH, only needs to provide 30 WETH from wallet
    uint256 attackerBalanceBefore = IERC20(WETH).balanceOf(attackerB);
    vm.prank(attackerB);
    router.exactInputSingle(ExactInputSingleParams({
        tokenIn: WETH, amountIn: 80 ether, ...
    }));
    // Attacker B only spent 30 WETH from their own balance
    assertEq(attackerBalanceBefore - IERC20(WETH).balanceOf(attackerB), 30 ether);
    // Router ETH balance is now 0 — User A's 50 ETH is gone
    assertEq(address(router).balance, 0);
}
```
