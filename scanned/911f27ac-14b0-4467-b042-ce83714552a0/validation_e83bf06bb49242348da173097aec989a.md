### Title
`addLiquidityExactShares` lacks deadline protection, enabling sandwich attacks on liquidity deposits — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

### Summary

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts no `deadline` parameter and performs no deadline check before executing the liquidity deposit. The swap router (`MetricOmmSimpleRouter`) correctly calls `_checkDeadline(params.deadline)` on every entry point, but the liquidity adder — which inherits only `PeripheryPayments`, not `MetricOmmSwapRouterBase` — has no equivalent guard. A pending `addLiquidityExactShares` transaction can therefore be held in the mempool and executed arbitrarily late, after an attacker has manipulated the pool's active-bin cursor.

### Finding Description

`addLiquidityExactShares` (both overloads) accepts `maxAmountToken0` / `maxAmountToken1` as token-amount caps but no `deadline`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol  L56-L68
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
}
``` [1](#0-0) 

Contrast with the router, which always checks the deadline before touching any pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol  L67-L68
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
``` [2](#0-1) 

In a bin-based AMM the token composition of each bin is determined by the pool's active-bin cursor at the moment of deposit. `addLiquidityExactShares` specifies exact `binIdxs` and `shares` but no cursor bounds (unlike `addLiquidityWeighted`, which accepts `minimalCurBin / maximalCurBin`). A stale, pending transaction therefore executes with whatever cursor position the pool happens to be in at inclusion time.

Attack sequence:

1. User submits `addLiquidityExactShares` targeting bins `[X-1, X, X+1]` while the active bin is `X`. Transaction sits in the mempool.
2. Attacker front-runs: swaps token0 → token1, moving the active bin from `X` to `X-5`. Bins `X-1`, `X`, `X+1` are now all above the cursor → they accept only token0.
3. User's transaction executes: all three bins receive only token0 (within the `maxAmountToken0` cap, so the callback does not revert).
4. Attacker back-runs: swaps token1 → token0, restoring the cursor to `X`. Bins `X-1` and `X` are now below the cursor → they hold only token1. The user's deposited token0 in those bins has been effectively sold at the manipulated price.
5. Attacker pockets the arbitrage profit; user's LP position is worth less than the tokens deposited.

The `maxAmountToken0 / maxAmountToken1` caps in the callback guard only against paying *more* than the user approved; they do not prevent the user from receiving a position whose market value is less than the tokens paid. [3](#0-2) 

### Impact Explanation

A user depositing liquidity via `addLiquidityExactShares` can have their tokens deposited into bins at a manipulated cursor position. The resulting LP position is worth materially less than the tokens spent. This is a direct loss of user principal with no recovery path, matching the "Critical/High/Medium direct loss of user principal or owed LP assets" impact gate.

### Likelihood Explanation

Any pending `addLiquidityExactShares` transaction visible in the public mempool is a target. No privileged access is required; the attacker only needs to execute two swaps around the victim's transaction. MEV bots routinely perform exactly this pattern on liquidity-provision calls that lack deadline or price-bound guards.

### Recommendation

Add a `uint256 deadline` parameter to both `addLiquidityExactShares` overloads and call `_checkDeadline(deadline)` (or an equivalent inline check) at the top of each function, mirroring the pattern already used in `MetricOmmSimpleRouter`. Additionally, consider exposing optional cursor-bound parameters (as `addLiquidityWeighted` already does) so callers can bound the acceptable pool state at execution time.

### Proof of Concept

```solidity
// Attacker contract (pseudocode)
function sandwichLiquidityAdd(
    address pool,
    address victim,
    LiquidityDelta calldata victimDeltas,
    uint256 victimMax0,
    uint256 victimMax1
) external {
    // Step 1: move cursor away from victim's target bins
    pool.swap(address(this), true, largeAmount, MIN_PRICE, "", "");

    // Step 2: victim's addLiquidityExactShares executes here (no deadline → no revert)
    // victim deposits all token0 into bins that are now above the cursor

    // Step 3: restore cursor, collect arbitrage
    pool.swap(address(this), false, largeAmount, MAX_PRICE, "", "");
    // attacker profits; victim's position is worth < tokens deposited
}
```

The absence of `_checkDeadline` in `addLiquidityExactShares` is the sole root cause: the transaction cannot be forced to expire before the attacker's setup is in place. [4](#0-3)
