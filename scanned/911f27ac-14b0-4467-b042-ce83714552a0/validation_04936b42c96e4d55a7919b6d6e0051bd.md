### Title
Router `pay()` uses contract-wide ETH balance without verifying caller's `msg.value` covers the swap, enabling theft of stranded ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` helper in `PeripheryPayments.sol` funds WETH swaps from `address(this).balance` without verifying that the ETH came from the current caller's `msg.value`. Any ETH left in the router by a prior caller who omitted `refundETH()` can be silently consumed by a subsequent caller who sends 0 ETH, causing direct loss of the prior user's principal.

---

### Finding Description

When `token == WETH` and `payer != address(this)`, `pay()` reads the router's entire native balance and uses it to wrap and forward WETH to the pool:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`address(this).balance` is not scoped to the current transaction's `msg.value`. It includes any ETH that accumulated from prior callers who sent excess ETH to any of the router's `payable` entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `unwrapWETH9`, `sweepToken`, `refundETH`, `selfPermit*`) and did not call `refundETH()` before their transaction ended.

The entry points that trigger this path are all `payable` and perform no check that `msg.value >= amountIn` for WETH swaps:

```solidity
// MetricOmmSimpleRouter.sol line 67
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
``` [2](#0-1) 

In the swap callback, `pay()` is called with `payer = msg.sender` (the original caller) and `token = tokenIn`. When `tokenIn == WETH` and the router holds leftover ETH, the `nativeBalance >= value` branch fires, the router wraps the leftover ETH, and the payer's WETH allowance is never touched:

```solidity
// MetricOmmSimpleRouter.sol lines 192-199
function _justPayCallback(int256 amount0Delta, int256 amount1Delta) private {
    pay(
      _getTokenToPay(),
      _getPayer(),
      msg.sender,
      uint256(MetricOmmSwapResults.extractPositiveAmount(amount0Delta, amount1Delta))
    );
}
``` [3](#0-2) 

The same `pay()` path is reachable through `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback()` when either pool token is WETH:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 172-177
if (amount0Delta > 0) {
    pay(token0, payer, msg.sender, amount0Delta);
}
if (amount1Delta > 0) {
    pay(token1, payer, msg.sender, amount1Delta);
}
``` [4](#0-3) 

---

### Impact Explanation

Direct loss of user principal. ETH sent by User A that is not recovered (e.g., because `refundETH()` was omitted from a multicall, or a downstream revert left ETH stranded) can be consumed by User B's WETH swap at zero cost to User B. The loss equals the stranded ETH amount and is permanent — User A has no recourse once the ETH is wrapped and forwarded to the pool on User B's behalf.

---

### Likelihood Explanation

Medium. Users routinely send excess ETH to routers to avoid slippage-induced reverts and rely on `refundETH()` to recover the surplus. Omitting `refundETH()` from a multicall, or having a partial revert that leaves ETH in the router, is a realistic user error. An attacker monitoring the router's ETH balance (trivially done on-chain) can immediately exploit any non-zero balance by calling `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`.

---

### Recommendation

Track the ETH contributed by the current call in transient storage at the entry of each `payable` swap function, and in `pay()` use only that tracked amount — not `address(this).balance` — when funding WETH payments. Alternatively, remove native-ETH support from `pay()` entirely and require callers to wrap ETH themselves before invoking the router, eliminating the ambiguity between "ETH I sent now" and "ETH left by someone else."

---

### Proof of Concept

1. User A calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `msg.value = 2 ETH`. The swap callback uses 1 ETH; 1 ETH remains in the router. User A omits `refundETH()`.
2. User B calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `msg.value = 0`.
3. The pool calls `metricOmmSwapCallback`; `_justPayCallback` calls `pay(WETH, userB, pool, 1e18)`.
4. `address(this).balance == 1e18 >= 1e18` → router wraps User A's 1 ETH and transfers WETH to the pool.
5. User B receives full swap output. User A's 1 ETH is permanently lost; User B paid nothing.
