### Title
Router `pay()` Consumes Entire Native ETH Balance, Enabling Cross-Transaction ETH Theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **total** native ETH balance — when wrapping ETH to WETH for a swap payment. It does not scope the ETH to the amount sent by the current caller. Any ETH left in the router from a prior transaction (or a prior multicall step that did not consume it) is silently consumed to subsidize a later user's WETH payment, causing the original depositor to lose ETH with no swap output in return.

---

### Finding Description

In `pay()`, when `token == WETH` and `payer != address(this)`, the function reads `nativeBalance = address(this).balance`: [1](#0-0) 

The partial-balance branch at lines 78–81 wraps **all** of `nativeBalance` into WETH and transfers it to the pool, then pulls only the remainder from the payer via `transferFrom`. There is no check that `nativeBalance` was contributed by the current caller.

ETH can be left in the router between transactions because every swap entry-point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) is `payable`: [2](#0-1) 

If a user calls `exactInputSingle{value: 1 ether}(params)` where `params.tokenIn` is **not** WETH, `pay()` takes the `else` branch (line 86) and does a plain `transferFrom` — the 1 ETH sent with the call is never touched and remains in the router. The `receive()` guard (line 32–34) only blocks bare ETH transfers; it does not prevent ETH from arriving via payable function calls. [3](#0-2) 

---

### Impact Explanation

**Direct fund loss for the ETH depositor.**

Concrete scenario:

1. User A calls `exactInputSingle{value: 1 ether}(params)` where `tokenIn = DAI`. The 1 ETH is ignored by `pay()` and stays in the router.
2. Attacker calls `exactInputSingle(params)` where `tokenIn = WETH`, `amountIn = 1.5 ether`. In the swap callback, `pay(WETH, attacker, pool, 1.5 ether)` is invoked.
3. `nativeBalance = 1 ether > 0` and `< 1.5 ether` → partial branch fires: wraps User A's 1 ETH into WETH, transfers it to the pool, then pulls only 0.5 WETH from the attacker.
4. User A loses 1 ETH and receives nothing. The attacker's swap settles correctly but at a 1 ETH discount funded by User A.

The same path applies to excess ETH sent for a WETH swap when the caller omits `refundETH()`.

---

### Likelihood Explanation

- Sending ETH with a non-WETH swap is a realistic user mistake (common pattern from Uniswap v3 where native ETH is the input token).
- The router's ETH balance is publicly visible on-chain; an attacker can monitor it and submit an exploit transaction immediately after any ETH-leaving transaction.
- No privileged role, malicious pool, or non-standard token is required.

---

### Recommendation

Scope the native ETH available for wrapping to `msg.value` (the ETH sent with the current call), not `address(this).balance`. Pass `msg.value` through the call stack or read it from transient storage alongside the other callback context fields. Alternatively, track a "reserved ETH" slot in transient storage that is set at the entry-point and cleared after `pay()` consumes it, so residual ETH from prior transactions is never eligible.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry test (conceptual)
function test_crossUserEthTheft() public {
    // User A sends 1 ETH with a DAI swap — ETH is stranded in router
    vm.deal(userA, 1 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
        tokenIn: DAI, tokenOut: USDC, pool: daiUsdcPool,
        amountIn: 1000e18, amountOutMinimum: 0, ...
    }));

    assertEq(address(router).balance, 1 ether); // ETH stranded

    // Attacker swaps 1.5 WETH, only approves 0.5 WETH
    deal(WETH, attacker, 0.5 ether);
    IERC20(WETH).approve(address(router), 0.5 ether);
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        tokenIn: WETH, tokenOut: USDC, pool: wethUsdcPool,
        amountIn: 1.5 ether, amountOutMinimum: 0, ...
    }));

    // Router consumed User A's 1 ETH; attacker only paid 0.5 WETH
    assertEq(address(router).balance, 0);
    assertEq(IERC20(WETH).balanceOf(attacker), 0);
    // Pool received 1.5 WETH worth of value; attacker got full swap output
}
```

The key assertion is that the pool receives 1.5 ETH worth of WETH while the attacker only spent 0.5 WETH, with the 1 ETH deficit silently drawn from User A's stranded deposit.

---

**Note on the question's specific proof idea:** The proposed step-1 mechanism ("send ETH via `refundETH` revert") is incorrect — `refundETH()` does not revert; it immediately returns all ETH to `msg.sender`, leaving the router with zero balance before step 2 runs. The cross-transaction path described above is the correct exploit vector.
