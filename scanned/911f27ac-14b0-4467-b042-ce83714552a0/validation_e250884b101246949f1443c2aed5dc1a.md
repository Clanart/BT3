### Title
Excess ETH sent to payable exact-output swap functions is permanently trapped and can be stolen by any caller via `refundETH()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` function in `PeripheryPayments.sol` uses `address(this).balance` to fund WETH payments. All swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are `payable`. For exact-output swaps, the user legitimately cannot know the precise ETH input in advance and must send an upper-bound amount. The surplus ETH is never refunded automatically and remains in the contract. Any subsequent caller can drain it by calling `refundETH()`, which unconditionally sends the entire contract ETH balance to `msg.sender`.

---

### Finding Description

**Root cause — `PeripheryPayments.pay()` consumes only the exact amount needed, leaving the rest:** [1](#0-0) 

When `token == WETH` and `nativeBalance >= value`, the function wraps exactly `value` ETH and transfers it to the pool. Any ETH above `value` that was sent with the transaction silently remains in the contract with no accounting of who deposited it.

**`refundETH()` sends the entire balance to whoever calls it:** [2](#0-1) 

There is no per-depositor accounting. The function sends `address(this).balance` to `msg.sender`, so any caller — not just the original depositor — can claim all trapped ETH.

**All swap entry points are `payable` and do not refund excess ETH:** [3](#0-2) 

`exactOutputSingle` and `exactOutput` are the most dangerous paths: the actual `amountIn` is determined only after the pool executes the swap inside the callback, so users must send an upper-bound ETH value. The function returns `amountIn` but never refunds `msg.value - amountIn`.

**The `receive()` guard does not prevent ETH from entering via `payable` function calls:** [4](#0-3) 

`receive()` only fires when ETH is sent with no calldata. ETH sent alongside a `payable` function call bypasses this guard entirely and is accepted by the EVM.

**Secondary theft vector — `pay()` partial-balance branch consumes prior users' trapped ETH:** [5](#0-4) 

When `0 < nativeBalance < value`, the function wraps ALL existing native balance (which may belong to a prior user) and pulls only the remainder from the current payer via `safeTransferFrom`. This silently transfers a prior user's trapped ETH to the pool on behalf of the current user, reducing the current user's WETH cost at the prior user's expense.

---

### Impact Explanation

Direct loss of user ETH principal. A user calling `exactOutputSingle` or `exactOutput` with `tokenIn = WETH` must send an upper-bound ETH amount because the exact input is unknown until the pool executes. The surplus is permanently trapped in the router and can be stolen by any address calling `refundETH()` in a separate transaction, or silently consumed by the next user's swap via the partial-balance branch of `pay()`.

---

### Likelihood Explanation

**High.** Exact-output swaps are a core, advertised feature. Sending excess ETH is not a user error — it is the only correct way to call `exactOutputSingle`/`exactOutput` with native ETH when the precise input is unknown. The exploit requires no special privileges: any EOA can call `refundETH()` at any time. A MEV bot monitoring the mempool can front-run the victim's transaction or back-run it in the same block to claim the surplus.

---

### Recommendation

1. **Auto-refund at the end of each payable swap function.** After `_clearExpectedCallbackPool()`, add:
   ```solidity
   if (address(this).balance > 0) _transferETH(msg.sender, address(this).balance);
   ```
2. **Alternatively, track per-caller ETH deposits in transient storage** and restrict `refundETH()` to return only the caller's own deposit.
3. **Do not use `address(this).balance` as a shared pool** in `pay()`. Instead, pass the exact ETH amount to use as a parameter, sourced from the transient context set at entry.

---

### Proof of Concept

**Theft via `refundETH()`:**

1. Alice calls `exactOutputSingle({tokenIn: WETH, amountOut: 100e6 USDC, amountInMaximum: 1e18, ...})` with `msg.value = 1e18`.
2. The pool swap executes; the callback calls `pay(WETH, Alice, pool, 0.8e18)`.
3. `pay()` sees `nativeBalance = 1e18 >= 0.8e18`; wraps 0.8 ETH, sends to pool. [6](#0-5) 
4. `exactOutputSingle` returns. `0.2 ETH` remains in the router with no owner record.
5. Bob calls `refundETH()`. The function sends `address(this).balance = 0.2 ETH` to Bob. [2](#0-1) 
6. Alice loses 0.2 ETH with no recourse.

**Theft via subsequent swap (partial-balance branch):**

1. Same setup: 0.2 ETH trapped in router after Alice's swap.
2. Bob calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18, ...})` with `msg.value = 0`.
3. Callback calls `pay(WETH, Bob, pool, 1e18)`.
4. `pay()` sees `nativeBalance = 0.2e18 > 0` but `< 1e18`; wraps Alice's 0.2 ETH, sends to pool, then pulls only 0.8 WETH from Bob. [5](#0-4) 
5. Bob receives a 0.2 ETH discount funded entirely by Alice's trapped ETH.
