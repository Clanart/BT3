### Title
Unrestricted `sweepToken` and `unwrapWETH9` Allow Anyone to Drain Router Token Balances to an Arbitrary Recipient - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

### Summary

`sweepToken` and `unwrapWETH9` in `PeripheryPayments.sol` are `public payable` with no caller restriction and accept a fully attacker-controlled `recipient` address. Any tokens that reside in the router — including WETH output routed through the contract for a WETH→ETH unwrap pattern — can be redirected to an attacker's address by anyone who calls these functions before the legitimate owner does.

### Finding Description

`PeripheryPayments.sweepToken` and `unwrapWETH9` transfer the router's entire balance of a token to a caller-supplied `recipient`:

```solidity
// PeripheryPayments.sol line 48
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}

// PeripheryPayments.sol line 37
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
```

Neither function checks `msg.sender` nor restricts `recipient`. The intended usage pattern (documented by the test suite) is to batch these inside a `multicall` call immediately after a swap that routes output to `address(router)`:

```
multicall([
    exactInputSingle(..., recipient: address(router)),
    unwrapWETH9(0, user)          // or sweepToken(token, 0, user)
])
```

When a user executes these as two separate transactions instead of a single atomic `multicall`, the WETH (or ERC-20) sits in the router between the two transactions. An attacker who observes the first transaction in the mempool can front-run the second with their own call to `unwrapWETH9(0, attacker)` or `sweepToken(token, 0, attacker)`, draining the full balance to themselves.

The `exactInput` multi-hop path in `MetricOmmSimpleRouter` explicitly routes intermediate tokens through `address(this)`:

```solidity
// MetricOmmSimpleRouter.sol line 103-106
_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
IMetricOmmPoolActions(pool).swap(
    i == last ? params.recipient : address(this),   // intermediate output lands in router
    ...
);
```

When `params.recipient == address(this)` on the final hop (the WETH→ETH pattern), the output WETH is held by the router until `unwrapWETH9` is called. If that call is not in the same `multicall`, the balance is exposed.

### Impact Explanation

Any user who routes swap output through the router and calls `sweepToken` / `unwrapWETH9` in a separate transaction loses their entire output amount to the front-runner. This is a direct, complete loss of user principal with no recovery path. The attacker needs zero capital and zero permissions.

### Likelihood Explanation

The `multicall` batching requirement is not enforced at the contract level and is not documented in the function NatSpec. Users interacting via scripts, integrations, or wallets that call `unwrapWETH9` / `sweepToken` as standalone transactions (a natural usage pattern) are fully exposed. The attack is trivially executable by any mempool observer.

### Recommendation

Restrict `sweepToken` and `unwrapWETH9` so that `recipient` can only be `msg.sender`, or add a `msg.sender`-only guard:

```solidity
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    require(recipient == msg.sender, "recipient must be caller");
    ...
}
```

Alternatively, make both functions `internal` and expose them only through `multicall`, ensuring atomicity is structurally enforced rather than relying on caller discipline.

### Proof of Concept

1. User calls `exactInputSingle(..., tokenOut: WETH, recipient: address(router))` — WETH lands in the router.
2. Attacker observes the transaction and immediately calls `router.unwrapWETH9(0, attacker)`.
3. The router's full WETH balance is unwrapped and sent to the attacker as ETH.
4. User's follow-up `unwrapWETH9(0, user)` call finds zero balance and receives nothing.
