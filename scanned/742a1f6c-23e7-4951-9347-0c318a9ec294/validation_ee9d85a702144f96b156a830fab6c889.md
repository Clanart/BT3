### Title
Unvalidated `tokens[]` Array in `exactInput` Allows Attacker to Drain Router-Held Residue via Disconnected Path — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`exactInput` accepts a caller-controlled `tokens[]` array that is stored verbatim as the per-hop `tokenToPay` in transient callback context. `_validatePath` only checks array lengths — it never calls `poolsAreConnected` and never verifies that `tokens[i]` matches the actual input token of `pools[i]`. The swap callback `_justPayCallback` blindly pays whatever token is in transient storage. For intermediate hops the payer is `address(this)`, so the callback transfers any ERC-20 the router holds. An attacker who supplies a deliberately disconnected `tokens[]` can redirect router-held residue (WETH, ERC-20, or ETH converted to WETH) as payment for an intermediate hop, strand their own intermediate output on the router, and recover it via the unrestricted `sweepToken` helper.

---

### Finding Description

**Root cause — `_validatePath` omits connectivity check:** [1](#0-0) 

`_validatePath` only enforces array-length invariants. `MetricOmmSwapPath.poolsAreConnected` exists and is capable of verifying that `hopOutputToken(pools[i]) == hopInputToken(pools[i+1])`, but it is never called here. [2](#0-1) 

**Root cause — `tokens[i]` stored as `tokenToPay` without on-chain validation:** [3](#0-2) 

For every hop `i > 0` the payer is `address(this)` and the token is `params.tokens[i]` — both fully attacker-controlled.

**Root cause — callback pays whatever token is in transient storage:** [4](#0-3) 

When `payer == address(this)`, `pay` does an unconditional `safeTransfer` of `_getTokenToPay()` from the router: [5](#0-4) 

**Root cause — `sweepToken` / `refundETH` are unrestricted:** [6](#0-5) 

Any caller can sweep any ERC-20 balance the router holds to any recipient.

---

### Attack Scenario (step-by-step)

Precondition: the router holds a residue of `tokenX` (e.g., a user sent excess ETH with a WETH swap and the `pay` path deposited only `amountIn` worth, leaving the surplus as ETH; or a user mistakenly transferred `tokenX` directly to the router; or a prior multicall left WETH on the router).

1. Attacker calls `exactInput` with:
   - `tokens = [tokenA, tokenX, tokenC]` — **disconnected**: `tokenX` is not the output of `pool0`
   - `pools = [pool(tokenA/tokenB), pool(tokenX/tokenC)]`
   - `zeroForOneBitMap` set so `pool0` outputs `tokenB`, `pool1` inputs `tokenX`
   - `amountIn` = N (tokenA)

2. **Hop 0**: callback context has `payer = msg.sender`, `tokenToPay = tokenA`. Attacker pays N tokenA → `pool0` sends N' tokenB to `address(this)` (router). ✓

3. **Hop 1**: callback context has `payer = address(this)`, `tokenToPay = tokenX`. `_justPayCallback` calls `pay(tokenX, address(this), pool1, N')`. Router transfers N' of its residue `tokenX` to `pool1`. `pool1` sends N'' tokenC to `params.recipient`. ✓

4. The `amountInActual < amount` guard passes because `pool1` consumed exactly N' of tokenX (exact-input swap): [7](#0-6) 

5. N' tokenB is now stranded on the router. Attacker calls `sweepToken(tokenB, 0, attacker)` to recover it.

6. **Net**: attacker spent N tokenA, received N'' tokenC + N' tokenB, and consumed N' of the router's residue tokenX — effectively stealing tokenX.

---

### Impact Explanation

Any ERC-20 or WETH residue held by the router is claimable by an unprivileged attacker through a crafted `exactInput` call. The attacker pays their own `tokens[0]` amount but recovers it via `sweepToken`, so the net loss is entirely the router's residue. This is a direct loss of user-attributable value (prior users' excess ETH/WETH or mistakenly transferred tokens) with no privileged access required.

---

### Likelihood Explanation

Routers routinely accumulate residue: users send `msg.value > amountIn` for WETH swaps (the `pay` function deposits only `amountIn`, leaving the surplus as ETH on the router), users forget to append `refundETH()` to a multicall, or tokens are sent to the router by mistake. The attack requires only a factory-registered pool for the residue token, which is a normal operational condition.

---

### Recommendation

Add a connectivity check inside `_validatePath` (or inline in `exactInput`) that calls `MetricOmmSwapPath.poolsAreConnected` and additionally asserts that `tokens[i]` equals `MetricOmmSwapPath.hopInputToken(pools[i], zeroForOne_i)` for every hop. This ensures the `tokenToPay` stored in transient context always matches the pool's actual input token, making a disconnected-path attack impossible regardless of router residue.

---

### Proof of Concept

```solidity
// Setup: router holds 100e18 tokenX (residue from a prior user)
// tokenA/tokenB pool and tokenX/tokenC pool are both factory-registered.

ExactInputParams memory params = ExactInputParams({
    tokens:         [tokenA, tokenX, tokenC],   // disconnected: tokenX ≠ output of pool0
    pools:          [pool_AB, pool_XC],
    extensionDatas: [bytes(""), bytes("")],
    zeroForOneBitMap: 0b01,                     // pool0: tokenA→tokenB, pool1: tokenX→tokenC
    amountIn:       50e18,                       // 50 tokenA
    amountOutMinimum: 0,
    recipient:      attacker,
    deadline:       block.timestamp + 1
});

router.exactInput(params);
// router now holds 50e18 tokenB (stranded from hop 0)
// attacker received tokenC (paid by router's tokenX residue)

router.sweepToken(tokenB, 0, attacker);
// attacker recovers tokenB; net cost = 0, net gain = tokenC
```

The `amountInActual < amount` check at line 115 does not block this because `pool_XC` consumes exactly `amount` of tokenX (exact-input semantics), satisfying the guard while paying the wrong token. [8](#0-7)

  function poolsAreConnected(address[] calldata pools, uint256 zeroForOneBitMap) internal view returns (bool) {
    uint256 last = pools.length - 1;
    for (uint256 i = 0; i < last; i++) {
      bool zeroForOne = resolveZeroForOneBitmap(zeroForOneBitMap, i);
      bool nextZeroForOne = resolveZeroForOneBitmap(zeroForOneBitMap, i + 1);
      if (hopOutputToken(pools[i], zeroForOne) != hopInputToken(pools[i + 1], nextZeroForOne)) {
        return false;
      }
    }
    return true;
  }
