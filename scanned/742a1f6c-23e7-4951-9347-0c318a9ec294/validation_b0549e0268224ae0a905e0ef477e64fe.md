### Title
Missing deadline check on `addLiquidityExactShares()` allows stale transactions to add liquidity at unfavorable oracle-shifted price levels — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` exposes `addLiquidityExactShares()` (both overloads) without any deadline parameter or deadline enforcement. The sibling router contract `MetricOmmSimpleRouter` consistently calls `_checkDeadline(params.deadline)` at the top of every swap entry point, but the liquidity adder has no equivalent guard. A pending `addLiquidityExactShares` transaction can sit in the mempool while the oracle price moves, then execute and deposit the user's tokens into bins that are now far from the live oracle price.

---

### Finding Description

`MetricOmmSimpleRouter` enforces a deadline on every user-facing function: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares()` has no such guard: [5](#0-4) [6](#0-5) 

The `addLiquidityWeighted` overloads have partial protection via `_validateBinAndBinPosition`, which reverts if the pool cursor has moved outside caller-specified bounds: [7](#0-6) 

But `addLiquidityExactShares` has **no** cursor-bounds check and **no** deadline. The only guards are `maxAmountToken0` / `maxAmountToken1` caps, which protect against paying too many tokens but do not protect against the oracle price having moved to a completely different level.

Because Metric OMM is an oracle-anchored AMM, the "active" bins are determined by the live oracle bid/ask. Bins that were near the oracle price when the user signed the transaction may be far from it by the time the transaction is mined.

---

### Impact Explanation

A user who calls `addLiquidityExactShares` targeting bins near the current oracle price can have their transaction delayed (gas spike, mempool congestion). When it eventually executes, the oracle price may have moved significantly. The user's tokens are deposited into bins that are now inactive — those bins will not be crossed by any swap and will earn zero fees. The user's capital is locked in a dead position until they manually remove it. Because the user specified exact bin indices and exact share amounts, the `maxAmountToken0`/`maxAmountToken1` caps do not help: the pool may still consume the full cap at the now-stale price levels.

This is a direct loss of the opportunity to earn fees and, in volatile markets, the user may have chosen not to add liquidity at all (or to different bins) had they known the oracle had moved. The severity matches the external report's medium classification: no principal is destroyed in a single block, but the user pays real token cost (gas + locked capital) for a position that provides no value.

---

### Likelihood Explanation

Any network congestion event (gas auction spike, block-space competition) can delay a liquidity transaction by minutes to hours. Metric OMM pools are oracle-anchored, so the oracle price can move substantially in that window. This is a normal operating condition, not an exotic attack. No adversary is required; the loss occurs from ordinary mempool delay.

---

### Recommendation

Add a `deadline` parameter to both `addLiquidityExactShares` overloads and call the same `_checkDeadline` helper already used by `MetricOmmSwapRouterBase`:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
+   uint256 deadline,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
+   _checkDeadline(deadline);
    _validateOwner(owner);
    ...
}
```

Apply the same change to the `msg.sender`-owner overload. For `addLiquidityWeighted`, the existing `_validateBinAndBinPosition` check provides meaningful protection, but adding a deadline there as well is consistent and cheap.

---

### Proof of Concept

1. Oracle price for ETH/USDC is $3 000. User calls `addLiquidityExactShares` targeting bins 10, 11, 12 (the three bins straddling $3 000). `maxAmountToken0 = 1 ETH`, `maxAmountToken1 = 3 000 USDC`.
2. Gas price spikes; the transaction sits in the mempool for 30 minutes.
3. Oracle price moves to $2 400. Bins 10–12 are now three bins above the active cursor; no swap will ever cross them.
4. Transaction is mined. The pool accepts the deposit into bins 10–12 (the bin indices are valid; the amounts are within the caps). The user has deposited 1 ETH + 3 000 USDC into permanently inactive bins.
5. The user earns zero fees. They must send a second transaction to remove liquidity, paying gas twice and having earned nothing.

Had a deadline of `block.timestamp + 5 minutes` been enforced, step 4 would have reverted and the user's funds would have remained in their wallet. [8](#0-7)
