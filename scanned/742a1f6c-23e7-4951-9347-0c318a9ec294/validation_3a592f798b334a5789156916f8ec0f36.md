### Title
Stale Router ETH from Unclaimed `msg.value` Funds a Subsequent User's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

The `pay()` function's hybrid WETH branch reads `address(this).balance` without any per-user accounting. If ETH from a prior transaction's `msg.value` remains in the router (because the sender never called `refundETH()`), a subsequent user's WETH swap will consume that stale ETH to partially or fully fund their payment, stealing it from the original depositor.

### Finding Description

The `pay()` function contains a three-branch WETH path: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` (line 74) and, if nonzero, wraps and forwards that ETH to the pool before pulling the remainder from `payer` via `transferFrom`. There is no check that the ETH balance belongs to the current caller.

The router's `receive()` guard only blocks *direct* ETH transfers from non-WETH addresses: [2](#0-1) 

However, ETH sent as `msg.value` in any `payable` function call bypasses `receive()` entirely and is credited to the contract. Every swap entry point is `payable`: [3](#0-2) [4](#0-3) 

So the attack path is:

1. **User A** calls `exactInputSingle` (or `multicall`) with `tokenIn = WETH` and `msg.value = 2 ETH`, but the swap only costs 1 ETH. The callback calls `pay(WETH, userA, pool, 1 ETH)`. `nativeBalance = 2 ETH ≥ 1 ETH`, so 1 ETH is wrapped and forwarded. **1 ETH remains in the router.**
2. User A does not call `refundETH()` (forgot, or the multicall omitted it).
3. **User B** calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 0.5 ETH`. The callback calls `pay(WETH, userB, pool, 0.5 ETH)`. `nativeBalance = 1 ETH ≥ 0.5 ETH`, so **User A's ETH** is wrapped and forwarded. User B's WETH `transferFrom` is never called.

User A loses 0.5 ETH; User B's swap is funded for free.

### Impact Explanation

Direct, permanent loss of ETH for User A. The ETH is not locked — it is actively transferred to a pool on behalf of an unrelated user. This satisfies the "direct loss of user principal" impact gate.

### Likelihood Explanation

- Sending excess `msg.value` is a standard pattern (users pad ETH to ensure the swap succeeds at any price).
- `refundETH()` is optional and not enforced by the router; users who call swap functions directly (not via a carefully constructed multicall) will routinely leave dust ETH.
- No privileged role or malicious setup is required; any two sequential transactions trigger it.

### Recommendation

In the `pay()` hybrid WETH branch, only consume `msg.value` that was explicitly sent in the **current** transaction. The standard fix (used by Uniswap v3's `PeripheryPayments`) is to track `msg.value` via a transient or per-call variable and consume only up to that amount, not `address(this).balance`. Alternatively, enforce that the router's ETH balance is zero at the start of every non-`multicall` entry point, or require `refundETH()` to be the last call in every multicall that involves ETH.

### Proof of Concept

```solidity
// 1. User A sends 2 ETH, swap costs 1 ETH, 1 ETH stranded in router
router.exactInputSingle{value: 2 ether}(paramsA); // tokenIn=WETH, amountIn=1e18
// User A forgets refundETH()
assertEq(address(router).balance, 1 ether);

// 2. User B swaps 0.5 WETH — router uses User A's ETH, never pulls from User B
uint256 balBefore = weth.balanceOf(userB);
router.exactInputSingle(paramsB); // tokenIn=WETH, amountIn=0.5e18, msg.value=0
assertEq(weth.balanceOf(userB), balBefore); // User B's WETH untouched
assertEq(address(router).balance, 0.5 ether); // 0.5 ETH consumed from User A's deposit
```
