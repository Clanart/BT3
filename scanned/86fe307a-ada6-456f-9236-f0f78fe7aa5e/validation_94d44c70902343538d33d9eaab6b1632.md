### Title
Missing `deadline` parameter in `addLiquidityWeighted` and `addLiquidityExactShares` allows stale-price liquidity deposits — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` exposes four public `addLiquidity*` entry points, none of which accept or enforce a `deadline`. The router (`MetricOmmSimpleRouter`) correctly calls `_checkDeadline(params.deadline)` at the top of every swap function, but the liquidity adder has no equivalent guard. A pending transaction can sit in the mempool and execute arbitrarily late, at a stale oracle price, causing the LP to deposit at a composition that is far from the price they intended.

---

### Finding Description

All four public entry points in `MetricOmmPoolLiquidityAdder` lack a `deadline` parameter: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

By contrast, every swap entry point in the router enforces a deadline before touching the pool: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The `addLiquidityWeighted` path is the most dangerous case. It performs a **probe** call to determine `need0`/`need1` at the current oracle price, then immediately scales shares and executes the real deposit: [9](#0-8) 

Both the probe and the paying add execute within the same transaction, so there is no intra-transaction gap. The vulnerability is that the **entire transaction** can be withheld by a validator or simply sit in a congested mempool for an extended period. When it finally lands, the oracle price may have moved substantially. The probe then reads the new (stale-to-the-user) price, scales shares accordingly, and the deposit is settled at a composition the user never intended.

The `maxAmountToken0` / `maxAmountToken1` caps are the only protection: [10](#0-9) 

These caps are analogous to `priceMarginBps` in the referenced external report — they bound the *quantity* spent but do not bound the *time* of execution. A user who sets generous caps (e.g., to avoid reverts from minor price moves) is fully exposed to late execution at an arbitrarily stale price.

The `_validateBinAndBinPosition` guard in `addLiquidityWeighted` provides partial protection only when the oracle cursor has moved outside the user-supplied bin bounds: [11](#0-10) 

However, this check is absent from `addLiquidityExactShares`, and even in `addLiquidityWeighted` it only catches cursor movement, not oracle price drift within the same bin.

---

### Impact Explanation

An LP submits `addLiquidityWeighted` intending to deposit at the current market price. The transaction is delayed (network congestion, validator withholding). When it executes, the oracle price has moved significantly. The probe computes `need0`/`need1` at the new price, shares are scaled to that composition, and the deposit settles at a price far from the user's intent. The LP immediately holds a position with embedded impermanent loss relative to the price at submission time. Because the `maxAmount` caps are set by the user to accommodate normal slippage, they do not prevent execution at a price that is minutes or hours stale. This constitutes direct loss of LP principal.

---

### Likelihood Explanation

Any pending transaction in a public mempool is subject to validator reordering or simple delay. On L2s (where this protocol is likely deployed given the oracle infrastructure), sequencer-controlled ordering makes deliberate delay trivial. A user who submits during a period of price volatility is at elevated risk. No special privilege is required; the trigger is a normal, valid user transaction.

---

### Recommendation

Add a `deadline` parameter to all four `addLiquidity*` entry points and validate it at the top of each function, mirroring the pattern already used in `MetricOmmSimpleRouter`:

```solidity
// In MetricOmmPoolLiquidityAdder
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    uint256 deadline,          // <-- add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _checkDeadline(deadline);  // <-- add
    ...
}
```

Apply the same change to both `addLiquidityWeighted` overloads and both `addLiquidityExactShares` overloads. The `_checkDeadline` helper already exists in `MetricOmmSwapRouterBase`; extract it to a shared base or utility so both contracts can use it.

---

### Proof of Concept

1. Alice calls `addLiquidityWeighted` with `maxAmountToken0 = 1000e18`, `maxAmountToken1 = 1000e18`, targeting the current oracle price of 1 token0 = 1 token1.
2. The transaction enters the mempool but is not included immediately (network congestion or validator delay).
3. The oracle price moves to 1 token0 = 2 token1 (token0 doubles in value).
4. The transaction is included. The probe reads the new price: `need0 = 500e18`, `need1 = 1000e18`.
5. Shares are scaled to this composition. Alice deposits 500 token0 and 1000 token1.
6. At the original price, 500 token0 was worth 500 token1, so Alice deposited 1500 token1-equivalent. At the new price, 500 token0 is worth 1000 token1, so her position is worth 2000 token1-equivalent — but she paid 1500 token1-equivalent at the old price. However, the position is now at the new price, meaning any reversion of the price move results in impermanent loss crystallized at the stale entry point.
7. More concretely: if Alice intended a 50/50 split at the old price but the price moved adversely, she ends up with a skewed position she did not consent to, with no recourse because no deadline was enforced.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
