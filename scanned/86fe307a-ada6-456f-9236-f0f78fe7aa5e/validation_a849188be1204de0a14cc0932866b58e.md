### Title
Payable swap and liquidity functions silently trap ETH sent for non-WETH operations, allowing any caller to steal it via `refundETH()` - (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol, metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

---

### Summary

Every swap entry-point in `MetricOmmSimpleRouter` and every liquidity entry-point in `MetricOmmPoolLiquidityAdder` is marked `payable`. When a caller sends ETH with a call whose input token is **not** WETH, the internal `pay()` helper falls through to a plain `safeTransferFrom` and never touches `address(this).balance`. The ETH is silently retained by the contract. Because `refundETH()` is an unrestricted `external payable` function that forwards the **entire** contract balance to `msg.sender`, any third party can immediately drain the trapped ETH.

---

### Finding Description

`PeripheryPayments.pay()` uses `address(this).balance` only when `token == WETH`:

```solidity
// PeripheryPayments.sol lines 69-88
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {          // ← only branch that consumes ETH
        uint256 nativeBalance = address(this).balance;
        ...
    } else {
        IERC20(token).safeTransferFrom(payer, recipient, value);  // ETH ignored
    }
}
```

All four router swap functions and all four liquidity-adder functions are `payable` with no guard that rejects non-zero `msg.value` when the operation does not involve WETH:

- `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput` — `MetricOmmSimpleRouter`
- `addLiquidityExactShares` (×2), `addLiquidityWeighted` (×2) — `MetricOmmPoolLiquidityAdder`
- `selfPermit`, `selfPermitIfNecessary`, `selfPermitAllowed`, `selfPermitAllowedIfNecessary` — `SelfPermit` (never consume ETH at all)

`refundETH()` has no access control and sends the full balance to the caller:

```solidity
// PeripheryPayments.sol lines 58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← any caller, full balance
    }
}
```

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers with no calldata; it does **not** prevent ETH from arriving via `msg.value` in a normal function call.

---

### Impact Explanation

A user who sends ETH alongside a non-WETH swap or liquidity call loses that ETH permanently to the first MEV bot that calls `refundETH()`. The loss is 1:1 with the ETH sent. No privileged action is required by the attacker; `refundETH()` is callable by any EOA or contract.

---

### Likelihood Explanation

- UI bugs, copy-paste errors, or wallet integrations that pre-populate `msg.value` for "gas" are realistic triggers.
- Users adding liquidity to a non-WETH pool (e.g., USDC/DAI) via `addLiquidityWeighted{value: X}` is a plausible mistake.
- MEV bots routinely monitor the mempool for exactly this pattern (unprotected `refundETH`-style drains). The window between the victim's transaction and the theft can be a single block.

---

### Recommendation

**Short term:** Add a `msg.value == 0` guard to every `payable` function that does not accept ETH as a payment leg. For functions that conditionally accept ETH (WETH path), add a post-execution check that `address(this).balance` has not increased unexpectedly, or enforce the refund inside the same call.

```solidity
modifier noNativeValue() {
    require(msg.value == 0, "unexpected ETH");
    _;
}
```

Apply `noNativeValue` to: `addLiquidityExactShares`, `addLiquidityWeighted`, `selfPermit*`, and any swap function whose `tokenIn` is statically known to be non-WETH.

**Long term:** Audit every `payable` entry-point and document explicitly whether it is ETH-accepting or ETH-rejecting. Consider making `refundETH()` callable only within a `multicall` context (e.g., via a reentrancy-style flag) so it cannot be weaponized as a standalone drain.

---

### Proof of Concept

```
1. Alice calls:
   router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
       tokenIn:  USDC,   // ← not WETH
       tokenOut: DAI,
       amountIn: 1_000e6,
       ...
   }));
   
   Execution path:
   → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, Alice, USDC)
   → pool.swap(...)  triggers metricOmmSwapCallback
   → _justPayCallback → pay(USDC, Alice, pool, amount)
   → token != WETH → safeTransferFrom(Alice, pool, amount)   // ETH untouched
   → function returns; 1 ETH remains in router

2. Bob (MEV bot) calls:
   router.refundETH();
   → balance = address(router).balance  // = 1 ETH
   → _transferETH(Bob, 1 ETH)           // Bob receives Alice's ETH

Alice loses 1 ETH. Bob gains 1 ETH. No privileged access required.
```

The identical scenario applies to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares{value: X}(...)` when neither pool token is WETH, and to all four `SelfPermit` functions which never consume ETH under any circumstances. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
