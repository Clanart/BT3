### Title
Unattributed Native ETH Balance in `pay()` Allows Any Caller to Steal Victim's Leftover ETH via WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses the router's entire `address(this).balance` as the ETH source for any WETH payment, with no per-transaction attribution. A victim who sends excess `msg.value` to a WETH `exactInputSingle` call and omits `refundETH()` leaves ETH stranded in the router. Any subsequent caller who swaps WETH with `value: 0` will have `pay()` consume that stranded ETH on their behalf, receiving swap output at the victim's expense.

---

### Finding Description

`pay()` in `PeripheryPayments.sol` handles WETH payments as follows: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — the router's **global** ETH balance — and uses it first, before pulling from the payer's WETH allowance. There is no mechanism that attributes which transaction deposited which ETH. Any ETH left in the router from a prior call is indistinguishable from ETH sent in the current call.

`exactInputSingle` is `payable`: [2](#0-1) 

It stores `msg.sender` as the payer in transient storage, then the pool triggers `_justPayCallback`: [3](#0-2) 

The callback calls `pay(_getTokenToPay(), _getPayer(), pool, amountOwed)`. When `tokenIn == WETH`, `pay()` checks `address(this).balance` first — regardless of who sent that ETH.

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses: [4](#0-3) 

It does **not** block ETH arriving as `msg.value` on a `payable` function call. ETH sent with `exactInputSingle{value: X}` enters the router's balance and persists after the call if not refunded.

---

### Impact Explanation

**Direct loss of user ETH.** The attacker calls `exactInputSingle{value: 0}` for a WETH swap of size `N`. `pay()` finds `address(this).balance >= N` (the victim's stranded ETH), wraps it to WETH, and sends it to the pool. The attacker receives the full swap output having spent nothing. The victim's ETH is permanently consumed.

The loss is bounded only by how much ETH the victim left in the router. A MEV bot can monitor the mempool for victim transactions that leave ETH residue and immediately follow with a stealing swap in the same block.

---

### Likelihood Explanation

- Users routinely send `msg.value > amountIn` to guarantee a WETH swap succeeds (e.g., slippage buffer on the input side), especially when calling `exactInputSingle` directly rather than via `multicall`.
- `refundETH()` is a separate, optional call. Omitting it is a common user error.
- The attack requires no special permissions, no malicious pool, and no non-standard tokens — only a public call to `exactInputSingle` with `tokenIn = WETH`.
- MEV searchers can automate detection and exploitation with zero marginal cost.

---

### Recommendation

Track ETH attribution per-call. The simplest fix is to record `msg.value` at the entry of each swap function and pass it explicitly into `pay()` as the available native budget, rather than reading `address(this).balance`. Alternatively, enforce that `address(this).balance` at the start of each top-level call equals zero (revert if not), so no cross-transaction ETH leakage is possible.

A minimal targeted fix for `pay()`:

