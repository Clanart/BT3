### Title
Cross-User ETH Consumption in `PeripheryPayments.pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay` uses the contract's entire native ETH balance to satisfy a WETH payment when `nativeBalance >= value`, without any per-user accounting. ETH left on the contract by a prior LP caller (from an overpaid `payable` liquidity add) is silently consumed by the next caller's payment, causing the prior LP to lose their ETH principal.

### Finding Description

`addLiquidityExactShares` is `payable`. When LP-A calls it with `msg.value = 5 ETH` for a WETH pool that only needs 2 ETH, `pay()` wraps 2 ETH and sends WETH to the pool. The remaining 3 ETH stays on the contract. [1](#0-0) 

When LP-B subsequently calls `addLiquidityExactShares` for a WETH pool needing 3 WETH (sending no ETH), the callback invokes `pay(WETH, LP-B, pool, 3e18)`. At line 75, `nativeBalance = 3 ETH >= value = 3 ETH`, so the function wraps LP-A's 3 ETH and transfers WETH to the pool — **without pulling any WETH from LP-B**. LP-B's WETH allowance is never touched. [2](#0-1) 

The `receive()` guard (only accepting ETH from WETH) does not prevent this: it only applies to plain ETH transfers, not ETH sent with a function call via `msg.value`. [3](#0-2) 

The transient `PayContextAlreadyActive` guard only prevents reentrancy within a single transaction; it does not prevent LP-B from calling in a separate transaction after LP-A's transaction completes. [4](#0-3) 

### Impact Explanation

LP-A suffers direct loss of ETH principal. LP-B adds liquidity for free. The loss is bounded only by how much ETH LP-A overpaid. No privileged role is required — LP-B is any ordinary caller.

### Likelihood Explanation

Any user who sends more ETH than the pool needs (a common pattern when the exact amount is unknown) leaves ETH on the contract. A frontrunner or any concurrent user can immediately consume it. The window is open from the end of LP-A's transaction until `refundETH()` is called, which is a separate transaction.

### Recommendation

Track per-user ETH deposits (e.g., a transient slot for `msg.value` credited to the current payer) and only use that credited amount in `pay()`. Alternatively, enforce that `pay()` only uses native ETH when `payer == msg.sender` and the credited ETH matches the current call's `msg.value`. The safest fix is to wrap `msg.value` into WETH immediately on entry (before any pool call) and treat it as an ERC-20 balance, eliminating the native-ETH branch in `pay()` for cross-user safety.

// Foundry test sketch
function test_crossUserEthConsumption() public {
    // LP-A adds liquidity to a WETH/token pool, overpaying by 3 ETH
    vm.deal(lpA, 5 ether);
    vm.prank(lpA);
    adder.addLiquidityExactShares{value: 5 ether}(
        pool, lpA, 0, deltas_needing_2eth, 5 ether, 0, ""
    );
    // 3 ETH remains on adder contract
    assertEq(address(adder).balance, 3 ether);

    // LP-B adds liquidity to a WETH pool needing 3 WETH, sends no ETH
    // LP-B has WETH approval but pay() never pulls it
    vm.prank(lpB);
    adder.addLiquidityExactShares(
        pool, lpB, 0, deltas_needing_3eth, 3 ether, 0, ""
    );

    // LP-A's 3 ETH is gone, LP-B paid nothing
    assertEq(address(adder).balance, 0);
    assertEq(weth.balanceOf(lpB), 0); // LP-B's WETH untouched
}
