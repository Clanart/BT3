The vulnerability is real and provable. The `pay` function in `PeripheryPayments.sol` uses `address(this).balance` — the router's **total** native ETH balance — with no per-user accounting. Any ETH stranded from a prior multicall that omitted `refundETH` is silently consumed by the next WETH-input swap.

### Title
Stranded ETH in Router Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses the router's entire native ETH balance when settling a WETH-input swap. Because `multicall` is payable and `refundETH` is optional, any ETH left over from a prior user's multicall is silently spent on behalf of the next caller who swaps with `tokenIn == WETH`.

### Finding Description

`pay` branches on `token == WETH` and reads `address(this).balance` as `nativeBalance`: [1](#0-0) 

If `nativeBalance >= value`, the function wraps and transfers the router's ETH to the pool — **without pulling any WETH from `payer`**: [2](#0-1) 

`refundETH` is a separate, optional call that sends **all** router ETH to `msg.sender`: [3](#0-2) 

`multicall` is payable and imposes no requirement to include `refundETH`: [4](#0-3) 

When `exactInputSingle` is called with `tokenIn == WETH`, the payer stored in transient context is `msg.sender`, but `pay` ignores that payer entirely whenever `nativeBalance >= value`: [5](#0-4) 

### Impact Explanation

**Direct loss of user ETH.** User A's stranded ETH is consumed to settle User B's swap. User B pays zero WETH from their wallet; User A's ETH is gone. The router ends with zero balance, and User A has no recourse. This is a direct principal loss above any Sherlock High threshold.

### Likelihood Explanation

- Forgetting `refundETH` in a multicall is a common integration mistake (the test suite itself demonstrates the correct pattern, implying it is easy to omit).
- MEV bots can monitor the mempool for multicalls that omit `refundETH` and front-run the refund with a WETH `exactInputSingle` for exactly `router.balance`.
- No special permissions, no malicious pool, no non-standard tokens required.

### Recommendation

Track per-call ETH credit in transient storage (set to `msg.value` at entry, decremented as it is consumed) and use only that credit — not `address(this).balance` — inside `pay`. Alternatively, revert in `pay` if `nativeBalance > 0` and the current call did not supply ETH (i.e., `msg.value == 0`), preventing cross-user ETH consumption.

### Proof of Concept

```
1. User A: router.multicall{value: 1 ether}([
       exactInputSingle(tokenIn=WETH, amountIn=0.9 ether, ...)
       // refundETH omitted
   ])
   → router.balance == 0.1 ether

2. User B: router.exactInputSingle{value: 0}(
       tokenIn=WETH, amountIn=0.1 ether, ...
   )
   → pay(WETH, userB, pool, 0.1 ether) is called
   → nativeBalance (0.1 ether) >= value (0.1 ether)
   → router wraps 0.1 ether of User A's ETH and transfers WETH to pool
   → User B's WETH allowance is never touched

3. Assert: router.balance == 0
           User A lost 0.1 ETH
           User B paid 0 WETH from wallet
```
