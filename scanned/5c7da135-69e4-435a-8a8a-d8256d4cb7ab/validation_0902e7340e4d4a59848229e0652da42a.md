### Title
`pay()` ignores router-held WETH balance when settling an external-payer WETH swap, causing payer overcharge and stealable stranded funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` helper in `PeripheryPayments.sol` checks only `address(this).balance` (native ETH) when deciding how much WETH to pull from an external payer. It never inspects `IERC20(WETH).balanceOf(address(this))`. When a multicall step leaves WETH on the router and a subsequent step uses WETH as input, the router ignores its own WETH balance, pulls the full amount from the user's wallet, and leaves the earlier WETH stranded — where any caller can steal it via the public `sweepToken` helper.

---

### Finding Description

`pay()` in `PeripheryPayments.sol` (lines 69–88) handles the WETH branch as follows:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;          // ← only native ETH checked
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // ← payer overcharged
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value); // ← payer charged full amount
    }
}
``` [1](#0-0) 

The function never reads `IERC20(WETH).balanceOf(address(this))`. If the router already holds `W` WETH (deposited by a prior multicall step), that balance is completely ignored. The payer is charged the full `value` from their own wallet, and the `W` WETH remains stranded on the router.

The `sweepToken` and `unwrapWETH9` helpers are both `public payable` with no access control and accept a caller-chosen `recipient`:

```solidity
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert ...;
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
``` [2](#0-1) 

Any address can call `sweepToken(WETH, 0, attacker)` to drain the stranded WETH.

The callback context for `exactInputSingle` always sets the payer to `msg.sender`:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
``` [3](#0-2) 

So even inside a multicall, each `exactInputSingle` step treats the original caller as the payer, never the router's own balance (unless `payer == address(this)` is set explicitly, which only happens for intermediate hops in `exactInput`). [4](#0-3) 

---

### Impact Explanation

**Direct loss of user principal.** Alice's WETH (obtained from step 1 of a multicall and held on the router) is never credited toward step 2's payment obligation. She is charged the full swap amount from her own wallet. The router's WETH is then immediately stealable by any third party via `sweepToken`. Alice loses `min(W, V)` WETH where `W` is the router's pre-existing WETH balance and `V` is the swap's required input.

---

### Likelihood Explanation

The multicall pattern of "swap token → WETH to router, then swap WETH → token2" is the standard Uniswap v3-style flow explicitly documented and tested in this codebase (see `test_multicall_tokenForWeth_thenUnwrapEth`). Any user following this pattern with two separate `exactInputSingle` calls instead of a single `exactInput` multihop triggers the bug. No privileged access is required. The attacker only needs to watch the mempool for a stranded WETH balance and call `sweepToken`. [5](#0-4) 

---

### Recommendation

In the WETH branch of `pay()`, read the router's existing WETH balance and apply it before pulling from the external payer:

```solidity
} else if (token == WETH) {
    uint256 wethBalance  = IERC20(WETH).balanceOf(address(this));
    uint256 nativeBalance = address(this).balance;

    if (wethBalance >= value) {
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        uint256 remaining = value - wethBalance;
        if (wethBalance > 0) IERC20(WETH).safeTransfer(recipient, wethBalance);
        if (nativeBalance >= remaining) {
            IWETH9(WETH).deposit{value: remaining}();
            IERC20(WETH).safeTransfer(recipient, remaining);
        } else if (nativeBalance > 0) {
            IWETH9(WETH).deposit{value: nativeBalance}();
            IERC20(WETH).safeTransfer(recipient, nativeBalance);
            IERC20(WETH).safeTransferFrom(payer, recipient, remaining - nativeBalance);
        } else {
            IERC20(WETH).safeTransferFrom(payer, recipient, remaining);
        }
    }
}
```

### Proof of Concept

```
Setup:
  - pool_A: token1/WETH
  - pool_B: WETH/token2
  - Alice has token1 and WETH, both approved to router

Attack (Alice's multicall):
  calls[0] = exactInputSingle(token1 → WETH, amountIn=X, recipient=address(router))
              → router receives W WETH

  calls[1] = exactInputSingle(WETH → token2, amountIn=V, recipient=alice)
              → pay(WETH, alice, pool_B, V) is called
              → nativeBalance = 0
              → falls through to safeTransferFrom(alice, pool_B, V)
              → Alice is charged V WETH from her own wallet
              → router's W WETH is untouched and stranded

After multicall:
  router.balanceOf(WETH) == W   ← stranded

Attacker (Bob, separate tx):
  sweepToken(WETH, 0, bob)
  → Bob receives W WETH

Alice's loss: W WETH (she paid V from wallet instead of max(0, V-W))
```
