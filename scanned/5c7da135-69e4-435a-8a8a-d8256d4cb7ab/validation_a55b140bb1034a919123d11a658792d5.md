### Title
Router Native ETH Balance Consumed by Any Caller to Satisfy WETH Payment Obligations — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
The `pay()` function in `PeripheryPayments` uses the router's entire native ETH balance to satisfy WETH payment obligations without attributing that ETH to the specific user who deposited it. Any caller can consume ETH stranded on the router from a prior multicall step to satisfy their own WETH payment obligation, receiving a swap output without paying from their own funds.

### Finding Description
When `token == WETH` and `payer != address(this)`, `pay()` checks `address(this).balance` and uses it to satisfy the payment before pulling from `payer` via `transferFrom`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // uses router's ETH, never touches payer
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // only pulls remainder
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

The router's `receive()` only accepts ETH from WETH, but `multicall` is `payable` and ETH sent with it is held by the router contract. If a user sends `multicall{value: X}` with a swap that only consumes `Y < X` of ETH and omits a `refundETH` call, the remaining `X - Y` ETH is stranded on the router. A subsequent caller who specifies `tokenIn = WETH` will have their callback's `pay()` call consume that stranded ETH — the `nativeBalance >= value` branch fires, the router deposits the stranded ETH as WETH and transfers it to the pool, and the attacker's `payer` is never charged via `transferFrom`.

The transient callback context correctly binds the expected pool and payer, but the `pay()` function's ETH-first logic ignores the attribution of that ETH to any particular user. The invariant broken is: *each user's WETH payment obligation must be satisfied from their own funds*. The router's native ETH balance is treated as a shared, unattributed pool.

### Impact Explanation
Direct loss of user principal. User A loses the stranded ETH (up to the full `msg.value` they sent minus what the swap consumed). Attacker B receives the full swap output without spending any ETH or WETH. The loss is bounded only by the amount of ETH stranded on the router, which can be arbitrarily large if User A sends a large `msg.value`.

### Likelihood Explanation
Medium. Requires User A to omit `refundETH` from their multicall — a common mistake when composing native ETH swaps. The exploit is then trivially executable by any unprivileged caller in a subsequent transaction. No special permissions, no malicious setup, no non-standard tokens required.

### Recommendation
Do not use the router's ambient native ETH balance to satisfy a payment obligation attributed to an external `payer`. Instead, only use `msg.value` (tracked explicitly per-call) for native ETH payments. One approach: track the ETH consumed from `msg.value` in transient storage and only allow `pay()` to use that tracked amount, not `address(this).balance`. Alternatively, require callers to always include `refundETH` by enforcing that `address(this).balance == 0` at the end of each top-level swap entry point.

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
