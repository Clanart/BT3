### Title
Unguarded `refundETH` allows any caller to steal excess ETH left on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.refundETH()` transfers the router's entire native ETH balance to `msg.sender` with no access control. Because `multicall{value: X}` leaves unspent ETH on the router when a user omits the `refundETH` step, any third party can call `refundETH()` in a subsequent transaction and receive that ETH.

### Finding Description
When a user calls `multicall{value: 1 ETH}([exactInputSingle(..., amountIn=0.5 ETH, tokenIn=WETH)])` without appending a `refundETH` call, the `pay()` function wraps only the 0.5 ETH needed for the swap and leaves the remaining 0.5 ETH as native ETH on the router contract. The `receive()` guard does not prevent this — it only blocks plain ETH transfers, not `msg.value` attached to function calls.

After the user's transaction settles, `address(router).balance == 0.5 ETH`. An attacker then calls `refundETH()` directly (or via their own multicall). The function reads `address(this).balance`, finds 0.5 ETH, and calls `_transferETH(msg.sender, balance)` — sending the full balance to the attacker with no check that the attacker was the original depositor.

### Impact Explanation
Direct loss of user principal. The victim loses the excess ETH they sent with their swap transaction. The attacker gains it. No privileged role is required; the attack is fully unprivileged and can be executed by anyone monitoring the mempool or the router's ETH balance.

### Likelihood Explanation
The intended usage pattern (include `refundETH` in every multicall that sends excess ETH) is not enforced at the contract level. Users who call `exactInputSingle` or `exactOutputSingle` directly with `msg.value > amountIn` — without wrapping in a multicall that ends with `refundETH` — are silently vulnerable. Front-running or block-watching makes this trivially exploitable.

### Recommendation
`refundETH` should only be callable within a `multicall` context and should refund to the original `msg.sender` of the outer `multicall`. One approach: record the top-level `msg.sender` in transient storage at the start of `multicall` and require `refundETH` to use that stored address rather than the immediate `msg.sender`. Alternatively, restrict `refundETH` so it can only be reached via `delegatecall` from `multicall` (e.g., check `address(this) == router` is false, which is always true under delegatecall — or use a reentrancy-style flag set only during multicall execution).

### Proof of Concept
```
1. User calls router.multicall{value: 1 ether}([
       abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ether, ...
       })))
       // refundETH omitted
   ]);
   // pay() wraps 0.5 ETH → WETH, sends to pool. 0.5 ETH remains on router.

2. Attacker calls router.refundETH();
   // balance = 0.5 ETH > 0 → _transferETH(attacker, 0.5 ETH)
   // Attacker receives 0.5 ETH. User's excess ETH is gone.
```
