### Title
Stranded ETH Stolen via Permissionless `refundETH` After Non-WETH `multicall{value}` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`multicall` is `payable` and accepts ETH via function call (bypassing the `receive()` guard). When `addLiquidityExactShares` is called for a non-WETH ERC20 pool, `pay()` uses `safeTransferFrom` and never touches `address(this).balance`. The ETH remains in the contract. `refundETH()` has no access control and sends the full balance to `msg.sender`, so any caller — including an attacker — can drain it.

### Finding Description

**Step 1 — ETH enters the contract via `multicall`.**

`multicall` is declared `public payable`: [1](#0-0) 

Because the call carries calldata, the `receive()` fallback is never invoked. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only fires on plain ETH transfers with no calldata: [2](#0-1) 

So `multicall{value: X}(...)` deposits X ETH into the contract regardless of the pool type.

**Step 2 — For a non-WETH pool, `pay()` never consumes the ETH.**

In the callback, `pay()` is called with `token0` and `token1` from the pool's immutables. For a non-WETH ERC20 pool, neither token matches `WETH`, so the `else` branch executes: [3](#0-2) 

`safeTransferFrom` pulls ERC20 tokens from the payer. `address(this).balance` is never read or decremented. The X ETH sent with the multicall remains in the contract after the call returns.

**Step 3 — `refundETH()` is permissionless.** [4](#0-3) 

There is no check that `msg.sender` is the original depositor. Any EOA or contract can call `refundETH()` and receive the full `address(this).balance`.

### Impact Explanation

Direct loss of user ETH principal. The user's X ETH is transferred to the attacker. The liquidity operation itself succeeds (ERC20 tokens are pulled correctly), so the user receives LP shares but loses the ETH they mistakenly attached to the call. This meets the "direct loss of user principal" threshold.

### Likelihood Explanation

The `multicall{value}` pattern is the documented and tested ETH-input path for WETH pools (see `test_multicall_ethInput_exactShares`). A user who copies this pattern for a non-WETH pool — or who uses a frontend/SDK that does not gate ETH attachment on pool token type — will strand ETH. An attacker monitoring the mempool can front-run the user's own `refundETH` call, or simply call it after the multicall settles.

### Recommendation

In `refundETH()`, restrict the recipient to a caller-supplied address that was recorded at multicall entry, or add a `msg.sender`-equality check against the initiating address. Alternatively, at the start of `multicall`, record `msg.sender` in transient storage and enforce it in `refundETH` for the duration of the call.

### Proof of Concept

```
// 1. User calls (no refundETH in the batch):
adder.multicall{value: 1 ether}([

// adder.balance == 1 ether; user's ERC20 LP position is created

// 2. Attacker calls:

// attacker receives 1 ether; user's ETH is gone

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
