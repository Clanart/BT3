### Title
Prior user's stranded ETH silently consumed by subsequent WETH swap in `PeripheryPayments.pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses `address(this).balance` to determine how much native ETH to wrap toward a WETH payment. This balance includes ETH left on the router by any prior caller who sent ETH with a non-WETH swap (or any payable entry point that did not consume it). A subsequent WETH swap silently wraps and forwards that stranded ETH, stealing it from its original owner.

### Finding Description
`PeripheryPayments.pay` contains a hybrid ETH/WETH branch: [1](#0-0) 

When `token == WETH` and `0 < nativeBalance < value`, the router wraps the **entire** `address(this).balance` and sends it to the pool, then pulls only `value - nativeBalance` from the payer via `safeTransferFrom`. The balance check at line 74 is not scoped to the current transaction's `msg.value`; it reflects the cumulative ETH held by the contract.

ETH can be stranded on the router because every swap entry point is `payable`: [2](#0-1) [3](#0-2) 

If a user calls `exactInputSingle{value: 0.5 ether}(...)` with `tokenIn != WETH`, the `pay` function takes the ERC20 path (line 86) and never touches `msg.value`. The 0.5 ETH remains on the router. The `receive()` guard only blocks bare ETH transfers, not ETH attached to a function call: [4](#0-3) 

### Impact Explanation
**Direct loss of user principal.** User A's stranded ETH is wrapped and forwarded to the pool as part of User B's swap payment. User A loses their ETH with no recourse; User B's WETH debit is reduced by the stolen amount. This satisfies the contest's High threshold: unprivileged attacker (User B) causes direct fund loss to another user (User A) through a standard public router entry point.

### Likelihood Explanation
Any user who sends ETH with a non-WETH swap (a natural mistake given all entry points are `payable`) creates the precondition. An attacker can monitor the mempool or router balance and immediately follow with a WETH swap sized to consume the stranded ETH. No special permissions or setup are required.

### Recommendation
Scope the native-ETH contribution in `pay` to the current transaction's `msg.value` rather than `address(this).balance`. One approach: pass `msg.value` as an explicit parameter through the call stack and use `min(msgValue, value)` as the ETH contribution, leaving any surplus for the caller's explicit `refundETH()` call. Alternatively, track per-call ETH in transient storage (consistent with the protocol's existing EIP-1153 usage) and zero it out after each top-level entry point.

### Proof of Concept
1. User A calls `exactInputSingle{value: 0.5 ether}(params)` where `params.tokenIn` is a plain ERC20 (not WETH). The `pay` function routes to `safeTransferFrom`; the 0.5 ETH is never consumed and stays on the router.
2. User B calls `exactInputSingle(params2)` where `params2.tokenIn = WETH` and `params2.amountIn = 1e18`. User B has approved the router for 1e18 WETH.
3. Inside `_justPayCallback` → `pay(WETH, userB, pool, 1e18)`:
   - `nativeBalance = address(this).balance = 0.5e18` (User A's ETH).
   - Branch `nativeBalance > 0` is taken (line 78).
   - Router wraps 0.5e18 of User A's ETH and sends it to the pool.
   - Router pulls only `1e18 - 0.5e18 = 0.5e18` WETH from User B.
4. Result: User A's 0.5 ETH is gone; User B paid 0.5e18 WETH instead of 1e18 WETH; the pool received the correct 1e18 WETH-equivalent. User A has no way to recover their ETH.
