The call chain is fully traceable. Here is the analysis:

**Step 1 — Router calls pool.swap with itself as `msg.sender`**

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly. [1](#0-0) 

**Step 2 — Pool passes `msg.sender` (the router) as `sender` to `_beforeSwap`**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, ...)`. At this point `msg.sender` is the **router's address**, not the original user's address. [2](#0-1) 

**Step 3 — Extension receives router address as `sender`**

`ExtensionCalling._beforeSwap` encodes and forwards that `sender` (router address) to the extension. [3](#0-2) 

**Step 4 — SwapAllowlistExtension checks the router, not the user**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router address. The original user's address is never consulted. [4](#0-3) 

This confirms the described path is real. The vulnerability produces **two symmetric broken invariants**:

| Pool admin action | Actual outcome |
|---|---|
| Allowlists individual users, not the router | Allowlisted users are fully blocked when routing through `MetricOmmSimpleRouter` (DoS) |
| Allowlists the router to enable router-mediated swaps | Every user — including non-allowlisted ones — can bypass the allowlist entirely |

Both are directly reachable by any unprivileged caller with no trusted-owner assumption required.

### Title
`SwapAllowlistExtension` checks router address instead of original user — allowlist DoS and bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` receives the direct caller of `pool.swap` as `sender`. When swaps are routed through `MetricOmmSimpleRouter`, that direct caller is the router contract, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, breaking the per-user gating invariant in both directions.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [5](#0-4) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [6](#0-5) 

`SwapAllowlistExtension.beforeSwap` then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
```
where `msg.sender` = pool and `sender` = router. The user's address is never read. [4](#0-3) 

`MetricOmmSimpleRouter.exactInputSingle` stores `msg.sender` only in the transient callback context (for payment), but passes nothing to the pool that would identify the original user to the extension: [1](#0-0) 

### Impact Explanation
**DoS path**: A pool admin allowlists individual users via `setAllowedToSwap(pool, user, true)`. Those users call `exactInputSingle` through the router. The extension checks `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap`. Every allowlisted user is permanently blocked from using the router, even though they are individually permitted.

**Bypass path**: To unblock router users, the admin must allowlist the router address. This grants every address — including non-allowlisted ones — the ability to bypass the allowlist by routing through `MetricOmmSimpleRouter`, defeating the extension's entire purpose.

### Likelihood Explanation
Any pool that deploys `SwapAllowlistExtension` with per-user allowlisting and expects users to interact via the canonical router hits this immediately. No special conditions, no privileged access, and no non-standard tokens are required. The router is a public, permissionless contract.

### Recommendation
Pass the original user's address through the swap call chain so extensions can gate on it. One approach: add an optional `originator` field to the pool's `swap` signature (or encode it in `callbackData`/`extensionData`) and have the router populate it with `msg.sender`. `SwapAllowlistExtension.beforeSwap` should then check `originator` (when non-zero) rather than `sender`.

Alternatively, document clearly that `sender` is always the direct caller of `pool.swap`, and require pool admins to allowlist router contracts rather than end users — but this must be prominently surfaced because it inverts the expected semantics.

### Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin allowlists `user` but NOT `router`
swapExtension.setAllowedToSwap(address(pool), user, true);

// 3. User calls router — reverts with NotAllowedToSwap
vm.prank(user);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: user,
    deadline: type(uint256).max,
    priceLimitX64: 0,
    extensionData: ""
}));
// → reverts: NotAllowedToSwap
// because extension sees sender = address(router), not address(user)

// 4. Direct pool call by user succeeds (user IS allowlisted)
vm.prank(user);
pool.swap(user, true, 1000, 0, "", "");
// → succeeds: allowedSwapper[pool][user] == true
```
