Looking at the `addLiquidityWeighted` function in `MetricOmmPoolLiquidityAdder.sol` and comparing it to the external bug's pattern (no minimum output guard on a deposit that produces a variable number of shares depending on live on-chain state):

**Key observation:** `addLiquidityWeighted` performs a two-step probe-then-pay flow entirely within one transaction. The probe reveals `need0`/`need1` at the current pool price, the scale factor `min(max0/need0, max1/need1)` is computed, and the real add executes. The number of shares minted per token deposited is entirely determined by the pool cursor at execution time. There is no `minSharesOut` (or equivalent) parameter. The `_validateBinAndBinPosition` guard only checks that the cursor lies within a user-supplied bin/position window — it does not pin the price within that window.

### Title
Lack of Minimum Shares Output in `addLiquidityWeighted` Allows Sandwich-Induced LP Share Loss — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`addLiquidityWeighted` exposes users to unbounded share-count slippage: the number of LP shares minted per token deposited depends on the live pool price at execution time, yet no minimum-shares-out guard exists. A sandwich attacker (or any price movement within the user's cursor window) can cause the user to receive materially fewer shares than simulated, constituting a direct loss of LP principal.

---

### Finding Description

`addLiquidityWeighted` operates in two on-chain steps inside a single transaction:

1. **Probe** — calls `addLiquidity` with `KIND_PROBE`; the callback always reverts with `LiquidityProbe(need0, need1)`, revealing how many token units the weight vector requires at the current pool cursor.
2. **Pay** — rescales the weight vector by `scaleWad = min(max0/need0, max1/need1)` and calls `addLiquidity` with `KIND_PAY`. [1](#0-0) 

The rescaling factor — and therefore the shares minted per token — is entirely a function of `need0`/`need1`, which are read from the pool's live price cursor at execution time. [2](#0-1) 

The only price-range guard is `_validateBinAndBinPosition`, which checks that the cursor's bin index and intra-bin position fall within `[minimalCurBin, maximalCurBin]`. It does **not** constrain the price to a narrow band within that window. [3](#0-2) 

If an attacker moves the pool price to an extreme within the user's window before the transaction lands, `need0`/`need1` shift, the scale factor drops, and the user deposits the same token amounts but receives fewer shares. No parameter in either `addLiquidityWeighted` overload allows the user to revert on insufficient shares output. [4](#0-3) 

The function returns only `(amount0Added, amount1Added)` — the token amounts consumed — not the shares minted, so the caller cannot even observe the share shortfall from the return value alone. [5](#0-4) 

---

### Impact Explanation

A user who deposits up to `maxAmountToken0`/`maxAmountToken1` tokens receives an LP position whose share count is smaller than expected. When they later remove liquidity, they recover fewer underlying tokens than they would have at the simulated price. The loss is proportional to the price deviation inside the cursor window and is directly realized as fewer tokens on withdrawal — a concrete loss of deposited principal.

---

### Likelihood Explanation

Medium. Any block-level price movement that keeps the cursor inside the user's window is sufficient — no special privilege is required. Users who set wide cursor windows to avoid spurious reverts are most exposed. MEV bots routinely perform sandwich attacks on liquidity-add transactions on EVM chains.

---

### Recommendation

Add a `minAmount0Out` / `minAmount1Out` pair (or a per-bin `minTotalShares`) to both `addLiquidityWeighted` overloads. After `_addLiquidity` returns, revert if the actual amounts (or derived shares) fall below the user-specified minimums:

```solidity
(amount0Added, amount1Added) = _addLiquidity(...);
if (amount0Added < minAmount0Out || amount1Added < minAmount1Out)
    revert InsufficientSharesOutput(amount0Added, amount1Added, minAmount0Out, minAmount1Out);
```

---

### Proof of Concept

1. Alice simulates off-chain: depositing 100 WETH + 200,000 USDC at the current price yields 1,000 shares across her chosen bins.
2. Bob (attacker) front-runs Alice's `addLiquidityWeighted` transaction, swapping to move the pool price upward while keeping the cursor inside Alice's `[minimalCurBin, maximalCurBin]` window. `_validateBinAndBinPosition` passes.
3. Alice's transaction executes: the probe now returns a higher `need0` (more WETH required per weight unit at the new price), so `scaleWad` drops. Alice deposits ~100 WETH + ~200,000 USDC but receives only ~750 shares.
4. Bob back-runs to restore the price, capturing the spread.
5. Alice's LP position is worth ~25% less than she expected; she has no recourse because no minimum-shares check exists. [6](#0-5)

  function _scaleWeightsToShares(LiquidityDelta calldata w, uint256 max0, uint256 max1, uint256 need0, uint256 need1)
    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
    uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;

    uint256 n = w.binIdxs.length;
    out.binIdxs = new int256[](n);
    out.shares = new uint256[](n);
    for (uint256 i; i < n; i++) {
      out.binIdxs[i] = w.binIdxs[i];
      out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
      if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
    }
  }
