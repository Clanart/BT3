### Title
`exactInput` multi-hop router reverts on valid partial-fill swaps due to strict `amountInActual < amount` guard — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`MetricOmmSimpleRouter.exactInput()` enforces a strict per-hop check that the pool consumed **at least** the full requested input. The MetricOmm pool is documented and test-proven to return partial fills (consuming **less** than `amountSpecified`) whenever the drift limit or liquidity boundary is reached. When this happens on any hop of a multi-hop path, the router reverts unconditionally, making the entire multi-hop exact-input path unusable under normal market conditions.

---

### Finding Description

In `exactInput`, after each pool call the router runs:

```solidity
// MetricOmmSimpleRouter.sol L114-115
int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);
```

`amount` is the full input requested for that hop (positive `int128`). `amountInActual` is the amount the pool actually consumed. The pool's own swap engine is explicitly designed to stop early and return a partial fill when the drift cap (MAX_DRIFT = 5 %) is reached or all liquidity bins are exhausted — the pool does **not** revert in those cases; it simply returns `amount1Delta < amountSpecified`. [1](#0-0) 

The pool's own test suite confirms this behaviour explicitly:

```
// metric-core/test/MetricOmmPool.swap.t.sol
test_exactInput_consumesLess_whenDriftLimitHit_token1ForToken0
  → assertLt(amount1Delta, hugeAmountIn, "Should consume less than specified when drift limit hit")

test_swap_insufficientLiquidity
  → assertLt(amount1, hugeAmountIn, "Should not use all input due to drift limit")
``` [2](#0-1) [3](#0-2) 

The same strict-equality pattern appears in the multi-hop exact-output callback:

```solidity
// MetricOmmSimpleRouter.sol L230-232
int128 amountOut = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0DeltaReturned, amount1DeltaReturned);
if (amountOut != amountToPay) revert InvalidOutputAmountAtHop(tradesLeft, amountOut, amountToPay);
``` [4](#0-3) 

Here the intermediate pool is called with exact-output semantics (`-amountToPay`). If that pool cannot fill the full amount (liquidity exhausted), it returns a partial fill, `amountOut < amountToPay`, and the strict `!=` check reverts the entire transaction.

The root cause is structurally identical to the seeded bug: an intermediate swap amount is assumed to be consumed in full, but the pool legitimately stops early, and the router treats that as an error rather than a valid partial fill.

---

### Impact Explanation

Any multi-hop `exactInput` or `exactOutput` call that routes through a pool operating near its drift cap or near liquidity exhaustion will revert unconditionally, even when the user's `amountOutMinimum` / `amountInMaximum` slippage guard would have been satisfied. The multi-hop swap path — the primary UX surface of the periphery — becomes unreliable or completely unusable under normal high-volume or low-liquidity market conditions. This matches the "unusable swap flows" impact class.

---

### Likelihood Explanation

The drift limit (MAX_DRIFT = 5 %) is a normal, expected pool behaviour triggered by any sufficiently large swap relative to the pool's liquidity. It is not an edge case: the pool's own test suite has dedicated tests for it. Any multi-hop route that passes through a pool where the first or intermediate hop is large relative to available liquidity will trigger the revert. No privileged access or malicious setup is required; an ordinary user submitting a valid multi-hop swap is sufficient.

---

### Recommendation

**For `exactInput`:** Replace the strict `amountInActual < amount` revert with acceptance of partial fills. Propagate the actual output (`extractAmountOut`) as the input to the next hop, and let the final `amountOutMinimum` guard protect the user:

```solidity
// Remove:
if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

// Keep only:
amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
```

**For `_exactOutputIterateCallback`:** Replace the strict `!=` check with a `<` check (partial fill is acceptable as long as the final output satisfies the user's minimum), or propagate the actual amount received and let the outer `InvalidOutputAmount` guard on the final hop enforce the invariant.

---

### Proof of Concept

1. Deploy two pools, each with modest liquidity (e.g., enough for a 1 000-unit swap but not a 10 000-unit swap).
2. Call `exactInput` with `amountIn = 10_000`, `amountOutMinimum = 0`, routing through both pools.
3. The first pool hits its drift limit and returns `amount1Delta = 8_000 < 10_000`.
4. The router evaluates `amountInActual (8_000) < amount (10_000)` → `true` → reverts with `InvalidInputAmountAtHop(0, 8000, 10000)`.
5. The swap fails even though the user set `amountOutMinimum = 0` and would have accepted any output.

This mirrors the seeded bug exactly: `interim_usdc_out ≠ interim_usdc_in` → `Error::InterimSwapNotEq` in Superposition; `amountInActual < amount` → `InvalidInputAmountAtHop` in Metric OMM. [5](#0-4)
