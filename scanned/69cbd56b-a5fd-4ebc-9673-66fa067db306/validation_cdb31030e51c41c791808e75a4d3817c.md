### Title
Unprotected `refundETH()`, `unwrapWETH9()`, and `sweepToken()` Allow Any Caller to Steal ETH and Tokens Stranded on the Router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.refundETH()`, `unwrapWETH9()`, and `sweepToken()` are public/external with no access control and no caller attribution. The internal `pay()` function, when settling a WETH swap, consumes only the exact swap amount from the router's native ETH balance and leaves any excess as stranded ETH. Any attacker can immediately call `refundETH()` in a separate transaction to drain that stranded ETH to themselves.

---

### Finding Description

`PeripheryPayments.pay()` handles WETH payment by first consuming the router's native ETH balance:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L73-84
} else if (token == WETH) {
  uint256 nativeBalance = address(this).balance;
  if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
  } else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
  } else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
  }
```

When `nativeBalance >= value`, exactly `value` wei is deposited and forwarded; the remainder `nativeBalance - value` is left as native ETH on the router with no attribution to the original sender.

The three public helpers that drain the router's balance have no access control and no caller-identity check:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L58-63
function refundETH() external payable override {
  uint256 balance = address(this).balance;
  if (balance > 0) {
    _transferETH(msg.sender, balance);   // sends to whoever calls, not the original payer
  }
}
```

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
  uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
  ...
  if (balanceWETH > 0) {
    IWETH9(WETH).withdraw(balanceWETH);
    _transferETH(recipient, balanceWETH);  // recipient is attacker-controlled
  }
}
```

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
  uint256 balanceToken = IERC20(token).balanceOf(address(this));
  ...
  if (balanceToken > 0) {
    IERC20(token).safeTransfer(recipient, balanceToken);  // recipient is attacker-controlled
  }
}
```

---

### Impact Explanation

**Direct loss of user ETH principal.** A user who calls `exactInputSingle{value: X}(amountIn: Y)` directly (not via multicall) where `X > Y` will have `X - Y` wei stranded on the router after the swap callback settles. Any attacker can immediately call `refundETH()` in the next transaction and receive the full stranded balance. The user has no recourse.

The same applies to `unwrapWETH9` and `sweepToken`: any WETH or ERC-20 tokens stranded on the router (e.g., from a token→WETH swap where `recipient: address(router)` was set but the unwrap step was omitted) can be swept to an attacker-controlled address by any caller.

---

### Likelihood Explanation

- **Trigger is fully unprivileged**: any EOA or contract can call `refundETH()`, `unwrapWETH9()`, or `sweepToken()` at any time.
- **Stranding is easy**: a user calling any payable swap function directly with `msg.value > amountIn` (a natural mistake when using WETH as input) will strand the excess. The `pay()` function silently leaves the remainder on the router.
- **No front-running required**: the attacker simply monitors the mempool for transactions that leave ETH on the router and calls `refundETH()` in the next block.
- **No special setup**: the attacker needs no approvals, no position, and no privileged role.

---

### Recommendation

1. **`refundETH()`**: Record the original `msg.sender` of the outermost multicall in transient storage and restrict `refundETH()` to only refund to that recorded address, or require it to be called only within a multicall context.

2. **`unwrapWETH9()` / `sweepToken()`**: These are intentionally caller-specified-recipient helpers for use inside a multicall. Add a transient-storage guard that ensures they can only be called within an active multicall initiated by the same transaction, or at minimum document that they must never be called outside a multicall and add a reentrancy/context check.

3. **`pay()` excess ETH**: After settling the WETH payment, if `nativeBalance > value`, immediately refund `nativeBalance - value` back to the payer rather than leaving it on the router.

### Proof of Concept

```
Setup:
  - Router deployed with WETH address
  - Pool exists for WETH/Token1

Attack:
  1. User calls:
       router.exactInputSingle{value: 2 ether}(
         ExactInputSingleParams({
           tokenIn: WETH,
           amountIn: 1000,   // only 1000 wei needed
           ...
         })
       )

  2. Inside the swap callback, pay(WETH, user, pool, 1000) is called:
       nativeBalance = 2 ether  (from msg.value)
       value         = 1000
       → deposits 1000 wei as WETH, sends to pool
       → 2 ether - 1000 wei remains as native ETH on router

  3. exactInputSingle() returns. Router holds 2 ether - 1000 wei.

  4. Attacker (any address) calls:
       router.refundETH()
       → address(router).balance = 2 ether - 1000 wei
       → _transferETH(attacker, 2 ether - 1000 wei)

  Result: User loses 2 ether - 1000 wei. Attacker gains it.
```
