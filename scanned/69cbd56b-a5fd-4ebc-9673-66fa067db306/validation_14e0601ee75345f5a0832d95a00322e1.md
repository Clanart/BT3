### Title
Unguarded `unwrapWETH9` Allows Any Caller to Steal All Router-Held WETH as ETH â€” (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.unwrapWETH9` is `public payable` with no access control. It reads the router's entire WETH balance, withdraws it via `IWETH9.withdraw`, and forwards the resulting ETH to a **caller-supplied** `recipient`. Any address can call it at any time with `amountMinimum = 0` and an attacker-controlled `recipient`, draining whatever WETH the router currently holds.

### Finding Description

`unwrapWETH9` contains no `msg.sender` check, no ownership guard, and no per-user accounting:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // recipient is fully attacker-controlled
    }
}
``` [1](#0-0) 

The intended usage pattern is `multicall([exactInputSingle(..., recipient: address(router)), unwrapWETH9(0, user)])` â€” WETH is transiently deposited to the router within one atomic transaction and immediately unwrapped. However, the function is exposed as a standalone public entry point with no atomicity enforcement.

WETH can realistically be stranded on the router via:

1. **Direct ERC20 transfer**: `weth.transfer(address(router), amount)` succeeds â€” the `receive()` guard only blocks raw ETH, not ERC20 transfers.
2. **Swap with `recipient: address(router)` without a paired `unwrapWETH9`**: A user who calls `exactInputSingle` directly (outside `multicall`) with `recipient = address(router)` leaves WETH on the router permanently.
3. **Failed or partial multicall**: If a multicall reverts after the swap step but before `unwrapWETH9`, the WETH remains on the router.

In all three cases, any attacker can immediately call `unwrapWETH9(0, attacker)` and receive the full balance as ETH.

The same structural issue applies to `sweepToken`, which is also `public payable` with no access control and sweeps the router's entire balance of any ERC20 to a caller-specified recipient. [2](#0-1) 

### Impact Explanation
Direct theft of user principal. Any WETH (or ERC20 via `sweepToken`) that reaches the router â€” whether by user error, partial multicall failure, or deliberate deposit â€” can be fully drained by an unprivileged attacker to an arbitrary address. There is no minimum amount; `amountMinimum = 0` bypasses the only guard.

### Likelihood Explanation
The intended multicall pattern is atomic and safe in the happy path. However, the attack surface is permanently open: any non-atomic interaction (direct swap call, ERC20 transfer to router, or failed multicall) creates a drainable balance. A MEV bot monitoring the mempool or router balance can exploit this with a single call at negligible cost.

### Recommendation
Restrict `unwrapWETH9` and `sweepToken` so that only `msg.sender` can be the `recipient`, or add a `onlyAuthorized`/`onlySelf` modifier so these helpers can only be invoked via `delegatecall` from `multicall` (not as standalone external calls). The simplest fix:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    require(recipient == msg.sender, "recipient must be caller");
    // ... rest unchanged
}
```

Alternatively, enforce that `recipient` is always `msg.sender` at the interface level, matching the security model of the `refundETH` function which already hard-codes `msg.sender` as the beneficiary. [3](#0-2)
