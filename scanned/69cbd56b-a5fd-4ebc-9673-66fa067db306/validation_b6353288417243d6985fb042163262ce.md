### Title
Synthetic Two-Feed Ratio in `AnchoredPriceProvider` Combines Legs Without Cross-Timestamp Validation, Enabling Bad-Price Execution — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider._getBidAndAskPrice()` supports a synthetic ratio mode (`quoteFeedId != 0`) that divides `price(baseFeedId)` by `price(quoteFeedId)` to produce a cross-rate (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). Each leg is individually staleness-checked against `MAX_REF_STALENESS`, but the two legs' `refTime` values are never compared to each other. The two feeds can therefore be up to `MAX_REF_STALENESS` apart in time, producing a synthetic mid that is materially wrong during volatile periods. Because the band is anchored to this wrong mid, the clamp does not correct the error. Swaps execute at the corrupted bid/ask, causing direct trader fund loss.

---

### Finding Description

In `_getBidAndAskPrice()`:

```solidity
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
if (!ok) return (0, type(uint128).max);

bytes32 _quote = quoteFeedId;
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
    if (!ok2 || mid2 == 0) return (0, type(uint128).max);
    mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // ratio from two independent timestamps
    spreadBps += spreadBps2;
}
return _computeBidAsk(mid, spreadBps);
``` [1](#0-0) 

`_readLeg` returns `refTime` as its third value, but the caller discards it (`,` placeholder). Each leg is individually gated by `MAX_REF_STALENESS` inside `_readLeg`:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [2](#0-1) 

No code ever computes `|refTime_base − refTime_quote|` or enforces a maximum inter-leg skew. `MAX_REF_STALENESS` is an immutable that can be set up to 7 days:

```solidity
if (_maxRefStaleness > 7 days) revert MaxRefStalenessOutOfBounds();
MAX_REF_STALENESS = _maxRefStaleness;
``` [3](#0-2) 

The band clamp in `_computeBidAsk` is derived from the already-corrupted `mid`:

```solidity
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
``` [4](#0-3) 

Because the band is centered on the wrong mid, clamping does not restore correctness — it only prevents the custom source from diverging further from the already-wrong anchor.

---

### Impact Explanation

The pool calls `getBidAndAskPrice()` at swap time and executes every bin step at the returned bid/ask. A corrupted synthetic ratio shifts the entire bid/ask band by the same percentage error. Traders who swap against the pool receive or pay tokens at the wrong rate:

- **Overpriced synthetic** (base feed fresh, quote feed stale from a period when the quote asset was cheaper): traders buying the base asset overpay in quote tokens — direct loss of quote tokens proportional to the price error.
- **Underpriced synthetic** (base feed stale, quote feed fresh): traders selling the base asset receive fewer quote tokens than fair value — direct loss.

The loss magnitude equals `(|T_base − T_quote| / MAX_REF_STALENESS) × price_volatility_of_quote_asset`. For a 1-hour `MAX_REF_STALENESS` and 5 % ETH hourly volatility, a trader swapping $100 k of BTC/ETH loses ~$5 k. This is a direct, per-swap loss of principal, matching the "bad-price execution" impact gate.

---

### Likelihood Explanation

The trigger is fully unprivileged: any user who calls `exactInputSingle` or `exactOutputSingle` through the router while the two feeds happen to have different `refTime` values (the normal case — feeds update independently) is affected. No special setup is required. The condition is persistent whenever the two underlying oracles (e.g., Pyth Lazer for BTC/USD and Chainlink Data Streams for ETH/USD) update at different cadences, which is their standard operating mode.

---

### Recommendation

After both legs pass their individual staleness checks, enforce a maximum inter-leg timestamp skew:

```solidity
uint256 MAX_LEG_SKEW = 60; // e.g., 60 seconds, set as immutable
...
(uint256 mid,  uint256 spreadBps,  uint256 refTime1, bool ok)  = _readLeg(baseFeedId);
(uint256 mid2, uint256 spreadBps2, uint256 refTime2, bool ok2) = _readLeg(_quote);
if (!ok || !ok2 || mid2 == 0) return (0, type(uint128).max);

uint256 skew = refTime1 > refTime2 ? refTime1 - refTime2 : refTime2 - refTime1;
if (skew > MAX_LEG_SKEW) return (0, type(uint128).max);
```

This mirrors the fix recommended in the Ion Protocol report and ensures the synthetic ratio is only computed from contemporaneous prices.

### Proof of Concept

**Setup**: `AnchoredPriceProvider` deployed with `baseFeedId = BTC/USD`, `quoteFeedId = ETH/USD`, `MAX_REF_STALENESS = 3600` (1 hour).

**State at swap time**:
- `price(BTC/USD)` → mid = 60 000e8, refTime = block.timestamp (fresh)
- `price(ETH/USD)` → mid = 2 000e8, refTime = block.timestamp − 3 599 (1 second before staleness cutoff, passes `_isStale`)

**Expected fair BTC/ETH mid**: 60 000 / 2 000 = 30.0

**Actual ETH price 3 599 seconds ago** (assume ETH was 10 % lower): mid2_actual_now = 2 200e8, but the stale feed returns 2 000e8.

**Computed synthetic mid**: 60 000e8 × 1e8 / 2 000e8 = 30.0 (using stale ETH price)

**True mid**: 60 000 / 2 200 ≈ 27.27

**Error**: ~10 % overpricing of BTC/ETH. A trader buying 10 BTC pays 300 ETH instead of 272.7 ETH — a loss of 27.3 ETH (~$60 k at current prices) in a single swap, with no oracle manipulation required, only the normal lag between two independently-updating feeds.

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
