### Title
Missing Zero-Address Validation on `recipient` in `unwrapWETH9` Permanently Burns User ETH - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

### Summary
`unwrapWETH9` accepts a caller-supplied `recipient` address and forwards all unwrapped ETH to it via a low-level `.call`. No guard prevents `recipient == address(0)`. On the EVM, `address(0).call{value: v}("")` succeeds and returns `(true, "")`, so the ETH is silently and permanently burned with no revert.

### Finding Description
`unwrapWETH9` in `PeripheryPayments` is the standard post-swap step for ETH-output flows. After a token→WETH swap whose output is directed to the router, the user calls `unwrapWETH9(amountMinimum, recipient)` (typically inside a `multicall`) to convert the router's WETH balance to native ETH and forward it. [1](#0-0) 

The function performs no validation on `recipient` before forwarding funds: [2](#0-1) 

`_transferETH` executes `to.call{value: value}("")`. When `to == address(0)`, the EVM treats this as a successful call (returns `true`), so `ok` is `true`, the `ETHTransferFailed` revert is never triggered, and the ETH is permanently destroyed. A grep across all periphery contracts confirms there is no `recipient != address(0)` guard anywhere in the call path.

The same structural gap exists in `sweepToken`, but standard ERC20 implementations (including OpenZeppelin) revert on `transfer(address(0), ...)`, so that path is self-protecting for in-scope tokens. The ETH path has no such backstop.

### Impact Explanation
Any ETH that the router holds at the time `unwrapWETH9` is called with `recipient == address(0)` is permanently burned. In the canonical multicall pattern — `exactInput*(…, recipient: address(router))` followed by `unwrapWETH9(0, address(0))` — the entire swap output is lost. This is a direct, irreversible loss of user principal with no recovery path.

### Likelihood Explanation
The `recipient` field is a plain `address` parameter supplied by the caller. A user who mistakenly passes `address(0)` (e.g., a zero-initialized variable in an integration, a UI bug, or a copy-paste error) loses all funds in that multicall with no on-chain protection. The function is `public payable` and callable by any EOA or contract. No privileged role or malicious setup is required — ordinary user error is sufficient.

### Recommendation
Add a zero-address guard at the top of `unwrapWETH9` (and symmetrically in `sweepToken` for defence-in-depth):

```solidity
error ZeroRecipient();

function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    if (recipient == address(0)) revert ZeroRecipient();   // ← add this
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
```

Apply the same guard to `sweepToken`.

### Proof of Concept

1. Pool is seeded with token1/WETH liquidity.
2. User approves the router for token1.
3. User calls `router.multicall([...])` with two sub-calls:
   - `exactInputSingle({tokenIn: token1, tokenOut: WETH, recipient: address(router), amountIn: 3000, ...})`
   - `unwrapWETH9(0, address(0))`
4. The first sub-call succeeds; the router now holds `X` WETH.
5. `unwrapWETH9` withdraws `X` WETH → router holds `X` ETH.
6. `_transferETH(address(0), X)` executes `address(0).call{value: X}("")` → returns `(true, "")`.
7. `ok == true`, no revert. `X` ETH is permanently burned.
8. User receives zero ETH; `address(router).balance == 0`. [1](#0-0) [2](#0-1)
