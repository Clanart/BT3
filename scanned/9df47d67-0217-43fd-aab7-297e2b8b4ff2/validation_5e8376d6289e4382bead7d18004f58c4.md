### Title
Stranded ETH on the router is silently consumed to partially fund a subsequent user's WETH input payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay` reads `address(this).balance` — the **entire** router ETH balance — when deciding how much native ETH to use for a WETH payment. It contains no accounting of which ETH belongs to which caller. Any ETH left on the router from a prior transaction is silently consumed to partially fund a later user's WETH input, causing the prior user to lose their ETH and the later user to underpay.

---

### Finding Description

`pay()` contains the following branch for WETH payments from an external payer: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}(); // ← consumes ALL stranded ETH
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` is the **global** router balance, not the ETH attributable to the current call. ETH can be stranded on the router across transaction boundaries when a user sends excess `msg.value` in a payable call and omits `refundETH()`. The `receive()` guard only blocks direct ETH pushes; it does not prevent `msg.value` from accumulating across separate transactions. [2](#0-1) 

**Attack path (cross-transaction):**

1. **User A** calls `exactInputSingle` (WETH tokenIn, `amountIn = 100`) with `msg.value = 150`. The callback fires `pay(WETH, A, pool, 100)`. Since `nativeBalance (150) >= value (100)`, exactly 100 ETH is deposited and transferred; **50 ETH remains stranded** on the router. User A does not call `refundETH()`.

2. **User B** calls `exactInputSingle` (WETH tokenIn, `amountIn = 200`) with `msg.value = 0`. The callback fires `pay(WETH, B, pool, 200)`. `nativeBalance = 50 > 0`, so the router deposits User A's 50 ETH as WETH, transfers it to the pool, then pulls only 150 WETH from User B via `safeTransferFrom`.

Result: User A loses 50 ETH; User B pays 150 WETH instead of 200 WETH.

**Attack path (same multicall, different logical users):**

`multicall` is `payable` and uses `delegatecall`, so `msg.value` is shared across all steps. [3](#0-2) 

An attacker can craft a multicall where step 1 intentionally strands ETH (e.g., a native-ETH swap that leaves residual) and step 2 is a WETH `exactInputSingle` for a different payer address, consuming the residual ETH to reduce the step-2 payer's cost.

---

### Impact Explanation

Direct loss of user principal. A user who sends excess ETH to the router and omits `refundETH()` permanently loses that ETH to the next caller who performs a WETH input swap. The router provides no per-caller ETH accounting, so the theft is silent and requires no privileged access.

---

### Likelihood Explanation

- Users routinely send excess `msg.value` when the exact WETH input amount is not known ahead of time, relying on `refundETH()` as a cleanup step.
- Omitting `refundETH()` is a common integration mistake.
- An attacker can monitor the router's ETH balance on-chain and immediately submit a WETH `exactInputSingle` to drain it.
- No special permissions, malicious pool setup, or non-standard tokens are required.

---

### Recommendation

Track the ETH attributable to the current call rather than using the global balance. One approach: snapshot `address(this).balance` at the start of each top-level payable entry point (storing it in transient storage), and in `pay()` use only `min(snapshot, value)` of native ETH, not the full `address(this).balance`. Alternatively, require that WETH payments always use `safeTransferFrom` and handle native-ETH wrapping only via an explicit `wrapETH` step that the caller opts into with a known amount.

### Proof of Concept

```solidity
// Foundry test sketch
function test_strandedEthConsumedByLaterWethSwap() public {
    // User A sends 150 ETH, swaps 100 WETH worth, leaves 50 ETH stranded
    vm.deal(userA, 150 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 150 ether}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 100 ether, ...})
    );
    // userA forgets refundETH(); 50 ETH sits on router
    assertEq(address(router).balance, 50 ether);

    // User B swaps 200 WETH, sends no ETH
    uint256 wethBefore = IERC20(WETH).balanceOf(userB);
    vm.prank(userB);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 200 ether, ...})
    );
    uint256 wethSpent = wethBefore - IERC20(WETH).balanceOf(userB);

    // User B only spent 150 WETH; 50 ETH from User A subsidized the rest
    assertEq(wethSpent, 150 ether);          // underpaid by 50
    assertEq(address(router).balance, 0);    // User A's ETH is gone
}
```
