### Title
Missing intermediate-token validation in `exactInput` allows draining router's stranded ERC20 balances — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

### Summary
`exactInput` sets the callback payment token to `params.tokens[i]` for each hop without verifying that `tokens[i]` equals the actual output token of `pools[i-1]`. An attacker can supply a mismatched `tokens[1]` pointing to any token the router holds, causing the router to drain that token to an attacker-chosen pool while the legitimate intermediate token is stranded.

### Finding Description
In `exactInput`, for hop `i ≥ 1`, the transient callback context is set as:

```solidity
_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, address(this), params.tokens[i]);
```

`params.tokens[i]` is attacker-controlled. When pool `i` calls `metricOmmSwapCallback`, `_justPayCallback` executes:

```solidity
pay(_getTokenToPay(), _getPayer(), msg.sender, amount);
// → IERC20(params.tokens[i]).safeTransfer(pool_i, amount)
```

No code validates that `params.tokens[i]` equals the output token of `params.pools[i-1]`. The attacker selects `pools[1]` to be a factory pool whose input token is `wrongIntermediateToken`, so the pool's balance check passes. The router's balance of `wrongIntermediateToken` is transferred to pool1; the correct intermediate token from hop 0 is stranded in the router.

### Impact Explanation
Any ERC20 token balance held by the router (stranded from failed multi-hop swaps, accidental transfers, or dust from prior swaps) can be drained by any caller via a crafted `exactInput` path. The `sweepToken` function's existence confirms the router is expected to hold residual balances. Loss is bounded by the router's current balance of the targeted token.

### Likelihood Explanation
The router transiently holds intermediate tokens during multi-hop swaps. Any failed transaction (e.g., slippage revert after hop 0 completes) leaves tokens stranded. The attack requires only: (a) the router holds some balance of token X, and (b) a factory pool exists with token X as input — both are realistic conditions on any active deployment.

### Recommendation
After each hop `i ≥ 1`, validate that `params.tokens[i]` equals the actual output token of `params.pools[i-1]`. Concretely, query the pool's token pair and assert the direction matches `tokens[i]`:

```solidity
// After hop i-1 completes, before setting callback context for hop i:
require(
    MetricOmmSwapPath.resolveOutputToken(params.pools[i-1], params.zeroForOneBitMap, i-1)
        == params.tokens[i],
    "token mismatch at hop"
);
```

Alternatively, derive the full path from pool token pairs rather than accepting `tokens[]` as user input.

### Proof of Concept
