### Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User-Approved Tokens via Callback - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

### Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address from the caller without verifying it against the factory registry. A malicious pool contract can exploit the `metricOmmModifyLiquidityCallback` settlement path to pull any ERC-20 tokens the user has approved to the adder, up to the user-supplied `maxAmountToken0`/`maxAmountToken1` caps, and redirect them to itself.

### Finding Description

`MetricOmmSimpleRouter` validates every pool address against the factory before storing it in transient context:

```solidity
// MetricOmmSwapRouterBase.sol L82-85
function _requireExpectedCallbackCaller(address caller) internal view {
    TransientCallbackPool.requireCaller(caller);
    if (!FACTORY.isPool(caller)) revert IMetricOmmSimpleRouter.InvalidPool(caller);
}
``` [1](#0-0) 

`MetricOmmPoolLiquidityAdder` performs no such check. Its own NatSpec acknowledges this explicitly:

> "This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement." [2](#0-1) 

`_addLiquidity` stores the caller-supplied `pool` address directly into transient context and then calls `addLiquidity` on it: [3](#0-2) 

When the pool calls back `metricOmmModifyLiquidityCallback`, the callback:
1. Accepts `msg.sender` as the expected pool (it matches because it was stored without factory validation).
2. Fetches token addresses by calling `IMetricOmmPool(msg.sender).getImmutables()` — fully controlled by the malicious pool.
3. Calls `pay(token0, payer, msg.sender, amount0Delta)` and `pay(token1, payer, msg.sender, amount1Delta)`, pulling tokens from the user (`payer = msg.sender` of the original call) to the malicious pool. [4](#0-3) 

The `pay` function performs no additional validation — it issues `safeTransferFrom(payer, recipient, value)` directly: [5](#0-4) 

The malicious pool controls:
- Which tokens are reported as `token0`/`token1` via `getImmutables()`.
- The exact `amount0Delta`/`amount1Delta` values passed to the callback (up to the user-supplied caps).

### Impact Explanation

A user who has approved `MetricOmmPoolLiquidityAdder` for USDC and WETH (or any other tokens) and is tricked into calling `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of `token0` and `maxAmountToken1` of `token1` — both chosen by the attacker — in a single transaction. The tokens are transferred directly to the attacker-controlled pool contract. This is a direct, complete loss of user principal with no recovery path.

### Likelihood Explanation

The trigger requires the user to supply a malicious pool address. This is achievable via:
- A phishing frontend that substitutes a legitimate pool address with a malicious one.
- Social engineering (e.g., "add liquidity to this new pool").
- Any integration that forwards a user-supplied pool address to the adder.

The `addLiquidityWeighted` variant additionally calls `_validateBinAndBinPosition` on the malicious pool before the probe, but the malicious pool can return any slot0 values to pass that check. [6](#0-5) 

### Recommendation

Add a factory registry check in `_addLiquidity` (or at each public entry point) mirroring the pattern already used in `MetricOmmSwapRouterBase`:

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires storing the factory address as an immutable in `MetricOmmPoolLiquidityAdder`, consistent with `MetricOmmSwapRouterBase`.

### Proof of Concept

```solidity
// MaliciousPool.sol
contract MaliciousPool {
    address immutable attacker;
    address immutable victimToken; // e.g. USDC

    constructor(address _attacker, address _victimToken) {
        attacker = _attacker;
        victimToken = _victimToken;
    }

    // Returns attacker-chosen tokens
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = victimToken;
        imm.token1 = victimToken; // or any second token
    }

    // Called by LiquidityAdder._addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta memory, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Call back with KIND_PAY and max amounts
        IMetricOmmModifyLiquidityCallback(msg.sender)
            .metricOmmModifyLiquidityCallback(
                1_000_000e6, // amount0Delta = maxAmountToken0 (USDC)
                0,
                abi.encode(uint8(1)) // KIND_PAY
            );
        return (1_000_000e6, 0);
    }
}

// Attack
// 1. Victim approves LiquidityAdder for 1_000_000 USDC
// 2. Attacker tricks victim into calling:
liquidityAdder.addLiquidityExactShares(
    address(maliciousPool),
    victim,
    0,
    deltas,
    1_000_000e6, // maxAmountToken0
    0,           // maxAmountToken1
    ""
);
// Result: 1_000_000 USDC transferred from victim to maliciousPool
