### Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale-Price Liquidity Addition — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` exposes four public liquidity-entry points — two overloads of `addLiquidityExactShares` and two of `addLiquidityWeighted` — none of which accept or enforce a deadline. The router (`MetricOmmSimpleRouter`) correctly calls `_checkDeadline(params.deadline)` at the top of every swap entry point, but the liquidity adder has no equivalent guard. A pending transaction can sit in the mempool and execute arbitrarily late, at an oracle price that has moved significantly from the user's intent.

---

### Finding Description

`MetricOmmSimpleRouter` enforces a deadline on every user-facing swap: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`MetricOmmPoolLiquidityAdder` has no such check on any of its four entry points: [5](#0-4) [6](#0-5) 

The `addLiquidityWeighted` path does call `_validateBinAndBinPosition`, which checks the pool cursor against caller-supplied bounds: [7](#0-6) 

However, this only rejects execution when the cursor has moved *outside* the user's stated range. If the oracle price moves within that range — or if the user supplied wide bounds — the transaction executes at the new, stale price with no recourse.

For `addLiquidityExactShares`, there is no cursor-bounds check at all; the only protection is the `maxAmountToken0`/`maxAmountToken1` caps, which limit *how much* is spent but do not prevent liquidity from being placed at the wrong oracle price.

---

### Impact Explanation

Metric OMM is oracle-anchored: bin prices are determined by the external oracle at execution time, not by reserve ratios. A user who submits a liquidity transaction expecting the current oracle price may have that transaction mined minutes or hours later at a materially different price. Their shares are minted into bins that are now off-market, producing immediate impermanent loss proportional to the oracle price movement during the delay. The `maxAmount` caps do not protect against this: they cap token spend, not price exposure.

---

### Likelihood Explanation

Any period of mempool congestion, gas-price spike, or deliberate MEV withholding can delay a liquidity transaction. This is a standard attack surface for any DeFi periphery contract that lacks a deadline. The attacker does not need any special privilege; they only need to observe a pending transaction and wait for a favorable oracle move before including it.

---

### Recommendation

Add a `uint256 deadline` parameter to all four `addLiquidity*` entry points and call the same `_checkDeadline` helper used by the router at the top of each function, before any state reads or pool calls. This mirrors the pattern already established in `MetricOmmSimpleRouter`.

---

### Proof of Concept

1. Alice calls `addLiquidityWeighted` with `minimalCurBin = -5`, `maximalCurBin = 5`, and `maxAmountToken0 = 1000 USDC`, `maxAmountToken1 = 1 ETH` when ETH oracle price is \$2 000.
2. The transaction is not mined immediately (gas too low, mempool congestion).
3. The ETH oracle price rises to \$2 500 within the same block range.
4. The transaction is mined. `_validateBinAndBinPosition` passes because the cursor is still within `[-5, 5]`.
5. The probe runs at \$2 500; `_scaleWeightsToShares` computes shares based on the new price.
6. Alice's liquidity is added at \$2 500 bins. She immediately holds a position worth less in ETH terms than she intended, with no ability to have prevented the execution. [8](#0-7)

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
