### Title
`exactInputSingle` Missing Zero-Output Guard Allows Silent Input Token Loss When `amountOutMinimum = 0` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`exactInputSingle` lacks the explicit `amountOut > 0` guard that its sibling `exactInput` enforces. When the pool's fixed-point oracle math rounds a dust-sized input to zero output, the router accepts the result silently if the caller supplied `amountOutMinimum = 0`, consuming the user's input tokens while delivering nothing.

---

### Finding Description

`exactInput` (multi-hop) contains an explicit guard after each hop:

```solidity
// MetricOmmSimpleRouter.sol line 120
if (amount <= 0) revert InvalidSwapDeltas();
``` [1](#0-0) 

`exactInputSingle` (single-hop) has no equivalent check. Its only post-swap guard is the caller-supplied minimum:

```solidity
// MetricOmmSimpleRouter.sol lines 81-83
int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
amountOut = MetricOmmSwapInputs.int128ToUint128(out);
if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);
``` [2](#0-1) 

When `amountOutMinimum = 0` (an explicitly supported value per the interface) and the pool's Q64.64 oracle math rounds the output to zero, the evaluation `0 < 0` is `false` — no revert occurs.

The callback guard does not save the user either:

```solidity
// MetricOmmSimpleRouter.sol line 47
if (amount0Delta <= 0 && amount1Delta <= 0) revert InvalidSwapDeltas();
``` [3](#0-2) 

This only reverts when **both** deltas are non-positive. A swap where the pool collects a positive input delta (`amount0Delta > 0`) but delivers zero output (`amount1Delta == 0`) passes the check, the callback pays the pool, and the transaction completes successfully with the user holding zero output tokens.

The README explicitly states rounding favors the pool (output rounds down), making zero-output for dust inputs a design-level possibility, not a theoretical edge case. [4](#0-3) 

---

### Impact Explanation

A user who calls `exactInputSingle` with a small `amountIn` and `amountOutMinimum = 0` loses their entire input amount with no output received. The pool's balance increases (it received the input via the callback payment path), the recipient receives nothing, and the transaction emits no revert. This is a direct, silent loss of user principal with no recovery path.

---

### Likelihood Explanation

`amountOutMinimum = 0` is a valid, documented parameter value. Integrators, aggregators, and users testing swaps routinely omit slippage protection. The condition is reachable by any unprivileged caller with no special setup. The asymmetry between `exactInput` (guarded) and `exactInputSingle` (unguarded) indicates the developers recognized the zero-output scenario as real — the guard was simply omitted from the single-hop path.

---

### Recommendation

Add an explicit zero-output guard in `exactInputSingle`, mirroring the guard already present in `exactInput`:

```solidity
int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
if (out <= 0) revert InvalidSwapDeltas();          // ← add this
amountOut = MetricOmmSwapInputs.int128ToUint128(out);
if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);
```

---

### Proof of Concept

1. Deploy a pool with an oracle price such that 1 wei of `tokenIn` maps to 0 wei of `tokenOut` under Q64.64 rounding (e.g., price >> 1, input = 1).
2. Approve the router for `tokenIn`.
3. Call:
   ```solidity
   router.exactInputSingle(ExactInputSingleParams({
       pool: pool,
       tokenIn: tokenIn,
       tokenOut: tokenOut,
       zeroForOne: true,
       amountIn: 1,           // dust amount
       amountOutMinimum: 0,   // no slippage guard
       recipient: attacker,
       deadline: block.timestamp + 1,
       priceLimitX64: 0,
       extensionData: ""
   }));
   ```
4. Observe: `tokenIn` balance of caller decreases by 1, `tokenOut` balance of recipient unchanged, transaction succeeds.
5. The pool's `tokenIn` reserve increased; the user's funds are permanently lost. [5](#0-4)

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
