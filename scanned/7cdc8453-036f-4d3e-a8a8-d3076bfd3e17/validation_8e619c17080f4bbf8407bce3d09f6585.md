### Title
Pool Pause Mechanism Ineffective for `addLiquidity` and `removeLiquidity` — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool` defines a `whenNotPaused` modifier and correctly applies it to `swap`, but omits it from `addLiquidity` and `removeLiquidity`. When the pool is paused (by the pool admin at level 1 or by the protocol owner at level 2), users can still add and remove liquidity, defeating the entire purpose of the pause mechanism for those flows.

### Finding Description

`MetricOmmPool` stores a `pauseLevel` (0 = active, 1 = admin-paused, 2 = protocol-paused) and exposes a `whenNotPaused` modifier that reverts when `pauseLevel != 0`. [1](#0-0) 

The `swap` function correctly carries this modifier: [2](#0-1) 

However, `addLiquidity` and `removeLiquidity` do not: [3](#0-2) [4](#0-3) 

The factory's `pausePool` and `protocolPausePool` entry points set `pauseLevel` to 1 and 2 respectively, but neither has any effect on the liquidity paths: [5](#0-4) 

### Impact Explanation

The pause mechanism is the primary emergency brake for the protocol. It is intended to freeze all user-facing pool activity when an oracle is compromised, a price anomaly is detected, or a security incident is in progress. Because `addLiquidity` and `removeLiquidity` bypass the pause:

1. **LP drain during emergency**: When the pool is paused because the oracle is returning manipulated prices, LPs can still call `removeLiquidity`. The removal is settled against the current (potentially corrupted) bin balances, so LPs who race to exit first may receive disproportionate token amounts relative to their share, leaving remaining LPs with a shortfall — a direct loss of LP principal.

2. **Liquidity injection into a compromised pool**: Users can still call `addLiquidity` against a paused pool. Tokens are pulled from the user via the callback and deposited into a pool whose state may be under active attack. When the pool is later unpaused and the oracle corrected, the bin cursor may have shifted, causing the newly added liquidity to be worth less than what was deposited.

3. **Broken invariant for protocol-level pause**: The protocol owner's `protocolPausePool` (level 2) is specifically designed to override even the pool admin and freeze the pool completely. The missing modifier means this stronger pause level is equally ineffective for liquidity operations.

### Likelihood Explanation

The pause mechanism is explicitly designed for emergency use. Any time the admin or protocol owner triggers a pause — a realistic and expected operational event — both `addLiquidity` and `removeLiquidity` remain fully open to any caller. No special privileges or unusual conditions are required to exploit this; any LP or user can call these functions during a pause.

### Recommendation

Add the `whenNotPaused` modifier to both `addLiquidity` and `removeLiquidity`:

```diff
  function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
- ) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
+ ) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {

  function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
+   whenNotPaused
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
  {
```

### Proof of Concept

1. Deploy a pool with a mutable price provider.
2. Pool admin calls `factory.pausePool(pool)` → `pauseLevel` becomes 1.
3. Confirm `pool.swap(...)` reverts with `PoolPaused`.
4. Call `pool.addLiquidity(...)` directly — it succeeds, tokens are pulled from the caller.
5. Call `pool.removeLiquidity(...)` directly — it succeeds, tokens are returned to the caller.
6. The pause has no effect on either liquidity function, confirming the invariant is broken.
