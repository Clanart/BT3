### Title
Excess native ETH sent to `exactOutputSingle` / `exactInputSingle` is permanently stranded on the router and claimable by any caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`exactOutputSingle` and `exactInputSingle` are `payable` functions. When a user sends `msg.value > actualAmountIn` (the standard pattern for exact-output swaps), the `pay()` helper wraps only the exact amount owed to the pool and leaves the remainder as raw ETH on the router. Neither swap function refunds the surplus. Any subsequent caller of the permissionless `refundETH()` function drains the entire router ETH balance, stealing the user's excess.

### Finding Description

`PeripheryPayments.pay()` handles native-ETH-as-WETH input as follows: [1](#0-0) 

When `nativeBalance >= value`, only `value` wei is wrapped and forwarded to the pool. The surplus `nativeBalance - value` is never returned to the caller. Neither `exactOutputSingle` nor `exactInputSingle` contains a post-swap refund step: [2](#0-1) 

The `refundETH()` helper exists but is permissionless — it sends the **entire** router ETH balance to `msg.sender`, not to the original depositor: [3](#0-2) 

### Impact Explanation

A user who calls `exactOutputSingle{value: amountInMaximum}(...)` with `tokenIn = WETH` loses `amountInMaximum - actualAmountIn` ETH. The stranded ETH sits on the router until any third party (e.g., a MEV searcher watching the mempool) calls `refundETH()` and receives it. This is a direct, permanent loss of user principal with no recovery path for the victim.

The same loss applies to `exactInput{value: X}` and `exactOutput{value: X}` whenever `X > actualAmountIn`.

### Likelihood Explanation

The exact-output pattern universally requires sending `amountInMaximum` upfront. Users who call `exactOutputSingle` directly (without wrapping in a `multicall` that appends `refundETH()`) — a natural and common integration pattern — will always lose the surplus. The functions are `payable` and accept ETH without restriction, giving no indication that the caller must manually reclaim the change.

### Recommendation

Add an automatic ETH refund at the end of each `payable` swap entry point, or revert if `msg.value` exceeds the amount actually consumed:

```solidity
// In exactOutputSingle, after amountIn is known:
if (tokenIn == WETH) {
    uint256 refund = msg.value - amountIn;
    if (refund > 0) _transferETH(msg.sender, refund);
}
```

Alternatively, enforce `require(msg.value == 0 || tokenIn == WETH)` and add a post-swap `refundETH` call inside the function body so callers cannot accidentally strand ETH.

### Proof of Concept

1. Pool is `WETH/TOKEN1`. Current price requires `1000 wei` WETH to buy the desired output.
2. User calls `exactOutputSingle{value: 5000}(params)` with `amountInMaximum = 5000`, `tokenIn = WETH`.
3. Pool triggers `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, user, pool, 1000)`.
4. Inside `pay`: `nativeBalance = 5000 >= value = 1000` → wraps and sends exactly `1000` wei WETH to pool. [4](#0-3) 
5. `exactOutputSingle` returns `amountIn = 1000`. The remaining `4000 wei` sits on the router. [5](#0-4) 
6. MEV bot calls `refundETH()` in the next block → receives `4000 wei`. User's `4000 wei` is permanently lost. [3](#0-2)

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
