### Title
Excess native ETH stranded by `pay()` is stealable by any caller via unguarded `refundETH()` — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

---

### Summary

When a user calls `exactOutputSingle` or `exactInputSingle` with `msg.value` greater than the actual swap input required, the `pay()` function deposits only the exact required amount as WETH and silently leaves the remainder on the router. Because `refundETH()` is an unguarded public function that sends the **entire** router ETH balance to `msg.sender`, any attacker can steal the stranded ETH in a subsequent transaction.

---

### Finding Description

The `pay()` function handles native ETH payments for WETH-input swaps by inspecting `address(this).balance` at callback time:

```solidity
// PeripheryPayments.sol lines 73-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    }
```

When `nativeBalance >= value`, it deposits exactly `value` ETH as WETH and forwards it to the pool. The surplus `nativeBalance - value` ETH is left on the router with no automatic refund and no attribution to the original sender. [1](#0-0) 

The recovery function `refundETH()` is `external` with zero access control:

```solidity
// PeripheryPayments.sol lines 58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

It sends the **entire** router ETH balance to `msg.sender`, regardless of who deposited it. Any caller in any subsequent transaction can drain all stranded ETH. [2](#0-1) 

The swap entry points `exactOutputSingle` and `exactInputSingle` are both `payable` and perform no automatic refund after the callback settles: [3](#0-2) 

The intended safe pattern — wrapping the swap and `refundETH()` together in a `multicall` — is demonstrated in tests but is **not enforced** by the contract. A user who calls `exactOutputSingle` directly (e.g., from a simple wallet, an aggregator, or any non-multicall integration) will strand the excess ETH with no protection. [4](#0-3) 

---

### Impact Explanation

A user who calls `exactOutputSingle{value: X}` where `X > amountIn` loses `X - amountIn` ETH permanently to any attacker who calls `refundETH()` in the next block. The loss is proportional to the user's ETH buffer — a common and reasonable practice for exact-output swaps where the precise input is not known in advance. This is a direct loss of user principal with no recovery path for the victim.

---

### Likelihood Explanation

Medium-High. Exact-output swaps with a native ETH buffer are a standard DeFi UX pattern. Any user or integrator that calls `exactOutputSingle` or `exactOutput` directly (outside a multicall) will strand ETH. An attacker needs only to watch the mempool for transactions that leave a non-zero ETH balance on the router and immediately call `refundETH()` in the next block. No special privileges, approvals, or setup are required.

---

### Recommendation

Auto-refund any remaining native ETH balance to `msg.sender` at the end of `exactOutputSingle`, `exactOutput`, and `exactInputSingle` when `tokenIn == WETH`:

```solidity
// After _clearExpectedCallbackPool():
uint256 remaining = address(this).balance;
if (remaining > 0) _transferETH(msg.sender, remaining);
```

Alternatively, track the original depositor in transient storage and restrict `refundETH()` to only refund to that address. Do **not** rely solely on the multicall pattern as a safety net, since the swap functions are independently callable and `payable`.

---

### Proof of Concept

1. User calls `exactOutputSingle{value: 2 ETH}` with `tokenIn = WETH`, `amountInMaximum = 2 ETH`, and the pool requires only `1 ETH` as actual input.
2. Inside `metricOmmSwapCallback`, `_justPayCallback` calls `pay(WETH, user, pool, 1 ETH)`.
3. `pay()` sees `nativeBalance = 2 ETH >= value = 1 ETH`, deposits 1 ETH as WETH, and sends it to the pool. The remaining 1 ETH stays on the router.
4. `exactOutputSingle` returns; no refund is issued. Router holds 1 ETH.
5. Attacker calls `refundETH()` in the next transaction. `refundETH()` sends `address(this).balance = 1 ETH` to the attacker.
6. User has paid 2 ETH but received only the swap output worth 1 ETH of input. Net loss: 1 ETH. [5](#0-4) [6](#0-5)

    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();

        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
