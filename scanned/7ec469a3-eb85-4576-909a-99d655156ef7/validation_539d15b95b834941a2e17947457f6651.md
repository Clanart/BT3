### Title
`setPoolAdminFeeDestination` Redirects Already-Accrued Admin Fees to New Destination Without Prior Collection — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-parameter setter in the factory calls `collectFees` before mutating state; this one does not. All fees earned under the old destination are silently redirected to the new one on the next collection, causing a direct, quantifiable loss for the old fee recipient.

---

### Finding Description

The factory enforces a consistent pattern: before any fee-split parameter is changed, the current accrued balance is collected using the **old** parameters so that each party receives exactly what it earned.

`setPoolProtocolFee` collects first: [1](#0-0) 

`setPoolAdminFees` also collects first: [2](#0-1) 

`setPoolAdminFeeDestination` does **not**: [3](#0-2) 

After the destination is overwritten, every subsequent call to `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee` passes the **new** `poolAdminFeeDestination[pool]` to `collectFees`: [4](#0-3) 

Inside `collectFees`, the admin share of both spread-surplus and notional-fee accumulators is transferred to whatever `adminFeeDestination_` is supplied at call time: [5](#0-4) 

The notional accumulators (`notionalFeeToken0Scaled`, `notionalFeeToken1Scaled`) are exact on-chain balances that were earned swap-by-swap under the old destination. The spread surplus is the real token balance above `binTotals`. Both are fully redirected to the new address.

---

### Impact Explanation

The old admin fee destination — which may be a DAO treasury, a separate multisig, or any address distinct from the pool admin — permanently loses all fees it earned before the destination change. The loss is bounded only by how long the pool has been running and how large its fee rate is; for an active pool with a 1 % notional fee and significant volume, this can be material. The tokens are not destroyed; they are transferred to the new destination, so the new address receives a windfall it did not earn.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that can call `setPoolAdminFeeDestination` at any time without a timelock. The admin fee destination is explicitly documented as a separate treasury address ("Coordinate with treasury operations"). Any admin rotation, treasury migration, or malicious pool admin can trigger the loss. No special market conditions are required.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** stored config and the **old** destination before overwriting `poolAdminFeeDestination[pool]`.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before changing it.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool runs for N blocks; `notionalFeeToken0Scaled = 500 USDC-scaled`, spread surplus = 300 USDC-scaled. Old destination = `treasury_A`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, treasury_B)`. No fees are collected; `poolAdminFeeDestination[pool]` is overwritten.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `treasury_B` and passes it to `collectFees`.
4. `collectFees` transfers the admin share of both the 500 and 300 units to `treasury_B`.
5. `treasury_A` receives nothing despite having earned those fees. `treasury_B` receives a windfall it did not earn.
