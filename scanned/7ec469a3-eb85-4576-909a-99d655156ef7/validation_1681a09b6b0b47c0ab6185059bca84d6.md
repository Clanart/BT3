### Title
Public Unauthenticated `sweepToken` and `unwrapWETH9` Allow Any Caller to Drain Router-Held Balances to an Arbitrary Recipient — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`sweepToken` and `unwrapWETH9` in `PeripheryPayments` are `public payable` with zero access control and accept a fully attacker-controlled `recipient` address. Any caller can invoke them at any time and redirect the router's entire ERC-20 or WETH balance to themselves. This is the direct structural analog of the external report's `rescue`/`blacklist` pattern: critical fund-moving functions are public and unauthenticated in the base contract, relying entirely on the assumption that callers will only use them inside an atomic `multicall`. That assumption is not enforced on-chain.

---

### Finding Description

`PeripheryPayments` exposes two public helpers:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L37-L55
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // ← attacker-supplied recipient
    }
}

function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);  // ← attacker-supplied recipient
    }
}
```

Neither function checks `msg.sender`, requires a role, or validates that `recipient` is the original depositor. They transfer the router's **entire** balance of the target asset unconditionally.

The router accumulates real token balances in two production paths:

1. **Multi-hop `exactInput`**: intermediate output tokens are sent to `address(this)` between hops (`i == last ? params.recipient : address(this)`). If a user submits the swap as a standalone transaction with `recipient = address(this)` (intending to follow up with `unwrapWETH9` or `sweepToken`), the tokens sit on the router until the next call.

2. **WETH payment with excess `msg.value`**: the `pay` helper deposits exactly `value` ETH as WETH and forwards it to the pool; any `msg.value` surplus remains on the router as native ETH. A user who does not include `refundETH` in the same multicall leaves ETH on the router, and `refundETH` itself sends to `msg.sender` — meaning an attacker who calls it first claims the ETH.

In both cases, a single unprivileged external call to `sweepToken(token, 0, attacker)` or `unwrapWETH9(0, attacker)` drains the full balance to the attacker.

---

### Impact Explanation

Any ERC-20 or WETH balance that reaches the router — through a standalone swap with `recipient = address(this)`, through a multicall that omits the cleanup step, or through any other legitimate deposit path — can be stolen in full by an unprivileged attacker in a single transaction. There is no minimum amount, no time lock, and no authentication. The loss is direct and immediate.

---

### Likelihood Explanation

The intended usage pattern (atomic `multicall` containing swap + sweep/unwrap) is safe, but it is not enforced. Users who:
- submit a swap with `recipient = address(this)` as a standalone transaction, or
- compose a multicall that omits the cleanup step (e.g., due to a UI bug, partial revert handling, or manual construction)

leave funds on the router that any observer can immediately claim. MEV bots routinely monitor router balances and can back-run any transaction that leaves residue.

---

### Recommendation

Mirror the fix recommended in the external report: make `sweepToken` and `unwrapWETH9` `internal`, and expose them only through a `public` wrapper that enforces `recipient == msg.sender` (or an equivalent caller-binding check). This ensures that only the address that initiated the multicall can direct where the swept value goes, eliminating the arbitrary-recipient theft vector while preserving the intended cleanup functionality.

```solidity
// Suggested replacement in PeripheryPayments
function sweepToken(address token, uint256 amountMinimum) public payable {
    _sweepToken(token, amountMinimum, msg.sender);
}

function unwrapWETH9(uint256 amountMinimum) public payable {
    _unwrapWETH9(amountMinimum, msg.sender);
}
```
