### Title
`addLiquidity` and `removeLiquidity` bypass the emergency pause — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap` is guarded by `whenNotPaused`, but `addLiquidity` and `removeLiquidity` carry no such guard. When the pool is paused (e.g., because the oracle is compromised or a price anomaly is detected), swaps are correctly blocked, yet liquidity deposits and withdrawals continue to execute, undermining the entire purpose of the emergency pause.

---

### Finding Description

`MetricOmmPool` defines a `whenNotPaused` modifier and a `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused): [1](#0-0) [2](#0-1) 

`swap` correctly applies the modifier: [3](#0-2) 

But `addLiquidity` and `removeLiquidity` do not: [4](#0-3) [5](#0-4) 

The periphery `MetricOmmPoolLiquidityAdder` routes directly to `pool.addLiquidity`, so it inherits the same gap: [6](#0-5) 

---

### Impact Explanation

**Scenario — oracle compromise triggers pause, LP funds are lost on unpause:**

1. The oracle begins returning a manipulated price. The protocol or admin sets `pauseLevel = 1` or `2` to halt swaps.
2. Because `addLiquidity` is not gated, an LP (or an attacker acting as LP) deposits tokens into the paused pool via `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` or `addLiquidityWeighted`. The pool accepts the deposit and mints shares.
3. When the pause is lifted, swaps resume against the (still-bad or newly-reset) oracle price. The freshly deposited LP principal is immediately exposed to adversarial swaps at the bad price, draining the LP's funds.

**Scenario — selective LP exit during pause:**

1. Pool is paused due to an in-progress exploit that has moved the bin cursor to an unfavorable position for remaining LPs.
2. An informed LP calls `removeLiquidity` (which is not blocked) and withdraws their share of tokens before other LPs can react.
3. Remaining LPs are left holding a pool with a skewed cursor and reduced liquidity depth, suffering a disproportionate loss when the pool is unpaused and swaps resume.

Both scenarios represent direct loss of LP principal — the first through forced exposure to bad-price swaps, the second through inequitable exit during an emergency.

---

### Likelihood Explanation

- The pause mechanism is explicitly designed for emergencies (oracle failure, price anomaly, exploit in progress). These are real operational events.
- `addLiquidity` is callable by any address (no `msg.sender == owner` restriction), and `removeLiquidity` is callable by any position owner.
- The periphery `MetricOmmPoolLiquidityAdder` is the standard user-facing entry point and passes calls through without any pause check of its own.
- No additional privilege is required; any user can trigger the gap.

---

### Recommendation

Add `whenNotPaused` to both `addLiquidity` and `removeLiquidity` in `MetricOmmPool.sol`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) { ... }

function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (...) { ... }
```

If allowing LPs to exit during a pause is an intentional design choice for `removeLiquidity`, that decision should be explicitly documented and the asymmetry with `addLiquidity` should still be resolved.

---

### Proof of Concept

```
1. Deploy pool with a mutable price provider.
2. Call pool.setPause(1) via factory (admin pause).
3. Attempt pool.swap(...) → reverts with PoolPaused. ✓
4. Call MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, ...) →
   succeeds, tokens transferred into paused pool. ✗
5. Call pool.removeLiquidity(owner, salt, deltas, "") →
   succeeds, tokens transferred out of paused pool. ✗
6. Call pool.setPause(0) (unpause).
7. Attacker swaps against the pool at the oracle price that caused the pause →
   LP funds deposited in step 4 are drained.
```

Steps 4 and 5 demonstrate that the pause is not enforced on liquidity operations, directly contradicting the invariant that a paused pool halts all state-changing token flows.
