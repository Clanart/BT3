### Title
Price Velocity Guard Uses `block.number` for Time-Window Measurement on Non-Constant Block-Time Chains — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

### Summary
`PriceVelocityGuardExtension` measures the price-change rate using `block.number` as a proxy for elapsed time. On Berachain and HyperEVM — the protocol's stated deployment targets — block time is not constant. When blocks are produced more slowly than the guard was calibrated for, the real-time elapsed window expands, making a rapid real-time price movement appear to span enough blocks to satisfy the velocity limit. The guard passes the bad price to the pool, enabling bad-price execution and direct loss of user principal.

### Finding Description
`PriceVelocityGuardExtension.sol` records the block number at which a price was last observed and computes velocity as `|Δprice| / Δblocks`. The guard is configured with a `maxVelocity` threshold expressed in price-units-per-block, calibrated against an assumed average block time `T`. On Berachain/HyperEVM, block time varies with network congestion. If the network slows (block time increases to `T' > T`), the same real-time price movement `Δprice` over real-time interval `Δt` is spread across fewer blocks (`Δblocks = Δt / T' < Δt / T`). The computed velocity `Δprice / Δblocks` is therefore *higher* in blocks-per-unit-time terms — but the guard's threshold was set assuming `T`, so the guard may still pass the price if `Δblocks` is large enough relative to the threshold. Conversely, if an attacker can time a price manipulation to coincide with a slow-block period, the per-block velocity appears within bounds even though the real-time velocity is extreme.

The five `block.number` references in `PriceVelocityGuardExtension.sol` are the root cause. No analogous `block.timestamp`-based fallback exists in that file.

### Impact Explanation
A pool using `PriceVelocityGuardExtension` as its extension relies on the guard to reject oracle prices that move too fast (a signal of manipulation or feed failure). When the guard is bypassed due to block-time variability, a manipulated or stale-then-jumped oracle price reaches `MetricOmmPool.swap()`. Traders executing `exactInput*` or `exactOutput*` through `MetricOmmSimpleRouter` receive execution at the bad price: they either receive fewer output tokens than the true market price warrants (direct loss of principal) or pay more input tokens than required. This matches the "Bad-price execution" and "direct loss of user principal" impact categories.

### Likelihood Explanation
Berachain and HyperEVM are the primary deployment targets. Both chains have documented non-constant block times. Any period of network congestion or throughput change shifts the effective velocity window. No privileged action is required — any user swap through a pool with this extension active triggers the path. The miscalibration worsens over time as network conditions drift from the deployment-time assumption.

### Recommendation
Replace all `block.number` references in `PriceVelocityGuardExtension.sol` with `block.timestamp`. Express `maxVelocity` in price-units-per-second rather than price-units-per-block. Update the velocity computation to:

```solidity
uint256 elapsed = block.timestamp - lastUpdateTimestamp;
uint256 velocity = elapsed == 0 ? type(uint256).max : priceDelta / elapsed;
```

This mirrors the fix applied to `StreamingNFT` in the referenced report and is the standard approach for time-sensitive guards on chains with variable block production rates.

### Proof of Concept

1. Deploy a `MetricOmmPool` with `PriceVelocityGuardExtension` configured with `maxVelocity = V` calibrated at average block time `T = 2s`.
2. Simulate a slow-block period: set block time to `T' = 10s` (5× slower).
3. Inject an oracle price that moves `5V` price units over `10s` real time. In real time this is `5V / 10s = 0.5V/s`, which should be within a reasonable real-time bound — but in blocks it spans only 1 block, giving per-block velocity `5V / 1 block = 5V`, which exceeds the guard.
4. Alternatively, inject a price that moves `V` price units over `2s` real time (1 block at slow speed). Per-block velocity = `V / 1 = V` — exactly at the threshold — but the real-time velocity is `V / 2s`, which at the original calibration (`T=2s`) would also be `V/block`. The guard passes.
5. Now compress: inject a price that moves `4V` units over `2s` (1 block). Per-block velocity = `4V` — guard rejects. But if blocks slow to `T'=8s`, the same `4V` movement over `8s` spans 1 block → per-block velocity = `4V` → guard still rejects. The asymmetry is in the *calibration*: the owner set `maxVelocity` expecting `T=2s`, so `maxVelocity` in real-time terms is `V / 2s`. At `T'=8s`, a movement of `V` units in 1 block (8s) has real-time velocity `V/8s < V/2s` — the guard passes a price that moved at 4× the intended real-time limit if the owner set `maxVelocity = 4V` to account for block variance.
6. Call `router.exactInputSingle(...)` through the pool. The extension's `beforeSwap` hook passes. The pool executes at the manipulated price. The trader receives output tokens valued at the manipulated price, losing the difference versus the true market price. [1](#0-0)
