### Title
Fee-on-Transfer Token Accounting Discrepancy Enables Undercollateralization and Fund Drainage - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

The EVM `initTransfer` function trusts the caller-supplied `amount` parameter when emitting the `InitTransfer` event, without verifying the actual token balance change after `safeTransferFrom`. For fee-on-transfer ERC20 tokens, the bridge locks fewer tokens than the amount recorded in the event. NEAR's `fin_transfer_callback` then mints the full event-stated amount, creating unbacked supply and breaking bridge collateralization.

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native ERC20 tokens (non-bridge, non-custom-minter) as follows:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

Immediately after, the event is emitted using the caller-supplied `amount`, not the actual balance delta:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-specified, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

On the NEAR side, `fin_transfer_callback` decodes the prover result (which reflects the on-chain event) and mints the full event-stated amount:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

There is no balance-before/balance-after check anywhere in `initTransfer`. [4](#0-3) 

Token registration is permissionless: `logMetadata` on EVM accepts any `tokenAddress` with no whitelist, and `deploy_token` on NEAR is similarly open to any caller when unpaused. [5](#0-4) 

### Impact Explanation

For a fee-on-transfer token with a `k%` fee:

- Bridge locks `amount × (1 - k%)` EVM tokens.
- NEAR mints `amount` tokens.
- The bridge is immediately undercollateralized by `amount × k%` per transfer.

When a user bridges back the full NEAR-minted amount, the EVM bridge attempts `safeTransfer(recipient, amount)` but only holds `amount × (1 - k%)`. If other users have deposited the same token, the shortfall is covered by their deposits — i.e., the attacker drains other users' locked funds. This is a direct, concrete loss of bridged assets matching the **High** allowed impact: *balance/accounting corruption that breaks bridge collateralization*.

### Likelihood Explanation

- `logMetadata` and `deploy_token` are permissionless, so any attacker can register a fee-on-transfer token without admin involvement.
- Fee-on-transfer tokens exist in production (e.g., tokens with transfer taxes, some rebasing tokens).
- The attack requires no privileged access, no leaked keys, and no oracle manipulation.
- The only prerequisite is that the same token has been deposited by other users (to drain), or the attacker is willing to accept unbacked NEAR-side tokens that can partially redeem against the pool.

### Recommendation

Record the actual balance change rather than trusting the caller-supplied `amount`:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
```

Alternatively, explicitly document and enforce a token whitelist that excludes fee-on-transfer tokens.

### Proof of Concept

1. Attacker deploys `FeeToken` — a standard ERC20 with a 10% transfer fee.
2. Attacker calls `OmniBridge.logMetadata(FeeToken)` on EVM (permissionless). Event `LogMetadata` is emitted.
3. Attacker submits proof to NEAR `deploy_token` — NEAR-side `FeeToken` wrapper is deployed.
4. Victim deposits 1,000 `FeeToken` via `initTransfer(FeeToken, 1000, ...)`:
   - `safeTransferFrom` transfers 1,000; bridge receives 900 (10% fee taken by token).
   - `InitTransfer` event records `amount = 1000`.
5. NEAR `fin_transfer_callback` mints 1,000 NEAR-side `FeeToken` to victim.
6. Attacker calls `initTransfer(FeeToken, 1000, ...)`:
   - Bridge receives 900; event records 1,000.
   - NEAR mints 1,000 to attacker.
7. Attacker bridges back 1,000 NEAR-side tokens → EVM bridge calls `safeTransfer(attacker, 1000)`.
   - Bridge holds 1,800 total (900 from victim + 900 from attacker).
   - Attacker receives 1,000 EVM tokens — 100 of which came from the victim's deposit.
8. Victim attempts to bridge back 1,000 NEAR tokens → bridge only has 800 left → transfer fails or victim receives less than deposited.

Net result: attacker extracted 100 `FeeToken` from the victim's locked balance with no privileged access. [1](#0-0)
