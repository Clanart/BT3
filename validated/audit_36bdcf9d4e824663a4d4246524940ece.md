### Title
Rejected L1 Handler Transactions Permanently Lost During L1 Provider Catch-Up — (`crates/apollo_l1_provider/src/l1_provider.rs`)

---

### Summary

When the L1 provider enters its catch-up phase, the `rejected_txs` argument is silently dropped before reaching `apply_commit_block`. Every L1 handler transaction that failed execution during a caught-up block is therefore marked `Committed` instead of `Rejected`, permanently preventing re-proposal. The developers left a `TODO` acknowledging the gap. This is the sequencer-native analog of the PufferVault "slash mid-withdrawal" bug: a two-step process (propose → commit) has an intermediate state that is not correctly reconciled when an external event (catch-up) interrupts the normal path, leaving the accounting permanently wrong.

---

### Finding Description

**Normal path** — `commit_block` → `apply_commit_block` with both sets:

```rust
fn apply_commit_block(
    &mut self,
    consumed_txs: IndexSet<TransactionHash>,
    rejected_txs: IndexSet<TransactionHash>,
) {
    let (rejected_and_consumed, committed_txs): (Vec<_>, Vec<_>) =
        consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));
    self.tx_manager.commit_txs(&committed_txs, &rejected_and_consumed);
    ...
}
``` [1](#0-0) 

The batcher always sends the full `consumed_l1_handler_tx_hashes` (which includes failed/rejected L1 handlers) together with the `rejected_l1_handler_tx_hashes` subset:

```rust
let rejected_l1_handler_tx_hashes = rejected_tx_hashes
    .iter()
    .copied()
    .filter(|tx_hash| consumed_l1_handler_tx_hashes.contains(tx_hash))
    .collect();

self.l1_provider_client
    .commit_block(consumed_l1_handler_tx_hashes, rejected_l1_handler_tx_hashes, height)
    .await;
``` [2](#0-1) 

**Catch-up path** — `commit_block` routes to `accept_commit_while_catching_up`, which receives only `committed_txs` (the full consumed set) and **drops `rejected_txs` entirely**:

```rust
if self.state.is_catching_up() {
    return self.accept_commit_while_catching_up(committed_txs, height);
}
``` [3](#0-2) 

Inside `accept_commit_while_catching_up`, the call to `apply_commit_block` passes an empty `IndexSet` for `rejected_txs`:

```rust
// TODO(guyn): check what about rejected txs here and in the backlog?
Equal => self.apply_commit_block(committed_txs, Default::default()),
``` [4](#0-3) 

Because `rejected_txs` is empty, the partition in `apply_commit_block` places every consumed L1 handler transaction — including those that failed execution — into `committed_txs`, and `commit_txs` marks them all `Committed`:

```rust
for &tx_hash in committed_txs {
    self.create_record_if_not_exist(tx_hash);
    self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
}
``` [5](#0-4) 

The same defect applies to the backlog: `add_commit_block_to_backlog` stores only `committed_txs`, so backlogged blocks processed after catch-up completes also lose the rejected-tx distinction. [6](#0-5) 

---

### Impact Explanation

The intended behavior is explicit in the test suite: a rejected L1 handler transaction must remain in `Rejected` state and return `ValidationStatus::Validated` so it can be re-proposed in a later block.

```rust
#[case::validate_rejected_transaction(tx_hash!(1), ValidationStatus::Validated)]
``` [7](#0-6) 

After the catch-up bug fires, the same transaction is in `Committed` state and `validate_tx` returns `AlreadyIncludedOnL2`:

```rust
TransactionState::Committed => {
    InvalidValidationStatus::AlreadyIncludedOnL2.into()
}
``` [8](#0-7) 

The transaction is permanently removed from the `proposable_index` and will never be included in any future block. The corresponding L1 message is silently lost: the L1 contract recorded it as sent, but L2 will never successfully execute it. This is a wrong L1-message / wrong sequencer state outcome that matches the **High** impact category: *"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."*

---

### Likelihood Explanation

Catch-up is triggered whenever the L1 provider's `current_height` does not match the height sent by the batcher — a routine occurrence at node startup, after a crash-restart, or when the batcher races ahead. Any block that (a) contains a rejected L1 handler transaction and (b) is processed during catch-up will silently corrupt the provider's state. No privileged access is required; the condition arises from normal operational sequencing.

---

### Recommendation

Pass `rejected_txs` through the catch-up path:

1. Change `accept_commit_while_catching_up` to accept a second `rejected_txs: IndexSet<TransactionHash>` parameter.
2. Forward it from `commit_block` at the call site.
3. Pass it to `apply_commit_block` instead of `Default::default()`.
4. Store `rejected_txs` alongside `committed_txs` in the `commit_block_backlog` so that backlogged blocks are also corrected when the backlog is drained.

---

### Proof of Concept

1. Start the L1 provider at height `N`. The batcher is already at height `N+2`.
2. Batcher calls `commit_block(consumed=[tx_A, tx_B], rejected=[tx_A], height=N+2)`. The provider's height is `N`, so it enters `CatchingUp` state.
3. Batcher calls `commit_block(consumed=[tx_A, tx_B], rejected=[tx_A], height=N)` (the block at the provider's current height). `accept_commit_while_catching_up` is reached; `rejected_txs` is dropped; `apply_commit_block([tx_A, tx_B], [])` is called.
4. Both `tx_A` and `tx_B` are marked `Committed`. `tx_A` should be `Rejected`.
5. Provider transitions to `Pending`. Future `validate(tx_A)` returns `AlreadyIncludedOnL2`.
6. `tx_A` is never re-proposed. The L1 message it represents is permanently lost. [9](#0-8)

    fn accept_commit_while_catching_up(
        &mut self,
        committed_txs: IndexSet<TransactionHash>,
        new_height: BlockNumber,
    ) -> L1ProviderResult<()> {
        let current_height = self.current_height;
        debug!(
            "Catchupper processing commit-block at height: {new_height}, current height is \
             {current_height}"
        );
        match new_height.cmp(&current_height) {
            // This is likely a bug in the batcher/sync, it should never be _behind_ the provider.
            Less => {
                // TODO(guyn): check if this is reliable: old blocks can have txs that were
                // committed then consumed and deleted. We should probably decide to always log and
                // ignore old blocks or always return an error.
                let diff_from_already_committed: Vec<_> = committed_txs
                    .iter()
                    .copied()
                    .filter(|&tx_hash| !self.tx_manager.is_committed(tx_hash))
                    .collect();

                if diff_from_already_committed.is_empty() {
                    error!(
                        "Duplicate commit block: commit block for {new_height:?} already \
                         received, and all committed transaction hashes already known to be \
                         committed."
                    );
                    return Ok(());
                } else {
                    // This is either a configuration error or a bug in the
                    // batcher/sync/catching up code.
                    error!(
                        "Duplicate commit block: commit block for {new_height:?} already \
                         received, with DIFFERENT transaction_hashes: \
                         {diff_from_already_committed:?}"
                    );
                    Err(L1ProviderError::UnexpectedHeight {
                        expected_height: current_height,
                        got: new_height,
                    })?
                }
            }
            // TODO(guyn): check what about rejected txs here and in the backlog?
            Equal => self.apply_commit_block(committed_txs, Default::default()),
            // We're still syncing, backlog it, it'll get applied later.
            Greater => {
                self.catchupper.add_commit_block_to_backlog(committed_txs, new_height);
                // No need to check the backlog or catchup completion, since those are only
                // applicable if we just increased the provider's height, like in the `Equal` case.
                return Ok(());
            }
        };
