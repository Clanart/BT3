### Title
Unbounded Predecessor Replay in `TraceTx` Enables Unbounded Computation DoS via Public Tracing RPC — (File: `x/evm/keeper/grpc_query.go`)

---

### Summary

The `TraceTx` gRPC handler iterates over `req.Predecessors` with no size cap and no timeout, executing each predecessor as a full EVM state transition via `ApplyMessageWithConfig`. An unprivileged caller can submit a `debug_traceTransaction` JSON-RPC request carrying thousands of high-gas predecessor transactions, forcing the node into unbounded computation with no protection mechanism.

---

### Finding Description

In `TraceTx`, before tracing the target transaction, the handler replays every entry in `req.Predecessors` to reconstruct block state:

```go
// x/evm/keeper/grpc_query.go  lines 542–563
for i, tx := range req.Predecessors {
    ethTx := tx.AsTransaction()
    msg, err := core.TransactionToMessage(ethTx, signer, cfg.BaseFee)
    if err != nil {
        k.Logger(ctx).Debug("trace: skipping predecessor, ...")
        continue
    }
    ...
    rsp, err := k.ApplyMessageWithConfig(ctx, msg, cfg, true)
    if err != nil {
        k.Logger(ctx).Error("trace: predecessor replay failed, ...")
        continue
    }
    cfg.TxConfig.LogIndex += uint(len(rsp.Logs))
}
``` [1](#0-0) 

Three compounding defects are present:

**1. No bound on predecessor count.** There is no check analogous to the `MaxSimulateBlocks` guard used in `SimulateV1`. Any number of predecessors is accepted.

**2. No gas cap applied to predecessors.** `EthCall` and `EstimateGas` both clamp execution gas via `k.queryMaxGasLimit`:

```go
// grpc_query.go lines 269–276
gasCap := req.GasCap
if k.queryMaxGasLimit != GasNoLimit {
    if gasCap == 0 {
        gasCap = k.queryMaxGasLimit
    } else if k.queryMaxGasLimit < gasCap {
        gasCap = k.queryMaxGasLimit
    }
}
``` [2](#0-1) 

The predecessor loop calls `core.TransactionToMessage` directly from the raw transaction, inheriting whatever gas limit the attacker embedded in the transaction object — up to the full block gas limit — with no cap applied.

**3. The timeout only covers the final traced transaction, not the predecessor loop.** The `context.WithTimeout` is set up inside `prepareTrace`, which is called *after* the predecessor loop completes:

```go
// grpc_query.go lines 503–507
result, _, err := k.prepareTrace(ctx, cfg, msg, req.GetTraceConfig(), false)
``` [3](#0-2) 

```go
// grpc_query.go lines 787–796  (inside prepareTrace)
deadlineCtx, cancel := context.WithTimeout(ctx.Context(), timeout)
defer cancel()
go func() {
    <-deadlineCtx.Done()
    if errors.Is(deadlineCtx.Err(), context.DeadlineExceeded) {
        tracer.Stop(errors.New("execution timeout"))
    }
}()
``` [4](#0-3) 

The predecessor replay loop runs entirely outside any deadline context.

This is the direct analog of the external report: an unbounded collection (`req.Predecessors`) is iterated with O(n) full-EVM work per element, with no minimum/maximum validation, causing unbounded resource consumption.

---

### Impact Explanation

An attacker submits a `debug_traceTransaction` JSON-RPC request (which maps to `TraceTx` gRPC) with, for example, 10 000 predecessor transactions each carrying the maximum block gas limit. The node executes all 10 000 full EVM state transitions synchronously before even beginning the actual trace. This exhausts CPU and memory on the serving node.

If the targeted node is a validator running the JSON-RPC server (a common deployment pattern), the sustained CPU starvation can cause it to miss block proposals and prevote/precommit rounds, directly contributing to deterministic validator consensus failure. This matches the allowed High impact: *"Public JSON-RPC, gRPC, simulation, tracing… path… exposes a reachable route to the impacts above"* (chain halt / consensus failure).

---

### Likelihood Explanation

`debug_traceTransaction` is a standard, unauthenticated JSON-RPC endpoint exposed by every Ethermint-based node that enables the debug namespace. No special privileges, tokens, or on-chain funds are required. The attacker only needs to craft a JSON payload with a large `Predecessors` array. The attack is cheap: predecessor transactions can be minimal (e.g., simple ETH transfers) yet still force full `ApplyMessageWithConfig` execution per entry.

---

### Recommendation

1. **Cap predecessor count.** Add a guard at the top of the predecessor loop (or in the proto validation layer):
   ```go
   const maxTracePredecessors = 128
   if len(req.Predecessors) > maxTracePredecessors {
       return nil, status.Errorf(codes.InvalidArgument,
           "too many predecessors: %d > %d", len(req.Predecessors), maxTracePredecessors)
   }
   ```

2. **Apply `queryMaxGasLimit` to each predecessor message**, mirroring the cap already applied in `EthCall`/`EstimateGas`.

3. **Wrap the predecessor loop in the same deadline context** used for the final trace, so the entire `TraceTx` call is bounded by `defaultTraceTimeout` (or the caller-supplied timeout).

---

### Proof of Concept

1. Attacker constructs 10 000 signed Ethereum transactions, each with `gasLimit = blockGasLimit` (e.g., 30 000 000), targeting a deployed contract that performs heavy computation.
2. Attacker sends a single JSON-RPC call:
   ```json
   {
     "method": "debug_traceTransaction",
     "params": ["0xdeadbeef...", {"predecessors": [<tx1>, <tx2>, ..., <tx10000>]}]
   }
   ```
3. The node enters the predecessor loop in `TraceTx` and calls `ApplyMessageWithConfig` 10 000 times, each consuming up to 30 M gas worth of CPU.
4. The node becomes unresponsive for the duration of the replay. If it is a validator, it misses block proposals and consensus votes during this window.
5. The attacker can repeat the request continuously to maintain the DoS.
