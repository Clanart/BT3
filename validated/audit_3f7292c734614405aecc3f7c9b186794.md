### Title
Integer Wrapping in `check_time_locks` Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Timelocks When `nowrap=false` ŌĆö (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false`, the relative-timelock checks for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` use `wrapping_add` instead of `saturating_add`. A coin whose puzzle outputs a near-maximum relative timelock value (e.g., `height_relative = 0xFFFF_FFFF` or `seconds_relative = 0xFFFF_FFFF_FFFF_FFFF`) causes the addition to wrap around to a value smaller than the current height/timestamp, making the guard condition trivially false and the spend unconditionally accepted ŌĆö bypassing the intended timelock entirely.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces relative timelocks with two arithmetic modes controlled by the `nowrap` boolean:

```rust
// nowrap=true path (correct ŌĆö saturating)
if prev_transaction_block_height < unspent.confirmed_block_index.saturating_add(height_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}

// nowrap=false path (vulnerable ŌĆö wrapping)
} else if prev_transaction_block_height < unspent.confirmed_block_index.wrapping_add(height_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

The condition parser in `conditions.rs` accepts `ASSERT_HEIGHT_RELATIVE` arguments up to `u32::MAX` (`0xFFFF_FFFF`, encoded as the 5-byte CLVM atom `0x00FFFFFFFF`) and `ASSERT_SECONDS_RELATIVE` arguments up to `u64::MAX` (`0xFFFF_FFFF_FFFF_FFFF`, encoded as the 9-byte atom `0x00FFFFFFFFFFFFFFFF`). Both are within the `sanitize_uint` acceptance window (`max_size+1` bytes when the leading byte is `0x00` to avoid sign-bit interpretation): [2](#0-1) [3](#0-2) 

When `height_relative = 0xFFFF_FFFF` and `confirmed_block_index = C`:

```
C.wrapping_add(0xFFFF_FFFF) = C - 1   (mod 2^32, for any C > 0)
```

The guard becomes `prev_height < C - 1`. Since `prev_height >= C` always holds (the coin was confirmed at height `C`), this is always `false`, so the function returns `Ok(())` ŌĆö the spend is accepted immediately, regardless of the intended ~4-billion-block wait.

The same arithmetic applies to `seconds_relative`:

```
coin_timestamp.wrapping_add(0xFFFF_FFFF_FFFF_FFFF) = coin_timestamp - 1
``` [4](#0-3) 

The test suite explicitly documents and confirms this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- accepted when nowrap=false
)]
``` [5](#0-4) [6](#0-5) 

The function is exposed as a Python binding (`py_check_time_locks`) and registered in the wheel module, making `nowrap` a caller-supplied parameter from the Python full-node layer: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**High ŌĆö Timelock/condition validation bypass enables unauthorized spend acceptance.**

Any coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFFF)` or `ASSERT_SECONDS_RELATIVE(0xFFFF_FFFF_FFFF_FFFF)` as a security constraint (e.g., a time-locked vault, clawback coin, or any puzzle relying on a near-maximum relative delay) can be spent immediately when the full node calls `check_time_locks` with `nowrap=false`. The intended lock ŌĆö effectively "never spendable within the chain's lifetime" ŌĆö is silently nullified. If different nodes use different `nowrap` values for the same block height, they will disagree on spend validity, causing consensus divergence and a potential chain split.

---

### Likelihood Explanation

The `nowrap` parameter exists precisely because `nowrap=false` (wrapping) was the original production behavior, and `nowrap=true` (saturating) is a later correction. The Python full-node code controls which mode is used per block height. Any block range where `nowrap=false` is still applied is directly exploitable by any unprivileged spend bundle that includes a coin with a near-maximum relative timelock condition. The CLVM encoding required (`0x00FFFFFFFF` for height, `0x00FFFFFFFFFFFFFFFF` for seconds) is trivially constructable by any puzzle author.

---

### Recommendation

1. **Remove the `nowrap=false` branch entirely** once the fork height that mandates `nowrap=true` is finalized and all nodes have upgraded. Until then, document the exact block height at which `nowrap` transitions and ensure no new coins rely on large relative timelocks before that height.
2. **Alternatively**, reject `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` values that would wrap (i.e., values where `confirmed_index + relative > type_max`) during condition parsing, returning `AssertHeightRelativeFailed` / `AssertSecondsRelativeFailed` immediately, regardless of `nowrap`.
3. **Add a consensus-layer guard** in `sanitize_uint` or the condition parser to cap `height_relative` at a safe maximum (e.g., `u32::MAX / 2`) to eliminate the wrapping class of inputs entirely.

---

### Proof of Concept

**Step 1 ŌĆö Craft a CLVM puzzle** that outputs condition `(82 . (0x00FFFFFFFF))` (opcode 82 = `ASSERT_HEIGHT_RELATIVE`, argument = `0xFFFF_FFFF` encoded as 5-byte positive integer).

**Step 2 ŌĆö Create and confirm a coin** spending that puzzle at block height `C`.

**Step 3 ŌĆö Immediately attempt to spend the coin** (at block height `C` or `C+1`). The intended timelock requires waiting until height `C + 0xFFFF_FFFF Ōēł 4.3 billion`.

**Step 4 ŌĆö Call `check_time_locks` with `nowrap=false`:**

```

height_relative       = 0xFFFF_FFFF
wrapping sum          = C.wrapping_add(0xFFFF_FFFF) = C - 1
guard: prev_height (C) < C - 1  ŌåÆ  false  ŌåÆ  Ok(())  ŌåÉ spend accepted
```

**Step 5 ŌĆö With `nowrap=true` (correct behavior):**

```

guard: prev_height (C) < u32::MAX  ŌåÆ  true  ŌåÆ  Err(AssertHeightRelativeFailed)  ŌåÉ spend rejected
```

pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }
