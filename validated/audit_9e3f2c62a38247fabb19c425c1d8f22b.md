### Title
`next_l2_gas_price` and `l2_gas_consumed` Omitted from Block Hash Enables P2P Peer to Corrupt Fee-Market State — (`crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

### Summary

The `gas_prices_to_hash` function deliberately omits `l2_gas_consumed` and `next_l2_gas_price` from the block hash computation (explicit TODO at line 416). Both fields are present in `BlockHeaderWithoutHash` and transmitted in P2P `SignedBlockHeader` messages. Because they are not committed to in the block hash, a malicious P2P peer can send a `SignedBlockHeader` with a valid block hash but an attacker-controlled `next_l2_gas_price`. When a syncing sequencer node accepts this header via `try_sync`, it unconditionally sets its internal `l2_gas_price` to the peer-supplied value, corrupting the fee market for every subsequent block it builds or validates.

### Finding Description

**Root cause — missing commitment:**

`gas_prices_to_hash` in `block_hash_calculator.rs` computes the single `gas_prices_hash` field that is chained into the block hash. For Starknet ≥ 0.13.4 it hashes only `l1_gas_price`, `l1_data_gas_price`, and `l2_gas_price`:

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(...) -> Vec<Felt> {
    if block_hash_version >= &BlockHashVersion::V0_13_4 {
        vec![
            HashChain::new()
                .chain(&STARKNET_GAS_PRICES0)
                .chain(&l1_gas_price.price_in_wei.0.into())
                ...
                .chain(&l2_gas_price.price_in_fri.0.into())
                .get_poseidon_hash(),
        ]
    }
``` [1](#0-0) 

The current production version is 0.14.2 (confirmed in test fixtures), so the TODO threshold has already passed, yet both fields remain absent from the hash.

`PartialBlockHashComponents`, the struct that feeds `calculate_block_hash`, carries `l1_gas_price`, `l1_data_gas_price`, and `l2_gas_price` but has no `l2_gas_consumed` or `next_l2_gas_price` fields: [2](#0-1) 

By contrast, `BlockHeaderWithoutHash` — the authoritative in-memory header — carries both fields: [3](#0-2) 

They are also serialised into the P2P `SignedBlockHeader` protobuf (fields 18 and 19): [4](#0-3) 

And are stored in `StorageBlockHeader`: [5](#0-4) 

**Attacker-controlled data path — `try_sync`:**

When the consensus orchestrator falls behind and calls `try_sync`, it fetches a `SyncBlock` from the state-sync service (which in turn receives it from a P2P peer), then unconditionally reads `next_l2_gas_price` from the unverified field and writes it into `self.l2_gas_price`:

```rust
self.l2_gas_price = max(
    sync_block.block_header_without_hash.next_l2_gas_price,
    VersionedConstants::latest_constants().min_gas_price,
);
``` [6](#0-5) 

The only guard is a floor at `min_gas_price`; there is no upper bound and no cross-check against the deterministic fee-market formula. The block hash check that precedes this assignment passes because `next_l2_gas_price` is not part of the hash.

`self.l2_gas_price` is then propagated to every subsequent block via `update_l2_gas_price` and `calculate_next_l2_gas_price`: [7](#0-6) 

The corrupted price is also written into the blob sent to the central service: [8](#0-7) 

**Downstream economic effects:**

The gateway's `validate_resource_bounds` compares each incoming transaction's `max_price_per_unit` against the current `l2_gas_price`: [9](#0-8) 

A manipulated `l2_gas_price` therefore directly controls which transactions are admitted or rejected.

### Impact Explanation

- **Incorrect fee/gas/resource accounting with economic impact (Critical scope):** `next_l2_gas_price` is the price used to charge L2 gas fees for the next block. An attacker-supplied value causes every fee calculation, balance deduction, and resource-bound check for that block to use a wrong price.
- **Mempool/gateway admission accepts invalid or rejects valid transactions (High scope):** If the injected price is inflated, all transactions whose `max_price_per_unit` is below the inflated value are rejected by `validate_resource_bounds`, effectively halting transaction ingestion. If deflated (above `min_gas_price`), transactions that should fail the price floor are admitted.
- **RPC fee estimation returns authoritative-looking wrong value (High scope):** Fee estimation queries use the stored `l2_gas_price`; after a sync event the returned estimates are wrong for the entire next block.

### Likelihood Explanation

`try_sync` is invoked whenever the consensus context detects it is behind — a routine occurrence on restart or network partition. Any node participating in the P2P network can serve `SignedBlockHeader` messages. The block hash verification passes because `next_l2_gas_price` is absent from the hash. No additional privilege is required; the attack is reachable by any peer that can respond to a sync request before the honest peer does.

### Recommendation

1. Add `l2_gas_consumed` and `next_l2_gas_price` to `gas_prices_to_hash` and to `PartialBlockHashComponents`, as the existing TODO at line 416 already prescribes, and activate this for the current version (≥ 0.14.0).
2. As a defence-in-depth measure, after reading `next_l2_gas_price` from a synced header, recompute the expected value from `l2_gas_consumed` and the previous block's `l2_gas_price` using `calculate_next_base_gas_price` and reject the header if the values diverge beyond a tolerance.

### Proof of Concept

1. A malicious P2P peer constructs a `SignedBlockHeader` for block N. It computes the correct `block_hash` (which does not cover `next_l2_gas_price`) and sets `next_l2_gas_price` to `u128::MAX`.
2. The syncing sequencer node calls `state_sync_client.get_block(N)`. The state-sync layer verifies the block hash — which passes — and returns the header.
3. `try_sync` executes:
   ```rust
   self.l2_gas_price = max(u128::MAX, min_gas_price); // = u128::MAX
   ```
4. On the next call to `set_height_and_round`, the sequencer proposes a block with `l2_gas_price = u128::MAX`.
5. The gateway's `validate_resource_bounds` rejects every incoming transaction because no transaction can set `max_price_per_unit = u128::MAX`.
6. The sequencer produces empty blocks indefinitely; all RPC fee estimates return `u128::MAX × gas_amount`, which is economically nonsensical.

Alternatively, setting `next_l2_gas_price` to just above `min_gas_price` causes the sequencer to accept transactions at a price far below the correct market rate, enabling fee-free execution for the attacker's transactions while honest users' transactions (priced at the real market rate) are processed at a loss to the sequencer.
