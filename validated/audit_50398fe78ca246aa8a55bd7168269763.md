### Title
`verify_block_signature` Is Never Called in the Production Central Sync Path — Blocks with Arbitrary Signatures Accepted - (File: `crates/apollo_central_sync/src/lib.rs`)

### Summary

The `verify_block_signature` function defined in `crates/starknet_api/src/block.rs` is never invoked in the production central sync path. The `verify_blocks` flag in `SyncConfig` only gates `track_sequencer_public_key_changes()` — a key-rotation detector — not actual cryptographic signature verification. Consequently, any block served by the feeder gateway, including blocks carrying all-zero or forged `BlockSignature` values, is accepted and committed to storage without any ECDSA check. The deployed `state_sync_config.json` further sets `verify_blocks: false`, disabling even the key-rotation guard.

---

### Finding Description

**The dead-code gap.** `verify_block_signature` is defined at: [1](#0-0) 

It computes `poseidon_hash(block_hash, state_diff_commitment)` and delegates to `verify_message_hash_signature`. A `grep` across the entire repository shows it appears only in the definition file and in `block_test.rs` — it is never called from any production code path.

**What `verify_blocks` actually does.** In `GenericStateSync::sync_while_ok`, the flag gates only the key-change tracker: [2](#0-1) 

`track_sequencer_public_key_changes` fetches the sequencer public key and returns an error if it changes between calls: [3](#0-2) 

It never passes the fetched key to `verify_block_signature`. The `sequencer_pub_key` field on `GenericStateSync` is populated solely for change-detection, not for cryptographic verification.

**`store_block` stores without verification.** Every block that arrives via the central sync stream is committed unconditionally: [4](#0-3) 

The `signature` argument is written straight to the `block_signatures` table with no ECDSA check.

**Deployed default disables even the key-rotation guard.** The production deployment config sets: [5](#0-4) 

`"verify_blocks": false` — so `track_sequencer_public_key_changes` is never called either, and `sequencer_pub_key` remains `None` for the lifetime of the sync process.

**`BlockSignature::default()` is all-zero.** The `Signature` struct derives `Default`, giving `r = Felt::ZERO, s = Felt::ZERO`: [6](#0-5) 

A malicious feeder gateway can serve any block with `BlockSignature::default()` and it will be stored without complaint.

---

### Impact Explanation

The central sync path feeds the node's MDBX storage, which is the authoritative backing store for the RPC layer. A MITM attacker on the HTTP connection to the feeder gateway (the URL is plain HTTP-configurable, not TLS-pinned in code) can inject blocks carrying:

- A wrong `state_root` (global Patricia root) in the header
- A wrong `state_diff_commitment` (the value that should be signed)
- Wrong storage values in the accompanying state diff
- Wrong transaction or event commitments

All of these are stored and subsequently served by the JSON-RPC server as authoritative values. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The trigger requires either a MITM on the feeder gateway HTTP connection or a compromised feeder gateway endpoint. The deployed default (`verify_blocks: false`) means the node never even detects a key rotation, let alone verifies a signature. Any node running central sync in the default configuration is fully exposed.

---

### Recommendation

1. **Call `verify_block_signature` inside `store_block`** before committing to storage, using the `sequencer_pub_key` already tracked by `track_sequencer_public_key_changes`. Reject blocks whose signature does not verify.
2. **Guard `store_block` against a `None` sequencer public key** when `verify_blocks` is `true` — refuse to store any block until the key has been fetched at least once.
3. **Change the deployed default** of `verify_blocks` from `false` to `true` in `state_sync_config.json` and `replacer_state_sync_config.json`.
4. **Add a non-zero public key assertion** analogous to the EVM recommendation: before using `sequencer_pub_key` for verification, assert it is not the zero felt.

---

### Proof of Concept

```
1. Start a node with central sync enabled and the default config
   (verify_blocks: false, or even verify_blocks: true).

2. Stand up a malicious feeder gateway that returns:
   - A valid-looking block header with an arbitrary state_root
     (e.g., GlobalRoot(Felt::ONE)) and state_diff_commitment = Felt::ZERO
   - BlockSignature { r: Felt::ZERO, s: Felt::ZERO }  // default, all-zero

3. The node calls store_block(block_number, block, signature).
   No call to verify_block_signature is made.
   The block is committed to MDBX storage.

4. Query starknet_getStateUpdate or starknet_getStorageAt via RPC.
   The RPC returns the attacker-controlled state_root and storage values
   as authoritative Starknet state.
```

The root cause is that `verify_block_signature` — the direct analog of the zero-address signer check — is defined but never wired into the production sync loop, leaving the commitment invariant `stored_state_root == sequencer_signed_state_root` entirely unenforced.

/// Verifies that the the block header was signed by the expected sequencer.
pub fn verify_block_signature(
    sequencer_pub_key: &SequencerPublicKey,
    signature: &BlockSignature,
    state_diff_commitment: &GlobalRoot,
    block_hash: &BlockHash,
) -> Result<bool, BlockVerificationError> {
    let message_hash = Poseidon::hash_array(&[block_hash.0, state_diff_commitment.0]);
    verify_message_hash_signature(&message_hash, &signature.0, &sequencer_pub_key.0).map_err(
        |err| BlockVerificationError::BlockSignatureVerificationFailed {
            block_hash: *block_hash,
            error: err,
        },
    )
}
