### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash corresponds to a previously declared contract class. This is directly analogous to the siloed-asset report: just as Morpho failed to check a special asset property before listing it (enabling a state that breaks future operations), the OS fails to check class existence before replacing it, enabling a state where the contract is permanently non-functional and all held funds are frozen.

---

### Finding Description

In `execute_replace_class`, after deducting gas, the function reads `class_hash` directly from the syscall request and immediately writes it into the contract's `StateEntry` in `contract_state_changes` — with **no check** that this hash exists in the declared class trie (`contract_class_changes`).

The code itself contains an explicit acknowledgment of this gap:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

Compare this to `execute_declare_transaction`, which enforces that a class can only be declared once and that the compiled class hash is non-zero before writing to `contract_class_changes`:

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

No equivalent validation exists in `execute_replace_class`. The `class_hash` field in `StateEntry` is updated to an arbitrary felt value that may never have been declared, with no cross-reference to `contract_class_changes`. [3](#0-2) 

---

### Impact Explanation

Once a contract's `class_hash` in `contract_state_changes` is set to an undeclared hash, every subsequent call to that contract — whether via `call_contract`, `invoke`, or `l1_handler` — will attempt to dispatch execution against a class that does not exist in the class commitment trie. The OS has no code to execute for that hash. The contract becomes permanently non-functional.

Any ERC-20 tokens, ETH bridged via L1→L2 messages, or other assets held in the contract's storage are permanently inaccessible. There is no recovery path: the class hash in state cannot be reset without another successful `replace_class` call, which itself requires the contract to be callable.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attacker-controlled entry path is direct and requires no privileged access:

1. An unprivileged contract deployer deploys a contract that exposes a `replace_class` entry point (or whose constructor calls it).
2. Users deposit funds into the contract (e.g., it presents itself as a vault or bridge adapter).
3. The deployer invokes the contract, triggering `replace_class` with an arbitrary undeclared felt (e.g., `1` or any value not in `contract_class_changes`).
4. The OS processes `execute_replace_class` without validation and commits the new `StateEntry` with the bogus class hash.
5. The contract is permanently bricked; all funds are frozen.

Additionally, a malicious sequencer can include any transaction that triggers this syscall on a contract holding funds, since the OS proof will be valid regardless of whether the class hash is declared. The TODO comment in the production code confirms this is a known, unmitigated gap.

---

### Recommendation

Before writing the new `StateEntry`, perform a lookup in `contract_class_changes` to assert the replacement class hash has been declared with a non-zero compiled class hash:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant already enforced in `execute_declare_transaction` and closes the gap identified in the TODO.

---

### Proof of Concept

1. Deploy `VaultContract` with a `drain_class` external entry point that calls `replace_class(0x1)` (an undeclared hash).
2. Users call `deposit()` on `VaultContract`, transferring tokens into its storage.
3. Attacker calls `drain_class()`.
4. OS executes `execute_replace_class`:
   - Reads `class_hash = 0x1` from request.
   - Skips the missing declared-class check (TODO line 898).
   - Writes `StateEntry(class_hash=0x1, ...)` to `contract_state_changes`.
5. Block is proven and accepted on L1 — the proof is valid because the OS enforces no constraint on `class_hash`.
6. All subsequent `call_contract` or `invoke` to `VaultContract` fail: the OS cannot find class `0x1` in the class trie.
7. All deposited tokens are permanently frozen. [4](#0-3)
