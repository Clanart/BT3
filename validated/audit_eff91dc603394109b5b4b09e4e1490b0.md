### Title
Blacklisted ERC20 Recipient Permanently Freezes User Funds in EVM `finTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.sol::finTransfer`, when the destination token is a native ERC20 with a blacklist mechanism (e.g., USDC, USDT), a recipient who is blacklisted after initiating a NEAR→EVM bridge transfer causes every `finTransfer` call to revert. Because the MPC signature is cryptographically bound to the specific recipient address and there is no cancel or redirect mechanism in the NEAR bridge contract, the user's tokens are permanently locked in the NEAR bridge with no recovery path.

---

### Finding Description

`OmniBridge.sol::finTransfer` finalizes cross-chain transfers from NEAR to EVM. For native ERC20 tokens (tokens that are not bridge-deployed, not ERC1155, and not custom-minter tokens), the function releases held tokens via a push-based `safeTransfer`:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
```

`SafeERC20.safeTransfer` reverts if the underlying `transfer` returns `false` or reverts. For tokens like USDC and USDT that implement an operator-controlled blacklist, if `payload.recipient` is blacklisted, the call reverts unconditionally.

Because `completedTransfers[payload.destinationNonce] = true` is set at line 287 — before the transfer — and Solidity reverts roll back all state changes, the nonce is **not** permanently consumed when the transfer fails. The transfer can theoretically be retried, but:

1. **The MPC signature is bound to the specific recipient.** The Borsh-encoded payload hashed at lines 289–309 includes `payload.recipient`. Any attempt to change the recipient requires a new MPC signature, which requires a new transfer message on NEAR.
2. **There is no cancel mechanism on NEAR.** The NEAR bridge contract stores the transfer in `pending_transfers` and provides no public function for the user to cancel the transfer and reclaim their tokens. The only removal paths are `sign_transfer_callback` (fee-zero path), `claim_fee_callback`, and `remove_fin_transfer` (called only from `fin_transfer_send_tokens_callback` for NEAR-destination transfers).
3. **There is no redirect mechanism.** Neither the EVM contract nor the NEAR contract allows changing the recipient of a signed, in-flight transfer.

The result: the user's tokens are permanently locked in the NEAR bridge contract.

The same structural issue exists in `starknet/src/omni_bridge.cairo::fin_transfer` (lines 250–262), where `_set_transfer_finalised` is called before the ERC20 `transfer`, and a failed `assert(success, 'ERR_TRANSFER_FAILED')` reverts the nonce marking, leaving the transfer permanently unfinalizeable.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

A user who bridges native USDC (or any blacklistable ERC20) from NEAR back to EVM and is subsequently blacklisted on that token will have their funds permanently locked in the NEAR bridge contract. The tokens were burned/locked on NEAR when `ft_transfer_call` was called with the `InitTransfer` message. There is no on-chain path to recover them.

---

### Likelihood Explanation

**Low-Medium.** USDC blacklisting is a real, actively used mechanism by Circle (e.g., for OFAC-sanctioned addresses). The bridge explicitly supports native ERC20 tokens including USDC. A user could be blacklisted between the time they initiate a transfer on NEAR and the time the relayer calls `finTransfer` on EVM. The window can be minutes to hours depending on relayer latency. No attacker action is required — the blacklisting is performed by the token issuer.

---

### Recommendation

Replace the push-based delivery with a pull-based pattern for the native ERC20 branch:

```solidity
// Instead of reverting on failed transfer, record the claimable balance
try IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount) {
    // success
} catch {
    claimable[payload.recipient][payload.tokenAddress] += payload.amount;
    emit TransferFailed(payload.recipient, payload.tokenAddress, payload.amount);
}
```

Add a `claim(address token)` function allowing the recipient to pull their tokens when they are no longer blacklisted. Alternatively, implement a `cancelTransfer` function on the NEAR bridge that allows the original sender to reclaim tokens if the EVM finalization has not succeeded within a timeout window.

---

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR (bridged from EVM previously).
2. Alice calls `ft_transfer_call` on the NEAR USDC contract with an `InitTransfer` message targeting her EVM address. The NEAR bridge receives and locks the tokens; `init_transfer` stores the transfer in `pending_transfers` and returns `U128(0)` (no refund).
3. A relayer calls `sign_transfer` on NEAR; the MPC network signs a `TransferMessagePayload` that includes Alice's EVM address as `recipient`.
4. Circle blacklists Alice's EVM address (e.g., due to a regulatory order).
5. The relayer calls `finTransfer` on `OmniBridge.sol` with the MPC signature and the payload.
6. At line 287, `completedTransfers[nonce] = true` is set. At line 351, `IERC20(usdc).safeTransfer(alice_evm, 10000e6)` is called. USDC's `transfer` returns `false` (blacklisted). `safeTransfer` reverts. The entire transaction reverts, including the nonce marking.
7. Every subsequent `finTransfer` call with the same payload reverts identically.
8. Alice's 10,000 USDC is permanently locked in the NEAR bridge contract. There is no `cancelTransfer` function. The MPC signature cannot be reused with a different recipient. Alice's funds are irrecoverable.
