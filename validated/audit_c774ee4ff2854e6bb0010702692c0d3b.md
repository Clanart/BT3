### Title
EIP-7702 `SetCodeAuthorizations` Bypass `EnableCreate` Governance Flag, Allowing Unauthorized Code Installation on EOAs — (File: `x/evm/keeper/state_transition.go`)

---

### Summary

When the `EnableCreate` governance parameter is set to `false` to block new code deployment, EIP-7702 `SetCodeAuthorizations` embedded in call transactions still install delegation code on EOA accounts. The `EnableCreate` guard only checks `msg.To == nil` (classic contract creation), leaving the EIP-7702 code-mutation path entirely unchecked.

---

### Finding Description

`ApplyMessageWithConfig` in `x/evm/keeper/state_transition.go` enforces the `EnableCreate` and `EnableCall` governance flags at lines 352–356:

```go
// return error if contract creation or call are disabled through governance
if !cfg.Params.EnableCreate && msg.To == nil {
    return nil, errorsmod.Wrap(types.ErrCreateDisabled, "failed to create new contract")
} else if !cfg.Params.EnableCall && msg.To != nil {
    return nil, errorsmod.Wrap(types.ErrCallDisabled, "failed to call contract")
}
``` [1](#0-0) 

The `EnableCreate` guard fires **only** when `msg.To == nil`. EIP-7702 transactions always have `msg.To != nil` (they are call transactions carrying `SetCodeAuthorizations`). Consequently, when `EnableCreate=false` and `EnableCall=true`, execution falls through to the `else` branch at lines 483–518, where `applyAuthorization` → `setAuthorizationDelegation` is called unconditionally:

```go
if msg.SetCodeAuthorizations != nil {
    for _, auth := range msg.SetCodeAuthorizations {
        authority, err := k.applyAuthorization(&auth, stateDB)
        ...
    }
}
``` [2](#0-1) 

`setAuthorizationDelegation` writes delegation code directly onto the authority account:

```go
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
``` [3](#0-2) 

There is no `EnableCreate` check anywhere in `applyAuthorization`, `applyDurableAuthorization`, or `setAuthorizationDelegation`. [4](#0-3) 

The durable path (committed to a separate cache context that survives EVM failure) also has no such check:

```go
if commit && cfg.DurableSetCodeAuthorizationCtx != nil && len(validAuths) > 0 {
    durableStateDB := statedb.NewWithParams(...)
    for _, va := range validAuths {
        k.applyDurableAuthorization(&va.auth, va.authority, durableStateDB)
    }
    if err := durableStateDB.Commit(); err != nil { ... }
}
``` [5](#0-4) 

---

### Impact Explanation

When a chain operator or governance action sets `EnableCreate=false` (e.g., during a security pause, upgrade window, or to permanently restrict code deployment), any unprivileged user can still install EIP-7702 delegation code on EOA accounts by submitting a Prague-era call transaction with `SetCodeAuthorizations`. This constitutes **unauthorized account/code mutation** that the governance flag was explicitly designed to prevent. The installed delegation pointer causes the EOA to behave as a smart contract (delegating all calls to the target address), which is functionally equivalent to deploying code on that account.

This matches the allowed High impact: *"EIP-7702 authorization bypass enabling unauthorized account/code mutation."*

---

### Likelihood Explanation

- Requires Prague fork to be active (EIP-7702 enabled) and `EnableCreate=false` with `EnableCall=true` — a realistic governance state during chain upgrades or security incidents.
- The attacker only needs to sign a valid EIP-7702 authorization for an account they control (their own EOA), then submit a standard call transaction. No privileged access, no leaked keys, no validator collusion.
- The entry path is a normal unprivileged `MsgEthereumTx` through the standard ante handler and `ApplyTransaction` → `ApplyMessageWithConfig`.

---

### Recommendation

Add an `EnableCreate` check in `ApplyMessageWithConfig` before processing `SetCodeAuthorizations`, since installing delegation code is semantically equivalent to code deployment:

```go
if msg.SetCodeAuthorizations != nil {
    if !cfg.Params.EnableCreate {
        return nil, errorsmod.Wrap(types.ErrCreateDisabled,
            "EIP-7702 code delegation requires EnableCreate")
    }
    for _, auth := range msg.SetCodeAuthorizations {
        ...
    }
}
``` [6](#0-5) 

---

### Proof of Concept

1. Governance submits `MsgUpdateParams` setting `EnableCreate=false`, `EnableCall=true`. Prague fork is active.
2. Attacker holds EOA `A` (nonce=0) and wants to install delegation to existing contract `C`.
3. Attacker signs `SetCodeAuthorization{ChainID: current, Address: C, Nonce: 0}` with key of `A`.
4. Attacker submits `MsgEthereumTx` with `To = <any existing address>` (non-nil) and `SetCodeAuthorizations = [above auth]`.
5. In `ApplyMessageWithConfig`: `msg.To != nil` → `EnableCreate` check is **skipped**; `EnableCall=true` → no block.
6. `applyAuthorization` validates the signature, increments nonce of `A`, and calls `stateDB.SetCode(A, delegation(C), ...)`.
7. `durableStateDB.Commit()` persists the code to the KV store.
8. Account `A` now has delegation code pointing to `C`, bypassing the `EnableCreate=false` governance restriction. [7](#0-6) [8](#0-7)

// applyAuthorization validates and applies an EIP-7702 code delegation to the
// state, returning the recovered authority so callers can reuse it without a
// second (expensive) signature recovery.
func (k *Keeper) applyAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (common.Address, error) {
	authority, err := k.validateAuthorization(auth, stateDB)
	if err != nil {
		return authority, err
	}

	// If the account already exists in state, refund the new account cost
	// charged in the intrinsic calculation.
	if stateDB.Exist(authority) {
		stateDB.AddRefund(params.CallNewAccountGas - params.TxAuthTupleGas)
	}

	k.setAuthorizationDelegation(auth, authority, stateDB)
	return authority, nil
}
```

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
