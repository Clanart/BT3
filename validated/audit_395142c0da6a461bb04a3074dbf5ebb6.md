### Title
`batch_insert` Bypasses Duplicate-Key/Hash Guards, Silently Corrupting the Shared `key_to_index` Cache and Enabling Forged DataLayer Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces. For every leaf beyond the first two, it calls `insert_entry_to_blob` directly, which silently overwrites existing entries in the shared `BlockStatusCache::key_to_index` and `leaf_hash_to_index` HashMaps. The result is a cache that points to the wrong tree index for a given key, while the orphaned first leaf remains in the blob. Subsequent `get_proof_of_inclusion` calls generate proofs for the wrong leaf, and the Merkle root is computed over a structurally invalid tree containing duplicate keys.

---

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing to the blob:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` only for the first two leaves (when `leaf_count <= 1`). For every subsequent leaf it bypasses those guards entirely and calls `insert_entry_to_blob` directly:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    // ... build block ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` always calls `add_leaf`, which unconditionally overwrites `key_to_index` and `leaf_hash_to_index` via `HashMap::insert`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

Both `key_to_index` and `leaf_hash_to_index` are **global** (shared across all leaves in the tree), exactly analogous to the shared `_ownedTokensIndex` in the ERC1155 report. When a caller supplies a duplicate `KeyId` in the batch, the second `add_leaf` call silently overwrites the first entry in `key_to_index`. The first leaf is now orphaned in the blob (unreachable through the cache), while the cache points to the second leaf's index. [4](#0-3) 

---

### Impact Explanation

`get_proof_of_inclusion` resolves the leaf index exclusively through `key_to_index`:

```rust
let mut index = *self
    .block_status_cache
    .get_index_by_key(key)
    .ok_or(Error::UnknownKey(key))?;
``` [5](#0-4) 

After a duplicate-key `batch_insert`, `key_to_index[K]` points to the second (wrong) leaf. A proof of inclusion is then generated for that wrong leaf, not for the key-value pair the caller believes it is proving. The Merkle root is computed over a tree that contains two leaves with the same key — an invalid state — so the root itself is incorrect. Any verifier that trusts this root and the generated proof accepts a forged inclusion claim.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is a public method exposed directly to Python callers via the wheel binding: [6](#0-5) 

Any Python caller (or downstream DataLayer client) that passes a list containing a repeated `KeyId` — whether by mistake or maliciously — triggers the corruption. No privileged role or key material is required. The `deltas.rs` code path also calls `batch_insert` with externally-sourced node data, widening the attack surface. [7](#0-6) 

---

### Recommendation

Add the same duplicate guards inside the bulk loop of `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic ...
}
```

Alternatively, refactor `batch_insert` to call the validated `insert` path for every leaf, accepting the minor performance cost in exchange for correctness, or add a pre-pass that deduplicates and validates the input vector before any blob writes occur.

---

### Proof of Concept

1. Create a `MerkleBlob` and insert two seed leaves so `leaf_count == 2` (the bulk path is now active).
2. Call `batch_insert` with a vector that contains `KeyId(42)` twice (different values/hashes).
3. Observe that `key_to_index[42]` now points to the second leaf's index; the first leaf is still in the blob but unreachable.
4. Call `get_proof_of_inclusion(KeyId(42))` — the returned proof is for the second leaf, not the first.
5. Call `check_integrity()` — it returns `IntegrityKeyToIndexCacheLength` because `leaf_count` (3, counting both duplicates) does not equal `key_to_index.len()` (2, since the second insert overwrote the first).
6. The Merkle root computed by `calculate_lazy_hashes` covers the invalid tree, making any proof generated from it untrustworthy. [2](#0-1) [8](#0-7) [9](#0-8)
