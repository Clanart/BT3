### Title
`From<CommitmentStateDiff> for ThinStateDiff` Always Produces Empty `deprecated_declared_classes`, Causing Wrong `StateDiffCommitment` and Block Hash for Blocks Containing Cairo 0 Class Declarations — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally hardcodes `deprecated_declared_classes: Vec::new()`. This is the direct sequencer analog of `isClosed()` always returning `false`: a field that must be populated from execution state is instead always set to a fixed empty value. Because `calculate_state_diff_hash` chains `deprecated_declared_classes` into the Poseidon hash that becomes `StateDiffCommitment`, any block that contains a Declare V0 (Cairo 0) transaction will produce a wrong `StateDiffCommitment`, a wrong `state_diff_length`, and therefore a wrong block hash.

---

### Finding Description

**Root cause — hardcoded empty field in the commitment path**

`CommitmentStateDiff` has no `deprecated_declared_classes` field. When the blockifier converts it to `ThinStateDiff` the field is unconditionally set to the empty vector:

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash:
                commitment_state_diff.class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),
        }
    }
}
``` [1](#0-0) 

**Where the empty field is consumed**

`calculate_state_diff_hash` chains `deprecated_declared_classes` into the Poseidon hash that becomes `StateDiffCommitment`:

```rust
hash_chain =
    chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
``` [2](#0-1) 

`StateDiffCommitment` is then chained into `calculate_block_hash` as `block_commitments.state_diff_commitment.0.0`: [3](#0-2) 

**Why the field should be non-empty**

When a Declare V0 transaction executes, the blockifier records the class hash in `StateMaps::declared_contracts` (set to `true`) but does **not** add an entry to `StateMaps::compiled_class_hashes` (Cairo 0 classes have no compiled class hash). The `From<StateMaps> for CommitmentStateDiff` conversion copies only `compiled_class_hashes` → `class_hash_to_compiled_class_hash`; `declared_contracts` is silently dropped: [4](#0-3) 

Consequently the Cairo 0 class hash never reaches `CommitmentStateDiff`, and the subsequent `ThinStateDiff` conversion cannot recover it — it always writes `Vec::new()`.

The central-sync path, by contrast, explicitly populates the field: [5](#0-4) 

This confirms that `deprecated_declared_classes` is expected to be non-empty for blocks containing Cairo 0 declarations.

**Second corrupted value — `state_diff_length`**

`ThinStateDiff::len()` counts every entry in `deprecated_declared_classes`. With the field always empty the `state_diff_length` stored in the block header is under-counted by exactly the number of Cairo 0 class declarations in the block. `state_diff_length` feeds `concat_counts`, which is chained into the block hash as `concatenated_counts`: [6](#0-5) 

---

### Impact Explanation

For every block that contains at least one Declare V0 transaction:

1. `StateDiffCommitment` is computed over a `ThinStateDiff` that is missing the declared class hashes → wrong commitment value.
2. `state_diff_length` is under-counted → wrong `concatenated_counts`.
3. Both wrong values are chained into `calculate_block_hash` → the sequencer stores and propagates a wrong block hash.
4. The wrong block hash becomes the `previous_block_hash` input for the next block, cascading the error forward.
