### Title
Unregistered Token Deposits Cause Permanent Fund Lock in `initTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts any ERC-20 token address without verifying that the token is registered in the cross-chain system. When a user deposits a token that has no corresponding NEAR-side registration, the tokens are locked in the EVM contract and the NEAR bridge's finalization callback panics, making the funds irrecoverable with no on-chain rescue path.

---

### Finding Description

`initTransfer` routes token handling through three branches:

1. `customMinters[tokenAddress] != address(0)` → burn via custom minter
2. `isBridgeToken[tokenAddress]` → burn
3. **else** → `safeTransferFrom(msg.sender, address(this), amount)` — lock in contract [1](#0-0) 

Branch 3 is the native-token lock path (EVM-origin tokens bridged to NEAR). There is **no check** that `ethToNearToken[tokenAddress]` is non-empty, i.e., that the token has been registered on the NEAR side via `logMetadata` + NEAR token deployment. Any arbitrary ERC-20 token passes through this branch and is silently locked. [2](#0-1) 

On the NEAR side, `fin_transfer_callback` unconditionally calls:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [3](#0-2) 

If the token was never registered, `token_decimals` returns `None` and the callback panics. The proof-based `fin_transfer` call on NEAR will always revert for this token, so the transfer can never be finalized.

The EVM contract has no `rescue`, `withdraw`, or `removeFunds` function. Recovery requires a UUPS contract upgrade by an admin — a centralized, manual intervention with no guarantee of execution. [4](#0-3) 

The identical pattern exists in the StarkNet bridge: `init_transfer` transfers any non-bridge-token to the contract without checking registration. [5](#0-4) 

---

### Impact Explanation

Any user who calls `initTransfer` (or StarkNet `init_transfer`) with a token that has not been registered through the `logMetadata` → NEAR deploy pipeline will have their tokens permanently locked in the bridge contract. There is no automated refund, no on-chain rescue function, and no timeout mechanism. The funds are irrecoverable without a privileged contract upgrade.

This matches: **Critical — Permanent freezing / irrecoverable lock of user funds in bridge vault flows.**

---

### Likelihood Explanation

The entry point is fully permissionless — any user can call `initTransfer` with any ERC-20 address. The `logMetadata` step is not enforced on-chain before `initTransfer`, so a user who calls `initTransfer` before the NEAR-side token is deployed (or with a token that was never registered) will lose funds. This is a realistic mistake for any user interacting with a newly listed or self-registered token.

---

### Recommendation

Add a whitelist check in `initTransfer` for the native-token lock path. Before locking tokens, require that the token has a registered NEAR mapping:

```solidity
} else {
    require(
        bytes(ethToNearToken[tokenAddress]).length > 0,
        "ERR_TOKEN_NOT_REGISTERED"
    );
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Apply the same guard in the StarkNet `init_transfer` for the non-bridge-token path.

---

### Proof of Concept

1. Deploy or use any ERC-20 token `T` that has **not** been registered via `logMetadata` + NEAR token deployment (i.e., `ethToNearToken[T] == ""`).
2. Approve `OmniBridge` to spend `T`.
3. Call `OmniBridge.initTransfer(T, amount, 0, 0, "near:recipient.near", "")`.
4. `T` is transferred to `OmniBridge` via the `else` branch. `InitTransfer` event is emitted.
5. A relayer attempts `fin_transfer` on NEAR with a proof of the EVM event.
6. `fin_transfer_callback` calls `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` — panics because `T` is not in `token_decimals`.
7. The NEAR call reverts. The EVM tokens remain locked. No refund path exists on EVM. [6](#0-5) [3](#0-2)
