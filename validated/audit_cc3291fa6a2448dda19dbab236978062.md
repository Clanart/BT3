### Title
Integer Division Truncation in `NewDynamicFeeChecker` Causes Zero-Fee Cosmos Transactions When BaseFee Is Zero - (File: `ante/evm/fee_checker.go`)

### Summary
`NewDynamicFeeChecker` computes `feeCap` via integer division `fee / gas`. When `fee < gas`, this truncates to 0. If `baseFee` is also 0 (a reachable state when `NoBaseFee = true` or when `MinGasPrice = 0` and blocks are below the gas target), the fee sufficiency check passes, `effectivePrice` resolves to 0, and `effectiveFee = 0 * gas = 0`. The `DeductFeeDecorator` then skips deduction entirely, allowing Cosmos SDK transactions to execute with zero fees collected despite a non-zero fee being declared in the transaction.

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` derives the per-gas fee cap from the transaction's declared fee and gas limit using integer (truncating) division:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // line 83
```

When `fee < gas` (e.g., `fee = 999`, `gas = 1000`), `feeCap` truncates to `0`.

The subsequent sufficiency check is:

```go
if feeCap.LT(baseFeeInt) { ... }   // line 86
```

When `baseFee = 0`, `feeCap (0) < baseFeeInt (0)` is `false`, so the check passes silently.

`EffectiveGasPrice` is then called:

```go
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
// = min(maxPriorityPrice + 0, 0) = 0
``` [1](#0-0) 

`EffectiveGasPrice` computes `min(tipCap + baseFee, feeCap) = min(MaxInt64 + 0, 0) = 0`, so `effectiveFee = 0 * gas = 0`. [2](#0-1) 

In `DeductFeeDecorator.AnteHandle`, the returned `effectiveFee` (0 coins) replaces the original fee and is passed to `checkDeductFee`:

```go
fee, priority, err = dfd.txFeeChecker(ctx, tx)   // effectiveFee = 0
...
if err := dfd.checkDeductFee(ctx, tx, fee); err != nil { ... }
``` [3](#0-2) 

Inside `checkDeductFee`, the deduction is gated on `!fee.IsZero()`:

```go
if !fee.IsZero() {
    err := evmkeeper.DeductFees(dfd.bankKeeper, ctx, deductFeesFromAcc, fee)
    ...
}
``` [4](#0-3) 

`sdk.Coins{{Denom: "aevmos", Amount: 0}}.IsZero()` returns `true`, so `DeductFees` is never called. The user's account is not charged.

**When is `baseFee = 0` reachable?**

`types.GetBaseFee` (used at line 56 of `fee_checker.go`) returns `new(big.Int)` (i.e., 0) when London HF is active but `NoBaseFee = true`:

```go
func GetBaseFee(...) *big.Int {
    if !IsLondon(ethCfg, height) { return nil }
    baseFee := feemarketParams.GetBaseFee()   // nil when NoBaseFee=true
    if baseFee == nil { return new(big.Int) } // returns 0
    return baseFee
}
``` [5](#0-4) 

`NoBaseFee = true` is a standard deployment configuration for chains that want EIP-1559 transaction format without dynamic fee adjustment. Additionally, when `NoBaseFee = false` and `MinGasPrice = 0`, the fee market can drive `baseFee` to 0 over time as blocks stay below the gas target. [6](#0-5) 

### Impact Explanation

Any unprivileged user can submit a Cosmos SDK transaction (e.g., `MsgSend`, governance votes, IBC messages) with a declared fee of `gas - 1` (or any value `< gas`) and pay **zero fees** when `baseFee = 0`. The fee collector module receives nothing. This is a direct ante handler fee mis-accounting bug: valid user funds that should be collected as fees are never deducted, and the transaction commits successfully. This matches the allowed High impact: *"ante handler bug that permits valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

`NoBaseFee = true` is a common and documented configuration. The `GetBaseFee` code path that returns 0 in this case is the normal code path, not an edge case. Any chain running with London HF enabled and `NoBaseFee = true` is permanently vulnerable. The attacker input is trivial: submit any Cosmos tx with `fee < gas` (e.g., `fee = 1 aevmos`, `gas = 21000`). No special privileges, keys, or network conditions are required.

### Recommendation

Replace the truncating integer division with a ceiling division so that `feeCap` is never rounded below the true per-gas price:

```go
// ceil(fee / gas) to avoid truncation to 0
feeCap := fee.Add(sdkmath.NewIntFromUint64(gas - 1)).Quo(sdkmath.NewIntFromUint64(gas))
```

Alternatively, compute `effectiveFee` directly from the original `fee` coins (not from the truncated `feeCap`) when `baseFee = 0`, or add an explicit guard rejecting transactions where `fee > 0` but `feeCap` rounds to 0.

### Proof of Concept

**Setup:** Chain with London HF enabled, `NoBaseFee = true` (or `MinGasPrice = 0` and baseFee has decayed to 0).

**Transaction:** Cosmos `MsgSend` with:
- `gas = 21000`
- `fee = 20999 aevmos` (one less than gas, so `fee / gas = 0`)

**Trace through `NewDynamicFeeChecker`:**
1. `baseFee = types.GetBaseFee(...) = big.NewInt(0)` (because `NoBaseFee = true`)
2. `feeCap = 20999 / 21000 = 0` (integer truncation)
3. `feeCap (0) < baseFeeInt (0)` → `false` → check passes
4. `effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = min(MaxInt64, 0) = 0`
5. `effectiveFee = sdk.Coins{{Denom: "aevmos", Amount: 0}}`
6. `checkDeductFee`: `fee.IsZero() = true` → `DeductFees` skipped
7. Transaction commits; user's account is not charged; fee collector receives 0.

**Result:** A transaction declaring `20999 aevmos` in fees executes for free.
