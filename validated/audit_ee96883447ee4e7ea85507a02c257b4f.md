### Title
Wrapping vs. Saturating Arithmetic Bifurcation in `check_time_locks` Enables Timelock Bypass and Consensus Divergence — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` accepts a `nowrap: bool` parameter that switches all four relative-timelock additions between `saturating_add` (`nowrap=true`) and `wrapping_add` (`nowrap=false`). For overflow-inducing timelock values the two modes produce opposite accept/reject decisions on the same spend bundle. Any node or software version that passes a different `nowrap` value than the rest of the network will diverge on the validity of such spends, and a node running with `nowrap=false` will silently accept spends whose timelocks have wrapped to a trivially small threshold, bypassing the intended lock entirely.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` validates four relative-timelock conditions (`height_relative`, `seconds_relative`, `before_height_relative`, `before_seconds_relative`). For each one the arithmetic is branched on `nowrap`:

```rust
// lines 55-68
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { return Err(...AssertHeightRelativeFailed); }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { return Err(...AssertHeightRelativeFailed); }
}
```

The same pattern repeats for `seconds_relative` (lines 70–78), `before_height_relative` (lines 79–98), and `before_seconds_relative` (lines 100–112).

**Concrete overflow scenario — `height_relative`:**

| Parameter | Value |
|---|---|
| `confirmed_block_index` | `1` |
| `height_relative` (from spend) | `0xFFFF_FFFF` (u32::MAX) |
| `prev_transaction_block_height` | `200` |

- `nowrap=true` (saturating): `1u32.saturating_add(0xFFFF_FFFF) = 0xFFFF_FFFF`. Check: `200 < 0xFFFF_FFFF` → **Err** (spend rejected — correct).
- `nowrap=false` (wrapping): `1u32.wrapping_add(0xFFFF_FFFF) = 0`. Check: `200 < 0` → **false** → **Ok** (spend accepted — timelock bypassed).

The same inversion exists for `seconds_relative` (u64 domain) and the `before_*` variants flip in the opposite direction.

The `nowrap` parameter is not a consensus constant embedded in `ConsensusConstants`; it is a caller-supplied boolean threaded through the Python binding (`py_check_time_locks`, lines 122–141) and therefore subject to per-node or per-version configuration drift.

---

### Impact Explanation

**Timelock bypass (High):** A node running with `nowrap=false` accepts a spend whose `height_relative` or `seconds_relative` is set to a value that wraps the addition to a small threshold. A coin that should be locked for ~4 billion blocks (or ~584 billion years for `seconds_relative = u64::MAX`) can be spent immediately. This enables unauthorized spend acceptance — a direct match for the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

**Consensus divergence (Critical path):** If any subset of the network passes `nowrap=true` while another passes `nowrap=false`, the two groups will reach opposite conclusions about the validity of the same spend bundle. The block containing that spend will be accepted by one partition and rejected by the other, producing a committed chain split — a match for the Critical impact: *"deterministic consensus divergence."*

---

### Likelihood Explanation

- `height_relative` and `seconds_relative` are parsed from attacker-controlled CLVM atoms in the spend solution; any value up to `u32::MAX` / `u64::MAX` is syntactically valid.
- The `nowrap` parameter is not enforced by a consensus constant; it is passed at call-site, making version-skew or misconfiguration realistic.
- The overflow threshold is exactly one integer addition away from any confirmed height/timestamp, so the craft is trivial.
- The Python test suite explicitly documents and exercises both divergent outcomes, confirming the bifurcation is reachable.

---

### Recommendation

1. **Remove the `nowrap` parameter entirely.** Choose one semantic — saturating addition — and hard-code it. Saturating is the safe choice: it makes large relative timelocks effectively permanent rather than wrapping to zero.
2. If a legacy-wrapping mode must be preserved for historical block replay, gate it on a consensus-constant block height (stored in `ConsensusConstants`) rather than a caller-supplied boolean, so all nodes agree deterministically.
3. Add a consensus-layer assertion that `height_relative` and `seconds_relative` values that would overflow the addition are rejected at condition-parse time (in `conditions.rs`), eliminating the ambiguity at the source.
