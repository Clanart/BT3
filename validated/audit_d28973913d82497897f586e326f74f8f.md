### Title
Inverted NEP-141 Callback Success Condition Releases Fee Tokens on Rejected BTC Withdrawal — (`near/omni-bridge/src/btc.rs`)

### Summary

The `submit_transfer_to_btc_connector_callback` function in `btc.rs` has its success/failure branches swapped relative to NEP-141 semantics. When the BTC connector **rejects** a transfer (returns a non-zero refund), the bridge pays the fee to the relayer. When the connector **accepts** (returns zero), the bridge re-inserts the transfer as pending. This is the exact inverse of correct behavior, allowing a trusted relayer to collect fee tokens without a confirmed BTC withdrawal.

### Finding Description

In NEP-141, `ft_transfer_call` resolves to the **refund amount** — the number of tokens the receiver did **not** consume:
- `Ok(U128(0))` → all tokens consumed → **SUCCESS** (connector accepted the transfer)
- `Ok(U128(n > 0))` → n tokens refunded → **REJECTION** (connector rejected the transfer)

The callback at `btc.rs` lines 111–125 reads:

```rust
if matches!(call_result, Ok(result) if result.0 > 0) {
    // Triggered when connector REFUNDED tokens = REJECTION
    let token_fee = transfer_msg.fee.fee.0;
    self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)  // ← pays fee on REJECTION
} else {
    // Triggered when connector CONSUMED tokens = SUCCESS
    self.add_transfer_message(transfer_msg, transfer_owner.clone()); // ← re-inserts on SUCCESS
    ...
}
``` [1](#0-0) 

The correct logic should be the opposite: pay the fee on success (`result.0 == 0`), and re-insert the transfer on rejection (`result.0 > 0`).

The caller `submit_transfer_to_utxo_chain_connector` removes the transfer from `pending_transfers` **before** the cross-contract call at line 84:

```rust
self.remove_transfer_message(transfer_id);
``` [2](#0-1) 

Then it calls `ft_transfer_call` to the BTC connector with `amount = transfer.amount - fee`: [3](#0-2) 

When the connector rejects and the token contract refunds `amount - fee` back to the bridge, the callback fires with `result.0 > 0`, triggering `send_fee_internal`. The fee tokens are released to the relayer, but the transfer record was already deleted and is never re-inserted. The `amount - fee` tokens are returned to the bridge contract with no corresponding pending transfer entry — effectively frozen.

### Impact Explanation

- **Unauthorized release of BTC-backed fee tokens**: The relayer receives the fee without any confirmed on-chain BTC UTXO settlement.
- **User funds frozen**: The `amount - fee` portion is refunded to the bridge by the token contract, but since the transfer was removed from `pending_transfers` and the callback does not re-insert it (it only re-inserts on the success branch), those tokens are stuck with no recovery path for the user without admin intervention.

This matches the critical impact: *unauthorized release of BTC-backed fee tokens to the relayer without a confirmed on-chain withdrawal*.

### Likelihood Explanation

The entry point `submit_transfer_to_utxo_chain_connector` is gated by `#[trusted_relayer]`. [4](#0-3) 

From the relayer staking system, becoming a trusted relayer requires staking 1,000 NEAR and waiting a 7-day period — it auto-promotes without DAO approval: [5](#0-4) 

This is a permissioned but open role. Any actor who stakes 1,000 NEAR and waits 7 days can exploit this. The BTC connector can legitimately return a non-zero refund for various reasons (invalid UTXO parameters, insufficient gas fee, etc.), making the trigger condition realistic in normal operation.

### Recommendation

Swap the branches in `submit_transfer_to_btc_connector_callback`:

```rust
if matches!(call_result, Ok(result) if result.0 == 0) {
    // Connector consumed all tokens = SUCCESS → pay fee
    let token_fee = transfer_msg.fee.fee.0;
    self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)
} else {
    // Connector refunded tokens = REJECTION → re-insert transfer
    let required_storage_balance =
        self.add_transfer_message(transfer_msg, transfer_owner.clone());
    self.update_storage_balance(
        transfer_owner,
        required_storage_balance,
        NearToken::from_yoctonear(0),
    );
    PromiseOrValue::Value(())
}
```

### Proof of Concept

1. A trusted relayer calls `submit_transfer_to_utxo_chain_connector` for a valid pending BTC-bound transfer.
2. The bridge removes the transfer from `pending_transfers` and calls `ft_transfer_call` on the BTC token contract, forwarding `amount - fee` to the connector.
3. The BTC connector's `ft_on_transfer` returns `U128(amount - fee)` (full refund = rejection).
4. The token contract refunds `amount - fee` back to the bridge and resolves `ft_transfer_call` with `Ok(U128(amount - fee))`.
5. `submit_transfer_to_btc_connector_callback` fires with `call_result = Ok(U128(n > 0))`.
6. The `if` branch matches → `send_fee_internal` is called → relayer receives fee tokens.
7. The `else` branch (re-insert) is **not** taken → the transfer is permanently gone from `pending_transfers`.
8. The user's `amount - fee` tokens sit in the bridge with no pending transfer record.

Unit test assertion: call `submit_transfer_to_btc_connector_callback` with `call_result = Ok(U128(1))` and assert that `send_fee_internal` is **not** called and the transfer **is** re-inserted into `pending_transfers`. The current code fails both assertions. [6](#0-5)
