### Title
Rejected L1 Handler Transactions Permanently Excluded from Future Blocks During Catch-Up — (`File: crates/apollo_l1_provider/src/l1_provider.rs`)

### Summary

During the L1 provider's catch-up phase, the `rejected_txs` set passed by the batcher is silently dropped. Every L1 handler transaction that was consumed (included in a block) but rejected during execution is incorrectly promoted to `Committed` state instead of `Rejected` state. `Committed` transactions are permanently barred from future blocks; `Rejected` transactions are supposed to remain re-proposable. The result is that any L1 handler transaction that fails execution in a block processed during catch-up is permanently lost from the sequencer's view, even though the L1 message was never successfully delivered on L2.

### Finding Description

**Normal flow (correct):**

In `commit_block`, when the provider is in steady state, both `committed_txs` and `rejected_txs` are forwarded to `apply_commit_block`:

```rust
// l1_provider.rs line 322
self.apply_commit_block(committed_txs, rejected_txs);
```

Inside `apply_commit_block`, the two sets are partitioned and passed to `commit_txs`:

```rust
// l1_provider.rs lines 371-373
let (rejected_and_consumed, committed_txs): (Vec<_>, Vec<_>) =
    consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));
self.tx_manager.commit_txs(&committed_txs, &rejected_and_consumed);
```

`commit_txs` marks truly-committed transactions as `Committed` and rejected-but-consumed transactions as `Rejected`. `Rejected` transactions remain validatable (`is_validatable()` returns `true`) and return `ValidationStatus::Validated` when a future validator queries them.

**Catch-up flow (broken):**

When the provider is in `CatchingUp` state, `commit_block` routes to `accept_commit_while_catching_up`, which only accepts `committed_txs` — the `rejected_txs` argument is silently dropped at the call site:

```rust
// l1_provider.rs line 315
return self.accept_commit_while_catching_up(committed_txs, height);
```

Inside `accept_commit_while_catching_up`, both the `Equal` branch and the backlog-replay loop call `apply_commit_block` with an empty `rejected_txs`:

```rust
// l1_provider.rs line 427
Equal => self.apply_commit_block(committed_txs, Default::default()),
// ...
// l1_provider.rs line 464
self.apply_commit_block(committed_block.committed_txs, Default::default());
```

The `CommitBlockBacklog` struct stores only `committed_txs` and `height`; there is no field for `rejected_txs`, so the information is unrecoverable once backlogged.

The developer-acknowledged TODO at line 426 confirms this is a known gap:

```rust
// TODO(guyn): check what about rejected txs here and in the backlog?
Equal => self.apply_commit_block(committed_txs, Default::default()),
```

**Consequence in `TransactionManager`:**

`commit_txs` receives the rejected-but-consumed hashes in the `committed_txs` slice (because `rejected_txs` is empty, the partition puts everything into `committed_txs`). Each such hash is marked `Committed`:

```rust
// transaction_manager.rs lines 154-157
for &tx_hash in committed_txs {
    self.create_record_if_not_exist(tx_hash);
    self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
}
```

`is_validatable()` returns `false` for `Committed` transactions, so any future `validate()` call returns `AlreadyIncludedOnL2`, permanently blocking re-inclusion.

### Impact Explanation

A rejected L1 handler transaction represents an L1→L2 message that was included in a block but whose execution failed. The correct behavior is to keep it in `Rejected` state so it can be re-proposed and re-executed in a later block. After this bug, the transaction is silently promoted to `Committed`, making it indistinguishable from a successfully executed message. The L1 message is permanently undelivered on L2 with no error surfaced to the operator or the L1 sender. This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

Catch-up is triggered automatically whenever the L1 provider's `current_height` diverges from the batcher's committed height — a routine occurrence during node startup, restart after a crash, or any brief network partition. Any block that contains a rejected L1 handler transaction and is committed while the provider is in `CatchingUp` state will silently corrupt the provider's records. No privileged access or adversarial action is required; normal operational conditions are sufficient.

### Recommendation

1. Extend `accept_commit_while_catching_up` to accept and forward `rejected_txs`:

```rust
fn accept_commit_while_catching_up(
    &mut self,
    committed_txs: IndexSet<TransactionHash>,
    rejected_txs: IndexSet<TransactionHash>,  // add this
    new_height: BlockNumber,
) -> L1ProviderResult<()>
```

2. Extend `CommitBlockBacklog` to store `rejected_txs` alongside `committed_txs`.

3. Pass the real `rejected_txs` to every `apply_commit_block` call inside `accept_commit_while_catching_up`, including the backlog-replay loop.

4. Update the call site in `commit_block` to forward `rejected_txs`:

```rust
return self.accept_commit_while_catching_up(committed_txs, rejected_txs, height);
```

### Proof of Concept

```
1. Start an L1 provider at height 0.
2. Trigger catch-up by calling commit_block([], [], height=5) → provider enters CatchingUp.
3. Prepare a block at height 0 that contains:
     consumed_l1_handler_tx_hashes = {tx_A}
     rejected_l1_handler_tx_hashes = {tx_A}   // tx_A was consumed but execution reverted
4. Call commit_block({tx_A}, {tx_A}, height=0) while provider is CatchingUp.
   → accept_commit_while_catching_up({tx_A}, height=0) is called; rejected_txs dropped.
   → apply_commit_block({tx_A}, {}) marks tx_A as Committed.
5. Complete catch-up; provider transitions to Pending.
6. Call validate(tx_A, current_height).
   → Returns AlreadyIncludedOnL2 (Committed state).
   → tx_A can never be re-proposed, even though its L1 message was never delivered on L2.
```

    pub fn commit_txs(
        &mut self,
        committed_txs: &[TransactionHash],
        rejected_txs: &[TransactionHash],
    ) {
        self.rollback_staging();

        for &tx_hash in committed_txs {
            self.create_record_if_not_exist(tx_hash);
            self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
        }
        for &tx_hash in rejected_txs {
            self.with_record(tx_hash, |r| r.mark_rejected()).expect(
                "Storage inconsistency: a transaction sent to the batcher was removed \
                 unexpectedly.",
            );
        }
    }
