### Title
Precommit Vote Message Digest Omits `chain_id`, Enabling Cross-Chain Signature Replay in BFT Consensus - (File: crates/apollo_signature_manager/src/signature_manager.rs)

### Summary

`build_precommit_vote_message_digest` constructs the ECDSA message digest for BFT consensus precommit votes using only a domain separator and the raw `block_hash` bytes. It omits `chain_id`. Because the block hash itself is also computed without `chain_id`, a precommit vote signature produced on one Starknet network (e.g., `SN_MAIN`) is cryptographically valid on any other Starknet network (e.g., `SN_SEPOLIA`) where the same block hash value appears. When vote-signature verification is wired in — explicitly planned via a `TODO` in `handle_vote` — an attacker can replay a validator's precommit from one chain to manufacture a false quorum on another.

---

### Finding Description

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs` builds the signed payload as:

```
blake2s( b"PRECOMMIT_VOTE" || block_hash_bytes )
``` [1](#0-0) 

No `chain_id` is mixed in. The corresponding signing entry-point `sign_precommit_vote` and the verification library function `verify_precommit_vote_signature` both call this same helper, so both sides of the protocol share the same chain-agnostic digest. [2](#0-1) [3](#0-2) 

The block hash itself is computed by `calculate_block_hash` as a Poseidon hash over: version constant, block number, state root, sequencer address, timestamp, packed counts, state-diff/tx/event/receipt commitments, gas prices, Starknet version, a zero field, and parent block hash — **no `chain_id`**. [4](#0-3) 

The Cairo OS `calculate_block_hash` function confirms the same omission: [5](#0-4) 

Because neither the block hash nor the precommit message digest binds a chain identifier, the same `(block_hash, validator_signature)` pair is valid on every Starknet network that shares the same validator key set.

Vote-signature verification is currently deferred with an explicit `TODO`:

```rust
// TODO(Asmaa): verify the signature
``` [6](#0-5) 

The signing infrastructure (`SignatureManager`, `sign_precommit_vote`) is already wired into the production node. Once the TODO is resolved and `verify_precommit_vote_signature` is called inside `handle_vote`, the replay path becomes immediately exploitable.

---

### Impact Explanation

An attacker who observes a valid precommit vote `(block_hash H, signature σ_V)` from validator `V` on network A can submit the identical `(H, σ_V)` to network B. If network B happens to be processing a block whose hash equals `H` (possible because block hashes are chain-agnostic), `verify_precommit_vote_signature` will accept `σ_V` as a legitimate vote from `V` on network B. This allows the attacker to:

- Inflate the apparent precommit quorum on network B using votes that were never cast there.
- Potentially drive `upon_decision` to emit `SMRequest::DecisionReached` for a block that the real validator set of network B never agreed upon.

This maps to the **High** impact category: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"* — the consensus layer binds a validator's signature to the wrong chain context, causing the wrong block commitment to be accepted.

---

### Likelihood Explanation

- **Starknet Mainnet and Sepolia share the same validator key infrastructure** in test/staging scenarios; the same operator key can appear on both networks.
- Block hashes are deterministic and chain-agnostic; a collision between networks is not hypothetical — it is guaranteed to occur for genesis-adjacent blocks or any block whose inputs happen to match.
- The attack requires only passive observation of the P2P vote broadcast channel on one network and active injection on another — both are unprivileged operations.
- The vulnerability is latent today but becomes live the moment the `TODO(Asmaa)` is resolved, which is an explicitly planned code change.

---

### Recommendation

Include `chain_id` in `build_precommit_vote_message_digest`:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    chain_id: &ChainId,
) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let chain_id_bytes = chain_id.as_hex().into_bytes();
    let mut message = Vec::with_capacity(
        PRECOMMIT_VOTE.len() + block_hash.len() + chain_id_bytes.len()
    );
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    message.extend_from_slice(&chain_id_bytes);
    MessageDigest(blake2s_to_felt(&message))
}
```

Propagate `chain_id` through `sign_precommit_vote`, `verify_precommit_vote_signature`, and the `SignatureManagerRequest::SignPrecommitVote` variant. Apply the same fix to `build_peer_identity_message_digest` for defence-in-depth. [7](#0-6) 

---

### Proof of Concept

1. **Setup**: Validator `V` operates on both `SN_MAIN` and `SN_SEPOLIA` with the same ECDSA key pair (common in staging).

2. **Capture**: On `SN_MAIN`, block `N` is produced with `block_hash = H`. Validator `V` calls `sign_precommit_vote(H)`, producing signature `σ`. The message digest is `blake2s(b"PRECOMMIT_VOTE" || H_bytes)` — no chain binding. [1](#0-0) 

3. **Collision**: On `SN_SEPOLIA`, a block is produced whose `calculate_block_hash` output also equals `H` (possible because `chain_id` is absent from the block hash formula). [8](#0-7) 

4. **Replay**: The attacker injects the vote `Vote { block_hash: H, signature: σ, voter: V, ... }` into the `SN_SEPOLIA` P2P vote channel.

5. **Acceptance**: `handle_vote` on `SN_SEPOLIA` (once the TODO is resolved) calls `verify_precommit_vote_signature(H, σ, V.public_key)`. It computes `blake2s(b"PRECOMMIT_VOTE" || H_bytes)` — identical to the Mainnet digest — and the ECDSA check passes. [3](#0-2) 

6. **Outcome**: `V`'s replayed vote is counted toward the precommit quorum on `SN_SEPOLIA`, potentially triggering `SMRequest::DecisionReached` for a block that `V` never voted for on Sepolia. [9](#0-8)

func calculate_block_hash{poseidon_ptr: PoseidonBuiltin*}(
    block_info: BlockInfo*,
    header_commitments: BlockHeaderCommitments*,
    gas_prices_hash: felt,
    state_root: felt,
    previous_block_hash: felt,
    starknet_version: felt,
) -> felt {
    static_assert BlockInfo.SIZE == 3;
    static_assert BlockHeaderCommitments.SIZE == 5;

    let hash_state = hash_init();
    with hash_state {
        hash_update_single(BLOCK_HASH_VERSION);
        hash_update_single(block_info.block_number);
        hash_update_single(state_root);
        hash_update_single(block_info.sequencer_address);
        hash_update_single(block_info.block_timestamp);
        hash_update_single(header_commitments.packed_lengths);
        hash_update_single(header_commitments.state_diff_commitment);
        hash_update_single(header_commitments.transaction_commitment);
        hash_update_single(header_commitments.event_commitment);
        hash_update_single(header_commitments.receipt_commitment);
        hash_update_single(gas_prices_hash);
        hash_update_single(starknet_version);
        hash_update_single(0);
        hash_update_single(previous_block_hash);
    }

    let block_hash = hash_finalize(hash_state=hash_state);
    return block_hash;
}

    pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {

        trace!("Received {:?}", vote);

        if vote.height != height {

                );
                return VecDeque::new();

                // Conflict - ignore and record.

                return VecDeque::new();

                // Vote is new, proceed to process it.
            }
