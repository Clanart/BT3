### Title
Hardcoded `failure_flag=0` in `execute_deploy` Syscall Prevents Detection of Constructor Failures — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall implementation in the StarkNet OS unconditionally writes `failure_flag=0` in the response header, regardless of whether the deployed contract's constructor succeeded or reverted. This is the direct analog of the Pendle hardcoded-zero vulnerability: a protection signal is hardcoded to a value that always indicates success, stripping callers of the ability to detect and react to failure. Any contract that uses the `deploy` syscall and makes state-altering decisions based on the reported outcome is vulnerable to permanent fund freezing.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–555), after calling `deploy_contract`, the OS writes the syscall response header with `failure_flag` unconditionally set to `0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The inline TODO comment explicitly acknowledges the incompleteness: *"support failures"*. The `deploy_contract` function is called with `remaining_gas` and returns `(retdata_size, retdata)`. When the constructor reverts, the revert log (which `execute_deploy` holds as an implicit argument) correctly unwinds the constructor's state changes — so the contract is not actually deployed. However, the response header still reports `failure_flag=0` (success), and the constructor's revert data is placed in `constructor_retdata_start/end` indistinguishably from normal return data.

The calling contract receives:
- `failure_flag = 0` → "deployment succeeded"
- `contract_address` → the address of a contract that does not exist in state
- `constructor_retdata` → the revert reason, presented as normal return data

There is no way for the calling contract to distinguish a successful deployment from a reverted one, because the only authoritative signal — `failure_flag` — is hardcoded to zero.

Compare this to the Pendle pattern:
- Pendle: `minLpOut = 0`, `minTokenOut = 0` → caller cannot enforce a minimum output
- StarkNet OS: `failure_flag = 0` → caller cannot detect a failed deployment

Both cases hardcode a protection value to zero, removing the caller's ability to guard against an adverse outcome.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A factory or vault contract that:
1. Calls `deploy` to create a child contract,
2. Receives `failure_flag=0` and stores the returned `contract_address` as a live deployment,
3. Locks user funds into a state that requires the child contract to exist for release,

will permanently freeze those funds. Because the constructor reverted, the child contract has no class hash in state (class_hash = 0). Any subsequent call to that address will fail with "entry point not found." If the parent contract has no recovery path (a common design for trustless vaults and escrows), the funds are irrecoverable.

The scenario is not hypothetical: factory patterns, proxy deployers, and escrow-on-deploy contracts are standard DeFi primitives on StarkNet. All of them are affected by this OS-level bug because the OS is the authoritative source of the `failure_flag` signal.

---

### Likelihood Explanation

**Medium-High.** The `deploy` syscall is a standard, widely-used primitive. Any contract that:
- Accepts a user-supplied `class_hash` to deploy (e.g., a permissionless factory), **or**
- Deploys a contract whose constructor can revert under attacker-controlled conditions (e.g., constructor reads from storage or calldata),

is exploitable. An unprivileged attacker needs only to:
1. Declare a class whose constructor reverts under a chosen condition.
2. Trigger a victim factory to deploy that class (e.g., by calling a public `create_vault(class_hash)` function).
3. The factory receives `failure_flag=0`, stores the dead address, and locks deposited funds.

No privileged access, leaked keys, or network-level attack is required.

---

### Recommendation

Remove the hardcoded `failure_flag=0`. Propagate the actual constructor outcome:

```cairo
// Capture whether the constructor reverted.
let (is_reverted, retdata_size, retdata) = deploy_contract(...);

assert [response_header] = ResponseHeader(
    gas=remaining_gas,
    failure_flag=is_reverted   // propagate actual result
);
```

If the constructor reverted, the `contract_address` field in `DeployResponse` should be set to `0` (or omitted) to prevent callers from storing a dead address.
