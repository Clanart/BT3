Audit Report

## Title
Permissionless `refundETH()` allows any caller to drain excess ETH left on the router by a victim — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is an unrestricted `external` function that unconditionally transfers the router's entire native ETH balance to `msg.sender`. When a user sends `msg.value` exceeding `amountIn` to a payable swap entry point, `pay()` deposits only the exact swap amount as WETH and leaves the remainder on the router. Any subsequent caller of `refundETH()` in a separate transaction receives that stranded ETH.

## Finding Description
`refundETH()` contains no access control:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // no caller check
    }
}
```

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, deposits exactly `value` and leaves the surplus on the router:

```solidity
// PeripheryPayments.sol L75-77
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable swap functions. After `exactInputSingle{value: 1 ether}(amountIn=0.5 ETH, tokenIn=WETH)` completes, 0.5 ETH remains on the router across transaction boundaries. The `multicall` dispatcher uses `delegatecall`, so `msg.sender` is preserved within a single multicall — but `refundETH` is independently callable by any EOA or contract in a subsequent transaction, where `msg.sender` is the attacker.

The intended safe pattern requires the user to compose a multicall that includes `refundETH` as the last call (as shown in `test_multicall_ethInput_exactInputSingle_refundsUnusedEth`), but this is not enforced at the contract level. A user calling `exactInputSingle` directly with excess `msg.value`, or omitting `refundETH` from a multicall, leaves ETH permanently exposed.

## Impact Explanation
Direct, unconditional loss of user principal (native ETH). Any ETH stranded on the router between transactions is immediately claimable by any EOA or contract with no preconditions. This is a Critical/High direct loss of user funds meeting Sherlock thresholds.

## Likelihood Explanation
Users sending native ETH for WETH swaps must manually compose a multicall including `refundETH` as the last call. Calling `exactInputSingle` directly with `msg.value > amountIn` — a natural usage pattern — leaves ETH on the router. MEV bots routinely monitor for unprotected ETH balances on router contracts and can backrun the victim's transaction in the same block. The attack requires no special privileges, no oracle manipulation, and no complex setup.

## Recommendation
Restrict `refundETH` to the active `multicall` context by recording `msg.sender` at `multicall` entry in transient storage and checking it inside `refundETH`. Alternatively, accept an explicit `recipient` parameter that the caller must supply (matching Uniswap v4's approach), or auto-refund excess ETH at the end of each swap entry point rather than relying on a separate call.

## Proof of Concept
```
1. User calls:
   router.exactInputSingle{value: 1 ether}(
       ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ETH, ...
       })
   );
   // pay() deposits exactly 0.5 ETH as WETH; 0.5 ETH stays on router.

2. Attacker (next tx, same block via backrun):
   router.refundETH();
   // refundETH sends address(this).balance (0.5 ETH) to attacker.

3. Assert: attacker.balance increased by 0.5 ETH; user lost 0.5 ETH.
```

Foundry test: deploy router, fund swapper with 1 ETH, call `exactInputSingle{value: 1 ether}` with `amountIn = 0.5 ETH` and `tokenIn = WETH`, then call `router.refundETH()` from a separate attacker address and assert `attacker.balance == 0.5 ETH` and `address(router).balance == 0`.
