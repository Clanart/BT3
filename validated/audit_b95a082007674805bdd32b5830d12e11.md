### Title
Non-EVM-denom Bank Balances Not Cleared on SELFDESTRUCT, Recoverable via CREATE2 Redeploy — (File: x/evm/statedb/statedb.go)

### Summary
When a contract self-destructs, `Commit()` only burns the EVM-native denom balance. Non-EVM-denom bank balances (IBC tokens, CosmWasm bridge tokens) held by the destroyed address are explicitly left untouched and remain in the Cosmos bank module. A subsequent CREATE2 redeploy at the same deterministic address can access these orphaned balances via a bank precompile, constituting unauthorized recovery of Cosmos bank funds.

### Finding Description
In `x/evm/statedb/statedb.go`, the `Commit()` function processes self-destructed accounts by burning only the EVM denom balance before calling `DeleteAccount`:

```go

The Cosmos SDK bank module stores balances keyed by address independently of the auth account. Removing the auth account via `RemoveAccount` leaves any non-EVM-denom bank balances (IBC denoms, bridged tokens) intact at that address. When a new contract is deployed at the same address via CREATE2, the new auth account is created at the same address and the pre-existing bank balances become accessible to it.

The analog to the external report is exact:
- **External report**: `deleteOperators` removes the operator from governance but leaves `operatorRewards[operator]` untouched → operator can still call `claimRewards()`.
- **Ethermint**: `DeleteAccount` removes the contract from EVM state but leaves non-EVM-denom bank balances untouched → a new contract at the same CREATE2 address can claim those balances via a bank precompile.

### Impact Explanation
On any Ethermint-based chain that (a) allows IBC or bridged tokens to be held at contract addresses and (b) exposes a bank precompile (e.g., Cronos's `x/bank` precompile), an attacker can:

1. Deploy a contract via CREATE2 at a deterministic address.
2. Arrange for that contract to hold non-EVM-denom tokens (IBC, bridged assets).
3. Trigger `SELFDESTRUCT` on the contract (EIP-6780: only effective if created in the same tx, or on pre-Cancun chains unconditionally).
4. Observe that the EVM denom balance is burned but IBC tokens remain at the address in the bank module.
5. Redeploy a new contract at the same CREATE2 address.
6. Call the bank precompile from the new contract to transfer the orphaned IBC tokens to an attacker-controlled address.

This is unauthorized theft of Cosmos bank funds through Ethermint transaction execution, matching the Critical/High allowed impact scope.

### Likelihood Explanation
The entry path (SELFDESTRUCT + CREATE2) is available to any unprivileged user via standard EVM opcodes. The precondition — a chain with non-EVM-denom tokens accessible to contracts via a bank precompile — is satisfied by multiple production deployments built on Ethermint. The code comment explicitly acknowledges the gap exists, confirming it is a reachable state rather than a theoretical edge case.

### Recommendation
In `Commit()`, after burning the EVM denom balance and before calling `DeleteAccount`, iterate over all bank module balances at the destroyed address and burn or redirect them:

```go
// Drain ALL bank balances, not just the EVM denom.
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrapf(err, "failed to burn post-selfdestruct %s balance", coin.Denom)
    }
}
```

Alternatively, transfer non-EVM-denom balances to a designated burn or community-pool address rather than attempting to subtract them (which requires the bank module to support burning arbitrary denoms).

### Proof of Concept

```
1. Deploy FactoryContract via CREATE2 with salt S.
   FactoryContract deploys ChildContract at deterministic address A = CREATE2(factory, S, initHash).

2. Transfer 1000 IBC/uatom to address A via the bank precompile (or IBC transfer).
   Verify: bankKeeper.GetBalance(A, "ibc/...") == 1000.

3. Call ChildContract.destroy() → triggers SELFDESTRUCT.
   Commit() burns s.evmDenom balance at A, calls DeleteAccount(A).
   Verify: bankKeeper.GetBalance(A, "ibc/...") == 1000  ← still present (bug).
   Verify: accountKeeper.GetAccount(A) == nil           ← auth account gone.

4. Redeploy ChildContract at address A via FactoryContract.redeployChild(S).
   A new EthAccount is created at address A.

5. From the new ChildContract, call bankPrecompile.transfer(A, attacker, "ibc/...", 1000).
   Verify: attacker receives 1000 IBC tokens.
   Verify: bankKeeper.GetBalance(A, "ibc/...") == 0.
```

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
