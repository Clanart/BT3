### Title
Nonce Underflow After Failed `evm.Create()` Enables Sender Nonce Replay - (File: x/evm/keeper/state_transition.go)

### Summary
In `ApplyMessageWithConfig`, the `nestedCreates` calculation performs an unchecked uint64 subtraction that underflows when `evm.Create()` fails and reverts. This sets the sender's committed nonce to `msg.Nonce` — one less than the ante-handler-incremented value — allowing the same nonce to be reused in a subsequent transaction.

### Finding Description

In `ApplyMessageWithConfig`, the contract-creation branch reads the sender's nonce before and after calling `evm.Create()` to account for nested `CREATE` opcodes: [1](#0-0) 

```go
if contractCreation {
    oldNonce := stateDB.GetNonce(sender)          // = msg.Nonce + 1 (ante-handler pre-incremented)
    stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to msg.Nonce before evm.Create
    ret, _, leftoverGas, vmErr = evm.Create(...)
    afterCreateNonce := stateDB.GetNonce(sender)
    nestedCreates := afterCreateNonce - msg.Nonce - 1   // ← underflows when Create reverts
    stateDB.SetNonce(sender, oldNonce+nestedCreates, ...)
}
```

When `evm.Create()` fails (constructor reverts, address collision, out-of-gas, etc.), go-ethereum reverts to the snapshot it took **before** incrementing the caller's nonce. Because Ethermint set the nonce to `msg.Nonce` before the call, the snapshot captures `msg.Nonce`, and after revert `afterCreateNonce == msg.Nonce`.

The subtraction then underflows in uint64 arithmetic:

```
nestedCreates = msg.Nonce - msg.Nonce - 1 = uint64(-1) = math.MaxUint64
```

The final `SetNonce` call:

```
oldNonce + math.MaxUint64
= (msg.Nonce + 1) + (2^64 - 1)
= msg.Nonce + 2^64
= msg.Nonce   (mod 2^64)
```

The stateDB is committed unconditionally regardless of `vmErr`: [2](#0-1) 

So `stateDB.Commit()` writes nonce `msg.Nonce` to the Cosmos account keeper — one less than the `msg.Nonce + 1` the ante-handler set. The ante-handler's nonce increment is silently undone.

The nonce validation in `CheckAndSetEthSenderNonce` checks `txNonce != expectedNonce`: [3](#0-2) 

With the keeper nonce now at `msg.Nonce`, the attacker can immediately submit another transaction with nonce `msg.Nonce`, which passes the check.

### Impact Explanation

**High.** This is a nonce verification bypass. After any failed contract creation, the sender's nonce is decremented by 1. The attacker can:

1. Reuse nonce `N` for a second transaction, breaking the one-use-per-nonce invariant.
2. Deploy a contract at the same `CREATE` address as a previously failed attempt (same sender + same nonce → same `crypto.CreateAddress` result), enabling address squatting or overwriting a slot that other contracts expected to be empty.
3. Repeatedly trigger the bug to keep the nonce at a fixed value, preventing the account's nonce from advancing and disrupting dependent protocol logic.

### Likelihood Explanation

**High.** Any unprivileged user can trigger this by submitting a contract creation transaction whose constructor reverts (e.g., a single `REVERT` opcode). The ante-handler accepts the transaction (nonce and fee checks pass before execution), `evm.Create()` fails during execution, and the bug fires unconditionally. No special privileges, governance access, or validator collusion are required.

### Recommendation

Guard the `nestedCreates` calculation against the revert case. When `evm.Create()` fails and reverts, `afterCreateNonce` equals `msg.Nonce` (no net increment), so the correct nonce to restore is `oldNonce`:

```go
afterCreateNonce := stateDB.GetNonce(sender)
if afterCreateNonce > msg.Nonce {
    nestedCreates := afterCreateNonce - msg.Nonce - 1
    stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
} else {
    // evm.Create() failed and reverted; restore the ante-handler nonce
    stateDB.SetNonce(sender, oldNonce, tracing.NonceChangeUnspecified)
}
```
