### Title
`starknet_estimateFee` / `starknet_simulateTransactions` / `starknet_estimateMessageFee` with `block_id = Pending` passes the committed block number N as `block_context_block_number` instead of the pending block number N+1, causing the execution context to expose the wrong block number to contracts - (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When any of the execution-based RPC methods (`starknet_estimateFee`, `starknet_simulateTransactions`, `starknet_estimateMessageFee`) is called with `block_id = Pending`, the code correctly overlays the pending state diff on top of the committed state, but incorrectly uses the latest **committed** block number N as the `block_context_block_number`. The pending block is block N+1. Any contract that reads `get_execution_info().block_info.block_number` during the simulated execution will observe N instead of N+1, producing an authoritative-looking but wrong fee estimate or simulation result.

---

### Finding Description

**Root cause ŌĆö `get_accepted_block_number` collapses `Pending` to `Latest`:** [1](#0-0) 

```rust
BlockId::Tag(Tag::Latest | Tag::Pending) => {
    get_latest_block_number(txn)?.ok_or_else(|| ErrorObjectOwned::from(BLOCK_NOT_FOUND))?
}
```

Both `Latest` and `Pending` resolve to the same committed block number N. This is intentional for state-lookup purposes (the pending block has no committed number yet), but the returned value is then reused verbatim as the **block context block number** passed to the execution engine.

**How the wrong value propagates in `estimate_fee`:** [2](#0-1) 

```rust
let maybe_pending_data = if let BlockId::Tag(Tag::Pending) = block_id {
    Some(client_pending_data_to_execution_pending_data(...))   // ŌåÉ pending state diff
} else { None };

let block_number = get_accepted_block_number(&storage_txn, block_id)?; // ŌåÉ returns N
let state_number = StateNumber::unchecked_right_after_block(block_number); // ŌåÉ N+1 Ō£ō
// block_number (= N) is forwarded as block_context_block_number Ōåō
exec_estimate_fee(
    ..., maybe_pending_data, state_number,
    block_number,   // ŌåÉ block_context_block_number = N  Ō£Ś (should be N+1)
    ...
)
```

The same pattern appears in `simulate_transactions` and `estimate_message_fee`: [3](#0-2) [4](#0-3) 

**What the module documentation says the correct pairing should be:** [5](#0-4) 

> "if you want to simulate transactions at the **beginning of block 10**, you should use `state_number = 10` and `block_context_block_number = 10`."

For the pending block (block N+1):
- `state_number = N+1` ŌĆö correctly set via `unchecked_right_after_block(N)`.
- `block_context_block_number` should be **N+1** ŌĆö but is set to **N**.

**How the wrong block number reaches contract execution:** [6](#0-5) 

`execute_transactions` constructs `CachedState` with the correct `state_number` (N+1) and the correct pending overlay, but then calls `create_block_context` with `block_context_block_number = N`. The resulting `BlockContext` carries block number N, which is what the Cairo VM exposes to contracts via the `get_execution_info` syscall.

**Analog to the external bug:**

| External (Rio) | Sequencer |
|---|---|
| `getTotalBalanceForAsset` sums committed EigenLayer shares **plus** uncommitted deposit-pool funds | `estimate_fee(Pending)` uses committed block number N as the execution context **while** overlaying uncommitted pending state diffs |
| Deposit-pool funds are not yet converted to shares at the current price | Block N is not the block in which the pending transactions will actually execute |
| Result: over/under-estimated available shares | Result: wrong `block_number` in `get_execution_info`, wrong `get_block_hash` syscall output |

---

### Impact Explanation

Any contract whose fee or execution outcome depends on `get_execution_info().block_info.block_number` (e.g., time-locked logic, epoch-based branching, block-hash-dependent randomness) will receive block number N instead of N+1 during fee estimation or simulation on the pending block. The RPC returns an authoritative-looking wrong value. A user who relies on this estimate may submit a transaction with an incorrect fee bound, causing it to revert or be under-priced.

Additionally, the `get_block_hash` syscall, which returns the hash of `block_number - 1`, will return the hash of block N-2 instead of N-1 when called from within a pending-block simulation.

**Matches:** *High ŌĆö RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

The trigger is unprivileged: any caller of `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_estimateMessageFee` with `block_id = Pending` hits this path. Contracts that read block number are common (e.g., ERC-4626-style vaults, governance contracts with voting windows, any contract using `get_execution_info` for replay protection). The discrepancy is always exactly 1 block, making it subtle and unlikely to be caught by casual testing.

---

### Recommendation

When `block_id` is `Pending`, increment the resolved block number by one before using it as `block_context_block_number`:

```rust
let block_number = get_accepted_block_number(&storage_txn, block_id)?;
let block_context_block_number = if matches!(block_id, BlockId::Tag(Tag::Pending)) {
    block_number.unchecked_next()   // N+1 for the pending block
} else {
    block_number
};
let state_number = StateNumber::unchecked_right_after_block(block_number); // unchanged
```

Apply the same fix in `simulate_transactions` and `estimate_message_fee`.

---

### Proof of Concept

1. Deploy a contract with an entry point that returns `get_execution_info().block_info.block_number`.
2. Suppose the latest committed block is N.
3. Call `starknet_estimateFee` with `block_id = Pending` targeting that entry point.
4. The execution engine receives `block_context_block_number = N`; the contract returns N.
5. Call `starknet_estimateFee` with `block_id = Latest` (also resolves to N).
6. Both calls return the same block number N ŌĆö confirming that the pending block is indistinguishable from the latest committed block inside the execution context, even though the pending block is N+1.
7. Any fee difference that would arise from the contract taking a different code path at block N+1 is invisible to the estimator, producing a wrong fee.
