## Core Finding

### `verify_path` is a membership proof, not a position proof

`verify_path` in `core/primitives/src/merkle.rs` simply recomputes the Merkle root from the item hash and the path, then checks equality with the expected root: [1](#0-0) 

It does **not** verify which leaf index the item occupies. The `MerklePath` encodes sibling hashes and Left/Right directions, but the function never checks that those directions correspond to a specific index.

A separate function `verify_path_with_index` exists that does enforce position: [2](#0-1) 

`set_state_header` uses `verify_path`, not `verify_path_with_index`.

### The check in `set_state_header` [3](#0-2) 

This proves only that `ChunkHashHeight(chunk_hash, height_included)` is **somewhere** in the Merkle tree rooted at `chunk_headers_root`. It does not prove the chunk occupies the leaf index corresponding to the requested `shard_id`.

There is also no explicit check anywhere in `set_state_header` that `chunk.shard_id() == shard_id`.

### The attack

In a 2-shard block, the Merkle tree over `[ChunkHashHeight(shard0), ChunkHashHeight(shard1)]` produces:
- `path_0 = [{hash: H(shard1_chunk_hash_height), direction: Right}]`
- `path_1 = [{hash: H(shard0_chunk_hash_height), direction: Left}]`

An attacker requesting state for `shard_id=1` sends:
- `chunk` = shard 0's chunk (valid, passes `validate_chunk_proofs`)
- `chunk_proof` = `path_0` (shard 0's Merkle proof)

The check evaluates:
```
verify_path(chunk_headers_root, path_0, ChunkHashHeight(shard0_hash, shard0_height))
= combine_hash(H(shard0_chunk_hash_height), H(shard1_chunk_hash_height)) == root  ✓
```

This passes. The existing `test_incorrect_path` test in `merkle.rs` only tests using item `i` with the path for item `(i+1)%3` — a cross-item mismatch. The attack uses item 0 with its own correct path 0, just claiming it belongs to shard 1. [4](#0-3) 

### What gets stored

After passing all checks, the header is stored: [5](#0-4) 

`StateHeaderKey(shard_id=1, sync_hash)` now holds shard 0's chunk. Downstream, `set_state_part` reads this header and extracts `prev_state_root` from it: [6](#0-5) 

State parts for shard 1 are validated against shard 0's `prev_state_root`. The wrong trie state is installed for shard 1.

### Title
Missing shard-index position check in `set_state_header` Merkle proof allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` uses `verify_path` (membership-only) instead of `verify_path_with_index` (position-enforcing) to validate the chunk Merkle proof. A peer can supply shard 0's chunk and its valid Merkle proof in response to a shard 1 state sync header request. The check passes, and shard 0's `prev_state_root` is stored as the authoritative state root for shard 1.

### Finding Description
In `ChainStateSyncAdapter::set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 394–403), the Merkle proof for the chunk is verified with `verify_path`, which only proves the `ChunkHashHeight` tuple is a leaf in the tree — not that it is the leaf at the shard-index position corresponding to the requested `shard_id`. There is no subsequent check that `chunk.shard_id() == shard_id`. An attacker peer can therefore present shard A's chunk (with its valid proof) in response to a state sync header request for shard B, and the function will accept and persist it.

### Impact Explanation
The corrupted `StateHeaderKey(shard_B, sync_hash)` entry contains shard A's `prev_state_root`. All subsequent `set_state_part` calls for shard B validate parts against shard A's trie root. The node installs shard A's state as shard B's state, causing incorrect execution, failed state root verification at finalization, or silent divergence depending on whether the node later cross-checks the installed root against the canonical block.

### Likelihood Explanation
Any peer participating in state sync can send a `ShardStateSyncResponseHeader`. No validator or operator privilege is required. The attack requires knowledge of the canonical block's chunk hashes (publicly available from block headers) and the ability to construct a valid `ShardStateSyncResponseHeader` for the wrong shard, which is straightforward given the block data.

### Recommendation
Replace `verify_path` with `verify_path_with_index` at lines 394–403 and 416–425, passing the `prev_shard_index` derived from `shard_layout.get_shard_index(shard_id)` and the total number of shards as `num_merklized_parts`. Additionally, add an explicit guard: `if chunk.shard_id() != shard_id { return Err(...) }`.

### Proof of Concept
In a multi-shard test-loop with 2 shards:
1. Obtain the canonical `ShardStateSyncResponseHeader` for shard 0 (chunk, chunk_proof, prev_chunk_header, prev_chunk_proof, receipts, state_root_node all for shard 0).
2. Call `set_state_header(shard_id=1, sync_hash, header_from_shard_0)`.
3. Assert it returns `Ok(())`.
4. Read back `StateHeaderKey(shard_id=1, sync_hash)` from `DBCol::StateHeaders`.
5. Assert the stored `prev_state_root` equals shard 0's canonical `prev_state_root`, not shard 1's.
