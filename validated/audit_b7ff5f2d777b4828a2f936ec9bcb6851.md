### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Same-Chain Signature Replay - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The OS-level hash function `compute_meta_tx_v0_hash` omits a nonce from the signed message, allowing any observer who captures a valid meta-transaction signature to replay it an unlimited number of times on the same chain. Because the OS does not enforce replay protection for this transaction type, a malicious relayer or on-chain observer can drain funds from any account that relies on the `meta_tx_v0` syscall for gasless/delegated execution.

---

### Finding Description

`compute_meta_tx_v0_hash` constructs the hash that a user must sign to authorize a v0 meta-transaction:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

Contrast this with every other transaction type. For regular account transactions, `hash_tx_common_fields` explicitly commits the nonce into the Poseidon hash state:

```cairo
poseidon_hash_update_single(item=common_fields.chain_id);
poseidon_hash_update_single(item=common_fields.nonce);   // ← nonce present
``` [2](#0-1) 

And for deprecated v0 invoke transactions, the nonce is passed as `additional_data`:

```cairo
func deprecated_get_transaction_hash{hash_ptr: HashBuiltin*}(
    ...
    additional_data_size: felt,
    additional_data: felt*,   // nonce goes here for regular txs
) -> (tx_hash: felt) {
``` [3](#0-2) 

For `compute_meta_tx_v0_hash`, `additional_data_size=0` and `additional_data=cast(0, felt*)` are hardcoded, so no nonce is ever committed. The OS-level nonce enforcement (`check_and_increment_nonce`) applies only to top-level account transactions, not to meta-transaction syscalls:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    if (tx_info.version == 0) {
        return ();   // version-0 path skips nonce entirely
    }
    ...
}
``` [4](#0-3) 

The `meta_tx_v0` syscall is therefore the only transaction-like primitive in the OS where the protocol itself provides zero replay protection at the hash level.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who signs a `meta_tx_v0` message (e.g., to authorize a gasless token transfer or approve a DeFi operation) produces a signature over a hash that is fully determined by `(contract_address, entry_point_selector, calldata, chain_id)`. Because none of these fields change between submissions, any party who observes the signed message on-chain can re-submit it an arbitrary number of times. Each replay executes the same entry point with the same calldata, allowing an attacker to, for example, repeatedly drain a token balance or repeatedly trigger an approval-based withdrawal until the victim's account is empty.

---

### Likelihood Explanation

The signed message is broadcast on-chain the first time it is used, making it trivially observable by any node operator, mempool watcher, or on-chain contract. No privileged access is required. The attacker only needs to copy the calldata and signature from the first successful transaction and re-submit them. The likelihood is therefore **high** for any account that uses the `meta_tx_v0` syscall for financial operations.

---

### Recommendation

Include a per-account, monotonically increasing nonce in the `compute_meta_tx_v0_hash` computation by passing it as `additional_data`, mirroring the pattern used by all other transaction types:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        chain_id=chain_id,
        additional_data_size=1,
        additional_data=&nonce,   // commit nonce into hash
    );
    return tx_hash;
}
```

The OS must then enforce that the nonce matches and increments the account's stored nonce after each successful `meta_tx_v0` execution, analogous to `check_and_increment_nonce` for regular transactions.

---

### Proof of Concept

1. **Victim** signs a `meta_tx_v0` message authorizing `transfer(attacker, 1000_STRK)` on their ERC-20 account. The signed hash is `H(INVOKE_PREFIX, 0, victim_contract, transfer_selector, calldata, 0, chain_id)`.
2. **Relayer** submits the transaction; it succeeds and 1000 STRK is transferred to the attacker.
3. **Attacker** observes the signature on-chain. The hash inputs are identical for any future submission because no nonce was committed.
4. **Attacker** re-submits the identical `(calldata, signature)` pair. The OS recomputes the same hash via `compute_meta_tx_v0_hash`, the signature verifies again, and another 1000 STRK is transferred.
5. Steps 4–5 repeat until the victim's balance is zero. No privileged access, no key compromise, and no additional conditions are required beyond observing the first transaction. [1](#0-0)

func deprecated_get_transaction_hash{hash_ptr: HashBuiltin*}(
    tx_hash_prefix: felt,
    version: felt,
    contract_address: felt,
    entry_point_selector: felt,
    calldata_size: felt,
    calldata: felt*,
    max_fee: felt,
    chain_id: felt,
    additional_data_size: felt,
    additional_data: felt*,
) -> (tx_hash: felt) {
    let (hash_state_ptr) = hash_init();
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=tx_hash_prefix);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=version);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=contract_address);
    let (hash_state_ptr) = hash_update_single(
        hash_state_ptr=hash_state_ptr, item=entry_point_selector
    );
    let (hash_state_ptr) = hash_update_with_hashchain(
        hash_state_ptr=hash_state_ptr, data_ptr=calldata, data_length=calldata_size
    );
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=max_fee);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=chain_id);

    let (hash_state_ptr) = hash_update(
        hash_state_ptr=hash_state_ptr, data_ptr=additional_data, data_length=additional_data_size
    );

    let (tx_hash) = hash_finalize(hash_state_ptr=hash_state_ptr);

    return (tx_hash=tx_hash);
