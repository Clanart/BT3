### Title
Unrestricted `sweepToken` and `unwrapWETH9` Allow Anyone to Steal Router-Held Tokens to an Arbitrary Recipient — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.sweepToken` and `PeripheryPayments.unwrapWETH9` are `public payable` with no caller restriction and accept a fully attacker-controlled `recipient` address. Any tokens or WETH that land on the router or liquidity adder between transactions can be redirected to an arbitrary address by any unprivileged caller.

### Finding Description

`sweepToken` and `unwrapWETH9` are inherited by both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder`. Neither function checks who the caller is or whether `recipient` is the rightful owner of the funds:

```solidity
// PeripheryPayments.sol lines 37-55
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // recipient is caller-supplied
    }
}

function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken); // recipient is caller-supplied
    }
}
``` [1](#0-0) 

Tokens land on the router in two well-defined, non-error paths:

1. **Multi-hop exact-input intermediate accumulation.** `exactInput` routes intermediate output to `address(this)` for every hop except the last. If the user sets `recipient: address(router)` on the final hop (the standard pattern before calling `unwrapWETH9`), WETH sits on the router until the unwrap call. [2](#0-1) 

2. **Native ETH partial consumption.** `pay` wraps only the exact amount needed and leaves the surplus ETH on the router, expecting a subsequent `refundETH` call. [3](#0-2) 

When these two steps are not bundled into a single atomic `multicall`, the router holds user-owned value between transactions. Because `sweepToken` and `unwrapWETH9` impose no caller check and forward the entire balance to a caller-supplied `recipient`, any third party can drain those funds.

`refundETH` sends to `msg.sender`, so an attacker calling it also captures any stranded ETH: [4](#0-3) 

The same `sweepToken`/`unwrapWETH9` surface is exposed on `MetricOmmPoolLiquidityAdder`, which also inherits `PeripheryPayments`: [5](#0-4) 

### Impact Explanation

Any tokens or ETH that reside on the router or liquidity adder between transactions are fully claimable by an unprivileged caller. The attacker supplies themselves as `recipient` and receives the entire contract balance. This is a direct, complete loss of user principal with no recovery path.

### Likelihood Explanation

The two-transaction pattern (swap → unwrap, or swap → refund) is the natural way to interact with the router without constructing a `multicall` payload. Integrators, wallets, and scripts that issue sequential transactions rather than a single `multicall` expose their users to this attack. A mempool-watching bot can backrun any transaction that leaves value on the router and drain it in the next block. No special privilege or setup is required.

### Recommendation

Restrict `sweepToken` and `unwrapWETH9` so that `recipient` must equal `msg.sender`, or add an `onlyAuthorized` / `checkCaller` guard that prevents third parties from specifying an arbitrary destination. Alternatively, remove the `recipient` parameter entirely and always send to `msg.sender`, matching the safer design already used by `refundETH`.

### Proof of Concept

```
1. Alice calls router.exactInputSingle({tokenIn: USDC, tokenOut: WETH,
       recipient: address(router), amountIn: 1000e6, ...})
   → Pool sends 0.5 WETH to address(router).

2. Alice intends to call router.unwrapWETH9(0, alice) in a second transaction.

3. Bob (attacker) observes Alice's first transaction in the mempool.
   After it confirms, Bob immediately calls:
       router.unwrapWETH9(0, bob)
   → router.balanceOf(WETH) = 0.5 WETH → withdrawn and sent to bob as ETH.

4. Alice's second transaction reverts with InsufficientWETH(0, 0).
   Bob has stolen 0.5 WETH worth of ETH. Alice's 1000 USDC is gone.
```

The same attack applies to any ERC-20 via `sweepToken(token, 0, attacker)` and to stranded native ETH via `refundETH()`.
