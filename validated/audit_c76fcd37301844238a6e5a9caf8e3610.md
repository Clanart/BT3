### Title
Arbitrary `contract_address` in `execute_meta_tx_v0` Enables Replay of Victim's Signed Meta-Transactions Without Nonce Consumption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall in the StarkNet OS accepts a fully attacker-controlled `contract_address` with no authorization check and hardcodes `nonce=0` in the meta-transaction hash without ever incrementing the target contract's nonce. Any deployed contract can invoke this syscall targeting an arbitrary victim address. If the victim has ever signed a version-0 meta-transaction (e.g., via a relayer), the attacker can replay that public signature indefinitely, causing the victim's `__execute__` entry point to re-execute the same calls — draining funds.

---

### Finding Description

`execute_meta_tx_v0` is a syscall callable by any Sierra contract. It accepts a `MetaTxV0Request` containing an attacker-supplied `contract_address` and `signature`, then:

1. Computes a meta-transaction hash via `compute_meta_tx_v0_hash` — which does **not** include a nonce (`additional_data_size=0`).
2. Constructs a new `TxInfo` with `version=0`, `nonce=0`, `max_fee=0`, and `account_contract_address=contract_address` (the victim).
3. Directly invokes `__execute__` on the victim contract with the attacker-supplied calldata and signature, bypassing `__validate__` entirely.
4. **Never checks or increments the victim contract's nonce.**

The OS enforces nonce replay protection for all regular account transactions via `check_and_increment_nonce`. No equivalent protection exists for `meta_tx_v0` targets.

**Root cause — no authorization check on `contract_address`:** [1](#0-0) 

```cairo

local selector = request.selector;
...

    write_failure_response(...);

}
// No check: is the caller authorized to act on behalf of contract_address?
```

**Root cause — nonce hardcoded to 0, never incremented:** [2](#0-1) 

```cairo

    version=0,
    account_contract_address=contract_address,

    nonce=0,          // hardcoded; victim's on-chain nonce is never read or incremented
    ...

```

**Root cause — hash excludes nonce, enabling replay:** [3](#0-2) 

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address, entry_point_selector, calldata, calldata_size, chain_id
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // nonce is NOT included in the hash
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
```

**Any contract can invoke this syscall — no caller restriction:** [4](#0-3) 

```cairo

```

### Impact Explanation

**Critical — Direct loss of funds.**

Once a victim has submitted any meta-transaction via a relayer (making the signature public on-chain), an attacker can:

- Deploy a malicious contract.
- Submit a transaction invoking the malicious contract.
- From within that contract, call `meta_tx_v0` with `contract_address=victim`, the victim's previously-seen calldata, and the victim's previously-seen signature.
- The OS computes the same hash (no nonce in hash), the victim's `__execute__` verifies the signature as valid, and executes the embedded calls (e.g., ERC-20 `transfer`).
- Because the victim's nonce is never incremented by the OS for meta-transactions, this replay can be repeated in every subsequent block until the victim's token balance is zero.

The victim has no on-chain mechanism to invalidate the replayed signature short of upgrading their account contract.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is a live, production syscall reachable by any deployed contract with no access control. Any user who has ever used a relayer-based meta-transaction flow has a public signature on-chain. The attacker requires no privileged access, no leaked key, and no operator cooperation — only the ability to deploy a contract and submit a transaction.

---

### Recommendation

1. **Include a nonce in `compute_meta_tx_v0_hash`** — add a per-target-contract nonce to `additional_data` so each signed meta-transaction can only be executed once.
2. **Check and increment the target contract's nonce** in `execute_meta_tx_v0` before calling `contract_call_helper`, analogous to `check_and_increment_nonce` used for regular account transactions.
3. **Restrict the caller** — verify that the calling contract is explicitly authorized to relay on behalf of `contract_address`, or require the target contract to opt-in to meta-transaction relaying.

---

### Proof of Concept

**Setup:**
- Victim contract `V` at address `0xVICTIM` holds 1000 STRK.
- Victim previously used a relayer: submitted a meta-transaction with calldata `[transfer(attacker, 100)]` and signature `SIG`. This is public on-chain.

**Attack:**
1. Attacker deploys malicious contract `M`.
2. Attacker submits an invoke transaction calling `M.__execute__`.
3. `M.__execute__` issues the `meta_tx_v0` syscall with:
   - `contract_address = 0xVICTIM`
   - `selector = __execute__`
   - `calldata = [transfer(attacker, 100)]`
   - `signature = SIG` (copied from the public relayer transaction)
4. OS computes `meta_tx_hash(0xVICTIM, __execute__, calldata, chain_id)` — identical to the original hash since nonce is excluded.
5. OS calls `V.__execute__` with the replayed `TxInfo`; `V` verifies `SIG` as valid and executes `transfer(attacker, 100)`.
6. Victim's nonce is **not** incremented by the OS.
7. Attacker repeats steps 2–6 in every block until `V`'s balance is zero.
