### Title
Parallel `burn.and(mint)` in `swap_migrated_token` Allows Unauthorized New-Token Mint Without Destroying Old Tokens — (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

`swap_migrated_token` uses `burn.and(mint)` to schedule both the old-token burn and the new-token mint as **parallel, independent** NEAR promises. Because `ft_on_transfer` immediately returns `U128(0)` (keeping all old tokens in the bridge) and `.detach()`es the joint promise, there is no callback that checks whether the burn succeeded before the mint executes. If the burn fails for any reason, the mint still runs, creating new tokens without destroying old ones — a direct unauthorized mint of bridged assets.

---

### Finding Description

The entry point is `ft_on_transfer` in `near/omni-bridge/src/lib.rs`:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrValue::Value(U128(0))   // bridge keeps all old tokens immediately
}
``` [1](#0-0) 

`swap_migrated_token` then constructs the joint promise:

```rust
fn swap_migrated_token(&mut self, sender_id: AccountId, old_token: AccountId, amount: U128) -> Promise {
    let new_token = self.migrated_tokens.get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);
    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
    burn.and(mint)   // ← both execute in parallel, no dependency
}
``` [2](#0-1) 

In NEAR, `Promise::and()` schedules both sub-promises to execute **independently**. Neither sub-promise's success or failure affects the other. There is no callback attached to the joint promise (it is `.detach()`ed), so no code ever checks whether the burn succeeded before the mint runs.

The `burn` function on `OmniToken` withdraws from `env::predecessor_account_id()` (the bridge contract):

```rust
fn burn(&mut self, amount: U128) {
    self.assert_controller();
    self.token.internal_withdraw(&env::predecessor_account_id(), amount.into());
}
``` [3](#0-2) 

If this call panics (gas exhaustion, old-token contract upgraded/broken, or any other runtime failure), the `mint` on the new token still executes and deposits `amount` of new tokens to the user.

**Sequence of events leading to double supply:**

1. User calls `ft_transfer_call` on the old token, sending `amount` to the bridge with `SwapMigratedToken` msg.
2. Old token contract calls `ft_on_transfer` on the bridge.
3. Bridge returns `U128(0)` — it keeps all `amount` of old tokens.
4. `burn.and(mint)` is detached; both run in parallel.
5. Burn panics (e.g., gas exhaustion, old token contract fault).
6. Mint succeeds — user receives `amount` of new tokens.
7. Bridge still holds `amount` of old tokens (never destroyed).

Net result: `amount` of old tokens remain in circulation (held by bridge, not burned) **and** `amount` of new tokens are minted to the user — total supply is inflated by `amount`.

---

### Impact Explanation

This is a **Critical** impact: direct unauthorized mint of bridged assets. The new token's total supply grows by `amount` without a corresponding reduction in the old token's supply. Bridge collateralization is broken — the new token is now unbacked by the amount that was supposed to be burned. An attacker who can reliably trigger burn failure (e.g., by tuning gas to exhaust the burn sub-promise while leaving the mint sub-promise with sufficient gas) can mint arbitrary quantities of the new bridged token for free.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who holds old (migrated) tokens. The user controls:

- **Gas attached** to `ft_transfer_call`: by attaching just enough gas for the `ft_on_transfer` frame and the mint sub-promise, but not enough for the burn sub-promise, the burn panics while the mint succeeds. NEAR's promise scheduler executes both sub-promises of `.and()` independently with their allocated gas slices.
- **Timing**: if the old token contract is ever upgraded or temporarily broken, any in-flight swap call during that window exploits the same desync.

The `ft_on_transfer` returning `U128(0)` before the burn/mint settle is the structural enabler — the bridge irrevocably keeps the old tokens at that point, so a failed burn leaves them permanently stranded while new tokens are already minted.

---

### Recommendation

Replace `burn.and(mint)` with a sequential chain: burn first, then mint only in a callback that verifies the burn succeeded.

```rust
fn swap_migrated_token(&mut self, sender_id: AccountId, old_token: AccountId, amount: U128) -> Promise {
    let new_token = self.migrated_tokens.get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);

    ext_token::ext(old_token)
        .burn(amount)
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(MINT_TOKEN_GAS)
                .swap_migrated_token_callback(new_token, sender_id, amount),
        )
}

#[private]
pub fn swap_migrated_token_callback(&mut self, new_token: AccountId, sender_id: AccountId, amount: U128) {
    // Check burn succeeded; panic (refund) if not
    require!(
        env::promise_results_count() == 1
            && matches!(env::promise_result(0), PromiseResult::Successful(_)),
        "Burn failed; aborting mint"
    );
    ext_token::ext(new_token).mint(sender_id, amount, None).detach();
}
```

Additionally, `ft_on_transfer` should propagate the promise result (return the promise index) rather than immediately returning `U128(0)` and detaching, so that a failed swap can refund the old tokens to the sender.

### Proof of Concept

1. DAO calls `migrate_deployed_token(origin_chain, old_token, new_token)` — registers the migration.
2. Attacker holds `X` units of `old_token`.
3. Attacker calls `old_token.ft_transfer_call(bridge, X, '{"SwapMigratedToken": null}')` attaching gas calibrated so that the burn sub-promise exhausts its gas slice but the mint sub-promise has sufficient gas.
4. `ft_on_transfer` returns `U128(0)` — bridge now holds `X` old tokens.
5. `burn.and(mint)` executes: burn panics (out of gas), mint succeeds.
6. Attacker's balance of `new_token` increases by `X`.
7. Bridge's balance of `old_token` remains `X` (never burned).
8. `new_token` total supply is inflated by `X` with no backing — bridge collateralization is broken. [4](#0-3) [1](#0-0)
