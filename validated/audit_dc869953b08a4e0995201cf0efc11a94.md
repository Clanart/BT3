### Title
User Tokens Permanently Locked/Burned When Transfer Amount Normalizes to Zero — (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR→EVM bridge transfer with an amount smaller than the decimal normalization factor and zero fee, their tokens are permanently burned or locked in the bridge. The zero-amount guard exists only in `sign_transfer` — after tokens are already consumed — and no cancel/refund path exists for the stuck transfer message.

### Finding Description

`normalize_amount` performs floor integer division to convert an amount from origin decimals to destination decimals:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

If `amount < 10^(origin_decimals − decimals)`, the result is `0`.

The bridge flow is:

**Step 1 — `init_transfer` / `init_transfer_internal`:** Tokens are burned (bridged tokens) or locked (native tokens) before any normalization check occurs. [2](#0-1) 

**Step 2 — `sign_transfer`:** The relayer later calls `sign_transfer`, which normalizes the amount and checks for zero:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

When `normalize_amount` returns `0`, `sign_transfer` reverts. The transfer message remains in storage permanently. There is no public `cancel_transfer` or refund function — the only removal paths are `sign_transfer_callback` (requires a successful signing) and `claim_fee_callback` (requires a proof of finalization on the destination chain), neither of which is reachable.

The code comment itself acknowledges the floor-division behavior:

> *"When fee = 0, dust stays locked/burned."* [4](#0-3) 

However, the comment refers to sub-unit *remainder* dust. The scenario here is the **entire amount** being lost when it is smaller than the normalization factor — a materially different and more severe case.

### Impact Explanation

A user's tokens are permanently burned or locked in the bridge with no recovery path. The transfer message is stuck in storage forever because `sign_transfer` will always revert for this transfer. This constitutes an irrecoverable lock of user funds in the bridge flow.

**Impact class:** Critical — Permanent freezing / irrecoverable lock of user funds in bridge flow.

### Likelihood Explanation

This is realistic for any token registered with a large decimal difference. For example:

- A token with `origin_decimals = 24` (e.g., a high-precision ERC-20) normalized to `decimals = 18` on NEAR gives `diff_decimals = 6`, threshold = `1,000,000`.
- Any user sending fewer than `1,000,000` of the smallest unit with `fee = 0` will permanently lose their tokens.
- The only input-side guard is `fee < amount` (line 554–557), which passes for `fee = 0, amount = 1`. [5](#0-4) 

The attacker-controlled entry path is the standard public `ft_transfer_call` → `init_transfer` flow, callable by any unprivileged token holder.

### Recommendation

Move the normalization check to `init_transfer` (or `init_transfer_internal`), **before** tokens are burned or locked:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
// Only then proceed to burn/lock tokens
```

This mirrors the fix recommended in the reference report: revert early rather than allowing the asset-consuming operation to proceed when the resulting transferable amount is zero.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (diff = 6, threshold = 1,000,000).
2. User calls `ft_transfer_call` on the NEAR bridge with `amount = 500_000`, `fee = 0`, recipient on EVM.
3. `init_transfer` passes the `fee < amount` check (0 < 500,000). ✓
4. `init_transfer_internal` burns/locks the 500,000 tokens. Tokens are gone.
5. Relayer calls `sign_transfer` for this transfer.
6. `normalize_amount(500_000, {origin_decimals:24, decimals:18}) = 500_000 / 1_000_000 = 0`.
7. `require!(amount_to_transfer > 0)` panics → `sign_transfer` reverts.
8. Transfer message remains in storage. No cancel function exists. User's 500,000 tokens are permanently lost. [3](#0-2) [6](#0-5) [1](#0-0)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
