### Title
Unprotected `new` Initialization Allows Any Caller to Seize DAO Role and MPC Signer Before Legitimate Deployer - (File: near/omni-bridge/src/lib.rs)

### Summary
The NEAR `omni-bridge` contract's `new` initialization function carries no caller access control. The `#[init]` attribute only prevents re-initialization once state exists; it does not restrict which account may call `new` first. If deployment and initialization occur in separate transactions — which NEAR's protocol permits — any attacker who calls `new` before the legitimate deployer becomes the DAO super-admin and sets an arbitrary `mpc_signer`, gaining full control over the bridge's signing authority and enabling unauthorized minting of bridged assets on every connected chain.

### Finding Description
`Contract::new` in `near/omni-bridge/src/lib.rs` (lines 285–314) accepts two caller-supplied parameters (`mpc_signer`, `wnear_account_id`) and unconditionally grants the DAO role and super-admin status to `env::predecessor_account_id()` — whoever calls the function first.

```rust
#[init]
pub fn new(mpc_signer: AccountId, wnear_account_id: AccountId) -> Self {
    let mut contract = Self {
        ...
        mpc_signer,
        ...
    };
    contract.acl_init_super_admin(near_sdk::env::predecessor_account_id());
    contract.acl_grant_role(Role::DAO.into(), near_sdk::env::predecessor_account_id());
    contract
}
```

In NEAR, a `DeployContract` action and a `FunctionCall` action can be placed in the same transaction (atomic) or in two separate transactions. The code imposes no constraint that forces atomicity. If a deployer submits the deploy transaction first and the init transaction second, the contract exists on-chain in an uninitialized state between those two transactions. Any account can call `new` during that window with attacker-chosen `mpc_signer` and `wnear_account_id` values.

The same pattern exists in `near/token-deployer/src/lib.rs` (lines 45–56), where `new` accepts `controller`, `dao`, and `global_code_hash` with no caller restriction, but the `omni-bridge` instance is the higher-impact target because it directly controls the MPC signing key used to authorize cross-chain transfers.

### Impact Explanation
An attacker who wins the initialization race:

1. Becomes the DAO super-admin (`acl_init_super_admin` + `acl_grant_role(Role::DAO)`), gaining unrestricted administrative control over the bridge.
2. Sets `mpc_signer` to an account they control. All subsequent `sign_transfer` and `log_metadata` calls route through this attacker-controlled signer.
3. As DAO, can call any role-gated admin function: register arbitrary provers for any chain, add arbitrary factories, set token deployer accounts, etc.
4. With a self-controlled prover registered, the attacker can supply fabricated `ProverResult::InitTransfer` data through `fin_transfer`, causing the bridge to mint NEAR-side tokens against non-existent locked funds.
5. With a self-controlled `mpc_signer`, the attacker can produce valid ECDSA signatures over arbitrary `TransferMessagePayload` structs, which EVM/Solana/StarkNet bridges accept as proof of a legitimate NEAR-side lock, enabling unbacked minting on those chains.

This constitutes direct unauthorized minting of bridged assets and permanent corruption of bridge collateralization — matching the Critical and High impact tiers in the allowed scope.

### Likelihood Explanation
Moderate-low. NEAR supports atomic deploy+init in a single transaction, and many deployment scripts use this pattern. However:
- The code itself does not enforce atomicity; nothing prevents a two-transaction deployment.
- Deployment scripts, CI pipelines, or manual deployments that separate the `DeployContract` and `FunctionCall` steps create the window.
- An attacker needs only to observe a deployed-but-uninitialized contract account on-chain (detectable by querying contract state) and submit a `new` call before the legitimate deployer's second transaction is included in a block.
- NEAR block times are ~1 second, giving a narrow but real window, especially if the deployer pauses between steps or uses a script that fails mid-way.

### Recommendation
Restrict the `new` function to a trusted caller before state is set. Options in increasing strength:

1. **Require predecessor equals current account** (self-call pattern): only callable via a batch transaction where the contract calls itself.
2. **Hardcode or derive the authorized deployer** from `env::current_account_id().get_parent_account_id()` (as `omni-token` already does correctly in `near/omni-token/src/lib.rs` lines 51–59).
3. **Enforce atomic deployment** at the SDK/tooling level by always combining `DeployContract` + `FunctionCall` in a single signed transaction in all deployment scripts.

The `omni-token` contract demonstrates the correct pattern:
```rust
require!(
    env::predecessor_account_id().as_str() == deployer_account,
    "Only the deployer account can init this contract"
);
```
Apply the same guard to `omni-bridge` and `token-deployer`.

### Proof of Concept

1. Attacker monitors NEAR for a newly deployed `omni-bridge` contract account whose state is empty (uninitialized).
2. Attacker submits a transaction calling `omni-bridge.near::new(attacker_mpc_account, attacker_wnear_account)` before the legitimate deployer's init transaction lands.
3. `#[init]` succeeds (state was empty); attacker's `predecessor_account_id` is granted DAO + super-admin.
4. Legitimate deployer's `new` call now panics ("contract is already initialized") — the bridge is permanently under attacker control.
5. Attacker calls `register_prover` (DAO-gated) to register their own prover contract for the EVM chain.
6. Attacker calls `fin_transfer` with a fabricated borsh-encoded `ProverResult::InitTransfer` payload; their prover returns success.
7. Bridge mints NEAR-side tokens to the attacker with no corresponding EVM-side lock.
8. Alternatively, attacker uses their `mpc_signer` to produce a valid signature over a crafted `TransferMessagePayload` and calls `finTransfer` on the EVM `OmniBridge`, minting ERC-20 bridge tokens with no NEAR-side deposit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
