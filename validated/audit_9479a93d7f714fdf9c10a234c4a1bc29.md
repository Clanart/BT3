### Title
`ProofOfInclusion::valid()` Does Not Verify Against a Trusted Root Hash, Enabling Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only checks internal self-consistency of the proof chain. It never compares the computed root against any externally-trusted root hash. The final guard `existing_hash == self.root_hash()` is a tautology: after the loop, `existing_hash` is always equal to `self.root_hash()` by construction. An attacker can craft a completely fabricated `ProofOfInclusion` for any arbitrary `node_hash` (claiming any key-value pair is in the tree), and `valid()` will return `true`. Any caller that relies solely on `proof.valid()` to verify DataLayer state can be deceived into accepting forged inclusion proofs.

### Finding Description

**Root cause — `valid()` is a tautological self-consistency check, not a root-anchored proof:**

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs:40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← always equals layer.combined_hash here
    }

    existing_hash == self.root_hash()      // ← always true after the loop
}
```

`root_hash()` is defined as:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs:32-38
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself, not from any trusted source
    } else {
        self.node_hash
    }
}
```

After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which the loop already asserted equals `layer.combined_hash`. `root_hash()` returns that same `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` — the final check adds no security.

**Missing parameter — no trusted root is ever supplied to `valid()`:**

The function signature is `pub fn valid(&self) -> bool`. There is no `expected_root: Hash` parameter. The proof is self-referential: it proves only that its own internal hash chain is consistent, not that the chain terminates at any root committed on-chain.

**Attacker-controlled entry path:**

The `ProofOfInclusion` struct is a public, constructible, serializable type exposed through the Python wheel:

```python
# wheel/python/chia_rs/datalayer.pyi:245
def __new__(cls, node_hash: bytes32, layers: list[ProofOfInclusionLayer]) -> ProofOfInclusion: ...
# also:
@classmethod
def from_bytes(cls, blob: bytes) -> Self: ...
```

An attacker can:
1. Choose any target `node_hash` (e.g., the hash of a key-value pair they want to falsely claim is in the store).
2. Build a single-layer `ProofOfInclusionLayer` with `other_hash = <any 32 bytes>`, `other_hash_side = Right`, and `combined_hash = calculate_internal_hash(node_hash, Right, other_hash)`.
3. Construct `ProofOfInclusion(node_hash=target, layers=[forged_layer])`.
4. Call `proof.valid()` → returns `True`.
5. `proof.root_hash()` returns the attacker-chosen `combined_hash`, which is not the real on-chain root.

Any verifier that calls only `proof.valid()` without separately asserting `proof.root_hash() == trusted_on_chain_root` will accept this forged proof.

**The analog to the external report's vulnerability class:**

Just as `createGauge4Pool()` accepted arbitrary token addresses without sanitizing them against the intended DAI/USDC/USDT/cash values, `ProofOfInclusion::valid()` accepts arbitrary proof data without validating it against a trusted root. Both functions are missing the critical external anchor check.

### Impact Explanation

**High — DataLayer Merkle proof logic lets untrusted input prove invalid state.**

A DataLayer client or verifier that calls `proof.valid()` as its sole verification step will accept forged inclusion proofs for arbitrary key-value pairs. This allows an attacker to:
- Falsely prove that a key-value pair exists in a DataLayer store when it does not.
- Falsely prove that a specific value is associated with a key when a different value is actually committed.

Since DataLayer stores are used for off-chain data with on-chain root commitments (the root hash is stored in a singleton coin), accepting a forged proof means accepting state that was never committed on-chain. This corrupts the integrity guarantee of the DataLayer and enables unauthorized state claims.

### Likelihood Explanation

The API design actively encourages the vulnerable usage pattern. The method is named `valid()`, which implies completeness. The `root_hash()` method is a separate call that a developer must know to make and compare against a trusted source. The fuzz target at `crates/chia-datalayer/fuzz/fuzz_targets/proofs_of_inclusion.rs` calls only `proof.valid()` with no root comparison, demonstrating that even the project's own test infrastructure uses the incomplete pattern. Any DataLayer client that follows this pattern is vulnerable.

### Recommendation

Add a `trusted_root: &Hash` parameter to `valid()` and replace the tautological final check with a comparison against the trusted root:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // ← compare against externally-supplied trusted root
}
```

Update all call sites — including the Python binding `py_valid`, the fuzz target, and all tests — to supply the trusted root obtained from `MerkleBlob::get_root_hash()` or from the on-chain committed value.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from hashlib import sha256

# Target: claim that this arbitrary hash is in the tree
target_node_hash = bytes([0xAB] * 32)

# Forge a single-layer proof
other_hash = bytes([0xCD] * 32)
# combined_hash = sha256(b"\x02" + other_hash + target_node_hash)  (Side.Left means other is left)
combined_hash = sha256(b"\x02" + other_hash + target_node_hash).digest()

forged_layer = ProofOfInclusionLayer(
    other_hash_side=0,          # Side.Left
    other_hash=other_hash,
    combined_hash=combined_hash,
)

forged_proof = ProofOfInclusion(
    node_hash=target_node_hash,
    layers=[forged_layer],
)

assert forged_proof.valid()          # True — forged proof passes validation
assert forged_proof.root_hash() == combined_hash  # attacker-controlled root
# No real MerkleBlob was ever consulted.
```

The forged proof passes `valid()` despite `target_node_hash` never having been inserted into any `MerkleBlob`. The `root_hash()` it reports is entirely attacker-controlled and will not match any legitimate on-chain commitment, but callers that omit the root comparison will not detect the forgery.
