Audit Report

## Title
Stranded ETH in payable router functions is silently consumed by any subsequent WETH-input swap caller via `pay()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` to subsidize WETH payments before pulling from the actual payer. Because all public entry points on `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are `payable` with no `msg.value == 0` guard when `tokenIn != WETH`, ETH sent alongside a non-WETH swap is silently accepted and left stranded. Any subsequent caller whose `tokenIn` is WETH will have their payment fully or partially subsidized by that stranded ETH, permanently losing the original depositor's funds.

## Finding Description
`PeripheryPayments.pay()` contains the following WETH branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
``` [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function unconditionally uses any ETH sitting in the contract to wrap and forward to the pool, only pulling from the actual payer for the shortfall.

ETH can accumulate through any of the `payable` entry points — `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput` on `MetricOmmSimpleRouter`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

And `addLiquidityExactShares`, `addLiquidityWeighted` on `MetricOmmPoolLiquidityAdder`: [6](#0-5) [7](#0-6) 

The `receive()` guard only blocks **direct** ETH transfers from non-WETH addresses: [8](#0-7) 

It is **not** invoked when ETH is sent alongside a `payable` function call. None of the swap or liquidity entry points check `msg.value == 0` when the input token is not WETH, so ETH sent with a non-WETH swap is silently accepted and left stranded in the contract.

## Impact Explanation
Direct, permanent loss of user principal. Once ETH is stranded, any unprivileged caller executing a WETH-input swap with `amountIn ≤ address(this).balance` pays nothing from their own wallet: the contract wraps the stranded ETH and forwards it to the pool on their behalf. The original depositor's ETH is unrecoverable. For partial amounts, the attacker receives a proportional subsidy. Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` share the same `pay()` implementation, so the attack surface covers swaps and liquidity additions alike. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation
No special privilege is required. Sending ETH with a non-WETH swap is a well-documented user error in Uniswap-style routers. The `multicall` pattern (also `payable`) actively encourages batching ETH-value calls, making it easy to omit the `refundETH()` step and leave surplus ETH stranded. The exploit is repeatable by any unprivileged address the moment stranded ETH exists in the contract.

## Recommendation
1. **Reject `msg.value` when `tokenIn != WETH`** in each swap and liquidity entry point, or add a modifier enforcing `msg.value == 0` unless the first token in the path is WETH.
2. **Do not use `address(this).balance` as a payment source in `pay()`** unless the ETH was explicitly sent in the current transaction. Gate the native-balance branch on a `msgValue` parameter passed down from the entry point rather than a raw balance read.
3. Alternatively, track ETH credited to the current call explicitly (e.g., pass `msg.value` through the call stack) and use only that tracked value in the WETH branch of `pay()`.

## Proof of Concept
```
1. Alice calls exactInputSingle{value: 1 ether}(params) where params.tokenIn = USDC.
   - Swap callback calls pay(USDC, alice, pool, amount).
   - token != WETH → ETH branch skipped; USDC pulled from Alice via transferFrom.
   - 1 ETH remains stranded: address(router).balance == 1 ether.

2. Bob calls exactInputSingle{value: 0}(params) where params.tokenIn = WETH, params.amountIn = 1 ether.
   - Swap callback calls pay(WETH, bob, pool, 1 ether).
   - nativeBalance = 1 ether >= 1 ether → contract wraps Alice's ETH, transfers WETH to pool.
   - Bob receives full swap output without spending any WETH or ETH.
   - Alice's 1 ETH is permanently lost.
```
