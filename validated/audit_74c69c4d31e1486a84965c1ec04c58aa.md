### Title
Missing `chunk.shard_id()` binding in `set_state_header` allows cross-shard state-root substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` validates that a supplied chunk is Merkle-included in the block but **never checks that `chunk.shard_id() == shard_id`**. An unprivileged peer can send a `ShardStateSyncResponseHeader` for shard A whose embedded chunk belongs to shard B. All five validation steps pass, the header is stored under `StateHeaderKey(shard_id=A, sync_hash)`, and every subsequent `set_state_part` call for shard A validates parts against shard B's `prev_state_root`, installing the wrong trie nodes.

---

### Finding Description

`set_state_header` performs five checks:

**Step 1-2 — `validate_chunk_proofs`** [1](#0-0) [2](#0-1) 

`validate_chunk_proofs` verifies the chunk's internal hash, tx-root, and receipts-root. It does **not** inspect `chunk.shard_id()`.

**Step 3a — Merkle inclusion check** [3](#0-2) 

`verify_path` confirms that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's `chunk_headers_root` Merkle tree. A standard Merkle path encodes only left/right sibling hashes; the verifier follows the path and checks the root. A chunk from shard B with its own valid proof for position B passes this check. **The position in the tree is never mapped back to `shard_id`.**

**Step 4 — Receipt proof validation** [4](#0-3) 

The receipt hash is computed as `ReceiptList(shard_id, receipts)` — using the *requested* `shard_id` (A), not the chunk's shard. An attacker who supplies valid incoming-receipt proofs for shard A (all publicly available on-chain) passes this check regardless of which chunk is embedded.

**Step 5 — `validate_state_root_node`** [5](#0-4) 

The node is validated against `chunk_inner.prev_state_root()` — shard B's root. No shard identity is checked.

**Storage** [6](#0-5) 

The header is stored under `StateHeaderKey(shard_id=A, sync_hash)` even though it contains shard B's chunk and shard B's `prev_state_root`.

---

### Impact Explanation

`set_state_part` reads the stored header and derives the state root directly from the embedded chunk: [7](#0-6) 

Because the stored header carries shard B's `prev_state_root`, every state part submitted for shard A is validated against shard B's trie root. Parts that are valid for shard B pass; parts that are valid for shard A fail. `apply_state_part` then installs shard B's trie nodes under shard A's `shard_uid`, corrupting shard A's state permanently. The node will subsequently produce incorrect block-validation results for shard A and diverge from the canonical chain.

---

### Likelihood Explanation

All data the attacker needs is publicly available:

| Required item | Source |
|---|---|
| Chunk B body + Merkle proof for position B | Any full node / block data |
| Valid incoming-receipt proofs for shard A | Any full node / block data |
| Shard B's `state_root_node` | Any full node |

The attacker only needs to be a connected peer. `set_state_header` is called on any network-delivered `ShardStateSyncResponseHeader` with no source-trust check. The victim node is in state-sync mode (catching up), so it is actively soliciting headers from peers.

---

### Recommendation

After extracting the chunk at the top of `set_state_header`, add an explicit shard-identity guard:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: reject cross-shard substitution
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same guard to `prev_chunk_header` if present.

---

### Proof of Concept

Write a cargo test in `chain/chain` that:

1. Builds a two-shard genesis and advances to a sync-hash block.
2. Calls `get_state_response_header(shard_id=1, sync_hash)` to obtain a legitimate header for shard 1.
3. Calls `set_state_header(shard_id=0, sync_hash, header_for_shard_1)`.
4. Asserts the call returns `Err(...)`.

Under the current code the call returns `Ok(())`, demonstrating that the wrong state root is accepted and stored for shard 0.

```rust
pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
```
