Audit Report

## Title
Stranded ETH from Excess `msg.value` in Payable Swap Functions Is Claimable by Any Subsequent Caller — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` settles WETH obligations by consuming `address(this).balance` — the router's entire native ETH balance — rather than only the ETH sent by the current caller. When a user calls any `payable` swap function with more ETH than the swap consumes, the surplus is silently stranded on the router with no attribution. Any subsequent caller can steal it either by executing a zero-`msg.value` WETH swap (whose cost is borne by the stranded ETH) or by calling the unrestricted `refundETH()` directly.

## Finding Description

`pay()` at [1](#0-0)  reads `address(this).balance` — the router's global ETH balance — and uses it to settle the current swap's WETH obligation. The `payer` argument is completely ignored when `nativeBalance >= value`; only `value` wei is consumed and the remainder stays on the router with no record of who deposited it.

Every public swap entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) is `payable` and routes through `_justPayCallback` → `pay()` inside the pool's swap callback: [2](#0-1) 

None of these functions refund unused ETH after the swap completes. For example, `exactOutputSingle` returns `amountIn` but never refunds `msg.value - amountIn`: [3](#0-2) 

`refundETH()` exists as a standalone public function with no access control — any caller receives the full router ETH balance regardless of who deposited it: [4](#0-3) 

The `receive()` guard (only accepts ETH from WETH) does not protect against this: payable function calls bypass `receive()` entirely, so users can freely send excess ETH via swap entry points. [5](#0-4) 

## Impact Explanation

Direct theft of user ETH principal. A victim's stranded ETH is used to fund an attacker's WETH swap at zero cost to the attacker, or is drained outright via `refundETH()`. The loss equals `msg.value_victim − amountActuallyConsumed`, which is unbounded. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation

Medium–High. `exactOutputSingle` and `exactOutput` are the primary triggers: users cannot know the exact ETH required before execution (pool state is dynamic), so they routinely send a conservative excess. `exactInputSingle` and `exactInput` are also affected when users send more ETH than `amountIn`. The attack requires only monitoring the router's ETH balance on-chain and submitting a zero-value WETH swap or a bare `refundETH()` call in the same or a subsequent block — no special privileges required.

## Recommendation

1. **Track per-call ETH delta**: Record `address(this).balance` at the start of each swap entry point and pass only the delta as available native ETH to `pay()`, rather than the full `address(this).balance`.
2. **Auto-refund surplus**: At the end of each `payable` swap function, refund any remaining `address(this).balance` to `msg.sender`.
3. **Restrict `refundETH`**: Remove it as a standalone public function or restrict it to the original depositor; expose it only through `multicall` where the caller is the same transaction initiator.

## Proof of Concept

**Step 1 — Victim strands ETH:**

Alice calls `exactOutputSingle{value: 2 ether}(params)` where the pool only requires 1 ETH of WETH input. Inside the swap callback, `_justPayCallback` calls `pay(WETH, Alice, pool, 1 ether)`. Since `address(this).balance = 2 ether ≥ 1 ether`, exactly 1 ETH is deposited as WETH and sent to the pool. The remaining **1 ETH stays on the router**. `exactOutputSingle` returns without refunding.

**Step 2a — Attacker steals via swap:**

Bob calls `exactInputSingle{value: 0}(params)` with `params.tokenIn = WETH`, `params.amountIn = 1 ether`. Inside the callback, `pay(WETH, Bob, pool, 1 ether)` is called. `address(this).balance = 1 ether` (Alice's stranded ETH) `≥ 1 ether`. Alice's 1 ETH is deposited as WETH and sent to the pool. Bob receives tokens worth 1 ETH having sent zero ETH and holding zero WETH.

**Step 2b — Attacker steals via `refundETH`:**

Bob calls `refundETH()` directly after Alice's transaction. Since there is no access control, `_transferETH(msg.sender, 1 ether)` sends Alice's 1 ETH to Bob.
