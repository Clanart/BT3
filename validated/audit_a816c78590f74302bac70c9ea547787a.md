### Title
P2P Sync State Diff Accepted Without `state_diff_commitment` Hash Verification, Corrupting Global Root and Block Hash — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

The P2P sync client assembles a `ThinStateDiff` from peer-supplied chunks and validates only that the total chunk count equals `header.state_diff_length`. It never computes `calculate_state_diff_hash` on the assembled diff and compares the result to the authenticated `header.state_diff_commitment` that is already stored from the signed block header. A single malicious peer can therefore inject arbitrary storage values, class hashes, or nonces — as long as the total element count is correct — and the corrupted diff is written to storage, propagated to the Patricia-trie committer, and used to derive the global root and block hash.

---

### Finding Description

**Analog mapping.** In the external report, `_graduate()` calls `addLiquidityETH` with `minAmountToken = X1` (the exact desired amount). An attacker deposits 1 wei of WETH directly into the pair and calls `sync()`, shifting the reserve by a tiny amount. The strict equality check then fails. The invariant that was assumed to hold — "no one can pre-seed the pool" — was never enforced before the critical operation.

In the sequencer the same structural gap exists:

| External | Sequencer |
|---|---|
| Pair reserve is the authoritative value | `state_diff_commitment` in the signed header is the authoritative value |
| Attacker calls `sync()` to inject 1 wei | Malicious peer sends chunks with correct `len()` but wrong field values |
| `addLiquidityETH` uses strict `minOut = X1` | Commitment manager uses the stored diff directly to compute the Patricia root |
| Missing pre-check: `skim()` + reserve comparison | Missing post-check: `calculate_state_diff_hash(result) == header.state_diff_commitment` |

**Root cause — `parse_data_for_block` in `state_diff.rs`.**

The function reads `target_state_diff_len` from the stored header and loops until `current_state_diff_len == target_state_diff_len`. After the loop it calls `validate_deprecated_declared_classes_non_conflicting` and returns `Ok(Some((result, block_number)))`. There is no call to `calculate_state_diff_hash` and no comparison to the header's `state_diff_commitment`. [1](#0-0) 

The header that was stored earlier carries the authenticated `state_diff_commitment` (field 7 of `SignedBlockHeader`, covered by the consensus signatures): [2](#0-1) 

The commitment is persisted into the block header table and is therefore readable at the time the state diff is assembled: [3](#0-2) 

**Downstream propagation.** The corrupted `ThinStateDiff` is written to storage unconditionally: [4](#0-3) 

The commitment manager then reads this diff from storage and submits it to the committer: [5](#0-4) 

The committer computes the Patricia root from the diff and stores it as the authoritative global root: [6](#0-5) 

The same diff is also used to compute `state_diff_commitment` inside `calculate_block_commitments`, which feeds `PartialBlockHashComponents` and ultimately the block hash: [7](#0-6) [8](#0-7) 

The same gap exists in the central-sync path and is explicitly acknowledged: [9](#0-8) 

---

### Impact Explanation

A syncing node that accepts a peer-supplied state diff with correct length but wrong content will:

1. Store wrong storage values, nonces, and class hashes — all RPC calls (`starknet_getStorageAt`, `starknet_getStateUpdate`) return wrong authoritative data.
2. Compute a wrong global root — the Patricia trie is built from the corrupted diff, so `new_root` in every subsequent block header is wrong.
3. Compute a wrong `state_diff_commitment` inside `calculate_block_commitments` — the block hash derived from `PartialBlockHashComponents` diverges from the canonical chain.
4. Feed wrong state to SNOS proof inputs — `CommitmentStateDiff` and `AccessedKeys` are derived from the stored diff, so any proof generated for that block is invalid.

This matches **Critical — Wrong state, storage value, or revert result from blockifier/syscall/execution logic for accepted input** and **High — RPC execution or pending view returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

Any peer in the P2P network can send state diff chunks. The only structural constraint enforced today is that the total `len()` of all chunks equals `target_state_diff_len`. A peer can satisfy this constraint while substituting arbitrary storage values, nonces, or class hashes. No cryptographic check prevents this. The `state_diff_commitment` that would catch the substitution is already present in storage but is never consulted.

---

### Recommendation

After the assembly loop in `parse_data_for_block`, retrieve the stored `state_diff_commitment` from the header and verify the assembled diff against it before returning:

```rust
// After the assembly loop, before returning Ok(Some(...)):
let stored_commitment = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("header must exist")
    .state_diff_commitment
    .ok_or(P2pSyncClientError::OldHeaderInStorage {
        block_number,
        missing_field: "state_diff_commitment",
    })?;

let computed_commitment = calculate_state_diff_hash(&result);
if computed_commitment != stored_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffCommitment {
        block_number,
        expected: stored_commitment,
        got: computed_commitment,
    }));
}
```

Apply the same fix in `store_state_diff` in `apollo_central_sync/src/lib.rs` to resolve the acknowledged TODO. [10](#0-9)

### Proof of Concept

1. A syncing node connects to a malicious peer.
2. The malicious peer sends a valid signed `BlockHeader` for block N (obtained from any honest peer). The header contains the correct `state_diff_commitment` and `state_diff_length = K`.
3. The malicious peer then serves the state diff stream for block N. It sends `K` elements (satisfying the length check) but substitutes one storage entry — e.g., changes the ERC-20 balance of address `A` from `V_correct` to `V_attacker`.
4. `parse_data_for_block` accepts the diff: `current_state_diff_len == target_state_diff_len` and no duplicate deprecated classes.
5. `write_to_storage` calls `append_state_diff(block_number, corrupted_diff)`.
6. The commitment manager submits the corrupted diff to the committer; the committer builds the Patricia trie from it and stores a wrong global root.
7. `starknet_getStorageAt(A, slot)` on the synced node returns `V_attacker`. `starknet_getStateUpdate(block_N)` returns a `new_root` that does not match the canonical chain. Any proof generated for block N is invalid because the SNOS input state is wrong.

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
