### Title
`initTransfer1155` Accepts ERC1155 Transfers Without Verifying `logMetadata1155` Was Called, Causing Irrecoverable Lock of User Funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`initTransfer1155` in `OmniBridge.sol` does not verify that `logMetadata1155` was previously called for the `(tokenAddress, tokenId)` pair before accepting ERC1155 tokens into the bridge. A user can initiate an ERC1155 bridge transfer without first registering the token's `multiTokens` mapping, causing the ERC1155 tokens to be irrecoverably locked in the bridge contract because the NEAR bridge cannot process the corresponding `InitTransfer` event.

### Finding Description
The ERC1155 bridging flow requires two distinct steps:

**Step 1 — Registration** (`logMetadata1155`): Sets `multiTokens[deterministicToken] = {tokenAddress, tokenId}` and emits a `LogMetadata` event. This is the prerequisite that allows the NEAR bridge to know about the token and process incoming transfers.

**Step 2 — Transfer** (`initTransfer1155`): Transfers ERC1155 tokens to the bridge and emits an `InitTransfer` event with `deterministicToken` as the token address.

The vulnerability is that `initTransfer1155` performs **no check** that `multiTokens[deterministicToken]` is populated before accepting the ERC1155 tokens:

```solidity
// OmniBridge.sol lines 439-490
function initTransfer1155(
    address tokenAddress,
    uint256 tokenId,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

    // Tokens transferred unconditionally — no check that multiTokens[deterministicToken] is set
    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");

    emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, ...);
}
```

When the relayer subsequently calls `fin_transfer` on NEAR with proof of this `InitTransfer` event, the NEAR bridge's `fin_transfer_callback` attempts to look up the token's decimals:

```rust
// near/omni-bridge/src/lib.rs lines 715-718
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

Since `logMetadata1155` was never called, the token is not registered in `token_decimals`, and the NEAR transaction panics and rolls back. The ERC1155 tokens remain locked in the EVM bridge.

**Recovery is blocked on all paths:**

1. **`finTransfer` on EVM** requires a valid MPC signature (`ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress`). MPC signatures are only generated by the NEAR bridge's `sign_transfer` for NEAR→EVM outbound transfers. No such signature exists for a failed EVM→NEAR transfer.

2. **`fin_transfer` on NEAR** is gated by `#[trusted_relayer]`, so the user cannot call it directly to retry.

3. **No admin rescue function** exists in `OmniBridge.sol` to recover stuck ERC1155 tokens.

The only recovery path is: user calls `logMetadata1155`, then `deploy_token` on NEAR, then asks the relayer to retry `fin_transfer`. This depends entirely on relayer cooperation and is not guaranteed.

### Impact Explanation
ERC1155 tokens sent via `initTransfer1155` without a prior `logMetadata1155` call are permanently locked in the `OmniBridge` contract. There is no on-chain mechanism for the user to recover them unilaterally. This matches the **Critical** impact class: irrecoverable lock of user funds in bridge flows.

### Likelihood Explanation
`logMetadata1155` and `initTransfer1155` are both permissionless public functions. Any unprivileged user can call `initTransfer1155` directly without calling `logMetadata1155` first — the contract imposes no ordering constraint. A user unfamiliar with the two-step registration requirement (which is not enforced on-chain) will lose their ERC1155 tokens. The `evm/SECURITY.md` does not list this as a known or accepted issue.

### Recommendation
Add a guard in `initTransfer1155` that requires the `multiTokens` mapping to be populated before accepting tokens:

```solidity
function initTransfer1155(
    address tokenAddress,
    uint256 tokenId,
    uint128 amount,
    uint128 fee,
    uint128 nativeFee,
    string calldata recipient,
    string calldata message
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    if (fee >= amount) {
        revert InvalidFee();
    }

    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

    // ADD: require logMetadata1155 was called first
    if (multiTokens[deterministicToken].tokenAddress == address(0)) {
        revert ERC1155MappingMismatch(); // or a new dedicated error
    }

    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
    ...
}
```

### Proof of Concept

1. Alice holds ERC1155 token `(tokenAddress=0xABC, tokenId=7)`.
2. Alice calls `initTransfer1155(0xABC, 7, 10, 0, 0, "alice.near", "")` **without** first calling `logMetadata1155(0xABC, 7)`.
3. `deriveDeterministicAddress(0xABC, 7)` → `deterministicToken = 0xDEF` (hash-derived address).
4. `IERC1155(0xABC).safeTransferFrom(alice, bridge, 7, 10, "")` succeeds — 10 tokens now held by bridge.
5. `InitTransfer` event emitted with `tokenAddress = 0xDEF`.
6. Relayer submits proof of this event to NEAR bridge via `fin_transfer`.
7. NEAR `fin_transfer_callback` executes: `self.token_decimals.get(&OmniAddress::Eth(0xDEF))` returns `None` → panics with `TokenDecimalsNotFound` → entire NEAR transaction rolls back.
8. `multiTokens[0xDEF]` on EVM is `{address(0), 0}` — the mapping was never set.
9. `finTransfer` on EVM cannot be called without a valid MPC signature, which the NEAR bridge will never produce for this unregistered token.
10. Alice's 10 ERC1155 tokens are permanently locked in `OmniBridge`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
