### Title
ERC1155 Tokens Permanently Locked in OmniBridge When Recipient Does Not Implement `IERC1155Receiver` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`finTransfer` in `OmniBridge.sol` uses `IERC1155.safeTransferFrom` to deliver ERC1155 tokens to the recipient. If the recipient is a contract that does not implement `IERC1155Receiver`, the call always reverts. Because the recipient address is immutably encoded in the MPC-signed payload, the transfer can never succeed, and the ERC1155 tokens held by the bridge are permanently irrecoverable.

---

### Finding Description

In `finTransfer`, when the incoming transfer involves an ERC1155 token (identified by a non-zero `multiToken.tokenAddress`), the bridge executes:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
``` [1](#0-0) 

Per the ERC1155 standard, `safeTransferFrom` calls `onERC1155Received` on the recipient if it is a contract. If the recipient contract does not implement `IERC1155Receiver`, the call reverts with no fallback path.

The `payload.recipient` is part of the MPC-signed `TransferMessagePayload`: [2](#0-1) 

Once the MPC has signed a payload containing a specific `recipient`, that address is fixed. No mechanism exists to re-route the transfer to a different address. The nonce guard (`completedTransfers[payload.destinationNonce] = true`) is set before the transfer: [3](#0-2) 

Although the revert rolls back the nonce flag (since the whole transaction reverts), the underlying problem remains: every subsequent attempt to call `finTransfer` with the same signed payload will hit the same `safeTransferFrom` revert. The ERC1155 tokens locked in the bridge during the original `initTransfer1155` call: [4](#0-3) 

...have no alternative release path and are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

The ERC1155 tokens deposited into `OmniBridge` via `initTransfer1155` are held by the contract. When the user attempts to bridge them back to EVM with a contract recipient that lacks `IERC1155Receiver`, `finTransfer` reverts unconditionally. The NEAR-side tokens were already burned to initiate the return leg. The EVM-side ERC1155 tokens remain locked in the bridge with no admin escape hatch or alternative withdrawal function, resulting in permanent loss.

---

### Likelihood Explanation

**Medium-High.** Many widely-used contract wallets, multisigs (e.g., older Gnosis Safe versions), protocol vaults, and DAO treasuries do not implement `IERC1155Receiver`. A user who specifies any such address as the EVM recipient when bridging ERC1155 tokens back from NEAR will trigger this condition. No special attacker knowledge is required — it is a normal user action.

---

### Recommendation

Replace `safeTransferFrom` with a pattern that does not enforce the receiver callback on the recipient side. Since ERC1155 has no plain `transferFrom`, the recommended approach is to use a **pull-based withdrawal pattern**: credit the amount to the recipient in an internal mapping and let the recipient call a `claim` function. Alternatively, wrap the `safeTransferFrom` in a `try/catch` and, on failure, store the pending transfer for the recipient to claim later:

```solidity
try IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
) {
    // success
} catch {
    pendingClaims[payload.recipient][multiToken.tokenAddress][multiToken.tokenId] += payload.amount;
}
```

This ensures the nonce is consumed and the transfer is finalized on-chain, while allowing the recipient to retrieve tokens via a separate `claim` call.

---

### Proof of Concept

1. Alice holds ERC1155 token `(tokenAddress=0xABC, tokenId=1, amount=100)` on EVM.
2. Alice calls `initTransfer1155(0xABC, 1, 100, ...)` — tokens are transferred to `OmniBridge` via `safeTransferFrom`.
3. NEAR side mints 100 wrapped tokens to Alice's NEAR account.
4. Alice initiates a return transfer on NEAR, specifying `recipient = 0xVAULT` (a contract without `IERC1155Receiver`).
5. MPC signs the `TransferMessagePayload` with `recipient = 0xVAULT`.
6. Anyone calls `finTransfer(sig, payload)`.
7. Execution reaches line 324: `IERC1155(0xABC).safeTransferFrom(bridge, 0xVAULT, 1, 100, "")`.
8. `0xVAULT` does not implement `onERC1155Received` → call reverts.
9. The signed payload is immutable; every retry with the same payload reverts identically.
10. The 100 ERC1155 tokens remain locked in `OmniBridge` forever with no recovery path.
