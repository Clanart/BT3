### Title
`charge_fee()` Does Not Enforce That `actual_fee` Is Bounded by Actual Gas Consumed — Sequencer Can Charge Up to `max_fee` Regardless of Real Usage - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `charge_fee()`, the `actual_fee` transferred from the user to the sequencer is loaded from a sequencer-controlled hint (`%{ LoadActualFee %}`) and is only constrained to be `<= max_fee`. There is no OS-enforced constraint that `actual_fee <= actual_gas_consumed * price_per_unit`. The `remaining_gas` value — which is precisely tracked throughout validate and execute phases — is never passed to `charge_fee` and never used to bound the charged amount. A sequencer can set `actual_fee = max_fee` for every transaction regardless of actual gas usage, and the OS proof will remain valid.

---

### Finding Description

In `execute_invoke_function_transaction`, the OS carefully tracks gas consumption:

```cairo
let initial_user_gas_bound = get_initial_user_gas_bound(common_tx_fields=common_tx_fields);
let remaining_gas = initial_user_gas_bound;

// Validate.
with remaining_gas {
    cap_remaining_gas(max_gas=VALIDATE_MAX_SIERRA_GAS);
    let pre_validate_gas = remaining_gas;
    run_validate(...);
}
let validate_gas_consumed = pre_validate_gas - remaining_gas;
tempvar remaining_gas = initial_user_gas_bound - validate_gas_consumed;

// Execute.
with remaining_gas {
    cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
    non_reverting_select_execute_entry_point_func(...);
}

// Charge fee — remaining_gas is NOT passed here.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
```

After execution, `remaining_gas` encodes exactly how much L2 gas was consumed. However, `charge_fee` receives no information about `remaining_gas`. Inside `charge_fee`:

```cairo
local low_actual_fee;
%{ LoadActualFee %}   // sequencer-supplied hint, unconstrained by gas consumed
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    amount=Uint256(low=low_actual_fee, high=0),
);

// Verify that the charged amount is not larger than the transaction's max_fee field.
assert_nn_le(calldata.amount.low, max_fee);
```

The only Cairo-enforced constraint is `actual_fee <= max_fee`. The `max_fee` is computed as:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

This is the user's worst-case budget, not the actual cost. The OS proof enforces no relationship between `actual_fee` and actual gas consumed. The same pattern applies to `execute_deploy_account_transaction` and `execute_declare_transaction`. The developers themselves acknowledge this gap with a TODO at line 808:

```cairo
// TODO(Yoni): calculate the gas consumed and use it to charge fee (for all transactions).
```

---

### Impact Explanation

**Critical — Direct loss of funds.**

A sequencer can set `low_actual_fee = max_fee` for every transaction in every block. The OS proof will be valid because the only enforced constraint (`actual_fee <= max_fee`) is satisfied. The ERC20 transfer executes on-chain, permanently moving the excess tokens from the user to the sequencer. Users have no recourse: the proof attests to the correctness of the block, including the fee transfer. The excess `max_fee - fair_fee` is never returned to the user — a direct analog to the original report's "excess ETH not refunded."

---

### Likelihood Explanation

High. The exploit requires no special access: any sequencer producing a valid block can set `actual_fee = max_fee` for all transactions. The hint `%{ LoadActualFee %}` is sequencer-controlled and the OS imposes no lower bound relative to gas consumed. The attack is silent, produces a valid proof, and scales linearly with transaction volume.

---

### Recommendation

Pass `remaining_gas` (post-execution) into `charge_fee` and enforce:

```
actual_fee <= actual_l2_gas_consumed * l2_gas_price
           + actual_l1_gas_consumed * l1_gas_price
           + actual_l1_data_gas_consumed * l1_data_gas_price
```

where `actual_l2_gas_consumed = initial_user_gas_bound - remaining_gas`. This mirrors the mitigation in the original report: compute the exact amount owed and enforce it as a ceiling, not just `max_fee`.

---

### Proof of Concept

1. User submits an invoke transaction with `l2_gas_bounds.max_amount = 1_000_000` and `l2_gas_bounds.max_price_per_unit = 1000`. `max_fee = 1_000_000_000`.
2. The transaction actually consumes 1,000 L2 gas units. Fair fee = `1,000 * 1,000 = 1_000_000`.
3. Sequencer sets `low_actual_fee = 1_000_000_000` (= `max_fee`) in the `%{ LoadActualFee %}` hint.
4. `assert_nn_le(1_000_000_000, 1_000_000_000)` passes.
5. ERC20 transfer of `1_000_000_000` tokens executes from user to sequencer.
6. OS proof is generated and verified as valid.
7. User is overcharged by `999_000_000` tokens with no recourse.
