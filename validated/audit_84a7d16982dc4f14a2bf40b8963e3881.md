Looking at the actual code to verify each step of the claim.

Audit Report

## Title
Surplus ETH Sent to `exactOutputSingle`/`exactInputSingle` Is Permanently Claimable by Any Caller via Unguarded `refundETH()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`exactOutputSingle` and `exactInputSingle` are `payable` and accept native ETH as WETH input. The internal `pay()` function wraps only the exact pool-requested `amountIn`, leaving any `msg.value` surplus in the router with no automatic refund. `refundETH()` carries no access control and sends the entire ETH balance to whoever calls it first, enabling any third party to steal surplus ETH left by a legitimate user.

## Finding Description

**Surplus ETH enters the router.** Both swap entry points are `payable`:

`MetricOmmSimpleRouter.sol` L67: `exactInputSingle` is `external payable`
`MetricOmmSimpleRouter.sol` L130: `exactOutputSingle` is `external payable`

Inside the swap callback, `_justPayCallback` calls `pay()` with the pool-determined `amountIn`, not with `msg.value`. In `PeripheryPayments.sol` L74–77:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
```

Only `value` (the exact pool-requested amount) is wrapped. Any `msg.value - value` remainder stays in the router. Neither `exactOutputSingle` nor `exactInputSingle` performs any surplus refund after the swap completes.

**Recovery helpers have no access control.** `refundETH()` at `PeripheryPayments.sol` L58–63:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

Any address can call this and receive the full ETH balance. `sweepToken()` and `unwrapWETH9()` similarly accept a caller-supplied `recipient` with no restriction.

**The design requires `multicall + refundETH()` but does not enforce it.** The test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` (L106–133) shows the intended pattern. However, `exactOutputSingle` is also called directly with ETH in `test_mixedNativeAndWeth_exactOutputSingle_wethForToken` (L72–104), confirming it is a supported entry point. There is no on-chain enforcement that `msg.value` equals the actual `amountIn`, and no automatic refund at the end of either function.

**Existing guards are insufficient.** The `amountInMaximum` check in `exactOutputSingle` (L145) only prevents overpayment to the pool; it does not refund the ETH difference back to the caller. The `_clearExpectedCallbackPool()` call (L146) only clears transient storage. No guard exists to prevent a third party from calling `refundETH()` between transactions.

## Impact Explanation
A user calling `exactOutputSingle{value: X}` where `X > actual amountIn` loses `X - amountIn` ETH permanently to any front-runner who calls `refundETH()` in the same block. This is a direct loss of user principal with no recovery path — the original sender has no priority claim over the surplus. The same applies to WETH output left in the router when `recipient = address(router)` is used without a same-transaction `unwrapWETH9` or `sweepToken` call.

## Likelihood Explanation
- `exactOutputSingle` is `payable` and the mixed-ETH path is explicitly tested and documented as supported.
- For `exactOutputSingle`, users cannot know the exact `amountIn` without a prior quoter call; sending a conservative `msg.value` overshoot is the natural UX pattern, routinely leaving surplus ETH.
- `refundETH()` is a public, zero-argument function with no access control — trivially callable by MEV bots monitoring router balances.
- No NatSpec warning, no on-chain guard, and no automatic refund exist to prevent this.

## Recommendation
1. **Auto-refund surplus ETH** at the end of `exactOutputSingle` and `exactInputSingle` when `tokenIn == WETH`:
   ```solidity
   uint256 surplus = address(this).balance;
   if (surplus > 0) _transferETH(msg.sender, surplus);
   ```
2. **Restrict `refundETH()`** to only send to `msg.sender` within a multicall context, or record the original depositor in transient storage and enforce it.
3. **Add a `msg.sender` guard** to `sweepToken` and `unwrapWETH9` so that only the address that initiated the multicall can specify a recipient.

## Proof of Concept
```
1. Pool state: exactOutputSingle(amountOut=1500) costs amountIn=1000 wei WETH.

2. Alice calls:
     router.exactOutputSingle{value: 2000}(
       ExactOutputSingleParams({
         tokenIn: WETH, amountOut: 1500, amountInMaximum: 3000, ...
       })
     );

3. Inside metricOmmSwapCallback → _justPayCallback → pay(WETH, Alice, pool, 1000):
     nativeBalance = 2000 >= 1000
     WETH.deposit{value: 1000}()
     WETH.transfer(pool, 1000)
     // 1000 wei ETH remains in router

4. exactOutputSingle returns. Alice's swap is complete.
   Router holds 1000 wei ETH.

5. Bob (same block) calls:
     router.refundETH();
     // _transferETH(Bob, 1000) — no access control

6. Alice lost 1000 wei ETH. Bob gained 1000 wei ETH.
   Alice cannot recover it.
```
