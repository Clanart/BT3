### Title
Meta-Transaction V0 Hash Excludes Nonce, Enabling Unbounded Signature Replay - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes a transaction hash with `additional_data_size=0`, permanently omitting any nonce. The OS simultaneously hardcodes `nonce=0` in the synthesized `TxInfo`, explicitly skips nonce enforcement for version-0 transactions, and never invokes `__validate__`. The result is that the same `(contract_address, calldata, chain_id)` tuple always produces an identical hash, and any signature over that hash is valid for unlimited replays with no OS-level protection.

---

### Finding Description

**Root cause — hash construction without nonce:**

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        ...
        chain_id=chain_id,
        additional_data_size=0,          // ← nonce never hashed
        additional_data=cast(0, felt*),
    );
``` [1](#0-0) 

`deprecated_get_transaction_hash` hashes only `[prefix, version, contract_address, selector, calldata_hash, max_fee, chain_id]` — no nonce field exists in the output: [2](#0-1) 

**Compounding factor 1 — nonce hardcoded to 0 in synthesized TxInfo:**

In `execute_meta_tx_v0`, the `new_tx_info` presented to the target contract always carries `nonce=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // ← always zero, every invocation
    ...
);
``` [3](#0-2) 

**Compounding factor 2 — OS nonce enforcement explicitly skipped for version 0:**

`check_and_increment_nonce` contains an early return for any version-0 transaction:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [4](#0-3) 

**Compounding factor 3 — `__validate__` is never called:**

`execute_meta_tx_v0` calls `contract_call_helper` directly, which invokes `select_execute_entry_point_func` targeting `__execute__`. The `run_validate` path (which calls `__validate__`) is never reached: [5](#0-4) 

The target contract's `__execute__` entry point therefore receives `tx_info.nonce = 0` and `tx_info.transaction_hash = H(prefix, 0, contract_address, __execute__, calldata, 0, chain_id)` — identical values on every replay.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any contract that uses the `meta_tx_v0` syscall to authorize a financial operation (e.g., a token transfer, a withdrawal, an approval) based on a user-provided signature is permanently vulnerable to replay. Because the OS-computed hash is identical for the same `(contract_address, calldata, chain_id)` on every block, and because the OS never increments a nonce or calls `__validate__`, an attacker who observes one valid meta-tx v0 call can re-submit it an unlimited number of times. Each replay executes `__execute__` with the same authenticated hash and signature, draining funds until the target account is empty.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is a public protocol feature callable by any contract. An attacker only needs to:
1. Observe one on-chain meta-tx v0 call (fully visible in transaction calldata).
2. Re-invoke the same outer contract with the same parameters and signature.

No privileged access, key compromise, or Sybil attack is required. The replay succeeds unconditionally at the OS level.

---

### Recommendation

Include a per-account, monotonically incrementing nonce in `compute_meta_tx_v0_hash` by passing it as `additional_data`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        chain_id=chain_id,
        additional_data_size=1,
        additional_data=&nonce,   // ← include nonce in hash
    );
```

Additionally, `execute_meta_tx_v0` must read the current nonce from `contract_state_changes` for `contract_address`, pass it into the hash, and call `check_and_increment_nonce` (or an equivalent) after execution — analogous to how `execute_invoke_transaction` calls `check_and_increment_nonce` at line 311 of `transaction_impls.cairo`. [6](#0-5) 

---

### Proof of Concept

1. Deploy an account contract `A` that holds funds and whose `__execute__` verifies a signature over `get_execution_info().tx_info.transaction_hash`.
2. A legitimate caller invokes a contract `C` that issues `meta_tx_v0(contract_address=A, selector=__execute__, calldata=[transfer, victim, amount], signature=sig)`.
3. The OS computes `hash = H(INVOKE_PREFIX, 0, A, __execute__, [transfer, victim, amount], 0, chain_id)` and calls `A.__execute__` with `tx_info.transaction_hash = hash`, `tx_info.nonce = 0`, `tx_info.signature = sig`.
4. `A.__execute__` verifies `sig` over `hash` — passes. Funds transferred.
5. Attacker re-invokes `C` (or any contract) with identical parameters and `sig`.
6. OS recomputes the **same** `hash` (no nonce, no state change to `A`'s nonce). `A.__execute__` verifies `sig` over the same `hash` — passes again.
7. Step 5–6 repeats until `A` is drained.
