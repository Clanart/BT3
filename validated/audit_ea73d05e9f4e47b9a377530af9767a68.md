### Title
Unbounded `max_price_per_unit` in `compute_max_possible_fee` Causes Felt Arithmetic Overflow, Breaking Fee Enforcement - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` multiplies user-controlled `max_amount` and `max_price_per_unit` fields in raw felt arithmetic with no upper-bound validation on `max_price_per_unit`. The result is passed directly to `assert_nn_le`, which relies on the range-check builtin and requires both operands to be in `[0, 2^128 − 1]`. A transaction sender can set `max_price_per_unit` to a value that makes the product exceed `2^128`, causing the range-check to fail and the OS block proof to abort.

---

### Finding Description

**Root cause — missing upper bound on `max_price_per_unit`**

`ResourceBounds` (defined in `new_syscalls.cairo`) stores `max_amount` and `max_price_per_unit` as plain `felt` fields with no intrinsic bounds. [1](#0-0) 

During transaction-hash computation, `pack_resource_bounds` enforces `max_amount ≤ 2^64 − 1` but only checks `assert_nn(max_price_per_unit)` — a non-negativity check with **no upper bound**: [2](#0-1) 

Later, `compute_max_possible_fee` multiplies these fields directly in felt arithmetic: [3](#0-2) 

The returned `max_fee` felt is then used as the upper bound in: [4](#0-3) 

`assert_nn_le(a, b)` internally calls `assert_nn(b − a)`, which writes `b − a` into the range-check builtin. The range-check builtin only accepts values in `[0, 2^128 − 1]`. If `max_fee ≥ 2^128 + actual_fee`, the range-check fails and the entire OS execution aborts.

**Concrete overflow scenario**

| Field | Value |
|---|---|
| `max_amount` | `2^64 − 1` (maximum allowed by `pack_resource_bounds`) |
| `max_price_per_unit` | `2^65` (passes `assert_nn`, no upper bound) |
| Product in felt | `(2^64 − 1) × 2^65 = 2^129 − 2^65 ≈ 2^129` |
| `actual_fee` | any small value, e.g. `1000` |
| `max_fee − actual_fee` | `≈ 2^129 − 1000 ≥ 2^128` → **range-check fails** |

The transaction hash computation (`pack_resource_bounds`) accepts `max_price_per_unit = 2^65` because `assert_nn(2^65)` passes. The OS therefore accepts the signed transaction as structurally valid, then aborts during fee enforcement.

---

### Impact Explanation

When the OS aborts inside `charge_fee` → `assert_nn_le`, the entire block proof fails. The sequencer cannot produce a valid STARK proof for any block that contains such a transaction. If the sequencer's off-chain mempool validation mirrors the OS's permissive check (`max_price_per_unit ≥ 0` only), it will admit the transaction, and every subsequent proof attempt for that block will abort — **total network shutdown** until the transaction is manually excluded.

This matches the allowed impact: **High — Network not being able to confirm new transactions**.

---

### Likelihood Explanation

- The attack requires only submitting a signed V3 transaction with `max_price_per_unit ≥ 2^65`. No privileged access is needed.
- The OS's own `pack_resource_bounds` explicitly validates `max_amount ≤ 2^64 − 1` but deliberately omits an upper bound on `max_price_per_unit`. Any sequencer that replicates the OS validation logic will admit the transaction.
- The discrepancy between the hash-time check (`assert_nn`) and the fee-time arithmetic (`assert_nn_le` on the product) is a latent inconsistency that is easy to miss in off-chain validation.

---

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds`, consistent with the existing bound on `max_amount`:

```cairo
// In pack_resource_bounds (transaction_hash.cairo):
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // ADD THIS
```

Alternatively, add a range check on the result of `compute_max_possible_fee` before passing it to `assert_nn_le`:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
assert_nn(max_fee);  // ensures max_fee ∈ [0, 2^128 − 1]
```

The bound `2^128 − 1` is appropriate because `assert_nn_le` uses the 128-bit range-check builtin.

---

### Proof of Concept

1. Craft a V3 `invoke` transaction with:
   - `max_amount` (L1 gas) = `2^64 − 1`
   - `max_price_per_unit` (L1 gas) = `2^65`
   - All other resource bounds = 0
2. Sign the transaction. `pack_resource_bounds` accepts it (`assert_nn(2^65)` passes).
3. Submit to the sequencer. The sequencer's mempool check mirrors the OS check and admits it.
4. Sequencer builds a block containing this transaction and calls the OS prover.
5. OS executes `compute_max_possible_fee`:
   - `max_fee = (2^64 − 1) × 2^65 = 2^129 − 2^65`
6. OS executes `assert_nn_le(actual_fee, max_fee)`:
   - Internally: `assert_nn(max_fee − actual_fee)` = `assert_nn(2^129 − 2^65 − actual_fee)`
   - `2^129 − 2^65 − actual_fee ≥ 2^128` → **range-check builtin rejects the value**
7. OS proof generation aborts. Block cannot be finalized. Network halts. [5](#0-4) [6](#0-5) [2](#0-1)

```text
// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
