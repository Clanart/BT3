### Title
Missing Duplicate-Key/Hash Validation in `MerkleBlob::batch_insert` Allows DataLayer Tree Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. An attacker-controlled batch containing repeated `KeyId` or `Hash` values is written directly into the blob without any uniqueness check, corrupting the Merkle tree structure, invalidating root hashes, and enabling forged proofs of inclusion or exclusion.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) enforces two invariants before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`MerkleBlob::batch_insert` takes a different code path. When the tree already holds two or more leaves (`leaf_count > 1`), the bootstrap branch that calls `self.insert()` is skipped entirely. Every item in the input vector is then written directly via `insert_entry_to_blob` with **no duplicate check at all**:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        ...
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even when the tree has ≤1 leaves, only the first two items are popped and inserted through the checked path; all remaining items in the vector still bypass the guards. [3](#0-2) 

The Python binding `py_batch_insert` accepts arbitrary caller-supplied `keys_values` and `hashes` lists and forwards them directly to `batch_insert` without any pre-validation: [4](#0-3) 

The result is that two or more leaf nodes with the same `KeyId` (or the same `Hash`) can coexist in the blob. The `block_status_cache` maps each key to exactly one index, so the second duplicate silently overwrites the cache entry, leaving an orphaned node in the blob that is invisible to the cache but still participates in hash computation. The tree root is therefore computed over a structurally inconsistent set of nodes.

---

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

1. **Corrupted root hash.** The internal hash tree is built over all blob nodes, including the orphaned duplicate. The resulting root hash does not correspond to any valid set of key-value pairs, so any downstream consumer that trusts the root (e.g., a DataLayer proof verifier) operates on a false commitment.

2. **Forged proofs of inclusion.** `get_proof_of_inclusion` walks the blob from the cache-recorded index. Because the cache points to the second (overwriting) entry, a proof for the first (orphaned) entry can never be generated, yet the orphaned node still contributes to the root. A verifier that checks `proof.valid()` against the corrupted root will accept or reject proofs inconsistently. [5](#0-4) 

3. **`check_integrity` divergence.** The integrity checker counts leaf nodes in the blob and compares against the cache length. A duplicate key causes `leaf_count != key_to_index_cache_length`, triggering `IntegrityKeyToIndexCacheLength` — but only if `check_integrity` is explicitly called. Normal DataLayer operation does not call it on every write. [6](#0-5) 

---

### Likelihood Explanation

The Python binding is the primary entry point. Any Python caller (DataLayer node software, a peer-sync handler, or a test harness) that passes a list containing a repeated `KeyId` or `Hash` to `batch_insert` triggers the bug. No privileged role is required; the check is simply absent. The existing test suite (`test_batch_insert`) only exercises unique-key batches and does not cover the duplicate case, so the gap has not been caught by testing. [7](#0-6) 

---

### Recommendation

Add the same duplicate guards to the fast path inside `batch_insert` before calling `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing blob write ...
}
```

Alternatively, pre-deduplicate the input vector and verify it against the existing cache before entering the fast path. Add a regression test that calls `batch_insert` with a repeated key and asserts the error.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Seed the tree with 2 leaves so leaf_count > 1 and the
    // bootstrap branch in batch_insert is skipped entirely.
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &sha256(i), InsertLocation::Auto {})
            .unwrap();
    }

    // batch_insert with a duplicate key — KeyId(99) appears twice.
    // Neither entry passes through contains_key(), so both are written.
    blob.batch_insert(vec![
        ((KeyId(99), ValueId(1)), sha256(99)),
        ((KeyId(99), ValueId(2)), sha256(100)), // same key, different hash
    ])
    .unwrap(); // succeeds — no error raised

    // The blob now contains two leaf nodes with KeyId(99).
    // check_integrity() will report IntegrityKeyToIndexCacheLength.
    blob.calculate_lazy_hashes().unwrap();
    assert!(blob.check_integrity().is_err()); // tree is corrupted
}

        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }

        let mut batch: Vec<((KeyId, ValueId), Hash)> = vec![];

        for i in pre_inserts..(pre_inserts + count) {
