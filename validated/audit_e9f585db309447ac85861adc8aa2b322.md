### Title
Native ETH Recipient Revert Permanently Blocks `finTransfer` Finalization, Locking Source-Chain Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.finTransfer()`, when the bridged token is native ETH (`payload.tokenAddress == address(0)`), the contract delivers ETH to `payload.recipient` via a raw `.call{value: ...}("")`. If the recipient is a contract that reverts on ETH receipt, the entire `finTransfer` transaction reverts. Because the `completedTransfers` nonce is only written in the same transaction, it is also rolled back, leaving the nonce permanently unconsumed. No relayer can ever successfully finalize the transfer, and the corresponding locked tokens on NEAR are irrecoverably frozen.

---

### Finding Description

`OmniBridge.finTransfer()` handles native ETH delivery as follows:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287 — reverts with tx

// ...signature verification...

if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();            // line 322
}
``` [1](#0-0) 

The nonce guard and the ETH transfer are in the same atomic transaction. If `payload.recipient` is a contract with no `receive()` / `fallback()` function, or one that deliberately reverts, the `revert FailedToSendEther()` unwinds the entire call, including the `completedTransfers[payload.destinationNonce] = true` write at line 287.

The result is:
- The destination nonce is never consumed.
- Every subsequent relay attempt hits the same revert.
- The `FinTransfer` event is never emitted, so the NEAR-side `claim_fee` (which requires proof of that event) can never succeed.
- The pending transfer record on NEAR is never removed, and the locked tokens are never released.

There is no WETH wrapper, no fallback delivery path, and no cancel/refund mechanism on the NEAR side that can unilaterally release a pending transfer without a valid EVM-side proof. [2](#0-1) 

---

### Impact Explanation

**Permanent freezing of user funds** — the locked tokens on NEAR can never be released because `finTransfer` will always revert for that specific `(recipient, nonce)` pair, and no alternative settlement path exists. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

The recipient address is chosen by the user when initiating the transfer on NEAR. Any of the following realistic scenarios triggers the freeze:

1. **Intentional self-grief / extortion**: A user deliberately sets a reverting contract as recipient to lock their own funds and blame the protocol.
2. **Accidental**: A user specifies a multisig, DAO treasury, or DeFi contract that has no `receive()` function (common for contracts that only accept ERC-20 tokens).
3. **Malicious third-party**: An attacker who controls the recipient contract can block finalization of any transfer directed to that address, e.g., to prevent a competitor from receiving bridged ETH.

No privileged access is required. The entry point is the public `ft_on_transfer` / `init_transfer` flow on NEAR, which is callable by any token holder.

---

### Recommendation

Wrap native ETH as WETH before delivery, mirroring the fix recommended in the reference report. If the recipient cannot receive raw ETH, WETH is always transferable via `safeTransfer`. A concrete pattern:

```solidity
if (payload.tokenAddress == address(0)) {
    IWETH(wethAddress).deposit{value: payload.amount}();
    IERC20(wethAddress).safeTransfer(payload.recipient, payload.amount);
}
```

Alternatively, adopt a pull-payment model: credit the recipient's claimable balance and let them withdraw separately, so a single bad recipient cannot block protocol state.

---

### Proof of Concept

1. **NEAR side**: User calls `ft_on_transfer` with an `InitTransferMsg` whose `recipient` is `0xDeadContract` — a deployed contract whose `receive()` always reverts.
2. **NEAR side**: The bridge locks the user's tokens and stores a `TransferMessage` in `pending_transfers`.
3. **EVM side**: Relayer constructs a valid MPC-signed `TransferMessagePayload` with `tokenAddress = address(0)` and `recipient = 0xDeadContract`, then calls `OmniBridge.finTransfer(signatureData, payload)`.
4. **EVM side**: Execution reaches line 319; `0xDeadContract.call{value: amount}("")` reverts; `FailedToSendEther` is thrown; the entire transaction reverts including the `completedTransfers` write at line 287.
5. **Result**: Every retry by any relayer produces the same revert. The `FinTransfer` event is never emitted. The NEAR-side `claim_fee` callback requires `ProverResult::FinTransfer` proof, which can never be produced. The user's tokens remain locked in `pending_transfers` indefinitely with no recovery path.
