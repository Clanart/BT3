### Title
Empty `check_is_reverted` Provides No Proof Constraint on Invoke Transaction Revert Status, Enabling Malicious Prover to Suppress Valid Executions - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is completely empty. The `is_reverted` flag that gates whether an invoke transaction's `__execute__` entry point runs is set entirely by an unconstrained hint (`%{ IsReverted %}`). Because no Cairo constraint verifies this hint, a malicious prover can mark any valid invoke transaction as reverted, skip its execution, and still produce a valid proof accepted by L1. This is a direct analog to the reported lending pool bug: both use the wrong (or absent) condition to gate a critical state transition.

---

### Finding Description

In `transaction_impls.cairo`, the pre-execution revert decision for invoke transactions is:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    // Align the stack with the `if` branch to avoid revoked references.
    tempvar range_check_ptr = range_check_ptr;
    tempvar remaining_gas = remaining_gas;
    ...
}
``` [1](#0-0) 

The `is_reverted` value is provided entirely by the hint `%{ IsReverted %}`. The function called to validate it is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This function is **completely empty**. It accepts `is_reverted` as a parameter and immediately returns, imposing zero Cairo constraints. There is no assertion, no range check, no comparison against any execution result.

Contrast this with `execute_entry_point.cairo`, where the revert status from an *actual* execution is properly constrained by the Cairo proof system through the entry point's `failure_flag` return value:

```cairo
local is_reverted = entry_point_return_values.failure_flag;
``` [3](#0-2) 

The pre-execution `is_reverted` hint in `transaction_impls.cairo` has no such grounding. When `is_reverted == TRUE`, the entire `__execute__` entry point is skipped. Fee charging still proceeds unconditionally via `charge_fee`:

```cairo
// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [4](#0-3) 

The `charge_fee` function enforces only that the charged amount does not exceed `max_fee`; it does not condition on whether execution actually occurred:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [5](#0-4) 

**Analog to the external report**: The lending pool bug uses a strict debt threshold (`userDebt > DUST_THRESHOLD`) instead of the correct health factor check to gate `closeLiquidation`. Here, the OS uses an unconstrained hint instead of a verified execution result to gate `__execute__`. Both are cases of the wrong (or absent) condition controlling a critical state transition.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious prover/sequencer can:

1. Accept a valid invoke transaction from a user.
2. Set `is_reverted = TRUE` in the `%{ IsReverted %}` hint.
3. Skip the `__execute__` entry point entirely — the transaction's intended state changes (token transfers, contract calls, etc.) are never applied.
4. Still charge the user the full fee via `charge_fee`.
5. Produce a valid Cairo proof, because `check_is_reverted` imposes no constraint — the proof verifier on L1 cannot distinguish this from a legitimately reverted transaction.

The user loses fees and their transaction's effects are permanently suppressed. Because the proof is valid, L1 accepts the incorrect state root. The damage is irreversible on-chain.

At scale, a malicious sequencer can suppress all invoke transactions in a block, rendering the network unable to confirm new transactions — mapping also to the "High: network not being able to confirm new transactions" impact category.

---

### Likelihood Explanation

The ZK proof system exists precisely to make the sequencer trustless. The threat model assumes the prover may be adversarial; the proof is supposed to prevent cheating. The empty `check_is_reverted` directly defeats this guarantee for the revert-status decision of every invoke transaction. Any sequencer operator who controls proof generation can exploit this without any additional capability (no key leak, no Sybil attack, no external dependency). The entry path is a standard invoke transaction submitted by any user.

---

### Recommendation

`check_is_reverted` must impose a real Cairo constraint. The correct approach depends on the intended semantics:

**Option A** — Remove the pre-execution revert hint entirely. Always run `__execute__` and let the actual `failure_flag` from `execute_entry_point` determine the revert status. This is the most conservative fix and eliminates the unconstrained hint.

**Option B** — If skipping execution for pre-known reverts is a deliberate optimization, the hint must be verified against a committed, constrained value (e.g., a sequencer-signed execution trace commitment that is itself verified in Cairo). The empty body of `check_is_reverted` must be replaced with a real assertion.

```diff
 func check_is_reverted(is_reverted: felt) {
-    return ();
+    // is_reverted must be 0 or 1.
+    assert is_reverted * (is_reverted - 1) = 0;
+    // Additional: verify against a constrained execution result, not just a hint.
 }
```

At minimum, the boolean range check prevents non-boolean hint values, but the deeper fix requires grounding `is_reverted` in a verifiable execution result.

---

### Proof of Concept

1. Prover receives a valid invoke transaction (e.g., an ERC-20 transfer).
2. In the `%{ IsReverted %}` hint handler, the prover sets `is_reverted = 1` (TRUE).
3. `check_is_reverted(1)` is called — returns immediately, no constraint fires.
4. The `if (is_reverted == FALSE)` branch is not taken; `__execute__` is skipped.
5. `charge_fee` runs and deducts the fee from the user's account (via the ERC-20 transfer syscall on the fee token).
6. The Cairo proof is generated. No constraint was violated (the empty function added no assertions).
7. The proof is submitted to and accepted by the L1 verifier.
8. The user's intended transfer never happened; their fee is gone; the state root reflects this incorrect state permanently.
