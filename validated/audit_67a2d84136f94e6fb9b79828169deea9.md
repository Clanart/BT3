### Title
EIP-7702 Authorization Durability Mechanism Commits Delegation to Third-Party EOA Without Caller Ownership Verification - (File: x/evm/keeper/state_transition.go)

### Summary

In `ApplyTransaction`, when an EIP-7702 `SetCode` transaction carries authorizations and hooks are registered, a "durable" authorization context is created to persist the authority's nonce bump and code delegation even if the EVM call or post-hook later fails. The durable commit path (`applyDurableAuthorization`) skips all re-validation and writes the delegation unconditionally. Critically, the outer transaction sender (`msg.From`) is never required to be the same as the `authority` recovered from the authorization signature. Any unprivileged sender can include a valid pre-signed authorization from a third-party EOA and force that EOA's nonce to be bumped and its code to be set to an arbitrary delegation — permanently — even when the EVM call itself reverts.

### Finding Description

The vulnerability class from the external report is: **a resource (NFT/account) is locked/mutated on behalf of a caller who does not own it, because ownership is not verified before the mutation is applied.**

The analog in Ethermint is the EIP-7702 durable authorization path.

**Step 1 — Validation only checks the authorization signature, not the outer sender:**

In `validateAuthorization`, the `authority` is recovered from the authorization's own ECDSA signature (`auth.Authority()`). There is no check that `authority == msg.From` (the outer transaction sender). [1](#0-0) 

**Step 2 — The durable commit path skips all re-validation:**

`applyDurableAuthorization` calls `setAuthorizationDelegation` directly, bypassing `validateAuthorization` entirely. It writes `SetNonce(authority, auth.Nonce+1)` and `SetCode(authority, delegation)` unconditionally. [2](#0-1) 

**Step 3 — The durable commit is triggered even when the EVM call fails (vmErr set) or the post-hook fails:**

In `ApplyTransaction`, `commitDurableAuthorization()` is called whenever `!tmpCtxCommitted`, which is true in two cases: (a) the EVM call reverted (`res.Failed()` is true, so `commit()` is never called), and (b) the post-hook failed (same outcome). In both cases the durable authorization is committed to the parent `ctx` store. [3](#0-2) 

**Step 4 — The durable context is a sibling cache branch of the parent `ctx`, so its commit writes directly to the block-level store:**

```go
durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()
``` [4](#0-3) 

**Step 5 — The durable commit happens before the main stateDB commit in `ApplyMessageWithConfig`:**

The durable stateDB is committed inside `ApplyMessageWithConfig` at line 510, before `evm.Call` even executes. So even if the EVM call reverts and the main stateDB is discarded, the durable commit has already written to the durable context's store. [5](#0-4) 

**The missing check — no ownership verification between `msg.From` and `authority`:**

Neither `validateAuthorization`, `applyAuthorization`, nor `ApplyMessageWithConfig` verify that the outer transaction sender is the same as the recovered `authority`. This is the direct analog of the NFT crafting bug: the caller does not need to own the account being delegated. [6](#0-5) 

### Impact Explanation

An unprivileged attacker who obtains a valid pre-signed `SetCodeAuthorization` from any EOA (e.g., a victim who signed an authorization for a different purpose, or whose signed authorization was leaked) can:

1. Construct an EIP-7702 type-4 transaction with `msg.From = attacker`, `msg.To = any_address`, and `authorizationList = [victim_signed_auth]`.
2. Deliberately make the EVM call revert (e.g., send to a contract that always reverts, or send to an address with no code).
3. The durable authorization commit fires unconditionally because `tmpCtxCommitted = false`.
4. The victim's account nonce is bumped and their code is set to the attacker-chosen delegation — permanently committed to chain state.

This constitutes **unauthorized account/code mutation** of a third-party EOA: the victim's account code is overwritten with an arbitrary delegation without their consent at the time of the transaction, and their nonce is incremented. This maps to the allowed High impact: "EIP-7702 authorization... signer verification bypass enabling... unauthorized account/code mutation."

Additionally, if the victim's account already has a delegation installed (from a prior legitimate use), the attacker can overwrite it with a different delegation target, redirecting all future calls to the victim's EOA to attacker-controlled code.

### Likelihood Explanation

- EIP-7702 authorizations are signed off-chain and may be reused or leaked.
- The attacker only needs to submit a standard type-4 Ethereum transaction — no privileged access required.
- The attack is triggered by a normal `eth_sendRawTransaction` RPC call.
- The condition `commitDurableAuthorization != nil && !tmpCtxCommitted` is reached on every EVM revert when hooks are registered, which is the default production configuration.
- Likelihood: **High** — any holder of a valid pre-signed authorization can exploit this.

### Recommendation

Add an ownership check in `validateAuthorization` or in the authorization processing loop in `ApplyMessageWithConfig` to verify that the outer transaction sender (`msg.From`) matches the recovered `authority`, or alternatively restrict the durable commit to only fire when the EVM call succeeds (i.e., `!res.Failed()`). The EIP-7702 specification itself does not require durability across EVM failures for third-party authorizations; the durable path was introduced to handle the Ethermint-specific post-hook failure case, but it should not apply when the authority is not the transaction sender.

Concretely, in `ApplyMessageWithConfig`, before calling `applyAuthorization`, add:

```go
if auth.Authority() != msg.From {
    // skip: outer sender does not own this authority
    continue
}
```

Or, restrict `commitDurableAuthorization` to only be called when `res.Failed()` is false (i.e., only on post-hook failure, not on EVM revert), and add the ownership check for the non-self-delegation case.

### Proof of Concept

1. Victim (`V`) signs a `SetCodeAuthorization` for nonce `N` delegating to address `D`. The signed auth is leaked or reused.
2. Attacker (`A`) constructs a type-4 transaction:
   - `from = A`
   - `to = 0xDEAD` (a contract that always reverts with `INVALID`)
   - `authorizationList = [V_signed_auth]`
   - `nonce = A's current nonce`
3. Attacker submits via `eth_sendRawTransaction`.
4. Ante handler passes (A has sufficient balance, valid signature on the outer tx).
5. In `ApplyMessageWithConfig`:
   - `validateAuthorization` recovers `authority = V`, checks `V`'s nonce matches `N` — passes.
   - `applyAuthorization` sets `V`'s nonce to `N+1` and code to `delegation(D)` in the main stateDB.
   - The durable stateDB also writes `V`'s nonce and code to `durableAuthorizationCtx`.
   - `evm.Call` to `0xDEAD` reverts → `vmErr != nil` → `res.Failed() = true`.
   - Main stateDB is committed (the stateDB commit still runs at line 603 even on vmErr).
6. Back in `ApplyTransaction`:
   - `res.Failed()` is true → `commit()` is never called → `tmpCtxCommitted = false`.
   - `commitDurableAuthorization != nil && !tmpCtxCommitted` → `commitDurableAuthorization()` fires.
   - V's nonce bump and code delegation are written to the parent block store.
7. Result: V's EOA now has `code = delegation(D)` and `nonce = N+1` — permanently, without V's consent at transaction time. [7](#0-6) [8](#0-7) [9](#0-8)

func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
	// Verify chain ID is null or equal to current chain ID.
	if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
		return authority, core.ErrAuthorizationWrongChainID
	}
	// Limit nonce to 2^64-1 per EIP-2681.
	if auth.Nonce+1 < auth.Nonce {
		return authority, core.ErrAuthorizationNonceOverflow
	}
	// Validate signature values and recover authority.
	authority, err = auth.Authority()
	if err != nil {
		return authority, fmt.Errorf("%w: %v", core.ErrAuthorizationInvalidSignature, err)
	}
	// Check the authority account
	//  1) doesn't have code or has exisiting delegation
	//  2) matches the auth's nonce
	//
	// Note it is added to the access list even if the authorization is invalid.
	stateDB.AddAddressToAccessList(authority)
	code := stateDB.GetCode(authority)
	if _, ok := types.ParseDelegation(code); len(code) != 0 && !ok {
		return authority, core.ErrAuthorizationDestinationHasCode
	}
	if have := stateDB.GetNonce(authority); have != auth.Nonce {
		return authority, core.ErrAuthorizationNonceMismatch
	}
	return authority, nil
}

func (k *Keeper) applyAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (common.Address, error) {
	authority, err := k.validateAuthorization(auth, stateDB)
	if err != nil {
		return authority, err
	}

	// If the account already exists in state, refund the new account cost
	// charged in the intrinsic calculation.
	if stateDB.Exist(authority) {
		stateDB.AddRefund(params.CallNewAccountGas - params.TxAuthTupleGas)
	}

	k.setAuthorizationDelegation(auth, authority, stateDB)
	return authority, nil
}

// setAuthorizationDelegation writes the nonce bump and delegation code for a
// validated authorization.
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}

func (k *Keeper) ApplyTransaction(ctx sdk.Context, msgEth *types.MsgEthereumTx) (*types.EVMResult, error) {
	ethTx := msgEth.AsTransaction()
	cfg, err := k.EVMConfig(ctx, k.eip155ChainID, ethTx.Hash())
	if err != nil {
		return nil, errorsmod.Wrap(err, "failed to load evm config")
	}

	msg := msgEth.AsMessage(cfg.BaseFee)
	// snapshot to contain the tx processing and post processing in same scope
	var commit func()
	var commitDurableAuthorization func()
	tmpCtxCommitted := false
	tmpCtx := ctx
	if k.hooks != nil {
		// Create a cache context to revert state when tx hooks fails,
		// the cache context is only committed when both tx and hooks executed successfully.
		// Didn't use `Snapshot` because the context stack has exponential complexity on certain operations,
		// thus restricted to be used only inside `ApplyMessage`.
		tmpCtx, commit = ctx.CacheContext()

		// Keep the EIP-7702 authorization effects in a separate cache so they
		// survive a later EVM/post-hook failure that discards tmpCtx. Only needed
		// for txs that actually carry authorizations.
		if len(msg.SetCodeAuthorizations) > 0 {
			var durableAuthorizationCtx sdk.Context
			durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()
			cfg.DurableSetCodeAuthorizationCtx = &durableAuthorizationCtx
		}
	}

	// pass true to commit the StateDB
	res, err := k.ApplyMessageWithConfig(tmpCtx, msg, cfg, true)
	if err != nil {
		return nil, errorsmod.Wrap(err, "failed to apply ethereum core message")
	}

	logs := types.LogsToEthereum(res.Logs)

	// Compute block bloom filter
	if len(logs) > 0 {
		bloom := ethtypes.Bloom{}
		for _, log := range logs {
			bloom.Add(log.Address.Bytes())
			for _, topic := range log.Topics {
				bloom.Add(topic[:])
			}
		}
		k.SetTxBloom(tmpCtx, bloom.Big())
	}

	var contractAddr common.Address
	if msg.To == nil {
		contractAddr = crypto.CreateAddress(msg.From, msg.Nonce)
	}

	receipt := &ethtypes.Receipt{
		Type:            ethTx.Type(),
		PostState:       nil, // TODO: intermediate state root
		Logs:            logs,
		TxHash:          cfg.TxConfig.TxHash,
		ContractAddress: contractAddr,
		GasUsed:         res.GasUsed,
		BlockHash:       cfg.TxConfig.BlockHash,
		BlockNumber:     cfg.BlockNumber,
	}

	if !res.Failed() {
		receipt.Status = ethtypes.ReceiptStatusSuccessful
		// Only call hooks if tx executed successfully.
		if err = k.PostTxProcessing(tmpCtx, msg, receipt); err != nil {
			// If hooks return error, revert the whole tx.
			res.VmError = types.ErrPostTxProcessing.Error()
			k.Logger(ctx).Error("tx post processing failed", "error", err)

			// If the tx failed in post processing hooks, we should clear the logs
			res.Logs = nil
		} else if commit != nil {
			// PostTxProcessing is successful, commit the tmpCtx
			commit()
			tmpCtxCommitted = true
			// Since the post-processing can alter the log, we need to update the result
			res.Logs = types.NewLogsFromEth(receipt.Logs)
		}
	}

	// Get the tracer and add OnGasChange hook for gas refund
	leftoverGas := msg.GasLimit - res.GasUsed

	// refund gas in order to match the Ethereum gas consumption instead of the default SDK one.
	if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
		return nil, errorsmod.Wrapf(err, "failed to refund leftover gas to sender %s", msg.From)
	}

	tracer := cfg.GetTracer()
	if tracer != nil && tracer.OnGasChange != nil {
		tracer.OnGasChange(leftoverGas, 0, tracing.GasChangeTxLeftOverReturned)
	}

	totalGasUsed, err := k.AddTransientGasUsed(ctx, res.GasUsed)
	if err != nil {
		return nil, errorsmod.Wrap(err, "failed to add transient gas used")
	}

	if commitDurableAuthorization != nil && !tmpCtxCommitted {
		commitDurableAuthorization()
	}

	// reset the gas meter for current cosmos transaction
	k.ResetGasMeterAndConsumeGas(ctx, totalGasUsed)

	return res, nil
}
