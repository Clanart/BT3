Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal excess ETH left in `MetricOmmPoolLiquidityAdder` after a direct `addLiquidityExactShares` call — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` sends the contract's entire ETH balance to `msg.sender` with no access control. Because `addLiquidityExactShares` is `payable` and `pay()` wraps only the exact amount the pool requests, any ETH surplus from a direct (non-`multicall`) invocation remains in the contract and is immediately claimable by any third party via `refundETH()`.

## Finding Description

`refundETH()` is unconditionally callable by any address: [1](#0-0) 

ETH enters legitimately via `addLiquidityExactShares{value: X}(...)`. Inside `pay()`, only the exact pool-requested amount is wrapped and forwarded; any surplus stays in the contract: [2](#0-1) 

`_addLiquidity` performs no ETH refund after the pool call returns: [3](#0-2) 

The `receive()` guard only blocks bare ETH transfers from non-WETH addresses; it does not prevent ETH from entering via `payable` function calls: [4](#0-3) 

The intended safe pattern is `multicall([addLiquidityExactShares(...), refundETH()])`, which is atomic and preserves `msg.sender` via `delegatecall`. However, there is no enforcement that `addLiquidityExactShares` must be called through `multicall`: [5](#0-4) 

## Impact Explanation

Any ETH remaining in the contract after a direct `addLiquidityExactShares` call is immediately claimable by any address via `refundETH()`. The user loses the full surplus with no recovery path. This is a direct, complete loss of user principal (native ETH).

## Likelihood Explanation

MEV bots routinely monitor the mempool for exactly this pattern. A user who calls `addLiquidityExactShares{value: X}(...)` directly — a natural usage given the function is `payable` and publicly documented — and sends more ETH than the pool needs will lose the surplus in the same block. The attack requires no special privileges, no malicious pool, and no non-standard tokens.

## Recommendation

Add an automatic ETH refund at the end of `_addLiquidity`:

```solidity
function _addLiquidity(...) internal returns (...) {
    _setPayContext(...);
    try ... {
        ...
        _clearPayContext();
    } catch { ... }
    uint256 leftover = address(this).balance;
    if (leftover > 0) _transferETH(payer, leftover);
}
```

Alternatively, restrict `refundETH()` so it can only be called via `delegatecall` (i.e., from within `multicall`), preventing standalone invocations.

## Proof of Concept

1. WETH pool exists with token0 = WETH, token1 = ERC20.
2. User calls `adder.addLiquidityExactShares{value: 2 ether}(pool, salt, deltas, 2e18, 0, "")` directly (not via `multicall`). Pool only needs 1 ETH worth of WETH.
3. `pay()` wraps and forwards exactly 1 ETH; 1 ETH remains in `adder`.
4. Attacker calls `adder.refundETH()` in the same or next block.
5. Attacker receives 1 ETH. User's surplus is permanently lost.
