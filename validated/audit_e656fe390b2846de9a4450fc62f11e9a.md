### Title
Integer Wrapping in Relative Timelock Arithmetic Allows `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Bypass — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary
`check_time_locks` supports two arithmetic modes controlled by the `nowrap: bool` parameter. When `nowrap=false`, relative timelock additions use `wrapping_add` instead of `saturating_add`. A crafted CLVM spend bundle containing `ASSERT_SECONDS_RELATIVE(u64::MAX)` or `ASSERT_HEIGHT_RELATIVE(u32::MAX)` causes the addition to wrap around to a value smaller than the coin's own confirmation timestamp/height, making the "not yet elapsed" guard evaluate to `false` and the check pass unconditionally — bypassing the timelock entirely.

### Finding Description

`check_time_locks` branches on `nowrap` for every relative condition:

```rust
// nowrap=false path — wrapping arithmetic
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

With `seconds_relative = u64::MAX`:

```
unspent.timestamp.wrapping_add(u64::MAX)
  = unspent.timestamp.wrapping_sub(1)
  = unspent.timestamp − 1
```

The guard becomes `timestamp < unspent.timestamp − 1`. Because `timestamp ≥ unspent.timestamp` is always true (a coin cannot be spent before it is confirmed), this comparison is always `false`, so no error is returned and the spend is accepted regardless of how recently the coin was created.

The same wrapping bypass applies to `ASSERT_HEIGHT_RELATIVE` with `height_relative = u32::MAX`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
```

`confirmed_block_index.wrapping_add(u32::MAX) = confirmed_block_index − 1`, so the check always passes.

The tests in the same file explicitly confirm this behaviour:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- nowrap=false: bypass
)]
```

On the condition-parsing side, `sanitize_uint` with an 8-byte limit accepts `u64::MAX` as `SanitizedUint::Ok(u64::MAX)` and stores it as `seconds_relative = Some(u64::MAX)`, so the crafted value passes the parser without error.

### Impact Explanation

A coin whose puzzle emits `(ASSERT_SECONDS_RELATIVE . 0xffffffffffffffff)` is intended to be unspendable for ~584 billion years. When `check_time_locks` is called with `nowrap=false`, that coin can be spent in the very next block. This is a **timelock validation bypass** that enables unauthorized coin spend acceptance — matching the High impact category: *"timelock … validation bypass enables unauthorized spend acceptance."*

### Likelihood Explanation

`check_time_locks` is exposed as a Python binding (`py_check_time_locks`) with `nowrap` as a caller-supplied argument. Any full-node code path that invokes this binding with `nowrap=False` for a block containing the crafted spend is vulnerable. The condition value `u64::MAX` is a valid CLVM atom (8 bytes, no leading zeros required beyond canonical encoding), so it is trivially embeddable in a spend bundle by any unprivileged user. No special keys, governance access, or network-level position is required.

### Recommendation

1. Remove the `nowrap=false` / `wrapping_add` branch from `check_time_locks` entirely, or gate it behind a hard-fork height check inside the function itself rather than delegating the choice to callers.
2. If backward compatibility with pre-hard-fork blocks is required, validate that `nowrap=false` is never reachable for blocks at or above the hard-fork activation height.
3. Add a parser-level rejection for `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` values that would wrap when added to any plausible coin timestamp/height (e.g., reject values ≥ `u64::MAX − MAX_EXPECTED_TIMESTAMP`).

### Proof of Concept

Craft a CLVM spend whose condition list contains:

```
(ASSERT_SECONDS_RELATIVE . 0xffffffffffffffff)
```

Call `check_time_locks` with `nowrap=false`, any `timestamp ≥ unspent.timestamp`, and the coin record for the spending coin. The function returns `Ok(())` — the timelock is bypassed — as confirmed by the existing test case `height_relative_wrap` / `seconds_relative_wrap` in the test suite.
