### Title
Missing Class Hash Existence Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by a calling contract corresponds to a previously declared class. Any contract can replace its own class hash with an arbitrary, undeclared value. Once committed to state, all subsequent calls to that contract will fail at the compiled-class lookup stage, permanently freezing any funds the contract holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads the new `class_hash` from the request and unconditionally writes it into the contract's `StateEntry` — with no check that the hash was ever declared. The code itself contains an explicit TODO acknowledging the missing guard:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

When any subsequent call is made to the affected contract, `execute_entry_point` performs a two-step lookup:

1. `dict_read` on `contract_class_changes` for the contract's (now-invalid) class hash — returns `0` for an undeclared hash.
2. `find_element` on `compiled_class_facts_bundle.compiled_class_facts` searching for compiled class hash `0` — this element does not exist, causing the proof to fail and the contract to be permanently uncallable. [2](#0-1) 

The permission model deviation is direct: the OS enforces that a class can only be *declared* once (`prev_value=0` guard in `dict_update` for class changes), but it imposes **no symmetric constraint** on `replace_class` — the operation that changes a live contract's class. A declared-class check is enforced on the write path but absent on the upgrade path, creating an asymmetric and incomplete permission boundary.

---

### Impact Explanation

Any contract that calls `replace_class` with an undeclared hash becomes permanently uncallable at the OS level. There is no recovery path: the state transition is committed, the class hash is invalid, and no entry point can ever be dispatched again. All tokens or assets held by the contract are permanently frozen. This matches **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender:

- No special role, key, or operator access is required.
- Any user can deploy a contract (e.g., a token vault or escrow) that internally calls `replace_class` with an arbitrary felt.
- A malicious deployer can advertise the contract as legitimate, collect user deposits, then trigger the `replace_class` call to brick the contract.
- The syscall is available to all Cairo 1 contracts via the standard syscall interface; no elevated privilege is needed to invoke it.

---

### Recommendation

Before updating `contract_state_changes` in `execute_replace_class`, verify that `class_hash` exists in `contract_class_changes` (i.e., its compiled class hash is non-zero, meaning it was previously declared). The check should mirror the lookup already performed in `execute_entry_point`:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This resolves the asymmetry between the declaration path (which enforces `prev_value=0`) and the upgrade path (which currently enforces nothing about the target class).

---

### Proof of Concept

1. Attacker deploys contract `V` with a legitimate interface (e.g., a token vault). Users deposit funds.
2. Attacker calls a function in `V` that internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (never declared on-chain).
3. The OS executes `execute_replace_class`: gas is deducted, `StateEntry` for `V` is updated with `class_hash=0xdeadbeef`, no validation occurs. [3](#0-2) 
4. State is committed. Any subsequent call to `V` reaches `execute_entry_point`, which calls `dict_read(contract_class_changes, 0xdeadbeef)` → returns `0`, then `find_element(compiled_class_facts, key=0)` → element not found → proof failure / permanent revert. [4](#0-3) 
5. All funds deposited in `V` are permanently frozen with no recovery mechanism.
