### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Arbitrary Class Hash Injection - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS Cairo program does not verify that the caller-supplied class hash corresponds to a previously declared contract class. An attacker-controlled contract can call `replace_class` with an arbitrary, undeclared class hash. If the sequencer's blockifier also lacks this check (consistent with the acknowledged TODO), the OS will commit the corrupted class hash to state. Any subsequent block that attempts to execute the affected contract will cause `find_element` to panic inside the OS, making the block unprovable and halting the network.

---

### Finding Description

In `execute_replace_class`, after deducting gas, the OS reads `request.class_hash` directly from the syscall request and writes it into `contract_state_changes` without any check that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment explicitly acknowledges the missing check. The deadline of `1/1/2026` has passed (today is 2026-07-03) and the check remains absent.

When a subsequent block calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

`dict_read` on `contract_class_changes` returns `0` for an undeclared class hash (Cairo dict default). `find_element` is a **hard assertion** — it panics if the key is absent from the array. There is no graceful revert path here, unlike `search_sorted_optimistic`. A panic in the OS Cairo program means the block cannot be proven.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

Once a contract's class hash is set to an undeclared value and committed to state, every subsequent block that includes any call to that contract (including reverted calls, since `find_element` is reached before the revert path) will cause the OS program to panic. A panicking OS produces no valid proof output, making the block permanently unprovable. The sequencer cannot skip the contract call because the state already reflects the corrupted class hash. This constitutes a permanent, targeted network halt for any block touching the affected contract.

---

### Likelihood Explanation

**Medium.** The vulnerability requires two conditions:

1. The OS does not validate the class hash in `replace_class` — **confirmed** by the explicit TODO at line 898.
2. The sequencer's blockifier also does not validate the class hash in `replace_class` at the time of transaction inclusion — **plausible**, because the TODO is in the OS code (the authoritative specification), suggesting the check was never implemented in either layer.

An unprivileged attacker only needs to deploy a contract that calls `replace_class` with an arbitrary felt value and submit a transaction invoking it. No privileged access, leaked keys, or operator cooperation is required beyond normal transaction submission.

---

### Recommendation

Add an explicit validation in `execute_replace_class` that the supplied `class_hash` exists in `contract_class_changes` (i.e., has been declared in the current or a prior block) before updating the state entry. This mirrors the allowlist approach recommended in the external report: new class hashes should be rejected by default unless they are known-declared classes.

```cairo
// After reading class_hash from request:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("replace_class: class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This ensures the OS enforces the same invariant as the sequencer, closing the gap between the two layers.

---

### Proof of Concept

1. **Attacker deploys** a Cairo 1 contract containing:
   ```rust
   // pseudocode
   fn exploit(ref self: ContractState) {
       starknet::replace_class_syscall(0xdeadbeef_undeclared_hash).unwrap();
   }
   ```
2. **Attacker submits** an invoke transaction calling `exploit()`. The OS's `execute_replace_class` accepts `class_hash = 0xdeadbeef_undeclared_hash` without validation and writes it to `contract_state_changes`.
3. **Block N is proven** successfully (the `replace_class` itself does not trigger `find_element`).
4. **In Block N+1**, any transaction that calls the attacker's contract (or any contract that `call_contract` into it) causes `execute_entry_point` to call `dict_read` on `contract_class_changes` with key `0xdeadbeef_undeclared_hash`, receiving `0`. `find_element` then searches for compiled class hash `0`, finds nothing, and **panics**.
5. **Block N+1 is unprovable.** The sequencer cannot produce a valid proof. The network halts on any block touching the affected contract. [3](#0-2) [4](#0-3)
