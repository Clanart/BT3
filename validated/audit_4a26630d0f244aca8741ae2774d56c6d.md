### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

The EVM `OmniBridge.initTransfer` function uses the caller-supplied `amount` parameter directly in the emitted `InitTransfer` event without measuring the actual tokens received by the contract. For fee-on-transfer (FOT) ERC-20 tokens, the bridge vault receives `amount - transferFee` but the cross-chain message records `amount`, causing the NEAR side to mint or unlock more tokens than the bridge actually holds.

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native ERC-20 tokens (non-bridge-token, non-custom-minter path) by pulling tokens from the caller:

```solidity
// OmniBridge.sol lines 407-411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
```

Immediately after, it emits the cross-chain event using the caller-supplied `amount`:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-supplied, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
```

For a fee-on-transfer token, `safeTransferFrom` deducts a fee during the transfer, so the bridge contract receives only `amount - transferFee`. The emitted event, however, records the full `amount`. The NEAR-side bridge (or Wormhole relayer) reads this event and processes a `fin_transfer` for the full `amount`, minting or unlocking `amount` tokens to the destination recipient. The EVM vault is now short by `transferFee` per transaction.

There is no balance-before/balance-after check anywhere in `initTransfer` to detect the discrepancy. [1](#0-0) [2](#0-1) 

### Impact Explanation

Every `initTransfer` call with a FOT token creates a collateral deficit equal to the transfer fee. After N such transfers, the bridge holds `N * transferFee` fewer tokens than the sum of outstanding cross-chain claims. When users bridge back from NEAR to EVM, `finTransfer` calls `IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount)` using the full recorded amount. The last users to redeem will find the vault drained, resulting in permanent loss of funds — a direct match for the allowed impact: **balance/accounting corruption that breaks bridge collateralization**. [3](#0-2) 

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with a FOT ERC-20 token address. The bridge imposes no whitelist on which ERC-20 tokens can be deposited via the native lock path (the `else` branch is reached for any token that is neither a bridge token nor has a custom minter). A single user can trigger the bug repeatedly, and the deficit compounds with each call. [4](#0-3) 

### Recommendation

Measure the actual received amount using a balance-before/balance-after pattern:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived (cast to uint128) in place of amount for the event and cross-chain message
```

Alternatively, explicitly document that FOT tokens are unsupported and add a token allowlist enforced on-chain.

### Proof of Concept

1. Deploy or use any ERC-20 with a 1% transfer fee (e.g., a token that deducts 1% on every `transferFrom`).
2. Call `OmniBridge.initTransfer(fotToken, 1000, 0, 0, "near:alice.near", "")`.
3. The bridge receives 990 tokens (`safeTransferFrom` deducts 10 as fee).
4. The emitted `InitTransfer` event records `amount = 1000`.
5. The NEAR bridge processes the proof and mints/unlocks 1000 tokens to `alice.near`.
6. Repeat 100 times: bridge holds 99,000 tokens but 100,000 tokens are outstanding on NEAR.
7. When the 100th user tries to bridge back 1000 tokens to EVM, `finTransfer` attempts `safeTransfer(recipient, 1000)` but the vault is short — the call reverts or drains the last legitimate user's funds. [5](#0-4) [6](#0-5)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
