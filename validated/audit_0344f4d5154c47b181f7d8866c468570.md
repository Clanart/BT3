### Title
Missing `chunk.shard_id()` == `shard_id` guard in `set_state_header` enables cross-shard state-root substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for shard A that embeds a valid chunk whose `shard_id` is shard B. Because `verify_path` is position-agnostic and no explicit shard-identity check exists, the function stores the header under `StateHeaderKey(A, sync_hash)` with chunk B's `prev_state_root`. Every subsequent call to `set_state_part` for shard A then validates parts against chunk B's state root, making all legitimate shard-A parts fail. An attacker who also supplies valid shard-B parts can cause the syncing node to install shard B's trie as shard A's state, diverging from the canonical chain.

---

### Finding Description

**Entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five checks before writing to `DBCol::StateHeaders`:

**Check 1-2 — `validate_chunk_proofs`** [1](#0-0) [2](#0-1) 

`validate_chunk_proofs` verifies the chunk's internal hash, tx root, and receipts root. It never inspects `chunk.shard_id()` against the `shard_id` parameter. A fully valid chunk for shard B passes this check.

**Check 3a — `verify_path` against `chunk_headers_root`** [3](#0-2) [4](#0-3) 

`verify_path` computes `compute_root_from_path(path, hash(item)) == root`. It is purely position-agnostic: it does not check *which* leaf index the path encodes. The Merkle tree over `chunk_headers_root` is built from all shards' `ChunkHashHeight` values ordered by shard index. Supplying chunk B's hash together with the Merkle path for shard B's leaf position satisfies this check even when `shard_id == A`. [5](#0-4) 

**Check 3b — `verify_path` for `prev_chunk`** [6](#0-5) 

Same position-agnostic flaw applies to the previous chunk proof.

**Check 4 — receipt proofs** [7](#0-6) 

Receipt proofs are verified against the `shard_id` *parameter* (shard A), not `chunk.shard_id()`. The attacker supplies legitimate on-chain receipt proofs for shard A; these are public data.

**Check 5 — `validate_state_root_node`** [8](#0-7) [9](#0-8) 

`validate_state_root_node` checks `hash(&state_root_node.data) == state_root` and that `memory_usage` matches the deserialized node. The `state_root` used here is `chunk_inner.prev_state_root()` — chunk B's state root. An attacker running a full node tracking shard B has the actual root node bytes for chunk B's state and can supply a valid `StateRootNode`.

**Storage — the invariant violation** [10](#0-9) 

The header is written under `StateHeaderKey(A, sync_hash)` with chunk B's embedded data. The invariant `StateHeaderKey(A, H).chunk.shard_id() == A` is broken.

**Downstream corruption in `set_state_part`** [11](#0-10) 

`set_state_part` reads the stored header, extracts `state_root = chunk_B.prev_state_root()`, and calls `validate_state_part(shard_id_A, state_root_B, part_id, part)`. Legitimate shard-A parts do not match `state_root_B` and are rejected. If the attacker supplies valid shard-B parts, they pass validation and are stored under `StatePartKey(sync_hash, A, part_id)`.

**Downstream corruption in `set_state_finalize`** [12](#0-11) 

`set_state_finalize` calls `apply_chunk` with `RuntimeStorageConfig::new(chunk_header.prev_state_root(), true)` — chunk B's state root — and `shard_uid_A`. Shard B's trie (installed via `apply_state_part`) is applied as shard A's state. The resulting `chunk_extra` carries chunk B's post-state root as shard A's authoritative state root.

**Subsequent block processing fails** [13](#0-12) 

When the node processes the next block for shard A, `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.state_root()` (chunk B's post-state root) against `chunk_header.prev_state_root()` (shard A's canonical state root). The mismatch causes `Error::InvalidStateRoot`, and the node cannot advance for shard A.

---

### Impact Explanation

The concrete corrupted value is `StateHeaderKey(A, H).chunk.prev_state_root()` = chunk B's `prev_state_root` instead of shard A's. This has two tiers of impact:

1. **Guaranteed DoS on state sync for shard A**: All legitimate shard-A parts fail `validate_state_part` because they do not match chunk B's state root. The syncing node cannot complete state sync for shard A.

2. **State corruption (if attacker supplies shard-B parts)**: Shard B's trie is installed under shard A's `shard_uid`. `set_state_finalize` applies chunk B's transactions to shard B's state, producing a wrong `chunk_extra` for shard A. The node diverges from the canonical chain and cannot process any subsequent block for shard A.

---

### Likelihood Explanation

The attacker must:
1. Run a full node tracking shard B (to obtain the `StateRootNode` and state parts for shard B — no privileged access required).
2. Be reachable as a peer and be selected by the syncing node's network layer as the state-sync responder for shard A.

The network layer enforces that the response comes from the same peer that was requested: [14](#0-13) 

An attacker who advertises state-sync capability for shard A (any peer can do this) and is selected will have their response accepted without shard-identity verification. No validator, chunk-producer, or operator privileges are required.

---

### Recommendation

Add an explicit shard-identity guard immediately after extracting the chunk in `set_state_header`:

```rust

``` [15](#0-14) 

Additionally, consider using `verify_path_with_index` (which checks the leaf position) instead of `verify_path` for the `chunk_headers_root` Merkle proof, binding the proof to the correct shard index: [16](#0-15)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
