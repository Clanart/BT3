### Title
`ProofOfInclusion::valid()` Never Anchors to a Trusted Root — Forged DataLayer Inclusion Proofs Always Pass — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` only verifies internal self-consistency of the proof chain. Its final check is a tautology: it compares `existing_hash` against `self.root_hash()`, which returns the very same `last.combined_hash` that `existing_hash` was just set to in the loop. No trusted external root is ever consulted. Any attacker who can supply a `ProofOfInclusion` object — trivially possible through the Python/Streamable bindings — can forge a proof of inclusion for any key/value pair against any claimed tree root, and `valid()` will return `true`.

---

### Finding Description

`ProofOfInclusion` is defined in `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs` and carries:

```
node_hash: Hash                    // claimed leaf hash
layers: Vec<ProofOfInclusionLayer> // each layer: other_hash_side, other_hash, combined_hash
```

The `valid()` method walks the layers, computing each `calculated_hash` and checking it equals `layer.combined_hash`, then sets `existing_hash = calculated_hash`. After the loop it checks:

```rust
existing_hash == self.root_hash()
```

But `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same field just verified in the loop
    } else {
        self.node_hash
    }
}
```

After the last loop iteration, `existing_hash` was set to `calculated_hash`, which was already asserted equal to `layer.combined_hash`. So `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash` — always `true`. The final guard is a no-op.

The function never accepts a `trusted_root: Hash` parameter and never compares the computed root against any externally-known, authoritative tree root. It only proves that the proof's own fields are internally consistent.

`ProofOfInclusion` and its `valid()` method are exposed directly to Python via `PyStreamable` (enabling deserialization from untrusted bytes) and `#[pyo3(name = "valid")]`:

```rust
#[cfg_attr(
    feature = "py-bindings",
    pyclass(get_all, from_py_object),
    derive(PyJsonDict, PyStreamable)
)]
pub struct ProofOfInclusion { ... }

#[pyo3(name = "valid")]
pub fn py_valid(&self) -> bool {
    self.valid()
}
```

Any Python caller that receives a `ProofOfInclusion` from a peer, deserializes it, and calls `.valid()` to decide whether a DataLayer key/value pair is present in a store will be deceived.

---

### Impact Explanation

This matches: **High. DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

A forged `ProofOfInclusion` can convince a verifier that an arbitrary key/value pair exists in a DataLayer store whose root the attacker does not control. In DataLayer's trust model, proofs of inclusion are the mechanism by which clients verify data without holding the full tree. Accepting forged proofs allows an attacker to:

- Claim any key maps to any value in a store they do not own.
- Bypass DataLayer state verification in light clients or cross-store operations that rely on `proof.valid()` as the sole check.

---

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type exposed over the Python boundary. Any code path that receives a proof from an untrusted peer (e.g., a DataLayer sync or light-client query) and calls `proof.valid()` is immediately exploitable. The attacker needs only to construct a single-layer `ProofOfInclusion` where `combined_hash = calculate_internal_hash(node_hash, side, any_other_hash)` — a trivial SHA-256 computation — and `valid()` returns `true` unconditionally.

---

### Recommendation

`valid()` must accept a `trusted_root: &Hash` parameter (the root obtained from a locally-verified, authoritative source) and replace the final tautological check:

```rust
// BEFORE (tautology — always true after loop)
existing_hash == self.root_hash()

// AFTER (anchors to externally-trusted root)
existing_hash == *trusted_root
```

The Python binding `py_valid()` must be updated to require the trusted root as an argument. All call sites must supply the root from a source that cannot be influenced by the proof provider.

### Proof of Concept

```python
from chia_rs import ProofOfInclusion, ProofOfInclusionLayer, Hash, Side
import hashlib

# Attacker wants to forge: "key K maps to value V in store with root R"
# They pick any node_hash (representing their fake leaf)
fake_node_hash = bytes([0xAA] * 32)
fake_other_hash = bytes([0xBB] * 32)

# Compute combined_hash honestly so the chain is internally consistent
h = hashlib.sha256()
h.update(b"\x02")          # internal_hash prefix
h.update(fake_node_hash)   # Side.Right: hash is left, other is right
h.update(fake_other_hash)
fake_combined = h.digest()

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=Hash(fake_other_hash),
    combined_hash=Hash(fake_combined),
)
proof = ProofOfInclusion(node_hash=Hash(fake_node_hash), layers=[layer])

# valid() returns True — no trusted root was ever checked
assert proof.valid() == True
# proof.root_hash() == fake_combined, which the attacker chose freely
```
