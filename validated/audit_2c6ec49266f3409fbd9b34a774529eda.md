### Title
EIP-7702 Authorization State Committed Even When EVM Execution Fails Without Hooks - (File: x/evm/keeper/state_transition.go)

### Summary

In `ApplyTransaction`, the durable EIP-7702 authorization context (`commitDurableAuthorization`) is only created when `k.hooks != nil`. When no hooks are registered, the authorization effects (authority nonce bump + delegation code) are committed directly into the main `StateDB` via `stateDB.Commit()` inside `ApplyMessageWithConfig`, even when the EVM call itself fails (reverts with a `vmErr`). This means a failed EVM execution still permanently persists the EIP-7702 authorization state ŌĆö the authority's nonce is incremented and delegation code is installed ŌĆö even though the EVM call that was supposed to use that delegation reverted.

### Finding Description

In `ApplyTransaction`, the durable authorization context is only set up when hooks are present:

```go
if k.hooks != nil {
    tmpCtx, commit = ctx.CacheContext()
    if len(msg.SetCodeAuthorizations) > 0 {
        var durableAuthorizationCtx sdk.Context
        durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()
        cfg.DurableSetCodeAuthorizationCtx = &durableAuthorizationCtx
    }
}
```

When `k.hooks == nil`, `cfg.DurableSetCodeAuthorizationCtx` is `nil` and no `tmpCtx` cache branch is created. The `ApplyMessageWithConfig` call runs directly on `ctx`.

Inside `ApplyMessageWithConfig`, the authorization processing path is:

```go
if commit && cfg.DurableSetCodeAuthorizationCtx != nil && len(validAuths) > 0 {
    // durable path: write to separate ctx
    durableStateDB := statedb.NewWithParams(*cfg.DurableSetCodeAuthorizationCtx, ...)
    ...
    durableStateDB.Commit()
}
// then evm.Call() runs
ret, leftoverGas, vmErr = evm.Call(sender, *msg.To, msg.Data, leftoverGas, ...)
```

When `cfg.DurableSetCodeAuthorizationCtx == nil` (no hooks), the authorization effects (`SetNonce`, `SetCode`) are applied directly to the main `stateDB`. The `stateDB.Commit()` at the end of `ApplyMessageWithConfig` then flushes **all** dirty state ŌĆö including the authorization nonce/code changes ŌĆö unconditionally, regardless of whether `vmErr != nil`.

The `StateDB.Commit()` function does not check `vmErr`:

```go
if commit {
    if err := stateDB.Commit(); err != nil { ... }
}
```

The EVM snapshot/revert mechanism only reverts EVM-level state changes made *inside* `evm.Call()`. The authorization effects applied *before* `evm.Call()` via `applyAuthorization` are written to the `stateDB` journal and are **not** reverted by the EVM's internal snapshot, because no snapshot was taken before `applyAuthorization` is called. They are therefore included in the `stateDB.Commit()` flush even when `vmErr != nil`.

The result: an EIP-7702 type-4 transaction whose EVM call reverts (e.g., the target contract always reverts) still permanently:
1. Increments the authority's nonce (`auth.Nonce + 1`)
2. Installs the delegation code on the authority account

This is the analog of the "failed proposal can be committed again" pattern: a failed EVM execution still commits a sub-operation (authorization) that should only be durable when the overall transaction succeeds, or at minimum should be idempotent and not exploitable.

**The no-hooks path is the common production path** ŌĆö hooks are an optional extension. The durable authorization mechanism was specifically designed to handle the hooks-present case (where `tmpCtx` might be discarded on hook failure), but the no-hooks path was left without equivalent protection against the EVM-revert case.

### Impact Explanation

An attacker can craft an EIP-7702 type-4 transaction where:
- The authorization list contains a valid authorization for a victim EOA (authority) that the attacker controls the signing key for, delegating to a malicious contract
- The `to` target of the outer transaction is a contract that always reverts

The EVM call reverts, so the transaction appears to fail (status=0, `VmError` set). However, the authority's nonce is permanently bumped and the delegation code is installed. On subsequent transactions, the authority EOA now executes the malicious delegate's code, enabling unauthorized code execution under the authority's identity and balance.

This constitutes an **EIP-7702 authorization bypass**: the delegation is installed without a successful EVM execution, enabling unauthorized account/code mutation. Any future call to the authority address will execute the attacker-installed delegate code, potentially draining the authority's balance.

### Likelihood Explanation

- Requires only a valid EIP-7702 type-4 transaction with a valid authorization signature for the target authority
- The attacker must control the authority's signing key (i.e., the authority is the attacker's own account, or the attacker has obtained the authority's private key)
- The outer transaction's `to` target simply needs to revert
- No privileged access, governance, or validator compromise required
- Reachable via standard `eth_sendRawTransaction` JSON-RPC on any Prague-enabled Ethermint chain
- The no-hooks path is the default deployment configuration

### Recommendation

The `stateDB.Commit()` path must not flush authorization state when `vmErr != nil`. Options:

1. **Take a snapshot before `applyAuthorization`** and revert to it when `vmErr != nil`, so authorization effects are rolled back on EVM failure (matching go-ethereum's behavior where authorization effects are part of the EVM state transition and revert with it).

2. **Extend the durable authorization pattern to the no-hooks path**: always create a `DurableSetCodeAuthorizationCtx` (not only when `k.hooks != nil`), and only commit it after confirming `vmErr == nil`.

3. **Check `vmErr` before `stateDB.Commit()`** and skip committing authorization-related dirty state when the EVM call failed.

The correct fix aligns with EIP-7702 spec: authorization effects should be durable across EVM-level reverts (the nonce bump prevents replay), but the current code goes further by also installing the delegation code even on revert, which is not spec-compliant and is exploitable.

### Proof of Concept

1. Deploy a contract `AlwaysRevert` at address `0xDEAD` that executes `REVERT` unconditionally.
2. Generate a key pair `(authorityKey, authority)`.
3. Sign an EIP-7702 authorization: `auth = SignSetCode(authorityKey, {chainID, address: maliciousDelegate, nonce: 0})`.
4. Submit a type-4 transaction: `{to: 0xDEAD, authorizationList: [auth], nonce: senderNonce}`.
5. The EVM call to `0xDEAD` reverts ŌåÆ `vmErr != nil`, `res.Failed() == true`.
6. Observe: `stateDB.GetNonce(authority) == 1` and `stateDB.GetCode(authority) == delegation(maliciousDelegate)`.
7. Submit any subsequent transaction calling `authority` ŌĆö it now executes `maliciousDelegate`'s code under `authority`'s context, draining `authority`'s balance to the attacker.

The root cause is at: [1](#0-0) 

The durable authorization context is only created when hooks are present, leaving the no-hooks path unprotected. [2](#0-1) 

Authorization effects are committed to the durable stateDB before `evm.Call()` runs, with no rollback on `vmErr`. [3](#0-2) 

`stateDB.Commit()` flushes all dirty state unconditionally, including authorization nonce/code changes, regardless of `vmErr`. [4](#0-3) 

`setAuthorizationDelegation` writes nonce and code to stateDB before EVM execution, with no snapshot taken beforehand.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
