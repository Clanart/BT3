### Title
Missing Zero/Undeclared Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any `class_hash` value — including zero — without validating that it is non-zero or corresponds to a declared contract class. This is the direct analog of the original report's vulnerability class: a function assumes non-zero inputs but does not enforce it. A malicious contract deployer can call `replace_class(0)`, permanently freezing any funds held by the affected contract.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `class_hash` directly from the syscall request and writes it into the state without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `TODO` comment explicitly acknowledges the missing check. There is no `assert_not_zero(class_hash)` and no lookup to confirm the class is declared.

When `class_hash = 0` is committed to the state, any future call to that contract enters `execute_entry_point`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // = 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // = 0
);
``` [2](#0-1) 

Since class `0` is never declared, `compiled_class_hash` resolves to `0` (the default dict value), and `find_element` panics because `0` is not present in `compiled_class_facts`. The OS cannot generate a valid proof for any block containing a call to the affected contract. The sequencer is forced to exclude all such calls, permanently locking the contract's state and any funds it holds.

This is structurally identical to the original report: `GeoEmaAndCumSmaPump` assumed non-zero reserves and silently substituted `1` when given `0`, distorting oracle state. Here, `execute_replace_class` assumes a valid declared class hash but enforces nothing, allowing zero to be committed to the global state root, distorting contract reachability permanently.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class(0)` is executed and the block is proven, the state root commits `class_hash = 0` for the affected contract. There is no recovery path: the contract is uncallable, any ERC-20 tokens or native assets it holds are inaccessible forever, and the OS will reject any proof that includes a call to it.

---

### Likelihood Explanation

An unprivileged **contract deployer** can:

1. Deploy a contract whose bytecode contains a `replace_class(0)` call reachable via a public entry point.
2. Attract user deposits (e.g., by presenting as a yield vault or bridge).
3. Trigger the `replace_class(0)` call via a normal invoke transaction.

No privileged role, leaked key, or operator cooperation is required. The OS is the last line of defense and it performs no validation. The sequencer cannot prevent the state update once the transaction is included; it can only refuse future calls to the broken contract, which is precisely the freeze mechanism.

---

### Recommendation

Inside `execute_replace_class`, before writing to `contract_state_changes`, add:

1. `assert_not_zero(class_hash)` — reject the zero class hash explicitly.
2. A lookup into `contract_class_changes` (or a dedicated declared-class registry) to assert that `class_hash` maps to a non-zero compiled class hash, confirming the class was declared in the current or a prior block.

This mirrors the fix implied by the original report's mitigation: revert the operation when the critical input is zero.
