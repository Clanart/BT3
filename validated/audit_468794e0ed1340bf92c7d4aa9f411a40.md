### Title
Unchecked Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary
`compute_max_possible_fee` performs unchecked felt-field arithmetic over user-controlled resource bounds. A crafted transaction can make this function return `0` via modular overflow, causing `charge_fee` to exit early and process the transaction without collecting any fee. The resulting OS proof is valid, meaning the sequencer receives no fee for real computation.

### Finding Description
In `charge_fee` (lines 111–165 of `transaction_impls.cairo`), the OS computes the maximum chargeable fee and immediately short-circuits if it is zero:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
```

`compute_max_possible_fee` (lines 87–102) performs pure felt arithmetic with no overflow guard:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit +
       l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip) +
       l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is modulo the StarkNet field prime `P = 2²⁵¹ + 17·2¹⁹² + 1`. The only upstream bounds enforced on the inputs (inside `pack_resource_bounds` / `hash_fee_fields`, called during transaction-hash computation) are:

- `max_amount ≤ 2⁶⁴ − 1`
- `max_price_per_unit ∈ [0, (P−1)/2]` (enforced by `assert_nn`)
- `tip ≤ 2⁶⁴ − 1`

A single product `max_amount × max_price_per_unit` can reach `(2⁶⁴−1) × (P−1)/2 ≈ 2⁶³ · P`, which wraps around the field many times. An attacker can therefore choose three pairs of `(max_amount, max_price_per_unit)` and a `tip` such that the three-term sum is exactly `0 mod P`. When `compute_max_possible_fee` returns `0`, `charge_fee` returns immediately at line 123–125 without executing the ERC-20 transfer, and the transaction is processed fee-free.

The `low_actual_fee` hint (`%{ LoadActualFee %}`) and the subsequent `assert_nn_le(calldata.amount.low, max_fee)` check (line 135) are never reached, so there is no secondary guard.

### Impact Explanation
**Critical — Direct loss of funds.** The sequencer receives zero fee tokens for a transaction that consumed real L1/L2/data-gas resources. At scale, an attacker can spam the network with zero-cost transactions, draining sequencer revenue and potentially causing economic collapse of the fee market. Because the OS proof is valid (the Cairo constraints are satisfied), the state transition is accepted on-chain with no fee deducted from the sender's balance.

### Likelihood Explanation
Any unprivileged transaction sender controls `resource_bounds` and `tip` directly in the signed transaction body. Solving for an overflow-to-zero combination is straightforward modular arithmetic (e.g., choose `l1_gas_bounds` and `l2_gas_bounds` terms that sum to `P − term3`, where `term3` is chosen freely). No privileged access, leaked key, or third-party compromise is required. The attack is fully self-contained in a single crafted V3 transaction.

### Recommendation
Replace the unchecked felt multiplication in `compute_max_possible_fee` with range-checked arithmetic that asserts no intermediate product or sum exceeds a safe bound (e.g., `2¹²⁸`). Concretely:

1. After computing each product `max_amount × max_price_per_unit`, assert the result fits within a known safe range using `assert_nn_le(product, MAX_FEE_BOUND)`.
2. Assert the final sum is also within the same bound before returning.
3. Alternatively, enforce tighter bounds on `max_price_per_unit` (e.g., `≤ 2⁶⁴ − 1`) so that no product can exceed `2¹²⁸ − 1`, making overflow impossible in practice.

### Proof of Concept

**Goal:** craft a V3 transaction where `compute_max_possible_fee` returns `0 mod P`.

Let `P = 2²⁵¹ + 17·2¹⁹² + 1`.

Choose:
- L1 gas: `max_amount_1 = 1`, `max_price_1 = P − 2` → term₁ = `P − 2 ≡ −2 mod P`
- L2 gas: `max_amount_2 = 1`, `max_price_2 = 1`, `tip = 0` → term₂ = `1`
- L1 data gas: `max_amount_3 = 1`, `max_price_3 = 1` → term₃ = `1`

Sum = `(P − 2) + 1 + 1 = P ≡ 0 mod P`.

All values satisfy the enforced bounds (`max_amount ≤ 2⁶⁴−1`, `max_price ≤ (P−1)/2` — note `P−2 < (P−1)/2` is false for large P; adjust with `max_amount_1 = 2`, `max_price_1 = (P−1)/2` → term₁ = `P−1 ≡ −1`, then term₂ = 1, term₃ = 0 → sum = 0). The signed transaction is submitted; the OS executes it, `compute_max_possible_fee` returns `0`, `charge_fee` returns at line 123 without any ERC-20 transfer, and the block proof is accepted with zero fee paid. // Returns the maximum possible fee that can be charged for the transaction.
```text
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
