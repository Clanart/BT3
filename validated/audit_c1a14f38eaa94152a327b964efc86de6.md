### Title
`migrate_deployed_token` Fails to Update Multi-Chain `token_id_to_address` and `locked_tokens` Entries, Permanently Freezing In-Flight User Transfers - (File: near/omni-bridge/src/lib.rs)

---

### Summary

`migrate_deployed_token` in the NEAR omni-bridge contract updates token registry entries only for the `origin_chain` dimension of `token_id_to_address`, and does not migrate `locked_tokens` entries from `old_token` to `new_token`. Any `pending_transfers` that were created before the migration and reference `old_token` can never be completed: `sign_transfer` and `claim_fee` both call `get_token_address(destination_chain, old_token)` which returns `None` after migration, causing an irrecoverable panic. User funds (burned tokens or locked collateral) are permanently frozen.

---

### Finding Description

`migrate_deployed_token` performs the following state updates: [1](#0-0) 

It removes `(origin_chain, old_token)` from `token_id_to_address` and inserts `(origin_chain, new_token)`. It updates `token_address_to_id` to point the foreign address to `new_token`. It records the mapping in `migrated_tokens`.

**What it does NOT do:**
1. It does not update `token_id_to_address` entries for chains other than `origin_chain`. For a token with multi-chain registrations (e.g., `(ChainKind::Sol, old_token)` → Solana address), those entries remain keyed by `old_token`.
2. It does not migrate `locked_tokens` entries from `(chain_kind, old_token)` to `(chain_kind, new_token)`. [2](#0-1) 

The `locked_tokens` map is keyed by `(ChainKind, AccountId)`: [3](#0-2) 

After migration, `locked_tokens` still holds entries under `old_token`. New transfers using `new_token` call `lock_tokens_if_needed(chain, new_token, amount)`, which checks `locked_tokens.get(&(chain, new_token))` → `None` → returns `LockAction::Unchanged` (no lock recorded), silently breaking collateralization accounting. [4](#0-3) 

**The critical breakage for in-flight transfers:**

`sign_transfer` resolves the destination token address for a pending transfer: [5](#0-4) 

`get_token_id` for `OmniAddress::Near(old_token)` returns `old_token` directly: [6](#0-5) 

`get_token_address(destination_chain, old_token)` then calls: [7](#0-6) 

After migration, `token_id_to_address.get(&(destination_chain, old_token))` returns `None` because `migrate_deployed_token` removed the `(origin_chain, old_token)` entry. The `unwrap_or_else` branch panics with `FailedToGetTokenAddress`. The same panic occurs in `claim_fee_callback`: [8](#0-7) 

---

### Impact Explanation

**Permanent freezing of user funds (Critical).**

A user who calls `ft_transfer_call` with `BridgeOnTransferMsg::InitTransfer` before the migration:
- Their tokens are burned from `old_token` (for deployed/bridged tokens) via `burn_tokens_if_needed`
- The transfer is stored in `pending_transfers` with `token: OmniAddress::Near(old_token)`
- `locked_tokens` is incremented under `(destination_chain, old_token)`

After `migrate_deployed_token` is called:
- `sign_transfer` panics → the MPC signature cannot be obtained → the transfer cannot be finalized on the destination chain
- `claim_fee` panics → the relayer cannot claim fees → the pending transfer entry is never removed
- The burned tokens are gone; the pending transfer is permanently stuck; user funds are irrecoverable

Additionally, `locked_tokens` accounting is permanently desynchronized: future transfers using `new_token` do not update any lock entry (since `(chain, new_token)` has no entry), breaking bridge collateralization invariants.

---

### Likelihood Explanation

`migrate_deployed_token` is a DAO-callable function intended for legitimate token upgrades. The NEAR bridge is a live system with continuous user activity. It is realistic that pending transfers exist at the time of any migration. The DAO may not be able to atomically drain all pending transfers before calling `migrate_deployed_token`, especially since `sign_transfer` requires MPC signing (an asynchronous off-chain step). Any transfer that is in the `pending_transfers` map at migration time is permanently frozen.

---

### Recommendation

1. **Short term:** Before calling `migrate_deployed_token`, require that no `pending_transfers` exist for `old_token` (or pause the bridge and drain all pending transfers first). Alternatively, add logic inside `migrate_deployed_token` to:
   - Migrate all `locked_tokens` entries from `old_token` to `new_token` for every chain kind.
   - Update all `token_id_to_address` entries for `old_token` across all chain kinds (not just `origin_chain`).

2. **Long term:** Implement a migration strategy that handles in-flight state atomically, or make `sign_transfer` and `claim_fee` aware of the `migrated_tokens` map so they can resolve `old_token` → `new_token` transparently for pending transfers.

---

### Proof of Concept

**State before migration:**
- `deployed_tokens` contains `old_token`
- `token_id_to_address[(ChainKind::Eth, old_token)]` = `0xETH_ADDR`
- `locked_tokens[(ChainKind::Eth, old_token)]` = `1_000_000`
- `pending_transfers[{Near, nonce=42}]` = `TransferMessage { token: OmniAddress::Near(old_token), amount: 500_000, recipient: OmniAddress::Eth(0xUSER), ... }`

**User action (before migration):** Alice calls `ft_transfer_call` on `old_token` → `ft_on_transfer` → `init_transfer` → `init_transfer_internal` burns 500_000 tokens from `old_token` and inserts the pending transfer.

**DAO action:** DAO calls `migrate_deployed_token(ChainKind::Eth, old_token, new_token)`.

**After migration:**
- `token_id_to_address[(ChainKind::Eth, old_token)]` is **removed**
- `token_id_to_address[(ChainKind::Eth, new_token)]` = `0xETH_ADDR`
- `locked_tokens[(ChainKind::Eth, old_token)]` = `1_000_000` (unchanged, stale)
- `pending_transfers[{Near, nonce=42}]` still exists with `token: OmniAddress::Near(old_token)`

**Relayer calls `sign_transfer({Near, nonce=42})`:**
```
get_token_id(OmniAddress::Near(old_token)) → old_token
get_token_address(ChainKind::Eth, old_token)
  → token_id_to_address.get(&(ChainKind::Eth, old_token))
  → None
  → panic!("ERR_FAILED_TO_GET_TOKEN_ADDRESS")
```

Alice's 500_000 tokens are burned and permanently unrecoverable. The pending transfer entry remains in storage indefinitely. [9](#0-8) [10](#0-9) [11](#0-10)

    pub fn migrate_deployed_token(
        &mut self,
        origin_chain: ChainKind,
        old_token: AccountId,
        new_token: AccountId,
    ) {
        require!(
            env::attached_deposit() >= NEP141_DEPOSIT,
            BridgeError::NotEnoughAttachedDeposit.as_ref()
        );

        require!(
            self.deployed_tokens.remove(&old_token),
            BridgeError::OldTokenNotDeployed.as_ref(),
        );
        require!(
            self.deployed_tokens.insert(&new_token),
            BridgeError::TokenExists.as_ref()
        );
        self.deployed_tokens_v2.remove(&old_token);
        self.deployed_tokens_v2.insert(&new_token, &origin_chain);

        let origin_address = self
            .token_id_to_address
            .remove(&(origin_chain, old_token.clone()))
            .near_expect(BridgeError::FailedToGetTokenAddress);

        require!(
            self.token_id_to_address
                .insert(&(origin_chain, new_token.clone()), &origin_address)
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );

        self.token_address_to_id
            .insert(&origin_address, &new_token)
            .near_expect(BridgeError::ExpectedToOverwriteTokenAddress);

        require!(
            self.migrated_tokens
                .insert(&old_token, &new_token)
                .is_none(),
            BridgeError::TokenAlreadyMigrated.as_ref()
        );

        ext_token::ext(new_token.clone())
            .with_static_gas(STORAGE_DEPOSIT_GAS)
            .with_attached_deposit(NEP141_DEPOSIT)
            .storage_deposit(&env::current_account_id(), Some(true))
            .detach();

        env::log_str(
            &OmniBridgeEvent::MigrateTokenEvent {
                old_token_id: old_token,
                new_token_id: new_token,
            }
            .to_log_string(),
        );
    }
