Looking at the full validation chain in `set_state_header` and the Merkle proof semantics, I need to determine whether the missing `chunk.shard_id() == shard_id` check is exploitable.

### Title
Missing shard_id binding in `set_state_header` allows cross-shard state corruption during state sync — (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for shard A that embeds shard B's `ShardChunk` because no check asserts `chunk.shard_id() == shard_id`. The Merkle-inclusion proof used in step 3a is **non-positional** (`verify_path`, not `verify_path_with_index`), so a valid proof for shard B's chunk against `chunk_headers_root` passes even though the chunk belongs to a different shard slot. The corrupted header is then persisted under `StateHeaderKey(shard_id_A, sync_hash)`, causing all subsequent state-part validation and application to use shard B's state root while writing into shard A's `ShardUId`.

---

### Finding Description

**Entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called from the `StateHeaderValidationRequest` handler after a header is downloaded from a peer.

**Missing guard**: There is no assertion of the form `chunk.shard_id() == shard_id` anywhere in `set_state_header`. The five checks that exist are:

1. **`validate_chunk_proofs`** — verifies hash self-consistency and body (tx root, receipts root). Does not inspect `shard_id`. [1](#0-0) 

2. **Step 3a `verify_path`** — verifies `ChunkHashHeight(chunk_B.chunk_hash(), chunk_B.height_included())` is somewhere in `chunk_headers_root`. `verify_path` is **non-positional**: it only checks that `compute_root_from_path(path, hash(item)) == root`. Shard B's chunk IS in the block's Merkle tree (at shard B's index), so a valid proof for it passes regardless of which shard was requested. [2](#0-1) [3](#0-2) [4](#0-3) 

3. **Step 3b `verify_path`** — same non-positional proof for `prev_chunk_header`. Shard B's prev chunk is also in the block, so this passes. [5](#0-4) 

4. **Steps 4a–4g receipt proofs** — validated using the outer `shard_id` (A) via `ReceiptList(shard_id, receipts)`. The attacker supplies valid on-chain receipt proofs for shard A. Step 4g checks the chain ends at `prev_chunk_header.height_included()` (shard B's prev chunk height). In NEAR's current deployment where all shards produce chunks at every block, shard A and shard B have identical prev-chunk heights, so the range matches. [6](#0-5) 

5. **Step 5 `validate_state_root_node`** — validates the `state_root_node` against `chunk_inner.prev_state_root()` (shard B's state root). Checks only hash and `memory_usage`, not `shard_id`. An attacker tracking shard B supplies a valid node. [7](#0-6) 

**Storage**: The corrupted header is written under `StateHeaderKey(shard_id_A, sync_hash)`. [8](#0-7) 

**Downstream propagation**:

- `set_state_part` reads the stored header for shard A, extracts shard B's `prev_state_root`, and validates incoming parts against it. [9](#0-8) 

- `apply_state_part` derives `shard_uid` from `shard_id_A` but applies trie changes computed from shard B's state root, writing shard B's trie data into shard A's column. [10](#0-9) 

- `set_state_finalize` reads the header for shard A, gets shard B's chunk, and applies it with `shard_uid` derived from `shard_id_A`, producing a `ChunkExtra` with shard B's post-state root stored under shard A's key. [11](#0-10) 

---

### Impact Explanation

After a successful attack, the syncing node's shard A trie is populated with shard B's state data. `set_state_finalize` applies shard B's chunk (transactions, receipts) to shard A's `ShardUId`, producing a `ChunkExtra` whose `state_root` is shard B's post-state root stored under shard A's key. The node then attempts to continue block processing for shard A against this wrong state root, causing it to diverge from the canonical chain. Flat storage for shard A is also initialized from shard B's chunk's `prev_block()`, further corrupting the flat-state index.

---

### Likelihood Explanation

**Constraints that must hold simultaneously**:

1. The attacker must be the peer that responds to the victim's `StateRequestHeader`. In NEAR's P2P state sync, any peer can respond — no trusted-peer restriction is enforced at the protocol level.
2. The attacker must track shard B to supply a valid `state_root_node`. Nodes tracking all shards (common for archival/RPC nodes) satisfy this trivially.
3. Shard A and shard B must have the same prev-chunk height. In NEAR mainnet with all shards producing chunks at every block, this is always true.

All three conditions are routinely satisfied. The attack requires no validator key, no block-producer key, and no admin access — only the ability to be a responding peer during state sync.

---

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind chunk to the requested shard
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        format!("set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
                chunk.shard_id(), shard_id).into()
    ));
}
```

Apply the same guard to `prev_chunk_header.shard_id()`. This closes the gap before any Merkle proof is evaluated. [12](#0-11)

```rust
//        incoming_receipts_proofs: valid_shard_0_receipts,

//        state_root_node: shard_1_state_root_node,
//    }
// 6. Call set_state_header(shard_id=0, sync_hash, crafted_header).
// 7. Assert: StateHeaders[StateHeaderKey(0, sync_hash)].chunk.shard_id() == 1  (corruption confirmed)
// 8. Assert: set_state_header returned Ok(()) without error.
```

The test would confirm that `set_state_header` returns `Ok(())` and that the stored header's embedded chunk carries `shard_id = 1` under the key for shard 0, demonstrating the missing binding check.

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
```
