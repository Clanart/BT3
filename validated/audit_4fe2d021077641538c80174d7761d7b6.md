### Title
Nonce Collision via Reentrancy in `OmniBridge.initTransfer()` Causes Permanent Fund Freezing — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer()` increments `currentOriginNonce` before external token calls but reads `currentOriginNonce` **again as a state variable** after those calls when passing it to `initTransferExtension()` and `emit`. A malicious or ERC777-compatible token can reenter `initTransfer()` during the token transfer, causing `currentOriginNonce` to be incremented a second time. The outer call then emits the same nonce as the inner call, creating a nonce collision that permanently freezes the outer call's tokens in the bridge with no corresponding cross-chain release.

---

### Finding Description

In `OmniBridge.initTransfer()`, `currentOriginNonce` is incremented at line 381 before any external calls:

```solidity
currentOriginNonce += 1;   // line 381 — incremented first
``` [1](#0-0) 

However, the external token transfer calls follow immediately after:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(   // external call — reentrancy possible here
        msg.sender,
        address(this),
        amount
    );
}
``` [2](#0-1) 

After those external calls, both `initTransferExtension()` and `emit` read `currentOriginNonce` **as a live state variable**, not a locally captured snapshot:

```solidity
initTransferExtension(
    msg.sender,
    tokenAddress,
    currentOriginNonce,   // reads state variable AFTER external calls
    ...
);

emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,   // reads state variable AFTER external calls
    ...
);
``` [3](#0-2) 

If a reentrant call to `initTransfer()` occurs during the token transfer (e.g., via an ERC777 `tokensToSend` hook or a malicious ERC20 `transferFrom`), `currentOriginNonce` is incremented again inside the inner call. When execution returns to the outer call, both `initTransferExtension()` and `emit` use the now-updated `currentOriginNonce` (N+1) instead of the original value (N). This causes two `InitTransfer` events to be emitted with the **same nonce N+1**, while nonce N is never emitted.

The same structural flaw exists in `initTransfer1155()`:

```solidity
currentOriginNonce += 1;   // line 448

IERC1155(tokenAddress).safeTransferFrom(   // external call — reentrancy possible
    msg.sender,
    address(this),
    tokenId,
    amount,
    ""
);

initTransferExtension(
    msg.sender,
    deterministicToken,
    currentOriginNonce,   // reads state variable AFTER external call
    ...
);
``` [4](#0-3) 

In the Wormhole variant (`OmniBridgeWormhole`), `initTransferExtension` encodes `originNonce` into the Wormhole payload and publishes it:

```solidity
Borsh.encodeUint64(originNonce),   // the corrupted nonce is published to Wormhole
``` [5](#0-4) 

There is no `ReentrancyGuard` or `nonReentrant` modifier on either `initTransfer` or `initTransfer1155`.

---

### Impact Explanation

The NEAR side (or Wormhole relay) processes the first event with nonce N+1 and rejects the second as a duplicate nonce. The tokens from the outer call are permanently locked in the bridge contract with no corresponding NEAR-side release — an irrecoverable lock of user funds. Nonce N is also permanently skipped, which can corrupt sequential nonce accounting on the NEAR side.

This matches the **Critical** allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

- `initTransfer()` accepts **any** ERC20 token address with no whitelist restriction on the `else` branch (non-bridge, non-custom-minter tokens).
- An attacker can deploy a malicious ERC20 whose `transferFrom` reenters `initTransfer()` before completing the transfer.
- ERC777 tokens (widely deployed, e.g., imBTC, CHAI) provide a built-in `tokensToSend` hook on the sender that fires during `safeTransferFrom`, giving a natural reentrancy vector without deploying a custom token.
- No privileged access is required. Any unprivileged bridge user can call `initTransfer()` with a token they control.

---

### Recommendation

1. **Capture `currentOriginNonce` in a local variable** before any external calls and use that local variable in `initTransferExtension()` and `emit`:

```solidity
currentOriginNonce += 1;
uint64 nonce = currentOriginNonce;   // snapshot before external calls

// ... external token transfer calls ...

initTransferExtension(msg.sender, tokenAddress, nonce, ...);
emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, nonce, ...);
```

2. **Add OpenZeppelin's `ReentrancyGuard`** and apply `nonReentrant` to both `initTransfer()` and `initTransfer1155()`.

Apply the same fix to `initTransfer1155()`.

---

### Proof of Concept

**Setup**: Attacker deploys `MaliciousToken` — an ERC20 whose `transferFrom(from, to, amount)` reenters `OmniBridge.initTransfer(legitimateToken, 1, 0, 0, recipient, "")` before completing the transfer.

**Steps**:

1. Attacker calls `initTransfer(MaliciousToken, X, 0, 0, recipient, "")`.
2. `currentOriginNonce` increments: N-1 → **N**.
3. `safeTransferFrom(attacker, bridge, X)` is called on `MaliciousToken`.
4. `MaliciousToken.transferFrom` reenters `initTransfer(legitimateToken, 1, 0, 0, recipient, "")`.
5. `currentOriginNonce` increments: N → **N+1**.
6. Inner call completes: `initTransferExtension(..., currentOriginNonce=N+1, ...)` and `emit InitTransfer(..., N+1, ...)` — **correct for inner call**.
7. Outer call resumes after `safeTransferFrom` returns.
8. Outer call executes: `initTransferExtension(..., currentOriginNonce=N+1, ...)` and `emit InitTransfer(..., N+1, ...)` — **wrong nonce, should be N**.
9. Two `InitTransfer` events with nonce **N+1** are emitted in the same transaction. Nonce **N** is never emitted.
10. NEAR/Wormhole relay processes nonce N+1 once (inner call, 1 token) and rejects the second submission (outer call, X tokens) as a duplicate nonce.
11. The outer call's **X tokens are permanently locked** in the bridge with no cross-chain release. Nonce N is permanently skipped.
