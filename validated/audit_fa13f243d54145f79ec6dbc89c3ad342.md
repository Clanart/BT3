### Title
Missing Validation on `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary `class_hash` from the caller without validating that it is non-zero or corresponds to a declared class. A malicious or buggy contract can call `replace_class(0)`, setting its own class hash to the uninitialized sentinel value, permanently rendering itself unexecutable and freezing all funds held in its storage. The OS code itself contains an explicit TODO acknowledging this missing check.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract state with no validation:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The `class_hash` field is taken verbatim from the user-supplied syscall request. There is no `assert_not_zero(class_hash)` and no check that `class_hash` exists in `contract_class_changes`. The TODO comment at line 898 explicitly acknowledges the missing declared-class check. [2](#0-1) 

This is directly analogous to the reported `_updateFeeWallet` bug: a critical address/hash field is accepted and committed to state without any validity check, allowing it to be set to a zero/sentinel value.

For comparison, `deploy_contract.cairo` correctly enforces that a freshly deployed contract must have `UNINITIALIZED_CLASS_HASH` (i.e., 0) before deployment, confirming that 0 is the sentinel for "no code": [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

If a contract's `class_hash` is set to `0` (`UNINITIALIZED_CLASS_HASH`), every subsequent call to that contract will fail at the class-dispatch level because there is no code to execute. Any ERC-20 token balances, ETH, or other assets stored in that contract's storage become permanently inaccessible. There is no recovery path: the OS has no mechanism to forcibly restore a class hash once it has been overwritten.

---

### Likelihood Explanation

**Medium.**

Any deployed contract can issue the `replace_class` syscall on itself. A malicious contract can be deliberately designed to:
1. Accept user deposits.
2. Trigger `replace_class(0)` (or any undeclared hash) under attacker-controlled conditions.

This requires no privileged role, no leaked key, and no external dependency. The attacker only needs to deploy a contract and convince users to interact with it — a realistic social-engineering or DeFi-integration scenario. The missing check is reachable by any unprivileged transaction sender whose contract issues this syscall.

---

### Recommendation

Inside `execute_replace_class`, add the following validations before updating state:

1. **Assert non-zero**: `assert_not_zero(class_hash)` — prevents setting the class to the uninitialized sentinel.
2. **Assert declared**: Verify that `class_hash` exists as a key in `contract_class_changes` (or the global class registry), ensuring only previously declared classes can be substituted.

This mirrors the fix recommended for the original `_updateFeeWallet` bug: validate the input before committing it to state.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts token deposits and exposes a `drain()` function.
2. `drain()` internally calls the `replace_class` syscall with `class_hash = 0`.
3. Users deposit tokens into `MaliciousVault` (e.g., via a DeFi integration).
4. Attacker calls `drain()`. The OS executes `execute_replace_class`, writing `class_hash = 0` into `MaliciousVault`'s `StateEntry` with no validation.
5. All future calls to `MaliciousVault` (including withdrawal functions) fail because `class_hash = 0` means no executable code.
6. All deposited tokens are permanently frozen in `MaliciousVault`'s storage, with no recovery path.

The root cause is entirely within the scoped file `syscall_impls.cairo` at the `execute_replace_class` function, lines 878–916. [4](#0-3)
