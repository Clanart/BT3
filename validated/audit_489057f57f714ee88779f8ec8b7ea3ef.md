### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The OS-level `execute_replace_class` syscall handler does not verify that the caller-supplied `class_hash` corresponds to a class that has actually been declared on-chain. A contract can invoke `replace_class` with an arbitrary, undeclared, or zero class hash. The OS accepts and commits this state transition without any validation. Once committed, all future calls to that contract fail at class dispatch, permanently freezing any funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the caller-supplied `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no check that the hash exists in `contract_class_changes`:

```cairo
// syscall_impls.cairo, lines 896–910
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The inline TODO at line 898 explicitly acknowledges the missing check:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
``` [2](#0-1)

``` [4](#0-3) 

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

Once a contract's `class_hash` in `contract_state_changes` is set to an undeclared value (e.g., `0` or any arbitrary felt), the OS has no declared class to dispatch future calls against. Every subsequent call to that contract will fail at class resolution. If the contract is an account contract holding STRK or ETH, or any contract acting as a vault or escrow, all assets it controls become permanently inaccessible. The state transition is committed to the proven block output and cannot be reversed — there is no upgrade path for a contract whose class hash points to nothing.

---

### Likelihood Explanation

**Low-Medium.** The `replace_class` syscall is callable by any contract on itself — no privileged role is required. A contract author who makes a mistake (passes `0` or a hash of a class not yet declared), or a malicious contract that deliberately self-destructs by calling `replace_class(0)`, triggers this path. The OS is the last line of defense and currently provides none. The acknowledged TODO confirms the team is aware the check is absent.

---

### Recommendation

Inside `execute_replace_class` (both in `syscall_impls.cairo` and `deprecated_execute_syscalls.cairo`), before writing `new_state_entry`, perform a lookup of `class_hash` in `contract_class_changes` and assert it is non-zero (i.e., the class has been declared). Concretely:

```cairo
// After: let class_hash = request.class_hash;
// Add:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("replace_class: class hash is not declared") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the existing guard in `execute_declare_transaction` which uses `prev_value=0` to enforce that a class may only be declared once, and `assert_not_zero(compiled_class_hash)` to reject zero hashes. [5](#0-4) 

---

### Proof of Concept

1. Deploy a contract `VaultContract` that holds user funds and exposes a `self_replace(class_hash: felt)` external function that calls the `replace_class` syscall with the provided argument.
2. Submit an invoke transaction calling `self_replace(0)` (or any felt value not present in `contract_class_changes`).
3. The OS dispatches to `execute_replace_class` in `syscall_impls.cairo`. At line 896, `class_hash = 0` is read from the request. The TODO at line 898 confirms no declared-class check is performed. `dict_update` writes `StateEntry { class_hash: 0, ... }` into `contract_state_changes`.
4. The block is proven and finalized. `VaultContract`'s on-chain class hash is now `0`.
5. Any subsequent call to `VaultContract` (e.g., to withdraw funds) fails: the OS reads `class_hash = 0` from state, finds no matching entry in `contract_class_changes`, and cannot dispatch the call.
6. All funds held by `VaultContract` are permanently frozen with no recovery path.
