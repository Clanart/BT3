### Title
Non-atomic `burn.and(mint)` in `swap_migrated_token` causes permanent token loss on mint failure — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`swap_migrated_token` schedules a `burn` and a `mint` as two **parallel, independent** NEAR promises via `.and()`, then `.detach()`es the combined promise with no failure callback. If the `mint` promise fails for any reason (recipient not registered for storage in the new token contract, gas exhaustion, contract panic), the `burn` has already been committed and the user's old tokens are permanently destroyed with no new tokens ever minted.

---

### Finding Description

In `ft_on_transfer`, the `SwapMigratedToken` branch calls:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
``` [1](#0-0) 

Returning `U128(0)` means the bridge retains all transferred old tokens. `swap_migrated_token` then constructs:

```rust
let burn = ext_token::ext(old_token).burn(amount);
let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
burn.and(mint)
``` [2](#0-1) 

In NEAR's promise model, `promise_a.and(promise_b)` schedules **both sub-promises to execute in parallel and independently**. Neither sub-promise is conditioned on the success of the other. The `.detach()` call means there is no `.then()` callback to detect or recover from a failure.

The `mint` implementation in `omni-token` with `msg = None` calls `self.token.internal_deposit(&account_id, amount.into())`: [3](#0-2) 

`internal_deposit` (NEAR NEP-141 standard) **panics** if `account_id` is not registered (i.e., has no storage deposit in the new token contract). This is a normal, expected condition for any user who has not yet called `storage_deposit` on the new token.

The `burn` implementation withdraws from `env::predecessor_account_id()` (the bridge): [4](#0-3) 

Since `burn` and `mint` are parallel and independent, `burn` executes and commits regardless of whether `mint` succeeds or fails.

---

### Impact Explanation

If `mint` fails:
- The old tokens are permanently burned from the bridge's balance (the bridge received them and returned 0, so they are now the bridge's to burn).
- No new tokens are minted to the user.
- There is no callback, no retry, no refund path.
- The user's funds are irrecoverably destroyed.

This matches the **Critical** impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

The failure condition is **realistic and common**:
- A user who holds old migrated tokens and calls `ft_transfer_call` with `SwapMigratedToken` without having first registered (called `storage_deposit`) in the new token contract will trigger the failure.
- This requires no privileged access. Any holder of old migrated tokens can reach this path.
- The `migrated_tokens` mapping is populated by admin, but the swap itself is fully open to any user.
- Gas exhaustion on the `mint` call (only `MINT_TOKEN_GAS = 5 TGas` is allocated) is another realistic failure vector. [5](#0-4) 

---

### Recommendation

Replace the fire-and-forget `burn.and(mint)` with a sequenced, callback-guarded flow:

1. First execute `burn`.
2. In the burn callback (verifying success), execute `mint`.
3. In the mint callback (verifying success), finalize. If `mint` fails, re-mint the old tokens back to the user or revert the burn via a compensating mint of old tokens.

Alternatively, reverse the order: mint new tokens first, and only burn old tokens in the mint success callback. This is the safer direction since a failed mint leaves old tokens intact.

---

### Proof of Concept

```
1. Admin registers old_token → new_token in migrated_tokens (privileged setup, not part of attack).
2. Attacker holds `amount` of old_token.
3. Attacker does NOT call storage_deposit on new_token (so they are unregistered).
4. Attacker calls: old_token.ft_transfer_call(
       receiver_id = bridge,
       amount = X,
       msg = '"SwapMigratedToken"'
   )
5. Bridge's ft_on_transfer fires:
   - Returns U128(0) → bridge keeps X old tokens.
   - Schedules burn.and(mint), detaches.
6. burn executes: old_token.burn(X) → bridge's X old tokens destroyed. ✓
7. mint executes: new_token.mint(attacker, X, None)
   → internal_deposit(attacker, X) panics (attacker not registered) → mint fails.
8. No callback exists to detect the failure.
9. Assert: attacker's old_token balance decreased by X. ✓
   Assert: attacker's new_token balance = 0. ✓
   → X tokens permanently destroyed.
```
