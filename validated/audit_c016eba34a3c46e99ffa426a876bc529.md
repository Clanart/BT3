### Title
`VerifyEthAccount` Does Not Account for Cumulative Cost Across Multiple `MsgEthereumTx` Messages in the Same Cosmos Tx - (File: `ante/eth.go`)

### Summary
`VerifyEthAccount` iterates over all `MsgEthereumTx` messages in a single Cosmos tx and checks each message's cost against the **same committed balance** read from the store, without subtracting the cost already "reserved" by previous messages in the same iteration. This is the direct analog of the external report's vulnerability class: pending/in-flight state is not accounted for when computing the amount check.

### Finding Description
In `VerifyEthAccount` (`ante/eth.go:76–106`), for each `MsgEthereumTx` in the tx, the function calls `evmKeeper.GetBalance` at line 101 and then `keeper.CheckSenderBalance` at line 102. Both calls are independent of any prior message in the loop:

```go
for _, msg := range tx.GetMsgs() {
    // ...
    balance := evmKeeper.GetBalance(ctx, from, evmDenom)   // always reads the same committed balance
    if err := keeper.CheckSenderBalance(sdkmath.NewIntFromBigIntMut(balance.ToBig()), ethTx); err != nil {
        return errorsmod.Wrap(err, "failed to check sender balance")
    }
}
```

`GetBalance` delegates to `k.bankKeeper.GetBalance` which reads from the KV store (`x/evm/keeper/keeper.go:297–299`). No running tally of already-checked costs is maintained across loop iterations.

The same pattern appears in `CheckEthCanTransfer` (`ante/eth.go:213–253`), where `canTransfer` at line 242 reads the same committed balance for every message without accounting for value already "reserved" by prior messages:

```go
if value.Sign() > 0 && !canTransfer(ctx, evmKeeper, evmParams.EvmDenom, from, value) { ... }
```

```go
func canTransfer(...) bool {
    balance := evmKeeper.GetBalance(ctx, sdk.AccAddress(from.Bytes()), denom)
    return balance.ToBig().Cmp(amount) >= 0
}
```

The ante handler execution order in `newEthAnteHandler` (`evmd/ante/handler_options.go:131–143`) is:
1. `VerifyEthAccount` — balance pre-check (no deduction)
2. `CheckEthCanTransfer` — value pre-check (no deduction)
3. `CheckEthGasConsume` — actual fee deduction via `DeductTxCostsFromUserBalance` → `DeductFees` → `SendCoinsFromAccountToModuleVirtual`

`DeductFees` (`x/evm/keeper/utils.go:207–216`) uses `SendCoinsFromAccountToModuleVirtual`, a virtual/ObjectStore write path. `GetBalance` reads from the KV store, not the ObjectStore. Consequently, within the `CheckEthGasConsume` loop, the balance read for message N does not reflect the fee already deducted for messages 1…N-1 in the same iteration, mirroring the same root cause as in `VerifyEthAccount`.

### Impact Explanation
A sender with balance `B` can craft a Cosmos tx containing `N` `MsgEthereumTx` messages each with individual cost `C`, where `B ≥ C` but `B < N·C`. All `N` messages pass `VerifyEthAccount` and `CheckEthCanTransfer` because each check reads the same committed balance `B`. In `CheckEthGasConsume`, if `SendCoinsFromAccountToModuleVirtual` reads from the KV store (not the ObjectStore) for its balance enforcement, all `N` fee deductions also succeed, causing the sender's balance to be over-debited or the fee collector to be over-credited relative to the sender's actual funds — a mis-accounting of user fees. This matches the allowed High impact: **ante handler bug that permits valid user funds/fees to be mis-accounted**.

### Likelihood Explanation
Ethermint explicitly supports multi-message Cosmos txs containing multiple `MsgEthereumTx` entries (the nonce check in `CheckAndSetEthSenderNonce` increments sequentially per message). Any unprivileged user can submit such a tx via the standard broadcast endpoint. No special role, governance action, or validator collusion is required.

### Recommendation
In `VerifyEthAccount`, maintain a per-sender running total of costs already checked and subtract it from the balance before each `CheckSenderBalance` call:

```go
cumulativeCost := new(big.Int)
for _, msg := range tx.GetMsgs() {
    // ...
    balance := evmKeeper.GetBalance(ctx, from, evmDenom)
    adjustedBalance := new(big.Int).Sub(balance.ToBig(), cumulativeCost)
    if err := keeper.CheckSenderBalance(sdkmath.NewIntFromBigInt(adjustedBalance), ethTx); err != nil {
        return errorsmod.Wrap(err, "failed to check sender balance")
    }
    cumulativeCost.Add(cumulativeCost, ethTx.Cost())
}
```

Apply the same fix to `CheckEthCanTransfer`'s `canTransfer` call. Additionally, verify that `SendCoinsFromAccountToModuleVirtual` enforces balance constraints against the ObjectStore (not the KV store) so that sequential deductions within `CheckEthGasConsume` are correctly bounded.

### Proof of Concept
1. Sender account has balance `B = 100` wei.
2. Attacker crafts a Cosmos tx with two `MsgEthereumTx` messages from the same sender, each with `gasLimit * gasPrice = 60` wei (cost `C = 60`).
3. `VerifyEthAccount` loop, iteration 1: `GetBalance` → 100; `CheckSenderBalance(100, cost=60)` → passes.
4. `VerifyEthAccount` loop, iteration 2: `GetBalance` → 100 (same KV read, no deduction tracked); `CheckSenderBalance(100, cost=60)` → passes.
5. Both messages clear the ante handler balance gate despite total cost (120) exceeding balance (100).
6. `CheckEthGasConsume` deducts 60 for msg1 via `SendCoinsFromAccountToModuleVirtual`; if the virtual store balance is not visible to the next KV read, deducts 60 again for msg2 — sender's effective balance goes to −20, mis-accounting 60 wei of fees.

func CheckSenderBalance(
	balance sdkmath.Int,
	tx *ethtypes.Transaction,
) error {
	cost := tx.Cost()

	if cost.Sign() < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInvalidCoins,
			"tx cost (%s) is negative and invalid", cost,
		)
	}

	if balance.IsNegative() || balance.BigInt().Cmp(cost) < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInsufficientFunds,
			"sender balance < tx cost (%s < %s)", balance, tx.Cost(),
		)
	}
	return nil
}

// DeductFees deducts fees from the given account.
func DeductFees(bankKeeper types.BankKeeper, ctx sdk.Context, acc sdk.AccountI, fees sdk.Coins) error {
	if !fees.IsValid() {
		return errorsmod.Wrapf(errortypes.ErrInsufficientFee, "invalid fee amount: %s", fees)
	}
	if ctx.BlockHeight() > 0 {
		if err := bankKeeper.SendCoinsFromAccountToModuleVirtual(ctx, acc.GetAddress(), authtypes.FeeCollectorName, fees); err != nil {
			return errorsmod.Wrap(errortypes.ErrInsufficientFunds, err.Error())
		}
	}
	return nil
}
