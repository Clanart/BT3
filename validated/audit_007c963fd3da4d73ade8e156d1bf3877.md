Audit Report

## Title
Router `pay()` WETH Branch Consumes Stranded Native ETH Without Per-Caller Accounting, Enabling Free Swaps at Prior Users' Expense — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` uses `address(this).balance` as the source of ETH for WETH swaps without any per-call accounting tying that balance to the caller who deposited it. When a user sends excess `msg.value` to a payable swap entry point and omits the trailing `refundETH` call, the residual ETH remains on the router and is freely consumed by the next caller's WETH-input swap, giving that caller a full swap output at zero cost while the prior user permanently loses their ETH.

## Finding Description
`PeripheryPayments.pay()` (lines 73–84) handles WETH payments by reading `address(this).balance` and, when `nativeBalance >= value`, wrapping exactly `value` wei and forwarding the resulting WETH to the pool — the `payer` argument is never touched:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer ignored
    } ...
}
```

The `receive()` guard (line 32–34) only blocks *direct* ETH transfers from non-WETH addresses; it does not prevent ETH from arriving via `msg.value` in the payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, etc.). There is no check anywhere that `msg.value` for the *current* top-level call is sufficient to cover the WETH amount being paid, and no transient-storage slot tracks per-call ETH credit.

**Exploit path:**

1. Alice calls `exactInputSingle{value: 1000}(amountIn=500, tokenIn=WETH, ...)`. During the callback, `pay()` sees `nativeBalance=1000 >= 500`, wraps 500 wei, and sends WETH to the pool. 500 wei remains on the router. Alice does not call `refundETH`.
2. Bob calls `exactInputSingle{value: 0}(amountIn=500, tokenIn=WETH, ...)`. `_setNextCallbackContext` records Bob as payer. During the callback, `pay()` sees `nativeBalance=500 >= 500`, wraps Alice's 500 wei, sends WETH to the pool, and returns the swap output to Bob. Bob pays nothing; Alice's 500 wei is permanently transferred to the pool.

The transient callback context correctly records Bob as `payer`, but `pay()` never calls `safeTransferFrom(payer, ...)` in the `nativeBalance >= value` branch, so the payer field is irrelevant to the actual fund movement.

## Impact Explanation
Direct loss of user principal. Any ETH stranded on the router is consumed by the next WETH-input swap regardless of who deposited it. The attacker receives the full swap output without spending any ETH or WETH. The victim's ETH is permanently transferred to the pool as payment for the attacker's trade. This satisfies the "Critical/High direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The precondition — ETH left on the router — arises naturally when a user sends `msg.value` larger than `amountIn` (e.g., to avoid a revert if the pool fills partially) and omits the trailing `refundETH` call, or when a `multicall` batch is constructed without a final `refundETH` step. Both patterns are common in practice. Once ETH is stranded, any observer can exploit it in the very next block with a zero-cost WETH swap requiring no special privileges.

## Recommendation
**Short term:** In the `nativeBalance >= value` branch, verify that the ETH credit for the current call (tracked in transient storage alongside the existing payer/token context) covers at least `value`, and deduct from that credit rather than from the raw contract balance. Alternatively, require `msg.value >= value` at the `exactInput*` entry point when `tokenIn == WETH`.

**Long term:** Enforce an invariant that `address(this).balance == 0` at the start of every non-payable entry point, or revert in `pay()` when `nativeBalance > 0` and the current call's `msg.value` is zero, preventing silent consumption of residual ETH.

## Proof of Concept
```
Block N:
  Alice calls exactInputSingle{value: 1000}(
      tokenIn=WETH, amountIn=500, recipient=Alice, ...
  )
  → pool.swap() fires metricOmmSwapCallback
  → _justPayCallback → pay(WETH, Alice, pool, 500)
      nativeBalance = 1000 >= 500
      router wraps 500 wei, sends WETH to pool  ← 500 wei of Alice's ETH used
      500 wei remains on router
  Alice does NOT call refundETH.

Block N+1:
  Bob calls exactInputSingle{value: 0}(
      tokenIn=WETH, amountIn=500, recipient=Bob, ...
  )
  → pool.swap() fires metricOmmSwapCallback
  → _justPayCallback → pay(WETH, Bob, pool, 500)
      nativeBalance = 500 >= 500
      router wraps 500 wei, sends WETH to pool  ← Alice's remaining 500 wei used
      Bob receives TOKEN output, pays nothing
```

A Foundry integration test can reproduce this by deploying the router with a mock WETH and pool, executing Alice's swap with excess `msg.value`, then executing Bob's swap with `value: 0` and asserting Bob receives output while the router balance drops to zero and Alice's refund is unrecoverable.
