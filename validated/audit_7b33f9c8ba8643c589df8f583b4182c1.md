### Title
`set_state_header` accepts a peer-supplied chunk for any shard without binding it to the requested `shard_id` — (File: `chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` validates that the peer-supplied chunk is internally consistent and is included in the block's Merkle root, but **never checks that `chunk.shard_id()` equals the `shard_id` parameter**. A malicious peer can supply a valid chunk for shard X when shard Y was requested. All existing checks pass, the header is stored under `StateHeaderKey(Y, sync_hash)` but carries shard X's state root, and the node subsequently reconstructs shard X's state instead of shard Y's state.

---

### Finding Description

In `set_state_header` the function receives two peer-controlled inputs: `shard_id` (extracted from `StateResponseInfo::shard_id()` in the network message) and `shard_state_header` (the full header body). It then runs four checks:

1. `validate_chunk_proofs(&chunk, …)` — validates the chunk's internal transaction/receipt roots.
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — proves the chunk is somewhere in the block's chunk-headers Merkle tree.
3. Receipt-proof chain validation.
4. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the state root node.

None of these checks compare `chunk.shard_id()` against the `shard_id` parameter. The header is then persisted under `StateHeaderKey(shard_id, sync_hash)`: [1](#0-0) 

A block with N shards contains N chunks, one per shard. The Merkle proof for shard X's chunk is a valid proof against `chunk_headers_root` — it just proves inclusion at shard X's position, not at shard Y's position. The code never checks which position the proof covers. A malicious peer can therefore:

- Provide `shard_id = Y` in the `StateResponseInfo` envelope.
- Provide a `shard_state_header` whose embedded chunk is shard X's chunk (with a valid Merkle proof for shard X's position).

All four checks pass because shard X's chunk is genuinely valid and genuinely in the block. The header is stored under `StateHeaderKey(Y, sync_hash)` but contains shard X's `prev_state_root`. [2](#0-1) 

The missing guard is:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other("set_shard_state failed: chunk shard_id mismatch".into()));
}
```

The downstream `set_state_part` then retrieves the stored header, extracts the (wrong) state root, and validates incoming parts against it: [3](#0-2) 

Parts for shard X pass `validate_state_part(shard_id=Y, state_root=shard_X_root, …)` because the Merkle proof is against shard X's root. The node accumulates shard X's trie data under shard Y's key and applies it as shard Y's state.

---

### Impact Explanation

A validator node that state-syncs shard Y ends up with shard X's trie applied as shard Y's state. When it subsequently produces chunks for shard Y it uses the wrong state root, generating invalid chunks. Depending on protocol enforcement this leads to chunk rejection, inability to advance the chain for that shard, and potential slashing of the validator's stake. The corrupted state is written to `DBCol::StateParts` and `DBCol::StatePartsApplied`, so the damage persists across restarts until a full re-sync.

---

### Likelihood Explanation

Any network peer can serve `StateResponseInfo` messages. A peer that advertises itself via a signed `SnapshotHostInfo` (which only requires the peer's own node key — no validator stake) and is selected by the syncing node as a snapshot source can execute this attack. The `ClientActor` handler validates `sync_hash` but not `shard_id`: [4](#0-3) 

If the malicious peer is the sole advertiser for the current epoch's snapshot, or if it can win the priority-score race in `SnapshotHostsCache`, it is selected unconditionally. No validator role or privileged position is required.

---

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind the peer-supplied chunk to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This mirrors the existing `shard_ids.contains(&shard_id)` guard in `get_state_response_part` and closes the analogous gap on the receiving side. [5](#0-4) 

---

### Proof of Concept

1. Network has 4 shards (IDs 0–3). Syncing node needs state for shard 1 (`shard_id=1`).
2. Malicious peer advertises itself as a snapshot host via a signed `SnapshotHostInfo { shards: [0,1,2,3], sync_hash: H, epoch_height: E }`.
3. Syncing node selects the malicious peer and sends `StateRequestHeader { shard_id: 1, sync_hash: H }`.
4. Malicious peer responds with `StateResponseInfo { shard_id: 1, sync_hash: H, state_response: <header_for_shard_0> }` — the header contains shard 0's chunk and a valid Merkle proof for shard 0's position in the block.
5. `set_state_header(shard_id=1, sync_hash=H, header_for_shard_0)` is called.
   - `validate_chunk_proofs` passes (shard 0's chunk is internally valid).
   - `verify_path(chunk_headers_root, proof_for_shard_0, ChunkHashHeight(shard_0_hash, …))` passes (shard 0 is in the block).
   - Receipt proofs and state root node pass (all consistent with shard 0).
6. Header stored under `StateHeaderKey(1, H)` with shard 0's `prev_state_root`.
7. Syncing node requests parts for shard 1; malicious peer supplies shard 0's parts.
8. `set_state_part(shard_id=1, …)` validates each part against shard 0's state root — all pass.
9. Node applies shard 0's trie as shard 1's state. Shard 1's state is now corrupted.
10. Node produces chunks for shard 1 using shard 0's state root → invalid chunks → slashing risk.

        let sync_block_header = self.chain_store.get_block_header(&sync_hash)?;

        let chunk = shard_state_header.cloned_chunk();
        let prev_chunk_header = shard_state_header.cloned_prev_chunk_header();

        // 1-2. Checking chunk validity
