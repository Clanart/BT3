### Title
Missing Deprecated Class Body Hash Verification in P2P Sync Allows Malicious Peer to Corrupt Cairo 0 Bytecode — (`crates/apollo_p2p_sync/src/client/class.rs`)

---

### Summary

The p2p sync client accepts deprecated (Cairo 0) contract class bodies from peers and stores them under a caller-supplied `class_hash` key without verifying that `hash(body) == class_hash`. A malicious peer can substitute an arbitrary bytecode body under any deprecated class hash that legitimately appears in a synced state diff, causing the node to permanently store and execute the wrong Cairo 0 bytecode for every contract whose class is that hash.

---

### Finding Description

**Entrypoint — `parse_data_for_block`**

When the p2p sync client receives a `(ApiContractClass::DeprecatedContractClass(body), class_hash)` tuple from a peer, the only check performed is membership: does `class_hash` appear in the state diff's `deprecated_declared_classes` set? [1](#0-0) 

There is no computation of `hash(body)` and no comparison against `class_hash`. The body is accepted as-is.

**Storage path — `write_to_storage`**

The accepted `(class_hash, body)` pair is forwarded directly to the class manager: [2](#0-1) 

**Class manager — `add_deprecated_class`**

The class manager stores the body under the provided `class_id` with no hash derivation: [3](#0-2) 

**Cached storage — `set_deprecated_class`**

The storage layer writes the body under `class_id` unconditionally (the only guard is a cache-hit early-return, not a hash check): [4](#0-3) 

**Contrast with Sierra (`add_class`)**

For Cairo 1 classes, the class manager *derives* the key by computing `sierra_class.calculate_class_hash()` from the body itself, so the peer-supplied hash is never used as the storage key: [5](#0-4) 

There is no equivalent derivation for deprecated classes. The peer-supplied `class_hash` is used verbatim as the storage key.

**Acknowledged gap**

A TODO on the Sierra path notes that even there the returned hash is not yet cross-checked against the state diff: [6](#0-5) 

No equivalent TODO exists for the deprecated class path, meaning the gap is not even tracked.

**SNOS check does not protect the execution path**

The SNOS hint processor does verify the hash at proof-generation time: [7](#0-6) 

However, this check runs only during SNOS proof generation, not during blockifier execution or RPC serving. A syncing node that does not run SNOS (the common case for full nodes) will never trigger this guard. Even on a node that does run SNOS, the wrong body is already stored and served to the blockifier before SNOS runs.

---

### Impact Explanation

After the corrupt write, every call to `get_executable(D)` returns body `B` (the attacker-supplied body) instead of the legitimate preimage of `D`. The blockifier executes `B` as the Cairo 0 bytecode for class `D`, producing wrong execution results for every transaction that invokes a contract whose class is `D`. This maps directly to the allowed impact: **"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution"** and **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic."**

For a node serving RPC, fee estimation, simulation, and tracing all return authoritative-looking wrong values. For a sequencer node, wrong block production follows.

---

### Likelihood Explanation

Any peer that the syncing node connects to can trigger this. The attacker only needs to be a reachable p2p peer and to know a `class_hash` that will appear in an upcoming state diff (which is public information on-chain). No operator privileges are required.

---

### Recommendation

In `parse_data_for_block`, after receiving a deprecated class body, compute its hash using `compute_deprecated_class_hash` and compare it against the peer-supplied `class_hash`. Reject the peer (return `BadPeerError`) if they do not match. This mirrors the implicit protection that `add_class` provides for Sierra classes by deriving the key from the body.

Alternatively, `add_deprecated_class` in the class manager can be changed to compute and return the hash from the body (analogous to `add_class`), and the p2p sync layer can then verify the returned hash against the state diff entry.

---

### Proof of Concept

```rust
// Pseudocode for a Rust unit test
let mut class_manager = ClassManager::new_for_testing(...);
let real_class_hash = ClassHash(felt!("0xDEAD"));

// Attacker body: any valid DeprecatedContractClass whose hash != real_class_hash
let attacker_body = RawExecutableClass::try_from(
    ContractClass::V0(DeprecatedContractClass { /* arbitrary */ ..Default::default() })
).unwrap();

// This call succeeds with no error — no hash check performed
class_manager.add_deprecated_class(real_class_hash, attacker_body.clone()).unwrap();

// The node now serves the wrong bytecode for real_class_hash
let stored = class_manager.get_executable(real_class_hash).unwrap();
assert_eq!(stored, Some(attacker_body)); // wrong body stored under real_class_hash
```

The existing test `class_manager_get_executable` already demonstrates this pattern — it passes an arbitrary `deprecated_class_hash` and body with no hash relationship enforced: [8](#0-7)
