### Title
Re-entrancy in `initTransfer1155` via Malicious ERC1155 Token Causes Nonce Collision and Bridge Accounting Corruption — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155` increments `currentOriginNonce` at the top of the function but then reads it again from storage **after** an external `IERC1155.safeTransferFrom` call. Because any user can supply an arbitrary ERC1155 token address, a malicious token can re-enter `initTransfer1155` during `safeTransferFrom`, causing the outer call to emit an `InitTransfer` event with the nonce that was already consumed by the inner call. This produces two `InitTransfer` events sharing the same nonce while one nonce is permanently skipped, corrupting the bridge's cross-chain accounting.

---

### Finding Description

In `initTransfer1155`:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;          // (1) nonce incremented to N
    ...
    IERC1155(tokenAddress).safeTransferFrom(   // (2) external call — re-entrancy window
        msg.sender, address(this), tokenId, amount, ""
    );
    ...
    initTransferExtension(
        msg.sender, deterministicToken,
        currentOriginNonce,            // (3) reads storage AFTER external call
        ...
    );
    emit BridgeTypes.InitTransfer(
        msg.sender, deterministicToken,
        currentOriginNonce,            // (4) reads storage AFTER external call
        ...
    );
}
``` [1](#0-0) 

`currentOriginNonce` is a storage variable. It is incremented to `N` at step (1), but steps (3) and (4) read it from storage again after the external call at step (2). There is no local snapshot of the nonce before the external call, and there is no `ReentrancyGuard` or `nonReentrant` modifier anywhere in the contract. [2](#0-1) 

`initTransfer1155` accepts **any** caller-supplied `tokenAddress` with no whitelist check, so an attacker can freely supply a malicious ERC1155 contract whose `safeTransferFrom` implementation calls back into `initTransfer1155` before returning. [3](#0-2) 

The same structural flaw exists in `initTransfer` for the plain-ERC20 path: `currentOriginNonce` is read from storage at lines 418 and 430 after the external `safeTransferFrom` / `burn` calls at lines 394–412, making it exploitable via ERC-777 `tokensToSend` hooks or any custom ERC-20 with a transfer callback. [4](#0-3) 

---

### Impact Explanation

**Attack trace:**

| Step | `currentOriginNonce` | Action |
|------|----------------------|--------|
| Outer call enters `initTransfer1155(MaliciousERC1155, …)` | N | nonce incremented |
| `MaliciousERC1155.safeTransferFrom` fires re-entrant call `initTransfer1155(RealToken, …)` | N+1 | inner call increments nonce |
| Inner call completes: `initTransferExtension(…, N+1, …)` + `emit InitTransfer(nonce=N+1)` | N+1 | Wormhole/event published for N+1 |
| Outer call resumes: `initTransferExtension(…, currentOriginNonce=N+1, …)` + `emit InitTransfer(nonce=N+1)` | N+1 | **duplicate** Wormhole/event published for N+1 |

Results:
- **Nonce N is permanently skipped** — no `InitTransfer` event is ever emitted for it.
- **Two `InitTransfer` events carry nonce N+1** — one for the legitimate inner transfer, one for the attacker's outer (fake) transfer.
- In the `OmniBridgeWormhole` variant, two Wormhole messages are published with the same `originNonce`. If the NEAR-side `fin_transfer` / `add_fin_transfer` replay guard is bypassed or the messages arrive out of order, the attacker's fake outer transfer can be finalized on NEAR, minting unbacked tokens (critical direct theft / unauthorized mint). [5](#0-4) 

Even with NEAR-side replay protection, the duplicate nonce N+1 event permanently corrupts the bridge's event log and the skipped nonce N breaks sequential nonce integrity, which relayers and provers depend on for correct cross-chain accounting. [6](#0-5) 

---

### Likelihood Explanation

- `initTransfer1155` is a public, permissionless function — any bridge user can call it with an arbitrary `tokenAddress`.
- Deploying a malicious ERC1155 contract requires no special privileges.
- No reentrancy guard exists anywhere in `OmniBridge` or `OmniBridgeWormhole`.
- The `onERC1155Received` guard (`operator != address(this)`) only blocks direct sends to the bridge; it does not prevent the malicious token's `safeTransferFrom` from calling back into `initTransfer1155` before invoking the callback. [7](#0-6) 

---

### Recommendation

1. **Capture the nonce as a local variable before any external call** and use that local copy in `initTransferExtension` and `emit`:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce;   // snapshot before external call
    ...
    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
    ...
    initTransferExtension(msg.sender, deterministicToken, nonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, nonce, ...);
}
```

Apply the same fix to `initTransfer`.

2. **Add `ReentrancyGuardUpgradeable`** from OpenZeppelin and apply `nonReentrant` to both `initTransfer` and `initTransfer1155` as defense-in-depth.

---

### Proof of Concept

```solidity
contract MaliciousERC1155 is ERC1155 {
    OmniBridge bridge;
    IERC1155   realToken;
    uint256    realTokenId;
    bool       reentered;

    function safeTransferFrom(
        address from, address to, uint256 id, uint256 amount, bytes memory
    ) public override {
        if (!reentered) {
            reentered = true;
            // Re-enter with the real token — inner call uses nonce N+1
            bridge.initTransfer1155{value: msg.value}(
                address(realToken), realTokenId, amount, 0, 0, "attacker.near", ""
            );
        }
        // Outer call resumes; currentOriginNonce is now N+1 (wrong)
        // emit InitTransfer(nonce=N+1) is fired again by the outer call
    }
}

// Attack:
// 1. Deploy MaliciousERC1155 pointing at bridge + realToken
// 2. Call bridge.initTransfer1155(address(malicious), ...)
//    → outer call: nonce N assigned, safeTransferFrom triggers re-entry
//    → inner call: nonce N+1, real tokens locked, InitTransfer(N+1) emitted
//    → outer call resumes: InitTransfer(N+1) emitted again (duplicate)
// Result: nonce N skipped; two events with nonce N+1 published to Wormhole/NEAR
```
