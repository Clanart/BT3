### Title
Unmetered `ForEachStorage` Iteration in `DeleteAccount` During `stateDB.Commit()` Enables On-Demand Block-Processing Stall via SELFDESTRUCT — (File: `x/evm/keeper/statedb.go`)

---

### Summary

`keeper.DeleteAccount()` calls `ForEachStorage()` to clear every storage slot of a self-destructed contract. This call is made from `statedb.Commit()` after EVM gas accounting is already complete. Because the EIP-3529 SELFDESTRUCT opcode carries a fixed gas cost (5,000 gas) that is entirely independent of the number of storage slots held by the contract, an attacker can accumulate an arbitrarily large number of storage slots across many prior blocks and then trigger their deletion in a single transaction. The resulting KV-store iteration and deletion work is proportional to the accumulated slot count, not to the gas paid in the destructing transaction.

---

### Finding Description

**Root-cause chain:**

1. `ApplyTransaction` (`x/evm/keeper/state_transition.go:163`) calls `ApplyMessageWithConfig` with `commit=true`.
2. Inside `ApplyMessageWithConfig` (`state_transition.go:602`), after the EVM has finished executing and all EVM gas has been accounted for, `stateDB.Commit()` is called.
3. `statedb.Commit()` (`x/evm/statedb/statedb.go:800–825`) iterates over every dirty address. For any address marked `selfDestructed`, it calls `keeper.DeleteAccount(cacheCtx, obj.Address())`.
4. `keeper.DeleteAccount()` (`x/evm/keeper/statedb.go:209–212`) calls `ForEachStorage`, which opens a KV-store prefix iterator over the entire storage trie of the contract and issues a `SetState(ctx, addr, key, nil)` delete for every slot found.

```
// x/evm/keeper/statedb.go:209-212
k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
    k.SetState(ctx, addr, key, nil)
    return true
})
```

5. `ForEachStorage` (`x/evm/keeper/statedb.go:58–74`) is an unbounded KV-store prefix iterator — it visits every slot stored under the contract's address prefix with no early-exit limit.

**Why the iteration is unmetered:**

The EVM gas for SELFDESTRUCT is a flat 5,000 gas (post-EIP-3529; storage-clearing refunds were removed). EVM gas accounting finishes before `Commit()` is entered. The `Commit()` path runs against `origCtx` (or a `CacheContext` of it), whose gas meter is the SDK transaction gas meter. Critically, EVM execution itself does not consume SDK gas — the SDK gas meter is not decremented during the EVM `Call`/`Create` execution loop. Therefore, when `Commit()` begins, the SDK gas meter still has nearly the full transaction gas limit available, and the N KV-store deletes issued by `ForEachStorage` consume that budget at the SDK store's per-operation rate rather than at any EVM-proportional rate. If the SDK context uses an infinite gas meter (the standard Ethermint pattern for EVM execution contexts), the iteration is completely unmetered.

**Attacker-controlled amplification:**

- Over K prior blocks, the attacker deploys a contract and fills it with N storage slots via SSTORE (paying EVM gas proportional to N, spread across K blocks).
- In a single subsequent block, the attacker submits one SELFDESTRUCT transaction paying only 5,000 EVM gas.
- `Commit()` then performs N KV-store deletes — work proportional to N, not to the 5,000 gas paid in the current block.
- N is bounded only by the cumulative gas the attacker has spent across all prior blocks, not by the current block's gas limit.

---

### Impact Explanation

The `ForEachStorage` loop in `DeleteAccount` performs one KV-store iterator step and one KV-store delete per slot. With N = 100,000 slots (achievable by spending ~2,100 × 100,000 = 210M gas across prior blocks at cold-SSTORE prices), the commit phase of a single SELFDESTRUCT transaction performs 100,000 unmetered store deletes. This can stall block finalization for an extended, attacker-controlled duration, constituting a deterministic, on-demand chain halt reachable by any unprivileged user.

Impact category matched: **Critical — valid unprivileged transaction in block-processing path can halt the chain or cause deterministic validator consensus failure.**

---

### Likelihood Explanation

- No privileged role, governance action, or validator compromise is required.
- The attacker only needs to deploy a contract, fill its storage over multiple blocks, and submit one SELFDESTRUCT call.
- SELFDESTRUCT remains available on chains that have not yet activated EIP-6780 (which restricts it to same-transaction creation), and even under EIP-6780 the same pattern applies if the contract is created and destructed in the same transaction with pre-filled storage from a prior deployment at the same address.
- The cost to mount the attack is proportional to the number of slots filled, but the per-block impact is amplified because the destructing transaction pays only 5,000 gas while causing O(N) commit-phase work.

---

### Recommendation

Replace the full-scan `ForEachStorage` deletion in `DeleteAccount` with lazy/deferred storage clearing:

1. **Tombstone approach**: Instead of iterating and deleting all slots immediately, write a single "account deleted" tombstone key. On any subsequent read of a storage slot for that address, check for the tombstone and return zero. A background or end-of-block pruning job can clear the slots with proper gas accounting.
2. **Charge EVM gas proportional to storage size before SELFDESTRUCT**: Before marking the account as self-destructed, count the number of storage slots (or maintain a per-account slot counter) and charge EVM gas proportional to that count, so the cost of the commit-phase iteration is always covered by the transaction's gas limit.
3. **Bound the per-transaction deletion**: Cap the number of slots deleted per `DeleteAccount` call and defer the remainder to subsequent blocks, similar to how some chains handle large state deletions.

---

### Proof of Concept

```go
// Deploy a contract that fills its own storage:
//   for i := 0; i < N; i++ { SSTORE(i, 1) }
// Spread across multiple blocks to stay within per-block gas limits.
// Then in one block submit:
//   SELFDESTRUCT(beneficiary)
//
// Observe that statedb.Commit() in that block calls:
//   keeper.DeleteAccount(cacheCtx, contractAddr)
//     -> ForEachStorage(ctx, contractAddr, func(key, _) { SetState(ctx, contractAddr, key, nil) })
// which iterates and deletes all N slots with no EVM-gas bound,
// stalling block finalization proportionally to N.
```

func (k *Keeper) ForEachStorage(ctx sdk.Context, addr common.Address, cb func(key, value common.Hash) bool) {
	store := ctx.KVStore(k.storeKey)
	prefix := types.AddressStoragePrefix(addr)

	iterator := storetypes.KVStorePrefixIterator(store, prefix)
	defer iterator.Close()

	for ; iterator.Valid(); iterator.Next() {
		key := common.BytesToHash(iterator.Key())
		value := common.BytesToHash(iterator.Value())

		// check if iteration stops
		if !cb(key, value) {
			return
		}
	}
}

func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
