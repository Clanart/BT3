### Title
`ProofOfInclusion::valid()` Does Not Verify Against a Trusted Root Hash, Enabling Forged Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` only checks the internal self-consistency of the proof chain. It never compares the computed root against any externally-supplied trusted root. Because `root_hash()` is derived entirely from the proof's own fields, the final equality check is a tautology: it is always `true` whenever the loop completes without an early return. An attacker can construct a `ProofOfInclusion` for any arbitrary leaf hash that passes `valid()` without the leaf existing in the real tree.

**The tautology:** After the `for` loop, `existing_hash` holds the last `calculated_hash`. The loop only continues when `calculated_hash == layer.combined_hash`, so on exit `existing_hash == self.layers.last().combined_hash`. But `root_hash()` also returns `self.layers.last().combined_hash`. Therefore the final check `existing_hash == self.root_hash()` is **always `true`** when the loop completes without an early return.

The function never accepts a trusted root hash parameter. It has no way to compare the computed root against the actual tree root stored in the `MerkleBlob`. Any internally-consistent proof — regardless of what tree it claims to belong to — passes `valid()`.

This is structurally identical to the H-02 pattern: just as `SpigotLib.claimRevenue` read `self.settings[revenueContract].ownerSplit` without checking registration (getting a default zero), `valid()` reads `self.root_hash()` without checking against a trusted external root (getting a self-referential value that is always equal to the computed hash).

### Impact Explanation

Any code that calls `proof.valid()` on an externally-received `ProofOfInclusion` to decide whether a DataLayer key-value pair is present in a tree will accept forged proofs. An attacker can:

1. Pick any `node_hash` (representing a fake leaf with any `KeyId`/`ValueId`).
2. Build a chain of `ProofOfInclusionLayer` values where each `combined_hash` is correctly computed from the previous hash and a chosen `other_hash`.
3. Serialize the struct via `Streamable` and send it to a verifier.
4. The verifier calls `valid()`, which returns `true`.

This lets untrusted input prove invalid DataLayer state — matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`ProofOfInclusion` is a network-exchanged object (it is `Streamable` and Python-exposed). Any DataLayer peer that receives a proof from an untrusted source and calls `valid()` without separately verifying the root hash is exploitable. The construction of a valid forgery requires only computing SHA-256 hashes in a chain — no secret knowledge is needed.

---

### Recommendation

`valid()` must accept a trusted root hash and compare against it:

```rust
pub fn valid_for_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root  // compare against externally-supplied root
}
```

The no-argument `valid()` should either be removed or clearly documented as only checking internal consistency (not authenticity). All call sites that verify externally-received proofs must be updated to supply the trusted root from the local `MerkleBlob`.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forge_proof(fake_node_hash: Hash, fake_sibling_hash: Hash) -> ProofOfInclusion {
    // Compute a combined_hash from the two fake hashes
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash, Side::Left, &fake_sibling_hash
    );
    ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: fake_sibling_hash,
            combined_hash: combined,  // internally consistent
        }],
    }
}

fn main() {
    let fake_node_hash = [0xAA; 32];
    let fake_sibling_hash = [0xBB; 32];
    let forged = forge_proof(fake_node_hash, fake_sibling_hash);
    // Returns true even though this node is not in any real tree
    assert!(forged.valid());
}
```

The forged proof passes `valid()` because the final check compares `existing_hash` against `self.root_hash()`, which is `self.layers.last().combined_hash` — the same value that `existing_hash` was just set to. [2](#0-1)
