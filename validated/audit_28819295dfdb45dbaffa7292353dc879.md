### Title
Native SOL `native_fee` Permanently Locked in `sol_vault` with No Withdrawal Path - (File: `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

When a user calls `init_transfer` or `init_transfer_sol` on the Solana bridge with a non-zero `native_fee`, the SOL is transferred into the `sol_vault` PDA. However, the `sol_vault` is only ever debited during `finalize_transfer_sol` to pay the **transfer amount** to the recipient — never to pay out the `native_fee` portion. There is no instruction in the program that withdraws accumulated `native_fee` SOL from `sol_vault` to a fee recipient. The `native_fee` SOL accumulates in `sol_vault` permanently with no recovery path.

---

### Finding Description

In `init_transfer`, when `payload.native_fee > 0`, the full `native_fee` lamports are transferred from the user into `sol_vault`:

```rust
// solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs:75-86
if payload.native_fee > 0 {
    transfer(
        CpiContext::new(
            self.common.system_program.to_account_info(),
            Transfer {
                from: self.user.to_account_info(),
                to: self.sol_vault.to_account_info(),
            },
        ),
        payload.native_fee,
    )?;
}
```

Similarly, in `init_transfer_sol`, both `amount` and `native_fee` are deposited together into `sol_vault`:

```rust
// solana/programs/bridge_token_factory/src/instructions/user/init_transfer_sol.rs:39-53
transfer(..., payload.native_fee.checked_add(payload.amount...)?)?;
```

The only instruction that ever withdraws from `sol_vault` is `finalize_transfer_sol`, which transfers exactly `data.amount` to the recipient:

```rust
// solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer_sol.rs:79-89
transfer(
    CpiContext::new_with_signer(..., Transfer { from: self.sol_vault, to: self.recipient }),
    data.amount.try_into()...?,
)?;
```

The `native_fee` portion is never extracted from `sol_vault`. The `FinalizeTransferResponse` records a `fee_recipient` field, but no SOL is ever sent to that address from `sol_vault`. There is no admin withdrawal instruction, no fee-claim instruction, and no other code path that debits `sol_vault` for the `native_fee` component.

Searching the entire `solana/programs/bridge_token_factory/src/instructions/` directory confirms there is no instruction that transfers `native_fee` lamports out of `sol_vault` to any fee recipient.

---

### Impact Explanation

Every `init_transfer` or `init_transfer_sol` call with `native_fee > 0` permanently locks that SOL in `sol_vault`. Since `sol_vault` is a PDA owned by the bridge program and there is no instruction to withdraw the accumulated fees, the SOL is irrecoverable. Over time, as users pay native fees to incentivize relayers, all of those lamports accumulate in `sol_vault` and are permanently frozen. This matches the **Critical** impact class: permanent freezing / irrecoverable lock of user funds in the bridge vault flow.

---

### Likelihood Explanation

`native_fee` is a first-class parameter in `InitTransferPayload` and is actively used in the bridge protocol to compensate relayers for cross-chain gas costs. Any user who sets `native_fee > 0` when calling `init_transfer` or `init_transfer_sol` triggers this loss. This is a normal, expected usage path — not an edge case.

---

### Recommendation

Add an instruction (e.g., `withdraw_native_fee`) that allows the designated fee recipient (as recorded in the Wormhole VAA / `FinalizeTransferResponse.fee_recipient`) or an admin to withdraw accumulated `native_fee` lamports from `sol_vault`. Alternatively, during `finalize_transfer_sol`, after transferring `data.amount` to the recipient, also transfer the corresponding `native_fee` to `data.fee_recipient` from `sol_vault`.

---

### Proof of Concept

1. User calls `init_transfer` with `native_fee = 1_000_000` lamports (0.001 SOL).
2. `init_transfer.rs` line 76–85 transfers 1,000,000 lamports from user → `sol_vault`.
3. NEAR side processes the transfer, signs a `FinalizeTransferPayload` with `fee_recipient = Some("relayer.near")`.
4. Relayer calls `finalize_transfer_sol` — only `data.amount` is transferred to the recipient; `native_fee` is never sent anywhere.
5. `sol_vault` balance is now `initial_balance + 1_000_000 - data.amount`. The 1,000,000 lamport `native_fee` remains permanently in `sol_vault`.
6. Repeated across all bridge users, the total locked SOL grows unboundedly with no recovery mechanism. [1](#0-0) [2](#0-1) [3](#0-2)
