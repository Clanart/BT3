Audit Report

## Title
`addLiquidityExactShares` lacks deadline protection, enabling sandwich attacks on liquidity deposits — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

Both overloads of `addLiquidityExactShares` in `MetricOmmPoolLiquidityAdder` accept no `deadline` parameter and perform no expiry check before executing the liquidity deposit. A pending transaction can be held in the mempool and executed after an attacker has manipulated the pool's active-bin cursor, causing the user's tokens to be deposited into bins at a manipulated composition. The resulting LP position is worth materially less than the tokens spent, constituting a direct loss of user principal.

## Finding Description

`addLiquidityExactShares` (both overloads, L56–81) performs only structural validation (`_validateOwner`, `_validateDeltas`) before calling `_addLiquidity`. There is no deadline parameter and no call to any expiry guard:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L56-68
function addLiquidityExactShares(
    address pool, address owner, uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
}
```

By contrast, every entry point in `MetricOmmSimpleRouter` calls `_checkDeadline` (defined in `MetricOmmSwapRouterBase` L91–94) as its first action. `MetricOmmPoolLiquidityAdder` inherits only `PeripheryPayments`, not `MetricOmmSwapRouterBase`, so `_checkDeadline` is entirely absent.

The pool's `LiquidityLib.addLiquidity` determines token composition from the live cursor at execution time (L86–107): bins strictly above `curBinIdx` receive only token0; bins strictly below receive only token1; the active bin receives a mix proportional to `curPosInBin`. Because `addLiquidityExactShares` fixes `binIdxs` and `shares` but not the cursor, the token composition of the deposit is fully determined by whatever cursor position the pool happens to be in at inclusion time.

The only existing guard — the `maxAmountToken0`/`maxAmountToken1` callback check (L165–167) — prevents paying *more* than the cap but does not prevent the user from receiving a position whose market value is less than the tokens paid.

`addLiquidityWeighted` explicitly documents and enforces cursor bounds (`minimalCurBin`/`maximalCurBin` via `_validateBinAndBinPosition`, L104/137) as the mitigation for exactly this class of manipulation. `addLiquidityExactShares` has neither cursor bounds nor a deadline.

**Exploit path:**
1. User submits `addLiquidityExactShares` targeting bins `[X-1, X, X+1]` while the active bin is `X`. Transaction sits in the mempool.
2. Attacker front-runs: swaps token0 → token1, moving the cursor from `X` to `X-5`. Bins `X-1`, `X`, `X+1` are now all above the cursor.
3. User's transaction executes: `LiquidityLib` sees `binIdx > curBinIdxCache` for all three bins → all receive only token0 (within `maxAmountToken0` cap, so callback does not revert).
4. Attacker back-runs: swaps token1 → token0, moving the cursor back through bins `X-5` to `X+1`. The cursor traversal consumes the user's deposited token0 from bins `X-1`, `X`, `X+1` at the manipulated (lower) price, leaving those bins holding token1.
5. Attacker pockets the arbitrage spread; user's LP position holds token1 acquired at a worse-than-market rate — worth less than the token0 deposited.

## Impact Explanation

The user suffers a direct loss of deposited principal: token0 is effectively sold at a manipulated price through the LP position, with no recovery path. The loss scales with the size of the deposit and the magnitude of the cursor manipulation. This matches the "direct loss of user principal" impact gate.

## Likelihood Explanation

Any `addLiquidityExactShares` transaction visible in the public mempool is a target. No privileged access is required; the attacker only needs to execute two swaps around the victim's transaction. MEV bots routinely perform this pattern on liquidity-provision calls that lack deadline or price-bound guards. The attack is repeatable on every such pending transaction.

## Recommendation

Add a `uint256 deadline` parameter to both `addLiquidityExactShares` overloads and call `_checkDeadline(deadline)` (or an equivalent inline `block.timestamp` check) at the top of each function, mirroring the pattern in `MetricOmmSwapRouterBase._checkDeadline`. Additionally, consider exposing optional cursor-bound parameters (as `addLiquidityWeighted` already does via `minimalCurBin`/`maximalCurBin`) so callers can bound the acceptable pool state at execution time.

## Proof of Concept

```solidity
// Foundry fork test outline
function test_sandwich_addLiquidityExactShares() public {
    // Setup: pool with active bin X=0, user targets bins [-1, 0, 1]
    // 1. Attacker front-runs: swap token0→token1, move cursor to X=-5
    pool.swap(attacker, true, largeAmount, MIN_PRICE, "", "");
    // 2. User's addLiquidityExactShares executes (no deadline → no revert)
    //    LiquidityLib sees bins -1,0,1 all > curBinIdx(-5) → all receive only token0
    adder.addLiquidityExactShares(address(pool), 0, deltas, maxToken0, maxToken1, "");
    // 3. Attacker back-runs: swap token1→token0, cursor moves through bins -1,0,1
    //    consuming user's token0 at manipulated price
    pool.swap(attacker, false, largeAmount, MAX_PRICE, "", "");
    // Assert: user's position value < token0 deposited
    uint256 positionValue = computePositionValue(pool, user, 0, bins);
    assertLt(positionValue, token0Deposited);
}
```

The absence of `_checkDeadline` in `addLiquidityExactShares` is the root cause: the transaction cannot be forced to expire before the attacker's setup is in place.
