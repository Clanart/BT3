### Title
Integer Division Truncation in `NewDynamicFeeChecker` Enables Zero-Fee Cosmos Transaction Execution - (File: `ante/evm/fee_checker.go`)

### Summary

`NewDynamicFeeChecker` derives the per-gas fee cap for Cosmos (non-EVM) transactions via integer division `feeCap = fee / gas`. When a user submits a transaction with `fee < gas`, the truncation produces `feeCap = 0`. If the chain runs with London HF enabled but the fee market disabled (`NoBaseFee = true`), `baseFee` is non-nil but equals `0`, so the `feeCap < baseFee` guard passes silently. The downstream `EffectiveGasPrice` call then returns `0`, and `effectiveFee = 0 * gas = 0` coins are deducted — the transaction executes for free.

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` computes the per-gas fee cap for Cosmos SDK transactions:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // line 83 — integer truncation
baseFeeInt := sdkmath.NewIntFromBigInt(baseFee)

if feeCap.LT(baseFeeInt) {                          // line 86
    return nil, 0, errorsmod.Wrapf(...)
}
```

`sdkmath.Int.Quo` truncates toward zero. Any transaction where `fee < gas` (e.g., `fee = 1 aphoton`, `gas = 2`) produces `feeCap = 0`. [1](#0-0) 

The guard on line 86 only rejects when `feeCap < baseFee`. When `baseFee = 0`, the condition `0 < 0` is false, so the check is silently bypassed.

`baseFee = 0` (non-nil) is the documented, reachable state when London HF is active but the fee market module is disabled. `x/evm/types/utils.go:GetBaseFee` explicitly returns `new(big.Int)` (zero) when `feemarketParams.GetBaseFee()` returns nil:

```go
func GetBaseFee(height int64, ethCfg *params.ChainConfig, feemarketParams *feemarkettypes.Params) *big.Int {
    if !IsLondon(ethCfg, height) {
        return nil
    }
    baseFee := feemarketParams.GetBaseFee()
    if baseFee == nil {
        return new(big.Int)   // ← zero, not nil; triggers dynamic-fee path
    }
    return baseFee
}
``` [2](#0-1) 

`feemarketParams.GetBaseFee()` returns nil whenever `NoBaseFee = true`: [3](#0-2) 

This is confirmed by the keeper test: `{"enable london HF, not enable feemarket", true, false, big.NewInt(0)}`. [4](#0-3) 

With `feeCap = 0` and `baseFee = 0`, `EffectiveGasPrice` computes:

```go
func EffectiveGasPrice(baseFee, feeCap, tipCap *big.Int) *big.Int {
    return ethermint.BigMin(new(big.Int).Add(tipCap, baseFee), feeCap)
    // = min(MaxInt64 + 0, 0) = 0
}
``` [5](#0-4) 

`effectiveFee = 0 * gas = 0`. The `DeductFeeDecorator` deducts nothing: [6](#0-5) 

The `MinGasPriceDecorator` that precedes `DeductFeeDecorator` in `newCosmosAnteHandler` short-circuits when `MinGasPrice = 0` (the default), providing no backstop: [7](#0-6) 

`DynamicFeeChecker = true` is hardcoded in the production app setup: [8](#0-7) 

The `NewDynamicFeeChecker` is wired into both `newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712`: [9](#0-8) 

### Impact Explanation

Any unprivileged user can submit a Cosmos SDK transaction (e.g., `MsgSend`, governance, staking) with `fee = 1 aphoton` and `gas = 2` (or any `fee < gas` ratio). The ante handler computes `effectiveFee = 0` and deducts nothing. The transaction is included in a block and committed with zero fee paid. This is a direct mis-accounting of user fees and permits otherwise-invalid (underpaying) transactions to commit, matching the allowed High impact: *"ante handler bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

The vulnerable configuration — London HF enabled, `NoBaseFee = true`, `MinGasPrice = 0` — is the default for chains that enable London HF without activating the fee market module. The production app sets `DynamicFeeChecker = true` unconditionally. No special privileges are required; any account can craft the malicious transaction. The attack is trivially reproducible with a standard Cosmos SDK tx builder by setting `gasLimit = 2` and `feeAmount = 1`.

### Recommendation

Replace the truncating integer division with a ceiling division or an explicit minimum-fee check before computing `feeCap`:

```go
// Option A: reject if fee < gas (feeCap would be 0 and baseFee is 0)
if fee.IsZero() && baseFeeInt.IsZero() {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, "fee cannot be zero when baseFee is zero")
}

// Option B: use ceiling division so feeCap rounds up, not down
feeCap := fee.Add(sdkmath.NewIntFromUint64(gas - 1)).Quo(sdkmath.NewIntFromUint64(gas))
```

Additionally, add an explicit guard: if `feeCap == 0` and `baseFee == 0`, reject the transaction rather than silently accepting it with zero effective fee.

### Proof of Concept

**Preconditions:** London HF active, `NoBaseFee = true`, `MinGasPrice = 0`, `DynamicFeeChecker = true` (all defaults or production settings).

1. Attacker constructs a Cosmos `MsgSend` transaction with `gasLimit = 1_000_000` and `feeAmount = sdk.NewCoins(sdk.NewCoin("aphoton", sdkmath.NewInt(1)))`.
2. `NewDynamicFeeChecker` is invoked:
   - `baseFee = types.GetBaseFee(...)` → `new(big.Int)` (0, non-nil) because `NoBaseFee = true`.
   - `feeCap = 1 / 1_000_000 = 0` (integer truncation).
   - `feeCap.LT(baseFeeInt)` → `0 < 0` → **false**, guard bypassed.
   - `effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = min(MaxInt64, 0) = 0`.
   - `effectiveFee = 0 * 1_000_000 = 0 aphoton`.
3. `DeductFeeDecorator.checkDeductFee`: `fee.IsZero()` → true → **no deduction**.
4. Transaction is committed. Attacker paid 0 fees; the 1 aphoton remains in their account.
