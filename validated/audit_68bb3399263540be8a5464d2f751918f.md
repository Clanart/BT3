### Title
Missing Zero-Address Recipient Validation in `init_transfer` Causes Permanent Irrecoverable Fund Lock — (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge's `init_transfer` function accepts any `OmniAddress` recipient that is not on the `Near` chain, but never checks whether the recipient is the zero address for the destination chain. A user who specifies `OmniAddress::Eth(H160::ZERO)` (i.e., EVM `address(0)`) as the recipient will have their tokens burned or locked on NEAR, while every subsequent `finTransfer` call on the EVM side will revert because OpenZeppelin's ERC20 rejects minting to `address(0)`. The funds are permanently irrecoverable.

### Finding Description

`init_transfer` in `near/omni-bridge/src/lib.rs` performs only one recipient check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

There is no call to `recipient.is_zero()`, even though the method is fully implemented in `OmniAddress`:

```rust
pub fn is_zero(&self) -> bool {
    match self {
        Self::Eth(address) | Self::Arb(address) | ... => address.is_zero(),
        Self::Sol(address) | Self::Fogo(address) => address.is_zero(),
        Self::Strk(address) => address.is_zero(),
        ...
    }
}
``` [2](#0-1) 

After the check passes, `init_transfer_internal` is called, which:
1. Burns the tokens on NEAR (for deployed/bridge tokens) via `burn_tokens_if_needed`, or
2. Locks them in the bridge via `lock_tokens_if_needed`. [3](#0-2) 

The `InitTransferEvent` is emitted, the MPC signs the payload (it signs whatever is requested without validating the recipient), and a relayer submits `finTransfer` on the EVM side with `payload.recipient = address(0)`.

On the EVM side, `finTransfer` dispatches to:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(
        payload.recipient,   // address(0)
        payload.amount
    );
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,   // address(0)
        payload.amount
    );
}
``` [4](#0-3) 

OpenZeppelin's ERC20 `_mint` unconditionally reverts when `account == address(0)` (`ERC20InvalidReceiver`). `safeTransfer` to `address(0)` also reverts in standard ERC20 implementations. The `finTransfer` call will revert on every attempt, the destination nonce is never marked used, and the NEAR-side tokens remain burned/locked with no recovery path.

There is no `cancel_transfer` or admin-rescue function in the NEAR bridge contract. The `set_locked_tokens` admin function can correct the accounting counter, but burned tokens (deployed bridge tokens) are irrecoverable. [5](#0-4) 

The same issue applies to `OmniAddress::Sol(SolAddress::ZERO)` (Solana system program) and `OmniAddress::Strk(H256::ZERO)` (StarkNet zero address), where tokens would be minted to uncontrolled accounts or cause reverts.

### Impact Explanation

- **For deployed bridge tokens**: Tokens are permanently burned on NEAR via `burn_tokens_if_needed`. No tokens are ever minted on EVM. Total supply is permanently reduced. Funds are irrecoverable.
- **For native (locked) tokens**: Tokens are permanently locked in the NEAR bridge contract. `finTransfer` on EVM always reverts. No standard recovery path exists.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` on a NEAR token with an `InitTransferMsg` containing a zero-address recipient. This can happen accidentally (e.g., a frontend bug, a misconfigured script, or a user copy-pasting a blank address) or deliberately. No special role or privilege is required.

### Recommendation

Add a zero-address check in `init_transfer` immediately after the chain-kind check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
require!(
    !init_transfer_msg.recipient.is_zero(),
    BridgeError::InvalidRecipient.as_ref()  // add this error variant
);
``` [1](#0-0) 

Similarly, add a zero-address guard in the EVM `initTransfer` for the `recipient` string parameter (parse and reject `"eth:0x0000...0000"` patterns), and in the Solana `InitTransferPayload` validation.

### Proof of Concept

1. User holds 1000 units of a deployed bridge token on NEAR.
2. User calls `ft_transfer_call` on the token contract with:
   - `receiver_id`: omni-bridge contract
   - `amount`: 1000
   - `msg`: `{"InitTransfer": {"recipient": "eth:0x0000000000000000000000000000000000000000", "fee": "0", "native_token_fee": "0"}}`
3. `init_transfer` passes the chain-kind check (Eth ≠ Near).
4. `init_transfer_internal` burns 1000 tokens on NEAR via `burn_tokens_if_needed`.
5. `locked_tokens[(ChainKind::Eth, token_id)]` is incremented by 1000.
6. `InitTransferEvent` is emitted; MPC signs the payload.
7. Relayer calls `finTransfer` on EVM with `payload.recipient = address(0)`.
8. `IBridgeToken(tokenAddress).mint(address(0), 1000)` → reverts with `ERC20InvalidReceiver(address(0))`.
9. Step 7–8 can be retried indefinitely; it always reverts.
10. The 1000 tokens are permanently burned on NEAR. The user has lost their funds with no recovery path. [6](#0-5) [7](#0-6) [8](#0-7)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
