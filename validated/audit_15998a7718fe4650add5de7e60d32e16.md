### Title
Non-EVM-Native Token Balances Orphaned After `SELFDESTRUCT` — Recoverable via CREATE2 Redeploy (`x/evm/statedb/statedb.go`)

### Summary
When a contract self-destructs in Ethermint, `StateDB.Commit()` correctly burns the EVM-denom balance and removes the auth account. However, non-EVM-native tokens (IBC, CosmWasm-bridged assets) held by the destroyed address are **never drained from the Cosmos bank module**. The auth account is deleted but the bank balances for those denominations remain at the address. A subsequent CREATE2 redeploy at the same address creates a fresh auth account that can immediately access those orphaned balances, enabling theft of Cosmos bank funds.

### Finding Description

The root cause is in `StateDB.Commit()`:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    ...
    s.keeper.SubBalance(cacheCtx, cosmosAddr, coin)
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil { ... }
writeCache()
``` [1](#0-0) 

`DeleteAccount` removes the auth account and EVM storage, but explicitly does **not** touch the bank balance:

```go
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
    ...
    k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool { ... })
    k.accountKeeper.RemoveAccount(ctx, acct)
    ...
}
``` [2](#0-1) 

The `Commit()` loop only queries and burns `s.evmDenom`. Any other denomination held by the contract address in the Cosmos bank is silently left behind. The code comment at line 813 explicitly acknowledges this gap: [3](#0-2) 

The analog to the Maia report is direct: just as removing a `UniswapV3Gauge` from the factory did not propagate the removal to the `UniswapV3Staker`, removing an EVM account via `SELFDESTRUCT` does not propagate the removal to the Cosmos bank module for non-EVM-native denominations.

### Impact Explanation

After `SELFDESTRUCT` commits:
- The EVM auth account is gone (`GetAccount` returns `nil`).
- The EVM-denom bank balance is burned (the fix applied in this repo).
- **All other bank denominations remain at the address with no owner.**

A CREATE2 factory can redeploy a contract at the same deterministic address in a subsequent transaction. The new deployment creates a fresh `EthAccount` at that address. The Cosmos bank module associates balances with addresses, not account objects — so the new account immediately "inherits" the orphaned IBC/bridge balances. The new contract owner can then transfer those funds out via a precompile or native Cosmos transaction.

This constitutes unauthorized transfer of Cosmos bank funds through Ethermint's stateDB commit logic.

### Likelihood Explanation

The attack is reachable by any unprivileged user who can:
1. Deploy a contract via CREATE2 (standard EVM operation).
2. Arrange for non-EVM-native tokens to reach the contract address (e.g., via an IBC transfer to the EVM address, which is a standard Cosmos operation).
3. Call `SELFDESTRUCT` on the contract (requires the contract to expose this path, which is common in proxy/factory patterns).
4. Redeploy at the same CREATE2 address in a later transaction.

No privileged role, governance action, or validator compromise is required. The scenario is realistic in any Ethermint chain that supports IBC and has contracts that hold IBC-bridged assets.

### Recommendation

In `StateDB.Commit()`, after burning the EVM denom, iterate all bank balances held by the destroyed address and burn every denomination, not just `s.evmDenom`:

```go
// Burn ALL denominations, not just evmDenom
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
```

Alternatively, expose a `BurnAllBalances` method on the bank keeper and call it before `DeleteAccount`. This mirrors the recommendation in the Maia report to add an `afterCreateRemoved` callback that propagates the removal to all dependent components.

### Proof of Concept

1. Deploy a CREATE2 factory contract `F` on an Ethermint chain that supports IBC.
2. From `F`, deploy child contract `C` at salt `S` → deterministic address `A`.
3. Via a Cosmos IBC transfer, send 1000 `uatom` to address `A` (standard Cosmos bank send to an EVM address).
4. Call `C.selfdestruct()` — EVM auth account at `A` is deleted, EVM-denom balance is burned, but `uatom` balance at `A` in the bank module remains.
5. From `F`, redeploy `C` at salt `S` → same address `A`. A new `EthAccount` is created at `A`.
6. From the new `C`, call a bank precompile or issue a Cosmos `MsgSend` from `A` to the attacker's address — the 1000 `uatom` are transferred out.

The orphaned balance is confirmed by the code comment at `statedb.go:813–815` and the `DeleteAccount` implementation at `keeper/statedb.go:194` which explicitly states "balance should be cleared separately" but the clearing loop only covers `s.evmDenom`. [4](#0-3) [5](#0-4)

```go
// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
```
