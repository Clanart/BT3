Audit Report

## Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale Liquidity Deposits at Unfavorable Prices — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` exposes four public liquidity-add entry points — two overloads of `addLiquidityExactShares` and two of `addLiquidityWeighted` — none of which accept or enforce a deadline. A transaction pending in the mempool can execute after significant price movement, depositing the caller's tokens into bins that are now far from the current price. The `maxAmountToken0`/`maxAmountToken1` caps bound only the quantity of tokens spent, not the price at which they are committed, leaving users exposed to stale-price LP deployment with no time-based revert path.

## Finding Description
`MetricOmmSwapRouterBase._checkDeadline` is defined at lines 91–94 and called as the first statement in all four swap entry points of `MetricOmmSimpleRouter` (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`).

`MetricOmmPoolLiquidityAdder` does not inherit from `MetricOmmSwapRouterBase` and has no `deadline` parameter or `_checkDeadline` call anywhere in its four public entry points:

- `addLiquidityExactShares(address pool, address owner, …)` — lines 56–68: only calls `_validateOwner` and `_validateDeltas`, then proceeds directly to `_addLiquidity`.
- `addLiquidityExactShares(address pool, …)` — lines 71–81: only calls `_validateDeltas`, then proceeds to `_addLiquidity`.
- `addLiquidityWeighted(address pool, address owner, …)` — lines 88–116: calls `_validateBinAndBinPosition`, which reads `slot0` at probe time and reverts if the cursor is outside user-specified bounds. This is not equivalent to a deadline: the cursor can remain within bounds while the pool's deposit ratio shifts materially.
- `addLiquidityWeighted(address pool, …)` — lines 123–149: same cursor-bounds check, same limitation.

The callback at lines 152–178 enforces `amount0Delta > max0 || amount1Delta > max1` but this only caps quantity, not price. A user who submits `addLiquidityExactShares` targeting bins near the current price can have their transaction delayed. If the cursor moves significantly before inclusion, the targeted bins are now entirely out-of-range; the pool requests only the single-sided token for those bins, and the user's tokens are deposited at the stale price. The user holds LP shares whose composition reflects the old price, not the current market price, and must pay gas to withdraw and redeploy capital.

For `addLiquidityWeighted`, the cursor-bounds check (`_validateBinAndBinPosition`) provides partial mitigation only when the user sets tight bounds. With wide bounds, or when the cursor stays within bounds while the deposit ratio shifts, the same stale-price exposure applies.

## Impact Explanation
This is a bad-price execution issue applied to liquidity addition. In a bin-based AMM, bins below the cursor hold only token1 and bins above hold only token0; only the active bin holds both. A user targeting bins near the current price who is delayed in the mempool will have their tokens deposited into bins that are now entirely out-of-range. Their capital earns zero fees (out-of-range bins are inactive), and when those bins eventually become active, the user's tokens are used to trade at the stale bin price rather than the current market price — constituting a direct loss of LP principal relative to fair-market value. The `maxAmount` caps do not prevent this: they bound quantity, not price. This matches the "bad-price execution" and "direct loss of user principal" allowed impact categories.

## Likelihood Explanation
Any user calling `addLiquidityExactShares` directly through a public mempool is exposed. No privileged actor is required. Network congestion, gas price spikes, or deliberate transaction ordering by a block builder can delay execution by minutes to hours. The `addLiquidityWeighted` path is partially mitigated by cursor bounds but remains vulnerable when bounds are wide or when the deposit ratio shifts within bounds. The condition is repeatable and requires no special attacker capability.

## Recommendation
Add a `deadline` parameter to all four public entry points in `MetricOmmPoolLiquidityAdder` and call `_checkDeadline(deadline)` as the first statement, mirroring the pattern in `MetricOmmSimpleRouter`. Since `MetricOmmPoolLiquidityAdder` does not inherit from `MetricOmmSwapRouterBase`, either duplicate the one-liner or extract it to a shared base:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    uint256 deadline,          // ← add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _checkDeadline(deadline);  // ← add
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
}
```

Apply the same change to both `addLiquidityWeighted` overloads.

## Proof of Concept
1. Pool cursor is at bin 0 (balanced, token0/token1 price = 1:1).
2. Alice calls `addLiquidityExactShares` targeting bins `[-1, 0, 1]` with `maxAmountToken0 = 1000e18`, `maxAmountToken1 = 1000e18`. Transaction enters the public mempool.
3. A large swap moves the cursor to bin 5 (price has risen sharply). Bins `[-1, 0, 1]` are now entirely below the cursor and hold only token1.
4. Alice's transaction is included. The pool callback requests `amount0Delta = 0`, `amount1Delta = 3000e18` (all token1, single-sided). The callback at line 165 passes the cap check (`3000e18 <= maxAmountToken1`).
5. Alice's 3000 token1 are deposited into bins that are now far out-of-range. Her LP shares earn zero fees and, when those bins eventually activate, her token1 will be used to buy token0 at the old (lower) bin prices — a direct loss relative to current market value.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
