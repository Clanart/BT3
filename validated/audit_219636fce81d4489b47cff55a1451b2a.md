### Title
No Recovery Mechanism for Tokens Accidentally Sent via Direct Transfer — (`near/omni-bridge/src/lib.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The Omni Bridge contracts on NEAR and EVM are designed to accept tokens only through bridge-aware transfer functions (`ft_transfer_call` → `ft_on_transfer` on NEAR; `initTransfer` on EVM). If a user accidentally sends tokens via a direct transfer function (`ft_transfer` on NEAR, or a direct ERC20 `transfer()` call on EVM), the tokens arrive at the bridge contract but no bridge record is created and no event is emitted. Neither contract contains an admin rescue function to recover such tokens, making them permanently and irrecoverably locked.

---

### Finding Description

**NEAR side:**

The bridge contract implements `ft_on_transfer`, which is invoked only when a token contract's `ft_transfer_call` is used. [1](#0-0) 

If a user instead calls `ft_transfer(receiver_id = bridge, amount = X)` on a NEP-141 token contract, the token contract transfers `X` tokens to the bridge address but **never calls `ft_on_transfer`**. The bridge receives the tokens silently: no `TransferMessage` is stored, no `InitTransferEvent` is emitted, and no nonce is consumed. The tokens are permanently locked inside the bridge with no on-chain path to recover them.

The bridge has no admin rescue or withdrawal function for NEP-141 tokens. The entire contract state for token handling is: [2](#0-1) 

There is no `rescue_ft`, `withdraw_token`, or equivalent function anywhere in `lib.rs`, `storage.rs`, `token_lock.rs`, or `migrate.rs`.

**EVM side:**

`initTransfer` uses `safeTransferFrom` to pull tokens from the caller: [3](#0-2) 

If a user directly calls `transfer(bridge_address, amount)` on an ERC20 token (bypassing `initTransfer`), the tokens arrive at the bridge but no `InitTransfer` event is emitted and no nonce is recorded. `OmniBridge.sol` contains no `rescueERC20`, `sweepToken`, or equivalent admin function. The full contract is 599 lines and has no token-recovery path. [4](#0-3) 

---

### Impact Explanation

Any tokens sent to the bridge via the wrong transfer function are permanently and irrecoverably locked. There is no admin rescue path, no governance override, and no upgrade path that would allow recovery without a full contract migration. This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

On NEAR, the distinction between `ft_transfer` (simple transfer, no callback) and `ft_transfer_call` (transfer with callback) is non-obvious to many users and is not enforced by wallets or block explorers. A user who manually constructs a transaction or uses a generic wallet UI may call `ft_transfer` to the bridge address, believing it initiates a bridge transfer. On EVM, users who have previously approved the bridge may call `transfer()` directly instead of going through `initTransfer`. Both are realistic user-error paths with no on-chain safeguard.

---

### Recommendation

Add an admin-only rescue function on both chains:

- **NEAR**: Add a `rescue_ft(token_id: AccountId, amount: U128, receiver: AccountId)` function gated by `Role::DAO`. To prevent misuse, require that the rescued amount does not exceed the bridge's actual token balance minus the sum of all locked/pending amounts for that token.
- **EVM**: Add a `rescueERC20(address token, uint256 amount, address receiver) external onlyRole(DEFAULT_ADMIN_ROLE)` function. Guard it so it cannot be used to drain tokens that are legitimately held as collateral (i.e., track or bound the rescue amount against the bridge's accounting).

This mirrors the fix applied in the referenced ImmutableBundle report: allow recovery of tokens that are not associated with any active bridge record.

---

### Proof of Concept

**NEAR:**
1. Alice holds 1000 USDC (a NEP-141 token registered with the bridge).
2. Alice calls `usdc.ft_transfer({ receiver_id: "omni-bridge.near", amount: "1000" })` instead of `usdc.ft_transfer_call({ receiver_id: "omni-bridge.near", amount: "1000", msg: "{...}" })`.
3. The USDC contract transfers 1000 tokens to `omni-bridge.near`. `ft_on_transfer` is never called.
4. No `InitTransferEvent` is emitted. No `TransferMessage` is stored. `current_origin_nonce` is unchanged.
5. Alice's 1000 USDC are permanently locked in the bridge with no recovery path.

**EVM:**
1. Bob holds 500 WETH and has previously approved the bridge.
2. Bob calls `weth.transfer(omniBridgeAddress, 500e18)` directly instead of calling `omniBridge.initTransfer(weth, 500e18, ...)`.
3. The WETH contract transfers 500 WETH to the bridge. No `InitTransfer` event is emitted. `currentOriginNonce` is unchanged.
4. Bob's 500 WETH are permanently locked in `OmniBridge.sol` with no recovery path.

    #[pause(except(roles(Role::DAO, Role::UnrestrictedDeposit)))]
    pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
        let token_id = env::predecessor_account_id();
        let parsed_msg: BridgeOnTransferMsg = serde_json::from_str(&msg)
            .or_else(|_| serde_json::from_str(&msg).map(BridgeOnTransferMsg::InitTransfer))
            .near_expect(BridgeError::ParseMsg);

        // We can't trust sender_id to pay for storage as it can be spoofed.
        let signer_id = env::signer_account_id();
        let promise_or_promise_index_or_value = match parsed_msg {
            BridgeOnTransferMsg::InitTransfer(init_transfer_msg) => {
                self.init_transfer(sender_id, signer_id, token_id, amount, init_transfer_msg)
            }
            BridgeOnTransferMsg::FastFinTransfer(fast_fin_transfer_msg) => {
                self.fast_fin_transfer(token_id, amount, signer_id, fast_fin_transfer_msg)
            }
            BridgeOnTransferMsg::UtxoFinTransfer(utxo_fin_transfer_msg) => self.utxo_fin_transfer(
                token_id,
                amount,
                &signer_id,
                &sender_id,
                utxo_fin_transfer_msg,
            ),
            BridgeOnTransferMsg::SwapMigratedToken => {
                self.swap_migrated_token(sender_id, token_id, amount)
                    .detach();
                PromiseOrPromiseIndexOrValue::Value(U128(0))
            }
        };

        promise_or_promise_index_or_value.as_return();
    }
