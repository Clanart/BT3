### Title
Permanent Token Lock via Dust-Amount Transfer — (`near/omni-bridge/src/lib.rs`)

### Summary
`init_transfer` validates only `fee < amount`, but `sign_transfer` additionally requires `normalize_amount(amount - fee, decimals) > 0`. For any token whose destination has fewer decimals than the origin (`origin_decimals > decimals`), a user can set `fee = amount - 1`, making `amount_without_fee = 1`, which normalizes to `0`. The transfer is stored and tokens are locked, but `sign_transfer` always panics. No user-accessible refund path exists, and `update_transfer_fee` can only increase the fee, making recovery impossible without a privileged contract upgrade.

### Finding Description
The root cause is a missing pre-condition in `init_transfer`: it does not verify that `normalize_amount(amount - fee, decimals) > 0` before locking tokens. The check that enforces this invariant lives exclusively in `sign_transfer`.

Concrete example with a token bridging from NEAR (18 decimals) to EVM (6 decimals), `diff_decimals = 12`:

- User calls `ft_transfer_call` with `amount = 5` (5 raw units), `fee = 4`
- `init_transfer` check: `4 < 5` ✓ — tokens locked, transfer stored
- `sign_transfer`: `normalize_amount(5 - 4, Decimals{origin:18, dest:6}) = 1 / 10^12 = 0` → panic `InvalidAmountToTransfer`
- `update_transfer_fee`: new fee must be `>= 4`, so fee can only go to `4` (no change) — no escape
- Tokens are permanently locked

The amount can be arbitrarily large: `fee = amount - 1` always satisfies `fee < amount` regardless of `amount`.

### Impact Explanation
User funds are permanently frozen in the bridge contract with no protocol-level recovery path. Matches: **Critical — Permanent freezing / irrecoverable lock of user funds**.

### Likelihood Explanation
Reachable by any unprivileged user via the public `ft_transfer_call` → `ft_on_transfer` → `init_transfer` path. Affects any token registered with `origin_decimals > decimals` (common for EVM tokens bridged to NEAR). Can be triggered accidentally (user sets a very high fee) or deliberately (griefing one's own funds, or a confused user).

### Recommendation
Add the normalization check inside `init_transfer` before locking tokens:

```rust
// In init_transfer, after building transfer_message:
let token_address = /* look up destination token address */;
if let Some(decimals) = self.token_decimals.get(&token_address) {
    let net = transfer_message.amount.0
        .checked_sub(transfer_message.fee.fee.0)
        .expect("fee < amount already checked");
    require!(
        Self::normalize_amount(net, decimals) > 0,
        BridgeError::InvalidAmountToTransfer.as_ref()
    );
}
```

Alternatively, enforce `amount - fee >= 10^(origin_decimals - decimals)` as a minimum net-amount constraint at `init_transfer` time.

### Proof of Concept
