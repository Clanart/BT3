### Title
Unauthorized Class Declaration via BOOTSTRAP Path Bypasses Signature Verification — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function contains a special-cased "BOOTSTRAP" path that unconditionally declares a class hash into the OS state **without running any signature verification or account validation**. The conditions that gate this path are entirely user-controlled transaction fields. An unprivileged sender can craft a declare transaction that satisfies these conditions and pre-populate `contract_class_changes` with an arbitrary `compiled_class_hash` for any target `class_hash`, permanently blocking the legitimate declaration of that class.

---

### Finding Description

Inside `execute_declare_transaction`, after computing the transaction hash, the following branch is evaluated: [1](#0-0) 

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
```

When this branch is taken, the function returns **immediately** — skipping:

- `check_and_increment_nonce` (nonce enforcement)
- The `__validate_declare__` entry point (account signature verification)
- `charge_fee` (fee payment)

All four gating conditions are user-controlled fields embedded in the transaction itself:

| Condition | Source |
|---|---|
| `sender_address == 'BOOTSTRAP'` | Arbitrary felt set by the submitter |
| `tx_info.nonce == 0` | Transaction nonce field |
| `tx_info.version == 3` | Always true for V3 declare txs |
| `max_possible_fee == 0` | All resource bounds set to zero |

The `sender_address` and `compiled_class_hash` are loaded from the hint `%{ DeclareTxFields %}` and are committed to by the transaction hash (verified by `%{ AssertTransactionHash %}`), but **no private-key ownership of the `'BOOTSTRAP'` address is ever checked**. [2](#0-1) 

The normal declare path, by contrast, always runs `__validate_declare__` and charges a fee before writing to `contract_class_changes`: [3](#0-2) 

The `dict_update` call uses `prev_value=0`, which enforces that a class hash can only be declared **once**: [4](#0-3) 

This is the direct analog to the CoverRight `deployCRToken` bug: the factory used only `(_poolId, _expiry, _generation)` as a key and allowed anyone to pre-populate the mapping before the privileged function ran. Here, the OS uses only `class_hash` as the key and allows anyone to pre-populate `contract_class_changes` before the legitimate declare transaction runs.

---

### Impact Explanation

**High — Network not being able to confirm new transactions.**

During the bootstrapping phase (when the chain is being initialized and no account contracts exist yet), the gateway cannot perform stateful sender validation against a deployed contract at the `'BOOTSTRAP'` address. An attacker who submits a BOOTSTRAP declare transaction first wins the race:

1. Attacker submits a declare transaction for a critical system class hash (e.g., the fee token class, the account class) with `sender_address = 'BOOTSTRAP'`, `nonce = 0`, `version = 3`, all resource bounds = 0, and an arbitrary (invalid) `compiled_class_hash`.
2. The OS writes `class_hash → attacker_compiled_class_hash` into `contract_class_changes` with `prev_value = 0`.
3. The legitimate sequencer-issued BOOTSTRAP declare for the same `class_hash` now fails because `prev_value` is no longer 0.
4. The critical system contract can never be correctly declared. Any subsequent deployment of that class hash resolves to the attacker's bogus `compiled_class_hash`, causing all executions against it to fail.
5. If the fee token or account class is poisoned this way, no transaction can be validated or charged, halting the network.

---

### Likelihood Explanation

**Low.** The attack window is narrow: it is only viable during the bootstrapping phase before any account contracts are deployed, when the gateway cannot reject the transaction on the basis of a missing sender contract. Once the system is bootstrapped and the gateway enforces sender-contract existence, the `'BOOTSTRAP'` address (a fixed felt literal) will not correspond to any deployed contract and the gateway will reject the transaction. The attacker must also race the legitimate sequencer bootstrap transactions.

---

### Recommendation

Add an explicit access-control assertion before the BOOTSTRAP branch, or remove the branch entirely and require all class declarations — including bootstrap ones — to go through the standard signature-verified path using a pre-deployed privileged account. At minimum, assert that the `sender_address` matches a hard-coded, operator-controlled bootstrap key rather than a publicly known felt literal:

```cairo
// Example mitigation: replace the felt literal check with a
// configurable, operator-controlled bootstrap address stored in
// block_context or os_global_context.
assert sender_address = block_context.bootstrap_operator_address;
```

Alternatively, require that even BOOTSTRAP transactions carry a valid ECDSA signature from a known operator key, verified before the `dict_update` is performed.

---

### Proof of Concept

1. Before the network bootstraps, craft a StarkNet V3 declare transaction with:
   - `sender_address` = felt value of `'BOOTSTRAP'` (ASCII: `0x424F4F545354524150`)
   - `nonce` = `0`
   - `version` = `3`
   - All resource bounds (`l1_gas`, `l2_gas`, `l1_data_gas`) `max_amount` and `max_price_per_unit` = `0`
   - `class_hash` = the known class hash of the fee token (or any critical system class)
   - `compiled_class_hash` = any non-zero felt (e.g., `1`)

2. Submit this transaction. The gateway, unable to validate the sender during bootstrapping, forwards it to the sequencer.

3. The OS executes `execute_declare_transaction`. The BOOTSTRAP branch is entered. `validate_declare` is never called. `dict_update` writes `fee_token_class_hash → 1` into `contract_class_changes`.

4. The legitimate sequencer BOOTSTRAP declare for the same `class_hash` now hits `dict_update` with `prev_value=0` but finds `prev_value=1`, causing an assertion failure and reverting the legitimate declaration.

5. The fee token class is permanently poisoned. No fee transfers can execute. The network cannot confirm any new transactions. [1](#0-0)
