### Title
`batch_insert` Bypasses Duplicate Key/Hash Guards, Corrupting DataLayer Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces. Supplying a batch that contains a key already present in the tree (or a repeated key within the batch itself) silently writes a second leaf node for that key into the blob, corrupting the tree structure and producing an incorrect Merkle root. Proofs of inclusion derived from the corrupted tree are then invalid, allowing a DataLayer store owner to commit a root that does not faithfully represent the stored data.

### Finding Description

`MerkleBlob::insert` guards against duplicate state before writing:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` only routes the **first two items** through `insert` (and only when the tree currently has ≤ 1 leaf). Every subsequent item — and every item when the tree already has ≥ 2 leaves — is written directly via `insert_entry_to_blob` with no duplicate check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else {
            return Ok(());
        };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // guarded
    }
}

for ((key, value), hash) in keys_values_hashes {          // NO guard
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

When a duplicate key is written, `block_status_cache.key_to_index` (a `HashMap`) silently overwrites the old index with the new one, leaving the original leaf node orphaned in the blob but no longer reachable through the cache. The subtree built from `indexes` is then grafted into the live tree via `insert_subtree_at_key`, producing a root hash that incorporates the duplicate leaf. The `check_integrity` routine will subsequently detect the mismatch between `leaf_count` and `key_to_index_cache_length`, but only if called explicitly — it is not called automatically after `batch_insert`.

The Python binding exposes `batch_insert` directly to callers:

```python
# tests/test_datalayer.py  line 334
merkle_blob.batch_insert(kv_ids, hashes)
```

### Impact Explanation

A DataLayer store owner who calls `batch_insert` with a repeated `KeyId` (or with a `KeyId` already present in the tree) obtains a `MerkleBlob` whose root hash does not correspond to a valid, consistent key-value mapping. Any `ProofOfInclusion` generated from this blob will either prove membership for the wrong value or fail verification against the committed root. This satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`batch_insert` is the primary bulk-load path used in production DataLayer sync (see `tests/test_datalayer.py` lines 334, 520–521). Any caller — including a Python-level DataLayer client — can supply a list containing a repeated key. No privilege is required; the check is simply absent from the fast path.

### Recommendation

Add the same guards that `insert` uses at the top of the unchecked loop in `batch_insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing insert_entry_to_blob logic …
}
```

Alternatively, call `check_integrity` (or a lightweight variant) after every `batch_insert` to catch corruption before the root is committed.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

def h(n):
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate so leaf_count > 1, bypassing the guarded path entirely
blob.insert(KeyId(0), ValueId(0), h(0))
blob.insert(KeyId(1), ValueId(1), h(1))

# batch_insert with a key already in the tree — no error raised
blob.batch_insert(
    [(KeyId(0), ValueId(99))],   # duplicate key 0
    [h(99)],
)

blob.calculate_lazy_hashes()
# The root now encodes key 0 twice with different values.
# A proof of inclusion for key 0 will reference whichever leaf
# the corrupted cache points to, not the authoritative one.
proof = blob.get_proof_of_inclusion(KeyId(0))
print(proof.valid())   # may print True against a root that is structurally invalid
