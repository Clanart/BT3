### Title
Reentrancy in `initTransfer()` Causes Nonce Collision and Permanent Token Freeze - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer()` increments `currentOriginNonce` before external token calls but reads it back from storage *after* those calls when emitting the `InitTransfer` event and calling `initTransferExtension`. A reentrant call during the external token transfer increments `currentOriginNonce` a second time, causing both the outer and inner call to emit `InitTransfer` with the same nonce. The outer call's original nonce is never emitted, permanently freezing the tokens locked by that call in the bridge with no corresponding NEAR-side record.

### Finding Description

In `OmniBridge.initTransfer()`, `currentOriginNonce` is incremented at line 381 before any external call — which is correct CEI intent. However, the incremented value is **never captured into a local variable**. Instead, `currentOriginNonce` is read directly from storage again at lines 418 and 430, after the external token operations complete:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;          // line 381: incremented, but not saved locally
    ...
    IERC20(tokenAddress).safeTransferFrom(   // line 407: external call — reentrancy window
        msg.sender, address(this), amount
    );
    ...
    initTransferExtension(
        msg.sender, tokenAddress,
        currentOriginNonce,           // line 418: reads storage — may be stale after reentry
        ...
    );
    emit BridgeTypes.InitTransfer(
        msg.sender, tokenAddress,
        currentOriginNonce,           // line 430: reads storage — same problem
        ...
    );
}
``` [1](#0-0) 

There is no `nonReentrant` guard on `initTransfer`. [2](#0-1) 

**Execution trace with ERC777 token (has `tokensToSend` hook):**

1. Outer call: `currentOriginNonce` 0 → **1**
2. `safeTransferFrom` fires the ERC777 `tokensToSend` hook on the attacker's contract
3. Reentrant call: `currentOriginNonce` 1 → **2**; reentrant `safeTransferFrom` completes; emits `InitTransfer(nonce=2)` ✓
4. Control returns to outer call; `safeTransferFrom` completes
5. Outer call reads `currentOriginNonce` from storage = **2**; emits `InitTransfer(nonce=2)` ← **collision**

Result: nonce 1 is never emitted; two events carry nonce 2.

The same pattern exists in `initTransfer1155`, where `currentOriginNonce` is incremented at line 448 but read from storage again at line 471 after the `IERC1155.safeTransferFrom` external call. [3](#0-2) 

The CLAUDE.md explicitly states the invariant that is broken: *"Every `originNonce` is incremented atomically. A nonce must never be reusable"* and *"The NEAR side relies solely on these events — any missing or ambiguous field means lost funds or spoofable transfers."* [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of user funds in the bridge.**

- The outer call locks `amount` tokens in the bridge contract via `safeTransferFrom`. No `InitTransfer` event with nonce 1 is ever emitted. The NEAR side never learns about this transfer. Those tokens are permanently frozen with no recovery path.
- Two `InitTransfer` events with the same `(originChain, originNonce=2)` are emitted. The NEAR side uses `TransferId { origin_chain, origin_nonce }` as a unique key. The second event will be rejected as a duplicate, so the reentrant call's transfer is processed once — but the outer call's locked tokens remain frozen forever.
- The `locked_tokens` accounting on the NEAR side becomes permanently inconsistent: tokens are held in the EVM bridge with no corresponding NEAR-side record to unlock them.

This directly matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

**Medium.** The attacker needs a token with a reentrancy hook reachable through `transferFrom`. ERC777 tokens satisfy this natively (the `tokensToSend` hook fires during `transferFrom`). The `else` branch of `initTransfer` (lines 406–412) accepts **any** ERC20-compatible token — there is no allowlist check for native (non-bridge) tokens. An attacker deploys an ERC777 token, calls `initTransfer` with it, and the hook reenters `initTransfer`. No privileged access is required. The attacker does pay for both token transfers but achieves permanent freeze of the outer call's tokens and nonce corruption. [5](#0-4) 

### Recommendation

Capture `currentOriginNonce` into a local variable immediately after incrementing it, and use only the local variable for all subsequent reads within the same call:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce;  // capture locally
    ...
    // external calls here
    ...
    initTransferExtension(msg.sender, tokenAddress, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
}
```

Apply the same fix to `initTransfer1155`. Additionally, consider adding a `ReentrancyGuard` (`nonReentrant`) modifier to both `initTransfer` and `initTransfer1155` as defense-in-depth.

### Proof of Concept

**Setup:** Deploy a malicious ERC777 token `MaliciousToken` whose `tokensToSend` hook calls `OmniBridge.initTransfer(address(maliciousToken), amount, ...)` once (guarded by a flag to prevent infinite recursion).

**Steps:**
1. Attacker approves OmniBridge to spend `MaliciousToken`.
2. Attacker calls `OmniBridge.initTransfer(address(maliciousToken), amount, 0, 0, "attacker.near", "")`.
3. `currentOriginNonce` becomes 1.
4. `safeTransferFrom` fires `tokensToSend` on attacker's contract.
5. Hook calls `initTransfer(address(maliciousToken), amount, 0, 0, "attacker.near", "")` again.
6. `currentOriginNonce` becomes 2. Reentrant `safeTransferFrom` completes. `InitTransfer(nonce=2)` emitted.
7. Outer `safeTransferFrom` completes. Outer call reads `currentOriginNonce=2`. Emits `InitTransfer(nonce=2)`.

**Observable state:**
- Transaction logs contain two `InitTransfer` events both with `originNonce=2`.
- No `InitTransfer` event with `originNonce=1` exists anywhere.
- Bridge holds `2 * amount` of `MaliciousToken`; NEAR side will only process one transfer (nonce 2); `amount` tokens are permanently frozen.
