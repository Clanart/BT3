### Title
`execute_replace_class` Syscall Bypasses Class Existence Constraint, Enabling Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS does not validate that the new class hash corresponds to a declared contract class. Any contract can replace its class with an arbitrary, undeclared felt value, violating the OS-level invariant that every contract's class hash must point to a declared class. Subsequent calls to the affected contract will fail permanently, freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the new class hash directly from the syscall request and writes it into the contract state without any existence check: [1](#0-0) 

The critical lines are:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment explicitly acknowledges the missing validation. The OS commits the new class hash to the Patricia state tree unconditionally.

**Structural analogy to the external report:** In the external report, `SharesCooldown.finalize` takes a special code path that skips the `minimumJrtSrtRatio` guard enforced by `maxWithdrawInner`. Here, `execute_replace_class` takes the only code path available for class replacement and that path has no guard at all — the class existence invariant is simply absent. In both cases a specific OS/protocol function bypasses a critical safety constraint, allowing the system to commit an invalid state.

The class existence invariant is enforced everywhere else in the OS:

- `execute_declare_transaction` uses `dict_update` with `prev_value=0` to ensure a class is declared exactly once. [2](#0-1) 
- `compute_class_commitment` hashes and commits the class tree, which only contains declared classes. [3](#0-2) 

`execute_replace_class` is the sole code path that writes a class hash into a contract's `StateEntry` without going through the class declaration machinery, creating a bypass identical in structure to the reported vulnerability.

---

### Impact Explanation

After `replace_class` is called with an undeclared hash `H`:

1. The contract's `StateEntry.class_hash` is committed to the global state as `H`. [4](#0-3) 
2. Every future call to the contract attempts to look up class `H` in the compiled-class tree; the lookup fails because `H` was never declared.
3. All entry points of the contract become permanently unreachable.
4. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently frozen with no recovery path.

**Impact class:** Critical — permanent freezing of funds.

---

### Likelihood Explanation

The attack surface is broad:

- **Malicious contract developer:** Deploy a contract that accepts user deposits and exposes any function that internally calls `replace_class` with an arbitrary felt. After attracting deposits, trigger the call. No special privilege is required; any account can deploy a contract and invoke its entry points.
- **Buggy upgrade path:** A legitimate upgradeable contract whose upgrade function does not validate the new class hash (e.g., reads it from storage or calldata without checking `contract_class_changes`) can accidentally or maliciously set an undeclared hash, freezing all held funds.
- **Reentrancy / callback abuse:** A contract that invokes an untrusted external callback before completing its own logic could be tricked into calling `replace_class` with an attacker-supplied hash via the callback.

The entry point is fully unprivileged: `execute_replace_class` is reachable by any contract execution, which is reachable by any transaction sender.

---

### Recommendation

Inside `execute_replace_class`, before writing the new `StateEntry`, verify that `class_hash` is present in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);  // 0 means undeclared.
```

This mirrors the guard already applied in `compute_class_commitment` and `execute_declare_transaction`, closing the bypass.

---

### Proof of Concept

1. Declare and deploy contract `Vault` (class `A`) that:
   - Accepts ERC-20 deposits from users.
   - Exposes `freeze()` which calls the `replace_class` syscall with `class_hash = 0xDEAD`.
2. Users deposit funds; `Vault` accumulates a balance.
3. Attacker (or anyone who can call `freeze()`) invokes `freeze()`.
4. The OS executes `execute_replace_class`:
   - `request.class_hash = 0xDEAD` is read from the syscall buffer.
   - No existence check is performed (the TODO line is a no-op).
   - `dict_update` writes `StateEntry(class_hash=0xDEAD, ...)` for `Vault`'s address.
5. The new state root is committed with `Vault.class_hash = 0xDEAD`.
6. Any subsequent `call_contract` or `invoke` targeting `Vault` fails: the OS cannot find a compiled class for `0xDEAD`.
7. All deposited funds are permanently frozen. [5](#0-4)

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
