### Title
Missing Upper-Bound Validation on `max_price_per_unit` Enables Fee-Computation Overflow to Zero, Allowing Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`pack_resource_bounds` validates `max_amount` with a tight upper-bound (`assert_nn_le(..., 2**64 - 1)`) but validates `max_price_per_unit` only with `assert_nn` (non-negative, i.e., `≤ (PRIME-1)/2 ≈ 2^250`). Because `compute_max_possible_fee` performs plain felt arithmetic on these values, an attacker can craft resource-bounds fields whose product sum wraps to exactly `0 mod PRIME`. When `max_fee == 0`, `charge_fee` returns immediately without executing the ERC-20 transfer, so the transaction is executed with zero fee charged.

---

### Finding Description

**Root cause — `pack_resource_bounds` (transaction_hash.cairo, lines 103–107):**

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // ✓ tight bound
    assert_nn(resource_bounds.max_price_per_unit);            // ✗ only ≥ 0; no upper bound
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
``` [1](#0-0) 

`max_price_per_unit` is therefore accepted in the range `[0, (PRIME−1)/2]`, i.e., up to ≈ 2^250.

**Overflow path — `compute_max_possible_fee` (transaction_impls.cairo, lines 87–101):**

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit +
           l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip) +
           l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
``` [2](#0-1) 

All arithmetic is felt arithmetic (mod PRIME). With `max_amount ≤ 2^64−1` and `max_price_per_unit ≤ (PRIME−1)/2`, the product of a single term can reach `(2^64−1)·(PRIME−1)/2 ≈ 2^314`, which wraps around PRIME many times. An attacker can choose values so the three-term sum is exactly `0 mod PRIME`.

**Concrete example:**

| Field | Value |
|---|---|
| `l1_gas` (max_amount) | `2` |
| `l1_price` (max_price_per_unit) | `(PRIME − 1) / 2` |
| `l2_gas` | `1` |
| `l2_price` | `1` |
| `tip` | `0` |
| `l1_data_gas` | `0` |

Computation: `2 · (PRIME−1)/2 + 1·1 + 0 = (PRIME−1) + 1 = PRIME ≡ 0 (mod PRIME)`.

All individual field checks pass: `l1_gas = 2 ≤ 2^64−1` ✓; `l1_price = (PRIME−1)/2 ≥ 0` ✓ (assert_nn passes); `tip = 0 ≤ 2^64−1` ✓.

**Early-exit in `charge_fee` (transaction_impls.cairo, lines 121–125):**

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

When `max_fee == 0`, the function returns before the ERC-20 transfer is executed. The sequencer receives no fee. The OS proof is still valid.

---

### Impact Explanation

- **Direct loss of funds (Critical):** The sequencer's fee revenue is the economic mechanism that compensates block producers and deters spam. An attacker who can force `max_fee = 0` at the OS level causes the sequencer to execute real computation (state changes, storage writes, syscalls) without receiving any payment. Repeated exploitation drains sequencer revenue and breaks the fee market.
- **Network halt (High):** Because the OS accepts such transactions as fully valid (the proof verifies), an attacker can flood the network with zero-cost transactions. With no fee barrier, the mempool and block-building pipeline can be saturated, preventing legitimate transactions from being confirmed — a total network shutdown.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft the specific `max_price_per_unit` values. The arithmetic is straightforward modular arithmetic over a known prime. The OS enforces no upper bound on `max_price_per_unit`, so the crafted transaction passes all OS-level checks. If the sequencer's mempool uses the same overflow-prone `compute_max_possible_fee` logic (or any logic that does not enforce `max_price_per_unit ≤ 2^128`), the transaction will be accepted and included in a block. The resulting proof is valid and will be accepted by the L1 verifier.

---

### Recommendation

In `pack_resource_bounds`, replace the loose `assert_nn` with a tight upper-bound check matching the semantic width of the field (128 bits, consistent with how `max_amount` is bounded to 64 bits):

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
-   assert_nn(resource_bounds.max_price_per_unit);
+   assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
``` [1](#0-0) 

With `max_price_per_unit ≤ 2^128−1` and `max_amount ≤ 2^64−1`, the maximum value of any single product term is `(2^64−1)·(2^128−1) < 2^192 ≪ PRIME`, so no overflow is possible and `compute_max_possible_fee` is sound.

---

### Proof of Concept

1. Construct a v3 Invoke transaction with the following resource bounds:
   - L1 gas: `max_amount = 2`, `max_price_per_unit = (PRIME − 1) / 2`
   - L2 gas: `max_amount = 1`, `max_price_per_unit = 1`
   - L1 data gas: `max_amount = 0`, `max_price_per_unit = 0`
   - `tip = 0`
