### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Breaks Bridge Collateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` locks external ERC20 tokens by calling `safeTransferFrom(msg.sender, address(this), amount)` and then unconditionally emits `InitTransfer` with the caller-supplied `amount`. For fee-on-transfer (deflationary) tokens, the bridge receives fewer tokens than `amount`, but the cross-chain event claims the full `amount` was locked. The NEAR side uses this event as the sole source of truth to mint or release tokens, creating unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.sol::initTransfer`, the "lock" branch for plain ERC20 tokens (not bridge tokens, not custom minters) is:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Immediately after, the event is emitted with the caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← user-supplied, not actual received balance
    fee,
    nativeFee,
    recipient,
    message
);
```

No balance snapshot is taken before or after the transfer to verify the actual amount received. For a fee-on-transfer token (one that burns or redirects a percentage on every `transfer`/`transferFrom` call), the bridge's actual token balance increases by `amount - deflation_fee`, while the emitted event asserts `amount` was locked.

The NEAR side reads `InitTransfer` events as the authoritative proof of locked collateral and mints or releases exactly `amount` tokens to the recipient. There is no on-chain reconciliation between the event value and the bridge's actual ERC20 balance.

The same structural issue exists in the `finTransfer` unlock path:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
```

Here the bridge attempts to release `payload.amount` but the recipient receives `payload.amount - deflation_fee`; the bridge's balance is drained by the full `payload.amount` (including the burned portion), accelerating undercollateralization.

The StarkNet bridge has the identical pattern in `init_transfer`:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
```

No post-transfer balance check is performed before the `InitTransfer` event is emitted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token creates a deficit: the bridge holds `amount - deflation_fee` tokens but the NEAR side mints `amount` tokens. The deficit compounds with each transfer. Eventually the bridge's ERC20 balance is insufficient to honor all outstanding redemptions. Users who attempt to bridge back from NEAR to EVM will find the bridge unable to release their tokens, resulting in **permanent freezing of user funds** and **broken bridge collateralization** — matching the High impact category: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

The bridge does not whitelist or restrict which ERC20 tokens can be used in `initTransfer`. Any user can call `initTransfer` with any ERC20 address. Fee-on-transfer tokens exist on mainnet (e.g., tokens with redistribution mechanics, reflection tokens, tokens with protocol fees). A user can trigger this path without any privileged access, either accidentally or deliberately to drain the bridge's collateral over time.

---

### Recommendation

Use a balance-before / balance-after pattern to determine the actual received amount, and use that value — not the caller-supplied `amount` — in the emitted event and in all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived in the InitTransfer event and fee validation
```

Apply the same fix to `finTransfer` (measure actual transferred-out amount) and to the StarkNet `init_transfer`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported, and add an on-chain guard (e.g., a token allowlist or a post-transfer balance assertion that reverts if `actualReceived != amount`).

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FeeToken` that burns 2% on every `transferFrom` call.
2. User mints 10,000 `FeeToken` and approves `OmniBridge`.
3. User calls `OmniBridge.initTransfer(FeeToken, 10_000, 0, 0, "alice.near", "")`.
4. `safeTransferFrom` executes: bridge receives `9,800` tokens (200 burned).
5. `InitTransfer` event emits `amount = 10_000`.
6. NEAR relayer observes the event and mints `10,000` wrapped `FeeToken` to `alice.near`.
7. Bridge is now undercollateralized by 200 tokens.
8. Repeat 49 more times: bridge holds `490,000` tokens but has issued `500,000` NEAR-side tokens.
9. The 50th user to bridge back from NEAR to EVM cannot be paid — their funds are permanently frozen in the bridge. [5](#0-4) [6](#0-5)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
