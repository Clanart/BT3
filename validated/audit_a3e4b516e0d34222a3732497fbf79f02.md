Audit Report

## Title
Unrestricted `sweepToken` and `unwrapWETH9` Allow Arbitrary Recipient Drain of Router-Held Tokens — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.sweepToken` and `unwrapWETH9` are `public payable` with no caller restriction and accept a fully attacker-controlled `recipient` address. Any tokens or WETH that land on the router or liquidity adder between transactions — via multi-hop intermediate accumulation or surplus native ETH from `pay` — can be redirected to an arbitrary address by any unprivileged caller, resulting in complete loss of user principal.

## Finding Description
Both functions are confirmed in the production code with no access control:

`unwrapWETH9` (lines 37–45) withdraws the entire WETH balance and forwards it to a caller-supplied `recipient` with no check that `recipient == msg.sender` or that the caller is the rightful owner:
```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
```

`sweepToken` (lines 48–55) transfers the entire ERC-20 balance to a caller-supplied `recipient` with no restriction:
```solidity
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
```

Tokens land on the router via two confirmed production paths:

**Path 1 — Multi-hop exact-input intermediate accumulation:** In `exactInput` (line 106), every intermediate hop sends output to `address(this)`. If the user sets `params.recipient = address(router)` on the final hop (the standard pre-unwrap pattern), WETH sits on the router after the transaction completes. A second transaction calling `unwrapWETH9(0, attacker)` drains it entirely.

**Path 2 — Surplus native ETH from `pay`:** In `pay` (lines 74–77), when `nativeBalance >= value`, exactly `value` is wrapped and transferred; the remainder stays as ETH on the router. `refundETH` (lines 58–63) sends to `msg.sender`, so an attacker calling it captures the stranded ETH.

Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` inherit `PeripheryPayments` and expose this surface. No existing guard prevents a third party from specifying themselves as `recipient`.

## Impact Explanation
Direct, complete loss of user principal. Any tokens or ETH residing on the router or liquidity adder between transactions are fully claimable by an unprivileged caller who supplies themselves as `recipient`. There is no recovery path once the drain occurs. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
The two-transaction pattern (swap → unwrap, or swap → refund) is the natural interaction model for integrators, wallets, and scripts that do not construct a `multicall` payload. A mempool-watching bot requires no special privilege or setup: it observes any transaction that leaves value on the router, then immediately calls `sweepToken(token, 0, attacker)` or `unwrapWETH9(0, attacker)` in the next block. The attack is repeatable and fully automated.

## Recommendation
Restrict `sweepToken` and `unwrapWETH9` so that `recipient` must equal `msg.sender`, eliminating the ability for a third party to redirect funds. Alternatively, remove the `recipient` parameter entirely and always send to `msg.sender`, matching the safer design already used by `refundETH`. This ensures only the caller who initiates the sweep or unwrap can receive the funds.

## Proof of Concept
1. Alice calls `router.exactInput({..., recipient: address(router), ...})` — pool sends 0.5 WETH to `address(router)`.
2. Alice intends to call `router.unwrapWETH9(0, alice)` in a second transaction.
3. Bob observes Alice's first transaction confirm. Bob immediately calls `router.unwrapWETH9(0, bob)`.
4. Router's WETH balance (0.5 WETH) is withdrawn and sent to Bob as ETH.
5. Alice's second transaction reverts with `InsufficientWETH(0, 0)`. Bob has stolen 0.5 WETH. Alice's input tokens are gone.

The same attack applies to any ERC-20 via `sweepToken(token, 0, attacker)` and to stranded native ETH via `refundETH()`.
