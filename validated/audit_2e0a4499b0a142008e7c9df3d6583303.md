### Title
`ASSERT_SECONDS_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` Time-Lock Bypass and Incorrect Rejection via Integer Wrapping in `check_time_locks()` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is invoked with `nowrap=false` (the pre-soft-fork / legacy consensus path), relative time-lock comparisons use `wrapping_add` instead of saturating arithmetic. An unprivileged spender can craft a coin whose `ASSERT_SECONDS_RELATIVE` condition carries a value large enough to wrap the 64-bit deadline below the current timestamp, making the time lock appear already satisfied and allowing the coin to be spent immediately. The symmetric `ASSERT_BEFORE_SECONDS_RELATIVE` case wraps in the opposite direction, causing a coin that should be spendable to be permanently rejected.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` implements two arithmetic modes controlled by the `nowrap: bool` parameter:

```rust
// lines 70-78
if let Some(seconds_relative) = spend.seconds_relative {
    if nowrap {
        if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
            return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
        }
    } else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
        return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
    }
}
``` [1](#0-0) 

And for `ASSERT_BEFORE_SECONDS_RELATIVE`:

```rust
// lines 100-111
if let Some(before_seconds_relative) = spend.before_seconds_relative {

        if timestamp >= unspent.timestamp.saturating_add(before_seconds_relative) {

``` [2](#0-1) 

The same wrapping pattern applies to `ASSERT_HEIGHT_RELATIVE` / `ASSERT_BEFORE_HEIGHT_RELATIVE` using `u32::wrapping_add`. [3](#0-2) 

The `seconds_relative` field is a `u64` parsed from the CLVM condition `ASSERT_SECONDS_RELATIVE`. The condition parser accepts any value up to `u64::MAX` (8-byte unsigned integer): [4](#0-3) 

The `SpendBundleConditions` struct stores `seconds_relative` as `u64`: [5](#0-4) 

**Attack path — `ASSERT_SECONDS_RELATIVE` bypass:**

Let `T` = `unspent.timestamp` (coin confirmation timestamp), `s` = attacker-chosen `seconds_relative`.

With `nowrap=false`:
```
deadline = T.wrapping_add(s)
check fails (coin NOT spendable) iff: current_timestamp < deadline
```

If the attacker sets `s = u64::MAX`, then `deadline = T.wrapping_add(u64::MAX) = T - 1`.

For any `current_timestamp >= T - 1` (i.e., any time after coin creation), the check `current_timestamp < T - 1` is **false**, so the error is **not** raised and the coin is immediately spendable — regardless of the intended `u64::MAX`-second lock.

More generally, the attacker picks `s = u64::MAX - T + k + 1` for any small `k`, causing `deadline = k`, which is always less than any realistic timestamp.

**Attack path — `ASSERT_BEFORE_SECONDS_RELATIVE` incorrect rejection:**

With `s = u64::MAX` and `T = 1000`, `deadline = 999`. The check `current_timestamp >= 999` is immediately true, so the coin is **permanently rejected** even though the intended deadline is `T + u64::MAX` (effectively never).

The test suite explicitly documents both divergences: [6](#0-5) 

---

### Impact Explanation

**`ASSERT_SECONDS_RELATIVE` bypass (High):** A coin whose puzzle enforces a relative time lock via `ASSERT_SECONDS_RELATIVE` can be spent immediately by choosing a `seconds_relative` value that wraps the 64-bit deadline below the current timestamp. In multi-party protocols (payment channels, escrow, vaults, clawback constructs) where one party's refund path is protected by a relative time lock, the counterparty can bypass the lock and claim funds before the agreed window expires. This matches the allowed impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

**`ASSERT_BEFORE_SECONDS_RELATIVE` DoS (High):** A coin with a large `before_seconds_relative` value is permanently unspendable under `nowrap=false`, corrupting the coin's intended lifecycle.

---

### Likelihood Explanation

The `nowrap=false` path is the active consensus path for all blocks processed before the soft fork that introduced `nowrap=true` activates. The Python full node passes `nowrap` based on block height; until activation, every block uses the wrapping path. The attacker-controlled entry is a standard CLVM condition (`ASSERT_SECONDS_RELATIVE`) accepted in any spend bundle — no privileged access is required. The condition value `u64::MAX` (or any wrap-inducing value) is a valid 8-byte unsigned integer accepted by `sanitize_uint`. [7](#0-6) 

---

### Recommendation

1. Remove the `nowrap=false` / `wrapping_add` branch entirely from `check_time_locks`. The `nowrap=true` / `saturating_add` path is the correct semantic for all relative time-lock conditions: a deadline that would overflow `u64` is treated as "never" (for `ASSERT_SECONDS_RELATIVE`) or "always valid" (for `ASSERT_BEFORE_SECONDS_RELATIVE`), which matches user intent.
2. If backward compatibility with pre-fork blocks is required, gate the wrapping path strictly on a block-height check inside Rust rather than relying on the Python caller to pass the correct boolean.
3. Apply the same fix to the `u32` height-relative checks (`confirmed_block_index.wrapping_add(height_relative)` / `wrapping_add(before_height_relative)`).

---

### Proof of Concept

```
Coin creation timestamp T = 1_000_000
seconds_relative = 0xFFFFFFFFFFFFFFFF  (u64::MAX, valid 8-byte CLVM atom)

wrapping_add(1_000_000, u64::MAX) = 999_999

current_timestamp = 1_000_001
check (nowrap=false): 1_000_001 < 999_999  →  false  →  no error  →  coin IS spendable

Expected (nowrap=true): saturating_add(1_000_000, u64::MAX) = u64::MAX
check: 1_000_001 < u64::MAX  →  true  →  AssertSecondsRelativeFailed  →  coin NOT spendable
```

A spend bundle containing a coin whose puzzle outputs `(ASSERT_SECONDS_RELATIVE . 0xFFFFFFFFFFFFFFFF)` passes `check_time_locks` immediately under `nowrap=false`, bypassing the intended `u64::MAX`-second relative time lock. [1](#0-0) [2](#0-1)
