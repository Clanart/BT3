### Title
`batch_insert` Bypasses Duplicate-Key Guard Present in `insert`, Enabling Silent Merkle Tree Root Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` inserts leaf nodes directly into the raw blob via the low-level `insert_entry_to_blob` helper, completely skipping the duplicate-key check that `MerkleBlob::insert` enforces through `block_status_cache`. A caller (including Python code via the exposed `batch_insert` binding) that supplies a batch containing a repeated `KeyId` silently produces a blob with two leaves sharing the same key, yielding a corrupted Merkle root and a cache/blob state disagreement that invalidates all subsequent proof-of-inclusion operations.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` (lines 570–656) takes a completely different code path for the bulk of its work. When the tree already has more than one leaf, **every** item in the supplied batch is written through `insert_entry_to_blob` without any call to `block_status_cache.contains_key`:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... LeafNode { hash, key, value } ... };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
```

Even in the special case where the tree has ≤ 1 leaves, only the **last two** items (popped from the end of the vector) are routed through `insert()` with its guard; all earlier items in the batch still bypass the check.

`BlockStatusCache::add_leaf` (lines 188–192) inserts into `key_to_index` with a plain `HashMap::insert`, which silently overwrites a prior entry for the same key. Consequently, after a batch containing a duplicate `KeyId`:

1. The **blob** contains two distinct leaf blocks with the same `KeyId`.
2. The **cache** (`key_to_index`) retains only the last-written index for that key.
3. `get_root_hash` / `calculate_lazy_hashes` traverse the blob and incorporate **both** leaves into the Merkle root computation.
4. `get_proof_of_inclusion` consults only the cache and therefore constructs a proof anchored to **one** of the two leaves — a proof that verifies against a root that was computed from a different tree shape.
5. `check_integrity` (lines 861–874) detects the leaf-count mismatch between blob and cache and returns an error, but it is never called automatically after `batch_insert`.

The Python binding exposes `batch_insert` directly:

```python
def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...
```

No Python-level deduplication is enforced before the call reaches Rust.
