### Title
Non-EVM-Native Token Balances Not Burned on SELFDESTRUCT — Orphaned Cosmos Bank Funds Recoverable via CREATE2 Redeploy - (File: x/evm/statedb/statedb.go)

### Summary
`StateDB.Commit()` burns only the EVM-denom balance when processing a self-destructed account. Non-EVM-native tokens (IBC, CosmWasm bridge, etc.) held by the destroyed address at the Cosmos bank layer are never drained. Because the Cosmos bank stores balances by raw address bytes — independent of the auth account — those tokens survive the `DeleteAccount` call and remain accessible to whoever recreates the contract at the same address via CREATE2.

### Finding Description
`Keeper.DeleteAccount` removes the auth account record and EVM storage but explicitly does not touch the bank balance: [1](#0-0) 

The fix introduced in PR #896 added a pre-`DeleteAccount` drain inside `StateDB.Commit()`, but it is scoped exclusively to `s.evmDenom`: [2](#0-1) 

The code comment at that site explicitly acknowledges the gap:

> "Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm bridge) held by the destroyed address are not drained and may remain as orphaned bank balances." [3](#0-2) 

After `writeCache()` commits the `DeleteAccount` result, the auth account is gone but every non-EVM-denom bank balance for that address persists in the Cosmos bank KVStore. When a new contract is deployed at the same address (deterministic via CREATE2), a fresh auth account is created at the identical raw bytes, and the bank module immediately associates the surviving balances with the new account. [4](#0-3) 

### Impact Explanation
An attacker who controls a CREATE2 factory can:
1. Deploy a child contract at a salt-predictable address.
2. Induce third parties to send IBC or other Cosmos-native tokens to the contract's Cosmos address (`sdk.AccAddress(contractAddr.Bytes())`).
3. Call `SELFDESTRUCT` on the child (valid under EIP-6780 when created in the same tx, or unconditionally on pre-Cancun chains). The EVM-denom balance is burned; IBC balances are not.
4. Redeploy the child at the same CREATE2 address in a subsequent transaction.
5. The new contract's Cosmos address is identical; the bank module returns the previously orphaned IBC balances to it. The attacker drains them.

This constitutes unauthorized theft of Cosmos bank funds through Ethermint EVM transaction execution, matching the Critical impact tier.

### Likelihood Explanation
- IBC tokens are routinely sent to EVM contract addresses on Cosmos EVM chains (e.g., via IBC transfer to the bech32 form of the contract's address, or via a Cosmos SDK `MsgSend`).
- CREATE2 address prediction is trivial and fully unprivileged.
- The attacker only needs to control the factory contract; no validator collusion or governance action is required.
- The entire exploit fits in two ordinary EVM transactions.

### Recommendation
In `StateDB.Commit()`, after burning the EVM-denom balance, iterate over **all** bank module denominations held by the destroyed address and burn each one before calling `DeleteAccount`. The Cosmos SDK `bankKeeper.GetAllBalances(ctx, cosmosAddr)` returns the full multi-denom balance; each coin can be burned via `SendCoinsFromAccountToModule` + `BurnCoins`. This mirrors the complete accounting cleanup that `removeToken` in the external report was supposed to perform before removing tracked state.

### Proof of Concept
```
// Tx 1 – single transaction via SelfDestructExploitFactory pattern
factory.attackInOneTx(salt, {value: evmAmount})
// → child deployed, SELFDESTRUCT called, EVM denom burned, IBC balance orphaned

// Off-chain: send IBC tokens to child's Cosmos address before Tx 1,
// or interleave with a Cosmos MsgSend in the same block.

// Tx 2
factory.redeployChild(salt)
// → child recreated at same address; bank returns orphaned IBC balance
child.drain()  // attacker recovers IBC tokens
```

The existing integration test `SelfDestructExploitFactory.attackInOneTx` / `redeployChild` in `tests/integration_tests/hardhat/contracts/SelfDestructExploit.sol` already models the CREATE2 redeploy path; extending it to include a pre-destruction IBC `MsgSend` to the child address would demonstrate the theft directly. func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)
