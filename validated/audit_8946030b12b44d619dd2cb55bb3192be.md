### Title
Nonce Replay via Batched `MsgEthereumTx` with `unsafeUnOrderedTx=true` — (`ante/eth.go`)

### Summary
When `UnsafeUnorderedTx=true` is configured, `CheckAndSetEthSenderNonce` skips all nonce verification. An unprivileged sender can batch multiple `MsgEthereumTx` messages with the **same nonce** in a single Cosmos tx. All messages pass the ante handler and all execute in the EVM, violating the invariant that each nonce is used exactly once per account.

### Finding Description
`CheckAndSetEthSenderNonce` iterates over all messages in the batch. When `unsafeUnOrderedTx=true`, the entire nonce-check block is skipped unconditionally — including any check for duplicate nonces within the same batch. The `NewCachedAccountGetter` returns the same in-memory account object for the same address, so each iteration reads the already-incremented sequence and increments it again. After the ante handler, the account sequence is `N + len(msgs)` regardless of what nonces the messages carry.

The EVM execution layer (`ApplyMessageWithConfig`) calls `evm.Call` directly, not `core.ApplyMessage`, so go-ethereum's `preCheck` nonce validation is never invoked. Both messages execute with `msg.Nonce=0`.

### Impact Explanation
An attacker who controls an account can submit a single Cosmos tx containing two (or more) `MsgEthereumTx` messages with identical nonces. Both execute in the EVM. For CALL transactions this enables double-execution of any state-changing operation (double-claiming rewards, double-minting, double-transferring). The nonce uniqueness invariant — the primary replay-protection mechanism in Ethereum — is broken for every sender on chains where this flag is enabled.

### Likelihood Explanation
Requires `UnsafeUnorderedTx=true` to be set by the operator. This is a production `HandlerOptions` field, not a test-only flag. Any chain that enables it is fully exposed. The attacker needs only a funded account and the ability to submit a Cosmos tx with a batched `MsgEthereumTx` payload — no special privileges required.

### Recommendation
Even when `unsafeUnOrderedTx=true`, the loop in `CheckAndSetEthSenderNonce` must reject duplicate nonces within the same batch. A simple per-batch seen-nonce set (keyed by `(address, nonce)`) should be checked before the sequence increment, returning `ErrInvalidSequence` on collision. This preserves the unordered-across-txs semantics while preventing intra-batch replay.

### Proof of Concept
1. Configure chain with `UnsafeUnorderedTx=true`.
2. Fund sender account `A` with sequence=0.
3. Construct a Cosmos tx with two `MsgEthereumTx` messages, both from `A`, both with `nonce=0`, both calling a contract that emits an event or transfers value.
4. Broadcast the tx.
5. Observe: both messages execute (two events emitted / value transferred twice); sender sequence is 2.
6. Assert: the nonce `0` was used twice, violating the replay-protection invariant.

func NewCachedAccountGetter(ctx sdk.Context, ak evmtypes.AccountKeeper) AccountGetter {
	accounts := make(map[string]sdk.AccountI, 1)
	return func(addr sdk.AccAddress) sdk.AccountI {
		acc := accounts[string(addr)]
		if acc == nil {
			acc = ak.GetAccount(ctx, addr)
			if acc == nil {
				// we create a new account in memory if it doesn't exist,
				// which is only set to store when updated.
				acc = ak.NewAccountWithAddress(ctx, addr)
			}
			accounts[string(addr)] = acc
		}
		return acc
	}
}
