Let me verify the exact call path and check if there's any nil guard before the `v.Sign()` call.

### Title
Nil-pointer panic in `NewRPCTransactionFromTx` via EIP-2930 AccessListTx with V=0 yParity — (`rpc/types/utils.go`, `x/evm/types/access_list_tx.go`, `x/evm/types/tx_data.go`)

---

### Summary

An EIP-2930 AccessListTx signed with yParity V=0 (a valid even-parity signature) triggers a nil-pointer dereference panic in the public JSON-RPC layer. The root cause is a lossy round-trip: `SetSignatureValues` stores `big.NewInt(0).Bytes()` as an empty byte slice `[]byte{}`, and `rawSignatureValues` reconstructs that as `nil` instead of `big.NewInt(0)`. When the RPC layer later calls `v.Sign()` on the nil `*big.Int`, the process panics.

---

### Finding Description

**Step 1 — V=0 stored as empty bytes.**

`SetSignatureValues` stores `v.Bytes()` unconditionally when `v != nil`:

```go
if v != nil {
    tx.V = v.Bytes()   // big.NewInt(0).Bytes() == []byte{}
}
``` [1](#0-0) 

For `v = big.NewInt(0)`, `v.Bytes()` returns `[]byte{}` (the canonical big-endian encoding of zero is the empty byte string). So `tx.V` is set to an empty slice, not `nil`.

**Step 2 — Empty bytes reconstructed as nil.**

`rawSignatureValues` only creates a `*big.Int` when the byte slice is non-empty:

```go
func rawSignatureValues(vBz, rBz, sBz []byte) (v, r, s *big.Int) {
    if len(vBz) > 0 {
        v = new(big.Int).SetBytes(vBz)
    }
    ...
    return v, r, s
}
``` [2](#0-1) 

When `vBz = []byte{}`, `len(vBz) == 0`, so `v` is never assigned and is returned as `nil`. The original `big.NewInt(0)` is irreversibly lost.

**Step 3 — `AsEthereumData` propagates nil V.**

`AsEthereumData` calls `GetRawSignatureValues()` → `rawSignatureValues` and places the nil `v` directly into the reconstructed `ethtypes.AccessListTx`:

```go
v, r, s := tx.GetRawSignatureValues()
return &ethtypes.AccessListTx{
    ...
    V: v,   // nil
    ...
}
``` [3](#0-2) 

**Step 4 — Nil pointer dereference in `NewRPCTransactionFromTx`.**

`NewRPCTransaction` calls `msg.AsTransaction()` (which internally calls `AsEthereumData()`), then passes the resulting `*ethtypes.Transaction` to `NewRPCTransactionFromTx`. Inside that function, `tx.RawSignatureValues()` returns the nil `v` from the go-ethereum `AccessListTx`, and the very next line dereferences it unconditionally:

```go
v, r, s := tx.RawSignatureValues()
...
yparity := hexutil.Uint64(v.Sign()) //#nosec G115
``` [4](#0-3) 

`v` is nil; `v.Sign()` is a nil-pointer dereference → **panic**.

---

### Impact Explanation

Any public JSON-RPC call that returns full transaction details for a committed EIP-2930 tx with V=0 will panic the RPC server process. Affected endpoints include `eth_getTransactionByHash`, `eth_getBlockByHash`/`eth_getBlockByNumber` with `fullTx=true`, `eth_getTransactionReceipt` (via the indexer path), and `eth_simulateV1` when `returnFullTransactions=true`. A single such transaction permanently breaks those endpoints for that transaction, and if the panic is unrecovered it can crash the RPC server entirely.

This matches the allowed High impact: **"Public JSON-RPC, gRPC, simulation, tracing, receipt/log, or indexer path feeds incorrect consensus-critical data into transaction execution or exposes a reachable route to the impacts above."**

---

### Likelihood Explanation

V=0 is a fully valid yParity value for EIP-2930 (Berlin+) transactions — roughly half of all EIP-2930 signatures will have V=0. No special privileges are required; any user submitting an EIP-2930 transaction whose signature happens to have even parity triggers this. The bug is deterministic and reproducible.

---

### Recommendation

Fix `SetSignatureValues` to preserve the zero value explicitly, or fix `rawSignatureValues` to return `new(big.Int)` (i.e., `big.NewInt(0)`) for an empty byte slice instead of nil. The cleanest fix is in `rawSignatureValues`:

```go
func rawSignatureValues(vBz, rBz, sBz []byte) (v, r, s *big.Int) {
    v = new(big.Int).SetBytes(vBz)   // SetBytes([]byte{}) == 0, not nil
    r = new(big.Int).SetBytes(rBz)
    s = new(big.Int).SetBytes(sBz)
    return v, r, s
}
```

Alternatively, add a nil guard before `v.Sign()` in `NewRPCTransactionFromTx` as a defence-in-depth measure.

### Proof of Concept

```go
// Construct a go-ethereum AccessListTx with V=0 (valid even-parity yParity)
inner := &ethtypes.AccessListTx{
    ChainID: big.NewInt(1),
    Nonce:   0,
    V:       big.NewInt(0),   // valid yParity=0
    R:       big.NewInt(1),
    S:       big.NewInt(1),
}
ethTx := ethtypes.NewTx(inner)

// Round-trip through Ethermint proto
txData, _ := evmtypes.NewTxDataFromTx(ethTx)   // calls newAccessListTx -> SetSignatureValues(v=0)
altx := txData.(*evmtypes.AccessListTx)
assert(len(altx.V) == 0)                        // V stored as []byte{}

v, _, _ := altx.GetRawSignatureValues()
assert(v == nil)                                // rawSignatureValues returns nil for []byte{}

// Reconstruct go-ethereum tx
reconstructed := ethtypes.NewTx(altx.AsEthereumData())
rv, _, _ := reconstructed.RawSignatureValues()
assert(rv == nil)                               // nil V in reconstructed tx

// Trigger the panic
_ = rv.Sign()                                   // nil pointer dereference PANIC
```
