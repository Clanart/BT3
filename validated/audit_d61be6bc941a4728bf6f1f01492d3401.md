### Title
Unconstrained `is_reverted` Hint Allows Prover to Arbitrarily Skip or Force Transaction Execution - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted` in `execution_constraints.cairo` is a complete no-op: it accepts the `is_reverted` parameter but performs zero validation. The `is_reverted` flag is populated exclusively by a prover-controlled hint (`%{ IsReverted %}`), and because `check_is_reverted` adds no Cairo constraints, the value is entirely unconstrained in the proof system. A malicious sequencer/prover can set `is_reverted` to any value and still produce a valid proof, allowing them to skip the execute phase of invoke transactions (while still charging fees) or suppress L1-handler execution (leaving L1→L2 messages unconsumed and funds frozen).

---

### Finding Description

**Root cause — `check_is_reverted` is a no-op:**

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

The function signature accepts `is_reverted` but immediately returns, adding no assertion, range-check, or any other Cairo constraint on the value.

**How `is_reverted` is set and used in `execute_invoke_function_transaction`:**

```cairo
local is_reverted;
%{ IsReverted %}          // hint: prover writes any felt here
check_is_reverted(is_reverted);   // no-op — no constraint added
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // execute phase is entirely skipped
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}
// Fee is charged regardless of the branch taken.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

Because `check_is_reverted` is a no-op, the Cairo verifier cannot distinguish a proof where `is_reverted = 0` from one where `is_reverted = 1`. Both satisfy all constraints.

**L1-handler transactions: `check_is_reverted` is not even called:**

```cairo
local is_reverted;
%{ IsReverted %}
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();          // returns before consume_l1_to_l2_message
}
...
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [3](#0-2) 

For L1 handlers, `is_reverted` is set by a hint and used directly with no constraint at all — not even the no-op call is present.

**Analogy to the original report:** Just as `addLiquidity` in VaderRouter accepted `amountAMin`/`amountBMin` but never enforced them, `check_is_reverted` accepts `is_reverted` but never enforces it. In both cases, a critical guard parameter is silently ignored, enabling manipulation of the protocol's execution path.

---

### Impact Explanation

**Attack vector A — Invoke transaction, prover sets `is_reverted = 1` (TRUE) for a valid transaction:**

1. User submits a valid invoke transaction (e.g., ERC-20 transfer).
2. `run_validate` succeeds; gas is consumed.
3. Prover writes `is_reverted = 1` via the `%{ IsReverted %}` hint.
4. `check_is_reverted` does nothing; the proof is valid.
5. The `else` branch is taken: `__execute__` is skipped entirely.
6. `charge_fee` still runs and deducts fees from the user.
7. The user's intended state change (e.g., token transfer) never occurs.

**Result:** Fees are permanently deducted with no corresponding execution — **direct loss of funds**.

**Attack vector B — L1-handler transaction, prover sets `is_reverted = 1` for a valid L1→L2 message:**

1. User deposits tokens on L1, triggering an L1→L2 message.
2. The sequencer includes the L1 handler in a block.
3. Prover writes `is_reverted = 1`; the function returns before `consume_l1_to_l2_message`.
4. The OS output does not record the message as consumed.
5. The proof is valid (no constraint on `is_reverted`).
6. The L1 verifier accepts the proof; the L1 message remains pending indefinitely.
7. The user's L2 balance is never credited; funds are locked on L1.

**Result:** User's bridged funds are inaccessible on both L1 (message pending, not yet cancellable) and L2 (balance not updated) — **permanent freezing of funds**.

Both impacts are within the allowed StarkNet protocol impact scope.

---

### Likelihood Explanation

Exploiting this requires the sequencer to act as a malicious prover. In StarkNet's current architecture the sequencer is centralized, but the entire purpose of the Cairo proof system is to make sequencer behavior verifiable and trustless — the L1 verifier is supposed to reject any proof that does not faithfully represent execution. This soundness bug breaks that guarantee: a malicious sequencer can produce a proof that the L1 verifier accepts, yet which misrepresents whether a transaction was reverted. The likelihood is therefore bounded by the trustworthiness of the sequencer operator, but the protocol-level guarantee is completely absent.

---

### Recommendation

`check_is_reverted` must add a concrete Cairo constraint that ties `is_reverted` to the actual outcome of the entry-point execution. The standard pattern is to read the failure flag directly from the `EntryPointReturnValues` struct (which is already constrained by the proof) and assert equality:

```cairo
func check_is_reverted(is_reverted: felt) {
    // `is_reverted` must equal the failure_flag returned by the entry point.
    // The failure_flag is already part of the constrained Cairo trace.
    assert is_reverted = is_reverted * is_reverted;  // enforce boolean
    // ... assert against the actual entry-point return value
}
```

More concretely, the `is_reverted` value should be derived directly from `entry_point_return_values.failure_flag` (already constrained in `execute_entry_point`) rather than from an independent hint, or the hint value must be asserted equal to the constrained failure flag. The same fix must be applied to `execute_l1_handler_transaction`, which currently uses the hint with no call to `check_is_reverted` at all.

---

### Proof of Concept

1. Sequencer receives a valid user invoke transaction (e.g., `transfer(recipient, amount)`).
2. Sequencer runs the OS Cairo program. During `execute_invoke_function_transaction`, after `run_validate` succeeds, the sequencer's hint provider writes `is_reverted = 1` for `%{ IsReverted %}`.
3. `check_is_reverted(1)` is called — returns immediately, no constraint added.
4. The `else` branch executes: `__execute__` is skipped; `charge_fee` runs and deducts fees.
5. The sequencer generates a STARK proof. The verifier checks all Cairo constraints — none of them involve `is_reverted`, so the proof is valid.
6. The proof is submitted to L1; the L1 verifier accepts it.
7. On-chain state reflects: nonce incremented, fee deducted, but no token transfer occurred.
8. The user has lost the fee with no corresponding execution.

The same steps apply to L1 handlers with `is_reverted = 1`, replacing step 7 with: L1 message not consumed, user's bridged deposit inaccessible.
