### Title
Deterministic Token Account Pre-Creation Blocks Wrapped Token Deployment Permanently — (File: `near/token-deployer/src/lib.rs`)

---

### Summary

The `deploy_token` function in `near/token-deployer/src/lib.rs` uses NEAR's `Promise::new(account_id).create_account()` to create the wrapped token account. Because the target account ID is fully deterministic and publicly computable, any unprivileged attacker can pre-create that account by sending a tiny NEAR deposit to it before the bridge's `deploy_token` flow executes. When `create_account()` is subsequently called on an already-existing account, the Promise action fails. Because the bridge contract in `near/omni-bridge/src/lib.rs` commits state changes (`deployed_tokens.insert`) **before** the cross-contract call, the token is permanently recorded as deployed in bridge state while the actual on-chain token contract never exists — making the token irrecoverably undeployable.

---

### Finding Description

**Root cause — `near/token-deployer/src/lib.rs`, lines 60–73:** [1](#0-0) 

```rust
#[payable]
#[access_control_any(roles(Role::Controller, Role::LegacyController))]
pub fn deploy_token(&mut self, account_id: AccountId, metadata: &BasicMetadata) -> Promise {
    Promise::new(account_id)
        .create_account()          // ← fails if account already exists
        .transfer(env::attached_deposit())
        .use_global_contract(self.global_code_hash)
        .function_call(...)
}
```

`create_account()` in NEAR's Promise API is equivalent to `system_instruction::create_account` on Solana: it unconditionally fails if the target account already exists (has any balance or state). There is no `create_account_if_not_exists` primitive used here.

**Deterministic account ID — `near/omni-bridge/src/lib.rs`, lines 2408–2411:** [2](#0-1) 

```rust
let prefix = token_address.get_token_prefix();
let token_id: AccountId = format!("{prefix}.{deployer}")
    .parse()
    .unwrap_or_else(...);
```

The account ID is derived entirely from the public token address and the publicly known deployer contract address. Any observer can compute it before the bridge does.

**State committed before cross-contract call — `near/omni-bridge/src/lib.rs`, lines 2414–2453:** [3](#0-2) 

`self.add_token(...)` and `self.deployed_tokens.insert(&token_id)` are called and committed to state before the `ext_deployer::ext(deployer).deploy_token(...)` cross-contract call is dispatched. If the deployer's `create_account()` fails, the callback receives a failed result, but the bridge's `deployed_tokens` set already contains the token ID. Any subsequent attempt to deploy the same token hits: [4](#0-3) 

```rust
require!(
    self.deployed_tokens.insert(&token_id),
    BridgeError::TokenExists.as_ref()
);
```

…and is permanently rejected. The token is stuck: recorded as deployed in bridge state, but the actual token contract does not exist.

---

### Impact Explanation

An attacker can permanently prevent any specific cross-chain token from ever being deployed as a wrapped asset on NEAR. Once the bridge state records the token as deployed (but the actual account creation failed), no recovery path exists without a contract upgrade. All user funds that were locked on the source chain (Solana, EVM, StarkNet) in anticipation of bridging that token to NEAR become permanently unclaimable on the NEAR side. This matches the allowed impact: **Permanent freezing / irrecoverable lock of user or protocol funds in bridge or token flows.**

---

### Likelihood Explanation

The attack requires only:
1. Monitoring the source chain for a `log_metadata` / `deploy_token` Wormhole message (public).
2. Computing the deterministic NEAR account ID (trivial, fully public inputs).
3. Sending a minimal NEAR deposit (~0.001 NEAR) to that account ID before the relayer submits `deploy_token` on NEAR.

No privileged access, no key compromise, no colluding validators. The attacker only needs to front-run the relayer, which is straightforward given that Wormhole message finalization introduces a multi-second to multi-minute delay before the relayer can act. Likelihood is **High**.

---

### Recommendation

Replace the `create_account()` primitive with a pattern that tolerates a pre-existing account, analogous to what Anchor does on Solana (`transfer` + `allocate` + `assign`). On NEAR this means:

1. Check whether the target account already exists before calling `create_account()`.
2. If it exists and is empty (no code, no meaningful state), transfer the deposit and deploy the global contract directly without `create_account()`.
3. Alternatively, use a non-deterministic or attacker-unguessable account ID suffix (e.g., include a nonce or hash of the Wormhole sequence number).
4. Ensure the bridge's `deploy_token_internal` does **not** commit `deployed_tokens.insert` until the deployer callback confirms success, so a failed deployment can be retried.

---

### Proof of Concept

1. A NEAR token `usdc.near` is registered on Solana and a `log_metadata` Wormhole message is emitted.
2. The bridge computes the target NEAR account ID, e.g., `sol-usdc.token-deployer.near`.
3. Attacker sends `0.001 NEAR` to `sol-usdc.token-deployer.near` via a standard transfer, creating the account.
4. The relayer calls `deploy_token` on the NEAR bridge with a valid proof.
5. `deploy_token_internal` runs: `deployed_tokens.insert("sol-usdc.token-deployer.near")` is committed to state.
6. The cross-contract call to `token-deployer.near::deploy_token("sol-usdc.token-deployer.near", ...)` executes `Promise::new(...).create_account()` — this fails because the account already exists.
7. The callback receives a failed result; the bridge state already has the token marked as deployed.
8. Any future `deploy_token` call for the same token hits `BridgeError::TokenExists` and is rejected.
9. All SOL-side USDC locked for bridging to NEAR is permanently unclaimable.
