### Title
Missing Class Existence Check in `execute_replace_class` Allows Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` does not verify that the new class hash supplied to the `replace_class` syscall corresponds to a previously declared contract class. Any contract can call `replace_class` with an arbitrary, undeclared class hash. The OS accepts the state update without validation, permanently making the contract uncallable and freezing any funds held within it.

---

### Finding Description

The analog to the ERC20Helper missing existence check is: just as calling a non-existent EVM contract silently succeeds, calling `replace_class` with a non-existent StarkNet class hash silently succeeds — with the consequence that the contract becomes permanently uncallable.

**Root cause — `syscall_impls.cairo`, `execute_replace_class` (lines 877–916):**

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The function reads `request.class_hash` and immediately writes it into `contract_state_changes` with no check that the class hash is present in `contract_class_changes` (i.e., was ever declared). The developer-acknowledged TODO at line 898 explicitly confirms this check is missing.

The same pattern exists in the deprecated path:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    ...
    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
    dict_update{dict_ptr=contract_state_changes}(...);
    ...
}
``` [2](#0-1) 

**What happens on the next call to the bricked contract:**

In `execute_entry_point`, the OS reads the compiled class hash for the contract's class hash from `contract_class_changes`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [3](#0-2) 

If `class_hash` was set to an undeclared value, `dict_read` returns 0 (`UNINITIALIZED_CLASS_HASH`). The OS then calls `find_element` looking for a compiled class with hash 0:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [4](#0-3) 

No such compiled class exists, so every future call to the contract fails. The contract is permanently bricked. `UNINITIALIZED_CLASS_HASH` is defined as 0: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract holding ERC-20 tokens or ETH (via the fee token) that has its class replaced with an undeclared class hash becomes permanently uncallable. All assets stored in that contract's storage are irrecoverably frozen. The state commitment will encode the invalid class hash, and no future block can recover the contract because the class hash cannot be resolved to executable bytecode.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is callable by any deployed contract — no privileged role is required. Realistic scenarios include:

1. **Malicious contract**: An attacker deploys a contract, deposits victim funds into it (e.g., via a DeFi protocol), then calls `replace_class(0xDEAD)` to permanently freeze those funds.
2. **Buggy upgrade path**: A legitimate contract with an on-chain upgrade mechanism (e.g., governed by a DAO vote) is tricked into voting for an invalid class hash, permanently bricking itself and all funds it holds.
3. **Griefing**: An attacker self-destructs their own contract after receiving funds from others (e.g., a shared escrow), freezing all deposited assets.

The missing check is explicitly acknowledged in the codebase via a TODO comment, confirming the developers are aware it is absent.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the class hash has been declared by checking its presence in `contract_class_changes`:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This check should be added in both:
- `execute_replace_class` in `syscall_impls.cairo` (line 898)
- `execute_replace_class` in `deprecated_execute_syscalls.cairo` (line 311)

---

### Proof of Concept

1. Attacker deploys `VaultContract` which holds user funds and exposes an `upgrade(class_hash)` function that calls `replace_class`.
2. Users deposit tokens into `VaultContract`.
3. Attacker calls `VaultContract.upgrade(0xDEADBEEF)` — an arbitrary value that was never declared.
4. The OS executes `execute_replace_class`: reads `class_hash = 0xDEADBEEF`, finds no validation, writes `StateEntry(class_hash=0xDEADBEEF, ...)` into `contract_state_changes`. The syscall returns success.
5. The block is proven and finalized. The on-chain state now records `VaultContract.class_hash = 0xDEADBEEF`.
6. Any subsequent call to `VaultContract` causes `execute_entry_point` to call `dict_read(contract_class_changes, 0xDEADBEEF)` → returns 0, then `find_element(..., key=0)` → fails (no compiled class with hash 0 exists in the block's facts bundle).
7. All user funds in `VaultContract` are permanently frozen with no recovery path.
