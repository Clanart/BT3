### Title
Missing shard_id binding in `set_state_header` allows cross-shard state installation via Merkle position confusion â€” (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a given `shard_id` but never verifies that the embedded chunk's `shard_id` field matches the requested shard. The `verify_path` call only proves the chunk hash exists *somewhere* in the block's Merkle tree â€” it does not bind the proof to a specific leaf index (shard slot). A malicious peer can present shard 0's chunk as the header for shard 1, pass all validation checks, and cause the syncing node to install shard 0's state into shard 1's trie.

### Finding Description

**Entrypoint:** `set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called from the state-sync downloader after receiving a `ShardStateSyncResponseHeader` from a peer.

**Root cause â€” no shard_id binding check:** [1](#0-0) 

The function extracts `chunk` from the header and immediately calls `validate_chunk_proofs`, which only checks internal consistency (hash, tx root, receipts root) â€” it never checks that `chunk.shard_id()` equals the `shard_id` parameter: [2](#0-1) 

Then `verify_path` is called to prove the chunk is in the block: [3](#0-2) 

**`verify_path` does not check leaf position.** It only checks that `compute_root_from_path(path, hash(item)) == root`: [4](#0-3) 

In a 2-shard block, the Merkle tree has two leaves: `[chunk_shard0, chunk_shard1]`. The path for index 0 is `[{hash(chunk_shard1), Right}]`. If an attacker supplies `chunk = chunk_shard0` with `chunk_proof = path_for_index_0`, then `verify_path(chunk_headers_root, path_for_index_0, ChunkHashHeight(chunk_shard0.hash, ...))` returns `true` â€” because chunk_shard0 IS genuinely in the tree at index 0. The function has no way to know the caller requested shard_id=1 (index 1).

**No shard_id check anywhere in the function.** After all checks pass, the header is stored under the attacker-controlled key: [5](#0-4) 

**Receipt proof check (step 4) is not a blocker.** The receipt proofs are for incoming receipts *to* the claimed shard_id=1. These are valid on-chain public data the attacker can obtain and supply correctly, independent of which chunk they embed: [6](#0-5) 

**`prev_chunk_header` check is not a blocker.** The attacker supplies shard 0's previous chunk header with its index-0 Merkle proof, which also passes `verify_path` for the same reason.

**`state_root_node` check (step 5) is not a blocker.** The attacker supplies shard 0's `state_root_node`, which is consistent with shard 0's `prev_state_root` embedded in the chunk: [7](#0-6) 

**Downstream propagation â€” `set_state_part`:** Reads the stored header, extracts `state_root = chunk_shard0.prev_state_root()`, and validates parts against it. The attacker supplies shard 0's state parts (public data), which are valid for shard 0's state root. `validate_state_part(shard_id=1, state_root=shard0_root, ...)` passes because the parts are cryptographically valid for that root: [8](#0-7) 

**Downstream propagation â€” `set_state_finalize`:** Uses `shard_id=1` to derive `shard_uid=1`, but applies shard 0's chunk with shard 0's `prev_state_root`. Since the state parts were installed into shard 1's trie slot using shard 0's data, `apply_chunk` succeeds and shard 1's trie is populated with shard 0's state: [9](#0-8) 

### Impact Explanation

A syncing node's shard 1 trie is silently populated with shard 0's account data. After `set_state_finalize`, the node believes it has valid state for shard 1 but is actually operating on shard 0's state. This causes incorrect transaction processing, wrong account balances, and divergence from the canonical chain for all shard 1 operations on that node.

### Likelihood Explanation

Any peer that a syncing node contacts for state sync can trigger this. State sync peers are not required to be validators or trusted operators â€” any reachable node can serve state sync responses. The attack requires only public on-chain data (shard 0's chunk, its Merkle proof, shard 0's state parts, and valid receipt proofs for shard 1).

### Recommendation

Add an explicit shard_id binding check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check to `prev_chunk_header.shard_id()`. Additionally, consider using `verify_path_with_index` (which validates the Left/Right direction sequence against the expected leaf index) instead of `verify_path` for the chunk Merkle proof, to enforce positional binding: [10](#0-9) 

### Proof of Concept

In a two-shard `TestEnv`:
1. Produce several blocks so both shards have chunks.
2. Obtain `get_state_response_header(shard_id=0, sync_hash)` â†’ `header_shard0`.
3. Obtain valid receipt proofs for shard_id=1 from the chain.
4. Construct a `ShardStateSyncResponseHeaderV2` with:
   - `chunk` = `header_shard0.chunk` (shard 0's chunk)
   - `chunk_proof` = `header_shard0.chunk_proof` (Merkle path at index 0)
   - `prev_chunk_header` / `prev_chunk_proof` = shard 0's previous chunk data
   - `incoming_receipts_proofs` / `root_proofs` = valid proofs for shard_id=1
   - `state_root_node` = shard 0's state root node
5. Call `set_state_header(shard_id=1, sync_hash, forged_header)` on client[1] â€” assert it returns `Ok(())`.
6. Assert the stored header under `StateHeaderKey(1, sync_hash)` contains shard 0's `prev_state_root`.
7. Supply shard 0's state parts via `set_state_part(shard_id=1, ...)` â€” assert they pass validation.
8. Call `set_state_finalize(shard_id=1, sync_hash)` â€” assert shard 1's trie now contains shard 0's accounts.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
