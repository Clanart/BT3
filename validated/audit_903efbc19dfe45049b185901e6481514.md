### Title
`FakeProver` Placeholder Unconditionally Accepts Any Proof, Enabling Unauthorized eNear Token Minting During Unpaused Window — (`File: evm/src/eNear/contracts/FakeProver.sol`)

---

### Summary

`FakeProver` is a production-deployed contract that unconditionally returns `true` for every call to `proveOutcome()`, completely bypassing proof verification for the `eNear` token. It is embedded as the prover address directly in the `eNear` creation bytecode at deployment time. The only protection against arbitrary minting is an off-chain `adminPause` call that must be made after deployment — a step that is not atomically enforced in any on-chain initializer. During the window between `eNear` deployment (with `pausedFlags = 0`) and the manual pause call, any unprivileged caller can invoke `eNear.finaliseNearToEthTransfer` directly with attacker-crafted Borsh-encoded proof data, receive a `true` from `FakeProver`, and mint arbitrary eNear tokens to any address.

---

### Finding Description

**Root cause — `FakeProver` unconditionally returns `true`:**

`FakeProver.proveOutcome()` ignores both arguments and always returns `true`:

```solidity
// evm/src/eNear/contracts/FakeProver.sol
contract FakeProver is INearProver {
    function proveOutcome(bytes calldata, uint64) external pure returns (bool) {
        return true;
    }
}
``` [1](#0-0) 

**`FakeProver` is embedded as the prover in `eNear` at construction time:**

The deployment pipeline (`evm.smk`) first deploys `FakeProver`, then injects its address into the `eNear` creation bytecode via `sed` substitution before deploying `eNear`:

```python
# e2e-testing/snakefiles/evm.smk  lines 98-110
rule evm_create_enear_creation_file:
    input:
        fake_prover = rules.evm_deploy_fake_prover.output,
        ...
    shell: """
    cat {input.template} | \
    sed "s/<PROVER_ADDRESS>/$(cat {input.fake_prover} | jq -r .fakeProverAddress ...)" > {output}
    """
``` [2](#0-1) 

The test suite confirms this: `eNear` is constructed with `FakeProver` as its prover and `pausedFlags = UNPAUSED_ALL (0)`:

```typescript
// evm/tests/eNearProxy.test.ts  lines 40-48
eNear = await eNearContractFactory.deploy(
  ERC20_NAME, ERC20_SYMBOL,
  Buffer.from("eNearBridge", "utf-8"),
  await nearProver.getAddress(),   // <-- FakeProver
  minBlockAcceptanceHeight,
  eNearAdmin.address,
  UNPAUSED_ALL,                    // <-- no pause at construction
)
``` [3](#0-2) 

**The pause is a separate, non-atomic, off-chain step:**

The README documents the intended security model:

> We will pause the `finaliseNearToEthTransfer` and `transferToNear` functions, and only `eNearProxy`, as the admin, will have the ability to call these functions. [4](#0-3) 

But neither the `eNear` constructor nor any on-chain initializer in `ENearProxy` atomically sets the pause. The `e-near-set-admin` task in `scripts.ts` only sets the admin — it does not pause:

```typescript
// evm/src/eNear/scripts.ts  lines 35-42
task("e-near-set-admin", "Set the proxy as admin for eNear")
  .setAction(async (taskArgs, hre) => {
    const eNear = await ethers.getContractAt("IENear", taskArgs.enear)
    await eNear.adminSstore(9, ethers.zeroPadValue(taskArgs.newAdmin, 32))
    // No adminPause() call here
  })
``` [5](#0-4) 

**The `ENearProxy.finaliseNearToEthTransfer` legacy path is also unpaused by default:**

`ENearProxy.initialize` never sets `PAUSED_LEGACY_FIN_TRANSFER`, so `finaliseNearToEthTransfer` on the proxy is callable by anyone immediately after deployment. While this path does call the real prover, it then forwards the call to `eNear.finaliseNearToEthTransfer`, which re-verifies with `FakeProver` (always `true`):

```solidity
// evm/src/eNear/contracts/ENearProxy.sol  lines 80-90
function finaliseNearToEthTransfer(
    bytes memory proofData,
    uint64 proofBlockHeight
) external whenNotPaused(PAUSED_LEGACY_FIN_TRANSFER) {
    require(prover.proveOutcome(proofData, proofBlockHeight), "Proof should be valid");
    eNear.finaliseNearToEthTransfer(proofData, proofBlockHeight);
}
``` [6](#0-5) 

**Proof data format is publicly documented in `ENearProxy.mint`:**

The `mint` function reveals the exact Borsh encoding an attacker must replicate to craft a valid-looking proof accepted by `eNear`:

```solidity
// evm/src/eNear/contracts/ENearProxy.sol  lines 58-72
bytes memory fakeProofData = bytes.concat(
    new bytes(72),
    hex"01000000",
    abi.encodePacked(currentReceiptId),
    new bytes(24),
    abi.encodePacked(Borsh.swapBytes4(uint32(nearConnector.length))),
    abi.encodePacked(nearConnector),
    hex"022500000000",
    abi.encodePacked(Borsh.swapBytes16(amount)),
    abi.encodePacked(to),
    new bytes(280)
);
currentReceiptId += 1;
eNear.finaliseNearToEthTransfer(fakeProofData, 0);
``` [7](#0-6)

### Impact Explanation

**Critical — Direct unauthorized mint of bridged eNear tokens on EVM.**

During the deployment window (between `eNear` construction with `FakeProver` and the manual `adminPause` call), any unprivileged Ethereum address can:

1. Construct Borsh-encoded `proofData` encoding an arbitrary `to` address and `amount`, using any unused `receiptId`.
2. Call `eNear.finaliseNearToEthTransfer(proofData, 0)` directly.
3. `FakeProver.proveOutcome()` returns `true` unconditionally.
4. `eNear` mints the specified amount of eNear tokens to the attacker's address.

This results in unbacked eNear supply — tokens minted without any corresponding NEAR locked on the NEAR side — directly stealing value from legitimate eNear holders and breaking bridge collateralization.

---

### Likelihood Explanation

**High.** The attack requires no privileged key, no MPC threshold, no Wormhole guardian, and no chain-level attack. The proof data format is fully documented in `ENearProxy.mint`. The only barrier is the deployment window, which exists in every deployment of this system. Additionally, if the pause is ever lifted (e.g., during an upgrade or misconfiguration), the vulnerability is permanently re-exposed. The `FakeProver` contract has no owner, no access control, and cannot be disabled.

---

### Recommendation

1. **Atomic pause at construction**: Deploy `eNear` with `pausedFlags = PAUSE_FINALISE_FROM_NEAR | PAUSE_TRANSFER_TO_NEAR` set in the constructor, eliminating the deployment window entirely.
2. **Remove `FakeProver` from production scope**: Replace the `FakeProver`-based minting mechanism with a proper access-controlled mint function on `eNear` itself, or use a real prover with a dedicated admin-only bypass path that does not expose `finaliseNearToEthTransfer` to `FakeProver`.
3. **Enforce pause atomically in `ENearProxy.initialize`**: Call `eNear.adminPause(PAUSE_FINALISE_FROM_NEAR | PAUSE_TRANSFER_TO_NEAR)` inside `initialize` before returning, so the proxy deployment itself closes the window.
