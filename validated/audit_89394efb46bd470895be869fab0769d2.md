### Title
Missing Upper Bound on `max_price_per_unit` Enables Fee-Free Transaction Execution via Field Arithmetic Overflow — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`pack_resource_bounds` enforces an upper bound on `max_amount` (`≤ 2^64 − 1`) but only enforces a lower bound of zero on `max_price_per_unit` via `assert_nn`. Because `max_price_per_unit` is later used in unchecked field-arithmetic multiplication inside `compute_max_possible_fee`, a transaction sender can craft values that cause the entire fee expression to reduce to zero modulo the field prime, causing `charge_fee` to silently skip fee collection.

---

### Finding Description

In `pack_resource_bounds` (transaction_hash.cairo, lines 103–107):

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // upper-bounded ✓
    assert_nn(resource_bounds.max_price_per_unit);            // only lower-bounded ✗
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

`max_amount` is correctly bounded to `[0, 2^64 − 1]`. `max_price_per_unit` is only checked to be non-negative, meaning it can be any value in `[0, (P−1)/2]` where `P ≈ 2^251` is the StarkNet field prime. The intended semantic is that `max_price_per_unit` fits in 128 bits (the packing formula places it in the low 128 bits of a 256-bit word), but the missing `assert_nn_le(max_price_per_unit, 2^128 − 1)` leaves values up to `≈ 2^250` accepted.

These raw (unvalidated) values are then used directly in `compute_max_possible_fee` (transaction_impls.cairo, lines 95–101):

```cairo
tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is modular (field arithmetic). Because `max_price_per_unit` can be close to `P`, the products `max_amount * max_price_per_unit` can wrap around modulo `P`. A user can choose values such that the entire sum reduces to `0 mod P`.

`charge_fee` (transaction_impls.cairo, lines 121–125) then short-circuits:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);

if (max_fee == 0) {
    return ();
}
```

When `max_fee == 0`, the function returns immediately without executing the ERC-20 transfer, so no fee is charged and the transaction executes for free.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker who successfully crafts a transaction where `compute_max_possible_fee` returns `0` executes their transaction without paying any fee. The sequencer performs real computational work (state transitions, proof generation) but receives no compensation. At scale, this enables free-of-cost transaction spam that drains sequencer resources and can escalate to a network halt, matching both the "direct loss of funds" and "network not being able to confirm new transactions" impact categories.

---

### Likelihood Explanation

**Medium.** The attacker is an unprivileged transaction sender. The required `max_price_per_unit` values (on the order of `P / max_amount ≈ 2^187` for typical gas amounts) are visibly anomalous and an honest sequencer's mempool may filter them. However:

1. The OS Cairo code is the authoritative validity rule; it does not reject these values.
2. A sequencer that does not independently re-validate `max_price_per_unit < 2^128` before block inclusion will produce a valid proof for a fee-free transaction.
3. The field-arithmetic cancellation can be engineered with multiple resource-bound entries, making the individual price values appear less extreme.

---

### Recommendation

In `pack_resource_bounds`, add an explicit upper-bound assertion mirroring the one already present for `max_amount`:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

This matches the 128-bit slot reserved for `max_price_per_unit` in the packing formula and prevents field-arithmetic overflow in `compute_max_possible_fee`.

---

### Proof of Concept

Choose transaction parameters (all within `assert_nn` / `assert_nn_le` bounds):

| Field | Value |
|---|---|
| L1 gas `max_amount` (A) | `2` |
| L2 gas `max_amount` (B) | `1` |
| L1 data gas `max_amount` (C) | `0` |
| `tip` | `0` |
| L1 gas `max_price_per_unit` (a) | `(P+1)/4` ≈ `2^249` — passes `assert_nn` |
| L2 gas `max_price_per_unit` (b) | `(P−1)/2` ≈ `2^250` — passes `assert_nn` |

Fee computation:

```
max_fee = A*a + B*b + C*0
        = 2*(P+1)/4 + 1*(P−1)/2
        = (P+1)/2 + (P−1)/2
        = P
        ≡ 0  (mod P)
```

`charge_fee` sees `max_fee == 0` and returns without transferring any tokens. The transaction's `__execute__` entry point runs with L2 gas budget `B = 1` (or any chosen amount), and the sequencer receives zero fee. [1](#0-0) [2](#0-1) [3](#0-2)
