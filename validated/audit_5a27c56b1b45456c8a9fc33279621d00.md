Audit Report

## Title
Router `pay()` consumes stranded native ETH balance for WETH-input swaps, enabling theft of prior users' ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function reads `address(this).balance` — the router's total ETH — when settling WETH-leg payments, rather than the current transaction's `msg.value`. Any ETH left on the router from a prior user's overpayment is silently consumed to fund a subsequent caller's WETH swap. Additionally, the public `refundETH()` function transfers the entire ETH balance to any caller with no ownership attribution, enabling direct theft of stranded ETH.

## Finding Description
In `PeripheryPayments.sol`, `pay()` branches on `token == WETH` and reads `address(this).balance` as `nativeBalance`: [1](#0-0) 

When `nativeBalance >= value`, the router deposits its own ETH as WETH and forwards it to the pool without pulling any token from `payer`. The `payer` identity stored in transient context is irrelevant when the router's ETH balance is sufficient.

ETH accumulates on the router when a user calls a payable entry point such as `exactInputSingle` with `msg.value` exceeding `amountIn` and omits a `refundETH()` call in the same multicall: [2](#0-1) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent excess `msg.value` from sitting on the router after a swap completes: [3](#0-2) 

`refundETH()` compounds the issue by sending the entire ETH balance to `msg.sender` with no ownership check: [4](#0-3) 

**Exploit flow (Path A — free WETH swap):**
1. Victim calls `exactInputSingle{value: 2 ether}` with `amountIn = 1 ether`, `tokenIn = WETH`, without a `refundETH()` in the same multicall.
2. `pay()` sees `address(this).balance == 2 ether >= 1 ether`, deposits 1 ether as WETH, forwards to pool. Victim receives token-out.
3. 1 ether remains stranded on the router.
4. Attacker calls `exactInputSingle{value: 0}` with `amountIn = 1 ether`, `tokenIn = WETH`.
5. `pay()` sees `address(this).balance == 1 ether >= 1 ether`, deposits the stranded ETH as WETH, forwards to pool. Attacker receives token-out having paid 0 ETH and 0 WETH.

**Exploit flow (Path B — direct ETH theft):**
Attacker calls `refundETH()` in a standalone transaction; the entire stranded ETH balance is transferred to the attacker.

The callback path in `_justPayCallback` passes `_getPayer()` (the original `msg.sender`) to `pay()`, but this payer is never charged when `address(this).balance >= value`: [5](#0-4) 

## Impact Explanation
Both paths result in direct, unrecoverable loss of victim's principal ETH. Path A also constitutes a swap conservation failure: the attacker receives pool output tokens without providing any input, violating the invariant that the pool must receive the owed input amount from the trader. The loss is bounded only by the stranded ETH amount. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
ETH stranding is a realistic user error: direct (non-multicall) calls to `exactInputSingle` or other payable entry points with a safety buffer of ETH leave residue. The attack requires no special privilege, no oracle manipulation, and no pool state precondition — only a monitoring bot watching the router's ETH balance. Both attack paths are executable by any unprivileged EOA.

## Recommendation
1. **Track `msg.value` per entry point in transient storage.** At the start of each payable public function, record `msg.value` in a transient slot. In `pay`, use `min(transientMsgValue, value)` as the native contribution rather than `address(this).balance`.
2. **Alternatively, auto-refund excess ETH at the end of every non-multicall payable entry point**, so no ETH can survive past the transaction boundary.
3. **Add a caller-binding check to `refundETH()`.** Record the original `msg.sender` in transient storage at multicall entry and restrict `refundETH()` to that address, or document clearly that it is intentionally open and that users must always pair it in the same multicall.

## Proof of Concept
```solidity
// Setup: pool with WETH as token0, token1 as token1; victim and attacker are EOAs.

// Step 1 — victim overpays and omits refundETH()
vm.deal(victim, 2 ether);
vm.prank(victim);
router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
    pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
    zeroForOne: true, amountIn: 1 ether, amountOutMinimum: 0,
    recipient: victim, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
}));
// 1 ether is now stranded on the router.

// Step 2 — attacker steals via free swap (Path A)
uint256 attackerToken1Before = token1.balanceOf(attacker);
vm.prank(attacker);
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
    zeroForOne: true, amountIn: 1 ether, amountOutMinimum: 0,
    recipient: attacker, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
}));
assert(token1.balanceOf(attacker) > attackerToken1Before);
assert(address(router).balance == 0);

// OR Path B — direct ETH theft
vm.prank(attacker);
router.refundETH();
// attacker receives 1 ether; victim's overpayment is gone.
```
