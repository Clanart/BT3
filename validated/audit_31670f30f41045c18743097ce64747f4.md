### Title
FIFO Eviction of Bounded `AnteCache` Enables Targeted Nonce-Replacement DoS, Preventing Transaction Cancellation — (File: `ante/cache/antecache.go`)

### Summary
The `AnteCache` uses FIFO eviction when bounded. An attacker can deliberately fill the cache to evict a victim's `(address, nonce)` entry. Once evicted, the victim's replacement transaction (same nonce, higher fee) is rejected with `ErrInvalidSequence` because the CheckTx state has already incremented the account sequence for the original pending transaction. This prevents the victim from canceling or replacing their pending transaction, causing the original transaction to execute against the victim's will.

### Finding Description

The `AnteCache` in `ante/cache/antecache.go` is a bounded FIFO cache tracking `(address, nonce)` pairs for pending transactions. When the cache is full (`maxTx > 0`), `Set` evicts the oldest entry (front of the FIFO list) to make room: [1](#0-0) 

The `CheckAndSetEthSenderNonce` function in `ante/eth.go` uses the cache as a replacement shortcut:
- If `(address, nonce)` **is** in the cache → nonce check and sequence increment are **bypassed** (line 295)
- If `(address, nonce)` is **not** in the cache → nonce check is performed against the CheckTx state sequence [2](#0-1) 

When a transaction with nonce N is first admitted during CheckTx:
1. Nonce check passes (`N == store_sequence`)
2. Account sequence is incremented to `N+1` in the CheckTx state via `ak.SetAccount(ctx, acc)`
3. After the full ante chain succeeds, `(address, N)` is flushed into the cache [3](#0-2) [4](#0-3) 

The `AnteCache` is initialized in production with `mempoolMaxTxs` — a positive bounded integer — making FIFO eviction active: [5](#0-4) 

**Attack path:**

1. Victim submits `tx1` with nonce `N`. CheckTx state: `victim.sequence = N+1`. Cache: `{..., (victim, N)}`.
2. Attacker submits `maxTx` valid transactions from distinct addresses, each triggering a FIFO eviction. Eventually `(victim, N)` is evicted from the front of the list.
3. Victim submits `tx2` (replacement, same nonce `N`, higher fee).
4. In `CheckAndSetEthSenderNonce`:
   - `ex = nonceCache.Exists(victim, N)` → **false** (evicted)
   - Falls through to normal nonce check
   - `expectedNonce = acc.GetSequence()` → reads **N+1** from the persistent CheckTx state (incremented by `tx1`)
   - `txNonce = N`
   - `N != N+1` → **FAIL** with `ErrInvalidSequence` [6](#0-5) 

The victim's replacement transaction is permanently rejected. The original `tx1` remains in the mempool and will execute.

The test comment in `ante/cache/antecache_test.go` acknowledges the related failure mode: [7](#0-6) 

### Impact Explanation

This is a **High** impact ante handler / mempool bug. The victim cannot cancel or replace a pending transaction. If the victim submitted a high-value transfer and subsequently wants to cancel it (e.g., wrong recipient, changed intent), the attacker can prevent cancellation by filling the cache. The original transaction executes, causing the victim's funds to be transferred to an unintended destination — a direct mis-accounting of valid user funds. This maps to: *"ante handler, mempool… bug that permits… valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

**Medium.** The attacker needs to submit enough valid, fee-paying transactions from distinct addresses to fill the cache (`maxTx` slots). The attack is cheapest when the victim's entry is near the front of the FIFO list (i.e., the victim's tx has been pending for a long time). The attacker can monitor the mempool via `txpool_content` RPC to identify high-value pending transactions and time the eviction burst. No privileged access is required — only sufficient balance to pay fees for `maxTx` transactions.

### Recommendation

1. **LRU eviction instead of FIFO**: Entries accessed recently (i.e., replacement candidates) are less likely to be evicted. This directly mirrors the fix needed in the Morpho analog.
2. **Eviction callback to reset CheckTx state**: When `(address, nonce)` is evicted from the cache, reset the CheckTx state sequence for that address back to the committed store sequence, so subsequent replacement attempts are not blocked by the stale increment.

// bounded caches should continue tracking the latest nonce
// even after they reach capacity. Right now Set simply returns when
// len(cache) >= maxTx without signalling failure, so callers assume the nonce
// was cached. When that happens, a replacement tx never sees its nonce in the
// cache and gets rejected with ErrInvalidSequence, effectively disabling nonce
// replacement once a node hits maxTx and amplifying the leak documented above.
func TestAnteCache_DropNewEntriesWhenFull(t *testing.T) {
	antecache := cache.NewAnteCache(1)
	address := "cosmos1huydeevpz37sd9shv2gqf9p8unc0j89x59cn3c"

	antecache.Set(address, 1)
	antecache.Set(address, 2)

	require.True(t, antecache.Exists(address, 2), "cache should keep track of replacement nonce even when full")
}
