Audit Report

## Title
Excess ETH stranded in router by WETH `exactInputSingle` is claimable by any subsequent caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`exactInputSingle` (and all other `payable` swap entry points) accept arbitrary `msg.value` with no enforcement that `msg.value == amountIn` when `tokenIn == WETH`. The `pay` function wraps only the required `amountIn` worth of ETH, leaving any excess in the router. Because `refundETH()` unconditionally sends the router's entire ETH balance to `msg.sender`, any subsequent caller can steal the stranded ETH. The same stranded ETH can also be silently consumed by a subsequent WETH swap via the `nativeBalance >= value` branch in `pay`.

## Finding Description
`exactInputSingle` is declared `external payable` with no validation on `msg.value`: [1](#0-0) 

Inside the pool callback, `_justPayCallback` calls `pay(token=WETH, payer=user, recipient=pool, value=amountIn)`: [2](#0-1) 

`pay` reads `address(this).balance` as the available ETH pool. If `nativeBalance >= value`, it wraps exactly `value` ETH and transfers WETH to the pool — the remainder stays in the router with no accounting or refund: [3](#0-2) 

`refundETH()` then sends the router's **entire** ETH balance to whoever calls it, with no restriction to the original depositor: [4](#0-3) 

The `receive()` guard rejecting non-WETH ETH senders does not apply to ETH sent via `msg.value` in a function call — it only blocks plain ETH transfers: [5](#0-4) 

**Exact exploit path:**
1. User calls `exactInputSingle{value: 200 ether}(tokenIn=WETH, amountIn=100 ether, ...)`.
2. `pay` sees `nativeBalance=200 >= value=100`, wraps 100 ETH, sends WETH to pool. 100 ETH remains in router.
3. Attacker calls `refundETH()` in a subsequent transaction, receiving all 100 ETH.
4. Alternatively, a subsequent WETH swap with `msg.value=0` and `amountIn <= 100` silently consumes the stranded ETH via the same `nativeBalance >= value` branch, giving the attacker a free swap.

There is no per-transaction ETH accounting, no automatic post-swap refund, and no restriction on who may call `refundETH()`.

## Impact Explanation
Direct, permanent loss of user ETH equal to `msg.value - amountIn`. The stranded ETH is immediately claimable by any address calling `refundETH()` in a subsequent transaction, or silently consumed to subsidize a subsequent WETH swap. The swap itself succeeds, so the loss is invisible at the application layer unless the user monitors their ETH balance. This constitutes a direct loss of user principal with no privileged access required.

## Likelihood Explanation
Any user who calls a `payable` swap entry point with `tokenIn=WETH` and `msg.value > amountIn` without wrapping the call in `multicall([swap, refundETH])` is vulnerable. Wallets and integrators following the Uniswap v3 pattern of sending `msg.value = amountInMaximum` for exact-output WETH swaps will trigger this on every such trade. The attack requires no setup: the attacker monitors the mempool for stranded ETH and calls `refundETH()` after the victim's transaction confirms.

## Recommendation
Add a post-swap ETH balance check in each `payable` swap entry point and auto-refund any remainder to `msg.sender` before returning. Alternatively, enforce `msg.value == 0 || (tokenIn == WETH && msg.value == amountIn)` at entry. The safest fix is to track per-transaction ETH contributed via a transient variable and restrict `refundETH` to returning only that amount to `msg.sender`, preventing cross-transaction ETH theft.

## Proof of Concept
```solidity
function test_excessEthStranded() public {
    // User swaps 100 WETH, sends 200 ETH — 100 ETH stranded in router
    router.exactInputSingle{value: 200 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: wethUsdcPool,
            tokenIn: WETH,
            tokenOut: USDC,
            zeroForOne: true,
            amountIn: 100 ether,
            amountOutMinimum: 0,
            recipient: user,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    assertEq(address(router).balance, 100 ether);

    // Attacker claims stranded ETH with no prior interaction
    vm.prank(attacker);
    router.refundETH();
    assertEq(attacker.balance, 100 ether);
    assertEq(address(router).balance, 0);
}
```
