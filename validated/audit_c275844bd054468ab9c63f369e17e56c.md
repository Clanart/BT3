The code confirms the vulnerability. The key facts:

1. `exactInputSingle` is `payable` (line 67), so ETH sent as `msg.value` lands in the router.
2. `receive()` (lines 32-34) only blocks *direct* ETH transfers — `msg.value` via payable calls bypasses it entirely.
3. `pay()` (lines 74-81) reads `address(this).balance` globally with no per-transaction accounting. The hybrid branch at lines 78-81 wraps and forwards the router's full ETH balance for any WETH payment where `0 < nativeBalance < value`, regardless of who deposited it.
4. `refundETH()` (lines 58-63) sends the entire balance to `msg.sender` — if not called, ETH is permanently stranded until consumed by the next WETH swap.
5. `_justPayCallback` (lines 192-199) passes `_getPayer()` (the current caller) to `pay()`, but `pay()` never checks whether the router's ETH balance was deposited by that payer.

The attack path is fully reachable with no privileged access required.

Audit Report

## Title
Stale Router ETH Consumed by Subsequent WETH Swap via Hybrid `pay()` Path, Causing Direct Loss of Prior Depositor's ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` as a global pool of ETH when paying WETH. When `0 < nativeBalance < value`, it wraps and forwards the router's entire ETH balance to subsidize the current payer's obligation, without verifying that the ETH belongs to the current transaction's sender. Residual ETH from a prior over-funded WETH swap is silently consumed to reduce a subsequent caller's WETH obligation, causing direct loss of the prior depositor's ETH principal.

## Finding Description
`exactInputSingle` (and other payable swap functions) accept `msg.value`, which lands in the router. The `receive()` guard at line 32-34 only blocks direct ETH pushes — it does not prevent ETH accumulation via `msg.value`. When a user sends excess ETH (e.g., 2 ETH for a 1 WETH swap), the callback wraps exactly `amountIn` and leaves the remainder in the router. Without a `refundETH()` call, that ETH is stranded.

The hybrid branch in `pay()` at lines 78-81:
```solidity
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```
fires for any WETH payment where the router holds any ETH. It unconditionally wraps and forwards the router's full ETH balance, then pulls only the shortfall from the current payer. There is no check that `address(this).balance` was deposited by the current payer or in the current transaction.

**Exploit flow:**
1. User A calls `exactInputSingle{value: 2 ETH}(amountIn=1 ETH, tokenIn=WETH)`. Callback wraps 1 ETH, pays pool. 1 ETH remains in router. User A omits `refundETH()`.
2. User B calls `exactInputSingle(amountIn=1.5 WETH, tokenIn=WETH)` with no ETH. Callback invokes `pay(WETH, UserB, pool, 1.5e18)`. `nativeBalance = 1 ETH`, hybrid branch fires: router wraps 1 ETH → pool, pulls only 0.5 WETH from User B.
3. Pool receives correct 1.5 WETH (no revert). User A loses 1 ETH silently. User B pays 1 WETH less than owed.

## Impact Explanation
Direct loss of ETH principal for any user who leaves ETH in the router without immediately reclaiming it. The victim's transaction already succeeded; the theft occurs in a later, unrelated transaction with no on-chain signal. Severity: **High** — silent, direct loss of user funds with no recovery path.

## Likelihood Explanation
Realistic. Users commonly send a round ETH amount for WETH swaps and rely on `refundETH()` in a multicall. A dropped or failed `refundETH()` call, a simple EOA call without multicall, or any revert in a later multicall step leaves ETH stranded. The next WETH swap by any user triggers the theft automatically, requiring no special attacker capability — any ordinary swap suffices.

## Recommendation
Restrict the hybrid ETH path to cases where the router itself is the payer (mid-path hops), or track per-transaction ETH deposits and limit the hybrid path to that amount only:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    // Only use router's ETH balance when the router is paying (mid-path hop)
    if (payer == address(this) && nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```
Alternatively, use transient storage to record `msg.value` at entry and limit the hybrid path to that recorded amount only.
