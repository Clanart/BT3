### Title
Unauthenticated Class Declaration via BOOTSTRAP Sender Address Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The StarkNet OS `execute_declare_transaction` function contains a special-case "BOOTSTRAP" execution path that completely skips signature verification (`__validate_declare__`), nonce enforcement, fee charging, and sender-address existence checks. Any unprivileged user can craft a v3 declare transaction with `sender_address = 'BOOTSTRAP'`, `nonce = 0`, and all resource bounds set to zero to trigger this path, registering an arbitrary `compiled_class_hash` for a valid Sierra `class_hash` without any authorization.

---

### Finding Description

In `execute_declare_transaction`, before the normal validation flow, the OS checks:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the OS:

1. **Skips `__validate_declare__`** — the account contract's signature verification entry point is never called.
2. **Skips `check_and_increment_nonce`** — the nonce is not verified against the on-chain state.
3. **Skips `charge_fee`** — no fee is deducted.
4. **Skips the `dict_read` for `sender_address`** — there is no check that a contract is deployed at `sender_address`.
5. **Directly writes `compiled_class_hash` into `contract_class_changes`** — the mapping `class_hash → compiled_class_hash` is committed to state. [2](#0-1) 

In the normal declare flow (lines 778–828), all five of these steps are mandatory. The BOOTSTRAP branch silently bypasses all of them.

The `class_hash` **is** verified to be a valid Sierra class hash (via `finalize_class_hash`, which runs before the BOOTSTRAP check): [3](#0-2) 

However, the `compiled_class_hash` (the CASM hash) is **not** verified against the `class_hash` in the BOOTSTRAP path — only `assert_not_zero(compiled_class_hash)` is checked. In the normal path, the signature over the transaction hash (which commits to both `class_hash` and `compiled_class_hash`) provides this binding. The BOOTSTRAP path removes that binding entirely.

The four triggering conditions are all fully attacker-controlled fields of a declare transaction:

| Condition | Attacker control |
|---|---|
| `sender_address == 'BOOTSTRAP'` (felt `0x424F4F545354524150`) | Set in transaction |
| `tx_info.nonce == 0` | Set in transaction |
| `tx_info.version == 3` | Set in transaction |
| All resource bounds `max_amount * max_price == 0` | Set in transaction | [4](#0-3) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

An attacker who successfully triggers the BOOTSTRAP path can register an attacker-chosen `compiled_class_hash` for a legitimate Sierra `class_hash`. When `execute_entry_point` resolves a class for execution, it looks up `compiled_class_hash` from `contract_class_changes` and then finds the actual CASM bytecode via `compiled_class_facts`: [5](#0-4) 

If the attacker supplies a `compiled_class_hash` that maps to malicious CASM (included in the block's `compiled_class_facts` bundle), any contract subsequently deployed under the hijacked `class_hash` will execute the attacker's CASM. This CASM can drain the contract's token balances, redirect transfers, or corrupt storage — constituting direct loss of funds for users of those contracts.

The `prev_value=0` guard prevents overwriting an already-declared class, but it does not prevent declaring a brand-new class with a mismatched or malicious `compiled_class_hash` before any legitimate declaration occurs.

---

### Likelihood Explanation

**High.** All four triggering conditions are trivially satisfiable by any transaction sender. No privileged key, operator access, or hash-function break is required. The felt encoding of `'BOOTSTRAP'` (`0x424F4F545354524150`) is a known constant that any user can place in the `sender_address` field of a declare transaction. The sequencer's OS execution is the authoritative path for proof generation; if the OS accepts the transaction, the resulting state transition is provably valid on L1.

---

### Recommendation

Remove the BOOTSTRAP special-case branch entirely from the production OS, or gate it behind a Cairo-level constraint that cannot be satisfied by an unprivileged user (e.g., require `sender_address` to equal a hard-coded, privileged OS-reserved address that is provably undeployable by users, and add an explicit Cairo `assert` — not merely a hint — enforcing this). If bootstrapping is genuinely needed, it should be handled as a genesis-block mechanism outside the normal transaction execution path, not as a runtime bypass reachable by any declare transaction.

---

### Proof of Concept

1. Attacker crafts malicious CASM `M` and computes `compiled_class_hash_M = hash(M)`.
2. Attacker picks any valid Sierra class `S` (with `class_hash_S` not yet declared on-chain) and computes its component hashes to satisfy `finalize_class_hash`.
3. Attacker submits a v3 declare transaction:
   - `sender_address = 0x424F4F545354524150` (`'BOOTSTRAP'` as a felt)
   - `nonce = 0`
   - `version = 3`
   - All resource bounds: `max_amount = 0`, `max_price_per_unit = 0`
   - `class_hash = class_hash_S`
   - `compiled_class_hash = compiled_class_hash_M`
4. The OS enters the BOOTSTRAP branch at line 764, skips `__validate_declare__`, `check_and_increment_nonce`, and `charge_fee`, and writes `contract_class_changes[class_hash_S] = compiled_class_hash_M`.
5. A victim deploys a contract using `class_hash_S` (believing it to be a legitimate class). The OS resolves `compiled_class_hash_M` and executes malicious CASM `M`.
6. Malicious CASM `M` drains the victim contract's funds. [1](#0-0) [6](#0-5)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

        }
