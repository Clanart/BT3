The vulnerability is real and confirmed. Here is the full analysis:

### Title
Missing Transaction Hash Validation in P2P Sync Allows Malicious Peer to Corrupt Stored Transaction Bodies — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `FullTransaction` messages from an untrusted P2P peer and stores the peer-supplied `transaction` body and `transaction_hash` directly into storage without ever verifying that `hash(transaction) == transaction_hash`. The code even contains an explicit developer TODO acknowledging this gap. The result is that a malicious peer can bind an arbitrary transaction body to any transaction hash in the node's persistent storage, corrupting every downstream consumer of that data.

### Finding Description

In `parse_data_for_block`, each `FullTransaction { transaction, transaction_output, transaction_hash }` received from the network is destructured and its fields are pushed directly into `block_body` without any hash validation:

```rust
block_body.transactions.push(transaction);
block_body.transaction_outputs.push(transaction_output);
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

The only guard applied is a count check: the loop runs until `current_transaction_len == header.n_transactions`. There is no call to `validate_transaction_hash` anywhere in this path. [2](#0-1) 

`validate_transaction_hash` exists and is capable of detecting this mismatch: [3](#0-2) 

The resulting `(BlockBody, BlockNumber)` is then written to storage via `write_to_storage` → `append_body` → `write_transactions`. Inside `write_transactions`, the peer-supplied `tx` body is serialized to the mmap file via `file_handlers.append_transaction(tx)`, and the peer-supplied `tx_hash` is inserted into the `transaction_hash_to_idx` index — with no cross-check between the two: [4](#0-3) 

The header sync path (`HeaderStreamBuilder::parse_data_for_block`) does not verify the header's `transaction_commitment` against the actual transaction hashes received, so the peer can also supply arbitrary `transaction_hash` values that do not correspond to the header's commitment: [5](#0-4) 

### Impact Explanation

**Impact: High** (per audit scope — "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" and "RPC execution, tracing, simulation returns an authoritative-looking wrong value").

After the corrupt write, the storage invariant `hash(stored_tx_body) == stored_tx_hash` is broken for every transaction the attacker controls. Concretely:

- `starknet_getTransactionByHash(H)` returns the attacker-chosen body `T_evil` instead of the real transaction.
- `starknet_traceTransaction(H)` re-executes `T_evil`, producing a wrong trace with wrong calldata, wrong signature, wrong resource bounds, and wrong signer.
- `starknet_simulateTransactions` and fee estimation over the corrupted block return wrong values.
- The `transaction_hash_to_idx` index maps `H → TransactionIndex(block_number, offset)` correctly, so the lookup succeeds and the wrong body is served with full authority.

The actual chain state is not affected (state diffs are synced separately), so this is High rather than Critical.

### Likelihood Explanation

Any peer that the syncing node connects to can trigger this. No special privileges are required — the P2P sync protocol is designed to accept data from arbitrary network peers. The attack requires only that the peer respond to a `TransactionQuery` with crafted `FullTransaction` messages. The count guard (`n_transactions`) is trivially satisfied by sending the correct number of messages.

### Recommendation

Call `validate_transaction_hash` on each received `(transaction, transaction_hash)` pair before pushing to `block_body`, and return `ParseDataError::BadPeer` on mismatch. The function signature and chain-id are already available in the storage reader context. Additionally, after all transactions for a block are collected, verify that the Merkle/Patricia commitment over the collected `transaction_hashes` matches the `transaction_commitment` stored in the block header.

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }
