### Title
Stale Balance Check in `VerifyEthAccount` and `CheckEthCanTransfer` Permits Batched Transactions with Insufficient Total Balance to Pass Ante Handler — (`File: ante/eth.go`)

### Summary
`VerifyEthAccount` and `CheckEthCanTransfer` each loop over all `MsgEthereumTx` messages in a batched Cosmos transaction and validate each message's cost/value against the sender's on-chain balance. Neither function deducts the cost between iterations, so every message is validated against the same original (stale) balance. A sender with balance `B` can batch `N` messages each costing `C` where `B >= C` but `B < N*C`, and all per-message checks pass. The ante handler admits the batch; messages that exhaust the balance fail at the EVM execution layer and are committed to the block as failed transactions — an outcome the ante handler was supposed to prevent.

### Finding Description

In `ante/eth.go`, `VerifyEthAccount` iterates over every message and reads the on-chain balance fresh each time, but never subtracts the cost of the current message before moving to the next:

```go
for _, msg := range tx.GetMsgs() {
    // ...
    balance := evmKeeper.GetBalance(ctx, from, evmDenom)          // always reads original balance
    if err := keeper.CheckSenderBalance(..., ethTx); err != nil { // checks balance >= fee+value
        return ...
    }
}
```

`CheckEthCanTransfer`, which runs immediately after, has the same pattern for the value component:

```go
for _, msg := range tx.GetMsgs() {
    // ...
    if value.Sign() > 0 && !canTransfer(ctx, evmKeeper, evmParams.EvmDenom, from, value) {
        return errorsmod.Wrapf(...)
    }
}
```

`canTransfer` reads the live balance from the store each call but the store is never updated between ante-handler iterations (no deduction happens until `CheckEthGasConsume`, which runs later and only deducts fees, not values).

The ante-handler execution order in `evmd/ante/handler_options.go` is:
1. `VerifyEthAccount` — checks `balance >= fee + value` per message (stale)
2. `CheckEthCanTransfer` — checks `balance >= value` per message (stale)
3. `CheckEthGasConsume` — deducts fees cumulatively (actual store write)
4. `CheckAndSetEthSenderNonce`

Because steps 1 and 2 never update the balance between messages, a batch of `N` messages from the same sender passes both checks as long as each individual message cost is within the balance, regardless of the total.

### Impact Explanation

An attacker submits a single Cosmos transaction batching multiple `MsgEthereumTx` messages from the same EOA where:
- Each individual `fee + value ≤ balance` (so `VerifyEthAccount` passes per-message)
- Each individual `value ≤ balance` (so `CheckEthCanTransfer` passes per-message)
- Total fees ≤ balance (so `CheckEthGasConsume` passes and deducts all fees)
- Total `fee + value > balance` (the batch is economically invalid)

The ante handler admits the batch. During EVM execution, early messages succeed and drain the balance; later messages fail at the EVM level with an insufficient-funds VM error. These failed messages are committed to the block as failed transactions, consuming block space and gas that the sender was not entitled to spend. This constitutes an ante handler bug that permits invalid transactions to commit.

### Likelihood Explanation

Batched EVM transactions are a supported, documented Cosmos-SDK-specific feature in Ethermint (see `docs/architecture/adr-003-batch-evm-transactions.md` and `tests/integration_tests/test_batch.py`). Any unprivileged user can construct a batched transaction. The balance condition (each individual cost ≤ balance < total cost) is easy to engineer deliberately. Likelihood is **Medium**.

### Recommendation

In `VerifyEthAccount`, maintain a per-sender running total of costs across messages and validate the cumulative total against the balance rather than checking each message independently:

```go
cumulativeCost := make(map[string]*big.Int)
for _, msg := range tx.GetMsgs() {
    // ...
    from := msgEthTx.GetFrom()
    cost := ethTx.Cost()
    key := from.String()
    if cumulativeCost[key] == nil {
        cumulativeCost[key] = new(big.Int)
    }
    cumulativeCost[key].Add(cumulativeCost[key], cost)
    balance := evmKeeper.GetBalance(ctx, from, evmDenom)
    if balance.ToBig().Cmp(cumulativeCost[key]) < 0 {
        return errorsmod.Wrapf(errortypes.ErrInsufficientFunds, ...)
    }
}
```

Apply the same cumulative-value tracking in `CheckEthCanTransfer`.

### Proof of Concept

**Setup**: Sender balance = 100 wei, gasPrice = 1, gasLimit = 10 (fee = 10 wei per message), value = 60 wei per message. Batch two `MsgEthereumTx` messages from the same sender.

**Ante handler walk-through**:

1. `VerifyEthAccount` msg1: `GetBalance` → 100; `CheckSenderBalance(100, cost=70)` → 100 ≥ 70 ✓
2. `VerifyEthAccount` msg2: `GetBalance` → 100 (unchanged); `CheckSenderBalance(100, cost=70)` → 100 ≥ 70 ✓
3. `CheckEthCanTransfer` msg1: `GetBalance` → 100; 100 ≥ 60 ✓
4. `CheckEthCanTransfer` msg2: `GetBalance` → 100 (unchanged); 100 ≥ 60 ✓
5. `CheckEthGasConsume` msg1: deduct fee=10 → balance=90 ✓
6. `CheckEthGasConsume` msg2: deduct fee=10 → balance=80 ✓

Ante handler returns success. EVM execution:
- msg1 executes: transfers 60 wei → balance = 20 wei
- msg2 executes: attempts to transfer 60 wei → EVM error (insufficient funds), committed as failed tx
