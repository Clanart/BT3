### Title
`batch_insert` Silently Accepts Duplicate Keys, Corrupting DataLayer Merkle Tree Root and Invalidating Proofs - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key guard that `MerkleBlob::insert` enforces. When a batch of three or more items contains a key already present in the tree (or a key that appears twice within the same batch), the function silently writes a second leaf node for that key directly into the blob without error. This leaves the tree in a structurally corrupt state: the ghost leaf is wired into the tree and contributes to the Merkle root hash, but the in-memory cache points only to the second copy. The resulting root hash is wrong, proofs of inclusion/exclusion computed from it are invalid, and `check_integrity` will detect the cache/blob disagreement — but only if it is called.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicate keys at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` uses `insert` only for the first two items (when `leaf_count <= 1`). For every subsequent item it skips directly to `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { key, value, hash, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which does a plain `HashMap::insert` — silently overwriting the existing key-to-index mapping:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);   // overwrites silently
}
``` [3](#0-2) 

After the batch completes, the subtree built from all batch items (including the duplicate leaf) is attached to the main tree via `insert_subtree_at_key`. Both the original and the duplicate leaf are now physically present in the blob and wired into the tree, so both contribute to the Merkle root hash. The cache, however, only records the second copy's index. The first copy is a ghost node: unreachable through the cache, but still hashed into the root. [4](#0-3) 

The Python binding `py_batch_insert` exposes this path directly, checking only that the lengths of the key/value and hash vectors match — no duplicate-key validation:

```rust
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    if keys_values.len() != hashes.len() { ... }
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

---

### Impact Explanation

The corrupted Merkle root hash is the authoritative commitment to the DataLayer store's state. Any proof of inclusion or exclusion derived from it is wrong:

- A key that was overwritten in the cache can still be "proven" present (the cache finds the second copy), but the root hash it is proven against includes the ghost first copy — so the proof verifies against a root that does not correspond to the intended logical state.
- A key that exists only as a ghost (first copy, cache-evicted) cannot be proven present at all, even though it is hashed into the root — effectively a silent data loss analogous to the original report's lost rewards.
- `check_integrity` will catch the cache/blob disagreement, but it is not called automatically after `batch_insert`. [6](#0-5) 

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The attacker-controlled entry path is `py_batch_insert` (Python binding) or `batch_insert` (Rust API). Any caller that constructs a batch containing a repeated key — whether by mistake or by deliberate crafting — triggers the corruption silently. No privileged role, leaked key, or network-level attack is required. The only precondition is the ability to call `batch_insert` with a batch of three or more items where at least one key is a duplicate.

---

### Recommendation

Add a duplicate-key check at the top of `batch_insert` before any items are written to the blob. The simplest correct fix mirrors what `insert` already does: check `block_status_cache.contains_key(key)` for every item in the batch before beginning any insertions, and return `Err(Error::KeyAlreadyPresent())` on the first collision. Alternatively, build a temporary `HashSet` of keys seen within the batch itself and reject the entire batch if any intra-batch duplicate is detected.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 items so leaf_count > 1 before batch_insert runs.
    // This forces batch_insert to take the unchecked fast path for all batch items.
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &sha256_num(i), InsertLocation::Auto {}).unwrap();
    }

    // Batch with a duplicate key (KeyId(99) appears twice).
    let batch = vec![
        ((KeyId(99), ValueId(1)), sha256_num(99)),
        ((KeyId(100), ValueId(2)), sha256_num(100)),
        ((KeyId(99), ValueId(3)), sha256_num(999)), // duplicate key, different hash
    ];

    // Succeeds — no error returned despite duplicate key.
    blob.batch_insert(batch).unwrap();

    blob.calculate_lazy_hashes().unwrap();

    // check_integrity will fail: leaf count in blob != cache length,
    // because the ghost leaf is in the blob but the cache only tracks one copy.
    blob.check_integrity().expect_err("tree is corrupt");
}
```

The `batch_insert` call returns `Ok(())` despite the duplicate key. The subsequent `check_integrity` call exposes the corruption: the blob contains more leaf nodes than the cache tracks, and the Merkle root hash is computed over a tree that includes the ghost leaf. [7](#0-6) [8](#0-7)
