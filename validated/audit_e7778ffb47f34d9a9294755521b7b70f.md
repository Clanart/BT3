### Title
Unrestricted `receive()` Permanently Locks Any Directly Sent ETH With No Recovery Path - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.sol` contains a completely unrestricted `receive() external payable {}` function with no sender check and no corresponding ETH rescue/withdrawal mechanism. Any ETH sent directly to the contract — whether by a user error, an external integration, or a script — is permanently irrecoverable.

### Finding Description
The base `OmniBridge` contract declares:

```solidity
receive() external payable {}
```

at line 574 with no guard whatsoever. Unlike the reported pattern (which at least checked a boolean flag), this implementation accepts ETH from any caller unconditionally.

The contract handles native ETH bridging: users call `initTransfer(address(0), amount, ...)` with `msg.value = amount + nativeFee`, locking ETH in the contract. That ETH is later disbursed via `finTransfer` when a valid MPC-signed message arrives. However, `finTransfer` is the **only** ETH-outflow path, and it requires a valid ECDSA signature from `nearBridgeDerivedAddress`. There is no admin rescue function, no `withdrawETH`, and no sweep mechanism anywhere in the contract or its `OmniBridgeWormhole` subclass.

Consequently, any ETH that enters the contract through `receive()` — rather than through the tracked `initTransfer` flow — is permanently stuck. The contract's ETH balance grows, but the excess is never attributable to any bridge transfer and can never be released.

### Impact Explanation
ETH sent directly to the contract address (e.g., by a user who mistakes the bridge contract for a deposit address, by a DeFi integration that forwards ETH, or by a script that calls `address(bridge).transfer(amount)`) is permanently frozen. There is no privileged or unprivileged path to recover it. This constitutes an irrecoverable lock of user funds in the bridge contract.

Allowed impact matched: **Critical — Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows.**

### Likelihood Explanation
- Bridge contracts that handle native ETH are commonly targeted by direct ETH sends from wallets, scripts, and integrations.

### Title
Unrestricted `receive()` Function Permanently Locks ETH in OmniBridge Contract - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.sol` contains a completely unrestricted `receive() external payable {}` function with no sender check and no ETH rescue/withdrawal mechanism. Any ETH sent directly to the contract — whether by a user mistake, an external integration, or a protocol-level refund — is permanently locked with no recovery path.

### Finding Description
The `OmniBridge` contract at line 574 declares:

```solidity
receive() external payable {}
``` [1](#0-0) 

This function imposes zero restrictions on the sender or the amount. Unlike the original report's contract (which at least checked a boolean flag), `OmniBridge` accepts all ETH unconditionally.

The contract's only ETH-outflow path is inside `finTransfer`, which sends ETH to a recipient only when a valid MPC-signed message specifies `payload.tokenAddress == address(0)`:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [2](#0-1) 

There is no admin rescue function, no `withdrawETH`, and no other ETH-sending path in the contract. ETH received via `receive()` outside the normal `initTransfer` → `finTransfer` accounting cycle has no corresponding signed message that would ever release it, so it is permanently frozen.

The `initTransfer` path for native ETH enforces exact accounting — `extensionValue = msg.value - amount - nativeFee` must equal the Wormhole fee, and the base contract reverts if `value != 0` — so there is no overflow path through the normal bridge flow either: [3](#0-2) [4](#0-3) 

### Impact Explanation
Any ETH sent directly to the contract address — by a user who mistakes the bridge contract for a deposit address, by an external DeFi integration, or by a protocol-level ETH refund — is permanently irrecoverable. The contract holds no mechanism to release such ETH. This constitutes a **permanent, irrecoverable lock of user or protocol funds** in the bridge contract.

### Likelihood Explanation
The `OmniBridge` contract is a publicly deployed bridge that handles native ETH. It is realistic that:
- Users or wallets send ETH directly to the contract address (common mistake with bridge contracts).
- External integrations or aggregators forward ETH to the contract outside the `initTransfer` calldata path.
- Protocol-level ETH (e.g., Wormhole fee refunds, failed call returns) lands at the contract address.

All such ETH is silently accepted and permanently frozen.

### Recommendation
Either:
1. **Remove `receive()`** entirely if the contract does not need to accept ETH outside of its `payable` function calls (which already handle ETH correctly).
2. **Restrict the sender**: add a check such as `require(msg.sender == address(0), ...)` or revert unconditionally, since all legitimate ETH entry points are already `payable` named functions.
3. **Add an admin ETH rescue function** that allows the `DEFAULT_ADMIN_ROLE` to recover ETH that does not correspond to any pending bridge transfer.

### Proof of Concept
1. Deploy `OmniBridge` (or `OmniBridgeWormhole`).
2. Send ETH directly to the contract address from any EOA:
   ```solidity
   (bool ok,) = address(omniBridge).call{value: 1 ether}("");
   require(ok); // succeeds — receive() accepts it
   ```
3. Observe that `address(omniBridge).balance` increased by 1 ETH.
4. Attempt to recover the ETH: no function exists to do so. The ETH is permanently locked. [1](#0-0)
