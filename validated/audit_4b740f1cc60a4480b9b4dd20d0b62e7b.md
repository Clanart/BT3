### Title
`ProofOfInclusion::valid()` Does Not Verify Against an External Root Hash — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only checks the internal self-consistency of a proof's layer chain. The final guard `existing_hash == self.root_hash()` is a tautology: `root_hash()` returns `last.combined_hash`, which is exactly the value `existing_hash` was set to in the last loop iteration (after the loop already asserted `calculated_hash == layer.combined_hash`). No external, trusted root hash is ever compared. Any attacker who can supply a `ProofOfInclusion` value — via the Streamable deserializer or the Python binding — can forge a proof for an arbitrary key-value pair and have `valid()` return `true`.

### Finding Description

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← taken directly from the proof itself
    } else {
        self.node_hash
    }
}
```

`valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;   // existing_hash ← layer.combined_hash
    }
    existing_hash == self.root_hash()      // ← always true: both sides == last.combined_hash
}
```

After the loop, `existing_hash` equals the last `layer.combined_hash` (the loop already enforced this). `self.root_hash()` also returns the last `layer.combined_hash`. The final comparison is therefore always `true` when the loop passes, regardless of what root the proof actually commits to.

The struct is `Streamable` and fully exposed to Python:

```rust
#[derive(Clone, Debug, std::hash::Hash, Eq, PartialEq, Streamable)]
pub struct ProofOfInclusion {
    pub node_hash: Hash,
    pub layers: Vec<ProofOfInclusionLayer>,
}
```

An attacker can deserialize an arbitrary `ProofOfInclusion` from bytes (via `from_bytes` / `from_bytes_unchecked`) and call `proof.valid()` (or `py_valid()` from Python). As long as the layers are internally chained correctly — a trivial construction — `valid()` returns `true` for any claimed `node_hash` and any claimed root.

### Impact Explanation

Any caller that relies solely on `proof.valid()` to decide whether a key-value pair is present in a DataLayer tree can be deceived. An attacker supplies a forged `ProofOfInclusion` whose `node_hash` encodes an arbitrary (key, value) pair and whose layers form a self-consistent chain to an attacker-chosen root. `valid()` returns `true`. The verifier accepts the forged state as proven. This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

### Likelihood Explanation

The `ProofOfInclusion` type is Streamable, is exposed as a first-class Python type with `from_bytes`, `from_bytes_unchecked`, and `valid()` bindings, and is the only validation API the struct provides. Any Python consumer that calls `proof.valid()` without separately comparing `proof.root_hash()` against a trusted, externally-anchored root is vulnerable. The construction of a passing forged proof requires only arithmetic over the hash function — no key material or privileged access.

### Recommendation

`valid()` must accept a trusted external root hash and compare the computed root against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against the EXTERNAL trusted root
}
```

The no-argument `valid()` should either be removed or clearly documented as an internal-consistency-only helper that must never be used as a security gate.

### Proof of Concept

Construct a single-layer `ProofOfInclusion` for a fake leaf:

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker-chosen fake leaf hash (any key-value pair)
fake_node_hash = bytes([0xAA] * 32)
# Attacker-chosen sibling hash
sibling_hash   = bytes([0xBB] * 32)

# Compute combined_hash using the same calculate_internal_hash logic
# (left-side: node_hash, right-side: sibling_hash)
combined = hashlib.sha256(b"\x01" + fake_node_hash + sibling_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=1,          # sibling is on the right
    other_hash=sibling_hash,
    combined_hash=combined,     # self-consistent: valid() will accept this
)

proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])
assert proof.valid()            # True — forged proof accepted
# proof.root_hash() == combined, which is NOT the real tree root
```
