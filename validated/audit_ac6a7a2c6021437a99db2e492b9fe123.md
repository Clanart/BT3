### Title
`sign_transfer` Permanently Locks User Funds When `normalize_amount(amount - fee) == 0` Due to Decimal Truncation — (`near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` validates only that `fee < amount` (in NEAR's native decimal representation), but `sign_transfer` later applies `normalize_amount(amount - fee, decimals)` and requires the result to be `> 0`. For tokens whose `origin_decimals > decimals`, the floor-division in `normalize_amount` can truncate a non-zero `amount - fee` to zero, causing `sign_transfer` to always revert with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Because there is no cancel or refund path for a pending transfer, the user's tokens are permanently locked or burned in the bridge.

---

### Finding Description

**Step 1 — `init_transfer` accepts the transfer.**

`init_transfer` (called via `ft_transfer_call`) stores the transfer message and burns/locks the user's tokens after passing only one fee check:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This check is performed in NEAR's native decimal representation (e.g., 24 decimals for wNEAR). Tokens are then burned or locked: [2](#0-1) 

**Step 2 — `sign_transfer` applies decimal normalization and requires `> 0`.**

When a relayer later calls `sign_transfer`, the net amount is computed and normalized to the destination chain's decimal precision:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

`normalize_amount` uses floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

`amount_without_fee` is simply `amount - fee`: [5](#0-4) 

**Step 3 — The stuck state.**

If `amount - fee < 10^(origin_decimals - decimals)`, then `normalize_amount` returns `0`, and `sign_transfer` always panics. The transfer is stored in `pending_transfers` indefinitely with no user-accessible cancellation path.

`update_transfer_fee` only allows the sender to *increase* the fee (never decrease it), which makes the situation worse, not better:

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [6](#0-5) 

There is no `cancel_transfer`, `withdraw_transfer`, or user-facing emergency refund function in the contract.

---

### Impact Explanation

**Permanent freezing of user funds.** Once `init_transfer` succeeds, the user's tokens are burned (for deployed/bridged tokens) or locked in the bridge (for native tokens). If `normalize_amount(amount - fee) == 0`, `sign_transfer` will revert on every call, and there is no on-chain path for the user to recover their funds. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

The condition is reachable without malicious intent whenever:

1. A token has `origin_decimals > decimals` (e.g., a NEAR-native token with 24 origin decimals bridging to an EVM chain registered with 18 decimals — divisor = `10^6`).
2. The user sets `amount - fee < 10^(origin_decimals - decimals)`.

**Concrete example** (USDC-like token, `origin_decimals = 6`, `decimals = 0`, divisor = `10^6`):
- `amount = 500_000` (0.5 USDC in base units), `fee = 1`
- `fee < amount` → passes `init_transfer`
- `amount_without_fee = 499_999`
- `normalize_amount(499_999) = 499_999 / 1_000_000 = 0` → `sign_transfer` always reverts
- User's 0.5 USDC is permanently locked

The `Decimals` struct confirms `origin_decimals` and `decimals` can differ: [7](#0-6) 

Token registration stores both values: [8](#0-7) 

---

### Recommendation

Add a validation in `init_transfer` (before locking/burning tokens) that checks the post-fee, post-normalization amount is non-zero:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the fix suggested in the external report: validate the constraint that will be enforced later *before* committing the state change (locking/burning tokens).

---

### Proof of Concept

1. Register a token with `origin_decimals = 6`, `decimals = 0` (divisor = `10^6`).
2. Call `ft_transfer_call` → `init_transfer` with `amount = 500_000`, `fee = 1`.
   - Check `fee < amount` → `1 < 500_000` → passes.
   - Tokens are locked in the bridge.
3. Call `sign_transfer` for the resulting `transfer_id`.
   - `amount_without_fee() = 499_999`
   - `normalize_amount(499_999, Decimals { decimals: 0, origin_decimals: 6 }) = 499_999 / 1_000_000 = 0`
   - Panics: `ERR_INVALID_AMOUNT_TO_TRANSFER`
4. Call `update_transfer_fee` to try to reduce the fee — rejected because `fee.fee >= current_fee.fee` is required.
5. No other user-callable function exists to recover the locked tokens. Funds are permanently frozen.
