### Title
`update_l2_gas_price` Called Before `record_fee_proposal` Produces Stale `fee_actual` Floor in `next_l2_gas_price` ŌĆö (`crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

In `finalize_decision`, `update_l2_gas_price` is invoked before `record_fee_proposal`. Because `update_l2_gas_price` internally calls `compute_fee_actual` over the in-memory `fee_proposals_window`, and the current block's `fee_proposal_fri` has not yet been inserted into that window, the `fee_actual` floor used to compute `next_l2_gas_price` is one block stale. The resulting wrong `next_l2_gas_price` is persisted to storage, forwarded in the cende blob, and returned by RPC ŌĆö all as an authoritative value.

---

### Finding Description

In `finalize_decision` the two calls appear in this order:

```rust
self.update_l2_gas_price(height, l2_gas_used);          // line 517
self.record_fee_proposal(height, init.fee_proposal_fri); // line 518
``` [1](#0-0) 

`update_l2_gas_price` delegates to `calculate_next_l2_gas_price`, which calls `compute_fee_actual` over `self.fee_proposals_window`:

```rust
fn calculate_next_l2_gas_price(&self, height: BlockNumber, l2_gas_used: GasAmount) -> GasPrice {
    let fee_actual = compute_fee_actual(
        &self.fee_proposals_window,
        height,
        VersionedConstants::latest_constants().fee_proposal_window_size,
    );
    ...
}
``` [2](#0-1) 

`compute_fee_actual` for block `height` reads the window `[height ŌłÆ window_size, height ŌłÆ 1]`:

```rust
for source_height in (start..height.0).map(BlockNumber) {
    match fee_proposals_window.get(&source_height) { ... }
}
``` [3](#0-2) 

Because `record_fee_proposal(height, ...)` has not yet run, the current block's `fee_proposal_fri` is absent from the window. The `fee_actual` returned is therefore the median of `[height ŌłÆ window_size, height ŌłÆ 1]` rather than the correct `[height ŌłÆ window_size + 1, height]` that will be used when the next block is built.

This stale `fee_actual` is then used as the `effective_min` floor in the EIP-1559 adjustment:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [4](#0-3) 

The wrong `next_l2_gas_price` is then written into three authoritative outputs:

**1. Storage / state-sync block header** ŌĆö `update_state_sync_with_new_block` stores `self.l2_gas_price` (the freshly-updated wrong value) as `next_l2_gas_price` in `BlockHeaderWithoutHash`:

```rust
next_l2_gas_price: self.l2_gas_price,
``` [5](#0-4) 

This is serialized to MDBX via `StorageBlockHeader`: [6](#0-5) 

**2. Cende blob** ŌĆö `FeeMarketInfo.next_l2_gas_price` in `prepare_blob_for_next_height` carries the same wrong value:

```rust
fee_market_info: FeeMarketInfo {
    l2_gas_consumed: l2_gas_used,
    next_l2_gas_price: self.l2_gas_price,
},
``` [7](#0-6) 

**3. Syncing nodes** ŌĆö `try_sync` reads `next_l2_gas_price` from the stored header and sets `self.l2_gas_price` from it, propagating the wrong floor into the next block's gas price:

```rust
self.l2_gas_price = max(
    sync_block.block_header_without_hash.next_l2_gas_price,
    VersionedConstants::latest_constants().min_gas_price,
);
``` [8](#0-7) 

The `BlockHeaderWithoutHash` struct itself carries a `TODO` acknowledging `fee_proposal_fri` is not yet in the block hash, meaning the only binding of `fee_proposal_fri` to a commitment is through the proposal commitment (`Poseidon(partial_block_hash, fee_proposal_fri)`): [9](#0-8) 

The P2P wire format also explicitly warns the field is not part of the block hash and must be trusted: [10](#0-9) 

---

### Impact Explanation

**Impact: High ŌĆö RPC execution, fee estimation, and pending view return an authoritative-looking wrong value.**

`next_l2_gas_price` is the canonical value that:
- RPC clients query to estimate fees for the next block
- Syncing nodes use as the starting `l2_gas_price` for the next block
- The cende blob carries to the centralized prover pipeline

When the current block's `fee_proposal_fri` is higher than the oldest entry being rotated out of the window (a common scenario during fee increases), the stale `fee_actual` floor is lower than the correct one. `next_l2_gas_price` can therefore be computed below the floor that will actually be enforced when the next block is built, causing fee estimation to understate the true minimum gas price. Users who rely on `next_l2_gas_price` for fee bounds will submit transactions with insufficient resource bounds, leading to rejections or unexpected fee charges.

---

### Likelihood Explanation

**Likelihood: High.** The ordering bug fires on every block finalization where the current block's `fee_proposal_fri` shifts the window median relative to the block being rotated out. This is the normal operating condition once the network is past the bootstrap phase (`height Ōēź window_size`). No special attacker action is required; the bug is triggered by ordinary block production.

---

### Recommendation

Swap the two calls so the current block's `fee_proposal_fri` is recorded before the next gas price is computed:

```rust
// In finalize_decision:
self.record_fee_proposal(height, init.fee_proposal_fri); // record first
self.update_l2_gas_price(height, l2_gas_used);           // then compute with complete window
```

This ensures `compute_fee_actual` inside `calculate_next_l2_gas_price` sees the window `[height ŌłÆ window_size + 1, height]`, which is the same window that will be used when building block `height + 1`, making `next_l2_gas_price` consistent with the actual floor that will be enforced.

---

### Proof of Concept

Assume `window_size = 10`, current height = 100, and fee proposals for heights 90ŌĆō99 are all `5 gwei`. The current block (height 100) has `fee_proposal_fri = 20 gwei`.

**Current (buggy) execution:**
1. `update_l2_gas_price(100, gas_used)` is called.
2. `compute_fee_actual` reads heights 90ŌĆō99 ŌåÆ median = `5 gwei`.
3. `effective_min = max(config_min, 5 gwei)`.
4. EIP-1559 adjustment runs with floor `5 gwei`; `next_l2_gas_price` may be as low as `5 gwei`.
5. `record_fee_proposal(100, 20 gwei)` inserts height 100 into the window.
6. `next_l2_gas_price = 5 gwei` is stored in block 100's header and the cende blob.

**When block 101 is built:**
1. `compute_fee_actual(101)` reads heights 91ŌĆō100 ŌåÆ median shifts toward `20 gwei` (e.g., `5 gwei` still if only one entry changed, but the floor is now higher).
2. The actual `effective_min` for block 101 is higher than the `5 gwei` floor used to compute `next_l2_gas_price`.
3. RPC clients that queried `next_l2_gas_price` from block 100 received `5 gwei` but the actual minimum enforced in block 101 is higher ŌĆö fee estimation is wrong.

**Correct execution (after fix):**
1. `record_fee_proposal(100, 20 gwei)` inserts height 100 first.
2. `compute_fee_actual(100)` reads heights 91ŌĆō100 ŌåÆ median includes `20 gwei`.
3. `effective_min` is correctly elevated; `next_l2_gas_price` reflects the true floor.
4. RPC clients receive an accurate `next_l2_gas_price`.
