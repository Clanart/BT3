### Title
Integer Truncation in `NewDynamicFeeChecker` Collapses `feeCap` to Zero When `baseFee = 0`, Allowing Cosmos Transactions to Execute With Zero Fee Deduction - (`File: ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` computes the per-gas fee cap for Cosmos transactions using integer division (`fee.Quo(gas)`). When the on-chain `baseFee` has been driven to zero — which is possible whenever `MinGasPrice` is a fractional value less than 1, because `CalculateBaseFee` uses `MinGasPrice.TruncateInt()` as the lower bound — the truncated `feeCap = 0` passes the `feeCap < baseFee` guard, and `EffectiveGasPrice(0, 0, …) = 0`, so `effectiveFee = 0`. The `DeductFeeDecorator` then deducts nothing from the sender's account. The `MinGasPriceDecorator` that runs earlier only checks the *declared* fee field, not the effective deduction, so the two checks are decoupled and the bypass is complete.

---

### Finding Description

**Root cause 1 — `CalculateBaseFee` truncates `MinGasPrice` to zero:** [1](#0-0) 

```go
minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
```

`MinGasPrice` is a `sdkmath.LegacyDec` (18 decimal places). Any value in the range `(0, 1)` — e.g. `0.5`, `0.001`, `0.000001` — truncates to integer `0`. The base fee lower bound therefore becomes `0`, and the EIP-1559 adjustment loop can drive `baseFee` all the way to `0` during periods of low block utilisation.

**Root cause 2 — `NewDynamicFeeChecker` derives `feeCap` by integer division:** [2](#0-1) 

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // integer truncation
baseFeeInt := sdkmath.NewIntFromBigInt(baseFee)

if feeCap.LT(baseFeeInt) {                          // 0 < 0 → false when baseFee=0
    return nil, 0, errorsmod.Wrapf(...)
}
```

`sdkmath.Int.Quo` is truncating integer division. When `fee < gas` (which is always true when `minGasPrice < 1`, because `ceil(minGasPrice * gas) < gas`), `feeCap` rounds down to `0`. With `baseFee = 0`, the guard `feeCap.LT(baseFeeInt)` evaluates to `0 < 0 = false` and does not reject the transaction.

**Root cause 3 — `EffectiveGasPrice` with both inputs zero returns zero:** [3](#0-2) 

```go
func EffectiveGasPrice(baseFee *big.Int, feeCap *big.Int, tipCap *big.Int) *big.Int {
    return ethermint.BigMin(new(big.Int).Add(tipCap, baseFee), feeCap)
```

With `baseFee = 0` and `feeCap = 0`, `EffectiveGasPrice(0, 0, maxPriorityPrice) = min(maxPriorityPrice + 0, 0) = 0`. [4](#0-3)

```

### Impact Explanation

Every Cosmos transaction routed through `newCosmosAnteHandler` with `ExtensionOptionDynamicFeeTx` (or without any extension option, since the same handler is used) pays **zero fees** to the fee collector module when `baseFee = 0`. The fee collector receives nothing; the sender's balance is not reduced. This is a systematic mis-accounting of transaction fees: the chain's fee revenue is zero regardless of the configured `MinGasPrice`, and the economic spam-prevention mechanism is completely neutralised. This matches the allowed High impact: *"fee market, ante handler bug that permits valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

The preconditions are:

1. **`MinGasPrice` is a fractional value `(0, 1)`** — extremely common. Cosmos chains routinely set `MinGasPrice` to values like `0.001`, `0.025`, or `0.5` in their smallest denomination. The `validateMinGasPrice` function only rejects negative values, not sub-integer ones. [8](#0-7) 

2. **`baseFee` has reached `0`** — guaranteed to happen eventually on any chain where `MinGasPrice < 1`, because `TruncateInt()` makes the lower bound `0`. On a lightly-used chain or testnet, the base fee decreases every block until it hits `0`.

3. **Attacker submits a Cosmos tx with `fee = ceil(minGasPrice * gas)` and `gas > fee`** — trivially satisfied when `minGasPrice < 1`, since `ceil(minGasPrice * gas) < gas`.

No special privileges, no governance access, no validator collusion required. Any unprivileged account can exploit this once `baseFee = 0`.

---

### Recommendation

**Short term (two independent fixes, either sufficient):**

1. **Use ceiling division for `feeCap`** in `NewDynamicFeeChecker` so that a non-zero declared fee always produces a non-zero `feeCap`:
   ```go
   // Replace:
   feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
   // With (ceiling division):
   feeCap := fee.Add(sdkmath.NewIntFromUint64(gas - 1)).Quo(sdkmath.NewIntFromUint64(gas))
   ```

2. **Use `RoundInt()` instead of `TruncateInt()`** in `CalculateBaseFee` so that a fractional `MinGasPrice` is rounded to `1` rather than `0`, preventing the base fee from reaching `0`:
   ```go
   // Replace:
   minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
   // With:
   minGasPrice := params.MinGasPrice.Ceil().RoundInt().BigInt()
   ``` [1](#0-0) 

**Long term:** Add a zero-value guard in `NewDynamicFeeChecker` that rejects any transaction where `effectiveFee = 0` but `minGasPrice > 0`. Add fuzz tests covering the interaction between fractional `MinGasPrice`, `baseFee` decay, and `feeCap` computation.

### Proof of Concept

**Setup:** Chain with `MinGasPrice = 0.5` (a `LegacyDec`), London hardfork enabled, `baseFee` has decayed to `0` (guaranteed by `TruncateInt()` lower bound).

**Transaction:** Cosmos tx with `ExtensionOptionDynamicFeeTx`, `gas = 21000`, `fee = 10500 atevmos` (= `ceil(0.5 * 21000)`).

**Trace through ante handler chain:**

```
MinGasPriceDecorator:
  requiredFees = ceil(0.5 * 21000) = 10500
  feeCoins     = 10500
  10500 >= 10500 → PASS

NewDynamicFeeChecker (called inside DeductFeeDecorator):
  baseFee    = 0          (stored params, driven to 0 by TruncateInt lower bound)
  feeCap     = 10500 / 21000 = 0   (integer truncation, 10500 < 21000)
  0 < 0      → false → PASS guard
  effectivePrice = EffectiveGasPrice(0, 0, MaxInt64)
             = min(MaxInt64 + 0, 0) = 0
  effectiveFee   = 0 * 21000 = 0

DeductFeeDecorator.checkDeductFee:
  deducts 0 atevmos from sender
  fee collector receives 0 atevmos
```

**Result:** Transaction executes successfully. Sender's balance is unchanged. Fee collector receives nothing. The attack is repeatable for every subsequent transaction while `baseFee = 0`.
