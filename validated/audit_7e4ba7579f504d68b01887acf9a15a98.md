### Title
Chunk Producer Self-Endorses Own Chunk Without State Witness Validation — (`chain/client/src/stateless_validation/state_witness_producer.rs`)

---

### Summary

In `send_chunk_state_witness_to_chunk_validators`, when the chunk producer is also sampled as a chunk validator for the same chunk, the code explicitly bypasses state witness validation and immediately issues a self-endorsement. The self-endorsement stake is counted toward the 2/3 endorsement threshold without any independent verification of the chunk's correctness. This is the direct nearcore analog of the "bounty funder submitting to their own bounty" pattern: the entity that created the commitment (chunk producer) also acts as a validator of that commitment (chunk validator) without performing the required validation step.

---

### Finding Description

In `send_chunk_state_witness_to_chunk_validators` (`chain/client/src/stateless_validation/state_witness_producer.rs`), after producing a chunk and creating its state witness, the chunk producer checks whether it is also assigned as a chunk validator for that chunk: [1](#0-0) 

The comment at line 62 is explicit: **"Bypass state witness validation if we created state witness. Endorse the chunk immediately."** The code calls `send_chunk_endorsement_to_block_producers` and stores the endorsement in `chunk_endorsement_tracker` — the same path taken by honest validators who have completed full `validate_chunk_state_witness` execution — without running any of the following:

- `pre_validate_chunk_state_witness` (receipt proof verification, tx root check, etc.)
- `validate_chunk_state_witness` (full state transition re-execution)
- `validate_chunk_with_chunk_extra_and_roots` (the lighter alternative path)

The chunk validator assignment is produced by `get_chunk_validator_assignments`, which calls `epoch_info.sample_chunk_validators(height)`: [2](#0-1) 

`sample_chunk_validators` draws from `ValidatorMandates`, which is constructed from `all_validators` — a list that **includes chunk producers**: [3](#0-2) 

There is no exclusion of the chunk producer for the current slot from the chunk validator sample. Consequently, a chunk producer is regularly assigned as a chunk validator for the same chunk they produced, and the code at lines 57–72 of `state_witness_producer.rs` is a live, reachable path in production.

The endorsement produced by this path is structurally identical to one produced after full validation: it is signed with the validator's key, carries the correct `ChunkHash`, and passes `validate_chunk_endorsements_in_block`: [4](#0-3) 

Block producers collect it via `collect_chunk_endorsements` and count its stake toward the 2/3 endorsement threshold: [5](#0-4) 

---

### Impact Explanation

The endorsement threshold requires chunk validators whose cumulative stake exceeds 2/3 of the total assigned chunk-validator stake to independently verify the state witness before a chunk is considered endorsed. When the chunk producer self-endorses without validation, their mandate weight is subtracted from the independent-verification requirement.

Under `ValidatorMandates`, stake is distributed proportionally. A chunk producer holding, say, 20% of the total validator stake will receive approximately 20% of the chunk-validator mandate weight for any given shard/height. Their unvalidated self-endorsement therefore reduces the remaining independent stake needed from 2/3 to roughly 2/3 − 0.20 = 47%. A malicious chunk producer with large stake can produce an invalid chunk (e.g., with a corrupted state root or invalid receipts), self-endorse it without validation, and require only the remaining threshold from other validators — some of whom may be colluding or offline — to get the invalid chunk endorsed and included in a block.

The corrupted value is the `ChunkEndorsement` stored in `ChunkEndorsementTracker` under the chunk's `ChunkProductionKey`, carrying the chunk producer's signature over the `ChunkHash` of an invalid chunk, counted as validated stake. [6](#0-5) 

---

### Likelihood Explanation

The overlap between chunk producers and chunk validators is not an edge case. `ValidatorMandates::sample` draws from all validators proportionally to stake: [7](#0-6) 

High-stake validators — who are most likely to be chunk producers — also receive the most mandates and are therefore most likely to appear in the chunk validator assignment for any given shard/height. The self-endorsement path in `state_witness_producer.rs` is triggered every time this overlap occurs, which is a regular, deterministic outcome of the mandate sampling for large validators.

---

### Recommendation

Remove the self-endorsement bypass. When the chunk producer is also assigned as a chunk validator, it should process the state witness through the normal validation path (`start_validating_chunk` / `validate_chunk_state_witness`) before issuing an endorsement. The optimization of skipping validation for self-produced chunks breaks the independence invariant of the endorsement system and allows a malicious chunk producer to contribute unvalidated stake toward the endorsement threshold.

If the optimization is desired for performance, the chunk producer should at minimum run `validate_chunk_with_chunk_extra_and_roots` (which uses the locally stored `ChunkExtra` and is already used as a fast path in `start_validating_chunk`) before endorsing: [8](#0-7) 

This path performs real validation against committed state and does not require re-executing the full state transition.

---

### Proof of Concept

1. Validator `V` has high stake and is assigned as both chunk producer and chunk validator for shard `S` at height `H` (confirmed by `get_chunk_validator_assignments(...).contains(V.validator_id())`).
2. `V` calls `produce_chunk` and produces a chunk with a corrupted `state_root` or invalid receipts.
3. `V` calls `send_chunk_state_witness_to_chunk_validators`. The condition at line 57–60 of `state_witness_producer.rs` is true.
4. `V` calls `send_chunk_endorsement_to_block_producers` **without** calling `validate_chunk_state_witness`. The endorsement is stored in `chunk_endorsement_tracker` with `V`'s full mandate weight.
5. The state witness is distributed to other chunk validators. If enough of them are offline, colluding, or if `V`'s mandate weight is large enough that the remaining threshold is reachable with fewer honest validators, the invalid chunk accumulates sufficient endorsement stake.
6. A block producer calls `collect_chunk_endorsements`, finds the threshold met (including `V`'s unvalidated self-endorsement), and includes the invalid chunk in a block.
7. `validate_chunk_endorsements_in_block` passes because `V`'s signature over the `ChunkHash` is cryptographically valid — it does not check whether `V` actually validated the state witness. [1](#0-0) [9](#0-8)

        where

        {
            // Shuffling shard ids to avoid a bias towards lower ids, see [`ShuffledShardIds`]. We

            let shard_ids_for_mandates = ShuffledShardIds::new(rng, self.config.num_shards);
            let shard_ids_for_partials = ShuffledShardIds::new(rng, self.config.num_shards);

            let shuffled_mandates = self.shuffled_mandates(rng);
            let shuffled_partials = self.shuffled_partials(rng);

            // in `[0, num_shards)`, we take the elements of the vector with index `i` such that `i %
            // num_shards == shard_id`.
            //

            // collect the mandates with indices 1, 5, and 9.

            let mut stake_assignment_per_shard =

                let mandates_assignment =
