### Title
Recipient Not Bound by MPC Signature in `finalize_transfer` — Any Relayer Can Redirect Tokens to Arbitrary Account - (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary

The `FinalizeTransferPayload` struct does not include a `recipient` field. The MPC signature therefore does not commit to the recipient. Any relayer can take a valid MPC-signed `finalize_transfer` and substitute any `recipient` account, redirecting tokens (minted or unlocked) to an attacker-controlled address.

### Finding Description

**Step 1 — `recipient` is `UncheckedAccount` with no constraint.**

In `FinalizeTransfer`, the `recipient` account at line 51 is declared as:

```rust
/// CHECK: this can be any type of account
pub recipient: UncheckedAccount<'info>,
``` [1](#0-0) 

There is no `address`, `owner`, or any other Anchor constraint on `recipient`. The `token_account` is derived as the ATA of whatever `recipient` is passed:

```rust
associated_token::authority = recipient,
``` [2](#0-1) 

**Step 2 — `FinalizeTransferPayload` has no `recipient` field.**

The signed payload struct is:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
``` [3](#0-2) 

`recipient` and `mint` are only `AdditionalParams` for the *outgoing* `serialize_for_near` response back to NEAR — they are not part of the signed incoming payload at all. [4](#0-3) 

**Step 3 — `process` never validates `recipient.key()` against payload data.**

The `process` function uses `data: FinalizeTransferPayload` and never checks that `self.recipient.key()` matches any field in the payload. Tokens are minted or transferred directly to `self.token_account`, which is the ATA of the caller-supplied `recipient`: [5](#0-4) 

The same pattern applies to `FinalizeTransferSol`, where SOL is transferred directly to the unchecked `recipient`: [6](#0-5) [7](#0-6) 

### Impact Explanation

Because the MPC signature covers only `(destination_nonce, transfer_id, amount, fee_recipient)` and not the recipient pubkey, any relayer who obtains a valid MPC-signed payload for a legitimate transfer can resubmit it with an attacker-controlled `recipient`. The nonce prevents double-spend of the same payload, but the first (and only) finalization can be stolen. Tokens (SPL or SOL) are minted or unlocked to the attacker's account instead of the NEAR-designated recipient.

### Likelihood Explanation

Every relayer is unprivileged and submits `finalize_transfer` transactions publicly. The attacker only needs to front-run or race the legitimate relayer with the same valid MPC-signed payload but a different `recipient`. This is straightforward on Solana where transaction ordering is not guaranteed and MEV/front-running is possible.

### Recommendation

Include the `recipient` pubkey (and `mint`) inside `FinalizeTransferPayload` so they are covered by the MPC signature. On-chain, after signature verification, assert that `self.recipient.key() == data.recipient` and `self.mint.key() == data.mint`. This ensures the MPC-attested recipient is the only valid destination.

### Proof of Concept

1. NEAR user initiates a transfer to Solana recipient `R_legit`. MPC signs `FinalizeTransferPayload { destination_nonce: N, transfer_id: T, amount: A, fee_recipient: None }`.
2. Attacker intercepts the signed payload before the legitimate relayer submits it.
3. Attacker calls `finalize_transfer` with the same valid `SignedPayload` but passes `recipient = attacker_pubkey`.
4. Anchor derives `token_account` as ATA of `attacker_pubkey`. Signature verification passes (recipient is not in the signed data). Nonce `N` is marked used.
5. Tokens are minted/unlocked to attacker's ATA. `R_legit` receives nothing. The nonce is consumed so the legitimate relayer's subsequent attempt reverts.

---

This is a **real, bounty-grade critical vulnerability**. The root cause is that `FinalizeTransferPayload` omits the recipient, so the MPC signature provides no binding between the signed transfer intent and the on-chain destination account.
