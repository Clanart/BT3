### Title
`deploy_token` Signature Hash Omits Destination Chain ID, Enabling Cross-Chain Replay of Token Deployment Authorizations — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deploy_token` / `deployToken` functions on EVM, Starknet, and Solana construct the signed message hash from only the token metadata (type prefix, token ID, name, symbol, decimals) without including the destination chain ID. In contrast, the `finTransfer` / `fin_transfer` / `finalize_transfer` functions on all three chains explicitly embed the destination chain ID in the signed payload. This asymmetry means a single NEAR MPC signature authorizing token deployment on chain A is cryptographically valid on every other chain that shares the same `nearBridgeDerivedAddress` / `omni_bridge_derived_address`, enabling unauthorized cross-chain replay of token deployment.

---

### Finding Description

**EVM `OmniBridge.deployToken`** constructs the borsh-encoded message as:

```
PayloadType::Metadata | token | name | symbol | decimals
```

No `omniBridgeChainId` is included. [1](#0-0) 

Compare with **EVM `OmniBridge.finTransfer`**, which embeds `bytes1(omniBridgeChainId)` twice (once for the token chain, once for the recipient chain): [2](#0-1) 

**Starknet `MetadataPayload.to_borsh()`** (called by `deploy_token`) takes no `chain_id` parameter and encodes only `PayloadType::Metadata | token | name | symbol | decimals`: [3](#0-2) 

Compare with **Starknet `TransferMessagePayload.to_borsh(chain_id)`** (called by `fin_transfer`), which includes `chain_id` at two positions: [4](#0-3) 

The Starknet `deploy_token` call site passes no chain ID: [5](#0-4) 

While `fin_transfer` passes `self.omni_bridge_chain_id.read()`: [6](#0-5) 

**Solana `DeployTokenPayload::serialize_for_near`** serializes only the metadata prefix and fields, with no `SOLANA_OMNI_BRIDGE_CHAIN_ID`: [7](#0-6) 

Compare with **Solana `FinalizeTransferPayload::serialize_for_near`**, which writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` at two positions: [8](#0-7) 

---

### Impact Explanation

When NEAR MPC signs a `deploy_token` authorization for token `X` on chain A (e.g., EVM Ethereum), the resulting signature is byte-for-byte valid on every other chain (EVM BNB, Starknet, Solana) that uses the same `nearBridgeDerivedAddress`. An attacker can replay that signature on chain B to deploy the bridged token there without NEAR's explicit authorization for chain B. Once the token is deployed on chain B, it is registered in the bridge's canonical token mapping (`isBridgeToken` / `near_to_starknet_token` / wrapped mint PDA), making it indistinguishable from a legitimately authorized deployment. Subsequent `finTransfer` calls on chain B using valid transfer signatures will then mint tokens against this unauthorized deployment, creating unbacked supply on chain B.

This matches the allowed impact: **unauthorized token deployment** and **proof/signature verification bypass enabling unauthorized token deployment**. [9](#0-8) 

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who can observe a valid `deploy_token` transaction on one chain (e.g., by watching on-chain events or mempool) and submit the same `(signatureData, metadata)` arguments to the `deployToken` / `deploy_token` function on another chain. No privileged access, key compromise, or colluding validators are required. The only precondition is that the target token has not yet been deployed on the replay chain, which is the normal state before NEAR authorizes deployment there. [10](#0-9) 

---

### Recommendation

Include the destination chain ID in the `deploy_token` signed message hash on all chains, mirroring the pattern already used by `finTransfer` / `fin_transfer` / `finalize_transfer`.

- **EVM**: Add `bytes1(omniBridgeChainId)` to the `borshEncoded` concatenation in `deployToken`.
- **Starknet**: Add a `chain_id: u8` parameter to `MetadataPayload.to_borsh()` and append it to the borsh bytes; pass `self.omni_bridge_chain_id.read()` at the call site in `deploy_token`.
- **Solana**: Write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into `DeployTokenPayload::serialize_for_near` at an appropriate position (e.g., before the token field), matching the convention in `FinalizeTransferPayload`.

The NEAR side must correspondingly include the target chain ID when constructing the payload it signs, so that each chain-specific deployment authorization produces a distinct hash.

---

### Proof of Concept

1. NEAR MPC signs a `deploy_token` payload for token `"usdc.near"` (name=`"USD Coin"`, symbol=`"USDC"`, decimals=`6`) targeting EVM Ethereum (chain ID `0x02`). The signed hash is `keccak256(0x01 | borsh("usdc.near") | borsh("USD Coin") | borsh("USDC") | 0x06)` — no chain ID byte is present.

2. A relayer submits this transaction on Ethereum; `deployToken` succeeds and registers the EVM Ethereum bridge token.

3. An attacker copies the exact `signatureData` and `metadata` arguments and calls `deployToken` on EVM BNB (chain ID `0x04`). The EVM BNB `OmniBridge` computes the identical hash (no chain ID in the encoding), `ECDSA.recover` returns `nearBridgeDerivedAddress`, and the token is deployed on BNB without NEAR's authorization. [11](#0-10) 

4. The same replay works on Starknet: the attacker calls `deploy_token(signature, payload)` on the Starknet `OmniBridge`. `_verify_borsh_signature` computes `keccak256(MetadataPayload.to_borsh())` — identical to the EVM hash — and `verify_eth_signature` passes. [12](#0-11) 

5. The same replay works on Solana: the attacker calls `deploy_token` with the same signed payload. `SignedPayload::verify_signature` calls `DeployTokenPayload::serialize_for_near(())`, which produces the same byte sequence (no chain ID), and `secp256k1_recover` returns `derived_near_bridge_address`. [7](#0-6)

    function deployToken(
        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

        // slither-disable-next-line reentrancy-no-eth
        address bridgeTokenProxy = address(
            new ERC1967Proxy(
                tokenImplementationAddress,
                abi.encodeWithSelector(
                    BridgeToken.initialize.selector,
                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )
        );

        deployTokenExtension(
            metadata.token,
            bridgeTokenProxy,
            decimals,
            metadata.decimals
        );

        emit BridgeTypes.DeployToken(
            bridgeTokenProxy,
            metadata.token,
            metadata.name,
            metadata.symbol,
            decimals,
            metadata.decimals
        );

        isBridgeToken[address(bridgeTokenProxy)] = true;
        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
        nearToEthToken[metadata.token] = address(bridgeTokenProxy);

        return bridgeTokenProxy;
    }
