Audit Report

## Title
Router Aggregate ETH Balance Used Unattributed in `pay()` WETH Branch Allows Residual ETH Theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`pay()` in `PeripheryPayments.sol` uses `address(this).balance` — the router's total native ETH balance — when funding WETH payments. Because the router has no mechanism to attribute ETH to a specific caller, any ETH left in the router by a prior user (via an overpaid `multicall` without `refundETH()`) is silently consumed to subsidize a subsequent caller's WETH swap. The prior user's ETH is permanently lost; the attacker pays less WETH than the pool demands.

## Finding Description

`pay()` handles WETH payments through three branches keyed on `address(this).balance`:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
  uint256 nativeBalance = address(this).balance;
  if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
  } else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
  } else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
  }
```

**ETH accumulation path:** The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH transfers; it does not prevent ETH from being held when sent as `msg.value` in a `payable` function call. A user calling `multicall{value: 2 ether}` with a 1-ether WETH swap and no `refundETH()` leaves 1 ETH in the router. The test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` confirms this pattern is real and that `refundETH()` is optional and user-driven.

**Exploit path:**
1. Alice calls `multicall{value: 2 ether}` swapping only 1 ether of WETH, omitting `refundETH()`. Router retains 1 ETH.
2. Bob calls `exactInputSingle(tokenIn=WETH, amountIn=1 ether)` with 0 ETH sent.
3. `exactInputSingle` sets callback context with `payer = Bob` via `_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`.
4. The pool calls `metricOmmSwapCallback`, which calls `_justPayCallback`, which calls `pay(WETH, Bob, pool, 1 ether)`.
5. Inside `pay()`: `nativeBalance = 1 ether >= value = 1 ether` → first branch fires: router deposits its 1 ETH as WETH, transfers to pool, pulls **0 WETH** from Bob.
6. Bob receives full swap output. Alice's 1 ETH is permanently consumed.

If `nativeBalance = 0.5 ether < value = 1 ether`, the partial branch fires: Bob pays only 0.5 WETH instead of 1 WETH, and Alice's 0.5 ETH is consumed. In both cases, `pay()` never verifies that the ETH it spends was contributed by the current caller.

No existing guard prevents this: the callback caller check (`_requireExpectedCallbackCaller`) only validates that the callback comes from the expected pool, not that the ETH being spent belongs to the current payer.

## Impact Explanation

Direct loss of user principal. Any ETH left in the router by a prior user is permanently transferred to the pool as part of an attacker's WETH swap. The attacker receives the full swap output while paying less WETH than the pool demanded. This is a direct loss of the prior user's ETH with no recovery path, satisfying the HIGH severity threshold for direct loss of user principal.

## Likelihood Explanation

Medium. The attack requires residual ETH in the router, which is a realistic condition: `refundETH()` is optional and user-driven, and any user who sends excess ETH via `multicall` without including `refundETH()` leaves ETH behind. The test suite explicitly validates this pattern. Once residual ETH exists, the attack is trivially executable by any address with a WETH approval and requires no special privileges.

## Recommendation

Track the ETH contributed by the current call context rather than using the aggregate `address(this).balance`. Options:

1. **Pass `msg.value` through the call stack** and cap the ETH used in `pay()` to that amount, stored in transient storage alongside the existing callback context fields.
2. **Restrict the ETH-assisted branch to `payer == address(this)`** — i.e., only use the router's ETH balance when the router itself is the payer (meaning it already holds the funds on behalf of the current user from the same transaction). When `payer != address(this)`, always use `safeTransferFrom` for the full amount.

## Proof of Concept

```solidity
function test_residualEth_subsidizesAttackerWethSwap() public {
    // Simulate Alice forgetting refundETH: seed router with 0.5 ETH residual
    vm.deal(address(router), 0.5 ether);

    uint128 amountIn = 1 ether;
    uint256 attackerWethBefore = weth.balanceOf(attacker);

    vm.prank(attacker);
    // Attacker sends 0 ETH, has WETH approval for full 1 ether
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: amountIn,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));

    // Router ETH drained to zero
    assertEq(address(router).balance, 0, "router ETH drained");
    // Attacker only paid 0.5 WETH, not 1 WETH
    assertEq(attackerWethBefore - weth.balanceOf(attacker), 0.5 ether, "only 0.5 WETH pulled");
}
```

Note: `vm.deal(address(router), 0.5 ether)` simulates residual ETH; in production this accumulates naturally from users who omit `refundETH()`. The `receive()` guard does not block ETH held from `msg.value` in payable calls.
