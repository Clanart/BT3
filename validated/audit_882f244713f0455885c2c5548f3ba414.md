### Title
Hardcoded `failure_flag=0` in `execute_deploy` Syscall Unconditionally Masks Constructor Failures, Enabling Direct Loss of Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in `syscall_impls.cairo` always writes `failure_flag=0` (success) into the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is an acknowledged gap (marked with a `TODO` comment). Any on-chain contract that relies on the standard `failure_flag` field to gate post-deployment fund transfers will be silently misled, causing funds to be sent to an address whose constructor never completed — permanently freezing them.

---

### Finding Description

In `execute_deploy` the constructor is executed via `deploy_contract`, which internally uses the revert log to undo storage and class-hash changes if the constructor reverts. However, the success/failure outcome is never captured and the response header is unconditionally written with `failure_flag=0`:

```cairo
// syscall_impls.cairo  lines 527-539
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

Compare this with `contract_call_helper` in the same file, which correctly propagates the revert status:

```cairo
// syscall_impls.cairo  lines 432-433
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` path is reached from `execute_syscalls` whenever a contract issues a `DEPLOY_SELECTOR` syscall:

```cairo
// execute_syscalls.cairo  lines 165-173
if (selector == DEPLOY_SELECTOR) {
    execute_deploy(block_context=block_context, caller_execution_context=execution_context);
    ...
}
``` [3](#0-2) 

When the constructor reverts, the revert log rolls back the class-hash assignment and all storage writes for the new address, leaving it as an uninitialised zero-class-hash slot. Yet the OS writes `failure_flag=0` into the response, so the calling contract's Sierra/CASM code observes a successful deploy and proceeds accordingly.

---

### Impact Explanation

**Critical — Direct loss of funds / Permanent freezing of funds.**

A factory-style contract (a common DeFi primitive) that:

1. Calls `deploy` to create a new vault or pool contract,
2. Reads `failure_flag` from the response (always 0 → "success"),
3. Immediately transfers ERC-20 tokens or native assets to the returned `contract_address`,

will send funds to an address whose class hash is `0` (the constructor was reverted). The ERC-20 `transfer` call succeeds — it merely updates the token contract's storage — but there is no contract at the destination to ever withdraw those tokens. The funds are permanently frozen.

The `charge_fee` path in `transaction_impls.cairo` confirms that fees are charged even for reverted executions, so the transaction is accepted by the OS and committed to state:

```cairo
// transaction_impls.cairo  lines 360-361
// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [4](#0-3) 

The state update in `state.cairo` then commits the (incorrect) final state, including the ERC-20 balance at the dead address, to the Patricia tree: [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The entry path is fully unprivileged: any user can invoke a transaction that calls a contract which issues a `DEPLOY` syscall. Factory contracts that deploy-and-fund in a single transaction are standard in DeFi (liquidity pools, vaults, proxy factories). The constructor failure condition can be triggered by:

- Supplying constructor calldata that causes an arithmetic assertion to fail,
- Deploying a class whose constructor calls a syscall that runs out of gas,
- Any other revertible condition inside the constructor.

No privileged role, leaked key, or operator cooperation is required.

---

### Recommendation

Modify `execute_deploy` to capture the constructor's revert status from `deploy_contract` and propagate it into the response header, mirroring the pattern already used in `contract_call_helper`:

```cairo
// Proposed fix
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(...);
}
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

`deploy_contract` must also be updated to return `is_reverted`. Until this is done, calling contracts have no reliable way to detect constructor failures via the standard `failure_flag` mechanism.

---

### Proof of Concept

1. **Attacker deploys** a `MaliciousFactory` contract whose `deploy_and_fund` entry point:
   - Calls the `deploy` syscall with a class whose constructor always reverts (e.g., `assert 1 = 0`).
   - Reads `response.failure_flag` — receives `0` (success) due to the bug.
   - Calls `erc20.transfer(deployed_address, amount)` to fund the "new" contract.

2. **User calls** `MaliciousFactory.deploy_and_fund(victim_funds)`.

3. **OS execution**:
   - Constructor reverts → revert log rolls back class-hash assignment → `deployed_address` has `class_hash = 0`.
   - `execute_deploy` writes `failure_flag=0` → factory proceeds.
   - ERC-20 transfer succeeds → `victim_funds` credited to `deployed_address` in token storage.

4. **State committed**: `deployed_address` holds a non-zero ERC-20 balance but has no contract code. No `__execute__` entry point exists to withdraw the funds. **Funds are permanently frozen.**
