### Title
Integer Wrapping in `check_time_locks` Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Timelocks When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` contains two arithmetic paths controlled by the `nowrap` boolean parameter. When `nowrap=false`, the relative-timelock comparisons use `wrapping_add` instead of `saturating_add`. A crafted CLVM spend condition with a near-maximum `height_relative` or `seconds_relative` value causes the sum `confirmed_height + relative` to wrap around to a small integer, making the "not yet mature" guard evaluate to `false` and allowing the coin to be spent immediately — bypassing the intended timelock entirely.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces per-spend relative timelocks. The function accepts a `nowrap: bool` parameter that selects between two arithmetic strategies for the four relative-timelock conditions:

```
// nowrap=true  (safe path)
prev_transaction_block_height < unspent.confirmed_block_index.saturating_add(height_relative)

// nowrap=false (legacy/wrapping path)
prev_transaction_block_height < unspent.confirmed_block_index.wrapping_add(height_relative)
``` [1](#0-0) 

The same pattern applies to `seconds_relative`, `before_height_relative`, and `before_seconds_relative`. [2](#0-1) 

**Wrapping bypass for `ASSERT_HEIGHT_RELATIVE`:**

If a coin is confirmed at height `H` and the spend carries `height_relative = R` such that `H + R` overflows `u32`, the `wrapping_add` result is a small integer (e.g., `10 + 0xFFFF_FFF6 = 0`). The guard `prev_height < 0` is immediately false, so the spend is accepted at any block height — the timelock is silently discarded.

The test suite explicitly documents this divergence:

```
// 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok (bypass)
// saturating: clamps to u32::MAX, 15 < u32::MAX -> Err (correct)
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
``` [3](#0-2) 

**Wrapping bypass for `ASSERT_SECONDS_RELATIVE`:**

```
// 10000 + (u64::MAX - 9999) overflows to 0, wrapping: 10150 < 0 -> Ok (bypass)
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_D8F0), 105, None),
``` [4](#0-3) 

**False rejection for `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE`:**

The inverse condition (`>=`) means wrapping to a small value causes a valid spend to be incorrectly rejected, producing a consensus-divergence risk between nodes running different `nowrap` settings. [5](#0-4) 

The `nowrap` parameter is fully caller-controlled through the Python binding `py_check_time_locks`, which passes it unmodified to the Rust core: [6](#0-5) 

The Python binding is registered in the `chia_rs` module and is callable from the full node: [7](#0-6) 

The type stub confirms the public API surface: [8](#0-7) 

---

### Impact Explanation

When the full node calls `check_time_locks` with `nowrap=False` (the legacy pre-fork path), an attacker who controls a coin's puzzle can embed `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFF6)` or `ASSERT_SECONDS_RELATIVE(0xFFFF_FFFF_FFFF_D8F0)`. The wrapping arithmetic reduces the effective maturity threshold to zero, allowing the coin to be spent at any block height or timestamp. This constitutes an **unauthorized spend acceptance** — a timelock condition validation bypass. Coins that were intended to be locked for billions of blocks or seconds become immediately spendable.

The secondary impact (false rejection of `ASSERT_BEFORE_*_RELATIVE` with large values under `nowrap=False`) creates a **consensus divergence** risk: nodes that apply `nowrap=True` accept the spend while nodes applying `nowrap=False` reject it, potentially causing a chain split.

---

### Likelihood Explanation

The `nowrap` parameter was introduced to distinguish legacy (wrapping) behavior from corrected (saturating) behavior, strongly implying the full node passes `nowrap=False` for blocks below a hard-fork activation height. Any attacker who can create a coin on the pre-fork chain with a crafted puzzle can exploit this. The crafted `height_relative` or `seconds_relative` values are valid CLVM atoms (they fit within the 4-byte and 8-byte limits accepted by `sanitize_uint`) and pass condition parsing without error. [9](#0-8) 

---

### Recommendation

1. **Remove the `nowrap=false` path entirely** if the hard fork that introduced `nowrap=True` is now universally activated. The legacy wrapping behavior should not remain reachable for any block height.
2. If legacy block re-validation is still required, **add an explicit upper-bound check** on `height_relative` and `seconds_relative` before the addition: reject any value that would cause `confirmed_index + relative > u32::MAX` (or `u64::MAX`) rather than silently wrapping.
3. **Audit all call sites** in the Python full node that invoke `check_time_locks` to confirm `nowrap=False` is no longer passed for any live or replayable block range.

---

### Proof of Concept

Using the existing Python test harness:

```python
from chia_rs import check_time_locks, CoinRecord, Coin
from chia_rs.sized_ints import uint32, uint64

# Coin confirmed at height 10
coin = Coin(b'\x00'*32, b'\x00'*32, uint64(1_000_000_000))
record = CoinRecord(coin, uint32(10), uint32(0), False, uint64(10000))

# Craft a spend with ASSERT_HEIGHT_RELATIVE(0xFFFF_FFF6)
# Intended: coin matures at block 10 + 0xFFFF_FFF6 = 4294967296 (never)
# With nowrap=False: 10 + 0xFFFF_FFF6 wraps to 0; 15 < 0 is False -> PASSES
conds = make_test_conds(height_relative=0xFFFF_FFF6)

result_safe   = check_time_locks({coin.name(): record}, conds, uint32(15), uint64(10150), True)
result_legacy = check_time_locks({coin.name(): record}, conds, uint32(15), uint64(10150), False)

assert result_safe   == 13    # Correctly rejected (AssertHeightRelativeFailed)
assert result_legacy is None  # BYPASSED: coin spendable immediately
