Looking at the actual code in `set_state_header` and `validate_chunk_proofs`, I need to trace every guard that could block the cross-shard substitution.

**`validate_chunk_proofs` checks** (lines 22–66 of `chain/chain/src/validate.rs`):
- Chunk header hash self-consistency
- Transaction Merkle root
- Outgoing receipts root
- **No check of `chunk.shard_id()` against any external `shard_id` parameter**

**`set_state_header` checks** (lines 368–531 of `chain/chain/src/state_sync/adapter.rs`):

Step 3a — the Merkle inclusion check:
```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
)
```
`verify_path` reconstructs the Merkle root from the path and the leaf hash. It confirms the chunk is *somewhere* in the block's `chunk_headers_root` tree. It does **not** verify which position (index) the chunk occupies, and it does **not** compare that position to `shard_id`. A valid Merkle proof for shard B's position passes this check even when `shard_id=A` was requested.

Step 4e — receipt hash uses the *requested* `shard_id`, not the chunk's:
```rust
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
```
The attacker must supply valid incoming receipts for shard A. These are public on-chain data, so a peer who has synced block headers and blocks can construct them.

Step 5 — state root node validated against `chunk.prev_state_root()` (shard B's root), which the attacker controls and provides consistently.

**Storage** (lines 526–529):
```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```
The key uses the *requested* `shard_id=A`, but the stored value contains shard B's chunk with shard B's `prev_state_root`.

**`set_state_part`** (lines 534–561) then reads the stored header, extracts `state_root` from shard B's chunk, and calls `validate_state_part(shard_id=A, &state_root_B, part_id, part)`. Parts of shard B's trie are cryptographically valid against `state_root_B`, so they pass.

**`set_state_finalize`** applies shard B's state as shard A's state.

---

However, the attack has a **non-trivial construction requirement** that the question's framing understates:

The attacker must supply `incoming_receipts_proofs` that pass the full receipt proof chain (steps 4b–4g). These proofs must be for shard A's actual incoming receipts, verified against the canonical block's `prev_chunk_outgoing_receipts_root`. This is public chain data, but it requires the attacker to have synced the full block data and correctly construct the receipt Merkle proofs for shard A — not just replay shard B's data wholesale.

Additionally, the `prev_chunk_header` must also be a chunk that is genuinely included in the correct ancestor block (step 3b). The attacker can use shard B's prev chunk header with its valid Merkle proof, since that check also has no shard_id position binding.

**The missing guard is real and concrete**: nowhere in `set_state_header` is there a check equivalent to:
```rust
if chunk.shard_id() != shard_id { return Err(...); }
```

The exact corrupted value is: after state sync completes, shard A's installed state root = shard B's `prev_state_root` from the crafted chunk, not shard A's actual state root at `sync_hash`.

### Title
Missing shard_id cross-check in `set_state_header` allows malicious peer to install wrong shard's state root — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` for `shard_id=A` containing a chunk whose embedded `shard_id` is B. No guard compares the chunk's shard identity to the requested shard. The header is stored under `StateHeaderKey(A, sync_hash)` with shard B's `prev_state_root`, which is then used as the authoritative state root for all subsequent `set_state_part` and `set_state_finalize` calls for shard A.

### Finding Description
`validate_chunk_proofs` validates only internal chunk consistency (hash, tx root, receipts root) with no reference to the caller-supplied `shard_id`. [1](#0-0) 

The Merkle inclusion check in `set_state_header` verifies that the chunk hash appears somewhere in the block's `chunk_headers_root` tree, but `verify_path` does not bind the proof to a specific leaf index, so a proof for shard B's position passes when shard A is requested. [2](#0-1) 

The storage key uses the caller-supplied `shard_id=A`, but the stored value is the crafted header containing shard B's chunk and shard B's `prev_state_root`. [3](#0-2) 

`set_state_part` reads the stored header, extracts the state root from the embedded chunk (shard B's root), and validates incoming parts against it with `shard_id=A`. [4](#0-3) 

### Impact Explanation
After a successful attack, the victim node installs shard B's trie as shard A's state. The node's `ChunkExtra` for shard A will record shard B's state root. All subsequent chunk application for shard A will start from the wrong state, producing wrong execution results, wrong outgoing receipts, and a state root that diverges from the honest network. The node cannot participate correctly in shard A validation.

### Likelihood Explanation
Any peer reachable by the syncing node can respond to `StateRequestHeader` messages. No validator or operator privilege is required. The attacker needs to have synced block headers and the relevant blocks to construct valid receipt proofs for shard A — this is public chain data. The construction is non-trivial but entirely feasible for a motivated attacker. [5](#0-4) 

### Recommendation
Add an explicit shard_id check at the top of `set_state_header`, after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, bind the Merkle inclusion check to the expected shard index by using `verify_path_with_index` (already present in the codebase) rather than `verify_path`, so the proof must correspond to the correct leaf position. [6](#0-5) 

### Proof of Concept
1. Set up a two-shard network. Let shard A = shard_id 0, shard B = shard_id 1.
2. At the sync epoch boundary, obtain shard B's full chunk and its valid Merkle proof from the block's `chunk_headers_root`.
3. Obtain shard A's incoming receipt proofs from the canonical chain (public data).
4. Construct a `ShardStateSyncResponseHeaderV2` with: `chunk` = shard B's chunk, `chunk_proof` = shard B's Merkle proof, `prev_chunk_header` = shard B's prev chunk header with its Merkle proof, `incoming_receipts_proofs` = shard A's actual incoming receipts with valid proofs, `state_root_node` = shard B's state root node.
5. Call `set_state_header(shard_id=0, sync_hash, crafted_header)` on the victim node.
6. Assert the call returns `Ok(())`.
7. Call `set_state_part` with valid parts of shard B's trie; assert they pass.
8. Call `set_state_finalize(shard_id=0, sync_hash)`.
9. Assert that the installed state root for shard 0 equals shard B's `prev_state_root`, not shard A's.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;

        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
