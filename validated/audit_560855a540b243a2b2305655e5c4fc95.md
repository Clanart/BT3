### Title
State Sync Header Accepts Cross-Shard Chunk Without Binding Verification — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` validates that the supplied chunk is included in the block's Merkle tree, but never verifies that the chunk's embedded `shard_id` matches the `shard_id` parameter of the request. A malicious peer can supply a cryptographically valid chunk from shard X in response to a state-sync header request for shard Y. All validation checks pass, the header is stored under shard Y's key, and the node subsequently downloads, validates, and applies shard X's trie state into shard Y's storage — permanently corrupting shard Y's state on the syncing node.

---

### Finding Description

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from an untrusted peer and performs the following checks:

1. **`validate_chunk_proofs`** — verifies the chunk's internal consistency (encoded Merkle root, producer signature). Does not check which shard the chunk belongs to.
2. **`verify_path` against `chunk_headers_root`** — verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's chunk-headers Merkle tree. This proves the chunk is *somewhere* in the block, but the Merkle tree contains one leaf per shard; the proof is position-specific to shard X's index, not to the requested shard Y.
3. Receipt proof checks — use the caller-supplied `shard_id` (Y) to compute `ReceiptList(shard_id, receipts)`, so they bind to shard Y. These are public on-chain data the attacker can supply correctly.
4. **`validate_state_root_node`** — validates the state root node against `chunk_inner.prev_state_root()`, which is shard X's state root.

**The missing check**: at no point is `chunk.shard_id()` compared against the `shard_id` parameter. After all checks pass, the header is stored:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

The key uses the requested shard Y, but the stored header contains shard X's chunk and shard X's `prev_state_root`.

**Downstream consequence chain:**

- `run_state_sync_for_shard` reads `state_root = header.chunk_prev_state_root()` — this is shard X's state root.
- `ensure_shard_part_downloaded_single_attempt` downloads parts and validates them against shard X's state root — they pass because they are genuine shard X parts.
- Parts are stored under `StatePartKey(sync_hash, shard_Y, part_id)`.
- `apply_state_part` is called with `shard_id = Y` and `epoch_id`, computes `shard_uid` for shard Y, and writes shard X's trie nodes into shard Y's storage column.
- `set_state_finalize` → `chain_update.set_state_finalize` applies shard X's chunk (transactions, receipts) using `shard_uid` for shard Y, completing the state corruption.

**Attacker-controlled data path:**

```
Malicious peer → ShardStateSyncResponseHeader {
    chunk: shard_X_chunk,          // valid chunk, valid signature, valid Merkle proof at shard_X_index
    chunk_proof: shard_X_proof,    // valid Merkle path for shard X's position
    incoming_receipts_proofs: ..., // valid on-chain receipts for shard Y (public data)
    state_root_node: shard_X_node, // matches shard_X_chunk.prev_state_root
}
→ set_state_header(shard_id=Y, ...) → all checks pass
→ StateHeaderKey(shard_Y, sync_hash) stores shard X's header
→ state parts for shard X applied to shard Y's storage
```

---

### Impact Explanation

**Critical.** A single malicious peer reachable during state sync can cause the syncing node to permanently install shard X's trie state as shard Y's state. The node will subsequently:

- Produce chunks for shard Y using shard X's state root, which will be rejected by honest validators.
- Fail to make progress on shard Y, effectively halting participation in that shard.
- Potentially produce invalid state transitions that propagate if the node is a chunk producer.

The corrupted value is the entire trie stored under `shard_uid(Y)` in `DBCol::State` and `DBCol::FlatState`, and the `ChunkExtra` written for shard Y after `set_state_finalize`.

---

### Likelihood Explanation

State sync is triggered whenever a node bootstraps or falls behind. During this phase the node connects to arbitrary peers and accepts `ShardStateSyncResponseHeader` messages from them. Any peer that can respond to a `StateRequestHeader` message — which is any reachable node — can mount this attack. No stake, no special role, and no prior relationship is required.

---

### Recommendation

In `set_state_header`, immediately after extracting the chunk, assert that the chunk's embedded shard ID matches the requested shard ID:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This check should be placed before the Merkle path verification and before any further use of the chunk's fields.

---

### Proof of Concept

1. Syncing node S requests `StateRequestHeader { shard_id: 1, sync_hash }` from malicious peer P.
2. P responds with a `ShardStateSyncResponseHeader` containing:
   - `chunk`: the genuine chunk for shard 0 at `sync_hash - 1`, with its valid producer signature.
   - `chunk_proof`: the genuine Merkle path for shard 0's position in `sync_prev_block.chunk_headers_root`.
   - `incoming_receipts_proofs`: genuine on-chain incoming receipts for shard 1 (public data).
   - `state_root_node`: the genuine state root node for shard 0's `prev_state_root`.
3. S calls `set_state_header(shard_id=1, sync_hash, header)`:
   - `validate_chunk_proofs` passes (shard 0's chunk is internally valid).
   - `verify_path` passes (shard 0's chunk hash IS in the block's Merkle tree).
   - Receipt proofs pass (they are genuine receipts for shard 1).
   - `validate_state_root_node` passes (node matches shard 0's `prev_state_root`).
   - Header stored under `StateHeaderKey(shard_1, sync_hash)`.
4. S downloads state parts for shard 0's state root, validates them (they pass), stores them under `StatePartKey(sync_hash, shard_1, part_id)`.
5. `apply_state_part(shard_id=1, state_root=shard_0_root, ...)` writes shard 0's trie into shard 1's storage column.
6. `set_state_finalize(shard_id=1, ...)` applies shard 0's chunk to shard 1's (now shard-0-populated) storage.
