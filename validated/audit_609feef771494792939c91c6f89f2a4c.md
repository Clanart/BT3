### Title
Unvalidated Class Hash in `replace_class` Syscall Enables Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` accepts any arbitrary class hash from a calling contract without verifying that the hash corresponds to a previously declared contract class. This missing validation — explicitly acknowledged by a TODO in the code — allows any contract to permanently brick itself (and freeze all funds it holds) by replacing its class with an undeclared, non-executable hash. The OS will accept and prove this invalid state transition.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the OS processes the `replace_class` syscall as follows:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
```

The `class_hash` field comes directly from the syscall request — i.e., from the calling contract's memory — and is written into `contract_state_changes` with **no check** that it exists in `contract_class_changes` (the dictionary of declared classes). The TODO comment on line 898 explicitly confirms this invariant is unenforced.

By contrast, the `execute_declare_transaction` path in `transaction_impls.cairo` enforces `prev_value=0` and `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`, ensuring a class is declared exactly once with a valid compiled hash. The `replace_class` path has no equivalent guard.

The analog to the Trust.sol report is direct: just as Trust.sol grants blanket authority to any trusted account without fine-grained controls over what operations they may perform, the OS grants blanket authority to any contract to write any felt value as its class hash, without verifying the value is a legitimately declared class.

---

### Impact Explanation

If a contract calls `replace_class` with a hash that was never declared:

1. The OS writes the undeclared hash into `contract_state_changes` for that contract address.
2. The state update is committed and proven — the OS produces a valid proof for this invalid state.
3. All subsequent calls to the contract will fail: the execution layer will look up the class hash, find no corresponding compiled class, and revert every invocation.
4. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — no `__execute__` or any other entry point can run.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

- **Rug-pull vector**: An attacker deploys a contract that appears legitimate (e.g., a vault or staking contract), attracts user deposits, then calls `replace_class` with an arbitrary undeclared felt. The OS accepts the transaction. The contract is permanently bricked. Deposited funds are frozen forever.
- **Griefing via vulnerable contracts**: If any deployed contract exposes a function that calls `replace_class` with caller-supplied data (e.g., an upgradeable proxy with insufficient access control), an external attacker can trigger this path on a contract they do not own, freezing its funds.
- **No privileged role required**: The syscall is available to any contract during normal execution. No sequencer cooperation, no key compromise, and no trusted role is needed beyond deploying a contract.

The TODO comment confirms the developers are aware the check is absent, meaning this is not a theoretical concern — it is a confirmed missing enforcement in the current OS code.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the hash exists as a key in `contract_class_changes` (i.e., it was previously declared). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `request.class_hash` and assert the returned value is non-zero (a valid compiled class hash). This mirrors the invariant already enforced in `execute_declare_transaction`.

---

### Proof of Concept

1. Attacker deploys contract `C` with a function `brick()` that issues `replace_class(0xdeadbeef)` — an arbitrary undeclared felt.
2. Users deposit tokens into `C` (e.g., via a `deposit()` entry point).
3. Attacker calls `brick()`. The OS executes `execute_replace_class` in `syscall_impls.cairo`, writes `class_hash=0xdeadbeef` into `contract_state_changes` for address `C`, and produces a valid proof — **no check is performed against `contract_class_changes`**.
4. The block is finalized on L1 with this state.
5. All subsequent calls to `C` (including `withdraw()`) fail: the execution layer finds no compiled class for hash `0xdeadbeef`.
6. All deposited funds are permanently frozen with no recovery path.
