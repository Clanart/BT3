### Title
Non-EVM-Native Bank Balances (IBC/CosmWasm Tokens) Not Burned on Self-Destruct, Recoverable via CREATE2 Redeployment — (`x/evm/statedb/statedb.go`)

### Summary

When a contract that holds non-EVM-native tokens (IBC vouchers, CosmWasm bridge tokens, or any Cosmos bank denom other than the EVM denom) executes `SELFDESTRUCT`, `StateDB.Commit()` burns only the EVM denom balance. All other bank module balances remain at the destroyed address as orphaned funds. Because `DeleteAccount` removes only the auth account record (not bank balances), a subsequent CREATE2 redeployment at the same address creates a new auth account that inherits those orphaned balances, allowing the deployer to drain them.

### Finding Description

The commit loop in `StateDB.Commit()` (`statedb.go:800–825`) handles self-destructed accounts by:

1. Burning any remaining EVM-denom balance via `keeper.SubBalance(cacheCtx, cosmosAddr, evmDenomCoin)` (line 816–820).
2. Calling `keeper.DeleteAccount(cacheCtx, addr)` which removes code, storage, and the auth account record (lines 208–215 of `keeper/statedb.go`), but **never touches bank balances**.

The code comment at lines 813–815 explicitly acknowledges: *"Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm bridge) held by the destroyed address are not drained and may remain as orphaned bank balances."*

The `BankKeeper` interface (`x/evm/types/interfaces.go`) exposes no method to iterate or drain all denoms at an address, so there is no mechanism in the current commit path to handle non-EVM denoms.

After `DeleteAccount`, the Cosmos bank module still holds the IBC balance keyed by the raw address bytes. When a new contract is deployed at the same address via CREATE2 (deterministic: `keccak256(0xff ++ factory ++ salt ++ keccak256(initcode))`), a new `EthAccount` is created at that address. The bank module immediately associates the pre-existing IBC balance with this new account, making it spendable.

### Impact Explanation

An attacker who controls a contract (or can trigger its self-destruct path) that holds IBC vouchers or other non-EVM-native Cosmos bank tokens can:

1. Self-destruct the contract — EVM denom is burned, IBC tokens remain.
2. Redeploy at the same address via CREATE2 with the same factory/salt/initcode.
3. Call any drain function on the redeployed contract to transfer the IBC tokens to themselves.

This constitutes unauthorized theft of Cosmos bank funds (IBC vouchers, bridged assets) through Ethermint stateDB commit logic, matching the Critical impact scope.

### Likelihood Explanation

- **Contracts holding IBC tokens**: Real and common. Any Cosmos address (including contract addresses) can receive IBC vouchers via `MsgTransfer` or bank sends. DEX contracts, bridge escrows, and CosmWasm-EVM interop contracts routinely hold multi-denom balances.
- **Attacker controlling self-destruct**: Requires controlling the contract or a privileged function within it. This is a meaningful constraint, but malicious contract deployers or rug-pull scenarios satisfy it.
- **CREATE2 redeployment**: The deployer controls factory address, salt, and init code — all known to them. Redeployment is straightforward.
- **EIP-6780 (Cancun)**: Under EIP-6780, `SelfDestruct6780` only destroys if `newContract == true` (created in the same tx). This limits the window but does not eliminate it — the attack works in a single atomic transaction (deploy → fund → destroy → redeploy is achievable across two transactions, or within one if the chain has not enabled Cancun).

### Recommendation

In the `selfDestructed` branch of `StateDB.Commit()`, after burning the EVM denom, iterate over **all** bank balances at the destroyed address and burn or escrow them. This requires either:

- Adding a `GetAllBalances(ctx, addr) sdk.Coins` method to the `BankKeeper` interface and burning each coin, or
- Transferring all balances to a designated module account before `DeleteAccount`.

The `BankKeeper` interface (`x/evm/types/interfaces.go:46–55`) must be extended accordingly, and the `Keeper.SubBalance` / `Keeper.AddBalance` helpers already exist for per-coin operations.

### Proof of Concept

```

// Tx 2 (atomic):
//   a) Factory F deploys child C at CREATE2(F, S, I)
//   b) IBC tokens are sent to C via Cosmos bank (MsgSend or IBC transfer)
//   c) C.selfdestruct() is called
//      → EVM denom burned; IBC balance remains at addr(C) in bank module
//      → auth account for C deleted by DeleteAccount
// Tx 3:
//   a) Factory F redeploys child C' at CREATE2(F, S, I) — same address
//      → new EthAccount created at addr(C)
//      → bank module IBC balance is now owned by C'
//   b) C'.drain() transfers IBC tokens to attacker
//
// Assert: IBC denom balance at addr(C) is non-zero after Tx 2 commit.
// Assert: IBC denom balance at addr(C) is zero after Tx 3 drain.
```

Extending it to fund the child with an IBC denom before destruction and asserting the IBC balance post-commit would directly demonstrate this gap. [6](#0-5)

func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil

// TestSelfDestructPostDestructionBalanceBurned verifies that any balance credited to a
// self-destructed address within the same transaction is burned at commit time rather
// than left as an orphaned bank balance recoverable by recreating the address.
func (suite *StateDBTestSuite) TestSelfDestructPostDestructionBalanceBurned() {
	raw, ctx, keeper := setupTestEnv(suite.T())

	// Setup: create a contract account with initial balance and code.
	db := statedb.New(ctx, keeper, emptyTxConfig)
	db.CreateAccount(address)
	db.CreateContract(address)
	db.SetCode(address, []byte("contract code"), 0)
	db.AddBalance(address, uint256.NewInt(100), tracing.BalanceChangeTransfer)
	suite.Require().NoError(db.Commit())

	ctx, keeper = newTestKeeper(suite.T(), raw)

	// Phase 1: Self-destruct the contract; its initial balance (100) must be burned.
	db = statedb.New(ctx, keeper, emptyTxConfig)
	db.SelfDestruct(address)
	suite.Require().True(db.HasSelfDestructed(address))
	suite.Require().Equal(uint256.NewInt(0), db.GetBalance(address))

	// Phase 2: Send value to the already-destroyed address in the same transaction.
	// This simulates a CALL with value to a self-destructed contract.
	postDestructValue := uint256.NewInt(500)
	db.AddBalance(address, postDestructValue, tracing.BalanceChangeTransfer)
	suite.Require().Equal(postDestructValue, db.GetBalance(address))

	suite.Require().NoError(db.Commit())

	// After commit: account metadata must be gone.
	ctx, keeper = newTestKeeper(suite.T(), raw)
	suite.Require().Nil(keeper.GetAccount(ctx, address))

	// The post-destruction balance must be burned (zero), not preserved.
	cosmosAddr := sdk.AccAddress(address.Bytes())
	balance := keeper.GetBalance(ctx, cosmosAddr, "uphoton")
	suite.Require().True(balance.IsZero(), "post-selfdestruct balance must be burned at commit")
}
