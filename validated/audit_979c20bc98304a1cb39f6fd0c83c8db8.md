### Title
Panic in `ProofOfSpace::update_digest` on Invalid v2 Proof Bytes Accepted by `parse` — (`File: crates/chia-protocol/src/proof_of_space.rs`)

---

### Summary

`ProofOfSpace::parse` for version-1 (v2) proofs accepts any byte sequence as the `proof` field without cryptographic validation. `ProofOfSpace::update_digest`, called transitively by `FullBlock::hash()` and `RewardChainBlock::hash()` during consensus processing, calls `self.quality_string().expect(...)` on that proof. When the proof bytes are invalid, `quality_string()` returns `None` and the `.expect()` unconditionally panics. An unprivileged peer can send a crafted `FullBlock` or `RewardChainBlock` containing a v2 `ProofOfSpace` with garbage proof bytes, crashing any node that attempts to compute the block hash.

---

### Finding Description

`ProofOfSpace::parse` for `version == 1` reads `plot_index`, `meta_group`, `strength`, and `proof` from the wire, then only checks that exactly one of `pool_public_key` / `pool_contract_puzzle_hash` is set:

```rust
// crates/chia-protocol/src/proof_of_space.rs:316-337
} else if version == 1 {
    let plot_index = <u16 as Streamable>::parse::<TRUSTED>(input)?;
    let meta_group  = <u8  as Streamable>::parse::<TRUSTED>(input)?;
    let strength    = <u8  as Streamable>::parse::<TRUSTED>(input)?;
    let proof       = <Bytes as Streamable>::parse::<TRUSTED>(input)?;

    if pool_public_key.is_some() == pool_contract_puzzle_hash.is_some() {
        return Err(Error::InvalidPoS);
    }
    Ok(ProofOfSpace { ..., proof, ... })   // proof bytes never validated
}
```

No cryptographic check is performed on `proof`. Any byte sequence of any length is accepted.

`update_digest` for `version == 1` then calls:

```rust
// crates/chia-protocol/src/proof_of_space.rs:251-253
self.quality_string()
    .expect("internal error. Can't compute hash of invalid ProofOfSpace")
    .update_digest(digest);
```

`quality_string()` delegates to `chia_pos2::quality_string_from_proof`, which returns `None` when the proof bytes are cryptographically invalid:

```rust
// crates/chia-protocol/src/proof_of_space.rs:162-178
pub fn quality_string(&self) -> Option<Bytes32> {
    if self.version != 1 { return None; }
    let k_size = (self.proof.len() * 8 / 128) as u8;
    let plot_id = self.compute_plot_id().to_bytes();
    chia_pos2::quality_string_from_proof(
        &plot_id, k_size, self.strength, self.proof.as_slice()
    )
    .map(|quality| { ... })
}
```

When `quality_string()` returns `None`, `.expect()` panics. This panic propagates through:

- `RewardChainBlock::update_digest` → `self.proof_of_space.update_digest(digest)` (line 54)
- `FullBlock::update_digest` → `self.reward_chain_block.update_digest(digest)` (line 48)
- `Streamable::hash()` → `self.update_digest(&mut ctx)` (line 71 of `streamable.rs`)

`hash()` is called during consensus block processing on network-received data.

---

### Impact Explanation

Any full node that receives a crafted `FullBlock` or `RewardChainBlock` containing a v2 `ProofOfSpace` with invalid proof bytes and then calls `.hash()` on it — which occurs during normal consensus validation — will panic and crash. Because the panic is deterministic and triggered by a single crafted network message, an attacker can halt all reachable nodes simultaneously, causing a chain halt. This matches the Critical impact: "serialized network object can trigger deterministic consensus divergence, chain halt, or committed state corruption."

---

### Likelihood Explanation

The attack requires only the ability to send a `FullBlock` or `RewardChainBlock` message to a peer — standard unprivileged network access. No keys, governance, or privileged roles are needed. The crafted block must pass structural parsing (trivially satisfied) but does not need to be a valid block in any other sense. The v2 proof format is new (Chia 3.0), so the attack surface is active on any node that has upgraded.

---

### Recommendation

- **Short term**: In `ProofOfSpace::parse` for `version == 1`, validate the proof bytes immediately after parsing by calling `quality_string_from_proof` and returning `Err(Error::InvalidPoS)` if it returns `None`. This mirrors the structural check already present for the pool key/contract exclusivity.
- **Long term**: Replace the `.expect()` in `update_digest` with a graceful error path (e.g., return a sentinel hash or propagate a `Result`), so that a malformed object received from the network cannot cause a process-level panic in any code path.

---

### Proof of Concept

Craft a `ProofOfSpace` with `version = 1`, one of `pool_public_key`/`pool_contract_puzzle_hash` set, and `proof = b"\x00" * 128` (garbage). Embed it in a `RewardChainBlock` inside a `FullBlock`. Send the serialized `FullBlock` to a peer. When the peer calls `FullBlock::from_bytes(...)` followed by `.hash()` (or any code path that calls `update_digest`), the process panics:
