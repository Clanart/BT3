Now I have all the information needed for a thorough analysis. Let me check one more critical detail about how `verify_path` is used vs `verify_path_with_index`:

### Title
Missing shard_id binding check in `set_state_header` allows cross-shard state root substitution during state sync — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id=A` but never verifies that the embedded `ShardChunk`'s own `shard_id` field equals A. Because `verify_path` against `chunk_headers_root` is position-blind (it checks Merkle inclusion, not leaf index), an unprivileged peer can supply shard B's chunk with shard B's valid Merkle proof, pass all validation gates, and cause `set_state_finalize` to apply shard B's `prev_state_root` under shard A's `ShardUId`, permanently corrupting shard A's post-sync state.

---

### Finding Description

**Entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function signature accepts `shard_id: ShardId` as the caller-supplied target shard, then extracts the chunk from the peer-supplied header:

```rust
let chunk = shard_state_header.cloned_chunk();
``` [1](#0-0) 

**Guard 1 — `validate_chunk_proofs`:** Checks the chunk's internal hash, tx root, and receipts root. It never reads `chunk.shard_id()` and has no parameter for the expected shard. [2](#0-1) 

**Guard 2 — `verify_path` against `chunk_headers_root`:** Checks that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's Merkle tree. The primitive used is:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [3](#0-2) 

This is **position-blind**: it only checks that the path from the leaf reaches the root. It does not check which leaf index (i.e., which shard slot) the proof corresponds to. The position-aware variant `verify_path_with_index` exists but is not used here. [4](#0-3) 

The `chunk_headers_root` is built by merklizing `ChunkHashHeight` values for **all** shards in order: [5](#0-4) 

So a valid Merkle proof for shard B's chunk at leaf index B passes `verify_path` even when the node is syncing shard A.

**Guard 3 — receipt proofs (step 4):** Uses the caller-supplied `shard_id` (A) to hash receipts:

```rust
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
``` [6](#0-5) 

This checks incoming receipts *to shard A*, which are public block data. An attacker can construct valid receipt proofs for shard A independently of which chunk body is supplied.

**No check ever performed:** There is no line anywhere in `set_state_header` that compares `chunk.shard_id()` to the `shard_id` parameter. [7](#0-6) 

**Storage:** The accepted header is stored under `StateHeaderKey(shard_id=A, sync_hash)` — keyed by the *requested* shard, not the chunk's actual shard: [8](#0-7) 

**Finalization:** `set_state_finalize` (chain_update) retrieves this header and calls `apply_chunk` with `chunk_header.prev_state_root()` — which is shard B's state root — but with `shard_uid` derived from the requested shard A: [9](#0-8) 

Shard A's `ChunkExtra` is then written with shard B's post-apply state root, permanently corrupting shard A's state.

---

### Impact Explanation

The exact corrupted value is `chunk_header.prev_state_root()` sourced from shard B's chunk header, installed as the authoritative state root for shard A's `ShardUId` in `ChunkExtra`. Every subsequent block applied on top of shard A will use this wrong root, making shard A's trie permanently diverge from the canonical chain. The node will fail to produce or validate any shard A chunk after sync completes.

---

### Likelihood Explanation

State sync data is fetched from peers over the P2P network. Any peer — including an unprivileged one — can respond to state sync header requests. The `StateSyncDownloader` downloads the header from a `source` and forwards it to `set_state_header` for validation: [10](#0-9) 

No validator key or privileged role is required to serve a crafted `ShardStateSyncResponseHeader`. The attacker only needs to be a reachable peer at the time the victim node requests the header. All inputs needed to construct the malicious header (shard B's chunk, its Merkle proof, shard A's receipt proofs) are derivable from public chain data.

---

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the position-blind `verify_path` call for `chunk_headers_root` with `verify_path_with_index`, supplying the shard's index in the layout as the expected leaf position, to make the Merkle proof check position-aware. [11](#0-10) 

---

### Proof of Concept

Build a two-node integration test (similar to the existing state sync test at `integration-tests/src/tests/client/process_blocks.rs:1971`): [12](#0-11) 

1. Run a chain with ≥2 shards.
2. From node 0, obtain the legitimate `ShardStateSyncResponseHeader` for shard 0 and for shard 1.
3. Construct a malicious header: take shard 1's `chunk` and `chunk_proof` (valid Merkle proof for leaf index 1), but use shard 0's `incoming_receipts_proofs`, `root_proofs`, and `state_root_node`.
4. Call `node1.chain.state_sync_adapter.set_state_header(shard_id=0, sync_hash, malicious_header)`.
5. Assert the call returns `Err(...)` — currently it returns `Ok(())`, demonstrating the missing guard.
6. If `Ok(())`, proceed to `set_state_finalize` and assert that the resulting `ChunkExtra` state root for shard 0 equals the legitimate shard 0 state root — currently it will equal shard 1's state root, confirming corruption.

```rust
pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
