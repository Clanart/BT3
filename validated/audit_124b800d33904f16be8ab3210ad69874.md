### Title
UTXO Chain (BTC/Zcash) Address Non-Canonicalization Causes Permanent Fund Lock — (`near/omni-bridge/src/btc.rs`, `near/omni-types/src/lib.rs`)

---

### Summary

`OmniAddress::Btc` and `OmniAddress::Zcash` store recipient addresses as raw, unnormalized `String` values. When a user initiates a BTC or Zcash bridge transfer with a non-canonical address form (e.g., uppercase bech32 such as `TB1QXXX…`), the bridge stores it verbatim. Later, `submit_transfer_to_utxo_chain_connector` performs a **case-sensitive string equality check** between the stored address and the relayer-supplied `target_btc_address`. If the relayer uses the canonical lowercase form (`tb1qxxx…`), the check fails, the function panics, and the user's funds are permanently locked in the bridge with no cancellation path.

---

### Finding Description

`UTXOChainAddress` is defined as a plain `String` alias:

```rust
pub type UTXOChainAddress = String;
``` [1](#0-0) 

Both the `FromStr` parser and the `new_from_slice` constructor accept any UTF-8 string for BTC/Zcash addresses without normalization:

```rust
"btc"   => Ok(Self::Btc(recipient.to_string())),
"zcash" => Ok(Self::Zcash(recipient.to_string())),
``` [2](#0-1) 

```rust
ChainKind::Btc => Ok(Self::Btc(
    String::from_utf8(address.to_vec())
        .map_err(|e| format!("Invalid BTC address: {e}"))?,
)),
ChainKind::Zcash => Ok(Self::Zcash(
    String::from_utf8(address.to_vec())
        .map_err(|e| format!("Invalid ZCash address: {e}"))?,
)),
``` [3](#0-2) 

`OmniAddress` derives `PartialEq`/`Eq` via Rust's standard derive, meaning equality for `Btc`/`Zcash` variants is a **byte-for-byte string comparison**:

```rust
#[near(serializers=[borsh])]
#[derive(Debug, Clone, Hash, PartialEq, Eq)]
pub enum OmniAddress {
    ...
    Btc(UTXOChainAddress),
    Zcash(UTXOChainAddress),
    ...
}
``` [4](#0-3) 

In `submit_transfer_to_utxo_chain_connector`, the stored address is compared directly against the relayer-supplied `target_btc_address`:

```rust
if let Some(btc_address) = transfer.message.recipient.get_utxo_address() {
    if let TokenReceiverMessage::Withdraw { target_btc_address, .. } = message {
        require!(
            btc_address == target_btc_address,
            BridgeError::IncorrectTargetUtxoAddress.as_ref()
        );
``` [5](#0-4) 

Bitcoin bech32 addresses (BIP-0173) are **case-insensitive by specification**. `TB1QXXX…` and `tb1qxxx…` represent the same address. A user who provides an uppercase bech32 recipient (e.g., copied from a QR code or a wallet that outputs uppercase) will have their transfer stored with the uppercase string. A relayer using the canonical lowercase form will fail the `require!` check, causing a panic. The transfer remains in `pending_transfers` indefinitely.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

The user's tokens are transferred into the bridge at `ft_transfer_call` time. The pending transfer entry is only removed inside `submit_transfer_to_utxo_chain_connector` after the address check passes. If the check panics, the entry is never removed and no cancellation or refund function exists for UTXO-chain pending transfers. The user's bridged tokens are permanently locked.

---

### Likelihood Explanation

**Medium-High.**

- Bitcoin bech32 addresses are case-insensitive per BIP-0173; many wallets and block explorers output uppercase bech32.
- The bridge performs zero validation on the BTC/Zcash address string at ingestion — any UTF-8 string is accepted.
- Relayer implementations canonicalize to lowercase (the standard form), creating a systematic mismatch whenever a user supplies uppercase.
- No privileged access is required; any bridge user initiating a BTC/Zcash transfer can trigger this.

---

### Recommendation

Normalize BTC and Zcash address strings to lowercase at the point of ingestion — in both `OmniAddress::from_str` and `OmniAddress::new_from_slice` — before storing them:

```rust
"btc"   => Ok(Self::Btc(recipient.to_lowercase())),
"zcash" => Ok(Self::Zcash(recipient.to_lowercase())),
```

Apply the same normalization in `new_from_slice` for `ChainKind::Btc` and `ChainKind::Zcash`. This ensures that regardless of the case supplied by the user, the stored address and the relayer-supplied address will always compare equal for the same logical destination.

---

### Proof of Concept

1. User calls `ft_transfer_call` on the BTC token contract with:
   ```json
   {
     "recipient": "btc:TB1QXXX...",
     "fee": "100",
     "native_token_fee": "0"
   }
   ```
   The bridge stores `OmniAddress::Btc("TB1QXXX...")` in `pending_transfers`.

2. Relayer reads the pending transfer and constructs the withdrawal message using the canonical lowercase form:
   ```json
   {
     "Withdraw": {
       "target_btc_address": "tb1qxxx...",
       "input": [...],
       "output": [...]
     }
   }
   ```

3. Relayer calls `submit_transfer_to_utxo_chain_connector(transfer_id, msg, ...)`.

4. Inside the function, `btc_address` = `"TB1QXXX..."` (from stored transfer), `target_btc_address` = `"tb1qxxx..."` (from relayer msg).

5. `require!(btc_address == target_btc_address, ...)` evaluates `"TB1QXXX..." == "tb1qxxx..."` → **false** → panic with `IncorrectTargetUtxoAddress`.

6. The transfer remains in `pending_transfers` permanently. The user's tokens are irrecoverably locked. [5](#0-4) [2](#0-1)
