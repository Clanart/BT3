### Title
`fin_transfer_send_tokens_callback` Silently Ignores `ft_transfer`/`ft_transfer_call` Failures, Permanently Locking Bridged Funds - (File: near/omni-bridge/src/lib.rs)

---

### Summary

The NEAR bridge contract's `fin_transfer_send_tokens_callback` does not check whether the underlying `ft_transfer` or `ft_transfer_call` promise actually succeeded. When either call fails, `is_refund_required` returns `false`, the callback proceeds down the "success" path, the transfer is permanently recorded in `finalised_transfers`, lock-accounting is not reverted, and the user's tokens are irrecoverably locked with no retry path.

---

### Finding Description

**Vulnerability class:** Callback/state desync — a cross-contract call can fail due to external state changes, but the callback has no fallback mechanism, causing permanent fund lock.

**Root cause — `is_refund_required`:** [1](#0-0) 

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                    amount.0 == 0
                } else {
                    false   // unexpected result → treated as success
                }
            }
            Err(_) => false,   // ← CRITICAL: failed promise treated as success
        }
    } else {
        false   // ← CRITICAL: plain ft_transfer failure always treated as success
    }
}
```

Two distinct failure modes are silently swallowed:

1. **Plain `ft_transfer` (non-deployed token, no message):** `is_ft_transfer_call = false` → always returns `false` regardless of whether the promise succeeded or failed.
2. **`ft_transfer_call` (any token with a message):** if the promise result is `Err(_)` (i.e., `ft_on_transfer` panicked or the call failed), returns `false` — treated as success.

**State committed before `send_tokens` is called:**

`process_fin_transfer_to_near` calls `add_fin_transfer` first, permanently inserting the transfer ID into `finalised_transfers`: [2](#0-1) 

This insertion cannot be undone unless `remove_fin_transfer` is explicitly called — which only happens in the refund path of `fin_transfer_send_tokens_callback`: [3](#0-2) 

The refund path also calls `revert_lock_actions` to restore the `locked_tokens` counter. But because `is_refund_required` returns `false` for both failure modes above, the refund path is never reached on failure.

**`send_tokens` — the two failing paths:** [4](#0-3) 

- Non-deployed token, no message → `ft_transfer` (line 2106). If this promise fails, `is_refund_required(false)` = `false`.
- Any token with message → `ft_transfer_call` (line 2116). If `ft_on_transfer` panics, promise result is `Err`, `is_refund_required(true)` = `false`.

**The callback on failure goes to the "success" path:** [5](#0-4) 

It sends fees to the fee recipient and logs `FinTransferEvent` — as if the transfer succeeded — while the user's tokens are stuck in the bridge contract with no recovery mechanism.

---

### Impact Explanation

**For deployed tokens with a message (`ft_transfer_call` path):**
- `mint(recipient, amount, Some(msg))` mints tokens to the bridge contract, then calls `ft_transfer_call`.
- If `ft_on_transfer` panics, NEAR's runtime refunds the tokens back to the bridge contract.
- The bridge's callback treats this as success: `finalised_transfers` permanently records the transfer ID, no retry is possible, and the minted tokens are stuck in the bridge's own account on the token contract with no recovery function.

**For non-deployed tokens without a message (`ft_transfer` path):**
- The bridge holds real locked tokens. `unlock_tokens_if_needed` decrements `locked_tokens` before `send_tokens` is called.
- If `ft_transfer` fails, the callback does not call `revert_lock_actions`, so `locked_tokens` is permanently under-counted while the tokens remain in the bridge's account — breaking bridge collateralization accounting.
- The transfer ID is in `finalised_transfers`; no retry is possible.

In both cases: **user funds are permanently frozen with no recovery path** — matching the "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds" impact class.

---

### Likelihood Explanation

**`ft_transfer_call` failure (deployed token + message):** A recipient contract whose `ft_on_transfer` panics — whether due to a bug, an upgrade after the transfer was initiated on EVM, or deliberate self-harm — triggers this path. The recipient is specified in the proof and cannot be changed after the EVM-side `initTransfer`. Any panic in `ft_on_transfer` permanently locks the tokens.

**`ft_transfer` failure (non-deployed token, no message):** On NEAR, each promise is a separate receipt. Between `check_or_pay_ft_storage` (which deposits storage for the recipient) and the actual `ft_transfer` execution, the recipient can call `storage_unregister` on the token contract (valid because their balance is 0 at that point). This causes `ft_transfer` to fail with no storage registered, triggering the silent-success path.

Both paths are reachable by an unprivileged bridge user (the recipient of a cross-chain transfer).

---

### Recommendation

In `fin_transfer_send_tokens_callback`, explicitly check the promise result for the plain `ft_transfer` case, and treat a failed `ft_transfer_call` promise (`Err(_)`) as a refund condition rather than success:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
        Err(_) => true,  // any promise failure → refund
        Ok(value) if is_ft_transfer_call => {
            near_sdk::serde_json::from_slice::<U128>(&value)
                .map(|amount| amount.0 == 0)
                .unwrap_or(false)
        }
        Ok(_) => false,
    }
}
```

This mirrors the fix applied in the referenced Wagmi report: add a try-catch (here, an `Err` arm) so that any external call failure triggers the refund/revert path instead of silently succeeding.

---

### Proof of Concept

1. Deploy a NEAR contract `MaliciousReceiver` whose `ft_on_transfer` always panics.
2. On EVM, call `initTransfer` for a deployed bridge token, specifying `recipient = MaliciousReceiver` and a non-empty `message`.
3. Wait for the MPC signature and have a relayer call `fin_transfer` on NEAR.
4. `process_fin_transfer_to_near` runs:
   - `add_fin_transfer` inserts the transfer ID into `finalised_transfers` ← permanent
   - `send_tokens` calls `mint(MaliciousReceiver, amount, Some(msg))`
   - Token contract mints to bridge, calls `ft_transfer_call(MaliciousReceiver, ...)`
   - `MaliciousReceiver.ft_on_transfer` panics → promise result = `Failed`
5. `fin_transfer_send_tokens_callback` is called:
   - `is_refund_required(true)` → `Err(_) => false` → returns `false`
   - Goes to success path, logs `FinTransferEvent`, sends fee
6. State after: transfer ID in `finalised_transfers` (no retry possible), minted tokens stuck in bridge's account on token contract, no `remove_fin_transfer` called, no `revert_lock_actions` called.
7. Calling `fin_transfer` again with the same proof panics with `TransferAlreadyFinalised`.
8. **Tokens are permanently locked.** [6](#0-5) [1](#0-0) [7](#0-6) [2](#0-1)

    pub fn fin_transfer_send_tokens_callback(
        &mut self,
        #[serializer(borsh)] transfer_message: TransferMessage,
        #[serializer(borsh)] fee_recipient: &AccountId,
        #[serializer(borsh)] is_ft_transfer_call: bool,
        #[serializer(borsh)] storage_owner: &AccountId,
        #[serializer(borsh)] lock_actions: Vec<LockAction>,
    ) {
        let token = self.get_token_id(&transfer_message.token);

        if Self::is_refund_required(is_ft_transfer_call) {
            self.burn_tokens_if_needed(
                token.clone(),
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
            );

            self.revert_lock_actions(&lock_actions);

            self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);

            env::log_str(
                &OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string(),
            );
        } else {
            // Send fee to the fee recipient
            if transfer_message.fee.fee.0 > 0 {
                if self.is_deployed_token(&token) {
                    ext_token::ext(token)
                        .with_static_gas(MINT_TOKEN_GAS)
                        .mint(fee_recipient.clone(), transfer_message.fee.fee, None)
                        .detach();
                } else {
                    ext_token::ext(token)
                        .with_attached_deposit(ONE_YOCTO)
                        .with_static_gas(FT_TRANSFER_GAS)
                        .ft_transfer(fee_recipient.clone(), transfer_message.fee.fee, None)
                        .detach();
                }
            }

            if transfer_message.fee.native_fee.0 > 0 {
                let native_token_id = self.get_native_token_id(transfer_message.get_origin_chain());

                ext_token::ext(native_token_id)
                    .with_static_gas(MINT_TOKEN_GAS)
                    .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
                    .detach();
            }

            env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
        }
    }
