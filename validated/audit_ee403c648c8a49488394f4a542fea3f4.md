Audit Report

## Title
Excess ETH Sent for WETH Exact-Output Swaps Is Not Refunded and Is Permanently Stealable — (`metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
`PeripheryPayments.pay()` wraps only the exact ETH amount the pool requests (`value`) and leaves any surplus native ETH in the router contract. `refundETH()` is permissionless and sends the entire contract ETH balance to `msg.sender`, not the original depositor. Neither `exactOutputSingle` nor `exactOutput` calls `refundETH()` automatically, so any user who over-sends ETH (which is structurally required for exact-output swaps) loses the surplus to any third-party caller.

## Finding Description
In `PeripheryPayments.pay()`, the WETH branch wraps exactly `value` ETH and leaves `address(this).balance - value` untouched: [1](#0-0) 

`refundETH()` is a public, permissionless function that sends the full contract ETH balance to `msg.sender`: [2](#0-1) 

`exactOutputSingle` enforces the `amountInMaximum` guard and then clears context — with no ETH refund: [3](#0-2) 

`exactOutput` has the identical gap: [4](#0-3) 

`multicall` (lines 39–44) uses `delegatecall` and preserves `msg.sender`, so a user *could* batch `exactOutputSingle` + `refundETH()` — but this is neither enforced nor documented in the swap functions or their NatSpec. An ordinary single-call invocation of `exactOutputSingle` or `exactOutput` with ETH leaves the surplus permanently exposed.

## Impact Explanation
Any user who calls `exactOutputSingle` or `exactOutput` with `tokenIn == WETH` and `msg.value > actualAmountIn` loses the difference immediately. A MEV bot or any observer can call `refundETH()` in the same or next block and receive the stranded ETH. This is a direct, unconditional loss of user principal — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation
Exact-output swaps are a standard use case. Users must set `amountInMaximum` conservatively above the expected cost to avoid slippage reverts, making surplus ETH structurally guaranteed whenever the pool does not consume the full maximum. No special conditions, malicious tokens, or privileged roles are required — any ordinary user calling `exactOutputSingle` with native ETH triggers the loss.

## Recommendation
Add an automatic ETH refund at the tail of every payable exact-output entry point, before returning:

```solidity
// after _clearExpectedCallbackPool() in exactOutputSingle and exactOutput:
uint256 ethLeft = address(this).balance;
if (ethLeft > 0) _transferETH(msg.sender, ethLeft);
```

Alternatively, enforce that callers must batch `refundETH()` via `multicall` by documenting it explicitly in NatSpec and/or adding a check that reverts if `msg.value > amountIn` after the swap settles.

## Proof of Concept
1. Pool `P` has WETH as `token0` and USDC as `token1`. Current price implies 1 WETH → 2000 USDC.
2. Alice calls `exactOutputSingle` with `amountOut = 1800 USDC`, `amountInMaximum = 1 ETH`, `msg.value = 1 ETH`.
3. Pool executes; actual cost is `0.9 ETH`. `pay()` wraps `0.9 ETH` and sends it to the pool. `0.1 ETH` remains in the router.
4. `exactOutputSingle` checks `0.9 ETH <= 1 ETH` ✓, calls `_clearExpectedCallbackPool()`, and returns — without refunding the `0.1 ETH`.
5. Bob observes the transaction (mempool or next block), calls `refundETH()`, and receives Alice's `0.1 ETH`.

Alice loses `0.1 ETH` with no recourse once the swap transaction is mined without a batched `refundETH()`.
