Now I have enough context to analyze the analog. Let me look at the critical discrepancy I found between `get_state_response_part` and `compute_state_response_header`.### Title
Wrong Epoch's Shard Layout Used to Index `prev_block.chunks()` in `get_state_response_part` — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` derives the chunk index for `prev_block` using the **sync block's (new) epoch** shard layout, while `prev_block` belongs to the **previous epoch** and must be indexed with the **previous epoch's** shard layout. After any resharding event, the two layouts diverge: new child shard IDs carry indices that are out-of-range for the old block's chunk array, causing every state-part request for a new child shard to return `Error::InvalidShardId` (served as an empty response). Any syncing node that needs to reconstruct a new child shard's state after resharding is permanently blocked.

---

### Finding Description

`compute_state_response_header` (the sibling function that builds the state-sync header) explicitly separates the two layouts:

```rust
// compute_state_response_header — CORRECT
let sync_block_epoch_id = sync_block_header.epoch_id();          // new epoch
let prev_epoch_id       = sync_prev_block.header().epoch_id();   // old epoch
let prev_shard_layout   = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index    = prev_shard_layout.get_shard_index(shard_id)?;  // old layout
let chunk_header = sync_prev_block.chunks().get(prev_shard_index)…;      // old block
```

`get_state_response_part` does not:

```rust
// get_state_response_part — BUG
let epoch_id    = block.header().epoch_id();                     // new epoch only
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?; // new layout
let prev_block  = self.chain_store.get_block(header.prev_hash())?; // old block
let shard_index = shard_layout.get_shard_index(shard_id)?;      // index from NEW layout
let state_root  = prev_block.chunks().get(shard_index)           // applied to OLD block
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) [2](#0-1) 

After a shard split (e.g., old layout `[s0, s1, s2]` → new layout `[s0, s1, s2, s3]`):

| Request for | `shard_layout.get_shard_index(shard_id)` | `prev_block.chunks().get(index)` | Result |
|---|---|---|---|
| `s3` (new child) | 3 (new layout) | `None` — old block has only 3 chunks | `Error::InvalidShardId(s3)` → empty response |
| `s2` (parent, same index in both layouts) | 2 | parent chunk's `prev_state_root` | accidentally correct |

The `validate_sync_hash` gate in `StateRequestActor` only checks that the sync hash belongs to a known recent epoch; it does not prevent a request for a new child shard ID, which passes the `shard_ids.contains(&shard_id)` check because that check also uses the new epoch's shard IDs. ---

### Impact Explanation

Any node performing state sync (catchup or far-behind sync) after a resharding event requests state parts for every shard in the new epoch, including new child shards. The serving node's `get_state_response_part` returns an empty response for every part of every new child shard. The syncing node retries indefinitely but can never obtain valid parts, so it cannot reconstruct the child shard's state and cannot participate in the new epoch as a chunk producer or validator for that shard. This breaks the state-sync reconstruction invariant: the `state_root` embedded in the served `StatePart` must match the `chunk_prev_state_root` in the corresponding `ShardStateSyncResponseHeader`, but here the serving side cannot even produce a part — the commitment is broken at the source.

**Severity: High** — state sync for new child shards is permanently broken after resharding; affected validators miss the epoch and lose rewards; the network's ability to onboard new validators post-resharding is impaired.

---

### Likelihood Explanation

The bug fires on every resharding event. NEAR has already executed mainnet resharding (evidenced by `MAINNET_RESHARDING_BLOCK_HASHES` in `chain/chain/src/resharding/migrations.rs`) and dynamic resharding is an active protocol feature. Any node that is more than one epoch behind at the time of a shard split and attempts state sync will hit this path. No special privileges are required: the attacker (or ordinary syncing node) only needs to know the canonical `sync_hash` (publicly derivable from block headers) and the new child shard's ID (publicly derivable from the epoch's shard layout). [5](#0-4) 

---

### Recommendation

Mirror the two-layout pattern from `compute_state_response_header` inside `get_state_response_part`:

```rust
// After fetching prev_block:
let prev_epoch_id    = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index      = prev_shard_layout.get_shard_index(shard_id)?;
let state_root       = prev_block.chunks().get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

The existing `shard_ids.contains(&shard_id)` guard (which correctly uses the new epoch's shard IDs) should be kept; only the index derivation must switch to the previous epoch's layout.

---

### Proof of Concept

**Setup:** Resharding splits shard `s2` into `s2` (retained) and `s3` (new child). New epoch starts. A syncing node calls `StateRequestPart { shard_id: s3, sync_hash: <first block of new epoch>, part_id: 0 }`.

**Trace through `get_state_response_part`:**

1. `block = get_block(sync_hash)` — block in new epoch. [6](#0-5) 
2. `epoch_id = block.header().epoch_id()` — new epoch.
3. `shard_layout = get_shard_layout(new_epoch)` — layout `[s0, s1, s2, s3]`.
4. `shard_ids.contains(&s3)` → `true` (passes guard). [4](#0-3) 
5. `prev_block = get_block(sync_hash.prev_hash)` — last block of old epoch, has chunks `[c0, c1, c2]` (3 entries).
6. `shard_index = shard_layout.get_shard_index(s3)` → `3` (index in new layout). [7](#0-6) 
7. `prev_block.chunks().get(3)` → `None` (old block has only 3 chunks, indices 0–2). [8](#0-7) 
8. Returns `Err(Error::InvalidShardId(s3))`.
9. `StateRequestActor` converts this to an empty `StatePartOrHeader` response. [9](#0-8) 
10. Syncing node receives empty response, retries, and loops forever — state sync for `s3` never completes.

**Contrast:** `compute_state_response_header` for the same `shard_id = s3` would call `prev_shard_layout.get_shard_index(s3)` which itself returns `EpochError` (shard not in old layout), surfacing the error cleanly and consistently — but `get_state_response_part` never reaches that check because it uses the wrong layout and gets a different, silent failure mode (out-of-bounds chunk access).
