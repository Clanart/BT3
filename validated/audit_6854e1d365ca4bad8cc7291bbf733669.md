### Title
Unchecked Multiplication Overflow in `denormalize_amount` Permanently Freezes Bridged Funds — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`denormalize_amount` in the NEAR bridge contract performs a plain `u128` multiplication with no overflow guard. Because the release profile sets `overflow-checks = true`, any overflow causes a WASM trap. An unprivileged user can initiate a transfer on EVM with a large-but-valid `uint128` amount for any token whose `origin_decimals > decimals`, causing every subsequent `fin_transfer_callback` call for that transfer to panic. The tokens are already burned/locked on EVM, so the funds are permanently frozen.

---

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← plain multiplication, no checked_mul
}
``` [1](#0-0) 

The workspace release profile explicitly enables overflow trapping:

```toml
[profile.release]
overflow-checks = true
``` [2](#0-1) 

`denormalize_amount` is called unconditionally inside `fin_transfer_callback` before any state is written:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
...
fee: Self::denormalize_fee(&init_transfer.fee, decimals),
``` [3](#0-2) 

`denormalize_fee` also delegates to `denormalize_amount`:

```rust
fn denormalize_fee(fee: &Fee, decimals: Decimals) -> Fee {
    Fee {
        fee: U128(Self::denormalize_amount(fee.fee.0, decimals)),
``` [4](#0-3) 

The EVM `InitTransfer` event carries `uint128 amount` and `uint128 fee`:

```solidity
event InitTransfer(
    address indexed sender,
    address indexed tokenAddress,
    uint64 indexed originNonce,
    uint128 amount,
    uint128 fee,
    ...
``` [5](#0-4) 

The EVM `initTransfer` function accepts any `uint128 amount` and only checks `fee < amount`:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    uint128 fee,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    if (fee >= amount) { revert InvalidFee(); }
``` [6](#0-5) 

**Overflow condition:** For a token with `origin_decimals = 24` and `decimals = 18` (`diff_decimals = 6`), any `amount > u128::MAX / 10^6 ≈ 3.4 × 10^32` overflows. The full `uint128` range goes up to `≈ 3.4 × 10^38`, so the overflow window spans six orders of magnitude of valid EVM amounts. The same overflow applies to the fee field via `denormalize_fee`.

The same vulnerable call also appears in `claim_fee_callback`:

```rust
let denormalized_amount = Self::denormalize_amount(
    fin_transfer.amount.0,
    self.token_decimals
        .get(&token_address)
        .near_expect(BridgeError::TokenDecimalsNotFound),
);
``` [7](#0-6) 

---

### Impact Explanation

When `fin_transfer_callback` panics due to overflow, the NEAR transaction reverts with no state change. The proof for that specific EVM nonce is not marked as used, so the callback can be retried — but it will always panic for the same amount. The tokens were already burned or locked on EVM at `initTransfer` time. There is no refund path on EVM once `initTransfer` succeeds. The result is **permanent, irrecoverable freezing of user funds**.

This matches the allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

- Any token registered with `origin_decimals > decimals` is vulnerable (e.g., a 24-decimal origin token bridged to an 18-decimal NEAR token).
- The attacker only needs to call `initTransfer` on EVM with a valid `uint128` amount above the overflow threshold. No privileged access is required.
- The EVM bridge imposes no upper bound on `amount` beyond the `uint128` type limit.
- The attacker loses the tokens they transfer (they are burned/locked), but the attack permanently freezes those funds and blocks the NEAR finalization path for that nonce.
- A well-funded attacker or one who already holds a large position can execute this at any time.

---

### Recommendation

Replace the plain multiplication in `denormalize_amount` with a checked variant that returns an error instead of panicking:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount
        .checked_mul(10_u128.pow(diff_decimals))
        .near_expect(BridgeError::AmountOverflow)
}
```

Additionally, enforce an upper bound on `amount` in `fin_transfer_callback` before calling `denormalize_amount`, rejecting proofs whose amounts would overflow after scaling. Consider also adding a cap in the EVM `initTransfer` function to reject amounts that would overflow after NEAR-side denormalization, or document the safe maximum per token.

---

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. On EVM, call `initTransfer(tokenAddress, amount=2**127, fee=1, ...)`. This is a valid `uint128` value; EVM accepts it and burns/locks the tokens.
3. A relayer submits the proof to NEAR `fin_transfer`.
4. `fin_transfer_callback` executes:
   - `denormalize_amount(2^127, {decimals:18, origin_decimals:24})`
   - `= 2^127 * 10^6`
   - `≈ 1.7 × 10^38 * 10^6 = 1.7 × 10^44`
   - `u128::MAX ≈ 3.4 × 10^38` → **overflow**
5. With `overflow-checks = true`, the WASM runtime traps. The callback panics.
6. The transfer nonce is never marked finalized. The callback will panic on every retry.
7. The user's tokens are permanently frozen on EVM with no recovery path.
