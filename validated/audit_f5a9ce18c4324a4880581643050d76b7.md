### Title
Receipt Proof Commitment Desync at Resharding Boundary Breaks Chunk State Witness Validation — (File: `chain/chain/src/stateless_validation/state_witness.rs`)

---

### Summary

At a shard-split resharding boundary, `collect_source_receipt_proofs` calls `get_incoming_receipts_for_shard` with a filter that removes receipts targeting the sibling child shard, but retains the Merkle proof that was computed over the **full, unfiltered** parent-shard receipt set. This breaks the hash/root binding: the proof no longer verifies against the `prev_outgoing_receipts_root` committed in the block header. Every validator that receives the state witness for the first chunk of a child shard after resharding will reject it with a proof-verification failure, permanently preventing chunk endorsement at every resharding boundary.

The codebase explicitly acknowledges this as an open bug: [1](#0-0) 

---

### Finding Description

**Invariant that must hold.** For each `ReceiptProof(receipts, ShardProof { proof, … })` included in a `ChunkStateWitness`, the following must be true:

```
verify_path(prev_outgoing_receipts_root, proof, hash(receipts)) == true
```

where `prev_outgoing_receipts_root` is the value committed in the block header.

**How the invariant is broken at a resharding boundary.**

In epoch N, parent shard P produces outgoing receipts `R = R_L ∪ R_R`, where `R_L` targets child shard `S_L` and `R_R` targets child shard `S_R`. The Merkle proof stored in each `ReceiptProof` is computed over `R` (the full set), and the corresponding root `root_R = merkle_root(R)` is committed in the block header as `prev_outgoing_receipts_root`.

In epoch N+1, when the chunk producer for `S_L` calls `collect_source_receipt_proofs` to build the state witness for the first chunk of `S_L`:

1. `get_incoming_receipts_for_shard` is invoked with `ReceiptFilter::TargetShard` (or equivalent filtering).
2. The function strips `R_R` from the stored `ReceiptProof`, returning only `R_L`.
3. The `ShardProof.proof` field is **not recomputed** — it still proves `hash(R)`, not `hash(R_L)`.
4. The state witness is emitted containing `ReceiptProof(R_L, proof_for_R)`.

When a validator receives this witness and runs pre-validation:

```rust
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts)); // hash(R_L)
if !verify_path(*root, proof, &receipts_hash) {                               // root=root_R, proof=proof_for_R
    return Err(…);  // always fails: proof_for_R proves hash(R), not hash(R_L)
}
``` [2](#0-1) 

The proof verification always fails because `hash(R_L) ≠ hash(R)`. The witness is rejected, the chunk cannot be endorsed, and the block cannot be finalized with a new chunk for `S_L`.

The exact corrupted value is `ReceiptProof.shard_proof` inside the `source_receipt_proofs` map of `ChunkStateWitness` — it carries a Merkle path that was computed over the parent shard's full receipt set but is presented alongside a filtered subset. [3](#0-2)

### Impact Explanation

**Severity: High**

At every shard-split resharding boundary, the state witness for the first new chunk of each child shard is structurally invalid. No honest validator can endorse it. The network either:

- Treats the chunk as missing (empty), silently dropping all receipts that were supposed to be applied in that chunk — a permanent state divergence between what was executed and what was endorsed; or
- Stalls finalization for those shards until the chunk producer rotates and a new producer somehow avoids the same code path (it cannot — the bug is deterministic).

Because the bug fires for **every** child shard on **every** resharding event, a single resharding splits the network's ability to process receipts across the boundary, affecting all accounts whose receipts cross the epoch/shard boundary.

---

### Likelihood Explanation

The bug is deterministic and fires unconditionally at every shard-split resharding boundary when stateless validation is active. Static resharding events that occurred in production (1→4→5 shards) predated stateless validation, so the bug has not yet been triggered. However:

- Any future static resharding (protocol-version-gated shard layout change) will trigger it.
- Dynamic resharding (`ProtocolFeature::DynamicResharding`) triggers it automatically once shard memory exceeds the configured threshold — no privileged action required. [5](#0-4) 

The resharding threshold is a protocol parameter, not an operator secret. The bug requires no attacker; it fires on normal protocol operation.

---

### Recommendation

As the TODO comment itself prescribes: collect the **original, unfiltered** `ReceiptProof` (with its valid Merkle proof over the full parent-shard receipt set) and include it verbatim in the state witness. After the validator verifies the proof against the committed root, it should then filter out the receipts targeting the sibling shard before applying them to the state transition. The proof must be computed over the same receipt set that is hashed for verification — filtering must happen **after** proof verification, not before proof inclusion.

---

### Proof of Concept

1. Run a network with stateless validation enabled and a shard layout that will change (static or dynamic resharding configured).
2. Advance to the resharding epoch boundary; shard P splits into `S_L` and `S_R`.
3. Observe the chunk producer for `S_L` call `create_state_witness` → `collect_source_receipt_proofs` → `get_incoming_receipts_for_shard(ReceiptFilter::TargetShard)`.
4. Inspect the emitted `ChunkStateWitness.source_receipt_proofs`: each `ReceiptProof` for blocks in epoch N contains only `R_L` receipts but a Merkle proof computed over `R = R_L ∪ R_R`.
5. Observe every validator reject the witness at the `verify_path` check in pre-validation.
6. Confirm the chunk for `S_L` at height `epoch_N+1_start` is never endorsed. [1](#0-0) [6](#0-5)

pub fn get_incoming_receipts_for_shard(
    chain_store: &ChainStoreAdapter,
    epoch_manager: &dyn EpochManagerAdapter,
    target_shard_id: ShardId,
    target_shard_layout: &ShardLayout,
    block_hash: CryptoHash,
    last_chunk_height_included: BlockHeight,
    receipts_filter: ReceiptFilter,
) -> Result<Vec<ReceiptProofResponse>, Error> {
    let _span =
            tracing::debug_span!(target: "chain", "get_incoming_receipts_for_shard", ?target_shard_id, ?block_hash, last_chunk_height_included).entered();

    let mut ret = vec![];

    let mut current_shard_id = target_shard_id;
    let mut current_block_hash = block_hash;
    let mut current_shard_layout = target_shard_layout.clone();

    loop {
        let header = chain_store.get_block_header(&current_block_hash)?;

        if header.height() < last_chunk_height_included {
            panic!("get_incoming_receipts_for_shard failed");
        }

        if header.height() == last_chunk_height_included {
            break;
        }

        let prev_hash = header.prev_hash();
        let prev_shard_layout = epoch_manager.get_shard_layout_from_prev_block(prev_hash)?;

        if prev_shard_layout != current_shard_layout {
            let parent_shard_id = current_shard_layout.get_parent_shard_id(current_shard_id)?;
            tracing::info!(
                target: "chain",
                version = current_shard_layout.version(),
                prev_version = prev_shard_layout.version(),
                ?current_shard_id,
                ?parent_shard_id,
                "crossing epoch boundary with shard layout change, updating shard id"
            );
            current_shard_id = parent_shard_id;
            current_shard_layout = prev_shard_layout;
        }

        let maybe_receipts_proofs =
            chain_store.get_incoming_receipts(&current_block_hash, current_shard_id);
        let receipts_proofs = match maybe_receipts_proofs {
            Ok(receipts_proofs) => {
                tracing::debug!(
                    target: "chain",
                    "found receipts from block with missing chunks",
                );
                receipts_proofs
            }
            Err(err) => {
                tracing::debug!(
                    target: "chain",
                    ?err,
                    "could not find receipts from block with missing chunks"
                );

                // This can happen when all chunks are missing in a block
                // and then we can safely assume that there aren't any
                // incoming receipts. It would be nicer to explicitly check
                // that condition rather than relying on errors when reading
                // from the db.
                Arc::new(vec![])
            }
        };

        let filtered_receipt_proofs = match receipts_filter {
            ReceiptFilter::All => receipts_proofs,
            ReceiptFilter::TargetShard => Arc::new(filter_incoming_receipts_for_shard(
                &target_shard_layout,
                target_shard_id,
                receipts_proofs,
            )?),
        };

        ret.push(ReceiptProofResponse(current_block_hash, filtered_receipt_proofs));
        current_block_hash = *prev_hash;
    }

    Ok(ret)
