Audit Report

## Title
Unprivileged caller can steal stranded ETH via `PeripheryPayments.refundETH` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is `external payable` with no access control and unconditionally transfers `address(this).balance` to `msg.sender`. ETH accumulates on the router whenever a user calls any `payable` swap function with `msg.value` exceeding the actual swap cost and does not include `refundETH` in the same atomic `multicall`. Any third party can then drain that ETH in a subsequent transaction.

## Finding Description
`refundETH()` at lines 58–63 of `PeripheryPayments.sol` sends the router's full ETH balance to whoever calls it: [1](#0-0) 

The `receive()` guard at lines 32–34 only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in any `payable` function: [2](#0-1) 

Inside `pay()`, when `token == WETH` and `nativeBalance >= value`, only `value` wei is wrapped and forwarded; the remainder stays on the contract: [3](#0-2) 

`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` are all `payable` and perform no automatic ETH refund: [4](#0-3) 

The design intent — evidenced by `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` — is that users bundle `refundETH()` as the last call in their `multicall`, but this is never enforced: [5](#0-4) 

**Exploit path:**
1. Victim calls `exactInputSingle{value: 2 ether}(amountIn: 1000, tokenIn: WETH, ...)` directly (or via `multicall` without appending `refundETH`).
2. `pay()` wraps 1000 wei; `~2 ether − 1000 wei` remains on the router after the transaction settles.
3. Attacker calls `refundETH()` in a separate transaction; the entire router ETH balance is transferred to the attacker.

No privileged role, malicious pool, or non-standard token is required.

## Impact Explanation
Direct loss of user ETH principal. The attacker receives ETH that belongs to the victim. Loss magnitude equals `msg.value − amountIn` for the victim's transaction, which can be arbitrarily large. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
Moderate. `exactInputSingle` and the other swap functions are `payable` and callable directly without `multicall`. Users who call them directly with excess ETH, or who construct a `multicall` without a trailing `refundETH`, are immediately vulnerable. Attackers need only monitor the mempool or the router's ETH balance and call `refundETH()` in the next block.

## Recommendation
Either (a) record the original `msg.sender` in transient storage at `multicall` entry and restrict `refundETH` to that stored address, or (b) automatically refund `address(this).balance − consumed` to `msg.sender` at the end of each top-level swap function, eliminating the need for a separate `refundETH` call.

## Proof of Concept
```solidity
function test_attacker_steals_stranded_eth() public {
    address victim   = makeAddr("victim");
    address attacker = makeAddr("attacker");
    vm.deal(victim, 2 ether);

    // Victim swaps 1000 wei of WETH but sends 2 ether, omitting refundETH
    vm.prank(victim);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // ~2 ether - 1000 wei is now stranded on the router

    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();  // no access control — succeeds

    assertGt(attacker.balance, attackerBefore, "attacker stole victim ETH");
    assertEq(address(router).balance, 0);
}
```
