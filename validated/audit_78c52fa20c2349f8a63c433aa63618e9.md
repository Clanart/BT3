### Title
`get_state_response_part` Uses New-Epoch Shard Layout to Index Old-Epoch Block Chunks, Producing Wrong State Root During Resharding — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::get_state_response_part` resolves `shard_index` using the **new epoch's** shard layout, then uses that index to read `prev_state_root` from `prev_block.chunks()`, which belongs to the **old epoch**. During resharding the two layouts differ, so the index selects the wrong chunk and the wrong state root. The sibling function `compute_state_response_header` performs the identical lookup but correctly uses the **old epoch's** shard layout — an exact inversion of the coordinate system, directly analogous to the external report's DAI/ETH vs ETH/DAI mismatch.

### Finding Description

In `get_state_response_part`:

```rust
// epoch_id = epoch of sync_hash block  (NEW epoch)
let epoch_id = block.header().epoch_id();
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;   // NEW layout
...
let prev_block = self.chain_store.get_block(header.prev_hash())?;    // OLD epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;           // index from NEW layout
let state_root = prev_block
    .chunks()
    .get(shard_index)          // ← indexes OLD epoch's chunks with NEW epoch's index
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

`compute_state_response_header` performs the same lookup but correctly derives `prev_shard_layout` from `sync_prev_block.header().epoch_id()` (the **old** epoch) and uses that layout to compute `prev_shard_index`:

```rust
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let chunk_header = chunks.get(prev_shard_index)...;   // correct
``` [2](#0-1) 

The `state_root` extracted in `get_state_response_part` is then passed to both `get_state_root_node` (to derive `num_parts`) and `obtain_state_part` (to generate the actual trie data): [3](#0-2) 

`obtain_state_part_impl` uses `prev_hash` only to resolve `epoch_id` via `get_epoch_id_from_prev_block`; the trie data it reads is keyed entirely by `state_root`: [4](#0-3) 

So a wrong `state_root` propagates silently into the generated `StatePart`.

The `RuntimeAdapter::obtain_state_part` contract documents the expected invariant:

> `prev_hash` is a block whose **post** state root is `state_root`. [5](#0-4) 

When the shard layouts differ (resharding), the index from the new layout selects a **different** chunk in `prev_block`, violating this invariant.

### Impact Explanation

Two failure modes arise during resharding:

1. **Silent wrong state root** — if the new-epoch `shard_index` is in-bounds for the old-epoch chunk array but points to a different shard's chunk, `obtain_state_part` generates a trie part rooted at the wrong hash. The receiving node validates it against the correct root (from `set_state_part` / `compute_state_response_header`), the check fails, and state sync aborts.

2. **Out-of-bounds error** — if the new-epoch `shard_index` exceeds the old-epoch chunk count (e.g., a newly split shard has a higher index), `prev_block.chunks().get(shard_index)` returns `None` and the function returns `Err(InvalidShardId)`, immediately breaking state sync for that shard.

Either outcome prevents validators from completing catchup for newly assigned shards, blocking chunk production and reward accrual for the affected epoch.

### Likelihood Explanation

The bug is dormant when no resharding occurs (both layouts are identical, so `shard_index` is the same). It activates at every epoch boundary where dynamic resharding changes the shard layout — a scenario explicitly supported and tested in the codebase (`DynamicResharding`, `shuffle_shard_assignment_for_chunk_producers`). Any node serving state parts at such a boundary triggers the path. [6](#0-5) 

### Recommendation

Mirror `compute_state_response_header`: derive `prev_shard_layout` from `prev_block.header().epoch_id()` and use it — not `shard_layout` — to compute `shard_index` for indexing into `prev_block.chunks()`.

```rust
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

### Proof of Concept

1. Network reshards: old epoch has shards `[S0, S1, S2]`; new epoch splits `S2` into `[S2a, S2b]`, yielding `[S0, S1, S2a, S2b]`.
2. `sync_hash` is the first block of the new epoch; `prev_block` is the last block of the old epoch (3 chunks, indices 0–2).
3. New-epoch `shard_layout.get_shard_index(S2a)` returns index `2` (position of `S2a` in the new layout).
4. `prev_block.chunks().get(2)` returns the chunk for old shard `S2`, not `S2a`.
5. `state_root` = `S2.prev_state_root()` — wrong root.
6. `obtain_state_part` generates a trie part rooted at `S2`'s root.
7. The receiving node calls `set_state_part`, which validates against the root from `compute_state_response_header` (correctly derived from `S2a`'s root); validation fails.
8. State sync for `S2a` fails; the validator cannot produce chunks for `S2a` in the new epoch.

    pub fn get_state_response_part(
        &mut self,
        shard_id: ShardId,
        part_id: u64,
        sync_hash: CryptoHash,
    ) -> Result<StatePart, Error> {
        let _span = tracing::debug_span!(
            target: "sync",
            "get_state_response_part",
            %shard_id,
            part_id,
            ?sync_hash)
        .entered();
        let block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let header = block.header();
        let epoch_id = block.header().epoch_id();
        // Check cache
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id)).unwrap();
        if let Some(bytes) = self.chain_store.store_ref().get(DBCol::StateParts, &key) {
            metrics::STATE_PART_CACHE_HIT.inc();
            let state_part = StatePart::from_bytes(bytes.to_vec())?;
            return Ok(state_part);
        }
        metrics::STATE_PART_CACHE_MISS.inc();

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_ids = self.epoch_manager.shard_ids(epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let prev_block = self.chain_store.get_block(header.prev_hash())?;
        let shard_index = shard_layout.get_shard_index(shard_id)?;
        let state_root = prev_block
            .chunks()
            .get(shard_index)
            .ok_or(Error::InvalidShardId(shard_id))?
            .prev_state_root();
        let prev_hash = *prev_block.hash();
        let prev_prev_hash = *prev_block.header().prev_hash();
        let state_root_node = self
            .runtime_adapter
            .get_state_root_node(shard_id, &prev_hash, &state_root)
            .log_storage_error("get_state_root_node fail")?;
        let num_parts = get_num_state_parts(state_root_node.memory_usage);
        if part_id >= num_parts {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let current_time = Instant::now();
        let state_part = self
            .runtime_adapter
            .obtain_state_part(
                shard_id,
                &prev_prev_hash,
                &state_root,
                PartId::new(part_id, num_parts),
            )
            .log_storage_error("obtain_state_part fail")?;

        let elapsed_ms = (self.clock.now().signed_duration_since(current_time))
            .whole_milliseconds()
            .max(0) as u128;
        self.requested_state_parts
            .save_state_part_elapsed(&sync_hash, &shard_id, &part_id, elapsed_ms);

        // Cache the part data, but only if the corresponding header is also cached.
        // At epoch boundaries, clear_all_downloaded_parts() deletes all cached headers
        // and parts. Since serving runs on a separate actor, a part request can arrive
        // after the clear and re-create a StatePartKey without its StateHeaderKey,
        // which the storage validator treats as an inconsistency.
        let header_key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
        if self.chain_store.store_ref().exists(DBCol::StateHeaders, &header_key) {
            let mut store_update = self.chain_store.store().store_update();
            let bytes = state_part.to_bytes();
            store_update.set(DBCol::StateParts, &key, &bytes);
            store_update.commit();
        }

        Ok(state_part)
    }
