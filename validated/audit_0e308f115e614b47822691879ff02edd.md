### Title
Proposal Validator Uses Proposer-Supplied Past Timestamp for L1 Gas Price Lookup, Allowing Stale Prices to Be Committed to Block Hash — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` enforces only a **lower bound of `last_block_timestamp`** on `init_proposed.timestamp`, with no floor at `now`. It then passes that same proposer-controlled timestamp directly into `get_l1_prices_in_fri_and_wei`, which queries the L1 gas price ring buffer anchored at that timestamp. A malicious proposer can therefore supply a timestamp that is arbitrarily far in the past (bounded only by the ring buffer's history window), causing every validator to independently recompute the "expected" gas price from the same stale window and accept the proposal. The resulting L1 gas prices are committed into the block header and block hash, permanently recording incorrect fee parameters for every transaction in the block.

---

### Finding Description

**Root cause — missing `max(now, timestamp)` floor before gas-price lookup**

In `is_block_info_valid` the only lower-bound check on the proposed timestamp is:

```rust
// validate_proposal.rs:255
if init_proposed.timestamp < last_block_timestamp { … reject … }
```

There is no check of the form `init_proposed.timestamp >= now`. The upper-bound check (`> now + window_seconds`) is present, but the symmetric lower-bound check against `now` is absent.

Immediately after those checks the validator calls:

```rust
// validate_proposal.rs:286-292
let (l1_gas_prices_fri, _) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← proposer-controlled, can be in the past
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
).await;
```

`get_l1_prices_in_fri_and_wei` calls `get_price_info(BlockTimestamp(timestamp))`, which selects the ring-buffer window ending at `timestamp - lag_margin_seconds`:

```rust
// l1_gas_price_provider.rs:127-128
let index_last_timestamp_rev = samples.iter().rev().position(|data| {
    data.timestamp <= timestamp.saturating_sub(&self.config.lag_margin_seconds.as_secs())
});
```

The only staleness guard in `get_price_info` checks whether the timestamp is **too far in the future** relative to the newest sample:

```rust
// l1_gas_price_provider.rs:119
if timestamp.0 > (*last_timestamp + self.config.max_time_gap_seconds) { … }
```

There is no symmetric guard for timestamps that are **too far in the past**. Consequently, any timestamp that falls within the ring buffer's retained history is silently accepted and used to anchor the price window.

**Structural analog to the AeraVault bug**

| AeraVault `updateWeightsGradually` | Sequencer `is_block_info_valid` |
|---|---|
| Uses raw `startTime` for weight-change duration | Uses raw `init_proposed.timestamp` for gas-price window |
| Balancer internally applies `max(currentTime, startTime)` | Correct behavior requires `max(now, init_proposed.timestamp)` |
| Discrepancy lets `targetWeights[i]` exceed the allowed bound | Discrepancy lets proposer select any historical price window |
| Fix: clamp `startTime = max(block.timestamp, startTime)` before use | Fix: clamp `effective_ts = max(now, init_proposed.timestamp)` before lookup |

**Attack path**

1. Malicious proposer sets `init.timestamp = T_past` where `T_past >= last_block_timestamp` and `T_past < now`.
2. Proposer queries its own `get_price_info(T_past)` to obtain the gas prices that correspond to that historical window.
3. Proposer broadcasts `ProposalInit` with `timestamp = T_past` and the matching stale prices.
4. Each validator runs `is_block_info_valid`:
   - `T_past >= last_block_timestamp` → passes.
   - `T_past <= now + 1s` → passes (past timestamps always satisfy this).
   - `get_l1_prices_in_fri_and_wei(T_past, …)` → returns the **same stale prices** the proposer used.
   - `within_margin(proposed, expected, margin)` → passes.
5. Proposal is accepted. `decision_reached` commits the block with stale L1 gas prices.

The same missing floor exists in `try_sync`:

```rust
// sequencer_consensus_context.rs:844-846
if !(block_number == height
    && timestamp.0 >= last_block_timestamp          // no >= now check
    && timestamp.0 <= now + block_timestamp_window_seconds)
```

---

### Impact Explanation

The L1 gas prices (`l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`) are embedded in the committed block header and enter the block hash via `gas_prices_hash` in `calculate_block_hash`. Every transaction in the block has its fee computed against these prices. A proposer who selects a past window where L1 base fees or blob fees were significantly lower can cause all transactions in the block to be charged less than the actual current L1 cost, creating an economic shortfall for the protocol. Conversely, selecting a historically high window extracts excess fees from users. The committed block hash is permanently wrong relative to what honest nodes would compute using current prices.

Matches: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

Any validator node that wins the proposer slot can execute this attack without any special privilege beyond normal participation in consensus. The attack requires only modifying the timestamp field in `ProposalInit` before broadcasting; no cryptographic material or privileged access is needed. The window of exploitable past timestamps is bounded by the ring buffer's `storage_limit` (number of L1 blocks retained), which in practice covers many hours of history.

---

### Recommendation

In `is_block_info_valid`, clamp the effective timestamp to `max(now, init_proposed.timestamp)` before passing it to the gas-price provider, mirroring the AeraVault fix pattern:

```rust
// validate_proposal.rs — proposed fix
let now: u64 = clock.unix_now();
let last_block_timestamp = …;

if init_proposed.timestamp < last_block_timestamp { … }
if init_proposed.timestamp > now + block_info_validation.block_timestamp_window_seconds { … }

// Clamp: do not allow a past timestamp to select a stale price window.
let effective_timestamp = init_proposed.timestamp.max(now);

let (l1_gas_prices_fri, _) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    effective_timestamp,   // ← use now-floored value
    …
).await;
```

Apply the same `max(now, timestamp)` floor in `try_sync` before accepting a sync block's gas prices.

Optionally, add a symmetric lower-bound guard in `get_price_info` that rejects timestamps older than `last_sample_timestamp - some_max_past_gap`, analogous to the existing `max_time_gap_seconds` guard for future timestamps.

### Proof of Concept

```
Setup:
  - Ring buffer contains L1 samples for the last 6 hours.
  - Current time: now = T.
  - L1 base fee at T-4h: 10 gwei.  L1 base fee at T: 100 gwei.
  - Previous block timestamp: T - 10s.

Attack:
  1. Proposer sets init.timestamp = T - 4h (satisfies >= T-10s, <= T+1s).
  2. Proposer calls get_price_info(T - 4h) → returns 10 gwei.
  3. Proposer broadcasts ProposalInit{timestamp=T-4h, l1_gas_price_fri=<10gwei converted>}.
  4. Validators call is_block_info_valid:
       - T-4h >= T-10s  ✓
       - T-4h <= T+1s   ✓
       - get_l1_prices_in_fri_and_wei(T-4h) → 10 gwei  ✓
       - within_margin(10gwei, 10gwei, margin) ✓
  5. Block committed with l1_gas_price = 10 gwei instead of 100 gwei.
  6. All transactions in the block pay 10x less for L1 gas than the actual cost.
  7. Block hash includes the 10-gwei price; the committed state is permanently wrong.

async fn get_proposal_timestamp(
    override_timestamp: bool,
    batcher: &dyn BatcherClient,
    clock: &dyn Clock,
) -> u64 {
    if override_timestamp {
        match batcher.get_timestamp().await {
            Ok(timestamp) => return timestamp,
            Err(err) => {
                warn!("Failed to get timestamp from batcher, falling back to clock time: {err:?}");
            }
        }
    }
    clock.unix_now()
}
