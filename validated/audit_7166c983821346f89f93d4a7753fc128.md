### Title
Unvalidated Class Hash in `execute_replace_class` Allows Permanent Contract State Corruption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the caller-supplied class hash corresponds to a previously declared contract class. The OS unconditionally writes the unvalidated hash into the contract's state entry. Any contract can therefore permanently replace its class with an undeclared hash, making itself permanently unexecutable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `request.class_hash` directly from the syscall buffer and immediately commits it to `contract_state_changes` without checking whether that hash exists in `contract_class_changes` (the dictionary that maps declared class hashes to their compiled-class hashes):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The developer-inserted TODO at line 898 explicitly acknowledges the missing check. The `contract_class_changes` dictionary (populated only by `declare` transactions and the bootstrap path in `execute_declare_transaction`) is never consulted here. [2](#0-1) 

This is the direct analog of the reported loan-address bug: just as `initiateRollover` passed an unvalidated `priorLoan` address into `PoolLib::calculateRollover`, `execute_replace_class` passes an unvalidated `class_hash` into the state-update dictionary without confirming it is a legitimately declared class.

---

### Impact Explanation

Once `dict_update` commits the invalid class hash to `contract_state_changes`, the contract's on-chain state permanently records a class hash for which no compiled class exists. Every subsequent call to that contract will fail at class-lookup time inside the OS execution pipeline. There is no recovery path: the OS has no mechanism to "un-replace" a class hash after the state root is committed. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible — matching the **Critical: Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The `replace_class` syscall is available to every Cairo contract without restriction. A realistic trigger path requires a contract that:

1. Holds user funds (e.g., a vault, AMM, or bridge contract), **and**
2. Exposes an upgrade entry point that forwards a caller-supplied class hash to `replace_class` without its own pre-validation.

Such patterns exist in upgradeable contract designs. An unprivileged transaction sender who can call the upgrade entry point with an arbitrary felt value can supply a hash that was never declared, causing the OS to silently accept the corrupt state. Because the OS is the last enforcement layer and performs no check, even a contract that intends to validate the hash before calling `replace_class` cannot rely on the OS as a safety net if its own validation is bypassed (e.g., via a reentrancy or logic bug).

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, verify that it exists in `contract_class_changes`:

```cairo
// Verify the new class hash is a declared class.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already present for `declare` transactions, where `assert_not_zero(compiled_class_hash)` is enforced before any state update is written. [3](#0-2) 

---

### Proof of Concept

1. Attacker deploys contract **V** (e.g., a minimal vault) that accepts user deposits and exposes an `upgrade(new_class_hash: felt)` entry point that calls `replace_class(new_class_hash)` directly.
2. Legitimate users deposit funds into **V**.
3. Attacker submits an invoke transaction calling `upgrade(0xdeadbeef…)` where `0xdeadbeef…` is a felt value that was never passed through a `declare` transaction and therefore has no entry in `contract_class_changes`.
4. The OS executes `execute_replace_class`: gas is deducted, `dict_update` writes `class_hash = 0xdeadbeef…` into **V**'s `StateEntry`, and the function returns successfully with no error.
5. The new state root is committed to L1 with **V**'s class hash set to the undeclared value.
6. Any subsequent transaction targeting **V** fails at class resolution; the contract is permanently unexecutable.
7. All deposited funds are permanently frozen with no recovery path. [4](#0-3)
