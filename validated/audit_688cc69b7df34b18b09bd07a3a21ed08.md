Audit Report

## Title
Unguarded `refundETH` Drains Any Stranded ETH to Arbitrary Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` sends `address(this).balance` to `msg.sender` with no per-user accounting and no access control. The `pay()` function deliberately leaves surplus native ETH on the router when `nativeBalance > value`. Any ETH stranded between transactions — from any user's overpayment — is claimable in full by any external caller of `refundETH()`.

## Finding Description
`pay()` in `PeripheryPayments.sol` (L73–78) handles native ETH input by wrapping exactly `value` wei via `WETH.deposit{value: value}()` and transferring it to the pool. When `address(this).balance > value`, the surplus remains as raw ETH on the router — this is the intended design for the multicall refund pattern. `refundETH()` (L58–63) is `external payable` with no modifier, no mapping, and no transient-storage check; it unconditionally transfers `address(this).balance` to `msg.sender`. `multicall` (L39–44) is `payable` and uses `delegatecall`, so `msg.sender` is preserved within a single transaction, but the cross-transaction vector is the real attack: User A calls `multicall{value: 2 ether}` encoding only `exactInputSingle` with `amountIn = 1000` and omits `refundETH`. After the swap, `2 ether - 1000 wei` sits on the router. Attacker B calls `refundETH()` in a separate transaction and receives the full balance. The `receive()` guard (L32–34) only blocks direct ETH sends from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on any payable entry point (`multicall`, `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`). No transient storage or any other mechanism tracks per-caller ETH deposits.

## Impact Explanation
Direct loss of user principal. A victim who sends ETH overpayment without appending `refundETH` to their multicall — or who calls `exactInputSingle` directly with `msg.value > amountIn` — permanently loses the surplus to the first caller of `refundETH()`. This is a concrete, quantifiable theft of user funds meeting the Critical/High direct-loss-of-principal threshold.

## Likelihood Explanation
Moderate-to-high. The safe pattern requires users to manually compose a two-call multicall with `refundETH` as the second step, as shown in `test_multicall_ethInput_exactInputSingle_refundsUnusedEth`. Integrators, aggregators, or users calling `exactInputSingle` directly with excess ETH will routinely omit this step. A MEV bot monitoring the router's ETH balance or the mempool can reliably back-run any transaction that leaves ETH on the router, making exploitation repeatable and low-effort.

## Recommendation
Track per-caller ETH deposits in transient storage at the start of each payable entry point and enforce in `refundETH()` that only the depositor's own credited balance is returned. Alternatively, auto-refund surplus ETH at the end of each swap function (after `_clearExpectedCallbackPool`) rather than relying on a separate user-initiated call.

## Proof of Concept
```solidity
// Step 1: Victim sends ETH overpayment without refundETH
vm.prank(victim);
bytes[] memory calls = new bytes[](1);

router.multicall{value: 2 ether}(calls);
// Router now holds 2 ether - 1000 wei as raw ETH

// Step 2: Attacker steals it in a separate transaction
uint256 before = attacker.balance;
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance - before, 2 ether - 1000); // attacker receives victim's surplus
```
