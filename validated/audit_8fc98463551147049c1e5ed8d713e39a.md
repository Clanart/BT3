### Title
Unchecked `.detach()` on `send_tokens` in Fast-Transfer Finalization Causes Permanent Loss of Relayer Funds - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain` and `utxo_fin_transfer_fast`, when a fast-transfer relayer is being reimbursed during canonical finalization, `send_tokens(...)` is called with `.detach()` ŌĆö meaning the promise result is never observed. If the token transfer fails, the fast transfer is already marked as finalised and the origin-chain locked-token accounting is already updated, making the relayer's pre-paid funds permanently irrecoverable with no retry path.

---

### Finding Description

In `process_fin_transfer_to_other_chain` (called from `fin_transfer_callback` when the recipient is on a non-NEAR chain), the code checks whether a fast transfer was previously executed by a relayer. If so, it attempts to reimburse the relayer:

```rust
// near/omni-bridge/src/lib.rs:2028-2040
if let Some(relayer) = recipient {
    self.send_tokens(
        token,
        relayer,
        U128(transfer_message.amount_without_fee()...),
        "",
    )
    .detach();                                    // ŌåÉ result never checked
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());
}
```

Before this point, the function has already:
1. Called `add_fin_transfer` ŌĆö consuming the proof/transfer ID permanently (line 1985)
2. Called `unlock_tokens_if_needed` ŌĆö decrementing `locked_tokens` for the origin chain (lines 1997ŌĆō2001)
3. Called `lock_tokens_if_needed` for the fee on the destination chain (lines 2002ŌĆō2005)

After `.detach()`, `mark_fast_transfer_as_finalised` is called unconditionally. If `send_tokens` fails (e.g., `ft_transfer` panics because the relayer has no storage registered for the token, or the token contract reverts), the following state is left:

- The transfer ID is consumed ŌĆö `fin_transfer` cannot be called again
- `locked_tokens[origin_chain]` is decremented ŌĆö accounting shows tokens as released
- The fast transfer is marked finalised ŌĆö no retry is possible
- The relayer never receives their tokens ŌĆö their pre-paid funds are permanently lost
- The bridge holds the tokens with no accounting entry pointing to them

The same pattern exists in `utxo_fin_transfer_fast`:

```rust
// near/omni-bridge/src/lib.rs:2542-2548
self.send_tokens(
    fast_transfer.token_id.clone(),
    fast_transfer_status.relayer,
    amount,
    "",
)
.detach();
```

Here too, `mark_fast_transfer_as_finalised` (or `remove_fast_transfer`) is called before the promise resolves, with no failure callback.

By contrast, the analogous path for NEAR-recipient transfers correctly uses a callback:

```rust
// near/omni-bridge/src/lib.rs:1967-1977
self.send_tokens(...)
.then(
    Self::ext(env::current_account_id())
        .fin_transfer_send_tokens_callback(...)
)
```

And `fin_transfer_send_tokens_callback` properly handles failure by reverting lock actions and removing the fin transfer record.

---

### Impact Explanation

**Critical ŌĆö Permanent freezing / irrecoverable loss of relayer funds.**

A relayer who executed a fast transfer pre-paid the user from their own token balance. When the canonical `fin_transfer` arrives to reimburse them, if `send_tokens` fails silently:

- The relayer's pre-paid tokens are permanently lost ŌĆö no retry, no refund path
- The bridge's `locked_tokens` accounting is corrupted: origin-chain locked amount is decremented but the tokens remain in the bridge contract unaccounted for
- The transfer ID is consumed, so the proof cannot be re-submitted

This breaks bridge collateralization (accounting corruption) and causes irrecoverable loss of the relayer's funds.

---

### Likelihood Explanation

**Medium.** `send_tokens` for native (non-deployed) tokens calls `ft_transfer`, which panics if the recipient (relayer) has no storage registered for that specific token. A relayer may have storage for common tokens but not for every token they relay. For deployed tokens, `mint` is called, which can fail if the token contract is paused or has a bug. Both are realistic operational conditions, not theoretical edge cases.

---

### Recommendation

Replace `.detach()` with a proper failure-handling callback in both `process_fin_transfer_to_other_chain` and `utxo_fin_transfer_fast`. The callback should:
1. Check the promise result
2. On failure: revert `mark_fast_transfer_as_finalised`, restore `locked_tokens`, and remove the fin transfer record (or allow re-submission)
3. On success: proceed as today

This mirrors the existing pattern in `process_fin_transfer_to_near` ŌåÆ `fin_transfer_send_tokens_callback`.

---

### Proof of Concept

1. Relayer R executes a fast transfer for user U: R pre-pays U `1000 TOKEN` on NEAR, recording a `FastTransfer` entry.
2. The canonical `fin_transfer` proof arrives from EVM. `process_fin_transfer_to_other_chain` is called.
3. `add_fin_transfer` consumes the transfer ID. `unlock_tokens_if_needed` decrements `locked_tokens[Eth]` by 1000.
4. The fast transfer status is found; `send_tokens(token, R, 1000, "")` is called with `.detach()`.
5. `ft_transfer` to R fails (R has no storage for TOKEN on this token contract).
6. `mark_fast_transfer_as_finalised` is called ŌĆö the fast transfer record is permanently closed.
7. Result: R's 1000 TOKEN pre-payment is unrecoverable. The bridge holds 1000 TOKEN with no `locked_tokens` entry. The transfer ID is consumed. No retry is possible. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
