Audit Report

## Title
`addLiquidity` and `removeLiquidity` bypass the emergency pause — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool.swap` is protected by `whenNotPaused`, which reverts when `pauseLevel != 0`. However, `addLiquidity` and `removeLiquidity` carry only `nonReentrant` and no pause guard. When the pool is paused to halt swaps during an emergency, liquidity deposits and withdrawals continue to execute, undermining the invariant that a paused pool halts all state-changing token flows.

## Finding Description

`_checkNotPaused` reverts when `pauseLevel != 0`: [1](#0-0) 

The `whenNotPaused` modifier wraps that check: [2](#0-1) 

`swap` correctly applies it: [3](#0-2) 

`addLiquidity` does not — only `nonReentrant` is present: [4](#0-3) 

`removeLiquidity` does not either: [5](#0-4) 

The periphery `MetricOmmPoolLiquidityAdder` routes directly to `pool.addLiquidity` with no pause check of its own: [6](#0-5) 

## Impact Explanation

Two concrete loss paths exist within the allowed impact gate:

**Direct LP principal loss via forced bad-price exposure:** An oracle compromise triggers `pauseLevel = 1`. Because `addLiquidity` is unguarded, an LP (or attacker acting as LP) deposits tokens into the paused pool. On unpause, swaps resume against the manipulated oracle price and drain the freshly deposited principal. This is a direct loss of user principal above Sherlock thresholds.

**Inequitable LP exit during an in-progress exploit:** An informed LP calls `removeLiquidity` while the pool is paused (e.g., due to a bin-cursor skew from an ongoing exploit). They exit at full value while remaining LPs are locked out of swaps but not of withdrawals, leaving the pool with skewed depth and disproportionate losses for remaining LPs. This constitutes broken core pool functionality causing loss of LP assets.

Severity: **High** — direct loss of LP principal, reachable by any unprivileged LP or address with no special role.

## Likelihood Explanation

- `addLiquidity` has no `msg.sender` restriction; any address can call it.
- `removeLiquidity` is callable by any position owner (`msg.sender == owner`), which is any LP.
- `MetricOmmPoolLiquidityAdder` is the standard user-facing entry point and passes calls through without any pause check.
- The pause mechanism is explicitly designed for emergencies (oracle failure, price anomaly, exploit in progress) — these are real operational events, not theoretical.
- No additional privilege is required; the gap is reachable by any unprivileged user.

## Recommendation

Add `whenNotPaused` to both functions in `metric-core/contracts/MetricOmmPool.sol`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) { ... }

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (...) { ... }
```

If allowing LPs to exit during a pause is an intentional design choice for `removeLiquidity`, that decision must be explicitly documented and the asymmetry with `addLiquidity` must still be resolved.

## Proof of Concept

```
1. Deploy pool with a mutable price provider.
2. Call pool.setPause(1) via factory (admin pause).
3. Attempt pool.swap(...) → reverts with PoolPaused. ✓
4. Call MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, ...) →
   succeeds, tokens transferred into paused pool. ✗
5. Call pool.removeLiquidity(owner, salt, deltas, "") →
   succeeds, tokens transferred out of paused pool. ✗
6. Call pool.setPause(0) (unpause).
7. Attacker swaps against the pool at the oracle price that caused the pause →
   LP funds deposited in step 4 are drained.
```

Steps 4 and 5 are confirmed by the production code: neither `addLiquidity` nor `removeLiquidity` carries `whenNotPaused`, while `swap` does. The gap is directly observable in the modifier lists at the function signatures cited above.
