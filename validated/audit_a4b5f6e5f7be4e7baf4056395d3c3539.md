### Title
`finalize_cancellation` Deletes Committed L1 Handler Tx Record Without Guard, Enabling Re-Execution on L2 — (File: `crates/apollo_l1_provider/src/transaction_manager.rs`)

---

### Summary

`finalize_cancellation` in `TransactionManager` unconditionally deletes a transaction record when an L1 `TransactionCanceled` event arrives, even when the transaction is already in the `Committed` state (already included in an L2 block). The sibling function `mark_cancellation_request` (in `TransactionRecord`) has an explicit guard that skips state mutation for committed transactions, but `finalize_cancellation` has no equivalent guard. After the record is deleted, a scraper restart can re-add the same L1 handler transaction to the proposable index, allowing it to be re-proposed and re-executed in a future L2 block.

---

### Finding Description

**The guarded path — `mark_cancellation_request` in `transaction_record.rs`:** [1](#0-0) 

```rust
if self.is_committed() {
    warn!(
        "L1 handler transaction {tx_hash} was not marked for cancellation started on L2 \
         as it is already committed."
    )
} else {
    self.state = TransactionState::CancellationStartedOnL2;
}
```

When a tx is already `Committed`, the state is **not** changed to `CancellationStartedOnL2`. The record survives intact.

**The unguarded path — `finalize_cancellation` in `transaction_manager.rs`:** [2](#0-1) 

```rust
pub fn finalize_cancellation(&mut self, tx_hash: TransactionHash) {
    let Some(record) = self.records.get(&tx_hash) else { ... return; };

    if record.state != TransactionState::CancellationStartedOnL2 {
        warn!("...not in the cancellation started on L2 state, but in the {:?} state.", record.state);
    }
    // Unconditionally deletes — no guard for Committed state.
    self.with_record(tx_hash, |r| r.mark_cancellation_finalized_on_l1());
    self.records.remove(&tx_hash);
}
```

The warning is logged but the record is **deleted regardless** of whether the transaction is `Committed`.

**Re-entry via `add_tx`:**

After deletion, if the scraper restarts and re-scrapes the same L1 handler transaction (possible when `start_height` is below the block where the tx was committed), `add_tx` calls `create_record_if_not_exist`, which succeeds because the record no longer exists: [3](#0-2) 

A new record is created in the default `Pending` state. `maintain_indices` then adds it to `proposable_index` because `is_proposable()` returns `true` for `Pending` state. [4](#0-3) 

**`commit_txs` cannot detect the double-commit:** [5](#0-4) 

`commit_txs` calls `create_record_if_not_exist` before `mark_committed`. Since the old record was deleted and a new one was created with `committed = false`, the assertion in `mark_committed` passes: [6](#0-5) 

The L1 handler transaction is thus committed twice to L2.

**Validation does not block re-inclusion:**

`validate_tx` returns `ValidationStatus::Validated` for a `Pending` record that has not yet been staged in the current epoch: [7](#0-6) 

A validator node would accept the re-proposed transaction, allowing it to be finalized in a new block.

---

### Impact Explanation

A committed L1 handler transaction can be re-executed on L2, producing wrong state, wrong receipts, and wrong L1 message processing. This matches:

> **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

The L1 handler tx is executed twice on L2. Depending on the tx's logic (e.g., token minting, fund bridging), this can drain the L1 bridge contract or corrupt L2 state.

---

### Likelihood Explanation

The scenario requires:
1. An L1 handler tx to be committed to L2 block N before the cancellation timelock expires.
2. The L1 cancellation finalization event to arrive after the tx was committed (realistic: the timelock is configurable and can be days long).
3. The scraper to restart and re-scrape from a `start_height` below block N (realistic: `start_height` is the last *proved* L2 block height; if block N is not yet proved, `start_height < N`).

All three conditions are realistic in production. The scraper can restart due to crashes or planned maintenance. The cancellation timelock is long enough for a tx to be committed before it expires.

---

### Recommendation

Add a guard in `finalize_cancellation` to skip deletion when the transaction is already `Committed`:

```rust
pub fn finalize_cancellation(&mut self, tx_hash: TransactionHash) {
    let Some(record) = self.records.get(&tx_hash) else {
        info!("Attempted to finalize cancellation for non-existent transaction: {tx_hash}.");
        return;
    };

    // MISSING CHECK (analog to the guard in mark_cancellation_request):
    if record.state == TransactionState::Committed {
        warn!(
            "Attempted to finalize cancellation for already-committed transaction {tx_hash}. \
             Ignoring to prevent re-execution."
        );
        return;
    }

    if record.state != TransactionState::CancellationStartedOnL2 {
        warn!("...not in the cancellation started on L2 state, but in the {:?} state.", record.state);
    }
    self.with_record(tx_hash, |r| r.mark_cancellation_finalized_on_l1());
    self.records.remove(&tx_hash);
}
```

    pub fn add_tx(
        &mut self,
        tx: L1HandlerTransaction,
        block_timestamp: BlockTimestamp,
        scrape_timestamp: UnixTimestamp,
    ) {
        let tx_hash = tx.tx_hash;
        // If exists, return false and do nothing. If not, create the record as a HashOnly payload.
        let is_new_record = self.create_record_if_not_exist(tx_hash);
        // Replace a HashOnly payload with a Full payload. Do not update a Full payload.
        // A hash only payload can come from catching up from state sync, and then updated by
        // add_events from the scraper. However, if we get the same full tx twice (from the scraper)
        // it could indicate a double-scrape, and may cause the tx to be re-added to the proposable
        // index.
        self.with_record(tx_hash, move |record| match &record.tx {
            TransactionPayload::HashOnly(_) => {
                if !is_new_record {
                    info!(
                        "Transaction {tx_hash} already exists as a HashOnly payload. It was \
                         probably gotten via state sync component, and is now updated with a Full \
                         payload."
                    );
                }
                record.tx.set(tx, block_timestamp, scrape_timestamp);
            }
            TransactionPayload::Full { tx: _, created_at_block_timestamp: _, scrape_timestamp } => {
                warn!(
                    "Transaction {tx_hash} already exists as a Full payload, scraped at \
                     {scrape_timestamp}. This could indicate a double scrape. Ignoring the new \
                     transaction."
                );
            }
        });
    }

    pub fn finalize_cancellation(&mut self, tx_hash: TransactionHash) {
        let Some(record) = self.records.get(&tx_hash) else {
            info!(
                "Attempted to finalize cancellation for non-existent transaction: {tx_hash}. This \
                 can happen if the transaction was too old to be scraped (e.g. it was created \
                 before we started scraping)."
            );
            return;
        };

        // Regardless of the state of the tx in the record, if we get the cancellation event from
        // the L1 contract, we delete this tx from the records and from the proposable index, even
        // if it was Pending and ready to be proposed (which is not supposed to happen, hence the
        // warning).
        if record.state != TransactionState::CancellationStartedOnL2 {
            warn!(
                "Attempted to finalize cancellation for transaction {tx_hash} that is not in the \
                 cancellation started on L2 state, but in the {:?} state.",
                record.state
            );
        }
        // This will also call maintain_indices to remove the tx from the proposable index.
        self.with_record(tx_hash, |r| r.mark_cancellation_finalized_on_l1());
        self.records.remove(&tx_hash);
