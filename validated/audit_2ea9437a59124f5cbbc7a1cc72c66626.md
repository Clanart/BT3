### Title
Unchecked Field Arithmetic in `compute_max_possible_fee` Allows Fee Bypass via Field-Wrap Overflow - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary

`compute_max_possible_fee` performs unchecked multiplications of `max_amount * max_price_per_unit` for each resource type in Cairo's felt (prime field) arithmetic. Because `max_price_per_unit` has no upper-bound check, an attacker-controlled transaction can cause the product to wrap around the Stark prime, producing an artificially small (or zero) `max_fee`. When `max_fee == 0`, `charge_fee` returns immediately without charging any fee, giving the attacker free execution.

### Finding Description

`compute_max_possible_fee` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

The only validation applied to `max_price_per_unit` is `assert_nn` in `pack_resource_bounds`:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn` only constrains `max_price_per_unit` to `[0, (P-1)/2]` where `P` is the Stark prime (~2^251). There is **no upper bound** on `max_price_per_unit`. With `max_amount` up to `2^64 - 1` and `max_price_per_unit` up to `~2^250`, the product `max_amount * max_price_per_unit` can be up to `~2^314`, wrapping around the ~252-bit prime approximately `2^62` times. The attacker has full control over what the product reduces to modulo the prime.

The caller `charge_fee` then does:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
...
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

If `max_fee` wraps to 0, the function returns without charging any fee. If it wraps to a small value (e.g., 1), the sequencer is forced to charge at most 1 wei regardless of actual execution cost.

### Impact Explanation

**Direct loss of funds (Critical).** A transaction sender can craft a V3 transaction with `max_price_per_unit` values chosen so that `compute_max_possible_fee` returns 0 (or near-zero) due to field arithmetic wrap-around. The OS then executes the transaction without collecting any fee, causing the sequencer/protocol to lose the fees that should have been collected. Since every V3 transaction goes through this path, the attacker can repeatedly execute transactions for free, draining sequencer revenue and potentially enabling unbounded spam.

### Likelihood Explanation

**High.** The attacker is any V3 transaction sender — fully unprivileged. The attack requires only choosing specific `max_price_per_unit` values (computable offline via modular arithmetic). No special access, leaked keys, or operator cooperation is needed. The Stark prime is public, so computing the required `max_price_per_unit = k * modular_inverse(max_amount, P) mod P` for a target `k = 0` (or any small value) is trivial.

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds` (or in `compute_max_possible_fee`) to ensure the product cannot wrap around the field prime. For example, bound `max_price_per_unit` to `2^128 - 1` (matching the 128-bit high limb of a Uint256 fee), consistent with how `max_amount` is bounded to 64 bits:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

With both `max_amount <= 2^64 - 1` and `max_price_per_unit <= 2^128 - 1`, the product is at most `~2^192`, well within the ~252-bit prime, eliminating wrap-around.

### Proof of Concept

1. Attacker constructs a V3 transaction with:
   - `l1_gas_bounds.max_amount = 1`
   - `l1_gas_bounds.max_price_per_unit = P - 1` (i.e., `-1` in the field, but `assert_nn` rejects this since `P-1 > (P-1)/2`)
   
   Alternatively, choose `max_amount = 2`, `max_price_per_unit = (P+1)/2` (which is `(P-1)/2 + 1`... also rejected). 

   More concretely: choose `max_amount = 3` and `max_price_per_unit = (P - 1) / 3` (if divisible). Then `3 * (P-1)/3 = P - 1 ≡ -1 mod P`. Set the other two terms to sum to `+1` (e.g., `l2_gas_bounds.max_amount = 1`, `l2_gas_bounds.max_price_per_unit + tip = 1`, `l1_data_gas_bounds.max_amount = 0`). Total: `-1 + 1 + 0 = 0 mod P`.

2. The OS calls `compute_max_possible_fee`, which returns `0`.

3. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without charging any fee.

4. The transaction executes for free. The attacker repeats this for every transaction they submit. [4](#0-3)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
