### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Contract Freezing — (File: `execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary class hash from a contract's syscall request without verifying that the hash corresponds to a declared contract class. This allows a contract to permanently set its own class hash to an undeclared value, making the contract permanently unexecutable and freezing any funds it holds.

### Finding Description
In `syscall_impls.cairo`, the `execute_replace_class` function (lines 878–916) reads the new class hash directly from the syscall request and writes it into the contract state without any validation against the set of declared classes:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developer-acknowledged TODO comment at line 898 explicitly confirms the missing check. [2](#0-1) 

The declared class hashes are tracked in `contract_class_changes` (the class changes dictionary, populated via `dict_update` in `execute_declare_transaction`). [3](#0-2) 

The OS never cross-references the `replace_class` target hash against this dictionary before committing the state update. Once committed, the state entry for the contract permanently holds an undeclared class hash. Any subsequent execution attempt for that contract will fail to resolve the class in the compiled class facts bundle, making the contract permanently unexecutable.

### Impact Explanation
**Critical — Permanent freezing of funds.**

If a contract holds assets (ERC-20 token balances, ETH, or any other on-chain value) and its class hash is replaced with an undeclared value, those assets are permanently frozen:
- The contract cannot be called (class lookup fails at execution time).
- The class hash cannot be corrected (doing so requires calling `replace_class` again, which requires the contract to be executable).
- There is no recovery path at the protocol level.

### Likelihood Explanation
The `replace_class` syscall is accessible to any executing contract. A realistic attack path:

1. An attacker deploys a contract that appears to be a legitimate shared service (e.g., a vault, a multisig wallet, or a token contract).
2. Users deposit funds into the contract.
3. The attacker triggers `replace_class` with an arbitrary undeclared class hash (either via a backdoor in the contract's logic, or by exploiting a reentrancy/access-control bug in a legitimate contract).
4. The OS commits the state update with the invalid class hash — no protocol-level check prevents this.
5. All funds held by the contract are permanently frozen.

The attack requires the attacker to control or exploit a contract, but the OS-level missing validation is the necessary enabling condition. Without this gap, the OS would reject the syscall and the contract's class hash would remain valid.

### Recommendation
In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that `class_hash` is present in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for the given `class_hash` and assert the returned compiled class hash is non-zero (`assert_not_zero`).

### Proof of Concept

1. Deploy contract `C` holding user funds. `C` exposes a function that calls `replace_class(0xdeadbeef...)` where `0xdeadbeef...` is never declared.
2. Submit an invoke transaction calling that function.
3. The OS executes `execute_replace_class`: `class_hash = 0xdeadbeef...` is written into `contract_state_changes` for address `C` with no validation.
4. `state_update` squashes and commits the state — `C`'s `StateEntry.class_hash` is now `0xdeadbeef...`. [4](#0-3) 
5. Any subsequent transaction targeting `C` fails at class resolution — the class `0xdeadbeef...` is absent from `compiled_class_facts_bundle`. [5](#0-4) 
6. All funds in `C` are permanently inaccessible.
