### Title
Unchecked Felt Arithmetic Overflow in `compute_max_possible_fee` Enables Fee-Free Transaction Execution — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in the StarkNet OS performs raw felt arithmetic multiplication on user-supplied `ResourceBounds` fields without any range checks. Because Cairo felt arithmetic is modular (mod the Stark prime P ≈ 2²⁵¹), an attacker can craft resource bound values whose sum wraps to exactly 0 mod P. When `max_fee == 0`, `charge_fee` returns immediately without executing the ERC-20 transfer, allowing the transaction to execute with no fee paid. The OS proof remains valid, and L1 accepts the state transition with the fee token balance unchanged.

---

### Finding Description

`compute_max_possible_fee` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

All six fields (`max_amount`, `max_price_per_unit` for each of the three resource types) are loaded from the transaction via hint `%{ LoadCommonTxFields %}` and are never range-checked in the OS. The multiplication and addition are pure felt operations, so the result is taken modulo P.

`charge_fee` then does:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
...
assert_nn_le(calldata.amount.low, max_fee);
``` [2](#0-1) 

If `max_fee` is 0, the function returns without executing the ERC-20 transfer. No Cairo constraint is violated; the proof is sound.

**Concrete overflow example:**

Set:
- `l1_gas_bounds.max_amount = 1`, `l1_gas_bounds.max_price_per_unit = P − 1`
- `l2_gas_bounds.max_amount = 1`, `l2_gas_bounds.max_price_per_unit = 1`
- `tip = 0`, `l1_data_gas_bounds.max_amount = 0`

OS felt computation:
```
1 * (P−1) + 1 * (1 + 0) + 0 = P − 1 + 1 = P ≡ 0 (mod P)
```

`max_fee = 0` → `charge_fee` returns immediately → no fee charged.

The sequencer's off-chain validator, using 64- or 128-bit integer arithmetic, computes `max_fee = P` (a large positive number) and accepts the transaction as having a valid (enormous) fee authorization. The discrepancy between the sequencer's arithmetic and the OS's felt arithmetic is the root cause.

The `ResourceBounds` values are committed to in the transaction hash (`compute_invoke_transaction_hash`), so the proof is consistent end-to-end — the OS simply never charges the fee. [3](#0-2) 

---

### Impact Explanation

**Direct loss of funds.** The fee recipient (sequencer / protocol treasury) does not receive the fee for the transaction. The fee token (STRK) balance of the sender is not decremented in the proven state transition. Because the OS proof is valid and accepted by the L1 verifier, the on-chain state reflects no fee payment. An attacker can repeat this for every transaction they submit, permanently draining fee revenue from the protocol with no on-chain recourse.

---

### Likelihood Explanation

The attack requires:
1. Crafting specific `ResourceBounds` values — the math is elementary (one example shown above).
2. The sequencer accepting the transaction — likely, because the sequencer's non-modular arithmetic computes a large positive `max_fee` and sees no reason to reject.
3. No privileged access, no key compromise, no network-level attack.

The attacker is an ordinary transaction sender. The only barrier is knowing the Stark prime P, which is public.

---

### Recommendation

Add explicit range checks on `max_amount` and `max_price_per_unit` for each resource type before computing the fee. Enforce that each field fits within a safe bound (e.g., `< 2^64`) using `assert_nn_le`, so that no product can overflow the felt field. Alternatively, compute `max_fee` using a checked multi-limb multiplication that asserts the result does not exceed `2^128 − 1` before using it in `assert_nn_le`.

```cairo
// Example guard (to be added before the return statement):
assert_nn_le(l1_gas_bounds.max_amount, MAX_RESOURCE_AMOUNT);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_RESOURCE_PRICE);
// ... repeat for l2 and l1_data bounds ...
```

---

### Proof of Concept

1. Construct a v3 Invoke transaction with:
   - `resource_bounds[L1_GAS] = {max_amount: 1, max_price_per_unit: P−1}`
   - `resource_bounds[L2_GAS] = {max_amount: 1, max_price_per_unit: 1}`
   - `resource_bounds[L1_DATA] = {max_amount: 0, max_price_per_unit: 0}`
   - `tip = 0`
2. Sign and submit to the sequencer. The sequencer computes `max_fee = P` (non-modular) and accepts.
3. The OS executes `compute_max_possible_fee`:
   - `1 * (P−1) + 1 * (1 + 0) + 0 * 0 = P ≡ 0 (mod P)`
4. `charge_fee` sees `max_fee == 0` and returns immediately.
5. The ERC-20 transfer is never executed; the fee token balance is unchanged.
6. The OS proof is generated and verified on L1 — no constraint is violated.
7. The attacker's transaction executed for free; the fee recipient received nothing. [4](#0-3)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
```
