### Title
Incorrect `||` Operator in `check_sync_progress` Causes Spurious `NoProgress` Events, Repeatedly Interrupting State Diff Sync and Serving Stale RPC State - (File: `crates/apollo_central_sync/src/lib.rs`)

---

### Summary

`check_sync_progress` uses `||` to combine three independent staleness checks. Because headers naturally stop advancing once the node reaches chain tip, the `header_marker == new_header_marker` sub-expression is permanently `true` in steady-state, making the entire `||` expression always `true`. This fires a `SyncEvent::NoProgress` every 300 seconds even while state-diff and compiled-class sync are actively making progress, causing the sync loop to restart and leaving the node's storage — and therefore every RPC read — perpetually behind.

---

### Finding Description

In `check_sync_progress` at line 1029 of `crates/apollo_central_sync/src/lib.rs`:

```rust
if header_marker==new_header_marker || state_marker==new_state_marker || is_casm_stuck {
    yield SyncEvent::NoProgress;
}
```

The intent of the function (confirmed by its own comment: *"This function is used to check if the sync is stuck"* and the developer TODO *"fix the bug and remove this function"*) is to detect when **all** sync pipelines have simultaneously stalled. The correct operator is `&&`.

With `||`, the condition fires whenever **any single** pipeline has not advanced during the 300-second polling window (`SLEEP_TIME_SYNC_PROGRESS`). In normal operation:

- **Headers** are downloaded first and reach chain tip before state diffs. Once there, `header_marker == new_header_marker` is permanently `true` because no new blocks arrive during the 300-second window.
- **State diffs** and **compiled classes** may still be actively catching up.

Because `true || <anything>` is always `true`, `SyncEvent::NoProgress` is yielded on every iteration of the check loop, even while state-diff and CASM sync are making forward progress.

`process_sync_event` maps `SyncEvent::NoProgress` directly to `Err(StateSyncError::NoProgress)`:

```rust
SyncEvent::NoProgress => Err(StateSyncError::NoProgress),
```

`StateSyncError::NoProgress` is classified as **recoverable** in `is_recoverable`:

```rust
StateSyncError::NoProgress
| StateSyncError::CentralSourceError(_)
| ... => true,
```

So `run` sleeps for `recoverable_error_sleep_duration` (3 seconds in production config) and calls `sync_while_ok` again. The new call recreates all streams from scratch. After 300 more seconds, `check_sync_progress` fires again, and the cycle repeats indefinitely.

---

### Impact Explanation

Every 303 seconds (300 s check interval + 3 s recovery sleep), the entire sync loop is torn down and restarted. During the 3-second gap, no state diffs or compiled classes are written to storage. More critically, the restart resets the in-flight streams, so any partially-downloaded batch of state diffs is discarded and must be re-fetched.

For a node that has fully synced headers but is still catching up on state diffs (the common case during initial sync or after a restart), the state-diff marker lags behind the header marker indefinitely. Every RPC call that reads state — `starknet_getStorageAt`, `starknet_getNonce`, `starknet_getClassHashAt`, `starknet_call`, fee estimation, simulation — goes through `verify_synced_up_to`, which checks the state marker. Queries for blocks that have headers but not yet state diffs return `BlockNotFound` or stale storage values, even though the block is fully available on-chain and the node's header storage is current.

This matches: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The condition is triggered automatically during normal, unprivileged operation. Any node that has finished downloading block headers but is still processing state diffs — which is the standard state during initial sync of a full-archive node — will hit this path. No attacker action is required; the bug is self-triggering. The developers have already acknowledged it with two TODO comments in the same function.

---

### Recommendation

Change `||` to `&&` on line 1029 so that `NoProgress` is only emitted when **all three** pipelines are simultaneously stuck:

```rust
// Before (buggy):
if header_marker==new_header_marker || state_marker==new_state_marker || is_casm_stuck {

// After (correct):
if header_marker==new_header_marker && state_marker==new_state_marker && is_casm_stuck {
```

This mirrors the fix recommended in the analogous Lido report and matches the function's stated purpose of detecting a fully-stuck sync, not a partially-caught-up one.

---

### Proof of Concept

1. Start a fresh node configured with `store_sierras_and_casms_block_threshold = 0` (the production default in `state_sync_config.json`).
2. Allow headers to sync to chain tip (header marker stops advancing).
3. State diffs are still behind (state marker is still advancing).
4. After 300 seconds, `check_sync_progress` wakes up. It reads:
   - `new_header_marker == header_marker` → `true` (headers at tip, no new blocks in 300 s)
   - `new_state_marker != state_marker` → `false` (state diffs progressing)
   - `is_casm_stuck` → `false` (CASM progressing)
5. Condition: `true || false || false` = `true` → `SyncEvent::NoProgress` is yielded.
6. `process_sync_event` returns `Err(StateSyncError::NoProgress)`.
7. `run` logs a warning, sleeps 3 seconds, restarts `sync_while_ok`.
8. All in-flight state-diff streams are discarded; re-fetching begins from scratch.
9. After another 300 seconds, step 4 repeats.
10. During each 3-second restart window, RPC calls for blocks whose state diffs have not yet been written return `BlockNotFound` or stale storage values, despite the block header being present in storage.
