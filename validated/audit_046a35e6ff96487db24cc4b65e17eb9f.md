Audit Report

## Title
`pay()` and `refundETH()` consume unattributed router ETH balance, enabling theft of stranded ETH — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's aggregate native ETH balance — with no per-caller attribution when settling WETH-input swaps. Any ETH left on the router from a prior payable call that omitted `refundETH()` is silently consumed by the next WETH-input swap. Additionally, `refundETH()` is public and sends the entire router ETH balance to `msg.sender` with no ownership check, allowing any caller to drain stranded ETH directly.

## Finding Description
`PeripheryPayments.pay()` (lines 73–84) reads `address(this).balance` as the ETH source for WETH wrapping:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // aggregate, no attribution
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value); // payer's safeTransferFrom never called
    } else if (nativeBalance > 0) {
        ...
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`refundETH()` (lines 58–63) sends the entire router balance to any caller:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
``` [2](#0-1) 

The `receive()` guard (lines 32–34) only blocks direct ETH pushes from non-WETH addresses, but `msg.value` in any `payable` function bypasses it entirely. [3](#0-2) 

All swap entry points — `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` — are `payable`, so ETH sent with any of them and not consumed persists on the router across transaction boundaries. [4](#0-3) 

`_justPayCallback` feeds `pay()` with `payer = _getPayer()` (the current swapper) and `token = _getTokenToPay()`. When `address(this).balance >= value`, the `safeTransferFrom` on the payer is never executed — the router's pooled ETH is wrapped and forwarded instead. [5](#0-4) 

## Impact Explanation
Direct loss of user principal. A user who sends ETH in a payable router call and omits `refundETH()` permanently loses the unspent ETH. An unprivileged attacker recovers it by either calling `refundETH()` directly (receives the full stranded balance) or calling `exactInputSingle` with `tokenIn = WETH` (the stranded ETH is wrapped and forwarded to the pool at the attacker's direction, with the attacker's own WETH allowance never touched). Both paths require zero privilege and cause direct, irreversible loss of the victim's principal. [6](#0-5) 

## Likelihood Explanation
The pattern of sending `msg.value` to a router and relying on `refundETH()` at the end of a multicall is standard DeFi UX. Users and integrators routinely omit the refund step, especially when building multicalls programmatically or when a revert in a later step causes a retry without the refund call. The router's `multicall` uses `delegatecall`, so any step that reverts mid-flight leaves ETH on the router. The exploit requires no special privilege — a single public call to `refundETH()` suffices, and the WETH-swap path requires only a normal swap with `tokenIn = WETH`. [4](#0-3) 

## Recommendation
Track the ETH contributed by the current top-level call in transient storage (store `msg.value` at `multicall`/swap entry and decrement as it is consumed by `pay()`). In `pay()`, replace `address(this).balance` with the transient per-call ETH budget. Alternatively, require callers to wrap ETH to WETH before calling the router and remove the native-ETH branch from `pay()` entirely, relying solely on `safeTransferFrom`. [7](#0-6) 

## Proof of Concept
```
1. Alice calls router.multicall{value: 1 ether}([
       exactInputSingle(tokenIn=WETH, amountIn=1000, ...)
       // refundETH() omitted
   ])
   → 1 ether - 1000 wei ≈ 1 ETH stranded on router.

2a. Bob calls router.refundETH()
    → _transferETH(Bob, ~1 ETH)
    → Alice loses ~1 ETH, Bob gains it. No privilege required.

2b. Alternatively, Bob calls router.exactInputSingle(tokenIn=WETH, amountIn=500_000, ...)
    → _justPayCallback fires: pay(WETH, Bob, pool, 500_000)
    → address(this).balance ≈ 1 ETH >= 500_000
    → router wraps 500_000 wei of Alice's ETH and sends WETH to pool
    → Bob's safeTransferFrom never called; Bob pays 0 from his wallet.
``` [8](#0-7)

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);

        IERC20(WETH).safeTransfer(recipient, nativeBalance);

      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
