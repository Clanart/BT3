The vulnerability is real and exploitable. Here is the complete analysis:

### Title
Permissionless `refundETH()` allows any caller to steal excess ETH left on the router by a victim — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`refundETH()` has no access control and unconditionally sends the router's entire ETH balance to `msg.sender`. Because `pay()` only consumes the exact swap amount from `msg.value`, any excess ETH sent by a user remains on the router after the swap call ends. An attacker who calls `refundETH()` in a subsequent transaction receives that ETH.

### Finding Description

`refundETH()` is implemented as:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // no caller check
    }
}
``` [1](#0-0) 

The `pay()` helper, when `tokenIn == WETH`, deposits exactly the swap-required `value` from the router's native balance and leaves the remainder untouched:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [2](#0-1) 

So after `exactInputSingle{value: 1 ether}(amountIn=0.5 ETH, tokenIn=WETH)` completes, `0.5 ETH` remains on the router. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable swap functions. [3](#0-2) 

The intended safe pattern — including `refundETH` in the same `multicall` — is shown in tests: [4](#0-3) 

But `refundETH` is a standalone `external` function callable by anyone at any time. There is no mechanism that restricts it to the original depositor or to the same transaction.

### Impact Explanation

Any ETH left on the router between transactions is immediately claimable by any EOA or contract. The victim loses the full excess amount. This is a direct, unconditional transfer of user principal to an arbitrary third party — no oracle, pool state, or privileged role is required.

### Likelihood Explanation

Users sending native ETH for WETH swaps must manually compose a multicall that includes `refundETH` as the last call. Omitting it (e.g., calling `exactInputSingle` directly with `msg.value > amountIn`, or building an incomplete multicall) leaves ETH on the router. MEV bots routinely monitor for unprotected ETH balances on router contracts and can backrun the victim's transaction in the same block.

### Recommendation

Restrict `refundETH` to the active `multicall` context by recording `msg.sender` at `multicall` entry in transient storage and checking it inside `refundETH`, or change `refundETH` to accept an explicit `recipient` parameter that the caller must supply (matching Uniswap v4's approach). Alternatively, auto-refund excess ETH at the end of each swap entry point rather than relying on a separate call.

### Proof of Concept

```
1. User calls:
   router.exactInputSingle{value: 1 ether}(
       ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ETH, ...
       })
   );
   // pay() deposits exactly 0.5 ETH as WETH; 0.5 ETH stays on router.

2. Attacker (next tx, same block via backrun):
   router.refundETH();
   // refundETH sends address(this).balance (0.5 ETH) to attacker.

3. Assert: attacker.balance increased by 0.5 ETH; user lost 0.5 ETH.
```
