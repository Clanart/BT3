### Title
Stale Pending Class Returned by `starknet_getClass` After Epoch Transition — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

`get_class` for `BlockId::Tag(Tag::Pending)` reads directly from `self.pending_classes` without first calling `read_pending_data()` to validate that the pending epoch is still current. Every other pending-aware RPC handler calls `read_pending_data()` first. During the window between a new block being finalized and the pending sync restarting, `pending_classes` retains classes from the superseded epoch while `read_pending_data()` would return a synthetic empty `PendingData`. An unprivileged RPC caller can therefore receive a class body that is not declared in any current pending state diff.

---

### Finding Description

**Entrypoint — `get_class` (lines 603–609):** [1](#0-0) 

When `block_id` is `Tag::Pending`, the function immediately reads from `self.pending_classes` and returns early if the class is found — with no call to `read_pending_data()`.

**Staleness guard — `read_pending_data` (lines 1570–1593):** [2](#0-1) 

This function returns a synthetic empty `PendingData` whenever `pending_data.block.parent_block_hash() != latest_header.block_hash`. Every other pending-aware handler (`get_block`, `get_nonce`, `maybe_get_class_hash_at`) calls this guard before touching any pending state. [3](#0-2) [4](#0-3) [5](#0-4) 

**`pending_classes` is NOT cleared on `PendingSyncFinished` — `get_pending_data` (lines 160–164):** [6](#0-5) 

When the pending sync detects a new finalized block (`new_pending_parent_hash != latest_block_hash`), it returns `PendingSyncFinished` immediately. The `pending_classes.clear()` call at line 180 is only reached when the new pending data has a *different* parent hash but still matches `latest_block_hash` — a condition that is mutually exclusive with the early return above. [7](#0-6) 

**The race window:**

| State | Value |
|---|---|
| `latest_header.block_hash` | `H_N` (new finalized block) |
| `pending_data.block.parent_block_hash()` | `H_{N-1}` (stale, not yet updated) |
| `pending_classes` | contains classes declared in the old epoch (parent = `H_{N-1}`) |

During this window:
- `read_pending_data()` returns a synthetic empty `PendingData` (state diff has zero declared classes).
- `get_class(pending, class_hash)` returns the stale class from `pending_classes` without consulting `read_pending_data()`.

The window persists until the sync loop restarts `sync_pending_data` with the new `latest_block_hash` and the first new pending data fetch triggers the `pending_classes.clear()` path.

---

### Impact Explanation

An unprivileged RPC caller invoking `starknet_getClass("pending", class_hash)` receives a class body that:
- Is not declared in the current pending state diff (which is effectively empty).
- Belongs to a superseded pending epoch that may never be finalized.

The response is indistinguishable from a legitimate pending class declaration. Clients relying on this for fee estimation, simulation, or contract introspection will operate on a wrong class body for the pending block. This matches the allowed impact: **High — RPC pending view returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

The window is bounded by the sync loop's polling interval and the time for the pending source to return new data with the updated parent hash. On a live network with ~6-second block times and a non-zero `sleep_duration`, this window is routinely several seconds wide on every block transition — making it reliably triggerable by any caller who polls `starknet_getClass(pending, ...)` around block boundaries.

---

### Recommendation

In `get_class`, call `read_pending_data()` before reading from `pending_classes`, exactly as every other pending-aware handler does. If `read_pending_data()` returns a synthetic empty `PendingData` (i.e., the parent hash did not match), skip the `pending_classes` lookup entirely and fall through to the finalized-state path (or return `CLASS_HASH_NOT_FOUND` if the class is not in finalized state).

```rust
// Proposed fix sketch
let block_id = if let BlockId::Tag(Tag::Pending) = block_id {
    let txn = self.storage_reader.begin_ro_txn().map_err(internal_server_error)?;
    let pending = read_pending_data(&self.pending_data, &txn).await?;
    // Only consult pending_classes if pending data is current
    // (read_pending_data already returns empty state_diff when stale,
    //  but pending_classes is a separate lock — guard it here too)
    let declared: std::collections::HashSet<_> = pending.state_update.state_diff
        .declared_classes.iter().map(|e| e.class_hash).collect();
    let maybe_class = self.pending_classes.read().await.get_class(class_hash);
    if let Some(class) = maybe_class {
        if declared.contains(&class_hash) {
            return class.clone().try_into().map_err(internal_server_error);
        }
    }
    BlockId::Tag(Tag::Latest)
} else { block_id };
```

Alternatively, the simpler fix is to check `pending_data.block.parent_block_hash() == latest_header.block_hash` before the `pending_classes` read, mirroring the guard already inside `read_pending_data`.
