### Title
`get_state_response_part` Uses New-Epoch Shard Layout to Index Previous-Epoch Block Chunks, Breaking State-Root Binding at Resharding Boundaries — (File: `chain/chain/src/state_sync/adapter.rs`)

### Summary

`get_state_response_part` resolves the `shard_index` for `prev_block.chunks()` using the **sync block's (new) epoch shard layout**, while the companion function `compute_state_response_header` correctly uses the **previous epoch's shard layout** (`prev_shard_layout`). At a resharding boundary the two layouts differ, so the part-serving path either reads the wrong chunk's `prev_state_root` (if an existing shard's index shifted) or returns `Error::InvalidShardId` for a new child shard (index out of bounds). Either outcome breaks the invariant that every served part must be consistent with the state root committed in the header, causing state sync to fail for any shard affected by the layout change.

### Finding Description

`compute_state_response_header` (the header path) explicitly fetches the previous epoch's layout and uses it to index `prev_block.chunks()`:

```rust
// adapter.rs lines 93-101
let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;   // ← OLD layout
let chunk_header = chunks.get(prev_shard_index)                         // ← correct index
    .ok_or(Error::InvalidShardId(shard_id))?;
```

`get_state_response_part` (the part-serving path) skips this step and uses the **new** epoch's layout instead:

```rust
// adapter.rs lines 295, 305, 311-316
let epoch_id = block.header().epoch_id();                               // sync block = NEW epoch
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;     // NEW layout
// ...
let prev_block = self.chain_store.get_block(header.prev_hash())?;      // OLD epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;             // ← NEW layout index
let state_root = prev_block
    .chunks()
    .get(shard_index)                                                   // ← wrong index on OLD block
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

When the sync block is the first block of a post-resharding epoch:

- **New child shard** (e.g., shard_id=4 in a 5-shard layout, where `prev_block` has only 4 chunks): `shard_layout.get_shard_index(4)` = 4, `prev_block.chunks().get(4)` = `None` → `Error::InvalidShardId`. The part cannot be served at all.
- **Existing shard whose index shifted** (e.g., shard B moved from index 1 to index 2 because a new child was inserted before it): `shard_layout.get_shard_index(B)` = 2, `prev_block.chunks().get(2)` = chunk for the *old* shard C → wrong `state_root`. Parts are generated against the wrong root and fail the receiving node's validation (`validate_state_part` checks against the header's committed root).

The header's committed `state_root` is always computed via the correct `prev_shard_layout` path. The part-serving path uses a different root. The invariant `part.validate(header.state_root)` is broken.

### Impact Explanation

Any node performing state sync at a resharding epoch boundary — a planned, recurring event — will receive parts that either cannot be generated (Error) or fail the mandatory `validate_state_part` check on the receiving side. State sync for the affected shard stalls. Nodes that need to join or re-join the network after a resharding event cannot complete synchronization for the affected shards, preventing them from participating in consensus or serving data.

### Likelihood Explanation

Resharding is a scheduled protocol event. Every node that was offline during a resharding epoch and attempts state sync afterward hits this path. The serving node's part cache (`DBCol::StateParts`) is populated lazily; the first request for each part triggers the buggy computation. The bug is deterministically triggered by any legitimate `StateRequestPart` message for a shard whose layout changed at the sync boundary.

### Recommendation

Mirror the header path: fetch `prev_shard_layout` from the previous epoch and use it to compute `prev_shard_index` before indexing `prev_block.chunks()`:

```rust
// get_state_response_part — fix
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;  // ← use OLD layout
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

This aligns `get_state_response_part` with `compute_state_response_header` and restores the invariant that both paths derive the state root from the same epoch context.

### Proof of Concept

1. Run a network through a resharding event (epoch N → N+1 with a shard split).
2. Bring a fresh node online after the resharding; it selects a sync_hash in epoch N+1.
3. The fresh node sends `StateRequestPart` for a new child shard to a serving node.
4. The serving node calls `get_state_response_part(child_shard_id, part_id, sync_hash)`.
5. `epoch_id` = N+1 epoch; `shard_layout` = N+1 layout (5 shards); `prev_block` = last block of epoch N (4 shards).
6. `shard_layout.get_shard_index(child_shard_id)` = 4; `prev_block.chunks().get(4)` = `None`.
7. Serving node returns `Error::InvalidShardId`; fresh node cannot obtain the part; state sync stalls.

Alternatively, if the child shard's index in the new layout falls within the old chunk count (e.g., index 2 in a 4→5 shard split where the child was inserted mid-list), the serving node returns a part computed against the wrong `state_root`. The receiving node's `validate_state_part` call (in `set_state_part` at `adapter.rs:544-553` and in `ensure_shard_part_downloaded_single_attempt` at `downloader.rs:174-189`) rejects it, and state sync stalls.
