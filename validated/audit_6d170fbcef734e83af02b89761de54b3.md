### Title
Stale Base Fee Snapshot in `NewDynamicFeeChecker` Allows Cosmos Native Transactions to Bypass Current Fee Market Requirement - (File: `ante/evm/fee_checker.go`)

### Summary

`NewDynamicFeeChecker`, used to validate fees for Cosmos native transactions, captures `feemarketParams` (including `BaseFee`) at ante handler **construction time** rather than reading it fresh at **transaction execution time**. Because the EIP-1559 base fee changes every block, any Cosmos native transaction submitted after the base fee has risen can be accepted with fees calibrated to the old, lower base fee, bypassing the current fee market requirement.

### Finding Description

In `evmd/ante/handler_options.go`, `newCosmosAnteHandler` is called once (at app initialization) with a fixed `ctx`. It reads `feemarketParams` from the store at that moment and passes a pointer to the local copy into `NewDynamicFeeChecker`:

```go

    feemarketParams := options.FeeMarketKeeper.GetParams(ctx)   // captured once
    ...
    txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)

`types.GetBaseFee` simply reads `feemarketParams.GetBaseFee()` from the captured struct: [3](#0-2) 

By contrast, the **EVM transaction** path calls `EVMBlockConfig` on every invocation, which reads fresh params from the KV store (with per-block object-store caching): [4](#0-3) 

The base fee is updated every block in `feemarket` `BeginBlock`: [5](#0-4)

The `NewDynamicFeeChecker` closure never re-reads from the store, so its `feemarketParams.BaseFee` is permanently frozen at the value present when the ante handler was constructed.

### Impact Explanation

When `DynamicFeeChecker` is enabled for Cosmos native transactions, the base fee check in `NewDynamicFeeChecker` uses a stale snapshot. If the EIP-1559 base fee has risen since app startup (the normal case on a live, congested chain), any Cosmos native transaction whose fee satisfies the old (lower) base fee will pass the `feeCap >= baseFeeInt` check and have its effective fee computed against the stale base fee. The transaction is accepted and committed even though it underpays relative to the current fee market. This is a fee market ante handler bug that permits invalid transactions to commit and causes user funds/fees to be mis-accounted — matching the **High** impact category.

### Likelihood Explanation

- The `DynamicFeeChecker` flag is a deployment option; when enabled (the intended production path for EIP-1559-aware Cosmos txs), the bug is always present.
- The base fee changes every block; on any active chain the stale value diverges from the live value within seconds of startup.
- No special privileges are required: any unprivileged user can submit a Cosmos native transaction with fees calibrated to the old base fee.
- The attacker only needs to observe the current base fee (public via `eth_gasPrice` / `feemarket` query) and the stale construction-time base fee (observable from node startup logs or genesis) to craft an underpaying transaction.

### Recommendation

Replace the construction-time parameter snapshot with a live read inside the `NewDynamicFeeChecker` closure. The simplest fix is to pass the `FeeMarketKeeper` interface into the checker and call `GetBaseFee(ctx)` on each invocation, mirroring the pattern used by `newEthAnteHandler` via `EVMBlockConfig`:

```go
func NewDynamicFeeChecker(ethCfg *params.ChainConfig, evmParams *types.Params, fmKeeper FeeMarketKeeper) authante.TxFeeChecker {
    return func(ctx sdk.Context, tx sdk.Tx) (sdk.Coins, int64, error) {
        ...
        baseFee := fmKeeper.GetBaseFee(ctx)  // live read per invocation
```

Alternatively, align `newCosmosAnteHandler` with `newEthAnteHandler` by making it a closure that re-reads params on every call rather than capturing them at construction time.

### Proof of Concept

1. Chain starts with `BaseFee = 1_000_000_000` (genesis default). `newCosmosAnteHandler` is called; `feemarketParams.BaseFee = 1_000_000_000` is frozen in the `NewDynamicFeeChecker` closure.
2. Several blocks pass with high gas usage. `feemarket.BeginBlock` updates `BaseFee` to `2_000_000_000` in the KV store.
3. Attacker submits a Cosmos native transaction with `fee = 1_000_000_000 * gasLimit` (satisfying the stale base fee).
4. `NewDynamicFeeChecker` evaluates `feeCap = fee / gas = 1_000_000_000 >= baseFeeInt (stale) = 1_000_000_000` → passes.
5. The transaction is accepted and committed despite paying only half the current required base fee.
6. An EVM transaction with the same gas price would be rejected by `CheckEthGasConsume` because `EVMBlockConfig` reads the live `BaseFee = 2_000_000_000`. [7](#0-6)

func GetBaseFee(height int64, ethCfg *params.ChainConfig, feemarketParams *feemarkettypes.Params) *big.Int {
	if !IsLondon(ethCfg, height) {
		return nil
	}
	baseFee := feemarketParams.GetBaseFee()
	// should not be nil if london hardfork enabled
	if baseFee == nil {
		return new(big.Int)
	}
	return baseFee
}

func (k *Keeper) BeginBlock(ctx sdk.Context) error {
	baseFee := k.CalculateBaseFee(ctx)

	// return immediately if base fee is nil
	if baseFee == nil {
		return nil
	}

	k.SetBaseFee(ctx, baseFee)

	defer func() {
		telemetry.SetGauge(float32(baseFee.Int64()), "feemarket", "base_fee") //nolint:staticcheck
	}()

	// Store current base fee in event
	ctx.EventManager().EmitEvents(sdk.Events{
		sdk.NewEvent(
			types.EventTypeFeeMarket,
			sdk.NewAttribute(types.AttributeKeyBaseFee, baseFee.String()),
		),
	})
	return nil
