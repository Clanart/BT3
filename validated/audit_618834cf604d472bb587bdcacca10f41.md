### Title
`deploy` Syscall Hardcodes `failure_flag=0`, Silently Masking Constructor Failures and Enabling Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

### Summary

The `execute_deploy` function in `syscall_impls.cairo` unconditionally writes `failure_flag=0` in the syscall response header, regardless of whether the deployed contract's constructor succeeded or failed. This is structurally analogous to the TokenVesting bug: just as `isActive=false` silently blocked withdrawal of already-vested tokens, `failure_flag=0` silently blocks the calling contract from detecting a failed deployment. Any contract that deploys a sub-contract and then transfers funds to the returned address — a standard factory pattern — will permanently freeze those funds if the constructor reverted.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–553), the OS calls `deploy_contract` and then unconditionally writes the response with `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
...
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `DeployResponse` is then written with the `contract_address` regardless of constructor outcome:

```cairo
assert [response] = DeployResponse(
    contract_address=contract_address,
    constructor_retdata_start=retdata,
    constructor_retdata_end=retdata + retdata_size,
);
``` [2](#0-1) 

When the constructor fails, `execute_entry_point` creates a fresh revert log, runs `call_execute_syscalls`, and then calls `handle_revert` to undo all state changes made by the constructor — including the class hash assignment for the newly deployed address: [3](#0-2) 

After `handle_revert`, the contract at the computed address has `class_hash=0` (no code). Yet the OS has already told the calling contract: "deployment succeeded, here is the address." The calling contract has no protocol-level mechanism to detect the failure because `failure_flag` is the only signal, and it is always `0`.

Compare with `contract_call_helper`, which correctly propagates `is_reverted` into `failure_flag`: [4](#0-3) 

The `deploy` path has no equivalent propagation — the TODO comment confirms this is a known gap, not an intentional design choice.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A factory contract that follows the standard pattern of deploying a sub-contract and then transferring tokens to it cannot distinguish a failed deployment from a successful one. After a failed constructor:

- The deployed address has `class_hash=0` — no entry points exist.
- Any ERC-20 tokens transferred to that address via `transfer` are recorded in the fee token's storage under that key.
- No function can ever be called on the address to recover them.
- The state is final and irreversible on-chain.

This matches the TokenVesting analog exactly: in that bug, `isActive=false` made vested tokens permanently unwithdrawable; here, `failure_flag=0` makes tokens sent to a failed-deployment address permanently unrecoverable.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender:

1. An attacker declares a class whose constructor always reverts (e.g., `assert 1 = 0`).
2. The attacker calls a victim factory contract (or deploys their own) that: (a) calls the `deploy` syscall with the malicious class, (b) transfers tokens to the returned address.
3. Because `failure_flag=0`, the factory contract's conditional logic (if any) cannot branch on failure.
4. Tokens are transferred to the address with `class_hash=0`.

Alternatively, the attacker can supply minimal gas to cause an out-of-gas failure in any constructor, triggering the same outcome without a specially crafted class. Factory contracts are a ubiquitous pattern in DeFi protocols built on StarkNet, making this a realistic and high-likelihood attack surface.

---

### Recommendation

Propagate the constructor's `is_reverted` return value into the `failure_flag` field of the `ResponseHeader`, mirroring the pattern already used in `contract_call_helper`:

```cairo
// After deploy_contract returns, check is_reverted.
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

If the constructor reverted, the `contract_address` field in `DeployResponse` should be set to `0` (or a sentinel) so the calling contract cannot accidentally use it. The revert of the constructor's state changes (via `handle_revert`) is already correctly implemented; only the response reporting is missing.

---

### Proof of Concept

```
// Attacker-declared class (constructor always fails):
@constructor
func constructor{...}(...) {
    assert 1 = 0;  // always reverts
    return ();
}

// Victim factory contract (standard pattern):
@external
func deploy_and_fund{...}(class_hash: felt, salt: felt, amount: Uint256) {
    // Deploy sub-contract
    let (contract_address) = deploy(
        class_hash=class_hash,
        contract_address_salt=salt,
        constructor_calldata_size=0,
        constructor_calldata=cast(0, felt*),
        deploy_from_zero=0,
    );
    // failure_flag is always 0 — factory cannot detect failure
    // Transfers tokens to the address, believing it is a valid contract
    IERC20.transfer(token_address, contract_address, amount);
    // `contract_address` has class_hash=0; tokens are permanently frozen
    return ();
}
```
