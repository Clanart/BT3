### Title
Permanent Fund Lock When Recipient Is Blacklisted on Destination EVM Chain — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` transfers tokens directly to `payload.recipient` with no error-handling fallback. If the recipient address is blacklisted in the destination token contract (e.g., USDC) after the cross-chain transfer is initiated but before it is finalized, every call to `finTransfer` will revert. Because the MPC-signed payload encodes the recipient immutably and no recovery function exists, the source-chain funds (already burned or locked) are permanently irrecoverable.

---

### Finding Description

`OmniBridge.finTransfer` marks the destination nonce as used and then unconditionally transfers tokens to `payload.recipient`: [1](#0-0) 

```solidity
completedTransfers[payload.destinationNonce] = true;
```

Then, for a non-bridge ERC-20 (e.g., native USDC locked in the bridge): [2](#0-1) 

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
```

`SafeERC20.safeTransfer` propagates any revert from the token contract. USDC (and other tokens with a `notBlacklisted` modifier) will revert if `payload.recipient` is blacklisted. Because the entire transaction reverts, `completedTransfers[payload.destinationNonce]` is also rolled back, so the nonce is never consumed — but this provides no relief: the signed payload is produced by the MPC network and encodes `payload.recipient` immutably. There is no function to re-sign the payload with a different recipient, and no admin or user-facing withdrawal function to rescue the locked collateral.

The same revert-on-delivery risk exists for:
- **Native ETH delivery** (line 319): `payload.recipient.call{value: payload.amount}("")` — a contract recipient that rejects ETH causes `FailedToSendEther` permanently.
- **ERC-1155 delivery** (line 324): `safeTransferFrom` to a recipient that does not implement `onERC1155Received`. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Tokens burned or locked on the source chain (NEAR or another EVM chain) cannot be recovered:
- On NEAR, `ft_on_transfer` → `init_transfer_internal` burns bridged tokens or locks native tokens before any cross-chain message is sent.
- On EVM, `initTransfer` burns bridge tokens (`BridgeToken.burn`) or transfers ERC-20 collateral into the bridge contract before emitting the cross-chain event. [4](#0-3) 

Once the source-side action is irreversible and the MPC signature is issued for a blacklisted recipient, the funds are permanently locked with no on-chain escape hatch.

---

### Likelihood Explanation

**Medium.** USDC blacklisting is an active, real-world mechanism exercised by Circle. Cross-chain finalization on the NEAR bridge can take minutes to hours (EVM→NEAR) or longer depending on relayer latency. Any window between `initTransfer` and `finTransfer` is sufficient for a blacklisting event to occur. The scenario is not theoretical: Circle has blacklisted thousands of addresses on mainnet.

---

### Recommendation

1. **Wrap token delivery in a try/catch** (Solidity `try`/`catch` on external calls, or a low-level `call` with return-value inspection). On failure, credit the amount to an internal `claimable[recipient][token]` mapping instead of reverting.
2. **Add a `claimBlacklistedFunds(address token, address to)` function** callable by the original recipient (or an admin) once the blacklist is lifted, mirroring the fix described in the referenced report.
3. **Alternatively**, allow the NEAR bridge to re-sign a corrected payload with a fallback recipient when the original recipient is provably blacklisted, though this requires protocol-level changes.

---

### Proof of Concept

**Scenario (NEAR → EVM, USDC):**

1. Alice holds bridged USDC on NEAR and calls `ft_transfer_call` to the NEAR bridge with `recipient = Bob_EVM_address`.
2. The NEAR bridge burns Alice's bridged USDC and stores the pending transfer.
3. The MPC network signs a `TransferMessagePayload` with `recipient = Bob_EVM_address` and `tokenAddress = USDC_EVM`.
4. Before the relayer submits `finTransfer` on EVM, Circle blacklists `Bob_EVM_address` in the USDC contract.
5. The relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
6. Execution reaches:
   ```solidity
   IERC20(USDC_EVM).safeTransfer(Bob_EVM_address, amount);
   // reverts: "Blacklistable: account is blacklisted"
   ``` [5](#0-4) 
7. The transaction reverts. `completedTransfers[nonce]` is rolled back.
8. Every subsequent attempt to call `finTransfer` with the same signed payload reverts identically.
9. No function exists in `OmniBridge.sol` to redirect or recover the funds.
10. Alice's USDC is permanently burned on NEAR with no recourse. [6](#0-5)

        bytes calldata signatureData,

        completedTransfers[payload.destinationNonce] = true;

        bytes memory borshEncoded = bytes.concat(

            Borsh.encodeUint64(payload.destinationNonce),
            bytes1(payload.originChain),
            Borsh.encodeUint64(payload.originNonce),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.tokenAddress),
            Borsh.encodeUint128(payload.amount),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.recipient),
            bytes(payload.feeRecipient).length == 0 // None or Some(String) in rust
                ? bytes("\x00")
                : bytes.concat(
                    bytes("\x01"),
                    Borsh.encodeString(payload.feeRecipient)
                ),
            bytes(payload.message).length == 0
                ? bytes("")
                : Borsh.encodeBytes(payload.message)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();

        MultiTokenInfo memory multiToken = multiTokens[payload.tokenAddress];
