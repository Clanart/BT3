### Title
EIP-7702 Authorization Signature Replay Across Chains via Zero ChainID — (`File: x/evm/keeper/set_code_authorizations.go`)

### Summary

The `validateAuthorization` function in Ethermint's EIP-7702 implementation explicitly permits authorizations signed with a zero (`0`) chain ID. Such authorizations are valid on **every** Ethermint-based chain simultaneously, allowing an attacker to replay a victim's signed EIP-7702 delegation on any other chain where the victim's account nonce matches, installing arbitrary code at the victim's EOA address without their consent on that chain.

### Finding Description

In `validateAuthorization`, the chain ID check is:

```go
if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

When `auth.ChainID` is zero, the condition short-circuits and the check is **skipped entirely**. The authorization is then validated purely by nonce and signature, with no binding to any specific chain. [1](#0-0) 

This is intentional per EIP-7702 (zero chain ID means "valid on all chains"), but it creates a direct replay vector: a single signed authorization with `chainId=0` can be submitted on any EVM chain where the authority's nonce matches `auth.Nonce`. The nonce check at line 39 only prevents reuse on the **same** chain after the nonce is consumed; it does not prevent replay on a different chain where the nonce has not yet been consumed.

The `setAuthorizationDelegation` function then unconditionally installs the delegation code and bumps the nonce:

```go
stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), ...)
``` [2](#0-1) 

The ante handler's `ValidateEthBasic` performs no per-authorization chain ID check — it only validates that the authorization list is non-empty and that the outer transaction is not a contract creation: [3](#0-2) 

The outer EIP-7702 transaction itself is EIP-155 protected (chain-ID bound), but the **inner authorization tuples** are independently signed and can carry `chainId=0`. The outer transaction's chain ID binding does not protect the inner authorization signatures.

### Impact Explanation

An attacker who observes a victim's EIP-7702 authorization signed with `chainId=0` on one Ethermint chain (e.g., chain A) can replay that exact authorization tuple inside a new outer EIP-7702 transaction on a different Ethermint chain (chain B), provided the victim's account nonce on chain B equals `auth.Nonce`. The result is that the attacker installs arbitrary delegation code (`ef0100<delegate_address>`) at the victim's EOA on chain B without the victim's knowledge or consent. The victim's EOA on chain B then executes the attacker-chosen delegate's code for all subsequent calls, enabling:

- Theft of all EVM-denom funds held by the victim's EOA on chain B (the delegate can drain via `CALL(CALLER, SELFBALANCE, ...)`).
- Unauthorized mutation of the victim's account code on chain B.

This matches the allowed High impact: **EIP-7702 authorization signer verification bypass enabling replay and unauthorized account/code mutation**.

### Likelihood Explanation

- Zero-chainID EIP-7702 authorizations are a documented feature of the EIP-7702 spec and are produced by standard tooling (e.g., `web3.py`, `ethers.js`) when users explicitly set `chainId=0` for cross-chain portability.
- An attacker only needs to observe a victim's zero-chainID authorization on any public chain (mempool, block explorer, or RPC) and replay it on another chain where the victim's nonce matches.
- No privileged access is required. The attacker submits a standard EIP-7702 transaction via `eth_sendRawTransaction`.
- Nonce matching is the only barrier, and for fresh accounts or accounts with predictable nonce sequences, this is trivially satisfied.

### Recommendation

1. **Reject zero-chainID authorizations at the ante handler level** in `ValidateEthBasic` (or in a dedicated EIP-7702 authorization validation step), unless the operator explicitly opts in to cross-chain authorizations.
2. Alternatively, add a per-authorization chain ID check in `validateAuthorization` that rejects `chainId=0` when the node is configured to require chain-ID binding:
   ```go
   if auth.ChainID.IsZero() && !allowCrossChainAuthorizations {
       return authority, core.ErrAuthorizationWrongChainID
   }
   ```
3. Document clearly that zero-chainID authorizations are accepted and warn users/wallets against producing them.

### Proof of Concept

**Setup:**
- Chain A: Ethermint chain ID `9000`, victim account `V` at nonce `0`.
- Chain B: Ethermint chain ID `9001`, victim account `V` at nonce `0` (same key, fresh account).

**Step 1 — Victim signs a zero-chainID authorization on chain A:**
```python
auth = {"chainId": 0, "address": attacker_delegate, "nonce": 0}
signed_auth = victim.sign_authorization(auth)
```

**Step 2 — Attacker observes `signed_auth` (from mempool or block on chain A).**

**Step 3 — Attacker submits a replay transaction on chain B:**
```python
setcode_tx = {
    "chainId": 9001,   # chain B — outer tx is chain-bound
    "type": 4,
    "to": victim_address,
    "nonce": attacker_nonce_on_chain_B,
    "authorizationList": [signed_auth],  # victim's zero-chainID auth replayed
    ...
}
```

**Step 4 — `validateAuthorization` on chain B:**
- `auth.ChainID.IsZero()` → `true` → chain ID check skipped.
- `stateDB.GetNonce(victim)` == `0` == `auth.Nonce` → nonce check passes.
- `setAuthorizationDelegation` installs `ef0100<attacker_delegate>` at victim's address on chain B. [4](#0-3) [5](#0-4) 

**Step 5 — Victim's EOA on chain B now executes attacker-controlled code, enabling fund drainage.**

func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
	// Verify chain ID is null or equal to current chain ID.
	if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
		return authority, core.ErrAuthorizationWrongChainID
	}
	// Limit nonce to 2^64-1 per EIP-2681.
	if auth.Nonce+1 < auth.Nonce {
		return authority, core.ErrAuthorizationNonceOverflow
	}
	// Validate signature values and recover authority.
	authority, err = auth.Authority()
	if err != nil {
		return authority, fmt.Errorf("%w: %v", core.ErrAuthorizationInvalidSignature, err)
	}
	// Check the authority account
	//  1) doesn't have code or has exisiting delegation
	//  2) matches the auth's nonce
	//
	// Note it is added to the access list even if the authorization is invalid.
	stateDB.AddAddressToAccessList(authority)
	code := stateDB.GetCode(authority)
	if _, ok := types.ParseDelegation(code); len(code) != 0 && !ok {
		return authority, core.ErrAuthorizationDestinationHasCode
	}
	if have := stateDB.GetNonce(authority); have != auth.Nonce {
		return authority, core.ErrAuthorizationNonceMismatch
	}
	return authority, nil
}
```

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
