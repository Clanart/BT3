### Title
Dust-Amount Transfer Permanently Locks User Funds When `normalize_amount` Rounds to Zero — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` accepts any non-zero token amount that satisfies `fee < amount`, burns or locks the tokens immediately, and stores the pending transfer. However, `sign_transfer` later calls `normalize_amount(amount_without_fee, decimals)` and hard-reverts if the result is zero. For tokens whose NEAR representation has more decimals than the destination chain (e.g., `origin_decimals = 24`, `decimals = 18`), any amount smaller than `10^(origin_decimals − decimals)` normalizes to zero. Because there is no user-callable cancel or refund path for a pending transfer, the user's tokens are permanently burned or locked with no recovery.

---

### Finding Description

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

For a token with `origin_decimals = 24` and `decimals = 18`, `diff_decimals = 6`, so any `amount_without_fee < 1_000_000` normalizes to `0`.

`init_transfer` only validates `fee.fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

It then immediately burns (deployed tokens) or locks (native tokens) the full amount inside `init_transfer_internal`:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [3](#0-2) 

Later, when the relayer calls `sign_transfer`, the normalized amount is checked:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

If `amount_without_fee < 10^diff_decimals`, this `require!` always panics. The transfer message remains in storage indefinitely, and there is no public cancel or refund function that would return the burned/locked tokens to the user.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

A user who initiates a transfer with a dust amount (below the decimal normalization threshold) has their tokens immediately burned or locked. `sign_transfer` will revert every time it is called for that transfer ID. No recovery path exists: there is no user-callable cancel, no timeout-based refund, and no DAO escape hatch that restores the specific user's tokens. The funds are irrecoverably lost.

---

### Likelihood Explanation

**Low-to-Medium.** The condition is reachable by any unprivileged user who calls `ft_transfer_call` with a small amount. It is most likely to occur:

- For tokens with a large decimal gap (e.g., NEAR native token: `origin_decimals = 24`, EVM: `decimals = 18`, threshold = 1,000,000 yoctoNEAR ≈ 10⁻¹⁸ NEAR).
- Accidentally, when a user sends a dust balance left over from a previous operation.
- Intentionally, by a griefing attacker who sends dust on behalf of a victim (if the token's `ft_transfer_call` allows a third-party sender).

The decimal gap is a protocol-level property, not a configuration error, so the vulnerable condition exists for any supported token pair where `origin_decimals > decimals`.

---

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) **before** burning or locking tokens:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    token_id.clone(),
).near_expect(BridgeError::FailedToGetTokenAddress);

let decimals = self.token_decimals
    .get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);

let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing check in `sign_transfer` but enforces it at the entry point, before any state mutation, so the user's tokens are never consumed for an undeliverable transfer.

---

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 24`, `decimals = 18` (6-decimal gap, threshold = 1,000,000 units).

1. User calls `ft_transfer_call` on the NEAR token contract with `amount = 500_000` (below threshold), `msg` encoding `InitTransferMsg { fee: U128(0), ... }`.
2. `ft_on_transfer` → `init_transfer` → `require!(0 < 500_000)` passes.
3. `init_transfer_internal` burns 500,000 tokens (deployed token) or locks them; stores transfer message; returns `U128(0)` (no refund).
4. Relayer calls `sign_transfer(transfer_id, ...)`.
5. `normalize_amount(500_000, Decimals{decimals:18, origin_decimals:24}) = 500_000 / 1_000_000 = 0`.
6. `require!(0 > 0, ...)` → **panic: `InvalidAmountToTransfer`**.
7. Transfer message persists in `pending_transfers`; tokens are permanently burned/locked; no recovery path exists.
