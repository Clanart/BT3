### Title
`ProofOfInclusion::valid()` Does Not Enforce Leaf-Node Invariant, Allowing Forged Inclusion Proof for Internal Subtree — (`crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` only verifies hash-chain consistency from `node_hash` up to the root. It does **not** check that `node_hash` corresponds to a leaf node. An attacker who knows the tree structure can deserialize a `ProofOfInclusion` (via `Streamable`/Python bindings) with `node_hash` set to an internal node hash and construct consistent layers, causing `valid()` to return `true` for a hash that is not a leaf.

---

### Finding Description

`ProofOfInclusion::valid()` is the sole public verification method: [1](#0-0) 

It starts from `self.node_hash`, iterates layers calling `calculate_internal_hash`, and returns `true` if the final hash equals `root_hash()`. There is no check that `node_hash` is a leaf hash rather than an internal node hash.

Internal node hashes are computed with a `\x02` prefix: [2](#0-1) 

Leaf hashes are stored as externally-provided values with no distinguishing prefix enforced by the tree. The `valid()` function treats both identically — it simply uses `node_hash` as the starting point for the hash chain.

`ProofOfInclusion` derives `Streamable` and `PyStreamable`, meaning it can be deserialized from attacker-controlled bytes: [3](#0-2) 

The Python binding exposes `py_valid()` which directly calls `valid()`: [4](#0-3) 

`get_proof_of_inclusion()` correctly sets `node_hash` to a leaf hash: [5](#0-4) 

But nothing prevents a caller from constructing a `ProofOfInclusion` with `node_hash` set to an internal node hash and passing it to `valid()`.

---

### Impact Explanation

A verifier that calls `proof.valid()` as the sole check — which is the natural and documented usage — would accept a proof where `node_hash` is an internal subtree hash, not a specific key-value leaf. The verifier is deceived into believing a specific leaf is proven when it is not. This falls under the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion**.

Note: `get_node_by_hash()` uses `leaf_hash_to_index` and would reject an internal hash: [6](#0-5) 

However, this is a separate function. `valid()` itself provides no such guard, and verifiers that do not call `get_node_by_hash` afterward are fully exposed.

---

### Likelihood Explanation

DataLayer tree data is typically public. An attacker can observe the tree, extract any internal node hash, construct consistent `ProofOfInclusionLayer` entries chaining that hash to the root, and present the forged `ProofOfInclusion` to any verifier that relies solely on `proof.valid()`. The `Streamable` and Python binding interfaces make this trivially constructable without any privileged access.

---

### Recommendation

`valid()` must verify that `node_hash` is not an internal node hash. The simplest fix is to check that `node_hash` does not equal any value produced by `internal_hash` — practically, this means the tree must distinguish leaf hashes from internal hashes at the type or prefix level. Options:

1. **Domain-separate leaf hashes**: prefix leaf hashes with `\x01` (mirroring the `\x02` prefix for internal nodes) so that `valid()` can assert `node_hash` was not produced by `internal_hash`.
2. **Add a `is_leaf: bool` field** to `ProofOfInclusion` and verify it in `valid()`, with `get_proof_of_inclusion` always setting it to `true`.
3. **Validate in `valid()`** that `node_hash` is present in the `leaf_hash_to_index` cache — but this requires passing the `MerkleBlob` reference, which changes the API.

---

### Proof of Concept

```python
# Given a MerkleBlob with at least 2 leaves, extract an internal node hash.
# The root (index 0) is an internal node with a known hash.
# Construct ProofOfInclusion(node_hash=root_hash, layers=[]) — zero layers means
# root_hash() == node_hash, so valid() trivially returns True.

from chia_rs import ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob

# Build a tree with two leaves
blob = MerkleBlob(b"")
blob.insert(key_id_1, val_id_1, leaf_hash_1, ...)
blob.insert(key_id_2, val_id_2, leaf_hash_2, ...)
blob.calculate_lazy_hashes()

root_hash = blob.get_hash_at_index(0)  # internal node hash

# Forged proof: node_hash is the root (internal), zero layers
forged = ProofOfInclusion(node_hash=root_hash, layers=[])
assert forged.valid()  # True — no leaf check performed
# root_hash is NOT a leaf hash; no key-value pair corresponds to it
```

For a deeper tree, the attacker picks any internal node hash `I` and constructs layers from `I` up to the root using the real sibling hashes, all of which are publicly observable. `valid()` returns `true` for the forged proof. [1](#0-0)

    pub fn valid(&self) -> bool {
        let mut existing_hash = self.node_hash;

        for layer in &self.layers {
            let calculated_hash = crate::calculate_internal_hash(
                &existing_hash,
                layer.other_hash_side,
                &layer.other_hash,
            );

            if calculated_hash != layer.combined_hash {
                return false;
            }

            existing_hash = calculated_hash;
        }

        existing_hash == self.root_hash()
    }

pub fn internal_hash(left_hash: &Hash, right_hash: &Hash) -> Hash {
    let mut hasher = Sha256::new();
    hasher.update(b"\x02");
    hasher.update(left_hash.0);
    hasher.update(right_hash.0);

    Hash(Bytes32::new(hasher.finalize()))
}
