Audit Report

## Title
Implicit ETH auto-consumption in `pay` lets any caller steal ETH left on the router by prior users — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments::pay` reads `address(this).balance` to cover WETH swap payments without distinguishing ETH contributed by the current caller from ETH stranded by prior callers. Any ETH left on the router from a previous payable call that omitted `refundETH()` is silently consumed by the next WETH-swap caller, resulting in direct ETH theft at zero cost to the attacker.

## Finding Description
The WETH branch of `pay` (lines 73–84) reads the router's total native balance and uses it to wrap and forward ETH before pulling from the registered `payer`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

There is no accounting for which caller deposited the ETH currently held by the contract.

**ETH accumulation path**: The `receive()` guard at lines 32–34 only blocks plain ETH transfers (no calldata). It does **not** block ETH sent as `msg.value` with function calls. All swap entry points — `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` — are `payable`. [2](#0-1) [3](#0-2) 

When a user calls `exactInputSingle{value: 1 ether}` with `amountIn = 0.5 ether`, `pay` wraps exactly 0.5 ETH and sends it to the pool. The remaining 0.5 ETH stays on the contract with no owner record. The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` confirms this pattern and shows that `refundETH()` must be explicitly appended — it is never automatic. [4](#0-3) 

`pay` is invoked from both callback paths:
- `_justPayCallback` (single-hop exact-input/output, line 192–199)
- `_exactOutputIterateCallback` final leg (line 207–213) [5](#0-4) [6](#0-5) 

In both cases `_getPayer()` returns the original `msg.sender` stored in transient storage, but the WETH branch bypasses the payer entirely when `nativeBalance > 0`.

**Exploit flow:**
1. User A calls `exactInputSingle{value: 1 ether}(amountIn=0.5 ether, tokenIn=WETH)` without appending `refundETH()`. After the call, `address(router).balance == 0.5 ether`.
2. Attacker calls `exactInputSingle{value: 0}(amountIn=0.5 ether, tokenIn=WETH)` with no WETH balance and no approval.
3. Pool calls `metricOmmSwapCallback`; router calls `pay(WETH, attacker, pool, 0.5 ether)`.
4. `nativeBalance = 0.5 ether >= value` → router wraps User A's 0.5 ETH and transfers WETH to the pool.
5. Attacker receives output tokens. User A loses 0.5 ETH. Attacker spent nothing.

## Impact Explanation
Direct, measurable loss of user ETH principal. The victim loses the full stranded amount; the attacker pays zero ETH and zero WETH. This is a concrete fund-loss above Sherlock thresholds for any non-trivial swap size, satisfying the "direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The precondition is a user sending excess `msg.value` without appending `refundETH()` — a realistic and documented user error in Uniswap-style routers (the codebase's own tests show `refundETH()` must be manually included). An attacker can passively monitor the router's ETH balance on-chain or watch mempool multicall transactions and back-run any that leave a residual. No privileged access, special tokens, or malicious setup is required.

## Recommendation
Track the ETH budget contributed by the current top-level call rather than reading the total contract balance. Two concrete approaches:
1. Record `msg.value` in a transient slot at each payable entry point and deduct from it inside `pay`, reverting if the current call's ETH budget is exhausted.
2. Pass `msg.value` as an explicit parameter through the call chain into `pay`, and use only that value in the WETH branch — always pulling any remainder from the payer via `safeTransferFrom`, eliminating the implicit `address(this).balance` read entirely.

## Proof of Concept
