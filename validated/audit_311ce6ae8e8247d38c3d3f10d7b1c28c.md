### Title
`batch_insert` Bypasses Duplicate Key/Hash Validation Present in `insert` - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::insert()` enforces two invariants before writing a leaf: no duplicate `KeyId` and no duplicate leaf `Hash`. `MerkleBlob::batch_insert()` calls `insert()` for at most the last two items in the batch (and only when the tree has ≤1 existing leaves), then writes all remaining items directly via `insert_entry_to_blob()` with no duplicate validation at all. An attacker-controlled batch containing duplicate keys or hashes silently corrupts the Merkle tree, producing an invalid root and enabling forged inclusion proofs.

### Finding Description

`MerkleBlob::insert()` guards both uniqueness invariants before writing:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert()` takes a different path for all but the last two items:

```rust
// blob.rs:578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        // only these two items go through insert() with duplicate checks
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}

for ((key, value), hash) in keys_values_hashes {
    // ALL remaining items bypass KeyAlreadyPresent / HashAlreadyPresent checks
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

When the tree already holds ≥2 leaves the `if` block is skipped entirely, so **every** item in the batch bypasses both checks. When the tree holds ≤1 leaves, only the last two items (popped via `.pop()`) are validated; all earlier items in the `Vec` are written unchecked.

The Python binding `py_batch_insert` (lines 1503–1519) passes the caller-supplied `keys_values` and `hashes` vectors directly into `batch_insert` with no pre-filtering, making this reachable from untrusted Python input.

### Impact Explanation

Inserting a duplicate `KeyId` produces two leaf nodes with the same key in the blob. The `block_status_cache.key_to_index` map (a `HashMap<KeyId, TreeIndex>`) will map the key to whichever index was written last, silently losing the earlier entry. The Merkle root computed over the corrupted tree is wrong. Any `get_proof_of_inclusion` call for the shadowed entry will either fail or return a proof that does not match the committed root, enabling forged inclusion/exclusion proofs and committed DataLayer state corruption.

Inserting a duplicate leaf `Hash` similarly corrupts `leaf_hash_to_index`, breaking hash-based lookups and delta-file operations that rely on `leaf_hash_to_index` for correctness.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

The Python binding `py_batch_insert` is a public API callable by any DataLayer client. A caller that passes a batch of ≥3 items (or any batch against a tree with ≥2 existing leaves) with a repeated `KeyId` or `Hash` triggers the bug with no special privileges. The condition is easy to reach accidentally (e.g., retrying a failed batch) or deliberately.

### Recommendation

Add the same duplicate checks that `insert()` performs at the top of the bulk loop in `batch_insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, pre-validate the entire input `Vec` for internal duplicates before writing any node, mirroring the fix applied in the Juicebox report (check both directions before committing).

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Pre-populate with 2 leaves so leaf_count >= 2 → the if-block is skipped entirely
blob.insert(KeyId(100), ValueId(100), hashlib.sha256(b"seed100").digest())
blob.insert(KeyId(200), ValueId(200), hashlib.sha256(b"seed200").digest())

# Now batch_insert with a duplicate KeyId(100) — no KeyAlreadyPresent error raised
dup_key   = KeyId(100)          # already in the tree
new_key   = KeyId(300)
blob.batch_insert(
    [(dup_key, ValueId(999)), (new_key, ValueId(300))],
    [hashlib.sha256(b"dup").digest(), hashlib.sha256(b"seed300").digest()],
)
# Tree now contains two leaf nodes with KeyId(100); root hash is corrupted.
blob.calculate_lazy_hashes()
proof = blob.get_proof_of_inclusion(dup_key)

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
