### Title
Missing State Diff Commitment Verification in P2P Sync Allows Malicious Peer to Corrupt `contract_storage` Table — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

The p2p sync client assembles a `ThinStateDiff` from peer-delivered `StateDiffChunk` messages and writes it directly to the `contract_storage` table via `append_state_diff` → `write_storage_diffs` without ever computing `calculate_state_diff_hash` on the assembled diff and comparing it to the `state_diff_commitment` stored in the block header. A malicious peer can substitute any storage value V' for (C, K) while preserving the total entry count, pass all existing guards, and cause `get_storage_at` to return V' for every subsequent query against that slot.

### Finding Description

**Entrypoint — `parse_data_for_block`** in `crates/apollo_p2p_sync/src/client/state_diff.rs`: [1](#0-0) 

The function reads `target_state_diff_len` from the stored header's `state_diff_length` field and loops until `current_state_diff_len == target_state_diff_len`. The only guards applied are:

1. **Count check**: total entry count must equal `state_diff_length` from the header.
2. **Duplicate-key check**: `unite_state_diffs` returns `BadPeerError::ConflictingStateDiffParts` if the same key appears twice.
3. **Deprecated-class dedup**: `validate_deprecated_declared_classes_non_conflicting`.

There is **no call** to `calculate_state_diff_hash` and **no comparison** against `header.state_diff_commitment`. The assembled `ThinStateDiff` is returned as-is. [2](#0-1) 

`write_to_storage` immediately calls `append_state_diff(block_number, tampered_diff)` with no further validation.

**Storage write — `write_storage_diffs`** in `crates/apollo_storage/src/state/mod.rs`: [3](#0-2) 

Every `(address, key) → value` pair from the tampered diff is written verbatim to the `contract_storage` table keyed by `((ContractAddress, StorageKey), BlockNumber)`. No commitment check exists here either.

**The commitment that should be checked** is `header.state_diff_commitment`, a Poseidon hash computed by `calculate_state_diff_hash`: [4](#0-3) 

This hash covers the full `storage_diffs` content (addresses, keys, and values). The p2p sync stores this commitment in the header table but never uses it to validate the state diff body.

**Central sync has the same gap**, explicitly acknowledged with a TODO: [5](#0-4) 

**Header signature verification is also absent** in the p2p sync header handler — `parse_data_for_block` for headers only checks block number ordering and signature vector length, never calling `verify_block_signature`: [6](#0-5) 

This means an attacker can also serve a fabricated header with an arbitrary `state_diff_commitment`, but even without that, serving a header with the correct commitment and a tampered diff body is sufficient because the body hash is never checked.

### Impact Explanation

Once V' is written to `contract_storage` at key `((C, K), block_N)`, every call to `get_storage_at(state_after_block_N, C, K)` returns V' instead of V. This value is consumed directly by the blockifier's `get_storage_at` syscall during transaction execution, corrupting execution results for every contract that reads that storage slot in any block after N. The impact is **Critical**: wrong storage value returned by blockifier syscall, corrupting execution results for all contracts reading that slot.

### Likelihood Explanation

Any node that can establish a p2p connection and respond to `StateDiff` protocol queries can execute this attack. No operator, validator, or sequencer privileges are required. The attack is silent — no error is raised, no peer is reported, and the tampered value is committed to durable storage.

### Recommendation

In `parse_data_for_block` (or immediately before `write_to_storage`), after assembling the full `ThinStateDiff`, compute:

```rust
let calculated = calculate_state_diff_hash(&result);
let expected = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("header must exist")
    .state_diff_commitment
    .ok_or(P2pSyncClientError::OldHeaderInStorage { ... })?;
if calculated != expected {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffHash { ... }));
}
```

The same check should be added in `store_state_diff` in `crates/apollo_central_sync/src/lib.rs` to resolve the existing TODO.

### Proof of Concept

The attack path in production code:

1. Attacker peer serves a `SignedBlockHeader` for block N with `state_diff_length = 1` and the real `state_diff_commitment` (or any value, since signatures are not verified).
2. Attacker peer serves one `StateDiffChunk::ContractDiff` with `contract_address = C`, `storage_diffs = {K: V'}` where V' ≠ V (the committed value). Entry count = 1 = `target_state_diff_len`. [7](#0-6) 
3. `unite_state_diffs` inserts `(C, K) → V'` into `result.storage_diffs`. No duplicate, no conflict. [8](#0-7) 
4. `parse_data_for_block` returns `Ok(Some((result, block_number)))` — no hash check performed.
5. `write_to_storage` calls `append_state_diff(N, result)` → `write_storage_diffs` writes `((C, K), N) → V'` to `contract_storage`. [9](#0-8) 
6. Any subsequent `get_storage_at(state_after_N, C, K)` returns V', not V, violating the commitment invariant.

fn write_storage_diffs<'env>(
