### Title
Cross-Chain Replay of `deploy_token` Signed Payload Due to Missing Chain-ID Binding — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` signed by the NEAR MPC for `deploy_token` / `deployToken` does not include any chain identifier or contract address. This is inconsistent with `finTransfer` / `fin_transfer` / `finalize_transfer`, which correctly bind the signed payload to a specific destination chain via `omniBridgeChainId`. Because the same NEAR MPC derived address signs for all supported chains, a valid `deploy_token` signature obtained on one chain (e.g., Ethereum mainnet) can be replayed verbatim on any other chain that shares the same MPC key (e.g., Arbitrum, Base, StarkNet, Solana), constituting an unauthorized token deployment without a fresh MPC signature for the target chain.

---

### Finding Description

**Root cause — EVM (`OmniBridge.deployToken`, L142–153):**

The borsh-encoded message that is hashed and verified against `nearBridgeDerivedAddress` is:

```
PayloadType.Metadata | token | name | symbol | decimals
```

No `omniBridgeChainId`, no contract address. [1](#0-0) 

**Root cause — StarkNet (`MetadataPayloadImpl::to_borsh`, L36–44):**

`deploy_token` calls `_verify_borsh_signature(ref self, @payload.to_borsh(), signature)`. The `to_borsh()` method encodes:

```
PayloadType::Metadata | token | name | symbol | decimals
```

No chain ID is passed; compare with `TransferMessagePayloadImpl::to_borsh(chain_id)` which explicitly encodes `chain_id` twice. [2](#0-1) [3](#0-2) 

**Root cause — Solana (`DeployTokenPayload::serialize_for_near`, L19–26):**

The signed serialization is:

```
IncomingMessageType::Metadata | token | name | symbol | decimals
```

No `SOLANA_OMNI_BRIDGE_CHAIN_ID`, no program ID; compare with `FinalizeTransferPayload::serialize_for_near` which writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` at both the token and recipient positions. [4](#0-3) [5](#0-4) 

**The inconsistency is explicit.** `finTransfer` on EVM encodes `omniBridgeChainId` at L294 and L297; `fin_transfer` on StarkNet passes `self.omni_bridge_chain_id.read()` into `to_borsh(chain_id)` at L253; `finalize_transfer` on Solana writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` at L30 and L35. `deploy_token` on all three chains omits this binding entirely. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The NEAR MPC uses a single derived address (`nearBridgeDerivedAddress` / `omni_bridge_derived_address` / `derived_near_bridge_address`) that is shared across all supported chains. A `deploy_token` signature produced by the MPC for chain A is byte-for-byte valid on chain B because the signed payload contains no chain-specific field. An attacker who observes a `deployToken` transaction on Ethereum can immediately submit the identical `signatureData` + `metadata` calldata to the Arbitrum (or Base, or any other EVM) bridge contract and have it accepted. The same applies cross-ecosystem: an EVM `deploy_token` signature is structurally identical to a StarkNet or Solana one (same Borsh prefix + same fields), so cross-ecosystem replay is also possible if the MPC key is shared.

Concrete consequences:
- **Unauthorized token deployment**: A bridge token is deployed on a target chain without the NEAR MPC ever signing a message for that chain, bypassing the intended authorization gate.
- **Mapping corruption / griefing**: Once the token is deployed via replay, `nearToEthToken[metadata.token]` / `near_to_starknet_token` is populated. Any subsequent legitimate deployment attempt for the same token on that chain reverts with `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`, permanently blocking the correct deployment flow for that token on that chain.

This matches the allowed High impact: "Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized transfer finalization, **token deployment**, or message execution." [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

- The NEAR MPC is explicitly designed to sign for all chains with the same key; this is the protocol's trust model.
- All `deployToken` transactions are public on-chain. An attacker needs only to copy the calldata from a confirmed transaction on chain A and submit it to chain B.
- No privileged access, no leaked keys, no colluding parties required — only a public mempool/explorer read and a standard transaction submission.
- The attack is especially realistic when the protocol expands to a new EVM chain: the first `deployToken` call on the existing chain immediately becomes a valid replay payload for the new chain.

---

### Recommendation

Include the destination chain identifier and, ideally, the bridge contract address in the signed `MetadataPayload`, mirroring the existing pattern used by `finTransfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain binding
    Borsh.encodeAddress(address(this)), // add contract binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**StarkNet:** Pass `self.omni_bridge_chain_id.read()` into `MetadataPayloadImpl::to_borsh()` and encode it, matching `TransferMessagePayloadImpl::to_borsh(chain_id)`.

**Solana:** Write `SOLANA_OMNI_BRIDGE_CHAIN_ID` (and optionally the program ID) into `DeployTokenPayload::serialize_for_near`, matching `FinalizeTransferPayload::serialize_for_near`.

The NEAR MPC signing logic must be updated in lockstep to include these fields when producing `deploy_token` signatures.

### Proof of Concept

1. Observe a confirmed `deployToken(signatureData, metadata)` transaction on Ethereum mainnet bridge (`OmniBridge` at address `X`).
2. Copy `signatureData` and `metadata` verbatim.
3. Submit `deployToken(signatureData, metadata)` to the Arbitrum mainnet bridge (`OmniBridge` at address `Y`, same `nearBridgeDerivedAddress`).
4. The EVM bridge at step 3 computes:
   ```
   keccak256(0x01 | borsh(token) | borsh(name) | borsh(symbol) | decimals)
   ```
   — identical to what was computed on Ethereum — and `ECDSA.recover` returns `nearBridgeDerivedAddress`. Signature check passes.
5. The token is deployed on Arbitrum and registered in `nearToEthToken` / `ethToNearToken`.
6. Any subsequent legitimate `deployToken` call for the same `metadata.token` on Arbitrum reverts with `"ERR_TOKEN_EXIST"`, permanently blocking the correct deployment. [10](#0-9) [11](#0-10) [2](#0-1) [12](#0-11) [4](#0-3)

        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata

            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)

            revert InvalidSignature();
        }

        );
