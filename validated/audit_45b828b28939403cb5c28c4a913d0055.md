Audit Report

## Title
Unguarded `refundETH` allows any caller to steal excess ETH left on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` transfers the router's entire native ETH balance to `msg.sender` with no access control. Because `multicall{value: X}` leaves unspent ETH on the router when a user omits the `refundETH` step, any third party can call `refundETH()` in a subsequent transaction and receive that ETH, causing direct loss of user principal.

## Finding Description
`refundETH()` reads `address(this).balance` and unconditionally calls `_transferETH(msg.sender, balance)` with no check that `msg.sender` was the original depositor. [1](#0-0) 

When `tokenIn == WETH` and the router holds native ETH, `pay()` wraps exactly `value` wei via `IWETH9.deposit{value: value}()` and transfers it to the pool. Any ETH above `value` remains on the contract as unwrapped native ETH. [2](#0-1) 

The `receive()` guard only reverts plain ETH transfers from non-WETH senders; it does not prevent ETH from being attached as `msg.value` to `multicall` or any other payable function call. [3](#0-2) 

`multicall` uses `delegatecall`, so `msg.value` is visible to every sub-call in the batch. After the batch completes, any unspent ETH remains on the router with no record of who deposited it. [4](#0-3) 

The test suite confirms the intended pattern — users must include `refundETH` as the final call in their multicall batch — but nothing in the contract enforces this. [5](#0-4) 

Exploit flow:
1. User calls `router.multicall{value: 1 ether}([exactInputSingle(..., amountIn: 0.5 ether, tokenIn: WETH)])` without appending `refundETH`.
2. `pay()` wraps 0.5 ETH → WETH and sends it to the pool. The remaining 0.5 ETH stays on the router.
3. Attacker calls `router.refundETH()`. `address(this).balance == 0.5 ETH > 0`, so `_transferETH(attacker, 0.5 ETH)` executes. The attacker receives the victim's excess ETH.

## Impact Explanation
Direct loss of user principal. The victim permanently loses the excess ETH they sent with their swap transaction. The attacker gains it with no privileged role required. This meets the Critical/High direct-loss-of-user-principal threshold under the contest rules.

## Likelihood Explanation
Any user who sends `msg.value > amountIn` without appending `refundETH` is silently vulnerable. This is a realistic mistake: users calling `exactInputSingle` or `exactOutputSingle` directly with a safety buffer, or constructing multicall batches programmatically and omitting the refund step. An attacker monitoring the mempool or the router's ETH balance can front-run or back-run the victim's transaction to drain the excess immediately.

## Recommendation
Record the top-level `msg.sender` in transient storage at the start of `multicall` and require `refundETH` to use that stored address rather than the immediate `msg.sender`. Alternatively, set a reentrancy-style flag in transient storage only during `multicall` execution and revert in `refundETH` if the flag is not set, ensuring `refundETH` can only be reached as a delegatecall sub-call within a multicall batch initiated by the original depositor.

## Proof of Concept
```solidity
// 1. User sends excess ETH without refundETH
bytes[] memory calls = new bytes[](1);
calls[0] = abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
    tokenIn: WETH, amountIn: 0.5 ether, ...
})));
router.multicall{value: 1 ether}(calls);
// pay() wraps 0.5 ETH → WETH. 0.5 ETH remains on router.

// 2. Attacker drains it
vm.prank(attacker);
router.refundETH();
// address(this).balance == 0.5 ETH → _transferETH(attacker, 0.5 ETH)
assertEq(attacker.balance, 0.5 ether); // attacker received victim's ETH
```
