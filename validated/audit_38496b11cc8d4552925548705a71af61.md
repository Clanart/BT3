### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Enforcement — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` contains a `nowrap: bool` parameter that selects between saturating and wrapping arithmetic for relative timelock checks. When `nowrap=false` (the legacy path, still reachable via the Python binding), an attacker-controlled near-maximum `height_relative` or `seconds_relative` value causes integer wraparound, making the computed deadline smaller than the current block height. The timelock check then passes unconditionally, allowing a coin to be spent immediately despite an intended multi-billion-block or multi-billion-second lock.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on `nowrap`:

```rust
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { ... }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { ... }
}
``` [1](#0-0) 

The same pattern applies to `seconds_relative`: [2](#0-1) 

When `nowrap=false`:

- `confirmed_block_index = 100`, `height_relative = 0xFFFF_FFFF` (u32::MAX)
- `100u32.wrapping_add(0xFFFF_FFFF) = 99`
- `prev_height < 99` → **false** for any `prev_height ≥ 99` → timelock silently passes

The intended semantics of `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` is "do not spend for ~4 billion more blocks." Wrapping collapses that to a deadline already in the past.

The same overflow applies to `ASSERT_SECONDS_RELATIVE` with `u64::MAX`:

- `coin_timestamp = 1000`, `seconds_relative = u64::MAX`
- `1000u64.wrapping_add(u64::MAX) = 999`
- `timestamp < 999` → false for any `timestamp ≥ 999` → timelock passes [2](#0-1) 

The repository's own unit tests document this divergence explicitly:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [3](#0-2) 

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),
)]
``` [4](#0-3) 

The Python binding `py_check_time_locks` forwards `nowrap` directly from the Python caller without any height-gating or default enforcement: [5](#0-4) 

This means any Python full-node code that calls `check_time_locks` with `nowrap=False` — whether for pre-fork blocks, during a transition window, or due to a misconfiguration — is exposed to the bypass. The `nowrap` decision is not encoded in `ConsensusFlags` and is not derived from block height inside `get_flags_for_height_and_constants`: [6](#0-5) 

The inverse overflow also exists for `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE`: wrapping causes a far-future deadline to collapse to a value already exceeded, incorrectly blocking spends that should be valid: [7](#0-6) 

---

### Impact Explanation

**High — Timelock condition validation bypass enables unauthorized spend acceptance.**

A coin whose puzzle emits `(ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF)` is intended to be unspendable for ~4 billion blocks. Under `nowrap=false`, the wrapping arithmetic makes the effective required height smaller than the coin's confirmation height, so the coin can be spent in the very next block. Any coin protected solely by a near-maximum relative timelock is immediately spendable by its key-holder without waiting, violating the contract encoded in the puzzle. If such coins hold value (e.g., vesting schedules, time-locked custody, DeFi primitives), the bypass allows premature extraction.

The inverse (wrapping `ASSERT_BEFORE_HEIGHT_RELATIVE`) can render coins permanently unspendable, constituting a coin-burn equivalent for any coin whose puzzle uses a large before-height guard.

---

### Likelihood Explanation

The `nowrap` parameter is caller-supplied from Python with no Rust-side enforcement of which value is correct for a given block height. Both `True` and `False` are tested and supported. Any full-node deployment that passes `nowrap=False` — including legacy nodes, nodes during a soft-fork transition window, or nodes that have not yet updated their Python call sites — is vulnerable. The attacker's entry path is fully unprivileged: craft a CLVM puzzle that outputs `(ASSERT_HEIGHT_RELATIVE <near-max-value>)`, fund the coin, and spend it immediately.

---

### Recommendation

1. **Remove the `nowrap=false` code path entirely** once the soft fork activating saturating arithmetic is finalized, or gate it inside Rust using a block-height threshold derived from `ConsensusConstants`, so the Python caller cannot accidentally select the wrong arithmetic mode.
2. **Add `nowrap` as a `ConsensusFlags` bit** and derive it from `get_flags_for_height_and_constants`, eliminating the caller-supplied boolean.
3. **Reject near-maximum relative timelock values at condition parse time** (in `parse_conditions` / `sanitize_uint`) if they would overflow under either arithmetic mode, making the condition immediately invalid rather than silently changing semantics.

---

### Proof of Concept

```
# Coin puzzle (CLVM): outputs (ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF)
# Coin confirmed at block height 100.
# Attacker submits spend at block height 101.

# With nowrap=False:
#   100u32.wrapping_add(0xFFFF_FFFF) = 99
#   101 < 99  →  False  →  no error  →  spend accepted

# With nowrap=True (correct):
#   100u32.saturating_add(0xFFFF_FFFF) = u32::MAX = 4_294_967_295
#   101 < 4_294_967_295  →  True  →  AssertHeightRelativeFailed
```

The same applies to `ASSERT_SECONDS_RELATIVE 0xFFFF_FFFF_FFFF_FFFF` with `u64` wrapping. Both cases are confirmed by the existing unit tests at lines 276–306 of `check_time_locks.rs`. [8](#0-7) [9](#0-8)

pub fn get_flags_for_height_and_constants(
    prev_tx_height: u32,
    constants: &ConsensusConstants,
) -> ConsensusFlags {
    //  the hard-fork initiated with 2.0. To activate June 2024
    //  * costs are ascribed to some unknown condition codes, to allow for
    // soft-forking in new conditions with cost
    //  * a new condition, SOFTFORK, is added which takes a first parameter to
    //    specify its cost. This allows soft-forks similar to the softfork
    //    operator
    //  * BLS operators introduced in the soft-fork (behind the softfork
    //    guard) are made available outside of the guard.
    //  * division with negative numbers are allowed, and round toward
    //    negative infinity
    //  * AGG_SIG_* conditions are allowed to have unknown additional
    //    arguments
    //  * Allow the block generator to be serialized with the improved clvm
    //   serialization format (with back-references)

    // The soft fork initiated with 2.5.0. The activation date is still TBD.
    // Adds a new keccak256 operator under the softfork guard with extension 1.
    // This operator can be hard forked in later, but is not included in a hard fork yet.

    // In hard fork 2, we enable the keccak operator outside the softfork guard
    let mut flags = ConsensusFlags::empty();
    if prev_tx_height >= constants.hard_fork2_height {
        flags |= ConsensusFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD
            | ConsensusFlags::COST_CONDITIONS
            | ConsensusFlags::ENABLE_SECP_OPS
            | ConsensusFlags::RELAXED_BLS;
    }

    if prev_tx_height >= constants.soft_fork8_height {
        flags |= ConsensusFlags::DISABLE_OP;
    }

    if prev_tx_height >= constants.soft_fork9_height {
        flags |= ConsensusFlags::SIMPLE_GENERATOR
            | ConsensusFlags::CANONICAL_INTS
            | ConsensusFlags::LIMIT_SPENDS;
    }
    flags
}
