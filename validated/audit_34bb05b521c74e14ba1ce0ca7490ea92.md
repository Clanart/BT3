### Title
Integer Overflow in `check_time_locks` `wrapping_add` Path Enables Relative Timelock Bypass — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary
`check_time_locks` contains two arithmetic paths for relative timelock evaluation, selected by the `nowrap: bool` parameter. When `nowrap=false`, `wrapping_add` is used instead of `saturating_add`. A crafted spend bundle with a large `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` condition value causes the addition to wrap around to a small integer, making the timelock check trivially pass and allowing the coin to be spent before the intended block height or timestamp.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` evaluates relative timelocks using two distinct arithmetic modes:

```rust
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { ... }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { ... }
}
``` [1](#0-0) 

The `ASSERT_HEIGHT_RELATIVE` check enforces: `prev_height < confirmed_height + height_relative` → reject (spend too early). With `nowrap=false`, if `confirmed_height + height_relative` overflows `u32`, the result wraps to a small value. For example:

- `confirmed_block_index = 10`, `height_relative = 0xFFFF_FFF6`
- `10u32.wrapping_add(0xFFFF_FFF6) = 0` (wraps to zero)
- `prev_height < 0` is always `false` → the check passes at **any** block height

The same overflow applies to `ASSERT_SECONDS_RELATIVE` using `u64::wrapping_add`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
``` [2](#0-1) 

The condition parser in `conditions.rs` accepts `height_relative` up to `u32::MAX` (4 bytes, `PositiveOverflow` is rejected but `0xFFFF_FFFF` is valid) and `seconds_relative` up to `u64::MAX` (8 bytes): [3](#0-2) 

The Python binding exposes `nowrap` as a caller-controlled parameter, meaning the Python full node determines which arithmetic mode is used: [4](#0-3) 

The test suite explicitly documents and confirms the bypass behavior for `nowrap=false`:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [5](#0-4) 

And in the Python test suite:

```python
# 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok
# saturating: clamps to u32::MAX, 15 < u32::MAX -> Err
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
``` [6](#0-5) 

### Impact Explanation

This is a **timelock validation bypass**. Any coin with `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` can be spent before the intended block height or timestamp when the full node calls `check_time_locks` with `nowrap=false`. This enables unauthorized spend acceptance — a coin that should be locked for thousands of blocks can be spent immediately by crafting a condition value that wraps the addition to zero or a small number. This matches the allowed High impact: *"timelock or coin-id validation bypass enables unauthorized spend acceptance."*

### Likelihood Explanation

The `nowrap` parameter is passed from the Python full node layer, which is outside this repository. If the full node uses `nowrap=False` for any block validation path (e.g., for pre-hard-fork blocks or as a legacy mode), the bypass is reachable by any unprivileged user who can submit a spend bundle. The condition value is fully attacker-controlled CLVM output. The maximum valid `height_relative` is `u32::MAX` and `seconds_relative` is `u64::MAX`, both of which are accepted by `sanitize_uint` and cause wrapping with any non-zero `confirmed_block_index` or `timestamp`.

### Recommendation

Remove the `nowrap=false` / `wrapping_add` code path entirely from `check_time_locks`. Always use `saturating_add` for relative timelock arithmetic. If backward compatibility with old blocks is required, document the exact block height at which `nowrap` transitions and ensure no attacker-controlled spend bundle can be validated under the wrapping mode after that height.

### Proof of Concept

A spend bundle containing `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFF6)` for a coin confirmed at block height `10`, evaluated with `nowrap=false`:

```
confirmed_block_index = 10
height_relative       = 0xFFFF_FFF6  (= 4294967286, valid u32, accepted by sanitize_uint)
10u32.wrapping_add(0xFFFF_FFF6)      = 0x1_0000_0000 mod 2^32 = 0

check: prev_height < 0  →  false at every block height
result: Ok(())  →  spend accepted, timelock bypassed
```

The same pattern applies to `ASSERT_SECONDS_RELATIVE` with `u64` arithmetic: a coin with `timestamp = T` and `seconds_relative = u64::MAX - T + 1` wraps to `0`, bypassing the seconds-relative timelock at any timestamp.
