### Title
EIP-7702 Authorization with Zero Chain ID Accepted Cross-Chain, Enabling Unauthorized EOA Code Delegation Replay - (File: `x/evm/keeper/set_code_authorizations.go`)

### Summary

The `validateAuthorization` function in Ethermint's EIP-7702 implementation explicitly permits authorizations signed with a zero chain ID (`auth.ChainID.IsZero()`). This is by design per the EIP-7702 spec, but it creates a direct analog to the external report's signature replay vulnerability: a valid EIP-7702 authorization signed with `chainID=0` carries no chain-binding, so the same signed authorization tuple can be replayed on any Ethermint-based chain that shares the same authority account nonce. An unprivileged attacker who obtains such a zero-chain-ID authorization (e.g., from a testnet, another Ethermint fork, or a user who signed one for portability) can submit it on any target chain to forcibly install arbitrary delegation code on the authority's EOA, mutating its code without the authority's consent on that specific chain.

### Finding Description

In `x/evm/keeper/set_code_authorizations.go`, `validateAuthorization` performs the following chain ID check:

```go
if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

When `auth.ChainID` is zero, the entire chain ID check is bypassed. The authorization is then validated only by:
1. Nonce overflow check (trivially satisfied for any normal nonce)
2. ECDSA signature recovery (`auth.Authority()`)
3. Authority account code check (no existing non-delegation code)
4. Nonce match against current state (`stateDB.GetNonce(authority) == auth.Nonce`)

The nonce check (step 4) is the only remaining guard. However, this is exactly the analog of the external report's vulnerability: the signed data (`chainID=0`, `address`, `nonce`) contains no chain-specific binding. If an authority account has the same nonce on two different Ethermint-based chains (which is common for fresh accounts or accounts with identical transaction histories), a zero-chain-ID authorization signed on one chain is valid and replayable on any other chain.

The entry path is fully unprivileged: any user can submit an EIP-7702 `SetCodeTx` (type 4) via `eth_sendRawTransaction` → `SendRawTransaction` in `rpc/backend/call_tx.go` → `broadcastTx` → ante handler → `ApplyTransaction` → `ApplyMessageWithConfig` → `applyAuthorization` → `validateAuthorization`. The outer transaction's chain ID is checked by the EIP-155 signer, but the inner authorization list's chain ID check is the one that is bypassed when zero.

The `setAuthorizationDelegation` function then unconditionally writes the delegation:

```go
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
```

This installs arbitrary code (`0xef0100 || auth.Address`) at the authority's EOA address, permanently changing its execution behavior.

### Impact Explanation

**High severity** — matches the allowed impact: "Ethereum transaction, EIP-7702 authorization, nonce, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation."

A victim EOA that signed a zero-chain-ID EIP-7702 authorization on any chain (e.g., a testnet, a different Ethermint fork, or even intentionally for multi-chain use) can have that authorization replayed on any other Ethermint chain where the account's nonce matches. The result is:
- The victim's EOA has its code forcibly set to a delegation pointing to an attacker-chosen contract address.
- All subsequent calls to the victim's EOA execute the delegated contract's code in the victim's context, giving the attacker full control over the victim's storage, balance, and execution.
- If the delegated contract is malicious (e.g., a drainer), the victim's entire EVM-denom balance can be stolen in the next transaction that calls the victim's address.

### Likelihood Explanation

Zero-chain-ID authorizations are explicitly permitted by the EIP-7702 specification as a "wildcard" for multi-chain use. Users and wallets that want to deploy the same delegation across multiple chains in a single signing operation will produce exactly these authorizations. Any such authorization observed on-chain (on any EVM chain) can be extracted and replayed on Ethermint chains where the authority's nonce matches. The attacker needs only:
1. A valid zero-chain-ID authorization (observable from any public chain or testnet).
2. Knowledge that the authority account has the matching nonce on the target chain.
3. Enough gas to submit the outer EIP-7702 transaction (no privileged access required).

This is a realistic, low-effort attack for any Ethermint-based chain that shares user accounts with other EVM chains.

### Recommendation

The zero-chain-ID bypass is intentional per EIP-7702 and cannot simply be removed. The mitigation is to ensure that zero-chain-ID authorizations are treated as a distinct risk class. Options:

1. **Governance parameter to reject zero-chain-ID authorizations**: Add an `AllowZeroChainIDAuthorizations` parameter (defaulting to `false`) to `x/evm/types/params.go` and check it in `validateAuthorization` before the `IsZero` bypass.
2. **Require non-zero chain ID in `ValidateEthBasic`**: In `ante/interfaces/setup.go`, add a check that all authorization list entries in a `SetCodeTx` have a non-zero chain ID matching the current chain, rejecting zero-chain-ID authorizations at the ante handler level.
3. **Document and warn**: At minimum, document that zero-chain-ID authorizations are accepted and that users should not sign them unless they accept cross-chain replay risk.

Option 2 is the most conservative and directly closes the replay path without breaking the EIP-7702 spec for single-chain use.

### Proof of Concept

1. On chain A (any EVM chain, e.g., a testnet), victim `V` signs an EIP-7702 authorization with `chainID=0`, `address=attacker_contract`, `nonce=N`. This produces `(v, r, s)`.

2. Attacker observes this authorization on-chain (or obtains it off-chain).

3. On target Ethermint chain B, victim `V` has nonce `N` (fresh account or matching history).

4. Attacker constructs an EIP-7702 `SetCodeTx` (type 4) with:
   - Outer tx signed with chain B's chain ID (passes EIP-155 check)
   - `authorizationList = [{chainID: 0, address: attacker_contract, nonce: N, v, r, s}]`
   - `to`: any address (e.g., attacker's own address)

5. Attacker submits via `eth_sendRawTransaction`.

6. In `validateAuthorization`:
   - `auth.ChainID.IsZero()` → `true` → chain ID check skipped
   - `auth.Authority()` → recovers `V`'s address (valid signature)
   - `stateDB.GetNonce(V) == N` → passes
   - Authorization applied: `V`'s code set to `0xef0100 || attacker_contract`

7. All future calls to `V`'s address execute `attacker_contract`'s code in `V`'s context. If `attacker_contract` contains a `SELFDESTRUCT` or balance-draining call, `V`'s funds are stolen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
