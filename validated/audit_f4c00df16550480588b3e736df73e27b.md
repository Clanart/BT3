### Title
DataLayer `ProofOfInclusion::valid()` Does Not Bind Computed Root to an External Trusted Root — Forged Inclusion Proofs Always Pass — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks the internal hash-chain consistency of the proof against itself. It derives the "root" it checks against from the proof's own last `combined_hash` field — not from any externally-trusted tree root. An attacker who can supply a serialized `ProofOfInclusion` (via the `Streamable`/`from_py_object` Python binding) can trivially forge a self-consistent proof that passes `valid()` for any key, without that key being present in the actual DataLayer tree.

### Finding Description

`ProofOfInclusion::valid()` is implemented as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The final check `existing_hash == self.root_hash()` is a **tautology**: after the loop, `existing_hash` holds the last `calculated_hash`, which was already verified to equal `last.combined_hash`. And `root_hash()` returns exactly `last.combined_hash`. So the final comparison is always `true` if the loop completes without returning `false`. The method never compares the computed root against any externally-trusted expected root.

`ProofOfInclusion` is `Streamable` and annotated `from_py_object`, making it fully deserializable from untrusted bytes via the Python binding: [3](#0-2) 

The Python binding exposes `valid()` directly: [4](#0-3) 

And `ProofOfInclusion` is exposed as a constructable, deserializable Python class: [5](#0-4) 

### Impact Explanation

An attacker can craft a `ProofOfInclusion` with arbitrary `node_hash`, `other_hash`, and `combined_hash` values that are internally consistent (trivially, since the attacker controls all fields) but prove inclusion of a key that is **not** in the actual DataLayer tree. The simplest exploit: supply a zero-layer proof with `node_hash = H(fake_key)` and `layers = []`. Then `root_hash()` returns `H(fake_key)`, the loop does not execute, and `existing_hash == self.root_hash()` is `H(fake_key) == H(fake_key)` = `true`. `valid()` returns `true` for a completely fabricated proof.

Any Python-layer DataLayer consumer that calls `proof.valid()` without also independently checking `proof.root_hash() == expected_tree_root` will accept forged inclusion proofs, allowing untrusted input to prove invalid state — matching the allowed High impact: **DataLayer Merkle proof logic lets untrusted input prove invalid state**.

### Likelihood Explanation

The `valid()` method name strongly implies a complete validity check. There is no documentation, assertion, or API-level guard requiring callers to also check `root_hash()` against an external trusted root. The `ProofOfInclusion` type is fully deserializable from bytes via the Python binding, making the attacker-controlled entry path direct and unprivileged. Any DataLayer client that receives a proof over the network and calls `.valid()` is vulnerable.

### Recommendation

`valid()` must accept an `expected_root: Hash` parameter and compare the computed root against it as the final step:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    // ... existing hash-chain loop ...
    existing_hash == *expected_root
}
```

Alternatively, add a separate `valid_for_root(&self, expected_root: &Hash) -> bool` method and deprecate the no-argument form. The Python binding must be updated accordingly. The `root_hash()` accessor can remain for informational use but must not be the sole anchor for proof validity.
