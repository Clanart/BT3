### Title
`check_is_reverted` Enforces No Constraint on Hint-Provided `is_reverted` Flag, Enabling Malicious Prover to Skip Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is supposed to validate that the hint-provided `is_reverted` flag is consistent with the actual execution state. However, the function body is entirely empty — it returns immediately without enforcing any constraint. This breaks the soundness of the OS proof, allowing a malicious prover (sequencer) to set `is_reverted = TRUE` for any invoke transaction, causing the execute step to be skipped while fees are still charged, resulting in direct loss of funds for users.

---

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`, lines 338–361), after the validation step succeeds, the `is_reverted` flag is loaded from a hint (`%{ IsReverted %}`) and then `check_is_reverted(is_reverted)` is called:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    // Align the stack with the `if` branch to avoid revoked references.
    ...
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

However, `check_is_reverted` in `execution_constraints.cairo` is completely empty:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

Since `is_reverted` is hint-provided and `check_is_reverted` enforces **no Cairo constraint**, the value is completely unconstrained in the proof. A malicious prover can set `is_reverted` to any non-zero value for any invoke transaction, causing the execute step to be skipped. Critically, `charge_fee` is called unconditionally **after** the `if-else` block, so fees are charged regardless of whether execution was skipped. [3](#0-2) 

This is the direct analog of the external report's `isRequestFailed()` bug: just as `isRequestFailed()` was supposed to accurately determine if a request failed but returned incorrect results (allowing `startDraw()` to be called when it shouldn't), `check_is_reverted()` is supposed to validate the `is_reverted` flag but does nothing — allowing the execute step to be bypassed when it should not be.

The contrast with `non_reverting_select_execute_entry_point_func` is instructive: that function explicitly asserts `is_reverted = 0` to enforce its invariant:

```cairo
assert is_reverted = 0;
``` [4](#0-3) 

No equivalent constraint exists for the `is_reverted` flag in `execute_invoke_function_transaction`.

---

### Impact Explanation

**Critical — Direct loss of funds.** A malicious sequencer can:

1. Include a user's invoke transaction in a block.
2. Set `is_reverted = TRUE` in the `%{ IsReverted %}` hint.
3. `check_is_reverted` does nothing — no proof constraint is violated.
4. The execute step is skipped entirely.
5. `charge_fee` is still called, charging the user up to `max_fee`.
6. The STARK proof is valid and accepted by the L1 verifier.
7. The user loses funds: fees are charged but the transaction is never executed.

The whole purpose of the StarkNet OS proof is to make the sequencer trustless — the proof is supposed to enforce that the sequencer cannot deviate from protocol rules. The empty `check_is_reverted` breaks this guarantee for every invoke transaction.

---

### Likelihood Explanation

The entry path is any user submitting an invoke transaction. Any sequencer that controls hint generation can exploit this without producing an invalid proof. In StarkNet's security model, the sequencer is explicitly **not** supposed to be a trusted party — the proof is the trust anchor. The missing constraint means the proof fails to fulfill this role for the `is_reverted` decision.

---

### Recommendation

Implement `check_is_reverted` to enforce that `is_reverted` is consistent with the actual execution state. At minimum, the function should assert that `is_reverted` is a valid boolean (0 or 1), and ideally should enforce that `is_reverted = 1` only when the transaction was legitimately reverted according to protocol-defined conditions (e.g., insufficient gas after validation). A minimal fix:

```cairo
func check_is_reverted(is_reverted: felt) {
    // Enforce that is_reverted is a boolean (0 or 1).
    assert is_reverted * (is_reverted - 1) = 0;
    return ();
}
```

A complete fix would additionally constrain that `is_reverted = 1` only when the protocol legitimately permits a revert at this stage.

---

### Proof of Concept

1. User submits an invoke transaction with `max_fee = X` (e.g., 1 ETH).
2. Sequencer includes the transaction in a block.
3. Validation step (`run_validate`) succeeds — nonce is incremented.
4. Sequencer sets `is_reverted = 1` in the `%{ IsReverted %}` hint.
5. `check_is_reverted(1)` returns immediately — no constraint added to the proof.
6. The `else` branch is taken — execute step is skipped.
7. `charge_fee` is called — user is charged up to `X`.
8. Proof is generated; the L1 verifier accepts it (no violated constraints).
9. On-chain state reflects: nonce incremented, fee deducted, transaction not executed.
10. User loses up to `X` with no recourse.
