Audit Report

## Title
Stranded ETH on Router Consumed by Subsequent WETH Swaps via `pay()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function in `PeripheryPayments.sol` uses `address(this).balance` — the router's entire native ETH balance — when settling WETH payments, rather than tracking only the ETH contributed by the current caller. Any ETH left on the router by a prior user (e.g., from an over-estimated `msg.value` without a `refundETH()` call) is silently consumed by the next caller whose `tokenIn` is WETH, resulting in direct loss of the prior user's ETH principal.

## Finding Description
In `PeripheryPayments.pay()` (L74–84), the WETH branch reads `address(this).balance` and uses it to deposit and transfer WETH on behalf of the current payer:

```solidity
uint256 nativeBalance = address(this).balance;   // entire router balance
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

This assumes all ETH on the router was placed there by the current transaction. That assumption is violated whenever a prior caller sent `msg.value` exceeding their `amountIn` and did not call `refundETH()`.

The `receive()` guard at L32–34 (`if (msg.sender != WETH) revert NotWETH()`) only blocks unsolicited direct ETH transfers; it does **not** prevent ETH from accumulating via `msg.value` in the payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, etc.).

The exploit path:
1. **User A** calls `exactInputSingle{value: 1 ether}(... amountIn: 0.5 ether ...)` with `tokenIn = WETH`. `pay()` deposits 0.5 ETH as WETH and sends it to the pool. The remaining 0.5 ETH stays on the router because no `refundETH()` was called.
2. **User B** calls `exactInputSingle{value: 0}(... amountIn: 0.5 ether ...)` with `tokenIn = WETH`. `pay()` sees `nativeBalance = 0.5 ether >= value = 0.5 ether`, deposits User A's ETH as WETH, and sends it to the pool. User B's WETH allowance is never touched.

`_justPayCallback` (L192–199) calls `pay()` with `_getPayer()` as the payer, but `pay()` never verifies that the ETH it uses actually came from that payer in the current call.

The same flaw applies to `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback` when token0 or token1 is WETH.

## Impact Explanation
Direct loss of user principal. User A's stranded ETH is consumed to settle User B's WETH payment obligation. User A loses the stranded ETH; User B receives swap output without spending any of their own funds. The loss is bounded only by the amount of ETH stranded, which can be arbitrarily large. This meets the Critical/High threshold for direct loss of user principal under Sherlock contest rules.

## Likelihood Explanation
Sending a round `msg.value` larger than the exact `amountIn` is a common UX pattern (wallets and integrators over-estimate to avoid reverts). Omitting `refundETH()` from a multicall or calling the router directly without a refund step is a realistic user mistake. The exploit requires no special privilege — any address can call `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0` to consume whatever ETH is currently on the router. The exploit is repeatable across blocks as long as ETH remains stranded.

## Recommendation
Track the ETH contributed by the current call in transient storage at the start of each payable entry point (e.g., store `msg.value` in a transient slot) and limit the WETH branch of `pay()` to that per-call budget rather than `address(this).balance`. Alternatively, enforce `address(this).balance == 0` at the start of every swap entry point (reverting if residue exists), or automatically refund surplus ETH at the end of every payable entry point.

## Proof of Concept
```solidity
// Step 1 — Strand ETH (User A, Transaction 1):
router.exactInputSingle{value: 1 ether}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(wethToken1Pool), tokenIn: address(WETH), tokenOut: address(token1),
        zeroForOne: true, amountIn: 0.5 ether, amountOutMinimum: 0,
        recipient: userA, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    })
);
// Router now holds 0.5 ETH belonging to User A (no refundETH called).

// Step 2 — Steal ETH (User B, Transaction 2):
router.exactInputSingle{value: 0}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(wethToken1Pool), tokenIn: address(WETH), tokenOut: address(token1),
        zeroForOne: true, amountIn: 0.5 ether, amountOutMinimum: 0,
        recipient: userB, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    })
);
// pay() sees nativeBalance = 0.5 ether >= value = 0.5 ether.
// Deposits User A's 0.5 ETH as WETH, sends to pool for User B's benefit.
// User B's WETH allowance is never touched. User A's 0.5 ETH is gone.
```
