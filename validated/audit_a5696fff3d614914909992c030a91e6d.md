### Title
Removed Custom Token Accepted in `initTransfer`, Permanently Locking User Funds — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol` exposes a `removeCustomToken()` admin function that deletes a token from all bridge mappings. However, `initTransfer()` performs no validation that the supplied `tokenAddress` is still a currently-registered token. After removal, `initTransfer()` silently falls through to the native-lock branch, locking user tokens in the bridge contract with no corresponding NEAR-side mapping and no recovery path.

---

### Finding Description

`removeCustomToken()` deletes four mappings for a token:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol lines 120-127
function removeCustomToken(address tokenAddress) external onlyRole(DEFAULT_ADMIN_ROLE) {
    delete isBridgeToken[tokenAddress];
    delete nearToEthToken[ethToNearToken[tokenAddress]];
    delete ethToNearToken[tokenAddress];
    delete customMinters[tokenAddress];
}
``` [1](#0-0) 

After this call, `initTransfer()` evaluates the removed token through its three-branch conditional:

```solidity
// lines 394-412
if (customMinters[tokenAddress] != address(0)) {          // false — deleted
    ...
} else if (isBridgeToken[tokenAddress]) {                  // false — deleted
    ...
} else {
    IERC20(tokenAddress).safeTransferFrom(                 // ← executes: tokens LOCKED
        msg.sender, address(this), amount
    );
}
``` [2](#0-1) 

The function contains **no check** that `ethToNearToken[tokenAddress]` is non-empty (i.e., that the token still has a valid NEAR-side mapping). The only guards are `whenNotPaused` and `fee >= amount`. [3](#0-2) 

The root cause mirrors the original report exactly: the validation checks the *current runtime state* of the token (is it a bridge token? is it a custom minter?) but does not verify that the token is still *registered and supported* by the protocol. Because `removeCustomToken()` only clears the EVM-side mappings, the token address is no longer zero-checked anywhere — it simply falls through to the lock branch.

There is no `rescue`, `sweep`, or emergency-withdrawal function in the contract for ERC-20 tokens, so any tokens locked this way are irrecoverable. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who calls `initTransfer()` with a removed custom token has their ERC-20 tokens transferred into the bridge contract (`safeTransferFrom` to `address(this)`). The `InitTransfer` event is emitted with a `tokenAddress` that has no `ethToNearToken` mapping. The NEAR bridge cannot finalize the transfer for an unregistered token, so the funds are permanently locked in the EVM bridge with no on-chain recovery mechanism.

A secondary accounting-corruption impact exists if the NEAR side still retains the token mapping at the time of the transfer: the EVM side now holds the tokens as a *locked* balance (lock/release model) while the NEAR side mints them (burn/mint model), breaking bridge collateralization — an unbacked NEAR-side supply is created.

---

### Likelihood Explanation

**Medium.** The precondition is an admin calling `removeCustomToken()`, which is a legitimate protocol operation (e.g., deprecating a compromised or delisted token). Once that happens:

- Any user unaware of the removal can call `initTransfer()` with the removed token.
- A race condition between the admin's removal transaction and a user's in-flight `initTransfer()` transaction is realistic on any congested block.
- No front-running or special knowledge is required; the user simply uses the token as they did before.

---

### Recommendation

Add a registration check at the top of `initTransfer()` for non-ETH, non-ERC1155 tokens:

```solidity
require(
    tokenAddress == address(0) ||
    multiTokens[tokenAddress].tokenAddress != address(0) ||
    customMinters[tokenAddress] != address(0) ||
    isBridgeToken[tokenAddress] ||
    bytes(ethToNearToken[tokenAddress]).length > 0,
    "ERR_TOKEN_NOT_REGISTERED"
);
```

This ensures that only tokens with a current, valid NEAR-side mapping (or explicitly supported token types) can be used to initiate a transfer, directly mirroring the recommendation in the original report to validate that a currency is still accepted before allowing the operation to proceed.

---

### Proof of Concept

1. Admin deploys a custom token `T` and registers it: `addCustomToken("near.token", T, minter, 18)`.
   - State: `isBridgeToken[T] = true`, `ethToNearToken[T] = "near.token"`, `customMinters[T] = minter`.
2. Admin decides to delist `T`: `removeCustomToken(T)`.
   - State: all four mappings cleared. NEAR bridge still has `"near.token"` registered (or has also removed it).
3. User (unaware of removal) calls `initTransfer(T, 1000e18, 0, 0, "alice.near", "")`.
   - `customMinters[T] == address(0)` → skip.
   - `isBridgeToken[T] == false` → skip.
   - `else` branch executes: `IERC20(T).safeTransferFrom(user, bridge, 1000e18)` — tokens locked.
   - `InitTransfer` event emitted with `tokenAddress = T`.
4. Relayer attempts `fin_transfer` on NEAR. NEAR bridge rejects (token not registered) or processes it (creating unbacked supply).
5. User's 1000e18 tokens are permanently locked in the EVM bridge. No rescue function exists. [1](#0-0) [5](#0-4)
