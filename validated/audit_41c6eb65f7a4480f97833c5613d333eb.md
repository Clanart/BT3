### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` updates a contract's `class_hash` in the state without verifying that the supplied `class_hash` is actually declared in the contract class tree. An unprivileged user can call `replace_class` with an arbitrary, undeclared class hash, permanently rendering their contract (including an account contract holding funds) unexecutable. This is the direct analog of the PoolFactory bug: a missing guard before an irreversible state mutation that permanently severs a critical resource link.

---

### Finding Description

In `execute_replace_class` (the new-syscall path), the OS accepts the caller-supplied `class_hash` and writes it directly into `contract_state_changes` with no check that a corresponding class exists in the class tree:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0)

After the state update is committed, the contract's `class_hash` field in the Patricia state tree permanently points to a hash for which no class body exists. Every subsequent call to that contract will fail at class-lookup time, with no recovery path.

The revert log does record the old class hash for rollback within the same transaction:

```cairo
assert [revert_log] = RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=state_entry.class_hash);
``` [3](#0-2) 

But once the transaction succeeds and the block is committed, the revert log is discarded and the corrupt state is final.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

An account contract that calls `replace_class` with an undeclared hash becomes permanently unexecutable: the OS cannot resolve the class body needed to run any entry point, including `__execute__` and `__validate__`. All assets held by or controlled through that account are irrecoverably frozen. There is no admin escape hatch in the OS to forcibly reset a contract's class hash after block finalization.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, publicly accessible syscall callable by any contract on itself. No privileged role is required. The trigger can be:

1. **Accidental**: A contract developer passes a class hash that has not yet been declared (e.g., off-by-one in a migration script), permanently bricking the contract.
2. **Deliberate self-harm**: A user intentionally freezes their own account to grief a protocol that depends on it (e.g., a multisig or escrow contract).
3. **Reentrancy-style confusion**: A contract that calls `replace_class` mid-execution with a hash it believes is declared, but which was never actually committed to the class tree.

The missing check is acknowledged by the development team itself (the `TODO` comment), confirming awareness that this guard is absent.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current block) or in the committed class tree root. Concretely, add a lookup analogous to:

```cairo
// Ensure the class is declared before allowing replace_class.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);  // or check the Patricia tree root
```

This mirrors the fix applied to the PoolFactory bug: add a guard that asserts the critical invariant (class is declared / ACL ownership is returned) before allowing the irreversible state mutation (class replacement / self-destruct).

---

### Proof of Concept

1. User deploys account contract `A` at address `0xABC` with a valid declared class `C1`.
2. User sends an invoke transaction from `A` that calls the `replace_class` syscall with `class_hash = 0xDEAD` (a hash never declared on-chain).
3. `execute_replace_class` in `syscall_impls.cairo` (line 906) writes `StateEntry(class_hash=0xDEAD, ...)` into `contract_state_changes` for address `0xABC` with no validation.
4. The transaction succeeds; the block is finalized; the Patricia state tree now records `class_hash=0xDEAD` for `0xABC`.
5. Any subsequent transaction targeting `0xABC` causes the OS to look up class `0xDEAD` in the class tree, find nothing, and fail at entry-point resolution.
6. All funds held by `0xABC` are permanently frozen with no recovery mechanism.
