Audit Report

## Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale-Price Liquidity Addition — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

All four public entry points of `MetricOmmPoolLiquidityAdder` — two overloads each of `addLiquidityExactShares` and `addLiquidityWeighted` — accept no `deadline` parameter and perform no time-based expiry check. A pending transaction can sit in the mempool and execute arbitrarily late, placing LP shares into bins priced by an oracle that has moved materially from the user's intent. `MetricOmmSimpleRouter` already establishes the correct pattern with `_checkDeadline` on every swap entry point, but the liquidity adder has no equivalent guard.

## Finding Description

`MetricOmmSimpleRouter` calls `_checkDeadline(params.deadline)` at the top of every user-facing swap function: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

None of the four entry points in `MetricOmmPoolLiquidityAdder` have any equivalent check: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The `addLiquidityWeighted` overloads call `_validateBinAndBinPosition`, which reads the pool's current cursor from `slot0` and reverts if it falls outside caller-supplied bounds: [9](#0-8) 

This guard is insufficient for two reasons. First, if the oracle price moves but the cursor remains within the user's stated range, the check passes and execution proceeds at the new price. Second, `addLiquidityExactShares` has no cursor-bounds check at all — the only protection is the `maxAmountToken0`/`maxAmountToken1` caps enforced in the callback: [10](#0-9) 

Those caps limit how much is spent, not at what oracle price the shares are minted.

## Impact Explanation

Metric OMM is oracle-anchored: bin prices are set by the external oracle at execution time, not by reserve ratios. A user who submits a liquidity transaction at oracle price P may have it mined at price P′ after an arbitrary delay. Their shares are minted into bins priced at P′, producing immediate impermanent loss proportional to |P′ − P|. The `maxAmount` caps do not protect against this: they bound token spend, not price exposure. This constitutes a direct loss of user LP principal with no on-chain recourse, meeting the Medium/High threshold for direct loss of user assets.

## Likelihood Explanation

Any period of mempool congestion, gas-price spike, or deliberate MEV withholding can delay a liquidity transaction. No special privilege is required — any unprivileged LP calling either `addLiquidityExactShares` or `addLiquidityWeighted` is exposed. The attack is repeatable and requires only that an attacker observe a pending transaction and wait for a favorable oracle move before including it.

## Recommendation

Add a `uint256 deadline` parameter to all four `addLiquidity*` entry points and call the same `_checkDeadline` helper used by `MetricOmmSimpleRouter` at the top of each function, before any state reads or pool calls. This mirrors the pattern already established in `MetricOmmSimpleRouter` and ensures transactions cannot execute after the user's intended expiry.

## Proof of Concept

1. Alice calls `addLiquidityWeighted` with `minimalCurBin = -5`, `maximalCurBin = 5`, `maxAmountToken0 = 1000 USDC`, `maxAmountToken1 = 1 ETH` when the ETH oracle price is $2,000.
2. The transaction is not mined immediately due to low gas or mempool congestion.
3. The ETH oracle price rises to $2,500 within the same block range.
4. The transaction is mined. `_validateBinAndBinPosition` passes because the cursor is still within `[-5, 5]`.
5. The probe executes at $2,500; `_scaleWeightsToShares` computes shares based on the new price.
6. Alice's liquidity is added at $2,500 bins. She immediately holds a position worth less in ETH terms than she intended, with no ability to have prevented the execution.

A Foundry fork test can reproduce this by: (a) submitting the `addLiquidityWeighted` call, (b) advancing the oracle price via the oracle admin, and (c) executing the call in the same test — confirming that `_validateBinAndBinPosition` does not revert and shares are minted at the new price.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
