### Title
Unverified Hint-Supplied Block Hash Written to Storage Without Chain Proof — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/os_utils.cairo`)

---

### Summary

`write_block_number_to_block_hash_mapping` in `os_utils.cairo` writes a historical block hash to the block hash contract's storage using a value supplied entirely by a prover-controlled hint (`GetOldBlockNumberAndHash`), with **no cryptographic verification** that the supplied hash is the legitimate hash for that block number. The OS itself acknowledges this with an explicit TODO comment. Because the STARK proof is supposed to be the trust anchor for safety — meaning even a malicious prover should not be able to forge state — this missing enforcement is a protocol-level flaw. Any contract calling the `get_block_hash` syscall will receive the forged value, enabling fund theft in contracts that rely on historical block hashes for security decisions.

---

### Finding Description

In `write_block_number_to_block_hash_mapping` (`os_utils.cairo`, lines 48–85), the OS computes `old_block_number = current_block_number - STORED_BLOCK_HASH_BUFFER` and then fetches the corresponding hash via a hint:

```cairo
// Currently, the block hash mapping is not enforced by the OS.
// TODO(Yoni, 1/1/2026): output this hash.
local old_block_hash;
%{ GetOldBlockNumberAndHash %}
``` [1](#0-0) 

The hint-supplied `old_block_hash` is then written directly into the block hash contract's storage dict:

```cairo
assert [storage_ptr] = DictAccess(key=old_block_number, prev_value=0, new_value=old_block_hash);
``` [2](#0-1) 

There is **no Cairo-level assertion, hash chain proof, or commitment** linking `old_block_hash` to any previously verified block. The prover can supply any arbitrary felt value and the STARK proof will still be valid, because the OS does not constrain this value.

This storage entry is exactly what `read_block_hash_from_storage` reads when `execute_get_block_hash` is called by a contract:

```cairo
assert [storage_ptr] = DictAccess(
    key=block_number, prev_value=expected_block_hash, new_value=expected_block_hash
);
``` [3](#0-2) 

The `execute_get_block_hash` syscall handler reads `response.block_hash` from the prover-supplied syscall segment and validates it against the storage entry:

```cairo
let block_hash = response.block_hash;
read_block_hash_from_storage(block_number=request_block_number, expected_block_hash=block_hash);
``` [4](#0-3) 

Since both the storage write (via `GetOldBlockNumberAndHash`) and the syscall response (via the syscall segment) are prover-controlled, the prover can make them consistent with each other while both being forged. The dict squashing will succeed because `prev_value=0` (first write) and `prev_value=expected_block_hash` (read) are consistent within the same block's execution.

A second, compounding gap exists in `get_block_hashes` (`block_hash.cairo`, lines 55–82): `previous_block_hash`, `header_commitments`, `gas_prices_hash`, and `starknet_version` are all hint-supplied with no OS-level verification:

```cairo
local previous_block_hash;
// Currently, the header commitments and gas prices are not computed by the OS.
// TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
local header_commitments: BlockHeaderCommitments*;
local gas_prices_hash;
local starknet_version;
%{ GetBlockHashes %}
``` [5](#0-4) 

The `OsOutputHeader` comment in `output.cairo` also acknowledges the `new_block_hash` field is not enforced:

```cairo
// Currently, the block hash is not enforced by the OS.
new_block_hash: felt,
``` [6](#0-5) 

And `get_block_os_output_header` explicitly states:

```cairo
// NOTE: both the previous block hash and previous state root are guessed, and the OS
// does not verify their consistency (unlike the new hash and root).
// The consumer of the OS output should verify both.
``` [7](#0-6) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `get_block_hash` syscall is a first-class StarkNet protocol primitive. Contracts that use it for security-critical decisions — such as cross-chain bridges verifying L1 block inclusion, proof-verification contracts, or commit-reveal schemes — will receive a forged hash that passes all OS-level consistency checks. A malicious prover can craft a forged `old_block_hash` that makes a target contract believe a specific historical block contained a specific state (e.g., a deposit, a commitment, or a proof), causing the contract to release funds to the attacker. Because the STARK proof verifies correctly (the OS does not constrain the hash), the L1 verifier accepts the proof, and the forged state becomes canonical on-chain.

---

### Likelihood Explanation

The vulnerability is exploitable by whoever controls the prover/sequencer role. In StarkNet's current architecture this is Starkware (centralized), but the protocol's stated roadmap is decentralization. The TODO deadline of `1/1/2026` in the source code confirms this is a known, unresolved gap. Any contract deployed today that uses `get_block_hash` for fund-gating logic is at risk the moment the sequencer role becomes adversarial or is decentralized without this fix in place. The attack requires no special transaction, no leaked key, and no external dependency — only the ability to provide a hint value, which is inherent to the prover role.

---

### Recommendation

Require a cryptographic hash chain proof inside `write_block_number_to_block_hash_mapping` that links `old_block_hash` to the current block's verified state root or block hash, analogous to the fix recommended in the external report. Concretely:

1. **Output `old_block_hash` in the OS output** (as the TODO at line 65 of `os_utils.cairo` indicates) so the L1 verifier can check it against the on-chain block hash registry.
2. **Enforce `previous_block_hash` inside the OS** in `get_block_hashes` (`block_hash.cairo`) by linking it to the previously verified block's output rather than accepting it from an unconstrained hint.
3. Until both are done, treat any contract using `get_block_hash` for fund-gating as operating under an unverified assumption.

---

### Proof of Concept

1. Attacker deploys a contract `Victim` on StarkNet that holds funds and releases them when `get_block_hash(N)` returns a specific value `H_forged` (e.g., a commit-reveal scheme or bridge verification).
2. Attacker (as malicious prover) constructs a block where `write_block_number_to_block_hash_mapping` is called with `old_block_number = N` and supplies `old_block_hash = H_forged` via the `GetOldBlockNumberAndHash` hint.
3. In the same block, attacker includes a transaction calling `Victim` which internally calls `get_block_hash(N)`.
4. `execute_get_block_hash` reads `response.block_hash = H_forged` from the prover-supplied syscall segment and calls `read_block_hash_from_storage(N, H_forged)`.
5. `read_block_hash_from_storage` asserts `DictAccess(key=N, prev_value=H_forged, new_value=H_forged)` — consistent with the write in step 2.
6. Dict squashing succeeds: write entry has `prev_value=0, new_value=H_forged`; read entry has `prev_value=H_forged, new_value=H_forged`. No contradiction.
7. The STARK proof is generated and accepted by the L1 verifier. `Victim` releases funds to the attacker. [8](#0-7) [9](#0-8) [10](#0-9)

// Writes the hash of the (current_block_number - buffer) block under its block number in the
// dedicated contract state, where buffer=STORED_BLOCK_HASH_BUFFER.
func write_block_number_to_block_hash_mapping{range_check_ptr, contract_state_changes: DictAccess*}(
    block_context: BlockContext*
) {
    alloc_locals;
    tempvar old_block_number = block_context.block_info_for_execute.block_number -
        STORED_BLOCK_HASH_BUFFER;
    let is_old_block_number_non_negative = is_nn(old_block_number);
    if (is_old_block_number_non_negative == FALSE) {
        // Not enough blocks in the system - nothing to write.
        return ();
    }

    // Fetch the (block number -> block hash) mapping contract state.
    local state_entry: StateEntry*;
    %{ GetBlockHashMapping %}

    // Currently, the block hash mapping is not enforced by the OS.
    // TODO(Yoni, 1/1/2026): output this hash.
    local old_block_hash;
    %{ GetOldBlockNumberAndHash %}

    // Update mapping.
    assert state_entry.class_hash = 0;
    assert state_entry.nonce = 0;
    tempvar storage_ptr = state_entry.storage_ptr;
    assert [storage_ptr] = DictAccess(key=old_block_number, prev_value=0, new_value=old_block_hash);
    let storage_ptr = storage_ptr + DictAccess.SIZE;
    %{ WriteOldBlockToStorage %}

    // Update contract state.
    tempvar new_state_entry = new StateEntry(class_hash=0, storage_ptr=storage_ptr, nonce=0);
    dict_update{dict_ptr=contract_state_changes}(
        key=BLOCK_HASH_CONTRACT_ADDRESS,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
}

// Verifies that the block hash for the given block number equals the expected block hash,
// by adding a read entry to the contract_state_changes dict with it.
func read_block_hash_from_storage{contract_state_changes: DictAccess*}(
    block_number: felt, expected_block_hash: felt
) {
    // Fetch the block hash contract state.
    tempvar state_entry: StateEntry*;
    // Fetch a state_entry in this hint. Validate it in the update that comes next.
    %{ GetBlockHashMapping %}

    // Read from storage.
    tempvar storage_ptr = state_entry.storage_ptr;
    assert [storage_ptr] = DictAccess(
        key=block_number, prev_value=expected_block_hash, new_value=expected_block_hash
    );
    let storage_ptr = storage_ptr + DictAccess.SIZE;

    // Update the state.
    dict_update{dict_ptr=contract_state_changes}(
        key=BLOCK_HASH_CONTRACT_ADDRESS,
        prev_value=cast(state_entry, felt),
        new_value=cast(
            new StateEntry(
                class_hash=state_entry.class_hash, storage_ptr=storage_ptr, nonce=state_entry.nonce
            ),
            felt,
        ),
    );

    return ();
}
