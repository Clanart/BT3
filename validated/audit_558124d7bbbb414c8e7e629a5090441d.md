Audit Report

## Title
Stale Router ETH Consumed by Subsequent WETH Swap via Hybrid `pay()` Path, Causing Direct Loss of Prior Depositor's ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` hybrid WETH branch in `PeripheryPayments.sol` unconditionally consumes the router's entire native ETH balance to partially fund any WETH payment, without verifying that the ETH belongs to the current payer. Residual ETH from a prior over-funded WETH swap is silently consumed to subsidize the next WETH swap caller, causing direct loss of the prior depositor's ETH principal.

## Finding Description
`PeripheryPayments.pay()` (L73-84) checks `address(this).balance` globally. When `0 < nativeBalance < value` and `token == WETH`, the hybrid branch at L78-81 wraps and forwards the router's full ETH balance to the pool, then pulls only the shortfall from `payer` via `safeTransferFrom`. There is no check that the native balance was deposited by the current `payer` or in the current transaction.

ETH accumulates in the router because `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` are all `payable`. When a user sends `msg.value > amountIn` (e.g., 2 ETH for a 1 WETH swap), the callback wraps exactly `amountIn` and the remainder stays in the router. The `receive()` guard (L32-34) only blocks direct ETH transfers via `call{value}("")`; it does not prevent ETH from arriving as `msg.value` in a `payable` function call.

Exploit flow:
1. User A calls `exactInputSingle{value: 2 ETH}(amountIn=1 ETH, tokenIn=WETH)`. The callback (`_justPayCallback`, L192-199) calls `pay(WETH, UserA, pool, 1 ETH)`. `nativeBalance=2 ETH >= value=1 ETH`, so the router wraps 1 ETH and sends it to the pool. 1 ETH remains in the router. User A omits `refundETH()`.
2. User B calls `exactInputSingle(amountIn=1.5 WETH, tokenIn=WETH)` with no ETH. The callback calls `pay(WETH, UserB, pool, 1.5 ETH)`. `nativeBalance=1 ETH > 0` and `1 ETH < 1.5 ETH`, so the hybrid branch fires: router wraps 1 ETH → sends to pool, then pulls only 0.5 WETH from User B.
3. User A loses 1 ETH. User B pays 1 WETH less than owed. The pool receives the correct 1.5 WETH total — no revert, no pool insolvency, theft is silent. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

## Impact Explanation
Direct loss of ETH principal for any user who leaves ETH in the router without immediately reclaiming it via `refundETH()`. The victim's transaction already succeeded; the theft occurs silently in a later, unrelated transaction by any other user performing a WETH swap. This meets the "Critical/High direct loss of user principal" impact gate. Severity: **High**.

## Likelihood Explanation
Realistic and repeatable. The intended usage pattern (multicall with `refundETH()` as the last step) is documented but not enforced. A bare `exactInputSingle{value: X}(amountIn < X)` call without multicall, a dropped `refundETH()` step, or a revert in a later multicall step all strand ETH. Any subsequent WETH swap by any user with `amountIn > router.balance` automatically triggers the theft. No privileged access or special setup is required. [5](#0-4) 

## Recommendation
In the hybrid WETH branch, restrict use of the router's native ETH balance to cases where `payer == address(this)` (i.e., mid-path hops where the router is paying from its own balance). For external payers, either:
- Remove the hybrid path entirely and require callers to pre-wrap ETH themselves, or
- Only consume `address(this).balance` when `payer == address(this)`, or
- Track per-transaction ETH deposits via transient storage at entry and limit the hybrid path to that recorded amount only. [6](#0-5)
