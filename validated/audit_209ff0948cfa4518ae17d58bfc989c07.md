### Title
`batch_insert` Fast-Path Skips Duplicate-Key Guard, Corrupting Merkle Tree Cache and Enabling False Proof of Inclusion — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`batch_insert` has two code paths. When the tree already has `leaf_count >= 2`, **all** entries bypass the duplicate-key guard entirely. When `leaf_count <= 1`, only the last two entries (popped from the vector) go through `insert()` with its guard; every earlier entry in the vector also bypasses it. The fast path calls `insert_entry_to_blob` → `add_leaf`, which calls `HashMap::insert` and silently overwrites the existing `key_to_index` entry for a duplicate key. The result is two live leaf nodes for the same `KeyId` in the blob, with the cache pointing only to the later one.

---

### Finding Description

**Guard in `insert()` (protected path):** [1](#0-0) 

`insert()` calls `block_status_cache.contains_key(key)` and returns `Err(KeyAlreadyPresent)` on collision.

**`batch_insert` fast path — no guard:** [2](#0-1) 

When `leaf_count <= 1`, only the two entries popped from the **end** of the vector go through `insert()`. All remaining entries (the front of the vector) go directly to `insert_entry_to_blob`. When `leaf_count >= 2`, the entire `insert()` branch is skipped and **every** entry in the batch goes through the unguarded path.

**`add_leaf` silently overwrites:** [3](#0-2) 

`HashMap::insert` returns the displaced value but `add_leaf` discards it. A second leaf with the same `KeyId` silently replaces the cache entry, leaving the first leaf reachable in the tree but invisible to the cache.

**Concrete trigger (tree already has 2+ leaves):**

```
blob.batch_insert(vec![
    ((KeyId(1), ValueId(10)), hash_a),
    ((KeyId(1), ValueId(20)), hash_b),   // duplicate
])
```

Both leaves are written to the blob and wired into the tree structure. `key_to_index[KeyId(1)]` ends up pointing to the second leaf (v=20). The first leaf (v=10) is still reachable from the root but the cache disagrees about its index.

---

### Impact Explanation

`get_proof_of_inclusion(key1)` resolves the key through the cache to the second leaf and builds a valid Merkle path to the root: [4](#0-3) 

The returned proof is structurally valid (the path from the second leaf to the root is correct), but the root hash was computed over a tree that contains **two** leaves for the same key. Any verifier checking the proof against the committed root hash will accept it, yet the proof attests to a value that was never the sole canonical value for that key. This satisfies the "DataLayer Merkle proof/blob/delta logic … lets untrusted input prove invalid state" criterion.

---

### Likelihood Explanation

`py_batch_insert` (the Python binding) performs no duplicate-key pre-validation: [5](#0-4) 

It only checks that `keys_values.len() == hashes.len()`. Any caller — including higher-level DataLayer code that assembles a batch from user-submitted records — that passes a list containing a repeated `KeyId` will silently corrupt the tree. The bug is triggered by ordinary API misuse, not by memory corruption or privileged access.

---

### Recommendation

Add a duplicate-key check at the top of the fast-path loop in `batch_insert`, mirroring the guard in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Alternatively, pre-deduplicate or sort-and-check the input vector before entering the loop. The `add_leaf` function itself could also be hardened to return an error if `HashMap::insert` returns `Some(...)`, making the invariant impossible to violate silently from any call site.

        &self,
        key: KeyId,
    ) -> Result<proof_of_inclusion::ProofOfInclusion, Error> {

            .block_status_cache
            .get_index_by_key(key)
            .ok_or(Error::UnknownKey(key))?;

        let node = self
            .get_node(index)?
            .expect_leaf("key to index mapping should only have leaves");

        let parents = self.get_lineage_blocks_with_indexes(index)?;
        let mut layers: Vec<proof_of_inclusion::ProofOfInclusionLayer> = Vec::new();
        let mut parents_iter = parents.iter();
        // first in the lineage is the index itself, second is the first parent
        parents_iter.next();
        for (next_index, block) in parents_iter {
            if block.metadata.dirty {
                return Err(Error::Dirty(*next_index));
            }
            let parent = block
                .node
                .expect_internal("all nodes after the first should be internal");
            let sibling_index = parent.sibling_index(index)?;
            let sibling_block = self.get_block(sibling_index)?;
            let sibling = sibling_block.node;
            let layer = proof_of_inclusion::ProofOfInclusionLayer {
                other_hash_side: parent.get_sibling_side(index)?,
                other_hash: sibling.hash(),
                combined_hash: parent.hash,
            };
            layers.push(layer);
            index = *next_index;
        }

        Ok(proof_of_inclusion::ProofOfInclusion {
            node_hash: node.hash,
            layers,
        })

    ) -> PyResult<()> {
