### Title
Permissionless `log_metadata` + missing token whitelist in `init_transfer` enables unbacked mint on NEAR — (`starknet/src/omni_bridge.cairo`)

### Summary
An unprivileged attacker can deploy a malicious ERC20 on StarkNet that returns `true` from `transfer_from` without moving any balance, register it via the permissionless `log_metadata` flow, and call `init_transfer` to emit a valid `InitTransfer` event with no real token lock. NEAR's `fin_transfer_callback` then mints tokens backed by nothing.

### Finding Description
`OmniBridge::init_transfer` accepts any `token_address`. For non-bridge tokens it dispatches `IERC20Dispatcher::transfer_from` and asserts the boolean return value. It does not verify that `token_address` is in any registry (`starknet_to_near_token`, `near_to_starknet_token`, or any allowlist). A contract that returns `true` from `transfer_from` with no state change satisfies this check.

`log_metadata` is intentionally permissionless and emits a `LogMetadata` event from the bridge contract (the registered factory). NEAR's `deploy_token` (also open to anyone) accepts a proof of this event, verifies only the emitter address, and calls `deploy_token_internal` → `add_token`, which inserts the malicious StarkNet address into `token_decimals`, `token_address_to_id`, and `token_id_to_address`.

Once registered, a proof of the fraudulent `InitTransfer` event submitted to NEAR's `fin_transfer` passes all checks in `fin_transfer_callback`: factory check, `token_decimals` lookup, and `get_token_id`. Because the token's origin chain is StarkNet and the transfer also originates from StarkNet, `unlock_tokens_if_needed` returns `Unchanged` (no collateral required). The bridge mints the full amount to the recipient.

### Impact Explanation
Unbacked token supply is created on NEAR (or any destination chain reachable via the second leg). The attacker can repeat this for arbitrary amounts, inflating supply of the bridged token without any corresponding lock or burn on StarkNet. This breaks bridge collateralization.

### Likelihood Explanation
All steps except the final `fin_transfer` call are fully permissionless. `fin_transfer` requires a `#[trusted_relayer]`, but production relayers are automated and process every valid event emitted by the registered bridge contract. The `InitTransfer` event is cryptographically valid (it was genuinely emitted by the bridge), so a relayer has no basis to reject it. The attack is repeatable and requires only the ability to deploy a contract on StarkNet.

### Recommendation
1. In `init_transfer`, require that `token_address` is either a bridge token (`is_bridge_token`) **or** present in `starknet_to_near_token` (i.e., registered via `deploy_token`). Reject unknown token addresses.
2. Alternatively, maintain an explicit allowlist of accepted native tokens and enforce it in `init_transfer`.
3. On the NEAR side, `fin_transfer_callback` should additionally verify that the `(origin_chain, token_address)` pair has a non-zero locked balance or is a deployed token with a valid mint authority, before minting.

            let mut res = syscalls::call_contract_syscall(
                token, selector!("name"), call_data.span(),
            )
                .unwrap_syscall();

            let name = if res.len() == 1 {
                // Old standard (felt252)
                let name = OptionTrait::expect(
                    Serde::<felt252>::deserialize(ref res), 'Could not deserialize name',
                );
                utils::felt252_to_string(name)
            } else {
                // New standard (ByteArray)
                OptionTrait::expect(
                    Serde::<ByteArray>::deserialize(ref res), 'Could not deserialize name',
                )
            };

                token, selector!("symbol"), call_data.span(),
