Audit Report

## Title
`pay()` consumes entire contract ETH balance instead of per-call `msg.value`, enabling theft of stranded ETH via WETH-input swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` when handling WETH-input payments, consuming the router's entire native ETH balance rather than only the ETH forwarded by the current caller. Because all swap entry-points are `payable` and ETH can accumulate between transactions, any subsequent WETH-input swap silently consumes a prior user's stranded ETH, causing direct loss of that user's funds with zero cost to the attacker.

## Finding Description
`pay()` at [1](#0-0)  reads `address(this).balance` and, if it covers the required `value`, wraps and forwards that ETH without any check that it belongs to the current caller.

The `receive()` guard at [2](#0-1)  only blocks direct ETH sends from non-WETH addresses. It does **not** prevent ETH from entering via the `payable` swap entry-points: `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` are all declared `payable` at [3](#0-2)  and [4](#0-3) .

The callback path `metricOmmSwapCallback` → `_justPayCallback` at [5](#0-4)  calls `pay()` with only the payer address and required amount — `msg.value` is never threaded through, so `pay()` has no way to distinguish ETH belonging to the current call from ETH left by prior callers.

Exploit flow:
1. Alice calls `exactInputSingle{value: 2 ether}` with `tokenIn = WETH`, swap costs 1 ether, and omits `refundETH()`. Router now holds 1 ether of Alice's ETH.
2. Bob calls `exactInputSingle{value: 0}` with `tokenIn = WETH`, swap costs 1 ether. Bob has approved the router for 1 WETH but sends no ETH.
3. Inside `_justPayCallback` → `pay(WETH, Bob, pool, 1e18)`: `address(this).balance == 1 ether` (Alice's), branch `nativeBalance >= value` is taken, Alice's ETH is wrapped and sent to the pool, Bob's WETH allowance is never touched.
4. Bob receives full swap output; Alice's 1 ETH is permanently lost.

## Impact Explanation
Direct loss of user principal. The victim loses stranded ETH to an unprivileged attacker who pays nothing for a full swap. The pool's accounting remains internally consistent (it receives the correct WETH), so the loss is entirely borne by the victim. This meets the Critical/High direct loss of user principal impact gate.

## Likelihood Explanation
The standard usage pattern is `multicall([exactInputSingle, refundETH])`. A dropped or mis-ordered `refundETH` call, or any `msg.value` slightly above the actual swap cost (e.g., to handle price movement), leaves ETH stranded. An attacker can monitor the router's on-chain ETH balance and immediately follow any such transaction with a zero-value WETH-input swap. No special role or permission is required.

## Recommendation
Pass `msg.value` (or a per-call snapshot) into `pay()` as an `availableNative` parameter rather than reading `address(this).balance`. Decrement it as ETH is consumed within a single call. Alternatively, store `msg.value` in a transient storage slot at each entry-point and zero it after use, so `pay()` can never spend more ETH than the current caller provided. The `refundETH()` function at [6](#0-5)  already demonstrates the correct pattern of returning the full balance to `msg.sender`, but it is optional and caller-initiated, making it an insufficient guard.

## Proof of Concept
1. Deploy router with WETH address.
2. Alice calls `exactInputSingle{value: 2 ether}(params)` where `params.tokenIn = WETH`, `params.amountIn = 1 ether`. She omits `refundETH()`. `address(router).balance == 1 ether`.
3. Bob calls `exactInputSingle{value: 0}(params)` where `params.tokenIn = WETH`, `params.amountIn = 1 ether`. Bob has approved router for 1 WETH.
4. Assert: Bob receives full swap output; `address(router).balance == 0`; Bob's WETH balance is unchanged; Alice's 1 ETH is gone.
