Audit Report

## Title
ETH Sent to Payable Swap/Liquidity Functions With Non-WETH `tokenIn` Is Silently Trapped and Stealable — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
When a caller attaches `msg.value > 0` to any `payable` swap or liquidity entry-point while `tokenIn` is not WETH, `PeripheryPayments.pay()` falls into its ERC-20 `safeTransferFrom` branch and never reads `address(this).balance`, leaving the ETH stranded in the router. The public, permissionless `refundETH()` function then allows any third party to drain the full contract ETH balance, resulting in direct loss of user funds.

## Finding Description
`PeripheryPayments.pay()` (L69–88) has three branches. When `token != WETH` and `payer != address(this)`, it executes only `IERC20(token).safeTransferFrom(payer, recipient, value)` (L86) and never inspects `address(this).balance`. All swap entry-points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and all liquidity entry-points (`addLiquidityExactShares`, `addLiquidityWeighted`) are declared `payable`, so the EVM accepts any attached ETH without reverting. The `receive()` guard at L32–34 only blocks plain ETH transfers (no calldata); it does not prevent `msg.value` from being accepted on function calls. After the swap or liquidity operation completes, the ETH remains in the router. `refundETH()` at L58–63 is an unrestricted public function that sends `address(this).balance` to `msg.sender`, enabling any caller — including an MEV bot — to immediately drain the stranded ETH. A secondary variant exists for WETH swaps: when `msg.value > amountIn`, Branch 2 wraps exactly `value` (L76) and leaves the surplus `msg.value - amountIn` in the router, also drainable via `refundETH()`.

## Impact Explanation
Direct loss of user ETH equal to `msg.value`. For the non-WETH case, 100% of attached ETH is lost. For the WETH-with-excess case, the surplus above `amountIn` is lost. This constitutes a direct loss of user principal, satisfying the Critical/High/Medium allowed impact gate for direct loss of user funds above Sherlock thresholds.

## Likelihood Explanation
Medium. A user may accidentally attach ETH when swapping non-WETH tokens due to a frontend bug, copy-paste error, or interface confusion. The attack requires no special permissions: any MEV bot monitoring the mempool can observe the victim's transaction, wait for inclusion, and immediately call `refundETH()` in the next block, or front-run with higher gas in the same block. The attack is trivially repeatable.

## Recommendation
1. **Reject ETH for non-WETH swaps**: At the start of each payable swap/liquidity function, add:
   ```solidity
   if (params.tokenIn != WETH && msg.value > 0) revert ETHNotAccepted();
   ```
2. **Reject excess ETH for WETH swaps**: For WETH swaps, add:
   ```solidity
   if (msg.value > params.amountIn) revert ExcessETH(msg.value, params.amountIn);
   ```
3. **Or make non-WETH functions non-payable**: Remove `payable` from functions that do not need to accept ETH, so the EVM itself reverts on any attached value.

## Proof of Concept
**Non-WETH case (100% ETH loss):**
```
1. Alice calls exactInputSingle{value: 1 ether}(params)
   where params.tokenIn = USDC (not WETH), params.amountIn = 1000 USDC

2. Inside metricOmmSwapCallback → _justPayCallback → pay(USDC, Alice, pool, 1000)
   → token != WETH → else branch (L85-86) → safeTransferFrom(Alice, pool, 1000)
   → address(this).balance (1 ether) is never touched

3. Swap completes. Router holds 1 ether.

4. Bob (MEV bot) calls refundETH() (L58-63) → receives 1 ether.

5. Alice loses 1 ether.
```

**WETH-with-excess case (partial ETH loss):**
```
1. Alice calls exactInputSingle{value: 2 ether}(params)
   where params.tokenIn = WETH, params.amountIn = 1 ether

2. pay(WETH, Alice, pool, 1 ether):
   nativeBalance (2 ether) >= value (1 ether) → deposit{value: 1 ether}() (L76)
   → transfer 1 WETH to pool
   → 1 ether surplus remains in router

3. Bob calls refundETH() → receives 1 ether surplus.

4. Alice loses 1 ether.
```
