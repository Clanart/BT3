Audit Report

## Title
`addLiquidity` Missing `whenNotPaused` Guard Allows Deposits Into Paused Pools - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool` defines a `pauseLevel` state variable and a `whenNotPaused` modifier that reverts when `pauseLevel != 0`. The modifier is applied to `swap()` but is absent from `addLiquidity()`. When a pool is paused in response to an oracle manipulation or security incident, any user — including those routing through `MetricOmmPoolLiquidityAdder` — can still deposit tokens into the compromised pool, placing their principal at immediate risk.

## Finding Description
`MetricOmmPool` declares `pauseLevel` and `whenNotPaused`: [1](#0-0) [2](#0-1) [3](#0-2) 

`swap()` correctly applies the modifier: [4](#0-3) 

`addLiquidity()` does not — it only carries `nonReentrant`: [5](#0-4) 

The `_beforeAddLiquidity` hook in `ExtensionCalling` only dispatches to optional pool extensions and performs no pause check of its own: [6](#0-5) 

The periphery `MetricOmmPoolLiquidityAdder._addLiquidity` calls `pool.addLiquidity` directly with no pause-state check: [7](#0-6) 

Both `addLiquidityExactShares` and `addLiquidityWeighted` route through this internal function: [8](#0-7) [9](#0-8) 

There is no compensating guard anywhere in the call chain.

## Impact Explanation
When a pool is paused due to an active exploit or oracle failure, the pause is intended to halt all pool activity that could expose user funds to the compromised state. Because `addLiquidity` bypasses `whenNotPaused`, users depositing after a pause is triggered — whether directly or via the periphery adder — transfer tokens into a pool whose price or accounting may be actively manipulated. Newly deposited principal is immediately at risk of being drained or permanently mispriced, constituting a direct loss of user principal. This meets the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation
The factory can pause any pool at any time via `setPause()`. Pausing is a routine emergency response. No special permissions or unusual conditions are required on the attacker's side — any user who calls `addLiquidityExactShares` or `addLiquidityWeighted` after a pause is triggered loses funds. The periphery is the standard user-facing entry point, making accidental deposits by ordinary users highly likely during an incident.

## Recommendation
Add `whenNotPaused` to `addLiquidity()` in `MetricOmmPool.sol`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` may intentionally remain unguarded to allow LPs to exit during a pause, but new deposits must be blocked.

## Proof of Concept
1. Factory admin calls `pool.setPause(1)` to pause the pool due to a suspected oracle issue.
2. User calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, ...)`.
3. `_addLiquidity` calls `IMetricOmmPoolActions(pool).addLiquidity(...)` — no `whenNotPaused` check fires.
4. Tokens are transferred from the user into the paused pool via the callback.
5. If the pool was paused due to an active exploit, the deposited tokens are immediately at risk of being drained.
