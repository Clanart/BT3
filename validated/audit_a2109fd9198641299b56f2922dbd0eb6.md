### Title
Sender Can Permanently Lock Bridge Funds by Maximizing Fee Before `sign_transfer` Finalization — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `update_transfer_fee` function allows the original sender of a pending NEAR-origin transfer to increase the token fee up to `amount - 1` at any time before `sign_transfer` is called. Because fees can only be increased (never decreased), and because `sign_transfer` panics with `InvalidAmountToTransfer` when decimal normalization reduces `amount_without_fee` to zero, a sender who raises the fee to `amount - 1` can render the transfer permanently unsignable. With no cancel or admin-recovery path for stuck pending transfers, the locked tokens are irrecoverable.

---

### Finding Description

**Root cause — `update_transfer_fee` allows unbounded fee increase by the sender:** [1](#0-0) 

The only upper bound on `fee.fee` is `fee.fee < transfer.message.amount`, so the sender can raise it to `amount - 1`. The constraint also enforces monotonicity (`fee.fee >= current_fee.fee`), so the fee can never be lowered after being raised.

**Root cause — `sign_transfer` panics when normalized amount is zero:** [2](#0-1) 

`normalize_amount` converts the NEAR-side amount to the destination chain's decimal precision. For tokens with a large decimal gap (e.g., a 24-decimal NEAR token bridged to a 6-decimal EVM token, normalization factor = 10¹⁸), any `amount_without_fee < 10¹⁸` normalizes to 0, triggering the `require!` panic.

**Root cause — no recovery path for stuck transfers:**

`remove_transfer_message` is only called in two places: [3](#0-2) 

(only when `fee.is_zero()`) and inside `claim_fee_callback`, which requires a proof that the transfer was already finalized on the destination chain. A transfer that was never signed can never produce such a proof, so neither path is reachable.

---

### Impact Explanation

**Permanent freezing of user funds.** Once the sender raises `fee.fee` to `amount - 1`:

1. `amount_without_fee = 1`.
2. For any token whose NEAR decimals exceed the destination-chain decimals by ≥ 1 (e.g., 24 vs. 6), `normalize_amount(1, decimals) = 0`.
3. Every subsequent call to `sign_transfer` panics at `InvalidAmountToTransfer`.
4. The transfer message remains in `pending_transfers` indefinitely.
5. The locked tokens in the bridge contract are irrecoverable.

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

**Medium.** The scenario is reachable by any unprivileged sender of a NEAR-origin transfer:

- The sender calls `ft_transfer_call` to initiate a transfer with a small amount (e.g., `amount = 10¹⁸` for a 24-decimal token) and `fee = 0`.
- The sender then calls `update_transfer_fee` to set `fee.fee = amount - 1 = 10¹⁸ - 1`.
- `normalize_amount(1, decimals)` returns 0 for any normalization factor > 1.
- `sign_transfer` permanently fails.

This can also occur accidentally: a user who sets a fee of 1 unit on a small transfer may not realize that decimal normalization will make the remaining 1 unit unsignable. The monotonicity constraint means there is no self-correction.

---

### Recommendation

1. **Add a lower bound check in `sign_transfer`** (or in `update_transfer_fee`) that rejects any fee update that would cause `normalize_amount(amount_without_fee, decimals) == 0`, preventing the transfer from entering an unsignable state.
2. **Add a sender-callable cancel/refund function** that allows the original sender to reclaim locked tokens for a pending transfer that has not yet been signed, subject to a time-lock or relayer-consent guard.
3. **Enforce a minimum `amount_without_fee` at fee-update time** equal to the normalization factor for the destination chain, so that the post-normalization amount is always ≥ 1.

---

### Proof of Concept

```
1. Token: NEAR-native token with 24 decimals, destination EVM with 6 decimals.
   Normalization factor = 10^(24-6) = 10^18.

2. Sender calls ft_transfer_call → init_transfer:
     amount = 10^18 + 1,  fee.fee = 0,  fee.native_fee = 0

3. Sender calls update_transfer_fee:
     new fee.fee = 10^18   (= amount - 1, satisfies fee.fee < amount)
     diff_native_fee = 0   (native_fee unchanged, no deposit required)
   → Stored fee.fee is now 10^18.

4. Relayer calls sign_transfer(transfer_id, fee_recipient, None):
     amount_without_fee = (10^18 + 1) - 10^18 = 1
     normalize_amount(1, decimals{factor=10^18}) = 1 / 10^18 = 0  (integer division)
     require!(0 > 0, ...)  →  PANIC: InvalidAmountToTransfer

5. Transaction reverts. Transfer message stays in pending_transfers.
   fee.fee cannot be lowered (monotonicity constraint).
   No cancel path exists.
   → 10^18 + 1 units of the token are permanently locked in the bridge.
``` [4](#0-3) [5](#0-4)

    #[payable]

    pub fn update_transfer_fee(&mut self, transfer_id: TransferId, fee: UpdateFee) {
        match fee {
            UpdateFee::Fee(fee) => {
                let mut transfer = self.get_transfer_message_storage(transfer_id);

                require!(
                    transfer.message.origin_transfer_id.is_none(),
                    BridgeError::UpdateFeeNotAllowedForTransfer.as_ref()
                );

                let current_fee = transfer.message.fee;
                require!(
                    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
                    BridgeError::InvalidFee.as_ref()
                );

                require!(
                    fee.fee == current_fee.fee
                        || OmniAddress::Near(env::predecessor_account_id())
                            == transfer.message.sender,
                    BridgeError::SenderCanUpdateTokenFeeOnly.as_ref()
                );

                let diff_native_fee = fee
                    .native_fee
                    .0
                    .checked_sub(current_fee.native_fee.0)
                    .near_expect(BridgeError::LowerFee);

                require!(
                    NearToken::from_yoctonear(diff_native_fee) == env::attached_deposit(),
                    BridgeError::InvalidAttachedDeposit.as_ref()
                );

                transfer.message.fee = fee;
                self.insert_raw_transfer(transfer.message.clone(), transfer.owner);

                env::log_str(
                    &OmniBridgeEvent::UpdateFeeEvent {
                        transfer_message: transfer.message,
                    }
                    .to_log_string(),
                );
            }
            UpdateFee::Proof(_) => {
                env::panic_str(BridgeError::UnsupportedFeeUpdateProof.to_string().as_str())
            }
        }
    }

    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };

        let payload = near_sdk::env::keccak256_array(
            transfer_payload
                .encode_hashable()
                .near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_TRANSFER_CALLBACK_GAS)
                    .sign_transfer_callback(transfer_payload, &transfer_message.fee),
            )
    }
