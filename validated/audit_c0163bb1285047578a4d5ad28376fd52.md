### Title
Permanent Fund Lock via Push-Strategy ETH Delivery in `finTransfer()` When Recipient Rejects ETH — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `finTransfer()` uses a push-strategy to deliver native ETH to the recipient. If the recipient is a contract that cannot receive ETH, the entire transaction reverts (including the nonce marking). Because the recipient is fixed inside the MPC-signed payload and cannot be changed by the relayer, and because the NEAR side has no cancel/refund path for a pending transfer, the user's NEAR-side tokens are permanently locked with no recovery mechanism.

---

### Finding Description

`finTransfer()` processes native ETH delivery (`tokenAddress == address(0)`) as follows:

```solidity
completedTransfers[payload.destinationNonce] = true;          // (1) mark nonce used

// ... signature verification ...

if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}(""); // (2) push ETH
    if (!success) revert FailedToSendEther();                             // (3) revert all
}
``` [1](#0-0) 

When step (3) fires, Solidity reverts the entire transaction, including the nonce write at step (1). The nonce is therefore **not** consumed, and the relayer can retry. However, the recipient address is part of the Borsh-encoded payload that the MPC signature covers: [2](#0-1) 

The relayer cannot substitute a different recipient without invalidating the signature. If the recipient is a contract that always reverts on ETH receive (no `receive()` function, deliberately reverting fallback, or a contract upgraded to reject ETH), every retry will fail identically.

On the NEAR side, `init_transfer` burns/locks the user's tokens and records a `TransferMessage` in `pending_transfers`. There is no `cancel_transfer` or user-facing refund function: [3](#0-2) 

The `claim_fee` path is for relayers only and requires a valid `FinTransfer` proof from EVM — which can never be produced if `finTransfer` always reverts: [4](#0-3) 

The result is a double lock: NEAR tokens are permanently burned/locked in `pending_transfers`, and the ETH remains stranded in the EVM contract.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.** The user's NEAR-side tokens are burned or locked and cannot be recovered. The ETH in the EVM contract is also unreachable by the user. No admin escape hatch or user-facing cancel function exists. This directly matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

The scenario is reachable by any unprivileged bridge user who specifies an EVM contract recipient that cannot receive ETH:

- A smart-contract wallet or multisig without a `receive()` function (common for Gnosis Safe deployments on certain chains).
- A contract that was upgraded after the transfer was initiated to remove ETH acceptance.
- A contract that deliberately reverts on ETH receive (a malicious contract the user was socially-engineered into using as recipient).

As smart-contract wallet adoption grows, users increasingly bridge to contract addresses rather than EOAs, making this scenario progressively more likely.

---

### Recommendation

1. **Pull-strategy**: Instead of pushing ETH directly, store the amount in a per-recipient claimable balance (`mapping(address => uint256) public claimableEth`). Add a separate `claimEth()` function the recipient calls to withdraw. This eliminates the revert-on-receive failure mode entirely.
2. **Fallback storage on failure**: If the direct `call{value}` fails, store the ETH in a claimable balance rather than reverting, so the nonce is consumed and the NEAR side can treat the transfer as finalized.
3. **NEAR-side cancel path**: Add a `cancel_transfer` function on NEAR that allows a user to reclaim locked tokens for a transfer whose `pending_transfers` entry has been present beyond a timeout, providing a recovery path when EVM finalization is permanently impossible.

---

### Proof of Concept

1. User calls `init_transfer` on NEAR, locking 1 NEAR-ETH worth of tokens, specifying as EVM recipient a contract `MaliciousReceiver` whose `receive()` always reverts.
2. NEAR tokens are burned; `TransferMessage` is stored in `pending_transfers`.
3. Relayer obtains MPC signature over the payload (which includes `MaliciousReceiver` as recipient) and calls `finTransfer` on EVM.
4. `completedTransfers[nonce] = true` is written.
5. `MaliciousReceiver.call{value: amount}("")` reverts.
6. `revert FailedToSendEther()` fires; all state (including nonce) rolls back.
7. Relayer retries indefinitely — same outcome every time.
8. User's NEAR tokens remain permanently locked in `pending_transfers`. No `cancel_transfer` exists. Funds are irrecoverable. [5](#0-4) [6](#0-5)
