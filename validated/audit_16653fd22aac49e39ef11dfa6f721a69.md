### Title
`nativeFee` ETH Paid by Bridge Users Is Permanently Locked in `OmniBridgeWormhole` With No Distribution or Withdrawal Path — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer` and `initTransfer1155` accept ETH for `nativeFee` from every bridging user, subtract it from `msg.value` before forwarding the remainder to Wormhole, but never transfer the `nativeFee` ETH to any relayer or return it to the user. There is no withdrawal function. The ETH accumulates in `OmniBridgeWormhole` permanently.

---

### Finding Description

In `OmniBridge.initTransfer`, the contract computes `extensionValue` — the ETH forwarded to Wormhole — by subtracting `nativeFee` from `msg.value`:

**ERC-20 path:**
```solidity
extensionValue = msg.value - nativeFee;   // nativeFee stays in contract
``` [1](#0-0) 

**Native ETH path:**
```solidity
extensionValue = msg.value - amount - nativeFee;  // amount + nativeFee stay in contract
``` [2](#0-1) 

**ERC-1155 path:**
```solidity
uint256 extensionValue = msg.value - nativeFee;   // nativeFee stays in contract
``` [3](#0-2) 

In `OmniBridgeWormhole.initTransferExtension`, only `value` (i.e., `extensionValue`) is forwarded to Wormhole:

```solidity
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [4](#0-3) [5](#0-4) 

The `nativeFee` ETH is never paid to the relayer in `finTransfer` — that function only sends `payload.amount` to `payload.recipient` for native-ETH transfers, and pays `msg.value` to Wormhole for the outbound message: [6](#0-5) [7](#0-6) 

The contract has a bare `receive()` confirming it can hold ETH, and neither `OmniBridge.sol` nor `OmniBridgeWormhole.sol` contains any `withdrawFees`, `sweep`, or admin-rescue function: [8](#0-7) 

The `nativeFee` value is emitted in the `InitTransfer` event and encoded in the Wormhole payload, but the ETH itself has no on-chain disbursement path: [9](#0-8) [10](#0-9) 

---

### Impact Explanation

Every user who calls `initTransfer` or `initTransfer1155` with `nativeFee > 0` permanently loses that ETH. The ETH accumulates in `OmniBridgeWormhole` and is irrecoverable by the user. No relayer receives it either, so the incentive mechanism is broken end-to-end. This matches the allowed critical impact: **permanent freezing / irrecoverable lock of user funds in a bridge flow**.

---

### Likelihood Explanation

`initTransfer` is the primary public entry point for all EVM-side bridge transfers. Any user who sets `nativeFee > 0` to incentivize a relayer triggers the lock. Because the `nativeFee` parameter is a normal, documented part of the bridge API, real users will routinely supply non-zero values. The entry path requires no special role or privilege.

---

### Recommendation

1. In `finTransfer`, pay `nativeFee` (recoverable from the signed payload or a separate accounting mapping) to `msg.sender` (the relayer) before or after the asset transfer.
2. Alternatively, add an admin-only `withdrawAccumulatedFees(address recipient)` function so locked ETH can be rescued.
3. Enforce the invariant that the contract holds no ETH beyond locked native-asset bridge amounts by validating `address(this).balance` before and after each operation, analogous to the fix described in the referenced report (commit `870521d`).

---

### Proof of Concept

1. Alice calls `initTransfer(USDC, 100e6, 0, 1e18 /*nativeFee*/, "near:alice.near", "")` sending `msg.value = 1e18 + wormholeFee`.
2. `extensionValue = msg.value - 1e18 = wormholeFee` → Wormhole receives `wormholeFee`.
3. `1e18` ETH remains in `OmniBridgeWormhole`. No function exists to withdraw it.
4. Repeat for every bridging user. The contract balance grows monotonically with no disbursement path.
5. Relayers, seeing no on-chain ETH reward for calling `finTransfer`, have no economic incentive to relay, potentially stalling bridge finalization — compounding the impact from a pure accounting loss into a liveness failure.
