### Title
Integer Overflow in Relative Timelock Arithmetic Allows Timelock Bypass — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
When `check_time_locks` is invoked with `nowrap=false` (the pre-softfork consensus path), it uses `wrapping_add` for relative timelock deadline computation. A CLVM spend embedding a near-maximum `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` value causes the computed deadline to wrap around to a value smaller than the current block timestamp/height, making the timelock check pass immediately. This is a direct analog to the external report's "missing restriction" class: just as `processCommitment` lacked a check for `commitmentHalfLife`, `check_time_locks` under `nowrap=false` fails to enforce the intended relative timelock when the operand is crafted to overflow.

### Finding Description
In `check_time_locks`, the relative timelock checks branch on the `nowrap` flag:

```rust
// ASSERT_HEIGHT_RELATIVE path (nowrap=false branch)
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

```rust
// ASSERT_SECONDS_RELATIVE path (nowrap=false branch)
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

When `height_relative` is chosen such that `confirmed_block_index.wrapping_add(height_relative)` overflows `u32`, the result wraps to a small value (e.g., `100u32.wrapping_add(0xffff_ffff) = 99`). The check `prev_height < 99` is then `false` for any `prev_height >= 99`, so no error is returned and the spend is accepted — even though the intended timelock is ~4 billion blocks. The same applies to `seconds_relative` with `u64` wrapping.

The codebase's own test suite explicitly documents and confirms this behavior:

```rust
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),  // <-- timelock bypassed under nowrap=false
)]
``` [3](#0-2) 

```rust

The `nowrap=false` path is the pre-softfork consensus rule. Any block validated under old consensus rules (or any caller that incorrectly passes `nowrap=false` for new blocks) is subject to this bypass.

### Impact Explanation
A CLVM puzzle embedding `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a near-maximum value (crafted to overflow when added to the coin's confirmation timestamp/height) will have its timelock silently bypassed under `nowrap=false`. The coin becomes immediately spendable at any block, regardless of the intended lock duration. This enables unauthorized spend acceptance — a coin the puzzle author intended to lock for an astronomically long period can be spent at the next block. This matches the **High** impact category: timelock condition validation bypass enabling unauthorized spend acceptance.

### Likelihood Explanation
The attacker-controlled entry path is a CLVM puzzle/solution submitted as part of a spend bundle. No privileged access is required. The attacker only needs to craft a puzzle that outputs `ASSERT_SECONDS_RELATIVE` with a value in the range `[u64::MAX - coin_timestamp + 1, u64::MAX]`. The `nowrap=false` path is active for all blocks below the soft-fork activation height, and remains present in the codebase as a reachable code path whenever the Python caller passes `nowrap=False`.

### Recommendation
Replace `wrapping_add` with `saturating_add` unconditionally for all relative timelock arithmetic, removing the `nowrap` branch entirely. The `nowrap=false` path preserves a broken consensus rule that was already identified as incorrect. Alternatively, enforce `nowrap=true` at the Rust level and remove the `nowrap` parameter from the public API, delegating any backward-compatibility handling to a higher layer that does not call into this function for old blocks.

### Proof of Concept
Coin confirmed at height 100, timestamp 1000. Attacker crafts a CLVM puzzle emitting:

```
(ASSERT_SECONDS_RELATIVE . 0xffff_ffff_ffff_ffff)
```

Under `nowrap=false`:
```
deadline = 1000u64.wrapping_add(0xffff_ffff_ffff_ffff) = 999
check: timestamp(2000) < 999  →  false  →  no error  →  spend accepted
```

Under `nowrap=true` (correct behavior):
```
deadline = 1000u64.saturating_add(0xffff_ffff_ffff_ffff) = u64::MAX
check: timestamp(2000) < u64::MAX  →  true  →  AssertSecondsRelativeFailed
```

The spend that should be locked for ~585 billion years is accepted immediately when `nowrap=false` is active. [7](#0-6)
