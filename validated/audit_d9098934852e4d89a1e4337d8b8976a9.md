Audit Report

## Title
`MetricOmmPoolLiquidityAdder` accepts arbitrary pool addresses without factory validation, enabling a malicious pool to drain user-approved tokens — (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in `addLiquidityExactShares` and `addLiquidityWeighted` and stores it as the trusted callback caller without verifying it against `MetricOmmPoolFactory.isPool`. A malicious contract implementing `IMetricOmmPoolActions.addLiquidity` can call back `metricOmmModifyLiquidityCallback`, pass the `msg.sender == expectedPool` guard (since it is the stored pool), and pull up to `maxAmountToken0`/`maxAmountToken1` from the user's approved balance. The router (`MetricOmmSimpleRouter`) is **not** affected — it already calls `_requireFactoryPool(pool)` inside `_setNextCallbackContext` and `FACTORY.isPool(caller)` inside `_requireExpectedCallbackCaller`, providing two layers of factory validation.

## Finding Description

`MetricOmmPoolLiquidityAdder` has no `factory` reference. Its constructor accepts only `weth`: [1](#0-0) 

The NatSpec comment on the contract explicitly acknowledges the missing guard: [2](#0-1) 

`_addLiquidity` stores the caller-supplied `pool` as the expected callback caller, then calls `pool.addLiquidity(...)`: [3](#0-2) 

In `metricOmmModifyLiquidityCallback`, the only guard is `msg.sender == expectedPool` — which is the attacker-controlled address — and then tokens are pulled from the user up to `max0`/`max1`: [4](#0-3) 

The token addresses are also read from `msg.sender` (the malicious pool) via `getImmutables()`, so the attacker controls which tokens are pulled: [5](#0-4) 

By contrast, `MetricOmmSwapRouterBase._setNextCallbackContext` calls `_requireFactoryPool(pool)` before storing any pool address, and `_requireExpectedCallbackCaller` additionally calls `FACTORY.isPool(caller)` at callback time — the router is not vulnerable: [6](#0-5) [7](#0-6) 

The factory's `isPool` registry is already populated and queryable: [8](#0-7) 

## Impact Explanation

A user who calls `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 from their approved balance. The malicious pool's `addLiquidity` implementation calls back `metricOmmModifyLiquidityCallback` requesting the full cap amounts; the adder pays them from the user and receives no LP position in return. This is a direct loss of user principal, meeting the Medium/High threshold under the allowed impact gate.

## Likelihood Explanation

Deploying a malicious contract implementing `IMetricOmmPoolActions` is trivial and permissionless. Users interacting via front-ends that resolve pool addresses are susceptible to phishing UIs or compromised integrations supplying the malicious address. Users commonly grant large or unlimited ERC-20 approvals to liquidity adder contracts, maximising the drained amount. The on-chain exploit path requires no privileged access.

## Recommendation

Add a `factory` address at construction to `MetricOmmPoolLiquidityAdder` and apply the same `isPool` check used by the router before `_setPayContext` is called:

```solidity
IMetricOmmPoolFactory internal immutable FACTORY;

constructor(address weth, address factory) PeripheryPayments(weth) {
    FACTORY = IMetricOmmPoolFactory(factory);
}

function _requireLegitimatePool(address pool) internal view {
    if (!FACTORY.isPool(pool)) revert NotARegisteredPool(pool);
}
```

Call `_requireLegitimatePool(pool)` at the top of `addLiquidityExactShares` and `addLiquidityWeighted` before any interaction with the pool address.

## Proof of Concept

```solidity
contract MaliciousPool {
    address adder;
    address victim;
    address token0;
    address token1;

    // Implements IMetricOmmPoolActions.addLiquidity
    function addLiquidity(
        address, uint80, LiquidityDelta calldata, bytes calldata callbackData, bytes calldata
    ) external returns (uint256, uint256) {
        // Call back the adder: requests full cap amounts
        IMetricOmmModifyLiquidityCallback(adder).metricOmmModifyLiquidityCallback(
            drainAmount0, drainAmount1, callbackData
        );
        return (drainAmount0, drainAmount1);
    }

    // getImmutables() returns attacker-controlled token0/token1
    function getImmutables() external view returns (PoolImmutables memory) {
        return PoolImmutables({token0: token0, token1: token1, ...});
    }
}

// Attack:
// 1. Attacker deploys MaliciousPool with token0=USDC, token1=USDT
// 2. Victim approves adder for USDC and USDT (e.g., 10_000 each)
// 3. Victim calls adder.addLiquidityExactShares({
//        pool: MaliciousPool,
//        maxAmountToken0: 10_000,
//        maxAmountToken1: 10_000, ...
//    })
// 4. Adder stores MaliciousPool as expectedPool, calls MaliciousPool.addLiquidity(...)
// 5. MaliciousPool calls back metricOmmModifyLiquidityCallback(10_000, 10_000, ...)
// 6. Guard passes: msg.sender == expectedPool == MaliciousPool
// 7. Adder reads token0/token1 from MaliciousPool.getImmutables() → USDC/USDT
// 8. Adder pays 10_000 USDC and 10_000 USDT from victim to MaliciousPool
// 9. Victim loses funds, receives no LP position
```
