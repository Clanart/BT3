### Title
Blacklisted ERC-20 Recipient in `finTransfer` Causes Permanent Fund Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` transfers native ERC-20 tokens (e.g., USDC) directly to `payload.recipient` in a single atomic call. If that address is blacklisted by the token contract, every invocation reverts. Because the recipient is embedded in an immutable MPC-signed payload, no alternative delivery path exists, and the user's funds are permanently locked on the source chain.

---

### Finding Description

In `OmniBridge.finTransfer`, after signature verification and nonce marking, the contract attempts to deliver tokens directly to the user-supplied recipient:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
``` [1](#0-0) 

The nonce is marked used at the top of the function, before any transfer:

```solidity
completedTransfers[payload.destinationNonce] = true;
``` [2](#0-1) 

Because `safeTransfer` is called inside the same transaction, a revert from the token contract (e.g., USDC blacklist check) rolls back the entire transaction, including the nonce assignment. The nonce is therefore **not** permanently consumed — but this provides no relief.

The `payload.recipient` is part of the Borsh-encoded, MPC-signed `TransferMessagePayload`:

```solidity
bytes memory borshEncoded = bytes.concat(
    ...
    bytes1(omniBridgeChainId),
    Borsh.encodeAddress(payload.recipient),
    ...
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

The recipient is cryptographically bound to the signed payload. There is no mechanism in the contract to substitute an alternative recipient, no pull-withdrawal pattern, and no admin rescue function for stuck transfers. Every future call to `finTransfer` with the same signed payload will revert identically.

The source-chain funds (locked or burned on NEAR when `ft_on_transfer` / `init_transfer` was called) cannot be recovered because the NEAR bridge has no refund path once a transfer is in-flight toward EVM.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

- Tokens are locked or burned on NEAR at `initTransfer` time.
- The only release mechanism on EVM is `finTransfer`, which is permanently blocked for a blacklisted recipient.
- No admin function, no alternative delivery, no refund path exists.
- The user's funds are unrecoverable without a contract upgrade.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

USDC is explicitly supported as a bridgeable native ERC-20 (it falls into the `else` branch — not a bridge token, not ETH, not ERC-1155). USDC's blacklist is actively maintained by Circle. A user can be blacklisted:

- After initiating the transfer but before finalization (race condition).
- Because they provided a recipient address that is already blacklisted (e.g., a smart contract wallet later sanctioned).
- Due to regulatory action against the user between the two-chain settlement window.

The scenario requires no privileged access and no attacker — it is a natural consequence of the protocol's design when interacting with tokens that have transfer restrictions.

---

### Recommendation

Replace the direct-push transfer with a **pull-withdrawal (claim) pattern** for native ERC-20 tokens:

1. In `finTransfer`, instead of calling `safeTransfer` to `payload.recipient`, record a claimable balance: `claimable[payload.recipient][payload.tokenAddress] += payload.amount`.
2. Add a separate `claimTokens(address tokenAddress)` function that lets the recipient pull their funds.

This isolates a blacklisted recipient's failure to their own claim, leaving the nonce consumed and all other users unaffected. The same pattern should be applied to the ETH delivery path (`payload.recipient.call{value: payload.amount}("")`) at line 319, which has an analogous failure mode for contracts that reject ETH. [4](#0-3) 

---

### Proof of Concept

**Setup:** USDC is a native ERC-20 bridged via Omni Bridge (held in the `OmniBridge` contract on EVM, locked on NEAR).

1. Alice initiates a USDC transfer from NEAR to EVM, specifying her EVM address as `recipient`. Her tokens are locked on NEAR via `ft_on_transfer` → `init_transfer`.
2. Circle blacklists Alice's EVM address (e.g., due to a regulatory action).
3. The MPC network signs a `TransferMessagePayload` with `recipient = Alice_EVM`, `tokenAddress = USDC`.
4. A relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
5. Execution reaches line 351: `IERC20(USDC).safeTransfer(Alice_EVM, amount)`.
6. USDC's `transfer` reverts because `Alice_EVM` is blacklisted.
7. The entire transaction reverts. The nonce is not consumed.
8. Every subsequent call to `finTransfer` with the same signed payload reverts identically — the recipient is immutable in the signature.
9. Alice's USDC is permanently locked on NEAR. There is no refund, no redirect, and no admin rescue path in the contract. [5](#0-4)
