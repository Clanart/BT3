### Title
`ApplyMessage` Bypasses `PostTxProcessing` Hooks on Native-Module EVM Calls with Committed State — (`File: x/evm/keeper/state_transition.go`)

---

### Summary

`ApplyMessage`, the documented public API for native Cosmos modules to execute EVM calls with committed state, never invokes `PostTxProcessing`. `ApplyTransaction` — the path for user-submitted Ethereum transactions — does invoke `PostTxProcessing`. Any security invariant enforced exclusively through the `EvmHooks` interface is therefore silently bypassed whenever a native module calls `ApplyMessage(..., commit=true)`.

---

### Finding Description

`ApplyTransaction` wraps `ApplyMessageWithConfig` inside a cache context and, on success, calls `PostTxProcessing` before committing:

```go
// x/evm/keeper/state_transition.go  lines 229-245
if !res.Failed() {
    receipt.Status = ethtypes.ReceiptStatusSuccessful
    // Only call hooks if tx executed successfully.
    if err = k.PostTxProcessing(tmpCtx, msg, receipt); err != nil {
        res.VmError = types.ErrPostTxProcessing.Error()
        ...
        res.Logs = nil
    } else if commit != nil {
        commit()          // ← state committed only after hooks pass
        tmpCtxCommitted = true
        ...
    }
}
```

`ApplyMessage`, by contrast, is a thin wrapper that calls `ApplyMessageWithConfig` and returns immediately — no hook invocation, no cache-context guard:

```go
// x/evm/keeper/state_transition.go  lines 277-290
func (k *Keeper) ApplyMessage(ctx sdk.Context, msg *core.Message,
    tracer *tracing.Hooks, commit bool) (*types.EVMResult, error) {
    cfg, err := k.EVMConfig(ctx, k.eip155ChainID, common.Hash{})
    ...
    result, err := k.ApplyMessageWithConfig(ctx, msg, cfg, commit)
    ...
    return result, nil   // ← PostTxProcessing never called
}
```

The code comment on `ApplyMessageWithConfig` explicitly lists "Called by other native modules directly" as scenario 3, confirming this is the intended integration point for native modules:

```
// # Different Callers
// It's called in three scenarios:
// 1. `ApplyTransaction`, in the transaction processing flow.
// 2. `EthCall/EthEstimateGas` grpc query handler.
// 3. Called by other native modules directly.
```

When `commit=true` is passed, `ApplyMessageWithConfig` calls `stateDB.Commit()` and persists all EVM state changes to the KV store — without any hook ever running.

---

### Impact Explanation

`PostTxProcessing` is the sole mechanism Ethermint exposes for post-execution enforcement: token-pause checks, ERC-20 ↔ Cosmos coin accounting, access-control gates, and similar invariants are all implemented there by integrating chains. Because `ApplyMessage` never calls `PostTxProcessing`, any native module that routes an EVM call through `ApplyMessage(..., true)` commits EVM state changes while completely skipping every registered hook. An attacker who can trigger such a native-module path (e.g., via a cross-chain message, governance execution, or any Cosmos SDK `MsgServer` that internally calls `ApplyMessage`) can:

- Bypass a token-pause check and execute a withdrawal that should be blocked.
- Bypass an ERC-20 ↔ bank-coin accounting hook, minting or burning Cosmos-native coins without the corresponding EVM-side debit/credit.
- Bypass any other hook-enforced invariant while the EVM state is permanently committed.

This matches the "High" allowed impact: *EVM state transition bug that permits invalid transactions to commit*.

---

### Likelihood Explanation

The vulnerability is structural and unconditional: every call to `ApplyMessage(..., true)` from any native module bypasses hooks by design. No special attacker capability is required beyond the ability to submit a Cosmos SDK transaction that reaches a native module using `ApplyMessage`. The ZetaChain audit finding demonstrates that real integrations do exactly this. Any Ethermint-based chain that (a) registers hooks for security enforcement and (b) has native modules calling `ApplyMessage` is affected.

---

### Recommendation

`ApplyMessage` should mirror `ApplyTransaction`'s hook-invocation pattern when `commit=true`:

1. If `k.hooks != nil` and `commit == true`, wrap the call in a cache context.
2. After `ApplyMessageWithConfig` returns a non-failed result, call `k.PostTxProcessing`.
3. Commit the cache context only if `PostTxProcessing` succeeds; otherwise surface the hook error as a `VmError` and discard the cache context.

Alternatively, document `ApplyMessage` as explicitly hook-free and require callers that need hook enforcement to use `ApplyTransaction` or to call `PostTxProcessing` themselves after `ApplyMessage` returns.

---

### Proof of Concept

The structural bypass is directly visible in the source:

**`ApplyTransaction` — hooks run, state committed only after hooks pass:** [1](#0-0) 

**`ApplyMessage` — no hook call, state committed unconditionally when `commit=true`:** [2](#0-1) 

**`ApplyMessageWithConfig` comment confirming native-module use as scenario 3:** [3](#0-2) 

**`stateDB.Commit()` called unconditionally inside `ApplyMessageWithConfig` when `commit=true`:** [4](#0-3) 

**`PostTxProcessing` keeper method — only reachable from `ApplyTransaction`, never from `ApplyMessage`:** [5](#0-4) 

A native module that calls `k.EvmKeeper.ApplyMessage(ctx, msg, nil, true)` will commit all EVM state changes while every registered `EvmHooks` implementation is silently skipped. The ZetaChain M-26 finding is the exact real-world exploitation of this pattern: the crosschain module called the EVM below the hook layer, bypassing the ZRC20 pause check that lived in `PostTxProcessing`.

func (k *Keeper) ApplyMessage(ctx sdk.Context, msg *core.Message, tracer *tracing.Hooks, commit bool) (*types.EVMResult, error) {

// PostTxProcessing delegate the call to the hooks. If no hook has been registered, this function returns with a `nil` error
func (k *Keeper) PostTxProcessing(ctx sdk.Context, msg *core.Message, receipt *ethtypes.Receipt) error {
	if k.hooks == nil {
		return nil
	}
	return k.hooks.PostTxProcessing(ctx, msg, receipt)
}
