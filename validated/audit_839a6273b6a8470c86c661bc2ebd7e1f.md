### Title
Missing Chain ID in `deployToken` Signed Payload Enables Cross-Chain Signature Replay for Unauthorized Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies a MPC/NEAR-bridge signature over a borsh-encoded payload that **omits the destination chain ID (`omniBridgeChainId`)**. Because the same `nearBridgeDerivedAddress` is used across all deployed EVM chains, a valid `deployToken` signature produced for one chain (e.g., Ethereum) can be replayed verbatim on any other EVM chain (e.g., Arbitrum, BSC) to deploy an unauthorized bridge token, permanently poisoning that chain's token registry and blocking any future legitimate deployment of the same token.

---

### Finding Description

`OmniBridge.deployToken` constructs the signed message as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The signed payload contains only `PayloadType.Metadata | token | name | symbol | decimals`. **`omniBridgeChainId` is entirely absent.** Compare this with `finTransfer`, which correctly binds the payload to the destination chain:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
``` [2](#0-1) 

The `nearBridgeDerivedAddress` is the NEAR bridge's MPC-derived Ethereum address. Because the NEAR bridge is a single contract on NEAR, it uses the same MPC key — and therefore the same `nearBridgeDerivedAddress` — for every EVM chain it is deployed on. This is confirmed by the initializer, which accepts `nearBridgeDerivedAddress` as a constructor parameter shared across deployments: [3](#0-2) 

Once a `deployToken` signature is broadcast on chain A (it is publicly visible in calldata), any party can submit the identical `(signatureData, metadata)` tuple to the `OmniBridge` contract on chain B. The signature check passes because the hash is identical, and the function proceeds to:

1. Deploy a new `BridgeToken` proxy on chain B.
2. Write `nearToEthToken[metadata.token] = bridgeTokenProxy` and `isBridgeToken[bridgeTokenProxy] = true`.
3. Emit a Wormhole message (in `OmniBridgeWormhole`) that notifies the NEAR bridge of the new token address on chain B. [4](#0-3) 

After step 3, the NEAR bridge registers the replayed token address for chain B via `deploy_token_callback`, which trusts the factory address check but has no way to detect that the EVM signature was replayed from another chain: [5](#0-4) 

The `deployToken` function then guards against a second deployment with:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [6](#0-5) 

This means once the attacker's replay succeeds, **no legitimate deployment of the same token on chain B is ever possible again** — the slot is permanently occupied by the attacker-controlled proxy.

The same structural omission exists in the Solana `DeployTokenPayload.serialize_for_near`, which also serializes only `IncomingMessageType::Metadata | token | name | symbol | decimals` with no chain identifier: [7](#0-6) 

---

### Impact Explanation

**High — Proof/signature verification bypass enabling unauthorized token deployment and token-mapping corruption.**

- An attacker deploys an unauthorized bridge token on any EVM chain using a signature intended for a different chain.
- The `nearToEthToken` mapping on the target chain is permanently poisoned: the legitimate NEAR bridge can never deploy the correct token there (`ERR_TOKEN_EXIST`).
- If the Wormhole relay propagates the `DeployToken` message, the NEAR bridge registers the attacker-controlled proxy as the canonical address for that token on the target chain, causing all future `finTransfer` mints to go to the attacker-controlled contract.
- This matches the allowed impact: **"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment"** and **"token-mapping corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

**High.** The attack requires no privileged access:

1. The attacker monitors any EVM chain for a successful `deployToken` call and extracts `(signatureData, metadata)` from calldata — both are public.
2. The attacker submits the same call to the `OmniBridge` on any other EVM chain where the same `nearBridgeDerivedAddress` is configured.
3. No key material, no MPC collusion, no oracle manipulation is needed.

The only precondition is that at least one legitimate `deployToken` has been executed on any chain, which is a normal operational event.

---

### Recommendation

Include `omniBridgeChainId` (and optionally the contract address) in the borsh-encoded payload that is signed by the NEAR bridge MPC key:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // <-- add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to the Solana `DeployTokenPayload.serialize_for_near` by prepending `SOLANA_OMNI_BRIDGE_CHAIN_ID` before the token fields, mirroring the pattern already used in `DeployTokenResponse.serialize_for_near`: [8](#0-7) 

The NEAR bridge MPC signing logic must be updated to include the target chain ID when producing `deployToken` signatures.

---

### Proof of Concept

1. Token `wrap.near` is legitimately deployed on Ethereum Sepolia. The transaction calldata contains `signatureData` (65 bytes) and `metadata = {token: "wrap.near", name: "Wrapped NEAR", symbol: "wNEAR", decimals: 24}`.

2. Attacker copies `signatureData` and `metadata` verbatim and calls `OmniBridge.deployToken(signatureData, metadata)` on Arbitrum Sepolia, where the same `nearBridgeDerivedAddress` is configured.

3. The hash computed on Arbitrum is:
   ```
   keccak256(0x01 || borsh("wrap.near") || borsh("Wrapped NEAR") || borsh("wNEAR") || 0x18)
   ```
   This is **identical** to the hash computed on Ethereum because `omniBridgeChainId` is not included.

4. `ECDSA.recover` returns `nearBridgeDerivedAddress` → signature check passes.

5. A new `BridgeToken` proxy is deployed on Arbitrum. `nearToEthToken["wrap.near"]` on Arbitrum now points to the attacker-triggered proxy.

6. The `OmniBridgeWormhole.deployTokenExtension` emits a Wormhole message with `omniBridgeChainId = <Arbitrum>` and the new proxy address. The NEAR bridge processes this and registers the proxy as the canonical `wrap.near` address on Arbitrum.

7. Any subsequent attempt by the legitimate NEAR bridge to deploy `wrap.near` on Arbitrum reverts with `ERR_TOKEN_EXIST`. The token mapping is permanently corrupted.
