### Title
`ProofOfSpace` v0 parse accepts neither-pool-field-set, but `compute_plot_id()` panics on that input — invariant violation in consensus-critical path (`crates/chia-protocol/src/proof_of_space.rs`)

---

### Summary

The `ProofOfSpace::parse` implementation is lenient for version=0 (v1 proofs): it accepts serialized inputs where **both** `pool_public_key` and `pool_contract_puzzle_hash` are `None`. However, `compute_plot_id()` on such a struct unconditionally calls `compute_plot_id_v1`, which contains an explicit `panic!` when neither field is set. This breaks the invariant that every successfully-parsed `ProofOfSpace` is safe to call `compute_plot_id()` on.

---

### Finding Description

**Parse is lenient for version=0:**

In `ProofOfSpace::parse`, the guard that rejects the neither-set case only fires for `version == 1`: [1](#0-0) 

For `version == 0`, the code falls straight through to `Ok(ProofOfSpace { ... })` with no such check: [2](#0-1) 

The test suite explicitly documents and exercises this lenient behavior: [3](#0-2) 

**`compute_plot_id()` panics on the neither-set case:**

`compute_plot_id()` for version=0 calls `compute_plot_id_v1` unconditionally: [4](#0-3) 

`compute_plot_id_v1` contains a hard `panic!` when both pool fields are `None`: [5](#0-4) 

---

### Impact Explanation

An attacker can craft a serialized `RewardChainBlock` (or any block/header containing a `ProofOfSpace`) with a version=0 proof where both pool fields are absent. The block deserializes without error. Any downstream code — including the Python full node via the `py_compute_plot_id` binding — that calls `compute_plot_id()` on the parsed struct will trigger the `panic!`. [6](#0-5) 

Under pyo3, a Rust `panic!` is caught and re-raised as a Python `PanicException`. If the full node's consensus validation path does not explicitly handle this exception class, the validation coroutine/thread crashes. Depending on the full node's error handling, this can cause the node to drop the peer connection, enter an error loop, or crash entirely. Because the block parses successfully, every node that attempts to validate it will hit the same panic, making the impact potentially network-wide.

The `update_digest` path for version=0 does **not** call `compute_plot_id()`, so hashing alone is safe — the panic is only reachable when `compute_plot_id()` is explicitly invoked, which the consensus validation path does. [7](#0-6) 

---

### Likelihood Explanation

- The malformed input is trivially constructable: set the version=0 prefix byte to `0x00` (neither pool field set, version=0) and fill the remaining fields with valid-length bytes.
- No privileged access, key material, or network position is required.
- The parse succeeds, so the block reaches the validation stage where `compute_plot_id()` is called.
- The `panic!` is unconditional and deterministic.

---

### Recommendation

Add the same mutual-exclusivity guard for version=0 that already exists for version=1. In `ProofOfSpace::parse`, after the `version == 0` branch constructs the struct, reject the neither-set case:

```rust
if version == 0 {
    let size = ...;
    let proof = ...;
    if pool_public_key.is_none() && pool_contract_puzzle_hash.is_none() {
        return Err(Error::InvalidPoS);
    }
    Ok(ProofOfSpace { ... })
}
```

Alternatively, replace the `panic!` in `compute_plot_id_v1` with a `Result` return type and propagate the error, making `compute_plot_id()` return `Result<Bytes32>` throughout.

The test case `v0_neither_lenient` in the roundtrip suite should be moved to the `parse_rejects` suite. [8](#0-7) 

---

### Proof of Concept

```rust
use chia_protocol::ProofOfSpace;
use chia_traits::Streamable;
use std::io::Cursor;

// Craft a minimal v0 ProofOfSpace with neither pool field set.
// Byte layout (simplified):
//   [32 bytes challenge]
//   [0x00]              <- pool_public_key absent
//   [0x00]              <- prefix byte: version=0, pool_contract absent
//   [48 bytes plot_pk]
//   [0x20]              <- size = 32
//   [0x00 0x00 0x00 0x01 0x80] <- proof (length-prefixed)
let mut buf = vec![0u8; 32];   // challenge
buf.push(0x00);                 // pool_public_key = None
buf.push(0x00);                 // prefix: version=0, contract=None
buf.extend_from_slice(&[0u8; 48]); // plot_public_key (dummy, will fail G1 decode in practice)
// ... fill remaining fields ...

let pos = ProofOfSpace::parse::<false>(&mut Cursor::new(&buf)).unwrap();
// parse succeeds — no error

pos.compute_plot_id(); // PANICS: "invalid proof of space. Neither pool pk nor contract puzzle hash set"
```
