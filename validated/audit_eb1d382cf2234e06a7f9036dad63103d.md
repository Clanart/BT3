### Title
`init_transfer` Accepts Transfers That Normalize to Zero, Permanently Freezing User Funds - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The `init_transfer` function in the NEAR bridge contract does not validate that `normalize_amount(amount_without_fee, decimals) > 0` before locking user tokens. When a user initiates a transfer with an `amount_without_fee` below the decimal normalization threshold (`10^(origin_decimals - decimals)`), the tokens are permanently locked because `sign_transfer` will always revert with `InvalidAmountToTransfer`, and no recovery path exists.

### Finding Description

The vulnerability class from the external report is **integer division truncation producing zero with no upstream guard**, causing incorrect protocol behavior. The analog in Omni Bridge is structurally identical: `normalize_amount` uses floor division and can return 0 for valid non-zero inputs, but the zero-check guard exists only in `sign_transfer` — not in `init_transfer` where tokens are actually locked.

**Step 1 — Token locking in `init_transfer`:**

The only fee validation is `fee.fee < amount`. There is no check that `normalize_amount(amount - fee, decimals) > 0`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

**Step 2 — `normalize_amount` uses floor division:**

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [2](#0-1) 

For a token with `origin_decimals = 24` and `decimals = 6`, `diff_decimals = 18`. Any `amount_without_fee < 10^18` normalizes to 0.

**Step 3 — `sign_transfer` guards against zero but provides no recovery:**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

This guard causes `sign_transfer` to always revert for the affected transfer. The transfer message is never removed, and the locked tokens are never returned.

**Step 4 — No cancel/refund path exists:**

The only code paths that call `remove_transfer_message` are `claim_fee_callback` and `sign_transfer_callback` — both of which require a successful `sign_transfer` execution first. Since `sign_transfer` always reverts for this transfer, neither path is reachable. [4](#0-3) [5](#0-4) 

### Impact Explanation

User funds are permanently frozen in the NEAR bridge contract. The tokens are locked at `init_transfer` time, but `sign_transfer` will always revert with `InvalidAmountToTransfer`, and no cancel or refund mechanism exists. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

This is reachable by any unprivileged bridge user. For tokens with a large decimal difference (e.g., NEAR-native tokens with 24 decimals bridged to EVM chains with 6 decimals, giving `diff_decimals = 18`), any transfer where `amount - fee < 10^18` triggers the freeze. A user transferring a small amount (e.g., `amount = 2`, `fee = 1`) with no knowledge of the normalization threshold will permanently lose their funds. The protocol provides no warning or minimum-amount enforcement at deposit time.

### Recommendation

Add a validation in `init_transfer` (or in `init_transfer_internal`) that `normalize_amount(amount_without_fee, decimals) > 0` before accepting and locking the user's tokens:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This check should be performed at the point where the `TransferMessage` is first stored and tokens are locked, not only at signing time.

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 6` (diff = 18).
2. Alice calls `ft_transfer_call` on the token contract, sending `amount = 2` with `fee = 1` and a valid EVM recipient.
3. `init_transfer` accepts the transfer: `fee (1) < amount (2)` passes. Alice's 2 tokens are locked.
4. A relayer calls `sign_transfer` for Alice's transfer.
5. `normalize_amount(amount_without_fee=1, {origin_decimals:24, decimals:6})` = `1 / 10^18 = 0`.
6. `require!(amount_to_transfer > 0, ...)` panics — `sign_transfer` reverts.
7. The transfer message remains in storage. No cancel function exists. Alice's 2 tokens are permanently frozen. [6](#0-5) [2](#0-1)
