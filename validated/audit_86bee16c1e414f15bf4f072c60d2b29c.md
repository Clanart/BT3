### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS program does not validate that the new class hash provided by a contract corresponds to a previously declared class. An unprivileged contract caller can invoke `replace_class` with an arbitrary, undeclared felt value as the class hash. The OS accepts this state transition without complaint, committing an invalid class hash to the contract's state entry. Any subsequent call to that contract will fail irrecoverably at the OS level because no compiled class can be found for the bogus hash, permanently freezing any funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested new class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash is a known, declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing guard. The `class_hash` value (`request.class_hash`) is fully attacker-controlled — it is whatever felt the calling contract places in the syscall request. No lookup into `contract_class_changes` or `compiled_class_facts_bundle` is performed to confirm the hash is declared.

When a future transaction later attempts to call this contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

Because the stored class hash is undeclared, `dict_read` returns 0 (the default), and `find_element` with key `0` will not find a matching entry in the compiled class facts bundle, causing the OS execution to fail. The contract is permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is overwritten with an undeclared value, the contract can never be called again within a valid OS execution. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are permanently inaccessible. There is no recovery path: `replace_class` cannot be called again because calling the contract itself fails, and no other mechanism exists to reset the class hash from outside the contract.

---

### Likelihood Explanation

**Medium-High.** The attack requires only that an attacker control a contract (trivially achieved by deploying one) and that the contract holds funds belonging to other users. A realistic scenario: a malicious developer deploys a contract that appears to be a legitimate vault or token, attracts user deposits, then calls `replace_class(0xdeadbeef)` in a single transaction. The OS accepts the state transition unconditionally. The funds are frozen in the same block. There is no guardian or time-delay mechanism to intervene — the analog to the AdminVoting bug is exact: the missing check means the harmful action and its irreversible consequence occur atomically, with no window for cancellation.

---

### Recommendation

Inside `execute_replace_class`, before writing the new `StateEntry`, verify that `class_hash` is a declared class by confirming that `contract_class_changes` contains a non-zero compiled class hash for it:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);  // Revert if class is not declared.
```

This mirrors the lookup already performed in `execute_entry_point` and closes the gap identified by the TODO comment.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts deposits and exposes a `drain()` entry point that internally calls `replace_class(0x1)` (an undeclared hash).
2. Users deposit funds into `MaliciousVault`.
3. Attacker submits an invoke transaction calling `drain()`.
4. Inside the OS, `execute_replace_class` is reached. [1](#0-0) 
5. No validation of `class_hash = 0x1` is performed (the TODO guard is absent). The `dict_update` writes `StateEntry(class_hash=0x1, ...)` for `MaliciousVault`'s address.
6. The OS proof is generated and accepted on L1. The committed state now records `class_hash = 0x1` for `MaliciousVault`.
7. Any subsequent transaction targeting `MaliciousVault` reaches `execute_entry_point`, which calls `dict_read` on `contract_class_changes` with key `0x1`, gets `compiled_class_hash = 0`, then calls `find_element` with key `0`. [2](#0-1) 
8. `find_element` fails to locate a compiled class fact with hash `0`, causing OS execution to abort. The contract is permanently uncallable and all deposited funds are frozen.
