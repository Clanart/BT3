### Title
Unbounded Stale L1 Gas Price Propagation via Unchecked `previous_block_info` Fallback — (File: `crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

`get_l1_prices_in_fri_and_wei_and_conversion_rate()` has a three-tier fallback: (1) live L1 scraper + oracle, (2) `previous_block_info`, (3) hardcoded defaults. When the L1 scraper stops updating, `get_price_info()` correctly fires `StaleL1GasPricesError`. The code then silently falls back to `previous_block_info` **with no check on how old that data is**. Because `previous_block_info` is refreshed every committed block from the committed block's own prices, stale prices self-replicate block-by-block for the entire duration of the outage. Both proposer and validator execute the same fallback path, so the `within_margin` check in `is_block_info_valid()` passes, and every block committed during the outage carries wrong L1 gas prices that are hashed into the block header commitment.

---

### Finding Description

**Root cause — `utils.rs` lines 183–208:**

```rust
// One or both (oracle/provider) have failed to fetch ...
if let Some(block_info) = previous_block_info {
    let prev_l1_gas_price_wei = block_info.l1_prices_wei.clone();
    let prev_l1_gas_price = block_info.l1_prices_fri.clone();
    let eth_to_fri_rate = calculate_eth_to_fri_rate(block_info);
    match eth_to_fri_rate {
        Ok(eth_to_fri_rate) => {
            return (prev_l1_gas_price, prev_l1_gas_price_wei, eth_to_fri_rate);
        }
        ...
    }
}
```

There is no check of the form `if current_timestamp - block_info.timestamp > MAX_FALLBACK_AGE_SECONDS`. The `PreviousBlockInfo` struct carries a `timestamp` field but it is never compared against the current wall-clock time in this path.

**Staleness detection exists only at the provider level, not at the fallback level.** `get_price_info()` checks:

```rust
if timestamp.0 > (*last_timestamp + self.config.max_time_gap_seconds) {
    return Err(L1GasPriceProviderError::StaleL1GasPricesError { ... });
}
```

This correctly detects that the scraper has stopped. But the error is caught by the caller and silently replaced with `previous_block_info`, which itself carries the last-committed block's prices — prices that were already stale when that block was built.

**Self-replication chain:**

| Block | `previous_block_info` source | Prices used |
|-------|------------------------------|-------------|
| N (last good) | block N-1 (live scraper) | correct |
| N+1 | block N (fallback) | stale by 1 block |
| N+2 | block N+1 (fallback) | stale by 2 blocks |
| … | … | unboundedly stale |

**Validation bypass:** `is_block_info_valid()` in `validate_proposal.rs` calls the same `get_l1_prices_in_fri_and_wei()` function with the same `previous_block_info`. Both proposer and validator therefore compute the identical stale price, so `within_margin()` always returns `true` and the proposal is accepted.

**Commitment path:** The accepted prices flow into `ProposalInit` fields `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`, which are then passed to `gas_prices_to_hash()` in `block_hash_calculator.rs` and hashed into the block header commitment:

```rust
HashChain::new()
    .chain(&STARKNET_GAS_PRICES0)
    .chain(&l1_gas_price.price_in_wei.0.into())
    .chain(&l1_gas_price.price_in_fri.0.into())
    .chain(&l1_data_gas_price.price_in_wei.0.into())
    .chain(&l1_data_gas_price.price_in_fri.0.into())
    ...
    .get_poseidon_hash()
```

---

### Impact Explanation

Every transaction executed during the outage has its fee computed against the wrong `l1_gas_price` and `l1_data_gas_price`. If L1 gas prices spike during the outage, users are systematically undercharged; if they drop, users are overcharged. The wrong prices are permanently committed into the block hash, making the state diff and all downstream proof inputs (SNOS, transaction prover) reflect incorrect fee accounting. This matches **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** (Critical).

---

### Likelihood Explanation

The L1 gas price scraper can stop updating due to L1 node unavailability, network partition, or scraper bugs — all realistic operational events. The `max_time_gap_seconds` threshold is the only guard, and once it fires the fallback takes over silently with no operator-visible error beyond a `warn!` log line. The outage can persist for an arbitrary number of blocks before the scraper recovers.

---

### Recommendation

Add an age check before accepting `previous_block_info` as a fallback. For example:

```rust
if let Some(block_info) = previous_block_info {
    // Reject if the previous block's prices are themselves too old.
    if timestamp.saturating_sub(block_info.timestamp) > MAX_FALLBACK_AGE_SECONDS {
        warn!("previous_block_info is too old ({} seconds), using defaults",
              timestamp.saturating_sub(block_info.timestamp));
    } else {
        let eth_to_fri_rate = calculate_eth_to_fri_rate(block_info);
        ...
    }
}
```

`MAX_FALLBACK_AGE_SECONDS` should be set to a small multiple of the expected block time so that a brief scraper hiccup is tolerated but a prolonged outage forces the system to use the hardcoded minimum prices (which are at least honest about being defaults) rather than silently propagating arbitrarily old market prices.

---

### Proof of Concept

1. L1 gas price scraper stops (e.g., L1 RPC node goes offline).
2. After `max_time_gap_seconds`, `get_price_info(timestamp)` returns `StaleL1GasPricesError`.
3. `get_l1_prices_in_fri_and_wei_and_conversion_rate()` enters the `previous_block_info` branch at line 185 of `utils.rs` — no age check is performed.
4. Proposer builds block N+1 with `l1_gas_price_fri` = value from block N's header.
5. Validator calls the same function with the same `previous_block_info`; computes the identical stale price; `within_margin()` at line 302 of `validate_proposal.rs` returns `true`; proposal is accepted.
6. `decision_reached()` updates `previous_block_info` to block N+1's prices (which are the same stale values).
7. Block N+2 repeats steps 3–6 with prices now two blocks stale.
8. After the scraper recovers, all blocks N+1 … N+k have permanently committed wrong `l1_gas_price_{fri,wei}` and `l1_data_gas_price_{fri,wei}` values in their block hashes, with all transaction fees computed against those wrong prices.
