### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key Validation, Silently Corrupting Tree Root and Proof State - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for all entries beyond the first two when the tree already has ≥ 2 leaves. Supplying a batch that contains a repeated `KeyId` (or repeated `Hash`) silently writes multiple leaf nodes with the same key into the blob, overwrites the `BlockStatusCache` entry for that key, and produces a root hash that is computed over "orphaned" leaves that can never be proven via the normal proof API. The resulting `MerkleBlob` is internally inconsistent: its root hash commits to data that no proof can reach.

---

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `insert` only for the first two items when the tree has ≤ 1 leaf (lines 578–585). For every subsequent item it calls `insert_entry_to_blob` directly (lines 587–603), which bypasses both guards entirely:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no dup check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` and silently overwrites any existing entry for the same `KeyId` or `Hash`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
```

After a batch with a repeated key `K`:
- The blob contains **two** `LeafNode` blocks with `key == K`.
- `block_status_cache.key_to_index[K]` points only to the **last** inserted leaf.
- The first leaf is "orphaned": it is present in the blob, contributes to the Merkle root hash computation, but is unreachable through any public API (`get_proof_of_inclusion`, `get_keys_values`, `get_key_index`).
- The root hash therefore commits to a leaf that no proof can cover, making every subsequent proof of inclusion or exclusion against that root potentially invalid.

---

### Impact Explanation

The corrupted `MerkleBlob` has a root hash that is computed over orphaned leaves. Any consumer that:
1. Stores the root hash as a commitment to the DataLayer state, and
2. Later verifies proofs of inclusion or exclusion against that root,

will accept proofs that are structurally valid against the root but do not reflect the true set of key-value pairs. Specifically:
- A proof of **exclusion** for key `K` can be generated (because the cache says `K` is present, so exclusion proofs are blocked — but for a *different* key whose orphaned leaf occupies a tree position that shifts the exclusion boundary), or the orphaned leaf can cause `generate_proof` to return an incorrect result.
- The root hash diverges from what `get_keys_values()` returns, breaking the invariant that the root is a cryptographic commitment to the visible key-value set.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

---

### Likelihood Explanation

`batch_insert` is exposed directly as a Python binding (`py_batch_insert`) callable by any Python code that holds a `MerkleBlob`. In the DataLayer sync protocol, a node receives batches of key-value-hash triples from peers and inserts them via `batch_insert`. A malicious peer can craft a batch containing a repeated `KeyId` (or `Hash`). No validation in the Python binding or the Rust fast path prevents this. The attacker needs only network access to a DataLayer node that accepts sync data.

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the start of the fast path in `batch_insert`, before any `insert_entry_to_blob` call. The simplest fix is to check `block_status_cache.contains_key(key)` and `block_status_cache.contains_leaf_hash(&hash)` for each item in the loop (lines 587–603), returning `Err(Error::KeyAlreadyPresent())` or `Err(Error::HashAlreadyPresent())` on violation — mirroring the guards already present in `insert`. Additionally, validate that the batch itself contains no internal duplicates before any mutation of the blob.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(Vec::new()).unwrap();

// Pre-populate so leaf_count > 1 (fast path is taken for subsequent items)
let h0 = Hash(Bytes32::new([0u8; 32]));
let h1 = Hash(Bytes32::new([1u8; 32]));
blob.insert(KeyId(0), ValueId(0), &h0, chia_datalayer::InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &h1, chia_datalayer::InsertLocation::Auto {}).unwrap();

// batch_insert with duplicate KeyId(2) — no error is returned
let h2a = Hash(Bytes32::new([2u8; 32]));
let h2b = Hash(Bytes32::new([3u8; 32]));
blob.batch_insert(vec![
    ((KeyId(2), ValueId(10)), h2a),
    ((KeyId(2), ValueId(20)), h2b),  // duplicate key, different hash
]).unwrap();  // succeeds silently

blob.calculate_lazy_hashes().unwrap();

// Root hash is computed over 4 leaves (including the orphaned first KeyId(2) leaf)
let root = blob.get_root_hash();

// But get_keys_values only returns 3 entries — KeyId(2) appears once
let kv = blob.get_keys_values().unwrap();
assert_eq!(kv.len(), 3);  // orphaned leaf is invisible

// get_proof_of_inclusion returns a proof for the second KeyId(2) leaf only;
// the orphaned first leaf contributes to `root` but cannot be proven.
let proof = blob.get_proof_of_inclusion(KeyId(2)).unwrap();
assert!(proof.valid());  // valid against root, but root is over a corrupted tree

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }

        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
