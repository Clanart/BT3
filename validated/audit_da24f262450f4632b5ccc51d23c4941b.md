### Title
Non-EVM-Denom Bank Balances Orphaned on SELFDESTRUCT Account Deletion — (`x/evm/statedb/statedb.go`)

### Summary

When a contract selfdestructs, `StateDB.Commit()` only burns the EVM-native denom balance and removes the auth account. Non-EVM-denom tokens (IBC, CosmWasm bridge tokens) held by the destroyed contract are explicitly left behind as orphaned bank balances. If the same address is later recreated via CREATE2, the new contract gains access to those orphaned tokens, enabling theft of Cosmos bank funds.

### Finding Description

In `StateDB.Commit()`, the selfdestruct path burns only the EVM denom balance and then calls `DeleteAccount()`:

```go
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
``` [1](#0-0) 

The code itself documents the gap:

> "Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm bridge) held by the destroyed address are not drained and may remain as orphaned bank balances." [2](#0-1) 

`DeleteAccount()` in the keeper removes auth-module metadata and contract storage, but the comment explicitly states it does **not** touch bank balances:

```go
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
    ...
    k.accountKeeper.RemoveAccount(ctx, acct)
    ...
}
``` [3](#0-2) 

The Cosmos SDK bank module stores balances independently of the auth module. After `RemoveAccount`, the bank-module KV entry for non-EVM-denom tokens at the destroyed address persists. When the address is recreated via CREATE2, `SetAccount` calls `k.accountKeeper.NewAccountWithAddress(ctx, cosmosAddr)`, restoring an auth account at the same address: [4](#0-3) 

The new account now has a valid auth entry at the same address, and the bank module will permit it to spend the previously orphaned non-EVM-denom tokens.

This is the direct structural analog to the external report: just as `ShortToken._burn()` checks only `shortAmount == 0` without verifying `collateralAmount == 0`, `StateDB.Commit()` checks only the EVM denom balance without verifying that all bank-module balances are zero before deleting the account.

### Impact Explanation

**Loss-of-funds path (High):** Any contract that holds non-EVM-denom tokens (received via a native precompile such as a bank or IBC precompile using `ExecuteNativeAction`) and then selfdestructs will permanently orphan those tokens. The tokens are stuck at the address with no owner, matching the external report's scenario of collateral permanently locked in a burned position.

**Theft path (Critical):** An attacker who controls a contract at a deterministic CREATE2 address can:
1. Accumulate non-EVM-denom tokens in the contract (e.g., user deposits via a precompile).
2. Selfdestruct the contract (EIP-6780 allows this when created in the same tx).
3. Redeploy the contract at the identical CREATE2 address in a subsequent transaction.
4. The new contract's auth account is restored at the same address; the bank module's orphaned balance is now accessible and can be drained via native precompile calls.

This constitutes unauthorized theft of Cosmos bank funds through Ethermint's stateDB/native action logic.

### Likelihood Explanation

Ethermint explicitly supports stateful native precompiles (e.g., Cronos bank/IBC precompiles) that transfer non-EVM-denom tokens to contract addresses via `ExecuteNativeAction`. Any deployment that uses such precompiles and allows contracts to hold non-EVM-denom balances is exposed. The loss-of-funds path requires only that a contract holding such tokens selfdestructs — a realistic scenario in DeFi protocols with upgrade/migration patterns. The theft path additionally requires CREATE2 redeployment, which is a standard pattern. The codebase already contains integration tests for the CREATE2 selfdestruct recovery scenario for the EVM denom, confirming the attack surface is understood; the non-EVM-denom gap is the unaddressed residual.

### Recommendation

In `StateDB.Commit()`, before calling `DeleteAccount()`, drain **all** bank-module balances at the destroyed address, not just the EVM denom. Use `bankKeeper.GetAllBalances(ctx, cosmosAddr)` to enumerate every denom and burn each coin:

```go
if obj.selfDestructed {
    cosmosAddr := sdk.AccAddress(obj.Address().Bytes())
    cacheCtx, writeCache := s.origCtx.CacheContext()
    // Drain ALL bank balances, not just the EVM denom.
    allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
    if !allBalances.IsZero() {
        if err := s.keeper.BurnAllBalances(cacheCtx, cosmosAddr, allBalances); err != nil {
            return errorsmod.Wrap(err, "failed to burn post-selfdestruct balances")
        }
    }
    if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
        return errorsmod.Wrap(err, "failed to delete account")
    }
    writeCache()
}
```

The `Keeper` interface should be extended with a `GetAllBalances` method, and the burn logic should handle each denom (send to module, then burn, or use a dedicated burn mechanism per denom type).

### Proof of Concept

1. Deploy a factory contract via CREATE2 with a known salt. The factory deploys a child contract at a deterministic address `childAddr`.
2. Via a native bank/IBC precompile (`ExecuteNativeAction`), transfer IBC tokens (denom `ibc/...`) to `childAddr`.
3. In the same transaction, call `SELFDESTRUCT` on the child (EIP-6780 applies; child was created in this tx).
4. `StateDB.Commit()` burns the EVM denom balance at `childAddr` but leaves the IBC token balance in the bank module.
5. `DeleteAccount()` removes the auth account for `childAddr`.
6. In a new transaction, call the factory's `redeployChild(salt)` — CREATE2 redeploys the child at the same `childAddr`.
7. `SetAccount` creates a new auth account at `childAddr`.
8. Call the bank precompile's `transfer` method from `childAddr` to drain the orphaned IBC tokens to the attacker's address.
9. The bank module permits the transfer because `childAddr` now has a valid auth account and the IBC balance is non-zero.

The existing test `TestSelfDestructPostDestructionBalanceBurned` confirms the EVM-denom path is fixed but does not cover non-EVM-denom tokens, leaving this gap undetected. [5](#0-4)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account

// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
```

// TestSelfDestructPostDestructionBalanceBurned verifies that any balance credited to a
// self-destructed address within the same transaction is burned at commit time rather
// than left as an orphaned bank balance recoverable by recreating the address.
func (suite *StateDBTestSuite) TestSelfDestructPostDestructionBalanceBurned() {
	raw, ctx, keeper := setupTestEnv(suite.T())

	// Setup: create a contract account with initial balance and code.
	db := statedb.New(ctx, keeper, emptyTxConfig)
	db.CreateAccount(address)
	db.CreateContract(address)
	db.SetCode(address, []byte("contract code"), 0)
	db.AddBalance(address, uint256.NewInt(100), tracing.BalanceChangeTransfer)
	suite.Require().NoError(db.Commit())

	ctx, keeper = newTestKeeper(suite.T(), raw)

	// Phase 1: Self-destruct the contract; its initial balance (100) must be burned.
	db = statedb.New(ctx, keeper, emptyTxConfig)
	db.SelfDestruct(address)
	suite.Require().True(db.HasSelfDestructed(address))
	suite.Require().Equal(uint256.NewInt(0), db.GetBalance(address))

	// Phase 2: Send value to the already-destroyed address in the same transaction.
	// This simulates a CALL with value to a self-destructed contract.
	postDestructValue := uint256.NewInt(500)
	db.AddBalance(address, postDestructValue, tracing.BalanceChangeTransfer)
	suite.Require().Equal(postDestructValue, db.GetBalance(address))

	suite.Require().NoError(db.Commit())

	// After commit: account metadata must be gone.
	ctx, keeper = newTestKeeper(suite.T(), raw)
	suite.Require().Nil(keeper.GetAccount(ctx, address))

	// The post-destruction balance must be burned (zero), not preserved.
	cosmosAddr := sdk.AccAddress(address.Bytes())
	balance := keeper.GetBalance(ctx, cosmosAddr, "uphoton")
	suite.Require().True(balance.IsZero(), "post-selfdestruct balance must be burned at commit")
}
