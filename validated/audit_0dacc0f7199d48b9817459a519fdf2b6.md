### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary `class_hash` from the caller and commits it directly to contract state without verifying that a compiled class with that hash has ever been declared on-chain. This allows any unprivileged transaction sender to permanently freeze the funds of any contract they can cause to invoke `replace_class` with an undeclared hash.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` (lines 877–916) reads `class_hash` from the raw syscall request and writes it unconditionally into the contract's `StateEntry`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The inline TODO comment is the codebase's own acknowledgement that the check is absent. No range check, no lookup against `contract_class_changes`, and no lookup against the compiled-class-facts bundle is performed before the state is mutated.

After this update is committed, every future call to the contract will attempt to resolve the compiled class for `class_hash`. Because the OS resolves classes through the hint-provided compiled-class-facts bundle (populated by the sequencer from declared classes only), an undeclared hash produces an unresolvable lookup, making the contract permanently uncallable.

The `execute_replace_class` function is dispatched from `execute_syscalls` for any contract that emits the `replace_class` syscall selector — there is no privilege gate.

---

### Impact Explanation

Any ERC-20 balance, NFT, or other asset stored in the contract's storage becomes permanently inaccessible. The class hash is a committed on-chain state field; the StarkNet protocol provides no administrative override or recovery path once it is set to an undeclared value. This satisfies the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The `replace_class` syscall is reachable by any unprivileged transaction sender through a contract they control or can influence:

- A **buggy contract** that computes a class hash off-chain and passes it before declaring the class will self-freeze.
- A **shared contract** (multisig, vault, DAO) with a logic flaw that lets an attacker supply an arbitrary argument to `replace_class` will freeze all depositors' funds in a single transaction.
- The OS provides **zero protocol-level guard**, so the entire burden falls on every contract author to never pass an undeclared hash — a fragile assumption at scale.

---

### Recommendation

Before committing the state update, verify that `class_hash` is non-zero and is present either in `contract_class_changes` (declared in the current block) or in the pre-existing compiled-class-facts bundle. If the hash is not found, write a failure response via `write_failure_response` and return without mutating state, consistent with how other invalid-argument cases are handled throughout `syscall_impls.cairo`.

---

### Proof of Concept

1. Deploy contract `C` that holds user funds and exposes a public function `freeze()`.
2. `freeze()` emits the `replace_class` syscall with `class_hash = 0` (or any felt that has never been declared).
3. The OS executes `execute_replace_class`; the TODO-guarded path writes `class_hash = 0` into `C`'s `StateEntry` without validation.
4. The transaction succeeds; the state is committed on-chain.
5. Any subsequent call to `C` (e.g., a withdrawal) causes the OS to attempt to resolve compiled class `0`, which does not exist in the compiled-class-facts bundle.
6. The call cannot be executed; all funds in `C` are permanently frozen with no recovery path.
