### Title
Missing Class Hash Declaration Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS does not validate that the new class hash corresponds to a declared contract class before updating the contract's state. This is an explicitly acknowledged missing check (TODO comment). Any contract can call `replace_class` with an arbitrary undeclared class hash, permanently bricking itself and freezing any funds it holds. The same flaw exists in the deprecated syscall path.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` (line 878) updates a contract's class hash in `contract_state_changes` without verifying the new hash is declared:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is the explicit acknowledgment that this validation is absent. The function accepts any `class_hash` value from the contract's syscall request and writes it directly into state.

When a future transaction calls the bricked contract, `execute_entry_point` runs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the class hash stored in state was never declared, `dict_read` returns 0 (the Cairo dict default), and `find_element` with key `0` fails to locate any compiled class. Every future call to the contract fails permanently. The identical missing validation exists in the deprecated path: [3](#0-2) 

This is structurally analogous to the external report: a mechanism (`replace_class`) exists and is callable, but the required guard (declaration check) is absent — the OS has no code to enforce it — leaving the resulting state permanently unrecoverable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, every subsequent call to that contract fails at the OS level. There is no recovery path: the OS has no syscall or transaction type that can restore a contract's class hash without executing the contract itself. Any ERC-20 tokens, ETH, or other assets held by the contract are permanently inaccessible.

---

### Likelihood Explanation

**Low-medium.**

The scenario requires a contract to invoke `replace_class` with an undeclared hash. This can occur:

- **Accidentally**: A contract computes a class hash with an off-by-one error or uses a hash before the corresponding `declare` transaction is confirmed.
- **Maliciously (self-inflicted rug)**: A contract is designed to accept deposits and then call `replace_class` with a garbage hash, permanently locking deposited funds.
- **Via a contract vulnerability**: A legitimate contract exposes a `replace_class` call path that an unprivileged attacker can trigger with attacker-controlled calldata.

The `replace_class` transaction itself succeeds and is included in the block. Only subsequent calls to the contract fail, so the sequencer's pre-inclusion simulation does not catch the freeze.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the result is non-zero before proceeding. Apply the same fix to the deprecated `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Deploy contract **C** holding user funds (e.g., an ERC-20 vault).
2. Submit an invoke transaction that causes **C** to call the `replace_class` syscall with an arbitrary undeclared hash `H` (e.g., `0xdeadbeef`).
3. The OS executes `execute_replace_class`: the TODO check is absent, so `contract_state_changes[C].class_hash` is updated to `H` with no validation. [1](#0-0) 
4. The transaction is included in the block and the state root is updated.
5. Any subsequent call to **C** reaches `execute_entry_point`, which does `dict_read(contract_class_changes, H)` → returns `0` (undeclared), then `find_element(..., key=0)` → fails. [4](#0-3) 
6. All calls to **C** permanently revert. All funds in **C** are permanently frozen with no recovery path.
