### Title
Empty `check_is_reverted` Allows Prover to Arbitrarily Skip Invoke Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a complete no-op. It is called in `execute_invoke_function_transaction` to validate the hint-supplied `is_reverted` flag that controls whether a transaction's execution step is run. Because the function imposes zero constraints, a malicious prover can freely set `is_reverted = TRUE` for any valid invoke transaction, causing its execution to be silently skipped while fees are still charged. The resulting proof remains cryptographically valid and is accepted by the L1 verifier, making this a soundness bug in the ZK proof system — not merely a sequencer-level policy issue.

---

### Finding Description

**Root cause — empty validation function:**

`check_is_reverted` in `execution_constraints.cairo` is defined as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

It accepts `is_reverted` as a parameter and immediately returns, adding **no Cairo constraints** whatsoever. [1](#0-0) 

**How `is_reverted` is consumed in `execute_invoke_function_transaction`:**

```cairo
local is_reverted;
%{ IsReverted %}          // hint: prover supplies any felt
check_is_reverted(is_reverted);   // no-op — zero constraints added
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // Execution is skipped entirely.
    ...
}

// Charge fee — called unconditionally, regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

The `is_reverted` value is loaded from the hint `%{ IsReverted %}`. In Cairo's ZK model, hints are non-deterministic prover inputs; the Cairo constraints are the only mechanism that can force a hint to be correct. Because `check_is_reverted` adds no constraints, `is_reverted` is **entirely unconstrained** — the prover may set it to any value.

**Fee is charged unconditionally:**

`charge_fee` is called after the `if/else` block, meaning fees are deducted from the user's account regardless of whether execution was skipped. [3](#0-2) 

**No indirect constraint closes the gap:**

The downstream `state_update` in `os.cairo` commits whatever state changes were recorded during execution. If execution was skipped, no state changes are recorded for that transaction, and `state_update` commits an empty delta — it does not verify that the delta is consistent with the transaction's expected outcome. [4](#0-3) 

**Analogy to the reported BeaconKit bug:**

| BeaconKit | StarkNet OS |
|---|---|
| `createProcessProposalResponse` accepts all proposals when error is non-fatal (no enforcement) | `check_is_reverted` accepts any `is_reverted` value (no enforcement) |
| Invalid block passes into `FinalizeBlock` and is processed | Invalid "reverted" flag passes into fee charging and is accepted |
| All errors wrapped as `WrapNonFatal` | `check_is_reverted` is a no-op |

Both are **validation-bypass** bugs where a guard function that should reject invalid inputs instead silently accepts them.

---

### Impact Explanation

**Direct loss of funds (Critical).**

A malicious prover sets `is_reverted = TRUE` for a valid user invoke transaction. The execution step is skipped — no state changes occur, the user's intended action is not performed — but `charge_fee` still executes an ERC-20 transfer from the user's account to the sequencer. The resulting STARK proof is valid (no Cairo assertion is violated), so the L1 verifier accepts it and the state transition is finalized on Ethereum. The user's funds are permanently lost with no recourse.

This is a **ZK soundness bug**: the proof system is supposed to prevent a malicious prover from producing a valid proof for an invalid state transition. The empty `check_is_reverted` breaks this guarantee.

---

### Likelihood Explanation

The StarkNet sequencer is the prover. If the sequencer is compromised or acts maliciously, it can exploit this bug against any or all users in any block. Because the proof is valid, neither the L1 verifier nor any on-chain mechanism can detect or reject the attack. The attack is silent, scalable (applies to every invoke transaction in a block), and leaves no on-chain evidence distinguishing it from a legitimately reverted transaction.

---

### Recommendation

Implement `check_is_reverted` to add a real Cairo constraint that ties `is_reverted` to a verifiable execution outcome. At minimum, the function must ensure that if `is_reverted = TRUE`, there exists a provable reason (e.g., out-of-gas, failed `__validate__`, entry-point not found) that is itself constrained by the proof. One approach is to require that a reverted transaction's gas consumption or failure code is committed to the output and verified against the execution trace, making it impossible to mark a successful execution as reverted without breaking a constraint.

---

### Proof of Concept

1. User submits a valid invoke transaction (e.g., a token transfer).
2. Malicious sequencer sets hint `%{ IsReverted %}` → `is_reverted = 1` (TRUE).
3. `check_is_reverted(1)` is called — returns immediately, no constraint added.
4. `if (is_reverted == FALSE)` is `FALSE`; the `else` branch is taken; `non_reverting_select_execute_entry_point_func` is **never called**.
5. `charge_fee(...)` is called unconditionally — ERC-20 transfer deducts fee from user.
6. No execution state changes are recorded; `state_update` commits an empty delta for this transaction.
7. The STARK proof is generated and verified on L1 — **proof is valid**, state transition accepted.
8. User's fee is gone; their transaction was never executed. [1](#0-0) [2](#0-1)
