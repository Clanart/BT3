Audit Report

## Title
Public Unauthenticated `sweepToken` and `unwrapWETH9` Allow Any Caller to Drain Router-Held Balances to an Arbitrary Recipient — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`sweepToken` and `unwrapWETH9` in `PeripheryPayments` are `public payable` with zero access control and accept a fully caller-controlled `recipient` address. Any unprivileged external caller can invoke them at any time to redirect the router's entire ERC-20 or WETH balance to themselves. The router accumulates real token balances during multi-hop `exactInput` execution and whenever a user routes a swap with `recipient = address(this)` outside an atomic multicall.

## Finding Description
Both functions are confirmed `public payable` with no `msg.sender` check, role guard, or recipient validation:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L37-55
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // attacker-supplied recipient
    }
}

function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);  // attacker-supplied recipient
    }
}
```

The router accumulates real balances through the confirmed production path in `exactInput`: intermediate hop outputs are sent to `address(this)` (`i == last ? params.recipient : address(this)` at L106), and the final output also lands on the router if `params.recipient == address(this)`. A user who sets `recipient = address(this)` intending to follow up with `unwrapWETH9` in a separate transaction (or whose multicall is partially constructed) leaves tokens on the router. An attacker then calls `sweepToken(token, 0, attacker)` or `unwrapWETH9(0, attacker)` to drain the full balance. The `amountMinimum = 0` argument bypasses the only guard present. No existing check in either function or in the router prevents this.

## Impact Explanation
Direct, complete loss of any ERC-20 or WETH balance held by the router. The stolen amount equals the full router balance of the targeted token — there is no floor, no time lock, and no authentication. This is a direct loss of user principal meeting Critical/High Sherlock thresholds.

## Likelihood Explanation
The intended safe usage is an atomic multicall (swap + sweep/unwrap in one transaction), but this is not enforced on-chain. Users who submit a swap with `recipient = address(this)` as a standalone transaction, or who compose a multicall that omits the cleanup step due to a UI bug or manual construction error, leave funds on the router. MEV bots routinely monitor router balances and can back-run any such transaction in the same block. The attack requires no special privilege, no capital, and is repeatable indefinitely.

## Recommendation
Make `sweepToken` and `unwrapWETH9` internal and expose them only through public wrappers that hard-bind `recipient` to `msg.sender`:

```solidity
function sweepToken(address token, uint256 amountMinimum) public payable {
    _sweepToken(token, amountMinimum, msg.sender);
}

function unwrapWETH9(uint256 amountMinimum) public payable {
    _unwrapWETH9(amountMinimum, msg.sender);
}
```

This preserves the intended cleanup functionality while ensuring only the address that initiated the multicall can direct where swept value goes.

## Proof of Concept
1. Deploy router with valid WETH and factory.
2. Alice calls `exactInput` with `recipient = address(router)` as a standalone transaction (intending to call `unwrapWETH9` next). The final hop output (WETH) lands on the router at L106.
3. Attacker observes the pending or confirmed transaction and calls `router.unwrapWETH9(0, attacker)`.
4. `balanceWETH = router's full WETH balance`; WETH is withdrawn and ETH is sent to attacker. Alice receives nothing.

Foundry test outline:
```solidity
vm.prank(alice);
router.exactInput(params); // params.recipient = address(router)
// router now holds WETH output

vm.prank(attacker);
router.unwrapWETH9(0, attacker); // drains entire balance to attacker
assertEq(attacker.balance, swapOutput);
```
