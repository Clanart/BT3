### Title
`descendant_of_tracked_shard_cache` Keyed Only by `ShardId` Returns Stale Shard-Tracking Decisions Across Epoch Boundaries During Resharding — (`chain/epoch-manager/src/shard_tracker.rs`)

### Summary

`ShardTracker::check_if_descendant_of_tracked_shard` caches its result in a `HashMap<ShardId, bool>` keyed solely by the numeric `ShardId`. The correct answer depends on both `ShardId` **and** `EpochId`, because the same numeric `ShardId` value appears in both pre- and post-resharding shard layouts (with different `ShardUId` versions). Once the cache is populated for one epoch, every subsequent query for the same `ShardId` — regardless of which epoch is being asked about — returns the stale cached value. This causes `should_catch_up_shard` to skip mandatory state sync, leaving a validator without the trie state it needs to produce or validate chunks in the next epoch.

### Finding Description

`ShardTracker` holds two caches for the `TrackedShardsConfig::Shards` variant:

```
tracked_accounts_shard_cache: Arc<SyncLruCache<EpochId, BitMask>>   // keyed by EpochId ✓
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<ShardId, bool>>> // keyed by ShardId only ✗
``` [1](#0-0) 

The lookup in `check_if_descendant_of_tracked_shard` ignores the `epoch_id` argument entirely when a cache entry already exists:

```rust
pub fn check_if_descendant_of_tracked_shard(
    &self,
    shard_id: ShardId,
    tracked_shards: &Vec<ShardUId>,
    epoch_id: &EpochId,          // ← passed in but never used for cache lookup
) -> Result<bool, EpochError> {
    if let Some(is_tracked) = self.descendant_of_tracked_shard_cache.lock().get(&shard_id) {
        return Ok(*is_tracked);  // ← returns stale value for a different epoch
    }
    ...
    self.descendant_of_tracked_shard_cache.lock().insert(shard_id, is_tracked);
    Ok(is_tracked)
}
``` [2](#0-1) 

The underlying implementation `check_if_descendant_of_tracked_shard_impl` correctly uses `epoch_id` to resolve the shard layout and trace ancestry: [3](#0-2) 

**Concrete trigger sequence** (node configured with `TrackedShardsConfig::Shards([ShardUId { shard_id: 1, version: 1 }])`, i.e., tracking the post-resharding version of shard 1):

At the last block of pre-resharding epoch T, `should_catch_up_shard` is called for `ShardId(1)`:

```rust
fn should_catch_up_shard(&self, prev_hash: &CryptoHash, shard_id: ShardId) -> bool {
    if !self.will_care_about_shard(prev_hash, shard_id) { ... }  // Step 1: queries epoch T+1
    if self.cares_about_shard(prev_hash, shard_id) { return false; } // Step 2: queries epoch T
    let tracked_before = self.cared_about_shard_in_prev_epoch_from_prev_hash(...); // Step 3
    !tracked_before
}
``` [4](#0-3) 

- **Step 1** (`will_care_about_shard` → epoch T+1): `ShardId(1, version=1)` is directly in `tracked_shards`. `check_if_descendant_of_tracked_shard_impl` returns `true`. Cache is written: `{ShardId(1): true}`.
- **Step 2** (`cares_about_shard` → epoch T): `ShardId(1)` exists in epoch T's layout as `ShardId(1, version=0)` — a completely different shard, not a descendant of the tracked post-resharding shard. The correct answer is `false`. But the cache returns `true` (stale hit from Step 1).
- **Result**: `cares_about_shard` returns `true` → `should_catch_up_shard` returns `false` → **state sync is skipped**.

The node enters epoch T+1 without the trie state for `ShardId(1, version=1)` and cannot apply or validate chunks for that shard.

The same stale cache also corrupts `should_apply_chunk`: [5](#0-4) 

and the resharding manager's child-shard filter: [6](#0-5) 

### Impact Explanation

A validator node configured with `TrackedShardsConfig::Shards` that crosses a resharding epoch boundary silently skips mandatory state sync for one or more shards. Without the correct trie state the node cannot apply chunks (`apply_new_chunk` / `apply_old_chunk`), cannot produce chunk state witnesses, and cannot endorse chunks. This results in missed chunk production, missed endorsements, and eventual validator kickout. The corrupted cache entry persists for the lifetime of the process because `descendant_of_tracked_shard_cache` is never cleared or invalidated.

### Likelihood Explanation

`TrackedShardsConfig::Shards` is a documented, production-supported configuration used by archival nodes and validators that track a fixed subset of shards. Resharding (shard splits) is an active protocol feature with multiple integration tests already in the repository. The bug fires automatically at the first epoch boundary after a resharding event — no adversarial input is required. The only prerequisite is that the same numeric `ShardId` value exists in both the pre- and post-resharding shard layouts, which is guaranteed by NEAR's shard layout derivation (non-split shards retain their `ShardId` with an incremented version).

### Recommendation

Change the cache key from `ShardId` to `(ShardId, EpochId)`, matching the pattern already used by `tracked_accounts_shard_cache`:

```rust
// Before
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<ShardId, bool>>>

// After
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<(ShardId, EpochId), bool>>>
```

Update `check_if_descendant_of_tracked_shard` to use `(shard_id, *epoch_id)` as the lookup and insertion key. This is consistent with how `tracked_accounts_shard_cache` already scopes its bitmask per `EpochId`. [7](#0-6) 

### Proof of Concept

```
Setup:
  - Two-shard pre-resharding layout: ShardId(0,v0), ShardId(1,v0)
  - Post-resharding layout (ShardId(0,v0) splits): ShardId(0,v1), ShardId(1,v1), ShardId(2,v1)
  - Node config: TrackedShardsConfig::Shards([ShardUId { shard_id: 1, version: 1 }])
    (tracking the post-resharding version of shard 1)

At the last block of pre-resharding epoch T:

1. should_catch_up_shard(prev_hash, ShardId(1)) is called.

2. will_care_about_shard → tracks_shard_at_epoch(ShardId(1), epoch_T+1):
   - shard_layout for T+1 contains ShardId(1,v1) ✓
   - check_if_descendant_of_tracked_shard_impl: ShardId(1,v1) ∈ tracked_shards → true
   - Cache written: {ShardId(1): true}

3. cares_about_shard → tracks_shard_at_epoch(ShardId(1), epoch_T):
   - shard_layout for T contains ShardId(1,v0) ✓ (exists, different shard)
   - check_if_descendant_of_tracked_shard: cache hit → returns true  ← BUG
   - Correct answer: ShardId(1,v0) is NOT a descendant of ShardId(1,v1) → false

4. should_catch_up_shard returns false (no state sync).

5. Node enters epoch T+1 without trie state for ShardId(1,v1).
   apply_new_chunk fails; chunk production and endorsement impossible.
```

        &self,
        mode: ApplyChunksMode,
        prev_hash: &CryptoHash,
        shard_id: ShardId,
    ) -> bool {
        let cares_about_shard_this_epoch = self.cares_about_shard(prev_hash, shard_id);
        let cares_about_shard_next_epoch = self.will_care_about_shard(prev_hash, shard_id);
        let cared_about_shard_prev_epoch =
            self.cared_about_shard_in_prev_epoch_from_prev_hash(prev_hash, shard_id);
        match mode {
            // next epoch's shard states are not ready, only update this epoch's shards plus shards we will care about in the future
            // and already have state for
            ApplyChunksMode::NotCaughtUp => {
                cares_about_shard_this_epoch
                    || (cares_about_shard_next_epoch && cared_about_shard_prev_epoch)
            }
            // update both this epoch and next epoch
            ApplyChunksMode::IsCaughtUp => {
                cares_about_shard_this_epoch || cares_about_shard_next_epoch
            }
            // catching up next epoch's shard states, do not update this epoch's shard state
            // since it has already been updated through ApplyChunksMode::NotCaughtUp
            ApplyChunksMode::CatchingUp => {
                let syncing_shard = !cares_about_shard_this_epoch
                    && cares_about_shard_next_epoch
                    && !cared_about_shard_prev_epoch;
                syncing_shard
            }
        }
    }

    fn should_catch_up_shard(&self, prev_hash: &CryptoHash, shard_id: ShardId) -> bool {
        // Won't care about it next epoch, no need to state sync it.
        if !self.will_care_about_shard(prev_hash, shard_id) {
            return false;
        }
        // Currently tracking the shard, so no need to state sync it.
        if self.cares_about_shard(prev_hash, shard_id) {
            return false;
        }

        // Now we need to state sync it unless we were tracking the parent in the previous epoch,
        // in which case we don't need to because we already have the state, and can just continue applying chunks

        let tracked_before =
            self.cared_about_shard_in_prev_epoch_from_prev_hash(prev_hash, shard_id);
        !tracked_before
    }

fn check_if_descendant_of_tracked_shard_impl(
    shard_id: ShardId,
    tracked_shards: &Vec<ShardUId>,
    epoch_id: &EpochId,
    epoch_manager: &Arc<dyn EpochManagerAdapter>,
) -> Result<bool, EpochError> {
    let tracked_shards: HashSet<ShardUId> = tracked_shards.into_iter().cloned().collect();
    let protocol_version = epoch_manager.get_epoch_protocol_version(epoch_id)?;
    let shard_layout = epoch_manager.get_shard_layout(&epoch_id)?;

    // `ShardLayoutV3` stores all ancestor shards, no need to iterate through protocol versions
    if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
        let ancestors = HashSet::from_iter(ancestors);
        return Ok(!ancestors.is_disjoint(&tracked_shards));
    }

    let mut shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        // We explicitly track `shard_id` (the shard is a descendant of itself).
        return Ok(true);
    }

    // `shard_uid` does not belong to `tracked_shards`, but it might be a descendant of one.
    // Iterate through consecutive pairs of historical shard layouts (newest to oldest) to trace
    // the ancestry. Each pair represents a resharding transition.
    let layout_history = epoch_manager.get_shard_layout_history(protocol_version, None);
    for window in layout_history.windows(2) {
        let current_layout = &window[0];
        let prev_layout = &window[1];
        let Some(parent_shard_id) = current_layout.try_get_parent_shard_id(shard_uid.shard_id())?
        else {
            debug_assert!(
                false,
                "Parent shard is missing for shard {} in shard layout {:?}",
                shard_uid, current_layout,
            );
            return Ok(false);
        };
        shard_uid = ShardUId::from_shard_id_and_layout(parent_shard_id, &prev_layout);
        if tracked_shards.contains(&shard_uid) {
            return Ok(true);
        }
    }
    Ok(false)
}
