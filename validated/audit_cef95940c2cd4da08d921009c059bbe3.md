Audit Report

## Title
Stranded Router ETH Consumed by Any WETH Swap, Enabling Theft of User Funds - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary

The `pay()` function in `PeripheryPayments.sol` settles WETH swap payments using `address(this).balance` — the router's total native ETH balance — without verifying that the ETH was deposited by the current payer. ETH stranded on the router from a prior transaction (e.g., excess `msg.value` from a partial fill with no `refundETH()` call) is silently consumed by the next WETH swap, transferring value from the original depositor to the attacker at zero cost.

## Finding Description

In `pay()`, when `token == WETH` and `payer != address(this)`, the function branches on `address(this).balance` rather than pulling from `payer`:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
  uint256 nativeBalance = address(this).balance;
  if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
  } else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
  } else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
  }
}
```

The `payer` argument is only consulted when the router holds zero ETH. If the router holds any ETH, it is used unconditionally regardless of who deposited it.

ETH accumulates on the router whenever a user sends `msg.value` via a payable entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) and does not include a `refundETH()` call in the same transaction. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers; it does not prevent ETH from arriving via `msg.value` on payable functions. None of the swap entry points automatically refund excess ETH.

The `multicall` entry point uses `delegatecall`, so `msg.value` is shared across all sub-calls in a bundle, but `refundETH()` is a separate, optional call that users frequently omit.

The callback path (`metricOmmSwapCallback`) validates only that `msg.sender` is the expected pool via `_requireExpectedCallbackCaller`; it does not validate that the ETH used to settle the payment was deposited by the current payer. An attacker who initiates any WETH swap while the router holds ETH will have their payment settled from that balance.

## Impact Explanation

Direct loss of user ETH principal. A victim who sends ETH to the router for a WETH swap and omits `refundETH()` loses their unspent ETH to the next party who calls any WETH swap through the router. The attacker receives the full swap output without paying the corresponding input cost. This is a Critical/High direct loss of user principal.

## Likelihood Explanation

Medium. The precondition — ETH stranded on the router — is a realistic and documented user error in Uniswap v3-style periphery. Users calling `exactInputSingle` with `msg.value` for WETH swaps and omitting `refundETH()` are the victim class. An attacker can monitor the router's ETH balance on-chain and immediately exploit any nonzero balance with a single `exactInputSingle` call sending 0 ETH, requiring no special privileges or setup.

## Recommendation

In `pay()`, when `token == WETH` and `payer != address(this)`, do not use `address(this).balance` to settle the payment. Reserve the native-ETH wrapping path exclusively for `payer == address(this)` (i.e., the router is paying from its own balance as an intermediate hop). For external payers, always pull WETH directly via `safeTransferFrom`. Alternatively, track per-user ETH deposits in transient storage and only allow each user's swap to consume their own deposited ETH.

## Proof of Concept

1. Victim calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 100`, sends `msg.value = 100 ETH`. The swap partially fills (price limit hit), consuming only 80 ETH. The victim does not include `refundETH()`. 20 ETH remains on the router.
2. Attacker calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 20`, sends `msg.value = 0`.
3. The pool calls `metricOmmSwapCallback` on the router. The router calls `pay(WETH, attacker, pool, 20)`.
4. Inside `pay()`: `address(this).balance == 20 >= 20`, so the router wraps the victim's 20 ETH and transfers WETH to the pool. The `payer` (attacker) is never consulted.
5. The attacker receives the swap output without spending any ETH or WETH.
6. The victim's 20 ETH is permanently lost.
