### Title
`starknet_estimateFee` / `starknet_simulateTransactions` omit `l2_gas_consumed` from `FeeEstimation`, causing authoritative-looking wrong `overall_fee` description and forcing hardcoded L2 gas bounds - (File: crates/apollo_rpc_execution/src/objects.rs)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but deliberately omits `l2_gas_consumed`. The `overall_fee` field is correctly computed internally (it includes L2 gas cost via `GasVector::cost()`), but the OpenRPC schema describes `overall_fee` as equalling only `gas_consumed*gas_price + data_gas_consumed*data_gas_price` — a formula that is wrong for any V3 transaction that consumes L2 gas. Clients who rely on the schema formula to verify or reconstruct the fee receive a systematically lower value than the actual fee charged. Clients who need to set `l2_gas.max_amount` in V3 resource bounds have no way to obtain the correct consumed amount from the fee estimation RPC, and are forced to use hardcoded safe upper bounds — the exact pattern the external report identifies as dangerous.

### Finding Description

**Root cause — missing field in `tx_execution_output_to_fee_estimation`:** [1](#0-0) 

The function reads `gas_vector.l1_gas` and `gas_vector.l1_data_gas` into the response but silently drops `gas_vector.l2_gas`. The `l2_gas_price` field is populated, creating an asymmetric response: price without quantity.

**The `FeeEstimation` struct and its acknowledged TODO:** [2](#0-1) 

The inline comment `// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.` confirms the developers know the field is missing and that the rounding relationship between the two is unverified.

**The schema description of `overall_fee` is wrong for V3 transactions:** [3](#0-2) 

The schema states `overall_fee` equals `gas_consumed*gas_price + data_gas_consumed*data_gas_price`. For any V3 transaction with non-zero L2 gas, the actual value returned is `l1_gas*l1_gas_price + l1_data_gas*l1_data_gas_price + l2_gas*l2_gas_price` (computed correctly via `GasVector::cost()`): [4](#0-3) 

The discrepancy between the schema formula and the actual value is exactly `l2_gas_consumed × l2_gas_price`, which for a typical Cairo 1 transaction (e.g., `l2_gas_consumed = 0xb56b6`, `l2_gas_price = 0x1dcd65000` as seen in test fixtures) is a substantial fraction of the total fee.

**Hardcoded upper-bound workaround documented in the prover:** [5](#0-4) 

Because `starknet_estimateFee` does not return `l2_gas_consumed`, the codebase itself documents that callers must either use the value from `starknet_estimateFee` (which does not expose it) or fall back to the hardcoded safe upper bound `100000000` — the direct Starknet analog of the hardcoded `gas: 500000` / `gas: 8000000` values in the external report.

### Impact Explanation

Any client calling `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction:

1. **Receives an authoritative-looking wrong formula**: the schema says `overall_fee = gas_consumed*gas_price + data_gas_consumed*data_gas_price`. For V3 transactions the actual charged fee is higher by `l2_gas_consumed × l2_gas_price`. A client that reconstructs the fee from the returned fields will compute a value lower than what the sequencer will actually charge.

2. **Cannot set correct `l2_gas.max_amount` resource bounds**: without `l2_gas_consumed` in the response, clients must either guess or use a hardcoded ceiling. Setting it too low causes the transaction to revert with `Out of gas`; setting it to the hardcoded `100_000_000` wastes resources and may cause pre-validation to reject the transaction if the implied fee exceeds the account balance.

This matches the impact category: **High — RPC fee estimation returns an authoritative-looking wrong value**.

### Likelihood Explanation

Every V3 (`INVOKE`/`DECLARE`/`DEPLOY_ACCOUNT`) transaction targeting a Cairo 1 contract consumes L2 gas. The Starknet network has been migrating to V3 transactions as the default. Any wallet, SDK, or dApp that calls `starknet_estimateFee` and uses the returned fields to construct resource bounds for a V3 transaction is affected. The trigger requires no special privilege — it is the normal fee-estimation flow.

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`, resolving the existing TODO.
2. Update the `FEE_ESTIMATE` OpenRPC schema description of `overall_fee` to include the L2 gas term: `gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*l2_gas_price`.
3. Remove the hardcoded `100_000_000` fallback in `virtual_snos_prover.rs` once the field is available from the RPC.

### Proof of Concept

1. Submit a V3 `INVOKE` transaction targeting any Cairo 1 contract to `starknet_estimateFee`.
2. Observe the response: `l2_gas_price` is non-zero, but `l2_gas_consumed` is absent.
3. Compute `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` using the returned fields.
4. Compare to `overall_fee`: the difference equals `l2_gas_consumed × l2_gas_price` (visible in test fixture: `l2_gas_consumed = 0xb56b6`, `l2_gas_price = 0x1dcd65000`, `overall_fee = 0x151eb86f3ed400`). [6](#0-5) 

The fixture confirms `l2_gas_consumed = 0xb56b6` and `overall_fee = 0x151eb86f3ed400`, but the `FeeEstimation` struct returned by the Apollo RPC would expose only `gas_consumed = 0x0`, `data_gas_consumed = 0x80`, and `l2_gas_price = 0x1dcd65000` — with no `l2_gas_consumed` field — making it impossible for a client to verify or reproduce the `overall_fee` from the schema formula.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```

```rust
    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
```
