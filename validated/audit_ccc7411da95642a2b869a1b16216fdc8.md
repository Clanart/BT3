### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Allows Undercollateralized Minting - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` records and emits the caller-supplied `amount` parameter rather than the actual tokens received by the contract. For fee-on-transfer ERC20 tokens registered in the bridge, the contract receives fewer tokens than `amount`, but the emitted `InitTransfer` event carries the full `amount`. The NEAR side then mints or releases the full `amount` to the recipient, creating unbacked supply and breaking bridge collateralization.

### Finding Description
In `OmniBridge.initTransfer`, when the token is a plain ERC20 (not a bridge token and not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // user-supplied
);
```

No balance snapshot is taken before or after the transfer. The function then emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // user-supplied, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
```

The NEAR bridge listens to this event and uses the emitted `amount` to determine how many tokens to mint or release on the destination chain. If the ERC20 token deducts a transfer fee, the bridge contract receives `amount - transferFee` but the event records `amount`. The NEAR side then finalizes a transfer of the full `amount`, minting more tokens than are backed by the EVM vault.

This is the direct analog of the external report's vulnerability class: a balance variable (here, the emitted bridged amount) is set from a caller-supplied input rather than from the actual post-transfer balance, causing the accounting to diverge from reality.

### Impact Explanation
Any fee-on-transfer ERC20 token that is registered in the bridge (via `ethToNearToken`/`nearToEthToken` mappings) is affected. For each such transfer:

- EVM vault receives: `amount - transferFee`
- NEAR mints/releases: `amount`

The shortfall (`transferFee`) is unbacked. Repeated transfers accumulate the deficit. When users later bridge back from NEAR to EVM, the vault cannot cover all redemptions, permanently breaking collateralization for that token. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation
Any unprivileged bridge user can trigger this by calling `initTransfer` with a fee-on-transfer token that has been registered in the bridge. No privileged access is required beyond the token being registered. Fee-on-transfer tokens (e.g., tokens with auto-staking or deflationary mechanics) are a well-known token class and are plausible candidates for cross-chain bridging.

### Recommendation
Record the actual balance change rather than trusting the caller-supplied `amount`:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    amount = uint128(balanceAfter - balanceBefore);
}
```

Use the adjusted `amount` in the subsequent `initTransferExtension` call and `InitTransfer` event emission so the NEAR side only mints what was actually received.

### Proof of Concept

1. Admin registers a fee-on-transfer ERC20 token `FeeToken` (5% transfer fee) in the bridge via `ethToNearToken`/`nearToEthToken`.
2. User calls `initTransfer(FeeToken, 1000, 0, 0, "near-recipient", "")`.
3. `safeTransferFrom` moves 1000 tokens from user; `FeeToken` deducts 5%, so bridge receives 950.
4. `InitTransfer` event is emitted with `amount = 1000`.
5. NEAR relayer observes the event and calls `fin_transfer` on NEAR with `amount = 1000`.
6. NEAR bridge mints 1000 tokens to `near-recipient`.
7. Bridge vault holds 950 tokens but has issued 1000 — 50 tokens of unbacked supply per transfer.
8. Repeated transfers drain the vault's backing ratio until redemptions from NEAR to EVM fail for lack of funds. [1](#0-0) [2](#0-1) [3](#0-2)
