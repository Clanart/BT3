Audit Report

## Title
Unguarded Shared ETH Balance in `pay()` Lets Any Caller Drain Stranded Native ETH from the Router — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` unconditionally consumes the router's global `address(this).balance` to settle WETH-input swaps without verifying that the current caller contributed that ETH. Any ETH left in the router from a prior incomplete `multicall` (missing `refundETH()`) can be stolen by an unprivileged attacker who initiates a WETH-input swap for the same amount, receiving output tokens while paying nothing.

## Finding Description
`pay()` at lines 69–88 of `PeripheryPayments.sol` branches on `address(this).balance` when `token == WETH`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never charged
}
``` [1](#0-0) 

When `nativeBalance >= value`, the router wraps its own ETH and forwards WETH to the pool. The `payer` argument (the original `msg.sender` stored in transient storage) is never consulted. There is no per-transaction accounting of who contributed the ETH.

ETH accumulates in the router when any `payable` entry point is called with `msg.value` exceeding the consumed amount and the caller omits `refundETH()`. The `receive()` guard:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

only blocks direct ETH pushes; it does not prevent ETH from accumulating via `msg.value` in payable functions like `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall`.

The callback path that reaches `pay()`:

```solidity
function _justPayCallback(int256 amount0Delta, int256 amount1Delta) private {
    pay(_getTokenToPay(), _getPayer(), msg.sender,
        uint256(MetricOmmSwapResults.extractPositiveAmount(amount0Delta, amount1Delta)));
}
``` [3](#0-2) 

`_getPayer()` returns the attacker's address from transient storage, but `pay()` ignores it entirely when the router's ETH balance is sufficient.

`exactInputSingle` is `payable` and sets the payer to `msg.sender` but enforces no `refundETH()`: [4](#0-3) 

`multicall` is also `payable` with no forced refund: [5](#0-4) 

## Impact Explanation
Direct, permanent loss of user principal. Any ETH stranded in the router (a common integration mistake when using `multicall{value: X}` without appending `refundETH()`) is immediately claimable by any attacker who calls a WETH-input swap for `amountIn ≤ router.balance`. The attacker receives output tokens worth the full stranded ETH while paying nothing. This is a swap conservation failure: the pool receives WETH (wrapped from the router's own balance) without the attacker paying, and the prior user's ETH is permanently stolen. Severity: Critical.

## Likelihood Explanation
The `multicall` + `refundETH` pattern is the documented usage, but omitting `refundETH` is a common integration mistake. The router's ETH balance is publicly readable on-chain. Any MEV bot can monitor for non-zero balances and atomically exploit the window in the same block. The attack requires no special permissions, no token approvals, no privileged role, and no prior setup — only the ability to call a public swap function.

## Recommendation
Track per-transaction ETH contributions in transient storage. At each `payable` entry point, record `msg.value` in a transient slot. In `pay()`, consume only from that recorded amount (decrementing it) and fall through to `safeTransferFrom` for any remainder, rather than reading the global `address(this).balance`. Alternatively, remove the native-ETH shortcut from `pay()` entirely and require callers to wrap ETH via `WETH.deposit()` before calling the router.

## Proof of Concept

```solidity
// Step 1 – User A leaves 0.5 ETH stranded in the router
router.multicall{value: 1 ether}([
    abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        pool: pool, tokenIn: WETH, tokenOut: token1,
        zeroForOne: true, amountIn: 0.5 ether,
        amountOutMinimum: 0, recipient: userA,
        deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    })))
    // ← no refundETH(); 0.5 ETH stays in router
]);

// Step 2 – Attacker exploits the stranded ETH (zero ETH sent, zero WETH approval)
router.exactInputSingle(ExactInputSingleParams({
    pool: pool, tokenIn: WETH, tokenOut: token1,
    zeroForOne: true, amountIn: 0.5 ether,
    amountOutMinimum: 0, recipient: attacker,
    deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
}));
// Pool calls metricOmmSwapCallback → _justPayCallback → pay(WETH, attacker, pool, 0.5e18)
// address(this).balance == 0.5 ETH >= 0.5 ETH → router wraps its own ETH, sends WETH to pool
// Attacker receives token1 worth 0.5 ETH; User A's 0.5 ETH is permanently stolen.
```
