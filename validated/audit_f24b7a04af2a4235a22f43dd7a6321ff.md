Audit Report

## Title
`MetricOmmPoolLiquidityAdder` Accepts Arbitrary Pool Address Without Factory Verification, Enabling Malicious Pool to Drain Victim Tokens Up to Max Caps - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary

`MetricOmmPoolLiquidityAdder` stores a caller-supplied `pool` address into transient storage via `_setPayContext` with no factory membership check. A malicious pool trivially passes the `msg.sender == expectedPool` guard in `metricOmmModifyLiquidityCallback`, then returns attacker-controlled token addresses from `getImmutables()`, causing the contract to execute `safeTransferFrom(victim, maliciousPool, amount)` for up to `maxAmountToken0`/`maxAmountToken1`. The contract's own NatSpec explicitly acknowledges this gap.

## Finding Description

`MetricOmmSwapRouterBase` holds an immutable `FACTORY` reference and calls `_requireFactoryPool(pool)` before writing any pool into transient storage in `_setNextCallbackContext`, `_initCallbackContextforRecursiveOutput`, and `_updateCallbackContextforRecursiveOutput`. [1](#0-0) [2](#0-1) [3](#0-2) 

`MetricOmmPoolLiquidityAdder` has no factory reference. Its constructor accepts only `weth`, and `_setPayContext` writes the caller-supplied `pool` directly into transient storage with no membership check: [4](#0-3) [5](#0-4) 

The NatSpec explicitly acknowledges this: [6](#0-5) 

In `metricOmmModifyLiquidityCallback`, the only caller check is `msg.sender != expectedPool`. Because `expectedPool` was set to the attacker-supplied address, a malicious pool trivially passes: [7](#0-6) 

After passing the check, the callback calls `IMetricOmmPool(msg.sender).getImmutables()` to obtain `token0`/`token1`, then calls `pay(token0, payer, msg.sender, amount0Delta)` and `pay(token1, payer, msg.sender, amount1Delta)`. A malicious pool controls both the token addresses and the delta values (bounded only by `max0`/`max1`): [8](#0-7) 

## Impact Explanation

A victim who has pre-approved `MetricOmmPoolLiquidityAdder` for token spending (required for normal LP use) and is tricked into calling `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 in a single transaction. Stolen tokens are transferred directly to the attacker's contract via `safeTransferFrom`. This is a direct, unrecoverable loss of user principal — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation

The attack requires the victim to invoke the adder with an attacker-controlled pool address. This is achievable via phishing UI substituting a malicious pool for a legitimate one, a poisoned off-chain pool registry, or address-similarity spoofing. Active LPs always have the adder pre-approved (standard workflow), so the approval prerequisite is always satisfied for the target population. The sibling `MetricOmmSwapRouterBase` already has the tooling to prevent this, confirming the omission is an oversight.

## Recommendation

Add a factory reference to `MetricOmmPoolLiquidityAdder` mirroring `MetricOmmSwapRouterBase`, and call `_requireFactoryPool(pool)` inside `_setPayContext` before writing to transient storage:

```solidity
// Constructor:
IMetricOmmPoolFactory internal immutable FACTORY;
constructor(address weth, address factory) PeripheryPayments(weth) {
    if (factory == address(0)) revert InvalidFactory();
    FACTORY = IMetricOmmPoolFactory(factory);
}

// _setPayContext:
function _setPayContext(address pool, address payer, uint256 maxAmountToken0, uint256 maxAmountToken1) internal {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool); // add this guard
    if (_tloadAddress(T_SLOT_PAY_POOL) != address(0)) revert PayContextAlreadyActive();
    _tstoreAddress(T_SLOT_PAY_POOL, pool);
    _tstoreAddress(T_SLOT_PAY_PAYER, payer);
    _tstore(T_SLOT_PAY_MAX0, maxAmountToken0);
    _tstore(T_SLOT_PAY_MAX1, maxAmountToken1);
}
```

## Proof of Concept

1. Attacker deploys `MaliciousPool` implementing `addLiquidity` and `getImmutables`. `getImmutables()` returns `token0 = stolenToken`. `addLiquidity` immediately calls back `metricOmmModifyLiquidityCallback(maxAmount, 0, abi.encode(KIND_PAY))`.
2. Victim (active LP with adder approved for `stolenToken`) is tricked into calling `addLiquidityExactShares(maliciousPool, victim, 0, deltas, 1_000e18, 0, "")`.
3. `_addLiquidity` → `_setPayContext(maliciousPool, victim, 1_000e18, 0)` — no factory check, pool stored.
4. `maliciousPool.addLiquidity(...)` is called; malicious pool immediately calls back `metricOmmModifyLiquidityCallback(1_000e18, 0, abi.encode(KIND_PAY))`.
5. Callback: `expectedPool = maliciousPool`, `msg.sender = maliciousPool` → `msg.sender != expectedPool` check passes. `1_000e18 <= max0` check passes.
6. `maliciousPool.getImmutables()` returns `token0 = stolenToken`.
7. `pay(stolenToken, victim, maliciousPool, 1_000e18)` → `IERC20(stolenToken).safeTransferFrom(victim, maliciousPool, 1_000e18)`.
8. Victim loses 1,000 tokens with no recovery path.
