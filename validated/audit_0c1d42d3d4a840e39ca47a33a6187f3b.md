### Title
ERC1155 Multi-Address Token Causes Irrecoverable Lock of User Funds via `initTransfer1155` Address-Identity Mismatch — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` computes a deterministic token identifier solely from the caller-supplied `tokenAddress`, but never validates that `multiTokens[deterministicToken]` is populated. `logMetadata1155` is the only function that populates `multiTokens`. For an ERC1155 token deployed behind a proxy (two valid entry-point addresses: proxy `P` and implementation `I`), calling `logMetadata1155(P, tokenId)` and `initTransfer1155(I, tokenId, amount)` produces two different deterministic addresses. The `multiTokens` entry exists only for the proxy-derived address. When `finTransfer` is later called with the implementation-derived deterministic address, it finds `multiTokens[deterministicB].tokenAddress == address(0)`, falls through to the ERC20 branch, and calls `IERC20(deterministicB).safeTransfer(...)` — which reverts because `deterministicB` is a computed hash address with no code. Because the nonce is marked consumed **before** the transfer attempt, the `finTransfer` can never be retried. The user's ERC1155 tokens are permanently locked in the bridge.

---

### Finding Description

**Vulnerability class**: Token-mapping collision / asset-accounting bug — address-based identity check fails for multi-address (proxy + implementation) ERC1155 tokens.

**`logMetadata1155`** populates `multiTokens[deterministicToken]` keyed on `deterministicToken = keccak256(tokenAddress, tokenId)`:

```solidity
// OmniBridge.sol:238-247
address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
MultiTokenInfo storage multiToken = multiTokens[deterministicToken];
if (multiToken.tokenAddress == address(0)) {
    multiToken.tokenAddress = tokenAddress;
    multiToken.tokenId = tokenId;
}
``` [1](#0-0) 

**`initTransfer1155`** computes the same deterministic address from the caller-supplied `tokenAddress`, locks the ERC1155 tokens, and emits `InitTransfer` — but **never checks** that `multiTokens[deterministicToken]` is set:

```solidity
// OmniBridge.sol:453-464
address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
// ... emits InitTransfer with deterministicToken — no multiTokens validation
``` [2](#0-1) 

**`finTransfer`** marks the destination nonce consumed **before** attempting the token release, then dispatches on `multiTokens[payload.tokenAddress]`:

```solidity
// OmniBridge.sol:287, 315, 323, 350-354
completedTransfers[payload.destinationNonce] = true;   // nonce burned here
MultiTokenInfo memory multiToken = multiTokens[payload.tokenAddress];
// ...
} else if (multiToken.tokenAddress != address(0)) {    // ERC1155 path — skipped
    IERC1155(multiToken.tokenAddress).safeTransferFrom(...);
} else {
    IERC20(payload.tokenAddress).safeTransfer(recipient, amount); // reverts
}
``` [3](#0-2) [4](#0-3) 

`deriveDeterministicAddress` is a pure hash of `(tokenAddress, tokenId)`:

```solidity
// OmniBridge.sol:576-584
return address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))));
``` [5](#0-4) 

Because the deterministic address is a hash output, no contract code exists at it. The ERC20 fallback `safeTransfer` call reverts. The nonce is already consumed and cannot be replayed. The locked ERC1155 tokens have no recovery path.

---

### Impact Explanation

**Permanent, irrecoverable lock of user ERC1155 funds.**

- The ERC1155 tokens are transferred into the bridge contract at `initTransfer1155` time.
- The NEAR side mints a corresponding wrapped token to the user (based on the `InitTransfer` event).
- When the user attempts to bridge back, `finTransfer` consumes the destination nonce at line 287 before the transfer executes.
- The subsequent `IERC20(deterministicB).safeTransfer(...)` call reverts because `deterministicB` has no code.
- The nonce is permanently spent; no retry is possible.
- The ERC1155 tokens remain locked in the bridge with no admin escape hatch.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

**Low.** The scenario requires:
1. An ERC1155 token deployed behind a proxy (two valid entry-point addresses). This is a realistic deployment pattern (e.g., OpenZeppelin `TransparentUpgradeableProxy` or UUPS wrapping an ERC1155 implementation).
2. `logMetadata1155` called with one address (e.g., proxy `P`) while `initTransfer1155` is called with the other (e.g., implementation `I`), or `initTransfer1155` called with no prior `logMetadata1155` at all.

No privileged role is required. Both `logMetadata1155` and `initTransfer1155` are permissionless public functions. [6](#0-5) [7](#0-6) 

---

### Recommendation

Add a guard in `initTransfer1155` that requires `multiTokens[deterministicToken]` to already be populated before accepting the deposit:

```solidity
function initTransfer1155(
    address tokenAddress,
    uint256 tokenId,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    if (fee >= amount) revert InvalidFee();

    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

    // Require logMetadata1155 to have been called first with this exact address
    MultiTokenInfo storage info = multiTokens[deterministicToken];
    if (info.tokenAddress == address(0)) {
        // First deposit: populate the mapping atomically
        info.tokenAddress = tokenAddress;
        info.tokenId = tokenId;
    } else if (info.tokenAddress != tokenAddress || info.tokenId != tokenId) {
        revert ERC1155MappingMismatch();
    }
    // ... rest of function
}
```

Alternatively, require `multiTokens[deterministicToken].tokenAddress != address(0)` as a precondition and revert if not set, forcing callers to call `logMetadata1155` first. This ensures the address used in `initTransfer1155` is always the same address stored in `multiTokens`, so `finTransfer` will always find the correct ERC1155 mapping regardless of whether the token has multiple entry-point addresses.

### Proof of Concept

1. Deploy a proxy-wrapped ERC1155 token. Proxy address = `P`, implementation address = `I`. Both `P` and `I` accept `safeTransferFrom` calls (the implementation is a standalone ERC1155 contract).

2. Call `logMetadata1155(P, tokenId)`:
   - `deterministicA = keccak256(P, tokenId)`
   - `multiTokens[deterministicA] = {P, tokenId}`
   - NEAR registers token A.

3. User calls `initTransfer1155(I, tokenId, 100, 0, 0, "user.near", "")`:
   - `deterministicB = keccak256(I, tokenId)` — **different from `deterministicA`**
   - Bridge receives 100 ERC1155 tokens.
   - `InitTransfer` emitted with `tokenAddress = deterministicB`.
   - NEAR mints 100 of token B to user.
   - `multiTokens[deterministicB]` is **never set**.

4. User bridges token B back to EVM. NEAR MPC signs `finTransfer` with `payload.tokenAddress = deterministicB, amount = 100`.

5. `finTransfer` executes:
   - Line 287: `completedTransfers[nonce] = true` — **nonce consumed**.
   - Line 315: `multiToken = multiTokens[deterministicB]` → `{address(0), 0}`.
   - Line 323: `multiToken.tokenAddress != address(0)` → **false**, ERC1155 path skipped.
   - Lines 331, 337: `customMinters` and `isBridgeToken` checks → **false**.
   - Line 351: `IERC20(deterministicB).safeTransfer(user, 100)` → **REVERTS** (no code at `deterministicB`).

6. Transaction reverts. Nonce is permanently spent. The 100 ERC1155 tokens are irrecoverably locked in the bridge. [8](#0-7) [9](#0-8) [10](#0-9) [5](#0-4)
