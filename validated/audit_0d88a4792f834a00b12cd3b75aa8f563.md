Audit Report

## Title
WETH-native double counting in `pay()` silently consumes stranded ETH from prior callers — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses the router's entire native ETH balance to settle WETH-denominated swaps without verifying that the ETH was contributed by the current transaction's `msg.sender`. Because all public swap and liquidity entry points are `external payable` and never auto-refund excess `msg.value`, a victim who over-sends ETH strands the surplus on the router. Any subsequent WETH swap by any caller silently spends that surplus, giving the second caller a free or discounted settlement at the first caller's expense.

## Finding Description
`PeripheryPayments.pay()` (lines 73–84) checks `address(this).balance` and uses it unconditionally to cover WETH payments:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

There is no accounting of which caller deposited the ETH currently held by the contract. The `receive()` guard (lines 32–34) only blocks plain ETH pushes from non-WETH addresses: [2](#0-1) 

It does **not** block `msg.value` arriving through a payable function call. Every public entry point — `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput` (all in `MetricOmmSimpleRouter`), and both overloads of `addLiquidityExactShares` and `addLiquidityWeighted` (in `MetricOmmPoolLiquidityAdder`) — is `external payable` and performs no automatic ETH refund after execution: [3](#0-2) [4](#0-3) 

The only recovery path is the opt-in `refundETH()` (lines 58–63), which callers must explicitly invoke (e.g., inside a `multicall`): [5](#0-4) 

If a caller omits `refundETH()`, surplus ETH is permanently stranded on the router and is available to any subsequent WETH swap.

## Impact Explanation
Direct, permanent loss of user principal (ETH). A victim who sends `V` ETH for a swap requiring `A < V` loses `V − A` ETH. The attacker receives the full swap output for zero cost. The loss scales with the surplus and is bounded only by the victim's wallet balance. This constitutes a direct loss of user principal and a swap conservation failure (the pool receives WETH that was not paid by the registered payer for that transaction).

## Likelihood Explanation
- All swap and liquidity entry points are `external payable`, making it natural for callers to attach ETH.
- Users routinely over-send ETH to avoid reverts under slippage or partial-fill uncertainty.
- `refundETH()` is opt-in; direct calls to swap functions without a `multicall` wrapper will not trigger it.
- An attacker needs only a single `eth_getBalance` RPC call to detect stranded ETH on the router, then submits a WETH swap of matching size — no special privilege, no malicious setup, no non-standard token required.
- The attack is repeatable across any number of victims and any WETH pool.

## Recommendation
Track the ETH contributed by the current transaction in transient storage: store `msg.value` at each public entry point and deduct from it inside `pay()`, consuming native ETH only up to the amount the current caller deposited in this transaction. Alternatively, auto-call `refundETH()` (or an internal equivalent) at the end of every public entry point to return any remaining `address(this).balance` to `msg.sender`. A third option is to disallow native ETH entirely on swap entry points and require callers to pre-wrap ETH to WETH before calling the router.

## Proof of Concept
```
Block N:
  Victim calls exactInputSingle{value: 2 ETH}(
      pool=WETH/TOKEN1, tokenIn=WETH, amountIn=1 ETH, ...
  )
  → metricOmmSwapCallback fires → pay(WETH, victim, pool, 1 ETH)
  → nativeBalance = 2 ETH >= value = 1 ETH
  → router deposits 1 ETH → WETH, transfers to pool
  → 1 ETH remains on router; victim does NOT call refundETH()

Block N (same or next):
  Attacker calls exactInputSingle{value: 0}(
      pool=WETH/TOKEN1, tokenIn=WETH, amountIn=1 ETH, ...
  )
  → metricOmmSwapCallback fires → pay(WETH, attacker, pool, 1 ETH)
  → nativeBalance = 1 ETH >= value = 1 ETH
  → router deposits victim's 1 ETH → WETH, transfers to pool
  → attacker receives TOKEN1 output, pays 0 WETH from own balance

Result: victim loses 1 ETH permanently; attacker receives full swap output for free.
```
