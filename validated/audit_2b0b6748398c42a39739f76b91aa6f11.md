Looking at the production code carefully to trace the exact validation path.

### Title
Missing `chunk.shard_id() == requested_shard_id` guard in `set_state_header` allows cross-shard state corruption — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` verifies that the embedded chunk is *some* chunk in the block's `chunk_headers_root` merkle tree, but never verifies that it is the chunk for the *requested* shard. An unprivileged peer can supply shard X's chunk (with shard X's valid merkle proof) in response to a state-sync request for shard Y. The header passes all validation and is stored under `StateHeaderKey(Y, sync_hash)`. When `set_state_finalize(Y, sync_hash)` runs, it reads that header, derives `shard_uid` for Y, but applies shard X's transactions and state root, installing shard X's post-state as shard Y's state in the chain store.

---

### Finding Description

**Entrypoint:** `set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called from the `StateHeaderValidationRequest` handler in `chain/client/src/client_actor.rs`.

The critical guard is at lines 394–403:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) {
    return Err(Error::Other("set_shard_state failed: chunk isn't included into block".into()));
}
``` [1](#0-0) 

`verify_path` only checks that `ChunkHashHeight(chunk_hash, height_included)` hashes to the given root via the supplied merkle path. It does **not** check the leaf index (position) in the tree: [2](#0-1) 

The `chunk_headers_root` is a merkle tree of all chunks ordered by shard index: [3](#0-2) 

If an attacker provides shard X's chunk with shard X's valid merkle path, `verify_path` returns `true` — shard X's chunk IS in the tree, just at position X, not position Y. The code never asserts `chunk.shard_id() == shard_id`. There is no `verify_path_with_index` call that would bind the proof to the expected leaf index.

After all checks pass, the header is persisted:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [4](#0-3) 

`set_state_finalize` then reads this header and applies it: [5](#0-4) 

Inside `chain_update.set_state_finalize`, `shard_uid` is correctly derived from the *requested* `shard_id=Y`, but `chunk_header.prev_state_root()` and all transactions come from shard X's chunk: [6](#0-5) 

The state parts for shard X's state root are installed under shard Y's UID via `apply_state_part` (called with `shard_id=Y` but the state root from the poisoned header). `apply_chunk` then succeeds because the trie data is present, and shard Y's `ChunkExtra` is written with shard X's `outgoing_receipts_root`, `gas_used`, `balance_burnt`, and `new_root`.

---

### Impact Explanation

Shard Y's committed state root, `ChunkExtra`, and flat storage are replaced with shard X's data. Any subsequent block processing for shard Y reads the wrong `ChunkExtra` (wrong state root, wrong gas, wrong receipts root), causing chain-level divergence. Validators tracking shard Y will reject blocks that reference the correct shard Y state root, and the corrupted node will fork off the canonical chain or stall.

---

### Likelihood Explanation

Any peer the syncing node connects to can serve a poisoned `ShardStateSyncResponseHeader`. No validator key or privileged role is required — the attacker only needs to hold the public block data (shard X's chunk, its merkle proof, and shard X's state parts), all of which are available on-chain. The attack is triggered during normal state sync, which every node performs when joining or catching up.

---

### Recommendation

After the `verify_path` call at line 394, add an explicit shard identity check:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Alternatively, replace `verify_path` with `verify_path_with_index` using the expected shard index derived from the epoch's shard layout, binding the proof to the correct leaf position.

---

### Proof of Concept

In an integration test with two shards:

1. Produce a chain with `shard_id=0` and `shard_id=1`.
2. On the syncing client, call `state_sync_adapter.set_state_header(shard_id=1, sync_hash, header_with_shard_0_chunk)` where `header_with_shard_0_chunk` is a real `ShardStateSyncResponseHeader` built from shard 0's chunk and shard 0's merkle proof (both available from the serving client).
3. Install shard 0's state parts via `apply_state_part(shard_id=1, shard_0_state_root, ...)`.
4. Call `chain.set_state_finalize(shard_id=1, sync_hash)`.
5. Assert that the resulting `ChunkExtra` for shard 1 contains shard 0's `outgoing_receipts_root` and `state_root`.

The test will pass (no error is returned), demonstrating that shard 1's state has been silently replaced with shard 0's data.
