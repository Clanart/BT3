### Title
Stale RO-Snapshot in `read_pending_data` Silently Replaces Real Pending State Diff with Empty Diff During Batch-Write Window — (`File: crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When storage batching is enabled (`batch_size > 1`), `read_pending_data` derives the chain tip from a read-only MDBX snapshot that lags behind the writer's in-memory persistent transaction. During the batch accumulation window the snapshot tip hash does not match the real pending block's `parent_block_hash`, so the function silently discards the real pending state diff and returns a synthetic empty one. Every RPC method that calls `read_pending_data` with `block_id = "pending"` — `estimate_fee`, `simulate_transactions`, `trace_transaction`, `trace_block_transactions`, `get_state_update`, `starknet_call` — then operates on the wrong state and returns authoritative-looking wrong values.

### Finding Description

`StorageWriter::begin_rw_txn` accumulates writes inside a persistent MDBX transaction and only flushes to disk when `commit_counter >= batch_size`. [1](#0-0)  `StorageReader::begin_ro_txn` opens a fresh read-only snapshot of the **committed** data, so it cannot see any writes that are still in the pending batch. [2](#0-1) 

The storage tests explicitly confirm this: with `batch_size = 5`, four committed writes are invisible to a concurrent reader until the fifth write triggers the flush. [3](#0-2) 

`read_pending_data` re-derives the chain tip from a `StorageTxn<'_, Mode>` (an RO snapshot) and compares it to `pending_data.block.parent_block_hash()`:

```rust
let latest_header = match get_latest_block_number(txn)? {
    Some(n) => get_block_header_by_number(txn, n)?,
    None => /* genesis stub */,
};
if pending_data.block.parent_block_hash() == latest_header.block_hash {
    Ok((*pending_data).clone())          // real pending data
} else {
    Ok(PendingData {                     // synthetic fallback
        state_update: ClientPendingStateUpdate {
            old_root: latest_header.block_header_without_hash.state_root,
            state_diff: Default::default(),   // ← EMPTY
        },
        ...
    })
}
``` [4](#0-3) 

During the batch accumulation window the RO snapshot's tip is the last **flushed** block (N−k), while the pending data correctly points to the real tip (N). The hashes differ, so the `else` branch fires and returns an empty state diff anchored on the stale root.

The `sync_pending_data` function in `apollo_central_sync` explicitly documents why re-reading the tip from an RO snapshot is wrong under batching and requires the caller to pass the accurate tip instead:

> "NOT a value re-read from a read-only storage snapshot: with batched writes the committed-to-disk view can lag the channel marker, which would anchor pending data on a stale block and starve the pending feed." [5](#0-4) 

`read_pending_data` does not apply this fix; it re-reads the tip from the RO snapshot every time.

### Impact Explanation

Every RPC handler that calls `read_pending_data` with `block_id = "pending"` is affected:

- `estimate_fee` — fee is computed against empty pending state diff; nonces, balances, and storage values from the real pending block are invisible. [6](#0-5) 
- `simulate_transactions` — simulation runs against wrong state. [7](#0-6) 
- `trace_transaction` / `trace_block_transactions` — traces are produced from wrong state. [8](#0-7) 
- `get_state_update("pending")` — returns empty state diff and wrong `old_root`. [9](#0-8) 
- `starknet_call("pending")` — contract call executes against wrong state.

The returned values look authoritative (no error, valid JSON-RPC response) but are computed from a state that is missing all storage writes, nonce increments, and class deployments that occurred in the unflushed batch. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

Batching is a production feature controlled by `BatchConfig` in the node configuration. [10](#0-9)  Any deployment with `batch_size > 1` is affected. The window during which the bug is active is not a narrow race: it spans the entire period between the first write of a new batch and the flush at `commit_counter == batch_size`. With `batch_size = 5` this is four full block-write cycles. Any client querying `"pending"` during this window receives wrong data. No special privileges or timing are required; any unprivileged RPC caller triggers the wrong path.

### Recommendation

`read_pending_data` must not re-derive the chain tip from an RO snapshot. It should accept the accurate tip hash as a parameter (mirroring the pattern already used in `sync_pending_data`), or call `StorageWriter::flush_pending_writes` before opening the RO snapshot. The simplest fix is to pass the tip hash from the caller's authoritative source (e.g., the channel marker or the writer's own in-memory state) rather than re-reading it from the potentially stale MDBX snapshot.

### Proof of Concept

1. Configure storage with `batch_size = 5`.
2. Sync writes blocks 0–3 (four writes; batch not yet flushed — all invisible to RO readers). [11](#0-10) 
3. `sync_pending_data` updates `pending_data` so that `pending_data.block.parent_block_hash() == block_3.block_hash`.
4. Client calls `starknet_estimateFee` with `block_id = "pending"`.
5. `read_pending_data` opens `begin_ro_txn()` → snapshot sees only the last flushed block (e.g., block −1 / genesis).
6. `get_latest_block_number(txn)` returns `None` (or the last flushed block number, not 3). [12](#0-11) 
7. `latest_header.block_hash` ≠ `pending_data.block.parent_block_hash()` → `else` branch fires.
8. `read_pending_data` returns synthetic `PendingData` with `state_diff: Default::default()` and `old_root` from the stale block.
9. `exec_estimate_fee` runs against this empty pending state; all storage writes, nonce changes, and class deployments from blocks 0–3 are invisible.
10. The returned fee estimate is wrong (e.g., a transaction that would revert due to a nonce conflict in the real pending state appears valid, or a balance check passes against a stale balance). [13](#0-12)

    /// Changes the batch size at runtime.
    ///
    /// If the pending batch has already reached the new (smaller) threshold
    /// (`commit_counter >= batch_size`), it is flushed immediately rather than waiting for the
    /// next commit. In particular, lowering the size to 1 with pending writes flushes them now.
    /// This upholds the invariant that `batch_size == 1` never has a pending batch — required
    /// because `StorageTxnRW`'s `Drop` discards the persistent transaction at `batch_size == 1`,
    /// so a subsequent read-only transaction must never be able to drop committed-but-unflushed
    /// writes.
    pub fn set_batch_size(&mut self, batch_size: usize) -> StorageResult<()> {
        let mut guard = self.shared_state.lock().unwrap();
        if guard.commit_counter >= batch_size {
            Self::flush_pending_locked(&self.file_writers, &mut guard)?;
        }
        guard.batch_size = batch_size;
        Ok(())
    }

            .expect("Failed to append header")
            .commit()
            .expect("Failed to commit");

    for i in 0..(batch_size - 1) {

            .begin_ro_txn()
            .expect("Failed to begin read transaction")
            .get_block_header(BlockNumber(i as u64))
            .expect("Failed to get header");
        assert!(header.is_none(), "Header {} should not be visible before batch commit", i);
    }

    // Write one more header to reach batch_size.
    writer
        .begin_rw_txn()
        .expect("Failed to begin transaction")

            &create_test_header_for_batching(BlockNumber((batch_size - 1) as u64)),
        )
        .expect("Failed to append header")
        .commit()
        .expect("Failed to commit");

    // Now all headers should be visible.
    for i in 0..batch_size {
        let header = reader
            .begin_ro_txn()
            .expect("Failed to begin read transaction")
            .get_block_header(BlockNumber(i as u64))
            .expect("Failed to get header");
        assert!(header.is_some(), "Header {} should be visible after batch commit", i);
        assert_eq!(header.unwrap().block_header_without_hash.block_number, BlockNumber(i as u64));

            block: PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock {

                starknet_version: latest_header

                    .starknet_version
                    .to_string(),
                ..Default::default()
            }),

                old_root: latest_header.block_header_without_hash.state_root,
