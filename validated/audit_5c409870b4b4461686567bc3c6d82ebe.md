### Title
Integer Division Truncation in `NewDynamicFeeChecker` Causes Systematic Fee Under-Collection and Potential Zero-Fee Bypass — (`File: ante/evm/fee_checker.go`)

### Summary
`NewDynamicFeeChecker` computes the per-gas fee cap via integer division (`fee / gas`) before multiplying back to derive `effectiveFee`. Due to truncation, the fee actually deducted from the sender is always `≤ fee`, and in the extreme case where `fee < gas`, the computed `feeCap` collapses to zero, allowing a transaction to pass with `effectiveFee = 0` whenever `baseFee = 0`.

### Finding Description

In `ante/evm/fee_checker.go`, the `NewDynamicFeeChecker` function computes the per-gas fee cap using integer (truncating) division: [1](#0-0) 

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // integer division — truncates
```

`fee` is the total fee amount in the EVM denom (`sdk.Int`), and `gas` is the `uint64` gas limit. `sdkmath.Int.Quo` performs truncating integer division. The result `feeCap` is then used to derive the effective price and the effective fee: [2](#0-1) 

```go
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))

effectiveFee := sdk.Coins{{
    Denom:  denom,
    Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)),   // multiply after divide
}}
```

The `DeductFeeDecorator` in `ante/evm/nativefee.go` **replaces** the original `feeTx.GetFee()` with the value returned by the fee checker, and deducts only that amount: [3](#0-2) 

```go
fee := feeTx.GetFee()
if !simulate {
    fee, priority, err = dfd.txFeeChecker(ctx, tx)   // fee is overwritten with effectiveFee
    ...
}
if err := dfd.checkDeductFee(ctx, tx, fee); err != nil { ... }
``` [4](#0-3) 

Only `effectiveFee` (not the original `feeTx.GetFee()`) is sent to `evmkeeper.DeductFees`.

**Two concrete consequences of the truncation:**

**1. Systematic fee under-collection (always present):**
For any Cosmos tx, the deducted fee is `floor(fee / gas) * gas`, not `fee`. The difference `fee mod gas` (up to `gas − 1` units) is never collected. For example, with `fee = 21_000_000_000_001` and `gas = 21_000`, `feeCap = 1_000_000_000` and `effectiveFee = 21_000_000_000_000` — 1 unit is silently dropped every tx.

**2. Zero-fee bypass when `baseFee = 0`:**
When `fee < gas` (e.g., `fee = 1`, `gas = 1_000_000`), `feeCap = 0`. The sufficiency check: [5](#0-4) 

```go
if feeCap.LT(baseFeeInt) { ... }   // 0 < 0 == false → passes when baseFee == 0
```

passes when `baseFee = 0`. Then `effectivePrice = min(0 + tip, 0) = 0`, so `effectiveFee = 0 * gas = 0`. The `DeductFeeDecorator` receives `effectiveFee = 0` and the `!fee.IsZero()` guard skips the deduction entirely — the sender pays nothing.

### Impact Explanation

This is a **fee mis-accounting bug** in the ante handler. The `effectiveFee` returned by `NewDynamicFeeChecker` is always `≤ fee` due to truncation, meaning the fee collector module systematically receives less than the sender committed to pay. In the zero-fee scenario (`fee < gas`, `baseFee = 0`), a Cosmos tx using the EVM denom can be submitted and executed with zero fee deducted, bypassing the fee market entirely. This fits the allowed High impact: *"ante handler bug that permits valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

- The systematic under-collection affects **every** Cosmos tx routed through `NewDynamicFeeChecker` when `fee mod gas ≠ 0`.
- The zero-fee bypass requires `baseFee = 0`. This occurs when `NoBaseFee = true` in feemarket params (a governance-settable parameter) **and** `types.GetBaseFee` returns `0` rather than `nil` in that mode. If it returns `nil`, the code falls back to `checkTxFeeWithValidatorMinGasPrices` and the bypass is blocked. The exact return value of `GetBaseFee` under `NoBaseFee = true` was not fully confirmed in this analysis.
- Any unprivileged user can craft a Cosmos tx with `fee < gas` to trigger the truncation-to-zero path.

### Recommendation

Perform multiplication before division. Compute the required total fee as `baseFee * gas` (integer, no truncation) and compare directly against the submitted `fee`, rather than deriving a truncated per-gas `feeCap`:

```go
// Instead of:
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
if feeCap.LT(baseFeeInt) { ... }

// Prefer:
requiredFee := baseFeeInt.Mul(sdkmath.NewIntFromUint64(gas))
if fee.LT(requiredFee) { ... }
```

For the effective fee, use `sdkmath.LegacyDec` arithmetic to avoid fractional truncation, or use `fee` directly as the deducted amount when `fee ≤ effectivePrice * gas` is not possible to exceed.

### Proof of Concept

```
Cosmos tx parameters:
  fee    = 1          (1 unit of EVM denom)
  gas    = 1_000_000  (1M gas limit)
  baseFee = 0         (NoBaseFee = true or chain at genesis)

Step 1: feeCap = fee.Quo(gas) = 1 / 1_000_000 = 0   (integer truncation)
Step 2: feeCap.LT(baseFeeInt) → 0 < 0 → false        (check passes)
Step 3: effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = min(0+MaxInt64, 0) = 0
Step 4: effectiveFee = 0 * 1_000_000 = 0
Step 5: DeductFeeDecorator receives fee=0; !fee.IsZero() == false → no deduction
Result: Tx executes with 0 fees deducted despite sender specifying fee=1
```

For the systematic case:
```
fee = 21_000_000_000_001, gas = 21_000, baseFee = 1_000_000_000
feeCap = 21_000_000_000_001 / 21_000 = 1_000_000_000  (1 unit truncated)
effectivePrice = min(1_000_000_000 + tip, 1_000_000_000) = 1_000_000_000
effectiveFee = 1_000_000_000 * 21_000 = 21_000_000_000_000
Charged: 21_000_000_000_000 instead of 21_000_000_000_001 (1 unit lost per tx)
```
