### Title
Unchecked Multiplication Overflow in `denormalize_amount` Permanently Freezes User Funds in `fin_transfer_callback` - (File: near/omni-bridge/src/lib.rs)

### Summary
`denormalize_amount` performs an unchecked `u128` multiplication. When a user initiates a transfer on an origin chain with a sufficiently large amount, the multiplication overflows and panics inside `fin_transfer_callback`. Because the origin-chain tokens are already burned/locked and the NEAR callback always panics for that amount, the user's funds are permanently irrecoverable.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

Both `10_u128.pow(diff_decimals)` and the subsequent multiplication are unchecked. NEAR contracts are compiled with `overflow-checks = true`, so any overflow causes a runtime panic.

This function is called unconditionally inside `fin_transfer_callback`:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
    fee: Self::denormalize_fee(&init_transfer.fee, decimals),
    ...
};
``` [2](#0-1) 

`denormalize_fee` also calls `denormalize_amount` on the fee field, creating a second overflow surface on the same code path. [3](#0-2) 

The `Decimals` struct stores `decimals` (bridge-normalized, capped at 18) and `origin_decimals` (the token's native precision): [4](#0-3) 

When `origin_decimals > decimals` (e.g., a NEAR token with 24 decimals bridged to EVM normalized to 18), `diff_decimals = 6` and the overflow threshold is `u128::MAX / 10^6 ≈ 3.4 × 10^32`. Any `init_transfer.amount` above this threshold causes a deterministic panic.

### Impact Explanation

When `fin_transfer_callback` panics:
- NEAR rolls back all state changes from that callback invocation.
- No transfer-ID finalization record is written, so the proof cannot be replayed successfully (the overflow is deterministic for the same amount).
- The user's tokens on the origin chain are already burned (bridge tokens) or locked in the EVM/Solana vault — there is no refund path on the origin side.

The result is **permanent, irrecoverable loss of the user's bridged assets**, matching the Critical impact class: "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."

The same overflow surface exists in `claim_fee_callback`: [5](#0-4) 

If triggered there, the stored `TransferMessage` has already been removed (`remove_transfer_message` at line 1094), so the fee is permanently unclaimable and the accounting record is gone.

### Likelihood Explanation

**Low.** The overflow requires `amount > u128::MAX / 10^diff_decimals`. For `diff_decimals = 6` (the common NEAR-24 → EVM-18 case), the threshold is ~340 trillion tokens in 18-decimal units. This is unrealistic for most tokens but is reachable for high-supply tokens (e.g., meme tokens with quadrillion-unit supplies) or tokens with larger `diff_decimals`. No privileged role is required — any user who initiates a transfer on the origin chain with a sufficiently large amount triggers the condition.

### Recommendation

Replace the unchecked arithmetic in `denormalize_amount` with checked operations and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Option<u128> {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.checked_pow(diff_decimals)?;
    amount.checked_mul(multiplier)
}
```

All call sites (`fin_transfer_callback`, `fast_fin_transfer`, `claim_fee_callback`, `denormalize_fee`) should handle the `None` case with an explicit error (e.g., `near_expect(BridgeError::AmountOverflow)`) rather than a panic, so the callback fails gracefully without permanently consuming the user's proof or funds.

### Proof of Concept

1. Token is registered with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. User calls `initTransfer` on EVM with `amount = u128::MAX / 10^6 + 1` (≈ 3.4 × 10^32 in 18-decimal units). Tokens are burned on EVM.
3. Relayer submits the proof to NEAR via `fin_transfer`.
4. `fin_transfer_callback` is invoked; it calls `denormalize_amount(amount, decimals)`.
5. `amount * 10^6` overflows `u128`; NEAR runtime panics and rolls back the callback.
6. The transfer ID is never marked finalized; the same proof always panics on retry.
7. User's EVM tokens are permanently burned with no corresponding NEAR credit. [1](#0-0) [6](#0-5)
