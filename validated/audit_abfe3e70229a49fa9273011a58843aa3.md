### Title
Prover Replacement via `add_prover`/`remove_prover` Permanently Locks In-Flight EVM→NEAR User Funds — (File: near/omni-bridge/src/lib.rs)

---

### Summary

The NEAR bridge's `add_prover` and `remove_prover` DAO functions change the prover contract for a chain with immediate effect and no mechanism to handle in-flight transfers. When the prover is replaced or removed, any pending EVM→NEAR transfers (tokens already locked on EVM, not yet finalized on NEAR) can no longer be finalized, permanently locking user funds on EVM with no user-facing recovery path.

---

### Finding Description

The `Contract` struct stores a `provers: UnorderedMap<ChainKind, AccountId>` mapping that holds the prover contract address for each chain. [1](#0-0) 

The DAO can update this mapping via two functions: [2](#0-1) 

Both functions take immediate effect with no grace period, no check for in-flight transfers, and no drain mechanism.

Every incoming EVM→NEAR finalization call goes through `fin_transfer`, which calls `verify_proof`: [3](#0-2) 

If the prover for `chain_kind` has been removed, `verify_proof` panics with `ProverForChainKindNotRegistered`. If it has been replaced with an incompatible prover (different proof format, e.g., switching from EVM light-client to Wormhole), the new prover rejects the submitted proof. [4](#0-3) 

The EVM bridge has no `cancelTransfer` or refund function. Once a user calls `initTransfer` on EVM, their tokens are locked and the only release path is `finTransfer` on EVM (which requires a NEAR MPC signature) or `fin_transfer` on NEAR (which requires a valid prover). There is no user-facing escape hatch. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds in the EVM bridge vault.**

A user who has called `initTransfer` on EVM (tokens locked) but whose `fin_transfer` on NEAR has not yet been processed is entirely dependent on the prover being present and compatible. If the prover is removed or replaced with an incompatible one before finalization:

- `fin_transfer` on NEAR panics or fails permanently.
- The user's tokens remain locked in the EVM bridge with no self-service recovery.
- The EVM bridge has no refund or cancel mechanism.
- Recovery requires the DAO to re-add the old prover — if the old prover contract is no longer available or the DAO does not act, the lock is irrecoverable.

This matches the allowed impact: *"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

The DAO legitimately upgrades provers during protocol evolution (e.g., switching from EVM light-client provers to Wormhole-based provers, or patching a buggy prover). The normal upgrade path is to call `add_prover(chain, new_prover_id)`, which immediately replaces the old prover via `UnorderedMap::insert`. Any `fin_transfer` submitted after this point uses the new prover. If the new prover requires a different proof format (e.g., Wormhole VAA vs. Merkle receipt proof), all in-flight transfers submitted with the old format fail permanently. The DAO may not realize that in-flight transfers exist at the moment of the upgrade. [6](#0-5) 

---

### Recommendation

1. **Timelock / grace period**: Do not apply prover changes immediately. Queue the change and apply it only after a delay sufficient for all in-flight transfers to complete.
2. **In-flight transfer drain check**: Before replacing or removing a prover, verify (or require off-chain confirmation) that no unfinalized transfers exist for that chain.
3. **User-facing cancel/refund on EVM**: Add a `cancelTransfer` function on the EVM bridge that allows users to reclaim locked tokens if a corresponding NEAR-side finalization proof is not submitted within a timeout window.
4. **Dual-prover support**: Allow both old and new provers to be registered simultaneously during a migration window, so in-flight transfers can still be finalized with the old prover while new transfers use the new one.

---

### Proof of Concept

1. Alice calls `initTransfer` on the EVM `OmniBridge`, locking 10,000 USDC. The EVM bridge emits an `InitTransfer` event.
2. The DAO calls `add_prover(ChainKind::Eth, new_wormhole_prover_id)` to upgrade from the EVM light-client prover to a Wormhole-based prover. The old prover is immediately replaced.
3. A relayer constructs a Merkle receipt proof (old format) and calls `fin_transfer` on the NEAR bridge with `chain_kind = ChainKind::Eth`.
4. `verify_proof` dispatches to `new_wormhole_prover_id`, which expects a Wormhole VAA, not a Merkle proof. The prover returns an error.
5. `fin_transfer_callback` receives a failed promise result and panics with `ERR_INVALID_PROOF`.
6. Alice's 10,000 USDC remains locked in the EVM bridge. There is no `cancelTransfer` function. Alice cannot recover her funds without DAO intervention to re-add the old prover. [7](#0-6) [3](#0-2) [8](#0-7)
