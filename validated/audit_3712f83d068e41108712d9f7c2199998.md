### Title
No Mechanism to Update `light_client` in `EvmProver` ÔÇö Permanent Freezing of All EVMÔćĺNEAR Bridge Finalization - (File: `near/omni-prover/evm-prover/src/lib.rs`)

---

### Summary

The `EvmProver` contract stores a single `light_client: AccountId` set immutably at initialization. Every call to `verify_proof` makes a mandatory cross-contract call to that address. There is no admin, DAO, or governance function to update the `light_client` address. If the light client contract is ever deprecated, migrated to a new account ID, or becomes permanently unresponsive, `verify_proof` will always fail and all EVMÔćĺNEAR bridge finalization (`fin_transfer`) will be permanently frozen.

The same structural flaw exists in `WormholeOmniProverProxy`, where `prover_account` is equally immutable.

---

### Finding Description

`EvmProver` is initialized with a `light_client` account ID and exposes only three methods: `init`, `verify_proof`, and `verify_proof_callback`. There is no setter, no role-gated update function, and no migration path for the `light_client` field. [1](#0-0) 

Every `verify_proof` call unconditionally calls `evm_client::ext(self.light_client.clone()).block_hash_safe(...)`: [2](#0-1) 

In `verify_proof_callback`, if the light client returns anything other than `Some(expected_block_hash)` ÔÇö including a cross-contract call failure caused by the light client being broken ÔÇö the proof is rejected: [3](#0-2) 

The omni-bridge's `fin_transfer` calls `self.verify_proof(args.chain_kind, args.prover_args)` which dispatches to the registered prover for the chain kind: [4](#0-3) 

The same pattern exists in `WormholeOmniProverProxy`: `prover_account` is set at `init` with no update path: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

If the EVM light client contract (e.g., `client-eth2.sepolia.testnet`) is deprecated, migrated to a new NEAR account ID, or becomes permanently unresponsive due to a botched upgrade, every call to `EvmProver::verify_proof` will fail. Since `fin_transfer` in the omni-bridge depends on a successful proof verification, all in-flight EVMÔćĺNEAR transfers become permanently unclaimable. Users who have already locked funds on the EVM side cannot finalize receipt on NEAR ÔÇö a permanent, irrecoverable freeze of bridged assets.

This matches: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

NEAR light client contracts are actively maintained infrastructure. A contract migration to a new account ID (e.g., versioned deployments like `client-eth2-v2.near`) is a realistic operational event. The NEAR ecosystem has precedent for contract account migrations. Unlike Chainlink aggregators (which the original report acknowledged as unlikely to fail), NEAR light client contracts are smaller, less battle-tested infrastructure that can be replaced or deprecated. Any such migration without a corresponding `EvmProver` update would silently break all EVM bridge finalization.

---

### Recommendation

Add a privileged (DAO-gated) setter for `light_client` in `EvmProver` and `prover_account` in `WormholeOmniProverProxy`, ideally with a timelock or at minimum an access-controlled role:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn set_light_client(&mut self, new_light_client: AccountId) {
    self.light_client = new_light_client;
}
```

Apply the same pattern to `WormholeOmniProverProxy::prover_account`. This mirrors the standard mitigation recommended in the original report: allow governance to update the external dependency address.

---

### Proof of Concept

1. `EvmProver` is deployed with `light_client = "client-eth2.near"`.
2. The NEAR light client team migrates to `"client-eth2-v2.near"` and decommissions the old account.
3. Any user calls `fin_transfer` on the omni-bridge for an EVMÔćĺNEAR transfer.
4. The omni-bridge dispatches to `EvmProver::verify_proof`.
5. `EvmProver` calls `evm_client::ext("client-eth2.near").block_hash_safe(...)` ÔÇö the call fails (account deleted or unresponsive).
6. `verify_proof_callback` receives a `PromiseError`, returns `Err(ProverError::InvalidBlockHash)`.
7. `fin_transfer_callback` panics with `BridgeError::InvalidProofMessage`.
8. All EVMÔćĺNEAR finalization is permanently blocked. There is no function in `EvmProver` to update `light_client` to the new address. The contract must be redeployed and the omni-bridge's prover registry updated ÔÇö but `EvmProver` itself provides no recovery path. [7](#0-6) [8](#0-7) [3](#0-2)
