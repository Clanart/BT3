### Title
wNEAR Not Unwrapped to Native NEAR When `msg` Is Non-Empty in `send_tokens` — (File: near/omni-bridge/src/lib.rs)

---

### Summary

The `send_tokens` function in the NEAR bridge contract only unwraps wNEAR to native NEAR when the `msg` parameter is empty. When a user bridges wNEAR to a NEAR recipient with a non-empty `msg`, the bridge skips the unwrap path and instead calls `ft_transfer_call` on the wNEAR contract, sending wNEAR as-is. For regular NEAR account recipients (the common case), this call fails and the wNEAR is refunded to the bridge, where it cannot be burned (wNEAR is not a deployed bridge token), causing permanent freezing of user funds.

---

### Finding Description

In `send_tokens`, the wNEAR-to-native-NEAR unwrap path is gated by a compound condition:

```rust
if token == self.wnear_account_id && msg.is_empty() {
    // Unwrap wNEAR → native NEAR
    ext_wnear_token::ext(self.wnear_account_id.clone())
        .near_withdraw(amount)
        .then(near_withdraw_callback(recipient, ...))
} else if is_deployed_token {
    // mint
} else if msg.is_empty() {
    // ft_transfer
} else {
    // ft_transfer_call  ← wNEAR with non-empty msg falls here
    ext_token::ext(token)
        .ft_transfer_call(recipient, amount, None, msg.to_string())
}
```

When `token == wnear_account_id` but `msg` is non-empty, the unwrap branch is skipped. Since wNEAR (`wrap.near`) is not a deployed bridge token, `is_deployed_token` is false. The code falls to the final `else` branch and calls `ft_transfer_call` on the wNEAR contract with the raw wNEAR amount.

For a regular NEAR account recipient (which does not implement `ft_on_transfer`), the wNEAR contract panics during the callback, refunds the full wNEAR amount back to the bridge, and the `fin_transfer_send_tokens_callback` is invoked with `is_ft_transfer_call = true`. The callback then calls `burn_tokens_if_needed`, but since wNEAR is not a deployed bridge token, no burn occurs. The wNEAR remains locked in the bridge contract with no recovery path.

For a contract recipient that does implement `ft_on_transfer`, the contract receives wNEAR instead of native NEAR. Any downstream logic expecting native NEAR (e.g., staking, payable calls) will fail or behave incorrectly.

The `msg` field is fully user-controlled: it is set by the user on the source chain (EVM `message` parameter, Solana `msg` field, Starknet `message` field) and is faithfully propagated into the NEAR `TransferMessage.msg` field, which is then passed directly to `send_tokens`.

---

### Impact Explanation

**Permanent freezing of user funds (Critical):** A user who bridges wNEAR from EVM/Solana/Starknet to a regular NEAR account with a non-empty `msg` will have their wNEAR permanently stuck in the bridge contract. The bridge cannot burn wNEAR (it is not a deployed bridge token), and there is no admin recovery path for stranded non-deployed tokens.

**Incorrect token delivery (High):** A user bridging wNEAR to a contract recipient with a non-empty `msg` receives wNEAR instead of native NEAR. Any protocol expecting native NEAR will malfunction, causing loss of the bridged value from the user's perspective.

Both outcomes match the allowed impact class: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

The `msg` field is a documented, first-class feature of the bridge protocol (used for cross-contract DeFi composability on NEAR). Any user who wants to bridge wNEAR and immediately interact with a NEAR DeFi protocol in the same transaction will set a non-empty `msg`. This is a realistic and expected usage pattern. The vulnerability is triggered by normal user behavior with no special privileges required.

---

### Recommendation

Remove the `msg.is_empty()` guard from the wNEAR unwrap condition. wNEAR should always be unwrapped to native NEAR regardless of whether `msg` is set. If a downstream cross-contract call is needed, it should be performed after the native NEAR transfer (e.g., by attaching the NEAR and calling the recipient contract with the message in the `near_withdraw_callback`):

```rust
// Before (broken):
if token == self.wnear_account_id && msg.is_empty() { ... }

// After (fixed):
if token == self.wnear_account_id {
    // Always unwrap wNEAR to native NEAR.
    // If msg is non-empty, forward the native NEAR to the recipient
    // via a payable cross-contract call in near_withdraw_callback.
    ext_wnear_token::ext(self.wnear_account_id.clone())
        .near_withdraw(amount)
        .then(near_withdraw_callback(recipient, amount, msg))
}
```

### Proof of Concept

1. On EVM, user calls `initTransfer(wNEAR_token, amount, 0, 0, "alice.near", "some_defi_message")`.
2. The EVM bridge emits `InitTransfer` with `message = "some_defi_message"`.
3. Relayer submits proof to NEAR `fin_transfer`.
4. `fin_transfer_callback` → `process_fin_transfer_to_near` → `send_tokens(wnear_account_id, "alice.near", amount, "some_defi_message")`.
5. Condition `token == wnear_account_id && msg.is_empty()` is **false** (msg is non-empty).
6. `is_deployed_token(wnear_account_id)` is **false**.
7. `msg.is_empty()` is **false**.
8. Falls to `ft_transfer_call("alice.near", amount, None, "some_defi_message")`.
9. wNEAR contract calls `ft_on_transfer` on `alice.near` (a regular account) → **panics**.
10. wNEAR refunded to bridge. `fin_transfer_send_tokens_callback` called with `is_ft_transfer_call = true`.
11. `burn_tokens_if_needed(wnear_account_id, amount)` — wNEAR is not a deployed token, no burn occurs.
12. `FailedFinTransferEvent` emitted. wNEAR permanently stuck in bridge. User's funds are lost. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
