Audit Report

## Title
`pay()` consumes aggregate router ETH balance instead of per-call `msg.value`, enabling stranded ETH to subsidize subsequent WETH swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay()` function reads `address(this).balance` (line 74) to determine how much native ETH is available for WETH wrapping. Because this reflects the router's total ETH balance across all prior transactions, any ETH left on the router by a previous caller — who sent excess `msg.value` and omitted `refundETH()` — is silently consumed to subsidize the next WETH swap. The prior user loses their stranded ETH and the subsequent user receives a proportional discount on their WETH payment.

## Finding Description

In `PeripheryPayments.sol`, `pay()` branches on `address(this).balance` when `token == WETH` and `payer != address(this)`:

```solidity
// PeripheryPayments.sol L74-84
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [1](#0-0) 

The router holds no per-user ETH accounting. `address(this).balance` is the aggregate of all ETH sent via `msg.value` across all prior payable calls that was not yet refunded.

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

It does **not** prevent ETH accumulation via `msg.value` in payable function calls — `msg.value` sent with `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, or `multicall` bypasses `receive()` entirely and increases `address(this).balance`. [3](#0-2) [4](#0-3) [5](#0-4) 

**Exploit path:**

1. User A calls `exactInputSingle{value: 1 ETH}(params)` with `tokenIn=WETH`, `amountIn=0.5 ETH`. The pool callback triggers `pay(WETH, UserA, pool, 0.5 ETH)`. `nativeBalance = 1 ETH ≥ 0.5 ETH` → wraps 0.5 ETH, sends to pool. 0.5 ETH remains on the router. User A does not call `refundETH()`.
2. Attacker calls `exactInputSingle(params)` with `tokenIn=WETH`, `amountIn=1 ETH`, no `msg.value`, but with 0.5 WETH approved. `pay(WETH, Attacker, pool, 1 ETH)` is called. `nativeBalance = 0.5 ETH` (0 < 0.5 < 1) → wraps User A's 0.5 ETH, sends to pool, then `safeTransferFrom(Attacker, pool, 0.5 WETH)`. Attacker pays only 0.5 WETH for a 1 ETH WETH swap.

The corrupted value is `nativeBalance` at line 74: it reads the router's global ETH balance instead of the ETH budget belonging to the current caller. [6](#0-5) 

## Impact Explanation

Direct loss of user principal. The prior user's stranded ETH is consumed by a subsequent user's WETH swap. The prior user loses up to 100% of their stranded ETH; the subsequent user receives a proportional discount. This satisfies the contest's Critical/High/Medium direct-loss-of-principal gate. Severity is Medium-to-High depending on contest thresholds, as it requires a precondition (stranded ETH on the router) but that precondition is easily triggered accidentally or adversarially.

## Likelihood Explanation

**Medium.** Two realistic unprivileged triggers exist:

1. **Accidental**: A user calls a payable swap entry point with more ETH than required and omits `refundETH()` from their multicall. The excess ETH is immediately at risk in the next block.
2. **Adversarial front-run**: A victim broadcasts `refundETH()` to reclaim stranded ETH. An attacker observing the mempool front-runs it with a WETH swap where `amountIn` equals the stranded amount, consuming the victim's ETH while paying less WETH from their own account.

Both triggers require no special privileges — only a WETH approval on the router for the adversarial case.

## Recommendation

Track the ETH available for the current transaction using transient storage (EIP-1153). At the start of each payable entry point, store `msg.value` in a transient slot and decrement it as ETH is consumed inside `pay()`. Replace the `address(this).balance` read with the transient per-call budget so that only ETH explicitly sent in the current call can be used to wrap WETH. Alternatively, pass the per-call ETH budget explicitly through the call stack rather than reading the live contract balance.

## Proof of Concept

```solidity
// Foundry test sketch
function test_strandedEthSubsidizesAttacker() public {
    // Setup: UserA sends 1 ETH but only needs 0.5 ETH for swap
    vm.deal(userA, 1 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 0.5 ether, ...})
    );
    // UserA omits refundETH(); 0.5 ETH stranded on router

    assertEq(address(router).balance, 0.5 ether);

    // Attacker has 0.5 WETH approved, sends no ETH
    deal(WETH, attacker, 0.5 ether);
    IERC20(WETH).approve(address(router), 1 ether);
    vm.prank(attacker);
    router.exactInputSingle( // no msg.value
        ExactInputSingleParams({tokenIn: WETH, amountIn: 1 ether, ...})
    );
    // Attacker paid only 0.5 WETH; UserA's 0.5 ETH was consumed
    assertEq(address(router).balance, 0);
    assertEq(IERC20(WETH).balanceOf(attacker), 0); // spent only 0.5 WETH
}
```
