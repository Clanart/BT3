### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezal of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from a contract without verifying that the hash corresponds to a previously declared class. This is an acknowledged missing check (marked with a `TODO`). An attacker who can trigger `replace_class` on a contract holding funds — either through their own contract or by exploiting a bug in a legitimate contract — can set the contract's class hash to an undeclared value, permanently freezing all funds held by that contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `class_hash` value is fully attacker-controlled (it is the `request.class_hash` field written by the calling contract into the syscall segment). There is no call to `dict_read` on `contract_class_changes` to confirm the hash was ever declared, and no `find_element` lookup against `compiled_class_facts_bundle` to confirm a compiled class exists for it.

Contrast this with `execute_declare_transaction`, which enforces that the class hash is the output of `finalize_class_hash` (a cryptographic commitment over the class components) and that `compiled_class_hash != 0`: [2](#0-1) [3](#0-2) 

No equivalent enforcement exists in `execute_replace_class`.

When a subsequent call is made to the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [4](#0-3) 

If the class hash set by `replace_class` was never declared, `dict_read` returns `0` (the default for uninitialized dict entries), and `find_element` fails to locate a compiled class with hash `0`, making the contract permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value via `replace_class`, no entry point of that contract can ever be executed again. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are permanently inaccessible. The OS commits this invalid state to the Merkle tree without objection, and there is no recovery mechanism.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is callable only from within a contract's own execution context. The attacker must either:

1. **Own a contract holding third-party funds** (e.g., a token contract or vault they deployed) and deliberately call `replace_class` with an invalid hash — a self-inflicted but protocol-permitted freeze of deposited user funds.
2. **Exploit an access-control bug in a legitimate contract** that exposes a `replace_class` path without sufficient guards, passing an attacker-chosen hash.

The OS is the last line of defense for protocol invariants. The missing validation means that any contract-level access-control failure around `replace_class` has no OS-level backstop, making the impact of such bugs irreversible.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, the OS must verify that the hash corresponds to a declared class. Concretely, perform a `dict_read` on `contract_class_changes` for the requested `class_hash` and assert the result is non-zero (i.e., a compiled class hash was previously registered for it):

```cairo
// Validate that the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant enforced in `execute_declare_transaction` and closes the gap identified by the existing `TODO` comment. [5](#0-4) 

---

### Proof of Concept

1. **Attacker deploys Contract A** — a token vault that accepts deposits from users and exposes an (insufficiently guarded) `upgrade(class_hash: felt)` function that calls `replace_class(class_hash)`.

2. **Users deposit funds** into Contract A.

3. **Attacker calls `upgrade(0xdeadbeef)`** — a felt value that has never been passed to a `declare` transaction and therefore does not exist in `contract_class_changes`.

4. **OS executes `execute_replace_class`** — reads `request.class_hash = 0xdeadbeef`, skips the missing validation, and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`. The state is committed to the Merkle tree.

5. **Any subsequent call to Contract A** causes `execute_entry_point` to perform `dict_read(key=0xdeadbeef)` on `contract_class_changes`, receiving `0` (undeclared), then `find_element(..., key=0)` fails to locate a compiled class — the contract is permanently uncallable.

6. **All user funds are frozen** with no recovery path.
