### Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` to Measure Price Velocity, Making the Guard Ineffective on Fast-Block Networks — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` measures the time between oracle price observations in **blocks** rather than **seconds**. On any network where the average block time differs from the implicit assumption baked into `maxChangePerBlockE18` (e.g., Arbitrum ~250 ms/block, Optimism/Base ~2 s/block vs. Ethereum mainnet ~12 s/block), the effective per-second price-change allowance is proportionally wrong. On fast-block L2s the guard becomes orders-of-magnitude too permissive, allowing a compromised oracle to move the price far faster in wall-clock time than the pool admin intended, draining LP funds before the guard triggers.

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` records and compares oracle mid-prices using `block.number`:

```solidity
// PriceVelocityGuardExtension.sol  lines 55-70
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);   // ← block-based clock

if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;   // ← block delta, not seconds

        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
        uint256 actualSq  = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);

        if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
    }
}
``` [1](#0-0) 

The struct and admin setter also hard-code the block-based unit:

```solidity
struct PriceVelocityState {
    uint128 lastMidPriceX64;
    uint64  lastUpdateBlock;          // ← stored as block number
    uint64  maxChangePerBlockE18;     // ← parameter named "per block"
}
``` [2](#0-1) 

`setLastMidPrice` also writes `block.number`:

```solidity
s.lastUpdateBlock = uint64(block.number);
``` [3](#0-2) 

The allowed deviation formula is:

```
allowedSq = maxChangePerBlockE18² × (1 + blockDiff)
```

This is calibrated assuming one block ≈ some fixed wall-clock interval. On Ethereum mainnet (~12 s/block) a pool admin might set `maxChangePerBlockE18` to allow, say, 0.1% per block (≈ 0.5%/min). On Arbitrum (~250 ms/block) the same parameter allows 0.1% per 250 ms ≈ **24%/min** — 48× more permissive in real time.

---

### Impact Explanation

The `PriceVelocityGuardExtension` is the sole on-chain mechanism protecting LP funds from a rapid oracle price manipulation attack. When the guard is miscalibrated due to block-time mismatch:

1. A compromised or manipulated oracle can move the mid-price at a real-time rate far exceeding the pool admin's intent without triggering `PriceVelocityExceeded`.
2. Swaps execute at the manipulated price, draining LP token balances from affected bins.
3. The error is **not cumulative in the same direction as Compound's bug** — each swap resets `lastUpdateBlock` — but the per-swap allowance is permanently wrong for the chain's block time, so every swap window is equally over-permissive.

LP principal (token0 and token1 bin balances) is directly at risk. The guard's `beforeSwap` hook is the only code path that could block such a swap; if it passes, the pool executes the trade and transfers tokens out.

---

### Likelihood Explanation

The protocol's deployment infrastructure explicitly targets multiple chains (the `smart-contracts-poc` layer includes multi-chain orchestration scripts). Deploying `PriceVelocityGuardExtension` on any L2 with block times shorter than Ethereum mainnet — which covers virtually every major L2 — silently breaks the guard. Pool admins have no on-chain signal that their `maxChangePerBlockE18` value is miscalibrated for the target chain; the parameter name itself implies a per-block unit, so admins will naturally set it based on block-count intuition rather than seconds.

---

### Recommendation

Replace the block-based clock with `block.timestamp` throughout `PriceVelocityGuardExtension`:

1. Rename `lastUpdateBlock` → `lastUpdateTimestamp` (store `uint64(block.timestamp)`).
2. Rename `maxChangePerBlockE18` → `maxChangePerSecondE18` (or per-minute, per admin preference).
3. Compute `timeDiff = block.timestamp - prevTimestamp` and use it in the allowed-deviation formula instead of `blockDiff`.
4. Update `setLastMidPrice` to write `block.timestamp`.
5. Update the interface struct, events, and admin setter accordingly.

This mirrors the fix recommended in the external report: decouple the guard from Ethereum's average block time so it behaves identically on every EVM-compatible chain.

---

### Proof of Concept

**Setup:** Deploy `PriceVelocityGuardExtension` on Arbitrum (block time ≈ 250 ms). Pool admin sets `maxChangePerBlockE18 = 1e15` (0.1% per block), intending to cap price movement at ~0.5%/min.

**Attack:**
1. Attacker controls or compromises the oracle.
2. In block N, oracle reports `midPrice = P`.
3. In block N+1 (250 ms later), oracle reports `midPrice = P × 1.001` (0.1% move). Guard passes — `blockDiff = 1`, `allowedSq = maxChange² × 2`.
4. Attacker repeats every block. After 60 seconds (240 blocks on Arbitrum), the cumulative price move is `(1.001)^240 ≈ 1.27` — a 27% move — while the guard has passed every single swap.
5. On Ethereum mainnet with the same parameter, 60 seconds = 5 blocks → only 0.5% cumulative move before the guard would trigger.

The guard provides **zero additional protection** beyond what the oracle's own rate of change already imposes, because on fast-block chains the per-block allowance is so large that no realistic oracle update rate can exceed it.
