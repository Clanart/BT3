### Title
Dead-Code `check_is_reverted` Constraint Enables Prover to Skip Invoke Transaction Execution While Charging Fees â€” (File: `execution/execution_constraints.cairo`)

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a no-op that performs zero validation. It is the sole Cairo constraint guarding the hint-provided `is_reverted` value in `execute_invoke_function_transaction`. Because the function does nothing, the prover can freely set `is_reverted = TRUE` for any invoke transaction, causing the `__execute__` entry point to be skipped while fees are still charged and the nonce is still incremented â€” a provable direct loss of user funds encoded in a valid OS proof.

### Finding Description

`check_is_reverted` at lines 20â€“22 of `execution_constraints.cairo`:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

The function accepts `is_reverted` as a parameter and immediately returns without any assertion, range check, or constraint of any kind. It is structurally identical to the reported bug's dead-code deviation check: the function exists, is called, and enforces nothing. [1](#0-0) 

In `execute_invoke_function_transaction` in `transaction_impls.cairo`, the flow is:

```cairo
local is_reverted;
%{ IsReverted %}           // Prover-controlled hint â€” no constraint
check_is_reverted(is_reverted);  // No-op: does nothing
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    non_reverting_select_execute_entry_point_func(...);
} else {
    // Stack alignment only â€” no execution occurs
}
// Fee is charged regardless of is_reverted
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

The `is_reverted` value is loaded from the hint `%{ IsReverted %}` **before any execution occurs**. The only validation is `check_is_reverted`, which is a no-op. There is no Cairo constraint that ties `is_reverted` to the actual execution outcome. The prover can therefore set `is_reverted = TRUE` for any invoke transaction, causing:

1. The `__execute__` entry point to be skipped entirely
2. Fees to still be charged via `charge_fee`
3. The nonce to still be incremented via `check_and_increment_nonce` (called before the `is_reverted` check at line 311) [3](#0-2) 

**Secondary path â€” L1 handler early return with no validation at all:**

In `execute_l1_handler_transaction`, `is_reverted` is loaded from a hint and used directly with no `check_is_reverted` call whatsoever:

```cairo
local is_reverted;
%{ IsReverted %}
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();   // Skips hash computation, nonce check, message consumption, and execution
}
``` [4](#0-3) 

This early return skips `consume_l1_to_l2_message`, meaning the L1-to-L2 message is not recorded in the OS output, yet the transaction slot is consumed. A prover can silently drop any L1 handler without the L1 verifier detecting it.
