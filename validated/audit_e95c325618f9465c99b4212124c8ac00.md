### Title
DataLayer `ProofOfInclusion::valid()` Returns `true` for Empty Layers — Forged Inclusion Proof Accepted for Any Node Hash - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate unconditionally returns `true` when the `layers` vector is empty, regardless of the value of `node_hash`. Because `root_hash()` falls back to `node_hash` when `layers` is empty, the final equality check `existing_hash == self.root_hash()` trivially holds. An attacker can construct a `ProofOfInclusion { node_hash: <any_value>, layers: [] }` that passes validation, forging a proof of inclusion for any claimed key against any tree root equal to `node_hash`.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        // ... hash chain verification ...
    }

    existing_hash == self.root_hash()
}
``` [1](#0-0) 

When `self.layers` is empty, the `for` loop body never executes, so `existing_hash` remains `self.node_hash`. The `root_hash()` method returns:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash   // <-- fallback when layers is empty
    }
}
``` [2](#0-1) 

With empty `layers`, `root_hash()` returns `self.node_hash`, so the final check becomes `self.node_hash == self.node_hash`, which is always `true`. No tree traversal or hash chain is verified.

`ProofOfInclusion` is a `Streamable` type exposed to Python via `pyclass(get_all, from_py_object)` with `from_bytes`, `from_bytes_unchecked`, and `parse_rust` constructors: [3](#0-2) 

It is registered in the Python module: [4](#0-3) 

The Python type stub confirms `ProofOfInclusion` is constructible with arbitrary `node_hash` and `layers`: [5](#0-4) 

An attacker who knows the expected tree root `R` constructs:

```python
forged = ProofOfInclusion(node_hash=R, layers=[])
assert forged.valid()          # True — always
assert forged.root_hash() == R # True — trivially
```

Any verifier that only checks `proof.valid() and proof.root_hash() == expected_root` accepts this forged proof, falsely concluding that the leaf with hash `R` is included in the tree.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

In the DataLayer protocol, peers exchange `ProofOfInclusion` objects to prove that a key-value pair exists in a shared tree. A malicious peer can send a serialized `ProofOfInclusion` with `layers=[]` and `node_hash` set to the known root. Any verifier that calls only `proof.valid()` and `proof.root_hash() == expected_root` will accept the forged proof, allowing the attacker to falsely assert inclusion of arbitrary data in the DataLayer tree without possessing the actual key-value pair.

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type with public Python-accessible constructors. Any DataLayer peer interaction that involves receiving and verifying a proof from an untrusted party is a reachable entry path. The attack requires only knowledge of the expected tree root (which is typically public) and the ability to send a crafted serialized `ProofOfInclusion`.

### Recommendation

Add a minimum-length check in `valid()` to reject proofs with empty `layers` unless the tree is provably a single-element tree (which requires additional context). At minimum:

```rust
pub fn valid(&self) -> bool {
    if self.layers.is_empty() {
        // A proof with no layers is only valid if the tree has exactly one
        // element and node_hash IS the root. This cannot be verified without
        // external context, so reject it here.
        return false;
    }
    // ... existing hash chain verification ...
}
```

Alternatively, require callers to separately verify that `proof.node_hash` matches the expected leaf hash (computed from the claimed key and value) before trusting `proof.valid()`. Document this requirement explicitly in the API.
