### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Freezal of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by a contract is actually a declared class. Any contract can call `replace_class` with an arbitrary, undeclared class hash, permanently setting its own class to a non-existent implementation. All funds held by that contract become permanently frozen because no future entry point can be dispatched.

### Finding Description
In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads the caller's current `StateEntry` and writes a new `StateEntry` with the caller-supplied `class_hash` directly into `contract_state_changes`, with no check that the supplied hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The inline TODO comment explicitly acknowledges the missing validation. The `contract_class_changes` dictionary (which maps class hashes to compiled class hashes) is never consulted. The OS accepts and commits any felt value as the new class hash. [1](#0-0) 

The analog to the external report is direct: just as `setCeilingStrategy` / `setPricingStrategy` allowed a privileged party to swap out critical crowdsale parameters mid-run without restriction, `execute_replace_class` allows any contract to swap out its own class hash (the parameter governing all future dispatch) to an arbitrary value without restriction.

### Impact Explanation
**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, every subsequent call to that contract will fail at class-lookup time because no compiled class exists for that hash. The contract's storage and token balances become permanently inaccessible. There is no recovery path: the class hash is committed to the proven state update, and no future transaction can re-enter the contract to restore it.

### Likelihood Explanation
The attack surface is reachable by any unprivileged contract deployer or user:

1. A user deploys a contract (or interacts with an existing upgradeable contract that exposes `replace_class`).
2. The contract calls the `replace_class` syscall with an arbitrary felt (e.g., `0xdeadbeef`) that has never been declared.
3. The OS writes the invalid class hash into `contract_state_changes` without rejection.
4. The state update is proven and committed on-chain.
5. All funds in the contract are permanently frozen.

No privileged key, operator cooperation, or network-level attack is required. The only prerequisite is the ability to deploy or interact with a contract — the baseline capability of any StarkNet user.

### Recommendation
Before writing the new `StateEntry`, verify that `class_hash` exists in `contract_class_changes` (i.e., has a non-zero compiled class hash entry). The check should mirror the validation already performed during `execute_declare_transaction`, where `dict_update` with `prev_value=0` enforces that only previously declared classes are accepted. Concretely, perform a `dict_read` on `contract_class_changes` keyed by `request.class_hash` and assert the result is non-zero before proceeding. [2](#0-1) 

### Proof of Concept

1. Deploy a contract `VictimVault` that holds ETH/ERC-20 tokens and exposes an `upgrade(new_class_hash)` entry point that calls `replace_class(new_class_hash)`.
2. Fund `VictimVault` with tokens.
3. Call `upgrade(0x1)` — a class hash that has never been declared on-chain.
4. The OS executes `execute_replace_class`:
   - Gas is deducted successfully.
   - `request.class_hash = 0x1` is written into `contract_state_changes` for `VictimVault`'s address.
   - No lookup into `contract_class_changes` occurs; the TODO guard is absent.
5. The block is proven and the state update is committed.
6. Any subsequent call to `VictimVault` (e.g., `withdraw`) fails: the OS looks up class hash `0x1` in the compiled class table, finds nothing, and the execution aborts.
7. All tokens in `VictimVault` are permanently frozen.

The root cause is exclusively in `execute_replace_class` in `syscall_impls.cairo` at the acknowledged TODO on line 898, which is a necessary step in every `replace_class` syscall execution path. [3](#0-2)
