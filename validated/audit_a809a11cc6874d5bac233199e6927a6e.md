Audit Report

## Title
Stranded Excess `msg.value` Stolen via Unguarded `refundETH()` or Zero-Value WETH Swap — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
Every `payable` swap entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) accepts ETH but never refunds the surplus. `pay()` settles WETH obligations from `address(this).balance` — the router's entire native balance — ignoring the `payer` argument when sufficient ETH is present. Any ETH stranded from a prior caller is immediately claimable by any subsequent caller either via a zero-value WETH swap or a direct call to the unguarded `refundETH()`.

## Finding Description
`PeripheryPayments.pay()` (lines 73–84) reads `uint256 nativeBalance = address(this).balance` and, when `nativeBalance >= value`, wraps exactly `value` wei from the router's total balance and transfers WETH to the pool — the `payer` argument is completely ignored. No swap function refunds unused ETH after the callback settles. `refundETH()` (lines 57–63) sends `address(this).balance` to `msg.sender` with no access control. The `receive()` guard (lines 32–34) only blocks plain ETH transfers from non-WETH addresses; ETH arriving via `payable` function calls bypasses it entirely, so surplus ETH is freely stranded. The `_justPayCallback` path (lines 192–199) is the direct call site that invokes `pay()` with the victim's `payer` address, which is then ignored when native balance suffices.

**Exploit path A (refundETH drain):**
1. Alice calls `exactOutputSingle{value: 2 ether}(...)` where the pool consumes only 1 ether of WETH input. `pay()` deposits 1 ether; 1 ether remains on the router.
2. Bob calls `refundETH()` in any subsequent transaction. He receives Alice's 1 ether with no preconditions.

**Exploit path B (zero-value WETH swap):**
1. Same setup — 1 ether stranded on router.
2. Bob calls `exactInputSingle{value: 0}(params)` with `tokenIn = WETH`, `amountIn = 1 ether`. Inside `_justPayCallback`, `pay(WETH, Bob, pool, 1 ether)` is called. `address(this).balance = 1 ether >= 1 ether`, so Alice's ETH is wrapped and sent to the pool. Bob receives tokens; Bob spent nothing.

Existing guards are insufficient: `amountInMaximum`/`amountOutMinimum` slippage checks only bound the swap amounts, not the ETH source; the callback pool authentication only verifies the caller is the expected pool, not that the payer's ETH is being used.

## Impact Explanation
Direct loss of user ETH principal. The loss equals `msg.value_victim − amountActuallyConsumed`, which is unbounded. Both attack paths are executable by any unprivileged caller with no setup beyond monitoring the router's ETH balance. This meets the Critical/High threshold for direct loss of user principal under Sherlock contest rules.

## Likelihood Explanation
`exactOutputSingle` and `exactOutput` are the primary triggers: users routinely send conservative excess ETH because the exact input amount is unknown before execution. `exactInputSingle` and `exactInput` are also affected when users send more ETH than `amountIn`. The `refundETH()` drain requires only a single call with no ETH and no swap — it is the lowest-friction attack path. Both paths are repeatable across every transaction that strands ETH.

## Recommendation
1. **Track per-call ETH delta**: snapshot `address(this).balance - msg.value` at entry to each `payable` swap function and pass only `msg.value` (or the delta) as the available native ETH to `pay()`, rather than `address(this).balance`.
2. **Auto-refund surplus**: at the end of each `payable` swap function, refund `address(this).balance` to `msg.sender`.
3. **Restrict `refundETH`**: remove it as a standalone public function or gate it to the original depositor; expose it only through `multicall` where the caller is the same transaction initiator.

## Proof of Concept
