### Title
`execute_deploy` Syscall Unconditionally Returns `failure_flag=0`, Masking Constructor Failures — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in the StarkNet OS always writes `failure_flag=0` into the syscall response regardless of whether the deployed contract's constructor succeeded or reverted. Because `revert_log` is an implicit argument of `execute_deploy` and is passed through to `deploy_contract`, constructors can and do revert — undoing all state changes — yet the caller contract is told the deployment succeeded. This prevents any caller from detecting a failed deployment, causing it to proceed with incorrect state assumptions and enabling direct loss of funds.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–553), after calling `deploy_contract`, the response header is unconditionally written with `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` function returns only `(retdata_size, retdata)` — it does not surface an `is_reverted` flag to the caller. The TODO comment explicitly acknowledges this: *"support failures."*

Critically, `revert_log` is declared as an implicit argument of `execute_deploy`:

```cairo
func execute_deploy{
    ...
    revert_log: RevertLogEntry*,
    ...
}(...)
``` [2](#0-1) 

Because `revert_log` is an implicit argument, it is threaded through to `deploy_contract` automatically. This is consistent with how `deploy_contract` is called in `execute_deploy_account_transaction`, where a fresh `revert_log` is explicitly initialized before the call:

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [3](#0-2) 

This confirms that `deploy_contract` participates in the revert mechanism. When a constructor reverts, `handle_revert` is invoked (as defined in `execute_entry_point.cairo`), undoing all state changes — the contract is **not** registered in the state. Yet the syscall response always reports `failure_flag=0`. [4](#0-3) 

The result is a permanent state inconsistency: the OS tells the caller "deployment succeeded" while the actual on-chain state has no contract at the computed address.

---

### Impact Explanation

**Critical. Direct loss of funds.**

A factory contract (a common DeFi primitive) that:
1. Calls the `deploy` syscall to create a sub-contract, and
2. Transfers funds to the newly deployed address

…is unconditionally vulnerable. If the constructor reverts:
- The contract is **not** deployed (state is rolled back via `handle_revert`)
- The factory receives `failure_flag=0` and believes deployment succeeded
- The factory transfers funds to the address — which has no contract
- The funds are sent to an empty address and are effectively frozen or claimable by whoever later deploys at that address

This is a protocol-level bug: no contract can work around it, because the OS itself suppresses the failure signal.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. A factory contract that uses the `deploy` syscall and sends funds to the deployed address after the call — a standard DeFi factory pattern.
2. A constructor that can be made to revert by an attacker-controlled input (e.g., by controlling the `constructor_calldata` passed to the factory, or by deploying a class whose constructor conditionally panics).

Both conditions are realistic. Any unprivileged transaction sender can trigger this by calling a vulnerable factory contract with crafted calldata.

---

### Recommendation

Modify `deploy_contract` to return an `is_reverted` flag (mirroring the return signature of `execute_entry_point`). In `execute_deploy`, use this flag to set `failure_flag` in the response header:

```cairo
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This is exactly what the existing TODO comment anticipates.

---

### Proof of Concept

1. Declare a contract class `FailingClass` whose constructor unconditionally reverts (e.g., `assert 1 = 0`).
2. Deploy a factory contract `Factory` that:
   - Accepts a `class_hash` and `calldata` argument
   - Calls `deploy(class_hash, calldata, ...)`
   - Immediately transfers 1000 STRK to the computed contract address
3. Call `Factory` with `class_hash = FailingClass`.
4. **Observed behavior (buggy):** The constructor reverts; the contract is not deployed; `failure_flag=0` is returned; the factory transfers 1000 STRK to the empty address; funds are frozen.
5. **Expected behavior (correct):** `failure_flag=1` is returned; the factory detects failure and does not transfer funds.
