Audit Report

## Title
Unrestricted `refundETH` / `sweepToken` / `unwrapWETH9` allow any caller to drain surplus ETH and tokens stranded in the router â€” (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH`, `sweepToken`, and `unwrapWETH9` carry no caller restriction and accept attacker-controlled recipients. ETH surplus accumulates in the router whenever a user calls `exactOutputSingle` (or `exactInputSingle`) directly with `msg.value` exceeding the actual `amountIn` consumed, because `pay` wraps only the exact `value` needed and leaves the remainder as raw ETH in the contract. Any third party can immediately drain that balance by calling `refundETH()`.

## Finding Description

**Unrestricted settlement helpers:**

`refundETH` sends the entire native ETH balance to `msg.sender` with no check on who the caller is. [1](#0-0) 

`sweepToken` and `unwrapWETH9` accept an arbitrary `recipient` with no caller check. [2](#0-1) 

**How ETH accumulates:**

Inside `pay`, when `token == WETH` and the contract's native balance covers the required `value`, only exactly `value` is wrapped and forwarded to the pool. Any `msg.value` sent above `value` is left as raw ETH in the router:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // wraps only `value`
    IERC20(WETH).safeTransfer(recipient, value);
    // surplus nativeBalance - value stays in router
}
``` [3](#0-2) 

**No auto-refund in `exactOutputSingle`:**

`exactOutputSingle` is `payable` and returns after clearing the callback context with no `refundETH()` call. A user performing an exact-output WETH swap cannot know the precise `amountIn` before execution; they must send at least `amountInMaximum` as ETH to guarantee the swap succeeds. The difference `msg.value âˆ’ amountIn` is stranded. [4](#0-3) 

**`receive()` does not protect against this:** The `receive()` guard only blocks direct ETH transfers from non-WETH addresses. It does not affect ETH arriving as `msg.value` in a function call, which is stored in the contract balance before `pay` is ever invoked. [5](#0-4) 

**Exploit path:**
1. Victim calls `exactOutputSingle{value: amountInMaximum}(...)` with WETH as `tokenIn`.
2. The swap callback invokes `pay(..., value = amountIn)` where `amountIn < amountInMaximum`.
3. `pay` wraps only `amountIn` ETH; surplus `amountInMaximum âˆ’ amountIn` remains in the router.
4. Attacker (any EOA or contract) calls `router.refundETH()` in the same or next block and receives the full surplus.

The same surface applies to `sweepToken(token, 0, attacker)` for any ERC-20 balance that reaches the router (e.g., WETH output from a tokenâ†’WETH swap with `recipient: address(router)` not immediately followed by `unwrapWETH9` in the same multicall).

**Existing guards are insufficient:** The only intended protection is the off-chain convention of wrapping calls in `multicall([swap, refundETH])`, shown in tests but never enforced on-chain. [6](#0-5) 

## Impact Explanation
Direct loss of user principal. The stranded amount equals `msg.value âˆ’ amountIn`, which can be as large as `amountInMaximum`. An attacker monitoring the mempool can front-run or back-run any `exactOutputSingle` call that sends excess ETH and claim the entire surplus via `refundETH()`. `sweepToken(token, 0, attacker)` drains any ERC-20 balance. This is a Critical/High direct theft of user funds with no protocol-side mitigation.

## Likelihood Explanation
Any unprivileged EOA or contract can trigger the exploit with a single call. The precondition â€” surplus ETH in the router â€” arises naturally whenever a user calls `exactOutputSingle` directly (not via `multicall`) with `msg.value > amountIn`, which is the standard safe-slippage pattern. No special permissions, flash loans, or complex setup are required. The attack is repeatable on every such transaction.

## Recommendation
Add an auto-refund at the end of every `payable` swap entry point:

```solidity
function exactOutputSingle(...) external payable returns (uint256 amountIn) {
    // ... existing logic ...
    _clearExpectedCallbackPool();
    // refund any unused ETH to msg.sender
    uint256 surplus = address(this).balance;
    if (surplus > 0) _transferETH(msg.sender, surplus);
}
```

Apply the same pattern to `exactInputSingle`, `exactInput`, and `exactOutput`. Alternatively, restrict `refundETH`, `sweepToken`, and `unwrapWETH9` to `msg.sender == tx.origin` or add a `checkDeadline`-style reentrancy guard that limits who may call them outside of a `multicall` context. The cleanest fix is the auto-refund approach, which eliminates the stranding window entirely.

## Proof of Concept

```solidity
function test_attacker_steals_surplus_eth() public {
    address attacker = makeAddr("attacker");
    uint128 amountOut = 1_500;
    (uint256 quotedIn,) = quoter.quoteHypotheticalExactOutputSingle(
        address(pool), true, amountOut, 0, TEST_BID_X64, TEST_ASK_X64
    );
    // User sends 2x the quoted amount as ETH (safe-slippage pattern)
    uint256 sentValue = quotedIn * 2;

    vm.prank(swapper);
    router.exactOutputSingle{value: sentValue}(
        IMetricOmmSimpleRouter.ExactOutputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountOut: amountOut,
            amountInMaximum: uint128(sentValue),
            recipient: recipient,
            deadline: type(uint256).max,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // Surplus ETH is now stranded in the router
    assertGt(address(router).balance, 0, "surplus stranded");

    // Attacker drains it
    vm.prank(attacker);
    router.refundETH();
    assertGt(attacker.balance, 0, "attacker received stolen ETH");
    assertEq(address(router).balance, 0, "router drained");
}
```
