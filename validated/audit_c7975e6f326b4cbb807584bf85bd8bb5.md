### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new `class_hash` supplied by the caller is a declared class. An unprivileged contract deployer can invoke `replace_class` with an arbitrary, undeclared class hash, permanently corrupting the contract's state and freezing any funds held within it. The missing check is explicitly acknowledged by a developer TODO comment in the production code.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` reads the requested `class_hash` from the syscall request and writes it directly into `contract_state_changes` without verifying that the hash exists in `contract_class_changes` (i.e., that it was previously declared via a Declare transaction):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The same omission exists in the deprecated path: [2](#0-1) 

The analog to the external report is exact: just as entities could be used without being "initiated," a contract class can be set on a contract without being "declared." The `contract_class_changes` dictionary is the StarkNet OS equivalent of the `"initiated"` slot — it records which class hashes are legitimately declared — but `execute_replace_class` never reads from it to validate the incoming hash.

For comparison, `deploy_contract` correctly enforces the analogous invariant on the contract side by asserting `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` before writing: [3](#0-2) 

No such guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to an undeclared value, every subsequent call to that contract will fail at class resolution time because the OS cannot find the class bytecode. Any ERC-20 tokens, ETH, STRK, or other assets stored in the contract's storage become permanently inaccessible. The state commitment will include the corrupted `class_hash`, and no future transaction can recover the contract because `replace_class` itself requires the contract to be callable.

Setting `class_hash = 0` (`UNINITIALIZED_CLASS_HASH`) is a particularly severe sub-case: `get_contract_state_hash` treats a contract with `class_hash == 0` and non-zero storage or nonce as a non-empty leaf, so the state tree remains consistent while the contract is permanently bricked. [4](#0-3) 

---

### Likelihood Explanation

**High.** The attack requires only:

1. Deploying a contract (permissionless on StarkNet).
2. Sending a single invoke transaction that calls `replace_class(undeclared_hash)`.

No privileged role, leaked key, or external dependency is needed. The attacker controls the `class_hash` field in the syscall request entirely. The explicit `TODO` comment at line 898 confirms the check was intentionally deferred, meaning the vulnerability is present in the current production OS program.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that it exists as a key in `contract_class_changes` (i.e., its compiled class hash is non-zero). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the result is non-zero, mirroring the `prev_value=0` guard used in `execute_declare_transaction`: [5](#0-4) 

Apply the same fix to the deprecated `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Attacker deploys contract `A` containing a `replace_class` call in one of its entry points, and deposits funds into it.
2. Attacker sends an invoke transaction targeting contract `A`, passing `class_hash = 0xdeadbeef` (never declared).
3. `execute_replace_class` in `syscall_impls.cairo` reads `class_hash = 0xdeadbeef`, skips the missing declared-class check (line 898 TODO), and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
4. The OS produces a valid proof and updates the global state root to include the corrupted entry.
5. All subsequent calls to contract `A` fail: the OS cannot resolve class `0xdeadbeef` to any bytecode.
6. All funds in contract `A`'s storage are permanently frozen with no recovery path.
