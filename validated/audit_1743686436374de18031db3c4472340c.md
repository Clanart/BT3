### Title
Hardcoded `L1_HANDLER_L2_GAS_MAX_AMOUNT` Permanently Freezes L1→L2 Message Funds When Handler Exceeds Gas Budget - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/constants.cairo`)

---

### Summary

`L1_HANDLER_L2_GAS_MAX_AMOUNT` is a compile-time constant baked into the OS program that sets the **only** gas budget for every L1 handler transaction. Because `non_reverting_select_execute_entry_point_func` is used for L1 handler execution, any handler that exceeds this budget makes the block **unprovable**. The sequencer's only escape is to mark the transaction as reverted, which leaves the L1→L2 message permanently unconsumed. Any L1 funds locked in a bridge contract waiting for that message to be consumed are permanently frozen.

---

### Finding Description

In `constants.cairo`, the gas budget for L1 handler execution is a single compile-time constant:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;  // 100M gas
``` [1](#0-0) 

This constant is used as the **sole and fixed** gas allocation for every L1 handler in `execute_l1_handler_transaction`:

```cairo
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [2](#0-1) 

The critical detail is the use of `non_reverting_select_execute_entry_point_func`, which internally asserts `is_reverted = 0`:

```cairo
let (is_reverted, retdata_size, retdata, is_deprecated) = select_execute_entry_point_func{...}(...);
assert is_reverted = 0;
``` [3](#0-2) 

If the L1 handler runs out of gas, `select_execute_entry_point_func` returns `is_reverted = 1`, the assertion fails, and the **entire OS proof becomes invalid**. The sequencer cannot include such a transaction in any provable block.

The sequencer's only alternative is to mark the transaction as reverted via the `%{ IsReverted %}` hint, which causes an early return **before** `consume_l1_to_l2_message` is called:

```cairo
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
// ...
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [4](#0-3) 

The message is never consumed. It remains in the L1→L2 message queue indefinitely. Any L1 funds locked pending that message's consumption are permanently frozen.

The disparity between gas limits is stark: regular `__execute__` gets `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` (1.1B gas), while L1 handlers get only `100,000,000` (100M gas) — an 11× difference. [5](#0-4) 

At `STEP_GAS_COST = 100`, 100M gas allows only 1 million VM steps. A handler performing ~2,224 storage writes (`STORAGE_WRITE_GAS_COST = 44,970`) or ~1,092 contract calls (`CALL_CONTRACT_GAS_COST = 91,560`) exhausts the budget. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any L1→L2 bridge or protocol whose L1 handler requires more than 100M gas will have its messages permanently stuck. The L1 contract locks user funds until the corresponding L2 message is consumed. Since the handler can never be included in a provable block, the message is never consumed, and the L1 funds are permanently frozen. This affects all users who sent funds through such a bridge, not just the deployer.

---

### Likelihood Explanation

**Medium.** The 100M gas cap is 11× smaller than the execute gas cap. Any L1 handler that performs batch operations, multiple storage writes, or chains of contract calls can realistically exceed this limit. A contract deployer (unprivileged role) can deliberately or inadvertently deploy such a handler. An L1/L2 message sender (unprivileged role) triggers the freeze by sending a message. The constant is immutable in the compiled OS program — it cannot be adjusted without a full protocol upgrade, meaning any affected messages remain frozen until a new OS version is deployed and the message is retried.

---

### Recommendation

1. Align `L1_HANDLER_L2_GAS_MAX_AMOUNT` with `EXECUTE_MAX_SIERRA_GAS` (1.1B), or at minimum provide a substantially larger budget that reflects realistic handler complexity.
2. If the low cap is intentional, implement a graceful revert path for L1 handlers that run out of gas (analogous to how regular transactions handle reverts), so that the message can be retried or cancelled rather than permanently stuck.
3. Consider making the L1 handler gas limit a configurable OS parameter rather than a compile-time constant, so it can be adjusted via governance without a full OS upgrade.

---

### Proof of Concept

1. Attacker (contract deployer) deploys an L2 contract whose `@l1_handler` function performs >1M VM steps (e.g., iterates over a large array in storage, or chains 1,100+ contract calls).
2. Attacker (or victim) sends an L1→L2 message targeting that contract, locking ETH/ERC20 tokens in the L1 bridge.
3. The sequencer attempts to execute the L1 handler with `remaining_gas = 100_000_000`.
4. The handler exhausts gas; `select_execute_entry_point_func` returns `is_reverted = 1`.
5. `assert is_reverted = 0` fails → block is unprovable if the transaction is included.
6. Sequencer marks the transaction as reverted → early return before `consume_l1_to_l2_message`.
7. The L1→L2 message remains unconsumed in the queue forever.
8. The L1 bridge funds are permanently frozen — no mechanism in the OS allows the message to be processed with more gas.
