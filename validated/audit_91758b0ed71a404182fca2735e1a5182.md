### Title
Missing Balance-Before/After Check in `initTransfer` Allows Fee-on-Transfer Tokens to Undercollateralize the Bridge - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.sol`'s `initTransfer` function accepts a caller-supplied `amount` parameter and uses it directly in the emitted `InitTransfer` event without verifying the actual number of tokens received by the contract. For ERC20 tokens with transfer fees (fee-on-transfer tokens), the bridge records a higher amount than it physically holds, causing the cross-chain accounting to diverge and the bridge to become undercollateralized.

### Finding Description

In `initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract pulls tokens from the caller and emits the event using the caller-supplied `amount`:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← requested amount, not verified received amount
    );
}
...
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ← same unverified value relayed cross-chain
    fee,
    nativeFee,
    recipient,
    message
);
```

No balance snapshot is taken before the transfer, and no balance snapshot is compared after it. The `amount` value in the `InitTransfer` event is therefore the caller's declared intent, not the actual tokens received.

The `InitTransfer` event is the authoritative source that the NEAR bridge reads to determine how many tokens to release to the recipient. On NEAR, `fin_transfer_callback` denormalizes and releases `amount - fee` tokens. If the EVM bridge only holds `amount - transfer_fee` (where `transfer_fee` is the token's internal fee), the EVM vault is undercollateralized by exactly `transfer_fee` per bridging round-trip.

The analogous pattern from the external report (H-02) is precisely this: the accounting mechanism trusts a declared value rather than measuring the actual balance change, causing the recorded amount to diverge from the real custody.

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

Each `initTransfer` call with a fee-on-transfer token records more value than the bridge holds. When users later bridge back from NEAR to EVM, the EVM bridge attempts to release the full recorded amount but lacks sufficient token balance. Repeated use drains the EVM vault's real reserves relative to its recorded liabilities, eventually causing legitimate withdrawals to fail (permanent freezing of user funds) or, if the shortfall is small enough per transaction, silently breaking the 1:1 collateralization invariant across the bridge.

### Likelihood Explanation

**Medium.** The `else` branch in `initTransfer` accepts any ERC20 token that is not a registered bridge token or custom-minter token — there is no allowlist or explicit rejection of fee-on-transfer tokens. Fee-on-transfer tokens exist in production (e.g., various reflection/deflationary tokens). Any user can call `initTransfer` with such a token without any privileged access. The attacker does not need a leaked key, colluding MPC signers, or any other privileged capability.

### Recommendation

Capture the contract's token balance before and after the `safeTransferFrom` call, and use the measured delta as the canonical `amount` for the event and all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived (cast to uint128 with overflow check) in the event and storage
```

This mirrors the fix recommended in H-02: measure the real balance change rather than trusting the caller-supplied value.

### Proof of Concept

1. Deploy or use any ERC20 token that deducts a 1% fee on every transfer (fee-on-transfer token). Ensure it is not registered as a bridge token or custom-minter token in `OmniBridge`.
2. Call `OmniBridge.initTransfer(feeToken, 1_000_000, 0, 0, "near:recipient.near", "")`.
3. `safeTransferFrom` transfers 1,000,000 tokens from the caller, but the token's internal fee deducts 10,000, so the bridge receives only 990,000 tokens.
4. The `InitTransfer` event is emitted with `amount = 1_000_000`.
5. The NEAR bridge relayer picks up the event and calls `fin_transfer` on NEAR, releasing 1,000,000 tokens to `recipient.near`.
6. The EVM bridge now holds 990,000 tokens but has a recorded liability of 1,000,000 — a 10,000-token shortfall per operation.
7. Repeating this 99 times exhausts the bridge's real reserves entirely while NEAR-side supply remains fully backed on paper, permanently freezing the last legitimate user's withdrawal. [1](#0-0) [2](#0-1) [3](#0-2)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
