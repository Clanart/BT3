### Title
Integer Truncation in `MinGasPrice` Lower-Bound Enforcement Allows Transactions Below the Intended Minimum Fee Floor - (File: `x/feemarket/keeper/eip1559.go`)

---

### Summary

`CalculateBaseFee` in `x/feemarket/keeper/eip1559.go` enforces `MinGasPrice` as a lower bound on the computed base fee by calling `params.MinGasPrice.TruncateInt().BigInt()`. Because `MinGasPrice` is stored as an `sdk.Dec` (an 18-decimal-precision rational), any fractional value (e.g., `0.5 aevmos/gas`) is silently truncated to `0`, making the lower-bound enforcement completely ineffective. A transaction whose effective gas price is above `0` but below the intended `MinGasPrice` will pass the base-fee floor check and be included in a block, bypassing the global minimum fee protection.

---

### Finding Description

`MinGasPrice` is stored as `sdkmath.LegacyDec` (18-decimal precision). In `CalculateBaseFee`:

```go
// x/feemarket/keeper/eip1559.go line 103
minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
```

`TruncateInt()` discards the fractional part. If governance sets `MinGasPrice` to any value in `(0, 1)` — for example `0.5` — the truncation yields `0`, and the base fee is allowed to fall to `0`. The same truncation appears in `rpc/backend/utils.go` `CalcBaseFee` at line 160:

```go
minGasPrice := p.MinGasPrice.TruncateInt().BigInt()
return ethermint.BigMax(baseFee, minGasPrice)
```

The `MinGasPriceDecorator` in `ante/cosmos/min_gas_price.go` and `CheckEthMinGasPrice` in the same file correctly use the full `sdk.Dec` value for their per-transaction checks. However, the base-fee floor in `CalculateBaseFee` is the consensus-level enforcement that prevents the base fee itself from dropping below `MinGasPrice`. When this floor is zeroed out by truncation, the base fee can fall to `0` even when `MinGasPrice > 0`, and any transaction with `effectiveGasPrice > 0` will pass the ante-handler's `feeCap >= baseFee` check (line 86 of `ante/evm/fee_checker.go`), because `baseFee` is now `0`.

The `NewDynamicFeeChecker` in `ante/evm/fee_checker.go` computes:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // integer division, truncates
...
if feeCap.LT(baseFeeInt) { return error }
```

With `baseFeeInt = 0`, any non-zero `feeCap` passes. The `CheckEthMinGasPrice` check uses the full `sdk.Dec` `MinGasPrice`, but it is only called from the `EthMinGasPriceDecorator`, which is a separate decorator. If the base fee has already been set to `0` by `CalculateBaseFee`, the `NewDynamicFeeChecker` path (used for Cosmos-wrapped EVM txs via `newCosmosAnteHandler`) will accept any fee above `0`, bypassing the intended minimum.

Additionally, `feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))` at line 83 of `ante/evm/fee_checker.go` performs integer division, silently truncating the per-gas price. A user who submits `fee = baseFee * gas - 1` (one unit below the required total) will have `feeCap = baseFee - 1` after truncation only if `gas > 1`, but if `fee = baseFee * gas` and `gas` is large, the truncation can produce `feeCap = baseFee - epsilon` rounded down, causing a valid fee to be incorrectly rejected or an underpaying fee to be accepted depending on rounding direction.

---

### Impact Explanation

**High.** The primary impact is a fee market accounting bug: the base fee floor is silently zeroed when `MinGasPrice` has a fractional value, allowing transactions to be included at an effective gas price below the governance-set minimum. This means:

1. Validators accept and include transactions that should be rejected by the global minimum fee policy.
2. Fee revenue is mis-accounted: the effective fee deducted from users can be lower than the intended minimum, constituting a valid-user-funds mis-accounting scenario.
3. The base fee can collapse to `0` across all blocks once the truncated floor takes effect, permanently disabling the fee market floor until governance corrects the parameter.

This matches the allowed impact: **"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."**

---

### Likelihood Explanation

`MinGasPrice` is a governance parameter of type `sdk.Dec`. Chains routinely set sub-integer gas prices (e.g., `0.001 aevmos` or `0.25 uatom`) to allow low-cost transactions while still preventing spam. Any such setting silently truncates to `0`. The bug is triggered by any governance proposal that sets `MinGasPrice` to a non-integer positive value — a realistic and common operational scenario. No privileged access beyond governance participation is required.

---

### Recommendation

Replace `TruncateInt()` with a ceiling operation (`Ceil().RoundInt()`) when converting `MinGasPrice` to a `*big.Int` for use as a lower bound. This ensures the base fee never falls below the intended minimum:

```go
// x/feemarket/keeper/eip1559.go line 103
minGasPrice := params.MinGasPrice.Ceil().RoundInt().BigInt()
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
```

Apply the same fix in `rpc/backend/utils.go` line 160. Additionally, add a validation in `validateMinGasPrice` that warns or rejects fractional values if the intent is always integer wei amounts, or document explicitly that fractional values are supported and the ceiling semantics are intentional.

---

### Proof of Concept

1. Governance sets `MinGasPrice = sdk.NewDecWithPrec(5, 1)` (i.e., `0.5`).
2. `CalculateBaseFee` is called at `BeginBlock`. The base fee is currently `1`.
3. Block gas used is below target, so `baseFeeDelta > 0` and the formula computes `newBaseFee = parentBaseFee - baseFeeDelta = 0`.
4. `minGasPrice := params.MinGasPrice.TruncateInt().BigInt()` → `minGasPrice = 0`.
5. `ethermint.BigMax(0, 0) = 0` → base fee is set to `0`.
6. In the next block, `NewDynamicFeeChecker` evaluates `feeCap.LT(baseFeeInt)` where `baseFeeInt = 0`. Any `feeCap >= 0` passes.
7. A transaction with `fee = 0` (or any positive amount) is accepted and included, bypassing the `0.5` minimum gas price floor.
8. The `EthMinGasPriceDecorator` (`CheckEthMinGasPrice`) would still catch this for direct EVM transactions, but the `NewDynamicFeeChecker` path used for Cosmos-wrapped transactions does not call `CheckEthMinGasPrice` — it only checks `feeCap >= baseFee`, which is `0 >= 0 = true`.
