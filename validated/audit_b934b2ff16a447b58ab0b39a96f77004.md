### Title
Unsigned `ContractCodeResponse::V1` Accepted Without Sender Verification, Allowing Any Peer to Inject Arbitrary Contract Code into Witness Reconstruction Cache — (`chain/client/src/stateless_validation/validate.rs`)

---

### Summary

`validate_contract_code_response` skips all sender-identity checks when `ProtocolFeature::SignedContractCodeResponse` is not yet enabled. Any network peer can send a `ContractCodeResponse::V1` carrying arbitrary contract bytes for any relevant `ChunkProductionKey`, and the bytes are unconditionally stored into the `PartialEncodedStateWitnessTracker` contract-code cache. When the cache entry finalizes, witness validation runs against the injected (wrong) code, produces a mismatched state root, and the validator never sends a chunk endorsement.

---

### Finding Description

`validate_contract_code_response` in `chain/client/src/stateless_validation/validate.rs` performs only a height/epoch relevance check and then conditionally verifies the responder's signature:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)
}
``` [1](#0-0) 

When `SignedContractCodeResponse` is not enabled, the `ContractCodeResponse::V1` variant carries no signature field and `verify_signature` always returns `false` for it — but the check is never reached: [2](#0-1) 

The handler in `PartialWitnessActor` passes every response that clears the relevance gate directly into `store_accessed_contract_codes`:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    if !validate_contract_code_response(...)?. is_relevant() { return Ok(()); }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
``` [3](#0-2) 

`store_accessed_contract_codes` feeds a `CacheUpdate::AccessedContractCodes` into the per-shard `CacheEntry`, which calls `try_finalize()`. If the entry already holds enough witness parts (i.e., the legitimate partial-witness messages arrived first), finalization triggers immediately with the injected contract bytes, causing `validate_chunk_state_witness_impl` to apply the wrong code and produce a state-root mismatch: [4](#0-3) [5](#0-4) 

The `SignedContractCodeResponse` feature was introduced precisely to close this gap, but until the protocol version that enables it is activated in production, the V1 path is live and unguarded.

---

### Impact Explanation

A malicious peer that can observe partial-witness traffic (which is broadcast to all chunk validators) can:

1. Detect a new chunk being distributed (via `PartialEncodedStateWitness` messages on the network).
2. Construct a `ContractCodeResponse::V1` with a valid `ChunkProductionKey` and arbitrary (wrong) compressed contract bytes.
3. Send it to one or more targeted chunk validators before the legitimate response arrives.
4. The validator stores the injected bytes, finalizes the cache entry, runs witness validation with wrong contract code, gets a state-root mismatch, and never emits a `ChunkEndorsement`.

If enough validators are targeted, the chunk fails to accumulate the required endorsement quorum, stalling block production for that shard. The attacker needs no stake, no validator key, and no privileged network position — only the ability to send a P2P message to a validator node.

---

### Likelihood Explanation

The attack requires:
- The `SignedContractCodeResponse` protocol feature to be inactive (i.e., the current production protocol version predates it).
- The ability to send a `ContractCodeResponse` message to a validator node before the legitimate chunk producer's response arrives — a narrow but realistic race window, since the validator's `ContractCodeRequest` is sent over the network and the response path is not authenticated.

Both conditions are simultaneously satisfiable in any epoch where the feature flag is not yet enabled.

---

### Recommendation

Remove the feature-flag guard around `validate_witness_contract_code_response_signature` in `validate_contract_code_response`. The signature check should be unconditional: if the response is V1 (unsigned), it must be rejected outright rather than accepted silently. Alternatively, accelerate the activation of `ProtocolFeature::SignedContractCodeResponse` so that V1 responses are no longer generated or accepted on the network.

---

### Proof of Concept

**Attacker steps (no validator key required):**

1. Connect to the NEAR P2P network as an ordinary peer.
2. Observe `PartialEncodedStateWitness` messages to learn the current `ChunkProductionKey` (`epoch_id`, `shard_id`, `height_created`) for an active chunk.
3. Craft a `ContractCodeResponse::V1`:
   ```
   ContractCodeResponseV1 {
       next_chunk: ChunkProductionKey { epoch_id, shard_id, height_created },
       compressed_contracts: <compress(arbitrary_wrong_wasm_bytes)>,
   }
   ```
4. Send this message directly to a targeted chunk validator node.
5. `validate_contract_code_response` passes (relevance check only, no signature check for V1).
6. `decompress_contracts()` succeeds.
7. `store_accessed_contract_codes(key, wrong_contracts)` stores the injected bytes.
8. `try_finalize()` triggers (if witness parts already present) → `validate_chunk_state_witness_impl` applies wrong contract code → state-root mismatch → `Error::InvalidChunkStateWitness` → no endorsement sent.

**Exact corrupted value:** The `Vec<CodeBytes>` stored under `ChunkProductionKey` in `PartialEncodedStateWitnessTracker`'s `CacheEntry::accessed_contracts` field, replacing the legitimate contract code with attacker-controlled bytes. [1](#0-0) [3](#0-2) [6](#0-5)

pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,
    V2(ContractCodeResponseV2) = 1,
}

impl ContractCodeResponse {
    pub fn encode(
        next_chunk: ChunkProductionKey,
        contracts: &Vec<CodeBytes>,
        signer: &ValidatorSigner,
        protocol_version: ProtocolVersion,
    ) -> std::io::Result<Self> {
        if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
            ContractCodeResponseV2::encode(next_chunk, contracts, signer).map(Self::V2)
        } else {
            ContractCodeResponseV1::encode(next_chunk, contracts).map(Self::V1)
        }
    }

    pub fn chunk_production_key(&self) -> &ChunkProductionKey {
        match self {
            Self::V1(v1) => &v1.next_chunk,
            Self::V2(v2) => &v2.inner.next_chunk,
        }
    }

    /// Account that produced this response. Available only for signed variants.
    pub fn responder(&self) -> Option<&AccountId> {
        match self {
            Self::V1(_) => None,
            Self::V2(v2) => Some(&v2.inner.responder),
        }
    }

    pub fn decompress_contracts(&self) -> std::io::Result<Vec<CodeBytes>> {
        let compressed_contracts = match self {
            Self::V1(v1) => &v1.compressed_contracts,
            Self::V2(v2) => &v2.inner.compressed_contracts,
        };
        compressed_contracts.decode().map(|(data, _size)| data)
    }

    /// Verifies the signature for signed variants. Returns `false` for
    /// unsigned variants since there is nothing to verify.
    pub fn verify_signature(&self, public_key: &PublicKey) -> bool {
        match self {
            Self::V1(_) => false,
            Self::V2(v2) => v2.verify_signature(public_key),
        }
    }
