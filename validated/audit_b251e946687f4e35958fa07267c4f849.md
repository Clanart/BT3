### Title
Missing Upper Bound on `max_price_per_unit` in `pack_resource_bounds` Enables Transaction Hash Collisions — (File: `transaction_hash/transaction_hash.cairo`)

---

### Summary

The `pack_resource_bounds` function enforces a lower bound on `max_price_per_unit` but **no upper bound**. This allows two structurally distinct `ResourceBounds` inputs to produce the same packed felt, and therefore the same transaction hash. Because the transaction hash is the signed commitment, a valid user signature becomes reusable across two distinct transactions with different resource bounds.

---

### Finding Description

In `pack_resource_bounds`, the packing formula is:

```cairo
(resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
    resource_bounds.max_price_per_unit;
```

`max_amount` is correctly bounded to `[0, 2^64 − 1]` via `assert_nn_le`. However, `max_price_per_unit` is only checked with `assert_nn`, which in Cairo permits values up to `(P−1)/2 ≈ 2^251`. [1](#0-0) 

Because `max_price_per_unit` contributes at coefficient 1 and `max_amount` contributes at coefficient `2^128`, any `max_price_per_unit ≥ 2^128` creates an arithmetic carry into the `max_amount` bits. Concretely:

- `{max_amount=A, max_price_per_unit=2^128 + Y}` produces packed felt `(resource * 2^64 + A + 1) * 2^128 + Y`
- `{max_amount=A+1, max_price_per_unit=Y}` produces the **identical** packed felt

Both inputs pass the current validation (`assert_nn_le(max_amount, 2^64−1)` and `assert_nn(max_price_per_unit)`), yet they are structurally different `ResourceBounds` structs that hash identically. [2](#0-1) 

This packed felt feeds directly into `hash_fee_fields`, which feeds into `hash_tx_common_fields`, which is the foundation of every v3 transaction hash (`compute_invoke_transaction_hash`, `compute_declare_transaction_hash`, `compute_deploy_account_transaction_hash`). [3](#0-2) 

---

### Impact Explanation

The transaction hash is the exact data signed by the user's account contract in `__validate__`. If two transactions share the same hash, a signature produced for T1 is cryptographically valid for T2.

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;

func hash_fee_fields{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    tip: felt, resource_bounds: ResourceBounds*, n_resource_bounds: felt
) -> felt {
    alloc_locals;

    let (local data_to_hash: felt*) = alloc();
    assert data_to_hash[0] = tip;
    assert_nn_le(tip, 2 ** 64 - 1);

    static_assert L1_GAS_INDEX == 0;
    static_assert L2_GAS_INDEX == 1;
    static_assert L1_DATA_GAS_INDEX == 2;

    with_attr error_message("Invalid number of resource bounds: {n_resource_bounds}.") {
        assert n_resource_bounds = 3;
    }

    // L1 gas.
    let l1_gas_bounds = resource_bounds[L1_GAS_INDEX];
    assert l1_gas_bounds.resource = L1_GAS;
    assert data_to_hash[1] = pack_resource_bounds(l1_gas_bounds);

    // L2 gas.
    let l2_gas_bounds = resource_bounds[L2_GAS_INDEX];
    assert l2_gas_bounds.resource = L2_GAS;
    assert data_to_hash[2] = pack_resource_bounds(l2_gas_bounds);

    // L1 data gas.
    let l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];
    assert l1_data_gas_bounds.resource = L1_DATA_GAS;
    assert data_to_hash[3] = pack_resource_bounds(l1_data_gas_bounds);

    let (hash) = poseidon_hash_many(n=n_resource_bounds + 1, elements=data_to_hash);
    return hash;
