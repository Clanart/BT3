### Title
`deployToken` / `deploy_token` Signed Payload Lacks Destination Chain ID, Enabling Cross-Chain Signature Replay for Unauthorized Token Deployment — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `MetadataPayload` signed by the NEAR MPC signer for `deployToken` / `deploy_token` does not include a destination chain ID. Because the same MPC signing path (`"bridge-1"`) is used for all destination chains, the resulting `nearBridgeDerivedAddress` is identical across EVM, Solana, and StarkNet. A signature produced for one chain is therefore cryptographically valid on every other chain. Any observer who captures the publicly emitted signature can replay it to deploy the token on any chain where it was not intended.

---

### Finding Description

**NEAR signing side** — `log_metadata_callback` constructs and signs a `MetadataPayload` that contains only `{prefix, token, name, symbol, decimals}` — no destination chain ID:

```rust
// near/omni-bridge/src/lib.rs  lines 341-360
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,   // ← no chain ID
};
...
ext_signer::ext(self.mpc_signer.clone())
    .sign(SignRequest {
        payload,
        path: SIGN_PATH.to_owned(),   // "bridge-1" — same for every chain
        key_version: 0,
    })
```

The resulting signature is then **publicly broadcast** on-chain:

```rust
// near/omni-bridge/src/lib.rs  lines 375-383
env::log_str(
    &OmniBridgeEvent::LogMetadataEvent {
        signature,
        metadata_payload,
    }
    .to_log_string(),
);
```

**EVM verification** — `deployToken` hashes only `{PayloadType.Metadata, token, name, symbol, decimals}`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 142-153
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)          // ← no omniBridgeChainId
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

**Contrast with `finTransfer`** — which correctly binds the signature to the destination chain:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 289-308
bytes memory borshEncoded = bytes.concat(
    ...
    bytes1(omniBridgeChainId),   // ← chain ID present
    Borsh.encodeAddress(payload.tokenAddress),
    ...
);
```

**StarkNet** (`starknet/src/omni_bridge.cairo` lines 202–209) and **Solana** (`solana/programs/bridge_token_factory/src/state/message/deploy_token.rs` lines 19–27) have the identical omission — their `MetadataPayload` Borsh encoding is `{1u8, token, name, symbol, decimals}` with no chain ID, and the Borsh wire format is byte-for-byte identical across all three chains.

Because `SIGN_PATH = "bridge-1"` is constant, the MPC-derived ECDSA key — and therefore `nearBridgeDerivedAddress` / `derived_near_bridge_address` — is the same on every destination chain. A signature that passes `ECDSA.recover(...) == nearBridgeDerivedAddress` on Ethereum will pass the identical check on Base, Arbitrum, Solana, and StarkNet.

---

### Impact Explanation

An attacker who observes the publicly emitted `LogMetadataEvent` on NEAR can extract `(signature, metadata_payload)` and call `deployToken` / `deploy_token` on **any** destination chain, deploying the bridge-controlled token without authorization. Concretely:

1. The bridge contract on the unintended chain gains mint/burn authority over the newly deployed token.
2. The legitimate deployment of that token on the same chain is permanently blocked (`ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`), because the idempotency check prevents a second deployment with corrected metadata.
3. If the NEAR bridge later processes the `DeployToken` event from the unintended chain, it may inadvertently register the token for that chain, opening a path for `finTransfer` minting on a chain the protocol never intended to support.

This matches the allowed impact: **High — signature/MPC verification bypass enabling unauthorized token deployment**.

---

### Likelihood Explanation

The `LogMetadataEvent` containing the full `(signature, metadata_payload)` is emitted as a public NEAR transaction log every time `log_metadata` is called. Any NEAR RPC observer — including any relayer, indexer, or unprivileged user — can extract the signature with zero privilege. Replaying it on another chain requires only a standard transaction call to `deployToken`. No special access, leaked key, or colluding party is needed.

---

### Recommendation

Include the destination chain ID in the signed `MetadataPayload`, mirroring the pattern already used in `TransferMessagePayload`:

```rust
// NEAR side: add destination chain to MetadataPayload
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    destination_chain: chain_kind,   // ← add this
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
```

All destination-chain verifiers (`deployToken` on EVM, `deploy_token` on Solana and StarkNet) must then include the chain's own ID in the hash they verify, so a signature produced for chain A is rejected on chain B.

---

### Proof of Concept

1. User calls `log_metadata("usdc.near")` on the NEAR bridge.
2. NEAR bridge fetches USDC metadata, constructs `MetadataPayload{token="usdc.near", name="USD Coin", symbol="USDC", decimals=6}`, and requests an MPC signature over `keccak256(borsh(payload))` using path `"bridge-1"`.
3. MPC returns signature `σ`. NEAR emits `LogMetadataEvent { signature: σ, metadata_payload }` — publicly visible on-chain.
4. A relayer legitimately calls `deployToken(σ, payload)` on Ethereum (chain ID 1). USDC bridge token is deployed at address `T_eth`.
5. Attacker observes `σ` from step 3. Attacker calls `deployToken(σ, payload)` on Base (chain ID 8453). The EVM bridge on Base computes the identical `keccak256(borsh(payload))`, recovers the same `nearBridgeDerivedAddress`, and passes signature verification — deploying an unauthorized USDC bridge token `T_base` with the Base bridge contract as mint authority.
6. The legitimate Base deployment is now permanently blocked (`ERR_TOKEN_EXIST`). If the NEAR bridge later indexes the Base `DeployToken` event, it may register `T_base` as the canonical USDC address for Base, enabling `finTransfer` minting on a chain the protocol never authorized.
