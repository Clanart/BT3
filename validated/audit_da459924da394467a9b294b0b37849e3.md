### Title
Cairo 0 Class from Reverted Block Served Without Declaration Check on Cache Hit — (`crates/blockifier/src/state/state_reader_and_contract_manager.rs`)

### Summary

`get_compiled_from_class_manager` explicitly skips the declaration check for `RunnableCompiledClass::V0` on a cache hit. The identical revert-stale-cache scenario that the code guards against for Cairo 1 classes is left completely unguarded for Cairo 0 classes. A Cairo 0 class declared in a reverted block remains in the `GlobalContractCache` and is returned unconditionally on the next cache hit, while a cache miss for the same class hash correctly returns `StateError::UndeclaredClassHash`.

### Finding Description

In `get_compiled_from_class_manager`:

```rust
match &runnable_class {
    RunnableCompiledClass::V0(_) => {}          // ← no check at all
    _ => {
        // The Cairo1 class is cached; verify it is declared,
        // since existence in the cache does not guarantee that
        // (it might contain a declared class from a reverted block, for example).
        if !self.state_reader.is_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
}
``` [1](#0-0) 

The developer comment at lines 76–78 explicitly names the reverted-block scenario as the motivation for the V1 check. The V0 arm is an empty no-op — the same risk is present but unmitigated.

On a **cache miss**, the call falls through to `self.state_reader.get_compiled_classes(class_hash)`. For `SyncStateReader`, that calls `get_contract_class_from_client`, which calls `is_class_declared_at` (covers both Cairo 0 and Cairo 1) and returns `StateError::UndeclaredClassHash` if the class is not in the committed state:

```rust
fn get_contract_class_from_client(&self, class_hash: ClassHash) -> StateResult<ContractClass> {
    let is_class_declared = self.runtime
        .block_on(self.state_sync_client.is_class_declared_at(self.block_number, class_hash))
        ...;
    if !is_class_declared {
        return Err(StateError::UndeclaredClassHash(class_hash));
    }
    ...
}
``` [2](#0-1) 

The two paths therefore produce different results for the same class hash H after a revert:

| Path | Result |
|---|---|
| Cache hit (V0) | `Ok(RunnableCompiledClass::V0(...))` — stale class served |
| Cache miss | `Err(StateError::UndeclaredClassHash(H))` — correct |

The `is_declared` method used for the V1 guard calls `is_cairo_1_class_declared_at`, which returns `false` for Cairo 0 classes by design:

```rust
fn is_declared(&self, class_hash: ClassHash) -> StateResult<bool> {
    self.runtime.block_on(
        self.state_sync_client.is_cairo_1_class_declared_at(self.block_number, class_hash),
    )...
}
``` [3](#0-2) 

This means even if the V0 arm were changed to call `is_declared`, it would always return `false` for Cairo 0 and still serve the stale class. The correct fix requires calling `is_class_declared_at` (not `is_declared`) for V0 classes, or adding a dedicated `is_cairo_0_class_declared_at` check.

### Impact Explanation

A Cairo 0 class declared in a reverted block is served to execution, fee estimation, simulation, or gateway validation without any declaration check. The executed bytecode is wrong relative to the committed state. This satisfies: **"Critical. Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."**

In the gateway/RPC path, this causes fee estimation and simulation to execute against a class that does not exist in the committed state, returning authoritative-looking wrong values. In the block execution path (if the same `StateReaderAndContractManager` wraps a block-execution state reader sharing the same `GlobalContractCache`), a transaction could execute a reverted Cairo 0 class, producing a wrong execution result that is committed to the block.

### Likelihood Explanation

Requires a block revert (chain reorg or sequencer-side revert) that leaves a Cairo 0 class in the `GlobalContractCache`. The cache is not cleared on revert. After the revert, any request for that class hash that hits the cache before eviction triggers the bug. Block reverts are uncommon in normal operation but are a defined part of the protocol. The cache is large (`GLOBAL_CONTRACT_CACHE_SIZE_FOR_TEST` is 600 entries) and LRU-eviction is slow for infrequently-used classes, so the stale entry can persist for many subsequent blocks. [4](#0-3) 

### Recommendation

Apply the same declaration guard to the V0 branch. Because `is_declared` only covers Cairo 1, a separate check is needed:

```rust
match &runnable_class {
    RunnableCompiledClass::V0(_) => {
        // Same revert-stale-cache risk as Cairo 1.
        // is_declared() only covers Cairo 1, so use is_class_declared_at directly.
        if !self.state_reader.is_cairo_0_class_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
    _ => {
        if !self.state_reader.is_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
}
```

Alternatively, extend the `FetchCompiledClasses` trait with a unified `is_class_declared` method that covers both Cairo 0 and Cairo 1, backed by `is_class_declared_at` in `SyncStateReader`, and apply it to both arms.

### Proof of Concept

Concrete call sequence demonstrating the divergence:

1. Block N: `DECLARE` transaction for Cairo 0 class H is executed. `set_and_compile(H, CompiledClasses::V0(...))` populates the `GlobalContractCache`. [5](#0-4) 

2. Block N is reverted. The `GlobalContractCache` is **not** cleared. H is no longer in the committed state; `is_class_declared_at(N+1, H)` returns `false`.

3. Block N+1, cache-hit path: `get_runnable(&H, ...)` returns `Some(RunnableCompiledClass::V0(...))`. The V0 arm is a no-op. `Ok(V0)` is returned and the class is executed. [6](#0-5) 

4. Block N+1, cache-miss path (cache evicted): `get_compiled_classes(H)` → `get_contract_class_from_client(H)` → `is_class_declared_at(N+1, H)` → `false` → `Err(UndeclaredClassHash(H))`. [2](#0-1) 

Steps 3 and 4 produce opposite results for the same class hash H at the same block height, depending solely on whether the cache was warm.
