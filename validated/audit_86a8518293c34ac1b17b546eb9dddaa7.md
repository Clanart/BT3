Audit Report

## Title
Unattributed Router ETH Balance Consumed by Subsequent WETH Swaps, Stealing Prior User's Refund â€” (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` uses `address(this).balance` as a global ETH pool when `token == WETH`, with no per-user attribution. Any ETH left on the router by a prior user (excess `msg.value` from a WETH swap where `refundETH` was not called) is silently consumed to fund a subsequent user's WETH swap, causing the prior user to permanently lose their refund.

## Finding Description

`exactInputSingle` is `payable` but imposes no constraint that `msg.value == params.amountIn`. [1](#0-0) 

When `tokenIn == WETH`, the pool callback fires `_justPayCallback`, which calls `pay(WETH, payer, pool, amountIn)` where `amountIn` is the pool-determined delta â€” not `msg.value`. [2](#0-1) 

Inside `pay`, `nativeBalance = address(this).balance` captures the **total** router ETH balance, not the current caller's contribution. [3](#0-2) 

The `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does not fire for ETH arriving as `msg.value` in a `payable` call, so excess ETH from User A's swap silently accumulates on the router. [4](#0-3) 

When a subsequent user calls `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`, `pay` reads the leftover ETH as `nativeBalance > 0` and uses it to fund the new swap â€” either fully (lines 75â€“77) or partially (lines 78â€“81) â€” without pulling from the new user's WETH allowance. The prior user's ETH is permanently consumed with no refund path. [5](#0-4) 

## Impact Explanation

Direct loss of user principal. The prior user's ETH (which should be refundable via `refundETH`) is silently transferred to a pool on behalf of a different user. The amount is bounded only by how much excess ETH the prior user sent. This satisfies the Critical/High direct-loss-of-user-principal gate.

## Likelihood Explanation

Medium. The precondition is that a user sends `msg.value > amountIn` without calling `refundETH` in the same transaction. This is a realistic mistake for users who call `exactInputSingle` directly rather than via `multicall` + `refundETH`. No privileged attacker setup is required â€” any subsequent WETH swap drains the leftover balance automatically.

## Recommendation

Track only the ETH that arrived with the current call, not the total router balance. Capture `msg.value` at the entry point and pass it down to `pay`, or compare `address(this).balance` before and after the call. Alternatively, enforce `msg.value == 0` when `tokenIn` is not the native-ETH sentinel and require a separate native-ETH swap path that validates the exact amount.

## Proof of Concept

```solidity

//    â†’ pay() wraps 100e18 ETH, sends WETH to pool
//    â†’ 50e18 ETH remains on router; User A does NOT call refundETH

// 2. Attacker calls exactInputSingle: tokenIn=WETH, amountIn=50e18, msg.value=0
//    â†’ pay(): nativeBalance = 50e18 >= 50e18
//    â†’ wraps 50e18 ETH (User A's), sends WETH to pool
//    â†’ NO transferFrom on attacker's WETH balance
```
