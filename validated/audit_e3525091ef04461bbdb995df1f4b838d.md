### Title
ERC1155 Deterministic Address Collision Enables Theft of Bridged ERC1155 Assets - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.deriveDeterministicAddress` maps each ERC1155 `(tokenAddress, tokenId)` pair to a 160-bit address by truncating a 256-bit keccak256 hash. Because the output space is only 160 bits, an attacker who finds two distinct `(tokenAddress, tokenId)` pairs that hash to the same deterministic address can deposit worthless tokens and withdraw valuable ERC1155 tokens locked in the bridge.

### Finding Description

`deriveDeterministicAddress` is defined as:

```solidity
function deriveDeterministicAddress(
    address tokenAddress,
    uint256 tokenId
) public pure returns (address) {
    return address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))));
}
``` [1](#0-0) 

This takes the first 20 bytes of a 32-byte keccak256 hash, discarding 96 bits. For any 160-bit output address `D`, there are `2^96` distinct inputs that produce it after truncation. The birthday paradox means a collision between two attacker-chosen inputs can be found with approximately `2^80` hash evaluations.

The attacker controls both inputs:
- `tokenAddress`: 160 bits — they can deploy contracts at chosen addresses via CREATE2
- `tokenId`: 256 bits — any arbitrary value

Total controllable input: 416 bits, making the collision search space very favorable.

**Attack flow:**

1. Legitimate users call `logMetadata1155(A1, T1)`, setting `multiTokens[D] = {A1, T1}` where `D = deriveDeterministicAddress(A1, T1)`. The bridge accumulates `A1:T1` ERC1155 tokens from users bridging to NEAR. [2](#0-1) 

2. Attacker finds `(A2, T2)` such that `deriveDeterministicAddress(A2, T2) == D`. `A2` is a malicious ERC1155 contract that mints tokens freely.

3. Attacker calls `initTransfer1155(A2, T2, large_amount, ...)`. This function does **not** check `multiTokens` — it simply computes the deterministic address and emits `InitTransfer` with `deterministicToken = D`. [3](#0-2) 

4. The NEAR side processes the `InitTransfer` event, seeing token identifier `D` and crediting the attacker with `large_amount` of `D` tokens.

5. Attacker bridges back from NEAR. The MPC network signs a `finTransfer` with `payload.tokenAddress = D, amount = large_amount`.

6. `finTransfer` looks up `multiTokens[D]` and finds `{A1, T1}` (the legitimate valuable token), then releases `large_amount` of `A1:T1` tokens to the attacker. [4](#0-3) 

The attacker deposited worthless `A2:T2` tokens and withdrew valuable `A1:T1` tokens.

### Impact Explanation

**Critical — Direct theft of ERC1155 assets locked in the bridge.**

All ERC1155 tokens locked in `OmniBridge` for any token whose deterministic address collides with an attacker-found pair can be drained. The bridge holds custody of these tokens and releases them solely based on `multiTokens[payload.tokenAddress]`, which is keyed by the collision-prone 160-bit deterministic address.

### Likelihood Explanation

The collision requires approximately `2^80` keccak256 evaluations (birthday paradox over a 160-bit space). As established in the referenced Napier/KyberSwap audit discussions, this attack costs on the order of $1–2M at current compute prices using the naive algorithm, and significantly less using the optimized algorithm described in [this paper](https://hackmd.io/Vzhp5YJyTT-LhWm_s0JQpA) (~`2^92` operations with ~160 TB memory). The cost decreases as compute capabilities advance. Given that the bridge is intended to be a long-lived, immutable-style contract holding significant ERC1155 value, the risk is material and grows over time.

### Recommendation

Replace the hash-truncation approach with a registry-based mapping. Instead of deriving a pseudo-address from a hash, maintain an explicit bidirectional mapping between `(tokenAddress, tokenId)` pairs and assigned identifiers, and require `logMetadata1155` to be called before `initTransfer1155` is accepted:

```solidity
// In logMetadata1155: assign a sequential or content-addressed ID
// stored in a mapping(bytes32 => MultiTokenInfo) keyed by keccak256(abi.encode(tokenAddress, tokenId))
// and a reverse mapping from that key to a registered bool

// In initTransfer1155: require the (tokenAddress, tokenId) pair to be registered
bytes32 key = keccak256(abi.encode(tokenAddress, tokenId));
require(registeredTokens[key], "ERC1155 token not registered");
```

Alternatively, use the full 32-byte `keccak256` output as the token identifier on the NEAR side rather than truncating to 160 bits, eliminating the collision surface entirely.

// Attacker generates pairs (A2_i, T2_i) where A2_i are CREATE2-deployed contracts
// and T2_i are arbitrary uint256 values.
// For each pair, compute: D_i = bytes20(keccak256(abi.encodePacked(A2_i, T2_i)))
// Store all D_i values. After ~2^80 iterations, by birthday paradox,
// D_j == D_k for some j != k, or D_j == D (the target legitimate token's address).

//    -> InitTransfer emitted with tokenAddress = D
// 2. NEAR credits attacker with 1000e18 of token D
// 3. Attacker bridges back: finTransfer called with payload.tokenAddress = D, amount = 1000e18
//    -> multiTokens[D] = {A1, T1}  (legitimate valuable token)
//    -> Bridge releases 1000e18 of A1:T1 to attacker
