### Title
`start_block` Missing `CatchingUp` State Guard Allows Premature Proposal/Validation, Enabling Re-Inclusion of Already-Committed L1 Handler Transactions - (File: `crates/apollo_l1_provider/src/l1_provider.rs`)

---

### Summary

`L1Provider::start_block` guards only against `Uninitialized` state. It does not guard against `CatchingUp` state. When the provider is catching up (its analog to "paused/emergency"), a batcher call to `start_block` with the matching `current_height` silently overwrites the `CatchingUp` state to `Propose` or `Validate` before the commit-block backlog is applied. Transactions already committed in backlogged blocks remain `Pending` in the provider and are returned by `get_txs`, causing double-inclusion of L1 handler transactions in a new block proposal and producing a wrong L1 message set, wrong state root, and wrong block commitment.

---

### Finding Description

`L1Provider::start_block` performs only two guards before overwriting `self.state`:

```rust
pub fn start_block(
    &mut self,
    height: BlockNumber,
    state: SessionState,
) -> L1ProviderResult<()> {
    if self.state.is_uninitialized() {          // only guard
        return Err(L1ProviderError::Uninitialized);
    }
    self.check_height_with_error(height)?;      // height == current_height
    self.state = state.into();                  // overwrites CatchingUp → Propose/Validate
    self.tx_manager.start_block();
    Ok(())
}
``` [1](#0-0) 

There is no guard for `CatchingUp`. Compare with `get_txs` and `validate`, which both enforce the correct active state via a `match self.state` arm that returns `UnexpectedProviderState` for any other value — but `start_block` never reaches that pattern. [2](#0-1) 

The `CatchingUp` state is entered when `commit_block` receives a height that is ahead of `current_height`. The provider spawns an async sync task (`l2_sync_task`) that replays historical blocks by calling `commit_block` for each height from `current_height` up to `target_height`. Blocks committed by the batcher *after* catching up started are stored in `commit_block_backlog`. The backlog is applied — and the state transitions to `Pending` — only when `catchupper.is_caught_up(self.current_height)` returns `true`, i.e., when `current_height > target_height`. [3](#0-2) [4](#0-3) 

**Exploit path:**

1. Provider is `Pending` at `current_height = H`.
2. Batcher calls `commit_block(H+N)` (height mismatch). Provider enters `CatchingUp`, sets `target_height = H+N`, starts sync task. Batcher also enqueues blocks `H+N+1`, `H+N+2` into the backlog.
3. Sync task replays blocks `H … H+N` via `commit_block`. After processing block `H+N`, `current_height = H+N+1`. Provider is still `CatchingUp` because `is_caught_up` requires `current_height > target_height = H+N`, which is now true — but the backlog has not been applied yet (it is applied only inside `accept_commit_while_catching_up`, which is only triggered by the *next* `commit_block` call).
4. Batcher calls `start_block(H+N+1, Propose)`. Height check passes (`current_height == H+N+1`). `self.state` is overwritten from `CatchingUp` to `Propose`. `tx_manager.start_block()` rolls back staging.
5. Batcher calls `get_txs`. The provider is now in `Propose` state. Transactions committed in backlogged blocks `H+N+1`, `H+N+2` are still `Pending` in `tx_manager` (backlog never applied). They pass the cooldown cutoff and are returned.
6. Batcher includes these already-committed L1 handler transactions in the new block proposal. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Double-including an L1 handler transaction means the sequencer proposes a block containing an L1→L2 message that was already consumed in a prior committed block. The blockifier executes it again, producing a wrong execution result, wrong storage writes, wrong events, and a wrong state root. The resulting block commitment and state diff are incorrect. This falls under:

> **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

and

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

`CatchingUp` is a startup/recovery state that occurs every time the node restarts or falls behind. The window between the sync task completing its last `commit_block` (setting `current_height = target_height + 1`) and the backlog being applied (which requires one more `commit_block` call to trigger `accept_commit_while_catching_up`) is a real, observable race. The batcher independently drives `start_block` calls and has no mechanism to detect that the provider is still in `CatchingUp`. Likelihood is **Low** (requires the batcher to call `start_block` in this narrow window with a non-empty backlog), matching the external report's low-likelihood classification.

---

### Recommendation

Add an explicit `CatchingUp` guard at the top of `start_block`, mirroring the pattern used in `get_txs` and `validate`:

```rust
pub fn start_block(
    &mut self,
    height: BlockNumber,
    state: SessionState,
) -> L1ProviderResult<()> {
    if self.state.is_uninitialized() {
        return Err(L1ProviderError::Uninitialized);
    }
    if self.state.is_catching_up() {
        return Err(L1ProviderError::UnexpectedProviderState {
            expected: ProviderState::Pending,
            found: self.state,
        });
    }
    self.check_height_with_error(height)?;
    self.state = state.into();
    self.tx_manager.start_block();
    Ok(())
}

### Proof of Concept

```rust
#[tokio::test]
async fn start_block_during_catching_up_bypasses_backlog() {
    use apollo_l1_provider_types::SessionState::Propose;

    // Provider starts at height 5, batcher is at height 7 → triggers CatchingUp.
    const STARTUP_HEIGHT: BlockNumber = BlockNumber(5);
    const TARGET_HEIGHT: BlockNumber = BlockNumber(6);

    // tx_1 was committed in the backlogged block (height 7).
    let tx_1 = l1_handler(1);
    let catchupper = make_catchupper!(backlog: [7 => [tx_hash!(1)]]);

    let mut provider = L1ProviderContentBuilder::new()
        .with_catchupper(catchupper)
        .with_txs([tx_1.clone()])          // tx_1 is still Pending in tx_manager
        .with_state(ProviderState::Uninitialized)
        .build_into_l1_provider();

    provider.initialize(STARTUP_HEIGHT, vec![]).await.unwrap();
    // Trigger CatchingUp; sync task will replay heights 5 and 6.
    call_commit_block_to_start_catching_up_and_expect_error(&mut provider, TARGET_HEIGHT);

    // Simulate sync task completing: commit heights 5 and 6.
    commit_block_no_rejected(&mut provider, &[], STARTUP_HEIGHT);          // height 5 → current=6
    commit_block_no_rejected(&mut provider, &[], STARTUP_HEIGHT.unchecked_next()); // height 6 → current=7

    // Provider is still CatchingUp; backlog (height 7, tx_1) not yet applied.
    assert!(provider.state.is_catching_up());
    assert_eq!(provider.current_height, BlockNumber(7));

    // BUG: start_block succeeds during CatchingUp because height matches.
    provider.start_block(BlockNumber(7), Propose).unwrap();
    assert_eq!(provider.state, ProviderState::Propose); // CatchingUp overwritten!

    // get_txs returns tx_1, which was already committed in the backlogged block 7.
    let txs = provider.get_txs(10, BlockNumber(7)).unwrap();
    assert!(txs.iter().any(|tx| tx.tx_hash == tx_1.tx_hash),
        "Already-committed tx_1 re-proposed due to missing CatchingUp guard");
}
