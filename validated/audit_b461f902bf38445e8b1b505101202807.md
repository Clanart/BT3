Looking at the code carefully, I need to trace every guard in `set_state_header` to determine whether the chunk's internal `shard_id` is ever compared against the `shard_id` parameter.

### Title
Missing shard_id binding in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` never checks that the `chunk` embedded in the `ShardStateSyncResponseHeader` has a `shard_id` matching the `shard_id` parameter. The Merkle proof check only verifies that the chunk hash appears *somewhere* in the block's `chunk_headers_root` tree (which covers all shards), not that it occupies the leaf position for the requested shard. A malicious state-sync peer can therefore supply shard B's chunk as the body of a header nominally for shard A, pass every guard, and cause `set_state_part` to validate and install state parts against shard B's `prev_state_root` into shard A's trie.

### Finding Description

**Entry point:** `set_state_header(shard_id=A, sync_hash, header_with_shard_B_chunk)` in `chain/chain/src/state_sync/adapter.rs`.

The five guards executed before the header is persisted are:

| Guard | What it checks | Checks `chunk.shard_id() == shard_id`? |
|---|---|---|
| `validate_chunk_proofs` | Internal hash/tx/receipt-root consistency of the chunk | No |
| `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk_hash, height_included))` | Chunk hash is *somewhere* in the block's all-shard Merkle tree | No |
| `verify_path(prev_block.chunk_headers_root, prev_chunk_proof, ...)` | Prev chunk hash is *somewhere* in prev block's Merkle tree | No |
| Receipt-proof loop (steps 4a–4g) | Receipts are valid for `shard_id` (parameter A) | N/A — uses parameter, not chunk field |
| `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` | `hash(node.data) == chunk.prev_state_root` | No |

The grep search over `adapter.rs` returns zero matches for `chunk.shard_id()`, confirming the field is never read during validation. [1](#0-0) 

The `chunk_headers_root` in a block header is a Merkle tree over **all** shards' `ChunkHashHeight` tuples: [2](#0-1) 

Because shard B's chunk is a legitimate leaf in that tree, the attacker's `chunk_proof` for shard B's position verifies against the same root that shard A's proof would use. The check is position-agnostic. [3](#0-2) 

`validate_chunk_proofs` only checks the chunk's internal hash, tx root, and outgoing-receipts root — it never reads `shard_id`: [4](#0-3) 

After the header is stored, `set_state_part` blindly reads `chunk.take_header().take_inner().prev_state_root()` from whatever was stored under `StateHeaderKey(shard_A_id, sync_hash)` and uses it as the validation root: [5](#0-4) 

`validate_state_part_impl` only checks the trie Merkle proof against the supplied `state_root`; `shard_id` is used only for metrics: [6](#0-5) 

### Impact Explanation

After a successful attack:

1. `StateHeaderKey(shard_A_id, sync_hash)` stores a header whose embedded chunk belongs to shard B.
2. `set_state_part` extracts shard B's `prev_state_root` and accepts any state parts that are valid for shard B's trie.
3. `apply_state_part` writes those trie nodes into shard A's `ShardUId` storage (the `shard_id` parameter routes the write to the correct column): [7](#0-6) 

The post-sync state root for shard A is shard B's state root. The node will subsequently fail to apply blocks for shard A because its trie does not match the canonical chain's expected state root for shard A.

### Likelihood Explanation

The attacker is any peer that the syncing node contacts for state data. No validator, chunk-producer, or operator privileges are required. All inputs the attacker needs are public chain data:

- Shard B's full chunk (available from any archive node)
- The Merkle proof for shard B's position in the block (computable from the block body)
- Shard A's receipt proofs (public chain data)
- Shard B's `StateRootNode` (obtainable from any node serving shard B)
- State parts valid for shard B's state root (obtainable from any node serving shard B)

The only practical constraint is that shard A and shard B must have the same `prev_chunk.height_included` so that the receipt-proof range check (step 4g) passes. In a healthy network where every shard produces a chunk at every block, this is always true. [8](#0-7) 

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: enforce the commitment invariant
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check to `prev_chunk_header.shard_id()`. This closes the gap without touching any other validation logic. [9](#0-8) 

### Proof of Concept

```
1. Node N is syncing shard A (shard_id=0) at sync_hash H.
2. Attacker peer P:
   a. Fetches shard B's (shard_id=1) full ShardChunk from the block at H-1.
   b. Computes the Merkle proof for shard B's leaf in block(H-1).chunk_headers_root.
   c. Fetches shard A's incoming_receipts_proofs and root_proofs (public chain data).
   d. Fetches shard B's StateRootNode (hash(node.data) == shard_B.prev_state_root).
   e. Constructs ShardStateSyncResponseHeader{chunk=shard_B_chunk, chunk_proof=shard_B_proof,
      prev_chunk_header=shard_B_prev, prev_chunk_proof=shard_B_prev_proof,
      incoming_receipts_proofs=shard_A_receipts, root_proofs=shard_A_root_proofs,
      state_root_node=shard_B_node}.
3. P sends this header to N for shard_id=0.
4. set_state_header(0, H, header) passes all five guards and stores the header under
   StateHeaderKey(0, H).
5. P sends state parts valid for shard B's prev_state_root.
6. set_state_part(0, H, part_id, part) reads shard B's prev_state_root from the stored
   header, validate_state_part passes, parts are written to shard A's trie column.
7. After set_state_finalize, shard A's trie contains shard B's state.
   The node diverges from the canonical chain on the next block application for shard A.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
