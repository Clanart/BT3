### Title
Silent Return Instead of Revert on Storage Balance Failure in `utxo_fin_transfer_to_near_callback` Permanently Freezes User Funds — (File: near/omni-bridge/src/lib.rs)

---

### Summary

In `utxo_fin_transfer_to_near_callback`, when the recipient's storage balance check fails, the function silently logs an error, removes the fin-transfer record, and returns the amount — instead of reverting. This mirrors the external report's pattern exactly: a critical guard condition is failed, but the function exits silently rather than reverting, leaving the protocol in an incorrect state. Because the UTXO funds are already consumed on the source chain and the fin-transfer record is removed (preventing replay), the user's bridged assets are permanently frozen.

---

### Finding Description

`utxo_fin_transfer_to_near_callback` (lines 975–1012) is the NEAR-side callback that finalises a UTXO-chain (BTC/ZEC) → NEAR transfer. It first checks whether the recipient has sufficient storage balance registered for the token:

```rust
if !Self::check_storage_balance_result(0) {
    env::log_str(BridgeError::StorageRecipientOmitted.to_string().as_str());
    self.remove_fin_utxo_transfer(
        &utxo_fin_transfer_msg.get_transfer_id(origin_chain),
        storage_owner,
    );
    return PromiseOrValue::Value(amount);
}
``` [1](#0-0) 

When the check fails, the function:
1. Emits a log string (not a revert/panic).
2. Calls `remove_fin_utxo_transfer`, which deletes the transfer record from contract state.
3. Returns `PromiseOrValue::Value(amount)` — a silent early exit.

This is structurally identical to the external report's pattern (`update_timestamp; return;`): a threshold/guard check fails, state is mutated, and the function exits without reverting.

**The critical contrast** is with `fast_fin_transfer_to_near_callback` (lines 845–848), which handles the same storage-balance guard for fast transfers using a hard revert:

```rust
require!(
    Self::check_storage_balance_result(0),
    BridgeError::StorageRecipientOmitted.as_ref()
);
``` [2](#0-1) 

The `fast_fin_transfer_to_near_callback` path correctly reverts, allowing the relayer to retry with a proper storage deposit. The UTXO path does not.

---

### Impact Explanation

- The UTXO (BTC/ZEC) is already consumed on the source chain before this callback fires; it cannot be reclaimed.
- `remove_fin_utxo_transfer` deletes the transfer record from contract state. Once removed, the transfer ID cannot be re-submitted (the record that would allow replay no longer exists, and re-submitting the same proof would either be rejected as a duplicate or treated as a new transfer with a different ID).
- The user receives no tokens on NEAR and cannot recover their source-chain funds.
- **Result: permanent, irrecoverable freeze of bridged user funds** — matching the allowed Critical/High impact class "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows." [1](#0-0) 

---

### Likelihood Explanation

This is reachable by any unprivileged bridge user or relayer:

- A relayer submits a UTXO fin-transfer proof without first registering storage for the recipient account on the token contract.
- `check_storage_balance_result(0)` returns `false`.
- The silent-return path fires, removing the record and freezing funds.

No privileged access is required. The relayer role is open to any participant, and storage registration is a separate, easily-missed step.

---

### Recommendation

Replace the silent-return block with a hard revert, consistent with `fast_fin_transfer_to_near_callback`:

```rust
require!(
    Self::check_storage_balance_result(0),
    BridgeError::StorageRecipientOmitted.as_ref()
);
```

A revert preserves the fin-transfer record and allows the relayer to retry after registering storage for the recipient, without any fund loss.

---

### Proof of Concept

1. User initiates a UTXO (BTC/ZEC) → NEAR bridge transfer; the UTXO is consumed on the source chain.
2. A relayer submits the fin-transfer proof via `fin_transfer` / the UTXO finalization path.
3. The bridge calls `check_or_pay_ft_storage` for the recipient, but the relayer omits or underfunds the storage deposit action.
4. `utxo_fin_transfer_to_near_callback` fires; `check_storage_balance_result(0)` returns `false`.
5. The function logs `StorageRecipientOmitted`, calls `remove_fin_utxo_transfer` (deleting the record), and returns `PromiseOrValue::Value(amount)`.
6. No tokens are minted/transferred to the recipient on NEAR.
7. The transfer record is gone; the proof cannot be re-submitted to recover funds.
8. User's UTXO funds are permanently frozen. [3](#0-2)

    #[private]
    pub fn utxo_fin_transfer_to_near_callback(
        &mut self,
        token_id: AccountId,
        recipient: AccountId,
        amount: U128,
        utxo_fin_transfer_msg: UtxoFinTransferMsg,
        origin_chain: ChainKind,
        storage_owner: &AccountId,
    ) -> PromiseOrValue<U128> {
        if !Self::check_storage_balance_result(0) {
            env::log_str(BridgeError::StorageRecipientOmitted.to_string().as_str());
            self.remove_fin_utxo_transfer(
                &utxo_fin_transfer_msg.get_transfer_id(origin_chain),
                storage_owner,
            );
            return PromiseOrValue::Value(amount);
        }

        self.send_tokens(
            token_id.clone(),
            recipient,
            amount,
            &utxo_fin_transfer_msg.msg,
        )
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(RESOLVE_UTXO_FIN_TRANSFER_GAS)
                .resolve_utxo_fin_transfer(
                    token_id,
                    amount,
                    utxo_fin_transfer_msg,
                    origin_chain,
                    storage_owner,
                ),
        )
        .into()
    }
