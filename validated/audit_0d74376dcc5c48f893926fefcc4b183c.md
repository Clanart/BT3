### Title
Constructor Revert Not Enforced in `execute_deploy` Syscall — OS Assumes Non-Reverting Constructors While Blockifier Supports Failures — (File: `execution/syscall_impls.cairo`, `execution/deploy_contract.cairo`)

---

### Summary

The StarkNet OS `execute_deploy` syscall unconditionally asserts `failure_flag=0` and `is_reverted=0` for constructor execution. The OS has no mechanism to handle a reverting constructor gracefully. Meanwhile, the blockifier (sequencer execution engine) is designed to support constructor failures by returning a failure response to the caller. This mismatch means a transaction that the blockifier considers valid — one where a `deploy` syscall's constructor reverts and the caller handles the failure — cannot be proven by the OS, causing proof generation to fail and halting the network.

---

### Finding Description

**Root cause in `deploy_contract.cairo`:** [1](#0-0) 

```cairo
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);
...
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
```

The `assert is_reverted = 0` is a Cairo constraint. If the constructor reverts (`is_reverted = 1`), this constraint is unsatisfiable — the proof is invalid. The comment says "deprecated deploy syscalls", but `deploy_contract` is called by **both** the deprecated and the new `execute_deploy` syscall.

**Root cause in `syscall_impls.cairo` (new `deploy` syscall):** [2](#0-1) 

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The `TODO` comment explicitly acknowledges that constructor failures are not yet supported. The response header always writes `failure_flag=0`, meaning the OS always reports the deploy as successful regardless of the constructor outcome.

**Contrast with the deprecated deploy syscall in `deprecated_execute_syscalls.cairo`:** [3](#0-2) 

The deprecated deploy syscall also calls `deploy_contract` and inherits the same `assert is_reverted = 0` constraint.

**The `non_reverting_select_execute_entry_point_func` pattern confirms the design intent:** [4](#0-3) 

The OS uses `non_reverting_select_execute_entry_point_func` (which asserts `is_reverted = 0`) only for contexts where reverts are explicitly disallowed by design (validate, charge_fee). The `deploy` syscall is **not** supposed to be in this category — it is a user-initiated syscall that should propagate failures to the caller — yet it enforces the same hard constraint.

---

### Impact Explanation

The blockifier (sequencer) is designed to handle constructor reverts gracefully: when a constructor reverts, the `deploy` syscall returns a failure response to the calling contract, which may handle it and continue executing. The transaction can succeed from the blockifier's perspective.

When the sequencer includes such a transaction in a block and the OS attempts to generate a proof:

1. The OS calls `deploy_contract` for the constructor.
2. The constructor reverts → `is_reverted = 1`.
3. `assert is_reverted = 0` is unsatisfiable.
4. No valid proof can be generated for the block.
5. The block cannot be finalized on L1.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

The attack requires only two unprivileged operations:

1. **Declare** a contract class whose constructor always reverts (a standard `declare` transaction, no special privileges).
2. **Invoke** a transaction that calls the `deploy` syscall with that class hash, and whose `__execute__` function handles the deploy failure gracefully (e.g., catches the failure flag and continues).

The blockifier would execute this transaction successfully (deploy fails, caller handles it, transaction succeeds). The sequencer includes it in a block. The OS cannot prove the block.

This is directly reachable by any unprivileged transaction sender. No trusted role, leaked key, or operator cooperation is required.

---

### Recommendation

1. **Remove `assert is_reverted = 0` from `deploy_contract`** and instead propagate the `is_reverted` flag back to `execute_deploy`.
2. **In `execute_deploy` (`syscall_impls.cairo`)**, write `failure_flag=is_reverted` in the response header when the constructor reverts, consistent with how `contract_call_helper` handles reverts for `call_contract`.
3. **Implement the revert log handling** for the deploy syscall analogous to `execute_call_contract` in `syscall_impls.cairo` (lines 217–231), which properly brackets caller/callee entries in the revert log.

---

### Proof of Concept

**Step 1**: Declare a class `RevertingConstructor` whose constructor body contains `assert 1 = 0` (always reverts).

**Step 2**: Deploy an attacker contract `AttackerCaller` whose `__execute__` function:
```
let (contract_address) = deploy(class_hash=REVERTING_CONSTRUCTOR_HASH, ...);
// blockifier returns failure_flag=1 here; caller continues
return ();  // transaction succeeds
```

**Step 3**: Submit an `invoke` transaction calling `AttackerCaller.__execute__`.

**Blockifier result**: The `deploy` syscall fails (constructor reverts), `failure_flag=1` is returned to the caller, the caller returns successfully. The sequencer marks the transaction as **succeeded** and includes it in a block.

**OS result**: When proving the block, `deploy_contract` reaches `assert is_reverted = 0` with `is_reverted = 1`. The constraint is unsatisfiable. **No valid proof can be generated. The block is stuck. Network halts.**
