Audit Report

## Title
Residual ETH in Router Silently Subsidizes Subsequent WETH Swaps, Stealing Prior User's Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses `address(this).balance` — the router's total native ETH balance — when deciding how much ETH to wrap for a WETH payment. Because `multicall` and `exactInputSingle` are `payable`, any user who sends excess ETH without appending `refundETH()` leaves ETH stranded in the router. The next WETH swapper's `pay` call silently consumes that stranded ETH, permanently transferring it to the pool on behalf of the new swapper, while the new swapper pays proportionally less WETH from their own allowance.

## Finding Description

`PeripheryPayments.pay` contains three branches for WETH payments: [1](#0-0) 

The middle branch (`nativeBalance > 0 && nativeBalance < value`) wraps **all** native ETH currently held by the router and forwards it to the pool, then pulls only `value - nativeBalance` from the payer's WETH allowance. It does not verify that the native ETH belongs to the current caller or was sent in the current transaction.

ETH can accumulate in the router via `msg.value` on the `payable` entry points: [2](#0-1) [3](#0-2) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable entry points: [4](#0-3) 

Exploit path:
1. User A calls `multicall{value: 1 ether}([exactInputSingle(tokenIn=WETH, amountIn=0.3 ether)])` without `refundETH()`. The swap consumes 0.3 ETH; 0.7 ETH remains stranded in the router.
2. User B calls `exactInputSingle(tokenIn=WETH, amountIn=1 ether)`. This triggers `pay(WETH, userB, pool, 1 ether)`. `nativeBalance = 0.7 ETH`, which is `> 0` and `< 1 ETH`, so the middle branch fires: wraps User A's 0.7 ETH, sends it to the pool, then pulls only 0.3 WETH from User B's allowance. The pool receives the correct 1 ETH, but User A's 0.7 ETH is permanently consumed without their consent.

## Impact Explanation

User A suffers a direct, irreversible loss of principal: their stranded ETH is wrapped and forwarded to the pool on behalf of User B's swap. User B receives a discount funded entirely by User A. The pool is not insolvent (it receives the correct amount), but User A has no recourse. This is a direct loss of user principal through a public, unprivileged swap path, meeting the High impact threshold.

## Likelihood Explanation

The pattern of `multicall{value}(swap)` without `refundETH()` is a common integration mistake explicitly anticipated by the test suite: [5](#0-4) 

Any user who sends excess ETH and omits `refundETH()` — or whose transaction reverts after ETH is already in the router — creates the vulnerable state. An attacker can monitor the mempool for such transactions and immediately follow with a WETH swap to claim the stranded ETH.

## Recommendation

In the partial-ETH branch, only consume native ETH that was sent by the current caller in the current transaction (i.e., `msg.value`). One approach: record `msg.value` in transient storage at each payable entry point and cap the native ETH consumed in `pay` to that recorded amount, ignoring any residual balance from prior transactions. Alternatively, revert at entry if `address(this).balance > msg.value`, ensuring no cross-user ETH leakage is possible.

## Proof of Concept

```
1. User A: router.multicall{value: 1 ether}([exactInputSingle(tokenIn=WETH, amountIn=0.3 ether)])
   → pay() sees nativeBalance=1 ETH >= value=0.3 ETH → wraps 0.3 ETH, sends to pool
   → router.balance = 0.7 ETH (stranded, no refundETH call)

2. User B: router.exactInputSingle(tokenIn=WETH, amountIn=1 ether)
   → pay(WETH, userB, pool, 1 ether)
   → nativeBalance=0.7 ETH > 0, < 1 ETH → middle branch
   → wraps 0.7 ETH (User A's), safeTransfer(pool, 0.7 WETH)
   → safeTransferFrom(userB, pool, 0.3 WETH)
   → pool receives 1 ETH ✓, but userB paid only 0.3 WETH
   → User A loses 0.7 ETH with no recourse
```
