### Title
Non-EVM-Denom Bank Balances of Self-Destructed Contracts Are Untracked and Stealable via CREATE2 Redeploy — (`File: x/evm/statedb/statedb.go`)

### Summary

When an EVM contract self-destructs, Ethermint's `StateDB.Commit()` only burns the EVM-native denomination balance. Non-EVM-denom tokens (IBC assets, CosmWasm bridge tokens, etc.) held by the destroyed address are explicitly left as orphaned bank balances. Because `DeleteAccount` removes only the auth-module account record and never touches the bank module, a new contract redeployed at the same address via CREATE2 inherits those orphaned balances and can drain them — constituting unauthorized theft of Cosmos bank funds through EVM state-transition logic.

---

### Finding Description

In `StateDB.Commit()`, the self-destruct branch reads:

`DeleteAccount` removes the auth-module record but never touches the bank module's `(address, denom) → amount` store. The Cosmos SDK bank module stores balances keyed purely by address; when a new account is later created at the same address, it inherits whatever bank balances remain.

The `SelfDestruct` function marks the account and burns the EVM-denom balance present at destruction time: [3](#0-2) 

But any non-EVM-denom coins (IBC, etc.) are never touched.

---

### Impact Explanation

An attacker who controls a factory contract can:

1. Deploy a contract via CREATE2 (known salt, known factory address).
2. Attract users to deposit IBC or other non-EVM-denom tokens into the contract (e.g., via a bank-send precompile or IBC transfer to the contract address).
3. Call `SELFDESTRUCT` on the contract (pre-EIP-6780 rules) or arrange creation+destruction in one transaction (post-EIP-6780 / `SelfDestruct6780`).
4. The EVM denom is burned; the auth account is deleted; IBC tokens remain in the bank module at that address.
5. Redeploy a new contract at the identical address using the same CREATE2 salt.
6. The new contract inherits the orphaned IBC bank balance and can transfer it to the attacker via a native-action precompile.

This constitutes **unauthorized theft of Cosmos bank funds through Ethermint EVM state-transition logic**, matching the Critical allowed impact.

---

### Likelihood Explanation

- Any unprivileged user can deploy a contract and control its self-destruct logic.
- CREATE2 redeployment at the same address is a standard EVM pattern.
- IBC tokens are routinely sent to contract addresses in Cosmos EVM chains.
- The comment in the code acknowledges the limitation but incorrectly characterizes the tokens as merely "orphaned" (stuck), not recognizing that redeployment makes them accessible.
- No governance, validator, or privileged role is required.

---

### Recommendation

In `StateDB.Commit()`, before calling `DeleteAccount`, iterate over **all** bank-module denominations held by the self-destructed address and burn or escrow them, not only the EVM denom. Concretely:

```go
// Drain ALL denominations, not just evmDenom
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        // or send to a community pool / governance escrow
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-evm balance")
    }
}
```

Alternatively, block CREATE2 redeployment to addresses that previously held a self-destructed account with residual non-EVM balances, or require explicit governance recovery of such orphaned funds.

---

### Proof of Concept

```
Block N:
  Attacker deploys Factory via CREATE2 salt=0xAA (factory address = F)
  Factory.deploy(salt=0xBB) → contract C at deterministic address A

Block N+1..M:
  Users call IBC transfer to address A (or use bank-send precompile)
  → bank module: A holds 1000 uatom

Block M+1:
  Attacker calls C.selfDestruct()
  → StateDB.SelfDestruct(A): burns EVM denom, marks selfDestructed=true
  → StateDB.Commit(): SubBalance burns EVM denom only; DeleteAccount removes auth record
  → bank module: A still holds 1000 uatom (UNTRACKED, not burned)

Block M+2:
  Attacker calls Factory.deploy(salt=0xBB) again
  → new contract C' deployed at same address A
  → auth module: new account created at A
  → bank module: A still holds 1000 uatom → now owned by C'

Block M+3:
  C'.drainIBC(attacker) calls bank-send precompile
  → 1000 uatom transferred to attacker
```

func (s *StateDB) SelfDestruct(addr common.Address) uint256.Int {
	stateObject := s.getStateObject(addr)
	var prevBalance uint256.Int
	if stateObject == nil {
		return prevBalance
	}
	prevBalance = *(stateObject.Balance())
	s.journal.append(selfDestructChange{
		account:     &addr,
		prev:        stateObject.selfDestructed,
		prevbalance: new(uint256.Int).Set(&prevBalance),
	})
	stateObject.markSelfDestructed()
	// clear balance
	balance := s.GetBalance(addr)
	if balance.Sign() > 0 {
		s.SubBalance(addr, balance, tracing.BalanceDecreaseSelfdestructBurn)
	}
	return prevBalance
}

func (k *Keeper) SubBalance(ctx sdk.Context, addr sdk.AccAddress, coin sdk.Coin) (uint256.Int, error) {
	coins := sdk.NewCoins(coin)
	prevBalance := k.GetBalance(ctx, addr, coin.Denom)
	if err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	if err := k.bankKeeper.BurnCoins(ctx, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	return prevBalance, nil
