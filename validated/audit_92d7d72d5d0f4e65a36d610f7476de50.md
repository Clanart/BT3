### Title
Silent `NonzeroGasPrice::MIN` (= 1) Substitution for Zero-Stored Gas Prices Causes Wrong RPC Fee Estimation and Simulation Results - (`File: crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

In `create_block_context_for_execution`, when building a `BlockContext` for RPC execution against any finalized block whose stored gas prices are zero (pre-0.13.1 blocks for `l1_data_gas_price`; pre-0.14.x blocks for `l2_gas_price`), all six gas-price fields are silently replaced with `NonzeroGasPrice::MIN` = `GasPrice(1)`. This is the direct sequencer analog of the external report's `GetExchangeRateBase` returning `sdk.OneDec()` for USD-named assets: a condition-based substitution of a hardcoded sentinel value instead of the real stored value. The resulting `BlockContext` is fed into the blockifier for `starknet_estimateFee`, `starknet_simulateTransactions`, and `starknet_call`, causing those RPC endpoints to return authoritative-looking but incorrect fee and execution results.

---

### Finding Description

`crates/apollo_rpc_execution/src/lib.rs` lines 379–396:

```rust
// TODO(yair): What to do about blocks pre 0.13.1 where the data gas price were 0?
gas_prices: GasPrices {
    eth_gas_prices: GasPriceVector {
        l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_wei)
            .unwrap_or(NonzeroGasPrice::MIN),          // MIN = GasPrice(1)
        l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_wei)
            .unwrap_or(NonzeroGasPrice::MIN),
        l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_wei)
            .unwrap_or(NonzeroGasPrice::MIN),
    },
    strk_gas_prices: GasPriceVector {
        l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_fri)
            .unwrap_or(NonzeroGasPrice::MIN),
        l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_fri)
            .unwrap_or(NonzeroGasPrice::MIN),
        l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_fri)
            .unwrap_or(NonzeroGasPrice::MIN),
    },
},
```

`NonzeroGasPrice::MIN` is defined as `NonzeroGasPrice(GasPrice(1))` in `crates/starknet_api/src/block.rs` line 522.

The substitution fires whenever `NonzeroGasPrice::new(price)` returns `Err`, i.e., whenever the stored price is `GasPrice(0)`. This happens for:

- **`l1_data_gas_price`** on all blocks before Starknet v0.13.1 (data gas was not tracked; the field was stored as 0).
- **`l2_gas_price`** on all blocks before Starknet v0.14.x (L2 gas was not tracked; the field was stored as 0).

The `BlockContext` built with these fabricated prices is then passed directly into the blockifier for execution. The blockifier uses `block_info.gas_prices` to:

1. Compute the fee charged per unit of each gas type (`get_fee_by_gas_vector`).
2. Determine whether the sender's balance covers the fee (`get_balance_and_if_covers_fee`).
3. Populate `TransactionReceipt.fee` and `TransactionExecutionInfo` returned to the caller.

Because the substituted price is `1` (not `0`), the fee calculation is non-zero and plausible-looking, but it is not the price that was actually in effect for that block. The contrast with the consensus/batcher path is instructive: `convert_to_sn_api_block_info` in `crates/apollo_consensus_orchestrator/src/utils.rs` lines 299–303 calls `NonzeroGasPrice::new(...)?` and propagates the error rather than silently substituting a sentinel.

---

### Impact Explanation

Any caller of `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_call` that targets a finalized block with zero stored gas prices receives a fee estimate or execution result computed against gas price = 1 wei/fri instead of the actual historical price. The returned value is authoritative-looking (it is a well-formed `FeeEstimation` or `TransactionExecutionInfo`), but it is wrong. Concretely:

- **`starknet_estimateFee`** at a pre-0.13.1 block returns a non-zero data-gas fee component that did not exist at that block height.
- **`starknet_simulateTransactions`** at such a block returns a simulation whose `actual_fee` and resource accounting are computed against the wrong price vector.
- **`starknet_call`** at such a block executes with a `BlockContext` whose gas prices differ from the real historical context, potentially changing execution outcomes for contracts that inspect gas prices via syscalls.

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Pre-0.13.1 and pre-0.14.x blocks exist on both Starknet mainnet and Sepolia testnet. Any unprivileged user can trigger the substitution by issuing a standard JSON-RPC call with a `block_id` pointing to any such block. No special permissions, keys, or network position are required. The open TODO comment (`// TODO(yair): What to do about blocks pre 0.13.1 where the data gas price were 0?`) confirms the substitution is not a deliberate design choice but an unresolved placeholder.

---

### Recommendation

Replace the silent `unwrap_or(NonzeroGasPrice::MIN)` fallback with an explicit error path, mirroring the consensus path:

```rust
l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_wei)
    .map_err(|_| ExecutionError::ZeroGasPriceInStoredBlock {
        block: block_context_number,
        field: "l1_data_gas_price_wei",
    })?,
```

Alternatively, if backward-compatible simulation of pre-0.13.1 blocks is required, reconstruct the historically correct price from the block's `max_fee` and actual gas consumed rather than substituting an arbitrary sentinel.

---

### Proof of Concept

1. Identify any mainnet block with `starknet_version < "0.13.1"` (e.g., block 500 000). Its stored `l1_data_gas_price` and `l2_gas_price` are `GasPrice(0)`.
2. Issue `starknet_estimateFee` with any v3 transaction body and `"block_id": {"block_number": 500000}`.
3. The node calls `create_block_context_for_execution` with `maybe_pending_data = None`.
4. `NonzeroGasPrice::new(GasPrice(0))` returns `Err`; `.unwrap_or(NonzeroGasPrice::MIN)` substitutes `GasPrice(1)` for all six fields.
5. The blockifier computes `fee = gas_vector.cost(price_vector_of_1, tip)` — a non-zero, plausible-looking but historically incorrect value.
6. The RPC response contains a well-formed `FeeEstimation` whose `overall_fee` and `gas_price` fields are wrong, with no indication of the substitution. [1](#0-0) [2](#0-1) [3](#0-2)
