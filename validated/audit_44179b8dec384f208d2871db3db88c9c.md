### Title
Missing Token Registration Validation in `initTransfer` Enables Irrecoverable Fund Lock — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

Both the EVM and StarkNet `initTransfer` entry points accept any token address without verifying that the token is registered in the bridge (i.e., has a corresponding NEAR-side mapping and decimals entry). A user who calls `initTransfer` with an unregistered token will have their tokens locked or burned on the source chain, while the NEAR `fin_transfer_callback` will unconditionally panic because the token has no registered decimals. There is no refund or recovery path in either contract. The funds are permanently frozen.

---

### Finding Description

**EVM — `OmniBridge.sol::initTransfer` (lines 373–437)**

The function dispatches on three token categories — custom minter, bridge token, or plain ERC20 — but never checks whether `ethToNearToken[tokenAddress]` is non-empty (i.e., the token is registered):

```solidity
// lines 394–412 — no registration guard before any branch
if (customMinters[tokenAddress] != address(0)) {
    ...burn via custom minter...
} else if (isBridgeToken[tokenAddress]) {
    BridgeToken(tokenAddress).burn(msg.sender, amount);   // tokens destroyed
} else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount); // tokens locked
}
```

For a plain ERC20 that has never been registered via `deployToken` or `addCustomToken`, the tokens are transferred into the bridge contract and an `InitTransfer` event is emitted. The NEAR bridge then attempts to finalize the transfer. [1](#0-0) 

**StarkNet — `omni_bridge.cairo::init_transfer` (lines 281–331)**

The same pattern exists on StarkNet. The function checks only `is_bridge_token` (which tests `starknet_to_near_token.read(token_address).len() > 0`) to decide burn vs. lock, but never asserts that the token is registered before proceeding:

```cairo
if self.is_bridge_token(token_address) {
    IBridgeTokenDispatcher { contract_address: token_address }.burn(caller, amount.into());
} else {
    let success = IERC20Dispatcher { contract_address: token_address }
        .transfer_from(caller, get_contract_address(), amount.into());
    assert(success, 'ERR_TRANSFER_FROM_FAILED');
}
``` [2](#0-1) 

**NEAR — `fin_transfer_callback` panics on unregistered token (lines 715–718)**

When the relayer submits the proof on NEAR, `fin_transfer_callback` immediately panics if the token has no registered decimals:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [3](#0-2) 

The NEAR transaction reverts. The source-chain tokens remain locked (or burned) with no refund mechanism. Neither `OmniBridge.sol` nor `omni_bridge.cairo` exposes a `rescue`, `emergencyWithdraw`, or equivalent recovery function. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds in bridge flows.**

- For plain ERC20 / ERC20 (StarkNet) tokens: tokens are transferred into the bridge contract and cannot be recovered because there is no admin rescue path and the NEAR finalization will always fail.
- For bridge tokens (`isBridgeToken == true` on EVM, `is_bridge_token` on StarkNet): tokens are **burned** on the source chain. Even if an admin later registers the token on NEAR, the burned supply is gone — the bridge's collateralization is broken.
- The `BridgeError::TokenDecimalsNotFound` error is a hard panic, not a graceful refund trigger. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged user can trigger this:

1. **Accidental**: A user holds a legitimate ERC20 (e.g., a newly listed token) that has not yet been registered via `deployToken`. They call `initTransfer` assuming any ERC20 is bridgeable — a reasonable assumption given the function accepts arbitrary `tokenAddress`.
2. **Intentional self-harm / griefing**: An attacker can permanently destroy their own bridge tokens by calling `initTransfer` with a bridge token whose NEAR-side registration has been removed or was never completed, breaking the bridge's token supply invariant.

The EVM `CLAUDE.md` security invariant states: *"InitTransfer must only be emitted in a code path where tokens have already been burned/locked"* — but it does not require the token to be registered, leaving this gap open. [6](#0-5) 

---

### Recommendation

Add a registration guard at the top of `initTransfer` on both EVM and StarkNet before any token movement:

**EVM (`OmniBridge.sol`):**
```solidity
require(
    tokenAddress == address(0) ||
    bytes(ethToNearToken[tokenAddress]).length > 0,
    "ERR_TOKEN_NOT_REGISTERED"
);
```

**StarkNet (`omni_bridge.cairo`):**
```cairo
assert(
    self.starknet_to_near_token.read(token_address).len() > 0,
    'ERR_TOKEN_NOT_REGISTERED'
);
```

This mirrors the check already present in `deployToken` (`require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")`) and is the direct analog of the missing `borrowGuardianPaused` check in the Morpho/Compound report. [7](#0-6) 

---

### Proof of Concept

**EVM path:**

1. Deploy any ERC20 `UnregisteredToken` on the EVM chain. Do **not** call `deployToken` or `addCustomToken`.
2. Approve `OmniBridge` to spend `1000 UnregisteredToken`.
3. Call `OmniBridge.initTransfer(address(UnregisteredToken), 1000, 0, 0, "victim.near", "")`.
4. `safeTransferFrom` succeeds; `InitTransfer` event is emitted; `currentOriginNonce` increments.
5. Relayer submits `fin_transfer` on NEAR with the proof.
6. `fin_transfer_callback` executes `token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` → **panic**.
7. NEAR transaction reverts. The 1000 `UnregisteredToken` remain locked in `OmniBridge` forever.

**StarkNet path (identical structure):**

1. Deploy any ERC20 on StarkNet not registered via `deploy_token`.
2. Call `omni_bridge.init_transfer(token_address, 1000, 0, 0, "victim.near", "")`.
3. `transfer_from` succeeds; `InitTransfer` event emitted.
4. NEAR `fin_transfer_callback` panics on `TokenDecimalsNotFound`.
5. Tokens permanently locked in the StarkNet bridge contract. [8](#0-7) [9](#0-8) [10](#0-9)
