Audit Report

## Title
Unguarded `sweepToken` and `unwrapWETH9` with Arbitrary `recipient` Allow Anyone to Drain Router/Adder Balances - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`sweepToken` and `unwrapWETH9` in `PeripheryPayments` are `public payable` with no access control and accept a fully caller-controlled `recipient` address. Any ERC-20 or WETH balance that lands on `MetricOmmSimpleRouter` or `MetricOmmPoolLiquidityAdder` can be redirected to an attacker's address by any unprivileged caller, resulting in direct loss of user principal.

## Finding Description
`PeripheryPayments.sweepToken` (L48–55) and `unwrapWETH9` (L37–45) are both declared `public payable` with no `msg.sender` check and accept an arbitrary `recipient` argument. There is no modifier, role check, or deadline guard restricting who may call them or where funds are sent.

`MetricOmmSimpleRouter` inherits `PeripheryPayments` directly:
```
contract MetricOmmSimpleRouter is MetricOmmSwapRouterBase, PeripheryPayments, SelfPermit, IMetricOmmSimpleRouter
```
As does `MetricOmmPoolLiquidityAdder`:
```
contract MetricOmmPoolLiquidityAdder is IMetricOmmPoolLiquidityAdder, PeripheryPayments
```

The `receive()` guard (L32–34) only blocks direct ETH transfers from non-WETH senders; it does not prevent ERC-20 tokens from being sent directly to the contract. Any ERC-20 balance on the router is immediately drainable by an unprivileged caller via `sweepToken(token, 0, attacker)`. Similarly, any WETH balance is drainable via `unwrapWETH9(0, attacker)`.

The existing payment test (`test_sweepToken_sendsFullBalance`, L57–67) confirms that `sweepToken` transfers the full router balance to any specified address with no caller restriction.

Two concrete exploit paths exist:
1. **Accidental ERC-20 transfer**: A user mistakenly sends tokens to the router. An attacker calls `sweepToken(token, 0, attacker)` and receives the full balance. Even if the victim attempts recovery first, an attacker can front-run with a different `recipient`.
2. **Two-step WETH/ETH pattern**: A user routes swap output to `address(router)` intending to follow up with `unwrapWETH9`. An attacker observing the mempool front-runs the second transaction with `unwrapWETH9(0, attacker)`, stealing the full WETH balance.

## Impact Explanation
Direct loss of user principal in ERC-20 tokens or ETH/WETH. Any balance that transiently or accidentally lands on `MetricOmmSimpleRouter` or `MetricOmmPoolLiquidityAdder` is immediately and completely drainable by any unprivileged EOA. The exact corrupted value is the full ERC-20 token balance (`IERC20(token).balanceOf(address(this))`) or full WETH balance (`IERC20(WETH).balanceOf(address(this))`). This meets the Critical/High direct loss of user principal threshold under the contest's allowed impact gate.

## Likelihood Explanation
No special privilege, setup, or precondition is required beyond a balance existing on the router. Accidental ERC-20 transfers to router addresses are a well-documented real-world occurrence. The two-step pattern is a natural mistake for users unfamiliar with the multicall requirement. The `receive()` guard provides a false sense of security (it only blocks ETH, not ERC-20). Any EOA can call `sweepToken` or `unwrapWETH9` with an arbitrary `recipient` at any time, and the attack is repeatable and front-runnable.

## Recommendation
Restrict `sweepToken` and `unwrapWETH9` so that the `recipient` is always `msg.sender`, eliminating the front-running redirect vector. Remove the `recipient` parameter from both functions and replace all uses of `recipient` with `msg.sender`. Alternatively, add an `onlyAuthorized` modifier or enforce that `recipient == msg.sender` with an explicit check. The `refundETH` function already correctly sends to `msg.sender` and requires no change.

## Proof of Concept
```solidity
// 1. Alice accidentally sends 1000 USDC to the router
usdc.transfer(address(router), 1000e6);

// 2. Attacker (Bob) calls sweepToken with his own address as recipient
// No access control, no msg.sender check, arbitrary recipient accepted
router.sweepToken(address(usdc), 0, bob);

// Result: Bob receives 1000 USDC, Alice loses her funds
assertEq(usdc.balanceOf(bob), 1000e6);
assertEq(usdc.balanceOf(address(router)), 0);
```

The existing test `test_sweepToken_sendsFullBalance` (L57–67 of `MetricOmmSimpleRouter.payments.t.sol`) already demonstrates this: it mints tokens directly to the router and confirms `sweepToken` transfers the full balance to any specified `recipient` with no caller restriction.   function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);

    if (balanceWETH > 0) {
      IWETH9(WETH).withdraw(balanceWETH);
      _transferETH(recipient, balanceWETH);
    }
  }

  /// @inheritdoc IPeripheryPayments
  function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);

    if (balanceToken > 0) {
      IERC20(token).safeTransfer(recipient, balanceToken);
    }
  }
