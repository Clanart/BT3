### Title
`ProofOfInclusion::valid()` Checks Against Its Own Internal Hash Instead of an External Trusted Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle proof module contains a tautological final check: it compares the computed hash against `self.root_hash()`, which is derived from the proof's own `combined_hash` field — the same value already verified in the loop. The method never compares against any externally-provided trusted root hash. This means `valid()` only verifies internal self-consistency of the proof structure, not that the proof corresponds to any particular committed tree root. An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key-value pair is included) and internally-consistent `layers`, and `valid()` will return `true`.

### Finding Description

In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`, the `ProofOfInclusion` struct has two methods:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- derived from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // existing_hash = last layer.combined_hash after loop
    }

    existing_hash == self.root_hash()  // tautological: last.combined_hash == last.combined_hash
}
```

After the loop completes without returning `false`, `existing_hash` equals the last `layer.combined_hash` (because the loop already verified `calculated_hash == layer.combined_hash` for every layer). `self.root_hash()` also returns `last.combined_hash`. So the final check is always `last.combined_hash == last.combined_hash` — unconditionally `true`. [1](#0-0) 

The analog to the reported bug is exact: in the original report, `minPayoutSize` is checked against `truePayoutSize` (before fee deduction) rather than `payoutAfterFee` (the value actually received). Here, `valid()` checks against `self.root_hash()` (an intermediate value derived from the proof itself) rather than an externally-provided trusted root hash (the value that actually matters for security).

`ProofOfInclusion` is a `Streamable` type exposed via Python bindings with `from_bytes()`, `valid()`, and `root_hash()` methods. [2](#0-1)  The Python stub confirms both methods are part of the public API. [3](#0-2) 

### Impact Explanation

Any caller that receives a `ProofOfInclusion` from an untrusted source and calls only `proof.valid()` to verify it is not actually verifying that the proof corresponds to any committed tree root. An attacker can:

1. Choose any arbitrary `node_hash` (claiming any key-value pair is included in the DataLayer tree).
2. Build a chain of `ProofOfInclusionLayer` values where each `combined_hash` is computed correctly from the previous step (internally consistent).
3. Serialize the forged `ProofOfInclusion` via `to_bytes()` and send it to a verifier.
4. The verifier calls `proof.valid()` → `true`, and then trusts the `node_hash` as proven.

This lets untrusted input prove invalid DataLayer state — forged inclusion of arbitrary key-value pairs — matching the allowed High impact: "DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."

### Likelihood Explanation

Medium. The `ProofOfInclusion` struct is a Streamable type with a public Python API. The misleading name `valid()` strongly implies complete proof validation, making it likely that callers use it as the sole check without separately comparing `proof.root_hash()` against a trusted root. The fuzz target and all tests call `proof.valid()` alone without any external root comparison. [4](#0-3) [5](#0-4) 

### Recommendation

The `valid()` method must accept an externally-provided trusted root hash and compare against it, rather than against `self.root_hash()`:

```diff
-pub fn valid(&self) -> bool {
+pub fn valid(&self, trusted_root: &Hash) -> bool {
     let mut existing_hash = self.node_hash;

     for layer in &self.layers {
         let calculated_hash = crate::calculate_internal_hash(
             &existing_hash,
             layer.other_hash_side,
             &layer.other_hash,
         );

         if calculated_hash != layer.combined_hash {
             return false;
         }

         existing_hash = calculated_hash;
     }

-    existing_hash == self.root_hash()
+    existing_hash == *trusted_root
 }
```

All call sites (including the Python binding `py_valid`) must be updated to pass the known trusted root hash obtained from a trusted source (e.g., the on-chain committed root).

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to forge inclusion of arbitrary node_hash
fake_node_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)

# Build one internally-consistent layer:
# combined_hash = SHA256(b"\x02" + other_hash + fake_node_hash)
h = hashlib.sha256()
h.update(b"\x02")
h.update(other_hash)
h.update(fake_node_hash)
combined = h.digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,   # Side::Left
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated from any real tree
assert forged_proof.valid(), "Forged proof accepted!"
# root_hash() returns the attacker-controlled combined hash
print("Claimed root:", forged_proof.root_hash().hex())
```

The tautological final check `existing_hash == self.root_hash()` passes because both sides equal `combined` — the attacker-chosen value. [6](#0-5) [7](#0-6)
