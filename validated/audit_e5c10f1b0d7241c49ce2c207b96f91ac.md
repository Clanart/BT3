### Title
Missing `recipient` Validation in `initTransfer` Enables Permanent Freezing of User Funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` on EVM (and `init_transfer` on StarkNet) accept an arbitrary `recipient` string without validating it is non-empty or a parseable cross-chain address. A user who submits an empty or malformed recipient will have their tokens permanently locked or burned on the origin chain with no recovery path, because the NEAR destination contract will fail to parse the recipient and reject finalization.

### Finding Description
In `OmniBridge.initTransfer`, the only input validation performed is `fee >= amount`. The `recipient` string parameter is passed directly into the Wormhole message payload without any format or emptiness check. [1](#0-0) 

On the NEAR side, `fin_transfer_callback` decodes the prover result and expects `init_transfer.recipient` to be a valid `OmniAddress`.
