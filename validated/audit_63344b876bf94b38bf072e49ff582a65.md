### Title
Missing Validation of New Class Hash in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by a contract corresponds to a previously declared class. This allows any contract to replace its class with an undeclared class hash, permanently freezing all funds held by the contract. The missing check is explicitly acknowledged in the code via a TODO comment.

---

### Finding Description

**Vulnerability class:** Two-step state initialization without atomicity enforcement — analogous to the external report's pattern where pool reserve initialization and liquidity supply happen in separate, unlinked steps.

In `execute_replace_class` at lines 878–916 of `syscall_impls.cairo`, the function accepts any `class_hash` value from the `ReplaceClassRequest` and updates the contract's `StateEntry` without verifying that the class hash was previously declared via a declare transaction. The code itself contains an explicit TODO acknowledging this missing check: [1](#0-0) 

The function then unconditionally writes the unvalidated class hash into `contract_state_changes`: [2](#0-1) 

This is the direct analog to the external report's vulnerability: class declaration (Step 1) and class assignment via `replace_class` (Step 2) are separate, unlinked steps. The OS enforces no invariant that Step 1 must precede Step 2 for the same class hash.

When a subsequent call is made to the contract with the invalid class hash, `execute_entry_point` reads the class hash and looks up the compiled class hash in `contract_class_changes`: [3](#0-2) 

Since the class was never declared, `dict_read` on `contract_class_changes` returns `0` (the default value for an uninitialized dict entry, as set by `dict_new()` in `initialize_state_changes`): [4](#0-3) 

`find_element` then fails to locate a compiled class with hash `0` in `compiled_class_facts_bundle`, causing the OS execution to abort. All subsequent calls to the contract revert, permanently freezing its funds. The committed state cannot be unwound.

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

A contract that calls `replace_class` with an undeclared class hash will have its funds permanently frozen. The state is committed to the blockchain with the invalid class hash. All future calls to the contract fail at the OS level because `find_element` cannot locate the compiled class. There is no recovery path — the funds are irrecoverably locked.

---

### Likelihood Explanation

**Medium.** Any deployed contract can invoke the `replace_class` syscall — it requires no privileged role. A malicious contract owner can intentionally freeze funds by calling `replace_class` with an undeclared class hash after users have deposited funds (e.g., into a vault, token contract, or multisig). The attack requires only the ability to deploy a contract and call a function within it, both of which are available to any unprivileged user.

---

### Recommendation

Add a validation check in `execute_replace_class` to verify that the new class hash corresponds to a declared class before updating the contract state. This can be done by reading from `contract_class_changes` and asserting the compiled class hash is non-zero:

```cairo
// Verify the new class hash was declared.
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the atomicity fix recommended in the external report: both steps (declaration and assignment) must be validated together before the state transition is committed.

---

### Proof of Concept

1. Attacker deploys a contract (class A) that appears to be a legitimate vault or token contract.
2. Users deposit funds into the contract.
3. Attacker calls a function in the contract that invokes `replace_class(undeclared_hash)`, where `undeclared_hash` is a felt value not present in `contract_class_changes`.
4. `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`) accepts the call, updates `contract_state_changes` with `class_hash = undeclared_hash`, and commits the state to the blockchain.
5. In subsequent blocks, when users attempt to withdraw funds, `execute_entry_point` (lines 154–166 of `execute_entry_point.cairo`) reads the contract's class hash = `undeclared_hash`, performs `dict_read` on `contract_class_changes` returning `0`, and `find_element` fails to locate compiled_class_hash = `0`.
6. All calls to the contract revert at the OS level. The funds are permanently frozen with no recovery mechanism.
