### Title
Undeclared Class Hash Accepted in `execute_replace_class` Without Validation — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary `class_hash` value from the caller without verifying that the hash corresponds to a previously declared class. This is directly analogous to the ICO report's pattern: a setter that mutates critical state without sanity-checking the new value. A malicious or buggy contract can permanently replace its own class hash with an undeclared value, bricking itself and permanently freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. By contrast, `execute_declare_transaction` enforces `assert_not_zero(compiled_class_hash)` and uses `dict_update` with `prev_value=0` to ensure a class is declared exactly once before it enters `contract_class_changes`: [2](#0-1) 

There is no cross-check between the `class_hash` supplied to `replace_class` and the set of hashes present in `contract_class_changes`. Any felt value — including one that was never declared — is accepted and committed to the global state via `state_update`. [3](#0-2) 

---

### Impact Explanation

Once `state_update` commits the squashed `contract_state_changes` to the Patricia Merkle Tree, the contract's class hash in the global state is permanently set to the undeclared value. [4](#0-3) 

Any subsequent `call_contract`, `library_call`, or direct invocation of that contract will attempt to look up the class by this hash. Since no compiled class fact exists for it, execution will fail unconditionally. There is no OS-level recovery path. All funds (ERC-20 balances, ETH, or any token held in the contract's storage) become permanently inaccessible.

**Impact class: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is a standard, permissionless StarkNet syscall callable by any contract from within its own execution context. No privileged role is required. A malicious contract developer can:

1. Deploy a contract that accepts user deposits.
2. At any point, call `replace_class` with an arbitrary felt (e.g., `1`) as the new class hash.
3. The OS processes the syscall in `execute_replace_class`, writes the invalid hash to `contract_state_changes`, and commits it.
4. All subsequent calls to the contract revert. Deposited funds are permanently frozen.

The attacker-controlled entry path is a standard `invoke` transaction — the most common transaction type on StarkNet — requiring no special access.

---

### Recommendation

Inside `execute_replace_class`, after reading `class_hash` from the request, verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). This mirrors the validation already present in `execute_declare_transaction`, which enforces `prev_value=0` to guarantee a class is registered before use. A `dict_read` on `contract_class_changes` with the new `class_hash` as key, followed by an `assert_not_zero` on the returned compiled class hash, would close the gap. [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys contract `VaultWithTrap` that:
   - Accepts ERC-20 deposits from users (stores balances in its own storage).
   - Exposes a `trap()` entry point that calls `replace_class(class_hash=0x1)` — a felt that was never declared.

2. Users deposit funds into `VaultWithTrap`. The contract's storage now holds their balances.

3. Attacker calls `trap()` via an `invoke` transaction. The OS executes `execute_replace_class`:
   - `class_hash = 0x1` is read from the request.
   - The TODO check is absent; no validation against `contract_class_changes` occurs.
   - `dict_update` writes `StateEntry(class_hash=0x1, ...)` for `VaultWithTrap`'s address.

4. `state_update` squashes and commits this to the Patricia tree. [6](#0-5) 

5. Any subsequent call to `VaultWithTrap` (e.g., `withdraw`) fails: the OS looks up class `0x1` in the compiled class facts bundle and finds nothing. All user funds are permanently frozen.
