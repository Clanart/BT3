### Title
Missing Declared-Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS Cairo program accepts any arbitrary class hash as a replacement without verifying that the hash corresponds to a declared class. An unprivileged contract deployer can exploit this to permanently render a contract inaccessible, freezing all funds it holds. The missing check is explicitly acknowledged in the code with a TODO comment.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS processes the `replace_class` syscall by writing the caller-supplied `class_hash` directly into `contract_state_changes` with no validation that the hash exists in `contract_class_changes`:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly confirms the missing invariant enforcement. Once the contract's class hash is overwritten with an undeclared value, every subsequent call to that contract will fail at the class-lookup stage because no valid class implementation can be provided to the prover. The contract becomes permanently inaccessible, and any ERC-20 token balances or other assets stored in its storage slots are permanently frozen.

This is structurally analogous to the reported staking vulnerability: just as staking to an address that cannot call `unstake` permanently locks funds, replacing a contract's class with an undeclared hash permanently locks all funds the contract holds, with no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value:
- No valid execution proof can be generated for any call to the contract (the prover cannot supply a class that does not exist).
- The contract's storage — including any ERC-20 balances, vault positions, or other assets — is permanently inaccessible.
- The L1 verifier accepts the resulting proof because the OS itself accepted the state transition without objection.
- There is no recovery mechanism: `replace_class` cannot be called again because the contract is already broken.

---

### Likelihood Explanation

The `replace_class` syscall is available to any deployed contract. Exploitation requires only that a contract calls `replace_class` with a class hash that is not present in `contract_class_changes`. This can occur via:

1. **Intentional attack**: A malicious deployer creates a contract holding shared funds (e.g., a vault or multisig) and calls `replace_class` with a fabricated hash, permanently freezing depositors' funds.
2. **Exploiting a legitimate contract**: Any contract that exposes a `replace_class` path callable by an unprivileged user (e.g., an upgradeable contract with insufficient access control) can be targeted by an attacker supplying an undeclared class hash.
3. **Accidental misconfiguration**: A developer passes an incorrect class hash (e.g., a Sierra hash instead of a CASM hash, or a hash from a different chain) and the OS silently accepts it.

The OS provides zero protection in all three cases.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, add an explicit check that the hash exists in `contract_class_changes`:

```cairo
// Verify the replacement class is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant already enforced during `declare` transactions (line 816–818 of `transaction_impls.cairo`), where `assert_not_zero(compiled_class_hash)` is required before updating `contract_class_changes`. [2](#0-1) 

---

### Proof of Concept

1. **Deploy a contract** (e.g., a simple vault that accepts ERC-20 deposits) using the `deploy` syscall or a `deploy_account` transaction.
2. **Deposit funds** into the vault contract so it holds a non-zero ERC-20 balance in its storage.
3. **Call `replace_class`** from within the vault contract, supplying a `class_hash` value that has never been declared on-chain (e.g., `0xdeadbeef` or any random felt).
4. **Observe**: `execute_replace_class` in `syscall_impls.cairo` writes the undeclared hash into `contract_state_changes` without any validation.
5. **Attempt any subsequent call** to the vault contract. The OS will look up the class hash (`0xdeadbeef`) in `contract_class_changes`, find nothing, and no valid proof can be generated.
6. **Result**: The vault contract is permanently inaccessible. All deposited funds are permanently frozen. The state transition is accepted by the L1 verifier because the OS proof is valid — the OS never checked the class hash.

The root cause is solely within the scoped file at lines 896–913 of `syscall_impls.cairo`, confirmed by the explicit `TODO` at line 898. [3](#0-2)
