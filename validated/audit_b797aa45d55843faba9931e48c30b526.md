### Title
Declare Transactions Bypass `max_l2_gas_amount` Cap in Stateless Validator, Admitting Unbounded L2 Gas Claims — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces a configurable `max_l2_gas_amount` ceiling for Invoke and DeployAccount transactions but explicitly skips that check for Declare transactions. A test in the same crate confirms this is the intended (but unfinished) behavior. Because the Starknet OS uses `l2_gas.max_amount` directly as the initial execution gas budget, a Declare transaction carrying `max_l2_gas_amount = u64::MAX` is admitted through the gateway and mempool with an execution gas budget that is orders of magnitude larger than the protocol operator intended.

---

### Finding Description

In `validate_resource_bounds`, the guard is written as an empty `if`-branch that silently skips the cap for Declare:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // nothing
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The companion test `valid_l2_gas_amount_on_declare` explicitly asserts that a Declare transaction with `max_amount = 200` passes when the configured limit is `100`: [2](#0-1) 

Meanwhile, the same amount is rejected for Invoke and DeployAccount: [3](#0-2) 

On the OS side, `get_initial_user_gas_bound` feeds `resource_bounds[L2_GAS_INDEX].max_amount` directly into the execution context as the initial gas budget: [4](#0-3) 

So the value that bypasses the gateway cap becomes the gas ceiling the OS enforces during execution of the Declare transaction.

The `min_gas_price` check does apply to Declare transactions, so `l2_gas.max_price_per_unit` must be ≥ `min_gas_price`. However, `max_possible_fee` uses saturating arithmetic: [5](#0-4) 

This means a Declare with `max_l2_gas_amount = u64::MAX` and `max_price_per_unit = min_gas_price` saturates to `Fee::MAX` for the L2 component, which the stateful balance check will reject — **unless** the attacker sets `max_price_per_unit = 0`. The `min_gas_price` check in the stateless validator would block that only if `min_gas_price > 0`. If the operator configures `min_gas_price = 0` (or the check is disabled via `validate_resource_bounds = false`), a Declare with `max_l2_gas_amount = u64::MAX` and `max_price_per_unit = 0` passes every gateway check, enters the mempool, and reaches the OS with a `u64::MAX` initial gas budget.

---

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A Declare transaction whose `l2_gas.max_amount` exceeds `config.max_l2_gas_amount` is admitted by the gateway and mempool without error. The OS then initialises execution with that oversized gas budget. While the block-level bouncer caps actual gas consumed, the inflated initial budget can allow the Declare's `__validate_declare__` entry point to run significantly more Sierra gas than the operator's `max_l2_gas_amount` policy permits, defeating the DoS-protection intent of that cap.

Additionally, any RPC fee-estimation call (`starknet_estimateFee`) for such a Declare will compute a `max_possible_fee` that saturates to `Fee::MAX`, returning an authoritative-looking but wrong fee estimate to callers.

---

### Likelihood Explanation

The condition is reachable by any unprivileged user who can submit a Declare transaction to the gateway. No special privilege, malicious peer, or external oracle is required. The bypass is unconditional for all Declare transactions — the empty branch fires on every call regardless of the transaction's content. The only mitigating factor is that `min_gas_price > 0` forces a non-zero L2 gas price, which causes the saturating fee to hit `Fee::MAX` and fail the stateful balance check. Operators running with `min_gas_price = 0` or `validate_resource_bounds = false` are fully exposed.

---

### Recommendation

Remove the empty Declare branch and apply the same `max_l2_gas_amount` ceiling to all transaction types:

```rust
// Apply to all transaction types uniformly.
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Update `valid_l2_gas_amount_on_declare` to expect an error when the amount exceeds the limit, and add a positive test for Declare transactions that stay within the limit.

---

### Proof of Concept

1. Configure the gateway with `validate_resource_bounds = true`, `max_l2_gas_amount = 1_000_000`, `min_gas_price = 0`.
2. Construct a V3 Declare transaction with:
   - `l2_gas.max_amount = u64::MAX`
   - `l2_gas.max_price_per_unit = 0`
   - `l1_gas.max_amount = 1`, `l1_gas.max_price_per_unit = 1` (to satisfy the non-zero fee check)
3. Submit via `starknet_addDeclareTransaction`.
4. **Observed**: the stateless validator returns `Ok(())` — the `max_l2_gas_amount` check is silently skipped for Declare.
5. **Expected**: the validator should return `MaxGasAmountTooHigh { gas_amount: u64::MAX, max_gas_amount: 1_000_000 }`.
6. The transaction enters the mempool and, when sequenced, the OS initialises the Declare's execution context with `initial_gas = u64::MAX`, far exceeding the operator's intended cap.

The existing test `valid_l2_gas_amount_on_declare` already documents and asserts this bypass as passing behavior: [6](#0-5)
