### Title
Meta-Transaction v0 Hash Lacks Nonce — Signature Replay Enables Repeated Unauthorized Execution - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall implementation computes a version-0 transaction hash that contains no nonce and performs no replay-protection check. Because `check_and_increment_nonce` explicitly skips all version-0 transactions, and because no "used hash" set is maintained, the same `(contract_address, calldata, signature)` tuple can be submitted an unlimited number of times across different blocks, causing repeated unauthorized execution of the target account's `__execute__` entry point.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` is a syscall that any executing contract can invoke. It:

1. Accepts a caller-supplied `contract_address`, `calldata`, and `signature`.
2. Computes a version-0 transaction hash via `compute_meta_tx_v0_hash`.
3. Constructs a synthetic `TxInfo` with `version=0` and `nonce=0`.
4. Calls the target contract's `__execute__` entry point with that synthetic context.

**Root cause — no nonce in the hash:**

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0`, meaning the nonce is never mixed into the hash:

```cairo
// transaction_hash.cairo L302-L313
let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    tx_hash_prefix=INVOKE_HASH_PREFIX,
    version=0,
    contract_address=contract_address,
    entry_point_selector=entry_point_selector,
    calldata_size=calldata_size,
    calldata=calldata,
    max_fee=0,
    chain_id=chain_id,
    additional_data_size=0,          // <-- nonce is absent
    additional_data=cast(0, felt*),
);
``` [1](#0-0) 

**Root cause — nonce check is explicitly skipped for version 0:**

`check_and_increment_nonce` contains an early return for any transaction whose version is 0, so the target contract's on-chain nonce is never read or incremented:

```cairo
// execute_transaction_utils.cairo L64-L67
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [2](#0-1) 

**Root cause — no used-hash registry in `execute_meta_tx_v0`:**

After computing `meta_tx_hash` and constructing the synthetic `TxInfo`, the function proceeds directly to `contract_call_helper` with no check or write to any "seen hashes" data structure:

```cairo
// syscall_impls.cairo L332-L393
let meta_tx_hash = compute_meta_tx_v0_hash(...);
...
tempvar new_tx_info = new TxInfo(
    version=0,
    nonce=0,          // always zero
    transaction_hash=meta_tx_hash,
    ...
);
...
contract_call_helper(remaining_gas=remaining_gas, ...);
// No dict_update or assertion that meta_tx_hash was not seen before.
``` [3](#0-2) 

Because the hash is a pure function of `(contract_address, selector, calldata, chain_id)`, any attacker who observes a valid `meta_tx_v0` invocation can replay the identical syscall in any future transaction, obtaining the same hash and the same signature validity.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A typical use of `meta_tx_v0` is gasless/delegated execution: a user signs a version-0 hash authorizing a specific calldata (e.g., a token transfer). An attacker who captures that `(calldata, signature)` pair can replay the syscall in subsequent transactions. Each replay causes the target account's `__execute__` to run with the same calldata and the same valid signature, executing the authorized action (e.g., transferring tokens) again and again until the account is drained.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is reachable by any unprivileged transaction sender: a caller simply deploys or invokes a contract that issues the syscall. All inputs (`contract_address`, `calldata`, `signature`) are observable on-chain from the first legitimate use. No privileged access, leaked key, or network-level attack is required — the attacker only needs to copy the syscall arguments from a prior block and re-submit them.

---

### Recommendation

Enforce replay protection at the OS level inside `execute_meta_tx_v0`. Two complementary approaches:

1. **Include a nonce in the hash.** Pass the target contract's current on-chain nonce as `additional_data` to `compute_meta_tx_v0_hash` (analogous to how `compute_l1_handler_transaction_hash` passes the L1 nonce), and increment that nonce after a successful call.

2. **Mark the hash as used.** Maintain a storage set (e.g., a `DictAccess` keyed by `meta_tx_hash`) and assert `prev_value=0` before writing `new_value=1`, ensuring each hash can only be consumed once — mirroring the pattern already used for class declarations (`prev_value=0` enforces single-use).

---

### Proof of Concept

1. **Block N:** Legitimate caller invokes a contract that issues `meta_tx_v0` with:
   - `contract_address = VICTIM`
   - `calldata = [transfer(ATTACKER, 1000_STRK)]`
   - `signature = <valid ECDSA sig over hash(VICTIM, __execute__, calldata, chain_id)>`
   - The OS computes `meta_tx_hash` (no nonce), calls `VICTIM.__execute__`, transfer succeeds.

2. **Block N+1 … N+k:** Attacker submits a transaction from any contract that re-issues the identical `meta_tx_v0` syscall with the same `contract_address`, `calldata`, and `signature`.
   - `compute_meta_tx_v0_hash` returns the same hash (inputs unchanged).
   - `check_and_increment_nonce` returns immediately because `version == 0`.
   - No used-hash check exists.
   - `VICTIM.__execute__` runs again, transferring another 1000 STRK to the attacker.

3. Repeat step 2 until `VICTIM`'s balance is zero — **complete loss of funds** with no additional attacker capability beyond observing the chain.
