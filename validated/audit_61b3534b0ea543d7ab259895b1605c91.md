### Title
Unchecked Multiplication Overflow in `denormalize_amount` Permanently Freezes User Funds During Cross-Chain Finalization - (File: near/omni-bridge/src/lib.rs)

### Summary

The `denormalize_amount` helper in the NEAR bridge contract performs an unchecked `u128` multiplication that can overflow when finalizing large cross-chain transfers for tokens whose origin chain has more decimal places than the normalized NEAR-side representation. Because `overflow-checks = true` is set in the workspace release profile, the overflow causes a contract panic, permanently blocking finalization of the transfer and freezing the user's already-locked or burned origin-chain funds.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

The `Decimals` struct stores two `u8` fields:

```rust
pub struct Decimals {
    pub decimals: u8,         // normalized (EVM bridge token, ≤ 18)
    pub origin_decimals: u8,  // actual origin-chain token decimals
}
``` [2](#0-1) 

When a token is registered via `bind_token_callback`, `decimals` is set to the EVM bridge token's normalized decimals (≤ 18) and `origin_decimals` is set to the actual EVM token decimals:

```rust
self.add_token(
    &deploy_token.token,
    &deploy_token.token_address,
    deploy_token.decimals,         // normalized, ≤ 18
    deploy_token.origin_decimals,  // actual EVM decimals
);
``` [3](#0-2) 

`denormalize_amount` is called in `fin_transfer_callback` with the raw amount from the cross-chain proof:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [4](#0-3) 

It is also called in `fast_fin_transfer` and `claim_fee_callback`: [5](#0-4) [6](#0-5) 

The workspace release profile explicitly enables overflow checks:

```toml
[profile.release]
overflow-checks = true
``` [7](#0-6) 

This means any overflow in `amount * (10_u128.pow(diff_decimals))` causes a **contract panic / abort**, not silent wrapping.

**Overflow condition:**

For a token with `origin_decimals = 24` and `decimals = 18` (`diff = 6`), the multiplication overflows when:

```
amount > u128::MAX / 10^6  ≈  3.4 × 10^32
```

In 18-decimal EVM bridge token units, `3.4 × 10^32` corresponds to approximately **340 trillion tokens** in a single transfer — large but within the `uint128` range accepted by the EVM `initTransfer` function, which takes a raw `uint128 amount` with no normalization before emitting the event:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // raw uint128, no normalization
    fee, nativeFee, recipient, message
);
``` [8](#0-7) 

For tokens with a larger `diff` (e.g., `diff = 18` for `decimals = 6`, `origin_decimals = 24`), the overflow threshold drops to `amount > 3.4 × 10^20`, which in 6-decimal units is still ~340 trillion tokens, but the raw uint128 value is much smaller and more easily reached.

### Impact Explanation

When `fin_transfer_callback` panics due to overflow:

1. The NEAR transaction reverts — the transfer is never recorded in contract state.
2. The user's tokens on the origin chain are already locked or burned (the EVM `initTransfer` already executed).
3. Every subsequent finalization attempt for the same proof will hit the same overflow and panic again.
4. The transfer is **permanently unfinalizeable** — the user's funds are irrecoverably frozen.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

The overflow requires a single transfer whose raw EVM amount exceeds `u128::MAX / 10^diff`. For the most common case (`diff = 6`), this is ~340 trillion tokens. While large, this is within the uint128 range and is reachable for high-supply tokens (e.g., SHIB, PEPE, or similar meme tokens with quadrillion-scale supplies). For `diff = 18`, the threshold in raw uint128 units is lower and more easily reached. No privileged access is required — any user can call `initTransfer` on the EVM bridge with an arbitrary `uint128` amount.

### Recommendation

Replace the unchecked multiplication in `denormalize_amount` with a checked variant and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Option<u128> {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount.checked_mul(10_u128.pow(diff_decimals))
}
```

All call sites (`fin_transfer_callback`, `fast_fin_transfer`, `claim_fee_callback`, `denormalize_fee`) should handle the `None` case by panicking with a descriptive error message (e.g., `BridgeError::AmountOverflow`) rather than letting an arithmetic panic propagate. This converts a silent, unrecoverable failure into an explicit, diagnosable rejection.

### Proof of Concept

1. Register a token via `bind_token` with `decimals = 18` and `origin_decimals = 24` (e.g., a NEAR-native token bridged to EVM, where the EVM bridge token is normalized to 18 decimals).
2. On the EVM bridge, call `initTransfer` with `amount = 4 × 10^32` (a valid `uint128`, representing 400 trillion tokens in 18-decimal units). The EVM bridge locks/burns the tokens and emits `InitTransfer`.
3. Submit the proof to NEAR's `fin_transfer`.
4. `fin_transfer_callback` calls `denormalize_amount(4e32, {decimals:18, origin_decimals:24})`:
   - `diff_decimals = 6`
   - `4e32 * 10^6 = 4e38 > u128::MAX (~3.4e38)` → **overflow → panic**
5. The NEAR transaction fails. The user's EVM tokens are already burned. The transfer can never be finalized. Funds are permanently frozen. [1](#0-0) [9](#0-8)
