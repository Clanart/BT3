### Title
Missing Post-Normalization Amount Validation in `init_transfer` Causes Permanent Freeze of User Funds - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `init_transfer` function validates only that `fee < amount` using pre-normalization values. It does not verify that `normalize_amount(amount - fee) > 0`. When a user bridges a small amount with a non-zero fee such that `(amount - fee) < 10^(origin_decimals - decimals)`, tokens are burned/locked at `init_transfer` time, but `sign_transfer` will always revert with `ERR_INVALID_AMOUNT_TO_TRANSFER` because the normalized amount is 0. The user's tokens are permanently frozen with no recovery mechanism.

---

### Finding Description

**Step 1 — Fee check at `init_transfer` uses pre-normalization values only.**

In `init_transfer`, the only fee validation is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This check passes as long as `fee < amount`, regardless of what the normalized amount will be.

**Step 2 — Tokens are burned/locked immediately in `init_transfer_internal`.**

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [2](#0-1) 

The tokens are burned (for deployed/bridged tokens) or locked (for native tokens) before `sign_transfer` is ever called.

**Step 3 — `sign_transfer` computes the normalized amount and checks it is > 0.**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

**Step 4 — `normalize_amount` uses floor division.**

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

If `amount - fee < 10^(origin_decimals - decimals)`, then `normalize_amount(amount - fee) = 0` due to floor division. `sign_transfer` will always revert.

**Step 5 — `sign_transfer` is `#[trusted_relayer]`; the user cannot call it.** [5](#0-4) 

The user cannot call `sign_transfer` themselves. The relayer, seeing that the call would fail, will not call it. The transfer message stays in `pending_transfers` indefinitely, but the tokens are already burned/locked. There is no `cancel_transfer` or refund mechanism in the contract.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For deployed (bridged) tokens: `burn_tokens_if_needed` fires a detached burn promise. The tokens are destroyed and cannot be recovered.

For native tokens: `lock_tokens_if_needed` increases `locked_tokens`. The tokens are locked in the bridge with no standard recovery path.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Low to medium.** Any unprivileged user can trigger this by calling `ft_transfer_call` with a small amount and a non-zero fee such that `amount - fee < 10^diff_decimals`. The threshold depends on the decimal difference:

- NEAR (24 decimals) → ETH (18 decimals): `diff = 6`, threshold = `1,000,000` base units
- NEAR (24 decimals) → Solana (9 decimals): `diff = 15`, threshold = `10^15` base units

For tokens with large decimal differences (e.g., NEAR → Solana), the vulnerable range is very wide. A user bridging any amount less than `10^15` base units with a non-zero fee would trigger this.

The user does not need to be malicious — this can happen accidentally when a user sets a small fee on a small transfer.

---

### Recommendation

Add a check in `init_transfer` (before burning/locking tokens) that verifies the post-normalization amount is non-zero. This requires looking up the destination token address and its decimals at `init_transfer` time:

```rust
// After computing transfer_message and before init_transfer_internal:
if let Some(token_address) = self.get_token_address(
    init_transfer_msg.get_destination_chain(),
    self.get_token_id(&OmniAddress::Near(token_id.clone())),
) {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

### Proof of Concept

**Setup**: Token registered with `origin_decimals = 24`, `decimals = 18` (diff = 6, `10^diff = 1,000,000`).

1. User calls `ft_transfer_call` with `amount = 1,000,001` and `fee = 2`.
2. `init_transfer` check: `2 < 1,000,001` → passes.
3. `amount_without_fee = 1,000,001 - 2 = 999,999`.
4. `normalize_amount(999,999) = 999,999 / 1,000,000 = 0` (floor division).
5. `init_transfer_internal` burns/locks the user's `1,000,001` tokens.
6. Relayer calls `sign_transfer` → `require!(0 > 0)` → panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. User's `1,000,001` tokens are permanently lost. The transfer message remains in `pending_transfers` with no recovery path. [6](#0-5) [7](#0-6) [3](#0-2) [4](#0-3)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
