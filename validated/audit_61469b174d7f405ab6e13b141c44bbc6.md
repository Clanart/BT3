### Title
Unchecked Constructor Revert in `execute_deploy` Syscall Causes Proof Failure — (`execution/syscall_impls.cairo`, `execution/deploy_contract.cairo`)

---

### Summary

The `execute_deploy` syscall implementation in the StarkNet OS unconditionally hardcodes `failure_flag=0` in the response header and delegates to `deploy_contract`, which hard-asserts `is_reverted = 0`. There is no code path that handles a reverting constructor. An unprivileged user can trigger this by submitting an invoke transaction that calls the `deploy` syscall with a class whose constructor reverts. The sequencer (blockifier) will include the outer transaction as a reverted transaction, but the OS Cairo program will fail to generate a valid proof because the hard assertion is violated. This results in a permanent inability to prove the block — a total network halt.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–539), the OS calls `deploy_contract` and then unconditionally writes a success response:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

Inside `deploy_contract` (`deploy_contract.cairo`, lines 82–92), the constructor is executed and then the return value is hard-asserted to be non-reverted:

```cairo
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [2](#0-1) 

The `assert is_reverted = 0` is a Cairo constraint, not a hint. If the constructor reverts, `is_reverted` will be `1`, and this constraint is unsatisfiable — the proof is invalid regardless of what the sequencer provides as hints.

Critically, `execute_entry_point` calls `call_execute_syscalls` even for reverted outer transactions (to process the syscall segment), meaning `execute_deploy` is reached and `deploy_contract` is called even when the outer call has already reverted:

```cairo
with syscall_ptr {
    call_execute_syscalls(
        block_context=block_context,
        execution_context=execution_context,
        syscall_ptr_end=entry_point_return_values.syscall_ptr,
    );
}
``` [3](#0-2)

### Impact Explanation

When the OS Cairo program encounters a reverting constructor inside `deploy_contract`, the hard `assert is_reverted = 0` constraint is violated. This makes the STARK proof for the entire block unprovable. No valid proof can be generated for the block, so the block cannot be finalized on L1. The network cannot confirm any new transactions until the block is abandoned or the issue is resolved — a total network shutdown matching the **High: Network not being able to confirm new transactions** impact category.

---

### Likelihood Explanation

Any unprivileged user can:
1. Declare a contract class whose constructor unconditionally reverts (e.g., `assert 1 = 0`).
2. Submit an invoke transaction calling any contract that invokes the `deploy` syscall with that class hash.

The blockifier (sequencer execution engine) will simulate the transaction, observe the constructor revert, mark the outer invoke as reverted, and include it in the block as a standard reverted transaction (charging gas up to the revert point). This is normal StarkNet behavior for reverted transactions. The OS then processes the syscall segment of the reverted outer call, reaches `execute_deploy`, calls `deploy_contract`, re-executes the constructor (which reverts), and hits the unsatisfiable constraint. The block cannot be proven.

---

### Recommendation

Remove the hard `assert is_reverted = 0` from `deploy_contract` and instead propagate the `is_reverted` flag back to `execute_deploy`. In `execute_deploy`, write `failure_flag=is_reverted` in the `ResponseHeader` and write a `FailureReason` when `is_reverted != 0`, mirroring the pattern already used in `contract_call_helper`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This is already acknowledged by the existing TODO comment. The fix should also ensure that when the constructor reverts, the state changes introduced by `deploy_contract` (the `dict_update` that sets the new class hash) are rolled back via the revert log.

---

### Proof of Concept

1. Declare a contract class `RevertingClass` whose constructor body is `assert 1 = 0` (always reverts).
2. Deploy a contract `Caller` that, in its `__execute__` entry point, calls `deploy(class_hash=RevertingClass, ...)`.
3. Submit an invoke transaction targeting `Caller.__execute__`.
4. The blockifier simulates: constructor reverts → `deploy` syscall fails → `Caller` reverts → outer invoke is marked reverted and included in the block.
5. The OS processes the block: reaches `execute_deploy` for the reverted `Caller` syscall segment → calls `deploy_contract` → re-executes the constructor → constructor reverts → `assert is_reverted = 0` is violated.
6. No valid STARK proof can be generated for the block. The network halts.
