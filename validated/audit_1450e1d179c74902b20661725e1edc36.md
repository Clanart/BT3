### Title
Silent Mint on Destroyed Bridge Token Contract Permanently Freezes Cross-Chain Funds — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` calls `IBridgeToken.mint` and `ICustomMinter.mint` — both void-returning functions — via high-level Solidity calls with no contract existence check. If the target contract has no deployed bytecode (e.g., the implementation was selfdestructed or the custom minter was destroyed), the EVM returns `success=true` with empty returndata. Because Solidity only reverts on returndata decode failure for functions with return values, a void `mint` call to a codeless address silently succeeds. The destination nonce is permanently consumed, no tokens are minted, and the user's cross-chain transfer is irrecoverably lost.

---

### Finding Description

In `OmniBridge.finTransfer`, after signature verification and nonce consumption, the contract dispatches token delivery through one of several branches:

**Branch 1 — bridge token mint (lines 337–349):**
```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount
        );
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount,
            payload.message
        );
    }
}
```

**Branch 2 — custom minter mint (lines 331–336):**
```solidity
} else if (customMinters[payload.tokenAddress] != address(0)) {
    ICustomMinter(customMinters[payload.tokenAddress]).mint(
        payload.tokenAddress,
        payload.recipient,
        payload.amount
    );
}
```

`IBridgeToken.mint` and `ICustomMinter.mint` are both declared `void` (no return value): [1](#0-0) [2](#0-1) 

In Solidity, a high-level call to a void function on an address with zero bytecode:
1. The EVM executes `CALL` to the codeless address → returns `(success=1, returndata_size=0)`.
2. Solidity checks `success == true` → passes.
3. Because the function is `void`, Solidity performs **no returndata length check** and does not revert.
4. Execution continues normally.

The nonce is marked used **before** the mint call: [3](#0-2) 

So the sequence is: nonce consumed → mint silently no-ops → event emitted → transfer irrecoverable.

By contrast, the `safeTransfer`/`safeTransferFrom` paths (line 351, 395, 407) use OpenZeppelin `SafeERC20` which internally calls `Address.functionCall`, which **does** check `address.code.length > 0` before calling. The `mint` paths have no equivalent guard. [4](#0-3) 

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

When `finTransfer` is called for a bridge token whose contract has been destroyed:
- The `completedTransfers[destinationNonce]` flag is set to `true` permanently.
- `mint` silently no-ops — zero tokens reach the recipient.
- The origin-chain funds (already burned or locked) cannot be recovered because the destination nonce is consumed and cannot be replayed.
- The user loses the full bridged amount with no recourse.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

`BridgeToken` is a UUPS proxy. The proxy itself cannot be selfdestructed (no such function), but the **implementation** contract can be selfdestructed if it is ever called directly with an unprotected path (e.g., a re-initialization exploit on the bare implementation). If the implementation is selfdestructed, `delegatecall` from the proxy goes to `address(0)`, which returns `(success=1, empty)` — the proxy forwards this back to `OmniBridge`, which sees a successful void call.

For `customMinters`: these are arbitrary contracts set by the admin. Any custom minter that contains a `selfdestruct` path (intentional or via a bug) can be destroyed, triggering the silent-success path for all future `finTransfer` calls routed through it.

Likelihood is **medium**: requires a contract destruction event, which is uncommon but has occurred in production (e.g., Parity multisig, various UUPS implementation attacks). The bridge's long operational lifetime increases exposure.

---

### Recommendation

Add an explicit contract existence check before each void `mint`/`burn` call that does not go through `SafeERC20`. The cheapest approach is an inline `EXTCODESIZE` guard:

```solidity
function _requireHasCode(address addr) internal view {
    require(addr.code.length > 0, "ERR_NO_CONTRACT");
}
```

Call it immediately before each unguarded dispatch:

```solidity
// Before IBridgeToken mint
_requireHasCode(payload.tokenAddress);
IBridgeToken(payload.tokenAddress).mint(...);

// Before ICustomMinter mint
address minter = customMinters[payload.tokenAddress];
_requireHasCode(minter);
ICustomMinter(minter).mint(...);
```

Alternatively, use OpenZeppelin's `Address.functionCall` wrapper for all external calls, which already performs this check, consistent with how `SafeERC20` is used elsewhere in the same contract.

### Proof of Concept

1. A `BridgeToken` is deployed and registered: `isBridgeToken[tokenProxy] = true`.
2. An attacker (or bug) causes the `BridgeToken` implementation to be selfdestructed. The proxy address still has code (the proxy bytecode), but `delegatecall` to the destroyed implementation returns `(1, 0)`.
3. A user initiates a cross-chain transfer on the origin chain; the MPC signs a `finTransfer` payload with `tokenAddress = tokenProxy`.
4. Anyone calls `OmniBridge.finTransfer(sig, payload)`:
   - Line 287: `completedTransfers[destinationNonce] = true` — nonce consumed.
   - Line 339: `IBridgeToken(tokenProxy).mint(recipient, amount)` — proxy delegates to destroyed impl → `(success=1, returndata="")` → Solidity sees void success → no revert.
   - Line 359: `FinTransfer` event emitted.
5. The recipient receives zero tokens. The nonce is permanently used. The origin-chain funds are gone. The transfer cannot be retried. [5](#0-4) [6](#0-5) [7](#0-6)
