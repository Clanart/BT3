### Title
`HyperliquedBridgeToken` 3-arg `mint` unconditionally routes all minted tokens to `_systemAddress`, enabling permanent misdirection of bridged assets via non-empty `message` field — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HyperliquedBridgeToken.mint(address,uint256,bytes)` mints tokens to the recipient then immediately transfers the full amount to `_systemAddress` via `_update`. `OmniBridge.finTransfer` selects this 3-arg overload whenever `payload.message.length > 0`. Because the `message` field originates from attacker-controlled `initTransfer` input and is included verbatim in the MPC-signed payload, any caller who can produce a valid `DestinationChainMsg`-formatted `msg` string causes the recipient to receive zero tokens while the full amount is parked at `_systemAddress`.

---

### Finding Description

**Root cause — `HlBridgeToken.sol` lines 76–83:**

```solidity
function mint(
    address account,
    uint256 value,
    bytes memory          // parameter is unnamed and never read
) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);   // immediately drains recipient
}
```

`_mint` credits `account` with `value` tokens. The immediately following `_update(account, _systemAddress, value)` transfers the entire balance away, leaving `account` at zero. The `bytes memory` argument is completely ignored — any non-empty byte string produces the same outcome.

**Branching logic — `OmniBridge.sol` lines 337–349:**

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient, payload.amount, payload.message   // 3-arg path
        );
    }
}
```

The sole criterion for choosing the 3-arg path is `payload.message.length > 0`. There is no check that the token is a `HyperliquedBridgeToken`, no validation of the message content, and no guard against the destructive side-effect of the override.

**Message propagation — `near/omni-bridge/src/lib.rs` lines 487–499:**

```rust
let message = DestinationChainMsg::from_json(&transfer_message.msg)
    .and_then(|s| s.destination_msg())
    .unwrap_or_default();
```

The NEAR bridge converts the `msg` string from the `InitTransfer` event into the `message` bytes that are Borsh-encoded and MPC-signed. If the attacker supplies a well-formed `DestinationChainMsg` JSON (a public type in `omni-types`), `destination_msg()` returns non-empty bytes, and the MPC signs a payload with `message.length > 0`. If parsing fails, `unwrap_or_default()` yields empty bytes — but the format is discoverable from the open-source codebase.

**Recovery is blocked.** Tokens parked at `_systemAddress` can only be released by `_systemAddress` itself calling `coreReceiveWithData` (line 114: `if (msg.sender != _systemAddress) revert NotSystemAddress()`). The recipient has no recourse.

---

### Impact Explanation

The intended recipient of a cross-chain transfer receives **zero tokens**. The full `payload.amount` is minted and immediately transferred to `_systemAddress`. From the recipient's perspective this is indistinguishable from theft: their expected balance is zero, the tokens are held by a system address they cannot interact with, and the source-chain tokens have already been burned/locked. This satisfies **Critical — direct theft / unauthorized misdirection of bridged assets**.

---

### Likelihood Explanation

- `initTransfer` is a public, permissionless function on any supported source chain.
- The `message` field is `string calldata` with no length or content restriction.
- The `DestinationChainMsg` type is defined in the open-source `omni-types` crate; its JSON schema is publicly readable.
- The NEAR MPC signs the payload as constructed — it has no knowledge of whether the destination token is a `HyperliquedBridgeToken` or whether the message is semantically valid for HyperCore.
- No privileged role, leaked key, or colluding validator is required.

---

### Recommendation

1. **Remove the unconditional `_update` from the 3-arg `mint`.** The HyperCore accounting model should be implemented in a dedicated function (e.g., `mintToHyperCore`) that is called only through a verified HyperCore-specific path, not through the generic `IBridgeToken.mint` interface.

2. **Alternatively, validate the message in `OmniBridge.finTransfer`** before dispatching to the 3-arg path: confirm the token implements a HyperCore-aware interface and that the message is a well-formed HyperCore routing payload.

3. **Do not use `payload.message.length > 0` as the sole dispatch criterion** for a mint overload that has destructive side-effects on the recipient's balance.

### Proof of Concept

```solidity
// 1. Deploy HyperliquedBridgeToken, set owner = OmniBridge
// 2. Register token: isBridgeToken[token] = true, customMinters[token] = address(0)

// 3. Attacker calls on source chain:
omniBridge.initTransfer(
    tokenAddress,
    1e18,
    0,          // fee
    0,          // nativeFee
    "evm:0xRecipient",
    '{"destination_msg":"0x01"}'  // valid DestinationChainMsg JSON → non-empty bytes
);

// 4. NEAR MPC signs TransferMessagePayload with message = <non-empty bytes>
// 5. Relayer calls:
omniBridge.finTransfer(signature, payload);
// → isBridgeToken branch taken
// → payload.message.length > 0 → 3-arg mint called
// → _mint(recipient, 1e18) then _update(recipient, _systemAddress, 1e18)

// 6. Assert:
assert(token.balanceOf(recipient) == 0);
assert(token.balanceOf(_systemAddress) == 1e18);
```
