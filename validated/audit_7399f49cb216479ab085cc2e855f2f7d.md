### Title
Failed State-Part Upload Unconditionally Commits `AllDumped` Progress Marker — (`File: nearcore/src/state_sync.rs`)

### Summary

`StateDumper::check_parts_upload()` calls `dump.await_parts_upload()` which returns `(ShardId, anyhow::Result<()>)`. The `Err` branch only logs the failure. Immediately after the `match`, `set_state_sync_dump_progress(...AllDumped...)` is written to the DB unconditionally — regardless of whether the upload succeeded or failed. This is the exact structural analog of the Timelock bug: a side-effect that should only fire on success fires on both success and failure.

### Finding Description

In `nearcore/src/state_sync.rs`, `check_parts_upload()`:

```rust
async fn check_parts_upload(&mut self) {
    let CurrentDump::InProgress(dump) = &mut self.current_dump else {
        return std::future::pending().await;
    };
    let (shard_id, result) = dump.await_parts_upload().await;

    match result {
        Ok(()) => { tracing::info!(..., "shard dump finished"); }
        Err(error) => { tracing::error!(..., "shard dump failed"); }  // ← only logs
    }

    // ← runs unconditionally, even when result == Err
    self.chain.chain_store().set_state_sync_dump_progress(
        shard_id,
        Some(StateSyncDumpProgress::AllDumped {
            epoch_id: dump.epoch_id,
            epoch_height: dump.epoch_height,
        }),
    );

    if dump.dump_state.is_empty() {
        self.current_dump = CurrentDump::Done(dump.epoch_id);
    }
}
``` [1](#0-0) 

`dump.await_parts_upload()` resolves from the `oneshot::Receiver<anyhow::Result<()>>` stored in `ShardDump.upload_parts`. The sender side, `dump_shard_state()`, explicitly sends `Err` on the first upload failure and returns early:

```rust
while let Some(result) = tasks.next().await {
    if result.is_err() {
        let _ = sender.send(result);   // ← sends Err
        return;
    }
}
``` [2](#0-1) 

So `Err` is a reachable, non-hypothetical result. When it arrives, `check_parts_upload` still writes `AllDumped` to `DBCol::BlockMisc` under the key `STATE_SYNC_DUMP:<shard_id>`.

On the next node restart, `init()` calls `check_old_progress()`, which reads every stored progress entry. `AllDumped` maps to `done = true`:

```rust
StateSyncDumpProgress::AllDumped { epoch_id, .. } => (epoch_id, true),
``` [3](#0-2) 

And `check_old_progress` removes the shard from the pending dump set when `done == true`:

```rust
} else if done {
    dump.dump_state.remove(&shard_id);
    senders.remove(&shard_id);
}
``` [4](#0-3) 

The shard is permanently skipped for that epoch. No retry is ever attempted.

### Impact Explanation

`StateSyncDumpProgress::AllDumped` is the commitment that all state parts for a shard/epoch pair are present in external storage and available for download. When this marker is written after a failed upload, the commitment is false: the parts are absent from external storage, but the dumper node believes (and persists) that they are complete.

Any node attempting state sync for that epoch will request those parts from external storage, find them missing, and be unable to complete state sync. Because the dumper will never retry (the `AllDumped` marker prevents it), the missing parts remain absent for the entire epoch window. This can block new nodes from joining the network and prevent crashed nodes from recovering via state sync.

### Likelihood Explanation

The failure path is reachable through ordinary infrastructure events: transient S3 errors, network timeouts, permission errors, or a state snapshot not yet being ready (explicitly called out in the code comment at line 185–187). No attacker action is required. The broken invariant fires automatically whenever any single part upload task returns an error. [5](#0-4) 

### Recommendation

Gate the `set_state_sync_dump_progress` call on the upload result:

```rust
match result {
    Ok(()) => {
        tracing::info!(..., "shard dump finished");
        self.chain.chain_store().set_state_sync_dump_progress(
            shard_id,
            Some(StateSyncDumpProgress::AllDumped {
                epoch_id: dump.epoch_id,
                epoch_height: dump.epoch_height,
            }),
        );
    }
    Err(error) => {
        tracing::error!(..., "shard dump failed");
        // Leave progress as InProgress so the next restart retries.
    }
}
```

On failure, the progress entry should remain `InProgress` (or be explicitly reset to `InProgress`) so that `check_old_progress` on the next restart does not skip the shard.

### Proof of Concept

1. Configure a node with state sync dump enabled pointing to an S3 bucket.
2. Arrange for the S3 `put_object` call to fail for one part (e.g., revoke write permissions mid-dump, or inject a network fault).
3. `dump_shard_state` sends `Err` on the `oneshot` channel.
4. `check_parts_upload` receives `Err`, logs it, then writes `AllDumped` to the DB.
5. Restart the node. `init()` → `check_old_progress()` reads `AllDumped` → removes the shard from `dump_state` → no re-dump occurs.
6. A second node attempts state sync for that epoch. It requests the missing parts from external storage. The parts are absent. State sync fails permanently for that epoch on this dumper node. [6](#0-5) [7](#0-6)
