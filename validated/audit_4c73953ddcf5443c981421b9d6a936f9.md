### Title
Missing Shard-Index Position Verification in `set_state_header` Allows Cross-Shard State Installation — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` verifies that the supplied chunk is *a member* of the `chunk_headers_root` Merkle tree but never verifies *which position* (shard index) that chunk occupies. Because `verify_path` is a pure membership proof with no index binding, a malicious peer can supply shard I's real chunk (with its real Merkle path) in response to a state-sync request for shard J. The function stores the header under `StateHeaderKey(shard_J, sync_hash)`, and the subsequent `set_state_part` call extracts shard I's `prev_state_root` to validate all parts for shard J, ultimately installing shard I's trie as shard J's state.

---

### Finding Description

**Entrypoint and target symbol**

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` (lines 368–531) is the production entrypoint. It is called from `client_actor.rs` when a syncing node receives a `ShardStateSyncResponseHeader` from a peer.

**The missing guard**

The function performs five checks:

1. `validate_chunk_proofs` — verifies the chunk's internal hash, tx-root, and receipts-root consistency. It does **not** check `chunk.shard_id()` against the caller-supplied `shard_id`. [1](#0-0) 

2. `verify_path` for the chunk — verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the `chunk_headers_root` Merkle tree. The `MerklePath` type is `Vec<MerklePathItem { hash, direction }>` — it encodes **no leaf index**. The function therefore proves only membership, not position. [2](#0-1) [3](#0-2) 

3. `verify_path` for `prev_chunk_header` — same position-blind membership proof. [4](#0-3) 

4. Receipt-proof chain — uses the caller-supplied `shard_id` (J) to hash receipts, so the attacker satisfies this with shard J's real on-chain incoming receipts and their real Merkle proofs. [5](#0-4) 

5. `validate_state_root_node` — validates the `state_root_node` against `chunk_inner.prev_state_root()`, which is shard I's state root. The attacker supplies shard I's real state root node, so this passes. [6](#0-5) 

**No check of the form `chunk.shard_id() == shard_id` exists anywhere in the function.**

**Storage under the wrong key**

After all checks pass, the header is stored under `StateHeaderKey(shard_id=J, sync_hash)` — keyed by the caller-supplied shard J, not by the chunk's actual shard I. [7](#0-6) 

**Propagation into `set_state_part`**

`set_state_part` reads the stored header for shard J, extracts `prev_state_root` from the chunk in that header (which is shard I's `prev_state_root`), and validates every state part for shard J against that wrong root. [8](#0-7) 

The exact corrupted value is: **`state_root` used in `validate_state_part` for shard J equals shard I's `prev_state_root`**, causing the syncing node to accept shard I's trie data as shard J's trie and install it via `set_state_finalize`.

---

### Impact Explanation

A syncing node that accepts the crafted header will:
- Validate all state parts for shard J against shard I's state root (parts for shard I pass; parts for shard J fail or are never provided).
- Install shard I's trie as shard J's trie in `set_state_finalize`.
- Operate with a permanently corrupted shard J state, producing wrong execution results, wrong receipts, and diverging from the canonical chain.

The `ShardStateSyncResponseHeaderV2` documentation confirms the invariant that is violated: the chunk must be `B_prev.chunks[shard_id]`, i.e., the chunk at the shard's index in the previous block. [9](#0-8) 

---

### Likelihood Explanation

**Practical constraints that must be satisfied:**

1. The attacker must be a peer that the syncing node accepts state-sync responses from (any network peer in the p2p state-sync path).
2. The receipt-proof chain (step 4) must end at `prev_chunk_header.height_included()`. Since `prev_chunk_header` is shard I's previous chunk, this height must match the range of shard J's incoming receipts. In a healthy network where all shards produce chunks at every height, shard I and shard J have previous chunks at the same height, so the attacker can supply shard J's real incoming receipts for the correct range.
3. All data the attacker needs (shard I's chunk, its Merkle proof, shard J's incoming receipts and their proofs, shard I's state root node) is public on-chain data.

The attack is therefore constructible from public chain data with no forged signatures and no privileged access.

---

### Recommendation

Add an explicit shard-index binding check immediately after the `verify_path` call for the chunk. Resolve the shard layout for the sync block's epoch, compute `expected_shard_index = shard_layout.get_shard_index(shard_id)`, and verify that the Merkle proof path corresponds to that index (e.g., by using a position-aware proof variant such as `verify_path_with_index`, which already exists in the codebase). [10](#0-9) 

Alternatively, add a direct check: `chunk.shard_id() == shard_id`, rejecting the header if the chunk's embedded shard ID does not match the requested shard.

The server-side header construction in `get_state_response_header` already correctly selects the chunk at `prev_shard_index`; the receiver-side validation must enforce the same binding. [11](#0-10)

### Proof of Concept

```
Setup: two-shard network, shard_I = ShardId(0), shard_J = ShardId(1), sync_hash = H.

1. Obtain from the chain:
   - chunk_I: the real ShardChunk for shard 0 at B_prev (block before H)
   - chunk_proof_I: the Merkle path for chunk_I in B_prev.chunk_headers_root
   - prev_chunk_I: the real previous ShardChunk for shard 0
   - prev_chunk_proof_I: its Merkle path
   - incoming_receipts_J: the real incoming receipts for shard 1 from sync_hash down to prev_chunk_I.height_included()
   - root_proofs_J: the corresponding root proofs (real on-chain data)
   - state_root_node_I: the real StateRootNode for shard 0's prev_state_root

2. Construct:
   header = ShardStateSyncResponseHeaderV2 {
       chunk: chunk_I,
       chunk_proof: chunk_proof_I,
       prev_chunk_header: Some(prev_chunk_I),
       prev_chunk_proof: Some(prev_chunk_proof_I),
       incoming_receipts_proofs: incoming_receipts_J,
       root_proofs: root_proofs_J,
       state_root_node: state_root_node_I,
   }

3. Send header to syncing node as response to its request for shard_J's header.

4. Node calls set_state_header(shard_id=shard_J, sync_hash=H, header).
   - validate_chunk_proofs(chunk_I) → Ok (internal consistency passes)
   - verify_path(chunk_headers_root, chunk_proof_I, ChunkHashHeight(chunk_I)) → true (chunk_I IS in the tree)
   - verify_path for prev_chunk_I → true
   - receipt proofs for shard_J → true (real data)
   - validate_state_root_node(state_root_node_I, chunk_I.prev_state_root) → Ok
   - Stored under StateHeaderKey(shard_J, H) ← wrong shard

5. Assert: get_state_header(shard_J, H).chunk.shard_id() == shard_I  ← invariant violated

6. Call set_state_part(shard_J, H, part_id, part_for_shard_I):
   - state_root = stored_header.chunk.prev_state_root() == shard_I's state root
   - validate_state_part(shard_J, shard_I_state_root, part) → Ok
   - Shard J's trie is overwritten with shard I's state.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)

/// Let B[h] be the block with hash h.
/// Let shard_id be the shard ID of the shard this header is meant for
/// As a shorthand,let B_sync = B[sync_hash], B_prev = B[B_sync.prev_hash]
///
/// Also let B_chunk be the block with height B_prev.chunks[shard_id].height_included
/// that is an ancestor of B_sync. So, the last block with a new chunk before B_sync.
/// And let B_prev_chunk = B[B_chunk.prev_hash]. So, the block before the last block with a new chunk before B_sync.
///
/// Given these definitions, the meaning of fields are explained below.
#[derive(Debug, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize, ProtocolSchema)]
pub struct ShardStateSyncResponseHeaderV2 {
    /// The chunk whose header in included as B_prev.chunks[shard_id]
    /// This chunk will be applied after downloading state
    pub chunk: ShardChunk,
    /// A merkle path for (Self::chunk.hash, Self::chunk.height_included), verifiable
    /// against B_prev.chunk_headers_root
    pub chunk_proof: MerklePath,
    /// This is None if sync_hash is the genesis hash. Otherwise, it's B_prev_chunk.chunks[shard_id]
    pub prev_chunk_header: Option<ShardChunkHeader>,
    /// A merkle path for (Self::prev_chunk_header.hash, Self::prev_chunk_header.height_included), verifiable
    /// against B_prev_chunk.chunk_headers_root
    pub prev_chunk_proof: Option<MerklePath>,
    /// This field contains the incoming receipts for shard_id for B_sync and B_prev_chunk.
    /// So, this field has at most two elements.
    /// These receipts are used to apply `chunk` after downloading state
    pub incoming_receipts_proofs: Vec<ReceiptProofResponse>,
    /// This field contains the info necessary to verify that the receipt proofs in Self::incoming_receipts_proofs
    /// are actually the ones referenced on chain
    ///
    /// The length of this field is the same as the length of Self::incoming_receipts_proofs, and elements
    /// of the two at a given index are taken together for verification. For a given index i,
    /// root_proofs[i] is a vector of the same length as incoming_receipts_proofs[i].1 , which itself is a
    /// vector of receipt proofs for all "from_shard_ids" that sent receipts to shard_id. root_proofs[i][j]
    /// contains a merkle root equal to the prev_outgoing_receipts_root field of the corresponding chunk
    /// included in the block with hash incoming_receipts_proofs[i].0, and a merkle path to verify it against
    /// that block's prev_chunk_outgoing_receipts_root field.
    pub root_proofs: Vec<Vec<RootProof>>,
    /// The state root with hash equal to B_prev.chunks[shard_id].prev_state_root.
    /// That is, the state root node of the trie before applying the chunks in B_prev
    pub state_root_node: StateRootNode,
}
