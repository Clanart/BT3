### Title
Relative Timelock Bypass via Integer Wrapping in `check_time_locks` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
The `check_time_locks` function contains a legacy code path (`nowrap=false`) that uses `wrapping_add` for relative timelock arithmetic. An unprivileged spender can craft a `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` condition with a near-maximum integer value that wraps around to zero, causing the timelock check to always pass and the coin to be spendable immediately regardless of the intended lock duration.

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic modes for relative timelock evaluation:

- `nowrap=true` (post-softfork): uses `saturating_add` — correct behavior
- `nowrap=false` (legacy/pre-softfork): uses `wrapping_add` — vulnerable [1](#0-0) 

For `ASSERT_HEIGHT_RELATIVE`, the check is:

```
prev_transaction_block_height < unspent.confirmed_block_index.wrapping_add(height_relative)
```

If a coin is confirmed at height `H` and the spend asserts `height_relative = u32::MAX - H + 1`, then:

```
H.wrapping_add(u32::MAX - H + 1) = 0
```

The check becomes `prev_height < 0u32`, which is always `false` for any `u32` value, so the check always passes — the timelock is bypassed entirely.

The same applies to `ASSERT_SECONDS_RELATIVE` with `u64` arithmetic: [2](#0-1) 

The test suite explicitly documents and confirms this behavior: [3](#0-2) [4](#0-3) 

The Python-level tests also confirm the bypass: [5](#0-4) 

The `nowrap` flag is exposed directly through the Python binding: [6](#0-5) 

### Impact Explanation

A coin protected by `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a crafted near-maximum value can be spent immediately when the `nowrap=false` path is active. This is a **timelock validation bypass enabling unauthorized spend acceptance** — matching the High impact category. The bypass is deterministic and requires no privileged access.

### Likelihood Explanation

The `nowrap=false` path is the pre-softfork legacy behavior. The `ConsensusConstants` struct exposes `soft_fork8_height: 8_655_000` and `soft_fork9_height: 0xffff_ffff` (placeholder, not yet activated): [7](#0-6) 

If the softfork controlling `nowrap` has not yet activated (or if `nowrap=false` is still passed by the Python full node for any spend path), the bypass is reachable by any unprivileged spender who can craft a spend bundle with a max-value relative condition. The attacker-controlled entry point is the `height_relative` or `seconds_relative` field in a CLVM puzzle output condition.

**Note**: The exact production call site of `check_time_locks` is in the Python `chia-blockchain` layer (not within `chia_rs` itself), so the precise softfork activation height controlling `nowrap` could not be confirmed from this repository alone. If `nowrap=true` is now universally enforced for all new spends, the exploitable surface is limited to historical block re-validation.

### Recommendation

1. Remove the `nowrap=false` / `wrapping_add` branch entirely once the governing softfork has activated on mainnet. The `saturating_add` path is the correct semantic for all relative timelocks.
2. Until removal, add a hard assertion or compile-time gate ensuring `nowrap=true` is always passed for mempool and new block validation.
3. Audit all call sites in `chia-blockchain` to confirm `nowrap=false` is never passed for post-softfork heights.

### Proof of Concept

Craft a spend bundle for a coin confirmed at height `H` with condition:

```
ASSERT_HEIGHT_RELATIVE(u32::MAX - H + 1)
```

When evaluated with `nowrap=false`:

```
H.wrapping_add(u32::MAX - H + 1) = 0
prev_height < 0u32  →  always false  →  check passes
```

The coin is spendable at any block height, bypassing the intended ~4-billion-block timelock. The same construction applies to `ASSERT_SECONDS_RELATIVE` using `u64::MAX - timestamp + 1`. [8](#0-7) [9](#0-8)

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
