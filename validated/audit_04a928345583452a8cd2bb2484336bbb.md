Audit Report

## Title
Router-held native ETH is claimable by any caller via `refundETH` or WETH-input swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` transfers the router's entire native ETH balance to `msg.sender` with no per-user attribution or access control. Separately, `pay()` uses `address(this).balance` — the router's total ETH — to cover any WETH-input swap callback, regardless of which user deposited it. ETH stranded by one user's overpayment in a `payable` swap call is immediately claimable by any subsequent caller, causing direct loss of user principal.

## Finding Description
**ETH stranding:** All four swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are `external payable`. A user calling `exactInputSingle{value: X}(amountIn: Y, tokenIn: WETH)` where `Y < X` causes the swap callback to invoke `pay(WETH, user, pool, Y)`. Inside `pay`, `nativeBalance = address(this).balance` equals `X` (the full `msg.value`), so only `Y` ETH is deposited as WETH; the remaining `X - Y` ETH stays on the router after the function returns. The `receive()` guard (line 32–34) only blocks direct ETH transfers from non-WETH addresses — it does not prevent ETH from arriving via `payable` function calls.

**Theft via `refundETH`:** Any EOA or contract can call `refundETH()` in a subsequent transaction. It reads `address(this).balance` (line 59) and transfers the entire amount to `msg.sender` (line 61) with no check that the caller deposited any of it.

**Theft via free WETH swap:** An attacker calls any WETH-input swap with `msg.value = 0`. Inside `pay()`, `nativeBalance = address(this).balance` equals the victim's stranded ETH. If `nativeBalance >= value`, the router deposits the victim's ETH as WETH (line 76–77) and sends it to the pool — the attacker's own WETH allowance is never touched, giving them a free swap.

**No existing guard is sufficient:** The `receive()` guard does not apply to `payable` function calls. There is no per-transaction ETH budget, no transient storage tracking, and no enforcement that `msg.value == amountIn`.

## Impact Explanation
Direct loss of user principal (native ETH). A user who over-sends ETH in any `payable` swap call without including a `refundETH` step in the same `multicall` permanently loses the excess to the next caller. The loss is bounded only by the overpayment amount; for large swaps with loose `msg.value` estimates this can be substantial. This meets the Sherlock High/Medium threshold for direct loss of user funds by an unprivileged attacker.

## Likelihood Explanation
All swap functions are `payable`, so users and integrators routinely send ETH directly. The `refundETH` step is opt-in and not enforced by the router; users unfamiliar with the Uniswap v3 multicall pattern will omit it. The attack requires no special privilege — any EOA can call `refundETH()` or submit a WETH-input swap in the next block. The router's ETH balance is publicly readable on-chain, making victim detection trivial and the attack repeatable.

## Recommendation
1. **Track per-transaction ETH budget via transient storage.** Store `msg.value` at the start of each top-level call and deduct from it inside `pay`. Revert or auto-refund any unused budget at the end of the call.
2. **Gate `refundETH` to the depositor's tracked budget** rather than `address(this).balance`, so only the caller's own deposited ETH is returned.
3. **Alternatively**, enforce `msg.value == 0` for non-ETH-input paths and require WETH callers to use `multicall{value}` with an explicit `refundETH` step, documented as a hard invariant and enforced on-chain.
