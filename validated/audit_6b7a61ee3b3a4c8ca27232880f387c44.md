Audit Report

## Title
Cross-User ETH Isolation Failure in `PeripheryPayments.pay` — Stranded ETH from One User Subsidises Another's WETH Input — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` reads `address(this).balance` — the router's entire native ETH balance — when settling a WETH-input swap. Because `multicall` is `payable` and `refundETH()` is a separate, optional call, ETH sent by User A in a prior transaction can remain in the router and be silently consumed to cover User B's WETH obligation in a later, independent transaction. User A suffers a direct, permanent loss of ETH with no corresponding output.

## Finding Description
The vulnerable branch in `pay` reads the full contract balance without any per-call or per-user accounting:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

ETH becomes stranded because `multicall` is `payable` and any ETH sent with a call that does not consume it (e.g., a pure ERC-20 swap) remains in the router:

```solidity
// MetricOmmSimpleRouter.sol L39
function multicall(bytes[] calldata data) public payable override returns (bytes[] memory results) {
```

`refundETH()` is a separate, optional call that users must explicitly include — it is not enforced at the end of any swap entry point:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

The `receive()` guard (L32-34) only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on any payable entry point (`multicall`, `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). There is no mechanism that caps `nativeBalance` to the ETH contributed by the current caller.

Exploit path:
1. User A calls `multicall{value: 1 ether}([exactInputSingle({tokenIn: tokenA, tokenOut: tokenB, ...})])` — no `refundETH()` included. The swap settles in ERC-20; 1 ETH remains in the router.
2. User B calls `exactInputSingle({tokenIn: WETH, tokenOut: tokenX, amountIn: 2 ether, ...})`.
3. The pool callback fires → `pay(WETH, userB, pool, 2 ether)`: `nativeBalance = 1 ether` → wraps and transfers 1 ETH as WETH to the pool, then pulls only 1 WETH from User B via `transferFrom`.
4. User A's 1 ETH is gone; User B's WETH cost is halved; the pool receives the correct 2 WETH.

## Impact Explanation
Direct, permanent loss of User A's ETH principal. User A's ETH is wrapped into WETH and transferred to a pool to settle User B's swap obligation; User A receives nothing in return. This is a cross-user fund-theft scenario satisfying the "direct loss of user principal" criterion at Critical/High severity. The exact corrupted value is `address(this).balance` being consumed for a payer who did not contribute it.

## Likelihood Explanation
No privileged access is required. Users routinely omit `refundETH()` from multicalls (e.g., when the swap consumed less than the sent ETH, or when the token was not WETH). Any subsequent WETH-input swap by any user will silently drain the stranded balance. A griever can deliberately exploit this by observing a stranded-ETH transaction in the mempool and immediately following with a WETH swap sized to consume exactly the stranded amount, making the attack repeatable and low-cost.

## Recommendation
Replace `address(this).balance` with only the ETH attributable to the current call. One approach: record `msg.value` in transient storage at each swap entry point and use that stored value as the cap in `pay()`:

```solidity
// In pay(), replace:
uint256 nativeBalance = address(this).balance;
// With:
uint256 nativeBalance = _currentMsgValue(); // stored in transient storage at swap entry
```

Alternatively, auto-refund any remaining ETH at the end of each top-level swap entry point, or enforce that `refundETH()` is always the last call in any `multicall` that sends ETH.

## Proof of Concept
1. Deploy `MetricOmmSimpleRouter` with real WETH, a factory-registered `tokenA/tokenB` pool (non-WETH), and a `WETH/tokenX` pool.
2. User A calls `multicall{value: 1 ether}([exactInputSingle({tokenIn: tokenA, tokenOut: tokenB, amountIn: ..., ...})])` — no `refundETH()`. Assert `address(router).balance == 1 ether`.
3. User B approves 1 WETH to the router, then calls `exactInputSingle({tokenIn: WETH, tokenOut: tokenX, amountIn: 2 ether, ...})`.
4. Pool callback fires → `pay(WETH, userB, pool, 2 ether)`: wraps 1 ETH from router balance, pulls 1 WETH from User B.
5. Assert: `address(router).balance == 0`; User B's WETH decreased by 1 (not 2); User A's ETH balance permanently reduced by 1 ETH with no output received.
