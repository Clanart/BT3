### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not validate that the new class hash supplied via the `replace_class` syscall corresponds to a previously declared contract class. An unprivileged contract deployer can exploit this to permanently brick any contract they control — freezing all user funds held within it — by replacing the contract's class hash with an arbitrary undeclared value.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function unconditionally writes the caller-supplied `class_hash` into the contract state without any on-chain verification that the hash is declared:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The inline `TODO` comment at line 898 explicitly acknowledges the missing check. The `class_hash` field is taken directly from `request.class_hash`, which is fully attacker-controlled contract-supplied syscall data. No constraint is imposed on its value.

After the state is committed with the undeclared class hash, any subsequent call to the bricked contract reaches `execute_entry_point`, which performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the class hash is undeclared, `dict_read` returns 0 (the default for an unwritten key), and `find_element` — a non-optional Cairo search that asserts the element exists — fails to locate a compiled class with hash 0. This makes any block containing a call to the bricked contract **unprovable**. The sequencer is forced to permanently exclude all transactions targeting that contract, making the freeze irreversible.

By contrast, `execute_declare_transaction` correctly enforces `prev_value=0` and `assert_not_zero(compiled_class_hash)` to ensure only valid, non-zero compiled class hashes are registered: [3](#0-2) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

An attacker deploys a contract that:
1. Accepts deposits from other users (e.g., a fake DeFi vault or wallet factory).
2. Contains a hidden or time-locked function that calls `replace_class(1)` — where `1` is an arbitrary undeclared class hash.
3. After accumulating user funds, triggers the `replace_class` call.

Once committed, the contract's class hash in the global state is permanently set to the undeclared value. No withdrawal, transfer, or any other entry point can ever be executed again. All user funds are **permanently frozen** with no recovery path.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack requires no privileged access:
- Any user can deploy a contract on StarkNet.
- The `replace_class` syscall is callable from any contract's own execution context.
- The attacker fully controls the `class_hash` argument passed to the syscall.
- No signature, authority check, or ownership proof is required beyond being the executing contract.

The attack is low-cost, deterministic, and irreversible once executed.

---

### Recommendation

Add a validation step in `execute_replace_class` to verify the new class hash is declared before committing the state update. Specifically, perform a lookup in `contract_class_changes` and assert the result is non-zero:

```cairo
// Verify that the new class hash is declared.
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the protection already present in `execute_declare_transaction` and closes the gap identified by the existing TODO comment.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract with a `deposit()` function and a hidden `brick()` function that calls `replace_class(1)`.
2. Users call `deposit()`, transferring funds into `MaliciousVault`.
3. Attacker calls `brick()`. The OS executes `execute_replace_class` with `class_hash = 1` (undeclared). No validation occurs. The state is updated.
4. The block is proven successfully (the `replace_class` call itself succeeds).
5. In any subsequent block, any transaction calling `MaliciousVault` causes `execute_entry_point` to call `dict_read` on class hash `1`, returning compiled class hash `0`, then `find_element` fails — the block is unprovable.
6. The sequencer permanently excludes all calls to `MaliciousVault`. All deposited funds are frozen forever.
