### Title
Missing Zero-Address Validation for Transfer Recipient Causes Permanent Fund Lock — (`near/omni-bridge/src/lib.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
The NEAR bridge's `init_transfer` function accepts any `OmniAddress` as a recipient, including zero addresses (e.g., `OmniAddress::Eth(H160::ZERO)`), without validation. When a user initiates a transfer to a zero EVM address, tokens are locked or burned on NEAR, but the corresponding EVM `finTransfer` either permanently reverts (for bridge tokens, because OpenZeppelin's `_mint(address(0))` reverts) or silently destroys ETH (for native ETH transfers). The nonce is never consumed in the revert case, making the NEAR-side lock irrecoverable.

### Finding Description

**Root cause — NEAR `init_transfer` (near/omni-bridge/src/lib.rs):**

The only recipient validation is a chain-kind check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

`OmniAddress::Eth(H160::ZERO)` passes this check because its chain kind is `Eth`, not `Near`. The `OmniAddress::is_zero()` helper exists in the type library but is never called here: [2](#0-1) 

**Propagation — EVM `finTransfer` (evm/src/omni-bridge/contracts/OmniBridge.sol):**

The MPC signs the transfer message including the zero recipient. The relayer then calls `finTransfer` with `payload.recipient = address(0)`. No zero-address guard exists before the transfer dispatch:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(
        payload.recipient,   // address(0)
        payload.amount
    );
``` [3](#0-2) 

`BridgeToken.mint` calls OpenZeppelin `_mint(address(0), amount)`, which reverts unconditionally per the ERC-20 standard. The nonce is marked used *before* the transfer: [4](#0-3) 

Because the entire transaction reverts, `completedTransfers[payload.destinationNonce]` is also rolled back. The EVM finalization will always revert, the nonce is never consumed, and the NEAR-side lock is permanent.

For the native-ETH path (`payload.tokenAddress == address(0)`), `address(0).call{value: amount}("")` succeeds silently, burning the ETH with no revert and consuming the nonce: [5](#0-4) 

**StarkNet `fin_transfer` (starknet/src/omni_bridge.cairo):**

The same pattern exists — no zero-address check on `payload.recipient` before minting or transferring: [6](#0-5) 

### Impact Explanation

| Scenario | NEAR-side | Destination-side | Recoverability |
|---|---|---|---|
| Bridge token → EVM address(0) | Tokens locked/burned | `finTransfer` always reverts; nonce never consumed | **Irrecoverable** |
| Native ETH → EVM address(0) | NEAR tokens locked | ETH sent to address(0), nonce consumed | **Irrecoverable** |
| Bridge token → StarkNet 0x0 | Tokens locked/burned | Behavior token-dependent; likely revert or burn | **Irrecoverable** |

There is no timeout, refund, or admin-recovery path in the NEAR bridge for a transfer whose destination-chain finalization permanently fails.

This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

A user can trigger this by:
1. Accidentally copy-pasting an uninitialized address (all-zeros) as the EVM recipient.
2. Programmatically constructing an `InitTransferMsg` with `recipient = OmniAddress::Eth(H160::ZERO)` — a valid Rust value that the type system permits.

The `OmniAddress::new_zero` constructor is used in test helpers within the codebase itself, confirming zero addresses are constructible and passable: [7](#0-6) 

No privileged role is required. Any token holder can trigger this via the public `ft_transfer_call` → `init_transfer` path.

### Recommendation

1. **In NEAR `init_transfer`** — add a zero-address guard using the existing `is_zero()` method:
   ```rust
   require!(
       !init_transfer_msg.recipient.is_zero(),
       BridgeError::InvalidRecipient.as_ref()
   );
   ``` [1](#0-0) 

2. **In EVM `finTransfer`** — add a defense-in-depth check:
   ```solidity
   require(payload.recipient != address(0), "InvalidRecipient");
   ``` [8](#0-7) 

3. **In StarkNet `fin_transfer`** — add:
   ```cairo
   assert(!payload.recipient.is_zero(), 'ERR_ZERO_RECIPIENT');
   ``` [9](#0-8)
