### Title
Missing Zero-Value Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts `class_hash = 0` from an unprivileged contract without any validation. Because `UNINITIALIZED_CLASS_HASH = 0` is the sentinel value for a non-deployed contract, setting a live contract's class hash to zero permanently breaks all future entry-point dispatch for that contract, freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the caller-supplied `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without checking whether it is zero:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;          // line 896 — no zero check

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash,                    // zero is accepted here
        storage_ptr=state_entry.storage_ptr,
        nonce=state_entry.nonce,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The sentinel value `UNINITIALIZED_CLASS_HASH = 0` is defined in `commitment.cairo`:

```cairo
const UNINITIALIZED_CLASS_HASH = 0;
``` [2](#0-1) 

This same constant is used in `deploy_contract.cairo` to assert that a deployment target is uninitialized before writing to it:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
``` [3](#0-2) 

After `replace_class(0)` succeeds, any subsequent call to the affected contract reaches `execute_entry_point`, which performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash          // key = 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,                  // compiled_class_hash = 0
);
``` [4](#0-3) 

`find_element` is a hard-failing Cairo primitive: if no compiled class with hash `0` exists in the bundle (none ever will, because `0` is the uninitialized sentinel), the OS proof generation aborts. The contract becomes permanently uncallable.

The TODO comment at line 898 of `syscall_impls.cairo` explicitly acknowledges the missing check:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
``` [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds token balances or other assets in its storage and calls `replace_class(0)` (whether maliciously or due to a bug) will have its `StateEntry.class_hash` committed as `0` on-chain. Because `0` maps to no valid compiled class, every subsequent call to that contract — including withdrawal or transfer entry points — will fail at the OS dispatch level. The storage (and any funds it represents) is permanently inaccessible. This matches the "permanent freezing of funds" impact category.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is callable by any deployed contract without privilege. A malicious contract author can deliberately invoke `replace_class(0)` to grief users who have deposited funds into the contract (e.g., a vault or escrow). A buggy contract could also trigger this accidentally. No operator or trusted-role access is required; the attacker only needs to be a contract deployer.

---

### Recommendation

Add an `assert_not_zero` guard on `class_hash` inside `execute_replace_class`, immediately after reading the request field, before any state mutation:

```cairo
let class_hash = request.class_hash;
with_attr error_message("replace_class: class_hash must not be zero.") {
    assert_not_zero(class_hash);
}
```

This mirrors the pattern already used in `deploy_contract.cairo` (line 45) where `assert_not_zero` is applied to the product of reserved-address differences before any state write. Additionally, the existing TODO should be resolved to also verify that `class_hash` corresponds to a previously declared class. [6](#0-5) 

---

### Proof of Concept

1. **Attacker deploys a malicious contract** whose body contains a call to the `replace_class` syscall with `class_hash = 0`.
2. **Attacker (or victim) deposits funds** into the contract (e.g., via an ERC-20 transfer to the contract's address).
3. **Attacker invokes the malicious entry point.** The OS executes `execute_replace_class` with `request.class_hash = 0`. No zero-check exists; `dict_update` commits `StateEntry { class_hash: 0, ... }` for the contract address.
4. **Any subsequent call to the contract** (e.g., a withdrawal) reaches `execute_entry_point`. `dict_read(key=0)` on `contract_class_changes` returns `0`. `find_element(..., key=0)` finds no matching `CompiledClassFact` and the OS aborts proof generation for that call.
5. **Result:** The contract is permanently uncallable at the OS level. All funds stored in its storage slots are frozen with no recovery path.

    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;
