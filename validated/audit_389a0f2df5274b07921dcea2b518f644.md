The `verify_path` implementation: [3](#0-2) 

`verify_path` only checks that `compute_root_from_path(path, hash(item)) == root`. It does **not** check the leaf's position/index in the tree. The `MerklePath` encodes sibling hashes and left/right directions, but the function does not enforce that the proof corresponds to a specific position (e.g., shard 0's slot).

**Concrete example with 2 shards:**

The `chunk_headers_root` is `combine_hash(hash(ChunkHashHeight(chunk0)), hash(ChunkHashHeight(chunk1)))`.

- Proof for shard 0 (position 0): `[{hash: hash(chunk1), direction: Right}]`
- Proof for shard 1 (position 1): `[{hash: hash(chunk0), direction: Left}]`

If an attacker provides `chunk1` with its position-1 proof `[{hash: hash(chunk0), direction: Left}]`, `verify_path` computes `combine_hash(hash(chunk0), hash(chunk1)) == root` — **this passes**, even though the requested `shard_id` is 0.

### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows cross-shard state root substitution via position-agnostic Merkle proof — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to a different shard than the requested `shard_id`. The Merkle proof check (`verify_path` against `chunk_headers_root`) only verifies set-membership, not leaf position, so a chunk from shard 1 passes the proof check for a shard-0 request. The header is then stored under `StateHeaderKey(requested_shard_id=0, sync_hash)` carrying shard 1's `prev_state_root`, corrupting state finalization for shard 0.

### Finding Description
In `set_state_header` (`chain/chain/src/state_sync/adapter.rs:368`), the function receives a `shard_id` parameter and a `ShardStateSyncResponseHeader`. It extracts the embedded chunk and runs five validation steps, but **none of them check `chunk.shard_id() == shard_id`**:

1. `validate_chunk_proofs` checks internal chunk consistency only.
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` checks that the chunk hash appears somewhere in the block's chunk Merkle tree, but `verify_path` is position-agnostic — it does not enforce that the proof corresponds to the shard-0 leaf slot.
3. Receipt proof validation uses the `shard_id` parameter (0), not `chunk.shard_id()`, so valid shard-0 receipts from the real chain satisfy it.
4. `validate_state_root_node` validates the attacker-supplied node against `chunk_inner.prev_state_root()` (shard 1's root), which the attacker controls.

After all checks pass, the header is stored under `StateHeaderKey(shard_id=0, sync_hash)` with shard 1's `prev_state_root`.

### Impact Explanation
A node performing state sync for shard 0 stores a header whose `prev_state_root` belongs to shard 1. `set_state_finalize` then calls `apply_chunk` with `shard_uid` for shard 0 but with shard 1's state root, causing the node to reconstruct and operate on the wrong state trie for shard 0. This violates the commitment invariant that `StateHeaderKey(shard_id, sync_hash)` maps to a header whose chunk's `shard_id` equals the key's shard. The corrupted node will diverge from the canonical chain.

### Likelihood Explanation
Any peer on the NEAR network can serve state sync responses — this is an unprivileged role. A malicious peer that responds to a state sync request before legitimate peers can inject the malicious header. The attack requires no validator or admin privileges, only the ability to act as a state-sync peer and knowledge of the target block's chunk Merkle tree (which is public on-chain data).

### Recommendation
Add an explicit shard ID check immediately after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be inserted before any other validation at line 376 of `chain/chain/src/state_sync/adapter.rs`.
