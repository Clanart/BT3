### Title
Wrapping Arithmetic Timelock Bypass via Unbounded `height_relative`/`seconds_relative` Condition Values — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` uses `wrapping_add` (when `nowrap=false`) for relative timelock enforcement. Because `sanitize_uint` imposes no upper cap below `u32::MAX` for `height_relative` or `u64::MAX` for `seconds_relative`, an unprivileged spender can embed a maximum-value relative timelock condition in a CLVM puzzle. The wrapping addition collapses the required future height/timestamp to a value already in the past, making the timelock trivially satisfied and allowing immediate coin spending.

---

### Finding Description

In `crates/chia-consensus/src/check_time_locks.rs`, the `check_time_locks` function enforces `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` conditions using two arithmetic modes controlled by the `nowrap: bool` parameter: [1](#0-0) 

When `nowrap=false` (the pre-soft-fork consensus path), the check is:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [2](#0-1) 

The condition parsing in `crates/chia-consensus/src/conditions.rs` accepts `height_relative` up to `u32::MAX` (4-byte `sanitize_uint`) and `seconds_relative` up to `u64::MAX` (8-byte `sanitize_uint`) with no tighter upper bound: [3](#0-2) [4](#0-3) 

The `sanitize_uint` function itself only rejects values that exceed the byte-width limit; it does not enforce any domain-specific upper cap: [5](#0-4) 

**Concrete arithmetic exploit:**

| Variable | Value |
|---|---|
| `confirmed_block_index` | `100` |
| `height_relative` | `0xFFFF_FFFF` (u32::MAX) |
| `wrapping_add` result | `100u32.wrapping_add(0xFFFF_FFFF) = 99` |
| `prev_transaction_block_height` | `200` |
| Check: `200 < 99` | **false → no error → spend accepted** |

The test suite explicitly documents and confirms this behavior: [6](#0-5) [7](#0-6) 

The same wrapping bypass applies to `ASSERT_SECONDS_RELATIVE` via `wrapping_add` on `u64`: [8](#0-7) 

A symmetric but opposite flaw exists for `ASSERT_BEFORE_HEIGHT_RELATIVE(u32::MAX)` with `nowrap=false`: the wrapping causes the deadline to appear already passed, **permanently bricking** a coin that should remain spendable for ~4 billion blocks: [9](#0-8) 

---

### Impact Explanation

**High.** Timelock condition bypass enables unauthorized spend acceptance.

Any CLVM puzzle that relies solely on `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` for access control (e.g., time-locked payments, vesting contracts, or any puzzle spendable by anyone after a deadline) can be spent immediately when `nowrap=false` is active and the condition argument is set to `u32::MAX` / `u64::MAX`. The wrapping collapses the required future block height or timestamp to a value already satisfied, so the consensus layer accepts the spend without waiting the intended duration. This matches the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

The inverse flaw (`ASSERT_BEFORE_HEIGHT_RELATIVE(u32::MAX)` with `nowrap=false`) permanently bricks coins, matching the external report's "important protocol functionality can be bricked" impact.

---

### Likelihood Explanation

**Low–Medium.** The `nowrap=false` path is the pre-soft-fork consensus behavior. While the Python full node controls the `nowrap` flag, any block validated before the `nowrap=true` soft fork activates uses wrapping arithmetic. An attacker who knows the wrapping behavior can craft a CLVM puzzle outputting `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFFF)` — this is entirely unprivileged input requiring no special role, key, or network position. The attacker only needs to submit a valid spend bundle containing such a puzzle.

---

### Recommendation

1. **Add an upper cap in condition parsing.** In `parse_args` (`conditions.rs`), after `sanitize_uint` returns `SanitizedUint::Ok(r)` for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE`, reject values above a safe maximum (e.g., `u32::MAX / 2` for heights, a reasonable epoch ceiling for seconds).
2. **Unconditionally use `saturating_add`.** Remove the `nowrap` branch and always use `saturating_add` in `check_time_locks`. The saturating path correctly rejects `u32::MAX` height-relative conditions as unsatisfiable.
3. **Enforce the cap at the `sanitize_uint` call site** for these specific conditions rather than relying on the arithmetic mode flag.

---

### Proof of Concept

The existing test in `check_time_locks.rs` is itself the proof of concept — it is a passing test that confirms the bypass:

```rust
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)), // nowrap=true
    Ok(()),  // nowrap=false: TIMELOCK BYPASSED
)]
``` [6](#0-5) 

A spend bundle containing a puzzle that outputs `(ASSERT_HEIGHT_RELATIVE . 0xFFFFFFFF)` submitted to a node running with `nowrap=false` will pass `check_time_locks` at any block height ≥ `confirmed_block_index − 1`, regardless of how recently the coin was created.

```rust
pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    // strip the leading zero byte if there is one
    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }

    Ok(SanitizedUint::Ok(u64_from_bytes(buf)))
