### Title
Unchecked `ft_transfer` Failure in `send_fee_internal` Causes Permanent Fee Token Lock and `locked_tokens` Accounting Corruption - (File: `near/omni-bridge/src/lib.rs`)

### Summary
In `send_fee_internal`, `locked_tokens` is decremented and the pending transfer message is removed **before** confirming that the `ft_transfer` to the fee recipient succeeds. Because no `.then()` callback is attached to the returned promise, a failed `ft_transfer` leaves the bridge in a permanently corrupted state: the fee tokens are irrecoverably stuck in the bridge contract and the `locked_tokens` collateral accounting is permanently understated.

### Finding Description

`send_fee_internal` is the terminal step of the `claim_fee` flow. Its call chain is:

```
claim_fee  →  claim_fee_callback  →  send_fee_internal
```

Inside `claim_fee_callback`, two irreversible state mutations happen **before** the token transfer is attempted:

1. **Line 1094** – `remove_transfer_message` permanently removes the pending transfer entry from `pending_transfers`.
2. **Line 2684** – `unlock_tokens_if_needed` permanently decrements `locked_tokens` for the destination chain.

Then, for non-deployed (natively locked) tokens, `send_fee_internal` issues an `ft_transfer` call and returns it as a bare `PromiseOrValue` with **no `.then()` error-handling callback**:

```rust
// near/omni-bridge/src/lib.rs  lines 2693-2697
ext_token::ext(token)
    .with_static_gas(FT_TRANSFER_GAS)
    .with_attached_deposit(ONE_YOCTO)
    .ft_transfer(fee_recipient, U128(token_fee), None)
    .into()
```

In NEAR's async execution model, state changes committed inside a callback receipt are **not rolled back** if a subsequent promise returned from that callback fails. Therefore, if `ft_transfer` panics or returns an error (e.g., recipient lacks a storage deposit for the token), the two state mutations above remain committed permanently.

### Impact Explanation

Two simultaneous impacts occur:

1. **Permanent freezing of fee tokens.** The fee amount (up to the full transfer amount minus the recipient's share) remains in the bridge contract with no recovery path. The `pending_transfers` entry has been deleted, so `claim_fee` cannot be retried. Only a DAO-privileged `transfer_token_as_dao` call could rescue the tokens, but this requires off-chain detection and manual intervention.

2. **`locked_tokens` accounting corruption.** `locked_tokens[(destination_chain, token_id)]` is decremented by `token_fee` even though no tokens left the bridge. This understates the bridge's locked collateral for that chain, breaking the invariant that `locked_tokens` equals the sum of all in-flight transfer amounts for that chain. Subsequent `unlock_tokens` calls that rely on this counter can underflow or produce incorrect results, potentially allowing more tokens to be released than are actually backed.

This matches the allowed impact classes: **permanent freezing of user/protocol funds** and **accounting corruption that breaks bridge collateralization**.

### Likelihood Explanation

`ft_transfer` on a NEP-141 token fails whenever the recipient account has no storage deposit registered with that token contract. A relayer calling `claim_fee` for a token they have never interacted with, or for a newly deployed token, will not have a storage deposit. This is a realistic, unprivileged-user-reachable condition: any relayer who submits a valid `claim_fee` proof but lacks a storage deposit for the specific token triggers the bug. No special access or colluding parties are required.

### Recommendation

Attach a `.then()` callback after the `ft_transfer` (and `mint`) calls in `send_fee_internal` that checks the promise result. If the transfer failed, the callback should:
- Re-insert the transfer message into `pending_transfers` (or store a dedicated "claimable fee" record), and
- Re-increment `locked_tokens` by `token_fee` to restore the correct accounting.

Alternatively, verify that the fee recipient has a sufficient storage deposit **before** calling `unlock_tokens_if_needed` and `remove_transfer_message`, so the function can revert cleanly without committing partial state.

### Proof of Concept

1. Relayer R submits a valid `claim_fee` proof for a non-deployed token T bridged from Ethereum. R has never registered a storage deposit with token T's contract.
2. `claim_fee_callback` executes:
   - `remove_transfer_message(transfer_id)` → pending transfer deleted. [1](#0-0) 
   - `send_fee_internal(...)` is called. [2](#0-1) 
3. Inside `send_fee_internal`:
   - `unlock_tokens_if_needed(destination_chain, &token, token_fee)` → `locked_tokens` decremented. [3](#0-2) 
   - `ft_transfer(fee_recipient, U128(token_fee), None)` is dispatched and returned as a bare promise. [4](#0-3) 
4. The `ft_transfer` receipt panics: R has no storage deposit → NEP-141 contract rejects the transfer.
5. NEAR runtime commits the state from step 3 (both mutations) regardless of the `ft_transfer` failure. No `.then()` callback exists to revert them. [5](#0-4) 
6. Result: `locked_tokens` is permanently understated by `token_fee`; `token_fee` tokens are permanently stuck in the bridge contract; the transfer message is gone with no retry path. [6](#0-5)

```rust
    fn send_fee_internal(
        &mut self,
        transfer_message: &TransferMessage,
        fee_recipient: AccountId,
        token_fee: u128,
    ) -> PromiseOrValue<()> {
        if transfer_message.fee.native_fee.0 != 0 {
            let origin_chain = transfer_message.origin_transfer_id.as_ref().map_or_else(
                || transfer_message.get_origin_chain(),
                |origin_transfer_id| origin_transfer_id.origin_chain,
            );

            if origin_chain.is_utxo_chain() {
                env::panic_str(BridgeError::NativeFeeForUtxoChain.to_string().as_str())
            } else if origin_chain == ChainKind::Near {
                Promise::new(fee_recipient.clone())
                    .transfer(NearToken::from_yoctonear(transfer_message.fee.native_fee.0))
                    .detach();
            } else {
                ext_token::ext(self.get_native_token_id(origin_chain))
                    .with_static_gas(MINT_TOKEN_GAS)
                    .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
                    .detach();
            }
        }

        let token = self.get_token_id(&transfer_message.token);
        env::log_str(
            &OmniBridgeEvent::ClaimFeeEvent {
                transfer_message: transfer_message.clone(),
            }
            .to_log_string(),
        );

        self.unlock_tokens_if_needed(transfer_message.get_destination_chain(), &token, token_fee);

        if token_fee > 0 {
            if self.is_deployed_token(&token) {
                ext_token::ext(token)
                    .with_static_gas(MINT_TOKEN_GAS)
                    .mint(fee_recipient, U128(token_fee), None)
                    .into()
            } else {
                ext_token::ext(token)
                    .with_static_gas(FT_TRANSFER_GAS)
                    .with_attached_deposit(ONE_YOCTO)
                    .ft_transfer(fee_recipient, U128(token_fee), None)
                    .into()
            }
        } else {
            PromiseOrValue::Value(())
        }
    }
```
