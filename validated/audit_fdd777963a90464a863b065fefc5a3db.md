Audit Report

## Title
Excess ETH from exact-output swaps is permanently trapped and drainable by any caller via `refundETH()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`exactOutputSingle` and `exactOutput` are `payable` but never auto-refund surplus ETH. Because the exact input amount for an exact-output swap is only known after pool execution, callers must send an upper-bound `msg.value`; the unused portion remains in the router. `refundETH()` has no per-depositor accounting and unconditionally sends `address(this).balance` to `msg.sender`, allowing any third party to drain all trapped ETH.

## Finding Description

**`pay()` wraps only the exact amount needed, leaving surplus in the contract.**
When `token == WETH` and `nativeBalance >= value`, the function wraps exactly `value` and transfers it to the pool. Any ETH above `value` that arrived with the transaction silently remains in the router with no ownership record. [1](#0-0) 

**`refundETH()` sends the entire contract balance to whoever calls it.**
There is no per-depositor accounting. Any EOA or contract can call this function at any time and receive all ETH currently held by the router. [2](#0-1) 

**`exactOutputSingle` and `exactOutput` are `payable` and perform no post-swap ETH refund.**
Both functions return after `_clearExpectedCallbackPool()` without checking or returning `address(this).balance - amountIn`. The actual `amountIn` is only determined inside the pool callback, so callers must send an upper-bound value; the surplus is never returned. [3](#0-2) [4](#0-3) 

**The `receive()` guard does not protect against ETH entering via `payable` function calls.**
`receive()` fires only when ETH is sent with empty calldata. ETH sent alongside any `payable` function call is accepted by the EVM without invoking `receive()`, so the `msg.sender != WETH` check is irrelevant for this attack path. [5](#0-4) 

**Secondary vector — partial-balance branch silently consumes prior users' trapped ETH.**
When `0 < nativeBalance < value`, `pay()` wraps ALL existing native balance (which may belong to a prior depositor) and pulls only the remainder from the current payer via `safeTransferFrom`. This transfers a prior user's trapped ETH to the pool on behalf of the current user, reducing the current user's cost at the prior user's expense. [6](#0-5) 

## Impact Explanation
Direct loss of user ETH principal. A user calling `exactOutputSingle` or `exactOutput` with `tokenIn = WETH` must send an upper-bound `msg.value` because the precise input is unknown until pool execution. The surplus is permanently trapped in the router and can be stolen by any address calling `refundETH()` in a subsequent transaction, or silently consumed by the next user's swap via the partial-balance branch. This meets the Sherlock critical/high threshold for direct loss of user principal with no recovery path.

## Likelihood Explanation
High. Exact-output swaps are a core advertised feature. Sending excess ETH is not a user error — it is the only correct way to call these functions with native ETH when the precise input is unknown. The exploit requires no special privileges: any EOA can call `refundETH()` at any time. A MEV bot monitoring the mempool can back-run the victim's transaction in the same block to claim the surplus before the victim can call `refundETH()` themselves.

## Recommendation
1. **Auto-refund at the end of each payable swap function.** After `_clearExpectedCallbackPool()`, add:
   ```solidity
   if (address(this).balance > 0) _transferETH(msg.sender, address(this).balance);
   ```
2. **Alternatively, track per-caller ETH deposits in transient storage** and restrict `refundETH()` to return only the caller's own deposit, preventing cross-user drainage.
3. **Do not use `address(this).balance` as a shared pool in `pay()`.** Pass the exact ETH amount to use as a parameter sourced from the transient context set at entry, so no prior user's ETH can be consumed.

## Proof of Concept

**Theft via `refundETH()`:**
1. Alice calls `exactOutputSingle({tokenIn: WETH, amountOut: 100e6 USDC, amountInMaximum: 1e18})` with `msg.value = 1e18`.
2. Pool executes; callback calls `pay(WETH, Alice, pool, 0.8e18)`.
3. `pay()` sees `nativeBalance = 1e18 >= 0.8e18`; wraps 0.8 ETH, sends to pool. 0.2 ETH remains in router.
4. `exactOutputSingle` returns. No refund is issued.
5. Bob calls `refundETH()`. Receives `address(this).balance = 0.2 ETH`. Alice loses 0.2 ETH with no recourse.

**Theft via subsequent swap (partial-balance branch):**
1. Same setup: 0.2 ETH trapped in router after Alice's swap.
2. Bob calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18})` with `msg.value = 0`.
3. Callback calls `pay(WETH, Bob, pool, 1e18)`.
4. `pay()` sees `nativeBalance = 0.2e18 > 0` but `< 1e18`; wraps Alice's 0.2 ETH, sends to pool, then pulls only 0.8 WETH from Bob via `safeTransferFrom`.
5. Bob receives a 0.2 ETH discount funded entirely by Alice's trapped ETH.
