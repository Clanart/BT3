### Title
Missing Normalized-Amount Validation at Transfer Initiation and Fee Update Allows Permanent Fund Lock — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `sign_transfer` function enforces `normalize_amount(amount - fee, decimals) > 0`, but neither the transfer initiation path (`ft_on_transfer` → `init_transfer_internal`) nor `update_transfer_fee` validates this constraint. For tokens whose on-chain decimal precision differs between NEAR and the destination chain, a user can create a transfer where `sign_transfer` permanently reverts with `InvalidAmountToTransfer`, irrecoverably locking their tokens in the bridge with no cancellation path.

---

### Finding Description

The bridge normalizes token amounts to account for decimal differences between chains. For example, a token with 24 decimals on NEAR and 18 decimals on EVM has `diff_decimals = 6`, so amounts are divided by `10^6` before being signed for EVM delivery.

`sign_transfer` computes the normalized amount and hard-reverts if it is zero:

```rust
// near/omni-bridge/src/lib.rs:475-485
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` uses floor division:

```rust
// near/omni-bridge/src/lib.rs:2784-2787
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

If `amount - fee < 10^diff_decimals`, the result is `0` and `sign_transfer` always panics.

However, the only fee constraint enforced at initiation and at fee-update time is `fee < amount`:

```rust
// near/omni-bridge/src/lib.rs:399-402  (update_transfer_fee)
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

This allows `fee = amount - 1`, leaving `amount - fee = 1`, which normalizes to `0` for any token with `diff_decimals ≥ 1`.

---

### Impact Explanation

Once a transfer is initiated, the user's tokens are locked or burned inside the bridge. The only forward path is `sign_transfer`. If `sign_transfer` always reverts, the transfer is permanently stuck. There is no `cancel_transfer` or refund function for outbound NEAR-origin transfers. The user's funds are irrecoverably locked.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows.**

---

### Likelihood Explanation

Any user bridging a token with `origin_decimals > decimals` (e.g., a 24-decimal NEAR token to an 18-decimal EVM token, `diff_decimals = 6`) who sets a fee within `10^diff_decimals` of the transfer amount will trigger this. This can happen:

1. **At initiation**: user sets `amount = 10^diff_decimals + 1`, `fee = 10^diff_decimals` — passes `fee < amount`, but `normalize_amount(1) = 0`.
2. **Via `update_transfer_fee`**: user initiates with a valid fee, then raises it to `amount - 1` — passes `fee < amount`, but `normalize_amount(1) = 0`.

The sender is the only party who can raise the token fee (enforced by `SenderCanUpdateTokenFeeOnly`), so this is self-inflicted, but the contract provides no guard against it and no recovery path.

---

### Recommendation

Add a normalized-amount check at both transfer initiation and fee update, not only at signing time. At the point where `decimals` are available (they are stored in `self.token_decimals`), validate:

```rust
require!(
    Self::normalize_amount(amount.0 - fee.fee.0, decimals) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This should be enforced in `init_transfer_internal` (or the `ft_on_transfer` handler) and in `update_transfer_fee`, mirroring the guard already present in `sign_transfer`.

---

### Proof of Concept

**Setup:** Token registered with `Decimals { origin_decimals: 24, decimals: 18 }` (diff = 6, divisor = 1,000,000).

**Step 1 — Initiate transfer:**
```
amount = 1_000_001   (NEAR token units)
fee    = 1_000_000
```
Check `fee < amount` → `1_000_000 < 1_000_001` → **passes**.
Tokens are locked/burned.

**Step 2 — Relayer calls `sign_transfer`:**
```rust
amount_without_fee() = 1_000_001 - 1_000_000 = 1
normalize_amount(1, diff=6) = 1 / 1_000_000 = 0   // floor division
require!(0 > 0, ...)  // PANICS: InvalidAmountToTransfer
```

**Step 3 — No recovery:**
`sign_transfer` reverts on every call. No cancel/refund function exists for outbound transfers. The 1,000,001 tokens are permanently locked in the bridge.

**Alternatively via `update_transfer_fee`:**
```
Initial: amount = 2_000_000, fee = 0  → normalize(2_000_000) = 2  ✓
Update:  fee = 1_999_999              → passes fee < amount check
         normalize(1) = 0            → sign_transfer always reverts
```

    pub fn update_transfer_fee(&mut self, transfer_id: TransferId, fee: UpdateFee) {
        match fee {
            UpdateFee::Fee(fee) => {
                let mut transfer = self.get_transfer_message_storage(transfer_id);

                require!(
                    transfer.message.origin_transfer_id.is_none(),
                    BridgeError::UpdateFeeNotAllowedForTransfer.as_ref()
                );

                let current_fee = transfer.message.fee;
                require!(
                    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
                    BridgeError::InvalidFee.as_ref()
                );

                require!(
                    fee.fee == current_fee.fee
                        || OmniAddress::Near(env::predecessor_account_id())
                            == transfer.message.sender,
                    BridgeError::SenderCanUpdateTokenFeeOnly.as_ref()
                );

                let diff_native_fee = fee
                    .native_fee
                    .0
                    .checked_sub(current_fee.native_fee.0)
                    .near_expect(BridgeError::LowerFee);

                require!(
                    NearToken::from_yoctonear(diff_native_fee) == env::attached_deposit(),
                    BridgeError::InvalidAttachedDeposit.as_ref()
                );

                transfer.message.fee = fee;
                self.insert_raw_transfer(transfer.message.clone(), transfer.owner);

                env::log_str(
                    &OmniBridgeEvent::UpdateFeeEvent {
                        transfer_message: transfer.message,
                    }
                    .to_log_string(),
                );
            }
            UpdateFee::Proof(_) => {
                env::panic_str(BridgeError::UnsupportedFeeUpdateProof.to_string().as_str())
            }
        }
    }
