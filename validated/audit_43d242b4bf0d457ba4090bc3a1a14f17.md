### Title
Wrong-Epoch Shard Layout Used to Index Previous-Epoch Chunks in `get_state_response_part` — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` resolves the `state_root` for a requested shard by indexing into the **previous epoch's** block chunks using a `shard_index` derived from the **new epoch's** shard layout. During a resharding event the two layouts differ, so the index is wrong: for child shards whose new-layout index exceeds the old chunk count the function returns `Error::InvalidShardId`, and for child shards whose new-layout index coincidentally falls within bounds it silently reads the wrong chunk's `prev_state_root`. The companion function `compute_state_response_header` performs the identical lookup correctly by first obtaining `prev_shard_layout` and using that to compute the index.

---

### Finding Description

`get_state_response_part` builds the state root it will use to generate a state part as follows:

```rust
// chain/chain/src/state_sync/adapter.rs  lines 295, 305, 310-316
let epoch_id = block.header().epoch_id();          // ← NEW epoch
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;  // ← NEW layout

let prev_block = self.chain_store.get_block(header.prev_hash())?;   // ← PREV epoch's last block
let shard_index = shard_layout.get_shard_index(shard_id)?;          // ← index in NEW layout
let state_root = prev_block
    .chunks()
    .get(shard_index)                              // ← indexes PREV epoch's chunks with NEW index
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

`prev_block` is the last block of the **previous** epoch; its chunks are ordered by the **previous** epoch's shard layout. Using the new epoch's `shard_index` to address those chunks is the wrong context.

`compute_state_response_header` performs the same lookup correctly:

```rust
// chain/chain/src/state_sync/adapter.rs  lines 93-101
let shard_layout     = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;  // new
let prev_epoch_id    = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;      // prev
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;               // ← correct
let chunk_header     = chunks.get(prev_shard_index)...;
``` [2](#0-1) 

During a resharding epoch the two layouts diverge. Concretely, if the old epoch had shards `[0, 1]` (indices 0, 1) and the new epoch has shards `[0, 2, 3]` (indices 0, 1, 2) after splitting shard 1:

| Requested `shard_id` | `shard_layout.get_shard_index` (new) | `prev_block.chunks().get(…)` | Result |
|---|---|---|---|
| 2 (left child) | 1 | chunk for old shard 1 | coincidentally correct |
| 3 (right child) | 2 | `None` (only 2 old chunks) | `Error::InvalidShardId` |

The `obtain_state_part` call that follows passes `prev_prev_hash` (the grandparent of the sync block) as the `prev_hash` context to the runtime:

```rust
// lines 328-336
let state_part = self.runtime_adapter.obtain_state_part(
    shard_id,
    &prev_prev_hash,   // grandparent of sync block
    &state_root,       // ← derived from wrong chunk above
    PartId::new(part_id, num_parts),
)
``` [3](#0-2) 

Inside `obtain_state_part_impl` the runtime resolves `shard_uid` from `prev_hash` via `get_epoch_id_from_prev_block`:

```rust
// chain/chain/src/runtime/mod.rs  lines 506-510
let epoch_id = self.epoch_manager.get_epoch_id_from_prev_block(prev_hash)?;
let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, &epoch_id)?;
let trie_with_state =
    self.tries.get_trie_with_block_hash_for_shard(shard_uid, *state_root, &prev_hash, true);
``` [4](#0-3) 

When `state_root` is wrong (from the wrong chunk) the trie lookup either fails or returns trie nodes that do not match the state root committed in the `ShardStateSyncResponseHeader`, so the receiving node's `set_state_part` validation rejects every part.

---

### Impact Explanation

During any resharding epoch, nodes serving state parts via `get_state_response_part` will:

1. Return `Error::InvalidShardId` for every child shard whose new-layout index exceeds the old chunk count (all "right" children), making those parts permanently unservable.
2. Silently serve parts built from the wrong `state_root` for child shards whose index coincidentally falls in bounds, causing the receiving node's `validate_state_part` check to fail.

Both outcomes prevent state sync from completing for the affected shards. Validators rotating onto those shards cannot catch up, and fresh nodes bootstrapping during a resharding epoch cannot join the network. This is a liveness failure scoped to resharding epochs.

---

### Likelihood Explanation

The bug is triggered automatically whenever:
- A resharding epoch boundary is crossed (shard layout changes), **and**
- Any node requests state parts for a new child shard from a peer that does not have the part cached in `DBCol::StateParts`.

The cache bypass path (lines 298-302) is the only escape: if the part was previously computed and cached before the epoch boundary, the cached bytes are returned without re-deriving the state root. However, the cache is cleared at epoch boundaries (`clear_all_downloaded_parts`), so the buggy path is hit on every fresh request during a resharding epoch. [5](#0-4) 

---

### Recommendation

Mirror the pattern used in `compute_state_response_header`: obtain the **previous** epoch's shard layout and use it to compute the shard index when accessing `prev_block.chunks()`.

```rust
// get_state_response_part — corrected state_root derivation
let prev_epoch_id     = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index  = prev_shard_layout.get_shard_index(shard_id)
    .or_else(|_| {
        // During resharding shard_id is a child; map to its parent.
        let parent = prev_shard_layout.get_parent_shard_id(shard_id)?;
        prev_shard_layout.get_shard_index(parent)
    })?;
let state_root = prev_block
    .chunks()
    .get(prev_shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

The resharding case (child shard not present in old layout) must additionally map `shard_id` to its parent shard in the previous layout, consistent with how `compute_state_response_header` handles `prev_shard_index` via `prev_shard_layout`.

---

### Proof of Concept

1. Configure a testnet with `epoch_length = 10` and a scheduled resharding (e.g., split shard 1 into shards 2 and 3).
2. Let the chain advance past the resharding boundary so the new epoch is active.
3. Start a fresh node that needs to state-sync the new epoch's shards.
4. The fresh node sends `StateRequestPart { shard_id: 3, sync_hash, part_id: 0 }` to a peer.
5. The peer enters `get_state_response_part`:
   - `shard_layout` = new epoch layout (3 shards, indices 0-2)
   - `shard_index = shard_layout.get_shard_index(3)` = 2
   - `prev_block.chunks().get(2)` = `None` (old layout had only 2 chunks)
   - Returns `Error::InvalidShardId(3)`
6. The fresh node never receives a valid state part for shard 3 and cannot complete state sync. [6](#0-5) [7](#0-6)
