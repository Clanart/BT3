Audit Report

## Title
`exactOutputSingle` and `exactOutput` strand excess native ETH on the router, claimable by any address â€” (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
When a user calls `exactOutputSingle` or `exactOutput` with native ETH as `msg.value` for a WETH-input swap, the `pay()` helper in `PeripheryPayments` wraps and forwards only the actual `amountIn` to the pool. The remainder (`msg.value âˆ’ amountIn`) is silently left on the router. Neither function issues a refund before returning. Because `refundETH()` is a public, access-control-free function that sends the entire router ETH balance to `msg.sender`, any third party can drain the stranded ETH in a subsequent transaction.

## Finding Description
`exactOutputSingle` is `payable` and accepts native ETH for WETH-input swaps. [1](#0-0) 

After the pool swap settles, `_justPayCallback` calls `pay(WETH, originalCaller, pool, amountIn)` where `amountIn` is the actual cost determined by the pool. [2](#0-1) 

Inside `pay()`, the WETH branch wraps only `value` (the actual `amountIn`) from `address(this).balance`, leaving any excess ETH on the contract: [3](#0-2) 

`exactOutputSingle` then checks the slippage cap and returns â€” with no ETH refund: [4](#0-3) 

The same omission exists in `exactOutput`, which also returns after `_getExactOutputAmountIn()` and `_clearExpectedCallbackPool()` with no refund: [5](#0-4) 

`refundETH()` is `external` with no caller check â€” it sends the full router ETH balance to whoever calls it: [6](#0-5) 

The `receive()` guard only blocks direct ETH deposits from non-WETH addresses; it does not prevent `msg.value` from accumulating through `payable` swap entry points: [7](#0-6) 

**Exploit flow:**
1. User calls `exactOutputSingle{value: 2 ETH}(...)` with `amountInMaximum = 2 ETH` and `tokenIn = WETH`.
2. Pool settles with `amountIn = 1 ETH`; `pay()` wraps and sends 1 ETH to the pool.
3. 1 ETH remains on the router; `exactOutputSingle` returns without refunding it.
4. Attacker calls `refundETH()` in a subsequent transaction and receives the user's 1 ETH.

## Impact Explanation
This is a direct, unconditional loss of user principal. Any user who calls `exactOutputSingle` or `exactOutput` directly (not via a `multicall` that also includes `refundETH()`) with `msg.value > actual amountIn` permanently loses the difference unless they self-rescue before an attacker does. The stranded ETH is immediately claimable by any address with no preconditions. This meets the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
Exact-output swaps are specifically designed for users who set `amountInMaximum` as a slippage cap above the expected cost. It is standard and expected practice to send more ETH than the exact cost. Any user calling these functions directly (without `multicall`) triggers the vulnerability unconditionally. No special attacker capability is required â€” a single public call to `refundETH()` suffices. The vulnerability is repeatable on every such swap.

## Recommendation
Add an automatic ETH refund at the end of `exactOutputSingle` and `exactOutput` after `amountIn` is known:

```solidity
// At the end of exactOutputSingle / exactOutput:
uint256 excess = address(this).balance;
if (excess > 0) {
    _transferETH(msg.sender, excess);
}
```

This mirrors the pattern used by Uniswap v3's `SwapRouter`. Alternatively, if the `multicall` pattern is intended to be mandatory, add a NatSpec requirement and a `require(msg.value == 0 || isMulticall)` guard, but the automatic refund is strongly preferred.

## Proof of Concept

```solidity
// 1. User calls exactOutputSingle directly (not via multicall):
uint256 amountIn = router.exactOutputSingle{value: 2 ether}(
    IMetricOmmSimpleRouter.ExactOutputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountOut: 1500,
        amountInMaximum: uint128(2 ether),
        recipient: user,
        deadline: block.timestamp + 60,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// amountIn == ~1 ether; 1 ether excess remains on router

// 2. Assert excess is stranded:
assertEq(address(router).balance, 1 ether);

// 3. Attacker drains it:
vm.prank(attacker);
router.refundETH();
assertEq(address(attacker).balance, 1 ether);
assertEq(address(router).balance, 0);
```
