### Title
Missing Shard-ID Binding in `set_state_header` Allows Cross-Shard State Substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer without verifying that the chunk embedded in the header belongs to the requested `shard_id`. A malicious peer can serve a header whose inner chunk belongs to a different shard, causing the syncing node to store the wrong shard's state root under the target shard's DB key, then download and apply state parts that reconstruct the wrong shard's trie into the target shard's storage.

---

### Finding Description

`set_state_header` receives two independent identifiers: the caller-supplied `shard_id` and the `ShardStateSyncResponseHeader` (which contains a `ShardChunk` whose inner header carries its own `shard_id` field). The function performs five validation steps:

1. `validate_chunk_proofs` — verifies the chunk's internal hash, tx-root, and receipts-root consistency. It does **not** inspect `chunk.shard_id()`.
2. `verify_path` against `sync_prev_block_header.chunk_headers_root()` — proves the chunk hash appears somewhere in the block's chunk-header Merkle tree. The proof encodes the leaf's position implicitly through its path directions, but the code never checks that this position equals the shard index of the requested `shard_id`.
3. `prev_chunk` inclusion proof — same structural issue.
4. Incoming-receipt proofs — uses the caller-supplied `shard_id` to hash receipts (`ReceiptList(shard_id, receipts)`), but the attacker can supply the correct receipts for the target shard independently of which chunk is embedded.
5. `validate_state_root_node` — validates the state-root node against `chunk_inner.prev_state_root()`, i.e., the embedded chunk's own state root, not the target shard's expected state root.

After all checks pass, the header is stored under `StateHeaderKey(shard_id, sync_hash)`:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

The stored header now carries shard B's `prev_state_root` but is indexed as shard A's header.

Downstream, `set_state_part` reads this header to obtain the `state_root` used for part validation:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// validates part against shard B's state_root
self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part)
```

`validate_state_part_impl` ignores `shard_id` entirely and only checks the trie against `state_root`:

```rust
fn validate_state_part_impl(&self, state_root: &StateRoot, part_id: PartId, part: &StatePart)
    -> StatePartValidationResult {
    match Trie::validate_state_part(state_root, part_id, partial_state) { ... }
}
```

So state parts for shard B pass validation. They are then applied via `apply_state_part` using shard A's `shard_uid`:

```rust
let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;
tries.apply_all(&trie_changes, shard_uid, &mut store_update);
flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
```

Shard B's trie nodes are written into shard A's storage namespace, completely replacing shard A's state with shard B's data.

The missing check is simply:

```rust
// NEVER PRESENT in set_state_header:
if chunk.shard_id() != shard_id { return Err(...); }
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

### Impact Explanation

**High.** A syncing node (validator catching up to a new shard assignment) has its entire shard-A state replaced with shard-B's trie data. The node cannot produce valid chunks for shard A, cannot pass stateless validation, and must wipe and re-sync. If the attack targets all validators simultaneously catching up to a new shard (common after shard shuffling), chunk production for that shard stalls for the duration of the epoch. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

**Medium.** Any peer on the network can respond to `StateRequestHeader` messages. The syncing node caches the first accepted header and never re-downloads it:

```rust
let existing_header = get_state_header_if_exists_in_storage(&store, sync_hash, shard_id)?;
if let Some(header) = existing_header { return Ok(header); }
```

A malicious peer that responds faster than honest peers wins the race. No special privilege (validator key, chunk-producer role) is required. The attacker only needs to be a reachable peer and to know the `sync_hash` (publicly visible on-chain). [7](#0-6) [8](#0-7) 

---

### Recommendation

Add an explicit shard-ID binding check immediately after extracting the chunk in `set_state_header`, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind the chunk's own shard_id to the requested shard_id
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, verify that the Merkle proof position corresponds to the expected shard index by checking `chunk.shard_id()` against the shard layout's index for the requested `shard_id`, mirroring the position-aware lookup already done in `compute_state_response_header`:

```rust
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
// assert chunk is at position shard_index in the Merkle tree
``` [9](#0-8) [10](#0-9)

### Proof of Concept

**Setup**: Two-shard network. Shard 0 has a large non-empty trie (many accounts). Shard 1 has an empty trie (`EMPTY_ROOT`). A validator V is catching up to shard 0 after a shard-shuffle epoch boundary.

**Attack**:

1. Attacker A is a peer of V. V sends `StateRequestHeader { shard_id: 0, sync_hash }`.

2. A constructs a malicious `ShardStateSyncResponseHeaderV2`:
   - `chunk`: the actual shard-1 chunk from `sync_prev_block` (valid chunk, passes `validate_chunk_proofs`)
   - `chunk_proof`: the Merkle proof for shard-1's chunk in `sync_prev_block.chunk_headers_root()` (valid, passes `verify_path`)
   - `incoming_receipts_proofs`: the actual incoming receipts for shard 0 (valid, passes step 4)
   - `state_root_node`: shard-1's state root node (valid for shard-1's `prev_state_root`)

3. V calls `set_state_header(shard_id=0, sync_hash, malicious_header)`. All five checks pass. The header is stored under `StateHeaderKey(0, sync_hash)` with shard-1's `prev_state_root = EMPTY_ROOT`.

4. V downloads state parts. `set_state_part` reads `state_root = EMPTY_ROOT`. `validate_state_part` with `EMPTY_ROOT` and `part_id=0/1` returns `Valid` for an empty part (the empty-root fast path in `apply_state_part_impl` returns immediately).

5. `set_state_finalize` applies shard-1's chunk (empty transactions, empty receipts) to shard-0's `shard_uid`. Shard 0's `ChunkExtra` is written with `state_root = EMPTY_ROOT`.

6. V now believes shard 0 has an empty state. All subsequent chunk production for shard 0 fails: V cannot find any account data, all transactions are rejected, and V's chunks are invalid. [11](#0-10) [12](#0-11) [13](#0-12) [14](#0-13)

    fn apply_state_part(
        &self,
        shard_id: ShardId,

        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,

            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()

        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =

        self.precompile_contracts(epoch_id, contract_codes)?;
