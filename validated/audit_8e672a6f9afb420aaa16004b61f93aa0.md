### Title
`execute_replace_class` Accepts Undeclared Class Hash Without Validation, Enabling Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. This is an explicit, acknowledged missing precondition check (marked with a `TODO`). Any unprivileged contract can call `replace_class` with an arbitrary, undeclared class hash, committing an inconsistent state to the global state tree. Any funds held in the affected contract are permanently frozen because no valid proof can ever be generated for a future call to it.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), the function reads the requested new class hash from the syscall request and immediately updates `contract_state_changes` with it: [1](#0-0) 

The critical missing check is explicitly acknowledged by the developers: [2](#0-1) 

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

The function proceeds unconditionally to write the new (potentially invalid) class hash into the contract's `StateEntry` and commits it to `contract_state_changes`: [3](#0-2) 

This updated state entry is then squashed and committed to the global Patricia Merkle tree during `state_update`: [4](#0-3) 

The `StateEntry` struct stores `class_hash` as a plain felt with no on-chain validation of its existence: [5](#0-4) 

By contrast, `execute_declare_transaction` enforces that a declared class hash is the result of a valid Sierra class hash calculation before writing to `contract_class_changes`: [6](#0-5) 

`execute_replace_class` has no equivalent guard. The precondition — that the new class hash must correspond to a declared class — is documented but never enforced.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` in the committed state tree points to an undeclared class, no valid execution proof can ever be generated for a call to that contract. The Cairo OS loads contract classes via hints provided by the prover; if the class does not exist in the declared class registry, the prover cannot supply it, and any transaction targeting the contract is unprovable. The contract becomes permanently non-executable. All funds (ETH, ERC-20 tokens, or any other assets) held in the contract's storage are irrecoverably frozen at the protocol level — no upgrade, no recovery, no escape hatch.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is callable by any contract from any unprivileged transaction. A malicious contract deployer can:
1. Deploy a contract that accepts user deposits (e.g., a yield vault, bridge, or escrow).
2. Accumulate user funds.
3. Issue a single `replace_class` syscall with an arbitrary undeclared felt (e.g., `1`).

The OS accepts the state transition without complaint. The contract's class_hash is now invalid in the committed global state. All deposited funds are permanently frozen. No special privilege, leaked key, or operator access is required beyond the ability to deploy and call a contract — which is available to any network participant.

---

### Recommendation

In `execute_replace_class` (`syscall_impls.cairo`), before updating `contract_state_changes`, verify that the requested `class_hash` exists in `contract_class_changes` (i.e., it was declared in the current block) or in the pre-existing class commitment tree. This mirrors the invariant already enforced in `execute_declare_transaction`, where `prev_value=0` ensures a class is declared at most once and only with a valid hash: [7](#0-6) 

The fix should resolve the existing `TODO` by performing a `dict_read` on `contract_class_changes` (and/or the pre-state class tree) to confirm the new class hash is non-zero and declared before allowing the state update.

---

### Proof of Concept

1. Attacker deploys contract `V` with a legitimate class hash `C_valid` (declared).
2. Users deposit funds into contract `V` (e.g., via a `deposit` entry point).
3. Attacker sends an invoke transaction that calls `replace_class(class_hash=0xdeadbeef)` from within contract `V`.
4. `execute_replace_class` in `syscall_impls.cairo` (line 896) reads `class_hash = 0xdeadbeef` from the request.
5. The missing check (line 898 TODO) means no validation occurs.
6. `dict_update` (line 906) writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
7. `state_update` in `state.cairo` squashes and commits this entry to the global Patricia tree.
8. The global state root now encodes contract `V` with `class_hash = 0xdeadbeef`.
9. Any future transaction targeting contract `V` requires the prover to supply the class for `0xdeadbeef`. No such class exists. The proof cannot be generated.
10. All user funds in contract `V` are permanently frozen with no recovery path.
