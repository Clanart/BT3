### Title
Unnecessary `payable` Modifiers on `logMetadata`, `logMetadata1155`, `deployToken`, and `finTransfer` in Base `OmniBridge` Permanently Lock Caller ETH — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol` declares four public functions as `payable` — `logMetadata`, `logMetadata1155`, `deployToken`, and `finTransfer` — but in the base contract their extension hooks (`logMetadataExtension`, `deployTokenExtension`, `finTransferExtension`) are empty no-ops. Any ETH sent with these calls is accepted by the contract and permanently locked, because no withdrawal path exists.

---

### Finding Description

`OmniBridge` is a non-abstract, deployable contract. Its four public `payable` functions delegate ETH handling to internal virtual extension hooks:

- `logMetadata` (L224) calls `logMetadataExtension` (L229), which is an empty virtual at L272–277.
- `logMetadata1155` (L234) calls `logMetadataExtension` (L257), same empty virtual.
- `deployToken` (L135) calls `deployTokenExtension` (L174), which is an empty virtual at L197–202.
- `finTransfer` (L279) calls `finTransferExtension` (L357), which is an empty virtual at L369–371. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

In `OmniBridgeWormhole`, these hooks forward `msg.value` to `_wormhole.publishMessage{value: msg.value}(...)`, which is the intended use. But in the base `OmniBridge`, the hooks are empty, so `msg.value` is silently swallowed. [5](#0-4) [6](#0-5) 

The contract has a `receive()` fallback (L574) and no `withdraw` or ETH-recovery function anywhere. The only ETH egress is the `finTransfer` ETH-transfer path (L317–322), which requires a valid MPC signature and a matching `payload.tokenAddress == address(0)` payload — it is not a general recovery mechanism. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Any ETH sent by a caller to `logMetadata`, `logMetadata1155`, `deployToken`, or `finTransfer` on a base `OmniBridge` deployment is permanently irrecoverable. There is no admin sweep, no rescue function, and no upgrade path that would allow recovery without a full proxy upgrade. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**. [1](#0-0) [9](#0-8) 

---

### Likelihood Explanation

`OmniBridge` is not declared `abstract`, so it is directly deployable. A deployment on a chain that does not use Wormhole (or any future chain integration that subclasses `OmniBridge` without overriding the extension hooks) exposes all four functions. Additionally, `logMetadata` and `logMetadata1155` have no access control — any token holder or bridge user can call them. A user who observes that `OmniBridgeWormhole` requires ETH for these calls (the Wormhole fee) may reasonably send ETH when calling the same function on a base deployment, permanently losing it. [10](#0-9) [11](#0-10) 

---

### Recommendation

Remove the `payable` modifier from `logMetadata`, `logMetadata1155`, `deployToken`, and `finTransfer` in `OmniBridge.sol`. If a subclass needs to accept ETH (e.g., for Wormhole fees), override the function in the subclass and add `payable` there. Alternatively, add a `require(msg.value == 0)` guard in each base-contract function body to revert on accidental ETH sends. [1](#0-0) [12](#0-11) 

---

### Proof of Concept

1. Deploy `OmniBridge` directly (not `OmniBridgeWormhole`).
2. Call `logMetadata(someERC20)` with `{value: 1 ether}`.
3. The call succeeds: `logMetadataExtension` is a no-op, `msg.value` is never forwarded or returned.
4. The contract balance increases by 1 ETH.
5. No function in `OmniBridge` allows recovery of this ETH — it is permanently locked.

The same applies to `logMetadata1155`, `deployToken` (with a valid MPC-signed metadata payload), and `finTransfer` (with a valid MPC-signed transfer payload). [2](#0-1) [4](#0-3) [3](#0-2)
