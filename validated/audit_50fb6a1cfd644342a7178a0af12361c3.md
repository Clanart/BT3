### Title
`sweepToken` and `unwrapWETH9` send the router's entire balance to any caller-supplied recipient with no ownership check, enabling front-running theft of router-held tokens ‚Äî (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`sweepToken` and `unwrapWETH9` are `public payable` functions that transfer the router's **entire** ERC-20 or WETH balance to an arbitrary caller-supplied `recipient` address. There is no check that `recipient == msg.sender` or that the caller was the one who deposited the tokens. Any actor who observes a pending sweep transaction in the mempool can front-run it and redirect the full balance to themselves.

---

### Finding Description

`PeripheryPayments.sweepToken` and `PeripheryPayments.unwrapWETH9` both operate on the router's full token balance and accept an arbitrary `recipient` parameter:

```solidity
// PeripheryPayments.sol
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);   // ‚Üê entire balance, any recipient
    }
}

function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);                  // ‚Üê entire balance, any recipient
    }
}
```

Neither function verifies that `msg.sender` is the depositor, nor that `recipient == msg.sender`. The router's balance is a shared, unattributed pool.

Tokens legitimately land on the router when a user routes a swap with `recipient = address(router)` ‚Äî the intended pattern for a WETH-output swap followed by `unwrapWETH9`, or any ERC-20 output followed by `sweepToken`. If the user submits the swap and the sweep as two separate transactions (rather than a single atomic `multicall`), the window between them is exploitable.

This is the direct Metric OMM analog of the ENS `makeCommitment` bug: just as the ENS commitment hash omits the `owner` field so that Eve can repurpose Alice's commitment to register a domain for herself, the router's sweep functions omit any depositor binding so that Eve can repurpose Alice's router balance to sweep tokens to herself.

---

### Impact Explanation

A user who routes a swap with `recipient = address(router)` and then calls `unwrapWETH9` or `sweepToken` in a separate transaction loses **100% of the swap output**. The attacker gains the full balance. This is a direct loss of user principal with no partial recovery.

---

### Likelihood Explanation

- The attack requires Alice to submit the sweep as a separate transaction rather than inside a `multicall`. This is a realistic condition: users interacting through simple EOA scripts, third-party frontends that do not compose `multicall`, or programmatic integrators that call each step individually are all exposed.
- The attacker only needs to monitor the public mempool for calls to `unwrapWETH9` or `sweepToken` on the router and submit the same call with `recipient = attacker` at a higher gas price ‚Äî a trivial MEV operation.
- No special privileges, no malicious setup, and no non-standard tokens are required.

---

### Recommendation

Restrict `sweepToken` and `unwrapWETH9` so that the `recipient` must equal `msg.sender`, or remove the `recipient` parameter entirely and always send to `msg.sender`:

```solidity
function sweepToken(address token, uint256 amountMinimum) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(msg.sender, balanceToken);
    }
}
```

Alternatively, if an arbitrary `recipient` is intentionally supported (e.g., for contract integrators), document clearly that these functions must always be composed inside an atomic `multicall` with the preceding swap, and add a NatSpec warning that standalone calls are unsafe.

---

### Proof of Concept

1. Alice wants to swap `tokenA ‚Üí WETH` and receive native ETH.
2. Alice calls `exactInputSingle({tokenIn: tokenA, tokenOut: WETH, recipient: address(router), amountIn: X, ...})` ‚Äî WETH lands on the router.
3. Alice submits a second transaction: `unwrapWETH9(X, Alice)`.
4. Eve observes Alice's pending `unwrapWETH9` call in the mempool.
5. Eve submits `unwrapWETH9(0, Eve)` with a higher gas price.
6. Eve's transaction executes first: the router's entire WETH balance is unwrapped and sent to Eve.
7. Alice's transaction executes next: `balanceWETH == 0`, `amountMinimum == X`, revert `InsufficientWETH`.
8. Alice loses 100% of her swap output; Eve gains it.

The same attack applies to `sweepToken` for any ERC-20 output token.
