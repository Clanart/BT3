### Title
EIP-7702 Authorization Nonce Consumed and Delegation Committed Even When EVM Execution Fails (Post-Hook Failure Path) — (`File: x/evm/keeper/state_transition.go`)

### Summary

Ethermint's `ApplyTransaction` deliberately commits EIP-7702 authorization effects (authority nonce bump + delegation code) to a **durable, separate cache context** that survives a post-hook failure that discards the main `tmpCtx`. This means an unprivileged attacker who controls the outer transaction sender can force an authority's nonce to be permanently incremented and a delegation to be permanently installed, even though the EVM call itself is rolled back and the transaction is reported as failed (`VmError = ErrPostTxProcessing`). The authority's account state is mutated without the authority's EVM call succeeding, and without any access control on who may submit the outer transaction.

---

### Finding Description

In `ApplyTransaction`, when hooks are registered, two separate cache contexts are created:

1. `tmpCtx` — the main execution context, committed only if both EVM execution and post-hooks succeed.
2. `durableAuthorizationCtx` — a sibling cache context whose sole purpose is to persist EIP-7702 authorization effects even if `tmpCtx` is later discarded. [1](#0-0) 

Inside `ApplyMessageWithConfig`, after the authorization list is validated and applied to the main `stateDB`, the same effects are **replayed** onto a fresh `durableStateDB` backed by `durableAuthorizationCtx`, and that `durableStateDB` is committed immediately: [2](#0-1) 

Back in `ApplyTransaction`, after the EVM call returns, if the post-hook fails (setting `res.VmError = ErrPostTxProcessing` and leaving `tmpCtxCommitted = false`), the code at line 266–268 commits the durable authorization context unconditionally: [3](#0-2) 

The `setAuthorizationDelegation` function writes both the nonce bump and the delegation code: [4](#0-3) 

There is **no access control** on who may be the outer transaction sender (`msg.From`). Any unprivileged account can craft a type-4 (EIP-7702) transaction that:
- Names a victim EOA as the authority in the authorization list (signed by the victim's key — this is the normal EIP-7702 flow).
- Targets a contract that will cause the post-hook to fail (e.g., a contract that emits a log recognized by a registered hook that always returns an error).

The result: the victim's nonce is permanently incremented and their account code is permanently set to the delegation pointer, even though the EVM call is rolled back and the transaction receipt shows `status=0`.

The analog to the external report is exact: just as `notifyFeeReceived` can be called by anyone to corrupt event authenticity, here **any unprivileged outer sender** can trigger the durable authorization commit path, permanently mutating a third-party authority's account state without a successful EVM execution.

---

### Impact Explanation

**High — EIP-7702 authorization/nonce/signer verification bypass enabling unauthorized account/code mutation.**

1. **Nonce burn**: The authority's nonce is permanently incremented by 1. This invalidates any pending transactions the authority has signed with the old nonce, effectively griefing the authority's transaction queue.
2. **Unauthorized code delegation**: The authority's account code is permanently set to `0xef0100 || delegate_address` (the EIP-7702 delegation prefix). Any subsequent call to the authority's address will execute the delegate's code in the authority's context, which can drain balances, modify storage, etc.
3. **Replay prevention bypass**: Because the nonce is consumed, the authority cannot re-use the same authorization tuple. However, the delegation itself persists, meaning the attacker has permanently altered the authority's on-chain behavior.

The authority's EVM-denom balance is protected (the drain call is rolled back with `tmpCtx`), but the nonce and code mutation are committed via the durable path. This matches the **High** impact category: "EIP-7702 authorization, nonce, or signer verification bypass enabling unauthorized account/code mutation."

---

### Likelihood Explanation

**Medium-High.** The preconditions are:

1. The chain must have Prague hardfork enabled (EIP-7702 active) — this is a production configuration.
2. The chain must have at least one registered `EvmHooks` implementation that can be made to fail (common in production deployments that integrate with native modules via hooks).
3. The attacker must obtain a valid signed EIP-7702 authorization from the victim — this is the standard EIP-7702 flow; the victim signs an authorization intending it to be used in a specific transaction, but the attacker can front-run or reuse it in a different outer transaction.
4. The attacker must target a contract that causes the post-hook to fail.

The authorization signature is a standard off-chain artifact. If a victim broadcasts a signed authorization (e.g., via a dApp), an attacker can extract it from the mempool and wrap it in a new outer transaction targeting a hook-failing contract.

---

### Recommendation

The durable authorization commit should only fire when the **main execution path** (EVM call + post-hooks) succeeds, not when it fails. The current logic at line 266–268:

```go
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
```

commits the durable context precisely when `tmpCtx` was **not** committed — i.e., on failure. This is inverted from what security requires.

The correct behavior is to commit the durable authorization **only when `tmpCtxCommitted` is true** (i.e., both EVM execution and post-hooks succeeded), or alternatively to remove the durable path entirely and accept that authorization effects are rolled back with the rest of the transaction on post-hook failure (matching go-ethereum's behavior, where there are no post-hooks). [3](#0-2)

### Proof of Concept

**Setup:**
- Chain has Prague hardfork enabled.
- A post-hook is registered that fails when it sees a specific log topic (e.g., a hook that processes ERC-20 transfer events and returns an error for a specific token).
- Victim (`authority`) has nonce 0 and signs an EIP-7702 authorization: `{chainID, delegate=0xDEAD, nonce=0}`.

**Attack:**
1. Attacker obtains the victim's signed authorization (from mempool, dApp, or social engineering).
2. Attacker constructs a type-4 transaction:
   - `from`: attacker's address
   - `to`: a contract that emits the log topic that triggers the hook failure
   - `authorizationList`: `[victim_signed_auth]`
3. Attacker submits the transaction.

**Execution path in `ApplyTransaction`:**
- `tmpCtx, commit = ctx.CacheContext()` — main context branched.
- `durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()` — durable context branched.
- `ApplyMessageWithConfig` runs: authorization validated, `applyAuthorization` sets victim's nonce to 1 and code to `0xef0100||0xDEAD` in `stateDB`; `applyDurableAuthorization` replays the same into `durableStateDB`, which is committed immediately into `durableAuthorizationCtx`. [2](#0-1) 

- EVM call executes, emits the hook-triggering log, succeeds at EVM level.
- `PostTxProcessing` hook returns error → `res.VmError = ErrPostTxProcessing`, `tmpCtxCommitted = false`, `tmpCtx` is discarded.
- Line 266: `commitDurableAuthorization != nil && !tmpCtxCommitted` → **true** → `commitDurableAuthorization()` is called, flushing victim's nonce=1 and delegation code into the parent `ctx`. [3](#0-2) 

**Result:** Transaction receipt shows `status=0` (failed). Victim's nonce is now 1 and their account code is `0xef0100||0xDEAD`. The attacker has permanently delegated the victim's account to an arbitrary contract without the victim's EVM call ever succeeding.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
