### Title
`ProofOfInclusion::valid()` Uses User-Supplied `combined_hash` as Authoritative Root Instead of Externally-Committed Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate contains a tautological final check: it compares the computed hash against `self.root_hash()`, which itself returns the user-supplied `combined_hash` from the last proof layer. This means `valid()` only verifies internal self-consistency of the proof structure and never validates the proof against an externally-committed tree root. An attacker can construct an internally-consistent `ProofOfInclusion` for a key that does not exist in the actual committed tree, and `valid()` will return `true`.

---

### Finding Description

`ProofOfInclusion` is a `Streamable`-serializable struct with Python bindings (`from_py_object`), meaning it can be constructed from arbitrary untrusted bytes. [1](#0-0) 

Each `ProofOfInclusionLayer` carries a `combined_hash` field that is entirely user-supplied: [2](#0-1) 

`root_hash()` returns the `combined_hash` from the **last user-supplied layer**, not any externally-committed root: [3](#0-2) 

`valid()` iterates through layers, verifying that each `calculated_hash == layer.combined_hash`, then sets `existing_hash = calculated_hash`. After the loop, `existing_hash` equals the last `calculated_hash`, which was already verified to equal the last `layer.combined_hash`. The final check `existing_hash == self.root_hash()` therefore compares `last.combined_hash` against `last.combined_hash` — a tautology that is always `true` if the loop completes: [4](#0-3) 

The method never accepts an expected root parameter and never compares against any externally-committed tree root. The Python-exposed `valid()` method is the sole validation gate visible to callers.

---

### Impact Explanation

An attacker who can submit a `ProofOfInclusion` to any DataLayer consumer that calls `proof.valid()` without separately asserting `proof.root_hash() == committed_root` can forge a proof of inclusion for an arbitrary key-value pair in an arbitrary tree. The forged proof will be internally consistent (each `combined_hash` is correctly computed from the attacker-chosen `other_hash` values), but it proves membership in a tree the attacker constructed, not the actual committed tree. This allows untrusted input to prove invalid state — matching the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

---

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` and exposed to Python via `from_py_object`. Any Python DataLayer consumer that deserializes a proof from a peer and calls `proof.valid()` as the sole check is vulnerable. The existing test suite calls `proof.valid()` without a root comparison, demonstrating the pattern is established and likely replicated in production consumers: [5](#0-4) 

---

### Recommendation

`valid()` should accept the expected committed root as a parameter and compare `existing_hash` against it instead of against `self.root_hash()`:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == *expected_root  // compare against actual committed root
}
```

Alternatively, remove `combined_hash` from `ProofOfInclusionLayer` entirely (it is redundant given it is always recomputed) and have `root_hash()` return the final computed hash rather than a stored field.

---

### Proof of Concept

1. Obtain the actual committed tree root `R_actual` for a DataLayer store.
2. Construct a fresh `MerkleBlob` containing a fabricated key `K_fake` not present in the actual store.
3. Call `get_proof_of_inclusion(K_fake)` on the fabricated blob to obtain an internally-consistent `ProofOfInclusion` with root `R_fake ≠ R_actual`.
4. Serialize this proof via `Streamable` and submit it to a DataLayer consumer.
5. The consumer deserializes it into a `ProofOfInclusion` and calls `proof.valid()`.
6. `valid()` returns `true` because the loop verifies internal consistency and the final check `existing_hash == self.root_hash()` compares `R_fake` against `R_fake` — a tautology.
7. The consumer accepts the forged proof of inclusion for `K_fake` in the actual committed tree. [4](#0-3)
