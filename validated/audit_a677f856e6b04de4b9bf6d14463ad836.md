### Title
Missing Validation of `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary `class_hash` from the caller without verifying that the hash corresponds to a previously declared class. This is directly acknowledged by a `TODO` comment in the code. An attacker can supply an undeclared class hash, permanently replacing a contract's class with one that does not exist in the class registry. Any subsequent call to that contract will fail at the OS level because the class cannot be resolved, permanently freezing any funds held in the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract state without any check against `contract_class_changes` (the dict tracking declared classes):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `TODO` comment at line 898 explicitly acknowledges the missing validation. Compare this to `execute_declare_transaction` in `transaction_impls.cairo`, which rigorously validates the class hash pre-image via `finalize_class_hash` before accepting a class declaration. No equivalent guard exists in `execute_replace_class`.

The `execute_deploy` path in `syscall_impls.cairo` similarly uses `request.class_hash` directly, but the constructor execution will fail at runtime if the class is absent. For `replace_class`, the OS silently commits the invalid class hash to state with no runtime failure at the point of the syscall — the failure is deferred to the next call into the contract, at which point the state is already committed and irreversible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared hash:
1. The OS state permanently records the invalid class hash for that contract address.
2. Every subsequent call to the contract (including any entry point that would allow recovery or withdrawal) will fail because the OS cannot resolve the class.
3. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible.

There is no recovery path: the `replace_class` change is committed to `contract_state_changes` and propagated to the Merkle state root. The revert log entry written at line 912 only helps if the *current* transaction reverts — but the syscall itself succeeds (gas is deducted, response header is written with `failure_flag=0`), so the transaction does not revert.

---

### Likelihood Explanation

**Medium.**

The attacker-controlled entry path requires:
1. A contract that exposes a public or semi-public upgrade/replace-class code path (common in upgradeable DeFi contracts, multisigs, proxies).
2. The attacker being able to call that function and supply an arbitrary `class_hash` argument.

This is not hypothetical: upgradeable contract patterns on StarkNet routinely expose `replace_class` via governance or owner functions. An attacker who can call such a function (e.g., via a governance exploit, a reentrancy, or a misconfigured access control) can supply `0x1` or any other undeclared felt as the class hash. The OS will accept it unconditionally.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, verify that the hash exists in `contract_class_changes`:

```cairo
let class_hash = request.class_hash;

// Validate that the class hash has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the implicit guarantee enforced during `execute_declare_transaction`, where `dict_update` with `prev_value=0` ensures a class is only declared once and always has a valid compiled class hash.

---

### Proof of Concept

1. Declare a valid class `C` and deploy a contract `A` using class `C`. Fund contract `A` with tokens.
2. Contract `A` exposes a public `upgrade(new_class_hash: felt)` function that calls the `replace_class` syscall.
3. Attacker calls `upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. The OS executes `execute_replace_class`: gas is deducted, `contract_state_changes` is updated with `class_hash=0xdeadbeef`, a revert log entry is written, and the syscall returns success.
5. The transaction is included in a block. The state root now records `class_hash=0xdeadbeef` for contract `A`.
6. Any subsequent `call_contract` or `invoke` targeting contract `A` causes the OS to look up class `0xdeadbeef` in `contract_class_changes`, find nothing, and fail.
7. All funds in contract `A` are permanently frozen. [1](#0-0) [2](#0-1)
