Audit Report

## Title
Excess native ETH sent to `exactOutputSingle`/`exactOutput` is silently stranded and stealable via permissionless `refundETH()` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
`exactOutputSingle` and `exactOutput` are `payable` and accept native ETH for WETH-input swaps. The internal `pay()` helper wraps only the exact amount the pool requests, leaving any `msg.value` surplus sitting in the router. Neither function issues a refund. Because `refundETH()` is permissionless and forwards the entire router ETH balance to `msg.sender`, any third party can drain the stranded ETH in a follow-up transaction.

## Finding Description
Both entry points are `payable`: [1](#0-0) [2](#0-1) 

When `tokenIn == WETH` and the router holds native ETH, `pay()` wraps exactly `value` (the pool-requested amount), not the full `address(this).balance`: [3](#0-2) 

Any `msg.value` above `actualAmountIn` is never consumed. Both functions return after `_clearExpectedCallbackPool()` with no refund logic: [4](#0-3) [5](#0-4) 

`refundETH()` is fully permissionless — it sends the entire router ETH balance to whoever calls it: [6](#0-5) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only applies to bare ETH transfers, not to `payable` function calls, so it does not prevent users from attaching ETH to `exactOutputSingle`/`exactOutput`. [7](#0-6) 

## Impact Explanation
Direct loss of user principal. A user calling `exactOutputSingle{value: amountInMaximum}` where the pool fills at a lower `actualAmountIn` loses `amountInMaximum - actualAmountIn` ETH to the first caller of `refundETH()`. This meets the Critical/High direct-loss-of-user-principal threshold: the loss is proportional to the user's slippage budget and is fully extractable by an unprivileged attacker in the same or next block.

## Likelihood Explanation
For exact-output swaps the caller cannot know `actualAmountIn` before execution, so sending `msg.value = amountInMaximum` is the natural and expected pattern. The functions are individually `payable` and callable without multicall. No NatSpec warns that excess ETH is not refunded. MEV bots routinely monitor router contracts for stranded ETH, making extraction near-certain whenever the condition arises.

## Recommendation
Add an automatic refund at the end of both exact-output entry points:

```solidity
// after _clearExpectedCallbackPool() in exactOutputSingle and exactOutput
uint256 leftover = address(this).balance;
if (leftover > 0) _transferETH(msg.sender, leftover);
```

Alternatively, enforce that callers must use multicall (`exactOutputSingle` + `refundETH`) when paying with native ETH by rejecting `msg.value > 0` on non-multicall paths, and document this prominently in NatSpec.

## Proof of Concept
1. Victim calls `router.exactOutputSingle{value: 1 ether}(params)` with `tokenIn = WETH`, `amountInMaximum = 1 ether`; pool fills at `actualAmountIn = 0.6 ether`.
2. `pay()` wraps 0.6 ETH → WETH and transfers to pool; 0.4 ETH remains in router. [3](#0-2) 
3. `exactOutputSingle` returns; router holds 0.4 ETH with no owner record. [4](#0-3) 
4. Attacker calls `router.refundETH()`; router sends 0.4 ETH to attacker. [6](#0-5) 
5. Victim loses 0.4 ETH.

```solidity
contract Thief {
    function steal(address router) external {
        IPeripheryPayments(router).refundETH();
        payable(msg.sender).transfer(address(this).balance);
    }
    receive() external payable {}
}
```
