### Title
Live `transaction_ttl` Update Retroactively Expires In-Flight Mempool Transactions - (`crates/apollo_mempool/src/mempool.rs`, `crates/apollo_mempool/src/transaction_pool.rs`)

---

### Summary

The mempool's `transaction_ttl` is a live-updatable dynamic config parameter. Transactions store only their `submission_time` at insertion; expiry is computed on-the-fly as `now - transaction_ttl`. When an operator shortens `transaction_ttl` via the `ConfigManager` hot-reload path, every in-flight transaction whose age exceeds the new (shorter) TTL is immediately and silently dropped — even though those transactions were submitted under a longer TTL guarantee. Conversely, extending the TTL keeps transactions alive past the deadline users expected.

---

### Finding Description

**Root cause — dynamic TTL applied retroactively to stored submission times**

At insertion, `TimedTransactionMap::insert` records only the wall-clock `submission_time`:

```rust
// crates/apollo_mempool/src/transaction_pool.rs:406-413
fn insert(&mut self, tx: TransactionReference) -> Option<SubmissionID> {
    let submission_id = SubmissionID {
        submission_time: self.clock.now(),
        tx_hash: tx.tx_hash,
        batching_time: None,
    };
    ...
}
```

No expiry deadline is stored. Every expiry check recomputes the cutoff from the *current* `transaction_ttl`:

```rust
// crates/apollo_mempool/src/mempool.rs:836
let submission_cutoff_time = self.clock.now() - self.config.dynamic_config.transaction_ttl;
```

```rust
// crates/apollo_mempool/src/transaction_pool.rs:431-432
let split_off_value = SubmissionID {
    submission_time: self.clock.now() - duration,   // duration = current TTL
    ...
};
```

**Live-update path**

`ConfigManagerRunner` watches config files and periodically polls; on any change it calls `set_node_dynamic_config`, which updates the `ConfigManager`'s `latest_node_dynamic_config`. The mempool wrapper fetches and applies the new value before each request:

```rust
// crates/apollo_mempool/src/communication.rs:110-117
async fn update_dynamic_config(&mut self) {
    let mempool_dynamic_config = self
        .config_manager_client
        .get_mempool_dynamic_config()
        .await
        .expect("Should be able to get mempool dynamic config");
    self.mempool.update_dynamic_config(mempool_dynamic_config);
}
```

```rust
// crates/apollo_mempool/src/mempool.rs:689-691
pub(crate) fn update_dynamic_config(&mut self, mempool_dynamic_config: MempoolDynamicConfig) {
    self.config.dynamic_config = mempool_dynamic_config;
}
```

**Concrete scenario**

| Time | Event |
|------|-------|
| T=0 | Tx submitted; `submission_time = 0`; TTL = 300 s; user expects expiry at T=300 |
| T=200 | Operator shortens TTL to 100 s via config file |
| T=200 | `ConfigManagerRunner` detects change, pushes new `MempoolDynamicConfig` |
| T=200 | Next `add_tx` / `get_txs` call triggers `update_dynamic_config` then `remove_expired_txs` |
| T=200 | `submission_cutoff = now(200) − 100 = T=100`; tx submitted at T=0 < T=100 → **dropped** |

The transaction is silently evicted 100 s before the user-visible deadline.

---

### Impact Explanation

**High — Mempool/gateway admission rejects valid transactions before sequencing.**

A transaction that passed all gateway validation checks and was accepted into the mempool with an implicit TTL guarantee is silently dropped before it can be sequenced. From the submitter's perspective the transaction simply disappears; no error is returned. The submitter may resubmit, causing duplicate-nonce confusion, or miss a time-sensitive operation entirely.

The inverse (TTL extension) keeps stale transactions — with outdated gas prices or nonces that have since been superseded on-chain — alive in the pool, where they can be returned by `get_txs` and fed to the batcher, potentially causing unnecessary execution failures or incorrect fee accounting.

---

### Likelihood Explanation

The `transaction_ttl` field is explicitly placed in `MempoolDynamicConfig` (not `MempoolStaticConfig`), signalling operator intent to change it at runtime. The production deployment overlay already overrides it from the default 60 s to 300 s, confirming the parameter is actively tuned. The `ConfigManagerRunner` hot-reload loop runs continuously; any config-file edit or periodic poll that changes `transaction_ttl` immediately triggers the retroactive expiry on the next mempool operation.

---

### Recommendation

Store the absolute expiry deadline at insertion time rather than recomputing it from the current TTL:

```rust
// In SubmissionID
pub(crate) struct SubmissionID {
    pub(crate) submission_time: DateTime,
    pub(crate) expiry_time: DateTime,   // = submission_time + ttl_at_insertion
    tx_hash: TransactionHash,
    pub(crate) batching_time: Option<DateTime>,
}
```

In `TimedTransactionMap::insert`, capture the TTL at the moment of insertion and compute `expiry_time = now + ttl`. In `remove_txs_older_than`, compare `now` against the stored `expiry_time` instead of recomputing the cutoff from the current TTL. This mirrors the fix suggested in BendDAO M-17: store the deadline, not the start time.

---

### Proof of Concept

```
1. Start mempool with transaction_ttl = 300 s.
2. Submit Tx A at T=0  →  submission_time=0, expected expiry T=300.
3. Submit Tx B at T=150 →  submission_time=150, expected expiry T=450.
4. At T=200, write mempool_config.dynamic_config.transaction_ttl = 100
   to the watched config file.
5. ConfigManagerRunner detects the change and calls set_node_dynamic_config.
6. At T=201, any add_tx or get_txs call triggers update_dynamic_config
   (TTL now = 100 s) then remove_expired_txs.
7. submission_cutoff = 201 − 100 = 101.
   Tx A (submission_time=0)   < 101  →  DROPPED  (expected to live until T=300)
   Tx B (submission_time=150) > 101  →  survives (correct)
8. Tx A is gone from the mempool with no error returned to its submitter.
```

    pub fn remove_txs_older_than(
        &mut self,
        duration: Duration,
        exclude_txs: &HashMap<ContractAddress, Nonce>,
    ) -> Vec<TransactionReference> {
        let split_off_value = SubmissionID {
            submission_time: self.clock.now() - duration,
            tx_hash: Default::default(),
            batching_time: None,
        };
        let old_txs = self.txs_by_submission_time.split_off(&split_off_value);

        let mut removed_txs = Vec::new();
        for (submission_id, tx) in old_txs.into_iter() {
            if exclude_txs.get(&tx.address).is_some_and(|nonce| tx.nonce < *nonce) {
                // The transaction should be preserved. Add it back.
                self.txs_by_submission_time.insert(submission_id, tx);
            } else {
                self.hash_to_submission_id.remove(&tx.tx_hash).expect(
                    "Transaction should have a submission ID if it is in the timed transaction \
                     map.",
                );
                removed_txs.push(tx);
            }
        }

        removed_txs
    }
