### Title
Per-Message Balance Check Without Cumulative Cost Tracking Admits Under-Funded Multi-`MsgEthereumTx` Transactions - (File: `ante/eth.go`)

### Summary

`VerifyEthAccount` and `CheckEthCanTransfer` in `ante/eth.go` each iterate over every `MsgEthereumTx` in a single Cosmos transaction and validate the sender's balance against each message's cost **individually**. The balance is re-read from the store on every iteration but is never decremented between iterations within those functions. A Cosmos transaction that wraps N `MsgEthereumTx` messages whose individual costs each fit within the sender's balance, but whose **aggregate** cost exceeds it, passes all ante-handler checks and is admitted to the mempool. Only during block execution does the shortfall surface, wasting block space and enabling a mempool-flooding DoS.

### Finding Description

`VerifyEthAccount` (the balance-sufficiency gate for EVM transactions) reads the sender's balance once per loop iteration and calls `CheckSenderBalance` against `tx.Cost()` (= `value + gasLimit × gasPrice`) for each message independently: [1](#0-0) 

`CheckSenderBalance` only compares the snapshot balance against the single-message cost: [2](#0-1) 

Because neither function accumulates a running total of costs already "committed" to earlier messages in the same loop, the same balance value is compared against every message's cost in isolation.

`CheckEthCanTransfer` has the identical structural defect for the value-transfer component: [3](#0-2) 

`canTransfer` reads the live store balance and compares it to the single-message value with no awareness of sibling messages: [4](#0-3) 

`CheckEthGasConsume` does deduct fees sequentially (so a balance that cannot cover even one message's fee is caught), but it only deducts the **fee** component, not the **value** component: [5](#0-4) 

After `CheckEthGasConsume` reduces the balance by the sum of all fees, `VerifyEthAccount` and `CheckEthCanTransfer` still compare the post-fee balance against each message's remaining cost/value individually, with no cumulative subtraction between iterations.

**Concrete example**

| | Msg 1 | Msg 2 | Total |
|---|---|---|---|
| value | 60 | 60 | 120 |
| fee | 1 | 1 | 2 |
| cost | 61 | 61 | 122 |

Sender balance = 100. After `CheckEthGasConsume` deducts fees: balance = 98.

- `VerifyEthAccount` Msg 1: 98 ≥ 61 ✓
- `VerifyEthAccount` Msg 2: 98 ≥ 61 ✓ (same 98, not 37)
- `CheckEthCanTransfer` Msg 1: 98 ≥ 60 ✓
- `CheckEthCanTransfer` Msg 2: 98 ≥ 60 ✓ (same 98, not 38)

Ante handler accepts the transaction. Actual total cost = 122 > 100. During EVM execution Msg 2 fails with insufficient funds.

### Impact Explanation

The ante handler is the sole admission gate for the mempool and for block inclusion. Accepting a transaction whose aggregate cost exceeds the sender's balance means:

1. **Invalid transactions enter the mempool** — they appear valid to every node, consuming mempool slots that could hold legitimate transactions.
2. **Block space is wasted** — the proposer includes the transaction; during `DeliverTx` the later messages fail (EVM-level revert), consuming gas and block capacity without producing useful state transitions.
3. **Repeatable DoS** — an attacker with a modest balance can continuously submit N-message transactions (each message individually affordable, collectively not), saturating the mempool and degrading throughput for all users.

This matches the allowed High impact: *"ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

- Ethermint explicitly supports multi-`MsgEthereumTx` Cosmos transactions (the ante handler iterates `tx.GetMsgs()` with no upper bound on count).
- No privileged role is required; any unprivileged account can craft such a transaction.
- The attack requires only a small balance and standard transaction submission via the JSON-RPC or gRPC endpoint.
- The structural defect is present in both `VerifyEthAccount` and `CheckEthCanTransfer`, so patching one alone is insufficient.

### Recommendation

Introduce a per-sender running cost accumulator inside both `VerifyEthAccount` and `CheckEthCanTransfer`. After each message's individual check passes, subtract that message's cost (or value) from a local `remainingBalance` variable and use that reduced value for the next iteration instead of re-reading the unchanged store balance.

```go
// VerifyEthAccount – proposed fix sketch
remainingBalance := evmKeeper.GetBalance(ctx, from, evmDenom)
for _, msg := range tx.GetMsgs() {
    ethTx := msgEthTx.AsTransaction()
    if err := keeper.CheckSenderBalance(sdkmath.NewIntFromBigIntMut(remainingBalance.ToBig()), ethTx); err != nil {
        return errorsmod.Wrap(err, "failed to check sender balance")
    }
    // deduct this message's cost from the running balance
    remainingBalance.Sub(remainingBalance, uint256.MustFromBig(ethTx.Cost()))
}
```

Apply the same pattern in `CheckEthCanTransfer` for the value component.

### Proof of Concept

1. Deploy an Ethermint node.
2. Fund account `A` with balance = 100 wei.
3. Construct a Cosmos transaction with `ExtensionOptionsEthereumTx` containing two `MsgEthereumTx` messages, both signed by `A`:
   - Msg 1: `to = B`, `value = 60`, `gasLimit = 21000`, `gasPrice = 1` → cost = 60 + 21000
   - Msg 2: `to = C`, `value = 60`, `gasLimit = 21000`, `gasPrice = 1` → cost = 60 + 21000
   - (Adjust values so each individual cost < 100 but combined > 100.)
4. Submit via `eth_sendRawTransaction` or gRPC `BroadcastTx`.
5. Observe `CheckTx` returns OK — the transaction is admitted to the mempool.
6. Wait for the transaction to be included in a block; observe that Msg 2 fails with an EVM insufficient-funds error while Msg 1 succeeds, confirming the ante handler admitted a transaction whose aggregate cost exceeded the sender's balance. [1](#0-0) [3](#0-2) [2](#0-1)

func CheckSenderBalance(
	balance sdkmath.Int,
	tx *ethtypes.Transaction,
) error {
	cost := tx.Cost()

	if cost.Sign() < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInvalidCoins,
			"tx cost (%s) is negative and invalid", cost,
		)
	}

	if balance.IsNegative() || balance.BigInt().Cmp(cost) < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInsufficientFunds,
			"sender balance < tx cost (%s < %s)", balance, tx.Cost(),
		)
	}
	return nil
