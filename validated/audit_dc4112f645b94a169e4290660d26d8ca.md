### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall in the StarkNet OS does not validate that the supplied class hash corresponds to a declared contract class before immediately committing the state change. This is the direct analog to the external report's vulnerability class: an **immediate, unvalidated parameter change** that creates an irreversible harmful state — here, permanently freezing any funds held by the affected contract.

### Finding Description
In `syscall_impls.cairo`, `execute_replace_class` reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no on-chain existence check. The code itself contains an explicit TODO acknowledging the missing guard:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The state change is **immediate and unconditional** — exactly the pattern from the external report. When the OS subsequently tries to execute any call to this contract, it resolves the class hash through `execute_entry_point.cairo`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the stored class hash is undeclared, `compiled_class_hash` resolves to 0 (the default dict value), `find_element` finds no matching fact, and every future call to the contract reverts with `ENTRY_POINT_NOT_FOUND`. The contract's storage — including all token balances or ETH equivalents — becomes permanently inaccessible.

### Impact Explanation
**Permanent freezing of funds.** Any contract holding user assets (vault, ERC-20 token, DeFi protocol) can have its class replaced with an undeclared hash. Once replaced, all entry points become unreachable and all assets stored in the contract's storage are permanently frozen with no recovery path. This matches the "Permanent freezing of funds" critical impact category.

### Likelihood Explanation
A malicious contract deployer — an explicitly allowed unprivileged attacker type — can deploy a contract that:
1. Accepts user deposits (funds stored in contract storage)
2. Exposes a public function that calls `replace_class` with an arbitrary undeclared hash

After users deposit funds, the deployer (or any caller of the public function) triggers the class replacement. The OS immediately commits the invalid class hash with no validation. All subsequent calls fail. This is a realistic rug-pull or griefing vector. The likelihood is **medium**: it requires users to interact with a malicious contract, but the attack is trivially executable and the damage is irreversible.

### Recommendation
Implement the missing validation before committing the class hash update. Specifically, verify that `contract_class_changes` contains a non-zero compiled class hash for the given `class_hash` value (i.e., the class has been declared). If the class is undeclared, write a failure response and return without modifying state — mirroring how other syscalls handle invalid inputs.

### Proof of Concept
1. Deploy `MaliciousVault` with:
   - `deposit()` — accepts user funds into contract storage
   - `freeze(invalid_hash)` — calls `replace_class(invalid_hash)` where `invalid_hash` is any undeclared felt
2. Users call `deposit()`, transferring funds to `MaliciousVault`
3. Attacker calls `freeze(0xDEAD)` — triggers `execute_replace_class` with `class_hash = 0xDEAD`
4. OS writes `class_hash = 0xDEAD` into `contract_state_changes` with no validation
5. All subsequent calls to `MaliciousVault` resolve `compiled_class_hash = 0`, fail `find_element`, and revert with `ENTRY_POINT_NOT_FOUND`
6. All user funds are permanently frozen

This is the direct analog to the external report: just as `adjustPrice` immediately changes the price parameter without gradual ramping or validation, `execute_replace_class` immediately changes the class hash without validating the new value — creating an irreversible harmful state exploitable by an unprivileged contract deployer.
