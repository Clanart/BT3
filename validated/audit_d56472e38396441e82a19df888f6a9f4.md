### Title
Unauthenticated `ProofManager::set_proof` Allows Pre-Poisoning of Proof Cache, Permanently Bypassing Cryptographic Proof Verification for Any `ProofFacts` Hash — (`crates/apollo_proof_manager/src/proof_manager.rs`)

---

### Summary

`ProofManager::set_proof` accepts arbitrary `(ProofFacts, Proof)` pairs from any caller with no authentication. The proof store uses **first-writer-wins** semantics: once a proof is persisted for a given `facts_hash`, it can never be overwritten. `TransactionConverter::run_proof_verification` skips `starknet_proof_verifier::verify_proof` entirely when `contains_proof` returns `true`. An attacker who can reach the proof manager's remote server (bound to `0.0.0.0` in distributed deployments) can pre-poison the store for any predictable `ProofFacts` hash with an invalid proof blob, permanently disabling cryptographic verification for every future transaction that carries those proof facts.

---

### Finding Description

**Root cause — no access control on `set_proof`:**

`ProofManager::set_proof` performs no caller authentication:

```rust
pub async fn set_proof(
    &self,
    proof_facts: ProofFacts,
    proof: Proof,
) -> Result<(), FsProofStorageError> {
    if self.contains_proof(proof_facts.clone()).await? {
        return Ok(());          // ← first writer wins; subsequent writes silently dropped
    }
    let facts_hash = proof_facts.hash();
    self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
    self.cache.insert(facts_hash, proof);
    Ok(())
}
``` [1](#0-0) 

The `ComponentRequestHandler` implementation dispatches `SetProof` directly to this function with no identity check: [2](#0-1) 

In the distributed deployment the proof manager's remote server binds to `0.0.0.0`, making it reachable from any host on the same network segment.

**First-writer-wins in `FsProofStorage::write_proof_atomically`:**

The filesystem layer uses an atomic rename. The comment explicitly states the assumption that "proofs are deterministic for a given facts_hash" — an assumption that is false when the store is unauthenticated:

```rust
// Since proofs are deterministic for a given facts_hash, the existing proof
// is identical and we can safely treat this as success.
``` [3](#0-2) 

Once a fake proof is written, no subsequent call to `set_proof` can replace it because `contains_proof` returns `true` and the function returns early.

**Verification skip in `run_proof_verification`:**

Both the gateway flow (`spawn_proof_verification`) and the consensus flow (`spawn_verify_and_store_proof`) share this logic:

```rust
let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

if contains_proof {
    return Ok(false);   // ← cryptographic verification permanently skipped
}

tokio::task::spawn_blocking(move || {
    starknet_proof_verifier::verify_proof(proof_facts, proof)
})
``` [4](#0-3) 

**Fake proof propagated to peers:**

`convert_internal_rpc_tx_to_rpc_tx` retrieves the stored proof from the proof manager and embeds it in the outgoing RPC transaction that is broadcast to peers: [5](#0-4) 

---

### Impact Explanation

An attacker pre-stores an invalid proof blob for a chosen `ProofFacts` hash. Every subsequent Invoke V3 transaction that carries those proof facts is admitted by the gateway and the consensus converter **without any cryptographic proof verification**. The fake proof bytes are then propagated to peer nodes, which will reject the transaction when they attempt their own verification, causing consensus disruption and potential transaction loss. The blockifier's `validate_proof_facts` only checks the public inputs (block hash, program hash, config hash) — it never inspects the proof bytes — so it provides no compensating control.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The proof manager's remote server is bound to `0.0.0.0` in the distributed deployment configuration. Any process on the same network (or a compromised internal node) can issue a `SetProof` RPC call. The `ProofFacts` structure is deterministic and predictable from public transaction data. The attack requires only a single unauthenticated write before the legitimate proof arrives — a direct analog to the frontrunning pattern in the original report.

---

### Recommendation

1. **Authenticate callers of `SetProof`**: Only the gateway and consensus converter should be permitted to write proofs. Add a caller-identity check (e.g., a shared secret, mTLS, or an allowlist of source addresses) to the `RemoteComponentServer` handling `ProofManagerRequest::SetProof`.

2. **Verify before storing, not after**: Move `starknet_proof_verifier::verify_proof` to execute *before* `set_proof` is called, so the proof manager only ever stores cryptographically valid proofs. The current optimization (skip verification if already stored) is only safe when the store is write-protected.

3. **Remove the false determinism assumption**: The comment in `write_proof_atomically` ("proofs are deterministic for a given facts_hash") is only true in a trusted environment. Document or enforce the invariant that only verified proofs may be written.

---

### Proof of Concept

```
# 1. Attacker observes a pending Invoke V3 tx with known proof_facts
#    (e.g., from the public mempool or by constructing valid SNOS fields
#    for a known block hash/number).

# 2. Attacker sends SetProof to the unauthenticated proof manager remote server
#    with the same proof_facts but an all-zero (invalid) proof blob.
grpcurl -plaintext <proof_manager_host>:<port> \
  ProofManagerService/SetProof \
  '{"proof_facts": [...], "proof": "AAAA..."}'

# 3. ProofManager stores the fake proof (first writer wins).
#    contains_proof(proof_facts) now returns true.

# 4. Legitimate Invoke V3 tx with those proof_facts arrives at the gateway.
#    run_proof_verification:
#      contains_proof → true → return Ok(false)   ← verify_proof never called

# 5. Gateway accepts the transaction. convert_internal_rpc_tx_to_rpc_tx
#    retrieves the fake proof from the proof manager and embeds it in the
#    outgoing RPC transaction broadcast to peers.

# 6. Peers receive the transaction, call verify_proof on the fake bytes → fail.
#    Transaction is rejected at peers; consensus is disrupted.
```

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }
