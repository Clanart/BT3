Audit Report

## Title
Unpartitioned `address(this).balance` in `pay()` allows stranded ETH to be consumed by any subsequent WETH-input swap — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` helper in `PeripheryPayments` uses `address(this).balance` — the router's entire, unpartitioned native ETH balance — when settling WETH payments. Because no per-call ETH budget is tracked, ETH left in the router from a prior user's transaction (e.g., excess ETH sent with a WETH swap before `refundETH` is called) is indistinguishable from ETH legitimately sent by the current caller. An attacker can call `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`, causing the router to wrap and forward the victim's stranded ETH to settle the attacker's swap at zero cost.

## Finding Description

**Root cause — `pay` reads the global ETH balance without per-call partitioning:**

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value); // payer is never touched
    } ...
}
``` [1](#0-0) 

When `nativeBalance >= value`, the `payer` argument is completely ignored — no `safeTransferFrom` is issued. Any ETH already sitting in the router satisfies the payment.

**ETH stranding is a normal operational condition.** Every swap entry point is `payable`: [2](#0-1) 

The `receive()` guard only blocks bare ETH transfers (no calldata); it does **not** block ETH attached to `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `unwrapWETH9`, `sweepToken`, `refundETH`, or any `selfPermit*` variant: [3](#0-2) 

Users routinely overpay ETH with WETH swaps and rely on a subsequent `refundETH` call to recover the surplus. `refundETH` itself sends the entire balance to `msg.sender` with no per-user accounting: [4](#0-3) 

**Transient storage tracks callback context but NOT per-call ETH contribution.** `TransientCallbackPool` stores pool address, callback mode, payer, and token-to-pay across four transient slots, but there is no slot recording `msg.value` at entry: [5](#0-4) 

The callback path that invokes `pay` is `_justPayCallback`, which reads `_getPayer()` (the attacker's address) and `_getTokenToPay()` (WETH) from transient storage, then calls `pay`. Because `address(this).balance` covers the required amount, `payer` is never consulted: [6](#0-5) 

The same vulnerability exists in `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback` when token0 or token1 is WETH: [7](#0-6) 

## Impact Explanation

Direct loss of user principal (native ETH). The victim loses ETH they sent to the router; the attacker receives a WETH swap settled entirely at the victim's expense. The loss is bounded only by the amount of ETH stranded in the router at the time of the attack, which can equal the full `amountIn` of a prior WETH swap — an unbounded, user-controlled amount. This meets the Critical/High threshold of direct loss of user principal.

## Likelihood Explanation

ETH stranding is a **normal operational condition**: users routinely send excess ETH with WETH swaps and rely on a subsequent `refundETH` call to recover the surplus. Any window between the swap and the `refundETH` call — including separate transactions, failed multicall legs, or simple user error — leaves ETH exploitable. An attacker can monitor the router's ETH balance on-chain and front-run the `refundETH` call with a zero-cost WETH swap. No special privileges are required; any unprivileged trader can execute this.

## Recommendation

Track per-call ETH contributions in transient storage (EIP-1153, already used throughout this codebase via `TransientCallbackPool`). At the start of each `payable` entry point, record `msg.value` in a dedicated transient slot (e.g., `T_MSG_VALUE_SLOT`). In `pay`, consume only up to the recorded per-call ETH budget — not the entire `address(this).balance`. Clear the slot after the call. This ensures ETH from one user's call cannot be consumed by another user's callback.

## Proof of Concept

1. **User A** calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18, ...})` sending `1.5e18` ETH as `msg.value`.
2. The pool callback fires; `pay(WETH, userA, pool, 1e18)` is called. `address(this).balance == 1.5e18 >= 1e18`, so the router wraps 1 ETH and sends WETH to the pool. Swap succeeds. **0.5e18 ETH remains in the router.**
3. User A submits a `refundETH()` transaction to recover the 0.5 ETH.
4. **Attacker** observes the pending `refundETH` in the mempool and front-runs it with `exactInputSingle({tokenIn: WETH, amountIn: 0.5e18, ...})` sending **0 ETH**.
5. The pool callback fires; `pay(WETH, attacker, pool, 0.5e18)` is called. `address(this).balance == 0.5e18 >= 0.5e18`, so the router wraps User A's 0.5 ETH and sends WETH to the pool. **Attacker's swap settles at zero cost.**
6. User A's `refundETH()` executes: `address(this).balance == 0`, nothing is returned. **User A loses 0.5 ETH.**
