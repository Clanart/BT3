### Title
Missing Declared Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts an arbitrary `class_hash` from the contract request and writes it directly to the contract state without verifying that the hash corresponds to a declared contract class. A self-acknowledged TODO comment in the code confirms this check is absent. Any contract can call `replace_class` with an undeclared class hash, permanently rendering itself uncallable and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS reads `request.class_hash` and immediately writes it into the contract's `StateEntry` via `dict_update`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

There is no `assert_not_zero`, no lookup into `contract_class_changes`, and no membership check against `compiled_class_facts`. The TODO at line 898 explicitly acknowledges the missing precondition.

When a future call is made to the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If `class_hash` was replaced with an undeclared value, `dict_read` returns 0 (the default for an absent key), and `find_element` with `key=0` will fail to locate any compiled class fact, causing an OS-level assertion failure. Because the OS is the prover-side authority, this makes the affected contract permanently uncallable at the protocol level.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared hash and the block is committed, there is no recovery path:

- The contract cannot be called (the OS cannot resolve its class).
- `replace_class` cannot be called again to fix it (calling the contract is itself broken).
- Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible.

This matches the allowed critical impact: *Permanent freezing of funds*.

---

### Likelihood Explanation

**Medium.**

The entry path is reachable by any unprivileged contract deployer:

1. A user deploys a contract whose logic calls `replace_class(arbitrary_undeclared_felt)`.
2. The OS processes the syscall without validating the hash.
3. The committed state now contains an invalid class hash for that contract.

This can occur through:
- A deliberate griefing attack (attacker deploys a contract, collects user funds, then calls `replace_class(0)` to freeze them).
- An accidental bug in an upgradeable contract that passes the wrong hash to `replace_class`.

The blockifier may independently reject such transactions, but the OS — the authoritative prover — does not enforce this invariant, leaving the protocol unsound at the proof layer.

---

### Recommendation

Add an explicit precondition check in `execute_replace_class` before writing the new class hash to state. Specifically, verify that `request.class_hash` has a non-zero entry in `contract_class_changes` (i.e., it has been declared in the current or a prior block):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(
    key=request.class_hash
);
assert_not_zero(compiled_class_hash);
```

This mirrors the check already performed in `execute_entry_point` and closes the gap acknowledged by the TODO comment.

---

### Proof of Concept

1. **Deploy** a contract `VaultWithTrap` that:
   - Accepts ETH/token deposits from users.
   - Exposes an `owner_upgrade(new_hash: felt)` function that calls `replace_class(new_hash)`.

2. **Collect funds**: Users deposit assets into `VaultWithTrap`.

3. **Trigger**: The owner (or a bug in the contract) calls `owner_upgrade(0x1337DEADBEEF)` where `0x1337DEADBEEF` is not a declared class hash.

4. **OS processing**: `execute_replace_class` writes `class_hash=0x1337DEADBEEF` into the contract's `StateEntry` with no validation. [3](#0-2) 

5. **Block committed**: The state with the invalid class hash is finalized.

6. **Freeze**: Any subsequent call to `VaultWithTrap` causes `execute_entry_point` to call `dict_read(key=0x1337DEADBEEF)` → returns 0 → `find_element(key=0)` → OS assertion failure. [2](#0-1) 

7. **Result**: All deposited funds are permanently frozen. No withdrawal, transfer, or upgrade is possible.
