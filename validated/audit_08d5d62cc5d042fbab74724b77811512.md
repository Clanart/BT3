### Title
Unconstrained Hint-Supplied `is_reverted` Flag Allows Sequencer to Skip Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`, `execution/transaction_impls.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a complete no-op. The `is_reverted` flag controlling whether a transaction's `__execute__` entry point runs is set exclusively by a prover hint (`%{ IsReverted %}`) with zero Cairo constraints enforcing its correctness. A malicious prover can set `is_reverted = TRUE` for any valid invoke transaction, causing execution to be silently skipped while the fee charge still executes — a direct loss of user funds provable on L1.

---

### Finding Description

In `execution_constraints.cairo`, the function that is supposed to validate the pre-execution revert flag is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

It accepts the flag and immediately returns — no assertion, no range check, no state comparison. [1](#0-0) 

In `execute_invoke_function_transaction`, the flag is loaded from a hint and passed to this no-op:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // skip execution entirely
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}
``` [2](#0-1) 

Immediately after this branch, `charge_fee` is called unconditionally regardless of `is_reverted`:

```cairo
// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [3](#0-2) 

The `charge_fee` function computes `max_possible_fee` from the transaction's resource bounds and executes an ERC20 transfer to the sequencer address — this transfer runs through `non_reverting_select_execute_entry_point_func` and its state changes are committed. [4](#0-3) 

Note that `run_validate` (signature verification) runs before the `is_reverted` branch, so the transaction is cryptographically valid — only execution is suppressed. [5](#0-4) 

The same pattern applies to `execute_l1_handler_transaction`, where `is_reverted` is hint-supplied and unconstrained, and an early return skips `consume_l1_to_l2_message`, leaving L1→L2 message funds in limbo: [6](#0-5) 

---

### Impact Explanation

**Direct loss of funds (Critical):** A malicious prover sets `is_reverted = TRUE` for any valid invoke transaction. The STARK proof is valid — all Cairo constraints are satisfied because `check_is_reverted` imposes none. The L1 verifier accepts the proof. The block is finalized with the user's fee transferred to the sequencer but the user's intended state changes (token transfers, contract interactions, etc.) never applied. The user permanently loses the fee with no recourse.

**Permanent freezing of funds (Critical):** For L1 handler transactions, setting `is_reverted = TRUE` causes the function to return before `consume_l1_to_l2_message` is called. The L1→L2 message is never recorded in the OS output. Funds locked on L1 as part of the message deposit are never released to L2, and the message cannot be re-executed (it was included in a finalized block). This constitutes permanent fund freezing.

---

### Likelihood Explanation

The StarkNet OS exists precisely to make the sequencer trustless — the proof is supposed to cryptographically enforce correct execution. The empty `check_is_reverted` function is a direct failure of this guarantee. Any sequencer operator who controls the prover process can exploit this without any additional access, leaked keys, or external dependencies. The exploit requires only: (1) running the sequencer, (2) setting the `IsReverted` hint to `TRUE` for targeted transactions. The resulting proof is indistinguishable from an honest proof to the L1 verifier.

---

### Recommendation

The `check_is_reverted` function must enforce that the hint-supplied `is_reverted` value is consistent with verifiable on-chain state. Concretely:

1. **For the pre-execution revert path:** Remove the pre-execution `is_reverted` hint entirely. Instead, always attempt execution and derive revert status from the actual entry point return value (`failure_flag`), as is already done inside `execute_entry_point`. The sequencer optimization of skipping execution for known-reverted transactions must be replaced with a provable mechanism.

2. **If a pre-execution skip is required for performance:** The OS must add Cairo constraints that prove the transaction *would* revert — for example, by running validation and asserting it fails, or by checking a committed revert reason against state. The hint alone is insufficient.

3. **For L1 handlers:** Apply the same fix. The `is_reverted` flag for L1 handlers must be constrained or eliminated.

---

### Proof of Concept

1. User submits a valid invoke transaction (e.g., ERC-20 transfer of 1000 STRK). Transaction passes `run_validate` (signature is valid, nonce is correct).
2. Malicious sequencer sets the `IsReverted` hint to a non-zero value for this transaction in the OS execution trace.
3. The OS executes `check_is_reverted(is_reverted=1)` — the function returns immediately with no constraint violation.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` (the `__execute__` call) is skipped.
5. `charge_fee` executes normally, transferring the user's max fee to the sequencer address via ERC-20 transfer.
6. The STARK proof is generated and verified on L1. The L1 verifier accepts it — all constraints hold.
7. The block is finalized. The user's 1000 STRK transfer never happened. The user's fee is gone. The sequencer collected the fee for zero work.
