### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Enforcement (`nowrap=false` Path) — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

In `check_time_locks`, when the `nowrap` flag is `false` (the legacy pre-fork code path), relative timelock comparisons use `wrapping_add` instead of `saturating_add`. When a coin's `confirmed_block_index + height_relative` (u32) or `timestamp + seconds_relative` (u64) overflows, the result wraps to a small value. The subsequent less-than comparison then evaluates to `false`, causing the timelock check to silently pass — allowing a coin protected by `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` to be spent immediately after confirmation, bypassing the intended lock entirely.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` contains two branches for each relative timelock condition, selected by the `nowrap: bool` parameter:

- `nowrap=true` (safe path): uses `saturating_add`, which clamps at the type maximum, so the comparison always reflects the intended semantics.
- `nowrap=false` (legacy path): uses `wrapping_add`, which silently wraps on overflow.

For `ASSERT_HEIGHT_RELATIVE` under `nowrap=false`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

If `confirmed_block_index + height_relative` overflows `u32`, the result wraps to a small value (possibly 0 or 9). The check `prev_height < small_value` evaluates to `false`, so no error is returned and the timelock is considered satisfied — even though the required height has not been reached.

The same flaw applies to `ASSERT_SECONDS_RELATIVE` with `u64` wrapping:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

The conditions parser in `conditions.rs` accepts `ASSERT_HEIGHT_RELATIVE` values up to `u32::MAX` (`0xFFFFFFFF`) and `ASSERT_SECONDS_RELATIVE` values up to `u64::MAX` (`0xFFFFFFFFFFFFFFFF`); values exceeding the type width are rejected at parse time with `AssertHeightRelativeFailed` / `AssertSecondsRelativeFailed`. This means the maximum valid values are exactly those that trigger overflow when added to any non-zero `confirmed_block_index` or `timestamp`. [3](#0-2) 

The inverse direction — `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE` — is also affected: overflow wraps to a small value, causing `prev_height >= small_value` to evaluate to `true`, permanently blocking a coin from being spent even though the before-height has not been reached. [4](#0-3) 

The wrapping behavior is explicitly documented and tested in the test suite, confirming it is a known divergence between the two paths: [5](#0-4) 

---

### Impact Explanation

**High.** An attacker who controls a coin's puzzle can embed `ASSERT_HEIGHT_RELATIVE(r)` where `r` is chosen so that `confirmed_block_index + r` overflows `u32`. Under `nowrap=false`, the timelock check passes immediately after the coin is confirmed, regardless of the actual block height. This constitutes a **timelock validation bypass enabling unauthorized spend acceptance**: a coin that was intended to be locked for ~4 billion blocks (or effectively forever) can be spent in the very next block.

The same applies to `ASSERT_SECONDS_RELATIVE` with a `u64` overflow, bypassing time-based locks.

---

### Likelihood Explanation

The `nowrap=false` path is the legacy behavior preserved for backward compatibility with pre-fork blocks. The `nowrap` parameter is passed from the Python full node layer (exposed via the `py_check_time_locks` Python binding). As long as any block height range in the active chain uses `nowrap=false`, an attacker can craft a spend bundle with an overflowing relative timelock and have it accepted by nodes running that code path. The attacker-controlled entry is a standard unprivileged CLVM puzzle — no special privileges, keys, or network position are required. [6](#0-5) 

---

### Recommendation

1. **Remove the `nowrap=false` branch entirely** once the soft fork that mandates `saturating_add` semantics has activated on all supported networks. Until then, gate the `nowrap=false` path behind an explicit height check so it is only reachable for blocks below the fork height.
2. **Add a parser-level guard**: reject `ASSERT_HEIGHT_RELATIVE` values where `confirmed_block_index + height_relative` would overflow, or clamp `height_relative` to `u32::MAX - confirmed_block_index` at parse time.
3. **Audit all callers** of `py_check_time_locks` in the Python full node to confirm `nowrap=true` is passed for all blocks at or above the fork activation height.

### Proof of Concept

**Setup**: Coin confirmed at block height `10` (u32). Attacker embeds `ASSERT_HEIGHT_RELATIVE(0xFFFFFFFF)` in the puzzle.

**Arithmetic under `nowrap=false`**:
```
confirmed_block_index = 10
height_relative       = 0xFFFFFFFF
wrapping_add result   = (10 + 0xFFFFFFFF) mod 2^32 = 9
```

**Check**:
```
prev_transaction_block_height (e.g., 15) < 9  →  false  →  no error returned
```

The timelock is bypassed. The coin is spendable at block 11 (or any block ≥ 10), despite the puzzle asserting a relative lock of ~4.3 billion blocks.

This is confirmed by the existing test vector:

```python
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok (no error = bypass)
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
#                                               ^    ^
#                              nowrap=True: Err  nowrap=False: None (passes!)
``` [7](#0-6)
