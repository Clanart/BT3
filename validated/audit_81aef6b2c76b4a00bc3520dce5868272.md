### Title
Missing `shard_id` Binding in `set_state_header` Allows Cross-Shard State Substitution During State Sync - (File: chain/chain/src/state_sync/adapter.rs)

### Summary

`set_state_header` in `ChainStateSyncAdapter` accepts a `ShardStateSyncResponseHeader` from a peer and validates that the embedded chunk is included in the sync block via a merkle proof, but never verifies that the chunk's `shard_id` matches the requested `shard_id`. A malicious peer can supply a valid chunk for shard Y (with a genuine merkle proof) in response to a state sync request for shard X. The header passes all checks and is stored under `StateHeaderKey(shard_id=X, sync_hash)`. Subsequent `set_state_part` and `apply_state_part` calls then install shard Y's trie under shard X's `ShardUId`, corrupting the syncing node's state for shard X.

### Finding Description

In `set_state_header`, the function receives a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader`. The validation sequence is:

1. `validate_chunk_proofs(&chunk, ...)` — validates the chunk's internal merkle proofs (tx root, receipts root). Does not check `shard_id`.
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk hash is a leaf in the block's chunk-headers merkle tree. The proof path encodes the leaf's position (shard Y's index), not the requested shard X's index. The check passes for any valid chunk in the block. [1](#0-0) 

There is no check of the form `chunk.shard_id() != shard_id → reject`. After passing validation, the header is committed: [2](#0-1) 

The key is stored as `StateHeaderKey(shard_id=X, sync_hash)` regardless of the chunk's actual shard. `set_state_part` then retrieves this header and extracts `state_root` from the chunk (shard Y's state root), validates parts against it, and stores them under `StatePartKey(sync_hash, shard_id=X, part_id)`: [3](#0-2) 

`apply_state_part` is called with the requested `shard_id=X`, so shard Y's trie nodes are written under shard X's `ShardUId`: [4](#0-3) 

Finally, `set_state_finalize` → `chain_update.set_state_finalize` applies shard Y's chunk body (transactions, receipts) to shard X's storage, completing the state substitution: [5](#0-4) 

The `ShardStateSyncResponseHeader` struct carries no `shard_id` field of its own — the binding between the header and the requested shard exists only in the caller-supplied parameter, which is never cross-checked against the embedded chunk: [6](#0-5) 

### Impact Explanation

A syncing validator node (performing catchup for a newly assigned shard X) installs shard Y's full trie state under shard X's `ShardUId`. The node then produces chunks for shard X starting from shard Y's state root. These chunks are invalid and are rejected by all other validators. The affected validator cannot produce valid chunks for shard X for the entire epoch, losing all chunk-production rewards. If the attack is applied to multiple validators simultaneously during a catchup epoch, it can reduce the number of valid chunk producers for shard X below the minimum, stalling that shard's progress.

The corrupted value is the `prev_state_root` committed to `ChunkExtra` for shard X — it becomes shard Y's state root, permanently diverging from the canonical chain state for shard X on this node.

### Likelihood Explanation

Any network peer can respond to a state sync header request. The syncing node contacts peers and accepts the first valid response. An attacker operating a reachable peer node that tracks shard Y can intercept or race the legitimate response. The attacker needs only:
- A valid `ShardChunk` for shard Y (available to any node tracking shard Y)
- A valid merkle proof for shard Y's chunk in the sync block (computable from the block)
- Valid state parts for shard Y's state root (available to any node that has shard Y's state)

No privileged role is required. The attack is fully constructible by an unprivileged peer.

### Recommendation

In `set_state_header`, after extracting the chunk from the header, add an explicit shard binding check before the merkle proof verification:

```rust
let chunk = shard_state_header.cloned_chunk();
// Bind the chunk to the requested shard_id
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the binding check already present in `compute_state_response_header` (server side), which validates `shard_id` against the epoch's shard layout before selecting the chunk: [7](#0-6) 

### Proof of Concept

1. Attacker runs a peer node tracking shard Y (shard_id = 1) for the current epoch.
2. Victim node begins state sync for shard X (shard_id = 0) and sends a `StateRequestHeader { shard_id: 0, sync_hash }` to the attacker's peer.
3. Attacker constructs a `ShardStateSyncResponseHeader` where:
   - `chunk` = the attacker's locally stored `ShardChunk` for shard Y at `sync_prev_block`
   - `chunk_proof` = the merkle proof for shard Y's chunk at shard Y's index in `sync_prev_block.chunk_headers_root`
   - `state_root_node` = shard Y's `StateRootNode`
   - `incoming_receipts_proofs` / `root_proofs` = valid proofs for shard Y's receipts
4. Victim calls `set_state_header(shard_id=0, sync_hash, crafted_header)`. The `verify_path` check passes (shard Y's chunk is genuinely in the block). No `chunk.shard_id() == shard_id` check exists. Header stored under `StateHeaderKey(0, sync_hash)`.
5. Victim downloads state parts. `set_state_part` extracts `state_root` from the stored header (shard Y's state root) and validates parts against it. Attacker provides valid parts for shard Y's state root. Parts stored under `StatePartKey(sync_hash, 0, part_id)`.
6. `apply_state_part(shard_id=0, state_root=shard_Y_root, ...)` writes shard Y's trie nodes under shard X's `ShardUId`.
7. `set_state_finalize(shard_id=0, sync_hash)` applies shard Y's chunk body to shard X's storage. Victim's shard 0 state is now shard 1's state.
8. Victim produces chunks for shard 0 from shard 1's state root — all rejected by the network.

pub struct ShardStateSyncResponseHeaderV2 {
    /// The chunk whose header in included as B_prev.chunks[shard_id]
    /// This chunk will be applied after downloading state
    pub chunk: ShardChunk,
    /// A merkle path for (Self::chunk.hash, Self::chunk.height_included), verifiable
    /// against B_prev.chunk_headers_root
    pub chunk_proof: MerklePath,
    /// This is None if sync_hash is the genesis hash. Otherwise, it's B_prev_chunk.chunks[shard_id]
    pub prev_chunk_header: Option<ShardChunkHeader>,
    /// A merkle path for (Self::prev_chunk_header.hash, Self::prev_chunk_header.height_included), verifiable
    /// against B_prev_chunk.chunk_headers_root
    pub prev_chunk_proof: Option<MerklePath>,
    /// This field contains the incoming receipts for shard_id for B_sync and B_prev_chunk.
    /// So, this field has at most two elements.
    /// These receipts are used to apply `chunk` after downloading state
    pub incoming_receipts_proofs: Vec<ReceiptProofResponse>,
    /// This field contains the info necessary to verify that the receipt proofs in Self::incoming_receipts_proofs
    /// are actually the ones referenced on chain
    ///
    /// The length of this field is the same as the length of Self::incoming_receipts_proofs, and elements
    /// of the two at a given index are taken together for verification. For a given index i,
    /// root_proofs[i] is a vector of the same length as incoming_receipts_proofs[i].1 , which itself is a
    /// vector of receipt proofs for all "from_shard_ids" that sent receipts to shard_id. root_proofs[i][j]
    /// contains a merkle root equal to the prev_outgoing_receipts_root field of the corresponding chunk
    /// included in the block with hash incoming_receipts_proofs[i].0, and a merkle path to verify it against
    /// that block's prev_chunk_outgoing_receipts_root field.
    pub root_proofs: Vec<Vec<RootProof>>,
    /// The state root with hash equal to B_prev.chunks[shard_id].prev_state_root.
    /// That is, the state root node of the trie before applying the chunks in B_prev
    pub state_root_node: StateRootNode,
}
