### Title
Non-Atomic Burn-and-Mint in `swap_migrated_token` Allows Permanent Token Loss — (`near/omni-bridge/src/lib.rs`)

### Summary

`swap_migrated_token` constructs `burn.and(mint)` and immediately `.detach()`s the combined promise with no failure callback. In NEAR's async model, `Promise::and()` schedules two independent cross-contract calls — they are **not atomic**. If `mint` panics (e.g., because the recipient has no storage registration on the new token), the `burn` has already executed and cannot be reverted. The user's old tokens are permanently destroyed with no new tokens minted and no refund path.

---

### Finding Description

**Entry point** — `ft_on_transfer` in `near/omni-bridge/src/lib.rs`:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
``` [1](#0-0) 

Returning `U128(0)` tells the old token contract to keep all transferred tokens at the bridge (no refund). The promise is `.detach()`ed — no callback is ever registered.

**The non-atomic promise** — `swap_migrated_token`:

```rust
fn swap_migrated_token(&mut self, sender_id: AccountId, old_token: AccountId, amount: U128) -> Promise {
    let new_token = self.migrated_tokens.get(&old_token).near_expect(BridgeError::TokenNotMigrated);
    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
    burn.and(mint)
}
``` [2](#0-1) 

`burn.and(mint)` in NEAR SDK schedules two **independent** cross-contract calls. They execute in separate receipts. If `mint` panics, `burn` is **not** rolled back — NEAR has no cross-contract atomicity.

**Why `mint` can panic** — `omni-token/src/lib.rs`:

```rust
fn mint(&mut self, account_id: AccountId, amount: U128, msg: Option<String>) -> PromiseOrValue<U128> {
    self.assert_controller();
    // msg is None in swap_migrated_token
    self.token.internal_deposit(&account_id, amount.into());
    PromiseOrValue::Value(amount)
}
``` [3](#0-2) 

`internal_deposit` is the standard NEAR FT implementation, which **panics** if `account_id` has no storage registration on the new token. No pre-check for storage registration is performed before the burn.

**`migrate_deployed_token` only registers the bridge itself**, not users:

```rust
ext_token::ext(new_token.clone())
    .with_static_gas(STORAGE_DEPOSIT_GAS)
    .with_attached_deposit(NEP141_DEPOSIT)
    .storage_deposit(&env::current_account_id(), Some(true))
    .detach();
``` [4](#0-3) 

---

### Impact Explanation

Any user who calls `old_token.ft_transfer_call(bridge, amount, "SwapMigratedToken")` without having previously registered storage on the new token will have their old tokens permanently burned with zero new tokens minted. There is no callback, no refund path, and no recovery mechanism. This constitutes **irrecoverable loss of user funds** — matching the Critical impact category: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds*.

---

### Likelihood Explanation

This is reachable by any unprivileged user. The `SwapMigratedToken` message variant is public and requires no special role. Storage registration on the new token is a separate, non-obvious prerequisite that users are not forced to complete before initiating the swap. The failure mode is silent (no event, no revert visible to the user at the `ft_transfer_call` level).

---

### Recommendation

Replace the fire-and-forget `burn.and(mint).detach()` pattern with a sequenced callback:

1. **Burn first**, then in the burn callback, **check success** before calling mint.
2. Alternatively, **mint first** (which can fail safely before any burn), then burn in the mint callback.
3. Add a **failure callback** on the combined promise that re-mints old tokens to the sender if mint fails.
4. Optionally, check `new_token.storage_balance_of(sender_id)` before proceeding and panic early (before burn) if storage is absent.

---

### Proof of Concept

```
1. DAO calls migrate_deployed_token(Near, old_token, new_token)
   → migrated_tokens[old_token] = new_token
   → bridge registers itself on new_token, but NOT the user

2. User (alice.near) has NOT called new_token.storage_deposit(alice.near)

3. Alice calls:
   old_token.ft_transfer_call(bridge, 1000, None, '"SwapMigratedToken"')

4. bridge.ft_on_transfer(alice, 1000, "SwapMigratedToken") executes:
   → swap_migrated_token creates burn.and(mint).detach()
   → returns U128(0) → old_token keeps 1000 at bridge (no refund)

5. Receipt 1: old_token.burn(1000) called by bridge
   → bridge's balance of old_token: 1000 → 0  ✓ (succeeds)

6. Receipt 2: new_token.mint(alice, 1000, None) called by bridge
   → internal_deposit(alice, 1000) panics: "The account alice.near is not registered"
   → mint receipt fails ✗

7. No callback exists to detect the failure.

Result: Alice's 1000 old tokens are burned. 0 new tokens minted. Funds permanently lost.
Assert: old_token.ft_balance_of(bridge) == 0, new_token.ft_balance_of(alice) == 0.
```
