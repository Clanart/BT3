### Title
Deploy Syscall Always Reports Success Regardless of Constructor Failure — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in `syscall_impls.cairo` unconditionally writes `failure_flag=0` in the syscall response header after calling `deploy_contract`, regardless of whether the constructor execution actually succeeded or failed. This is the direct StarkNet analog of the THORChain bug: just as THORChain emits `TransferOut` with the original `to` address even when the ETH transfer bounced back, the StarkNet OS always tells the calling contract that a deployment succeeded even when the constructor reverted.

---

### Finding Description

In `execute_deploy` (lines 527–553 of `syscall_impls.cairo`), after invoking `deploy_contract`, the OS writes the syscall response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The `deploy_contract` call returns `(retdata_size, retdata)`, which can carry constructor revert data. However, neither the return values nor any revert status is consulted when writing the response. The embedded TODO comment — `"support failures"` — is an explicit developer acknowledgment that constructor failures are not handled.

Contrast this with `contract_call_helper` (lines 404–448 of the same file), which correctly propagates the `is_reverted` flag from `select_execute_entry_point_func` into the `ResponseHeader`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

The `execute_deploy` path has no equivalent check.

The vulnerability class is identical to the THORChain H-02 report: **an output/event is committed with a "success" status even when the underlying operation failed**, causing the consumer of that output to act on incorrect information.

---

### Impact Explanation

When a contract calls the `deploy` syscall and the constructor reverts (e.g., out-of-gas, failed assertion, missing entry point), the OS still writes `failure_flag=0` to the response. The calling contract's Sierra/Cairo code reads this flag to decide whether the deployment succeeded. Because the flag is always 0, the calling contract:

1. Believes the deployment succeeded.
2. May record the returned `contract_address` as a valid, initialized contract.
3. May subsequently transfer ERC-20 tokens (STRK, ETH, or any token) to that address, expecting a functioning contract.

If the constructor's state changes were reverted (class_hash assignment rolled back via the revert log), the target address has no associated class. Tokens transferred to a zero-class address via the ERC-20 `transfer` syscall are recorded in the token contract's storage as belonging to that address, but no contract exists at that address to spend them. The result is **permanent freezing of funds** — matching the Critical impact tier.

---

### Likelihood Explanation

The trigger path is fully unprivileged:

- Any user can deploy a contract whose constructor can fail (e.g., a constructor that asserts a condition on its calldata, or one that runs out of the gas budget).
- Any contract that calls `deploy` and then acts on the returned address (a common pattern for factory contracts) is affected.
- The constructor failure can be induced by the deployer themselves by crafting calldata that causes the constructor to revert, or it can occur naturally due to gas exhaustion.

No privileged role, leaked key, or operator cooperation is required.

---

### Recommendation

Mirror the pattern used in `contract_call_helper`. Capture the revert status from `deploy_contract` and propagate it into the response header:

```cairo
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

let response_header = cast(syscall_ptr, ResponseHeader*);
let syscall_ptr = syscall_ptr + ResponseHeader.SIZE;

// Correctly propagate constructor failure.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

`deploy_contract` must also be updated to return an `is_reverted` flag (analogous to `select_execute_entry_point_func`), and the `DeployResponse` body should carry the failure reason when `is_reverted=1`, consistent with how `CallContractResponse` handles failures.

---

### Proof of Concept

1. Declare a class whose constructor always reverts (e.g., `assert 1 = 0`).
2. Deploy a factory contract that calls `deploy(class_hash, ...)` and, on success (`failure_flag == 0`), immediately calls `IERC20(token).transfer(deployed_address, amount)`.
3. Invoke the factory. The OS executes the constructor, which reverts; the revert log rolls back the class_hash assignment. However, `execute_deploy` writes `failure_flag=0`.
4. The factory contract reads `failure_flag=0`, concludes success, and calls `transfer`.
5. The ERC-20 contract records `amount` tokens as belonging to `deployed_address`.
6. No contract exists at `deployed_address` (class_hash = 0 after revert). The tokens are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2)
