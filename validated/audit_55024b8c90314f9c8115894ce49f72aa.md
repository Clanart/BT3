### Title
Missing Recipient Validation in `initTransfer()` Allows Permanent Lock of User Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The `initTransfer()` function in both the EVM and StarkNet Omni Bridge contracts performs no validation that the `recipient` parameter is non-empty. A user who supplies an empty or malformed recipient string will have their tokens burned or locked on the source chain, while the corresponding cross-chain transfer can never be finalized on NEAR because the recipient address is unparseable. There is no on-chain recovery path, resulting in permanent loss of user funds.

---

### Finding Description

**EVM — `OmniBridge.sol` `initTransfer()`**

The function accepts a `string calldata recipient` parameter and performs only two checks before irreversibly consuming user tokens:

```solidity
// OmniBridge.sol lines 382-384
if (fee >= amount) {
    revert InvalidFee();
}
```

No check is made that `recipient` is non-empty or syntactically valid as an `OmniAddress`. Immediately after this guard, tokens are either burned (bridge tokens) or transferred into the contract (native tokens), and the `InitTransfer` event is emitted with the empty recipient string.

```solidity
// OmniBridge.sol lines 404-412
} else if (isBridgeToken[tokenAddress]) {
    BridgeToken(tokenAddress).burn(msg.sender, amount);
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

The emitted event is the only record the NEAR bridge uses to reconstruct the transfer. When the relayer submits the proof to the NEAR bridge's `fin_transfer_callback`, the prover decodes the `recipient` field as an `OmniAddress`. An empty string fails this parse, causing the callback to panic. Because the EVM bridge has no refund or recovery function, the tokens are permanently locked.

**StarkNet — `omni_bridge.cairo` `init_transfer()`**

The same pattern exists. The function checks `amount > 0` and `fee < amount`, but performs no validation on `recipient: ByteArray`:

```cairo
// omni_bridge.cairo lines 290-293
assert(!_is_paused(@self, PAUSE_INIT_TRANSFER), 'ERR_INIT_TRANSFER_PAUSED');
assert(amount > 0, 'ERR_ZERO_AMOUNT');
assert(fee < amount, 'ERR_INVALID_FEE');
```

Tokens are then burned or transferred before the event is emitted, with no recipient check:

```cairo
// omni_bridge.cairo lines 300-307
if self.is_bridge_token(token_address) {
    IBridgeTokenDispatcher { contract_address: token_address }
        .burn(caller, amount.into());
} else {
    let success = IERC20Dispatcher { contract_address: token_address }
        .transfer_from(caller, get_contract_address(), amount.into());
    assert(success, 'ERR_TRANSFER_FROM_FAILED');
}
```

**Contrast with NEAR bridge**

The NEAR bridge's `init_transfer()` explicitly validates the recipient chain before any state mutation:

```rust
// near/omni-bridge/src/lib.rs lines 531-534
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
```

The EVM and StarkNet bridges have no equivalent guard.

**Note on Solana**: The Solana `SECURITY.md` explicitly acknowledges the same missing recipient validation as a known low-severity issue for that chain. No equivalent acknowledgment exists for the EVM or StarkNet bridges.

---

### Impact Explanation

**Severity: High — Permanent freezing / irrecoverable lock of user funds.**

When a user calls `initTransfer` with an empty or malformed `recipient`:

1. Tokens are burned (bridge tokens) or locked in the contract (native/ERC-20 tokens) on the source chain — this is irreversible.
2. The `InitTransfer` event is emitted with the invalid recipient.
3. The NEAR bridge relayer submits the proof; `fin_transfer_callback` attempts to parse the recipient as an `OmniAddress` and panics.
4. Because the NEAR bridge callback fails, no `TransferMessage` is stored and no destination-chain settlement is ever produced.
5. The EVM and StarkNet bridges have no `refund`, `rescue`, or `recover` function. The only recovery path requires privileged admin action (manually minting tokens back), which is not guaranteed and is not part of the protocol's normal operation.

The result is permanent, irrecoverable loss of the bridged amount for the user.

---

### Likelihood Explanation

**Medium.** The `recipient` field is a free-form string parameter with no UI-enforced format. A user interacting directly with the contract (e.g., via a script, a third-party integration, or a wallet that does not validate the field) can trivially supply `""`. Accidental omission of the recipient in programmatic integrations is a realistic scenario. The attack requires no privilege and no special knowledge — any unprivileged bridge user can trigger it on themselves or, if they control the call, on behalf of another address.

---

### Recommendation

Add an explicit non-empty recipient check at the top of `initTransfer()` in both the EVM and StarkNet bridges, before any token transfer or burn occurs:

**EVM (`OmniBridge.sol`)**:
```solidity
require(bytes(recipient).length > 0, "ERR_EMPTY_RECIPIENT");
```

**StarkNet (`omni_bridge.cairo`)**:
```cairo
assert(recipient.len() > 0, 'ERR_EMPTY_RECIPIENT');
```

Optionally, add a lightweight format check (e.g., that the string contains a `:` separator matching the `chain:address` `OmniAddress` format) to catch obviously malformed values before funds are consumed.

---

### Proof of Concept

**EVM scenario:**

1. Alice holds 1000 USDC bridged tokens on Ethereum.
2. Alice (or a buggy integration) calls:
   ```solidity
   OmniBridge.initTransfer(
       usdcBridgeToken,
       1000e6,   // amount
       0,        // fee
       0,        // nativeFee
       "",       // recipient — empty string
       ""        // message
   );
   ```
3. `fee >= amount` check passes (`0 < 1000e6`). No other guard fires.
4. `BridgeToken(usdcBridgeToken).burn(alice, 1000e6)` executes — Alice's tokens are destroyed.
5. `InitTransfer` event is emitted with `recipient = ""`.
6. Relayer submits proof to NEAR bridge `fin_transfer`.
7. NEAR bridge `fin_transfer_callback` decodes the proof; `OmniAddress::from_str("")` fails; callback panics.
8. No `TransferMessage` is stored; no MPC signature is ever produced; no destination-chain settlement occurs.
9. Alice's 1000 USDC is permanently lost with no on-chain recovery path.
