### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key Guard, Corrupting Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces for all items beyond the first two, allowing an attacker-controlled batch to silently insert duplicate `KeyId`/`Hash` entries. The resulting tree is structurally inconsistent: the `block_status_cache` is out of sync with the blob, the computed Merkle root is wrong, and any proof of inclusion or exclusion derived from it is invalid.

### Finding Description

`MerkleBlob::insert` enforces two guards before writing a leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` routes only the **last two** items in the input vector through `insert` (and only when the tree currently has ≤ 1 leaves). Every other item is written directly via `insert_entry_to_blob`, which performs no key-uniqueness or hash-uniqueness check:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?; // guarded
    }
}

for ((key, value), hash) in keys_values_hashes {   // remaining items — no guard
    let new_leaf_index = self.get_new_index();
    // ... build Block ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When the tree already has ≥ 2 leaves the `if` branch is skipped entirely, so **all** items in the batch bypass the guard.

Consequences of inserting a duplicate `KeyId`:
- Two leaf nodes with the same key exist in the blob at different tree indexes.
- `block_status_cache.key_to_index` maps the key to whichever index was written last, orphaning the earlier node.
- `check_integrity` detects the mismatch (`IntegrityKeyToIndexCacheIndex` / `IntegrityKeyToIndexCacheLength`), but integrity checking is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`).
- The internal-node hashes computed during the batch-insert phase (`internal_hash(&hashes[0], &hashes[1])`) incorporate the duplicate leaf hash, producing a root that does not correspond to the intended key-value set. [3](#0-2) 

The Python binding `py_batch_insert` is the direct unprivileged entry point:

```rust
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    // ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

### Impact Explanation

A caller who supplies a batch containing a repeated `KeyId` (or repeated `Hash`) silently corrupts the Merkle tree root. Any `get_proof_of_inclusion` call on the tree thereafter returns a proof that does not match the true committed state, satisfying the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

The Python binding is the primary public API surface for DataLayer operations. Any code path that constructs a batch from external (e.g., network-received) key-value pairs without pre-deduplicating them can trigger the bug. The `test_double_insert_fails` test only exercises the single-item `insert` path, not `batch_insert`. [5](#0-4) 

### Recommendation

Add the same duplicate checks at the top of `batch_insert` (or inside the fast-path loop) before calling `insert_entry_to_blob`:

```rust
for ((key, value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Alternatively, deduplicate the input vector before processing, or extend `insert_entry_to_blob` to update and validate the cache on every write.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "little")).digest()

# Build a batch of 4 items where key 0 appears twice (positions 0 and 2)
keys_values = [(KeyId(0), ValueId(0)),
               (KeyId(1), ValueId(1)),
               (KeyId(0), ValueId(99)),   # duplicate key
               (KeyId(2), ValueId(2))]
hashes = [h(0), h(1), h(100), h(2)]

# This succeeds — no error raised despite the duplicate KeyId(0)
blob.batch_insert(keys_values, hashes)
blob.calculate_lazy_hashes()

# The tree now has an inconsistent root; proof for key 0 reflects
# whichever leaf was written last, not the intended committed state.
proof = blob.get_proof_of_inclusion(KeyId(0))
# proof.valid() may return True but against a corrupted root
```

The first two items are popped from the end of the vector and processed via `insert` (guarded). The remaining two items — including the duplicate `KeyId(0)` — are written directly via `insert_entry_to_blob` with no guard, silently corrupting the tree.
