### Title
Fee-on-Transfer Token Accounting Error in `initTransfer` Enables Unbacked Minting on Destination Chain - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without verifying the actual number of tokens received by the contract. For fee-on-transfer ERC20 tokens, the bridge receives `amount - fee` but emits and proves `amount`, causing the destination chain (NEAR or other) to release more tokens than were ever locked, breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer`, when the token is a plain ERC20 (not a bridge token, not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-controlled parameter
);
``` [1](#0-0) 

Immediately after, the function emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // same caller-controlled parameter, not actual received balance
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The contract never measures the balance delta before and after the `safeTransferFrom`. For a fee-on-transfer ERC20 (a token that deducts a percentage on every transfer), the bridge receives `amount - transferFee` but the event records `amount`.

On the NEAR side, `fin_transfer_callback` reads `init_transfer.amount` directly from the proven event data and uses it to determine how many tokens to release or mint to the recipient:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

The NEAR bridge then calls `send_tokens` with `amount_without_fee()` derived from this inflated `amount`, releasing more tokens than were ever locked on EVM. [4](#0-3) 

---

### Impact Explanation

**Critical — balance/accounting corruption that breaks bridge collateralization.**

For every `initTransfer` call with a fee-on-transfer token:
- EVM bridge holds: `amount - transferFee`
- NEAR bridge releases: `amount`

The bridge is undercollateralized by `transferFee` per transfer. An attacker can repeatedly bridge fee-on-transfer tokens to drain the bridge's reserves on the EVM side relative to the supply minted on NEAR, or to receive more tokens on NEAR than they deposited on EVM. This constitutes unauthorized minting of unbacked bridged assets.

---

### Likelihood Explanation

Fee-on-transfer ERC20 tokens are a well-known token class (e.g., USDT on some chains historically, STA, PAXG, deflationary tokens). The `initTransfer` function has no allowlist or blocklist for token types — any ERC20 whose NEAR token ID has been registered via `logMetadata`/`deployToken` can be used. An unprivileged user only needs to call `initTransfer` with such a token; no special role is required. [5](#0-4) 

---

### Recommendation

Measure the actual received balance using a before/after balance check:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported by adding a registry check or by using `actualReceived` as the bridged amount in the emitted event.

---

### Proof of Concept

1. Deploy or identify a fee-on-transfer ERC20 token `T` with a 5% transfer fee, registered in the Omni Bridge.
2. Attacker calls `initTransfer(T, 1000, 0, 0, "near:attacker.near", "")`.
3. `safeTransferFrom` transfers 1000 from attacker; bridge receives 950 (5% fee deducted by token).
4. `InitTransfer` event is emitted with `amount = 1000`.
5. Relayer submits proof to NEAR `fin_transfer`.
6. `fin_transfer_callback` reads `amount = 1000` from proof, releases 1000 tokens to `attacker.near`.
7. Bridge is undercollateralized: holds 950 on EVM, has issued 1000 on NEAR.
8. Repeat to drain the bridge or accumulate unbacked supply. [6](#0-5) [7](#0-6)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
