### Title
Cross-Chain Replay of Unprotected Legacy Transactions When `allow_unprotected_txs=true` — (`ante/interfaces/setup.go`, `evmd/ante/handler_options.go`, `rpc/backend/call_tx.go`)

### Summary

When `allow_unprotected_txs=true`, a legacy transaction signed with `HomesteadSigner` (no chain-id in the signature) passes all ante-handler and RPC validation on any Ethermint chain with the same setting. Because the signature commits to no chain-id, the identical raw bytes can be broadcast to a second chain and will commit there too, draining the sender's funds on both chains without their consent.

### Finding Description

**Guard 1 — `ValidateEthBasic`** (`ante/interfaces/setup.go:126-130`):

```go
if !allowUnprotectedTxs && !tx.Protected() {
    return errorsmod.Wrapf(...)
}
```

When `allowUnprotectedTxs=true` the entire protection check is skipped. An unprotected tx (V=27/28) proceeds unconditionally. [1](#0-0) 

**Guard 2 — `VerifyEthSig`** (`evmd/ante/handler_options.go:122-124`):

```go
ethSigner := ethtypes.MakeSigner(blockCfg.ChainConfig, blockCfg.BlockNumber, blockCfg.BlockTime)
if err := evmante.VerifyEthSig(tx, ethSigner); err != nil { ... }
```

With `EIP155Block=0`, `MakeSigner` returns an `EIP155Signer`. However, go-ethereum's `EIP155Signer.Sender()` explicitly falls back to `HomesteadSigner{}.Sender(tx)` when `!tx.Protected()`. This means the unprotected tx's sender is recovered correctly and `VerifySender` passes — on **any** chain, regardless of chain-id. [2](#0-1) [3](#0-2) [4](#0-3) 

**Guard 3 — `SendRawTransaction` chain-id check** (`rpc/backend/call_tx.go:154-157`):

```go
// Reject transactions targeting a different chain.
if tx.Protected() && tx.ChainId().Cmp(b.chainID) != 0 {
    return common.Hash{}, fmt.Errorf("incorrect chain-id; ...")
}
```

The chain-id check is **conditional on `tx.Protected()`**. For an unprotected tx, this branch is never entered, so no chain-id validation occurs at the RPC layer either. [5](#0-4) 

The `FromSignedEthereumTx` call immediately after uses `LatestSignerForChainID(b.chainID)`, which again falls back to HomesteadSigner for unprotected txs, so sender recovery succeeds on any chain. [6](#0-5) 

### Impact Explanation

An attacker who holds funds on two Ethermint chains both configured with `allow_unprotected_txs=true` can:
1. Sign a single legacy tx with `HomesteadSigner` (no chain-id).
2. Broadcast it to Chain A — it commits, transferring funds.
3. Broadcast the **identical raw bytes** to Chain B — it commits again, transferring funds from the same address on Chain B without the sender's intent.

This is unauthorized fund transfer via signature/chain-id verification bypass — a High-severity impact under the allowed scope.

### Likelihood Explanation

- `allow_unprotected_txs` defaults to `false` in both the on-chain params (`DefaultAllowUnprotectedTxs`) and the node config, so both chains must have explicitly opted in. [7](#0-6) 
- The nonce must match on the target chain (e.g., fresh accounts, or chains forked from the same state).
- The scenario is realistic for testnets, chains forked from a common genesis, or any operator who enables the flag for compatibility with pre-EIP-155 tooling.

### Recommendation

1. **Remove the HomesteadSigner fallback path for unprotected txs when EIP-155 is active.** When `EIP155Block=0`, `VerifyEthSig` should explicitly reject any tx where `!tx.Protected()`, regardless of `allowUnprotectedTxs`. The `allowUnprotectedTxs` flag should only gate RPC submission, not consensus-level signature verification.
2. **Make the chain-id check in `SendRawTransaction` unconditional** — reject unprotected txs at the RPC layer even when `allowUnprotectedTxs=true`, or at minimum log a clear warning.
3. **Alternatively**, if unprotected txs must be supported, bind them to the chain by including the chain-id in a wrapper layer before committing to state.

### Proof of Concept

```go
// Chain A: chainID=9000, allow_unprotected_txs=true, EIP155Block=0
// Chain B: chainID=9001, allow_unprotected_txs=true, EIP155Block=0

privKey, _ := ethsecp256k1.GenerateKey()
from := common.BytesToAddress(privKey.PubKey().Address())

// Sign with HomesteadSigner — no chain-id in signature
legacyTx := ethtypes.NewTx(&ethtypes.LegacyTx{
    Nonce: 0, To: &victim, Value: big.NewInt(1e18), Gas: 21000, GasPrice: big.NewInt(1e9),
})
signer := ethtypes.HomesteadSigner{}
signed, _ := ethtypes.SignTx(legacyTx, signer, privKey.ToECDSA())
raw, _ := signed.MarshalBinary()

// Submit to Chain A → commits (nonce 0, funds drained)
chainA_rpc.SendRawTransaction(raw)

// Submit SAME raw bytes to Chain B → also commits (same nonce 0, same signature valid)
chainB_rpc.SendRawTransaction(raw)

// Both chains accepted the tx; funds drained on both without sender's intent for Chain B
```

func VerifyEthSig(tx sdk.Tx, signer ethtypes.Signer) error {
	for _, msg := range tx.GetMsgs() {
		msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
		if !ok {
			return errorsmod.Wrapf(errortypes.ErrUnknownRequest, "invalid message type %T, expected %T", msg, (*evmtypes.MsgEthereumTx)(nil))
		}

		if err := msgEthTx.VerifySender(signer); err != nil {
			return errorsmod.Wrapf(errortypes.ErrorInvalidSigner, "signature verification failed: %s", err.Error())
		}
	}

	return nil
}

	}
