### Title
Excess ETH Sent to Payable Bridge Functions Is Irrecoverably Forwarded to Wormhole Without Refund - (File: evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol)

---

### Summary

In `OmniBridgeWormhole`, multiple `payable` entry points (`deployToken`, `logMetadata`, `logMetadata1155`, `finTransfer`, `initTransfer`, `initTransfer1155`) forward ETH to `_wormhole.publishMessage` without verifying that the amount forwarded equals exactly `_wormhole.messageFee()`. Any ETH sent in excess of the Wormhole fee is permanently transferred to the Wormhole core contract, which does not refund overpayments. There is no refund path in `OmniBridgeWormhole` either. The excess ETH is irrecoverably lost to the caller.

---

### Finding Description

`OmniBridgeWormhole` overrides four internal extension hooks. Three of them (`deployTokenExtension`, `logMetadataExtension`, `finTransferExtension`) blindly forward the full `msg.value` to Wormhole:

```solidity
// deployTokenExtension (line 63)
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);

// logMetadataExtension (line 87)
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);

// finTransferExtension (line 109)
_wormhole.publishMessage{value: msg.value}(wormholeNonce, messagePayload, _consistencyLevel);
```

These are called from the `payable` public functions `deployToken`, `logMetadata`, `logMetadata1155`, and `finTransfer` respectively. None of those callers enforce `msg.value == _wormhole.messageFee()`.

The fourth override, `initTransferExtension`, forwards `value` (which equals `msg.value - nativeFee` for ERC20, or `msg.value - amount - nativeFee` for native ETH) to Wormhole:

```solidity
// initTransferExtension (line 143)
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

Again, there is no check that `value == _wormhole.messageFee()`. If the caller sends more ETH than required, the surplus is forwarded to Wormhole and is unrecoverable.

Wormhole's `publishMessage` accepts any ETH amount ≥ `messageFee()` and does not refund the difference. `OmniBridgeWormhole` has no sweep or rescue function for ETH, and the `receive()` fallback in the base `OmniBridge` contract only allows ETH to accumulate, not to be recovered by users.

---

### Impact Explanation

Any user or relayer who calls `deployToken`, `logMetadata`, `logMetadata1155`, `finTransfer`, `initTransfer`, or `initTransfer1155` with `msg.value` exceeding the exact Wormhole fee (plus `amount`/`nativeFee` where applicable) permanently loses the excess ETH. The ETH is forwarded to the Wormhole core contract, which retains it. This constitutes irrecoverable loss of user funds and qualifies as **High — fee/value misdirection that permanently removes user ETH from the bridge flow**.

---

### Likelihood Explanation

This is reachable by any unprivileged bridge user or relayer. Frontends and integrators must compute the exact Wormhole `messageFee()` at call time; any rounding, stale read, or UI error causes overpayment. The Wormhole fee can also change between the time a user signs a transaction and when it is mined, making exact-fee submission non-trivial. The `finTransfer` path is particularly exposed because relayers routinely call it and may pad ETH for gas safety.

---

### Recommendation

Add an explicit equality check in each extension before forwarding ETH to Wormhole:

```solidity
uint256 fee = _wormhole.messageFee();
require(msg.value == fee, "incorrect wormhole fee");
_wormhole.publishMessage{value: fee}(...);
```

For `initTransferExtension`, check `value == _wormhole.messageFee()` before the `publishMessage` call. Alternatively, refund any surplus ETH to `msg.sender` after the Wormhole call.

---

### Proof of Concept

1. Wormhole `messageFee()` returns `0.001 ether`.
2. A user calls `logMetadata(tokenAddress)` with `msg.value = 0.002 ether` (e.g., due to a stale frontend estimate).
3. `logMetadataExtension` executes `_wormhole.publishMessage{value: 0.002 ether}(...)`.
4. Wormhole accepts the call (amount ≥ fee) and retains the full `0.002 ether`.
5. `OmniBridgeWormhole` has no refund logic; the user's `0.001 ether` overpayment is permanently lost.

The same path applies to `deployToken` [1](#0-0) , `logMetadataExtension` [2](#0-1) , `finTransferExtension` [3](#0-2) , and `initTransferExtension` [4](#0-3) , all of which forward ETH to Wormhole without verifying the exact fee. The public entry points `deployToken`, `logMetadata`, `logMetadata1155`, and `finTransfer` are all declared `payable` with no `msg.value` equality guard. [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
