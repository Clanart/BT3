### Title
Integer Wrapping in `check_time_locks` Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Timelocks — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap = false` (the pre-soft-fork legacy path), relative timelock conditions (`ASSERT_HEIGHT_RELATIVE`, `ASSERT_SECONDS_RELATIVE`) are evaluated using `wrapping_add`. An attacker-controlled CLVM spend can supply a near-maximum value for `height_relative` (e.g., `0xFFFF_FFFF`, encodable as a valid 5-byte CLVM atom) causing the sum `confirmed_block_index + height_relative` to wrap around to a value smaller than `prev_transaction_block_height`, making the timelock check trivially pass. The coin is accepted as spendable before its required height, bypassing the timelock entirely.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces relative timelocks for each spend in a bundle. When `nowrap = false`, it uses `wrapping_add` for the addition of `confirmed_block_index` and `height_relative`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

If `height_relative = 0xFFFF_FFFF` and `confirmed_block_index = C`, then `wrapping_add(C, 0xFFFF_FFFF) = C - 1` (mod 2³²). The guard `prev_height < C - 1` is `false` for any `prev_height >= C` (which is always true for a confirmed coin), so the error branch is never taken and the spend is accepted unconditionally.

The same wrapping flaw applies to `ASSERT_SECONDS_RELATIVE` with `u64::MAX`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

The inverse condition `ASSERT_BEFORE_HEIGHT_RELATIVE` is also affected: wrapping causes a spend with a far-future deadline to be incorrectly rejected. [3](#0-2) 

**Attacker-controlled input path:** The condition value `height_relative` originates from a CLVM atom parsed by `sanitize_uint` with `max_size = 4`. The value `0xFFFF_FFFF` is reachable by encoding it as the 5-byte atom `[0x00, 0xFF, 0xFF, 0xFF, 0xFF]` — a valid CLVM positive integer with a leading zero byte to prevent sign-bit interpretation. `sanitize_uint` accepts this and returns `Ok(0xFFFF_FFFF)`, which is then cast to `u32`:

```rust
SanitizedUint::Ok(r) => Ok(Condition::AssertHeightRelative(r as u32)),
``` [4](#0-3) 

The repository's own test suite explicitly documents and confirms this wrapping bypass:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- timelock bypassed when nowrap=false
)]
``` [5](#0-4) 

The Python integration tests confirm the same for both `height_relative` and `seconds_relative`:

```python
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
# 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF), 105, None),
``` [6](#0-5) 

---

### Impact Explanation

This is a **High** impact finding matching the allowed category: *"Signature, aggregate signature, synthetic key, puzzle type, condition, timelock, or coin-id validation bypass enables unauthorized spend acceptance."*

Any puzzle that enforces a relative height or time lock via `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` can be bypassed when `nowrap = false` is the active evaluation mode. A spend bundle containing a coin whose puzzle emits (or accepts from the solution) a near-maximum `height_relative` value will pass timelock validation immediately, regardless of the actual block height. This enables premature coin spending — breaking vesting schedules, escrow contracts, or any protocol relying on relative timelocks.

---

### Likelihood Explanation

The `nowrap = false` path is the legacy pre-soft-fork behavior exposed through the Python binding `py_check_time_locks`. It is a reachable production path: the `nowrap` parameter is passed from the Python full-node layer, and nodes operating before soft-fork activation use `nowrap = false`. The attacker input is a single CLVM atom (`[0x00, 0xFF, 0xFF, 0xFF, 0xFF]`) embedded in a spend bundle — no privileged access, key material, or network-level capability is required.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for all relative timelock comparisons, regardless of the `nowrap` flag. The `nowrap = true` branch already uses `saturating_add` correctly. The `nowrap = false` branch should be updated to match, or the `nowrap` parameter should be removed entirely now that the correct behavior is established:

```rust
// ASSERT_HEIGHT_RELATIVE — always use saturating_add
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [7](#0-6) 

Apply the same fix to `seconds_relative`, `before_height_relative`, and `before_seconds_relative`.

---

### Proof of Concept

**Setup:** A coin confirmed at block height `C = 100`. The puzzle emits `(ASSERT_HEIGHT_RELATIVE . 0xFFFF_FFFF)` where `0xFFFF_FFFF` is encoded as the 5-byte CLVM atom `[0x00, 0xFF, 0xFF, 0xFF, 0xFF]`.

**Expected behavior:** The spend should be rejected until block height `100 + 4,294,967,295` — effectively never spendable.

**Actual behavior with `nowrap = false`:**
```
confirmed_block_index = 100
height_relative       = 0xFFFF_FFFF = 4,294,967,295
wrapping_add(100, 4294967295) = 99   (mod 2^32)

Check: prev_height (200) < 99  →  false  →  no error raised
Result: spend ACCEPTED at block 200
```

The timelock is completely bypassed. The same calculation applies for any `prev_height >= confirmed_block_index`, which is always true for a spendable coin. [7](#0-6) [8](#0-7)
