Audit Report

## Title
Partial-ETH branch in `pay()` silently consumes stranded router ETH to subsidize a subsequent WETH swap — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` contains a middle branch (lines 78–81) that fires when `token == WETH`, `payer != address(this)`, and `0 < address(this).balance < value`. This branch unconditionally wraps and forwards the router's entire ETH balance as a partial payment, then pulls only the remainder from the actual payer's allowance. Because the router holds no per-user ETH attribution, any ETH left on the router by a prior user (who sent `msg.value` but omitted `refundETH()`) is silently consumed by the next WETH swap, causing a direct, irreversible loss of principal for the original depositor.

## Finding Description
`pay()` at `metric-periphery/contracts/base/PeripheryPayments.sol` lines 69–88 has three branches for `token == WETH && payer != address(this)`:

```
nativeBalance >= value  →  wrap value ETH, pull nothing from payer          (L75–77)
nativeBalance > 0       →  wrap nativeBalance ETH, pull remainder from payer (L78–81) ← root cause
nativeBalance == 0      →  pull all from payer                               (L82–84)
```

The middle branch drains the router's full ETH balance regardless of which user deposited it. The call chain that reaches this branch is:

1. `exactInputSingle` (L67–86) stores `msg.sender` as payer via `_setNextCallbackContext` (L71).
2. The pool calls back `metricOmmSwapCallback` (L46–62), which dispatches to `_justPayCallback` (L192–199).
3. `_justPayCallback` calls `pay(_getTokenToPay(), _getPayer(), msg.sender, amount)`.
4. If `tokenIn == WETH` and the router holds any residual ETH, the partial branch fires, consuming that ETH before touching the payer's allowance.

The `receive()` guard (L32–34) only blocks direct ETH transfers from non-WETH addresses; it does not prevent ETH accumulation via `msg.value` on any `payable` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `unwrapWETH9`, `sweepToken`, `refundETH`). `refundETH()` is not enforced and must be composed manually.

## Impact Explanation
Direct, irreversible loss of user principal (ETH). User A sends `msg.value = V`, swaps `A < V` worth of WETH; the callback wraps exactly `A` ETH and the remaining `V − A` stays on the router. If User A omits `refundETH()`, User B's subsequent `exactInputSingle` with `tokenIn = WETH` and `amountIn = X > V − A` triggers the partial branch: the router wraps `V − A` of User A's ETH and sends it to the pool, then pulls only `X − (V − A)` WETH from User B. User B receives the full swap output while paying less than owed; User A's `V − A` ETH is permanently lost. The pool is made whole, so no pool insolvency occurs, but the loss to User A is concrete and unrecoverable. This meets the Critical/High direct loss-of-user-principal threshold.

## Likelihood Explanation
- `refundETH()` is opt-in and not enforced by any router entry point; omitting it is a common mistake in manual calls or incomplete multicalls.
- Any `payable` entry point can leave residual ETH on the router.
- The exploit requires no privileged access: any unprivileged address can call `exactInputSingle` with `tokenIn = WETH` to trigger the partial branch.
- The router's ETH balance is publicly readable on-chain; an attacker can monitor it and time the call immediately after a victim's transaction.
- The condition is repeatable: every stranded-ETH event creates a new opportunity.

## Recommendation
Replace the partial-ETH branch with the Uniswap v3 pattern — only use router ETH when it fully covers the payment; otherwise fall through to `safeTransferFrom`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

This eliminates cross-user ETH consumption while preserving the intended "pay with native ETH" convenience for users who send exactly the right `msg.value`.

## Proof of Concept
```
Initial state:
  router.balance = 0
  User A WETH balance = 1000e18
  User B WETH balance = 1000e18, allowance to router = 1000e18

Step 1 – User A strands ETH:
  User A calls exactInputSingle{value: 200e18}(
      pool, tokenIn=WETH, amountIn=100e18, ...
  )
  Callback: pay(WETH, UserA, pool, 100e18)
    nativeBalance=200e18 >= value=100e18 → wraps 100e18 ETH, sends to pool  (L75–77)
  After swap: router.balance = 100e18  (User A forgot refundETH)

Step 2 – User B exploits:
  User B calls exactInputSingle(
      pool, tokenIn=WETH, amountIn=150e18, ...
  )
  Callback: pay(WETH, UserB, pool, 150e18)
    nativeBalance=100e18 > 0, < 150e18 → partial branch fires:             (L78–81)
      wraps 100e18 ETH → sends to pool
      safeTransferFrom(UserB, pool, 50e18)  ← only 50e18 pulled from User B

Result:
  User A lost 100e18 ETH (stranded, now consumed by User B's swap)
  User B paid 50e18 WETH instead of 150e18 WETH
  Pool received correct 150e18 WETH (no pool loss)
```
