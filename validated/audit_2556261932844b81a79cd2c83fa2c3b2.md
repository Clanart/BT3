### Title
Non-EVM-native Cosmos bank tokens held by self-destructed contracts are not burned at `StateDB.Commit()`, leaving orphaned bank balances — (`x/evm/statedb/statedb.go`)

---

### Summary

When `StateDB.Commit()` processes a self-destructed contract, it only burns the EVM-denom balance. Non-EVM-native Cosmos bank tokens (IBC tokens, CosmWasm bridge tokens) held by the destroyed address are explicitly skipped and remain as orphaned bank balances after `DeleteAccount()` removes the auth account metadata. Any unprivileged user who can trigger `SELFDESTRUCT` on a contract holding such tokens causes those funds to be permanently mis-accounted in the Cosmos bank module.

---

### Finding Description

In `x/evm/statedb/statedb.go`, the `Commit()` function's self-destruct handling loop (lines 800–825) performs two operations for each self-destructed object:

1. Burns only the EVM denom balance via `s.keeper.SubBalance(cacheCtx, cosmosAddr, coin)` — where `coin` is constructed exclusively from `s.evmDenom`.
2. Deletes the auth account metadata via `s.keeper.DeleteAccount(cacheCtx, obj.Address())`.

The code itself documents the gap:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
``` [1](#0-0) 

The burn logic reads only the EVM denom balance and constructs a single coin for that denom:

```go

    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
``` [2](#0-1) 

After `DeleteAccount()` removes the auth account record, the Cosmos bank module retains any non-EVM-denom balances at that address. In Cosmos SDK, bank balances are stored independently of auth accounts, so these tokens persist as orphaned entries with no owning account.

The `Keeper.SubBalance` interface only operates on a single `sdk.Coin` (one denom at a time), and no enumeration of all bank balances for the destroyed address is performed anywhere in the commit path: [3](#0-2) 

The `Keeper.AddBalance` / `Keeper.SubBalance` pair in `x/evm/keeper/statedb.go` confirms that only the EVM denom is ever minted or burned through the stateDB path: [4](#0-3) 

---

### Impact Explanation

Non-EVM-native Cosmos bank tokens (e.g., IBC-transferred USDC, ATOM, or any CosmWasm bridge token) held by a self-destructed contract address are permanently orphaned in the bank module. The auth account is deleted, but the bank balances remain at the address with no owning account. These funds are:

1. **Permanently inaccessible to the original owner** — the contract is gone and no EVM mechanism exists to recover non-EVM-denom bank balances.
2. **Potentially recoverable by an attacker** — if the contract was deployed via CREATE2, a malicious actor can redeploy a new contract at the same deterministic address. The new account created at that address would inherit the orphaned bank balances, enabling unauthorized access to funds that should have been destroyed.

This matches the allowed High impact: *"EVM state transition… bug that permits… valid user funds/fees to be mis-accounted."* It also touches the Critical boundary: *"Cosmos bank funds… through… stateDB/native action logic."*

---

### Likelihood Explanation

- In a Cosmos-EVM environment, IBC tokens are routinely sent to contract addresses (e.g., via IBC transfer to an EVM contract address, or via a CosmWasm bridge).
- `SELFDESTRUCT` (and EIP-6780 same-transaction destruction) is reachable by any unprivileged user who owns or can call a destructible contract.
- CREATE2 address reuse is a well-known, documented technique requiring only knowledge of the factory address and salt — both of which are on-chain and observable.
- The code comment confirms this is a known gap, not a theoretical edge case.

---

### Recommendation

In `StateDB.Commit()`, when processing a self-destructed account, enumerate **all** bank balances held by the destroyed address across all denoms and burn them before calling `DeleteAccount()`. This requires exposing a `GetAllBalances(ctx, addr)` method on the `statedb.Keeper` interface and iterating over the result to call `SubBalance` for each non-zero denom. Alternatively, use `bankKeeper.SpendableCoins` or `bankKeeper.GetAllBalances` directly in the keeper's `DeleteAccount` implementation to drain all balances atomically.

---

### Proof of Concept

1. Deploy contract `A` via `CREATE2` from factory `F` with salt `S`. Contract `A` holds EVM-denom and also receives IBC-USDC via a Cosmos bank send to its address.
2. In a single EVM transaction (satisfying EIP-6780), call `A.destroy()` — `SELFDESTRUCT` fires, `SelfDestruct()` burns the EVM-denom balance in-memory and marks `obj.selfDestructed = true`.
3. `StateDB.Commit()` is called. The self-destruct branch at line 800 runs:
   - `GetBalance(cacheCtx, cosmosAddr, s.evmDenom)` → burns EVM denom. ✓
   - IBC-USDC balance is **never queried or burned**. ✗
   - `DeleteAccount(cacheCtx, obj.Address())` removes auth account metadata.
4. After commit: `eth_getBalance(A)` returns `0` (EVM denom burned), but `bank.GetBalance(A, "ibc/...")` returns the original IBC-USDC amount — orphaned.
5. Attacker calls `F.redeploy(S)` — CREATE2 recreates a contract at the same address `A`. The Cosmos SDK auth module creates a new `EthAccount` at `A`. The new account now has access to the orphaned IBC-USDC bank balance.
6. The new contract at `A` calls a precompile or native action to transfer the IBC-USDC to the attacker, completing the theft. [5](#0-4)

func (k *Keeper) AddBalance(ctx sdk.Context, addr sdk.AccAddress, coin sdk.Coin) (uint256.Int, error) {
	coins := sdk.NewCoins(coin)
	prevBalance := k.GetBalance(ctx, addr, coin.Denom)
	if err := k.bankKeeper.MintCoins(ctx, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.ModuleName, addr, coins); err != nil {
		return uint256.Int{}, err
	}
	return prevBalance, nil
}

func (k *Keeper) SubBalance(ctx sdk.Context, addr sdk.AccAddress, coin sdk.Coin) (uint256.Int, error) {
	coins := sdk.NewCoins(coin)
	prevBalance := k.GetBalance(ctx, addr, coin.Denom)
	if err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	if err := k.bankKeeper.BurnCoins(ctx, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	return prevBalance, nil
```
