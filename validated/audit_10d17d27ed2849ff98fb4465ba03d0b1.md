### Title
`ProofOfInclusion::valid()` Performs Internal Consistency Only — Missing Trusted-Root Comparison Allows Forged DataLayer Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate checks only that the proof's own fields are internally self-consistent. It never compares the computed root against any externally-trusted root hash. Because `ProofOfInclusion` is `Streamable` (deserializable from untrusted bytes) and exposed via Python bindings, an attacker can craft a structurally valid but entirely fabricated proof that passes `valid()` while proving inclusion of a key-value pair that does not exist in the real tree.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [1](#0-0) 

`root_hash()` returns the last `combined_hash` stored inside the proof itself:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The final check `existing_hash == self.root_hash()` is trivially satisfied: after the loop, `existing_hash` is exactly `last.combined_hash`, which is what `root_hash()` returns. The function therefore only verifies that each layer's hash chains correctly from the previous one — it never compares the resulting root against any externally-supplied, trusted root hash.

The struct is `Streamable` and fully exposed via Python bindings: [3](#0-2) [4](#0-3) 

An attacker can trivially construct a passing proof for any arbitrary `node_hash`:

1. Choose any `node_hash` (e.g., hash of a fabricated key-value pair not in the real tree).
2. Choose any `other_hash` and `other_hash_side`.
3. Compute `combined_hash = calculate_internal_hash(node_hash, side, other_hash)`.
4. Construct `ProofOfInclusion { node_hash, layers: [ProofOfInclusionLayer { other_hash_side: side, other_hash, combined_hash }] }`.
5. `proof.valid()` returns `true`.

The resulting `proof.root_hash()` is the attacker-chosen `combined_hash`, not the real tree's root. Any verifier that calls only `proof.valid()` — without separately asserting `proof.root_hash() == trusted_root` — accepts the forged proof.

The analog to the external report is direct: in `OperatorStakingPool._withdraw`, the accounting state is updated but the token transfer is never executed. Here, `valid()` executes the internal hash-chain check but the critical action — comparing against the trusted external root — is never performed.

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion.**

A malicious DataLayer peer can send a crafted `ProofOfInclusion` (serialized via `Streamable`) that is internally consistent but proves inclusion of a key-value pair that does not exist in the real tree. Any verifier that calls `proof.valid()` as its sole validity gate accepts the forged proof, allowing an attacker to convince a peer that arbitrary data is committed in the DataLayer tree when it is not. This enables false state attestation across DataLayer participants.

### Likelihood Explanation

The `ProofOfInclusion` type is `Streamable`, exposed via Python bindings with `valid()` and `root_hash()` as separate, independent methods. The API design makes it natural to call only `valid()` without the additional `root_hash()` check. The fuzz target and internal tests always generate proofs from the trusted blob and call `valid()` without an external root check, reinforcing the pattern. [5](#0-4) 

Any DataLayer consumer that receives a `ProofOfInclusion` over the network and calls only `proof.valid()` is exploitable by an unprivileged peer.

### Recommendation

`valid()` must accept a trusted root hash parameter and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root
}
```

The existing `valid()` method (which only checks internal consistency) should either be removed or clearly documented as insufficient for security-critical verification. All callers that receive `ProofOfInclusion` from untrusted sources must use the root-anchored variant.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};
use chia_datalayer::merkle::blob::calculate_internal_hash;

// Fabricate a proof for a key that does NOT exist in any real tree.
let fake_node_hash = Hash([0xAA; 32]);
let other_hash    = Hash([0xBB; 32]);
let side          = Side::Right;

// Compute a combined_hash that makes the proof internally consistent.
let combined_hash = calculate_internal_hash(&fake_node_hash, side, &other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: side,
        other_hash,
        combined_hash,
    }],
};

// valid() returns true even though this key is not in any real tree.
assert!(forged_proof.valid());

// The "root" is entirely attacker-controlled.
assert_eq!(forged_proof.root_hash(), combined_hash);
