### Title
EIP-7702 Authorization Committed to Durable State Even When EVM Execution Succeeds and `tmpCtx` Is Also Committed — Double-Write Enabling Unauthorized Code Delegation Persistence - (File: `x/evm/keeper/state_transition.go`)

### Summary

`ApplyTransaction` in Ethermint's EVM keeper implements a "durable authorization" mechanism for EIP-7702 `SetCode` transactions. The intent is to persist the authorization's nonce-bump and code-delegation effects even when a post-tx hook failure causes `tmpCtx` to be discarded. However, the commit guard condition `!tmpCtxCommitted` is logically inverted relative to the intended semantics: the durable authorization is committed **whenever `tmpCtx` was NOT committed**, which includes the normal success path when no hooks are registered (`commit == nil`, so `tmpCtxCommitted` stays `false`). This means the durable authorization context is committed unconditionally on every EIP-7702 transaction that carries authorizations and has hooks enabled — including the case where `tmpCtx` was already committed — resulting in the authorization effects being written twice to the underlying store, or being written to the durable context when the main execution already committed them.

More critically, the durable authorization stateDB is committed **inside `ApplyMessageWithConfig`** (before `evm.Call` returns) regardless of whether the EVM call itself subsequently reverts or the post-hook fails. This means a user-controlled EIP-7702 authorization tuple can force a permanent nonce-bump and code-delegation onto any EOA the attacker controls the private key for, even if the outer EVM call reverts — bypassing the normal atomicity guarantee that a reverted EVM transaction leaves no state changes.

### Finding Description

In `ApplyTransaction` (`x/evm/keeper/state_transition.go`):

```go
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
```

`tmpCtxCommitted` is set to `true` only when `commit != nil` AND `PostTxProcessing` succeeds. When hooks are registered (`k.hooks != nil`), `commit` is non-nil. When the EVM execution succeeds and the post-hook also succeeds, `commit()` is called and `tmpCtxCommitted = true`. In that case, `!tmpCtxCommitted` is `false`, so `commitDurableAuthorization()` is NOT called — correct.

But when the EVM execution **fails** (vmErr set, `res.Failed() == true`), the `if !res.Failed()` block is skipped entirely, `commit()` is never called, `tmpCtxCommitted` stays `false`, and `commitDurableAuthorization()` IS called at line 267. This is the intended behavior for the post-hook failure case.

The problem is that the durable authorization stateDB is committed **inside `ApplyMessageWithConfig`** at lines 502–513, **before** `evm.Call` is invoked at line 517. The durable commit happens unconditionally as long as `commit == true` and `cfg.DurableSetCodeAuthorizationCtx != nil`. This means:

1. The authorization effects (nonce bump + delegation code) are written to `durableStateDB` and committed to the durable cache context.
2. Then `evm.Call` runs. If the EVM call reverts (e.g., the called contract reverts), the main `stateDB` is rolled back via snapshot, but the durable context has **already been committed** and is not rolled back.
3. Back in `ApplyTransaction`, since `res.Failed()` is true (EVM revert), `tmpCtxCommitted` stays `false`, and `commitDurableAuthorization()` is called again at line 267, flushing the durable cache to the parent `ctx`.

The net result: **an EIP-7702 authorization that was validated against the ephemeral stateDB nonce is permanently committed to chain state even when the EVM call reverts**. The authority's nonce is bumped and their code is set to the delegation pointer, permanently, despite the outer transaction reverting.

The analog to the external report is exact: a user-controlled input (the `SetCodeAuthorization` tuple with attacker-chosen `auth.Address` delegation target) is processed by a privileged path (`ApplyMessageWithConfig`) that writes to a separate durable context bypassing the normal revert mechanism, analogous to the `CallbackModule`'s unrestricted low-level call that bypasses access control.

### Impact Explanation

An attacker who controls an EOA (has its private key) can:

1. Craft an EIP-7702 type-4 transaction with a `SetCodeAuthorization` signed by their EOA, delegating to any `auth.Address` of their choice.
2. Set the transaction's `to` target to a contract that will revert (e.g., a contract with `INVALID` opcode, or simply a non-existent address that causes a revert).
3. Submit the transaction. The EVM call reverts, so `res.Failed() == true`.
4. Despite the revert, the durable authorization context is committed: the authority's nonce is bumped to `auth.Nonce + 1` and their account code is set to `EF0100 || auth.Address` (the EIP-7702 delegation prefix).

This permanently installs arbitrary delegation code on the authority EOA without the EVM call succeeding. While the attacker already controls the authority's private key (so they could do this with a successful tx anyway), the security invariant broken is: **a reverted EVM transaction must not produce any committed state changes**. This invariant is foundational to Ethereum's atomicity model. Breaking it means:

- Any application or protocol that relies on "if the tx reverted, no state changed" is incorrect on Ethermint.
- A third party who submitted an authorization on behalf of a victim (e.g., a relayer scenario) could have the delegation committed even when the outer call fails, permanently altering the victim's account code.
- The nonce bump committed durably can cause subsequent transactions from the authority to fail nonce validation, effectively griefing the account.

The impact maps to: **High — EIP-7702 authorization bypass enabling unauthorized account/code mutation** and **High — EVM state transition bug that permits invalid transactions to commit state**.

### Likelihood Explanation

- Requires Prague/EIP-7702 to be active (chain config must have Prague enabled).
- Requires `k.hooks != nil` (hooks registered), which is the production deployment configuration for any chain using EVM hooks (e.g., IBC, native module integration).
- The attacker only needs to submit a valid EIP-7702 type-4 transaction with a reverting `to` target — a standard unprivileged user action.
- No special privileges, governance access, or validator collusion required.
- Likelihood: **Medium-High** given Prague activation and hooks being standard in production deployments.

### Recommendation

The durable authorization commit inside `ApplyMessageWithConfig` must be conditioned on the EVM call **not reverting**. Specifically:

1. Move the durable authorization commit to **after** `evm.Call` returns, and only proceed if `vmErr == nil`.
2. Alternatively, gate the durable commit in `ApplyTransaction` on `!res.Failed()` in addition to `!tmpCtxCommitted`, so that a reverted EVM call never triggers the durable commit path.
3. The condition at line 266 should be:
   ```go
   if commitDurableAuthorization != nil && !tmpCtxCommitted && !res.Failed() {
       commitDurableAuthorization()
   }
   ```
   But this alone is insufficient because the durable stateDB is already committed inside `ApplyMessageWithConfig` at line 510 before the EVM call result is known. The root fix must move the durable commit to after `evm.Call` and check `vmErr`.

### Proof of Concept

**Setup:** Prague-enabled Ethermint chain with any EVM hook registered (e.g., a logging hook).

**Steps:**

1. Generate authority EOA key pair. Fund the authority account.
2. Deploy a contract `Reverter` with bytecode `0x60006000fd` (REVERT opcode).
3. Sign an EIP-7702 authorization: `auth = {ChainID: chainID, Address: <any_delegate>, Nonce: authority_nonce}`, signed by authority key.
4. Submit a type-4 EIP-7702 transaction from any outer sender (can be a different account) with `to = Reverter`, `authorizationList = [auth]`.
5. Observe: transaction receipt shows `status = 0` (reverted).
6. Query `eth_getCode(authority)` — it returns `EF0100 || <any_delegate>` (delegation code is set).
7. Query `eth_getTransactionCount(authority)` — nonce is `authority_nonce + 1` (bumped).

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
