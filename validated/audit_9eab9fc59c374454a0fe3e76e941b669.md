### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezal of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS Cairo program accepts an arbitrary class hash from the caller without verifying that the hash corresponds to a previously declared contract class. An unprivileged user can call `replace_class` with any undeclared hash, permanently setting their contract's class to an invalid value. Any funds held in that contract become permanently frozen because all future calls to the contract will fail when the OS attempts to resolve the undeclared class hash to a compiled class.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The explicit `TODO` comment at line 898 acknowledges the missing check. The OS unconditionally commits the new class hash to state. [1](#0-0) 

When any future transaction calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the class hash stored by `replace_class` was never declared, `dict_read` returns 0 (the uninitialized default), and `find_element` fails with an assertion error because no compiled class fact with hash 0 exists. The OS cannot produce a valid proof for any block containing a call to the affected contract. The sequencer is forced to permanently exclude all transactions targeting that contract, making any funds it holds irrecoverable.

This is the direct analog of the "fake LPToken" vulnerability class: just as `supportMarket()` accepted an unverified contract instance that could later drain funds, `execute_replace_class` accepts an unverified class hash that permanently disables the contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, the state change is committed on-chain and cannot be reversed by any unprivileged action. All ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible. The OS-level proof system enforces the corrupted state, so no future block can legitimately execute the contract.

---

### Likelihood Explanation

Any unprivileged user who can deploy or interact with a contract can trigger this. The attack requires only a single `replace_class` syscall with an arbitrary felt value as the class hash — no special permissions, leaked keys, or operator cooperation are needed. The TODO comment in the production code confirms the check is absent and was intentionally deferred.

---

### Recommendation

Before committing the new class hash in `execute_replace_class`, verify that the hash exists as a key in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` with the requested `class_hash` and assert the returned compiled class hash is non-zero before updating the state entry. This mirrors the validation already implicitly enforced during `execute_entry_point` — making it explicit at the point of `replace_class` prevents the invalid state from ever being committed.

---

### Proof of Concept

1. Attacker deploys contract `C` holding user funds (e.g., an ERC-20 vault).
2. Attacker sends a transaction invoking a function on `C` that internally calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (any value not present in `contract_class_changes`).
3. `execute_replace_class` in `syscall_impls.cairo` (lines 896–910) writes `class_hash=0xdeadbeef` into `contract_state_changes` for contract `C` with no validation.
4. The block is proven and finalized. The on-chain state now records `C.class_hash = 0xdeadbeef`.
5. Any subsequent transaction calling `C` causes `execute_entry_point` (lines 154–166 of `execute_entry_point.cairo`) to call `dict_read(contract_class_changes, 0xdeadbeef)` → returns 0, then `find_element(..., key=0)` → assertion failure.
6. The sequencer cannot include any transaction targeting `C` in a provable block. All funds in `C` are permanently frozen.
