### Title
`CREATE`-based Bridge Token Deployment Enables Reorg-Induced Token-Address Mismatch and Permanent Fund Freeze — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken` deploys each bridge token proxy via the `CREATE` opcode (`new ERC1967Proxy(...)`). The resulting address is determined solely by the OmniBridge contract's nonce. On reorg-susceptible EVM chains (Arbitrum, Optimism, Polygon), a block reorganization that reorders two or more concurrent `deployToken` transactions causes each token to land at a different address than the one published in its Wormhole VAA. Because `OmniBridgeWormhole` is deployed with `consistencyLevel = 0`, Wormhole guardians sign the VAA immediately—before any reorg can be detected. Once the NEAR bridge processes the stale VAA, the token-address binding is permanently wrong, and every subsequent `finTransfer` for the affected token fails, freezing user funds irreversibly.

---

### Finding Description

`OmniBridge.deployToken` (lines 162–172) deploys each wrapped-token proxy with a plain `new ERC1967Proxy(...)`:

```solidity
address bridgeTokenProxy = address(
    new ERC1967Proxy(          // CREATE — address = f(OmniBridge.nonce)
        tokenImplementationAddress,
        abi.encodeWithSelector(BridgeToken.initialize.selector, ...)
    )
);
``` [1](#0-0) 

The address of `bridgeTokenProxy` depends entirely on OmniBridge's contract-level nonce. Immediately after deployment, `OmniBridgeWormhole.deployTokenExtension` publishes a Wormhole message that encodes this address as the canonical EVM token address for the corresponding NEAR token:

```solidity
bytes memory payload = bytes.concat(
    bytes1(uint8(MessageType.DeployToken)),
    Borsh.encodeString(token),
    bytes1(omniBridgeChainId),
    Borsh.encodeAddress(tokenAddress),   // ← nonce-derived address
    ...
);
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
``` [2](#0-1) 

The deployment script hard-codes `consistencyLevel = 0`:

```typescript
const consistencyLevel = 0
``` [3](#0-2) 

`consistencyLevel = 0` instructs Wormhole guardians to attest the message after **zero** additional confirmations—i.e., immediately upon observing the transaction in a block. This means a signed VAA is available before the block achieves any finality.

**Reorg scenario (two concurrent deployments):**

| Step | Block N (original) | Block N′ (post-reorg) |
|------|-------------------|----------------------|
| OmniBridge nonce 1 | `deployToken(tokenX)` → proxy at `0xAAAA` | `deployToken(tokenY)` → proxy at `0xAAAA` |
| OmniBridge nonce 2 | `deployToken(tokenY)` → proxy at `0xBBBB` | `deployToken(tokenX)` → proxy at `0xBBBB` |

Wormhole guardians sign VAA₁ = `(tokenX → 0xAAAA)` and VAA₂ = `(tokenY → 0xBBBB)` from Block N before the reorg. After the reorg, the EVM state has `nearToEthToken[tokenX] = 0xBBBB` and `isBridgeToken[0xAAAA] = false`.

The NEAR bridge processes VAA₁ and records `tokenX ↔ 0xAAAA`. Any subsequent `finTransfer` from NEAR to EVM for tokenX supplies `tokenAddress = 0xAAAA`, which hits the `else` branch in `finTransfer`:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
}
``` [4](#0-3) 

`0xAAAA` is now tokenY's proxy (not a vault holding tokenX), so the transfer either reverts or sends the wrong token. Users who locked tokenX on NEAR cannot recover their funds on EVM.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.** Once the NEAR bridge has committed the stale VAA binding (`tokenX → 0xAAAA`), there is no on-chain mechanism to correct it. Every `finTransfer` for tokenX will reference the wrong EVM address. Tokens locked on the NEAR side of the bridge become unclaimable. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

- Arbitrum, Optimism, and Polygon—all supported EVM targets—are documented as reorg-susceptible (Optimistic Rollup fraud proofs can revert confirmed blocks; Polygon has experienced multi-minute reorgs historically).
- `consistencyLevel = 0` eliminates the confirmation buffer that would otherwise protect against reorgs.
- Two concurrent `deployToken` calls in the same block is a realistic operational scenario (e.g., batch token onboarding by a relayer).
- No attacker-controlled input is required; the reorg itself is the trigger. The attacker only needs to observe the stale VAA and submit it to the NEAR bridge before the correct VAA is processed.

---

### Recommendation

Replace `new ERC1967Proxy(...)` with `new ERC1967Proxy{salt: ...}(...)` (i.e., `CREATE2`) using a deterministic salt derived from the token identifier:

```solidity
bytes32 salt = keccak256(abi.encodePacked(metadata.token));
address bridgeTokenProxy = address(
    new ERC1967Proxy{salt: salt}(
        tokenImplementationAddress,
        abi.encodeWithSelector(BridgeToken.initialize.selector, ...)
    )
);
```

With `CREATE2`, the proxy address is a pure function of `(OmniBridge address, salt, initcode hash)` and is independent of nonce. A reorg cannot change the address, so the Wormhole VAA and the on-chain mapping will always agree. Additionally, consider raising `consistencyLevel` to a value appropriate for each chain's finality guarantees.

### Proof of Concept

1. Relayer submits two transactions in the same block on Polygon:
   - Tx₁: `deployToken(sig₁, {token: "usdc.near", ...})`
   - Tx₂: `deployToken(sig₂, {token: "wbtc.near", ...})`
2. Block B is mined; OmniBridge nonce 1 → `usdc.near` proxy at `0xAAAA`; nonce 2 → `wbtc.near` proxy at `0xBBBB`.
3. Wormhole guardians (consistency 0) immediately sign VAA₁ = `(usdc.near → 0xAAAA)` and VAA₂ = `(wbtc.near → 0xBBBB)`.
4. Polygon reorg reverts Block B; Block B′ includes Tx₂ before Tx₁.
5. Post-reorg EVM state: `usdc.near → 0xBBBB`, `wbtc.near → 0xAAAA`, `isBridgeToken[0xAAAA] = true` (wbtc), `isBridgeToken[0xBBBB] = true` (usdc).
6. Attacker submits VAA₁ to the NEAR bridge; NEAR bridge records `usdc.near ↔ 0xAAAA`.
7. User bridges 1000 USDC from NEAR: NEAR locks 1000 USDC, relayer calls `finTransfer` on EVM with `tokenAddress = 0xAAAA`.
8. `isBridgeToken[0xAAAA]` is `true` (it is wbtc's proxy), so `IBridgeToken(0xAAAA).mint(recipient, 1000)` is called—minting 1000 wBTC instead of USDC, or reverting due to decimal/supply constraints. The user's 1000 USDC on NEAR is permanently locked with no valid redemption path. [1](#0-0) [5](#0-4) [3](#0-2)
