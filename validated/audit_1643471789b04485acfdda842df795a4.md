### Title
`execute_meta_tx_v0` Syscall Allows Unauthorized `__execute__` Invocation on Arbitrary Account Contracts Without OS-Level Signature Validation — (`execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall handler in `syscall_impls.cairo` allows any executing contract to invoke the `__execute__` entry point of an **arbitrary target account contract** without the OS calling `__validate__` first and without verifying the provided signature at the protocol level. This breaks the fundamental OS-enforced invariant that `__validate__` always precedes `__execute__`, enabling an unprivileged attacker to trigger execution on behalf of any account whose `__execute__` does not internally re-validate the signature.

---

### Finding Description

In the normal StarkNet OS transaction flow (`execute_invoke_function_transaction`), the OS enforces a strict two-phase sequence:

1. `run_validate` → calls `__validate__` on the account contract (signature check)
2. `non_reverting_select_execute_entry_point_func` → calls `__execute__` [1](#0-0) 

The `execute_meta_tx_v0` syscall handler, however, **skips `__validate__` entirely** and directly calls `contract_call_helper` (which calls `__execute__`) on an attacker-specified `contract_address`: [2](#0-1) 

The new `TxInfo` constructed for the target contract has:
- `version = 0` (which causes `run_validate` to return early if it were called — see `execute_transaction_utils.cairo` line 128)
- `nonce = 0` (no nonce check or increment is performed)
- `signature_start/end` = **attacker-supplied** values
- `account_contract_address = contract_address` (the victim's address)
- `caller_address = ORIGIN_ADDRESS` (address 0) [3](#0-2) 

The OS performs no cryptographic verification of the provided signature before dispatching to the target's `__execute__`. The only restriction is that `selector` must equal `EXECUTE_ENTRY_POINT_SELECTOR`: [4](#0-3) 

This is directly analogous to the reported `burn()` vulnerability: just as `MINTER_BURNER_ROLE` could destroy tokens from **any** address without their consent, `execute_meta_tx_v0` allows **any** contract to invoke `__execute__` on **any** account address without the account owner's authorization being enforced at the OS level.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Account contracts (especially v0/legacy accounts) are designed under the invariant that `__validate__` is always called by the OS before `__execute__`. Their `__execute__` implementations do not re-validate the signature, trusting the OS to have already done so. An attacker exploiting this syscall can:

1. Supply arbitrary `calldata` (e.g., `transfer(attacker_address, victim_balance)`) targeting the victim's account.
2. Supply a garbage or zero-length signature.
3. The victim's `__execute__` runs the attacker-supplied calls with full account authority, draining all assets held by or approved to the victim account.

Additionally, since `nonce = 0` is hardcoded and no nonce increment occurs on the target account, the attack can be replayed across multiple transactions until the victim's account is emptied.

---

### Likelihood Explanation

**High.** The syscall is available to any contract callable by any unprivileged transaction sender. No special role, key, or operator access is required. The attacker only needs to:

1. Deploy (or reuse) a contract that issues the `meta_tx_v0` syscall.
2. Submit a standard invoke transaction.

The condition that the target account does not re-validate the signature in `__execute__` is satisfied by the majority of deployed v0 account contracts (OpenZeppelin Cairo 0.x, Argent v0, Braavos v0), which rely entirely on the OS-enforced `__validate__` → `__execute__` ordering.

---

### Recommendation

The OS must enforce one of the following mitigations in `execute_meta_tx_v0`:

1. **Call `__validate__` on the target contract before `__execute__`**, passing the meta-transaction hash and the provided signature, and assert the return value equals `VALIDATED`. This mirrors the standard transaction flow.
2. **Restrict the syscall** so it can only be called by a designated, trusted relayer contract class (e.g., verified via `class_hash` check), reducing the attack surface to a privileged path.
3. **Remove the syscall** if the use case can be served by standard invoke transactions with the user as the sender.

---

### Proof of Concept

**Attacker contract (pseudocode Cairo):**
```cairo
// Attacker deploys this contract and invokes it via a standard invoke transaction.
func __execute__{syscall_ptr: felt*, ...}(calls: felt*) {
    // Target: victim's account address
    // Calldata: transfer(attacker_address, MAX_UINT256)
    // Signature: empty / garbage
    meta_tx_v0(
        contract_address=VICTIM_ACCOUNT_ADDRESS,
        selector=EXECUTE_ENTRY_POINT_SELECTOR,
        calldata=[transfer_call_to_attacker],
        signature=[]
    );
    return ();
}
```

**OS execution path:**
1. `execute_invoke_function_transaction` → attacker's `__execute__` runs.
2. Attacker's contract issues `meta_tx_v0` syscall.
3. `execute_meta_tx_v0` in `syscall_impls.cairo` constructs `new_tx_info` with `version=0`, `nonce=0`, empty signature, `account_contract_address=VICTIM`.
4. `contract_call_helper` calls victim's `__execute__` directly — **no `__validate__` call**.
5. Victim's `__execute__` (v0 account) executes the attacker-supplied transfer call.
6. Victim's funds are transferred to the attacker. [5](#0-4) [6](#0-5)
