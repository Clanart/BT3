### Title
Missing `l2_gas_consumed` in `FeeEstimation` Causes Incorrect Fee Breakdown in RPC Fee Estimation Response - (File: `crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct and its constructor `tx_execution_output_to_fee_estimation` expose `l2_gas_price` (the rate) but omit `l2_gas_consumed` (the amount). For L1 gas and L1 data gas, both the amount and the rate are returned, allowing clients to verify `overall_fee`. For L2 gas, only the rate is returned. The struct's own doc comment encodes the wrong formula (`overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`), omitting the L2 gas term entirely. Any client that reconstructs the fee from the returned fields will compute a value lower than `overall_fee` whenever `l2_gas > 0`.

### Finding Description

In `tx_execution_output_to_fee_estimation`:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),   // L1 amount ✓
    l1_gas_price,                                // L1 rate  ✓
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1-data amount ✓
    l1_data_gas_price,                           // L1-data rate ✓
    l2_gas_price,                                // L2 rate only — amount MISSING
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes l2_gas cost
    unit: tx_execution_output.price_unit,
})
```

`overall_fee` is taken from `receipt.fee`, which is computed by `GasVector::cost()`:

```rust
for (gas, price, resource) in [
    (self.l1_gas,      gas_prices.l1_gas_price,      Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas,      tipped_l2_gas_price,           Resource::L2Gas),  // included
] {
    let cost = gas.checked_mul(price.get())...
```

So `overall_fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * l2_gas_price`. The returned `FeeEstimation` exposes `l2_gas_price` but not `l2_gas_consumed`, making the breakdown irreconcilable with `overall_fee` for any V3 transaction that consumes L2 gas. The struct's doc comment at line 108–109 explicitly states the wrong formula, cementing the misleading contract.

The TODO at line 104 acknowledges this gap: `// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.` [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

**High. RPC fee estimation returns an authoritative-looking wrong value.**

A client calling `starknet_estimateFee` or `starknet_simulateTransactions` receives a `FeeEstimation` object with `l2_gas_price` but no `l2_gas_consumed`. The doc comment asserts `overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`. Any client that follows this formula will compute a fee that is **lower** than the actual `overall_fee` by exactly `l2_gas_consumed * l2_gas_price`. For V3 transactions with significant L2 gas usage (e.g., the test record shows `l2_gas_consumed = 0xb56b6` at `l2_gas_price = 0x1dcd65000`), the discrepancy is material. A wallet or SDK that sets resource bounds based on the reconstructed breakdown — rather than the opaque `overall_fee` — will under-provision L2 gas, causing transaction failures or incorrect fee assertions. [4](#0-3) 

### Likelihood Explanation

**High.** Every V3 transaction that executes Cairo code consumes L2 gas. The `GasVectorComputationMode::All` path (used for V3 transactions) always populates `gas_vector.l2_gas`. The missing field is structural — it is absent from the `FeeEstimation` struct definition itself, so every call to `tx_execution_output_to_fee_estimation` for a V3 transaction produces an incomplete response. No special attacker action is required; any user submitting a V3 transaction and calling `starknet_estimateFee` will receive the misleading breakdown. [5](#0-4) 

### Recommendation

1. Add `l2_gas_consumed: Felt` to the `FeeEstimation` struct, populated from `gas_vector.l2_gas.0.into()`.
2. Correct the doc comment to reflect the true formula: `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * l2_gas_price`.
3. Update all serialization snapshot tests and RPC spec conformance tests to include the new field. [6](#0-5) 

### Proof of Concept

Using the existing test record values:
- `l2_gas_consumed = 0xb56b6 = 743094`
- `l2_gas_price = 0x1dcd65000 = 8000000000`
- `l1_gas_consumed = 0x0`, `l1_data_gas_consumed = 0x80 = 128`, `l1_data_gas_price = 0x3e8 = 1000`
- `overall_fee = 0x151eb86f3ed400 = 5999999999999999488` (fri)

Client reconstruction using the documented formula:
```
fee_from_fields = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price
               = 0 * l1_gas_price + 128 * 1000
               = 128000
```

Actual `overall_fee` includes L2 gas:
```
actual = 128000 + 743094 * 8000000000 = 128000 + 5944752000000000 ≈ 5.94 × 10^15 fri
```

The client-computed value (`128000`) is off by a factor of ~46 million from the actual fee. A resource-bounds check or wallet balance check based on the reconstructed value would be catastrophically wrong. [7](#0-6) [8](#0-7)

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}

    /// Computes the cost (in fee token units) of the gas vector (panicking on overflow).
    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
