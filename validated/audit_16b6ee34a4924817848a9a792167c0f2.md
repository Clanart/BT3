### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Enforcement When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` uses `wrapping_add` for all four relative time/height conditions when the `nowrap` flag is `false`. An unprivileged user can embed a crafted `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` value in a CLVM puzzle that wraps the addition to a small number, making the time-lock check trivially pass. Simultaneously, nodes that have activated the `nowrap=true` path evaluate the same spend differently, producing deterministic consensus divergence between old and new nodes on any block containing such a spend.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between `saturating_add` (new, correct) and `wrapping_add` (legacy) for relative condition arithmetic:

```rust
// crates/chia-consensus/src/check_time_locks.rs lines 70-78
if let Some(seconds_relative) = spend.seconds_relative {
    if nowrap {
        if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
            return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
        }
    } else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
        return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
    }
}
``` [1](#0-0) 

The same pattern applies to `height_relative`, `before_height_relative`, and `before_seconds_relative`: [2](#0-1) 

When `nowrap=false`, the addition `unspent.timestamp.wrapping_add(seconds_relative)` wraps modulo `u64::MAX + 1`. If an attacker sets `seconds_relative` to `u64::MAX - coin_birth_timestamp + 1`, the sum wraps to `0`. The guard `timestamp < 0` is always `false`, so the condition returns `Ok(())` — the time lock is silently bypassed regardless of how much time has elapsed.

The inverse holds for `ASSERT_BEFORE_SECONDS_RELATIVE`: a wrapping value collapses the deadline to `0`, making `timestamp >= 0` always `true`, permanently blocking a coin from being spent.

The project's own test suite documents and confirms this divergence explicitly:

```
# 10000 + (u64::MAX - 9999) overflows to 0, wrapping: 10150 < 0 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_D8F0), 105, None),
# 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF), 105, None),
``` [3](#0-2) 

`nowrap=true` → `Err` (time lock enforced); `nowrap=false` → `Ok` (time lock bypassed). The same spend is accepted by legacy nodes and rejected by updated nodes.

---

### Impact Explanation

**Timelock bypass (High):** A CLVM puzzle embedding `ASSERT_SECONDS_RELATIVE = u64::MAX` (or any value that wraps `coin_birth_timestamp + seconds_relative` below `current_timestamp`) passes the staleness check immediately under `nowrap=false`. Any protocol relying on relative time locks for vaults, escrows, or rate-limiting is undermined.

**Consensus divergence (Critical):** Because `nowrap` is not derived from `ConsensusFlags` or `get_flags_for_height_and_constants` — it is passed separately from the Python layer — any block containing a spend with a wrapping relative condition will be accepted by nodes running `nowrap=false` and rejected by nodes running `nowrap=true`. This is deterministic, reproducible consensus divergence triggered by a single crafted spend bundle. [4](#0-3) 

`get_flags_for_height_and_constants` returns `ConsensusFlags` but has no `nowrap` equivalent, meaning the flag is not governed by the same fork-height mechanism as all other consensus rule changes. [5](#0-4) 

---

### Likelihood Explanation

- The attacker only needs to publish a coin whose puzzle outputs `ASSERT_SECONDS_RELATIVE` with a near-`u64::MAX` value. No privileged access, key material, or validator cooperation is required.
- The wrapping values are trivially computable given the coin's birth timestamp (available on-chain).
- The divergence window exists for any block height range where some nodes pass `nowrap=false` and others pass `nowrap=true`.

---

### Recommendation

1. **Remove the `nowrap` branch entirely.** The `saturating_add` path is the correct semantic for all relative conditions. The `wrapping_add` path has no valid consensus use case — it allows bypassing time locks and creates divergence.
2. If backward compatibility with pre-fork blocks is required, gate the `nowrap=false` path behind an explicit, height-checked `ConsensusFlags` bit (analogous to `hard_fork_height`) so all nodes agree deterministically.
3. Alternatively, reject at condition-parse time any `seconds_relative` or `height_relative` value that would overflow when added to the maximum possible coin birth timestamp.

---

### Proof of Concept

Coin confirmed at timestamp `T = 10_000`. Attacker sets `seconds_relative = u64::MAX - 9_999` in the puzzle output.

Under `nowrap=false`:
```
unspent.timestamp.wrapping_add(seconds_relative)
= 10_000u64.wrapping_add(u64::MAX - 9_999)
= 10_000 + 18_446_744_073_709_541_616  (mod 2^64)
= 0
```
Check: `current_timestamp < 0` → `false` → `Ok(())`. Time lock bypassed at any block timestamp.

Under `nowrap=true`:
```
unspent.timestamp.saturating_add(seconds_relative)
= u64::MAX
```
Check: `current_timestamp < u64::MAX` → `true` → `Err(AssertSecondsRelativeFailed)`. Time lock enforced.

The same spend bundle produces `Ok` on legacy nodes and `Err` on updated nodes — a chain split. [1](#0-0) [6](#0-5)

    //    negative infinity
    //  * AGG_SIG_* conditions are allowed to have unknown additional

    // Adds a new keccak256 operator under the softfork guard with extension 1.
    // This operator can be hard forked in later, but is not included in a hard fork yet.
