### Title
`batch_insert` Bypasses Duplicate Hash Guard for Items Beyond the First Two, Corrupting `leaf_hash_to_index` Cache — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`batch_insert` uses two distinct insertion paths. The last two items in the input vector are popped and routed through `insert()`, which enforces the `HashAlreadyPresent` guard. All remaining items (the front of the vector) are inserted directly via `insert_entry_to_blob()` with no duplicate-hash check. A caller supplying a batch where a hash appearing in the front of the vector also appears in the tail will silently produce a tree containing two leaves with the same hash, corrupting the `leaf_hash_to_index` cache and violating the uniqueness invariant.

---

### Finding Description

**Root cause — split insertion paths in `batch_insert`:**

When `leaf_count <= 1` at call time, `batch_insert` pops the last two entries and routes them through `self.insert()`: [1](#0-0) 

`insert()` explicitly checks for duplicate hashes before proceeding: [2](#0-1) 

All remaining items (the front of the vector) are then iterated and written directly via `insert_entry_to_blob()`: [3](#0-2) 

`insert_entry_to_blob()` calls `block_status_cache.add_leaf()` unconditionally: [4](#0-3) 

`add_leaf()` performs a bare `HashMap::insert`, which silently overwrites any existing entry for the same hash: [5](#0-4) 

**Concrete trigger sequence (empty tree, 3-item batch):**

```
batch_insert([(key1, val1, H), (key2, val2, H2), (key3, val3, H)])
```

1. `pop()` → `(key3, val3, H)` → `insert()` → H registered in `leaf_hash_to_index`
2. `pop()` → `(key2, val2, H2)` → `insert()` → H2 registered
3. Remaining `[(key1, val1, H)]` → `for` loop → `insert_entry_to_blob` → `add_leaf` → `leaf_hash_to_index.insert(H, new_index)` **silently overwrites** the index stored for H

Result: 3 leaf nodes exist in the blob, but `leaf_hash_to_index` contains only 2 entries. The leaf for `key3` is now unreachable by hash lookup; the entry for H points to `key1`'s leaf instead.

**`check_integrity` detection — but not automatic:**

`check_just_integrity` does compare `leaf_count` against `leaf_hash_to_index_cache_length` and would return `IntegrityLeafHashToIndexCacheLength` for this state: [6](#0-5) 

However, `check_integrity_on_drop` is only `true` in test builds: [7](#0-6) 

In production, `check_integrity` is never called automatically after `batch_insert`, so the corrupted state persists.

**Exposed via Python binding:**

The Python-facing `py_batch_insert` passes caller-supplied hashes directly into `batch_insert` with no pre-validation: [8](#0-7) 

---

### Impact Explanation

The DataLayer Merkle tree's `leaf_hash_to_index` cache is corrupted: two leaves share hash H in the blob, but the map points only to one. Hash-based lookups (`get_index_by_leaf_hash`) return the wrong leaf. Proof-of-inclusion generation for the displaced leaf will either fail or produce a proof for the wrong node. The Merkle root computed from the tree structure may appear structurally valid while the semantic invariant (all leaf hashes unique) is broken, allowing untrusted input to produce invalid DataLayer state that passes structural checks but fails hash-based verification.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The Python binding is directly callable by DataLayer node software processing external publisher data. In a DataLayer subscription scenario, a malicious publisher can craft a batch update containing a repeated hash at a position that survives into the `for`-loop path (i.e., not among the last two items). No cryptographic capability is required — the hash field is caller-supplied and not derived internally from the key/value pair. The trigger is a single `batch_insert` call with ≥3 items and a repeated hash at index 0 matching one at index ≥2.

---

### Recommendation

Add a duplicate-key and duplicate-hash check at the top of `batch_insert` before any insertion occurs, mirroring the guards in `insert()`. Alternatively, route all items through `insert()` or add an explicit `contains_leaf_hash` / `contains_key` check inside the `for` loop before calling `insert_entry_to_blob`.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn h(b: u8) -> Hash { Hash(Bytes32::new([b; 32])) }

let mut blob = MerkleBlob::new(vec![]).unwrap();

// H appears at index 0 AND index 2; index 2 is popped first (goes through insert()),
// index 0 goes through the unchecked for-loop path.
let batch = vec![
    ((KeyId(1), ValueId(1)), h(0xAA)),  // index 0 — for-loop path, no check
    ((KeyId(2), ValueId(2)), h(0xBB)),  // index 1 — popped second via insert()
    ((KeyId(3), ValueId(3)), h(0xAA)),  // index 2 — popped first via insert()
];

// No error raised — duplicate hash 0xAA silently accepted
blob.batch_insert(batch).unwrap();

// check_integrity detects the corruption
assert!(blob.check_integrity().is_err());
```
