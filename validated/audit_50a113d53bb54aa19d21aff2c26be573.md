### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Validation, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. An unprivileged caller supplying a batch with repeated `KeyId` or `Hash` values causes multiple leaf nodes with the same key to be written into the binary blob, silently corrupting the `BlockStatusCache` and producing a Merkle root hash that does not faithfully represent the stored key-value set. Subsequent `get_proof_of_inclusion` calls return proofs anchored to a corrupted root, enabling forged inclusion proofs against any verifier that trusts that root.

### Finding Description

`MerkleBlob::insert` enforces two guards before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path for all items beyond the first two (or for all items when the tree already has ≥ 2 leaves). It writes leaf blocks directly via `insert_entry_to_blob` with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which silently overwrites the `key_to_index` and `leaf_hash_to_index` HashMap entries on collision:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index); // silent overwrite
}
``` [3](#0-2) 

The result is that the binary blob contains two (or more) leaf nodes for the same key, but `key_to_index` only records the last-written one. The subtree built from these leaves — and the final root hash — incorporates all duplicate leaves, while `get_proof_of_inclusion` can only generate a proof for the last-written leaf. The root hash therefore does not correspond to any consistent key-value mapping.

The condition that routes items through the safe `insert()` path is:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [4](#0-3) 

When the tree already has ≥ 2 leaves, this block is skipped entirely and **every** item in the batch bypasses validation. When the tree has 0–1 leaves, only the last two items (popped from the end) are checked; all earlier items are unchecked.

The Python binding exposes this path directly:

```python
def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...
``` [5](#0-4) 

The Rust binding:

```rust
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [6](#0-5) 

### Impact Explanation

A corrupted Merkle root means:

1. **Forged inclusion proofs**: `get_proof_of_inclusion` returns a structurally valid proof (it passes `proof.valid()`) for the last-written duplicate leaf, but the root hash was computed from a tree that also contains the orphaned duplicate. Any external verifier trusting that root accepts a proof that does not faithfully represent the stored state.

2. **Corrupted tree root**: The root hash no longer represents a consistent key-value set. DataLayer consumers (e.g., cross-chain bridges, oracles, or light clients) that rely on the root to verify state receive a false commitment.

3. **Cache/blob divergence**: `block_status_cache.key_to_index.len()` is smaller than the actual leaf count in the blob. `check_integrity()` would detect this, but it is not called automatically after `batch_insert`. [7](#0-6) 

This matches the allowed impact: **High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

**Medium.** The attacker must be able to supply input to `batch_insert` — either directly via the Python API or indirectly through a DataLayer update flow that does not pre-deduplicate keys. No privileged role is required. The tree must have ≥ 2 existing leaves (or the batch must have ≥ 3 items) to reach the unchecked path, which is the normal operating state of any non-trivial DataLayer store.

### Recommendation

Add duplicate-key and duplicate-hash checks inside `batch_insert` before writing each leaf in the unchecked loop, mirroring the guards in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing write logic
}
```

Alternatively, refactor `batch_insert` to call `insert` for every item (accepting the performance trade-off), or add a pre-pass that deduplicates and validates the entire input vector before any writes begin.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(Vec::new()).unwrap();

    // Seed the tree with 2 leaves so leaf_count > 1 → all batch items bypass checks
    blob.insert(KeyId(1), ValueId(1), &Hash(Bytes32::new([1u8; 32])), InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(2), ValueId(2), &Hash(Bytes32::new([2u8; 32])), InsertLocation::Auto {}).unwrap();

    // Batch with duplicate KeyId(3) — both entries bypass the duplicate guard
    let batch = vec![
        ((KeyId(3), ValueId(10)), Hash(Bytes32::new([3u8; 32]))),
        ((KeyId(3), ValueId(99)), Hash(Bytes32::new([4u8; 32]))), // duplicate key
    ];
    blob.batch_insert(batch).unwrap(); // succeeds — no error

    // Tree blob now contains two leaf nodes for KeyId(3)
    // key_to_index only records the last one (ValueId(99))
    // Root hash is computed from a tree containing BOTH leaves → corrupted root
    // check_integrity() will fail:
    blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheLength or similar
}

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
