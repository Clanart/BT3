Audit Report

## Title
`payable` liquidity entry points silently trap ETH on non-WETH pools, enabling permissionless theft via `refundETH()` - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
All four liquidity entry points in `MetricOmmPoolLiquidityAdder` are declared `payable`, but the internal `pay()` helper only consumes native ETH when the pool token is exactly `WETH`. For pools whose tokens are both non-WETH ERC-20s, any ETH sent with the call is silently left in the contract. Because `refundETH()` unconditionally forwards the entire contract ETH balance to `msg.sender` with no access control, a front-runner can call it immediately after the victim's transaction and steal the trapped ETH.

## Finding Description
All four external entry points are marked `payable`:

- `addLiquidityExactShares(pool, owner, ...)` [1](#0-0) 
- `addLiquidityExactShares(pool, salt, ...)` [2](#0-1) 
- `addLiquidityWeighted(pool, owner, ...)` [3](#0-2) 
- `addLiquidityWeighted(pool, salt, ...)` [4](#0-3) 

Inside the callback, `pay()` is the sole ETH consumer. It only wraps and forwards native ETH when `token == WETH`; for any other token it falls through to the `else` branch and calls `safeTransferFrom`, completely ignoring `address(this).balance`: [5](#0-4) 

ETH sent with a call on a non-WETH pool therefore accumulates in the contract. `refundETH()` then sends the **entire** balance to whoever calls it, with no binding to the original depositor and no access control: [6](#0-5) 

The `receive()` guard that rejects direct ETH transfers from non-WETH addresses does not apply here — it only fires on bare ETH sends, not on ETH sent alongside a `payable` function call: [7](#0-6) 

The interface NatSpec explicitly trains users to send ETH with liquidity calls for WETH pools, making it natural for a user switching to a non-WETH pool to repeat the same pattern: [8](#0-7) 

## Impact Explanation
Direct loss of user principal (ETH). The victim's liquidity is added correctly via ERC-20 pull, but any ETH sent with the call is permanently lost to the first caller of `refundETH()`. There is no recovery path for the victim once the front-runner has claimed the balance. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
- The interface NatSpec explicitly documents the ETH/multicall pattern for WETH pools, training users to send ETH with liquidity calls.
- A user switching from a WETH pool to a non-WETH pool without changing their call pattern silently loses ETH.
- `refundETH()` is a public, zero-argument function trivially callable by MEV bots monitoring the mempool.
- For `addLiquidityWeighted`, the probe-then-pay flow widens the window: the probe reverts (no ETH consumed), then the paying add executes, leaving ETH in the contract throughout both sub-calls. [9](#0-8) 

## Recommendation
Remove `payable` from all four liquidity-adder entry points. ETH-for-WETH flows should be handled exclusively through `multicall{value: ...}([addLiquidityExactShares(...), refundETH()])`, where the `multicall` wrapper (which remains `payable`) is the sole ETH entry point. This matches the pattern already used by the swap router and eliminates the accidental-ETH trap entirely.

## Proof of Concept
```solidity
// Pool: token0 = USDC, token1 = DAI (neither is WETH)

// Step 1 – Victim calls addLiquidityExactShares and accidentally sends 1 ETH
liquidityAdder.addLiquidityExactShares{value: 1 ether}(
    pool, owner, salt, deltas, maxAmount0, maxAmount1, ""
);
// pay() takes the safeTransferFrom path (token != WETH), ignores msg.value
// 1 ETH now sits in MetricOmmPoolLiquidityAdder

// Step 2 – Front-runner (Bob) sees the victim's tx in the mempool,
//           submits a higher-gas refundETH() call
liquidityAdder.refundETH();   // msg.sender = Bob
// _transferETH(Bob, 1 ether) executes; Bob receives victim's 1 ETH
```

The victim's liquidity is added correctly (ERC-20 pull succeeds), but the 1 ETH is permanently lost to the front-runner.
