### Title
`normalize_amount` Floor-Division Truncation Permanently Freezes User Funds When `amount_without_fee` Falls Below Decimal Scaling Factor — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The `init_transfer` entry-point validates only that `fee < amount`, but does **not** validate that `normalize_amount(amount - fee) > 0`. Because `normalize_amount` uses floor division, any `amount_without_fee` that is smaller than the decimal scaling factor (`10^(origin_decimals − decimals)`) silently truncates to zero. The downstream `sign_transfer` call then panics with `InvalidAmountToTransfer`, and because there is no cancel/refund path for a pending transfer, the user's locked or burned tokens are permanently irrecoverable.

---

### Finding Description

**`normalize_amount` uses floor division:** [1](#0-0) 

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))   // floor division — remainder silently lost
}
```

**`init_transfer` only checks `fee < amount`, not that the normalized net amount is non-zero:** [2](#0-1) 

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

Tokens are locked/burned and the `TransferMessage` is stored in `pending_transfers` immediately after this check passes.

**`sign_transfer` later rejects the transfer if the normalized net amount is zero:** [3](#0-2) 

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

**`amount_without_fee` is a simple subtraction:** [4](#0-3) 

```rust
pub fn amount_without_fee(&self) -> Option<u128> {
    self.amount.0.checked_sub(self.fee.fee.0)
}
```

**There is no cancel or refund path for a pending transfer.** `remove_transfer_message` is only called from `claim_fee_callback`, which requires a proof of finalization on the destination chain — a proof that can never exist because the transfer was never signed. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Once `init_transfer` succeeds, the user's tokens are burned (bridged tokens) or locked (native tokens) inside the bridge contract. If `sign_transfer` always panics for that transfer ID, the funds are permanently frozen with no on-chain recovery mechanism.

---

### Likelihood Explanation

**Realistic.** The condition is triggered whenever:

```
amount_without_fee  <  10^(origin_decimals − decimals)
```

Concrete example with a NEAR-native token (24 decimals) bridged to an EVM chain where it is represented with 6 decimals (scaling factor = `10^18`):

| Parameter | Value |
|---|---|
| `origin_decimals` | 24 |
| `decimals` (EVM) | 6 |
| Scaling factor | `10^18` |
| User sets `amount` | `10^18 − 1` (sub-unit dust) |
| User sets `fee` | `0` |
| `amount_without_fee` | `10^18 − 1` |
| `normalize_amount(...)` | `(10^18 − 1) / 10^18 = 0` ← **zero** |

The `init_transfer` guard (`fee < amount`) passes. Tokens are burned. `sign_transfer` panics. Funds are gone.

A second vector: a user sets a non-zero fee that brings `amount_without_fee` just below the scaling threshold. For example, `amount = 10^18 + 1`, `fee = 2` → `amount_without_fee = 10^18 − 1` → normalized to 0.

A third vector: a relayer with permission to call `update_transfer_fee` can raise the fee on an existing pending transfer (the code enforces fee can only increase, not decrease) until `amount_without_fee` falls below the scaling factor, permanently bricking the transfer for the victim.

---

### Recommendation

Add a validation at `init_transfer` time (before locking/burning tokens) that the normalized net amount is strictly positive:

```rust
let decimals = /* look up decimals for destination chain */;
let normalized_net = Self::normalize_amount(
    amount.0.checked_sub(init_transfer_msg.fee.fee.0)
        .near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized_net > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

Apply the same guard inside `update_transfer_fee` when the fee is raised, to prevent a relayer from bricking an existing pending transfer.

---

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 24`, `decimals = 6` (scaling factor `10^18`).

1. User calls `ft_transfer_call` with `amount = 10^18 − 1`, `fee = 0`, valid EVM recipient.
2. `init_transfer` guard: `0 < 10^18 − 1` → passes. Tokens burned. Message stored in `pending_transfers`.
3. Relayer calls `sign_transfer(transfer_id, ...)`.
4. `normalize_amount(10^18 − 1, {decimals:6, origin_decimals:24})` = `(10^18 − 1) / 10^18` = **0**.
5. `require!(0 > 0, ...)` → **panics** with `InvalidAmountToTransfer`.
6. No `cancel_transfer` exists. `claim_fee_callback` requires a destination-chain finalization proof that can never be produced. User's `10^18 − 1` yocto-tokens are permanently frozen.
