### Title
Malicious P2P Peer Stores Arbitrary Transactions Unchecked Against `transaction_commitment` — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

`TransactionStreamFactory::parse_data_for_block` uses the peer-supplied `n_transactions` field from the stored header as the sole loop bound, then accepts exactly that many peer-supplied transactions with no verification against the `transaction_commitment` stored in the same header. A single malicious peer can therefore bind any set of transactions to any block number, producing a stored `BlockBody` whose hashes are permanently inconsistent with the committed `transaction_commitment`.

---

### Finding Description

**Step 1 — Header acceptance without commitment verification.**

`HeaderStreamBuilder::parse_data_for_block` validates only two things before storing a peer-supplied `SignedBlockHeader`:

- `block_number` matches the expected sequence number
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` [1](#0-0) 

Neither the `block_hash`, nor the `transaction_commitment`, nor the `n_transactions` field is verified against any external source. The header — including its peer-controlled `n_transactions` and `transaction_commitment` — is written verbatim to storage via `append_header`. [2](#0-1) 

**Step 2 — Transaction count derived from the peer-controlled header.**

`TransactionStreamFactory::parse_data_for_block` reads `target_transaction_len` directly from the just-stored header's `n_transactions` field: [3](#0-2) 

The loop then runs until exactly `target_transaction_len` peer-supplied transactions have been received.

**Step 3 — No transaction hash or commitment validation.**

Each received `FullTransaction` is pushed into `block_body` without any check that its hash matches the `transaction_commitment` in the header. The code itself acknowledges this gap: [4](#0-3) 

The `transaction_hash` field is also peer-supplied and is pushed directly into `block_body.transaction_hashes` without recomputation.

**Step 4 — Corrupted body written to storage.**

The assembled `(BlockBody, BlockNumber)` is committed unconditionally: [5](#0-4) 

---

### Impact Explanation

After the attack the node's storage contains:

| Field | Value |
|---|---|
| `header.transaction_commitment` | Peer-chosen value `X` |
| `body.transactions[0..N]` | N peer-chosen arbitrary transactions |
| `body.transaction_hashes[0..N]` | N peer-chosen arbitrary hashes |

The stored `transaction_commitment` does not commit to the stored transactions. Any RPC call that reads the block body (`get_block_with_txs`, `get_transaction_by_hash`, `get_transaction_receipt`, etc.) returns authoritative-looking but fabricated transaction data. This matches the **High** impact tier: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

It also satisfies the **Critical** commitment-binding tier: *"Unprivileged-user-triggered … transaction commitment … path binds valid-looking block data to the wrong commitment"* — the stored `transaction_commitment` is permanently mismatched with the stored `BlockBody`.

---

### Likelihood Explanation

Any reachable P2P peer requires no credentials. The attack requires only that the victim node selects the malicious peer as its sync source for a given block range, which is the normal operating mode of the sync client.

---

### Recommendation

1. **Recompute and verify `transaction_commitment` after collecting all transactions.** After the collection loop, compute `calculate_transaction_commitment` over the received `transaction_hashes` and compare it against `header.transaction_commitment`. Reject the peer on mismatch.
2. **Recompute each transaction hash from its content** rather than trusting the peer-supplied `transaction_hash` field (the existing TODO at line 88 tracks this).
3. **Verify `n_transactions` against `concatenated_counts`** in the header (the existing TODO at `header.rs` line 140 tracks this). [6](#0-5) 

---

### Proof of Concept

Concrete call path:

```
malicious peer
  → sends SignedBlockHeader { n_transactions: 3, transaction_commitment: FAKE_COMMITMENT, block_number: 0, signatures: [valid_sig] }
  → HeaderStreamBuilder::parse_data_for_block  [header.rs:104-120]  ← only checks block_number and sig count
  → SignedBlockHeader::write_to_storage → append_header(0, header)  [header.rs:36-39]  ← stores n_transactions=3, transaction_commitment=FAKE
  → sends FullTransaction { tx: TX_A, hash: HASH_A }, FullTransaction { tx: TX_B, hash: HASH_B }, FullTransaction { tx: TX_C, hash: HASH_C }
  → TransactionStreamFactory::parse_data_for_block  [transaction.rs:62-66]  ← reads target=3 from stored header
  → loop collects TX_A, TX_B, TX_C without any commitment check  [transaction.rs:86-90]
  → (BlockBody, 0)::write_to_storage → append_body(0, body)  [transaction.rs:36]  ← stores fabricated body

assert stored_body.transactions == [TX_A, TX_B, TX_C]          // true
assert recompute_tx_commitment(stored_body) == FAKE_COMMITMENT  // false → commitment mismatch confirmed
```
