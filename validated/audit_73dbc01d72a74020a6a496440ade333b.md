### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Unbounded Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function in the StarkNet OS computes a hash for meta-transaction v0 syscalls without including any nonce. Simultaneously, `check_and_increment_nonce` explicitly skips nonce enforcement for version 0. Together, these two gaps mean that any signature produced for a `meta_tx_v0` call can be replayed an unlimited number of times by any unprivileged actor who observes it on-chain, leading to direct loss of funds.

---

### Finding Description

`compute_meta_tx_v0_hash` is defined as:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        ...
        additional_data_size=0,          // ← no nonce appended
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

Compare this with `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1, additional_data=&nonce` to bind the hash to a single use: [2](#0-1) 

The second half of the root cause is in `check_and_increment_nonce`, which explicitly opts version-0 transactions out of all nonce enforcement:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

Because `compute_meta_tx_v0_hash` produces version 0 hashes and `check_and_increment_nonce` is a no-op for version 0, the OS provides **zero replay protection** for `meta_tx_v0` signatures at the protocol level. The hash is fully determined by `(contract_address, entry_point_selector, calldata, chain_id)` — all of which are static for a given intended call.

For contrast, every modern account transaction (Invoke v3, Declare v3, DeployAccount v3) correctly includes the nonce via `hash_tx_common_fields`: [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `meta_tx_v0` syscall is designed to let a contract execute a user-signed call on the user's behalf (a meta-transaction relay pattern). If a user signs a `meta_tx_v0` authorizing, for example, a token transfer of 100 STRK, any party who observes that signature on-chain can submit the identical `(contract_address, selector, calldata, sig)` tuple again and again. Each replay executes the same transfer. Because the OS never checks or increments a nonce for version-0 meta transactions, there is no protocol-enforced bound on the number of replays. The user's account is drained until it is empty or the attacker stops.

---

### Likelihood Explanation

**Medium.**

- Signatures for `meta_tx_v0` calls are broadcast as part of transaction calldata and are therefore publicly visible to any network observer.
- No privileged access, leaked key, or operator cooperation is required — the attacker only needs to copy the already-public signature and re-submit it.
- The only natural limit is the account balance; once it is exhausted the replay reverts, but by then the damage is done.
- The attack is fully automatable and requires no timing precision.

---

### Recommendation

1. Add a `nonce` parameter to `compute_meta_tx_v0_hash` and include it in the hash via `additional_data_size=1, additional_data=&nonce`, mirroring the pattern already used in `compute_l1_handler_transaction_hash`.
2. Remove or condition the early-return in `check_and_increment_nonce` so that meta-tx v0 calls are subject to the same nonce increment as all other transaction types, or maintain a separate per-contract meta-tx nonce counter that is checked and incremented by the syscall implementation.
3. Alternatively, if the v0 format is intentionally nonce-less for backward compatibility, the OS should maintain a `used_meta_tx_hashes` set and reject any hash that has already been consumed, analogous to the `usedListings` mapping in the external report.

---

### Proof of Concept

1. User calls the `meta_tx_v0` syscall with `contract_address=TOKEN`, `entry_point_selector=transfer`, `calldata=[recipient, 100]` and signs the resulting hash `H = compute_meta_tx_v0_hash(TOKEN, transfer, [recipient,100], chain_id)`.
2. The transaction is included in a block. The signature `sig` is now public in the transaction calldata.
3. Attacker constructs a new transaction that invokes the same contract with the same `(TOKEN, transfer, [recipient,100], sig)` tuple.
4. The OS recomputes `H' = compute_meta_tx_v0_hash(TOKEN, transfer, [recipient,100], chain_id)` — identical to `H` because no nonce is mixed in.
5. Signature verification passes (`H' == H`, same sig).
6. `check_and_increment_nonce` is called with `version=0` and immediately returns without touching state.
7. The transfer executes again. Steps 3–7 repeat until the user's balance is zero. [5](#0-4) [6](#0-5)

// Verifies that the transaction's nonce matches the contract's nonce and increments the
// latter.
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
}
