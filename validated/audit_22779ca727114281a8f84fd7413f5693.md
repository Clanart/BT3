### Title
Fire-and-Forget Relayer Reimbursement in Fast-Transfer Finalization Causes Permanent Fund Loss and Accounting Corruption - (File: near/omni-bridge/src/lib.rs)

### Summary
In `process_fin_transfer_to_other_chain`, when a fast-transfer relayer has pre-funded a recipient on the destination chain, the bridge permanently updates `locked_tokens` accounting and marks the fast transfer as finalised **before** confirming the relayer reimbursement succeeded. The reimbursement is issued via `send_tokens(...).detach()` with no failure callback. If the token transfer fails, the relayer permanently loses their pre-funded amount and the `locked_tokens` accounting is irrecoverably corrupted. The same fire-and-forget pattern also silently drops the relayer fee in `fin_transfer_send_tokens_callback`.

### Finding Description

**Root cause — `process_fin_transfer_to_other_chain`:**

The function executes three irreversible state mutations before the actual token transfer is confirmed:

```rust
// Step 1: Unlock tokens from origin chain — state mutated
self.unlock_tokens_if_needed(
    transfer_message.get_origin_chain(),
    &token,
    transfer_message.amount.0,
);
// Step 2: Lock fee tokens on destination chain — state mutated
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token,
    transfer_message.fee.fee.into(),
);

// Step 3: Fire-and-forget reimbursement — NO callback, NO failure handling
self.send_tokens(token, relayer, U128(amount_without_fee), "")
    .detach();                                          // ← detached promise
// Step 4: Mark finalised — prevents any retry
self.mark_fast_transfer_as_finalised(&fast_transfer.id());
```

If `send_tokens` fails (e.g., `ft_transfer` panics because the relayer account has no storage registered for the token, or `mint` fails because the token contract is paused):

- For **non-deployed tokens**: `ft_transfer` reverts inside the token contract; the tokens remain in the bridge but `locked_tokens[origin][token]` has already been decremented. The bridge now holds tokens that are invisible to its own accounting — a collateralization desync.
- For **deployed tokens**: `mint` fails; no tokens are created, but `locked_tokens` has already been updated on both origin and destination chains. The accounting permanently overstates the destination-chain locked balance.
- The fast transfer is permanently marked as finalised via `mark_fast_transfer_as_finalised`, so neither the relayer nor anyone else can retry the reimbursement.

**Secondary instance — `fin_transfer_send_tokens_callback`:**

After the main recipient transfer succeeds, the relayer fee is also sent with `.detach()`:

```rust
ext_token::ext(token)
    .with_attached_deposit(ONE_YOCTO)
    .with_static_gas(FT_TRANSFER_GAS)          // fixed gas budget
    .ft_transfer(fee_recipient.clone(), transfer_message.fee.fee, None)
    .detach();                                  // ← no callback
```

If `ft_transfer` fails (e.g., `FT_TRANSFER_GAS` is insufficient for the token contract, or the fee recipient has no storage), the relayer's fee is silently dropped. The `FinTransferEvent` has already been emitted and the transfer record removed, so there is no recovery path.

### Impact Explanation

- **Permanent, irrecoverable loss of the relayer's pre-funded amount** — the relayer advanced up to `amount_without_fee` tokens on the destination chain and cannot be reimbursed once the fast transfer is marked finalised. This matches the "permanent freezing / unclaimable settlement of protocol funds" critical impact class.
- **`locked_tokens` accounting corruption** — the bridge's collateralization invariant (locked tokens ≥ outstanding obligations) is broken. For non-deployed tokens the bridge holds untracked surplus; for deployed tokens the destination-chain locked balance is overstated. This matches the "accounting corruption that breaks bridge collateralization" high impact class.
- The fast transfer cannot be re-processed because `is_unified_transfer_finalised` will return `true` for any subsequent attempt.

### Likelihood Explanation

- Any valid proof submission by any unprivileged user triggers `fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_other_chain`. No privileged role is required.
- `ft_transfer` on NEAR fails with a panic if the recipient account has no storage registered for the token. A relayer operating across many tokens may not have pre-registered storage for every token they handle.
- The fixed `FT_TRANSFER_GAS` budget in `fin_transfer_send_tokens_callback` may be insufficient for token contracts with non-trivial `ft_transfer` logic, causing a silent drop.
- The `TODO` comment at line 2065 (`// TODO: not all send_tokens callbacks has the same gas`) confirms the developers are aware of gas budget uncertainty.

### Recommendation

1. Replace `.detach()` with a chained callback in `process_fin_transfer_to_other_chain`. On failure, revert `locked_tokens` mutations and unmark the fast transfer as finalised so the relayer can retry.
2. In `fin_transfer_send_tokens_callback`, replace the detached fee transfer with a tracked promise. On failure, store the unclaimed fee and expose a separate `retry_fee_claim` entry point.
3. Never update finalisation state (`mark_fast_transfer_as_finalised`) before the token transfer outcome is confirmed.

### Proof of Concept

1. User initiates a transfer of 1 000 USDC from Ethereum to Solana with a 10 USDC fee.
2. Fast-transfer relayer pre-funds the Solana recipient with 990 USDC from their own balance.
3. Anyone submits the Ethereum proof via `fin_transfer`.
4. `process_fin_transfer_to_other_chain` executes:
   - `locked_tokens[Eth][USDC]` decremented by 1 000 ✓
   - `locked_tokens[Sol][USDC]` incremented by 10 (fee) ✓
   - `send_tokens(USDC, relayer, 990, "").detach()` — relayer has no USDC storage on NEAR → `ft_transfer` panics inside the token contract → promise result is `Failed`, silently ignored
   - `mark_fast_transfer_as_finalised(...)` — fast transfer permanently closed
5. Outcome:
   - Relayer permanently loses 990 USDC (pre-funded on Solana, never reimbursed on NEAR).
   - Bridge holds 1 000 USDC but `locked_tokens[Eth][USDC]` = 0 — accounting desync.
   - Fast transfer is finalised; `is_unified_transfer_finalised` returns `true` for any retry attempt, blocking recovery. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        &self,
        token: AccountId,
        recipient: AccountId,
        amount: U128,
        msg: &str,

            .saturating_sub(SEND_TOKENS_CALLBACK_GAS) // TODO: not all send_tokens callbacks has the same gas.

            // Unwrap wNEAR and transfer NEAR tokens

                .with_static_gas(WNEAR_WITHDRAW_GAS)
                .with_attached_deposit(ONE_YOCTO)
                .near_withdraw(amount)
                .then(
                    Self::ext(env::current_account_id())
                        .with_static_gas(NEAR_WITHDRAW_CALLBACK_GAS)
                        .near_withdraw_callback(recipient, NearToken::from_yoctonear(amount.0)),
                )
        } else if is_deployed_token {
            let deposit = if msg.is_empty() {
                NO_DEPOSIT
            } else {
                ONE_YOCTO
            };

                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
