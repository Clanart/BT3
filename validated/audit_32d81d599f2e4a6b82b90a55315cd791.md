### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Fund Freeze — (`File: execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the caller-supplied `class_hash` corresponds to a previously declared contract class. Any contract can replace its own class hash with an arbitrary, undeclared value. Because the OS will subsequently be unable to resolve the class for that contract, all funds held in the contract's storage become permanently inaccessible.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash exists in `contract_class_changes` (the dictionary of declared classes):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
```

The developer-acknowledged TODO comment confirms the missing check: `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.`

The `class_hash` field of `ReplaceClassRequest` is fully attacker-controlled — it is whatever the calling contract writes into the syscall buffer. The OS performs no cross-reference against `contract_class_changes` before committing the update to state.

This is structurally identical to the M-14 pattern: a caller-supplied routing parameter (pool ID / class hash) is decoded and applied directly to state without being validated against a stored authoritative set of valid values.

---

### Impact Explanation

After `replace_class` is executed with an undeclared `class_hash`:

1. The contract's on-chain state entry now records an undeclared class hash.
2. Any subsequent call to the contract requires the OS to look up the compiled class for that hash. Because the hash was never declared, no compiled class exists.
3. All entry points of the contract become permanently unreachable.
4. All ERC-20 balances, ETH, or other assets stored in the contract's storage slots are permanently frozen — no withdrawal, transfer, or recovery path exists.

This satisfies the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract on itself. Realistic trigger paths include:

- A contract with an upgradeable pattern (e.g., a proxy or multisig) that has a flaw in its access-control logic, allowing an attacker to supply an arbitrary class hash.
- A malicious contract deployed specifically to receive user deposits and then self-brick via `replace_class` with a garbage hash, permanently locking deposited funds.
- An honest contract whose upgrade logic contains a bug (e.g., missing selector check) that an attacker exploits to pass an undeclared hash.

Because the OS provides no safety net — the validation gap is entirely at the protocol level — any of these paths results in irreversible state corruption. The attacker entry point is an unprivileged transaction sender who either controls or can influence a contract that calls `replace_class`.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, the OS must verify that the hash exists in `contract_class_changes`. Concretely, a `dict_read` on `contract_class_changes` with `key=class_hash` should be performed, and the result must be non-zero (i.e., a compiled class hash has been registered for it). If the lookup returns zero, the syscall should write a failure response instead of updating state.

---

### Proof of Concept

1. Attacker deploys contract `V` that holds user funds (e.g., users call `deposit()` sending ETH).
2. Attacker calls any entry point on `V` that internally invokes the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared hash).
3. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xdeadbeef` from the request buffer.
   - Skips the missing declared-class check.
   - Writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes` for contract `V`.
4. The block is proven and the state root is updated with `V`'s class hash set to `0xdeadbeef`.
5. Any subsequent call to `V` (e.g., `withdraw()`) causes the OS to look up the compiled class for `0xdeadbeef`, finds nothing, and the call fails.
6. All funds deposited into `V` are permanently frozen.
