### Title
Missing Token-Registration Validation in `init_transfer` Allows Permanent Lock of User Funds — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `init_transfer` function in the NEAR bridge contract does not validate that the transferred token is registered for the user-specified destination chain before locking the user's tokens. A user can initiate a transfer to a chain where the token has no registered address mapping, causing the transfer to be permanently stuck in `pending_transfers` with no recovery path, because every subsequent `sign_transfer` call will panic.

---

### Finding Description

In `init_transfer` (lines 523–619 of `near/omni-bridge/src/lib.rs`), the only pre-storage validations are:

1. `recipient.get_chain() != ChainKind::Near` [1](#0-0) 
2. `fee.fee < amount` [2](#0-1) 

There is **no check** that the token (`token_id = env::predecessor_account_id()`) is registered for the destination chain in `token_id_to_address`. The transfer is unconditionally stored in `pending_transfers` and the user's tokens are locked in the bridge. [3](#0-2) 

Later, when `sign_transfer` is called, it attempts to resolve the token address for the destination chain:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [4](#0-3) 

`get_token_address` performs a lookup in `token_id_to_address` keyed by `(ChainKind, AccountId)`. [5](#0-4) 

If the token is not registered for the destination chain, this returns `None` and `sign_transfer` **always panics**. The transfer is permanently stuck in `pending_transfers`. No cancellation or refund path is present in the reviewed code — `remove_transfer_message` is only reachable via `sign_transfer_callback` (zero-fee path) or `claim_fee_callback`, both of which require a successful MPC signature that can never be obtained. [6](#0-5) 

The analog to the external report is direct: just as `newActionProposal()` accepted any `typeStr` string without validating it against the finite set of supported types — causing proposals that could never be finalized — `init_transfer` accepts any destination chain without validating the token is mapped there, causing transfers that can never be signed.

---

### Impact Explanation

User tokens are permanently locked in the bridge contract. The transfer occupies a slot in `pending_transfers` and can never advance to `finalised_transfers`. This is an irrecoverable lock of user funds, matching the **Critical/High** impact category: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

Any unprivileged user can trigger this via the standard NEP-141 `ft_transfer_call` flow — no privileged access is required. A realistic scenario: token X is registered for Ethereum but not yet for Solana (a common state during phased rollouts). A user who specifies `"sol:..."` as the recipient will have their tokens permanently locked. The `OmniAddress` type accepts any syntactically valid Solana address, so no parse error prevents the call from reaching `init_transfer`. [7](#0-6) 

---

### Recommendation

Add a token-registration check inside `init_transfer` before storing the transfer, so the `ft_on_transfer` call fails and the tokens are automatically refunded to the sender:

```rust
require!(
    self.get_token_address(
        init_transfer_msg.get_destination_chain(),
        token_id.clone(),
    ).is_some(),
    BridgeError::TokenNotFound.as_ref()
);
```

This mirrors the validation already present in `sign_transfer` but moves it to the entry point where a refund is still possible.

---

### Proof of Concept

1. Token `usdc.near` is registered for `ChainKind::Eth` (`token_id_to_address[(Eth, usdc.near)] = eth:0xA0b8...`) but **not** for `ChainKind::Sol`.
2. User calls `usdc.near::ft_transfer_call(bridge_id, 1000, '{"InitTransfer":{"recipient":"sol:7xKXtg2CW87d97TXJSDpbD5jBkheTqA83TZRuJosgAsU","fee":"0","native_token_fee":"0"}}')`.
3. `ft_on_transfer` → `init_transfer` → no token-registration check → transfer stored in `pending_transfers`, returns `U128(0)` → 1000 USDC locked in bridge.
4. Relayer calls `sign_transfer(transfer_id, ...)` → `get_token_address(Sol, usdc.near)` returns `None` → `env::panic_str(FailedToGetTokenAddress)`.
5. Transfer is permanently stuck; user's 1000 USDC are irrecoverably locked. [8](#0-7)

    fn init_transfer(
        &mut self,
        sender_id: AccountId,
        signer_id: AccountId,
        token_id: AccountId,
        amount: U128,
        init_transfer_msg: InitTransferMsg,
    ) -> PromiseOrPromiseIndexOrValue<U128> {
        require!(
            init_transfer_msg.recipient.get_chain() != ChainKind::Near,
            BridgeError::InvalidRecipientChain.as_ref()
        );

        self.current_origin_nonce += 1;
        let destination_nonce =
            self.get_next_destination_nonce(init_transfer_msg.get_destination_chain());

        let transfer_message = TransferMessage {
            origin_nonce: self.current_origin_nonce,
            token: OmniAddress::Near(token_id),
            amount,
            recipient: init_transfer_msg.recipient,
            fee: Fee {
                fee: init_transfer_msg.fee,
                native_fee: init_transfer_msg.native_token_fee,
            },
            sender: OmniAddress::Near(sender_id),
            msg: init_transfer_msg.msg.map(String::from).unwrap_or_default(),
            destination_nonce,
            origin_transfer_id: None,
        };
        require!(
            transfer_message.fee.fee < transfer_message.amount,
            BridgeError::InvalidFee.as_ref()
        );
