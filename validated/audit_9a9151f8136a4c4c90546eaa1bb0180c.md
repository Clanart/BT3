### Title
Permanent Fund Freeze When Transfer Amount Normalizes to Zero in `sign_transfer` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR→Foreign transfer with an amount that, after fee subtraction, is smaller than the decimal normalization factor, `sign_transfer` permanently reverts with `InvalidAmountToTransfer`. Because `sign_transfer` is the only path to release a pending transfer and no cancel/rescue mechanism exists, the user's tokens are irrecoverably locked in the bridge contract.

---

### Finding Description

**Root cause — `normalize_amount` floor-divides to zero:**

`sign_transfer` in `near/omni-bridge/src/lib.rs` is the sole mechanism for a relayer to produce the MPC-signed payload that unlocks funds on the destination chain. It calls `normalize_amount` to convert the NEAR-side amount to the destination chain's decimal precision:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

Immediately after, `sign_transfer` enforces a hard revert if the normalized result is zero:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

For any token where `origin_decimals > decimals` (e.g., a NEAR token with 24 decimals bridged to Ethereum where it is represented with 18 decimals, giving a normalization factor of 10^6), any transfer whose `amount_without_fee` is less than `10^(origin_decimals − decimals)` will normalize to 0 and cause `sign_transfer` to panic on every invocation.

**Why funds are permanently locked:**

The user's tokens are already transferred to the bridge contract at the `ft_transfer_call` → `ft_on_transfer` → `init_transfer` stage, before `sign_transfer` is ever called: [3](#0-2) 

The `pending_transfers` map is only cleaned up via `remove_transfer_message`, which is called in:
1. `sign_transfer_callback` — only when `fee.is_zero()`, and only after a *successful* MPC signing (which never happens if `sign_transfer` panics before reaching the MPC call).
2. `claim_fee_callback` — requires a cryptographic proof from the destination chain that the transfer was finalized there, which is impossible if `sign_transfer` never succeeds. [4](#0-3) 

There is no `cancel_transfer`, admin rescue, or any other function that removes a pending transfer entry and refunds the user. The transfer is permanently stuck.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

Any user who initiates a NEAR→Foreign transfer with an amount (after fee) below the decimal normalization threshold has their tokens permanently locked in the bridge contract. The bridge's `pending_transfers` storage retains the entry indefinitely, and no on-chain path exists to recover the funds. This matches the allowed impact class: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** The condition requires a token registered with `origin_decimals > decimals` — a common and intentional configuration for NEAR-native tokens (typically 24 decimals) bridged to EVM chains (typically 18 decimals), yielding a normalization factor of 10^6. Any user transferring fewer than 10^6 base units of such a token (i.e., less than 1 micro-token in NEAR terms) triggers the freeze. This is reachable by any unprivileged user calling `ft_transfer_call` directly, with no special access required.

---

### Recommendation

**Option 1 (preferred):** Validate the normalized amount *before* accepting the transfer in `init_transfer`. If `normalize_amount(amount_without_fee, decimals) == 0`, reject the transfer immediately and return the tokens to the sender, rather than storing an unreleasable pending transfer.

**Option 2:** Add a `cancel_transfer` function that allows the original sender (or an admin) to remove a pending transfer from `pending_transfers` and refund the locked tokens, analogous to the fix recommended in the original report — move the blocking check so it does not permanently trap funds.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (normalization factor = 10^6).
2. User calls `ft_transfer_call` with `amount = 500_000` (< 10^6) and `fee = 0`.
3. `init_transfer` stores the transfer in `pending_transfers`; the 500_000 tokens are now held by the bridge contract. [5](#0-4) 
4. Relayer calls `sign_transfer` for this `transfer_id`.
5. `normalize_amount(500_000, Decimals { decimals: 18, origin_decimals: 24 })` = `500_000 / 1_000_000` = **0** (floor division). [1](#0-0) 
6. `require!(0 > 0, BridgeError::InvalidAmountToTransfer)` panics — the transaction reverts. [6](#0-5) 
7. Every subsequent `sign_transfer` call for this transfer ID produces the same panic.
8. `claim_fee` cannot be called (no destination-chain proof exists).
9. The 500_000 tokens are permanently locked with no recovery path.

        &mut self,
        sender_id: AccountId,
        signer_id: AccountId,
        token_id: AccountId,
        amount: U128,

        require!(
            init_transfer_msg.recipient.get_chain() != ChainKind::Near,
            BridgeError::InvalidRecipientChain.as_ref()
        );

        self.current_origin_nonce += 1;
        let destination_nonce =
            self.get_next_destination_nonce(init_transfer_msg.get_destination_chain());

        let transfer_message = TransferMessage {
            origin_nonce: self.current_origin_nonce,
            token: OmniAddress::Near(token_id),
            amount,
            recipient: init_transfer_msg.recipient,
            fee: Fee {
                fee: init_transfer_msg.fee,
                native_fee: init_transfer_msg.native_token_fee,
            },
            sender: OmniAddress::Near(sender_id),
            msg: init_transfer_msg.msg.map(String::from).unwrap_or_default(),
            destination_nonce,
            origin_transfer_id: None,
        };
        require!(
            transfer_message.fee.fee < transfer_message.amount,
            BridgeError::InvalidFee.as_ref()
        );

        let required_storage_balance =
            self.required_balance_for_init_transfer_message(transfer_message.clone());

        let message_storage_account_id = transfer_message
            .calculate_storage_account_id(init_transfer_msg.external_id.map(String::from));

        // Choose storage payer or whether to yield execution until storage is available
        if self
            .try_to_transfer_balance_from_message_account(
                &message_storage_account_id,
                NearToken::from_yoctonear(init_transfer_msg.native_token_fee.0),
                &signer_id,
                required_storage_balance,
            )
            .is_ok()
            || (self.has_storage_balance(
                &signer_id,
                required_storage_balance.saturating_add(NearToken::from_yoctonear(
                    init_transfer_msg.native_token_fee.0,
                )),
            ) && (init_transfer_msg.native_token_fee.0 == 0
                || !self.acl_has_role(Role::NativeFeeRestricted.into(), signer_id.clone())))
        {
            PromiseOrPromiseIndexOrValue::Value(
                self.init_transfer_internal(transfer_message, signer_id),
            )
        } else {
            let promise_index = env::promise_yield_create(
                "init_transfer_resume",
                json!({
                    "transfer_message": transfer_message,
                    "message_storage_account_id": message_storage_account_id,
                    "storage_owner": signer_id,
                })
                .to_string()
                .as_bytes(),
                INIT_TRANSFER_RESUME_GAS,
                GasWeight(0),
                PROMISE_REGISTER_ID,
            );

            let yield_id: CryptoHash = env::read_register(PROMISE_REGISTER_ID)
                .near_expect(BridgeError::ReadPromiseRegister)
                .try_into()
                .near_expect(BridgeError::ReadPromiseYieldId);

            let required_storage_balance = self.add_promise(&message_storage_account_id, &yield_id);

            self.update_storage_balance(
                env::current_account_id(),
                required_storage_balance,
                NearToken::from_yoctonear(0),
            );

            env::log_str(&format!(
                "Yield init transfer until storage is available at {message_storage_account_id}"
            ));

            PromiseOrPromiseIndexOrValue::PromiseIndex(promise_index)
        }
    }
