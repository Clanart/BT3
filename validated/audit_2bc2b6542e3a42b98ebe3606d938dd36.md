However, the **Amino path** (`legacyDecodeAminoSignDoc`) has no such guard, and the `VerifySignature` function in the ante handler constructs `txBytes` directly from `tx.GetTimeoutHeight()` without any rejection of non-zero values.

## Attack Analysis

**Precondition:** User signs T1 (legacy EIP-712, Amino path) with `timeout_height = N > 0`, intending the tx to expire at block N.

**Attack:**
1. Attacker observes T1 (e.g., from mempool or a failed/expired submission) and extracts `extOpt.FeePayerSig`.
2. Attacker constructs T2 with identical `msgs`, `fees`, `memo`, `sequence`, `account_number`, `chain_id`, but `timeout_height = 0`.
3. Attacker submits T2 with T1's signature in `FeePayerSig`.
4. `VerifySignature` calls `LegacyWrapTxToTypedData` with T2's `txBytes` (which includes `timeout_height: "0"`).
5. Because `timeout_height` is absent from the `Tx` type schema, the EIP-712 hash is identical to T1's hash.
6. `ethcrypto.Ecrecover(sigHash, feePayerSig)` recovers the correct public key.
7. T2 passes all signature checks and executes.

**What the attacker can change:** Only `timeout_height`. The `msgs`, `fee`, `memo`, `sequence`, `chain_id`, and `account_number` fields ARE in the schema and would change the hash if modified. The question's broader claim about changing msgs or fees is **not supported** — those fields are protected.

**Actual impact:** The attacker can strip a user's intended transaction expiry, causing a tx the user intended to be time-limited to execute indefinitely. This is a partial EIP-712 authorization bypass enabling replay beyond the user's intended expiry window.

## Severity Assessment

The impact is real but **narrow**:
- The same msgs execute with the same fees — no unauthorized fund transfer or different action.
- The only bypassed constraint is `timeout_height`.
- Practical harm: a user who signs a time-sensitive tx (e.g., a delegation intended to expire before a governance vote) loses control over the expiry window.

This fits the "High" category as an EIP-712 authorization bypass enabling replay, but the scope is limited to timing control, not to forged execution of different messages or fee theft.

The non-legacy EIP-712 path (`GetEIP712BytesForMsg`) has the same schema omission but is protected upstream by the `TimeoutHeight != 0` rejection in `decodeProtobufSignDoc`. The legacy Amino path has no such protection.

### Title
Legacy EIP-712 `timeout_height` Omission Allows Signature Replay Beyond User-Intended Expiry — (`ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

### Summary
The legacy EIP-712 `Tx` type schema omits `timeout_height`, so two transactions differing only in that field produce identical EIP-712 hashes. An attacker who observes a signed legacy EIP-712 transaction with a non-zero `timeout_height` can reuse its signature on a crafted transaction with `timeout_height = 0`, bypassing the user's intended expiry.

### Finding Description
`extractMsgTypes` in `ethereum/eip712/eip712_legacy.go` defines the `Tx` EIP-712 struct without a `timeout_height` field (explicitly commented out). `VerifySignature` in `ante/cosmos/eip712.go` constructs `txBytes` via `legacytx.StdSignBytes` with `tx.GetTimeoutHeight()`, which includes `timeout_height` in the JSON. `LegacyWrapTxToTypedData` unmarshals this JSON into `txData` (the EIP-712 message), but since `timeout_height` is absent from the type schema, `TypedDataAndHash` ignores it. Two transactions with identical fields except `timeout_height` produce the same signable hash.

### Impact Explanation
An attacker can take a valid legacy EIP-712 signature for a transaction with `timeout_height = N` and replay it against a crafted transaction with `timeout_height = 0`. The user's intended expiry constraint is stripped. The same msgs and fees execute, so there is no unauthorized fund transfer, but the user loses control over when their transaction can be submitted.

### Likelihood Explanation
Requires the legacy EIP-712 path (`ExtensionOptionsWeb3Tx`), which is deprecated but still active in `newLegacyCosmosAnteHandlerEip712`. The attacker must observe a signed transaction with a non-zero `timeout_height` before it expires. This is realistic via mempool observation.

### Recommendation
Add `timeout_height` back to the `Tx` type schema in `extractMsgTypes` (`ethereum/eip712/eip712_legacy.go`) and ensure `StdSignBytes` always produces a non-empty `timeout_height` value (e.g., `"0"` when unset) so the field is consistently hashed. Alternatively, add a rejection guard for `timeout_height != 0` in the Amino decode path, mirroring the existing guard in `legacyDecodeProtobufSignDoc`.

### Proof of Concept
Construct two Amino `StdSignDoc` payloads identical except `timeout_height` (`"0"` vs `"100"`). Call `LegacyGetEIP712BytesForMsg` on both. Observe that both return identical byte slices, confirming the hash collision. Sign one; verify the signature passes for the other via `pubKey.VerifySignature`.
