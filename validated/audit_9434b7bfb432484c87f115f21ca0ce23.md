### Title
Unvalidated P2P Transaction Hashes Corrupt L1 Provider Committed Set via `l2_sync_task` — (`crates/apollo_l1_provider/src/catchupper.rs`)

### Summary

During the L1 provider catchup phase, `l2_sync_task` calls `StateSync::get_block` which reads transaction hashes directly from storage. Those hashes were written by the p2p sync client without verifying them against the `transaction_commitment` stored in the block header. A malicious p2p peer can supply `L1Handler` transactions with attacker-chosen hashes; those hashes flow unmodified into `l1_provider_client.commit_block(...)`, corrupting `TransactionManager.records` for block N.

### Finding Description

**Step 1 — Missing validation in p2p body sync.**

`TransactionStreamFactory::parse_data_for_block` in `crates/apollo_p2p_sync/src/client/transaction.rs` accepts `transaction_hash` from the peer and pushes it directly into `block_body.transaction_hashes` with an explicit unresolved TODO:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

The header for the same block is stored with a `transaction_commitment` field (written by `append_header`), but no code ever cross-checks the stored body hashes against that commitment. [2](#0-1) 

**Step 2 — `StateSync::get_block` trusts storage unconditionally.**

`StateSync::get_block` reads `get_block_transactions_with_hash` from storage, classifies each entry by transaction type, and returns the result. It reads `block_header_commitments` from the header but never compares `transaction_commitment` against the actual hashes it just read:

```rust
for (tx, tx_hash) in block_transactions_with_hash {
    match tx {
        Transaction::L1Handler(_) => l1_transaction_hashes.push(tx_hash),
        _ => account_transaction_hashes.push(tx_hash),
    }
}
let block_header_commitments = Option::<BlockHeaderCommitments>::try_from(&block_header)?;
Ok(SyncBlock { ..., l1_transaction_hashes, block_header_commitments })
``` [3](#0-2) 

The `SyncBlock` type documents this trust assumption explicitly:

> "Blocks that came from the state sync are trusted. Therefore, SyncBlock doesn't contain data needed for verifying the block" [4](#0-3) 

**Step 3 — `l2_sync_task` forwards the corrupted hashes to `commit_block`.**

```rust
l1_provider_client
    .commit_block(
        block.l1_transaction_hashes.into_iter().collect(),
        l1_handler_rejected_tx_hashes,
        current_height,
    )
    .await
    .unwrap();
``` [5](#0-4) 

`commit_block` calls `apply_commit_block` which writes the supplied hashes into `TransactionManager.records` as `Committed` for block N with no further verification. [6](#0-5) 

### Impact Explanation

Two concrete corruptions result:

1. **Phantom commits.** If the attacker supplies hashes that match real pending L1Handler txs (all L1 events are public), those txs are marked `Committed` in `TransactionManager.records` without ever being executed on L2. The L1 provider will never re-propose them, silently skipping L1→L2 messages.

2. **Stale pending set.** If the attacker supplies hashes that do not match any real pending tx, the real L1Handler txs for block N remain `Pending` and are re-proposed in future blocks, potentially causing double-execution of L1→L2 messages (subject to whether the execution layer guards against re-consumption).

Both outcomes cause the L1 provider's committed set to diverge from the canonical chain, meeting the "High" impact threshold: valid L1Handler transactions are rejected from (or incorrectly admitted to) sequencing.

### Likelihood Explanation

- Exploitable only during the catchup phase (node startup), which is a bounded but real window.
- Requires a malicious p2p peer that the node connects to during sync — no operator privileges needed.
- L1 event hashes are public, so the attacker can craft maximally damaging payloads (targeting real pending hashes).
- The missing guard is explicitly acknowledged with a TODO, confirming it is a known gap that has not been closed.

### Recommendation

In `TransactionStreamFactory::parse_data_for_block`, after collecting all transactions for a block, recompute the `transaction_commitment` from the collected `(transaction, transaction_hash)` pairs using `calculate_transaction_commitment` and compare it against the value stored in the block header (`block_header.transaction_commitment`). Reject the peer (return `BadPeerError`) if the values differ. This closes the gap noted by the existing TODO.

Separately, `StateSync::get_block` should assert that the hashes it reads from storage are consistent with the stored `transaction_commitment` before returning a `SyncBlock`, so that even a storage-layer inconsistency is caught before it reaches the L1 provider.

### Proof of Concept

1. Start a sequencer node configured to use p2p sync.
2. Intercept the p2p transaction response for block N and replace the `transaction_hash` field of each `L1Handler` `FullTransaction` with attacker-chosen values (e.g., the hashes of real pending L1Handler txs scraped from L1 events).
3. Allow the node to enter catchup mode (trigger by sending a `commit_block` for a height above the provider's current height).
4. Observe that `l2_sync_task` calls `commit_block` with the attacker-chosen hashes.
5. Assert that `TransactionManager.records` now marks those hashes as `Committed` for block N, while the real L1Handler tx hashes remain `Pending` — or, if the attacker used real pending hashes, that those txs are now permanently skipped.

The integration test described in the question (write a block to storage with mismatched `transaction_commitment` vs actual L1Handler tx hashes, call `StateSync::get_block`, assert the returned `l1_transaction_hashes` do not match the commitment field) would directly demonstrate the missing guard.
