### Title
Unverified `prev_block_hash` and Unverified `header_commitments` in Block Hash Computation Break Inter-Block Linkage — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/block_hash.cairo`)

---

### Summary

In `get_block_hashes` (`block_hash.cairo`), the `previous_block_hash` and all five `BlockHeaderCommitments` fields (`transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`, `packed_lengths`) are set exclusively by prover-controlled hints with **no Cairo-level assertion** enforcing their correctness. The only "check" present — `%{ CheckBlockHashConsistency %}` — is a hint (Python code), not a Cairo constraint, and therefore cannot enforce anything at the proof level. This is the direct structural analog of the reported vulnerability: a linkage check that exists in form but is not enforced.

---

### Finding Description

**Root cause — `block_hash.cairo`, `get_block_hashes` (lines 55–81):**

```cairo

    previous_block_hash: felt, new_block_hash: felt
) {
    alloc_locals;
    local previous_block_hash;                        // (1) guessed, never asserted
    // Currently, the header commitments and gas prices are not computed by the OS.
    // TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
    local header_commitments: BlockHeaderCommitments*; // (2) guessed, never asserted
    local gas_prices_hash;                             // (3) guessed, never asserted
    local starknet_version;                            // (4) guessed, never asserted

    %{ GetBlockHashes %}   // hint sets all four locals above

    let block_hash = calculate_block_hash(
        block_info=block_info,
        header_commitments=header_commitments,
        gas_prices_hash=gas_prices_hash,
        state_root=state_root,
        previous_block_hash=previous_block_hash,
        starknet_version=starknet_version,
    );

    %{ CheckBlockHashConsistency %}  // hint only — NOT a Cairo assertion

    return (previous_block_hash=previous_block_hash, new_block_hash=block_hash);
}
``` [1](#0-0) 

Every input to `calculate_block_hash` except `block_info` (block number, sequencer address, timestamp) and `state_root` is prover-supplied with no constraint. The `new_block_hash` is therefore a function of arbitrary prover-chosen values. Critically, `previous_block_hash` — the field that chains blocks together — is never verified against any committed prior block output.

### Impact Explanation

**Unintended chain split (network partition) — High.**

Because `prev_block_hash` is never constrained by a Cairo assertion, a prover can produce a valid proof for block N that claims to descend from any arbitrary block hash, not necessarily the actual block N-1. Two independently submitted proofs for the same block number can each claim different parent hashes and both pass OS-level verification. This creates a fork at the proof layer: two divergent chains of blocks, each with valid proofs, can coexist. Nodes that accept proofs without independently re-verifying the `prev_block_hash` chain will split.

Additionally, because `header_commitments` (including `transaction_commitment` and `event_commitment`) are entirely prover-supplied, the `new_block_hash` does not cryptographically commit to the actual transactions executed. A proof can be replayed or reused across different transaction sets that share the same `state_root`, analogous to the replay attack described in the external report.

---

### Likelihood Explanation

The OS code contains explicit `TODO` comments acknowledging both missing checks, with target dates of 2026 and 2027. Until those TODOs are resolved, any prover (in a decentralized proving system where the prover role is not exclusively held by a single trusted party) can exploit this. The vulnerability requires no special privilege beyond the ability to submit a proof — the standard prover role. The code path is exercised for every block.

---

### Recommendation

1. **Enforce `prev_block_hash` with a Cairo assertion** inside `get_block_hashes`: after computing `new_block_hash` for block N-1, assert that the `previous_block_hash` supplied for block N equals that computed value. Remove the delegation to "the consumer."
2. **Compute `header_commitments` inside the OS** from the actual transaction, event, receipt, and state-diff data produced during block execution, rather than accepting them from a hint. The TODO at line 61 of `block_hash.cairo` tracks this.
3. **Enforce `old_block_hash`** in `write_block_number_to_block_hash_mapping` by deriving it from the committed OS output of the corresponding prior block rather than accepting it from a hint.
4. Replace `%{ CheckBlockHashConsistency %}` with a Cairo `assert` statement so the check is part of the proof constraint system, not merely advisory Python code.

---

### Proof of Concept

1. Prover executes block N legitimately, producing a valid `state_root_N`.
2. In `get_block_hashes`, the prover's hint `GetBlockHashes` sets `previous_block_hash = H_fake` (any value other than the actual block N-1 hash) and sets `header_commitments` to arbitrary values (e.g., all zeros).
3. `calculate_block_hash` computes `new_block_hash` from these prover-chosen inputs. No Cairo assertion rejects this.
4. `%{ CheckBlockHashConsistency %}` runs as Python but cannot add a constraint to the proof; the proof is generated with `prev_block_hash = H_fake` in the output.
5. The OS output header is serialized with `prev_block_hash = H_fake` and `new_block_hash = H_N_fake`.
6. A second prover independently does the same for block N, setting `previous_block_hash = H_real` (the actual block N-1 hash) and `header_commitments` to the real values, producing `new_block_hash = H_N_real`.
7. Both proofs are valid at the OS level. Nodes that accept either proof now hold divergent views of the chain — a network partition.
8. Separately, the prover sets `old_block_hash` in `write_block_number_to_block_hash_mapping` to an arbitrary value; any contract calling `get_block_hash(old_block_number)` receives the fabricated hash, potentially corrupting application-layer logic that depends on historical block hashes for security decisions.
