### Title
Push-Transfer to Blacklisted Recipient Permanently Freezes Bridged Native Tokens — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`finTransfer` in `OmniBridge.sol` uses a push-transfer pattern to deliver native ERC-20 tokens (e.g. USDC) directly to the MPC-signed recipient. If that recipient is on the USDC (or any token-issuer) blacklist, every call to `finTransfer` reverts. Because the MPC signature commits to the exact recipient and there is no emergency-processing path, pull-pattern fallback, or on-chain refund route back to NEAR, the bridged funds are permanently irrecoverable.

### Finding Description

`finTransfer` executes the following sequence:

1. Checks and marks the nonce used (`completedTransfers[payload.destinationNonce] = true`) — line 287.
2. Verifies the MPC signature over a payload that encodes `tokenAddress`, `amount`, and `recipient` — line 311.
3. For non-bridge, non-multitoken ERC-20s (the `else` branch), calls `IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount)` — line 351. [1](#0-0) [2](#0-1) 

If `safeTransfer` reverts (e.g. USDC blacklists the recipient), the entire transaction reverts, including the nonce write. The nonce is therefore never consumed — but the MPC signature is permanently bound to the blacklisted recipient. The MPC will not re-sign for a different recipient because the NEAR-side transfer record is immutable once the `InitTransfer` event is emitted and the tokens are locked/burned. There is no emergency-processing function, no pull-pattern claimable balance, and no admin rescue path in `OmniBridge.sol`. [3](#0-2) 

The EVM `SECURITY.md` does not list this scenario as a known or accepted issue. [4](#0-3) 

The same structural flaw exists in the StarkNet `fin_transfer`: `_set_transfer_finalised` is called before the `IERC20Dispatcher.transfer`, and if the transfer fails the whole Cairo transaction reverts, leaving the nonce unconsumed and the funds undeliverable with no fallback. [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who bridges USDC (or any token with issuer-controlled transfer restrictions) from NEAR to EVM to a recipient that is, or becomes, blacklisted will have their tokens permanently locked:

- Tokens are already burned/locked on NEAR at `InitTransfer` time.
- Every `finTransfer` attempt on EVM reverts; the nonce is never consumed.
- The MPC signature cannot be reused for a different recipient.
- No on-chain mechanism exists to refund the NEAR-side locked balance.
- The contract is UUPS-upgradeable, so an admin *could* deploy a rescue, but no such path exists in the current code and it requires trusted intervention outside the protocol.

### Likelihood Explanation

**Medium.** USDC's blacklist is actively maintained by Circle. A recipient address can be blacklisted:

- After the user initiates the NEAR-side `InitTransfer` but before the EVM `finTransfer` is submitted (the MPC signing and relaying window can be minutes to hours).
- If the recipient is a smart contract that later gets sanctioned.
- If the user's address is added to the OFAC/Circle list between initiation and finalization.

USDC is one of the most commonly bridged stablecoins; the scenario is realistic and has precedent in other bridge incidents.

### Recommendation

Implement a pull-pattern fallback for native token delivery in `finTransfer`:

1. On transfer failure, instead of reverting, store the claimable amount in a `mapping(address => mapping(address => uint256)) public claimable` mapping.
2. Expose a `claim(address token)` function that lets the original recipient (or a governance-designated substitute) withdraw.
3. Alternatively, accept an optional `fallbackRecipient` in the MPC-signed payload so the MPC can sign a secondary delivery address when the primary is blocked.

This mirrors the "pull pattern for all token withdrawals" recommendation in the referenced Moloch report.

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR and calls `init_transfer` targeting her EVM address `0xAlice`. NEAR locks 10,000 USDC and emits `InitTransfer`.
2. Circle blacklists `0xAlice` (e.g. due to a regulatory action) before the MPC signature is submitted.
3. The MPC signs a `TransferMessagePayload` with `tokenAddress = USDC`, `amount = 10000e6`, `recipient = 0xAlice`.
4. Any relayer calls `OmniBridge.finTransfer(sig, payload)`.
5. Execution reaches line 351: `IERC20(USDC).safeTransfer(0xAlice, 10000e6)` — USDC reverts because `0xAlice` is blacklisted.
6. The entire transaction reverts. `completedTransfers[nonce]` remains `false`.
7. No alternative `finTransfer` call is possible (signature is bound to `0xAlice`).
8. Alice's 10,000 USDC is permanently locked — undeliverable on EVM, unrefundable on NEAR. [6](#0-5)

                : Borsh.encodeBytes(payload.message)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {

# Security Notes

## Design Decisions (Non-Issues)

These patterns have been reviewed and confirmed as intentional. Do not flag or "fix" them.

- **Fee-on-transfer tokens not supported**: `initTransfer` emits the requested `amount`, not the actual received balance. Fee-on-transfer and rebasing tokens are intentionally unsupported
- **`logMetadata` and `deployToken` are permissionless**: Anyone can call `logMetadata` for any ERC20, and anyone can submit a valid MPC signature to `deployToken`. This is by design — the bridge is fully permissionless
- **`ENearProxy.burn` uses empty NEAR recipient**: `eNear.transferToNear(amount, "")` is intentional — `transferToNear` is a legacy method used purely as a burn mechanism. The actual NEAR recipient is tracked in the OmniBridge `InitTransfer` event
- **`deployToken` signature has no chain ID**: Metadata signatures are intentionally chain-agnostic — one NEAR-side signature deploys the same token on all EVM chains

## Known Issues

Low-severity items acknowledged but not yet addressed:

- **`addCustomToken` can overwrite existing mappings** (H-01): Admin-only function. No existence check — calling with an already-mapped `nearTokenId` silently overwrites `nearToEthToken`. Accepted as operational risk
- **`pause(flags)` replaces all flags** (H-02): `_pause(flags)` does full replacement, not bitwise OR. Calling `pause(PAUSED_INIT_TRANSFER)` when `PAUSED_FIN_TRANSFER` is set will unpause finTransfer. Use `pauseAll()` for emergencies
- **`BridgeToken.initialize` stores metadata redundantly** (L-01): `__ERC20_init(name_, symbol_)` writes to parent storage that is never read (getters are overridden). Minor gas waste on init
- **`require` strings instead of custom errors** (L-02): Several locations use `require` with string messages instead of custom errors (`OmniBridge.sol:150,204,556`, `SelectivePausableUpgradable.sol:100,107`, `ENearProxy.sol:56,76,86`)
- **`OmniBridgeWormhole` has no `__gap`** (L-04): Three storage variables with no gap array. Safe as a leaf contract but would need a gap if inherited from
- **`PayloadType.ClaimNativeFee` defined but unused** (L-05): Enum value 2 is never referenced. Native fees are recovered via `finTransfer` with `tokenAddress=address(0)`
