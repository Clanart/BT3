Audit Report

## Title
Unattributed Router ETH Balance Enables Theft via Unguarded `refundETH` and Free Swaps via `pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay` function consumes `address(this).balance` without verifying the ETH belongs to the current payer, and `refundETH()` is `external` with no access control, sending the router's entire ETH balance to any caller. When a user calls `exactOutputSingle` directly with excess `msg.value` (the natural pattern, since exact input is unknown pre-execution), the unconsumed ETH is stranded on the router and immediately claimable by any subsequent caller, causing direct, unrecoverable loss of user principal.

## Finding Description

**Root cause 1 — `pay` uses unattributed `address(this).balance`:** [1](#0-0) 

When `token == WETH`, `pay` reads `address(this).balance` and uses it unconditionally. There is no check that the ETH on the router was deposited by the current `payer`. Any ETH left from a prior transaction is silently consumed to fund the current caller's swap.

**Root cause 2 — `refundETH` has no access control or attribution:** [2](#0-1) 

`refundETH()` is `external payable` with no guard. Any caller receives the router's entire ETH balance regardless of who deposited it.

**How ETH becomes stranded:**

`exactOutputSingle` is `external payable`: [3](#0-2) 

The actual `amountIn` is only known after the pool executes the swap. A user who sends `msg.value = X` but the pool requires only `Y < X` will have `X - Y` ETH stranded on the router after the call returns. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers — it does not prevent ETH from being received via payable function calls. [4](#0-3) 

The test suite confirms `exactOutputSingle{value: nativePart}` is called directly without a `multicall`+`refundETH` wrapper, establishing this as a supported usage pattern: [5](#0-4) 

(That specific test passes `_assertRouterEmpty()` only because `nativePart` is set to exactly `quotedIn / 2`, which is fully consumed. A user sending any overage — the natural conservative pattern — strands the remainder.)

**`sweepToken` and `unwrapWETH9` are equally unguarded:** [6](#0-5) 

Both are `public payable` with no access control and transfer the router's entire balance of the specified token to a caller-chosen `recipient`.

## Impact Explanation

**ETH theft via `refundETH`:** User A calls `exactOutputSingle{value: 1 ether}` where the pool requires 0.6 ETH. The `pay` function wraps 0.6 ETH and sends it to the pool; 0.4 ETH remains on the router. An attacker immediately calls `refundETH()` and receives 0.4 ETH. User A's 0.4 ETH is permanently lost with no recourse.

**Free swap via `pay`:** With 0.4 ETH stranded, an attacker calls `exactInputSingle{value: 0}` with `tokenIn = WETH` and `amountIn = 0.4 ether`. The `pay` function enters the `nativeBalance >= value` branch, wraps the stranded ETH, and sends it to the pool. The attacker receives swap output without spending any ETH or WETH.

Both vectors result in direct, unrecoverable loss of user principal — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation

`exactOutputSingle` is the highest-risk entry point: users cannot know the exact input amount before execution, so sending a conservative `msg.value` overage is the natural and expected pattern. The test suite itself demonstrates calling `exactOutputSingle{value: nativePart}` directly without a paired `refundETH`, confirming this is a supported usage path. MEV bots routinely monitor contract balances and can atomically steal stranded ETH in the same block. No special privileges, approvals, or setup are required beyond the victim's own transaction.

## Recommendation

1. **Track per-caller ETH deposits in transient storage.** When `msg.value > 0` is received in a payable swap function, record `(msg.sender → msg.value)` in a transient slot. In `refundETH`, only return the amount attributed to `msg.sender`, not the entire balance.
2. **Restrict `refundETH` to the attributed depositor.** At minimum, only refund the amount recorded for `msg.sender` in the transient attribution slot, not `address(this).balance`.
3. **Apply the same guard to `sweepToken` and `unwrapWETH9`.** Restrict these to `multicall`-only contexts with transient attribution, or require the caller to be the entity that deposited the tokens.
4. **In `pay`, consume only `msg.value` attributed to the current payer.** Read from the transient slot rather than `address(this).balance`.

## Proof of Concept

**ETH theft via `refundETH`:**

```solidity
// User A calls exactOutputSingle directly (not via multicall)
// Pool requires 0.6 ETH; user sends 1 ETH
router.exactOutputSingle{value: 1 ether}(ExactOutputSingleParams({
    tokenIn: WETH,
    amountOut: someTokenAmount,
    amountInMaximum: 1 ether,
    ...
}));
// pay() wraps 0.6 ETH, sends to pool; 0.4 ETH stranded on router

// Attacker (same or next block) calls refundETH — no access control
router.refundETH();
// Attacker receives 0.4 ETH; User A's funds permanently lost
```

**Free swap via `pay` consuming stranded ETH:**

```solidity
// 0.4 ETH stranded on router from User A above

router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn: WETH,
    amountIn: 0.4 ether,  // <= stranded balance
    ...
}));
// pay() sees nativeBalance (0.4 ETH) >= value (0.4 ETH)
// Wraps User A's 0.4 ETH, sends to pool
// Attacker receives swap output without spending any ETH or WETH
```

A Foundry test can reproduce this by: (1) calling `exactOutputSingle{value: X}` where `X > quotedIn`, (2) asserting `address(router).balance > 0`, (3) pranking as a second address and calling `refundETH()`, (4) asserting the attacker received the stranded ETH and the original user's balance is reduced by the full `X`.
