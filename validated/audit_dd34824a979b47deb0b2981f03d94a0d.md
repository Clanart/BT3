### Title
Missing shard_id cross-check in `set_state_header` allows malicious peer to install wrong shard's state - (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `shard_id` parameter and a `ShardStateSyncResponseHeader` but never asserts that the embedded chunk's own `shard_id` field equals the `shard_id` parameter. Every validation guard it runs is satisfiable with a genuine chunk from a *different* shard, so the header is stored under `StateHeaderKey(A, sync_hash)` while carrying shard B's `prev_state_root`. Subsequent `set_state_part` calls then validate and persist parts against shard B's state root under shard A's key, and `set_state_finalize` applies that state to shard A's trie.

---

### Finding Description

`set_state_header` performs five checks before writing to `DBCol::StateHeaders`:

**Check 1 – `validate_chunk_proofs`** [1](#0-0) 

`validate_chunk_proofs` verifies the chunk's internal hash, tx root, and receipts root. [2](#0-1)  It never inspects `chunk.shard_id()` against any external parameter. A well-formed chunk from shard B passes this check unconditionally.

**Check 2 – Merkle inclusion in `chunk_headers_root`** [3](#0-2) 

`verify_path` reduces to `compute_root_from_path(path, hash_of_item) == root`. [4](#0-3)  It proves the chunk hash appears *somewhere* in the Merkle tree; it does **not** bind the proof to a specific leaf index (shard position). Shard B's chunk is a genuine leaf of the same `chunk_headers_root`, so its Merkle path verifies correctly even when the caller claims `shard_id = A`.

**Check 3 – prev_chunk Merkle inclusion** [5](#0-4)  Same position-agnostic `verify_path` call; same bypass.

**Check 4 – Incoming receipts proofs** [6](#0-5) 

The receipts hash is computed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the *parameter* `shard_id = A`. [7](#0-6)  The attacker supplies the `receipts` and `root_proofs` fields. Because shard A's actual incoming receipts and their outgoing-receipts-root proofs are public on-chain data, the attacker can populate these fields with legitimate shard-A receipt data, satisfying the proof check while the embedded chunk still belongs to shard B.

**Check 5 – `validate_state_root_node`** [8](#0-7) 

Validates the supplied `state_root_node` against `chunk_inner.prev_state_root()` — shard B's state root. The attacker, who controls shard B's state sync data, can supply a valid root node for shard B.

**Storage** [9](#0-8) 

The header is written under `StateHeaderKey(shard_id=A, sync_hash)` carrying shard B's `prev_state_root`.

**`set_state_part` consequence** [10](#0-9) 

Reads the stored header for shard A, extracts shard B's state root, validates incoming parts against it, and stores them under `StatePartKey(sync_hash, A, part_id)`. Valid parts for shard B's trie pass `validate_state_part`.

---

### Impact Explanation

A syncing node installs shard B's full trie under shard A's identity. After `set_state_finalize` the node operates with the wrong state for shard A: it will produce incorrect chunk hashes, fail state-root checks against the canonical chain, and be unable to participate correctly in consensus for that shard. The corruption is persistent (written to `DBCol::StateParts` and applied to the trie).

---

### Likelihood Explanation

Any peer reachable by the syncing node can act as a state sync provider — no validator or operator privilege is required. All inputs needed to craft the malicious header (shard B's chunk, its Merkle path, shard A's receipt proofs, shard B's state root node) are obtainable from public chain data or a legitimate state sync provider. The attack requires no cryptographic forgery.

---

### Recommendation

Add an explicit shard-id guard immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: reject headers whose embedded chunk belongs to a different shard
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be placed before any other validation so that a mismatched chunk is rejected immediately. [11](#0-10)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
