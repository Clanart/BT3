### Title
Missing Declared-Class Validation in `replace_class` Syscall Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts any arbitrary felt value as the new class hash without verifying that the value corresponds to a declared contract class. A malicious or buggy contract can call `replace_class` with an undeclared class hash, permanently rendering itself uncallable and freezing any funds it holds. The OS itself acknowledges this gap with a `TODO` comment but does not enforce the check.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS reads `request.class_hash` directly from the syscall request and writes it into `contract_state_changes` without any validation that the hash corresponds to a class that has been declared on-chain:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `TODO` comment at line 898 explicitly acknowledges the missing check. The OS never consults `contract_class_changes` (the dict tracking newly declared classes in the current block) nor any prior-state class registry to confirm that `class_hash` is a valid, deployed class. Any felt value — including one that has never been declared — is accepted and committed to state.

Compare this with `execute_declare_transaction` in `transaction_impls.cairo` (lines 738–743), which does enforce that the class hash is the result of a valid Sierra class hash calculation:

```cairo
let expected_class_hash = finalize_class_hash(
    contract_class_component_hashes=contract_class_component_hashes
);
with_attr error_message("Invalid class hash pre-image.") {
    assert [class_hash_ptr] = expected_class_hash;
}
```

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and the block is proven and committed to state, the contract becomes permanently uncallable:

- Every subsequent invocation of the contract will fail at class resolution because the class does not exist.
- The `replace_class` syscall itself requires the contract to be callable, so the state cannot be self-corrected.
- Any ERC-20 token balances, ETH/STRK, or other assets held in the contract's storage are permanently inaccessible.
- Recovery requires a protocol-level emergency intervention (hard fork or governance override), which is outside normal operation.

---

### Likelihood Explanation

The `replace_class` syscall is available to every deployed contract with no caller restriction. Two realistic paths exist:

1. **Malicious deployer**: An attacker deploys a contract that accepts user deposits (e.g., a vault, pool, or staking wrapper), accumulates funds, and then calls `replace_class` with an arbitrary invalid felt as the class hash. All deposited funds are permanently frozen.

2. **Buggy upgrade logic**: A legitimate contract with an upgrade mechanism passes an incorrect class hash (e.g., a hash that was never declared, or a hash from a different chain). The OS silently accepts it, and the contract is bricked.

Both paths are reachable by an unprivileged transaction sender with no special role required.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, `execute_replace_class` must verify that the requested class hash is a declared class. Concretely:

1. Check `contract_class_changes` for a class declared in the current block.
2. If not found there, verify against the pre-block state (i.e., the class hash must have a non-zero compiled class hash in the global class registry).

This mirrors the validation already enforced in `execute_declare_transaction` and closes the gap acknowledged by the existing `TODO` comment.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts ERC-20 deposits and exposes a public `drain()` function.
2. Users deposit funds into `MaliciousVault`.
3. Attacker calls `drain()`, which internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary undeclared felt).
4. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xdeadbeef` from the request.
   - Skips the missing declared-class check (line 898 `TODO`).
   - Writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. The block is proven and committed. `MaliciousVault`'s on-chain class hash is now `0xdeadbeef`.
6. All subsequent calls to `MaliciousVault` fail at class resolution — the class does not exist.
7. All user deposits are permanently frozen with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2)

    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
