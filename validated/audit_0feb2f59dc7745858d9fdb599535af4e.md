### Title
Cross-shard state header substitution: `set_state_header` missing shard_id binding allows unprivileged peer to install shard X's trie into shard Y's flat storage — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id=Y` but never checks that the embedded chunk actually belongs to shard Y. Because `verify_path` only proves membership in the block's `chunk_headers_root` Merkle tree (not position), a peer can supply shard X's chunk — which is legitimately in that tree at position X — and the check passes. The stored header then carries shard X's `prev_state_root`, causing `set_state_part` and `apply_state_part` to write shard X's trie nodes and flat-state key/value pairs into shard Y's `ShardUId`-keyed storage.

---

### Finding Description

**Step 1 — Missing shard identity check in `set_state_header`.**

`validate_chunk_proofs` validates only the chunk's internal consistency (hash, tx root, receipts root). It does not compare `chunk.shard_id()` against the caller-supplied `shard_id`. [1](#0-0) 

The subsequent `verify_path` call proves that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is *somewhere* in the block's `chunk_headers_root` Merkle tree: [2](#0-1) 

`verify_path` is purely a root-recomputation check — it does not verify the leaf's index: [3](#0-2) 

Because `chunk_headers_root` is a Merkle root over **all** shards' chunks, shard X's chunk is a valid leaf at position X. Supplying shard X's chunk together with shard X's Merkle proof makes `verify_path` return `true` even when the caller requested shard Y. No code in `set_state_header` ever asserts `chunk.shard_id() == shard_id`. The grep confirms zero references to `chunk.shard_id()` in `adapter.rs`.

The header is then stored under `StateHeaderKey(Y, sync_hash)` with shard X's `prev_state_root` embedded: [4](#0-3) 

**Step 2 — `set_state_part` validates against the wrong (shard X) root.**

`set_state_part` reads the stored header for `(Y, sync_hash)` and extracts `state_root` from the chunk's `prev_state_root()` — which is now shard X's root. Parts for shard X validate correctly against that root: [5](#0-4) 

**Step 3 — `apply_state_part` writes shard X's trie keys into shard Y's flat storage.**

`Trie::apply_state_part` is called with shard X's `state_root` and shard X's parts, producing `trie_changes` and `flat_state_delta` whose keys belong to shard X's key space. Then `shard_uid` is derived from `(shard_id=Y, epoch_id)`, and the delta is applied to shard Y's flat storage: [6](#0-5) 

The `flat_state_delta` produced by `apply_state_part_impl` contains shard X's trie keys verbatim: [7](#0-6) 

**Step 4 — Incoming receipts check does not block the attack.**

The receipts check at step 4e hashes receipts with `shard_id` (Y), so the attacker must supply valid incoming receipts for shard Y. These are publicly available on-chain. In the common case where both shards include a chunk at every block, `height_included` is identical for both shards, so the height-range check at step 4g also passes: [8](#0-7) 

**Contrast with chunk distribution path.** `ShardsManager` does call `verify_chunk_shard_id` during normal chunk validation, but that guard is absent from the state-sync path: [9](#0-8) 

---

### Impact Explanation

After `apply_state_part` completes, shard Y's `DBCol::FlatState` and trie node storage contain shard X's key/value pairs keyed under shard Y's `ShardUId`. `set_state_finalize` then calls `apply_chunk` using shard X's `prev_state_root` on shard Y's `shard_uid`, which succeeds because the trie nodes are now present. The resulting `ChunkExtra` for shard Y records shard X's post-state root. Any subsequent query to shard Y's flat storage returns values from shard X's accounts, and shard Y's normal block processing diverges from the canonical chain.

---

### Likelihood Explanation

The syncing node downloads the state header from a single source and accepts the first header that passes validation — there is no multi-peer comparison: [10](#0-9) 

Any peer reachable by the syncing node can serve a malicious header. The attacker needs only knowledge of the canonical chain (to construct valid Merkle proofs and incoming receipts for shard Y), which is public. No validator or operator privilege is required.

---

### Recommendation

In `set_state_header`, after extracting the chunk, assert that the chunk's shard identity matches the requested shard:

```rust
// After: let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [11](#0-10) 

---

### Proof of Concept

1. Syncing node begins state sync for `shard_id=Y`, `sync_hash=H`.
2. Attacker peer intercepts the header request and responds with a `ShardStateSyncResponseHeader` containing:
   - Shard X's `ShardChunk` (with shard X's `prev_state_root = root_X`)
   - Shard X's Merkle proof from the block's `chunk_headers_root` (valid at position X)
   - Shard X's prev chunk and its Merkle proof
   - Valid incoming receipts for shard Y (same height range, publicly available)
   - A valid `StateRootNode` for `root_X`
3. `set_state_header(Y, H, malicious_header)` passes all checks; header stored under `StateHeaderKey(Y, H)`.
4. Attacker serves genuine shard X state parts; `set_state_part(Y, H, part_id, shard_X_part)` validates against `root_X` and stores under `StatePartKey(H, Y, part_id)`.
5. `apply_state_part(Y, root_X, part_id, shard_X_part, epoch_id)` writes shard X's trie nodes and flat-state entries into shard Y's `ShardUId`-keyed storage.
6. After `set_state_finalize`, querying shard Y's flat storage returns values from shard X's accounts.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
