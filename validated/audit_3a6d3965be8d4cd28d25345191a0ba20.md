### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Conditions — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` uses `wrapping_add` (when `nowrap=false`) for relative timelock arithmetic. An attacker who crafts a CLVM condition with a near-maximal `height_relative` or `seconds_relative` value causes the addition to overflow and wrap to a small integer, making the timelock check trivially pass. This bypasses `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` guards on any coin spend evaluated under the legacy (`nowrap=false`) path.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic modes for relative timelock evaluation:

```
// nowrap=true  (safe path)
confirmed_block_index.saturating_add(height_relative)

// nowrap=false (legacy path)
confirmed_block_index.wrapping_add(height_relative)
``` [1](#0-0) 

The guard logic is:

> **Reject** the spend if `prev_height < confirmed + height_relative`.

When `nowrap=false` and an attacker supplies `height_relative = 0xFFFF_FFFF`:

```
confirmed_block_index = 100
100u32.wrapping_add(0xFFFF_FFFF) = 99   // overflows, wraps to 99
```

The check becomes `prev_height < 99`. Any node at height ≥ 99 passes the check — the timelock that was supposed to require ~4 billion additional blocks is silently bypassed.

The same overflow applies to `seconds_relative` (u64):

```
timestamp = 1000
1000u64.wrapping_add(0xFFFF_FFFF_FFFF_FFFF) = 999   // wraps to 999
``` [2](#0-1) 

The tests explicitly confirm this behavior:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
height_relative=0xffff_ffff, nowrap=false → Ok(())   // timelock bypassed
seconds_relative=0xffff_ffff_ffff_ffff, nowrap=false → Ok(())  // timelock bypassed
``` [3](#0-2) 

The Python binding exposes this function directly to the consensus layer: [4](#0-3) 

The Python test suite also confirms the divergence between the two paths: [5](#0-4) 

---

### Impact Explanation

**High — Timelock validation bypass enables unauthorized spend acceptance.**

A coin whose puzzle enforces `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` (or any value that causes `confirmed + height_relative` to overflow u32) is intended to be unspendable for ~4 billion blocks. Under `nowrap=false`, the overflow makes the computed deadline wrap to a value already in the past, so the spend is accepted immediately. This matches the allowed impact:

> *High. Signature, aggregate signature, synthetic key, puzzle type, condition, timelock, or coin-id validation bypass enables unauthorized spend acceptance or replay.*

The symmetric effect on `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` is the inverse: overflow wraps the deadline to a small value, causing `prev_height >= small_value` to be true and **incorrectly rejecting** valid spends. This creates consensus divergence between nodes using different `nowrap` values for the same block. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The `nowrap=false` path is the legacy behavior preserved for backward-compatible validation of pre-fork blocks. Any spend bundle submitted to a node that evaluates it under `nowrap=false` (e.g., during re-validation of historical blocks, or if the Python consensus layer passes `nowrap=False` for any live block) is vulnerable. The attacker-controlled input (`height_relative` / `seconds_relative`) is a standard CLVM condition value embedded in a spend bundle — no privileged access is required to craft it.

---

### Recommendation

1. **Remove the `nowrap=false` branch entirely** once the hard-fork activating `saturating_add` semantics is finalized and all historical blocks have been re-validated under the new rules.
2. Until then, **add a range check** before the arithmetic: reject any `height_relative` value that would cause `confirmed_block_index + height_relative` to exceed `u32::MAX`, rather than silently wrapping.
3. Audit all call sites in the Python consensus layer to confirm `nowrap=True` is passed for every block at or above the fork height.

---

### Proof of Concept

Craft a spend bundle with a coin spend whose CLVM output includes:

```
(ASSERT_HEIGHT_RELATIVE . 0xFFFFFFFF)
```

Submit this spend bundle to a node that calls `check_time_locks(..., nowrap=false)`. The evaluation path is:

```
confirmed_block_index = C   (any value, e.g. 100)
height_relative       = 0xFFFF_FFFF

// wrapping_add:
C.wrapping_add(0xFFFF_FFFF) = C - 1   (wraps to C-1)

// check:
prev_height < (C - 1)   →  false for any prev_height ≥ C-1
// → no error returned → spend accepted despite timelock
```

The coin is spendable at any block height ≥ `confirmed_block_index - 1`, bypassing the intended ~4 billion block delay. [1](#0-0) [7](#0-6)
