## Vulnerability Analysis

Let me trace the exact code path to verify the claim.

### `insert()` — has the guard [1](#0-0) 

```rust
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

### `batch_insert()` — bulk path has NO guard [2](#0-1) 

The first `if` block (lines 578–585) only runs when `leaf_count <= 1`, and it calls `self.insert()` (with the hash guard) for at most 2 items via `pop()`. **All remaining items** — and all items when the tree already has ≥2 leaves — go through the bulk `for` loop (lines 587–603) which calls `insert_entry_to_blob` directly with zero duplicate-hash checking.

### `insert_entry_to_blob()` → `add_leaf()` — silently overwrites [3](#0-2) [4](#0-3) 

`add_leaf` calls `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`. `HashMap::insert` silently overwrites the previous value. So inserting a second leaf with hash `H` replaces `leaf_hash_to_index[H]` with the new index, making the first leaf unreachable by hash.

### `get_node_by_hash()` — reads the corrupted cache [5](#0-4) 

It reads directly from `block_status_cache.get_index_by_leaf_hash()`, so it will return the wrong `(KeyId, ValueId)` for `H`.

### `check_integrity()` would detect it — but only in tests [6](#0-5) 

The integrity check compares `leaf_count` (from tree traversal) against `leaf_hash_to_index_cache_length`. With 2 leaves sharing hash `H`, `leaf_count=2` but `leaf_hash_to_index_cache_length=1`, so it would error. However: [7](#0-6) 

`check_integrity_on_drop` is only `true` in `cfg!(test)`. In production it is `false`, so the corruption persists silently.

### Title
`batch_insert` bulk path omits duplicate-hash guard, corrupting `leaf_hash_to_index` cache and causing `get_node_by_hash` to return wrong leaf data — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`batch_insert` bypasses the `HashAlreadyPresent` guard for all items processed through its bulk path. Two leaves with identical hash `H` but different `(key, value)` pairs can be inserted. `add_leaf` silently overwrites `leaf_hash_to_index[H]`, making the first leaf permanently unreachable by hash and causing `get_node_by_hash(H)` to return the wrong `(KeyId, ValueId)`.

### Finding Description
`insert()` guards against duplicate hashes at lines 372–374. `batch_insert()` calls `insert()` for at most 2 items (via `pop()` from the end of the vector) only when `leaf_count <= 1`. All other items — and all items when the tree already has ≥2 leaves — go through the bulk `for` loop (lines 587–603), which calls `insert_entry_to_blob` directly. `insert_entry_to_blob` calls `add_leaf`, which calls `HashMap::insert` on `leaf_hash_to_index` without checking for an existing entry. The old mapping is silently overwritten.

### Impact Explanation
`get_node_by_hash` is the primary hash-keyed lookup for DataLayer state. After the corruption:
- `get_node_by_hash(H)` returns `(k1, v1)` instead of `(k2, v2)` (or vice versa)
- The displaced leaf is present in the blob and reachable by key, but permanently unreachable by hash
- The Merkle root itself is not corrupted (it is computed from the blob structure), but hash-based state lookups are wrong
- `check_integrity` would detect the `leaf_count` vs `leaf_hash_to_index_cache_length` mismatch, but it is disabled in production (`check_integrity_on_drop: cfg!(test)`)

This fits the allowed High impact: "DataLayer Merkle proof/blob/delta logic... lets untrusted input prove invalid state."

### Likelihood Explanation
Any caller with write access to a DataLayer store can trigger this by submitting a `batch_insert` with two entries sharing the same hash value (the hash is caller-supplied, not derived from key/value). The tree only needs ≥2 existing leaves for the entire batch to bypass the guard. No cryptographic capability is required.

### Recommendation
Add a duplicate-hash check in the bulk path of `batch_insert`, mirroring the guard in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing block construction
}
```

Alternatively, `add_leaf` could return an error if either `key_to_index` or `leaf_hash_to_index` already contains the key/hash, propagating the error up through `insert_entry_to_blob`.

### Proof of Concept
```rust
// Start with a tree that has ≥2 leaves so batch_insert skips the insert() path
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.insert(KeyId(0), ValueId(0), &hash_a, InsertLocation::AsRoot {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &hash_b, InsertLocation::Auto {}).unwrap();

// Now batch_insert with two entries sharing hash H
let shared_hash = Hash(...);
blob.batch_insert(vec![
    ((KeyId(2), ValueId(2)), shared_hash),
    ((KeyId(3), ValueId(3)), shared_hash),
]).unwrap(); // succeeds — no error

// leaf_hash_to_index[shared_hash] now points to KeyId(3)'s index
let (key, _value) = blob.get_node_by_hash(shared_hash).unwrap();
assert_eq!(key, KeyId(2)); // FAILS — returns KeyId(3)
// KeyId(2)'s leaf is in the blob but unreachable by hash
```
