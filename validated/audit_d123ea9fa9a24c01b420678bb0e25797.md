### Title
Unchecked Multiplication Overflow in `denormalize_amount` Permanently Locks User Funds on Source Chain — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`denormalize_amount` performs an unchecked `u128` multiplication. When a user calls `initTransfer` on EVM (or Starknet/Solana) with an amount large enough to overflow after denormalization, the NEAR `fin_transfer_callback` panics on every replay of that proof. Because the source-chain tokens are already burned/locked before the event is emitted, the funds become permanently irrecoverable.

---

### Finding Description

`denormalize_amount` in `near/omni-bridge/src/lib.rs` is:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

The workspace `Cargo.toml` enables overflow checks in the release profile:

```toml
[profile.release]
overflow-checks = true
``` [2](#0-1) 

So any multiplication that exceeds `u128::MAX` causes a **panic** (not silent wrap-around), which aborts the NEAR transaction.

`denormalize_amount` is called unconditionally in `fin_transfer_callback` on the amount extracted directly from the prover result:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

`init_transfer.amount` is populated verbatim from the EVM `InitTransfer` event log:

```rust
amount: near_sdk::json_types::U128(event.data.amount),
``` [4](#0-3) 

The EVM `initTransfer` function accepts any `uint128 amount` from the caller and emits it without normalization:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [5](#0-4) 

The `Decimals` struct stored per token has `origin_decimals` (NEAR-side decimals) and `decimals` (EVM-side decimals). For a NEAR-native token with 24 decimals bridged to EVM with 18 decimals, `diff_decimals = 6`. The overflow threshold is:

```
amount_overflow_threshold = u128::MAX / 10^6 ≈ 3.4 × 10^32
```

Any `uint128` value above this threshold, when passed to `initTransfer`, causes `denormalize_amount` to panic. For tokens with larger decimal differences (e.g., `diff_decimals = 24`), the threshold drops to `≈ 3.4 × 10^14` EVM-side units — a more reachable value.

`denormalize_fee` (called on the same code path at line 727) has the identical flaw for the fee field:

```rust
fee: Self::denormalize_fee(&init_transfer.fee, decimals),
``` [6](#0-5) 

---

### Impact Explanation

1. User calls `initTransfer(token, amount, fee, ...)` on EVM. Tokens are burned or locked in the bridge vault **before** the event is emitted — this is by design and enforced by the EVM invariant.
2. The `InitTransfer` event is permanently recorded on-chain.
3. A relayer submits the proof to NEAR via `fin_transfer`.
4. `fin_transfer_callback` calls `denormalize_amount(amount, decimals)`, which overflows and **panics**.
5. The NEAR transaction reverts. The proof is not consumed (no nonce is marked used), so the same proof can be replayed — but every replay will panic identically.
6. There is no EVM-side refund or cancellation path for a failed NEAR finalization.
7. The user's tokens are **permanently locked/burned** on EVM with no recovery.

This matches: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

- The condition requires `origin_decimals > decimals`, which is the standard configuration for NEAR-native tokens bridged to EVM (NEAR uses 24 decimals; EVM tokens typically use 18).
- For `diff_decimals = 6`, the overflow threshold is `≈ 3.4 × 10^32` in EVM token units — far beyond any realistic circulating supply, making accidental triggering practically impossible.
- For tokens with larger decimal differences (e.g., `diff_decimals = 24`), the threshold falls to `≈ 3.4 × 10^14` EVM units, which is reachable for high-supply tokens.
- A malicious user must actually hold and burn/lock the tokens to trigger this, so self-harm is required. However, a user who mistakenly submits a near-max `uint128` amount (e.g., due to a UI bug or deliberate griefing of a shared pool) would suffer permanent loss.
- Likelihood is **Low to Medium** depending on the token's decimal configuration, but the impact is irreversible.

---

### Recommendation

Replace the bare multiplication in `denormalize_amount` with a checked or saturating operation that returns an error instead of panicking:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Result<u128, BridgeError> {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.checked_pow(diff_decimals)
        .ok_or(BridgeError::AmountOverflow)?;
    amount.checked_mul(multiplier)
        .ok_or(BridgeError::AmountOverflow)
}
```

In `fin_transfer_callback`, propagate the error gracefully (e.g., refund the relayer and abort without consuming the proof, or reject the transfer with a logged error) rather than panicking. This prevents the proof from becoming permanently unprocessable.

---

### Proof of Concept

**Setup:** A NEAR-native token with `origin_decimals = 24` is bridged to EVM with `decimals = 18` (`diff_decimals = 6`).

**Attack steps:**

1. Attacker holds `amount = (u128::MAX / 10^6) + 1` units of the EVM bridged token (≈ `3.4 × 10^32` wei-equivalent).
2. Attacker calls:
   ```solidity
   OmniBridge.initTransfer(tokenAddress, amount, 0, 0, "attacker.near", "");
   ```
   EVM burns `amount` tokens and emits `InitTransfer(attacker, token, nonce, amount, 0, 0, ...)`.
3. Relayer submits the proof to NEAR via `fin_transfer(...)`.
4. `fin_transfer_callback` executes:
   ```rust
   // decimals = Decimals { decimals: 18, origin_decimals: 24 }
   // diff_decimals = 6
   // amount = (u128::MAX / 10^6) + 1
   // amount * 10^6 > u128::MAX → PANIC (overflow-checks = true)
   Self::denormalize_amount(init_transfer.amount.0, decimals)
   ```
5. NEAR transaction reverts. Tokens are permanently burned on EVM. Every subsequent relay attempt panics identically. [1](#0-0) [3](#0-2) [7](#0-6) [8](#0-7)
