### Title
Unprotected Bootstrap Declare Path Bypasses All Signature and Authorization Checks, Allowing Unauthorized Class Registration - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` in `transaction_impls.cairo` contains a special "bootstrap" code path (lines 761–776) that completely skips signature validation, nonce verification, and fee charging. The sole guard is a felt-literal comparison `sender_address == 'BOOTSTRAP'`. Because `sender_address` is a plain transaction field with no cryptographic binding, a malicious sequencer can craft a declare transaction that satisfies all four conditions and register any class hash → compiled class hash mapping into the global state without any account ownership or signature. This is the direct analog of the unprotected `initialize()` vulnerability: a privileged initialization path with no access control.

---

### Finding Description

Inside `execute_declare_transaction`, after the transaction hash is computed and `tx_info` is filled, the OS evaluates:

```cairo
// Do not run validate or perform any account-related actions for declare transactions that
// meet the following conditions.
// This flow is used for the sequencer to bootstrap a new system.
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately, having skipped:

1. **`run_validate`** — the account contract's `__validate_declare__` entry point, which performs ECDSA signature verification. [2](#0-1) 

2. **`check_and_increment_nonce`** — which verifies the transaction nonce matches the on-chain account nonce. [3](#0-2) 

3. **`charge_fee`** — which deducts fees from the sender's balance. [4](#0-3) 

The four conditions that gate this path are:

| Condition | Source |
|---|---|
| `sender_address == 'BOOTSTRAP'` | Plain felt field from `%{ DeclareTxFields %}` hint |
| `tx_info.nonce == 0` | Plain felt field from `%{ LoadCommonTxFields %}` hint |
| `tx_info.version == 3` | Plain felt field from `%{ TxVersion %}` hint |
| `max_possible_fee == 0` | All three resource bounds set to zero | [5](#0-4) 

None of these conditions require a cryptographic proof of ownership. `'BOOTSTRAP'` is the felt encoding of the ASCII string `BOOTSTRAP` (= `0x424f4f545354524150`). There is no deployed account at this address, no private key, and no signature required. The OS enforces no ownership over who may set `sender_address` to this value in a transaction.

Additionally, the branch ends with `%{ SkipTx %}` before returning, which instructs the execution helper to omit this transaction from the OS output entirely. The class change, however, **is** committed to `contract_class_changes` and therefore flows into the state diff and the final state root. [6](#0-5) 

The normal declare path (lines 778–827) enforces `check_and_increment_nonce`, runs `__validate_declare__`, and calls `charge_fee`. The bootstrap path bypasses all three. [7](#0-6) 

---

### Impact Explanation

**Primary impact: Critical — Direct loss of funds / High — Network shutdown**

The `dict_update` call uses `prev_value=0`, which enforces that only previously-undeclared class hashes can be registered. This means the attacker cannot overwrite an already-declared class. However, the attack surface is the **bootstrap window**: when a new StarkNet system is being initialized, critical system classes (fee token, account contracts, proxy contracts) have not yet been declared. A malicious sequencer can front-run the legitimate bootstrap declare transactions and register those class hashes with attacker-controlled `compiled_class_hash` values.

Consequences:
- **Fee token poisoned**: If the fee token class hash is declared with a malicious compiled class, all subsequent fee transfers execute attacker code. Funds paid as fees are stolen or frozen permanently.
- **Account class poisoned**: If a widely-used account class hash is declared with malicious code, every account deployed using that class executes attacker-controlled logic, enabling theft of all funds held by those accounts.
- **Network halt**: If a class required by the OS itself or by the sequencer's infrastructure is declared with a broken compiled class, the network cannot process further transactions.

Because the malicious class registration is hidden from the OS output (`%{ SkipTx %}`), the L1 verifier sees only the resulting state root change, with no on-chain transaction record to audit or dispute.

---

### Likelihood Explanation

**Medium-High**

- The attack requires a malicious sequencer (or a sequencer that has been compromised), since the sequencer controls which transactions are included in a block and provides the hints that populate `sender_address`, `nonce`, `version`, and `resource_bounds`.
- The bootstrap window is a one-time event per network deployment, but it is the highest-value target: all critical system classes are undeclared at that moment.
- No private key, no account, and no fee is required. The attacker only needs to craft a well-formed declare transaction with four specific field values.
- The OS itself provides no defense; the only protection would have to come from off-chain sequencer policy, which is outside the OS's enforcement boundary.

---

### Recommendation

Add a cryptographic access control check to the bootstrap path. Options include:

1. **Remove the bootstrap path entirely** and require all class declarations to go through the normal signed declare flow, even during bootstrapping.
2. **Bind the bootstrap path to a specific deployer key**: store the deployer's public key (or its hash) in the OS config (`StarknetOsConfig`) and require the bootstrap declare transaction to carry a valid signature from that key, verified before the `dict_update` call.
3. **Require the bootstrap sender address to be a real deployed account**: enforce `state_entry.class_hash != UNINITIALIZED_CLASS_HASH` for the `'BOOTSTRAP'` address before allowing the shortcut path, so that at minimum a deployed contract must authorize the action.

The core principle (mirroring the original report's recommendation) is: save the authorized deployer identity at OS initialization time and verify `sender == authorized_deployer` before executing any privileged initialization path.

---

### Proof of Concept

A malicious sequencer constructs the following declare transaction and includes it in a block during the bootstrap window:

```
sender_address  = 0x424f4f545354524150  // felt('BOOTSTRAP')
version         = 3
nonce           = 0
resource_bounds = [(L1_GAS, 0, 0), (L2_GAS, 0, 0), (L1_DATA_GAS, 0, 0)]  // max_fee = 0
class_hash      = <target system class hash, e.g. fee token class>
compiled_class_hash = <attacker's malicious compiled class hash>
signature       = []  // empty — never checked
```

The OS processes this transaction via `execute_declare_transaction`:

1. `sender_address == 'BOOTSTRAP'` → **true**
2. `tx_info.nonce == 0` → **true**
3. `tx_info.version == 3` → **true**
4. `compute_max_possible_fee(tx_info) == 0` → **true** (all bounds zero)
5. `assert_not_zero(compiled_class_hash)` → passes (attacker provides non-zero value)
6. `dict_update(key=fee_token_class_hash, prev_value=0, new_value=malicious_compiled_class_hash)` → **class registered**
7. `%{ SkipTx %}` → transaction omitted from OS output
8. `return ()` → no signature check, no nonce check, no fee

The legitimate bootstrap declare for the same `class_hash` subsequently fails because `prev_value=0` is no longer satisfied (the class is already declared). The network is initialized with the attacker's malicious fee token class. All subsequent fee payments execute attacker-controlled code, resulting in direct loss of funds for every transaction sender on the network. [1](#0-0) [8](#0-7)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
```
