### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer()` Inflates Cross-Chain Message Amount â€” (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer()` uses the caller-supplied `amount` parameter â€” not the actual tokens received â€” when constructing the cross-chain message. For fee-on-transfer tokens (e.g., stETH, USDT with fees enabled), the bridge locks fewer tokens than the amount encoded in the outbound message, causing the destination chain to release more tokens than the EVM bridge holds, breaking collateralization.

### Finding Description

In `OmniBridge.initTransfer()`, when the token is a plain ERC20 (not a bridge token or custom minter), the contract pulls tokens from the user: [1](#0-0) 

The original caller-supplied `amount` is then forwarded unchanged to `initTransferExtension()`: [2](#0-1) 

In `OmniBridgeWormhole.initTransferExtension()`, this `amount` is Borsh-encoded directly into the Wormhole message payload that instructs the destination chain how many tokens to release: [3](#0-2) 

For fee-on-transfer tokens, `safeTransferFrom` delivers `amount - fee` to the bridge, but the Wormhole message encodes `amount`. The destination chain (NEAR or another EVM) will finalize a transfer of the full `amount`, draining more from the bridge's reserves than were deposited.

The same inflated `amount` is also emitted in the `InitTransfer` event: [4](#0-3) 

### Impact Explanation

Each bridging operation with a fee-on-transfer token causes the bridge to under-collateralize by the transfer fee. Repeated use drains the bridge's EVM-side reserves, eventually making it unable to honor `finTransfer` redemptions for legitimate users â€” permanent freezing of funds or direct theft of bridge reserves. This matches: **High â€” Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value.**

### Likelihood Explanation

Any unprivileged user can call `initTransfer()` with a fee-on-transfer token address. The bridge does not whitelist or validate token behavior. stETH and similar rebasing/fee tokens are widely used and are natural candidates for bridging. Likelihood is **Medium** â€” requires a fee-on-transfer token to be used, but no special privilege is needed.

### Recommendation

Measure the actual received balance before and after the `safeTransferFrom` and use the delta as the bridged amount:

```diff
} else {
+   uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
+   amount = uint128(IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore);
}
```

Then `amount` (now the actual received amount) flows correctly into `initTransferExtension()` and the emitted event.

### Proof of Concept

1. A fee-on-transfer token `FOT` charges a 1% fee on every transfer.
2. Attacker (or any user) calls `OmniBridge.initTransfer(FOT, 1000e18, 0, 0, "near-recipient", "")`.
3. `safeTransferFrom` delivers `990e18` to the bridge; the bridge holds `990e18`.
4. `initTransferExtension` encodes `amount = 1000e18` into the Wormhole message.
5. NEAR's bridge finalizes the transfer and releases `1000e18` wrapped tokens to `near-recipient`.
6. The EVM bridge is now short `10e18` tokens per operation.
7. After enough operations, the bridge cannot honor `finTransfer` redemptions â€” funds are permanently frozen or stolen from other users' deposits.
