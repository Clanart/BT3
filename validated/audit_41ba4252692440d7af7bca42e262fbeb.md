### Title
Gateway `validate_proof_block_hash` Reads Stale State, Rejecting Valid Proof-Facts Transactions at the Boundary Block — (`crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

`validate_proof_block_hash` reads the block-hash contract storage from the **committed** (latest) state, but `validate_proof_block_number` permits proof block numbers up to `current_block_number − STORED_BLOCK_HASH_BUFFER`, where `current_block_number` is set to `latest_committed_block + 1` by the gateway. The hash for the boundary block (`latest_committed_block − STORED_BLOCK_HASH_BUFFER + 1`) is only written to storage by `pre_process_block` at the **start** of the next block's execution — after the gateway's state snapshot was taken. The gateway therefore reads `Felt::ZERO` for that slot, fails the `stored_block_hash != proof_block_hash` check, and rejects a transaction that the OS would accept.

---

### Finding Description

**Root cause — wrong state source for the block-hash lookup**

The gateway instantiates its block context with:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // N+1
``` [1](#0-0) 

So `current_block_number = N + 1` (where N is the latest committed block). `validate_proof_block_number` then computes:

```
max_allowed = (N + 1) − STORED_BLOCK_HASH_BUFFER   =   N − STORED_BLOCK_HASH_BUFFER + 1
``` [2](#0-1) 

A proof with `proof_block_number = N − STORED_BLOCK_HASH_BUFFER + 1` passes this range check.

`validate_proof_block_hash` then reads the stored hash from the **committed** state (block N):

```rust
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

if stored_block_hash != proof_block_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [3](#0-2) 

`pre_process_block` writes the hash of block `next_block_number − STORED_BLOCK_HASH_BUFFER` to storage **at the start of block N+1's execution**:

```rust
// crates/blockifier/src/blockifier/block.rs
state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
``` [4](#0-3) 

After block N is committed, storage contains hashes only up to block `N − STORED_BLOCK_HASH_BUFFER`. The hash for block `N − STORED_BLOCK_HASH_BUFFER + 1` has **not yet been written**. The gateway reads `Felt::ZERO` for that slot, and since `proof_block_hash ≠ Felt::ZERO` (the zero-hash guard already passed), the comparison fails and the transaction is rejected.

**The OS would accept the same transaction.** During block N+1 execution, `pre_process_block` runs first and writes the hash of block `N − STORED_BLOCK_HASH_BUFFER + 1` to storage. The OS `check_proof_facts` then reads the now-populated slot and the proof passes:

```cairo
// execution_constraints.cairo
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
read_block_hash_from_storage(
    block_number=os_output_header.base_block_number,
    expected_block_hash=os_output_header.base_block_hash,
);
``` [5](#0-4) 

**Analogy to M-27:** In M-27, `_minted = IERC20(_bathToken).balanceOf(address(this))` (a local token balance) was used as the guard for calling `_maxBorrow`, which internally required actual comptroller liquidity. The wrong source caused the guard to pass when it should not (or fail when it should not). Here, `validate_proof_block_number` uses `current_block_number = N+1` (the "next" block) to compute the allowed range, while `validate_proof_block_hash` reads from the committed state at block N. The two checks use **different effective block heights**, creating an off-by-one where the boundary block number passes the range check but its hash is absent from the state the hash-lookup uses.

---

### Impact Explanation

Any user who constructs a valid SNOS proof anchored to block `N − STORED_BLOCK_HASH_BUFFER + 1` (the maximum block number the range check permits) and submits an Invoke V3 transaction will have it **rejected at the gateway** with `"Block hash mismatch"`. The transaction is valid and would be accepted by the OS. This is a permanent denial-of-service for the boundary block: the condition persists until block N+2 is committed (at which point N+1 becomes the new boundary and the same problem shifts forward by one).

Impact: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

Any user of the client-side proving feature who anchors their proof to the most recent allowed block number triggers this. The boundary block advances with every new committed block, so the window is always present. No special privileges or network access are required; a normal Invoke V3 transaction with a valid proof suffices.

---

### Recommendation

Align the block-number upper bound in `validate_proof_block_number` with the block hashes that are actually present in the committed state. Either:

1. Use `current_block_number − STORED_BLOCK_HASH_BUFFER − 1` as `max_allowed` in the gateway path (i.e., do not add the `unchecked_next()` offset before passing to proof validation), or
2. Pre-populate the gateway's state with the block hash that `pre_process_block` would write for block N+1 before running `validate_proof_block_hash`.

The OS Cairo code is correct because `pre_process_block` runs before transaction execution within the same block. The gateway must mirror that invariant.

---

### Proof of Concept

1. Let `N = latest_committed_block`, `B = STORED_BLOCK_HASH_BUFFER = 10`.
2. The committed state (block N) contains the hash of block `N − 10` in the block-hash contract storage.
3. Construct a valid SNOS proof for block `N − 9` (= `N − B + 1`).
4. Submit an Invoke V3 transaction with `proof_block_number = N − 9` and `proof_block_hash = hash(block N−9)`.
5. Gateway `validate_proof_block_number`: `max_allowed = (N+1) − 10 = N − 9`. Check passes (`N−9 ≤ N−9`). [2](#0-1) 
6. Gateway `validate_proof_block_hash`: reads `state.get_storage_at(block_hash_contract_address, StorageKey::from(N−9))` → returns `Felt::ZERO` (not yet written). [3](#0-2) 
7. `Felt::ZERO ≠ hash(block N−9)` → gateway returns `InvalidProofFacts("Block hash mismatch")`. Transaction rejected.
8. If the same transaction were included in block N+1, `pre_process_block` would write `hash(block N−9)` to storage first, the OS check would pass, and the transaction would execute successfully. [6](#0-5)

pub fn pre_process_block(
    state: &mut dyn State,
    old_block_number_and_hash: Option<BlockHashAndNumber>,
    next_block_number: BlockNumber,
    os_constants: &OsConstants,
) -> StateResult<()> {
    let should_block_hash_be_provided =
        next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
    if let Some(BlockHashAndNumber { number, hash }) = old_block_number_and_hash {
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();
        let block_number_as_storage_key = StorageKey::from(number.0);
        state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
    } else if should_block_hash_be_provided {
        return Err(StateError::OldBlockHashNotProvided);
    }
