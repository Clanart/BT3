Audit Report

## Title
`_afterTimelock` uint32 truncation lets pool admin permanently bypass all timelocks — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`_afterTimelock` casts `block.timestamp + oracleStopLossConfig[pool_].timelock` to `uint32` without any cap on the stored timelock value. A pool admin who installs a crafted timelock causes every subsequent proposal's `pendingXxxExecuteAfter` to truncate to a small non-zero value, which `_requireElapsed` passes immediately, permanently eliminating the LP-protection window for all parameter changes.

## Finding Description
`_afterTimelock` performs an unchecked downcast:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

The addition executes in `uint256` (no revert), then the explicit `uint32(...)` cast silently truncates the lower 32 bits. There is no `_validateTimelock` anywhere in the codebase (confirmed by grep). `initialize` calls `_validateDrawdown` and `_validateDecay` but skips any timelock cap: [2](#0-1) 

`proposeOracleStopLossTimelock` likewise stores `newTimelock` without validation: [3](#0-2) 

**Exploit arithmetic (July 2026, `block.timestamp ≈ 1_753_000_000`):**

```
maliciousTimelock = type(uint32).max - block.timestamp + 2
                  = 4_294_967_295 - 1_753_000_000 + 2
                  = 2_541_967_297   // fits in uint32 ✓

block.timestamp + maliciousTimelock = 4_294_967_297
uint32(4_294_967_297) = 1           // lower 32 bits
```

`pendingXxxExecuteAfter = 1` passes the `!= 0` guard in `executeOracleStopLossTimelock`: [4](#0-3) 

And `_requireElapsed(1)` trivially passes since `block.timestamp ≈ 1.75 × 10⁹ > 1`: [5](#0-4) 

The same `_afterTimelock` is used for drawdown, decay, and watermark proposals — all are affected: [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation
Once the malicious timelock is installed (after one legitimate wait), every future parameter change — drawdown reduction, decay increase, watermark reset — executes in the same block as the proposal. LPs lose the reaction window the timelock is designed to provide. The pool admin can immediately set `drawdownE6 = 0` (disabling stop-loss) or zero watermarks, removing all value-loss guards while LPs have no time to exit. This is a direct admin-boundary break: pool admin bypasses timelocks, matching the allowed impact gate.

## Likelihood Explanation
Requires a malicious pool admin who waits through one legitimate timelock cycle to install the large timelock value. After that, the bypass is permanent and affects all future proposals with no further preconditions. The pool admin role is the sole actor; no unprivileged caller is needed.

## Recommendation
Add a maximum cap on the timelock value in both `initialize` and `proposeOracleStopLossTimelock`, analogous to `_validateDrawdown` and `_validateDecay`:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days; // ~31.5M seconds, well below uint32 overflow

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock` in `initialize` (alongside the existing drawdown/decay checks) and at the top of `proposeOracleStopLossTimelock` before storing the value.

## Proof of Concept
```solidity
// Foundry test sketch
uint32 ts = uint32(block.timestamp); // ~1_753_000_000
uint32 maliciousTimelock = type(uint32).max - ts + 2; // wraps sum to 1

// Verify truncation
assertEq(uint32(uint256(ts) + uint256(maliciousTimelock)), 1);

// Step 1: pool admin calls proposeOracleStopLossTimelock(pool, maliciousTimelock)
//         _afterTimelock uses current (legitimate) timelock → future executeAfter
// Step 2: wait existing timelock, call executeOracleStopLossTimelock
//         oracleStopLossConfig[pool].timelock = maliciousTimelock
// Step 3: pool admin calls proposeOracleStopLossDrawdown(pool, 0)
//         _afterTimelock returns uint32(block.timestamp + maliciousTimelock) = 1
//         pendingDrawdownExecuteAfter = 1
// Step 4: pool admin immediately calls executeOracleStopLossDrawdown
//         pendingDrawdownExecuteAfter != 0  → passes
//         block.timestamp < 1              → false → passes
//         drawdownE6 set to 0, stop-loss disabled with zero LP reaction time
```
