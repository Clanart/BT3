### Title
Corruptible Upgradability: `OmniBridge` and `BridgeToken` Lack Storage Gaps, Corrupting Child Contract Storage on Upgrade — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `evm/src/omni-bridge/contracts/BridgeToken.sol`)

---

### Summary

Two upgradeable base contracts in the EVM bridge — `OmniBridge` and `BridgeToken` — declare state variables but define no `__gap` storage reserve. Their respective child contracts (`OmniBridgeWormhole` and `HyperliquedBridgeToken`) append their own state variables immediately after the parent's variables. Any future upgrade that adds a new state variable to either parent will silently shift the child's variable slots, corrupting critical runtime state.

---

### Finding Description

**Chain 1 — `OmniBridge` → `OmniBridgeWormhole`**

`OmniBridge` is a UUPS-upgradeable contract that declares ten sequential storage variables:

```
mapping(address => string)  ethToNearToken        // slot N
mapping(string => address)  nearToEthToken         // slot N+1
mapping(address => bool)    isBridgeToken          // slot N+2
address                     tokenImplementationAddress // slot N+3
address                     nearBridgeDerivedAddress   // slot N+4
uint8                       omniBridgeChainId          // packed N+4
mapping(uint64 => bool)     completedTransfers         // slot N+5
uint64                      currentOriginNonce         // slot N+6
mapping(address => address) customMinters              // slot N+7
mapping(address => MultiTokenInfo) multiTokens         // slot N+8
```

`OmniBridge` declares **no `__gap`**.

`OmniBridgeWormhole` inherits `OmniBridge` and immediately appends:

```
IWormhole  _wormhole          // slot N+9
uint8      _consistencyLevel  // packed N+9
uint32     wormholeNonce      // slot N+10
```

If a future upgrade of `OmniBridgeWormhole` adds even one new variable to `OmniBridge`, the slots occupied by `_wormhole`, `_consistencyLevel`, and `wormholeNonce` shift by one. The proxy's storage still holds the old layout, so the new implementation reads garbage values for all three fields.

**Chain 2 — `BridgeToken` → `HyperliquedBridgeToken`**

`BridgeToken` declares `_name`, `_symbol`, `_decimals` with no `__gap`. `HyperliquedBridgeToken` inherits it and appends `_systemAddress` directly after `_decimals`. `_systemAddress` is the sole access-control guard for `coreReceiveWithData` and the accounting pivot for the 3-arg `mint`:

```solidity
// HlBridgeToken.sol line 114
if (msg.sender != _systemAddress) revert NotSystemAddress();
```

```solidity
// HlBridgeToken.sol line 82
_update(account, _systemAddress, value);
```

A corrupted `_systemAddress` either permanently locks `coreReceiveWithData` (if it becomes `address(0)`) or allows an arbitrary address to call it and trigger unauthorized token movements.

---

### Impact Explanation

**`OmniBridgeWormhole`:** A corrupted `_wormhole` address causes every call to `_wormhole.publishMessage` to revert or route to an unintended contract. This breaks `initTransfer`, `finTransfer`, and `deployToken` for all Wormhole-backed bridge operations, permanently freezing in-flight and future user funds in the bridge.

**`HyperliquedBridgeToken`:** A corrupted `_systemAddress` either (a) permanently locks `coreReceiveWithData`, freezing all HyperCore→HyperEVM redemptions, or (b) redirects the access-control check to an attacker-controlled address, enabling unauthorized `_update` calls that move tokens out of the system pool — an unauthorized release of bridged assets.

Both match the allowed impact scope:
- *Critical — Permanent freezing / irrecoverable lock of user funds in bridge flows.*
- *Critical — Unauthorized release or unauthorized mint of bridged assets.*

---

### Likelihood Explanation

Both `OmniBridge` and `BridgeToken` are actively developed UUPS-upgradeable contracts. The protocol has already demonstrated willingness to add new state variables (e.g., `multiTokens`, `customMinters` were added to `OmniBridge`; `_systemAddress` was added to the token layer). Any future feature addition — fee accounting, new token-type flags, additional role addresses — that requires a new storage variable in either parent will trigger this corruption silently, with no on-chain warning.

---

### Recommendation

Add an `__gap` array to every upgradeable base contract that has state variables and is inherited by a child that also has state variables:

```solidity
// In OmniBridge.sol, after all state variable declarations:
uint256[50] private __gap;

// In BridgeToken.sol, after _decimals:
uint256[50] private __gap;
```

Alternatively, migrate both contracts to EIP-7201 namespaced storage (as `SelectivePausableUpgradable` already does correctly), which eliminates the gap requirement entirely.

---

### Proof of Concept

**Storage layout collision for `OmniBridgeWormhole`:**

Current layout (before upgrade adding one variable to `OmniBridge`):
- Slot 9 → `_wormhole` (IWormhole address, 20 bytes) + `_consistencyLevel` (uint8, 1 byte)
- Slot 10 → `wormholeNonce` (uint32)

After upgrade where `OmniBridge` adds `address feeCollector` after `multiTokens`:
- Slot 9 → `feeCollector` (new OmniBridge variable)
- Slot 10 → `_wormhole` (now reads old `wormholeNonce` value — a `uint32` cast to `address`)
- Slot 11 → `_consistencyLevel` / `wormholeNonce` (garbage)

Every subsequent call to `_wormhole.publishMessage(...)` in `initTransferExtension`, `finTransferExtension`, and `deployTokenExtension` targets a nonsense address and reverts, permanently breaking all Wormhole bridge operations.
