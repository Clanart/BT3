### Title
Missing Nonce in `execute_meta_tx_v0` Hash Enables Unbounded Replay of Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall computes a transaction hash that excludes any nonce, and the OS explicitly skips nonce checking and incrementing for version-0 transactions. This allows any unprivileged attacker who has observed a valid meta-tx signature on-chain to replay the same meta-transaction an unlimited number of times within the same or future blocks, directly analogous to the M-07 multisig re-execution bug where the absence of a unique per-execution identifier allows the same action to fire repeatedly.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_meta_tx_v0` (lines 286–400) is a syscall callable by any contract during execution. It performs the following steps:

**Step 1 — Hash computation without a nonce:**

```cairo
let meta_tx_hash = compute_meta_tx_v0_hash(
    contract_address=contract_address,
    entry_point_selector=selector,
    calldata=calldata_start,
    calldata_size=calldata_size,
    chain_id=old_tx_info.chain_id,
);
``` [1](#0-0) 

The hash inputs are `contract_address`, `entry_point_selector`, `calldata`, and `chain_id`. **No nonce is passed.** The resulting hash is identical for every replay of the same call.

**Step 2 — `TxInfo` constructed with hardcoded `nonce=0` and `version=0`:**

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
``` [2](#0-1) 

**Step 3 — OS nonce enforcement is bypassed for version 0:**

In `execute_transaction_utils.cairo`, `check_and_increment_nonce` explicitly returns early for version-0 transactions:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

Furthermore, `check_and_increment_nonce` is never called inside `execute_meta_tx_v0` at all — it is only called at the top-level transaction dispatch (`execute_invoke_function_transaction`, `execute_deploy_account_transaction`, `execute_declare_transaction`). [4](#0-3) [5](#0-4) [6](#0-5) 

**Step 4 — Execution proceeds via `contract_call_helper`:**

The target contract's `__execute__` entry point is invoked with the nonce-free `meta_tx_hash` as the `transaction_hash` and `nonce=0`. The account contract's stored nonce in `contract_state_changes` is never incremented by the OS. [7](#0-6) 

**Consequence:** Because the `meta_tx_hash` is identical across replays (no nonce), and the OS never increments the account's nonce for version-0 meta-txs, the account contract's stored nonce remains 0 after every execution. A standard account contract that verifies the signature against `transaction_hash` and checks `tx_info.nonce` against its stored nonce will see `nonce=0` matching stored nonce=0 on every replay, and the signature will remain valid since the hash is unchanged.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any attacker who observes a valid `execute_meta_tx_v0` invocation on-chain (all data is public) can extract the `contract_address`, `calldata`, and `signature` fields. They can then deploy a contract that calls `execute_meta_tx_v0` with the same parameters in any subsequent block. Because the OS produces the same `meta_tx_hash` (no nonce), the same signature remains valid, and the target account's `__execute__` entry point fires again with identical effect. If the meta-tx encodes a token transfer, the attacker can drain the target account by replaying the transaction an arbitrary number of times.

---

### Likelihood Explanation

**High.** The attack requires no privileged access. The attacker only needs to:
1. Observe a valid meta-tx on-chain (all transaction data is public).
2. Deploy a contract that issues the `execute_meta_tx_v0` syscall with the observed parameters.

There is no off-chain coordination requirement, no key compromise, and no network-level attack needed. Any block that includes the attacker's outer transaction will re-execute the meta-tx.

---

### Recommendation

Include a nonce in the `compute_meta_tx_v0_hash` computation. The nonce should be sourced from the account contract's current on-chain nonce (read from `contract_state_changes`), verified against `tx_info.nonce`, and incremented after each successful meta-tx execution — mirroring the logic in `check_and_increment_nonce` for regular transactions. This ensures each meta-tx hash is unique and non-replayable, directly analogous to the nonce-based fix recommended in M-07.

---

### Proof of Concept

1. Alice's contract calls `execute_meta_tx_v0` with `contract_address=Alice`, `calldata=transfer(100 tokens → Bob)`, and `signature=σ`.
2. The OS computes `meta_tx_hash = H(Alice, __execute__, calldata, chain_id)` — no nonce.
3. Alice's `__execute__` runs, verifies `σ` over `meta_tx_hash`, and transfers 100 tokens to Bob. Alice's stored nonce remains 0 (OS skips increment for version 0).
4. An attacker observes `σ`, `calldata`, and `Alice` on-chain.
5. The attacker deploys `AttackerContract` which calls `execute_meta_tx_v0(contract_address=Alice, calldata=transfer(100→Bob), signature=σ)`.
6. The OS recomputes the same `meta_tx_hash` (identical inputs, no nonce). Alice's stored nonce is still 0; `tx_info.nonce=0` matches. `σ` is valid over the same hash.
7. Alice's `__execute__` fires again, transferring another 100 tokens to Bob.
8. Steps 5–7 repeat in every block until Alice's balance is zero.

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
```
