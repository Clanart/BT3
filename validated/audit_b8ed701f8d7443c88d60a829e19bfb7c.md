### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay — (File: `transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the OS-level hash for a v0 meta transaction without including a nonce. Because the hash is permanently valid for any given `(contract_address, entry_point_selector, calldata)` tuple, an attacker who observes one valid submission can replay the same signature an unlimited number of times, leading to direct loss of funds.

---

### Finding Description

In `transaction_hash.cairo`, `compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

This means the hash covers only: `INVOKE_HASH_PREFIX`, `version=0`, `contract_address`, `entry_point_selector`, `calldata`, `max_fee=0`, and `chain_id` — **no nonce**.

By direct contrast, `compute_l1_handler_transaction_hash` passes `additional_data_size=1` and `additional_data=&nonce`, binding the hash to a specific nonce and preventing replay: [2](#0-1) 

The `deprecated_get_transaction_hash` function itself is correct — it hashes `additional_data` when provided: [3](#0-2) 

The root cause is that `compute_meta_tx_v0_hash` deliberately passes zero-length additional data, leaving no replay-prevention field in the signed commitment. The OS has no separate mechanism to mark a meta transaction as consumed.

---

### Impact Explanation

Any valid `(signature, contract_address, entry_point_selector, calldata)` tuple produced by a user is permanently reusable. If the meta transaction encodes a token transfer or any fund-moving operation, an attacker can drain the signer's balance by replaying the same tuple repeatedly. This is **direct loss of funds (Critical)**.

---

### Likelihood Explanation

The attacker requires only one observed valid meta transaction — available from the public mempool or on-chain history. No privileged access, leaked key, or special role is needed. The replay is deterministic: the OS recomputes the identical hash on every submission and accepts the same signature unconditionally.

---

### Recommendation

Mirror the pattern used in `compute_l1_handler_transaction_hash` — add a `nonce` parameter and pass it as `additional_data`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,                        // ADD: nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=1,         // CHANGE: was 0
        additional_data=&nonce,         // CHANGE: was cast(0, felt*)
    );
    return tx_hash;
}
```

The OS must also track consumed meta transaction nonces per account to enforce single-use semantics, analogous to how `check_and_increment_nonce` is enforced for regular account transactions: [4](#0-3) 

---

### Proof of Concept

1. User signs a meta transaction authorizing a token transfer:
   `sig = sign(compute_meta_tx_v0_hash(token_contract, transfer_selector, [recipient, amount], chain_id))`
2. Relayer submits the meta transaction; the OS verifies `sig` against the nonce-free hash and executes the transfer.
3. Attacker copies `(sig, token_contract, transfer_selector, [recipient, amount])` from the chain.
4. Attacker re-submits the identical tuple. The OS recomputes the **same** hash (no nonce field), verifies `sig` successfully, and executes the transfer again.
5. Steps 3–4 repeat until the user's balance is exhausted — constituting direct, irreversible loss of funds.
