### Title
Unauthenticated `requester` Field in `SpicePartialDataRequest` Allows Any Peer to Redirect Witness/Receipt-Proof Data to an Arbitrary Account - (File: chain/client/src/spice/data_distributor_actor.rs)

---

### Summary

`SpiceDataDistributorActor::handle_partial_data_request` accepts a network-supplied `requester: AccountId` field without verifying that the network peer who sent the request is actually that account. A malicious peer can forge the `requester` field to any `AccountId`, causing the producer to send a full state witness or receipt proof (and its associated contract-access list) to an arbitrary account chosen by the attacker. The producer performs no check that the `requester` matches the actual sending peer, and the code itself contains a `TODO` acknowledging the missing check.

---

### Finding Description

`SpicePartialDataRequest` is a network message with two fields: `data_id` (identifying the witness or receipt proof) and `requester` (the `AccountId` to which the response should be sent). [1](#0-0) 

When a producer node receives this message, it is dispatched to `handle_partial_data_request`: [2](#0-1) 

The function:
1. Verifies that **the local node** is a producer for `data_id` — correct.
2. Fetches the data from the store — correct.
3. **Does not verify** that the peer who sent the request is actually the account named in `requester`.
4. Sends the full encoded data (witness or receipt proof) and, for witness requests, the full contract-access list, directly to the `requester` account.

The code even contains an explicit `TODO` at line 1023–1025 acknowledging the missing check:

```
// TODO(spice): Check that requester is one of the recipients and implement a
// lower-priority way for other nodes that aren't validators (e.g. rpc nodes) to get
// data they require.
``` [3](#0-2) 

The message is routed over the network via `T1MessageBody::SpicePartialDataRequest` and delivered to the producer's actor without any sender-identity binding: [4](#0-3) 

The `requester` field is a plain `AccountId` with no signature or proof that the sending peer controls that account. Any connected peer can set `requester` to any `AccountId` in the message.

The existing test `test_requesting_receipts_when_not_validator` explicitly demonstrates that a `"not-validator"` requester string causes the producer to serve the data to that arbitrary account: [5](#0-4) 

---

### Impact Explanation

**Confidentiality of state witnesses and receipt proofs is broken.** An attacker who is a connected peer can:

1. Send a `SpicePartialDataRequest` with `data_id` pointing to any valid block's witness or receipt proof, and `requester` set to any `AccountId` the attacker controls or can intercept.
2. The producer will encode and transmit the full state witness (or receipt proof) to that account.
3. For witness requests, the producer also sends the full `SpiceChunkContractAccesses` list to the forged `requester`. [6](#0-5) 

The data sent includes the full `SpiceData::StateWitness` (the complete chunk state witness) or `SpiceData::ReceiptProof` (the full receipt proof for a shard pair), which are sensitive pre-finalization data. The attacker can redirect this data to any account, bypassing the intended recipient set computed by `recipients_and_producers`.

**Impact rating: High.** The invariant that witness/receipt-proof data is only sent to the authorized recipient set (`_recipients` from `recipients_and_producers`) is broken. Any peer can cause a producer to exfiltrate data to an arbitrary account.

---

### Likelihood Explanation

**Medium.** The attack requires:
- A network connection to a producer node (any peer in the NEAR P2P network qualifies).
- Knowledge of a valid `block_hash` and `shard_id`/`from_shard_id`/`to_shard_id` for a recent block (publicly observable from the chain).
- No cryptographic material, no validator key, no privileged role.

The `SpicePartialDataRequest` message is a plain borsh-serialized struct with no signature. Any peer can craft and send it. The test `test_requesting_receipts_when_not_validator` confirms the producer serves the data to `"not-validator"` without error.

---

### Recommendation

In `handle_partial_data_request`, before serving data, verify that the `requester` is one of the authorized recipients for the requested `data_id`:

```rust
let (recipients, producers) = self.recipients_and_producers(&data_id, &block)?;
if !producers.contains(signer.validator_id()) {
    return Err(Error::Other("we do not produce requested data"));
}
// ADD: verify requester is an authorized recipient
if !recipients.contains(&requester) {
    return Err(Error::Other("requester is not an authorized recipient"));
}
```

Additionally, consider binding the `requester` field to the actual sending peer's identity (e.g., by requiring the request to carry a signature from the requester's validator key, or by using the network-layer peer identity as the implicit requester rather than a caller-supplied field). [7](#0-6) 

---

### Proof of Concept

The existing test `test_requesting_receipts_when_not_validator` already demonstrates the issue end-to-end:

1. A producer actor is created for a valid chunk producer account.
2. A `SpicePartialDataRequest` is sent with `requester: AccountId::from_str("not-validator").unwrap()` — an account that is not in the authorized recipient set.
3. The producer serves the full receipt proof data to `"not-validator"` without error.
4. The non-validator actor successfully receives and decodes the receipt proof. [8](#0-7) 

The same attack applies to state witnesses: send `SpicePartialDataRequest` with `data_id = SpiceDataIdentifier::Witness { block_hash, shard_id }` and `requester` set to any account. The producer will send the full witness and contract-access list to that account.

#[test]
#[cfg_attr(not(feature = "protocol_feature_spice"), ignore)]
fn test_requesting_receipts_when_not_validator() {
    let (genesis, chain) = setup(2, 1);
    let requester_chain = new_chain(&chain, &genesis);

    let block = latest_block(&chain);
    let receipt_proof = new_test_receipt_proof(&block);
    let mut store_update = chain.chain_store.store().store_update();
    save_receipt_proof(&mut store_update, block.hash(), &receipt_proof);
    store_update.commit();

    let producer = producers_of_receipt_proof(&chain, &block, &receipt_proof).swap_remove(0);
    let data_id = SpiceDataIdentifier::ReceiptProof {
        block_hash: *block.hash(),
        from_shard_id: receipt_proof.1.from_shard_id,
        to_shard_id: receipt_proof.1.to_shard_id,
    };
    let to_shard_id = receipt_proof.1.to_shard_id;

    let (outgoing_sc, mut outgoing_rc) = unbounded_channel();
    let mut actor = new_actor_for_account(outgoing_sc, &chain, &producer);

    let requester = AccountId::from_str("not-validator").unwrap();
    actor.handle(SpicePartialDataRequest { data_id, requester: requester.clone() });
    let (partial_data, recipients) = drain_outgoing_partial_data(&mut outgoing_rc).swap_remove(0);
    assert_eq!(recipients, HashSet::from([requester.clone()]));

    let to_shard_uid = {
        let shard_layout = chain.epoch_manager.get_shard_layout(block.header().epoch_id()).unwrap();
        ShardUId::from_shard_id_and_layout(to_shard_id, &shard_layout)
    };

    let (outgoing_sc, mut outgoing_rc) = unbounded_channel();
    let mut actor = ActorBuilder::new(Some(requester))
        .tracked_shards_config(TrackedShardsConfig::Shards(vec![to_shard_uid]))
        .build(outgoing_sc, &requester_chain);
    actor.handle(SpiceIncomingPartialData { data: partial_data });

    let message = outgoing_rc.try_recv().unwrap();
    assert_matches!(outgoing_rc.try_recv(), Err(TryRecvError::Empty));
    let OutgoingMessage::ExecutorIncomingUnverifiedReceipts(ExecutorIncomingUnverifiedReceipts {
        block_hash: reconstructed_block_hash,
        receipt_proof: reconstructed_receipt_proof,
    }) = message
    else {
        panic!();
    };
    assert_eq!(block.hash(), &reconstructed_block_hash);
    assert_eq!(receipt_proof, reconstructed_receipt_proof);
}
