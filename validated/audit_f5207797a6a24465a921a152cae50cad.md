Audit Report

## Title
Router Native ETH Balance Consumed by Any Caller to Satisfy WETH Payment Obligations — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay()` function in `PeripheryPayments` uses `address(this).balance` — the router's entire ambient ETH balance — to satisfy a WETH payment obligation attributed to an external `payer`. Because `multicall` is `payable` and ETH sent with it persists on the router between calls, any subsequent caller specifying `tokenIn = WETH` can have their payment obligation satisfied by stranded ETH from a prior user, receiving full swap output without spending any of their own funds.

## Finding Description
In `PeripheryPayments.pay()` (L69–88), when `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` and, if it is sufficient, deposits that ETH as WETH and transfers it to the recipient — never calling `transferFrom` on `payer`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer is never charged
}
```

The `receive()` guard (L32–34) only blocks direct ETH sends from non-WETH addresses, but `multicall` (L39–44) is `payable`, so ETH sent with a multicall is held by the router. If a user sends `multicall{value: X}` with a swap consuming only `Y < X` ETH and omits `refundETH`, the remaining `X - Y` ETH is stranded on the router.

A subsequent attacker calls `exactInputSingle` (L67–86) with `tokenIn = WETH` and `amountIn = X - Y`. The pool triggers `metricOmmSwapCallback`, which calls `_justPayCallback` (L192–199), which calls `pay(WETH, attacker, pool, X-Y)`. Since `address(this).balance == X-Y >= X-Y`, the first branch fires: the router deposits the stranded ETH as WETH and transfers it to the pool. The attacker's address is never charged via `transferFrom`. The transient callback context correctly binds the expected pool and payer, but `pay()` ignores ETH attribution entirely.

## Impact Explanation
Direct loss of user principal: User A loses up to their full stranded ETH (bounded only by `msg.value` minus what their swap consumed, which can be arbitrarily large). Attacker B receives the full swap output without spending any ETH or WETH. This constitutes both a direct loss of user funds and a swap conservation failure — the pool receives payment from an unattributed source while the attacker's obligation goes unsatisfied.

## Likelihood Explanation
Medium. The precondition — User A omitting `refundETH` from a native ETH multicall — is a common composition mistake, especially for users building multicalls manually or using integrations that do not enforce cleanup. Once ETH is stranded, the exploit is trivially executable by any unprivileged caller in a subsequent transaction with no special permissions, no malicious setup, and no non-standard tokens.

## Recommendation
Do not use `address(this).balance` to satisfy a payment obligation attributed to an external `payer`. Track ETH consumed from `msg.value` explicitly per top-level call (e.g., in transient storage) and only allow `pay()` to draw from that tracked amount. Alternatively, enforce that `address(this).balance == 0` at the end of each top-level payable entry point, or require callers to always include `refundETH` as the final multicall step.

## Proof of Concept
```
// Step 1: User A strands ETH on the router
// User A calls multicall{value: 1 ether} containing exactInputSingle
// with tokenIn=WETH, amountIn=0.5 ether, and no refundETH step.
// pay() fires: nativeBalance=1e18 >= 0.5e18 → deposits 0.5 ETH, transfers WETH to pool.
// After call: address(router).balance == 0.5 ether (stranded)

// Step 2: Attacker B calls exactInputSingle with tokenIn=WETH, amountIn=0.5 ether
// Attacker B sends 0 ETH and has 0 WETH approval.
// Pool triggers metricOmmSwapCallback → _justPayCallback →
//   pay(WETH, attackerB, pool, 0.5 ether):
//     nativeBalance = 0.5 ether >= 0.5 ether  → branch 1 fires
//     WETH.deposit{value: 0.5 ether}()         → stranded ETH becomes WETH
//     WETH.transfer(pool, 0.5 ether)           → pool receives payment
//     transferFrom(attackerB, ...) is NEVER called
// Attacker B receives full token output. User A's 0.5 ETH is permanently lost.
```
