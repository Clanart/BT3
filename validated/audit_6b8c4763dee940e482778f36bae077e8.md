### Title
`OmniBridge::deployToken` Signature Lacks Chain-ID Binding, Enabling Cross-Chain Replay for Unauthorized Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` (and its StarkNet equivalent in `starknet/src/omni_bridge.cairo`) verifies an MPC-produced signature over a `MetadataPayload` that contains no destination chain identifier. Because the same MPC key signs for every supported chain, a valid `deployToken` signature produced for one EVM chain (e.g., Ethereum) can be replayed verbatim on any other OmniBridge deployment (Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract, StarkNet), enabling unauthorized token deployment on chains the NEAR bridge never authorized.

---

### Finding Description

**Root cause — missing `omniBridgeChainId` in the signed metadata payload.**

In `OmniBridge.sol::deployToken`, the message that is hashed and verified against the MPC-derived signer is:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The signed payload contains only token metadata — no chain ID, no contract address, no nonce. Compare this with `finTransfer`, which correctly embeds `omniBridgeChainId` **twice** (once for the token address context and once for the recipient context):

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

The NEAR side generates the signature in `log_metadata_callback` using `MetadataPayload`, which is defined as:

```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
``` [3](#0-2) 

No destination chain field exists. The NEAR bridge signs this payload and emits it in a `LogMetadataEvent`, making the signature publicly observable on-chain.

The StarkNet bridge has the identical flaw. Its `MetadataPayload.to_borsh()` also omits any chain identifier:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [4](#0-3) 

And `deploy_token` on StarkNet verifies the same chain-agnostic payload:

```cairo
fn deploy_token(ref self: ContractState, signature: Signature, payload: MetadataPayload) {
    _verify_borsh_signature(ref self, @payload.to_borsh(), signature);
    ...
}
``` [5](#0-4) 

The Omni Bridge is deployed on at least eight distinct chains (`Eth`, `Arb`, `Base`, `Bnb`, `Pol`, `HyperEvm`, `Abs`, `Strk`), all sharing the same MPC-derived `nearBridgeDerivedAddress`. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` signature (emitted publicly in a NEAR `LogMetadataEvent`, or seen in a mempool for an EVM relay transaction) can:

1. Replay the signature on **any other** OmniBridge deployment (e.g., replay an Ethereum-intended signature on Arbitrum).
2. The token is deployed on the unintended chain with the correct metadata, and the `nearToEthToken` / `ethToNearToken` mappings are set on that chain's bridge.
3. The attacker then calls `bind_token` on NEAR with a proof of the `DeployToken` event emitted by the Arbitrum factory. Since `bind_token_callback` only checks that the emitter is a registered factory:

```rust
require!(
    self.factories.get(&deploy_token.emitter_address.get_chain())
        == Some(deploy_token.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
``` [7](#0-6) 

…the NEAR bridge registers the token for the attacker-chosen chain, enabling `finTransfer` flows to that chain.

**Concrete impact:** Unauthorized token deployment on unintended chains, followed by forced registration of that token in the NEAR bridge's canonical mapping. This breaks the protocol's controlled token-deployment process and can create unbacked token supply on chains that were not yet operationally ready (e.g., missing liquidity, missing relayer infrastructure, or pending security review). This matches the allowed impact class: *"signature verification bypass enabling unauthorized token deployment."*

---

### Likelihood Explanation

The `LogMetadataEvent` containing the MPC signature is emitted publicly on NEAR. Any observer can extract the signature and replay it on any other chain. No privileged access is required — `deployToken` is a permissionless function callable by anyone. The only precondition is that the target chain's OmniBridge is deployed and unpaused, which is the normal operational state.

---

### Recommendation

Include `omniBridgeChainId` in the Borsh-encoded `MetadataPayload` that is signed by the MPC, mirroring the pattern already used in `finTransfer`. Concretely:

**EVM (`OmniBridge.sol`):**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**NEAR (`omni-types/src/lib.rs`):**
```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
+   pub destination_chain: ChainKind,   // added field
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
```

The NEAR `log_metadata_callback` must be updated to accept a target chain and embed it in the payload before signing. The StarkNet `MetadataPayload.to_borsh()` must include the same chain byte.

---

### Proof of Concept

1. Relayer calls `log_metadata("usdc.near")` on the NEAR bridge, targeting Ethereum.
2. NEAR MPC signs `MetadataPayload { prefix: Metadata, token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 }` — no chain ID.
3. The signature is emitted in the public `LogMetadataEvent` on NEAR.
4. Attacker extracts `(signature, payload)` from the NEAR event.
5. Attacker calls `OmniBridgeArbitrum.deployToken(signature, payload)` on the Arbitrum OmniBridge.
6. Signature verification passes (same MPC key, same chain-agnostic payload). Token is deployed on Arbitrum.
7. Attacker calls `bind_token` on NEAR with a Wormhole/prover proof of the Arbitrum `DeployToken` event.
8. NEAR bridge registers `usdc.near ↔ ArbitrumTokenAddress` in its canonical mapping.
9. `finTransfer` flows to Arbitrum for `usdc.near` are now live — on a chain the protocol never authorized for this token.
