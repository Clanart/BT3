### Title
Legacy EIP-712 `timeout_height` Omission from Type Schema Allows Timeout Bypass — (`ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

---

### Summary

The legacy EIP-712 `Tx` type schema deliberately omits `timeout_height` from its field definitions. Because EIP-712 `hashStruct` only commits to fields declared in the type schema, a signed transaction's `timeout_height` value is never covered by the signature hash. An attacker who observes a victim's signed-but-expired legacy EIP-712 transaction can strip the `timeout_height` (set it to 0) and resubmit it with the original signature, bypassing the victim's intended expiry window.

---

### Finding Description

**Root cause — schema omission:**

In `extractMsgTypes`, the `Tx` type definition explicitly excludes `timeout_height`: [1](#0-0) 

The comment confirms this was intentional: `"timeout_height was removed because it was not getting filled with the legacyTx"`.

**The message data DOES contain `timeout_height`:**

In `VerifySignature`, `txBytes` is built by calling `legacytx.StdSignBytes` with `tx.GetTimeoutHeight()`: [2](#0-1) 

When `timeout_height > 0`, the Cosmos SDK `StdSignDoc` serializes it as `"timeout_height": "<N>"` in the JSON (it uses `omitempty`, so it only appears when non-zero). This JSON is then passed as `data` to `LegacyWrapTxToTypedData`: [3](#0-2) 

Inside `LegacyWrapTxToTypedData`, the JSON is unmarshaled into `txData` and set as `typedData.Message`: [4](#0-3) 

**EIP-712 hashing ignores undeclared fields:**

Per the EIP-712 spec, `hashStruct` encodes only the fields listed in the type definition. The `timeout_height` key present in `txData` (the message map) is silently ignored by `apitypes.TypedDataAndHash` because it is not declared in the `Tx` type. The resulting `sigHash` is therefore identical for any two transactions that differ only in `timeout_height`.

**Ante handler ordering confirms exploitability:**

`TxTimeoutHeightDecorator` runs at position 5, well before `LegacyEip712SigVerificationDecorator` at position 13: [5](#0-4) 

A crafted T2 with `timeout_height=0` passes the timeout check first (no timeout enforced), then passes signature verification (hash identical to T1).

---

### Impact Explanation

A victim signs T1 with `timeout_height=X` to limit its validity to blocks before X. If T1 is broadcast to the mempool but not included before block X, an attacker who observed T1 can:

1. Copy T1's `FeePayerSig` from `ExtensionOptionsWeb3Tx`.
2. Construct T2 identical to T1 but with `timeout_height=0`.
3. Submit T2 after block X.

T2 passes `TxTimeoutHeightDecorator` (no timeout) and `LegacyEip712SigVerificationDecorator` (same EIP-712 hash). The victim's transaction executes without their consent, after the expiry window they explicitly set. This constitutes an EIP-712 authorization bypass enabling unauthorized execution — a High-severity impact under the allowed scope.

---

### Likelihood Explanation

- The legacy EIP-712 path (`ExtensionOptionsWeb3Tx`) is still active in the codebase and routed through `newLegacyCosmosAnteHandlerEip712`.
- Mempool transactions are publicly observable; extracting `FeePayerSig` from a pending or expired transaction is trivial.
- The only precondition is that the victim's account sequence has not been incremented by another transaction between T1's expiry and the attacker's submission of T2.
- The code comment explicitly acknowledges the omission, confirming it is a known design choice rather than an oversight that was later corrected.

---

### Recommendation

Add `timeout_height` back to the `Tx` type schema in `extractMsgTypes`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    {Name: "timeout_height", Type: "string"},  // must be included
},
```

Ensure `StdSignBytes` always serializes `timeout_height` (even when 0, e.g., as `"0"`) so the field is always present in the message and covered by the hash. Alternatively, reject legacy EIP-712 transactions that carry a non-zero `timeout_height` until the schema is fixed.

---

### Proof of Concept

```
1. Alice signs T1: {msgs: [MsgDelegate(...)], fees: F, sequence: 5, timeout_height: 100}
   → EIP-712 hash H (timeout_height not in schema, ignored)
   → FeePayerSig = Sign(H, alice_privkey)

2. T1 is broadcast but block 100 passes without inclusion.

3. Attacker observes T1 in mempool, extracts FeePayerSig.

4. Attacker builds T2: {msgs: [MsgDelegate(...)], fees: F, sequence: 5, timeout_height: 0}
   → EIP-712 hash H' = H  (timeout_height still not in schema, still ignored)

5. Attacker submits T2 with FeePayerSig from step 1.

6. TxTimeoutHeightDecorator: timeout_height=0 → no timeout → PASS
7. LegacyEip712SigVerificationDecorator: Ecrecover(H', sig) == alice → PASS
8. T2 commits. Alice's delegation executes after her intended expiry.
```

Two transactions differing only in `timeout_height` produce the same `LegacyWrapTxToTypedData` hash because `timeout_height` is present in `txData` (the message) but absent from the `Tx` type schema, so `apitypes.TypedDataAndHash` never encodes it.
