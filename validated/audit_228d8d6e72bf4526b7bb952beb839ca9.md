### Title
Deflationary ERC20 Token Transfer Inflates Emitted `amount` in `initTransfer`, Breaking Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` transfers a caller-supplied `amount` of a native ERC20 token into the bridge vault, then unconditionally emits that same `amount` in the `InitTransfer` event. For deflationary tokens (tokens that deduct a fee on every transfer), the vault receives strictly less than `amount`, but the cross-chain message records the full `amount`. The destination chain then releases the full `amount` (minus bridge fee) to the recipient, creating an unbacked surplus that permanently undercollateralizes the EVM vault.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native ERC20 tokens in the `else` branch:

```solidity
// OmniBridge.sol lines 406–412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← requested amount, not actual received
    );
}
```

After the transfer, the function emits:

```solidity
// OmniBridge.sol lines 427–436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,            // ← still the caller-supplied value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
```

No pre/post balance check is performed. The `amount` in the event is the caller's input, not `balanceOf(address(this))` delta. The cross-chain relayer reads this event and instructs the destination chain to finalize a transfer of `amount - fee` tokens to the recipient. On the return path, `finTransfer` releases `payload.amount` from the vault:

```solidity
// OmniBridge.sol lines 351–354
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount   // ← full amount from the cross-chain message
    );
}
```

Because the vault received less than `amount` on deposit but is obligated to release `amount` on withdrawal, each deflationary-token bridge cycle drains the vault by the transfer-fee delta.

---

### Impact Explanation

Every `initTransfer` call with a deflationary token creates a shortfall equal to `amount × transfer_fee_rate` in the EVM vault. The destination chain mints or releases the full promised amount, so the bridge's cross-chain supply exceeds its EVM collateral. When users bridge back to EVM, `finTransfer` attempts to release the full `payload.amount`; once the cumulative shortfall exceeds the vault balance, legitimate users' return transfers revert permanently — their funds are irrecoverably locked on the destination chain.

This matches the allowed impact: **High — Balance/accounting corruption that breaks bridge collateralization or misdirects value**, and can escalate to **Critical — Permanent freezing of user funds** as the vault is progressively drained.

---

### Likelihood Explanation

The `initTransfer` function is fully permissionless — any address can call it with any ERC20 token address. No whitelist, registration, or privileged role is required to trigger the vulnerable path. Deflationary tokens (e.g., tokens with transfer taxes) are a well-known ERC20 variant in active use. A single user bridging such a token is sufficient to begin the undercollateralization; repeated use amplifies the shortfall until the vault is insolvent.

---

### Recommendation

Measure the actual received amount using a pre/post balance check and use that value — not the caller-supplied `amount` — in the emitted event and all downstream accounting:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 actualReceived = uint128(
        IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore
    );
    amount = uint128(actualReceived); // shadow with actual received
}
```

Then pass `amount` (now the actual received value) to `initTransferExtension` and the `InitTransfer` event. Alternatively, document that deflationary tokens are explicitly unsupported and add a token-level allowlist enforced on-chain.

---

### Proof of Concept

1. `DEFLT` is an ERC20 token with a 1% on-transfer fee (deflationary).
2. Alice calls `initTransfer(DEFLT, 1000, 0, 0, "near:alice.near", "")`.
3. `safeTransferFrom(Alice, bridge, 1000)` executes; bridge vault receives **990** (1% fee deducted by token).
4. `InitTransfer` event emits `amount = 1000` (caller-supplied, not actual received).
5. Relayer reads the event and submits a finalization to the destination chain for **1000** tokens.
6. Alice receives 1000 tokens on the destination chain; bridge vault holds only 990 on EVM — **10-token shortfall created**.
7. Bob, who previously bridged 1000 DEFLT to EVM legitimately, submits `finTransfer` to reclaim his tokens.
8. `finTransfer` calls `safeTransfer(Bob, 1000)`. If the vault has been drained below 1000 by accumulated shortfalls, this reverts — Bob's funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2)
