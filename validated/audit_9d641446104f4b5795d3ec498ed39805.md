### Title
EIP-7623 Floor Data Gas Missing from Ante Handler Intrinsic Gas Check Causes uint64 Underflow in `ApplyMessageWithConfig` - (File: `x/evm/keeper/state_transition.go`, `x/evm/keeper/gas.go`)

---

### Summary

`GetEthIntrinsicGas` in `x/evm/keeper/gas.go` calls `core.IntrinsicGas` without passing the `IsPrague` flag, so the EIP-7623 floor data gas is never enforced during ante handler validation. A Prague-era transaction with large zero-byte calldata and a `GasLimit` set to just above the standard (non-Prague) intrinsic gas passes all ante handler checks. During execution in `ApplyMessageWithConfig`, the EIP-7623 floor data gas can exceed `msg.GasLimit`, causing an unguarded uint64 underflow at line 550 of `state_transition.go`. The wrapped-around `leftoverGas` value is then caught at line 579 as `ErrGasOverflow`, causing the transaction to fail after fees have already been deducted.

---

### Finding Description

**Root cause — missing `IsPrague` in `GetEthIntrinsicGas`:**

`GetEthIntrinsicGas` passes only seven arguments to `core.IntrinsicGas`, omitting the Prague flag:

```go
// x/evm/keeper/gas.go:34-44
func (k *Keeper) GetEthIntrinsicGas(msg *core.Message, rules params.Rules, isContractCreation bool) (uint64, error) {
    return core.IntrinsicGas(
        msg.Data,
        msg.AccessList,
        msg.SetCodeAuthorizations,
        isContractCreation,
        rules.IsHomestead,
        rules.IsIstanbul,
        rules.IsShanghai,
        // rules.IsPrague is NOT passed
    )
}
``` [1](#0-0) 

Because `IsPrague` is not passed, `core.IntrinsicGas` never computes the EIP-7623 floor data gas component (`tokens × TOKEN_COST_FLOOR`). The ante handler therefore enforces only the standard pre-Prague intrinsic gas, allowing a transaction with `GasLimit < floorDataGas` to pass validation.

**Underflow site — `ApplyMessageWithConfig` line 550:**

After EVM execution and the refund step, the Prague branch applies the EIP-7623 floor:

```go
// x/evm/keeper/state_transition.go:543-554
if rules.IsPrague {
    floorDataGas, err := core.FloorDataGas(msg.Data)
    if err != nil {
        return nil, errorsmod.Wrap(err, "floor data gas")
    }
    if msg.GasLimit-leftoverGas < floorDataGas {
        prev := leftoverGas
        leftoverGas = msg.GasLimit - floorDataGas  // ← uint64 underflow when floorDataGas > msg.GasLimit
        ...
    }
}
temporaryGasUsed = msg.GasLimit - leftoverGas
``` [2](#0-1) 

The guard at line 548 (`msg.GasLimit - leftoverGas < floorDataGas`) is true whenever `floorDataGas > msg.GasLimit - leftoverGas`, which includes the case where `floorDataGas > msg.GasLimit`. The assignment at line 550 is **not** separately guarded against `floorDataGas > msg.GasLimit`. In Go, uint64 subtraction wraps silently; `leftoverGas` becomes `msg.GasLimit - floorDataGas + 2^64`, a near-maximal uint64 value.

The wrapped value is then caught at line 579:

```go
if msg.GasLimit < leftoverGas {
    return nil, errorsmod.Wrapf(types.ErrGasOverflow, "message gas limit < leftover gas (%d < %d)", ...)
}
``` [3](#0-2) 

This returns `ErrGasOverflow` from the message handler after the ante handler has already deducted fees.

---

### Impact Explanation

This is a **High** severity EVM state transition / ante handler accounting bug. A transaction that passes all ante handler checks (fees deducted from the sender) fails during EVM execution with `ErrGasOverflow` due to the uint64 underflow. The user's fees are consumed but the transaction produces no state change. This constitutes a mis-accounting of valid user funds/fees: the fee collector receives payment for a transaction that was never executed, with no refund path triggered because the error occurs after the ante handler completes.

Additionally, if a block proposer includes such a transaction and validators replay it, all validators hit the same `ErrGasOverflow` path deterministically, so the block is not halted — but the fee mis-accounting is committed to state on every node.

---

### Likelihood Explanation

Prague (EIP-4844 / Cancun+) support is present in this codebase (`rules.IsPrague` is checked at line 543). Once the chain activates Prague, any user can craft a transaction with:
- Large zero-byte calldata (e.g., 3 501+ zero bytes)
- `GasLimit` set to `21000 + n×4` (standard intrinsic, no EIP-7623 floor)

For zero bytes: standard data gas = `n×4`; EIP-7623 floor = `n×10`. With `n = 3501`, `floorDataGas = 35010 > GasLimit = 21000 + 14004 = 35004`. The condition is met with a minimal, easily constructed transaction. No special privileges are required.

---

### Recommendation

Pass `rules.IsPrague` to `core.IntrinsicGas` in `GetEthIntrinsicGas`:

```go
return core.IntrinsicGas(
    msg.Data,
    msg.AccessList,
    msg.SetCodeAuthorizations,
    isContractCreation,
    rules.IsHomestead,
    rules.IsIstanbul,
    rules.IsShanghai,
    rules.IsPrague,   // ← add this
)
```

Additionally, add an explicit guard before line 550 in `ApplyMessageWithConfig`:

```go
if floorDataGas > msg.GasLimit {
    return nil, errorsmod.Wrap(types.ErrGasOverflow, "floor data gas exceeds gas limit")
}
leftoverGas = msg.GasLimit - floorDataGas
```

### Proof of Concept

1. Chain is running with Prague rules active (`rules.IsPrague = true`).
2. Attacker constructs an EVM transaction with:
   - `msg.Data` = 3 501 zero bytes
   - `GasLimit` = `21000 + 3501×4 = 35004`
3. Ante handler calls `GetEthIntrinsicGas` → `core.IntrinsicGas` without `IsPrague` → intrinsic gas = `35004`. Check passes.
4. Fees = `35004 × gasPrice` are deducted from the sender.
5. `ApplyMessageWithConfig` executes; EVM runs (trivial call, minimal gas used), `leftoverGas` ≈ `35004`.
6. `core.FloorDataGas(msg.Data)` = `3501 × 10 = 35010 > 35004 = msg.GasLimit`.
7. Condition at line 548: `35004 - leftoverGas < 35010` → true.
8. Line 550: `leftoverGas = 35004 - 35010` → uint64 underflow → `leftoverGas = 2^64 - 6`.
9. Line 579: `35004 < 2^64 - 6` → true → returns `ErrGasOverflow`.
10. Transaction fails; fees already deducted; no refund. [1](#0-0) [4](#0-3)
