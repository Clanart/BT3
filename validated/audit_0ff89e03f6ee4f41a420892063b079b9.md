### Title
`fin_transfer_send_tokens_callback` Silently Ignores `ft_transfer` Failures, Permanently Locking Bridged Funds ŌĆö (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When finalizing a bridge transfer to NEAR for a **non-deployed token with no message** (e.g., native USDC, USDT), the `fin_transfer_send_tokens_callback` does not check whether the underlying `ft_transfer` promise succeeded or failed. If `ft_transfer` is rejected by the token contract (e.g., blacklisted recipient, paused token, or any non-standard restriction), the transfer is still permanently marked as finalized, locked tokens are never restored, and the user's bridged funds are irrecoverably lost.

---

### Finding Description

The finalization flow for NEAR-destined transfers is:

**Step 1 ŌĆö `process_fin_transfer_to_near` (line 1875):** `add_fin_transfer` is called **first**, permanently recording the transfer ID in `finalised_transfers`. This state is committed before any token movement. [1](#0-0) 

**Step 2 ŌĆö `send_tokens` (line 1957ŌĆō1966):** For non-deployed tokens with an empty `msg`, `send_tokens` dispatches `ft_transfer` (not `ft_transfer_call`). [2](#0-1) 

**Step 3 ŌĆö `fin_transfer_send_tokens_callback` (line 1692):** The callback is invoked after `send_tokens` resolves. It decides between a refund path and a success path via `is_refund_required(is_ft_transfer_call)`. [3](#0-2) 

**Step 4 ŌĆö `is_refund_required` (line 1784):** When `is_ft_transfer_call` is `false` (which is always the case when `msg` is empty, per line 1973), the function **unconditionally returns `false`** ŌĆö it never inspects the promise result. [4](#0-3) 

The `is_ft_transfer_call` flag is set at the call site as `!msg.is_empty()`: [5](#0-4) 

**Consequence:** When `ft_transfer` fails (promise rejected), the callback falls through to the **success path** (lines 1719ŌĆō1746), emits `FinTransferEvent`, and sends fees ŌĆö without ever calling `remove_fin_transfer` or `revert_lock_actions`. The transfer ID remains in `finalised_transfers` permanently, making any retry impossible. [6](#0-5) 

The refund path ŌĆö which is the only place `remove_fin_transfer` and `revert_lock_actions` are called ŌĆö is never reached: [7](#0-6) 

---

### Impact Explanation

**Critical ŌĆö Permanent irrecoverable lock of user funds.**

A user who bridges a non-deployed token (USDC, USDT, or any NEP-141 token with a blacklist, pause, or other transfer restriction) from EVM/Solana/StarkNet to NEAR, where the NEAR recipient address is unable to receive the token, will have their funds permanently locked:

- The source-chain tokens are already locked or burned at `initTransfer` time.
- The NEAR-side transfer is marked finalized (nonce consumed), so it cannot be re-submitted.
- The `ft_transfer` failure is silently treated as success.
- No recovery path exists in the contract.

---

### Likelihood Explanation

**Medium.** USDC and USDT ŌĆö both likely to be bridged ŌĆö maintain on-chain blacklists. A blacklisted NEAR address attempting to receive bridged USDC will trigger this path. Additionally, any token that is paused at the moment of finalization, or that enforces recipient whitelists, will trigger the same silent failure. The user need not be malicious; this is a normal operational scenario for compliant stablecoins.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, check the promise result even when `is_ft_transfer_call` is `false`. Use `env::promise_result_checked(0, ...)` or `env::promise_results_count()` / `env::promise_result(0)` to detect a failed `ft_transfer` promise. If the promise failed, execute the same refund path: call `revert_lock_actions`, `remove_fin_transfer`, and emit `FailedFinTransferEvent`. This mirrors the existing `ft_transfer_call` failure handling and ensures no transfer is permanently finalized without the recipient actually receiving funds.

---

### Proof of Concept

1. User holds USDC on EVM and calls `initTransfer` to bridge to NEAR recipient `alice.near`. Tokens are locked in the EVM bridge.
2. The NEAR USDC contract operator blacklists `alice.near`.
3. Relayer calls `fin_transfer` on the NEAR bridge with a valid proof.
4. `process_fin_transfer_to_near` runs: `add_fin_transfer` records the transfer ID as finalized (state committed). `send_tokens` dispatches `ft_transfer(alice.near, amount)` to the USDC contract.
5. USDC contract rejects the transfer (blacklisted). The promise result is `Failed`.
6. `fin_transfer_send_tokens_callback` is invoked. `is_ft_transfer_call = false` (empty msg). `is_refund_required(false)` returns `false` unconditionally.
7. Callback enters the success branch: emits `FinTransferEvent`, attempts to send fees. No `remove_fin_transfer`, no `revert_lock_actions`.
8. Transfer ID is permanently in `finalised_transfers`. Any retry of `fin_transfer` with the same proof reverts with `ERR_TRANSFER_ALREADY_FINALISED`.
9. `alice.near` never receives USDC. The EVM-locked USDC is permanently stranded. No DAO or user action can recover it through the normal bridge flow.
