Looking at the external report's vulnerability class: a **guard mechanism that exists in name but is not effective** — the wrong base class is inherited so the `_update` hook never enforces the paused state. I need to find an analogous pattern in the StarkNet OS scope: a validation function that is called but does not actually constrain anything.

The most direct analog is in `execution_constraints.cairo`.

### Title
Empty `check_is_reverted` Leaves Invoke Transaction Revert Flag Unconstrained — (`execution/execution_constraints.cairo`)

### Summary

`check_is_reverted` in `execution_constraints.cairo` has an empty body. It is called in `execute_invoke_function_transaction` to validate a hint-provided `is_reverted` flag, but because the function does nothing, the flag is entirely unconstrained in the proof. A malicious sequencer can freely set `is_reverted = TRUE` for any valid invoke transaction, causing the execute step to be silently skipped while the fee is still charged — a direct loss of funds for the user.

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`, lines 338–361), the OS loads `is_reverted` from a hint and then calls `check_is_reverted` before branching on the flag:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...)
} else {
    // Align the stack ...
    tempvar _dummy_return_value: ...;
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The call to `check_is_reverted` is the only place where the OS is supposed to constrain the hint-provided value. However, the implementation in `execution_constraints.cairo` is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

The function body is completely empty — it accepts the argument and immediately returns, imposing **zero constraints** on `is_reverted`. Because Cairo's soundness relies on explicit assertions, an unconstrained hint variable is freely choosable by the prover (sequencer). The file is named `execution_constraints.cairo`, making the intent clear: this function was meant to enforce a constraint but does not.

This is structurally identical to the external report: `PausableUpgradeable` is inherited and `pause()`/`unpause()` are callable, but because `_update` is never overridden, the paused state is never checked during transfers. Here, `check_is_reverted` is called, but because its body is empty, the revert state is never actually validated.

### Impact Explanation

**Direct loss of funds (Critical).**

Fee charging is unconditional — it happens regardless of the `is_reverted` branch:

```cairo
// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [3](#0-2) 

A malicious sequencer sets `is_reverted = TRUE` for any valid invoke transaction. The OS proof accepts this (no constraint is violated). The `__execute__` entry point is skipped, but the fee transfer executes. The user's account balance is debited for the fee, and their intended state change never occurs. This is a direct, provably-valid theft of user funds within the protocol's own proof system.

### Likelihood Explanation

Any sequencer running a modified hint implementation can exploit this on every invoke transaction in every block. The OS proof remains valid because no Cairo constraint is violated. The attack is silent, repeatable, and requires no special privilege beyond operating a sequencer node — which in a decentralized StarkNet is an open role. The OS proof is specifically designed to constrain sequencer behavior; this bug nullifies that protection for the execute-step revert flag.

### Recommendation

Implement `check_is_reverted` with an actual constraint. The function must verify that the hint-provided `is_reverted` value matches the true execution outcome. One approach is to derive `is_reverted` from the result of the execution itself rather than from a free hint, or to add a Cairo `assert` that ties the flag to a verifiable on-chain condition (e.g., the result of a dry-run stored in the state diff). At minimum, the function must not be a no-op.

### Proof of Concept

1. User submits a valid invoke transaction (e.g., a token transfer).
2. Malicious sequencer includes the transaction and sets `is_reverted = 1` in the `IsReverted` hint.
3. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
4. The `if (is_reverted == FALSE)` branch is not taken; `__execute__` is skipped entirely.
5. `charge_fee` executes unconditionally, debiting the user's fee token balance.
6. The OS proof is generated and verified on L1 — it is valid because no Cairo constraint was violated.
7. The user's fee is gone; their transaction had no effect. The sequencer can repeat this for every invoke transaction in every block.
