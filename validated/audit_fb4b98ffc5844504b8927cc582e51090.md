### Title
Integer Wrapping in Relative Timelock Arithmetic Allows `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Bypass — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` contains two arithmetic paths for relative timelock evaluation, controlled by the `nowrap: bool` parameter. When `nowrap = false` (the pre-soft-fork / legacy consensus path), the function uses `wrapping_add` to compute the deadline for `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE`. An attacker can supply a crafted large relative value that causes the addition to wrap to zero (or a small number), making the computed deadline appear to be in the past and causing the timelock check to pass immediately — a direct analog to the Solidity `uint128(block.timestamp + duration)` silent overflow.

---

### Finding Description

In `check_time_locks`, the `nowrap = false` branch uses `wrapping_add` for all four relative timelock conditions:

```rust
// ASSERT_SECONDS_RELATIVE (nowrap = false)
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

```rust
// ASSERT_HEIGHT_RELATIVE (nowrap = false)
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

For `ASSERT_SECONDS_RELATIVE`: if an attacker sets `seconds_relative = u64::MAX - unspent.timestamp + 1`, then:

```
unspent.timestamp.wrapping_add(seconds_relative) = 0
```

The check becomes `timestamp < 0`, which is always `false` for a `u64` timestamp. The error is never returned, so the spend is accepted immediately regardless of the current block timestamp. The coin is effectively unlocked at any time.

For `ASSERT_HEIGHT_RELATIVE`: the same wrapping applies with `u32` arithmetic. Setting `height_relative = u32::MAX - confirmed_block_index + 1` wraps the deadline to `0`, and `prev_height < 0` is always false.

The codebase's own tests explicitly document and confirm this behavior:

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),  // <-- bypass: spend accepted when it should be rejected
)]
```

The `nowrap = true` path correctly uses `saturating_add`, which clamps to `u64::MAX` and always produces an error for such large values. The `nowrap = false` path is the legacy consensus behavior used for blocks before the corresponding soft fork activates.

---

### Impact Explanation

When `nowrap = false` is the active consensus path (pre-soft-fork blocks), any unprivileged spender can bypass a relative timelock (`ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE`) by encoding a crafted large value in their CLVM puzzle solution. The coin is accepted as spendable at any block height or timestamp, defeating the intended lock period. This is a **timelock condition validation bypass enabling unauthorized spend acceptance**, matching the High impact category.

---

### Likelihood Explanation

The attacker-controlled entry path is direct: the `seconds_relative` / `height_relative` value is parsed from the CLVM spend solution via `sanitize_uint`, which accepts any value up to `u64::MAX` (8 bytes) without rejecting it as an overflow. The value is stored as-is and passed to `check_time_locks`. No privileged access, key material, or network position is required. Any wallet or puzzle author can craft such a spend. The only constraint is that `nowrap = false` must be the active consensus mode for the block being validated.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally in both the `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE` branches, removing the `nowrap` branching for these two conditions. The `nowrap = false` (wrapping) path should not be reachable for any block height in production consensus. If backward compatibility with pre-fork blocks is required, the soft-fork activation height for `nowrap = true` should be confirmed to cover all blocks where these conditions can appear.

```rust
// Before (nowrap = false branch):
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {

// After (always safe):
if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
```

### Proof of Concept

Using the Python binding exposed via `check_time_locks`:

```python
from chia_rs import check_time_locks, SpendBundleConditions, SpendConditions, CoinRecord
from chia.types.blockchain_format.sized_bytes import bytes32
from chia.util.ints import uint32, uint64

coin_timestamp = uint64(1_000_000)
coin_height = uint32(100)
current_timestamp = uint64(1_000_001)  # only 1 second later
current_height = uint32(101)

# Craft seconds_relative to wrap: u64::MAX - coin_timestamp + 1
# wrapping result: coin_timestamp + (u64::MAX - coin_timestamp + 1) = 0
seconds_relative = uint64(0xFFFF_FFFF_FFFF_FFFF - coin_timestamp + 1)

conds = make_test_conds(seconds_relative=seconds_relative)

# With nowrap=False (legacy path): returns None (no error) — BYPASS
result = check_time_locks(removals, conds, current_height, current_timestamp, False)
assert result is None  # spend accepted despite timelock

# With nowrap=True (safe path): returns error code 105
result = check_time_locks(removals, conds, current_height, current_timestamp, True)
assert result == 105  # AssertSecondsRelativeFailed — correctly rejected
