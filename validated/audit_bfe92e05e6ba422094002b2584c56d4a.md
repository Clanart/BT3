### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Guards, Corrupting DataLayer Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` uses a split code path that applies duplicate-key and duplicate-hash guards only to the last two items in the batch, silently writing duplicate leaf nodes for all earlier items. This corrupts the persistent Merkle tree root and invalidates all subsequent proof-of-inclusion and proof-of-exclusion operations.

### Finding Description

`MerkleBlob::insert` enforces two uniqueness invariants before writing any leaf: [1](#0-0) 

`batch_insert` is the bulk-insertion fast path. When the tree already has two or more leaves, it pops the **last two** items from the input vector and routes them through `insert` (which runs the guards), then iterates over **all remaining items** and writes them directly via `insert_entry_to_blob` with no duplicate check at all: [2](#0-1) 

Any item that is not among the last two in the vector bypasses both the `KeyAlreadyPresent` and `HashAlreadyPresent` guards entirely. If the caller supplies two entries with the same `KeyId` (or the same `Hash`) in positions 0…N-3, both are written as distinct leaf nodes into the blob without error.

This function is directly exposed as a Python binding: [3](#0-2) 

### Impact Explanation

Inserting a duplicate `KeyId` produces two leaf nodes in the blob that share the same key. The `block_status_cache` is updated by each `insert_entry_to_blob` call, so the second write overwrites the cache entry for that key, orphaning the first leaf. The tree's internal hash chain is then computed over a structure that contains a phantom leaf, producing a root hash that does not correspond to any valid set of key-value pairs. All subsequent `get_proof_of_inclusion` calls for that key operate against this corrupted root, and any proof generated can be verified against a root that does not reflect the true committed state. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`py_batch_insert` is the standard bulk-insertion entry point used by the Chia DataLayer service. Any caller that supplies a batch of three or more items where two share a `KeyId` — whether by mistake or by deliberate construction — triggers the bug. No privileged role is required; the Python binding accepts the vector directly from the caller.

### Recommendation

Add the same duplicate guards to the fast path in `batch_insert` before writing each leaf. The simplest fix is to check `block_status_cache.contains_key(key)` and `block_status_cache.contains_leaf_hash(&hash)` inside the `for ((key, value), hash) in keys_values_hashes` loop (lines 587–603), returning `Err(Error::KeyAlreadyPresent())` or `Err(Error::HashAlreadyPresent())` on collision, mirroring the checks already present in `insert`.

### Proof of Concept

```python
from chia_rs import MerkleBlob

blob = MerkleBlob(bytearray())

# Pre-populate so leaf_count > 1 (the guard path is skipped for items 0..N-3)
from chia_rs import KeyId, ValueId
import hashlib

def h(n): return hashlib.sha256(n.to_bytes(8, 'big')).digest()

# Insert 2 seed entries so leaf_count == 2
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(101), ValueId(101), h(101))

# Now batch_insert with 3+ items; items 0 and 1 share KeyId(0) — no guard fires
blob.batch_insert(
    [(KeyId(0), ValueId(0)), (KeyId(0), ValueId(99)), (KeyId(2), ValueId(2))],
    [h(0), h(99), h(2)]
)
# No exception raised; tree now contains two leaves with KeyId(0).
# The committed root is corrupted; proof-of-inclusion for KeyId(0)
# is non-deterministic and does not reflect a valid set state.
root = blob.get_root()
print("Corrupted root:", root.hex())
```

The `insert` single-item path correctly raises `KeyAlreadyPresentError` for the same input, confirming the guard is present but absent in the batch fast path. [4](#0-3) [5](#0-4)
