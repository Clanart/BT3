Audit Report

## Title
Missing Factory Validation in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User-Approved Tokens — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address from the caller and stores it directly into transient pay-context without verifying it against the factory registry. The callback `metricOmmModifyLiquidityCallback` validates only that `msg.sender` matches the stored pool address — which is attacker-controlled — and then reads token addresses from `msg.sender.getImmutables()`, enabling a malicious pool to drain any ERC-20 token the victim has approved to the adder, up to the caller-supplied `maxAmountToken0`/`maxAmountToken1` caps.

## Finding Description
`MetricOmmSwapRouterBase._requireFactoryPool` enforces factory membership on every pool before storing it in transient context: [1](#0-0) 

`MetricOmmPoolLiquidityAdder` has no `FACTORY` immutable and performs no equivalent check. The NatSpec explicitly documents this omission: [2](#0-1) 

`_addLiquidity` stores the caller-supplied `pool` directly into transient pay-context and calls `pool.addLiquidity(...)`: [3](#0-2) 

The callback validates only that `msg.sender == expectedPool` (the attacker-controlled address), then reads token addresses from `msg.sender.getImmutables()` and executes `pay(token0, payer, msg.sender, amount0Delta)`: [4](#0-3) 

Because `token0`/`token1` are read from the malicious pool's own `getImmutables()`, the attacker can return any token address, causing `pay` to execute `IERC20(stolenToken).safeTransferFrom(victim, maliciousPool, amount0Delta)`.

The same unvalidated path is reachable through `addLiquidityWeighted`, where the probe call also goes to the unvalidated pool: [5](#0-4) 

## Impact Explanation
Direct loss of user principal. Any ERC-20 token the victim has approved to `MetricOmmPoolLiquidityAdder` can be stolen up to the `maxAmountToken0`/`maxAmountToken1` values the victim passes. Because users must pre-approve the adder before adding liquidity, the approval surface is always present for active liquidity providers. The attacker receives the tokens directly in the malicious pool contract. This is a Critical/High direct loss of user principal meeting Sherlock thresholds.

## Likelihood Explanation
The attack requires only that a victim be induced to call `addLiquidityExactShares` or `addLiquidityWeighted` with an attacker-controlled `pool` address — a standard phishing or compromised-frontend vector. No privileged role, no special token, and no protocol-level setup is needed. The victim's prior approval to the adder (a normal prerequisite for using the contract) is the only precondition.

## Recommendation
Add a `FACTORY` immutable and a factory membership check at the top of `_addLiquidity` (and before the probe call in `addLiquidityWeighted`), mirroring the pattern already used in `MetricOmmSwapRouterBase`:

```solidity
IMetricOmmPoolFactory internal immutable FACTORY;

// At the start of _addLiquidity:
if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
```

## Proof of Concept
1. Victim approves `MetricOmmPoolLiquidityAdder` for 1000e18 USDC.
2. Attacker deploys `MaliciousPool` whose `getImmutables()` returns `token0 = USDC` and whose `addLiquidity` re-enters `metricOmmModifyLiquidityCallback` with `amount0Delta = 1000e18` and `callbackData = abi.encode(KIND_PAY)`.
3. Attacker calls `adder.addLiquidityExactShares(address(maliciousPool), victim, 0, deltas, 1000e18, 0, "")`.
4. `_addLiquidity` stores `maliciousPool` as `expectedPool` in transient storage and calls `maliciousPool.addLiquidity(...)`.
5. `MaliciousPool.addLiquidity` calls back into `metricOmmModifyLiquidityCallback(1000e18, 0, abi.encode(KIND_PAY))`.
6. Callback passes the `msg.sender == expectedPool` check (both are `maliciousPool`), reads `token0 = USDC` from `maliciousPool.getImmutables()`, and executes `pay(USDC, victim, maliciousPool, 1000e18)`.
7. Victim loses 1000 USDC.
