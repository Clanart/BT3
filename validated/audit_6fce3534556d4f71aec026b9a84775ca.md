### Title
`run_block_generator` silently accepts `SIMPLE_GENERATOR` + non-empty `block_refs` while `run_block_generator2` rejects with `TooManyGeneratorRefs` — (`crates/chia-consensus/src/run_block_generator.rs`)

### Summary
`run_block_generator` (v1) does not call `setup_generator_args` and therefore never enforces the `TooManyGeneratorRefs` guard when `SIMPLE_GENERATOR` is active. `run_block_generator2` (v2) does call `setup_generator_args` and correctly rejects any non-empty `block_refs` under `SIMPLE_GENERATOR`. A block crafted with `SIMPLE_GENERATOR` active and non-empty `block_refs` is accepted by v1 and rejected by v2, producing deterministic consensus divergence between nodes running different implementations.

### Finding Description
`setup_generator_args` enforces the invariant that `SIMPLE_GENERATOR` blocks must have no `block_refs`: [3](#0-2) 

`run_block_generator2` delegates to this function: [4](#0-3) 

`run_block_generator` (v1) instead builds args inline with no equivalent guard: [5](#0-4) 

Because the generator under `SIMPLE_GENERATOR` is a quoted list (`0xff 0x01 ...`), it ignores its arguments entirely. The ROM runs it successfully regardless of what `block_refs` were passed in, so v1 returns `Ok` while v2 returns `Err(TooManyGeneratorRefs)`.

### Impact Explanation
Any node using `run_block_generator` (v1) for consensus validation would accept a block that any node using `run_block_generator2` (v2) rejects. This is a deterministic, reproducible consensus split. The impact matches: *"Valid unprivileged CLVM program, spend bundle, block, proof, or serialized network object can trigger deterministic consensus divergence."*

### Likelihood Explanation
After the `SIMPLE_GENERATOR` hard fork activates, a farmer (an unprivileged network participant with block-production rights) can include non-empty `block_refs` in a block. No key compromise, governance access, or network-level attack is required. The split is triggered by a single crafted block.

### Recommendation
Add the same `SIMPLE_GENERATOR` + `block_refs` guard to `run_block_generator` (v1), either by calling `setup_generator_args` (and discarding the result if the ROM path doesn't need it), or by inserting an explicit early check mirroring lines 50–54 before the inline `block_refs` iteration at line 115.

### Proof of Concept
Call both functions with `SIMPLE_GENERATOR` active and a non-empty `block_refs` slice (e.g., `&[&[0u8, 1, 2, 3]]`) using a valid quoted generator (`0xff 0x01 ...`). Assert that `run_block_generator` returns `Ok` and `run_block_generator2` returns `Err` with `ErrorCode::TooManyGeneratorRefs`. The existing test infrastructure in `test_generators.rs` already exercises `run_block_generator2` for this case: [6](#0-5) 
but no equivalent test covers `run_block_generator` (v1) under the same conditions.
