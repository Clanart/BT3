Audit Report

## Title
`PeripheryPayments.pay()` consumes unscoped router ETH balance, enabling ETH theft and free WETH swaps â€” (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` â€” the router's entire native ETH balance â€” when settling a WETH swap, without scoping it to the current caller or transaction. Combined with an unrestricted `refundETH()` that sends the full contract ETH balance to any caller, ETH stranded on the router from a prior user's `msg.value` can be (A) stolen outright by any caller via `refundETH()`, or (B) silently consumed to fund a different user's WETH swap, giving that user free swap output without pulling their WETH.

## Finding Description
In `pay()`, when `token == WETH` and `payer != address(this)`, the function reads the raw contract balance:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer's WETH never pulled
} else if (nativeBalance > 0) { ... }
``` [1](#0-0) 

`address(this).balance` is not scoped to the current caller or the current transaction's `msg.value`. It includes any ETH left on the router from prior transactions. ETH arrives via `msg.value` on any `payable` function (`multicall`, `exactInputSingle`, `exactInput`, etc.). The `receive()` guard only blocks plain ETH transfers from non-WETH addresses â€” it does not prevent ETH from accumulating via `msg.value`: [2](#0-1) 

`refundETH()` is public, has no access control, and sends the entire contract ETH balance to `msg.sender`: [3](#0-2) 

For Attack Vector B, `exactInputSingle` sets `payer = msg.sender` in transient storage: [4](#0-3) 

The pool then calls `metricOmmSwapCallback`, which dispatches to `_justPayCallback`, which calls `pay(tokenIn, payer, pool, amount)`: [5](#0-4) 

If `tokenIn == WETH` and the router holds stranded ETH â‰Ą `amount`, the pool is paid from the stranded ETH and the attacker's WETH is never pulled.

## Impact Explanation
**Attack Vector A â€” direct ETH theft:** Any caller invokes `refundETH()` after ETH is stranded on the router. The entire router ETH balance is transferred to the attacker. The original depositor loses their ETH with no recourse. This is direct loss of user principal.

**Attack Vector B â€” free WETH swap:** Attacker calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = X`, no `msg.value`, no WETH approval. If the router holds â‰Ą X stranded ETH, `pay()` wraps that ETH and sends WETH to the pool. The attacker receives swap output without spending any of their own assets. The victim who stranded the ETH subsidises the attacker's trade.

Both vectors constitute direct loss of user principal, qualifying as High/Critical under Sherlock thresholds for any non-trivial ETH amount.

## Likelihood Explanation
ETH stranding is a realistic, recurring condition. Users calling `exactInputSingle` with ETH input via `multicall` must manually append `refundETH()`; omitting it is a single-call mistake. Aggregators and wallets batching ETH-input swaps may omit the refund step. A price-limit partial fill also leaves ETH on the router even when the user intended to spend it all. The attacker requires no special permissions, no token approvals, and no privileged role â€” only the ability to call a public function.

## Recommendation
1. Track `msg.value` in transient storage at `multicall` entry and deduct only the current-call allocation in `pay()`, rather than reading the raw `address(this).balance`.
2. Scope `refundETH()` to the current `msg.sender`'s tracked ETH allocation rather than the entire contract balance, or restrict it to within a multicall context where the caller's ETH contribution is known.
3. Alternatively, enforce via a revert guard that every ETH-input multicall ends with `refundETH()`, preventing ETH from persisting across transaction boundaries.
