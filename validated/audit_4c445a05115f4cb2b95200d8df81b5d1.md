### Title
Missing Duplicate-Key/Hash Guard in `batch_insert` Silently Corrupts Merkle Tree State - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a batch containing duplicate `KeyId` or `Hash` values is supplied (beyond the first two items), leaf nodes are written directly into the blob and the `BlockStatusCache` is silently overwritten, producing a structurally invalid Merkle tree whose root hash and proofs of inclusion are incorrect.

### Finding Description
`MerkleBlob::insert` enforces two invariants before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `self.insert()` (with those guards) only for the first two items when the tree has ≤ 1 leaf. All remaining items in the batch are written directly via `insert_entry_to_blob` with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs an unconditional `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

`HashMap::insert` silently overwrites the previous entry. The old leaf node remains in the blob at its original index but is now orphaned — the cache no longer tracks it. The tree now contains two leaf nodes sharing the same `KeyId` or `Hash`, violating the fundamental uniqueness invariant.

The Python binding `py_batch_insert` directly forwards caller-supplied vectors to `batch_insert` with no additional validation:

```rust
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

### Impact Explanation
After a batch with duplicate keys/hashes is processed:

1. The blob contains two leaf nodes with the same `KeyId` or `Hash` at different tree indices.
2. The `BlockStatusCache` maps the key/hash to only the last-written index; the first leaf is permanently orphaned.
3. `calculate_lazy_hashes()` computes the root hash over the corrupted tree structure, producing an incorrect root.
4. `get_proof_of_inclusion()` uses `block_status_cache.get_index_by_key()` to locate the leaf, so it will generate a proof for the last-written duplicate only — the orphaned leaf is unprovable and the root hash is wrong.

This matches: **High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation
The Python binding `MerkleBlob.batch_insert` is a public API callable by any DataLayer client. An unprivileged caller who supplies a list with a repeated `KeyId` or `Hash` (e.g., two entries with the same key but different values) triggers the corruption deterministically. No special privileges, keys, or network access are required. The `check_integrity_on_drop` guard that would catch this is disabled in non-test builds. [5](#0-4) 

### Recommendation
Add duplicate-key and duplicate-hash checks at the start of the bulk loop in `batch_insert`, mirroring the guards in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, validate the entire input batch for uniqueness before any writes begin, so the tree is never left in a partially-written corrupt state on error.

### Proof of Concept
```python
from chia_rs import MerkleBlob

blob = MerkleBlob(bytearray())

# Supply 4 items where item 0 and item 2 share the same KeyId (1)
# Items 3 and 4 are processed by the guarded self.insert() path (leaf_count <= 1)
# Items 0 and 2 go through the unguarded bulk loop
keys_values = [(1, 10), (2, 20), (3, 30), (1, 99)]  # KeyId 1 duplicated
hashes = [b'\x01'*32, b'\x02'*32, b'\x03'*32, b'\x04'*32]

# This should raise KeyAlreadyPresentError but does not
blob.batch_insert(keys_values, hashes)

# The tree now has two leaf nodes with KeyId=1; the cache points only to the
# last-written one. The root hash is computed over a structurally invalid tree.
blob.calculate_lazy_hashes()
root = blob.get_root_hash()
# root is now a hash of a corrupted tree — proofs derived from it are invalid
```
