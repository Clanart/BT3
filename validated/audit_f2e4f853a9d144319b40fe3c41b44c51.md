### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 Allows Emission of Unbacked `InitTransfer` Events — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer1155` makes an external call to an untrusted, attacker-supplied ERC1155 contract (`IERC1155(tokenAddress).safeTransferFrom(...)`) before emitting the `InitTransfer` event. A malicious ERC1155 can reenter `initTransfer1155` during that call, causing multiple `InitTransfer` events to be emitted with distinct nonces while no tokens (or fewer tokens than claimed) are actually locked in the bridge. The NEAR side treats every emitted `InitTransfer` event as proof that tokens are held, so it will mint NEAR-side tokens against events that have no EVM-side collateral.

### Finding Description

`initTransfer1155` executes in this order:

1. `currentOriginNonce += 1` — nonce incremented (line 448)
2. `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` — **external call to attacker-controlled contract** (lines 458–464)
3. `initTransferExtension(...)` — Wormhole message (lines 468–478)
4. `emit BridgeTypes.InitTransfer(...)` — event emitted (lines 480–489) [1](#0-0) 

The ERC1155 `safeTransferFrom` interface gives the token contract full control of execution before returning. A malicious ERC1155 can, inside its `safeTransferFrom` implementation:

1. Reenter `initTransfer1155` (before calling `onERC1155Received` on the bridge).
2. The reentrant call increments the nonce again (nonce = 2), calls `safeTransferFrom` a second time (which may simply return), then emits `InitTransfer` with nonce 2.
3. Control returns to the outer call, which then emits `InitTransfer` with nonce 1.

Result: two `InitTransfer` events with distinct nonces are emitted, but the malicious ERC1155 never actually transferred any tokens to the bridge.

The `onERC1155Received` guard (`operator != address(this)`) does **not** prevent this: the malicious ERC1155 can reenter `initTransfer1155` before invoking `onERC1155Received`, and when it does invoke it, `operator == address(this)` is satisfied because the bridge itself is the caller of `safeTransferFrom`. [2](#0-1) 

The `logMetadata1155` function is permissionless — any caller can register any ERC1155 token address in `multiTokens`, which is the prerequisite for the NEAR side to recognise the `deterministicToken` identifier emitted in `InitTransfer`. [3](#0-2) 

The protocol's own security invariant, documented in `evm/CLAUDE.md`, is:

> **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. If the token transfer reverts or is skipped, the event must not emit — the NEAR side will treat any emitted event as proof that tokens are held. [4](#0-3) 

This invariant is directly violated by the reentrancy path above.

### Impact Explanation

The NEAR bridge processes `InitTransfer` events from registered EVM factories and mints NEAR-side tokens for each unique nonce. Because each reentrant call produces a distinct nonce, the NEAR side sees N independent transfer events and mints N × `amount` tokens, while the EVM bridge holds 0 (or fewer) tokens as collateral. This breaks bridge collateralization: NEAR-side supply exceeds EVM-side locked supply, enabling the attacker to withdraw or sell unbacked bridged tokens.

This matches the allowed impact: **Critical — unauthorized mint of bridged assets** and **High — accounting corruption that breaks bridge collateralization**.

### Likelihood Explanation

The entry path is fully unprivileged:

1. Attacker deploys a malicious ERC1155 contract.
2. Attacker calls `logMetadata1155(maliciousERC1155, tokenId)` (no access control) to register the `deterministicToken` on EVM and emit a `LogMetadata` event for the NEAR side to process.
3. Once the NEAR side has deployed a token for `deterministicToken`, attacker calls `initTransfer1155(maliciousERC1155, tokenId, amount, ...)`.
4. The malicious ERC1155 reenters `initTransfer1155` N times during `safeTransferFrom`, emitting N `InitTransfer` events.
5. NEAR side mints N × `amount` tokens to the attacker's NEAR address.

No privileged keys, no admin access, and no external dependency compromise are required.

### Recommendation

Apply the Checks-Effects-Interactions pattern: emit `InitTransfer` (and call `initTransferExtension`) **before** the external `safeTransferFrom` call, or add OpenZeppelin's `ReentrancyGuard` (`nonReentrant` modifier) to `initTransfer1155`. The analogous `initTransfer` function for ERC20 tokens should also be reviewed for ERC777/callback-capable tokens.

```solidity
// Recommended order in initTransfer1155:
currentOriginNonce += 1;
// ... fee checks ...
// 1. Emit event and extension FIRST (state + effects)
emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...);
initTransferExtension(..., currentOriginNonce, ...);
// 2. Then perform the external token transfer (interaction)
IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
```

Or, equivalently, add `nonReentrant` from OpenZeppelin's `ReentrancyGuard`.

### Proof of Concept

```solidity
contract MaliciousERC1155 is ERC1155 {
    OmniBridge bridge;
    bool reentered;

    function safeTransferFrom(
        address from, address to, uint256 id, uint256 amount, bytes memory
    ) public override {
        if (!reentered) {
            reentered = true;
            // Reenter before transferring — no tokens actually move yet
            bridge.initTransfer1155(
                address(this), id, uint128(amount), 0, 0, "attacker.near", ""
            );
        }
        // Optionally transfer 0 tokens or just return — bridge never checks
        // onERC1155Received is called here with operator == address(bridge) ✓
        _doSafeTransferAcceptanceCheck(msg.sender, from, to, id, amount, "");
    }
}
```

Attack flow:
1. Deploy `MaliciousERC1155`, mint tokens to attacker.
2. Call `bridge.logMetadata1155(malicious, tokenId)` — permissionless registration.
3. Call `bridge.initTransfer1155(malicious, tokenId, 100, 0, 0, "attacker.near", "")`.
4. Two `InitTransfer` events fire (nonce 1 and nonce 2) for 100 tokens each.
5. NEAR side mints 200 tokens to `attacker.near`; bridge holds ≤ 100 tokens. [5](#0-4) [6](#0-5)
