### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from the caller without verifying that the hash corresponds to a previously declared contract class. This is the direct analog of the reported "accept messages from any origin" vulnerability class: just as the wallet extension accepted events without checking their source, the OS here accepts a class hash without checking its origin (i.e., whether it was ever declared). A contract can replace its own class with an undeclared hash, rendering itself permanently unexecutable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 878–916) reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no validation that the hash corresponds to a declared class. The code itself contains an explicit TODO acknowledging the missing check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The function's implicit argument list structurally prevents the validation from being performed — `contract_class_changes` (the dictionary of declared classes) is absent:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
``` [2](#0-1) 

By contrast, the `execute_deploy` syscall (line 452) also uses a caller-supplied `class_hash`, but a failed deployment reverts cleanly. `replace_class` is different: it commits the new hash to the persistent state unconditionally, so there is no recovery path.

The `contract_class_changes` dictionary — which maps declared class hashes to their compiled class hashes — is maintained throughout block execution and is available to other syscall handlers (e.g., `execute_deploy`), but was never threaded into `execute_replace_class`. [3](#0-2) 

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value and the block is committed, the OS has no compiled class facts for that hash. Every subsequent call to the contract will fail at the class-lookup stage. Because `replace_class` cannot be re-invoked (the contract is no longer executable), the state is irrecoverable. Any funds — tokens, ETH, or other assets — held by the contract are permanently frozen.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is available to every contract without restriction. The realistic attack path is:

1. A malicious deployer creates a contract that accepts user deposits (e.g., a fake yield vault or proxy).
2. After accumulating user funds, the deployer calls `replace_class` with an arbitrary felt value that has never been declared as a class.
3. The OS commits the invalid class hash to the state.
4. The contract is permanently bricked; all deposited funds are frozen.

A secondary path exists for any contract that exposes a `replace_class` call reachable by an unprivileged caller (e.g., a proxy with a missing access-control guard on its upgrade function). The OS's missing validation is the necessary protocol-level step that makes either path succeed — without it, the OS would reject the undeclared hash.

**Likelihood: Medium** (requires either a malicious deployer or a contract-level access-control bug, but the OS provides no safety net in either case).

---

### Recommendation

Add `contract_class_changes: DictAccess*` as an implicit argument to `execute_replace_class` and perform a lookup to confirm that `class_hash` exists as a key in `contract_class_changes` (i.e., `new_value != 0`) before updating `contract_state_changes`. This mirrors the existing pattern used during `execute_deploy`, where the class hash is expected to be resolvable to compiled class facts.

---

### Proof of Concept

1. Attacker deploys contract `C` (valid class, accepts deposits, owner-callable `replace_class` wrapper).
2. Users deposit funds into `C`; `C` accumulates a balance.
3. Attacker sends a transaction calling `C`'s upgrade function with `new_class_hash = 0xdeadbeef` (never declared).
4. `C` issues the `replace_class` syscall; the OS executes `execute_replace_class`, skips the missing validation, and writes `class_hash=0xdeadbeef` into `contract_state_changes` for `C`'s address.
5. The block is finalized; the state root now reflects `C.class_hash = 0xdeadbeef`.
6. Any future transaction targeting `C` fails: the OS looks up `0xdeadbeef` in the compiled class facts bundle, finds nothing, and cannot execute the entry point.
7. All user funds in `C` are permanently frozen with no recovery mechanism. [4](#0-3)
