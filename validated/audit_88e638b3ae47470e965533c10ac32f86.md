### Title
`compute_meta_tx_v0_hash` Produces Hash Identical to Deprecated Invoke v0 Transaction, Enabling Cross-Context Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` and the deprecated invoke v0 transaction hash computation both call `deprecated_get_transaction_hash` with the same set of parameters — `INVOKE_HASH_PREFIX`, `version=0`, `max_fee=0`, `additional_data_size=0` — producing an identical Pedersen hash for the same `(contract_address, selector, calldata, chain_id)` tuple. There is no type-domain separator distinguishing a meta-transaction hash from a regular v0 invoke transaction hash. A signature created for one context is cryptographically valid in the other, enabling an unprivileged attacker to replay a victim's v0 invoke signature via the `meta_tx_v0` syscall (or vice versa) without the victim's consent.

---

### Finding Description

**Root cause — identical hash domains:**

`compute_meta_tx_v0_hash` is defined as:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address, entry_point_selector, calldata*, calldata_size, chain_id
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,   // 'invoke'
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,                           // hardcoded zero
        chain_id=chain_id,
        additional_data_size=0,              // hardcoded zero
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

A deprecated invoke v0 transaction hash is computed by the same `deprecated_get_transaction_hash` function with:
- `tx_hash_prefix = INVOKE_HASH_PREFIX`
- `version = 0`
- `max_fee = 0` (when the user sets no fee)
- `additional_data_size = 0` (v0 carries no nonce in the hash) [2](#0-1) 

The two hash computations are **byte-for-byte identical** when `max_fee = 0`. No type tag, no context discriminator, and no nonce separates them.

**Execution path — `execute_meta_tx_v0`:**

The `execute_meta_tx_v0` syscall handler in `syscall_impls.cairo`:
1. Accepts an attacker-supplied `signature_start`/`signature_end` pair.
2. Calls `compute_meta_tx_v0_hash` to produce the hash.
3. Constructs a new `TxInfo` with `version=0`, `nonce=0`, and `transaction_hash=meta_tx_hash`.
4. Directly invokes the target contract's `__execute__` entry point with that `TxInfo`. [3](#0-2) 

No nonce is checked for v0 transactions anywhere in the OS:

```cairo
func check_and_increment_nonce{...}(tx_info: TxInfo*) {
    if (tx_info.version == 0) {
        return ();   // nonce check skipped entirely
    }
    ...
}
``` [4](#0-3) 

**The two vulnerable paths share the same hash with no domain separator:**

| Field | Regular v0 Invoke | `meta_tx_v0` |
|---|---|---|
| `tx_hash_prefix` | `INVOKE_HASH_PREFIX` | `INVOKE_HASH_PREFIX` |
| `version` | `0` | `0` |
| `max_fee` | `0` (when fee-free) | `0` (hardcoded) |
| `additional_data_size` | `0` | `0` (hardcoded) |
| `nonce` in hash | absent | absent |
| Type discriminator | **none** | **none** | [5](#0-4)

### Impact Explanation

**Direct loss of funds (Critical).**

An attacker who observes a victim's signed v0 invoke transaction (with `max_fee=0`) on-chain can:

1. Extract the victim's `(contract_address, calldata, signature)`.
2. Deploy a malicious contract.
3. Submit a regular transaction calling the malicious contract, which internally calls `meta_tx_v0` with the victim's signature and calldata.
4. The OS computes `meta_tx_hash = hash(INVOKE_PREFIX, 0, victim_addr, __execute__, calldata, 0, chain_id, [])` — identical to the original transaction hash.
5. The victim's `__execute__` verifies the signature against this hash — it passes.
6. The victim's `__execute__` runs the calldata (e.g., an ERC-20 transfer) again.
7. Because no nonce is checked for v0, the attacker can repeat this within the same block or across blocks until the victim's account is drained.

The reverse direction also holds: a signature produced for a `meta_tx_v0` (e.g., for a gasless-transaction service) can be submitted as a standalone v0 invoke transaction, executing the victim's `__execute__` without their intent.

---

### Likelihood Explanation

- v0 invoke transactions with `max_fee=0` are actively used in gasless-transaction patterns and by legacy account contracts.
- The `meta_tx_v0` syscall is reachable by any unprivileged contract deployed on the network.
- Signatures are publicly visible on-chain once a transaction is submitted.
- No special privilege, leaked key, or operator access is required — only the ability to deploy a contract and submit a transaction.

---

### Recommendation

Add a domain-separation constant to `compute_meta_tx_v0_hash` that is absent from the regular invoke v0 hash. The simplest fix is to include a dedicated `META_TX_V0_HASH_PREFIX` constant (distinct from `INVOKE_HASH_PREFIX`) as the `tx_hash_prefix`, or to include a non-zero `additional_data` field (e.g., a fixed type tag) that is not present in regular v0 invoke transactions:

```cairo
// In constants.cairo
const META_TX_V0_HASH_PREFIX = 'meta_tx_v0';  // new, distinct prefix

// In transaction_hash.cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(...) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=META_TX_V0_HASH_PREFIX,  // <-- changed
        ...
    );
    return tx_hash;
}
```

This mirrors the fix recommended for H-17: add a type field so that a signature for one operation cannot be replayed in the other.

### Proof of Concept

**Setup:**
- Victim account at address `V` uses a v0 account contract.
- Victim submits a v0 invoke transaction: transfer 100 STRK to attacker address `A`, with `max_fee=0`. Signature `σ` is visible on-chain.

**Attack:**
1. Attacker deploys contract `M` with the following logic in its `__execute__`:
   ```
   meta_tx_v0(contract_address=V, selector=__execute__, calldata=[transfer, A, 100], signature=σ)
   ```
2. Attacker submits a v3 invoke transaction calling `M.__execute__`.
3. Inside `M`, the `meta_tx_v0` syscall fires. The OS computes:
   ```
   hash = Pedersen(INVOKE_PREFIX, 0, V, __execute__, [transfer,A,100], 0, chain_id, [])
   ```
   This equals the original transaction hash that `σ` was created for.
4. The OS calls `V.__execute__` with `tx_info.transaction_hash = hash` and `tx_info.signature = σ`.
5. `V.__execute__` verifies `σ` against `hash` — valid. Executes the transfer: 100 STRK sent to `A`.
6. Attacker repeats step 2 until `V` is drained. No nonce prevents repetition. [6](#0-5) [1](#0-0)

func deprecated_get_transaction_hash{hash_ptr: HashBuiltin*}(

    contract_address: felt,

    calldata_size: felt,

    local calldata_size = request.calldata_end - calldata_start;

    let specific_base_gas_cost = (
        META_TX_V0_GAS_COST + META_TX_V0_CALLDATA_FACTOR_GAS_COST * calldata_size
    );
    let (success, remaining_gas) = reduce_syscall_base_gas(
        specific_base_gas_cost=specific_base_gas_cost, request_struct_size=MetaTxV0Request.SIZE

    if (success == FALSE) {

            calldata=calldata_start,
            calldata_size=calldata_size,
            chain_id=old_tx_info.chain_id,

    );

    let (deprecated_tx_info_ptr: DeprecatedTxInfo*) = alloc();
    tempvar execution_context: ExecutionContext* = new ExecutionContext(
        entry_point_type=ENTRY_POINT_TYPE_EXTERNAL,
        class_hash=state_entry.class_hash,
        calldata_size=calldata_size,
