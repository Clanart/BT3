### Title
Parallel `burn.and(mint)` in `swap_migrated_token` mints new tokens even when old-token burn fails, inflating bridged-token supply — (File: `near/omni-bridge/src/lib.rs`)

### Summary

`swap_migrated_token` fires a `burn` on the old token and a `mint` on the new token as two **parallel** NEAR promises (`burn.and(mint)`). Because the joint promise is immediately `.detach()`ed, the bridge never checks whether the burn succeeded. If the burn fails for any reason, the new tokens are still minted to the user while the old tokens remain in the bridge, inflating the bridged-token supply without a corresponding lock on the source chain.

### Finding Description

When a user calls `ft_transfer_call` on an old (pre-migration) bridge token with `msg = "SwapMigratedToken"`, the bridge receives the old tokens and dispatches:

```rust
// near/omni-bridge/src/lib.rs  lines 2738-2753
fn swap_migrated_token(
    &mut self,
    sender_id: AccountId,
    old_token: AccountId,
    amount: U128,
) -> Promise {
    let new_token = self
        .migrated_tokens
        .get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);

    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);

    burn.and(mint)   // ← both fire unconditionally
}
```

The caller site immediately detaches the result:

```rust
// near/omni-bridge/src/lib.rs  lines 275-279
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))  // ← no refund, no callback
}
```

In NEAR, `Promise::and` schedules both sub-promises to execute **independently**. Neither sub-promise is gated on the other's success. Because the joint promise is `.detach()`ed, the bridge never inspects the outcome. The `ft_on_transfer` return value of `U128(0)` means the old tokens are never refunded to the caller regardless of what happens.

**Failure path that inflates supply:**

1. User calls `ft_transfer_call` on `old_token` → bridge receives `amount` old tokens.
2. Bridge fires `burn(old_token, amount)` **and** `mint(new_token, user, amount)` in parallel.
3. `burn` fails (see likelihood section for realistic triggers).
4. `mint` still executes → user receives `amount` new tokens.
5. Old tokens remain in the bridge (not burned, not refunded).
6. Net: `amount` new tokens exist with no corresponding destruction of old tokens → supply inflated by `amount`.

The correct pattern is sequential: burn first, then mint only inside a callback that verifies the burn succeeded.

### Impact Explanation

For every unit of new token minted without a corresponding burn of old tokens, the total circulating supply of the new bridged token exceeds the amount locked/burned on the source chain. This breaks bridge collateralization: when users redeem new tokens, the bridge cannot back all outstanding supply, causing some redemptions to fail or receive less than owed. This maps directly to the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation

Any condition that causes the `burn` cross-contract call to fail triggers the vulnerability. Realistic triggers reachable without privileged access:

- **Old token contract paused**: if the old token has a pause mechanism (common in bridge tokens) and is paused for maintenance or an emergency, `burn` reverts while `mint` proceeds.
- **Bridge authorization revoked on old token**: if the old token contract was upgraded and the bridge's burn role was not re-granted, every swap attempt inflates supply.
- **Old token contract does not implement `burn`**: if the `migrated_tokens` mapping is ever populated with a token that lacks a `burn` method, the cross-contract call fails silently and the mint still fires.

A user with old tokens can trigger this at any time by calling `ft_transfer_call`; no privileged role is required.

### Recommendation

Replace the parallel `burn.and(mint)` with a sequential pattern: burn first, then mint inside a callback that checks the burn result:

```rust
fn swap_migrated_token(
    &mut self,
    sender_id: AccountId,
    old_token: AccountId,
    amount: U128,
) -> Promise {
    let new_token = self
        .migrated_tokens
        .get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);

    ext_token::ext(old_token)
        .burn(amount)
        .then(
            Self::ext(env::current_account_id())
                .swap_migrated_token_callback(sender_id, new_token, amount),
        )
}

#[private]
pub fn swap_migrated_token_callback(
    &mut self,
    sender_id: AccountId,
    new_token: AccountId,
    amount: U128,
) {
    // Only mint if burn succeeded; otherwise refund or panic
    require!(
        env::promise_result(0) == PromiseResult::Successful(vec![]),
        "Burn failed; aborting mint"
    );
    ext_token::ext(new_token).mint(sender_id, amount, None);
}
```

### Proof of Concept

1. DAO registers `old_token → new_token` in `migrated_tokens`.
2. Old token contract is paused (or bridge's burn authorization is revoked).
3. Attacker holds `N` units of `old_token`.
4. Attacker calls `old_token.ft_transfer_call(bridge, N, "SwapMigratedToken")`.
5. Bridge receives `N` old tokens; fires `burn(old_token, N)` (fails) and `mint(new_token, attacker, N)` (succeeds) in parallel.
6. `ft_on_transfer` returns `U128(0)` — no refund.
7. Attacker now holds `N` new tokens; bridge holds `N` old tokens (not burned).
8. New token total supply is inflated by `N` with no backing destruction. [1](#0-0) [2](#0-1)
