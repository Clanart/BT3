### Title
Unvalidated Constructor Return in `execute_deploy` Syscall Always Reports Success — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The OS-level `execute_deploy` function in `syscall_impls.cairo` unconditionally writes `failure_flag=0` in the syscall response header after invoking the constructor of a newly deployed contract, regardless of whether the constructor actually succeeded or reverted. This is the StarkNet OS analog of the reported "non-validated external call" class: the OS makes an external contract call (the constructor) and does not validate or propagate the outcome back to the caller.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–553), after calling `deploy_contract` (which internally executes the constructor entry point and may revert it via the revert log), the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): consider sharing code with call_contract_helper.
let response_header = cast(syscall_ptr, ResponseHeader*);
let syscall_ptr = syscall_ptr + ResponseHeader.SIZE;

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The developer TODO comment explicitly acknowledges that failure propagation is not yet implemented. Compare this with `contract_call_helper` in the same file, which correctly reads `is_reverted` from `select_execute_entry_point_func` and writes it into the response header:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` path lacks this check entirely. The `deploy_contract` call can internally revert the constructor's state changes via the revert log, but the response written to the syscall segment always signals success to the calling contract.

A secondary unvalidated-call instance exists in `execute_replace_class`, where the OS accepts an arbitrary class hash from the caller without verifying it corresponds to any declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
``` [3](#0-2)

### Impact Explanation

**Primary path — Permanent freezing of funds (Critical):**

When a calling contract (Contract A) invokes the `deploy` syscall to deploy Contract B, and Contract B's constructor reverts:

1. The OS reverts the constructor's storage writes via the revert log.
2. The OS still writes `failure_flag=0` into the syscall response.
3. Contract A's Sierra execution reads `failure_flag=0` and follows the success branch.
4. Contract A may transfer funds to Contract B's address (e.g., as part of an initialization flow).
5. Contract B exists at the address (class hash is set) but its storage is uninitialized (constructor reverted).
6. If Contract B's logic depends on constructor-initialized storage (e.g., an owner address, a token balance mapping), it cannot process incoming funds.
7. Funds sent to Contract B are permanently frozen with no recovery path.

**Secondary path — Proof/state inconsistency (High — chain split):**

If the sequencer (blockifier) correctly propagates `failure_flag=1` to the calling contract's execution trace, but the OS proof hardcodes `failure_flag=0`, the proven state transition diverges from the actual L2 state. The L1 verifier accepts the proof, but the L2 state is inconsistent with it — an unintended chain split.

---

### Likelihood Explanation

The attacker-controlled entry path is fully reachable by any unprivileged transaction sender:

- Any user can deploy a contract that calls the `deploy` syscall.
- Any user can craft constructor calldata that causes the constructor to revert (e.g., by violating an assertion inside the constructor).
- No privileged role, leaked key, or operator cooperation is required.
- The pattern of deploying a contract and immediately sending funds to it is common in DeFi protocols (factory patterns, vault deployments, etc.).

Likelihood is moderate: it requires a specific contract pattern (deploy + fund transfer in the same call), but this pattern is common enough in production contracts.

---

### Recommendation

1. **Propagate the constructor's revert status.** Refactor `execute_deploy` to mirror `contract_call_helper`: capture `is_reverted` from `deploy_contract` and write it into the `ResponseHeader.failure_flag` field instead of hardcoding `0`.

2. **Validate class hash in `execute_replace_class`.** Before updating `contract_state_changes`, verify that `request.class_hash` exists in `contract_class_changes` (i.e., it has been declared). This closes the second unvalidated-call path.

3. **Add a regression test** that deploys a contract with a reverting constructor via the `deploy` syscall and asserts that the response header contains `failure_flag=1`.

### Proof of Concept

**Trigger sequence (no privileged access required):**

```
// Contract A (attacker-deployed)
fn execute(...) {
    // Deploy Contract B whose constructor always panics
    let result = deploy(
        class_hash=REVERTING_CONTRACT_CLASS_HASH,
        calldata=[],
        salt=0,
        deploy_from_zero=false,
    );
    // OS writes failure_flag=0 regardless of constructor outcome
    // result.failure_flag == 0  <-- always, per OS hardcoding
    let deployed_addr = result.contract_address;

    // Send funds to the "successfully deployed" address
    IERC20(fee_token).transfer(deployed_addr, amount);
    // Funds are now frozen: Contract B's storage is uninitialized
    // (constructor reverted), so it cannot process or forward the tokens.
}
