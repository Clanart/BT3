### Title
`fast_fin_transfer_to_near_callback` Proceeds Without Re-Checking Finalization State, Enabling Double-Mint and Permanent Relayer Fund Loss - (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fast_fin_transfer` checks `is_unified_transfer_finalised` before scheduling an async cross-contract call (`check_or_pay_ft_storage`). The callback `fast_fin_transfer_to_near_callback` that executes in a later receipt does **not** re-check whether the transfer has since been finalized via the regular `fin_transfer` path. If `fin_transfer` is processed in the window between the two receipts, the callback proceeds unconditionally: it records the fast transfer, mints tokens to the recipient a second time (for deployed/bridged tokens), and burns the relayer's deposited tokens — leaving the relayer with no reimbursement path and the token supply inflated.

---

### Finding Description

**Entry path:**

A trusted relayer calls `ft_transfer_call` on a bridged token contract, passing a `FastFinTransferMsg` for a NEAR-recipient transfer. This triggers `ft_on_transfer` → `fast_fin_transfer`.

```rust

**Race window:** In NEAR, `check_or_pay_ft_storage` is a cross-contract call that creates a new receipt processed in a subsequent block. Between that receipt and the `fast_fin_transfer_to_near_callback` receipt, another transaction can be processed. A second relayer (or the same actor) can call `fin_transfer` for the same transfer ID during this window.

`process_fin_transfer_to_near` calls `add_fin_transfer` (inserts into `finalised_transfers`) and, finding no fast transfer entry yet, sends tokens to the original recipient:

```rust
let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
...
let fast_transfer_status = self.get_fast_transfer_status(&fast_transfer.id()); // → None
// sends to original recipient, not relayer
``` [5](#0-4) 

When `fast_fin_transfer_to_near_callback` then executes:
- `add_fast_transfer` succeeds (no entry exists yet in `fast_transfers`) [6](#0-5) 
- `send_tokens` mints tokens to the recipient **a second time**
- `resolve_fast_transfer` burns the relayer's deposited tokens and returns `U128(0)` (no refund)
- The fast transfer entry sits in `fast_transfers` with `finalised: false` permanently, because `add_fin_transfer` would now panic with `ERR_TRANSFER_ALREADY_FINALISED` on any future attempt to finalize it [7](#0-6) 

---

### Impact Explanation

**Critical — Unauthorized mint of bridged assets + irrecoverable lock of relayer funds.**

For a deployed (bridged) token:
- The recipient receives `amount_without_fee` tokens twice — once from the regular `fin_transfer` and once from the fast transfer callback — inflating the bridged token supply without a corresponding lock on the origin chain.
- The relayer's deposited tokens are burned with no reimbursement. The fast transfer entry in `fast_transfers` is permanently orphaned: the transfer is already in `finalised_transfers`, so no future `fin_transfer` can ever reimburse the relayer.

For a native (locked) token:
- The bridge sends the relayer's deposited tokens to the recipient as a second payment, draining bridge reserves beyond what was locked.
- The relayer's tokens are permanently locked in the bridge with no recovery path.

Both outcomes match the allowed impact scope: **unauthorized mint of bridged assets** and **permanent freezing / irrecoverable lock of user funds**.

---

### Likelihood Explanation

The race window is at least one NEAR block wide (the `check_or_pay_ft_storage` cross-contract call). In a live environment with multiple competing relayers, both `fast_fin_transfer` and `fin_transfer` for the same transfer can be submitted in the same block by different relayers. No privileged access, leaked key, or colluding MPC signers are required — only two honest relayers acting independently, or a single actor who submits both transactions. `fin_transfer` requires a valid proof, which is publicly available once the origin-chain transaction is finalized.

---

### Recommendation

Add a re-check of `is_unified_transfer_finalised` at the top of `fast_fin_transfer_to_near_callback`, before `add_fast_transfer` and `send_tokens` are called. If the transfer is already finalized, the callback should return the full amount as a refund (so `ft_resolve_transfer` returns the tokens to the relayer):

```rust
pub fn fast_fin_transfer_to_near_callback(...) -> Promise {
    require!(Self::check_storage_balance_result(0), ...);

    // Re-check: the transfer may have been finalized during the async gap
    if self.is_unified_transfer_finalised(&fast_transfer.transfer_id) {
        // Return amount so ft_resolve_transfer refunds the relayer
        return PromiseOrValue::Value(fast_transfer.amount); // or panic to trigger refund
    }
    ...
}
```

This mirrors the guard already present in `fast_fin_transfer` at line 778 and closes the TOCTOU window introduced by the async `check_or_pay_ft_storage` call.

### Proof of Concept

```
Setup:
- Token T is a deployed (bridged) token on NEAR, originating from Ethereum.
- Transfer ID = {origin_chain: Eth, origin_nonce: 42}
- Recipient = alice.near
- Amount = 100 T (normalized)

Step 1: Relayer A calls ft_transfer_call(bridge, 100, FastFinTransferMsg{transfer_id: {Eth,42}, recipient: alice.near, ...})
  → fast_fin_transfer: is_unified_transfer_finalised({Eth,42}) == false ✓
  → Schedules check_or_pay_ft_storage (async receipt, processed next block)

Step 2: [Next block, before fast_fin_transfer_to_near_callback receipt]
  Relayer B calls fin_transfer with valid EVM proof for transfer {Eth,42}
  → fin_transfer_callback → process_fin_transfer_to_near
  → add_fin_transfer({Eth,42}) → inserted into finalised_transfers ✓
  → get_fast_transfer_status({Eth,42}) → None (not yet recorded)
  → send_tokens(T, alice.near, 100) → mints 100 T to alice.near
  alice.near balance: 100 T

Step 3: fast_fin_transfer_to_near_callback executes
  → NO check for is_unified_transfer_finalised
  → add_fast_transfer({Eth,42}) → succeeds (not in fast_transfers yet)
  → send_tokens(T, alice.near, 100) → mints 100 T to alice.near AGAIN
  alice.near balance: 200 T  ← double mint

Step 4: resolve_fast_transfer executes
  → burn_tokens_if_needed(T, 100) → burns 100 T from bridge balance
  → returns U128(0) → no refund to Relayer A

Result:
  - alice.near: 200 T (should be 100 T) — 100 T unauthorized mint
  - Relayer A: 0 T (lost 100 T, no reimbursement possible)
  - fast_transfers[{Eth,42}] = {finalised: false} — permanently orphaned
  - finalised_transfers contains {Eth,42} — no future fin_transfer can reimburse Relayer A
```

    #[private]
    pub fn fast_fin_transfer_to_near_callback(
        &mut self,
        #[serializer(borsh)] fast_transfer: &FastTransfer,
        #[serializer(borsh)] storage_payer: AccountId,
        #[serializer(borsh)] relayer_id: AccountId,
    ) -> Promise {
        require!(
            Self::check_storage_balance_result(0),
            BridgeError::StorageRecipientOmitted.as_ref()
        );

        let OmniAddress::Near(recipient) = fast_transfer.recipient.clone() else {
            env::panic_str(BridgeError::InvalidState.to_string().as_str())
        };

        let required_balance = self
            .add_fast_transfer(fast_transfer, relayer_id, storage_payer.clone())
            .saturating_add(ONE_YOCTO);

        self.update_storage_balance(
            storage_payer,
            required_balance,
            NearToken::from_yoctonear(0),
        );

        env::log_str(
            &OmniBridgeEvent::FastTransferEvent {
                fast_transfer: fast_transfer.clone(),
                new_transfer_id: None,
            }
            .to_log_string(),
        );

        let amount_without_fee = U128(
            fast_transfer
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
        );
        self.send_tokens(
            fast_transfer.token_id.clone(),
            recipient,
            amount_without_fee,
            &fast_transfer.msg,
        )
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(RESOLVE_FAST_TRANSFER_GAS)
                .resolve_fast_transfer(
                    &fast_transfer.token_id,
                    &fast_transfer.id(),
                    amount_without_fee,
                    !fast_transfer.msg.is_empty(),
                ),
        )
    }
