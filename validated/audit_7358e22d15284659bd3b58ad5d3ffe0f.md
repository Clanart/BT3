### Title
Unvalidated `recipient` String in `initTransfer` Enables Permanent Irrecoverable Loss of Bridged Assets — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary
`OmniBridge.initTransfer` accepts an arbitrary `string calldata recipient` with zero format validation. Tokens are burned or locked on the EVM side before the `InitTransfer` event is emitted. If a user supplies a malformed or unparseable recipient address, the NEAR-side finalization fails, leaving the user's tokens permanently burned/locked on EVM with no on-chain recovery path. The same pattern exists in `starknet/src/omni_bridge.cairo`. Neither is acknowledged in the EVM or StarkNet security notes.

---

### Finding Description

**Root cause — EVM:**

`OmniBridge.initTransfer` (lines 373–437) accepts `string calldata recipient` and performs no validation on it whatsoever. [1](#0-0) 

The function first burns or locks the user's tokens: [2](#0-1) 

Then emits the `InitTransfer` event with the unvalidated `recipient` string: [3](#0-2) 

There is no check that `recipient` is non-empty, contains a recognized chain prefix (`near:`, `sol:`, `eth:`, etc.), or that the address portion is parseable.

**Root cause — StarkNet (same pattern):**

`OmniBridge.init_transfer` in Cairo accepts `recipient: ByteArray` with no validation, burns/locks tokens, then emits the event: [4](#0-3) 

**Why funds are lost:**

The NEAR bridge's `fin_transfer_callback` receives the prover result and attempts to use the `recipient` field (already parsed as `OmniAddress` by the prover). `OmniAddress::from_str` applies the following logic: [5](#0-4) 

If the recipient string is malformed (e.g., `""`, `"garbage"`, `"near:invalid!account"`), parsing fails. The `fin_transfer_callback` panics with `BridgeError::InvalidProofMessage`: [6](#0-5) 

The transfer is never stored on NEAR. The EVM-side burn/lock is already irreversible. There is no refund or revert path.

**Contrast with Solana (acknowledged):**

The Solana program has the identical gap — `InitTransferPayload.recipient` is a plain `String` with no validation: [7](#0-6) 

Tokens are burned/locked before the Wormhole message is posted: [8](#0-7) 

The Solana SECURITY.md explicitly acknowledges this: *"No validation of `recipient` string in `InitTransferPayload` — An invalid recipient causes the transfer to fail on the NEAR side after tokens are locked/burned on Solana. Manual intervention would be needed."* [9](#0-8) 

The **EVM** and **StarkNet** equivalents are **not acknowledged** in `evm/SECURITY.md`: [10](#0-9) 

---

### Impact Explanation
**Critical — Permanent irrecoverable lock of user funds in bridge flows.**

Any user who calls `initTransfer` on EVM (or `init_transfer` on StarkNet) with an invalid `recipient` string will have their tokens burned or locked with no on-chain recovery mechanism. The NEAR-side finalization fails, and the user's funds are permanently lost. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation
**Medium.** Cross-chain address formats are non-trivial (`"near:alice.near"`, `"sol:BXss9..."`, `"eth:0x..."`). A user unfamiliar with the required `chain:address` format — or one who pastes a bare address without a chain prefix — will trigger this silently. There is no contract-level enforcement, no revert-and-refund path, and no UI-layer guarantee. The `OmniAddress::from_str` default behavior (treating a colon-free string as an ETH address) means even a plausible-looking bare hex string for a non-ETH destination will be silently accepted on EVM and then fail on NEAR.

---

### Recommendation
1. **Validate `recipient` before burning/locking.** At minimum, require the string to contain a recognized chain prefix and a non-empty address portion. Reject the call if parsing would fail.
2. **Implement a refund/recovery mechanism.** If the NEAR-side finalization fails due to an invalid recipient, allow the user to reclaim their tokens on EVM (analogous to the fix applied to BitVMBridge in PR #10).
3. **Apply the same fix to StarkNet `init_transfer`.** The `recipient: ByteArray` parameter in `starknet/src/omni_bridge.cairo` has the identical gap.

---

### Proof of Concept

1. User calls `OmniBridge.initTransfer(tokenAddress, 1000, 0, 0, "invalid_recipient", "")` on EVM.
2. `BridgeToken(tokenAddress).burn(msg.sender, 1000)` executes — 1000 tokens are permanently burned on EVM.
3. `InitTransfer` event is emitted with `recipient = "invalid_recipient"`.
4. Relayer submits the EVM proof to NEAR `fin_transfer`.
5. NEAR prover attempts `OmniAddress::from_str("invalid_recipient")`: no colon → defaults to `("eth", "invalid_recipient")` → `H160::from_str("invalid_recipient")` → `Err("ERR_INVALID_HEX")`.
6. `fin_transfer_callback` receives an error from `decode_prover_result(0)` and panics with `BridgeError::InvalidProofMessage`.
7. Transfer is never stored on NEAR. User's 1000 tokens are permanently lost with no recovery path.

```markdown
# Security Notes

## Design Decisions (Non-Issues)

These patterns have been reviewed and confirmed as intentional. Do not flag or "fix" them.

- **Fee-on-transfer tokens not supported**: `initTransfer` emits the requested `amount`, not the actual received balance. Fee-on-transfer and rebasing tokens are intentionally unsupported
- **`logMetadata` and `deployToken` are permissionless**: Anyone can call `logMetadata` for any ERC20, and anyone can submit a valid MPC signature to `deployToken`. This is by design — the bridge is fully permissionless
- **`ENearProxy.burn` uses empty NEAR recipient**: `eNear.transferToNear(amount, "")` is intentional — `transferToNear` is a legacy method used purely as a burn mechanism. The actual NEAR recipient is tracked in the OmniBridge `InitTransfer` event
- **`deployToken` signature has no chain ID**: Metadata signatures are intentionally chain-agnostic — one NEAR-side signature deploys the same token on all EVM chains

## Known Issues

Low-severity items acknowledged but not yet addressed:

- **`addCustomToken` can overwrite existing mappings** (H-01): Admin-only function. No existence check — calling with an already-mapped `nearTokenId` silently overwrites `nearToEthToken`. Accepted as operational risk
- **`pause(flags)` replaces all flags** (H-02): `_pause(flags)` does full replacement, not bitwise OR. Calling `pause(PAUSED_INIT_TRANSFER)` when `PAUSED_FIN_TRANSFER` is set will unpause finTransfer. Use `pauseAll()` for emergencies
- **`BridgeToken.initialize` stores metadata redundantly** (L-01): `__ERC20_init(name_, symbol_)` writes to parent storage that is never read (getters are overridden). Minor gas waste on init
- **`require` strings instead of custom errors** (L-02): Several locations use `require` with string messages instead of custom errors (`OmniBridge.sol:150,204,556`, `SelectivePausableUpgradable.sol:100,107`, `ENearProxy.sol:56,76,86`)
- **`OmniBridgeWormhole` has no `__gap`** (L-04): Three storage variables with no gap array. Safe as a leaf contract but would need a gap if inherited from
- **`PayloadType.ClaimNativeFee` defined but unused** (L-05): Enum value 2 is never referenced. Native fees are recovered via `finTransfer` with `tokenAddress=address(0)`
```
