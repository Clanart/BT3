### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Unbacked Cross-Chain Minting ŌĆö (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge::initTransfer` accepts an arbitrary ERC20 token and emits an `InitTransfer` event carrying the caller-supplied `amount`. For fee-on-transfer tokens the bridge actually receives `amount ŌłÆ fee`, but the cross-chain message records the full `amount`. The NEAR bridge reads that recorded value from the proof and mints/releases the full `amount` on the destination chain, creating unbacked supply.

---

### Finding Description

In `OmniBridge::initTransfer`, the regular ERC20 lock path is:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 406-411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ŌåÉ caller-supplied, not verified post-transfer
);
```

Immediately after, the function emits:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ŌåÉ same caller-supplied value, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
```

`SafeERC20.safeTransferFrom` only checks the boolean return value; it does **not** verify the post-transfer balance increase. For a fee-on-transfer token the bridge silently receives `amount ŌłÆ transfer_fee` while the event records `amount`.

The NEAR bridge's `fin_transfer_callback` reconstructs the transfer message directly from the proof of that event:

```rust
// near/omni-bridge/src/lib.rs  lines 722-732
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
```

`init_transfer.amount` is the value extracted from the on-chain `InitTransfer` log ŌĆö the original `amount`, not the actual tokens held. The NEAR side then mints or releases that full `amount` to the recipient.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token creates a permanent collateral shortfall: the EVM bridge holds `amount ŌłÆ fee` tokens but the NEAR side mints `amount` tokens. Repeating this inflates the circulating supply of the bridged token beyond what is locked, breaking bridge collateralization. An attacker who controls or deploys a fee-on-transfer token that is accepted by the bridge (the non-`isBridgeToken`, non-`customMinters` path imposes no token whitelist) can mint an unbounded amount of unbacked bridged tokens on NEAR, then redeem them for real assets held in the NEAR bridge ŌĆö constituting direct theft of protocol funds.

This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

The entry point is fully unprivileged: any address may call `initTransfer`. The regular ERC20 path has no token whitelist ŌĆö any address that is neither in `isBridgeToken` nor in `customMinters` is accepted. Fee-on-transfer tokens are a well-established token pattern (deflationary tokens, tokens with protocol fees, etc.). No special role, leaked key, or colluding party is required.

---

### Recommendation

Record the actual balance increase rather than trusting the caller-supplied `amount`:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "zero received");
// use actualReceived in place of amount for the event and all downstream accounting
```

Alternatively, restrict the non-bridge-token path to a whitelist of known-good tokens, or document and enforce that fee-on-transfer tokens are not supported (and add a runtime check that reverts if `actualReceived != amount`).

---

### Proof of Concept

1. Deploy `FeeToken`, an ERC20 that deducts a 10 % fee on every `transferFrom` (keeps 90 % of `amount` in the recipient, burns 10 %).
2. Approve `OmniBridge` for 1 000 `FeeToken` units.
3. Call `OmniBridge.initTransfer(FeeToken, 1000, 0, 0, "near-recipient.near", "")`.
4. `safeTransferFrom` succeeds; bridge receives **900** tokens.
5. `InitTransfer` event is emitted with `amount = 1000`.
6. Relayer submits proof to NEAR bridge; `fin_transfer_callback` reads `init_transfer.amount = 1000` and mints **1 000** bridged `FeeToken` to `near-recipient.near`.
7. Bridge holds 900 tokens; 1 000 are in circulation on NEAR ŌĆö **100 tokens unbacked**.
8. Repeat N times; each iteration inflates supply by 100 tokens.
9. Redeem the accumulated unbacked tokens via a normal `fin_transfer` back to EVM, draining real tokens from the bridge vault. [1](#0-0) [2](#0-1) [3](#0-2)
