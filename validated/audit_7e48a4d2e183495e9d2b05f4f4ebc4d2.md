### Title
Reentrancy via ERC777/ERC20 Callback in `initTransfer` Causes `currentOriginNonce` Staleness, Resulting in Nonce Collision and Permanent Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer` increments `currentOriginNonce` at the top of the function but reads it again from storage — after an external ERC20 `safeTransferFrom` call — when passing it to `initTransferExtension` and `emit`. If the ERC20 token implements a transfer callback (e.g., ERC777 `tokensToSend` hook), a reentrant call increments `currentOriginNonce` again before the outer call reads it. Both the outer and inner calls then publish Wormhole messages and emit `InitTransfer` events carrying the same (final) `originNonce`, causing a nonce collision on NEAR that permanently locks the outer transfer's tokens in the bridge.

### Finding Description

In `OmniBridge.initTransfer`, the execution order is:

1. `currentOriginNonce += 1` — storage write (L381)
2. `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` — external call (L407–411)
3. `initTransferExtension(..., currentOriginNonce, ...)` — reads `currentOriginNonce` from storage (L415–425)
4. `emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...)` — reads `currentOriginNonce` from storage (L427–436) [1](#0-0) 

Steps 3 and 4 read `currentOriginNonce` from storage **after** the external call at step 2. For ERC777 tokens (which are ERC20-compatible), `safeTransferFrom` triggers the `tokensToSend` hook on the sender before completing the transfer. A sender contract can exploit this hook to reenter `initTransfer`:

- **Outer call**: `currentOriginNonce` → 1; external call fires hook → reentrant call begins
- **Reentrant call**: `currentOriginNonce` → 2; external call (no further reentrancy); `initTransferExtension(nonce=2)` publishes Wormhole message with `originNonce=2`; emits `InitTransfer(originNonce=2)` ✓
- **Outer call resumes**: `initTransferExtension(currentOriginNonce=2)` publishes Wormhole message with `originNonce=2` ✗ (should be 1); emits `InitTransfer(currentOriginNonce=2)` ✗ (should be 1)

Both Wormhole messages carry `originNonce=2`. NEAR processes the first and rejects the second as a duplicate nonce. The outer transfer's tokens (already transferred to the bridge at step 2) are permanently locked with no corresponding valid NEAR-side settlement.

The same structural flaw exists in `initTransfer1155`, where `currentOriginNonce` is also read from storage after `IERC1155.safeTransferFrom`. [2](#0-1) 

In `OmniBridgeWormhole`, `initTransferExtension` passes `currentOriginNonce` (stale) directly into the Borsh-encoded Wormhole payload, making the corruption cross-chain. [3](#0-2) 

### Impact Explanation

The outer transfer's ERC20 tokens are irrecoverably locked in the bridge contract. NEAR's nonce deduplication causes it to reject the second `InitTransfer` message with the same `originNonce`, so no mint or unlock is ever issued for the outer transfer. This is a **permanent, irrecoverable fund lock** matching the Critical/High allowed impact class.

Additionally, the Wormhole message for the outer transfer carries a corrupted `originNonce` (the inner call's nonce), constituting **accounting corruption** that breaks bridge collateralization: the EVM bridge holds more locked tokens than NEAR's ledger accounts for.

### Likelihood Explanation

ERC777 tokens are ERC20-compatible and widely deployed (e.g., imBTC, CHAI). Any such token that is bridgeable (not a `isBridgeToken` or `customMinters` entry) reaches the vulnerable `safeTransferFrom` path. An attacker needs only to:

1. Hold an ERC777 token that is accepted by the bridge on both EVM and NEAR sides.
2. Deploy a contract implementing the ERC777 `tokensToSend` hook that calls `initTransfer` again.
3. Call `initTransfer` once to trigger the reentrancy.

No privileged access, no compromised keys, and no colluding parties are required. The entry path is fully unprivileged.

### Recommendation

Capture `currentOriginNonce` into a local variable immediately after incrementing it, before any external call, and use only the local variable in `initTransferExtension` and `emit`:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce; // capture before external calls
    ...
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...
    initTransferExtension(msg.sender, tokenAddress, nonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, nonce, ...);
}
```

Apply the same fix to `initTransfer1155`. Alternatively, add OpenZeppelin's `ReentrancyGuard` (`nonReentrant` modifier) to both functions.

### Proof of Concept

```
Attacker contract (implements ERC777 tokensToSend hook):

function tokensToSend(...) external {
    if (!reentered) {
        reentered = true;
        // Reentrant call — currentOriginNonce becomes 2
        OmniBridge(bridge).initTransfer(token, innerAmount, fee, 0, recipient, "");
    }
}

Attack sequence:
1. attacker.attack() calls OmniBridge.initTransfer(token, outerAmount, fee, 0, recipient, "")
2. OmniBridge: currentOriginNonce = 1
3. OmniBridge: safeTransferFrom(attacker, bridge, outerAmount)
   → ERC777 fires tokensToSend on attacker contract
   → Attacker calls OmniBridge.initTransfer(token, innerAmount, ...)
     → OmniBridge: currentOriginNonce = 2
     → safeTransferFrom(attacker, bridge, innerAmount) [no further hook]
     → initTransferExtension(nonce=2) → Wormhole msg: originNonce=2 ✓
     → emit InitTransfer(nonce=2) ✓
   ← tokensToSend returns
4. OmniBridge: initTransferExtension(currentOriginNonce=2) → Wormhole msg: originNonce=2 ✗
5. OmniBridge: emit InitTransfer(currentOriginNonce=2) ✗

Result:
- Two Wormhole messages with originNonce=2
- NEAR processes first (innerAmount minted), rejects second
- outerAmount tokens permanently locked in OmniBridge with no NEAR-side settlement
``` [4](#0-3)
