### Title
Relative Timelock Bypass via Integer Wrapping in `check_time_locks` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
The `check_time_locks` function contains two arithmetic paths for relative timelock evaluation, controlled by a `nowrap` boolean. When `nowrap=false`, it uses `wrapping_add` for both `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` checks. An attacker who can submit a spend with a maximum-value relative condition (`0xFFFFFFFF` for height, `0xFFFFFFFFFFFFFFFF` for seconds) causes the addition to wrap to a value smaller than the coin's confirmation index/timestamp, making the timelock appear already satisfied and allowing the spend to be accepted immediately.

### Finding Description

In `check_time_locks`, the relative height check has two branches:

```rust
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { ... }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { ... }
}
``` [1](#0-0) 

The same dual-path pattern applies to `seconds_relative`, `before_height_relative`, and `before_seconds_relative`: [2](#0-1) 

**Concrete overflow scenario for `ASSERT_HEIGHT_RELATIVE`:**

- Coin confirmed at block `100`; `height_relative = 0xFFFF_FFFF` (maximum `u32`)
- `100u32.wrapping_add(0xFFFF_FFFF) = 99` (wraps to 99)
- Check: `prev_transaction_block_height < 99` → `False` at any height ≥ 99
- Result: timelock considered satisfied; spend accepted immediately

The project's own test suite explicitly documents and asserts this divergence:

```rust
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [3](#0-2) 

The same wrapping bypass is documented for `ASSERT_SECONDS_RELATIVE`: [4](#0-3) 

The `nowrap` parameter is exposed directly to Python via `py_check_time_locks`: [5](#0-4) 

The `height_relative` and `seconds_relative` values originate from CLVM conditions parsed by `parse_args` / `sanitize_uint`. The parser accepts any value up to `0xFFFF_FFFF` (4-byte max) for height and `0xFFFF_FFFF_FFFF_FFFF` (8-byte max) for seconds — both are valid, unprivileged inputs from a spend bundle: [6](#0-5) [7](#0-6) 

### Impact Explanation

When `nowrap=false` is active, any spend bundle containing `ASSERT_HEIGHT_RELATIVE(0xFFFFFFFF)` or `ASSERT_SECONDS_RELATIVE(0xFFFFFFFFFFFFFFFF)` passes the timelock check at any block height or timestamp. A coin whose puzzle enforces a relative timelock (e.g., a vesting contract, a singleton with a cooldown, or any puzzle that uses relative conditions to gate spending) can be spent immediately, defeating the timelock entirely. This is a **timelock condition validation bypass enabling unauthorized spend acceptance**, matching the High impact tier.

### Likelihood Explanation

The `nowrap` boolean is passed from the Python full node layer. The `nowrap=false` path (wrapping) is the pre-existing behavior retained for backward compatibility with pre-fork block validation. If the Python full node passes `nowrap=False` for any block height range still reachable by new spend submissions (i.e., the fork activating `nowrap=True` has not yet fully activated or is not enforced for all code paths), an attacker can submit a crafted spend bundle with a max-value relative condition and have it accepted. The attacker-controlled input is entirely within the CLVM condition list of a spend bundle — no privileged access is required beyond possession of the spending key for the coin.

### Recommendation

1. Remove the `nowrap=false` / `wrapping_add` branch entirely from `check_time_locks`. Use `saturating_add` unconditionally for all relative timelock arithmetic.
2. If backward compatibility with historical blocks is required, apply the wrapping path only to read-only re-validation of already-committed historical blocks, never to mempool acceptance or new block validation.
3. Add a consensus-level guard that rejects any `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value that would cause `confirmed_index + relative > u32::MAX` (or `u64::MAX`), independent of the arithmetic path chosen.

### Proof of Concept

1. Craft a puzzle: `(AND (AGG_SIG_ME <pk>) (ASSERT_HEIGHT_RELATIVE 4294967295))` — the relative lock is `0xFFFFFFFF` blocks.
2. Create a coin with this puzzle at block height `N`.
3. At block height `N+1` (or any height ≥ `N-1` after wrapping), sign and submit a spend of this coin.
4. Call `check_time_locks` with `nowrap=false`.
5. `confirmed_block_index.wrapping_add(0xFFFF_FFFF)` = `N - 1`; the check `prev_height < N-1` is `False`; the spend is accepted with no error — the ~4-billion-block timelock is bypassed entirely.

pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    // strip the leading zero byte if there is one
    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }

    Ok(SanitizedUint::Ok(u64_from_bytes(buf)))
