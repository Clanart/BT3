### Title
Recipient-Controlled `ft_on_transfer` Rejection Removes Replay Protection, Enabling Cross-Chain Double-Spend — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a NEAR-destined `fin_transfer` uses `ft_transfer_call` (non-empty `msg`), the recipient contract's `ft_on_transfer` can deliberately return the full token amount (rejecting all tokens). This causes `fin_transfer_send_tokens_callback` to call `remove_fin_transfer`, which **removes the transfer ID from `finalised_transfers`** — the sole replay-prevention guard. The same proof can then be re-submitted by a trusted relayer, causing the bridge to process the transfer a second time and deliver tokens twice from a single cross-chain event.

---

### Finding Description

The `process_fin_transfer_to_near` function first marks the transfer as finalized via `add_fin_transfer` (inserting into `finalised_transfers`), then calls `send_tokens`, which — when `msg` is non-empty — issues an `ft_transfer_call` to the recipient contract. [1](#0-0) [2](#0-1) 

After `ft_transfer_call` completes, `fin_transfer_send_tokens_callback` is invoked. It calls `is_refund_required`, which returns `true` when the `ft_resolve_transfer` result is `0` — meaning the recipient's `ft_on_transfer` returned the full amount (rejected all tokens) or panicked. [3](#0-2) 

When `is_refund_required` returns `true`, the callback calls `remove_fin_transfer`: [4](#0-3) 

`remove_fin_transfer` calls `self.finalised_transfers.remove(transfer_id)`, permanently erasing the replay-prevention record: [5](#0-4) 

After removal, `add_fin_transfer` will succeed again for the same `(origin_chain, origin_nonce)` pair, because `finalised_transfers.insert` only panics on duplicate: [2](#0-1) 

---

### Impact Explanation

A trusted relayer that observes a `FailedFinTransferEvent` (emitted at line 1717) may automatically retry the same proof. On the second submission:

- For **non-deployed (locked) tokens**: `revert_lock_actions` restores the locked-token counter on the first rejection, so the locker has sufficient balance for a second unlock-and-send. The recipient receives the full amount twice.
- For **deployed (bridged) tokens**: `burn_tokens_if_needed` burns the rejected tokens on the first attempt. On the second attempt, `mint` is called again, creating new unbacked supply.

Both cases constitute a **cross-chain double-spend** or **unbacked supply creation**, matching the High impact category. [6](#0-5) 

---

### Likelihood Explanation

The attacker must:
1. Be the sender of a cross-chain transfer and set `msg` to a non-empty string pointing to their own NEAR contract as recipient — fully within the sender's control via `init_transfer`.
2. Have their recipient contract's `ft_on_transfer` return the full token amount on the first delivery attempt.
3. Have a trusted relayer re-submit the same proof.

Step 3 is the main constraint. However, the bridge emits `FailedFinTransferEvent` on rejection, which relayer infrastructure may treat as a signal to retry. The attacker can also socially engineer a retry by claiming a transient failure. The `msg` field is embedded in the on-chain proof and is set by the user at transfer initiation time, making step 1 and 2 fully attacker-controlled with no privileged access required. [7](#0-6) [8](#0-7) 

---

### Recommendation

Do **not** remove the transfer from `finalised_transfers` on a failed `ft_transfer_call`. The finalization record should be permanent once set. Instead, on rejection:

- Keep the transfer in `finalised_transfers` (do not call `remove_fin_transfer`).
- Revert lock actions and burn/refund tokens as currently done.
- Emit the `FailedFinTransferEvent` to signal the failure.
- If re-delivery is desired, implement a separate, explicit re-delivery mechanism that does not re-use the original proof's transfer ID as the replay key, or require a new signed authorization. [4](#0-3) 

---

### Proof of Concept

1. Attacker initiates a transfer on the source chain (e.g., Ethereum) with `msg = "<attacker_near_contract>"` and `recipient = <attacker_near_contract>`.
2. Trusted relayer calls `fin_transfer` with the valid proof. `add_fin_transfer` inserts `(Eth, nonce_N)` into `finalised_transfers`.
3. Bridge calls `ft_transfer_call(attacker_contract, amount, msg)`. Attacker's `ft_on_transfer` returns `amount` (full rejection).
4. `fin_transfer_send_tokens_callback` fires: `is_refund_required` returns `true` (result is `U128(0)`). `remove_fin_transfer` removes `(Eth, nonce_N)` from `finalised_transfers`. `revert_lock_actions` restores locked tokens. `FailedFinTransferEvent` is emitted.
5. Relayer observes `FailedFinTransferEvent` and re-submits the same proof. `add_fin_transfer` succeeds (no duplicate). Bridge calls `ft_transfer_call` again.
6. Attacker's `ft_on_transfer` returns `0` (accepts all tokens). Attacker receives the full bridged amount.
7. Attacker has received tokens twice from a single source-chain transfer, with the locker drained or unbacked supply minted.

    pub fn fin_transfer_send_tokens_callback(
        &mut self,
        #[serializer(borsh)] transfer_message: TransferMessage,
        #[serializer(borsh)] fee_recipient: &AccountId,
        #[serializer(borsh)] is_ft_transfer_call: bool,
        #[serializer(borsh)] storage_owner: &AccountId,
        #[serializer(borsh)] lock_actions: Vec<LockAction>,
    ) {
        let token = self.get_token_id(&transfer_message.token);

        if Self::is_refund_required(is_ft_transfer_call) {
            self.burn_tokens_if_needed(
                token.clone(),
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
            );

            self.revert_lock_actions(&lock_actions);

            self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);

            env::log_str(
                &OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string(),
            );
        } else {
            // Send fee to the fee recipient
            if transfer_message.fee.fee.0 > 0 {
                if self.is_deployed_token(&token) {
                    ext_token::ext(token)
                        .with_static_gas(MINT_TOKEN_GAS)
                        .mint(fee_recipient.clone(), transfer_message.fee.fee, None)
                        .detach();
                } else {
                    ext_token::ext(token)
                        .with_attached_deposit(ONE_YOCTO)
                        .with_static_gas(FT_TRANSFER_GAS)
                        .ft_transfer(fee_recipient.clone(), transfer_message.fee.fee, None)
                        .detach();
                }
            }

            if transfer_message.fee.native_fee.0 > 0 {
                let native_token_id = self.get_native_token_id(transfer_message.get_origin_chain());

                ext_token::ext(native_token_id)
                    .with_static_gas(MINT_TOKEN_GAS)
                    .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
                    .detach();
            }

            env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
        }
    }

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()

    }

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
