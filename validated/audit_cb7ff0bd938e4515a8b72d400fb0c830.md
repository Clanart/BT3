### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay for `meta_tx_v0` Syscall — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes the signed hash for the `meta_tx_v0` syscall without including any nonce. Simultaneously, `check_and_increment_nonce` explicitly skips all nonce enforcement for version-0 transactions. Together, these two facts mean that any signature a user produces for a `meta_tx_v0` call can be replayed an unlimited number of times by any on-chain observer, enabling repeated execution of the same call on behalf of the victim and direct loss of funds.

---

### Finding Description

**Root cause — no nonce in the hash:**

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

The fields committed to by the hash are therefore only: `INVOKE_HASH_PREFIX`, `version=0`, `contract_address`, `entry_point_selector`, `calldata`, `max_fee=0`, and `chain_id`. **No nonce is present.** Compare this with every other transaction type (invoke v3, declare v3, deploy-account v3, L1-handler), all of which include a nonce either through `CommonTxFields.nonce` or through `additional_data=&nonce`. [2](#0-1) [3](#0-2) 

**Root cause — nonce enforcement explicitly bypassed for version 0:**

`check_and_increment_nonce` — the sole OS-level replay-prevention gate — returns immediately without any check or state update when `tx_info.version == 0`: [4](#0-3) 

Because `compute_meta_tx_v0_hash` hard-codes `version=0`, every `meta_tx_v0` execution falls into this early-return path. No per-account counter is read, checked, or incremented.

**Contrast with all other account transactions:**

Every other account transaction type calls `check_and_increment_nonce` after filling `tx_info`, and the nonce is part of the signed hash: [5](#0-4) [6](#0-5) 

The `meta_tx_v0` path has no equivalent call.

---

### Impact Explanation

The `meta_tx_v0` syscall is designed to let a contract execute a call on behalf of a user who has signed the corresponding hash. Because the hash contains no nonce and the OS enforces no nonce, a signature that was legitimately produced and broadcast once is permanently valid. An attacker who observes the signature on-chain can submit it again and again, causing the same call (e.g., a token transfer, an approval, a withdrawal) to execute repeatedly on behalf of the victim. This constitutes **direct, unbounded loss of funds** — a Critical impact under the allowed scope.

---

### Likelihood Explanation

The attacker's entry path requires no privileges:

1. Any **unprivileged transaction sender** can observe a victim's `meta_tx_v0` signature in a confirmed block (it is public).
2. The attacker deploys or reuses any contract that accepts and forwards a `meta_tx_v0` call.
3. The attacker submits the replay transaction; the OS computes the identical hash (same inputs, no nonce), verifies the signature successfully, and executes the call.

No key compromise, no operator collusion, and no race condition is required. The window of opportunity is unlimited — the signature remains valid forever because there is no state that invalidates it.

---

### Recommendation

1. Add a `nonce` parameter to `compute_meta_tx_v0_hash` and pass it as `additional_data` to `deprecated_get_transaction_hash` (mirroring the L1-handler pattern at line 234).
2. Maintain a per-account nonce mapping in the OS specifically for `meta_tx_v0` executions, separate from the account transaction nonce.
3. In the `meta_tx_v0` syscall handler, read the expected nonce, assert it matches the value committed to in the hash, and increment it atomically — analogous to `check_and_increment_nonce` for regular transactions.
4. Remove or guard the blanket `version == 0` early-return in `check_and_increment_nonce` so that future version-0 paths cannot accidentally inherit the same bypass.

---

### Proof of Concept

1. **Victim** signs a `meta_tx_v0` hash committing to `(contract_address=TokenContract, selector=transfer, calldata=[attacker, 1000])`. The hash is `H = Pedersen(INVOKE_PREFIX, 0, TokenContract, transfer_selector, H(calldata), 0, chain_id)`.
2. Victim submits the transaction; it is included in a block. The signature `σ` is now publicly visible.
3. **Attacker** constructs a new transaction calling the same contract with the same `(selector, calldata, σ)`.
4. The OS calls `compute_meta_tx_v0_hash` with the same inputs → produces the same `H`. Signature verification passes.
5. `check_and_increment_nonce` is reached with `version=0` → returns immediately, no state change.
6. The transfer executes: 1000 tokens leave the victim's account.
7. Attacker repeats steps 3–6 until the victim's balance is zero. Each iteration costs only the attacker's gas fee.

// Verifies that the transaction's nonce matches the contract's nonce and increments the
// latter.
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
}
