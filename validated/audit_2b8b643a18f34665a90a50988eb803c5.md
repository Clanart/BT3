### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Breaks EVM Bridge Collateralization — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

The EVM `OmniBridge.initTransfer` function accepts a caller-supplied `amount` parameter, transfers that nominal amount from the user via `safeTransferFrom`, and immediately emits an `InitTransfer` event carrying that same nominal `amount`. No balance-before/balance-after check is performed. For fee-on-transfer ERC-20 tokens (tokens that deduct a percentage during every `transferFrom`), the bridge vault receives strictly less than `amount`, yet the cross-chain message carries the full nominal value. When the transfer is finalized on NEAR, the full nominal `amount` is released to the recipient, making the EVM bridge vault permanently undercollateralized.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` path for non-bridge, non-custom-minter ERC-20 tokens is:

```solidity
// lines 406-411
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // nominal amount, not verified against actual receipt
    );
}
```

Immediately after, the event is emitted with the same nominal value:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // nominal amount — may exceed actual balance received
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

The `InitTransfer` event is the authoritative cross-chain message that the NEAR relayer reads to finalize the transfer. The NEAR bridge's `fin_transfer` path releases exactly the `amount` encoded in the proven event to the recipient. [3](#0-2) 

Because no actual-balance check is performed on the EVM side, every `initTransfer` call with a fee-on-transfer token creates a deficit: the vault holds `amount - transfer_fee` but the cross-chain message promises `amount`. Each such transfer widens the undercollateralization gap.

---

### Impact Explanation

**Allowed impact class: Balance/accounting corruption that breaks bridge collateralization.**

The EVM bridge vault accumulates a growing shortfall. The first users to bridge back from NEAR to EVM receive their full entitlement, draining the vault faster than it was filled. Subsequent users attempting to withdraw find the vault insolvent and their funds permanently locked in the bridge — an irrecoverable loss of user funds.

The severity is **Critical** when the token is widely used (e.g., a reflection-mechanic token or a token with an enabled transfer fee), because a single large deposit can immediately render the vault unable to service all outstanding obligations.

---

### Likelihood Explanation

- Any unprivileged user can call `initTransfer` with any ERC-20 token address; there is no allowlist enforced at the contract level for the non-bridge-token path.
- Fee-on-transfer ERC-20 tokens are a well-known, deployed token class (reflection tokens, tokens with configurable fees, Token-2022 transfer-fee extension equivalents on EVM, etc.).
- The attacker does not need any special role, leaked key, or colluding party — they simply call `initTransfer` with a qualifying token.
- The deficit compounds silently with every such deposit; no on-chain alarm is raised.

---

### Recommendation

Replace the fixed-`amount` transfer with a balance-before/balance-after pattern and use the measured delta as the canonical bridged amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "zero received");
// use actualReceived (cast to uint128 with overflow check) in place of amount
// for the event and all downstream accounting
```

This mirrors the standard recommendation for Yearn-style integrations: measure what was actually received rather than trusting the nominal parameter.

---

### Proof of Concept

1. Alice deploys or uses an existing fee-on-transfer ERC-20 token `FOT` with a 1% transfer fee.
2. Alice approves `OmniBridge` for 1,000 `FOT` and calls `initTransfer(FOT, 1000, 0, 0, "near:bob.near", "")`.
3. `safeTransferFrom` executes: the bridge vault receives 990 `FOT` (1% fee deducted); Alice's balance decreases by 1,000.
4. The `InitTransfer` event is emitted with `amount = 1000`.
5. The NEAR relayer picks up the event and calls `fin_transfer` with `amount = 1000`.
6. The NEAR bridge mints/releases 1,000 `FOT`-equivalent tokens to `bob.near`.
7. Bob bridges back: the NEAR bridge burns 1,000 tokens and the EVM bridge is asked to release 1,000 `FOT`.
8. The vault only holds 990 `FOT`; the release either fails (Bob's funds are permanently locked) or, if other deposits exist, drains them (other users' funds are stolen).
9. Repeating step 2–6 N times creates an N×10 `FOT` deficit, eventually making the vault completely insolvent. [4](#0-3)
