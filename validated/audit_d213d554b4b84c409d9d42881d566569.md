### Title
`batch_insert()` Bypasses Duplicate-Key Guards, Corrupting DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert()` skips the `KeyAlreadyPresent` / `HashAlreadyPresent` duplicate-key checks that `insert()` enforces. When a caller supplies a batch containing a key that already exists in the tree (or a key that appears twice within the same batch), duplicate leaf nodes are silently written into the blob. The in-memory `block_status_cache` silently overwrites its `key_to_index` entry for the duplicate, leaving an orphaned leaf in the blob that is invisible to the cache. The resulting Merkle tree is structurally corrupt: the committed root hash is computed over a tree that contains phantom leaves, and any inclusion proof derived from it misrepresents the actual key-value state.

---

### Finding Description

`insert()` (lines 362–413) guards against duplicates at the very top of its body:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert()` (lines 570–657) calls `self.insert()` only for the first two items when the tree has ≤ 1 existing leaves (lines 578–584). Every subsequent item in the batch is written directly to the blob via `insert_entry_to_blob()` (line 601) **without any duplicate check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { key, value, hash, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob()` (lines 1013–1030) always calls `block_status_cache.add_leaf()`:

```rust
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    ...
}
```

`add_leaf()` (lines 188–192) uses `HashMap::insert`, which **silently overwrites** the existing entry for a duplicate key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites old entry
    self.leaf_hash_to_index.insert(leaf.hash, index);   // overwrites old entry
}
```

After the overwrite:
- The blob contains **two** leaf nodes with the same key (at indexes I₁ and I₂).
- The cache's `key_to_index` map points only to I₂.
- The leaf at I₁ is an orphan: it is part of the blob's byte array and is included in hash computations, but is invisible to all cache-based lookups (`get_leaf_by_key`, `get_proof_of_inclusion`, `delete`, etc.).

The root hash is then computed over a tree that contains this phantom leaf, so the committed root no longer faithfully represents the key-value set.

---

### Impact Explanation

This maps to the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Concrete consequences:

1. **Root hash corruption.** `calculate_lazy_hashes()` computes the root over all nodes reachable from index 0, including the orphaned duplicate leaf. The resulting root hash does not correspond to the intended key-value set.

2. **Inclusion proof invalidity.** `get_proof_of_inclusion()` (lines 1155–1196) builds a proof by walking the cache-tracked path. For the duplicate key it returns a proof anchored to I₂, but the blob also contains a structurally connected leaf at I₁ that contributes to ancestor hashes. The proof's `valid()` check (lines 40–58 of `proof_of_inclusion.rs`) may pass against the corrupted root while misrepresenting the actual committed state.

3. **Silent delete failure.** `delete()` (lines 736–790) resolves the key through the cache to I₂ and removes it, but the orphaned leaf at I₁ remains in the blob and continues to influence the root hash. A subsequent `calculate_lazy_hashes()` will incorporate I₁, producing a root that still reflects the "deleted" key.

4. **Cache/blob divergence detected only on explicit `check_integrity()`.** `check_integrity_on_drop` is `false` in production (line 328: `cfg!(test)`), so the corruption is not automatically detected.

---

### Likelihood Explanation

`batch_insert()` is exposed directly through the Python bindings (`py_batch_insert`, lines 1503–1519) and is the primary bulk-load API for the DataLayer. Any caller — including DataLayer clients that supply user-controlled key-value pairs — can trigger this path by:

- Passing a batch that contains a key already present in the tree (when `leaf_count > 1`), or
- Passing a batch with an internally repeated key (any batch size ≥ 3 when `leaf_count ≤ 1`, or any batch size ≥ 1 when `leaf_count > 1`).

No privileged role is required; the Python binding accepts the batch directly from the caller.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path of `batch_insert()` that `insert()` already enforces. Before writing each leaf in the loop (line 587), check:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, deduplicate the input vector before processing and verify no key in the batch collides with any key already in the tree. The `BlockStatusCache::contains_key` and `contains_leaf_hash` methods (lines 170–176) are already available for this purpose.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 (batch_insert bulk path is active)
blob.insert(KeyId(1), ValueId(1), h(1))
blob.insert(KeyId(2), ValueId(2), h(2))

# batch_insert with a key (KeyId(1)) that already exists — no error raised
blob.batch_insert(
    [(KeyId(1), ValueId(99)), (KeyId(3), ValueId(3)), (KeyId(4), ValueId(4))],
    [h(99), h(3), h(4)],
)

blob.calculate_lazy_hashes()

# Root hash is now computed over a tree containing a phantom duplicate leaf
root = blob.get_root_hash()
print("Corrupted root:", root.hex())

# check_integrity reveals the corruption
try:
    blob.check_integrity()
    print("Integrity OK (unexpected)")
except Exception as e:
    print("Integrity failure (expected):", e)
```

The `check_integrity()` call will raise `IntegrityKeyToIndexCacheIndex` or `IntegrityKeyToIndexCacheLength`, confirming that the blob and cache have diverged.     fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
