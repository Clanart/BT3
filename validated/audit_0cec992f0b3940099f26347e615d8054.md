### Title
Solana `deploy_token` Missing Pause Check Allows Token Deployment and Wormhole Message Posting During Emergency Pause — (`solana/programs/bridge_token_factory/src/lib.rs`)

### Summary

The Solana bridge program's `deploy_token` instruction has no pause guard, while the EVM and StarkNet counterparts both define and enforce a dedicated `PAUSED_DEPLOY_TOKEN` flag. When the bridge is paused via `ALL_PAUSED`, `deploy_token` remains callable, creates on-chain mint and Metaplex metadata accounts, and posts a Wormhole confirmation message back to NEAR — all state-changing cross-chain operations that the pause mechanism is intended to block.

### Finding Description

The Solana `constants.rs` defines only two pause flags:

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED;
``` [1](#0-0) 

There is no `DEPLOY_TOKEN_PAUSED` bit. The `deploy_token` entry point in `lib.rs` performs no pause check before proceeding:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
``` [2](#0-1) 

Compare with `init_transfer` and `finalize_transfer`, which both gate on the pause flags: [3](#0-2) [4](#0-3) 

The EVM bridge enforces `whenNotPaused(PAUSED_DEPLOY_TOKEN)` on `deployToken`: [5](#0-4) 

The StarkNet bridge asserts `!_is_paused(@self, PAUSE_DEPLOY_TOKEN)` at the top of `deploy_token`: [6](#0-5) 

`initialize_token_metadata` in `deploy_token.rs` does three things: initialises a new SPL mint PDA, creates Metaplex metadata, and calls `self.common.post_message(payload)?` to post a Wormhole VAA back to NEAR confirming the deployment: [7](#0-6) 

### Impact Explanation

When the bridge is paused (e.g., `ALL_PAUSED` set by the pausable admin in response to a security incident), any relayer holding a valid MPC-signed `DeployTokenPayload` can still call `deploy_token` on Solana. This:

1. Permanently creates a new wrapped-token mint PDA and Metaplex metadata account on Solana.
2. Posts a Wormhole message to NEAR recording the new `(near_token_id → solana_mint)` mapping.

Once the Wormhole VAA is processed by NEAR, the token mapping is established on both sides. When the bridge is later unpaused, `finalize_transfer` can immediately mint tokens for this newly registered token — exactly the "plan during pause, execute after unpause" pattern the pause mechanism is designed to prevent. If the pause was triggered because of a compromised MPC key or a bad metadata payload (wrong decimals, wrong token ID), the attacker can still cement a malicious token mapping on Solana and NEAR before the incident is resolved, leading to accounting corruption or misdirected value when the bridge resumes.

This matches: **High — Wormhole message execution and unauthorized token deployment bypassing the pause control, enabling token-mapping corruption that breaks bridge collateralization or misdirects value.**

### Likelihood Explanation

Any relayer (unprivileged, permissionless role) who holds a valid MPC-signed `DeployTokenPayload` — including one obtained before the pause was triggered — can call `deploy_token` at any time. No special privilege beyond a valid signature is required. The window is the entire duration of the pause.

### Recommendation

Add a `DEPLOY_TOKEN_PAUSED` constant and enforce it in `deploy_token`, mirroring the EVM and StarkNet implementations:

```rust
// In constants.rs
pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;

// In lib.rs deploy_token
pub fn deploy_token(ctx: Context<DeployToken>, data: SignedPayload<DeployTokenPayload>) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
        error::ErrorCode::Paused
    );
    // ...
}
```

### Proof of Concept

1. Admin calls `pause` (sets `ALL_PAUSED`) on Solana in response to a security incident.
2. `init_transfer` and `finalize_transfer` are now blocked.
3. Relayer holds a valid MPC-signed `DeployTokenPayload` for token `bad.near` with incorrect decimals.
4. Relayer calls `deploy_token` on Solana — succeeds with no pause check.
5. A new SPL mint is created and a Wormhole VAA is posted to NEAR recording `bad.near → <mint>`.
6. NEAR processes the VAA and records the mapping.
7. Admin unpauses the bridge.
8. Relayer calls `finalize_transfer` with a valid MPC signature, minting tokens under the corrupted decimal mapping to an attacker-controlled recipient.
