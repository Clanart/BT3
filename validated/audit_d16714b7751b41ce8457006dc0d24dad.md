### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Validation, Corrupting DataLayer Merkle Tree Roots and Proofs of Inclusion — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` silently skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces whenever the tree already contains two or more leaves. Any caller — including the public Python binding `py_batch_insert` — can supply a batch containing repeated `KeyId`s or `Hash`es and the function will write all of them into the blob without error, silently overwriting `BlockStatusCache` entries and leaving orphaned leaf nodes in the raw blob. The resulting tree has a corrupted root hash and produces structurally invalid `ProofOfInclusion` objects, satisfying the allowed High impact: *DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state*.

---

### Finding Description

`MerkleBlob::insert` enforces two guards before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path for all items beyond the first two when the tree already has `leaf_count > 1`. It calls `insert_entry_to_blob` directly, which performs **no** duplicate checking:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` unconditionally calls `block_status_cache.add_leaf(index, leaf)`:

```rust
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    ...
}
``` [3](#0-2) 

`BlockStatusCache` stores `key_to_index: HashMap<KeyId, TreeIndex>` and `leaf_hash_to_index: HashMap<Hash, TreeIndex>`. A `HashMap::insert` on a duplicate key silently overwrites the previous entry. The old leaf node remains physically present in the blob at its original index but is no longer reachable through the cache. The new leaf node is tracked at a different index. The two nodes are wired into the tree at different positions, so the internal-node hashes computed during `calculate_lazy_hashes` reflect a tree that contains both nodes, while the cache only knows about one of them. [4](#0-3) 

The Python binding exposes this path without any additional guard:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

The public type stub confirms the binding is part of the stable API:

```python
def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...
``` [6](#0-5)

### Impact Explanation

After a `batch_insert` call with duplicate keys on a tree that already has ≥ 2 leaves:

1. **Corrupted root hash** — `calculate_lazy_hashes` walks the full blob and computes internal-node hashes over both the orphaned and the live leaf, producing a root that does not correspond to any valid set of key-value pairs.
2. **Invalid `ProofOfInclusion`** — `get_proof_of_inclusion` looks up the key in `key_to_index`, finds the *new* leaf's index, and builds a proof path through the tree. Because the orphaned leaf is still wired into the tree at a different position, the sibling hashes along the path are wrong and `ProofOfInclusion::valid()` returns `false` for a key that is genuinely present.
3. **Forged exclusion** — a verifier that checks `proof.valid()` against the published root hash will reject a legitimately included key, effectively proving exclusion of an included item.

`ProofOfInclusion::valid()` recomputes hashes bottom-up and compares against `combined_hash` stored in each layer:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(...);
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [7](#0-6) 

---

### Likelihood Explanation

The Python binding `py_batch_insert` is the primary interface used by the DataLayer node software when bulk-loading key-value pairs from a store. A DataLayer store owner (an unprivileged role — anyone can create a store) who publishes a dataset containing a repeated `KeyId` will cause every subscribing node to call `batch_insert` with that duplicate, silently corrupting each subscriber's local Merkle tree. No special privilege beyond store ownership is required.

The existing test `test_double_insert_fails` only exercises the single-item `insert` path and does not cover `batch_insert` with duplicates, so the gap is untested. [8](#0-7) 

---

### Recommendation

Add duplicate-key and duplicate-hash checks inside the fast path of `batch_insert` before calling `insert_entry_to_blob`. The simplest fix is to collect all incoming keys and hashes into a temporary `HashSet` and cross-check against `block_status_cache` before writing any node:

```rust
for ((key, value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Additionally, add a test that calls `batch_insert` with a duplicate key on a tree that already has ≥ 2 leaves and asserts that an error is returned.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import os

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 (triggers the fast path)
kv = [(KeyId(i), ValueId(i)) for i in range(2)]
hs = [os.urandom(32) for _ in range(2)]
blob.batch_insert(kv, hs)
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# Now batch_insert a third item that duplicates KeyId(0)
# Under insert() this would raise KeyAlreadyPresentError; batch_insert silently accepts it.
dup_hash = os.urandom(32)
blob.batch_insert([(KeyId(0), ValueId(99))], [dup_hash])
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# Root hash has changed even though no new unique key was added
assert root_before != root_after, "root corrupted"

# Proof for KeyId(0) is now structurally invalid
proof = blob.get_proof_of_inclusion(KeyId(0))
assert not proof.valid(), "proof invalid after duplicate batch_insert"
