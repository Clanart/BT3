### Title
Unbounded `waiting_for_block` HashMap in `SpiceChunkValidatorActor` Enables Memory Exhaustion DoS - (`chain/client/src/spice/chunk_validator_actor.rs`)

### Summary

`SpiceChunkValidatorActor` stores incoming `SpiceChunkStateWitness` and `SpiceChunkContractAccesses` messages in an unbounded `HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>` called `waiting_for_block` when the referenced block is not yet in the store. Unlike the non-Spice orphan witness path, no size limit, distance-from-head check, or eviction policy is applied before insertion. Any peer can send witnesses referencing fabricated block hashes that will never arrive, causing the map to grow without bound until the validator node is killed by OOM.

### Finding Description

`SpiceChunkValidatorActor` declares `waiting_for_block` as a plain `HashMap` with no capacity bound:

```rust
// chain/client/src/spice/chunk_validator_actor.rs:63-64
waiting_for_block:
    HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>,
``` [1](#0-0) 

When `process_chunk_state_witness` is called and `witness_processing_readiness` returns `NotReady` (block not in store), the witness is pushed directly into the map with no guard:

```rust
// Line 267
self.waiting_for_block.entry(chunk_id.block_hash).or_default().0.push(witness);
``` [2](#0-1) 

The same applies to `SpiceChunkContractAccesses` when the block is not yet available:

```rust
// Line 523
self.waiting_for_block.entry(chunk_id.block_hash).or_default().1.push(accesses);
``` [3](#0-2) 

The code itself acknowledges both omissions with identical TODO comments:

> `// TODO(spice): Implement additional checks (size limit, distance to head) before adding witness to `waiting_for_block`. See non-spice handle_orphan_witness().` [4](#0-3) [5](#0-4) 

Entries are only removed when the matching block arrives via `process_waiting_for_block`:

```rust
let Some((witnesses, contract_accesses)) = self.waiting_for_block.remove(&block_hash)
``` [6](#0-5) 

For fabricated block hashes that never arrive, entries are **never removed**. There is no periodic cleanup, no height-based eviction, and no total-size cap.

By contrast, the non-Spice orphan witness path enforces:
- A distance-from-head range check (`ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD`)
- A per-witness size cap (`max_orphan_witness_size`)
- An LRU cache with a fixed capacity (`default_orphan_state_witness_pool_size`) [7](#0-6) 

The `partial_chunk_data` field uses an `LruCache` bounded by `MAX_PENDING_CHUNKS = 24`, and the code comment explains the ~3 GiB memory budget that bound provides. `waiting_for_block` has no equivalent bound. [8](#0-7) 

### Impact Explanation

Each `SpiceChunkStateWitness` can be up to `MAX_UNCOMPRESSED_STATE_WITNESS_SIZE` (64 MiB). An attacker sending N witnesses referencing distinct non-existent block hashes causes `waiting_for_block` to hold N × 64 MiB of data. When the validator node's process memory is exhausted, the OS kills it (OOM). The validator is then unable to produce chunk endorsements, permanently removing it from Spice consensus until manually restarted. Because the attack is cheap (send-and-forget network messages) and the cleanup path never fires for fake hashes, the DoS is persistent across the node's uptime.

**Severity: High** — permanent loss of a validator's ability to participate in Spice chunk validation.

### Likelihood Explanation

Any network peer can send `SpiceChunkStateWitnessMessage` to a validator node. No signature on the witness is verified before the message is stored in `waiting_for_block`; signature verification only occurs later inside `spice_pre_validate_chunk_state_witness`, which is called only when the block arrives. Because the block never arrives for fabricated hashes, the signature check is never reached. The attack requires only the ability to open a TCP connection to the validator and send P2P messages — no stake, no validator role, no privileged access.

### Recommendation

Apply the same guards used by the non-Spice orphan witness path before inserting into `waiting_for_block`:

1. **Distance check**: reject witnesses whose `height_created` is outside `ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD` relative to the current chain head.
2. **Size check**: reject witnesses exceeding `max_orphan_witness_size`.
3. **Capacity cap**: replace the plain `HashMap` with a bounded structure (e.g., an LRU map keyed by block hash, with a per-block-hash Vec length limit). When the cap is reached, drop the oldest or smallest entry.
4. **Periodic cleanup**: evict entries whose referenced block height has fallen below the final head height.

The TODO comments at lines 266 and 522 already identify this gap; the fix is to resolve them before the Spice feature is enabled in production.

### Proof of Concept

1. Attacker connects to a Spice validator node as a network peer.
2. Attacker generates `K` distinct random 32-byte values as fake `block_hash` values.
3. For each fake `block_hash`, attacker crafts a `SpiceChunkStateWitnessMessage` with `chunk_id.block_hash` set to that value and sends it to the validator.
4. `witness_processing_readiness` returns `NotReady` for each (block not in store).
5. Each witness is pushed into `waiting_for_block` without any size or count check.
6. With `K` large enough (e.g., K = 50 witnesses × 64 MiB each = 3.2 GiB), the validator process is killed by OOM.
7. The validator stops producing chunk endorsements; Spice consensus for its assigned shards stalls until the node is restarted.
8. The attacker can repeat the attack immediately after restart, maintaining the DoS indefinitely. [9](#0-8) [10](#0-9) [11](#0-10)

    fn process_waiting_for_block(
        &mut self,
        block: Arc<Block>,
        signer: Arc<ValidatorSigner>,
    ) -> Result<(), Error> {
        let block_hash = *block.header().hash();
        let Some((witnesses, contract_accesses)) = self.waiting_for_block.remove(&block_hash)
        else {
            return Ok(());
        };

        // Process deferred contract accesses first so that partial_chunk_data entries
        // have their `missing` set before witnesses try to finalize.
        for accesses in contract_accesses {
            self.handle_spice_contract_accesses(accesses)?;
        }

        let prev_hash = *block.header().prev_hash();
        let prev_block = self.chain_store.get_block(&prev_hash)?;
        if self.core_reader.get_block_execution_results(prev_block.header())?.is_none() {
            tracing::debug!(
                target: "spice_chunk_validator",
                ?prev_hash,
                "process_waiting_for_block: new block is available, but some of the prev block execution results are still missing");
            // Put witnesses back; contract accesses already went through
            // handle_spice_contract_accesses and are now in partial_chunk_data.
            self.waiting_for_block.entry(block_hash).or_default().0 = witnesses;
            return Ok(());
        };
