### Title
`batch_insert` Bypasses Duplicate-Hash Guard for All Items Beyond the Last Two — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` uses `.pop()` to bootstrap the first two items through `insert()` (which enforces `HashAlreadyPresent`), but all remaining items in the vector are written directly via `insert_entry_to_blob` with no duplicate-hash check. `add_leaf` uses `HashMap::insert`, which silently overwrites the existing `leaf_hash_to_index` entry. The result is two live leaf nodes sharing the same hash in the blob, a corrupted `leaf_hash_to_index` map, and an incorrect tree root hash.

### Finding Description

`batch_insert` has two distinct code paths:

**Path 1 — guarded (last two items in the vector):** [1](#0-0) 

Each item popped here goes through `insert()`, which explicitly rejects duplicate hashes: [2](#0-1) 

**Path 2 — unguarded (all remaining items):** [3](#0-2) 

No `contains_leaf_hash` check exists here. `insert_entry_to_blob` calls `add_leaf`: [4](#0-3) 

And `add_leaf` uses `HashMap::insert`, which silently overwrites: [5](#0-4) 

There is no error, no panic — the old `leaf_hash_to_index` entry for the duplicate hash is silently replaced with the new index.

### Impact Explanation

After a successful exploit:

1. **Two leaf nodes with the same hash exist in the blob.** The blob's byte content contains both, but `leaf_hash_to_index` only maps the hash to the last-written index.
2. **`get_node_by_hash` returns ambiguous/wrong results** — the first leaf (key3 in the PoC) is permanently unreachable by hash lookup. [6](#0-5) 
3. **The tree root hash is corrupted.** Internal node hashes are computed from their children's hashes; two leaves with identical hashes produce incorrect subtree hashes, making the committed Merkle root invalid.
4. **Merkle inclusion proofs are broken.** `get_proof_of_inclusion` for the orphaned leaf (key3) will either fail or produce a proof that does not verify against the corrupted root.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is exposed through the Python bindings: [7](#0-6) 

The `Hash` is an explicit caller-supplied parameter in the function signature: [8](#0-7) 

Any caller that controls the hash values — including through the Python/wasm DataLayer API — can supply a vector where an early item shares a hash with one of the last two items. No cryptographic collision is required; the attacker simply passes the same `Hash` value for two different `(KeyId, ValueId)` pairs.

### Recommendation

Add a duplicate-hash check inside the `for` loop in `batch_insert`, mirroring the guard already present in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing block construction and insert_entry_to_blob call
}
```

Alternatively, refactor `batch_insert` to route all items through `insert()` for the guard, or add the check inside `add_leaf` itself and return a `Result`.

### Proof of Concept

```rust
use chia_datalayer::{MerkleBlob, KeyId, ValueId, Hash};

let mut blob = MerkleBlob::new(vec![]).unwrap();
let hash_a = Hash::from([0xAA; 32]);
let hash_b = Hash::from([0xBB; 32]);

// Vector: [item0=(key1,hash_A), item1=(key2,hash_B), item2=(key3,hash_A)]
// pop() takes item2 first → insert(key3, hash_A) → OK, hash_A in cache
// pop() takes item1 next  → insert(key2, hash_B) → OK
// for loop processes item0 → insert_entry_to_blob(key1, hash_A) → NO CHECK
//   add_leaf silently overwrites leaf_hash_to_index[hash_A] = key1's index
let entries = vec![
    ((KeyId(1), ValueId(1)), hash_a),  // processed last, bypasses guard
    ((KeyId(2), ValueId(2)), hash_b),
    ((KeyId(3), ValueId(3)), hash_a),  // popped first, sets hash_A in cache
];
blob.batch_insert(entries).unwrap(); // succeeds silently — no error raised

// Now two leaves with hash_A exist; get_node_by_hash returns only key1's data
// key3 is permanently unreachable by hash; tree root is corrupted
```
