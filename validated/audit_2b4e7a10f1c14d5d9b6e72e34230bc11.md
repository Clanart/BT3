### Title
Missing Pause Check in `deploy_token` Allows Token Deployment When Solana Bridge Is Fully Paused - (File: `solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana `bridge_token_factory` program exposes a `pause()` instruction that is intended to halt all bridge operations in an emergency. However, the `deploy_token` instruction never checks the `paused` flag before executing. This means that even when the bridge is fully paused, any caller holding a valid MPC-signed `DeployTokenPayload` can still deploy new wrapped tokens on Solana, creating token mappings and posting Wormhole messages back to NEAR. The EVM and StarkNet bridge implementations both enforce a deploy-token-specific pause flag; the Solana program is missing this guard entirely.

---

### Finding Description

The Solana bridge program defines two pause-flag constants and enforces them on transfer operations: [1](#0-0) 

```
INIT_TRANSFER_PAUSED  = 1 << 0
FINALIZE_TRANSFER_PAUSED = 1 << 1
ALL_PAUSED = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED
```

There is **no** `DEPLOY_TOKEN_PAUSED` constant defined anywhere in the Solana program.

The `pause()` instruction sets `config.paused = ALL_PAUSED`: [2](#0-1) 

`finalize_transfer`, `finalize_transfer_sol`, `init_transfer`, and `init_transfer_sol` all correctly gate on the paused flag before proceeding: [3](#0-2) [4](#0-3) 

However, `deploy_token` performs **no pause check whatsoever**: [5](#0-4) 

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

By contrast, the EVM bridge guards `deployToken` with `whenNotPaused(PAUSED_DEPLOY_TOKEN)`: [6](#0-5) 

And the StarkNet bridge asserts `!_is_paused(@self, PAUSE_DEPLOY_TOKEN)` at the top of `deploy_token`: [7](#0-6) 

The Solana `deploy_token` instruction:
1. Creates a new wrapped SPL mint on Solana (a permanent on-chain state change).
2. Posts a Wormhole message back to NEAR, which NEAR's bridge will process to register the new token mapping.

Both of these effects occur even when `config.paused == ALL_PAUSED`. [8](#0-7) 

---

### Impact Explanation

**High — Unauthorized token deployment during an emergency pause.**

The pause mechanism is the protocol's primary emergency-stop control. When an admin pauses the Solana bridge (e.g., in response to a detected MPC key compromise or a critical bug), the intent is to halt all bridge operations. Because `deploy_token` bypasses the pause:

- A relayer or attacker holding a pre-signed `DeployTokenPayload` (obtained before the pause, or via a compromised MPC key — the exact scenario that motivates a pause) can still call `deploy_token` and create a new wrapped token on Solana.
- The resulting Wormhole message will propagate to NEAR and trigger token registration there, establishing a token mapping that the admin cannot prevent.
- If the MPC key is compromised (the most common reason to pause), the attacker can craft arbitrary `DeployTokenPayload` messages, sign them with the compromised key, and deploy fake or duplicate token mappings on Solana even while the bridge is paused. This can corrupt the token-address mapping and enable subsequent unbacked minting once the bridge is unpaused.

This fits the allowed impact: **High — Proof/MPC verification bypass enabling unauthorized token deployment or message execution**, and **High — Token-mapping corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

**High.** The pause mechanism is clearly intended to cover all user-facing bridge operations. The omission of a pause check in `deploy_token` is a straightforward implementation gap — the flag constant does not even exist in the Solana program, unlike the EVM and StarkNet implementations. Any emergency pause of the Solana bridge will silently fail to stop token deployment. The entry path is fully permissionless (any caller can submit a signed payload).

---

### Recommendation

1. Add a `DEPLOY_TOKEN_PAUSED` constant to `constants.rs`:

```rust
pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
```

2. Add a pause guard at the top of `deploy_token` in `lib.rs`, mirroring the pattern used for `finalize_transfer` and `init_transfer`:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
        error::ErrorCode::Paused
    );
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

3. Similarly add a pause check to `log_metadata` if token registration via native token logging should also be halted during an emergency.

---

### Proof of Concept

1. Admin calls `pause()` on the Solana bridge program (sets `config.paused = ALL_PAUSED = 0x03`).
2. Verify that `init_transfer` and `finalize_transfer` now revert with `ErrorCode::Paused`.
3. Obtain a valid MPC-signed `DeployTokenPayload` (e.g., one that was in-flight before the pause, or one signed by a compromised MPC key).
4. Call `deploy_token` with the signed payload.
5. Observe that the instruction succeeds: a new SPL mint is created on Solana and a Wormhole message is posted — despite the bridge being fully paused.
6. Confirm that the EVM bridge rejects the equivalent call with `"Pausable: paused"` when `PausedDeployToken` is set, demonstrating the inconsistency. [5](#0-4) [1](#0-0)
