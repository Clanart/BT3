Audit Report

## Title
Router Native ETH Balance Consumed by Any Caller to Satisfy WETH Payment Obligations — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay()` function in `PeripheryPayments` uses `address(this).balance` — the router's entire ambient ETH balance — to satisfy WETH payment obligations without attributing that ETH to the specific caller who deposited it. Because `multicall` is `payable`, any ETH sent with a multicall that is not fully consumed and not refunded via `refundETH` remains stranded on the router. A subsequent unprivileged caller specifying `tokenIn = WETH` can have their swap's `pay()` call consume that stranded ETH, receiving full swap output without spending any of their own funds.

## Finding Description
`PeripheryPayments.pay()` (L69–88) contains the following logic when `token == WETH` and `payer != address(this)`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never charged
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [1](#0-0) 

The `receive()` guard (L32–34) only blocks direct ETH transfers from non-WETH addresses, but `multicall` is `payable` (L39), so ETH sent with a `multicall{value: X}` call is held by the router without restriction. [2](#0-1) [3](#0-2) 

When `exactInputSingle` is called with `tokenIn = WETH`, it sets the callback payer to `msg.sender` via `_setNextCallbackContext` (L71), then the pool calls back into `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, attackerB, pool, value)`. [4](#0-3) [5](#0-4) 

If `address(this).balance >= value`, the first branch fires: the router deposits stranded ETH as WETH and transfers it to the pool. `safeTransferFrom(attackerB, ...)` is never called. The transient callback context correctly binds the payer, but `pay()` ignores that binding when native ETH is available — it treats the router's entire ETH balance as a shared, unattributed pool.

## Impact Explanation
Direct loss of user principal. User A loses up to their full stranded ETH (the portion of `msg.value` not consumed by their swap and not refunded). Attacker B receives the full swap output without spending any ETH or WETH. The loss is bounded only by the amount of ETH stranded on the router, which can be arbitrarily large. This meets the "Critical/High direct loss of user principal" threshold.

## Likelihood Explanation
Medium. The precondition — User A omitting `refundETH` from a native ETH multicall — is a common composition mistake and is not enforced against by the router. Once ETH is stranded, the exploit is trivially executable by any unprivileged caller in a subsequent transaction with no special permissions, no malicious setup, and no non-standard tokens.

## Recommendation
Do not use `address(this).balance` to satisfy a payment obligation attributed to an external `payer`. Track ETH consumed from `msg.value` explicitly in transient storage per top-level call, and only allow `pay()` to draw from that tracked amount. Alternatively, enforce that `address(this).balance == 0` at the end of each top-level swap entry point, or require `refundETH` to be the final step of any `multicall` that sends ETH.

## Proof of Concept

```
// Step 1: User A strands ETH on the router
// User A calls multicall{value: 1 ether} containing a single exactInputSingle
// that only needs 0.5 ether, and omits refundETH.
// After the call: address(router).balance == 0.5 ether

// Step 2: Attacker B calls exactInputSingle with tokenIn = WETH, amountIn = 0.5 ether
// Attacker B sends no ETH and holds no WETH / no approval.
// Pool calls metricOmmSwapCallback → _justPayCallback →
//   pay(WETH, attackerB, pool, 0.5 ether):
//     nativeBalance = 0.5 ether >= 0.5 ether  →  branch 1 fires
//     WETH.deposit{value: 0.5 ether}()         →  stranded ETH becomes WETH
//     WETH.transfer(pool, 0.5 ether)           →  pool receives payment
//     safeTransferFrom(attackerB, ...) is NEVER called
// Attacker B receives full token output.
// User A's 0.5 ETH is permanently lost.
```
