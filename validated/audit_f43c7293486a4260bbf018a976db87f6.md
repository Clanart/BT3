### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay of Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the transaction hash for `meta_tx_v0` syscall executions without including any nonce or replay-distinguishing field (`additional_data_size=0`). The OS also explicitly skips nonce enforcement and `__validate__` for version-0 transactions. Because the hash is a pure function of `(contract_address, selector, calldata, chain_id)`, any attacker who observes a valid `meta_tx_v0` signature can replay it an unlimited number of times by deploying a contract that issues the same syscall with the captured signature, causing the target account's `__execute__` to re-run the same authorized action (e.g., a fund transfer) without the account owner's consent.

---

### Finding Description

`compute_meta_tx_v0_hash` is defined as:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

Compare this to `compute_l1_handler_transaction_hash`, which correctly includes a nonce via `additional_data_size=1, additional_data=&nonce`: [2](#0-1) 

The OS then uses this nonce-free hash as the authoritative `transaction_hash` in the `TxInfo` presented to the target account contract's `__execute__`: [3](#0-2) 

Two additional OS-level safeguards are explicitly disabled for version-0 transactions:

**1. Nonce enforcement is skipped:**
```cairo
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [4](#0-3) 

**2. `__validate__` is skipped:**
```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
``` [5](#0-4) 

The `TxInfo` delivered to `__execute__` also hardcodes `nonce=0`: [6](#0-5) 

The result: the hash is identical for every replay of the same `(contract_address, selector, calldata, chain_id)` tuple, the OS enforces no nonce, and the account contract's `__execute__` receives a `TxInfo` that is indistinguishable from the original invocation.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker who observes a valid `meta_tx_v0` signature authorizing a fund transfer (e.g., an ERC-20 `transfer` wrapped in `__execute__` calldata) can replay it by deploying a contract that issues the `meta_tx_v0` syscall with the same `(contract_address, selector, calldata, signature)`. The target account's `__execute__` will compute the same `transaction_hash`, verify the same signature successfully, and execute the transfer again. This can be repeated indefinitely, draining the victim account.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no leaked keys, and no operator cooperation. Any unprivileged transaction sender can:
1. Observe a `meta_tx_v0` syscall in a past block (all calldata and signatures are public on-chain).
2. Deploy a contract that re-issues the syscall with the captured parameters.
3. Submit a transaction calling that contract.

There are no preconditions beyond the victim having previously used the `meta_tx_v0` syscall.

---

### Recommendation

Include a nonce (or a block-scoped unique identifier) in `compute_meta_tx_v0_hash` by passing it through `additional_data`, mirroring the pattern used in `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,           // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← include nonce in hash
    );
    return tx_hash;
}
```

The OS must also enforce nonce uniqueness for `meta_tx_v0` executions (e.g., by tracking used nonces in contract storage or by using the parent transaction's nonce combined with a per-call counter).

---

### Proof of Concept

1. **Victim** calls a contract `C` which issues `meta_tx_v0` targeting account `A` with calldata encoding `transfer(attacker, 100_STRK)` and a valid signature `σ` over `hash(INVOKE_PREFIX, 0, A, __execute__, calldata, 0, chain_id)`.

2. **Attacker** reads the syscall parameters from the block (public data): `(A, __execute__, calldata, σ)`.

3. **Attacker** deploys contract `Evil` whose only action is to call `meta_tx_v0(contract_address=A, selector=__execute__, calldata=calldata, signature=σ)`.

4. **Attacker** submits a transaction invoking `Evil`.

5. The OS computes `meta_tx_hash = compute_meta_tx_v0_hash(A, __execute__, calldata, chain_id)` — identical to step 1 because no nonce is included.

6. Account `A`'s `__execute__` receives `TxInfo{transaction_hash=meta_tx_hash, nonce=0, version=0, signature=σ}`, verifies `σ` against `meta_tx_hash` (valid), and executes `transfer(attacker, 100_STRK)` again.

7. Steps 3–6 can be repeated until `A`'s balance is exhausted.
