### Title
Wrong Shard Layout Used to Derive `shard_index` When Serving State Parts at Epoch Boundaries — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` resolves `shard_index` using the **new epoch's** shard layout, then uses that index to access `prev_block.chunks()`, which belongs to the **previous epoch** and is indexed by the **previous epoch's** shard layout. At a resharding boundary (V2/V3 layouts), the two layouts differ, so the wrong chunk header — and therefore the wrong `prev_state_root` — is selected to generate state parts. The companion function `compute_state_response_header` performs the same lookup correctly by explicitly fetching the previous epoch's layout.

---

### Finding Description

`get_state_response_part` in `chain/chain/src/state_sync/adapter.rs` resolves the shard index as follows:

```rust
// line 295
let epoch_id = block.header().epoch_id();          // sync block = first block of NEW epoch
// ...
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;  // NEW epoch layout
// ...
let prev_block = self.chain_store.get_block(header.prev_hash())?;   // last block of OLD epoch
let shard_index = shard_layout.get_shard_index(shard_id)?;          // index in NEW layout ← BUG
let state_root = prev_block
    .chunks()
    .get(shard_index)                                                // indexed by OLD layout
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

`compute_state_response_header`, which produces the committed `state_root` stored in the `ShardStateSyncResponseHeader`, does this correctly:

```rust
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;   // ← correct
let chunk_header = chunks.get(prev_shard_index)...;
``` [2](#0-1) 

In `ShardLayoutV0`/`V1`, `shard_id == shard_index`, so the two layouts agree and the bug is latent. In `ShardLayoutV2`/`V3` (used with dynamic resharding), `shard_id` and `shard_index` are decoupled: [3](#0-2) [4](#0-3) 

When a shard split occurs (e.g., old layout `[S0, S1, S2, S3]` → new layout `[S0, S1a, S1b, S2, S3]`):

- Shard `S2` has old index `2`, new index `3`.
- `get_state_response_part` computes `shard_index = 3` (new layout), then reads `prev_block.chunks()[3]` — which is `S3`'s chunk header, not `S2`'s.
- The `state_root` extracted is `S3`'s `prev_state_root`, not `S2`'s.
- The header (from `compute_state_response_header`) commits `S2`'s `prev_state_root`.

The `set_state_part` validation then compares the generated part against the committed state root:

```rust
let state_root = *chunk.take_header().take_inner().prev_state_root();
if matches!(
    self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part),
    StatePartValidationResult::Invalid
) { ... }
``` [5](#0-4) 

Because the part was generated against the wrong state root, validation fails and state sync is broken for every shard whose index shifts at the resharding boundary.

---

### Impact Explanation

**High.** Any node attempting to state-sync a shard whose `shard_index` changes across a resharding epoch boundary will receive state parts that fail `validate_state_part`. The node cannot complete state sync and cannot participate in the network for that shard. This is a broken reconstruction invariant: the `state_root` committed in the `ShardStateSyncResponseHeader` does not match the `state_root` used to generate the parts served by `get_state_response_part`.

---

### Likelihood Explanation

**Medium.** The bug is latent under `ShardLayoutV0`/`V1` (current mainnet, where `shard_id == shard_index`). It activates under `ShardLayoutV2`/`V3` with dynamic resharding, which is an actively developed and planned production feature. Every resharding event that shifts the index of at least one existing shard triggers the mismatch. No special attacker capability is required — any node requesting state parts at a resharding boundary will encounter the failure.

---

### Recommendation

In `get_state_response_part`, resolve `shard_index` using the **previous epoch's** shard layout (the epoch that owns `prev_block`), mirroring the pattern in `compute_state_response_header`:

```rust
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

---

### Proof of Concept

Consider a two-epoch scenario with `ShardLayoutV2`:

1. **Old epoch** (4 shards): `[S0(idx=0), S1(idx=1), S2(idx=2), S3(idx=3)]`
2. **New epoch** (5 shards, `S1` split): `[S0(idx=0), S1a(idx=1), S1b(idx=2), S2(idx=3), S3(idx=4)]`

A node requests `get_state_response_part(shard_id=S2_id, part_id=0, sync_hash=<first block of new epoch>)`:

- `epoch_id` = new epoch → `shard_layout` = new layout
- `shard_layout.get_shard_index(S2_id)` = **3** (new index)
- `prev_block.chunks().get(3)` = **S3's chunk header** (old layout index 3 = S3)
- `state_root` = S3's `prev_state_root` ← **wrong**

Meanwhile, `compute_state_response_header(shard_id=S2_id, sync_hash=...)`:

- `prev_shard_layout.get_shard_index(S2_id)` = **2** (old index)
- `sync_prev_block.chunks().get(2)` = **S2's chunk header**
- `state_root` = S2's `prev_state_root` ← **correct**, committed in header

`set_state_part` validates the part (generated for S3's root) against the header's committed root (S2's root) → `StatePartValidationResult::Invalid` → state sync fails. [6](#0-5) [7](#0-6)

    pub fn compute_state_response_header(
        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<ShardStateSyncResponseHeader, Error> {
        // Consistency rules:
        // 1. Everything prefixed with `sync_` indicates new epoch, for which we are syncing.
        // 1a. `sync_prev` means the last of the prev epoch.
        // 2. Empty prefix means the height where chunk was applied last time in the prev epoch.
        //    Let's call it `current`.
        // 2a. `prev_` means we're working with height before current.
        // 3. In inner loops we use all prefixes with no relation to the context described above.
        let sync_block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let sync_block_header = sync_block.header();
        let sync_block_epoch_id = sync_block_header.epoch_id();
        let shard_ids = self.epoch_manager.shard_ids(sync_block_epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }

        // The chunk was applied at height `chunk_header.height_included`.
        // Getting the `current` state.
        // TODO(current_epoch_state_sync): check that the sync block is what we would expect. So, either the first
        // block of an epoch, or the first block where there have been two new chunks in the epoch
        let sync_prev_block = self.chain_store.get_block(sync_block_header.prev_hash())?;

        let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
        let prev_epoch_id = sync_prev_block.header().epoch_id();
        let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
        let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;

        // Chunk header here is the same chunk header as at the `current` height.
        let sync_prev_hash = sync_prev_block.hash();
        let chunks = sync_prev_block.chunks();
        let chunk_header = chunks.get(prev_shard_index).ok_or(Error::InvalidShardId(shard_id))?;

    pub fn get_state_response_part(
        &mut self,
        shard_id: ShardId,
        part_id: u64,
        sync_hash: CryptoHash,
    ) -> Result<StatePart, Error> {
        let _span = tracing::debug_span!(
            target: "sync",
            "get_state_response_part",
            %shard_id,
            part_id,
            ?sync_hash)
        .entered();
        let block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let header = block.header();
        let epoch_id = block.header().epoch_id();
        // Check cache
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id)).unwrap();
        if let Some(bytes) = self.chain_store.store_ref().get(DBCol::StateParts, &key) {
            metrics::STATE_PART_CACHE_HIT.inc();
            let state_part = StatePart::from_bytes(bytes.to_vec())?;
            return Ok(state_part);
        }
        metrics::STATE_PART_CACHE_MISS.inc();

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_ids = self.epoch_manager.shard_ids(epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let prev_block = self.chain_store.get_block(header.prev_hash())?;
        let shard_index = shard_layout.get_shard_index(shard_id)?;
        let state_root = prev_block
            .chunks()
            .get(shard_index)
            .ok_or(Error::InvalidShardId(shard_id))?
            .prev_state_root();
        let prev_hash = *prev_block.hash();
        let prev_prev_hash = *prev_block.header().prev_hash();
        let state_root_node = self
            .runtime_adapter
            .get_state_root_node(shard_id, &prev_hash, &state_root)
            .log_storage_error("get_state_root_node fail")?;
        let num_parts = get_num_state_parts(state_root_node.memory_usage);
        if part_id >= num_parts {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let current_time = Instant::now();
        let state_part = self
            .runtime_adapter
            .obtain_state_part(
                shard_id,
                &prev_prev_hash,
                &state_root,
                PartId::new(part_id, num_parts),
            )
            .log_storage_error("obtain_state_part fail")?;

        let elapsed_ms = (self.clock.now().signed_duration_since(current_time))
            .whole_milliseconds()
            .max(0) as u128;
        self.requested_state_parts
            .save_state_part_elapsed(&sync_hash, &shard_id, &part_id, elapsed_ms);

        // Cache the part data, but only if the corresponding header is also cached.
        // At epoch boundaries, clear_all_downloaded_parts() deletes all cached headers
        // and parts. Since serving runs on a separate actor, a part request can arrive
        // after the clear and re-create a StatePartKey without its StateHeaderKey,
        // which the storage validator treats as an inconsistency.
        let header_key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
        if self.chain_store.store_ref().exists(DBCol::StateHeaders, &header_key) {
            let mut store_update = self.chain_store.store().store_update();
            let bytes = state_part.to_bytes();
            store_update.set(DBCol::StateParts, &key, &bytes);
            store_update.commit();
        }

        Ok(state_part)
    }

    pub fn get_shard_index(&self, shard_id: ShardId) -> Result<ShardIndex, ShardLayoutError> {
        match self {
            // In V0 & V1 the shard id and shard index are the same.
            Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
            // In V2 & V3 the shard id and shard index are **not** the same.
            Self::V2(v2) => v2.get_shard_index(shard_id),
            Self::V3(v3) => v3.get_shard_index(shard_id),
        }
    }
