### Title
`#[trusted_relayer]` Guard on `claim_fee` Permanently Locks Fees for Non-Relayer Fee Recipients - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The `claim_fee` function in the NEAR omni-bridge contract is gated by `#[trusted_relayer]`, which prevents any account that is not an active trusted relayer from calling it. However, `sign_transfer` allows a trusted relayer to designate **any** NEAR account as `fee_recipient` — including accounts that are not trusted relayers. When such a non-relayer fee_recipient attempts to call `claim_fee` to collect their earned fee, the `#[trusted_relayer]` guard rejects the call before the proof is even verified, permanently locking the fee tokens in the bridge.

### Finding Description

`sign_transfer` is a `#[trusted_relayer]`-gated function that accepts an arbitrary `fee_recipient: Option<AccountId>` parameter:

```rust
// near/omni-bridge/src/lib.rs:447-500
pub fn sign_transfer(
    &mut self,
    transfer_id: TransferId,
    fee_recipient: Option<AccountId>,
    fee: &Option<Fee>,
) -> Promise {
    ...
    let transfer_payload = TransferMessagePayload {
        ...
        fee_recipient,   // any NEAR account ID, no restriction
        ...
    };
``` [1](#0-0) 

This `fee_recipient` is embedded in the MPC-signed payload and eventually emitted as part of the `FinTransfer` event on the destination chain (EVM, Solana, StarkNet). When the fee_recipient later tries to collect their fee on NEAR, they must call `claim_fee`:

```rust
// near/omni-bridge/src/lib.rs:1054-1064
#[payable]
#[trusted_relayer]          // ← blocks any non-trusted-relayer caller
#[pause(except(roles(Role::DAO)))]
pub fn claim_fee(&mut self, #[serializer(borsh)] args: ClaimFeeArgs) -> Promise {
``` [2](#0-1) 

Inside `claim_fee_callback`, the contract correctly verifies the caller is the fee_recipient:

```rust
require!(
    fee_recipient == *predecessor_account_id,
    BridgeError::OnlyFeeRecipientCanClaim.as_ref()
);
``` [3](#0-2) 

But this inner check is never reached for a non-trusted-relayer fee_recipient — the `#[trusted_relayer]` macro rejects the call at the entry point. The `bypass_roles` configured for the macro are `Role::DAO` and `Role::UnrestrictedRelayer`:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedRelayer),
    manager_roles(Role::DAO, Role::RelayerManager),
    config_roles(Role::DAO)
)]
``` [4](#0-3) 

Even DAO cannot rescue the fee, because the `fee_recipient == predecessor_account_id` check inside the callback would still fail if DAO tried to call on behalf of the real fee_recipient.

The pending transfer (which holds the fee) remains in `pending_transfers` indefinitely with no admin recovery path, since `remove_transfer_message` is only called from within `claim_fee_callback`. [5](#0-4) 

### Impact Explanation

A trusted relayer can legitimately set `fee_recipient` to a non-trusted-relayer NEAR account — for example, a treasury contract, a multisig, a DAO account, or any other address. Once the transfer is finalized on the destination chain and the `FinTransfer` event is emitted with that fee_recipient, the fee tokens are permanently locked in the NEAR bridge contract. The fee_recipient cannot call `claim_fee`, and no other party can claim on their behalf. This constitutes an **irrecoverable lock of bridge fee funds**.

### Likelihood Explanation

The scenario is realistic and requires no malicious intent. A relayer may direct fees to a shared treasury, a smart contract, or a non-relayer operational account. The `fee_recipient` parameter is explicitly optional and unrestricted in `sign_transfer`. There is no documentation or on-chain enforcement preventing a trusted relayer from setting a non-relayer fee_recipient. Any such configuration permanently forfeits the fee.

### Recommendation

Remove the `#[trusted_relayer]` guard from `claim_fee`. The function already enforces that only the legitimate fee_recipient (as proven by the destination-chain event) can claim, via the `fee_recipient == predecessor_account_id` check inside `claim_fee_callback`. The proof verification is the correct and sufficient access control here. The `#[trusted_relayer]` guard is redundant and overly restrictive, directly analogous to the `(!rHolderStatus.isHolder)` check in the referenced external report.

### Proof of Concept

1. DAO or RelayerManager activates `treasury.near` as a non-relayer account (no staking, no trusted-relayer status).
2. A trusted relayer calls `sign_transfer(transfer_id, fee_recipient = Some("treasury.near"), fee = Some(...))`. The MPC signs the payload embedding `fee_recipient = "treasury.near"`.
3. A caller submits `finTransfer` on the EVM bridge. The EVM emits `FinTransfer(..., feeRecipient = "treasury.near")`.
4. `treasury.near` calls `claim_fee` on NEAR with a valid proof of the EVM `FinTransfer` event.
5. The `#[trusted_relayer]` macro checks `is_trusted_relayer("treasury.near")` → `false`. The call panics/reverts before reaching `claim_fee_callback`.
6. The pending transfer for `transfer_id` remains in `pending_transfers` with the fee locked. No path exists to recover it: `claim_fee` is the only function that calls `remove_transfer_message` for this transfer, and it is permanently inaccessible to `treasury.near`. [2](#0-1) [6](#0-5) [1](#0-0)

    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };
