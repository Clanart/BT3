### Title
`deployed_tokens_v2` State Erased on Contract Migration, Corrupting Token Origin-Chain Accounting - (File: `near/omni-bridge/src/migrate.rs`)

### Summary
The NEAR bridge's `migrate()` function discards the existing `deployed_tokens_v2` mapping and replaces it with a fresh empty `LookupMap`, even though the old state carries the same field. After migration, `get_token_origin_chain` falls back to a name-based heuristic that can panic for non-standard token names or silently return the wrong origin chain, breaking the `lock_tokens_if_needed` / `unlock_tokens_if_needed` invariant that underpins bridge collateralization.

### Finding Description

In `near/omni-bridge/src/migrate.rs`, the `migrate()` function reads `OldState` and constructs the new `Contract`. `OldState` contains `deployed_tokens_v2: LookupMap<AccountId, ChainKind>` — the authoritative mapping from every deployed token account ID to its origin chain. Instead of forwarding `old_state.deployed_tokens_v2`, the migration silently drops it and allocates a brand-new empty map: [1](#0-0) [2](#0-1) 

The critical line is:

```rust
deployed_tokens_v2: LookupMap::new(StorageKey::DeployedTokensV2),  // old_state.deployed_tokens_v2 silently dropped
```

After migration, every call to `get_token_origin_chain` misses the cache and falls through to a name-prefix heuristic: [3](#0-2) 

The heuristic has two failure modes:

1. **Panic** — if a token's account ID does not match any known prefix (e.g., a legacy token registered via `add_deployed_tokens()` with a non-standard name), the function hits `env::panic_str(&BridgeError::CannotDetermineOriginChain.as_ref())`.

2. **Silent wrong answer** — if a token's name prefix does not match its actual origin chain (e.g., a Solana-origin token whose account ID starts with `"eth"`), the heuristic returns the wrong `ChainKind`, which is then **cached back into `deployed_tokens_v2`** at line 1447, making the corruption permanent for the lifetime of that deployment.

Both failure modes propagate directly into the token-locking invariant: [4](#0-3) 

`lock_tokens_if_needed` and `unlock_tokens_if_needed` skip locking when `get_token_origin_chain(token) == chain_kind`. A wrong origin chain causes the bridge to skip locking on a non-origin chain (over-releasing collateral) or to lock on the origin chain (double-counting). A panic causes every `init_transfer` and `fin_transfer` for that token to revert.

### Impact Explanation

**Accounting corruption / bridge collateralization break (High):** If the heuristic returns the wrong origin chain for any token, `lock_tokens_if_needed` silently skips locking on the destination chain. Subsequent `fin_transfer` calls then call `unlock_tokens_if_needed` on a counter that was never incremented, either reverting (freezing the transfer) or, if the key happens to exist with a stale value, decrementing it below the true locked amount. This breaks the invariant that `locked_tokens[(chain, token)]` equals the total outstanding bridged supply on that chain, enabling unbacked minting or irrecoverable fund lock.

**Permanent freeze (High):** For any token whose account ID does not match a known prefix, every bridge operation that touches `get_token_origin_chain` panics after migration. Pending `TransferMessage` entries stored in `pending_transfers` for such tokens become unclaimable — the relayer cannot call `sign_transfer` or `fin_transfer` without triggering the panic, permanently freezing user funds until the DAO manually re-registers the token via `add_deployed_tokens()`.

### Likelihood Explanation

This is **guaranteed to occur on every execution of `migrate()`** as long as `deployed_tokens_v2` is non-empty before migration — exactly the same structural guarantee as the LivenessGuard report. The `deployed_tokens_v2` map is populated by every call to `deploy_token_internal`, `add_deployed_tokens`, and `migrate_deployed_token`, so any production deployment with bridged tokens will have a non-empty map. The silent-wrong-answer path is triggered whenever a token's account ID prefix does not exactly match its registered origin chain, which is possible for any token registered through the admin `add_deployed_tokens` path with a non-standard name.

### Recommendation

Carry the existing mapping forward instead of allocating a new one:

```rust
// In migrate()
deployed_tokens_v2: old_state.deployed_tokens_v2,  // was: LookupMap::new(StorageKey::DeployedTokensV2)
```

This is the direct analog of the LivenessGuard recommendation: initialize from the previous state rather than resetting it.

### Proof of Concept

1. Before migration, the bridge has token `"legacy-wrapped.deployer.near"` registered via `add_deployed_tokens()` with `origin_chain = ChainKind::Sol`. Its entry exists in `deployed_tokens_v2`.

2. DAO upgrades the contract and calls `migrate()`. `deployed_tokens_v2` is reset to empty; `deployed_tokens` still contains `"legacy-wrapped.deployer.near"`.

3. A user calls `ft_on_transfer` with `"legacy-wrapped.deployer.near"` to initiate a transfer to Ethereum. `init_transfer_internal` calls `lock_tokens_if_needed(ChainKind::Eth, "legacy-wrapped.deployer.near", amount)`.

4. `lock_tokens_if_needed` calls `get_token_origin_chain("legacy-wrapped.deployer.near")`. The `deployed_tokens_v2` cache is empty. The token is in `deployed_tokens`. The name does not start with any known prefix → `env::panic_str("CannotDetermineOriginChain")`.

5. The `ft_on_transfer` call reverts. The user's tokens are returned, but any already-stored `TransferMessage` for this token (from a prior `fin_transfer` path) is now unclaimable, freezing those funds until the DAO intervenes.

    pub fn get_token_origin_chain(&mut self, token: &AccountId) -> ChainKind {
        if let Some(origin_chain) = self.deployed_tokens_v2.get(token) {
            return origin_chain;
        }

        if let Some(origin_chain) = self.get_utxo_chain_by_token(token) {
            return origin_chain;
        }

        if !self.deployed_tokens.contains(token) {
            return ChainKind::Near;
        }

        let origin_chain = match token.as_str() {
            s if s.starts_with("eth")
                || s.contains("factory.bridge.near")
                || s.contains("factory.sepolia.testnet") =>
            {
                ChainKind::Eth
            }
            s if s.starts_with("base") => ChainKind::Base,
            s if s.starts_with("arb") => ChainKind::Arb,
            s if s.starts_with("bnb") => ChainKind::Bnb,
            s if s.starts_with("pol") => ChainKind::Pol,
            s if s.starts_with("hlevm") => ChainKind::HyperEvm,
            s if s.starts_with("abs") => ChainKind::Abs,
            s if s.starts_with("sol") => ChainKind::Sol,
            s if s.starts_with("fogo") => ChainKind::Fogo,
            s if s.starts_with("strk") || s.starts_with("starknet") => ChainKind::Strk,
            _ => env::panic_str(&BridgeError::CannotDetermineOriginChain.as_ref()),
        };

        if !origin_chain.is_utxo_chain() {
            self.deployed_tokens_v2.insert(token, &origin_chain);
        }

        origin_chain
    }
