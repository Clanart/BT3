### Title
StorageBalance `total` Not Updated When Crediting Message Account Funds — Permanent Lock of User NEAR Deposits - (File: near/omni-bridge/src/storage.rs)

### Summary

In `try_to_transfer_balance_from_message_account`, when the balance of a pre-funded message account is transferred to the storage payer, only `storage.available` is incremented. The paired field `storage.total` is never updated. This breaks the invariant `total >= available` that the rest of the storage accounting system relies on, causing the credited NEAR tokens to become permanently unwithdrawable — a direct analog to the "Disparity In Rewards Update" class where two coupled accounting fields diverge because one update site is missed.

---

### Finding Description

`StorageBalance` has two fields that must always move in tandem:

- `total` — the gross NEAR deposited by the account (used as the ceiling in `storage_withdraw`)
- `available` — the portion not yet consumed by storage reservations

Every other write site in the contract updates both fields together. `storage_deposit` (lines 159–160) does:

```rust
storage.total = storage.total.saturating_add(amount);
storage.available = storage.available.saturating_add(amount);
```

`storage_withdraw` (lines 194–205) subtracts from both:

```rust
storage.total = storage.total.checked_sub(to_withdraw)…;
storage.available = storage.available.checked_sub(to_withdraw)…;
```

`try_to_transfer_balance_from_message_account` (lines 260–290) is the sole exception. It credits the message account's full balance to the storage payer's `available` but leaves `total` unchanged:

```rust
storage.available = storage.available.saturating_add(balance.total);
// storage.total is never touched
self.accounts_balances.insert(storage_payer, &storage);
self.accounts_balances.remove(account_id);
``` [1](#0-0) 

After this call, `storage.available > storage.total` by exactly `balance.total` (the amount that was in the message account).

---

### Impact Explanation

`storage_withdraw` defaults `to_withdraw` to `storage.available` when no explicit amount is given:

```rust
let to_withdraw = amount.unwrap_or(storage.available);
storage.total = storage.total.checked_sub(to_withdraw).near_expect(…);
``` [2](#0-1) 

Because `storage.total < storage.available` after the buggy credit, any attempt to withdraw the full available balance panics at the `checked_sub` on `total`. The user can only withdraw up to the original `total`, never recovering the NEAR that was transferred from the message account. The message account itself is deleted (`accounts_balances.remove(account_id)`), so those tokens are permanently locked inside the bridge contract with no recovery path.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows.**

---

### Likelihood Explanation

The message-account mechanism is the documented path for users who pre-deposit native fees before calling `init_transfer`. The function is called in two reachable, unprivileged code paths:

1. `init_transfer` (direct path, lines 566–573 in `lib.rs`) — triggered by any `ft_on_transfer` call with an `InitTransfer` message when the message account has sufficient balance.
2. `init_transfer_resume` (yield-resume path, lines 635–643 in `lib.rs`) — triggered when the yield callback fires after storage becomes available. [3](#0-2) [4](#0-3) 

Every user who pre-funds a message account and successfully completes an `init_transfer` via this path will have their deposited NEAR permanently locked. No privileged access is required.

---

### Recommendation

Update `try_to_transfer_balance_from_message_account` to increment `storage.total` alongside `storage.available`, mirroring every other write site in the contract:

```rust
storage.total = storage.total.saturating_add(balance.total);
storage.available = storage.available.saturating_add(balance.total);
``` [5](#0-4) 

---

### Proof of Concept

1. Alice calls `storage_deposit` with `account_id = message_account_id` and attaches 1 NEAR. The message account now has `{ total: 1 NEAR, available: ~0.998 NEAR }`.
2. Alice calls `ft_on_transfer` with an `InitTransfer` message referencing `message_storage_account_id`. The bridge calls `try_to_transfer_balance_from_message_account(message_account_id, native_fee, alice, required_storage)`.
3. Inside the function: `storage.available += 1 NEAR` (message account total). `storage.total` is unchanged. The message account entry is deleted.
4. Alice's `StorageBalance` is now `{ total: T, available: T + 1 NEAR }` where `T` is her prior total.
5. Alice calls `storage_withdraw(None)` — defaults to withdrawing `available = T + 1 NEAR`. The line `storage.total.checked_sub(T + 1 NEAR)` underflows and panics. The 1 NEAR is permanently locked.
6. Alice can only successfully call `storage_withdraw(Some(T))` to recover her original deposit, never the 1 NEAR from the message account. [1](#0-0) [6](#0-5)

    pub(crate) fn try_to_transfer_balance_from_message_account(
        &mut self,
        account_id: &AccountId,
        native_fee: NearToken,
        storage_payer: &AccountId,
        required_storage_payer_balance: NearToken,
    ) -> Result<(), StorageError> {
        let balance = self
            .accounts_balances
            .get(account_id)
            .ok_or(StorageError::MessageAccountNotRegistered)?;

        if balance.total < native_fee {
            return Err(StorageError::NotEnoughBalanceForFee);
        }

        let mut storage = self
            .accounts_balances
            .get(storage_payer)
            .ok_or(StorageError::SignerNotRegistered)?;

        storage.available = storage.available.saturating_add(balance.total);

        if storage.available < required_storage_payer_balance.saturating_add(native_fee) {
            return Err(StorageError::SignerNotEnoughBalance);
        }

        self.accounts_balances.insert(storage_payer, &storage);
        self.accounts_balances.remove(account_id);
        Ok(())
    }
