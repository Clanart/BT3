### Title
Migration v5 Uses `store.Has()` as Boolean Value, Incorrectly Enabling `AllowUnprotectedTxs` and Bypassing EIP-155 Replay Protection — (File: x/evm/migrations/v5/migrate.go)

---

### Summary

The v5 store migration reads three boolean EVM parameters by calling `store.Has(key)` instead of reading and unmarshaling the actual stored bytes. Because the legacy params module persists every parameter key regardless of its `true`/`false` value, `store.Has()` always returns `true` for any key that was ever written. After the migration runs, `AllowUnprotectedTxs` is unconditionally set to `true` even when the chain had it explicitly set to `false`, silently disabling EIP-155 replay protection for all subsequent blocks.

---

### Finding Description

In `x/evm/migrations/v5/migrate.go`, lines 34–36:

```go
params.EnableCreate        = store.Has(v0types.ParamStoreKeyEnableCreate)
params.EnableCall          = store.Has(v0types.ParamStoreKeyEnableCall)
params.AllowUnprotectedTxs = store.Has(v0types.ParamStoreKeyAllowUnprotectedTxs)
```

The Cosmos SDK legacy params subspace stores every registered parameter as a marshaled byte value under its key, including boolean `false` (encoded as `0x00`). The key is present in the store whether the value is `true` or `false`. `store.Has(key)` therefore returns `true` for both cases — it only tests key existence, not the actual value.

The correct approach (used for the non-boolean params in the same function) is to call `store.Get(key)` and unmarshal the result:

```go
// correct pattern used for chainConfig and extraEIPs
chainCfgBz := store.Get(v0types.ParamStoreKeyChainConfig)
cdc.MustUnmarshal(chainCfgBz, &chainConfig)
```

For the boolean fields the migration should do the equivalent: read the raw bytes and decode them. Instead it substitutes a key-existence test, which is semantically wrong.

The default value of `AllowUnprotectedTxs` is `false` (`DefaultAllowUnprotectedTxs = false` in `x/evm/types/params.go`). On any chain that initialised params at genesis (which calls `SetParams` → `store.Set(key, marshaledFalse)`), the key is present. After the v5 migration the field is written as `true`, permanently overriding the governance-set value.

---

### Impact Explanation

`AllowUnprotectedTxs` gates the ante-handler check in `ante/evm/` that rejects non-EIP-155 transactions (transactions without a chain-ID in their signature). When it is `true`, the ante handler accepts legacy pre-EIP-155 signatures. This means:

1. An attacker can submit a transaction signed without a chain ID on this chain.
2. The same raw transaction is valid on every other EVM chain that accepts unprotected transactions (e.g., Ethereum Classic, testnets, other Cosmos EVM chains).
3. Cross-chain replay is therefore possible: a transaction legitimately signed on chain A can be replayed on this chain, draining the signer's funds without their knowledge.

This matches the allowed High impact: *"Ethereum transaction, EIP-155 authorization, chain-id, or signer verification bypass enabling replay."*

Additionally, if `EnableCreate` or `EnableCall` were governance-disabled (`false`) before the upgrade, they are silently re-enabled, allowing contract creation or calls that governance explicitly prohibited.

---

### Likelihood Explanation

- The v5 migration runs automatically for every node during the chain upgrade; no attacker action is required to trigger it.
- The default `AllowUnprotectedTxs = false` means virtually every production chain is affected: the key is written at genesis, so `store.Has()` returns `true`.
- After the upgrade completes, any unprivileged user can immediately submit non-EIP-155 transactions; no special privilege is needed.
- The window is permanent (until a corrective governance proposal re-sets the param to `false`), and the chain may not notice because the migration produces no error.

---

### Recommendation

Replace the three `store.Has()` calls with proper byte-read-and-unmarshal logic, mirroring the pattern already used for `chainConfig` and `extraEIPs`:

```go
// Instead of:
params.EnableCreate = store.Has(v0types.ParamStoreKeyEnableCreate)

// Use:
if bz := store.Get(v0types.ParamStoreKeyEnableCreate); len(bz) > 0 {
    var b bool
    // amino bool: 0x00 = false, 0x01 = true
    params.EnableCreate = bz[0] != 0
}
```

Apply the same fix to `EnableCall` and `AllowUnprotectedTxs`.

---

### Proof of Concept

1. Chain genesis sets `AllowUnprotectedTxs = false` (the default). The legacy params subspace writes key `AllowUnprotectedTxs → 0x00` to the KV store.
2. Chain upgrade triggers `MigrateStore` in `x/evm/migrations/v5/migrate.go`.
3. Line 36 executes: `params.AllowUnprotectedTxs = store.Has(v0types.ParamStoreKeyAllowUnprotectedTxs)`. The key exists → returns `true`.
4. The migrated params are serialised and written to `types.KeyPrefixParams` with `AllowUnprotectedTxs = true`.
5. From the next block onward, `GetParams` returns `AllowUnprotectedTxs = true`.
6. The ante handler's EIP-155 check is skipped for non-EIP-155 transactions.
7. Attacker broadcasts a pre-EIP-155 transaction (signed without chain ID) that was originally created on Ethereum mainnet. The transaction is accepted, and the original signer's funds on this chain are transferred without their consent. [1](#0-0) [2](#0-1) [3](#0-2)
