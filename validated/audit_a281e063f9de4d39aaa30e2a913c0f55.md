### Title
Unconstrained `is_reverted` Hint Allows Sequencer to Bypass Invoke Transaction Execution While Charging Fees — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted` in `execution_constraints.cairo` is a complete no-op. It accepts the `is_reverted` hint value but performs zero validation, leaving it entirely unconstrained in the Cairo proof. A malicious sequencer can set `is_reverted = TRUE` for any invoke transaction, skip its `__execute__` entry point, still charge fees, and produce a proof that the verifier accepts as valid. This is the direct structural analog of the nucypher bug: a verification function that silently succeeds regardless of input, followed by a caller that trusts its output.

---

### Finding Description

**Vulnerability class:** Crypto verification flaw / state-transition bypass — unconstrained witness value in the proof system.

In `execute_invoke_function_transaction`, the `is_reverted` value is loaded from a hint and then passed to `check_is_reverted`:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // stack alignment only, no execution
    ...
}
// Charge fee — unconditional.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

`check_is_reverted` is defined as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

In Cairo's proof system, hint values (`%{ ... %}`) are provided by the prover (the sequencer) and are **not** constrained by the verifier unless an explicit assertion is made. `check_is_reverted` adds no assertion whatsoever. Therefore `is_reverted` is a free witness: the sequencer can assign it any value and the resulting proof remains valid.

The structural parallel to the nucypher report is exact:

| nucypher | StarkNet OS |
|---|---|
| `verify_from` returns decrypted data without verifying the signature | `check_is_reverted` returns without checking `is_reverted` |
| `set_policy` proceeds with unverified data, corrupting state | `execute_invoke_function_transaction` branches on unconstrained value, skipping execution |
| `attach_kfrag_to_saved_arrangement` has no idempotency guard | `charge_fee` is called unconditionally regardless of the unconstrained branch |

---

### Impact Explanation

**Critical — Permanent freezing of funds / Direct loss of funds.**

When a sequencer sets `is_reverted = TRUE` for an invoke transaction:

1. The `__execute__` entry point is **not** called — no state changes from the transaction are recorded in `contract_state_changes`.
2. `charge_fee` is called **unconditionally** after the branch, deducting fees from the user's account.
3. The Cairo proof is valid because `check_is_reverted` adds no constraint.
4. The L1 verifier accepts the proof.

The user's transaction never executes. Their funds are drained via fees. The state committed on-chain reflects a world where the transaction was reverted, even though the sequencer accepted it. This is permanent and unrecoverable: the nonce is incremented by `check_and_increment_nonce` (called before the branch), so the transaction cannot be replayed. [3](#0-2) 

The reverse direction is equally dangerous: setting `is_reverted = FALSE` for a transaction that the sequencer's off-chain execution determined should revert forces execution of a transaction that should have failed, potentially draining contract balances.

---

### Likelihood Explanation

The sequencer is the entity that supplies all hint values to the Cairo prover. No additional privilege, key leak, or external dependency is required. The sequencer already controls `%{ IsReverted %}` as part of normal operation. The only barrier is intent. The StarkNet OS proof exists precisely to constrain the sequencer's behavior; this bug removes that constraint for the revert flag.

---

### Recommendation

`check_is_reverted` must be implemented to actually constrain `is_reverted`. The correct approach is to derive `is_reverted` from a verifiable source — for example, by checking whether the execution trace produced a revert marker in the output segment, or by asserting that `is_reverted` is boolean (`assert is_reverted * (1 - is_reverted) = 0`) and then tying it to the actual execution outcome recorded in the state diff. The function's location in `execution_constraints.cairo` and its name make clear it was intended to enforce a constraint; the body must be filled in.

---

### Proof of Concept

1. User submits an invoke transaction (e.g., a token transfer).
2. Sequencer receives the transaction, increments the nonce off-chain, and sets `%{ IsReverted %}` to return `1` (TRUE) in the hint.
3. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called; no state changes from the transfer are written to `contract_state_changes`.
5. `charge_fee` executes unconditionally, deducting fees from the user's account via an ERC-20 transfer.
6. `check_and_increment_nonce` has already incremented the nonce, so the transaction cannot be resubmitted.
7. The OS generates a proof over this trace. The verifier accepts it because no constraint was violated.
8. The block is finalized on L1. The user's transfer never happened; their fee is gone; the nonce is consumed. [2](#0-1) [4](#0-3) [5](#0-4)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
