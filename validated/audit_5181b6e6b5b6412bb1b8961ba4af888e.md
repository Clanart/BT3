### Title
`ProofOfInclusion::valid()` Final Root Check Is Always True, Enabling Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` contains a final check `existing_hash == self.root_hash()` that is **always true** and therefore dead code. The method only verifies internal chain consistency of the proof, never against any external trusted root hash. An attacker can construct a `ProofOfInclusion` with arbitrary `node_hash` and `layers` that passes `valid()` while proving membership in a completely fabricated tree root, not the actual committed DataLayer root.

---

### Finding Description

The `valid()` method in `ProofOfInclusion` is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // ← set to calculated_hash
    }

    existing_hash == self.root_hash()     // ← always true
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same value as existing_hash after loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**Why the final check is always true:**

In the last loop iteration, the code:
1. Computes `calculated_hash`
2. Asserts `calculated_hash == layer.combined_hash` (returns `false` if not)
3. Sets `existing_hash = calculated_hash`

After the loop, `existing_hash` equals the last `calculated_hash`, which equals the last `layer.combined_hash` (guaranteed by the in-loop check). `root_hash()` returns exactly `last.combined_hash`. Therefore `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash`, which is unconditionally `true`.

**The analog to the original bug:** In the Symmetrical report, the solvency check used pre-adjustment `lockedValues` (the wrong reference) instead of post-adjustment values. Here, `valid()` uses the proof's own internal `combined_hash` as the root reference (the wrong reference) instead of an external, trusted committed root. In both cases, the validation passes against a value that the attacker controls or that does not reflect the true committed state.

**Forged proof construction:** An attacker can trivially construct a passing proof:

```
node_hash  = H(fake_key || fake_value)
layers     = [ { other_hash_side: Right,
                 other_hash:      H(anything),
                 combined_hash:   internal_hash(node_hash, H(anything)) } ]
```

- Loop check: `calculated_hash == combined_hash` ✓ (by construction)
- Final check: `existing_hash == root_hash()` ✓ (always true)
- `proof.root_hash()` returns the attacker-chosen `combined_hash`

The proof passes `valid()` and claims membership in a root the attacker chose, not the actual DataLayer tree root.

The Python binding exposes both `valid()` and `root_hash()` as separate methods: [3](#0-2) 

If a caller uses only `proof.valid()` without separately asserting `proof.root_hash() == known_committed_root`, the forged proof is accepted as genuine.

---

### Impact Explanation

Matches: **High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

A forged `ProofOfInclusion` that passes `valid()` can be used to assert that an arbitrary key-value pair is present in the DataLayer when it is not. Any system that gates actions on `proof.valid()` alone — without separately checking the root against the on-chain committed root — will accept fabricated state as genuine DataLayer state.

---

### Likelihood Explanation

The Python binding exposes `valid()` as a standalone method named to suggest completeness. The method's name and signature give no indication that a separate root-hash comparison is required. Any Python caller that writes `if not proof.valid(): raise ...` without also checking `proof.root_hash() == committed_root` is vulnerable. The attacker-controlled entry path is any interface that accepts a serialized `ProofOfInclusion` from an untrusted source and validates it with `valid()`.

---

### Recommendation

The `valid()` method must accept the expected root hash as a parameter and compare against it, rather than deriving the root from the proof itself:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against external trusted root
}
```

The existing `valid()` method (which only checks internal consistency) should either be removed or clearly documented as insufficient for security-critical verification. All callers must be updated to supply the committed root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};
use chia_datalayer::merkle::blob::internal_hash;

// Attacker-chosen hashes
let fake_node_hash = Hash(/* H(fake_key || fake_value) */);
let fake_other    = Hash(/* arbitrary */);
let fake_combined = internal_hash(&fake_node_hash, &fake_other); // right-side

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash:      fake_other,
        combined_hash:   fake_combined,
    }],
};

assert!(forged_proof.valid());           // passes — always true
assert_eq!(forged_proof.root_hash(), fake_combined); // attacker-controlled root
// The proof "proves" fake_node_hash is in a tree rooted at fake_combined,
// which has nothing to do with the actual committed DataLayer root.
