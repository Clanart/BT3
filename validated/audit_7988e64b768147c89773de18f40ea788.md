### Title
Native ETH Permanently Frozen in `OmniBridge` When Recipient Contract Cannot Accept ETH — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` delivers native ETH to the recipient via a low-level `.call{value: amount}("")`. If the recipient is a contract that cannot accept ETH (no `receive`/`fallback`, or one that reverts on receipt), every `finTransfer` attempt for that payload reverts with `FailedToSendEther`. Because the recipient address is immutably encoded in the MPC-signed payload and there is no admin rescue path for ETH held by the bridge, the ETH locked in the contract from the corresponding `initTransfer` is permanently irrecoverable.

---

### Finding Description

In `OmniBridge.finTransfer`, the native-ETH branch is:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The nonce guard is set before the transfer attempt:

```solidity
completedTransfers[payload.destinationNonce] = true;
``` [2](#0-1) 

When `FailedToSendEther` is thrown the entire transaction reverts, so `completedTransfers` is also rolled back. The nonce is therefore never consumed, and the relayer can retry indefinitely — but every retry will fail identically if the recipient is a non-ETH-accepting contract.

The ETH balance that `finTransfer` draws from was deposited into the bridge by earlier `initTransfer` calls:

```solidity
if (tokenAddress == address(0)) {
    if (fee != 0) { revert InvalidFee(); }
    extensionValue = msg.value - amount - nativeFee;
}
``` [3](#0-2) 

The contract has a bare `receive() external payable {}` to accept ETH, but no corresponding admin withdrawal or rescue function exists anywhere in the contract. [4](#0-3) 

A grep for `rescue`, `withdrawEth`, `withdrawNative`, and `recoverEth` across all EVM contracts returns zero results, confirming no built-in recovery path exists.

On the NEAR side, once a user's tokens are burned/locked by `init_transfer_internal`, there is no `cancel_transfer` or equivalent function to reverse the outbound leg: [5](#0-4) 

The recipient address is baked into the MPC-signed `TransferMessagePayload` and cannot be changed after signing: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who initiates a NEAR→EVM (or EVM→EVM via NEAR relay) transfer of native ETH and specifies a contract recipient that cannot accept ETH will:

1. Have their source-side tokens permanently burned/locked (no cancel path on NEAR).
2. Cause the destination-side ETH to be permanently frozen inside the `OmniBridge` contract, because `finTransfer` will revert on every attempt and there is no admin rescue function.

The ETH is irrecoverable without a contract upgrade, which is an out-of-band privileged action not guaranteed to occur.

---

### Likelihood Explanation

**Medium.** The scenario is reachable by any unprivileged user who:

- Specifies a smart-contract wallet, multisig, or protocol contract as the EVM recipient that lacks a `receive()` function or whose `receive()` reverts (e.g., access-controlled, paused, or self-destructed contracts).
- Makes a typo and provides a contract address instead of an EOA.

No special role, leaked key, or colluding party is required. The user only needs to call `ft_transfer_call` (NEAR side) or `initTransfer` (EVM side) with a problematic recipient address.

---

### Recommendation

1. **Wrap-and-store pattern**: If the low-level ETH call fails, store the amount in a per-recipient claimable mapping instead of reverting, allowing the recipient (or an admin) to pull the ETH later.
2. **Admin rescue function**: Add a role-gated `rescueEth(address to, uint256 amount)` function so that ETH stuck due to non-accepting recipients can be recovered and re-routed.
3. **Recipient validation**: Consider restricting `tokenAddress == address(0)` recipients to EOAs only (check `recipient.code.length == 0`), or document clearly that contract recipients must implement `receive()`.

---

### Proof of Concept

1. User calls `initTransfer(address(0), 1 ether, 0, 0, "near:recipient.near", "")` on EVM, sending 1 ETH. ETH is locked in `OmniBridge`.
2. NEAR processes the proof; MPC signs a `TransferMessagePayload` with `recipient = <ContractNoReceive>` (a contract with no `receive()` function).
3. Relayer calls `finTransfer(sig, payload)` on EVM.
4. Execution reaches line 319: `(bool success,) = ContractNoReceive.call{value: 1 ether}("")` → `success == false`.
5. `revert FailedToSendEther()` is thrown; `completedTransfers[nonce]` is rolled back.
6. Every subsequent retry by any relayer produces the same revert.
7. The 1 ETH remains in `OmniBridge` with no admin function to extract it. The user's source-side tokens are already burned/locked with no cancel path. [1](#0-0) [7](#0-6)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
