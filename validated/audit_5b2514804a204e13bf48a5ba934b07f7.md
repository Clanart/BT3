### Title
`ProofOfInclusion::valid()` Does Not Verify Root Against Committed Tree Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` in the DataLayer crate checks only the internal hash-chain consistency of the proof, deriving the root entirely from the proof's own fields. It never accepts or compares against an externally committed/known root. Any caller that relies solely on `proof.valid()` to authenticate a DataLayer inclusion proof can be deceived by a crafted, internally-consistent proof with a fabricated root and fabricated leaf claim.

### Finding Description

`ProofOfInclusion` is the DataLayer's proof-of-inclusion type, exposed to Python via `pyclass(get_all, from_py_object)` and `Streamable`, meaning it can be deserialized from untrusted bytes or constructed directly from Python.

The `root_hash()` method derives the root entirely from the proof's own last layer: [1](#0-0) 

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- comes from the proof itself
    } else {
        self.node_hash
    }
}
```

The `valid()` method then walks the hash chain and compares the final computed hash against `self.root_hash()` — which is also derived from the proof: [2](#0-1) 

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(...);
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()  // compares against self-derived root
}
```

There is no parameter for a trusted external root. `valid()` returns `true` for any proof whose internal hash chain is self-consistent, regardless of whether the root matches the actual committed on-chain DataLayer store root.

The Python binding exposes both `valid()` and `root_hash()` as separate methods: [3](#0-2) 

The type is also `Streamable` and carries `from_py_object`, making it fully constructible from untrusted input: [4](#0-3) 

All existing callers — the Rust tests, the Python tests, and the fuzz target — call `proof.valid()` without separately asserting `proof.root_hash() == known_committed_root`: [5](#0-4) [6](#0-5) 

### Impact Explanation

An attacker who can supply a `ProofOfInclusion` to any code path that calls `proof.valid()` without also checking `proof.root_hash() == committed_root` can:

1. Claim any arbitrary key-value pair is present in a DataLayer store.
2. Construct a `ProofOfInclusion` with a fabricated `node_hash` (the claimed leaf) and a single layer whose `combined_hash` is computed as `internal_hash(node_hash, attacker_chosen_sibling)`.
3. `valid()` returns `true` because the chain is internally consistent.
4. The client accepts the forged inclusion proof.

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `ProofOfInclusion` type is exposed to Python via `pyclass(get_all, from_py_object)` and `Streamable`, so it can be deserialized from arbitrary bytes or constructed directly in Python. The method name `valid()` strongly implies complete proof validation, making it likely that downstream Python consumers of the DataLayer API will call only `proof.valid()` and omit the separate `proof.root_hash() == committed_root` check. The Linea ENS bug was exploited in exactly this pattern — a proof was verified for internal consistency but not anchored to the committed state.

### Recommendation

1. Add a `valid_for_root(expected_root: &Hash) -> bool` method (or modify `valid()` to accept the committed root as a mandatory parameter) that compares the computed root against the caller-supplied committed root before returning `true`.
2. Deprecate or remove the zero-argument `valid()` from the public Python API, or rename it to `internally_consistent()` to make clear it does not authenticate against any committed state.
3. All callers — including the fuzz target and Python tests — should be updated to pass the known committed root.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
from chia_rs.sized_bytes import bytes32
import hashlib

# Attacker wants to forge a proof that fake_leaf is in the store
fake_leaf = bytes32(b'\xaa' * 32)
sibling   = bytes32(b'\xbb' * 32)

# Compute a fake combined_hash that is internally consistent
combined = bytes32(hashlib.sha256(b'\x02' + fake_leaf + sibling).digest())

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=sibling,
    combined_hash=combined,
)
proof = ProofOfInclusion(node_hash=fake_leaf, layers=[layer])

# valid() returns True even though this root was never committed on-chain
assert proof.valid(), "Forged proof passes validation"
# proof.root_hash() == combined  (attacker-controlled, not the real store root)
```

`proof.valid()` returns `True` for this entirely fabricated proof because `valid()` never compares against any externally committed root. [2](#0-1)
