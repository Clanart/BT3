### Title
Missing Zero-Address Check for `nearBridgeDerivedAddress_` in `OmniBridge.initialize` Permanently Freezes Bridge Finalization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initialize` accepts `nearBridgeDerivedAddress_` without any zero-address validation. This address is the sole authorization gate for `finTransfer` and `deployToken`. If it is set to `address(0)` at deployment, every subsequent call to those functions permanently reverts, making all user funds that were locked or burned on NEAR/Solana/StarkNet irrecoverable on EVM.

### Finding Description

`OmniBridge.initialize` stores three caller-supplied addresses without validation: [1](#0-0) 

`nearBridgeDerivedAddress_` is assigned directly to `nearBridgeDerivedAddress` with no zero-address guard. This field is then used as the only authorization check in both `finTransfer` and `deployToken`: [2](#0-1) [3](#0-2) 

The project uses OpenZeppelin v5.4.0: [4](#0-3) 

In OZ v5.x, `ECDSA.recover` reverts on malformed signatures and returns a non-zero address for any valid signature. Therefore, if `nearBridgeDerivedAddress == address(0)`, the comparison `ECDSA.recover(hashed, signatureData) != address(0)` is always `true` for any valid signature, causing every `finTransfer` and `deployToken` call to revert with `InvalidSignature`. There is no recovery path: `setNearBridgeDerivedAddress` is `onlyRole(DEFAULT_ADMIN_ROLE)`, but if `_adminAddress` was also set to `address(0)` (or `_msgSender()` was the zero address), the admin role is unrecoverable and the contract is permanently bricked. [5](#0-4) 

The same pattern exists in `OmniBridgeWormhole.initializeWormhole` for `wormholeAddress` (no zero-address check), and in `ENearProxy.initialize` for `_eNear`, `_prover`, and `_adminAddress`: [6](#0-5) [7](#0-6) 

The CLAUDE.md security invariant explicitly states: *"No token release without signature: Never mint, transfer, or unlock tokens to a recipient without first verifying a valid MPC signature … it is the only authorization gate for finTransfer."* A zero `nearBridgeDerivedAddress` makes this gate permanently closed. [8](#0-7) 

### Impact Explanation

If `nearBridgeDerivedAddress_` is initialized to `address(0)`:

- `finTransfer` always reverts → tokens locked or burned on NEAR, Solana, or StarkNet can never be claimed on EVM. All in-flight user funds are permanently frozen.
- `deployToken` always reverts → no bridged token contracts can be deployed, blocking the entire bridge bootstrapping.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

The `initialize` function is `public initializer` and is called exactly once at deployment time by the deployer. A deployment script error, copy-paste mistake, or misconfigured environment variable passing `address(0)` for `nearBridgeDerivedAddress_` would silently succeed (no revert at initialization) and leave the bridge permanently non-functional. The external report's referenced finding (Linea TokenBridge) had the identical likelihood profile and was accepted as a valid issue. The absence of a guard means there is no safety net for this class of human error.

### Recommendation

Add explicit zero-address checks at the top of `OmniBridge.initialize` (and analogously in `OmniBridgeWormhole.initializeWormhole` and `ENearProxy.initialize`):

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    require(tokenImplementationAddress_ != address(0), "Zero tokenImplementation");
    require(nearBridgeDerivedAddress_ != address(0), "Zero nearBridgeDerivedAddress");
    // ... rest of init
}
```

Similarly guard `wormholeAddress` in `initializeWormhole`, and `_eNear`, `_prover`, `_adminAddress` in `ENearProxy.initialize`.

### Proof of Concept

1. Deploy `OmniBridge` proxy calling `initialize(validImpl, address(0), chainId)`. Transaction succeeds; `nearBridgeDerivedAddress` is now `address(0)`.
2. A user on NEAR locks 1000 USDC and initiates a cross-chain transfer to EVM.
3. Relayer calls `finTransfer(validSignature, payload)`.
4. `ECDSA.recover(hashed, validSignature)` returns the legitimate MPC signer address (non-zero).
5. `legitimateSigner != address(0)` → `true` → `revert InvalidSignature()`.
6. The nonce is marked used at line 287 **before** the signature check at line 311: [9](#0-8) 

Wait — actually the nonce IS marked used before the signature check (line 287 before line 311), so the nonce is consumed but the transfer never completes. The user's funds are locked on NEAR with no corresponding release on EVM, and the nonce is burned, making the transfer permanently unclaimable even if the address were later corrected via `setNearBridgeDerivedAddress` (because `completedTransfers[payload.destinationNonce]` is already `true`).
