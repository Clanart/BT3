### Title
`L1Provider::start_block` Does Not Check for `CatchingUp` State, Allowing Block Proposal/Validation with Incomplete L1 Transaction Records — (`crates/apollo_l1_provider/src/l1_provider.rs`)

---

### Summary

`L1Provider::start_block` guards only against `Uninitialized` state. It does not guard against `CatchingUp` state. When the provider is mid-sync and its `current_height` happens to equal the height the batcher supplies, `start_block` succeeds, transitions the provider to `Propose` or `Validate`, and the batcher proceeds to call `get_txs` or `validate` against a `TransactionManager` whose records are incomplete. L1 handler transactions that were already consumed or cancelled in blocks the provider has not yet synced can be proposed into a new block, and valid pending transactions can be silently omitted.

---

### Finding Description

`L1Provider` maintains a `ProviderState` enum with five variants: `Uninitialized`, `CatchingUp`, `Pending`, `Propose`, and `Validate`. [1](#0-0) 

`CatchingUp` is entered when `commit_block` receives a height that does not match `current_height`; the provider spawns a background sync task that advances `current_height` by replaying blocks from L2 state sync, and the `tx_manager` is updated incrementally as each synced block is applied. [2](#0-1) 

Every other state-sensitive method correctly handles `CatchingUp`. `commit_block` explicitly branches on `is_catching_up()` and routes to `accept_commit_while_catching_up`: [3](#0-2) 

`get_txs` and `validate` enforce their required states (`Propose` / `Validate`) via a `match` that returns `UnexpectedProviderState` for anything else, including `CatchingUp`. [4](#0-3) 

`start_block`, however, only checks `is_uninitialized()` and then `check_height_with_error`. There is no `is_catching_up()` guard: [5](#0-4) 

During catching up the background sync task advances `current_height` one block at a time via `apply_commit_block`. At any intermediate height H where `current_height == H`, a concurrent call to `start_block(H, Propose)` or `start_block(H, Validate)` passes both guards and writes `self.state = ProviderState::Propose` (or `Validate`), overwriting `CatchingUp`. The provider is now in a proposing/validating session backed by a `tx_manager` that has not yet processed all blocks up to the sync target.

The batcher's `validate_block` path explicitly ignores errors from `start_block` and proceeds regardless: [6](#0-5) 

This means even a soft error path does not protect the downstream `get_txs` / `validate` calls from operating on stale state.

---

### Impact Explanation

The `tx_manager` records which L1 handler transactions are `Pending` (proposable), `Committed`, `Consumed`, or `Cancelled`. During catching up, blocks between `start_height` and the sync target have not yet been applied, so:

- Transactions consumed or cancelled in those unsynced blocks still appear as `Pending` in the `tx_manager`.
- `get_txs` returns them as valid candidates for the new block proposal.
- `validate` returns `Validated` for them instead of `ConsumedOnL1` / `CancelledOnL2`.

An L1 handler transaction that was already consumed on L1 (i.e., the corresponding L1→L2 message was already processed) gets re-included in a new block. This constitutes a double-execution of an L1 message: the L1 contract's consumed-message accounting is violated, and the resulting L2 state root diverges from the correct one. Conversely, transactions that should be excluded (because they were cancelled during the unsynced window) are included, and transactions that should be included may be missing if the sync has not yet surfaced them.

**Matching impact scope:** *High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing* (the L1 provider is the admission gate for L1 handler transactions; it admits stale/invalid ones and rejects valid ones when `start_block` bypasses the `CatchingUp` guard).

---

### Likelihood Explanation

`CatchingUp` is entered on every node restart and on any height skew between the batcher and the provider. The sync task advances `current_height` one block at a time, so for every block in the catch-up range there is a window during which `current_height` equals the batcher's next proposal height. The batcher calls `start_block` as part of its normal `propose_block` / `validate_block` flow without any back-off for provider state. No existing test covers `start_block` called while in `CatchingUp` state.

---

### Recommendation

Add an `is_catching_up()` guard at the top of `start_block`, mirroring the pattern used in `commit_block`:

```rust
pub fn start_block(
    &mut self,
    height: BlockNumber,
    state: SessionState,
) -> L1ProviderResult<()> {
    if self.state.is_uninitialized() {
        return Err(L1ProviderError::Uninitialized);
    }
    if self.state.is_catching_up() {
        return Err(L1ProviderError::UnexpectedProviderState {
            expected: ProviderState::Pending,
            found: self.state,
        });
    }
    self.check_height_with_error(height)?;
    self.state = state.into();
    self.tx_manager.start_block();
    Ok(())
}
```

Add a corresponding test asserting that `start_block` returns `UnexpectedProviderState` when the provider is in `CatchingUp` state, analogous to the existing `pending_state_returns_error` test. [7](#0-6) 

---

### Proof of Concept

1. Provider initializes at `start_height = H`. State: `Pending`, `current_height = H`.
2. Batcher calls `commit_block(height = H+5)`. Height mismatch → provider calls `start_catching_up(H+5)`. State: `CatchingUp`, sync task spawned targeting `H+5`.
3. Sync task applies blocks H, H+1, H+2 via `apply_commit_block`. `current_height` advances to `H+3`. Blocks H through H+2 contained L1 handler tx `T` which was consumed on L1 at block H+1; `tx_manager` has not yet processed that block.
4. Batcher calls `start_block(height = H+3, state = Propose)`.
   - `is_uninitialized()` → false.
   - `check_height_with_error(H+3)` → OK (`current_height == H+3`).
   - `self.state = ProviderState::Propose`. **`CatchingUp` is silently overwritten.**
5. Batcher calls `get_txs(n, H+3)`. Provider is now in `Propose` state; `tx_manager` returns tx `T` as `Pending` because block H+1 (which consumed it) has not been applied yet.
6. Tx `T` is included in the new block proposal. The L1 message is double-executed. The resulting L2 state root is wrong. [5](#0-4) [3](#0-2) [2](#0-1)

pub enum ProviderState {

    Propose,

}
