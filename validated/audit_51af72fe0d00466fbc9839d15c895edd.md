### Title
`finTransfer` Permits Bridge Contract as Recipient, Permanently Locking User Funds and Corrupting Token Supply Accounting — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol`'s `finTransfer` function contains no guard preventing `payload.recipient` from equalling `address(this)`. When a user initiates a cross-chain transfer on NEAR and specifies the EVM OmniBridge contract address as the destination recipient, the MPC key signs the payload as-is, the relayer submits it to EVM, and the tokens or ETH are delivered to the bridge contract itself. Because the destination nonce is consumed before the transfer executes, the transfer is permanently finalised with the funds irrecoverably locked inside the bridge. For bridge-token flows this additionally mints unbacked supply on EVM while the NEAR-side burns the corresponding tokens, breaking bridge collateralisation.

---

### Finding Description

In `finTransfer`, the destination nonce is marked used unconditionally before any asset movement: [1](#0-0) 

Immediately after, assets are dispatched to `payload.recipient` with no check that the recipient is not the bridge contract itself:

**ETH path** — the bridge has an unconditional `receive()` fallback, so the call succeeds and ETH stays in the contract: [2](#0-1) 

**Non-bridge ERC-20 path** — tokens are transferred back into the bridge's own balance: [3](#0-2) 

**Bridge-token path** — new tokens are minted directly to the bridge contract: [4](#0-3) 

The `receive()` fallback that makes the ETH path silently succeed: [5](#0-4) 

On the NEAR side, `init_transfer` only rejects recipients whose chain is `ChainKind::Near`; an EVM address that happens to be the OmniBridge contract passes validation: [6](#0-5) 

The MPC signer signs whatever `TransferMessagePayload` the NEAR bridge constructs, including a recipient equal to the OmniBridge EVM address. No layer in the pipeline rejects this.

For the bridge-token case the accounting corruption is concrete: `IBridgeToken.mint(address(this), amount)` increases EVM total supply, while on NEAR the corresponding tokens were already burned or unlocked. The bridge now carries unbacked EVM supply with no NEAR collateral behind it.

---

### Impact Explanation

- **Permanent freezing of user funds (Critical):** The destination nonce is consumed and `completedTransfers[nonce] = true` is set before the transfer executes. There is no admin recovery path for a user whose funds were sent to the bridge contract. The user's NEAR-side tokens are burned/locked and the EVM-side tokens are irrecoverable.
- **Bridge collateralisation corruption (High):** For bridge tokens, minting to `address(this)` inflates EVM token supply without any backing on NEAR. This breaks the 1:1 peg invariant the bridge relies on.

---

### Likelihood Explanation

Any user can trigger this by specifying the OmniBridge EVM contract address as the `recipient` field in `InitTransferMsg` when calling `ft_transfer_call` on NEAR. The MPC key signs the payload without inspecting the recipient. A malicious frontend or a simple user mistake is sufficient. No privileged role is required.

---

### Recommendation

Add a recipient sanity check at the top of `finTransfer`, before the nonce is consumed:

```solidity
require(
    payload.recipient != address(this),
    "Recipient cannot be the bridge contract"
);
```

Optionally, enforce the same constraint on the NEAR side inside `init_transfer` by rejecting any recipient whose serialised address matches a known bridge contract address, though the EVM-side guard is the critical fix.

---

### Proof of Concept

1. Attacker (or unwitting user) calls `ft_transfer_call` on a NEAR ERC-20-bridged token contract, passing an `InitTransferMsg` whose `recipient` field is the EVM OmniBridge contract address (e.g. `Eth(0xABCD…OmniBridge)`).
2. NEAR bridge's `init_transfer` accepts it — the only chain check is `recipient.get_chain() != ChainKind::Near`, which passes for an EVM address. [6](#0-5) 
3. A relayer calls `sign_transfer`; the MPC key signs the `TransferMessagePayload` including `recipient = OmniBridge address`.
4. Relayer calls `finTransfer` on EVM with the valid MPC signature.
5. Signature verification passes at line 311. [7](#0-6) 
6. `completedTransfers[payload.destinationNonce] = true` — nonce permanently consumed. [8](#0-7) 
7. For a bridge token: `IBridgeToken(payload.tokenAddress).mint(address(this), payload.amount)` executes successfully, minting tokens to the bridge contract. [9](#0-8) 
8. Tokens are permanently locked inside the bridge. The user's NEAR-side tokens were burned. EVM total supply is inflated with no NEAR backing. No replay is possible because the nonce is consumed.
