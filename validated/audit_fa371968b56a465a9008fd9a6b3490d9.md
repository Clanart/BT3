### Title
Tokens Permanently Locked When `normalize_amount` Returns Zero in `sign_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR→EVM bridge transfer with an amount smaller than the decimal normalization divisor, the tokens are locked/burned during `init_transfer` but `sign_transfer` permanently panics with `InvalidAmountToTransfer`, leaving the funds irrecoverably locked with no cancel path.

### Finding Description

The bridge uses `normalize_amount` to convert a NEAR-side token amount to the destination chain's decimal precision via floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

For a NEAR token with `origin_decimals = 24` and a bridged EVM token with `decimals = 18`, the divisor is `10^6`. Any transfer where `amount_without_fee() < 10^6` will produce `normalize_amount(...) == 0`.

In `sign_transfer`, this zero result triggers a hard panic:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

However, the tokens were already locked or burned in a **prior transaction** during `init_transfer`. The `init_transfer` function only validates `fee < amount` — it does not check whether `normalize_amount(amount - fee) > 0`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [3](#0-2) 

Once the transfer message is stored in `pending_transfers` and the tokens are locked, `sign_transfer` will **always** panic for this transfer. There is no cancel or refund mechanism — `remove_transfer_message` is only reachable via `sign_transfer_callback` (which requires a successful MPC signing) or `claim_fee_callback` (which requires a finalized destination-chain proof). Neither path is reachable when `sign_transfer` itself panics before reaching the MPC call.

### Impact Explanation

User funds are permanently frozen in the NEAR bridge contract. The transfer message occupies storage in `pending_transfers` indefinitely and the locked/burned tokens can never be recovered. This matches the **Critical** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

### Likelihood Explanation

Any unprivileged user who calls `ft_transfer_call` with an amount below the decimal normalization threshold triggers this condition. For a 24-decimal NEAR token bridging to an 18-decimal EVM token, the threshold is `10^6` base units — a realistic dust amount. The user receives no warning at deposit time; the failure only surfaces later when a relayer attempts `sign_transfer`.

### Recommendation

Add a pre-check in `init_transfer` (or its internal helper) that validates `normalize_amount(amount - fee, decimals) > 0` **before** locking or burning tokens. If the normalized amount would be zero, revert immediately and return the tokens to the sender. This mirrors the Napier fix: reject the deposit at entry rather than failing mid-flow after funds are committed.

```rust
// In init_transfer, after computing transfer_message:
let decimals = self.token_decimals.get(&token_address)...;
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee()...,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

### Proof of Concept

1. Token `foo.near` is registered with `origin_decimals = 24`, bridged EVM token has `decimals = 18` (divisor = `10^6`).
2. User calls `ft_transfer_call` with `amount = 500_000` (< `10^6`) and `fee = 0`. The `fee < amount` check passes. Tokens are locked in the bridge. Transfer message stored in `pending_transfers`.
3. Relayer calls `sign_transfer` for this transfer. `normalize_amount(500_000, {24, 18}) = 500_000 / 1_000_000 = 0`. The `require!(amount_to_transfer > 0, ...)` panics.
4. No MPC signing occurs. `sign_transfer_callback` is never reached. Transfer message remains in `pending_transfers` forever.
5. User's 500,000 base units of `foo.near` are permanently locked with no recovery path. [4](#0-3) [5](#0-4) [3](#0-2)
