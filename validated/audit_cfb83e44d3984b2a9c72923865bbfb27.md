### Title
Silent ERC1155 Transfer to Non-Contract Address Enables Unauthorized Token Minting on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` calls `IERC1155(tokenAddress).safeTransferFrom(...)` without verifying that `tokenAddress` is a deployed contract. In Solidity, a high-level call to a void-returning function on an address with no code silently succeeds (the EVM CALL returns `success=true` with empty returndata; since there is no return value to ABI-decode, no revert occurs). Combined with the permissionless `logMetadata1155` function — which also performs no contract existence check — an attacker can register a fake ERC1155 token, trigger NEAR-side token deployment, then fake an arbitrary-amount transfer, causing NEAR to mint unbacked bridged tokens.

---

### Finding Description

**Root cause — `initTransfer1155`:**

```solidity
// OmniBridge.sol L458-464
IERC1155(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    tokenId,
    amount,
    ""
);
``` [1](#0-0) 

`IERC1155` is a bare interface — there is no `SafeERC1155` equivalent in OpenZeppelin that adds a contract-existence check. When `tokenAddress` has no deployed code, the EVM CALL opcode returns `(success=true, returndata=empty)`. Because `safeTransferFrom` is declared `void`, Solidity does not attempt to decode any return data and does not revert. The call silently succeeds.

**Root cause — `logMetadata1155`:**

```solidity
// OmniBridge.sol L234-270
function logMetadata1155(address tokenAddress, uint256 tokenId) external payable {
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
    MultiTokenInfo storage multiToken = multiTokens[deterministicToken];
    if (multiToken.tokenAddress == address(0)) {
        multiToken.tokenAddress = tokenAddress;
        multiToken.tokenId = tokenId;
    }
    ...
    emit BridgeTypes.LogMetadata(deterministicToken, Strings.toHexString(tokenAddress), "", 0);
}
``` [2](#0-1) 

`logMetadata1155` makes no calls to `tokenAddress` whatsoever — it only stores the address and emits a `LogMetadata` event. It succeeds for any address, including non-contracts.

**NEAR-side token registration:**

The NEAR bridge's `deploy_token_callback` accepts a `LogMetadata` proof from any registered factory (the OmniBridge contract is a registered factory) and deploys a new NEP-141 token for the `deterministicToken` address:

```rust
// near/omni-bridge/src/lib.rs L1159-1174
let chain = metadata.emitter_address.get_chain();
require!(
    self.factories.get(&chain) == Some(metadata.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
self.deploy_token_internal(chain, &metadata.token_address, ...)
``` [3](#0-2) 

**NEAR-side transfer finalization:**

`fin_transfer_callback` mints tokens to the recipient once the factory check passes and the token is registered:

```rust
// near/omni-bridge/src/lib.rs L708-713
require!(
    self.factories
        .get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
``` [4](#0-3) 

---

### Impact Explanation

An attacker can mint an arbitrary quantity of bridged NEP-141 tokens on NEAR without locking any real ERC1155 tokens on EVM. This directly breaks bridge collateralization: the NEAR-side supply is unbacked. The minted tokens can be transferred or used within the NEAR ecosystem, constituting unauthorized mint of bridged assets.

**Allowed impact matched:** *Critical — Direct theft, unauthorized release, or unauthorized mint of native or bridged assets.*

---

### Likelihood Explanation

- Both `logMetadata1155` and `initTransfer1155` are fully permissionless — no role, whitelist, or approval is required.
- The attacker only needs to choose any address with no deployed code (e.g., a fresh EOA address, or a CREATE2 address before deployment).
- The only external dependency is a relayer submitting the two proofs to NEAR, which is the normal bridge operation and can be done by the attacker themselves if they run a relayer node.
- No privileged access, leaked keys, or colluding parties are required.

---

### Recommendation

Add a contract existence check in `initTransfer1155` before the ERC1155 transfer:

```solidity
if (tokenAddress.code.length == 0) {
    revert NotAContract();
}
```

Optionally, add the same check in `logMetadata1155` to prevent registration of non-contract addresses from the start.

For defense-in-depth, consider using a `SafeERC1155` wrapper (analogous to OpenZeppelin's `SafeERC20`) that verifies `address(token).code.length > 0` before any transfer call.

---

### Proof of Concept

1. Attacker selects `nonContractAddress` — any address with no deployed code (e.g., `address(0xdead...)`).

2. Attacker calls:
   ```solidity
   bridge.logMetadata1155(nonContractAddress, tokenId);
   // Succeeds: no call to nonContractAddress, populates multiTokens[deterministicToken],
   // emits LogMetadata(deterministicToken, "0xdead...", "", 0)
   ``` [2](#0-1) 

3. Relayer (or attacker) submits the `LogMetadata` EVM proof to NEAR `deploy_token`. NEAR verifies the factory, deploys a new NEP-141 token for `deterministicToken`. [5](#0-4) 

4. Attacker calls:
   ```solidity
   bridge.initTransfer1155(
       nonContractAddress,
       tokenId,
       1_000_000e18,   // arbitrary large amount
       0,              // fee
       0,              // nativeFee
       "attacker.near",
       ""
   );
   // IERC1155(nonContractAddress).safeTransferFrom(...) silently succeeds
   // (no code at address → EVM CALL returns success=true, empty returndata,
   //  void return type → no ABI decode → no revert)
   // Emits InitTransfer(deterministicToken, 1_000_000e18, "attacker.near")
   ``` [6](#0-5) 

5. Relayer submits the `InitTransfer` EVM proof to NEAR `fin_transfer`. NEAR verifies the factory, looks up `deterministicToken` (registered in step 3), and mints `1_000_000e18` NEP-141 tokens to `attacker.near` — with zero real ERC1155 tokens ever locked. [7](#0-6)
