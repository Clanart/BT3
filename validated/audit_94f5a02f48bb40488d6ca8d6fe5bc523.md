Audit Report

## Title
Unguarded `refundETH()` Allows Any Caller to Steal ETH Stranded by Direct Payable Swap Calls — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` sends the router's entire ETH balance to `msg.sender` with no access control. When a user calls `exactOutputSingle` (or any payable swap function) directly with excess `msg.value`, the `pay()` function wraps only the exact required amount as WETH, leaving the remainder as raw ETH in the router. Any subsequent caller of `refundETH()` can steal this stranded ETH.

## Finding Description
The root cause is a combination of two behaviors. First, in `pay()` (`PeripheryPayments.sol` L73-78), when `token == WETH` and `nativeBalance >= value`, only `value` wei is wrapped via `IWETH9.deposit{value: value}()` and forwarded to the pool — the surplus `msg.value - value` remains as raw ETH in the router after the swap function returns. Second, `refundETH()` (`PeripheryPayments.sol` L58-63) unconditionally transfers `address(this).balance` to `msg.sender` with no per-user accounting, no access control, and no reentrancy guard.

The `receive()` guard (`PeripheryPayments.sol` L32-34) is not a mitigation: it only reverts plain ETH transfers (direct `send`/`transfer`/`call{value: x}("")`). ETH sent as `msg.value` through a `payable` function call like `exactOutputSingle{value: 2 ether}(...)` never triggers `receive()`.

The intended safe pattern — `multicall{value}([swap, refundETH()])` — is atomic and returns excess ETH to the original caller. But `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` are all `external payable` and callable directly. A user calling `exactOutputSingle{value: 2 ether}(amountOut: 1000)` directly will have `2 ether - actualAmountIn` stranded in the router after the transaction. An attacker then calls `refundETH()` in any subsequent transaction to drain it.

## Impact Explanation
Direct loss of user ETH principal. For `exactOutputSingle`, users routinely send a buffer above the expected input because the exact input is unknown before execution. The stolen amount equals `msg.value - actualAmountIn`, which can be arbitrarily large. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
Medium. The swap functions are `external payable` and the direct-call pattern is natural, especially for `exactOutputSingle` where the exact input is unknown. A mempool-watching attacker needs only to observe a direct payable swap call and call `refundETH()` in the next block. No special privileges or setup are required.

## Recommendation
Add an automatic ETH refund at the end of each payable swap function: after the swap completes, if `address(this).balance > 0`, transfer it back to `msg.sender`. Alternatively, restrict `refundETH()` to only be callable via `delegatecall` from `multicall` by checking `address(this) == implementation`. The simplest fix is to add `if (address(this).balance > 0) _transferETH(msg.sender, address(this).balance);` at the end of `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Proof of Concept
```solidity
function test_refundETH_stealsStrandedETH() public {
    address userA = makeAddr("userA");
    address attacker = makeAddr("attacker");
    vm.deal(userA, 2 ether);

    // userA calls exactOutputSingle directly with 2 ETH buffer;
    // actual amountIn is pool-determined (much less than 2 ETH).
    vm.prank(userA);
    router.exactOutputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactOutputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountOut: 1_000,
            amountInMaximum: 2 ether,
            recipient: userA,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // 2 ETH - actualAmountIn is now stranded in the router.
    assertGt(address(router).balance, 0, "ETH stranded");

    // Attacker steals it in a subsequent transaction.
    uint256 before = attacker.balance;
    vm.prank(attacker);
    router.refundETH();
    assertGt(attacker.balance, before, "attacker stole userA's ETH");
    assertEq(address(router).balance, 0, "router drained");
}
```
