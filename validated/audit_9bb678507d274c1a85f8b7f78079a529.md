### Title
Asymmetric Base in `within_margin` Allows Proposer to Inflate L1 Gas Price Beyond Intended Tolerance — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `within_margin` function used to validate a proposer's L1 gas price against the local oracle computes the allowed deviation band using the **proposed** price as the base, not the **oracle reference** price. When the proposed price exceeds the oracle price, the band widens proportionally, permitting a proposer to embed an L1 gas price that deviates more than `l1_gas_price_margin_percent` from the oracle value. The inflated price is then committed into the block header and used for all fee calculations in that block.

---

### Finding Description

`is_block_info_valid` in `validate_proposal.rs` calls `within_margin` to check that the proposer's `l1_gas_price_fri` and `l1_data_gas_price_fri` are close enough to the validator's own oracle reading:

```rust
within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
``` [1](#0-0) 

The function is:

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← base is number1 (proposed)
    number1.0.abs_diff(number2.0) <= margin
}
``` [2](#0-1) 

`number1` is the **proposed** price; `number2` is the **oracle reference** price. The margin is `number1 * percent / 100`, so when `proposed > oracle`, the margin is computed from the larger value, making the band wider than intended — exactly the same asymmetry as the Olympus `_isDeviating` bug.

**Concrete example** (`margin_percent = 10`):

| Scenario | Proposed | Oracle | Margin computed | `abs_diff` | Passes? | True deviation from oracle |
|---|---|---|---|---|---|---|
| Proposed = oracle + 10 % | 1100 | 1000 | 1100 × 10 % = **110** | 100 | ✓ | 10 % |
| Proposed = oracle + 11.1 % | 1111 | 1000 | 1111 × 10 % = **111.1** | 111 | ✓ | **11.1 %** |
| Proposed = oracle − 10 % | 900 | 1000 | 900 × 10 % = **90** | 100 | ✗ | 10 % (incorrectly rejected) |

The upward direction is more permissive than intended; the downward direction is more restrictive than intended.

The oracle-derived reference price is obtained via `get_l1_prices_in_fri_and_wei`, which applies `apply_fee_transformations` (tip, multiplier, min/max clamp) before the comparison. [3](#0-2) 

The validated `l1_gas_price_fri` is then passed directly into `convert_to_sn_api_block_info` and used as the authoritative `l1_gas_price` for the block's `GasPrices`, which drives every fee calculation executed by the blockifier for that block. [4](#0-3) 

---

### Impact Explanation

The L1 gas price embedded in the block header is the price at which L1 gas is charged to every transaction in the block. An inflated `l1_gas_price_fri` causes the blockifier to charge users more STRK than the oracle-sanctioned price warrants. Because the block is committed to consensus and stored, the wrong price becomes the authoritative on-chain value for that block, affecting receipts, fee refunds, and any downstream proof or sync consumer that reads the block's gas price.

This matches: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

Any validator that wins a proposal round can exploit this. In a decentralized validator set, the proposer role rotates, so no special privilege beyond being a validator is required. The deviation is bounded by the margin percentage (e.g., ~1 % extra above the intended 10 % ceiling), but it is systematic and repeatable every block the attacker proposes.

---

### Recommendation

Compute the margin from the **oracle reference** price (`number2`), not the proposed price (`number1`):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number2.0 * margin_percent) / 100;  // base = oracle reference
    number1.0.abs_diff(number2.0) <= margin
}
```

This makes the allowed band symmetric around the oracle value: `oracle × (1 ± margin_percent/100)`, regardless of which direction the proposed price deviates.

---

### Proof of Concept

Using the existing unit-test harness in `validate_proposal_test.rs`:

```rust
// Demonstrates the asymmetric over-permissiveness upward
assert!(within_margin(GasPrice(1111), GasPrice(1000), 10));
// |1111 - 1000| = 111, margin = 1111 * 10 / 100 = 111 → passes
// But true deviation = (1111 - 1000) / 1000 = 11.1 % > 10 %

// Demonstrates the asymmetric over-restrictiveness downward
assert!(!within_margin(GasPrice(900), GasPrice(1000), 10));
// |900 - 1000| = 100, margin = 900 * 10 / 100 = 90 → fails
// But true deviation = (1000 - 900) / 1000 = 10 % = exactly the limit
``` [5](#0-4) 

The existing test cases `big_number_in_margin(1000, 1050, 10, true)` and `big_number_out_of_margin(1000, 1150, 10, false)` only test the case where `proposed < oracle`, which is the more-restrictive direction and does not expose the bug. No test covers `proposed > oracle` near the boundary, so the asymmetry is currently undetected by the test suite.

fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```
