### Title
`extended_initial_reads` in `RpcStorageProofsProvider::get_storage_proofs` Omits Execution `compiled_class_hashes`, Producing Incomplete Classes-Trie `commitment_facts` in Transaction Prover Proof Inputs - (File: `crates/starknet_transaction_prover/src/running/storage_proofs.rs`)

---

### Summary

`RpcStorageProofsProvider::get_storage_proofs` assembles `extended_initial_reads` from two sources — the RPC proof leaf data (class hashes, nonces) and execution storage reads — but silently drops `execution_data.initial_reads.compiled_class_hashes`. Because `create_facts_db_from_storage_proof` is called with this incomplete map, `insert_compiled_class_leaves` inserts zero compiled-class leaves into the facts DB. The resulting `classes_trie_commitment_info.commitment_facts` is missing every compiled-class leaf preimage that was read during execution. A code comment acknowledges the gap and attempts a workaround by appending RPC inner-node facts, but inner nodes are not leaf preimages; the leaves remain absent. The `compiled_class_hashes` are only added to `extended_initial_reads` later, in `runner.rs`, after `commitment_infos` is already finalised — too late to affect the facts DB.

---

### Finding Description

**Two-location state split (the silo analog)**

`VirtualBlockExecutionData` carries two distinct sources of compiled-class information:

1. `execution_data.initial_reads.compiled_class_hashes` — every `(ClassHash → CompiledClassHash)` pair that the blockifier read from the classes trie while executing the transactions.
2. `classes.class_hash_to_compiled_class_hash` — the same mapping fetched from the classes provider in `runner.rs`.

**The omission in `get_storage_proofs`** [1](#0-0) 

```rust
let mut extended_initial_reads = StateMaps::default();
for (leaf, addr) in
    rpc_proof.contracts_proof.contract_leaves_data.iter().zip(&query.contract_addresses)
{
    extended_initial_reads.class_hashes.insert(*addr, ClassHash(leaf.class_hash));
    extended_initial_reads.nonces.insert(*addr, Nonce(leaf.nonce));
}
// Include storage values from execution.
extended_initial_reads.storage.extend(&execution_data.initial_reads.storage);
```

`compiled_class_hashes` from `execution_data.initial_reads` is never copied into `extended_initial_reads`. The field stays at its `Default` value of an empty map.

**Downstream effect: `insert_compiled_class_leaves` inserts nothing**

`create_facts_db_from_storage_proof` is called immediately with this incomplete map: [2](#0-1) 

Inside, `insert_compiled_class_leaves` iterates over `initial_reads.compiled_class_hashes`: [3](#0-2) 

Because the map is empty, no compiled-class leaf preimages are written to the facts DB.

**The workaround is insufficient**

The code acknowledges the gap and tries to patch it: [4](#0-3) 

```rust
// The created commitment infos doesn't have the compiled class hashes,
// as a result it doesn't have the classes trie commitment info.
// We complement it with the RPC proof facts.
let classes_rpc_facts =
    flatten_preimages(&Self::rpc_nodes_to_preimage_map(&rpc_proof.classes_proof));
commitment_infos.classes_trie_commitment_info.commitment_facts.extend(classes_rpc_facts);
```

`classes_rpc_facts` contains only the inner Merkle nodes from the RPC proof, not the leaf preimages (compiled-class hash values). The OS needs leaf preimages to traverse the trie to the leaves it read during execution. They remain absent.

**`compiled_class_hashes` added too late**

In `runner.rs`, the compiled-class hashes are added to `extended_initial_reads` only after `commitment_infos` is already finalised: [5](#0-4) 

```rust
let mut extended_initial_reads = storage_proofs.extended_initial_reads;
// Add class hash to compiled class hash mappings from the classes provider.
extended_initial_reads
    .compiled_class_hashes
    .extend(&classes.class_hash_to_compiled_class_hash);
```

The `commitment_infos` that was computed without the leaves is already baked into `StorageProofs`; this late addition does not retroactively fix the facts DB.

**The `VirtualOsBlockInput` receives wrong proof inputs** [6](#0-5) 

`contract_class_commitment_info` is set from `storage_proofs.commitment_infos.classes_trie_commitment_info`, which has the incomplete `commitment_facts`. The OS uses this map as its preimage database when verifying the classes trie. Any class hash that was read during execution but whose leaf preimage is absent from `commitment_facts` cannot be verified.

---

### Impact Explanation

The Starknet OS verifies the classes Patricia trie by traversing from the root using preimages stored in `commitment_facts`. For every compiled-class hash read during execution, the OS must locate the corresponding leaf preimage. Because those preimages are missing, the OS either:

- **Fails to produce a valid proof** for any Invoke transaction that calls a contract (all non-trivial Invoke transactions), causing valid transactions to be unprovable — a High-severity denial of the proving path; or
- **Accepts a wrong classes-trie root** if the missing leaves allow the trie traversal to terminate at a wrong node, producing an incorrect `contract_class_commitment_info.updated_root` that is fed into the SNOS input — a Critical-severity wrong state value.

This matches: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* and potentially *"Critical. Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."*

---

### Likelihood Explanation

Every Invoke transaction that calls any deployed contract causes the blockifier to read at least one compiled-class hash (the callee's class). This populates `execution_data.initial_reads.compiled_class_hashes` with at least one entry. The omission therefore fires on every non-trivial Invoke transaction processed by the transaction prover, with no special preconditions required from the caller.

---

### Recommendation

In `get_storage_proofs`, after populating `extended_initial_reads` with storage values, also copy the compiled-class hashes from execution initial reads:

```rust
// Include storage values from execution.
extended_initial_reads.storage.extend(&execution_data.initial_reads.storage);
// Include compiled class hashes from execution (needed for classes-trie leaf preimages).
extended_initial_reads.compiled_class_hashes
    .extend(&execution_data.initial_reads.compiled_class_hashes);
```

This ensures `insert_compiled_class_leaves` populates the facts DB with the correct leaf preimages before `create_commitment_infos_with_state_changes` is called, making `classes_trie_commitment_info.commitment_facts` complete. The late addition in `runner.rs` can then be removed or kept as a no-op.

---

### Proof of Concept

1. Submit any Invoke transaction that calls a deployed Sierra contract (e.g., a simple ERC-20 `transfer`).
2. The blockifier reads the contract's compiled-class hash; `execution_data.initial_reads.compiled_class_hashes` is non-empty.
3. `get_storage_proofs` builds `extended_initial_reads` without `compiled_class_hashes`; `insert_compiled_class_leaves` inserts zero leaves.
4. `create_commitment_infos_with_state_changes` produces `classes_trie_commitment_info` whose `commitment_facts` contains only inner nodes from the RPC proof, with no leaf preimages for the executed class.
5. `VirtualOsBlockInput.contract_class_commitment_info` carries this incomplete map.
6. The OS, when verifying the classes trie for the executed class hash, cannot find the leaf preimage in `commitment_facts` and either panics, produces a wrong root, or silently accepts an invalid proof.

fn insert_compiled_class_leaves(
    db_map: &mut DbHashMap,
    initial_reads: &StateMaps,
) -> Result<(), ProofProviderError> {
    for class_hash_value in initial_reads.compiled_class_hashes.values() {
        let compiled_class_hash = CompiledClassHash(class_hash_value.0);
        let leaf_hash = TreeHashFunctionImpl::compute_leaf_hash(&compiled_class_hash);
        insert_leaf_node(db_map, leaf_hash, compiled_class_hash, &EmptyKeyContext)?;
    }

    Ok(())
}
