### Title
Missing Uniqueness Enforcement in `MerkleBlob::batch_insert` Bulk Path Allows Duplicate Key Insertion and Tree Root Corruption - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` contains a fast-path that bypasses the duplicate-key and duplicate-hash guards present in the single-item `insert` function. When the tree already holds two or more leaves, every item in a batch is written directly to the blob without any uniqueness check, allowing an attacker-controlled caller to insert duplicate `KeyId` or `Hash` values. The result is a structurally corrupt Merkle tree whose root hash is computed over phantom duplicate leaves, breaking proof-of-inclusion consistency.

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing any leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` has two code paths:

1. **Bootstrap path** (when `leaf_count <= 1`): pops the last two items from the vector and calls `self.insert(...)`, which does enforce the guards.
2. **Bulk path** (all remaining items, or all items when `leaf_count > 1`): iterates the vector and calls `self.insert_entry_to_blob(...)` directly — **no call to `contains_key` or `contains_leaf_hash`**.

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        ...
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When a duplicate `KeyId` is written via the bulk path, `BlockStatusCache::add_leaf` silently overwrites the previous `key_to_index` entry with the new `TreeIndex`, leaving the first leaf node present in the blob but invisible to the cache. Both leaves are still incorporated into the internal-node hash chain, so the computed root hash diverges from what the cache believes the tree contains.

The `BlockStatusCache` is built from the blob on construction and enforces uniqueness there:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 109-114
if key_to_index.insert(leaf.key, index).is_some() {
    return Err(Error::KeyAlreadyPresent());
}
if leaf_hash_to_index.insert(leaf.hash, index).is_some() {
    return Err(Error::HashAlreadyPresent());
}
``` [3](#0-2) 

This means a tree corrupted by `batch_insert` will fail to reload from disk, but the in-memory state is already corrupt and can be used to generate and export proofs before the inconsistency is detected.

The Python binding exposes this path directly:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518

pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,

    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

### Impact Explanation

A corrupt root hash means that any `ProofOfInclusion` generated from the tree after the duplicate insertion is computed against a root that does not correspond to the canonical key-value set. Downstream consumers that trust the root hash (e.g., DataLayer sync peers) will accept proofs that prove membership in a tree whose root they cannot independently reproduce from the declared key-value set. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."** [5](#0-4) 

### Likelihood Explanation

The Python binding `MerkleBlob.batch_insert` is a public API. Any caller — including untrusted DataLayer sync code that assembles batches from network-received data — can supply a list containing a repeated `KeyId`. The bug is triggered whenever the tree already has two or more leaves (the common steady-state case), because in that branch the bootstrap path is skipped entirely and every item goes through the unchecked bulk loop.

### Recommendation

Add the same duplicate guards to the bulk path of `batch_insert` that exist in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing blob write
}
```

Alternatively, pre-validate the entire input vector for uniqueness before any writes begin, and also cross-check against keys already present in the tree.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n):
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 → bootstrap path is skipped
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(200), ValueId(200), h(200))

# batch_insert bulk path: all three items bypass uniqueness checks
# KeyId(100) already exists; KeyId(999) appears twice in the batch
blob.batch_insert(
    [(KeyId(999), ValueId(1)), (KeyId(999), ValueId(2)), (KeyId(300), ValueId(300))],
    [h(1), h(2), h(300)],
)

# Tree now contains two leaf nodes with KeyId(999) but cache tracks only one.
# Root hash is computed over the duplicate-bearing tree → corrupted root.
root = blob.get_root_hash()
# proof_of_inclusion for KeyId(999) reflects only one of the two leaves;
# the other leaf silently inflates the root hash.
proof = blob.get_proof_of_inclusion(KeyId(999))
assert proof.valid()   # passes, but root is wrong
