Audit Report

## Title
Ex-Owner Retains Updater Privilege After Ownership Transfer, Enabling Persistent `confidenceParam` Manipulation — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`)

## Summary
`transferProviderOwnership` in both `PriceProviderFactory` and `AnchoredProviderFactory` updates `providerOwner[provider]` and the `_providersByCreator` enumerable sets but never clears `isUpdater[provider][previousOwner]`. Because `_requireUpdater` independently accepts any address where `isUpdater[provider][msg.sender] == true`, an ex-owner who self-granted updater rights before transferring retains the ability to call `setConfidence` indefinitely, manipulating the bid-ask spread on pools that use the provider.

## Finding Description
`transferProviderOwnership` at [1](#0-0)  updates `providerOwner[provider]` and the creator sets but performs no write to `isUpdater`. `_requireUpdater` at [2](#0-1)  passes independently for any address where `isUpdater[provider][msg.sender] == true`, with no reference to current ownership. `setConfidence` at [3](#0-2)  calls `_requireUpdater` then directly invokes `PriceProvider.setConfidenceParam`. The exploit sequence is:

1. Attacker calls `createPriceProvider(...)` — permissionless; becomes `providerOwner[provider]`.
2. Attacker calls `grantUpdater(provider, attacker)` — sets `isUpdater[provider][attacker] = true`. [4](#0-3) 
3. Attacker calls `transferProviderOwnership(provider, victim)` — `providerOwner[provider]` becomes `victim`, but `isUpdater[provider][attacker]` is never cleared. [5](#0-4) 
4. Attacker calls `factory.setConfidence([provider], [value])` — `_requireUpdater` passes because `isUpdater[provider][attacker]` is still `true`; `setConfidenceParam` is invoked on the provider.

The identical flaw exists in `AnchoredProviderFactory.transferProviderOwnership` at [6](#0-5)  with the same `setConfidence` / `_requireUpdater` pair. [7](#0-6) 

The existing test `testOldOwnerCannotUpdateAfterTransfer` only covers the case where the old owner did **not** pre-grant themselves as updater and therefore fails `_requireUpdater` on both branches; it does not exercise the self-grant-then-transfer path. [8](#0-7) 

## Impact Explanation
`confidenceParam` is used in `_getBidAskFrom` to compute `delta = midPrice * confidence / CONFIDENCE_BASE`, then `bid = midPrice - delta` and `ask = midPrice + delta`. Widening `confidenceParam` toward `CONFIDENCE_MAX` lowers the bid toward zero and raises the ask, causing every swap against pools using this provider to execute at a worse price than the oracle mid warrants — a direct bad-price execution impact on traders. Narrowing it to zero collapses the spread to only `marginStep`, enabling arbitrage extraction from the pool. This matches the allowed impact category of bad-price execution: unclamped bid/ask quotes reaching pool swaps.

## Likelihood Explanation
Provider creation is fully permissionless via `createPriceProvider`. Any address can execute the four-step sequence with no privileged preconditions. The new owner receives no on-chain signal that a stale `isUpdater` entry exists — the `ProviderOwnershipTransferred` event carries only `provider`, `previousOwner`, and `newOwner`; `isUpdater` state is not surfaced. [9](#0-8)  The new owner must proactively enumerate and revoke all prior updaters, which is neither enforced nor documented. The attack is repeatable every `CONFIDENCE_COOLDOWN` interval for as long as the new owner has not explicitly called `revokeUpdater`.

## Recommendation
In `transferProviderOwnership` (both factories), explicitly revoke the previous owner's updater entry before updating `providerOwner`:

```solidity
if (isUpdater[provider][previousOwner]) {
    isUpdater[provider][previousOwner] = false;
    emit UpdaterRevoked(provider, previousOwner);
}
```

Alternatively, implement a two-step ownership acceptance pattern so the incoming owner can audit and revoke updaters before taking control.

## Proof of Concept
```solidity
function testExOwnerRetainsUpdaterAfterTransfer() public {
    vm.warp(100);
    address p = _create(FEED_A);          // owner creates provider

    factory.grantUpdater(p, owner);        // owner self-grants as updater

    factory.transferProviderOwnership(p, creatorB); // transfer to new owner

    // owner is no longer providerOwner[p], but isUpdater[p][owner] == true
    address[] memory providers = new address[](1);
    providers[0] = p;
    uint256[] memory values = new uint256[](1);
    values[0] = 900_000;                   // widen spread aggressively

    // This should revert but does NOT — ex-owner still passes _requireUpdater
    factory.setConfidence(providers, values);
    assertEq(PriceProvider(p).confidenceParam(), 900_000);
}
```
