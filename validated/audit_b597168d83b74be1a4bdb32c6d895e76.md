Looking at the exact validation logic in `set_state_header` and the honest construction in `compute_state_response_header`, I need to determine whether the shard-index binding between `root_proofs[i][j].root` and `from_shard_id` is enforced.

**Honest construction** (server side): [1](#0-0) 

The honest builder uses `from_shard_index` to look up the exact `prev_outgoing_receipts_root` for that shard and the Merkle path at that index. The binding is: `root_proofs[i][j].root == block.chunks()[from_shard_index].prev_outgoing_receipts_root()`.

**Validation** (receiver side, steps 4e and 4f): [2](#0-1) 

Step 4e checks: `verify_path(root, proof, receipts_hash)` — receipts are in the trie rooted at `root`.
Step 4f checks: `verify_path(block_header.prev_chunk_outgoing_receipts_root(), block_proof, root)` — `root` is *some* valid leaf in the block's outgoing-receipts Merkle tree.

**The gap:** Neither step verifies that the leaf position encoded in `block_proof` corresponds to `from_shard_id`'s shard index. The Merkle proof `block_proof` implicitly encodes a position (via left/right directions), but the validator never extracts or checks that position against `from_shard_id`.

**The Pigeonhole argument is broken:** [3](#0-2) 

The comment claims uniqueness of `from_shard_id` values (step 4d) is sufficient to prove all receipts are present. But the `root` values in `root_proofs[i]` are **not** required to be unique. An attacker can reuse shard X's `(root=P_X, block_proof=proof_for_P_X)` for a different `from_shard_id=Y`, while providing shard X's receipts R_X and a valid proof for R_X under P_X. All checks pass:

- 4d: `from_shard_id` values X and Y are distinct ✓
- 4e for Y: `verify_path(P_X, proof_for_R_X_in_P_X, hash(ReceiptList(shard_A, R_X)))` ✓
- 4f for Y: `verify_path(block_outgoing_root, proof_for_P_X, P_X)` ✓

Shard Y's actual receipts R_Y are never included. The malicious header is stored to `DBCol::StateHeaders`: [4](#0-3) 

Then `set_state_finalize` collects and applies the receipts from the stored header: [5](#0-4) 

R_Y is absent from `receipt_proof_responses`, so `apply_chunk` runs without shard Y's cross-shard receipts. There is no post-application state-root check in `set_state_finalize` that would catch the mismatch before the corrupted state is committed.

**Attacker capability:** The attacker only needs to be a network peer that the syncing node selects for state download. All inputs (R_X, proof for R_X in P_X, P_X, proof for P_X in the block's outgoing root) are obtainable by requesting an honest `ShardStateSyncResponseHeader` from any honest node and then substituting the entry for `from_shard_id=Y`.

### Title
Missing shard-index binding in `root_proofs` validation allows receipt suppression during state sync — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` step 4f verifies that `root_proofs[i][j].root` is *a* valid leaf in the block's `prev_chunk_outgoing_receipts_root` Merkle tree but never checks that the leaf's position corresponds to `receipt_proofs[j].from_shard_id`. An unprivileged peer can substitute shard X's `(root, block_proof)` for the entry attributed to `from_shard_id=Y`, causing the syncing node to omit shard Y's incoming receipts when applying the chunk after state sync.

### Finding Description
The honest builder at `compute_state_response_header` constructs `root_proofs[i][j]` by looking up `block.chunks()[from_shard_index].prev_outgoing_receipts_root()` and `block_receipts_proofs[from_shard_index]`, binding the root to the correct shard index. The validator in `set_state_header` does not reconstruct or verify this binding. Step 4f only calls `verify_path(block_header.prev_chunk_outgoing_receipts_root(), block_proof, root)`, which confirms `root` is *some* committed leaf, not that it is the leaf at the position for `from_shard_id`. Because `root` values in `root_proofs[i]` are also not required to be distinct, an attacker can reuse shard X's `(P_X, proof_for_P_X)` for `from_shard_id=Y` while supplying shard X's receipts R_X and a valid inner proof. The Pigeonhole argument (uniqueness of `from_shard_id`) no longer guarantees coverage of all shards' receipts.

### Impact Explanation
The malicious header is persisted to `DBCol::StateHeaders`. `set_state_finalize` then calls `apply_chunk` with the attacker-supplied receipt list, which omits R_Y. The resulting state root diverges from the canonical committed value. The syncing node operates on a corrupted state until it fails to validate subsequent blocks, requiring a full re-sync. A targeted attack against a validator node during state sync could cause it to produce invalid blocks and be slashed.

### Likelihood Explanation
The attacker requires only network-peer access to a syncing node and read access to any honest node's state-sync response (all public data). No validator key, stake, or privileged role is needed. The attack is constructive: every required value (R_X, inner proof, P_X, block_proof) is directly readable from an honest `ShardStateSyncResponseHeader`.

### Recommendation
In `set_state_header`, after step 4f, verify that `root_proofs[i][j].root` equals the `prev_outgoing_receipts_root` of the chunk for `from_shard_id` in block B. Concretely: resolve `from_shard_id` to its shard index in the block's shard layout, fetch the block (which is already trusted via the chain store), and assert `root == block.chunks()[from_shard_index].prev_outgoing_receipts_root()`. Additionally, enforce uniqueness of `root` values within `root_proofs[i]` to close the reuse vector independently.
