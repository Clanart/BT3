### Title
Unsafe `IERC20Metadata` Calls in `logMetadata()` Revert for Non-Standard Tokens, Permanently Freezing Bridged Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.logMetadata()` calls `IERC20Metadata(tokenAddress).name()`, `.symbol()`, and `.decimals()` with no safe-decode wrapper. For tokens that return `bytes32` instead of `string` (e.g., Sai/MakerDAO, WBTC on some deployments), the ABI decoder reverts. Because `logMetadata()` is the only permissionless path to register a native EVM token on the NEAR side, and because `initTransfer()` has no prerequisite check that `logMetadata()` was ever called, a user can lock tokens in the bridge that can never be finalized on NEAR — permanently freezing their funds.

---

### Finding Description

`logMetadata()` is declared `external payable` with no access-control modifier, making it callable by any unprivileged user:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name    = IERC20Metadata(tokenAddress).name();    // line 225
    string memory symbol  = IERC20Metadata(tokenAddress).symbol();  // line 226
    uint8  decimals       = IERC20Metadata(tokenAddress).decimals(); // line 227
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

`IERC20Metadata` is the standard OpenZeppelin interface that expects `name()` and `symbol()` to return `string`. When the EVM ABI decoder receives a raw `bytes32` value (as returned by Sai, early WBTC, and other pre-EIP-20 tokens), `abi.decode` reverts unconditionally. There is no `try/catch`, no low-level `staticcall`, and no `returnDataToString()` fallback.

The same unsafe pattern appears in `addCustomToken()` (lines 99–101), but that function is `onlyRole(DEFAULT_ADMIN_ROLE)` and therefore out of scope for an unprivileged attacker path. [2](#0-1) 

`initTransfer()` — the function that actually locks user tokens — has **no check** that `logMetadata()` was ever successfully called for the token:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount, ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...
    emit BridgeTypes.InitTransfer(...);
}
``` [3](#0-2) 

The `LogMetadata` event is the signal the NEAR relayer uses to deploy a wrapped token on NEAR. Without it, the NEAR side has no wrapped token to mint against an incoming `InitTransfer` message, so the cross-chain transfer can never be finalized. [4](#0-3) 

---

### Impact Explanation

**Permanent freezing of user funds (Critical/High).**

A user who calls `initTransfer()` for a non-standard token (one whose `name()` returns `bytes32`) will have their tokens locked in the bridge contract. Every subsequent call to `logMetadata()` for that token reverts, so the NEAR side never receives the metadata event, never deploys a wrapped token, and can never finalize the transfer. The locked ERC-20 tokens are irrecoverable without an admin intervention that itself requires a contract upgrade or a separate rescue path — neither of which exists today.

---

### Likelihood Explanation

Several real, widely-held ERC-20 tokens are known to return `bytes32` for `name()` and `symbol()`: Sai (the original MakerDAO single-collateral DAI), WBTC on certain deployments, and others documented in the BoringCrypto audit trail. Any user who attempts to bridge one of these tokens — or any future token with the same non-compliance — triggers the freeze. The entry path (`initTransfer`) is fully permissionless and requires no special role.

---

### Recommendation

Replace the direct `IERC20Metadata` calls in `logMetadata()` with a safe low-level wrapper that handles both `string` and `bytes32` return values, following the BoringCrypto `safeSymbol`/`safeName` pattern:

```solidity
function _safeTokenName(address token) internal view returns (string memory) {
    (bool success, bytes memory data) =
        token.staticcall(abi.encodeWithSignature("name()"));
    if (!success || data.length == 0) return "";
    if (data.length == 32) return string(abi.encodePacked(bytes32(abi.decode(data, (bytes32)))));
    return abi.decode(data, (string));
}
```

Apply the same pattern for `symbol()`. For `decimals()`, a similar `staticcall` guard is advisable since some tokens omit it entirely.

Additionally, consider adding a guard in `initTransfer()` that requires `logMetadata()` to have been successfully called (e.g., by storing a `registeredTokens` mapping set inside `logMetadata()`), so tokens cannot be locked before they are registerable.

---

### Proof of Concept

1. Deploy a token whose `name()` returns `bytes32("Sai Stablecoin v1.0")` (32 raw bytes, not ABI-encoded as `string`).
2. Call `OmniBridge.initTransfer(saiAddress, 1e18, 0, 0, "alice.near", "")` — succeeds; 1 Sai is locked in the bridge.
3. Call `OmniBridge.logMetadata(saiAddress)` — **reverts** at line 225 because `abi.decode(data, (string))` fails on a 32-byte non-length-prefixed payload.
4. The NEAR relayer never sees a `LogMetadata` event for Sai; no wrapped token is deployed on NEAR.
5. The NEAR bridge contract receives the `InitTransfer` Wormhole/MPC message but cannot mint (no token contract exists); the transfer is unclaimable.
6. The 1 Sai remains permanently locked in `OmniBridge` with no recovery path available to the user. [1](#0-0) [5](#0-4)
