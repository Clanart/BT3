### Title
Blacklisted ERC-20 Recipient Permanently Freezes Cross-Chain Funds in `finTransfer` — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` transfers tokens directly to `payload.recipient` with no fallback. If the recipient address is blacklisted by the token contract (e.g., USDC, USDT) or is a contract that rejects ETH, every call to `finTransfer` for that signed payload will revert. Because the recipient is cryptographically bound in the MPC-signed payload, no alternative delivery path exists. The origin-chain transfer is already finalized, so the bridged funds are permanently irrecoverable.

---

### Finding Description

`finTransfer` in `OmniBridge.sol` follows this sequence:

1. Check and mark `completedTransfers[payload.destinationNonce] = true` (line 287).
2. Verify the MPC signature over a Borsh-encoded payload that includes `payload.recipient` (lines 289–313).
3. Deliver tokens to `payload.recipient` via one of four paths (lines 317–355).

For native ERC-20 tokens the delivery is:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
```

`safeTransfer` (OpenZeppelin) reverts on any failure, including a USDC/USDT blacklist hit. When it reverts the entire transaction reverts, including the nonce write at line 287, so the nonce is not permanently consumed. However, the signed payload encodes `payload.recipient` as a fixed field; the MPC will not re-sign the same logical transfer with a different recipient. The NEAR-side `fin_transfer_callback` has already called `add_fin_transfer`, consuming the origin nonce and burning/locking the source tokens. There is no admin rescue function, no pull-pattern fallback, and no mechanism to redirect the delivery to an alternative address.

The same permanent-freeze applies to the ETH path (line 322, `revert FailedToSendEther()`) when the recipient is a contract that rejects plain ETH sends, and to the ERC-1155 path (line 324) when the recipient does not implement `onERC1155Received`. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

- The origin-chain tokens (NEAR side) are burned or locked inside the NEAR bridge contract by `fin_transfer_callback` before the EVM leg is attempted.
- The EVM bridge holds the native ERC-20 tokens (deposited during `initTransfer` on EVM) or would mint bridge tokens; neither can be released.
- Because the MPC signature covers the exact recipient, no valid alternative `finTransfer` call can be constructed.
- There is no admin sweep, no pull-pattern claim, and no on-chain mechanism to redirect the delivery.
- Funds are permanently frozen in the bridge with no recovery path. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

**Medium.**

USDC and USDT both maintain on-chain blacklists enforced at the token level. A user's address can be blacklisted by Circle or Tether at any time, including after a bridge transfer is initiated but before the EVM `finTransfer` is executed (cross-chain finality windows are minutes to hours). The scenario requires no privileged access: any ordinary bridge user whose address is subsequently blacklisted triggers the freeze. The ETH-rejection variant is also reachable by any smart-contract wallet that does not implement a payable fallback.

---

### Recommendation

Replace the direct-push delivery with a pull pattern for the native ERC-20 (and ETH) paths:

```solidity
// Instead of reverting on failed delivery, credit a claimable balance:
mapping(address => mapping(address => uint256)) public claimable;

// In finTransfer, replace safeTransfer with:
try IERC20(payload.tokenAddress).transfer(payload.recipient, payload.amount) {
    // success — nothing extra needed
} catch {
    claimable[payload.recipient][payload.tokenAddress] += payload.amount;
    emit DeliveryFailed(payload.recipient, payload.tokenAddress, payload.amount);
}

// Add a permissionless claim function:
function claim(address tokenAddress) external {
    uint256 amount = claimable[msg.sender][tokenAddress];
    require(amount > 0, "nothing to claim");
    claimable[msg.sender][tokenAddress] = 0;
    IERC20(tokenAddress).safeTransfer(msg.sender, amount);
}
```

Apply the same pattern to the ETH path (store in a `claimableEth` mapping and let the recipient pull via `claim(address(0))`). This preserves the "state before external calls" invariant documented in `CLAUDE.md` while eliminating the permanent-freeze risk. [5](#0-4) 

---

### Proof of Concept

1. Alice holds USDC on NEAR and initiates a bridge transfer to her EVM address `0xAlice` via `initTransfer` on NEAR. The NEAR bridge burns her USDC and records the transfer; `fin_transfer_callback` calls `add_fin_transfer`, consuming the origin nonce.

2. Circle blacklists `0xAlice` (e.g., for regulatory compliance) before the relayer submits the EVM leg.

3. The relayer calls `OmniBridge.finTransfer(signatureData, payload)` where `payload.recipient = 0xAlice` and `payload.tokenAddress = USDC`.

4. Execution reaches line 351:
   ```solidity
   IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
   ```
   USDC's `transfer` reverts because `0xAlice` is blacklisted. `safeTransfer` propagates the revert.

5. The entire transaction reverts. `completedTransfers[nonce]` is rolled back to `false`.

6. Every subsequent attempt to call `finTransfer` with the same MPC-signed payload reverts identically. No new MPC signature covering a different recipient can be obtained for the same origin transfer.

7. Alice's USDC is permanently locked: burned on NEAR, undeliverable on EVM, with no admin rescue or pull-claim mechanism available. [6](#0-5) [7](#0-6)
