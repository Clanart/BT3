### Title
Missing `token::authority = user` Constraint on `from` Allows Delegate to Bridge Victim's Tokens — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` account struct does not enforce that `user` is the owner of the `from` token account. Because SPL Token's `transfer_checked` accepts an approved delegate as a valid authority, any address that holds a delegate allowance on a victim's token account can call `init_transfer` with `from=victim_ata`, drain the victim's tokens into the bridge vault, and emit a Wormhole VAA crediting an attacker-controlled NEAR recipient.

---

### Finding Description

The `from` account in `InitTransfer` is constrained only to match the mint and token program:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint. The `process()` function then issues a `transfer_checked` CPI with `authority: self.user.to_account_info()`:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),
            authority: self.user.to_account_info(),
            mint: self.mint.to_account_info(),
        },
    ),
    ...
)?;
``` [2](#0-1) 

SPL Token's `transfer_checked` succeeds when `authority` is either the token account **owner** or an **approved delegate** with sufficient delegated amount. The bridge enforces only the former case at the Anchor constraint level — it does not. Therefore, if `user` is a delegate (not the owner) of `from`, the CPI succeeds.

The Wormhole VAA is then posted with `self.user.key()` as the sender and the attacker-supplied `payload.recipient` as the NEAR recipient:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
``` [3](#0-2) 

The same pattern applies to the burn path (bridged tokens), where `authority: self.user.to_account_info()` is passed to `burn` with no ownership check on `from`. [4](#0-3) 

---

### Impact Explanation

A victim's tokens are irreversibly locked in the bridge vault (native path) or burned (bridged path), and a Wormhole VAA is emitted crediting an attacker-controlled NEAR address. The victim receives nothing. This is direct, unrecoverable theft of user funds across chains.

---

### Likelihood Explanation

The precondition is that the attacker holds a delegate allowance on the victim's token account. This is not self-grantable — the victim must have previously called `approve`. However, this is a realistic and common state:

- Users routinely approve DEX routers, yield aggregators, and other DeFi contracts as delegates.
- A malicious or compromised contract that has been approved as a delegate can exploit this path.
- The attacker needs only to sign a valid Solana transaction — no privileged role, no leaked key, no oracle manipulation.

---

### Recommendation

Add `token::authority = user` to the `from` account constraint in `InitTransfer`:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This ensures Anchor rejects any `from` account whose owner is not `user`, eliminating the delegate-abuse path entirely.

---

### Proof of Concept

1. Victim holds `victim_ata` with 1,000 tokens and has previously called `approve(attacker_pubkey, 1000)` on it (e.g., for a DEX interaction).
2. Attacker constructs a transaction calling `init_transfer` with:
   - `from = victim_ata`
   - `user = attacker` (signs the transaction)
   - `payload.recipient = "attacker.near"`
   - `payload.amount = 1000`, `payload.fee = 1`
3. Anchor validates `from`: mint matches ✓, token_program matches ✓ — no ownership check.
4. `transfer_checked` CPI executes: SPL Token checks that `attacker` is a delegate of `victim_ata` with allowance ≥ 1000 — succeeds.
5. 1,000 tokens move from `victim_ata` to the bridge vault.
6. Wormhole VAA is posted encoding `sender=attacker_pubkey`, `recipient="attacker.near"`.
7. NEAR side finalizes the transfer, crediting `attacker.near` with the tokens.
8. Victim's tokens are gone; attacker receives them on NEAR.
