### Title
Missing Chain-ID Domain Separation in `deployToken` / `deploy_token` Signed Hash Enables Cross-Chain Signature Replay — (Files: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The Borsh-encoded hash that the NEAR MPC signs for `deployToken` / `deploy_token` does **not** include the destination chain ID, while the analogous `finTransfer` / `fin_transfer` hash **does**. Because the same NEAR MPC-derived address (`nearBridgeDerivedAddress`) is used across every destination chain, a single MPC-signed `deployToken` signature is valid on every chain simultaneously. An unprivileged relayer can replay a signature obtained for one chain (e.g., Ethereum) on any other chain (Arbitrum, Starknet, Solana) to deploy an unauthorized bridged-token contract without a fresh MPC signing round.

---

### Finding Description

**Asymmetry between `deployToken` and `finTransfer` hashing:**

`finTransfer` (EVM) encodes `omniBridgeChainId` twice in the Borsh payload before hashing:

```solidity
bytes1(omniBridgeChainId),   // destination chain
...
bytes1(omniBridgeChainId),   // recipient chain
``` [1](#0-0) 

`deployToken` (EVM) encodes only `(PayloadType.Metadata, token, name, symbol, decimals)` — **no chain ID**:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

The same omission exists on Starknet. `MetadataPayload::to_borsh()` encodes only `(PayloadType::Metadata, token, name, symbol, decimals)`:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
``` [3](#0-2) 

While `TransferMessagePayload::to_borsh()` explicitly binds to `chain_id`: [4](#0-3) 

On Solana, `DeployTokenPayload::serialize_for_near()` serializes only `IncomingMessageType::Metadata` followed by `(token, name, symbol, decimals)` — again, no chain ID:

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    self.serialize(&mut writer)?;
``` [5](#0-4) 

Note that `SOLANA_OMNI_BRIDGE_CHAIN_ID` **is** written in the outgoing `DeployTokenResponse`, but is absent from the incoming `DeployTokenPayload` that is actually signature-verified. [6](#0-5) 

**Root cause:** The `MetadataPayload` struct carries no chain-ID field and none of the three chain implementations inject one before hashing. [7](#0-6) 

---

### Impact Explanation

The NEAR MPC system derives a **single** secp256k1 key pair; the same `nearBridgeDerivedAddress` / `omni_bridge_derived_address` / `derived_near_bridge_address` is configured on every destination chain. Because the `deployToken` hash is chain-agnostic, one MPC signature is cryptographically valid on all chains simultaneously.

An attacker who observes a valid `deployToken` transaction on chain A can submit the identical `(signatureData, metadata)` tuple to chain B, C, … and deploy the bridged-token contract there without any new MPC authorization. The only on-chain guard is the "token already exists" check, which only prevents a second deployment on the **same** chain. [8](#0-7) [9](#0-8) 

Once an unauthorized bridged-token contract exists on an unintended chain, the NEAR relayer infrastructure may begin signing `finTransfer` messages for that chain (because it sees a deployed token), enabling minting of unbacked tokens. This constitutes unauthorized token deployment and potential unbacked supply — matching the "High: proof/signature/MPC verification bypass enabling unauthorized token deployment" impact class.

---

### Likelihood Explanation

**Medium.** The preconditions are:
1. The bridge is deployed on at least two chains sharing the same MPC-derived address (the standard multi-chain deployment model).
2. A `deployToken` transaction for a given token has been broadcast on any one of those chains (publicly observable on-chain).
3. The attacker submits the same calldata to another chain — a single permissionless transaction requiring no special access.

No key compromise, no colluding MPC signers, and no privileged role is required. Any public observer of chain A can execute the replay on chain B.

---

### Recommendation

1. **Add `destination_chain_id` to `MetadataPayload`** (or inject it at hash-construction time, as `finTransfer` already does) on all three chains so the signed hash is chain-specific.

   EVM example — mirror the `finTransfer` pattern:
   ```solidity
   bytes memory borshEncoded = bytes.concat(
       bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
       bytes1(omniBridgeChainId),          // add this
       Borsh.encodeString(metadata.token),
       ...
   );
   ```

2. **Add an `expiredAt` timestamp** to `MetadataPayload` (as the referenced report recommends) so that a stale MPC signature cannot be replayed months later after the bridge configuration changes.

3. Ensure the NEAR side includes the destination chain ID when constructing the message it submits to the MPC for signing, so the fix is end-to-end.

---

### Proof of Concept

1. NEAR MPC signs a `deployToken` for `usdc.near` on Ethereum (chain ID `0x01`). The signed Borsh blob is `[0x01, <token_borsh>, <name_borsh>, <symbol_borsh>, <decimals>]` — identical for every chain.
2. The relayer submits `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Ethereum. Token deployed at address `T_eth`.
3. Attacker copies `sig` and the same `metadata` calldata and calls `deployToken` on the Starknet bridge. `_verify_borsh_signature` recovers the same MPC address from the same hash → passes. `near_to_starknet_token["usdc.near"]` is zero → passes. Token `T_starknet` is deployed.
4. Starknet bridge now has an unauthorized `usdc.near` bridged token. If the NEAR relayer subsequently signs `finTransfer` messages destined for Starknet (because it observes the `DeployToken` event), it will mint `T_starknet` tokens against any cross-chain transfer, creating unbacked supply on Starknet.
