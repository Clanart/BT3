Looking at the actual code, I need to verify three things: (1) whether `verify_path` checks position, (2) whether `set_state_header` checks `chunk.shard_id() == shard_id`, and (3) whether `validate_chunk_proofs` checks shard identity.

**Finding: The gap is real.**

---

### Title
Missing Shard-Position Binding in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` in `ChainStateSyncAdapter` accepts a `ShardStateSyncResponseHeader` for `shard_id=A` that contains a genuine `ShardChunk` from `shard_id=B`. The `verify_path` call proves only that the chunk is *somewhere* in `chunk_headers_root`, not that it occupies the merkle-tree leaf index corresponding to shard A. No code in the function compares `chunk.shard_id()` against the `shard_id` parameter. The corrupted header is then persisted under `StateHeaderKey(shard_id=A, sync_hash)` carrying shard B's `prev_state_root`, causing the node to download and apply shard B's state under shard A's identity.

### Finding Description

**`verify_path` does not check position.**

`verify_path` in `core/primitives/src/merkle.rs` simply recomputes the root from the path and compares it to the expected root:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [1](#0-0) 

It does not check which leaf index the path corresponds to. The function `verify_path_with_index` exists precisely to add that check, and is used in `validate_part` for chunk parts, but is **not** used in `set_state_header`. [2](#0-1) 

**`set_state_header` uses `verify_path` without position binding.**

The check at lines 394–403 proves only that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is *some* leaf in `sync_prev_block_header.chunk_headers_root()`. A valid merkle proof for shard B's leaf position passes this check even when `shard_id=A` was requested:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { ... }
``` [3](#0-2) 

**No `chunk.shard_id() == shard_id` check anywhere in `set_state_header`.**

Reading the entire function (lines 368–532), the `shard_id` parameter is used only to key the storage write at line 527. It is never compared against `chunk.shard_id()`. [4](#0-3) 

**`validate_chunk_proofs` does not check shard identity.**

The call at line 380 validates the chunk's internal hash, `tx_root`, and `outgoing_receipts_root`. It does not verify that the chunk belongs to the requested shard: [5](#0-4) 

**The same position-blindness applies to `prev_chunk_proof`.**

The second `verify_path` call (lines 416–425) for `prev_chunk_header` has the identical gap. [6](#0-5) 

**Corrupted value stored.**

The header is written under `StateHeaderKey(shard_id=A, sync_hash)` but contains shard B's chunk and shard B's `prev_state_root`. Downstream, `set_state_part` reads `prev_state_root` directly from the stored chunk header and validates parts against it: [7](#0-6) 

`set_state_finalize` then applies the downloaded state to shard A's `ShardUId` using shard B's state root, installing the wrong state. [8](#0-7) 

### Impact Explanation

A syncing node installs shard B's trie state under shard A's `ShardUId`. The node's shard A state is permanently corrupted: it will compute wrong state roots, fail chunk validation, and be unable to participate in consensus for shard A. The attack is silent — no error is returned, and the node proceeds normally until it tries to produce or validate a chunk.

### Likelihood Explanation

Any peer that tracks shard B can mount this attack. No validator or operator privilege is required. The attacker only needs:
1. A valid `ShardChunk` for shard B (available to any node tracking shard B).
2. The merkle proof for shard B's leaf in `sync_prev_block.chunk_headers_root` (computable from the block).
3. Valid incoming receipts for shard A (publicly available on-chain).
4. Shard B's `StateRootNode` (available to any node holding shard B's state).

State sync is specifically the phase where a node trusts external peers for data it does not yet have locally, making this the highest-risk window.

### Recommendation

In `set_state_header`, after extracting the chunk, add an explicit shard identity check:

```rust
let chunk_shard_id = chunk.shard_id();
if chunk_shard_id != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
        chunk_shard_id, shard_id
    )));
}
```

Additionally, replace the `verify_path` calls with `verify_path_with_index`, supplying the shard index derived from the epoch's shard layout, so the merkle proof is bound to the correct leaf position. Apply the same fix to the `prev_chunk_proof` verification.

### Proof of Concept

Build a two-shard test-loop state sync test:

1. Produce enough blocks to reach a sync point with two shards (shard 0 and shard 1).
2. From the honest node, obtain `sync_prev_block` and compute the `chunk_headers_root` merkle tree.
3. Extract the genuine `ShardChunk` for shard 1 and its merkle proof at leaf index 1.
4. Construct a `ShardStateSyncResponseHeaderV2` with:
   - `chunk` = shard 1's chunk
   - `chunk_proof` = shard 1's merkle proof (leaf index 1)
   - `incoming_receipts_proofs` = valid receipts for shard 0 (copied from the honest header)
   - `state_root_node` = shard 1's `StateRootNode`
5. Call `set_state_header(shard_id=0, sync_hash, forged_header)` on the syncing node — assert it returns `Ok(())`.
6. Read back the stored header under `StateHeaderKey(shard_id=0, sync_hash)` and assert `stored_header.chunk_prev_state_root() == shard_1_prev_state_root` (not shard 0's), confirming the corruption.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
