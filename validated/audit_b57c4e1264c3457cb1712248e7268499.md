### Title
Unsigned Field-Element Subtraction Underflow in `check_proof_facts` Causes Unprovable Block (Network Halt) — (File: `execution/execution_constraints.cairo`)

---

### Summary

`check_proof_facts` in `execution_constraints.cairo` performs the subtraction `current_block_number - STORED_BLOCK_HASH_BUFFER` directly in Cairo's `felt` (field element) arithmetic without first verifying that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. When `current_block_number < 10` (the first 10 blocks of the network), the subtraction wraps around to a huge field element near the prime, causing the subsequent `assert_nn_le` range check to fail. This makes the block unprovable — a network halt.

---

### Finding Description

In `execution_constraints.cairo`, `check_proof_facts` validates that a user-supplied proof-facts block number is old enough:

```cairo
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
``` [1](#0-0) 

`STORED_BLOCK_HASH_BUFFER = 10` is a compile-time constant. [2](#0-1) 

In Cairo, `felt` arithmetic is modular over the Stark prime P (~2²⁵¹). When `current_block_number < 10`, the expression `current_block_number - STORED_BLOCK_HASH_BUFFER` does **not** revert — it silently wraps to `P - (10 - current_block_number)`, a value near 2²⁵¹. The `assert_nn_le(a, b)` builtin then checks that `b - a` is in `[0, 2¹²⁸)`. Since `b ≈ P ≫ 2¹²⁸`, the range check fails, making the entire OS execution invalid and the block unprovable.

The same subtraction pattern appears in two other places in the codebase, but **both are correctly guarded**:

1. `write_block_number_to_block_hash_mapping` in `os_utils.cairo` explicitly calls `is_nn(old_block_number)` and returns early if the result is negative:

```cairo
tempvar old_block_number = block_context.block_info_for_execute.block_number -
    STORED_BLOCK_HASH_BUFFER;
let is_old_block_number_non_negative = is_nn(old_block_number);
if (is_old_block_number_non_negative == FALSE) {
    return ();
}
``` [3](#0-2) 

2. `execute_get_block_hash` in `syscall_impls.cairo` uses a hint-based guard before the same subtraction, writing a failure response instead of panicking:

```cairo
if (is_block_number_in_block_hash_buffer != FALSE) {
    assert_lt(current_block_number, request_block_number + STORED_BLOCK_HASH_BUFFER);
    write_failure_response(...);
    return ();
}
assert_nn_le(request_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER);
``` [4](#0-3) 

`check_proof_facts` has **no such guard**.

The function is called from `execute_invoke_transaction` with user-supplied `proof_facts_size` and `proof_facts` loaded from the transaction via the `TxProofFacts` hint: [5](#0-4) 

The function short-circuits only when `proof_facts_size == 0`: [6](#0-5) 

An attacker who submits an invoke transaction with `proof_facts_size > 0` during blocks 0–9 triggers the unguarded subtraction.

---

### Impact Explanation

The OS is a Cairo program whose execution produces a STARK proof. A range-check failure inside the OS makes the block **unprovable**: no valid proof can be generated for a block that includes such a transaction. This constitutes a **total network halt** for the duration of the first 10 blocks — the sequencer cannot finalize any block that contains the malicious transaction, and must either exclude it (if detected off-chain) or be stuck.

**Impact class: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

- The vulnerability window is the first 10 blocks of the network (block numbers 0–9).
- Any unprivileged user who can submit an invoke transaction with non-empty `proof_facts` (a feature exposed for client-side proving / virtual OS) can trigger it.
- No special privileges, leaked keys, or trusted roles are required.
- The attacker only needs to know the network is in its early blocks and submit one transaction.

---

### Recommendation

Mirror the guard used in `write_block_number_to_block_hash_mapping`. Before the subtraction, verify that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. If not, treat the proof facts as invalid (return an error or revert the transaction):

```cairo
// Guard: if current_block_number < STORED_BLOCK_HASH_BUFFER, no valid proof facts
// can exist yet — reject immediately.
with_attr error_message("Block number too low for proof facts.") {
    assert_nn_le(STORED_BLOCK_HASH_BUFFER, current_block_number);
}
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

Alternatively, use `is_nn` (as done in `os_utils.cairo`) to branch safely before the subtraction.

---

### Proof of Concept

1. Network launches at block 0. `STORED_BLOCK_HASH_BUFFER = 10`.
2. Attacker submits an invoke transaction at block 5 with `proof_facts_size = ProofHeader.SIZE + VirtualOsOutputHeader.SIZE` and a crafted `proof_facts` array where `base_block_number = 0`.
3. The sequencer includes the transaction in block 5.
4. The OS executes `check_proof_facts` with `current_block_number = 5`.
5. `current_block_number - STORED_BLOCK_HASH_BUFFER = 5 - 10 = P - 5` (field wrap).
6. `assert_nn_le(0, P - 5)` checks that `(P - 5) - 0 = P - 5` is in `[0, 2¹²⁸)`. It is not — range check fails.
7. The OS execution is invalid; no valid STARK proof can be produced for block 5.
8. The network cannot confirm block 5 or any subsequent block until the transaction is excluded.
