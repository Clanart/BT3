### Title
Non-EVM-Denom Bank Balances Not Burned on SELFDESTRUCT — Orphaned Funds Recoverable via CREATE2 Address Recreation - (File: x/evm/statedb/statedb.go)

### Summary

The `StateDB.Commit()` function in Ethermint only burns the EVM-native denomination balance when processing a self-destructed account. Non-EVM-native tokens (IBC tokens, CosmWasm bridge tokens, or any other Cosmos bank denomination) held by the destroyed contract address are explicitly skipped and left as orphaned bank balances. An attacker can recover these orphaned funds by recreating a contract at the same deterministic CREATE2 address after the original contract self-destructs.

### Finding Description

In `x/evm/statedb/statedb.go`, the `Commit()` function handles self-destructed accounts by burning only the EVM denom balance and then calling `DeleteAccount`. The code explicitly acknowledges the gap:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
``` [1](#0-0) 

`DeleteAccount` removes the auth account metadata and contract storage, but never touches the Cosmos bank module balances for non-EVM denoms:

```go
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
    // clear storage
    k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool { ... })
    // remove auth account
    k.accountKeeper.RemoveAccount(ctx, acct)
    // NOTE: bank balances for non-EVM denoms are NOT cleared
}
``` [2](#0-1) 

In the Cosmos SDK bank module, balances are stored keyed by address independently of the auth account. Removing the auth account does not zero out bank balances for other denominations. When a new account is subsequently created at the same address (via CREATE2), it inherits those orphaned bank balances.

The analog to the external report is direct: just as `yveCrv` was omitted from the `protected[]` array and could be swept, non-EVM-denom tokens are omitted from the burn loop in `Commit()` and can be recovered.

### Impact Explanation

An attacker who controls a contract holding non-EVM-denom tokens (e.g., IBC-transferred assets, tokens sent via `bank.MsgSend` to the contract address, or tokens received through a native precompile) can:

1. Self-destruct the contract (triggering `SELFDESTRUCT` or `SELFDESTRUCT6780` within the same transaction under EIP-6780).
2. Observe that `DeleteAccount` removes the auth account but leaves non-EVM bank balances intact.
3. Redeploy a new contract at the identical CREATE2 address.
4. The new contract's account now has access to the orphaned non-EVM-denom bank balances.

This constitutes unauthorized recovery of Cosmos bank funds that should have been burned or redistributed upon contract destruction. The impact matches: **unauthorized theft/transfer of Cosmos bank funds through Ethermint stateDB commit logic**.

### Likelihood Explanation

- Non-EVM tokens can reach a contract address via `bank.MsgSend`, IBC transfers, or native precompile calls — all standard, unprivileged operations.
- CREATE2 address reuse is a well-known Ethereum pattern; the attacker fully controls the salt and init code.
- The self-destruct path is reachable by any unprivileged EVM transaction.
- The code comment explicitly acknowledges this gap, confirming it is a known unresolved issue in the production path.

### Recommendation

In `StateDB.Commit()`, when processing a `selfDestructed` account, iterate over **all** bank denominations held by the address (not just `s.evmDenom`) and burn or redirect them before calling `DeleteAccount`. The Cosmos SDK `BankKeeper` exposes `GetAllBalances(ctx, addr)` which returns all coins held by an address. Each non-zero balance should be burned (sent to the module account and burned) or handled according to chain policy before the auth account is removed.

```go
// Burn ALL denominations, not just evmDenom
allBalances := bankKeeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if err := bankKeeper.SendCoinsFromAccountToModule(cacheCtx, cosmosAddr, types.ModuleName, sdk.NewCoins(coin)); err != nil {
        return err
    }
    if err := bankKeeper.BurnCoins(cacheCtx, types.ModuleName, sdk.NewCoins(coin)); err != nil {
        return err
    }
}
```

### Proof of Concept

1. Deploy a factory contract `F` via CREATE2 with a known salt.
2. From `F`, deploy child contract `C` at a deterministic CREATE2 address `A`.
3. Send IBC tokens (e.g., `ibc/...` denom) to address `A` via `bank.MsgSend` from any account.
4. In the same transaction (EIP-6780 path) or a subsequent transaction, call `SELFDESTRUCT` on `C`.
5. `StateDB.Commit()` burns only the EVM denom at `A`; the IBC token balance at `A` in the bank module is untouched.
6. `DeleteAccount` removes the auth account for `A`.
7. Redeploy `C` at address `A` using the same CREATE2 salt and init code.
8. The new account at `A` can now call `bank.MsgSend` or a native precompile to transfer the orphaned IBC tokens to any address.

The attacker has recovered tokens that should have been destroyed with the contract. [3](#0-2) [4](#0-3)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
