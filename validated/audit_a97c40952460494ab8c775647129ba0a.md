### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Corruption — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to a different shard than the `shard_id` parameter. Every cryptographic check in the function passes because `verify_path` is a pure membership proof (no position/index check), `validate_chunk_proofs` is an internal-consistency check only, and the receipt proofs are validated against the `shard_id` parameter rather than `chunk.shard_id()`. The function then writes the corrupted header under `StateHeaderKey(shard_id, sync_hash)`, binding the wrong shard's state root to the target shard's DB key. Downstream calls to `set_state_part` and `apply_state_part` use that state root to validate and install trie nodes, permanently installing the wrong shard's state into the target shard's storage.

### Finding Description

**Entrypoint and data flow**

A syncing node calls `set_state_header(shard_id=0, sync_hash, header)` where `header` is received from a peer over the network. The peer controls every field of `ShardStateSyncResponseHeader`; the local node controls only `shard_id` and `sync_hash`.

**Validation steps and why each one passes with a cross-shard header**

| Step | Code | What it checks | Why it passes |
|---|---|---|---|
| 1–2 | `validate_chunk_proofs` | chunk hash self-consistency, tx root, receipts root | No shard_id field checked [1](#0-0)  |
| 3a | `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(...))` | chunk hash+height is a member of the block's merkle tree | `verify_path` is a pure membership proof; it does not check the leaf's position/index in the tree [2](#0-1)  |
| 3b | `verify_path(prev_block.chunk_headers_root, prev_chunk_proof, ...)` | same membership check for prev chunk | Same reason; attacker supplies shard 1's prev chunk and its valid proof [3](#0-2)  |
| 4 | receipt proof loop | receipts are valid for `shard_id` (the parameter) | `receipts_hash = hash(ReceiptList(shard_id, receipts))` uses the parameter, not `chunk.shard_id()`; attacker supplies real shard-0 receipt proofs from the chain [4](#0-3)  |
| 5 | `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` | state root node matches chunk's prev_state_root | `chunk_inner` is shard 1's chunk; attacker supplies shard 1's state root node, which is valid against shard 1's state root [5](#0-4)  |

**The missing guard**

There is no check of the form `chunk.shard_id() == shard_id` anywhere in `set_state_header`. [6](#0-5) 

**Storage write with wrong binding**

After all checks pass, the header is stored:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [7](#0-6) 

`StateHeaderKey(0, sync_hash)` now maps to a header whose `chunk.prev_state_root()` is shard 1's state root.

**Downstream corruption in `set_state_part`**

`set_state_part` reads the stored header and extracts `state_root` from the chunk:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// validates parts against shard-1's state root, then writes to DBCol::StateParts keyed by shard_id=0
``` [8](#0-7) 

Parts from shard 1's trie validate correctly against shard 1's state root, so they are accepted and stored under `StatePartKey(sync_hash, 0, part_id)`. `apply_state_part` then installs shard 1's trie nodes into shard 0's storage.

**Attacker-controlled data required**

All of the following are publicly available chain data, obtainable by any peer node without validator privileges:
- Shard 1's `ShardChunk` and its merkle proof in the block's `chunk_headers_root`
- Shard 1's prev chunk header and its merkle proof
- Shard 0's incoming receipt proofs (for the `shard_id` parameter check)
- Shard 1's `StateRootNode`

### Impact Explanation

A syncing node that accepts this header will install shard 1's entire trie into shard 0's storage. The node will subsequently fail to process any blocks for shard 0 (wrong state root), silently diverge from the canonical chain, or crash. The corruption is written to `DBCol::StateHeaders` and `DBCol::StateParts` and persists across restarts.

### Likelihood Explanation

Any peer that can respond to a state sync header request can trigger this. State sync is performed by nodes that are not tracking a shard and need to catch up; the provider does not need to be a validator. The attack requires constructing a valid-looking cross-shard header from real chain data, which is straightforward for any node that has synced the chain.

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} != expected shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, consider using `verify_path_with_index` (which also checks the leaf's position in the tree) instead of `verify_path` for the `chunk_headers_root` check, to enforce that the chunk is at the index corresponding to `shard_id`. [9](#0-8) 

### Proof of Concept

A cargo test would:
1. Build a two-shard genesis and produce a few blocks.
2. Obtain a valid `ShardStateSyncResponseHeader` for `shard_id=1` via `get_state_response_header(1, sync_hash)`.
3. Call `set_state_header(0, sync_hash, header_for_shard_1)` on a second client.
4. Assert the call returns `Ok(())` (demonstrating the missing guard).
5. Read `DBCol::StateHeaders` at key `StateHeaderKey(0, sync_hash)` and assert `header.chunk_prev_state_root() != expected_shard_0_state_root`.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
