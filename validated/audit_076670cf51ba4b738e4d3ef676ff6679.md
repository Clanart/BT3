### Title
Integer Overflow Divergence in Relative Timelock Arithmetic via Large Condition Values — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary
`check_time_locks` accepts a caller-controlled `nowrap: bool` parameter that switches between `wrapping_add` and `saturating_add` when evaluating relative timelock conditions (`ASSERT_HEIGHT_RELATIVE`, `ASSERT_SECONDS_RELATIVE`, `ASSERT_BEFORE_HEIGHT_RELATIVE`, `ASSERT_BEFORE_SECONDS_RELATIVE`). For maximum-width values (`u32::MAX` for height, `u64::MAX` for seconds), the two arithmetic modes produce **opposite** accept/reject outcomes for the same spend bundle. Any node calling `check_time_locks` with a different `nowrap` value than its peers will reach a different consensus decision, enabling deterministic chain split via an unprivileged spend.

### Finding Description
`check_time_locks` branches on `nowrap` for every relative condition:

```rust
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { return Err(...AssertHeightRelativeFailed); }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { return Err(...AssertHeightRelativeFailed); }
}
``` [1](#0-0) 

The same pattern repeats for `seconds_relative`, `before_height_relative`, and `before_seconds_relative`. [2](#0-1) 

The upstream condition parser (`parse_args` in `conditions.rs`) accepts any 4-byte unsigned integer for height conditions and any 8-byte unsigned integer for seconds conditions via `sanitize_uint`. The maximum accepted values — `u32::MAX` (0xffff_ffff) for height, `u64::MAX` (0xffff_ffff_ffff_ffff) for seconds — are valid and pass through to `check_time_locks` without rejection. [3](#0-2) [4](#0-3) 

The Python binding `py_check_time_locks` forwards `nowrap` directly from the caller with no consensus-constant derivation: [5](#0-4) 

### Impact Explanation
The test suite explicitly documents the divergence for all four relative conditions:

| Condition | Value | `nowrap=true` | `nowrap=false` |
|---|---|---|---|
| `height_relative` | `0xffff_ffff` | **Err** (saturates → u32::MAX, rejected) | **Ok** (wraps → 99, accepted) |
| `seconds_relative` | `0xffff_ffff_ffff_ffff` | **Err** | **Ok** |
| `before_height_relative` | `0xffff_ffff` | **Ok** | **Err** |
| `before_seconds_relative` | `0xffff_ffff_ffff_ffff` | **Ok** | **Err** | [6](#0-5) [7](#0-6) 

A spend bundle containing `ASSERT_HEIGHT_RELATIVE = u32::MAX` is simultaneously valid (accepted) on nodes calling `check_time_locks(..., nowrap=false)` and invalid (rejected) on nodes calling `check_time_locks(..., nowrap=true)`. Because `nowrap` is not derived from any consensus constant visible in chia_rs, any implementation disagreement or version skew in how callers set this flag produces a deterministic, reproducible chain split for the same block.

The impact matches: **High — valid unprivileged spend bundle triggers deterministic consensus divergence**.

### Likelihood Explanation
The attacker-controlled entry path is fully reachable by any unprivileged coin owner:

1. Craft a CLVM puzzle that emits `(ASSERT_HEIGHT_RELATIVE . 0xffffffff)` as a condition.
2. `sanitize_uint` with `max_size=4` accepts `0xffffffff` as a valid 4-byte positive integer and returns `SanitizedUint::Ok(0xffffffff)`.
3. The parsed `u32::MAX` value is stored in `spend.height_relative` and passed to `check_time_locks`.
4. Nodes with `nowrap=false` accept; nodes with `nowrap=true` reject.

No privileged role, leaked key, or network-level attack is required. The only precondition is that the network contains nodes with differing `nowrap` settings — a realistic scenario during any soft-fork transition or version skew.

### Recommendation
1. **Remove the `nowrap` parameter** from `check_time_locks` and unconditionally use `saturating_add`. Saturating semantics are strictly safer: a coin confirmed at height `H` with `ASSERT_HEIGHT_RELATIVE = u32::MAX` should never be spendable before block `u32::MAX`, which saturating arithmetic correctly enforces.
2. If the wrapping behavior must be preserved for historical block replay, derive the arithmetic mode exclusively from a consensus constant (e.g., a soft-fork activation height stored in `ConsensusConstants`) rather than from a caller-supplied boolean, so all nodes compute the same value for the same block.
3. Add a consensus-layer guard: reject any spend bundle containing a relative timelock condition whose value, when added to the coin's confirmed height/timestamp, would overflow the field width, before the value reaches `check_time_locks`.

pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    // strip the leading zero byte if there is one
    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }

    Ok(SanitizedUint::Ok(u64_from_bytes(buf)))
