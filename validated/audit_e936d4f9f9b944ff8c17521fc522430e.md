### Title
Missing Class Hash Existence Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS does not validate that the supplied class hash corresponds to a declared contract class before committing the state update. This is the direct analog of M-08's "missing prerequisite check before a state update": just as Rubicon's approval functions wrote a new allowance without first resetting to zero (causing USDT to revert), the OS here writes a new class hash without first confirming it is declared — permanently bricking any contract that supplies an undeclared hash and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the requested class hash from the syscall request and immediately commits it to `contract_state_changes` with no existence check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The inline TODO comment is the OS developers' own acknowledgment that this check is absent. Any felt value — including `0` or an arbitrary undeclared hash — is accepted and written as the contract's new class hash. Once committed, the contract has no resolvable entry points; every subsequent call to it will fail at class lookup. The state transition is irreversible: there is no mechanism to undo a committed `replace_class` after the block is proven.

Compare this with `execute_declare_transaction`, which enforces `prev_value=0` to guarantee a class is declared at most once:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

`execute_replace_class` has no symmetric guard on the *new* value side.

---

### Impact Explanation

If a contract's class hash is replaced with an undeclared value, the contract becomes permanently inaccessible. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are permanently frozen with no recovery path. This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The syscall is reachable by any contract that exposes an upgrade path callable by an unprivileged sender (e.g., a multisig with a compromised signer, a governance contract with a flash-loan-exploitable quorum, or a contract whose `replace_class` guard has a logic bug). The OS is the last line of defense; without the existence check, a single malformed `replace_class` call finalised in a proven block permanently destroys the contract. The explicit TODO confirms the developers are aware the guard is missing, raising the probability that it will be triggered before the fix lands.

---

### Recommendation

Before committing the new class hash, verify it exists in `contract_class_changes` (or a canonical class registry). The check should mirror the pattern already used in `execute_declare_transaction`:

```cairo
// Assert the class hash is declared (exists in contract_class_changes).
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This ensures `replace_class` can only target a class that has already been successfully declared, eliminating the ability to brick a contract with an arbitrary felt.

---

### Proof of Concept

1. Deploy contract `Vault` holding 1000 STRK. `Vault` exposes `upgrade(new_class_hash)` with a flawed access-control check (e.g., any caller with a valid signature, or a governance path exploitable via flash loan).
2. Attacker calls `Vault.upgrade(0xdeadbeef)` where `0xdeadbeef` is not declared.
3. Inside `Vault`, the Sierra runtime issues `replace_class(0xdeadbeef)`.
4. The OS executes `execute_replace_class`; the TODO-guarded check is absent, so `dict_update` writes `class_hash=0xdeadbeef` into `contract_state_changes`.
5. The block is proven and finalised on L1.
6. All subsequent calls to `Vault` fail: the OS resolves `class_hash=0xdeadbeef`, finds no entry in `contract_class_changes`, and reverts.
7. The 1000 STRK are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2)

    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
