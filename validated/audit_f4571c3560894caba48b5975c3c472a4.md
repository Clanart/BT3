Audit Report

## Title
Router's Accumulated ETH Balance Is Silently Consumed by Any Subsequent WETH Swap, Causing Prior ETH Depositors to Lose Funds - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay` function in `PeripheryPayments.sol` uses `address(this).balance` unconditionally when settling WETH payments, without verifying that the ETH belongs to the current transaction's sender. ETH left in the router by a prior user who called a `payable` entry point (e.g., `exactInputSingle`) without atomically calling `refundETH` is silently consumed by the next WETH swap caller, who pays nothing from their own balance. The original depositor loses their stranded ETH with no revert, no event, and no recourse.

## Finding Description
`exactInputSingle` is declared `external payable`, so a user can attach `msg.value` directly without going through `multicall`. [1](#0-0) 

When `pay` is called during the swap callback, it reads `address(this).balance` and, if that balance covers the required `value`, wraps and forwards the ETH without ever pulling from the actual payer: [2](#0-1) 

The `receive()` guard only blocks plain ETH transfers; it does not prevent `msg.value` from accumulating via payable function calls: [3](#0-2) 

The intended recovery path is `multicall([exactInputSingle(...), refundETH()])`, as shown in the test suite: [4](#0-3) 

However, `refundETH` is a separate call. A user who calls `exactInputSingle{value: X}(amountIn: Y)` directly (Y < X) leaves `X - Y` ETH stranded. Any subsequent WETH swap caller with `amountIn <= X - Y` triggers branch 1 of `pay`, consuming the stranded ETH and paying nothing from their own WETH balance.

The `else if (nativeBalance > 0)` branch (lines 78–81) makes the drain unconditional even for partial balances: even 1 wei of stranded ETH is consumed by the next WETH swap, with the shortfall pulled from the new caller's WETH. This branch is absent from the Uniswap v3 reference implementation.

The same `pay` function is inherited by `MetricOmmPoolLiquidityAdder`, exposing the same drain path for WETH liquidity additions: [5](#0-4) 

## Impact Explanation
Direct, silent, irreversible loss of user principal (ETH). A user who sends excess ETH with a WETH swap and does not atomically call `refundETH` loses the unspent ETH to the next WETH swap caller. The loss equals the stranded ETH amount — up to the full `msg.value` if the user sent ETH with a non-WETH swap by mistake. This is a Critical/High direct loss of user funds meeting Sherlock contest thresholds.

## Likelihood Explanation
The trigger requires no privilege: any unprivileged user can call `exactInputSingle` directly with `msg.value` (a natural pattern for ETH-in swaps), and any subsequent WETH swapper drains the stranded ETH. The window between the first call and a `refundETH` call is exploitable by front-runners or any WETH swapper in the same or a later block. The `else if (nativeBalance > 0)` branch makes the drain unconditional once any ETH accumulates.

## Recommendation
1. **Track per-transaction ETH via transient storage**: At each `payable` entry point, record `msg.value` in a transient slot. In `pay`, consume only from that recorded value, not from `address(this).balance`. This prevents cross-transaction ETH leakage.
2. **Remove the partial-ETH branch**: Revert to Uniswap v3 behavior — if `address(this).balance < value`, pull the full amount from the payer via `safeTransferFrom`. This eliminates the novel drain vector while preserving the ETH-wrapping convenience for the `nativeBalance >= value` case.
3. **Enforce the `multicall` + `refundETH` pattern**: If the partial-ETH branch is kept, add a guard that reverts if `address(this).balance` exceeds `msg.value` at call time, ensuring only the current transaction's ETH is used.

## Proof of Concept
```solidity
// Step 1: User A swaps WETH→token1, sends 1 ETH, amountIn = 0.5 ETH
// User A calls exactInputSingle directly (not via multicall), does NOT call refundETH
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    pool: pool, tokenIn: WETH, tokenOut: token1,
    zeroForOne: true, amountIn: 0.5 ether, amountOutMinimum: 0,
    recipient: userA, deadline: ..., priceLimitX64: 0, extensionData: ""
}));
// pay(): nativeBalance (1 ETH) >= value (0.5 ETH) → branch 1
// Wraps 0.5 ETH, sends WETH to pool. 0.5 ETH remains in router.
assert(address(router).balance == 0.5 ether);

// Step 2: User B swaps WETH→token1, amountIn = 0.5 ETH, sends 0 ETH
// User B has approved router for WETH but pay() never calls safeTransferFrom
router.exactInputSingle(ExactInputSingleParams({
    pool: pool, tokenIn: WETH, tokenOut: token1,
    zeroForOne: true, amountIn: 0.5 ether, amountOutMinimum: 0,
    recipient: userB, deadline: ..., priceLimitX64: 0, extensionData: ""
}));
// pay(): nativeBalance (0.5 ETH) >= value (0.5 ETH) → branch 1
// Router wraps User A's 0.5 ETH and sends WETH to pool; User B's WETH never pulled
assert(address(router).balance == 0);
// User A lost 0.5 ETH; User B received the swap for free
```
