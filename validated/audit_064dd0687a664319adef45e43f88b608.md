Audit Report

## Title
Unrestricted `sweepToken` and `unwrapWETH9` Allow Any Caller to Redirect Router Token Balances to Arbitrary Address — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`sweepToken` and `unwrapWETH9` in `PeripheryPayments.sol` are `public payable` and accept a caller-supplied `recipient` with no `msg.sender` restriction. Any address can invoke either function at any time to redirect the entire ERC-20 or WETH balance held by `MetricOmmSimpleRouter` or `MetricOmmPoolLiquidityAdder` to an attacker-controlled address. The most direct exploit path is the two-transaction WETH-unwrap pattern, where a user's swap output sitting in the router between transactions is front-run and stolen before the user's `unwrapWETH9` call executes.

## Finding Description
`PeripheryPayments` exposes two unguarded public entry points:

```solidity
// L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // recipient is attacker-supplied
    }
}

// L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);  // recipient is attacker-supplied
    }
}
```

Neither function checks `msg.sender` nor constrains `recipient`. By contrast, `refundETH` (L58-63) correctly hard-codes `msg.sender` as the destination, demonstrating the intended pattern was known but not applied consistently.

**Two-transaction exploit path (primary):**
A user who does not batch via `multicall` may call `exactInputSingle` with `recipient = address(router)` to receive WETH output in the router, intending to follow up with `unwrapWETH9(minAmount, user)`. Between these two transactions, an attacker calls `unwrapWETH9(0, attacker)`, draining the entire WETH balance. The user's subsequent call either reverts (if `amountMinimum > 0`) or silently sends nothing.

**Multi-hop `exactInput` intermediate token path (secondary):**
In `exactInput`, intermediate hops send output to `address(this)` (L106). While this is within a single transaction and consumed in the next hop, any failed multicall step or partial revert that leaves tokens in the router exposes them to `sweepToken` in a subsequent call.

The broken invariant is: tokens transiently held by the router on behalf of a user must only be retrievable by that user, not by an arbitrary third party.

## Impact Explanation
Direct loss of user principal. In the two-transaction WETH-unwrap scenario, the user's full swap output (WETH) is stolen with no recovery path. In the ERC-20 sweep scenario, any token balance held by the router is equally exposed. This constitutes a Critical/High direct loss of user funds, satisfying the allowed impact gate for direct loss of user principal above Sherlock thresholds.

## Likelihood Explanation
The two-transaction pattern is a realistic user behavior; not all users batch via `multicall`. MEV bots routinely scan for non-zero token balances in periphery contracts and sweep them within the same block. No special role, permit, or prior approval is required — a single external call with `amountMinimum = 0` suffices. The attack is repeatable and fully automated.

## Recommendation
Remove the `recipient` parameter from both functions and hard-code `msg.sender` as the destination, matching the pattern already used by `refundETH`:

```solidity
function unwrapWETH9(uint256 amountMinimum) public payable {
    address recipient = msg.sender;
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) { IWETH9(WETH).withdraw(balanceWETH); _transferETH(recipient, balanceWETH); }
}

function sweepToken(address token, uint256 amountMinimum) public payable {
    address recipient = msg.sender;
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) IERC20(token).safeTransfer(recipient, balanceToken);
}
```

## Proof of Concept
```
1. Alice calls exactInputSingle{value: 1 ether}(
       pool=WETH/USDC, tokenIn=WETH, amountIn=1e18,
       recipient=address(router)   // intends to follow up with unwrapWETH9
   )
   → Pool receives 1 WETH; router holds WETH output from pool.

2. Bob (attacker) observes Alice's pending Tx 2
   (unwrapWETH9(1e18, alice)) in the mempool.

3. Bob front-runs with:
       router.unwrapWETH9(0, bob)
   → Router's entire WETH balance is unwrapped and sent to Bob.

4. Alice's Tx 2 executes; router WETH balance is now 0.
   If amountMinimum > 0: reverts with InsufficientWETH.
   If amountMinimum == 0: silently sends nothing.
   Alice loses her full swap output.
```

Foundry test outline:
- Deploy router with mock WETH and pool.
- Call `exactInputSingle` with `recipient = address(router)`, confirm WETH lands in router.
- From a second address, call `unwrapWETH9(0, attacker)`, assert attacker receives full balance.
- Assert original caller receives nothing on their subsequent `unwrapWETH9` call.
