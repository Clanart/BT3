### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS does not verify that the new class hash provided by a contract is actually declared (i.e., exists in `contract_class_changes`). This is directly analogous to M-14: just as `changeController` accepted a controller whose `vaultFactory` back-reference was never validated, `execute_replace_class` accepts an arbitrary class hash with no back-reference check against the declared class registry. Any contract can replace its class with an undeclared hash, rendering itself permanently uncallable and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS updates `contract_state_changes` with the caller-supplied `class_hash` without any check that this hash exists in `contract_class_changes` (the declared class registry). The developers themselves acknowledge this gap with an explicit TODO: [1](#0-0) 

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

After this update is committed, any subsequent call to the contract reaches `execute_entry_point`, which performs a two-step lookup:

**Step 1** — read compiled class hash from the class registry: [2](#0-1) 

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

If `execution_context.class_hash` is not declared, `dict_read` returns `0` (the default for an uninitialized dict entry). `find_element` then searches for a compiled class with hash `0`. If none exists, `find_element` raises a Cairo assertion failure, crashing the OS prover for that block.

Even if the sequencer's blockifier defensively reverts every subsequent call to the broken contract (preventing the OS crash path), the contract's committed on-chain state permanently carries the invalid class hash. The contract becomes uncallable in all future blocks, and any funds it holds are irrecoverably frozen.

The missing validation that **should** exist is analogous to the check enforced in `execute_declare_transaction`, which uses `prev_value=0` to guarantee a class is only registered once and is always backed by a real compiled class: [3](#0-2) 

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract that replaces its class with an undeclared hash becomes permanently uncallable. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irrecoverably frozen. There is no recovery path: the only way to fix the class hash would be to call `replace_class` again, but calling the contract at all is impossible once the class is invalid.

---

### Likelihood Explanation

- The `replace_class` syscall is callable by **any contract**, with no privilege requirement. An unprivileged user can deploy a contract and invoke `replace_class` with an arbitrary felt value as the class hash.
- The OS explicitly acknowledges the missing check via the TODO comment dated 2026-01-01, confirming this is a known, unmitigated gap in the current production code.
- The attack requires no special sequencer cooperation: if the blockifier also lacks this check (consistent with the OS TODO), the transaction is included normally. Even if the blockifier reverts subsequent calls, the corrupted state is already committed.

---

### Recommendation

In `execute_replace_class`, add a validation that the supplied `class_hash` exists in `contract_class_changes` before updating the contract state. Concretely, perform a `dict_read` on `contract_class_changes` keyed by `request.class_hash` and assert the result is non-zero (i.e., a compiled class hash has been registered for it). This mirrors the existing pattern in `execute_declare_transaction` and resolves the acknowledged TODO.

---

### Proof of Concept

1. Attacker deploys `MaliciousContract` containing an external function that calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is never declared.
2. Attacker (or any user) sends an invoke transaction calling that function.
3. The OS processes `execute_replace_class`: no declared-class check is performed; `contract_state_changes` is updated with `class_hash = 0xdeadbeef`.
4. The block is proven successfully — the OS accepts the invalid state transition.
5. In all subsequent blocks, any call to `MaliciousContract` causes `execute_entry_point` to call `dict_read(contract_class_changes, 0xdeadbeef)` → returns `0`, then `find_element(..., key=0)` → assertion failure (OS crash) or the blockifier reverts the call.
6. All funds held by `MaliciousContract` are permanently frozen with no recovery mechanism.
