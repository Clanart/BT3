Audit Report

## Title
`PeripheryPayments.pay()` consumes entire router native ETH balance for WETH settlement, enabling theft of residual ETH — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` — the router's total native ETH balance — when settling a WETH-input swap, rather than scoping to the current call's `msg.value`. Because all router entry points are `payable` and the `receive()` guard only blocks direct non-WETH ETH sends (not ETH attached to payable calls), native ETH can accumulate on the router. A subsequent caller routing through any factory-registered WETH pool can consume that accumulated ETH to fund their own swap, stealing the prior caller's funds.

## Finding Description
In `PeripheryPayments.pay()` (lines 73–84), when `token == WETH` and `payer != address(this)`, the function reads the router's full native balance:

```solidity
uint256 nativeBalance = address(this).balance;   // entire router balance, not msg.value
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [1](#0-0) 

This branch is reachable through `exactInput` and `exactInputSingle` whenever `params.tokens[0] == WETH` (or `params.tokenIn == WETH`). In `exactInput`, hop 0 sets the payer to `msg.sender` and the token to `params.tokens[i]`:

```solidity
_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
``` [2](#0-1) 

All entry points (`exactInput`, `exactInputSingle`, `exactOutput`, `exactOutputSingle`, `multicall`) are `payable`, so ETH attached to any call is retained on the router. The `receive()` guard only blocks direct ETH sends from non-WETH addresses; it does not prevent ETH accumulation from payable function calls: [3](#0-2) 

The factory validation in `_setNextCallbackContext` → `_requireFactoryPool` only checks that the pool is registered; it does not prevent an attacker from using a legitimate WETH pool to trigger the vulnerable branch: [4](#0-3) 

**Exploit flow:**
1. Alice calls `exactInput(tokens=[WETH, USDC], amountIn=0.5e18)` with `msg.value=1e18`. `pay()` wraps 0.5 ETH and pays the pool. Alice receives USDC. The remaining 0.5 ETH stays on the router because Alice did not call `refundETH()`.
2. Attacker calls `exactInput(tokens=[WETH, USDC], amountIn=0.5e18)` with `msg.value=0` and zero WETH approval. `pay()` reads `address(this).balance = 0.5e18`, wraps Alice's ETH, and pays the pool. Attacker receives USDC. Alice's 0.5 ETH is permanently lost.

The `_validatePath` function only enforces array-length consistency and does not verify that `params.tokens[i]` matches the actual pool token, so the attacker can freely supply `params.tokens[0] = WETH` for any factory-registered WETH pool: [5](#0-4) 

## Impact Explanation
Direct loss of user principal. Any native ETH stranded on the router — from any payable call where `refundETH()` was not invoked — is consumed to settle a subsequent caller's WETH swap obligation. The attacker receives output tokens without providing any ETH or WETH. The victim's ETH is permanently lost. This constitutes a critical direct loss of user funds above Sherlock thresholds.

## Likelihood Explanation
The `multicall` pattern (compose `exactInput` + `refundETH`) is the safe path, but users calling `exactInput` directly with `msg.value > amountIn` — or sending ETH with any payable call while routing non-WETH tokens — will routinely leave residual ETH. An attacker can monitor `address(router).balance` on-chain and exploit it atomically in the same block. No privileged access, malicious pool, or non-standard token behavior is required; only a factory-registered WETH pool is needed.

## Recommendation
Scope the native ETH budget to the current call's `msg.value`. One approach: snapshot `address(this).balance - msg.value` at entry and treat only the delta attributable to `msg.value` as spendable. Alternatively, pass a decreasing ETH budget counter into `pay()` and consume only from that budget, falling back to `safeTransferFrom` for any remainder. The `refundETH()` helper alone is insufficient because it relies on callers to invoke it correctly.

// Foundry fork test outline
function test_stealResidualETH() public {
    // 1. Alice sends msg.value=1e18, amountIn=0.5e18 through WETH→USDC pool
    vm.prank(alice);
    router.exactInput{value: 1e18}(ExactInputParams({
        tokens: [WETH, USDC],
        pools: [wethUsdcPool],
        extensionDatas: [bytes("")],
        zeroForOneBitMap: ...,
        amountIn: 0.5e18,
        amountOutMinimum: 0,
        recipient: alice,
        deadline: block.timestamp + 1
    }));
    // Alice forgets refundETH(); 0.5 ETH remains on router
    assertEq(address(router).balance, 0.5e18);

    // 2. Attacker sends msg.value=0, zero WETH approval
    vm.prank(attacker);
    router.exactInput{value: 0}(ExactInputParams({
        tokens: [WETH, USDC],
        pools: [wethUsdcPool],
        extensionDatas: [bytes("")],
        zeroForOneBitMap: ...,
        amountIn: 0.5e18,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1
    }));
    // Attacker receives USDC; Alice's 0.5 ETH is consumed
    assertEq(address(router).balance, 0);
    assertGt(IERC20(USDC).balanceOf(attacker), 0);
}
```
