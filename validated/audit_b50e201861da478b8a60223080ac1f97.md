### Title
Wrong Epoch's Shard Layout Used to Index Previous Block's Chunks in `get_state_response_part` at Resharding Boundary ŌĆö (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` resolves `shard_index` using the **new** epoch's `shard_layout`, then uses that index to read `prev_block.chunks()`, which is ordered by the **previous** epoch's layout. `compute_state_response_header` correctly uses `prev_shard_layout` for the same lookup. At a resharding boundary the two layouts diverge: unchanged shards whose position shifted receive a `state_root` extracted from the wrong chunk slot, producing state parts that fail `validate_state_part` on the receiving node and permanently breaking state sync for every shard whose index moved.

---

---

### Impact Explanation

Every node attempting to state-sync into the epoch immediately following a resharding event will fail to apply state parts for all shards whose `shard_index` changed. Because the `state_root` embedded in the served part does not match the `state_root` committed in the `ShardStateSyncResponseHeader`, `validate_state_part` rejects every part. The syncing node cannot advance past the resharding epoch boundary. This is a **Critical** correctness break: it renders state sync non-functional at the exact epoch transition where it is most needed (new nodes joining after a shard split).

---

### Likelihood Explanation

The bug is latent until the first dynamic resharding event on a network running `ShardLayoutV3`. Once resharding occurs, every state-syncing node is affected for every shard whose index shifted. No special attacker action is required; normal state sync triggers the path. The condition is deterministic and reproducible on every resharding epoch boundary.

---

### Recommendation

In `get_state_response_part`, mirror the two-layout pattern used in `compute_state_response_header`: fetch the previous block's epoch ID, obtain `prev_shard_layout` from it, and use `prev_shard_layout.get_shard_index(shard_id)` to index into `prev_block.chunks()`. The corrected lines 305ŌĆō316 should read:

```rust
let prev_block   = self.chain_store.get_block(header.prev_hash())?;
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(prev_epoch_id)?;
let shard_index  = prev_shard_layout.get_shard_index(shard_id)?;
let state_root   = prev_block.chunks().get(shard_index)
                       .ok_or(Error::InvalidShardId(shard_id))?
                       .prev_state_root();
```

This aligns the index computation with `compute_state_response_header` and restores the invariant that both functions read the same chunk slot for the same `shard_id`.

---

### Proof of Concept

1. Configure a testnet with `ShardLayoutV3` dynamic resharding enabled.
2. Trigger a shard split (e.g., by exceeding the memory threshold for one shard).
3. At the resharding epoch boundary, start a new node and initiate state sync.
4. The new node requests the `ShardStateSyncResponseHeader` for any shard whose index shifted (e.g., shard_id=2 in the example above). The header is computed correctly via `compute_state_response_header` using `prev_shard_layout`.
5. The new node then requests state parts via `get_state_response_part`. The serving node computes `shard_index` using the new epoch's layout, reads `prev_block.chunks().get(wrong_index).prev_state_root()`, and generates parts against the wrong root.
6. `set_state_part` on the receiving node calls `validate_state_part` with the header's correct root; every part fails validation.
7. State sync loops indefinitely, never completing for the affected shards.
