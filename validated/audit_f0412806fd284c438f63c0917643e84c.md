### Title
`removeCustomToken` Deletes Routing State While In-Flight Transfers Exist, Permanently Locking User Funds ŌĆö (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`removeCustomToken()` atomically deletes `customMinters[tokenAddress]` and `isBridgeToken[tokenAddress]` with no guard against in-flight transfers. `finTransfer()` routes token delivery by reading those same mappings at execution time. If the mappings are cleared between a user's `initTransfer` (which burned the tokens via the custom minter, leaving the bridge with zero balance) and the corresponding `finTransfer`, the routing falls through to an unconditional `IERC20.safeTransfer` call against a bridge contract that holds no balance of that token. The call reverts, the nonce is already consumed, and the user's funds are permanently unclaimable.

---

### Finding Description

`initTransfer` for a custom-minter token transfers the user's tokens to the custom minter and burns them there:

```solidity
// OmniBridge.sol lines 394-403
if (customMinters[tokenAddress] != address(0)) {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender, customMinters[tokenAddress], amount
    );
    ICustomMinter(customMinters[tokenAddress]).burn(tokenAddress, amount);
}
```

After this call the bridge contract holds **zero** balance of `tokenAddress`. The user's tokens are gone from EVM; the NEAR side is expected to mint the equivalent.

`removeCustomToken` can be called at any time by `DEFAULT_ADMIN_ROLE`:

```solidity
// OmniBridge.sol lines 120-127
function removeCustomToken(address tokenAddress) external onlyRole(DEFAULT_ADMIN_ROLE) {
    delete isBridgeToken[tokenAddress];
    delete nearToEthToken[ethToNearToken[tokenAddress]];
    delete ethToNearToken[tokenAddress];
    delete customMinters[tokenAddress];
}
```

There is no check for outstanding nonces, pending transfers, or any balance held by the bridge. The deletion is unconditional.

`finTransfer` routes delivery by reading the current state of those mappings:

```solidity
// OmniBridge.sol lines 331-355
} else if (customMinters[payload.tokenAddress] != address(0)) {   // (A) now address(0)
    ICustomMinter(customMinters[payload.tokenAddress]).mint(...);
} else if (isBridgeToken[payload.tokenAddress]) {                  // (B) now false
    IBridgeToken(payload.tokenAddress).mint(...);
} else {
    IERC20(payload.tokenAddress).safeTransfer(                     // (C) reached
        payload.recipient, payload.amount
    );
}
```

With both mappings cleared, branch (C) executes. The bridge has no balance ŌåÆ `safeTransfer` reverts. The nonce was already marked used at line 287 (`completedTransfers[payload.destinationNonce] = true`) before the routing block, so the signed payload cannot be replayed. The user's funds are permanently locked.

---

### Impact Explanation

**Critical ŌĆö Permanent, irrecoverable lock of user funds.**

The user burned real ERC-20 tokens on EVM via `initTransfer`. The NEAR side will have already processed the `InitTransfer` event and burned/locked the corresponding NEAR tokens. Neither side can recover: the EVM nonce is consumed and the routing branch that would mint/transfer tokens no longer executes. There is no recovery path in the contract ŌĆö no admin rescue function, no refund mechanism, no way to re-route a consumed nonce.

---

### Likelihood Explanation

`removeCustomToken` is a live admin function with no in-flight-transfer guard. An admin performing routine token management (e.g., replacing a custom minter, deprecating a token integration) could call it while transfers are in flight without any on-chain warning. The window between `initTransfer` and `finTransfer` spans multiple blocks (NEAR finality + MPC signing latency), making the race condition realistic. The eNEAR custom token is a documented production use case, making this a realistic scenario.

---

### Recommendation

Mirror the fix recommended in the external report: either

1. **Remove `removeCustomToken` entirely** and make the custom-token registration immutable (analogous to making `bribeVault` immutable), or
2. **Add a drain-before-remove guard**: require that no in-flight transfers reference the token before deletion. Because the EVM contract has no on-chain pending-transfer registry, the practical mitigation is to require the admin to pause `PAUSED_INIT_TRANSFER` and `PAUSED_FIN_TRANSFER`, wait for all in-flight transfers to settle (verified off-chain), and only then call `removeCustomToken`.

A minimal code-level guard would be to add a `currentOriginNonce` snapshot check or require an explicit "no pending transfers" attestation, but the safest fix is to remove the setter entirely and use contract upgrades for token-registration changes.

---

2. User calls `initTransfer(eNEAR, 1000e18, 0, 0, "alice.near", "")`. Tokens are transferred to `eNearProxy` and burned. Bridge holds 0 eNEAR. `currentOriginNonce` becomes N.

4. Admin calls `removeCustomToken(eNEAR)`. `customMinters[eNEAR]` ŌåÆ `address(0)`, `isBridgeToken[eNEAR]` ŌåÆ `false`.
