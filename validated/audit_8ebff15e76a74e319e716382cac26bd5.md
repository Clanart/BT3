Audit Report

## Title
Router address substitution breaks SwapAllowlistExtension per-user gating — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the address passed by the pool's `swap` function. When `MetricOmmSimpleRouter` is used, the pool receives the router as `msg.sender` and unconditionally forwards it as `sender` to the extension. The original user's address is never visible to the extension, making per-user allowlisting non-functional for all router-mediated swaps.

## Finding Description
`MetricOmmPool.swap` unconditionally passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // always the immediate caller — router when routed
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool address, `sender` = router address. The user's address is stored only in transient callback context for payment purposes (`_setNextCallbackContext`) and is never forwarded to `pool.swap` as the initiating address. A pool admin who calls `setAllowedToSwap(pool, userAddress, true)` to permit a specific user will find that `allowedSwapper[pool][routerAddress]` is `false`, causing all router-mediated swaps to revert with `NotAllowedToSwap` even for explicitly allowlisted users.

## Impact Explanation
Complete DoS of the router-based swap path for all users of any pool that deploys `SwapAllowlistExtension` with a non-open allowlist. Allowlisted users cannot execute swaps through `MetricOmmSimpleRouter`, losing access to slippage protection, multi-hop routing, and WETH wrapping. The only workarounds are calling `pool.swap()` directly (bypassing router safety features) or allowlisting the router address (which defeats per-user gating entirely). This constitutes broken core swap functionality as defined in the contest scope.

## Likelihood Explanation
Any pool that deploys `SwapAllowlistExtension` with per-user allowlisting — the primary use case of the extension — and expects users to interact via `MetricOmmSimpleRouter` will trigger this. The extension is a production periphery contract explicitly designed for this purpose. No special attacker capability is required; any allowlisted user attempting a normal router swap will be blocked.

## Recommendation
Either: (1) the pool should expose the original initiator's address separately from `msg.sender` (e.g., via a dedicated `initiator` parameter or transient storage); (2) the router should pass the original user's address through `extensionData` and `SwapAllowlistExtension` should decode and check it; or (3) `SwapAllowlistExtension` documentation and `setAllowedToSwap` should clearly state that when a router is used, the router's address must be allowlisted rather than individual user addresses.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` attached.
2. Pool admin calls `setAllowedToSwap(pool, userAddress, true)` — user is allowlisted, router is not.
3. User calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(recipient, zeroForOne, amount, ...)` — pool passes `router_address` as `sender` to `_beforeSwap`.
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router_address]` → `false` → reverts `NotAllowedToSwap`.
6. User is blocked despite being explicitly allowlisted. Foundry test: deploy pool + extension, configure allowlist for user EOA only, call `router.exactInputSingle` from that EOA, assert revert `NotAllowedToSwap`.
