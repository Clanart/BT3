### Title
`MerkleBlob::upsert` Missing Duplicate-Hash Guard Corrupts `leaf_hash_to_index` Cache — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::upsert` does not check whether `new_hash` already belongs to a different existing leaf before executing its update path. This allows `leaf_hash_to_index` to be silently overwritten, causing `get_node_by_hash` to resolve a live leaf's hash to a different leaf's data, and leaving the blob in a state where two physical leaves share the same hash.

### Finding Description

`insert` guards against duplicate hashes at lines 372–374: [1](#0-0) 

`upsert`'s update path (key already present) has no equivalent guard: [2](#0-1) 

Exact execution trace for `upsert(key_A, value_A', hash_B)` when leaf A (`key_A`, `hash_A`) and leaf B (`key_B`, `hash_B`) both exist:

1. **`remove_leaf(&leaf_A)`** (line 798): removes `hash_A → idx_A` and `key_A → idx_A` from the cache, marks `idx_A` free. [3](#0-2) 

2. **`insert_entry_to_blob(idx_A, block_with_hash_B)`** (line 803): writes the updated block to the blob, then calls `add_leaf(idx_A, leaf{hash_B})`. [4](#0-3) 

3. **`add_leaf`** (line 192): `leaf_hash_to_index.insert(hash_B, idx_A)` silently **overwrites** the existing `hash_B → idx_B` entry with `hash_B → idx_A`. [5](#0-4) 

Post-upsert state:
- `leaf_hash_to_index = { hash_B: idx_A }` — leaf B's hash now resolves to leaf A's slot
- The blob at `idx_B` still contains leaf B with `hash_B` — two physical leaves share the same hash
- `key_to_index = { key_A: idx_A, key_B: idx_B }` — both keys remain registered

### Impact Explanation

`get_node_by_hash` is a pure cache lookup: [6](#0-5) 

After the corruption, `get_node_by_hash(hash_B)` reads `idx_A` from the cache and returns `(key_A, value_A')` instead of `(key_B, value_B)`. Any DataLayer consumer that resolves leaf data by hash — including proof-of-inclusion verification — will receive the wrong key/value pair for `hash_B`. The blob also contains two leaves with identical hashes, making the Merkle tree structurally invalid. `check_integrity` would catch this (it checks `leaf_count == leaf_hash_to_index_cache_length`), but it is only called in test builds: [7](#0-6) 

### Likelihood Explanation

The `upsert` method is exposed directly through the Python bindings: [8](#0-7) 

The hash parameter is caller-supplied (`new_hash: bytes32`). Any code path that calls `upsert` with a hash value sourced from external/untrusted input — or that reuses a hash across two different keys — triggers the corruption with a single call. No hash collision is required; the attacker only needs to supply the hash of an already-existing leaf as the `new_hash` for a different key.

### Recommendation

Add the same guard that `insert` uses, at the start of `upsert`'s update path, after `remove_leaf` but before `insert_entry_to_blob`:

```rust
if self.block_status_cache.contains_leaf_hash(new_hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, check before `remove_leaf` so the cache is not modified on failure.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();
let hash_a = Hash(Bytes32::new([1u8; 32]));
let hash_b = Hash(Bytes32::new([2u8; 32]));
blob.insert(KeyId(1), ValueId(10), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(20), &hash_b, InsertLocation::Auto {}).unwrap();

// Upsert key 1 with hash_b (already owned by key 2)
blob.upsert(KeyId(1), ValueId(99), &hash_b).unwrap();

// Cache is now corrupted: hash_b resolves to key 1's slot
let (key, value) = blob.get_node_by_hash(hash_b).unwrap();
assert_eq!(key, KeyId(1));   // returns key 1, not key 2
assert_eq!(value, ValueId(99)); // key 2's data is unreachable by hash
```
