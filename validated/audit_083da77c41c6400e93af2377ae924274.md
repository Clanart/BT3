### Title
P2P Sync Client Stores Peer-Supplied Gas Prices Verbatim Without Block-Hash Consistency Verification — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary

`HeaderStreamBuilder::parse_data_for_block` and `write_to_storage` accept and persist a `SignedBlockHeader` from any p2p peer with no verification that the gas price fields (`l1_gas_price`, `l1_data_gas_price`, `l2_gas_price`) are consistent with the `block_hash` field. A malicious peer can supply the real `block_hash` (and a valid signature over it) while injecting arbitrary gas prices, which are stored verbatim and subsequently served by RPC as authoritative values.

### Finding Description

**`parse_data_for_block`** performs exactly two checks on the received `SignedBlockHeader`:
1. Block number ordering [1](#0-0) 
2. Signature vector length == 1 [2](#0-1) 

Neither check touches gas prices. **`write_to_storage`** then calls `append_header` with the peer-supplied `block_header` directly, with no intermediate validation: [3](#0-2) 

The function `verify_block_signature` exists in the codebase: [4](#0-3) 

However, a `grep_search` confirms it is **never called** anywhere in the p2p sync client path — only in its own unit test. Even if it were called, it only verifies `Poseidon(block_hash, state_diff_commitment)` against the signature. It does **not** verify that the `block_hash` field was correctly computed from the gas prices. A malicious peer can supply the real `block_hash` (which carries a valid signature) while setting gas prices to arbitrary values — the signature check would still pass.

The gas prices are direct inputs to `gas_prices_to_hash`, which is chained into `calculate_block_hash`: [5](#0-4) 

The `block_hash` field in `BlockHeader` is a separate stored field — it is whatever the peer sends, not recomputed from the stored gas prices. So the stored `block_hash` remains the real one, but the stored gas prices are inconsistent with it.

### Impact Explanation

The stored gas prices are served by RPC as the authoritative prices for fee estimation and execution for that block. Any client querying `starknet_getBlockWithTxHashes`, `starknet_estimateFee`, or similar endpoints will receive the attacker-controlled gas prices. This satisfies:

- **High**: RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.
- **Critical** (if economic impact is realized): Incorrect L1 gas price effect with economic impact on fee calculations.

The `block_hash` field itself is **not** corrupted (the real hash is stored), so the specific "wrong block hash" claim in the question's proof idea is inaccurate. The actual damage is wrong gas prices in storage and RPC.

### Likelihood Explanation

Any node that connects to a malicious p2p peer during sync is affected. No cryptographic guard prevents this: the signature covers only `Poseidon(block_hash, state_diff_commitment)`, not the gas price fields independently. The attack requires no special privileges — any peer on the p2p network can serve headers.

### Recommendation

In `parse_data_for_block`, after receiving a `SignedBlockHeader`:
1. Recompute the block hash from all `block_header_without_hash` fields (including gas prices) plus the commitment fields, using `calculate_block_hash` / `gas_prices_to_hash`.
2. Assert the recomputed hash equals `signed_block_header.block_header.block_hash`.
3. Only then call `verify_block_signature` against the stored sequencer public key.

This two-step guard ensures that neither gas prices nor any other header field can be tampered with independently of the block hash commitment.

### Proof of Concept

1. Intercept or impersonate a p2p peer serving block N.
2. Fetch the real `SignedBlockHeader` for block N (real `block_hash`, real `signatures`, real commitments).
3. Replace `block_header_without_hash.l1_gas_price.price_in_wei` with `GasPrice(999999)`.
4. Serve this modified header to a syncing node.
5. The node's `parse_data_for_block` accepts it (block number matches, signature count == 1).
6. `write_to_storage` → `append_header` persists the tampered gas price.
7. Query the node's RPC for block N gas price — it returns `999999` wei.
8. Confirm: `calculate_block_hash` with stored prices ≠ `calculate_block_hash` with correct prices (for both V0_13_2 raw-felt path and V0_13_4 Poseidon path), demonstrating the stored gas prices are inconsistent with the stored `block_hash`. [6](#0-5)

/// Verifies that the the block header was signed by the expected sequencer.
pub fn verify_block_signature(
    sequencer_pub_key: &SequencerPublicKey,
    signature: &BlockSignature,
    state_diff_commitment: &GlobalRoot,
    block_hash: &BlockHash,
) -> Result<bool, BlockVerificationError> {
    let message_hash = Poseidon::hash_array(&[block_hash.0, state_diff_commitment.0]);
    verify_message_hash_signature(&message_hash, &signature.0, &sequencer_pub_key.0).map_err(
        |err| BlockVerificationError::BlockSignatureVerificationFailed {
            block_hash: *block_hash,
            error: err,
        },
    )
}

pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> {
    if block_hash_version >= &BlockHashVersion::V0_13_4 {
        vec![
            HashChain::new()
                .chain(&STARKNET_GAS_PRICES0)
                .chain(&l1_gas_price.price_in_wei.0.into())
                .chain(&l1_gas_price.price_in_fri.0.into())
                .chain(&l1_data_gas_price.price_in_wei.0.into())
                .chain(&l1_data_gas_price.price_in_fri.0.into())
                .chain(&l2_gas_price.price_in_wei.0.into())
                .chain(&l2_gas_price.price_in_fri.0.into())
                .get_poseidon_hash(),
        ]
    } else {
        vec![
            l1_gas_price.price_in_wei.0.into(),
            l1_gas_price.price_in_fri.0.into(),
            l1_data_gas_price.price_in_wei.0.into(),
            l1_data_gas_price.price_in_fri.0.into(),
        ]
    }
}
