### Title
Cross-Chain Replay of `deployToken` / `deploy_token` MPC Signature Due to Missing Destination Chain ID in `MetadataPayload` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `near/omni-bridge/src/lib.rs`)

---

### Summary

The `MetadataPayload` signed by the NEAR MPC for token deployment does not include a destination chain identifier. Because the same MPC signing path (`"bridge-1"`) is used for all destination chains, the resulting derived Ethereum address is identical across all EVM chains and Starknet. A signature produced for deploying a token on one chain (e.g., Ethereum) is therefore cryptographically valid on every other chain that shares the same derived address (e.g., Arbitrum, Base, Starknet). Any observer of on-chain transactions can extract the signature and replay it on an unintended chain, causing unauthorized token deployment.

---

### Finding Description

**NEAR side — signing (`near/omni-bridge/src/lib.rs`, `near/omni-types/src/lib.rs`)**

`log_metadata_callback` constructs a `MetadataPayload` and signs it with the MPC signer using the constant path `"bridge-1"` for every destination chain:

```rust
// near/omni-bridge/src/lib.rs  lines 341-365
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload)...
);
ext_signer::ext(self.mpc_signer.clone())
    .sign(SignRequest { payload, path: SIGN_PATH.to_owned(), key_version: 0 })
```

`SIGN_PATH = "bridge-1"` is a single constant for all chains. The `MetadataPayload` struct contains no chain ID field:

```rust
// near/omni-types/src/lib.rs  lines 694-702
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
```

**EVM side — verification (`evm/src/omni-bridge/contracts/OmniBridge.sol`)**

`deployToken` reconstructs and verifies the same chain-ID-free payload:

```solidity
// OmniBridge.sol  lines 142-153
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

No `omniBridgeChainId` byte appears here — contrast with `finTransfer` (lines 294, 297) which correctly embeds `bytes1(omniBridgeChainId)` twice.

**Starknet side — verification (`starknet/src/bridge_types.cairo`, `starknet/src/omni_bridge.cairo`)**

`MetadataPayload.to_borsh()` produces the identical byte sequence with no chain ID:

```cairo
// starknet/src/bridge_types.cairo  lines 36-44
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into()); // 0x01
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

`deploy_token` verifies against `omni_bridge_derived_address` (an `EthAddress`), which is derived from the same MPC key/path as every EVM chain's `nearBridgeDerivedAddress`:

```cairo
// starknet/src/omni_bridge.cairo  lines 202-205
fn deploy_token(ref self: ContractState, signature: Signature, payload: MetadataPayload) {
    _verify_borsh_signature(ref self, @payload.to_borsh(), signature);
    ...
}
```

**Solana side** uses `IncomingMessageType::Metadata` (value `1`) and the same field order, but verifies against a 64-byte uncompressed public key rather than an Ethereum address, so it is not cross-replayable with EVM/Starknet.

---

### Impact Explanation

Because the signed payload is byte-for-byte identical across all EVM chains and Starknet, and the same MPC key (path `"bridge-1"`) produces the same derived Ethereum address on all of them, a single `deployToken` signature is valid on every EVM-compatible OmniBridge instance and on the Starknet OmniBridge simultaneously.

An attacker who observes a successful `deployToken` transaction on Ethereum can immediately replay the calldata (signature + payload) on Arbitrum, Base, or Starknet, deploying the same bridge token on those chains without any authorization from the NEAR MPC or bridge operators. The only on-chain guard is the "token already exists" check, which only prevents re-deployment on the *same* chain.

Consequences:
- Unauthorized bridge tokens are deployed on chains the operators have not yet enabled, potentially before the bridge infrastructure (relayers, provers, fee configuration) is ready.
- Once deployed, the token is registered in the bridge's token mapping and cannot be easily removed; any subsequent legitimate `finTransfer` for that chain will use the attacker-triggered deployment.
- If operators intended to register a *native* token (e.g., canonical USDC) rather than a bridge-minted one, the pre-deployment forces the bridge to use the bridge-minted version instead, permanently altering the token model for that chain.

This matches the allowed impact: **High — Proof/signature/MPC verification bypass enabling unauthorized token deployment.**

---

### Likelihood Explanation

- All `deployToken` transactions are public on-chain; the signature and payload are trivially extractable from any block explorer.
- No privileged access is required; any EOA can call `deployToken` on any chain.
- The attacker only needs to copy the calldata from one chain's transaction and submit it to another chain's OmniBridge contract.
- The window is open from the moment the first deployment is confirmed until the token is deployed on every other chain.

---

### Recommendation

Include the destination chain ID in the signed `MetadataPayload`, mirroring the pattern already used in `finTransfer`/`TransferMessagePayload`. On the NEAR side, pass the destination chain kind into `log_metadata_callback` and embed it in the payload before signing. On each destination chain, include the chain's own ID byte in the borsh-encoded data before hashing.

**NEAR types** — add a `destination_chain` field to `MetadataPayload`:
```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub destination_chain: ChainKind, // NEW
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
```

**EVM** — prepend `omniBridgeChainId` in `deployToken`, matching `finTransfer`:
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD
    Borsh.encodeString(metadata.token),
    ...
);
```

**Starknet** — pass and embed `chain_id` in `MetadataPayload.to_borsh()`:
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append_byte(chain_id);  // ADD
    ...
}
```

### Proof of Concept

1. Operator calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Ethereum OmniBridge. Transaction is mined; `sig` and payload are public.
2. Attacker copies the exact `signatureData` and `metadata` calldata.
3. Attacker calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum OmniBridge.
4. Arbitrum's `ECDSA.recover(keccak256(borshEncoded), sig)` returns the same `nearBridgeDerivedAddress` (identical key, identical payload bytes) → signature check passes.
5. `isBridgeToken[nearToEthToken["usdc.near"]]` is `false` on Arbitrum → existence check passes.
6. A new `BridgeToken` proxy is deployed on Arbitrum and registered as the canonical bridge token for `"usdc.near"` — without any NEAR MPC authorization for Arbitrum.
7. Repeat for Starknet's `deploy_token` with the same signature. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
