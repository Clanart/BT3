### Title
Unvalidated `recipient` String in `initTransfer` Enables Permanent EVM Token Lock — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` on EVM accepts a raw `string calldata recipient` with no format validation. Tokens are burned or locked on EVM before the recipient is ever checked. When the NEAR side later parses the emitted `InitTransfer` event, an unparseable recipient string causes `fin_transfer_callback` to panic and the transfer to become permanently irrecoverable. The same root cause exists in the Solana `init_transfer` / `init_transfer_sol` entrypoints (explicitly acknowledged in `solana/SECURITY.md` as unaddressed) and in StarkNet's `init_transfer`.

---

### Finding Description

**Source-chain entrypoint — no recipient validation:**

`OmniBridge.initTransfer()` accepts `string calldata recipient` and immediately burns or locks tokens, then emits `InitTransfer` with the raw string: [1](#0-0) 

The only guard is `fee >= amount`. There is no check that `recipient` is a well-formed `OmniAddress` string (e.g. `"eth:0x..."`, `"near:alice.near"`, `"sol:<base58>"`).

**NEAR-side parsing — hard failure on invalid recipient:**

When a relayer submits the EVM proof, the NEAR EVM prover parses the `InitTransfer` log and calls:

```rust
recipient: event.data.recipient.parse().map_err(stringify)?,
``` [2](#0-1) 

`OmniAddress::from_str()` requires the format `"chain:address"` with a supported chain prefix and a valid address for that chain: [3](#0-2) 

If parsing fails, the `TryFromLog` conversion returns `Err`, the prover promise fails, and `fin_transfer_callback` panics:

```rust
let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) else {
    env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str())
};
``` [4](#0-3) 

**No recovery path exists.** The EVM tokens are already burned or locked in the contract. There is no admin unlock, no retry mechanism, and no refund path for a transfer whose NEAR-side proof parse fails. The `completedTransfers` nonce is never marked (the callback panicked), so the same invalid VAA cannot even be replayed to attempt recovery.

**Identical root cause on Solana** — `InitTransferPayload.recipient` is an unvalidated `String`: [5](#0-4) 

`process()` checks only fee and amount; no recipient validation: [6](#0-5) 

The Solana `SECURITY.md` explicitly acknowledges this as an unaddressed known issue, but classifies it as "low severity" — a misclassification given the actual impact of permanent fund loss: [7](#0-6) 

**Identical root cause on StarkNet** — `init_transfer` accepts `recipient: ByteArray` with no format validation before burning/locking tokens and emitting the event: [8](#0-7) 

The StarkNet prover similarly calls `recipient_str.parse().map_err(stringify)?`: [9](#0-8) 

---

### Impact Explanation

Any user who calls `initTransfer` (EVM), `init_transfer`/`init_transfer_sol` (Solana), or `init_transfer` (StarkNet) with a malformed `recipient` string will have their tokens permanently locked or burned on the source chain with no recovery path. The NEAR `fin_transfer_callback` panics on the invalid proof, leaving no pending transfer record, no retry payload, and no refund mechanism. This matches **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows**.

---

### Likelihood Explanation

The `recipient` field is a free-form string on all three source chains. A user mistake — wrong chain prefix, unsupported chain name, invalid address encoding, or a plain typo — is sufficient to trigger the loss. No special privileges are required. The Solana SECURITY.md already confirms the scenario is reachable and unaddressed. The EVM and StarkNet paths are equally reachable by any unprivileged user calling the public `initTransfer` entrypoint.

---

### Recommendation

Validate the `recipient` string on the source chain before burning or locking tokens:

- **EVM**: Parse `recipient` against the expected `OmniAddress` format (e.g. require a known `chain:` prefix and a syntactically valid address) and revert if invalid.
- **Solana**: Add a validation step in `InitTransfer::process()` that checks `payload.recipient` is a parseable `OmniAddress` string before locking/burning tokens.
- **StarkNet**: Add equivalent validation in `init_transfer` before the token burn/lock.

Alternatively, implement a source-chain refund/revert mechanism so that if the NEAR-side proof parse fails, the source-chain tokens can be unlocked via an admin or user-initiated recovery call.

---

### Proof of Concept

**EVM path:**

1. Call `OmniBridge.initTransfer(tokenAddress, 1e18, 0, 0, "not_a_valid_omni_address", "")`. The function passes the `fee >= amount` check, burns/locks `1e18` tokens, and emits `InitTransfer` with `recipient = "not_a_valid_omni_address"`.
2. A relayer submits the EVM proof to NEAR via `fin_transfer`.
3. The EVM prover calls `event.data.recipient.parse()` → `OmniAddress::from_str("not_a_valid_omni_address")` → defaults to chain `"eth"`, tries to parse `"not_a_valid_omni_address"` as an EVM hex address → `Err("ERR_INVALID_HEX")`.
4. `TryFromLog` returns `Err`; the prover promise fails.
5. `fin_transfer_callback` hits the `else` branch: `env::panic_str(BridgeError::InvalidProofMessage...)`.
6. The NEAR transaction reverts. No `completedTransfers` entry is written. No pending transfer is stored. The EVM tokens remain permanently locked/burned with no recovery path.

**Solana path (same outcome):**

1. Call `init_transfer` with `payload.recipient = "garbage"`. Tokens are locked in the vault PDA or burned from the bridged mint. Wormhole message posted.
2. NEAR prover parses the VAA: `transfer.recipient.parse()` → `Err`.
3. `fin_transfer_callback` panics. Tokens permanently lost.

            let origin_nonce = self.current_origin_nonce.read() + 1;

                            token_address,
                            origin_nonce,
                            amount,
                            fee,
                            native_fee,
                            recipient,

                        },
