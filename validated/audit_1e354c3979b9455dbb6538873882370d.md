### Title
Unchecked Return Value from `mint()` in `finTransfer()` After Nonce Consumption Causes Permanent Fund Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.sol`, the `finTransfer()` function marks the destination nonce as consumed **before** calling `mint()` on `ICustomMinter` or `IBridgeToken`. If either `mint()` call returns `false` (silent failure, no revert), the nonce is permanently spent but no tokens are delivered to the recipient. The user's bridged assets become irrecoverable.

---

### Finding Description

`finTransfer()` follows this sequence:

1. **Line 287** — nonce is marked used unconditionally:
   ```solidity
   completedTransfers[payload.destinationNonce] = true;
   ```

2. **Lines 332–336** — `ICustomMinter.mint()` return value is not checked:
   ```solidity
   ICustomMinter(customMinters[payload.tokenAddress]).mint(
       payload.tokenAddress,
       payload.recipient,
       payload.amount
   );
   ```

3. **Lines 339–349** — `IBridgeToken.mint()` return value is not checked:
   ```solidity
   IBridgeToken(payload.tokenAddress).mint(
       payload.recipient,
       payload.amount
   );
   // or with message:
   IBridgeToken(payload.tokenAddress).mint(
       payload.recipient,
       payload.amount,
       payload.message
   );
   ```

Both `ICustomMinter` and `IBridgeToken` are custom interfaces. If their `mint()` implementations return a `bool` to signal success/failure (rather than reverting on failure), a `false` return goes entirely undetected. The nonce at line 287 is already consumed, so the call cannot be retried. The `FinTransfer` event is still emitted (line 359–366), giving the appearance of a successful settlement. ---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

- The user locked assets on the origin chain (NEAR, Solana, StarkNet, or another EVM chain) during `initTransfer`.
- `finTransfer()` is called on the destination EVM chain with a valid MPC signature.
- The nonce is consumed at line 287, but `mint()` silently returns `false`.
- No tokens reach the recipient; the nonce cannot be replayed; the origin-chain lock is already finalized.
- The user's funds are permanently lost with no recovery path.

---

### Likelihood Explanation

**Medium.** The trigger requires a `customMinter` or `bridgeToken` whose `mint()` can return `false` without reverting — a realistic scenario for any non-standard or upgradeable token integrated via the custom-minter registry. Any bridge user whose token uses such a minter is exposed on every `finTransfer` call. No privileged access is required; the attacker-controlled entry is the public `finTransfer()` function callable by any relayer or user with a valid MPC-signed payload.

---

### Recommendation

After each `mint()` call, check the return value and revert if it indicates failure. Use a pattern such as:

```solidity
bool ok = ICustomMinter(customMinters[payload.tokenAddress]).mint(
    payload.tokenAddress,
    payload.recipient,
    payload.amount
);
if (!ok) revert MintFailed();
```

Apply the same check to both `IBridgeToken.mint()` overloads. Alternatively, require that all registered minters revert on failure rather than returning `false`, and enforce this in the minter registration logic.

---

### Proof of Concept

1. Register a custom minter whose `mint()` always returns `false` (does not revert).
2. A user initiates a cross-chain transfer on the origin chain, locking tokens.
3. A relayer calls `finTransfer()` on the EVM destination with the valid MPC signature.
4. `completedTransfers[nonce]` is set to `true` (line 287).
5. `ICustomMinter.mint()` is called and returns `false` — no tokens minted, no revert.
6. `FinTransfer` event is emitted; the transaction succeeds on-chain.
7. The user receives nothing. The nonce is spent. The origin-chain lock is permanent. Funds are irrecoverably lost. [3](#0-2)
