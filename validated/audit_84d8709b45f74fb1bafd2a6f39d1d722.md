### Title
Unauthenticated `ProofManager::set_proof` Endpoint Enables Proof-Slot Poisoning, Permanently Bypassing Proof Verification for Matching `proof_facts` — (`crates/apollo_proof_manager/src/proof_manager.rs`)

---

### Summary

The `ProofManager` exposes a `SetProof` RPC endpoint via `RemoteComponentServer` with no caller authentication and no proof verification. The `TransactionConverter::run_proof_verification` function skips cryptographic proof verification entirely when `contains_proof` returns `true`. An attacker with network access to the proof manager's HTTP/2 endpoint can pre-populate a `proof_facts` slot with an invalid (unverified) proof blob. Every subsequent transaction carrying those same `proof_facts` will have its proof verification silently skipped by both the gateway and consensus validator paths, causing the node to accept and propagate transactions whose ZK proofs were never verified.

---

### Finding Description

**Analog mapping.** The LayerZero bug's core structure is: (1) a shared queue keyed only by endpoint address, (2) any UA can write to it without sender isolation, (3) an invalid item at the head permanently blocks the queue. The sequencer analog is: (1) a shared proof store keyed only by `Poseidon(proof_facts)`, (2) any caller can write to it without authentication, (3) a pre-written invalid proof permanently occupies the slot and causes all subsequent verification to be skipped.

**Root cause — `ProofManager::set_proof` stores without verifying:** [1](#0-0) 

`set_proof` checks `contains_proof` and returns `Ok(())` immediately if the slot is occupied. It never calls `verify_proof`. The `RemoteComponentServer` wrapping it adds no authentication layer. [2](#0-1) 

**Root cause — `run_proof_verification` skips verification when slot is occupied:** [3](#0-2) 

Lines 403–407 are the critical path: if `contains_proof` returns `true`, the function returns `Ok(false)` without ever calling `starknet_proof_verifier::verify_proof`. The caller (`spawn_proof_verification` for the gateway path, `spawn_verify_and_store_proof` for the consensus path) treats `Ok(false)` as "already verified — skip" and proceeds to accept the transaction.

**Gateway path — verification skipped, invalid proof silently retained:** [4](#0-3) 

The `VerificationHandle` task resolves `Ok(())` whether verification ran or was skipped. The gateway then calls `store_proof_in_proof_manager(F, P_legitimate)`: [5](#0-4) 

But `ProofManager::set_proof` sees `contains_proof(F) = true` (the attacker's `P_invalid` is already there) and returns early — the legitimate proof is silently discarded.

**Consensus path — same skip applies to validators:** [6](#0-5) 

Validators call `run_proof_verification` on the same `proof_facts`; `contains_proof` returns `true`; verification is skipped; the transaction is accepted.

**Proof retrieval returns the poisoned value:** [7](#0-6) 

When the internal transaction is converted back to an RPC transaction for consensus propagation, `get_proof(&tx.proof_facts)` retrieves `P_invalid` from the proof manager and embeds it in the outgoing `RpcTransaction`.

**`proof_facts` hash is a simple Poseidon chain — no binding to caller identity:** [8](#0-7) 

The key is `Poseidon(proof_facts[0], proof_facts[1], …)`. Any caller who knows the target `proof_facts` vector can compute the same key and pre-occupy the slot.

---

### Impact Explanation

**Matching scope: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A transaction whose ZK proof was never verified is admitted by the gateway, propagated through consensus, and included in a committed block. The `perform_pre_validation_stage` check validates `block_number`, `block_hash`, `program_hash`, and `config_hash` fields inside `proof_facts`: [9](#0-8) 

but it does **not** verify the proof bytes `P`. Those bytes are only checked by `verify_proof` in the converter — which is the step being bypassed. The accepted transaction carries `proof_facts` whose claimed virtual-OS execution output is not backed by any valid cryptographic proof.

---

### Likelihood Explanation

The `RemoteComponentServer` for the proof manager is a plain HTTP/2 endpoint with no application-level authentication: [10](#0-9) 

In a distributed deployment any node or process that can reach the proof manager's network port can issue a `SetProof` request. The attacker needs only to know (or guess) a `proof_facts` vector that a future transaction will use — which is predictable because `SnosProofFacts` is constructed from public chain data (`block_number`, `block_hash`, `program_hash`, `config_hash`): [11](#0-10) 

---

### Recommendation

1. **Authenticate `SetProof` callers.** The `RemoteComponentServer` for the proof manager should require mTLS or a shared secret so only the gateway and consensus orchestrator can write proofs.
2. **Verify before storing.** `ProofManager::set_proof` should call `verify_proof(proof_facts, proof)` before persisting, making the store itself the authoritative verification point regardless of caller.
3. **Do not skip verification on cache hit.** `run_proof_verification` should not treat `contains_proof = true` as proof of validity. It should either re-verify or require that the store itself enforces validity (point 2 above).
4. **Isolate proof channels by submitter.** Analogous to the LayerZero fix (`Channels<phantom UA>` scoped by UA address), scope the proof store by the submitting account address or transaction hash so one submitter cannot pre-occupy a slot for another.

---

### Proof of Concept

```
# 1. Attacker learns the proof_facts F that a future client-side-proving tx will use.
#    F = [PROOF_VERSION, VIRTUAL_SNOS, program_hash, VIRTUAL_OS_OUTPUT_VERSION,
#          block_number, block_hash, config_hash, ...]
#    All fields are public chain data.

# 2. Attacker sends SetProof(F, P_invalid) directly to the RemoteProofManagerServer HTTP/2 port.
#    No authentication is required. P_invalid is arbitrary bytes.
#    ProofManager::set_proof stores P_invalid under Poseidon(F) without calling verify_proof.

# 3. Legitimate user submits InvokeV3 tx with proof_facts=F and valid proof P to the gateway.

# 4. Gateway calls run_proof_verification(F, P):
#      contains_proof(F) → true  (attacker's P_invalid is stored)
#      → returns Ok(false)  [verification SKIPPED]
#    verification_task resolves Ok(()) → gateway accepts the transaction.

# 5. Gateway calls store_proof_in_proof_manager(F, P):
#      ProofManager::set_proof: contains_proof(F) → true → returns Ok(()) silently.
#    P_invalid remains stored; P is discarded.

# 6. Transaction enters mempool, is proposed, and validators repeat step 4:
#      run_proof_verification(F, P_invalid_retrieved):
#        contains_proof(F) → true → skipped → Ok(false)
#    All validators accept without verifying.

# 7. convert_internal_rpc_tx_to_rpc_tx retrieves P_invalid from proof manager
#    and embeds it in the outgoing RpcTransaction — an authoritative-looking
#    transaction with an invalid ZK proof is committed to the chain.
```

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
