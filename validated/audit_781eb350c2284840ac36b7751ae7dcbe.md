### Title
`batch_insert` Bypasses Duplicate-Key Guard, Corrupting DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash existence checks that `MerkleBlob::insert` enforces. When a key already present in the tree (or repeated within the same batch) is supplied to `batch_insert`, a second leaf node for that key is silently written into the blob. The result is a structurally corrupt Merkle tree whose root hash is computed over both the old and the new leaf, invalidating all subsequent proofs of inclusion and the committed DataLayer state.

### Finding Description

`MerkleBlob::insert` (the single-item path) opens with two hard guards:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` (the bulk path) calls `self.insert` only for the first two items when the tree has ≤ 1 existing leaf:

```rust
// blob.rs:578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← guarded
    }
}
```

Every remaining item is written directly via `insert_entry_to_blob` with **no existence check**:

```rust
// blob.rs:587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` then calls `block_status_cache.add_leaf`, which silently overwrites the cache entry for the duplicate key:

```rust
// blob.rs:188-193  (BlockStatusCache::add_leaf)
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // ← HashMap::insert silently replaces
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

After `batch_insert` completes the subtree-linking phase (lines 607–654), the tree contains **two leaf nodes for the same key**: the original leaf (still connected to its parent in the blob) and the newly inserted leaf (connected via the new subtree). The cache now points only to the new leaf, so the old leaf is an unreachable orphan from the cache's perspective but is still reachable from the root via the original parent pointer.

`check_integrity` (lines 821–887) is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`); it is not called in production after `batch_insert`.

### Impact Explanation

The Merkle root is computed over both leaves (old and new) because `calculate_lazy_hashes` traverses the blob from the root and the old leaf is still reachable through its parent. The root hash therefore reflects a tree that contains the same key twice with potentially different values and hashes. Consequences:

- **Corrupted DataLayer root**: the committed root hash no longer represents a valid key-value mapping.
- **Invalid proofs of inclusion**: `get_proof_of_inclusion(key)` navigates to the new leaf (via the cache), producing a proof that is internally consistent but anchored to a root that also includes the orphaned old leaf — the proof's `root_hash()` does not match the true root of the tree.
- **Proof-of-exclusion forgery**: a verifier comparing the root against an expected value will see a mismatch, allowing an attacker to deny the existence of a key that is actually present, or assert inclusion of a key under a root that was never committed.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`py_batch_insert` is a first-class Python binding exposed on `MerkleBlob`:

```rust
// blob.rs:1503-1519
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    …
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

It is declared in the public type stub (`wheel/python/chia_rs/datalayer.pyi:331`) and exercised directly in integration tests (`tests/test_datalayer.py:334`). Any caller that passes a key already present in the tree — or passes the same key twice within one batch — triggers the corruption. The chia-blockchain Python node calls this path during DataLayer store updates; a malformed or replayed update record is sufficient to trigger it without any privileged access.

### Recommendation

Add the same existence checks at the top of the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing insertion logic
}
```

Alternatively, refactor `batch_insert` to call the guarded `insert` path for every item (accepting the small per-item overhead), or add a pre-pass that deduplicates and validates the input batch before any blob mutation begins.

### Proof of Concept

```rust
// Tree already has 3 leaves (leaf_count > 1), so the guarded path is skipped.
let mut blob = MerkleBlob::new(vec![]).unwrap();
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
}

// batch_insert with a key that already exists (KeyId(0)) — no error is returned.
blob.batch_insert(vec![
    ((KeyId(0), ValueId(99)), sha256_num(&99i64)),  // duplicate key
]).unwrap();

blob.calculate_lazy_hashes().unwrap();

// check_integrity is only active in test builds; in production this passes silently.
// The root hash now covers two leaves for KeyId(0), corrupting the committed state.
let proof = blob.get_proof_of_inclusion(KeyId(0)).unwrap();
// proof.root_hash() != actual root because the old leaf is still in the tree.
assert!(proof.valid()); // passes — but the root it validates against is wrong
```
