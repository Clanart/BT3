Audit Report

## Title
Stranded ETH on Router Consumed by Subsequent User's WETH Payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` reads `address(this).balance` as an unaccounted global pool when settling WETH payments. ETH left on the router by a prior user who omitted `refundETH()` is silently consumed to subsidize a subsequent user's WETH swap, causing direct, irrecoverable loss of the first user's ETH principal.

## Finding Description
The WETH branch of `pay` (lines 73–84) reads the router's entire native balance with no per-call budget tracking:

```solidity
uint256 nativeBalance = address(this).balance;   // global, not per-user
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

`exactInputSingle` is `payable` and performs no automatic refund of excess `msg.value`; the protocol's own test suite documents that users must append `refundETH()` to a multicall to recover unused ETH. If a user calls `exactInputSingle` directly (not via multicall) with `msg.value > amountIn`, the surplus ETH remains on the router after the call completes.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks bare ETH transfers; it does not prevent ETH from accumulating via `payable` function calls.

**Exploit path:**
1. User A calls `exactInputSingle{value: 2 ETH}(amountIn=1 ETH, tokenIn=WETH, ...)` without `refundETH()`. The swap consumes 1 ETH; 1 ETH remains on the router.
2. User B calls `exactInputSingle(amountIn=1.5 ETH, tokenIn=WETH, ...)` with no ETH, relying on WETH allowance.
3. The pool triggers `metricOmmSwapCallback`, which calls `_justPayCallback`, which calls `pay(WETH, userB, pool, 1.5 ETH)`.
4. `nativeBalance = 1 ETH`: satisfies `nativeBalance > 0 && nativeBalance < value`.
5. Router wraps and forwards User A's 1 ETH as WETH to the pool; pulls only 0.5 ETH worth of WETH from User B via `transferFrom`.
6. User B receives a full 1.5 ETH swap while paying only 0.5 ETH from their allowance. User A's 1 ETH is permanently lost.

No existing guard prevents this: the transient callback context records User B as `payer`, the `receive()` check is irrelevant, and there is no per-call ETH budget anywhere in the call stack.

## Impact Explanation
Direct loss of user principal: User A's ETH is irreversibly transferred to a pool to settle User B's swap. The router holds no record of whose ETH it holds, so there is no recovery path. This is an unauthorized consumption of another user's ETH resulting in direct fund loss, meeting the High severity threshold.

## Likelihood Explanation
- The protocol's own test (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) demonstrates the pattern of sending excess ETH and relying on `refundETH()` to clean up, confirming the residual-ETH scenario is a normal usage path.
- Any user who calls `exactInputSingle` directly (not via multicall) with `msg.value > amountIn` will leave residual ETH on the router.
- An MEV bot or griefer can monitor the router's ETH balance and immediately follow with a WETH swap to capture the subsidy.
- No special permissions, malicious pool setup, or non-standard tokens are required — only two sequential public router calls.

## Recommendation
Track per-call ETH entitlement rather than using the global balance. Record `msg.value` at the top-level entry point (e.g., in transient storage during `exactInputSingle` / `multicall`) and pass the available ETH budget explicitly into `pay`, decrementing it as ETH is consumed and reverting if the per-call budget is exceeded. Alternatively, enforce that `address(this).balance` at the start of `pay` equals exactly the ETH that arrived in the current top-level call by storing and comparing `msg.value` in transient storage.

## Proof of Concept
```solidity
function test_strandedEthSubsidizesOtherUser() public {
    // User A sends 2 ETH but swaps only 1 ETH worth of WETH, no refundETH
    vm.prank(userA);
    router.exactInputSingle{value: 2 ether}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 1 ether, ...})
    );
    assertEq(address(router).balance, 1 ether); // 1 ETH stranded

    uint256 userBWethBefore = weth.balanceOf(userB);
    // User B swaps 1.5 ETH worth of WETH, sends no ETH
    vm.prank(userB);
    router.exactInputSingle(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 1.5 ether, ...})
    );

    // User B's WETH allowance reduced by only 0.5 ETH (not 1.5 ETH)
    assertEq(userBWethBefore - weth.balanceOf(userB), 0.5 ether);
    // User A's 1 ETH is gone
    assertEq(address(router).balance, 0);
}
```

The root cause is confirmed at `PeripheryPayments.sol` lines 74–81 where `address(this).balance` is read with no per-user accounting, and at `MetricOmmSimpleRouter.sol` line 67 where `exactInputSingle` is `payable` with no automatic excess-ETH refund.
