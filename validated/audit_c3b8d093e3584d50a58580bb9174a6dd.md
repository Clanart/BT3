### Title
Unbound `ChunkEndorsementV2` Inner/Metadata Signatures Enable Cache-Poisoning to Suppress Validator Endorsements — (`File: core/primitives/src/stateless_validation/chunk_endorsement.rs`)

---

### Summary

`ChunkEndorsementV2` carries two independently-signed payloads: `inner` (containing only `chunk_hash`) and `metadata` (containing `shard_id`, `epoch_id`, `height_created`, `account_id`). Neither payload commits to the other. An unprivileged network peer who has collected any two valid endorsements from the same validator can splice them — taking `inner` from one and `metadata` from the other — to produce a structurally valid message that passes all signature checks. When delivered to a block producer before the legitimate endorsement, this poisons the per-slot endorsement cache and causes the legitimate endorsement to be silently dropped, preventing that validator's vote from being counted.

---

### Finding Description

**Structural gap.** `ChunkEndorsementV2` is defined as:

```
inner            = { chunk_hash, signature_differentiator }   ← signed by `signature`
metadata         = { account_id, shard_id, epoch_id,
                     height_created }                          ← signed by `metadata_signature`
``` [1](#0-0) 

The `verify()` method checks both signatures against the same public key but never asserts that `inner.chunk_hash` corresponds to the chunk identified by `metadata.(shard_id, epoch_id, height_created)`:

```rust
fn verify(&self, public_key: &PublicKey) -> bool {
    let inner    = borsh::to_vec(&self.inner).unwrap();
    let metadata = borsh::to_vec(&self.metadata).unwrap();
    self.signature.verify(&inner, public_key)
        && self.metadata_signature.verify(&metadata, public_key)
}
``` [2](#0-1) 

**Splice construction.** Given two legitimately broadcast endorsements from the same validator V:

| | E1 (chunk C1, hash CH1, slot S1/E1/H1) | E2 (chunk C2, hash CH2, slot S2/E2/H2) |
|---|---|---|
| inner | `{chunk_hash: CH1}` + `sig1` | `{chunk_hash: CH2}` + `sig2` |
| metadata | `{S1,E1,H1,V}` + `msig1` | `{S2,E2,H2,V}` + `msig2` |

An attacker constructs `E_forged = { inner: E1.inner, signature: E1.signature, metadata: E2.metadata, metadata_signature: E2.metadata_signature }`. Both `sig1.verify(inner1, V.pubkey)` and `msig2.verify(meta2, V.pubkey)` return `true`, so `E_forged.verify()` passes.

**Cache-poisoning path.** `process_chunk_endorsement` first checks whether the cache already holds an entry for `(chunk_production_key, account_id)`:

```rust
if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
    tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
    return Ok(());
}
``` [3](#0-2) 

If `E_forged` arrives first, it is stored under key `(S2,E2,H2)` with value `(CH1, sig1)`. When the legitimate `E2` subsequently arrives, the early-return fires and `E2` is silently discarded.

**Endorsement never counted.** `collect_chunk_endorsements` filters cached entries by `chunk_hash == chunk_header.chunk_hash()`:

```rust
.filter(|(_, (chunk_hash, _))| chunk_hash == chunk_header.chunk_hash())
``` [4](#0-3) 

Because the poisoned entry carries `CH1 ≠ CH2`, it is excluded. Validator V's endorsement for chunk C2 is never counted toward the endorsement threshold.

**Validation does not cross-check.** `validate_chunk_endorsement` calls `validate_chunk_relevant_as_validator` (using metadata's `chunk_production_key`) and then `validate_chunk_endorsement_signature` (which calls `verify()` above). Neither step checks that `inner.chunk_hash` is the canonical hash for the slot described by `metadata`. [5](#0-4) 

---

### Impact Explanation

If an attacker can deliver `E_forged` to the block producer before the legitimate endorsement for a sufficient number of validators (exceeding the endorsement-stake threshold), the block producer cannot collect enough endorsements to include the chunk. This is a targeted liveness attack: chunk production stalls for the targeted slot without any on-chain evidence of misbehavior, and the block producer has no mechanism to distinguish a poisoned cache entry from a genuine duplicate.

---

### Likelihood Explanation

Endorsements are broadcast over the P2P network. Any connected peer accumulates a pool of valid endorsements from all validators over time. The attacker only needs:
1. One prior endorsement from validator V (trivially collected from broadcast traffic).
2. To observe V's endorsement for the target slot and deliver `E_forged` to the block producer before the legitimate message arrives.

The race window is the network propagation delay between V and the block producer. A well-connected adversary (e.g., a node co-located with the block producer, or one on the routing path) can reliably win this race. No privileged role is required.

---

### Recommendation

Bind `inner` and `metadata` cryptographically so that neither can be substituted independently. The simplest fix is to include the `chunk_hash` inside `ChunkEndorsementMetadata` (or vice-versa, include the routing key inside `ChunkEndorsementInnerV1`) so that a single signature covers both. Alternatively, replace the two-signature design with a single signature over a combined struct containing `chunk_hash`, `shard_id`, `epoch_id`, `height_created`, and `account_id`. [6](#0-5) 

---

### Proof of Concept

```
// Attacker has collected from broadcast traffic:
//   E1: validator V endorsed chunk C1 (chunk_hash=CH1) at slot (S_any, E_any, H_any)
//   E2: validator V endorsed chunk C2 (chunk_hash=CH2) at slot (S2, E2, H2)

let E_forged = ChunkEndorsementV2 {
    inner:              E1.inner,              // chunk_hash = CH1
    signature:          E1.signature,          // valid sig over CH1 by V
    metadata:           E2.metadata,           // shard=S2, epoch=E2, height=H2, account=V
    metadata_signature: E2.metadata_signature, // valid sig over (S2,E2,H2,V) by V
};

// E_forged.verify(V.public_key()) == true  ← both checks pass independently

// Attacker sends E_forged to block producer before E2 arrives.
// Block producer stores (CH1, sig1) under cache key (S2, E2, H2) for account V.
// Legitimate E2 arrives → early-return fires → E2 dropped.
// collect_chunk_endorsements for C2 filters by chunk_hash==CH2 → CH1≠CH2 → V not counted.
```

pub struct ChunkEndorsementV2 {

        let inner = borsh::to_vec(&self.inner).unwrap();
        let metadata = borsh::to_vec(&self.metadata).unwrap();
        self.signature.verify(&inner, public_key)
            && self.metadata_signature.verify(&metadata, public_key)
    }
}

        target: "stateless_validation",
        "validate_chunk_endorsement",
        height = endorsement.chunk_production_key().height_created,

        endorsement.account_id(),

    Ok(ChunkRelevance::Relevant)
