### Title
Constructor Revert in `deploy` Syscall Causes Block-Level Panic Instead of Graceful Failure — (File: `execution/deploy_contract.cairo`, `execution/syscall_impls.cairo`)

---

### Summary

The `deploy` syscall in the StarkNet OS Cairo program does not handle constructor reverts. When a Sierra contract constructor reverts during a `deploy` syscall, the OS asserts `is_reverted = 0` (panicking the Cairo VM) instead of returning a graceful `failure_flag=1` response to the caller. This is structurally analogous to the original report's try-catch that silently swallows a callback failure: the outer operation (block execution) proceeds as if the inner call succeeded, but the inner state transition (constructor execution) was never properly handled. The result is a block-level proof failure — a network halt.

---

### Finding Description

**Root cause in `deploy_contract.cairo`:**

The function `deploy_contract` is the shared deployment primitive called both from the `execute_deploy` syscall and from `execute_deploy_account_transaction`. It writes the new contract's class hash to state *before* invoking the constructor, then calls `select_execute_entry_point_func` for the constructor: [1](#0-0) [2](#0-1) [3](#0-2) 

If the constructor reverts (`is_reverted = 1`), the `assert is_reverted = 0` on line 91 causes the Cairo VM to panic. There is no revert path: the class hash written to state at lines 56–66 is not rolled back before the panic, and the block proof becomes invalid.

**Root cause in `syscall_impls.cairo`:**

The `execute_deploy` syscall calls `deploy_contract` and then unconditionally hardcodes `failure_flag=0` in the response header: [4](#0-3) 

The comment on line 538 (`// TODO(Yoni, 1/1/2026): support failures.`) explicitly acknowledges that failure handling is unimplemented. Unlike `contract_call_helper`, which correctly sets `failure_flag=is_reverted` in the response: [5](#0-4) 

…the `deploy` syscall has no equivalent path. The caller contract can never observe a `failure_flag=1` from `deploy`; instead, the OS panics.

**Contrast with `non_reverting_select_execute_entry_point_func`:**

The `assert is_reverted = 0` pattern is intentional in `non_reverting_select_execute_entry_point_func` (used for `__validate__`, fee charging, L1 handlers — contexts where revert is a protocol violation): [6](#0-5) 

But `deploy_contract` is explicitly *not* in that category — its own TODO comment says "handle failures" — making the `assert is_reverted = 0` there a bug, not a design choice.

---

### Impact Explanation

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

When a user-submitted transaction triggers the `deploy` syscall with a constructor that reverts:

1. `deploy_contract` panics via `assert is_reverted = 0`.
2. The Cairo VM execution of the block fails.
3. The block proof cannot be generated.
4. The sequencer cannot finalize the block.
5. If the sequencer's off-chain simulation (Rust execution layer) handles constructor reverts gracefully — returning `failure_flag=1` — while the Cairo OS panics, the sequencer will include the transaction in a block that the Cairo prover then cannot prove. This is the direct analog to the original report: the simulation "succeeds" (like the try-catch swallowing the error), but the authoritative state-transition layer (the Cairo OS proof) fails.

---

### Likelihood Explanation

The sequencer simulates transactions before inclusion. If the simulation uses the same Cairo OS code path, it would also panic and exclude the transaction. However:

- The sequencer's execution layer is a Rust implementation. If it handles constructor reverts by returning a failure response (the *correct* behavior per the TODO), while the Cairo OS panics (the *current* behavior), there is a simulation/proof divergence.
- An attacker can deliberately deploy a contract whose constructor reverts conditionally (e.g., based on block number or storage state that differs between simulation and proving time), causing the divergence to be triggered on-chain.
- The TODO deadline of `1/1/2026` has passed (today is 2026-07-03), meaning this known gap is still present in production code.

---

### Recommendation

In `deploy_contract.cairo`, replace the unconditional `assert is_reverted = 0` with proper failure propagation — returning `(is_reverted, retdata_size, retdata)` to the caller and rolling back the state entry written before the constructor call via the existing `revert_log` mechanism (which is already threaded through the function).

In `execute_deploy` (`syscall_impls.cairo`), replace the hardcoded `failure_flag=0` with `failure_flag=is_reverted`, mirroring the pattern in `contract_call_helper`.

---

### Proof of Concept

1. Declare a Sierra contract class whose constructor always reverts (e.g., `panic_with_felt252('fail')`).
2. Submit an invoke transaction from an account that calls `deploy` syscall targeting that class.
3. The sequencer's Rust simulation returns `failure_flag=1` for the `deploy` syscall and marks the outer transaction as succeeded (caller contract handles the failure).
4. The sequencer includes the transaction in a block.
5. The Cairo OS prover executes `deploy_contract` → `select_execute_entry_point_func` returns `is_reverted=1` → `assert is_reverted = 0` panics.
6. Block proof generation fails; the network cannot advance.
