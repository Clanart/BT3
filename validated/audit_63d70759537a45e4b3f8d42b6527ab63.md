Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain residual ETH stranded on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` is an unrestricted `external payable` function that transfers the router's entire ETH balance to `msg.sender` with no access control. Because all four swap entry-points are `payable` and the `pay()` helper wraps only the exact amount the pool callback requests, any surplus native ETH sent by a user remains on the contract and is immediately claimable by an arbitrary caller in a separate transaction.

## Finding Description

`refundETH()` unconditionally sends the full contract ETH balance to `msg.sender`: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, wraps only the pool-requested `value` and leaves the remainder on the contract: [2](#0-1) 

All four swap entry-points are `external payable`, meaning users can and do send ETH with swap calls: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The `receive()` guard only blocks bare ETH pushes from non-WETH addresses; it does not prevent ETH from arriving via `msg.value` in any payable call, so the stranding path is always open: [7](#0-6) 

The intended safe pattern is `multicall{value}([swap(...), refundETH()])`, confirmed by the test suite: [8](#0-7) 

However, no enforcement exists that `refundETH()` must be called atomically within the same multicall. A user who calls `exactInputSingle{value: V}(amountIn: A)` where `V > A` — a documented and expected usage pattern — leaves `V − A` ETH on the router. Any address can then call `refundETH()` in a separate transaction and receive that ETH. The NatSpec explicitly documents the multicall+refundETH pattern as the expected flow, confirming that users are expected to send excess ETH regularly: [9](#0-8) 

Contrast with `unwrapWETH9` and `sweepToken`, which both accept a caller-supplied `recipient` parameter, preventing redirection by a third party: [10](#0-9) [11](#0-10) 

`refundETH()` is the only payment helper that uses `msg.sender` as the recipient, making it uniquely exploitable by a third-party caller.

## Impact Explanation

Direct loss of user ETH principal. Any residual native ETH on the router — from overpayment or a price-limit-truncated exact-input swap — is immediately claimable by an unprivileged attacker. The victim receives no refund and the attacker receives the full stranded balance. This constitutes a direct principal loss above Sherlock Medium thresholds.

## Likelihood Explanation

Moderate-to-high. The `pay()` function explicitly handles the `nativeBalance >= value` case, which is a normal and documented usage pattern. The NatSpec documents the multicall+refundETH pattern as the expected flow, meaning users are expected to send excess ETH regularly. A MEV bot monitoring the mempool can back-run any transaction that leaves ETH on the router. No special privileges or preconditions are required for the attacker.

## Recommendation

Replace the `msg.sender` recipient in `refundETH()` with a caller-supplied `address recipient` parameter, matching the pattern already used by `unwrapWETH9` and `sweepToken`:

```solidity
function refundETH(address recipient) external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(recipient, balance);
    }
}
```

This ensures that only the original transaction sender (who encodes `recipient` as themselves in the multicall) can direct where the refund goes, and an attacker calling in a separate transaction cannot redirect it to themselves.

## Proof of Concept

```solidity
function test_attacker_steals_residual_eth() public {
    address victim  = makeAddr("victim");
    address attacker = makeAddr("attacker");
    vm.deal(victim, 2 ether);

    // Victim calls exactInputSingle directly with 2 ETH but amountIn = 1000 wei.
    // pay() wraps 1000 wei; ~2 ETH - 1000 wei stays on router.
    vm.prank(victim);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 60,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // Router now holds ~2 ether - 1000 wei of residual ETH.
    assertGt(address(router).balance, 0, "residual eth on router");

    // Attacker calls refundETH() in a separate transaction.
    vm.prank(attacker);
    router.refundETH();

    // Attacker received victim's residual ETH.
    assertGt(attacker.balance, 0, "attacker stole eth");
    assertEq(address(router).balance, 0, "router drained");
}
```
