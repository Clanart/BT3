### Title
Child `ChunkExtra` Inherits Parent's `proposed_split` During Resharding, Causing Child Shard's First Chunk to Fail Validation — (File: `chain/chain/src/resharding/manager.rs`)

---

### Summary

In `process_memtrie_resharding_storage_update`, the child `ChunkExtra` is created by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field is never reset to `None`, so it silently inherits the parent shard's split proposal. When the child shard's first chunk is produced in the new epoch, the runtime computes `proposed_split = None` (child is freshly created, below any memory threshold), but the stored `child_chunk_extra.proposed_split` still carries the parent's non-`None` value. The invariant check in `validate_chunk_with_chunk_extra_and_receipts_root` then fires `InvalidChunkHeaderShardSplit`, permanently blocking the child shard from accepting its first chunk.

---

### Finding Description

In `chain/chain/src/resharding/manager.rs`, `process_memtrie_resharding_storage_update` creates the child `ChunkExtra` as follows:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// proposed_split is NOT reset — it is silently inherited from parent
``` [1](#0-0) 

The `proposed_split` field in `ChunkExtraV5` records the trie-split proposal computed during the **previous** chunk application: [2](#0-1) 

At the resharding block (last block of epoch N+1), `compute_proposed_split` is called for the parent shard. Because the parent shard is still above the memory threshold (that is why it was selected for resharding), it returns `Some(TrieSplit{...})`. This value is stored in `parent_chunk_extra.proposed_split`. The child `ChunkExtra` clones this value verbatim.

In epoch N+2, when the child shard's first chunk is produced, the runtime calls `compute_proposed_split` for the child shard. The child is freshly created and its memory usage is well below the threshold, so it returns `None`. The chunk header is signed with `proposed_split = None`.

`validate_chunk_with_chunk_extra_and_receipts_root` then compares:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [3](#0-2) 

`prev_chunk_extra.proposed_split()` = `Some(TrieSplit{...})` (inherited from parent)  
`chunk_header.proposed_split()` = `None` (freshly computed for child)  
→ **`InvalidChunkHeaderShardSplit` is returned, and the child shard's first chunk is rejected.**

The codebase itself acknowledges the root cause in the dynamic resharding documentation:

> "allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."



The stated mitigation (`min_epochs_between_resharding > 0`) prevents a *second* resharding from being scheduled too soon, but it does **not** prevent the parent from proposing a split in epoch N+1 (the preparation epoch). If the parent is still above the memory threshold in N+1, its final `ChunkExtra` will carry a non-`None` `proposed_split`, and the child will inherit it regardless of the cooldown.

---

### Impact Explanation

**High.** The child shard's very first chunk is permanently rejected by every validator running `validate_chunk_with_chunk_extra_and_receipts_root`. No chunk can ever be accepted for the child shard, halting all transaction processing on it. Because the child `ChunkExtra` is written atomically at the resharding block and there is no in-protocol repair path, the child shard remains blocked until a node restart with a patched binary or a manual store fix. All accounts assigned to the child shard lose access to their funds and contracts for the duration of the outage.

---

### Likelihood Explanation

**Medium.** The bug requires `ProtocolFeature::DynamicResharding` to be active. Once active, it fires whenever the parent shard's memory usage remains above the resharding threshold at the last block of the preparation epoch (N+1). Because the parent was selected for resharding precisely because it was above the threshold, and memory does not drop significantly in one epoch, this condition is the common case, not the exception. The scenario is therefore expected to trigger on every dynamic resharding event in production once the feature is enabled.

---

### Recommendation

After cloning the parent `ChunkExtra` and before saving it as the child's initial `ChunkExtra`, explicitly reset `proposed_split` to `None`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// ADD: child shard has no prior split proposal
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

This mirrors the correct pattern used in `ApplyChunkResult::to_chunk_extra`, which sets `proposed_split` from the freshly computed runtime result rather than inheriting any prior value: [6](#0-5) 

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` with `min_epochs_between_resharding = 1`.
2. Deploy enough contract data to a single shard to push its memory usage above `memory_usage_threshold`.
3. Wait for the epoch manager to select the shard for resharding (epoch N). The parent's `ChunkExtra.proposed_split` becomes `Some(TrieSplit{...})` at the last block of epoch N.
4. In epoch N+1 (preparation), the parent shard remains above the threshold; `compute_proposed_split` again returns `Some(TrieSplit{...})` at the last block of N+1. This value is stored in `parent_chunk_extra.proposed_split`.
5. At the epoch N+1/N+2 boundary, `process_memtrie_resharding_storage_update` clones the parent `ChunkExtra` into both child `ChunkExtra` records. Both children inherit `proposed_split = Some(TrieSplit{...})`.
6. In epoch N+2, the chunk producer for the left child shard calls `compute_proposed_split` → returns `None` (child memory is half of parent, below threshold). The chunk header is signed with `proposed_split = None`.
7. `validate_chunk_with_chunk_extra_and_receipts_root` compares `Some(TrieSplit{...})` (stored) vs `None` (header) → `InvalidChunkHeaderShardSplit`. The chunk is rejected by all validators. The child shard is permanently halted.
