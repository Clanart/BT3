### Title
Unchecked Return Value of `non_reverting_select_execute_entry_point_func` in `charge_fee` Allows Silent Fee Transfer Failure — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in `transaction_impls.cairo` calls `non_reverting_select_execute_entry_point_func` to execute the ERC20 fee transfer but completely discards its return value `(retdata_size, retdata, is_deprecated)`. Other callers of the same function within the same file explicitly capture and assert on the return value. If the fee transfer execution fails or reverts internally, the OS has no mechanism to detect this and continues processing the transaction as if the fee was successfully paid — resulting in the sequencer receiving no fee for the executed transaction.

---

### Finding Description

`charge_fee` is responsible for deducting the transaction fee from the user's account by invoking the ERC20 `transfer` entry point on the fee token contract:

```cairo
// transaction_impls.cairo lines 160-163
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
```

The return tuple `(retdata_size, retdata, is_deprecated)` is silently dropped. The function is named "non_reverting" because it handles internal reverts without propagating them to the caller — meaning if the ERC20 transfer reverts, the state changes are rolled back internally, but execution continues normally in the caller.

This is directly inconsistent with every other call site of the same function in the same file:

- `execute_declare_transaction` (lines 804–812): captures and asserts `retdata[0] = VALIDATED`
- `execute_deploy_account_transaction` (lines 677–684): captures and asserts `retdata[0] = VALIDATED`

Both of these callers explicitly check the return value to confirm the execution produced the expected result. `charge_fee` does not.

The analog to the original report is exact: just as `VaultCore._redeem` calls `CompoundStrategy.withdraw` and ignores a zero return (continuing to burn OUSD), `charge_fee` calls the ERC20 transfer entry point and ignores the return value — continuing to finalize the transaction even if the fee was never transferred.

---

### Impact Explanation

`non_reverting_select_execute_entry_point_func` is designed to handle internal reverts without aborting the OS. If the ERC20 fee transfer reverts (for any reason), the function rolls back the storage changes to the fee token contract internally and returns — but `charge_fee` does not inspect the return data to detect this failure. The OS then proceeds to finalize the transaction (state committed, nonce incremented, outputs written) without the sequencer having received any fee.

This constitutes **direct loss of funds**: the sequencer executes and finalizes a transaction for which it receives zero compensation. At scale or under targeted conditions, this drains sequencer revenue and undermines the economic security of the network.

---

### Likelihood Explanation

The fee token (STRK or ETH) is a specific, trusted ERC20 contract. Under normal conditions, if the user's balance passes the pre-execution validation check, the transfer should succeed. However, the following realistic conditions can cause the transfer to fail silently:

1. **Gas exhaustion**: `DEFAULT_INITIAL_GAS_COST` is a fixed constant. If the fee token contract's `transfer` implementation consumes more gas than this budget (e.g., after an upgrade adding hooks or logic), the call runs out of gas and reverts internally — undetected by `charge_fee`.
2. **Fee token contract upgrade**: The fee token is upgradeable. A new implementation with a bug or a changed `transfer` signature could cause silent failures.
3. **Edge-case arithmetic in the ERC20**: Rounding or overflow in the fee token's balance arithmetic at boundary values could cause a revert that is swallowed.

Likelihood is **medium** given the dependency on the fee token contract's behavior, but the attack surface is reachable by any transaction sender whose fee transfer happens to trigger one of the above conditions.

---

### Recommendation

Capture and validate the return value of `non_reverting_select_execute_entry_point_func` inside `charge_fee`. At minimum, assert that the execution did not produce an error retdata (i.e., the transfer did not revert). If the fee transfer fails, the transaction should be aborted rather than silently finalized:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
// Assert the transfer did not revert (retdata_size == 0 for success, or check specific return).
assert retdata_size = 0;  // or appropriate success check for the ERC20 transfer
```

Additionally, audit all other call sites of `non_reverting_select_execute_entry_point_func` (e.g., `execute_invoke_function_transaction` line 345, `execute_l1_handler_transaction` line 446) for the same pattern.

---

### Proof of Concept

1. Deploy a transaction (invoke, L1 handler, or deploy-account) where the fee token's `transfer` call reverts internally — for example, by triggering a gas exhaustion in the fee token contract's `transfer` function (e.g., via a fee token upgrade that adds expensive logic).
2. `non_reverting_select_execute_entry_point_func` handles the revert internally: fee token storage changes are rolled back, the function returns normally.
3. `charge_fee` ignores the return value and returns normally.
4. The OS finalizes the transaction: nonce is incremented, state is committed, outputs are written.
5. The sequencer's fee token balance is **not increased** — the fee was never paid.
6. The attacker's transaction executes at zero cost.
