### Title
STRK Native Fees Permanently Locked in StarkNet OmniBridge — No Withdrawal Mechanism - (File: starknet/src/omni_bridge.cairo)

---

### Summary

The StarkNet `OmniBridge` contract collects STRK tokens as native fees during `init_transfer`, transferring them into the contract itself. However, the contract exposes no function to withdraw or distribute these accumulated STRK tokens. They are permanently locked with no on-chain recovery path in the current code.

---

### Finding Description

In `init_transfer`, when a user specifies `native_fee > 0`, the contract pulls STRK tokens from the caller directly into `get_contract_address()`:

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
``` [1](#0-0) 

The entire `IOmniBridge` interface is:

```cairo
fn log_metadata(...)
fn deploy_token(...)
fn fin_transfer(...)
fn init_transfer(...)
fn upgrade_token(...)
fn set_pause_flags(...)
fn pause_all(...)
fn get_token_address(...)
fn is_bridge_token(...)
fn is_transfer_finalised(...)
``` [2](#0-1) 

None of these functions withdraw or distribute accumulated STRK tokens. The `fin_transfer` function only transfers `payload.token_address` tokens to `payload.recipient` — it has no special case for STRK fee distribution: [3](#0-2) 

**Contrast with EVM**: The EVM `OmniBridge.sol` explicitly handles native fee recovery via `finTransfer` with `tokenAddress = address(0)`, which sends accumulated ETH to the fee recipient. The StarkNet contract has no equivalent mechanism. [4](#0-3) 

On the NEAR side, when a StarkNet-originated transfer is finalized, `send_fee_internal` **mints** a wrapped STRK token on NEAR for the relayer — it does not transfer the actual STRK tokens held in the StarkNet contract: [5](#0-4) 

This means the STRK tokens collected on StarkNet are never distributed through the normal fee-claim flow. They accumulate in the contract indefinitely.

---

### Impact Explanation

Every `init_transfer` call with `native_fee > 0` on StarkNet deposits STRK tokens into the bridge contract. These tokens have no withdrawal path in the current contract. All accumulated STRK native fee revenue is permanently locked — irrecoverable without a contract upgrade. This matches the allowed impact: **Critical — permanent freezing/irrecoverable lock of protocol funds in bridge flow**.

---

### Likelihood Explanation

High. Native fees are a standard part of the bridge protocol — users pay them on every cross-chain transfer to compensate relayers. Any user calling `init_transfer` with `native_fee > 0` contributes to the locked balance. This is a routine protocol operation, not an edge case.

---

### Recommendation

Add an admin-gated withdrawal function for accumulated STRK (and any other ERC-20 tokens held by the contract):

```cairo
fn withdraw_token(
    ref self: ContractState,
    token_address: ContractAddress,
    recipient: ContractAddress,
    amount: u128,
) {
    self.accesscontrol.assert_only_role(DEFAULT_ADMIN_ROLE);
    let success = IERC20Dispatcher { contract_address: token_address }
        .transfer(recipient, amount.into());
    assert(success, 'ERR_WITHDRAW_FAILED');
}
```

Alternatively, mirror the EVM approach: allow `fin_transfer` with `token_address = strk_token_address` to distribute accumulated STRK fees to a designated fee recipient, signed by the NEAR MPC.

---

### Proof of Concept

1. User calls `init_transfer` on StarkNet with `native_fee = 1_000_000` (1 STRK).
2. The contract executes `transfer_from(caller, get_contract_address(), 1_000_000)` — STRK is now held by the bridge.
3. On NEAR, `send_fee_internal` mints a wrapped STRK token for the relayer on NEAR — the actual STRK on StarkNet is untouched.
4. Protocol team attempts to recover accumulated STRK fees from the StarkNet contract.
5. No function exists to do so. The STRK balance in the contract grows with every fee-paying transfer and is permanently locked. [6](#0-5) [7](#0-6)

            let origin_nonce = self.current_origin_nonce.read() + 1;
