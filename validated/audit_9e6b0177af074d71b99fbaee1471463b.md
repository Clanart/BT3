### Title
Unvalidated `l1_gas_price_wei` in `ProposalInit` Causes Wrong `l2_gas_price_wei` Committed to Block Hash - (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates only the FRI-denominated L1 gas prices against the oracle and explicitly discards the computed wei prices without comparing them to the proposer-supplied `l1_gas_price_wei` / `l1_data_gas_price_wei`. A malicious proposer can set `l1_gas_price_wei` to an arbitrary value while keeping `l1_gas_price_fri` within the accepted margin. `convert_to_sn_api_block_info` then derives `eth_to_fri_rate` from the manipulated wei price, producing a wrong `l2_gas_price_wei`. For Starknet version ≥ 0.13.4, `gas_prices_to_hash` commits both `l1_gas_price.price_in_wei` and `l2_gas_price.price_in_wei` into the block hash, so the wrong values propagate into the `PartialBlockHash` / `proposal_commitment` that every validator accepts and that is ultimately committed to L1.

---

### Finding Description

**Step 1 – Validation gap.**

In `is_block_info_valid` the validator fetches oracle-derived wei prices but immediately discards them:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
```

Only the FRI prices are checked:

```rust
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, l1_gas_price_margin_percent))
```

`init_proposed.l1_gas_price_wei` and `init_proposed.l1_data_gas_price_wei` are never compared to the oracle. [1](#0-0) 

**Step 2 – Denomination mismatch in `convert_to_sn_api_block_info`.**

`PreviousBlockInfo::from(init)` copies the unvalidated `init.l1_gas_price_wei` directly into the struct used to derive `eth_to_fri_rate`:

```rust
impl From<&ProposalInit> for PreviousBlockInfo {
    fn from(init: &ProposalInit) -> Self {
        Self {
            l1_prices_wei: L1PricesInWei {
                l1_gas_price: init.l1_gas_price_wei,   // ← attacker-controlled
                ...
            },
            l1_prices_fri: L1PricesInFri {
                l1_gas_price: init.l1_gas_price_fri,   // ← oracle-validated
                ...
            },
        }
    }
}
``` [2](#0-1) 

`calculate_eth_to_fri_rate` then computes:

```
eth_to_fri_rate = l1_gas_price_fri × WEI_PER_ETH / l1_gas_price_wei
``` [3](#0-2) 

Because `l1_gas_price_fri` is oracle-validated but `l1_gas_price_wei` is attacker-controlled, the derived rate is wrong. `l2_gas_price_wei` is then computed from this wrong rate:

```rust
let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)
``` [4](#0-3) 

**Step 3 – Both wrong values enter the block hash.**

`PartialBlockHashComponents::new` reads gas prices directly from the `BlockInfo` produced by `convert_to_sn_api_block_info`:

```rust
l1_gas_price: block_info.gas_prices.l1_gas_price_per_token(),   // contains attacker l1_gas_price_wei
l2_gas_price: block_info.gas_prices.l2_gas_price_per_token(),   // contains derived wrong l2_gas_price_wei
``` [5](#0-4) 

For version ≥ 0.13.4, `gas_prices_to_hash` chains both wei values into the Poseidon hash that becomes the `gas_prices_hash` field of the block hash:

```rust
.chain(&l1_gas_price.price_in_wei.0.into())
...
.chain(&l2_gas_price.price_in_wei.0.into())
``` [6](#0-5) 

Because every validator runs the same `convert_to_sn_api_block_info` on the same `ProposalInit`, all nodes converge on the same wrong `PartialBlockHash` / `proposal_commitment`, the proposal passes consensus, and the corrupted block hash is committed to L1.

**Concrete denomination-mismatch example:**

| Value | Oracle truth | Attacker-supplied |
|---|---|---|
| `l1_gas_price_wei` | 100 gwei | **200 gwei** |
| `l1_gas_price_fri` | 100 fri (within margin) | 100 fri (passes check) |
| Derived `eth_to_fri_rate` | 10^18 | **5×10^17** (halved) |
| Derived `l2_gas_price_wei` | X wei | **2X wei** (doubled) |
| Block-hash gas-prices field | correct | **wrong** |

This is the direct sequencer analog of the external report: a USD-denominated amount (`poolBadDebt`) was used as a raw token amount without oracle conversion; here, a FRI-denominated price is used to derive a wei amount using an oracle-bypassed rate, and the wrong wei amount is committed to the block hash without any oracle cross-check.

---

### Impact Explanation

**Wrong block hash committed to L1 (Critical).** `l1_gas_price.price_in_wei` and `l2_gas_price.price_in_wei` are both chained into the Poseidon block hash for version ≥ 0.13.4. A manipulated `l1_gas_price_wei` produces a wrong `PartialBlockHash` that is accepted by all validators and stored on-chain. The SNOS proof is generated over this wrong hash; the L1 verifier accepts it, permanently anchoring an incorrect gas-price commitment.

**Wrong fee for ETH-paying transactions (Critical economic).** The `BlockInfo` passed to the blockifier has `eth_gas_prices.l1_gas_price = l1_gas_price_wei` (attacker value) and `eth_gas_prices.l2_gas_price = l2_gas_price_wei` (derived wrong value). Deprecated v0/v1 transactions whose fees are denominated in wei are charged against these wrong prices, producing incorrect `actual_fee` values that are also committed to the receipt tree.

---

### Likelihood Explanation

Any consensus validator whose turn it is to propose a block can trigger this. No special privilege beyond being a scheduled proposer is required. The `l1_gas_price_wei` field is freely settable in the `ProposalInit` protobuf message; the only constraint is that `l1_gas_price_fri` stays within the oracle margin, which is easy to satisfy independently.

---

### Recommendation

In `is_block_info_valid`, after computing `_l1_gas_prices_wei`, compare the proposed `l1_gas_price_wei` and `l1_data_gas_price_wei` against the oracle-derived wei values using the same `within_margin` check already applied to the FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

if !(within_margin(init_proposed.l1_gas_price_wei,     l1_gas_prices_wei.l1_gas_price,      margin)
  && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

This mirrors the existing FRI validation at lines 302–318 and closes the denomination-mismatch gap before the wei values are used to derive `eth_to_fri_rate` and committed to the block hash.

---

### Proof of Concept

1. Attacker is the scheduled proposer for block N.
2. Oracle reports `l1_gas_price_wei = 100`, `eth_to_fri_rate = 10^18`, so `l1_gas_price_fri = 100`.
3. Attacker constructs `ProposalInit` with `l1_gas_price_fri = 100` (passes `within_margin`) and `l1_gas_price_wei = 200` (not validated).
4. Every validator calls `is_block_info_valid`: FRI check passes; wei values are discarded (`_l1_gas_prices_wei`). [7](#0-6) 
5. Every validator calls `convert_to_sn_api_block_info(init)`:
   - `PreviousBlockInfo::from(init)` stores `l1_gas_price_wei = 200`. [2](#0-1) 
   - `calculate_eth_to_fri_rate` → `100 × 10^18 / 200 = 5×10^17` (half the real rate). [3](#0-2) 
   - `l2_gas_price_wei = l2_gas_price_fri × 10^18 / 5×10^17 = 2 × correct_value`. [8](#0-7) 
6. Batcher executes block with `eth_gas_prices.l1_gas_price = 200` and `eth_gas_prices.l2_gas_price = 2×correct`; ETH-paying transactions
