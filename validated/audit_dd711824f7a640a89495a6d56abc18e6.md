### Title
EIP-712 Legacy Signature Does Not Commit to `timeout_height`, Enabling Replay of Expired Transactions — (`ante/cosmos/eip712.go`, `ethereum/eip712/eip712_legacy.go`)

### Summary
The legacy EIP-712 signature verification path builds Amino sign bytes that include `timeout_height` when non-zero, but the `Tx` EIP-712 type schema permanently omits that field. Because EIP-712 hashing ignores message fields not declared in the type schema, the computed `sigHash` is identical regardless of the `timeout_height` value in the actual transaction. An attacker who obtains a broadcast-but-unexecuted signed tx can strip or alter its `TimeoutHeight` without invalidating the signature.

### Finding Description
In `VerifySignature` (`ante/cosmos/eip712.go:197-207`), `txBytes` is produced by `legacytx.StdSignBytes(..., tx.GetTimeoutHeight(), ...)`. When `TimeoutHeight != 0`, the resulting Amino JSON contains `"timeout_height": "100"`. This JSON is passed to `LegacyWrapTxToTypedData`, which unmarshals it into `txData` (a raw `map[string]interface{}`) and assigns it as `typedData.Message`. However, the `Tx` type schema defined in `extractMsgTypes` (`eip712_legacy.go:129-138`) does not list `timeout_height` — the field is commented out. The go-ethereum `TypedDataAndHash` implementation only encodes fields that appear in the declared type; extra fields in the message map are silently ignored. Therefore the EIP-712 hash is invariant to `timeout_height`.

### Impact Explanation
A user who signs a tx with `TimeoutHeight=N` intends the tx to be invalid after block N. If the tx is not included before block N (e.g., due to congestion), the user's sequence number is unchanged. An attacker who observed the signed tx in the mempool or via a failed broadcast can re-encode the protobuf tx body with `TimeoutHeight=0`, resubmit it, and the `VerifySignature` check passes because the EIP-712 hash is the same. The tx executes with the user's funds/messages, bypassing the expiry the user relied on. This is a concrete EIP-712 authorization bypass enabling replay of transactions the signer intended to expire.

### Likelihood Explanation
The attacker only needs to observe a signed legacy EIP-712 tx in the public mempool (standard for any Cosmos chain) and re-encode the protobuf `TxBody` with a different `TimeoutHeight`. No privileged access, key material, or validator cooperation is required. The sequence number constraint is satisfied automatically because the original tx was never executed.

### Recommendation
Restore `timeout_height` to the `Tx` EIP-712 type schema in both `eip712_legacy.go` and `types.go`. The comment "was not getting filled with the legacyTx" reflects a historical state that is no longer accurate — `StdSignBytes` is called with `tx.GetTimeoutHeight()` and does emit the field when non-zero. Alternatively, reject legacy EIP-712 txs that carry a non-zero `TimeoutHeight` in the ante handler until the type schema is corrected.

### Proof of Concept
1. Construct a legacy EIP-712 tx with `TimeoutHeight=500` and sign it.
2. Compute `TypedDataAndHash` for that tx → hash H₁.
3. Construct an identical tx with `TimeoutHeight=0` (or any other value) using the same signature.
4. Compute `TypedDataAndHash` → hash H₂.
5. Assert H₁ == H₂ — the hashes are equal because `timeout_height` is absent from the `Tx` type schema.
6. Submit the modified tx (with `TimeoutHeight=0`) after block 500; `VerifySignature` passes and the tx commits.
