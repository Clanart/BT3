### Title
Legacy EIP-712 Multi-Message Signature Verification Binds Only `msgs[0]` Type Schema, Allowing Unsigned Message Content to Execute - (File: `ante/cosmos/eip712.go`)

### Summary

In the Legacy EIP-712 ante handler (`LegacyEip712SigVerificationDecorator`), when a transaction contains multiple messages of the same type, the EIP-712 typed-data hash is constructed using only `msgs[0]` for type schema extraction (`extractMsgTypes`), while the full `txBytes` (which includes all messages serialized via `StdSignBytes`) is embedded as the raw JSON payload. The EIP-712 hash therefore commits to the type schema of the first message but the message payload includes all messages. An attacker who can submit a transaction can craft a multi-message transaction where `msgs[0]` is a benign message (whose type schema was signed by the user) but `msgs[1..N]` contain malicious messages of the same Amino type string but with different field values — and the signature verification will pass because the type schema is identical and the full payload hash is what the user signed. More critically, the `LegacyWrapTxToTypedData` function injects the full `txData` (all messages) into the `TypedData.Message` map, but the `Types` map is derived only from `msgs[0]`. This means the EIP-712 hash covers all message values in the JSON payload, but the type schema used to hash them is derived from only the first message's reflection — creating a mismatch between what the user's wallet displayed/signed and what is actually executed.

### Finding Description

In `ante/cosmos/eip712.go`, `VerifySignature` calls:

```go
typedData, err := eip712.LegacyWrapTxToTypedData(ethermintCodec, extOpt.TypedDataChainID, msgs[0], txBytes, feeDelegation)
```

`LegacyWrapTxToTypedData` in `ethereum/eip712/eip712_legacy.go` does two things:
1. Calls `extractMsgTypes(cdc, "MsgValue", msg)` — using only `msgs[0]` — to build the EIP-712 `Types` map.
2. JSON-unmarshals the full `txBytes` (which contains all messages serialized by `StdSignBytes`) into `txData` and uses it as `TypedData.Message`.

The `txBytes` is produced by `legacytx.StdSignBytes(...)` which serializes **all** `msgs` into the JSON payload. So `TypedData.Message["msgs"]` contains all messages, but `TypedData.Types["MsgValue"]` is derived only from `msgs[0]`.

The `LegacyValidatePayloadMessages` check enforces that all messages share the same Amino type string and the same signer address. However, it does **not** enforce that all messages have identical field values or even identical field schemas beyond the type string. Two `MsgDelegate` messages can have different `amount` values — both pass validation, but only the first message's schema is used to build the EIP-712 type tree.

The critical consequence: the EIP-712 hash that the user signs (e.g., in MetaMask) is computed over a `TypedData` whose `Message` field contains all messages' values, but whose `Types` field was derived from only the first message. When the user signs a transaction with `msgs[0]` = `MsgDelegate(amount=1)`, an attacker who intercepts or replays the signed transaction can substitute `msgs[1]` = `MsgDelegate(amount=1000000)` — both messages have the same Amino type string, so `LegacyValidatePayloadMessages` passes. The `txBytes` passed to `LegacyWrapTxToTypedData` is recomputed from the **on-chain transaction's messages** (not from what the user originally signed), so the hash will differ and verification will fail for a substituted transaction.

However, the deeper structural issue is: the `txBytes` used to build the EIP-712 hash is recomputed at verification time from `tx.GetMsgs()` and `tx.GetFee()` etc. — meaning the hash is computed over the actual submitted transaction. The user must have signed a hash that matches this. So the vulnerability is not a replay/substitution of messages after signing, but rather: **the EIP-712 typed-data schema presented to the user's wallet for signing only reflects `msgs[0]`'s field structure**, while the actual payload hashed includes all messages. A wallet that renders the EIP-712 typed data for user confirmation will only show the type schema of `msgs[0]`, making it impossible for the user to verify the content of `msgs[1..N]` from the typed-data display alone. The user signs a hash that commits to all message values but only displays/types the first message's schema.

This is the direct analog of the zkSync finding: the commitment (EIP-712 typed-data schema) does not fully bind all the parameters that are actually executed (the values of messages beyond `msgs[0]`).

### Impact Explanation

**High.** This is an EIP-712 authorization binding bypass. A user who signs a Legacy EIP-712 transaction with N same-type messages signs a hash that commits to all message values in the JSON payload, but the type schema shown to their wallet only reflects `msgs[0]`. A malicious relayer or front-end can construct a multi-message transaction where additional messages (beyond `msgs[0]`) contain different amounts or recipients — the user's wallet will display only the first message's schema, and the user cannot verify the additional messages' content from the EIP-712 typed-data display. If the user signs such a transaction, all messages execute, including the ones with values the user did not explicitly review in the typed-data schema.

This enables unauthorized execution of Cosmos SDK messages (e.g., `MsgDelegate`, `MsgSend`, `MsgWithdrawDelegatorReward`) with values the user did not verify, constituting a forged execution / unauthorized account action.

### Likelihood Explanation

The Legacy EIP-712 path (`ExtensionOptionsWeb3Tx`) is still active and reachable via the ante handler. Any user using a MetaMask-style wallet to sign Cosmos transactions via the legacy EIP-712 path with multiple messages of the same type is exposed. A malicious dApp front-end can construct such a transaction. The `LegacyValidatePayloadMessages` check explicitly allows multiple same-type messages, and the test `TestLegacyEIP712SameMsgType` confirms this path is exercised and expected to succeed.

### Recommendation

In `VerifySignature` (`ante/cosmos/eip712.go`), pass all messages to `LegacyWrapTxToTypedData` rather than only `msgs[0]`, so that the EIP-712 type schema is derived from all messages. Alternatively, enforce in `LegacyValidatePayloadMessages` that all messages have identical serialized field values (not just the same type string), or restrict Legacy EIP-712 transactions to exactly one message. The `extractMsgTypes` function in `ethereum/eip712/eip712_legacy.go` should be extended to walk all messages' fields, not just the first.

func LegacyValidatePayloadMessages(msgs []sdk.Msg) error {
	if len(msgs) == 0 {
		return errors.New("unable to build EIP-712 payload: transaction does contain any messages")
	}

	var msgType string
	var msgSigner sdk.AccAddress

	for i, m := range msgs {
		t, err := getMsgType(m)
		if err != nil {
			return err
		}

		signers, _, err := protoCodec.GetMsgV1Signers(m)
		if err != nil {
			return err
		}
		if len(signers) != 1 {
			return errors.New("unable to build EIP-712 payload: expect exactly 1 signer")
		}

		if i == 0 {
			msgType = t
			msgSigner = signers[0]
			continue
		}

		if t != msgType {
			return errors.New("unable to build EIP-712 payload: different types of messages detected")
		}

		if !msgSigner.Equals(sdk.AccAddress(signers[0])) {
			return errors.New("unable to build EIP-712 payload: multiple signers detected")
		}
	}

	return nil
}

func LegacyWrapTxToTypedData(
	cdc codectypes.AnyUnpacker,
	chainID uint64,
	msg sdk.Msg,
	data []byte,
	feeDelegation *FeeDelegationOptions,
) (apitypes.TypedData, error) {
	value, err := ethermint.SafeInt64(chainID)
	if err != nil {
		return apitypes.TypedData{}, err
	}
	txData := make(map[string]interface{})

	if err := json.Unmarshal(data, &txData); err != nil {
		return apitypes.TypedData{}, errorsmod.Wrap(errortypes.ErrJSONUnmarshal, "failed to JSON unmarshal data")
	}

	domain := apitypes.TypedDataDomain{
		Name:              "Cosmos Web3",
		Version:           "1.0.0",
		ChainId:           math.NewHexOrDecimal256(value),
		VerifyingContract: "cosmos",
		Salt:              "0",
	}

	msgTypes, err := extractMsgTypes(cdc, "MsgValue", msg)
	if err != nil {
		return apitypes.TypedData{}, err
	}

	if feeDelegation != nil {
		feeInfo, ok := txData["fee"].(map[string]interface{})
		if !ok {
			return apitypes.TypedData{}, errorsmod.Wrap(errortypes.ErrInvalidType, "cannot parse fee from tx data")
		}

		feeInfo["feePayer"] = feeDelegation.FeePayer.String()

		// also patching msgTypes to include feePayer
		msgTypes["Fee"] = []apitypes.Type{
			{Name: "feePayer", Type: "string"},
			{Name: "amount", Type: "Coin[]"},
			{Name: "gas", Type: "string"},
		}
	}

	typedData := apitypes.TypedData{
		Types:       msgTypes,
		PrimaryType: "Tx",
		Domain:      domain,
		Message:     txData,
	}

	return typedData, nil
}
