### Title
Felt Arithmetic Underflow in `check_proof_facts` Causes OS Program Panic on Early Blocks — (`execution_constraints.cairo`)

### Summary

`check_proof_facts` in `execution_constraints.cairo` performs an unguarded felt subtraction `current_block_number - STORED_BLOCK_HASH_BUFFER` (where `STORED_BLOCK_HASH_BUFFER = 10`). When `current_block_number < 10`, this subtraction wraps around in the Cairo field to a value near the field prime P ≈ 2²⁵¹. The subsequent `assert_nn_le` call then fails because the wrapped value exceeds the range-check domain [0, 2¹²⁸), causing the OS program to panic and making it impossible to generate a valid block proof.

### Finding Description

In `execution_constraints.cairo`, the function `check_proof_facts` validates the `proof_facts` field of an invoke transaction: [1](#0-0) 

```cairo
// Validate that the proof facts block number is not too recent.
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

`STORED_BLOCK_HASH_BUFFER` is the constant `10`: [2](#0-1) 

When `current_block_number < 10`, the expression `current_block_number - STORED_BLOCK_HASH_BUFFER` wraps around in the Cairo field to `P - (10 - current_block_number)`, a value near 2²⁵¹. Cairo's `assert_nn_le(a, b)` internally calls `assert_le(a, b)`, which places `b - a` into a range-check cell. Since range checks only accept values in [0, 2¹²⁸), a `b` value near 2²⁵¹ causes an immediate range-check failure, panicking the OS program.

This is called from `execute_invoke_transaction` with user-supplied `proof_facts`: [3](#0-2) 

```cairo
check_proof_facts(
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
    current_block_number=block_context.block_info_for_execute.block_number,
    virtual_os_config_hash=block_context.os_global_context.virtual_os_config_hash,
);
```

The `proof_facts_size` and `proof_facts` are loaded from the transaction via hints (`TxProofFacts`). Any user can submit an invoke transaction with a non-zero `proof_facts_size`.

**Contrast with the correctly guarded sibling:** `execute_get_block_hash` performs the same subtraction at line 753 but is protected by a hint-driven branch that intercepts the case where `current_block_number < STORED_BLOCK_HASH_BUFFER` before reaching the subtraction: [4](#0-3) 

`check_proof_facts` has no equivalent guard.

### Impact Explanation

When the OS program panics, the prover cannot produce a valid STARK proof for the block. The sequencer must discard the block and retry. If an attacker repeatedly submits invoke transactions with non-empty `proof_facts` during the first 10 blocks of a new network deployment, every block that includes such a transaction will fail to prove, halting block finalization. This matches the **High — Network not being able to confirm new transactions (total network shutdown)** impact class.

### Likelihood Explanation

The trigger window is the first 10 blocks of any new StarkNet network deployment (block numbers 0–9). Within that window, any unprivileged user who submits an invoke transaction with a non-zero `proof_facts` field triggers the panic. The `proof_facts` field is a standard part of the invoke transaction format and requires no special privilege to populate. The sequencer has no in-protocol mechanism to reject such transactions before attempting to prove the block.

### Recommendation

Mirror the guard used in `execute_get_block_hash`. Before computing the subtraction, check whether `current_block_number >= STORED_BLOCK_HASH_BUFFER` and return early (or write a failure response) if not:

```cairo
// Guard against underflow on early blocks.
let is_block_old_enough = is_nn(current_block_number - STORED_BLOCK_HASH_BUFFER);
if (is_block_old_enough == FALSE) {
    // No valid proof facts can exist yet; reject.
    assert 1 = 0;  // or write appropriate failure
    return ();
}
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

Alternatively, restructure the comparison to use only addition, analogous to the fix suggested in the external report:

```cairo
// Equivalent: base_block_number + STORED_BLOCK_HASH_BUFFER <= current_block_number
assert_nn_le(
    os_output_header.base_block_number + STORED_BLOCK_HASH_BUFFER, current_block_number
);
```

This eliminates the subtraction entirely and is safe for all block numbers.

### Proof of Concept

1. Deploy a new StarkNet network (block number starts at 0).
2. During blocks 0–9, submit an invoke transaction (version 3) with `proof_facts_size > 0` and any non-empty `proof_facts` payload that passes the earlier header checks (valid `proof_version`, `proof_variant`, `program_hash`).
3. The sequencer includes the transaction in a block.
4. The OS program reaches `check_proof_facts` → computes `current_block_number - 10` → felt wraps to `P - (10 - current_block_number)` → `assert_nn_le` places this value into a range-check cell → range-check rejects it (value > 2¹²⁸) → OS program panics.
5. The prover cannot generate a proof; the block cannot be finalized.
6. Repeat for every block in the 0–9 window to sustain the halt. [5](#0-4)

    // Proof version and variant are for future compatibility.

    }

    // Validate that the proof facts block number is not too recent.
    // (This is a sanity check - the following non-zero check ensures that the block hash is
