Audit Report

## Title
Prior User's Stranded ETH Silently Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses `address(this).balance` — the router's **total** native ETH balance — when deciding how much ETH to wrap for a WETH payment. Any ETH left on the router from a prior user (e.g., excess `msg.value` not reclaimed via `refundETH`) is silently consumed as part of a subsequent user's WETH swap, permanently stealing the prior user's ETH. An attacker can actively exploit this by calling a WETH swap with `msg.value = 0` and `amountIn` equal to the stranded ETH amount.

## Finding Description

In `PeripheryPayments.pay`, the WETH branch reads the router's entire native balance: [1](#0-0) 

`nativeBalance` at line 74 is `address(this).balance` — the router's total ETH, not the current transaction's `msg.value`. It includes ETH deposited by prior callers in earlier transactions.

The `multicall` function is `payable` and batches arbitrary calls via `delegatecall`: [2](#0-1) 

`exactInputSingle` and `exactOutputSingle` are also `payable`: [3](#0-2) [4](#0-3) 

The `receive()` guard only blocks direct ETH sends from non-WETH addresses; it does not prevent ETH accumulation via `msg.value` in payable swap functions: [5](#0-4) 

`refundETH()` is an opt-in call that returns the entire router balance to `msg.sender`: [6](#0-5) 

If a user omits `refundETH()` after an `exactOutputSingle` with ETH overpayment, the excess ETH remains on the router indefinitely and is available to any subsequent caller's `pay()` invocation.

**Active theft path:** Attacker observes router holds `X` ETH (via on-chain balance check), calls `exactInputSingle(WETH, amountIn=X, msg.value=0)`. The swap callback fires `pay(WETH, attacker, pool, X)`. At line 75, `nativeBalance >= value` is true, so the router wraps all `X` ETH and sends it to the pool; the attacker receives output tokens and pays zero WETH from their own balance.

## Impact Explanation

Direct, permanent loss of user principal (ETH). Any ETH stranded on the router — from `exactOutputSingle` overpayment, `exactInputSingle` with unused `msg.value`, or any payable multicall without `refundETH` — can be fully extracted by an unprivileged attacker in a single transaction. This is a complete theft of stranded ETH with no recovery path. Severity: **Critical**.

## Likelihood Explanation

- `exactOutputSingle` with ETH overpayment is a standard usage pattern; the exact consumed amount is only known after the swap settles, making overpayment routine.
- Omitting `refundETH()` from a `multicall` bundle is a common user mistake.
- Active exploitation requires no special privileges: any caller can check `address(router).balance` on-chain and immediately drain it with a zero-`msg.value` WETH swap sized to the stranded amount.
- The attack is repeatable, atomic, and requires no mempool monitoring — a simple on-chain balance check suffices.

## Recommendation

Scope the ETH budget in `pay` to only the ETH contributed by the current transaction. Options:

1. Thread `msg.value` through the call stack and use it instead of `address(this).balance` in `pay`.
2. Record `address(this).balance - msg.value` at entry to each top-level payable function and compute the per-call ETH budget as `address(this).balance - snapshot`.
3. Auto-refund excess ETH at the end of every swap entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) rather than relying on the caller to include `refundETH()`.

## Proof of Concept

1. User A calls `exactOutputSingle(tokenIn=WETH, amountOut=Y)` with `msg.value = 1e18`; pool consumes only `0.5e18`. User A omits `refundETH()`. Router now holds `0.5e18` ETH.
2. Attacker calls `exactInputSingle(pool, tokenIn=WETH, amountIn=0.5e18, msg.value=0)`.
3. Pool callback fires `pay(WETH, attacker, pool, 0.5e18)`.
4. At line 74: `nativeBalance = address(this).balance = 0.5e18`. At line 75: `nativeBalance >= value` → router calls `WETH.deposit{value: 0.5e18}()` and transfers `0.5 WETH` to pool.
5. Attacker receives output tokens; their WETH balance is unchanged; User A's `0.5 ETH` is permanently gone.
