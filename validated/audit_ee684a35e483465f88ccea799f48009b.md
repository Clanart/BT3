### Title
`compute_meta_tx_v0_hash` Omits Nonce from Digest, Enabling Signature Replay Attacks — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` constructs the transaction hash that account contracts use to verify meta-transaction signatures. It calls `deprecated_get_transaction_hash` with `additional_data_size=0`, meaning **no nonce is committed to the hash**. Because the hash is fully determined by `(contract_address, selector, calldata, chain_id)`, any signature produced over it is permanently replayable by any party who has observed it on-chain.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` (lines 295–315) delegates to `deprecated_get_transaction_hash` with the nonce slot explicitly zeroed out:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // ← nonce absent
        additional_data=cast(0, felt*),  // ← nonce absent
    );
    return tx_hash;
}
``` [1](#0-0) 

Compare this with `compute_l1_handler_transaction_hash`, which correctly passes the nonce as `additional_data`:

```cairo
additional_data_size=1,
additional_data=&nonce,
``` [2](#0-1) 

In `execute_meta_tx_v0`, the OS constructs the `TxInfo` with `nonce=0` hardcoded and never calls `check_and_increment_nonce`:

```cairo
tempvar new_tx_info = new TxInfo(
    ...
    transaction_hash=meta_tx_hash,
    nonce=0,                   // ← always zero, never incremented
    ...
);
``` [3](#0-2) 

The resulting `meta_tx_hash` is then placed into `TxInfo.transaction_hash` and exposed to the target account contract's `__execute__` entry point via `get_execution_info()`. Account contracts that verify the caller's signature against `transaction_hash` (the intended use-case for meta-transactions) will accept any replay of the same `(contract_address, selector, calldata, chain_id)` tuple because the hash is identical across all invocations. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The meta_tx_v0 syscall is designed for gasless/relayer patterns where a user pre-signs a payload and a relayer submits it. Because the hash commits to no nonce, a single user signature authorises unlimited future executions of the same calldata. An attacker who extracts the signature from any historical transaction can replay it to:

- Drain ERC-20 balances by replaying a `transfer` calldata.
- Replay any state-mutating call the user originally authorized once.

There is no OS-level guard (no nonce increment, no uniqueness check) that would prevent this.

---

### Likelihood Explanation

**High.** All meta_tx_v0 signatures are broadcast on-chain as part of the outer transaction's calldata. Any observer can extract them. The attacker only needs to deploy a contract that issues the `meta_tx_v0` syscall with the captured signature and identical calldata — a fully unprivileged operation available to any contract deployer on StarkNet. [5](#0-4) 

---

### Recommendation

Include the nonce in the meta_tx_v0 hash, mirroring the pattern used by `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,               // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        chain_id=chain_id,
        additional_data_size=1,        // ← commit to nonce
        additional_data=&nonce,
    );
    return tx_hash;
}
```

Additionally, `execute_meta_tx_v0` must:
1. Accept and validate a caller-supplied nonce.
2. Call `check_and_increment_nonce` on the target contract's state entry to enforce single-use semantics.
3. Pass the nonce into `compute_meta_tx_v0_hash` and into `TxInfo.nonce`.

---

### Proof of Concept

1. **User action**: User signs a meta_tx_v0 payload for `(target_contract, __execute__, calldata=[transfer, victim_addr, 1000], chain_id)`. The resulting `meta_tx_hash = H(prefix, 0, target_contract, __execute__, calldata, 0, chain_id)` is fixed.

2. **First relay**: A legitimate relayer submits an outer transaction containing the `meta_tx_v0` syscall with the user's signature. The transfer executes. The signature is now visible on-chain.

3. **Replay**: An attacker deploys `MaliciousRelayer` which calls `meta_tx_v0(contract_address=target_contract, selector=__execute__, calldata=[transfer, victim_addr, 1000], signature=captured_sig)`. The OS computes the identical `meta_tx_hash` (no nonce), places the captured signature in `TxInfo.signature`, and calls `__execute__` on the target contract.

4. **Outcome**: The target account contract verifies `captured_sig` against `meta_tx_hash` — which matches — and executes the transfer again. Steps 3–4 repeat until the account is drained. [6](#0-5) [7](#0-6)
