Audit Report

## Title
Synthetic Ratio Precision Loss Truncates Mid to Zero, Permanently Halting Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
In `AnchoredPriceProvider._getBidAndAskPrice()`, the synthetic ratio mid price is computed as `Math.mulDiv(mid, ORACLE_DECIMALS, mid2)` where `ORACLE_DECIMALS = 1e8`. When the base/quote price ratio is below `1e-8`, integer division truncates the result to `0`. No zero-check follows; the zero mid propagates into `_computeBidAsk`, which returns the `(0, type(uint128).max)` failure sentinel, causing `getBidAndAskPrice()` to revert with `FeedStalled()` on every swap call while both oracle feeds remain entirely valid.

## Finding Description
In `_getBidAndAskPrice()` (line 267), when `quoteFeedId != bytes32(0)`:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
// ORACLE_DECIMALS = 1e8
// No zero-check here
spreadBps += spreadBps2;
return _computeBidAsk(mid, spreadBps);  // mid may be 0
```

`_computeBidAsk(0, spreadBps)` calls `_bandEdge(0, BPS_BASE_U - half, Floor)` at line 309, which evaluates `Math.mulDiv(0, Q64 * edgeFactor, STEP_DENOM, Floor) = 0`. The guard at line 311 fires:

```solidity
if (refBid == 0 || refAsk > type(uint128).max || refBid >= refAsk) {
    return (0, type(uint128).max);
}
```

The sentinel `(0, type(uint128).max)` is returned to `getBidAndAskPrice()`, which reverts at line 216:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
```

Every swap on the pool reverts. The existing guard at line 265 (`if (!ok2 || mid2 == 0)`) only protects against a zero denominator, not a zero result after division.

## Impact Explanation
Any pool whose `AnchoredPriceProvider` is deployed in synthetic two-feed mode with a base/quote price ratio below `1e-8` has all swap execution permanently blocked while that price condition holds. LPs cannot earn fees; traders cannot execute swaps. This satisfies the "broken core pool functionality causing unusable swap flows" impact gate. Withdrawals are unaffected (the sentinel is fail-closed for swaps only), but the core trading function of the pool is entirely disabled by valid oracle data.

## Likelihood Explanation
The `quoteFeedId` parameter is an explicit constructor input for synthetic ratio quoting. A pool can be legitimately deployed with a reasonable price ratio and later reach the truncation condition if the base token price crashes (e.g., a meme token depreciating against BTC or ETH). No privileged action or malicious setup is required — the condition is triggered purely by market price movements reflected in valid oracle data. The condition persists for as long as the price ratio remains below `1e-8`, which can be indefinite.

## Recommendation
Add an explicit zero-check immediately after the ratio computation:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max);
```

For a more robust fix, increase internal precision by scaling the numerator before dividing (e.g., multiply by `1e18` instead of `1e8`), then adjust `STEP_DENOM` and `_bandEdge` accordingly so the downstream Q64 conversion remains correct. This extends the representable ratio range from `≥ 1e-8` to `≥ 1e-18`.

## Proof of Concept
Deploy `AnchoredPriceProvider` with:
- `baseFeedId` → feed returning `mid = 100` (`$0.000001` at 8 decimals)
- `quoteFeedId` → feed returning `mid = 6_000_000_000_000` (`$60,000` at 8 decimals)

```
syntheticMid = Math.mulDiv(100, 1e8, 6_000_000_000_000)
             = 10_000_000_000 / 6_000_000_000_000
             = 0  (integer truncation)
```

`_computeBidAsk(0, spreadBps)` → `refBid = _bandEdge(0, …) = 0` → returns `(0, type(uint128).max)`.

`getBidAndAskPrice()` reverts with `FeedStalled()`. Every subsequent `swap()` call on the pool reverts despite both oracle feeds returning fresh, valid data.
