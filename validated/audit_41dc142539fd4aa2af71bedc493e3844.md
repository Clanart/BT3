### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows a malicious peer to install wrong-shard state under a target shard's key — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id` but never asserts that the embedded chunk's `shard_id()` matches the requested shard. A malicious peer can serve a header containing shard S2's chunk (with a valid Merkle proof against the block's `chunk_headers_root`) in response to a state sync request for shard S1. All existing validations pass, the header is stored under `StateHeaderKey(S1, sync_hash)`, and subsequent `set_state_part` calls extract S2's `prev_state_root` and apply it as S1's state root — installing S2's state under S1's shard key.

### Finding Description

`set_state_header` performs four validation steps before writing to `DBCol::StateHeaders`:

**Step 1 — `validate_chunk_proofs`** checks internal chunk consistency (header hash, tx root, receipts root). It takes no `shard_id` parameter and never compares `chunk.shard_id()` to anything external. [1](#0-0) 

**Step 2 — `verify_path` against `chunk_headers_root`** proves the chunk's `ChunkHashHeight` is a member of the block's Merkle tree. The tree is built by iterating all chunks in shard-index order and hashing `ChunkHashHeight(chunk_hash, height_included)`. The proof encodes left/right directions but `verify_path` only checks that the path reconstructs the root — it does not verify which leaf index (shard position) the chunk occupies. [2](#0-1) [3](#0-2) [4](#0-3) 

**Step 3 — receipt proof validation** hashes receipts as `ReceiptList(shard_id, receipts)` using the *requested* `shard_id` (S1), not the chunk's `shard_id`. An attacker can supply valid S1 incoming-receipt proofs (public chain data) alongside S2's chunk. [5](#0-4) 

**Step 4 — `validate_state_root_node`** checks the `state_root_node` against `chunk_inner.prev_state_root()` — S2's state root. An attacker who controls a state sync provider has S2's state and can supply a valid node. [6](#0-5) 

**Missing check**: nowhere in the function is `chunk.shard_id() == shard_id` asserted.

The header is then written under `StateHeaderKey(shard_id=S1, sync_hash)` with S2's chunk embedded: [7](#0-6) 

`set_state_part` later reads this header, extracts `prev_state_root` from S2's chunk, and uses it as the state root for shard S1: [8](#0-7) 

### Impact Explanation

The syncing node applies state parts validated against S2's `prev_state_root` and stores them under S1's `StatePartKey`. When `set_state_finalize` runs for shard S1, it reads S2's chunk from the stored header and applies S2's state to S1's shard slot. The node ends up with S2's committed state installed under S1's shard identity — a concrete, permanent corruption of the node's state database for shard S1. [9](#0-8) 

### Likelihood Explanation

Any peer that can respond to state sync header requests can exploit this. State sync providers are not required to be validators or hold any special privilege — the syncing node requests headers from available peers. The attacker needs only: (a) a valid chunk for S2 that is included in the target block (public chain data), (b) the Merkle proof for S2's chunk in that block (computable from the block), (c) valid incoming-receipt proofs for S1 (public chain data), and (d) a valid `state_root_node` for S2's state (available to any full node tracking S2). No cryptographic forgery is required.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// Guard: the chunk must belong to the shard being synced.
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be inserted at approximately line 377, before any further validation, so that a mismatched chunk is rejected before any work is done. [10](#0-9)

    // 1. Checking chunk.header.hash

        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid

        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
