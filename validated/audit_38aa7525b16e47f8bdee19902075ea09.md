### Title
Large Integer JSON Serialization Causes Cross-Language Coin Amount Disagreement - (File: `crates/chia-traits/src/to_json_dict.rs`)

### Summary

`Coin.amount` (`u64`) and other large integer fields (`u128` such as `weight`) are serialized as raw JSON numbers — not strings — in both the Python `to_json_dict()` path and the Rust `serde` path. Any value exceeding 2^53−1 (≈ 9 XCH in mojos) is silently corrupted by IEEE 754 double-precision rounding when parsed by JavaScript/WASM consumers, producing a different coin amount and therefore a different coin ID than the Rust/Python consensus layer computes.

### Finding Description

The `to_json_primitive!` macro in `crates/chia-traits/src/to_json_dict.rs` registers `u64` and `u128` as direct Python integer objects: [1](#0-0) 

`u64` and `u128` are both covered by this macro. When Python serializes the resulting `PyAny` to JSON (e.g., via `json.dumps()`), it emits a bare JSON number with no string quoting.

The `Coin` struct carries `amount: u64`: [2](#0-1) 

The Python test suite explicitly confirms that `uint64(0xFFFFFFFFFFFFFFFF)` round-trips through `to_json_dict()` as a raw JSON number: [3](#0-2) 

The Rust `serde` path (used by the WASM target and any Rust JSON consumer) also emits `amount` as a bare JSON number, as shown in the `SpendBundle` serde test: [4](#0-3) 

`serde_json` serializes `u64` as a JSON integer by default. Any value above 2^53−1 = 9,007,199,254,740,991 (≈ 9 XCH) is outside the IEEE 754 safe integer range. JavaScript's `JSON.parse()`, browsers, and Node.js will silently round such values to the nearest representable double. For example, `9007199254740993` becomes `9007199254740992`.

The coin ID is computed by hashing `parent_coin_info || puzzle_hash || amount_bytes`: [5](#0-4) 

A JavaScript/WASM client that parses a JSON-serialized coin with a large amount will compute a different coin ID than the Rust/Python consensus layer, because the amount bytes fed into the hash differ.

The WASM target is explicitly designed for web browser environments:



Protocol types such as `NewPeakWallet` carry `weight: uint128`, which in practice far exceeds 2^53 and is also serialized as a raw JSON number: [6](#0-5) 

### Impact Explanation

This is a **High** impact finding matching the allowed scope: *"Streamable/CLVM/serde/Python/wasm boundary parsing bug causes non-canonical bytes, integer confusion, hash mismatch, or cross-language disagreement in consensus-critical data."*

A JavaScript/WASM client that:
1. Receives a JSON-serialized `Coin` or `SpendBundle` from a Python full node
2. Parses `amount` as a JavaScript `number`
3. Computes the coin ID or validates a spend

will silently operate on a wrong amount for any coin worth more than ≈ 9 XCH, producing a wrong coin ID. This causes cross-language disagreement on which coins exist, which spends are valid, and which coin IDs to track — all consensus-critical operations.

### Likelihood Explanation

Coin amounts above 2^53 mojos (≈ 9 XCH) are routine in the Chia ecosystem. The JSON API is the primary interoperability surface between the Rust/Python full node and external clients (web wallets, explorers, WASM-based tools). Any JavaScript consumer of `to_json_dict()` output or `serde_json`-serialized protocol messages is affected without any special attacker action — a normal large-value coin triggers the bug.

### Recommendation

- **Short term:** Serialize `u64` and `u128` fields as JSON strings in `to_json_dict()` and in the `serde` implementations for all protocol types. Accept both strings and numbers in `from_json_dict()` / `Deserialize` for backward compatibility.
- **Long term:** Adopt a canonical JSON encoding convention (e.g., all integers > 32 bits as decimal strings) across the Python, Rust, and WASM surfaces, and document it explicitly. Validate this in the existing JSON round-trip tests by asserting the serialized type is a string for large integer fields.

### Proof of Concept

```python
import json
from chia_rs import Coin
from chia_rs.sized_ints import uint64

# Amount just above 2^53 — a normal ~9 XCH coin
amount = uint64(9_007_199_254_740_993)  # 2^53 + 1
coin = Coin(b'\x00' * 32, b'\x01' * 32, amount)

# Python serializes correctly (Python int is arbitrary precision)
d = coin.to_json_dict()
json_str = json.dumps(d)
# json_str contains: "amount": 9007199254740993

# JavaScript / Node.js parses this as:
# JSON.parse('{"amount": 9007199254740993}').amount
# => 9007199254740992   <-- silently wrong (2^53, not 2^53+1)

# The JavaScript client now computes a different coin ID:
# sha256(parent || puzzle_hash || encode(9007199254740992))
# != sha256(parent || puzzle_hash || encode(9007199254740993))
# => cross-language coin ID mismatch for a routine large-value coin
```

    pub fn coin_id(&self) -> Bytes32 {
        let mut hasher = Sha256::new();
        hasher.update(self.parent_coin_info);
        hasher.update(self.puzzle_hash);

        let amount_bytes = self.amount.to_be_bytes();
        if self.amount >= 0x8000_0000_0000_0000_u64 {
            hasher.update([0_u8]);
            hasher.update(amount_bytes);
        } else {
            let start = match self.amount {
                n if n >= 0x0080_0000_0000_0000_u64 => 0,
                n if n >= 0x8000_0000_0000_u64 => 1,
                n if n >= 0x0080_0000_0000_u64 => 2,
                n if n >= 0x8000_0000_u64 => 3,
                n if n >= 0x0080_0000_u64 => 4,
                n if n >= 0x8000_u64 => 5,
                n if n >= 0x80_u64 => 6,
                n if n > 0 => 7,
                _ => 8,
            };
            hasher.update(&amount_bytes[start..]);
        }

        let coin_id: [u8; 32] = hasher.finalize().as_slice().try_into().unwrap();
        Bytes32::new(coin_id)
    }
