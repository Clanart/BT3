### Title
Missing Class Hash Validation in `replace_class` Syscall Allows Permanent Freezing of Contract Funds - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` does not validate that the new class hash supplied by a contract corresponds to a declared class. An unprivileged contract can call `replace_class` with an arbitrary, undeclared class hash, permanently rendering itself uncallable and freezing any funds it holds. The missing check is explicitly acknowledged with a `TODO` comment in the code.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), the OS processes the `replace_class` syscall by writing the caller-supplied `class_hash` directly into `contract_state_changes` with no verification that the hash corresponds to a declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `class_hash` value originates from `request.class_hash`, which is fully attacker-controlled (set by the calling contract's own code). The TODO comment explicitly acknowledges the absent validation.

When a subsequent transaction calls this contract, `execute_entry_point` performs the following lookup chain:

**Step 1** — reads the compiled class hash for the (invalid) class hash from `contract_class_changes`. Because the invalid class hash was never declared, `dict_read` returns the default value `0`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [2](#0-1) 

**Step 2** — calls `find_element` to locate the compiled class for `compiled_class_hash = 0`. `find_element` asserts the element exists; if `0` is not a valid compiled class hash (which it is not in practice), the assertion fails and the block becomes unprovable:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

The state change (invalid class hash) is committed to the block before any subsequent call is attempted. Once committed, there is no OS-level mechanism to restore the contract's class hash without the contract's own code executing — which is now impossible. All funds held in the contract are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 tokens, NFTs, or other assets held in a contract that has replaced its class with an undeclared hash are irrecoverably locked. The contract becomes permanently uncallable at the OS level. No withdrawal, transfer, or recovery function can be invoked. There is no protocol-level escape hatch.

---

### Likelihood Explanation

**Medium.**

- The `replace_class` syscall is available to every deployed contract with no privilege requirement.
- A malicious contract can intentionally call `replace_class(arbitrary_felt)` to freeze its own funds (e.g., as part of a rug-pull or griefing attack against users who deposited into it).
- A buggy contract could accidentally supply an undeclared hash.
- The missing validation is explicitly flagged with a `TODO` comment dated `1/1/2026`, confirming the gap is known and unresolved.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that `request.class_hash` is a declared class by checking that `contract_class_changes` contains a non-zero compiled class hash entry for it. This mirrors the lookup already performed in `execute_entry_point` and would reject any `replace_class` call referencing an undeclared class, preventing permanent self-bricking of contracts.

---

### Proof of Concept

1. Deploy a contract `MaliciousVault` that accepts user deposits and holds funds.
2. The contract's logic includes a call to `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. The OS processes the `replace_class` syscall in `execute_replace_class` — no validation occurs; the state entry for `MaliciousVault` is updated with `class_hash = 0xdeadbeef`.
4. The block is proven successfully (the replace itself is valid from the OS's perspective at this point).
5. Any subsequent transaction invoking `MaliciousVault` causes `execute_entry_point` to call `dict_read` on `contract_class_changes` for key `0xdeadbeef`, receiving `0` (undeclared). `find_element` then fails to locate compiled class hash `0`, making the block unprovable.
6. The sequencer must permanently exclude all transactions targeting `MaliciousVault`. All deposited funds are frozen with no recovery path.
