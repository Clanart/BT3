### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Check for All Items When Tree Has ≥ 2 Leaves, Silently Corrupting Tree State and Root Hash - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` only calls the guarded `insert()` path (which checks for duplicate keys) for the last two items in the input when the tree has ≤ 1 existing leaves. For all remaining items — and for **all** items when the tree already has ≥ 2 leaves — it calls `insert_entry_to_blob()` directly, which silently overwrites the `block_status_cache` entry for any key that already exists. This leaves the old leaf node orphaned in the blob (still referenced by its parent in the tree structure) while the cache now points to the new duplicate leaf. The result is a tree with two leaves sharing the same key, a corrupted `block_status_cache`, and a root hash that diverges from what the cache-based API reports.

### Finding Description

In `batch_insert`, the fast path for items beyond the bootstrap phase writes leaf nodes directly to the blob without any uniqueness check: [1](#0-0) 

When `leaf_count() <= 1`, only the last two items are popped and routed through `self.insert()`, which enforces uniqueness: [2](#0-1) 

All remaining items — and **all** items when `leaf_count() > 1` — bypass `insert()` entirely and go directly to `insert_entry_to_blob()`: [3](#0-2) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf()`, which performs an unconditional `HashMap::insert` on `key_to_index` — silently overwriting any existing mapping for the same key: [4](#0-3) [5](#0-4) 

By contrast, `insert()` explicitly guards against this: [6](#0-5) 

After a duplicate-key `batch_insert` completes:

1. The blob contains **two** leaf nodes with the same `KeyId`: the original at index `I_old` (still wired into the tree via its parent pointer) and the new one at index `I_new` (in the freshly-attached batch subtree).
2. `key_to_index` maps the key to `I_new` only; `I_old` is not in `free_indexes` either, so it is invisible to both the cache and the free-list.
3. `calculate_lazy_hashes()` traverses the actual tree structure and computes a root hash that includes **both** leaves.
4. `get_proof_of_inclusion(K)` uses the cache to locate `I_new` and generates a proof anchored to that leaf — but the root hash it must match against was computed from a tree that also contains `I_old`.
5. `get_keys_values()` returns only one entry for key `K`, hiding the duplicate from the caller.

The Python binding `py_batch_insert` exposes this path directly: [7](#0-6) 

### Impact Explanation

This is a **High** impact DataLayer finding. The corrupted tree produces a root hash that does not correspond to any valid, consistent key-value set. A DataLayer store that commits this root hash to the blockchain has committed a state that:

- Cannot be faithfully reconstructed from the key-value API (the hidden duplicate leaf is unreachable via `get_keys_values` or `get_proof_of_inclusion`).
- Produces proofs of inclusion that may fail verification against the committed root, or that prove membership of a key at a position that does not match the committed root.
- Causes cross-node disagreement: nodes that rebuild the tree from the key-value set will compute a different root than the node that ran `batch_insert` with a duplicate.

This satisfies the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is a public API exposed through Python bindings and used by the DataLayer subsystem for bulk operations. Any caller — including one constructing a DataLayer update — can supply a key list that contains a key already present in the tree. No privilege is required. The bug is triggered whenever the tree has ≥ 2 existing leaves and the batch contains at least one duplicate key, or when the tree has ≤ 1 leaves and the batch contains ≥ 3 items with a duplicate among the first `n-2`.

### Recommendation

Add a duplicate-key (and duplicate-hash) check inside the fast-path loop of `batch_insert`, mirroring the guard in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, validate the entire input batch for uniqueness (both against each other and against the existing tree) before any mutation begins, so the operation is atomic with respect to the invariant.

### Proof of Concept

// Tree already has 2 leaves (key=0, key=1), so leaf_count() > 1

blob.insert(KeyId(0), ValueId(0), &sha256_num(&0i64), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &sha256_num(&1i64), InsertLocation::Auto {}).unwrap();

// key=0 already exists; batch_insert does NOT reject it
blob.batch_insert(vec![
    ((KeyId(0), ValueId(99)), sha256_num(&99i64)),  // duplicate key!
    ((KeyId(2), ValueId(2)), sha256_num(&2i64)),

// blob.check_integrity() will return an error (IntegrityKeyToIndexCacheLength mismatch)

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
