### Title
Gateway Stateful Validator Enforces Minimum Gas Price Threshold Only on L2 Gas, Skipping L1 Gas and L1 Data Gas — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `StatefulTransactionValidator` enforces a minimum gas-price threshold (derived from the previous block's L2 gas price) only against `l2_gas.max_price_per_unit`. The two secondary resource prices — `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` — are never checked at the gateway admission stage. A transaction carrying a zero or dust L1/L1-data gas price passes every gateway check and is admitted to the mempool, but is then rejected by the blockifier's pre-validation at execution time. The code itself acknowledges the gap with an explicit TODO.

---

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` fetches the previous block's L2 gas price and delegates to `validate_tx_l2_gas_price_within_threshold`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:196-216
async fn validate_resource_bounds(
    &self,
    executable_tx: &ExecutableTransaction,
) -> StatefulTransactionValidatorResult<()> {
    if self.config.validate_resource_bounds {
        let previous_block_l2_gas_price = self
            .gateway_fixed_block_state_reader
            .get_block_info()
            .await?
            .gas_prices
            .strk_gas_prices
            .l2_gas_price;          // ← only L2 price is fetched
        self.validate_tx_l2_gas_price_within_threshold(
            executable_tx.resource_bounds(),
            previous_block_l2_gas_price,
        )?;
    }
    Ok(())
}

The stateless validator has the same asymmetry — it enforces an absolute floor only on `l2_gas.max_price_per_unit`:

```rust
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(...MaxGasPriceTooLow...);
}
// l1_gas and l1_data_gas prices: unchecked
``` [3](#0-2) 

By contrast, the blockifier's `check_fee_bounds` (called at execution time, not at admission) validates **all three** resource prices against the actual block prices:

```rust
// L1Gas, L1DataGas, L2Gas all checked:
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { resource, ... });
}
``` [4](#0-3) 

The mempool's own gas-price threshold mechanism is also keyed exclusively on L2 gas price (`max_l2_gas_price`), so a transaction with a zero L1 or L1-data gas price is never demoted or evicted by the mempool either.

The structural parallel to M-20 is exact:

| M-20 (Notional) | Sequencer analog |
|---|---|
| `accountDebtUnderlying` checked against `minAccountBorrowSize` | `l2_gas.max_price_per_unit` checked against threshold |
| `accountDebtOne`, `accountDebtTwo` **not** checked | `l1_gas.max_price_per_unit`, `l1_data_gas.max_price_per_unit` **not** checked |
| Dust secondary debts → insolvency | Dust secondary gas prices → invalid txs admitted to mempool |

---

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Any unprivileged user can craft a V3 `AllResources` transaction with a valid `l2_gas.max_price_per_unit` (to pass the gateway) but with `l1_gas.max_price_per_unit = 0` and/or `l1_data_gas.max_price_per_unit = 0`. The transaction:

1. Passes stateless validation (only L2 gas price floor is enforced).
2. Passes stateful validation (only L2 gas price threshold is enforced).
3. Is admitted to the mempool.
4. Fails at blockifier pre-validation with `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }` — wasting sequencer execution resources and polluting the mempool with transactions that can never succeed.

At scale this enables cheap, targeted mempool flooding: the attacker pays only the L2 gas price floor while the sequencer bears the cost of executing and reverting every such transaction.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no special contract, and no knowledge of internal state. Any user who can submit a V3 transaction can set `l1_gas.max_price_per_unit` to zero. The gap is acknowledged in the codebase itself (`TODO(Arni): Consider running this validation for all gas prices`), confirming it is a known-but-unaddressed omission rather than an intentional design choice.

---

### Recommendation

Extend `validate_tx_l2_gas_price_within_threshold` (or introduce a symmetric helper) to also validate `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against their respective previous-block prices and the configured `min_gas_price_percentage` threshold. The previous block's L1 and L1-data gas prices are already available on the `BlockInfo` struct returned by `get_block_info()`.

```rust
// Fetch all three prices:
let prices = block_info.gas_prices.strk_gas_prices;
// Then apply the same threshold check to l1_gas and l1_data_gas
// as is currently applied to l2_gas.
```

The stateless validator's absolute floor check should be extended in the same way.

---

### Proof of Concept

1. Construct a V3 `InvokeTransaction` with `AllResources` bounds:
   - `l2_gas.max_price_per_unit` = current block L2 gas price (passes gateway threshold)
   - `l1_gas.max_price_per_unit` = `0`
   - `l1_data_gas.max_price_per_unit` = `0`
2. Submit via the gateway HTTP endpoint.
3. **Expected (current behaviour):** Gateway accepts the transaction; it enters the mempool.
4. **At sequencing time:** Blockifier `check_fee_bounds` fires `InsufficientResourceBounds { errors: [MaxGasPriceTooLow { resource: L1Gas, max_gas_price: 0, actual_gas_price: <block_price> }] }` and the transaction is reverted/rejected.
5. **Expected (fixed behaviour):** Gateway rejects the transaction at step 2 with `GAS_PRICE_TOO_LOW` before it ever reaches the mempool.
