### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key and Duplicate-Hash Validation, Corrupting DataLayer Merkle Tree State — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that the single-item `insert` function enforces. When the tree already contains two or more leaves, every item in a batch is written directly to the blob without any uniqueness check. An unprivileged caller supplying a batch with a repeated `KeyId` or repeated `Hash` silently corrupts the Merkle tree, producing an incorrect root hash and enabling forged or invalid proofs of inclusion.

### Finding Description

`MerkleBlob::insert` (single-item path) enforces two invariants before writing anything:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a completely different code path for the bulk portion of the batch:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 570-603
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    let mut indexes = vec![];

    if self.block_status_cache.leaf_count() <= 1 {
        for _ in 0..2 {
            let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                return Ok(());
            };
            self.insert(key, value, &hash, InsertLocation::Auto {})?;  // validated
        }
    }
    // ↓ ALL remaining items — or ALL items when tree already has ≥2 leaves — bypass validation
    for ((key, value), hash) in keys_values_hashes {
        let new_leaf_index = self.get_new_index();
        let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value }) };
        self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // no key/hash check
        indexes.push(new_leaf_index);
    }
``` [2](#0-1) 

Two distinct bypass scenarios exist:

1. **Tree has ≥ 2 leaves (most common production state):** The `if self.block_status_cache.leaf_count() <= 1` guard is false, so the entire batch is written via `insert_entry_to_blob` with zero uniqueness checks.
2. **Tree has ≤ 1 leaf and batch has ≥ 3 items:** The last two items are validated via `insert`; the first N-2 items (processed in the `for` loop) are not.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1519

pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2) 

When a duplicate `KeyId` is inserted, the `block_status_cache` key-to-index map is overwritten, leaving an orphaned leaf node in the blob that is unreachable through the cache but still participates in hash computation. When a duplicate `Hash` is inserted, two leaf nodes share the same hash, making the tree's root hash non-unique and breaking the one-to-one correspondence between leaf hashes and tree positions that `ProofOfInclusion::valid()` relies on. [4](#0-3) 

### Impact Explanation

A corrupted `MerkleBlob` produces an incorrect root hash. Any `ProofOfInclusion` generated from the corrupted tree will either:
- Claim a key is present when it is not (forged inclusion), or
- Produce a root hash that does not match the canonical tree, causing valid proofs to be rejected (denial of valid state).

Both outcomes directly satisfy the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion API and is called by the DataLayer sync path. Any caller that controls the key/value/hash inputs — including a malicious DataLayer peer or a crafted delta sync payload — can trigger this with a batch of ≥ 3 items containing a repeated key or hash. No privileged access is required; the Python binding is public.

### Recommendation

Add the same uniqueness guards to the bulk path in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, deduplicate the input vector before processing and return an error on any collision, consistent with the single-item `insert` contract.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
from chia.types.blockchain_format.sized_bytes import bytes32

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 (bypasses the guarded path)
blob.insert(KeyId(1), ValueId(1), bytes32(b'\x01' * 32))
blob.insert(KeyId(2), ValueId(2), bytes32(b'\x02' * 32))

# batch_insert with a duplicate KeyId (key=3 appears twice)
# Single insert would raise KeyAlreadyPresentError on the second key=3
blob.batch_insert(
    [(KeyId(3), ValueId(3)), (KeyId(3), ValueId(99))],  # duplicate key
    [bytes32(b'\x03' * 32), bytes32(b'\x04' * 32)],
)

# Tree is now corrupt: two leaf nodes share KeyId(3),
# block_status_cache maps key=3 to only one of them,
# the other is an orphan that still affects the root hash.
blob.calculate_lazy_hashes()
root = blob.get_root_hash()
# root is now non-canonical; proofs generated from it are unreliable
proof = blob.get_proof_of_inclusion(KeyId(3))
# proof.valid() may return True for the wrong leaf hash
