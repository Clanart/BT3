### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Enables Bridge Undercollateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` uses the caller-supplied `amount` parameter directly in the emitted `InitTransfer` event and in all downstream accounting, without measuring the actual token balance received after `safeTransferFrom`. For fee-on-transfer ERC20 tokens, the bridge vault receives `amount − fee` but the cross-chain message credits the user with the full `amount`, permanently breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer()`, when the token is a plain ERC20 (not a bridge token and not a custom minter), the following sequence occurs:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 406-436
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← user-supplied value
    );
}
// ...
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,           // ← same user-supplied value, not actual received
    fee, nativeFee, recipient, message, extensionValue
);

emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,           // ← emitted as the canonical bridged amount
    fee, nativeFee, recipient, message
);
```

The `InitTransfer` event is the authoritative cross-chain message that the NEAR bridge reads to credit the user on the destination chain. Because `amount` is used verbatim — never replaced by a before/after balance delta — any token that deducts a fee during `transferFrom` causes the EVM vault to hold fewer tokens than the NEAR side believes it does. [1](#0-0) [2](#0-1) 

The same pattern is present in the Solana bridge: `init_transfer.rs` calls `transfer_checked` with `payload.amount` and then serializes that same `payload.amount` into the Wormhole message sent to NEAR, without accounting for any Token-2022 transfer-fee extension deduction. [3](#0-2) 

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token creates a deficit:

- EVM vault receives: `amount − transfer_fee`
- NEAR credits user: `amount`

The deficit accumulates with each transfer. Eventually, when users bridge tokens back from NEAR to EVM, the vault cannot cover the full redemption. This is a **direct break of bridge collateralization** — the EVM-side reserve is systematically smaller than the NEAR-side supply, enabling the last redeemers to be unable to withdraw their funds (permanent freezing of a portion of user funds).

This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

- Fee-on-transfer ERC20 tokens exist today (e.g., STA, PAXG, various reflection tokens). USDT has a fee mechanism that is currently set to zero but can be enabled by its owner.
- Any such token that passes through `logMetadata` and gets registered on the NEAR bridge becomes exploitable immediately.
- The entry path requires no privileged access: any user can call `initTransfer` with a publicly available fee-on-transfer token.
- The attacker does not need to do anything special — simply bridging the token repeatedly is sufficient to widen the collateral gap.

---

### Recommendation

Replace the user-supplied `amount` with the actual received amount by measuring the balance delta around the `safeTransferFrom` call:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    amount = uint128(balanceAfter - balanceBefore); // use actual received
}
```

Apply the same fix to `initTransfer1155` if ERC-1155 tokens with fee hooks are in scope, and to the Solana `init_transfer.rs` for Token-2022 tokens with the transfer-fee extension.

Additionally, consider adding a token allowlist that explicitly rejects fee-on-transfer tokens at registration time.

---

### Proof of Concept

1. Deploy or use an existing fee-on-transfer ERC20 token `FeeToken` with a 1% transfer fee.
2. Register `FeeToken` on the Omni Bridge via `logMetadata` and complete the NEAR-side token binding.
3. Call `OmniBridge.initTransfer(FeeToken, 1_000_000, 0, 0, "near_recipient.near", "")`.
4. The bridge vault receives `990_000` tokens (1% fee deducted).
5. The `InitTransfer` event emits `amount = 1_000_000`.
6. NEAR bridge processes the event and mints/unlocks `1_000_000` `FeeToken` units for `near_recipient.near`.
7. Repeat steps 3–6 many times. Each iteration widens the collateral gap by `10_000` tokens.
8. When any user attempts to bridge `1_000_000` tokens back from NEAR to EVM, the vault is short. After enough iterations, the vault is fully drained while NEAR-side supply remains unbacked, permanently freezing the remaining users' funds. [4](#0-3)
