### Title
Unbounded `waiting_for_block` Vec Growth in `SpiceChunkValidatorActor` Causes Validator OOM — (`File: chain/client/src/spice/chunk_validator_actor.rs`)

---

### Summary

`SpiceChunkValidatorActor` stores incoming `SpiceChunkStateWitness` and `SpiceChunkContractAccesses` messages in an unbounded `Vec` inside `waiting_for_block` when the referenced block is not yet in the store. No size limit, distance-from-head check, or eviction policy is applied before insertion. Any network peer can flood a validator with witnesses referencing fabricated block hashes that will never arrive, causing the `waiting_for_block` `HashMap` to grow without bound until the validator process is killed by OOM. The non-Spice path has explicit protections for the identical scenario; the Spice path has none, and the code itself carries a `TODO` acknowledging the gap.

---

### Finding Description

`SpiceChunkValidatorActor` owns the field:

```rust
waiting_for_block:
    HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>,
``` [1](#0-0) 

When `process_chunk_state_witness` is called and `witness_processing_readiness` returns `NotReady` (the block identified by `chunk_id.block_hash` is absent from the store), the witness is unconditionally appended:

```rust
self.waiting_for_block.entry(chunk_id.block_hash).or_default().0.push(witness);
``` [2](#0-1) 

The same pattern applies to `SpiceChunkContractAccesses` when `get_epoch_id` returns `MissingBlock`:

```rust
self.waiting_for_block.entry(chunk_id.block_hash).or_default().1.push(accesses);
``` [3](#0-2) 

Both insertion sites carry an explicit `TODO` acknowledging the missing guards:

> `// TODO(spice): Implement additional checks (size limit, distance to head) before adding witness to `waiting_for_block`. See non-spice handle_orphan_witness().` [4](#0-3) 

The only pre-insertion check is that the local node holds a validator signer — it says nothing about the sender or the content of the witness:

```rust
let Some(signer) = self.validator_signer.get() else { ... return; };
if let Err(err) = self.process_chunk_state_witness(witness, signer) { ... }
``` [5](#0-4) 

Witness signature validation (`spice_pre_validate_chunk_state_witness`) is only reached after the block is available — it is never reached for the deferred path. [6](#0-5) 

Entries in `waiting_for_block` are only removed when the matching block arrives via `ProcessedBlock` or `ExecutionResultEndorsed`. Entries keyed on fabricated hashes that never correspond to a real block are never evicted. [7](#0-6) 

**Contrast with the non-Spice path.** `handle_orphan_witness` in `ChunkValidationActor` enforces:
1. A height-distance window (`ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD`).
2. A per-witness byte-size cap (`max_orphan_witness_size`).
3. An LRU cache with a hard capacity (`OrphanStateWitnessPool`). [8](#0-7) 

`OrphanStateWitnessPool` is keyed on `ChunkProductionKey` (shard + height), so a second witness for the same slot evicts the first rather than appending. [9](#0-8) 

None of these protections exist in the Spice path.

---

### Impact Explanation

An attacker sends a continuous stream of `SpiceChunkStateWitnessMessage` packets, each carrying a distinct random `block_hash` in `chunk_id`. Every message passes the only guard (local validator signer present), fails the block-lookup, and appends a full `SpiceChunkStateWitness` object — which can be up to the protocol's uncompressed witness size limit — to a new `Vec` in `waiting_for_block`. Because the keys are random hashes that will never match an arriving block, no eviction ever fires. The `HashMap` grows monotonically until the validator process is killed by the OS OOM killer. This crashes the validator, removing it from consensus participation and, if enough validators are targeted simultaneously, halting block production for the affected shards.

---

### Likelihood Explanation

The Spice feature is gated behind `protocol_feature_spice`, so the attack surface is active only when that feature flag is enabled. Once enabled, any peer reachable over the NEAR P2P network can send `SpiceChunkStateWitnessMessage` to a validator node; no stake, no validator key, and no valid witness content is required. The attacker only needs to know the target's peer address (publicly discoverable via the network's routing table) and to send messages faster than the validator can process them. The attack requires no on-chain action and leaves no on-chain trace.

---

### Recommendation

Apply the same three-layer defence that the non-Spice orphan-witness path uses before inserting into `waiting_for_block`:

1. **Distance-from-head check** — reject witnesses whose `block_hash` height is outside a small window around the current chain head (mirrors `ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD`).
2. **Per-witness size cap** — reject witnesses whose serialised size exceeds a configured maximum (mirrors `max_orphan_witness_size`).
3. **Bounded, keyed storage** — replace `Vec<SpiceChunkStateWitness>` with an LRU structure keyed on `(block_hash, shard_id)` so that a second witness for the same slot replaces the first rather than appending, and total capacity is bounded (mirrors `OrphanStateWitnessPool`).

Additionally, add a total-byte budget across all entries in `waiting_for_block` so that even within the allowed height window an attacker cannot exhaust memory by sending many distinct shard witnesses for the same absent block.

### Proof of Concept

```
// Pseudocode — attacker node on the NEAR P2P network
for i in 0..1_000_000 {
    let fake_block_hash = CryptoHash::hash_bytes(&i.to_le_bytes());
    let witness = SpiceChunkStateWitness::new(
        SpiceChunkId { block_hash: fake_block_hash, shard_id: ShardId::new(0) },
        /* large base_state to maximise per-entry cost */
        SpiceChunkStateTransition { base_state: PartialState::TrieValues(vec![0u8; 1_000_000]), .. },
        HashMap::new(),
        CryptoHash::default(),
        vec![],
        ChunkExecutionResultHash(CryptoHash::default()),
        CryptoHash::default(),
        None,
    );
    network.send_to_validator(target_peer, SpiceChunkStateWitnessMessage { witness, raw_witness_size: 0 });
}
// Each message passes the only guard (local validator_signer.get().is_some()),
// fails chain_store.get_block(fake_block_hash) → DBNotFoundErr,
// and appends the witness to waiting_for_block[fake_block_hash].0
// without any size or count check.
// The HashMap grows by ~1 MiB per iteration; at 1 M iterations → ~1 TiB
// (in practice OOM occurs within seconds at realistic witness sizes).
```

The exact corrupted state is `SpiceChunkValidatorActor::waiting_for_block`: a `HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>` whose entries for fabricated block hashes are never evicted, growing until process termination. [10](#0-9)

    witness_cache: LruCache<ChunkProductionKey, CacheEntry>,

struct CacheEntry {
    witness: ChunkStateWitness,
    _metrics_tracker: OrphanWitnessMetricsTracker,
}

impl OrphanStateWitnessPool {
    /// Create a new `OrphanStateWitnessPool` with a capacity of `cache_capacity` witnesses.
    /// The `Default` trait implementation provides reasonable defaults.
    pub fn new(cache_capacity: usize) -> Self {
        if cache_capacity > 128 {
            tracing::warn!(
                target: "client",
                cache_capacity,
                "orphan state witness cache capacity is larger than expected, this might lead to performance problems"
            );
        }

        OrphanStateWitnessPool {
            witness_cache: LruCache::new(NonZeroUsize::new(cache_capacity).unwrap()),
        }
    }

    /// Add an orphaned chunk state witness to the pool. The witness will be put in a cache and it'll
    /// wait there for the block that's required to process it.
    /// It's expected that this `ChunkStateWitness` has gone through basic validation - including signature,
    /// shard_id, size, epoch_id and distance from the tip. The pool would still work without it, but without
    /// validation it'd be possible to fill the whole cache with spam.
    /// `witness_size` is only used for metrics, it's okay to pass 0 if you don't care about the metrics.
    pub fn add_orphan_state_witness(&mut self, witness: ChunkStateWitness, witness_size: usize) {
        // Insert the new ChunkStateWitness into the cache
        let cache_key = witness.chunk_production_key();
        let metrics_tracker = OrphanWitnessMetricsTracker::new(&witness, witness_size);

        if let Some((_, ejected_entry)) = self.witness_cache.push(cache_key, cache_entry) {
