Audit Report

## Title
Router `pay()` consumes stranded native ETH to settle WETH-input swaps, enabling theft of prior users' ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function reads `address(this).balance` — the router's total ETH — when settling WETH-leg payments, rather than the current transaction's `msg.value`. ETH overpaid by any user in a payable entry point accumulates on the router and is silently consumed to fund a subsequent caller's WETH swap. Additionally, `refundETH()` transfers the entire ETH balance to any caller with no ownership binding, enabling direct theft.

## Finding Description
In `PeripheryPayments.sol` L74, `pay` reads `uint256 nativeBalance = address(this).balance`. When `nativeBalance >= value` (L75-78), it deposits exactly `value` of the router's ETH as WETH and forwards it to the pool — without pulling any token from `payer`. The `payer` stored in transient context is irrelevant when the router's ETH balance is sufficient.

ETH accumulates on the router whenever a user calls a payable entry point (e.g., `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) with `msg.value > amountIn` and does not include a `refundETH()` call in the same multicall. The `receive()` guard at L32-34 only blocks direct ETH pushes from non-WETH addresses; it does not prevent excess `msg.value` from sitting on the router after a swap completes, because payable function calls bypass `receive()`.

A subsequent attacker can exploit this in two ways:

**Path A — free WETH swap:** Attacker calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=S)` where `S` equals the stranded ETH. `pay` sees `address(this).balance == S >= S`, deposits the stranded ETH as WETH, and forwards it to the pool. Attacker receives token-out without spending any ETH or WETH.

**Path B — direct ETH theft:** Attacker calls `refundETH()` standalone. `refundETH` at L58-63 sends `address(this).balance` to `msg.sender` with no check that the caller is the original depositor.

Existing guards are insufficient: the `receive()` guard does not prevent overpayment via `msg.value`; the transient-storage payer context is bypassed entirely when the native balance branch is taken; and `multicall` performs no automatic refund.

## Impact Explanation
Direct, unrecoverable loss of user principal ETH. Any ETH overpaid by a victim is permanently consumed either by a free WETH swap (attacker receives equivalent token-out) or by direct `refundETH()` drain. Loss is bounded only by the victim's overpayment. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
ETH is stranded whenever a user calls a payable router function with `msg.value > amountIn` and omits `refundETH()` — a realistic user error for direct (non-multicall) calls with a safety ETH buffer. The attack requires no special privilege, no oracle manipulation, and no pool state precondition. A monitoring bot watching the router's ETH balance can execute either path atomically.

## Recommendation
1. **Track `msg.value` per entry point in transient storage.** At the start of each payable public function, record `msg.value` in a transient slot. In `pay`, use `min(transientMsgValue, value)` as the native contribution rather than `address(this).balance`.
2. **Alternatively, auto-refund excess ETH** at the end of every non-multicall payable entry point so no ETH survives past the transaction boundary.
3. **Bind `refundETH()` to the original caller.** Record `msg.sender` in transient storage at multicall entry and restrict `refundETH()` to that address, or auto-refund within the same multicall frame.
