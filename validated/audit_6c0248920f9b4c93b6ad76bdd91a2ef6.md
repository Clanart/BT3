### Title
Fee-on-Transfer Token Accounting Discrepancy Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records and broadcasts the caller-supplied `amount` as the canonical transfer value, but for fee-on-transfer ERC20 tokens the bridge actually receives `amount − transfer_fee`. The NEAR side credits the full `amount`, creating an unbacked surplus on NEAR that permanently undercollateralizes the EVM vault. Later withdrawers cannot be paid in full, resulting in irrecoverable fund loss.

---

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge-deployed token nor a custom-minter token, the bridge locks it via a plain `safeTransferFrom`:

```solidity
// OmniBridge.sol L406-L411
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← requested amount, not actual received amount
    );
}
```

For fee-on-transfer tokens (e.g., Tether with a non-zero fee, PAXG, or any token that deducts a percentage on every transfer), the bridge receives `amount − fee_deducted` tokens, but the function continues to use the original `amount` in all downstream accounting:

1. **Event emission** — `InitTransfer` is emitted with `amount` (L427–L436), not the actual received balance.
2. **Wormhole message** — `initTransferExtension` in `OmniBridgeWormhole.sol` encodes `amount` verbatim into the cross-chain payload (L136), which NEAR reads as the authoritative deposit size.

On the withdrawal side, `finTransfer` releases exactly `payload.amount` from the vault:

```solidity
// OmniBridge.sol L351-L354
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount   // ← full amount, but vault is short
    );
}
```

Because the vault holds less than the sum of all credited `amount` values, the vault becomes progressively undercollateralized. The first withdrawers are paid in full; later withdrawers receive a revert, permanently freezing their funds.

---

### Impact Explanation

Each fee-on-transfer deposit creates a deficit of `transfer_fee` tokens in the EVM vault. After N deposits the vault is short by `N × transfer_fee`. When the cumulative deficit exceeds the vault balance, `safeTransfer` in `finTransfer` reverts, making those withdrawals permanently unclaimable. This matches the allowed impact class: **balance/accounting corruption that breaks bridge collateralization** and **permanent freezing / irrecoverable lock of user funds**.

---

### Likelihood Explanation

The bridge explicitly supports arbitrary ERC20 tokens via `logMetadata` / `deployToken` and the `acceptedEthTokens`-style open path. Tether (USDT) already has a fee mechanism that is currently set to zero but is upgradeable by its owner; PAXG charges a 0.02% fee on every transfer today. Any user who bridges one of these tokens triggers the bug without any privileged access. Likelihood is **Medium-High** given the open token set and the existence of live fee-charging tokens.

---

### Recommendation

Measure the actual received amount using a before/after balance check and use that value for all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
// or: use actualReceived as the canonical amount going forward
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not accepted (e.g., via an allowlist with a known-safe token registry).

---

### Proof of Concept

1. USDT owner enables a 1% transfer fee (a live, non-hypothetical capability).
2. Alice calls `initTransfer(USDT, 1000e6, 0, 0, "near:alice", "")`.
3. Bridge receives `990e6` USDT; vault balance = `990e6`.
4. `InitTransfer` event and Wormhole message both record `amount = 1000e6`.
5. NEAR credits Alice with `1000e6` USDT equivalent.
6. Alice calls `finTransfer` on NEAR to withdraw back to EVM; NEAR signs a payload for `1000e6`.
7. `finTransfer` on EVM calls `safeTransfer(alice, 1000e6)` — succeeds (vault had `990e6` but Alice is first).
8. Vault is now at `−10e6` effective deficit; the next depositor's withdrawal will revert.
9. Repeat: after 99 such round-trips the vault is fully drained; the 100th user's `finTransfer` reverts permanently.
