### Title
Stale `BLOCK_HASH_TABLE_ADDRESS` Lower Bound in `ContractAddress::validate()` Admits OS-Reserved Addresses 0x2 and 0x3 as Valid Sender Addresses — (`crates/starknet_api/src/core.rs`)

### Summary

`ContractAddress::validate()` uses `BLOCK_HASH_TABLE_ADDRESS` (0x1) as the sole lower-bound sentinel for OS-reserved addresses. The OS has since added two more reserved addresses — `ALIAS_CONTRACT_ADDRESS` (0x2) and `RESERVED_CONTRACT_ADDRESS` (0x3) — but the lower bound was never updated. As a result, the gateway's stateless validator admits Invoke and Declare transactions whose `sender_address` is 0x2 or 0x3, which are OS-reserved and must never be used as account addresses.

### Finding Description

In `crates/starknet_api/src/core.rs`, `ContractAddress::validate()` enforces:

```rust
pub fn validate(&self) -> Result<(), StarknetApiError> {
    let value = self.0.0;
    let l2_address_upper_bound = Felt::from(*L2_ADDRESS_UPPER_BOUND);
    if (value > BLOCK_HASH_TABLE_ADDRESS.0.0) && (value < l2_address_upper_bound) {
        return Ok(());
    }
    Err(StarknetApiError::OutOfRange { string: format!("[0x2, {l2_address_upper_bound})") })
}
```

The condition `value > BLOCK_HASH_TABLE_ADDRESS.0.0` (i.e., `value > 0x1`) passes for any address ≥ 0x2. [1](#0-0) 

The OS, however, defines three reserved addresses that must never be used as user-facing contract addresses:

```
BLOCK_HASH_CONTRACT_ADDRESS = 0x1   // block number → block hash mapping
ALIAS_CONTRACT_ADDRESS      = 0x2   // stateful compression alias mapping
RESERVED_CONTRACT_ADDRESS   = 0x3   // future reserved
``` [2](#0-1) 

The OS's own deploy guard explicitly enumerates all three:

```cairo
assert_not_zero(
    (contract_address - ORIGIN_ADDRESS) *
    (contract_address - BLOCK_HASH_CONTRACT_ADDRESS) *
    (contract_address - ALIAS_CONTRACT_ADDRESS) *
    (contract_address - RESERVED_CONTRACT_ADDRESS),
);
``` [3](#0-2) 

The Rust-side `validate()` was written when only 0x1 was reserved. When 0x2 and 0x3 were added to the OS, the lower bound was not updated — the exact same pattern as the ZkSync `CURRENT_MAX_PRECOMPILE_ADDRESS` bug.

The stateless validator calls `validate()` directly for every Invoke and Declare transaction:

```rust
fn validate_contract_address(tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    let sender_address = match tx {
        RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => tx.sender_address,
        RpcTransaction::DeployAccount(_) => return Ok(()),
        RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => tx.sender_address,
    };
    Ok(sender_address.validate()?)
}
``` [4](#0-3) 

A transaction with `sender_address = 0x2` passes this check and is forwarded to the mempool.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Any unprivileged actor can craft an Invoke or Declare transaction with `sender_address = 0x2` (ALIAS_CONTRACT_ADDRESS) or `0x3` (RESERVED_CONTRACT_ADDRESS). The stateless validator passes it, the mempool admits it, and the batcher picks it up. At execution time the blockifier finds no user-deployed class at those addresses and the transaction fails — but it has already consumed mempool and batcher resources and may be included in a block as a failed transaction, polluting the block body and state diff. More critically, if the alias contract (0x2) ever has a class hash set by the OS pre-processing step, an admitted transaction could attempt to call into it, potentially corrupting the alias storage that drives stateful state-diff compression and therefore the state root.

### Likelihood Explanation

Trivially reachable: any external caller can submit an RPC transaction with `sender_address = 0x2`. No privileged access, no special network position, and no prior state is required. The only barrier is that the transaction ultimately fails on-chain, but the admission-level invariant is broken before that point.

### Recommendation

Update `ContractAddress::validate()` to use the highest OS-reserved address as the lower bound:

```diff
-    if (value > BLOCK_HASH_TABLE_ADDRESS.0.0) && (value < l2_address_upper_bound) {
+    // Lower bound must be above all OS-reserved addresses (0x1, 0x2, 0x3).
+    const RESERVED_CONTRACT_ADDRESS: Felt = Felt::from_hex_unchecked("0x3");
+    if (value > RESERVED_CONTRACT_ADDRESS) && (value < l2_address_upper_bound) {
```

Alternatively, derive the lower bound from `OsContractAddresses::reserved_contract_address()` so it stays in sync with the versioned constants automatically.

### Proof of Concept
