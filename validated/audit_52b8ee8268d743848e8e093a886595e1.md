Looking at the exact code path in `StateDB.Commit()` and the `SelfDestruct6780` implementation, plus the `DeleteAccount` function.

### Title
Non-EVM-Denom (IBC) Bank Balances Not Drained on SELFDESTRUCT ÔÇö Orphaned Balance Recoverable via CREATE2 Redeployment - (`x/evm/statedb/statedb.go`)

### Summary

`StateDB.Commit()` only burns the EVM-native denom when processing a selfdestructed account. Non-EVM-native bank balances (IBC tokens, CosmWasm bridge tokens) held by the destroyed address are silently skipped. After `DeleteAccount` removes the auth account and storage, those balances remain as orphaned entries in the Cosmos bank module. An attacker who controls the CREATE2 parameters can redeploy a contract at the identical deterministic address and immediately access the previously-orphaned IBC tokens.

---

### Finding Description

In `StateDB.Commit()`, the selfdestruct branch reads:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, ...)
    s.keeper.SubBalance(cacheCtx, cosmosAddr, coin)
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil { ... }
writeCache()
``` [1](#0-0) 

`GetBalance` is called with `s.evmDenom` only ÔÇö a single hard-coded denom. Any other denom held by the address is never queried, never subtracted, and never burned.

`DeleteAccount` confirms this: it clears EVM storage slots and removes the auth account record, but the function comment explicitly states "NOTE: balance should be cleared separately" ÔÇö and only the EVM denom is cleared separately. [2](#0-1) 

`SelfDestruct6780` enforces EIP-6780: it only marks the account for deletion when `stateObject.newContract == true`, i.e., the contract was created in the **same** EVM transaction. [3](#0-2) 

---

### Impact Explanation

After `Commit()`:
- The auth account is gone (`RemoveAccount`).
- EVM storage is wiped.
- The EVM denom balance is burned.
- **IBC token balances remain in the bank module at the now-deleted address.**

When an attacker redeploys a contract at the same CREATE2 address (same factory + same salt + same init-code hash), the bank module creates a fresh `EthAccount` at that address. The pre-existing IBC balance is now owned by the new contract, which the attacker controls. The attacker can then call any transfer/withdraw function to drain those tokens.

This violates the invariant that a selfdestructed account must have zero total bank balance after commit.

---

### Likelihood Explanation

**Exploitability depends on hardfork configuration:**

**Pre-Cancun chains (SELFDESTRUCT always destroys):**
The full attack is reachable with three unprivileged transactions:
1. EVM tx: `CREATE2` deploy contract at address `A`.
2. Cosmos tx: `bank.MsgSend` IBC tokens to address `A` (permissionless for non-blocked addresses).
3. EVM tx: call the contract's selfdestruct function Ôćĺ `StateDB.Commit()` burns only `evmDenom`, IBC tokens survive Ôćĺ `DeleteAccount` removes the auth record.
4. EVM tx: `CREATE2` redeploy at same address Ôćĺ drain IBC tokens.

**Post-Cancun chains (EIP-6780 active):**
The sequence above does **not** work as written because `SelfDestruct6780` returns `(balance, false)` when the contract was created in a prior transaction ÔÇö the account is never marked `selfDestructed`, so `Commit()` never calls `DeleteAccount`. The attack requires all four steps (CREATE2 + bank.Send via precompile + SELFDESTRUCT) to occur within a **single** EVM transaction, which requires a custom `bank.SendCoins` precompile using `ExecuteNativeAction`. Base Ethermint ships no such precompile, but the `ExecuteNativeAction` infrastructure is present and chains built on Ethermint (e.g., Cronos) do register bank precompiles. [3](#0-2) 

---

### Recommendation

In the `obj.selfDestructed` branch of `Commit()`, iterate over **all** bank balances held by `cosmosAddr` (not just `s.evmDenom`) and burn or transfer each one before calling `DeleteAccount`. The Cosmos SDK `BankKeeper` exposes `GetAllBalances(ctx, addr)` for this purpose. Wrap the full drain + delete in the existing `CacheContext` so the operation remains atomic.

---

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
```
