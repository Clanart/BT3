### Title
`compute_meta_tx_v0_hash` Omits Nonce, Enabling Unlimited Signature Replay Against Any Target Account - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that deliberately omits a nonce and passes `additional_data_size=0`. The OS also explicitly skips nonce checking and incrementing for version-0 transactions. Together, these two design choices mean that any valid meta-tx signature is permanently reusable: an attacker who observes a single on-chain meta-tx can replay it an unlimited number of times by submitting new outer invoke transactions, each time causing the target account's `__execute__` to run with the same calldata and the same verified signature.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

The resulting hash is a pure function of `(INVOKE_PREFIX, 0, contract_address, __execute__, calldata, 0, chain_id)`. It is identical on every block, for every outer transaction, for every caller.

The Rust implementation in `syscall_base.rs` mirrors this exactly — it constructs an `InvokeTransactionV0` with `nonce` absent from the struct and calls `calculate_transaction_hash`, which resolves to `get_common_invoke_transaction_v0_hash`:

```rust
let transaction_hash = InvokeTransactionV0 {
    max_fee: Fee(0),
    signature: signature.clone(),
    contract_address,
    entry_point_selector,
    calldata,
}
.calculate_transaction_hash(
    &self.context.tx_context.block_context.chain_info.chain_id,
    &signed_tx_version(&TransactionVersion::ZERO, ...),
)?;
``` [2](#0-1) 

`get_common_invoke_transaction_v0_hash` chains only `INVOKE`, `version`, `contract_address`, `entry_point_selector`, `calldata_hash`, `max_fee`, and `chain_id` — no nonce field: [3](#0-2) 

The second half of the vulnerability is in `check_and_increment_nonce`, which explicitly returns early for version-0 transactions without touching the target contract's nonce:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [4](#0-3) 

`execute_meta_tx_v0` in `syscall_impls.cairo` hardcodes `nonce=0` in the synthetic `TxInfo` it builds for the target contract:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    account_contract_address=contract_address,
    ...
    nonce=0,
    ...
);
``` [5](#0-4) 

Because the hash never changes and the nonce is never incremented, the target account's `__validate__` always receives the same `(transaction_hash, nonce=0)` pair. Any account contract that verifies the signature over `transaction_hash` will accept the same signature on every replay.

---

### Impact Explanation

An attacker who observes a single successful `meta_tx_v0` call on-chain learns the tuple `(contract_address, calldata, signature)`. They can then submit arbitrarily many new outer invoke transactions (paying only their own outer-tx fees) that call `execute_meta_tx_v0` with the same parameters. Each replay:

1. Passes the OS-level selector check (`selector == __execute__`).
2. Recomputes the identical meta-tx hash.
3. Presents `nonce=0` to the target account's `__validate__`, which the account cannot distinguish from the original invocation because the target contract's nonce is never incremented by meta-txs.
4. Executes `__execute__` with the original calldata — transferring tokens, updating storage, or performing any other state-changing operation the original signer intended to happen only once.

This maps to: **Critical — Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce logic.**

---

### Likelihood Explanation

- The attacker needs no special privilege: any funded account can submit outer invoke transactions.
- The original meta-tx parameters are fully visible on-chain after the first execution.
- The replay cost is only the outer transaction's gas fee, which is bounded and predictable.
- No cryptographic break is required; the original signature is reused verbatim.

---

### Recommendation

Include a replay-prevention field in the meta-tx hash. The most natural choice is to bind the hash to the target contract's current nonce at the time of signing, mirroring how `compute_l1_handler_transaction_hash` passes `additional_data_size=1, additional_data=&nonce`:

```cairo
// In compute_meta_tx_v0_hash, add a nonce parameter and pass it as additional_data:
additional_data_size=1,
additional_data=&nonce,
``` [6](#0-5) 

The OS must then also increment the target contract's nonce after a successful meta-tx execution (removing the version-0 early-return guard in `check_and_increment_nonce` for the meta-tx path, or adding a separate increment call in `execute_meta_tx_v0`).

---

### Proof of Concept

1. Deploy an account contract `A` at address `addr_A` that holds 100 STRK.
2. Sign a meta-tx: `sig = ECDSA_sign(key_A, compute_meta_tx_v0_hash(addr_A, __execute__, transfer_50_STRK_to_attacker, chain_id))`.
3. Submit outer invoke tx T1 from any funded account calling `execute_meta_tx_v0(addr_A, __execute__, transfer_50_STRK_to_attacker, sig)`. This executes and transfers 50 STRK.
4. Submit outer invoke tx T2 (new nonce for the outer account, but same meta-tx parameters). The meta-tx hash is identical; `nonce=0` is presented to `A.__validate__`; the target contract's nonce is still 0 (never incremented). The signature verifies. Another 50 STRK is transferred.
5. Repeat until `A` is drained.

The test in `crates/starknet_os_flow_tests/src/tests.rs` confirms that the meta-tx hash is computed without a nonce and that `nonce=0` is always the value seen inside the meta-tx execution context: [7](#0-6)
