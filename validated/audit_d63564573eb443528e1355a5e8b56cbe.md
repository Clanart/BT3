### Title
State Sync Header `shard_id` Not Verified Against Embedded Chunk's `shard_id` — (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`set_state_header` accepts a caller-supplied `shard_id` parameter and an attacker-controlled `ShardStateSyncResponseHeader`, but never checks that `chunk.shard_id() == shard_id`. A malicious peer can embed a chunk from shard X inside a header response for shard Y. All existing checks pass, and the node stores shard X's `prev_state_root` under shard Y's storage key. Subsequent state-part download and application then write shard X's trie into shard Y's `shard_uid`, permanently corrupting the syncing node's state.

---

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` performs the following checks on the received `ShardStateSyncResponseHeader`:

1. **Chunk internal validity** — `validate_chunk_proofs` verifies the chunk's own body/header consistency.
2. **Chunk inclusion in block** — a merkle path is verified against `sync_prev_block_header.chunk_headers_root()` using `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())`.
3. **Incoming receipts validity** — each `ReceiptProof` is hashed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` and verified against the block's `prev_chunk_outgoing_receipts_root`.
4. **State root node validity** — `validate_state_root_node` checks the node against `chunk_inner.prev_state_root()`.

**What is never checked**: that `chunk.shard_id() == shard_id`.

The `chunk_headers_root` merkle tree contains one leaf per shard. A chunk from shard X is a valid leaf in that tree. The merkle proof in step 2 only proves "this chunk is somewhere in the block", not "this chunk is the one for the requested shard". Because `chunk.shard_id()` is never compared to the `shard_id` parameter, an attacker can substitute any shard's chunk.

The receipts check (step 3) uses the caller-supplied `shard_id` (= Y) to compute the hash, so it correctly validates receipts destined for shard Y regardless of which chunk is embedded. The `to_shard_id` field inside `ShardProof` is explicitly discarded:

```rust
let ShardProof { from_shard_id, to_shard_id: _, proof } = shard_proof;
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
```

This means an attacker can satisfy the receipts check by supplying the real incoming receipts for shard Y (publicly available on-chain), while embedding shard X's chunk (and therefore shard X's `prev_state_root`).

The corrupted state root is then committed to `DBCol::StateHeaders` under `StateHeaderKey(shard_id=Y, sync_hash)`:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

`set_state_part` later reads this header, extracts `chunk.prev_state_root()` (shard X's root), and validates incoming state parts against it. `validate_state_part_impl` only takes `state_root`, `part_id`, and `part` — it does not re-check `shard_id`. Parts for shard X's trie pass validation and are written into shard Y's `shard_uid` by `apply_state_part`.

---

### Impact Explanation

**Critical.** A syncing node ends up with shard X's trie committed as shard Y's state. Every subsequent chunk execution for shard Y operates on the wrong state root, producing invalid execution results. If the node is a chunk producer or validator for shard Y, it will produce or endorse invalid chunks, leading to slashing or ejection. The corruption is silent — no error is returned — and persists across restarts because it is written to the node's database.

---

### Likelihood Explanation

**High.** Any peer on the network can respond to a state sync header request. The attack requires only publicly available on-chain data (a real chunk from shard X and the real incoming receipts for shard Y). No privileged role is needed. State sync is triggered automatically when a node falls behind, making the attack window predictable and repeatable.

---

### Recommendation

Add an explicit binding check immediately after extracting the chunk from the header, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// Bind the embedded chunk to the requested shard.
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Apply the same check to `prev_chunk_header` if present:

```rust
if let Some(ref prev_chunk_header) = prev_chunk_header {
    if prev_chunk_header.shard_id() != shard_id {
        byzantine_assert!(false);
        return Err(Error::Other("set_shard_state failed: prev_chunk shard_id mismatch".into()));
    }
}
```

### Proof of Concept

**Setup**: two-shard network (shard 0, shard 1). Node B is state-syncing shard 1 from node A (malicious).

**Attacker constructs a malicious `ShardStateSyncResponseHeader` for shard 1**:

- `chunk` = the real `ShardChunk` for **shard 0** from `sync_prev_block` (fetched from any honest peer or archive node).
- `chunk_proof` = the merkle path for shard 0's `ChunkHashHeight` in `sync_prev_block.chunk_headers_root()` — valid by construction.
- `prev_chunk_header` = the real `ShardChunkHeader` for **shard 0** from the block before `chunk.height_included()`.
- `prev_chunk_proof` = the corresponding merkle path — valid by construction.
- `incoming_receipts_proofs` = the real incoming receipts for **shard 1** (fetched from any honest peer) — these satisfy `ReceiptList(shard_id=1, receipts)` hashing and the block's `prev_chunk_outgoing_receipts_root` check.
- `root_proofs` = the real root proofs for those shard-1 receipts — valid by construction.
- `state_root_node` = shard 0's real `StateRootNode` (consistent with `chunk.prev_state_root()`).

**Node B calls `set_state_header(shard_id=1, sync_hash, malicious_header)`**:

- `validate_chunk_proofs` → passes (shard 0's chunk is internally valid).
- Chunk merkle proof → passes (shard 0's chunk is in `chunk_headers_root`).
- Receipts verification → passes (receipts are the real shard-1 receipts, hash computed with `shard_id=1`).
- State root node → passes (matches shard 0's `prev_state_root`).
- **No check on `chunk.shard_id() == 1`.**
- `DBCol::StateHeaders[StateHeaderKey(1, sync_hash)]` is written with shard 0's `prev_state_root`.

**Node B then downloads state parts** (from the same attacker) for shard 0's trie. `set_state_part` validates them against shard 0's state root → passes. `apply_state_part` writes shard 0's trie nodes into `shard_uid` for shard 1.

**Result**: Node B's shard 1 state is shard 0's trie. All subsequent shard-1 execution on node B is incorrect. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
