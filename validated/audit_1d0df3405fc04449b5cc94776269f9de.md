### Title
Ineffective Revert-Status Constraint Allows Sequencer to Skip Transaction Execution While Charging Fees - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted` in `execution_constraints.cairo` is a complete no-op. It is called in `execute_invoke_function_transaction` to validate the hint-supplied `is_reverted` flag, but it imposes zero Cairo constraints on that value. Because the flag is set exclusively by an unconstrained hint (`%{ IsReverted %}`), a sequencer can set it to any felt value and still produce a valid proof. Setting it to `1` for a valid transaction causes `__execute__` to be skipped while fees are still charged, resulting in direct loss of funds for the user.

---

### Finding Description

In `execution_constraints.cairo`, the function `check_is_reverted` is defined as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

It accepts `is_reverted` but immediately returns without asserting anything. This is the direct analog of the zBanc bug: a protection function is called, but it never sets (or in this case, never constrains) the value it is supposed to guard.

In `execute_invoke_function_transaction`, the flow is:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    tempvar _dummy_return_value: ...;
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

The sequence is:
1. `is_reverted` is declared as a `local` variable.
2. The hint `%{ IsReverted %}` fills it — hints are prover-supplied and are not constrained by the proof system.
3. `check_is_reverted(is_reverted)` is called — but it is a no-op, so no Cairo assertion constrains `is_reverted`.
4. The `if (is_reverted == FALSE)` branch in Cairo 0 does not assert that `is_reverted` is boolean; it merely branches.
5. `charge_fee` is called unconditionally after the branch.

Because `is_reverted` is never constrained, the prover can assign it any felt value and the resulting proof remains valid.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A sequencer that sets `is_reverted = 1` for a transaction that would succeed causes:
- `non_reverting_select_execute_entry_point_func` (the `__execute__` call) to be skipped entirely.
- `charge_fee` to still execute, deducting the full fee from the user's account.

The user pays fees but their transaction has no on-chain effect. Because the OS proof is supposed to be the trustless guarantee that the sequencer cannot lie about execution results, the absence of any constraint on `is_reverted` directly breaks this guarantee. The proof verifier on L1 accepts the proof as valid even though the state transition is incorrect.

---

### Likelihood Explanation

The StarkNet OS proof is the mechanism that constrains the sequencer. The entire security model relies on the proof preventing a sequencer from producing an incorrect state transition. The `check_is_reverted` no-op removes this constraint for the revert flag. Any sequencer running a modified prover can exploit this without any additional capability. The entry path is a standard user-submitted invoke transaction — no special privilege is required from the user's side; the vulnerability is in the OS's failure to constrain the sequencer.

---

### Recommendation

Replace the no-op `check_is_reverted` with a real Cairo constraint. At minimum, assert that `is_reverted` is boolean:

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (is_reverted - 1) = 0;
    return ();
}
```

Ideally, derive `is_reverted` from a verifiable execution result rather than a bare hint, or add an assertion that ties the hint value to a provable execution outcome (e.g., the return value of the entry point call).

---

### Proof of Concept

1. User submits a valid invoke transaction (e.g., an ERC-20 transfer that would succeed).
2. Sequencer runs the transaction off-chain, confirms it succeeds, but sets `is_reverted = 1` in the hint `%{ IsReverted %}`.
3. The OS Cairo program executes: `check_is_reverted(1)` returns immediately (no-op); the `if (is_reverted == FALSE)` branch takes the `else` path, skipping `__execute__`; `charge_fee` runs and deducts the fee.
4. The sequencer generates a STARK proof over this execution trace. Because `is_reverted` is unconstrained, the proof is valid.
5. The L1 verifier accepts the proof. The state update is committed: the user's fee balance is reduced, but the intended state change (e.g., the token transfer) never occurred.
6. The user has lost funds with no recourse, and the proof provides no evidence of wrongdoing.
