### Title
Missing Chain ID in `deployToken` Signed Payload Enables Cross-Chain Signature Replay — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies an ECDSA signature over a Borsh-encoded payload that contains **no chain identifier**. Because all EVM deployments of `OmniBridge` share the same `nearBridgeDerivedAddress` (derived from the NEAR MPC key), a valid `deployToken` signature obtained on one EVM chain (e.g., Ethereum) is cryptographically identical and replayable on every other EVM chain (e.g., Arbitrum, Polygon) that deploys the same contract. This is the direct analog of the GnosisBase domain-separation flaw: the source/destination domain is not bound into the signed message.

---

### Finding Description

In `OmniBridge.sol`, `deployToken` constructs the signed payload as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);

if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The payload encodes only: `PayloadType`, `token`, `name`, `symbol`, `decimals`. **No chain ID is included.**

Contrast this with `finTransfer`, which explicitly binds the payload to the destination chain by embedding `omniBridgeChainId` twice:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
    Borsh.encodeUint64(payload.destinationNonce),
    bytes1(payload.originChain),
    Borsh.encodeUint64(payload.originNonce),
    bytes1(omniBridgeChainId),          // destination chain bound here
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),          // and here
    Borsh.encodeAddress(payload.recipient),
    ...
``` [2](#0-1) 

`omniBridgeChainId` is a per-chain constant set at initialization:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    ...
    omniBridgeChainId = omniBridgeChainId_;
``` [3](#0-2) 

Because `nearBridgeDerivedAddress` is derived from the NEAR MPC key — which is shared across all EVM deployments — the same ECDSA signature is valid on every chain. The `deployToken` payload is chain-agnostic, so a signature produced for Ethereum is equally valid on Arbitrum, Base, Polygon, or any other EVM chain running `OmniBridge`.

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (the signature is public on-chain) can replay the identical `(signatureData, metadata)` call on chain B. The `ECDSA.recover` check passes because the hashed payload is identical. The `ERR_TOKEN_EXIST` guard only blocks replay if the token is **already** deployed on chain B; if it has not yet been deployed there, the replay succeeds.

This results in:

- **Unauthorized token deployment** on chain B: a bridge token is created and registered in `nearToEthToken` / `ethToNearToken` / `isBridgeToken` mappings without NEAR bridge authorization for that chain.
- **Token mapping pre-emption**: the attacker's deployment occupies the `nearToEthToken[metadata.token]` slot on chain B. When the NEAR bridge later attempts to officially deploy the same token on chain B, the call reverts with `ERR_TOKEN_EXIST`, **permanently blocking** the legitimate deployment.
- **Accounting desync**: if the NEAR bridge processes the `DeployToken` event emitted by the attacker's replay and registers the chain-B address, subsequent `finTransfer` calls on chain B will mint against an address the NEAR bridge did not intend to authorize, breaking bridge collateralization.

This matches the allowed impact: **High — Proof/signature verification bypass enabling unauthorized token deployment**, and potentially **Critical — Permanent freezing of the legitimate token deployment flow on chain B**.

---

### Likelihood Explanation

- The attacker needs only to observe a `deployToken` transaction on any live EVM chain (fully public, no privileged access required).
- The attacker needs chain B to be a supported EVM chain with `OmniBridge` deployed and the same `nearBridgeDerivedAddress`.
- No special role, leaked key, or colluding party is required.
- The attack is executable by any unprivileged user who can submit a transaction.

Likelihood: **Medium-High** — trivially executable once a signature exists on any chain.

---

### Recommendation

Include `omniBridgeChainId` in the `deployToken` signed payload, exactly as `finTransfer` does:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD THIS
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The corresponding NEAR-side signing in `log_metadata_callback` must also include the target chain ID in `MetadataPayload` so that signatures are chain-specific. [4](#0-3) 

---

### Proof of Concept

1. NEAR bridge operator calls `log_metadata` on NEAR for token `token.near`. The MPC signer produces signature `S` over `keccak256(Metadata || "token.near" || "Token" || "TKN" || 18)`.
2. A relayer submits `deployToken(S, metadata)` on **Ethereum** (chain A, `omniBridgeChainId = 1`). The token is deployed at address `0xAAA...` and the event `DeployToken` is emitted.
3. An attacker observes `S` on-chain (fully public).
4. The attacker calls `deployToken(S, metadata)` on **Arbitrum** (chain B, `omniBridgeChainId = 2`). The hashed payload is identical (no chain ID), so `ECDSA.recover` returns `nearBridgeDerivedAddress`. The check passes.
5. `isBridgeToken[nearToEthToken["token.near"]]` is `false` on Arbitrum (token not yet deployed there), so `ERR_TOKEN_EXIST` does not trigger.
6. A new bridge token is deployed at `0xBBB...` on Arbitrum. `nearToEthToken["token.near"] = 0xBBB...` and `isBridgeToken[0xBBB...] = true` are set.
7. When the NEAR bridge later attempts to officially deploy `token.near` on Arbitrum, the call reverts: `isBridgeToken[nearToEthToken["token.near"]]` is now `true`. The legitimate deployment is permanently blocked on Arbitrum.
