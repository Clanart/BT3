Audit Report

## Title
Unpartitioned `address(this).balance` in `pay` allows any WETH-input swap to consume stranded ETH from prior callers — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` helper in `PeripheryPayments` uses `address(this).balance` — the router's entire, unpartitioned native ETH balance — when settling a WETH payment. Because no per-call ETH budget is tracked, ETH left in the router from any prior user's transaction is indistinguishable from ETH legitimately sent by the current caller. An attacker can call `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`, causing the router to wrap and forward a victim's stranded ETH to settle the attacker's swap at zero cost.

## Finding Description
In `PeripheryPayments.pay` (lines 73–84), when `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` and, if it covers `value`, wraps and transfers that ETH to the recipient without pulling anything from `payer`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);  // payer never touched
}
```

Every swap entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) is `payable`. The `receive()` guard only rejects direct ETH sends (no calldata); ETH attached to any of these calls is accepted freely. When a user sends excess ETH with a WETH swap (e.g., 1.5 ETH for a 1 ETH swap), the surplus 0.5 ETH remains in the router after the callback settles. The `TransientCallbackPool` transient storage tracks the pool, callback mode, payer address, and token to pay — but has no slot recording `msg.value` or any per-call ETH budget. Consequently, when a subsequent attacker calls `exactInputSingle({tokenIn: WETH, amountIn: 0.5e18})` with `msg.value = 0`, the callback fires `pay(WETH, attacker, pool, 0.5e18)`, finds `address(this).balance == 0.5e18 >= 0.5e18`, and wraps the victim's ETH to settle the attacker's swap. The only existing guard in the callback path — `_requireExpectedCallbackCaller` — validates the pool identity, not the ETH source. `refundETH` then returns nothing to the victim because the balance is zero.

## Impact Explanation
Direct loss of user principal (native ETH). The victim loses the full amount of ETH stranded in the router; the attacker receives a complete WETH swap settled at zero cost. The loss is bounded only by the stranded ETH amount, which equals the excess `msg.value` sent by the victim — an unbounded, user-controlled quantity. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation
ETH stranding is a normal operational condition: users routinely send excess ETH with WETH swaps to avoid computing the exact amount off-chain, then call `refundETH` in a separate transaction or as a subsequent multicall leg. Any window between the swap and the `refundETH` call — including separate transactions or failed multicall legs — leaves ETH exploitable. An attacker can monitor the router's ETH balance on-chain and front-run the `refundETH` call with a zero-cost WETH swap. The attack requires no special privileges, no malicious token, and no unusual preconditions beyond a victim having stranded ETH in the router.

## Recommendation
Track per-call ETH contributions in transient storage (EIP-1153, already used throughout this codebase via `TransientCallbackPool`). At the start of each `payable` entry point, record `msg.value` in a dedicated transient slot (e.g., `T_MSG_VALUE_SLOT`). In `pay`, when `token == WETH`, consume only up to the recorded per-call ETH budget rather than `address(this).balance`. Decrement the budget by the amount consumed. Clear the slot after the call completes. This ensures ETH from one user's call cannot be consumed by another user's callback.

## Proof of Concept
1. **User A** calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18, ...})` sending `1.5e18` ETH as `msg.value`. The callback fires; `pay(WETH, userA, pool, 1e18)` wraps 1 ETH and sends WETH to the pool. Swap succeeds. **0.5e18 ETH remains in the router.**
2. User A submits `refundETH()` to recover the 0.5 ETH surplus.
3. **Attacker** observes the pending `refundETH` in the mempool and front-runs it with `exactInputSingle({tokenIn: WETH, amountIn: 0.5e18, ...})` sending **0 ETH**.
4. The callback fires; `pay(WETH, attacker, pool, 0.5e18)` is called. `address(this).balance == 0.5e18 >= 0.5e18`, so the router wraps User A's 0.5 ETH and sends WETH to the pool. **Attacker's swap settles at zero cost.**
5. User A's `refundETH()` executes: `address(this).balance == 0`, nothing is returned. **User A loses 0.5 ETH.**

Foundry test: deploy router with WETH, fund User A's swap with 1.5e18 ETH, assert 0.5e18 remains in router, have attacker call `exactInputSingle` with 0 ETH and `amountIn = 0.5e18`, assert attacker received WETH output and router balance is 0, assert User A's `refundETH` returns 0.
