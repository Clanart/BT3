### Title
Permanently Locked ETH Due to `receive()` With No Withdrawal Function - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol` declares `receive() external payable {}` at line 574, allowing ETH to be sent directly to the contract. No `withdraw`, `rescue`, or `recoverEth` function exists anywhere in the EVM contract suite. Any ETH deposited via the bare `receive()` fallback — or sent with calls to `payable` functions whose `msg.value` is not fully consumed — is permanently irrecoverable.

---

### Finding Description

`OmniBridge.sol` exposes a bare `receive()` function:

```solidity
receive() external payable {}   // line 574
``` [1](#0-0) 

A comprehensive search across all EVM contracts (`evm/src/**/*.sol`) for any of `withdraw`, `rescue`, `recoverEth`, or `sweep` returns **zero matches**. There is no administrative or user-facing path to recover ETH that enters the contract outside the normal `initTransfer` / `finTransfer` accounting.

Additionally, three `external payable` functions in the base `OmniBridge` contract accept `msg.value` but have no-op extension hooks in the base class:

| Function | Line | Extension hook in base `OmniBridge` |
|---|---|---|
| `deployToken` | 135–195 | `deployTokenExtension` → empty virtual |
| `logMetadata` | 224–232 | `logMetadataExtension` → empty virtual |
| `logMetadata1155` | 234–270 | `logMetadataExtension` → empty virtual | [2](#0-1) [3](#0-2) [4](#0-3) 

In `OmniBridgeWormhole`, the overrides forward `msg.value` to `_wormhole.publishMessage`, so those three functions are safe in the Wormhole deployment. However, the `receive()` fallback remains unguarded in both the base and Wormhole variants, and no withdrawal path exists in either. [5](#0-4) 

---

### Impact Explanation

Any ETH that enters the contract via the `receive()` fallback is permanently locked. This includes:

- Users who accidentally send ETH directly to the contract address (a common mistake when interacting with bridge contracts).
- Any protocol-level ETH accumulation from rounding, fee residuals, or failed `finTransfer` ETH sends that leave a remainder.

Because the contract holds real ETH for native-ETH bridge flows (`finTransfer` with `tokenAddress == address(0)` sends ETH to recipients), the contract balance is non-trivial in production. ETH deposited outside the `initTransfer` accounting is silently absorbed with no recovery path.

**Impact class:** Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows. [6](#0-5) 

---

### Likelihood Explanation

- The `receive()` function is publicly reachable by any EOA or contract with no preconditions.
- Accidental ETH sends to contract addresses are a well-documented, recurring user error in EVM ecosystems.
- The bridge contract is a high-value target that users interact with directly; the probability of at least one misdirected ETH send over the contract's lifetime is high.
- No privileged role or special condition is required to trigger the lock.

---

### Recommendation

1. **Remove `receive() external payable {}`** if the contract does not need to accept plain ETH transfers. ETH for native bridging enters via `initTransfer` (which is already `payable`), so the bare fallback is unnecessary.
2. If the `receive()` function must be retained (e.g., for Wormhole refunds), add an admin-only ETH rescue function:

```solidity
function rescueEth(address payable to, uint256 amount)
    external
    onlyRole(DEFAULT_ADMIN_ROLE)
{
    (bool ok, ) = to.call{value: amount}("");
    require(ok, "ETH transfer failed");
}
```

3. Consider removing the `payable` modifier from `logMetadata`, `logMetadata1155`, and `deployToken` in the base `OmniBridge` contract, since the base class extension hooks are no-ops and any `msg.value` sent to those functions in a non-Wormhole deployment is permanently lost. [1](#0-0) 

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` (or base `OmniBridge`) to a testnet.
2. From any EOA, send 1 ETH directly to the contract address (triggers `receive()`).
3. Observe that the contract balance increases by 1 ETH.
4. Attempt to recover the ETH — no `withdraw`, `rescue`, or equivalent function exists. The ETH is permanently locked.
5. Confirm: grep the entire `evm/src/` tree for `withdraw`, `rescue`, `recoverEth`, `sweep` — zero matches. [7](#0-6)
