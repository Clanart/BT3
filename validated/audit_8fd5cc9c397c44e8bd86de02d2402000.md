### Title
Integer Underflow in `nestedCreates` Nonce Reconciliation Resets Sender Nonce on Failed CREATE, Enabling Transaction Replay - (File: x/evm/keeper/state_transition.go)

### Summary

In `ApplyMessageWithConfig`, when processing a contract-creation (`contractCreation`) message, Ethermint resets the stateDB nonce to `msg.Nonce` before calling `evm.Create()`, then reconciles the nonce afterward using an unsigned subtraction. If `evm.Create()` returns early with `ErrInsufficientBalance` (or `ErrDepth`) **before** incrementing the nonce, the reconciliation arithmetic underflows as `uint64`, wrapping the sender's committed nonce back to its pre-transaction value. This allows the same nonce to be reused in a future block — a transaction replay.

### Finding Description

In `x/evm/keeper/state_transition.go`, the `contractCreation` branch of `ApplyMessageWithConfig` performs the following nonce management:

```go
oldNonce := stateDB.GetNonce(sender)          // e.g. N+1 (ante handler pre-incremented)
stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to N
ret, _, leftoverGas, vmErr = evm.Create(...)   // may return ErrInsufficientBalance
afterCreateNonce := stateDB.GetNonce(sender)   // still N if evm.Create returned early
nestedCreates := afterCreateNonce - msg.Nonce - 1  // N - N - 1 = uint64 underflow = 2^64-1
stateDB.SetNonce(sender, oldNonce+nestedCreates, ...) // (N+1) + (2^64-1) = N (mod 2^64)
``` [1](#0-0) 

In go-ethereum's `evm.create()`, both `ErrDepth` and `ErrInsufficientBalance` are returned **before** the nonce increment:

```go
if evm.depth > int(params.CallCreateDepth) {
    return nil, common.Address{}, gas, ErrDepth          // nonce NOT incremented
}
if !evm.Context.CanTransfer(evm.StateDB, caller.Address(), value) {
    return nil, common.Address{}, gas, ErrInsufficientBalance  // nonce NOT incremented
}
// ... only here is nonce incremented
evm.StateDB.SetNonce(caller.Address(), nonce+1, ...)
```

When either early-return fires, `afterCreateNonce == msg.Nonce`, causing the `uint64` subtraction `afterCreateNonce - msg.Nonce - 1` to wrap to `2^64 - 1`. Adding this to `oldNonce` (= `msg.Nonce + 1` after the ante handler's single increment) wraps back to `msg.Nonce`. The stateDB is then committed with this corrupted nonce value. [2](#0-1) 

The ante handler increments the nonce for every message in a batch upfront: [3](#0-2) 

The `CheckEthCanTransfer` check evaluates each message's value against the balance **independently at ante time**, before any execution has drained the balance: [4](#0-3) 

This means a batch transaction can pass ante validation even when a later message's value transfer will fail at execution time.

### Impact Explanation

After the batch transaction is committed, the sender's nonce is `msg.Nonce` (= N+1 for the second message) instead of the correct `N+2`. The attacker can now submit any transaction with nonce `N+1` in a future block — including a replay of the original CREATE transaction (once balance is restored) or any other transaction at that nonce. This constitutes a **nonce bypass enabling replay and unauthorized account/code mutation**, matching the High impact category. A contract deployed via replay would be at the same deterministic address (`keccak256(sender, N+1)`), potentially colliding with or overwriting an expected deployment.

### Likelihood Explanation

The attack requires a batch Cosmos transaction (two `MsgEthereumTx` in one envelope), which is a supported but discouraged feature per ADR-003.



The attacker controls both messages and their values. No privileged role is needed. The ante handler's per-message balance check (not cumulative) makes the precondition reachable by any unprivileged user who can craft a batch transaction where message 1 drains the balance needed by message 2's CREATE value.

### Recommendation

Guard the subtraction against underflow before computing `nestedCreates`. If `afterCreateNonce <= msg.Nonce` (i.e., `evm.Create()` did not increment the nonce), treat `nestedCreates` as zero and restore `oldNonce` directly:

```go
afterCreateNonce := stateDB.GetNonce(sender)
var nestedCreates uint64
if afterCreateNonce > msg.Nonce+1 {
    nestedCreates = afterCreateNonce - msg.Nonce - 1
}
stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
```

This ensures that a failed `evm.Create()` (which does not increment the nonce) leaves the sender's nonce at `oldNonce` (the ante-handler-incremented value), preserving replay protection.

### Proof of Concept

1. Sender account has balance `B` (enough for fees + value of msg2).
2. Attacker submits a batch Cosmos transaction containing:
   - **msg1** (nonce N): CALL to a contract that transfers all of sender's balance to attacker-controlled address.
   - **msg2** (nonce N+1): CREATE with `value = B` (contract creation sending ETH).
3. Ante handler passes: at ante time, balance `B >= B` for both messages independently.
4. Ante handler increments nonce to `N+2`.
5. msg1 executes: balance drained to ~0.
6. msg2 executes:
   - `oldNonce = N+2`
   - `stateDB.SetNonce(sender, N+1, ...)` (reset to msg.Nonce)
   - `evm.Create()` → `CanTransfer` fails → returns `ErrInsufficientBalance` (nonce stays at N+1)
   - `afterCreateNonce = N+1`
   - `nestedCreates = (N+1) - (N+1) - 1 = 2^64 - 1`
   - `stateDB.SetNonce(sender, (N+2) + (2^64-1), ...) = stateDB.SetNonce(sender, N+1, ...)`
7. stateDB committed: sender nonce = N+1 (should be N+2).
8. Attacker refunds balance to sender, then replays msg2 (nonce N+1) in a future block — it now succeeds, deploying a contract at `keccak256(sender, N+1)`. [1](#0-0) [5](#0-4)
