### Title
Double-Mint via Proof Replay After `remove_fin_transfer` Clears Replay Guard — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a deployed (mint/burn) token transfer to NEAR uses `ft_transfer_call` and the recipient's `ft_on_transfer` returns the full amount (triggering the refund path), `fin_transfer_send_tokens_callback` calls `remove_fin_transfer`, which unconditionally removes the `transfer_id` from `finalised_transfers`. Because all three provers are stateless (no used-proof tracking), the same proof can immediately be submitted again via a second `fin_transfer` call, minting the bridged tokens a second time.

---

### Finding Description

**Step 1 — `fin_transfer` entry point**

`fin_transfer` is gated by `#[trusted_relayer]`, which is permissionlessly obtainable by staking 1000 NEAR and waiting the activation period. Any account can become a trusted relayer. [1](#0-0) 

**Step 2 — `add_fin_transfer` inserts the transfer_id as the sole replay guard**

`process_fin_transfer_to_near` calls `add_fin_transfer`, which panics with `ERR_TRANSFER_ALREADY_FINALISED` if the `transfer_id` is already present. This is the **only** replay protection. [2](#0-1) [3](#0-2) 

**Step 3 — `send_tokens` for deployed tokens mints to bridge then calls `ft_transfer_call`**

For a deployed token with a non-empty `msg`, `send_tokens` calls `mint(bridge, amount, Some(msg))`. The `omni-token` `mint` implementation mints to `env::predecessor_account_id()` (the bridge) and then calls `ft_transfer_call` to the recipient. [4](#0-3) [5](#0-4) 

**Step 4 — Attacker-controlled recipient returns full amount from `ft_on_transfer`**

When `ft_on_transfer` returns the full transferred amount, NEP-141's `ft_resolve_transfer` returns 0 (nothing used). The promise result of `send_tokens` is therefore `U128(0)`.

**Step 5 — `is_refund_required` returns `true`**

`is_refund_required` checks `amount.0 == 0`. The code comment explicitly confirms: *"The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`"*. [6](#0-5) 

**Step 6 — `burn_tokens_if_needed` fires a detached burn, then `remove_fin_transfer` clears the guard**

In the refund branch of `fin_transfer_send_tokens_callback`:
1. `burn_tokens_if_needed` fires a `.detach()`-ed burn promise — fire-and-forget, not awaited.
2. `remove_fin_transfer` **unconditionally removes** the `transfer_id` from `finalised_transfers`. [7](#0-6) [8](#0-7) [9](#0-8) 

**Step 7 — All provers are stateless; no proof-level replay protection exists**

The EVM prover, Wormhole proxy, and MPC prover all verify the cryptographic proof but maintain **no state** about previously verified proofs. They will accept the same proof bytes a second time. [10](#0-9) [11](#0-10) [12](#0-11) 

**Step 8 — Second `fin_transfer` with the same proof succeeds**

After `remove_fin_transfer` clears the guard:
- `verify_proof` succeeds (prover is stateless).
- `add_fin_transfer` succeeds (`transfer_id` is no longer in `finalised_transfers`).
- `send_tokens` mints the bridged tokens again.
- The recipient accepts this time (returns `0` from `ft_on_transfer`).

The attacker now holds tokens backed by only one source-chain lock/burn event.

---

### Impact Explanation

**Critical — unauthorized double-mint of bridged assets on NEAR.**

For every deployed (mint/burn) bridge token, an attacker who controls the recipient NEAR account can cause the bridge to mint the same amount twice for a single source-chain event. The first mint is refunded to the bridge and burned (detached), but the second mint is accepted by the recipient. The token supply on NEAR exceeds the locked/burned supply on the source chain by exactly the transfer amount.

---

### Likelihood Explanation

- The attacker only needs to control a NEAR account that implements `ft_on_transfer` (trivial).
- Becoming a trusted relayer is permissionless: stake 1000 NEAR, wait the activation period.
- Alternatively, a legitimate relayer observing `FailedFinTransferEvent` may retry the proof automatically, requiring no relayer collusion — just the attacker controlling the recipient.
- The proof is valid and will pass all cryptographic checks on the second submission.
- No race condition is required; the second `fin_transfer` can be submitted in any subsequent block after the first callback settles.

---

### Recommendation

**Do not remove the `transfer_id` from `finalised_transfers` in the refund path.** The entry in `finalised_transfers` is the only replay guard. Instead:

1. Keep the `transfer_id` in `finalised_transfers` permanently (or until an explicit DAO-controlled cleanup).
2. If storage refund is needed, use a separate "failed transfers" set or a status flag rather than deletion.
3. Alternatively, move replay protection into the provers themselves (record used proof hashes), so the bridge's `finalised_transfers` is not the single point of failure.

---

### Proof of Concept

```
1. Source chain: legitimate init_transfer for deployed token T, amount A, recipient = attacker_contract.near, msg = "some_msg".
2. Relayer calls fin_transfer(proof_1) on NEAR bridge.
3. fin_transfer_callback → process_fin_transfer_to_near:
     - add_fin_transfer(transfer_id) → finalised_transfers = {transfer_id}
     - send_tokens → mint(bridge, A, Some("some_msg")) → ft_transfer_call(attacker_contract, A, "some_msg")
4. attacker_contract.ft_on_transfer returns A (full refund).
   ft_resolve_transfer returns 0. Bridge holds A tokens.
5. fin_transfer_send_tokens_callback: is_refund_required = true (amount == 0).
     - burn_tokens_if_needed(bridge, A).detach()   ← fire-and-forget
     - remove_fin_transfer(transfer_id)             ← finalised_transfers = {}
     - emit FailedFinTransferEvent
6. Attacker (trusted relayer, staked 1000 NEAR) calls fin_transfer(proof_1) again.
7. verify_proof(proof_1) succeeds (prover is stateless).
   add_fin_transfer(transfer_id) succeeds (set is empty).
   send_tokens → mint(bridge, A, Some("some_msg")) → ft_transfer_call(attacker_contract, A, "some_msg")
8. attacker_contract.ft_on_transfer returns 0 (accepts tokens).
   ft_resolve_transfer returns A.
9. fin_transfer_send_tokens_callback: is_refund_required = false → FinTransferEvent emitted.
10. Attacker holds A tokens. Source chain locked/burned A tokens once. Net: A unbacked tokens minted.
```

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }

    pub fn verify_proof(&self, #[serializer(borsh)] input: Vec<u8>) -> Result<Promise, String> {
        let args = EvmVerifyProofArgs::try_from_slice(&input)
            .map_err(|_| ProverError::ParseArgs.to_string())?;

        let evm_proof = args.proof;
        let header: BlockHeader = rlp::decode(&evm_proof.header_data).map_err(|e| e.to_string())?;
        let log_entry: LogEntry =
            rlp::decode(&evm_proof.log_entry_data).map_err(|e| e.to_string())?;
        let receipt: Receipt = rlp::decode(&evm_proof.receipt_data).map_err(|e| e.to_string())?;

        // Verify log_entry included in receipt
        let log_index_usize = usize::try_from(evm_proof.log_index).map_err(|e| e.to_string())?;
        require!(receipt.logs[log_index_usize] == log_entry);

        // Verify receipt included into header
        let data = Self::verify_trie_proof(
            header.receipts_root.0,
            rlp::encode(&evm_proof.receipt_index).to_vec(),
            &evm_proof.proof,
        );

        if evm_proof.receipt_data != data {
            return Err(ProverError::InvalidProof.to_string());
        }

        // Verify block header was in the bridge
        Ok(evm_client::ext(self.light_client.clone())
            .with_static_gas(BLOCK_HASH_SAFE_GAS)
            .block_hash_safe(header.number.as_u64())
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(VERIFY_PROOF_CALLBACK_GAS)
                    .verify_proof_callback(
                        args.proof_kind,
                        evm_proof.log_entry_data,
                        header
                            .hash
                            .ok_or_else(|| ProverError::HashNotSet.to_string())?
                            .0,
                    ),
            ))
    }

    pub fn verify_proof(&self, #[serializer(borsh)] input: Vec<u8>) -> Promise {
        let args = MpcVerifyProofArgs::try_from_slice(&input).near_expect(ProverError::ParseArgs);

        let sign_payload = ForeignTxSignPayload::try_from_slice(&args.sign_payload)
            .near_expect(ProverError::ParseArgs);

        let ForeignTxSignPayload::V1(ref payload_v1) = sign_payload;

        let chain_kind = Self::request_to_chain_kind(&payload_v1.request)
            .near_expect(ProverError::UnsupportedChain);

        let finality = self
            .finalities
            .get(&chain_kind)
            .near_expect(ProverError::UnsupportedChain);

        require!(
            Self::request_matches_finality(&payload_v1.request, finality),
            ProverError::FinalityMismatch.as_ref()
        );

        let request_args = VerifyForeignTransactionRequestArgs {
            request: payload_v1.request.clone(),
            domain_id: DomainId(FOREIGN_TX_DOMAIN_ID),
            payload_version: ForeignTxPayloadVersion::V1,
        };

        ext_mpc_contract::ext(self.mpc_contract_id.clone())
            .with_static_gas(VERIFY_FOREIGN_TX_GAS)
            .with_attached_deposit(ONE_YOCTO)
            .verify_foreign_transaction(request_args)
            .then(
                Self::ext(near_sdk::env::current_account_id())
                    .with_static_gas(VERIFY_CALLBACK_GAS)
                    .verify_callback(args.proof_kind, args.sign_payload, chain_kind),
            )
    }
