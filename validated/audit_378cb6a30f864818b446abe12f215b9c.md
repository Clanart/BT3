### Title
Unconstrained `is_reverted` Flag Enables Fee Charging Without Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is an empty stub that performs no validation. The `is_reverted` flag, loaded from a prover-controlled hint, is used to decide whether to execute a transaction's `__execute__` entry point, but is never constrained by any Cairo assertion. A malicious prover can set `is_reverted = TRUE` for any transaction, causing the OS to skip execution while still charging the user's fee — resulting in direct loss of funds with a cryptographically valid proof.

---

### Finding Description

In `execute_invoke_function_transaction`, the execution flow is:

1. Run `__validate__` (non-reverting)
2. Load `is_reverted` from hint `%{ IsReverted %}`
3. Call `check_is_reverted(is_reverted)` — **does nothing**
4. Conditionally skip `__execute__` if `is_reverted != FALSE`
5. Always call `charge_fee` [1](#0-0) 

The `check_is_reverted` function that is supposed to validate this flag: [2](#0-1) 

The function body is entirely empty — it accepts `is_reverted: felt` and immediately returns, placing **zero constraints** on the value. In Cairo's ZK proof model, only explicit `assert` statements constrain witness values. An unconstrained hint value can be set to anything by the prover while still producing a valid proof.

The `charge_fee` function is called unconditionally after the conditional execute block: [3](#0-2) 

`charge_fee` executes an ERC20 transfer from the user's account to the sequencer address, deducting the actual fee from the user's balance regardless of whether execution occurred: [4](#0-3) 

The same unconstrained `is_reverted` pattern applies to `execute_deploy_account_transaction`, which also uses `check_is_reverted` indirectly through the same constraints module.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious prover sets `is_reverted = TRUE` via the `%{ IsReverted %}` hint for a transaction that should succeed. The OS skips the `__execute__` entry point entirely (no state changes from the user's intended logic), but `charge_fee` still runs and deducts the fee from the user's ERC20 balance. The resulting state transition — fee deducted, execution absent — is committed to the global state root and accepted by the L1 verifier because the proof is cryptographically valid (no assertion was violated). The user permanently loses their fee with no corresponding execution benefit. This is the direct analog to the OmoVault bug: funds (fees) are consumed while the corresponding accounting action (execution) is silently absent, corrupting the expected state.

---

### Likelihood Explanation

**Low.** Exploiting this requires the prover/sequencer to deliberately set the hint incorrectly. In the current StarkNet architecture the sequencer is a centralized trusted party. However, the entire purpose of the OS ZK program is to make the sequencer's behavior trustless and verifiable — the OS is the enforcement layer. The empty `check_is_reverted` breaks this guarantee at the protocol level: the OS program fails to constrain a security-critical execution decision, meaning the proof system cannot detect or prevent this manipulation. Any future decentralized or adversarial prover scenario makes this immediately exploitable.

---

### Recommendation

Replace the empty `check_is_reverted` stub with a real constraint. The OS should derive `is_reverted` from the actual execution outcome rather than accepting it as an unconstrained hint. One approach: always run the execute entry point through `select_execute_entry_point_func` (which returns `is_reverted` as a constrained output of the execution), and use that return value directly rather than loading it from a separate hint. If a hint-based optimization is required for performance, the hint value must be verified against the constrained execution result via an `assert` statement.

---

### Proof of Concept

1. User submits a valid `INVOKE_FUNCTION` transaction (e.g., a token transfer).
2. The OS runs `__validate__` successfully (non-reverting, constrained).
3. Malicious prover sets `is_reverted = 1` via `%{ IsReverted %}`.
4. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
5. The `if (is_reverted == FALSE)` branch is skipped; `__execute__` is never called.
6. `charge_fee` executes: ERC20 `transfer(sequencer, actual_fee)` runs against the fee token contract, deducting the fee from the user's balance.
7. The prover generates a valid STARK proof — no Cairo assertion was violated.
8. The L1 verifier accepts the proof and the state root is updated.
9. **Result**: User's fee balance is reduced; their intended transaction was never executed. Funds are permanently lost. [2](#0-1) [5](#0-4) [6](#0-5)
