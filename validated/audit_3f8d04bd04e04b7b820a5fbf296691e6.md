### Title
Cross-Chain Replay of Legacy (Non-EIP-155) Transactions When `allowUnprotectedTxs=true` — (`rpc/backend/call_tx.go`, `ante/interfaces/setup.go`)

### Summary

When `allowUnprotectedTxs=true` is set (either as an on-chain EVM param or as the RPC node config), the chain-id check in `SendRawTransaction` is gated exclusively on `tx.Protected()`. A legacy transaction signed with `HomesteadSigner` (V=27/28, no chain-id in the signature hash) has `Protected()==false`, so the chain-id guard is never evaluated. The same raw bytes are accepted and committed on any Ethermint node with any chain-id that has this flag enabled.

---

### Finding Description

**RPC layer — `rpc/backend/call_tx.go`:**

```go
// check the local node config in case unprotected txs are disabled
if !b.UnprotectedAllowed() && !tx.Protected() {
    return common.Hash{}, errors.New("only replay-protected (EIP-155) transactions allowed over RPC")
}

// Reject transactions targeting a different chain.
if tx.Protected() && tx.ChainId().Cmp(b.chainID) != 0 {
    return common.Hash{}, fmt.Errorf("incorrect chain-id; expected %d, got %d", b.chainID, tx.ChainId())
}
``` [1](#0-0) 

When `allowUnprotectedTxs=true`, the first guard passes for any unprotected tx. The second guard is conditioned on `tx.Protected()`, which is `false` for a Homestead-signed tx — so the chain-id comparison is **never reached**.

**Ante handler — `ante/interfaces/setup.go`:**

```go
if !allowUnprotectedTxs && !tx.Protected() {
    return errorsmod.Wrapf(
        errortypes.ErrNotSupported,
        "rejected unprotected Ethereum transaction. ...")
}
``` [2](#0-1) 

The ante handler mirrors the same logic: it only rejects unprotected txs when the param is `false`. When `true`, unprotected txs flow through with no chain-id binding check at any layer.

**Signature verification — `ante/sigverify.go`:**

`VerifyEthSig` calls `msgEthTx.VerifySender(signer)` where `signer = LatestSignerForChainID(chainID)`. [3](#0-2) 

For a Homestead-signed tx (V=27/28), go-ethereum's `LatestSignerForChainID` falls back to Homestead signing when recovering the sender — the chain-id is not part of the hash. Signature recovery succeeds identically on chain A and chain B, recovering the same sender address regardless of which chain-id the node uses.

---

### Impact Explanation

An attacker who submits a single Homestead-signed transaction to chain A (e.g., a token transfer or contract call) can replay the **identical raw bytes** on chain B. Both chains commit the same state mutation: nonce increment, value transfer, or contract execution. This is a direct chain-id / signer verification bypass enabling unauthorized account mutation and replay across chains.

**Impact:** High — matches "Ethereum transaction, EIP-155/EIP-712/EIP-7702 authorization, nonce, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation."

---

### Likelihood Explanation

`allowUnprotectedTxs` defaults to `false`: [4](#0-3) 

However, it is an on-chain governance parameter that any chain operator or governance vote can enable. Testnets, devnets, and chains that want to support legacy tooling routinely enable it. The attack requires no privileges beyond the ability to submit a transaction to the public RPC endpoint of two such nodes.

---

### Recommendation

Add an explicit chain-id check for **unprotected** transactions as well. Since a Homestead tx carries no chain-id, the node should either:

1. **Reject all unprotected txs unconditionally** (remove `allowUnprotectedTxs` or hard-code it to `false`), or
2. **Bind unprotected txs to the current chain** by refusing to accept them if the node's chain-id differs from a canonical "legacy" chain-id (e.g., only allow on chain-id 1), or
3. At minimum, document clearly that enabling `allowUnprotectedTxs=true` on any chain with chain-id ≠ 1 creates a replay surface and should never be done on a production chain.

The fix in `SendRawTransaction` would be:

```go
if !tx.Protected() {
    // Unprotected txs carry no chain-id; they are replayable across all chains.
    // Only permit them if explicitly allowed AND we are on the canonical legacy chain.
    if !b.UnprotectedAllowed() {
        return common.Hash{}, errors.New("only replay-protected (EIP-155) transactions allowed over RPC")
    }
    // Optionally: reject if chainID != 1
}
```

The same guard must be mirrored in `ValidateEthBasic` in `ante/interfaces/setup.go`.

---

### Proof of Concept

```go
// 1. Create a legacy tx with HomesteadSigner (no chain-id)
key, _ := crypto.GenerateKey()
signer := types.HomesteadSigner{}
tx, _ := types.SignTx(types.NewTransaction(
    0, toAddr, big.NewInt(1e18), 21000, big.NewInt(1e9), nil,
), signer, key)
raw, _ := tx.MarshalBinary()

// 2. Submit to Node A (chain-id=9000, allowUnprotectedTxs=true)
clientA.Call("eth_sendRawTransaction", hexutil.Encode(raw))
// → accepted, committed

// 3. Submit identical bytes to Node B (chain-id=9001, allowUnprotectedTxs=true)
clientB.Call("eth_sendRawTransaction", hexutil.Encode(raw))
// → also accepted, committed — same nonce consumed, same value transferred
```

`tx.Protected()` returns `false` for both submissions, so the chain-id guard at line 155 of `call_tx.go` is never evaluated on either node. [5](#0-4)
