### Title
Excess `msg.value` permanently lost in Wormhole fee payment across all payable bridge entry points — (`File: evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

### Summary
`OmniBridgeWormhole` forwards `msg.value` (or a derived residual) directly to `_wormhole.publishMessage` without verifying it equals exactly `_wormhole.messageFee()`. Because the Wormhole core contract accepts any amount ≥ `messageFee()` and does not refund the surplus, any ETH sent above the required fee is permanently lost in the Wormhole contract.

### Finding Description
Four extension hooks in `OmniBridgeWormhole` call `_wormhole.publishMessage` with an unchecked ETH value:

**`deployTokenExtension` (line 63)** — called from the `payable` `deployToken`:
```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
``` [1](#0-0) 

**`logMetadataExtension` (line 87)** — called from the `payable` `logMetadata` / `logMetadata1155`:
```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
``` [2](#0-1) 

**`finTransferExtension` (line 109)** — called from the `payable` `finTransfer`:
```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, messagePayload, _consistencyLevel);
``` [3](#0-2) 

**`initTransferExtension` (line 143)** — called from the `payable` `initTransfer` with a residual `value`:
```solidity
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [4](#0-3) 

In `initTransfer`, the `value` forwarded to Wormhole is computed as:
- ERC-20 path: `extensionValue = msg.value - nativeFee`
- Native ETH path: `extensionValue = msg.value - amount - nativeFee` [5](#0-4) 

None of these sites check `value == _wormhole.messageFee()`. The Wormhole core contract's `publishMessage` accepts any `msg.value >= messageFee()` and does not return the surplus, so any excess ETH is irrecoverably absorbed.

### Impact Explanation
Any ETH sent above `_wormhole.messageFee()` is permanently lost — it cannot be recovered by the user or the protocol. This affects every public bridge entry point (`initTransfer`, `finTransfer`, `deployToken`, `logMetadata`, `logMetadata1155`). The most impactful path is `initTransfer` for native ETH bridging, where a user who accidentally sends `msg.value = amount + nativeFee + messageFee() + ε` loses `ε` forever with no on-chain revert or warning.

### Likelihood Explanation
Low — requires a user or integration to send more ETH than the exact required amount. This can realistically occur when: (a) a front-end or SDK miscalculates the Wormhole fee, (b) the Wormhole `messageFee()` changes between fee estimation and transaction submission, or (c) a user manually constructs the transaction with a rounded-up value.

### Recommendation
Validate that the ETH forwarded to Wormhole equals exactly `_wormhole.messageFee()` before each `publishMessage` call. For the extension hooks that receive a pre-computed `value`, add:

```diff
+ require(value == _wormhole.messageFee(), "Incorrect Wormhole fee");
  _wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

For the hooks that use raw `msg.value`:

```diff
+ require(msg.value == _wormhole.messageFee(), "Incorrect Wormhole fee");
  _wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
```

### Proof of Concept
1. Wormhole `messageFee()` returns `1e15` (0.001 ETH).
2. User calls `finTransfer(signatureData, payload)` with `msg.value = 2e15` (0.002 ETH) — e.g., due to a front-end rounding up.
3. `finTransferExtension` executes `_wormhole.publishMessage{value: 2e15}(...)`.
4. Wormhole accepts the call (2e15 ≥ 1e15), consumes 1e15 as fee, and silently absorbs the extra 1e15.
5. The transaction succeeds; the user's extra 1e15 wei is permanently lost with no revert or event.

The same path applies to `deployToken`, `logMetadata`, `logMetadata1155`, and `initTransfer`.
