### Title
`PostTxProcessing` Hook Fires Before `RefundGas` State Update, Violating CEI and Enabling Systematic Gas-Refund Mis-accounting — (File: x/evm/keeper/state_transition.go)

---

### Summary

In `ApplyTransaction`, the `PostTxProcessing` external hook callback is invoked **before** `RefundGas` credits the leftover-gas refund back to the sender. Any hook that reads or acts on the sender's EVM-denom balance observes a value that is `leftoverGas × gasPrice` lower than the balance that will actually be committed, causing systematic mis-accounting of user funds on every successful EVM transaction.

---

### Finding Description

`ApplyTransaction` in `x/evm/keeper/state_transition.go` executes in this order:

```
line 194  ApplyMessageWithConfig(tmpCtx, …, commit=true)   // EVM runs; stateDB committed to tmpCtx
line 232  k.PostTxProcessing(tmpCtx, msg, receipt)         // ← external hook fires HERE
line 241  commit()                                          // tmpCtx flushed into ctx
line 252  k.RefundGas(ctx, msg, leftoverGas, evmDenom)     // ← gas refund credited AFTER hook
line 261  k.AddTransientGasUsed(ctx, res.GasUsed)
line 271  k.ResetGasMeterAndConsumeGas(ctx, totalGasUsed)
```

`PostTxProcessing` is the documented Ethermint extension point for chains to react to EVM transactions (see `EvmHooks` interface, `x/evm/types/interfaces.go` line 75-78). It receives `tmpCtx` — a cache context that already contains the EVM execution results — but `RefundGas` has not yet run. `RefundGas` calls `bankKeeper.SendCoinsFromModuleToAccountVirtual` to transfer `leftoverGas × gasPrice` coins from the fee-collector module back to `msg.From` (`x/evm/keeper/gas.go` lines 67-82). That transfer only lands in `ctx` at line 252, **after** the hook has already completed.

The hook therefore reads the sender's EVM-denom balance as:

```
balance_in_hook = initial − fee_paid − value_sent + value_received
```

while the balance that will actually be committed is:

```
balance_final = initial − fee_paid − value_sent + value_received + (leftoverGas × gasPrice)
```

The gap equals the gas refund, which can be up to `gasLimit/2 × gasPrice` (EIP-3529 cap). For a transaction with `gasLimit = 1,000,000`, `gasUsed = 100,000`, and `gasPrice = 1 Gwei`, the hook underestimates the sender's balance by **900,000 Gwei** of the EVM denom.

This is the direct analog of the Timeswap H-07 finding: an external callback fires in the middle of the function while important state updates (the gas refund) still remain.

---

### Impact Explanation

**High — EVM gas-refund mis-accounting of user funds.**

The `PostTxProcessing` hook is the primary Ethermint integration surface used by every chain built on Ethermint (e.g., Evmos) to implement token incentives, staking rewards, bridge callbacks, and balance-gated logic. Because the hook fires before `RefundGas`:

- A hook that **mints reward tokens proportional to the sender's post-tx balance** will systematically under-mint by an amount proportional to the gas refund.
- A hook that **enforces a minimum balance threshold** (e.g., "only process if sender holds ≥ X tokens") will incorrectly reject transactions from senders whose balance only clears the threshold after the refund is applied.
- A hook that **burns or transfers tokens based on the observed balance** will operate on a stale value, permanently mis-accounting user funds on every successful EVM transaction.

The mis-accounting is not a one-off edge case — it occurs on **every successful EVM transaction** where `leftoverGas > 0`, which is the normal case for any transaction that does not consume its entire gas limit.

---

### Likelihood Explanation

**High likelihood of triggering; medium-to-high likelihood of material impact.**

- Every successful EVM transaction with unused gas triggers the ordering bug.
- The `PostTxProcessing` hook is the standard Ethermint extension point; chains built on Ethermint routinely implement hooks that read balances or make balance-dependent decisions.
- An unprivileged user can maximize the refund gap by submitting a transaction with a high `gasLimit` relative to actual `gasUsed` (e.g., a simple ETH transfer with `gasLimit = 1,000,000`).
- No special privileges are required: any account can submit an EVM transaction.

---

### Recommendation

Move `RefundGas` (and `AddTransientGasUsed` / `ResetGasMeterAndConsumeGas`) **before** the `PostTxProcessing` call so the hook always observes the sender's final, post-refund balance:

```go
// 1. Refund gas first — state is now final
leftoverGas := msg.GasLimit - res.GasUsed
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil { … }

totalGasUsed, err := k.AddTransientGasUsed(ctx, res.GasUsed)
…
k.ResetGasMeterAndConsumeGas(ctx, totalGasUsed)

// 2. Then fire the hook on the fully-settled state
if !res.Failed() {
    if err = k.PostTxProcessing(tmpCtx, msg, receipt); err != nil { … }
    …
    commit()
}
```

This follows the Checks-Effects-Interactions pattern: all state effects are applied before any external callback is invoked.

---

### Proof of Concept

**Setup:** A chain built on Ethermint registers a `PostTxProcessing` hook that mints 1 reward token for every 1000 units of EVM-denom the sender holds at the time of the hook call.

**Attack / Trigger (unprivileged):**

1. Sender holds exactly 1,000,000 units of EVM-denom.
2. Sender submits an EVM transaction: `gasLimit = 500,000`, `gasPrice = 2`, `gasUsed = 10,000`.
3. `ApplyMessageWithConfig` runs; EVM deducted `500,000 × 2 = 1,000,000` units as the upfront fee during the ante handler. Post-EVM balance in `tmpCtx` = `0`.
4. `PostTxProcessing` fires. Hook reads balance = `0`. Mints `0` reward tokens.
5. `commit()` flushes `tmpCtx` → `ctx`.
6. `RefundGas` credits `(500,000 − 10,000) × 2 = 980,000` units back to sender. Final balance = `980,000`.

**Expected behavior (if CEI were respected):** Hook should have seen balance = `980,000` and minted `980` reward tokens.

**Actual behavior:** Hook sees balance = `0` and mints `0` reward tokens. The sender permanently loses `980` reward tokens they were entitled to.

```go
func (k *Keeper) RefundGasWithPrice(
	ctx sdk.Context,
	msg *core.Message,
	leftoverGas uint64,
	gasPrice *big.Int,
	denom string,
) error {
	if gasPrice == nil {
		gasPrice = new(big.Int)
	}

	// Return EVM tokens for remaining gas, exchanged at the original rate.
	remaining := new(big.Int).Mul(new(big.Int).SetUint64(leftoverGas), gasPrice)

	switch remaining.Sign() {
	case -1:
		// negative refund errors
		return errorsmod.Wrapf(types.ErrInvalidRefund, "refunded amount value cannot be negative %d", remaining.Int64())
	case 1:
		// positive amount refund
		refundedCoins := sdk.Coins{sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(remaining))}

		// refund to sender from the fee collector module account, which is the escrow account in charge of collecting tx fees
		err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
		if err != nil {
			err = errorsmod.Wrapf(errortypes.ErrInsufficientFunds, "fee collector account failed to refund fees: %s", err.Error())
			return errorsmod.Wrapf(err, "failed to refund %d leftover gas (%s)", leftoverGas, refundedCoins.String())
		}
	default:
		// no refund, consume gas and update the tx gas meter
	}

	return nil
}
```
