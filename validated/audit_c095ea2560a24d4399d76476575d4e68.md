### Title
Pending `init_transfer` for a deployed token becomes permanently irrecoverable after `migrate_deployed_token` removes the old token mapping — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR→EVM bridge transfer for a deployed (EVM-originated) token, the token is burned immediately and a `TransferMessage` is stored in `pending_transfers`. If the DAO subsequently calls `migrate_deployed_token` before `sign_transfer` is processed, the old token's destination-chain address mapping is deleted. Every subsequent call to `sign_transfer` for that pending transfer panics with `FailedToGetTokenAddress`, and no cancellation path exists. The burned tokens are permanently lost.

---

### Finding Description

**Step 1 — User initiates a transfer and tokens are burned.**

`ft_transfer_call` → `init_transfer` → `init_transfer_internal` burns the user's deployed token immediately: [1](#0-0) 

`burn_tokens_if_needed` fires a detached cross-contract call to burn the tokens: [2](#0-1) 

The `TransferMessage` is stored in `pending_transfers` with `token: OmniAddress::Near(old_token_id)`.

**Step 2 — DAO migrates the token, deleting the old mapping.**

`migrate_deployed_token` removes `old_token` from `deployed_tokens` and, critically, removes the entry `token_id_to_address[(origin_chain, old_token)]`: [3](#0-2) 

The new mapping `token_id_to_address[(origin_chain, new_token)] = origin_address` is inserted, but the old key is gone.

**Step 3 — `sign_transfer` panics for the in-flight transfer.**

`sign_transfer` resolves the destination-chain token address by calling `get_token_address(destination_chain, old_token_id)`: [4](#0-3) 

`get_token_address` performs a direct lookup of `token_id_to_address[(destination_chain, old_token_id)]`: [5](#0-4) 

Because `migrate_deployed_token` deleted that key, the lookup returns `None`, and `sign_transfer` panics unconditionally. The pending transfer entry remains in `pending_transfers` forever, and there is no cancel or refund function anywhere in the contract.

---

### Impact Explanation

The impact is **permanent, irrecoverable loss of user funds**:

- The user's deployed tokens are burned at `init_transfer_internal` time (before any migration occurs).
- After migration, `sign_transfer` always panics for the affected `TransferMessage`.
- No cancel, refund, or rescue path exists in the contract.
- The burned tokens cannot be re-minted because the MPC signature is never produced.
- The corresponding EVM-side release never happens.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Token migration is an explicitly supported DAO operation (`migrate_deployed_token` exists precisely for this purpose). A migration can legitimately occur while users have in-flight transfers — there is no on-chain check or grace period that prevents this. Any user who submitted a NEAR→EVM transfer for the old token between the `ft_transfer_call` and the migration call loses their funds. The window can be minutes to hours depending on relayer latency and MPC signing queue depth.

---

### Recommendation

Before executing `migrate_deployed_token`, the contract should verify that no pending transfers reference `old_token`. Alternatively, add a cancellation path that allows the sender to reclaim their tokens when `sign_transfer` cannot be completed (e.g., because the token mapping no longer exists). A minimal fix is to keep the old `token_id_to_address` entry alive (or add a forwarding alias) so that `sign_transfer` can still resolve the destination address for in-flight transfers.

---

### Proof of Concept

```
1. Alice holds 1000 eth.usdc.near (a deployed token, origin: Ethereum USDC).

2. Alice calls ft_transfer_call on eth.usdc.near:
     receiver_id = omni_bridge.near
     amount      = 1000
     msg         = InitTransfer { recipient: Eth(alice_eth_addr), fee: 0, ... }

   → init_transfer_internal:
       burn_tokens_if_needed("eth.usdc.near", 1000)   // tokens burned
       pending_transfers[{Near, nonce=42}] = TransferMessage {
           token: OmniAddress::Near("eth.usdc.near"), ...
       }

3. DAO calls migrate_deployed_token(ChainKind::Eth, "eth.usdc.near", "eth.newusdc.near"):
   → token_id_to_address.remove((Eth, "eth.usdc.near"))   // KEY DELETED
   → token_id_to_address.insert((Eth, "eth.newusdc.near"), eth_usdc_addr)

4. Relayer calls sign_transfer({ origin_chain: Near, origin_nonce: 42 }, ...):
   → get_token_id(OmniAddress::Near("eth.usdc.near")) = "eth.usdc.near"
   → get_token_address(ChainKind::Eth, "eth.usdc.near")
       = token_id_to_address.get((Eth, "eth.usdc.near"))
       = None                                            // DELETED IN STEP 3
   → env::panic_str("ERR_FAILED_TO_GET_TOKEN_ADDRESS")  // PERMANENT PANIC

5. Alice's 1000 eth.usdc.near are burned. No EVM release. No refund. Funds lost.
```
