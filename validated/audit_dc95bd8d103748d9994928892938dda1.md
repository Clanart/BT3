### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt as a new class hash without verifying that the hash corresponds to a previously declared class. In contrast, `execute_declare_transaction` enforces that a class hash is valid before committing it to state. This asymmetry — the same invariant enforced in one path but absent in another — allows any contract to permanently replace its own class hash with an undeclared (non-existent) value, rendering the contract permanently non-executable and freezing all funds it holds.

---

### Finding Description

In `execute_declare_transaction` (`transaction_impls.cairo`, lines 738–743), the OS enforces that a class hash is the genuine result of a Sierra class hash computation before writing it to `contract_class_changes`:

```cairo
let expected_class_hash = finalize_class_hash(
    contract_class_component_hashes=contract_class_component_hashes
);
with_attr error_message("Invalid class hash pre-image.") {
    assert [class_hash_ptr] = expected_class_hash;
}
```

Additionally, `dict_update` with `prev_value=0` enforces that a class may only be declared once, and `assert_not_zero(compiled_class_hash)` ensures the compiled class hash is non-trivial.

In `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), the OS reads the new class hash directly from the syscall request and writes it to `contract_state_changes` with **no validation** that the hash corresponds to any declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `class_hash` value is taken verbatim from `request.class_hash` (line 896) with no check against `contract_class_changes` or any declared class registry. The TODO comment at line 898 explicitly acknowledges this missing check.

Once this state transition is committed, any future attempt by the OS to execute the contract will attempt to look up the compiled class for the invalid hash. Since no such class exists, execution will fail at the hint level in every subsequent block. The contract's state — including all stored funds — is permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash becomes permanently non-executable. All ERC-20 balances, NFTs, or other assets held in that contract's storage are irreversibly frozen. The state commitment (Patricia Merkle Tree root) will reflect the contract's existence with an invalid class hash, but no valid proof can ever execute that contract again. This satisfies the "permanent freezing of funds" critical impact category.

---

### Likelihood Explanation

**Medium.**

The `REPLACE_CLASS_SELECTOR` syscall is available to any Cairo contract at runtime. An attacker who controls a contract (e.g., a malicious DeFi protocol, a rug-pull contract, or a contract with a compromised upgrade key) can call `replace_class` with an arbitrary felt (e.g., `1`) as the new class hash. No privileged role, leaked key, or external dependency is required beyond deploying a contract and calling the syscall. The attack is a single transaction.

---

### Recommendation

Before committing the new class hash in `execute_replace_class`, verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). Specifically, perform a `dict_read` on `contract_class_changes` for the new `class_hash` and assert that the returned compiled class hash is non-zero. This mirrors the invariant already enforced in `execute_declare_transaction`.

---

### Proof of Concept

1. Attacker deploys a contract `VaultContract` that accepts user deposits (ETH/ERC-20 tokens) and exposes an `upgrade(new_class_hash)` function that calls the `replace_class` syscall.
2. Users deposit funds into `VaultContract`.
3. Attacker calls `upgrade(1)` — passing the felt value `1` as the new class hash, which is not a declared class.
4. `execute_replace_class` in `syscall_impls.cairo` (lines 896–910) writes `class_hash=1` to `contract_state_changes` without any validation.
5. The block is proven and the state root is updated. `VaultContract`'s state entry now has `class_hash=1`.
6. In any subsequent block, any transaction targeting `VaultContract` causes the OS to attempt to look up the compiled class for hash `1`. No such class exists; the hint fails and no valid proof can be generated for any execution of `VaultContract`.
7. All user funds in `VaultContract` are permanently frozen.
