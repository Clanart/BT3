### Title
Uninitialized `proposed_split` in Child `ChunkExtra` After Resharding Causes Deterministic Chunk Validation Failure — (File: `chain/chain/src/resharding/manager.rs`)

---

### Summary

During resharding, child `ChunkExtra` objects are created by cloning the parent's `ChunkExtra` without resetting the `proposed_split` field. The parent's final chunk (the resharding boundary block) carries `proposed_split = Some(TrieSplit)`. Each child shard inherits this value. When the first chunk of a child shard is produced in the new epoch, the chunk producer freshly computes `proposed_split = None`. The validation gate in `validate_chunk_with_chunk_extra_and_receipts_root` compares the stored `prev_chunk_extra.proposed_split()` (`Some(TrieSplit)`) against the header's `chunk_header.proposed_split()` (`None`) and returns `InvalidChunkHeaderShardSplit`, stalling chunk production for every child shard after every dynamic resharding event.

---

### Finding Description

In `chain/chain/src/resharding/manager.rs`, the child `ChunkExtra` is constructed by cloning the parent and updating only two fields:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

The `proposed_split` field — added in `ChunkExtraV5` — is never reset:

```rust
pub struct ChunkExtraV5 {
    ...
    pub proposed_split: Option<TrieSplit>,
}
``` [2](#0-1) 

The parent's `ChunkExtra` at the resharding boundary block has `proposed_split = Some(TrieSplit {...})` because the runtime computed and stored the split proposal during chunk application near the epoch end. The child inherits this value verbatim.

The validation path that fires for the first chunk of each child shard is:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [3](#0-2) 

`prev_chunk_extra.proposed_split()` returns the inherited `Some(TrieSplit)`. The chunk producer freshly computes `proposed_split = None` for the new child shard (no split is needed; the resharding cooldown enforced by `min_epochs_between_resharding > 0` also suppresses any new proposal). The mismatch is deterministic and unavoidable.

The codebase's own architecture document explicitly acknowledges this uninitialized field:

> **`chain/chain/src/resharding/manager.rs:249`** — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field). [4](#0-3) 

And the consequence is documented:

> a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`. [5](#0-4) 

---

### Impact Explanation

Every dynamic resharding event produces two child shards whose stored `ChunkExtra.proposed_split` is `Some(TrieSplit)` instead of `None`. The first chunk header produced for each child shard carries `proposed_split = None`. The deterministic mismatch causes `validate_chunk_with_chunk_extra_and_receipts_root` to return `InvalidChunkHeaderShardSplit` for every validator that processes the chunk. No endorsement is issued; the child shard's chunk production is permanently blocked until the node's stored `ChunkExtra` is corrected out-of-band. Because both child shards are affected simultaneously, the entire post-resharding shard layout is stalled.

**Severity: High** — complete loss of liveness for all child shards after every dynamic resharding event.

---

### Likelihood Explanation

The bug fires on every dynamic resharding event where the parent shard's boundary-block chunk carries a non-`None` `proposed_split`. By design, `proposed_split` is set precisely at the boundary block (the block that triggers the split), so the condition is always true when dynamic resharding is active. The `min_epochs_between_resharding > 0` guard prevents back-to-back reshardings but does not prevent the first occurrence. Dynamic resharding is gated behind `ProtocolFeature::DynamicResharding`; once that feature is enabled in production, every resharding event triggers the bug.

**Likelihood: High** (conditional on `DynamicResharding` being enabled).

---

### Recommendation

In `chain/chain/src/resharding/manager.rs`, after cloning the parent `ChunkExtra`, explicitly reset `proposed_split` to `None` for each child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Reset proposed_split: child shards start fresh; inheriting the parent's
// split proposal causes InvalidChunkHeaderShardSplit on the first child chunk.
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

Additionally, resolve the broader `TODO(resharding): set all fields of ChunkExtra` to ensure future `ChunkExtra` versions are also fully initialized for child shards.

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` and configure `DynamicReshardingConfig` with `force_split_shards` containing a target shard.
2. Run the chain until the forced resharding boundary block is finalized. At this block, the parent shard's `ChunkExtra` has `proposed_split = Some(TrieSplit { boundary_account, ... })`.
3. Observe that `chain/chain/src/resharding/manager.rs` creates child `ChunkExtra` objects by cloning the parent without resetting `proposed_split`.
4. In the new epoch (N+2), the chunk producer for each child shard calls `compute_proposed_split`, which returns `None` (resharding cooldown active, fresh shard).
5. The produced chunk header carries `proposed_split = None`.
6. Every validator calls `validate_chunk_with_chunk_extra_and_receipts_root`, which compares `prev_chunk_extra.proposed_split() = Some(TrieSplit {...})` against `chunk_header.proposed_split() = None` and returns `Err(Error::InvalidChunkHeaderShardSplit(...))`.
7. No endorsements are issued; both child shards are permanently stalled.

The `DYNAMIC_RESHARDING_VALIDATION_FAILURES` Prometheus counter with label `chunk_header` will increment on every validator for every block, confirming the failure. [6](#0-5) [3](#0-2) [7](#0-6)
