### Title
Missing Duplicate-Key/Hash Validation in `batch_insert` Bypasses Invariants Enforced by `insert` — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces, allowing attacker-controlled input to silently insert multiple leaf nodes sharing the same `KeyId` or `Hash`. The resulting tree is structurally corrupt: its Merkle root is wrong, and any inclusion proof derived from it is invalid.

---

### Finding Description

`insert` (lines 362–413) enforces two invariants before writing a leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` (lines 570–657) calls `insert` for **at most the last two items** in the input vector, and only when `leaf_count <= 1`. All remaining items are written directly via `insert_entry_to_blob` with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When `leaf_count > 1`, the guarded first loop is skipped entirely, so **every** item in the batch bypasses the duplicate checks: [3](#0-2) 

**Concrete bypass scenario (3-item batch on an empty tree):**

1. `batch_insert([(key1,v1,h1), (key2,v2,h2), (key1,v3,h3)])` is called.
2. `leaf_count == 0 ≤ 1` → first loop pops `(key1,v3,h3)` then `(key2,v2,h2)` and inserts them via `insert()` — both succeed.
3. The remaining vector `[(key1,v1,h1)]` is processed by the unchecked loop — `key1` is already present but no error is raised; a second leaf with `key1` is written into the blob.

The `BlockStatusCache.key_to_index` map now maps `key1` to only the last-written index, while the blob contains two leaf nodes with `key1`. The tree is structurally inconsistent. [4](#0-3) 

---

### Impact Explanation

A corrupted blob produces an incorrect Merkle root. Any `ProofOfInclusion` generated from the corrupted tree via `get_proof_of_inclusion` will not correspond to the actual committed key-value state. Verifiers that check `proof.valid()` and compare `proof.root_hash()` against the on-chain root will either accept a proof for a non-existent key-value pair or reject a proof for a legitimately stored one — both constitute forged or corrupted DataLayer state. [5](#0-4) 

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is a `pub` method on `MerkleBlob` and is the primary bulk-load path for DataLayer tree construction. DataLayer store owners supply the key-value pairs that populate the tree; if the DataLayer ingestion layer does not independently deduplicate keys before calling `batch_insert`, a store owner can trigger this path with a crafted batch. The `insert` path correctly rejects duplicates, so the gap is specific to the batch code path.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards inside the bulk loop of `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, refactor `batch_insert` to route every item through `insert` (accepting the performance trade-off) or add a pre-pass that deduplicates and validates the input vector before any blob writes.

---

### Proof of Concept

```rust
use chia_datalayer::{KeyId, ValueId, Hash, MerkleBlob};

let mut blob = MerkleBlob::new(Vec::new()).unwrap();

let key   = KeyId(1);
let hash1 = Hash(/* some 32-byte value */);
let hash2 = Hash(/* different 32-byte value */);

// Three-item batch: first item duplicates the key of the third item.
// The third item is popped first (via `insert`, which succeeds),
// then the second item is popped (via `insert`, which succeeds),
// then the first item is written via the unchecked loop — no error.
blob.batch_insert(vec![
    ((key, ValueId(1)), hash1),          // duplicate key — bypasses check
    ((KeyId(2), ValueId(2)), hash2),
    ((key, ValueId(3)), Hash(/* ... */)),
]).unwrap();  // succeeds — tree is now corrupt

// blob now contains two leaf nodes with KeyId(1).
// The Merkle root is incorrect; any proof derived from it is invalid.
``` [6](#0-5)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
