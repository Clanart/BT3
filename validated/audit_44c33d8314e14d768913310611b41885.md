### Title
Reentrancy via ERC777 `tokensToSend` Hook Causes Nonce Collision and Permanent Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
In `OmniBridge.initTransfer`, `currentOriginNonce` is incremented but never captured into a local variable before an external `safeTransferFrom` call. If the token is ERC777-compatible, the `tokensToSend` hook fires on the sender before the transfer completes, giving the attacker a re-entry window. A re-entrant inner call increments `currentOriginNonce` a second time; when the outer call resumes, it reads the already-mutated counter and publishes a Wormhole message and emits an event with the same nonce as the inner call. This nonce collision causes the outer call's tokens to be permanently locked in the bridge with no recovery path.

### Finding Description
`OmniBridge.initTransfer` (lines 373–437) follows this sequence:

1. `currentOriginNonce += 1;` — counter is incremented (e.g., to N+1) but **not stored in a local variable**.
2. `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);` — external call that, for ERC777 tokens, triggers the sender's registered `tokensToSend` hook **before** the transfer settles.
3. `initTransferExtension(msg.sender, tokenAddress, currentOriginNonce, ...)` — reads `currentOriginNonce` directly from storage.
4. `emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, currentOriginNonce, ...)` — also reads `currentOriginNonce` directly from storage.

Because steps 3 and 4 read `currentOriginNonce` from storage rather than from a local snapshot, any mutation of that variable between steps 1 and 3 is silently picked up.

An attacker whose ERC777 token fires `tokensToSend` can re-enter `initTransfer` during step 2:

- **Outer call** increments nonce to N+1, then yields control to the hook.
- **Inner (re-entrant) call** increments nonce to N+2, completes fully, publishes Wormhole message with nonce N+2, emits event with nonce N+2.
- **Outer call** resumes; `currentOriginNonce` is now N+2. It publishes a second Wormhole message with nonce N+2 and emits a second event with nonce N+2.

Nonce N+1 is never published. Nonce N+2 appears in two independent Wormhole messages carrying different amounts. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
On the NEAR side, `fin_transfer_callback` uses `finalised_transfers` (a set keyed by `TransferId`) to prevent replay. The first Wormhole message carrying nonce N+2 (the inner call's) is finalized; the second (the outer call's) is rejected as a duplicate. The outer call's tokens — already transferred into the bridge contract — have no corresponding finalized transfer on NEAR and no on-chain recovery mechanism. They are permanently locked. This satisfies the **Critical** impact criterion: *irrecoverable lock of user funds in bridge flows*. [4](#0-3) 

### Likelihood Explanation
The attack requires:
1. An ERC777 token (or any token with a pre-transfer callback to the sender) that is accepted by the bridge.
2. The attacker to control the `tokensToSend` implementer registered for their address in the ERC1820 registry.

ERC777 is a published standard; bridges routinely accept arbitrary ERC20-compatible tokens. No privileged role is required. The bridge has no reentrancy guard. Likelihood is **medium**: it depends on an ERC777 token being bridged, which is a realistic scenario.

### Recommendation
Capture `currentOriginNonce` in a local variable immediately after incrementing it, and use only that local variable in all subsequent reads within the same function call:

```solidity
currentOriginNonce += 1;
uint64 originNonce = currentOriginNonce;   // snapshot before any external call
...
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
...
initTransferExtension(msg.sender, tokenAddress, originNonce, ...);
emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
```

Additionally, consider adding OpenZeppelin's `ReentrancyGuard` to `OmniBridge` as a defense-in-depth measure.

### Proof of Concept

**Setup:** An ERC777 token `T` is registered with the bridge. The attacker deploys a contract `A` that implements `tokensToSend` and is registered as the ERC1820 implementer for itself.

**Initial state:** `currentOriginNonce = N`.

1. `A` calls `OmniBridge.initTransfer(T, 1000, ...)`.
2. Bridge executes `currentOriginNonce = N+1`.
3. Bridge calls `T.safeTransferFrom(A, bridge, 1000)`.
4. `T` fires `A.tokensToSend(...)` before completing the transfer.
5. Inside `tokensToSend`, `A` calls `OmniBridge.initTransfer(T, 1, ...)` (inner call).
6. Bridge executes `currentOriginNonce = N+2`.
7. Inner call completes: 1 token locked, Wormhole message published with `originNonce = N+2`, event emitted with nonce N+2.
8. Control returns to outer call; `T.safeTransferFrom` completes; 1000 tokens are now locked.
9. Outer call executes `initTransferExtension(..., currentOriginNonce=N+2, ...)` — **wrong nonce**.
10. Outer call emits `InitTransfer(..., currentOriginNonce=N+2, ...)` — **wrong nonce**.

**Result on NEAR:**
- Inner call's Wormhole message (nonce N+2, amount 1) is finalized first; 1 token released.
- Outer call's Wormhole message (nonce N+2, amount 1000) is rejected: `finalised_transfers` already contains nonce N+2.
- 1000 tokens are permanently locked in the EVM bridge with no recovery path.
- Nonce N+1 is permanently skipped. [5](#0-4) [6](#0-5)
