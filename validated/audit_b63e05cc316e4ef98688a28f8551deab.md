### Title
`chain_gas_consumed` Silently Drops Actual L2 Gas from Receipt Commitment, Producing Wrong Block Hash for v0.14.2+ Blocks - (`crates/starknet_api/src/block_hash/receipt_commitment.rs`)

---

### Summary

`chain_gas_consumed` accepts a `GasVector` that carries a non-zero `l2_gas` field for every Starknet v0.14.2+ transaction, but unconditionally chains `Felt::ZERO` instead of `gas_consumed.l2_gas`. The result is a structurally wrong `ReceiptCommitment` that propagates into the block hash. This is the direct sequencer analog of the external report's pattern: an outer interface that accepts a value (ETH / L2 gas) while the inner implementation silently ignores it.

---

### Finding Description

`calculate_receipt_hash` calls `chain_gas_consumed` to commit the three gas dimensions of a transaction receipt:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)                    // ← always zero, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
``` [1](#0-0) 

The comment reads *"L2 gas consumed (In the current RPC: always 0)"*, but this was written before Starknet v0.14.2 introduced L2 gas billing. The production block fixture `block_post_0_14_2.json` shows every transaction carrying a non-zero `total_gas_consumed.l2_gas` (e.g. `76014735`): [2](#0-1) 

The `GasVector` type that is passed in has a fully populated `l2_gas` field:

```rust
// crates/starknet_api/src/execution_resources.rs  lines 106-111
pub struct GasVector {
    pub l1_gas: GasAmount,
    pub l1_data_gas: GasAmount,
    pub l2_gas: GasAmount,   // ← populated by blockifier, silently dropped above
}
``` [3](#0-2) 

The `apollo_starknet_client` reader populates `gas_consumed` directly from `total_gas_consumed`, which includes L2 gas: [4](#0-3) 

`calculate_receipt_commitment` is called from `calculate_block_commitments`, whose result feeds directly into `BlockHeaderCommitments.receipt_commitment` and then into `calculate_block_hash`: [5](#0-4) [6](#0-5) 

`BlockExecutionArtifacts::new` calls `calculate_block_commitments` and stores the result in `partial_block_hash_components`, which is later committed to storage and broadcast to consensus peers: [7](#0-6) 

---

### Impact Explanation

Every block produced on Starknet v0.14.2+ contains transactions with non-zero L2 gas. For each such block the sequencer computes a `receipt_commitment` that is wrong — it commits to `l2_gas = 0` for every transaction regardless of actual consumption. This wrong commitment is embedded in the block hash returned by `calculate_block_hash`. Any external verifier (SNOS prover, L1 verifier, sync peer) that independently computes the receipt commitment using the actual L2 gas values will derive a different block hash, breaking proof verification and cross-node agreement.

This matches the allowed impact: **Wrong receipt, or revert result from blockifier/syscall/execution logic for accepted input** (Critical), and **RPC execution or simulation returns an authoritative-looking wrong value** (High).

---

### Likelihood Explanation

The trigger condition is any block on Starknet v0.14.2 or later that contains at least one transaction consuming L2 gas — which is every normal block since v0.14.2 launched. No special attacker action is required; the bug fires unconditionally on normal transaction traffic.

---

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual `l2_gas` field:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // ← use actual L2 gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

Verify that the SNOS receipt-commitment Cairo code uses the same field ordering and that the regression test in `receipt_commitment_test.rs` is updated with a non-zero `l2_gas` fixture.

---

### Proof of Concept

1. Take any transaction from `block_post_0_14_2.json` with `total_gas_consumed.l2_gas = 76014735`.
2. Construct a `ReceiptElement` with `gas_consumed = GasVector { l1_gas: 0, l1_data_gas: 128, l2_gas: 76014735 }`.
3. Call `calculate_receipt_hash` on it — the hash chains `Felt::ZERO` for L2 gas.
4. Manually compute the hash with `76014735` in the L2 gas slot — the result differs.
5. The sequencer's `receipt_commitment` stored in `BlockHeaderCommitments` and used in `calculate_block_hash` therefore diverges from any independently computed value, producing a wrong block hash for every v0.14.2+ block.
