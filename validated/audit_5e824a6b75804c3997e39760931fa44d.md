### Title
Unconditional `assert is_reverted = 0` in `deploy_contract` Causes OS Proof Failure on Constructor Revert - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deploy_contract.cairo`)

---

### Summary

`deploy_contract.cairo` contains a hard `assert is_reverted = 0` after invoking the constructor. This assertion is shared by both the deprecated and new (Cairo 1) deploy syscall paths. If a constructor reverts — whether explicitly or due to out-of-gas — the OS Cairo program panics and cannot generate a valid STARK proof for the block, causing a total network halt.

---

### Finding Description

`deploy_contract` is the shared function used by all deploy paths in the OS:

- `execute_deploy_syscall` (deprecated path, `deprecated_execute_syscalls.cairo` line 299)
- `execute_deploy` (new Cairo 1 path, `syscall_impls.cairo` line 527)
- `execute_deploy_account_transaction` (`transaction_impls.cairo` line 643)

After invoking the constructor via `select_execute_entry_point_func`, the function unconditionally asserts:

```cairo
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

The comment explicitly names only the deprecated path, but the same function body — and therefore the same assertion — is reached from the new `execute_deploy` path in `syscall_impls.cairo`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [2](#0-1) 

`select_execute_entry_point_func` legitimately returns `is_reverted = 1` for Cairo 1 constructors that set `failure_flag` in their return values (out-of-gas, explicit panic, etc.): [3](#0-2) 

When `is_reverted = 1` is returned, the `assert is_reverted = 0` in `deploy_contract` fails as a Cairo constraint, making the OS unable to produce a valid proof for any block that contains such a transaction.

The same assertion is also reached from `execute_deploy_account_transaction`, where the constructor runs under a capped gas budget (`VALIDATE_MAX_SIERRA_GAS`), making out-of-gas reverts trivially achievable: [4](#0-3) 

---

### Impact Explanation

A failed Cairo assertion inside the OS program means the prover cannot generate a valid STARK proof for the block. The sequencer includes failed transactions in blocks (they still pay fees), so a block containing a deploy transaction with a reverting constructor cannot be proven. This constitutes a **total network shutdown**: the network cannot confirm any new transactions until the OS is patched and redeployed.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

Any unprivileged user can:
1. Declare a Cairo 1 contract class whose constructor explicitly panics (or is designed to run out of gas under `VALIDATE_MAX_SIERRA_GAS`).
2. Submit a `DeployAccount` transaction or an `Invoke` transaction that calls the `deploy` syscall targeting that class.
3. The sequencer includes the transaction in a block as a failed transaction.
4. The OS hits `assert is_reverted = 0` and cannot prove the block.

No privileged access, leaked keys, or external dependencies are required. The attack is cheap and deterministic.

---

### Recommendation

`deploy_contract` must handle the `is_reverted = 1` case instead of asserting it away. When the constructor reverts:

1. Do **not** assert `is_reverted = 0`; instead branch on it.
2. Revert the state changes already written to `contract_state_changes` (the class hash update at line 62–66) using the revert log entries already appended.
3. Return the revert status to the caller (`execute_deploy`, `execute_deploy_syscall`, `execute_deploy_account_transaction`) so each can write the correct `failure_flag` in the response header.

The TODO comment at line 28 (`// TODO(Yoni, 1/1/2027): handle failures.`) and the TODO at `syscall_impls.cairo` line 538 (`// TODO(Yoni, 1/1/2026): support failures.`) confirm this is a known gap — but the current code is not safe to ship as-is because the assertion makes the OS unprovable on any constructor revert. [5](#0-4) 

---

### Proof of Concept

1. **Declare** a Cairo 1 contract whose constructor body is:
   ```rust
   #[constructor]
   fn constructor(ref self: ContractState) {
       panic!("revert");
   }
   ```
2. **Submit** a `DeployAccount` transaction (or an `Invoke` that calls `deploy` syscall) targeting this class hash.
3. The sequencer simulates the transaction, observes the constructor revert, and includes it in a block as a failed transaction (fees are charged).
4. The OS processes the block, reaches `deploy_contract`, calls `select_execute_entry_point_func`, receives `is_reverted = 1`.
5. `assert is_reverted = 0` at `deploy_contract.cairo:91` fails as a Cairo constraint.
6. The prover cannot generate a valid STARK proof for the block → **network halt**. [6](#0-5)
