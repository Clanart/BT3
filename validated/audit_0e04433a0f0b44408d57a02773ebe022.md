### Title
Recipient contract callback in `finTransfer()` can permanently freeze bridged ERC-1155 tokens or native ETH — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` makes two external calls that invoke attacker/recipient-controlled code before the transfer is finalized. For the ERC-1155 path, `IERC1155.safeTransferFrom` triggers `onERC1155Received` on the recipient contract; for the native-ETH path, a low-level `.call` is made to the recipient. If either call reverts, the entire `finTransfer()` transaction reverts. Because there is no fallback delivery path and no admin rescue mechanism, the corresponding source-chain funds (already locked or burned) are permanently irrecoverable.

---

### Finding Description

In `OmniBridge.finTransfer()`, after signature verification and nonce marking, two delivery branches make external calls directly to `payload.recipient`:

**ERC-1155 branch** (lines 323–330):
```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,   // attacker-controlled contract
    multiToken.tokenId,
    payload.amount,
    ""
);
```
`safeTransferFrom` mandates that if `payload.recipient` is a contract, it must implement `IERC1155Receiver.onERC1155Received`. A recipient contract that:
- does not implement the interface, or
- implements it but always reverts, or
- implements it but consumes all remaining gas

will cause `finTransfer()` to revert on every call.

**Native-ETH branch** (lines 317–322):
```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
```
A recipient contract without a `receive()` function, or one whose `receive()` reverts, causes `success == false`, triggering `revert FailedToSendEther()`.

In both cases the `completedTransfers[payload.destinationNonce] = true` write at line 287 is also reverted (it is part of the same transaction), so the nonce is never consumed. The `finTransfer()` call can be retried indefinitely, but it will always revert for the same `payload.recipient`. There is no admin function to redirect the delivery to a different address or to rescue the locked tokens/ETH from the bridge contract.

The `payload.recipient` is an EVM address embedded in the MPC-signed cross-chain message. It was originally specified by the user who called `initTransfer` on the source chain. Once the source-chain transfer is committed (tokens burned or locked), the recipient address is immutable in the signed payload.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

- For ERC-1155 tokens: the tokens are held by the bridge contract (`address(this)`) and can never be delivered. No withdrawal or rescue path exists.
- For native ETH: the ETH is held by the bridge contract and can never be delivered. No withdrawal or rescue path exists.

The source-chain assets (burned bridge tokens, or locked native tokens) are already consumed. The destination-chain delivery is permanently blocked. Funds are irrecoverably lost.

---

### Likelihood Explanation

**Medium-to-High.**

The scenario requires only that `payload.recipient` is a contract that cannot accept ERC-1155 tokens or ETH. This is a common real-world condition:

- Many smart contract wallets, multisigs, DAOs, and protocol contracts do not implement `IERC1155Receiver`.
- Many contracts lack a `receive()` function or have one that reverts under certain conditions.
- A user bridging to a contract address they control (e.g., a multisig) may not realize the contract lacks ERC-1155 support.
- No privileged access is required; any user who initiates a cross-chain transfer to such an address triggers the freeze.

---

### Recommendation

Apply the same fix pattern recommended in the referenced report: never deliver directly to the recipient in a way that invokes recipient-controlled code during the finalization transaction. Instead, use a pull-payment (withdraw) pattern:

1. For ERC-1155: transfer to `address(this)` (the bridge) and record `withdrawable[collection][tokenId] = recipient`. Add a `withdrawERC1155()` function the recipient can call separately.
2. For native ETH: record `pendingEth[recipient] += amount`. Add a `withdrawEth()` function.

This decouples delivery failure from finalization, ensuring the nonce is always consumed and funds are always claimable.

```diff
// ERC-1155 path
- IERC1155(multiToken.tokenAddress).safeTransferFrom(
-     address(this),
-     payload.recipient,
-     multiToken.tokenId,
-     payload.amount,
-     ""
- );
+ // tokens already held by address(this); record claimable balance
+ withdrawableERC1155[multiToken.tokenAddress][multiToken.tokenId][payload.recipient] += payload.amount;

// ETH path
- (bool success, ) = payload.recipient.call{value: payload.amount}("");
- if (!success) revert FailedToSendEther();
+ pendingEth[payload.recipient] += payload.amount;
```

### Proof of Concept

1. Alice holds ERC-1155 tokens on NEAR and initiates a cross-chain transfer to her EVM multisig contract (which does not implement `IERC1155Receiver`).
2. The NEAR side locks/burns Alice's tokens and the MPC produces a signed `TransferMessagePayload` with `payload.recipient = <multisig address>`.
3. A relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
4. Execution reaches line 324: `IERC1155(...).safeTransferFrom(address(this), payload.recipient, ...)`.
5. The ERC-1155 contract calls `payload.recipient.onERC1155Received(...)`. The multisig has no such function → revert.
6. The entire `finTransfer()` transaction reverts. `completedTransfers[nonce]` is also reverted.
7. Any subsequent call to `finTransfer()` with the same payload repeats steps 3–6 indefinitely.
8. Alice's tokens are permanently locked in the bridge with no recovery path.
