### Title
`canTransfer` Ante-Handler Check Uses Total Bank Balance Instead of Spendable Balance, Admitting Guaranteed-Fail Transactions - (`ante/eth.go`, `x/evm/keeper/keeper.go`)

### Summary

`CheckEthCanTransfer` (and `VerifyEthAccount`) in the ante handler use `bankKeeper.GetBalance` (total balance, including locked Cosmos vesting tokens) to decide whether a sender can cover the EVM transaction value. However, the actual EVM-level transfer path — `statedb.Transfer` → `keeper.Transfer` → `bankKeeper.SendCoins` and `statedb.SubBalance` → `keeper.SubBalance` → `bankKeeper.SendCoinsFromAccountToModule` — enforces spendability. The result is a systematic discrepancy: ante-handler admission checks pass for accounts whose EVM-denom tokens are locked in a Cosmos vesting schedule, while the execution-time transfer always fails. The transaction is included in the block, the nonce is permanently incremented, and the user's gas fee is consumed — all for a transfer that was guaranteed to fail.

### Finding Description

**Root cause — `GetBalance` reads total, not spendable, bank balance:**

`x/evm/keeper/keeper.go:297-299`:
```go
func (k *Keeper) GetBalance(ctx sdk.Context, addr sdk.AccAddress, denom string) uint256.Int {
    balance := k.bankKeeper.GetBalance(ctx, addr, denom).Amount.BigInt()
    return *uint256.MustFromBig(balance)
}
```
`bankKeeper.GetBalance` returns the **total** coin amount for the address, including tokens that are locked under a Cosmos vesting schedule. It does **not** call `bankKeeper.SpendableCoins` / `bankKeeper.SpendableCoin`.

**Ante-handler balance checks use this inflated figure:**

`ante/eth.go:256-259` (`canTransfer`):
```go
func canTransfer(ctx sdk.Context, evmKeeper interfaces.EVMKeeper, denom string, from common.Address, amount *big.Int) bool {
    balance := evmKeeper.GetBalance(ctx, sdk.AccAddress(from.Bytes()), denom)
    return balance.ToBig().Cmp(amount) >= 0
}
```
Called from `CheckEthCanTransfer` (`ante/eth.go:242`):
```go
if value.Sign() > 0 && !canTransfer(ctx, evmKeeper, evmParams.EvmDenom, from, value) {
    return errorsmod.Wrapf(errortypes.ErrInsufficientFunds, ...)
}
```
`VerifyEthAccount` (`ante/eth.go:101-104`) also uses `evmKeeper.GetBalance` for the sender-balance check.

**Execution-time paths enforce spendability:**

`x/evm/keeper/statedb.go:92-101` (`SubBalance`):
```go
if err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, coins); err != nil {
    return uint256.Int{}, err
}
```
`x/evm/keeper/statedb.go:76-78` (`Transfer`):
```go
func (k *Keeper) Transfer(ctx sdk.Context, sender, recipient sdk.AccAddress, coins sdk.Coins) error {
    return k.bankKeeper.SendCoins(ctx, sender, recipient, coins)
}
```
Both `SendCoinsFromAccountToModule` and `SendCoins` in the Cosmos SDK bank module internally call `SpendableCoins` and reject locked vesting tokens.

**Error propagation — `s.err` aborts `Commit()`:**

`x/evm/statedb/statedb.go:447-449`:
```go
if err := s.ExecuteNativeAction(..., func(ctx sdk.Context) error {
    return s.keeper.Transfer(ctx, senderAddr, recipientAddr, coins)
}); err != nil {
    s.err = err
}
```
`x/evm/statedb/statedb.go:753-756`:
```go
if s.err != nil {
    return s.err
}
```
A non-`ErrStateConflict` error from `Commit()` propagates as a cosmos-level error from `ApplyMessageWithConfig`, meaning the transaction fails after the ante handler has already incremented the nonce and deducted gas fees.

### Impact Explanation

An unprivileged user holding a Cosmos vesting account whose EVM-denom tokens are still locked can craft an EVM transaction with a `value` field that exceeds their spendable balance but not their total balance. The ante handler's `canTransfer` and `VerifyEthAccount` checks pass (total balance ≥ value), the transaction is included in the block, the sender's nonce is permanently incremented, and the gas fee is consumed. The EVM execution then fails because `bankKeeper.SendCoins` rejects the locked tokens. The user's gas fees are mis-accounted (paid for a transaction that was structurally guaranteed to fail), and the nonce is advanced without a successful transfer — a state the user cannot recover from without submitting a replacement transaction at the next nonce.

This matches the allowed High impact: **"ante handler bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."**

### Likelihood Explanation

Any chain deploying Ethermint that also uses Cosmos vesting accounts (a standard Cosmos SDK feature used for team/investor token lockups) is affected. A user with a vesting account whose EVM-denom tokens are partially or fully locked can trigger this with a standard `eth_sendRawTransaction` call — no special privileges required. The condition (locked EVM-denom vesting tokens + enough spendable tokens to cover gas) is realistic in any chain that distributes EVM-denom tokens via vesting schedules.

### Recommendation

Replace `bankKeeper.GetBalance` with `bankKeeper.SpendableCoin` (or `SpendableCoins`) in `Keeper.GetBalance` (or introduce a separate `GetSpendableBalance` method used by the ante handler and `canTransfer`):

```go
// x/evm/keeper/keeper.go
func (k *Keeper) GetBalance(ctx sdk.Context, addr sdk.AccAddress, denom string) uint256.Int {
    // Use SpendableCoin so locked vesting tokens are excluded
    balance := k.bankKeeper.SpendableCoin(ctx, addr, denom).Amount.BigInt()
    return *uint256.MustFromBig(balance)
}
```

This aligns the ante-handler admission check with what the bank module will actually permit at execution time, preventing guaranteed-fail transactions from being admitted to the block.

### Proof of Concept

1. Alice holds a Cosmos vesting account. Her EVM-denom balance: 100 tokens total, 0 currently spendable (all locked), plus 10 spendable tokens from a separate source for gas.
2. Alice submits an EVM transaction: `value = 50`, `gasLimit * gasPrice = 5`.
3. **Ante handler — `VerifyEthAccount`**: `GetBalance` returns 110 (total). `CheckSenderBalance(110, tx)` passes.
4. **Ante handler — `CheckEthGasConsume`**: `DeductTxCostsFromUserBalance` deducts 5 spendable tokens. Passes.
5. **Ante handler — `CheckEthCanTransfer`**: `canTransfer` calls `GetBalance` → 105 (total). `105 >= 50` → passes. Nonce incremented.
6. **EVM execution — `Transfer`**: `bankKeeper.SendCoins(alice, recipient, 50)` → fails: "50 tokens are locked".
7. `s.err` is set. `stateDB.Commit()` returns the error. `ApplyMessageWithConfig` returns a cosmos-level error.
8. **Result**: Alice's nonce is permanently incremented, 5 tokens of gas fees are consumed, and no transfer occurred. The transaction was admitted to the block despite being structurally impossible to execute.

	coins := sdk.NewCoins(coin)
	prevBalance := k.GetBalance(ctx, addr, coin.Denom)
	if err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	if err := k.bankKeeper.BurnCoins(ctx, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err
	}
	return prevBalance, nil
