### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking and Fund Freeze — (`execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared (and compiled) contract class before committing the replacement to state. Any contract can therefore replace its own class hash with an arbitrary, undeclared value. Once that happens, every future call to the contract will fail irrecoverably inside the OS prover, permanently freezing any funds the contract holds.

---

### Finding Description

In `execute_replace_class` the OS reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any existence check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developer-inserted TODO at line 898 explicitly acknowledges the missing check. [2](#0-1) 

When any future transaction calls the now-replaced contract, `execute_entry_point` performs the following two-step lookup:

**Step 1** — read the compiled class hash from `contract_class_changes` using the (undeclared) class hash as key:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [3](#0-2) 

Because the class was never declared, the key is absent from `contract_class_changes` and `dict_read` returns `0`.

**Step 2** — look up the compiled class fact by that hash:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [4](#0-3) 

`find_element` is a Cairo standard-library function that **panics** (hard assertion failure) when the key is not present. Because `compiled_class_hash` is `0` and no compiled class fact with hash `0` exists in the bundle, every subsequent call to the bricked contract causes the OS to panic. The sequencer's simulation detects this and permanently excludes all such transactions, leaving the contract — and every token or asset it holds — inaccessible forever.

The `replace_class` syscall is dispatched unconditionally to any executing contract:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
``` [5](#0-4) 

No privilege or role is required.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, the contract is permanently non-executable. Any ERC-20 balances, NFTs, or other assets stored in that contract's storage become permanently inaccessible. There is no recovery path: the class hash in state cannot be corrected without a new `replace_class` call, which itself requires executing the (now-broken) contract.

---

### Likelihood Explanation

The attack surface is broad:

- **Any contract** can issue the `replace_class` syscall — no admin key or privileged role is required.
- A malicious contract author can deploy a contract that accepts user deposits and then calls `replace_class` with an arbitrary undeclared hash to rug-pull depositors.
- A buggy upgrade script that passes the wrong hash (e.g., a Sierra class hash instead of the correct compiled-class hash) triggers the same outcome accidentally.
- The TODO comment in the production code confirms the developers are aware the check is absent, meaning the window of exposure is open until the fix is shipped.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). A `dict_read` returning a non-zero compiled class hash is sufficient proof of declaration. If the hash is absent, write a failure response and return without modifying state — mirroring the pattern already used for out-of-gas conditions in the same function.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts ETH/token deposits from users.
2. After accumulating funds, the attacker calls any entry point that internally issues `replace_class(0xdeadbeef…)` where `0xdeadbeef…` is a felt that has never been declared on-chain.
3. The OS executes `execute_replace_class`: gas is deducted, `contract_state_changes[MaliciousVault] = StateEntry(class_hash=0xdeadbeef…, …)` is written, the transaction succeeds and is included in a block.
4. Any user who now tries to withdraw calls `MaliciousVault.__execute__`. The sequencer simulates the call: `execute_entry_point` reads `class_hash = 0xdeadbeef…` from state, does `dict_read(contract_class_changes, 0xdeadbeef…) → 0`, then `find_element(compiled_class_facts, 0)` panics. Simulation fails; the transaction is never included.
5. All deposited funds are permanently frozen with no recovery path.
