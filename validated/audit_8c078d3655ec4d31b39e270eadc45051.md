Audit Report

## Title
Stranded ETH in Router Consumed by Subsequent WETH Swap Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` — the router's entire native ETH balance — when deciding how much ETH to wrap into WETH. Because the router's payable entry points accept arbitrary `msg.value` and `pay()` wraps only `value` (not the full balance), any surplus ETH sent by one caller remains in the router and is silently consumed to fund a subsequent caller's WETH payment, with no `transferFrom` charged to that later caller.

## Finding Description

`pay()` at [1](#0-0)  reads `address(this).balance` as `nativeBalance` and uses it to fund a WETH wrap without verifying that the ETH was deposited by the current `payer`. The three branches are:

- `nativeBalance >= value` → wrap `value` from router balance, no `transferFrom` from payer
- `nativeBalance > 0` → wrap partial from router, pull remainder from payer
- else → pull all from payer via `transferFrom`

`exactInputSingle` is `payable` [2](#0-1)  and calls `pay()` inside the pool's swap callback via `_justPayCallback` [3](#0-2) , which wraps only `value` (the owed amount), not the full `msg.value`. If Alice sends `1 ETH` for a `0.9 ETH` swap, `pay()` wraps `0.9 ETH` and `0.1 ETH` remains in the router after the call returns.

The `receive()` guard [4](#0-3)  only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable functions. The stranded ETH is then available to any subsequent caller whose `pay(WETH, ...)` call finds `nativeBalance >= value`.

## Impact Explanation

Direct loss of user principal. Alice's surplus ETH is permanently transferred to fund Bob's swap — Bob receives WETH-backed pool output without any `transferFrom` deducted from his account. This meets the "direct loss of user principal" threshold at Critical/High severity. The attack is permissionless: Bob requires no approvals, no special role, and no coordination with Alice.

## Likelihood Explanation

The precondition — ETH stranded in the router — is realistic and common:

1. Users calling `exactInputSingle{value: X}` with `X > amountIn` and omitting a trailing `refundETH()`.
2. `multicall` batches [5](#0-4)  that include a WETH swap but omit `refundETH()` as the final call.

The router provides no on-chain enforcement that `refundETH()` is called. A griefing bot can monitor the mempool for transactions that leave ETH in the router and front-run the victim's `refundETH()` with a zero-value WETH swap.

## Recommendation

Replace `address(this).balance` in `pay()` with a caller-scoped `msgValue` parameter that is snapshotted at the payable entry point and threaded through to the callback:

```solidity
// In pay(), replace address(this).balance with a caller-scoped msgValue parameter
function pay(address token, address payer, address recipient, uint256 value, uint256 msgValue) internal {
    ...
    } else if (token == WETH) {
        uint256 nativeBalance = msgValue; // only current caller's ETH
        ...
    }
}
```

Alternatively, at the start of every payable entry point, revert if `address(this).balance > msg.value` (i.e., the router already holds pre-existing ETH), or require callers to wrap ETH to WETH themselves before calling the router.

## Proof of Concept

```solidity
function test_strandedEthStolenByNextSwapper() public {
    // Alice sends 1 ETH but only needs 0.9 ETH for her swap
    vm.prank(alice);
    router.exactInputSingle{value: 1 ether}(
        ExactInputSingleParams({
            pool: pool, tokenIn: WETH,
            amountIn: 0.9 ether, amountOutMinimum: 0,
            recipient: alice, ...
        })
    );
    // pay() wraps 0.9 ETH; 0.1 ETH remains stranded
    assertEq(address(router).balance, 0.1 ether);

    // Bob swaps with value=0, amountIn=0.1 ETH (tokenIn=WETH)
    // pay() sees nativeBalance=0.1 ETH >= value=0.1 ETH → wraps Alice's ETH
    vm.prank(bob);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({
            pool: pool, tokenIn: WETH,
            amountIn: 0.1 ether, amountOutMinimum: 0,
            recipient: bob, ...
        })
    );

    // Bob received output without paying; Alice's 0.1 ETH is gone
    assertEq(address(router).balance, 0);
}
```

The root cause is at [6](#0-5)  where `address(this).balance` is used without verifying it was deposited by the current payer.
