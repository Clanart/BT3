### Title
Unbounded `waiting_for_block` accumulation in `SpiceChunkValidatorActor` allows memory exhaustion via unverified deferred messages — (`chain/client/src/spice/chunk_validator_actor.rs`)

---

### Summary

`SpiceChunkValidatorActor` stores incoming `SpiceChunkStateWitness` and `SpiceChunkContractAccesses` messages in an unbounded `HashMap<CryptoHash, (Vec<…>, Vec<…>)>` when the referenced block is not yet in the store. No size cap, no distance-from-head guard, and — critically for contract-accesses messages — no signature verification is performed before the data is appended. An attacker can reference arbitrary fake block hashes that will never arrive, causing the map to grow without bound and exhausting validator memory.

---

### Finding Description

`SpiceChunkValidatorActor` declares:

```rust
waiting_for_block:
    HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>,
``` [1](#0-0) 

When a witness arrives for a block not yet in the store, it is pushed unconditionally:

```rust
self.waiting_for_block.entry(chunk_id.block_hash).or_default().0.push(witness);
``` [2](#0-1) 

When a `SpiceChunkContractAccesses` message arrives for an unknown block, the epoch-id lookup fails, and the message is stored **before any signature check**:

```rust
Err(EpochError::EpochOutOfBounds(_)) | Err(EpochError::MissingBlock(_)) => {
    // TODO(spice): Implement additional checks (size limit, distance to head) …
    self.waiting_for_block.entry(chunk_id.block_hash).or_default().1.push(accesses);
    return Ok(());
}
``` [3](#0-2) 

The signature verification that would reject a forged sender only runs after the epoch-id is resolved — which requires the block to be present. For a fake block hash the block never arrives, so the entry is never cleaned up and the signature is never checked.

The non-SPICE path (`ChunkValidationActor::handle_orphan_witness`) enforces three guards that are entirely absent here:

1. **Distance from head** — `ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD: Range<BlockHeight> = 2..6`
2. **Size cap** — `max_orphan_witness_size`
3. **Bounded LRU cache** — `OrphanStateWitnessPool` backed by `LruCache` [4](#0-3) [5](#0-4) 

The SPICE actor's own TODO comments acknowledge the missing guards and point to the non-SPICE path as the reference:

> `// TODO(spice): Implement additional checks (size limit, distance to head) before adding witness to waiting_for_block. See non-spice handle_orphan_witness().` [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A network peer can send an unbounded stream of `SpiceChunkContractAccesses` messages (each containing up to `MAX_CONTRACTS_PER_REQUEST` code hashes) referencing arbitrary fake block hashes. Because the block is absent, the epoch-id lookup fails, the signature is never verified, and each message is appended to the `waiting_for_block` map. The map grows without bound, exhausting the validator's heap. A memory-exhausted validator cannot process witnesses, cannot emit `SpiceChunkEndorsement` messages, and therefore cannot participate in SPICE consensus for any shard it is assigned to.

---

### Likelihood Explanation

Any peer connected to the validator can send `SpiceChunkContractAccessesMessage`. The message is forwarded directly from `SpiceDataDistributorActor` to `SpiceChunkValidatorActor` without any pre-filtering:

```rust
impl Handler<SpiceChunkContractAccessesMessage> for SpiceDataDistributorActor {
    fn handle(&mut self, msg: SpiceChunkContractAccessesMessage) {
        self.contract_accesses_validator_sender.send(msg);
    }
}
``` [8](#0-7) 

No rate-limiting, no size check, and no signature check is applied before the message reaches the actor and is stored. The attacker only needs a single TCP connection to the target validator.

---

### Recommendation

Apply the same guards used by the non-SPICE orphan-witness path before appending to `waiting_for_block`:

1. **Verify the signature immediately** using any plausible epoch's validator set (e.g., the current or previous epoch) before deferring. Reject messages whose signature cannot be verified.
2. **Enforce a distance-from-head bound** on the block hash's implied height. Reject messages referencing blocks more than a small constant (e.g., 6) blocks ahead of the current head.
3. **Cap the total number of deferred entries** per block hash and globally, using an LRU structure analogous to `OrphanStateWitnessPool`.
4. **Enforce a per-message size limit** on `SpiceChunkContractAccesses` (number of code hashes × hash size) before storing.

---

### Proof of Concept

1. Attacker connects to a SPICE validator node as a network peer.
2. Attacker generates `N` distinct random 32-byte values as fake `block_hash` values.
3. For each fake `block_hash`, attacker constructs a `SpiceChunkContractAccesses` message with `MAX_CONTRACTS_PER_REQUEST` code hashes and any signature bytes (signature is not checked before storage).
4. Attacker sends all `N` messages to the validator.
5. Each message passes the `get_epoch_id` lookup with `EpochError::MissingBlock`, enters the `waiting_for_block` map, and is never evicted.
6. The validator's heap grows by approximately `N × MAX_CONTRACTS_PER_REQUEST × 32` bytes plus map overhead, with no upper bound.
7. Once memory is exhausted, the validator OOMs and stops emitting endorsements. [9](#0-8) [10](#0-9)

        if accesses.contracts().len() > MAX_CONTRACTS_PER_REQUEST {
            tracing::debug!(
                target: "spice_chunk_validator",
                ?chunk_id,
                num_contracts = accesses.contracts().len(),

            return Ok(());

        // Verify signature of accesses message against any chunk producer for the shard

                // Block not in store yet — defer until it arrives.

                    target: "spice_chunk_validator",
                    ?chunk_id,
