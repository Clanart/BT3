### Title
Missing Expiry in `deployToken` Signed Payload Allows Stale Signature Replay to Deploy Token with Wrong Decimals — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies a NEAR MPC signature over a `MetadataPayload` that contains no nonce and no expiry timestamp. A valid signature issued by NEAR MPC for a given token's metadata remains valid indefinitely. If NEAR MPC ever re-signs corrected metadata for the same token (e.g., to fix a wrong decimal value), an attacker who holds the original signature can front-run the legitimate deployment and permanently lock the token into the wrong configuration.

---

### Finding Description

`deployToken` constructs its signed hash from only four fields:

```
PayloadType.Metadata | token | name | symbol | decimals
``` [1](#0-0) 

There is no nonce field and no expiry/deadline field in this hash. The only replay guard is the `isBridgeToken` mapping check:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [2](#0-1) 

This means:

1. Once NEAR MPC emits a signature for a `MetadataPayload`, that signature is valid forever — there is no on-chain mechanism to expire or invalidate it.
2. If NEAR MPC later re-signs corrected metadata for the same `token` ID (e.g., fixing `decimals` from 18 to 6), **both** the old and new signatures are simultaneously valid.
3. An attacker who observed the original (incorrect) signature — e.g., from a NEAR event log, a failed EVM transaction, or a mempool broadcast — can submit it before the legitimate deployer uses the corrected one.
4. Because the `isBridgeToken` check fires after the first successful deployment, the corrected signature can never be used afterward.

Contrast this with `finTransfer`, which correctly includes `destinationNonce` in the signed payload and tracks it in `completedTransfers`: [3](#0-2) 

`deployToken` has no equivalent protection.

---

### Impact Explanation

Deploying a bridged token with wrong `decimals` permanently breaks the bridge's decimal normalization for that token. All subsequent `finTransfer` calls will mint or release amounts scaled to the wrong decimal precision, directly corrupting the collateralization ratio between the NEAR-side locked supply and the EVM-side minted supply. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

The `_normalizeDecimals` function caps at 18 but does not validate correctness: [4](#0-3) 

Once the token is deployed with wrong decimals, there is no upgrade path for the decimal value itself (only name/symbol can be updated via `setMetadata`), making the corruption permanent.

---

### Likelihood Explanation

**Low.** The attack requires:
- NEAR MPC to have signed at least two different `MetadataPayload` instances for the same `token` ID (e.g., an initial incorrect signing followed by a correction).
- The attacker to have obtained the first (incorrect) signature before the token is deployed on EVM.
- The attacker to submit the stale signature before the legitimate deployer submits the corrected one.

NEAR MPC signing events are observable on-chain (NEAR event logs), so the attacker does not need privileged access to obtain the old signature. The attack window is the gap between the first signing and the first EVM deployment.

---

### Recommendation

1. **Add a nonce to `MetadataPayload`**: Include a per-token or global nonce in the signed hash and track used nonces in storage, so each signature can only be consumed once.
2. **Add an `expiredAt` timestamp to `MetadataPayload`**: Include a deadline in the signed hash and reject calls where `block.timestamp > expiredAt`. This bounds the validity window of any issued signature.

Both fields must be part of the Borsh-encoded hash to be covered by the MPC signature.

---

### Proof of Concept

1. NEAR MPC signs `MetadataPayload{token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 18}` (incorrect — should be 6). The signature `sig_old` is emitted in a NEAR event log.
2. NEAR MPC discovers the error and signs corrected `MetadataPayload{token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6}`, producing `sig_new`.
3. Attacker observes `sig_old` from the NEAR event log and calls `deployToken(sig_old, {token: "usdc.near", ..., decimals: 18})` on EVM before the legitimate relayer submits `sig_new`.
4. The `ECDSA.recover` check passes (signature is valid), `isBridgeToken[nearToEthToken["usdc.near"]]` is `false` (token not yet deployed), so the token is deployed with `decimals = 18`.
5. `isBridgeToken[bridgeTokenProxy]` is now `true`. The legitimate relayer's call with `sig_new` reverts with `"ERR_TOKEN_EXIST"`.
6. All future `finTransfer` calls for USDC mint amounts 10^12× too large relative to the NEAR-side locked supply, breaking collateralization. [5](#0-4)

    function deployToken(
        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

        // slither-disable-next-line reentrancy-no-eth
        address bridgeTokenProxy = address(
            new ERC1967Proxy(
                tokenImplementationAddress,
                abi.encodeWithSelector(
                    BridgeToken.initialize.selector,
                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )
        );

        deployTokenExtension(
            metadata.token,
            bridgeTokenProxy,
            decimals,
            metadata.decimals
        );

        emit BridgeTypes.DeployToken(
            bridgeTokenProxy,
            metadata.token,
            metadata.name,
            metadata.symbol,
            decimals,
            metadata.decimals
        );

        isBridgeToken[address(bridgeTokenProxy)] = true;
        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
        nearToEthToken[metadata.token] = address(bridgeTokenProxy);

        return bridgeTokenProxy;
    }

    function _normalizeDecimals(uint8 decimals) internal pure returns (uint8) {
        uint8 maxAllowedDecimals = 18;
        if (decimals > maxAllowedDecimals) {
            return maxAllowedDecimals;
        }
