### Title
Integer Underflow in `nestedCreates` Nonce Reconciliation Corrupts Sender Nonce After EVM CREATE — (`File: x/evm/keeper/state_transition.go`)

---

### Summary

In `ApplyMessageWithConfig`, the nonce reconciliation logic after `evm.Create()` computes `nestedCreates` via an unchecked unsigned subtraction. If `evm.Create()` reverts (leaving `afterCreateNonce == msg.Nonce`), the subtraction `afterCreateNonce - msg.Nonce - 1` wraps around to `uint64` max (`2^64 - 1`). The subsequent `stateDB.SetNonce(sender, oldNonce + nestedCreates, ...)` then sets the sender's nonce to an astronomically large value, permanently corrupting the account's nonce in committed state.

---

### Finding Description

In `ApplyMessageWithConfig`, the `contractCreation` branch resets the stateDB nonce to `msg.Nonce` before calling `evm.Create()`, then reconciles the nonce afterward:

```go
// x/evm/keeper/state_transition.go lines 468-482
if contractCreation {
    oldNonce := stateDB.GetNonce(sender)
    stateDB.SetNonce(sender, msg.Nonce, tracing.NonceChangeUnspecified)
    ret, _, leftoverGas, vmErr = evm.Create(sender, msg.Data, leftoverGas, uint256.MustFromBig(msg.Value))
    // evm.Create() increments nonce from msg.Nonce to (msg.Nonce + 1 + nestedCreates)
    // We need: oldNonce + nestedCreates
    afterCreateNonce := stateDB.GetNonce(sender)
    nestedCreates := afterCreateNonce - msg.Nonce - 1   // <-- UNSAFE uint64 subtraction
    stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
}
```

The comment assumes `evm.Create()` always increments the nonce by at least 1 (to `msg.Nonce + 1`). However, `evm.Create()` can fail before it increments the nonce — for example, when the contract address already exists (collision), when the depth limit is exceeded, or when the EVM returns an error before the nonce bump. In those cases, `afterCreateNonce == msg.Nonce`, and the subtraction becomes:

```
nestedCreates = msg.Nonce - msg.Nonce - 1 = 0 - 1 = 2^64 - 1  (uint64 wrap)
```

Then:
```
stateDB.SetNonce(sender, oldNonce + (2^64 - 1), ...)
```

This sets the sender's nonce to `oldNonce + 2^64 - 1`, which wraps to `oldNonce - 1` (or a very large value depending on `oldNonce`). This corrupted nonce is then committed to the Cosmos KV store via `stateDB.Commit()`.

The `vmErr` from `evm.Create()` is **not checked before the nonce reconciliation** — the code proceeds to compute and apply the nonce correction regardless of whether the EVM call succeeded or failed.

---

### Impact Explanation

**High — EVM state transition bug that permits invalid nonce state to commit, mis-accounting user nonce.**

Once the sender's nonce is set to `2^64 - 1` (or near it), all subsequent transactions from that account will fail nonce verification (`txNonce != expectedNonce`) in `CheckAndSetEthSenderNonce`, permanently bricking the account. The account's funds become inaccessible via normal EVM transactions. This constitutes permanent loss of account control — equivalent to a fund-locking impact — triggered by a single unprivileged contract creation transaction that causes a CREATE collision or depth-limit failure.

Additionally, if `oldNonce + nestedCreates` overflows `uint64`, the resulting nonce value is unpredictable and may allow nonce reuse, enabling replay of previously-valid transactions.

---

### Likelihood Explanation

A CREATE collision is reachable by any unprivileged user:
1. An attacker can front-run a victim's contract deployment by deploying to the same address first (using `CREATE2` or by predicting the victim's `CREATE` address from `keccak256(rlp([sender, nonce]))`).
2. The victim's `evm.Create()` call then fails because the target address already has code, returning without incrementing the nonce.
3. The nonce reconciliation underflows, corrupting the victim's nonce.

Alternatively, a user can trigger this themselves by crafting a contract whose constructor causes a CREATE at maximum call depth, causing the inner CREATE to fail without a nonce increment.

---

### Recommendation

Check `vmErr` (or compare `afterCreateNonce >= msg.Nonce + 1`) before computing `nestedCreates`. If `evm.Create()` did not increment the nonce, skip the reconciliation or handle the zero-nested-creates case explicitly:

```go
afterCreateNonce := stateDB.GetNonce(sender)
var nestedCreates uint64
if afterCreateNonce > msg.Nonce {
    nestedCreates = afterCreateNonce - msg.Nonce - 1
}
stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
```

---

### Proof of Concept

1. Attacker deploys a contract at address `A` using `CREATE2`.
2. Victim submits a `MsgEthereumTx` with `To == nil` (contract creation), where the derived address is also `A` (same sender + nonce).
3. In `ApplyMessageWithConfig`:
   - `oldNonce = stateDB.GetNonce(sender)` = e.g. `N+1` (post-ante-handler increment)
   - `stateDB.SetNonce(sender, msg.Nonce=N, ...)` — reset to `N`
   - `evm.Create()` fails because address `A` already has code; nonce is **not** incremented; `afterCreateNonce = N`
   - `nestedCreates = N - N - 1 = 2^64 - 1` (uint64 underflow)
   - `stateDB.SetNonce(sender, (N+1) + (2^64-1), ...)` = `stateDB.SetNonce(sender, N, ...)` if wrapping, or a huge value
4. `stateDB.Commit()` persists the corrupted nonce.
5. All future transactions from the victim fail with `invalid nonce`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

func CheckAndSetEthSenderNonce(
	ctx sdk.Context, tx sdk.Tx, ak evmtypes.AccountKeeper, unsafeUnOrderedTx bool, accountGetter AccountGetter, nonceCache *cache.AnteCache,
) ([]cache.TxNonce, error) {
	var pending []cache.TxNonce

	for _, msg := range tx.GetMsgs() {
		msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
		if !ok {
			return nil, errorsmod.Wrapf(errortypes.ErrUnknownRequest, "invalid message type %T, expected %T", msg, (*evmtypes.MsgEthereumTx)(nil))
		}

		tx := msgEthTx.AsTransaction()

		from := msgEthTx.GetFrom()
		acc := accountGetter(from)
		if acc == nil {
			return nil, errorsmod.Wrapf(
				errortypes.ErrUnknownAddress,
				"account %s is nil", common.BytesToAddress(from.Bytes()),
			)
		}
		expectedNonce := acc.GetSequence()
		txNonce := tx.Nonce()
		fromStr := from.String()

		// if flag is set, we bypass nonce all check verification
		if !unsafeUnOrderedTx {
			ex := nonceCache.Exists(fromStr, txNonce)
			// to support tx replacement, we check if the transaction nonce exists in the cache and if yes we skip
			// nonce verification, and we don't set the sequence
			// We allow skip verification only during CheckTx to keep sequence safe during the execution.
			if ctx.IsCheckTx() && !ctx.IsReCheckTx() && ex {
				continue
			}

			// nonce verification, the sequence needs to be in order
			if txNonce != expectedNonce {
				// delete in case of recheck tx
				if ex {
					nonceCache.Delete(fromStr, txNonce)
				}
				return nil, errorsmod.Wrapf(
					errortypes.ErrInvalidSequence,
					"invalid nonce; got %d, expected %d", txNonce, expectedNonce,
				)
			}

			if ctx.IsCheckTx() {
				if !ctx.IsReCheckTx() {
					pending = append(pending, cache.TxNonce{Address: fromStr, Nonce: txNonce})
				}
			} else if ex {
				// delete in case of deliver tx
				nonceCache.Delete(fromStr, txNonce)
			}
		}

		// increase sequence of sender
		if err := acc.SetSequence(expectedNonce + 1); err != nil {
			return nil, errorsmod.Wrapf(err, "failed to set sequence to %d", acc.GetSequence()+1)
		}

		ak.SetAccount(ctx, acc)
	}

	return pending, nil
}
