### Title
Detached Burn Cross-Contract Call Silently Fails, Leaving Source Tokens Unburned While Destination Chain Mints тАФ (`File: near/omni-bridge/src/lib.rs`)

### Summary
`burn_tokens_if_needed` in the NEAR omni-bridge fires a cross-contract `burn` call to the deployed token contract using `.detach()`, allocating only 3 TGas. Because the call is detached, its result is never observed. If the burn fails for any reason (gas exhaustion, token contract panic, etc.), the bridge has already committed the transfer state and the MPC will sign the transfer, allowing the destination chain to mint tokens while the source tokens remain unburned in the bridge contract тАФ creating unbacked supply.

### Finding Description
In `near/omni-bridge/src/lib.rs`, the helper `burn_tokens_if_needed` is defined as:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)   // 3 TGas
            .burn(amount)
            .detach();                          // fire-and-forget, result never checked
    }
}
``` [1](#0-0) [2](#0-1) 

`BURN_TOKEN_GAS` is `Gas::from_tgas(3)` тАФ 3 TGas. The target `OmniToken::burn` calls `internal_withdraw`, a storage-modifying operation that itself requires gas for trie reads and writes. 3 TGas is extremely tight for a cross-contract call that must cover scheduling overhead plus a storage-modifying function. [3](#0-2) 

This function is called in three critical paths, all of which have already committed bridge state before the detached burn fires:

**Path 1 тАФ `init_transfer_internal`** (the primary outbound transfer path): [4](#0-3) 

The sequence is:
1. `add_transfer_message` тАФ transfer is recorded in `pending_transfers` (state committed).
2. `burn_tokens_if_needed` тАФ detached burn fires, result ignored.
3. `lock_tokens_if_needed` тАФ locked-token accounting updated.
4. Returns `U128(0)` to the token contract тАФ bridge keeps all tokens.

**Path 2 тАФ `fast_fin_transfer_to_other_chain`**: [5](#0-4) 

**Path 3 тАФ `resolve_fast_transfer`** (callback after fast-transfer send): [6](#0-5) 

In all three paths, the bridge state (transfer recorded, nonce incremented, locked-token map updated) is fully committed before the detached burn executes. If the burn fails, the bridge has no mechanism to detect or revert.

### Impact Explanation
When the burn fails silently:

- The bridge holds the source deployed tokens (they were transferred in via `ft_transfer_call` and the bridge returned `U128(0)` keeping them).
- The transfer message is recorded and the MPC will sign it.
- The destination chain (EVM, Solana, StarkNet) receives a valid MPC-signed transfer and mints the corresponding bridged tokens to the recipient.
- The source token's `ft_total_supply` is inflated тАФ the tokens still exist in the bridge's account but were supposed to be destroyed.
- The bridge's `locked_tokens` accounting is updated as if the burn succeeded, breaking the collateralization invariant.

This constitutes **unauthorized minting of bridged assets** (tokens minted on the destination chain without the source tokens being burned) and **balance/accounting corruption that breaks bridge collateralization** тАФ matching the Critical and High impact tiers.

### Likelihood Explanation
3 TGas is the allocation for a cross-contract call that must cover:
- NEAR runtime scheduling overhead for the cross-contract call itself (~1тАУ2 TGas).
- `assert_controller` (storage read).
- `internal_withdraw` (storage read + write for the account balance trie node).

Under normal conditions this is borderline; under any elevated storage cost or token contract complexity it will fail. Additionally, if the token contract is upgraded (the controller can call `set_metadata`, `attach_full_access_key`, etc.) or if the bridge's account is not registered in the token contract, the burn will panic and the detached call will silently fail. Any unprivileged user who initiates a transfer of a deployed token triggers this path.

### Recommendation
1. **Remove `.detach()`** and instead chain the burn as a proper promise with a callback that verifies success. If the burn fails, the callback should revert the transfer message and locked-token state, and return the tokens to the sender.
2. **Increase `BURN_TOKEN_GAS`** to at least 10тАУ15 TGas to safely cover the cross-contract call overhead plus `internal_withdraw`.
3. Alternatively, restructure the flow so the burn is performed *before* the transfer message is committed, making the commit conditional on burn success.

### Proof of Concept
1. Alice holds 1,000 bridge-minted NEAR tokens (origin: ETH, deployed by the omni-bridge).
2. Alice calls `ft_transfer_call` on the token contract, sending 1,000 tokens to the bridge with an `InitTransfer` message targeting an ETH address.
3. The bridge's `ft_on_transfer` тЖТ `init_transfer` тЖТ `init_transfer_internal` executes:
   - Records the transfer in `pending_transfers` (nonce N).
   - Fires `burn(1000)` with `.detach()` and 3 TGas тАФ the call runs out of gas and fails silently.
   - Updates `locked_tokens` for ETH chain.
   - Returns `U128(0)` тАФ bridge keeps the 1,000 tokens.
4. A relayer calls `sign_transfer` for nonce N; the MPC signs the payload.
5. Alice (or the relayer) submits the MPC signature to the EVM `OmniBridge.finTransfer`, which mints 1,000 ETH-side tokens to Alice's ETH address.
6. Alice now holds 1,000 tokens on ETH. The NEAR bridge contract also holds 1,000 unburned tokens. The `ft_total_supply` of the NEAR deployed token is inflated by 1,000, and the bridge's collateral is unbacked by that amount. [1](#0-0) [7](#0-6) [3](#0-2)
