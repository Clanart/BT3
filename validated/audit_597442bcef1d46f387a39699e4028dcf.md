## Key Code Examined

**`set_state_header`** in `chain/chain/src/state_sync/adapter.rs` (lines 368–531) performs these checks:

1. `validate_chunk_proofs(&chunk, ...)` — verifies internal chunk consistency (hash, tx root, receipts root). **No shard_id check.**
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), chunk_proof, &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk is *somewhere* in the block's Merkle tree.
3. Receipt proof validation using `shard_id` (the requested shard A).
4. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the state root node against the chunk's `prev_state_root`.

**`verify_path`** in `core/primitives/src/merkle.rs` (lines 113–115):

## Vulnerability Assessment

### Title
Missing shard-index binding in `set_state_header` Merkle proof check allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` verifies that the provided chunk is *present* in the block's `chunk_headers_root` Merkle tree, but does not verify that it is at the *position* corresponding to the requested `shard_id`. An attacker controlling a state-sync peer can supply a `ShardStateSyncResponseHeader` whose inner chunk belongs to shard B while the outer request is for shard A. All existing checks pass, the header is stored under `StateHeaderKey(shard_A_id, sync_hash)`, and subsequent `set_state_part` calls validate shard A's state parts against shard B's `prev_state_root`, installing shard B's state into shard A's trie.

### Finding Description

**Step 1 — Attacker constructs a crafted header:**
- `chunk`: a valid, on-chain chunk for shard B (internally consistent, passes `validate_chunk_proofs`)
- `chunk_proof`: the legitimate Merkle path for chunk B's position in the block's `chunk_headers_root`
- `prev_chunk_header` / `prev_chunk_proof`: shard B's previous chunk and its valid Merkle proof
- `incoming_receipts_proofs` / `root_proofs`: valid receipt proofs for shard A (public chain data)
- `state_root_node`: valid state root node for shard B's `prev_state_root`

**Step 2 — `set_state_header(shard_A_id, sync_hash, crafted_header)` is called:**

- `validate_chunk_proofs` passes — chunk B is internally valid. [1](#0-0) 
- `verify_path(chunk_headers_root, chunk_B_proof, ChunkHashHeight(chunk_B_hash, ...))` passes — chunk B IS in the block's Merkle tree; `verify_path` does not check position. [2](#0-1) 
- `verify_path` for `prev_chunk` passes for the same reason. [3](#0-2) 
- Receipt proofs pass — attacker supplies valid proofs for shard A's receipts (public data). [4](#0-3) 
- `validate_state_root_node(state_root_node, chunk_B.prev_state_root())` passes — attacker supplies a valid node for shard B's root. [5](#0-4) 
- Header stored under `StateHeaderKey(shard_A_id, sync_hash)`. [6](#0-5) 

**Step 3 — `set_state_part(shard_A_id, sync_hash, part_id, part)` is called:**

- Retrieves the stored header (containing chunk B).
- Extracts `state_root` from chunk B's `prev_state_root` (shard B's root).
- Calls `validate_state_part(shard_A_id, shard_B_root, part_id, part)`.
- Attacker supplies state parts valid for shard B's trie; they pass validation and are written to shard A's DB column. [7](#0-6) 

**The missing guard:** `verify_path_with_index` exists in the codebase and enforces positional binding, but `set_state_header` uses plain `verify_path`. There is no explicit `chunk.shard_id() == shard_id` check anywhere in the function. [8](#0-7) [9](#0-8) 

### Impact Explanation

After a successful attack, the syncing node installs shard B's state into shard A's trie. The node's post-sync state root for shard A will be shard B's `prev_state_root`, which will not match the canonical chain's expected state root for shard A. The node will be unable to participate correctly in block production or validation for shard A, and any account/contract state queries against shard A will return shard B's data.

### Likelihood Explanation

The attacker must control a state-sync peer that the victim node selects. State-sync peers are selected from the network without strong authentication. An attacker running a node can position themselves as a state-sync provider. The construction requires only public on-chain data (chunk bodies, Merkle paths, receipt proofs) — no private keys or validator privileges are needed.

### Recommendation

Replace the positionally-unbound `verify_path` call with `verify_path_with_index`, passing the shard index corresponding to `shard_id` as the expected position. Additionally, add an explicit guard:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other("set_shard_state failed: chunk shard_id mismatch".into()));
}
```

This should be checked immediately after `cloned_chunk()` is called, before any further validation.

### Proof of Concept

Integration test strategy:
1. Set up a two-shard network; produce blocks until epoch boundary.
2. Obtain a legitimate `ShardStateSyncResponseHeader` for shard B.
3. Construct a crafted header: keep chunk B and its proof, but supply shard A's receipt proofs and a valid `state_root_node` for shard B.
4. Call `set_state_header(shard_A_id, sync_hash, crafted_header)` on a syncing client.
5. Assert it returns `Ok(())` (demonstrating the missing check).
6. Call `set_state_part` with shard B's state parts.
7. Assert the installed state root for shard A equals shard B's `prev_state_root`, not shard A's.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?

        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
