### Title
Missing `state_diff_commitment` Validation in P2P State Diff Assembly Allows Malicious Peer to Inject Corrupted State — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`parse_data_for_block` assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and validates only that the summed `chunk.len()` equals `target_state_diff_len` from the stored header, plus an intra-field duplicate check. It never computes `calculate_state_diff_hash` on the assembled result and never compares it against the `state_diff_commitment` already stored in the header. A malicious peer can craft two `ContractDiff` chunks for the same address that set **different fields** (one sets `class_hash`, the other sets `storage_diffs`), satisfy the length check, pass the conflict check, and cause the node to persist a `ThinStateDiff` with a wrong class hash and spurious storage entries.

---

### Finding Description

**`StateDiffChunk::len` counts fields, not addresses.** [1](#0-0) 

A `ContractDiff` with `class_hash=Some(X)` and empty `storage_diffs` has `len()=1`. A second `ContractDiff` for the **same address** with `class_hash=None` and `storage_diffs={k:v}` also has `len()=1`. Together they sum to 2, satisfying `target_state_diff_len=2`.

**`unite_state_diffs` only rejects same-field duplicates.** [2](#0-1) 

- Chunk 1 (`class_hash=Some(X)`): inserts `deployed_contracts[addr]=X` — no prior entry, no error.
- Chunk 2 (`storage_diffs={k:v}`): addr not yet in `state_diff.storage_diffs` → takes the `None` branch and inserts `storage_diffs[addr]={k:v}` — no error.

The conflict guard fires only when the **same field** is set twice for the same address. Cross-field splits for the same address are silently merged.

**No commitment hash check after assembly.** [3](#0-2) 

After the length check passes, `validate_deprecated_declared_classes_non_conflicting` is the only remaining guard. There is no call to `calculate_state_diff_hash` and no comparison against `header.state_diff_commitment`. The assembled `ThinStateDiff` is written directly to storage: [4](#0-3) 

**`verify_state_diff_hash` exists only in the committer, not in the P2P sync path.** [5](#0-4) 

That guard is gated on `self.config.verify_state_diff_hash` and lives in the batcher/committer path. The P2P sync client has no equivalent.

---

### Impact Explanation

The syncing node stores a `ThinStateDiff` with:
- `deployed_contracts[addr] = X` (attacker-chosen class hash, not the committed `Y`)
- `storage_diffs[addr] = {k: v}` (spurious storage entry not in the committed diff)

Every subsequent RPC call (`starknet_getClassHashAt`, `starknet_getStorageAt`), fee estimation, simulation, and execution trace for that contract address will read the wrong class hash and wrong storage value from this node's authoritative storage. If the node is used as a sequencer's state backend or proof input, the wrong state root propagates into proof production.

---

### Likelihood Explanation

Any peer on the P2P network can send crafted `StateDiffChunk` messages. No privilege is required. The attack requires knowing `target_state_diff_len` for a block (available from the synced header) and crafting two `ContractDiff` messages whose `len()` values sum to that target. The `state_diff_commitment` in the header is the exact cryptographic guard that should prevent this, but it is never consulted during state diff assembly.

---

### Recommendation

After the assembly loop completes and `current_state_diff_len == target_state_diff_len`, compute and verify the commitment:

```rust
let computed = calculate_state_diff_hash(&result);
let expected = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_commitment
    .ok_or(...)?;
if computed != expected {
    return Err(ParseDataError::BadPeer(BadPeerError::StateDiffCommitmentMismatch { ... }));
}
```

This mirrors the existing `verify_state_diff_hash` logic in the committer. [5](#0-4)

### Proof of Concept

```
committed diff for block N:
  deployed_contracts: {addr: Y}
  storage_diffs: {}
  → state_diff_length = 1, state_diff_commitment = H(Y, no-storage)

attacker sends (target_state_diff_len = 1 is too tight; use target=2 with a real block
where committed diff has deployed_contracts:{addr:Y} + nonces:{addr:N} → len=2):

  Chunk 1: ContractDiff { contract_address: addr, class_hash: Some(X), nonce: None, storage_diffs: {} }
           → len() = 1
  Chunk 2: ContractDiff { contract_address: addr, class_hash: None,    nonce: None, storage_diffs: {k:v} }
           → len() = 1

  current_state_diff_len = 2 == target_state_diff_len ✓
  no ConflictingStateDiffParts (different fields) ✓
  no commitment check ✓

stored ThinStateDiff:
  deployed_contracts: {addr: X}   ← wrong class hash
  storage_diffs:      {addr: {k:v}} ← spurious storage
  nonces:             {}           ← missing committed nonce

starknet_getClassHashAt(addr) → X  (should be Y)
starknet_getStorageAt(addr, k) → v  (should be 0)
```
