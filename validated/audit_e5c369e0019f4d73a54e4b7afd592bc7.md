### Title
`deploy_token` Borsh Payload Lacks Chain-ID Domain Binding, Enabling Cross-Chain Signature Replay — (`starknet/src/omni_bridge.cairo`, `starknet/src/bridge_types.cairo`)

---

### Summary

`deploy_token` hashes and verifies a Borsh-encoded payload that contains **no chain identifier and no contract address**. A signature produced by the MPC key for one Starknet deployment is therefore cryptographically valid on every other Starknet deployment that shares the same `omni_bridge_derived_address`. An unprivileged attacker who observes a valid `deploy_token` call on chain A can replay the identical `(signature, payload)` pair on chain B to deploy a bridge token without MPC authorization for that chain.

`fin_transfer` is **not** affected by the same gap: it passes `omni_bridge_chain_id` into `to_borsh`, so the chain ID is bound into the signed bytes.

---

### Finding Description

**`_verify_borsh_signature`** (the single verification primitive for both entrypoints):

```cairo
fn _verify_borsh_signature(
    ref self: ContractState, borsh_bytes: @ByteArray, signature: Signature,
) {
    let message_hash_le = compute_keccak_byte_array(borsh_bytes);
    let message_hash = reverse_u256_bytes(message_hash_le);
    let sig = signature_from_vrs(signature.v, signature.r, signature.s);
    verify_eth_signature(message_hash, sig, self.omni_bridge_derived_address.read());
}
``` [1](#0-0) 

It hashes raw Borsh bytes with no EIP-712 domain separator, no contract address, and no Ethereum chain ID. The only domain binding comes from what the caller passes as `borsh_bytes`.

**`deploy_token` call site** passes `payload.to_borsh()` with no chain argument:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(), signature);
``` [2](#0-1) 

**`MetadataPayload::to_borsh()`** serializes only `[PayloadType::Metadata, token, name, symbol, decimals]`:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    let mut borsh_bytes: ByteArray = Default::default();
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [3](#0-2) 

No `chain_id`, no contract address, no action tag beyond the generic `PayloadType::Metadata` byte.

**Contrast with `fin_transfer`**, which correctly binds `chain_id`:

```cairo
_verify_borsh_signature(
    ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature,
);
``` [4](#0-3) 

`TransferMessagePayload::to_borsh(chain_id)` embeds `chain_id` into the serialized bytes (lines 67 and 70), so a `fin_transfer` signature is chain-scoped. [5](#0-4) 

---

### Impact Explanation

Because `MetadataPayload::to_borsh()` omits `chain_id` and contract address, the Keccak digest — and therefore the MPC signature — is **identical** across every Starknet bridge deployment that shares the same `omni_bridge_derived_address`. An attacker can:

1. Observe a valid `deploy_token(sig, payload)` transaction on Starknet mainnet.
2. Submit the same `(sig, payload)` to a Starknet testnet (or any parallel Starknet L2) bridge contract configured with the same MPC-derived address.
3. The `verify_eth_signature` check passes because the hash is identical; the token is deployed on the target chain without the MPC ever authorizing it for that chain.

This constitutes **unauthorized token deployment** — a listed High impact ("Proof, signature, MPC … verification bypass enabling … token deployment").

Secondary concern: because no contract address is bound, two bridge contracts on the **same** chain with the same `omni_bridge_chain_id` and `omni_bridge_derived_address` would also accept each other's `deploy_token` signatures.

---

### Likelihood Explanation

The Omni Bridge MPC key is shared across chains by design (the same `omni_bridge_derived_address` is configured on every chain). Signatures are publicly visible on-chain. The replay requires no privileged access — only the ability to call `deploy_token` on the target chain, which is a public entrypoint. [6](#0-5) 

---

### Recommendation

Bind `omni_bridge_chain_id` (and optionally `get_contract_address()`) into `MetadataPayload::to_borsh()`, mirroring what `TransferMessagePayload::to_borsh(chain_id)` already does:

```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    let mut borsh_bytes: ByteArray = Default::default();
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append_byte(chain_id);          // ← add this
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

And update the call site in `deploy_token`:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature);
```

### Proof of Concept

```
// Chain A (mainnet) — attacker observes this transaction
deploy_token(
  signature = { v, r, s },   // MPC-signed over keccak([1, "usdc.near", "USD Coin", "USDC", 6])
  payload   = { token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 }
)

// Chain B (testnet) — attacker replays identical call
deploy_token(
  signature = { v, r, s },   // same bytes — hash is identical, no chain_id in payload
  payload   = { token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 }
)
// → verify_eth_signature passes; token deployed on chain B without MPC authorization for chain B
```
