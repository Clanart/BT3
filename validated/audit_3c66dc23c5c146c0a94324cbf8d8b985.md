### Title
Missing `chunk.shard_id() == shard_id` Binding in `set_state_header` Allows Cross-Shard State Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from any peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the embedded chunk's `shard_id` field matches the `shard_id` key parameter. The Merkle-inclusion check uses `verify_path` (not `verify_path_with_index`), which proves only that the chunk exists somewhere in the block's `chunk_headers_root` tree — not that it occupies the position corresponding to the requested shard. An unprivileged peer can therefore supply shard B's chunk header (with a valid Merkle proof for shard B's position) in response to a shard A state-sync request, causing the node to store `StateHeaderKey(A, sync_hash) → header_for_B`, then validate and apply shard B's state parts into shard A's trie.

### Finding Description

**Entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called via `Handler<StateHeaderValidationRequest>` in `chain/client/src/client_actor.rs`.

**Missing guard:** The function extracts `chunk = shard_state_header.cloned_chunk()` at line 376 but never asserts `chunk.shard_id() == shard_id`. [1](#0-0) 

**Merkle check does not bind position:** The `verify_path` call at lines 394–403 proves `ChunkHashHeight(chunk_B.hash, height)` is a leaf in `chunk_headers_root`, but `verify_path` is defined as `compute_root_from_path(path, hash_borsh(item)) == root` — it carries no index constraint. [2](#0-1) 

The position-aware variant `verify_path_with_index` (which calls `verify_path_matches_index`) exists but is not used here. [3](#0-2) 

**Receipt proof check is bypassable:** Step 4 hashes receipts as `ReceiptList(shard_id, receipts)` using the *requested* shard A id, not the chunk's actual shard id. An attacker can supply the real on-chain incoming receipts for shard A (publicly available) to satisfy this check, while still embedding shard B's chunk. [4](#0-3) 

**State root node check is shard-agnostic:** Step 5 validates `state_root_node` against `chunk_inner.prev_state_root()` — both come from shard B's chunk, so the check passes trivially. [5](#0-4) 

**Storage commit uses the wrong key:** The header is stored under `StateHeaderKey(shard_id=A, sync_hash)` regardless of the chunk's actual shard. [6](#0-5) 

**`validate_state_part` ignores `shard_id`:** The wrapper passes `shard_id` only to metrics; `validate_state_part_impl` calls `Trie::validate_state_part(state_root, part_id, partial_state)` with no shard binding, so shard B's parts validate against shard B's `state_root` even when called with `shard_id=A`. [7](#0-6) 

**`apply_state_part` writes to shard A's storage slot:** It derives `shard_uid` from the `shard_id` parameter (A), so shard B's trie nodes are committed into shard A's column. [8](#0-7) 

### Impact Explanation

Shard A's trie is permanently overwritten with shard B's state. All account balances, contract state, and receipts for shard A are replaced with shard B's data. Subsequent block processing for shard A will compute wrong state roots, causing the node to diverge from the canonical chain and produce invalid chunks if it is a chunk producer.

### Likelihood Explanation

Any peer that can respond to a `StateRequestHeader` message can execute this attack. State sync is performed by nodes that are catching up after being offline or joining the network — a common operational event. The attacker needs only to be a reachable peer and to know the `sync_hash` (which is deterministic and publicly derivable from the epoch boundary). No validator or operator privileges are required.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the `verify_path` call with `verify_path_with_index`, supplying the expected shard index derived from `shard_layout.get_shard_index(shard_id)`, to cryptographically bind the chunk to its correct position in the block.

### Proof of Concept

Integration test outline (2-shard V1 layout):

1. Produce several blocks with 2 shards so both shards have committed chunks.
2. Obtain `sync_hash` at an epoch boundary.
3. Call `get_state_response_header(shard_id=1, sync_hash)` to get shard 1's legitimate header.
4. Collect the real incoming receipts for shard 0 from the chain store.
5. Construct a crafted `ShardStateSyncResponseHeader` containing shard 1's chunk, shard 1's prev_chunk, shard 1's Merkle proof (at position 1), shard 0's incoming receipts (with valid proofs), and shard 1's `state_root_node`.
6. Call `set_state_header(shard_id=0, sync_hash, crafted_header)` — assert it returns `Ok(())`.
7. Assert `get_state_header(0, sync_hash).chunk().shard_id() == 1` (the stored header belongs to shard 1).
8. Supply shard 1's state parts via `set_state_part(shard_id=0, ...)` — assert each returns `Ok(())`.
9. Call `set_state_finalize(shard_id=0, sync_hash)` and verify shard 0's trie now contains shard 1's state root, confirming the corruption.
