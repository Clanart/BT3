### Title
Sub-Threshold Transfer Amount Causes Permanent Fund Lock with No Recovery Path - (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR→foreign-chain transfer with an `amount_without_fee` smaller than the decimal normalization divisor (`10^(origin_decimals - decimals)`), `init_transfer_internal` locks or burns the user's tokens immediately, but `sign_transfer` will always revert with `ERR_INVALID_AMOUNT_TO_TRANSFER` because `normalize_amount` floors to zero. No cancel or refund function exists for pending transfers, so the user's funds are permanently frozen.

### Finding Description

The bridge normalizes token amounts when crossing decimal-precision boundaries. `normalize_amount` performs integer floor division: [1](#0-0) 

`sign_transfer` correctly guards against a zero normalized amount: [2](#0-1) [3](#0-2) 

However, this guard fires **after** `init_transfer_internal` has already committed the user's tokens — burning deployed tokens or locking native tokens: [4](#0-3) 

The `init_transfer` entry point only validates `fee < amount`; it does not verify that `normalize_amount(amount - fee) > 0`: [5](#0-4) 

`amount_without_fee` is a simple subtraction with no normalization check: [6](#0-5) 

There is no public cancel or refund function for a pending transfer. `update_transfer_fee` can only increase the fee (not decrease it, and cannot change the amount): [7](#0-6) 

**Concrete example** — NEAR token with `origin_decimals = 24`, destination EVM token with `decimals = 18` (divisor = 10^6):

1. User calls `ft_transfer_call` with `amount = 500_000` (< 10^6).
2. `init_transfer_internal` burns/locks 500,000 tokens; `TransferMessage` is stored.
3. Relayer calls `sign_transfer` → `normalize_amount(500_000, {24, 18}) = 500_000 / 1_000_000 = 0` → panics.
4. No recovery path exists; 500,000 tokens are permanently lost.

The documented "dust stays locked/burned" known issue covers only the **remainder** after a successful normalization (e.g., 1 token out of 1,000,001). This is a distinct case: the **entire** transferred amount normalizes to zero, the transfer can never be completed, and the funds are irrecoverable.

### Impact Explanation

Permanent freezing of user funds. Tokens are burned (for deployed/bridged tokens) or locked indefinitely (for native tokens) in the bridge contract with no mechanism to recover them. This matches the **Critical/High** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

### Likelihood Explanation

Any user who sends an amount below the normalization threshold triggers this. For a 24→18 decimal token pair the threshold is 10^6 base units — a small but non-negligible amount for high-precision tokens. The condition is reachable by any unprivileged bridge user via the standard `ft_transfer_call` entry point with no special preconditions.

### Recommendation

Add a normalization check at the point of token commitment, before locking or burning, inside `init_transfer` or `init_transfer_internal`. Specifically, look up the destination token's `Decimals` and verify `normalize_amount(amount_without_fee, decimals) > 0` before proceeding. If the check fails, return the full amount as a refund (the same pattern already used when storage balance is insufficient at line 1847).

```rust
// In init_transfer, after resolving token_address and decimals:
let normalized = Self::normalize_amount(
    amount_without_fee,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 24`, destination EVM address registered with `decimals = 18`.

1. Alice calls `ft_transfer_call` on the NEAR token contract, transferring `amount = 999_999` to the bridge with `fee = 0` and an EVM recipient.
2. `init_transfer` passes the `fee < amount` check (0 < 999,999). `init_transfer_internal` is called.
3. `burn_tokens_if_needed` burns Alice's 999,999 tokens (if deployed token) or `lock_tokens_if_needed` locks them.
4. `TransferMessage` is stored in `pending_transfers`.
5. Relayer calls `sign_transfer`:
   - `normalize_amount(999_999, {origin_decimals: 24, decimals: 18})` = `999_999 / 10^6` = **0**
   - `require!(0 > 0, ...)` → **panics** with `ERR_INVALID_AMOUNT_TO_TRANSFER`
6. No cancel function exists. Alice's 999,999 tokens are permanently lost. [3](#0-2) [8](#0-7) [1](#0-0)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
