### Title
Hardcoded `'BOOTSTRAP'` String Used as Authentication Bypass in `execute_declare_transaction` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS uses the hardcoded felt literal `'BOOTSTRAP'` as a sender address "password" to bypass all authentication, nonce enforcement, and fee payment for declare transactions. This is a direct analog to the `founderHash`/`privatePhrase` bug: the "secret" is a publicly visible constant in the source code, providing zero real security.

---

### Finding Description

In `execute_declare_transaction`, lines 761–776 implement a special fast-path:

```cairo
// Do not run validate or perform any account-related actions for declare transactions that
// meet the following conditions.
// This flow is used for the sequencer to bootstrap a new system.
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        // Declare the class hash and skip the rest of the transaction.
        // Note that prev_value=0 enforces that a class may be declared only once.
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the OS **completely skips**:

1. `check_and_increment_nonce` — nonce replay protection is not enforced.
2. `non_reverting_select_execute_entry_point_func` with `VALIDATE_DECLARE_ENTRY_POINT_SELECTOR` — the account contract's `__validate_declare__` signature check is never run.
3. `charge_fee` — no fee is deducted from any account. [2](#0-1) 

The four conditions required to trigger the bypass are all trivially satisfiable by any party who can submit a transaction:

| Condition | Value |
|---|---|
| `sender_address` | `'BOOTSTRAP'` = felt `0x424f4f545354524150` (ASCII, public) |
| `tx_info.nonce` | `0` |
| `tx_info.version` | `3` |
| `compute_max_possible_fee(tx_info)` | `0` (set all `max_amount` and `max_price_per_unit` fields to zero) |

`compute_max_possible_fee` returns zero when all three resource-bound amounts and prices are zero:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [3](#0-2) 

The `AssertTransactionHash` check at line 732 is a **hint** (`%{ ... }`), not a Cairo constraint. Hints are prover-supplied and are not part of the STARK proof. They impose no cryptographic enforcement on the transaction hash. [4](#0-3) 

The only remaining Cairo-enforced check is that the declared class hash is a valid Sierra class hash pre-image (`finalize_class_hash`), which is a correctness check, not an authorization check. [5](#0-4) 

---

### Impact Explanation

**Critical. Direct loss of funds.**

An attacker who can get a BOOTSTRAP-format declare transaction included in a proven block can register an arbitrary contract class in `contract_class_changes` without owning any account, without a valid signature, and without paying any fee. Once a malicious class is registered on-chain, it can be deployed via `deploy` syscalls. Any user who subsequently interacts with a contract deployed from that class (e.g., a token, a bridge, a vault) is exposed to arbitrary logic chosen by the attacker — enabling direct theft of funds held by those contracts or their callers.

The `dict_update` with `prev_value=0` prevents overwriting an already-declared class, but does not prevent declaring a brand-new class with attacker-controlled bytecode. [6](#0-5) 

---

### Likelihood Explanation

The "password" (`'BOOTSTRAP'`) is hardcoded in the public source code of the OS program. Any party who can submit a transaction to the sequencer — or who operates a sequencer node in a future decentralized sequencer setting — can craft the exact four-field combination required. No key material, privileged access, or brute force is needed. The conditions are deterministic and fully specified by the source code.

---

### Recommendation

Remove the `'BOOTSTRAP'` fast-path entirely from the OS Cairo code. Bootstrapping of initial system classes should be handled at the sequencer/operator layer before the OS proof system is active, not as a permanently open backdoor inside the proven OS. If a bootstrapping mechanism is genuinely required inside the OS, it must be gated on a cryptographic commitment (e.g., a hash of an authorized public key committed in `BlockContext`) rather than a publicly known string literal.

---

### Proof of Concept

1. Construct a v3 declare transaction with:
   - `sender_address = 0x424f4f545354524150` (felt encoding of `'BOOTSTRAP'`)
   - `nonce = 0`
   - `version = 3`
   - All resource bounds (`l1_gas`, `l2_gas`, `l1_data_gas`) `max_amount = 0`, `max_price_per_unit = 0`
   - `class_hash` = hash of any attacker-controlled Sierra class
   - `compiled_class_hash` = corresponding compiled class hash (non-zero)
   - No valid signature required (the `__validate_declare__` entry point is never called)

2. Submit the transaction. The sequencer includes it in a block.

3. The OS Cairo execution reaches line 764, evaluates all four conditions as true, calls `dict_update` to register the class, and returns — skipping nonce increment, signature validation, and fee charging entirely.

4. The attacker's class is now registered on-chain. They deploy a contract from it and drain any funds held by users who interact with it. [7](#0-6)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
