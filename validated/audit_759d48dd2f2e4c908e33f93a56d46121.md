### Title
Front-Running Class Declaration Permanently Poisons `compiled_class_hash` — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS enforces that a class can only be declared once (`prev_value=0`), but it does **not** verify that the supplied `compiled_class_hash` is the correct CASM compilation of the Sierra class identified by `class_hash`. An unprivileged attacker can front-run any pending declare transaction by submitting their own declare for the same `class_hash` with an attacker-controlled `compiled_class_hash`. Because re-declaration is permanently blocked, the class is irreversibly poisoned: any contract subsequently deployed under that class will execute the attacker's CASM, enabling direct loss of funds or permanent freezing of funds.

---

### Finding Description

In `execute_declare_transaction`, the Sierra class hash is rigorously verified: [1](#0-0) 

```cairo
let expected_class_hash = finalize_class_hash(
    contract_class_component_hashes=contract_class_component_hashes
);
with_attr error_message("Invalid class hash pre-image.") {
    assert [class_hash_ptr] = expected_class_hash;
}
```

However, the `compiled_class_hash` — the CASM counterpart — receives only a non-zero check before being committed to the class tree: [2](#0-1) 

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

The `prev_value=0` constraint is the critical enabler: it permanently prevents re-declaration of the same `class_hash`. The OS never verifies that `compiled_class_hash` is the correct CASM compilation of the Sierra class. The two values are independently supplied by the transaction sender.

**Structural analogy to the Uniswap report:**

| Uniswap v4 | StarkNet OS |
|---|---|
| Pool ID = `hash(token0, token1, fee, ...)` — excludes `sqrtPriceX96` | Class ID = `class_hash` (Sierra hash) — excludes `compiled_class_hash` |
| Anyone can initialize with any `sqrtPriceX96` | Anyone can declare with any `compiled_class_hash` (≠ 0) |
| Pool can only be initialized once | Class can only be declared once (`prev_value=0`) |
| Front-runner sets wrong price → LP liquidity drained | Front-runner sets wrong CASM hash → contracts execute malicious code |

**Attack path:**

1. Attacker monitors the mempool for a pending declare transaction targeting Sierra class `S` with `class_hash = H` and `compiled_class_hash = C_legitimate`.
2. The Sierra class `S` is public (it is part of the transaction). The attacker copies it to satisfy `finalize_class_hash`.
3. Attacker previously declared a separate class whose CASM has hash `C_malicious` (making the sequencer aware of that CASM).
4. Attacker submits a valid declare transaction: `{class_hash: H, compiled_class_hash: C_malicious}`, signed by their own account, with a higher fee.
5. Attacker's transaction is included first. The class tree now maps `H → C_malicious`.
6. The legitimate declare fails: `dict_update` panics because `prev_value` is no longer `0`.
7. Any contract subsequently deployed with class `H` will have the OS resolve `H → C_malicious` and execute the malicious CASM.

The class tree update is committed via `state_update` → `squash_class_changes` → `compute_class_commitment`, which writes the poisoned mapping into the global state root: [3](#0-2) 

Once the global state root is updated with the poisoned `compiled_class_hash`, the mapping is permanent and provably correct from the OS's perspective.

---

### Impact Explanation

**Critical — Direct loss of funds / Permanent freezing of funds.**

- **Direct loss of funds**: If `C_malicious` is the hash of attacker-controlled CASM, any contract deployed under class `H` will execute the attacker's code. The malicious CASM can drain the contract's token balances or redirect calls.
- **Permanent freezing of funds**: If `C_malicious` is a hash with no corresponding CASM in the sequencer's database, the class becomes permanently unexecutable. Any contract deployed under class `H` is frozen forever, as the sequencer cannot produce a valid proof for blocks that execute it.

In both cases the damage is irreversible because re-declaration is blocked by the `prev_value=0` invariant.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. Observing a pending declare transaction in the public mempool — straightforward.
2. Submitting a competing declare with a higher fee — standard front-running, no special privilege required.
3. Having previously declared a class whose CASM hash equals `C_malicious` — a one-time setup step that is itself a valid, fee-paying transaction.

No privileged role, leaked key, or operator access is needed. Any account holder can execute this attack against any class that has not yet been declared.

---

### Recommendation

The OS must verify that `compiled_class_hash` is the correct CASM compilation of the Sierra class before committing it to the class tree. Concretely:

1. **Preferred**: Compute the expected `compiled_class_hash` from the Sierra class components during declaration (analogous to how `finalize_class_hash` verifies the Sierra hash) and assert equality with the supplied value.
2. **Alternative**: Require the declarer to supply a proof-of-compilation (e.g., a Merkle witness linking the Sierra class to its CASM) and verify it in Cairo.

Without this check, the `prev_value=0` "declare-once" invariant becomes a liability rather than a protection, because it permanently locks in whatever `compiled_class_hash` was supplied first.

### Proof of Concept

```
// Setup (one-time, by attacker):
// Attacker declares a benign-looking class whose CASM is actually malicious.
// This makes compiled_class_hash = C_malicious known to the sequencer.
declare_tx(class=MALICIOUS_SIERRA, compiled_class_hash=C_malicious, sender=Attacker)
