### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the caller-supplied `class_hash` is a declared (non-zero) class before committing the state update. This is directly analogous to the reported Fraxlend bug: just as `_deposit()` makes an external call without checking `externalAssetVault != address(0)`, `execute_replace_class` updates contract state without checking `class_hash != 0` (or that the hash is declared). The result is that any contract can permanently replace its own class with an invalid hash, irreversibly bricking itself and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads `class_hash` directly from the syscall request and immediately writes it into the contract state dictionary — with no check that the hash is non-zero or corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The inline `TODO` comment explicitly acknowledges the missing validation. The OS unconditionally accepts any felt value — including `0` or any undeclared hash — as the new class for the contract. [1](#0-0) 

Compare this to the pattern used elsewhere in the OS. For example, `execute_call_contract` reads the class hash from state before dispatching, and the OS is expected to enforce that class hashes in state are valid declared classes. The `replace_class` path is the only place where an arbitrary caller-supplied hash bypasses this invariant entirely. [2](#0-1) 

---

### Impact Explanation

Once a contract's `class_hash` is set to `0` (or any undeclared hash), every subsequent call to that contract will fail at entry-point dispatch — the OS cannot locate any entry points for class `0`. The state update is committed to the Merkle trie and is irreversible. Any ERC-20 balances, liquidity positions, or other assets held by the contract are permanently inaccessible.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is callable by any Cairo 1 contract on itself. A malicious contract author who has attracted user deposits (e.g., a DeFi vault or token contract) can call `replace_class(0)` in a single transaction to permanently brick the contract and freeze all deposited funds. No privileged role, leaked key, or external dependency is required — only the ability to deploy and invoke a contract, which is available to any unprivileged transaction sender.

**Likelihood: Medium** — The attack requires a contract that (a) holds third-party funds and (b) is authored or controlled by the attacker, but this is a realistic DeFi scenario on StarkNet.

---

### Recommendation

Add an explicit validation in `execute_replace_class` that the supplied `class_hash` is non-zero and corresponds to a previously declared class (i.e., it exists in `contract_class_changes` with a non-zero compiled class hash). This mirrors the guard that should exist before any critical state-mutating operation, analogous to the `address(0)` check recommended in the Fraxlend report:

```cairo
let class_hash = request.class_hash;
// Ensure the class hash is non-zero and declared.
assert_not_zero(class_hash);
// Additionally verify class_hash exists in contract_class_changes (declared classes dict).
``` [3](#0-2) 

---

### Proof of Concept

1. Attacker deploys a Cairo 1 contract `VaultAttack` that accepts user deposits (ERC-20 `transfer_from`) and exposes a public `drain()` entry point.
2. Users deposit funds into `VaultAttack`, trusting it as a vault.
3. Attacker calls `drain()`, which internally issues the `replace_class` syscall with `class_hash = 0`.
4. The OS executes `execute_replace_class` at lines 877–916 of `syscall_impls.cairo`. No validation is performed; `dict_update` writes `StateEntry(class_hash=0, ...)` for `VaultAttack`'s address.
5. The state update is committed to the Patricia trie in `state_update` (`state/state.cairo`).
6. All subsequent calls to `VaultAttack` fail at entry-point dispatch (class `0` has no entry points). User funds are permanently frozen. [2](#0-1) [4](#0-3)
