### Title
Token Vault Permanently Frozen via Unvalidated Mint `freeze_authority` in `log_metadata` — (File: `solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

The `LogMetadata` instruction registers any native Solana token for bridging and creates the bridge vault for that token. It does not validate that the mint's `freeze_authority` is absent. A token deployer who retains `freeze_authority` can freeze the bridge vault after users have deposited funds, permanently locking all bridged assets.

---

### Finding Description

In `log_metadata.rs`, the `LogMetadata` accounts struct accepts any mint whose `mint_authority` is not the bridge authority, and unconditionally creates (or reuses) the vault PDA:

```rust
// log_metadata.rs lines 41-62
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,

#[account(
    init_if_needed,
    payer = common.payer,
    token::mint = mint,
    token::authority = authority,
    seeds = [VAULT_SEED, mint.key().as_ref()],
    bump,
    token::token_program = token_program,
)]
pub vault: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no constraint of the form `mint::freeze_authority = COption::None`. The SPL token program allows the `freeze_authority` of a mint to freeze **any** token account associated with that mint, including program-derived vault accounts. Setting `token::authority = authority` on the vault only controls transfer authority; it does not prevent the mint's `freeze_authority` from calling `freeze_account` on the vault.

Once the vault is frozen:

- `init_transfer.rs` line 90–102: the `transfer_checked` CPI from the user's account into the vault will fail.
- `finalize_transfer.rs` line 103–116: the `transfer_checked` CPI from the vault to the recipient will fail. [2](#0-1) [3](#0-2) 

Both the native-token deposit path and the withdrawal path are blocked, with no recovery mechanism in the bridge program.

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds in the bridge vault.**

Any user who has deposited a native Solana token into the vault before the freeze is applied loses access to those funds permanently. The bridge program has no `unfreeze_account` instruction and no admin escape hatch for frozen vaults. The vault PDA is owned by the token program, so the bridge authority cannot unilaterally unfreeze it; only the mint's `freeze_authority` can do so, and a malicious deployer has no incentive to cooperate.

---

### Likelihood Explanation

`log_metadata` is a permissionless instruction — any caller can register any mint. The only guard is `!mint.mint_authority.contains(authority.key)`, which prevents registering bridge-deployed wrapped tokens, not externally-issued tokens with a freeze authority. [4](#0-3) 

A malicious token deployer follows a straightforward path:
1. Deploy a Token-2022 (or classic SPL) mint retaining `freeze_authority`.
2. Call `log_metadata` to register the token and create the vault (permissionless).
3. Attract users to bridge the token via `init_transfer`.
4. Call `freeze_account` on the vault PDA.
5. All deposited funds are permanently locked; `finalize_transfer` reverts for every subsequent withdrawal attempt.

This requires no privileged bridge role, no MPC collusion, and no chain-level attack — only control of the `freeze_authority` of a self-issued token.

---

### Recommendation

Add a `freeze_authority` absence constraint on the mint in `LogMetadata`:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    constraint = mint.freeze_authority == COption::None
        @ ErrorCode::FreezeAuthorityNotAllowed,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

For Token-2022 mints, additionally inspect the raw extension list for dangerous extensions (`PermanentDelegate`, `MintCloseAuthority`, `TransferHook`) before creating the vault, since these can similarly render the vault inoperable or allow unauthorized token movement.

---

### Proof of Concept

1. Deploy a Token-2022 mint with `freeze_authority = attacker_keypair.pubkey()`.
2. Call `log_metadata` with this mint — succeeds because `mint_authority ≠ bridge_authority`.
3. User calls `init_transfer` depositing 1000 tokens into the vault — succeeds.
4. Attacker calls `spl_token_2022::instruction::freeze_account(vault_pda, mint, attacker_keypair)` — succeeds because attacker holds `freeze_authority`.
5. User calls `finalize_transfer` to withdraw — the `transfer_checked` CPI fails with `AccountFrozen`.
6. The 1000 tokens are permanently locked in the vault with no recovery path in the bridge program.
