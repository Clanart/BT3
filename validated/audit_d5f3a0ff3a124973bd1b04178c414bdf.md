### Title
Tokens Permanently Stuck in NEAR Bridge Contract When `ft_transfer_call` Recipient Returns Unused Tokens - (File: `near/omni-bridge/src/lib.rs`)

### Summary

When `fin_transfer` is executed on the NEAR bridge with a non-empty `msg` for a non-deployed (locked) token, the bridge calls `ft_transfer_call` on the token contract. Under NEAR's NEP-141 standard, the recipient's `ft_on_transfer` callback may return a non-zero "unused" amount, which the token contract automatically refunds back to the bridge. The bridge has no callback registered to handle these returned tokens, and no user-facing recovery mechanism exists. The cross-chain transfer nonce is already consumed, so the tokens are permanently irrecoverable by the original sender.

### Finding Description

In `near/omni-bridge/src/lib.rs`, the `send_tokens` helper function handles the case where a non-deployed token is sent with a non-empty message by calling `ft_transfer_call`:

```rust
// near/omni-bridge/src/lib.rs ~L2107-2117
} else {
    require!(
        ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
        BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
    );

    ext_token::ext(token)
        .with_attached_deposit(ONE_YOCTO)
        .with_static_gas(ft_transfer_call_gas)
        .ft_transfer_call(recipient, amount, None, msg.to_string())
}
```

Under NEP-141, `ft_transfer_call` works as follows:
1. The token contract transfers `amount` from the bridge to `recipient`.
2. The token contract calls `recipient.ft_on_transfer(bridge, amount, msg)`.
3. The recipient returns `unused_amount` (tokens it did not consume).
4. The token contract automatically refunds `unused_amount` back to the bridge.

The `send_tokens` function returns the `Promise` from `ft_transfer_call` with **no `.then()` callback** to inspect or act on the returned `unused_amount`. The bridge therefore has no awareness of how many tokens were refunded. The destination nonce is already marked as used before `send_tokens` is called (in `fin_transfer`), so the transfer cannot be re-finalized. The refunded tokens accumulate silently in the bridge's token balance with no user-facing claim or recovery path.

### Impact Explanation

Any tokens returned by the recipient's `ft_on_transfer` are permanently locked in the bridge contract. The originating cross-chain transfer is already finalized (nonce consumed), so the user on the source chain has already burned or locked their tokens. There is no mechanism for the user to reclaim the stuck tokens on NEAR, and no admin sweep function is visible in the production scope. This constitutes an irrecoverable loss of user funds matching the **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds"** impact class.

### Likelihood Explanation

This is reachable by any unprivileged bridge user who:
1. Initiates a cross-chain transfer to NEAR with a non-empty `message` field targeting a non-deployed (locked) token.
2. Specifies a NEAR recipient that is a contract implementing `ft_on_transfer` (e.g., any DeFi protocol, AMM, or vault).

Many legitimate NEAR DeFi contracts return unused tokens from `ft_on_transfer` when they cannot process the swap/deposit (e.g., slippage exceeded, pool full). This is standard NEP-141 behavior and does not require any malicious intent from the recipient contract. The user may simply be trying to bridge-and-swap in one step.

### Recommendation

After `ft_transfer_call` completes, chain a `.then()` callback that reads the returned `unused_amount` from the promise result. If `unused_amount > 0`, either:
- Re-transfer the unused tokens to the original recipient via a plain `ft_transfer` (no message), or
- Emit an event and store a claimable balance so the user can retrieve the unused tokens in a subsequent call.

This mirrors the standard pattern used by NEAR DEX contracts that wrap `ft_transfer_call` with a resolver callback.

### Proof of Concept

1. User on EVM locks 1000 USDC and initiates a transfer to NEAR with `message = '{"action":"swap","min_out":"950"}'`, recipient = a NEAR AMM contract.
2. Relayer calls `fin_transfer` on the NEAR bridge. The nonce is marked used. `send_tokens` is called.
3. `send_tokens` calls `token.ft_transfer_call(amm_contract, 1000, None, msg)`.
4. The AMM's `ft_on_transfer` attempts the swap but slippage is exceeded; it returns `unused_amount = 1000`.
5. The token contract refunds 1000 USDC to the bridge contract.
6. No callback fires on the bridge side. The bridge's USDC balance increases by 1000 silently.
7. The user's EVM-side tokens are already burned/locked. The NEAR-side nonce is consumed. The 1000 USDC are permanently stuck in the bridge with no recovery path. [1](#0-0) [2](#0-1)
