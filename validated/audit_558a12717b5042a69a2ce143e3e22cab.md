### Title
Hardcoded `L1_HANDLER_L2_GAS_MAX_AMOUNT` Gas Ceiling Permanently Freezes L1→L2 Message Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`L1_HANDLER_L2_GAS_MAX_AMOUNT` is hardcoded at `100_000_000` gas units in `constants.cairo` and is applied unconditionally as the sole gas budget for every L1 handler execution. L1→L2 message senders have no mechanism to specify a higher gas limit. If a deployed contract's L1 handler requires more gas than this ceiling, the message can never be consumed by the OS, permanently freezing any funds locked in the L1→L2 message queue for L1 contracts that do not implement a cancellation path.

---

### Finding Description

In `constants.cairo`, the gas ceiling for L1 handler execution is a compile-time constant: [1](#0-0) 

In `execute_l1_handler_transaction`, this constant is assigned directly as `remaining_gas` with no user-supplied override: [2](#0-1) 

The L1 handler `TxInfo` is constructed with empty resource bounds (`resource_bounds_start = cast(0, ResourceBounds*)`, `resource_bounds_end = cast(0, ResourceBounds*)`), confirming that no per-message gas specification is possible at the protocol level: [3](#0-2) 

`non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0`; if the handler runs out of gas the assertion fails and the proof is invalid: [4](#0-3) 

To avoid an invalid proof, the sequencer is forced to pre-mark the transaction as reverted via the `%{ IsReverted %}` hint. The early-return path at lines 387–390 exits **before** `consume_l1_to_l2_message` is called, so the message is never removed from the queue: [5](#0-4) 

Contrast this with account transactions, where the user-supplied `resource_bounds[L2_GAS_INDEX].max_amount` drives `initial_user_gas_bound` and the OS merely caps it: [6](#0-5) 

L1 handlers have no equivalent user-controlled field.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any L1→L2 message whose handler's actual gas consumption exceeds `L1_HANDLER_L2_GAS_MAX_AMOUNT` will be permanently stuck in the message queue. The sequencer must mark it reverted to produce a valid proof; the OS then skips message consumption; and the message can never be retried successfully because the gas ceiling is a compile-time constant baked into the OS program hash. For L1 bridge contracts that do not implement a `startL1ToL2MessageCancellation` / `cancelL1ToL2Message` flow, the deposited assets are irrecoverable.

---

### Likelihood Explanation

**Medium.** The entry path is fully unprivileged: any L1 address (EOA or contract) can send an L1→L2 message. A contract deployer can publish a L2 contract whose `@l1_handler` is deliberately or accidentally gas-intensive (e.g., large storage writes, complex Sierra loops, or deep call chains). Once such a contract exists and a user sends value to it via an L1→L2 message, the funds are frozen with no OS-level remedy. The 100 M gas ceiling, while generous for simple token-deposit handlers, is a fixed protocol constant that cannot adapt to more complex handler logic.

---

### Recommendation

1. Allow the L1 message sender to embed a `l2_gas_limit` field in the L1→L2 message header, and have the OS read it instead of the hardcoded constant (analogous to how account transactions use `resource_bounds[L2_GAS_INDEX].max_amount`).
2. If a per-message gas field is not feasible, expose `L1_HANDLER_L2_GAS_MAX_AMOUNT` as a governance-upgradeable OS config parameter rather than a compile-time constant, so it can be raised without requiring a full OS program upgrade.
3. Ensure the StarkNet core L1 contract enforces a cancellation window and documents it clearly so that L1 bridge integrators implement the cancellation path.

---

### Proof of Concept

1. Deploy a L2 contract whose `@l1_handler` performs `> 100_000_000` gas of work (e.g., 1 000 000 storage writes at `STORAGE_WRITE_GAS_COST = 44_970` each ≈ 44.97 B gas — well above the ceiling).
2. From L1, call `StarknetCore.sendMessageToL2(l2Contract, selector, payload)` with ETH or ERC-20 value attached.
3. The sequencer simulates the handler, finds it exceeds `L1_HANDLER_L2_GAS_MAX_AMOUNT`, and sets `IsReverted = TRUE`.
4. `execute_l1_handler_transaction` returns at line 389 before `consume_l1_to_l2_message` is reached.
5. The message remains unconsumed in every subsequent block; the deposited value is frozen.
6. If the L1 bridge contract has no `cancelL1ToL2Message` implementation, the funds are permanently lost.
