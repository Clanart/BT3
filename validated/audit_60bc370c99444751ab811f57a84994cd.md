### Title
`PriceVelocityGuardExtension` uses `block.number` for velocity measurement, making the guard bypassable on Arbitrum/Optimism — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` computes `blockDiff = block.number - prevBlock` to determine how much oracle mid-price movement is permitted per swap. On L2 networks (Arbitrum, Optimism), `block.number` does not advance with L2 block time; it reflects the L1 block number, which can jump by several units in a single L2 block at sync time. This inflates `blockDiff`, causing the guard to permit far more price movement than the pool admin intended, allowing a manipulated or stale oracle price to reach pool swaps.

---

### Finding Description

The velocity guard stores `lastUpdateBlock` as `uint64(block.number)` and computes the allowed price deviation as:

```
allowedSq = maxChange² × (1 + blockDiff)
```

where `blockDiff = block.number - lastUpdateBlock`. [1](#0-0) 

On Arbitrum, `block.number` returns the most recently synced L1 block number. L1 blocks are ~12 s apart; Arbitrum blocks are ~0.25 s apart. When the sequencer syncs, `block.number` can jump by N (e.g., 5) in a single Arbitrum block. A swap executed in that Arbitrum block sees `blockDiff = N`, so the guard permits `sqrt(1 + N)` times the intended per-block cap — e.g., `sqrt(6) ≈ 2.45×` instead of `sqrt(2) ≈ 1.41×` for a normal one-block gap.

The same state variable is written by `setLastMidPrice`, which also records `block.number`: [2](#0-1) 

The struct field `lastUpdateBlock` is typed `uint64` and is compared directly against `block.number`: [3](#0-2) 

---

### Impact Explanation

The `PriceVelocityGuardExtension` is the sole on-chain mechanism preventing a rapid oracle mid-price shift from reaching `beforeSwap`. When `blockDiff` is artificially large due to an L1 sync jump, the guard's `allowedSq` threshold is proportionally inflated. A price that should be rejected (e.g., 2.4× the per-block cap) passes the check. The pool then executes swaps at the manipulated bid/ask, causing traders to receive worse prices than the oracle/bin curve permits and LP assets to be drained through the price discrepancy. This matches the allowed impact: **bad-price execution — an unclamped bid/ask quote reaches a pool swap**.

---

### Likelihood Explanation

Arbitrum is a primary deployment target for AMM periphery contracts. The L1 block sync event is deterministic and observable on-chain; any actor monitoring the sequencer can time a swap to coincide with a sync that produces a large `blockDiff`. No privileged access is required — any caller of `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on `MetricOmmSimpleRouter` triggers `beforeSwap` through the pool, which invokes the extension. [4](#0-3) 

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Rename `lastUpdateBlock` → `lastUpdateTime`, `maxChangePerBlockE18` → `maxChangePerSecondE18`, and restate the invariant in seconds. `block.timestamp` is reliable on both Arbitrum and Optimism for short-term time measurement, whereas `block.number` is explicitly documented as unreliable for sub-minute windows on L2.

---

### Proof of Concept

**Setup:** Pool on Arbitrum with `PriceVelocityGuardExtension` attached, `maxChangePerBlockE18 = 0.01e18` (1 % per block).

1. At L1 block B, a swap occurs. `lastUpdateBlock = B`, `lastMidPriceX64 = P`.
2. The Arbitrum sequencer does not sync for 60 s (5 L1 blocks). During this time, `block.number` stays at B on Arbitrum.
3. At the next sync, `block.number` jumps to B+5 within a single Arbitrum block.
4. An attacker (or a colluding oracle updater) submits a swap with a new oracle price `P' = P × 1.024` (2.4 % move — above the 1 % per-block cap).
5. The guard computes:
   - `blockDiff = (B+5) - B = 5`
   - `allowedSq = (0.01e18)² × (1 + 5) = 6 × 10^{-4} e36`
   - `changeE18 = 0.024e18`; `actualSq = (0.024e18)² = 5.76 × 10^{-4} e36`
   - `actualSq (5.76e-4) ≤ allowedSq (6e-4)` → **guard passes**
6. On mainnet with a true `blockDiff = 1`:
   - `allowedSq = (0.01e18)² × 2 = 2 × 10^{-4} e36`
   - `actualSq (5.76e-4) > allowedSq (2e-4)` → **guard reverts**
7. The pool executes the swap at the 2.4 % inflated price; LPs absorb the loss. [5](#0-4)
