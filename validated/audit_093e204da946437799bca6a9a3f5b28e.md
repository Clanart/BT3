### Title
Stale `feemarketParams` Snapshot in `NewDynamicFeeChecker` Causes Incorrect Base Fee Enforcement for Cosmos SDK Transactions — (`ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` captures `feemarketParams *feemarkettypes.Params` as a pointer at ante-handler construction time. Every invocation of the returned closure reads `feemarketParams.BaseFee` from that frozen in-memory struct. Because `SetBaseFee` writes only to the KV store (creating a new `Params` value each time), the in-memory pointer is never updated. The result is that the base-fee check and the effective-fee deduction for Cosmos SDK transactions always use the initial base fee, not the current one.

---

The pointer stored in the closure still points to the original struct from construction time. The same pattern affects `MinGasPriceDecorator`, which stores `feemarketParams *feemarkettypes.Params` and reads `mpd.feemarketParams.MinGasPrice` instead of querying the keeper:

```go

``` [4](#0-3) 

The stale `baseFee` is then used for both the sufficiency check and the effective-fee computation:

```go

The live keeper path (`feeMarketKeeper.GetBaseFee(ctx)`) is available and used correctly for EVM transactions in `CheckEthGasConsume` → `VerifyFee`, but is never consulted by `NewDynamicFeeChecker`. [6](#0-5) 

---

### Impact Explanation

When the base fee has risen above the stale snapshot value (which happens whenever blocks are consistently above the gas target), a Cosmos SDK transaction with:

```
feeCap = stale_baseFee  (< actual_baseFee)
```

passes the `feeCap.LT(baseFeeInt)` check and is committed. The `effectiveFee` deducted equals `stale_baseFee * gas`, which is less than the required `actual_baseFee * gas`. The fee collector receives fewer EVM-denom tokens than the protocol requires, constituting a direct mis-accounting of user fees. Over many blocks of high utilization the divergence compounds: `CalculateBaseFee` can increase the base fee by up to `baseFee / BaseFeeChangeDenominator` per block. [7](#0-6) 

---

### Likelihood Explanation

The base fee is recalculated at every `BeginBlock`. Any chain running with blocks above the gas target will have a continuously rising base fee. The stale snapshot diverges from the live value immediately after the first block that changes the base fee. The entry path is any unprivileged Cosmos SDK transaction routed through `NewDynamicFeeChecker` (i.e., any non-EVM Cosmos tx on a London-enabled chain). No special privileges are required.

---

### Recommendation

Replace the static `feemarketParams` pointer with a live keeper query inside the closure:

```go
// Instead of: baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
// Use:
baseFee := feeMarketKeeper.GetBaseFee(ctx)
if baseFee == nil { return checkTxFeeWithValidatorMinGasPrices(ctx, feeTx) }
```

Pass `feeMarketKeeper interfaces.FeeMarketKeeper` to `NewDynamicFeeChecker` and `MinGasPriceDecorator` instead of (or in addition to) the static params pointer, and always read dynamic values (`BaseFee`, `MinGasPrice`) from the keeper at call time.

---

### Proof of Concept

```
Block 1:  baseFee = 1_000_000_000  (feemarketParams snapshot captured here)
...
Block 500: baseFee = 2_000_000_000  (stored in KV, but feemarketParams.BaseFee still = 1_000_000_000)

Attacker submits Cosmos SDK tx:
  fee    = 1_500_000_000 * gasLimit
  feeCap = fee / gasLimit = 1_500_000_000

NewDynamicFeeChecker check:
  feeCap (1_500_000_000) >= stale baseFee (1_000_000_000) → PASSES
  effectivePrice = min(1_000_000_000 + tip, 1_500_000_000) = 1_000_000_000 (if tip=0)
  effectiveFee deducted = 1_000_000_000 * gasLimit

Required fee = 2_000_000_000 * gasLimit
Underpayment = 1_000_000_000 * gasLimit  (50% shortfall)
```

The transaction is committed and the fee collector receives half the required fee.
