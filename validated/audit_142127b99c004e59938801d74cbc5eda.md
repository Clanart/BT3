I have read the full `set_state_header` implementation and the `verify_path` / `compute_chunk_headers_root` implementations. Here is my analysis.

---

## Core Code Facts

**`compute_chunk_headers_root`** builds a merkle tree ordered by shard index: [1](#0-0) 

Each leaf is `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())`. Position 0 = shard 0, position 1 = shard 1, etc.

**`verify_path`** only checks that the item, combined with the supplied path, reproduces the root — it does **not** check the leaf's position: [2](#0-1) 

Compare with `verify_path_with_index`, which does enforce position: [3](#0-2) 

**`set_state_header` step 3a** uses the position-blind `verify_path`: [4](#0-3) 

**`validate_chunk_proofs`** checks internal hash/tx/receipt consistency but never compares `chunk.shard_id()` against the `shard_id` parameter: [5](#0-4) 

**`set_state_header` stores the header** under the caller-supplied `shard_id`, not the chunk's embedded `shard_id`: [6](#0-5) 

**`set_state_part`** derives `state_root` from the stored header's chunk (shard X's root) and validates parts against it, while writing under `shard_id` = Y: [7](#0-6) 

**`set_state_finalize`** derives `shard_uid` from the `shard_id` parameter (Y), not from the chunk's embedded `shard_id`, and applies shard X's chunk transactions/state root to shard Y's `ShardUId`: [8](#0-7)

## Attack Feasibility

For the swap to pass **all five steps**:

| Step | Check | Passes with swapped chunk? |
|---|---|---|
| 1-2 | `validate_chunk_proofs` | Yes — shard 0's chunk is internally valid |
| 3a | `verify_path` vs `chunk_headers_root` | Yes — shard 0's chunk IS in the tree; position not checked |
| 3b | `verify_path` for prev_chunk | Yes — same reasoning |
| 4 | Receipt proofs for `shard_id=Y` | Yes — attacker supplies real receipt proofs for shard Y from public block data |
| 5 | `validate_state_root_node` vs chunk's `prev_state_root` | Yes — attacker supplies shard 0's valid state root node |

The receipt proof check at step 4 uses the `shard_id` **parameter** (Y), not the chunk's embedded `shard_id`: [9](#0-8) 

So the attacker constructs receipt proofs for shard Y independently of which chunk body is used — these are public chain data.

The chunk's signature is from the legitimate chunk producer for shard 0 and is never re-verified in `set_state_header` (signature verification lives in the shards manager, not here).

## Verdict

### Title
Missing shard-index position check in `set_state_header` allows cross-shard state header substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to shard X but is presented under `shard_id=Y`. All five validation steps pass because `verify_path` is position-blind and no code ever asserts `chunk.shard_id() == shard_id`. The header is persisted under `StateHeaderKey(Y, sync_hash)`, subsequent `set_state_part` calls validate parts against shard X's state root, and `set_state_finalize` installs shard X's trie under shard Y's `ShardUId` prefix.

### Finding Description
`verify_path` computes `compute_root_from_path(path, hash(item)) == root`. For a two-shard block the merkle tree has leaves `[H(shard0), H(shard1)]`. Shard 0's proof `[Right(H(shard1))]` satisfies `combine(H(shard0), H(shard1)) == chunk_headers_root`. Presenting shard 0's chunk with shard 0's proof to `set_state_header(shard_id=1, …)` passes step 3a because the root matches — the function never checks that the proof's leaf position equals `shard_layout.get_shard_index(shard_id)`. The same gap exists in step 3b for `prev_chunk`. No other guard in the function compares `chunk.shard_id()` to the `shard_id` parameter.

### Impact Explanation
A syncing node installs shard X's trie data under shard Y's `ShardUId` key. The stored `StateHeaderKey(Y, sync_hash)` contains shard X's `prev_state_root`. All subsequent `set_state_part` calls validate shard X's parts against shard X's root (they pass), and `set_state_finalize` applies shard X's transactions to shard X's state but writes the result to shard Y's chunk extra and flat storage. The node then operates with a permanently corrupted shard Y state, diverging from the canonical chain.

### Likelihood Explanation
Any peer that can respond to a state sync header request can mount this attack. State sync peers are not required to be validators. The attacker needs only public chain data: shard 0's chunk (broadcast on the network), its merkle proof (derivable from the block), and receipt proofs for shard Y (stored in the chain). No privileged access is required.

### Recommendation
In `set_state_header`, after step 3a, assert that the chunk's embedded `shard_id` matches the `shard_id` parameter:
```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```
Additionally, replace the position-blind `verify_path` call with `verify_path_with_index`, passing `shard_layout.get_shard_index(shard_id)?` as the expected leaf index, so the merkle proof is pinned to the correct position in the tree.

### Proof of Concept
In a two-shard test environment:
1. Obtain `shard_state_header_0 = get_state_response_header(shard_id=0, sync_hash)`.
2. Extract shard 0's chunk and its merkle proof from the block's `compute_chunk_headers_root()` at index 0.
3. Construct a `ShardStateSyncResponseHeaderV2` substituting shard 0's chunk and proof, but with valid receipt proofs for shard 1 (from the actual chain).
4. Call `set_state_header(shard_id=1, sync_hash, crafted_header)` — expect `Ok(())`.
5. Read back the stored header via `get_state_header(shard_id=1, sync_hash)` and assert `header.chunk_prev_state_root() == shard_0_prev_state_root`.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
