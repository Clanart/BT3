### Title
`MerkleBlob::batch_insert` Silently Accepts Duplicate Keys/Hashes, Corrupting DataLayer Merkle Tree State — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that the single-item `insert` function enforces. When the tree already has more than one leaf, **every** item in the batch is written directly to the blob via `insert_entry_to_blob` with no uniqueness check. When the tree has zero or one leaf, only the last two items (popped from the vector) go through the guarded `insert` path; all remaining items still bypass the check. Supplying a batch that contains a repeated `KeyId` or a repeated `Hash` silently succeeds (`Ok(())`), leaving the blob in a state where two leaf nodes share the same key or hash — an invariant the rest of the codebase assumes can never occur.

---

### Finding Description

`insert` (the single-item path) enforces uniqueness before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` takes a completely different code path for the bulk of its work:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        metadata: NodeMetadata { node_type: NodeType::Leaf, dirty: false },
        node: Node::Leaf(LeafNode {
            parent: Parent(None),
            hash,
            key,
            value,
        }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` writes raw bytes to the blob; it does **not** consult `block_status_cache.contains_key` or `block_status_cache.contains_leaf_hash`. The two-item bootstrap branch (lines 578-585) calls `self.insert` for at most two items, but the remainder of the batch — and the entire batch when `leaf_count > 1` — is written without any guard.

The analog to the external report is direct:

| External report | chia_rs analog |
|---|---|
| `Lock.claimGovFees` transfers funds to `Lock` (committed state) | `batch_insert` writes the first leaf(ves) to the blob (committed state) |
| `BondNFT.distribute` silently `return`s when `totalShares == 0 \|\| !allowedAsset` | `batch_insert` silently writes duplicate-key leaves via `insert_entry_to_blob` with no error |
| Funds are stuck in `Lock`, unrecoverable | Blob contains two leaves with the same `KeyId`/`Hash`; tree invariants are broken, root hash is wrong, proofs are invalid |

After the silent duplicate write, the `block_status_cache` key-to-index and hash-to-index maps are inconsistent with the blob. `get_proof_of_inclusion`, `get_node_by_hash`, and `calculate_lazy_hashes` all rely on those maps being accurate. The resulting `ProofOfInclusion` objects will either fail `valid()` or, worse, pass `valid()` while pointing to the wrong leaf — because `valid()` only checks internal chain consistency, not an externally-trusted root:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs  lines 40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // root_hash() is derived from the proof itself
}
```

---

### Impact Explanation

Matches **High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

A blob that contains two leaves sharing a `KeyId` has an undefined root hash (the hash depends on which duplicate the tree-building loop happened to place). Any downstream consumer that trusts the root hash (e.g., a DataLayer sync peer verifying a delta) receives a root that does not correspond to the canonical key-value set. Proofs of inclusion generated from the corrupted blob may pass `valid()` while asserting membership of a key at the wrong value, enabling a DataLayer node to prove invalid state to a peer.

---

### Likelihood Explanation

`batch_insert` is exposed directly to Python callers via `py_batch_insert` (lines 1503-1519) and is the primary bulk-write API used by the DataLayer. Any caller — including a DataLayer peer sending a delta — that supplies a batch with a repeated `KeyId` or `Hash` triggers the silent corruption. No privileged role is required; the Python binding accepts the vector as-is after only a length check.
