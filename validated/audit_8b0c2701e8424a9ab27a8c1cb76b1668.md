### Title
Missing Declared-Class Verification in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary `class_hash` value from the calling contract without verifying that the hash corresponds to a previously declared class. An attacker can replace a contract's class with an undeclared hash, rendering the contract permanently unexecutable and freezing any funds it holds. This is the direct StarkNet analog of the NFT report's root cause: a resource identifier is accepted without verifying the caller's right to use it or its existence.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The developer-acknowledged TODO at line 898 explicitly confirms the missing check: the OS does **not** verify that `request.class_hash` corresponds to any entry in `contract_class_changes` (the declared class registry). The new class hash is written directly into the contract's `StateEntry` without any existence check. [1](#0-0) 

When a future transaction calls this contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

If `class_hash` was never declared, `dict_read` returns 0 (the default uninitialized value), `find_element` fails to locate a compiled class fact for key `0`, and the entry point returns `ERROR_ENTRY_POINT_NOT_FOUND`. Every future call to the contract reverts permanently. [3](#0-2) 

The `replace_class` syscall is dispatched unconditionally for any executing contract: [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds ERC-20 tokens, ETH, or STRK can have its class replaced with an undeclared hash. Once replaced, no entry point on that contract can ever execute successfully again. All assets stored in the contract's storage become permanently inaccessible. There is no recovery path: the class hash is committed to the proven state, and no future transaction can call into the contract to withdraw funds.

---

### Likelihood Explanation

**Medium.** The attack requires the attacker to control a contract (deploy one or exploit a contract that exposes a path to `replace_class`). The attacker does not need any privileged role beyond being a normal transaction sender who deploys a contract. The attack is a single transaction. A realistic scenario is a malicious DeFi vault contract that accepts user deposits and then calls `replace_class(garbage_hash)` to freeze all deposited funds. The missing check is confirmed by the in-code TODO, indicating it is a known gap in the current implementation.

---

### Recommendation

Before writing the new `class_hash` into the contract's `StateEntry`, verify that the hash exists as a key in `contract_class_changes` (i.e., it has a non-zero compiled class hash entry). Concretely, add a check analogous to:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the existing pattern used in `execute_declare_transaction`, which enforces `assert_not_zero(compiled_class_hash)` before registering a class. [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` with a `deposit()` function and a `freeze()` function. `freeze()` internally calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (any value not declared on-chain).
2. Users call `deposit()`, sending STRK to `VaultContract`. Funds accumulate in the contract's storage.
3. Attacker sends a transaction invoking `VaultContract.freeze()`.
4. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`. No declared-class check is performed. The contract's `StateEntry.class_hash` is updated to `0xdeadbeef` in `contract_state_changes`.
5. The state transition is proven and committed to L1.
6. Any subsequent transaction calling `VaultContract` reaches `execute_entry_point`, which calls `dict_read(key=0xdeadbeef)` → returns `0`, then `find_element(key=0)` → fails to find a compiled class fact → returns `ERROR_ENTRY_POINT_NOT_FOUND`.
7. All user funds are permanently frozen with no recovery mechanism.
