Audit Report

## Title
Stranded Router ETH Consumed by Subsequent WETH Swaps, Causing Direct ETH Loss for Prior Depositors ŌĆö (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function reads `address(this).balance` globally with no per-transaction accounting. When a user sends excess `msg.value` to a payable swap function without calling `refundETH()`, the surplus ETH remains on the router indefinitely. Any subsequent WETH swap whose `amountIn` exceeds the router's current ETH balance will silently consume that stranded ETH, reducing the WETH pulled from the attacker's allowance. The original depositor loses their ETH with no recourse.

## Finding Description
In `PeripheryPayments.pay` (lines 73ŌĆō84), when `token == WETH` and `payer != address(this)`, the function reads the router's total native balance:

```solidity
uint256 nativeBalance = address(this).balance;
```

The partial branch (`0 < nativeBalance < value`) wraps all available ETH and pulls only the remainder from the payer:

```solidity
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```

`address(this).balance` is the **total** router balance ŌĆö it includes ETH from any prior transaction that was not refunded. There is no per-transaction ETH accounting.

**How ETH becomes stranded:** `exactInputSingle`, `exactOutputSingle`, `exactInput`, and `exactOutput` are all `payable` and accept arbitrary `msg.value` with no enforcement that `msg.value == amountIn`. When a user sends excess ETH (e.g., `{value: 3 ETH}` with `amountIn = 2 ETH`), the `pay` callback wraps exactly `amountIn` ETH (the `nativeBalance >= value` branch), leaving the surplus on the router. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses ŌĆö it does not prevent consumption of already-stranded ETH.

The test suite confirms this pattern: `refundETH()` is always an optional, separate multicall step, and excess ETH is explicitly left on the router when it is omitted.

**Exploit flow:**
1. User A calls `exactInputSingle{value: 3 ETH}` with `amountIn = 2 ETH` (WETH as tokenIn), omitting `refundETH()`. After the swap, 1 ETH remains on the router.
2. Attacker calls `exactInputSingle{value: 0}` with WETH as tokenIn, `amountIn = 2 ETH`.
3. Inside the callback, `pay(WETH, attacker, pool, 2e18)` is called. `nativeBalance = 1 ETH`.
4. The partial branch executes: wraps User A's 1 ETH ŌåÆ transfers 1 WETH to pool; pulls only 1 WETH from attacker.
5. Pool receives the correct 2 WETH ŌĆö no pool-level check reverts.
6. User A loses 1 ETH. Attacker pays 1 WETH instead of 2 WETH.

## Impact Explanation
Direct loss of user ETH principal by an unprivileged attacker. The loss is silent and undetectable from the pool's perspective because the pool invariant is satisfied. Any ETH stranded on the router is fully drainable by the next WETH swap whose `amountIn` exceeds the stranded amount. This constitutes a Critical/High direct loss of user funds above Sherlock thresholds.

## Likelihood Explanation
Sending excess ETH without `refundETH()` is a realistic user error, particularly for EOA callers who do not use multicall. All four swap entry points (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) are `payable` and impose no constraint that `msg.value == amountIn`. An attacker can monitor the mempool or on-chain state for any non-zero `address(router).balance` and immediately execute a WETH swap to drain it. The attack is repeatable, requires no special privileges, and is undetectable by the pool.

## Recommendation
Track the ETH belonging to the current transaction separately from any pre-existing balance. The simplest fix is to snapshot `address(this).balance - msg.value` at the start of each swap entry point and pass the per-transaction ETH amount into `pay` rather than re-reading `address(this).balance` inside the callback. Alternatively, automatically refund excess ETH at the end of each swap function, or revert if `address(this).balance` after the swap is non-zero and no `refundETH()` was called.

## Proof of Concept
```solidity
function test_strandedEthStolenByWethSwap() public {
    // Step 1: User A sends 3 ETH for a 2 ETH swap, forgets refundETH()
    vm.prank(userA);
    router.exactInputSingle{value: 3 ether}(
        ExactInputSingleParams({
            tokenIn: address(weth), amountIn: 2 ether, /* ... */
        })
    );
    // 1 ETH is now stranded on the router
    assertEq(address(router).balance, 1 ether);

    // Step 2: Attacker executes a 2 WETH swap with no ETH sent
    uint256 attackerWethBefore = weth.balanceOf(attacker);
    vm.prank(attacker);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({
            tokenIn: address(weth), amountIn: 2 ether, /* ... */
        })
    );

    // Attacker only spent 1 WETH, not 2
    assertEq(attackerWethBefore - weth.balanceOf(attacker), 1 ether);
    // Router ETH is now 0 ŌĆö User A's ETH was consumed
    assertEq(address(router).balance, 0);
}
```

The pool receives the full 2 WETH, so no pool-side check reverts. The loss is silent and undetectable from the pool's perspective.
