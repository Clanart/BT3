### Title
Unchecked Deploy Syscall Failure — Constructor Revert Silently Reported as Success - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` into the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. A calling contract that relies on `failure_flag` to determine deployment success will always observe success, even when the constructor silently failed and the contract was never actually deployed. Any funds subsequently routed to the "deployed" address are permanently frozen.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–555), after calling `deploy_contract`, the OS writes the syscall response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns `(retdata_size, retdata)` — the constructor's return data — but no `is_reverted` flag is surfaced. The TODO comment explicitly acknowledges that failure propagation is unimplemented. The `failure_flag` field in `ResponseHeader` is the sole mechanism by which a calling contract learns whether a `deploy` syscall succeeded or failed. Because it is always `0`, the calling contract has no protocol-level way to distinguish a successful deployment from a silently-failed one.

Compare this with `contract_call_helper`, which correctly propagates the revert status:

```cairo
if (is_reverted != FALSE) {
    assert retdata[retdata_size] = ERROR_ENTRY_POINT_FAILED;
    ...
}
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

`execute_deploy` lacks this same propagation.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A factory or escrow contract that calls the `deploy` syscall and then transfers tokens to the returned contract address (a standard pattern) will do so even when the constructor reverted and the contract was never deployed. The target address has `class_hash=0` (no code). Any ERC-20 balance assigned to that address is permanently inaccessible: no entry point exists to withdraw it, and the address cannot be called. The funds are frozen forever.

---

### Likelihood Explanation

The `deploy` syscall is a standard, publicly accessible StarkNet syscall. Any unprivileged contract deployer can:

1. Declare a class whose constructor reverts under attacker-controlled conditions (e.g., based on block number, caller, or any on-chain state).
2. Induce a legitimate factory/escrow contract to deploy that class.
3. The factory sees `failure_flag=0`, records the address as live, and routes funds to it.
4. Funds are frozen.

No privileged access, leaked keys, or operator cooperation is required. The attacker only needs to be a class declarer — an unprivileged role open to anyone on StarkNet.

---

### Recommendation

Propagate the constructor's revert status through `deploy_contract`'s return signature and write it into `failure_flag`, mirroring the pattern already used in `contract_call_helper`:

```cairo
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

The existing TODO comment (`// TODO(Yoni, 1/1/2026): support failures.`) confirms the developers are aware this is incomplete. It must be resolved before the syscall is safe for production use by arbitrary contracts.

---

### Proof of Concept

1. **Attacker** declares `MaliciousClass` whose constructor executes:
   ```
   if (get_caller_address() != ATTACKER_ADDRESS) { panic(); }
   ```
2. **Victim factory contract** calls `deploy(class_hash=MaliciousClass, ...)`.
3. Constructor panics (reverts). The OS's `deploy_contract` processes the revert via the revert log, undoing all state changes — the contract is never deployed.
4. `execute_deploy` writes `ResponseHeader(gas=..., failure_flag=0)` unconditionally. [3](#0-2) 
5. The factory reads `failure_flag=0`, concludes deployment succeeded, and calls `erc20.transfer(deployed_address, amount)`.
6. The ERC-20 balance is credited to an address with `class_hash=0`. No contract exists there. The funds are permanently frozen — matching the **Critical: Permanent freezing of funds** impact category.
