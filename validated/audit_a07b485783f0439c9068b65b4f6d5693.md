### Title
Insufficient Validation in EIP-712 Typed Data Decoding Allows Cross-Format Signature Acceptance Without Chain-ID Binding Check - (File: `ethereum/eip712/encoding.go`, `ethereum/eip712/encoding_legacy.go`, `crypto/ethsecp256k1/ethsecp256k1.go`)

### Summary

The `verifySignatureAsEIP712` function in `crypto/ethsecp256k1/ethsecp256k1.go` calls `GetEIP712BytesForMsg` and `LegacyGetEIP712BytesForMsg` to reconstruct the EIP-712 typed data from raw sign-doc bytes. Both reconstruction functions (`GetEIP712TypedDataForMsg` and `LegacyGetEIP712TypedDataForMsg`) attempt to decode the sign-doc bytes as **either** Amino JSON **or** Protobuf, accepting whichever decodes first without verifying that the decoded chain-ID matches the chain-ID embedded in the outer Cosmos SDK signing context. The chain-ID used to build the EIP-712 domain separator is taken entirely from the attacker-controlled sign-doc bytes, not from the consensus-validated `signerData.ChainID`. This is the direct analog of the `get_body_hash` report's Step 2/3 leniency: a field is checked for syntactic presence but not verified to be the *correct* value in the *correct* context.

### Finding Description

**Root cause — `verifySignatureAsEIP712` in `crypto/ethsecp256k1/ethsecp256k1.go`:**

```go
func (pubKey PubKey) verifySignatureAsEIP712(msg, sig []byte) bool {
    eip712Bytes, err := eip712.GetEIP712BytesForMsg(msg)
    ...
    if pubKey.verifySignatureECDSA(eip712Bytes, sig) {
        return true
    }
    legacyEIP712Bytes, err := eip712.LegacyGetEIP712BytesForMsg(msg)
    ...
    return pubKey.verifySignatureECDSA(legacyEIP712Bytes, sig)
}
```

`msg` here is the raw sign-doc bytes passed in by the Cosmos SDK's standard `SigVerificationDecorator`. The function reconstructs the EIP-712 typed data entirely from those bytes, with no external anchor to the consensus chain-ID.

**Inside `GetEIP712TypedDataForMsg` (`ethereum/eip712/encoding.go:65-78`):**

```go
func GetEIP712TypedDataForMsg(signDocBytes []byte) (apitypes.TypedData, error) {
    typedDataAmino, errAmino := decodeAminoSignDoc(signDocBytes)
    if errAmino == nil && isValidEIP712Payload(typedDataAmino) {
        return typedDataAmino, nil
    }
    typedDataProtobuf, errProtobuf := decodeProtobufSignDoc(signDocBytes)
    if errProtobuf == nil && isValidEIP712Payload(typedDataProtobuf) {
        return typedDataProtobuf, nil
    }
    ...
}
```

The function tries Amino first, then Protobuf. The chain-ID used to build the EIP-712 domain separator is extracted from the sign-doc itself:

- In `decodeAminoSignDoc` (`encoding.go:118-124`): `chainID, err := types.ParseChainID(aminoDoc.ChainID)` → `WrapTxToTypedData(chainID.Uint64(), signDocBytes)`
- In `decodeProtobufSignDoc` (`encoding.go:187-210`): `chainID, err := types.ParseChainID(signDoc.ChainId)` → `WrapTxToTypedData(chainID.Uint64(), signBytes)`

**The same pattern exists in `LegacyGetEIP712TypedDataForMsg` (`encoding_legacy.go:55-68`)** with `legacyDecodeAminoSignDoc` and `legacyDecodeProtobufSignDoc`, both extracting chain-ID from the sign-doc bytes.

**The critical gap:** Neither `GetEIP712TypedDataForMsg` nor `LegacyGetEIP712TypedDataForMsg` receives the expected chain-ID from the calling context. The `verifySignatureAsEIP712` function passes only `msg` (the raw sign-doc bytes) and `sig`. There is no parameter for the expected chain-ID, and no comparison is made between the chain-ID parsed from the sign-doc and the chain-ID from `signerData.ChainID` (which is the consensus-validated chain-ID from `ctx.ChainID()`).

**Contrast with the `LegacyEip712SigVerificationDecorator` path** (`ante/cosmos/eip712.go:228-229`), which *does* check `extOpt.TypedDataChainID != signerChainID.Uint64()`. The `verifySignatureAsEIP712` path used by the standard Cosmos SDK `SigVerificationDecorator` has no equivalent check.

**The dual-format acceptance leniency:** `GetEIP712TypedDataForMsg` accepts the sign-doc as either Amino or Protobuf without requiring the caller to specify which format is expected. An attacker can craft a sign-doc that is valid Amino JSON but embeds a different chain-ID than the one the node is running on. The Amino decode succeeds, the chain-ID from the attacker-controlled bytes is used to build the domain separator, and the signature is verified against that attacker-chosen domain.

### Impact Explanation

This is a **High** severity EIP-712 authorization bypass. An attacker who holds a valid EIP-712 signature over a Cosmos transaction on chain A (e.g., a testnet or another EVM chain with the same message content) can replay that transaction on chain B by submitting the sign-doc bytes with chain A's chain-ID embedded. The `verifySignatureAsEIP712` path will reconstruct the EIP-712 typed data using chain A's chain-ID (from the attacker-controlled bytes), compute the same domain separator the attacker signed, and accept the signature as valid — even though the node is running chain B.

This enables cross-chain replay of EIP-712-signed Cosmos transactions (e.g., `MsgSend`, governance votes, staking operations) without the signer's consent on the target chain, constituting an unauthorized account action / forged execution.

### Likelihood Explanation

The `verifySignatureAsEIP712` path is reached whenever an `ethsecp256k1` public key's `VerifySignature` is called and the direct ECDSA check fails. This is the standard path for EIP-712-signed Cosmos SDK transactions processed by the non-legacy ante handler chain (i.e., `newCosmosAnteHandler` with `ExtensionOptionDynamicFeeTx`). Any unprivileged user can submit a crafted transaction. The attacker needs only a valid EIP-712 signature from another chain with the same message content, which is obtainable by self-signing on a testnet or another EVM chain.

### Recommendation

1. **Pass the expected chain-ID into `GetEIP712BytesForMsg` / `LegacyGetEIP712BytesForMsg`** and assert that the chain-ID parsed from the sign-doc matches the expected value before using it to build the domain separator.

2. **In `verifySignatureAsEIP712`**, the function currently has no access to the expected chain-ID. The signature should be refactored so that the expected chain-ID (from `signerData.ChainID` or the consensus context) is threaded through to the EIP-712 reconstruction functions.

3. **Alternatively**, after reconstructing the typed data, extract the `Domain.ChainId` from the result and compare it against the expected chain-ID before returning the typed data.

4. **Tighten `isValidEIP712Payload`** to also validate that the domain chain-ID is non-zero and matches the expected value, analogous to the report's recommendation to tighten the `body_hash_index` bounds check.

### Proof of Concept

1. Attacker controls account `A` on chain `ethermint_9000-1` (target) and chain `ethermint_9001-1` (source).
2. Attacker constructs a `MsgSend` from account `A` to account `B` on chain `ethermint_9001-1` and signs it using EIP-712 (Amino JSON sign-doc with `"chain_id": "ethermint_9001-1"`). This produces a valid signature `sig`.
3. Attacker submits to chain `ethermint_9000-1` a Cosmos SDK transaction containing the same `MsgSend`, with the sign-doc bytes containing `"chain_id": "ethermint_9001-1"` (the source chain).
4. The node calls `pubKey.VerifySignature(signDocBytes, sig)`.
5. `verifySignatureECDSA` fails (the direct Keccak256 hash doesn't match).
6. `verifySignatureAsEIP712` is called. It calls `GetEIP712BytesForMsg(signDocBytes)`.
7. `decodeAminoSignDoc` parses `"chain_id": "ethermint_9001-1"` from the attacker-controlled bytes and builds the EIP-712 domain with chain-ID `9001`.
8. The domain separator matches what the attacker signed on chain `9001`.
9. `verifySignatureECDSA(eip712Bytes, sig)` returns `true`.
10. The transaction is accepted on chain `ethermint_9000-1` — a cross-chain replay.

// Verifies the signature as an EIP-712 signature by first converting the message payload
// to EIP-712 object bytes, then performing ECDSA verification on the hash. This is to support
// signing a Cosmos payload using EIP-712.
func (pubKey PubKey) verifySignatureAsEIP712(msg, sig []byte) bool {
	eip712Bytes, err := eip712.GetEIP712BytesForMsg(msg)
	if err != nil {
		return false
	}

	if pubKey.verifySignatureECDSA(eip712Bytes, sig) {
		return true
	}

	// Try verifying the signature using the legacy EIP-712 encoding
	legacyEIP712Bytes, err := eip712.LegacyGetEIP712BytesForMsg(msg)
	if err != nil {
		return false
	}

	return pubKey.verifySignatureECDSA(legacyEIP712Bytes, sig)
}
