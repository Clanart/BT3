### Title
`initTransfer1155` Does Not Update `multiTokens` Mapping, Causing Permanent Lock of ERC1155 Tokens - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`initTransfer1155` locks ERC1155 tokens in the bridge but never writes to the `multiTokens` mapping. If `logMetadata1155` was not called beforehand, the mapping entry for the deterministic token address remains empty. When `finTransfer` is later called to release the tokens, it cannot find the ERC1155 mapping and falls through to an ERC20 call on a non-contract address, reverting permanently. The locked ERC1155 tokens have no recovery path.

### Finding Description

The `OmniBridge` contract uses a `multiTokens` mapping to track the relationship between a deterministic pseudo-address and the real ERC1155 contract address plus token ID. This mapping is the only mechanism by which `finTransfer` knows to call `IERC1155.safeTransferFrom` instead of an ERC20 transfer when releasing tokens to a recipient.

`logMetadata1155` is the only function that populates `multiTokens`: [1](#0-0) 

However, `initTransfer1155` — the function that actually locks ERC1155 tokens in the bridge — never writes to `multiTokens`. It computes the deterministic address, transfers the ERC1155 tokens in, and emits the event, but leaves the mapping untouched: [2](#0-1) 

There is no on-chain guard requiring that `multiTokens[deterministicToken]` is already populated before `initTransfer1155` is called. Any user can call `initTransfer1155` directly without a prior `logMetadata1155`.

When `finTransfer` is later called with `payload.tokenAddress = deterministicToken`, it reads `multiTokens[payload.tokenAddress]` and finds an empty struct (`tokenAddress == address(0)`). The ERC1155 branch is skipped, and execution falls through to the final `IERC20.safeTransfer` call on the deterministic address — which is not a deployed contract — causing a revert: [3](#0-2) 

There is no admin rescue function for arbitrary ERC1155 tokens held by the bridge. The only release path is through `finTransfer` with a valid MPC signature, which is permanently broken for this token.

### Impact Explanation

ERC1155 tokens deposited via `initTransfer1155` without a prior `logMetadata1155` are permanently and irrecoverably locked in the bridge contract. The NEAR side may mint bridged tokens to the user, but the user can never bridge back because every `finTransfer` attempt for that deterministic token address will revert. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation

The entry path is fully unprivileged. Any token holder with an ERC1155 token can call `initTransfer1155` directly. The function imposes no precondition that `logMetadata1155` was called first. A user who is unaware of the off-chain prerequisite, or who deliberately skips it, triggers the lock. The `deriveDeterministicAddress` function is `public pure`, so the deterministic address is trivially computable off-chain, making it easy to verify the mapping is empty before calling. [4](#0-3) 

### Recommendation

Add a check inside `initTransfer1155` that requires `multiTokens[deterministicToken].tokenAddress != address(0)` before accepting the ERC1155 deposit. Alternatively, have `initTransfer1155` itself populate the `multiTokens` mapping (mirroring the logic in `logMetadata1155`) so the mapping is always consistent with the actual token custody state.

### Proof of Concept

```solidity

//    ERC1155 tokens are transferred to the bridge.
bridge.initTransfer1155(erc1155Address, tokenId, amount, 0, 0, "recipient.near", "");

// 2. multiTokens[deterministicToken] is still empty:
//    bridge.multiTokens(deterministicToken).tokenAddress == address(0)  ✓ (confirmed empty)

// 4. Recipient on NEAR initiates bridge-back. MPC signs a finTransfer payload

// 5. finTransfer is called on EVM. multiTokens lookup returns empty struct.
//    Falls through to:
//      IERC20(deterministicToken).safeTransfer(recipient, amount)
//    deterministicToken is not a contract → REVERT.

// 6. ERC1155 tokens remain permanently locked in the bridge.
//    No admin function exists to recover them.
```

```text
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        address deterministicToken = deriveDeterministicAddress(
            tokenAddress,
            tokenId
        );

        IERC1155(tokenAddress).safeTransferFrom(
            msg.sender,
            address(this),
            tokenId,
            amount,
            ""
        );

        uint256 extensionValue = msg.value - nativeFee;

        initTransferExtension(
            msg.sender,
            deterministicToken,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            deterministicToken,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
```

```text
    function deriveDeterministicAddress(
        address tokenAddress,
        uint256 tokenId
    ) public pure returns (address) {
        return
            address(
                bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId)))
            );
    }
```
