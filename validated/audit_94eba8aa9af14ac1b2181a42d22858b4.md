### Title
Panic in `ProofOfSpace::update_digest` via `.expect()` on `quality_string()` with invalid v2 proof bytes — (`crates/chia-protocol/src/proof_of_space.rs`)

### Summary

`ProofOfSpace::update_digest` calls `self.quality_string().expect(...)` for v2 proofs (version=1). `quality_string()` returns `None` when the proof bytes are cryptographically invalid. `parse()` accepts any proof bytes without validating them. An unprivileged attacker can craft a `RewardChainBlock` containing a v2 `ProofOfSpace` with invalid proof bytes, which parses successfully but panics when any code path calls `update_digest` (e.g., `Streamable::hash()`), crashing the node.

### Finding Description

**Root cause — `update_digest` panics on invalid proof:** [1](#0-0) 

```rust
self.quality_string()
    .expect("internal error. Can't compute hash of invalid ProofOfSpace")
    .update_digest(digest);
```

**`quality_string()` returns `None` for invalid proof bytes:** [2](#0-1) 

The function calls `chia_pos2::quality_string_from_proof(...)` which returns `None` when the proof is cryptographically invalid. The `version != 1` guard does not help here because `update_digest` only reaches this branch when `version == 1`.

**`parse()` does NOT validate proof bytes for v2:** [3](#0-2) 

For version 1, `parse()` reads `plot_index`, `meta_group`, `strength`, and `proof` bytes, and only checks that exactly one of `pool_public_key`/`pool_contract_puzzle_hash` is set. The proof bytes themselves are accepted unconditionally.

**`RewardChainBlock::update_digest` delegates to `ProofOfSpace::update_digest`:** [4](#0-3) 

**`Streamable::hash()` calls `update_digest`:** [5](#0-4) 

Any call to `hash()` on a `RewardChainBlock` containing an invalid v2 proof triggers the panic.

### Impact Explanation

A Rust `panic!` / `.expect()` in a non-`catch_unwind` context terminates the thread or the process. In a consensus-critical hashing path (block hash computation), this crashes the node. The impact is **chain halt via node crash** — matching the Critical allowed scope: *"Valid unprivileged serialized network object can trigger deterministic consensus divergence, chain halt, or committed state corruption."*

### Likelihood Explanation

- The attacker only needs to craft a well-formed `RewardChainBlock` wire message with a v2 `ProofOfSpace` whose `proof` field contains arbitrary bytes (e.g., all zeros, wrong length, or random data).
- `parse()` accepts it without error.
- Any node that computes the block hash (e.g., during header validation, block record construction, or peer relay) will panic.
- No privileged access, leaked keys, or network-level position is required.

### Recommendation

Replace the `.expect()` with a proper error return. `update_digest` should not be able to panic on parsed input. Options:

1. Change `update_digest` to return `Result<(), Error>` (requires trait change).
2. Validate the proof at `parse()` time for v2 proofs and return `Err(Error::InvalidPoS)` if `quality_string_from_proof` returns `None`.
3. Cache the `quality_string` result at construction/parse time so that an invalid proof is rejected before a `ProofOfSpace` object is ever created.

Option 2 is the least invasive: add a `quality_string_from_proof` call inside `parse()` for `version == 1` and return `Err(Error::InvalidPoS)` if it returns `None`.

    pub fn quality_string(&self) -> Option<Bytes32> {
        if self.version != 1 {
            return None;
        }

        let k_size = (self.proof.len() * 8 / 128) as u8;
        let plot_id = self.compute_plot_id().to_bytes();
        chia_pos2::quality_string_from_proof(&plot_id, k_size, self.strength, self.proof.as_slice())
            .map(|quality| {
                let mut sha256 = Sha256::new();
                sha256.update(chia_pos2::serialize_quality(
                    &quality.chain_links,
                    self.strength,
                ));
                sha256.finalize().into()
            })
    }
