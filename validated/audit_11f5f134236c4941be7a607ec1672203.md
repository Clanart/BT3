### Title
`set_state_finalize_on_height` Applies Old Chunks Without Saving `StateTransitionData`, Blocking State Witness Generation After State Sync — (File: `chain/chain/src/chain_update.rs`)

---

### Summary

After state sync completes, `set_state_finalize_on_height` applies old (missing) chunks to rebuild the chain state but never persists the `StateTransitionData` (partial storage proof) for those blocks. When the node later attempts to produce a state witness for a chunk whose implicit transitions pass through those blocks, `get_state_transition()` fails with "Missing transition state proof", permanently blocking state witness generation for that shard.

---

### Finding Description

**The analog pattern.** The external bug is: Component A grants an allowance (registers a commitment) but never credits the underlying balance; Component B later tries to draw on that balance and reverts because it is zero. In nearcore the same structure appears: `set_state_finalize_on_height` saves the `ChunkExtra` (the state-root commitment) but never saves the `StateTransitionData` (the partial-storage proof that is the "balance" needed to produce a state witness).

**Normal block processing saves both.** In `process_apply_chunk_result`, when `should_save_state_transition_data` is true, the proof is extracted from `apply_result` and persisted before anything else:

```rust
if should_save_state_transition_data {
    let apply_result = &mut new_chunk_result.apply_result;
    self.chain_store_update.save_state_transition_data(
        *block_hash,
        shard_id,
        apply_result.proof.take(),          // ← proof saved
        apply_result.applied_receipts_hash,
        mem::take(&mut apply_result.contract_updates),
    );
}
``` [1](#0-0) 

The same guard exists for `OldChunk` results: [2](#0-1) 

**State-sync finalization omits the save entirely.** `set_state_finalize_on_height` applies old chunks during state sync finalization. It saves flat-state changes, trie changes, and `ChunkExtra`, but the `apply_result.proof` is simply dropped:

```rust
let apply_result = self.runtime_adapter.apply_chunk(
    RuntimeStorageConfig::new(*chunk_extra.state_root(), true),
    ...
)?;
// flat storage, trie changes, chunk extra saved …
// apply_result.proof is NEVER extracted or saved
``` [3](#0-2) 

The proof is recorded by the runtime (every `apply_chunk` call wraps the trie with `recording_reads_with_proof_size_limit`), so the data exists in memory but is silently discarded. [4](#0-3) 

**The consumer fails.** When the node later tries to produce a state witness, `collect_state_transition_data` iterates over every block between the previous chunk's height and the current chunk's `prev_block_hash`, calling `get_state_transition` for each: [5](#0-4) 

`get_state_transition` reads `DBCol::StateTransitionData` and returns a hard error if the row is absent:

```rust
let stored_chunk_state_transition_data = self
    .store()
    .get_ser(DBCol::StateTransitionData, &get_block_shard_id(block_hash, shard_id))
    .ok_or_else(|| {
        Error::Other(format!(
            "Missing transition state proof for block {block_hash} and shard {shard_id}"
        ))
    })?;
``` [6](#0-5) 

Because `set_state_finalize_on_height` never wrote those rows, every call to `create_state_witness` that spans those blocks returns an error, and no state witness is ever distributed for that shard.

`save_state_transition_data` itself is a no-op when `partial_storage` is `None`, so the omission is silent — no panic, no log at error level, just a missing DB row: [7](#0-6) 

---

### Impact Explanation

A validator that completes state sync for a shard and then becomes a chunk producer for that shard in the next epoch will fail to generate state witnesses for any chunk whose implicit-transition chain passes through blocks applied by `set_state_finalize_on_height`. Because stateless validation requires the chunk producer to distribute a valid state witness, the failure is permanent until the node re-syncs or the blocks age out of the witness-generation window. This effectively disables stateless validation for the affected shard on that node, which is a **Critical** impact: chunk production continues but validators cannot validate the produced chunks.

---

### Likelihood Explanation

The scenario is routine: validator rotation causes nodes to track new shards every epoch, which triggers state sync followed by `set_state_finalize`. Any node that (a) completes state sync for a shard and (b) is assigned as chunk producer for that shard in the immediately following epoch will hit this path. With increasing shard counts and frequent validator rotation, this becomes a near-certain occurrence for active validators.

---

### Recommendation

In `set_state_finalize_on_height`, extract and persist the proof from `apply_result` before discarding the result, mirroring the pattern in `process_apply_chunk_result`:

```rust
// After apply_chunk succeeds, before saving chunk_extra:
self.chain_store_update.save_state_transition_data(
    *block_header.hash(),
    shard_id,
    apply_result.proof.take(),
    apply_result.applied_receipts_hash,
    apply_result.contract_updates.clone(),
);
```

The same fix should be applied to `set_state_finalize` (the new-chunk path), which also calls `apply_chunk_postprocessing` without passing a `should_save_state_transition_data` flag. [8](#0-7) 

---

### Proof of Concept

1. Start a two-node network with stateless validation enabled.
2. Let node B fall behind by more than two epochs so it triggers state sync.
3. After state sync completes, `set_state_finalize` and `set_state_finalize_on_height` run for the synced shard.
4. Assign node B as chunk producer for that shard in the next epoch.
5. Node B calls `create_state_witness` → `collect_state_transition_data` → `get_state_transition` for a block applied by `set_state_finalize_on_height`.
6. `get_ser(DBCol::StateTransitionData, ...)` returns `None` because the row was never written.
7. `get_state_transition` returns `Error::Other("Missing transition state proof for block … and shard …")`.
8. `create_state_witness` propagates the error; no state witness is sent; chunk validators cannot validate the chunk.

The exact corrupted value is the absent `DBCol::StateTransitionData` row keyed by `(block_hash || shard_id)` for every block height in the range `[chunk.height_included + 1, sync_hash_height - 1]` that was processed by `set_state_finalize_on_height`. [9](#0-8) [10](#0-9)

    pub fn set_state_finalize_on_height(
        &mut self,
        height: BlockHeight,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<bool, Error> {
        let _span =
            tracing::debug_span!(target: "sync", "set_state_finalize_on_height", height, %shard_id)
                .entered();
        // Note that block headers are already synced and can be taken
        // from store on disk.
        let block_header_result = get_block_header_on_chain_by_height(
            &self.chain_store_update.chain_store(),
            &sync_hash,
            height,
        );
        if let Err(_) = block_header_result {
            // No such height, go ahead.
            return Ok(true);
        }
        let block_header = block_header_result?;
        if block_header.hash() == &sync_hash {
            // Don't continue
            return Ok(false);
        }
        let block = self.chain_store_update.get_block(block_header.hash())?;

        let prev_hash = block_header.prev_hash();
        let prev_block_header = self.chain_store_update.get_block_header(prev_hash)?;

        let shard_uid =
            shard_id_to_uid(self.epoch_manager.as_ref(), shard_id, block_header.epoch_id())?;
        let chunk_extra = self.chain_store_update.get_chunk_extra(prev_hash, &shard_uid)?;
        let memtrie_pin = self
            .runtime_adapter
            .get_tries()
            .maybe_pin_memtrie_root(shard_uid, *chunk_extra.state_root())?;

        let apply_result = self.runtime_adapter.apply_chunk(
            RuntimeStorageConfig::new(*chunk_extra.state_root(), true),
            ApplyChunkReason::UpdateTrackedShard,
            ApplyChunkShardContext {
                shard_uid,
                last_validator_proposals: chunk_extra.validator_proposals(),
                gas_limit: chunk_extra.gas_limit(),
                is_new_chunk: false,
                on_post_state_ready: None,
                memtrie_pin,
            },
            ApplyChunkBlockContext::from_header(
                &block_header,
                prev_block_header.next_gas_price(),
                block.block_congestion_info(),
                block.block_bandwidth_requests(),
            ),
            &[],
            SignedValidPeriodTransactions::empty(),
        )?;
        let flat_storage_manager = self.runtime_adapter.get_flat_storage_manager();
        let store_update = flat_storage_manager.save_flat_state_changes(
            *block_header.hash(),
            *prev_block_header.hash(),
            height,
            shard_uid,
            apply_result.trie_changes.state_changes(),
        )?;
        self.chain_store_update.merge(store_update.into());
        self.chain_store_update.save_trie_changes(*block_header.hash(), apply_result.trie_changes);

        // The chunk is missing but some fields may need to be updated
        // anyway. Prepare a chunk extra as a copy of the old chunk
        // extra and apply changes to it.
        let mut new_chunk_extra = ChunkExtra::clone(&chunk_extra);
        *new_chunk_extra.state_root_mut() = apply_result.new_root;
        self.chain_store_update.save_chunk_extra(
            block_header.hash(),
            &shard_uid,
            new_chunk_extra.into(),
        );
        Ok(true)
    }

    pub fn save_state_transition_data(
        &mut self,
        block_hash: CryptoHash,
        shard_id: ShardId,
        partial_storage: Option<PartialStorage>,
        applied_receipts_hash: CryptoHash,

            self.state_transition_data.insert(

                StoredChunkStateTransitionData::V1(StoredChunkStateTransitionDataV1 {

                    receipts_hash: applied_receipts_hash,
                    contract_accesses: contract_accesses.into_iter().collect(),
                    contract_deploys: contract_deploys.into_iter().map(|c| c.into()).collect(),
                }),
            );
        }
    }

    pub fn set_state_finalize(
        &mut self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<(), Error> {
        let shard_state_header = self.state_sync_adapter.get_state_header(shard_id, sync_hash)?;
        let mut height = shard_state_header.chunk_height_included();
        let mut chain_update = self.chain_update();
        let shard_uid = chain_update.set_state_finalize(shard_id, sync_hash, shard_state_header)?;
        chain_update.commit()?;

        // We restored the state on height `shard_state_header.chunk.header.height_included`.
        // Now we should build a chain up to height of `sync_hash` block.
        loop {
            height += 1;
            let mut chain_update = self.chain_update();
            // Result of successful execution of set_state_finalize_on_height is bool,
            // should we commit and continue or stop.
            if chain_update.set_state_finalize_on_height(height, shard_id, sync_hash)? {
                chain_update.commit()?;
            } else {
                break;
            }
        }

        let flat_storage_manager = self.runtime_adapter.get_flat_storage_manager();
        if let Some(flat_storage) = flat_storage_manager.get_flat_storage_for_shard(shard_uid) {
            let header = self.get_block_header(&sync_hash)?;
            flat_storage.update_flat_head(header.prev_hash()).unwrap();
        }

        Ok(())
    }
