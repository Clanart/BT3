### Title
Returned Tokens Permanently Locked in Bridge When UTXO Connector Rejects Transfer - (File: near/omni-bridge/src/btc.rs)

### Summary
When a trusted relayer submits a pending UTXO transfer to the BTC connector via `submit_transfer_to_utxo_chain_connector`, the transfer message is removed from `pending_transfers` before the cross-contract call. If the connector's `ft_on_transfer` returns a non-zero amount (i.e., rejects the transfer), the callback `submit_transfer_to_btc_connector_callback` sends only the fee to the fee recipient but never returns the user's principal tokens — which are now held by the bridge contract — to the original depositor. The original user's address (`transfer_owner`) is available in the callback but is unused in the failure branch, causing permanent irrecoverable lock of user funds.

### Finding Description

In `submit_transfer_to_utxo_chain_connector`, the transfer message is removed from `pending_transfers` before the `ft_transfer_call` to the UTXO connector:

```rust
self.remove_transfer_message(transfer_id);   // transfer_owner is extracted here

ext_token::ext(btc_account_id)
    .ft_transfer_call(self.get_utxo_chain_connector(chain_kind), amount, None, msg)
    .then(
        Self::ext(env::current_account_id())
            .submit_transfer_to_btc_connector_callback(
                transfer.message,
                transfer.owner,   // original user passed to callback
                fee_recipient,
            ),
    )
``` [1](#0-0) 

In the callback, the NEP-141 `ft_transfer_call` result is inspected. When `result.0 > 0` (the connector returned tokens, meaning it rejected the transfer), the code only sends the fee to the fee recipient and does nothing with the returned principal:

```rust
if matches!(call_result, Ok(result) if result.0 > 0) {
    let token_fee = transfer_msg.fee.fee.0;
    self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)
} else {
    // success or hard failure: re-add transfer message
    let required_storage_balance =
        self.add_transfer_message(transfer_msg, transfer_owner.clone());
    ...
}
``` [2](#0-1) 

The `transfer_owner` (the original user) is present in the callback signature but is only used in the `else` branch (success/hard-failure). In the soft-rejection branch (`result.0 > 0`), `transfer_owner` is completely ignored. The returned tokens accumulate in the bridge contract's balance with no recovery path.

This is the direct analog of the CLGauge bug: just as CLGauge had the original staker's address available but derived the reward recipient from current NFT ownership (the contract itself), here the bridge has the original depositor's address (`transfer_owner`) available but fails to use it to return the rejected tokens, leaving them permanently in the contract.

### Impact Explanation

**Impact: Critical — Permanent irrecoverable lock of user funds.**

When the UTXO connector soft-rejects a transfer (returns `result.0 > 0` from `ft_on_transfer` without panicking):

1. The bridge sent `amount = total - fee` tokens to the connector.
2. The connector returned `result.0` tokens (up to `amount`) back to the bridge via NEP-141 refund.
3. The bridge sends `fee` tokens to the fee recipient.
4. The remaining `result.0` tokens sit in the bridge's token balance permanently — the transfer message was already deleted, there is no `cancel_transfer` or admin-recovery function in scope, and `transfer_owner` is not used to refund.

The user loses their principal (up to `total - fee` UTXO-chain tokens) with no recourse.

### Likelihood Explanation

**Likelihood: Medium.**

- The UTXO connector is a semi-trusted component. Its `ft_on_transfer` can legitimately return a non-zero amount (soft rejection) if, for example, the BTC transaction cannot be constructed (invalid UTXO inputs, gas fee mismatch after RBF, connector-side validation failure, or transient state).
- The relayer validates the message format on the NEAR side (lines 38–68 of btc.rs), but cannot guarantee the connector will accept it — connector-side state is external.
- Any user who initiates a BTC withdrawal is at risk whenever the connector soft-rejects the submission. [3](#0-2) 

### Recommendation

In the soft-rejection branch of `submit_transfer_to_btc_connector_callback`, return the principal tokens to `transfer_owner` (and optionally the fee as well, since the transfer did not complete):

```diff
 if matches!(call_result, Ok(result) if result.0 > 0) {
-    let token_fee = transfer_msg.fee.fee.0;
-    self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)
+    // Connector rejected the transfer; re-add the pending message so the
+    // user retains their funds and can retry or cancel.
+    let required_storage_balance =
+        self.add_transfer_message(transfer_msg, transfer_owner.clone());
+    self.update_storage_balance(
+        transfer_owner,
+        required_storage_balance,
+        NearToken::from_yoctonear(0),
+    );
+    PromiseOrValue::Value(())
 } else {
```

Alternatively, if the fee should still be paid on soft-rejection, explicitly transfer `result.0` tokens back to `transfer_owner` before sending the fee.

### Proof of Concept

1. Alice holds 1 000 000 wBTC (NEAR-side BTC wrapper) and calls `ft_transfer_call` on the BTC token contract, sending 1 000 000 tokens to the bridge with an `InitTransfer` message targeting a BTC address. The bridge stores the transfer message with `transfer_owner = Alice`.

2. A trusted relayer calls `submit_transfer_to_utxo_chain_connector(transfer_id, msg, ...)`. The bridge:
   - Removes Alice's transfer message from `pending_transfers` (line 84).
   - Calls `ft_transfer_call(connector, 990_000, None, msg)` (amount minus fee).

3. The UTXO connector's `ft_on_transfer` encounters a validation error (e.g., UTXO inputs are already spent) and returns `U128(990_000)` — a soft rejection.

4. The NEP-141 token contract refunds 990 000 tokens back to the bridge. The bridge's `submit_transfer_to_btc_connector_callback` is invoked with `result.0 = 990_000`.

5. The condition `matches!(call_result, Ok(result) if result.0 > 0)` is true. The bridge sends `fee = 10_000` tokens to the fee recipient via `send_fee_internal`.

6. The 990 000 returned tokens remain in the bridge's wBTC balance. Alice's transfer message no longer exists. Alice has no mechanism to recover her 990 000 tokens. They are permanently locked. [4](#0-3)

    #[private]
    pub fn submit_transfer_to_btc_connector_callback(
        &mut self,
        transfer_msg: TransferMessage,
        transfer_owner: AccountId,
        fee_recipient: AccountId,
        #[callback_result] call_result: &Result<U128, PromiseError>,
    ) -> PromiseOrValue<()> {
        if matches!(call_result, Ok(result) if result.0 > 0) {
            let token_fee = transfer_msg.fee.fee.0;
            self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)
        } else {
            let required_storage_balance =
                self.add_transfer_message(transfer_msg, transfer_owner.clone());

            self.update_storage_balance(
                transfer_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            );

            PromiseOrValue::Value(())
        }
    }
