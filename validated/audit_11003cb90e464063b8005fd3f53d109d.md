### Title
Missing Duplicate Key/Hash Validation in `MerkleBlob::batch_insert` Silently Corrupts Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash precondition checks that `MerkleBlob::insert` enforces. When a caller supplies a batch containing a key or leaf-hash that already exists in the tree, the duplicate leaf is silently written to the blob, producing an internally inconsistent tree whose root hash is wrong and whose inclusion/exclusion proofs are forged.

---

### Finding Description

`MerkleBlob::insert` enforces two guards before writing any leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`MerkleBlob::batch_insert` only routes items through `insert` (and therefore through those guards) for the first two elements when the tree currently has ≤ 1 leaf. All remaining elements are written directly via `insert_entry_to_blob` with **no duplicate check**:

```rust
// lines 578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← guarded
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← NO guard
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // writes directly
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` writes the raw block bytes and updates `block_status_cache` via `add_leaf`. `add_leaf` calls `HashMap::insert`, which **silently overwrites** the existing cache entry for a duplicate key rather than returning an error:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);   // overwrites silently
}
``` [3](#0-2) 

The result is:

| Layer | State after duplicate batch insert |
|---|---|
| **Blob (on-disk)** | Two leaf blocks with the same `KeyId` |
| **`block_status_cache`** | Only the *last* duplicate is tracked; the first is orphaned |
| **Merkle root** | Computed over both leaves → wrong hash |
| **`check_integrity`** | Detects mismatch only if explicitly called |

The `check_integrity` function confirms this would be caught post-hoc, but only if invoked:

```rust
if leaf_count != key_to_index_cache_length {
    return Err(Error::IntegrityKeyToIndexCacheLength(…));
}
``` [4](#0-3) 

The regular `insert` path is protected and the double-insert test confirms the guard works there: [5](#0-4) 

But no equivalent test exists for `batch_insert` with duplicate keys.

---

### Impact Explanation

A corrupted Merkle tree produces an incorrect root hash. Any inclusion or exclusion proof generated from this tree is invalid. Clients or other DataLayer nodes that verify state against the root will accept forged proofs. This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`MerkleBlob` and `batch_insert` are public APIs exposed through the `chia-datalayer` crate and the Python wheel. A DataLayer operator or any caller that can supply a crafted batch (e.g., a batch containing a key already present in the tree, or two entries with the same key) can trigger silent corruption. No privileged role is required beyond the ability to call `batch_insert`.

---

### Recommendation

Before writing each leaf in the bulk loop, add the same guards that `insert` uses:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing insert_entry_to_blob logic …
}
```

Alternatively, collect all keys and hashes from the batch into a temporary `HashSet` before writing anything, and reject the entire batch if any duplicate is found.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn sha256_of(n: u8) -> Hash {
    use sha2::{Digest, Sha256};
    let mut h = Sha256::new();
    h.update([n]);
    Hash(Bytes32::new(h.finalize().into()))
}

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate so leaf_count > 1, bypassing the two-item insert() path
    blob.insert(KeyId(10), ValueId(10), &sha256_of(10), InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(11), ValueId(11), &sha256_of(11), InsertLocation::Auto {}).unwrap();

    // Batch with a key that already exists (KeyId(10)) — no error returned
    let batch = vec![
        ((KeyId(10), ValueId(99)), sha256_of(99)),  // duplicate key!
        ((KeyId(12), ValueId(12)), sha256_of(12)),
    ];
    blob.batch_insert(batch).unwrap();  // succeeds silently

    // Tree now has two leaf blocks for KeyId(10); root hash is wrong.
    // check_integrity() will panic/error, but the root was already exported wrong.
    blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheLength
}
```
