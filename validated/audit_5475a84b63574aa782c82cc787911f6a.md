### Title
OS `execute_replace_class` Accepts Undeclared Class Hash Without Validation, Diverging from Blockifier and Producing Wrong State Root - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The Starknet OS Cairo implementation of `execute_replace_class` does not validate that the target class hash is declared before updating contract state. The blockifier's Rust implementation enforces this check and reverts the transaction on failure. This divergence means the OS produces a different (wrong) state root for any block containing a transaction that calls `replace_class` with an undeclared class hash, breaking the proof invariant.

---

### Finding Description

**Blockifier (Rust) — validates class hash:**

In `crates/blockifier/src/execution/deprecated_syscalls/hint_processor.rs`, the deprecated path explicitly reads the compiled class to ensure it is declared before allowing the replacement:

```rust
fn replace_class(...) {
    // Ensure the class is declared (by reading it).
    syscall_handler.state.get_compiled_class(request.class_hash)?;
    syscall_handler.state.set_class_hash_at(..., request.class_hash)?;
    Ok(ReplaceClassResponse {})
}
```

The Cairo 1 path (via `SyscallHandlerBase::replace_class`) also enforces this, as confirmed by the test `undeclared_class_hash` in `crates/blockifier/src/execution/syscalls/syscall_tests/replace_class.rs` which asserts the error `"is not declared"` when an undeclared hash is supplied.

**OS Cairo (SNOS) — missing validation, acknowledged by TODO:**

In `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`, the `execute_replace_class` function directly updates `contract_state_changes` with the caller-supplied class hash, with no existence check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The deprecated OS path in `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deprecated_execute_syscalls.cairo` has the same omission — no class hash existence check before updating `contract_state_changes`.

**Divergence path:**

| Layer | `replace_class(undeclared_hash)` outcome |
|---|---|
| Blockifier (Rust) | Syscall fails → transaction reverts → state unchanged |
| OS Cairo (SNOS) | No check → syscall succeeds → `contract_state_changes` updated → transaction does NOT revert |

Because the OS does not fail the syscall, the transaction does not trigger the revert log rollback. The OS computes a state root that includes the undeclared class hash written into the contract's `StateEntry`, while the blockifier's committed state root does not include this change.

---

### Impact Explanation

**Impact: Critical — Wrong state/class hash from blockifier/syscall/execution logic.**

The OS is the authoritative source for the SNOS proof. When the OS re-executes a block for proving, it independently applies `execute_replace_class` without validating the class hash. For any transaction that calls `replace_class` with an undeclared class hash:

1. The blockifier reverts the transaction; the committed state diff does not include the class hash change.
2. The OS does not revert the transaction; it writes the undeclared class hash into `contract_state_changes`.
3. The OS-computed global state root diverges from the blockifier-committed state root.
4. The proof is generated for a wrong state root, breaking the soundness of the proof system.

---

### Likelihood Explanation

Any user can submit a transaction calling `replace_class` with an arbitrary undeclared class hash. No special privilege is required. The blockifier will revert the transaction (fee is still charged), but the OS will not, creating the divergence on every such transaction. The TODO comment at line 898 explicitly acknowledges the missing check, confirming this is a known gap in the production code.

---

### Recommendation

Add a check in `execute_replace_class` in `syscall_impls.cairo` (and in `deprecated_execute_syscalls.cairo`) to verify that the supplied class hash corresponds to a declared contract class before updating `contract_state_changes`. The check should mirror the blockifier's behavior: if the class hash is not declared, the syscall should fail (write a failure response header) so the transaction reverts in the OS exactly as it does in the blockifier, preserving the state root invariant.

---

### Proof of Concept

1. Deploy a Cairo 1 contract `C` at address `A`.
2. From `C`, call `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. **Blockifier execution**: `get_compiled_class(0xdeadbeef)` returns `StateError::UndeclaredClassHash` → syscall fails → transaction reverts → state diff for block does not include any class hash change for `A`.
4. **OS re-execution for proof**: `execute_replace_class` in `syscall_impls.cairo` reads `state_entry` for `A` via `%{ GetContractAddressStateEntry %}`, constructs `new_state_entry` with `class_hash = 0xdeadbeef`, and calls `dict_update` — no failure, no revert.
5. The OS computes a contracts trie root that includes `A → class_hash = 0xdeadbeef`.
6. The blockifier-committed state root has `A → class_hash = original_hash`.
7. The proof is for the wrong global root; the proof invariant is broken. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
