### Title
Recipient Can Permanently Lock Bridged Funds by Changing ATA Authority Before `finalize_transfer` — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

In `FinalizeTransfer`, the `token_account` is constrained to be the ATA owned by `recipient`. If the recipient calls SPL Token's `SetAuthority` to reassign their ATA's owner to a key they do not control before `finalize_transfer` is executed, the Anchor constraint `associated_token::authority = recipient` will fail on every subsequent attempt. Because there is no admin rescue path in the bridge, the in-flight funds are permanently irrecoverable.

---

### Finding Description

`FinalizeTransfer` declares the recipient's token account as:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    associated_token::mint = mint,
    associated_token::authority = recipient,
    token::token_program = token_program,
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

The `recipient` field is an unchecked account whose public key is bound to the signed Wormhole payload via:

```rust
data.verify_signature(
    (ctx.accounts.mint.key(), ctx.accounts.recipient.key()),
    &ctx.accounts.common.config.derived_near_bridge_address,
)?;
``` [2](#0-1) 

This means the `recipient` key is fixed by the NEAR-side message and cannot be substituted by a relayer. The ATA address is therefore deterministic: `PDA(recipient, mint)`.

**Attack path:**

1. The recipient initiates a NEAR → Solana bridge transfer, specifying their Solana address. NEAR burns (or locks) the tokens and emits a signed Wormhole message.
2. Before any relayer calls `finalize_transfer`, the recipient calls SPL Token's `SetAuthority` on their existing ATA, changing the account owner to an address they do not control (e.g., `Pubkey::default()`).
3. A relayer calls `finalize_transfer`. Anchor's `init_if_needed` finds the ATA already exists at the deterministic address but its owner no longer equals `recipient`. The constraint `associated_token::authority = recipient` fails.
4. The transaction reverts atomically — the nonce is **not** consumed — but the ATA at the canonical address is permanently poisoned.
5. Every future call to `finalize_transfer` for this transfer will fail identically. The nonce can never be consumed, the Wormhole acknowledgment (`FinalizeTransferResponse`) is never posted back to NEAR, and the funds are irrecoverable. [3](#0-2) 

The same failure mode applies if the token's freeze authority (e.g., USDC's issuer) freezes the recipient's ATA: `transfer_checked` / `mint_to` will return `AccountFrozen`, the transaction reverts, and the transfer is permanently undeliverable. [4](#0-3) 

There is no admin-callable rescue instruction anywhere in the bridge:


The four admin instructions (`change_config`, `initialize`, `pause`, `update_metadata`) provide no mechanism to redirect or recover stuck in-flight funds.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

- **Bridged-token path (`mint_to`):** NEAR has already burned the user's bridged tokens. Solana never mints the counterpart. The supply is destroyed with no recovery.
- **Native-token path (`transfer_checked`):** Tokens are locked in the vault PDA on Solana. `finalize_transfer` can never release them. No admin withdrawal exists.

Both outcomes satisfy the allowed critical impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** The authority-change vector requires only a standard SPL Token `SetAuthority` call, which any token account owner can issue permissionlessly. A user could trigger this accidentally (e.g., a wallet migration script that reassigns token account authorities) or deliberately (self-griefing). The frozen-account vector additionally applies to any token with an active freeze authority (USDC, USDT, etc.) where the issuer freezes the recipient for compliance reasons — a realistic, real-world scenario for a production bridge.

---

### Recommendation

1. **Add an admin-callable rescue instruction** (analogous to the fix in the referenced report) that, given a valid signed `FinalizeTransferPayload` whose nonce has not yet been consumed, allows the admin to redirect the transfer to an alternate recipient address or return the funds to the NEAR-side escrow.
2. **Alternatively**, record the pending transfer on-chain before attempting the token delivery, so that a failed delivery does not silently discard the transfer — a relayer or the recipient can retry with a corrected account.
3. For the frozen-account case, consider checking `token_account.is_frozen()` before attempting the CPI and emitting a recoverable pending-state event rather than reverting, so the nonce can be consumed and the transfer can be claimed later once the account is unfrozen.

---

### Proof of Concept

```
1. Alice holds 100 bridged-USDC on NEAR and initiates a transfer to her Solana address S.
   NEAR burns 100 bUSDC and emits Wormhole VAA: {recipient: S, mint: M, amount: 100, nonce: N}.

2. Alice calls SPL Token SetAuthority on her ATA(S, M), setting the new owner to Pubkey::default().

3. Relayer calls finalize_transfer with accounts:
     recipient = S
     token_account = ATA(S, M)   ← exists, but owner ≠ S

4. Anchor constraint `associated_token::authority = recipient` fails.
   Transaction reverts. Nonce N is NOT marked used.

5. Relayer retries — same failure every time.
   Alice's 100 bUSDC are burned on NEAR and undeliverable on Solana.
   No admin instruction exists to recover them.
```

impl FinalizeTransfer<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        UsedNonces::use_nonce(
            data.destination_nonce,
            &self.used_nonces,
            &mut self.config,
            self.authority.to_account_info(),
            self.common.payer.to_account_info(),
            &Rent::get()?,
            self.system_program.to_account_info(),
        )?;

        if let Some(vault) = &self.vault {
            // Native version. We have a proof of token registration by vault existence
            transfer_checked(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    TransferChecked {
                        from: vault.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                        mint: self.mint.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
                self.mint.decimals,
            )?;
        } else {
            // Bridged version. May be a fake token with our authority set but it will be ignored on the near side
            require!(
                self.mint.mint_authority.contains(self.authority.key),
                ErrorCode::InvalidBridgedToken
            );

            mint_to(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    MintTo {
                        mint: self.mint.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
            )?;
        }

        let payload = FinalizeTransferResponse {
            token: self.mint.key(),
            amount: data.amount,
            fee_recipient: data.fee_recipient.unwrap_or_default(),
            transfer_id: data.transfer_id,
        }
        .serialize_for_near(())?;

        self.common.post_message(payload)?;

        Ok(())
    }
