### Title
Missing `transaction_commitment` Verification in P2P Transaction Sync Allows Attacker-Controlled Peer to Inject Wrong Transactions — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` collects exactly `header.n_transactions` transactions from an attacker-controlled p2p peer and stores them via `append_body` without ever verifying that `calculate_transaction_commitment(collected_txs) == header.transaction_commitment`. The stored body is permanently inconsistent with the header's commitment, and is served by RPC and fed to SNOS as authoritative block data.

---

### Finding Description

**Entry point — transaction sync:**

`parse_data_for_block` reads `target_transaction_len` from the stored header's `n_transactions` field, then loops collecting exactly that many `FullTransaction` items from the network peer: [1](#0-0) 

The peer-supplied `transaction_hash` is pushed directly into `block_body.transaction_hashes` with an explicit acknowledgement that it is unvalidated: [2](#0-1) 

After the loop, `(block_body, block_number)` is returned and written to storage unconditionally: [3](#0-2) 

`append_body` performs no commitment check — it stores whatever `BlockBody` it receives: [4](#0-3) 

**Entry point — header sync:**

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from a peer and only checks block-number ordering and signature-vector length. It performs **no cryptographic signature verification** against a known sequencer public key and **no block-hash verification**: [5](#0-4) 

`verify_block_signature` is never called anywhere in `crates/apollo_p2p_sync/`. This means an attacker who controls the header peer can also supply an arbitrary `transaction_commitment` and `n_transactions`.

**The commitment that is never checked:**

`calculate_transaction_commitment` builds a Poseidon Patricia root over `(tx_hash, tx_signature)` leaf pairs: [6](#0-5) 

This root is stored in `BlockHeaderCommitments.transaction_commitment` and is chained into the block hash: [7](#0-6) 

After `append_body` completes, the stored transaction hashes are permanently decoupled from the `transaction_commitment` field in the stored header.

---

### Impact Explanation

An attacker-controlled p2p peer can cause a syncing full node to store an arbitrary set of K transactions for any block, with those transactions bearing no relation to the `transaction_commitment` recorded in the block header. Downstream consumers — RPC (`get_block_with_txs`, `get_transaction_by_hash`, `simulate_transactions`, etc.) and the SNOS input builder — read transactions from storage and treat them as authoritative. The concrete corrupted value is: `calculate_transaction_commitment(stored_tx_hashes) ≠ header.transaction_commitment`.

This satisfies: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, and potentially **Critical — wrong transactions fed to SNOS/proof production**.

---

### Likelihood Explanation

Any node on the Starknet p2p network can act as a sync peer. No operator, validator, or admin privilege is required. The attack requires only that the victim node selects the attacker's peer for either the header or the transaction sub-protocol query. The missing check is explicitly flagged with a TODO in the source, confirming it is a known gap rather than an intentional design decision.

---

### Recommendation

After collecting all K transactions in `parse_data_for_block`, recompute the transaction commitment and compare it against `header.transaction_commitment` before returning `Ok(Some(...))`. Reject the peer (return `ParseDataError::BadPeer`) if the roots diverge. The same guard should be applied to event and receipt commitments. Additionally, `HeaderStreamBuilder::parse_data_for_block` should cryptographically verify the block signature against the known sequencer public key before storing the header.

---
