### Title
Missing shard-id binding in `set_state_header` allows cross-shard state-root substitution during state sync — (File: `chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a caller-supplied `shard_id` and a `ShardStateSyncResponseHeader` containing a chunk. It verifies the chunk is present in the block's `chunk_headers_root` via `verify_path`, but `verify_path` is position-agnostic (it does not call `verify_path_with_index`). There is no assertion that `chunk.shard_id() == shard_id`. A malicious peer can therefore supply the chunk for shard Y — with its valid merkle proof — in response to a state-header request for shard X. The header is then committed to `DBCol::StateHeaders` under key `StateHeaderKey(shard_id=X, sync_hash)` carrying shard Y's `prev_state_root`. All subsequent state-part validation and application for shard X is performed against shard Y's state root, permanently corrupting the syncing node's view of shard X.

### Finding Description

`set_state_header` performs five checks before persisting the header:

1. **Chunk self-consistency** (`validate_chunk_proofs`) — verifies the chunk's internal hash and body. No shard-id check.
2. **Chunk inclusion in block** — uses `verify_path` against `sync_prev_block_header.chunk_headers_root()`.
3. **Prev-chunk inclusion** — same `verify_path` pattern.
4. **Incoming receipt proofs** — validates receipts against the outer `shard_id` parameter.
5. **State-root-node validity** — validates the state root node from the chunk's `prev_state_root`.

The critical gap is in check 2. `verify_path` is defined as:

```rust
// core/primitives/src/merkle.rs:113
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
```

It only confirms the item hash appears somewhere in the tree. It does **not** call `verify_path_matches_index`, which is what `verify_path_with_index` does to enforce that the proof's Left/Right directions correspond to a specific leaf index. The block's `chunk_headers_root` is a merkle tree over all shard chunks ordered by shard index. A chunk from shard Y at index Y has a valid proof that passes `verify_path` regardless of what `shard_id` (X) the caller requested.

After passing all checks, the header is stored:

```rust
// chain/chain/src/state_sync/adapter.rs:527
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

`shard_id` here is the function parameter (X), but the chunk inside `shard_state_header` belongs to shard Y. The stored header now carries shard Y's `prev_state_root` under shard X's key.

`set_state_part` then retrieves this header and validates every incoming state part against shard Y's state root:

```rust
// chain/chain/src/state_sync/adapter.rs:541-545
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// validates part against state_root (shard Y's root)
self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part)
```

The syncing node downloads and applies shard Y's trie data as shard X's state. When `set_state_finalize_on_height` subsequently applies missing chunks for shard X on top of this wrong state, the resulting state roots diverge from the canonical chain. The node either stalls during catchup or, if it proceeds to chunk production, emits chunks with an incorrect `prev_state_root` that the rest of the network rejects.

### Impact Explanation

**High.** The syncing node's committed state for shard X is replaced with shard Y's trie. This breaks the fundamental invariant that `StateHeaderKey(shard_id=X, sync_hash)` binds to the state root of shard X at the sync boundary. Consequences:

- The node cannot complete state sync for shard X (catchup chunk application diverges from canonical state roots).
- If the node proceeds past catchup, it produces chunks with a wrong `prev_state_root`, which are rejected by all other validators.
- A chunk producer assigned to shard X is effectively knocked out of consensus for that shard until it re-syncs from scratch.
- The attack is repeatable: the malicious peer can serve the same poisoned header on every re-sync attempt.

### Likelihood Explanation

**Medium.** State sync responses are served by any reachable peer; there is no authentication of the responding node. Any network participant (including a non-validator) can intercept or respond to `StateRequestHeader` messages. The attacker only needs to know the `sync_hash` and `shard_id` being requested (both are visible on the network) and to have a valid chunk from a different shard in the same block (always available from the canonical chain). No privileged role is required.

### Recommendation

Add an explicit shard-id binding check immediately after `validate_chunk_proofs` in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match expected shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, replace the position-agnostic `verify_path` call with `verify_path_with_index`, supplying the shard's index within the epoch's shard layout, so the merkle proof is pinned to the correct leaf position:

```rust
let shard_index = shard_layout.get_shard_index(shard_id)?;
if !verify_path_with_index(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
    shard_index as u64,
    shard_layout.num_shards(),
) { ... }
```

### Proof of Concept

1. Honest node N begins state sync for `shard_id = 0` at `sync_hash = H`.
2. Block at `prev(H)` contains chunks for shards 0 and 1. Shard 1's chunk has `prev_state_root = R1`.
3. Malicious peer M intercepts the `StateRequestHeader { shard_id: 0, sync_hash: H }` message.
4. M constructs a `ShardStateSyncResponseHeader` containing shard 1's full chunk (valid, passes `validate_chunk_proofs`) and shard 1's merkle proof against `chunk_headers_root` (valid, passes `verify_path` because position is not checked).
5. N calls `set_state_header(shard_id=0, sync_hash=H, header_with_shard1_chunk)`. All five checks pass. The header is stored under `StateHeaderKey(shard_id=0, H)` with `prev_state_root = R1`.
6. N downloads state parts for shard 0, each validated against `R1` (shard 1's root). Parts for shard 1's trie pass; parts for shard 0's trie would fail, so M serves shard 1's parts.
7. N applies shard 1's trie as shard 0's state. `set_state_finalize_on_height` applies shard 0's missing chunks on top of shard 1's state; the resulting roots diverge from the canonical chain.
8. N is unable to complete catchup for shard 0 or produces invalid chunks, and must re-sync — where M can repeat the attack.
