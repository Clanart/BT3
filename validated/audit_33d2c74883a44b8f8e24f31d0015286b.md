### Title
Unauthenticated `RemoteProofManagerServer` `SetProof` Endpoint Bypasses Cryptographic Proof Verification, Causing Invalid Transactions to Be Accepted - (File: `crates/apollo_proof_manager/src/communication.rs`)

---

### Summary

The `ProofManager`'s `SetProof` RPC endpoint is exposed via an unauthenticated `RemoteComponentServer` bound to `0.0.0.0`. The proof-verification logic in `run_proof_verification` unconditionally skips `verify_proof` when `contains_proof` returns `true`. An attacker who can reach the proof manager's network port can pre-seed an arbitrary `(ProofFacts, garbage_proof)` pair, then submit an Invoke V3 transaction carrying those same `ProofFacts`; the gateway and consensus converter will skip cryptographic verification and accept the transaction as if it carried a valid proof.

---

### Finding Description

**Root cause — no access control on `SetProof`:**

`ProofManager::handle_request` dispatches every incoming `ProofManagerRequest::SetProof` directly to `self.set_proof(proof_facts, proof)` with no caller identity check: [1](#0-0) 

`ProofManager::set_proof` itself only checks whether the proof is already stored; it performs no cryptographic verification of the proof blob: [2](#0-1) 

**Network exposure — `RemoteProofManagerServer` bound to `0.0.0.0`:**

In both the distributed and hybrid deployment configurations the proof manager's remote server is configured with `bind_ip: "0.0.0.0"` and no TLS or authentication layer: [3](#0-2) 

The `RemoteComponentServer` connection handler contains no authentication check — it processes any incoming serialised `ProofManagerRequest`: [4](#0-3) 

**Verification bypass — `contains_proof` short-circuits `verify_proof`:**

`run_proof_verification` is the single shared verification path for both the gateway and consensus flows. When `contains_proof` returns `true` it returns immediately without calling `starknet_proof_verifier::verify_proof`: [5](#0-4) 

The gateway flow stores the proof only after this check passes, but the consensus flow (`spawn_verify_and_store_proof`) also relies on the same early-exit: [6](#0-5) 

**Structural proof-facts validation does not substitute for proof verification:**

`validate_proof_facts` in the blockifier validates the *fields* of the proof facts (allowed proof version, allowed program hash, block number within retention window, block hash matches stored value, config hash matches). It does not verify the cryptographic proof blob: [7](#0-6) 

These structural checks can all be satisfied with a legitimately constructed `ProofFacts` struct (using a real historical block hash and the correct program/config hashes) while the `Proof` blob itself remains garbage.

---

### Impact Explanation

An attacker who can reach the proof manager's TCP port (reachable from within the cluster in the distributed topology, or from any host in a misconfigured deployment) can:

1. Construct a `ProofFacts` that passes all blockifier structural checks (valid `block_number`, matching `block_hash` from the stored block-hash contract, allowed `program_hash`, correct `config_hash`).
2. Pair it with an arbitrary/garbage `Proof` blob.
3. Call `SetProof` directly on the `RemoteProofManagerServer` — no authentication required.
4. Submit an Invoke V3 transaction carrying those `ProofFacts`.
5. The gateway/consensus converter calls `contains_proof` → `true` → skips `verify_proof` → accepts the transaction.

The sequencer's mempool and consensus pipeline accept a transaction whose attached proof has never been cryptographically verified. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

- The `RemoteProofManagerServer` is a production component, enabled in both the distributed (`proof_manager.json`) and hybrid (`replacer_core.json`) deployment profiles.
- `bind_ip: "0.0.0.0"` is the default value in `config_schema.json`, meaning any node reachable on the network can connect.
- The `ProofManagerRequest` enum is serialised with `serde` and sent over plain HTTP/2 — no TLS, no token, no IP allowlist is enforced at the application layer.
- Constructing valid `ProofFacts` requires only public on-chain data (a historical block hash, the published program hash, and the config hash), all of which are observable by any Starknet participant.

---

### Recommendation

1. **Remove `SetProof` from the remote-facing API surface.** Only `GetProof` and `ContainsProof` need to be callable by remote components (gateway, batcher). `SetProof` should be callable only by the local gateway/consensus converter after successful `verify_proof`.
2. **Alternatively, verify inside `set_proof`.** Move the `verify_proof` call into `ProofManager::set_proof` so that any caller — local or remote — must supply a proof that passes cryptographic verification before it is stored.
3. **Restrict the remote server binding.** Change the default `bind_ip` from `0.0.0.0` to a cluster-internal address, and add mTLS or a network policy that limits which pods may connect to the proof manager port.

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
