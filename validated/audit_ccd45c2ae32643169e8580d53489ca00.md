### Title
EIP-7702 SetCode Transactions Bypass `EnableCreate` Governance Restriction - (File: `ante/interfaces/setup.go`)

### Summary
When `EnableCreate` is disabled via governance to freeze code installation on the chain, the guard `!enableCreate && tx.To() == nil` only blocks traditional `CREATE`-opcode contract deployments. EIP-7702 SetCode transactions always have `tx.To() != nil`, so they pass the guard unconditionally and proceed to install delegation bytecode on EOAs — the exact code-mutation operation the flag is meant to prevent.

### Finding Description
Both enforcement sites use the same incomplete predicate.

**Ante handler** — `ante/interfaces/setup.go` lines 116–120:

```go
// return error if contract creation or call are disabled through governance
if !enableCreate && tx.To() == nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "failed to create new contract")
} else if !enableCall && tx.To() != nil {
    return errorsmod.Wrap(evmtypes.ErrCallDisabled, "failed to call contract")
}
```

**State-transition layer** — `x/evm/keeper/state_transition.go` lines 352–356:

```go
if !cfg.Params.EnableCreate && msg.To == nil {
    return nil, errorsmod.Wrap(types.ErrCreateDisabled, "failed to create new contract")
} else if !cfg.Params.EnableCall && msg.To != nil {
    return nil, errorsmod.Wrap(types.ErrCallDisabled, "failed to call contract")
}
```

An EIP-7702 SetCode transaction has `tx.To() != nil` (it targets a call destination), so neither branch fires when only `EnableCreate` is disabled. Execution continues into `ApplyMessageWithConfig`, which processes the authorization list (`x/evm/keeper/state_transition.go` lines 484–513) and calls `setAuthorizationDelegation` (`x/evm/keeper/set_code_authorizations.go` lines 74–84):

```go
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
```

This writes `EF0100<delegate_address>` bytecode onto the authority EOA — a direct account-code mutation — despite `EnableCreate` being disabled.

The structural parallel to the external report is exact: just as `isIncreasing` only tests whether `sizeDelta` has the same sign as `self.size` (missing the net-direction flip), the `EnableCreate` guard only tests `tx.To() == nil` (missing the EIP-7702 path that also mutates account code).

### Impact Explanation
When governance disables `EnableCreate` (e.g., to freeze deployments during an emergency or chain upgrade), any unprivileged user can submit an EIP-7702 SetCode transaction to install delegation code on their own EOA. This:

1. Directly bypasses the governance-enforced code-mutation freeze.
2. Installs executable bytecode on an EOA, making it behave as a smart contract.
3. Allows the attacker to execute arbitrary logic (from the delegated contract) from an account that governance intended to keep code-free.

This matches the allowed High impact: *"EIP-7702 authorization… bypass enabling… unauthorized account/code mutation."*

### Likelihood Explanation
The attack requires only:
- `EnableCreate` is disabled (a realistic governance action during emergencies or upgrades).
- `EnableCall` remains enabled (the common case; disabling both simultaneously would halt the chain entirely).
- The attacker submits a valid EIP-7702 SetCode transaction signed by their own EOA.

No privileged access, leaked keys, or special conditions are needed. Any EOA holder can exploit this whenever the governance state satisfies the two conditions above.

### Recommendation
Add an explicit check for EIP-7702 SetCode transactions when `EnableCreate` is disabled, in both `ante/interfaces/setup.go` and `x/evm/keeper/state_transition.go`:

```go
if !enableCreate && tx.To() == nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "failed to create new contract")
} else if !enableCall && tx.To() != nil {
    return errorsmod.Wrap(evmtypes.ErrCallDisabled, "failed to call contract")
}
// Add:
if !enableCreate && len(tx.SetCodeAuthorizations()) > 0 {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled,
        "EIP-7702 code installation is disabled (EnableCreate is false)")
}
```

### Proof of Concept
1. Governance disables `EnableCreate` via `MsgUpdateParams`.
2. Attacker constructs an EIP-7702 SetCode transaction:
   - `tx.To` = any non-nil address (e.g., attacker's own EOA or any contract).
   - `authorizationList` = `[{chainId, delegateContractAddress, nonce, v, r, s}]` signed by the attacker's EOA key.
3. Ante handler evaluates `!enableCreate && tx.To() == nil` → **false** (since `tx.To() != nil`). Transaction is admitted.
4. `ApplyMessageWithConfig` evaluates the same predicate → **false**. Execution continues.
5. The authorization loop calls `applyAuthorization` → `setAuthorizationDelegation`, which executes `stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), ...)`.
6. Delegation bytecode is now committed to the authority EOA's code slot despite `EnableCreate` being disabled.
7. The attacker's EOA now executes the delegate contract's logic on every call, bypassing the governance freeze. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```go
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
```
