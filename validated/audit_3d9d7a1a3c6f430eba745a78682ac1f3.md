### Title
`set_state_part` Accepts Attacker-Controlled `PartId.total` Without Validating Against Header's `num_parts`, Causing Panic in `apply_state_part` - (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_part` validates an incoming state part using the caller-supplied `PartId` (including `total`), but never checks that `total` matches the `num_parts` committed in the stored state header. The storage key only encodes `part_id.idx`. When `apply_state_part` later uses the header's correct `num_parts` to compute trie boundaries, the stored part — validated against a different `total` — may not contain the required nodes, causing an unconditional panic via `.expect("apply_state_part is guaranteed to succeed when each part is valid")`.

---

### Finding Description

**Root cause — missing `num_parts` binding in `set_state_part`:**

---

### Impact Explanation

A malicious snapshot host (any node that advertises itself as having a state snapshot) can respond to a state part request for `part_id = 0` with a part crafted for `PartId { idx: 0, total: K }` where `K > N` (the header's `num_parts`). The part passes `validate_state_part` because it is internally consistent for that `K`. It is stored under the key for `idx=0`. When the syncing node applies all parts using the header's `N`, the apply step panics, crashing the state sync task. The node retries, but if the malicious peer is consistently selected (e.g., it is the only advertised snapshot host, or it wins peer selection repeatedly), state sync for that shard is permanently blocked. Nodes that cannot complete state sync cannot participate in the network.

---

### Likelihood Explanation

Any node that can join the NEAR network and advertise a snapshot (via `SnapshotHostEvent`) can execute this attack. No privileged role (validator, chunk producer) is required. The syncing node selects snapshot hosts from the advertised set; a malicious host that advertises availability for the target shard will be selected. The crafted part is trivially constructable: obtain the real part for `PartId { idx: 0, total: K }` from the trie (or compute it locally), which is a valid subset of the full trie.

---

### Recommendation

In `set_state_part`, after retrieving the stored header, assert that `part_id.total` matches `shard_state_header.num_state_parts()`:

```rust
pub fn set_state_part(
    &self,
    shard_id: ShardId,
    sync_hash: CryptoHash,
    part_id: PartId,
    part: &StatePart,
) -> Result<(), Error> {
    let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
    let num_parts = shard_state_header.num_state_parts();
    // ADD: reject parts whose total doesn't match the committed header
    if part_id.total != num_parts {
        return Err(Error::Other(format!(
            "set_state_part: part_id.total {} != header num_parts {}",
            part_id.total, num_parts
        )));
    }
    ...
}
```

Additionally, change `Trie::apply_state_part` to return `Result` instead of panicking, so that a mismatch produces a recoverable error rather than a process crash.

### Proof of Concept

1. Attacker runs a NEAR node and advertises a snapshot for shard `S` at `sync_hash`.
2. Victim node begins state sync; its header says `num_parts = N` (e.g., 100).
3. Victim requests `part_id = 0` from the attacker.
4. Attacker computes a valid part for `PartId { idx: 0, total: K }` with `K = 10 * N` (e.g., 1000). This part covers only the first `1/1000` of the trie and passes `validate_state_part` for that `PartId`.
5. Attacker sends this part in the `ShardStateSyncResponse`.
6. Victim calls `set_state_part(shard_id, sync_hash, PartId { idx: 0, total: 1000 }, &part)`. No check against `num_parts = 100`. Validation passes. Part stored under `StatePartKey(sync_hash, shard_id, 0)`.
7. Victim downloads parts 1–99 correctly and proceeds to apply all parts.
8. `apply_state_part` is called with `PartId { idx: 0, total: 100 }` and the stored bytes (covering only `[0, 1/1000)`). The iterator requests nodes in `[0, 1/100)` but the part only has `[0, 1/1000)`. `apply_state_part_impl` returns `Err(MissingTrieValue)`. `apply_state_part` panics. State sync task for shard `S` crashes.

    pub fn validate_state_part(
        state_root: &StateRoot,
        part_id: PartId,
        partial_state: PartialState,
    ) -> Result<(), StorageError> {
        let PartialState::TrieValues(nodes) = &partial_state;
        let num_nodes = nodes.len();
        let (trie, storage) = Trie::from_recorded_storage_with_storage::<true>(
            PartialStorage { nodes: partial_state },
            *state_root,
            false,
        );
        trie.visit_nodes_for_state_part(part_id)?;
        drop(trie);
        let storage = Arc::into_inner(storage).unwrap();
        if storage.visited_nodes.len() != num_nodes {
            // As all nodes belonging to state part were visited, there is some
            // unexpected data in downloaded state part.
            return Err(StorageError::UnexpectedTrieValue);
        }
        Ok(())
    }
