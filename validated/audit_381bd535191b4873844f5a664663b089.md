Audit Report

## Title
Stranded ETH on router subsidizes attacker's WETH swap via `pay()` native-balance priority — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's global ETH balance — to fund any WETH payment, regardless of which caller deposited that ETH. Because `exactInputSingle` is `payable` with no automatic refund and no `msg.value == amountIn` enforcement, excess ETH from a prior caller remains on the router and is available to any subsequent WETH swap, allowing an attacker to drain the victim's stranded ETH at zero personal cost.

## Finding Description
`exactInputSingle` (L67) is `payable` and performs no check that `msg.value` equals `params.amountIn`, nor does it call `refundETH()` before returning. Any excess ETH sent by a victim accumulates on the router.

When the attacker subsequently calls `exactInputSingle{value:0}` with `tokenIn = WETH`, the callback context is set with `payer = msg.sender` (attacker) and `tokenToPay = WETH` (L71). The pool's swap callback triggers `_justPayCallback` (L192–199), which calls `pay(WETH, attacker, pool, X)`.

Inside `pay()` (L73–84), the WETH branch reads `nativeBalance = address(this).balance`. This is the router's entire ETH balance, including the victim's stranded ETH. If `nativeBalance >= value`, the contract wraps that ETH and transfers WETH to the pool — never touching the attacker's WETH allowance or wallet. The victim's ETH is consumed to settle the attacker's swap.

The `receive()` guard (L32–34) only rejects direct ETH transfers from non-WETH addresses; it does not prevent ETH from entering via `msg.value` on payable functions, so the stranding precondition is trivially achievable.

## Impact Explanation
Direct theft of user principal. The victim's stranded ETH is consumed to fund the attacker's swap. The attacker receives real pool output tokens without spending any of their own assets. Loss equals `min(stranded_ETH, attacker_amountIn)`. This is a direct loss of user funds meeting Critical/High thresholds under Sherlock rules.

## Likelihood Explanation
ETH stranding is a common user mistake with payable routers — sending excess ETH without a `refundETH()` call in the same multicall. The attacker requires no special permissions, only a public `exactInputSingle` call with `tokenIn = WETH`. The attack is atomic, requires no front-running beyond observing the router's ETH balance on-chain, and is repeatable as long as ETH remains stranded.

## Recommendation
Restrict the native-ETH path in `pay()` to only fire when `payer == address(this)` (mid-path context), ensuring external callers cannot benefit from ETH deposited by others. Alternatively, track per-caller deposited ETH via transient storage keyed to `msg.sender` and only allow `pay()` to consume ETH attributed to the current payer. Additionally, add a post-swap `refundETH()` call or enforce `msg.value == amountIn` in `exactInputSingle` and other payable entry points.

## Proof of Concept
```solidity
// 1. Victim strands 0.5 ETH on router
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    tokenIn: WETH, amountIn: 0.5 ether, ...
}));
// Victim forgets refundETH(); router now holds 0.5 ETH

// 2. Attacker has 0 ETH, 0 WETH allowance
uint256 attackerWETHBefore = IERC20(WETH).balanceOf(attacker);
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn: WETH, amountIn: 0.5 ether, recipient: attacker, ...
}));
// pay(WETH, attacker, pool, 0.5e18):
//   nativeBalance = 0.5 ETH >= 0.5 ETH → wraps victim's ETH, sends WETH to pool
// Attacker receives pool output tokens

assert(IERC20(WETH).balanceOf(attacker) == attackerWETHBefore); // allowance untouched
assert(address(router).balance == 0);                            // victim's ETH gone
```
