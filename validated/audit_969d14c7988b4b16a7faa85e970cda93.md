### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract State Corruption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by a contract is actually declared in the `contract_class_changes` dictionary. This is an exact analog of the reported state-transition bypass: just as `checkTradeExpiry` lacked proper settlement-status guards and allowed re-entry into a terminal vault state, `execute_replace_class` lacks a class-declaration guard and allows a contract to transition into a permanently broken state. Any funds held by the affected contract are irreversibly frozen.

---

### Finding Description

In `execute_replace_class` the OS reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` with no check that the hash corresponds to a declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing guard. The function accepts any arbitrary `felt` as the new class hash and commits it to state without consulting `contract_class_changes`.

When a subsequent transaction calls the corrupted contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // ← the arbitrary, undeclared hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // ← 0 (default for undeclared key)
);
``` [2](#0-1) 

`dict_read` on an undeclared key returns the default value `0`. `find_element` then searches for a compiled class with hash `0`, which does not exist in any real deployment. `find_element` panics on a missing key, making the contract permanently unexecutable at the OS/proof level.

The state commitment machinery in `compute_contract_state_commitment` faithfully hashes and commits whatever `class_hash` is stored in the `StateEntry`, with no cross-check against the class tree: [3](#0-2) 

Once the block containing the `replace_class` call is proven and finalized, the invalid class hash is part of the canonical state root. There is no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared hash (whether by attacker design or by exploiting a bug in a shared contract) becomes permanently inaccessible. All ERC-20 balances, NFTs, or other assets held by that contract are frozen forever. If the affected contract is a shared vault, AMM, or multi-user escrow, every depositor loses access to their funds simultaneously.

---

### Likelihood Explanation

**Medium.** The missing check is explicitly flagged in a developer TODO comment dated 2026, confirming the team is aware of the gap but has not yet closed it. The entry path requires only that a contract call the `replace_class` syscall — a standard, unprivileged operation available to any Sierra contract. No privileged role, leaked key, or off-chain coordination is needed. A malicious contract deployer can trigger this deterministically.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, read the corresponding entry from `contract_class_changes` and assert it is non-zero (i.e., the class has been declared):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the guard already present in `execute_declare_transaction`, which asserts `compiled_class_hash != 0` before writing to `contract_class_changes`: [4](#0-3) 

---

### Proof of Concept

1. **Attacker deploys a contract** whose code contains a function `poison()` that calls `replace_class(0xdeadbeef)` — an arbitrary felt that has never been declared.
2. **Users deposit funds** into the contract (e.g., it presents itself as a yield vault).
3. **Attacker calls `poison()`**. The OS executes `execute_replace_class`, reads `class_hash = 0xdeadbeef` from the syscall request, and writes it directly to `contract_state_changes` with no validation.
4. **Block is proven and finalized.** The state root now encodes `class_hash = 0xdeadbeef` for the contract address.
5. **Any subsequent call to the contract** reaches `execute_entry_point`, which calls `dict_read` on `contract_class_changes` for key `0xdeadbeef`, receiving `0` (undeclared). `find_element` panics searching for compiled class hash `0`.
6. **The contract is permanently inaccessible.** All deposited funds are frozen with no recovery mechanism.
