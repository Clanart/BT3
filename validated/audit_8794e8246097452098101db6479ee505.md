### Title
Detached `burn_tokens_if_needed` Promise Leaves Bridge State Inconsistent on Burn Failure — (`near/omni-bridge/src/lib.rs`)

### Summary

`burn_tokens_if_needed` fires a cross-contract burn call with `.detach()`, meaning the result is never observed. If the burn sub-call fails for any reason, the bridge contract's state is already committed as if the burn succeeded: the transfer message is stored, locked-token accounting is updated, and the relayer can proceed to sign the transfer. The destination chain then mints new tokens against source tokens that were never actually destroyed, inflating total bridged supply.

### Finding Description

`burn_tokens_if_needed` is defined as:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result is never checked
    }
}
``` [1](#0-0) 

`BURN_TOKEN_GAS` is only 3 TGas: [2](#0-1) 

This function is called inside `init_transfer_internal` **after** the transfer message has already been stored and locked-token accounting updated:

```rust
// transfer message already stored above
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(...);
// InitTransferEvent emitted
``` [3](#0-2) 

The same pattern appears in `fast_fin_transfer_to_other_chain`, where the relayer's bridged tokens are supposed to be burned before a new cross-chain transfer message is created: [4](#0-3) 

And in `resolve_fast_transfer`, where the comment explicitly acknowledges the burn is needed to prevent double-minting:

```rust
// Burn the tokens to ensure the locked tokens are not double-minted
self.burn_tokens_if_needed(token_id.clone(), amount);
``` [5](#0-4) 

Because `.detach()` is used in all three call sites, a failed burn sub-call is silently ignored. The bridge has no mechanism to detect or recover from this failure.

### Impact Explanation

If the detached burn call fails (e.g., 3 TGas is exhausted by the token contract's storage operations, the token contract is paused, or any other runtime failure occurs):

1. The user's bridged tokens are transferred to the bridge contract via `ft_transfer_call` but **not burned**.
2. The `TransferMessage` is stored and the `InitTransferEvent` is emitted.
3. A trusted relayer calls `sign_transfer`, which succeeds because the transfer message exists.
4. The destination chain finalizes the transfer and **mints new tokens** for the user.
5. The source bridged tokens remain unburned inside the bridge contract, inflating total supply.

This breaks bridge collateralization: the destination chain's minted supply is no longer backed 1:1 by burned source tokens. Over repeated occurrences, the bridge's locked-token accounting diverges from reality, enabling effective double-spend of bridged assets.

### Likelihood Explanation

The `burn` function in `omni-token` is:

```rust
fn burn(&mut self, amount: U128) {
    self.assert_controller();
    self.token.internal_withdraw(&env::predecessor_account_id(), amount.into());
}
``` [6](#0-5) 

Under normal conditions this is simple enough for 3 TGas. However, the design flaw is structural: **no code path verifies the burn succeeded**. Any future change to the token contract (added logic, pause guard, storage migration) that increases gas consumption beyond 3 TGas would silently trigger the inconsistency. Additionally, if the token contract is ever paused at the moment of a bridge transfer, the burn fails silently while the transfer proceeds. Because the entry point (`ft_transfer_call` → `ft_on_transfer` → `init_transfer`) is open to any unprivileged user, the window is reachable without any privileged access.

### Recommendation

Replace the detached burn with a chained callback that verifies success before committing state, or move the burn call to a callback that executes only after the transfer message is confirmed to be valid and the burn result is known. At minimum, add a callback that reverts the stored transfer message and locked-token accounting if the burn sub-call fails:

```rust
ext_token::ext(token)
    .with_static_gas(BURN_TOKEN_GAS)
    .burn(amount)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(BURN_CALLBACK_GAS)
            .on_burn_callback(transfer_id),
    );
```

Also increase `BURN_TOKEN_GAS` from 3 TGas to a safer margin (e.g., 10–15 TGas) to account for future token contract complexity.

### Proof of Concept

1. Alice holds 1000 bridged USDC on NEAR (minted by the bridge, `is_deployed_token` = true).
2. Alice calls `ft_transfer_call` on the bridged USDC contract, sending 1000 tokens to the bridge with an `InitTransfer` message targeting an EVM chain.
3. The bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal` runs. The transfer message is stored and `burn_tokens_if_needed` fires a detached burn call.
4. The burn sub-call fails (e.g., token contract is paused, or 3 TGas is insufficient after a token upgrade). The bridge does not observe this failure.
5. The `InitTransferEvent` is emitted. A relayer calls `sign_transfer`; the MPC signs the payload.
6. The relayer submits the signed payload to the EVM bridge, which calls `finTransfer` and mints 1000 USDC for Alice on the EVM chain.
7. Alice now holds 1000 USDC on EVM. The 1000 bridged USDC tokens remain unburned inside the NEAR bridge contract, creating 1000 units of unbacked supply. [7](#0-6) [1](#0-0)

            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(

                required_storage_balance,

        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

                &token_id,

            return transfer_message.amount;
