### Title
`skip_stateful_validations` Over-Extended Scope Allows Signature Bypass for Factory-Deployed Accounts — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is designed to skip `__validate__` only when a `deploy_account` transaction is pending in the mempool (UX improvement for simultaneous deploy+invoke). However, the gating check `account_tx_in_pool_or_recent_block` returns `true` for **any** account known to the mempool — not exclusively accounts with a pending `deploy_account`. For accounts deployed via the `deploy` syscall (factory-deployed), which have `nonce=0` in state but are already live, an attacker can exploit this over-extension to inject a nonce=1 invoke transaction with an invalid signature that bypasses `__validate__` at the gateway.

### Finding Description

The function `skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` decides whether to skip the `__validate__` entry-point call for an incoming invoke transaction:

```rust

    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                // ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is incorrect. `account_tx_in_pool_or_recent_block` returns `true` whenever the account has **any** transaction in the pool or is known from a recent committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

**Attack scenario:**

1. Account `A` is deployed via the `deploy` syscall (factory deployment). It has contract code but `nonce=0` in state (factory deployment does not increment the nonce).
2. The legitimate owner submits a nonce=0 invoke for `A`. This passes `__validate__` (account has code, signature is valid) and enters the mempool. Now `account_tx_in_pool_or_recent_block(A) == true`.
3. An attacker submits a nonce=1 invoke for `A` with an **invalid or absent signature**.
4. `validate_nonce` passes: `0 <= 1 <= 0 + max_allowed_nonce_gap(200)`. [3](#0-2) 

5. `validate_by_mempool` passes: the mempool accepts the nonce gap.
6. `skip_stateful_validations` returns `true`: `tx.nonce()==1` ✓, `account_nonce==0` ✓, `account_tx_in_pool_or_recent_block==true` ✓.
7. `run_validate_entry_point` is called with `skip_validate=true`, so `execution_flags.validate = false` — `__validate__` (signature verification) is **never called**. [4](#0-3) 

8. The attacker's unsigned/invalid-signature nonce=1 invoke is accepted into the mempool and forwarded to the batcher.

The over-extension is structurally identical to the ENS M-01 bug: approval granted for one scope (deploy_account UX skip) bleeds into a different scope (already-deployed factory accounts), because the gating predicate (`account_tx_in_pool_or_recent_block`) does not distinguish between the two cases.

### Impact Explanation

The gateway accepts an invalid transaction — one that carries no valid account signature — into the mempool and passes it to the batcher for sequencing. This matches the High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

During block execution the batcher will call `__validate__` with `validate=true` (default execution flags), so the transaction will be rejected at that stage. However, the gateway has already admitted it, consuming mempool capacity and batcher resources, and the attacker can repeat this at scale for any factory-deployed account that has a pending nonce=0 invoke.

### Likelihood Explanation

Factory-deployed accounts (deployed via `deploy` syscall rather than `deploy_account`) are a standard Starknet pattern. Any such account with `nonce=0` in state and at least one pending invoke in the mempool is a valid target. The attacker needs no privileged access — only the ability to submit transactions to the public gateway. The `max_allowed_nonce_gap` default of 200 means the nonce=1 invoke always passes the nonce range check. [5](#0-4) 

### Recommendation

Replace the over-broad `account_tx_in_pool_or_recent_block` check with a predicate that specifically verifies a **`deploy_account` transaction** is pending for the address. Add a dedicated mempool query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, and use that in `skip_stateful_validations`. Alternatively, check whether the account's class hash is zero in state (i.e., the account is genuinely not yet deployed) before allowing the skip, which would exclude factory-deployed accounts from the bypass path.

### Proof of Concept

```
1. Deploy contract C via `deploy` syscall from some factory. C has code, nonce=0 in state.

2. Owner submits Invoke(sender=C, nonce=0, signature=VALID).
   → validate_nonce: 0 <= 0 <= 200 ✓
   → run_validate_entry_point: __validate__ called, passes ✓
   → Tx enters mempool. account_tx_in_pool_or_recent_block(C) == true.

3. Attacker submits Invoke(sender=C, nonce=1, signature=GARBAGE).
   → validate_nonce: 0 <= 1 <= 200 ✓
   → validate_by_mempool: passes ✓
   → skip_stateful_validations:
       tx.nonce()==1 ✓
       account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(C)==true ✓
       → returns true (skip __validate__)
   → run_validate_entry_point called with skip_validate=true:
       execution_flags.validate = false
       __validate__ is NOT called
   → Attacker's tx with GARBAGE signature accepted into mempool ✓

4. Batcher picks up attacker's tx, calls __validate__ with validate=true,
   __validate__ fails → tx rejected at execution time.
   But gateway has already admitted it, bypassing signature enforcement.
``` [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
