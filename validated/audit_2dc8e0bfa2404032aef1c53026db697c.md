### Title
Hardcoded `L1_HANDLER_L2_GAS_MAX_AMOUNT` Permanently Freezes L1→L2 Messages Whose Handlers Exceed the Fixed Gas Budget — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`L1_HANDLER_L2_GAS_MAX_AMOUNT` is a compile-time constant (`100_000_000`) that is the sole gas budget for every L1 handler execution. Unlike regular invoke transactions — where the user specifies `resource_bounds[L2_GAS_INDEX].max_amount` — L1 message senders have no mechanism to supply a higher gas limit. If an L1 handler's actual gas consumption exceeds this constant (due to contract complexity, gas-cost schedule changes in a future OS upgrade, or deliberate crafting), the sequencer is forced to permanently skip the message. The message is never consumed and can never be retried, permanently freezing any funds or state transitions that depended on it.

---

### Finding Description

In `execute_l1_handler_transaction`, the gas budget for every L1 handler is unconditionally set to the compile-time constant:

```cairo
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [1](#0-0) 

The constant is defined as:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;
``` [2](#0-1) 

`non_reverting_select_execute_entry_point_func` hard-asserts that the execution did not revert:

```cairo
assert is_reverted = 0;
``` [3](#0-2) 

This means the sequencer cannot include a block where an L1 handler runs out of gas and reverts — the Cairo proof would be invalid. The sequencer's only safe option is to pre-mark such a transaction as `is_reverted = TRUE` via the `%{ IsReverted %}` hint, which causes `execute_l1_handler_transaction` to return early **before** `consume_l1_to_l2_message` is called:

```cairo
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [4](#0-3) 

The L1→L2 message is therefore never consumed and never written to the OS output. It remains in the L1 message queue indefinitely with no mechanism to retry it at a higher gas limit, because the gas budget is a compile-time constant with no user-configurable override path.

By contrast, regular invoke transactions allow the sender to specify their own L2 gas bound via `resource_bounds[L2_GAS_INDEX].max_amount`, which is then capped by `EXECUTE_MAX_SIERRA_GAS`:

```cairo
let initial_user_gas_bound = get_initial_user_gas_bound(common_tx_fields=common_tx_fields);
``` [5](#0-4) 

L1 handler senders have no equivalent mechanism.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

L1→L2 messages are the canonical bridge path for moving assets from Ethereum to StarkNet. When a user deposits ETH or ERC-20 tokens via the StarkNet bridge, an L1→L2 message is enqueued and the corresponding L1 handler on L2 is expected to mint or credit the user's L2 balance. If the L1 handler's gas consumption exceeds `L1_HANDLER_L2_GAS_MAX_AMOUNT`, the message can never be executed. The deposited funds on L1 are locked in the bridge contract, and the L2 credit is never issued. There is no admin escape hatch, no retry with higher gas, and no protocol-level refund path — the funds are permanently frozen until a full OS upgrade changes the constant and the OS program hash is updated on L1.

---

### Likelihood Explanation

The likelihood is **medium-to-high** for the following reasons:

1. **Gas schedule changes**: Future OS upgrades may increase per-step or per-builtin gas costs (e.g., `STEP_GAS_COST`, `KECCAK_BUILTIN_GAS_COST`). An L1 handler that fits within 100M gas units today may exceed it after a cost schedule update, with no way to adjust the limit without another full upgrade.
2. **Complex L1 handlers**: Any L1 handler that performs heavy computation (e.g., Keccak, Secp256k1 operations, large storage writes) can approach or exceed the limit. `SECP256K1_MUL_GAS_COST = 8143850` means only ~12 such operations exhaust the entire budget.
3. **Unprivileged trigger**: Any L1 address can send an L1→L2 message. No special role is required to trigger this path. [6](#0-5) 

---

### Recommendation

Replace the compile-time constant `L1_HANDLER_L2_GAS_MAX_AMOUNT` with a value that is either:

1. **Supplied by the L1 message sender** as part of the L1→L2 message payload (analogous to how invoke transactions carry `resource_bounds`), capped by a protocol-level maximum; or
2. **Configurable via the OS global context** (e.g., stored in `BlockContext` or `os_global_context`) so that it can be updated through governance without requiring a full OS program hash upgrade.

This mirrors the fix applied in the referenced report: converting `MAX_SLIPPAGE` from a constant to an admin-settable `maxSlippage` variable.

---

### Proof of Concept

1. Deploy a StarkNet contract whose `@l1_handler` function performs 13 `secp256k1_mul` operations. Each costs `SECP256K1_MUL_GAS_COST = 8_143_850` gas, totalling ~105M — exceeding `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000`.
2. From L1, call the StarkNet bridge's `sendMessageToL2` targeting this contract, depositing ETH.
3. The sequencer simulates the L1 handler and finds it exceeds the gas budget. It sets `is_reverted = TRUE` via `%{ IsReverted %}`.
4. `execute_l1_handler_transaction` returns early at line 387–390 without calling `consume_l1_to_l2_message`.
5. The L1→L2 message is never consumed. The deposited ETH remains locked in the L1 bridge contract. No retry is possible because the gas budget is a compile-time constant — the message is permanently frozen. [7](#0-6) [8](#0-7)
