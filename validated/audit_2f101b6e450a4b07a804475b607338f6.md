### Title
Missing `max_l2_gas_amount` Validation for Declare Transactions Allows Oversized Transactions to Bypass Gateway Admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function explicitly skips the `max_l2_gas_amount` upper-bound check for `Declare` transactions while enforcing it for all other transaction types (`Invoke`, `DeployAccount`). A malicious actor can submit a `Declare` transaction with `max_l2_gas_amount = u64::MAX` (or any value above `config.max_l2_gas_amount`) and have it admitted to the mempool, bypassing the gateway's own admission control.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, the `validate_resource_bounds` method contains the following logic:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` branch is an intentional no-op: it matches Declare transactions and does nothing, falling through without the `max_l2_gas_amount` check. The `else if` branch only executes for non-Declare transactions. The TODO comment confirms this is a known gap.

The `validate_declare_tx` function that is called for Declare transactions validates Sierra version, class length, and entry-point ordering, but does not compensate for the missing gas-amount bound:

```rust
fn validate_declare_tx(
    &self,
    declare_tx: &RpcDeclareTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let contract_class = match declare_tx {
        RpcDeclareTransaction::V3(tx) => &tx.contract_class,
    };
    self.validate_sierra_version(&contract_class.sierra_program)?;
    self.validate_class_length(contract_class)?;
    self.validate_entry_points_sorted_and_unique(contract_class)?;
    Ok(())
}
```

The full gateway admission path in `add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)?` before the stateful validator, so a Declare transaction with an oversized `max_l2_gas_amount` passes stateless validation and proceeds to the mempool.

### Impact Explanation

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

A Declare transaction with `max_l2_gas_amount` far above `config.max_l2_gas_amount` is admitted to the mempool. The maximum fee is `max_l2_gas_amount × max_price_per_unit`; if the attacker sets `max_price_per_unit` to the minimum (1 fri), the fee remains payable while the gas-amount bound is violated. The mempool will queue and propagate the transaction. The bouncer may later reject it at block-building time, but the gateway's own admission invariant — that no transaction with `max_l2_gas_amount > config.max_l2_gas_amount` enters the mempool — is broken for Declare transactions. This asymmetry can be used to flood the mempool with oversized Declare transactions that other transaction types cannot inject.

### Likelihood Explanation

Any unprivileged user can craft a `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount` set to an arbitrary value. No special account, key, or network position is required. The code path is reachable via the public HTTP gateway endpoint (`starknet_addDeclareTransaction`).

### Recommendation

Remove the Declare-specific exemption and apply the same `max_l2_gas_amount` upper-bound check uniformly across all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The TODO comment at line 78–84 of `stateless_transaction_validator.rs` should be resolved by implementing this check rather than deferring it.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with a minimal Sierra contract class (passes `validate_sierra_version`, `validate_class_length`, `validate_entry_points_sorted_and_unique`).
2. Set `resource_bounds.l2_gas.max_amount = u64::MAX` and `resource_bounds.l2_gas.max_price_per_unit = 1`.
3. Submit via `starknet_addDeclareTransaction` to the gateway.
4. The stateless validator reaches `validate_resource_bounds`, enters the `if let RpcTransaction::Declare(_) = tx { }` no-op branch, and returns `Ok(())` without checking `max_l2_gas_amount`.
5. The transaction proceeds to the stateful validator and, if the sender has sufficient balance for the computed fee, is admitted to the mempool — violating the gateway's own `config.max_l2_gas_amount` admission bound that is enforced for every other transaction type. [1](#0-0) [2](#0-1) [3](#0-2)
