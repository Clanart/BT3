### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Contract Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS does not verify that the replacement class hash has been declared. This allows any contract to replace its own class with an arbitrary, undeclared class hash. Once replaced, the contract becomes permanently non-callable, freezing all funds it holds. The missing check is explicitly acknowledged in the source with a `TODO` comment.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the current `StateEntry` for the calling contract and writes a new `StateEntry` with the caller-supplied `class_hash` — without any validation that this hash exists in the `contract_class_changes` dictionary (i.e., that it was ever declared): [1](#0-0) 

The critical lines are:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `TODO` at line 898 is the OS developers' own acknowledgment that this invariant is unenfored. Compare this to the `execute_declare_transaction` path, which correctly enforces that a class can only be declared once by requiring `prev_value=0`: [2](#0-1) 

The `replace_class` path has no equivalent guard. Any felt value — including one never declared — is accepted as the new class hash and committed to state.

The `StateEntry` struct that is written into the Patricia commitment tree is: [3](#0-2) 

Once the undeclared hash is committed, `compute_contract_state_commitment` will hash it into the global state root: [4](#0-3) 

The state root now encodes a contract whose class does not exist. Any future transaction targeting this contract will fail at the proving layer because no valid class implementation can be provided for the undeclared hash. The sequencer cannot include such transactions, so the contract — and all funds it holds — is permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value:

1. The state root commits to a `class_hash` for which no compiled class exists.
2. Any call to the contract requires the prover to supply the class implementation; since none exists, the proof cannot be generated.
3. The sequencer is forced to exclude all transactions targeting this contract.
4. All ERC-20 balances, ETH, or other assets held in the contract's storage are permanently inaccessible.

This is structurally identical to the external report's impact: a state-machine assumption (class hash always points to a declared class) is violated, and the result is an irrecoverable freeze.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is callable by any contract on itself. Realistic attack vectors include:

1. **Malicious deployer pattern**: An attacker deploys a contract that appears to be a legitimate token or escrow, attracts user deposits, then calls a public `upgrade(new_class_hash)` function with an undeclared hash. All deposited funds are frozen.

2. **Missing access control in upgrade patterns**: Many protocols expose an `upgrade` function. If access control is absent or bypassable (a common contract-level bug), an attacker calls it with an undeclared hash. The OS provides no backstop check, so the state is permanently corrupted.

The OS is the last line of defense for protocol-level invariants. The absence of this check means the OS cannot prevent either scenario.

---

### Recommendation

In `execute_replace_class` (`syscall_impls.cairo`), before writing the new `StateEntry`, verify that `class_hash` exists in `contract_class_changes` with a non-zero `compiled_class_hash`. Concretely:

```cairo
// After reading class_hash from request:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);  // Ensures the class was declared.
```

This mirrors the invariant already enforced during declaration (`prev_value=0` in `dict_update`) and closes the gap acknowledged by the TODO comment.

---

### Proof of Concept

1. Declare and deploy a contract `Vault` that holds user funds and exposes:
   ```
   @external
   func upgrade(new_class_hash: felt) {
       replace_class(new_class_hash);  // No validation of new_class_hash
   }
   ```
2. Users deposit funds into `Vault`.
3. Attacker calls `Vault.upgrade(0xdeadbeef)` where `0xdeadbeef` has never been declared.
4. The OS's `execute_replace_class` accepts the call. `contract_state_changes` is updated with `class_hash=0xdeadbeef` for `Vault`'s address.
5. `state_update` squashes and commits this into the global state root via `compute_contract_state_commitment`. [5](#0-4) 

6. Any subsequent transaction calling `Vault` requires the prover to supply the class for `0xdeadbeef`. No such class exists; the proof cannot be generated.
7. The sequencer permanently excludes all calls to `Vault`. All user funds are frozen.

**Analogy to external report**: Just as the Linea bridge assumed a token address could only be native on one chain and provided no check for the other direction — allowing an attacker to exploit the assumption and freeze bridging — the StarkNet OS assumes `replace_class` will only be called with declared hashes and provides no enforcement, allowing an attacker to exploit the assumption and freeze contract funds.
