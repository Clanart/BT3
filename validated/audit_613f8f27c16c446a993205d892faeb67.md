### Title
Gateway Rejects Valid Proof-Facts Transactions at the `STORED_BLOCK_HASH_BUFFER` Boundary Due to State/Block-Number Mismatch — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`, `crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

The gateway increments the block number to N (the next block) for proof-facts validation but reads block-hash storage from the committed state at N-1. The hash of block `N - STORED_BLOCK_HASH_BUFFER` is only written during block N's `pre_process_block`, so it is absent from the N-1 state. A valid Invoke V3 transaction whose proof references block `N - STORED_BLOCK_HASH_BUFFER` passes the block-number check but fails the hash check with a zero-vs-real-hash mismatch, causing the gateway to permanently reject a transaction that would succeed during block N's execution.

---

### Finding Description

**The `STORED_BLOCK_HASH_BUFFER` timing overlap — analog to the TDelay race condition:**

In the external report, `TDelay` governs both when a strategy can be *activated* and when it can be *replaced*, creating an overlap at the boundary. In the sequencer, `STORED_BLOCK_HASH_BUFFER` governs both:

1. **When a proof-facts transaction is valid** (block number check): `proof_block_number <= current_block_number - STORED_BLOCK_HASH_BUFFER`
2. **When the block hash is written to the block-hash contract** (state write): at block N, `pre_process_block` writes the hash of block `N - STORED_BLOCK_HASH_BUFFER`

These two conditions overlap at the boundary: at block N, the hash of block `N - STORED_BLOCK_HASH_BUFFER` is *just being written* AND a proof referencing that block is *just becoming valid*. The gateway reads from the pre-block-N state, so it sees the hash as zero and rejects the transaction.

The hash of block `N - STORED_BLOCK_HASH_BUFFER` is written by `pre_process_block` at the start of block N's execution, **before** any transactions run:

```rust

---

### Impact Explanation

**High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Any Invoke V3 transaction carrying proof facts that reference block `N - STORED_BLOCK_HASH_BUFFER` (the most recent block allowed by the block-number check) will be permanently rejected by the gateway with a spurious "Block hash mismatch" error. The transaction is valid and would succeed during block N's execution, but it never reaches the mempool. This is a direct denial-of-service against the client-side proving feature for users who construct proofs against the most recent eligible block.

---

### Likelihood Explanation

**Medium.** The boundary case (`proof_block_number = N - STORED_BLOCK_HASH_BUFFER`) is the natural choice for a user who wants to prove against the most recent available block. The integration test in `integration_test_proof_flow.rs` explicitly waits for `BLOCK_PAST_HASH_BUFFER` blocks before submitting a proof transaction, suggesting the system designers are aware of the buffer requirement but the off-by-one at the gateway boundary is not guarded. [8](#0-7) 

---

### Recommendation

In `stateful_transaction_validator.rs`, tighten the effective block number used for proof-facts validation to the **committed** block number (N-1) rather than the next block number (N). Concretely, do not increment `block_info.block_number` before constructing the block context used for `validate_proof_facts`, or apply a separate, tighter bound:

```
proof_block_number <= (committed_block_number) - STORED_BLOCK_HASH_BUFFER
```

This ensures the hash of `proof_block_number` is guaranteed to be present in the committed state that the gateway reads from. Alternatively, apply `pre_process_block` to the gateway's cached state before running proof-facts validation so the hash is available.

---

### Proof of Concept

1. Chain is at committed block N-1 (e.g., N = 15, `STORED_BLOCK_HASH_BUFFER` = 10).
2. User constructs an Invoke V3 transaction with `proof_facts` referencing block `N - STORED_BLOCK_HASH_BUFFER = 5` with the correct block hash `H(5)`.
3. User submits to the gateway.
4. Gateway sets `block_number = N = 15`, reads state from committed block N-1 = 14.
5. `validate_proof_block_number`: `5 <= 15 - 10 = 5` → **passes**.
6. `validate_proof_block_hash`: reads `state_14[BLOCK_HASH_CONTRACT][5]` = `Felt::ZERO` (the hash of block 5 is written during block 15's `pre_process_block`, which has not run yet).
7. `Felt::ZERO != H(5)` → transaction **rejected** with `"Block hash mismatch for block 5"`.
8. During block 15's execution: `pre_process_block` writes `H(5)` to `BLOCK_HASH_CONTRACT[5]`, then the same transaction would **pass** `validate_proof_block_hash`.

    old_block_number_and_hash: Option<BlockHashAndNumber>,

        return Err(StateError::OldBlockHashNotProvided);
