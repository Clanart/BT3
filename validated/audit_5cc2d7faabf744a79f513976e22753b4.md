### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` does not validate that the new class hash supplied to the `replace_class` syscall corresponds to a previously declared contract class. This is directly analogous to the external report's vulnerability class: a security invariant check is insufficient — it validates one dimension (gas, address, etc.) but omits the critical identity check (declared class existence). Any contract that exposes an upgrade path accepting an attacker-controlled class hash can be permanently bricked, freezing all funds it holds.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the OS reads the new class hash from the syscall request and unconditionally writes it into `contract_state_changes` for the calling contract:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 is an explicit, in-code acknowledgment that the required check is absent. No lookup into `contract_class_changes` is performed to confirm the hash was ever declared.

Compare this with `execute_declare_transaction` in `transaction_impls.cairo`, which rigorously validates the class hash before writing it:

```cairo
let expected_class_hash = finalize_class_hash(
    contract_class_component_hashes=contract_class_component_hashes
);
with_attr error_message("Invalid class hash pre-image.") {
    assert [class_hash_ptr] = expected_class_hash;
}
// ...
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

Declared classes are stored in `contract_class_changes` with a non-zero `compiled_class_hash`. The `replace_class` path never consults this dictionary, so it accepts any arbitrary felt as a valid class hash.

When the OS subsequently tries to execute the contract (e.g., in `get_invoke_tx_execution_context`, which reads `state_entry.class_hash` and passes it into the entry-point dispatch), the compiled class lookup for the undeclared hash will fail. The contract becomes permanently non-executable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value via `replace_class`, every future call to that contract fails at the OS level during compiled-class resolution. The contract cannot be recovered: there is no mechanism to reset the class hash without the contract itself executing, which it can no longer do. Any ERC-20 tokens, ETH, or other assets held by the contract are permanently inaccessible.

---

### Likelihood Explanation

The `replace_class` syscall is the standard upgrade primitive for Cairo 1 contracts on StarkNet. Upgradeable account contracts and protocol vaults routinely expose an `upgrade(new_class_hash: ClassHash)` entry point that calls `replace_class` with caller-supplied input. If the contract-level validation of `new_class_hash` is absent or bypassable (e.g., the contract only checks ownership, not class existence), an attacker with the ability to call that entry point can supply an undeclared hash. The OS provides no backstop. This is a realistic, low-complexity attack path against any upgradeable contract that does not independently verify class declaration before calling `replace_class`.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, the OS must verify that the hash exists in `contract_class_changes` (i.e., its compiled class hash is non-zero). Concretely, inside `execute_replace_class`, add a `dict_read` on `contract_class_changes` keyed by `request.class_hash` and assert the returned value is non-zero:

```cairo
// Verify the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("replace_class: undeclared class hash.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant already enforced in `execute_declare_transaction` and closes the gap identified by the TODO comment.

---

### Proof of Concept

1. Deploy an upgradeable contract `Vault` that holds user funds and exposes:
   ```cairo
   fn upgrade(new_class_hash: ClassHash) {
       // Only checks caller is owner — does NOT verify new_class_hash is declared.
       replace_class_syscall(new_class_hash).unwrap();
   }
   ```
2. Attacker (or compromised owner) calls `Vault.upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared on-chain.
3. The OS executes `execute_replace_class`: no validation of `0xdeadbeef` against `contract_class_changes` is performed; the state entry for `Vault` is updated with `class_hash = 0xdeadbeef`.
4. In any subsequent block, any invoke transaction targeting `Vault` causes the OS to attempt compiled-class resolution for `0xdeadbeef`. The class is absent from the compiled class facts; execution fails unconditionally.
5. All funds inside `Vault` are permanently frozen with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;
