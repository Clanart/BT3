### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from an unprivileged peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the embedded chunk belongs to `shard_id`. The `verify_path` call proves only that the chunk's hash appears *somewhere* in the block's `chunk_headers_root` Merkle tree; it does not verify the chunk's position (shard index). An attacker can supply a chunk from shard Y with its valid Merkle proof while the node is syncing shard X, causing shard Y's `prev_state_root` to be stored under shard X's key and subsequently installed as shard X's trie state.

### Finding Description

**Entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs four categories of checks:

**Check 1 — `validate_chunk_proofs` (lines 380–385):** [1](#0-0) 

`validate_chunk_proofs` (in `chain/chain/src/validate.rs`) verifies the chunk's internal hash integrity, transaction root, and outgoing receipts root. It does **not** compare `chunk.shard_id()` to the caller-supplied `shard_id`. [2](#0-1) 

**Check 2 — `verify_path` for chunk inclusion (lines 394–403):** [3](#0-2) 

`verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` proves that the chunk's hash is a leaf in the block's Merkle tree. It does **not** verify which leaf index (shard position) the chunk occupies. A chunk from shard Y at index Y passes this check even when `shard_id=X` is passed.

**Check 3 — `verify_path` for `prev_chunk_header` (lines 416–425):** [4](#0-3) 

Same issue: proves membership, not position.

**Check 4 — Receipt proofs (lines 438–510):** [5](#0-4) 

The receipt hash is computed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the caller-supplied `shard_id=X`. An attacker can supply the actual on-chain receipt proofs for shard X (publicly available), satisfying this check independently of which chunk is embedded.

**Check 5 — `validate_state_root_node` (lines 512–523):** [6](#0-5) 

Validates the attacker-supplied `state_root_node` against `chunk_inner.prev_state_root()` — shard Y's state root. The attacker supplies shard Y's actual `state_root_node` (publicly available), so this passes.

**Storage (lines 526–529):** [7](#0-6) 

The header is stored under `StateHeaderKey(shard_id=X, sync_hash)` — using the caller-supplied `shard_id`, not the chunk's actual shard.

**Downstream propagation — `set_state_part` (lines 541–543):** [8](#0-7) 

`set_state_part` reads the stored header for `(shard_id=X, sync_hash)`, extracts `prev_state_root` from the embedded chunk (shard Y's root), and uses it to validate and store state parts. Shard Y's trie is then installed under shard X's `ShardUId`.

### Impact Explanation

A syncing node accepts shard Y's trie as shard X's canonical state. After `set_state_finalize`, shard X's `ChunkExtra` and flat storage are initialized from shard Y's data. All subsequent chunk application for shard X operates on the wrong state, producing wrong state roots, wrong receipts, and wrong transaction outcomes. The node diverges from the canonical chain and cannot re-join without a full resync.

### Likelihood Explanation

State sync is triggered whenever a node falls behind by more than one epoch. Any peer the syncing node connects to for state data can be the attacker — no validator or operator privilege is required. The attacker only needs to serve publicly available on-chain data (chunk Y, its Merkle proof, shard X's receipt proofs, shard Y's state root node) in a crafted `ShardStateSyncResponseHeader`.

### Recommendation

In `set_state_header`, after extracting the chunk, add an explicit shard identity check:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check to `prev_chunk_header.shard_id()`. This should be inserted immediately after `validate_chunk_proofs` succeeds (after line 385) and before the `verify_path` calls. [9](#0-8)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
