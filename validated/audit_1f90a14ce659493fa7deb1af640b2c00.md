### Title
`gc_spice_core_data` Unconditionally Destroys Content-Addressed `UncertifiedExecutionResults` Still Referenced by Live Blocks — (`chain/chain/src/garbage_collection.rs`)

---

### Summary

`gc_spice_core_data` deletes entries from `DBCol::UncertifiedExecutionResults` keyed by `ChunkExecutionResultHash` (a content hash) when GC-ing a block. Because that column is keyed by execution-result content rather than by block hash, the same entry is shared across every block whose chunk produced an identical `ChunkExecutionResult`. When GC removes the entry for an old block, any live (non-GC'd) block that shares the same content hash loses its backing store entry. A subsequent call to `record_block_core_statements` for a live block then hits an unconditional `.expect()` on the now-missing entry and panics, crashing the `SpiceCoreWriterActor`.

---

### Finding Description

`DBCol::UncertifiedExecutionResults` is keyed by `ChunkExecutionResultHash`, which is the hash of the `ChunkExecutionResult` value itself, not by block hash or shard id:

```rust
// core/primitives/src/utils.rs:235-237
pub fn get_uncertified_execution_results_key(hash: &ChunkExecutionResultHash) -> Vec<u8> {
    hash.0.as_ref().to_vec()
}
``` [1](#0-0) 

Entries are written with `insert_ser` (not refcounted) in `save_uncertified_execution_result`:

```rust
// core_writer_actor.rs:121-123
let key = get_uncertified_execution_results_key(&execution_result.compute_hash());
store_update.insert_ser(DBCol::uncertified_execution_results(), &key, &execution_result);
``` [2](#0-1) 

During GC, `gc_spice_core_data` collects every `execution_result_hash` referenced by the endorsements stored for the block being collected, then unconditionally deletes the corresponding `UncertifiedExecutionResults` entry:

```rust
// garbage_collection.rs:764-771
for hash in execution_result_hashes {
    // We cannot gc uncertified execution results right away since there may be
    // duplicates which isn't allowed by store.
    self.gc_col(
        DBCol::uncertified_execution_results(),
        &get_uncertified_execution_results_key(&hash),
    );
}
``` [3](#0-2) 

The comment itself acknowledges the duplicate-key problem but the code deletes the entry anyway. Because the key is content-addressed, any other live block whose chunk produced the same `ChunkExecutionResult` (same state root, same outgoing-receipts root) shares the identical key. After GC removes it, the entry is gone for all referencing blocks.

When a subsequent block is processed that carries endorsements for a live block's chunk, `record_block_core_statements` calls:

```rust
// core_writer_actor.rs:521-522
let execution_result = self.get_uncertified_execution_result(&chunk_execution_result_hash)
    .expect("for each endorsement we should save corresponding uncertified execution result");
``` [4](#0-3) 

`get_uncertified_execution_result` returns `None` because GC already deleted the entry, and the `.expect()` panics.

---

### Impact Explanation

The panic crashes the `SpiceCoreWriterActor` thread. This actor is the sole writer of Spice core state (endorsements, execution results, uncertified-chunk tracking). Its crash halts all further Spice certification processing on the node: no new execution results are certified, no `ExecutionResultEndorsed` notifications are sent to the chunk executor or chunk validator, and the node can no longer participate in Spice consensus. The corrupted invariant is:

> Every `ChunkExecutionResultHash` referenced by a live endorsement in `DBCol::Endorsements` must have a corresponding entry in `DBCol::UncertifiedExecutionResults`.

GC breaks this invariant unconditionally whenever an old block shares a content hash with a live block.

**Severity: High** — node-level liveness failure triggered by normal GC operation on idle shards.

---

### Likelihood Explanation

The trigger condition is two blocks — one old enough to be GC'd (≥ 5 epochs behind head) and one still live — whose chunks for the same shard produced identical `ChunkExecutionResult` values. This is routine for any shard that carries no transactions: consecutive empty chunks have the same state root and empty outgoing-receipts root, so their `ChunkExecutionResult` is byte-for-byte identical and hashes to the same `ChunkExecutionResultHash`. No adversarial action is required; the collision arises from normal chain operation on a quiet shard. GC runs every 500 ms and processes blocks automatically, so the deletion is guaranteed to occur once the old block ages out.

---

### Recommendation

`DBCol::UncertifiedExecutionResults` must not be keyed by content hash if entries are to be deleted per-block. Two options:

1. **Reference-count the column.** Change `DBCol::UncertifiedExecutionResults` to use `GcPolicy::DecrementRefcount`. Each `save_uncertified_execution_result` call increments the refcount; each GC call decrements it. The entry is only physically removed when the refcount reaches zero.

2. **Defer deletion entirely.** Remove the `uncertified_execution_results` deletion from `gc_spice_core_data` and instead run a separate GC pass that scans the column and removes entries whose hash is no longer referenced by any live endorsement in `DBCol::Endorsements`.

Either approach must ensure the invariant holds: an entry in `DBCol::UncertifiedExecutionResults` is present as long as any live endorsement references its hash.

---

### Proof of Concept

**Setup:**
- Shard S carries no transactions for an extended period.
- Block A (height H, epoch E-6) and Block B (height H′, epoch E-1) both produce an empty chunk for shard S. Both chunks have the same state root and empty outgoing-receipts root, so `ChunkExecutionResult` is identical and `hash(R_A) == hash(R_B) == H_R`.

**Steps:**

1. Validators endorse Block B's chunk. `record_chunk_endorsements_with_block` calls `save_uncertified_execution_result(R_B)`, writing `UncertifiedExecutionResults[H_R] = R`. [5](#0-4) 

2. GC runs. Block A is ≥ 5 epochs old. `gc_spice_core_data` iterates Block A's endorsements, collects `H_R`, and calls `gc_col(DBCol::uncertified_execution_results(), key(H_R))`, deleting the shared entry. [6](#0-5) 

3. Block C is produced. It contains endorsements for Block B's chunk (re-emitted because Block B's chunk is still uncertified). `handle_processed_block` → `record_block_core_statements` is called for Block C. [7](#0-6) 

4. Inside `record_block_core_statements`, the endorsements for Block B's chunk accumulate enough signatures to cross the threshold. The code calls:
   ```rust
   let execution_result = self.get_uncertified_execution_result(&H_R)
       .expect("for each endorsement we should save corresponding uncertified execution result");
   ```
   `get_uncertified_execution_result` returns `None` (entry was deleted in step 2). The `.expect()` panics, crashing `SpiceCoreWriterActor`. [4](#0-3)

    fn gc_spice_core_data(&mut self, block_hash: &CryptoHash, shard_layout: &ShardLayout) {
        if !cfg!(feature = "protocol_feature_spice") {
            return;
        }

        for shard_id in shard_layout.shard_ids() {
            let endorsement_keys: Vec<Box<[u8]>> = self
                .store()
                .iter_prefix(
                    DBCol::endorsements(),
                    &get_endorsements_key_prefix(block_hash, shard_id),
                )
                .map(|(key, _)| key)
                .collect();
            let mut execution_result_hashes = HashSet::new();
            for key in endorsement_keys {
                let endorsement: SpiceStoredVerifiedEndorsement =
                    self.store().get_ser(DBCol::endorsements(), &key).unwrap();
                execution_result_hashes.insert(endorsement.execution_result_hash);
                self.gc_col(DBCol::endorsements(), &key);
            }
            for hash in execution_result_hashes {
                // We cannot gc uncertified execution results right away since there may be
                // duplicates which isn't allowed by store.
                self.gc_col(
                    DBCol::uncertified_execution_results(),
                    &get_uncertified_execution_results_key(&hash),
                );
            }
            self.gc_col(
                DBCol::execution_results(),
                &get_execution_results_key(block_hash, shard_id),
            );
        }
        self.gc_col(DBCol::uncertified_chunks(), block_hash.as_ref());
        self.gc_col(DBCol::spice_endorsement_stats(), block_hash.as_ref());
    }
