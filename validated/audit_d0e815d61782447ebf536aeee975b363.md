### Title
P2P State Diff Sync Validates Entry Count But Not Commitment Hash, Enabling Malicious Peer to Inject Wrong State - (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
The P2P sync client assembles a block's `ThinStateDiff` from peer-supplied chunks and validates only that the total entry count matches the block header's `state_diff_length` field. It never verifies the assembled content against the `state_diff_commitment` (Poseidon hash) that is stored in the same block header and committed into the block hash. A malicious peer can supply chunks with the correct count but wrong values — wrong storage slot values, wrong nonces, wrong class hashes — and the syncing node will persist the corrupted state and serve it authoritatively through RPC.

The function reads `state_diff_length` (a raw integer count) from the block header but never reads `state_diff_commitment`. The `state_diff_commitment` is a Poseidon hash of the full state diff content; it is stored in the block header and is chained into the block hash:

```rust
.chain(&block_commitments.state_diff_commitment.0.0)
``` [2](#0-1) 

`unite_state_diffs` rejects duplicate keys but places no constraint on the *values* a peer supplies:

```rust
