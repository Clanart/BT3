### Title
`nativeFee` ETH Permanently Locked in EVM Bridge with No Withdrawal Mechanism - (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

Every user who calls `initTransfer` or `initTransfer1155` on the EVM bridge with a non-zero `nativeFee` sends ETH to the `OmniBridge` contract. That ETH is never forwarded to any recipient and has no withdrawal path, causing it to be permanently locked in the contract.

### Finding Description

In `OmniBridge.sol`, `initTransfer` is a `payable` function that accepts `nativeFee` as part of `msg.value`: [1](#0-0) 

The function computes `extensionValue = msg.value - nativeFee` and passes only `extensionValue` to `initTransferExtension`. In the base contract, `initTransferExtension` reverts if `value != 0`: [2](#0-1) 

This means for the base `OmniBridge`, `extensionValue` must be zero, so `msg.value == nativeFee`. The entire ETH sent by the user is the `nativeFee` — and it stays in the contract. The `nativeFee` value is only emitted in the `InitTransfer` event: [3](#0-2) 

It is never transferred to the relayer, a treasury, or any other address. There is no `withdraw`, `rescueETH`, or equivalent function anywhere in `OmniBridge.sol`.

The same issue applies to `initTransfer1155`: [4](#0-3) 

On the NEAR side, when a relayer calls `claim_fee`, the native fee is paid out by **minting** wrapped ETH tokens on NEAR — not by releasing the ETH locked in the EVM contract: [5](#0-4) 

This confirms the design: the EVM-side ETH is never meant to be released back through the normal fee-claim flow. It accumulates permanently.

### Impact Explanation

Every `nativeFee` ETH paid by bridge users on EVM is irrecoverably locked in the `OmniBridge` contract. There is no on-chain mechanism to recover it. Over time, as bridge usage grows, the locked ETH balance grows without bound. This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

The `nativeFee` parameter is the primary mechanism for users to incentivize relayers to process their cross-chain transfers. Any user who sets `nativeFee > 0` (the normal, intended usage) triggers this loss. This is a routine, unprivileged user action with no special preconditions.

### Recommendation

Add a withdrawal mechanism for accumulated `nativeFee` ETH, restricted to an admin role. Alternatively, forward the `nativeFee` ETH directly to the `feeRecipient` inside `finTransfer` when the origin chain is EVM, or track it in a claimable mapping keyed by nonce so the relayer can withdraw it on EVM after proof submission.

### Proof of Concept

1. User calls `initTransfer(tokenAddress, amount, fee, nativeFee=1 ETH, recipient, message)` with `msg.value = 1 ETH`.
2. `extensionValue = msg.value - nativeFee = 0`. `initTransferExtension` is called with `value=0` (no revert).
3. The 1 ETH remains in the `OmniBridge` contract balance.
4. The `InitTransfer` event is emitted with `nativeFee=1 ETH`.
5. On NEAR, the relayer calls `claim_fee`; `send_fee_internal` mints wrapped ETH tokens on NEAR to the relayer — the EVM ETH is never touched.
6. The 1 ETH is permanently locked. No function in `OmniBridge.sol` can release it. [6](#0-5) [7](#0-6)
