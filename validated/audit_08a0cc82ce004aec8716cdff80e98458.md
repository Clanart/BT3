### Title
Unregistered Token Accepted in `init_transfer` Causes Permanent Fund Lock — (`starknet/src/omni_bridge.cairo` and `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `init_transfer` function on both the Starknet and EVM bridge contracts accepts any arbitrary `token_address` without verifying that the token is registered in the bridge's token mapping. Tokens transferred to the bridge for an unregistered token address are permanently locked: the NEAR side panics when it cannot find the token's decimals, and no recovery path exists.

---

### Finding Description

**Starknet (`starknet/src/omni_bridge.cairo`, lines 281–331):**

`init_transfer` accepts any `token_address`. It only branches on `is_bridge_token` (which checks `starknet_to_near_token.read(token_address).len() > 0`) to decide burn vs. lock, but never asserts that the token is registered in the bridge's `starknet_to_near_token` / `near_to_starknet_token` mappings.

```cairo
fn init_transfer(
    ref self: ContractState,
    token_address: ContractAddress,   // ← no registration check
    amount: u128,
    ...
) {
    ...
    if self.is_bridge_token(token_address) {
        IBridgeTokenDispatcher { contract_address: token_address }.burn(caller, amount.into());
    } else {
        let success = IERC20Dispatcher { contract_address: token_address }
            .transfer_from(caller, get_contract_address(), amount.into());
        assert(success, 'ERR_TRANSFER_FROM_FAILED');
    }
    // Emits InitTransfer event regardless of registration status
    self.emit(Event::InitTransfer(InitTransfer { token_address, ... }))
}
``` [1](#0-0) 

`is_bridge_token` only returns `true` for bridge-deployed tokens; it does **not** cover native tokens registered via `bind_token`. An unregistered token passes the `else` branch, is transferred in, and an `InitTransfer` event is emitted. [2](#0-1) 

**EVM (`evm/src/omni-bridge/contracts/OmniBridge.sol`, lines 373–437):**

The same pattern: `initTransfer` accepts any `tokenAddress` and falls through to `safeTransferFrom` for any token not in `customMinters` or `isBridgeToken`, with no check against `ethToNearToken`. [3](#0-2) 

**NEAR side failure (`near/omni-bridge/src/lib.rs`, lines 715–718):**

When the relayer submits the proof of the `InitTransfer` event to NEAR, `fin_transfer_callback` looks up `token_decimals` for the emitted token address. For an unregistered token this map entry does not exist, causing a hard panic:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);  // panics
``` [4](#0-3) 

The NEAR transaction reverts, the source-chain tokens remain in the bridge contract, and there is no admin rescue function to recover them.

---

### Impact Explanation

Any user who calls `init_transfer` with a token address that has not been registered via `deploy_token` / `bind_token` will have their tokens permanently locked in the bridge contract. The NEAR side cannot process the proof, so the transfer is never completed and the tokens cannot be reclaimed. This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

- `init_transfer` is a fully public, permissionless function on both Starknet and EVM.
- No on-chain guard prevents a user from supplying an unregistered token address.
- A user may attempt to bridge a token that has not yet been registered, or an attacker may socially engineer a victim into calling `init_transfer` with a worthless/unregistered token address, causing the victim to permanently lose their legitimate tokens that were transferred in.
- The EVM `else` branch silently accepts any ERC-20 via `safeTransferFrom`, making the failure mode invisible at the source chain.

---

### Recommendation

Add a registration check at the start of `init_transfer` on both chains:

**Starknet:**
```cairo
// Require token is registered (either bridge token or native registered token)
assert(
    self.starknet_to_near_token.read(token_address).len() > 0,
    'ERR_TOKEN_NOT_REGISTERED'
);
```

**EVM:**
```solidity
require(
    bytes(ethToNearToken[tokenAddress]).length > 0
        || tokenAddress == address(0)
        || multiTokens[tokenAddress].tokenAddress != address(0),
    "Token not registered"
);
```

This mirrors the fix recommended in the original report: validate that the supplied token is one of the known/registered token types before accepting funds.

---

### Proof of Concept

1. Deploy any ERC-20 token `FakeToken` on Starknet (or EVM) that is **not** registered with the bridge.
2. Approve the Starknet bridge to spend `FakeToken`.
3. Call `init_transfer(fake_token_address, 1000, 0, 0, "victim.near", "")`.
4. The bridge accepts the transfer (no revert), emits `InitTransfer`, and increments the nonce.
5. A relayer submits the proof to NEAR `fin_transfer`.
6. NEAR `fin_transfer_callback` panics at `TokenDecimalsNotFound` because `fake_token_address` is absent from `token_decimals`.
7. The 1000 `FakeToken` units are permanently locked in the Starknet bridge with no recovery path. [5](#0-4) [6](#0-5)
