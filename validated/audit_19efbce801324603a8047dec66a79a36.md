### Title
Double-Finalization via `remove_fin_transfer` on `ft_transfer_call` Rejection Allows Same Proof to Be Replayed - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge's `fin_transfer_send_tokens_callback` removes the transfer ID from `finalised_transfers` when an `ft_transfer_call` recipient rejects the tokens. This erasure allows the same cross-chain proof to be submitted a second time, causing the bridge to mint or unlock tokens again from a single origin-chain event ‚Äî a direct double-spend.

### Finding Description

The `process_fin_transfer_to_near` flow executes in three sequential steps:

1. **Mark finalized** ‚Äî `add_fin_transfer` inserts `TransferId { origin_chain, origin_nonce }` into `finalised_transfers` (panics if already present).
2. **Unlock tokens** ‚Äî `unlock_tokens_if_needed` decrements `locked_tokens` for native tokens.
3. **Send tokens** ‚Äî `send_tokens` dispatches `ft_transfer_call` (when `msg` is non-empty) and chains `fin_transfer_send_tokens_callback`. [1](#0-0) 

Inside `fin_transfer_send_tokens_callback`, when `is_refund_required` returns `true` (the receiver's `ft_on_transfer` returned 0, meaning it used zero tokens):

```
burn_tokens_if_needed(...)      // burns returned deployed tokens
revert_lock_actions(...)        // re-locks native tokens
remove_fin_transfer(...)        // ‚Üê ERASES the finalization record
``` [2](#0-1) 

`remove_fin_transfer` calls `self.finalised_transfers.remove(transfer_id)`: [3](#0-2) 

After this removal, `add_fin_transfer` ‚Äî which guards against replay by requiring `finalised_transfers.insert(transfer_id)` to return `true` ‚Äî will succeed again for the identical `TransferId`: [4](#0-3) 

The same Wormhole VAA or light-client proof can therefore be submitted a second time through `fin_transfer`, passing all checks and triggering a second mint/unlock.

### Impact Explanation

**For deployed (bridged) tokens**: On the refund path the returned tokens are burned, reducing supply back to zero. On the second `fin_transfer` call the bridge mints a fresh batch and delivers them to the recipient. The origin-chain event locked only one unit of value; two units of bridged supply are now in circulation ‚Äî unbacked inflation.

**For native tokens**: On the refund path the tokens are re-locked. On the second `fin_transfer` call they are unlocked and sent again. The vault pays out twice for a single origin-chain lock.

Both cases match the allowed impact: *"Cross-chain replay, double-finalization, nonce reuse, or duplicate settlement that enables double-spend or unbacked supply."*

### Likelihood Explanation

The trigger condition is fully attacker-controlled:

1. The attacker deploys a NEAR contract whose `ft_on_transfer` initially returns `U128(0)` (reject all tokens).
2. The attacker initiates a transfer on the EVM side with `msg` pointing to that contract. The `msg` field is user-supplied and passes through the proof verbatim.
3. A relayer submits `fin_transfer`. The bridge finalizes, mints/unlocks, calls `ft_transfer_call`. The malicious contract rejects. The bridge erases the finalization record and emits `FailedFinTransferEvent`.
4. The relayer (observing the failure event) retries, or the attacker ‚Äî if holding a `Role::UnrestrictedRelayer` role ‚Äî submits directly. The bridge finalizes again and delivers tokens a second time.
5. The attacker's contract accepts on the second call.

No privileged key compromise is required. The only external dependency is a relayer retry, which is the expected operational behavior after a `FailedFinTransferEvent`.

### Recommendation

**Do not remove the finalization record on the refund path.** The `TransferId` must remain in `finalised_transfers` permanently once inserted. The refund path should only reverse the token-accounting side effects (re-lock / burn) without touching the replay-protection set.

```rust
// In fin_transfer_send_tokens_callback, refund branch:
self.burn_tokens_if_needed(token.clone(), ...);
self.revert_lock_actions(&lock_actions);
// REMOVE the following line ‚Äî do NOT erase the finalization record:
// self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
```

If storage-cost refund to the relayer is desired, it should be handled separately without touching `finalised_transfers`.

### Proof of Concept

```
1. Attacker deploys MaliciousReceiver on NEAR:
     ft_on_transfer(call 1) ‚Üí return U128(full_amount)  // reject
     ft_on_transfer(call 2) ‚Üí return U128(0)            // accept

2. Attacker calls EVM OmniBridge.initTransfer(token, amount, fee, "near:MaliciousReceiver", msg="trigger")
   ‚Üí EVM emits InitTransfer(origin_nonce=N, amount=X)

3. Relayer submits fin_transfer(proof_of_N):
     add_fin_transfer({Eth, N})          // inserted ‚úì
     unlock_tokens_if_needed(X)          // locked_tokens[Eth][token] -= X
     ft_transfer_call(MaliciousReceiver, X, msg)
     ‚Üí ft_on_transfer returns full_amount (reject)
     ‚Üí fin_transfer_send_tokens_callback:
         revert_lock_actions()           // locked_tokens[Eth][token] += X
         remove_fin_transfer({Eth, N})   // ‚Üê ERASED

4. Relayer retries fin_transfer(same proof_of_N):
     add_fin_transfer({Eth, N})          // succeeds again (was removed)
     unlock_tokens_if_needed(X)          // locked_tokens[Eth][token] -= X (again)
     ft_transfer_call(MaliciousReceiver, X, msg)
     ‚Üí ft_on_transfer returns 0 (accept)
     ‚Üí fin_transfer_send_tokens_callback: success path, tokens delivered

Result: X tokens delivered to attacker from a single origin-chain lock of X.
        For deployed tokens: 2X minted, X burned ‚Üí net X unbacked tokens in supply.
        For native tokens: vault paid out 2X for a single X lock.
```

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
    }

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
