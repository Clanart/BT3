### Title
`state_diff_length` in stored `BlockHeader` not cross-validated against `concatenated_counts` in `convert_sync_block_to_block_data` — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

When a validator receives a `SyncBlock` from a proposer via the consensus path, `convert_sync_block_to_block_data` independently derives `state_diff_length` from `sync_block.state_diff.len()` and stores it in the `BlockHeader`, while the `concatenated_counts` field (which encodes `state_diff_length` as part of the block hash) is taken directly from `header_commitments.concatenated_counts` without any cross-check. A malicious proposer can craft a `SyncBlock` where these two representations of the same quantity diverge. The stored `state_diff_length` then propagates through the node's RPC layer to the transaction prover, which recomputes `concatenated_counts` from the stored header field, producing a wrong block hash and breaking proof verification.

---

### Finding Description

**Root cause — two independent derivations of `state_diff_length` with no equality check:**

In `convert_sync_block_to_block_data`:

```rust
// crates/apollo_p2p_sync/src/client/header.rs  L130-L157
fn convert_sync_block_to_block_data(
    block_number: BlockNumber,
    sync_block: SyncBlock,
) -> SignedBlockHeader {
    let header_commitments = sync_block.block_header_commitments.expect(...);
    let n_events =
        extract_event_count_from_concatenated_counts(&header_commitments.concatenated_counts);
    // TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
    // concatenated_counts.
    SignedBlockHeader {
        block_header: BlockHeader {
            ...
            state_diff_commitment: Some(header_commitments.state_diff_commitment),
            state_diff_length: Some(sync_block.state_diff.len()),   // ← derived from payload
            ...
        },
        ...
    }
}
``` [1](#0-0) 

The `state_diff_length` stored in the header is `sync_block.state_diff.len()`. The `concatenated_counts` field (which encodes `state_diff_length` at bits 64–127 of a packed felt) is taken verbatim from `header_commitments.concatenated_counts` — the proposer's value. The code itself acknowledges the missing check with a TODO comment.

**How `concatenated_counts` encodes `state_diff_length`:**

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L374-L393
pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt {
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),   // ← packed into block hash
        ...
    ].concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}
``` [2](#0-1) 

**How the stored `state_diff_length` is later used to reconstruct `concatenated_counts`:**

The `BlockHeader` type has no `concatenated_counts` field. When downstream code (e.g., the transaction prover) needs `BlockHeaderCommitments`, it reconstructs `concatenated_counts` from the stored header fields:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L139-L166
impl TryFrom<&BlockHeader> for Option<BlockHeaderCommitments> {
    fn try_from(block_header: &BlockHeader) -> Result<Self, Self::Error> {
        ...
        concatenated_counts: concat_counts(
            block_header.n_transactions,
            block_header.n_events,
            state_diff_length,          // ← taken from stored header.state_diff_length
            block_header.block_header_without_hash.l1_da_mode,
        ),
        ...
    }
}
``` [3](#0-2) 

**Transaction prover path — wrong `concatenated_counts` → wrong block hash:**

`BaseBlockInfo::new` in the transaction prover fetches the block header via RPC and recomputes `concatenated_counts` from `header.state_diff_length`:

```rust
// crates/starknet_transaction_prover/src/running/virtual_block_executor.rs  L104-L115
let base_block_header_commitments = BlockHeaderCommitments {
    transaction_commitment: header.transaction_commitment,
    event_commitment: header.event_commitment,
    receipt_commitment: header.receipt_commitment,
    state_diff_commitment: header.state_diff_commitment,
    concatenated_counts: concat_counts(
        header.transaction_count,
        header.event_count,
        header.state_diff_length,   // ← sourced from RPC, which reads stored header
        header.l1_da_mode,
    ),
};
``` [4](#0-3) 

If the stored `state_diff_length` was poisoned by a malicious proposer, the prover computes a wrong `concatenated_counts`, which feeds into `calculate_block_hash` and produces a wrong block hash for the proof.

**P2P state-diff sync termination — wrong chunk count accepted:**

The P2P state-diff sync client uses the stored `state_diff_length` as the termination condition for chunk collection:

```rust
// crates/apollo_p2p_sync/src/client/state_diff.rs  L62-L104
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .state_diff_length
    .ok_or(...)?;

while current_state_diff_len < target_state_diff_len {
    ...
    current_state_diff_len += state_diff_chunk.len();
    unite_state_diffs(&mut result, state_diff_chunk)?;
}

if current_state_diff_len != target_state_diff_len {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffLength { ... }));
}
``` [5](#0-4) 

A poisoned `state_diff_length` causes the sync client to accept too many or too few chunks, storing an incorrect state diff.

---

### Impact Explanation

A malicious proposer (any validator in the rotation) crafts a `SyncBlock` where `sync_block.state_diff.len()` differs from the `state_diff_length` encoded in `header_commitments.concatenated_counts`. The validator stores the header with the wrong `state_diff_length`. This wrong value:

1. **Corrupts the transaction prover's block hash**: `BaseBlockInfo::new` recomputes `concatenated_counts` from the stored `state_diff_length`, producing a wrong block hash that fails L1 proof verification.
2. **Corrupts P2P state-diff sync**: Other nodes syncing from this validator use the wrong `state_diff_length` as the termination condition, accepting an incorrect state diff.
3. **Corrupts RPC responses**: `starknet_getBlockWithTxHashesParams` and related endpoints return the wrong `state_diff_length`, causing any downstream consumer (including the prover) to reconstruct a wrong `concatenated_counts`.

This matches: **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** (wrong state diff stored) and **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** (wrong block hash from prover).

---

### Likelihood Explanation

Any validator in the consensus rotation can become a proposer. The `SyncBlock` is a struct the proposer constructs and sends to validators. There is no signature or commitment binding `sync_block.state_diff.len()` to the `state_diff_length` inside `concatenated_counts` before the header is stored. The missing check is explicitly acknowledged in the codebase with a TODO comment, confirming no guard exists.

---

### Recommendation

In `convert_sync_block_to_block_data`, extract `state_diff_length` from `concatenated_counts` (using the same bit-extraction logic as `extract_event_count_from_concatenated_counts`) and assert it equals `sync_block.state_diff.len()` before constructing the `SignedBlockHeader`. Similarly extract and assert `n_transactions`. Reject the block (treat as bad peer) if either assertion fails.

```rust
let state_diff_length_from_counts =
    extract_state_diff_length_from_concatenated_counts(
        &header_commitments.concatenated_counts
    );
assert_eq!(
    state_diff_length_from_counts,
    sync_block.state_diff.len(),
    "state_diff_length in concatenated_counts does not match state_diff payload length"
);
```

An analogous extraction function for `state_diff_length` (bits 128–191 of the packed felt) should be added alongside the existing `extract_event_count_from_concatenated_counts`. [6](#0-5) 

---

### Proof of Concept

1. Malicious proposer constructs a `SyncBlock` with:
   - `state_diff`: a `ThinStateDiff` with `N` entries (e.g., `N = 5`)
   - `block_header_commitments.concatenated_counts`: a felt encoding `state_diff_length = N + K` (e.g., `K = 10`) via `concat_counts(tx_count, event_count, N + K, l1_da_mode)`

2. Validator calls `convert_sync_block_to_block_data`:
   - Stores `state_diff_length = Some(5)` in the header (from `sync_block.state_diff.len()`)
   - Stores `concatenated_counts` encoding `state_diff_length = 15` (from proposer)

3. Transaction prover fetches the header via RPC, reads `state_diff_length = 5`, computes:
   ```
   concatenated_counts = concat_counts(tx_count, event_count, 5, l1_da_mode)
   ```
   This differs from the on-chain `concatenated_counts` (which encodes 15), producing a wrong block hash.

4. The proof submitted to L1 carries the wrong block hash and fails the on-chain verifier.

pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt {
    let l1_data_availability_byte: u8 = match l1_data_availability_mode {
        L1DataAvailabilityMode::Calldata => 0,
        L1DataAvailabilityMode::Blob => 0b10000000,
    };
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),
        &[l1_data_availability_byte],
        &[0_u8; 7], // zero padding
    ]
    .concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}

pub fn extract_event_count_from_concatenated_counts(concatenated_counts: &Felt) -> usize {
    let bytes = concatenated_counts.to_bytes_be();
    // Extract bytes 8-15 (the event_count field)
    let event_count_bytes: [u8; 8] =
        bytes[8..16].try_into().expect("Felt should have at least 16 bytes");
    u64::from_be_bytes(event_count_bytes).try_into().expect("Expected event count to fit in usize")
}
