### Title
Fee-on-Transfer Token Accounting Discrepancy Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter in the emitted cross-chain event without verifying the actual balance received after `safeTransferFrom`. For fee-on-transfer tokens, the bridge receives `amount - fee` but instructs the NEAR side to credit the full `amount`, permanently undercollateralizing the bridge vault for that token.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native (non-bridge, non-custom-minter) ERC20 tokens by pulling `amount` from the caller: [1](#0-0) 

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Immediately after, the function emits `InitTransfer` and calls `initTransferExtension` — both using the caller-supplied `amount`, not the actual balance delta received by the contract: [2](#0-1) 

For fee-on-transfer tokens (e.g., PAXG, STA, or USDT with fee enabled), `safeTransferFrom(msg.sender, address(this), amount)` deducts a fee mid-transfer, so the bridge contract receives only `amount - transfer_fee`. However, the `InitTransfer` event — which drives the NEAR-side credit — carries the full `amount`.

`logMetadata` is permissionless: [3](#0-2) 

Any caller can register any ERC20 token (including fee-on-transfer tokens) with the bridge, and `initTransfer` imposes no whitelist check on the token address for the native ERC20 path.

In `OmniBridgeWormhole.sol`, `initTransferExtension` publishes the Wormhole message with the same unverified `amount`: [4](#0-3) 

The NEAR bridge (`near/omni-bridge/src/lib.rs`) reads the proven `amount` from the cross-chain message and credits it to the recipient at face value: [5](#0-4) 

When the recipient later bridges back, NEAR burns `amount` tokens and the EVM bridge is instructed to release `amount`, but the vault only holds `amount - transfer_fee`. Each such round-trip widens the collateral gap.

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

Every `initTransfer` call with a fee-on-transfer token inflates the NEAR-side supply relative to the EVM vault balance by exactly the token's transfer fee. After N transfers of size `amount` with fee rate `r`, the vault is short by `N * amount * r` tokens. When users attempt to bridge back, the bridge cannot honor withdrawals, causing permanent loss of funds for later redeemers. This matches the allowed impact: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

**Medium.** Fee-on-transfer tokens (PAXG, STA, and others) exist on mainnet. `logMetadata` is callable by any address with no role restriction, so any user can register such a token. `initTransfer` accepts any ERC20 on the native path without a whitelist. No special privilege or key compromise is required — a regular token holder initiating a bridge transfer is the only actor needed.

---

### Recommendation

Record the vault balance before and after `safeTransferFrom` and use the actual received delta as the canonical `amount` for the cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use `received` (cast to uint128) in place of `amount` for the event and extension call
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported by reverting when `received < amount`.

---

### Proof of Concept

1. Deploy or identify a fee-on-transfer ERC20 token `FOT` with a 1% transfer fee.
2. Call `logMetadata(FOT)` on `OmniBridge` — succeeds with no role check.
3. Approve `OmniBridge` for `1000 FOT`, then call `initTransfer(FOT, 1000, 0, 0, "near:recipient", "")`.
4. `safeTransferFrom` transfers `1000 FOT` but the bridge receives only `990 FOT` (1% fee taken by the token).
5. The emitted `InitTransfer` event carries `amount = 1000`.
6. The NEAR side processes the proof and credits `1000 FOT` to `near:recipient`.
7. `near:recipient` bridges back `1000 FOT`: NEAR burns `1000`, EVM bridge attempts `safeTransfer(recipient, 1000)` but only holds `990` — the transfer reverts or, if repeated across many users, the vault is drained before all redemptions can be honored.
8. Net result: `10 FOT` of unbacked supply per transfer, permanently breaking collateralization.
