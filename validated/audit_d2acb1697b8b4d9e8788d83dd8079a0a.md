### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows cross-shard state header substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` and stores it under `StateHeaderKey(shard_id, sync_hash)` using the **caller-supplied** `shard_id`, but never verifies that the embedded chunk's `shard_id()` matches that parameter. A malicious state-sync peer can serve shard A's chunk (with its valid Merkle proof) in response to a shard B request, causing the header to be stored under shard B's key and all subsequent state-part installation to use shard A's `prev_state_root` for shard B's storage.

### Finding Description

The critical path in `set_state_header`:

**Step 1 — Chunk proof validation** (`validate_chunk_proofs`): [1](#0-0) 

`validate_chunk_proofs` checks the chunk's internal hash consistency, tx root, and receipts root. It does **not** check `chunk.shard_id()` against any expected value.

**Step 2 — Merkle inclusion check** (`verify_path`): [2](#0-1) 

This verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the `chunk_headers_root` tree. The `verify_path` implementation: [3](#0-2) 

`verify_path` reconstructs the Merkle root by walking the path's Left/Right sibling hashes. It proves the leaf is **somewhere** in the tree, but does **not** verify the leaf's position/index corresponds to `shard_id`. There is no `verify_path_with_index` call here (that function exists at line 121 but is not used in `set_state_header`).

An attacker serving shard A's chunk provides shard A's valid `chunk_proof` (which encodes shard A's position via its sibling hashes). `verify_path` passes because shard A's `ChunkHashHeight` genuinely is in the tree at that position. The check is blind to which shard index the proof corresponds to.

**Step 3 — Storage under wrong key**: [4](#0-3) 

The header is stored under `StateHeaderKey(shard_id, sync_hash)` — the **caller-supplied** `shard_id` (e.g., 1), not `chunk.shard_id()` (e.g., 0). There is no guard of the form `if chunk.shard_id() != shard_id { return Err(...) }` anywhere in the function.

**Step 4 — Downstream corruption in `set_state_part`**: [5](#0-4) 

`set_state_part` calls `get_state_header(shard_id=1, sync_hash)`, retrieves the poisoned header (shard 0's chunk), extracts shard 0's `prev_state_root`, validates state parts against that root, and stores them under `StatePartKey(sync_hash, 1, part_id)`. Shard 0's trie nodes are then installed into shard 1's storage.

### Impact Explanation

After finalization, shard 1's `ChunkExtra.state_root` will reflect shard 0's state root. Any subsequent block processing for shard 1 will operate on the wrong trie, causing state root mismatches, failed chunk validation, and effective denial of service for shard 1 on the syncing node.

### Likelihood Explanation

State sync providers are not required to be validators or privileged nodes. Any peer selected as a state sync source (including external storage providers) can serve this malicious response. The syncing node has no other guard between receiving the peer response and calling `set_state_header`.

### Recommendation

Add an explicit shard ID consistency check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD THIS:
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [6](#0-5) 

### Proof of Concept

1. Set up a two-shard test network with a syncing node requesting state for shard 1.
2. Intercept the state sync response and substitute shard 0's `ShardStateSyncResponseHeader` (with shard 0's valid `chunk_proof` against `sync_prev_block_header.chunk_headers_root()`).
3. Call `set_state_header(shard_id=1, sync_hash, shard_0_header)`.
4. Assert `get_state_header(1, sync_hash).chunk().shard_id() == 0` — the wrong chunk is stored.
5. Complete state sync for shard 1 using shard 0's parts.
6. Assert that the installed `ChunkExtra.state_root` for shard 1 does not match the expected value from the canonical chain.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
