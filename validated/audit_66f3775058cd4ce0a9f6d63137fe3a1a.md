### Title
Missing Declared Class Validation in `replace_class` Syscall Enables Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not validate that the new class hash supplied to the `replace_class` syscall corresponds to a previously declared contract class. The OS maintains two separate dictionaries — one for declared classes (`contract_class_changes`) and one for contract state (`contract_state_changes`) — and writes to the latter without checking the former. This is the direct structural analog of M-05: a write to one storage location proceeds without verifying existence in the authoritative companion location. Any contract can exploit this to permanently set its class hash to an undeclared value, bricking itself and freezing all funds it holds.

---

### Finding Description

The StarkNet OS tracks two orthogonal pieces of state in separate dictionaries:

- `contract_class_changes`: maps `class_hash → compiled_class_hash`. A class hash is inserted here only when a valid `declare` transaction is processed. The `dict_update` call in `execute_declare_transaction` enforces `prev_value=0`, meaning a class hash can only be declared once and must not already exist.

- `contract_state_changes`: maps `contract_address → StateEntry`, where `StateEntry` contains `class_hash`, `storage_ptr`, and `nonce`. This is the live per-contract state.

The `replace_class` syscall is implemented in `execute_replace_class`:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The function writes `class_hash` — an attacker-controlled field from the syscall request — directly into `contract_state_changes` with no lookup into `contract_class_changes`. There is no assertion that `class_hash != 0`, no `dict_read` on `contract_class_changes`, and no range or existence check of any kind. The TODO comment is an explicit developer acknowledgment that this check is missing.

This is structurally identical to M-05: the "proposal metadata" (contract state, including class hash) is stored in one location (`contract_state_changes`), while the authoritative "counter" (the set of valid declared class hashes) lives in a separate location (`contract_class_changes`), and the write path does not consult the authoritative source.

By contrast, the `execute_declare_transaction` path correctly enforces `prev_value=0` on `contract_class_changes` to prevent re-declaration, and the `deploy_contract` path asserts `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` before writing. The `replace_class` path has no equivalent guard.

---

### Impact Explanation

If a contract calls `replace_class` with a class hash that was never declared (i.e., has no entry in `contract_class_changes`), the OS commits the new `StateEntry` with the invalid class hash to the global state. From that point forward:

1. Any transaction that invokes the contract causes the OS to look up the compiled class for the stored class hash in the `compiled_class_facts` bundle.
2. No compiled class exists for the undeclared hash; execution cannot proceed.
3. The contract is permanently unexecutable.
4. All funds (ERC-20 balances, NFTs, or any other assets) stored in the contract's storage are permanently inaccessible — no withdrawal, transfer, or recovery path exists.

This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The `replace_class` syscall is available to every Cairo contract without restriction. The attacker-controlled entry path requires no privileged role, no operator cooperation, and no external dependency:

1. A malicious deployer publishes a contract that accepts user deposits (e.g., a fake vault or staking contract).
2. Users deposit funds.
3. The deployer calls `replace_class` with any felt value that was never declared (e.g., `0x1`, or `UNINITIALIZED_CLASS_HASH = 0`).
4. The OS accepts the syscall, commits the invalid class hash, and the contract is permanently bricked.

No signature forgery, no 51% attack, and no privileged key is required. The only precondition is that the attacker controls a deployed contract, which is available to any network participant.

---

### Recommendation

Inside `execute_replace_class`, before writing to `contract_state_changes`, add a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert that the returned compiled class hash is non-zero:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the protection already present in `execute_declare_transaction`, which uses `prev_value=0` to enforce that only previously-absent class hashes can be declared, and closes the gap between the two dictionaries.

---

### Proof of Concept

1. **Deploy** a contract `Vault` that accepts ETH deposits from users. The contract is legitimate-looking and accumulates user funds.
2. **Users deposit** funds into `Vault`; balances are recorded in `Vault`'s storage.
3. **Attacker** (the deployer) sends a transaction from `Vault` that calls the `replace_class` syscall with `class_hash = 0x1` (a felt value that was never passed through `execute_declare_transaction`).
4. **OS execution**: `execute_replace_class` reads `request.class_hash = 0x1`, skips any existence check (the TODO path), and calls `dict_update` on `contract_state_changes` setting `Vault`'s `StateEntry.class_hash = 0x1`. The transaction succeeds and the state is committed.
5. **Post-commit**: Any subsequent transaction targeting `Vault` causes the OS to search `compiled_class_facts` for a compiled class with hash `0x1`. No such entry exists. Execution fails unconditionally.
6. **Result**: All user funds stored in `Vault`'s storage are permanently frozen with no recovery path.
