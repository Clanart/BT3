Audit Report

## Title
Stranded ETH in Router Subsidizes Subsequent User's WETH Input Payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses the router's entire `address(this).balance` as a WETH substitute for any caller whose `tokenIn` is WETH, with no per-user accounting. ETH stranded in the router from a prior user's `multicall{value: X}` (where `refundETH` was omitted) is silently consumed to cover a subsequent user's WETH input, permanently destroying the prior user's ETH.

## Finding Description

`pay` in `PeripheryPayments.sol` handles the `token == WETH, payer != address(this)` case by reading the global `address(this).balance`: [1](#0-0) 

`address(this).balance` carries no attribution to which user deposited it. Any ETH sitting in the router at the time of a WETH-input swap is consumed first, before pulling from the actual payer.

ETH can be stranded because `multicall` is `payable`: [2](#0-1) 

The `receive()` guard only blocks **direct** ETH transfers from non-WETH addresses: [3](#0-2) 

It does **not** block ETH arriving as `msg.value` on a `payable` function like `multicall`. ETH sent via `multicall{value: X}(...)` is held by the contract and remains in `address(this).balance` after the call if `refundETH` is not included. The callback path that triggers `pay` is: [4](#0-3) 

## Impact Explanation

Direct, irreversible loss of User A's ETH principal. User A calls `multicall{value: 1 ether}` for a non-ETH swap; the ETH is never consumed and no `refundETH` is included. User B then calls any WETH-input swap. Inside `pay(WETH, userB, pool, 2 ether)` with `nativeBalance = 1 ether`: the `else if (nativeBalance > 0)` branch wraps and transfers User A's 1 ETH to the pool, then pulls only `2 - 1 = 1 WETH` from User B. User A's 1 ETH is permanently destroyed; User B's swap succeeds with only half the required WETH approved. This is a direct loss of user principal meeting Critical/High Sherlock thresholds.

## Likelihood Explanation

`multicall` is the standard entry point for ETH-input swaps. Omitting `refundETH` is a realistic user or integrator mistake, especially since the `receive()` guard creates a false impression that ETH cannot be accidentally deposited. No front-running or special timing is required — any WETH swap by any user will silently consume stranded ETH. The stranded ETH persists indefinitely until consumed or reclaimed.

## Recommendation

Track per-call ETH attribution using transient storage (EIP-1153), or snapshot `address(this).balance` at the start of each top-level entry point and restrict `pay` to only use the delta introduced by the current call. A simpler mitigation: in `pay`, only use native ETH when `payer == address(this)` (mid-path), and always pull WETH via `transferFrom` from external payers, leaving ETH-wrapping exclusively to the user's own `multicall` context with explicit `refundETH`.

## Proof of Concept

```solidity
// 1. User A strands 1 ETH in the router
vm.prank(userA);
router.multicall{value: 1 ether}(
    [abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        tokenIn: address(token1),   // NOT WETH — ETH is never used
        tokenOut: address(token2),
        amountIn: 1000,
        ...
    })))]
);
// router now holds 1 ETH; userA.balance reduced by 1 ETH permanently

// 2. User B swaps WETH->X with only 1 WETH approved (needs 2)
weth.approve(address(router), 1 ether);  // only 1 WETH approved
vm.prank(userB);
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: address(weth),
    amountIn: 2 ether,
    ...
}));
// pay() uses 1 ETH (userA's) + pulls 1 WETH from userB
// swap succeeds; userA's 1 ETH is gone
assertEq(address(router).balance, 0);
assertEq(weth.balanceOf(userB), initialWeth - 1 ether); // only 1 WETH pulled, not 2
```
