### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay via `meta_tx_v0` Syscall — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes a transaction hash that deliberately omits any nonce or replay-prevention field (`additional_data_size=0`). The `execute_meta_tx_v0` syscall, callable by any unprivileged contract, uses this hash as the `transaction_hash` exposed to the target account's `__execute__` entry point. Because the hash is purely a function of `(contract_address, selector, calldata, chain_id)`, any valid v0 signature observed on-chain can be replayed indefinitely by an attacker to re-execute the same account action — including token transfers — causing direct loss of funds.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash/transaction_hash.cairo` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
``` [1](#0-0) 

The `execute_meta_tx_v0` syscall handler then constructs a `TxInfo` with `nonce=0` (hardcoded) and passes the nonce-free hash as `transaction_hash` to the target contract's `__execute__`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    transaction_hash=meta_tx_hash,
    nonce=0,          // ← always zero, never checked or incremented
    ...
);
``` [2](#0-1) 

No nonce check or increment is performed for the target contract. `check_and_increment_nonce` explicitly skips version-0 transactions:

```cairo
if (tx_info.version == 0) {
    return ();
}
``` [3](#0-2) 

Similarly, `run_validate` (which would run `__validate__`) is also skipped for version 0, so the only signature check is whatever the target account's `__execute__` performs against the nonce-free hash. [4](#0-3) 

The `meta_tx_v0` syscall is reachable from any contract execution via the standard syscall dispatch:

```cairo
assert selector = META_TX_V0_SELECTOR;
execute_meta_tx_v0(block_context=block_context, caller_execution_context=execution_context);
``` [5](#0-4) 

The attacker-controlled inputs are: `contract_address`, `selector`, `calldata`, and `signature` — all supplied inside the `MetaTxV0Request` struct by the calling contract.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any account contract that (a) supports v0 transaction execution in its `__execute__` entry point and (b) verifies the signature against the transaction hash will accept a replayed signature, because the hash is identical for the same `(contract_address, selector, calldata, chain_id)` tuple across all invocations. An attacker who has observed a historical v0 token-transfer signature (all signatures are public on-chain) can replay it via `meta_tx_v0` to drain tokens from the victim account repeatedly.

---

### Likelihood Explanation

**Medium-High.** V0 invoke transactions were the standard before v3 was introduced; many deployed account contracts have historical v0 signatures on-chain. The attacker only needs to:
1. Identify a victim account with a historical v0 transfer signature.
2. Deploy a contract that calls `meta_tx_v0` with the replayed signature and original calldata.
3. Submit a standard v3 transaction to invoke that contract.

No privileged access, leaked keys, or trusted-role compromise is required. The entire attack path is accessible to any unprivileged transaction sender.

---

### Recommendation

Include the nonce in the meta-transaction v0 hash. Specifically, `compute_meta_tx_v0_hash` should pass `additional_data_size=1` and `additional_data=&nonce`, where `nonce` is fetched from the target contract's on-chain state and incremented after each successful `meta_tx_v0` execution — mirroring the pattern used by `check_and_increment_nonce` for regular account transactions. [6](#0-5) 

---

### Proof of Concept

1. Victim account `A` previously sent a v0 invoke transaction transferring 100 tokens to address `B`. The signature `sig_v0` is public on-chain. The v0 hash is `H = hash(prefix, 0, A, __execute__, calldata, 0, chain_id)` — no nonce.

2. Attacker deploys contract `Evil` with the following logic in its `__execute__`:
   ```
   syscall meta_tx_v0(
       contract_address = A,
       selector = __execute__,
       calldata = <original transfer calldata>,
       signature = sig_v0   // replayed from chain history
   )
   ```

3. Attacker submits a v3 transaction calling `Evil.__execute__`.

4. The OS calls `execute_meta_tx_v0`. It computes `meta_tx_hash = compute_meta_tx_v0_hash(A, __execute__, calldata, chain_id)` — identical to the original `H` because no nonce is included.

5. Account `A`'s `__execute__` is invoked with `tx_info.transaction_hash = H` and `tx_info.signature = sig_v0`. The signature verifies correctly (same hash, same signature).

6. The token transfer executes again. The attacker repeats steps 3–6 until `A`'s balance is drained. [1](#0-0) [7](#0-6)

// Verifies that the transaction's nonce matches the contract's nonce and increments the
// latter.
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
