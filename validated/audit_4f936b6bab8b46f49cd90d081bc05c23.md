Audit Report

## Title
Excess native ETH attached to payable swap calls is permanently stranded and claimable by any caller via `refundETH()` - (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
Every swap entry-point on `MetricOmmSimpleRouter` is `payable`, but `pay()` wraps only the exact ETH amount the pool requests, leaving any surplus in the router. Because `refundETH()` is a public, unrestricted function that forwards the entire contract ETH balance to `msg.sender`, any third party can call it after a swap and claim the stranded excess ETH, resulting in direct loss of user principal.

## Finding Description
All four swap functions are `payable`: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

When `tokenIn == WETH`, the swap callback calls `_justPayCallback` → `pay()`. Inside `pay()`, the native ETH branch wraps only exactly `value`: [5](#0-4) 

The condition `nativeBalance >= value` accepts any surplus without reverting. Only `value` is wrapped and forwarded; `nativeBalance - value` remains in the router with no automatic refund.

`refundETH()` has no access control — it sends the entire contract ETH balance to whoever calls it: [6](#0-5) 

The `receive()` guard only blocks plain ETH transfers (no calldata) from non-WETH addresses; it does not prevent ETH from being attached to a function call such as `exactInputSingle{value: X}(...)`: [7](#0-6) 

There is no post-swap refund of excess ETH anywhere in the swap functions. [8](#0-7) 

## Impact Explanation
A user who calls `exactInputSingle{value: X}(params)` with `tokenIn == WETH` and `X > amountIn` loses `X - amountIn` ETH permanently unless they reclaim it via `refundETH()` before a front-runner does. Because `refundETH()` is unrestricted, any observer can call it and redirect the full stranded balance to themselves. This is a direct, unrecoverable loss of user principal — ETH is stolen, not merely locked. This meets the allowed impact gate of "Critical/High/Medium direct loss of user principal."

## Likelihood Explanation
Requires the user to attach more ETH than the swap consumes. This occurs via buggy integrations, manual over-funding, or slippage miscalculations. The intended safe pattern is `multicall([exactInputSingle(...), refundETH()])`, but a direct single call with excess ETH is a realistic user or integrator mistake. Front-running is straightforward: the attacker monitors the mempool for any transaction that leaves ETH in the router, then calls `refundETH()`. Likelihood is Low, but the exploit is trivially repeatable whenever the precondition is met.

## Recommendation
After each swap settles, refund any remaining native ETH balance to `msg.sender` inside each payable swap function:

```solidity
if (address(this).balance > 0) {
    _transferETH(msg.sender, address(this).balance);
}
```

Alternatively, tighten `pay()` to reject surplus ETH:

```diff
- if (nativeBalance >= value) {
+ if (nativeBalance == value) {
```

Or enforce that attached ETH equals exactly the consumed amount at the entry-point level.

## Proof of Concept
1. Pool has a WETH/USDC pair; `amountIn` for a given swap is `1 ETH`.
2. User calls `exactInputSingle{value: 2 ether}(params)` with `tokenIn = WETH`.
3. Pool triggers `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, user, pool, 1e18)`.
4. `pay()` sees `nativeBalance = 2e18 >= 1e18 = value`, wraps `1e18`, transfers to pool. `1e18` ETH remains in router.
5. Attacker observes the transaction (mempool or post-inclusion), calls `refundETH()`.
6. `refundETH()` sends the full `1e18` ETH balance to the attacker.
7. User receives correct swap output but loses `1 ETH` of excess input with no recovery path.
