### Title
Missing state-diff Poseidon-hash guard in P2P sync lets a malicious peer commit a fabricated `compiled_class_hash` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`StateDiffStreamBuilder::parse_data_for_block` accepts a peer-supplied `ThinStateDiff` whose only integrity check is that the assembled chunk-length equals the header's `state_diff_length` field. The `state_diff_commitment` (Poseidon hash) stored in the same block header is never read during state-diff sync. A malicious peer can therefore substitute any `compiled_class_hash` value for a real `class_hash`, have it written to the `compiled_class_hash` table, and cause every downstream consumer (classes trie, SNOS, RPC) to operate on the wrong leaf value.

---

### Finding Description

**Step 1 — length-only guard in `StateDiffStreamBuilder::parse_data_for_block`**

The function reads `state_diff_length` from the stored header and loops until the assembled chunk count matches it. [1](#0-0) 

It then calls `validate_deprecated_declared_classes_non_conflicting` and returns `Ok(Some((result, block_number)))`. [2](#0-1) 

At no point is `calculate_state_diff_hash(&result)` called and compared against `header.state_diff_commitment`. The `state_diff_commitment` field is present in the stored header but is simply ignored.

**Step 2 — fabricated `compiled_class_hash` is written verbatim to storage**

`write_to_storage` calls `append_state_diff` with the unverified diff: [3](#0-2) 

`append_state_diff` writes the attacker-controlled `compiled_class_hash` directly into the `compiled_class_hash` table: [4](#0-3) 

**Step 3 — `ClassStreamBuilder` trusts the already-stored (poisoned) state diff**

`ClassStreamBuilder::parse_data_for_block` reads the stored state diff and uses its `class_hash_to_compiled_class_hash` map only to check whether the key (`class_hash`) is present — the value (`compiled_class_hash`) is never verified: [5](#0-4) [6](#0-5) 

An open TODO in the same file explicitly acknowledges the missing check: [7](#0-6) 

So the attacker sends the correct Sierra body for `class_hash=X`; `ClassStreamBuilder` accepts it (key present in map), and the class manager stores the correct Sierra body — but the `compiled_class_hash` table already holds `WRONG_CCH`.

**Step 4 — concrete corrupted committed value**

`calculate_state_diff_hash` chains `(class_hash, compiled_class_hash)` pairs into the Poseidon hash: [8](#0-7) 

With `WRONG_CCH` stored, `calculate_state_diff_hash` on the stored diff produces a value that differs from the block header's `state_diff_commitment`. The classes trie leaf for `class_hash=X` becomes `Poseidon(CONTRACT_CLASS_LEAF_VERSION, WRONG_CCH)` instead of the correct leaf, corrupting the global state root.

---

### Impact Explanation

The `compiled_class_hash` table is the authoritative source for the classes trie and for execution-time class-hash lookups. A wrong value there means:

- The classes trie leaf for the affected class is `Poseidon(CONTRACT_CLASS_LEAF_VERSION, WRONG_CCH)`, producing a wrong global state root.
- Any SNOS proof generated by this node proves against the wrong leaf — the proof is invalid against the canonical root.
- RPC calls that return `compiled_class_hash` for the affected class return the attacker-chosen value.

This satisfies the **Critical** impact criterion: *Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution.*

---

### Likelihood Explanation

Any peer the syncing node connects to can mount this attack. No operator or validator privilege is required. The attacker only needs to:

1. Serve a `DeclaredClass` chunk with the correct `class_hash` and a fabricated `compiled_class_hash` whose Poseidon-length contribution equals 1 (same as the real entry).
2. Serve the correct Sierra body for that `class_hash` in the subsequent class-sync session.

Both streams are accepted without cryptographic binding to the block header commitment.

---

### Recommendation

In `StateDiffStreamBuilder::parse_data_for_block`, after assembling the full `ThinStateDiff`, read `header.state_diff_commitment` and assert:

```rust
let expected = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_commitment
    .ok_or(/* missing field error */)?;

let actual = calculate_state_diff_hash(&result);
if actual != expected {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffHash { block_number }));
}
```

This mirrors the guard already present in the committer: [9](#0-8) 

Additionally, in `ClassStreamBuilder::write_to_storage`, verify that the `CompiledClassHash` returned by `class_manager_client.add_class` matches the value stored in the state diff for that `class_hash`, closing the secondary gap noted in the TODO.
