### Title
`deployToken` Signed Payload Omits Destination Chain ID, Enabling Cross-Chain Replay of Token Deployment Signatures — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` signed hash used by `deployToken` across all bridge chains (EVM, StarkNet, Solana) does not include a destination chain identifier. In contrast, `finTransfer`'s `TransferMessagePayload` explicitly encodes `omniBridgeChainId` in its signed hash. Because the same NEAR MPC-derived key (`nearBridgeDerivedAddress`) is used across all deployments, a valid `deployToken` signature obtained from one chain can be replayed verbatim on any other chain where the bridge is deployed, deploying the token without per-chain authorization.

---

---

### Impact Explanation

**High — token-mapping corruption and unauthorized token deployment.**

The bridge is explicitly designed for multi-chain deployment (the `omniBridgeChainId` state variable exists for this purpose). Because the same NEAR MPC key signs for all chains, a `deployToken` signature observed on chain A is byte-for-byte valid on chain B.

Concrete harm:

1. **Decimal/accounting mismatch**: The NEAR bridge may intend to deploy token X on Arbitrum with `decimals=6` (after a correction), but an attacker replays the earlier Ethereum signature (which used `decimals=18`) on Arbitrum first. The `ERR_TOKEN_EXIST` guard then permanently blocks the correct deployment. The resulting decimal mismatch breaks bridge collateralization for all subsequent transfers of that token on Arbitrum.

2. **Unauthorized token deployment**: A token is deployed on a chain before the NEAR side has registered the corresponding token mapping for that chain, creating an orphaned bridge token that can receive minted supply without a corresponding locked balance on NEAR.

---

### Likelihood Explanation

All `deployToken` transactions are public on-chain. Any observer can extract `(signatureData, metadata)` from a confirmed transaction on chain A and submit it to the same function on chain B. No special access, key material, or privileged role is required. The bridge is already deployed on multiple EVM chains, making this immediately exploitable.

---

### Recommendation

Include the destination chain identifier in the `MetadataPayload` signed hash, consistent with how `finTransfer` already handles domain separation:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain ID
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `MetadataPayload.to_borsh()` in StarkNet and `DeployTokenPayload.serialize_for_near()` in Solana, and update the NEAR MPC signing path to include the target chain ID when producing `deployToken` signatures.

---

### Proof of Concept

1. NEAR MPC signer produces a `deployToken` signature for token `"usdc.near"` (name=`"USD Coin"`, symbol=`"USDC"`, decimals=`6`) targeting Ethereum (chain ID 1). The signed hash is `keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)`.

2. The transaction is submitted and confirmed on Ethereum. The signature and payload are now public.

3. Attacker calls `deployToken(signatureData, metadata)` on the Arbitrum deployment of `OmniBridge` with the identical `signatureData` and `metadata`.

4. The Arbitrum contract computes the identical hash (no chain ID in the preimage), `ECDSA.recover` returns `nearBridgeDerivedAddress`, and the signature check passes.

5. `nearToEthToken["usdc.near"]` is unset on Arbitrum, so `ERR_TOKEN_EXIST` does not fire.

6. A new `BridgeToken` proxy is deployed on Arbitrum and registered as the canonical bridge token for `"usdc.near"` — without any authorization from the NEAR bridge for the Arbitrum chain.
