### Title
Unchecked Felt Arithmetic in `compute_max_possible_fee` Enables Complete Fee Bypass via Field Modular Overflow — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in `transaction_impls.cairo` performs unchecked felt arithmetic on user-controlled `ResourceBounds` fields (`max_amount`, `max_price_per_unit`, `tip`). Because Cairo felt arithmetic is modulo the Stark prime P, a transaction sender can craft resource bounds whose products sum to exactly 0 (mod P). This causes `charge_fee` to hit its early-exit guard (`if (max_fee == 0) { return (); }`) and skip the ERC-20 fee transfer entirely — the user executes a transaction and pays nothing.

---

### Finding Description

**Root cause — `compute_max_possible_fee`** (lines 87–102):

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
         + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
         + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
``` [1](#0-0) 

`ResourceBounds.max_amount` and `ResourceBounds.max_price_per_unit` are raw `felt` values:

```cairo
struct ResourceBounds {
    resource: felt,
    max_amount: felt,
    max_price_per_unit: felt,
}
``` [2](#0-1) 

No bounds check is applied to these fields anywhere in the OS before they are used in arithmetic. The OS loads them from hints in `get_account_tx_common_fields` and places them directly into `TxInfo`: [3](#0-2) 

**Trigger — `charge_fee`** (lines 111–165):

```cairo
func charge_fee(...) {
    local tx_info: TxInfo* = tx_execution_context.execution_info.tx_info;
    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    if (max_fee == 0) {
        return ();          // ← fee skipped entirely
    }
    ...
    assert_nn_le(calldata.amount.low, max_fee);   // never reached
    ...ERC-20 transfer...
}
``` [4](#0-3) 

The only guard against over-charging is `assert_nn_le(actual_fee, max_fee)`. There is **no lower-bound check** — the analog of the UMA missing minimum collateral ratio. When `max_fee` wraps to 0, the guard is never reached and the fee transfer is silently skipped.

**Concrete overflow construction:**

The Stark prime is P = 2²⁵¹ + 17·2¹⁹² + 1. Choose:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `1` |
| `l1_gas_bounds.max_price_per_unit` | `P − 1` (i.e., felt `-1`) |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas_bounds.max_amount` | `0` |

Sum = `(P−1) + 1 + 0 = P ≡ 0 (mod P)` → `max_fee = 0`.

The user signs the transaction hash (which commits to these exact resource bounds via `hash_fee_fields` inside `hash_tx_common_fields`), so the signature check passes normally. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The fee ERC-20 transfer from the user's account to the sequencer is the protocol's sole mechanism for compensating block producers and preventing spam. Bypassing it means:

1. The sequencer receives zero fee for a fully-executed transaction — direct, provable loss of funds on every such transaction.
2. Because the OS produces a valid STARK proof for the fee-free execution, the on-chain verifier accepts the state transition. The loss is permanent and irreversible.
3. At scale, an attacker can flood the network with zero-cost transactions, exhausting sequencer resources and halting block production.

---

### Likelihood Explanation

**Medium.** The attacker must:
- Know the Stark prime (public knowledge).
- Compute resource bound values whose felt-arithmetic sum is 0 mod P (trivial algebra, shown above).
- Sign the crafted transaction with their own account key (no privileged access needed).

The gateway layer may apply off-chain heuristic limits on `max_price_per_unit`, but the OS itself enforces no such constraint. Any path that delivers a transaction directly to the prover/OS (e.g., a future permissionless proving mode, a custom sequencer, or a gateway misconfiguration) exposes the vulnerability. The OS must be sound independently of the gateway.

---

### Recommendation

Add explicit range checks on `max_amount` and `max_price_per_unit` inside `compute_max_possible_fee` (or at the point of loading in `get_account_tx_common_fields`) to assert each value fits within a safe bound (e.g., `[0, 2^64)` or `[0, 2^128)`):

```cairo
assert_nn_le(l1_gas_bounds.max_amount, MAX_RESOURCE_BOUND);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_RESOURCE_BOUND);
// ... repeat for l2 and l1_data bounds, and tip
```

With each operand bounded to `[0, 2^64)`, the maximum possible sum is `3 · (2^64)^2 = 3 · 2^128 ≪ P`, making field overflow impossible.

---

### Proof of Concept

1. **Craft transaction** with `resource_bounds`:
   - `L1_GAS`: `max_amount = 1`, `max_price_per_unit = P − 1`
   - `L2_GAS`: `max_amount = 1`, `max_price_per_unit = 1`, `tip = 0`
   - `L1_DATA`: `max_amount = 0`, `max_price_per_unit = 0`

2. **Compute transaction hash** (Poseidon over the above fields via `hash_fee_fields`). Sign with the account's private key.

3. **Submit transaction** (invoke, declare, or deploy-account).

4. **OS execution**:
   - `compute_max_possible_fee` evaluates `(1·(P−1)) + (1·(1+0)) + (0·0) = P−1+1 = P ≡ 0 (mod P)`.
   - `charge_fee` hits `if (max_fee == 0) { return (); }` at line 123 and exits.
   - No ERC-20 transfer is executed; the sequencer receives 0 fee.

5. **Proof is valid**: the STARK proof covers the full execution including the fee-skip branch. The on-chain verifier accepts it. The state transition is finalized with zero fee paid. [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
