### Title
Missing shard-index binding in `set_state_header` allows a malicious peer to install shard A's state under shard B's storage key - (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id` but never verifies that the embedded chunk belongs to that shard. `verify_path` only checks that a leaf hash traverses to the Merkle root; it does not check which position (index) in the tree the leaf occupies. A malicious peer can supply shard A's chunk with shard A's valid Merkle proof while claiming the response is for shard B. All validation passes, and the header — carrying shard A's `prev_state_root` — is stored under `StateHeaderKey(shard_id=B, sync_hash)`. Every subsequent `set_state_part` and `set_state_finalize` call then operates on shard A's state root for shard B's storage, corrupting shard B's state.

### Finding Description

**Entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five checks:

1. `validate_chunk_proofs(&chunk, ...)` — verifies the chunk's internal hash, `tx_root`, and `receipts_root`. Does **not** check `chunk.shard_id()`. [1](#0-0) 

2. `verify_path(*sync_prev_block_header.chunk_headers_root(), chunk_proof, &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk hash appears somewhere in the block's `chunk_headers_root` Merkle tree. [2](#0-1) 

3. A similar `verify_path` for `prev_chunk`. [3](#0-2) 

4. Receipt proof chain validation (uses the caller-supplied `shard_id`, not `chunk.shard_id()`). [4](#0-3) 

5. `validate_state_root_node` against `chunk_inner.prev_state_root()`. [5](#0-4) 

**The gap**: `verify_path` is defined as:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [6](#0-5) 

It only checks that the path from the leaf hash reaches the root. It does **not** check which position (left/right sequence) the leaf occupies, i.e., it does not enforce that the leaf is at index `shard_layout.get_shard_index(shard_id_B)`. The position-aware variant `verify_path_with_index` exists but is not used here. [7](#0-6) 

There is no check anywhere in `set_state_header` that `chunk.shard_id() == shard_id`. The header is then stored under the caller-supplied key:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [8](#0-7) 

**Attack construction** (2-shard network, shard A = index 0, shard B = index 1):

- `chunk_headers_root = merklize([ChunkHashHeight(chunk_A_hash, h_A), ChunkHashHeight(chunk_B_hash, h_B)])`
- Attacker sends `ShardStateSyncResponseHeader` with `shard_id=B` in the outer request, but `chunk = chunk_A` (internally valid, shard A's chunk) and `chunk_proof = Merkle path for position 0`.
- Check 1 (`validate_chunk_proofs`): passes — chunk A is internally consistent.
- Check 2 (`verify_path`): `verify_path(chunk_headers_root, path_for_pos_0, ChunkHashHeight(chunk_A_hash, h_A))` → **passes** — chunk A is genuinely at position 0.
- No check that position 0 corresponds to shard B's index (1).
- No check that `chunk.shard_id() == shard_id_B`.
- Receipt proofs for shard B are publicly available on-chain; attacker supplies them correctly → check 4 passes.
- `state_root_node` is shard A's, consistent with shard A's `prev_state_root` → check 5 passes.
- Header stored under `StateHeaderKey(shard_id=B, sync_hash)` carrying shard A's `prev_state_root`.

**Downstream propagation**:

`set_state_part` reads the stored header, extracts shard A's `state_root`, and calls `validate_state_part(shard_id=B, state_root_A, ...)`. The attacker provides shard A's state parts, which are valid against `state_root_A` → passes. [9](#0-8) 

`set_state_finalize` then applies shard A's chunk to shard B's `shard_uid`, writing shard A's trie data into shard B's storage. [10](#0-9) 

**Reachability**: State sync headers are downloaded from peers over the P2P network. Any peer can respond to state sync requests — no validator or operator privilege is required. [11](#0-10) 

### Impact Explanation

A syncing node that accepts a malicious `ShardStateSyncResponseHeader` will install shard A's state into shard B's storage. After `set_state_finalize`, the node's shard B trie contains shard A's data. The node will then operate on a permanently corrupted state for shard B: wrong account balances, wrong contract storage, wrong validator proposals. The node cannot detect this without an independent state root comparison against the canonical chain.

### Likelihood Explanation

Any peer reachable by a syncing node can serve state sync responses. The attacker only needs to:
1. Run a NEAR node connected to the syncing node.
2. Intercept or race the legitimate state sync response for shard B.
3. Construct the malicious header using publicly available block data (chunk A's full chunk and its Merkle proof are derivable from any archive node).

No cryptographic key material, validator stake, or privileged access is required. The attack is deterministic and requires no brute force.

### Recommendation

In `set_state_header`, immediately after extracting the chunk, add an explicit shard identity check:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the bare `verify_path` call with a position-aware check using `verify_path_with_index`, computing the expected index via `shard_layout.get_shard_index(shard_id)` and the total number of shards as `num_merklized_parts`. Apply the same fix to the `prev_chunk_proof` verification. [12](#0-11) 

### Proof of Concept

```rust

// 1. Obtain a valid ShardStateSyncResponseHeader for shard_A from a legitimate node.
// 2. Construct a malicious header:

    chunk: legitimate_header_for_shard_A.take_chunk(),   // shard A's chunk
    chunk_proof: legitimate_header_for_shard_A.chunk_proof().clone(), // proof for position 0
    prev_chunk_header: /* shard A's prev chunk header */,
    prev_chunk_proof: /* shard A's prev chunk proof */,
    incoming_receipts_proofs: /* correct receipts for shard_B (public) */,
    root_proofs: /* correct root proofs for shard_B (public) */,
    state_root_node: /* shard A's state_root_node */,
});
// 3. Call set_state_header with shard_id=B but the malicious header:
let result = chain.state_sync_adapter.set_state_header(shard_id_B, sync_hash, malicious_header);
// Expected (with fix): Err("chunk shard_id does not match")
// Actual (without fix): Ok(()) — header stored under StateHeaderKey(shard_B, sync_hash)
//                        with shard A's prev_state_root
assert!(result.is_err(), "should reject cross-shard header");

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
