### Title
Missing Uniqueness Validation in `batch_insert` Corrupts DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces, allowing a caller to silently write multiple leaf nodes with the same `KeyId` or `Hash` into the blob. This produces an incorrect Merkle root and invalidates all subsequent proofs of inclusion, matching the external report's class of "missing uniqueness validation on a list of inputs leads to Merkle tree corruption."

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When the tree already has ≥ 2 leaves, it pops the **last two** items from the input vector and routes them through `insert()` (which does check for duplicates). All **remaining** items are written directly to the blob in a tight loop with no uniqueness check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

If the input vector contains a repeated `KeyId` (e.g., `[(A,v1,h1), (A,v2,h2), B, C]`), both `A` entries are written as separate leaf nodes. The `block_status_cache.key_to_index` map ends up pointing to only one of them, while the blob contains both. The Merkle root is then computed over a tree that has two leaves for the same key, producing a root that is inconsistent with the cache's view of the tree.

The Python-facing binding passes the input directly without any pre-validation:

```rust
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2) 

### Impact Explanation

A corrupted blob produces a Merkle root that does not correspond to any consistent set of key-value pairs. Any `ProofOfInclusion` generated from the corrupted tree will fail `valid()` because the path hashes will not reconstruct the stored root. This satisfies the allowed High impact: **"DataLayer Merkle proof/blob/delta logic … corrupts tree roots, or lets untrusted input prove invalid state."**

`check_integrity()` would detect the inconsistency (`IntegrityKeyToIndexCacheLength` or `IntegrityLeafHashToIndexCacheLength`), but it is only called explicitly; it is not invoked automatically after `batch_insert` in production paths. [4](#0-3) 

### Likelihood Explanation

The Python binding `py_batch_insert` is exposed to any Python caller of the `chia_rs.datalayer` module. [5](#0-4) 

DataLayer sync code that receives key-value batches from the network and passes them to `batch_insert` without pre-deduplication is the realistic trigger. The attacker only needs to supply a batch containing a repeated `KeyId` or `Hash` in the non-tail portion of the list (i.e., not among the last two items, which are routed through the safe `insert()` path).

### Recommendation

Add uniqueness checks at the start of the bulk loop in `batch_insert`, mirroring the guards in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, pre-deduplicate the input vector before processing, or update the cache incrementally during the loop so that the second occurrence of a duplicate key is caught by the cache lookup.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def make_hash(n):
    return hashlib.sha256(n.to_bytes(8, 'big')).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so batch_insert takes the bulk path
blob.insert(KeyId(100), ValueId(100), make_hash(100))
blob.insert(KeyId(101), ValueId(101), make_hash(101))

# Batch with a duplicate KeyId(1) in the non-tail positions
# Last two items (KeyId(3), KeyId(4)) go through insert() and are fine.
# KeyId(1) appears twice in the bulk loop — no error is raised.
keys_values = [(KeyId(1), ValueId(1)),
               (KeyId(1), ValueId(99)),   # duplicate key, different value/hash
               (KeyId(2), ValueId(2)),
               (KeyId(3), ValueId(3)),
               (KeyId(4), ValueId(4))]
hashes = [make_hash(1), make_hash(99), make_hash(2), make_hash(3), make_hash(4)]

blob.batch_insert(keys_values, hashes)   # succeeds silently
blob.calculate_lazy_hashes()

# check_integrity() will raise because leaf_count != key_to_index_cache_length
blob.check_integrity()   # expected to fail, demonstrating tree corruption
```
