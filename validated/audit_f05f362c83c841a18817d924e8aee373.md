### Title
Missing Zero-Address Recipient Validation in `init_transfer` Allows Permanent Fund Lock - (File: near/omni-bridge/src/lib.rs)

### Summary
The NEAR bridge's `init_transfer` function accepts any `OmniAddress` as recipient without checking `is_zero()`. A user can supply a zero-address recipient (e.g., `eth:0x0000000000000000000000000000000000000000`), causing tokens to be permanently locked on NEAR while the corresponding EVM `finTransfer` either reverts (for ERC20/bridge tokens) or burns ETH (for native ETH), with no recovery path.

### Finding Description

`OmniAddress::is_zero()` is defined in `near/omni-types/src/lib.rs` and correctly identifies zero addresses across all supported chains: [1](#0-0) 

However, the `init_transfer` function in `near/omni-bridge/src/lib.rs` only validates that the recipient chain is not `ChainKind::Near` — it never calls `is_zero()`: [2](#0-1) 

The `InitTransferMsg` struct accepts any `OmniAddress` as recipient with no further constraints: [3](#0-2) 

After `init_transfer` succeeds, the transfer message is stored and the origin tokens are locked. The MPC signer then signs a `TransferMessagePayload` containing the zero-address recipient. On the EVM side, `finTransfer` dispatches to: [4](#0-3) 

- **Bridge tokens**: `IBridgeToken.mint(address(0), amount)` — OpenZeppelin ERC20 reverts on mint to zero address. The transaction reverts, the nonce is not consumed, but the NEAR-side tokens remain locked with no valid finalization path.
- **Native ERC20**: `IERC20.safeTransfer(address(0), amount)` — OpenZeppelin SafeERC20 reverts similarly.
- **Native ETH** (`tokenAddress == address(0)`): `address(0).call{value: amount}("")` succeeds — ETH is permanently burned.

The nonce-before-transfer ordering means ERC20 reverts unwind the nonce mark, but the NEAR-side locked balance is never decremented. The MPC will only ever sign the zero-address payload, so no valid `finTransfer` can ever succeed for that transfer.

### Impact Explanation

**Critical / High — Permanent irrecoverable lock of user funds.**

For ERC20 and bridge token transfers: tokens are locked on NEAR indefinitely. The MPC-signed payload is fixed to the zero-address recipient; no alternative finalization exists. For native ETH transfers: ETH held in the bridge vault is permanently burned. In both cases the `locked_tokens` accounting on NEAR is permanently inflated, corrupting bridge collateralization tracking for that chain/token pair.

### Likelihood Explanation

**Low-to-Medium.** A user must supply a zero-address OmniAddress (e.g., `"eth:0x0000000000000000000000000000000000000000"`) as recipient. This can happen by accident (copy-paste error, UI bug, programmatic integration mistake) or intentionally as a griefing vector against one's own funds. The `OmniAddress::FromStr` parser accepts zero addresses without error: [5](#0-4) 

The Solana SECURITY.md acknowledges an analogous issue ("No validation of `recipient` string in `InitTransferPayload`") as a known low-severity item requiring manual intervention: [6](#0-5) 

The NEAR-side variant is more severe because the recipient is a typed `OmniAddress` (not a raw string), so it parses successfully and proceeds through the full lock flow before the zero-address problem manifests on the destination chain.

### Recommendation

Add a zero-address guard at the top of `init_transfer` in `near/omni-bridge/src/lib.rs`, immediately after the chain-kind check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
require!(
    !init_transfer_msg.recipient.is_zero(),
    BridgeError::InvalidRecipientAddress.as_ref()
);
```

Apply the same guard in the EVM `initTransfer` (validate that `recipient` is a non-empty, non-zero-address string before burning/locking tokens) and in the Starknet `init_transfer`.

### Proof of Concept

1. User calls `ft_transfer_call` on NEAR with:
   ```json
   {
     "receiver_id": "omni-bridge.near",
     "amount": "1000000",
     "msg": "{\"InitTransfer\":{\"recipient\":\"eth:0x0000000000000000000000000000000000000000\",\"fee\":\"0\",\"native_token_fee\":\"0\"}}"
   }
   ```
2. `init_transfer` passes the `get_chain() != Near` check (chain is `Eth`), passes the fee check, and locks 1,000,000 tokens on NEAR.
3. MPC observes the `InitTransferEvent` and signs a `TransferMessagePayload` with `recipient = 0x0000000000000000000000000000000000000000`.
4. Relayer calls `finTransfer` on EVM with the signed payload.
5. For a bridge token: `IBridgeToken.mint(address(0), 1000000)` reverts → transaction reverts → nonce not consumed → relayer retries indefinitely, always reverting.
6. Tokens remain locked on NEAR with no valid finalization path. `locked_tokens[Eth][token]` is permanently inflated by 1,000,000.
