### Title
Uint64 Underflow in `nestedCreates` Calculation Corrupts Sender Nonce on Failed Contract Creation — (File: x/evm/keeper/state_transition.go)

### Summary
In `ApplyMessageWithConfig`, the nonce-restoration logic for contract creation computes `nestedCreates := afterCreateNonce - msg.Nonce - 1` using bare `uint64` arithmetic. When `evm.Create()` fails its internal balance check before it can increment the nonce, `afterCreateNonce` equals `msg.Nonce`, causing a silent integer underflow (`nestedCreates = MaxUint64`). The subsequent `stateDB.SetNonce(sender, oldNonce+nestedCreates, ...)` then wraps around and permanently commits the sender's nonce as `msg.Nonce` instead of `msg.Nonce + 1`, breaking the Ethereum invariant that a nonce is always incremented after a transaction is included in a block.

### Finding Description
In `ApplyMessageWithConfig`, the contract-creation branch explicitly resets the sender's nonce to `msg.Nonce` before calling `evm.Create()` (because the ante handler pre-increments it), then tries to restore the correct post-execution nonce:

```go
oldNonce := stateDB.GetNonce(sender)          // = msg.Nonce + 1 (pre-incremented by ante)
stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to msg.Nonce
ret, _, leftoverGas, vmErr = evm.Create(...)
afterCreateNonce := stateDB.GetNonce(sender)
nestedCreates := afterCreateNonce - msg.Nonce - 1   // ← uint64 subtraction, no underflow guard
stateDB.SetNonce(sender, oldNonce+nestedCreates, ...)
``` [1](#0-0) 

The comment assumes `evm.Create()` always increments the nonce at least once. However, go-ethereum's `evm.create()` performs a `CanTransfer` balance check **before** the nonce increment:

```
if !evm.Context.CanTransfer(evm.StateDB, caller.Address(), value) {
    return nil, common.Address{}, gas, ErrInsufficientBalance   // nonce NOT incremented
}
// ...
evm.StateDB.SetNonce(caller.Address(), nonce+1, ...)            // only reached on success
```

When the balance check fails, `afterCreateNonce` remains equal to `msg.Nonce`. The subtraction `afterCreateNonce - msg.Nonce - 1` underflows to `MaxUint64`. Then:

```
stateDB.SetNonce(sender, (msg.Nonce+1) + MaxUint64, ...)
  = stateDB.SetNonce(sender, msg.Nonce, ...)   // wraps to msg.Nonce, not msg.Nonce+1
```

`stateDB.Commit()` is called unconditionally even when `vmErr != nil`, so this corrupted nonce is persisted to the KV store. [2](#0-1) 

The ante handler (`CheckEthGasConsume`) only verifies and deducts gas fees; it does **not** verify that the sender's post-fee balance covers the transaction's `value` field. A user can therefore craft a contract-creation transaction where `value > balance - gas_fees`, pass the ante handler, and trigger the failure path inside `evm.Create()`. [3](#0-2) 

### Impact Explanation
The sender's nonce is committed as `msg.Nonce` instead of `msg.Nonce + 1` after a failed contract creation. This:

1. **Enables nonce replay**: the attacker can submit a second transaction with the same nonce `msg.Nonce` after the failed one, because the chain's recorded nonce was never advanced.
2. **Breaks the Ethereum nonce invariant**: any wallet, relayer, or contract that relies on the nonce being monotonically increasing after every included transaction will behave incorrectly.
3. **Allows nonce manipulation**: by deliberately triggering the failure, a user can hold their nonce at an arbitrary value and later submit a different transaction with that nonce (e.g., after receiving additional funds), effectively replacing a previously-broadcast transaction that other parties may have been relying on.

This matches the allowed High impact: *"nonce … verification bypass enabling replay … or unauthorized account/code mutation."*

### Likelihood Explanation
The trigger condition is straightforward and requires no special privilege:

- Submit a `MsgEthereumTx` contract-creation with `value` set to slightly more than `balance - (gasLimit × gasPrice)`.
- The ante handler accepts the transaction (gas fee check passes).
- After fee deduction, `balance < value`, so `evm.Create()` returns `ErrInsufficientBalance` without incrementing the nonce.
- The underflow fires and the corrupted nonce is committed.

Any unprivileged Ethereum account can craft such a transaction via the standard JSON-RPC `eth_sendRawTransaction` endpoint.

### Recommendation
Add an explicit underflow guard before computing `nestedCreates`. If `evm.Create()` did not increment the nonce (i.e., `afterCreateNonce <= msg.Nonce`), restore the nonce to `oldNonce` directly rather than attempting the subtraction:
