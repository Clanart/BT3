Audit Report

## Title
Stranded Router ETH Consumed by Subsequent WETH Swap Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's entire native ETH balance — when deciding how much ETH to wrap for a WETH payment. Because every swap entry point is `payable`, any `msg.value` exceeding `amountIn` that is not reclaimed via `refundETH()` remains in the router and is silently consumed by the next caller whose swap triggers the WETH branch of `pay()`, causing direct loss of the original sender's ETH.

## Finding Description

`pay()` in `PeripheryPayments.sol` has three branches for `token == WETH`:

```
nativeBalance = address(this).balance   // entire router balance, not msg.value
if nativeBalance >= value  → wrap from router only, no transferFrom from payer
if nativeBalance > 0       → wrap partial from router, pull remainder from payer
else                       → pull all from payer via transferFrom
``` [1](#0-0) 

The `receive()` guard at lines 32–34 only blocks *direct* ETH pushes (i.e., bare transfers not from WETH). It does **not** block `msg.value` attached to a `payable` function call such as `exactInputSingle{value: X}(...)`. [2](#0-1) 

When Alice calls `exactInputSingle{value: 1 ETH}` with `amountIn = 0.9 ETH`, the swap callback invokes `pay(WETH, alice, pool, 0.9 ETH)`. At that point `address(this).balance = 1 ETH >= 0.9 ETH`, so 0.9 ETH is wrapped and sent to the pool. The remaining 0.1 ETH stays in the router with no accounting of its owner. [3](#0-2) 

When Bob subsequently calls `exactInputSingle{value: 0}` with `amountIn = 0.1 ETH, tokenIn = WETH`, the callback invokes `pay(WETH, bob, pool, 0.1 ETH)`. `address(this).balance = 0.1 ETH >= 0.1 ETH` triggers the first branch: Alice's stranded ETH is wrapped and forwarded to the pool. No `transferFrom` is executed against Bob; he pays nothing. [4](#0-3) 

## Impact Explanation

Direct, permissionless loss of user ETH principal. Alice loses 0.1 ETH; Bob receives a free swap funded by Alice's funds. The attack requires no approvals, no special role, and no coordination. It meets the Sherlock High threshold: direct loss of user principal by an unprivileged caller through a reachable production code path.

## Likelihood Explanation

The precondition — ETH stranded in the router — is realistic and common:

1. Users calling any `payable` entry point with `msg.value > amountIn` and omitting a trailing `refundETH()`.
2. `multicall` batches that include a WETH swap but omit `refundETH()` as the final call. [5](#0-4) 

The router provides no on-chain enforcement that `refundETH()` is called. A griefing bot can monitor the mempool for transactions that leave ETH in the router and front-run the victim's `refundETH()` with a zero-value WETH swap.

## Recommendation

Replace `address(this).balance` in `pay()` with a caller-scoped `msgValue` parameter that is captured at the entry point and threaded through to the callback. Only the ETH the current caller deposited should be eligible for wrapping:

```solidity
// Capture at entry point
uint256 msgValue = msg.value;

// Pass through callback context and into pay()
function pay(address token, address payer, address recipient, uint256 value, uint256 msgValue) internal {
    ...
    } else if (token == WETH) {
        uint256 nativeBalance = msgValue; // only current caller's ETH
        ...
    }
}
```

Alternatively, assert `address(this).balance == msg.value` at the start of every payable entry point and revert if pre-existing ETH is detected.

## Proof of Concept

```solidity
function test_strandedEthStolenByNextSwapper() public {
    // Alice sends 1 ETH but only needs 0.9 ETH
    vm.prank(alice);
    router.exactInputSingle{value: 1 ether}(
        ExactInputSingleParams({ pool: pool, tokenIn: WETH, amountIn: 0.9 ether, ... })
    );
    // Alice omits refundETH(); 0.1 ETH is stranded
    assertEq(address(router).balance, 0.1 ether);

    // Bob swaps with value=0, amountIn=0.1 ETH (tokenIn=WETH)
    // pay() sees nativeBalance=0.1 ETH >= value=0.1 ETH → wraps Alice's ETH
    vm.prank(bob);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({ pool: pool, tokenIn: WETH, amountIn: 0.1 ether, ... })
    );

    // Bob received output without paying; Alice's 0.1 ETH is gone
    assertEq(address(router).balance, 0);
}
```

The root cause is at line 74–77 of `PeripheryPayments.sol`: `address(this).balance` is used without any verification that the balance was deposited by the current caller. [6](#0-5)
