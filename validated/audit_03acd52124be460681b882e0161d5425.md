### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary class hash from the caller without verifying that the hash corresponds to a declared class. This is an exact analog to the external report's vulnerability class: a state-mutating operation that is supposed to accept only valid, protocol-verified inputs instead accepts attacker-controlled arbitrary values. A contract deployer can exploit this to permanently freeze funds held by any contract they control, or that they can induce to call `replace_class`.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` without any validation:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;
    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}
    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. The `class_hash` field is taken verbatim from `request.class_hash` — a value fully controlled by the calling contract — and committed to the state without any cross-reference against the set of declared classes.

The downstream consequence is in `execute_entry_point`, which resolves the class hash through `dict_read` on `contract_class_changes` and then calls `find_element` to locate the compiled class:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

`find_element` asserts that the element exists. If the class hash set via `replace_class` is undeclared, `dict_read` returns 0 (the default for an uninitialized dict entry), and `find_element` with key `0` will fail with an assertion error. This makes it impossible for the sequencer to include any future transaction that calls the bricked contract — the OS cannot generate a valid proof for such a block. The contract becomes permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value:

1. The state transition is committed successfully (the `replace_class` transaction itself is valid and provable).
2. Every subsequent transaction that attempts to call the contract causes `find_element` to assert-fail inside the OS, making proof generation impossible.
3. The sequencer is forced to exclude all such transactions.
4. Any funds (tokens, ETH, etc.) held in the contract's storage are permanently inaccessible — no withdrawal, no recovery.

This is a one-way, irreversible operation at the OS level. There is no recovery path because calling the contract to issue a corrective `replace_class` is itself impossible once the class hash is invalid.

---

### Likelihood Explanation

**Medium.**

The attacker path requires only the ability to deploy a contract (an unprivileged role). A malicious deployer can:

1. Deploy a contract that appears legitimate and collects user deposits.
2. At any chosen moment, call `replace_class` with an arbitrary undeclared hash (e.g., `0xdeadbeef`).
3. The OS commits the invalid class hash to state without complaint.
4. All deposited funds are permanently frozen.

This is also reachable accidentally: any legitimate contract with a bug that allows an attacker to supply an arbitrary argument to `replace_class` (e.g., via a missing access-control check on an upgrade function) is equally vulnerable. The OS provides no safety net.

---

### Recommendation

Implement the check that the TODO comment acknowledges is missing. Before writing the new class hash to `contract_state_changes`, verify that `request.class_hash` exists as a key in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for `request.class_hash` and assert the returned compiled class hash is non-zero before proceeding with the state update.

---

### Proof of Concept

1. Attacker deploys contract `C` at address `A` with a valid class hash `H_valid`.
2. Users deposit funds into `C` (e.g., via a `deposit` entry point).
3. Attacker sends an invoke transaction that calls `C.__execute__`, which internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared hash).
4. The OS executes `execute_replace_class`: reads `request.class_hash = 0xdeadbeef`, skips any validation (the TODO check is absent), and calls `dict_update` to set `C`'s class hash to `0xdeadbeef` in `contract_state_changes`.
5. The block is proven and committed. The on-chain state now records `C`'s class hash as `0xdeadbeef`.
6. A user submits a withdrawal transaction targeting `C`. The sequencer attempts to prove it: `execute_entry_point` calls `dict_read(contract_class_changes, 0xdeadbeef)` → returns `0`; `find_element(..., key=0)` asserts and fails. The OS cannot produce a proof. The sequencer drops the transaction.
7. All funds in `C` are permanently frozen with no recovery path.
