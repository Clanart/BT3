### Title
`fin_transfer_send_tokens_callback` Does Not Detect `mint()` Failure for Deployed Tokens, Causing Permanent Fund Lock - (`near/omni-bridge/src/lib.rs`)

### Summary

In the NEAR bridge contract, when finalizing an inbound transfer to a NEAR recipient for a bridge-deployed token, the transfer is permanently marked as finalized in `finalised_transfers` before the asynchronous `mint()` cross-contract call completes. The callback `fin_transfer_send_tokens_callback` uses `is_refund_required` to detect failure, but this function returns `false` for any failed mint promise — it only checks for refund in the `ft_transfer_call` path and treats all other promise failures as success. If `mint()` panics for any reason, the callback logs `FinTransferEvent` as if the transfer succeeded, the transfer ID is permanently consumed, and the recipient never receives tokens. The source-chain tokens are already burned/locked and cannot be recovered.

### Finding Description

The NEAR `fin_transfer` flow for a NEAR recipient with a bridge-deployed token proceeds as follows:

1. `fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_near`
2. Inside `process_fin_transfer_to_near`, `add_fin_transfer` is called **synchronously**, inserting the `TransferId` into `finalised_transfers` before any token movement occurs.
3. `send_tokens` is called, which for deployed tokens schedules `ext_token::ext(token).mint(recipient, amount, ...)` as a cross-contract call.

**The bug:** When `is_ft_transfer_call = false` (no message, plain mint), `is_refund_required` unconditionally returns `false` without inspecting the promise result at all. When `is_ft_transfer_call = true` (mint with message), a failed promise returns `Err(_)` from `promise_result_checked`, which also maps to `false`. In both cases, a panicking `mint()` is silently treated as success.

The callback then proceeds to the `else` branch: it fires detached fee-mint calls and logs `FinTransferEvent` — as if the transfer completed successfully — while the recipient received nothing. [6](#0-5) 

Because `add_fin_transfer` already inserted the `TransferId` into `finalised_transfers`, any retry of `fin_transfer` with the same proof will panic with `TransferAlreadyFinalised`: [7](#0-6) 

There is no admin escape hatch or recovery path. The transfer is permanently consumed.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

- The user's tokens on the source chain (EVM, Solana, StarkNet) are already burned or locked at the time `fin_transfer` is called.
- If `mint()` panics on NEAR, the destination-side transfer is permanently finalized with no tokens delivered.
- The `TransferId` is consumed; no retry is possible.
- The `FinTransferEvent` is emitted, misleading off-chain indexers into believing the transfer succeeded.
- The user's funds are irrecoverably lost.

### Likelihood Explanation

`mint()` on a NEAR NEP-141 token can panic in several realistic scenarios:

1. **Gas exhaustion**: `send_tokens` allocates `MINT_TOKEN_GAS` for the mint call. If the token contract's `mint` implementation consumes more gas than allocated (e.g., due to an upgrade adding logic, or a complex `ft_transfer_call` receiver), the call panics. The gas budget is computed dynamically and can be zero if upstream gas is tight.
2. **Token contract paused or access-controlled**: If the deployed token contract is paused or its controller is changed, `mint` will panic.
3. **Token contract upgrade**: A breaking upgrade to the token contract that causes `mint` to panic under certain conditions.

Any of these conditions, combined with the fact that the transfer nonce is already consumed, results in permanent fund loss with no recovery path.

### Recommendation

In `fin_transfer_send_tokens_callback`, check the promise result for **all** cases (not just `ft_transfer_call`), and treat a failed promise as a failure requiring recovery. For deployed tokens, if `mint()` failed, the transfer should be removed from `finalised_transfers` so it can be retried, or an alternative recovery mechanism (e.g., a DAO-callable rescue function) should be provided.

Concretely, `is_refund_required` should be extended to also return `true` when the promise at index 0 failed (regardless of `is_ft_transfer_call`), and the refund path in `fin_transfer_send_tokens_callback` should handle the case where no tokens were minted (i.e., skip the `burn_tokens_if_needed` call for deployed tokens since nothing was minted).

### Proof of Concept

1. User bridges 100 USDC from Ethereum to NEAR. Tokens are locked in the EVM bridge.
2. Relayer calls `fin_transfer` on NEAR with a valid proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` inserts `TransferId{origin_chain: Eth, origin_nonce: N}` into `finalised_transfers`. [1](#0-0) 
   - `send_tokens` schedules `ext_token::ext(usdc_near).mint(recipient, 100, None)`. [3](#0-2) 
4. The USDC NEAR token contract panics in `mint` (e.g., gas exhaustion or contract paused).
5. `fin_transfer_send_tokens_callback` is called. `is_ft_transfer_call = false` (no message). `is_refund_required(false)` returns `false` immediately. [8](#0-7) 
6. Callback logs `FinTransferEvent`. Recipient has 0 tokens.
7. Relayer retries `fin_transfer` with the same proof → panics `TransferAlreadyFinalised`. [9](#0-8) 
8. User's 100 USDC is permanently locked in the EVM bridge with no recourse.

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }
