Looking at the external report's vulnerability class — **funds committed on the source side with no refund/failure-propagation mechanism when the destination operation fails** — I need to find an analog in the StarkNet OS where a resource is consumed or committed, but a failure on the "destination" side is silently swallowed, leaving no recovery path.

After reading the execution pipeline, I found a concrete instance in `execute_deploy` inside `syscall_impls.cairo`.

---

### Title
Deploy Syscall Hardcodes `failure_flag=0`, Silently Swallowing Constructor Reverts and Enabling Permanent Fund Loss — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

`execute_deploy` in the StarkNet OS always writes `failure_flag=0` in the syscall response header, regardless of whether the deployed contract's constructor actually reverted. A calling contract that sends tokens to the returned contract address after a "successful" deploy will permanently lose those tokens if the constructor silently failed, because the state was rolled back but the caller was never told.

### Finding Description

In `execute_deploy`, after invoking `deploy_contract` (which runs the constructor and may revert it via the revert log), the OS unconditionally writes a success response:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call is made with `remaining_gas` and the constructor execution context. If the constructor reverts, the revert log mechanism rolls back all state changes (class hash assignment, storage writes) for the new contract address. However, the response written back to the calling contract always has `failure_flag=0` and always includes the computed `contract_address`:

```cairo
assert [response] = DeployResponse(
    contract_address=contract_address,
    constructor_retdata_start=retdata,
    constructor_retdata_end=retdata + retdata_size,
);
``` [2](#0-1) 

The calling contract receives:
- `failure_flag = 0` (success)
- A valid-looking `contract_address`
- The constructor's retdata (which may be an error message, but the caller has no way to know)

The calling contract has **no protocol-level signal** that the deploy failed. It cannot distinguish a successful deploy from a silently-reverted one.

This is structurally identical to the external report's root cause: a resource is committed (tokens sent to the returned address), the destination operation fails (constructor reverts, state rolled back), and there is no mechanism to notify the caller or recover the committed resource.

The revert log correctly undoes the constructor's state changes: [3](#0-2) 

But this rollback is invisible to the calling contract because `failure_flag` is never set to `1`.

For comparison, `contract_call_helper` (used by `call_contract` and `library_call`) correctly propagates the revert flag:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [4](#0-3) 

The `deploy` syscall is the only syscall that does **not** propagate the failure flag.

### Impact Explanation

**Critical — Direct loss of funds.**

A factory contract that follows the common pattern of:
1. Calling `deploy` to create a new contract
2. Transferring tokens to the returned `contract_address`

will transfer tokens to an address whose class hash was rolled back to zero (no code). Those tokens are permanently unrecoverable: there is no contract at that address to handle a withdrawal, and no refund path exists in the OS output. This matches the "permanent freezing / direct loss of funds" impact class.

### Likelihood Explanation

Factory-and-fund patterns are ubiquitous in DeFi (e.g., AMM pair creation, vault deployment). Constructor reverts can be triggered by:
- Out-of-gas conditions (an unprivileged caller controls the gas forwarded)
- User-controlled constructor calldata that fails an assertion
- A class hash whose constructor always reverts under certain conditions

An unprivileged transaction sender can craft a transaction that triggers a constructor revert in any factory contract that accepts user-supplied constructor arguments, causing the factory to fund an undeployed address.

### Recommendation

Remove the hardcoded `failure_flag=0`. Track whether `deploy_contract` resulted in a revert (the revert log already records this) and propagate it:

```cairo
// Replace:
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);

// With:
let constructor_is_reverted = ...; // returned from deploy_contract
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=constructor_is_reverted);
```

The `deploy_contract` function should return an `is_reverted` flag (analogous to `execute_entry_point`'s return value), and the response header should reflect it. Calling contracts can then check `failure_flag` and revert or refund accordingly.

### Proof of Concept

1. Contract A (a factory) calls the `deploy` syscall with class hash X and user-supplied constructor calldata.
2. The constructor of class X reverts (e.g., `assert calldata[0] != 0` fails when attacker passes `0`).
3. The OS revert log rolls back the class hash assignment for `contract_address` — the address now has no code.
4. The OS writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` — reporting success to Contract A.
5. Contract A reads `failure_flag=0`, concludes the deploy succeeded, and calls `transfer(contract_address, 1000_tokens)`.
6. The ERC-20 transfer succeeds (the token contract has no knowledge of the deploy failure).
7. `contract_address` has no code; the 1000 tokens are permanently locked with no recovery path.
