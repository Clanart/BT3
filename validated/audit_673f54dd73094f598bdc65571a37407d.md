### Title
Missing Access Control on `execute_meta_tx_v0` Syscall Allows Any Contract to Drain Arbitrary Account Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall in the StarkNet OS has no restriction on which contract may invoke it. Any unprivileged contract can call this syscall to execute another account's `__execute__` entry point with fully attacker-controlled calldata, while simultaneously bypassing `__validate__` (signature verification) and nonce enforcement. This is a direct analog to the `lockUnits/unlockUnits` missing-access-control class: a privileged state-mutating operation is exposed to any caller without authorization.

---

### Finding Description

`execute_meta_tx_v0` is a syscall handler in the OS that, by design, calls a target contract's `__execute__` entry point with a synthesized version-0 transaction context. The function signature accepts any `caller_execution_context` and performs no check on who the caller is:

```cairo
func execute_meta_tx_v0{...}(
    block_context: BlockContext*,
    caller_execution_context: ExecutionContext*   // ← no restriction on caller
) {
```

The syscall dispatcher in `execute_syscalls.cairo` routes any contract's `META_TX_V0_SELECTOR` syscall directly to this handler with no guard:

```cairo
assert selector = META_TX_V0_SELECTOR;
execute_meta_tx_v0(block_context=block_context, caller_execution_context=execution_context);
```

Inside `execute_meta_tx_v0`, the synthesized execution context is built as follows:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,                                    // ← forces version 0
    account_contract_address=contract_address,    // ← victim's address
    max_fee=0,
    signature_start=request.signature_start,      // ← attacker-supplied
    signature_end=request.signature_end,
    transaction_hash=meta_tx_hash,
    nonce=0,                                      // ← nonce bypassed
    ...
);

tempvar execution_context: ExecutionContext* = new ExecutionContext(
    ...
    execution_info=new ExecutionInfo(
        ...
        caller_address=ORIGIN_ADDRESS,            // ← set to 0 (OS address)
        contract_address=contract_address,        // ← victim
        selector=selector,                        // ← must be __execute__
    ),
    ...
);
```

Three compounding bypasses result from this construction:

**1. `__validate__` is skipped.** In `run_validate` (`execute_transaction_utils.cairo` line 128), version-0 transactions unconditionally return without running signature verification:
```cairo
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
```

**2. Nonce is not checked.** In `check_and_increment_nonce` (`execute_transaction_utils.cairo` line 65), version-0 transactions are also exempted:
```cairo
if (tx_info.version == 0) {
    return ();
}
```

**3. `caller_address = ORIGIN_ADDRESS (0)`.** Standard account contracts (e.g., OpenZeppelin) guard `__execute__` by asserting `get_caller_address() == 0`, which is exactly what the OS normally provides. The synthesized context satisfies this check, so the account's own guard passes.

The only restriction present is that `selector` must equal `EXECUTE_ENTRY_POINT_SELECTOR` (`__execute__`), which is the most sensitive entry point of any account contract.

---

### Impact Explanation

An attacker can drain the entire token balance of any account on the network. The attacker constructs calldata instructing the victim's `__execute__` to transfer all tokens to the attacker's address. Because `__execute__` in standard account contracts does not re-verify the signature (that is `__validate__`'s responsibility, which is bypassed), and because `caller_address == 0` satisfies the account's own guard, the transfer executes unconditionally.

This is **Critical — Direct loss of funds**. Every account on the network is affected simultaneously; no account is safe.

---

### Likelihood Explanation

The attack requires only:
1. Deploying a malicious contract (permissionless on StarkNet).
2. Submitting a single invoke transaction that calls the malicious contract.
3. The malicious contract issues a `meta_tx_v0` syscall targeting any victim.

No privileged role, leaked key, or social engineering is required. The entry path is fully reachable by any unprivileged transaction sender.

---

### Recommendation

Add an explicit caller-authorization check at the top of `execute_meta_tx_v0`. Only contracts that have been explicitly granted the right to issue meta-transactions (e.g., a designated paymaster registry, or the account itself) should be permitted to invoke this syscall. A minimal fix:

```cairo
// Require that the caller is the target contract itself (self-meta-tx only),
// or an allowlisted paymaster address stored in block_context.
with_attr error_message("Unauthorized meta_tx_v0 caller.") {
    assert caller_execution_info.contract_address = contract_address;
}
```

Alternatively, if the syscall is intended for paymasters, introduce an allowlist in `BlockContext` and assert membership before proceeding.

---

### Proof of Concept

1. Attacker deploys `MaliciousContract` with a function `steal(victim, amount)` that issues:
   ```
   syscall: META_TX_V0
     contract_address = victim
     selector        = __execute__
     calldata        = [transfer(attacker, amount)]
     signature       = []   // empty; __validate__ is never called
   ```

2. Attacker submits `invoke(MaliciousContract, steal, [victim_address, victim_balance])`.

3. OS routes the `META_TX_V0` syscall to `execute_meta_tx_v0`:
   - Builds `new_tx_info` with `version=0`, `nonce=0`, `caller_address=0`.
   - Calls `contract_call_helper` → `select_execute_entry_point_func` → victim's `__execute__`.

4. Victim's `__execute__` sees `get_caller_address() == 0` ✓ and executes the transfer calldata, sending all funds to the attacker.

5. `__validate__` was never invoked; the victim's private key was never needed.
