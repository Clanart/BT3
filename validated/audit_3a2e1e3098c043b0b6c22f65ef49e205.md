### Title
`buyToken0InBinSpecifiedIn` fallback at lines 620-622 consumes full trader input without rescaling `out0Scaled`, breaking swap conservation and stealing from the trader — (`metric-core/contracts/libraries/SwapMath.sol`)

---

### Summary

In `SwapMath.buyToken0InBinSpecifiedIn`, a two-step approximation loop (analytical solution + one proportional refinement) is used to find the bin target position for an exact-input swap. When both steps still underestimate the required input, a fallback at lines 620-622 sets `totalIn1Scaled = amountSpecifiedRemainingScaled` **without updating `out0Scaled`**. Because the subsequent proportional-rescale guard at line 624 fires only when `totalIn1Scaled > amountSpecifiedRemainingScaled` (strict greater-than), the now-equal values cause it to be skipped entirely. The pool therefore charges the trader's full specified input while crediting them only the under-computed output, retaining the excess token1 as pool balance. The identical pattern exists in `buyToken1InBinSpecifiedIn` at lines 758-760.

---

### Finding Description

**Control flow in the `else` branch (line 579 onward):**

The outer check at line 577 establishes that the cost to traverse the full bin (`maxFinalBinPos`) **exceeds** `amountSpecifiedRemainingScaled`, so a partial fill is needed. [1](#0-0) 

The analytical solution at lines 581-596 computes an initial `targetPos` and the corresponding `out0Scaled` / `totalIn1Scaled`. [2](#0-1) 

The first refinement at lines 598-618 detects an underestimate (`totalIn1Scaled < amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos`), scales `targetPos` up proportionally, and recomputes both `out0Scaled` and `totalIn1Scaled`. [3](#0-2) 

If the recomputed `totalIn1Scaled` is **still** less than `amountSpecifiedRemainingScaled` and `targetPos` is still below `maxFinalBinPos`, the fallback fires:

```solidity
// lines 620-622
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;   // ← input bumped up
    // out0Scaled is NOT updated
}
``` [4](#0-3) 

After the fallback `totalIn1Scaled == amountSpecifiedRemainingScaled`. The proportional-rescale guard immediately below uses **strict greater-than**:

```solidity
// line 624
if (totalIn1Scaled > state.amountSpecifiedRemainingScaled) {   // FALSE — equal, not greater
    out0Scaled = (out0Scaled * state.amountSpecifiedRemainingScaled) / totalIn1Scaled;  // skipped
    ...
}
``` [5](#0-4) 

The settlement lines then execute with the mismatched pair:

```solidity
state.amountSpecifiedRemainingScaled -= totalIn1Scaled;  // = 0, full input consumed
state.amountCalculatedScaled         += out0Scaled;      // under-computed, not rescaled
``` [6](#0-5) 

The pool's token1 balance is increased by the full input minus protocol fee, while token0 is reduced only by the under-computed `out0Scaled`: [7](#0-6) 

**Why the fallback is reachable:**

The proportional scaling at line 602 is a linear approximation of a non-linear (geometric-mean price) relationship. When the price curve is convex over the interval `[currBinPos, maxFinalBinPos]`, both the analytical quadratic and the single linear refinement can consistently underestimate the required input, leaving `totalIn1Scaled < amountSpecifiedRemainingScaled` after both passes while `targetPos` remains strictly below `maxFinalBinPos`. Specific combinations of bin width (`lengthE6`), fee, and input amount reliably produce this state.

**Symmetric bug in `buyToken1InBinSpecifiedIn`:**

The same pattern appears at lines 758-760 (fallback) and 762-773 (rescale guard), affecting exact-input token0→token1 swaps identically. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

**Direct loss of trader principal.** The trader specifies an exact input amount and expects the maximum achievable output. When the fallback fires, the pool charges the full `amountSpecifiedRemainingScaled` but credits only `out0Scaled` computed for a smaller `targetPos`. The difference — `amountSpecifiedRemainingScaled − original_totalIn1Scaled` — is silently absorbed into the pool's token1 balance with no corresponding token0 given to the trader. This is a swap conservation failure: `amountIn × price > amountOut + fee` within rounding tolerance. The pool's token1 balance grows beyond what LP deposits and prior swaps justify, meaning the excess is effectively extracted from the trader.

---

### Likelihood Explanation

The fallback is a last-resort guard after two approximation passes. It is reachable whenever the price curve's non-linearity causes both the quadratic analytical estimate and the single proportional refinement to underestimate. Wide bins (large `lengthE6`), moderate-to-high fees, and input amounts that land in the middle of a bin are the most likely triggers. Any trader executing a standard exact-input token1→token0 swap through `_swapToken1ForToken0SpecifiedInput` can hit this path without any privileged access or special setup. [10](#0-9) 

---

### Recommendation

Replace the fallback with a correct resolution. Since the invariant at that point is `cost(targetPos) < remaining < cost(maxFinalBinPos)`, the right action is to advance to `maxFinalBinPos` and recompute `out0Scaled`:

```solidity
// Replace lines 620-622 with:
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    targetPos = maxFinalBinPos;
    out0Scaled = calculateOutputToken0FromBinPosition(binState.token0BalanceScaled, currBinPos, targetPos);
    finalPriceX64 = calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Ceil);
    avgPriceX64 = calculateArithmeticMean(startingPriceX64, finalPriceX64);
    in1WithoutFeeScaled = calculateRequiredToken(out0Scaled, avgPriceX64);
    totalIn1Scaled = grossInputWithBinFeeCeil(in1WithoutFeeScaled, onePlusBuyFeeX64);
}
```

This ensures the subsequent `if (totalIn1Scaled > amountSpecifiedRemainingScaled)` guard at line 624 fires correctly and proportionally rescales `out0Scaled` before settlement. Apply the symmetric fix to `buyToken1InBinSpecifiedIn` lines 758-760.

// Foundry unit test sketch
function test_fallback_conservation_violation() public pure {
    // Wide bin, moderate fee, input chosen to land in the underestimate zone
    BinState memory bin = BinState({
        token0BalanceScaled: 1e18,
        token1BalanceScaled: 0,
        lengthE6: 50_000,   // 5% wide bin
        addFeeBuyE6: 0,
        addFeeSellE6: 0
    });
    uint128 lower = uint128(Q64);
    uint128 upper = uint128(Q64 * 105 / 100);
    uint256 feeX64 = Math.mulDiv(3000, Q64, 1e6); // 0.3% fee

    // Pick an input that triggers the fallback (mid-bin, analytical underestimates)
    SwapMath.SwapState memory state = SwapMath.SwapState({
        amountSpecifiedRemainingScaled: /* crafted value */ 5e17,
        amountCalculatedScaled: 0,
        protocolFeeAmountScaled: 0,
        feeExclusiveInputScaled: 0
    });

    uint256 inputBefore = state.amountSpecifiedRemainingScaled;
    (, uint256 out0,,,) = SwapMath.buyToken0InBinSpecifiedIn(
        bin, 0, state, feeX64, lower, upper, type(uint128).max, 0
    );
    uint256 consumed = inputBefore - state.amountSpecifiedRemainingScaled;

    // Conservation invariant: consumed * avgPrice <= out0 + fee (within rounding)
    // If fallback fired: consumed == inputBefore but out0 < expected → assertion fails
    uint256 avgPrice = (lower + upper) / 2;
    assertLe(consumed, (out0 * avgPrice) >> 64 + feeAmount, "Conservation violated");
}
