### Title
Missing `set_withdraw_relayer_address` Exposure in Bridge Controller Permanently Locks Legacy Withdrawal Funds — (`near/omni-token/src/lib.rs`, `near/omni-bridge/src/lib.rs`)

---

### Summary

`OmniToken.set_withdraw_relayer_address()` is gated behind `assert_controller()`, meaning only the bridge contract (the controller) can call it. However, the bridge contract (`near/omni-bridge/src/lib.rs`) exposes **no function** that calls `set_withdraw_relayer_address` on any deployed token. The relayer address therefore can never be set. Any user who invokes the legacy Near Intents withdrawal flow — sending tokens to the token contract with a `WITHDRAW_TO:` memo — will have their tokens silently transferred to the token contract's own balance, where they are permanently irrecoverable.

---

### Finding Description

`OmniToken` implements a legacy bridging flow for Near Intents inside `ft_transfer`:

```rust
// near/omni-token/src/lib.rs  lines 194-207
pub fn ft_transfer(&mut self, receiver_id: AccountId, amount: U128, memo: Option<String>) {
    if receiver_id == env::current_account_id()
        && memo.as_ref().is_some_and(|m| m.starts_with(WITHDRAW_MEMO_PREFIX))
    {
        if let Some(withdraw_relayer) = Self::read_withdraw_relayer_address() {
            return self.token.ft_transfer(withdraw_relayer, amount, memo);
        }
    }
    self.token.ft_transfer(receiver_id, amount, memo);   // ← fallthrough
}
```

When `read_withdraw_relayer_address()` returns `None` (the relayer was never configured), the call falls through and executes `self.token.ft_transfer(token_contract_itself, amount, memo)`. The user's tokens are credited to the token contract's own NEP-141 balance.

The relayer address is set via:

```rust
// near/omni-token/src/lib.rs  lines 113-117
pub fn set_withdraw_relayer_address(&mut self, relayer: &AccountId) {
    self.assert_controller();   // only bridge can call
    env::storage_write(WITHDRAW_RELAYER_ADDRESS, &borsh::to_vec(relayer).unwrap());
}
```

`assert_controller()` enforces `env::predecessor_account_id() == self.controller`, and the controller is always the bridge contract (set at token initialization time by `token-deployer`).

A search of the entire bridge contract (`near/omni-bridge/src/lib.rs`) finds **zero calls** to `set_withdraw_relayer_address`. The bridge exposes `set_token_metadata`, `update_tokens_controller`, `transfer_token_as_dao`, and other token-management helpers, but nothing that reaches `set_withdraw_relayer_address`. The function is therefore permanently unreachable.

The only potential recovery path, `transfer_token_as_dao`, calls `ft_transfer` from the **bridge's** NEP-141 balance — not from the token contract's own balance — so it cannot rescue tokens locked inside the token contract itself.

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds in a token flow.**

Any user who follows the documented Near Intents legacy withdrawal pattern (sending tokens to the token contract address with a `WITHDRAW_TO:` memo) will lose their tokens permanently. The tokens land in the token contract's own NEP-141 balance. No function in the token contract, the bridge, or any other in-scope contract can transfer tokens out of that balance on behalf of the token contract. The lock is irrecoverable without a contract upgrade.

---

### Likelihood Explanation

**Medium.** The legacy flow is explicitly documented in the source code as "Legacy bridging flow used by Near Intents." Near Intents is an active product with real users. Any user or integration that was previously using this withdrawal pattern, or that reads the source code and attempts to use it, will trigger the lock. The attacker-controlled entry path is a standard `ft_transfer` call — no special privilege is required.

---

### Recommendation

Add a privileged function in the bridge contract that forwards the call to `set_withdraw_relayer_address` on a deployed token, analogous to the existing `set_token_metadata` pattern:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn set_token_withdraw_relayer(
    &mut self,
    token: AccountId,
    relayer: AccountId,
) -> Promise {
    require!(self.is_deployed_token(&token), "ERR_NOT_DEPLOYED_TOKEN");
    ext_token::ext(token)
        .with_static_gas(SET_METADATA_GAS)
        .set_withdraw_relayer_address(relayer)
}
```

Alternatively, if the legacy flow is intentionally deprecated, the `set_withdraw_relayer_address` function and the `WITHDRAW_TO:` branch in `ft_transfer` should be removed to prevent accidental fund loss.

---

### Proof of Concept

1. Bridge deploys an `OmniToken` via `token-deployer`. The controller is set to the bridge account. `WITHDRAW_RELAYER_ADDRESS` storage slot is empty.
2. Grep `near/omni-bridge/src/lib.rs` for `set_withdraw_relayer` → **zero matches**. The relayer address can never be set.
3. A Near Intents user calls `ft_transfer` on the token contract with `receiver_id = <token_contract_account>` and `memo = "WITHDRAW_TO:<destination>"`.
4. `read_withdraw_relayer_address()` returns `None`. The `if let Some(...)` branch is skipped.
5. `self.token.ft_transfer(<token_contract_account>, amount, memo)` executes, crediting the user's tokens to the token contract's own NEP-141 balance.
6. No function in the token contract, bridge, or any other in-scope contract can transfer tokens out of the token contract's own balance. Funds are permanently locked.
