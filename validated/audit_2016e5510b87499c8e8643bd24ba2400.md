### Title
Cross-Chain Replay of MPC `deployToken` Signature Enables Unauthorized Token Deployment on Unintended EVM Chains — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary
The `deployToken()` function on all EVM bridge contracts verifies an MPC-produced signature over a `MetadataPayload` that contains only `(PayloadType, token, name, symbol, decimals)`. Unlike `finTransfer()`, which binds the signed hash to the destination chain via `omniBridgeChainId`, the `deployToken` hash includes **no chain identifier**. Because the NEAR MPC key (`nearBridgeDerivedAddress`) is the same across all EVM deployments, a valid signature obtained for deploying a token on one EVM chain is cryptographically valid on every other EVM chain. An unprivileged attacker can replay the signature to deploy the token on chains the protocol never authorized, permanently blocking legitimate future deployment there.

---

### Finding Description

`OmniBridge.deployToken()` constructs the signed hash as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The `MetadataPayload` struct contains no chain discriminator:

```solidity
struct MetadataPayload {
    string token;
    string name;
    string symbol;
    uint8 decimals;
}
``` [2](#0-1) 

The NEAR-side `MetadataPayload` that the MPC signs is equally chain-agnostic:

```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
``` [3](#0-2) 

Contrast this with `finTransfer()`, which **does** embed `omniBridgeChainId` twice in the signed payload, providing chain-binding:

```solidity
bytes1(omniBridgeChainId),   // destination chain
Borsh.encodeAddress(payload.tokenAddress),
...
bytes1(omniBridgeChainId),   // recipient chain
Borsh.encodeAddress(payload.recipient),
``` [4](#0-3) 

The MPC key (`nearBridgeDerivedAddress`) is derived from the same NEAR MPC network for all EVM chains (Ethereum, Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract). A signature produced for one chain is therefore accepted by every other chain's `ECDSA.recover()` check.

The only replay guard in `deployToken` is:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [5](#0-4) 

This guard is per-chain state, so it does not prevent the same signature from being used on a different chain where the token has not yet been deployed.

The same issue exists in the StarkNet `deploy_token()`, where `MetadataPayload.to_borsh()` also omits any chain identifier:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
``` [6](#0-5) 

However, StarkNet uses a different signature scheme than EVM ECDSA, so cross-chain replay between EVM and StarkNet is not directly possible. The EVM-to-EVM replay is the primary attack surface.

---

### Impact Explanation

**Impact: High** — unauthorized token deployment on EVM chains.

An attacker who replays the signature on chain B achieves:

1. A bridge token contract is deployed on chain B with the bridge as owner (`isBridgeToken[proxy] = true`, `nearToEthToken[token] = proxy`).
2. The NEAR bridge has no record of this token on chain B (it was never registered via `bind_token` or `deploy_token` flow for chain B), so it will never produce MPC signatures for transfers to chain B for this token.
3. The legitimate protocol attempt to deploy the same token on chain B later fails permanently with `ERR_TOKEN_EXIST`.
4. The token is permanently undeployable on chain B through the normal bridge flow, freezing that chain's ability to bridge the asset.

This matches the allowed impact: **"Proof, signature, MPC … verification bypass enabling unauthorized … token deployment"** and **"Permanent freezing … of user or protocol funds in bridge … flows"**.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The `deployToken` calldata (including the MPC signature) is fully public on-chain once used on any EVM chain.
- Any observer can extract the `signatureData` and `MetadataPayload` from the transaction and submit them to any other EVM bridge contract.
- No privileged access, leaked key, or colluding MPC signers are required — only reading public blockchain data.
- The attacker's only cost is the gas to call `deployToken` on the target chain.
- The attack is permanent and irreversible once executed.

---

### Recommendation

1. **Include `omniBridgeChainId` in the `MetadataPayload` hash** for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD: chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

2. **Update the NEAR `MetadataPayload`** to include the destination chain kind, and update `log_metadata_callback` to sign per-chain payloads.

3. **Apply the same fix to StarkNet** `MetadataPayload.to_borsh()` by prepending the StarkNet chain ID.

---

### Proof of Concept

1. Call `log_metadata("usdc.near")` on the NEAR bridge. The MPC produces a signature `sig` over `keccak256(borsh(Metadata | "usdc.near" | "USD Coin" | "USDC" | 6))`.

2. A relayer legitimately calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on the **Ethereum** bridge. USDC is now deployed on Ethereum.

3. An attacker reads `sig` and the payload from the Ethereum transaction calldata (fully public).

4. The attacker calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on the **Arbitrum** bridge. `ECDSA.recover(keccak256(borsh(...)), sig) == nearBridgeDerivedAddress` passes because the hash is identical and the MPC key is the same. `isBridgeToken[nearToEthToken["usdc.near"]]` is `false` on Arbitrum, so the guard passes. A USDC bridge token is deployed on Arbitrum.

5. The NEAR bridge has no `token_id_to_address` entry for `usdc.near` on Arbitrum. It will never sign transfers to Arbitrum for USDC.

6. When the protocol later tries to legitimately deploy USDC on Arbitrum, `deployToken` reverts with `ERR_TOKEN_EXIST`. USDC is permanently undeployable on Arbitrum via the bridge.
