### Title
Unauthenticated Bootstrap Bypass in `execute_declare_transaction` Allows Unauthorized Class Declaration — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function contains a special-case bypass path that, when triggered by crafting a declare transaction with `sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and zero resource bounds, **completely skips all authorization checks** — including signature verification (`__validate_declare__`), nonce enforcement, and fee payment — and directly writes an arbitrary class hash into the protocol's class registry. No cryptographic proof of ownership over the sender address is required.

---

### Finding Description

In `transaction_impls.cairo`, `execute_declare_transaction` contains the following logic:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

The four conditions (`sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, `max_possible_fee == 0`) are all **plain felt-value comparisons on attacker-controlled transaction fields**. In Cairo, `'BOOTSTRAP'` is the ASCII-encoded felt `0x424F4F545354524150`. Any transaction sender can set their declare transaction's `sender_address` field to this value.

The normal declare path (lines 779–825) enforces:
1. `check_and_increment_nonce` — verifies the nonce matches the on-chain account state
2. Reading `state_entry` from `contract_state_changes` — verifies the sender has a deployed contract
3. Running `__validate_declare__` — executes the account contract's signature verification
4. `charge_fee` — deducts fees [2](#0-1) 

The bootstrap path skips **all four** of these steps. The `%{ AssertTransactionHash %}` annotation is a hint (not a Cairo constraint), so it adds no soundness guarantee. The only constraint in the bootstrap path is `assert_not_zero(compiled_class_hash)` and `prev_value=0` (class not previously declared). [3](#0-2) 

The `sender_address` is loaded from the hint `%{ DeclareTxFields %}` at line 715, which is sequencer-provided and not independently constrained by the OS before the bootstrap check. [4](#0-3) 

---

### Impact Explanation

A malicious sequencer (or any entity able to inject a transaction into a block) can:

1. **Declare an arbitrary class hash** with an arbitrary `compiled_class_hash` without owning any account contract, without a valid signature, and without paying fees.
2. **Deploy a contract** using the declared malicious class in a subsequent transaction.
3. **Steal funds** from users who interact with the deployed malicious contract (e.g., a fake token, a fake bridge endpoint, or a contract that mimics a trusted protocol contract).

The `prev_value=0` constraint prevents overwriting existing class declarations, but it does not prevent declaring new classes with attacker-chosen hashes. Because `execute_replace_class` also lacks a check that the new class hash is declared (see the `TODO` comment at line 898), a contract that has already been deployed can further replace its class with the attacker-declared hash. [5](#0-4) 

**Impact classification: Critical — Direct loss of funds.** The class registry is a root-of-trust for all contract execution in StarkNet. Unauthorized entries in it enable deployment of malicious contracts that can drain user funds.

---

### Likelihood Explanation

The conditions to trigger the bypass are entirely under the attacker's control at the transaction-field level:

- `sender_address = 0x424F4F545354524150` (felt encoding of `'BOOTSTRAP'`) — any felt value, freely chosen
- `nonce = 0` — freely chosen
- `version = 3` — freely chosen
- All resource bounds set to zero — freely chosen

No private key, no deployed contract, and no cryptographic proof is required. The only external dependency is that the sequencer includes the transaction in a block. A malicious or compromised sequencer can do this trivially. Because the OS is the proof-level enforcer and it accepts this path, the L1 verifier will accept the resulting proof.

---

### Recommendation

Remove the bootstrap bypass entirely, or gate it behind a cryptographic check that is enforced at the Cairo constraint level (not via a hint). If bootstrapping is a legitimate operational need, it should be handled via a separate, explicitly authorized mechanism — for example, by requiring the transaction to be signed by a well-known bootstrapper key whose public key is committed to in the OS config hash (`starknet_os_config_hash`), analogous to how `public_keys_hash` is used for other privileged operations. [6](#0-5) 

---

### Proof of Concept

1. Craft a declare transaction with:
   - `sender_address = 0x424F4F545354524150` (`'BOOTSTRAP'` as a felt)
   - `nonce = 0`
   - `version = 3`
   - `resource_bounds` all set to zero (so `max_possible_fee == 0`)
   - `class_hash` = hash of a malicious Sierra class
   - `compiled_class_hash` = hash of the corresponding malicious CASM

2. Submit the transaction to a sequencer (or inject it directly into a block being proven).

3. The OS enters the bootstrap branch at line 764, skips `check_and_increment_nonce`, skips `__validate_declare__`, skips `charge_fee`, and executes:
   ```cairo
   dict_update{dict_ptr=contract_class_changes}(
       key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
   );
   ```
   registering the malicious class without any authorization.

4. In a subsequent transaction, deploy a contract using the malicious class hash. The deployed contract can now execute arbitrary logic (e.g., drain an ERC-20 allowance from any user who calls it). [1](#0-0)
