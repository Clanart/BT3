### Title
Missing `state_diff_commitment` verification in `StateDiffStreamBuilder::parse_data_for_block` allows malicious P2P peer to persist arbitrary state diff — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`StateDiffStreamBuilder::parse_data_for_block` assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and validates only that the accumulated length equals `header.state_diff_length`. It never computes `calculate_state_diff_hash(&result)` and compares it to `header.state_diff_commitment`. Because `HeaderStreamBuilder::parse_data_for_block` also performs no block-hash or cryptographic-signature verification, a malicious peer fully controls both fields in the stored header. The assembled wrong diff is written unconditionally to the `state_diffs` table and subsequently served by `starknet_getStateUpdate` and consumed by the execution state reader.

---

### Finding Description

**Step 1 — Header accepted without block-hash or signature verification.**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks on the incoming `SignedBlockHeader`:

- block number matches the expected sequence number
- `signatures.len() == 1` [1](#0-0) 

Neither the `block_hash` field nor the `state_diff_commitment` / `state_diff_length` fields are validated against each other or against any cryptographic proof. `write_to_storage` then calls `append_header`, which stores every field verbatim: [2](#0-1) 

An attacker therefore controls the persisted `state_diff_length = N` and `state_diff_commitment = X` for any block number.

---

**Step 2 — State diff stream uses attacker-controlled length as the sole termination condition.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` directly from the stored header: [3](#0-2) 

It then loops, accumulating chunks, until `current_state_diff_len == target_state_diff_len`: [4](#0-3) 

After the loop, the only additional check is `validate_deprecated_declared_classes_non_conflicting`, which is a structural deduplication check, not a commitment check: [5](#0-4) 

**There is no call to `calculate_state_diff_hash` and no comparison against `header.state_diff_commitment`.** The assembled `ThinStateDiff` is returned as `Ok(Some((result, block_number)))` regardless of whether its hash matches the committed value.

---

**Step 3 — Wrong diff written to storage and served by RPC.**

`write_to_storage` calls `append_state_diff` unconditionally: [6](#0-5) 

The `state_diffs` table now contains a `ThinStateDiff` whose Poseidon hash `Y ≠ X = header.state_diff_commitment`. Any subsequent call to `starknet_getStateUpdate` or any execution path that reads storage diffs for this block will receive the attacker-injected content.

---

### Impact Explanation

A single malicious P2P peer can permanently corrupt the `state_diffs` table for any block it is asked to serve. The corrupted diff is then returned authoritatively by `starknet_getStateUpdate` (wrong storage slots, nonces, deployed contracts, declared classes) and is used by the execution state reader for fee estimation, tracing, and simulation. The mismatch between the stored diff and `header.state_diff_commitment` is never detected at rest.

Impact: **High** — RPC returns an authoritative-looking wrong value; execution state reader consumes wrong committed data.

---

### Likelihood Explanation

Any peer that the syncing node connects to and queries for headers and state diffs can mount this attack. No special privilege is required. The peer does not need to be a validator or sequencer. The attack is silent — no error is logged, no peer report is filed, and the wrong data is committed to durable storage.

---

### Recommendation

After the accumulation loop in `StateDiffStreamBuilder::parse_data_for_block`, compute the Poseidon hash of the assembled diff and compare it to the commitment stored in the header:

```rust
let computed = calculate_state_diff_hash(&result);
let expected = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_commitment
    .ok_or(...)?;
if computed != expected {
    return Err(ParseDataError::BadPeer(BadPeerError::StateDiffCommitmentMismatch { ... }));
}
```

Additionally, `HeaderStreamBuilder` should verify the block hash (recompute it from the header fields and compare to `block_header.block_hash`) and verify the consensus signature against that hash before storing the header.

---

### Proof of Concept

```
1. Attacker peer sends SignedBlockHeader for block 0:
     state_diff_length  = 2
     state_diff_commitment = X   (arbitrary, e.g. Felt::from(0xdeadbeef))
     block_hash         = anything (not verified)
     signatures         = [BlockSignature::default()]  (length check passes)

2. HeaderStreamBuilder accepts and stores the header.
   Storage now has: header[0].state_diff_length = 2, header[0].state_diff_commitment = X.

3. StateDiffStreamBuilder queries the attacker for block 0's state diff.
   Attacker sends two StateDiffChunk messages (total len = 2) whose union is:
     deployed_contracts: { 0xBAD => ClassHash(0xBAD) }
   calculate_state_diff_hash(assembled) = Y ≠ X.

4. parse_data_for_block returns Ok(Some((assembled_diff, BlockNumber(0)))).
   write_to_storage calls append_state_diff(0, assembled_diff).commit().

5. starknet_getStateUpdate(block_id=0) returns the attacker-injected diff.
   assert calculate_state_diff_hash(stored_diff) != header.state_diff_commitment  // Y ≠ X
```
