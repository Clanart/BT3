### Title
Strict Boolean Return Check on `IERC20Dispatcher` Causes Permanent Fund Freeze for Non-Compliant Tokens in `fin_transfer` - (File: starknet/src/omni_bridge.cairo)

---

### Summary

The StarkNet `OmniBridge` contract uses `IERC20Dispatcher` for ERC20 token transfers in both `fin_transfer` and `init_transfer`. This dispatcher strictly expects a `bool` return value from `transfer` and `transfer_from`. Tokens that do not return a boolean (e.g., older StarkNet tokens using the pre-SNIP-2 felt252-based standard with no return value) will cause the dispatcher to panic. In `fin_transfer`, this results in a permanent freeze of user funds already locked or burned on the source chain (NEAR, EVM, or Solana), because the finalization can never succeed for that token.

---

### Finding Description

In `starknet/src/omni_bridge.cairo`, the `fin_transfer` function handles incoming cross-chain transfers for non-bridge tokens by calling:

```cairo
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');
```

Similarly, `init_transfer` calls:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
```

`IERC20Dispatcher` is the OpenZeppelin Cairo standard dispatcher, which expects `transfer` and `transfer_from` to return a `bool` (serialized as a single felt252). If the target token contract does not return any value — as is the case with some older StarkNet tokens using the pre-SNIP-2 standard — the dispatcher will fail to deserialize the return data and panic, reverting the entire transaction.

Although `#[feature("safe_dispatcher")]` is declared on the impl block, the actual ERC20 calls use the regular `IERC20Dispatcher`, not `IERC20SafeDispatcher`. The safe dispatcher variant would allow catching panics; the regular one does not.

The EVM counterpart (`OmniBridge.sol`) correctly uses OpenZeppelin's `SafeERC20` library (`using SafeERC20 for IERC20`) with `safeTransfer` and `safeTransferFrom`, which handle non-returning tokens. The StarkNet contract has no equivalent protection.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

The attack path for `fin_transfer`:

1. A user initiates a transfer of a non-SNIP-2-compliant token from NEAR/EVM/Solana to StarkNet. Their tokens are locked or burned on the source chain.
2. A relayer (or the user) calls `fin_transfer` on StarkNet with the signed payload.
3. `_set_transfer_finalised` marks the nonce as used.
4. `IERC20Dispatcher.transfer()` panics because the token returns no boolean.
5. The entire transaction reverts — including the nonce marking — so the nonce is not permanently consumed.
6. However, every subsequent attempt to call `fin_transfer` for this token will also panic and revert.
7. The user's funds on the source chain are permanently locked with no path to recovery, because `fin_transfer` can never succeed for this token.

This satisfies the **Critical** impact class: irrecoverable lock of user funds in the bridge flow.

---

### Likelihood Explanation

**Medium.** StarkNet has a known ecosystem split between old-style ERC20 tokens (pre-SNIP-2, using felt252 return or no return) and new-style tokens (SNIP-2, returning `bool`). Any native StarkNet token that was registered in the bridge as a non-bridge-token and uses the old standard would trigger this. A user only needs to initiate a cross-chain transfer of such a token from a source chain to StarkNet — a fully unprivileged action — to permanently freeze their funds.

---

### Recommendation

Replace the raw `IERC20Dispatcher` calls with `IERC20SafeDispatcher` (Cairo's safe dispatcher variant), which wraps the call in a `Result` and allows graceful error handling instead of panicking. Alternatively, use low-level `call_contract_syscall` with explicit return-data validation, similar to how `log_metadata` already handles the old vs. new metadata standards.

```cairo
// Instead of:
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');

// Use safe dispatcher:
let result = IERC20SafeDispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
match result {
    Result::Ok(success) => assert(success, 'ERR_TRANSFER_FAILED'),
    Result::Err(_) => panic!("ERR_TRANSFER_FAILED"),
}
```

---

### Proof of Concept

1. Deploy a StarkNet ERC20 token whose `transfer` function returns no value (void return, pre-SNIP-2 style).
2. Register this token in the bridge as a non-bridge-token (it appears in the `else` branch of `fin_transfer`).
3. From NEAR or EVM, call `initTransfer` / `init_transfer` to lock/burn tokens and emit an `InitTransfer` event targeting StarkNet.
4. Obtain the MPC-signed `TransferMessagePayload` for the destination nonce.
5. Call `fin_transfer` on the StarkNet bridge with the valid signature and payload.
6. Observe: `IERC20Dispatcher.transfer()` panics due to missing boolean return → transaction reverts.
7. Repeat step 5 indefinitely — every attempt reverts. Source-chain funds are permanently frozen.
