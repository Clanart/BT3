### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Bricking and Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from the caller without verifying that the hash corresponds to a previously declared contract class. This is directly analogous to the external report's pattern of accepting `0` as a minimum bound with no enforcement — here, the OS accepts `0` (or any undeclared felt) as a valid new class hash with no validation. A malicious contract deployer can exploit this to permanently brick a contract and freeze all funds it holds.

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the new class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges this missing check. The `class_hash` value is fully attacker-controlled (it comes from the syscall request written by the calling contract).

When a future transaction calls the bricked contract, `execute_entry_point` performs:

1. `dict_read{dict_ptr=contract_class_changes}(key=undeclared_class_hash)` — returns `0` (default for uninitialized dict entries) because the class was never declared.
2. `find_element(... key=0)` — searches for a compiled class with hash `0` in `compiled_class_facts_bundle`. [2](#0-1) 

If no compiled class with hash `0` exists (the normal case), `find_element` causes a VM panic, making the entire OS execution fail and the block proof invalid. The contract becomes permanently uncallable.

### Impact Explanation

**Critical. Permanent freezing of funds.**

A malicious contract deployer can:
1. Deploy a contract that accepts user deposits (acting as a vault or escrow).
2. At any point, have the contract issue a `replace_class` syscall with an arbitrary undeclared class hash (e.g., `0x1`).
3. The OS commits this state change without validation.
4. All future calls to the contract fail at the OS level — not a graceful revert, but a VM panic that invalidates the block proof.
5. Any funds (ERC-20 tokens, ETH) held by the contract are permanently frozen with no recovery path.

### Likelihood Explanation

The attack requires deploying a contract and calling `replace_class` with an undeclared hash. Both operations are available to any unprivileged user. The TODO comment confirms the check is intentionally absent from the OS. The sequencer's blockifier would need to also lack this check for the state to be committed — but since the OS is the authoritative verifier and the TODO is in the OS, this gap exists at the protocol level. Any sequencer that does not independently enforce this constraint (relying on the OS to do so) would propagate the bricked state.

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it has been declared). Specifically, perform a `dict_read` on `contract_class_changes` with the new `class_hash` and assert the result is non-zero:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed in `execute_entry_point` and closes the gap identified by the TODO comment.

### Proof of Concept

1. Deploy a contract `VaultWithTrap` that:
   - Accepts ERC-20 deposits from users.
   - Exposes a `trap()` entry point that calls `replace_class(class_hash=0xDEAD)` where `0xDEAD` is not a declared class.
2. Users deposit funds into `VaultWithTrap`.
3. Attacker calls `trap()`. The OS executes `execute_replace_class` and writes `class_hash=0xDEAD` into `contract_state_changes` with no validation.
4. State is committed. `VaultWithTrap`'s class hash is now `0xDEAD`.
5. Any subsequent transaction calling `VaultWithTrap` reaches `execute_entry_point`, which calls `dict_read{dict_ptr=contract_class_changes}(key=0xDEAD)` → returns `0`, then `find_element(..., key=0)` → VM panic.
6. All deposited funds are permanently frozen. [3](#0-2)
