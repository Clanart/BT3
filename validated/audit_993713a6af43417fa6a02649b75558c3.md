### Title
Unauthorized `__execute__` Invocation via `execute_meta_tx_v0` Bypasses Account Signature Validation — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall in the StarkNet OS allows **any executing contract** to invoke the `__execute__` entry point of **any other account contract** with an attacker-supplied signature and calldata, without the OS ever calling `__validate__` on the target account. Standard account contracts do not re-verify the signature inside `__execute__` — they rely on `__validate__` having already done so. This creates a complete authorization bypass: an unprivileged attacker can drain any account's funds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_meta_tx_v0` (lines 286–400) implements a syscall that any contract can invoke. It:

1. Reads an arbitrary `contract_address`, `calldata`, and `signature` from the syscall request — all attacker-controlled.
2. Enforces only that `selector == EXECUTE_ENTRY_POINT_SELECTOR` (line 317); there is **no check** that the calling contract is authorized to act on behalf of `contract_address`.
3. Computes a new `meta_tx_hash` and constructs a synthetic `TxInfo` with `version=0`, `nonce=0`, and the attacker-supplied signature (lines 343–363).
4. Calls `contract_call_helper` (line 389), which directly invokes the target contract's `__execute__` entry point via `select_execute_entry_point_func`. [1](#0-0) 

The OS **never calls `__validate__`** on the target contract for this path. Compare this to the normal invoke flow in `execute_transaction_utils.cairo`, where `run_validate` is explicitly called before `__execute__`: [2](#0-1) 

And in `run_validate`, version-0 transactions skip validation entirely: [3](#0-2) 

Because `execute_meta_tx_v0` sets `version=0` in the synthetic `TxInfo`, even if `__validate__` were somehow invoked, it would be skipped. The target account's `__execute__` runs with no signature check enforced by the OS. [4](#0-3) 

The syscall is reachable by any contract, as confirmed by the syscall dispatch table in `execute_syscalls.cairo`: [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

Standard StarkNet account contracts (e.g., OpenZeppelin-style) perform signature verification exclusively in `__validate__`. Their `__execute__` function trusts that validation has already occurred and directly executes the provided multicall. An attacker who bypasses `__validate__` can supply arbitrary calldata — including ERC-20 `transfer` calls — and drain the victim account's entire token balance. Because `account_contract_address` in the synthetic `TxInfo` is set to the victim's address (line 345), the fee token transfer in `charge_fee` would also be attributed to the victim, not the attacker. [6](#0-5) 

---

### Likelihood Explanation

**High.** The attack requires only:
- Deploying a malicious contract (permissionless on StarkNet).
- Submitting a standard invoke transaction that calls the malicious contract.
- The malicious contract calls `execute_meta_tx_v0` with the victim's address and attacker-crafted calldata.

No privileged role, leaked key, or special network condition is needed. The syscall is available to every contract in every block.

---

### Recommendation

Add an explicit authorization check inside `execute_meta_tx_v0` before dispatching to `contract_call_helper`. At minimum, one of the following must hold:

1. **Require the caller to be the target contract itself** (i.e., `caller_execution_context.execution_info.contract_address == contract_address`), so only a contract can issue a meta-transaction on its own behalf.
2. **Invoke `__validate__` on the target contract** before calling `__execute__`, consistent with how top-level invoke transactions are processed in `execute_invoke_function_transaction`.
3. **Restrict the syscall** to a designated relayer registry or require an on-chain authorization from the target account.

---

### Proof of Concept

```
1. Attacker deploys MaliciousContract.
2. Attacker submits INVOKE_FUNCTION tx calling MaliciousContract.attack(victim_addr).
3. MaliciousContract.attack issues the meta_tx_v0 syscall:
      contract_address  = victim_addr          // any account on-chain
      selector          = __execute__ selector
      calldata          = [transfer(attacker, victim_balance)]
      signature         = []                   // arbitrary; __validate__ is never called
4. OS executes execute_meta_tx_v0:
      - Constructs TxInfo{version=0, account_contract_address=victim_addr, signature=[]}
      - Calls contract_call_helper → select_execute_entry_point_func → victim.__execute__
      - __validate__ is never invoked.
5. victim.__execute__ runs the attacker-supplied transfer call.
6. Victim's entire token balance is transferred to the attacker.
``` [7](#0-6)
