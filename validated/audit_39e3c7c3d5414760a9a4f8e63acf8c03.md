### Title
Solana `FinalizeTransfer` and `FinalizeTransferSol` do not validate the `recipient` account against the signed payload — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`, `finalize_transfer_sol.rs`)

---

### Summary

The Solana bridge's `FinalizeTransfer` and `FinalizeTransferSol` Anchor account structs accept a `recipient` account that is completely unconstrained — it is never validated against the `recipient` field inside the MPC-signed `FinalizeTransferPayload`. Any submitter of the finalization transaction can pass an arbitrary account as `recipient`, redirecting bridged tokens or native SOL to themselves while consuming the nonce and permanently blocking the legitimate recipient from claiming their funds.

---

### Finding Description

In `finalize_transfer.rs`, the Anchor accounts struct declares `recipient` as a bare `UncheckedAccount` with no constraint tying it to the payload:

```rust
/// CHECK: this can be any type of account
pub recipient: UncheckedAccount<'info>,

#[account(
    init_if_needed,
    payer = common.payer,
    associated_token::mint = mint,
    associated_token::authority = recipient,   // ← derived from unchecked recipient
    token::token_program = token_program,
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

The `process` function then mints or transfers tokens directly into `self.token_account`, which is the ATA of the unchecked `self.recipient`:

```rust
pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
    UsedNonces::use_nonce(data.destination_nonce, ...)?;   // nonce consumed here

    if let Some(vault) = &self.vault {
        transfer_checked(..., to: self.token_account.to_account_info(), ...)?;
    } else {
        mint_to(..., to: self.token_account.to_account_info(), ...)?;
    }
``` [2](#0-1) 

The signed `FinalizeTransferPayload` carries a `destination_nonce`, `amount`, `fee_recipient`, and `transfer_id` — all of which are used — but the payload's `recipient` field (the address the MPC network authorised to receive funds) is **never compared** to `self.recipient.key()`. The `#[instruction(data: SignedPayload<FinalizeTransferPayload>)]` decorator makes the payload available for account constraints, but no such constraint exists.

The identical flaw exists in `FinalizeTransferSol`, where native SOL is transferred directly to the unchecked `recipient`:

```rust
/// CHECK: this can be any type of account
#[account(mut)]
pub recipient: UncheckedAccount<'info>,
...
transfer(..., Transfer { from: self.sol_vault, to: self.recipient }, ..., data.amount)?;
``` [3](#0-2) 

**Analog to the external report**: The external report shows that `OmnichainStaking.onERC721Received` decodes the beneficiary address from attacker-controlled `data` bytes, allowing the staker to mint voting power to an arbitrary address. Here, the Solana bridge's `finalize_transfer` instruction accepts the token/SOL recipient as an attacker-controlled account argument that is never validated against the MPC-signed payload — the same class of "callback/state desync where attacker-controlled input overrides the authorised value."

---

### Impact Explanation

An attacker who observes a pending cross-chain transfer to Solana (e.g., from NEAR or EVM) can front-run or simply submit the `finalize_transfer` transaction themselves, substituting their own Solana account as `recipient`. The nonce is consumed by `UsedNonces::use_nonce`, so the legitimate recipient can never replay the same proof. The attacker receives the full bridged token amount or native SOL. This is **direct theft of bridged assets** — matching the Critical impact tier.

---

### Likelihood Explanation

The `finalize_transfer` instruction is permissionless (any payer can submit it). The signed payload and proof are observable on-chain or via relayer infrastructure. A malicious relayer, or any on-chain observer who front-runs the honest relayer, can trivially exploit this on every inbound Solana transfer. No privileged access, leaked key, or colluding MPC signer is required.

---

### Recommendation

Add an explicit account constraint in both `FinalizeTransfer` and `FinalizeTransferSol` that enforces `self.recipient.key()` equals the recipient encoded in the signed payload:

```rust
// In FinalizeTransfer
#[account(
    constraint = recipient.key() == Pubkey::from(data.payload.recipient)
        @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

```rust
// In FinalizeTransferSol
#[account(
    mut,
    constraint = recipient.key() == Pubkey::from(data.payload.recipient)
        @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

This mirrors how the EVM `finTransfer` correctly includes `payload.recipient` inside the ECDSA-signed Borsh blob so the recipient cannot be substituted. [4](#0-3) 

---

### Proof of Concept

1. Alice initiates a transfer of 1000 USDC from NEAR to her Solana address `alice_pubkey`.
2. The NEAR bridge emits an `InitTransfer` event; the MPC network signs a `FinalizeTransferPayload` containing `recipient = alice_pubkey`, `destination_nonce = N`, `amount = 1000`.
3. Attacker Eve observes the signed payload (from relayer mempool or NEAR events).
4. Eve calls `finalize_transfer` on Solana, passing the valid `SignedPayload<FinalizeTransferPayload>` but substituting `recipient = eve_pubkey` in the accounts list.
5. Anchor derives `token_account` as the ATA of `eve_pubkey` for the USDC mint — no constraint rejects this.
6. `UsedNonces::use_nonce(N, ...)` marks nonce `N` as used.
7. `mint_to(..., to: eve_ata, amount: 1000)` executes — Eve receives 1000 USDC.
8. Alice's transfer is permanently blocked; nonce `N` is consumed and cannot be replayed. [5](#0-4) [6](#0-5)
