Audit Report

## Title
Missing Factory Pool Validation in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Funds - (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` accepts an arbitrary caller-supplied `pool` address in all public entry points without verifying it against the factory registry. Because the callback guard only checks `msg.sender == expectedPool` — a condition trivially satisfied by the same malicious pool that was just called — any user who has approved the adder can have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 transferred directly from their wallet to the attacker-controlled pool.

## Finding Description
`MetricOmmSwapRouterBase` stores the factory as an immutable and calls `_requireFactoryPool(pool)` before writing any pool address into transient context, in `_setNextCallbackContext` (L30), `_initCallbackContextforRecursiveOutput` (L41), and `_updateCallbackContextforRecursiveOutput` (L46). [1](#0-0) 

`MetricOmmPoolLiquidityAdder` has no factory reference at all — its constructor accepts only `weth`. [2](#0-1) 

The NatDoc on the contract explicitly acknowledges the gap: [3](#0-2) 

`_addLiquidity` writes the caller-supplied `pool` directly into transient pay context and then calls `pool.addLiquidity(...)` with no prior validation: [4](#0-3) 

`metricOmmModifyLiquidityCallback` validates only `msg.sender == expectedPool`, which is trivially satisfied by the malicious pool that was just invoked: [5](#0-4) 

After passing that check, the callback reads `token0`/`token1` from `msg.sender` (the malicious pool) and calls `pay()` to pull real tokens from the victim: [6](#0-5) 

## Impact Explanation
Any user who has approved `MetricOmmPoolLiquidityAdder` and calls any `addLiquidityExactShares` or `addLiquidityWeighted` overload with a malicious pool address suffers a direct loss of up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 transferred from their wallet to the attacker. This is a direct loss of user principal with no recovery path, meeting the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
The trigger requires a user to call the adder with a malicious pool address (e.g., via a phishing UI, compromised front-end, or misleading integration) and to have pre-approved the adder. The adder is a public, permissionless contract intended for broad use. The router's analogous factory guard (`_requireFactoryPool`) demonstrates the protocol is aware of this class of risk and enforced it in the swap path but not here. The `addLiquidityWeighted` variant additionally lets the attacker control the probe revert values (`need0`, `need1`) to manipulate the share-scaling step before the paying callback fires, though the cap remains `max0`/`max1`. [7](#0-6) 

## Recommendation
Add a factory validation call at the top of `_addLiquidity` (or in each public entry point), mirroring the router pattern. This requires storing the factory address as an immutable in `MetricOmmPoolLiquidityAdder`, exactly as `MetricOmmSwapRouterBase` does: [8](#0-7) 

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

## Proof of Concept
1. Deploy `MaliciousPool(token0, token1, adderAddress)` implementing `addLiquidity` to immediately call back `metricOmmModifyLiquidityCallback(MAX0, MAX1, abi.encode(uint8(1)))`.
2. Victim approves `MetricOmmPoolLiquidityAdder` for `token0` and `token1`.
3. Attacker calls `adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, MAX0, MAX1, "")`.
4. `_addLiquidity` writes `maliciousPool` as `expectedPool` in transient context and calls `maliciousPool.addLiquidity(...)`.
5. `maliciousPool.addLiquidity` immediately calls back `metricOmmModifyLiquidityCallback(MAX0, MAX1, KIND_PAY)`.
6. `msg.sender == expectedPool` check passes (malicious pool is the expected pool).
7. Callback reads `token0`/`token1` from `msg.sender` (malicious pool) and executes `safeTransferFrom(victim, maliciousPool, MAX0)` and `safeTransferFrom(victim, maliciousPool, MAX1)`. [9](#0-8)
