### Title
`deploy_account` Address Fully Predictable and Frontrunnable via `deploy_from_zero=TRUE` Syscall — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The OS computes `deploy_account` contract addresses with a hardcoded `deployer_address=0`. The `deploy` syscall also accepts `deploy_from_zero=TRUE`, which identically zeroes the deployer address. Because both paths feed the same inputs into `get_contract_address`, any unprivileged attacker who observes a pending `deploy_account` transaction can preemptively occupy the victim's intended address by issuing a `deploy` syscall with the same `(salt, class_hash, constructor_calldata)` and `deploy_from_zero=TRUE`. If the victim pre-funded the address (the standard StarkNet account-deployment flow), those tokens are permanently transferred to the attacker's malicious contract.

---

### Finding Description

**Root cause — two code paths produce identical addresses:**

**Path 1 — `deploy_account` transaction** (`transaction_impls.cairo`, `prepare_constructor_execution_context`):

```cairo
let (contract_address) = get_contract_address(
    salt=contract_address_salt,
    class_hash=class_hash,
    constructor_calldata_size=constructor_calldata_size,
    constructor_calldata=constructor_calldata,
    deployer_address=0,          // ← always 0, hardcoded
);
``` [1](#0-0) 

**Path 2 — `deploy` syscall with `deploy_from_zero=TRUE`** (`syscall_impls.cairo`, `execute_deploy`):

```cairo
// Set deployer_address to 0 if request.deploy_from_zero is TRUE.
let deployer_address = (1 - deploy_from_zero) * caller_address;
// → deployer_address = 0 when deploy_from_zero = 1
``` [2](#0-1) 

The same pattern exists in the deprecated path: [3](#0-2) 

**Address formula** (`contract_address.cairo`):

```
address = hash(PREFIX, deployer_address=0, salt, class_hash, hash(constructor_calldata))
``` [4](#0-3) 

Because `deployer_address=0` in both paths, the resulting address is **identical** for the same `(salt, class_hash, constructor_calldata)` triple, regardless of who submits the transaction.

**Collision enforcement is a hard assertion, not a graceful revert** (`deploy_contract.cairo`):

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
assert state_entry.nonce = 0;
``` [5](#0-4) 

If the address is already occupied when `deploy_account` executes, this assertion fails, making the block proof invalid and forcing the sequencer to drop the victim's transaction entirely.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The standard StarkNet account-deployment flow requires the user to:
1. Compute the expected account address (from `salt`, `class_hash`, `constructor_calldata` with `deployer_address=0`).
2. **Pre-fund that address** with ERC-20 tokens (e.g., STRK/ETH for fees and initial balance).
3. Submit the `deploy_account` transaction.

An attacker who frontruns step 3 deploys a **malicious contract** (with a `transfer`-draining constructor or entry point) to the same address. The victim's `deploy_account` is then rejected. The victim's pre-funded tokens now reside in the attacker's contract, which the attacker can drain at will. There is no recovery path: the address is permanently occupied by the attacker's contract.

---

### Likelihood Explanation

**Medium-High.** The attack requires:

1. **Observing the victim's parameters**: `deploy_account` transactions are public in the sequencer mempool. `salt`, `class_hash`, and `constructor_calldata` are all plaintext fields.
2. **Submitting a `deploy` syscall with `deploy_from_zero=TRUE`** using the same parameters from any existing contract — no special privilege required.
3. **Transaction ordering**: The attacker's transaction must be processed before the victim's. In StarkNet's current centralized-sequencer model, an attacker can submit their transaction with higher fees or simply submit it before the victim's `deploy_account` is broadcast.

The pre-funding step (step 2 of the victim's flow) is itself a public on-chain event that reveals the target address, giving the attacker a clear signal and a timing window before the `deploy_account` is submitted.

---

### Recommendation

Include a deployment-type discriminator in the address computation for `deploy_account` transactions so that the address space for `deploy_account` is disjoint from the address space reachable via `deploy` syscall with `deploy_from_zero=TRUE`. For example, use a distinct prefix constant for `deploy_account` address derivation (analogous to the external report's fix of mixing `msg.sender` into the salt):

```cairo
// In prepare_constructor_execution_context, use a non-zero sentinel
// that cannot be produced by any live contract address:
const DEPLOY_ACCOUNT_DEPLOYER_SENTINEL = 'DEPLOY_ACCOUNT';

let (contract_address) = get_contract_address(
    salt=contract_address_salt,
    class_hash=class_hash,
    constructor_calldata_size=constructor_calldata_size,
    constructor_calldata=constructor_calldata,
    deployer_address=DEPLOY_ACCOUNT_DEPLOYER_SENTINEL,  // non-zero, unique
);
```

This ensures no `deploy` syscall (which uses either `caller_address` or `0`) can produce the same address as a `deploy_account` transaction.

### Proof of Concept

**Setup**: Attacker has an existing contract `AttackerContract` that can issue `deploy` syscalls.

**Step 1**: Victim computes their expected account address:
```
victim_address = hash(PREFIX, 0, salt_V, class_hash_V, hash(calldata_V))
```
This is fully deterministic from public parameters.

**Step 2**: Victim pre-funds `victim_address` with 1000 STRK tokens (standard flow).

**Step 3**: Victim broadcasts `deploy_account(salt=salt_V, class_hash=class_hash_V, constructor_calldata=calldata_V)`.

**Step 4**: Attacker observes the pending `deploy_account` transaction, extracts `(salt_V, class_hash_V, calldata_V)`, and calls `AttackerContract.deploy(salt=salt_V, class_hash=malicious_class_hash, deploy_from_zero=TRUE)` — or with the same `class_hash_V` if the attacker wants to match exactly. The attacker's transaction is submitted with higher fees and processed first.

**Step 5**: OS executes attacker's `deploy` syscall via `execute_deploy` → `deploy_contract`. The address `victim_address` is now occupied by the attacker's contract (`state_entry.class_hash` is set to `malicious_class_hash`). [6](#0-5) 

**Step 6**: OS processes victim's `deploy_account` transaction. `prepare_constructor_execution_context` computes the same `victim_address`. [7](#0-6) 

`deploy_contract` is called and hits:
```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;  // FAILS — address occupied
``` [5](#0-4) 

The victim's `deploy_account` is rejected. The 1000 STRK pre-funded at `victim_address` are now under the attacker's malicious contract's control and are drained.
