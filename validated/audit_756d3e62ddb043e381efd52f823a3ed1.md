### Title
ChunkEndorsementTracker silently drops canonical-fork endorsements when a stale-fork endorsement is cached first — (File: chain/client/src/stateless_validation/chunk_endorsement.rs)

---

### Summary

`ChunkEndorsementTracker::process_chunk_endorsement` deduplicates by `ChunkProductionKey` (`shard_id`, `epoch_id`, `height_created`) without including `chunk_hash`. When a validator legitimately endorses two different chunks at the same height (one on a stale fork, one on the canonical fork), whichever endorsement arrives first permanently occupies the cache slot. Any subsequent endorsement from the same validator for the same key — even for a different `chunk_hash` — is silently dropped. An unprivileged network participant can exploit this by relaying a validator's stale-fork endorsement to a block producer before the canonical endorsement arrives, preventing the block producer from collecting sufficient stake to produce a valid block.

---

### Finding Description

In `process_chunk_endorsement`, the deduplication guard at line 50 checks whether `account_id` is already present in the cache under the `ChunkProductionKey`:

```rust
if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
    tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
    return Ok(());
}
``` [1](#0-0) 

The `ChunkProductionKey` is `{shard_id, epoch_id, height_created}` — it does **not** include `chunk_hash`. [2](#0-1) 

When the guard fires, the function returns `Ok(())` without storing the new endorsement. The stored value is `(ChunkHash, Signature)`:

```rust
cache.get_or_insert_mut(key, || HashMap::new()).insert(
    account_id.clone(),
    (endorsement.chunk_hash(), endorsement.signature()),
);
``` [3](#0-2) 

When `collect_chunk_endorsements` is called by the block producer, it filters the cached entries by `chunk_hash`:

```rust
let validator_signatures = entry
    .into_iter()
    .filter(|(_, (chunk_hash, _))| chunk_hash == chunk_header.chunk_hash())
    .map(|(account_id, (_, signature))| (account_id, signature.clone()))
    .collect();
``` [4](#0-3) 

If the cached `chunk_hash` for a validator is `H_fork` but the block producer needs endorsements for `H_canonical`, that validator's endorsement is silently excluded from the stake count. Because the deduplication guard already fired, no subsequent endorsement from that validator for `H_canonical` will ever be stored.

The `ChunkEndorsementHandlerActor` is spawned with 4 concurrent threads, and the lock is released between the deduplication check and the insertion, creating an additional TOCTOU window: [5](#0-4)

### Impact Explanation

In a fork scenario, chunk validators receive state witnesses from competing chunk producers and are expected to endorse all valid chunks they observe. If an attacker relays stale-fork endorsements from enough validators (stake > 1/3 of the shard's assigned chunk validators) to the block producer before the canonical endorsements arrive, the block producer's `ChunkEndorsementTracker` cache is poisoned. The block producer calls `collect_chunk_endorsements` and finds insufficient stake for the canonical chunk. Because the deduplication guard permanently blocks the correct endorsements from entering the cache (for the lifetime of the LRU entry, up to 100 slots), the block producer cannot produce a valid block for that shard. This stalls chunk inclusion and, if sustained across multiple shards, stalls the chain.

---

### Likelihood Explanation

- **Unprivileged attacker**: Any network participant can observe and relay signed `ChunkEndorsement` messages. No validator key or special role is required.
- **Fork condition**: Forks occur naturally during network partitions or when competing chunk producers exist. NEAR's Doomslug minimizes but does not eliminate forks.
- **Validator behavior**: Chunk validators are expected to endorse all valid chunks they receive state witnesses for, including chunks on competing forks. This is not Byzantine behavior.
- **Relay race**: The attacker only needs to deliver the stale-fork endorsement to the block producer before the canonical endorsement arrives — a straightforward network-level advantage.

---

### Recommendation

Include `chunk_hash` in the deduplication key, or allow a validator's cached endorsement to be replaced when a new endorsement with a different `chunk_hash` and a valid signature arrives for the same `ChunkProductionKey`. Concretely, change the guard to:

```rust
if cache.peek(&key).is_some_and(|entry| {
    entry.get(account_id).is_some_and(|(cached_hash, _)| cached_hash == &endorsement.chunk_hash())
}) {
    return Ok(());
}
```

This allows the cache to be updated when the validator endorses a different chunk (e.g., on the canonical fork after having endorsed a stale fork), while still deduplicating exact duplicates.

---

### Proof of Concept

1. At height `H`, shard `S`, epoch `E`: two competing chunks exist — `C_fork` (hash `H_fork`) on a stale fork and `C_canonical` (hash `H_canonical`) on the canonical fork.
2. Validator `V` (assigned to shard `S` at height `H`) validates both state witnesses and broadcasts endorsements for both `H_fork` and `H_canonical`.
3. Attacker observes `V`'s endorsement for `H_fork` on the network and relays it to the block producer on the canonical fork before `V`'s endorsement for `H_canonical` arrives.
4. `process_chunk_endorsement` stores `V -> (H_fork, sig_fork)` under key `(S, E, H)`.
5. `V`'s endorsement for `H_canonical` arrives. The deduplication guard at line 50 fires (`V` is already in the cache for key `(S, E, H)`) and returns `Ok(())` without updating the stored `chunk_hash`.
6. The block producer calls `collect_chunk_endorsements` for `C_canonical`. The filter at line 109 excludes `V`'s entry because `H_fork != H_canonical`. `V`'s stake is not counted.
7. If the attacker repeats this for enough validators (stake > 1/3 of the shard's chunk validator set), the block producer cannot reach the endorsement threshold and cannot produce a valid block for shard `S`. [6](#0-5)

    pub fn process_chunk_endorsement(&self, endorsement: &ChunkEndorsement) -> Result<(), Error> {
        // Check if we have already received chunk endorsement from this validator.
        let key = endorsement.chunk_production_key();
        let account_id = endorsement.account_id();

        {
            let cache = self.chunk_endorsements.lock();
            if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
                tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
                return Ok(());
            }
        }

        // Validate the chunk endorsement and store it in the cache.
        match validate_chunk_endorsement(self.epoch_manager.as_ref(), endorsement, &self.store)? {
            ChunkRelevance::Relevant => {
                let mut cache = self.chunk_endorsements.lock();
                cache.get_or_insert_mut(key, || HashMap::new()).insert(
                    account_id.clone(),
                    (endorsement.chunk_hash(), endorsement.signature()),
                );
                let shard_id = endorsement.shard_id();
                metrics::CHUNK_ENDORSEMENTS_ACCEPTED
                    .with_label_values(&[&shard_id.to_string()])
                    .inc();
            }
            irrelevant => {
                metrics::CHUNK_ENDORSEMENTS_REJECTED
                    .with_label_values::<&str>(&[
                        endorsement.shard_id().to_string().as_str(),
                        irrelevant.into(),
                    ])
                    .inc();
            }
        };
        Ok(())
    }
