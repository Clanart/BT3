### Title
Unvalidated Cached Chunk-Forward Parts Poison `encoded_chunks` Cache, Permanently Blocking Block Processing — (`chain/chunks/src/shards_manager_actor.rs`)

### Summary

`process_partial_encoded_chunk_forward` silently caches a `PartialEncodedChunkForwardMsg` — including its attacker-supplied parts and `merkle_root` — when `validate_partial_encoded_chunk_forward` fails with `Error::ChainError(DBNotFoundErr(...))`. This happens whenever `get_chunk_producer_info_from_prev_block` cannot resolve the chunk producer (e.g., `prev_block_hash` not yet in the DB, or `EpochError::ChunkProducerNotInDB` under the EarlyKickout feature). The cached parts are later merged into `encoded_chunks` without any re-validation against the chunk header's real `encoded_merkle_root`. Because `merge_in_partial_encoded_chunk` uses `or_insert_with`, fake parts for a given `part_ord` permanently block real parts from being inserted. Once enough fake parts accumulate to meet the reconstruction threshold, Reed-Solomon decoding is attempted, fails the merkle-root check, and the cache entry is poisoned (`decode_failed = true`). All subsequent real parts are rejected, the request-pool entry is removed, and the node cannot process the block.

### Finding Description

**Step 1 — Signature check fails with `DBNotFoundErr`, forward is cached without validation:** [1](#0-0) 

`validate_partial_encoded_chunk_forward` calls `verify_chunk_header_signature_by_hash_and_parts`, which calls `get_chunk_producer_info_from_prev_block`. [2](#0-1) 

`get_chunk_producer_info_from_prev_block` returns `EpochError::MissingBlock` or `EpochError::ChunkProducerNotInDB`, both of which convert to `near_chain::Error::DBNotFoundErr`. [3](#0-2) 

The `DBNotFoundErr` arm in `process_partial_encoded_chunk_forward` caches the forward — including attacker-controlled `merkle_root` and `parts` — without the signature ever being verified. [4](#0-3) 

**Step 2 — Cached parts merged into `encoded_chunks` without re-validation:**

When the header arrives, `insert_header_if_not_exists_and_process_cached_chunk_forwards` pops the cached parts and merges them. The comment claims "we don't need any further validation for the forwarded part … validated via `validate_partial_encoded_chunk_forward`" — but this is false for the `DBNotFoundErr` path where that function returned an error. [5](#0-4) 

The parts are merged without checking them against the chunk header's `encoded_merkle_root`. [6](#0-5) 

**Step 3 — `or_insert_with` prevents real parts from overwriting fake ones:**

`merge_in_partial_encoded_chunk` uses `entry(part_ord).or_insert_with(...)`, so once a fake part occupies a slot, the real part for that `part_ord` is silently discarded when it arrives later. [7](#0-6) 

**Step 4 — Reconstruction with fake parts poisons the cache entry permanently:**

When `entry.parts.len() >= num_data_parts`, reconstruction is triggered. The fake parts fail the merkle-root check inside `decode_encoded_chunk_if_complete`. [8](#0-7) 

`mark_decode_failed` sets `decode_failed = true` and removes the chunk from `incomplete_chunks`. The request-pool entry is also removed. [9](#0-8) 

The poisoned entry rejects all subsequent real parts (`test_late_parts_rejected_after_decode_failure` confirms this). [10](#0-9) 

### Impact Explanation

Any unprivileged network peer can send a `PartialEncodedChunkForwardMsg` with:
- `chunk_hash`: a real chunk hash observed from the network
- `prev_block_hash`: any hash not yet in the local DB (trivially satisfied before the parent block is processed)
- `merkle_root`: an attacker-chosen value
- `parts`: fake parts whose merkle proofs are valid against the fake `merkle_root`
- `signature`: arbitrary (never verified in this path)

If the attacker sends `num_data_parts` (≈ 1/3 of total parts) fake parts before the real parts arrive, the cache entry is poisoned. The node sends `DecodedChunk::Invalid` to the client, cannot reconstruct the chunk, and cannot process the block. Because the poisoned entry blocks re-requests and rejects late real parts, the node is stuck until the entry is GC'd — which itself requires the chain head to advance, creating a potential deadlock for validators and chunk producers.

### Likelihood Explanation

The `DBNotFoundErr` path is a normal, expected code path (the metric `PARTIAL_ENCODED_CHUNK_FORWARD_CACHED_WITHOUT_PREV_BLOCK` is explicitly tracked). Any peer on the network can send a `PartialEncodedChunkForwardMsg`; no validator key or privileged role is required. The attack window opens as soon as a chunk hash is observable on the network (before the parent block is processed), which is the common case for forwarded parts.

### Recommendation

In `process_partial_encoded_chunk_forward`, when caching a forward due to `DBNotFoundErr`, store the raw forward message but **do not** treat it as validated. When `insert_header_if_not_exists_and_process_cached_chunk_forwards` later merges cached parts, re-validate each part's merkle proof against the chunk header's real `encoded_merkle_root` (using the existing `validate_part` helper) before inserting it into `encoded_chunks`. Alternatively, re-run `validate_partial_encoded_chunk_forward` at merge time (now that the prev_block is available) and discard any forward whose signature or parts fail. The comment claiming prior validation is correct should be removed or scoped only to the `UnknownChunk` path where validation actually succeeded.

### Proof of Concept

```
1. Attacker observes chunk_hash H for shard S at height N from the network.
2. Attacker crafts PartialEncodedChunkForwardMsg:
     chunk_hash      = H
     prev_block_hash = <any 32-byte value not in the local DB>
     merkle_root     = sha256("fake")
     parts           = [num_data_parts fake PartialEncodedChunkPart entries,
                        each with a valid merkle proof against the fake merkle_root]
     signature       = <arbitrary bytes>
     shard_id        = S
3. Attacker sends this message to the victim node.
4. In process_partial_encoded_chunk_forward:
     validate_partial_encoded_chunk_forward fails at verify_chunk_header_signature_by_hash_and_parts
     because get_chunk_producer_info_from_prev_block returns EpochError::MissingBlock
     → Error::ChainError(DBNotFoundErr(...))
     → insert_forwarded_chunk(forward) caches the fake parts.
5. The real header for H arrives (e.g., via process_chunk_header_from_block).
6. insert_header_if_not_exists_and_process_cached_chunk_forwards merges the fake parts
   into encoded_chunks without validating against header.encoded_merkle_root().
7. Real parts arrive via process_partial_encoded_chunk but are silently discarded
   (or_insert_with finds the fake parts already present).
8. entry.parts.len() >= num_data_parts → reconstruction triggered.
9. decode_encoded_chunk_if_complete: merkle_root of fake content ≠ header.encoded_merkle_root()
   → ChunkDecodeResult::Invalid.
10. mark_decode_failed poisons the entry; requested_partial_encoded_chunks.remove drops the request.
11. Client receives DecodedChunk::Invalid; node cannot process the block at height N.
```

    pub fn merge_in_partial_encoded_chunk(
        &mut self,
        parts: impl Iterator<Item = PartialEncodedChunkPart>,
        receipts: impl Iterator<Item = ReceiptProof>,
    ) -> HashSet<u64> {
        let mut previously_missing_part_ords = HashSet::new();
        for part_info in parts {
            let part_ord = part_info.part_ord;
            self.parts.entry(part_ord).or_insert_with(|| {
                previously_missing_part_ords.insert(part_ord);
                part_info
            });
        }

        for receipt in receipts {
            let shard_id = receipt.1.to_shard_id;
            self.receipts.entry(shard_id).or_insert_with(|| receipt);
        }
        previously_missing_part_ords
    }
