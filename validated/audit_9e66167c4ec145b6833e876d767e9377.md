Audit Report

## Title
Stranded Native ETH on Router Subsidizes Any Subsequent WETH Swap Caller — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` uses `address(this).balance` to fund WETH swaps without verifying the ETH was contributed by the current payer in the current transaction. ETH left on the router from a prior `exactOutputSingle` or `exactInputSingle` call (where `msg.value > amountIn`) can be silently consumed by any subsequent caller who sends `msg.value = 0` with `tokenIn = WETH`. The victim loses their stranded ETH; the attacker receives a fully subsidized swap output.

## Finding Description
`PeripheryPayments.pay` (lines 69–88) contains the following WETH branch:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

When `nativeBalance >= value`, the function wraps and forwards `value` ETH from the router's balance to the pool without pulling anything from `payer`. No check exists that the native ETH was contributed by the current payer.

**How ETH accumulates:** `exactOutputSingle` (lines 130–147) is `payable`. A user sends `msg.value = amountInMaximum` because the exact input is unknown. The pool determines `amountIn < amountInMaximum`; `pay` deposits only `amountIn` ETH, leaving `amountInMaximum − amountIn` stranded on the router. There is no automatic refund — `refundETH()` must be called explicitly (e.g., via `multicall`). The same residue arises from `exactInputSingle` (lines 67–86) when `msg.value > amountIn`.

**How the attacker drains it:** `exactInputSingle` is `payable` and imposes no `msg.value >= amountIn` check. An attacker calls it with `msg.value = 0`, `tokenIn = WETH`, `amountIn = <stranded amount>`. The transient callback context stores the attacker as `payer` and `WETH` as `tokenToPay` (via `_setNextCallbackContext`, line 71). The swap callback fires `_justPayCallback` → `pay(WETH, attacker, pool, value)`. Inside `pay`, `nativeBalance >= value` is true (victim's ETH covers it), so the router wraps the victim's ETH into WETH and forwards it to the pool. The attacker receives the full swap output at zero cost.

**Why existing guards fail:** The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`, lines 32–34) prevents arbitrary ETH injection via plain transfers, but ETH sent through `msg.value` in `payable` entry points bypasses it entirely. The transient callback context (`TransientCallbackPool`) stores pool, callbackMode, payer, and tokenToPay — but never `msg.value`, so there is no per-call ETH budget to cap against.

## Impact Explanation
Direct loss of user principal. The victim loses the entire ETH surplus they sent (up to `amountInMaximum − amountIn` for an exact-output swap). The attacker receives a fully subsidized swap output. This meets the Sherlock critical/high threshold for direct loss of user funds with no special privileges required.

## Likelihood Explanation
Medium-High. `exactOutputSingle` with native ETH input is a standard UX pattern. Any user who calls it directly (without a `multicall` + `refundETH()`) creates the residue. An attacker can monitor the mempool for such transactions and immediately follow with a zero-cost `exactInputSingle`. No privileged role, no special token, and no pool manipulation is required.

## Recommendation
Track the ETH contributed by the current payer in the transient callback context. Store `msg.value` alongside the payer address when setting the callback context in `exactInputSingle` / `exactOutputSingle` (via `_setNextCallbackContext`), and cap the native ETH used in `pay` to that stored value rather than the router's full `address(this).balance`. Alternatively, enforce `msg.value == amountIn` for WETH swaps and require users to always wrap via `multicall` + `refundETH()`.

## Proof of Concept
1. Deploy router with a WETH/TokenB pool.
2. **Victim:** Call `exactOutputSingle` with `tokenIn = WETH`, `amountOut = X`, `amountInMaximum = M`, `msg.value = M`. Pool consumes `amountIn = A < M`; router now holds `M − A` ETH.
3. **Attacker:** Call `exactInputSingle` with `tokenIn = WETH`, `amountIn = M − A`, `msg.value = 0`, `recipient = attacker`.
4. Callback fires: `pay(WETH, attacker, pool, M − A)`. `address(this).balance = M − A >= M − A`, so router wraps victim's ETH and sends WETH to pool.
5. Attacker receives `amountOut` of TokenB at zero cost. Victim's `M − A` ETH is gone.

Foundry test outline:
```solidity
vm.deal(victim, M);
vm.prank(victim);
router.exactOutputSingle{value: M}(params); // leaves M-A ETH on router

vm.prank(attacker);
router.exactInputSingle{value: 0}(attackParams); // drains M-A ETH, attacker gets output
assertEq(address(router).balance, 0);
assertGt(tokenB.balanceOf(attacker), 0);
```
