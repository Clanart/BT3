### Title
Unconditional `assert is_reverted = 0` After Constructor Execution Causes OS Proof-Generation Halt — (File: `execution/deploy_contract.cairo`)

---

### Summary

`deploy_contract` in `deploy_contract.cairo` calls `select_execute_entry_point_func` to invoke a contract constructor, then unconditionally asserts `assert is_reverted = 0`. If the constructor reverts for any reason, this Cairo assertion fails, making it impossible for the OS to produce a valid execution trace and therefore a valid proof for the block. Any block containing such a transaction cannot be finalized, constituting a network halt.

---

### Finding Description

In `deploy_contract.cairo`, after invoking the constructor:

```cairo
// Invoke the contract constructor.
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);
...
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

There is no conditional branch: if `is_reverted = 1`, the Cairo assertion fails and the OS execution trace is invalid. This is directly analogous to the external report's pattern — a mandatory sub-step (constructor execution) is assumed to always succeed, and no error path exists.

This function is invoked from two distinct call sites:

**Call site 1 — the `deploy` syscall** (`syscall_impls.cairo`):

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [2](#0-1) 

The `failure_flag=0` is hardcoded and the TODO comment explicitly acknowledges that failures are not yet supported. The OS panics before ever reaching this line if the constructor reverts.

**Call site 2 — `deploy_account` transaction** (`transaction_impls.cairo`):

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [3](#0-2) 

Again, no `is_reverted` check is performed after `deploy_contract` returns; a reverting constructor causes an OS panic.

The `execute_entry_point` function does process syscalls even for reverted outer entry points (to replay the execution faithfully):

```cairo
if (is_reverted != FALSE) {
    ...
    let revert_log = init_revert_log();
} else { ... }
...
with syscall_ptr {
    call_execute_syscalls(...);
}
``` [4](#0-3) 

This means that even if the outer transaction is reverted by the sequencer, the OS still replays the `deploy` syscall and hits the `assert is_reverted = 0` when the constructor had reverted.

---

### Impact Explanation

A failing Cairo `assert` produces an invalid execution trace. The prover cannot generate a STARK proof from an invalid trace. Any block containing a transaction where a `deploy` syscall's constructor reverted cannot be proven. The network cannot finalize that block, resulting in a **total network halt** (High severity per the allowed impact scope).

---

### Likelihood Explanation

The attack path is fully unprivileged:

1. An attacker declares a Sierra class whose constructor unconditionally panics/reverts.
2. The attacker sends a transaction invoking a contract that calls the `deploy` syscall with that class hash, or sends a `deploy_account` transaction using that class hash directly.
3. The sequencer's blockifier executes the transaction. If the blockifier handles the constructor revert gracefully (e.g., by reverting the outer transaction and including it in the block as a reverted tx — which is standard StarkNet behavior for reverted transactions), the transaction is included in the block.
4. The OS attempts to prove the block, replays the `deploy` syscall, encounters `is_reverted = 1` from the constructor, and hits `assert is_reverted = 0`, making proof generation impossible.

The TODO comment `// TODO(Yoni, 1/1/2026): support failures.` confirms the OS developers are aware this case is unhandled, strongly implying the blockifier may already handle it gracefully while the OS does not — exactly the discrepancy that creates the vulnerability.

---

### Recommendation

Replace the unconditional `assert is_reverted = 0` in `deploy_contract` with a conditional branch that propagates the failure back to the caller, analogous to how `contract_call_helper` handles reverts:

```cairo
if (is_reverted != FALSE) {
    // Revert the deployed contract's state changes.
    handle_revert(contract_address=contract_address, revert_log_end=revert_log);
    // Return failure to the caller (execute_deploy will write failure_flag=1).
    return (retdata_size=retdata_size, retdata=retdata, is_reverted=1);
}
```

The `execute_deploy` function in `syscall_impls.cairo` must also be updated to write `failure_flag=is_reverted` instead of the hardcoded `failure_flag=0`.

---

### Proof of Concept

1. Declare a Sierra class `RevertingClass` whose `constructor` entry point always panics.
2. Submit a `deploy_account` transaction (or an invoke transaction calling `deploy`) using `RevertingClass`.
3. The sequencer's blockifier executes the transaction; the constructor reverts; the transaction is included in the block as a reverted transaction.
4. The OS begins proving the block and reaches `deploy_contract` → `select_execute_entry_point_func` returns `is_reverted=1` → `assert is_reverted = 0` fails → the OS cannot produce a valid trace → the block cannot be proven → **network halt**.
