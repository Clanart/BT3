### Title
`batch_insert` Bypasses Duplicate-Key Enforcement via Ignored `HashMap::insert` Return Values, Corrupting DataLayer Merkle Tree State — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`BlockStatusCache::add_leaf` silently discards the `Option<V>` returned by both `HashMap::insert` calls. When `MerkleBlob::batch_insert` is invoked with three or more entries, all items except the last two bypass the duplicate-key/hash guard and flow directly into `insert_entry_to_blob` → `add_leaf`. A duplicate key or hash in that prefix silently overwrites the cache mapping while leaving the old leaf node intact in the blob, creating a permanent cache-blob divergence that corrupts the Merkle tree root and invalidates all subsequent inclusion/exclusion proofs.

### Finding Description

`BlockStatusCache::add_leaf` discards both `HashMap::insert` return values:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // Option<TreeIndex> dropped
    self.leaf_hash_to_index.insert(leaf.hash, index); // Option<TreeIndex> dropped
}
``` [1](#0-0) 

`HashMap::insert` returns `Some(old_value)` when the key already exists and silently overwrites it. Neither call checks this, so a duplicate key or hash is accepted without error.

`MerkleBlob::batch_insert` pops the **last two** items and routes them through `self.insert()` (which enforces uniqueness). All **preceding** items are written directly via `insert_entry_to_blob` with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {   // no uniqueness guard
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` unconditionally calls `add_leaf`:

```rust
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    Node::Internal(..) => self.block_status_cache.add_internal(index),
}
``` [3](#0-2) 

The Python binding `py_batch_insert` exposes this path directly to untrusted callers: [4](#0-3) 

### Impact Explanation

When a duplicate key appears in the non-checked prefix of a `batch_insert` call:

1. **Blob state**: two distinct leaf nodes with the same `KeyId` are written at different `TreeIndex` positions.
2. **Cache state**: `key_to_index` and `leaf_hash_to_index` are silently overwritten to point only to the later index; the earlier leaf node becomes unreachable through the cache but remains in the blob.
3. **Tree root**: `calculate_lazy_hashes` computes the root over the actual blob structure (which contains both leaves), producing a root that does not correspond to any valid tree.
4. **Proofs**: `get_proof_of_inclusion` uses the cache to locate a leaf and builds a proof path over the corrupted tree. The resulting proof is structurally inconsistent with the committed root, enabling acceptance of forged inclusion/exclusion proofs by any verifier that trusts the root.
5. **`check_integrity`** will detect the divergence, but only if explicitly called — it is not invoked automatically after `batch_insert`. [5](#0-4) 

### Likelihood Explanation

`py_batch_insert` is a public Python-binding method. Any caller — including untrusted DataLayer clients — can supply a list containing a repeated `KeyId` or `Hash` among the first N-2 entries of a batch of size ≥ 3. No authentication or privilege is required. The `BlockStatusCache::new` constructor does check for duplicates during deserialization, but `batch_insert` does not re-validate on mutation. [6](#0-5) 

### Recommendation

`add_leaf` should treat a non-`None` return from either `HashMap::insert` as a programming error and either return an `Err` or panic:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) -> Result<(), Error> {
    self.free_indexes.shift_remove(&index);
    if self.key_to_index.insert(leaf.key, index).is_some() {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.leaf_hash_to_index.insert(leaf.hash, index).is_some() {
        return Err(Error::HashAlreadyPresent());
    }
    Ok(())
}
```

Additionally, `batch_insert` should validate uniqueness of all input keys and hashes (against each other and against the existing tree) before writing any entry to the blob, mirroring the guard already present in `MerkleBlob::insert`. [7](#0-6) 

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

def h(n):
    return bytes(hashlib.sha256(n.to_bytes(8, "big")).digest())

blob = MerkleBlob(bytearray())

# Batch of 4 entries: first 2 bypass self.insert(); last 2 go through it.
# Entry at index 0 and index 1 share the same KeyId — duplicate in unchecked prefix.
dup_key   = KeyId(42)
dup_value = ValueId(1)
keys_values = [(dup_key, dup_value),          # duplicate — no guard
               (KeyId(99), ValueId(99)),       # duplicate — no guard
               (KeyId(100), ValueId(100)),     # checked by self.insert()
               (KeyId(101), ValueId(101))]     # checked by self.insert()
hashes = [h(0), h(1), h(2), h(3)]

# Second entry with same key — overwrites cache silently
keys_values[1] = (dup_key, ValueId(2))
hashes[1] = h(10)

blob.batch_insert(keys_values, hashes)
blob.calculate_lazy_hashes()

# check_integrity reveals the divergence
try:
    blob.check_integrity()
    print("INTEGRITY OK — cache-blob divergence undetected at this point")
except Exception as e:
    print(f"Integrity failure (expected): {e}")

# Proof of inclusion built on corrupted root — structurally invalid
proof = blob.get_proof_of_inclusion(blob.get_key_index(dup_key))
print("Proof layers:", proof)
```
