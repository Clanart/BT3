### Title
Relayer-Controlled `recipient` in `FinalizeTransferSol` Enables SOL Misdirection to Unspendable Accounts — (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer_sol.rs`)

---

### Summary

`FinalizeTransferSol::process` transfers SOL from `sol_vault` to a `recipient: UncheckedAccount` that is entirely relayer-supplied and never validated against the cross-chain payload. Because `FinalizeTransferPayload` carries no recipient field, any relayer can substitute an arbitrary account — including a program-owned PDA — as the destination, causing the user's SOL to be irrecoverably locked.

---

### Finding Description

The `FinalizeTransferSol` account struct declares the recipient as:

```rust
/// CHECK: this can be any type of account
#[account(mut)]
pub recipient: UncheckedAccount<'info>,
``` [1](#0-0) 

The only constraint is `mut`. There is no `owner`, `address`, or any other identity check.

The payload that arrives from NEAR via Wormhole is:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
``` [2](#0-1) 

There is **no `recipient` field**. The intended Solana recipient is never encoded in the Wormhole message that NEAR emits for a SOL finalization. Consequently, `process` performs the transfer with no cross-check:

```rust
transfer(
    CpiContext::new_with_signer(
        self.common.system_program.to_account_info(),
        Transfer {
            from: self.sol_vault.to_account_info(),
            to: self.recipient.to_account_info(),   // fully relayer-controlled
        },
        &[&[SOL_VAULT_SEED, &[self.config.bumps.sol_vault]]],
    ),
    data.amount.try_into()...?,
)?;
``` [3](#0-2) 

Contrast this with `init_transfer_sol`, where the outgoing Wormhole message **does** embed the NEAR recipient string in the payload:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    Pubkey::default(),
))?)?;
``` [4](#0-3) 

and `InitTransferPayload` carries `pub recipient: String` at field 7. [5](#0-4) 

The reverse path (NEAR → Solana) simply omits the recipient from the payload entirely, leaving the relayer free to supply any pubkey.

---

### Impact Explanation

The Solana System Program's `Transfer` instruction places no ownership constraint on the `to` account; it succeeds for any existing account, including program-owned PDAs. When a relayer passes a program-owned PDA as `recipient`:

- Lamports are credited to the PDA.
- The user has no authority over that PDA and cannot withdraw.
- Unless the owning program exposes a withdrawal instruction (which an attacker-chosen PDA would not), the SOL is permanently inaccessible.
- The nonce is consumed (`UsedNonces::use_nonce`), so the transfer cannot be re-finalized to the correct recipient. [6](#0-5) 

The same missing check also enables the more severe variant: the relayer passes their own system-owned wallet as `recipient`, constituting direct theft of the user's SOL.

---

### Likelihood Explanation

- No privileged role is required. Any account can be `common.payer` (the relayer); there is no allowlist or signer restriction on who may call `finalize_transfer_sol`.
- The Wormhole message is publicly observable on-chain. A racing or front-running relayer can intercept a legitimate finalization and submit it first with a substituted recipient.
- The nonce-consumption mechanism makes the attack one-shot and irreversible.

---

### Recommendation

Embed the intended Solana recipient `Pubkey` in `FinalizeTransferPayload` on the NEAR side, and add a constraint in `FinalizeTransferSol` that enforces the match:

```rust
#[account(
    mut,
    address = data.payload.recipient @ ErrorCode::InvalidArgs,
)]
pub recipient: SystemAccount<'info>,
```

Using `SystemAccount` (instead of `UncheckedAccount`) additionally enforces that the recipient is a wallet account owned by the System Program, preventing SOL from being sent to program-owned accounts that cannot spend it.

---

### Proof of Concept

```rust
// 1. Observe a pending FinalizeTransferPayload Wormhole VAA for a NEAR→Solana SOL transfer.
// 2. Derive or choose any program-owned PDA, e.g. a PDA of the bridge program itself.
let locked_pda = Pubkey::find_program_address(&[b"any_seed"], &bridge_program_id).0;

// 3. Submit finalize_transfer_sol with `recipient = locked_pda` before the legitimate relayer.
//    The nonce check passes (first use), the System Program transfer succeeds,
//    and the nonce is permanently consumed.
client.finalize_transfer_sol(signed_vaa, locked_pda).await?;

// 4. The legitimate user's SOL now sits in `locked_pda`.
//    No subsequent call can re-finalize (NonceAlreadyUsed).
//    The user's funds are irrecoverably locked.
```
