### Title
Division-Before-Multiplication in `NewDynamicFeeChecker` Causes Systematic Fee Under-Collection - (File: `ante/evm/fee_checker.go`)

### Summary
`NewDynamicFeeChecker` computes `feeCap` by integer-dividing the total submitted fee by gas, then multiplies the result back by gas to produce `effectiveFee`. This division-before-multiplication pattern causes the protocol to systematically collect less fee than the user submitted, mis-accounting fees on every Cosmos-native EIP-1559 transaction where `fee % gas ≠ 0`.

### Finding Description

In `ante/evm/fee_checker.go`, the `NewDynamicFeeChecker` function computes the effective fee for Cosmos transactions using EIP-1559 logic as follows:

```go
// line 83
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // integer division: truncates fee/gas
...
// line 91
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()),
)
// line 94-99
effectiveFee := sdk.Coins{{
    Denom:  denom,
    Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)),  // multiply back by gas
}}
``` [1](#0-0) 

`sdkmath.Int.Quo` performs integer (floor) division. When `fee` is not an exact multiple of `gas`, the truncation means:

```
feeCap  = fee / gas          (truncated, loses fee % gas)
effectivePrice ≤ feeCap
effectiveFee = effectivePrice * gas ≤ feeCap * gas = (fee/gas)*gas ≤ fee
```

The maximum underpayment per transaction is `gas − 1` units of the EVM denom. Because `effectiveFee` is the value returned to the Cosmos SDK `DeductFeeDecorator`, it is the exact amount deducted from the sender's account — the remainder `fee % gas` is never collected by the protocol.

The same sequential-division pattern also appears in `getTxPriority`:

```go
gasPrice := fee.Amount.QuoRaw(gas)                    // first division
amt := gasPrice.Quo(types.DefaultPriorityReduction)   // second division on already-truncated value
``` [2](#0-1) 

### Impact Explanation

Every Cosmos-native EIP-1559 transaction (those carrying `ExtensionOptionDynamicFeeTx`) where `fee % gas ≠ 0` results in the protocol collecting `fee % gas` fewer units of the EVM denom than the user committed to pay. This is a systematic, per-block fee mis-accounting in the ante handler. The maximum loss per transaction is `gas − 1` units (up to ~10^7 units of the smallest denomination for high-gas transactions). Across many transactions this accumulates into measurable underpayment of protocol fees, constituting mis-accounting of valid user funds in the ante handler path.

### Likelihood Explanation

This is triggered by any unprivileged user submitting a Cosmos-wrapped EIP-1559 transaction with a fee amount that is not an exact multiple of the gas limit — a common occurrence since wallets typically compute fees as `gasPrice × gasLimit` in integer arithmetic and the submitted coin amount may differ. No special privileges or conditions are required; the code path is exercised on every `DeliverTx` for such transactions.

### Recommendation

Avoid dividing before multiplying. Compute `effectiveFee` directly without first deriving `feeCap` via integer division:

1. Compute `effectiveGasPrice = EffectiveGasPrice(baseFee, fee/gas, maxPriorityPrice)` only for the comparison/rejection check.
2. Compute `effectiveFee = effectiveGasPrice * gas` using the un-truncated `effectiveGasPrice` derived from `baseFee` and `maxPriorityPrice` directly, not from the truncated `feeCap`.

Alternatively, use `sdkmath.LegacyDec` (which preserves 18 decimal places) for the intermediate `feeCap` computation before converting back to `Int` only at the final multiplication step, following the same pattern recommended in the GMX report (multiply all numerators first, then divide by the product of denominators).

### Proof of Concept

Suppose:
- `gas = 1_000_000`
- `baseFee = 1_000_000_000` (1 Gwei)
- User submits `fee = 1_000_000_000 * 1_000_000 + 999_999 = 1_000_000_999_999` units

Step-by-step through the code:
```
feeCap = 1_000_000_999_999 / 1_000_000 = 1_000_000  (truncated, loses 999_999)
effectivePrice = min(baseFee + tip, feeCap) = 1_000_000
effectiveFee = 1_000_000 * 1_000_000 = 1_000_000_000_000
```

User submitted `1_000_000_999_999` but is charged only `1_000_000_000_000`. The `999_999` units are never deducted — the protocol under-collects by `999_999` units on this single transaction. At scale (e.g., 1000 such transactions per block), the protocol loses ~10^9 units of the EVM denom per block.
