### Title
Malicious P2P Peer Can Store Body with Transaction Count Mismatched to `transaction_commitment` â€” (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The p2p sync header path accepts `SignedBlockHeader` messages from peers with **no block-hash recomputation, no cryptographic signature verification, and no cross-check between `n_transactions` and `transaction_commitment`**. The transaction sync path then blindly uses the stored `n_transactions` as the loop bound and stores whatever the peer sends without verifying the result against `transaction_commitment`. A malicious peer can therefore cause the node to store a body whose transaction count is attacker-chosen and whose stored `transaction_commitment` does not cover the stored transactions.

---

### Finding Description

**Guard inventory in `HeaderStreamBuilder::parse_data_for_block`**

The only two checks performed before the header is accepted are:

1. Block number ordering (lines 104â€“113)
2. Signature vector length == 1 (line 115) [1](#0-0) 

There is **no** call to `calculate_block_hash` to recompute and compare the `block_hash` field, **no** call to `verify_block_signature` to cryptographically validate the signature, and **no** comparison of `n_transactions` against `transaction_commitment` or `concatenated_counts`. A grep across the entire `crates/apollo_p2p_sync/` tree for `verify_block_signature` returns zero matches. The TODO comment in `convert_sync_block_to_block_data` explicitly acknowledges the missing check:

```

// concatenated_counts.
``` [2](#0-1) 

The accepted header is written to storage verbatim â€” `n_transactions`, `transaction_commitment`, and `block_hash` are all peer-supplied and unverified. [3](#0-2) 

**Guard inventory in `TransactionStreamFactory::parse_data_for_block`**

The loop bound is read directly from the stored (unverified) header field:

```rust
let target_transaction_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .n_transactions;
while current_transaction_len < target_transaction_len { â€¦ }
``` [4](#0-3) 

After the loop, the collected `BlockBody` is returned and written to storage with no comparison against `transaction_commitment`. The TODO on line 88 acknowledges that transaction hashes are not validated either. [5](#0-4) 

**Block hash structure confirms the mismatch is undetected**

`calculate_block_hash` chains `concatenated_counts` (which encodes `n_transactions`) and `transaction_commitment` into the Poseidon hash. [6](#0-5) 

A header with `n_transactions=5` but `transaction_commitment` over 2 transactions would have a wrong `block_hash`. Because the p2p sync client never recomputes the block hash, the peer can supply any `block_hash` value and it will be stored as authoritative.

**Attack chain**

1. Malicious peer sends `SignedBlockHeader` with `n_transactions=5`, `transaction_commitment` = root over 2 transactions, and an arbitrary `block_hash`.
2. `HeaderStreamBuilder::parse_data_for_block` passes both checks (correct block number, signature vector length == 1) and returns `Ok(Some(signed_block_header))`.
3. `write_to_storage` calls `append_header`, persisting the peer-supplied `n_transactions=5` and mismatched `transaction_commitment`.
4. `TransactionStreamFactory::parse_data_for_block` reads `target_transaction_len = 5` and loops until 5 transactions are received from the same peer.
5. All 5 transactions are pushed into `block_body` and written via `append_body` â€” no commitment recomputation occurs.
6. The stored body now has 5 transactions; the stored `transaction_commitment` covers only 2.

---

### Impact Explanation

Any downstream consumer that reads the stored header and body will observe a `transaction_commitment` that does not match the actual stored transactions. RPC endpoints serving `starknet_getBlockWithTxs` or `starknet_getTransactionByHash` will return an authoritative-looking response whose transaction count is attacker-chosen and whose commitment is inconsistent with the body. This maps to **High â€” RPC returns an authoritative-looking wrong value**.

The question's "Critical" classification is not supported by the allowed impact scope: the mismatch is a sync/storage integrity issue, not an execution-layer acceptance of an invalid transaction.

---

### Likelihood Explanation

Any peer that the node connects to for p2p sync can exploit this. No cryptographic capability is required â€” the attacker only needs to craft a protobuf `SignedBlockHeader` message with an inflated `n_transactions` field and a signature vector of length 1 (any bytes). The attack is deterministic and requires no brute force.

---

### Recommendation

1. **Recompute and verify `block_hash`** in `HeaderStreamBuilder::parse_data_for_block` before accepting a header. The block hash commits to both `concatenated_counts` (which encodes `n_transactions`) and `transaction_commitment`, so a single hash check closes both gaps simultaneously.
2. **Verify the block signature cryptographically** using `verify_block_signature` against the known sequencer public key.
3. **Recompute `transaction_commitment`** from the collected transactions at the end of `TransactionStreamFactory::parse_data_for_block` and compare it against the stored header's `transaction_commitment` before calling `write_to_storage`.
4. Resolve the acknowledged TODO: cross-check `n_transactions` and `state_diff_length` against `concatenated_counts` in `convert_sync_block_to_block_data`.

---

### Proof of Concept

```rust
// Pseudocode â€” store a header with n_transactions=5 but commitment over 2 txs,
// feed 5 FullTransactions through parse_data_for_block, assert the mismatch.

let two_tx_commitment = calculate_transaction_commitment::<Poseidon>(&two_leaf_elements);
let header = BlockHeader {
    n_transactions: 5,
    transaction_commitment: Some(two_tx_commitment),
    ..Default::default()
};
storage_writer.begin_rw_txn()?.append_header(BlockNumber(0), &header)?.commit()?;

// Feed 5 transactions through the mock response manager
// (parse_data_for_block loops until current_transaction_len == 5)
let (block_body, _) = parse_data_for_block(&mut mock_manager, BlockNumber(0), &reader)
    .await.unwrap().unwrap();

// Assert: 5 transactions stored, commitment does not match
assert_eq!(block_body.transactions.len(), 5);
let actual_commitment = calculate_transaction_commitment::<Poseidon>(&five_leaf_elements);
assert_ne!(actual_commitment, two_tx_commitment); // commitment covers only 2
```

The stored body will have 5 transactions while `transaction_commitment` covers only 2, violating the commitment invariant with no error raised anywhere in the production path.
