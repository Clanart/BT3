### Title
Unauthenticated `SetProof` on `RemoteProofManagerServer` Poisons Proof Cache and Bypasses Cryptographic Proof Verification — (File: `crates/apollo_proof_manager/src/communication.rs`)

---

### Summary

The `RemoteProofManagerServer` is deployed with `bind_ip: "0.0.0.0"` and **no authentication layer** of any kind. Its `handle_request` dispatcher forwards `ProofManagerRequest::SetProof` directly to `ProofManager::set_proof`, which stores arbitrary proof bytes keyed by `proof_facts.hash()` without cryptographic verification. Because `run_proof_verification` in `transaction_converter.rs` unconditionally skips `starknet_proof_verifier::verify_proof` whenever `contains_proof` returns `true`, an attacker who pre-populates the proof manager with invalid bytes for a known `proof_facts` hash causes every subsequent legitimate transaction carrying those same `proof_facts` to have its proof accepted without verification. The attacker's invalid bytes also permanently displace the user's valid proof (first-write-wins), so the invalid proof is the one retrieved and propagated to consensus.

---

### Finding Description

**Root cause — no access control on `SetProof`:**

`RemoteComponentServer` is a plain HTTP/2 server over raw TCP. Its `RemoteServerConfig` contains only `bind_ip`, `max_streams_per_connection`, `set_tcp_nodelay`, and keepalive fields — no TLS, no token, no caller identity. [1](#0-0) 

In the distributed deployment the proof manager is configured with `"remote_server_config.#is_none": false` and `"remote_server_config.bind_ip": "0.0.0.0"`, making it reachable from any pod in the cluster (or externally if the port is exposed). [2](#0-1) 

The request handler dispatches `SetProof` with zero access control: [3](#0-2) 

**Root cause — `set_proof` stores without verifying:**

`ProofManager::set_proof` stores the caller-supplied bytes keyed by `proof_facts.hash()` with a first-write-wins policy. No cryptographic check is performed here. [4](#0-3) 

**Root cause — `run_proof_verification` skips verification when proof is already stored:**

The only place where `starknet_proof_verifier::verify_proof` is called is `run_proof_verification`. It checks `contains_proof` first and returns `Ok(false)` (skipped) if the proof already exists, without ever inspecting the stored bytes. [5](#0-4) 

**Consequence — attacker's invalid proof displaces the user's valid proof:**

When the gateway later calls `store_proof_in_proof_manager` with the user's valid proof, `ProofManager::set_proof` silently returns early because the proof is already stored. [6](#0-5) 

**Consequence — invalid proof is retrieved for consensus propagation:**

`convert_internal_rpc_tx_to_rpc_tx` calls `get_proof` to reconstruct the full RPC transaction for consensus. It retrieves whatever bytes are stored — the attacker's invalid bytes. [7](#0-6) 

---

### Impact Explanation

An attacker who can reach the proof manager's HTTP/2 port (any pod in the cluster, or externally if the port is exposed) can:

1. Pre-store invalid proof bytes for a target `proof_facts` hash.
2. Cause every subsequent `InvokeV3` transaction carrying those `proof_facts` to pass through the gateway and consensus flow **without cryptographic proof verification**.
3. Ensure the invalid bytes are the ones included in the block and eventually submitted to L1, where they will fail on-chain verification.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The `validate_proof_facts` check in the blockifier validates the program hash, block hash, block number, and config hash embedded in `proof_facts`, but it does **not** verify the actual proof bytes cryptographically. [8](#0-7) 

The cryptographic check lives exclusively in `run_proof_verification`, which is bypassed.

---

### Likelihood Explanation

- The `RemoteProofManagerServer` is bound to `0.0.0.0` with no authentication in the production distributed deployment config.
- Any process reachable to the proof manager's port can call `SetProof` with arbitrary data.
- The attacker needs to know the `proof_facts` of a target transaction. These are visible in the mempool (they are part of the `RpcInvokeTransactionV3` payload), so a racing attacker can extract them and call `SetProof` before the gateway processes the transaction.
- The first-write-wins policy means a single successful pre-store permanently poisons that `proof_facts` slot.

---

### Recommendation

1. **Add authentication to `RemoteProofManagerServer`**: Require a shared secret or mTLS on all connections before dispatching any request variant.
2. **Verify before storing**: `ProofManager::set_proof` should call `starknet_proof_verifier::verify_proof` before persisting, so that the storage invariant "every stored proof is cryptographically valid for its `proof_facts`" holds regardless of the caller.
3. **Restrict `SetProof` to internal callers only**: If the remote interface must remain unauthenticated for other request variants, at minimum reject `SetProof` at the remote boundary and only allow it through the local channel.

                    .await
                    .map_err(|e| ProofManagerError::ProofStorage(e.to_string())),

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
