### Title
Non-BTC Token with BTC Recipient Causes Permanent EVM Fund Lock — (`near/omni-bridge/src/lib.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

The EVM `initTransfer` accepts any `(tokenAddress, recipient)` combination without validating that a BTC-format recipient is only used with the registered UTXO-chain native token. When the NEAR-side `fin_transfer_callback` processes such a transfer, `process_fin_transfer_to_other_chain` panics at the `require!(token == btc_account_id)` guard. Because NEAR panics roll back all state changes in the callback but cannot undo the already-executed EVM token lock/burn, the user's funds are permanently frozen with no refund path.

---

### Finding Description

**Step 1 — EVM `initTransfer` accepts arbitrary recipient strings**

`OmniBridge.sol::initTransfer` takes a `string calldata recipient` and performs only a fee check. It burns or locks the token, then emits `InitTransfer`. There is no validation that a `"btc:…"` recipient is paired with the registered BTC token. [1](#0-0) 

**Step 2 — EVM event is parsed; recipient string becomes `OmniAddress::Btc`**

`OmniAddress::from_str` parses `"btc:bc1qxxx"` into `OmniAddress::Btc("bc1qxxx")` without any token-compatibility check. [2](#0-1) 

**Step 3 — `fin_transfer_callback` routes to `process_fin_transfer_to_other_chain`**

Because the recipient is not `OmniAddress::Near`, the callback unconditionally calls `process_fin_transfer_to_other_chain`. There is no pre-check for token/recipient chain compatibility at this routing point. [3](#0-2) 

**Step 4 — `require!` panics; no refund is emitted**

Inside `process_fin_transfer_to_other_chain`, the contract fetches the registered UTXO-chain native token (`btc_account_id`) and asserts the transfer token matches it. For a USDC transfer with a BTC recipient, this assertion fails and the function panics. [4](#0-3) 

The panic rolls back all NEAR state changes (including `add_fin_transfer` at line 1985), so the transfer is never marked as finalized. Every subsequent relay attempt for the same EVM event will panic identically. No `RefundTransfer` or equivalent cross-chain refund event is emitted.

**Step 5 — EVM tokens are permanently locked**

The EVM `initTransfer` already burned bridge tokens (`BridgeToken.burn`) or transferred native tokens to the contract before emitting the event. These operations are irreversible from NEAR. There is no EVM-side mechanism to reclaim funds when the NEAR callback fails. [5](#0-4) 

---

### Impact Explanation

User funds (any ERC-20 token) are permanently frozen in the EVM bridge contract or permanently burned. The NEAR side will panic on every finalization attempt for the affected transfer ID, and no cross-chain refund event is ever produced. This matches the **Critical** impact class: irrecoverable lock of user funds in bridge flows.

---

### Likelihood Explanation

The attack requires only a single unprivileged call to the public `initTransfer` function with a non-BTC token address and a `"btc:…"` recipient string. No special role, leaked key, or colluding party is needed. A user could trigger this accidentally (e.g., copy-pasting a BTC address into a UI that does not enforce token/chain pairing) or intentionally as a griefing vector against themselves or others. The EVM contract provides no guard, and the NEAR contract provides no recovery path.

---

### Recommendation

**Option A (preferred) — Validate on EVM**: Parse the recipient prefix in `initTransfer` and revert if a UTXO-chain prefix (`"btc:"`, `"zcash:"`) is detected for a token that is not the registered UTXO-chain native token. This requires the EVM contract to maintain a mapping of UTXO-chain token addresses.

**Option B — Validate on NEAR before locking**: In `fin_transfer_callback`, before routing to `process_fin_transfer_to_other_chain`, check whether the recipient is a UTXO chain and whether the token matches the registered native token. If not, emit a cross-chain refund event (mint back on EVM) instead of panicking.

**Option C — Replace `require!` with graceful error**: Change the `require!` in `process_fin_transfer_to_other_chain` to a non-panicking branch that emits a `RefundTransfer` event, allowing the EVM side to release the locked tokens.

---

### Proof of Concept

```
1. Deploy USDC as a bridge token on EVM (or use any non-BTC ERC-20).
2. Call OmniBridge.initTransfer(
       usdcAddress,
       1_000_000,   // amount
       100,         // fee
       0,           // nativeFee
       "btc:bc1qar0srrr7xfkvy5l643lydnw9re59gtzzwf5mdq",  // BTC-format recipient
       ""
   )
   → USDC is burned/locked; InitTransfer event emitted.
3. Relay the EVM proof to NEAR via fin_transfer.
4. NEAR fin_transfer_callback calls process_fin_transfer_to_other_chain.
5. require!(token == btc_account_id) fails → panic.
6. Observe: transfer never finalized; no refund event; USDC permanently lost.
``` [6](#0-5) [1](#0-0)

        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue

            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
