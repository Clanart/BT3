### Title
Native Fee Added to Storage-Payer's Available Balance Without Deduction, Enabling Theft of Protocol NEAR — (`near/omni-bridge/src/storage.rs`)

### Summary

`try_to_transfer_balance_from_message_account` credits the entire message-account balance (which includes the native fee) to the storage-payer's `available` balance, but never subtracts the native-fee portion. Because `storage_withdraw` has no pending-transfer guard, the initiating user can immediately reclaim the native fee while the relayer is still owed that same amount from the contract's NEAR balance — draining the contract by `native_fee` per exploit.

### Finding Description

During an `init_transfer` with a non-zero `native_token_fee`, the user deposits `native_fee + min_message_storage` NEAR to a dedicated message account via `storage_deposit`. When `init_transfer_resume` fires, it calls `try_to_transfer_balance_from_message_account`:

```rust
// storage.rs L281
storage.available = storage.available.saturating_add(balance.total);

if storage.available < required_storage_payer_balance.saturating_add(native_fee) {
    return Err(StorageError::SignerNotEnoughBalance);
}

self.accounts_balances.insert(storage_payer, &storage);
self.accounts_balances.remove(account_id);
```

`balance.total` equals `native_fee + min_message_storage`. The function:
1. Adds the full `balance.total` (including `native_fee`) to `storage.available` of the storage-payer.
2. Validates that `storage.available >= required_storage_payer_balance + native_fee` — a correct sufficiency check.
3. Persists the inflated `storage.available` without ever subtracting `native_fee`.
4. Removes the message account.

Subsequently, `init_transfer_internal` calls `update_storage_balance` which deducts only `required_storage_payer_balance` (the transfer-message storage cost). The `native_fee` is never deducted from `storage.available`.

After `init_transfer_resume` completes, the storage-payer's available balance is:

```
original_available + native_fee + min_message_storage − required_storage_payer_balance
```

The `min_message_storage` portion is legitimately freed (message account removed). But `native_fee` is still owed to the relayer and will be paid from the contract's NEAR balance when `claim_fee` → `send_fee_internal` executes:

```rust
// lib.rs L2665-2667
Promise::new(fee_recipient.clone())
    .transfer(NearToken::from_yoctonear(transfer_message.fee.native_fee.0))
    .detach();
```

Because `storage_withdraw` has no pending-transfer guard:

```rust
// storage.rs L187-212
pub fn storage_withdraw(&mut self, amount: Option<NearToken>) -> StorageBalance {
    assert_one_yocto();
    let account_id = env::predecessor_account_id();
    ...
    Promise::new(account_id).transfer(to_withdraw).detach();
    storage
}
```

the storage-payer can call `storage_withdraw(native_fee)` immediately after `init_transfer_resume`, reclaiming the native fee. The relayer later also receives `native_fee` from the contract. The contract pays out `2 × native_fee` but only ever received `native_fee + min_message_storage` from the user.

### Impact Explanation

Every `init_transfer` with a non-zero `native_token_fee` allows the sender to recover the native fee via `storage_withdraw` while the relayer is still paid the same amount from the contract's NEAR balance. The contract's NEAR balance is depleted by `native_fee` per exploit. Repeated exploitation drains the contract's NEAR reserve, eventually preventing legitimate relayer fee payments and breaking bridge collateralization. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation

The attack requires no special role or privilege. Any user who initiates a NEAR-origin transfer with a non-zero `native_token_fee` can exploit this. The steps are fully within normal user interactions (call `ft_transfer_call`, `storage_deposit`, then `storage_withdraw`). The profit per exploit equals the native fee the user chose to set, incentivizing large native fees to maximize extraction.

### Recommendation

After adding `balance.total` to `storage.available`, subtract `native_fee` so the storage-payer is not credited with funds that belong to the relayer:

```rust
storage.available = storage.available
    .saturating_add(balance.total)
    .saturating_sub(native_fee);   // native_fee is owed to the relayer, not the payer
```

Alternatively, only add `balance.total - native_fee` (i.e., the freed storage portion) to `storage.available`, and handle the native fee separately as a reserved amount that is never reflected in the user's withdrawable balance.

### Proof of Concept

1. Sender has storage balance `A` (available) in the bridge contract.
2. Sender calls `ft_transfer_call` with `native_token_fee = N` (e.g., 1 NEAR). Contract yields.
3. Sender calls `storage_deposit` for the message account, attaching `N + min_storage`.
4. `init_transfer_resume` fires → `try_to_transfer_balance_from_message_account` sets sender's `available = A + N + min_storage`, then `init_transfer_internal` deducts `required_storage` → sender's `available = A + N + min_storage - required_storage`.
5. Sender immediately calls `storage_withdraw(N)` (attaching 1 yoctoNEAR). Contract transfers `N` NEAR to sender. Sender has recovered the native fee.
6. Relayer calls `claim_fee` → `send_fee_internal` transfers `N` NEAR to relayer from the contract's balance.
7. Net result: contract paid out `2N` NEAR but only received `N + min_storage` from the sender. Contract balance is short by `N`.
