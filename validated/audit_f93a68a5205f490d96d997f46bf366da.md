### Title
Missing Zero-Value Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash is non-zero or that it corresponds to a declared class. This allows any contract to replace its own class hash with `0` (or any undeclared hash), permanently making the contract unexecutable and freezing all funds it holds. The OS itself acknowledges this gap with an explicit TODO comment.

---

### Finding Description

In `execute_replace_class` (`execution/syscall_impls.cairo`, lines 878–916), the OS reads `class_hash` directly from the `ReplaceClassRequest` struct and immediately writes it into the contract's `StateEntry` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

Two checks are entirely absent:

1. **`class_hash != 0`** — there is no `assert_not_zero(class_hash)` guard.
2. **`class_hash` is declared** — there is no lookup into `contract_class_changes` to confirm the hash exists.

The TODO comment at line 898 explicitly acknowledges the missing check. Compare this to `execute_declare_transaction`, which does enforce `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`: [2](#0-1) 

The asymmetry is the root cause: declaration enforces non-zero compiled class hash, but class replacement enforces nothing.

This is the direct analog of M-08: just as `_delegate` in ERC721Votes accepted `address(0)` without a guard and corrupted vote accounting, `execute_replace_class` accepts `class_hash=0` without a guard and corrupts contract-class state.

---

### Impact Explanation

Once a contract's `StateEntry.class_hash` is set to `0` (or any undeclared hash), every subsequent call to that contract will fail at the class-lookup stage inside the OS execution engine — the OS cannot find an entry point for `class_hash=0` in the compiled class facts bundle. There is no recovery path: the state transition is committed to the global state root, and no syscall exists to undo a `replace_class`. Any ERC-20 tokens, ETH, or protocol-controlled funds held in that contract's storage are permanently inaccessible.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack path is reachable by an unprivileged user through any upgradeable contract that forwards a caller-supplied class hash to `replace_class`. This is a common pattern in proxy/upgrade architectures on StarkNet. The attacker supplies `class_hash=0`; the contract relays it to the OS syscall; the OS writes it unconditionally. No privileged role, leaked key, or operator cooperation is required. The OS's own TODO comment confirms the developers are aware the check is absent.

---

### Recommendation

Inside `execute_replace_class`, before updating `contract_state_changes`, add:

1. `assert_not_zero(class_hash)` — reject zero class hash immediately.
2. A lookup into `contract_class_changes` (or the compiled class facts bundle) to confirm `class_hash` is a declared class — reject undeclared hashes.

This mirrors the guard already present in `execute_declare_transaction` (`assert_not_zero(compiled_class_hash)`) and closes the symmetry gap.

---

### Proof of Concept

1. Deploy an upgradeable DeFi vault contract that holds user funds and exposes `upgrade(new_class_hash: felt)` which internally calls the `replace_class` syscall with the caller-supplied hash.
2. As an unprivileged user, call `upgrade(0)`.
3. The vault contract issues `replace_class(0)` to the OS.
4. `execute_replace_class` reads `class_hash = 0` from the request, finds no validation, and writes `StateEntry(class_hash=0, ...)` into `contract_state_changes`.
5. After state commitment, the vault's global state entry has `class_hash=0`.
6. Any subsequent call to the vault (withdraw, transfer, etc.) fails: the OS cannot resolve `class_hash=0` to any compiled class, so no entry point is found.
7. All user funds locked in the vault's storage are permanently frozen with no recovery path.
