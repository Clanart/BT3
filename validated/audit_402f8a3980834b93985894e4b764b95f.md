### Title
Missing Zero-Value Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts `class_hash = 0` without validation. Since `UNINITIALIZED_CLASS_HASH = 0` is the sentinel value for a contract with no code, any contract can irrevocably set its own class hash to zero, permanently bricking itself and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `request.class_hash` and writes it directly into the contract's `StateEntry` with no check that the value is non-zero:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The constant `UNINITIALIZED_CLASS_HASH = 0` is the canonical sentinel for "no class deployed": [2](#0-1) 

`deploy_contract` explicitly enforces that a contract's class hash must equal `UNINITIALIZED_CLASS_HASH` (0) before deployment, confirming that 0 is the "no code" state: [3](#0-2) 

The same missing validation exists in the deprecated path: [4](#0-3) 

The TODO comment at line 898 acknowledges the missing class existence check, but even a class-existence check would not catch `class_hash = 0` unless it explicitly rejects zero. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After a successful `replace_class(0)` call, the contract's `StateEntry.class_hash` is committed on-chain as 0. Any subsequent call to that contract will resolve to `class_hash = 0`, for which no entry points exist. The contract becomes permanently inert. All ERC-20 balances, NFTs, or other assets held by the contract are irrecoverably frozen. The state transition is accepted and committed by the OS with no revert path.

---

### Likelihood Explanation

Any deployed contract can issue the `replace_class` syscall with `class_hash = 0`. This requires no privileged role — it is reachable by:
- A buggy contract that passes an uninitialized variable as the class hash argument.
- A malicious contract deliberately self-destructing to freeze user funds.
- A griefing attack on a shared infrastructure contract (e.g., a multisig or vault).

The syscall is dispatched unconditionally through `execute_syscalls` whenever the selector matches `REPLACE_CLASS_SELECTOR`: [6](#0-5) 

---

### Recommendation

Add an `assert_not_zero(class_hash)` guard immediately after reading `request.class_hash` in `execute_replace_class`, before the state update:

```cairo
let class_hash = request.class_hash;
assert_not_zero(class_hash);  // Reject UNINITIALIZED_CLASS_HASH (0).
```

Apply the same fix to the deprecated variant in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Deploy a contract `Vault` holding user funds.
2. From within `Vault`, call the `replace_class` syscall with `class_hash = 0`.
3. The OS executes `execute_replace_class`; no validation fires; `dict_update` commits `StateEntry(class_hash=0, ...)` for `Vault`'s address.
4. The block is proven and the new state root (with `Vault.class_hash = 0`) is accepted on L1.
5. Any future `call_contract` targeting `Vault` resolves to `class_hash = 0`, finds no entry points, and reverts.
6. All funds in `Vault` are permanently frozen with no recovery path.
