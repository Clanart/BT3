### Title
Detached Promise After `unlock_tokens_if_needed` in Fast-Transfer Relayer Reimbursement Causes Permanent Accounting Corruption and Irrecoverable Relayer Fund Loss — (`near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain`, the bridge updates its internal `locked_tokens` accounting state (via `unlock_tokens_if_needed`) **before** dispatching the relayer reimbursement token transfer as a **detached promise** (`.detach()`). Because no callback is attached, a failed token transfer is silently ignored: the accounting decrement is permanent, the fast transfer is marked finalised, and the relayer's funds are irrecoverably lost. This is the direct NEAR analog of the Dexter call-injection/state-desync class described in the external report.

---

### Finding Description

`process_fin_transfer_to_other_chain` is invoked from `fin_transfer_callback` when the finalized transfer's recipient is on a non-NEAR chain. When a fast transfer was previously executed for the same transfer ID, the function takes the relayer-reimbursement branch:

**Step 1 — accounting state mutated first:** [1](#0-0) 

`unlock_tokens_if_needed` immediately decrements `locked_tokens[origin_chain][token]` by `transfer_message.amount`. This write is committed to contract state in the same NEAR transaction frame.

**Step 2 — external token transfer dispatched with `.detach()`:** [2](#0-1) 

`send_tokens(…).detach()` fires a cross-contract call to the token contract (`ft_transfer`, `mint`, or `near_withdraw`) with **no callback registered**. NEAR's promise model guarantees that a detached promise's failure is never surfaced to the caller; the calling function has already returned successfully.

**Step 3 — fast transfer permanently closed:** [3](#0-2) 

`mark_fast_transfer_as_finalised` is called unconditionally, regardless of whether the token transfer will succeed.

**Failure path for `send_tokens`:** [4](#0-3) 

For non-deployed, non-wNEAR tokens the path is `ft_transfer`, which requires the recipient (the fast-transfer relayer) to have a storage deposit registered with the token contract. If the relayer's account lacks that registration, the `ft_transfer` call panics and the promise fails silently.

**Contrast with the `process_fin_transfer_to_near` path**, which correctly attaches a callback: [5](#0-4) 

`fin_transfer_send_tokens_callback` is registered there and calls `revert_lock_actions` on failure: [6](#0-5) 

The "other chain" path has no equivalent revert guard.

---

### Impact Explanation

When `send_tokens` fails in the detached path:

1. **`locked_tokens` is permanently under-counted.** The bridge's internal ledger records fewer locked tokens for the origin chain than are actually held. This breaks bridge collateralization: the bridge can subsequently accept and finalize additional inbound transfers from the origin chain that it does not have sufficient backing for, enabling unbacked supply on the destination chain.

2. **The fast-transfer relayer's funds are irrecoverably lost.** The relayer already disbursed tokens to the end-recipient during the fast transfer. The reimbursement promise fails silently, and because `mark_fast_transfer_as_finalised` has already been called, no retry path exists. The relayer cannot recover their capital.

3. **The finalization record is consumed.** `add_fin_transfer` marks the transfer ID as used, so the proof cannot be re-submitted to trigger a fresh reimbursement attempt.

Matches: **Critical — permanent freezing / irrecoverable lock of user funds in bridge flows**, and **High — accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

The trigger condition — a fast-transfer relayer account that lacks a storage deposit for the bridged token — is realistic:

- Relayers may register storage on the token contract for the tokens they *send* during fast transfers but not for the token they expect to *receive* as reimbursement (e.g., a different wrapped asset).
- A relayer operating across many token types may miss a storage registration for a newly deployed bridge token.
- An adversary who observes a pending fast transfer can front-run the `fin_transfer` proof submission by deregistering (withdrawing) their storage deposit on the token contract, guaranteeing the reimbursement fails.

The `fin_transfer` entry point is restricted to trusted relayers (`#[trusted_relayer]`), but the fast-transfer execution that sets up the vulnerable state may be open to any participant. The trusted relayer submitting the proof is an innocent party; the accounting corruption and fund loss occur regardless of their correctness.

---

### Recommendation

Replace `.detach()` with a chained callback that calls `revert_lock_actions` on failure, mirroring the pattern already used in `process_fin_transfer_to_near`:

```rust
self.send_tokens(token, relayer, amount, "")
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(SEND_TOKENS_CALLBACK_GAS)
            .fast_transfer_reimbursement_callback(
                fast_transfer.id(),
                lock_actions,
            ),
    );
```

The callback should:
- On success: call `mark_fast_transfer_as_finalised`.
- On failure: call `revert_lock_actions` to restore `locked_tokens`, and leave the fast transfer record open so a retry can be attempted after the relayer registers storage.

---

### Proof of Concept

1. Relayer R executes a fast transfer for transfer ID T, sending 1000 USDC to the end-recipient on NEAR. R's account has no storage deposit on the USDC token contract for receiving tokens back.
2. A trusted relayer submits the proof for T via `fin_transfer`.
3. `fin_transfer_callback` → `process_fin_transfer_to_other_chain` is called.
4. `unlock_tokens_if_needed(Eth, usdc, 1000)` decrements `locked_tokens[(Eth, usdc)]` by 1000. State committed.
5. `send_tokens(usdc, R, 1000, "")` dispatches `ft_transfer(R, 1000)` as a detached promise.
6. The USDC token contract panics: R has no storage deposit. Promise fails silently.
7. `mark_fast_transfer_as_finalised(T)` has already been called. No retry is possible.
8. Result: `locked_tokens[(Eth, usdc)]` is 1000 lower than the actual bridge balance; R has lost 1000 USDC with no recovery path; the bridge's collateralization invariant is broken for all future Eth→NEAR USDC transfers.

    pub fn revert_lock_actions(&mut self, lock_actions: &[LockAction]) {
        for lock_action in lock_actions {
            match lock_action {
                LockAction::Locked {
                    chain_kind,
                    token_id,
                    amount,
                } => {
                    self.unlock_tokens(*chain_kind, token_id, *amount);
                }
                LockAction::Unlocked {
                    chain_kind,
                    token_id,
                    amount,
                } => {
                    self.lock_tokens(*chain_kind, token_id, *amount);
                }
                LockAction::Unchanged => {}
            }
        }
    }
