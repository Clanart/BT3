### Title
`ValidResourceBounds::AllResources` Silently Downgraded to `L1Gas` in P2P Protobuf Deserialization, Causing Wrong Transaction Hash Binding via `get_tip_resource_bounds_hash` — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserialization of `ValidResourceBounds` infers the variant from the *values* of `l2_gas` and `l1_data_gas` rather than from an explicit type tag. A V3 (`AllResources`) transaction whose `l2_gas` and `l1_data_gas` are both zero is silently re-typed to `ValidResourceBounds::L1Gas` after P2P propagation. Because `get_tip_resource_bounds_hash` hashes **2 resource bounds** for `L1Gas` and **3 resource bounds** for `AllResources`, the transaction hash bound to the deserialized transaction differs from the hash the sender signed, and the gas-vector computation mode switches from `All` to `NoL2Gas`, bypassing L2-gas enforcement.

---

### Finding Description

**Root cause — type label inferred from content, not from an explicit discriminant.**

In `crates/apollo_protobuf/src/converters/transaction.rs` the conversion from the wire format to the internal enum is:

```rust
// line 431
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 variant
} else {
    ValidResourceBounds::AllResources(...)       // ← 0.13.3+ variant
})
```

There is no explicit discriminant in the protobuf message that records which variant was originally used. Any `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero is silently re-classified as `L1Gas` on the receiving node. The test file itself acknowledges this:

```rust
// crates/apollo_protobuf/src/converters/consensus_test.rs line 26-27
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
```

**Stage 1 — accepted at submission (analogous to `createVault`).**

A user submits a V3 transaction via RPC. `RpcInvokeTransactionV3` carries `AllResourceBounds` (always the `AllResources` path). The gateway converts it to `ValidResourceBounds::AllResources`. `get_tip_resource_bounds_hash` hashes **3 bounds** (L1Gas + L2Gas + L1DataGas) and the sender signs over that hash. The transaction is admitted to the mempool and propagated over P2P.

**Stage 2 — wrong type bound at execution (analogous to `exercise`/`withdraw`).**

On the receiving sequencer node the protobuf bytes are decoded. Because `l2_gas == 0` and `l1_data_gas == 0`, the variant becomes `ValidResourceBounds::L1Gas`. Two downstream invariants are now broken:

1. **Wrong transaction hash.** `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the variant:

   ```rust
   // line 203-208
   resource_felts.extend(match resource_bounds {
       ValidResourceBounds::L1Gas(_) => vec![],                          // 2 bounds
       ValidResourceBounds::AllResources(all_resources) => {
           vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 bounds
       }
   });
   ```

   The hash the receiving node computes omits the `L1DataGas` field, producing a value different from the hash the sender signed. Any component that re-derives the hash (e.g., for signature verification, transaction commitment leaf, or SNOS input) will use the wrong value.

2. **Wrong gas-vector computation mode.** `get_gas_vector_computation_mode()` returns `GasVectorComputationMode::NoL2Gas` for `L1Gas` and `GasVectorComputationMode::All` for `AllResources`. With `NoL2Gas`, the blockifier does not track or enforce the L2-gas limit. A transaction that carries `l2_gas.max_amount = 0` and would be reverted under `All` mode (out-of-L2-gas) is instead executed to completion under `NoL2Gas` mode, producing a different state diff, receipt, and event set.

The Starknet OS Cairo code (`transaction_hash.cairo` line 123-124) always asserts `n_resource_bounds = 3`, so the OS and the Rust sequencer now disagree on the hash, breaking the proof invariant.

---

### Impact Explanation

- **Wrong transaction hash bound to the transaction** after P2P propagation: the hash used for the transaction-commitment Merkle leaf, the receipt commitment, and the SNOS proof input differs from the hash the sender signed. This is a direct violation of the "Transaction conversion or signature/hash logic binds the wrong hash" impact.
- **Wrong execution result**: with `NoL2Gas` mode, a transaction whose L2-gas limit is zero can consume unbounded L2 gas and succeed where it should revert, producing a wrong state diff, wrong receipt, and wrong event set — matching the "Wrong state, receipt, event, or revert result from blockifier/syscall/execution logic" impact.
- **Wrong fee/gas accounting**: the `NoL2Gas` mode skips L2-gas cost enforcement, matching the "Incorrect fee, gas, resource accounting" impact.

---

### Likelihood Explanation

Any V3 transaction submitted with `AllResourceBounds { l2_gas: 0, l1_data_gas: 0 }` triggers the bug. This is a valid, gateway-accepted transaction format (no gateway check requires non-zero L2 gas). The bug fires on every P2P hop for such transactions. An attacker can craft such a transaction deliberately; it also occurs naturally for users who set only an L1-gas bound.

---

### Recommendation

Add an explicit one-bit discriminant to the protobuf `ResourceBounds` message (or a separate `resource_bounds_version` field) so the deserializer can reconstruct the correct variant without inspecting values. Alternatively, always serialize and deserialize as `AllResources` for all V3 transactions and remove the value-based inference entirely. The TODO comment at line 426 (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) points to the intended fix; it should be completed and the value-based variant inference removed.

---

### Proof of Concept

1. Submit an `InvokeV3` transaction via RPC with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }
   ```
2. The gateway computes `get_tip_resource_bounds_hash` with `AllResources` → hash includes `L1DataGas` field → **hash_A** (3 bounds).
3. The transaction is serialized to protobuf and broadcast over P2P.
4. The receiving node calls `ValidResourceBounds::try_from(protobuf::ResourceBounds)`. Because `l2_gas.is_zero() && l1_data_gas.is_zero()` → variant becomes `L1Gas`.
5. The receiving node recomputes `get_tip_resource_bounds_hash` with `L1Gas` → hash omits `L1DataGas` → **hash_B** (2 bounds). `hash_A ≠ hash_B`.
6. The receiving node's batcher calls `get_gas_vector_computation_mode()` → `NoL2Gas`. L2-gas enforcement is skipped. A transaction body that consumes L2 gas executes successfully instead of reverting, producing a wrong state diff and wrong receipt.
7. The transaction commitment leaf in `calculate_block_commitments` is built from `hash_B`, while the SNOS proof input expects `hash_A`, breaking the proof.

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
