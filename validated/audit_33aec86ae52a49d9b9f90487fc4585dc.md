### Title
`ProofOfInclusion::valid()` Performs Incomplete Validation — Final Root Check Is a Tautology, Enabling Forged Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle proof subsystem performs only an internal self-consistency check. Its final verification step — `existing_hash == self.root_hash()` — is a tautology that is always `true` by construction. The function never validates the proof against an external trusted tree root. An attacker who supplies a crafted `ProofOfInclusion` (possible via the Streamable deserialization path and Python bindings) can make `valid()` return `true` for any internally consistent proof, regardless of what tree it actually corresponds to.

This is the direct analog of the Lido M-15 bug: just as `price()` stopped at the WstETH→stETH intermediate step and returned that value as if it were the final ETH price, `valid()` stops at the layer-consistency intermediate step and returns `true` as if that were a complete proof of inclusion in a specific tree.

---

### Finding Description

`ProofOfInclusion` is defined in `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`:

```rust
pub struct ProofOfInclusion {
    pub node_hash: Hash,

```

The `root_hash()` method derives the claimed root entirely from the proof itself:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // taken from the proof, not from a trusted source
    } else {
        self.node_hash              // also from the proof
    }
}
```

The `valid()` method:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // existing_hash == layer.combined_hash after this
    }

    existing_hash == self.root_hash()      // TAUTOLOGY: always true
}
```

After the loop, `existing_hash` equals the last `layer.combined_hash` (because the loop returns `false` if they ever differ). `self.root_hash()` also returns the last `layer.combined_hash`. Therefore `existing_hash == self.root_hash()` is **always `true`** — it adds zero validation.

When `layers` is empty: `existing_hash = self.node_hash` and `self.root_hash() = self.node_hash`, so the final check is `self.node_hash == self.node_hash` — again always `true`.

**The missing step** (analogous to the missing stETH→ETH conversion in Lido): `valid()` never checks that `root_hash()` matches an external, independently trusted tree root. The full validation chain should be:

1. Check each layer's `combined_hash` matches the calculated hash ✅ (done)
2. Check that the final `combined_hash` equals the known trusted tree root ❌ (missing — replaced by a tautology)

**Attacker-controlled entry path**: `ProofOfInclusion` derives `Streamable`, so it can be deserialized from arbitrary bytes via `from_bytes()`. It is fully exposed to Python via `py-bindings`:

```rust
#[cfg_attr(
    feature = "py-bindings",
    pyclass(get_all, from_py_object),
    derive(PyJsonDict, PyStreamable)
)]
#[derive(Clone, Debug, std::hash::Hash, Eq, PartialEq, Streamable)]
pub struct ProofOfInclusion { ... }
```

A Python caller receiving a `ProofOfInclusion` from an untrusted peer calls `proof.valid()` → `True`, then uses `proof.root_hash()` as the authoritative tree root. The attacker controls both values.

**Concrete forge construction**: An attacker constructs:

```
ProofOfInclusion {
    node_hash: H_leaf_fake,
    layers: [
        ProofOfInclusionLayer {
            other_hash_side: Right,
            other_hash: H_sibling,
            combined_hash: internal_hash(H_leaf_fake, H_sibling),  // attacker sets this
        }
    ]
}
```

`valid()` returns `true`. `root_hash()` returns `internal_hash(H_leaf_fake, H_sibling)` — a value entirely under attacker control. Any caller that does not separately verify `proof.root_hash() == trusted_root` accepts this as a valid proof of inclusion for the leaf `H_leaf_fake` in the tree with root `internal_hash(H_leaf_fake, H_sibling)`.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Any DataLayer consumer that calls `proof.valid()` and trusts the result — without separately asserting `proof.root_hash() == known_trusted_root` — can be made to accept a forged proof of inclusion. The attacker can claim that an arbitrary leaf hash is included in a tree with an attacker-chosen root. This corrupts the DataLayer state verification logic and allows untrusted input to prove invalid state.

---

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type exposed via Python bindings with `valid()` and `root_hash()` as the primary API surface. The fuzz target and all tests call only `proof.valid()` without a separate root check, establishing a usage pattern that omits the missing validation step. Any Python application that follows this pattern when processing proofs from untrusted network peers is exploitable with a single crafted serialized `ProofOfInclusion`.

---

### Recommendation

`valid()` must accept the trusted root as a parameter and verify against it, completing the full validation chain:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    // Complete the chain: verify against the externally trusted root
    existing_hash == *trusted_root
}
```

Alternatively, rename the current `valid()` to `is_internally_consistent()` and add a separate `valid_for_root(trusted_root: &Hash) -> bool` that performs the complete check. Update all callers — including the Python bindings — accordingly.

pub fn internal_hash(left_hash: &Hash, right_hash: &Hash) -> Hash {
    let mut hasher = Sha256::new();
    hasher.update(b"\x02");
    hasher.update(left_hash.0);
    hasher.update(right_hash.0);

    Hash(Bytes32::new(hasher.finalize()))
}

pub fn calculate_internal_hash(hash: &Hash, other_hash_side: Side, other_hash: &Hash) -> Hash {
    match other_hash_side {
        Side::Left => internal_hash(other_hash, hash),
        Side::Right => internal_hash(hash, other_hash),
    }
