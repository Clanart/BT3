Audit Report

## Title
Unprotected `refundETH()`, `unwrapWETH9()`, and `sweepToken()` Allow Any Caller to Drain Stranded ETH and Tokens from the Router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` deposits exactly `value` wei as WETH when `nativeBalance >= value`, leaving `nativeBalance - value` as native ETH on the router with no attribution. `refundETH()`, `unwrapWETH9()`, and `sweepToken()` are all public/external with no access control and no caller-identity check, so any address can immediately drain the stranded balance. A user who calls any payable swap function directly with `msg.value > amountIn` loses the excess to the first caller of `refundETH()`.

## Finding Description

`pay()` in `PeripheryPayments.sol` handles the WETH branch as follows:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L73-84
} else if (token == WETH) {
  uint256 nativeBalance = address(this).balance;
  if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
  } else if (nativeBalance > 0) { ... }
```

When `nativeBalance >= value`, exactly `value` wei is deposited and forwarded; the remainder `nativeBalance - value` is left as native ETH on the router with no record of who sent it.

The three public helpers that drain the router's balance have no access control:

- `refundETH()` (L58-63): sends `address(this).balance` to `msg.sender` — whoever calls it, not the original payer.
- `unwrapWETH9()` (L37-45): withdraws all router WETH and sends ETH to a caller-supplied `recipient`.
- `sweepToken()` (L48-55): transfers all router token balance to a caller-supplied `recipient`.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH pushes; it does not prevent ETH from arriving via `msg.value` in payable swap functions, so the stranding path is fully reachable.

The intended usage pattern (documented in test comments and demonstrated in `test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) is to bundle `refundETH()` as the last step of a `multicall`. Within a `multicall`, `delegatecall` preserves `msg.sender`, so `refundETH()` correctly refunds the original caller. However, there is no on-chain enforcement preventing a user from calling `exactInputSingle{value: X}` directly with `X > amountIn`, which strands the excess and exposes it to any attacker.

**Exploit path:**
1. User calls `router.exactInputSingle{value: 2 ether}(ExactInputSingleParams{tokenIn: WETH, amountIn: 1000, ...})` directly (not via multicall).
2. Inside the swap callback, `pay(WETH, user, pool, 1000)` is called. `nativeBalance = 2 ether`, `value = 1000`. The branch deposits 1000 wei as WETH and sends it to the pool. `2 ether - 1000 wei` remains as native ETH on the router.
3. `exactInputSingle` returns. Router holds `2 ether - 1000 wei`.
4. Attacker calls `router.refundETH()`. `_transferETH(attacker, 2 ether - 1000 wei)` executes. Attacker receives the full stranded balance.

The same applies to `unwrapWETH9` and `sweepToken` for WETH or ERC-20 tokens stranded when a user sets `recipient: address(router)` in a swap but omits the unwrap/sweep step from a multicall.

## Impact Explanation

Direct loss of user ETH principal. Any user who calls a payable swap function directly with `msg.value > amountIn` (a natural mistake when using WETH as input) loses the excess to the first caller of `refundETH()`. The stranded amount can be arbitrarily large — up to the full `msg.value` minus the swap amount. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation

- **Fully unprivileged**: any EOA or contract can call `refundETH()`, `unwrapWETH9()`, or `sweepToken()` at any time with no setup, approvals, or privileged role.
- **Stranding is easy**: calling any payable swap function directly with `msg.value > amountIn` is a natural user error, especially when estimating gas or using WETH as input.
- **No front-running required**: the attacker simply monitors the mempool for transactions that leave ETH on the router and calls `refundETH()` in the next block, or front-runs the original transaction.
- **Repeatable**: the attack can be executed against every such transaction independently.

## Recommendation

1. **`refundETH()`**: Record the original `msg.sender` of the outermost multicall in transient storage and restrict `refundETH()` to only refund to that recorded address, or enforce that it can only be called within an active multicall context.
2. **`unwrapWETH9()` / `sweepToken()`**: Add a transient-storage guard ensuring they can only be called within an active multicall initiated by the same transaction, preventing cross-transaction balance theft.
3. **`pay()` excess ETH**: After settling the WETH payment when `nativeBalance > value`, immediately refund `nativeBalance - value` back to the payer rather than leaving it on the router.

## Proof of Concept

```solidity
// Foundry test sketch
function test_refundETH_theft() public {
    uint128 amountIn = 1_000;
    uint256 msgValue = 2 ether;

    // User calls exactInputSingle directly (not via multicall) with excess ETH
    vm.prank(user);
    router.exactInputSingle{value: msgValue}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: user,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // Router now holds 2 ether - 1000 wei
    assertEq(address(router).balance, msgValue - amountIn);

    // Attacker drains it
    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();
    assertEq(attacker.balance - attackerBefore, msgValue - amountIn);
    assertEq(address(router).balance, 0);
}
