### Title
Missing Parallel-List Length Validation in `BLSCache.py_aggregate_verify` Silently Truncates Signature Verification Pairs - (File: `crates/chia-bls/src/bls_cache.rs`)

### Summary

The Python binding `py_aggregate_verify` on `BlsCache` accepts two independent lists — `pks` (public keys) and `msgs` (messages) — and zips them together without first asserting that their lengths are equal. Rust's `zip` silently truncates to the shorter iterator, so if the two lists have different lengths, the function verifies fewer `(pk, msg)` pairs than the caller intended and returns a result based on that reduced set. An aggregate signature that is valid only for the shorter subset will pass verification even though the caller supplied additional keys or messages that were never checked.

### Finding Description

In `crates/chia-bls/src/bls_cache.rs`, the `py_aggregate_verify` Python binding collects `pks` and `msgs` into two independent `Vec`s and then zips them:

```rust
Ok(self.aggregate_verify(pks.into_iter().zip(msgs), sig))
```

There is no guard of the form `if pks.len() != msgs.len() { return Err(...) }` before the zip. The same pattern appears in `py_evict`:

```rust
self.evict(pks.into_iter().zip(msgs));
```

The Rust-internal `BlsCache::aggregate_verify` takes a single `IntoIterator<Item = (Pk, Msg)>`, so the length-mismatch problem is confined to the Python-boundary wrappers that split the pair into two separate lists.

Two silent failure modes exist:

| Condition | Effect |
|---|---|
| `pks.len() > msgs.len()` | Extra public keys are silently dropped; signature verified against only `msgs.len()` pairs |
| `pks.len() < msgs.len()` | Extra messages are silently dropped; signature verified against only `pks.len()` pairs |

In both cases `aggregate_verify_gt` is called with fewer GT elements than the caller intended, and a signature that covers only the truncated subset will return `true`.

### Impact Explanation

`BLSCache.aggregate_verify` is the Python-exposed path used by `chia-blockchain`'s mempool manager to verify aggregate BLS signatures over spend-bundle conditions. If the Python caller constructs `pks` and `msgs` with unequal lengths — whether due to a bug in list-building logic or an attacker-influenced code path — the binding silently verifies a strict subset of the intended `(pk, msg)` pairs. A crafted aggregate signature that is valid only for that subset will pass, constituting an aggregate-signature validation bypass that could allow unauthorized spend acceptance.

The same truncation in `py_evict` can leave stale GT pairings in the cache, causing future calls to `aggregate_verify` to return cached `true` results for key-message pairs that should have been invalidated.

### Likelihood Explanation

The Python binding is a public API surface. Any Python code that builds `pks` and `msgs` through separate pipelines (filtering, deduplication, list comprehensions) can inadvertently produce lists of different lengths. Because the binding silently succeeds rather than raising an error, the mismatch is invisible to callers and will not surface in normal testing unless the exact length-mismatch case is explicitly exercised. The `py_evict` path is lower-severity but equally silent.

### Recommendation

1. Add an explicit length check at the top of both `py_aggregate_verify` and `py_evict`:
   ```rust
   if pks.len() != msgs.len() {
       return Err(PyValueError::new_err(
           "pks and msgs must have the same length",
       ));
   }
   ```
2. Consider changing the Python API to accept a single list of `(G1Element, bytes)` tuples, eliminating the possibility of a length mismatch at the call site.

### Proof of Concept

```python
from chia_rs import BLSCache, AugSchemeMPL

sk1 = AugSchemeMPL.key_gen(b'\x01' * 32)
sk2 = AugSchemeMPL.key_gen(b'\x02' * 32)
pk1, pk2 = sk1.get_g1(), sk2.get_g1()
msg1, msg2 = b"hello", b"world"

# Aggregate signature covers ONLY (pk1, msg1)
sig = AugSchemeMPL.sign(sk1, msg1)

cache = BLSCache()
# Caller intends to verify both (pk1,msg1) and (pk2,msg2),
# but passes one extra pk with no matching msg.
# zip truncates to len(msgs)=1, so only (pk1,msg1) is checked.
result = cache.aggregate_verify([pk1, pk2], [msg1], sig)
# result == True  — pk2/msg2 pair was never verified
assert result, "signature bypass: pk2 was silently ignored"
```
