### Title
Missing Registered-Token Check in `initTransfer` Permanently Locks User Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary
`OmniBridge.initTransfer` and `OmniBridge.initTransfer1155` accept any arbitrary ERC-20/ERC-1155 `tokenAddress` without verifying that the token is registered in the bridge (i.e., that `ethToNearToken[tokenAddress]` is non-empty). When a user calls `initTransfer` with an unregistered token, the EVM contract locks the tokens in the bridge, emits `InitTransfer`, and the NEAR-side `fin_transfer_callback` unconditionally panics with `TokenDecimalsNotFound` because no decimal mapping exists for the unknown token. There is no admin rescue path for locked native ERC-20 tokens, so the funds are permanently irrecoverable. The same pattern exists in `starknet/src/omni_bridge.cairo`.

---

### Finding Description

`initTransfer` in `OmniBridge.sol` performs three token-handling branches:

1. `customMinters[tokenAddress] != address(0)` → burn via custom minter
2. `isBridgeToken[tokenAddress]` → burn bridge token
3. **else** → `safeTransferFrom(msg.sender, address(this), amount)` — lock in bridge [1](#0-0) 

There is no guard before these branches that checks `bytes(ethToNearToken[tokenAddress]).length != 0`. Any ERC-20 that the caller has approved falls into branch 3 and is locked in the bridge contract.

The `ethToNearToken` mapping is only populated by `deployToken` (MPC-signed) or `addCustomToken` (admin-only). [2](#0-1) 

On the NEAR side, `fin_transfer_callback` immediately calls:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [3](#0-2) 

For any token not registered via `deploy_token` / `bind_token`, `token_decimals` has no entry for the EVM address, so this call panics unconditionally. The NEAR transaction reverts, no MPC signature is ever produced, and the EVM-locked tokens can never be released through `finTransfer` (which requires a valid MPC signature). [4](#0-3) 

The same missing check exists in Starknet's `init_transfer`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Tokens locked in the EVM bridge via the `else` branch of `initTransfer` can only be released by `finTransfer`, which requires a valid NEAR MPC signature over a `TransferMessagePayload`. Because the NEAR `fin_transfer_callback` panics before any state is written, no signature is ever requested from the MPC network. There is no emergency-withdraw or rescue function in `OmniBridge.sol` for arbitrary locked ERC-20 balances. The funds are permanently frozen.

---

### Likelihood Explanation

**High.** The function is fully permissionless and callable by any address. A user who has previously approved the bridge for a token that was later de-registered (e.g., `removeCustomToken` was called), or who simply passes the wrong token address, will silently lock their funds. The EVM contract provides no revert or warning for unregistered tokens. [6](#0-5) 

---

### Recommendation

Add a registration check at the top of `initTransfer` (and `initTransfer1155`) before any token movement:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    uint128 fee,
    uint128 nativeFee,
    string calldata recipient,
    string calldata message
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
+   require(
+       tokenAddress == address(0) ||
+       bytes(ethToNearToken[tokenAddress]).length != 0,
+       "ERR_TOKEN_NOT_REGISTERED"
+   );
    currentOriginNonce += 1;
    ...
}
```

Apply the same guard to `initTransfer1155` and to Starknet's `init_transfer` (check `starknet_to_near_token[token_address]` is non-zero).

---

### Proof of Concept

**Setup:** Deploy any ERC-20 token `FakeToken` that is not registered in `OmniBridge` (i.e., `ethToNearToken[fakeToken] == ""`).

**Steps:**

1. User calls `FakeToken.approve(omniBridgeAddress, 1000)`.
2. User calls `OmniBridge.initTransfer(fakeToken, 1000, 0, 0, "victim.near", "")` with `msg.value = 0`.
3. Branch 3 executes: `IERC20(fakeToken).safeTransferFrom(user, address(this), 1000)` — tokens are now locked in the bridge.
4. `InitTransfer` event is emitted with `tokenAddress = fakeToken`.
5. A relayer observes the event and calls NEAR `fin_transfer` with the EVM proof.
6. `fin_transfer_callback` executes: `self.token_decimals.get(&OmniAddress::Eth(fakeToken)).near_expect(BridgeError::TokenDecimalsNotFound)` — **panics**.
7. The NEAR transaction reverts. No MPC signature is ever produced. No `finTransfer` can ever be called on EVM.
8. The 1000 `FakeToken` units are permanently locked in `OmniBridge` with no recovery path. [7](#0-6) [3](#0-2)
