### Title
Missing Zero-Value Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation accepts any `class_hash` value — including `0` — without validating that the new hash is non-zero or corresponds to a declared class. A contract deployer can call `replace_class(0)` to permanently set a contract's class hash to an invalid value, making the contract permanently non-executable and freezing all funds it holds. The codebase itself acknowledges this missing check via an explicit TODO comment.

---

### Finding Description

In `execute_replace_class` at lines 878–916 of `syscall_impls.cairo`, the function reads the caller-supplied `class_hash` directly from the syscall request and writes it into the contract state with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO at line 898 explicitly acknowledges the missing check: *"Check that there is a declared contract class with the given hash."* [2](#0-1) 

There is no `assert_not_zero(class_hash)` guard, no lookup into `contract_class_changes` to confirm the hash is declared, and no range check preventing `class_hash = 0`. The state update is unconditional.

This is structurally identical to the H-01 pattern: a critical state-mutating operation (`_sendTokenToBribesProcessor` / `dict_update` on class hash) executes without checking whether the key parameter is zero.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to `0` (or any undeclared hash), every subsequent call to that contract will fail at the class-lookup stage. The OS resolves the class to execute by reading `state_entry.class_hash` and dispatching to the corresponding compiled class. Class hash `0` has no compiled class registered; execution cannot proceed. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — there is no code path that can transfer them out.

---

### Likelihood Explanation

A contract deployer is an explicitly listed unprivileged attacker in the scope. The attack requires only:

1. Deploying a contract (any class).
2. Attracting user deposits (e.g., presenting as a vault or bridge).
3. Issuing a single `replace_class(0)` syscall from within the contract.

The OS enforces no precondition on the new class hash. The attack is deterministic, requires no special privilege beyond contract deployment, and leaves no recovery path. The TODO comment confirms the developers are aware the check is absent.

---

### Recommendation

In `execute_replace_class`, before writing the new state entry, add:

1. **Non-zero check**: `assert_not_zero(class_hash)` — mirrors the fix recommended in H-01 (`assert bribesProcessor != 0`).
2. **Declared-class check**: verify that `class_hash` exists as a key in `contract_class_changes` (i.e., it was declared in this or a prior block), consistent with the existing TODO.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` with a valid class hash `C`.
2. Users deposit tokens into `MaliciousVault`; balances accumulate in its storage.
3. Attacker calls a function on `MaliciousVault` that internally issues the `replace_class` syscall with `class_hash = 0`.
4. `execute_replace_class` in `syscall_impls.cairo` (lines 878–916) reads `class_hash = 0` from the request, skips all validation (none exists), and calls `dict_update` setting `MaliciousVault`'s `class_hash` to `0` in `contract_state_changes`.
5. State is committed. `MaliciousVault` now has `class_hash = 0` on-chain.
6. Any subsequent call to `MaliciousVault` (e.g., `withdraw`) causes the OS to look up compiled class `0`, find nothing, and fail.
7. All deposited funds are permanently frozen with no recovery mechanism. [3](#0-2)
