### Title
Missing Duplicate-Key and Duplicate-Hash Validation in `batch_insert` Corrupts DataLayer Merkle Tree State - (File: crates/chia-datalayer/src/merkle/blob.rs)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. Supplying a batch that contains a repeated `KeyId` or a repeated leaf `Hash` silently writes multiple leaves with the same key or hash into the blob, producing an inconsistent `block_status_cache` and an incorrect Merkle root. Because the root is wrong, any subsequent inclusion or exclusion proof generated from the tree is invalid, and a verifier that trusts the root will accept forged state.

---

### Finding Description

`MerkleBlob::insert` opens with two mandatory guards before touching the blob:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`MerkleBlob::batch_insert` takes a different code path for all items beyond the first two. When `leaf_count > 1`, it iterates over the remaining entries and calls `insert_entry_to_blob` directly, with **no** duplicate-key or duplicate-hash check:

```rust
// blob.rs:587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Only the first two items (handled when `leaf_count <= 1`) go through the validated `insert` path: [3](#0-2) 

Consequences of inserting a duplicate key:

1. `insert_entry_to_blob` writes a new leaf block at a fresh index. The `block_status_cache` maps `key → index`; the second write overwrites the cache entry, so the first leaf is now orphaned in the blob but invisible to the cache.
2. The orphaned leaf is still counted in the blob's raw block array, so `check_integrity` will detect a mismatch between `leaf_count` and `key_to_index_cache_length`.
3. Internal-node hashes are computed from the subtree built over the batch (lines 607–648), so the final root hash is computed over a tree that contains the duplicate leaf, producing a root that does not correspond to the intended key-value set. [4](#0-3) 

The `batch_insert` entry point is exposed to Python callers: [5](#0-4) 

and is exercised directly from Python tests: [6](#0-5) 

---

### Impact Explanation

A corrupted Merkle root means every proof of inclusion or exclusion generated from the tree is computed against a wrong root. A verifier that trusts the stored root will accept proofs for keys that were never legitimately inserted, or reject proofs for keys that were. This maps directly to the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

---

### Likelihood Explanation

`batch_insert` is the performance-optimised bulk-load path and is called whenever a DataLayer store processes a set of updates in one shot. Any caller — including a Python-layer DataLayer client — that supplies a list containing a repeated key or hash triggers the bug without any special privilege. The regular `insert` path correctly rejects duplicates, so the gap is purely in the batch path.

---

### Recommendation

Add the same pre-flight checks to `batch_insert` that `insert` already enforces. Before the main loop at line 587, validate the entire incoming batch:

```rust
for ((key, _value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Additionally, check for intra-batch duplicates (two entries in the same batch sharing a key or hash) using a local `HashSet` over the batch before writing anything to the blob.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Seed with 2 entries so leaf_count > 1 and the fast path is taken
for i in 0i64..2 {
    blob.insert(KeyId(i), ValueId(i),
        &Hash(Bytes32::new([i as u8; 32])),
        chia_datalayer::InsertLocation::Auto {}).unwrap();
}

// batch_insert with a duplicate key — no error is returned
let dup_key = KeyId(0);
blob.batch_insert(vec![
    ((dup_key, ValueId(99)), Hash(Bytes32::new([0xAA; 32]))),
    ((KeyId(100), ValueId(100)), Hash(Bytes32::new([0xBB; 32]))),
]).unwrap(); // succeeds — should have returned KeyAlreadyPresent

// The tree is now corrupt: check_integrity will fail
blob.check_integrity().unwrap_err(); // panics / errors — inconsistent state
```

The duplicate `KeyId(0)` bypasses all guards in the bulk loop, writes a second leaf for the same key, corrupts the `block_status_cache`, and produces an incorrect root hash for any proof generated thereafter.
