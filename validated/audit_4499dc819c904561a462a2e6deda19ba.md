### Title
3-arg `mint` in `HlBridgeToken` Parks Recipient Tokens at `_systemAddress`, Permanently Freezing Funds on `finTransfer` with Non-Empty Message — (File: `evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`OmniBridge.finTransfer` dispatches to the 3-argument `IBridgeToken.mint(account, value, message)` overload whenever `payload.message.length > 0`. For `HlBridgeToken`, that overload mints tokens to `account` and then immediately transfers them away to `_systemAddress` via `_update`. The intended recipient ends up with zero tokens; the minted value is permanently stranded at the HyperLiquid system address, which is not user-controlled and can only be moved by the HL system itself.

---

### Finding Description

`OmniBridge.finTransfer` contains the following dispatch logic for bridge tokens:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount
        );
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount,
            payload.message   // ← 3-arg path
        );
    }
}
``` [1](#0-0) 

The 3-arg `mint` in `HlBridgeToken` is:

```solidity
function mint(
    address account,
    uint256 value,
    bytes memory
) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);  // ← immediately moves tokens away
}
``` [2](#0-1) 

The `_update(account, _systemAddress, value)` call is an ERC-20 internal transfer that moves the freshly minted tokens from `account` (the intended recipient) to `_systemAddress`. This is the HyperCore accounting model: the 3-arg overload is designed for HyperCore-side minting where the token supply on HyperEVM must be mirrored at `_systemAddress`. When called from `finTransfer` for a NEAR → HyperEVM delivery, the semantics are wrong: the recipient is an EVM address expecting to hold tokens, not a HyperCore participant.

After the call completes:
- `payload.recipient` balance: **0**
- `_systemAddress` balance: **+amount**

`_systemAddress` is the HyperLiquid system address. Tokens parked there can only be moved by the HL system via `coreReceiveWithData`, which is callable exclusively by `_systemAddress` itself. [3](#0-2) 

There is no user-accessible function to recover tokens from `_systemAddress`. The funds are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Any NEAR → HyperEVM `finTransfer` for a `HlBridgeToken`-backed asset where `payload.message` is non-empty results in the recipient receiving zero tokens. The minted amount is irrecoverably stranded at `_systemAddress`. The nonce is marked used (`completedTransfers[payload.destinationNonce] = true`) before the mint, so the transfer cannot be replayed or reversed through the bridge. [4](#0-3) 

---

### Likelihood Explanation

**High.** The `message` field is a first-class parameter in the bridge protocol, used for cross-chain contract calls and DeFi interactions. Any user who initiates a transfer from NEAR to HyperEVM with a non-empty `msg` field (e.g., to trigger a swap or contract call on arrival) will trigger this path. The `HlBridgeToken` is registered as a bridge token via `addCustomToken` or `deployToken`, so `isBridgeToken[tokenAddress]` is true and the dispatch reaches the 3-arg `mint`. No privileged access is required; any unprivileged bridge user can trigger this by including a message in their transfer. [5](#0-4) 

---

### Recommendation

The 3-arg `mint` in `HlBridgeToken` must not be used for EVM-side delivery. Options:

1. **Override `finTransfer` dispatch for `HlBridgeToken`**: Always call the 2-arg `mint` (which delivers tokens directly to the recipient) regardless of `message` length, and handle the message separately if needed.
2. **Override the 3-arg `mint` in `HlBridgeToken`** to behave identically to the 2-arg variant (i.e., do not call `_update` to `_systemAddress`), and introduce a separate function for HyperCore-side minting.
3. **Guard `finTransfer`**: Detect `HlBridgeToken` (e.g., via interface check) and always use the 2-arg path.

---

### Proof of Concept

1. A NEAR user initiates `ft_transfer_call` to the NEAR bridge targeting a HyperEVM address with `msg = "{"action":"swap","pool":"0x..."}"` (non-empty).
2. The NEAR bridge signs a `TransferMessagePayload` with `message = <encoded bytes>`.
3. A relayer calls `OmniBridge.finTransfer(signature, payload)` on HyperEVM.
4. `completedTransfers[payload.destinationNonce]` is set to `true` — transfer is now permanently consumed.
5. `isBridgeToken[payload.tokenAddress]` is `true` (HlBridgeToken is registered).
6. `payload.message.length > 0` → 3-arg `mint(payload.recipient, payload.amount, payload.message)` is called.
7. Inside `HlBridgeToken.mint`: `_mint(recipient, amount)` then `_update(recipient, _systemAddress, amount)`.
8. `recipient.balanceOf` = 0. `_systemAddress.balanceOf` += amount.
9. No bridge function allows the recipient to recover the tokens from `_systemAddress`.
10. Funds are permanently frozen. [2](#0-1) [6](#0-5)
