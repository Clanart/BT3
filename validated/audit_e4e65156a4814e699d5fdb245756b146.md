### Title
Unvalidated Peer-Supplied `transaction_hash` Stored and Served as Authoritative — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `transaction_hash` from a peer-supplied `FullTransaction` and pushes it directly into `block_body.transaction_hashes` without calling `validate_transaction_hash`. The body is then committed to storage verbatim. The stored attacker-chosen hash is subsequently served by the RPC layer and re-propagated to downstream p2p peers as authoritative.

### Finding Description

In `parse_data_for_block`, the peer-controlled `transaction_hash` field of each `FullTransaction` is appended to `block_body.transaction_hashes` with no validation:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

The `write_to_storage` implementation then calls `append_body`, which calls `write_transactions`. Inside `write_transactions`, the peer-supplied hash is stored in two places without recomputation:

1. `transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)` — the reverse-lookup index is keyed on the attacker-chosen felt.
2. `TransactionMetadata { tx_location, tx_output_location, tx_hash: *tx_hash }` — the per-transaction metadata record stores the attacker-chosen felt as the canonical hash. [2](#0-1) 

`validate_transaction_hash` exists in `crates/starknet_api/src/transaction_hash.rs` and can detect this mismatch, but it is never called on this path. [3](#0-2) 

The p2p server's `FetchBlockData for FullTransaction` reads stored hashes back from `get_block_transaction_hashes` and re-serves them to downstream peers, propagating the corruption. [4](#0-3) 

The RPC layer also reads stored hashes via `get_block_transaction_hashes` and serves them to clients. [5](#0-4) 

### Impact Explanation

**Correction to the question's commitment claim**: The `transaction_commitment` Merkle root is computed by the batcher/sequencer from execution results (`prepare_txs_hashing_data` → `calculate_block_commitments`) and stored in the block header *before* the body is synced. The syncing node does not recompute `transaction_commitment` from stored body hashes; the header's commitment field is authoritative and is not affected by the corrupted body hash. [6](#0-5) [7](#0-6) 

The actual, concrete impacts are:

- **RPC serves wrong transaction hash**: Any RPC call that returns transaction hashes for block N will return the attacker-chosen felt instead of `hash(Transaction, chain_id)`. This is an authoritative-looking wrong value.
- **Downstream p2p peers receive corrupted data**: The p2p server re-serves stored hashes, so any peer syncing from this node receives and stores the same corrupted hash, propagating the corruption transitively.
- **Reverse-lookup table is poisoned**: `transaction_hash_to_idx_table` is keyed on the attacker-chosen hash. Lookups by the correct protocol hash return nothing; lookups by the attacker hash return the wrong index.

### Likelihood Explanation

Any p2p peer that is the sole sync source for a block whose body has not yet been stored can trigger this. No special privileges are required. The precondition (peer is only available source, block body not yet stored) is the normal operating condition for a syncing full node.

### Recommendation

Before pushing `transaction_hash` into `block_body.transaction_hashes`, call `validate_transaction_hash` (from `starknet_api::transaction_hash`) against the stored block's `chain_id` and the received `transaction`. If validation fails, return `ParseDataError::BadPeer` and disconnect the peer. The existing TODO comment at line 88 already identifies this gap. [1](#0-0) 

### Proof of Concept

The existing `transaction_basic_flow` test in `transaction_test.rs` already demonstrates that whatever `transaction_hash` is supplied by the peer is stored and read back verbatim via `get_block_transaction_hashes`. [8](#0-7) 

A concrete exploit test would:
1. Construct a `FullTransaction` with a valid `InvokeV1` transaction body but `transaction_hash = felt!(0xdeadbeef)`.
2. Feed it through `parse_data_for_block` with a `storage_reader` whose header for block N has `n_transactions = 1`.
3. Call `write_to_storage`.
4. Assert `get_block_transaction_hashes(N)` returns `[felt!(0xdeadbeef)]`.
5. Assert `validate_transaction_hash(&transaction, &block_number, &chain_id, felt!(0xdeadbeef), &TransactionOptions::default())` returns `Ok(false)`.

All five assertions would pass, confirming the stored hash diverges from the protocol-computed hash with no guard preventing it.

pub fn validate_transaction_hash(

pub async fn calculate_block_commitments(

    starknet_version: &StarknetVersion,

        .iter()
