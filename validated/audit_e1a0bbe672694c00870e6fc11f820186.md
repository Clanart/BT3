### Title
Missing shard-ID binding in `set_state_header` allows cross-shard state-root substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from an untrusted peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the embedded chunk's `shard_id()` equals the requested `shard_id` parameter. A malicious peer can supply a header whose chunk belongs to shard B while the caller requests shard A, pass every existing validation gate, and corrupt `DBCol::StateHeaders` so that all subsequent `set_state_part` / `apply_state_part` calls install shard B's trie into shard A's `ShardUId`.

---

### Finding Description

The full validation sequence in `set_state_header` is:

**Step 1-2 — `validate_chunk_proofs`** [1](#0-0) 

This checks the chunk's internal hash, tx-root, and receipts-root. It validates that the chunk body is self-consistent. It does **not** compare `chunk.shard_id()` against the `shard_id` parameter.

**Step 3a — `verify_path` against `chunk_headers_root`** [2](#0-1) 

`verify_path` is a pure Merkle-path verifier: [3](#0-2) 

It computes `compute_root_from_path(path, hash(item)) == root`. The Left/Right directions in the path encode the leaf's *position* in the tree, so a proof for chunk B at shard-index 1 is only valid when combined with chunk B's hash — but it is valid for that combination regardless of what `shard_id` the caller requested. An attacker who supplies chunk B together with chunk B's genuine Merkle proof will pass this check even though the caller asked for shard A.

**Step 3b — `prev_chunk_header` proof** [4](#0-3) 

Verifies that `prev_chunk_header` is in the block at `chunk.height_included() - 1`. The attacker supplies shard B's genuine `prev_chunk_header`, which passes.

**Step 4 — incoming receipts** [5](#0-4) 

Receipt hashes are computed as `ReceiptList(shard_id, receipts)` — using the *requested* `shard_id` (shard A). The attacker provides shard A's actual on-chain incoming receipts, which are publicly available and pass all Merkle checks. The terminal height check (step 4g, line 507) compares `header.height()` against `prev_chunk_header.height_included()`. In a healthy network where every shard produces a chunk at every height, shard B's `prev_chunk_header.height_included()` equals shard A's, so this check passes.

**Step 5 — `validate_state_root_node`** [6](#0-5) 

Validates `hash(state_root_node.data) == chunk_inner.prev_state_root()`: [7](#0-6) 

`chunk_inner` is chunk B's inner, so `prev_state_root()` is shard B's state root. The attacker supplies shard B's genuine `state_root_node`, which passes.

**Storage — no shard-ID binding** [8](#0-7) 

The header (containing chunk B) is stored under `StateHeaderKey(shard_A, sync_hash)`. The key uses the caller's `shard_id`; the value contains chunk B's `prev_state_root`.

---

### Impact Explanation

`set_state_part` reads the stored header to obtain `state_root`: [9](#0-8) 

It then validates incoming parts against shard B's state root. The attacker supplies genuine shard B state parts, which pass `validate_state_part`.

`apply_state_part` then installs those parts: [10](#0-9) 

`shard_uid` is derived from the requested `shard_id` (shard A) and `epoch_id`. `trie_changes` and `flat_state_delta` come from shard B's trie. Both `tries.apply_all` and `flat_state_delta.apply_to_flat_state` write under shard A's `shard_uid`. The result is that shard B's complete trie is installed as shard A's canonical state, with shard A's `ShardUId` as the storage key.

---

### Likelihood Explanation

Any peer that can respond to `StateRequest` network messages can execute this attack. No validator, block-producer, or operator privilege is required. The attacker only needs:
- Knowledge of the block structure (public)
- Shard B's chunk and its genuine Merkle proof (derivable from any full node)
- Shard A's incoming receipts (public on-chain data)
- Shard B's `state_root_node` (obtainable from any archival node)

The attack is most straightforward when both shards produce chunks at every height (the common case), making the height-check in step 4g trivially satisfiable.

---

### Recommendation

Add an explicit shard-ID binding check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be inserted after line 376 (`let chunk = shard_state_header.cloned_chunk();`) and before `validate_chunk_proofs`. [11](#0-10) 

Similarly, `prev_chunk_header.shard_id()` should be verified to equal `shard_id`.

---

### Proof of Concept

In a two-shard test-loop environment:

1. Node A begins state sync for `shard_id = 0`, `sync_hash = H`.
2. Attacker intercepts the `StateResponse` network message.
3. Attacker replaces the `header.chunk` field with shard 1's chunk (which is genuinely included in the same block), and replaces `chunk_proof` with shard 1's valid Merkle proof against `sync_prev_block_header.chunk_headers_root()`.
4. Attacker keeps `incoming_receipts_proofs` as shard 0's actual receipts (valid for `shard_id=0`).
5. Attacker keeps `state_root_node` as shard 1's genuine root node.
6. Call `set_state_header(shard_id=0, sync_hash=H, tampered_header)`.
7. Assert: `stored_header.chunk_prev_state_root()` equals shard 1's `prev_state_root`, not shard 0's canonical `prev_state_root`.
8. Supply shard 1's genuine state parts; call `set_state_part` and `apply_state_part` for `shard_id=0`.
9. Assert: shard 0's trie storage now contains shard 1's key-value pairs.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)

    fn apply_state_part(
        &self,
        shard_id: ShardId,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,
    ) -> Result<(), Error> {
        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);
        let tries = self.get_tries();
        let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;
        let mut store_update = tries.store_update();
        tries.apply_all(&trie_changes, shard_uid, &mut store_update);
        tracing::debug!(target: "chain", %shard_id, values_count = %flat_state_delta.len(), "inserting values to flat storage");
        // TODO: `apply_to_flat_state` inserts values with random writes, which can be time consuming.
        //       Optimize taking into account that flat state values always correspond to a consecutive range of keys.
        flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
        self.precompile_contracts(epoch_id, contract_codes)?;
        store_update.commit();
        Ok(())
    }

                StateRootNodeValidationResult::Invalid

                    StateRootNodeValidationResult::Valid

                    StateRootNodeValidationResult::Invalid
                }
