### Title
Missing Chain ID in `MetadataPayload` Borsh Encoding Enables Cross-Chain Replay of `deployToken` MPC Signature — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`)

---

### Summary

The `MetadataPayload` Borsh encoding used to verify MPC signatures in `deployToken` omits the destination chain ID, while the `TransferMessagePayload` encoding used in `finTransfer` explicitly includes it twice. Because the same NEAR-derived ECDSA key (`nearBridgeDerivedAddress` / `omni_bridge_derived_address`) is used across all EVM deployments and Starknet, a valid `deployToken` signature observed on one chain can be replayed verbatim on any other chain that shares the same verifying key.

---

### Finding Description

**Vulnerability class**: Signature/domain-separation bypass — a protection (chain ID binding) is applied to one signed operation (`finTransfer`) but omitted from a related signed operation (`deployToken`), allowing the protection to be circumvented via the alternative path.

**EVM `deployToken` Borsh encoding** (`OmniBridge.sol` L142–149):

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
    // ← no omniBridgeChainId
);
``` [1](#0-0) 

**EVM `finTransfer` Borsh encoding** (`OmniBridge.sol` L289–308) — chain ID included twice:

```solidity
bytes1(omniBridgeChainId),   // destination chain
...
bytes1(omniBridgeChainId),   // recipient chain
``` [2](#0-1) 

**Starknet `MetadataPayload.to_borsh`** (`bridge_types.cairo` L36–44):

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    // ← no chain_id
}
``` [3](#0-2) 

**Starknet `TransferMessagePayload.to_borsh`** (`bridge_types.cairo` L61–84) — chain ID included twice:

```cairo
borsh_bytes.append_byte(chain_id);   // destination chain
...
borsh_bytes.append_byte(chain_id);   // recipient chain
``` [4](#0-3) 

Both EVM and Starknet verify the signature against the same NEAR-derived ECDSA address using `keccak256` of the identical Borsh byte sequence. Since the `MetadataPayload` encoding is identical across all chains (no chain discriminator), the resulting hash — and therefore the valid signature — is identical on every chain.

The only replay guard in `deployToken` is the "token already deployed" check:

- EVM: `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` [5](#0-4) 
- Starknet: `assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED')` [6](#0-5) 

This guard prevents same-chain replay but does nothing to prevent cross-chain replay, exactly mirroring the VotingEscrow pattern where the protection flag is set on one path but not enforced on the alternative path.

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (e.g., EVM Ethereum) can extract the raw signature and `MetadataPayload` and submit them to:

1. Any other EVM deployment of OmniBridge (Arbitrum, Base, etc.) sharing the same `nearBridgeDerivedAddress`.
2. The Starknet OmniBridge sharing the same `omni_bridge_derived_address`.

This deploys the bridged token on chains where NEAR never authorized deployment. If the NEAR side subsequently processes the `DeployToken` response event from the unauthorized chain (via its registered prover), it would register the token address for that chain, enabling `finTransfer` operations to that chain — potentially minting bridged tokens on a chain the protocol did not intend to support for that token. This constitutes **unauthorized token deployment** and a **signature/domain-separation bypass** enabling unauthorized message execution.

Allowed impact match: *High — Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment.*

---

### Likelihood Explanation

- The `deployToken` calldata (including the raw MPC signature) is publicly visible on-chain the moment the transaction is mined.
- Any observer can extract it and replay it on another chain in the next block.
- No privileged access, leaked keys, or colluding parties are required.
- The attacker only needs to pay gas on the target chain.

Likelihood: **High** — trivially executable by any unprivileged observer.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` Borsh encoding, matching the pattern already used in `TransferMessagePayload`:

**EVM** (`OmniBridge.sol` `deployToken`):
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet** (`bridge_types.cairo` `MetadataPayload.to_borsh`):
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
+   borsh_bytes.append_byte(chain_id);
    ...
}
```

The NEAR side must include the target chain ID when constructing the payload before MPC signing, and the receiving chain must supply its own `omniBridgeChainId` when verifying.

---

### Proof of Concept

1. NEAR MPC signs `MetadataPayload{token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6}` for deployment on EVM Ethereum (chain ID `0x02`).
2. The signed transaction is submitted to Ethereum's `OmniBridge.deployToken`. The `DeployToken` event is emitted; `USDC` is now deployed on Ethereum.
3. Attacker copies the exact `signatureData` and `metadata` calldata from the Ethereum transaction.
4. Attacker calls `OmniBridge.deployToken(signatureData, metadata)` on Arbitrum's OmniBridge (chain ID `0x05`).
5. Arbitrum's `ECDSA.recover(keccak256(borshEncoded), signatureData)` returns the same `nearBridgeDerivedAddress` — the Borsh bytes are identical because no chain ID is included.
6. The `require(!isBridgeToken[nearToEthToken["usdc.near"]])` check passes because `usdc.near` has not been deployed on Arbitrum yet.
7. A new `BridgeToken` proxy for `usdc.near` is deployed on Arbitrum without NEAR's authorization for that chain.
8. The same replay works against Starknet's `deploy_token` using the identical signature, since both chains hash the same Borsh bytes with `keccak256` and verify against the same NEAR-derived ECDSA address.

            },

        match self.message {
            Option::None => {},
            Option::Some(message) => { borsh_bytes.append(@borsh::encode_byte_array(message)); },
        }
