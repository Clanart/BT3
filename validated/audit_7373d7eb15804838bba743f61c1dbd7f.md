### Title
`batch_insert` Bypasses Duplicate-Key Guard, Corrupting DataLayer Merkle Tree Root and Proof Validity — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash existence checks that `MerkleBlob::insert` enforces, for all items beyond the first two in the batch. An unprivileged caller supplying a batch with a repeated `KeyId` (or a `KeyId` already present in the tree) silently writes a second leaf node into the blob, desynchronises the `block_status_cache`, and produces a corrupted Merkle root. Any proof of inclusion derived from the corrupted tree is invalid, and the tree fails its own integrity check.

---

### Finding Description

`MerkleBlob::insert` guards against re-insertion at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `self.insert()` (with those guards) only for the last one or two items popped from the input vector when the tree has ≤ 1 existing leaf. Every remaining item is written directly via `insert_entry_to_blob` with no existence check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf(index, leaf)`, which inserts into the `key_to_index` and `leaf_hash_to_index` HashMaps. A HashMap insert silently overwrites the existing entry for a duplicate key, so:

1. The blob now contains **two** leaf nodes sharing the same `KeyId`.
2. The cache's `key_to_index` points only to the **second** (later) leaf; the first leaf becomes a ghost node unreachable through the cache.
3. The actual leaf count in the blob exceeds the cache's key count, violating the invariant checked by `check_integrity`.
4. The Merkle root is computed over both leaves (including the ghost), producing a root hash that does not correspond to any valid key-set. [3](#0-2) 

When the tree has ≥ 2 existing leaves the `if self.block_status_cache.leaf_count() <= 1` branch is skipped entirely, so **all** items in the batch bypass the guard: [4](#0-3) 

The Python binding `py_batch_insert` forwards caller-supplied lists directly to `batch_insert` with no additional filtering: [5](#0-4) 

---

### Impact Explanation

The corrupted Merkle root is the authoritative commitment to the DataLayer store's key-value state. A root derived from a tree containing a ghost leaf does not faithfully represent the stored data. Proofs of inclusion generated against this root are structurally invalid: `get_proof_of_inclusion` resolves the key through the cache to the second leaf's index, producing a proof path that does not include the ghost leaf, yet the root was computed over both. A verifier recomputing the root from the proof will obtain a different value, causing proof verification to fail for legitimately inserted keys, or — depending on how the root is consumed — allowing a caller to present a proof for a key-value pair that is not the canonical entry for that key.

This matches: **High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The entry path is direct and unprivileged. Any caller with access to `MerkleBlob::batch_insert` (including via the Python `MerkleBlob.batch_insert` binding) can trigger the bug by passing a list containing a repeated `KeyId`, or a `KeyId` already present in the tree, as long as the duplicate is not among the last two items in the list (which are the only ones routed through the guarded `insert` path). No special permissions, keys, or chain state are required.

---

### Recommendation

Add the same existence checks at the top of the unchecked loop in `batch_insert`, or refactor the fast path to call `self.insert()` for every item. At minimum, before calling `insert_entry_to_blob` for a leaf, verify:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [2](#0-1)

### Proof of Concept

```rust
// Tree starts empty (leaf_count == 0, so the guarded branch fires for the last 2 items).
// Input: [A, B, C] where A and C share the same KeyId.
// batch_insert pops C (guarded insert, succeeds), pops B (guarded insert, succeeds),
// then iterates over [A] — no duplicate check — inserting a second leaf for A's KeyId.

let mut blob = MerkleBlob::new(vec![]).unwrap();
let dup_key = KeyId(42);
let batch = vec![
    ((dup_key,   ValueId(1)), Hash(Bytes32::new([0x01; 32]))),  // A — goes through unchecked loop
    ((KeyId(99), ValueId(2)), Hash(Bytes32::new([0x02; 32]))),  // B — guarded insert
    ((dup_key,   ValueId(3)), Hash(Bytes32::new([0x03; 32]))),  // C — guarded insert (succeeds, no prior entry yet)
];
blob.batch_insert(batch).unwrap();
// blob now has two leaf nodes with KeyId(42); check_integrity() will fail.
blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheIndex or IntegrityKeyToIndexCacheLength
``` [6](#0-5)

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }

    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
