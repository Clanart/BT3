### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the new class hash supplied by a contract is actually declared in the class tree before committing the state update. Any contract can set its own class hash to an undeclared value, making itself permanently inaccessible and irreversibly freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the new class hash from the syscall request and unconditionally writes it into the contract's `StateEntry` with no check that the hash corresponds to a declared class.

The code itself acknowledges the missing check with an explicit TODO: [1](#0-0) 

The update then proceeds without any guard: [2](#0-1) 

The revert-log entry records only the *old* class hash for potential intra-transaction revert: [3](#0-2) 

Once the transaction is finalized, the change is permanent. The `StateEntry` struct that drives all subsequent execution looks up the class hash to find the code to run: [4](#0-3) 

If `class_hash` points to an undeclared class, every future call to the contract fails at the entry-point dispatch stage because no compiled class exists for that hash. The contract is permanently bricked.

**Contrast with `deploy_contract`**, which *does* enforce that the target address is uninitialized before writing a class hash: [5](#0-4) 

The same discipline — asserting a valid class hash before committing — is absent in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After `replace_class` is called with an undeclared hash:

1. The contract's `StateEntry.class_hash` is committed to the undeclared value.
2. Every subsequent call to the contract fails: the OS cannot locate entry points for an undeclared class.
3. The contract cannot self-heal by calling `replace_class` again — that too requires executing code.
4. All ERC-20 balances (STRK, ETH) held by the contract are permanently inaccessible.

---

### Likelihood Explanation

The `replace_class` syscall is reachable by any unprivileged contract deployer. Concrete paths:

- **Direct self-attack**: An attacker deploys a contract, funds it, then calls `replace_class` with `0xdeadbeef`. Funds are frozen. While this harms only the attacker's own contract, it demonstrates the OS accepts the invalid state without error.
- **Governance / upgrade attack**: Many production contracts (multisigs, vaults, protocol treasuries) expose an upgrade path that calls `replace_class` with a hash supplied by governance or an admin. An attacker who can influence that input (e.g., via a governance proposal, a compromised admin key, or a reentrancy vector) can supply an undeclared hash and permanently freeze the contract's funds.
- **Accidental misuse**: A contract developer who passes a Sierra class hash instead of a compiled class hash, or makes an off-by-one error in hash computation, silently bricks their contract with no OS-level rejection.

---

### Recommendation

Before committing the `dict_update` in `execute_replace_class`, verify that `class_hash` is present in the declared class set. Concretely:

1. Perform a `dict_read` on `contract_class_changes` for the new `class_hash`; if the result is `UNINITIALIZED_CLASS_HASH` (0), the class was not declared in this block.
2. For classes declared in prior blocks, the hint layer must supply a Merkle proof of inclusion in the class Patricia tree, and the OS must verify it.
3. Reject (write a failure response) if neither check passes, consistent with how other invalid syscall arguments are handled.

---

### Proof of Concept

```
1. Declare class C (compiled_class_hash = H_C).
2. Deploy contract A using class C; fund A with 1000 STRK.
3. From A, call replace_class(class_hash=0xdeadbeef).
   - execute_replace_class reads class_hash = 0xdeadbeef.
   - No validation against contract_class_changes or the Patricia tree.
   - dict_update writes StateEntry{class_hash=0xdeadbeef, ...} for A.
   - Transaction succeeds; revert log records old hash C.
4. Attempt any call to A (e.g., transfer funds out).
   - OS looks up class 0xdeadbeef → not found → execution fails.
5. 1000 STRK are permanently frozen in A.
```

The root cause — the OS committing contract state with an unvalidated class hash — is the direct StarkNet analog of the Nibiru finding, where the OS committed EVM bytecode without setting a `codeHash` because the account-type check was absent.
