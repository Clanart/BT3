The vulnerability class is: **missing account/state constraint causing deposited/committed funds to become locked or inaccessible**. I need to find an analog in Ethermint where tokens committed to EVM state or bank state become inaccessible due to a missing constraint or validation.

### Title
Non-EVM-Denom Cosmos Bank Tokens Permanently Locked on Contract Self-Destruct — (File: x/evm/statedb/statedb.go)

---

### Summary

When a contract self-destructs, `StateDB.Commit()` only burns the EVM-denom balance before deleting the auth account. Non-EVM-denom tokens (IBC assets, CW-bridge tokens, etc.) held at the contract's Cosmos bank address are never drained. After `DeleteAccount` removes the auth account, those bank balances become permanently orphaned: no auth account exists to authorize outbound transfers, and no EVM account exists to receive calls.

---

### Finding Description

In `x/evm/statedb/statedb.go`, the `Commit()` function handles self-destructed accounts as follows: [1](#0-0) 

The code explicitly acknowledges the gap in a comment at line 813–815:

> "Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm bridge) held by the destroyed address are not drained and may remain as orphaned bank balances."

The drain logic reads only the EVM denom balance: [2](#0-1) 

Then `DeleteAccount` removes the auth account and all EVM storage: [3](#0-2) 

After `RemoveAccount` executes, the Cosmos address has no auth account. The Cosmos SDK bank module requires a valid auth account to authorize `SendCoins` or `SendCoinsFromAccountToModule`. Any non-EVM-denom bank balance at that address is therefore permanently inaccessible — it cannot be transferred, burned, or recovered through any standard path.

The missing constraint is the analog to the original report: just as the deposit instruction lacked a constraint enforcing an associated token account (making deposited tokens inaccessible to subsequent swap operations), `Commit()` lacks a constraint to drain all bank denominations before account deletion, making non-EVM-denom tokens permanently inaccessible after self-destruct.

### Impact Explanation

Any non-EVM-denom Cosmos bank tokens held by a self-destructed contract are permanently locked. This constitutes a permanent loss of Cosmos bank funds through Ethermint's stateDB/native action logic — matching the Critical/High allowed impact: *"EVM state transition … bug that permits … valid user funds/fees to be mis-accounted"* and *"Cosmos bank funds … through … stateDB/native action logic."*

Concrete scenarios:
- A DeFi contract accumulates IBC tokens (e.g., `ibc/...` denom) via a bank precompile or native action, then self-destructs. All IBC holdings are permanently orphaned.
- A user sends IBC tokens to a contract address via a permissionless Cosmos `MsgSend` before the contract self-destructs in the same block. Those tokens are lost.
- A malicious contract deployer intentionally self-destructs a contract holding user-deposited IBC assets, permanently locking them.

---

### Likelihood Explanation

**Medium.** The preconditions are:
1. A contract must hold non-EVM-denom tokens. This is reachable via: (a) bank precompiles that allow contracts to receive IBC tokens, (b) `ExecuteNativeAction` calls that transfer non-EVM-denom tokens to a contract, or (c) a permissionless Cosmos `MsgSend` to the contract's bech32 address.
2. The contract must self-destruct. This is a standard EVM operation callable by any contract that implements it.

Both conditions are reachable by an unprivileged user. Ethermint chains with bank or IBC precompiles (a common extension pattern) are directly exposed. The `ExecuteNativeAction` path in `statedb.go` is the production mechanism for native token operations: [4](#0-3) 

---

### Recommendation

Before calling `DeleteAccount`, iterate over all bank balances at the contract's Cosmos address and burn or redirect every denomination — not only the EVM denom. For example:

```go
// In Commit(), self-destruct branch — after burning evmDenom, before DeleteAccount:
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if coin.Denom == s.evmDenom {
        continue // already handled above
    }
    if err := s.keeper.BurnNonEvmCoins(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn non-evm-denom balance on selfdestruct")
    }
}
```

Alternatively, enforce at the precompile/native-action layer that non-EVM-denom tokens cannot be deposited into a contract address that does not have a guaranteed recovery path (the direct analog of the original report's remediation).

### Proof of Concept

1. Deploy contract `C` on an Ethermint chain with a bank precompile.
2. Via the bank precompile, transfer 1000 `ibc/ATOM` tokens into `C`'s balance.
3. Call `C.selfDestruct()` (or trigger EIP-6780 self-destruct in the same creation tx).
4. `StateDB.Commit()` executes: burns `C`'s EVM-denom balance, calls `DeleteAccount` — auth account removed, EVM storage cleared.
5. Query `bank.Balance(C_cosmos_addr, "ibc/ATOM")` → returns 1000 (balance still present).
6. Attempt `bank.Send(C_cosmos_addr, ...)` → fails: no auth account at `C_cosmos_addr`.
7. The 1000 `ibc/ATOM` are permanently orphaned.

The root cause is the unconditional single-denom drain at: [5](#0-4) 

with no corresponding loop over all bank denominations before the `DeleteAccount` call at: [6](#0-5)
