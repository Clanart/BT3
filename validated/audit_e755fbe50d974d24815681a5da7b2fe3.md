### Title
`BlockStatusCache::move_index` Fails to Update `key_to_index` and `leaf_hash_to_index` After Leaf Node Relocation During Deletion — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`BlockStatusCache::move_index` only marks the source slot as free in `free_indexes` but never updates `key_to_index` or `leaf_hash_to_index` for the moved node. When `MerkleBlob::delete` moves a **leaf** sibling to `TreeIndex(0)` (the root position) after removing the last internal parent, both index maps retain stale entries pointing to the now-freed source slot. Subsequent operations on the surviving leaf use the wrong index, and once a new insert reuses that freed slot, the surviving leaf's key and hash permanently resolve to the new leaf's data, corrupting all DataLayer Merkle proof and lookup operations.

---

### Finding Description

`BlockStatusCache` maintains three parallel data structures:

- `free_indexes` — set of available blob slots
- `key_to_index` — maps `KeyId → TreeIndex` for every leaf
- `leaf_hash_to_index` — maps `Hash → TreeIndex` for every leaf [1](#0-0) 

`move_index` is the only function responsible for updating the cache when a node physically relocates:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    // to be called _after_ having written to the destination index
    if self.free_indexes.contains(&source) {
        return Err(Error::MoveSourceIndexNotInUse(source));
    }
    if self.free_indexes.contains(&destination) {
        return Err(Error::MoveDestinationIndexNotInUse(destination));
    }

    self.free_indexes.insert(source);   // ← only free_indexes is updated

    Ok(())
}
``` [2](#0-1) 

It marks `source` as free but **never touches `key_to_index` or `leaf_hash_to_index`**.

`move_index` is called inside `delete` exactly when the deleted leaf's parent is the root (no grandparent), forcing the sibling to be physically relocated to `TreeIndex(0)`:

```rust
let Some(grandparent_index) = parent.parent.0 else {
    sibling_block.node.set_parent(Parent(None));
    let destination = TreeIndex(0);
    if let Node::Internal(node) = sibling_block.node {
        for child_index in [node.left, node.right] {
            self.update_parent(child_index, Some(destination))?;
        }
    }

    self.insert_entry_to_blob(destination, &sibling_block)?;
    self.block_status_cache
        .move_index(sibling_index, destination)?;   // ← stale maps not fixed

    return Ok(());
};
``` [3](#0-2) 

When the sibling is a **leaf** (the common case for a 2-leaf tree), after `move_index` returns:

| Map | Entry | Points to |
|---|---|---|
| `key_to_index[sibling.key]` | stale | `sibling_index` (now in `free_indexes`) |
| `leaf_hash_to_index[sibling.hash]` | stale | `sibling_index` (now in `free_indexes`) |
| blob at `TreeIndex(0)` | correct | sibling leaf data |

`get_leaf_by_key` resolves a key exclusively through `key_to_index`:

```rust
pub fn get_leaf_by_key(&self, key: KeyId) -> Result<(TreeIndex, LeafNode, Block), Error> {
    let index = *self
        .block_status_cache
        .get_index_by_key(key)
        .ok_or(Error::UnknownKey(key))?;
    let block = self.get_block(index)?;
    ...
}
``` [4](#0-3) 

So every subsequent call using `sibling.key` or `sibling.hash` reads from the stale (freed) slot.

When a new leaf C is later inserted, `get_new_index` pops `sibling_index` from `free_indexes` and `insert_entry_to_blob` calls `add_leaf(sibling_index, C)`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [5](#0-4) 

After this, `key_to_index[sibling.key]` still maps to `sibling_index`, which now holds C's data. Every lookup, proof, upsert, or delete for the surviving sibling silently operates on C's node instead.

The integrity check that would catch this is gated behind `#[cfg(any(test, debug_assertions))]` and is never called automatically in production release builds:

```rust
#[cfg(any(test, debug_assertions))]
impl Drop for MerkleBlob {
    fn drop(&mut self) {
        if self.check_integrity_on_drop {
            self.check_integrity()
                .expect("integrity check failed while dropping merkle blob");
        }
    }
}
``` [6](#0-5) 

`check_integrity_on_drop` defaults to `cfg!(test)`, so it is `false` in production: [7](#0-6) 

---

### Impact Explanation

After the stale index is overwritten by a new leaf:

- `get_leaf_by_key(sibling.key)` returns C's `(key, value, hash)` — wrong data for the surviving leaf.
- `get_node_by_hash(sibling.hash)` returns C's `(key, value)` — wrong data.
- Merkle inclusion proofs generated for `sibling.key` traverse from the wrong blob position, producing an invalid or forgeable proof path.
- `upsert(sibling.key, ...)` modifies C's node, silently corrupting C's stored value.
- `delete(sibling.key)` removes C's node, leaving the actual sibling leaf permanently orphaned and unremovable.
- The root hash computed from the tree is inconsistent with the key-value state reported by the cache, so any proof verified against the root will disagree with the cache's view of the tree.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The trigger condition — deleting one leaf from a 2-leaf tree — is a routine, low-privilege operation reachable through the public `MerkleBlob::delete` API and its Python binding `py_delete`. Any DataLayer store that shrinks to a single key after a deletion hits this path. No special privileges, keys, or network access are required. The corruption is silent in release builds and persists across all subsequent operations on the same `MerkleBlob` instance.

---

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf. The simplest fix is to pass the node being moved so the cache can update all three maps:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex, node: &Node) -> Result<(), Error> {
    if self.free_indexes.contains(&source) {
        return Err(Error::MoveSourceIndexNotInUse(source));
    }
    if self.free_indexes.contains(&destination) {
        return Err(Error::MoveDestinationIndexNotInUse(destination));
    }

    self.free_indexes.insert(source);

    // Fix: update leaf index maps for the moved node
    if let Node::Leaf(leaf) = node {
        self.key_to_index.insert(leaf.key, destination);
        self.leaf_hash_to_index.insert(leaf.hash, destination);
    }

    Ok(())
}
```

The call site in `delete` already has `sibling_block` in scope and can pass `&sibling_block.node`. [2](#0-1) [8](#0-7)

### Proof of Concept

```rust
use chia_datalayer::{InsertLocation, KeyId, MerkleBlob, ValueId};

fn sha256_num(n: i64) -> [u8; 32] { /* sha256 of n as little-endian bytes */ }

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    blob.check_integrity_on_drop = false; // disable debug guard

    // Insert two leaves: A and B
    let key_a = KeyId(1);
    let key_b = KeyId(2);
    let hash_a = sha256_num(1);
    let hash_b = sha256_num(2);
    blob.insert(key_a, ValueId(10), &hash_a, InsertLocation::Auto {}).unwrap();
    blob.insert(key_b, ValueId(20), &hash_b, InsertLocation::Auto {}).unwrap();

    // Delete leaf A → sibling B is moved to TreeIndex(0)
    // move_index() does NOT update key_to_index[B] or leaf_hash_to_index[B]
    blob.delete(key_a).unwrap();

    // Insert leaf C — reuses the freed slot that key_to_index[B] still points to
    let key_c = KeyId(3);
    let hash_c = sha256_num(3);
    blob.insert(key_c, ValueId(30), &hash_c, InsertLocation::Auto {}).unwrap();

    // get_leaf_by_key(B) now resolves to C's slot → returns C's data
    let (_, leaf, _) = blob.get_leaf_by_key(key_b).unwrap();
    assert_eq!(leaf.key, key_c);   // PASSES: B's key resolves to C's node
    assert_eq!(leaf.value, ValueId(30)); // PASSES: wrong value returned for B

    // check_integrity detects the corruption
    blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheIndex
}
```
