### Title
Missing Declared-Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the replacement class hash supplied by a contract is actually declared in the class trie. An explicit `TODO` comment in the production code acknowledges this missing check. As a result, any contract can irrevocably replace its own class hash with an undeclared (or zero) value, permanently bricking itself and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads the requested `class_hash` from the syscall request and unconditionally writes it into the contract's `StateEntry`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

There is **no lookup** into `contract_class_changes` or the global class trie to confirm that `class_hash` corresponds to a previously declared class. The TODO comment (dated 1/1/2026, now overdue) explicitly acknowledges this gap.

The analog to the original report's vulnerability class is direct: just as the Celo approver role can take an action (refusing to approve proposals) that permanently entrenches itself and blocks the only mechanism that could remove it, a StarkNet contract can invoke `replace_class` with an undeclared hash to permanently disable the only mechanism that could ever execute or upgrade it — its own class. Once the state is committed, no future transaction can execute any entry point on the contract (because the OS cannot resolve the class), and no further `replace_class` call can be made either.

The `revert_log` entry is written, so the change is reversible only if the enclosing transaction reverts. If the transaction succeeds, the state diff is committed and the replacement is permanent. [2](#0-1) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds user assets (ERC-20 tokens, ETH, NFTs, DeFi positions) and whose logic reaches a `replace_class` call with an attacker-controlled or zero class hash will be permanently bricked. Because the OS resolves the class at execution time from the state trie, and the trie now contains an undeclared hash, every future call to the contract will fail at the class-resolution step. There is no recovery path: the contract cannot execute a corrective `replace_class`, cannot transfer funds out, and cannot be upgraded. All funds are frozen indefinitely.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is the standard StarkNet upgrade mechanism. Any upgradeable contract that passes a caller-supplied or storage-read class hash to `replace_class` without first verifying it is declared is vulnerable. A malicious contract owner can exploit this intentionally (rug-pull variant: deploy a contract, attract user deposits, then call `replace_class(0)`). A buggy contract can trigger it accidentally. No special privilege is required beyond deploying a contract and submitting a transaction — both are available to any unprivileged user.

---

### Recommendation

Implement the missing check inside `execute_replace_class` before updating the state entry. Specifically, verify that `class_hash` exists as a key with a non-zero value in `contract_class_changes` (or, if not present there, in the pre-existing class trie). A minimal fix:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

If the class was declared in a prior block (not in the current block's `contract_class_changes`), the hint layer must also expose a trie-membership proof that the OS can verify.

---

### Proof of Concept

1. **Deploy** a contract `Vault` that holds user ERC-20 deposits and exposes an `upgrade(new_class_hash: felt)` entry point that calls `replace_class(new_class_hash)`.
2. **Attract** user deposits into `Vault`.
3. **Submit** an invoke transaction calling `Vault.upgrade(0)` (or any felt that is not a declared class hash).
4. The OS executes `execute_replace_class`:
   - Gas is deducted successfully.
   - `class_hash = 0` is written into `Vault`'s `StateEntry` with no validation.
   - The revert log records the old class hash.
   - The transaction succeeds; the state diff is committed.
5. **Any subsequent call** to `Vault` (withdraw, transfer, etc.) fails at class resolution because class hash `0` is not in the class trie.
6. All user funds in `Vault` are **permanently frozen**. [3](#0-2)

    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
