Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain surplus ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` transfers the router's entire ETH balance to `msg.sender` with no access control. ETH accumulates on the router when a user sends more `msg.value` than the swap requires, because `pay()` wraps only the exact amount needed and silently leaves the surplus. Any third party — including MEV bots — can call `refundETH()` in a subsequent transaction and steal that surplus, causing direct loss of user ETH principal.

## Finding Description
`refundETH()` is implemented with no caller restriction: [1](#0-0) 

ETH accumulates on the router via `msg.value` attached to payable entry points. Inside `pay()`, when `token == WETH` and `nativeBalance >= value`, exactly `value` is wrapped and the remainder stays on the contract: [2](#0-1) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent accumulation from `msg.value` on payable calls: [3](#0-2) 

`multicall()` executes sub-calls sequentially and returns without any automatic ETH refund, so a user who omits `refundETH()` from their call array leaves the surplus exposed indefinitely: [4](#0-3) 

Exploit path:
1. User calls `router.multicall{value: 1 ether}([exactInputSingle(..., amountIn: 0.5 ether, ...)])` — omitting `refundETH`.
2. `pay()` wraps 0.5 ETH; 0.5 ETH remains on the router.
3. Attacker calls `router.refundETH()` in a separate transaction.
4. `_transferETH(msg.sender, 0.5 ether)` sends the surplus to the attacker.

No existing guard prevents step 3. The `receive()` check is irrelevant because the attacker is not sending ETH — they are receiving it.

## Impact Explanation
Direct loss of user ETH principal. The victim loses the full surplus they sent above the swap amount. The loss is bounded only by how much excess ETH the user attached, which can be arbitrarily large. This meets the Critical/High direct-loss-of-user-principal threshold.

## Likelihood Explanation
`refundETH` is an opt-in call that users must explicitly append to their multicall array. Omitting it is a realistic mistake for users interacting directly with the contract or through integrations that do not enforce the pattern. The attack requires no special privileges, no malicious pool, and no non-standard token behavior — only a call to a public function. MEV bots routinely monitor for stranded ETH on router contracts and can execute the steal atomically in the same block.

## Recommendation
1. **Auto-refund in `multicall`**: after executing all sub-calls, if `address(this).balance > 0`, transfer the remainder back to `msg.sender` unconditionally.
2. **Transient-storage depositor tracking**: record `msg.sender` at the start of `multicall` in transient storage and restrict `refundETH` to that address within the same call context.

## Proof of Concept
The existing test `test_refundETH_sendsBalanceToCaller` already demonstrates the issue: an arbitrary `swapper` address (not the original depositor) calls `router.refundETH()` and receives the full router ETH balance: [5](#0-4)
