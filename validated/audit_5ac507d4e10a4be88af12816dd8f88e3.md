### Title
EVM Value Transfer to Blocked Module Address Surfaces as Cosmos-Level Error, Causing Fee Loss and Nonce Increment Without EVM Execution — (`x/evm/statedb/statedb.go`, `x/evm/keeper/state_transition.go`)

### Summary

When an EVM CALL transfers value to a Cosmos-blocked module address (e.g., the staking module account), `bankKeeper.SendCoins` fails inside `StateDB.Transfer`. The error is stored in `s.err` and surfaces only at `stateDB.Commit()` time as a cosmos-level error — not as a `VmError`. The only commit error converted to a VmError is `ErrStateConflict`. All other commit errors, including blocked-address rejections, propagate as cosmos-level failures from `ApplyMessageWithConfig` → `ApplyTransaction` → the message handler. Because the ante handler already committed fee deductions and nonce increments before message execution, the sender loses fees and has their nonce incremented with no EVM state change and no gas refund.

### Impact Explanation

- **Fee mis-accounting (High):** The sender's full `gasLimit * gasPrice` is deducted in the ante handler and never refunded. In a normal failed EVM tx (VmError), only `gasUsed * gasPrice` is charged and the remainder is refunded. Here, the refund path is bypassed entirely.
- **Nonce increment without EVM execution:** The sender's nonce is incremented by the ante handler. The EVM execution never completes. The tx is rejected at the message handler level, not included as a normal failed EVM tx with `status=0`.
- **Invariant broken:** EVM value transfers to blocked addresses must produce a `VmError` (status=0, included in block, gas used charged, remainder refunded). Instead they produce a cosmos-level rejection after ante handler state is committed.

### Likelihood Explanation

- Blocked module addresses are publicly derivable: `authtypes.NewModuleAddress("staking")`, `authtypes.NewModuleAddress("bonded_tokens_pool")`, etc.
- Any user who sends ETH value to such an address (accidentally or deliberately) triggers this path.
- The ante handler's `CheckEthCanTransfer` only checks sender balance, not recipient blocked status, so the tx passes ante validation unconditionally if the sender has sufficient funds.
- No privileged access required. Reachable via `eth_sendRawTransaction`.

### Recommendation

In `ApplyMessageWithConfig`, after `evm.Call`/`evm.Create` returns and before calling `stateDB.Commit()`, check `stateDB.Error()`. If it is non-nil, treat it as a VmError (set `vmErr = stateDB.Error()`, skip commit) rather than allowing it to surface from `Commit()` as a cosmos-level error. This mirrors how go-ethereum handles transfer failures — as EVM-level reverts, not infrastructure errors.

Alternatively, add a blocked-address pre-check in `CheckEthCanTransfer` for the recipient, so the tx is rejected before the ante handler commits fee deductions and nonce increments.

### Proof of Concept

1. Derive the staking bonded pool address: `sdk.AccAddress(authtypes.NewModuleAddress("bonded_tokens_pool"))`
2. Fund a sender EOA with sufficient EVM denom.
3. Send a raw EVM CALL transaction with `value > 0` and `to = <bonded_tokens_pool_address>`.
4. Observe: ante handler succeeds (nonce incremented, fees deducted). Message handler returns cosmos error `"failed to apply ethereum core message: failed to commit stateDB: ..."`.
5. Assert: sender nonce is `N+1`, sender balance decreased by full `gasLimit * gasPrice` (no refund), no EVM receipt with `status=0` is produced, and the tx is not included as a normal failed EVM tx.
