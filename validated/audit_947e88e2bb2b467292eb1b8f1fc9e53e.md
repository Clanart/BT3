### Title
Receipt Commitment Hardcodes L2 Gas as Zero, Producing Wrong Block Hash for L2-Gas-Consuming Transactions - (File: crates/starknet_api/src/block_hash/receipt_commitment.rs)

### Summary

`chain_gas_consumed` in `receipt_commitment.rs` unconditionally chains `Felt::ZERO` for the L2 gas component of every receipt leaf instead of reading `gas_consumed.l2_gas`. This is the direct sequencer analog of the CurveLPStaking bug: just as that controller hardcoded `_claim_rewards = false` and silently dropped the reward-claiming path, this function accepts a full `GasVector` (which carries `l2_gas`) but silently drops the L2 dimension, producing a structurally wrong receipt commitment for every block that contains L2-gas-consuming transactions.

### Finding Description

`calculate_receipt_commitment` builds a Poseidon Patricia tree whose leaves are produced by `calculate_receipt_hash`. That function delegates the gas portion to `chain_gas_consumed`:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO) // L2 gas consumed  ← hardcoded, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

The comment acknowledges the issue: *"L2 gas consumed (In the current RPC: always 0)"*. The function signature accepts `&GasVector`, which carries `l2_gas: GasAmount`, but the field is never read. The Starknet receipt-hash specification chains three gas dimensions (L2, L1, L1-data); the code chains only two real values and substitutes a constant zero for the first.

`GasVector.l2_gas` is non-zero in production. Starknet 0.14.x blocks already carry non-zero `total_gas_consumed.l2_gas` (e.g., `76014735` in `block_post_0_14_2.json`). The blockifier's own regression tests confirm non-zero L2 gas vectors (`l2_gas: GasAmount(637545)`) for executed transactions. `prepare_txs_hashing_data` feeds `info.output_for_hashing()` — which populates `TransactionOutputForHash.gas_consumed` from the execution receipt — directly into `calculate_block_commitments`, so the non-zero `l2_gas` value is present in the `GasVector` passed to `chain_gas_consumed` but is silently discarded.

The corrupted receipt commitment then propagates into the block hash through `calculate_block_hash`, which chains `block_commitments.receipt_commitment.0` as one of its inputs, and into the `StorageBlockHeader.receipt_commitment` field persisted to MDBX storage.

### Impact Explanation

Every block containing at least one transaction with non-zero L2 gas receives a wrong `receipt_commitment`. Because `receipt_commitment` is a direct input to `calculate_block_hash`, the block hash itself is wrong. Any external verifier, light client, or proof system that independently computes the receipt commitment using the actual L2 gas value will derive a different block hash than the sequencer, breaking cross-system agreement on the canonical chain state. The stored `receipt_commitment` returned by RPC (`starknet_getBlockWithTxHashes`, `starknet_getStateUpdate`) is an authoritative-looking wrong value.

**Impact category:** High — RPC returns an authoritative-looking wrong receipt commitment; also touches the Critical category of wrong receipt/block-hash from execution logic.

### Likelihood Explanation

Any Invoke V3 transaction executed under `GasVectorComputationMode::All` (the mode used for all post-0.13.x transactions with `AllResourceBounds`) produces non-zero `l2_gas`. No special privilege is required; any user submitting a standard transaction triggers the condition. The defect is present in every block produced by the sequencer once L2 gas accounting is active.

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual field value:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // was: Felt::ZERO
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

Update the corresponding regression test vectors in `receipt_commitment_test.rs` to use non-zero `l2_gas` inputs and verify the expected root changes accordingly.

### Proof of Concept

1. Submit any Invoke V3 transaction with `AllResourceBounds` to the sequencer.
2. After execution, `TransactionExecutionInfo.receipt.resources` will contain a `GasVector` with `l2_gas > 0` (confirmed by regression test showing `l2_gas: GasAmount(637545)`).
3. `prepare_txs_hashing_data` calls `info.output_for_hashing()`, populating `TransactionOutputForHash.gas_consumed.l2_gas` with the non-zero value.
4. `calculate_block_commitments` passes this `GasVector` to `calculate_receipt_commitment` → `calculate_receipt_hash` → `chain_gas_consumed`.
5. `chain_gas_consumed` chains `Felt::ZERO` instead of `gas_consumed.l2_gas`, producing a receipt leaf hash that differs from the correct value.
6. The resulting `receipt_commitment` stored in `StorageBlockHeader` and returned by RPC is wrong; any independent computation using the actual L2 gas value yields a different block hash.
