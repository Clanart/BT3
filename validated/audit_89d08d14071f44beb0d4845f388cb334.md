### Title
Detached `send_tokens` After Irreversible State Mutation in `utxo_fin_transfer_fast` Causes Permanent Relayer Fund Loss — (`near/omni-bridge/src/lib.rs`)

---

### Summary

In `utxo_fin_transfer_fast`, the fast-transfer state is irreversibly mutated (removed or marked finalised) **before** `send_tokens` is called with `.detach()`. If the token transfer to the relayer fails, the relayer permanently loses their fronted tokens with no recovery path. The developers themselves flagged this with a `// TODO: check how to deal with failed send_tokens` comment at the call site.

---

### Finding Description

`utxo_fin_transfer_fast` is invoked from `utxo_fin_transfer` when a UTXO-chain deposit arrives and a matching fast-transfer record already exists (i.e., a relayer previously fronted tokens to the user). The function is responsible for repaying that relayer.

The critical sequence is:

```
// near/omni-bridge/src/lib.rs  lines 2518–2561
fn utxo_fin_transfer_fast(...) {
    ...
    let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
        self.remove_fast_transfer(&fast_transfer.id());          // ← state mutated (record deleted)
        fast_transfer.amount
    } else {
        self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // ← state mutated (finalised = true)
        U128(fast_transfer.amount_without_fee()...)
    };

    self.send_tokens(
        fast_transfer.token_id.clone(),
        fast_transfer_status.relayer,
        amount,
        "",
    )
    .detach();   // ← fire-and-forget, no callback
    ...
}
```

The state mutation (`remove_fast_transfer` or `mark_fast_transfer_as_finalised`) is committed to storage **in the same transaction** before `send_tokens` is even scheduled. Because `.detach()` fires the promise without attaching any callback, a failure of the underlying `ft_transfer` or `mint` call is silently discarded.

If `send_tokens` fails:

- **Destination = NEAR**: `remove_fast_transfer` already deleted the record. The relayer cannot retry; the fast-transfer ID no longer exists.
- **Destination ≠ NEAR**: `mark_fast_transfer_as_finalised` set `finalised = true`. Any retry hits `ERR_FAST_TRANSFER_ALREADY_FINALISED`.

In both branches the relayer's fronted tokens are permanently unrecoverable.

The developers explicitly acknowledged the gap with the comment immediately before the call:

```rust
// TODO: check how to deal with failed send_tokens
return self.utxo_fin_transfer_fast(fast_transfer, status, utxo_fin_transfer_msg);
```

This is structurally identical to the Canto `processEvents` bug: a state change is committed (contract marked registered / fast-transfer marked finalised) while the operation that should accompany it (CSR store write / token transfer) can silently fail, leaving the two components permanently desynced.

---

### Impact Explanation

A relayer who fronted tokens for a user via `fast_fin_transfer` can have those tokens permanently frozen inside the bridge contract. The fast-transfer record is either deleted or locked in a finalised state, so no retry or recovery call is possible. This satisfies:

> **Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

The relayer's fronted capital is lost with no on-chain recourse.

---

### Likelihood Explanation

`send_tokens` dispatches either `ft_transfer` or `mint` on the token contract. Realistic failure conditions include:

- The relayer account has no storage registration for the token (NEP-141 requires it).
- The token contract is paused or temporarily unavailable.
- Insufficient gas allocated to the detached promise (the gas budget is computed dynamically and can be zero after saturation).

None of these require adversarial action; they can occur in normal operation. The `// TODO` comment confirms the developers have not yet implemented a guard.

---

### Recommendation

Replace the `.detach()` pattern with a chained callback that checks the result of `send_tokens` and reverts the state mutation on failure:

```rust
self.send_tokens(...)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(RESOLVE_UTXO_FAST_TRANSFER_GAS)
            .resolve_utxo_fast_transfer(&fast_transfer.id(), amount, destination_is_near),
    )
    .into()
```

`resolve_utxo_fast_transfer` should re-insert the fast-transfer record (or clear the `finalised` flag) if the token transfer failed, mirroring the existing `resolve_fast_transfer` and `resolve_utxo_fin_transfer` callbacks.

---

### Proof of Concept

1. Relayer fronts tokens for a user via `ft_transfer_call` → `ft_on_transfer` → `fast_fin_transfer` → `fast_fin_transfer_to_other_chain`. A `FastTransfer` record is stored with `finalised = false`.
2. The UTXO connector later calls `ft_transfer_call` → `ft_on_transfer` → `utxo_fin_transfer`. Because a matching fast-transfer record exists, `utxo_fin_transfer_fast` is called.
3. Inside `utxo_fin_transfer_fast`, `mark_fast_transfer_as_finalised` is called — the record is now `finalised = true` in storage.
4. `send_tokens(...).detach()` is called. The relayer's account lacks a storage deposit for the token; `ft_transfer` panics inside the token contract.
5. The panic is silently discarded (detached promise). The bridge transaction succeeds.
6. The relayer attempts to recover by calling any bridge function referencing the fast-transfer ID. Every path hits `ERR_FAST_TRANSFER_ALREADY_FINALISED` or `ERR_TRANSFER_NOT_EXIST`.

    fn remove_fast_transfer(&mut self, fast_transfer_id: &FastTransferId) {
        let storage_usage = env::storage_usage();
        let fast_transfer = self
            .fast_transfers
            .remove(fast_transfer_id)
            .map(storage::FastTransferStatusStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&fast_transfer.storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances
                .insert(&fast_transfer.storage_owner, &storage);
        }
    }

    fn utxo_fin_transfer_fast(
        &mut self,
        fast_transfer: FastTransfer,
        fast_transfer_status: FastTransferStatus,
        utxo_fin_transfer_msg: UtxoFinTransferMsg,
    ) -> PromiseOrPromiseIndexOrValue<U128> {
        require!(
            !fast_transfer_status.finalised,
            BridgeError::FastTransferAlreadyFinalised.as_ref()
        );

        let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
            self.remove_fast_transfer(&fast_transfer.id());
            fast_transfer.amount
        } else {
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
            // With transfers to other chain the fee will be claimed after finalization on the destination chain
            U128(
                fast_transfer
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            )
        };

        self.send_tokens(
            fast_transfer.token_id.clone(),
            fast_transfer_status.relayer,
            amount,
            "",
        )
        .detach();

        env::log_str(
            &OmniBridgeEvent::UtxoTransferEvent {
                token_id: fast_transfer.token_id,
                amount,
                utxo_transfer_message: utxo_fin_transfer_msg,
                new_transfer_id: None,
            }
            .to_log_string(),
        );

        PromiseOrPromiseIndexOrValue::Value(U128(0))
    }
