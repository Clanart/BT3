### Title
`ft_transfer` Failure in `fin_transfer_send_tokens_callback` Permanently Freezes User Funds — (`near/omni-bridge/src/lib.rs`)

### Summary

`fin_transfer_send_tokens_callback` never checks whether the underlying `ft_transfer` promise succeeded when `msg` is empty (`is_ft_transfer_call = false`). Because `is_refund_required` unconditionally returns `false` for that branch, any `ft_transfer` failure leaves the transfer permanently marked as finalized and the user's tokens irrecoverably locked inside the bridge contract.

### Finding Description

The NEAR bridge's `fin_transfer` flow for a NEAR-recipient transfer proceeds as follows:

1. `process_fin_transfer_to_near` calls `add_fin_transfer`, which inserts the transfer ID into `finalised_transfers` (making it non-replayable) and decrements `locked_tokens` via `unlock_tokens_if_needed`.
2. It then calls `send_tokens` and chains `fin_transfer_send_tokens_callback`.
3. Inside the callback, the only path that reverts state (re-locks tokens, removes the finalization record) is guarded by `is_refund_required`.

`is_refund_required` is:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => { ... amount.0 == 0 }
            Err(_) => false,   // promise failed → no refund
        }
    } else {
        false   // ← always false for plain ft_transfer
    }
}
```

When `msg` is empty, `is_ft_transfer_call = false` and `is_refund_required` returns `false` unconditionally — it never inspects the promise result at all. If `ft_transfer` fails (token contract paused, panics, gas exhaustion, etc.), the callback still takes the "success" else-branch:

```rust
} else {
    // Send fee to the fee recipient
    if transfer_message.fee.fee.0 > 0 { ... }
    env::log_str(&OmniBridgeEvent::FinTransferEvent { ... });
}
```

No call to `revert_lock_actions`, `remove_fin_transfer`, or any state rollback occurs. The result:

- `finalised_transfers` permanently contains the transfer ID → the proof cannot be re-submitted.
- `locked_tokens` was already decremented → accounting is corrupted.
- The tokens remain in the bridge contract with no mechanism to release them to the recipient.

The refund path that does exist (for `ft_transfer_call`) correctly calls `revert_lock_actions` and `remove_fin_transfer`, but this path is structurally unreachable for plain `ft_transfer`.

### Impact Explanation

**Critical — Permanent freezing of user funds.** A user who bridges tokens to NEAR (with no `msg`) can have their funds permanently locked in the bridge contract if the `ft_transfer` step fails for any reason. The transfer is marked finalized so it cannot be retried, and the locked-token accounting is corrupted. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

**Medium.** Plain `ft_transfer` (no `msg`) is the default path for ordinary token withdrawals. Failure conditions include: the token contract being paused at the moment of transfer, a token contract upgrade that introduces a panic, gas exhaustion inside the token contract, or the recipient account being deleted between the storage-deposit step and the transfer step. None of these require attacker action; they are realistic operational conditions.

### Recommendation

`is_refund_required` must also inspect the promise result for the `ft_transfer` (non-call) case:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => { ... }
            Err(_) => true,  // ft_transfer_call itself failed → revert
        }
    } else {
        // Check whether the plain ft_transfer promise succeeded
        env::promise_result(0) != PromiseResult::Successful(vec![])
        // or equivalently: env::promise_result_checked(0, 0).is_err()
    }
}
```

Alternatively, restructure `fin_transfer_send_tokens_callback` to always check `env::promise_result(0)` first and only proceed to the fee-payment path when the transfer succeeded.

### Proof of Concept

```
1. Relayer calls fin_transfer() with a valid proof for a NEAR-recipient transfer (msg = "").
   → process_fin_transfer_to_near() runs:
       add_fin_transfer()          // transfer ID inserted into finalised_transfers
       unlock_tokens_if_needed()   // locked_tokens decremented
       send_tokens(..., "")        // ft_transfer dispatched (not ft_transfer_call)
       .then(fin_transfer_send_tokens_callback(..., is_ft_transfer_call=false, ...))

2. ft_transfer fails (e.g., token contract is paused mid-flight).
   → NEAR runtime calls fin_transfer_send_tokens_callback with promise result = Failed.

3. is_refund_required(false) → returns false unconditionally.
   → else-branch executes: fee sent (or skipped), FinTransferEvent logged.
   → No revert_lock_actions(), no remove_fin_transfer().

4. State after:
   - finalised_transfers contains the transfer ID → proof cannot be re-submitted.
   - locked_tokens is lower than actual holdings → accounting corrupted.
   - Tokens sit in the bridge contract with no recovery path.
   - User's funds are permanently frozen.

        if is_ft_transfer_call {

                        // Unexpected case: don't refund
                        false
                    }
                }

            }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
