### Title
Missing Deadline Check in `addLiquidityExactShares` Allows Stale Liquidity Addition at Unfavorable Prices - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

### Summary

`MetricOmmPoolLiquidityAdder` exposes `addLiquidityExactShares` (both overloads) and `addLiquidityWeighted` (both overloads) with no deadline parameter. A transaction submitted at a favorable price can sit in the mempool indefinitely and execute after the oracle price has moved significantly, causing the user to add liquidity at wrong bin positions and suffer immediate principal loss. The router (`MetricOmmSimpleRouter`) consistently calls `_checkDeadline(params.deadline)` on every swap entry point, but the liquidity adder has no equivalent guard.

### Finding Description

`MetricOmmSimpleRouter` enforces a deadline on all four swap entry points: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts no deadline and has no cursor-bounds check either: [5](#0-4) 

`addLiquidityWeighted` adds a cursor-bounds check (`_validateBinAndBinPosition`) but still accepts no deadline, so a transaction can execute at any future time as long as the cursor happens to fall within the user-specified range: [6](#0-5) 

The `maxAmountToken0` / `maxAmountToken1` caps only bound the absolute token quantities pulled; they do not protect against the price at which those tokens are deposited. In a bin-based oracle AMM, the bin composition at execution time determines the economic value of the position. If the oracle price has moved between submission and execution, the user's tokens are deposited into bins that are now mispriced relative to the market, and an arbitrageur can immediately extract the difference.

### Impact Explanation

A user who submits `addLiquidityExactShares` during a period of low network activity (or with a low gas price) can have their transaction mined minutes or hours later after the oracle price has shifted. Their tokens are deposited into bins at the stale composition. An arbitrageur immediately swaps against those bins at the favorable rate, extracting value directly from the user's newly deposited principal. The loss is bounded by `maxAmountToken0` / `maxAmountToken1` but can approach those full caps in a large price move.

### Likelihood Explanation

Any user who submits a liquidity addition during mempool congestion or with a competitive gas price is exposed. No privileged access is required; the attacker is simply a MEV bot that monitors pending transactions and waits for the oracle to update before the liquidity transaction is mined. This is a standard sandwich/stale-tx pattern that is routinely exploited on mainnet.

### Recommendation

Add a `uint256 deadline` parameter to all `addLiquidityExactShares` and `addLiquidityWeighted` overloads and call the same `_checkDeadline` helper used by the router at the top of each function, before any state reads or external calls. This mirrors the fix applied to the Delay app (requiring the current timestamp to be before the relevant time boundary) and is consistent with the deadline enforcement already present in `MetricOmmSimpleRouter`.

### Proof of Concept

1. User calls `addLiquidityExactShares(pool, owner, salt, deltas, maxAmount0=1000e6, maxAmount1=1000e18, extensionData)` with a low gas price. The oracle bid/ask at submission time is `P`.
2. Transaction sits in the mempool. The oracle updates; the new price is `P' = 1.05 * P` (5 % move).
3. A validator mines the user's transaction. The pool now prices token0 5 % higher.
4. The callback pulls up to `maxAmount0` token0 and `maxAmount1` token1 from the user and deposits them into bins calibrated to the old price `P`.
5. An MEV bot immediately swaps against those bins at the favorable rate, extracting ~5 % of the deposited value from the user's position.
6. The user's LP position is worth materially less than the tokens they deposited, with no recourse. [7](#0-6)

  function metricOmmModifyLiquidityCallback(uint256 amount0Delta, uint256 amount1Delta, bytes calldata callbackData)
    external
    override
  {
    uint8 kind = abi.decode(callbackData, (uint8));
    if (kind == KIND_PROBE) {
      revert LiquidityProbe(amount0Delta, amount1Delta);
    }
    if (kind != KIND_PAY) revert InvalidCallbackKind();

    (address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
    if (expectedPool == address(0)) revert CallbackContextNotActive();
    if (msg.sender != expectedPool) revert InvalidCallbackCaller(msg.sender, expectedPool);
    if (amount0Delta > max0 || amount1Delta > max1) {
      revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
    }

    PoolImmutables memory imm = IMetricOmmPool(msg.sender).getImmutables();
    address token0 = imm.token0;
    address token1 = imm.token1;
    if (amount0Delta > 0) {
      pay(token0, payer, msg.sender, amount0Delta);
    }
    if (amount1Delta > 0) {
      pay(token1, payer, msg.sender, amount1Delta);
    }
    _clearPayContext();
