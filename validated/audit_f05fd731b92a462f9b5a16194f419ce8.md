### Title
Small-Amount Transfer Permanently Locks User Tokens Due to `normalize_amount` Floor-Division Returning Zero - (File: near/omni-bridge/src/lib.rs)

### Summary

When a user initiates a cross-chain transfer with an amount smaller than `10^(origin_decimals - decimals)`, the `normalize_amount` floor-division in `sign_transfer` returns 0, causing the function to panic with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Because the tokens are already locked at `init_transfer` time and no cancel/refund path exists for this case, the user's funds are permanently irrecoverable.

### Finding Description

**Root cause — `normalize_amount` floor-division:** [1](#0-0) 

The function divides by `10^diff_decimals`. For any `amount_without_fee < 10^diff_decimals`, the result is 0.

**`sign_transfer` panics after tokens are already locked:** [2](#0-1) 

The `require!(amount_to_transfer > 0)` guard fires and reverts the entire transaction. However, the user's tokens were locked during the earlier `ft_transfer_call` → `init_transfer` step, which stored the `TransferMessage` in `pending_transfers`. The revert of `sign_transfer` does not undo that lock.

**No cancel/refund path exists for this case:**

- `claim_fee_callback` requires a proof of finalization on the destination chain — impossible if `sign_transfer` always panics.
- `sign_transfer_callback` (which removes the message when `fee.is_zero()`) is never reached because the panic occurs before the MPC cross-contract call is dispatched. [3](#0-2) 

- No `cancel_transfer` or user-initiated refund function exists in the contract.

**The `init_transfer` entry point performs no normalization pre-check:** [4](#0-3) 

Only `fee < amount` is validated. There is no guard that `normalize_amount(amount_without_fee) > 0` before locking tokens.

**Concrete example:**

A token registered with `origin_decimals = 18`, `decimals = 6` (diff = 12). A user transfers `amount = 999_999_999_999` (< 10^12). Tokens are locked. Every subsequent `sign_transfer` call panics. Tokens are permanently locked.

The existing code comment acknowledges dust locking only for the *remainder* after normalization: [5](#0-4) 

But this case is distinct: the *entire* transfer amount normalizes to zero, not just a remainder.

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds** in the NEAR bridge contract. Any user who transfers an amount below the decimal-scaling threshold for a token with `origin_decimals > decimals` loses their tokens with no recovery path. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

The threshold scales with the decimal gap: for `diff_decimals = 12`, any transfer below ~0.000001 of the token's base unit is permanently locked. For tokens with larger gaps (e.g., diff = 15), the threshold rises to economically meaningful amounts.

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` with a small amount. No special role is required to initiate a transfer. The `#[trusted_relayer]` restriction on `sign_transfer` means the user cannot even attempt to self-rescue — only a trusted relayer can call it, and it will always panic for this transfer. The condition is reachable on any token pair where `origin_decimals > decimals`.

### Recommendation

Add a normalization pre-check at `init_transfer` time, before locking tokens:

```rust
require!(
    Self::normalize_amount(
        init_transfer_msg.fee.fee.0.checked_sub(amount.0)
            .unwrap_or(amount.0),
        decimals
    ) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Alternatively, add a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have never been signed, so that users are not permanently harmed by this edge case.

### Proof of Concept

1. Register a token with `origin_decimals = 18`, `decimals = 6` (diff = 12).
2. User calls `ft_transfer_call` with `amount = 500_000_000_000` (< 10^12), `fee = 0`.
3. Bridge locks 500_000_000_000 tokens; `TransferMessage` stored in `pending_transfers`.
4. Trusted relayer calls `sign_transfer` for this transfer.
5. `normalize_amount(500_000_000_000, Decimals { decimals: 6, origin_decimals: 18 })` = `500_000_000_000 / 10^12` = `0`.
6. `require!(0 > 0)` panics → `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Transfer message remains in `pending_transfers`; tokens remain locked forever.
8. No `cancel_transfer` exists; `claim_fee` requires destination-chain proof that can never be produced. [6](#0-5) [2](#0-1) [7](#0-6)

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
