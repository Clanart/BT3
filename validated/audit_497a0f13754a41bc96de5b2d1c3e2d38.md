### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay Attacks — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that omits any nonce or replay-protection field. The OS also hardcodes `nonce=0` in the resulting `TxInfo` and never calls `check_and_increment_nonce` for these calls. Because the hash is a pure deterministic function of `(contract_address, selector, calldata, chain_id)`, a signature produced for one `meta_tx_v0` execution is valid for every future re-execution of the same call. Any unprivileged actor who observes a valid `meta_tx_v0` on-chain can replay it an unlimited number of times, draining the target account.

---

### Finding Description

**Root cause — hash omits nonce (`transaction_hash.cairo`)**

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

The `additional_data` slot is exactly where the nonce lives for every other transaction type (e.g., L1-handler passes `additional_data_size=1, additional_data=&nonce`). [2](#0-1) 

**Root cause — OS hardcodes `nonce=0` and skips nonce increment (`syscall_impls.cairo`)**

`execute_meta_tx_v0` constructs the inner `TxInfo` with `nonce=0` (hardcoded) and never calls `check_and_increment_nonce`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // ← always zero, never incremented
    ...
);
``` [3](#0-2) 

`check_and_increment_nonce` explicitly skips version-0 transactions, so even if it were called it would be a no-op:

```cairo
func check_and_increment_nonce{...}(tx_info: TxInfo*) {
    if (tx_info.version == 0) {
        return ();   // ← no nonce check for v0
    }
    ...
}
``` [4](#0-3) 

**`MetaTxV0Request` carries no nonce field**

The syscall request struct itself provides no nonce slot, so there is no caller-supplied value that could be incorporated:

```cairo
struct MetaTxV0Request {
    contract_address: felt,
    selector: felt,
    calldata_start: felt*,
    calldata_end: felt*,
    signature_start: felt*,
    signature_end: felt*,
}
``` [5](#0-4) 

**Consequence**

The hash `H(INVOKE_PREFIX, 0, contract_address, __execute__, calldata, 0, chain_id)` is identical across every block and every caller. A signature that validates against this hash is permanently valid. The target account contract receives `tx_info.nonce = 0` on every invocation, so it cannot use the standard nonce field to distinguish a first call from a replay.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A `meta_tx_v0` call is the canonical mechanism for a relayer to execute a user-signed action (e.g., an ERC-20 transfer, a DeFi operation) on behalf of the user. Because the hash is nonce-free, any attacker who extracts the signature from a confirmed transaction can replay the identical call in any subsequent block. Each replay executes the same state-changing action (e.g., transfers the same token amount to the same recipient) against the victim's account. The victim's account balance is drained proportionally to the number of replays the attacker submits.

---

### Likelihood Explanation

**High.**

- All transaction data, including the signature passed in `MetaTxV0Request`, is public on-chain.
- The attacker only needs to deploy a contract that calls the `meta_tx_v0` syscall — no privileged access, no leaked key, no operator cooperation.
- The replay succeeds as long as the victim's account has a balance; the attacker can loop until it is empty.
- No OS-level guard exists to detect or block the replay.

---

### Recommendation

1. **Add a nonce field to `MetaTxV0Request`** and include it in `compute_meta_tx_v0_hash` via `additional_data_size=1, additional_data=&nonce` (mirroring the L1-handler pattern).
2. **Track and increment the nonce** for the target `contract_address` inside `execute_meta_tx_v0`, or maintain a separate per-`(contract_address, nonce)` bitmap to mark consumed nonces.
3. Alternatively, bind the meta-tx hash to the **outer transaction hash** (`old_tx_info.transaction_hash`) so that each outer transaction can produce at most one valid meta-tx execution.

---

### Proof of Concept

**Setup**

- Alice signs a `meta_tx_v0` authorising a transfer of 1000 STRK from her account (`alice_addr`) to `recipient`. The signature `sig_A` covers `H(INVOKE_PREFIX, 0, alice_addr, __execute__, calldata, 0, chain_id)`.
- A legitimate relayer submits the call; the transfer executes in block N.

**Replay**

```
// Attacker's contract (Cairo pseudocode)
func attack() {
    // Reuse Alice's public calldata and signature from block N.
    meta_tx_v0(
        contract_address = alice_addr,
        selector         = EXECUTE_ENTRY_POINT_SELECTOR,
        calldata         = alice_calldata,   // same as block N
        signature        = sig_A,            // same as block N
    );
}
```

**OS execution path**

1. `execute_meta_tx_v0` computes `meta_tx_hash = compute_meta_tx_v0_hash(alice_addr, __execute__, alice_calldata, chain_id)`.
2. Because no nonce is hashed, `meta_tx_hash` is identical to the one from block N.
3. `new_tx_info.nonce = 0` — same as block N.
4. Alice's `__execute__` entry point receives the same `(transaction_hash, signature, nonce=0)` tuple it saw in block N and passes signature verification.
5. The 1000 STRK transfer executes again.
6. The attacker repeats until Alice's balance reaches zero.
