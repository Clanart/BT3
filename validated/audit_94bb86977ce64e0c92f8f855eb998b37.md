### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Validation in Fast Path, Corrupting DataLayer Tree Roots - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` contains a fast-path code branch that writes leaf nodes directly to the blob via `insert_entry_to_blob` without consulting the `block_status_cache` duplicate-key guard that `insert` enforces. A caller supplying a batch containing a repeated `KeyId` — or a `KeyId` already present in the tree — silently produces a Merkle tree with two leaf nodes sharing the same key, yielding a corrupted root hash and invalidating all subsequent proofs of inclusion/exclusion.

---

### Finding Description

`MerkleBlob::insert` correctly rejects duplicates before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a completely different code path for all items when the tree already has two or more leaves (i.e., `leaf_count > 1`). In that case the `if self.block_status_cache.leaf_count() <= 1` branch is skipped entirely, and every item in the batch is written directly through `insert_entry_to_blob` with no duplicate check:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        ...
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even when the tree has ≤1 leaves, only the last two items (popped from the end of the vector) are routed through the safe `insert` path; all remaining items still take the unchecked fast path. [3](#0-2) 

The Python binding `py_batch_insert` exposes this function directly to callers with no additional validation:

`BlockStatusCache::add_leaf` uses `HashMap::insert`, which silently overwrites the previous index for a duplicate key:

```rust
// lines 188-192
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [6](#0-5) 

The result is that the blob contains two physical leaf nodes for the same key, but the cache tracks only the last one. The tree root hash is computed over both nodes, producing a value that no honest verifier can reproduce from a single-key proof path.

---

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Concretely:

1. **Corrupted root hash.** The internal hash of every ancestor node is computed over both duplicate leaves. The committed root hash diverges from what any correct single-key proof would reconstruct, making all proofs of inclusion for any key in the tree unverifiable.
2. **Orphaned leaf node.** After the duplicate is inserted, `key_to_index` maps the key to only the second leaf's index. The first leaf node exists in the blob but is unreachable through the cache. `check_integrity()` will detect the mismatch (`leaf_count != key_to_index_cache_length`), but only if explicitly called.
3. **Silent corruption if integrity check is omitted.** If the caller does not invoke `check_integrity()` after `batch_insert`, the corrupted tree is silently committed and used to generate proofs.

---

### Likelihood Explanation

- `batch_insert` / `py_batch_insert` is a public API called by DataLayer sync logic whenever a set of key-value pairs is committed.
- Any caller — including Python DataLayer code — that passes a batch containing a repeated `KeyId` (e.g., due to a race condition, a malformed delta, or an adversarial DataLayer peer supplying duplicate rows) triggers the corruption.
- No privileged role is required; the function accepts caller-supplied data without pre-validation.
- The single-item `insert` path correctly rejects duplicates, so the discrepancy is non-obvious and unlikely to be caught in code review.

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the top of `batch_insert`, mirroring the guards in `insert`:

```rust
pub fn batch_insert(
    &mut self,
    keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // also check for intra-batch duplicates
    let mut seen_keys = HashSet::new();
    let mut seen_hashes = HashSet::new();
    for ((key, _), hash) in &keys_values_hashes {
        if !seen_keys.insert(key) { return Err(Error::KeyAlreadyPresent()); }
        if !seen_hashes.insert(hash) { return Err(Error::HashAlreadyPresent()); }
    }
    // ... existing logic
}
```

Alternatively, route all items through the existing `insert` function (accepting the performance cost) or add a post-insert `check_integrity()` call that is enforced rather than optional.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import os

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 3 leaves so leaf_count > 1 (fast path is taken)
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), os.urandom(32))

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with a duplicate key — no error is raised
dup_key = KeyId(99)
blob.batch_insert(
    [(dup_key, ValueId(1)), (dup_key, ValueId(2))],  # same key twice
    [os.urandom(32), os.urandom(32)],
)

blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# root_after is computed over two leaf nodes with key=99;
# any proof of inclusion for key=99 will fail to reconstruct root_after
proof = blob.get_proof_of_inclusion(dup_key)
print("proof valid:", proof.valid())   # False — tree is corrupted

# check_integrity reveals the inconsistency
blob.check_integrity()  # raises IntegrityKeyToIndexCacheLength or similar
```

The `batch_insert` call succeeds silently, the root hash is corrupted, and `get_proof_of_inclusion` returns an invalid proof. The `insert` path (used for single items) would have raised `KeyAlreadyPresentError` for the second `dup_key` entry. [7](#0-6) [8](#0-7) [4](#0-3)
