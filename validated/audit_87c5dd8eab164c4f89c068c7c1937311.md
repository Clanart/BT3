### Title
Unbounded `name` and `symbol` Strings in `LogMetadataMessage` Allow Proof Submitter to Inflate Bridge State and Block Token Deployment — (`File: near/omni-types/src/prover_result.rs`)

---

### Summary

`LogMetadataMessage`, the proof-parsed result type for `LogMetadata` events, stores `name` and `symbol` as plain, unbounded `String` fields. A malicious EVM token deployer can set arbitrarily long token name/symbol values, then any proof submitter can relay the resulting `LogMetadata` proof to NEAR. Because no length cap is enforced on deserialization, the oversized strings propagate through the bridge's `log_metadata` handler unchecked, potentially causing the bridge to store or forward payloads that exceed NEAR's object/argument size limits, permanently blocking token deployment for that token address.

---

### Finding Description

`LogMetadataMessage` is defined in `near/omni-types/src/prover_result.rs`:

```rust
pub struct LogMetadataMessage {
    pub token_address: OmniAddress,
    pub name: String,      // ← unbounded
    pub symbol: String,    // ← unbounded
    pub decimals: u8,
    pub emitter_address: OmniAddress,
}
``` [1](#0-0) 

These fields are populated directly from the proof payload submitted by any proof submitter. The codebase already has a `BoundedString<const MAX: usize>` type that enforces length limits on both JSON and Borsh deserialization: [2](#0-1) 

This type is correctly applied to user-facing inputs — for example, `InitTransferMsg.msg` is typed as `BoundedString<MAX_INIT_TRANSFER_MSG_LEN>` and `InitTransferMsg.external_id` as `BoundedString<MAX_EXTERNAL_ID_LEN>`: [3](#0-2) 

However, `LogMetadataMessage.name` and `LogMetadataMessage.symbol` — which originate from proof data, not from direct user input — remain plain `String` with no length enforcement. The same gap exists in `InitTransferMessage.msg`: [4](#0-3) 

The `log_metadata` handler in `near/omni-bridge/src/lib.rs` processes `LogMetadataMessage` and passes `name`/`symbol` into `BasicMetadata`, which is then forwarded to the token deployer's `deploy_token` call: [5](#0-4) 

NEAR enforces a maximum object size (~4 MB) and maximum function-call argument size. An oversized `name` or `symbol` can cause the cross-contract call to `deploy_token` to fail, or — if the bridge stores any intermediate state containing the metadata (e.g., a pending callback record) — can inflate the bridge contract's own state object, blocking subsequent operations on that storage slot.

---

### Impact Explanation

A malicious EVM token deployer deploys an ERC-20 token whose `name()` or `symbol()` returns a multi-megabyte string. After the `LogMetadata` event is emitted on EVM, any proof submitter (unprivileged) relays the proof to NEAR. The bridge's `log_metadata` handler accepts the oversized strings without rejection. The resulting `deploy_token` cross-contract call carries an argument payload that exceeds NEAR's limits, causing it to fail. If the bridge stores a pending callback record containing the oversized metadata, that storage slot is permanently inflated, and the token can never be successfully deployed on NEAR. Users who have already initiated cross-chain transfers of that token to NEAR cannot receive their funds, constituting a permanent freeze of bridged assets for that token.

**Impact class:** High — permanent freezing / irrecoverable lock of user funds in the bridge flow for the affected token.

---

### Likelihood Explanation

- Any EVM token deployer (unprivileged) can set an arbitrarily long `name()`/`symbol()` return value.
- Any proof submitter (unprivileged) can relay the resulting `LogMetadata` proof to NEAR.
- No privileged access, key compromise, or chain-level attack is required.
- The attack is cheap and deterministic: deploy a token with a large name, emit the event, submit the proof.

---

### Recommendation

Replace the plain `String` fields in `LogMetadataMessage` (and `InitTransferMessage.msg`) with `BoundedString<N>` using appropriate limits, consistent with how `InitTransferMsg` already handles user-supplied strings:

```rust
// near/omni-types/src/prover_result.rs
pub struct LogMetadataMessage {
    pub token_address: OmniAddress,
    pub name: BoundedString<256>,    // e.g. 256 bytes
    pub symbol: BoundedString<32>,   // e.g. 32 bytes
    pub decimals: u8,
    pub emitter_address: OmniAddress,
}
```

`BoundedString`'s `BorshDeserialize` impl already rejects oversized inputs at deserialization time, so this change propagates the guard to the proof-parsing layer with no additional logic.

---

### Proof of Concept

1. Deploy an ERC-20 token on an EVM chain whose `name()` function returns a 3 MB string.
2. Call the bridge's `logMetadata` function on EVM to emit a `LogMetadata` event containing the oversized name.
3. Obtain a proof of that event (via the bridge's normal proof infrastructure).
4. Submit the proof to NEAR's `log_metadata` entry point as an unprivileged proof submitter.
5. The bridge deserializes `LogMetadataMessage` with `name: String` — no length check fires.
6. The bridge attempts to call `deploy_token` with a `BasicMetadata` argument containing the 3 MB name; the cross-contract call fails due to NEAR's argument size limit, or the bridge's pending-callback state is inflated beyond safe bounds.
7. The token can never be deployed on NEAR; any users who initiated transfers of that token to NEAR cannot claim their funds. [1](#0-0) [6](#0-5)

/// UTF-8 string whose byte length is bounded by `MAX`.
///
/// The bound is enforced on construction ([`BoundedString::new`] / [`FromStr`]) and on
/// both JSON and Borsh deserialization. Oversized inputs fail with
/// [`TypesError::StringTooLong`], preventing untrusted callers from passing arbitrarily
/// large strings that would inflate storage and gas costs or grow hashes without bound.
#[derive(Debug, Clone, Default, PartialEq, Eq, Hash, Serialize, BorshSerialize)]
#[serde(transparent)]
pub struct BoundedString<const MAX: usize>(String);

impl<const MAX: usize> BoundedString<MAX> {
    pub const MAX_LEN: usize = MAX;

    pub fn new(s: impl Into<String>) -> Result<Self, TypesError> {
        let s = s.into();
        if s.len() > MAX {
            return Err(TypesError::StringTooLong);
        }
        Ok(Self(s))
    }
```
