Audit Report

## Title
`PeripheryPayments.pay()` consumes residual contract ETH from prior transactions to subsidize the current payer's WETH obligation — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`pay()` reads `address(this).balance` at line 74 to determine how much ETH to wrap toward a WETH payment. This balance includes ETH stranded from prior payable calls, not just the current `msg.value`. A subsequent caller paying WETH can have their `safeTransferFrom` obligation reduced by the residual ETH, causing the prior depositor to permanently lose those funds with no recovery path.

## Finding Description
In `PeripheryPayments.pay()`, the WETH branch reads the total contract ETH balance:

```solidity
uint256 nativeBalance = address(this).balance;   // line 74 — total, not msg.value
```

ETH accumulates when a user calls any of the three `payable` entry points — `addLiquidityExactShares` (lines 56–68, 71–81), `addLiquidityWeighted` (lines 88–116, 123–148), or `multicall` (lines 42–47) — with `msg.value > 0` but the pool's WETH leg is not required for that call. In `metricOmmModifyLiquidityCallback` (lines 172–177), `pay()` is only invoked for tokens with `amountDelta > 0`; if WETH is not needed, the sent ETH is never consumed and stays in the contract.

The `receive()` guard at lines 32–34 only blocks `address.call{value}("")`-style transfers from non-WETH addresses. It does not block ETH arriving via `msg.value` in payable function calls, so residual ETH from legitimate-but-incomplete multicalls is a realistic accumulation source.

**Exploit flow:**
1. User A calls `addLiquidityExactShares{value: 0.5 ether}` on a pool where only token1 is needed. `pay()` is called for token1 only; 0.5 ETH stays in the adder. User A omits `refundETH`.
2. User B calls `addLiquidityExactShares` (no ETH sent) with WETH as token0 and `amount0Delta = 1e18`.
3. In `pay()`: `nativeBalance = 0.5e18`, which is `> 0` but `< 1e18`, so the `else if` branch fires (line 78–81): wraps 0.5 ETH → transfers 0.5e18 WETH to pool → pulls only `0.5e18` WETH from User B via `safeTransferFrom`.
4. Pool receives correct 1e18 WETH. User B's net cost is 0.5e18 WETH. User A's 0.5 ETH is permanently consumed; `refundETH` returns 0.

## Impact Explanation
User A suffers a direct, permanent loss of principal (ETH). The funds cannot be recovered via `refundETH` once consumed by `pay()`. The pool itself is not insolvent — it receives the correct WETH amount — but the payment invariant ("the registered payer must supply the full token amount") is violated: User B pays less than the required amount, subsidized by User A's stranded ETH. This constitutes a direct loss of user principal meeting Sherlock Medium/High thresholds.

## Likelihood Explanation
The precondition requires an innocent user to accidentally leave ETH in the contract, which is realistic: the `multicall` + `refundETH` pattern is easy to omit, and any call to a payable liquidity function on a pool where only token1 is needed silently strands the sent ETH. An attacker monitoring the contract's ETH balance can exploit the window before the victim calls `refundETH`. The `receive()` guard prevents deliberate ETH injection, so an attacker cannot self-seed profitably (net zero), but the victim's accidental deposit is exploitable.

## Recommendation
Track only the ETH sent in the current transaction. Replace `address(this).balance` with a locally captured `msg.value` passed into `pay()`, or snapshot the balance at the top of the outermost payable entry point and thread it through the call chain. This ensures residual ETH from prior transactions is never consumed on behalf of a new payer.

## Proof of Concept
1. Deploy `MetricOmmPoolLiquidityAdder` with a WETH/token1 pool.
2. Call `addLiquidityExactShares{value: 0.5 ether}(pool, owner, salt, token1OnlyDelta, max0, max1, "")` — pool only needs token1, so `pay()` is called for token1 only; 0.5 ETH stays in the adder. Do not call `refundETH`.
3. From a second account (User B), call `addLiquidityExactShares(pool, owner2, salt2, wethDelta, 1e18, max1, "")` with no ETH sent.
4. Observe: `pay()` wraps the 0.5 ETH residual and pulls only 0.5e18 WETH from User B via `transferFrom`.
