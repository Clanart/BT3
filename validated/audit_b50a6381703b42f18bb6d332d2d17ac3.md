### Title
Unrestricted `execute_meta_tx_v0` Syscall Allows Any Contract to Impersonate the OS-Level Privileged Caller and Bypass `__validate__` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall is callable by **any** Cairo 1 contract without restriction. It invokes `__execute__` on an arbitrary attacker-chosen contract address with `caller_address = ORIGIN_ADDRESS (0)` — the privileged top-level OS address — while **skipping `__validate__`** entirely. Account contracts that guard `__execute__` with `get_caller_address() == 0` and rely on `__validate__` for signature verification will execute attacker-crafted calls, resulting in direct loss of funds.

---

### Finding Description

`execute_meta_tx_v0` (lines 286–400 of `syscall_impls.cairo`) is dispatched by `execute_syscalls` for any contract that emits `META_TX_V0_SELECTOR`: [1](#0-0) 

The function accepts a fully user-controlled `MetaTxV0Request` containing `contract_address`, `calldata_start/end`, and `signature_start/end`. The only selector restriction is that the target must be `EXECUTE_ENTRY_POINT_SELECTOR`: [2](#0-1) 

It then constructs a new `ExecutionInfo` with `caller_address = ORIGIN_ADDRESS` (address 0) — the privileged address the OS uses for all top-level transactions: [3](#0-2) 

It then calls `contract_call_helper` directly, which invokes `__execute__` on the target **without calling `__validate__`**: [4](#0-3) 

There is **no allowlist, no caller check, and no restriction** on which contracts may invoke this syscall. Compare this to the standard invoke transaction flow in `execute_transaction_utils.cairo`, where `run_validate` is always called before execution: [5](#0-4) 

The standard account contract pattern checks `get_caller_address() == 0` in `__execute__` to confirm it is a top-level OS-initiated call, and relies on `__validate__` (enforced by the OS in the normal flow) for signature verification. Since `execute_meta_tx_v0` sets `caller_address = ORIGIN_ADDRESS` and skips `__validate__`, this guard is trivially satisfied by any malicious contract.

Additionally, the `new_tx_info` constructed inside `execute_meta_tx_v0` sets `version = 0` and `nonce = 0`: [6](#0-5) 

The `check_and_increment_nonce` function explicitly skips nonce enforcement for version 0: [7](#0-6) 

This means the attack is also **replayable** — the same crafted meta-tx can be submitted repeatedly with no nonce protection at the OS level.

---

### Impact Explanation

An attacker deploys a contract that calls `execute_meta_tx_v0` targeting any victim account, with calldata encoding arbitrary calls (e.g., ERC-20 `transfer` to the attacker). The victim's `__execute__` sees `caller_address == 0` (valid top-level OS call) and executes the crafted multicall without signature verification. This is **Critical — Direct loss of funds**.

---

### Likelihood Explanation

High. The attack requires only deploying a contract and submitting a single invoke transaction. No privileged access, leaked keys, social engineering, or operator collusion is required. The syscall is available to all Cairo 1 contracts with no allowlist or caller restriction. [8](#0-7) 

---

### Recommendation

- **Short term**: Add a caller restriction to `execute_meta_tx_v0` — only allow explicitly authorized relayer contracts to invoke this syscall, enforced at the OS level.
- **Long term**: Enforce that `__validate__` is called and returns `VALID` before `__execute__` is invoked in the meta_tx_v0 path, consistent with the standard transaction security model. Alternatively, do not set `caller_address = ORIGIN_ADDRESS` — use the calling contract's address instead, so account contracts can distinguish meta-tx calls from top-level OS calls and apply their own authorization logic.

---

### Proof of Concept

1. Mallory deploys `MaliciousRelayer`, a Cairo 1 contract that emits a `META_TX_V0_SELECTOR` syscall with:
   - `contract_address` = Alice's account address
   - `calldata` = ABI-encoded multicall: `[transfer(recipient=Mallory, amount=Alice_full_balance)]`
   - `signature` = empty array (or any arbitrary value)
   - `selector` = `EXECUTE_ENTRY_POINT_SELECTOR`

2. Mallory submits an invoke transaction calling `MaliciousRelayer.__execute__`.

3. The OS dispatches `execute_meta_tx_v0` via `execute_syscalls`: [1](#0-0) 

4. `execute_meta_tx_v0` constructs an `ExecutionContext` with `caller_address = ORIGIN_ADDRESS (0)` and calls Alice's `__execute__` directly via `contract_call_helper`, with no call to `__validate__`: [9](#0-8) 

5. Alice's `__execute__` checks `get_caller_address() == 0` → **passes**. It executes the multicall, transferring Alice's tokens to Mallory. `__validate__` was never called; no signature was verified.

6. Because `version = 0` and `nonce = 0`, the attack can be replayed in subsequent blocks until Alice's account is drained.

This is a direct analog to the report's `CallbackModule` vulnerability: a trusted OS-level execution context (`execute_meta_tx_v0`) performs arbitrary calls with user-controlled target and calldata, bypassing the protocol's access control mechanism (`__validate__`).
