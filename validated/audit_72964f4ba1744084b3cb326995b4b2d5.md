### Title
Unsigned cross-binding between `ChunkEndorsementV2` inner and metadata enables endorsement-cache poisoning — (`core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

`ChunkEndorsementV2` carries two independently-signed payloads: `inner` (signed over `chunk_hash`) and `metadata` (signed over `shard_id`, `epoch_id`, `height_created`, `account_id`). Because neither signature commits to the other payload, an attacker who has observed two valid endorsements from the same validator can splice the `inner` of one with the `metadata` of the other. The resulting message passes every signature check in `ChunkEndorsementV2::verify` and in `validate_chunk_endorsement`, yet it carries a `chunk_hash` that is inconsistent with the routing key derived from `metadata`. When this crafted message reaches a block producer's `ChunkEndorsementTracker` first, it occupies the validator's slot in the LRU cache; the subsequent legitimate endorsement is silently dropped by the "already received" early-exit guard, permanently excluding that validator's vote from the chunk's endorsement tally.

### Finding Description

`ChunkEndorsementV2` is defined as:

```rust
pub struct ChunkEndorsementV2 {
    inner: ChunkEndorsementInnerV1,   // chunk_hash + differentiator
    signature: Signature,             // signs inner only
    metadata: ChunkEndorsementMetadata, // shard_id, epoch_id, height_created, account_id
    metadata_signature: Signature,    // signs metadata only
}
``` [1](#0-0) 

The `verify` method checks both signatures independently:

```rust

    let inner    = borsh::to_vec(&self.inner).unwrap();
    let metadata = borsh::to_vec(&self.metadata).unwrap();
    self.signature.verify(&inner, public_key)
        && self.metadata_signature.verify(&metadata, public_key)

There is no check that `inner.chunk_hash` is consistent with `metadata.{shard_id, epoch_id, height_created}`. The routing key used by the entire endorsement pipeline is derived exclusively from `metadata`:

`validate_chunk_endorsement` calls `validate_chunk_endorsement_signature`, which calls `endorsement.verify(validator.public_key())`. Because `verify` only checks the two independent signatures and never cross-checks `inner.chunk_hash` against `metadata`, a crafted message with mismatched fields passes unconditionally. [5](#0-4) 

The cache insertion and the early-exit guard that drops duplicates both key on `chunk_production_key()` (from `metadata`) and `account_id` (from `metadata`):

```rust

### Impact Explanation

An attacker who has observed two valid endorsements from the same validator V — one for chunk C_A and one for chunk C_B — can construct a crafted endorsement by combining `inner` (and its `signature`) from C_A with `metadata` (and its `metadata_signature`) from C_B. Both signatures verify against V's public key. The crafted message is accepted by `validate_chunk_endorsement` and stored in the block producer's LRU cache under the key for C_B with `chunk_hash = H_A`. When the legitimate endorsement for C_B subsequently arrives (from V or any relay), the early-exit guard drops it. V's vote is permanently absent from C_B's endorsement tally. If the attacker repeats this for enough validators, the chunk falls below the endorsement threshold and cannot be included in a block, stalling chunk production for that shard.

### Likelihood Explanation

Endorsements are broadcast to all peers on the network. Any peer that has observed two endorsements from the same validator (for any two chunks that validator is assigned to) can mount this attack. The only timing requirement is that the crafted message reaches the target block producer before the legitimate endorsement. This is achievable by a well-connected adversary. No privileged role is required.

### Recommendation

Bind `inner` and `metadata` together in a single signed payload, or include `inner.chunk_hash` inside `metadata` (or vice versa) so that a single signature covers both. For example, sign `(inner, metadata)` jointly:

```rust
fn verify(&self, public_key: &PublicKey) -> bool {
    // Verify metadata signature as before
    let metadata_bytes = borsh::to_vec(&self.metadata).unwrap();
    if !self.metadata_signature.verify(&metadata_bytes, public_key) {
        return false;
    }
    // Verify inner signature, but also assert chunk_hash is consistent
    // with what the metadata implies (derive expected chunk_hash from metadata
    // and compare), OR sign (inner || metadata) jointly.
    let inner_bytes = borsh::to_vec(&self.inner).unwrap();
    self.signature.verify(&inner_bytes, public_key)
        && self.inner.chunk_hash == expected_chunk_hash_from(&self.metadata)
}
```

Alternatively, add `chunk_hash` to `ChunkEndorsementMetadata` and verify equality with `inner.chunk_hash` inside `verify`, so that any mismatch is caught before the message is cached.

### Proof of Concept

1. Observe two valid `ChunkEndorsement::V2` messages from validator V on the network: `E_A` (for chunk C_A, `chunk_hash = H_A`, `metadata = {shard=0, epoch=E1, height=100}`) and `E_B` (for chunk C_B, `chunk_hash = H_B`, `metadata = {shard=1, epoch=E2, height=200}`).
2. Construct crafted endorsement `E_crafted`:
   - `inner = E_A.inner` (chunk_hash = H_A), `signature = E_A.signature`
   - `metadata = E_B.metadata` (shard=1, epoch=E2, height=200), `metadata_signature = E_B.metadata_signature`
3. Send `E_crafted` to the block producer for shard 1 before `E_B` arrives.
4. `validate_chunk_endorsement` accepts `E_crafted`: both signatures verify against V's public key; `validate_chunk_relevant_as_validator` passes because V is a valid validator for (shard=1, epoch=E2, height=200).
5. Cache stores `(H_A, sig_A)` under key `(shard=1, epoch=E2, height=200)` for account V.
6. When `E_B` arrives, the early-exit guard fires (`entry.contains_key(V)` is true) and `E_B` is dropped.
7. `collect_chunk_endorsements` for C_B filters out the cached entry because `H_A ≠ H_B`; V's endorsement is absent from the tally.
