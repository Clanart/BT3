### Title
Incorrect `locked_tokens` Accounting in `fast_fin_transfer_to_other_chain` Causes Permanent Fund Freezing — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `fast_fin_transfer_to_other_chain`, only `amount_without_fee` is added to `locked_tokens`, but `claim_fee_callback` later subtracts the full `fee` from the same counter. This leaves `locked_tokens` understated by `fee` relative to the actual tokens committed to the destination chain. When a user subsequently tries to return those tokens from the destination chain back to NEAR, the `unlock_tokens_if_needed` call panics with `InsufficientLockedTokens`, permanently freezing the user's funds.

---

### Finding Description

**Vulnerability class:** Asset-accounting bug — gross balance used instead of correctly-encumbered balance, analogous to the `MultiRewardStaking.changeRewardSpeed` bug where the contract's token balance (including already-owed rewards) was used to compute remaining reward time.

**Root cause — inconsistent locking between the two transfer paths:**

**Path 1 — normal `init_transfer_internal`** locks the *full* amount (including fee):

```rust
// near/omni-bridge/src/lib.rs  ~line 1853
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,   // ← full amount, fee included
);
``` [1](#0-0) 

**Path 2 — `fast_fin_transfer_to_other_chain`** locks only `amount_without_fee`:

```rust
// near/omni-bridge/src/lib.rs  ~line 932
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,           // ← net amount, fee excluded
);
``` [2](#0-1) 

**`claim_fee_callback` always unlocks `fee` from `locked_tokens`** (confirmed by the test `test_claim_fee_decreases_locked_tokens_for_non_deployed_token`):

```rust
// near/omni-bridge/src/lib.rs  ~line 1131
let fee = transfer_message.amount.0 - denormalized_amount;
self.send_fee_internal(&transfer_message, fee_recipient, fee)
``` [3](#0-2) 

**Accounting after `claim_fee_callback`:**

| Path | Locked after init | Unlocked by claim_fee | Remaining in counter | Actual tokens on dest chain |
|---|---|---|---|---|
| Normal `init_transfer` | `amount` | `fee` | `amount − fee` = `amount_without_fee` ✓ | `amount_without_fee` |
| `fast_fin_transfer_to_other_chain` | `amount_without_fee` | `fee` | `amount_without_fee − fee` ✗ | `amount_without_fee` |

The counter is understated by exactly `fee` for every fast transfer to another chain.

**Return path failure:**

When a user sends tokens back from the destination chain to NEAR, `fin_transfer_callback` → `process_fin_transfer_to_near` calls `unlock_tokens_if_needed` with the full `amount_without_fee`. The `unlock_tokens` function enforces:

```rust
// near/omni-bridge/src/token_lock.rs  line 81
require!(
    available >= amount,
    TokenLockError::InsufficientLockedTokens.as_ref()
);
``` [4](#0-3) 

Because `available = amount_without_fee − fee < amount_without_fee = amount`, this panics and the transfer is permanently rejected. The test `test_fin_transfer_callback_near_success` confirms that `fin_transfer_callback` does unlock the full amount from `locked_tokens`: [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of user funds.** Any user who receives tokens on a non-NEAR destination chain (e.g., Solana, EVM) via a fast transfer and later attempts to bridge them back to NEAR will have their return transfer permanently rejected with `InsufficientLockedTokens`. The tokens are irrecoverably stranded on the destination chain because the NEAR bridge's `locked_tokens` counter is too low to permit the unlock. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**High.** The fast transfer path (`fast_fin_transfer_to_other_chain`) is a core, publicly-accessible feature. Any relayer performing a fast transfer to a non-NEAR chain triggers the undercount. The fee is a mandatory, non-zero parameter of every fast transfer. The bug is deterministic: every fast transfer to another chain followed by a `claim_fee_callback` leaves `locked_tokens` understated by exactly `fee`. Any subsequent return transfer by any user who received tokens via that fast transfer will fail.

---

### Recommendation

In `fast_fin_transfer_to_other_chain`, lock the full `fast_transfer.amount.0` (including fee) instead of `amount_without_fee`, to match the accounting in `init_transfer_internal`:

```rust
// Change:
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,
);

// To:
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    fast_transfer.amount.0,   // full amount, consistent with init_transfer_internal
);
```

This ensures that after `claim_fee_callback` subtracts `fee`, the remaining counter equals `amount_without_fee`, which correctly represents the tokens committed to the destination chain.

---

### Proof of Concept

**Setup:** ETH-origin token deployed on NEAR. Relayer performs a fast transfer: ETH → NEAR → SOL with `amount = 1000`, `fee = 100`, so `amount_without_fee = 900`.

1. `fast_fin_transfer_to_other_chain` is called.
   - `locked_tokens[(Sol, eth_token)]` += 900 → **counter = 900**
2. SOL finalizes the transfer (mints 900 tokens to recipient).
3. Relayer calls `claim_fee_callback` on NEAR.
   - `fee = transfer_message.amount (1000) − denormalized_amount (900) = 100`
   - `send_fee_internal` calls `unlock_tokens(Sol, eth_token, 100)`
   - **counter = 900 − 100 = 800** ← understated; should be 900
4. Recipient on SOL tries to bridge 900 tokens back to NEAR.
   - `fin_transfer_callback` → `process_fin_transfer_to_near`
   - `unlock_tokens_if_needed(Sol, eth_token, 900)` is called
   - `available (800) < amount (900)` → **panic: `ERR_INSUFFICIENT_LOCKED_TOKENS`**
   - Transfer permanently rejected. User's 900 tokens are frozen on SOL.
