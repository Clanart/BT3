### Title
Missing `shard_id` Binding Check in `set_state_header` Allows Cross-Shard State Corruption During State Sync - (File: chain/chain/src/state_sync/adapter.rs)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a caller-supplied `shard_id` parameter and a `ShardStateSyncResponseHeader` payload, but never verifies that the chunk embedded inside the header actually belongs to the supplied `shard_id`. A malicious peer can respond to a state-sync header request for shard X with a legitimately-signed, merkle-proven header for shard Y. All existing checks pass, and the header is stored under `StateHeaderKey(X, sync_hash)`. Subsequent `set_state_part` calls then validate parts against shard Y's state root and apply shard Y's trie data into shard X's storage, permanently corrupting the syncing node's state.

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` performs five validation steps before persisting the header:

1. `validate_chunk_proofs` — validates the chunk's internal merkle proofs (tx root, receipt root, etc.)
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — proves the chunk hash is somewhere in the block's chunk-headers merkle tree
3. Prev-chunk inclusion proof
4. Incoming-receipts proof chain
5. `validate_state_root_node` against `chunk_inner.prev_state_root()`

None of these steps check that `chunk.shard_id() == shard_id`. [1](#0-0) 

The merkle proof at step 2 proves the chunk is *somewhere* in the block's chunk-headers tree, but the tree contains one entry per shard. The proof path encodes the position of shard Y, not shard X. The code never extracts or compares that position against the `shard_id` argument. [2](#0-1) 

After all checks pass, the header is stored under the key `StateHeaderKey(shard_id, sync_hash)` — using the attacker-controlled `shard_id`, not the shard_id embedded in the chunk: [3](#0-2) 

`set_state_part` then retrieves this poisoned header, extracts `prev_state_root` from shard Y's chunk, and validates all incoming parts against shard Y's trie root: [4](#0-3) 

`apply_state_part` subsequently writes shard Y's trie nodes into shard X's `ShardUId` storage slot, completing the corruption: [5](#0-4) 

### Impact Explanation

After `set_state_finalize` completes, the syncing node holds shard Y's state under shard X's `ShardUId`. When the node later produces or validates chunks for shard X, it reads from the wrong trie root. Produced chunks will have an incorrect `prev_state_root`, causing them to be rejected by all honest validators. If the node is a chunk producer for shard X, it will be unable to produce valid chunks and will lose rewards or be slashed. The corruption is persistent (written to RocksDB) and survives restarts.

### Likelihood Explanation

State sync downloads headers from any reachable peer — there is no restriction to trusted validators. The `StateRequestActor` only validates that `sync_hash` belongs to a known recent epoch; it does not validate the `shard_id` in the response against the chunk inside the payload: [6](#0-5) 

An attacker needs only to: (a) be a reachable peer, (b) possess a legitimately-signed `ShardStateSyncResponseHeader` for any shard Y in the same epoch (trivially obtained by syncing normally), and (c) respond to the victim's `StateRequestHeader` for shard X with that header. No privileged role is required.

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk from the header in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This check should be placed before `validate_chunk_proofs` at line 380 of `chain/chain/src/state_sync/adapter.rs`. [7](#0-6) 

### Proof of Concept

1. Honest node A is syncing and requests `StateRequestHeader { shard_id: 0, sync_hash }`.
2. Malicious peer M has a valid `ShardStateSyncResponseHeader` for shard 1 (obtained by syncing normally). The header contains shard 1's chunk with a valid merkle proof placing it in the block.
3. M responds with this shard-1 header to A's shard-0 request.
4. A calls `set_state_header(shard_id=0, sync_hash, header_for_shard_1)`.
5. `validate_chunk_proofs` passes — shard 1's chunk is internally valid.
6. `verify_path` passes — shard 1's chunk hash IS in the block's `chunk_headers_root`.
7. No check fires for `chunk.shard_id() (=1) != shard_id (=0)`.
8. Header stored under `StateHeaderKey(0, sync_hash)` containing shard 1's `prev_state_root`.
9. A downloads and validates state parts for "shard 0" — they validate against shard 1's state root and pass.
10. `set_state_finalize` applies shard 1's trie into shard 0's `ShardUId` storage.
11. Node A now has shard 1's state installed as shard 0's state. All subsequent chunk production and validation for shard 0 fails.

    fn apply_state_part(

        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,

        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);

        self.precompile_contracts(epoch_id, contract_codes)?;

impl Handler<StateRequestHeader, Option<StatePartOrHeader>> for StateRequestActor {
    fn handle(&mut self, msg: StateRequestHeader) -> Option<StatePartOrHeader> {
        let StateRequestHeader { shard_id, sync_hash } = msg;
        let _timer = metrics::STATE_SYNC_REQUEST_TIME
            .with_label_values(&["StateRequestHeader"])
            .start_timer();
        let _span =
            tracing::debug_span!(target: "sync", "StateRequestHeader", ?shard_id, ?sync_hash)
                .entered();

        if self.throttle_state_sync_request() {
            tracing::debug!(target: "sync", "throttling state sync request for shard");
            metrics::STATE_SYNC_REQUESTS_THROTTLED_TOTAL.inc();
            return None;
        }

        if self.validate_sync_hash(&sync_hash) == SyncHashValidationResult::Rejected {
            metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL
                .with_label_values(&["header", "failed"])
                .inc();
            return None;
        }

        let header = self.state_sync_adapter.get_state_response_header(shard_id, sync_hash);
        let Ok(header) = header else {
            tracing::warn!(target: "sync", "cannot build state sync header");
            metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL
                .with_label_values(&["header", "failed"])
                .inc();
            return Some(new_header_response_empty(shard_id, sync_hash));
        };
        let ShardStateSyncResponseHeader::V2(header) = header else {
            tracing::warn!(target: "sync", "invalid state sync header format");
            metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL
                .with_label_values(&["header", "failed"])
                .inc();
            return None;
        };

        metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL.with_label_values(&["header", "success"]).inc();
        let response = new_header_response(shard_id, sync_hash, header);
        Some(response)
    }
