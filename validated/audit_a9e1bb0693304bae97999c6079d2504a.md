Audit Report

## Title
Router `pay()` Consumes Unattributed Native ETH to Settle Any Caller's WETH Swap, Enabling Theft of Stranded ETH — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay()` function in `PeripheryPayments.sol` reads `address(this).balance` as a shared, unattributed pool when settling WETH-input swaps. Any ETH stranded on the router from a prior user's excess `msg.value` (e.g., a `multicall` that omits `refundETH`) is silently consumed by the next caller's swap. The original depositor permanently loses their ETH; the attacker receives output tokens without spending any WETH or ETH of their own.

## Finding Description
`pay()` (lines 69–88 of `PeripheryPayments.sol`) handles WETH-input swaps by first checking the router's raw native balance:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

There is no check that the ETH on the router was deposited by the current caller. When `nativeBalance >= value`, the named `payer` is never touched.

ETH reaches the router legitimately via `msg.value` on any payable entry point. `multicall` (line 39) is `payable` and executes sub-calls via `delegatecall`, so a user can send excess ETH that is only partially consumed by their swap. The `receive()` guard (lines 32–34) only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from being stranded by a user who omits `refundETH`. `refundETH()` (lines 58–63) is a separate, optional call — nothing in the router enforces its inclusion.

`_justPayCallback` (lines 192–199) calls `pay()` with `_getPayer()` set to the original `msg.sender`, but `pay()` bypasses the payer entirely whenever native balance is sufficient.

## Impact Explanation
Direct loss of user principal (ETH). When the router's native balance covers the full swap amount, the named payer is never debited — the router pays entirely from its own (victim's) balance. The attacker receives output tokens without spending any WETH or ETH. The victim's stranded ETH is permanently consumed by the pool as WETH payment for a stranger's trade. This is a direct loss of user principal meeting Sherlock High/Medium thresholds.

## Likelihood Explanation
Medium. The precondition — ETH stranded on the router — occurs whenever a user sends excess `msg.value` in a `multicall` and omits `refundETH`, a realistic and documented usage pattern. Integrators building on top of the router are especially likely to omit the cleanup step. Once ETH is stranded, any observer can drain it in the next block with a single `exactInputSingle` call targeting a WETH pool, requiring zero WETH balance and zero WETH allowance.

## Recommendation
Track the ETH belonging to the current execution context rather than reading the raw contract balance. One approach: record `msg.value` in transient storage at each payable entry point and deduct from that tracked amount inside `pay()`, reverting if the tracked amount is exhausted. Alternatively, remove the native-ETH branch from `pay()` entirely and require callers to pre-wrap ETH to WETH before calling the router, relying solely on `safeTransferFrom`. Either change eliminates the unattributed-balance race.

## Proof of Concept
1. **Alice** calls `router.multicall{value: 2 ether}([exactInputSingle(WETH→token1, amountIn=1 ether, amountOutMinimum=0, ...)])` — she forgets to append `refundETH`. The swap consumes 1 ETH; 1 ETH remains on the router.
2. **Bob** (attacker, zero WETH balance, zero WETH allowance) calls `router.exactInputSingle(ExactInputSingleParams{pool: wethToken1Pool, tokenIn: WETH, amountIn: 1 ether, amountOutMinimum: 0, ...})`.
3. Inside `metricOmmSwapCallback`, `_justPayCallback` calls `pay(WETH, Bob, pool, 1 ether)`.
4. `address(this).balance == 1 ether >= 1 ether` → router wraps Alice's ETH into WETH and transfers it to the pool. `safeTransferFrom(Bob, ...)` is never called.
5. Bob receives token1 output. Alice's 1 ETH is gone. Bob spent nothing.
