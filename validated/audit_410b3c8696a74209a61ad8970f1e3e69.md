### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay Leading to Direct Loss of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the signed hash for the `meta_tx_v0` syscall without including any nonce or replay-protection field. Because the hash is purely a function of `(prefix, version=0, contract_address, entry_point_selector, calldata, max_fee=0, chain_id)`, a valid signature produced for one execution remains valid forever. Any party who observes the signature can replay the same meta-transaction an unlimited number of times, draining funds or corrupting state.

---

### Finding Description

In `transaction_hash.cairo` lines 295–315, `compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
```

Compare this to `compute_l1_handler_transaction_hash` (lines 220–238), which passes `additional_data_size=1, additional_data=&nonce` — the nonce is the sole replay-protection mechanism in the `deprecated_get_transaction_hash` scheme. The `meta_tx_v0` variant omits it entirely.

The `deprecated_get_transaction_hash` function (lines 68–100) hashes only the fields it receives; it has no internal nonce logic. The `additional_data` array is the designated slot for replay-protection data (nonce) in this hash scheme, and it is left empty for `meta_tx_v0`.

The vulnerability class is identical to the reported `IdRegistry` bug: a signed commitment is constructed without binding it to the current state (nonce), so the signature remains valid across state transitions and can be replayed.

---

### Impact Explanation

The `meta_tx_v0` syscall allows a contract to execute an arbitrary entry point on behalf of an account, authenticated by a signature over the hash. Because the hash does not include a nonce:

1. A victim account signs a `meta_tx_v0` authorizing a token transfer (or any state-mutating call).
2. The transaction executes once legitimately.
3. An attacker (any unprivileged party who observed the original transaction on-chain) re-submits the same `(contract_address, entry_point_selector, calldata, signature)` tuple in a new transaction.
4. The OS recomputes the identical hash and accepts the signature as valid.
5. The operation executes again — funds are transferred again.

This loop can be repeated until the victim's balance is exhausted. **Impact: Direct loss of funds (Critical).**

---

### Likelihood Explanation

- All data needed to replay (contract address, selector, calldata, signature) is publicly visible on-chain after the first execution.
- No privileged access, leaked key, or social engineering is required.
- The attacker only needs to submit a standard transaction invoking the `meta_tx_v0` syscall with the observed parameters.
- The OS Cairo program enforces no nonce check for this hash path, so the replay succeeds deterministically.

Likelihood is **High**: any on-chain observer can exploit this immediately after the first legitimate use of a `meta_tx_v0` signature.

---

### Recommendation

Include the account's current nonce (or a monotonically increasing counter specific to `meta_tx_v0`) in the hash preimage, exactly as `compute_l1_handler_transaction_hash` does:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,           // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← bind hash to current nonce
    );
    return tx_hash;
}
```

This mirrors the fix applied in the `IdRegistry` report: binding the signed commitment to the current state so that any state change (nonce increment) automatically invalidates prior signatures.

func deprecated_get_transaction_hash{hash_ptr: HashBuiltin*}(
    tx_hash_prefix: felt,
    version: felt,
    contract_address: felt,
    entry_point_selector: felt,
    calldata_size: felt,
    calldata: felt*,
    max_fee: felt,
    chain_id: felt,
    additional_data_size: felt,
    additional_data: felt*,
) -> (tx_hash: felt) {
    let (hash_state_ptr) = hash_init();
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=tx_hash_prefix);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=version);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=contract_address);
    let (hash_state_ptr) = hash_update_single(
        hash_state_ptr=hash_state_ptr, item=entry_point_selector
    );
    let (hash_state_ptr) = hash_update_with_hashchain(
        hash_state_ptr=hash_state_ptr, data_ptr=calldata, data_length=calldata_size
    );
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=max_fee);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=chain_id);

    let (hash_state_ptr) = hash_update(
        hash_state_ptr=hash_state_ptr, data_ptr=additional_data, data_length=additional_data_size
    );

    let (tx_hash) = hash_finalize(hash_state_ptr=hash_state_ptr);

    return (tx_hash=tx_hash);
