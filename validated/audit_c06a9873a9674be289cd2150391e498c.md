### Title
Fee-on-Transfer Token Accounting Mismatch Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `initTransfer` function records the caller-supplied `amount` in the emitted `InitTransfer` event without verifying the actual tokens received by the contract. For fee-on-transfer (deflationary) ERC-20 tokens, the contract receives `amount - fee` but the event — and therefore the cross-chain proof — records `amount`. The NEAR bridge reads this proof and releases the full `amount` on the destination chain, permanently undercollateralizing the bridge.

---

### Finding Description

In `initTransfer`, when the token is a plain ERC-20 (not a bridge token or custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← user-supplied parameter
);
``` [1](#0-0) 

Immediately after, without any balance reconciliation, the function emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← same user-supplied parameter, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The `InitTransfer` event definition confirms `amount` is a plain `uint128` field taken directly from the call parameter: [3](#0-2) 

The same flaw is present in the Wormhole variant. `OmniBridgeWormhole.initTransferExtension` encodes the passed-in `amount` directly into the Wormhole message payload: [4](#0-3) 

On the NEAR side, `fin_transfer_callback` decodes the proof and constructs a `TransferMessage` using `init_transfer.amount` — the value that came from the EVM event — without any independent verification of what was actually locked:

```rust
let transfer_message = TransferMessage {
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [5](#0-4) 

This `TransferMessage` is then used to release tokens to the recipient on the destination chain. The bridge has no mechanism to detect or correct the discrepancy between what was locked on EVM and what is released on NEAR.

---

### Impact Explanation

For every `initTransfer` call with a fee-on-transfer token:

- EVM bridge locks: `amount - transfer_fee`
- NEAR bridge releases: `amount`

The shortfall (`transfer_fee`) is a permanent loss of collateral from the bridge's EVM reserves. An attacker can exploit this repeatedly to drain the bridge's holdings of any fee-on-transfer token that has been whitelisted. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

Several widely-used tokens implement transfer fees (e.g., STA, PAXG, tokens with upgradeable fee logic). The bridge accepts arbitrary ERC-20 tokens for the non-bridge-token, non-custom-minter path. Any unprivileged user who holds such a token and calls `initTransfer` triggers the vulnerability. No special access is required; the entry point is fully public.

---

### Recommendation

Measure the actual balance change around the `safeTransferFrom` call and use that as the canonical bridged amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(received == amount, "Fee-on-transfer token not supported");
```

Alternatively, explicitly document and enforce that only standard (non-fee) ERC-20 tokens may be bridged, and add an on-chain allowlist check before accepting a token in `initTransfer`.

---

### Proof of Concept

1. A fee-on-transfer token `FEE` (5% fee on every transfer) is registered with the Omni Bridge on EVM.
2. Attacker calls `initTransfer(FEE, 1000, 0, 0, "near:attacker.near", "")`.
3. `safeTransferFrom` moves `1000` tokens from attacker; the token contract deducts 5%, so the bridge receives `950`.
4. `InitTransfer` event is emitted with `amount = 1000`.
5. A relayer submits the proof to NEAR's `fin_transfer`.
6. `fin_transfer_callback` reads `init_transfer.amount = 1000` from the proof and creates a `TransferMessage` for `1000` tokens.
7. NEAR releases `1000` tokens to `attacker.near`.
8. Bridge is short `50` tokens of `FEE` per iteration; repeating drains the bridge's EVM-side reserves entirely. [6](#0-5)
