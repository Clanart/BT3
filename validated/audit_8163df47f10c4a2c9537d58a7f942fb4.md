Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal excess ETH stranded on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` is an external, access-control-free function that unconditionally transfers the router's entire ETH balance to `msg.sender`. Because `pay()` deposits only the exact swap amount when native ETH is used as WETH input, any excess `msg.value` sent to a payable swap function (e.g., `exactInputSingle{value: X}` with `amountIn < X`) is left on the router with no attribution and is immediately claimable by any subsequent caller.

## Finding Description

`refundETH()` at L58–63 of `PeripheryPayments.sol` performs no ownership or context check:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

`exactInputSingle` (L67 of `MetricOmmSimpleRouter.sol`) is `external payable`, so it accepts arbitrary `msg.value`. Inside the swap callback, `pay()` is invoked with `value = amountIn`. At L73–77 of `PeripheryPayments.sol`, when `token == WETH` and `nativeBalance >= value`, only exactly `value` is deposited as WETH:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
```

The remainder (`msg.value - amountIn`) stays on the router. The `receive()` guard at L32–34 only blocks plain ETH transfers (no calldata) from non-WETH addresses; it does not prevent ETH from being attached to payable function calls. Any caller in a subsequent transaction can then invoke `refundETH()` and receive the full stranded balance. The test `test_refundETH_sendsBalanceToCaller` at L74–85 of `MetricOmmSimpleRouter.payments.t.sol` directly confirms this: it pre-loads the router with ETH via `vm.deal` and shows any arbitrary caller receives it all.

## Impact Explanation

Direct loss of user ETH principal. A user who sends `msg.value > amountIn` to `exactInputSingle` (or any other payable swap entry point) without appending `refundETH` in the same multicall permanently loses the excess ETH to any attacker who calls `refundETH()` in the next transaction. There is no minimum threshold — the attacker receives the full stranded balance. This meets the Critical/High direct loss of user principal threshold under the allowed impact gate.

## Likelihood Explanation

The documented and tested safe pattern (appending `refundETH` as the last call in a multicall) is not enforced at the contract level. Users calling `exactInputSingle{value}` directly without a multicall wrapper, or users who forget to append `refundETH`, will strand ETH. MEV bots monitoring mempool or block state can trivially detect a non-zero router ETH balance and call `refundETH()` atomically in the next block. The condition is reachable by any unprivileged caller with no special setup.

## Recommendation

Restrict `refundETH()` so it can only be called within a `multicall` context (e.g., enforce that execution is via `delegatecall` from `multicall`), or record the original `msg.sender` of the outermost `multicall` in transient storage and require `msg.sender == storedCaller` inside `refundETH()`. Alternatively, accept a `recipient` parameter validated against a transient payer context, mirroring how `unwrapWETH9` accepts a `recipient` but is called within the same atomic multicall.

## Proof of Concept

```
1. User calls router.exactInputSingle{value: 1 ether}(
       ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ether, ...
       })
   );
   // pay() deposits exactly 0.5 ETH as WETH → pool
   // 0.5 ETH remains on router (nativeBalance - value)

2. Attacker (separate tx) calls router.refundETH();
   // balance = 0.5 ETH → _transferETH(attacker, 0.5 ETH)
   // Attacker receives 0.5 ETH; user's excess is permanently lost.
```

Confirmed by `test_refundETH_sendsBalanceToCaller` (`MetricOmmSimpleRouter.payments.t.sol` L74–85) and the safe-pattern test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` (`MetricOmmSimpleRouter.native.t.sol` L106–133), which shows the only protection is the user voluntarily including `refundETH` in the same multicall.
