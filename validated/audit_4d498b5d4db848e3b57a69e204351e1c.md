### Title
ETH Transfer via Low-Level `call` to Non-Existent Recipient Silently Succeeds, Permanently Locking Bridged Funds — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.sol`, the `finTransfer` function delivers native ETH to `payload.recipient` via a bare low-level `call`. Per the EVM/Solidity specification, a low-level `call` returns `true` as its first return value when the target account is non-existent. The existing `if (!success) revert FailedToSendEther()` guard therefore cannot detect a delivery to a non-existent or self-destructed contract address. Because the destination nonce is irrevocably consumed *before* the ETH transfer, any ETH sent to such an address is permanently locked with no recovery path.

---

### Finding Description

In `finTransfer`, the destination nonce is marked used unconditionally before any asset movement:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287 — nonce consumed
```

Then, for native-ETH transfers (`payload.tokenAddress == address(0)`):

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();              // lines 319-322
```

The Solidity documentation states: *"The low-level functions `call`, `delegatecall` and `staticcall` return `true` as their first return value if the account called is non-existent, as part of the design of the EVM. Account existence must be checked prior to calling if needed."*

Consequently, if `payload.recipient` is:
- a contract address whose code was removed via `selfdestruct` (pre-EIP-6780 chains, or same-transaction destruction), or
- any address at which no code exists and no private key is known to the recipient,

the `call` returns `success = true`, the transaction completes, the nonce is permanently consumed, and the ETH is deposited at an address from which it can never be retrieved. [1](#0-0) 

---

### Impact Explanation

The destination nonce is marked used at line 287 before the ETH transfer executes. Once the transaction succeeds with `success = true` against a non-existent recipient, the nonce entry in `completedTransfers` is permanently set, making any retry impossible. The bridged ETH is irrecoverably locked at the dead address. This satisfies the allowed critical impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."* [2](#0-1) 

---

### Likelihood Explanation

A realistic trigger: a user whose EVM-side recipient is a contract wallet (e.g., a Gnosis Safe or account-abstraction wallet) initiates a NEAR→EVM ETH bridge transfer. If that contract wallet is subsequently self-destructed (security incident, upgrade migration, or user error) before the relayer finalises the transfer, `finTransfer` will silently succeed and the ETH will be locked. The bridge supports multiple EVM chains; chains that have not yet adopted EIP-6780 (which restricts `selfdestruct` to same-transaction use) remain fully exposed. The `payload.recipient` field is entirely user-controlled and is not validated for code existence anywhere in the contract.

---

### Recommendation

Before executing the low-level ETH send, verify that the recipient address either (a) has deployed code (`payload.recipient.code.length > 0`) or (b) is explicitly expected to be an EOA. A minimal fix:

```solidity
if (payload.tokenAddress == address(0)) {
    // Reject delivery to addresses with no code that are not plain EOAs
    // (i.e., self-destructed contracts whose code has been wiped)
    require(
        payload.recipient.code.length == 0 || /* is known EOA */ true,
        "FailedToSendEther: dead contract"
    );
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

A cleaner alternative is to use OpenZeppelin's `Address.sendValue`, which internally checks `address(this).balance >= amount` and reverts on failure, combined with an explicit `extcodesize` guard for contract recipients. Additionally, consider moving the nonce-consumption (`completedTransfers[payload.destinationNonce] = true`) to *after* the asset transfer succeeds, so that a failed delivery does not permanently consume the nonce.

---

### Proof of Concept

1. User holds a Gnosis Safe at `0xDEAD…` on an EVM chain supported by the bridge.
2. User initiates a NEAR→EVM transfer of native ETH, specifying `0xDEAD…` as `recipient` and `address(0)` as `tokenAddress`.
3. The Gnosis Safe at `0xDEAD…` is self-destructed (e.g., during a security incident) before the relayer submits `finTransfer`.
4. Relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
5. Line 287: `completedTransfers[payload.destinationNonce] = true` — nonce permanently consumed.
6. Line 319: `payload.recipient.call{value: payload.amount}("")` — target has no code; EVM returns `success = true`.
7. Line 322: `if (!success) revert FailedToSendEther()` — guard is not triggered.
8. Transaction succeeds. ETH is deposited at `0xDEAD…` (no code). No withdrawal is possible. Nonce is consumed. User's funds are permanently lost. [3](#0-2)
