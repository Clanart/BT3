### Title
Unregistered ERC20 Tokens Accepted in `initTransfer()` Without Registry Validation, Causing Permanent Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer()` accepts any arbitrary ERC20 token in its `else` branch without verifying the token is registered in the NEAR bridge's token registry. Tokens deposited for unregistered tokens are permanently locked in the EVM bridge because the NEAR `fin_transfer_callback` unconditionally panics with `TokenDecimalsNotFound` when it cannot find the token's decimal metadata, and no EVM-side refund or cancellation path exists.

---

### Finding Description

In `OmniBridge.initTransfer()`, the token-handling logic branches as follows: [1](#0-0) 

When `tokenAddress` is neither a `customMinters` entry nor an `isBridgeToken` entry, the contract falls through to the final `else` branch and unconditionally pulls the tokens from the caller into the bridge contract:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

There is no check that `tokenAddress` has ever been registered on the NEAR side (i.e., that `ethToNearToken[tokenAddress]` is non-empty, or that a `token_decimals` entry exists on NEAR for the corresponding `OmniAddress`). The function then emits `InitTransfer` and returns normally. [2](#0-1) 

On the NEAR side, when a relayer submits the proof for this transfer, `fin_transfer_callback` attempts to look up the token's decimal configuration: [3](#0-2) 

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

If the token was never registered via `log_metadata` + MPC signing on NEAR, this call panics unconditionally. The NEAR transaction reverts, but the EVM-side token transfer has already been committed. The EVM bridge contract has no `cancel`, `rescue`, or `refund` function that would allow recovery of tokens locked for a failed cross-chain transfer. The tokens are permanently stranded.

The identical pattern exists in StarkNet's `init_transfer()`: [4](#0-3) 

```cairo
} else {
    let success = IERC20Dispatcher { contract_address: token_address }
        .transfer_from(caller, get_contract_address(), amount.into());
    assert(success, 'ERR_TRANSFER_FROM_FAILED');
}
```

No check is made that `token_address` maps to a known NEAR token before accepting the deposit.

---

### Impact Explanation

Any user who calls `initTransfer` (or StarkNet `init_transfer`) with an ERC20 token that has not been registered on the NEAR bridge via the `log_metadata` → MPC-sign → `bind_token` flow will have their tokens permanently locked in the bridge contract. There is no cross-chain refund path: the EVM contract has no admin rescue function, and the NEAR side will always panic before it can issue a `sign_transfer` that would allow the EVM side to release funds via `finTransfer`. This constitutes **permanent freezing / irrecoverable lock of user funds in a bridge vault flow**, matching the Critical impact tier.

---

### Likelihood Explanation

The `initTransfer` function is fully public and callable by any user. A realistic scenario: a token deployer deploys a new ERC20 on EVM and calls `initTransfer` before completing the `log_metadata` registration on NEAR (e.g., due to UI confusion, front-running, or partial setup). The protocol provides no guard — no revert, no warning — at the EVM entry point. Likelihood is **Medium**: it requires a user to act on an unregistered token, which is a plausible mistake given the multi-step registration flow.

---

### Recommendation

Add a validation in `initTransfer()` (and StarkNet `init_transfer()`) that the token is registered before accepting the deposit. For the native-token path (the `else` branch), require that `ethToNearToken[tokenAddress]` is non-empty:

```solidity
} else {
    require(
        bytes(ethToNearToken[tokenAddress]).length > 0,
        "ERR_TOKEN_NOT_REGISTERED"
    );
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Apply the equivalent guard in StarkNet's `init_transfer` by asserting `starknet_to_near_token.read(token_address).len() > 0` before accepting the `transfer_from`.

---

### Proof of Concept

1. Deploy a new ERC20 token `T` on EVM. Do **not** call `log_metadata` / complete NEAR-side registration.
2. Approve the `OmniBridge` contract to spend `T`.
3. Call `OmniBridge.initTransfer(T, amount, 0, 0, "near-recipient.near", "")`.
   - The `else` branch executes: `T.safeTransferFrom(msg.sender, bridge, amount)` succeeds.
   - `InitTransfer` event is emitted with `tokenAddress = T`.
4. A relayer submits the proof to NEAR `fin_transfer`.
5. `fin_transfer_callback` reaches `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` — panics because `T` has no `token_decimals` entry.
6. The NEAR transaction reverts. The EVM transfer is final. `amount` of token `T` is permanently locked in `OmniBridge` with no recovery path. [5](#0-4) [3](#0-2)
