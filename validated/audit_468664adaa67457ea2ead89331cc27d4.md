### Title
`upsert` Missing `HashAlreadyPresent` Guard Corrupts `leaf_hash_to_index` Cache — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::upsert` does not check whether `new_hash` is already mapped to a different key's leaf. Calling `upsert(key_A, v, hash_B)` when `hash_B` already belongs to `key_B` silently overwrites `leaf_hash_to_index[hash_B]` to point at `key_A`'s index, leaving the blob with two leaves sharing the same hash and the cache in an inconsistent state.

### Finding Description
In `upsert` at line 798, `remove_leaf(&leaf)` removes the **old** hash of `key_A` from `leaf_hash_to_index`. Then at line 799, `leaf.hash` is set to `new_hash` (which is `hash_B`, already owned by `key_B`). At line 803, `insert_entry_to_blob` calls `add_leaf`, which does `self.leaf_hash_to_index.insert(leaf.hash, index)` — a plain `HashMap::insert` that overwrites the existing `hash_B → index_B` entry with `hash_B → index_A`. [4](#0-3) 

After this:
- The blob contains **two leaves** with `hash_B` (at `index_A` and `index_B`).
- `leaf_hash_to_index` maps `hash_B` only to `index_A`.
- `key_to_index[key_B]` still points to `index_B`, whose blob bytes still carry `hash_B`.

### Impact Explanation
- `get_node_by_hash(hash_B)` returns `(key_A, value_A)` instead of `(key_B, value_B)` — wrong key/value pair for a given leaf hash. [5](#0-4) 

- `check_integrity` fails with `IntegrityLeafHashToIndexCacheLength` because `leaf_count == 2` but `leaf_hash_to_index.len() == 1`. [6](#0-5) 

- The Merkle root computed by `calculate_lazy_hashes` reflects a tree with two leaves sharing the same hash, producing a corrupted root that does not correspond to any valid DataLayer state.
- Inclusion proofs generated via `get_proof_of_inclusion` for `key_B` will use the correct blob path but the cache-level lookup for `hash_B` resolves to `key_A`, enabling forged or misattributed proof responses.

This matches the High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation
`upsert` is a public method. In the DataLayer API, the `hash` parameter is caller-supplied and not validated against the key-value content inside `MerkleBlob`. Any caller who can issue two DataLayer operations — insert `key_B` with `hash_B`, then upsert `key_A` with `hash_B` — triggers the corruption. No privileges, key material, or hash collisions are required.

### Recommendation
Add the same guard present in `insert` at the top of `upsert`'s update branch:

```rust
// after remove_leaf, before updating leaf.hash:
if self.block_status_cache.contains_leaf_hash(new_hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0)
