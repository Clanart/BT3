### Title
Detached `send_tokens` in fast-transfer relayer repayment causes permanent irrecoverable loss of bridged assets - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `utxo_fin_transfer_fast` and `process_fin_transfer_to_other_chain`, when the bridge repays a fast-transfer relayer upon UTXO/EVM chain finalization, `send_tokens(...)` is called with `.detach()` — fire-and-forget — **after** the fast-transfer record has already been removed or marked finalised. If the token transfer fails for any reason, the relayer's fronted bridged assets are permanently locked in the bridge contract with no retry path. The codebase itself acknowledges this with an explicit `// TODO: check how to deal with failed send_tokens` comment at the call site.

---

### Finding Description

**Root cause — `utxo_fin_transfer_fast` (`near/omni-bridge/src/lib.rs:2518–2560`)**

```rust
let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
    self.remove_fast_transfer(&fast_transfer.id());   // record deleted FIRST
    fast_transfer.amount
} else {
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // marked final FIRST
    U128(fast_transfer.amount_without_fee()...)
};

self.send_tokens(
    fast_transfer.token_id.clone(),
    fast_transfer_status.relayer,
    amount,
    "",
)
.detach();   // ← fire-and-forget; failure is never observed
```

The fast-transfer record is mutated (removed or finalised) **before** `send_tokens` is dispatched. Because `.detach()` is used, the NEAR runtime never feeds the promise result back to the contract. If `send_tokens` fails — e.g., the relayer's account lacks a storage deposit for the token, the token contract is paused, or gas is exhausted — the relayer's tokens remain in the bridge contract forever. There is no callback, no re-insertion of the fast-transfer record, and no retry surface.

The call site in `utxo_fin_transfer` (line 2483–2485) carries an explicit developer acknowledgment:

```rust
if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
    // TODO: check how to deal with failed send_tokens
    return self.utxo_fin_transfer_fast(fast_transfer, status, utxo_fin_transfer_msg);
}
```

**Same pattern — `process_fin_transfer_to_other_chain` (`near/omni-bridge/src/lib.rs:2028–2040`)**

```rust
if let Some(relayer) = recipient {
    self.send_tokens(token, relayer, U128(...), "").detach();  // fire-and-forget
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // always runs
}
```

Here `mark_fast_transfer_as_finalised` executes unconditionally after the detached promise is scheduled. A failed `send_tokens` leaves the fast transfer permanently finalised with the relayer's tokens stranded.

**Contrast with the correctly handled BTC-connector callback (`near/omni-bridge/src/btc.rs:103–126`)**

```rust
pub fn submit_transfer_to_btc_connector_callback(...) -> PromiseOrValue<()> {
    if matches!(call_result, Ok(result) if result.0 > 0) {
        self.send_fee_internal(...)
    } else {
        // re-inserts the transfer so it can be retried
        let required_storage_balance =
            self.add_transfer_message(transfer_msg, transfer_owner.clone());
        ...
    }
}
```

The BTC-connector callback correctly re-inserts the transfer on failure. The fast-transfer repayment paths have no equivalent recovery.

---

### Impact Explanation

A fast-transfer relayer fronts real bridged assets (e.g., nBTC, wETH) to the user out of pocket. When the origin chain finalises, the bridge is supposed to repay the relayer from the locked collateral. If `send_tokens` fails silently:

- The fast-transfer record is gone (deleted or finalised) — no retry is possible.
- The bridged tokens remain locked inside the bridge contract indefinitely.
- The relayer suffers a total, irrecoverable loss of the fronted amount.
- The bridge's locked-token accounting (`locked_tokens`) is decremented (via `unlock_tokens_if_needed` in `process_fin_transfer_to_other_chain`) even though the tokens were never delivered, breaking collateral accounting.

This matches the allowed impact: **Critical — permanent freezing / irrecoverable lock of bridged assets in bridge flows**.

---

### Likelihood Explanation

`send_tokens` can fail in realistic, non-adversarial conditions:

1. **Missing storage deposit**: `ft_transfer` on a NEP-141 token requires the recipient to have a storage deposit. If the relayer's account was never registered on the specific bridge token, the transfer fails.
2. **Token contract paused**: Bridge tokens expose a pause mechanism; if the token is paused at the moment of repayment, `ft_transfer` / `mint` reverts.
3. **Gas exhaustion**: `utxo_fin_transfer_fast` is called inside `ft_on_transfer` with whatever gas remains. If the prepaid gas is tight, `send_tokens` may run out of gas.

None of these require malicious behaviour — they are ordinary operational conditions. The `// TODO` comment confirms the developers are aware the failure case is unhandled.

---

### Recommendation

Replace the `.detach()` pattern with a proper callback that re-inserts the fast-transfer record on failure, mirroring the existing `submit_transfer_to_btc_connector_callback` recovery logic:

```rust
self.send_tokens(token_id, relayer, amount, "")
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(FAST_TRANSFER_REPAY_CALLBACK_GAS)
            .fast_transfer_repay_callback(fast_transfer, fast_transfer_status),
    );
```

In the callback, if the promise failed, re-insert the fast-transfer record (un-finalise it) so the relayer can trigger repayment again once the blocking condition is resolved.

---

### Proof of Concept

1. Relayer calls `ft_transfer_call` on the bridge token with a `FastFinTransferMsg`, fronting nBTC to the NEAR recipient. Bridge records the fast transfer via `add_fast_transfer`.
2. BTC chain finalises; the UTXO connector calls `ft_transfer_call` on the bridge with a `UtxoFinTransferMsg`. Bridge dispatches `utxo_fin_transfer` → `utxo_fin_transfer_fast`.
3. Inside `utxo_fin_transfer_fast`, `remove_fast_transfer` deletes the record, then `send_tokens(...).detach()` is scheduled to repay the relayer.
4. The relayer's NEAR account has no storage deposit for the nBTC token contract (e.g., it was never registered). `ft_transfer` panics inside the token contract; the promise result is `Failed`.
5. Because `.detach()` was used, the bridge never sees the failure. The fast-transfer record is gone. The nBTC tokens remain in the bridge contract. The relayer has no recourse. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn submit_transfer_to_btc_connector_callback(
