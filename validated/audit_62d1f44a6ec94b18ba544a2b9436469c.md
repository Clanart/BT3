### Title
Missing Duplicate-Key and Duplicate-Hash Checks in `batch_insert()` Bypass Guards Present in `insert()` — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::insert()` enforces two invariants — `KeyAlreadyPresent` and `HashAlreadyPresent` — before writing a leaf. `MerkleBlob::batch_insert()` calls `insert()` for at most the last two items in the input vector (and only when the tree has ≤ 1 existing leaves), then writes all remaining items directly to the blob via `insert_entry_to_blob()` with no duplicate checks. An untrusted caller can supply a batch containing duplicate `KeyId`s or duplicate `Hash`es and silently corrupt the Merkle tree, producing a wrong root hash and enabling forged proofs of inclusion/exclusion.

### Finding Description

`insert()` enforces both guards before any write:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` calls `insert()` (with its guards) only for the last two items popped from the vector, and only when `leaf_count <= 1`:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items — the first `N-2` entries in the original vector, or **all** entries when the tree already has ≥ 2 leaves — are written directly:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

No `contains_key` or `contains_leaf_hash` check is performed for these items. The `block_status_cache` is not updated for them either, so subsequent calls to `insert()` will also fail to detect the duplicates.

The Python binding `py_batch_insert` directly forwards caller-supplied `keys_values` and `hashes` to `batch_insert()`:

```rust
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

### Impact Explanation

Inserting duplicate `KeyId`s or `Hash`es corrupts the binary Merkle tree structure: two leaves share the same key identity, internal node hashes are computed over the wrong children, and the root hash no longer reflects the true committed state. This directly enables:

- **Forged proofs of inclusion**: a key that was never legitimately inserted can be "proven" present using a path through a duplicate leaf.
- **Corrupted tree roots**: the root hash stored on-chain diverges from the actual key-value set, breaking all subsequent proof verification.
- **Cross-language disagreement**: Python callers and Rust callers computing proofs against the same blob will reach different conclusions about membership.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The `py_batch_insert` Python binding is a public API surface. Any Python caller — including DataLayer clients constructing bulk updates — can supply an arbitrary list of `(KeyId, ValueId)` pairs and `Hash` values. No privilege or key material is required. The bypass is triggered simply by including a duplicate key anywhere in a batch of three or more items, or in any batch when the tree already has two or more leaves.

### Recommendation

Move the duplicate-key and duplicate-hash checks into `batch_insert()` itself, applied to every item before it is written to the blob. For example:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing write logic ...
}
```

Alternatively, update `block_status_cache` immediately after each `insert_entry_to_blob` call so that the cache remains consistent and subsequent duplicate detection works correctly.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Pre-populate with 2 leaves so leaf_count >= 2 — the if-block is skipped entirely
blob.batch_insert(
    [(KeyId(1), ValueId(1)), (KeyId(2), ValueId(2))],
    [hashlib.sha256(b"1").digest(), hashlib.sha256(b"2").digest()],
)

# Now batch_insert with a duplicate KeyId(1) — no check is performed
# This silently inserts a second leaf with key=1, corrupting the tree
blob.batch_insert(
    [(KeyId(1), ValueId(99)), (KeyId(3), ValueId(3))],
    [hashlib.sha256(b"dup").digest(), hashlib.sha256(b"3").digest()],
)

blob.calculate_lazy_hashes()
# The root hash is now computed over a tree containing two leaves with KeyId(1),
# making any proof of inclusion for KeyId(1) ambiguous and the root invalid.
proof = blob.get_proof_of_inclusion(KeyId(1))
# proof.valid() may return True for a key whose committed state is undefined
```
