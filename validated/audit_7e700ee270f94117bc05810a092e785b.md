Audit Report

## Title
Permissionless `register()` Irrevocably Clears Admin Pool Blacklist, Defeating the Oracle Emergency-Stop — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

## Summary

`OracleBase.register()` is a permissionless, fee-gated function that unconditionally clears `blacklisted[pool]` for any pool it processes. The admin's `setBlacklist(pool, true)` emergency stop can therefore be atomically reversed by any unprivileged caller for 1 wei (the hardcoded default `registrationFee`). There is no `unregister()` function anywhere in the codebase, so once `registeredPool[feedId][pool]` is set to `true` it cannot be revoked through any admin path targeting only that pool.

## Finding Description

`OracleBase` exposes two relevant admin controls:

1. **`setBlacklist(address, bool)`** (lines 271–276) — `ADMIN_ROLE` only; sets `blacklisted[account]`.
2. **`register(bytes32, address, address)`** (lines 201–214) — `external payable`, no role check; unconditionally clears `blacklisted[pool]` if it is set.

The critical path inside `register()`:

```solidity
// OracleBase.sol lines 207-210
if (blacklisted[pool]) {
    blacklisted[pool] = false;          // ← clears admin blacklist
    emit BlacklistUpdated(pool, false);
}
```

The NatSpec at line 198 documents this as intentional: *"Paying also clears any blacklist on the pool."*

The `price()` function enforces both guards (lines 160–172):
- `notBlacklisted` modifier (line 76–80): checks `blacklisted[msg.sender]` (the price provider).
- `require(!blacklisted[pool], ...)` (line 167): checks the pool address.
- `require(registeredPool[feedId][pool], ...)` (line 168): checks registration.

An attacker targeting a blacklisted pool calls `register(feedId, pool, approvedFactory)` with 1 wei. This clears `blacklisted[pool]` atomically. `registeredPool[feedId][pool]` was already `true` (no-op). Both `price()` guards are now satisfied and oracle reads resume.

The existing test `test_register_thirdParty_redeems_andEnablesRead` (lines 468–487 of `OracleBaseAbuseProtection.t.sol`) explicitly confirms this path: a third party clears the blacklist and successfully calls `price()` afterward.

A grep across all `.sol` files confirms no `unregister` function exists in `providers/OracleBase.sol`, leaving no admin path to revoke `registeredPool[feedId][pool]` for a specific pool.

**Preconditions satisfied for any legitimate pool under exploit:**
- Pool was created by an approved factory → `approvedFactories.contains(factory)` passes.
- `IPoolFactory(factory).isPool(pool)` passes (pool is legitimate).
- `registrationFee` defaults to 1 wei (line 53).

## Impact Explanation

When a pool is actively being exploited (e.g., price-manipulation or callback-reentrancy), the admin's intended response is `setBlacklist(pool, true)`, causing every subsequent `price(feedId, pool)` call to revert with `Blacklisted(pool)`, halting oracle-dependent swaps. An unprivileged attacker defeats this for 1 wei by calling `register(feedId, pool, approvedFactory)`, clearing the blacklist atomically. Oracle price reads resume, swaps proceed, and the exploit continues draining LP funds. This is a direct admin-boundary break: an oracle role check (`ADMIN_ROLE`-gated blacklist) is bypassed by an unprivileged path, enabling continued bad-price execution and fund loss from the exploited pool.

Note: the admin can alternatively blacklist the price provider address (`msg.sender` in `price()`), which `register()` cannot clear since it only clears `blacklisted[pool]`. However, the pool-address blacklist path — the natural and documented emergency stop — is fully defeatable as described.

## Likelihood Explanation

- **Trigger cost**: 1 wei (hardcoded default `registrationFee`).
- **Caller restriction**: none — any EOA or contract qualifies.
- **Preconditions**: pool must be from an approved factory — already satisfied for any legitimate pool being exploited.
- **Attacker motivation**: high — the attacker is already profiting from the exploit and has every incentive to spend 1 wei to continue.
- **Repeatability**: unlimited; the attacker can re-call `register()` after every admin `setBlacklist(pool, true)` action.

## Recommendation

1. **Remove the blacklist-clearing side-effect from `register()`**. A blacklisted pool should not be re-registerable without explicit admin approval. At minimum, add:
   ```solidity
   require(!blacklisted[pool], Blacklisted(pool));
   ```
   at the top of `register()`.

2. **Add an `unregister()` function** so the admin can permanently revoke a pool's oracle access for a specific feed:
   ```solidity
   function unregister(bytes32 feedId, address pool) external onlyRole(ADMIN_ROLE) {
       registeredPool[feedId][pool] = false;
       emit PoolUnregistered(feedId, pool);
   }
   ```

3. **Require admin co-signature to re-register a previously blacklisted pool**, or gate re-registration with a separate admin-approved allowlist.

## Proof of Concept

```
1. Pool P is created by approvedFactory; registeredPool[feedId][P] = true (via register).
2. Pool P is found to be exploitable; admin calls setBlacklist(P, true).
   → price(feedId, P) reverts with Blacklisted(P); oracle-dependent swaps halt.
3. Attacker calls register(feedId, P, approvedFactory) with msg.value = 1 wei.
   → Inside register() (line 207-210): blacklisted[P] = false.
   → registeredPool[feedId][P] = true (already set, no-op).
4. price(feedId, P) succeeds again; swaps through P resume.
5. Attacker continues draining LP positions.
   Admin blacklist is permanently ineffective as long as attacker spends 1 wei per admin action.

Foundry test confirming step 3-4 already exists:
  smart-contracts-poc/test/oracles/OracleBaseAbuseProtection.t.sol
  test_register_thirdParty_redeems_andEnablesRead (lines 468-487)
```
