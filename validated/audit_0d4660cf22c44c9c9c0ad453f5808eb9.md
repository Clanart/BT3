Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal ETH stranded by exact-output swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` transfers the router's entire ETH balance to `msg.sender` with no access control. ETH is routinely stranded in the router when users call `exactOutputSingle` or `exactOutput` with `msg.value = amountInMaximum` and the pool's actual cost is lower. Any subsequent caller can drain the surplus before the original user reclaims it.

## Finding Description
`refundETH()` (L58–63) is `external payable` with no `msg.sender` check — it unconditionally sends `address(this).balance` to the caller. [1](#0-0) 

ETH enters the router via the `payable` modifier on `exactOutputSingle` (L130) — this bypasses the `receive()` guard at L32–34, which only blocks *direct* ETH pushes from non-WETH addresses, not ETH attached to a function call. [2](#0-1) 

Inside `pay()`, when `token == WETH`, the function wraps exactly `value` (the actual amountIn demanded by the pool) and leaves any surplus native ETH in the contract: [3](#0-2) 

`exactOutputSingle` enforces `amountIn <= amountInMaximum` but never refunds the difference: [4](#0-3) 

The intended safe usage pattern is to bundle `[exactOutputSingle(...), refundETH()]` inside a single `multicall`. However, `multicall` is not enforced — `exactOutputSingle` is independently callable as `external payable`, and nothing prevents a user from calling it standalone. When they do, the surplus ETH is stranded in the router until any third party calls `refundETH()` and claims it.

## Impact Explanation
Direct loss of user ETH principal equal to `msg.value − actual amountIn` for every exact-output ETH swap where the user does not atomically bundle `refundETH()` in the same `multicall`. This is a realistic loss path: exact-output swaps inherently produce surplus ETH whenever the pool's actual cost is below `amountInMaximum`. Meets Sherlock Medium/High threshold for direct loss of user funds.

## Likelihood Explanation
Moderate. Users who call `exactOutputSingle` or `exactOutput` directly (not via multicall) strand ETH on every swap where actual cost < maximum. MEV bots routinely monitor router contracts for stranded ETH. No special privilege or setup is required — any EOA can call `refundETH()`.

## Recommendation
Either (a) restrict `refundETH()` to only be callable within a `multicall` context (e.g., require `msg.sender == address(this)` after delegatecall), or (b) automatically refund excess ETH at the end of each exact-output swap function, or (c) add a revert guard that enforces `refundETH()` must be bundled in the same `multicall` as the swap.

## Proof of Concept
```solidity
// 1. User calls exactOutputSingle with msg.value = 2 ETH, amountInMaximum = 2 ETH
//    Pool only needs 1.5 ETH; pay() wraps 1.5 ETH → 0.5 ETH stays in router.
// 2. Attacker (separate tx) calls router.refundETH()
//    → attacker receives 0.5 ETH belonging to the user.
assertEq(attacker.balance, 0.5 ether);                              // passes
assertEq(user.balance, initialBalance - 1.5 ether - 0.5 ether);    // user lost excess
```
