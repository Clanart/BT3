### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key/Hash Existence Checks for All But the Last Two Items, Corrupting the DataLayer Merkle Tree — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` enforces duplicate-key and duplicate-hash guards only for the **last two** items in the input batch (via the guarded `insert()` path). All earlier items are written directly to the blob via `insert_entry_to_blob()` with no existence check. Supplying a batch with a repeated `KeyId` or `Hash` — or a key that already exists in the tree — silently inserts a second leaf node into the blob while overwriting the cache pointer, orphaning the first leaf and producing a structurally corrupt Merkle tree whose root hash is wrong and whose proofs of inclusion are invalid.

---

### Finding Description

`insert()` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert()` calls `insert()` for only the **last two** items when the tree has ≤ 1 existing leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { … };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // ← guarded
    }
}
// All remaining items (the first N-2 in the original input):
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) … };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← NO duplicate check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob()` unconditionally calls `block_status_cache.add_leaf()`:

```rust
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    …
}
```

`add_leaf()` uses `HashMap::insert`, which **silently overwrites** the existing entry:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);   // overwrites silently
}
```

When a duplicate key appears among the first N-2 items:
1. A second `LeafNode` with the same `KeyId` is appended to the blob at a new index.
2. The cache's `key_to_index` is updated to point to the new index; the old leaf node is now an **orphan** — still present in the blob and still referenced by its parent `InternalNode`, but invisible to the cache.
3. The tree-building loop then computes `internal_hash` over the orphaned leaf's hash, embedding it in the Merkle root.
4. The resulting root hash is wrong; any proof of inclusion derived from it is invalid.

The same flaw applies when a key in the batch already exists in the tree before `batch_insert` is called (for items 3+, there is no check against the pre-existing cache state either).

---

### Impact Explanation

The DataLayer Merkle tree is the authoritative structure for DataLayer inclusion/exclusion proofs. A corrupted root hash means:

- `get_proof_of_inclusion()` returns a proof that does not verify against the actual committed root.
- `upsert()` and `delete()` operating on the orphaned key will silently operate on the wrong node (the cache points to the duplicate, not the original).
- `check_integrity()` would detect the inconsistency (`leaf_count != key_to_index_cache_length`), but it is not called automatically during or after `batch_insert`.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is exposed directly as the Python binding `py_batch_insert` (lines 1503–1519), callable by any Python code that holds a `MerkleBlob`. The DataLayer service uses this path for bulk insertions. Any caller that passes a batch containing a repeated `KeyId` — whether by mistake or by adversarial construction — triggers the corruption. No privileged role is required; the Python API accepts the input without any pre-validation.

### Recommendation

Add the same existence guards inside the bulk loop in `batch_insert`, mirroring `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing insert_entry_to_blob call …
}
```

Alternatively, deduplicate the input batch before processing, or add a post-insert `check_integrity()` call (though the former is strongly preferred for early rejection).

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
