### Title
Stale `msg.Nonce` Overwrites EIP-7702 Authorization Nonce Increment in Simulator — (`x/evm/keeper/simulate.go`)

### Summary

In `applyCall` inside `simulate.go`, after EIP-7702 `applyAuthorization` increments the sender's nonce (for a self-delegation), the code unconditionally overwrites the nonce with the stale pre-execution value `msg.Nonce+1`. This is the direct analog of the external report: a snapshot of a value is captured, the underlying state is mutated, and the stale snapshot is then used to overwrite the updated state.

### Finding Description

In `applyCall` (`x/evm/keeper/simulate.go`, lines 430–451), the execution flow for a CALL transaction with EIP-7702 authorizations is:

```go
// Step 1: set nonce to msg.Nonce (snapshot captured here)
sim.state.SetNonce(msg.From, msg.Nonce, tracing.NonceChangeUnspecified)

// Step 2: apply authorizations — for self-delegation (authority == msg.From),
//         setAuthorizationDelegation calls:
//         stateDB.SetNonce(authority, auth.Nonce+1, ...)
//         which MODIFIES the sender's nonce in state
for _, auth := range msg.SetCodeAuthorizations {
    sim.keeper.applyAuthorization(&auth, sim.state)
}

// Step 3: execute call
ret, leftoverGas, vmErr = evm.Call(msg.From, *msg.To, msg.Data, leftoverGas, value)

// Step 4: STALE OVERWRITE — uses the original msg.Nonce, ignoring the
//         authorization nonce increment from Step 2
sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
``` [1](#0-0) 

`setAuthorizationDelegation` in `set_code_authorizations.go` unconditionally calls `stateDB.SetNonce(authority, auth.Nonce+1, ...)`: [2](#0-1) 

For a self-delegating EIP-7702 transaction where `authority == msg.From` and `auth.Nonce == msg.Nonce+1`, the correct post-execution nonce is `msg.Nonce+2`. But the stale `SetNonce(msg.From, msg.Nonce+1, ...)` at line 450 overwrites it to `msg.Nonce+1`.

**Contrast with the production path** in `ApplyMessageWithConfig` (`state_transition.go`): for CALL transactions, there is **no** `SetNonce` call after `evm.Call()`. The ante handler pre-increments the nonce, and `applyAuthorization` handles the authorization nonce bump — neither is overwritten. [3](#0-2) 

The simulator has no ante handler, so it manually manages the nonce. The bug is that line 450 uses the stale `msg.Nonce` snapshot rather than reading the current post-authorization nonce from state.

### Impact Explanation

The `eth_simulateV1` JSON-RPC endpoint uses `Simulator.Execute` → `applyCall`. When a simulation block contains an EIP-7702 self-delegation CALL followed by any further transaction from the same sender, the simulator's nonce state is off by one. In `validate=true` mode (the default for `eth_simulateV1`), the nonce check at lines 364–372 will reject the subsequent transaction with `ErrNonceTooHigh` even though it would succeed on-chain: [4](#0-3) 

This causes the public JSON-RPC simulation path to feed incorrect nonce state into simulated transaction execution, producing results that diverge from actual on-chain execution. Users and tooling relying on `eth_simulateV1` to predict multi-transaction outcomes (e.g., contract deployment addresses, gas estimates, revert/success status) will receive wrong answers for any block containing an EIP-7702 self-delegation CALL.

### Likelihood Explanation

EIP-7702 is a Prague/Pectra hardfork feature. Any user submitting a self-delegating EIP-7702 CALL via `eth_simulateV1` with a subsequent transaction in the same simulated block triggers the bug. The entry path is fully unprivileged: a standard JSON-RPC call with a crafted `authorizationList` containing a self-delegation. No special permissions are required.

### Recommendation

Replace the unconditional stale overwrite with a read of the current post-execution nonce from state:

```go
// Instead of:
sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)

// Use:
currentNonce := sim.state.GetNonce(msg.From)
if currentNonce < msg.Nonce+1 {
    sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
}
```

This mirrors the production path in `ApplyMessageWithConfig`, which does not overwrite the nonce after `evm.Call()` and instead relies on the already-updated state from `applyAuthorization`.

### Proof of Concept

1. Deploy a contract `DelegationTarget` at address `D`.
2. Construct an `eth_simulateV1` request with `validate: true` and two calls in one block from sender `S` (nonce=N):
   - **Call 0**: EIP-7702 type-4 tx, `to=S`, `nonce=N`, `authorizationList=[{address: D, nonce: N+1}]` (self-delegation). After execution, correct nonce = N+2; simulator sets it to N+1.
   - **Call 1**: Any tx from `S` with `nonce=N+2`.
3. Observe that Call 1 fails with `ErrNonceTooHigh: address S, tx: N+2 state: N+1` in the simulation, even though it would succeed on-chain.

The root cause is at `simulate.go:450` where `sim.state.SetNonce(msg.From, msg.Nonce+1, ...)` uses the stale pre-authorization nonce snapshot, overwriting the nonce increment applied by `applyAuthorization` at `set_code_authorizations.go:76`. [5](#0-4) [6](#0-5)

func (k *Keeper) applyAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (common.Address, error) {
	authority, err := k.validateAuthorization(auth, stateDB)
	if err != nil {
		return authority, err
	}

	// If the account already exists in state, refund the new account cost
	// charged in the intrinsic calculation.
	if stateDB.Exist(authority) {
		stateDB.AddRefund(params.CallNewAccountGas - params.TxAuthTupleGas)
	}

	k.setAuthorizationDelegation(auth, authority, stateDB)
	return authority, nil
}
