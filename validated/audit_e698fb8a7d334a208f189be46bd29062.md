### Title
Missing Normalized-Amount Bounds Check in `update_transfer_fee` Permanently Locks User Funds - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`update_transfer_fee` enforces only that `fee.fee < transfer.message.amount`, allowing the token fee to be raised to `amount - 1`. For any token whose on-chain decimals differ from its origin decimals (`origin_decimals > decimals`), the resulting `amount_without_fee = 1` normalizes to zero in `sign_transfer`, which then hard-panics with `InvalidAmountToTransfer`. Because the fee can only be monotonically increased and there is no cancel path, the locked tokens become permanently irrecoverable.

---

### Finding Description

`update_transfer_fee` in `near/omni-bridge/src/lib.rs` validates the new fee with a single bound:

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This permits `fee.fee = amount - 1`, leaving `amount_without_fee() = 1`.

`sign_transfer` then calls `normalize_amount` on that remainder:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` performs integer division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

For any token where `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridged to EVM where it is capped at 18, giving `diff_decimals = 6`), `normalize_amount(1, decimals) = 1 / 10^6 = 0`. The `require!(amount_to_transfer > 0, …)` guard then panics, aborting `sign_transfer`.

The fee monotonicity constraint means the fee can never be lowered after this point:

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

No cancel or refund path exists in the contract. The transfer message and its locked tokens remain in the bridge indefinitely.

The same missing check exists at `init_transfer` time:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [4](#0-3) 

---

### Impact Explanation

A user who sets `fee.fee` to any value satisfying `amount - fee.fee < 10^diff_decimals` (e.g., `fee = amount - 1` for any token with `diff_decimals ≥ 1`) causes every subsequent `sign_transfer` call for that transfer to revert with `InvalidAmountToTransfer`. Because the fee cannot be decreased and no cancel mechanism exists, the bridged tokens locked in the NEAR contract are permanently irrecoverable. This is an irrecoverable lock of user funds in the bridge vault flow, matching the Critical impact tier.

---

### Likelihood Explanation

The condition is reachable whenever a token has `origin_decimals > decimals`. NEAR-native tokens canonically carry 24 decimals while EVM bridge tokens are capped at 18 (`_normalizeDecimals`), giving `diff_decimals = 6` for the most common token class. A user transferring any amount smaller than `10^6` base units after fee deduction — or deliberately or accidentally setting a fee that leaves fewer than `10^6` units — triggers the lock. The `update_transfer_fee` function is public and callable by the transfer sender without any privileged role.

---

### Recommendation

Add a post-normalization check inside both `update_transfer_fee` and `init_transfer` to ensure the net amount after fee deduction normalizes to a positive value on the destination chain:

```rust
let normalized = Self::normalize_amount(
    transfer_message.amount.0 - fee.fee.0,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

Alternatively, enforce a minimum `amount_without_fee` floor equal to `10^diff_decimals` so that the normalized amount is always at least 1.

---

### Proof of Concept

1. Token `T` is registered with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. User calls `ft_transfer_call` → `init_transfer` with `amount = 1_000_000` (1 unit in 24-decimal representation) and `fee = 0`. Tokens are locked.
3. User calls `update_transfer_fee` with `fee.fee = 999_999` (allowed because `999_999 < 1_000_000`). No additional validation fires.
4. Trusted relayer calls `sign_transfer`.
   - `amount_without_fee() = 1_000_000 - 999_999 = 1`
   - `normalize_amount(1, Decimals { decimals: 18, origin_decimals: 24 }) = 1 / 10^6 = 0`
   - `require!(0 > 0, …)` → **panic: `ERR_INVALID_AMOUNT_TO_TRANSFER`**
5. The fee cannot be lowered (monotonicity check). No cancel function exists.
