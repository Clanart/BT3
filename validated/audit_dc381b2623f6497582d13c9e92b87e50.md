### Title
Unhandled Constructor Revert in `deploy_contract` Causes OS Proof Failure — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deploy_contract.cairo`)

---

### Summary

The `deploy_contract` function in the StarkNet OS unconditionally asserts that the constructor entry point did not revert (`assert is_reverted = 0`). This is incomplete error handling: the function calls `select_execute_entry_point_func`, which can legitimately return `is_reverted = 1` if the constructor fails, but the only handling is a hard assertion. When triggered, the Cairo VM cannot satisfy the constraint, the OS proof cannot be generated, and the block cannot be finalized — a network halt.

---

### Finding Description

In `deploy_contract.cairo`, after invoking the constructor via `select_execute_entry_point_func`, the OS performs:

```cairo
// Invoke the contract constructor.
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);

// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

The comment "The deprecated deploy syscalls do not support reverts" is misleading: `deploy_contract` is also called from the primary `execute_deploy_account_transaction` path in `transaction_impls.cairo`:

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [2](#0-1) 

The function's own docstring acknowledges the gap with a deferred TODO:

```cairo
// TODO(Yoni, 1/1/2027): handle failures.
func deploy_contract{...}(...)
``` [3](#0-2) 

`select_execute_entry_point_func` (called inside `deploy_contract`) routes through `execute_entry_point`, which returns `is_reverted = 1` for out-of-gas, entry-point-not-found, or any contract-level failure flag: [4](#0-3) 

When `is_reverted = 1` is returned and `assert is_reverted = 0` is evaluated, the Cairo VM raises an unsatisfiable constraint. The prover cannot produce a valid proof for the block.

The analog to the external report is exact:
- **External report**: `catch Error(string memory)` silently misses `revert EnforcedPause()` (a custom error), causing the system to continue in a broken state.
- **StarkNet OS**: `assert is_reverted = 0` silently misses the constructor-revert case, causing the OS proof to fail entirely instead of gracefully reverting the deployment.

---

### Impact Explanation

When `assert is_reverted = 0` fails inside `deploy_contract`, the Cairo VM cannot satisfy the constraint. The OS program cannot complete execution, no valid STARK proof can be generated for the block, and the block cannot be committed to the chain. Any block containing a `DEPLOY_ACCOUNT` transaction whose constructor reverts during OS execution is permanently unproducible.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

The sequencer simulates transactions before inclusion. However, a constructor's revert behavior can diverge between simulation time and OS execution time if another transaction in the same block modifies storage that the constructor reads. Concretely:

1. User A submits a `DEPLOY_ACCOUNT` transaction whose constructor reads a storage slot and reverts if it is non-zero.
2. User B submits a transaction that writes a non-zero value to that slot.
3. The sequencer simulates User A's transaction in isolation (slot is zero → constructor succeeds) and includes both transactions, ordering User B's first.
4. During OS execution, User B's transaction runs first, setting the slot to non-zero.
5. User A's constructor reads the non-zero value and sets `failure_flag = 1`.
6. `assert is_reverted = 0` fails → proof generation fails → block is permanently stuck.

This requires no privileged access: both users are ordinary transaction senders. The constructor logic is user-controlled (User A controls the class hash and calldata). The ordering is sequencer-controlled but not secret.

**Likelihood: Medium.**

---

### Recommendation

Replace the hard assertion with a proper revert handler, consistent with how `execute_entry_point` handles reverts elsewhere in the OS:

```diff
 let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
     block_context=block_context, execution_context=constructor_execution_context
 );

-// The deprecated deploy syscalls do not support reverts.
-assert is_reverted = 0;
-return (retdata_size=retdata_size, retdata=retdata);
+if (is_reverted != FALSE) {
+    handle_revert(
+        contract_address=contract_address,
+        revert_log_end=revert_log,
+    );
+    return (is_reverted=1, retdata_size=retdata_size, retdata=retdata);
+}
+return (is_reverted=0, retdata_size=retdata_size, retdata=retdata);
```

The callers (`execute_deploy_account_transaction` and any deploy-syscall handler) must then propagate and handle the `is_reverted` return value rather than assuming success.

---

### Proof of Concept

1. Declare a Sierra class whose constructor contains:
   ```
   // reads storage slot S; if S != 0, panic
   let val = storage_read(S);
   assert val == 0, 'constructor reverted';
   ```
2. Submit a `DEPLOY_ACCOUNT` transaction (User A) deploying this class.
3. Submit a separate invoke transaction (User B) that writes `1` to slot `S` of the same contract address (possible before deployment because the address is deterministic from salt + class hash).
4. Arrange for User B's transaction to precede User A's in the block (e.g., higher tip).
5. The sequencer simulates User A's constructor before User B's write → simulation passes.
6. The sequencer includes both transactions in the block with User B first.
7. During OS execution: User B writes `S = 1`; User A's constructor reads `S = 1` and sets `failure_flag = 1`; `assert is_reverted = 0` at `deploy_contract.cairo:91` fails; the OS cannot produce a proof; the block is permanently unproducible. [1](#0-0) [5](#0-4)
