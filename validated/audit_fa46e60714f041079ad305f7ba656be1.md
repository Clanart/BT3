### Title
Cross-Chain Signature Replay in `deploy_token` / `deployToken` — Missing Chain ID in `MetadataPayload` Hash (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The NEAR MPC signs a `MetadataPayload` that contains no chain identifier. The Borsh-encoded bytes hashed and signed are identical across EVM, Starknet, and Solana. A signature produced for deploying a token on one chain is therefore valid on every other chain, enabling an unprivileged attacker to replay it and deploy the same bridged token on any chain without NEAR's per-chain authorization.

---

### Finding Description

**NEAR signing side** — `near/omni-bridge/src/lib.rs` `log_metadata_callback`:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload)…
);
``` [1](#0-0) 

The `MetadataPayload` struct carries `[prefix=1, token, name, symbol, decimals]` — no chain ID field exists. [2](#0-1) 

**EVM verification** — `OmniBridge.sol` `deployToken`:

```solidity

``` [3](#0-2) 

No `omniBridgeChainId` byte is mixed into the hash. Compare with `finTransfer`, which explicitly encodes `bytes1(omniBridgeChainId)` twice: [4](#0-3) 

**Starknet verification** — `bridge_types.cairo` `MetadataPayloadTrait::to_borsh`:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into()); // 1
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [5](#0-4) 

Again no chain ID. Compare with `TransferMessagePayloadTrait::to_borsh`, which appends `chain_id` twice: [6](#0-5) 

The Starknet CLAUDE.md explicitly documents this design intent for transfers: *"Chain ID binding: Destination chain_id encoded in message hash (not in payload) — prevents cross-chain replay."* No equivalent protection exists for `deploy_token`.

**Solana verification** — `deploy_token.rs` `DeployTokenPayload::serialize_for_near`:

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;  // 0x01
    self.serialize(&mut writer)?;  // {token, name, symbol, decimals}
    …
}
``` [7](#0-6) 

`SOLANA_OMNI_BRIDGE_CHAIN_ID` is absent from the metadata hash, unlike `finalize_transfer.rs` which writes it twice: [8](#0-7) 

**Result**: The three byte sequences produced by EVM, Starknet, and Solana for the same token are byte-for-byte identical. The single NEAR MPC signature is therefore universally valid across all chains.

---

### Impact Explanation

An attacker who observes a valid `deploy_token` / `deployToken` transaction on any chain (the signature is on-chain and public) can immediately replay it on every other supported chain. This achieves:

1. **Unauthorized token deployment** on chains where NEAR has not yet authorized the token — directly matching the allowed impact class *"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment."*

2. **Permanent fund lock** — once the bridged token exists on an unauthorized chain, users who deposit into the bridge on that chain will have their tokens locked/burned. If NEAR has no registered prover or token mapping for that chain, the corresponding `fin_transfer` on NEAR will never be processed and the funds are irrecoverable — matching *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."*

3. **Token-mapping corruption** — deploying the same NEAR token ID on multiple chains simultaneously can corrupt the `nearToEthToken` / `near_to_starknet_token` mappings if the protocol later tries to register the token officially on those chains, since the slot is already occupied by the attacker-triggered deployment.

---

### Likelihood Explanation

- The `deploy_token` / `deployToken` call is a public, permissionless transaction. The signature is visible in calldata the moment the first legitimate deployment lands on any chain.
- No privileged access, leaked key, or colluding MPC signer is required. Any observer of the first deployment can replay it.
- The bridge supports many EVM chains (Eth, Arb, Base, BNB, Pol, HyperEvm, Abs) plus Solana and Starknet. A single signature is replayable across all of them.
- The only on-chain guard is the "token already deployed" check, which only prevents replay on the *same* chain. [9](#0-8) [10](#0-9) 

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` hash, exactly as `TransferMessagePayload` already does. On the NEAR signing side, pass the target `ChainKind` into `log_metadata_callback` and append it to the borsh bytes before hashing. On each destination chain, verify that the chain ID byte in the signed payload matches the contract's own chain ID. This mirrors the existing pattern in `finTransfer` / `fin_transfer` / `finalize_transfer` and closes the domain-separation gap.

---

### Proof of Concept

1. NEAR signs a `MetadataPayload` for token `"usdc.near"` with `name="USD Coin"`, `symbol="USDC"`, `decimals=6`. The resulting keccak256 hash and ECDSA signature `σ` are emitted on Ethereum when `deployToken` is called.

2. An attacker reads `σ` from the Ethereum transaction calldata.

3. The attacker calls `deployToken(σ, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on the Arbitrum `OmniBridge` contract. The Borsh encoding is byte-for-byte identical; `ECDSA.recover` returns `nearBridgeDerivedAddress`; the call succeeds and a new `BridgeToken` proxy is deployed on Arbitrum.

4. The attacker repeats step 3 for Base, BNB, Polygon, Starknet, and Solana using the same `σ`.

5. Users who see `USDC` listed on Arbitrum (or any other chain) and deposit into the bridge have their tokens locked. If NEAR has no prover or token mapping registered for Arbitrum, those tokens are permanently unrecoverable. [11](#0-10) [12](#0-11) [13](#0-12)

            assert(!_is_paused(@self, PAUSE_DEPLOY_TOKEN), 'ERR_DEPLOY_TOKEN_PAUSED');

            _verify_borsh_signature(ref self, @payload.to_borsh(), signature);

            let token_id_hash = compute_keccak_byte_array(@payload.token);
            let existing_token = self.near_to_starknet_token.read(token_id_hash);
            assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED');

            let decimals = _normalizeDecimals(payload.decimals);

            let mut constructor_calldata: Array<felt252> = array![];
            (payload.name.clone(), payload.symbol.clone(), decimals)
                .serialize(ref constructor_calldata);

            // Use the low part of the u256 hash to ensure it fits in felt252
            let salt: felt252 = token_id_hash.low.into();
            let (contract_address, _) = deploy_syscall(
                self.bridge_token_class_hash.read(), salt, constructor_calldata.span(), false,
            )
                .unwrap_syscall();

            self.starknet_to_near_token.write(contract_address, payload.token.clone());
            self.near_to_starknet_token.write(token_id_hash, contract_address);

            self
                .emit(
                    Event::DeployToken(
                        DeployToken {
                            token_address: contract_address,
                            near_token_id: payload.token,
                            name: payload.name,
                            symbol: payload.symbol,
                            decimals,
                            origin_decimals: payload.decimals,
                        },
                    ),
                )
