### Title
Receipt Commitment Silently Substitutes Zero for Actual L2 Gas, Producing a Wrong Block Hash for Every L2-Gas-Consuming Transaction — (File: `crates/starknet_api/src/block_hash/receipt_commitment.rs`)

---

### Summary

`chain_gas_consumed` in `receipt_commitment.rs` chains `Felt::ZERO` as the L2-gas-consumed field of every per-transaction receipt hash instead of the actual `gas_consumed.l2_gas` value. Because Starknet v0.14.x transactions routinely carry non-zero L2 gas (production blocks show values of ~76 million units per transaction), every such transaction produces a receipt hash that omits its real L2 gas consumption. The receipt commitment — a Patricia root over all per-transaction receipt hashes — is therefore systematically wrong for any block that contains L2-gas-consuming transactions. That wrong commitment is chained directly into the block hash via `calculate_block_hash`, so the block hash the sequencer produces diverges from what any independent verifier computing the commitment from actual execution data would derive.

---

### Finding Description

**Root cause — `chain_gas_consumed`:** [1](#0-0) 

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO) // L2 gas consumed   ← always 0, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

The function receives a `GasVector` that carries `l2_gas`, but chains the constant `Felt::ZERO` instead of `gas_consumed.l2_gas.into()`. The inline comment acknowledges the discrepancy ("In the current RPC: always 0"), but the actual execution data flowing into this function is non-zero.

**Data-flow path from execution to commitment:**

1. `TransactionExecutionInfo::output_for_hashing()` sets `gas_consumed: self.receipt.gas`, which is the full `GasVector` including `l2_gas` from the blockifier receipt. [2](#0-1) 

2. `prepare_txs_hashing_data()` in `block_builder.rs` calls `output_for_hashing()` for every executed transaction and packages the result into `TransactionHashingData`. [3](#0-2) 

3. `BlockExecutionArtifacts::new` calls `calculate_block_commitments()` with those hashing-data objects. [4](#0-3) 

4. `calculate_block_commitments()` spawns `calculate_receipt_commitment::<Poseidon>()`, which maps each element through `calculate_receipt_hash()`, which calls `chain_gas_consumed()` — dropping `l2_gas`. [5](#0-4) 

5. The resulting (wrong) `receipt_commitment` is stored in `BlockHeaderCommitments` and chained into the block hash. [6](#0-5) 

**Confirmation that `l2_gas` is non-zero in production:**

The `block_post_0_14_2.json` test fixture shows `"l2_gas": 76014735` in `total_gas_consumed` for every transaction in a v0.14.2 block. The `ExecutionResources → GasVector` conversion in the client reader maps `total_gas_consumed` directly to `gas_consumed`, so the `l2_gas` field reaching `chain_gas_consumed` is non-zero. [7](#0-6) 

**Analogy to the seed bug:**

The GMX bug added PnL to the pool value when it should have subtracted it — a wrong arithmetic operation on a critical accumulator. Here, the wrong *value* (constant zero) is substituted for the actual L2 gas in a critical hash accumulator. Both produce a committed value that diverges from the true state of the system.

---

### Impact Explanation

The receipt commitment is a direct input to `calculate_block_hash`. [8](#0-7) 

A wrong receipt commitment produces a wrong block hash. The `PartialBlockHash` (used as the `ProposalCommitment` in consensus) is derived from the same `PartialBlockHashComponents` that contains the wrong receipt commitment. [9](#0-8) 

Any external system — L1 verifier, proof system, or cross-chain bridge — that independently recomputes the receipt commitment from actual execution data (including real L2 gas) will derive a different block hash, causing verification failures. The `PartialBlockHash` stored and gossiped during consensus will also be wrong, so a validator that independently recomputes it from the execution output will disagree with the proposer.

**Scope match:** "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input" (Critical) and "RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value" (High).

---

### Likelihood Explanation

Every Starknet v0.14.x transaction that consumes L2 gas triggers this. Production block data confirms L2 gas values of ~76 million units per transaction. No special attacker capability is required — submitting any ordinary invoke transaction on a v0.14.x network is sufficient to produce a block whose receipt commitment is wrong.

---

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual L2 gas value:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // was: Felt::ZERO
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
``` [10](#0-9) 

---

### Proof of Concept

1. Execute any Starknet v0.14.x invoke transaction. The blockifier sets `receipt.gas.l2_gas` to the actual L2 gas consumed (e.g., 76,014,735).
2. Call `output_for_hashing()` on the resulting `TransactionExecutionInfo`. Observe `gas_consumed.l2_gas > 0`.
3. Feed the result into `calculate_receipt_hash()`. Observe that `chain_gas_consumed` chains `Felt::ZERO` for L2 gas regardless.
4. Independently compute the receipt hash using `gas_consumed.l2_gas` instead of zero. The two hashes differ.
5. Compute `calculate_receipt_commitment` for a block containing such transactions using both approaches. The two receipt commitments differ.
6. Feed both receipt commitments into `calculate_block_hash` with identical other inputs. The two block hashes differ — the sequencer's stored block hash diverges from the independently verified one.
