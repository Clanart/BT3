### Title
STRK Native Fees Permanently Locked in StarkNet Bridge Contract With No Recovery Path - (File: starknet/src/omni_bridge.cairo)

### Summary
The `init_transfer` function in the StarkNet `OmniBridge` contract collects `native_fee` STRK tokens by transferring them from the caller directly into the contract itself (`get_contract_address()`). The contract exposes no function — neither admin-gated nor otherwise — to withdraw, sweep, or redistribute these accumulated STRK tokens. Every STRK native fee paid by every bridge user is permanently and irrecoverably locked in the contract.

### Finding Description
In `starknet/src/omni_bridge.cairo`, `init_transfer` handles the `native_fee` parameter as follows:

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
```

The destination is `get_contract_address()` — the bridge contract itself. [1](#0-0) 

The full `IOmniBridge` interface exposes only: `log_metadata`, `deploy_token`, `fin_transfer`, `init_transfer`, `upgrade_token`, `set_pause_flags`, `pause_all`, `get_token_address`, `is_bridge_token`, `is_transfer_finalised`. [2](#0-1) 

None of these functions transfer STRK tokens out of the contract. A grep across all of `starknet/src/` confirms there is no `withdraw`, `rescue`, `sweep`, `recoverToken`, or equivalent function anywhere in the StarkNet source tree. The STRK tokens sent as `native_fee` have no exit path.

The `fin_transfer` function only transfers the bridged asset token to the recipient — it does not touch the accumulated STRK balance. [3](#0-2) 

### Impact Explanation
Every call to `init_transfer` with `native_fee > 0` permanently destroys that STRK value from the user's perspective and from the protocol's perspective. The fees accumulate in the contract with no mechanism for the relayer, admin, or any party to claim them. This matches the allowed critical impact: **permanent freezing / irrecoverable lock of user or protocol funds in bridge flows**.

### Likelihood Explanation
The `native_fee` parameter is a first-class field in the `InitTransfer` event and is part of the documented fee model. Any unprivileged bridge user who calls `init_transfer` with `native_fee > 0` triggers the loss immediately and unconditionally. No special conditions, timing, or attacker knowledge is required — the loss is automatic on every such call.

### Recommendation
Transfer the `native_fee` to a configurable fee-recipient address (e.g., a relayer or treasury address passed as a parameter or stored in contract state) rather than to `get_contract_address()`. Analogously to how the NEAR bridge sends fees to `fee_recipient` in `send_fee_internal`, the StarkNet contract should route native fees to a designated recipient at the time of `init_transfer`.

### Proof of Concept
1. User calls `init_transfer(token_address, 1000, 10, 500, "near:recipient", "")` on the StarkNet `OmniBridge`, where `native_fee = 500` STRK.
2. The contract executes `IERC20Dispatcher { contract_address: strk_token }.transfer_from(caller, get_contract_address(), 500)`. [4](#0-3) 
3. 500 STRK are now held by the `OmniBridge` contract address.
4. No function in `IOmniBridge` or the contract implementation can move these STRK tokens out. [2](#0-1) 
5. The 500 STRK are permanently locked. Repeating for every bridge user compounds the loss linearly with protocol usage.
