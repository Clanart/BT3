### Title
Missing Address-to-Leaf Binding Validation in Transaction Prover Silently Binds Wrong Class Hash and Nonce to Contract Addresses in OS Proof Inputs - (File: crates/starknet_transaction_prover/src/running/storage_proofs.rs)

### Summary

`RpcStorageProofsProvider::get_storage_proofs` in `storage_proofs.rs` and `insert_contract_leaves` in `committer_utils.rs` both assume positional correspondence between `contract_leaves_data[i]` and `contract_addresses[i]` in the RPC proof response, but neither function validates that each returned leaf actually belongs to the corresponding address. If the RPC node returns `contract_leaves_data` in a different order than the requested `contract_addresses` — due to a non-compliant or buggy implementation — the wrong `class_hash`, `nonce`, and `storage_root` are silently bound to each contract address in `extended_initial_reads`, producing wrong OS proof inputs and wrong Patricia trie commitment facts.

### Finding Description

**Root cause — `get_storage_proofs` (storage_proofs.rs:521–528):**

```rust
// Update initial reads with class hashes and nonces from the proof.
// We've validated the lengths match, so this zip is safe.
let mut extended_initial_reads = StateMaps::default();
for (leaf, addr) in
    rpc_proof.contracts_proof.contract_leaves_data.iter().zip(&query.contract_addresses)
{
    extended_initial_reads.class_hashes.insert(*addr, ClassHash(leaf.class_hash));
    extended_initial_reads.nonces.insert(*addr, Nonce(leaf.nonce));
}
```

The only guard is a length check at lines 512–519. The comment "We've validated the lengths match, so this zip is safe" is incorrect: equal lengths guarantee no out-of-bounds access, but do **not** guarantee that `leaf[i]` belongs to `contract_addresses[i]`. No cross-check is performed between the leaf's content and the address it is being assigned to. [1](#0-0) 

**Root cause — `insert_contract_leaves` (committer_utils.rs:187–198):**

```rust
for (leaf, _address) in
    rpc_proof.contracts_proof.contract_leaves_data.iter().zip(&query.contract_addresses)
{
    let contract_state = ContractState { ... };
    let leaf_hash = TreeHashFunctionImpl::compute_leaf_hash(&contract_state);
    insert_leaf_node(db_map, leaf_hash, contract_state, &EmptyKeyContext)?;
}
```

The address is explicitly discarded (`_address`). The leaf is inserted into the `FactsDb` keyed only by its content hash. If `leaf[i]` belongs to a different address than `contract_addresses[i]`, the wrong `ContractState` (class_hash, nonce, storage_root) is stored under the wrong hash, corrupting the FactsDb used for state root computation. [2](#0-1) 

**Root cause — `build_storage_commitment_infos` (storage_proofs.rs:453–458):**

```rust
let addr_to_leaf: HashMap<ContractAddress, _> = query
    .contract_addresses
    .iter()
    .zip(&rpc_proof.contracts_proof.contract_leaves_data)
    .map(|(addr, leaf)| (*addr, leaf))
    .collect();
```

Again, a positional zip with no address-to-leaf binding check. The `storage_root` extracted from the wrong leaf is used as `previous_root` and `updated_root` in the `CommitmentInfo` for each contract's storage trie. [3](#0-2) 

The code itself documents this as an unvalidated assumption in the `fetch_proofs` docstring:

> "This function relies on the Starknet JSON-RPC `get_storage_proof` method returning data in the same order as the input arrays … Validation is performed in `to_storage_proofs` to detect any violations of this assumption." [4](#0-3) 

However, the referenced "validation" only checks lengths, not positional correctness.

**Analog to the external bug:**

The external bug: the Dapp binds session state to Account 1 and never re-checks whether the active MetaMask account has changed to Account 2. The sequencer analog: the prover binds `class_hash`/`nonce`/`storage_root` from `leaf[i]` to `contract_addresses[i]` and never verifies that the leaf actually belongs to that address. In both cases, a stale or mismatched identity is silently accepted and propagated into downstream operations.

### Impact Explanation

`extended_initial_reads` with wrong `class_hash` and `nonce` per contract address is passed directly into `create_commitment_infos_with_state_changes`: [5](#0-4) 

Inside, `create_facts_db_from_storage_proof` uses `extended_initial_reads` to insert storage and compiled-class leaves, and `initial_reads_keys` (derived from `extended_initial_reads`) drives the Patricia trie path traversal for `create_commitment_infos`. Wrong class hashes in `initial_reads_keys` cause the committer to fetch proof paths for the wrong trie leaves, producing wrong `commitment_facts` in the `StateCommitmentInfos` returned to the OS block input. The OS uses these commitment facts to verify that execution read the correct state; wrong facts mean the proof attests to the wrong class hash and nonce for each affected contract. [6](#0-5) 

This falls under: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, and potentially **Critical — wrong class hash or contract code selected for execution** if the mismatched class hash is a valid declared class.

### Likelihood Explanation

The trigger requires the RPC node to return `contract_leaves_data` in a different order than the requested `contract_addresses`. This can occur with:

1. A non-compliant Starknet full node implementation (e.g., a custom node or a node with a sorting bug in `get_storage_proof`).
2. After chunking: `merge_storage_proofs` extends `contract_leaves_data` in chunk order (line 142). If a future change to `split_query` or `flatten_query` alters the ordering of contract addresses across chunks, the merged leaves would be in a different order than the original `contract_addresses` list. [7](#0-6) 

The root cause is entirely within the sequencer's own validation logic — the missing binding check is a sequencer-side defect, not a dependency-only behavior.

### Recommendation

After receiving the RPC proof, validate that each `ContractLeafData` actually belongs to the corresponding `ContractAddress` by verifying the Patricia trie membership proof for each leaf. At minimum, cross-check the leaf hash against the trie node at the contract's address:

1. In `get_storage_proofs`, after the length check, verify that `TreeHashFunctionImpl::compute_leaf_hash(&ContractState { nonce: leaf.nonce, class_hash: leaf.class_hash, storage_root: leaf.storage_root })` matches the leaf hash stored at `contract_addresses[i]` in the contracts trie (derivable from the `contracts_proof.nodes`).

2. In `insert_contract_leaves`, use the address (currently `_address`) to verify the leaf hash against the trie node at that address before inserting.

3. In `build_storage_commitment_infos`, replace the positional zip with an address-keyed lookup that verifies the leaf belongs to the address.

### Proof of Concept

```
1. Construct a VirtualBlockExecutionData with two contract addresses:
   - addr_A = 0x100 (class_hash_A, nonce_A, storage_root_A)
   - addr_B = 0x200 (class_hash_B, nonce_B, storage_root_B)

2. Call RpcStorageProofsProvider::get_storage_proofs with this data.

3. The RPC node (non-compliant) returns contract_leaves_data in reversed order:
   - contract_leaves_data[0] = { class_hash: class_hash_B, nonce: nonce_B, storage_root: storage_root_B }
   - contract_leaves_data[1] = { class_hash: class_hash_A, nonce: nonce_A, storage_root: storage_root_A }
   - contract_addresses in query = [addr_A, addr_B]

4. The length check at lines 512-519 passes (2 == 2).

5. The zip at lines 524-528 produces:
   - extended_initial_reads.class_hashes[addr_A] = class_hash_B  ← WRONG
   - extended_initial_reads.nonces[addr_A]       = nonce_B        ← WRONG
   - extended_initial_reads.class_hashes[addr_B] = class_hash_A  ← WRONG
   - extended_initial_reads.nonces[addr_B]       = nonce_A        ← WRONG

6. These wrong values are passed to create_commitment_infos_with_state_changes,
   which uses them to build the FactsDb and compute commitment_facts.

7. The OS block input now contains wrong class_hash and nonce for both contracts,
   causing the SNOS proof to attest to the wrong contract state.
``` [8](#0-7) [9](#0-8)

#[async_trait]
impl StorageProofProvider for RpcStorageProofsProvider {
    async fn get_storage_proofs(
        &self,
        block_number: BlockNumber,
        execution_data: &VirtualBlockExecutionData,
        config: &StorageProofConfig,
    ) -> Result<StorageProofs, ProofProviderError> {
        let query = Self::prepare_query(execution_data);

        let rpc_proof = self.fetch_proofs(block_number, &query).await?;

        // Validate that contract_leaves_data matches contract_addresses length.
        let leaves_len = rpc_proof.contracts_proof.contract_leaves_data.len();
        let addresses_len = query.contract_addresses.len();
        if leaves_len != addresses_len {
            return Err(ProofProviderError::InvalidProofResponse(format!(
                "Contract leaves length mismatch: expected {addresses_len} leaves for requested \
                 contracts, got {leaves_len}"
            )));
        }

        // Update initial reads with class hashes and nonces from the proof.
        // We've validated the lengths match, so this zip is safe.
        let mut extended_initial_reads = StateMaps::default();
        for (leaf, addr) in
            rpc_proof.contracts_proof.contract_leaves_data.iter().zip(&query.contract_addresses)
        {
            extended_initial_reads.class_hashes.insert(*addr, ClassHash(leaf.class_hash));
            extended_initial_reads.nonces.insert(*addr, Nonce(leaf.nonce));
        }

        // Include storage values from execution.
        extended_initial_reads.storage.extend(&execution_data.initial_reads.storage);

        let commitment_infos = match config.include_state_changes {
            true => {
                Self::create_commitment_infos_with_state_changes(
                    &rpc_proof,
                    &query,
                    &extended_initial_reads,
                    &execution_data.state_diff,
                )
                .await?
            }
            false => Self::create_commitment_infos_without_state_changes(&rpc_proof, &query)?,
        };

        Ok(StorageProofs { extended_initial_reads, commitment_infos })
    }

pub(crate) fn create_facts_db_from_storage_proof(
    rpc_proof: &RpcStorageProof,
    query: &RpcStorageProofsQuery,
    initial_reads: &StateMaps,
) -> Result<FactsDb<MapStorage>, ProofProviderError> {
    let mut db_map: DbHashMap = DbHashMap::new();

    insert_inner_nodes(&mut db_map, &rpc_proof.contracts_proof.nodes)?;
    insert_inner_nodes(&mut db_map, &rpc_proof.classes_proof)?;
    for storage_proof in &rpc_proof.contracts_storage_proofs {
        insert_inner_nodes(&mut db_map, storage_proof)?;
    }

    insert_contract_leaves(&mut db_map, rpc_proof, query)?;
    insert_storage_leaves(&mut db_map, initial_reads)?;
    insert_compiled_class_leaves(&mut db_map, initial_reads)?;

    // Add dummy nodes for orphan hashes (sibling hashes without preimages).
    add_dummy_nodes_for_orphan_hashes(&mut db_map, &rpc_proof.contracts_proof.nodes)?;
    add_dummy_nodes_for_orphan_hashes(&mut db_map, &rpc_proof.classes_proof)?;
    for storage_proof in &rpc_proof.contracts_storage_proofs {
        add_dummy_nodes_for_orphan_hashes(&mut db_map, storage_proof)?;
    }

    Ok(FactsDb::new(MapStorage(db_map)))
}
