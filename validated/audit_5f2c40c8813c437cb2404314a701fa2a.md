### Title
Unauthenticated BOOTSTRAP Declare Bypass Skips Signature Verification, Nonce, and Fee Enforcement — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` contains a special-case branch that, when `sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and all resource bounds are zero, **completely skips** `__validate_declare__` (signature verification), nonce increment, and fee charging, and directly writes a new class hash into `contract_class_changes`. Because `'BOOTSTRAP'` is merely a Cairo felt literal, any unprivileged user can craft a declare transaction satisfying these conditions. The OS will accept it as valid, allowing unauthorized class declarations with zero fees and no account ownership required.

---

### Finding Description

In `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the following branch executes:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

This early return bypasses three critical enforcement steps that appear later in the normal declare flow:

1. **`check_and_increment_nonce`** — nonce replay protection is skipped entirely.
2. **`non_reverting_select_execute_entry_point_func` with `VALIDATE_DECLARE_ENTRY_POINT_SELECTOR`** — the account's `__validate_declare__` entry point (signature check) is never called.
3. **`charge_fee`** — no fee is deducted from the sender. [2](#0-1) 

The condition `sender_address == 'BOOTSTRAP'` compares against the ASCII felt encoding of the string `"BOOTSTRAP"`. There is no on-chain ownership check, no deployed-contract check, and no privileged-key check. The `sender_address` field is loaded from the sequencer hint (`%{ DeclareTxFields %}`), and the transaction hash is computed over it — but the hash computation does not enforce that the sender is authorized; it only binds the fields together. [3](#0-2) 

The normal declare path calls `compute_declare_transaction_hash` and then `AssertTransactionHash`, which verifies the hash matches the sequencer's record. A malicious sequencer (or a sequencer that accepts the attacker's transaction) can include a declare with `sender_address='BOOTSTRAP'`, `nonce=0`, `version=3`, and zero resource bounds. The OS will compute the hash, verify it matches, enter the BOOTSTRAP branch, and write the class hash — all without any signature.

The `execute_transactions_inner` dispatcher routes any transaction typed `'DECLARE'` to `execute_declare_transaction` with no pre-check on the sender address: [4](#0-3) 

---

### Impact Explanation

**Direct loss of funds (Critical):** The sequencer is entitled to collect fees for every transaction it processes. The BOOTSTRAP path sets `max_possible_fee = 0` as a precondition and then returns before `charge_fee` is ever called. An attacker who submits (or colludes with a sequencer to include) BOOTSTRAP declare transactions causes the sequencer to perform real computational work — Patricia trie updates, class hash finalization, proof generation — for zero compensation. At scale, this is a direct, provable loss of funds from the sequencer's fee revenue.

**Unauthorized state mutation:** The `dict_update` on `contract_class_changes` with `prev_value=0` permanently registers a new `class_hash → compiled_class_hash` mapping in the protocol state. Once declared, a class cannot be undeclared. An attacker can populate the class registry with attacker-controlled `compiled_class_hash` values for chosen `class_hash` keys. Any subsequent `deploy_syscall` or `replace_class` syscall that references one of these class hashes will execute the attacker's compiled class, enabling fund theft from contracts that are later deployed or upgraded to that class. [5](#0-4) 

---

### Likelihood Explanation

The conditions are trivially satisfiable by any user:
- `sender_address = 'BOOTSTRAP'` — a known felt constant, no key required.
- `nonce = 0` — always satisfiable on first use.
- `version = 3` — the only supported non-deprecated version.
- All resource bounds = 0 — explicitly setting `max_amount = 0` for L1_GAS, L2_GAS, L1_DATA_GAS satisfies `max_possible_fee == 0`.

No privileged access, no leaked key, and no Sybil attack is required. The attacker only needs to submit a well-formed declare transaction to a sequencer. Because the OS itself accepts the transaction as valid, any sequencer that does not apply an out-of-band filter will include it.

---

### Recommendation

**Short term:** Add an explicit check that the `sender_address` is not the felt literal `'BOOTSTRAP'` unless the transaction originates from a trusted, operator-controlled source — or, more robustly, remove the BOOTSTRAP special case entirely and require all declare transactions to pass `__validate_declare__`. If the bootstrap flow is necessary for system initialization, gate it on a block-number range or a one-time flag in `BlockContext` rather than on a guessable felt value.

**Long term:** Document every special-case transaction path (BOOTSTRAP, version-0 skips, L1 handler nonce-free paths) with explicit threat-model analysis. Add OS-level integration tests that assert BOOTSTRAP-style transactions submitted by non-operator accounts are rejected.

---

### Proof of Concept

1. Attacker selects any valid Sierra class `C` and computes its `class_hash` and `compiled_class_hash` using the standard Poseidon-based `finalize_class_hash` algorithm.
2. Attacker constructs a declare transaction:
   - `tx_type = 'DECLARE'`
   - `sender_address = 'BOOTSTRAP'` (felt `0x424f4f545354524150`)
   - `nonce = 0`
   - `version = 3`
   - `resource_bounds = [{max_amount:0, max_price:0}, {max_amount:0, max_price:0}, {max_amount:0, max_price:0}]`
   - `class_hash = <hash of C>`, `compiled_class_hash = <casm hash of C>`
3. Attacker submits the transaction. The sequencer computes the transaction hash (binding all fields), includes it in a block, and provides the hint values.
4. The OS executes `execute_declare_transaction`:
   - Computes and verifies the transaction hash. ✓
   - Enters the `if (sender_address == 'BOOTSTRAP' ...)` branch. ✓
   - Calls `compute_max_possible_fee` → returns 0. ✓
   - Calls `dict_update(contract_class_changes, key=class_hash, prev_value=0, new_value=compiled_class_hash)`. ✓
   - Returns — **no signature check, no nonce increment, no fee charged**.
5. The class is now permanently registered in the protocol state. The attacker paid zero fees. The sequencer received zero compensation. Any future `deploy` or `replace_class` referencing this `class_hash` will execute the attacker's code. [6](#0-5)

        %{ SkipTx %}
        return ();

    local class_hash_ptr: felt*;
    local compiled_class_hash;
    local account_deployment_data_size;
    local account_deployment_data: felt*;
    %{ DeclareTxFields %}
    let common_tx_fields = get_account_tx_common_fields(
        block_context=block_context,
        tx_hash_prefix=DECLARE_HASH_PREFIX,
        sender_address=sender_address,
    );

    let poseidon_ptr = builtin_ptrs.selectable.poseidon;
    with poseidon_ptr {
        // Compute transaction hash.
        let transaction_hash = compute_declare_transaction_hash(
            common_fields=common_tx_fields,
            class_hash=[class_hash_ptr],
            compiled_class_hash=compiled_class_hash,
            account_deployment_data_size=account_deployment_data_size,
            account_deployment_data=account_deployment_data,

        // Ensure the given class hash is a result of a Sierra class hash calculation.

        let expected_class_hash = finalize_class_hash(
            contract_class_component_hashes=contract_class_component_hashes
        );
        with_attr error_message("Invalid class hash pre-image.") {
            assert [class_hash_ptr] = expected_class_hash;
        }
    }
    update_poseidon_in_builtin_ptrs(poseidon_ptr=poseidon_ptr);

    // Get the account transaction info.

        account_deployment_data_size=account_deployment_data_size,
        account_deployment_data=account_deployment_data,
        proof_facts_size=0,
        proof_facts=cast(0, felt*),
        tx_info_dst=tx_info,
        deprecated_tx_info_dst=deprecated_tx_info,
    );

    // Do not run validate or perform any account-related actions for declare transactions that

    // This flow is used for the sequencer to bootstrap a new system.

                key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
            );
            %{ SkipTx %}
            return ();
