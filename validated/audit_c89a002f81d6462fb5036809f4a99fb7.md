### Title
`deployToken` / `deploy_token` Signature Lacks Destination-Chain Domain Separation, Enabling Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The MPC-signed `MetadataPayload` used by `deployToken` (EVM), `deploy_token` (Starknet), and `deploy_token` (Solana) does not include any destination-chain identifier in the signed message. A valid signature produced by the NEAR MPC signer for one chain can be replayed verbatim to any other supported chain, causing unauthorized bridge-token deployment on chains where the token was never intended to be deployed. This permanently blocks legitimate future deployment on those chains and creates a state desync between NEAR and the replayed chain.

---

### Finding Description

The `finTransfer` path correctly binds the signed message to a specific chain by embedding `omniBridgeChainId` twice in the borsh-encoded payload: [1](#0-0) 

The `deployToken` path, however, constructs its signed message from only `PayloadType.Metadata || token || name || symbol || decimals` — no chain identifier whatsoever: [2](#0-1) 

The same omission exists in the Starknet `MetadataPayload::to_borsh()`, which serializes only the payload type, token string, name, symbol, and decimals — no `omni_bridge_chain_id`: [3](#0-2) 

Contrast this with `TransferMessagePayload::to_borsh()` on Starknet, which correctly receives and embeds `chain_id`: [4](#0-3) 

The Solana `DeployTokenPayload::serialize_for_near()` similarly omits any chain identifier from the signed bytes: [5](#0-4) 

Whereas `FinalizeTransferPayload::serialize_for_near()` on Solana correctly writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` before the token and recipient fields: [6](#0-5) 

The NEAR-side `MetadataPayload` struct itself also contains no chain field: [7](#0-6) 

---

### Impact Explanation

**Impact: High — Cross-chain replay enabling unauthorized token deployment and permanent blocking of legitimate deployment.**

The NEAR MPC signer produces one signature over `{prefix=1, token, name, symbol, decimals}`. Because no chain ID is bound into the message, this single signature is valid on every EVM chain (Ethereum, Base, Arbitrum, BNB, Polygon, HyperEVM, Abstract), on Starknet, and on Solana simultaneously.

An attacker who observes a valid `deployToken` signature on-chain (e.g., from an Ethereum transaction) can immediately replay it to every other chain. On each replayed chain:

1. A bridge token is deployed by the bridge contract (so it passes the `isBridgeToken` check).
2. `nearToEthToken[metadata.token]` is populated with the newly deployed address.
3. The NEAR bridge has no record of this deployment on the replayed chain, so no `finTransfer` will ever be signed for it — the token is permanently orphaned.
4. When the legitimate relayer later attempts to deploy the same token on that chain, the call reverts with `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`, permanently blocking the correct deployment. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

**Likelihood: High.**

- The MPC-signed metadata event is emitted publicly on NEAR and on the first chain where `deployToken` is called. Any observer can extract the signature and replay it.
- The bridge is deployed on multiple EVM chains sharing the same MPC-derived address (`nearBridgeDerivedAddress` / `omni_bridge_derived_address`), making cross-chain replay trivially possible.
- No special access or privileged role is required; the attacker only needs to submit a transaction on the target chain.
- The attack is permanent: once the token is deployed on a chain via replay, the legitimate deployment path is irrecoverably blocked.

---

### Recommendation

Embed the destination chain identifier into the `MetadataPayload` signed message, mirroring the pattern already used in `finTransfer`. Concretely:

- **EVM**: include `bytes1(omniBridgeChainId)` in the borsh-encoded bytes before or after the token string in `deployToken`.
- **Starknet**: pass `omni_bridge_chain_id` into `MetadataPayload::to_borsh()` and append it, as is already done in `TransferMessagePayload::to_borsh(chain_id)`.
- **Solana**: write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into `DeployTokenPayload::serialize_for_near()`, as is already done in `FinalizeTransferPayload::serialize_for_near()`.
- **NEAR signing**: add a `destination_chain: ChainKind` field to `MetadataPayload` so the MPC signer produces a distinct signature per target chain.

---

### Proof of Concept

1. Legitimate relayer calls `deployToken(sig, {token:"wrap.near", name:"Wrapped NEAR", symbol:"wNEAR", decimals:24})` on Ethereum. Transaction is mined; `sig` is now public.
2. Attacker submits the identical call `deployToken(sig, {token:"wrap.near", name:"Wrapped NEAR", symbol:"wNEAR", decimals:24})` to the Base OmniBridge.
3. Base OmniBridge reconstructs `keccak256(borsh(Metadata || "wrap.near" || "Wrapped NEAR" || "wNEAR" || 24))` — identical bytes to Ethereum — and `ECDSA.recover` returns `nearBridgeDerivedAddress`. Signature check passes.
4. A bridge token for `wrap.near` is deployed on Base, `nearToEthToken["wrap.near"]` is set, `isBridgeToken[newToken] = true`.
5. NEAR bridge has no record of this Base deployment; no `finTransfer` will ever be signed for it.
6. When the legitimate relayer later tries to deploy `wrap.near` on Base, the call reverts: `require(!isBridgeToken[nearToEthToken["wrap.near"]], "ERR_TOKEN_EXIST")`.
7. The Base bridge for `wrap.near` is permanently broken.
