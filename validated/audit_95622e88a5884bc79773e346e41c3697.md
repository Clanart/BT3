### Title
`ProofOfInclusion::valid()` Does Not Verify Against an External Trusted Root — Forged Inclusion Proofs Always Pass — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks the internal hash-chain consistency of the proof structure. It never compares the computed root against any externally-supplied trusted root. Because the final equality check inside `valid()` is a tautology (it compares `existing_hash` against `self.root_hash()`, which is defined as the last `combined_hash` that `existing_hash` was just set to), an attacker can craft a completely fabricated `ProofOfInclusion` — claiming any arbitrary key-value pair is included in any tree — and `valid()` will return `true`.

### Finding Description

`ProofOfInclusion` is a `Streamable` type, meaning it is serialized and deserialized across trust boundaries. Its sole verification API is `valid()`:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautology
}
``` [1](#0-0) 

The `root_hash()` helper is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which was just verified to equal `layer.combined_hash`. `root_hash()` returns that same `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` when the loop body does not short-circuit. The final line adds zero security.

The struct itself carries no external root field:

```rust
pub struct ProofOfInclusion {
    pub node_hash: Hash,
    pub layers: Vec<ProofOfInclusionLayer>,
}
``` [3](#0-2) 

An attacker who controls a serialized `ProofOfInclusion` can:

1. Set `node_hash` to the hash of any fabricated leaf (any key/value pair they wish to claim is present).
2. Build a chain of `ProofOfInclusionLayer` entries where each `combined_hash` is correctly computed from the previous hash and a chosen `other_hash` — this is trivially achievable with a single SHA-256 call per layer.
3. Submit this proof to any verifier that calls `proof.valid()`.

`valid()` returns `true` for this fabricated proof because the internal chain is consistent. The verifier has no way to distinguish it from a genuine proof without separately comparing `proof.root_hash()` against a known trusted root — a step the API does not enforce and that callers in the test suite do not perform:

```python
proof_of_inclusion = merkle_blob.get_proof_of_inclusion(kv_id)
assert proof_of_inclusion.valid()   # no root check
``` [4](#0-3) 

```rust
let proof = blob.get_proof_of_inclusion(key).unwrap();
assert!(proof.valid());   // no root check
``` [5](#0-4) 

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

Any DataLayer consumer that receives a `ProofOfInclusion` from an untrusted peer and calls only `valid()` to verify it will accept a completely fabricated proof. This allows an attacker to assert that arbitrary key-value data is present in a DataLayer store whose actual root they do not control — corrupting the authenticated state visible to that consumer.

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type explicitly designed for cross-node transmission. The `valid()` method is the only verification entry point exposed by the type. Its name strongly implies completeness. Any developer integrating DataLayer proof verification is likely to call `valid()` and consider the check done, because the API provides no other mechanism and the final equality check inside `valid()` appears to confirm the root. The tautological nature of that final check is non-obvious without careful reading.

### Recommendation

`valid()` must accept an external trusted root hash and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    // existing internal-consistency loop ...
    existing_hash == *trusted_root
}
```

Alternatively, remove the misleading `valid()` method and expose only `root_hash()`, requiring callers to perform the comparison explicitly. All call sites — including Python bindings, fuzz targets, and tests — must be updated to supply and check the trusted root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};
use chia_sha2::Sha256;

fn forge_proof(claimed_node_hash: Hash, fake_sibling: Hash) -> ProofOfInclusion {
    // Compute a fake combined_hash that is internally consistent
    let combined = calculate_internal_hash(&claimed_node_hash, Side::Left, &fake_sibling);
    ProofOfInclusion {
        node_hash: claimed_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: fake_sibling,
            combined_hash: combined,   // attacker computes this trivially
        }],
    }
}

fn main() {
    let fake_node_hash = [0xAA; 32];
    let fake_sibling   = [0xBB; 32];
    let proof = forge_proof(fake_node_hash, fake_sibling);

    // valid() returns true for a completely fabricated proof
    assert!(proof.valid(), "forged proof accepted");
    // proof.root_hash() is the attacker-controlled combined_hash,
    // not the real tree root — but valid() never checks the real root.
}
