### Title
`batch_insert` Skips Duplicate-Key Validation, Silently Corrupting the Merkle Tree Root and Enabling Forged Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `insert` enforces. When a batch contains a key already present in the tree (or a key that appears twice within the same batch), the function silently writes a second leaf node for that key into the blob, overwrites the cache entry for that key, and leaves the first leaf as an unreachable orphan that still contributes to the computed Merkle root. The resulting root hash is wrong, and any proof of inclusion generated against it is forged.

### Finding Description

`MerkleBlob::insert` opens with two guards: [1](#0-0) 

These guards prevent a key or leaf hash from appearing twice in the tree.

`batch_insert` calls `insert` (with its guards) only for the last two items popped from the vector when the tree currently has ≤ 1 leaf: [2](#0-1) 

Every other item — the entire batch when `leaf_count > 1`, and all items except the last two when `leaf_count ≤ 1` — is written directly via `insert_entry_to_blob` with no duplicate check: [3](#0-2) 

`insert_entry_to_blob` writes the raw block bytes to the blob and then calls `block_status_cache.add_leaf`, which calls `HashMap::insert` on `key_to_index` and `leaf_hash_to_index`: [4](#0-3) 

`HashMap::insert` silently overwrites the previous entry and returns the displaced value — which is discarded: [5](#0-4) 

**Result:** after a duplicate-key batch insert:

1. The blob contains **two leaf nodes** with the same `KeyId` at different `TreeIndex` positions.
2. The `key_to_index` cache points only to the **second** leaf; the first leaf is orphaned — not in `free_indexes`, not reachable by key, but physically present in the blob.
3. The batch subtree is built from all inserted indexes (including the duplicate) and attached to the existing tree, so the orphaned leaf's hash is folded into every ancestor's `internal_hash` computation all the way to the root.
4. The stored root hash is therefore wrong: it commits to data that the cache does not acknowledge.

This is the direct analog of the `witch.sol` bug: a second state-changing call overwrites the tracking record for the first, leaving the first's effects intact in the underlying store while making them invisible to all subsequent logic.

### Impact Explanation

- `get_proof_of_inclusion` looks up the leaf via `key_to_index` (finding only the second leaf), then walks the lineage to the root. The root hash it encounters was computed including the orphaned first leaf, so the proof is built against a corrupted root.
- `get_root_hash` returns the corrupted value, which is what the Chia node commits on-chain for the DataLayer store.
- A verifier checking a proof against the on-chain root will accept proofs that reflect the corrupted state, enabling forged inclusion or exclusion proofs.
- `check_integrity` will detect the inconsistency (leaf count in blob ≠ cache length), but only if explicitly called; it is not called automatically after `batch_insert`. [6](#0-5) 

### Likelihood Explanation

`batch_insert` is exposed directly to Python callers: [7](#0-6) [8](#0-7) 

Any caller that passes a batch containing a key already present in the tree (or a key repeated within the batch) triggers the bug. This can happen through programming error or deliberate malicious input. The tree must have ≥ 2 leaves for the entire batch to bypass guards, which is the normal operating state of any non-trivial DataLayer store.

### Recommendation

Add the same duplicate-key and duplicate-hash guards at the top of the unchecked loop in `batch_insert`, mirroring what `insert` does:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing code
}
```

Alternatively, build a local `HashSet` of keys and hashes seen within the current batch before writing anything, and reject the entire batch if any duplicate is found.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
from chia_protocol import bytes32
import hashlib

def h(n: int) -> bytes32:
    return bytes32(hashlib.sha256(n.to_bytes(8, "big")).digest())

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 3 leaves so leaf_count > 1 (bypasses the guarded path)
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), h(i))
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with a key (KeyId(0)) already in the tree — no error raised
blob.batch_insert(
    [(KeyId(0), ValueId(999))],   # duplicate key
    [h(999)],
)
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# root_after is corrupted: it includes the orphaned original leaf for KeyId(0)
# AND the new leaf for KeyId(0), but the cache only knows about the new one.
assert root_before != root_after  # root changed

# check_integrity reveals the inconsistency
try:
    blob.check_integrity()
    print("integrity OK (unexpected)")
except Exception as e:
    print(f"integrity failure: {e}")
    # IntegrityKeyToIndexCacheIndex or IntegrityKeyToIndexCacheLength
```

The `batch_insert` call succeeds without error. `check_integrity` then reports a mismatch between the blob's leaf count and the cache's `key_to_index` length, confirming the corrupted state. Any `get_proof_of_inclusion` call for `KeyId(0)` after this point returns a proof anchored to the wrong root.

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
