### Title
Silent Proof Manager Storage Failure in Gateway Permanently Loses Proof, Blocking Block Proposal — (`crates/apollo_gateway/src/gateway.rs`)

---

### Summary

When `store_proof_in_proof_manager` fails inside `store_proof_and_spawn_archiving`, the gateway logs the error and silently continues, sending the transaction to the mempool. The proof is permanently absent from the local `ProofManager`. Later, when the proposer tries to broadcast a block containing that transaction, `convert_internal_rpc_tx_to_rpc_tx` calls `get_proof`, receives `None`, and returns `ProofNotFound` — permanently blocking the block proposal for any block that includes this transaction.

---

### Finding Description

In `gateway.rs`, `store_proof_and_spawn_archiving` calls `store_proof_in_proof_manager`. On failure the match arm only logs:

```rust
Err(e) => {
    error!("Failed to set proof in proof manager: {}", e);
}
``` [1](#0-0) 

Execution then falls through: the GCS archive task is still spawned and the transaction is forwarded to the mempool. [2](#0-1) 

The `ProofManager` is the authoritative local store for the consensus flow. When the proposer later converts an internal transaction back to a consensus transaction for broadcasting, `convert_internal_rpc_tx_to_rpc_tx` unconditionally calls `get_proof` for any non-empty `proof_facts`:

```rust
let proof = if tx.proof_facts.is_empty() {
    Proof::default()
} else {
    self.get_proof(&tx.proof_facts).await?   // ← returns ProofNotFound
};
``` [3](#0-2) 

`get_proof` maps a `None` result to `TransactionConverterError::ProofNotFound`: [4](#0-3) 

There is no retry path and no mechanism to re-populate the `ProofManager` from the GCS archive. The proof is permanently lost.

---

### Impact Explanation

A transaction with proof facts that passed gateway validation (proof cryptographically verified) is admitted to the mempool. The batcher picks it up, the blockifier executes it (proof-facts validation in `perform_pre_validation_stage` reads from on-chain state, not the proof manager), and the transaction is included in the proposed block. When the proposer attempts to stream that block to validators via `convert_internal_consensus_tx_to_consensus_tx`, the `ProofNotFound` error propagates, the block proposal cannot be broadcast, and consensus stalls for that height. The transaction is permanently unsequenceable with no user-visible error and no retry mechanism.

This matches the **High** impact: *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing* — specifically, a transaction is admitted that the sequencer can never successfully propose.

---

### Likelihood Explanation

Any transient I/O error on the filesystem backing `FsProofStorage` (disk full, permission error, kernel interrupt) during `write_proof_atomically` will trigger this path. The gateway is a hot path; a single such failure permanently poisons the mempool entry. No operator action can recover the proof without restarting and resubmitting the transaction (which the mempool may reject as a duplicate nonce). [5](#0-4) 

---

### Recommendation

Replace the silent `error!` log with a hard failure: return an error from `store_proof_and_spawn_archiving` and propagate it through `add_tx_inner` so the gateway rejects the transaction. This mirrors the VUSD recommendation of "send VUSD back to the user if the call fails" — here, reject the transaction at the gateway boundary rather than accepting it into a state from which it can never be sequenced.

Alternatively, implement a retry loop (analogous to `retry_until_base_layer_success` used in the L1 scraper) before giving up on proof manager storage. [6](#0-5) 

---

### Proof of Concept

1. Submit an Invoke V3 transaction carrying valid `proof_facts` and `proof` to the gateway.
2. Gateway verifies the proof successfully (inside `convert_rpc_tx_to_internal_and_executable_txs`).
3. `store_proof_in_proof_manager` encounters an `FsProofStorageError::IoError` (e.g., inject a disk-full condition on the proof storage directory).
4. The `Err` arm at line 263–265 logs the error; execution continues.
5. The transaction is forwarded to the mempool and accepted.
6. The batcher proposes a block containing this transaction; the blockifier executes it successfully (proof-facts validation reads from on-chain state).
7. The proposer calls `convert_internal_rpc_tx_to_rpc_tx` to stream the block to validators.
8. `get_proof` returns `Ok(None)` → `Err(ProofNotFound)`.
9. The block proposal fails; consensus cannot advance at this height.
10. The transaction remains in the mempool permanently with no retry or eviction path.
