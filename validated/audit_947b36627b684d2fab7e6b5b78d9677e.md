### Title
`execute_send_message_to_l1` Charges Flat Gas Regardless of Payload Size, Undercounting Actual OS Work - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_send_message_to_l1` syscall handler charges a flat `SEND_MESSAGE_TO_L1_GAS_COST = 14470` regardless of the message payload size, while the actual OS work (`memcpy` of the payload into the output segment) scales linearly with `payload_size`. This is directly analogous to the external report: a variable-length data component is excluded from gas accounting. An unprivileged contract can send arbitrarily large L1 messages paying only the flat fee, forcing the OS prover to perform unbounded extra work at no additional cost to the attacker.

### Finding Description

In `execute_send_message_to_l1`, the gas deduction is performed before the payload is read:

```cairo
let success = reduce_syscall_gas_and_write_response_header(
    total_gas_cost=SEND_MESSAGE_TO_L1_GAS_COST, request_struct_size=SendMessageToL1Request.SIZE
);
```

After the flat fee is charged, the function computes `payload_size` and performs a `memcpy` of that many felts into the OS output segment:

```cairo
tempvar payload_size = request.payload_end - payload_start;
memcpy(dst=outputs.messages_to_l1 + MessageToL1Header.SIZE, src=payload_start, len=payload_size);
```

The `memcpy` consumes O(`payload_size`) Cairo VM steps, which are part of the proof. No per-element gas factor is applied.

Contrast this with `execute_deploy` and `execute_meta_tx_v0`, which both correctly apply a per-element factor for variable-length data:

```cairo
// execute_deploy:
let specific_base_gas_cost = DEPLOY_GAS_COST + DEPLOY_CALLDATA_FACTOR_GAS_COST * constructor_calldata_size;

// execute_meta_tx_v0:
let specific_base_gas_cost = META_TX_V0_GAS_COST + META_TX_V0_CALLDATA_FACTOR_GAS_COST * calldata_size;
```

`DEPLOY_CALLDATA_FACTOR_GAS_COST = 4850` and `META_TX_V0_CALLDATA_FACTOR_GAS_COST = 4850` are defined precisely to cover the OS-level per-element processing cost. `SEND_MESSAGE_TO_L1_GAS_COST` has no equivalent factor.

There is also no explicit bound check on `payload_size` inside `execute_send_message_to_l1`.

### Impact Explanation

An attacker deploys a contract that calls `send_message_to_l1` with a very large payload array. The user's L2 gas budget covers the cost of allocating the array in their contract, but the OS-level `memcpy` of that payload into the proof output is charged only the flat `14470` gas units regardless of payload size. The sequencer/prover must execute and prove the full `memcpy` at no additional cost to the attacker. By submitting many such transactions, an attacker can make blocks disproportionately expensive to prove relative to the fees collected, potentially causing the network to be unable to confirm new transactions (total network shutdown / block production halt).

### Likelihood Explanation

Any unprivileged contract can call `send_message_to_l1` with an arbitrarily large payload. No special role or privilege is required. The attacker only needs to pay the flat syscall fee plus the L2 gas for allocating the payload array in their own contract execution. The attack is cheap relative to the OS proving cost imposed.

### Recommendation

Apply a per-element gas factor to `send_message_to_l1` proportional to `payload_size`, consistent with the pattern used in `execute_deploy` and `execute_meta_tx_v0`:

```cairo
let specific_base_gas_cost = SEND_MESSAGE_TO_L1_GAS_COST +
    SEND_MESSAGE_TO_L1_PAYLOAD_FACTOR_GAS_COST * payload_size;
let (success, remaining_gas) = reduce_syscall_base_gas(
    specific_base_gas_cost=specific_base_gas_cost,
    request_struct_size=SendMessageToL1Request.SIZE
);
```

Also add a bound check on `payload_size` (e.g., `assert_nn_le(payload_size, SIERRA_ARRAY_LEN_BOUND - 1)`) before the gas deduction.

### Proof of Concept

1. Deploy a Cairo 1 contract with an entry point that calls `send_message_to_l1` with a payload of N felts (e.g., N = 100,000).
2. Submit an invoke transaction calling that entry point. The user's L2 gas budget covers the contract-side array allocation (O(N) steps at the contract level), and the OS charges exactly `SEND_MESSAGE_TO_L1_GAS_COST = 14470` for the syscall.
3. The OS `execute_send_message_to_l1` then executes `memcpy(..., len=N)`, consuming O(N) additional Cairo steps in the OS program that are not charged to the user.
4. Repeat across many transactions in a block. Each transaction imposes O(N) unpriced OS steps, making the block proof arbitrarily more expensive than the fees collected, eventually halting block production.
