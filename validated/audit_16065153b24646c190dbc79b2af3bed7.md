### Title
No User-Callable Cancel/Refund Mechanism for Pending NEAR→EVM Transfers After Tokens Are Burned - (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR→EVM transfer of a deployed (bridged) token, `init_transfer_internal` immediately and irrevocably burns the tokens on NEAR and stores the transfer in `pending_transfers`. There is no user-callable function to cancel the pending transfer and recover the burned tokens if the transfer cannot be finalized on the destination chain. This is a direct analog to H-11: the absence of a refund/cancel mechanism means any condition that prevents destination-chain finalization results in permanent, unrecoverable token loss for the user.

### Finding Description

The NEAR→EVM transfer flow for deployed (bridged) tokens proceeds as follows:

1. User calls `ft_transfer_call` → `ft_on_transfer` → `init_transfer` → `init_transfer_internal`
2. Inside `init_transfer_internal`, tokens are **immediately burned** via `burn_tokens_if_needed` (for deployed tokens) and the transfer is stored in `pending_transfers`
3. A relayer calls `sign_transfer` to obtain an MPC signature
4. The user/relayer submits the signature to the EVM bridge to finalize
5. The relayer calls `claim_fee` with proof of EVM finalization, which removes the entry from `pending_transfers`

The burn in step 2 is unconditional and irreversible:

```rust
// near/omni-bridge/src/lib.rs:1850-1857
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(
        transfer_message.get_destination_chain(),
        &token_id,
        transfer_message.amount.0,
    );
}
```

`burn_tokens_if_needed` fires a detached (fire-and-forget) burn cross-contract call with no rollback path:

```rust
// near/omni-bridge/src/lib.rs:1806-1813
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();
    }
}
```

The only function that removes a transfer from `pending_transfers` is `claim_fee_callback`, which requires a valid proof that the transfer was finalized on the destination chain:

```rust
// near/omni-bridge/src/lib.rs:1087-1094
require!(
    self.factories
        .get(&fin_transfer.emitter_address.get_chain())
        == Some(fin_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
...
let transfer_message = self.remove_transfer_message(fin_transfer.transfer_id);
```

There is no `cancel_transfer`, `withdraw_pending_transfer`, or any other user-callable function that would re-mint the burned tokens to the original sender if the transfer cannot be completed.

The `sign_transfer_callback` also does nothing on MPC failure — the transfer simply stays in `pending_transfers` indefinitely:

```rust
// near/omni-bridge/src/lib.rs:649-668
pub fn sign_transfer_callback(
    &mut self,
    #[callback_result] call_result: Result<SignatureResponse, PromiseError>,
    ...
) {
    if let Ok(signature) = call_result {
        if fee.is_zero() {
            self.remove_transfer_message(message_payload.transfer_id);
        }
        // emit event
    }
    // On Err: nothing happens, transfer stays in pending_transfers forever
}
```

### Impact Explanation

**Impact: Critical — Permanent freezing / irrecoverable lock of user funds.**

For any deployed (bridged) token transfer from NEAR to EVM:
- Tokens are burned on NEAR at initiation time
- If the transfer cannot be finalized on the destination chain for any reason, the burned tokens are permanently lost
- There is no on-chain mechanism for the user to recover their funds

Scenarios that cause permanent loss:
1. **Factory address update**: DAO calls `add_factory` with a new EVM factory address. All in-flight transfers initiated under the old factory address can no longer be claimed via `claim_fee` (the `emitter_address` in the proof won't match the new factory). The burned tokens are unrecoverable.
2. **Token mapping removal**: If the token address mapping is removed, `sign_transfer` panics on `FailedToGetTokenAddress`, making the transfer permanently unsignable while the tokens remain burned.
3. **Destination chain bridge permanently unavailable**: If the EVM bridge is paused indefinitely or upgraded incompatibly, no finalization proof can ever be produced, and the burned tokens are permanently lost.

For locked (non-deployed) tokens, the DAO can use `transfer_token_as_dao` as an emergency rescue. But for burned deployed tokens, **even the DAO cannot recover them** — the tokens no longer exist on-chain.

### Likelihood Explanation

The likelihood is **moderate**. The most realistic trigger is a factory address update during a bridge upgrade (a routine operational event). The Omni Bridge is actively developed and has already undergone migrations (evidenced by `migrate.rs`). Any upgrade that changes the EVM factory contract address would silently strand all in-flight transfers initiated before the upgrade. Users who happened to initiate a transfer during the upgrade window would permanently lose their tokens with no recourse.

### Recommendation

Add a user-callable `cancel_transfer` function that:
1. Verifies the caller is the original `sender` of the pending transfer
2. Removes the transfer from `pending_transfers`
3. For deployed tokens: mints the burned amount back to the sender
4. For locked tokens: transfers the locked amount back to the sender and decrements `locked_tokens`

A time-lock (e.g., transfer must be pending for at least N blocks) can be added to prevent abuse.

### Proof of Concept

1. Alice holds 100 wETH (a deployed bridged token) on NEAR.
2. Alice calls `ft_transfer_call` with `InitTransferMsg` targeting Ethereum. `init_transfer_internal` burns 100 wETH and stores the transfer in `pending_transfers`. [1](#0-0) 
3. The DAO upgrades the EVM bridge and calls `add_factory` with the new factory address, overwriting the old one. [2](#0-1) 
4. A relayer calls `sign_transfer` for Alice's transfer — this still succeeds because it only reads `token_address`, not the factory. [3](#0-2) 
5. Alice submits the MPC signature to the old EVM factory — but the old factory is no longer active, so the EVM transaction fails.
6. The relayer attempts `claim_fee` with a proof from the old factory. `claim_fee_callback` panics with `ERR_UNKNOWN_FACTORY` because `self.factories.get(chain) != Some(old_factory_address)`. [4](#0-3) 
7. Alice's transfer remains in `pending_transfers` forever. Her 100 wETH are burned and unrecoverable — there is no `cancel_transfer` function she can call. [5](#0-4)

    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };

        let payload = near_sdk::env::keccak256_array(
            transfer_payload
                .encode_hashable()
                .near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_TRANSFER_CALLBACK_GAS)
                    .sign_transfer_callback(transfer_payload, &transfer_message.fee),
            )
    }
