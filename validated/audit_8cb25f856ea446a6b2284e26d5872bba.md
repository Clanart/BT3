### Title
Integer Truncation in `NewDynamicFeeChecker` `feeCap` Computation Causes Systematic Fee Under-Collection — (File: `ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` derives the per-gas fee cap for Cosmos SDK transactions via integer (floor) division: `feeCap = fee / gas`. Because the result is truncated toward zero, the effective fee deducted from the sender is always `≤ fee − (fee mod gas)`. The remainder `fee mod gas` (up to `gas − 1` tokens) is never collected by the fee-collector module. Every Cosmos SDK transaction processed through this path contributes a small, systematic shortfall to the fee collector, directly analogous to the Umee rounding-direction bug.

---

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` computes the per-gas fee cap as:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // line 83 — integer (floor) division
```

`sdkmath.Int.Quo` is truncating integer division. For any `fee` that is not an exact multiple of `gas`, this produces `feeCap = ⌊fee / gas⌋ < fee / gas`.

The effective fee then becomes:

```go
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
effectiveFee := sdk.Coins{{Denom: denom, Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))}}
```

Because `effectivePrice ≤ feeCap = ⌊fee/gas⌋`, the maximum amount ever deducted is:

```
effectiveFee_max = ⌊fee/gas⌋ × gas = fee − (fee mod gas)
```

The residual `fee mod gas` tokens (0 to `gas − 1`) remain in the sender's account and are never forwarded to the fee collector. This is not a refund of unused gas — it is a structural loss caused by rounding direction.

The same truncation pattern appears in two additional locations that compound the effect:

- `x/feemarket/keeper/abci.go:74` — `gasWanted` stored per block is truncated: `sdkmath.LegacyMaxDec(limitedGasWanted, ...).TruncateInt().Uint64()`, systematically biasing the base-fee input downward each block.
- `x/feemarket/keeper/eip1559.go:103` — `minGasPrice := params.MinGasPrice.TruncateInt().BigInt()` truncates the `MinGasPrice` lower bound applied to the base fee, allowing the base fee to fall below the governance-set minimum when `MinGasPrice` has a fractional part.

---

### Impact Explanation

Every Cosmos SDK transaction routed through `NewDynamicFeeChecker` (the path used when `DynamicFeeChecker` is enabled in `newCosmosAnteHandler`) under-pays the fee collector by up to `gas − 1` tokens. For a transaction with `gas = 200 000` and a denomination with moderate value, the per-transaction shortfall can be material. Across a high-throughput chain, the cumulative loss to the fee collector (and therefore to validators and stakers who receive fee distributions) is unbounded and grows linearly with transaction volume. This matches the "valid user funds/fees to be mis-accounted" High impact criterion.

---

### Likelihood Explanation

The condition `fee mod gas ≠ 0` is true for virtually every real transaction — wallets and clients do not round submitted fees to exact multiples of the gas limit. The bug is therefore triggered on every Cosmos SDK transaction that passes through `NewDynamicFeeChecker`, making it a near-certain, continuous loss rather than a rare edge case. No special privileges or unusual inputs are required; any unprivileged user submitting a standard Cosmos SDK transaction triggers it.

---

### Recommendation

Replace the floor-division `feeCap` with a ceiling-division so that the rounding direction favors the protocol rather than the sender:

```go
// ceiling division: feeCap = ⌈fee / gas⌉
gasInt := sdkmath.NewIntFromUint64(gas)
feeCap := fee.Add(gasInt.SubRaw(1)).Quo(gasInt)
```

Alternatively, perform the minimum-fee check directly on the total fee without converting to a per-gas price:

```go
if fee.LT(baseFeeInt.Mul(gasInt)) {
    return nil, 0, errorsmod.Wrapf(...)
}
```

Apply the same ceiling-rounding fix to `x/feemarket/keeper/abci.go:74` and use `RoundInt()` (round-half-up) instead of `TruncateInt()` for `MinGasPrice` in `x/feemarket/keeper/eip1559.go:103` so the base-fee lower bound is never below the governance-set minimum.

### Proof of Concept

**Setup**: `baseFee = 1000`, `gas = 21000`, `tip = 0` (no extension option → `maxPriorityPrice = MaxInt64`, so `effectivePrice = feeCap`).

**Attacker submits**: `fee = 1000 × 21000 + 20999 = 21 020 999` with `gas = 21000`.

**Execution**:
```
feeCap = ⌊21 020 999 / 21 000⌋ = ⌊1000.999...⌋ = 1000
feeCap ≥ baseFee (1000 ≥ 1000) → passes
effectivePrice = min(1000 + MaxInt64, 1000) = 1000
effectiveFee   = 1000 × 21 000 = 21 000 000
```

**Result**: The sender submitted `21 020 999` tokens but is charged only `21 000 000`. The `20 999` token remainder is never collected. Repeated across N transactions, the fee collector is short `N × 20 999` tokens with zero additional cost to the sender (the remainder simply stays in their account). [1](#0-0) [2](#0-1) [3](#0-2)
