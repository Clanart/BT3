### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the maximum chargeable fee from user-controlled resource bounds using Cairo felt arithmetic (modular arithmetic mod P ≈ 2^251). No post-computation invariant check is performed on the result. An unprivileged transaction sender can craft resource bounds such that the sum of the three fee terms wraps to exactly 0 modulo P. When `max_fee == 0`, `charge_fee` returns immediately without charging any fee, allowing the transaction to execute for free.

---

### Finding Description

`compute_max_possible_fee` computes:

```
max_fee = l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
        + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
        + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit
``` [1](#0-0) 

All arithmetic is modular over the Cairo field prime P. The bounds enforced during transaction hash computation (via `pack_resource_bounds`) are:

- `max_amount` ∈ [0, 2^64 − 1]
- `max_price_per_unit` ∈ [0, (P−1)/2] (only `assert_nn` is applied)
- `tip` ∈ [0, 2^64 − 1] [2](#0-1) 

These bounds are **not sufficient** to prevent overflow in the products. The product `max_amount * max_price_per_unit` can reach up to (2^64 − 1) × (P−1)/2 ≈ 2^314, which wraps around P multiple times. No post-computation check is performed to verify that `max_fee` is within a valid range after the summation.

The result is used directly as the upper bound in `charge_fee`:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
...
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

If `max_fee` overflows to 0, `charge_fee` returns immediately — no fee is charged, no ERC-20 transfer is executed, and the transaction runs for free.

---

### Impact Explanation

**Critical. Direct loss of funds.**

The sequencer's off-chain fee validation (implemented in Rust with 256-bit arithmetic) computes a large, non-zero `max_fee` for the crafted transaction and includes it in the block. The OS proof, using felt arithmetic, computes `max_fee = 0` and skips fee collection entirely. The sequencer provides execution without receiving payment. At scale, repeated exploitation drains sequencer revenue and can lead to a network halt.

---

### Likelihood Explanation

**Medium.** The attack is fully deterministic and requires only crafting specific felt values for `max_price_per_unit` within the already-permitted range [0, (P−1)/2]. No privileged access, leaked keys, or external dependencies are required. Any V3 transaction sender can perform this.

---

### Recommendation

After computing `max_fee` in `compute_max_possible_fee`, add an invariant check that the result is within a protocol-defined upper bound (e.g., bounded by the maximum representable fee in the system). Alternatively, enforce tighter bounds on `max_price_per_unit` in `pack_resource_bounds` (e.g., `assert_nn_le(max_price_per_unit, MAX_PRICE_PER_UNIT)`) so that no product can overflow a felt. The missing check is the direct analog of the external report's recommendation: verify the computed value against its expected upper bound before using it as a constraint.

---

### Proof of Concept

Craft a V3 transaction with the following resource bounds (all pass existing validation):

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `2` |
| `l1_gas_bounds.max_price_per_unit` | `(P−1)/2` |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas_bounds.max_amount` | `0` |
| `l1_data_gas_bounds.max_price_per_unit` | `0` |

Computation inside `compute_max_possible_fee` (felt arithmetic mod P):

```
A = 2 × (P−1)/2 = P−1 ≡ −1  (mod P)
B = 1 × (1 + 0)  = 1
C = 0 × 0        = 0

max_fee = (−1) + 1 + 0 = 0  (mod P)
```

All bounds checks in `pack_resource_bounds` pass:
- `assert_nn_le(2, 2^64−1)` ✓
- `assert_nn((P−1)/2)` ✓ (exactly at the boundary)
- `assert_nn_le(0, 2^64−1)` ✓

The OS reaches `charge_fee`, computes `max_fee = 0`, and returns immediately at:

```cairo
if (max_fee == 0) {
    return ();
}
``` [4](#0-3) 

The transaction executes with zero fee charged. The sequencer's Rust-based off-chain validator, using 256-bit arithmetic, computed `max_fee = P` (a large positive number) and accepted the transaction — the discrepancy between off-chain and on-chain fee computation is the root cause.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
