### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Unbounded Replay of Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the hash of a version-0 meta-transaction without including any nonce or uniqueness identifier. Because `check_and_increment_nonce` unconditionally skips nonce enforcement for version-0 transactions, a valid meta-transaction signature is permanently reusable. Any attacker who observes a `meta_tx_v0` syscall can replay the identical signed payload an unlimited number of times across future blocks, causing direct loss of funds from the target account.

---

### Finding Description

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
// transaction_hash.cairo lines 295-315
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // ← no nonce committed
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
```

The resulting hash commits only to `(prefix, version=0, contract_address, selector, calldata, max_fee=0, chain_id)`. No nonce, block number, or any other replay-preventing field is included. [1](#0-0) 

In `execute_meta_tx_v0`, the OS constructs a `TxInfo` with `nonce=0` and `version=0`, then calls `contract_call_helper` directly — no nonce is checked or incremented: [2](#0-1) 

The reason no nonce check fires is explicit in `check_and_increment_nonce`:

```cairo
// execute_transaction_utils.cairo lines 63-67
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

The three conditions that together create the vulnerability:

1. **Hash omits nonce** — `additional_data_size=0` in `compute_meta_tx_v0_hash`.
2. **TxInfo carries `nonce=0`** — hardcoded in `execute_meta_tx_v0`.
3. **Nonce check is skipped for version 0** — `check_and_increment_nonce` returns immediately.

This is structurally identical to the H-07 report: the EVM contract checked `nonces[batchId]` but not `batchId` itself; here the OS checks the outer transaction's nonce but never checks or commits to any nonce for the inner meta-transaction.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who signs a meta-transaction to, for example, transfer ERC-20 tokens or approve a DeFi operation produces a signature that is valid for `(contract_address, selector, calldata, chain_id)` forever. Any party who observed the original on-chain `meta_tx_v0` syscall can re-submit the same `(contract_address, selector, calldata, signature)` tuple in a new outer invoke transaction in any subsequent block. The target account's `__execute__` entry point will verify the signature against the same hash and succeed, executing the transfer again. This can be repeated until the account is drained.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is a public, unprivileged protocol feature callable by any account contract. Every meta-transaction ever submitted is permanently replayable by any observer of the chain. No special access, leaked key, or privileged role is required — the attacker only needs to copy the `(contract_address, selector, calldata, signature)` fields from a previously confirmed transaction and wrap them in a new outer invoke.

---

### Recommendation

Include a nonce in the meta-transaction hash. The nonce must be:
1. Committed into `compute_meta_tx_v0_hash` via `additional_data`, analogous to how `compute_l1_handler_transaction_hash` passes `additional_data=&nonce`.
2. Sourced from the target contract's on-chain nonce (read from `contract_state_changes`) and incremented after each successful meta-transaction execution.
3. Enforced by removing the version-0 early-return in `check_and_increment_nonce`, or by adding a dedicated nonce check inside `execute_meta_tx_v0`.

Concretely, `compute_meta_tx_v0_hash` should be changed to:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,     // ← commit nonce
        additional_data=&nonce,
    );
    return tx_hash;
}
```

And `execute_meta_tx_v0` must read the target contract's current nonce from `contract_state_changes`, pass it to the hash function, and increment it afterward.

---

### Proof of Concept

**Setup**: Alice's account contract supports `meta_tx_v0` (gasless relaying). She signs a meta-transaction to transfer 100 tokens to Bob.

**Step 1 — Legitimate execution**: A relayer submits an outer v3 invoke that calls `meta_tx_v0` with:
- `contract_address = Alice`
- `selector = __execute__`
- `calldata = [transfer(Bob, 100)]`
- `signature = Alice_sig`

The OS computes `meta_tx_hash = H(prefix, 0, Alice, __execute__, calldata, 0, chain_id)` (no nonce). Alice's `__validate__` verifies `Alice_sig` against this hash. 100 tokens are transferred to Bob.

**Step 2 — Replay**: In the next block, the attacker (or the relayer itself) submits a new outer v3 invoke calling `meta_tx_v0` with the identical `(Alice, __execute__, calldata, Alice_sig)`. The OS recomputes the identical `meta_tx_hash` (nothing has changed — no nonce was committed). Alice's `__validate__` verifies `Alice_sig` again and succeeds. Another 100 tokens are transferred to Bob.

**Step 3 — Drain**: This is repeated until Alice's balance is zero. The attacker pays only the outer transaction gas; Alice's account has no defense because the signature is cryptographically valid for every replay. [4](#0-3) [5](#0-4)
