### Title
Unprivileged HyperCore user can permanently lock bridged tokens in `HyperliquedBridgeToken` by setting `recipient = address(this)` in `ACTION_TRANSFER` — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

### Summary
`coreReceiveWithData` dispatches `ACTION_TRANSFER` by decoding an arbitrary `address recipient` from user-supplied `data` and calling `_update(_systemAddress, recipient, amount)`. No guard prevents `recipient == address(this)`. A HyperCore user can craft `data = 0x00 || abi.encode(tokenContractAddress)` and trigger the system address to call this function, permanently moving tokens into the token contract with no recovery path.

### Finding Description
The `coreReceiveWithData` function in `HyperliquedBridgeToken` is callable only by `_systemAddress`, but on HyperLiquid any HyperCore user can invoke `sendToEvmWithData` targeting this contract, causing the system address to relay the call with user-controlled `data`. For `ACTION_TRANSFER`, the recipient address is decoded directly from `data[1:]` with no validation:

```solidity
address recipient = abi.decode(tail, (address));
_update(_systemAddress, recipient, amount);
```

Setting `recipient = address(this)` transfers `amount` tokens from the `_systemAddress` pool into the token contract's own balance. The token contract has no `transfer`, `approve`, or `initTransfer` function callable on its own behalf that would allow those tokens to exit. The only administrative escape (`BridgeToken.burn`) would destroy supply without emitting an `InitTransfer` event, leaving the NEAR-side backing permanently locked — so no recovery path is free of collateral damage.

### Impact Explanation
- The `_systemAddress` pool (which mirrors total HyperCore-side balance) is drained by `amount`.
- Tokens accumulate at `address(this)` with no reachable exit.
- All other HyperCore users whose tokens are represented in the pool lose the ability to withdraw to HyperEVM, because subsequent `_update(_systemAddress, legitimateRecipient, amount)` calls will revert once the pool is exhausted.
- This is a **permanent, irrecoverable lock** of bridged assets — Critical impact.

### Likelihood Explanation
Any HyperCore user holding a non-zero balance of the bridged token can execute this with a single `sendToEvmWithData` call. No privileged access, leaked key, or colluding party is required. The attacker sacrifices their own tokens but can drain the shared pool, griefing all other HyperCore holders of that token.

### Recommendation
Add a zero-address and self-address guard in the `ACTION_TRANSFER` branch:

```solidity
if (action == ACTION_TRANSFER) {
    address recipient = abi.decode(tail, (address));
    require(
        recipient != address(0) && recipient != address(this),
        "Invalid recipient"
    );
    _update(_systemAddress, recipient, amount);
}
```
