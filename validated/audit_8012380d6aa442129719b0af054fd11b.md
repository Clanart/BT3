### Title
Missing `prev_chunk_header.shard_id()` == `shard_id` guard in `set_state_header` lets a malicious peer corrupt the receipt window and install a wrong state root — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` verifies that `prev_chunk_header` is present in the block's chunk-header merkle tree (step 3b), but never asserts that the header belongs to the requested shard. Because `verify_path` is position-agnostic, an attacker can supply shard Y's chunk header with shard Y's valid merkle proof, pass the check, and substitute a different `height_included` as the lower bound for incoming-receipt collection. `set_state_finalize` then applies receipts from the wrong height window and stores a wrong `ChunkExtra` state root, permanently breaking the syncing node's ability to process subsequent blocks.

### Finding Description

**Entrypoint.** A syncing node calls `set_state_header` with a `ShardStateSyncResponseHeader` received from any peer over the network. No privilege is required to respond to a state-header request.

**Missing guard.** Step 3b (`adapter.rs:416-425`) calls:

```rust
if !verify_path(
    *prev_block_header.chunk_headers_root(),
    prev_chunk_proof,
    &ChunkHashHeight(prev_chunk_header.chunk_hash(), prev_chunk_header.height_included()),
) { … }
```

`verify_path` (`merkle.rs:113-119`) reconstructs the root from the leaf hash and the sibling path and compares it to the known root. It does **not** check which index the leaf occupies. An attacker who supplies shard Y's chunk header together with shard Y's genuine merkle proof (both obtained from public chain data) will satisfy this check, because shard Y's header really is in the tree at position Y.

There is no subsequent `prev_chunk_header.shard_id() == shard_id` assertion anywhere in `set_state_header`.

**Receipt-window corruption.** Step 4g (`adapter.rs:507`) enforces:

```rust
if header.height() != prev_chunk_header.map_or(0, |h| h.height_included()) { … }
```

This anchors the `incoming_receipts_proofs` range to shard Y's `height_included` (call it H_Y) rather than shard X's correct value H_X. The attacker supplies a receipt-proof sequence covering `[sync_hash … H_Y]` — all proofs are verifiable against real block headers and real outgoing-receipts roots (steps 4b–4f), so the entire header passes validation and is written to `DBCol::StateHeaders`.

**Wrong receipts applied.** `set_state_finalize` (`chain_update.rs:479-486`) reads the stored `incoming_receipts_proofs` and keeps those whose block height ≤ `chunk.height_included()`. If H_Y > H_X, receipts from `[H_X … H_Y)` are absent from the stored proofs and are never applied (missing receipts). If H_Y < H_X, receipts from `[H_Y … H_X)` are extra and are applied twice (double-applied receipts). Either way the `apply_chunk` call (`chain_update.rs:519-542`) produces a state root that diverges from the canonical one, and `apply_chunk_postprocessing` stores this wrong root as `ChunkExtra`.

**Downstream breakage.** Every subsequent block that the node tries to process will call `validate_chunk_with_chunk_extra` against the wrong `ChunkExtra` state root and fail, leaving the node permanently unable to advance its chain head without a full re-sync.

### Impact Explanation
A single malicious peer response causes the syncing node to store a wrong `ChunkExtra` state root for the synced shard. The node cannot process any further blocks for that shard and must wipe its state and re-sync. The attack is silent — the node logs no error during `set_state_header` because all proof checks pass.

### Likelihood Explanation
All inputs needed to craft the attack (chunk headers, merkle paths, receipt proofs, outgoing-receipts roots) are public chain data. Any peer that can respond to a `StateRequestHeader` network message — no validator key or special role required — can execute this. The syncing node accepts the first header that passes validation with no cross-peer comparison.

### Recommendation
Add an explicit shard-id check immediately after extracting `prev_chunk_header` in `set_state_header`:

```rust
if let Some(ref pch) = prev_chunk_header {
    if pch.shard_id() != shard_id {
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk_header shard_id mismatch".into(),
        ));
    }
}
```

Similarly audit the `chunk` field in step 3a for the same missing `chunk.shard_id() == shard_id` guard, since `verify_path` there is equally position-agnostic.

### Proof of Concept

1. Attacker observes that the victim is syncing shard X at `sync_hash`.
2. Attacker fetches from public chain data:
   - Shard X's chunk header and its merkle proof (for the main `chunk` field — correct).
   - Shard Y's chunk header and its merkle proof at position Y in the block one height before `chunk.height_included()` (for `prev_chunk_header`/`prev_chunk_proof`).
   - All `incoming_receipts_proofs` and `root_proofs` for blocks `[sync_hash … H_Y]` (valid, from real chain data).
3. Attacker sends this crafted `ShardStateSyncResponseHeader` to the victim.
4. `set_state_header` runs:
   - Step 1-2: `validate_chunk_proofs` passes (correct shard X chunk).
   - Step 3a: `verify_path` passes (shard X chunk is in the block).
   - Step 3b: `verify_path` passes (shard Y chunk header is in the block at position Y — no shard_id check).
   - Steps 4a-4g: all receipt proofs are valid for the `[sync_hash … H_Y]` range.
   - Step 5: `state_root_node` validates against shard X's `prev_state_root`.
   - Header is written to `DBCol::StateHeaders`.
5. `set_state_finalize` applies receipts from `[H_Y … chunk.height_included()]` instead of `[H_X … chunk.height_included()]`, producing a wrong state root stored in `ChunkExtra`.
6. Victim node fails to process any subsequent block for shard X.
