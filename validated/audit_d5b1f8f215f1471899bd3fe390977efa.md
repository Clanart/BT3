### Title
Unauthenticated Class Declaration via Hardcoded `'BOOTSTRAP'` Sender Address Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `execute_declare_transaction`, the OS contains a special-case path that completely skips signature validation, nonce enforcement, and fee charging when a transaction's `sender_address` equals the felt literal `'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and `max_possible_fee == 0`. Because `'BOOTSTRAP'` is simply an ASCII-encoded felt value with no cryptographic protection, any unprivileged user can craft a transaction satisfying all four conditions and declare arbitrary class hashes into the protocol's class registry without any authentication.

---

### Finding Description

In `execute_declare_transaction` at lines 761–776 of `transaction_impls.cairo`, the following branch exists:

```cairo
// Do not run validate or perform any account-related actions for declare transactions that
// meet the following conditions.
// This flow is used for the sequencer to bootstrap a new system.
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns **before**:
- `check_and_increment_nonce` (line 779) — nonce is never verified or incremented
- `non_reverting_select_execute_entry_point_func` with `VALIDATE_DECLARE_ENTRY_POINT_SELECTOR` (line 804) — `__validate_declare__` is never called
- `charge_fee` (line 822) — no fee is charged [2](#0-1) 

The `'BOOTSTRAP'` string in Cairo is a felt literal — the big-endian ASCII encoding of the bytes of "BOOTSTRAP". It is a fixed, publicly known felt value. There is no cryptographic secret, privileged key, or on-chain access-control check protecting this branch. Any transaction sender can set `sender_address` to this felt value.

The only guard against re-use is `dict_update` with `prev_value=0`, which prevents overwriting an already-declared class. However, because the nonce of the BOOTSTRAP address is **never incremented** (the function returns before `check_and_increment_nonce`), an attacker can submit an unlimited number of BOOTSTRAP transactions — each with `nonce=0` — declaring different class hashes, and all will succeed. [3](#0-2) 

---

### Impact Explanation

**Critical — Direct loss of funds / network shutdown.**

The StarkNet class registry is append-only per class hash (`prev_value=0` enforcement). During network bootstrapping, critical system contract classes (fee token, account contract, etc.) have not yet been declared. An attacker who front-runs the legitimate bootstrap sequence can:

1. Declare a malicious class hash for any not-yet-declared class.
2. If the sequencer or protocol subsequently deploys a contract using that class hash (e.g., the fee token or a system account), the deployed contract executes the attacker's malicious logic.
3. This enables direct theft of funds from users interacting with the compromised contract, or a complete halt of the network if a critical infrastructure contract is poisoned.

Because the nonce is never incremented for BOOTSTRAP transactions, the attacker can declare multiple malicious classes in a single block, maximizing the attack surface during the bootstrapping window.

---

### Likelihood Explanation

**High.** The triggering conditions are entirely attacker-controlled and require no privileged access:

- `sender_address = felt('BOOTSTRAP')` — a publicly known constant, no key required
- `nonce = 0` — trivially satisfiable
- `version = 3` — standard transaction version
- `max_possible_fee = 0` — all resource bounds set to zero

The attack is most effective in the first blocks of a new network deployment, exactly the window when the bootstrap mechanism is intended to be used. An attacker monitoring the mempool can front-run any legitimate bootstrap declare transaction.

---

### Recommendation

The bootstrap bypass must be restricted to a trusted, cryptographically authenticated identity. Options include:

1. **Remove the bypass entirely** and require all declare transactions to go through normal `__validate_declare__` flow, including during bootstrapping.
2. **Replace the felt-literal address check** with a cryptographically enforced mechanism — for example, requiring the transaction to be signed by a known sequencer key, or gating the bypass on a block-number range enforced in the block context.
3. **Enforce nonce increment** even in the bootstrap path, so that each BOOTSTRAP address can only declare one class per nonce, limiting the attack surface.

---

### Proof of Concept

An attacker constructs a v3 declare transaction with:

```
sender_address  = 0x424f4f545354524150  // felt('BOOTSTRAP')
nonce           = 0
version         = 3
l1_gas          = { max_amount: 0, max_price_per_unit: 0 }
l2_gas          = { max_amount: 0, max_price_per_unit: 0 }
l1_data_gas     = { max_amount: 0, max_price_per_unit: 0 }
tip             = 0
class_hash      = <hash of attacker-controlled malicious Sierra class>
compiled_class_hash = <corresponding CASM hash>
```

The OS evaluates the branch at line 764:

```
sender_address == 'BOOTSTRAP'  → true
tx_info.nonce == 0             → true
tx_info.version == 3           → true
compute_max_possible_fee(...)  → 0 (all bounds zero)
``` [4](#0-3) 

The OS executes `dict_update(key=class_hash, prev_value=0, new_value=compiled_class_hash)` and returns. No signature is verified, no fee is charged, no nonce is consumed. The attacker can repeat this with `nonce=0` for every additional class hash they wish to inject, since the BOOTSTRAP account nonce is never incremented. [5](#0-4)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
