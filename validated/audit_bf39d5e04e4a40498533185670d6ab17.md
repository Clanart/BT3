### Title
`Contract::migrate()` Silently Drops `relayers` and `relayer_config` State, Permanently Locking Relayer Stakes and Halting Bridge Operations - (`near/omni-bridge/src/migrate.rs`)

### Summary

The `Contract::migrate()` function in the NEAR omni-bridge reads an `OldState` struct that includes `relayers: LookupMap<AccountId, RelayerState>` and `relayer_config: RelayerConfig`, but constructs the new `Contract` without transferring these fields. Both are silently dropped. This permanently locks all relayer-staked NEAR tokens in inaccessible storage and resets the trusted-relayer registry to empty, causing all bridge operations that require a trusted relayer to fail post-migration.

### Finding Description

In `near/omni-bridge/src/migrate.rs`, the `OldState` struct (the pre-migration contract layout) includes two fields at lines 42–43:

```rust
pub relayers: LookupMap<AccountId, RelayerState>,
pub relayer_config: RelayerConfig,
```

The `migrate()` function at lines 50–78 reads this old state via `env::state_read::<OldState>()` and constructs a new `Self` (the current `Contract`). However, the construction block at lines 52–74 maps every other field from `old_state` but **never assigns `old_state.relayers` or `old_state.relayer_config`** to the new state. Both are dropped at the end of the `if let Some(old_state)` block.

The current `Contract` struct (in `lib.rs` lines 220–243) does not declare `relayers` or `relayer_config` as explicit fields — they are injected by the `#[trusted_relayer(...)]` proc-macro at lines 245–249. The macro manages its own storage for the relayer registry. After migration, the macro-managed relayer storage is initialized empty (no trusted relayers, default config), while the old `relayers` LookupMap data remains orphaned in contract storage under the `_Relayers` storage key prefix (visible in `StorageKey` enum at line 110, where the underscore prefix signals it is now deprecated/unreachable).

Relayer stakes are NEAR tokens deposited via `apply_for_trusted_relayer`. These tokens are held in the contract's account balance, tracked by the `relayers` LookupMap. After migration, the LookupMap is unreachable, so relayers cannot call `resign_trusted_relayer` to recover their stake. The tokens are permanently locked.

Additionally, the `relayer_config` (stake requirements, waiting period) is reset to defaults, and all previously trusted relayers lose their status. Every bridge function guarded by `#[trusted_relayer]` — including `fin_transfer`, `sign_transfer`, fast transfers, and UTXO operations — requires a trusted relayer and will reject calls until relayers re-apply and wait out the activation period.

### Impact Explanation

**Permanent fund lock (Critical/High):** All NEAR tokens staked by relayers before the migration are permanently locked in the contract. The `relayers` LookupMap that tracks stake ownership is orphaned; no code path in the new contract can read it or return funds to stakers.

**Bridge operation freeze (High):** Post-migration, the trusted-relayer registry is empty. All `fin_transfer`, `sign_transfer`, fast-transfer, and UTXO-transfer calls are gated by the `#[trusted_relayer]` guard. With no trusted relayers, these calls revert. Pending cross-chain transfers cannot be finalized, causing user funds to be stuck in the bridge until relayers re-stake and the waiting period elapses.

### Likelihood Explanation

This is a deterministic bug that fires on every execution of `migrate()`. The migration is a privileged, one-time upgrade step, but its effect is irreversible: once called, the old relayer state is gone and staked funds are permanently inaccessible. Any deployment that upgrades the contract and calls `migrate()` will trigger this.

### Recommendation

Transfer `old_state.relayers` and `old_state.relayer_config` into the new contract state. If the `#[trusted_relayer]` macro stores these under its own storage keys, the migration must explicitly copy the old LookupMap entries into the new storage prefix, or expose a post-migration function to re-register existing relayers. At minimum, the old `relayers` LookupMap must be iterated and each staker's NEAR returned before the map is abandoned.

```rust
// In migrate(), transfer relayer state instead of dropping it:
Self {
    // ... all existing fields ...
    // Pass old relayer data to macro-managed storage via an explicit migration step
}
// Then iterate old_state.relayers and refund each staker, or
// copy entries into the new relayer storage prefix.
```

### Proof of Concept

1. Deploy the omni-bridge contract with the old code (containing `relayers` and `relayer_config` as inline fields).
2. Have several relayers call `apply_for_trusted_relayer` with NEAR deposits; wait for them to become trusted.
3. Upgrade the contract code to the new version.
4. Call `migrate()`.
5. Observe: `is_trusted_relayer` returns `false` for all previously trusted relayers; `get_relayer_stake` returns `None` for all stakers; `resign_trusted_relayer` panics because the new relayer system has no record of them; the staked NEAR is permanently locked in the contract balance.
6. Attempt `fin_transfer` — it reverts because no trusted relayer exists.
