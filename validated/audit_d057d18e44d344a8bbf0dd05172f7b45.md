### Title
Empty `check_is_reverted` Allows Malicious Prover to Skip Invoke Transaction Execution While Charging Fee - (File: `execution/execution_constraints.cairo`)

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a no-op stub that imposes zero constraints on the hint-supplied `is_reverted` value used in `execute_invoke_function_transaction`. A malicious prover (sequencer) can freely set `is_reverted = TRUE` for any valid invoke transaction, causing the execute step to be silently skipped while the fee is still charged and a valid proof is produced. This is a direct analog to the external report's incomplete-condition-check class: just as the quorum check omitted `againstVotes`, the revert check here omits any actual verification.

---

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`), after validation runs successfully, the OS determines whether to execute the transaction body via a hint-provided flag:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    tempvar range_check_ptr = range_check_ptr;
    tempvar remaining_gas = remaining_gas;
    ...
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The `check_is_reverted` function that is supposed to constrain this value is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

The function body is entirely empty. It accepts `is_reverted` as an argument but writes no constraints, performs no assertions, and returns immediately. The Cairo proof system only enforces what is explicitly asserted; an unconstrained local variable set by a hint (`%{ IsReverted %}`) is a free witness that the prover can assign to any field element.

The fee is charged unconditionally after the branch:

```cairo
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [3](#0-2) 

The fee computation itself is bounded only by `max_fee`:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [4](#0-3) 

There is no path that ties the fee charge to whether execution actually occurred.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious sequencer can produce a valid STARK proof for a block in which any invoke transaction has its execute step suppressed (`is_reverted = TRUE`) while the fee is deducted from the user's account. Because `check_is_reverted` imposes no constraint, the proof verifier on L1 accepts the proof. The committed state root reflects:

- Fee deducted from the user's balance (real loss).
- No state changes from the intended execution (e.g., no token transfer, no contract interaction).

If the suppressed transaction was the only mechanism to withdraw or unlock funds held in a contract, those funds become permanently inaccessible — satisfying both the **direct loss of funds** and **permanent freezing of funds** impact categories.

---

### Likelihood Explanation

The attacker-controlled entry path is the sequencer itself, which is the sole entity that supplies hints to the OS Cairo program. The sequencer:

1. Constructs the block and provides all hint values.
2. Sets `is_reverted = TRUE` for a targeted invoke transaction.
3. Omits the execution trace for that transaction (no trace is needed since the branch is not taken).
4. Produces a valid proof — the Cairo verifier finds no violated constraint because `check_is_reverted` asserts nothing.

No leaked key, no Sybil attack, and no external dependency is required. The only prerequisite is operating as the sequencer, which in a decentralized or permissioned-but-compromised setting is a realistic threat. The OS is specifically designed to be the trust boundary that prevents even a malicious sequencer from producing a valid proof for incorrect execution; this function breaks that guarantee for invoke transactions.

---

### Recommendation

Replace the empty stub with a real constraint. The correct approach depends on the intended semantics of `is_reverted`:

- **If `is_reverted` should always be `FALSE` for invoke transactions** (since `run_validate` uses `non_reverting_select_execute_entry_point_func` which already asserts no revert): add `assert is_reverted = 0;` inside `check_is_reverted`, or remove the branch entirely and always execute.
- **If pre-execution revert is a legitimate state** (e.g., sequencer-side resource exhaustion): the OS must independently derive `is_reverted` from a verifiable condition (e.g., a gas check that can be proven in-circuit) rather than accepting it as a free hint.

---

### Proof of Concept

1. User submits an invoke transaction (e.g., `transfer(recipient, amount)`) with sufficient gas and fee.
2. Malicious sequencer includes the transaction in a block but sets the `IsReverted` hint to `TRUE`.
3. The OS Cairo program executes `execute_invoke_function_transaction`:
   - `run_validate` succeeds (validation always runs and is non-reverting).
   - `check_is_reverted(TRUE)` returns immediately — no assertion fires.
   - The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
   - `charge_fee` executes normally, deducting the fee from the user's account.
4. The resulting Cairo trace is valid; no constraint is violated.
5. The STARK proof is generated and verified on L1.
6. The committed state root reflects: fee deducted, token transfer never happened.
7. The user has lost the fee and their intended state change did not occur. If the call was a withdrawal, the funds remain locked in the contract with no recourse.
