### Title
Missing Cross-Category Class Hash Validation in P2P State Diff Assembly Produces Wrong State Diff Commitment and State Root — (File: `crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

The P2P sync client assembles `ThinStateDiff` from peer-supplied chunks without checking whether the same class hash appears in both `class_hash_to_compiled_class_hash` (Cairo 1 / Sierra) and `deprecated_declared_classes` (Cairo 0). This violates a documented but unenforced invariant in `StateDiff`. A malicious peer can craft a state diff where one class hash occupies both categories while keeping the total `len()` equal to the header's `state_diff_length`, bypassing the only length-based gate. The resulting stored state diff diverges from the honest one, producing a wrong `StateDiffCommitment` and, after the committer processes it, a wrong global state root.

---

### Finding Description

**Documented but unenforced invariant.**
`StateDiff` in `crates/starknet_api/src/state.rs` carries an explicit comment:

```
// Invariant: Class hashes of declared_classes and deprecated_declared_classes are exclusive.
// TODO(yair): Enforce this invariant.
``` [1](#0-0) 

`ThinStateDiff` inherits the same structural split — `class_hash_to_compiled_class_hash: IndexMap<ClassHash, CompiledClassHash>` and `deprecated_declared_classes: Vec<ClassHash>` — but no enforcement exists anywhere in the P2P sync path. [2](#0-1) 

**Missing cross-category check in `unite_state_diffs`.**
When a `StateDiffChunk::DeprecatedDeclaredClass` arrives, the handler simply pushes to the `Vec` with no check against `class_hash_to_compiled_class_hash`:

```rust
StateDiffChunk::DeprecatedDeclaredClass(deprecated_declared_class) => {
    state_diff.deprecated_declared_classes.push(deprecated_declared_class.class_hash);
}
``` [3](#0-2) 

By contrast, `DeclaredClass` and all `ContractDiff` fields do check for intra-category duplicates and return `BadPeerError::ConflictingStateDiffParts` on collision. The asymmetry is intentional for performance but leaves the cross-category invariant unguarded.

**Post-assembly validation is also insufficient.**
`validate_deprecated_declared_classes_non_conflicting` only checks that `deprecated_declared_classes` has no internal duplicates:

```rust
fn validate_deprecated_declared_classes_non_conflicting(
    state_diff: &ThinStateDiff,
) -> Result<(), BadPeerError> {
    if state_diff.deprecated_declared_classes.len()
        == state_diff.deprecated_declared_classes.iter().cloned().collect::<HashSet<_>>().len()
    {
        Ok(())
    } else {
        Err(BadPeerError::ConflictingStateDiffParts)
    }
}
``` [4](#0-3) 

It never intersects `deprecated_declared_classes` with `class_hash_to_compiled_class_hash`.

**Length check is the only remaining gate — and it can be satisfied.**
`ThinStateDiff::len()` counts both fields independently:

```rust
result += self.class_hash_to_compiled_class_hash.len();
result += self.deprecated_declared_classes.len();
``` [5](#0-4) 

If the honest state diff has `n` total entries, a malicious peer can craft a diff of the same length `n` by placing class hash `X` in both categories while omitting one legitimate entry from each category. The length check passes; the cross-category collision is invisible to all current validators.

**Downstream commitment and root corruption.**
`calculate_state_diff_hash` hashes `declared_classes` and `deprecated_declared_classes` as independent sequences: [6](#0-5) 

A class hash appearing in both sequences is hashed twice, producing a `StateDiffCommitment` that differs from the honest block header's commitment. When the committer subsequently processes this stored diff, it derives a wrong global root: [7](#0-6) 

The wrong root is then persisted under `ForestMetadataType::StateRoot` and returned by all state-root queries. [8](#0-7) 

Additionally, `append_state_diff` writes class hash `X` into both `declared_classes_block_table` and `deprecated_declared_classes_block_table`, corrupting the class-type registry used by execution and RPC: [9](#0-8) 

---

### Impact Explanation

A node syncing via P2P stores a state diff that violates the exclusive-category invariant. The resulting `StateDiffCommitment` diverges from the honest block header's commitment, and the global state root stored for that block is wrong. All subsequent RPC calls (`starknet_getStateUpdate`, `starknet_getClassAt`, `starknet_call`, fee estimation, simulation) that read from this node return authoritative-looking but incorrect values. Class hash `X` is simultaneously resolvable as a Cairo 1 class and a Cairo 0 class, which can cause wrong contract code to be selected for execution.

This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The trigger requires a malicious P2P peer. In a permissionless P2P network any node can connect, so this is an unprivileged position. The attacker must know the honest block's `state_diff_length` (available from the synced header) and craft a replacement diff of equal length. No cryptographic break is required; only knowledge of the protocol format. The missing check is a single-line omission in an otherwise well-guarded function.

---

### Recommendation

In `validate_deprecated_declared_classes_non_conflicting` (or directly in `unite_state_diffs`), add an intersection check:

```rust
fn validate_deprecated_declared_classes_non_conflicting(
    state_diff: &ThinStateDiff,
) -> Result<(), BadPeerError> {
    let deprecated_set: HashSet<_> =
        state_diff.deprecated_declared_classes.iter().cloned().collect();
    // Existing duplicate check
    if deprecated_set.len() != state_diff.deprecated_declared_classes.len() {
        return Err(BadPeerError::ConflictingStateDiffParts);
    }
    // NEW: cross-category exclusivity check
    for class_hash in state_diff.class_hash_to_compiled_class_hash.keys() {
        if deprecated_set.contains(class_hash) {
            return Err(BadPeerError::ConflictingStateDiffParts);
        }
    }
    Ok(())
}
```

This enforces the invariant documented in `StateDiff` and closes the gap noted by the `TODO(yair)` comment.

### Proof of Concept

1. Honest block N has `state_diff_length = 4` with:
   - `declared: [(A, casm_A), (B, casm_B)]`
   - `deprecated: [C, D]`

2. Malicious peer sends four chunks for block N:
   - `DeclaredClass { class_hash: A, compiled_class_hash: casm_A }` ✓ accepted
   - `DeclaredClass { class_hash: X, compiled_class_hash: casm_X }` ✓ accepted (X ≠ A)
   - `DeprecatedDeclaredClass { class_hash: X }` ✓ accepted (no intra-list duplicate yet)
   - `DeprecatedDeclaredClass { class_hash: D }` ✓ accepted

3. Assembled `ThinStateDiff`:
   - `class_hash_to_compiled_class_hash: {A→casm_A, X→casm_X}` (len 2)
   - `deprecated_declared_classes: [X, D]` (len 2)
   - Total `len() = 4` — matches header ✓

4. `validate_deprecated_declared_classes_non_conflicting`: `{X, D}` has no internal duplicates ✓

5. State diff stored. `X` is now in both `declared_classes_block_table` and `deprecated_declared_classes_block_table`. `B` and `C` are absent.

6. Committer computes `calculate_state_diff_hash` over the malicious diff → wrong `StateDiffCommitment` → wrong global root persisted for block N.

7. All RPC queries against block N return values derived from the wrong root.
