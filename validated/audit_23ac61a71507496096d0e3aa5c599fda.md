### Title
Unbounded `max_price_per_unit` in `pack_resource_bounds` Causes `compute_max_possible_fee` to Exceed `assert_nn_le` Range, Making Block Proof Generation Impossible - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`pack_resource_bounds` enforces only a non-negativity check (`assert_nn`) on `max_price_per_unit`, bounding it to `[0, 2**128 - 1]`. Because `max_amount` is independently bounded to `[0, 2**64 - 1]`, their product in `compute_max_possible_fee` can reach `≈ 2**192`. The subsequent `assert_nn_le(actual_fee, max_fee)` call in `charge_fee` requires `max_fee - actual_fee` to fit in a range-check cell (`[0, 2**128)`), which is impossible when `max_fee > 2**128`. Any block containing such a transaction cannot produce a valid OS proof.

---

### Finding Description

**`pack_resource_bounds`** validates resource bounds during transaction hash computation:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount ∈ [0, 2^64)
    assert_nn(resource_bounds.max_price_per_unit);            // max_price_per_unit ∈ [0, 2^128)
    ...
}
``` [1](#0-0) 

`assert_nn` uses a range-check builtin, bounding `max_price_per_unit` to `[0, 2**128 - 1]`. No tighter upper bound is enforced.

**`compute_max_possible_fee`** then multiplies these unbounded values:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [2](#0-1) 

With `max_amount = 2**64 - 1` and `max_price_per_unit = 2**128 - 1`, a single term evaluates to `(2**64 - 1) × (2**128 - 1) ≈ 2**192`. The total `max_fee` can reach `≈ 3 × 2**192`.

**`charge_fee`** then calls:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

`assert_nn_le(a, b)` decomposes to:
1. `assert_nn(a)` → `a ∈ [0, 2**128)`
2. `assert_nn(b - a)` → `b - a ∈ [0, 2**128)`

When `max_fee ≈ 2**192`, the value `max_fee - actual_fee ≈ 2**192` cannot fit in a range-check cell (`[0, 2**128)`). The Cairo program fails unconditionally for any `actual_fee` the sequencer could supply, making it impossible to generate a valid proof for any block containing this transaction.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

If the sequencer includes a transaction with `max_price_per_unit` large enough to push `max_fee` above `2**128`, the OS Cairo program fails at `assert_nn_le` inside `charge_fee`. Because `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` for fee charging, and the failure occurs before the ERC-20 transfer call (at the range-check assertion itself), the entire block proof is invalid. The sequencer cannot finalize the block, and no subsequent blocks can be proven until the problematic transaction is excluded.

---

### Likelihood Explanation

Any unprivileged V3 transaction sender controls `max_price_per_unit` directly. The value `max_price_per_unit = 3` with `max_amount = 2**64 - 1` already produces `max_fee ≈ 3 × 2**64 << 2**128`, which is safe. But `max_price_per_unit = 2**65` with `max_amount = 2**64 - 1` produces `max_fee ≈ 2**129 > 2**128`, triggering the failure. The transaction passes hash computation (since `assert_nn(2**65)` succeeds — `2**65 < 2**128`). The sequencer's off-chain mempool validation does not enforce the missing OS-level bound, so such a transaction can reach block inclusion. Likelihood is **medium**: requires the sequencer to not independently simulate the fee-charging step with the exact OS range-check semantics.

---

### Recommendation

In `pack_resource_bounds`, replace the loose `assert_nn` with a tighter upper bound that guarantees `max_amount × max_price_per_unit` fits within the range that `assert_nn_le` can handle. For example, bound `max_price_per_unit` to `[0, 2**64 - 1]` (matching `max_amount`), ensuring each fee term is at most `(2**64 - 1)^2 ≈ 2**128`, and the three-term sum stays within `[0, 3 × 2**128)`. Then update `charge_fee` to use a comparison that handles values up to `3 × 2**128`, or enforce a stricter per-term cap so the total `max_fee` stays within `[0, 2**128 - 1]`.

```cairo
// Recommended fix in pack_resource_bounds:
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 64 - 1);
```

### Proof of Concept

1. Craft a V3 invoke transaction with:
   - `L1_GAS: max_amount = 2**64 - 1, max_price_per_unit = 2**65`
   - `L2_GAS: max_amount = 1, max_price_per_unit = 1`
   - `L1_DATA_GAS: max_amount = 1, max_price_per_unit = 1`
   - `tip = 0`

2. Transaction hash computation calls `pack_resource_bounds`:
   - `assert_nn_le(2**64 - 1, 2**64 - 1)` ✓
   - `assert_nn(2**65)` ✓ (since `2**65 < 2**128`)
   - Hash is computed and matches; transaction is accepted into the mempool.

3. Sequencer includes the transaction in a block and runs the OS.

4. `compute_max_possible_fee` computes:
   `max_fee = (2**64 - 1) × 2**65 + 1 × 1 + 1 × 1 ≈ 2**129`

5. `charge_fee` calls `assert_nn_le(actual_fee, 2**129)`:
   - `assert_nn(actual_fee)` ✓ (sequencer sets `actual_fee = 0`)
   - `assert_nn(2**129 - 0)` → `2**129` does **not** fit in `[0, 2**128)` → **range check fails**

6. The Cairo OS program fails. The block proof is invalid. The network cannot finalize the block. [1](#0-0) [4](#0-3) [5](#0-4)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
