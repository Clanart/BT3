Audit Report

## Title
Leftover ETH on Router Subsidizes Attacker's WETH Swap via `PeripheryPayments.pay()` Partial-ETH Branch — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` contains a partial-ETH branch that consumes the router's entire native ETH balance — regardless of who deposited it — to subsidize any WETH swap. Because `multicall` is `payable` and imposes no automatic refund, excess ETH from a prior user's transaction persists on the router. A subsequent attacker calling `exactInputSingle` with `tokenIn=WETH` triggers the partial branch, which deposits the prior user's ETH as WETH and reduces the WETH pulled from the attacker by exactly that amount, causing a direct ETH principal loss for the prior user.

## Finding Description
`PeripheryPayments.pay()` at lines 73–84 contains three branches for WETH payment: [1](#0-0) 

The partial branch (lines 78–81) fires when `0 < address(this).balance < value`. It deposits all router ETH as WETH, transfers it to the pool, then pulls only `value - nativeBalance` WETH from the actual payer. There is no ownership check — any ETH on the router, regardless of who deposited it, is consumed.

ETH accumulates between transactions because:

1. `receive()` only reverts bare ETH transfers from non-WETH addresses; ETH sent with calldata (e.g., `multicall{value: X}(...)`) bypasses `receive()` entirely and is silently credited to the router. [2](#0-1) 

2. The full branch (`nativeBalance >= value`) deposits only the exact `value` needed, leaving any excess on the router. [3](#0-2) 

3. `multicall` is `payable` and imposes no automatic refund; `refundETH()` is opt-in. [4](#0-3) 

**Exploit flow:**
- User A calls `multicall{value: 1.5 ETH}([exactInputSingle(tokenIn=WETH, amountIn=1 ETH)])` without appending `refundETH()`. The full branch deposits 1 ETH; 0.5 ETH remains on the router.
- Attacker B calls `exactInputSingle(tokenIn=WETH, amountIn=1.5 ETH)` with no ETH sent. `exactInputSingle` sets `msg.sender` (attacker) as payer via `_setNextCallbackContext`. [5](#0-4) 
- The swap callback invokes `pay(WETH, attacker, pool, 1.5 ETH)`. With `nativeBalance = 0.5 ETH < 1.5 ETH`, the partial branch deposits User A's 0.5 ETH as WETH and pulls only 1 ETH of WETH from the attacker. [6](#0-5) 
- The pool receives the correct 1.5 ETH-equivalent input; User A's 0.5 ETH is permanently consumed.

## Impact Explanation
Direct ETH principal loss for an unprivileged user. User A loses 0.5 ETH with no recourse; the attacker receives a 1.5 ETH-equivalent swap while paying only 1 ETH of WETH. The pool accounting remains intact, so the loss falls entirely on User A. This constitutes a direct loss of user principal above Sherlock thresholds, qualifying as High severity.

## Likelihood Explanation
The prerequisite — a user omitting `refundETH()` from a native-ETH multicall — is a realistic and common mistake. The protocol provides no enforcement and the pattern is opt-in. MEV bots can monitor the router's ETH balance on-chain and immediately exploit any nonzero balance in the next block. The attack requires no special permissions, no malicious pool, no non-standard tokens, and is fully repeatable.

## Recommendation
Remove the partial-ETH subsidy branch entirely. Two safe alternatives:

1. **Track per-call ETH via transient storage**: Record `msg.value` at the top of each swap entry point and use only that amount in `pay()`, reverting if insufficient.
2. **Eliminate the partial branch**: Only use the router's ETH balance when `msg.value` was explicitly provided in the current call (i.e., `msg.value >= value`); otherwise fall through directly to `safeTransferFrom`.

Additionally, consider adding an automatic `refundETH()` at the end of each swap entry point to prevent ETH from accumulating across transactions.

## Proof of Concept
```solidity
function test_leftoverEthSubsidizesAttacker() public {
    uint256 amountIn = 1 ether;
    uint256 excess   = 0.5 ether;

    // User A: sends 1.5 ETH but only swaps 1 ETH worth; forgets refundETH()
    vm.deal(userA, amountIn + excess);
    vm.prank(userA);
    bytes[] memory callsA = new bytes[](1);
    callsA[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        ExactInputSingleParams({ tokenIn: address(weth), amountIn: amountIn, /* ... */ })
    );
    router.multicall{value: amountIn + excess}(callsA);
    // 0.5 ETH remains on router; full branch consumed only 1 ETH
    assertEq(address(router).balance, excess);

    // Attacker B: approves 1.5 ETH of WETH but only holds 1 ETH worth
    deal(address(weth), attacker, amountIn); // only 1 ETH of WETH
    vm.prank(attacker);
    weth.approve(address(router), amountIn + excess);
    router.exactInputSingle(
        ExactInputSingleParams({ tokenIn: address(weth), amountIn: amountIn + excess, /* ... */ })
    );

    // Router drained: User A's 0.5 ETH consumed
    assertEq(address(router).balance, 0);
    // Attacker's WETH fully spent (1 ETH), but received 1.5 ETH-equivalent swap output
    assertEq(weth.balanceOf(attacker), 0);
}
```
