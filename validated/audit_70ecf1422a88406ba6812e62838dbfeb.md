Looking at the actual code to validate each claim component.

Audit Report

## Title
Surplus native ETH sent to `exactOutputSingle` / `exactOutput` is permanently stranded and stealable via permissionless `refundETH()` - (File: metric-periphery/contracts/base/PeripheryPayments.sol, metric-periphery/contracts/MetricOmmSimpleRouter.sol)

## Summary

`exactOutputSingle` and `exactOutput` are `payable` and accept native ETH as a WETH substitute through `pay()`. When `msg.value` exceeds the actual `amountIn` determined by the pool, `pay()` wraps only the exact amount requested, leaving the surplus as raw ETH in the router with no automatic refund. Because `refundETH()` is a permissionless `external` function that sends the entire router ETH balance to `msg.sender`, any third party can call it in a subsequent transaction and steal the stranded ETH.

## Finding Description

`pay()` in `PeripheryPayments.sol` handles the WETH branch by wrapping exactly `value` (the pool-requested amount), not `address(this).balance`:

```solidity
// PeripheryPayments.sol L73-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // wraps only `value`
        IERC20(WETH).safeTransfer(recipient, value);
```

When `nativeBalance > value`, the difference `nativeBalance - value` remains as raw ETH in the router.

`exactOutputSingle` (L130-147) and `exactOutput` (L154-188) are both `payable` and contain no post-swap ETH refund logic. After the swap settles and `amountIn` is confirmed, execution ends with `_clearExpectedCallbackPool()` — no `address(this).balance` check, no `_transferETH` call.

`refundETH()` is a public, permissionless function with no access control:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) does not protect against this: it only applies to plain ETH transfers with empty calldata. ETH sent alongside a `payable` function call bypasses `receive()` entirely and is accepted by the EVM directly.

The exploit path:
1. User calls `exactOutputSingle{value: X}(...)` where `tokenIn == WETH` and `X > actualAmountIn`.
2. Pool callback fires; `_justPayCallback` calls `pay(WETH, user, pool, actualAmountIn)`.
3. `pay()` wraps exactly `actualAmountIn`, sends WETH to pool. `X - actualAmountIn` ETH remains in router.
4. Attacker calls `router.refundETH()` — receives the entire stranded balance.

The same pattern applies to `exactOutput` (multi-hop) and to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` / `addLiquidityWeighted`, which are also `payable` and inherit the same `pay()` / `refundETH()` pair.

## Impact Explanation

Direct loss of user principal (native ETH). The surplus is not dust — it scales with the size of the overpayment. A user who sends a 10–20% ETH buffer to avoid slippage reverts (a common pattern) loses that entire buffer to any attacker who monitors the mempool. This meets the "Critical/High/Medium direct loss of user principal above Sherlock thresholds" criterion.

## Likelihood Explanation

Two realistic, non-malicious triggers exist without any special attacker setup:

1. **Price movement between quote and execution**: A user quotes `amountIn` off-chain, sends that as `msg.value`, but the pool price moves favorably so the actual `amountIn` is smaller. The surplus is stranded.
2. **Deliberate buffer**: A user intentionally sends slightly more ETH than the quote to avoid `InputTooHigh` reverts. The buffer is stranded.

The attacker only needs to monitor the mempool for transactions that leave ETH in the router and call `refundETH()` in the same block. No privileged access, no special setup, fully repeatable.

## Recommendation

Add an automatic ETH refund at the end of `exactOutputSingle` and `exactOutput` when `tokenIn == WETH`:

```solidity
// After amountIn check in exactOutputSingle / exactOutput:
uint256 surplus = address(this).balance;
if (surplus > 0) _transferETH(msg.sender, surplus);
```

Alternatively, enforce that callers use `multicall([exactOutputSingle(...), refundETH()])` and add a `nonPayable` guard to the single-call entry points (breaking change). The same fix should be applied to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` and `addLiquidityWeighted`.

## Proof of Concept

```
Setup:
  Pool price: 1 ETH in → 1500 token1 out (quoted off-chain)

1. User calls:
     router.exactOutputSingle{value: 1.2 ether}(ExactOutputSingleParams{
         tokenIn: WETH, amountOut: 1500, amountInMaximum: 2 ether, ...
     })

2. Pool determines it needs 1.0 ETH. Callback fires:
     pay(WETH, user, pool, 1.0 ETH)
     → IWETH9.deposit{value: 1.0 ether}()
     → 0.2 ETH remains in router

3. exactOutputSingle returns. No refund. Router holds 0.2 ETH.

4. Attacker (watching mempool) calls:
     router.refundETH()
     → _transferETH(attacker, 0.2 ether)
     → Attacker receives 0.2 ETH. User's 0.2 ETH is permanently lost.

Foundry test skeleton:
  function test_exactOutputSingle_surplusEthStolenByAttacker() public {
      uint256 actualIn = /* quote */ ...;
      uint256 msgValue = actualIn + 0.2 ether;
      vm.prank(user);
      router.exactOutputSingle{value: msgValue}(...);
      // router now holds 0.2 ETH
      assertEq(address(router).balance, 0.2 ether);
      vm.prank(attacker);
      router.refundETH();
      assertEq(attacker.balance, 0.2 ether);
  }
```
