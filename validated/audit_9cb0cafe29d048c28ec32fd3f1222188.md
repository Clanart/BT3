### Title
Small-Amount Transfer Permanently Locks User Funds via `normalize_amount` Rounding to Zero — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a cross-chain transfer of a token whose NEAR-side decimals differ from its origin-chain decimals, sending an amount smaller than the decimal-scaling factor causes `normalize_amount` to return `0`. The `sign_transfer` function then permanently rejects the transfer with `ERR_INVALID_AMOUNT_TO_TRANSFER`, while the user's tokens are already irrecoverably locked or burned. There is no cancel or refund path for the user.

---

### Finding Description

The bridge stores two decimal values per token: `decimals` (the NEAR-side representation) and `origin_decimals` (the origin chain's representation). When `origin_decimals > decimals`, the bridge scales amounts down via integer floor division in `normalize_amount`:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

This is called inside `sign_transfer` after the user's tokens are already locked:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

The token locking/burning happens earlier, inside `init_transfer` / `ft_on_transfer`, before `sign_transfer` is ever called. Once tokens are locked, there is no user-callable cancel or refund function exposed in the contract. [3](#0-2) 

The `bind_token_callback` stores `decimals` and `origin_decimals` from the on-chain proof, which can legitimately differ (e.g., a NEAR token with 24 decimals bridged to an EVM chain with 18 decimals): [4](#0-3) 

The test environment confirms this pattern — `bind_token` is called with `destination_decimals=18` and `origin_decimals=24`, giving `diff_decimals = 6`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any user who sends fewer than `10^diff_decimals` base units of a token with a decimal gap will have their tokens permanently locked in the bridge (for native tokens) or burned (for deployed tokens), with no recovery path. For a token with `origin_decimals=24` and `decimals=18`, any transfer of fewer than `1,000,000` base units triggers this. The `sign_transfer` call will always revert, and the transfer message sits in storage indefinitely.

---

### Likelihood Explanation

**Medium.** Tokens with decimal normalization (`origin_decimals > decimals`) are a standard part of the bridge design — the Solana bridge explicitly caps decimals at `MAX_ALLOWED_DECIMALS` and records `origin_decimals` separately: [6](#0-5) 

Any user sending a "dust" amount — whether accidentally or as a test — on such a token will lose funds. The protocol does not validate the amount against the decimal scaling factor at `init_transfer` time.

---

### Recommendation

Add a minimum-amount check at `init_transfer` time, before tokens are locked, to ensure `normalize_amount(amount - fee, decimals) > 0`. This mirrors the existing check in `sign_transfer` but must be enforced earlier, at the point of token custody transfer:

```rust
let normalized = Self::normalize_amount(
    amount.0.checked_sub(init_transfer_msg.fee.0).ok_or(...)?,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

Alternatively, expose a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have never been signed.

---

### Proof of Concept

1. Token is registered via `bind_token` with `decimals=18`, `origin_decimals=24` → `diff_decimals=6`, scaling factor = `1_000_000`.
2. User calls `ft_transfer_call` with `amount=500_000` (< `1_000_000`) and `fee=0`.
3. Tokens are locked in the bridge contract inside `init_transfer_internal`.
4. Relayer calls `sign_transfer(transfer_id, ...)`.
5. `normalize_amount(500_000, Decimals{decimals:18, origin_decimals:24})` = `500_000 / 1_000_000` = **0**.
6. `require!(0 > 0, ...)` panics → `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. No relayer can ever sign this transfer. The user's `500_000` tokens are permanently locked with no cancel path. [7](#0-6) [8](#0-7)
