### Title
Missing Validation of `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the caller-supplied `class_hash` corresponds to a previously declared contract class. A contract can replace its own class with an arbitrary, undeclared class hash, permanently rendering itself inaccessible and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the user-controlled syscall request and writes it into `contract_state_changes` without any check that the hash exists in the declared-class registry (`contract_class_changes`):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. After the state update, any subsequent call to the contract triggers `execute_entry_point`, which does:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

If `class_hash` was never declared, `dict_read` returns the default value `0`, and `find_element` will fail to locate a compiled class, causing the OS to panic. The contract becomes permanently inaccessible.

This is the direct analog of M-03: just as the Zap contract accepted any ERC20 token address without validating it was wBTC or renBTC, the OS accepts any `class_hash` without validating it is a declared class.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds funds (e.g., a multi-sig wallet, a DeFi vault, or any account contract) and calls `replace_class` with an undeclared class hash becomes permanently inaccessible. No future transaction can execute against it because the OS will always fail to resolve the compiled class. Funds stored in the contract's storage or sent to its address are irrecoverably frozen. This is not a revert — the state change is committed to the block and cannot be undone.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is callable by any contract from within its own execution context — no privileged role is required. A malicious contract deployer can intentionally trigger this to grief users who interact with their contract (e.g., a fake DeFi protocol that calls `replace_class` with a garbage hash after collecting deposits). A buggy contract could also trigger this accidentally. The entry path is fully reachable by an unprivileged transaction sender.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., has a non-zero compiled class hash entry). The check should be:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This is already acknowledged by the development team via the TODO comment at line 898. [3](#0-2) 

---

### Proof of Concept

1. An attacker deploys a contract `VaultContract` that accepts user deposits (holds funds in storage).
2. The attacker's contract calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary, never-declared hash).
3. The OS processes `execute_replace_class`: no validation is performed; `contract_state_changes[VaultContract.address].class_hash` is set to `0xdeadbeef`.
4. The block is proven and committed. The state change is final.
5. Any subsequent transaction targeting `VaultContract` reaches `execute_entry_point`, which calls `dict_read{dict_ptr=contract_class_changes}(key=0xdeadbeef)` → returns `0`. `find_element` with key `0` fails (no compiled class with hash 0 exists).
6. All calls to `VaultContract` permanently revert at the OS level. All deposited funds are frozen forever. [1](#0-0) [4](#0-3)

        // Not enough gas to execute the syscall.
        return ();
