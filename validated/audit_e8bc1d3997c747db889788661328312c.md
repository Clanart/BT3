Audit Report

## Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens — (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address from the caller and stores it as the trusted callback caller in transient storage without verifying it against the factory registry. A victim who has approved the adder for their tokens can be tricked into calling `addLiquidityExactShares` or `addLiquidityWeighted` with an attacker-controlled pool address, causing the adder to transfer up to `maxAmountToken0`/`maxAmountToken1` directly from the victim's wallet to the malicious pool.

## Finding Description

The contract explicitly documents the missing guard at L19–21:

> "This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement." [1](#0-0) 

`addLiquidityExactShares` passes the caller-supplied `pool` directly to `_addLiquidity` with no factory check: [2](#0-1) 

`_addLiquidity` stores the attacker-controlled address as `expectedPool` in transient storage via `_setPayContext`, then calls `addLiquidity` on it: [3](#0-2) 

Inside `metricOmmModifyLiquidityCallback`, the only caller check is `msg.sender != expectedPool`. Because `expectedPool` was set to the malicious pool, this check passes. The malicious pool then calls back with the full user-supplied caps: [4](#0-3) 

The malicious pool also controls `getImmutables()`, so it can return any token addresses. The `pay` call then executes `safeTransferFrom(payer, msg.sender, amount)` where `payer` is the victim and `msg.sender` is the malicious pool: [5](#0-4) [6](#0-5) 

By contrast, `MetricOmmSwapRouterBase._setNextCallbackContext` calls `_requireFactoryPool(pool)` before storing any pool in transient context, and `_requireExpectedCallbackCaller` additionally checks `FACTORY.isPool(caller)` at callback time: [7](#0-6) [8](#0-7) 

`MetricOmmPoolLiquidityAdder` has no equivalent guard at any point in its call path.

## Impact Explanation

Any user who has approved `MetricOmmPoolLiquidityAdder` for token0 and/or token1 — a prerequisite for normal LP use — can have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 transferred directly to the attacker's malicious pool contract. The malicious pool never credits LP shares; the tokens are simply stolen. This is a direct, complete loss of user principal with no recovery path. Severity: **Critical**.

## Likelihood Explanation

The attack requires:
1. The victim has approved the adder (standard for any LP user).
2. The attacker tricks the victim into calling `addLiquidityExactShares(maliciousPool, ...)` — e.g., via a phishing UI that substitutes the pool parameter, or a malicious frontend presenting a fake pool as legitimate.

The adder is a shared, approved contract, making the social-engineering surface real and repeatable. No privileged access is required.

## Recommendation

Add a factory registry check inside `_addLiquidity` (or at each public entry point) before storing the pool in transient context, mirroring the pattern in `MetricOmmSwapRouterBase`:

```solidity
// In MetricOmmPoolLiquidityAdder._addLiquidity or each public entry point:
if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
```

This requires injecting the factory address at construction time, exactly as `MetricOmmSwapRouterBase` does via its `FACTORY` immutable.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

contract MaliciousPool {
    address public immutable token0;
    address public immutable token1;
    address public immutable adder;

    constructor(address _token0, address _token1, address _adder) {
        token0 = _token0; token1 = _token1; adder = _adder;
    }

    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token0; imm.token1 = token1;
    }

    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Callback with full caps; msg.sender == expectedPool check passes
        MetricOmmPoolLiquidityAdder(payable(adder))
            .metricOmmModifyLiquidityCallback(
                1_000_000e18, 1_000_000e18, abi.encode(uint8(1))
            );
        return (1_000_000e18, 1_000_000e18);
    }
}

// Attack sequence:
// 1. victim.approve(adder, token0, type(uint256).max)
// 2. victim.approve(adder, token1, type(uint256).max)
// 3. attacker deploys MaliciousPool(token0, token1, adder)
// 4. attacker tricks victim into calling:
//      adder.addLiquidityExactShares(
//          address(maliciousPool), victim, 0, deltas,
//          1_000_000e18, 1_000_000e18, ""
//      )
// 5. adder stores maliciousPool as expectedPool, calls maliciousPool.addLiquidity
// 6. maliciousPool calls back metricOmmModifyLiquidityCallback with full caps
// 7. msg.sender == expectedPool check passes (both are maliciousPool)
// 8. adder calls pay(token0, victim, maliciousPool, 1_000_000e18)
//    → safeTransferFrom(victim, maliciousPool, 1_000_000e18) ← STOLEN
// 9. adder calls pay(token1, victim, maliciousPool, 1_000_000e18) ← STOLEN
```
