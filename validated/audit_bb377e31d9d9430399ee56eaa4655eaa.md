### Title
Missing Chain-Domain Separator in `deployToken` Signature Enables Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
The MPC signature verified in `OmniBridge.deployToken` covers only `(PayloadType.Metadata, token, name, symbol, decimals)`. It contains no chain identifier (`omniBridgeChainId`, EVM `block.chainid`, or contract address). Any two `OmniBridge` deployments sharing the same `nearBridgeDerivedAddress` will accept the identical `(signatureData, MetadataPayload)` tuple, allowing an unprivileged relayer to replay a legitimately obtained signature from Chain A onto Chain B.

### Finding Description
In `OmniBridge.deployToken`, the Borsh-encoded message that is hashed and ECDSA-verified is:

```
PayloadType.Metadata || token || name || symbol || decimals
``` [1](#0-0) 

No field in this payload identifies the destination EVM chain. The only signer-identity check is `ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress`. If two `OmniBridgeWormhole` instances are initialized with the same `nearBridgeDerivedAddress` (a realistic scenario when the same NEAR MPC derivation path is reused across chains), the recovered address will match on both, and the signature passes on both.

The duplicate-deployment guard `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` only checks the local mapping of the instance being called. [2](#0-1) 

It provides no cross-chain protection.

By contrast, `finTransfer` correctly embeds `omniBridgeChainId` in its signed payload: [3](#0-2) 

The inconsistency is the root cause.

### Impact Explanation
1. A relayer obtains a valid `deployToken` signature for NEAR token `T` on Chain A.
2. The relayer submits the identical `(signatureData, MetadataPayload)` to Chain B's `deployToken`. The signature check passes because `nearBridgeDerivedAddress` is the same.
3. A new `BridgeToken` proxy is deployed on Chain B and registered in `isBridgeToken`, `ethToNearToken`, and `nearToEthToken`.
4. `OmniBridgeWormhole.deployTokenExtension` emits a Wormhole message to NEAR announcing the token at the new proxy address on Chain B. [4](#0-3) 

5. NEAR's bridge, upon processing this Wormhole message, registers the token as valid on Chain B. Subsequent `finTransfer` calls to Chain B (with signatures legitimately issued by NEAR for Chain B's `omniBridgeChainId`) will mint tokens on Chain B against collateral that was only locked on Chain A, enabling unbacked supply.

**Impact**: High — cross-chain replay enabling unauthorized token deployment and potential unbacked minting, matching the "duplicate settlement / unbacked supply" impact category.

### Likelihood Explanation
The precondition — two `OmniBridge` deployments sharing the same `nearBridgeDerivedAddress` — is realistic. The NEAR MPC system derives an Ethereum address from a NEAR account ID and a path string. If the same path is used for multiple EVM chains (e.g., `"bridge"` on both Ethereum and Arbitrum), the derived address is identical. The attacker needs only a single valid `deployToken` signature (obtainable by observing any prior legitimate deployment on Chain A) and the ability to call a public function on Chain B.

### Recommendation
Include the chain domain in the `deployToken` signed payload, consistent with how `finTransfer` already does it:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD THIS
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

This makes each `deployToken` signature valid for exactly one `omniBridgeChainId`, preventing cross-chain replay.

### Proof of Concept
1. Deploy two `OmniBridgeWormhole` instances: Chain A (`omniBridgeChainId = 1`) and Chain B (`omniBridgeChainId = 2`), both initialized with the same `nearBridgeDerivedAddress = SIGNER`.
2. Obtain a valid MPC signature `sig` over `keccak256(Metadata || "token.near" || "Token" || "TKN" || 18)` from `SIGNER`.
3. Call `chainA.deployToken(sig, {token: "token.near", name: "Token", symbol: "TKN", decimals: 18})` — succeeds, deploys proxy `addrA`.
4. Call `chainB.deployToken(sig, {token: "token.near", name: "Token", symbol: "TKN", decimals: 18})` — succeeds with the **identical** signature, deploys proxy `addrB`.
5. Assert `chainB.isBridgeToken[addrB] == true` and `chainB.nearToEthToken["token.near"] == addrB`.
6. Construct a valid `finTransfer` payload for Chain B (with `omniBridgeChainId = 2`, `tokenAddress = addrB`) signed by `SIGNER`, and call `chainB.finTransfer(...)` — mints tokens on Chain B without any corresponding lock on Chain B. [5](#0-4)
