### Title
`batch_insert` Skips Duplicate-Key/Hash Guards in Bulk Path, Corrupting DataLayer Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` contains two distinct code paths. When the tree already holds two or more leaves (`leaf_count > 1`), the function routes **every** item in the batch directly through `insert_entry_to_blob` without consulting `block_status_cache`. The duplicate-key and duplicate-hash guards that `insert` enforces are completely absent in this path. A caller who supplies a batch containing a `KeyId` that already exists in the tree — or that appears twice within the same batch — silently writes two leaf nodes with the same key into the blob, producing an incorrect Merkle root and invalidating all subsequent proofs of inclusion.

### Finding Description

`insert` (line 362) opens with two mandatory guards before touching the blob:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (line 570) has a branching structure:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // ← guards active
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← NO duplicate check
    let new_leaf_index = self.get_new_index();
    let new_block = Block { node: Node::Leaf(LeafNode { hash, key, value, .. }), .. };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

When `leaf_count > 1` (the common production case after any prior insertions), the `if` block is skipped entirely and **all** items in the batch are written through the unguarded loop. `insert_entry_to_blob` writes raw bytes to the blob; it does not call `block_status_cache.add_leaf`, so the cache is also not updated for these items. The subtree is then grafted onto the existing tree via `insert_subtree_at_key` (line 653), which does not re-validate key uniqueness.

**Concrete trigger:** a tree with ≥ 2 existing leaves, then:

```python
blob.batch_insert(
    [(KeyId(0), ValueId(99)), (KeyId(200), ValueId(200))],  # KeyId(0) already present
    [new_hash_a, new_hash_b]
)
```

Both items enter the unguarded loop. `KeyId(0)` is written a second time. The blob now contains two `LeafNode` entries with `key = 0`, violating the tree's fundamental uniqueness invariant.

### Impact Explanation

A tree with duplicate leaf keys has an incorrect root hash. `get_proof_of_inclusion` for the duplicated key will traverse to one of the two leaves non-deterministically; the proof it returns is either wrong or unverifiable against the committed root. Any DataLayer store whose root is derived from this blob has a forged or unprovable state. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

The Python binding `py_batch_insert` (line 1503) is a public API that accepts caller-supplied `keys_values` and `hashes` lists with no pre-validation beyond length matching. Any caller who can invoke `MerkleBlob.batch_insert` — including a DataLayer service endpoint reachable by untrusted clients — can supply a repeated `KeyId` and trigger the corruption. No cryptographic secret or privileged role is required.
