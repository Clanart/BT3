### Title
`batch_insert` Bypasses Duplicate Key/Hash Guards, Corrupting DataLayer Merkle Tree Roots - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces, allowing a caller to silently insert multiple leaf nodes with the same `KeyId` or the same leaf `Hash`. The resulting tree has an inconsistent `block_status_cache`, an incorrect Merkle root, and produces unreliable inclusion/exclusion proofs. The function is exposed directly through the Python binding `py_batch_insert`.

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different path. When the tree already has two or more leaves, the entire input vector bypasses `insert` entirely and goes straight to `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even in the `leaf_count() <= 1` branch, only the first two items are routed through `insert`; every subsequent item in the same batch still bypasses the guard. [3](#0-2) 

`BlockStatusCache::add_leaf` uses `HashMap::insert`, which silently overwrites the cache entry for a repeated key, leaving the earlier leaf node as an orphan in the raw blob:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
``` [4](#0-3) 

`BlockStatusCache::new` (called when loading a blob from disk) does detect the duplicate and returns `Error::KeyAlreadyPresent`, but only at load time: [5](#0-4) 

The integrity check (`check_integrity_on_drop`) is disabled in production builds:

```rust
check_integrity_on_drop: cfg!(test),
``` [6](#0-5) 

The Python binding `py_batch_insert` passes the caller-supplied vector directly to `batch_insert` with no additional deduplication: [7](#0-6) 

### Impact Explanation

A duplicate leaf node in the blob means:

1. **Corrupted Merkle root** — both leaf nodes contribute to the root hash, so the root no longer represents a valid set of unique key-value pairs.
2. **Forged inclusion proofs** — a proof path can be constructed to either copy of the duplicate leaf, making it possible to prove membership for a key-value pair that was logically overwritten or deleted.
3. **Inconsistent exclusion proofs** — the tree no longer satisfies the uniqueness invariant that exclusion proofs rely on; a verifier may accept or reject a proof incorrectly.
4. **Cross-node state divergence** — nodes that loaded the blob from disk (triggering `BlockStatusCache::new`) will fail with `KeyAlreadyPresent`, while the in-memory session that produced the blob continues operating on the corrupt state.

This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

`batch_insert` is the primary bulk-load path for DataLayer sync. Any DataLayer publisher (or any Python caller of `py_batch_insert`) that supplies a batch containing a repeated `KeyId` — whether maliciously or due to a bug — silently corrupts the subscriber's tree. No special privilege beyond being a DataLayer data provider is required. The fuzz target for `insert` correctly handles `KeyAlreadyPresent`, but there is no corresponding fuzz target for `batch_insert` with duplicate keys.



### Recommendation

Add the same duplicate guards at the top of the `batch_insert` loop that `insert` applies:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, pre-deduplicate the input vector and verify it against the existing cache before any blob writes begin. A fuzz target that exercises `batch_insert` with repeated keys should also be added.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate with 2 leaves so leaf_count() >= 2,
// causing batch_insert to skip ALL duplicate checks.
blob.insert(KeyId(0), ValueId(0), &hash_0, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &hash_1, InsertLocation::Auto {}).unwrap();

// batch_insert with a duplicate key — no error is returned.
blob.batch_insert(vec![
    ((KeyId(2), ValueId(2)), hash_2),
    ((KeyId(2), ValueId(99)), hash_2b),  // same KeyId, different value
]).unwrap();  // succeeds silently

// The blob now contains two leaf nodes with KeyId(2).
// The Merkle root is computed over both, making it invalid.
// Reloading from disk will fail with KeyAlreadyPresent,
// but the in-memory session continues with a corrupt tree.
blob.check_integrity().unwrap_err(); // integrity violation detected only here
```

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
