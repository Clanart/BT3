### Title
Password Parameter Completely Ignored in `personal_sendTransaction` and `personal_sign`, Enabling Unauthorized Signing from Any Keyring Account — (`rpc/namespaces/ethereum/personal/api.go`)

### Summary

Both `personal_sendTransaction` and `personal_sign` accept a password parameter per the Ethereum `personal_` namespace spec, but discard it entirely. The backend signs directly using the node's already-unlocked Cosmos SDK keyring with no password check whatsoever. Any caller who can reach the RPC endpoint and knows (or enumerates) a keyring address can sign transactions or arbitrary data from that account with any password string, including an empty one.

### Finding Description

`PrivateAccountAPI.SendTransaction` at `rpc/namespaces/ethereum/personal/api.go:131` and `PrivateAccountAPI.Sign` at line 145 both declare the password argument as `_ string`, discarding it before delegating to the backend. [1](#0-0) 

The backend `Backend.SendTransaction` (`rpc/backend/sign_tx.go:38`) performs only a keyring address lookup, then calls `msg.Sign(signer, b.clientCtx.Keyring)` — the password is never consulted. [3](#0-2) 

The backend `Backend.Sign` (`rpc/backend/sign_tx.go:128`) similarly performs only a keyring address lookup, then calls `b.clientCtx.Keyring.SignByAddress(...)` directly. [4](#0-3) 

The `personal` namespace is registered with `Public: false` in `rpc/apis.go:141`, but Ethermint's JSON-RPC server startup at `server/json_rpc.go:88` calls `rpcServer.RegisterName(api.Namespace, api.Service)` for every API returned by `GetRPCAPIsWithMempool` without inspecting the `Public` field. This means if `personal` appears in the operator's configured API list, it is fully reachable over HTTP/WS with no IPC restriction. [6](#0-5) 

### Impact Explanation

When the `personal` namespace is enabled, an unprivileged HTTP caller can:

1. Call `personal_listAccounts` to enumerate all keyring addresses held by the node.
2. Call `personal_sendTransaction` with any password (including `""`) to sign and broadcast a valid EIP-155 transaction from any of those addresses.
3. Call `personal_sign` with any password to produce a valid ECDSA signature over arbitrary data from any of those addresses.

This is a direct signer verification bypass: the password that is supposed to gate access to the private key is never checked. The result is unauthorized transaction submission and arbitrary message signing from node-managed accounts — matching the **High** impact category of signer verification bypass enabling forged execution and unauthorized account mutation.

### Likelihood Explanation

The `personal` namespace is not in the default API list, so nodes using the out-of-box configuration are not exposed. However:

- The namespace is explicitly documented and listed in `GetAPINamespaces()` as a supported option. [9](#0-8) 
- Operators enabling it (e.g., for wallet tooling, dApp backends, or following go-ethereum documentation) would reasonably expect the password to provide authentication, as the method comments state. [8](#0-7) 
- The `Public: false` flag gives a false sense of security — it is silently ignored by the HTTP server.
- There is no other authentication layer (no HTTP basic auth, no token) on the JSON-RPC server by default.

Likelihood is **medium** for any deployment that enables the `personal` namespace.

### Recommendation

- Implement actual password verification before signing. For the Cosmos SDK `file` keyring backend, use the password to re-derive or verify the key before calling `SignByAddress`. For other backends, require an explicit unlock step.
- Alternatively, remove the password parameter from the public API surface and document that the `personal` namespace relies solely on network-level access control (IPC/firewall), matching the actual implementation.
- Enforce the `Public: false` flag in the HTTP server registration path so that private APIs cannot be accidentally exposed over HTTP/WS regardless of the configured API list.

### Proof of Concept

```bash
# 1. Start an Ethermint node with personal namespace enabled
ethermintd start --json-rpc.api eth,net,web3,personal

# 2. Enumerate keyring accounts
curl -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: ["0xADDRESS"]

# 3. Send a transaction with a deliberately wrong password — succeeds
curl -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{"from":"0xADDRESS","to":"0xRECIPIENT","value":"0x1"}, "WRONG_PASSWORD"],
    "id":2
  }'
# Returns a transaction hash — transaction is signed and broadcast successfully

# 4. Sign arbitrary data with a wrong password — succeeds
curl -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0","method":"personal_sign",
    "params":["0xdeadbeef","0xADDRESS","WRONG_PASSWORD"],
    "id":3
  }'
# Returns a valid 65-byte ECDSA signature
```

The transaction at step 3 passes the full ante handler pipeline (EIP-155 signature verification, nonce check, balance check) because the signing itself is legitimate — the node's keyring signs it. The password bypass means the caller never needed to know the account's password to authorize the signing.

// Sign signs the provided data using the private key of address via Geth's signature standard.
func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
