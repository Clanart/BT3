Audit Report

## Title
Residual ETH in Router Consumed by Subsequent User's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` to determine how much native ETH to contribute toward a WETH payment. Because all swap entry-points are `payable` and no swap function automatically refunds excess ETH, ETH sent by one user can persist in the router across transaction boundaries. A subsequent user's WETH swap will silently consume that stranded ETH, causing the original sender to lose their ETH with no recourse.

## Finding Description
`pay()` contains three branches for `token == WETH` when `payer != address(this)`: [1](#0-0) 

The hybrid branch (`nativeBalance > 0 && nativeBalance < value`, line 78–81) wraps the router's **entire** native ETH balance and forwards it to the pool, then pulls only `value - nativeBalance` from `payer`'s WETH allowance. There is no check that the native ETH balance belongs to the current caller or was sent in the current transaction.

ETH accumulates in the router because:

1. All four swap entry-points are `payable`, so ETH sent alongside any call is silently accepted: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

2. `receive()` only blocks plain ETH transfers (no calldata); ETH sent alongside a function call bypasses it entirely: [6](#0-5) 

3. `refundETH()` is a separate, optional call — no swap function invokes it automatically: [7](#0-6) 

**Concrete residual-ETH vectors:**
- User A calls `exactInputSingle{value: 2 ETH}(...)` with `tokenIn = WETH` and `amountIn = 1 ETH`. The `nativeBalance >= value` branch wraps exactly 1 ETH; 1 ETH remains in the router.
- User A calls any swap with a non-WETH `tokenIn` but attaches ETH. The ETH is entirely unused and stays in the router.
- User A calls `multicall{value: X}(...)` without a `refundETH()` sub-call.

Once ETH is stranded, the next user whose WETH payment triggers the `nativeBalance > 0 && nativeBalance < value` branch has User A's ETH wrapped and forwarded to the pool on their behalf, reducing the WETH pulled from their own allowance. User A's ETH is irrecoverably consumed — `refundETH()` can no longer help because the ETH has already been wrapped and sent.

## Impact Explanation
Direct loss of user principal. User A's ETH is transferred to the pool to settle User B's swap. User A receives nothing in return. The pool receives the correct total so no pool-side invariant is broken, but the payer-attribution invariant is violated: User B's payment is partially funded by User A's funds. This constitutes a High-severity direct loss of user principal under Sherlock contest thresholds.

## Likelihood Explanation
Sending ETH with a payable router call without a paired `refundETH()` is a common integration mistake: frontend over-estimates ETH needed, a non-WETH swap is called with ETH attached, or a multicall omits the cleanup step. The router provides no on-chain protection. Any subsequent WETH swap of sufficient size will silently drain the residual balance. The condition is reachable by any unprivileged user with no special setup.

## Recommendation
In `pay()`, when `token == WETH` and `payer != address(this)`, only use native ETH that was explicitly sent in the current transaction. The simplest fix is to thread `msg.value` down from each entry-point and replace `address(this).balance` with that value. Alternatively, add an automatic `refundETH()` at the end of every swap entry-point to ensure no ETH persists across transaction boundaries. A third option is to remove the hybrid ETH/WETH path entirely and require callers to pre-wrap ETH before calling the router.

## Proof of Concept
```solidity
// Foundry test sketch
function test_staleEthConsumedBySubsequentWethSwap() public {
    // 1. User A calls exactInputSingle for a non-WETH pair, accidentally attaching 1 ETH.
    //    The ETH is unused; 1 ETH remains in the router.
    vm.deal(userA, 1 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
        pool: nonWethPool,
        tokenIn: token1,   // NOT WETH
        amountIn: 1_000,
        ...
    }));
    assertEq(address(router).balance, 1 ether); // ETH stuck

    // 2. User B swaps WETH -> token2, needing to pay 2 WETH.
    //    pay(WETH, userB, pool, 2e18) sees nativeBalance=1e18 > 0 and < 2e18.
    //    Router wraps 1 ETH (User A's) + pulls 1 WETH from User B.
    uint256 userBWethBefore = weth.balanceOf(userB);
    vm.prank(userB);
    router.exactInputSingle(ExactInputSingleParams({
        pool: wethPool,
        tokenIn: address(weth),
        amountIn: 2 ether,
        ...
    }));

    // User A's 1 ETH is gone — consumed for User B's swap
    assertEq(address(router).balance, 0);
    // User B only had 1 WETH pulled (not 2), benefiting from User A's ETH
    assertEq(userBWethBefore - weth.balanceOf(userB), 1 ether);
}
```
