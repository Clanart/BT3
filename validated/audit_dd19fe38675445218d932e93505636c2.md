### Title
Missing shard-position binding in `set_state_header` allows a malicious peer to install shard A's state root under shard B's `StateHeaderKey` — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader`, validates that the embedded chunk is *somewhere* in the block's `chunk_headers_root` Merkle tree via `verify_path`, but never asserts that the chunk's shard position in that tree equals the `shard_id` parameter used to key `StateHeaderKey(shard_id, sync_hash)`. A malicious peer can supply shard A's chunk (with its valid position-0 Merkle proof) while the syncing node calls `set_state_header(shard_id=B, ...)`, causing the header — and shard A's `prev_state_root` — to be stored under shard B's key. `set_state_part` then reads shard A's state root from that key and installs state parts under shard B's `StatePartKey`, initializing shard B with shard A's trie.

---

### Finding Description

**Entrypoint:** `set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five validation steps before writing to the DB:

**Step 1-2 — `validate_chunk_proofs`** (`chain/chain/src/validate.rs` lines 22-67): checks the chunk's internal hash, tx root, and receipts root. It does **not** check `chunk.shard_id() == shard_id`. [1](#0-0) 

**Step 3a — `verify_path` against `chunk_headers_root`** (lines 394-403): proves the chunk hash is *somewhere* in the block's chunk-header Merkle tree. `verify_path` is defined as:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [2](#0-1) 

It only checks that `compute_root_from_path(path, hash) == root`. It does **not** check which leaf index the item occupies. A sibling function `verify_path_with_index` exists and does check position via `verify_path_matches_index`, but it is not called here. [3](#0-2) 

The `chunk_headers_root` Merkle tree is ordered by shard index. Shard 0's chunk sits at leaf 0; shard 1's chunk sits at leaf 1. A valid Merkle proof for leaf 0 (shard 0's chunk) will pass `verify_path` against the same root, because the proof correctly reconstructs the root — the check is purely hash-equality, not position-equality. [4](#0-3) 

**Step 4 — receipt proof validation** (lines 447-503): uses the `shard_id` *parameter* (not the chunk's shard_id) to compute `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))`. An attacker can satisfy this by supplying the correct on-chain incoming receipts for shard B (public information). [5](#0-4) 

**Step 5 — `validate_state_root_node`** (lines 512-523): validates the supplied `state_root_node` against `chunk_inner.prev_state_root()` — the chunk's own state root, not shard B's expected state root. An attacker supplies shard A's state root node, which is consistent with shard A's chunk. [6](#0-5) 

**Storage** (lines 526-529): the header is written under `StateHeaderKey(shard_id, sync_hash)` — keyed by the *parameter* `shard_id`, not by the chunk's actual shard. [7](#0-6) 

**Downstream — `set_state_part`** (lines 534-560): reads the header for `shard_id`, extracts `state_root` from the stored chunk (shard A's root), validates incoming parts against that root, and writes them under `StatePartKey(sync_hash, shard_id, part_id)`. A malicious peer that also serves shard A's state parts will pass this validation, completing the installation of shard A's trie under shard B's key. [8](#0-7) 

---

### Impact Explanation

The exact corrupted value is **`shard_id` in `StateHeaderKey`**: a header whose embedded chunk belongs to shard A is stored under the DB key for shard B. All subsequent state-sync operations for shard B — part validation, trie installation, and `set_state_finalize` — operate against shard A's state root. The syncing node initializes shard B's trie with shard A's data, producing a permanently diverged local state for shard B.

---

### Likelihood Explanation

State sync responses are accepted from any peer on the network; no validator or operator privilege is required to respond to a `StateRequestHeader` message. The attack requires only that the malicious peer be selected as the state-sync provider for the victim node, which is a normal network-level condition during state sync. All five validation steps can be satisfied with on-chain public data (chunk bodies, Merkle paths, receipt proofs) without forging any cryptographic primitive.

---

### Recommendation

After extracting `chunk` from the header, add an explicit shard-identity guard before any further validation:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the bare `verify_path` call for the chunk Merkle proof with `verify_path_with_index`, supplying the shard's index in the epoch's shard layout as `part_idx` and the total number of shards as `num_merklized_parts`. This binds the proof to the correct leaf position and closes the positional ambiguity independently of the shard_id field check.

---

### Proof of Concept

```
1. Obtain a valid sync_hash for an epoch with ≥2 shards.
2. Fetch the real ShardStateSyncResponseHeaderV2 for shard 0
   (chunk_0, chunk_proof_0, prev_chunk_header_0, prev_chunk_proof_0,
    state_root_node_0).
3. Fetch the on-chain incoming receipt proofs for shard 1
   (incoming_receipts_proofs_1, root_proofs_1).
4. Construct a crafted ShardStateSyncResponseHeaderV2:
     chunk                  = chunk_0          // shard 0's chunk
     chunk_proof            = chunk_proof_0    // valid proof for leaf 0
     prev_chunk_header      = prev_chunk_header_0
     prev_chunk_proof       = prev_chunk_proof_0
     incoming_receipts_proofs = incoming_receipts_proofs_1  // shard 1's receipts
     root_proofs            = root_proofs_1
     state_root_node        = state_root_node_0
5. Call set_state_header(shard_id=1, sync_hash, crafted_header).
   Expected: Ok(()) — all five checks pass.
6. Call get_state_header(shard_id=1, sync_hash).
   Assert: returned header's chunk.shard_id() == 0  ← invariant violated.
7. Serve shard 0's state parts in response to shard 1 part requests.
   Call set_state_part(shard_id=1, ...) for each part.
   Expected: Ok(()) — parts validate against shard 0's state root.
8. Observe: shard 1's trie is now initialized with shard 0's state.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)

    pub fn set_state_part(
