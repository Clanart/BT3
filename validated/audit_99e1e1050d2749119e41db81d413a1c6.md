### Title
Integer Overflow in Relative Timelock Arithmetic Enables Timelock Bypass and Consensus Divergence — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary

The `check_time_locks` function uses `wrapping_add` when the `nowrap` flag is `false`, and `saturating_add` when it is `true`. Because the condition parser accepts `ASSERT_SECONDS_RELATIVE` values up to `u64::MAX` and `ASSERT_HEIGHT_RELATIVE` values up to `u32::MAX` without an upper bound, an attacker can craft a spend with an extreme relative timelock value that wraps around to a small number under `wrapping_add`, making the timelock appear already satisfied. Nodes running with `nowrap=false` accept the spend; nodes running with `nowrap=true` reject it, producing deterministic consensus divergence.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` evaluates relative timelock conditions for each spend in a bundle. For `ASSERT_HEIGHT_RELATIVE` it computes:

```rust
// nowrap=false path (legacy)
unspent.confirmed_block_index.wrapping_add(height_relative)
// nowrap=true path (new)
unspent.confirmed_block_index.saturating_add(height_relative)
``` [1](#0-0) 

The same pattern applies to `seconds_relative`, `before_height_relative`, and `before_seconds_relative`: [2](#0-1) 

The condition parser accepts `ASSERT_SECONDS_RELATIVE` with up to 8 bytes (values 0 through `u64::MAX`) and `ASSERT_HEIGHT_RELATIVE` with up to 4 bytes (values 0 through `u32::MAX`), with no upper bound enforced: [3](#0-2) 

The `sanitize_uint` function returns `SanitizedUint::Ok(r)` for any value that fits within the byte limit, so `u64::MAX` and `u32::MAX` are fully valid parsed values: [4](#0-3) 

The test suite explicitly documents the divergence:

```

    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),

#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },

``` [5](#0-4) 

**Concrete arithmetic for `ASSERT_HEIGHT_RELATIVE(u32::MAX)` with coin confirmed at height 100, checked at height 200:**

- `nowrap=false`: `100u32.wrapping_add(0xffff_ffff) = 99`. Check: `200 < 99` → false → `Ok(())` — timelock bypassed.
- `nowrap=true`: `100u32.saturating_add(0xffff_ffff) = u32::MAX`. Check: `200 < u32::MAX` → true → `Err(AssertHeightRelativeFailed)` — correctly rejected.

**For `ASSERT_BEFORE_HEIGHT_RELATIVE(u32::MAX)` the overflow inverts the other direction:**

- `nowrap=false`: `100u32.wrapping_add(0xffff_ffff) = 99`. Check: `200 >= 99` → true → `Err(AssertBeforeHeightRelativeFailed)` — a spend that should be valid (deadline is effectively never) is incorrectly rejected.
- `nowrap=true`: `100u32.saturating_add(0xffff_ffff) = u32::MAX`. Check: `200 >= u32::MAX` → false → `Ok(())` — correctly accepted. [6](#0-5) 

The Python binding exposes `nowrap` as a caller-controlled parameter, meaning different callers (or different software versions) can produce opposite verdicts on the same spend bundle: [7](#0-6) 

### Impact Explanation

Two distinct impacts:

1. **Timelock bypass (High):** On any node calling `check_time_locks` with `nowrap=false`, a spend bundle containing `ASSERT_SECONDS_RELATIVE(u64::MAX)` or `ASSERT_HEIGHT_RELATIVE(u32::MAX)` passes the timelock check immediately after the coin is confirmed (the wrapping sum underflows to a value already in the past). A coin that was intended to be locked for an astronomically long time is spendable at once.

2. **Consensus divergence (Critical):** Nodes using `nowrap=true` reject the same spend bundle that nodes using `nowrap=false` accept. Because `nowrap` is a runtime parameter passed from Python, a network where some nodes have upgraded to the new behavior and others have not will permanently fork on any block containing such a spend.

### Likelihood Explanation

- The attacker-controlled entry path is fully unprivileged: any CLVM puzzle can emit `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with an arbitrary value up to the type maximum.
- The condition parser imposes no upper bound beyond the byte-width limit, so `u64::MAX` and `u32::MAX` are accepted as valid parsed conditions.
- The divergence is deterministic and reproducible: the same bytes always produce opposite results under the two code paths.
- The `nowrap` transition period (old nodes vs. upgraded nodes) is the exact window of exposure.

### Recommendation

1. **Enforce an upper bound in the condition parser.** In `parse_args` for `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE`, reject values that would overflow when added to any plausible confirmation timestamp or block height. A safe upper bound for `height_relative` is `u32::MAX - max_block_height`; for `seconds_relative`, a value exceeding the maximum plausible Unix timestamp difference.

2. **Alternatively, unconditionally use `saturating_add`.** Remove the `nowrap` branch entirely and always saturate. This eliminates the divergence without requiring a parser change, and is the semantically correct behavior (a timelock of `u32::MAX` blocks should never be satisfiable).

3. **If the `nowrap` flag must remain for backward compatibility**, document it as a consensus-critical flag and ensure all nodes in the network transition simultaneously via a hard fork activation height, not as a caller-controlled runtime parameter.

### Proof of Concept

Craft a CLVM puzzle that returns the condition `(ASSERT_HEIGHT_RELATIVE . (0xffffffff))` (opcode 82, value = 4 bytes `0xff 0xff 0xff 0xff`). Submit a spend bundle spending a coin confirmed at block height 100 at current block height 200.

- `sanitize_uint(a, node, 4, ...)` returns `SanitizedUint::Ok(0xffff_ffff)` — no error.
- `check_time_locks(..., nowrap=false)`: `100u32.wrapping_add(0xffff_ffff) = 99`; `200 < 99` is false → spend accepted.
- `check_time_locks(..., nowrap=true)`: `100u32.saturating_add(0xffff_ffff) = 4294967295`; `200 < 4294967295` is true → spend rejected with `AssertHeightRelativeFailed`.

The same spend bundle is simultaneously valid and invalid depending on the `nowrap` flag, producing deterministic consensus divergence between nodes on opposite sides of the flag transition. [1](#0-0) [8](#0-7) [9](#0-8)

pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    // strip the leading zero byte if there is one
    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }

    Ok(SanitizedUint::Ok(u64_from_bytes(buf)))
