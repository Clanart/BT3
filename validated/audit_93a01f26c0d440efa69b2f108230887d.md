### Title
Integer Wrapping in `check_time_locks` Bypasses Relative Timelock Enforcement — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary

When `nowrap=false` (legacy/pre-fork mode), `check_time_locks` uses `wrapping_add` for relative timelock arithmetic. An attacker can craft a spend with `height_relative = u32::MAX` or `seconds_relative = u64::MAX` to wrap the sum to a value near zero, causing the timelock comparison to pass unconditionally and allowing the coin to be spent before its timelock expires.

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic paths for relative timelock enforcement:

- `nowrap=true` → `saturating_add` (correct: clamps at `u32::MAX`/`u64::MAX`, timelock always fails)
- `nowrap=false` → `wrapping_add` (legacy: overflows silently to a small value, timelock passes) [1](#0-0) 

For `ASSERT_HEIGHT_RELATIVE`, when `nowrap=false`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
```

If `confirmed_block_index = 10` and `height_relative = 0xFFFF_FFF6` (i.e., `u32::MAX - 9`), then `wrapping_add` produces `0`, and the check becomes `prev_height < 0`, which is always `false` → `Ok(())`. The timelock is bypassed. [2](#0-1) 

The same applies to `ASSERT_SECONDS_RELATIVE` with `u64` wrapping: [3](#0-2) 

The codebase's own tests confirm this behavior explicitly:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
height_relative = 0xffff_ffff → nowrap=false → Ok(())  ← timelock bypassed
seconds_relative = 0xffff_ffff_ffff_ffff → nowrap=false → Ok(()) ← timelock bypassed
``` [4](#0-3) 

The Python binding exposes `nowrap` as a caller-controlled parameter, meaning the full node determines which path is taken based on block height (pre-fork vs. post-fork): [5](#0-4) 

### Impact Explanation

This matches the **High** allowed impact: *Timelock condition validation bypass enables unauthorized spend acceptance.* A coin locked with `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` can be spent before its timelock expires when the node processes the spend under `nowrap=false`. The attacker does not need any key material — only the ability to submit a crafted spend bundle with a maximum-value relative timelock argument.

### Likelihood Explanation

`nowrap=false` is the legacy pre-fork path, still reachable for blocks below the hard-fork activation height. Any coin confirmed before the fork is eligible. The crafted condition value (`u32::MAX` or `u64::MAX`) is a valid CLVM atom that passes `sanitize_uint` without error (it is within the 4-byte / 8-byte range), so no prior validation step rejects it. [6](#0-5) 

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally in both the `height_relative` and `seconds_relative` branches, removing the `nowrap` bifurcation for these two checks. The `nowrap=false` path was introduced to preserve legacy behavior, but that legacy behavior is itself the vulnerability. If backward compatibility with pre-fork blocks is required, the correct fix is to treat any overflow as "timelock not yet satisfied" (i.e., saturate), which is what `nowrap=true` already does. [7](#0-6) 

### Proof of Concept

1. Create a coin confirmed at block height `10`.
2. Craft a spend bundle with condition `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFF6` (i.e., `u32::MAX - 9`).
3. Submit the spend bundle to a node running in legacy mode (`nowrap=False`).
4. The node computes `10u32.wrapping_add(0xFFFF_FFF6) = 0`.
5. The check `prev_height < 0` is `false` → `Ok(())`.
6. The spend is accepted at block height `15`, even though the intended timelock requires height `≥ 0xFFFF_FFFF` (effectively never, under correct arithmetic).

The test suite already encodes this exact scenario: [8](#0-7)
