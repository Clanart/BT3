### Title
Missing Finalization Re-Check in Async Callback Enables Double-Spend via Fast Transfer Race Condition - (File: near/omni-bridge/src/lib.rs)

### Summary
`fast_fin_transfer` checks `is_unified_transfer_finalised` before initiating an async cross-contract call, but the actual state mutation (`add_fast_transfer`) and token disbursement happen in the callback `fast_fin_transfer_to_near_callback`, which does **not** re-check finalization status. Between the initial check and the callback execution, `fin_transfer` can finalize the same transfer and pay the original recipient. The callback then proceeds to pay the recipient a second time, creating a double-spend and corrupting the bridge's locked-token accounting.

### Finding Description

In `fast_fin_transfer` (NEAR-recipient path), the finalization guard fires at line 778:

```rust
if self.is_unified_transfer_finalised(&fast_fin_transfer_msg.transfer_id) {
    env::panic_str(BridgeError::TransferAlreadyFinalised.to_string().as_str());
}
```

After this check, the function does **not** immediately register the fast transfer. Instead it schedules an async cross-contract call to check/pay FT storage, then chains `fast_fin_transfer_to_near_callback`:

```rust
Self::check_or_pay_ft_storage(&deposit_action, ...)
    .then(
        Self::ext(env::current_account_id())
            .fast_fin_transfer_to_near_callback(&fast_transfer, signer_id, relayer),
    )
```

The callback, which executes in a **separate NEAR receipt**, does not re-check finalization status:

```rust
pub fn fast_fin_transfer_to_near_callback(...) -> Promise {
    require!(Self::check_storage_balance_result(0), ...);
    // ← NO is_unified_transfer_finalised check here
    let required_balance = self
        .add_fast_transfer(fast_transfer, relayer_id, storage_payer.clone())
        ...;
    self.send_tokens(fast_transfer.token_id.clone(), recipient, amount_without_fee, ...)
```

Meanwhile, `process_fin_transfer_to_near` (called by `fin_transfer_callback`) first inserts into `finalised_transfers`, then checks `get_fast_transfer_status`. If the fast transfer has not yet been registered (because the callback hasn't run), it treats the transfer as a normal finalization and pays the original recipient:

```rust
let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
// ...
let fast_transfer_status = self.get_fast_transfer_status(&fast_transfer.id());
// fast_transfer_status is None → pays original recipient, not relayer
```

### Impact Explanation

**Race condition sequence:**

1. Relayer A calls `fast_fin_transfer` (via `ft_transfer_call`). The `is_unified_transfer_finalised` check passes. An async receipt for `check_or_pay_ft_storage` is queued.
2. Before the callback receipt executes, Relayer B calls `fin_transfer`. `process_fin_transfer_to_near` runs: `add_fin_transfer` inserts the transfer ID into `finalised_transfers`; `get_fast_transfer_status` returns `None` (fast transfer not yet registered); tokens are sent to the **original recipient**.
3. `fast_fin_transfer_to_near_callback` executes. It does not check `is_unified_transfer_finalised`. `add_fast_transfer` succeeds (fast transfer not in map). `send_tokens` sends tokens to the **recipient a second time**.

**Result:**
- The recipient receives `amount_without_fee` tokens twice.
- For native (locked) tokens: `unlock_tokens_if_needed` fires in `fin_transfer`, reducing `locked_tokens` by `amount`. The bridge then also disburses the relayer's pre-paid tokens, making the bridge insolvent by `amount_without_fee`.
- For bridged tokens: `fin_transfer` mints `amount_without_fee` tokens; the callback additionally transfers the relayer's tokens to the recipient, creating unbacked supply.

This matches the allowed impact: **double-finalization enabling double-spend or unbacked supply**.

### Likelihood Explanation

NEAR's asynchronous receipt model makes this race window real: any transaction submitted between the `check_or_pay_ft_storage` receipt and the `fast_fin_transfer_to_near_callback` receipt can interleave. Two independent honest relayers acting on the same transfer simultaneously (one doing fast-transfer, one doing standard finalization) can trigger this without any malicious coordination. The window is narrow but structurally guaranteed by the protocol's async design.

### Recommendation

Add a finalization re-check at the top of `fast_fin_transfer_to_near_callback`, mirroring the guard in `fast_fin_transfer`:

```rust
pub fn fast_fin_transfer_to_near_callback(
    &mut self,
    fast_transfer: &FastTransfer,
    storage_payer: AccountId,
    relayer_id: AccountId,
) -> Promise {
    require!(
        Self::check_storage_balance_result(0),
        BridgeError::StorageRecipientOmitted.as_ref()
    );
+   if self.is_unified_transfer_finalised(&fast_transfer.transfer_id) {
+       // Refund the relayer's tokens and abort
+       return self.send_tokens(fast_transfer.token_id.clone(), storage_payer, fast_transfer.amount, "");
+   }
    ...
```

This ensures that if `fin_transfer` completed between the initial call and the callback, the fast-transfer path aborts and the relayer's pre-paid tokens are returned rather than causing a double disbursement.
