### Title
Arbitrary ERC20 Token Address in `initTransfer` Without Whitelist Enables Fake Deposit and Unbacked Cross-Chain Mint — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `logMetadata` and `initTransfer` functions accept any arbitrary ERC20 token address with no whitelist validation. A malicious ERC20 contract can (a) return crafted metadata to register itself as a legitimate bridgeable token on NEAR, and (b) return `true` from `transferFrom` without moving any tokens, passing the `SafeERC20` check. The resulting `InitTransfer` event is indistinguishable from a legitimate one, causing the NEAR bridge to mint tokens for the attacker backed by zero real collateral.

---

### Finding Description

**Root cause — EVM side (`OmniBridge.sol`):**

`logMetadata` is permissionless and calls `name()`, `symbol()`, and `decimals()` on any caller-supplied address: [1](#0-0) 

`initTransfer` is also permissionless. For any token that is neither a registered bridge token nor has a custom minter, it falls into the unguarded `else` branch and calls `SafeERC20.safeTransferFrom` on the caller-supplied address: [2](#0-1) 

`SafeERC20.safeTransferFrom` succeeds if `transferFrom` returns `true` or returns no data. A malicious ERC20 contract can implement `transferFrom` to return `true` without moving any tokens, satisfying the check while depositing nothing.

There is no whitelist, `isBridgeToken` check, or any other guard on `tokenAddress` in either function.

**Root cause — NEAR side (`near/omni-bridge/src/lib.rs`):**

`deploy_token` on NEAR is permissionless (no role restriction, only a pause guard). It accepts a proof of a `LogMetadata` event and, after validating that the emitter is a registered factory, registers the token in `token_decimals`: [3](#0-2) [4](#0-3) 

`add_token` inserts the malicious ERC20 address into `token_decimals`, `token_id_to_address`, and `token_address_to_id`: [5](#0-4) 

`fin_transfer_callback` on NEAR then validates only that (1) the emitter is a registered factory and (2) the token is in `token_decimals`. Both conditions are satisfied after the attacker's setup: [6](#0-5) 

---

### Impact Explanation

**Critical — Direct unauthorized mint of bridged assets.**

An attacker can mint an arbitrary quantity of NEAR-side bridge tokens without depositing any real EVM-side collateral. The minted tokens are fully fungible and transferable. The bridge's collateral accounting is broken: the EVM bridge holds zero tokens while the NEAR side has minted a positive supply, permanently breaking collateralization for any user who later attempts to bridge back.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a malicious ERC20 contract (trivial, no special permissions).
2. Calling two permissionless EVM functions (`logMetadata`, `initTransfer`).
3. Submitting two permissionless NEAR calls (`deploy_token`, then waiting for a relayer to call `fin_transfer` — or calling it directly if the attacker operates a relayer).

No privileged role, leaked key, or colluding party is required. The entire attack is executable by any unprivileged EVM account.

---

### Recommendation

Implement a whitelist of permitted ERC20 token addresses for `initTransfer` and `logMetadata`, controlled by the contract owner/admin. Only tokens explicitly added to the whitelist should be accepted. This is the direct fix applied in the referenced external report (pull request #1).

Concretely:

- Add a `mapping(address => bool) public allowedTokens` storage variable.
- Add an `onlyRole(DEFAULT_ADMIN_ROLE)` function to add tokens to the whitelist.
- In `logMetadata` and `initTransfer`, add a `require(allowedTokens[tokenAddress], "ERR_TOKEN_NOT_ALLOWED")` guard before any external call to `tokenAddress`. [7](#0-6) 

---

### Proof of Concept

```
1. Attacker deploys MaliciousERC20:
   - name()     → "Fake USDC"
   - symbol()   → "FUSDC"
   - decimals() → 6
   - transferFrom(from, to, amount) → returns true, moves 0 tokens

2. Attacker calls OmniBridge.logMetadata(MaliciousERC20)
   → OmniBridge calls MaliciousERC20.name/symbol/decimals (returns crafted values)
   → Emits LogMetadata(MaliciousERC20, "Fake USDC", "FUSDC", 6)
   [Event is emitted by the registered OmniBridge factory — proof will be valid]

3. Attacker (or any relayer) calls NEAR deploy_token with proof of LogMetadata event
   → deploy_token_callback: factories.get(Eth) == OmniBridge ✓ (registered factory)
   → deploy_token_internal → add_token inserts MaliciousERC20 into token_decimals ✓

4. Attacker calls OmniBridge.initTransfer(
       MaliciousERC20, 1_000_000e6, 0, 0, "attacker.near", ""
   )
   → customMinters[MaliciousERC20] == address(0) → not custom minter
   → isBridgeToken[MaliciousERC20] == false → not bridge token
   → SafeERC20.safeTransferFrom(attacker, bridge, 1_000_000e6)
       → MaliciousERC20.transferFrom returns true, 0 tokens moved ✓
   → Emits InitTransfer(attacker, MaliciousERC20, nonce, 1_000_000e6, ...)

5. Relayer calls NEAR fin_transfer with proof of InitTransfer event
   → fin_transfer_callback:
       factories.get(Eth) == OmniBridge ✓
       token_decimals.get(MaliciousERC20) → Some(6) ✓ (registered in step 3)
   → NEAR mints 1_000_000 FUSDC tokens to attacker.near

Result: Attacker holds 1,000,000 NEAR-side tokens backed by 0 EVM-side collateral.
```
