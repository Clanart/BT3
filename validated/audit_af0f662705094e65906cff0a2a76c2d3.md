### Title
Integer Wrapping in `check_time_locks` Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Timelocks Under Legacy `nowrap=false` Path — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

When `check_time_locks` is invoked with `nowrap=false` (the pre-fork legacy path), relative timelock checks for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` use `wrapping_add` instead of `saturating_add`. An unprivileged spender can craft a CLVM puzzle that emits `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` (u32::MAX) or `ASSERT_SECONDS_RELATIVE 0xFFFF_FFFF_FFFF_FFFF` (u64::MAX). The wrapping addition produces a threshold smaller than the coin's confirmation height/timestamp, causing the guard to pass immediately and the timelock to be silently bypassed.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces relative timelocks by comparing the current block height/timestamp against `coin.confirmed_block_index + height_relative` (or `coin.timestamp + seconds_relative`). When `nowrap=false`, this addition is performed with `wrapping_add`:

```rust
// lines 64-68
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

```rust
// lines 75-77
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

The condition parser (`parse_args` in `conditions.rs`) accepts any `ASSERT_HEIGHT_RELATIVE` argument that fits in 4 bytes (up to `u32::MAX = 0xFFFF_FFFF`) and any `ASSERT_SECONDS_RELATIVE` argument that fits in 8 bytes (up to `u64::MAX`). Values exceeding those widths are rejected with `PositiveOverflow`, but the maximum in-range value is stored verbatim.

Concrete arithmetic (from the test suite at `check_time_locks.rs` lines 276–281):

- `confirmed_block_index = 100`, `height_relative = 0xFFFF_FFFF`
- `wrapping_add(100, 0xFFFF_FFFF) = 99` (wraps to 99)
- Guard: `prev_height < 99` → **false** at any block ≥ 99 → **no error returned**
- The spend is accepted at block 99, even though the intended lock is `100 + 4,294,967,295` blocks in the future.

The test case explicitly documents this divergence:

```
// nowrap=true  → Err(AssertHeightRelativeFailed)   (saturates, correctly rejects)
// nowrap=false → Ok(())                             (wraps, incorrectly accepts)
```

The same wrapping bypass applies to `seconds_relative` with `u64::MAX`.

### Impact Explanation

A spender who controls the puzzle (CLVM program) can emit `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` to create a coin that appears to carry a ~4-billion-block timelock but is actually spendable at any block height ≥ `confirmed_block_index − 1` when the validator uses `nowrap=false`. This is a **timelock validation bypass enabling unauthorized spend acceptance**, matching the High impact category: *"timelock or coin-id validation bypass enables unauthorized spend acceptance or replay."*

### Likelihood Explanation

The `nowrap` parameter is caller-controlled and is exposed directly through the Python binding `py_check_time_locks`. Any code path in chia-blockchain that invokes `check_time_locks` with `nowrap=False` (e.g., during a transition window, for pre-fork block re-validation, or due to a caller bug) is vulnerable. The attacker entry point requires only the ability to submit a spend bundle containing a puzzle that outputs the maximum-value relative timelock condition — a fully unprivileged operation. No special keys, roles, or network access beyond normal transaction submission are required.

### Recommendation

1. **Remove the `nowrap=false` branch entirely** once the soft-fork activating saturating arithmetic is considered final and all nodes have upgraded. Retaining the wrapping path as a permanently reachable code path is unnecessary and dangerous.
2. **If backward compatibility is still required**, gate the `nowrap=false` path strictly on block height (i.e., only apply it to blocks confirmed before the fork height), and never allow it for mempool/new-spend validation.
3. **Add a compile-time or runtime assertion** in `py_check_time_locks` that rejects `nowrap=false` for any block height above the fork activation height.

### Proof of Concept

Craft a puzzle that outputs:

```
(ASSERT_HEIGHT_RELATIVE . 0xFFFF_FFFF)
```

Parse it through `parse_args` / `parse_conditions` — the value is accepted as `height_relative = Some(0xFFFF_FFFF)` because it fits in 4 bytes.

Call `check_time_locks` with `nowrap=false`, `prev_transaction_block_height = 100`, and a coin record with `confirmed_block_index = 100`:

```
wrapping_add(100u32, 0xFFFF_FFFFu32) = 99u32
check: 100 < 99  →  false  →  Ok(())   ← timelock bypassed
```

The spend is accepted at block 100, despite the puzzle asserting a relative lock of `u32::MAX` blocks. The same construction with `seconds_relative = 0xFFFF_FFFF_FFFF_FFFF` bypasses the seconds-based relative timelock identically.
