### Title
Zero-Normalized Transfer Amount Permanently Locks/Burns User Tokens After `init_transfer_internal` Commits State Before `sign_transfer` Validates - (File: `near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR-side transfer of a token whose amount is too small to survive decimal normalization (i.e., `normalize_amount` returns 0), the bridge irrecoverably locks or burns the user's tokens. The zero-amount guard exists only in `sign_transfer`, which is called **after** `init_transfer_internal` has already committed the transfer message to storage and locked/burned the tokens. No public cancel or recovery path exists, leaving the funds permanently frozen.

### Finding Description

The NEAR bridge's `init_transfer_internal` function unconditionally locks or burns user tokens and stores the transfer message in `pending_transfers` before any check on whether the amount will survive normalization to the destination chain's decimal precision. [1](#0-0) 

Specifically, for a NEAR-origin token:
1. `add_transfer_message` inserts the transfer into `pending_transfers`.
2. `burn_tokens_if_needed` burns the tokens if the token is a deployed bridge token (irreversible).
3. `lock_tokens_if_needed` increments the `locked_tokens` counter.
4. Returns `U128(0)`, signaling the NEP-141 callback to keep all tokens (no refund). [2](#0-1) 

The zero-amount guard is placed only in `sign_transfer`, which is called later by a trusted relayer:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

The normalization function uses integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

Any amount strictly less than `10^(origin_decimals − decimals)` normalizes to 0. For example, a token registered with 24 NEAR-side decimals and 18 EVM-side decimals has a threshold of `10^6` base units. Sending fewer than `10^6` units causes `sign_transfer` to always revert with `BridgeError::InvalidAmountToTransfer`, while the tokens are already gone.

There is no public cancel or refund function. The internal `remove_transfer_message` helpers are not exposed to users: [5](#0-4) 

The `update_transfer_fee` function cannot help: it only allows increasing the fee (never decreasing it), and the fee is bounded strictly below the amount, so `amount_without_fee()` remains positive but still normalizes to 0. [6](#0-5) 

### Impact Explanation

- **Deployed (bridge-minted) tokens**: `burn_tokens_if_needed` destroys the tokens immediately and irreversibly. Even admin contract upgrades cannot recreate burned supply without minting new tokens, breaking collateralization.
- **Non-deployed (native NEAR) tokens**: tokens are held by the bridge contract and the `locked_tokens` counter is incremented. The user has no self-service recovery path; the funds are frozen until an admin upgrades the contract.

In both cases the transfer message remains in `pending_transfers` indefinitely, consuming storage and making the nonce permanently unusable for that transfer ID.

This matches: **Critical — Permanent freezing, irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

**Low.** The condition requires sending an amount below `10^(origin_decimals − decimals)` base units. For tokens with large decimal gaps (e.g., 24 NEAR decimals → 18 EVM decimals, threshold = 1,000,000 base units), this is a tiny but non-zero amount. For tokens with smaller gaps (e.g., 9 → 6, threshold = 1,000 base units), the risk is higher. A user could trigger this accidentally by sending "dust" amounts, or a malicious actor could grief another user by front-running a transfer with a crafted small amount (if the token allows it).

### Recommendation

Move the zero-normalized-amount check **into `init_transfer_internal`** (or into `init_transfer` before calling `init_transfer_internal`), before any tokens are locked or burned and before the transfer message is stored:

```rust
fn init_transfer_internal(
    &mut self,
    transfer_message: TransferMessage,
    storage_owner: AccountId,
) -> U128 {
    // NEW: validate normalized amount before committing any state
    if let OmniAddress::Near(ref token_id) = transfer_message.token {
        if let Some(token_address) = self.get_token_address(
            transfer_message.get_destination_chain(), token_id.clone()
        ) {
            if let Some(decimals) = self.token_decimals.get(&token_address) {
                let normalized = Self::normalize_amount(
                    transfer_message.amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                    decimals,
                );
                require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
            }
        }
    }
    // ... existing logic
}
```

This ensures that if the amount would normalize to 0, the NEP-141 callback returns the full amount as a refund before any state mutation occurs.

### Proof of Concept

1. Deploy a NEAR bridge token registered with `origin_decimals = 24` on NEAR and `decimals = 18` on EVM (threshold = `10^6` base units).
2. Call `ft_transfer_call` on the token contract with `amount = 500_000` (below threshold) and `msg = InitTransferMsg { recipient: <EVM address>, fee: 0, ... }`.
3. `ft_on_transfer` → `init_transfer` → `init_transfer_internal`:
   - Transfer message inserted into `pending_transfers`. [7](#0-6) 
   - `burn_tokens_if_needed` burns 500,000 tokens (if deployed token). [8](#0-7) 
   - Returns `U128(0)` — no refund issued. [9](#0-8) 
4. Relayer calls `sign_transfer(transfer_id, ...)`:
   - `normalize_amount(500_000, {decimals:18, origin_decimals:24})` = `500_000 / 1_000_000` = `0`. [4](#0-3) 
   - `require!(0 > 0, ...)` panics — transaction reverts. [10](#0-9) 
5. Transfer message remains in `pending_transfers`. Tokens are burned/locked. No public cancel function exists. Funds are permanently frozen.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();

            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
