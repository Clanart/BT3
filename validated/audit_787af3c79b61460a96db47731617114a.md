### Title
ERC721 Token Deposited via ERC20 `initTransfer` Path Causes Permanent Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` performs no token-type validation before pulling tokens from the caller. Because ERC721 shares the `transferFrom(address,address,uint256)` ABI signature with ERC20, an ERC721 NFT can be silently deposited through the ERC20 code path. The corresponding release path in `finTransfer` calls `safeTransfer(address,uint256)`, which does not exist on ERC721, causing every subsequent withdrawal attempt to revert. The NFT is permanently locked in the bridge with no admin recovery function.

---

### Finding Description

**Deposit path — `initTransfer`**

For any `tokenAddress` that is neither a bridge token nor a custom minter, the function executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← treated as ERC20 amount
);
``` [1](#0-0) 

ERC721 exposes `transferFrom(address from, address to, uint256 tokenId)`. The ABI selector and parameter layout are byte-for-byte identical to ERC20's `transferFrom`. When `amount` equals a token ID owned by `msg.sender`, the call succeeds and the NFT is transferred into the bridge. No ERC165 interface check, no whitelist, and no other guard prevents this.

**Release path — `finTransfer`**

The same token type falls through to:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [2](#0-1) 

ERC721 does not implement `transfer(address,uint256)`. `SafeERC20.safeTransfer` will revert on every call. There is no admin sweep, no emergency withdrawal, and no refund path in the contract, so the NFT is irrecoverably locked.

**No guard in `initTransfer`**

The full branching logic in `initTransfer` is:

```solidity
if (customMinters[tokenAddress] != address(0)) { … }
else if (isBridgeToken[tokenAddress])           { … }
else {
    IERC20(tokenAddress).safeTransferFrom(…);   // ← ERC721 lands here
}
``` [3](#0-2) 

Neither branch performs any token-standard check.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Once an ERC721 token is deposited via `initTransfer`, it cannot be released:

- `finTransfer` reverts because ERC721 has no `transfer(address,uint256)`.
- There is no admin recovery, emergency withdrawal, or sweep function in `OmniBridge`.
- The NEAR side will either reject the cross-chain message (token not registered → `TokenDecimalsNotFound` panic) or, if the NFT address was previously registered as an ERC20, mint unbacked NEAR-side tokens while the EVM-side asset remains permanently frozen.

This matches the allowed impact: *"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** Any unprivileged user can trigger this:

1. No role or whitelist check gates `initTransfer`.
2. The attacker only needs to own an ERC721 token and call `approve` on the bridge.
3. The function signature collision between ERC20 and ERC721 `transferFrom` is well-known and requires no special tooling to exploit.

The scenario is realistic for any ERC721 token whose address is not already registered as a bridge token or custom minter.

---

### Recommendation

Add an ERC165 interface check inside `initTransfer` before the ERC20 fallback branch:

```solidity
// Reject ERC721 tokens masquerading as ERC20
if (IERC165(tokenAddress).supportsInterface(type(IERC721).interfaceId)) {
    revert UnsupportedTokenType();
}
```

Alternatively, maintain an explicit ERC20 allowlist (separate from the ERC721/ERC1155 allowlist already implied by `multiTokens`) so that only pre-approved fungible tokens can be deposited via `initTransfer`.

---

### Proof of Concept

1. Attacker owns ERC721 token at `nftAddress` with `tokenId = 42`.
2. Attacker calls `nftAddress.approve(bridgeAddress, 42)`.
3. Attacker calls:
   ```solidity
   bridge.initTransfer(
       nftAddress,
       42,   // amount == tokenId
       0, 0,
       "attacker.near",
       ""
   );
   ```
4. Inside `initTransfer`, `nftAddress` is not a custom minter and not a bridge token, so the ERC20 fallback fires:
   ```solidity
   IERC20(nftAddress).safeTransferFrom(attacker, bridge, 42);
   ```
   ERC721's `transferFrom(attacker, bridge, 42)` executes successfully. NFT is now held by the bridge.
5. NEAR side receives the `InitTransfer` event. If the token is unregistered, `fin_transfer_callback` panics at `token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` — no NEAR-side settlement occurs, but the EVM-side NFT is already locked.
6. If `finTransfer` is ever called with `payload.tokenAddress = nftAddress, payload.amount = 42`, it executes:
   ```solidity
   IERC20(nftAddress).safeTransfer(recipient, 42);
   ```
   ERC721 has no `transfer(address,uint256)` → revert. NFT is permanently frozen in the bridge. [4](#0-3) [5](#0-4)
