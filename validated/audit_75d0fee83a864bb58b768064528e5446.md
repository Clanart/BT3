### Title
Reentrancy via ERC-777 Sender Hook Causes Permanent Fund Lock Through Nonce Desync in `initTransfer` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer` increments `currentOriginNonce` before the external token transfer (correct), but reads `currentOriginNonce` from storage again when emitting the `InitTransfer` event and calling `initTransferExtension` (after the external call). If a token with a sender hook (e.g., ERC-777) triggers a reentrant call to `initTransfer` during `safeTransferFrom`, the outer call's event is emitted with the wrong nonce — the same nonce already used by the inner call — permanently locking the outer call's tokens in the bridge with no recoverable event on NEAR.

### Finding Description

In `OmniBridge.initTransfer`, the nonce is incremented at the top of the function, but the `emit` and `initTransferExtension` call at the bottom read `currentOriginNonce` directly from storage rather than from a local snapshot: [1](#0-0) 

```solidity
currentOriginNonce += 1;   // incremented here (storage write)
```

Then, after the external token transfer: [2](#0-1) 

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [3](#0-2) 

```solidity
initTransferExtension(
    msg.sender, tokenAddress,
    currentOriginNonce,   // reads storage AFTER external call
    ...
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress,
    currentOriginNonce,   // reads storage AFTER external call
    ...
);
```

There is no `nonReentrant` guard on `initTransfer`. [4](#0-3) 

**Reentrancy trace with an ERC-777 token (plain ERC-20 path):**

| Step | `currentOriginNonce` | Action |
|------|----------------------|--------|
| Outer call enters | N → **N+1** | nonce incremented |
| `safeTransferFrom` fires ERC-777 `tokensToSend` hook | — | attacker re-enters `initTransfer` |
| Inner call enters | N+1 → **N+2** | nonce incremented again |
| Inner call completes token transfer | — | |
| Inner call emits `InitTransfer` with nonce **N+2** | — | valid event |
| Outer call's `safeTransferFrom` returns | — | |
| Outer call emits `InitTransfer` with `currentOriginNonce` = **N+2** | — | **wrong nonce — collision** |

Result: two `InitTransfer` events carry `originNonce = N+2`. Nonce N+1 is never emitted. The NEAR side will finalize the inner call's transfer (nonce N+2) and reject the outer call's duplicate (same nonce already finalized). The outer call's tokens are permanently locked in the bridge with no valid event for NEAR to process.

The same structural flaw exists in `initTransfer1155`: [5](#0-4) 

In `OmniBridgeWormhole`, `initTransferExtension` also publishes a Wormhole message using the stale `currentOriginNonce`, so the Wormhole message for the outer call would also carry the wrong nonce: [6](#0-5) 

### Impact Explanation

The outer call's tokens are irrecoverably locked in the bridge contract. The NEAR side has no valid `InitTransfer` event for nonce N+1 to process, and the duplicate nonce N+2 event is rejected by the `finalised_transfers` guard. There is no withdrawal or refund path. This matches **Critical — Permanent freezing / irrecoverable lock of user funds in the bridge**.

### Likelihood Explanation

ERC-777 tokens implement the ERC-20 interface and are accepted by `safeTransferFrom`. Real ERC-777 tokens exist on mainnet (e.g., imBTC, various DeFi tokens). Any such token registered in the bridge as a plain ERC-20 (not a bridge token, not a custom minter) is a live trigger. The attacker only needs to call `initTransfer` with such a token and have their `tokensToSend` hook re-enter `initTransfer`. No privileged access is required.

### Recommendation

Capture `currentOriginNonce` in a local variable immediately after incrementing it, before any external call, and use that local variable in all subsequent reads:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce;  // snapshot before external calls
    ...
    IERC20(tokenAddress).safeTransferFrom(...);
    ...
    initTransferExtension(msg.sender, tokenAddress, nonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, nonce, ...);
}
```

Apply the same fix to `initTransfer1155`. Additionally, consider adding a `nonReentrant` modifier to both functions as defense-in-depth.

### Proof of Concept

1. Deploy a malicious ERC-777 token `MalToken` that implements `tokensToSend`. In the hook, call `OmniBridge.initTransfer(MalToken, 50, 0, 0, recipient, "")`.
2. Register `MalToken` in the bridge as a plain ERC-20 (not a bridge token).
3. Attacker holds 100 `MalToken` and calls `initTransfer(MalToken, 50, 0, 0, recipient, "")`.
4. `currentOriginNonce` becomes N+1. `safeTransferFrom` fires the `tokensToSend` hook.
5. Hook re-enters `initTransfer` with 50 tokens. `currentOriginNonce` becomes N+2. Inner call completes and emits `InitTransfer` with nonce N+2.
6. Outer call resumes. Emits `InitTransfer` with `currentOriginNonce` = N+2 (collision).
7. NEAR processes nonce N+2 once (inner call). Outer call's 50 tokens are permanently locked. Nonce N+1 is never emitted.
