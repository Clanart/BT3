### Title
Unchecked ERC20 Fee Transfer Return Value in `charge_fee` Allows Silent Fee Bypass - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in the StarkNet OS executes an ERC20 `transfer()` call to collect transaction fees, but completely discards the return value of `non_reverting_select_execute_entry_point_func`. If the fee token transfer fails silently (reverts internally without propagating), the OS proceeds as if the fee was successfully collected. This is the direct Cairo/StarkNet analog of the "use safeTransferFrom" vulnerability class: a critical token transfer whose success is never verified.

---

### Finding Description

In `charge_fee` (`transaction_impls.cairo`, lines 111–165), the OS constructs an `ExecutionContext` targeting the fee token contract with the `transfer` selector and calls:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
``` [1](#0-0) 

The function `non_reverting_select_execute_entry_point_func` returns `(retdata_size, retdata, is_deprecated)`. In every other call site within the same file, these return values are captured and validated. For example, in `execute_deploy_account_transaction`:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(
    block_context=block_context, execution_context=validate_deploy_execution_context
);
...
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = VALIDATED;
}
``` [2](#0-1) 

And in `execute_declare_transaction`: [3](#0-2) 

In `charge_fee`, however, **all return values are silently dropped**. There is no assertion that:
1. The transfer did not revert (`is_reverted == 0`)
2. The ERC20 `transfer()` returned `true` (success indicator per the ERC20 standard)

The function is named "non_reverting" precisely because it catches inner reverts and surfaces them as return data rather than propagating them — meaning a failed transfer will not abort the outer OS execution. Since `charge_fee` ignores the return, a failed transfer is indistinguishable from a successful one at the OS level. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

If the ERC20 fee transfer fails silently (e.g., the user's fee token balance was drained by a prior transaction in the same block, or the fee token contract itself returns `false`), the OS generates a valid state-transition proof that commits the block without the fee having been transferred. The L1 verifier accepts this proof because the OS Cairo program itself does not enforce transfer success. The sequencer/protocol permanently loses the fee revenue for every such transaction, and the user's transaction is executed for free. This is a direct, provable loss of funds committed on-chain.

---

### Likelihood Explanation

An unprivileged transaction sender can trigger this by:

1. Submitting two transactions in the same block: (a) a transaction that drains their fee token balance to another address, and (b) a subsequent transaction that benefits from free execution.
2. The gateway performs pre-validation of balance at submission time, but the OS executes both transactions sequentially. After transaction (a) executes and drains the balance, transaction (b)'s `charge_fee` call will attempt the transfer, fail silently, and the OS will proceed.

The attacker controls the transaction content and ordering request. No privileged access, leaked keys, or external dependency compromise is required.

---

### Recommendation

Capture and validate the return value of `non_reverting_select_execute_entry_point_func` in `charge_fee`, consistent with how all other call sites handle it. Specifically:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);

// Verify the transfer succeeded.
if (is_deprecated == 0) {
    // ERC20 transfer must return (true,) i.e. a single felt equal to 1.
    assert retdata_size = 1;
    assert retdata[0] = 1;  // TRUE
}
```

This mirrors the pattern already used for `__validate__`, `__validate_declare__`, and `__validate_deploy__` entry points in the same file, and ensures that a failed fee transfer causes the OS to halt rather than silently proceed.

---

### Proof of Concept

1. User deploys account contract `A` with fee token balance of exactly `F` (the fee for one transaction).
2. User submits two transactions in the same block:
   - **Tx1**: Calls a contract that transfers all `F` tokens from `A` to address `B` (draining `A`'s balance).
   - **Tx2**: An invoke transaction from `A` with `max_fee = F`.
3. The sequencer includes both in a block. Tx1 executes first, draining `A`'s balance.
4. Tx2 executes. `charge_fee` is called. It constructs the ERC20 `transfer(sequencer, F)` call. The fee token's `transfer` reverts internally (insufficient balance). `non_reverting_select_execute_entry_point_func` returns with `is_reverted=1` but `charge_fee` discards this.
5. The OS continues, finalizes the block, and generates a valid proof.
6. The L1 verifier accepts the proof. Tx2's effects are committed on-chain. The sequencer received zero fees for Tx2. [1](#0-0)

// Charges a fee from the user.
// If max_fee is not 0, validates that the selector matches the entry point of an account contract
// and executes an ERC20 transfer on the behalf of that account contract.
//
// Arguments:
// block_context - a global context that is fixed throughout the block.
// tx_execution_context - The execution context of the transaction that pays the fee.
func charge_fee{
    range_check_ptr,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,
    outputs: OsCarriedOutputs*,
}(block_context: BlockContext*, tx_execution_context: ExecutionContext*) {
    alloc_locals;

    local tx_info: TxInfo* = tx_execution_context.execution_info.tx_info;
    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    if (max_fee == 0) {
        return ();
    }

    local low_actual_fee;
    %{ LoadActualFee %}
    local calldata: TransferCallData = TransferCallData(
        recipient=block_context.block_info_for_execute.sequencer_address,
        amount=Uint256(low=low_actual_fee, high=0),
    );

    // Verify that the charged amount is not larger than the transaction's max_fee field.
    assert_nn_le(calldata.amount.low, max_fee);

    // TODO(ilya, 01/01/2026): Consider caching the fee_token_class_hash.
    local fee_token_address = block_context.os_global_context.starknet_os_config.fee_token_address;
    let (fee_state_entry: StateEntry*) = dict_read{dict_ptr=contract_state_changes}(
        key=fee_token_address
    );
    let (__fp__, _) = get_fp_and_pc();
    // Use block_info directly from block_context, so that charge_fee will always run in
    // execute-mode rather than validate-mode.
    local execution_context: ExecutionContext = ExecutionContext(
        entry_point_type=ENTRY_POINT_TYPE_EXTERNAL,
        class_hash=fee_state_entry.class_hash,
        calldata_size=TransferCallData.SIZE,
        calldata=&calldata,
        execution_info=new ExecutionInfo(
            block_info=block_context.block_info_for_execute,
            tx_info=tx_info,
            caller_address=tx_info.account_contract_address,
            contract_address=fee_token_address,
            selector=TRANSFER_ENTRY_POINT_SELECTOR,
        ),
        deprecated_tx_info=tx_execution_context.deprecated_tx_info,
    );

    let remaining_gas = DEFAULT_INITIAL_GAS_COST;
    non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
        block_context=block_context, execution_context=&execution_context
    );
    return ();
}
