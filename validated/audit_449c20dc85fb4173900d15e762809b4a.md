Audit Report

## Title
Stranded ETH in payable router functions is silently consumed by any subsequent WETH-input swap caller via `pay()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` to fund WETH payments before pulling from the actual payer. Because all swap entry points on `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are `payable` with no guard rejecting `msg.value` when `tokenIn != WETH`, ETH sent with a non-WETH swap is silently accepted and stranded. Any subsequent caller whose `tokenIn` is WETH will have their payment subsidized by that stranded ETH, permanently losing the original depositor's funds.

## Finding Description
`PeripheryPayments.pay()` contains the following WETH branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
``` [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` and, if non-zero, wraps that native ETH and transfers it to the pool before (or instead of) pulling WETH from the actual payer.

The `receive()` guard only blocks **direct** ETH transfers (bare calls with no calldata): [2](#0-1) 

It is **not** invoked when ETH is sent alongside a `payable` function call. All four swap entry points are `payable` with no `msg.value == 0` enforcement when `tokenIn != WETH`: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` and `addLiquidityWeighted` are also `payable` and call the same `pay()` in their callback: [7](#0-6) [8](#0-7) 

Exploit path:
1. Alice calls `exactInputSingle{value: 1 ether}(params)` where `params.tokenIn = USDC`. The swap callback calls `pay(USDC, alice, pool, amount)`. Since `token != WETH`, the ETH branch is skipped; USDC is pulled from Alice. 1 ETH remains stranded (`address(router).balance == 1 ether`).
2. Bob calls `exactInputSingle{value: 0}(params)` where `params.tokenIn = WETH`, `params.amountIn = 1 ether`. The swap callback calls `pay(WETH, bob, pool, 1 ether)`. `nativeBalance = 1 ether >= 1 ether` → contract wraps Alice's ETH and transfers WETH to pool. Bob receives the swap output without spending any WETH or ETH. Alice's 1 ETH is permanently lost.

## Impact Explanation
Direct loss of user principal: Alice's ETH is permanently transferred to the pool on Bob's behalf. Bob receives full swap output at zero cost. The impact scales with the stranded ETH amount. Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` share the same `pay()` implementation, covering swaps and liquidity additions. This constitutes a Critical/High direct loss of user principal above Sherlock thresholds, matching the "direct loss of user principal" allowed impact.

## Likelihood Explanation
No special privilege is required. The pattern of sending ETH with a non-WETH swap, or sending excess ETH and omitting `refundETH()` from a multicall, is a well-documented user error in Uniswap-style routers. The `multicall` pattern actively encourages batching ETH-value calls, making it easy to omit the refund step. Any unprivileged address can trigger the exploit the moment stranded ETH exists in the router. The attack is repeatable and requires no front-running or special timing.

## Recommendation
1. **Reject `msg.value` when `tokenIn != WETH`** in each swap entry point, or add a modifier enforcing `msg.value == 0` unless the first token in the path is WETH.
2. **Do not use `address(this).balance` as a payment source in `pay()`** unless the ETH was explicitly sent in the same transaction. Gate the native-balance branch on a `msgValue` parameter passed down through the call stack rather than a raw balance read.
3. Alternatively, track `msg.value` at the entry point and pass it explicitly through the call stack, using that tracked value instead of `address(this).balance` in `pay()`.
