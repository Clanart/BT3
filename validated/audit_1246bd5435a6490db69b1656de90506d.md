### Title
Deposited NEAR Tokens Permanently Locked in Bridge Contract on Failed Token Deployment - (File: `near/omni-bridge/src/lib.rs`)

### Summary
In `deploy_token_by_deployer_callback`, when the external deployer call fails, the `attached_deposit` funds returned from the deployer are silently absorbed into the bridge contract with no refund path to the original caller. This is a direct analog to the M-05 pattern: funds sent for an operation that partially fails get permanently stuck in the intermediary contract.

### Finding Description
The `deploy_token` function is callable by any unprivileged user and requires a substantial `attached_deposit` to cover the cost of creating a new on-chain token account and deploying the token contract.

The call chain is:

1. User calls `deploy_token` with `attached_deposit` (several NEAR tokens).
2. `deploy_token_internal` retains `required_deposit = storage_byte_cost × storage_usage + NEP141_DEPOSIT` in the bridge contract, then calls the external deployer with `attached_deposit − required_deposit`. [1](#0-0) 

3. `deploy_token_by_deployer_callback` is invoked as the callback. [2](#0-1) 

When the deployer call **fails** (e.g., insufficient deposit to create the sub-account, invalid global code hash, or the token contract's `new` initializer panics), the NEAR runtime automatically returns `attached_deposit − required_deposit` to the bridge contract. The callback then only cleans up state and returns `PromiseOrValue::Value(())` — **there is no refund to the original caller**. [3](#0-2) 

A secondary loss occurs even on a **successful** deployer call: `deploy_token_by_deployer_callback` fires `storage_deposit` with `NEP141_DEPOSIT` attached but with **no callback**. If `storage_deposit` fails (e.g., the 3 TGas `STORAGE_DEPOSIT_GAS` budget is exhausted, or the token contract panics), `NEP141_DEPOSIT` is returned to the bridge contract with no mechanism to forward it back to the caller. [4](#0-3) 

The token deployer contract itself simply creates the account and transfers the attached deposit to it, so any failure there returns the full amount to the bridge contract. [5](#0-4) 

### Impact Explanation
**Critical — Permanent irrecoverable lock of user funds in the bridge contract.**

A user who calls `deploy_token` with a deposit of, say, 5 NEAR (typical for deploying a token sub-account) loses the entire `attached_deposit − required_deposit` if the deployer call fails. There is no admin withdrawal function visible in the contract, so the funds are permanently stuck. The `NEP141_DEPOSIT` (~1.25 mNEAR) is additionally lost on `storage_deposit` failure. Neither loss is recoverable by the user.

### Likelihood Explanation
**Moderate.** The deployer call can fail for several realistic reasons:
- The user sends slightly less than the actual cost to create the sub-account (the required amount depends on runtime storage pricing and can vary).
- The `global_code_hash` stored in the deployer is updated between the user's deposit calculation and the actual call.
- The token contract's `new` initializer panics (e.g., due to a metadata validation error).
- Gas exhaustion in the deployer's `DEPLOY_TOKEN_GAS` (50 TGas) budget.

`deploy_token` is open to any caller (no role restriction, only pausable by DAO). [6](#0-5) 

### Recommendation
Pass the original caller's `AccountId` as a parameter to `deploy_token_by_deployer_callback` and issue a refund in the failure branch:

```rust
pub fn deploy_token_by_deployer_callback(
    &mut self,
    token_address: &OmniAddress,
    token_id: AccountId,
    original_caller: AccountId,          // <-- add this
    refund_amount: NearToken,             // <-- add this (= attached_deposit - required_deposit)
) -> PromiseOrValue<()> {
    if env::promise_result_checked(0, usize::MAX).is_ok() {
        // ... existing storage_deposit logic ...
    } else {
        // clean up state
        self.deployed_tokens.remove(&token_id);
        // ...
        // Refund the caller
        Promise::new(original_caller).transfer(refund_amount);
        PromiseOrValue::Value(())
    }
}
```

For the `storage_deposit` success path, add a callback that refunds `NEP141_DEPOSIT` to the original caller if `storage_deposit` fails.

### Proof of Concept
1. Alice calls `deploy_token` with `attached_deposit = 5 NEAR` for a token whose deployer sub-account creation costs 4.9 NEAR.
2. `deploy_token_internal` retains `required_deposit ≈ 0.00125 NEAR` and sends `≈ 4.99875 NEAR` to the deployer.
3. The deployer call fails (e.g., the global code hash was just rotated by DAO).
4. `≈ 4.99875 NEAR` is returned to the bridge contract.
5. `deploy_token_by_deployer_callback` cleans up state and returns `()`.
6. Alice's `≈ 4.99875 NEAR` is permanently locked in the bridge contract with no recovery path. [7](#0-6) [2](#0-1)

    #[private]
    pub fn deploy_token_by_deployer_callback(
        &mut self,
        token_address: &OmniAddress,
        token_id: AccountId,
    ) -> PromiseOrValue<()> {
        if env::promise_result_checked(0, usize::MAX).is_ok() {
            ext_token::ext(token_id)
                .with_static_gas(STORAGE_DEPOSIT_GAS)
                .with_attached_deposit(NEP141_DEPOSIT)
                .storage_deposit(&env::current_account_id(), Some(true))
                .into()
        } else {
            self.deployed_tokens.remove(&token_id);
            self.deployed_tokens_v2.remove(&token_id);
            self.token_id_to_address
                .remove(&(token_address.get_chain(), token_id));
            self.token_address_to_id.remove(token_address);
            self.token_decimals.remove(token_address);
            PromiseOrValue::Value(())
        }
    }
