### Title
Unverified Hint-Sourced Block Hash Written to On-Chain State Enables Malicious Prover to Inject Arbitrary Block Hashes — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/os_utils.cairo`)

---

### Summary

The Cairo OS program loads the historical block hash (`old_block_hash`) from a single, unvalidated hint source and writes it directly to the on-chain block hash mapping contract (`BLOCK_HASH_CONTRACT_ADDRESS = 0x1`) with no Cairo-level assertion verifying its correctness. Simultaneously, the inputs used to compute the new block hash (`previous_block_hash`, `header_commitments`, `gas_prices_hash`, `starknet_version`) are also loaded from a single unvalidated hint source. A malicious prover can supply arbitrary values for all of these, injecting fabricated block hashes into the live chain state that every contract can read via `get_block_hash`.

---

### Finding Description

**Root Cause 1 — `write_block_number_to_block_hash_mapping` in `os_utils.cairo`:**

```cairo
// Currently, the block hash mapping is not enforced by the OS.
// TODO(Yoni, 1/1/2026): output this hash.
local old_block_hash;
%{ GetOldBlockNumberAndHash %}

// Update mapping.
assert state_entry.class_hash = 0;
assert state_entry.nonce = 0;
tempvar storage_ptr = state_entry.storage_ptr;
assert [storage_ptr] = DictAccess(key=old_block_number, prev_value=0, new_value=old_block_hash);
``` [1](#0-0) 

`old_block_hash` is loaded exclusively from the hint `%{ GetOldBlockNumberAndHash %}`. There is no Cairo `assert` that binds `old_block_hash` to any cryptographic commitment of block `old_block_number`. The value is then written verbatim into the state of `BLOCK_HASH_CONTRACT_ADDRESS` (`0x1`), which is the system contract that all user contracts query via `get_block_hash`. [2](#0-1) 

**Root Cause 2 — `get_block_hashes` in `block_hash.cairo`:**

```cairo
local previous_block_hash;
// Currently, the header commitments and gas prices are not computed by the OS.
// TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
local header_commitments: BlockHeaderCommitments*;
local gas_prices_hash;
local starknet_version;

%{ GetBlockHashes %}

let block_hash = calculate_block_hash(...);

%{ CheckBlockHashConsistency %}
``` [3](#0-2) 

All four inputs to `calculate_block_hash` — `previous_block_hash`, `header_commitments`, `gas_prices_hash`, and `starknet_version` — are loaded from a single hint call `%{ GetBlockHashes %}`. The subsequent `%{ CheckBlockHashConsistency %}` is Python hint code, not a Cairo `assert`; it is invisible to the proof verifier and can be omitted or overridden by a malicious prover. The `OsOutputHeader` comment confirms the block hash is not enforced:

```cairo
// Currently, the block hash is not enforced by the OS.
new_block_hash: felt,
``` [4](#0-3) 

The `get_block_os_output_header` function in `os_utils.cairo` also explicitly documents that the previous block hash and previous state root are guessed and not verified by the OS:

```cairo
// NOTE: both the previous block hash and previous state root are guessed, and the OS
// does not verify their consistency (unlike the new hash and root).
// The consumer of the OS output should verify both.
``` [5](#0-4) 

**Analog to External Report:**
The external report describes an off-chain worker that relies on a single, unvalidated Ethereum API as its sole source of truth for events, with no cross-validation. The analog here is that the Cairo OS program relies on a single, unvalidated hint source as its sole source of truth for block hash data, with no Cairo-level cross-validation. In both cases, the single data source can be manipulated to alter chain state.

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

The block hash mapping contract at address `0x1` is a system-level contract whose storage is populated exclusively by `write_block_number_to_block_hash_mapping`. Every block, the OS writes the hash of block `N - STORED_BLOCK_HASH_BUFFER` (i.e., `N - 10`) into this contract's storage under key `N - 10`. [6](#0-5) 

User contracts call `get_block_hash(block_number)` to retrieve these values. Because the Cairo OS program places no assertion on `old_block_hash`, a malicious prover can write any felt value for any historical block number into this mapping. Contracts that use `get_block_hash` as a source of unpredictability (e.g., lottery/randomness contracts), for replay protection, or for cross-contract authentication will receive attacker-controlled values. This enables the attacker to predict or force specific "random" outcomes, bypass replay guards, or forge authentication proofs — all of which can directly drain funds from those contracts.

---

### Likelihood Explanation

The ZK proof system's core security guarantee is that the prover is trustless: the Cairo program enforces all invariants, so even a malicious prover cannot produce a valid proof for an invalid state transition. The missing Cairo assertions break this guarantee. Any entity that operates a prover (currently StarkWare, but the protocol is designed for decentralization) can exploit this without any key compromise, phishing, or network-level attack — they simply supply a crafted hint value. The exploit requires no special user interaction and is triggered during normal block production.

---

### Recommendation

1. **Short term:** Add a Cairo `assert` in `write_block_number_to_block_hash_mapping` that verifies `old_block_hash` against a cryptographic commitment anchored in the proof (e.g., the state root of the previous block or a Merkle path). Remove the reliance on the single hint `%{ GetOldBlockNumberAndHash %}` as the sole source of truth.

2. **Short term:** In `get_block_hashes`, replace the hint-only `%{ CheckBlockHashConsistency %}` with a Cairo-level assertion that binds `previous_block_hash` to the previously proven block hash output. Compute `header_commitments` and `gas_prices_hash` inside Cairo rather than guessing them from hints (the existing TODOs at lines 61 and 65 of `block_hash.cairo` already acknowledge this).

3. **Long term:** Eliminate the pattern of loading security-critical values from a single hint source without Cairo-level cross-validation. Every value that influences the chain state must be either computed deterministically in Cairo or verified against a cryptographic commitment that is itself enforced by the proof.

---

### Proof of Concept

1. Attacker operates a prover for block `N`.
2. During execution of `write_block_number_to_block_hash_mapping`, the prover supplies hint `%{ GetOldBlockNumberAndHash %}` with `old_block_hash = ATTACKER_CHOSEN_VALUE` for block number `N - 10`.
3. The Cairo code at line 73 of `os_utils.cairo` writes `DictAccess(key=N-10, prev_value=0, new_value=ATTACKER_CHOSEN_VALUE)` into the state of `BLOCK_HASH_CONTRACT_ADDRESS`.
4. No Cairo assertion checks this value; the proof verifier accepts the proof as valid.
5. The fabricated block hash is committed to the on-chain state.
6. A target contract calls `get_block_hash(N - 10)` and receives `ATTACKER_CHOSEN_VALUE`.
7. If the contract uses this value for randomness or authentication, the attacker exploits the predictable/controlled output to drain funds. [7](#0-6) [3](#0-2)

    alloc_locals;
    tempvar old_block_number = block_context.block_info_for_execute.block_number -
        STORED_BLOCK_HASH_BUFFER;
    let is_old_block_number_non_negative = is_nn(old_block_number);
    if (is_old_block_number_non_negative == FALSE) {
        // Not enough blocks in the system - nothing to write.
        return ();
    }

    // Fetch the (block number -> block hash) mapping contract state.
    local state_entry: StateEntry*;
    %{ GetBlockHashMapping %}

    // Currently, the block hash mapping is not enforced by the OS.
    // TODO(Yoni, 1/1/2026): output this hash.
    local old_block_hash;
    %{ GetOldBlockNumberAndHash %}

    // Update mapping.
    assert state_entry.class_hash = 0;
    assert state_entry.nonce = 0;
    tempvar storage_ptr = state_entry.storage_ptr;

    let storage_ptr = storage_ptr + DictAccess.SIZE;

        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
