### Title
Unchecked Felt Arithmetic Overflow in `compute_max_possible_fee` Enables Complete Fee Bypass — (`execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` in `transaction_impls.cairo` performs unchecked felt-field arithmetic on user-controlled `ResourceBounds` values. Because Cairo felt arithmetic is implicitly modulo the field prime P ≈ 2²⁵¹, a crafted transaction can make the function return exactly `0`. The caller `charge_fee` treats a zero result as "no fee required" and returns immediately without executing the ERC-20 transfer, so the sequencer receives nothing for processing the transaction.

---

### Finding Description

`compute_max_possible_fee` computes the maximum chargeable fee as a plain felt sum of products:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

No `assert_nn_le`, `assert_le`, or range-check builtin is applied to any of the six user-supplied fields (`max_amount`, `max_price_per_unit` for each of the three resource types) or to `tip`. All six values are loaded from hints without bounds validation: [2](#0-1) 

The result is consumed by `charge_fee`, which short-circuits on zero:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

When `max_fee == 0` the ERC-20 `transfer` call that pays the sequencer is never executed: [4](#0-3) 

`charge_fee` is called for every account transaction type — invoke, declare, and deploy-account: [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Direct loss of funds (Critical).** The sequencer's fee revenue is the economic mechanism that compensates block producers and sustains the network. When `max_fee` wraps to zero, the OS proof is still valid — the prover can produce a correct proof for a block that contains fee-free transactions. The on-chain verifier accepts the proof, the state transition is finalised, and the sequencer address receives no ERC-20 tokens for those transactions. At scale, an attacker can drain all fee revenue from every block.

---

### Likelihood Explanation

Any V3 transaction sender can craft the six `ResourceBounds` fields freely; they are part of the signed transaction body and are loaded from hints with no OS-level range enforcement. A single arithmetic identity (shown in the PoC below) is sufficient. The attack requires no privileged role, no key leak, and no external dependency — only the ability to submit a transaction.

---

### Recommendation

Apply `assert_nn_le` (or equivalent range-check-backed bounds) to each `max_amount` and `max_price_per_unit` field immediately after they are loaded, before any arithmetic is performed. Concretely, enforce that each value fits within a safe integer range (e.g., `[0, 2^64)`) so that no product or sum can overflow the felt field. Additionally, assert that the final `max_fee` result is non-negative under standard integer semantics before using it as a guard.

---

### Proof of Concept

Let P = 3618502788666131213697322783095070105623107215331596699973092056135872020481 (the Cairo field prime).

Set the following transaction fields (all are user-controlled in a V3 transaction):

```
l1_gas_bounds.max_amount          = 1
l1_gas_bounds.max_price_per_unit  = P - 1   // i.e. felt(-1)
l2_gas_bounds.max_amount          = 1
l2_gas_bounds.max_price_per_unit  = 0
tip                               = 1
l1_data_gas_bounds.max_amount     = 0
l1_data_gas_bounds.max_price_per_unit = 0
```

`compute_max_possible_fee` evaluates:

```
= 1*(P-1) + 1*(0+1) + 0*0
= (P-1) + 1
= P
≡ 0  (mod P)
```

`max_fee` is `0`. `charge_fee` returns at line 123 without executing the ERC-20 transfer. The transaction is fully executed and its state changes are committed, but the sequencer receives zero fee tokens. The OS proof for this block is valid and will be accepted by the on-chain verifier. [8](#0-7)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
