### Title
Wrapping Integer Overflow in `check_time_locks` Bypasses Relative Timelock Conditions and Permanently Locks Coins — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `nowrap=false`, `check_time_locks` uses `wrapping_add` for relative timelock arithmetic on both `ASSERT_HEIGHT_RELATIVE`/`ASSERT_SECONDS_RELATIVE` and `ASSERT_BEFORE_HEIGHT_RELATIVE`/`ASSERT_BEFORE_SECONDS_RELATIVE` conditions. Integer overflow in `wrapping_add` produces two distinct consensus failures from unprivileged spend input:

1. **Timelock bypass**: A coin with `ASSERT_HEIGHT_RELATIVE(u32::MAX)` or `ASSERT_SECONDS_RELATIVE(u64::MAX)` becomes immediately spendable instead of being locked for billions of blocks/seconds.
2. **Permanent coin lock** (direct analog to the external report): A coin with `ASSERT_BEFORE_HEIGHT_RELATIVE(u32::MAX)` or `ASSERT_BEFORE_SECONDS_RELATIVE(u64::MAX)` becomes permanently unspendable — the deadline wraps to a value already in the past, so the check always fails and the coin can never be spent.

---

### Finding Description

In `check_time_locks`, the `nowrap` parameter selects between `saturating_add` (correct) and `wrapping_add` (legacy/buggy) for all four relative timelock checks:

**`ASSERT_HEIGHT_RELATIVE` — timelock bypass:**
```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```
With `confirmed_block_index = 100` and `height_relative = 0xFFFF_FFFF`:
- `wrapping_add(100, 0xFFFF_FFFF) = 99`
- Check: `prev_height < 99` → **false at height 99+**, so the spend is **allowed immediately**
- Intended: coin should be locked until block ~4,294,967,395 (effectively forever)

**`ASSERT_BEFORE_HEIGHT_RELATIVE` — permanent coin lock:**
```rust
} else if prev_transaction_block_height
    >= unspent.confirmed_block_index.wrapping_add(before_height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertBeforeHeightRelativeFailed));
}
```
With `confirmed_block_index = 100` and `before_height_relative = 0xFFFF_FFFF`:
- `wrapping_add(100, 0xFFFF_FFFF) = 99`
- Check: `prev_height >= 99` → **always true at height 99+**, so the spend **always fails**
- Intended: coin should be spendable until block ~4,294,967,395 (effectively always)

The same wrapping logic applies to `ASSERT_SECONDS_RELATIVE` (u64) and `ASSERT_BEFORE_SECONDS_RELATIVE` (u64).

The test suite explicitly documents and confirms both divergent behaviors: [1](#0-0) 

The `nowrap=false` path is preserved as a Python-callable binding: [2](#0-1) 

And is exposed in the Python API stub: [3](#0-2) 

The `nowrap` parameter is caller-controlled from Python; the Python full node decides which value to pass based on block height. The `get_flags_for_height_and_constants` function does not include a `nowrap` flag, meaning the Python layer independently controls this parameter: [4](#0-3) 

---

### Impact Explanation

**Timelock bypass (High):** A spend bundle containing a coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFFF)` or `ASSERT_SECONDS_RELATIVE(0xFFFF_FFFF_FFFF_FFFF)` is accepted by `check_time_locks` with `nowrap=false` at any block height ≥ 99 (or any timestamp ≥ 999), even though the intended timelock is ~4 billion blocks/seconds in the future. This is a timelock condition validation bypass enabling unauthorized spend acceptance — fitting the High impact category.

**Permanent coin lock (analog to external report):** A coin whose puzzle emits `ASSERT_BEFORE_HEIGHT_RELATIVE(0xFFFF_FFFF)` or `ASSERT_BEFORE_SECONDS_RELATIVE(0xFFFF_FFFF_FFFF_FFFF)` becomes permanently unspendable when `nowrap=false` is used. The wrapped deadline is already in the past, so the coin can never be spent again — directly analogous to the Opyn report where user funds were permanently locked because a required admin action was never taken. Here, the "admin action" is the hard-fork activation of `nowrap=true`; coins created before that activation with large `ASSERT_BEFORE_*_RELATIVE` values are permanently frozen.

---

### Likelihood Explanation

- The `nowrap=false` path is reachable via the Python binding `check_time_locks(..., nowrap=False)`, which the Python full node calls for pre-hard-fork block validation.
- The maximum-value relative timelock (`0xFFFF_FFFF` for height, `0xFFFF_FFFF_FFFF_FFFF` for seconds) is a valid CLVM integer representable as a 5-byte atom (`0x00FFFFFFFF` / `0x00FFFFFFFFFFFFFFFF`), accepted by `sanitize_uint` without error.
- Any unprivileged user can craft a spend bundle with a puzzle that emits these conditions; no privileged role or key is required.
- The Python test suite (`tests/test_check_time_locks.py`) confirms both the bypass and the permanent-lock outcomes are reachable with `nowrap=False`. [5](#0-4) 

---

### Recommendation

1. **Remove the `nowrap=false` / `wrapping_add` path entirely.** The `saturating_add` behavior is correct for all relative timelock arithmetic. The legacy wrapping path should not be reachable for any block height in production.
2. If backward compatibility with pre-hard-fork blocks is required, document precisely which block heights use `nowrap=false` and ensure no new spend bundles can be validated with that path.
3. Add a consensus-level check: if `before_height_relative` or `before_seconds_relative` would wrap to a value less than `confirmed_block_index` / `timestamp`, treat it as `u32::MAX` / `u64::MAX` (saturate) rather than wrapping.

---

### Proof of Concept

The test suite already encodes the exact exploit scenario:

```
// height_relative = u32::MAX, confirmed_block_index = 100, prev_height = 200
// nowrap=true  → saturating: 100 + u32::MAX = u32::MAX → 200 < u32::MAX → Err (correct)
// nowrap=false → wrapping:   100 + u32::MAX = 99       → 200 < 99      → Ok  (BYPASS)

// before_height_relative = u32::MAX, confirmed_block_index = 100, prev_height = 200
// nowrap=true  → saturating: 100 + u32::MAX = u32::MAX → 200 >= u32::MAX → Ok  (correct)
// nowrap=false → wrapping:   100 + u32::MAX = 99       → 200 >= 99       → Err (PERMANENT LOCK)
``` [6](#0-5) 

A concrete attacker-controlled spend bundle:
1. Craft a puzzle that emits `(ASSERT_HEIGHT_RELATIVE . 0x00FFFFFFFF)` (5-byte atom encoding of `u32::MAX`).
2. Submit the spend bundle to a node calling `check_time_locks(..., nowrap=False)`.
3. The coin is accepted as spendable at block height ≥ 99, bypassing the intended ~4-billion-block timelock.

For the permanent lock variant, replace with `ASSERT_BEFORE_HEIGHT_RELATIVE . 0x00FFFFFFFF`; the coin becomes unspendable at all future block heights when `nowrap=False` is active.

pub fn get_flags_for_height_and_constants(
    prev_tx_height: u32,
    constants: &ConsensusConstants,
) -> ConsensusFlags {
    //  the hard-fork initiated with 2.0. To activate June 2024
    //  * costs are ascribed to some unknown condition codes, to allow for
    // soft-forking in new conditions with cost
    //  * a new condition, SOFTFORK, is added which takes a first parameter to
    //    specify its cost. This allows soft-forks similar to the softfork
    //    operator
    //  * BLS operators introduced in the soft-fork (behind the softfork
    //    guard) are made available outside of the guard.
    //  * division with negative numbers are allowed, and round toward
    //    negative infinity
    //  * AGG_SIG_* conditions are allowed to have unknown additional
    //    arguments
    //  * Allow the block generator to be serialized with the improved clvm
    //   serialization format (with back-references)

    // The soft fork initiated with 2.5.0. The activation date is still TBD.
    // Adds a new keccak256 operator under the softfork guard with extension 1.
    // This operator can be hard forked in later, but is not included in a hard fork yet.

    // In hard fork 2, we enable the keccak operator outside the softfork guard
    let mut flags = ConsensusFlags::empty();
    if prev_tx_height >= constants.hard_fork2_height {
        flags |= ConsensusFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD
            | ConsensusFlags::COST_CONDITIONS
            | ConsensusFlags::ENABLE_SECP_OPS
            | ConsensusFlags::RELAXED_BLS;
    }

    if prev_tx_height >= constants.soft_fork8_height {
        flags |= ConsensusFlags::DISABLE_OP;
    }

    if prev_tx_height >= constants.soft_fork9_height {
        flags |= ConsensusFlags::SIMPLE_GENERATOR
            | ConsensusFlags::CANONICAL_INTS
            | ConsensusFlags::LIMIT_SPENDS;
    }
    flags
}
