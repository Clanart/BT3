### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` performs unchecked felt-field arithmetic over user-supplied `max_price_per_unit` values that have no upper bound. Because Cairo arithmetic is modular (mod the Stark prime P), a crafted transaction can make the function return exactly `0`, causing `charge_fee` to silently skip the ERC-20 transfer and execute the transaction for free. The resulting OS execution trace is fully valid and produces an accepted STARK proof.

---

### Finding Description

**Root cause — `pack_resource_bounds` imposes no upper bound on `max_price_per_unit`:** [1](#0-0) 

`assert_nn_le` bounds `max_amount` to `[0, 2^64-1]`, but `assert_nn` only constrains `max_price_per_unit` to `[0, (P-1)/2]` where P ≈ 2^251. No tighter ceiling is enforced.

**Vulnerable computation — `compute_max_possible_fee` sums products in felt arithmetic:** [2](#0-1) 

The return value is computed as:

```
l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
+ l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tip)
+ l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit
```

All arithmetic is modular. With `max_price_per_unit` up to `(P-1)/2` and `max_amount` up to `2^64-1`, individual products can reach `≈ 2^314`, wrapping around P many times. The sum of multiple such terms can be made to equal exactly `0 mod P`.

**Fee bypass — `charge_fee` exits immediately when `max_fee == 0`:** [3](#0-2) 

When `compute_max_possible_fee` returns `0`, `charge_fee` returns before executing the ERC-20 transfer, so no tokens are moved from the user to the sequencer.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any user can craft a V3 transaction whose resource-bound values cause `compute_max_possible_fee` to return `0` in felt arithmetic. The OS then produces a fully valid STARK proof in which the transaction was executed but zero fee was charged. The sequencer (and by extension the protocol) receives no fee revenue for that execution. Because the proof is valid, L1 accepts it. The user obtains arbitrary on-chain execution at zero cost.

---

### Likelihood Explanation

The attack requires only knowledge of the Stark prime P and basic modular arithmetic — no privileged access, no key material, no network-level capability. Any transaction sender can construct the required resource-bound values, sign the transaction (the hash commits to those values), and submit it. The sequencer's off-chain fee estimator may not replicate the felt-modular computation and could include the transaction believing it carries a non-zero fee.

---

### Recommendation

1. **Bound `max_price_per_unit` tightly.** Add `assert_nn_le(resource_bounds.max_price_per_unit, MAX_PRICE_PER_UNIT)` in `pack_resource_bounds`, where `MAX_PRICE_PER_UNIT` is chosen so that `(2^64 - 1) * MAX_PRICE_PER_UNIT * 3 < P` (ensuring the entire sum in `compute_max_possible_fee` cannot overflow).

2. **Assert `max_fee > 0` before the early-return guard.** After computing `max_fee`, add an explicit `assert_nn(max_fee - 1)` (or equivalent) to guarantee the result is a genuine positive integer, not a wrapped-around zero.

---

### Proof of Concept

Let P = 2^251 + 17·2^192 + 1 (Stark prime).

Craft a V3 transaction with:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `2` |
| `l1_gas_bounds.max_price_per_unit` | `(P − 1) / 2` |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas_bounds.max_amount` | `0` |
| `l1_data_gas_bounds.max_price_per_unit` | `0` |

**All Cairo range checks pass:**
- `assert_nn_le(2, 2^64 − 1)` ✓
- `assert_nn((P−1)/2)` ✓ — this is exactly the maximum non-negative felt value
- `assert_nn_le(1, 2^64 − 1)` ✓
- `assert_nn(1)` ✓
- `assert_nn_le(tip=0, 2^64 − 1)` ✓

**`compute_max_possible_fee` evaluates to:**

```
2 · (P−1)/2  +  1 · (1 + 0)  +  0 · 0
= (P − 1)    +  1
= P
≡ 0  (mod P)
```

`max_fee = 0`. `charge_fee` hits the early-return branch at line 123 and returns without executing the ERC-20 transfer. The transaction is fully executed with zero fee paid. The OS trace is constraint-satisfying and produces a valid STARK proof accepted by the L1 verifier.

```text
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
