Audit Report

## Title
`msg.value` Reuse via `delegatecall` in `OracleBase.multicall` Allows Bypassing Per-Registration Fee ŌĆö (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

## Summary
`OracleBase` inherits OpenZeppelin's `Multicall`, which dispatches each batched call via `delegatecall`. Because `delegatecall` preserves the original `msg.value` across every sub-call, an attacker can batch N `register()` calls while paying only one `registrationFee`, successfully registering N pools and bypassing the spam-deterrent fee for all but the first registration.

## Finding Description
`OracleBase` is declared as:

```solidity
contract OracleBase is AccessControl, Multicall, IOffchainOracle {
``` [1](#0-0) 

The import is OpenZeppelin's `Multicall`, whose `multicall` function iterates over calldata elements and dispatches each via `delegatecall`. `delegatecall` does not consume or split `msg.value` ŌĆö it forwards the full original `msg.value` to every sub-call. [2](#0-1) 

The `register` function is `payable` and enforces a per-call fee check:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
``` [3](#0-2) 

When `multicall([register(feedId1,pool1,factory), register(feedId2,pool2,factory)])` is called with `msg.value == registrationFee`, both `delegatecall` sub-calls read the same original `msg.value`. Both pass the `require`, both set `registeredPool[feedId][pool] = true`, and only one fee is collected by the contract. [4](#0-3) 

The `IMulticall` interface in this repository explicitly documents the `delegatecall` dispatch:

```solidity
/// @notice Executes each calldata element on this contract via delegatecall.
function multicall(bytes[] calldata data) external payable returns (bytes[] memory results);
``` [5](#0-4) 

## Impact Explanation
The `registrationFee` is the sole spam-deterrent for the `registeredPool` mapping, which gates on-chain oracle price reads via `price(feedId, pool)`. An attacker paying one fee can register an unbounded number of pools, causing direct loss of protocol fee revenue equal to `(N-1) * registrationFee` per N-registration batch, and fully defeating the spam-deterrent invariant. The fee is tunable by ADMIN and explicitly described as a spam deterrent. [6](#0-5) [7](#0-6) 

## Likelihood Explanation
Any unprivileged caller with two valid `(feedId, pool, factory)` tuples ŌĆö where `factory` is approved and `pool` is recognized by that factory ŌĆö can exploit this. No special role is required. The precondition (having legitimate pools) is the normal operating state of the system. The exploit is repeatable at will.

## Recommendation
Remove `payable` from `register` and collect fees via ERC-20 `transferFrom`, or track cumulative fees consumed within the `multicall` context using a transient storage counter decremented per sub-call. Alternatively, do not inherit `Multicall` on contracts with `payable` functions that check `msg.value`.

## Proof of Concept
```solidity
function test_multicall_fee_bypass() public {
    uint256 fee = oracle.registrationFee(); // e.g. 1 ether after ADMIN tunes it

    bytes[] memory calls = new bytes[](2);
    calls[0] = abi.encodeCall(oracle.register, (feedId1, pool1, factory));
    calls[1] = abi.encodeCall(oracle.register, (feedId2, pool2, factory));

    oracle.multicall{value: fee}(calls); // only one fee paid

    assertTrue(oracle.registeredPool(feedId1, pool1)); // registered
    assertTrue(oracle.registeredPool(feedId2, pool2)); // also registered ŌĆö free
    assertEq(address(oracle).balance, fee);            // only one fee collected
}
```
