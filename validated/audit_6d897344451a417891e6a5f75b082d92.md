Audit Report

## Title
Unrestricted `sweepToken` Allows Any Caller to Drain Router ERC20 Balance to Arbitrary Address — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.sweepToken` is `public payable` with no `msg.sender` check and an attacker-controlled `recipient` parameter. Any EOA can call it with `amountMinimum = 0` and drain the router's entire balance of any ERC20 token to an arbitrary address. The only guard — the `amountMinimum` floor — is trivially bypassed by passing `0`.

## Finding Description
`sweepToken` at line 48 performs no check on `msg.sender` and no check that `recipient == msg.sender`. [1](#0-0) 

By contrast, `refundETH` at line 58–63 hard-codes `msg.sender` as the destination, confirming the asymmetry is a defect rather than a design choice. [2](#0-1) 

The inheritance chain (`MetricOmmSwapRouterBase`, `SelfPermit`) adds no access control to `sweepToken`. The router can hold a non-zero ERC20 balance in realistic situations: tokens accidentally sent directly to the router (a common user error), or an attacker front-running a staged `multicall` where an intermediate step routes tokens through the router address. [3](#0-2) 

The `pay` internal function's `payer == address(this)` branch is intra-transaction only and does not leave a persistent balance, but the above two vectors do. [4](#0-3) 

## Impact Explanation
Any ERC20 balance held by the router — regardless of how it arrived — can be fully drained to an attacker-controlled address in a single call. This is a direct, complete loss of the affected token balance with no recovery path, meeting the High impact threshold.

## Likelihood Explanation
The call requires no special role, no pool interaction, and no prior state. The only precondition is that the router holds a non-zero ERC20 balance, achievable via direct transfer (common user error) or front-running a staged multicall. Likelihood is **Medium** (precondition required), impact is **High**, placing overall severity at **High**.

## Recommendation
Restrict `sweepToken` (and `unwrapWETH9`) so that `recipient` must equal `msg.sender`, mirroring the `refundETH` pattern:

```solidity
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
+   if (recipient != msg.sender) revert RecipientNotSender();
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
```

Alternatively, remove the `recipient` parameter entirely and always send to `msg.sender`.

## Proof of Concept
The existing test `test_sweepToken_sendsFullBalance` already demonstrates the unrestricted drain — the caller is not the depositor and `recipient` is an arbitrary address: [5](#0-4) 

Minimal PoC extending this pattern:
```solidity
// 1. Transfer 1000 USDC directly to router (simulates user error)
token1.mint(address(router), 1000);
assertEq(token1.balanceOf(address(router)), 1000);

// 2. Attacker (fresh EOA, no approvals, no role) calls:
vm.prank(attacker);
router.sweepToken(address(token1), 0, attacker);

// 3. Assert
assertEq(token1.balanceOf(attacker), 1000);   // attacker received all tokens
assertEq(token1.balanceOf(address(router)), 0); // router drained
```
