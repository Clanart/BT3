### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key Guard, Silently Corrupting DataLayer Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` uses a fast-path that writes leaf nodes directly to the blob via `insert_entry_to_blob` without performing the duplicate-key or duplicate-hash checks that `insert` enforces. When a caller supplies a key already present in the tree, or supplies the same key more than once within a batch of ≥3 new items, the tree blob ends up with two live leaf nodes for the same key. The in-memory cache (`block_status_cache`) silently overwrites its pointer to the old leaf, leaving the old leaf orphaned-but-structurally-connected. The resulting root hash is computed over a structurally corrupt tree, and proofs of inclusion generated from that root are invalid against any honest root.

---

### Finding Description

`MerkleBlob::insert` guards against duplicate keys and hashes before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path for all items beyond the first two. When the tree already has ≥2 leaves, the function skips the `insert` call entirely and writes each new leaf directly via `insert_entry_to_blob`:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` and silently overwrites any existing entry for the same key:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
``` [3](#0-2) 

The old leaf node at its original index remains physically present in the blob and is still wired into the tree's parent-child structure. The new leaf is also wired in as part of the freshly built subtree. After `calculate_lazy_hashes()`, the root hash is computed over a tree that contains **two** leaf nodes for the same key.

`check_integrity_on_drop` is only enabled in test builds (`cfg!(test)`), so production code never detects the corruption: [4](#0-3) 

The Python binding `py_batch_insert` exposes this path directly to callers: [5](#0-4)

### Impact Explanation

Two concrete corruption scenarios exist:

**Scenario A – pre-existing key in the tree:** A caller invokes `batch_insert` with a key that was already inserted by a prior `insert` or `batch_insert`. The old leaf remains in the blob and is part of the tree structure; the new leaf is also inserted. The root hash is computed over both leaves. Any proof of inclusion generated via `get_proof_of_inclusion` is anchored to this corrupt root. A verifier holding the honest (pre-corruption) root will reject the proof; a verifier that accepted the corrupt root will accept a proof for a value that was silently overwritten.

**Scenario B – intra-batch duplicate keys:** A batch of ≥3 items contains the same key twice. Both leaves are written to the blob and both are included in the new subtree. The cache points to the last one. The root hash is computed over a tree with two leaves for the same key.

In both cases the DataLayer Merkle tree root is corrupted: it commits to a state that no honest tree would produce. Downstream consumers that rely on the root hash to verify DataLayer state (e.g., via `ProofOfInclusion::valid()`) will either accept forged state or diverge from nodes that computed the root correctly. [6](#0-5) 

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion API exposed to Python callers and is used in all DataLayer batch-update workflows. Any DataLayer operation that re-inserts a key (e.g., an update implemented as insert-over-existing rather than upsert) will silently corrupt the tree if the batch contains ≥3 items and the tree already has ≥2 leaves. The missing guard is not documented as intentional; the single-item `insert` path explicitly rejects duplicates, making this a clear inconsistency. No special privilege is required — any DataLayer store owner can trigger the path.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the fast path in `batch_insert` before calling `insert_entry_to_blob`. Specifically, inside the loop at line 587, check `self.block_status_cache.contains_key(key)` and `self.block_status_cache.contains_leaf_hash(&hash)` and return `Err(Error::KeyAlreadyPresent())` / `Err(Error::HashAlreadyPresent())` on violation, mirroring the checks in `insert`. Additionally, consider enabling `check_integrity` in production (or at least after every `batch_insert`) to catch any future regressions.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so batch_insert uses the fast path
    let k0 = KeyId(0); let h0 = Hash(Bytes32::new([0u8; 32]));
    let k1 = KeyId(1); let h1 = Hash(Bytes32::new([1u8; 32]));
    blob.insert(k0, ValueId(0), &h0, InsertLocation::Auto {}).unwrap();
    blob.insert(k1, ValueId(1), &h1, InsertLocation::Auto {}).unwrap();

    // batch_insert with k0 already present — fast path, no duplicate check
    let k2 = KeyId(2); let h2 = Hash(Bytes32::new([2u8; 32]));
    let k3 = KeyId(3); let h3 = Hash(Bytes32::new([3u8; 32]));
    // k0 is a duplicate of the pre-existing key
    blob.batch_insert(vec![
        ((k2, ValueId(2)), h2),
        ((k3, ValueId(3)), h3),
        ((k0, ValueId(99)), Hash(Bytes32::new([99u8; 32]))), // duplicate!
    ]).unwrap(); // succeeds — should have returned KeyAlreadyPresent

    blob.calculate_lazy_hashes().unwrap();

    // Tree now has two leaf nodes for k0; root hash is corrupt.
    // check_integrity() (test-only) would return IntegrityKeyToIndexCacheLength.
    // In production, the corrupt root is silently committed.
    let proof = blob.get_proof_of_inclusion(k0).unwrap();
    assert!(proof.valid()); // valid against the corrupt root
}

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
