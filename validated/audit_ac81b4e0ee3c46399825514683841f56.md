### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Unbacked Mint on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` calls `safeTransferFrom` with the caller-supplied `amount` but emits the `InitTransfer` event (and publishes the Wormhole message) using that same caller-supplied `amount` rather than the actual tokens received. For fee-on-transfer ERC20 tokens, the contract locks `amount - transfer_fee` tokens while the cross-chain proof records `amount`. The NEAR bridge finalizes the transfer using the proof's `amount`, minting more tokens than were ever locked on EVM, creating unbacked supply.

---

### Finding Description

In `OmniBridge.initTransfer`, the non-bridge ERC20 path is:

```solidity
// OmniBridge.sol lines 407-411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied
);
```

Immediately after, the event is emitted with the same caller-supplied `amount`:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // NOT actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
```

In the Wormhole variant (`OmniBridgeWormhole.sol`), `initTransferExtension` publishes the same `amount` into the Wormhole message payload:

```solidity
// OmniBridgeWormhole.sol lines 136
Borsh.encodeUint128(amount),   // NOT actual received amount
```

The NEAR bridge's `fin_transfer_callback` reconstructs the `TransferMessage` directly from the proof's `init_transfer.amount` field and uses it to mint or unlock tokens:

```rust
// near/omni-bridge/src/lib.rs line 725
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
```

This `amount` then flows into `process_fin_transfer_to_near` → `send_tokens` → mint/transfer to recipient. The NEAR side has no independent way to verify how many tokens the EVM contract actually holds; it trusts the event entirely, as documented in `evm/CLAUDE.md`: *"The NEAR side relies solely on these events — any missing or ambiguous field means lost funds or spoofable transfers."*

The same pattern exists identically in `starknet/src/omni_bridge.cairo` lines 304–306 and 323, where `transfer_from` is called with `amount` and the `InitTransfer` event emits the same caller-supplied `amount`.

---

### Impact Explanation

**Critical — Unauthorized mint of bridged assets / bridge collateral undercollateralization.**

For every `initTransfer` call with a fee-on-transfer token:
- EVM bridge holds `amount - transfer_fee` tokens
- NEAR bridge mints `amount` tokens to the recipient

The difference (`transfer_fee`) is unbacked. Repeated calls drain the bridge's EVM-side collateral relative to the NEAR-side supply. When legitimate users later bridge back (NEAR → EVM), the EVM bridge cannot honor all redemptions because it holds fewer tokens than the total NEAR-side supply represents.

---

### Likelihood Explanation

Any ERC20 token with a transfer fee (e.g., tokens with deflationary mechanics, reflection tokens, or tokens that route fees to a treasury on transfer) that is registered as a bridgeable asset triggers this. The attacker needs only to call the public `initTransfer` function — no privileged access required. The attacker can be the same user bridging their own tokens; the profit comes from receiving more on NEAR than was locked on EVM.

---

### Recommendation

Measure the actual received amount using a balance-before/balance-after pattern for the non-bridge ERC20 path, and use that measured amount in the event and Wormhole payload:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    require(received <= type(uint128).max, "amount overflow");
    amount = uint128(received);   // rebind to actual received
}
```

Then `amount` used in `initTransferExtension(...)` and `emit BridgeTypes.InitTransfer(...)` will reflect what is actually held. Apply the same fix to `starknet/src/omni_bridge.cairo`'s `init_transfer`.

---

### Proof of Concept

1. Deploy or use any ERC20 token that deducts a 1% fee on every `transferFrom` (e.g., a reflection token).
2. Call `OmniBridge.initTransfer(feeToken, 1000e18, 0, 0, "alice.near", "")`.
3. The contract receives `990e18` tokens (1% fee deducted).
4. The `InitTransfer` event emits `amount = 1000e18`.
5. A relayer submits the proof to the NEAR bridge via `fin_transfer`.
6. `fin_transfer_callback` reads `init_transfer.amount = 1000e18` and mints `1000e18` NEAR-side tokens to `alice.near`.
7. The EVM bridge is now undercollateralized by `10e18` tokens per call.
8. Repeat until the EVM bridge's token balance is exhausted relative to NEAR-side supply, permanently freezing legitimate redemptions. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
