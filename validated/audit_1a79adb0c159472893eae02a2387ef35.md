### Title
Empty `check_is_reverted` Allows Prover to Bypass Invoke Transaction Execution While Charging Fees — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is completely empty. In `execute_invoke_function_transaction`, the `is_reverted` flag is loaded from an unconstrained hint and passed to this no-op function. Because no Cairo constraint is ever asserted on the hint value, the prover can freely declare any invoke transaction as reverted — skipping its `__execute__` entry point — while the fee-charge step still runs unconditionally. The resulting proof is valid, yet the user's intended state changes are never applied and their fee is consumed.

The critical asymmetry is:

- When `is_reverted == FALSE`, `non_reverting_select_execute_entry_point_func` is called, which internally asserts `is_reverted = 0` on the *actual* execution result — so the hint is cross-checked against reality.
- When `is_reverted != FALSE`, the execute entry point is **never called**. There is no actual execution whose result could contradict the hint. The hint is the only source of truth, and it is unconstrained.

`charge_fee` is called unconditionally after both branches, so the fee transfer always executes.
