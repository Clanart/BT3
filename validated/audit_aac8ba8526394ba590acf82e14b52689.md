### Title
Arithmetic Overflow in `denormalize_amount` Corrupts Bridge Transfer Amounts — (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

`denormalize_amount` in the NEAR bridge contract performs an unchecked multiplication `amount * (10_u128.pow(diff_decimals))`. When `diff_decimals ≥ 39`, `10_u128.pow(diff_decimals)` silently overflows `u128` in Rust release mode (or panics if overflow-checks are enabled), producing either a garbage amount used for token minting or a permanent transaction revert that freezes user funds.

---

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

`u128::MAX ≈ 3.4 × 10^38`, so `10_u128.pow(39)` already exceeds the type's capacity. Any token registered with `origin_decimals − decimals ≥ 39` triggers this overflow on every inbound transfer.

The `Decimals` struct stores two independent fields:

```rust
pub struct Decimals {
    pub decimals: u8,
    pub origin_decimals: u8,
}
``` [2](#0-1) 

The Solana bridge explicitly separates these two values when deploying a token. It caps the on-chain mint decimals at `MAX_ALLOWED_DECIMALS` (9 for SPL tokens) while preserving the original source-chain decimals:

```rust
let origin_decimals = metadata.decimals;
metadata.decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, metadata.decimals);
// ...
let payload = DeployTokenResponse {
    decimals: metadata.decimals,   // e.g. 9
    origin_decimals,               // e.g. 57
    ...
``` [3](#0-2) 

This `DeployTokenResponse` is sent to NEAR as a Wormhole message. NEAR's `bind_token` (permissionless when unpaused) processes it via `bind_token_callback`, which reads `DeployTokenMessage.decimals` and `DeployTokenMessage.origin_decimals`:

```rust
pub struct DeployTokenMessage {
    pub decimals: u8,
    pub origin_decimals: u8,
    ...
}
``` [4](#0-3) 

These values are stored verbatim in `token_decimals` via `add_token`. With `decimals = 9` and `origin_decimals = 57`, `diff_decimals = 48`, and `10_u128.pow(48)` overflows.

`denormalize_amount` is called in three critical paths:

**`fin_transfer_callback`** — converts the normalized amount back to original decimals before storing in `TransferMessage`: [5](#0-4) 

**`claim_fee_callback`** — recomputes the denormalized amount for fee settlement: [6](#0-5) 

**`fast_fin_transfer`** — denormalizes the fast-transfer amount: [7](#0-6) 

---

### Impact Explanation

**If overflow-checks are disabled (release default):** `10_u128.pow(48)` wraps to a small garbage value. The resulting `transfer_message.amount` is a corrupted number. For a bridged (deployed) token, NEAR calls `mint(recipient, corrupted_amount)`, issuing unbacked tokens. For a native NEAR token, it attempts `ft_transfer` of a wrong amount, breaking collateralization. This is a **Critical** direct-theft / unbacked-mint impact.

**If overflow-checks are enabled:** The NEAR runtime panics and reverts the transaction. Because the transfer is already recorded on Solana (funds locked/burned), but NEAR can never finalize it, user funds are **permanently frozen** — a Critical irrecoverable lock.

---

### Likelihood Explanation

- `bind_token` is permissionless (no role guard, only a pause check).
- An attacker deploys an ERC-20 with `decimals() = 57` on any supported EVM chain, logs its metadata, deploys it on Solana (which normalizes to 9), and submits the resulting Wormhole proof to NEAR's `bind_token`. All steps are available to any unprivileged user.
- Once the token is registered with the corrupted `Decimals`, every subsequent Solana→NEAR transfer of that token triggers the overflow.
- No privileged access, no key compromise, and no colluding validators are required.

---

### Recommendation

1. **Add an overflow guard in `denormalize_amount`:** assert or check that `diff_decimals < 39` (i.e., `10_u128.pow(diff_decimals) <= u128::MAX / amount`) before multiplying, and use `checked_mul` / `saturating_mul`.
2. **Validate `origin_decimals >= decimals` in `add_token`:** reject registrations where the subtraction would underflow or produce an unsafe exponent.
3. **Cap `diff_decimals`:** enforce a protocol-level maximum (e.g., 18) on `origin_decimals − decimals` at token registration time.
4. **Add fuzz tests** covering extreme decimal combinations (e.g., `decimals = 0`, `origin_decimals = 255`) to catch edge cases in both `normalize_amount` and `denormalize_amount`.

---

### Proof of Concept

1. Deploy `MaliciousToken` on Ethereum with `decimals() = 57`.
2. Call `logMetadata(maliciousToken)` on the EVM OmniBridge — emits a `LogMetadata` event.
3. Submit the Wormhole proof to Solana's `deploy_token`. Solana stores `decimals = 9`, `origin_decimals = 57` and emits a `DeployTokenResponse` Wormhole message.
4. Call NEAR's permissionless `bind_token` with the Solana Wormhole proof. NEAR stores `Decimals { decimals: 9, origin_decimals: 57 }` for the Solana mint address.
5. Transfer 1 unit of `MaliciousToken` from Solana to NEAR. The Solana bridge locks/burns 1 unit and emits an `InitTransfer` Wormhole message with `amount = 1`.
6. Submit the proof to NEAR's `fin_transfer`. NEAR calls:
   ```rust
   denormalize_amount(1, Decimals { decimals: 9, origin_decimals: 57 })
   // diff_decimals = 48
   // 10_u128.pow(48) overflows → garbage value
   // 1 * garbage = garbage amount minted to recipient
   ``` [1](#0-0) 
7. NEAR mints a corrupted (potentially astronomically large) amount of bridged tokens to the recipient, breaking bridge collateralization.
