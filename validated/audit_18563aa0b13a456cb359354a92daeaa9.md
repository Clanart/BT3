### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the caller-supplied class hash corresponds to a previously declared contract class. Any contract can replace its own class hash with an arbitrary, undeclared felt value. The OS produces a valid proof for this state transition, permanently rendering the contract non-executable and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the OS reads the new class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash is a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The explicit `TODO` comment at line 898 acknowledges this missing check. The `contract_class_changes` dictionary — which tracks declared class hashes — is not consulted at all. Any felt value, including `0x1` or any other undeclared hash, is accepted and committed to state.

This is the direct analog to the external report: just as `set_debt_ceiling` mints tokens to any address without checking it is a Controller or PegKeeper, `execute_replace_class` sets a contract's class to any hash without checking it is a declared class.

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value, every subsequent call to that contract fails at class resolution. There is no recovery path in the OS: the state is committed to L1 with a valid proof, and the contract is permanently non-executable. Any funds held by the contract — ERC-20 tokens, fee tokens, or any other assets — are permanently frozen with no mechanism for withdrawal.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack path requires only that an unprivileged user deploy a contract. The contract's own code can call the `replace_class` syscall with an arbitrary hash. The OS enforces no constraint on the hash value. The sequencer's off-chain execution layer is also expected to rely on the OS for this enforcement (as evidenced by the TODO), so the transaction would be included in a block and proven. No privileged role, leaked key, or malicious operator is required.

---

### Recommendation

Before writing the new `StateEntry`, verify that `class_hash` exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). The check should assert that a `dict_read` on `contract_class_changes` for `class_hash` returns a non-zero compiled class hash, consistent with how `execute_declare_transaction` enforces `prev_value=0` to prevent double-declaration.

---

### Proof of Concept

1. Attacker writes and deploys a contract `VaultMalicious` that:
   - Exposes a `deposit()` entry point that accepts and stores user funds.
   - Exposes a `rug()` entry point that calls `replace_class(class_hash=0x1)`.
2. Victims call `deposit()` and transfer funds to `VaultMalicious`.
3. Attacker calls `rug()`. The contract executes the `replace_class` syscall with `class_hash = 0x1`.
4. `execute_replace_class` in the OS writes `StateEntry(class_hash=0x1, ...)` for `VaultMalicious` into `contract_state_changes` — **no declared-class check is performed** (line 898 TODO).
5. The OS produces a valid STARK proof for this state transition; L1 accepts it.
6. All subsequent calls to `VaultMalicious` fail: the OS cannot resolve class `0x1` to any compiled class.
7. All deposited funds are permanently frozen.
