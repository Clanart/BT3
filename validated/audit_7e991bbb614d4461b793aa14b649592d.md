Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal ETH stranded in the router after a direct payable swap call — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` sends the router's entire native ETH balance to `msg.sender` with no access control. Any payable swap entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) accepts `msg.value` but never auto-refunds surplus ETH at the end of the call. When a user sends `msg.value > actual amountIn` (unavoidable for exact-output swaps), the residual ETH is permanently accessible to any subsequent caller of `refundETH()`, enabling direct theft of user principal.

## Finding Description

`refundETH()` unconditionally transfers the full contract ETH balance to `msg.sender`:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

There is no check that `msg.sender` is the original depositor, no per-caller accounting, and no reentrancy guard.

The `receive()` guard only blocks plain ETH transfers; it does not restrict `msg.value` on calls that carry calldata (i.e., any payable function call):

```solidity
// PeripheryPayments.sol L32-34
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
```

During the swap callback, `pay()` wraps **exactly** `value` ETH into WETH — only the precise amount the pool requests:

```solidity
// PeripheryPayments.sol L73-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
```

After the swap, neither `exactInputSingle` nor `exactOutputSingle` refunds any remaining ETH — both simply clear the callback context and return:

```solidity
// MetricOmmSimpleRouter.sol L83-86
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);
    _clearExpectedCallbackPool();
}
```

```solidity
// MetricOmmSimpleRouter.sol L145-147
    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
}
```

The surplus `msg.value − actual amountIn` sits in the router until any caller invokes `refundETH()`.

## Impact Explanation

Direct loss of user ETH principal. Any ETH sent as `msg.value` in excess of the swap's actual cost is permanently accessible to any address that calls `refundETH()`. The attacker receives the victim's ETH with no preconditions, no privileged role, and no special token behavior. This meets the High threshold for direct loss of user principal.

## Likelihood Explanation

- `exactOutputSingle` with ETH input is the primary trigger: the user cannot know the exact cost upfront and must send `msg.value >= amountInMaximum`, guaranteeing surplus ETH in the router after every such call.
- `exactInputSingle` called directly (not via `multicall`) with `msg.value > amountIn` also strands ETH.
- The attack requires no privileged access, no malicious pool, and no special token behavior.
- An attacker can monitor the mempool for transactions that leave ETH in the router and call `refundETH()` in the next block, or front-run the victim's own `refundETH()` call.
- The attack is repeatable on every such swap.

## Recommendation

At the end of each payable swap entry point, automatically refund any remaining `address(this).balance` to `msg.sender`:

```solidity
if (address(this).balance > 0) _transferETH(msg.sender, address(this).balance);
```

This matches the pattern already expected when users compose `multicall([swap(...), refundETH()])`, but makes it safe for direct single-function calls as well. Alternatively, bind `refundETH()` to a stored original-caller address set at swap entry and cleared on exit, but the auto-refund approach is simpler and eliminates the stranding entirely.

## Proof of Concept

```solidity
// Victim calls exactOutputSingle with excess ETH (cannot know exact cost upfront)
uint256 amountIn = router.exactOutputSingle{value: 1.001 ether}(
    IMetricOmmSimpleRouter.ExactOutputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountOut: someAmount,       // costs exactly 1 ETH
        amountInMaximum: 1.001 ether,
        recipient: victim,
        deadline: block.timestamp + 60,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// amountIn == 1 ETH; 0.001 ETH remains stranded in router

// Attacker (separate tx, next block)
uint256 attackerBefore = attacker.balance;
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance - attackerBefore, 0.001 ether); // attacker stole victim's ETH
```
