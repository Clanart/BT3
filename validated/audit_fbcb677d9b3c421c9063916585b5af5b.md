### Title
Detached `burn_tokens_if_needed` Promise Result Never Verified — Silent Burn Failure Enables Unbacked Supply or Double-Finalization - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`burn_tokens_if_needed` fires a cross-contract burn call with `.detach()`, meaning the result is never checked. If the burn fails for any reason (e.g., the 3 TGas allocation is insufficient, or the token contract panics), the bridge has already committed its state changes. In the `fin_transfer_send_tokens_callback` path this additionally removes the transfer from `finalised_transfers`, allowing the same proof to be re-submitted for double-minting. In the `init_transfer_internal` path, unburned tokens remain in the bridge's balance while the destination chain releases native assets, inflating supply.

---

### Finding Description

`burn_tokens_if_needed` is defined as:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result is silently discarded
    }
}
``` [1](#0-0) 

`BURN_TOKEN_GAS` is only **3 TGas**: [2](#0-1) 

A NEAR cross-contract call alone costs ~2.4 TGas in base overhead, leaving fewer than 0.6 TGas for the `burn()` body, which must read and write storage (account balance + total supply). This budget is extremely tight and can fail. Because `.detach()` is used, any failure is silently swallowed.

This function is called in four critical locations, all of which commit state **before** the burn result could be observed:

**Location 1 — `init_transfer_internal` (line 1851):** After the transfer message is added to `pending_transfers` and the nonce is incremented, `burn_tokens_if_needed` is called detached. If the burn fails, the `InitTransfer` event is still emitted and the transfer proceeds to signing and destination-chain finalization. [3](#0-2) 

**Location 2 — `fin_transfer_send_tokens_callback` (line 1703):** When `is_refund_required` is true (the `ft_transfer_call` to the recipient failed), the bridge calls `burn_tokens_if_needed` detached and then calls `remove_fin_transfer`, which **removes the transfer ID from `finalised_transfers`**: [4](#0-3) 

`remove_fin_transfer` deletes the entry from the replay-protection set: [5](#0-4) 

If the burn fails, the minted tokens remain in the bridge's balance AND the transfer ID is no longer in `finalised_transfers`. A relayer can re-submit the same proof, pass the `add_fin_transfer` guard (which requires the ID to be absent), and mint another batch of tokens — double-minting.

**Location 3 — `resolve_fast_transfer` (line 904):** The comment explicitly states "Burn the tokens to ensure the locked tokens are not double-minted," yet the burn is detached: [6](#0-5) 

**Location 4 — `fast_fin_transfer_to_other_chain` (line 932):** Same pattern — burn detached before state is committed. [7](#0-6) 

The `burn()` function on the token contract withdraws from `env::predecessor_account_id()` (the bridge): [8](#0-7)

### Impact Explanation

**`fin_transfer_send_tokens_callback` path (most severe):**
- A relayer finalizes a transfer to NEAR with a `msg` (triggering `ft_transfer_call`).
- The recipient contract's `ft_on_transfer` panics or returns 0 (unused amount), causing `is_refund_required = true`.
- The bridge calls `burn_tokens_if_needed` detached (3 TGas) and then removes the transfer ID from `finalised_transfers`.
- If the burn fails silently, the minted tokens remain in the bridge's balance and the transfer ID is gone from the replay-protection set.
- The same proof can be re-submitted; `add_fin_transfer` succeeds (ID absent), and another batch of tokens is minted.
- **Result: double-minting of bridged assets — unbacked supply on NEAR.**

**`init_transfer_internal` path:**
- User bridges deployed tokens from NEAR to EVM. Tokens are transferred to the bridge via `ft_transfer_call`.
- `burn_tokens_if_needed` is called detached. If it fails, 100 tokens remain in the bridge's balance.
- The transfer is recorded, signed, and finalized on EVM, releasing native assets.
- **Result: 100 tokens exist in the bridge's balance AND 100 native assets are released on EVM — inflated supply.**

Both impacts fall under: **Critical — unauthorized mint of bridged assets** and **High — double-finalization enabling double-spend or unbacked supply**.

---

### Likelihood Explanation

- `BURN_TOKEN_GAS = 3 TGas` is extremely tight. NEAR charges ~2.4 TGas base for a cross-contract call, leaving <0.6 TGas for the `burn()` body (storage read + write). Under any gas-price spike or runtime overhead, this can fail.
- The `fin_transfer_send_tokens_callback` failure path is reachable by any user who deploys a receiver contract whose `ft_on_transfer` panics or returns 0 — this is an unprivileged, user-controlled entry point.
- No special role or key is required; any bridge user can trigger the failure path by controlling the recipient contract.

---

### Recommendation

1. **Do not use `.detach()` for security-critical burns.** Chain the burn as a proper callback and verify its result before committing state.
2. **In `fin_transfer_send_tokens_callback`**, do not call `remove_fin_transfer` until the burn result is confirmed. Alternatively, keep the transfer in `finalised_transfers` permanently (as a replay-protection tombstone) and only remove it after a confirmed burn.
3. **Increase `BURN_TOKEN_GAS`** to at least 5–10 TGas to cover cross-contract overhead plus storage operations.
4. **Pattern fix for `init_transfer_internal`**: return the burn promise and chain subsequent state changes as a callback, reverting the transfer message if the burn fails.

---

### Proof of Concept

**Double-minting via `fin_transfer_send_tokens_callback`:**

1. Deploy a malicious receiver contract `evil.near` whose `ft_on_transfer` always panics.
2. Submit a valid `fin_transfer` proof for a deployed token, specifying `evil.near` as recipient with a non-empty `msg`.
3. Bridge mints tokens to itself, calls `ft_transfer_call(evil.near, amount, msg)`.
4. `evil.near`'s `ft_on_transfer` panics → `is_refund_required = true`.
5. `fin_transfer_send_tokens_callback` fires `burn_tokens_if_needed` detached (3 TGas — may fail) and calls `remove_fin_transfer`.
6. Transfer ID is removed from `finalised_transfers`; tokens remain in bridge's balance if burn failed.
7. Re-submit the identical proof → `add_fin_transfer` succeeds → bridge mints another batch.
8. Repeat for unbounded double-minting.

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
