### Title
Unauthorized BOOTSTRAP Declare Bypasses Signature Verification and Fee Payment, Enabling Class Hash Poisoning and Permanent Fund Freezing - (File: `execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS contains a special "BOOTSTRAP" code path that skips signature verification, nonce enforcement, and fee charging for any declare transaction whose `sender_address` equals the publicly known felt value `'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and all resource bounds set to zero. Because `'BOOTSTRAP'` is simply an ASCII-encoded felt constant with no cryptographic gating, any unprivileged actor who can get such a transaction included in a block can declare any class hash with an arbitrary `compiled_class_hash`, permanently poisoning the class registry.

---

### Finding Description

In `transaction_impls.cairo` at lines 761–776, the OS executes the following branch inside `execute_declare_transaction`:

```cairo
// Do not run validate or perform any account-related actions for declare transactions that
// meet the following conditions.
// This flow is used for the sequencer to bootstrap a new system.
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        // Declare the class hash and skip the rest of the transaction.
        // Note that prev_value=0 enforces that a class may be declared only once.
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
```

When this branch is taken, the OS:

1. **Skips `run_validate`** — no account signature is verified.
2. **Skips `check_and_increment_nonce`** — the nonce at address `'BOOTSTRAP'` is never incremented, so the same nonce `0` can be reused for every BOOTSTRAP declare.
3. **Skips `charge_fee`** — no ERC-20 transfer is executed; the sequencer collects nothing.
4. **Directly writes `compiled_class_hash` into `contract_class_changes`** — with the attacker-supplied value.

The guard conditions are:

| Condition | Attacker control |
|---|---|
| `sender_address == 'BOOTSTRAP'` | Freely set; `'BOOTSTRAP'` = `0x424f4f545354524150` is public |
| `tx_info.nonce == 0` | Freely set |
| `tx_info.version == 3` | Freely set |
| `max_possible_fee == 0` | Freely set (zero all resource bounds) |

None of these conditions require a cryptographic proof of identity. The comment acknowledges the intent ("used for the sequencer to bootstrap a new system"), but there is no enforcement mechanism — no sequencer signature, no allowlist, no on-chain authority check.

The analog to the Zivoe `depositReward(zero)` pattern is exact: just as anyone could call `depositReward` with `reward = 0` to manipulate the reward period, here anyone can call the BOOTSTRAP declare path with `max_possible_fee = 0` to manipulate the class registry without authorization. ### Impact Explanation

**Primary — Permanent freezing of funds (Critical):**

The `dict_update` call enforces `prev_value=0`, meaning each class hash can be declared exactly once. An attacker who front-runs a legitimate class declaration with a BOOTSTRAP transaction supplying a garbage or malicious `compiled_class_hash` permanently poisons that class hash entry. Any contract subsequently deployed under that class hash will fail at execution time (the `compiled_class_hash` stored in the class changes dict will not correspond to any valid compiled class known to the prover), locking all funds held by those contracts forever.

**Secondary — Direct loss of funds (Critical):**

Because `charge_fee` is bypassed entirely, the attacker declares classes for free. The `charge_fee` function only executes when `compute_max_possible_fee > 0`; with all resource bounds zeroed, it returns immediately. [2](#0-1) 

---

### Likelihood Explanation

- The felt value `'BOOTSTRAP'` is hardcoded in open-source Cairo and trivially known.
- The conditions (`nonce=0`, `version=3`, all bounds zero) are freely constructable by any transaction sender.
- In a decentralized sequencer environment, an attacker who is themselves a sequencer (or who can submit a well-formed transaction that a sequencer includes) can trigger this path.
- The nonce is never incremented in the BOOTSTRAP path, so the same attacker address can submit unlimited BOOTSTRAP declares in a single block, poisoning many class hashes at once.
- Front-running a class declaration is realistic: Sierra class hashes are deterministic and can be computed off-chain before submission. [3](#0-2) 

---

### Recommendation

Replace the magic-felt guard with a cryptographically enforced identity check. Options include:

1. **Sequencer signature**: Require the BOOTSTRAP transaction to carry a signature from a known sequencer public key, verified inside the OS before entering the BOOTSTRAP branch.
2. **Remove the BOOTSTRAP path**: Use a separate, out-of-band mechanism (e.g., genesis block initialization) that does not pass through the normal transaction execution loop.
3. **Allowlist via OS global context**: Store an authorized bootstrapper address in `OsGlobalContext` and assert `sender_address == block_context.os_global_context.bootstrapper_address`, where that address is set at OS initialization time and cannot be forged. [1](#0-0)

### Proof of Concept

**Setup**: Attacker observes that a legitimate developer is about to declare Sierra class `C` (class hash publicly computable from the Sierra source).

**Attack steps**:

1. Attacker constructs a declare transaction with:
   - `sender_address = 0x424f4f545354524150` (`'BOOTSTRAP'`)
   - `nonce = 0`
   - `version = 3`
   - `resource_bounds` for L1_GAS, L2_GAS, L1_DATA_GAS all set to `max_amount=0, max_price_per_unit=0`
   - `class_hash = C` (the target class hash)
   - `compiled_class_hash = 1` (any non-zero garbage value)

2. Attacker submits this transaction. The OS evaluates:
   - `sender_address == 'BOOTSTRAP'` ✓
   - `nonce == 0` ✓
   - `version == 3` ✓
   - `compute_max_possible_fee == 0` ✓ (all bounds zero)

3. OS skips `run_validate`, `check_and_increment_nonce`, `charge_fee`.

4. OS executes:
   ```cairo
   dict_update{dict_ptr=contract_class_changes}(
       key=C, prev_value=0, new_value=1  // garbage compiled_class_hash
   );
   ```

5. Legitimate developer's declare transaction for `C` now fails: `prev_value=0` assertion is violated because `C` is already registered.

6. Any contract deployed with class hash `C` will fail at execution time — the prover cannot find a compiled class matching `compiled_class_hash=1`. All funds in such contracts are permanently frozen. [3](#0-2) [4](#0-3) [2](#0-1)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
