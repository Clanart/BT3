The `calculate_state_diff_hash` function explicitly chains `deprecated_declared_classes` into the Poseidon hash, but `From<CommitmentStateDiff> for ThinStateDiff` always supplies an empty `Vec::new()` for that field. This is a direct structural analog to the external bug.

### Title
Wrong `state_diff_commitment` and `concatenated_counts` in block hash due to silently-dropped `deprecated_declared_classes` in `CommitmentStateDiff → ThinStateDiff` conversion — (File: `crates/blockifier/src/state/cached_state.rs`)

### Summary
`BlockExecutionArtifacts::new` converts `CommitmentStateDiff` to `ThinStateDiff` before computing block commitments. The conversion unconditionally sets `deprecated_declared_classes: Vec::new()`. Because `calculate_state_diff_hash` and `ThinStateDiff::len()` both consume that field, any block that contains a `DeclareV1` (Cairo 0 / deprecated class declaration) transaction will produce a wrong `state_diff_commitment` and a wrong `state_diff_length` inside `concatenated_counts`, corrupting the block hash.

### Finding Description

`calculate_state_diff_hash` hashes the full state diff as:

```
Poseidon("STARKNET_STATE_DIFF0", deployed_contracts, declared_classes,
         deprecated_declared_classes, 1, 0, storage_diffs, nonces)
``` [1](#0-0) 

`ThinStateDiff::len()` also counts `deprecated_declared_classes.len()` as part of the state-diff length that is packed into `concatenated_counts`: [2](#0-1) 

However, the only path through which the batcher produces a `ThinStateDiff` for commitment computation is `From<CommitmentStateDiff> for ThinStateDiff`, which hard-codes the field to an empty vector:

```rust
// TODO(AlonH): Remove this when the structure of storage diffs changes.
deprecated_declared_classes: Vec::new(),
``` [3](#0-2) 

`BlockExecutionArtifacts::new` calls this conversion immediately before `calculate_block_commitments`:

```rust
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),   // ← deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
``` [4](#0-3) 

`calculate_block_commitments` then passes this truncated diff to both `calculate_state_diff_hash` (producing the wrong `state_diff_commitment`) and `concat_counts` (producing the wrong `state_diff_length` packed into `concatenated_counts`): [5](#0-4) 

Both corrupted values are chained directly into `calculate_block_hash`: [6](#0-5) 

`CommitmentStateDiff` itself has no field for deprecated declared classes, and `From<StateMaps> for CommitmentStateDiff` does not populate one, so the information is structurally lost before it can reach the commitment path: [7](#0-6) 

### Impact Explanation

For every block that contains at least one `DeclareV1` transaction:

1. **Wrong `state_diff_commitment`** — `chain_deprecated_declared_classes` receives an empty slice instead of the actual declared class hashes, so the Poseidon hash diverges from the value the SNOS (Starknet OS) would compute over the real state diff.
2. **Wrong `concatenated_counts`** — `state_diff_length` is underestimated by exactly the number of deprecated declared classes, corrupting the packed field in the block hash.
3. **Wrong block hash** — both corrupted values are inputs to `calculate_block_hash`, so the final block hash is wrong.

A wrong block hash means the sequencer's proposal commitment (`PartialBlockHash`) diverges from what the SNOS verifies, causing proof verification to fail for any such block. This matches the **Critical** impact: *wrong state/commitment value from execution logic for accepted input*.

### Likelihood Explanation

`DeclareV1` transactions are standard, unprivileged Starknet transactions. Any user can broadcast one. The sequencer's own test suite exercises `declare_deprecated_class` flows, confirming the transaction type is accepted. Every such transaction that lands in a sequenced block silently corrupts the block commitment.

### Recommendation

`CommitmentStateDiff` must be extended with a `deprecated_declared_classes: Vec<ClassHash>` field, populated from `StateMaps` (which already tracks class declarations via `class_hashes`). The `From<CommitmentStateDiff> for ThinStateDiff` conversion must then forward this field instead of substituting `Vec::new()`. The TODO comment at line 699 of `cached_state.rs` acknowledges the gap; it must be resolved before blocks with deprecated class declarations are sequenced.

### Proof of Concept

1. Submit a `DeclareV1` transaction declaring a Cairo 0 class hash `C`.
2. The sequencer includes it in block `N`. The blockifier records `C` in `StateMaps::class_hashes` but not in any `deprecated_declared_classes` field (none exists).
3. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff)` → `deprecated_declared_classes = []`.
4. `calculate_state_diff_hash` hashes `deprecated_declared_classes = []` → commitment omits `C`.
5. `ThinStateDiff::len()` returns `N_actual - 1` (missing the one deprecated class entry).
6. `concat_counts` packs the underestimated length into `concatenated_counts`.
7. `calculate_block_hash` produces a hash that differs from the SNOS-computed hash for the same block.
8. Proof verification for block `N` fails; the sequencer has committed to an invalid block hash.

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
