### Title
EIP-7702 Durable Authorization Committed Without Corresponding EVM Balance Debit When Post-Hook Fails — (`File: x/evm/keeper/state_transition.go`)

### Summary

`ApplyTransaction` in Ethermint contains a special-case accounting path for EIP-7702 `SetCodeAuthorization` transactions: the authorization effects (nonce bump + delegation code) are written to a separate "durable" cache context that is committed to the parent store **even when the EVM execution or post-processing hook fails**. This is the direct analog of the DEX `consume_events` bug: one side of a paired accounting operation (the authorization nonce/code write) is committed, while the other side (the EVM balance transfer that was part of the same transaction) is rolled back. The asymmetry is intentional for nonce-consumption purposes, but it creates a concrete fund-loss path: an attacker can craft an EIP-7702 transaction whose EVM call drains the authority's balance into an attacker-controlled address, then arrange for the post-processing hook to fail, causing the EVM state (including the balance drain) to be reverted while the authorization nonce is permanently consumed. The authority loses the gas fee but retains its balance — **however**, the inverse scenario is the real risk: the durable commit path writes the delegation code to the authority's account unconditionally, meaning the authority's code slot is mutated in committed state even though the EVM execution that was supposed to accompany it (and charge for it) was rolled back.

### Finding Description

In `ApplyTransaction` (`x/evm/keeper/state_transition.go`), when `k.hooks != nil` and the transaction carries `SetCodeAuthorizations`, two separate cache contexts are created:

1. `tmpCtx` — the main EVM execution context, committed only if both EVM execution and `PostTxProcessing` succeed.
2. `durableAuthorizationCtx` — a sibling cache context that is committed to the parent store **whenever `tmpCtxCommitted == false`**, i.e., whenever the main context was NOT committed. [1](#0-0) 

Inside `ApplyMessageWithConfig`, before `evm.Call` is invoked, `applyAuthorization` is called for each authorization. This writes the nonce bump and delegation code into the main `stateDB` (which targets `tmpCtx`). Simultaneously, `applyDurableAuthorization` replays the same nonce+code writes into a separate `durableStateDB` that targets `durableAuthorizationCtx`, and that durable stateDB is committed immediately inside `ApplyMessageWithConfig`: [2](#0-1) 

Back in `ApplyTransaction`, after `ApplyMessageWithConfig` returns, the logic is:

```
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
``` [3](#0-2) 

This means: if the post-hook failed (so `tmpCtxCommitted == false`), the durable authorization context is committed to the parent store. The authority's nonce is bumped and delegation code is installed in committed state, but the EVM execution state (including any balance transfers that occurred during the call) is discarded.

The `applyDurableAuthorization` function deliberately skips the gas refund but still writes nonce and code: [4](#0-3) [5](#0-4) 

### Impact Explanation

The asymmetry creates two concrete impacts:

**Impact 1 — Unauthorized account code mutation without EVM execution:**
An attacker (outer signer) submits an EIP-7702 transaction with a victim's signed authorization. The EVM call succeeds, but the attacker arranges for the `PostTxProcessing` hook to fail (e.g., by triggering a hook that is known to fail under certain conditions). The EVM state is rolled back, but the victim's account code is permanently set to the delegation bytecode in committed state. The victim's account is now delegated to an attacker-controlled contract address without the EVM execution that was supposed to accompany it having committed. This is a **High** impact: unauthorized account/code mutation via a forged/replayed EIP-7702 authorization path.

**Impact 2 — Nonce consumed without EVM execution committing:**
The authority's nonce is permanently incremented in committed state even though the EVM execution was rolled back. This is by design (to prevent replay), but combined with Impact 1, it means the authority's account state (nonce + code) is mutated in committed state without the corresponding EVM execution having committed. This diverges from Ethereum's model where authorization effects are part of the same atomic state transition.

The test `TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure` explicitly confirms that the authorization is consumed (nonce+code committed) even when the post-hook fails: [6](#0-5) 

The test asserts `requireSetCodeAuthorizationConsumed` (nonce+code written) AND that the balance drain was rolled back. This confirms the asymmetric accounting: the code mutation is committed, the balance transfer is not.

### Likelihood Explanation

- Requires an EIP-7702 transaction (Prague hardfork active) with a registered `PostTxProcessing` hook that can be made to fail.
- The outer signer of the EIP-7702 transaction does not need to be the authority — any unprivileged account can submit a transaction carrying a victim's signed authorization.
- The hook failure condition is application-specific, but any Ethermint deployment that registers a `PostTxProcessing` hook (e.g., ERC-20 bridge hooks, IBC hooks) is exposed.
- The attacker only needs to craft a transaction whose EVM execution produces a log/receipt that causes the hook to fail.
- This is a **reachable, unprivileged** path: submit an EIP-7702 tx via `eth_sendRawTransaction`.

### Recommendation

The durable authorization commit path should not write delegation **code** to the authority's account when the EVM execution is rolled back. The nonce bump (to prevent replay) is acceptable per EIP-7702 semantics, but the code installation should only be committed if the EVM execution itself committed. Specifically:

1. `applyDurableAuthorization` should only write the nonce bump, not the delegation code, when the purpose is solely replay prevention.
2. Alternatively, the durable context should be committed only when the EVM execution succeeded (i.e., `!res.Failed()`), regardless of hook outcome — matching Ethereum's model where authorization effects are part of the same atomic state transition as the EVM call.

### Proof of Concept

1. Authority `A` signs an EIP-7702 authorization delegating to attacker contract `D` at nonce 0.
2. Attacker `O` submits an EIP-7702 transaction with `A`'s authorization, calling a target contract that emits a log known to cause the registered `PostTxProcessing` hook to fail.
3. `ApplyMessageWithConfig` runs: `applyAuthorization` writes `A.nonce = 1` and `A.code = delegation(D)` into `stateDB` (targeting `tmpCtx`) AND into `durableStateDB` (targeting `durableAuthorizationCtx`), which is immediately committed.
4. `evm.Call` executes successfully (EVM-level success).
5. `PostTxProcessing` fails → `tmpCtxCommitted = false` → `tmpCtx` is discarded.
6. `commitDurableAuthorization()` is called because `!tmpCtxCommitted`.
7. Committed state now has `A.nonce = 1` and `A.code = delegation(D)` — the delegation is installed — even though the EVM execution that was supposed to accompany it was rolled back.
8. `A`'s account is now delegated to `D` without `A` having consented to a successful EVM execution, and without the gas cost of the EVM execution having been charged (only intrinsic gas was charged). [7](#0-6) [4](#0-3)

// setAuthorizationDelegation writes the nonce bump and delegation code for a
// validated authorization.
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
