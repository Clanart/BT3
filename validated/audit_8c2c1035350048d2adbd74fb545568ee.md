### Title
Unbounded WebSocket Message Size, No HTTP Client Timeout, and Uncapped Response Buffering in `tcpGetAndSendResponse` — (`rpc/websockets.go`)

---

### Summary

`tcpGetAndSendResponse` in `rpc/websockets.go` contains three compounding resource-exhaustion defects reachable by any unauthenticated WebSocket client: (1) no WebSocket read-limit is ever set, so arbitrarily large inbound frames are accepted; (2) the forwarding `http.Client` carries no timeout, so goroutines can be held indefinitely; (3) the backend response is consumed with `io.ReadAll` and no `io.LimitReader`, so the entire response is buffered in heap memory.

---

### Finding Description

**Defect 1 — No WebSocket message-size limit**

`ServeHTTP` upgrades the connection and immediately enters `readLoop`. Neither the `websocket.Upgrader` nor the resulting `*websocket.Conn` ever calls `SetReadLimit`. The gorilla/websocket library's default read limit is `math.MaxInt64` (effectively unlimited). Every frame read at line 268 is accepted regardless of size: [1](#0-0) 

The raw byte slice `mb` is then passed verbatim to `tcpGetAndSendResponse` at lines 287, 303, and 392.

**Defect 2 — `http.Client` with no timeout**

Inside `tcpGetAndSendResponse`, a fresh `http.Client{}` is constructed with zero configuration: [2](#0-1) 

No `Timeout`, no `Transport` with `ResponseHeaderTimeout` or `IdleConnTimeout`. A slow or stalled backend holds the calling goroutine (and the WS connection's read loop) open indefinitely. An attacker who sends many concurrent large requests can exhaust the node's goroutine pool.

**Defect 3 — Uncapped `io.ReadAll` on the response body** [3](#0-2) 

The entire backend response is read into a single `[]byte` with no `io.LimitReader`. For methods that can return large payloads (e.g., `eth_getLogs` over a wide block range, `debug_traceTransaction`), this allocates heap proportional to the response size. Combined with Defect 1 (large request triggers large response) and Defect 2 (no timeout), an attacker can drive unbounded heap growth.

---

### Impact Explanation

All three defects are on the public WS path. The Ethermint WS server runs in the same OS process as the consensus node. A sustained OOM condition kills the process, halting the chain. Even short of OOM, goroutine exhaustion from no-timeout connections degrades or stops block processing. This maps to the allowed High/Critical impact: *public JSON-RPC path exposes a reachable route to node resource exhaustion / chain halt*.

---

### Likelihood Explanation

The WS endpoint is publicly reachable by design (it is the standard Ethereum JSON-RPC WS port). No authentication is required. The attacker needs only a TCP connection and the ability to send a WebSocket frame. No validator collusion, governance action, or privileged key is required. The attack is trivially scriptable.

---

### Recommendation

1. **Set a WebSocket read limit** immediately after upgrading the connection in `ServeHTTP`:
   ```go
   conn.SetReadLimit(4 * 1024 * 1024) // e.g., 4 MB
   ```

2. **Add a timeout to the HTTP client** in `tcpGetAndSendResponse`:
   ```go
   client := &http.Client{Timeout: 30 * time.Second}
   ```

3. **Cap the response body** with `io.LimitReader`:
   ```go
   body, err := io.ReadAll(io.LimitReader(resp.Body, maxResponseBytes))
   ```

4. **Propagate the WS connection context** into `http.NewRequestWithContext` instead of `context.Background()` so that a disconnecting client cancels the in-flight backend request.

---

### Proof of Concept

```python
import websocket, json, os

ws = websocket.create_connection("ws://<node>:8546")

# 50 MB params array
payload = json.dumps({
    "jsonrpc": "2.0",
    "method": "eth_call",
    "params": [{"data": "0x" + "aa" * (50 * 1024 * 1024)}, "latest"],
    "id": 1
})

ws.send(payload)          # accepted: no SetReadLimit
resp = ws.recv()          # goroutine held; heap grows
print(len(resp))
ws.close()
```

Repeat with 100 concurrent connections. Monitor node RSS; assert it does not grow unboundedly and that the node remains responsive to block proposals.
