### Title
Integer Wrapping in `check_time_locks` Bypasses Relative Timelock Conditions When `nowrap=false` - (File: crates/chia-consensus/src/check_time_locks.rs)

### Summary
The `check_time_locks` function uses `wrapping_add` for relative timelock arithmetic when the `nowrap` flag is `false`. A CLVM puzzle that emits `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a near-maximum integer value causes the addition to wrap around to a value smaller than the coin's confirmation timestamp, making the timelock check trivially pass. This is a direct analog to the "unsafe arithmetic producing unexpected values" class described in the external report.

### Finding Description
In `check_time_locks`, the relative timelock checks for `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE` branch on the `nowrap` boolean:

```rust
if let Some(seconds_relative) = spend.seconds_relative {
    if nowrap {
        if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
            return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
        }
    } else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
        return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
    }
}
``` [1](#0-0) 

When `nowrap=false` and `seconds_relative = u64::MAX` with `unspent.timestamp = T`:
- `wrapping_add(T, u64::MAX) = T - 1`
- The guard becomes `timestamp < T - 1`, which is `false` for any current timestamp
- The timelock passes immediately regardless of elapsed time

The same wrapping applies to `height_relative` (`u32::MAX`): [2](#0-1) 

The codebase's own tests explicitly document and confirm this bypass:

```
# 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF), 105, None),
``` [3](#0-2) 

where `None` (no error) is the expected result under `nowrap=False`, confirming the timelock is silently skipped.

The analog to the external report is precise: just as `BestDexLens` performed unsafe arithmetic that produced an unexpected small value from a large input, `wrapping_add` here converts a very large `seconds_relative` into a value smaller than the coin's own confirmation timestamp, inverting the intended comparison.

### Impact Explanation
An attacker who controls a CLVM puzzle can emit `ASSERT_SECONDS_RELATIVE` with value `0xFFFFFFFFFFFFFFFF`. If the full node calls `check_time_locks` with `nowrap=false`, the relative timelock is bypassed — the coin can be spent immediately after confirmation regardless of the intended waiting period. This is a timelock condition validation bypass enabling unauthorized spend acceptance, matching the High impact category: *"timelock or coin-id validation bypass enables unauthorized spend acceptance."*

### Likelihood Explanation
The `nowrap` parameter is caller-controlled and is exposed directly through the Python binding `py_check_time_locks`: [4](#0-3) 

The parameter exists to preserve backward-compatible behavior for pre-fork blocks. Any full-node code path that passes `nowrap=false` — including legacy block validation — is reachable by an unprivileged user who submits a spend bundle containing a puzzle that outputs a near-maximum relative timelock value. No privileged access, key material, or network-level attack is required.

### Recommendation
1. Audit all call sites of `check_time_locks` (and `py_check_time_locks`) to confirm whether `nowrap=false` is ever passed for blocks that accept new spend bundles.
2. If `nowrap=false` is only needed for replaying historical blocks, gate it behind a hard-fork height check so new spends always use `saturating_add`.
3. Consider removing the `nowrap` branch entirely once the transition height is finalized, replacing both paths with `saturating_add` unconditionally.
4. Add a consensus-level test that submits a spend with `ASSERT_SECONDS_RELATIVE = u64::MAX` and asserts it is rejected under all active validation paths.

### Proof of Concept
A CLVM puzzle that returns the condition list `((ASSERT_SECONDS_RELATIVE . 0xFFFFFFFFFFFFFFFF))`. When `check_time_locks` is called with `nowrap=false`:

```
coin confirmed at timestamp T
wrapping_add(T, 0xFFFFFFFFFFFFFFFF) = T - 1
check: current_timestamp < T - 1  →  false  →  no error returned
```

The timelock passes at any block timestamp, allowing the coin to be spent immediately after confirmation. The test vector at line 251 of `tests/test_check_time_locks.py` already demonstrates this outcome with `expected_wrap=None` (pass). [5](#0-4) [6](#0-5)

#[cfg(feature = "py-bindings")]
#[pyfunction]
#[pyo3(name = "check_time_locks")]
#[allow(clippy::needless_pass_by_value)] // pyo3 prefers pass_by_value
pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
