### Title
Fee-on-Transfer Token Accounting Mismatch Breaks Bridge Collateralization — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event and Wormhole message without verifying the actual amount received by the contract. For fee-on-transfer ERC20 tokens, the bridge holds less collateral than it commits to release on NEAR, permanently breaking collateralization.

### Finding Description

In `OmniBridge.initTransfer`, when the token is a plain ERC20 (not a bridge token and not a custom minter), the contract pulls tokens from the caller:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied, not verified post-transfer
);
``` [1](#0-0) 

Immediately after, the function emits the `InitTransfer` event and — in `OmniBridgeWormhole` — publishes a Wormhole message, both carrying the original `amount` value:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // <-- not actual received amount
    fee, nativeFee, recipient, message
);
``` [2](#0-1) 

`OmniBridgeWormhole.initTransferExtension` encodes and publishes the same `amount` to Wormhole: [3](#0-2) 

There is no balance-before / balance-after check anywhere in this path. If the ERC20 token deducts a transfer fee (e.g., USDT with fees enabled, STA, PAXG, or any rebasing/deflationary token), the contract receives `amount - fee_on_transfer` but the cross-chain message claims `amount` was locked.

The same pattern exists for the `customMinters` path: `safeTransferFrom` sends `amount` to the custom minter, then `burn(tokenAddress, amount)` is called with the full `amount` — if the token deducted a fee, the minter holds less than `amount` and the burn may fail or succeed with an inflated figure depending on the minter's implementation. [4](#0-3) 

### Impact Explanation

The NEAR bridge (or any destination chain) reads the committed `amount` from the signed/Wormhole message and releases that full amount to the recipient. The EVM bridge, however, only holds `amount - fee_on_transfer` as collateral. Every such transfer inflates the circulating supply of the bridged token on NEAR relative to the EVM-side reserve. When those tokens are eventually bridged back, the EVM bridge cannot satisfy the redemption, causing permanent loss of funds for later redeemers — a direct bridge collateralization break.

**Allowed impact matched:** *High — Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value.*

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 token address that is already registered (i.e., has a NEAR mapping). Fee-on-transfer tokens are a well-known ERC20 variant; USDT has the fee mechanism in its contract (currently set to zero but togglable by its owner). A single transfer with such a token is sufficient to create an undercollateralized position. No privileged access is required.

### Recommendation

Capture the contract's token balance before and after the `safeTransferFrom` call and use the delta as the canonical bridged amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "zero received");
// use actualReceived (cast to uint128) in place of amount for the event and Wormhole message
```

Apply the same pattern to the `customMinters` path (check the minter's balance delta before calling `burn`). Additionally, document which token properties (no transfer fees, no rebasing) are required for a token to be safely bridgeable, and enforce this at token-registration time.

### Proof of Concept

1. A fee-on-transfer ERC20 token `FOT` (1% fee) is registered in the bridge with a NEAR mapping.
2. Alice calls `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
3. `safeTransferFrom` moves 1000 FOT from Alice; the contract receives 990 FOT (1% fee deducted by the token).
4. The `InitTransfer` event (and Wormhole message in `OmniBridgeWormhole`) records `amount = 1000`.
5. The NEAR bridge processes the message and mints/releases 1000 FOT-equivalent tokens to `alice.near`.
6. Alice now holds 1000 NEAR-side tokens backed by only 990 EVM-side tokens.
7. Repeating this 100 times drains the EVM reserve entirely while NEAR-side supply remains inflated.
8. The last redeemer bridging back from NEAR to EVM finds the bridge insolvent and cannot withdraw their funds.

    function initTransferExtension(
        address sender,
        address tokenAddress,
        uint64 originNonce,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message,
        uint256 value
    ) internal override {
        bytes memory payload = bytes.concat(
            bytes1(uint8(MessageType.InitTransfer)),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(sender),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(tokenAddress),
            Borsh.encodeUint64(originNonce),
            Borsh.encodeUint128(amount),
            Borsh.encodeUint128(fee),
            Borsh.encodeUint128(nativeFee),
            Borsh.encodeString(recipient),
            Borsh.encodeString(message)
        );
        // slither-disable-next-line reentrancy-eth
        _wormhole.publishMessage{value: value}(
            wormholeNonce,
            payload,
            _consistencyLevel
        );

        wormholeNonce++;
