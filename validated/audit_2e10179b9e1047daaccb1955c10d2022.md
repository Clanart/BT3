### Title
Missing Zero Address Check for `nearBridgeDerivedAddress_` in `OmniBridge.initialize()` Freezes All Bridge Signature Verification — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initialize()` accepts `nearBridgeDerivedAddress_` without any zero address guard. This address is the sole ECDSA verification anchor for every cross-chain message (`finTransfer`, `deployToken`). If it is mistakenly set to `address(0)`, every subsequent call to those functions reverts with `InvalidSignature`, permanently blocking settlement of in-flight user transfers. The companion setter `setNearBridgeDerivedAddress()` carries the same omission.

---

### Finding Description

`OmniBridge.initialize()` stores three constructor-time parameters with no validation:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 72-86
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,   // ← no require(... != address(0))
    uint8 omniBridgeChainId_
) public initializer {
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;   // stored as-is
    ...
}
```

`nearBridgeDerivedAddress` is then used as the recovered-signer check in both critical paths:

```solidity
// finTransfer — line 311
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress)
    revert InvalidSignature();

// deployToken — line 151
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress)
    revert InvalidSignature();
```

`ECDSA.recover` (OpenZeppelin ≥ 4.8) always returns a non-zero address for a well-formed signature, so the inequality `someAddress != address(0)` is always `true`, causing both functions to unconditionally revert.

The post-deployment setter carries the same flaw:

```solidity
// lines 568-572
function setNearBridgeDerivedAddress(
    address nearBridgeDerivedAddress_
) external onlyRole(DEFAULT_ADMIN_ROLE) {
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;  // no zero check
}
```

---

### Impact Explanation

`nearBridgeDerivedAddress` is the only trust anchor for MPC-signed cross-chain messages. Setting it to `address(0)` at initialization (or later via the setter) causes:

- **`finTransfer()` always reverts** — users who have already locked or burned tokens on the source chain (NEAR, Solana, StarkNet) cannot receive the corresponding assets on the EVM side. Their source-chain funds are irrecoverably locked for the duration of the freeze.
- **`deployToken()` always reverts** — no new bridged token contracts can be deployed, blocking the entire token-deployment flow.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

While an admin can call `setNearBridgeDerivedAddress()` to restore a correct value, that setter itself has no zero-address guard, and any in-flight transfers whose source-chain funds were already committed during the freeze window remain at risk until the fix is applied and the relayer retries.

---

### Likelihood Explanation

Deployment of UUPS proxies typically involves a one-shot initializer call, often from a deployment script. A copy-paste error, wrong argument order, or a script bug that passes `address(0)` for `nearBridgeDerivedAddress_` is a realistic operational mistake. The `initializer` modifier prevents re-initialization, so there is no recovery path other than the admin setter — which itself lacks the same guard.

---

### Recommendation

Add explicit zero-address guards in both `initialize()` and `setNearBridgeDerivedAddress()`:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    require(tokenImplementationAddress_ != address(0), "zero tokenImpl");
    require(nearBridgeDerivedAddress_ != address(0), "zero nearBridgeAddr");
    ...
}

function setNearBridgeDerivedAddress(
    address nearBridgeDerivedAddress_
) external onlyRole(DEFAULT_ADMIN_ROLE) {
    require(nearBridgeDerivedAddress_ != address(0), "zero nearBridgeAddr");
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;
}
```

Apply the same pattern to `OmniBridgeWormhole.initializeWormhole()` for `wormholeAddress`.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` proxy and call `initializeWormhole` with `nearBridgeDerivedAddress = address(0)`.
2. A user on NEAR locks 1000 USDC and the MPC produces a valid `finTransfer` signature.
3. Relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
4. Inside `finTransfer`, `ECDSA.recover(hashed, signatureData)` returns the legitimate MPC address (e.g. `0xABCD…`).
5. Check: `0xABCD… != address(0)` → `true` → `revert InvalidSignature()`.
6. The call reverts; the user's NEAR-side tokens remain locked; the EVM-side nonce is never consumed, but the user cannot claim funds until an admin corrects the address.
