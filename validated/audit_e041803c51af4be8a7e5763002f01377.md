### Title
`MerkleBlob::batch_insert` Silently Accepts Duplicate Keys, Corrupting Tree Root and Invalidating Proofs - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a key that already exists in the tree (or a repeated key within the batch) is supplied to the bulk path of `batch_insert`, the duplicate leaf is written directly into the blob without error. The `BlockStatusCache` is silently overwritten to point to the new leaf, leaving the original leaf as an unreachable "ghost" node that still contributes to the computed root hash. The resulting root hash is therefore incorrect, and any proof of inclusion generated from it is invalid.

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against both duplicate keys and duplicate hashes before writing anything:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` uses `insert` only for the first two items when the tree has ≤ 1 existing leaf. All remaining items are written through a fast bulk path that calls `insert_entry_to_blob` directly, with no duplicate check at all:

```rust
for ((key, value), hash) in keys_values_hashes {   // no guard here
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

When a duplicate key reaches this path:

1. A second `LeafNode` with the same `KeyId` is written into the blob at a new index.
2. `insert_entry_to_blob` updates `BlockStatusCache`, so `key_to_index` is overwritten to point to the new (duplicate) leaf; the original leaf's index is no longer tracked.
3. The original leaf remains in the blob, is wired into the tree structure, and is included in every subsequent `calculate_lazy_hashes` traversal — so it contributes to the root hash.
4. The root hash now reflects a tree containing a key that appears twice, but the cache only knows about one of them.
5. `get_proof_of_inclusion` generates a proof against the cache-tracked leaf; that proof verifies against a root that was computed over a different (larger) tree.
6. The ghost leaf can never be deleted because `delete` looks up the key in the cache and finds only the duplicate.

The same flaw applies to duplicate hashes within a batch and to keys that collide with keys already present in a tree that has ≥ 2 leaves before `batch_insert` is called.

### Impact Explanation

The DataLayer Merkle tree root is committed on-chain. A corrupted root — one that silently includes a ghost leaf — means:

- Proofs of inclusion generated by `get_proof_of_inclusion` do not verify against the committed root, because the proof path was computed over a subtree that excludes the ghost leaf while the root was computed over a tree that includes it.
- A verifier that recomputes the root from a proof will obtain a different hash than the one stored on-chain, causing proof rejection or, if the verifier trusts the proof, acceptance of a proof for an incorrect state.
- The ghost leaf permanently inflates the tree, making the committed state diverge from the logical key-value set.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`batch_insert` is exposed as a public Python binding (`py_batch_insert`) and is the primary bulk-insertion API used by the DataLayer. Any caller that does not deduplicate its input before calling `batch_insert` — including callers processing data received from the network — can trigger the corruption. No privileged role is required; the attacker only needs to influence the set of key-value pairs submitted to a DataLayer store.

### Recommendation

Add the same duplicate guards to the bulk path of `batch_insert` that `insert` already enforces:

- Before writing each leaf in the `for ((key, value), hash) in keys_values_hashes` loop, check `self.block_status_cache.contains_key(key)` and `self.block_status_cache.contains_leaf_hash(&hash)`, returning `Err(Error::KeyAlreadyPresent())` or `Err(Error::HashAlreadyPresent())` respectively.
- Alternatively, deduplicate the input vector before entering the bulk path, or delegate all insertions through `insert` regardless of batch size.

A regression test that calls `batch_insert` with a key already present in a ≥ 2-leaf tree and asserts an error is returned should be added alongside the fix.

### Proof of Concept

```
1. Create a MerkleBlob and insert 3 distinct keys (K1, K2, K3) so leaf_count = 3.
2. Call batch_insert with [(K1, V_new, H_new)] — K1 already exists.
3. batch_insert enters the bulk path (leaf_count > 1), writes a second LeafNode
   for K1 at a new index, overwrites key_to_index[K1] in the cache.
4. Call calculate_lazy_hashes(); the traversal visits both K1 leaves, so the
   root hash is computed over 4 leaves.
5. Call get_proof_of_inclusion(K1); the proof is built from the cache-tracked
   (new) K1 leaf and its ancestors.
6. Verify the proof: proof.valid() returns false because the proof's root_hash
   is derived from a 3-leaf subtree path, while the actual root was computed
   over 4 leaves.
7. Observe that delete(K1) removes only the cache-tracked leaf; the ghost K1
   leaf remains in the blob and continues to affect the root hash.
