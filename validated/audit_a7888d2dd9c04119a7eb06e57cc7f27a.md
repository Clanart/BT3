## Title
**Bank Balance Flushed Without EVM Account Commit After createObjectChange Revert Clears `journal.dirties`** — (`x/evm/statedb/statedb.go`, `x/evm/statedb/journal.go`, `x/evm/statedb/native.go`)

---

## Summary

`nativeChange.Dirtied()` unconditionally returns `nil`, so `ExecuteNativeAction` (which backs `AddBalance`/`Transfer`) never adds the target address to `journal.dirties`. If a subsequent `createObjectChange` for the same address is later reverted, the revert logic decrements `dirties[addr]` to zero and deletes it from the map entirely. `Commit()` then skips the address in its account-commit loop while `flushNativeCacheLayers()` unconditionally flushes the bank balance change — leaving a committed bank balance with no corresponding EVM account.

---

## Finding Description

**`nativeChange.Dirtied()` returns `nil`:** [1](#0-0) 

This means `journal.append(nativeChange{...})` inside `ExecuteNativeAction` never increments `dirties[addr]` for the target address.

**`createObjectChange.Dirtied()` returns the account address:** [2](#0-1) 

So `CreateAccount(A)` sets `dirties[A] = 1`.

**`journal.Revert()` deletes the address when count reaches zero:** [3](#0-2) 

Reverting the `createObjectChange` decrements `dirties[A]` to 0 and removes A from the map entirely, because the prior `nativeChange` never contributed a count.

**`createObjectChange.Revert()` also deletes from `stateObjects`:** [4](#0-3) 

**`Commit()` skips addresses not in `sortedDirties()`:** [5](#0-4) 

**`flushNativeCacheLayers()` runs unconditionally before the account-commit loop:** [6](#0-5) [7](#0-6) 

The flush writes all native cache layers (including A's bank balance change) to the parent store regardless of whether A is in `sortedDirties()`.

## Exact Call Sequence

```
journal.entries = []
dirties = {}

1. AddBalance(A, amount)  →  ExecuteNativeAction  →  nativeChange appended
   nativeChange.Dirtied() = nil  →  dirties unchanged = {}

2. Snapshot(S1)  →  validRevisions[{id:1, journalIndex:1}]

3. CreateAccount(A)  →  createObjectChange appended
   createObjectChange.Dirtied() = &A  →  dirties[A] = 1
   stateObjects[A] = newObject(...)

4. RevertToSnapshot(S1)  →  journal.Revert(s, snapshot=1)
   i=1: createObjectChange.Revert()  →  delete(stateObjects, A)
        Dirtied()=&A  →  dirties[A]-- = 0  →  delete(dirties, A)
   journal.entries = [nativeChange]
   dirties = {}   ← A is gone
   stateObjects = {}  ← A is gone

5. Commit()
   sortedDirties() = []  →  account-commit loop: nothing for A
   flushNativeCacheLayers()  →  A's bank balance change IS written to parent store
   keeper.SetAccount(A, ...) is NEVER called
```

**Result:** A has a non-zero bank balance in the Cosmos bank module but no EVM account (`keeper.GetAccount(A) == nil`).

---

## Impact Explanation

- `eth_getBalance(A)` → non-zero (reads from bank module, which was flushed)
- `eth_getNonce(A)` → 0 (no EVM account)
- `eth_getCode(A)` → empty (no EVM account)
- `keeper.GetAccount(A)` → nil

The bank balance is permanently committed without a corresponding EVM account. Funds are mis-accounted: they exist in the bank but the EVM has no record of the account owning them. Any subsequent EVM operation that tries to debit A's balance will fail to find the account, and the funds become inaccessible through normal EVM paths.

This matches the **High** impact: *"EVM state transition... bug that permits... valid user funds/fees to be mis-accounted."*

---

## Likelihood Explanation

Reachable through a crafted smart contract submitted by an unprivileged attacker:

1. Contract C sends ETH to address A via a value-bearing `CALL` → `Transfer(C, A, v)` → `AddBalance(A, v)` → `nativeChange` (Dirtied=nil)
2. Contract C then attempts `CREATE2` targeting address A (e.g., A is a deterministic address with a pre-funded balance)
3. The EVM takes a snapshot, calls `CreateAccount(A)`, then executes init code that reverts (e.g., intentional `REVERT` opcode)
4. `RevertToSnapshot` removes A from `dirties` and `stateObjects`
5. `Commit()` flushes A's balance but skips A's account

This is a standard EVM execution pattern (pre-fund + failed CREATE2) that any contract can trigger.

---

## Recommendation

`nativeChange` must contribute to `journal.dirties` for the affected address. The simplest fix is to make `nativeChange` track the address(es) it modifies and return them from `Dirtied()`, so that a native balance change always keeps the address in `dirties` independently of any EVM object changes. Alternatively, `Commit()` should independently enumerate all addresses with pending native cache layer changes (not just `sortedDirties()`) and ensure each has a committed EVM account before flushing.

```go
// Unit test sketch (x/evm/statedb package)
func TestBankBalanceWithoutAccount(t *testing.T) {
    db := statedb.New(ctx, keeper, txConfig)
    addrA := common.HexToAddress("0xAAAA...")

    // Step 1: native balance change for A (nativeChange, Dirtied=nil)
    db.AddBalance(addrA, uint256.NewInt(1e18), tracing.BalanceChangeUnspecified)

    // Step 2: snapshot
    snap := db.Snapshot()

    // Step 3: create account for A
    db.CreateAccount(addrA)

    // Step 4: revert — removes A from dirties and stateObjects
    db.RevertToSnapshot(snap)

    // Step 5: commit
    require.NoError(t, db.Commit())

    // Invariant violation: bank balance non-zero, EVM account nil
    balance := keeper.GetBalance(ctx, sdk.AccAddress(addrA.Bytes()), evmDenom)
    account := keeper.GetAccount(ctx, addrA)
    require.True(t, balance.IsPositive(), "bank balance should be non-zero")
    require.Nil(t, account, "EVM account should not exist — but bank balance was committed")
    // This assert fires: funds exist in bank with no EVM account
}
```
