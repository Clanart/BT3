### Title
Finalization Record Removed on `ft_transfer_call` Failure Enables Cross-Chain Replay / Double-Finalization — (`near/omni-bridge/src/lib.rs`)

### Summary

When a NEAR-bound `fin_transfer` uses `ft_transfer_call` (triggered by a non-empty `msg` field) and the recipient contract rejects the tokens (returns the full amount from `ft_on_transfer`), the bridge removes the finalization record from `finalised_transfers`. This allows the same source-chain proof to be submitted a second time, causing the bridge to release tokens again for a single source-chain burn/lock event — a cross-chain double-finalization.

### Finding Description

The NEAR bridge contract tracks completed inbound transfers in `finalised_transfers: LookupSet<TransferId>`, where `TransferId = {origin_chain, origin_nonce}`. This set is the sole replay-protection gate for `fin_transfer`.

In `process_fin_transfer_to_near`, the finalization record is inserted first via `add_fin_transfer`, then tokens are sent to the recipient via `send_tokens`. When the transfer has a non-empty `msg`, `send_tokens` uses `ft_transfer_call`, which invokes `ft_on_transfer` on the recipient contract. The result is checked in `fin_transfer_send_tokens_callback`: [1](#0-0) 

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    self.burn_tokens_if_needed(token.clone(), ...);
    self.revert_lock_actions(&lock_actions);
    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
    env::log_str(&OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string());
}
```

`is_refund_required` returns `true` when the `ft_transfer_call` promise result deserializes to `U128(0)` — meaning the recipient's `ft_on_transfer` returned the full amount (all tokens unused/refunded): [2](#0-1) 

`remove_fin_transfer` unconditionally removes the `TransferId` from `finalised_transfers`: [3](#0-2) 

After removal, `add_fin_transfer` — which is the only guard against replay — will succeed again for the same `TransferId`: [4](#0-3) 

The `FailedFinTransferEvent` emitted after removal signals to relayers that the transfer failed and should be retried, making the replay a natural consequence of normal relayer behavior.

### Impact Explanation

**Impact: High — Cross-chain replay / double-finalization enabling double-spend.**

An attacker can receive tokens on NEAR twice for a single burn/lock event on the source chain (EVM, Solana, StarkNet). For deployed (bridged) tokens, the bridge mints tokens twice. For native tokens locked in the bridge, the bridge releases locked collateral twice, breaking bridge collateralization. The source chain's locked/burned supply is only debited once, creating unbacked supply on NEAR.

### Likelihood Explanation

**Likelihood: Low.**

The attack requires:
1. The attacker controls the recipient NEAR account (a contract they deploy).
2. The original `initTransfer` on the source chain includes a non-empty `message` field (attacker-controlled).
3. The attacker's contract returns the full token amount from `ft_on_transfer` on the first call, then accepts on the second.
4. A trusted relayer retries the transfer after observing `FailedFinTransferEvent` (standard relayer behavior).

Conditions 1–3 are fully attacker-controlled. Condition 4 is a realistic expectation of automated relayer infrastructure. The `#[trusted_relayer]` gate on `fin_transfer` means the attacker cannot directly replay without relayer cooperation, but the `FailedFinTransferEvent` is designed to trigger relayer retries, making this a realistic attack path.

### Recommendation

Do not remove the finalization record on `ft_transfer_call` failure. Instead, keep the `TransferId` in `finalised_transfers` permanently. If the token transfer to the recipient fails, the relayer or recipient should use a separate recovery mechanism (e.g., a claimable escrow) rather than re-opening the replay window. The finalization record must be treated as permanent once inserted.

### Proof of Concept

1. Attacker deploys a malicious NEAR contract `attacker.near` whose `ft_on_transfer` returns the full `amount` on the first call and `0` on the second call.
2. Attacker calls `initTransfer` on EVM with `recipient = "attacker.near"` and `message = "x"` (non-empty). EVM burns/locks `N` tokens.
3. Trusted relayer submits the EVM proof to NEAR `fin_transfer`.
4. `fin_transfer_callback` → `process_fin_transfer_to_near` → `add_fin_transfer` inserts `{Eth, nonce}` → `send_tokens` calls `ft_transfer_call` on `attacker.near`.
5. `attacker.near::ft_on_transfer` returns `U128(N)` (full refund). Tokens are returned to bridge.
6. `fin_transfer_send_tokens_callback`: `is_refund_required` = `true` → `remove_fin_transfer` removes `{Eth, nonce}` from `finalised_transfers`. `FailedFinTransferEvent` is emitted.
7. Relayer observes `FailedFinTransferEvent` and retries with the same proof.
8. `fin_transfer_callback` → `add_fin_transfer` succeeds (record was removed) → `send_tokens` calls `ft_transfer_call` on `attacker.near`.
9. `attacker.near::ft_on_transfer` returns `U128(0)` (accepts tokens). Attacker receives `N` tokens.
10. Attacker has received `N` tokens on NEAR; the source chain burned/locked `N` tokens only once. Net gain: `N` unbacked tokens. [5](#0-4) [6](#0-5) [7](#0-6)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
    }

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
