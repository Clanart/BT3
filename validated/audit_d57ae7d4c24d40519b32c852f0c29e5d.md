### Title
Temporally Misaligned L1 Gas Price and ETH→STRK Rate Combination Produces Wrong FRI Gas Prices in Block Header - (`crates/apollo_consensus_orchestrator/src/utils.rs`)

### Summary

`get_l1_prices_in_fri_and_wei_and_conversion_rate` in `crates/apollo_consensus_orchestrator/src/utils.rs` concurrently fetches two independent price feeds — the L1 gas price in wei (from the L1 scraper, lagged by `lag_margin_seconds = 600s`) and the ETH→STRK conversion rate (from an external oracle, quantized by `lag_interval_seconds = 900s`) — and multiplies them together without any validation that they reflect the same time window. The resulting `l1_gas_price_fri` and `l1_data_gas_price_fri` values are committed into the `ProposalInit` block header, hashed into the block hash via `gas_prices_to_hash`, and used for all transaction fee calculations in the block.

### Finding Description

In `get_l1_prices_in_fri_and_wei_and_conversion_rate`:

```rust
let (eth_to_fri_rate, price_info) = tokio::join!(
    l1_gas_price_provider_client.get_eth_to_fri_rate(timestamp),
    l1_gas_price_provider_client.get_price_info(BlockTimestamp(timestamp))
);
```

Both calls receive the same `timestamp`, but each applies its own independent lag:

**Source 1 — L1 gas price in wei** (`get_price_info`):
Selects L1 blocks whose timestamp ≤ `timestamp − lag_margin_seconds`. In production (`l1_gas_price_provider_config.json`): `lag_margin_seconds = 600s`. The returned price is the mean of the last 300 L1 blocks ending ~10 minutes before the L2 block timestamp.

**Source 2 — ETH→STRK rate** (`eth_to_fri_rate` in `EthToStrkOracleClient`):
```rust
let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
    .checked_div(self.config.lag_interval_seconds)
    .expect("lag_interval_seconds should be non-zero");
```
In production: `lag_interval_seconds = 900s`. The oracle queries the rate for the interval `[quantized_timestamp × 900, (quantized_timestamp+1) × 900)`, which corresponds to approximately `timestamp − 1800s` to `timestamp − 900s` — i.e., 15–30 minutes before the L2 block timestamp.

The two sources therefore reflect **different time windows** with a structural gap of at least 300 seconds (5 minutes) and up to 900+ seconds (15+ minutes). They are multiplied together without any cross-validation:

```rust
let l1_gas_prices_fri_result =
    L1PricesInFri::convert_from_wei(&prices_in_wei, eth_to_fri_rate);
```

The resulting `l1_gas_price_fri` and `l1_data_gas_price_fri` are placed into `ProposalInit` and flow through `convert_to_sn_api_block_info` into the `BlockInfo` used for execution. They are also committed into the block hash via `PartialBlockHashComponents::new` → `calculate_block_hash` → `gas_prices_to_hash`, which hashes all six gas price fields (wei and fri for l1, l1_data, l2) into `gas_prices_hash`.

The validator in `is_block_info_valid` recomputes the same misaligned combination using the same two sources, so it accepts the wrong prices without detecting the misalignment.

### Impact Explanation

During volatile ETH/STRK price periods, the temporal gap between the two sources causes the computed FRI gas price to diverge from the true contemporaneous value:

- If STRK is appreciating against ETH (ETH→STRK rate rising), the stale oracle rate underestimates the true rate, causing `l1_gas_price_fri` to be **understated**. Users pay less than the correct FRI fee; the sequencer under-collects fees.
- If STRK is depreciating (ETH→STRK rate falling), the stale oracle rate overestimates the true rate, causing `l1_gas_price_fri` to be **overstated**. Users overpay FRI fees.

The wrong FRI gas prices are:
1. Committed into the block hash (`gas_prices_hash` field) via `calculate_block_hash` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs`.
2. Used by the blockifier for all transaction fee calculations in the block.
3. Returned by `starknet_estimateFee` and `starknet_simulateTransactions` RPC calls, which read gas prices from the pending block info.
4. Stored in `StorageBlockHeader.l1_gas_price` and `l1_data_gas_price` (the `price_in_fri` fields), persisted to the storage layer.

This matches: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact** and **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**.

### Likelihood Explanation

The misalignment is **structural and always present** — it occurs on every block proposal because the two lag parameters are independently configured with different values (600s vs 900s). No special conditions or attacker action are required. The impact magnitude scales with ETH/STRK price volatility. During periods of high volatility (e.g., a 5–10% ETH/STRK move within 15 minutes, which is not uncommon), the FRI gas price error can be 5–10% or more. The validator accepts the wrong prices because it recomputes the same misaligned combination.

### Recommendation

Align the effective timestamps of the two data sources before combining them. Concretely:

1. **Expose the effective timestamp** from each source: `get_price_info` should return the timestamp of the last L1 block included in the mean; `eth_to_fri_rate` should return the `adjusted_timestamp` (`quantized_timestamp × lag_interval_seconds`) it actually queried.
2. **Validate temporal proximity**: after fetching both values, assert that `|t_gas_price − t_eth_strk_rate| ≤ max_allowed_skew`. If the skew exceeds the threshold, fall back to the previous block info rather than combining misaligned data.
3. **Alternatively**, unify the lag parameters: set `lag_margin_seconds` equal to `lag_interval_seconds` so both sources reflect the same time window, and document this coupling as a configuration invariant.

### Proof of Concept

**Setup**: Production config — `lag_margin_seconds = 600s`, `lag_interval_seconds = 900s`.

**Scenario**: At L2 block timestamp `T = 1_000_000`:

- `get_price_info(T)` returns the mean L1 gas price over blocks with timestamp ≤ `T − 600 = 999_400`. Suppose this yields `base_fee = 30 gwei`.
- `eth_to_fri_rate(T)` computes `quantized_timestamp = (T − 900) / 900 = 1110`, then queries the oracle for `adjusted_timestamp = 1110 × 900 = 999_000`. This reflects the ETH/STRK rate at `T − 1000s` (≈17 minutes ago). Suppose the rate at `T − 1000s` was `R_old`, while the current rate at `T − 600s` is `R_new = 1.08 × R_old` (STRK appreciated 8% in 7 minutes during a rally).

- **Computed** `l1_gas_price_fri = 30 gwei × R_old`
- **Correct** `l1_gas_price_fri = 30 gwei × R_new = 30 gwei × 1.08 × R_old`

The block header commits a FRI gas price that is **8% below the correct value**. All transactions in the block pay 8% less than the correct FRI fee. The block hash includes this wrong `gas_prices_hash`. `starknet_estimateFee` returns the wrong (8% low) FRI fee estimate to users. The validator accepts the proposal because it recomputes the same 8%-low value from the same misaligned sources.
