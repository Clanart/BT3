Audit Report

## Title
Unvalidated `pool` Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Caller-Approved Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address in every public entry-point without verifying it against the factory registry. Because the callback's only caller-identity check is `msg.sender == expectedPool`, and the malicious pool *is* the stored expected pool, a malicious contract can fire `metricOmmModifyLiquidityCallback` with attacker-controlled token addresses and drain up to `maxAmountToken0`/`maxAmountToken1` of any ERC-20 tokens from the victim's wallet via `safeTransferFrom`.

## Finding Description

Every public `addLiquidityExactShares` and `addLiquidityWeighted` overload forwards the caller-supplied `pool` directly to `_addLiquidity` with no factory validation: [1](#0-0) 

`_setPayContext` stores the unvalidated `pool` as the authoritative callback caller in transient storage, then immediately calls `pool.addLiquidity(...)`. The NatSpec at lines 19–21 explicitly acknowledges this omission: [2](#0-1) 

In `metricOmmModifyLiquidityCallback`, the only caller check is `msg.sender != expectedPool`: [3](#0-2) 

This check passes trivially for a malicious pool because it *is* the stored expected pool. The callback then calls `IMetricOmmPool(msg.sender).getImmutables()` to obtain `token0`/`token1`, which the malicious pool controls entirely: [4](#0-3) 

`pay` then issues `safeTransferFrom(payer, maliciousPool, amount)` for whichever tokens the malicious pool names: [5](#0-4) 

By contrast, `MetricOmmSwapRouterBase._setNextCallbackContext` always calls `_requireFactoryPool(pool)` before storing the callback context: [6](#0-5) 

And `_requireExpectedCallbackCaller` re-validates against the factory on every callback: [7](#0-6) 

The router path is safe; the liquidity adder path is not. The `MaxAmountExceeded` guard only bounds the amount, not the token identity, so the malicious pool can substitute arbitrary high-value tokens.

## Impact Explanation

Direct loss of user principal. A malicious pool can drain up to `maxAmountToken0` of any ERC-20 and `maxAmountToken1` of any other ERC-20 from the caller's wallet, provided the caller has approved those tokens to `MetricOmmPoolLiquidityAdder`. The victim receives zero liquidity. This is a Critical/High direct loss of user funds with no recovery path, meeting Sherlock contest thresholds.

## Likelihood Explanation

The attack requires a user to call `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address while having pre-approved tokens to `MetricOmmPoolLiquidityAdder`. Achievable via: (1) social engineering / phishing with a contract that looks like a legitimate pool; (2) a compromised front-end substituting the pool address; (3) user error. Because `MetricOmmPoolLiquidityAdder` is a shared periphery contract that users approve tokens to in advance, the attack surface is persistent across all users with outstanding approvals.

## Recommendation

Add a factory validation check at the top of `_addLiquidity`, mirroring the pattern in `MetricOmmSwapRouterBase`:

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires storing the factory address as an immutable in `MetricOmmPoolLiquidityAdder`, exactly as `MetricOmmSwapRouterBase` stores `IMetricOmmPoolFactory internal immutable FACTORY`. [8](#0-7) 

## Proof of Concept

1. Attacker deploys `MaliciousPool` implementing `IMetricOmmPoolActions.addLiquidity` and `IMetricOmmPool.getImmutables`:
   - `getImmutables()` returns `token0 = WBTC`, `token1 = LINK` (tokens the victim holds and has approved).
   - `addLiquidity(...)` immediately calls back `metricOmmModifyLiquidityCallback(max0, max1, abi.encode(KIND_PAY))`.

2. Victim has approved `MetricOmmPoolLiquidityAdder` for WBTC and LINK.

3. Attacker tricks victim into calling:
   ```solidity
   liquidityAdder.addLiquidityExactShares(
       maliciousPool, victim, 0, deltas, 1e8 /*1 WBTC*/, 1000e18 /*1000 LINK*/, ""
   );
   ```

4. `_addLiquidity` stores `maliciousPool` as expected callback caller (no factory check) and calls `maliciousPool.addLiquidity(...)`.

5. `MaliciousPool.addLiquidity` calls back `metricOmmModifyLiquidityCallback(1e8, 1000e18, abi.encode(KIND_PAY))`.

6. Callback passes `msg.sender == expectedPool`. `getImmutables()` returns WBTC/LINK. `pay(WBTC, victim, maliciousPool, 1e8)` and `pay(LINK, victim, maliciousPool, 1000e18)` execute via `safeTransferFrom`.

7. Attacker receives 1 WBTC + 1000 LINK from victim's wallet. Victim receives zero liquidity.
