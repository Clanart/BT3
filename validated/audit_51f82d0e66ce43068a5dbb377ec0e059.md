### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary, user-controlled `class_hash` from the syscall request without verifying that the hash corresponds to a class that has been declared on-chain. An unprivileged contract can call `replace_class` with any felt value, permanently corrupting its own `StateEntry.class_hash` to an undeclared value. Any subsequent call to that contract will fail at the OS level because no executable class exists for the stored hash, permanently freezing all funds held by the contract.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the OS reads `class_hash` directly from the user-supplied syscall request buffer and writes it unconditionally into the contract's `StateEntry` via `dict_update`:

```cairo
let class_hash = request.class_hash;   // line 896 — fully user-controlled

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The inline TODO comment at line 898 explicitly acknowledges that the missing check — "Check that there is a declared contract class with the given hash" — has not been implemented. No validation against `contract_class_changes` (the dict tracking declared classes) is performed anywhere in this function or in its call site in `execute_syscalls` (line 195–203 of `execute_syscalls.cairo`).

This is the direct analog of the LiFi bug: just as LiFi facets accepted a user-supplied `router` address and called `maxApproveERC20(token, router, amount)` without restricting it to known bridge addresses, the StarkNet OS accepts a user-supplied `class_hash` and writes it into the contract state without restricting it to declared classes.

The `StateEntry.class_hash` field is the sole pointer used by the OS to locate executable code for a contract. Once it is set to an undeclared value, the contract becomes permanently unexecutable. The state commitment machinery in `commitment.cairo` (`get_contract_state_hash`, lines 51–71; `hash_contract_state_changes`, lines 148–206) will faithfully commit the corrupted hash into the global state root, making the corruption permanent and provable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is replaced with an undeclared value:

1. The corrupted `StateEntry` is committed into the Patricia-Merkle contract state tree (`compute_contract_state_commitment`, `commitment.cairo` lines 76–111).
2. Every future transaction targeting that contract address will fail at the OS execution layer because no class body can be loaded for the stored hash.
3. All ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — there is no recovery path once the state root is finalized.

---

### Likelihood Explanation

**Medium.**

The attacker-controlled entry path is straightforward:

- Any unprivileged transaction sender who owns or controls a contract (including one they just deployed) can issue a `replace_class` syscall with an arbitrary felt as the new class hash.
- No special privilege, leaked key, or operator cooperation is required.
- The only prerequisite is that the attacker controls a contract (trivially satisfied by deploying one).

A realistic attack scenario: an attacker deploys a contract, convinces users to deposit funds into it (e.g., as a fake vault or bridge), then calls `replace_class(0xdeadbeef)` in the same or a subsequent transaction, permanently freezing all deposited funds.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (the dict of declared classes for the current block) or in the pre-existing class commitment tree. Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled class hash is non-zero. This is exactly what the existing TODO comment calls for and what the `execute_declare_transaction` path already enforces via `dict_update{dict_ptr=contract_class_changes}(key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash)`.

---

### Proof of Concept

1. Deploy contract `C` holding user funds.
2. From `C`, issue a `replace_class` syscall with `class_hash = 0x1` (any undeclared felt).
3. The OS executes `execute_replace_class(contract_address=C)`:
   - Line 896: `class_hash = 0x1` (from request, user-controlled).
   - Line 898: TODO check is absent — no validation performed.
   - Lines 902–910: `StateEntry` for `C` is updated with `class_hash=0x1`.
4. The block is proven; `commitment.cairo` commits the corrupted state root.
5. Any subsequent `call_contract` or `invoke` targeting `C` fails — the OS cannot load a class for hash `0x1`.
6. All funds in `C`'s storage are permanently frozen.

func get_contract_state_hash{hash_ptr: HashBuiltin*}(
    class_hash: felt, storage_root: felt, nonce: felt
) -> (hash: felt) {
    const CONTRACT_STATE_HASH_VERSION = 0;
    if (class_hash == UNINITIALIZED_CLASS_HASH) {
        if (storage_root == 0) {
            if (nonce == 0) {
                return (hash=0);
            }
        }
    }

    // Set res = H(H(class_hash, storage_root), nonce).
    let (hash_value) = hash2(class_hash, storage_root);
    let (hash_value) = hash2(hash_value, nonce);

    // Return H(hash_value, CONTRACT_STATE_HASH_VERSION). CONTRACT_STATE_HASH_VERSION must be in the
    // outermost hash to guarantee unique "decoding".
    let (hash) = hash2(hash_value, CONTRACT_STATE_HASH_VERSION);
    return (hash=hash);
}
