### Title
Integer Wrapping in Relative Timelock Arithmetic Enables Timelock Bypass and Consensus Divergence - (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

`check_time_locks` performs relative timelock arithmetic using either `wrapping_add` or `saturating_add` depending on a `nowrap` boolean parameter. When `nowrap=false` (the pre-fork path), an attacker-controlled CLVM condition value of `0xffff_ffff` (for `ASSERT_HEIGHT_RELATIVE`) or `0xffff_ffff_ffff_ffff` (for `ASSERT_SECONDS_RELATIVE`) causes the addition to wrap around to a value smaller than the coin's confirmation height/timestamp, making the timelock check pass when it must fail. The inverse wrapping occurs for `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE`, causing a spend that should be accepted to be rejected. Any node evaluating the same spend bundle with a different `nowrap` value reaches the opposite validity conclusion, producing deterministic consensus divergence.

### Finding Description

In `check_time_locks`, the relative timelock checks branch on `nowrap`:

```rust
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { return Err(...AssertHeightRelativeFailed); }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { return Err(...AssertHeightRelativeFailed); }
}

**Concrete arithmetic for `ASSERT_HEIGHT_RELATIVE = 0xffff_ffff`, coin confirmed at height 100, checked at height 200:**

- `nowrap=true`: `100.saturating_add(0xffff_ffff) = u32::MAX = 4294967295`. Check: `200 < 4294967295` → **true** → **Err** (correctly rejected; coin is locked for ~4 billion more blocks).
- `nowrap=false`: `100.wrapping_add(0xffff_ffff) = 99`. Check: `200 < 99` → **false** → **Ok** (incorrectly accepted; timelock silently bypassed).

**Concrete arithmetic for `ASSERT_BEFORE_HEIGHT_RELATIVE = 0xffff_ffff`, same scenario:**

- `nowrap=true`: `100.saturating_add(0xffff_ffff) = u32::MAX`. Check: `200 >= u32::MAX` → **false** → **Ok** (correctly accepted; deadline is effectively infinity).
- `nowrap=false`: `100.wrapping_add(0xffff_ffff) = 99`. Check: `200 >= 99` → **true** → **Err** (incorrectly rejected; valid spend refused).

The attacker-controlled entry path is the CLVM condition value. `ASSERT_HEIGHT_RELATIVE` is parsed via `sanitize_uint` with `max_size=4`, which accepts any 4-byte unsigned integer including `0xffff_ffff`: [6](#0-5) 

No additional constraint is applied to the parsed value before it reaches `check_time_locks`. The `SpendBundleConditions` struct stores `height_relative` as `Option<u32>` and `seconds_relative` as `Option<u64>`, both of which can hold their respective maximum values: [7](#0-6) 

### Impact Explanation

**Timelock bypass (nowrap=false path):** A spend bundle containing `ASSERT_HEIGHT_RELATIVE = 0xffff_ffff` is accepted immediately on any node running with `nowrap=false`, regardless of the coin's confirmation height. A puzzle that uses this condition to enforce a multi-billion-block lock (effectively a permanent lock) is silently bypassed. Any smart contract, singleton, or DID that relies on `ASSERT_HEIGHT_RELATIVE` with a large value for security is vulnerable to unauthorized spend on pre-fork nodes.

**Consensus divergence:** A single crafted spend bundle with `ASSERT_HEIGHT_RELATIVE = 0xffff_ffff` is accepted by nodes with `nowrap=false` and rejected by nodes with `nowrap=true`. The same bundle with `ASSERT_BEFORE_HEIGHT_RELATIVE = 0xffff_ffff` is accepted by `nowrap=true` nodes and rejected by `nowrap=false` nodes. Either crafted bundle causes a deterministic chain split between node populations with different `nowrap` settings. Because `nowrap` is passed from Python as a caller-controlled boolean rather than being derived internally from a consensus-committed activation height, any inconsistency in how the Python full node sets this flag — including during fork activation windows — is directly exploitable.

This matches the allowed Critical impact: "Valid unprivileged CLVM program, spend bundle, block, proof, or serialized network object can trigger deterministic consensus divergence" and the allowed High impact: "Timelock validation bypass enables unauthorized spend acceptance."

### Likelihood Explanation

The attacker requires no privileged access. Crafting a CLVM puzzle that outputs `(ASSERT_HEIGHT_RELATIVE . 0xffffffff)` is trivial and costs only the standard transaction fee. The wrapping behavior is not gated by any per-condition check — `sanitize_uint` accepts `0xffff_ffff` as a valid 4-byte unsigned integer, and no downstream clamp or rejection is applied before the value reaches `wrapping_add`. The divergence is deterministic and reproducible across any two nodes with differing `nowrap` values.

### Recommendation

1. **Remove the `nowrap` branch entirely.** Replace all `wrapping_add` calls with `saturating_add` unconditionally. The wrapping behavior has no correct use case: a relative timelock of `u32::MAX` blocks should never silently become a timelock of `confirmed_height - 1`.
2. **If the `nowrap` flag must remain for backward compatibility**, gate it on a consensus-committed activation height computed internally within `check_time_locks` (or its callers in `run_block_generator`), not on a caller-supplied boolean that can diverge across nodes.
3. **Add a pre-validation clamp** in `parse_args` / `parse_conditions` that rejects any `ASSERT_HEIGHT_RELATIVE` value that would overflow when added to any plausible confirmation height (e.g., reject values > `u32::MAX / 2`).

### Proof of Concept

**Attacker constructs a CLVM puzzle** that unconditionally outputs:
```
((ASSERT_HEIGHT_RELATIVE . 0xffffffff))
```
i.e., opcode 82 with argument bytes `ff ff ff ff`.

**Coin confirmed at block height H** (any value, e.g., H=100).

**Spend submitted at current block height P** (any value where P < H + u32::MAX without wrapping, e.g., P=200).

**On a node with `nowrap=false`:**
```
unspent.confirmed_block_index.wrapping_add(0xffff_ffff)
= 100u32.wrapping_add(4294967295)
= 99
prev_transaction_block_height < 99
= 200 < 99
= false  →  Ok(())  →  spend ACCEPTED
```

**On a node with `nowrap=true`:**
```
unspent.confirmed_block_index.saturating_add(0xffff_ffff)
= 100u32.saturating_add(4294967295)
= 4294967295  (u32::MAX)
prev_transaction_block_height < 4294967295
= 200 < 4294967295
= true  →  Err(AssertHeightRelativeFailed)  →  spend REJECTED
```

The same spend bundle is simultaneously valid on one class of nodes and invalid on another, producing a deterministic chain split. The attacker pays only the standard transaction fee.

pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    // strip the leading zero byte if there is one
    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }

    Ok(SanitizedUint::Ok(u64_from_bytes(buf)))
