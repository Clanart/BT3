### Title
Stranded ETH from a prior multicall is silently consumed to fund a different user's WETH swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses the raw `address(this).balance` to partially or fully fund a WETH payment on behalf of a named `payer`. Because ETH is not attributed to any specific user, ETH left on the router by a prior `multicall{value}` call (without `refundETH`) is silently consumed to subsidise a completely different user's swap, permanently destroying the original depositor's funds.

### Finding Description

`pay` contains the following WETH branch:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;          // ← unattributed global balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {                         // ← partial-ETH branch
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // ← payer only pays remainder
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

The function never checks that the ETH on the contract was deposited by `payer`. It treats the entire contract balance as freely available to cover any WETH payment.

**How ETH gets stranded:** `multicall` is `payable`, and so are `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`. The `receive()` guard only blocks plain ETH transfers (no calldata); a call with a function selector bypasses it entirely. Any user who sends `multicall{value: B}(...)` without including a `refundETH` call leaves `B` ETH on the router with no owner record. [2](#0-1) [3](#0-2) 

**How the stranded ETH is consumed:** When any subsequent caller swaps with `tokenIn = WETH`, the pool fires `metricOmmSwapCallback`, which calls `_justPayCallback`, which calls `pay(WETH, msg.sender, pool, value)`. If `address(this).balance > 0`, the partial-ETH branch fires and the stranded ETH is wrapped and forwarded to the pool — reducing the `transferFrom` pull on the actual `payer` by exactly `nativeBalance`. [4](#0-3) 

### Impact Explanation

**Direct, permanent loss of user principal.** userB's `B` ETH is consumed to fund userA's swap. userB receives nothing in return and has no recourse. userA's swap succeeds while only spending `V - B` WETH instead of `V`. This is a swap conservation failure (the pool receives the correct input, but it is sourced from the wrong party) and a direct loss of principal above any Sherlock threshold for a realistic `B`.

### Likelihood Explanation

The stranding condition is easy to reach:
- Any user who sends `multicall{value: B}(calls)` where `calls` does not include `refundETH` strands `B` ETH. This is the standard Uniswap v3 multicall pattern and users routinely omit the refund call when they believe the swap will consume exactly their ETH.
- Excess ETH from over-funded single calls (`exactInputSingle{value: B}(amountIn < B)`) also strands the difference.

An attacker can passively monitor the router's ETH balance and trigger the exploit with a single `exactInputSingle(WETH, amountIn = V)` call where they have only `V - B` WETH approved, requiring no privileged access and no malicious pool.

### Recommendation

Track ETH attribution per-call using transient storage. Store `msg.value` keyed to `msg.sender` at the entry point of each `payable` function, and in `pay` only consume ETH up to the amount attributed to `payer`, not the global `address(this).balance`. Alternatively, restrict the ETH-wrapping path to only fire when `payer == msg.sender` and `msg.value >= value` within the same top-level call, and revert if the balance is insufficient rather than silently drawing from an unattributed pool.

### Proof of Concept

```
1. userB calls router.multicall{value: 0.5 ether}([]) — no calls, 0.5 ETH stranded on router.
   router.balance == 0.5 ETH

2. userA approves router for 0.5 WETH (not 1 WETH).

3. userA calls router.exactInputSingle(
       pool=WETH/TOKEN pool,
       tokenIn=WETH,
       amountIn=1 ether,
       ...
   )

4. Pool fires metricOmmSwapCallback → _justPayCallback →
   pay(WETH, userA, pool, 1 ether)
     nativeBalance = 0.5 ether  (userB's stranded ETH)
     → deposit(0.5 ETH) → transfer(pool, 0.5 WETH)
     → transferFrom(userA, pool, 0.5 WETH)  ← succeeds, userA only had 0.5 WETH approved

5. Swap settles. userA receives full output. userB's 0.5 ETH is permanently lost.
```
