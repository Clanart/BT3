### Title
Integer Wrapping in Relative Timelock Validation Bypasses `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Enforcement — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false` (the pre-soft-fork code path), relative timelock conditions (`ASSERT_SECONDS_RELATIVE`, `ASSERT_HEIGHT_RELATIVE`) are evaluated using `wrapping_add`. An attacker who controls the CLVM puzzle output can supply a crafted large relative value that causes the addition to wrap around to a value smaller than the current timestamp/height, making the timelock check pass when it should fail. This allows a coin to be spent before its intended relative lock period has elapsed.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` flag for all four relative timelock conditions. When `nowrap=false`, it uses `wrapping_add` instead of `saturating_add`:

```rust
// ASSERT_SECONDS_RELATIVE, nowrap=false path
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

And for `ASSERT_HEIGHT_RELATIVE`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [2](#0-1) 

The condition parser (`sanitize_uint`) accepts any value from `0` to `u64::MAX` (8 bytes) for `ASSERT_SECONDS_RELATIVE` and `0` to `u32::MAX` (4 bytes) for `ASSERT_HEIGHT_RELATIVE`: [3](#0-2) 

So an attacker can supply `seconds_relative = u64::MAX - coin_timestamp + 1`, causing `coin_timestamp.wrapping_add(seconds_relative) = 0`. The check `current_timestamp < 0` is always `false`, so the spend is accepted unconditionally regardless of how recently the coin was created.

The inverse problem also exists for `ASSERT_BEFORE_SECONDS_RELATIVE` and `ASSERT_BEFORE_HEIGHT_RELATIVE` with `nowrap=false`: wrapping to a small value makes `timestamp >= small_value` always `true`, causing a false rejection of a legitimately timed spend: [4](#0-3) 

The test suite explicitly documents and confirms both behaviors: [5](#0-4) 

---

### Impact Explanation

**High — Timelock validation bypass enables unauthorized spend acceptance.**

When `nowrap=false` is active (pre-soft-fork blocks or nodes that have not yet activated the `nowrap` soft fork), an attacker who controls a CLVM puzzle can emit `ASSERT_SECONDS_RELATIVE` with a crafted large value. The wrapping addition produces a threshold smaller than the current block timestamp, causing `check_time_locks` to accept the spend as if the relative lock period has elapsed, even if the coin was just created. This directly bypasses the relative timelock mechanism, enabling unauthorized spend acceptance of coins that should be locked.

---

### Likelihood Explanation

The `nowrap` flag is a caller-controlled parameter exposed through both the Rust API and the Python binding (`py_check_time_locks`): [6](#0-5) [7](#0-6) 

The Chia full node passes `nowrap=False` for pre-soft-fork block validation. Any network where the soft fork has not yet activated, or any historical block re-validation path using `nowrap=False`, is exposed. The attacker-controlled input (`seconds_relative` up to `u64::MAX`) is accepted without restriction by `sanitize_uint` and flows directly into the wrapping arithmetic.

---

### Recommendation

1. **Remove the `nowrap=false` / `wrapping_add` path entirely** once the soft fork has activated on all supported networks. The `saturating_add` path (`nowrap=true`) is the correct behavior and should be the only path.
2. **Until removal**, add an explicit guard: if `coin_timestamp + seconds_relative` would overflow (i.e., `seconds_relative > u64::MAX - coin_timestamp`), treat the condition as unsatisfiable and return `AssertSecondsRelativeFailed` immediately, analogous to how `saturating_add` clamps to `u64::MAX`.
3. For `ASSERT_BEFORE_SECONDS_RELATIVE` / `ASSERT_BEFORE_HEIGHT_RELATIVE` with `nowrap=false`, overflow should be treated as the deadline being in the far future (condition trivially satisfied), not as a small wrapped value that causes false rejection.

---

### Proof of Concept

Given:
- `coin_timestamp = 1000` (from `CoinRecord.timestamp`)
- `seconds_relative = 0xFFFF_FFFF_FFFF_FFFF_D8F0` — wait, `seconds_relative` is `u64`, so use `0xFFFF_FFFF_FFFF_D8F0`
- `current_timestamp = 10150`

With `nowrap=false`:
```
1000u64.wrapping_add(0xFFFF_FFFF_FFFF_D8F0) = 0  (wraps to 0)
check: 10150 < 0  →  false  →  Ok(())  ← spend ACCEPTED
```

The test suite confirms this exact case: [5](#0-4) 

A CLVM puzzle that outputs `(80 . 0xFFFF_FFFF_FFFF_D8F0)` (opcode 80 = `ASSERT_SECONDS_RELATIVE`) on a coin confirmed at timestamp 1000 will pass `check_time_locks` with `nowrap=false` at any block timestamp, bypassing the relative timelock entirely.
