### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Locks Bridge Collateral Short - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer()` records the caller-supplied `amount` in the `InitTransfer` event rather than the actual tokens received by the contract. For fee-on-transfer ERC20 tokens, the bridge receives `amount − fee` but the NEAR side credits the recipient with the full `amount`, permanently undercollateralizing the bridge for that token.

### Finding Description
In `OmniBridge.initTransfer()`, when the token is neither a bridge token nor a custom-minter token, the contract pulls tokens from the caller and then emits the event using the raw `amount` argument:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← user-supplied; actual receipt may be less
    );
}
``` [1](#0-0) 

Immediately after, the event is emitted with the same unverified `amount`:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← not the balance delta; may exceed actual receipt
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

No balance-before / balance-after check is performed. The NEAR bridge reads the `amount` field from this event to determine how many tokens to credit the recipient on NEAR. For a fee-on-transfer ERC20, the EVM bridge holds `amount − transferFee` while NEAR credits `amount`, creating a permanent shortfall.

### Impact Explanation
Every fee-on-transfer `initTransfer` call leaves the EVM bridge holding fewer tokens than NEAR has credited. When any holder of the NEAR-side representation bridges back, the EVM `finTransfer` will attempt to release the full `amount` but the bridge's balance is insufficient. This either:
- Causes the `finTransfer` to revert (permanent freeze of user funds), or
- Drains collateral belonging to other users if the bridge has a surplus from other deposits.

This matches the allowed impact: **Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value.**

### Likelihood Explanation
Any ERC20 token that implements a transfer fee (a common pattern in DeFi) can be used with `initTransfer` — the function's `else` branch imposes no token whitelist. A user only needs to call `logMetadata` to register the token on-chain, then call `initTransfer`. The NEAR side processes the event amount verbatim. No privileged access is required.

### Recommendation
Track the actual balance delta after `safeTransferFrom` and use that value in the event:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    amount = uint128(received); // use actual received amount going forward
}
```

Alternatively, explicitly reject fee-on-transfer tokens by requiring `received == amount`.

### Proof of Concept
1. Deploy a fee-on-transfer ERC20 token `FeeToken` with a 1% transfer fee.
2. Call `OmniBridge.logMetadata(feeTokenAddress)` to register it.
3. Approve the bridge for `1000` tokens and call `initTransfer(feeTokenAddress, 1000, 0, 0, nearRecipient, "")`.
4. The bridge receives `990` tokens (1% fee deducted by the token contract).
5. The `InitTransfer` event is emitted with `amount = 1000`.
6. The NEAR bridge processes the event and mints `1000` units of the bridged token to `nearRecipient`.
7. `nearRecipient` bridges back `1000` units; `finTransfer` on EVM attempts `safeTransfer(recipient, 1000)` but the bridge only holds `990`, causing a revert — user funds are permanently frozen. [3](#0-2)
