### Title
`get_state_response_part` Uses New-Epoch Shard Layout to Index Previous-Epoch Block Chunks, Breaking State-Part/Header Root Binding at Resharding Boundaries — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`get_state_response_part` derives `shard_index` from the **sync block's (new) epoch shard layout** and uses it to index into `prev_block`'s chunk array, which is organized under the **previous epoch's shard layout**. At a resharding boundary the two layouts differ, so the function either reads the wrong chunk's `prev_state_root` or panics with `InvalidShardId`. `compute_state_response_header` — the function that produces the committed state root stored in the header — correctly uses `prev_shard_layout` for the same indexing step. The mismatch breaks the invariant that every state part must be derivable from the state root committed in the header.

### Finding Description

`compute_state_response_header` explicitly separates the two shard layouts:

```rust
// chain/chain/src/state_sync/adapter.rs  lines 93-101
let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;   // ← prev epoch
// ...
let chunk_header = chunks.get(prev_shard_index)...;
``` [1](#0-0) 

`get_state_response_part`, which serves the actual trie bytes, omits the second step and uses only the new epoch's layout:

```rust
// chain/chain/src/state_sync/adapter.rs  lines 295-316
let epoch_id = block.header().epoch_id();          // sync block = new epoch
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
// ...
let prev_block = self.chain_store.get_block(header.prev_hash())?;
let shard_index = shard_layout.get_shard_index(shard_id)?;  // ← new epoch, WRONG
let state_root = prev_block
    .chunks()
    .get(shard_index)                              // indexes old-epoch block with new-epoch index
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [2](#0-1) 

`prev_block` is the last block of the **old** epoch; its chunk array is ordered by the old epoch's shard layout. When a resharding event changes the number or ordering of shards, `shard_index` from the new layout either:

- **exceeds** `prev_block.chunks().len()` → `get_state_response_part` returns `Err(InvalidShardId)` for every part request, making the shard unsyncable; or  
- **falls within bounds but points to a different shard's chunk** → the function silently returns the wrong `state_root`, generates trie nodes rooted at that wrong hash, and uploads them to external storage or serves them to peers.

The downstream `set_state_part` validates each part against the state root extracted from the header:

```rust
// chain/chain/src/state_sync/adapter.rs  lines 541-553
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
if matches!(
    self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part),
    StatePartValidationResult::Invalid
) { ... return Err(...) }
``` [3](#0-2) 

Because the header's `state_root` was computed with the correct `prev_shard_layout` and the parts were generated with the wrong `shard_layout`, every part fails validation. The state sync for that shard cannot complete.

The same wrong `state_root` is also passed to `obtain_state_part_impl`, which uses it to look up trie nodes:

```rust
// chain/chain/src/runtime/mod.rs  lines 509-518
let trie_with_state =
    self.tries.get_trie_with_block_hash_for_shard(shard_uid, *state_root, &prev_hash, true);
let trie_nodes = self.tries.get_trie_nodes_for_part_from_snapshot(
    shard_uid, state_root, &prev_hash, part_id, trie_with_state,
);
``` [4](#0-3) 

A `state_root` from the wrong chunk will not be present in the snapshot, causing `obtain_state_part` to fail with a storage error.

### Impact Explanation

Any node performing state sync at a resharding epoch boundary — including the production state-dump path in `nearcore/src/state_sync.rs` — calls `get_state_response_part` for each shard in the new epoch. At the boundary, the function either errors out or produces parts that fail the header-root validation check. The result is that no state part for the affected shard can be successfully stored or applied, making the shard permanently unsyncable from that sync hash. Nodes that rely on external state sync (the majority of non-archival nodes joining after a resharding event) cannot bootstrap. [5](#0-4) 

### Likelihood Explanation

The bug is latent until a resharding event occurs. Once resharding is activated on mainnet or testnet, every state-sync request for the new epoch's shards at the resharding sync hash triggers the wrong-layout path. No special attacker capability is required; any node issuing a standard `StateRequestPart` message exercises the code path.

### Recommendation

Mirror the two-layout pattern from `compute_state_response_header` inside `get_state_response_part`:

```rust
// After fetching prev_block:
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;  // use prev epoch
let state_root = prev_block.chunks().get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?.prev_state_root();
```

This makes `get_state_response_part` use the same shard-index derivation as `compute_state_response_header`, restoring the invariant that parts are always rooted at the state root committed in the header.

### Proof of Concept

1. Deploy a network with resharding enabled (e.g., `ShardLayout::V2` → `ShardLayout::V3` transition).
2. Let the network advance past the resharding epoch boundary. The sync hash is the first block of the new epoch.
3. Call `get_state_response_header(child_shard_id, sync_hash)` — succeeds, returns a header whose `chunk_prev_state_root` is derived via `prev_shard_layout`.
4. Call `get_state_response_part(child_shard_id, 0, sync_hash)` — uses `shard_layout` (new epoch) to compute `shard_index`, indexes `prev_block.chunks()` with that index, obtains a different (or absent) chunk, and either returns `Err(InvalidShardId)` or a part rooted at the wrong `state_root`.
5. Attempt `set_state_part` with the returned part — `validate_state_part` fails because the part's root does not match the header's committed `state_root`.
6. State sync for `child_shard_id` is permanently broken for this sync hash. [6](#0-5) [7](#0-6)

    /// Computes ShardStateSyncResponseHeader.
    pub fn compute_state_response_header(
        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<ShardStateSyncResponseHeader, Error> {
        // Consistency rules:
        // 1. Everything prefixed with `sync_` indicates new epoch, for which we are syncing.
        // 1a. `sync_prev` means the last of the prev epoch.
        // 2. Empty prefix means the height where chunk was applied last time in the prev epoch.
        //    Let's call it `current`.
        // 2a. `prev_` means we're working with height before current.
        // 3. In inner loops we use all prefixes with no relation to the context described above.
        let sync_block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let sync_block_header = sync_block.header();
        let sync_block_epoch_id = sync_block_header.epoch_id();
        let shard_ids = self.epoch_manager.shard_ids(sync_block_epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }

        // The chunk was applied at height `chunk_header.height_included`.
        // Getting the `current` state.
        // TODO(current_epoch_state_sync): check that the sync block is what we would expect. So, either the first
        // block of an epoch, or the first block where there have been two new chunks in the epoch
        let sync_prev_block = self.chain_store.get_block(sync_block_header.prev_hash())?;

        let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
        let prev_epoch_id = sync_prev_block.header().epoch_id();
        let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
        let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;

        // Chunk header here is the same chunk header as at the `current` height.
        let sync_prev_hash = sync_prev_block.hash();
        let chunks = sync_prev_block.chunks();
        let chunk_header = chunks.get(prev_shard_index).ok_or(Error::InvalidShardId(shard_id))?;

    pub fn get_state_response_part(
        &mut self,
        shard_id: ShardId,
        part_id: u64,
        sync_hash: CryptoHash,
    ) -> Result<StatePart, Error> {
        let _span = tracing::debug_span!(
            target: "sync",
            "get_state_response_part",
            %shard_id,
            part_id,
            ?sync_hash)
        .entered();
        let block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let header = block.header();
        let epoch_id = block.header().epoch_id();
        // Check cache
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id)).unwrap();
        if let Some(bytes) = self.chain_store.store_ref().get(DBCol::StateParts, &key) {
            metrics::STATE_PART_CACHE_HIT.inc();
            let state_part = StatePart::from_bytes(bytes.to_vec())?;
            return Ok(state_part);
        }
        metrics::STATE_PART_CACHE_MISS.inc();

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_ids = self.epoch_manager.shard_ids(epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let prev_block = self.chain_store.get_block(header.prev_hash())?;
        let shard_index = shard_layout.get_shard_index(shard_id)?;
        let state_root = prev_block
            .chunks()
            .get(shard_index)
            .ok_or(Error::InvalidShardId(shard_id))?
            .prev_state_root();
        let prev_hash = *prev_block.hash();
        let prev_prev_hash = *prev_block.header().prev_hash();
        let state_root_node = self
            .runtime_adapter
            .get_state_root_node(shard_id, &prev_hash, &state_root)
            .log_storage_error("get_state_root_node fail")?;
        let num_parts = get_num_state_parts(state_root_node.memory_usage);
        if part_id >= num_parts {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let current_time = Instant::now();
        let state_part = self
            .runtime_adapter
            .obtain_state_part(
                shard_id,
                &prev_prev_hash,
                &state_root,
                PartId::new(part_id, num_parts),
            )
            .log_storage_error("obtain_state_part fail")?;

        let elapsed_ms = (self.clock.now().signed_duration_since(current_time))
            .whole_milliseconds()
            .max(0) as u128;
        self.requested_state_parts
            .save_state_part_elapsed(&sync_hash, &shard_id, &part_id, elapsed_ms);

        // Cache the part data, but only if the corresponding header is also cached.
        // At epoch boundaries, clear_all_downloaded_parts() deletes all cached headers
        // and parts. Since serving runs on a separate actor, a part request can arrive
        // after the clear and re-create a StatePartKey without its StateHeaderKey,
        // which the storage validator treats as an inconsistency.
        let header_key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
        if self.chain_store.store_ref().exists(DBCol::StateHeaders, &header_key) {
            let mut store_update = self.chain_store.store().store_update();
            let bytes = state_part.to_bytes();
            store_update.set(DBCol::StateParts, &key, &bytes);
            store_update.commit();
        }

        Ok(state_part)
    }

    async fn upload_state_part(self: Arc<Self>, part_idx: u64) -> anyhow::Result<()> {
        if !self.parts_missing.read().contains(&part_idx) {
            self.inc_parts_dumped();
            return Ok(());
        }
        let part_id = PartId::new(part_idx, self.num_parts);

        let state_part = loop {
            if self.canceled.load(Ordering::Relaxed) {
                return Ok(());
            }
            let _timer = metrics::STATE_SYNC_DUMP_ITERATION_ELAPSED
                .with_label_values(&[&self.shard_id.to_string()])
                .start_timer();
            let state_part = {
                let _permit = self.obtain_parts.acquire().await.unwrap();
                self.runtime.obtain_state_part(
                    self.shard_id,
                    &self.sync_prev_prev_hash,
                    &self.state_root,
                    part_id,
                )
            };
            match state_part {
                Ok(state_part) => {
                    break state_part;
                }
                Err(error) => {
                    // TODO: return non retryable errors.
                    tracing::warn!(
                        target: "state_sync_dump",
                        shard_id = %self.shard_id, epoch_height=%self.epoch_height, epoch_id=?&self.epoch_id, ?part_id, ?error,
                        "failed to obtain state part, retrying in 200 millis"
                    );
                    self.clock.sleep(Duration::milliseconds(200)).await;
                    continue;
                }
            }
        };
