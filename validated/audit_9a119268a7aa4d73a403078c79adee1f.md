### Title
Stale Lazy-Hash Root Returned by `get_root_hash()` / `get_proof_of_inclusion()` After Mutations Without `calculate_lazy_hashes()` — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob` uses a deferred ("lazy") hash-recomputation model: after any `insert`, `delete`, or `upsert`, only the affected lineage is marked dirty via `mark_lineage_as_dirty()`. The actual hash recomputation is deferred until the caller explicitly invokes `calculate_lazy_hashes()`. Both `get_root_hash()` and `get_proof_of_inclusion()` read hashes directly from the blob without triggering recomputation. If a caller omits `calculate_lazy_hashes()` between a mutation and a root-hash or proof query, the returned root hash and proof layers silently reflect the pre-mutation tree state. This is the direct structural analog of the EIP-712 stale `_CACHED_DOMAIN_SEPARATOR` bug: a cached/deferred value is consumed by a security-critical operation without being refreshed after the underlying state changes.

---

### Finding Description

**Lazy-hash deferral model**

After every structural mutation, `mark_lineage_as_dirty()` is called to flag internal nodes along the path to the root: [1](#0-0) [2](#0-1) [3](#0-2) 

The dirty nodes retain their old hash bytes in the blob until `calculate_lazy_hashes()` is explicitly called. No mutation path auto-triggers recomputation.

**`get_root_hash()` reads the stale blob**

`get_root_hash()` reads the hash stored at index 0 (the root block) directly from the blob. After a mutation that marks the root's lineage dirty, this returns the pre-mutation root hash — identical to the EIP-712 pattern where `_domainSeparatorV4()` returns `_CACHED_DOMAIN_SEPARATOR` without checking whether the underlying name has changed.

**`get_proof_of_inclusion()` builds layers from stale internal-node hashes**

`get_proof_of_inclusion()` walks from the target leaf up to the root, collecting sibling hashes and `combined_hash` values from the blob. If any ancestor's hash is dirty, the proof layers embed the stale `combined_hash` values. The self-consistency check `proof.valid()` passes because it recomputes each layer's hash using the (also stale) `other_hash` sibling values — the stale values are internally consistent with each other, just not with the actual current tree.

**Test code confirms the mandatory manual step**

The test suite explicitly calls `calculate_lazy_hashes()` before every proof query, confirming that omitting it yields incorrect results: [4](#0-3) [5](#0-4)

### Impact Explanation

**Allowed impact matched:** *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

Concretely:

1. **Stale root used as authoritative root.** A DataLayer node that calls `get_root_hash()` immediately after a deletion (without `calculate_lazy_hashes()`) obtains the pre-deletion root R₁. If R₁ is committed or broadcast as the current tree root, any proof of inclusion generated against R₁ — including proofs for keys that have since been deleted — will pass `proof.valid()` and match the committed root. This allows a deleted key to appear provably present.

2. **Proof generated against wrong root.** A proof generated by `get_proof_of_inclusion()` before `calculate_lazy_hashes()` carries stale `combined_hash` values and a stale `root_hash()`. A verifier comparing `proof.root_hash()` to the stale `get_root_hash()` sees a match and accepts the proof, even though the proof does not reflect the post-mutation tree.

3. **Two divergent root views.** Callers that call `calculate_lazy_hashes()` see root R₂; callers that do not see R₁. This is the exact two-view inconsistency described in the EIP-712 report (`DOMAIN_SEPARATOR()` vs `_domainSeparatorV4()`), and it can cause consensus divergence between DataLayer peers that apply the same mutations but query the root at different points.

---

### Likelihood Explanation

The Python bindings expose `get_root_hash()`, `get_proof_of_inclusion()`, `insert()`, `delete()`, and `upsert()` as independent methods. Nothing in the API contract prevents a caller — including the Chia Python full node — from calling `get_root_hash()` or `get_proof_of_inclusion()` without first calling `calculate_lazy_hashes()`. The omission is a single missing method call, not a complex exploit chain. Any DataLayer operation that inserts or deletes a key and then immediately reads the root hash or generates a proof is affected. [6](#0-5) 

---

### Recommendation

1. **Auto-recompute on read.** Make `get_root_hash()` and `get_proof_of_inclusion()` call `calculate_lazy_hashes()` internally (taking `&mut self`) before reading any hash from the blob. This mirrors the recommendation in the EIP-712 report: make the "cached" and "live" views identical by always recomputing on demand.

2. **Alternatively, enforce ordering via type state.** Introduce a `DirtyMerkleBlob` / `CleanMerkleBlob` type-state pattern so that `get_root_hash()` and `get_proof_of_inclusion()` are only callable on a `CleanMerkleBlob`, which is only obtainable after `calculate_lazy_hashes()` succeeds.

3. **At minimum, document and assert.** Add a `debug_assert!(!self.has_dirty_hashes())` guard at the entry of `get_root_hash()` and `get_proof_of_inclusion()` so that callers are alerted during development if they query before recomputing.

### Proof of Concept

```rust
// 1. Build a two-leaf tree and compute correct hashes.
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.insert(KeyId(1), ValueId(1), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(2), &hash_b, InsertLocation::Auto {}).unwrap();
blob.calculate_lazy_hashes().unwrap();
let root_before = blob.get_root_hash().unwrap(); // R1 — correct

// 2. Delete key 1 — marks lineage dirty, does NOT recompute.
blob.delete(KeyId(1)).unwrap();

// 3. Read root WITHOUT calling calculate_lazy_hashes().
let stale_root = blob.get_root_hash().unwrap(); // Still R1 — STALE
assert_eq!(root_before, stale_root); // passes — stale root equals pre-deletion root

// 4. Generate a proof for key 2 against the stale blob.
let proof = blob.get_proof_of_inclusion(KeyId(2)).unwrap();
assert!(proof.valid());                        // passes — internally consistent with stale hashes
assert_eq!(proof.root_hash(), stale_root);     // passes — both stale

// 5. Correct root (post-deletion, single-leaf tree) differs.
blob.calculate_lazy_hashes().unwrap();
let correct_root = blob.get_root_hash().unwrap(); // R2
assert_ne!(stale_root, correct_root);          // stale_root != correct_root

// Result: a verifier that committed stale_root (R1) as the authoritative root
// will accept the proof for key 2 — but the proof encodes the wrong tree state.
// Any peer that correctly recomputed to R2 will reject the same proof,
// causing DataLayer root divergence.

impl ProofOfInclusion {
    pub fn root_hash(&self) -> Hash {
        if let Some(last) = self.layers.last() {
            last.combined_hash
        } else {
            self.node_hash
        }
    }

    pub fn valid(&self) -> bool {
        let mut existing_hash = self.node_hash;

        for layer in &self.layers {
            let calculated_hash = crate::calculate_internal_hash(
                &existing_hash,
                layer.other_hash_side,
                &layer.other_hash,
            );

            if calculated_hash != layer.combined_hash {
                return false;
            }

            existing_hash = calculated_hash;
        }

        existing_hash == self.root_hash()
    }
