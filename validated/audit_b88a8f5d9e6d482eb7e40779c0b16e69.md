### Title
`batch_insert` Bypasses Duplicate Key/Hash Checks, Corrupting DataLayer Merkle Tree Roots — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a caller supplies a batch containing a key (or leaf hash) that already exists in the tree, or a key that appears more than once within the same batch, the duplicate leaf node is silently written into the blob. The resulting tree root commits to a phantom duplicate leaf, permanently corrupting the Merkle root and invalidating all subsequent proofs of inclusion/exclusion derived from it.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) opens with two mandatory guards:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `insert` for at most the last two items popped from the input vector (only when `leaf_count <= 1`). Every other item is written directly via `insert_entry_to_blob`, which performs **no duplicate check**:

```rust
for ((key, value), hash) in keys_values_hashes {   // no guard here
    let new_leaf_index = self.get_new_index();
    let new_block = Block { node: Node::Leaf(LeafNode { hash, key, value, .. }), .. };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // silently overwrites cache
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which calls `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`. `HashMap::insert` silently overwrites the previous entry without returning an error:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
```

After `batch_insert` returns:

- **Both** leaf nodes (original and duplicate) are physically present in the blob and are wired into the internal-node subtree that `batch_insert` constructs.
- The cache tracks only the **last** index for the duplicated key/hash.
- The Merkle root is computed over a tree that includes the phantom duplicate leaf, so the root hash is wrong.
- `get_proof_of_inclusion` returns a proof for the cache-tracked leaf; that proof validates against the corrupted root, but the root itself no longer faithfully represents the key-value mapping.
- `check_integrity` would detect the inconsistency (`leaf_count != key_to_index_cache_length`), but it is not called automatically after `batch_insert`.

When `leaf_count > 1` at call time, **all** items in the batch bypass duplicate checks, not just items beyond the first two.

---

### Impact Explanation

The DataLayer Merkle root is the value committed on-chain to represent a store's state. A corrupted root means:

- The on-chain commitment does not faithfully represent the key-value mapping.
- Proofs of inclusion generated from the corrupted tree validate against the corrupted root, but prove membership in a tree that contains phantom data.
- A party relying on the root to verify DataLayer state (e.g., a light client or a cross-store proof) receives a root that silently encodes extra, attacker-injected leaf content.
- Subsequent `delete` or `upsert` operations on the orphaned duplicate leaf are impossible (the cache does not know about it), leaving the blob in a permanently inconsistent state that `check_integrity` will flag as an error, potentially halting further DataLayer operations on that store.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion path used by the DataLayer node and is directly exposed to Python callers via `py_batch_insert`. Any caller that supplies a list of `(KeyId, ValueId, Hash)` triples controls the keys and hashes. A duplicate key within a batch of three or more items (or a key already present in a tree with two or more existing leaves) is sufficient to trigger the bug. No special privilege is required beyond the ability to submit DataLayer key-value updates.

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the top of the `batch_insert` loop, mirroring the guards in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing direct-insert logic
}
```

Alternatively, update `add_leaf` to return an error on collision instead of silently overwriting, so that any future code path that bypasses the outer guard still fails safely.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    // Build a tree with 2 existing leaves so leaf_count > 1
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    let h = |b: u8| Hash(Bytes32::new([b; 32]));

    blob.insert(KeyId(1), ValueId(1), &h(1), InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(2), ValueId(2), &h(2), InsertLocation::Auto {}).unwrap();
    blob.calculate_lazy_hashes().unwrap();
    let root_before = blob.get_root_hash().unwrap();

    // batch_insert with a duplicate key (KeyId(1) already exists)
    // No error is returned — the duplicate is silently accepted
    blob.batch_insert(vec![
        ((KeyId(3), ValueId(3)), h(3)),
        ((KeyId(1), ValueId(99)), h(99)),  // duplicate key, different value/hash
    ]).unwrap();

    blob.calculate_lazy_hashes().unwrap();
    let root_after = blob.get_root_hash().unwrap();

    // root_after encodes a tree with TWO leaves for KeyId(1)
    // check_integrity() will now fail with IntegrityKeyToIndexCacheLength
    assert!(blob.check_integrity().is_err());
}
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
