### Title
Insufficient Amount Validation in `init_transfer` Causes Permanent Fund Lock When `normalize_amount` Produces Zero ‚Äî (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` only checks `fee < amount` before locking user tokens. It does not verify that `amount_without_fee` will survive decimal normalization as a non-zero value. Later, `sign_transfer` applies `normalize_amount` (floor division) and hard-rejects with `InvalidAmountToTransfer` if the result is zero. Because no cancel/refund path exists for a stored transfer message, any transfer whose `amount_without_fee` is below the normalization unit is permanently locked in the bridge.

---

### Finding Description

The NEAR bridge contract stores a `TransferMessage` and locks the user's tokens during `init_transfer`. The only fee-related guard at that point is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This allows `fee = amount - 1`, leaving `amount_without_fee = 1`.

Later, when a relayer calls `sign_transfer`, the contract normalizes the net amount for the destination chain:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` is pure floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

For any token where `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridging to an EVM chain with 18 decimals, giving `diff_decimals = 6`), any `amount_without_fee < 10^6` normalizes to exactly 0. `sign_transfer` then panics unconditionally, and the transfer can never be finalized.

There is no public `cancel_transfer`, `refund_transfer`, or equivalent function. The only path that removes a stored transfer message is `remove_transfer_message`, which is called only from `claim_fee_callback` (requires a valid finalization proof from the destination chain ‚Äî impossible if the transfer was never signed) and `process_fin_transfer_to_near` (also requires a proof). The tokens are irrecoverably locked.

The same condition can be reached via `update_transfer_fee`, which permits the original sender to raise `fee` up to `amount - 1`:

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [4](#0-3) 

A user who raises their fee to `amount - 1` after a valid `init_transfer` will also permanently lock their own funds.

---

### Impact Explanation

User tokens are locked in the NEAR bridge locker at `init_transfer_internal` and can never be released. `sign_transfer` will always revert with `ERR_INVALID_AMOUNT_TO_TRANSFER`, and no recovery function exists. This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**. [5](#0-4) 

---

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user in two ways:

1. **Direct small-amount transfer**: A user transfers an amount below the normalization unit (e.g., fewer than `10^diff_decimals` base units of a NEAR-native token bridging to an 18-decimal EVM token). The `init_transfer` check passes; `sign_transfer` permanently fails.
2. **Fee escalation**: A user with a valid in-flight transfer calls `update_transfer_fee` to raise the token fee close to the full amount, reducing `amount_without_fee` below the normalization unit.

For tokens with large decimal gaps (e.g., 24-decimal NEAR tokens to 6-decimal EVM tokens, `diff_decimals = 18`), any transfer of fewer than `10^18` base units ‚Äî i.e., less than 1 whole token ‚Äî is permanently unrecoverable. This is a realistic amount for small or test transfers.

---

### Recommendation

Add a normalization-aware guard inside `init_transfer` (or `init_transfer_internal`) that rejects the transfer before locking tokens if `normalize_amount(amount_without_fee, decimals) == 0`. Alternatively, implement a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have not yet been signed, providing a recovery path for stuck transfers.

---

### Proof of Concept

**Setup**: Token registered with `origin_decimals = 24`, `decimals = 18` (diff = 6). Destination chain is Ethereum.

1. Alice calls `ft_transfer_call` ‚Üí `init_transfer` with `amount = 5` (5 yocto-units), `fee = 0`.
   - Check `fee < amount` ‚Üí `0 < 5` ‚úì passes.
   - `init_transfer_internal` locks 5 tokens in the bridge.
   - `TransferMessage` stored on-chain.

2. Relayer calls `sign_transfer` for Alice's transfer.
   - `amount_without_fee() = 5 - 0 = 5`.
   - `normalize_amount(5, Decimals { origin: 24, decimals: 18 }) = 5 / 10^6 = 0`.
   - `require!(0 > 0, ...)` ‚Üí **panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`**.

3. No other function can remove Alice's `TransferMessage` or unlock her tokens. Alice's 5 tokens are permanently locked. [2](#0-1) [3](#0-2) [1](#0-0)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
