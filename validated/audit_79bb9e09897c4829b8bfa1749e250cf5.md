### Title
Disabled TLS Certificate Validation in `BlockingCheckClient` Allows MITM/DNS-Rebinding Bypass of Transaction Blocking Policy, Causing Prover to Return Authoritative Proofs for Blocked Transactions ŌĆö (File: `crates/starknet_transaction_prover/src/blocking_check.rs`)

---

### Summary

The `BlockingCheckClient` in `starknet_transaction_prover` unconditionally disables TLS certificate validation via `danger_accept_invalid_certs(true)`. A network-level attacker ŌĆö including one using DNS rebinding ŌĆö can intercept the HTTPS connection to the external blocking-check service and always return `Allowed`, causing the prover to generate and return a cryptographically valid `ProveTransactionResult` (including `proof_facts`) for a transaction that the operator intended to block. Those proof facts are then accepted by the sequencer gateway's `run_proof_verification`, allowing the blocked transaction to be sequenced.

---

### Finding Description

`BlockingCheckClient::new()` builds a `reqwest` HTTP client with `.danger_accept_invalid_certs(true)`:

```rust
// crates/starknet_transaction_prover/src/blocking_check.rs  lines 72-77

    let http_client = reqwest::Client::builder()
        .danger_accept_invalid_certs(true)   // ŌåÉ accepts ANY certificate
        .build()
        .expect("Failed to build blocking check HTTP client");
    Self { http_client, url, timeout_millis, fail_open }
}
``` [1](#0-0) 

This client is used in `prove_with_blocking_check` to send `starknet_checkTransaction` to the configured `blocking_check_url` before returning a proof:

```rust

Because TLS certificate validation is disabled, any attacker who can intercept the TCP/TLS connection between the prover and the blocking-check endpoint ŌĆö via DNS rebinding, ARP poisoning, BGP hijack, or a compromised resolver ŌĆö can present a self-signed certificate and respond with:

```json
{"jsonrpc":"2.0","result":{},"id":1}
```

The `check_transaction` parser treats the absence of an `error` field as `BlockingCheckResult::Allowed`:

```rust
// blocking_check.rs  lines 118-125
match json_rpc_response.error {
    None => BlockingCheckResult::Allowed,
    Some(err) if err.code == BLOCKED_ERROR_CODE => BlockingCheckResult::Blocked,
    Some(err) => { ... BlockingCheckResult::Inconclusive }
}
``` [3](#0-2) 

`prove_with_blocking_check` then calls `run_and_prove`, which executes the virtual OS and generates a full `ProveTransactionResult`:

```rust

    pub proof: Proof,
    pub proof_facts: ProofFacts,

``` [4](#0-3) 

The `ProvingRpcServer` implementation in `rpc_impl.rs` has no authentication, no Host-header check, and no origin validation ŌĆö it accepts any caller:

```rust

The returned `proof_facts` are then embedded in an Invoke V3 transaction and submitted to the sequencer gateway. The gateway calls `run_proof_verification` ŌåÆ `starknet_proof_verifier::verify_proof`, which verifies the proof cryptographically. Because the proof is genuinely valid (the OS ran correctly), verification passes and the transaction is admitted to the mempool. [6](#0-5) 

The blockifier's `validate_proof_facts` checks `program_hash`, `block_hash`, `block_number`, and `config_hash` ŌĆö none of which involve the blocking-check policy ŌĆö so it also accepts the transaction: [7](#0-6) 

---

### Impact Explanation

The prover RPC (`starknet_proveTransaction`) returns an authoritative `ProveTransactionResult` containing cryptographically valid `proof_facts` for a transaction the operator explicitly intended to block. Because the gateway and blockifier validate only the cryptographic and structural correctness of the proof facts ŌĆö not the blocking-check policy ŌĆö the transaction is accepted and sequenced. This matches:

> **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

and

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

- The attack requires network-level interception of the TLS connection between the prover and the blocking-check service. This is achievable via DNS rebinding (the direct analog of the seed report), ARP poisoning on the same subnet, or a compromised DNS resolver ŌĆö all unprivileged from the prover's perspective.
- The default `blocking_check_fail_open: true` means even a simple network disruption (without MITM) already bypasses the check; the `danger_accept_invalid_certs(true)` makes active MITM equally trivial.
- The prover server binds to `0.0.0.0:3000` by default with no authentication, making it reachable from any network interface. [8](#0-7) 

---

### Recommendation

1. **Remove `danger_accept_invalid_certs(true)`** from `BlockingCheckClient::new()`. Use standard TLS certificate validation.
2. If the blocking-check service uses a self-signed certificate, supply a custom CA bundle via `.add_root_certificate(...)` instead of disabling all validation.
3. Add a `Host` header allowlist to the prover's HTTP middleware (analogous to the original report's recommendation) to prevent DNS-rebinding-based abuse of the unauthenticated `starknet_proveTransaction` endpoint.
4. Consider making `blocking_check_fail_open` default to `false` so that any check failure is conservative.

---

### Proof of Concept

1. Operator deploys the prover with `BLOCKING_CHECK_URL=https://check.internal/` and `BLOCKING_CHECK_FAIL_OPEN=false`.
2. Attacker poisons DNS for `check.internal` to resolve to `127.0.0.1` (DNS rebinding) or performs ARP poisoning on the prover's subnet.
3. Attacker's server presents a self-signed TLS certificate for `check.internal`; the prover accepts it because `danger_accept_invalid_certs(true)` is hardcoded.
4. Attacker's server responds to every `starknet_checkTransaction` POST with `{"jsonrpc":"2.0","result":{},"id":1}`.
5. Caller submits a transaction that the real blocking-check service would return code `10000` for.
6. `check_transaction` returns `BlockingCheckResult::Allowed`; `prove_with_blocking_check` proceeds to `run_and_prove`.
7. Prover returns a valid `ProveTransactionResult` with `proof_facts` for the blocked transaction.
8. Caller embeds `proof_facts` in an Invoke V3 transaction and submits it to the sequencer gateway.
9. `run_proof_verification` passes (proof is cryptographically valid); `validate_proof_facts` passes (program/block/config hashes are correct); transaction enters the mempool and is sequenced.

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
