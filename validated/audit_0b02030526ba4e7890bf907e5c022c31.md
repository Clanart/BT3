### Title
Fee Bypass via Unchecked Felt Arithmetic Overflow in `compute_max_possible_fee` — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in `transaction_impls.cairo` performs raw felt-field arithmetic on user-controlled resource bound values without any range-check enforcement. Because Cairo felt arithmetic is modular (mod the Stark prime P ≈ 2²⁵¹), a transaction sender can craft resource bound values such that the sum overflows to exactly 0 mod P. When `max_fee == 0`, `charge_fee` returns immediately without executing any ERC-20 fee transfer, allowing the transaction to execute entirely free of charge.

---

### Finding Description

**Root cause — `compute_max_possible_fee`** (`transaction_impls.cairo`, lines 87–102):

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit +
           l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip) +
           l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
```

All arithmetic is in the felt field. No `assert_nn_le` or range-check builtin call is made on any of the individual resource bound fields (`max_amount`, `max_price_per_unit`, `tip`) before or inside this function. The resource bounds are loaded from hints (`%{ LoadCommonTxFields %}` in `get_account_tx_common_fields`) and committed to only via the Poseidon transaction hash, which does not range-check its inputs.

**Triggering path — `charge_fee`** (`transaction_impls.cairo`, lines 111–165):

```cairo
func charge_fee{...}(block_context: BlockContext*, tx_execution_context: ExecutionContext*) {
    ...
    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    if (max_fee == 0) {
        return ();          // ← fee transfer is skipped entirely
    }
    ...
    assert_nn_le(calldata.amount.low, max_fee);   // never reached
    ...
    non_reverting_select_execute_entry_point_func(...);  // ERC-20 transfer never called
}
```

The early-return guard `if (max_fee == 0) { return (); }` is the intended zero-fee short-circuit for free transactions. However, because `compute_max_possible_fee` is not overflow-safe, an attacker can force this branch by making the felt sum wrap to 0.

**How the overflow is achieved:**

The Stark prime is P = 2²⁵¹ + 17·2¹⁹² + 1. A user constructs a V3 transaction with three resource bound entries whose products sum to exactly P (≡ 0 mod P). A concrete example:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | P − 1 |
| `l1_gas_bounds.max_price_per_unit` | 1 |
| `l2_gas_bounds.max_amount` | 1 |
| `l2_gas_bounds.max_price_per_unit` | 1 |
| `tip` | 0 |
| `l1_data_gas_bounds.max_amount` | 0 |

Sum = (P−1)·1 + 1·(1+0) + 0 = P ≡ 0 (mod P).

The user signs the Poseidon hash of this transaction (which includes these resource bounds), submits it to the sequencer, and the OS proves the execution without charging any fee.

**No OS-level range enforcement exists** on resource bound fields. The only downstream check, `assert_nn_le(calldata.amount.low, max_fee)` at line 135, is never reached when `max_fee == 0`.

---

### Impact Explanation

**Direct loss of funds.** The sequencer executes arbitrary computation (Invoke, DeployAccount, Declare) and receives zero fee payment. Because the OS produces a valid proof for this execution, the L1 verifier accepts it. The attack is not limited to a single transaction; any number of transactions can be crafted this way, permanently denying fee revenue to the sequencer and, by extension, to stakers/validators who depend on fee income.

---

### Likelihood Explanation

The attack requires only that the transaction sender:
1. Compute a set of resource bound values whose felt sum equals 0 mod P (trivially solvable with one degree of freedom).
2. Sign the resulting transaction hash (standard wallet operation).
3. Submit the transaction to any sequencer.

No privileged access, leaked keys, or third-party compromise is required. The sequencer has no Cairo-enforced reason to reject the transaction — the OS itself is the only authoritative checker, and it does not range-check resource bounds. The attack is fully reachable by any unprivileged transaction sender on any V3 transaction type.

---

### Recommendation

Add explicit range checks on each resource bound component before performing the fee arithmetic. Specifically, inside `compute_max_possible_fee` (or before it is called), assert that each `max_amount` fits in a u64 and each `max_price_per_unit` fits in a u128:

```cairo
assert_nn_le(l1_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_U128);
// ... repeat for l2 and l1_data bounds, and for tip
```

With valid u64 × u128 products, the maximum possible sum is approximately 3 × 2¹⁹² ≪ P, making overflow impossible.

---

### Proof of Concept

1. Construct a V3 Invoke transaction with:
   - `l1_gas_bounds = { max_amount: P-1, max_price_per_unit: 1 }`
   - `l2_gas_bounds = { max_amount: 1, max_price_per_unit: 1 }`, `tip = 0`
   - `l1_data_gas_bounds = { max_amount: 0, max_price_per_unit: 0 }`
2. Sign the Poseidon hash of this transaction with a valid account key.
3. Submit to the sequencer. The sequencer includes it in a block.
4. The OS calls `execute_invoke_function_transaction` → `charge_fee`.
5. `compute_max_possible_fee` returns `(P-1)*1 + 1*(1+0) + 0 = P ≡ 0`.
6. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits — no ERC-20 transfer occurs.
7. The transaction's `__execute__` entry point runs normally; the OS produces a valid proof; the sequencer receives 0 fee.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
