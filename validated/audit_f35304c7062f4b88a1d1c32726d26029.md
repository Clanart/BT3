### Title
Payload Serialization Inconsistency Between NEAR `sign_transfer` and Solana `FinalizeTransferPayload.serialize_for_near` Omits `message` Field from Hash, Permanently Freezing Transfers with Non-Empty Messages - (File: `solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`)

---

### Summary

The NEAR `sign_transfer` function conditionally includes a `message` field in the borsh-encoded payload that is hashed and MPC-signed. The Solana `FinalizeTransferPayload.serialize_for_near()` function, which reconstructs the same payload for signature verification, never serializes a `message` field. Any transfer to Solana that carries a non-empty `DestHexMsg` message will produce a hash mismatch between the NEAR-signed payload and the Solana-verified payload, causing signature verification to permanently fail and the user's funds to be irrecoverably frozen.

---

### Finding Description

**NEAR signing side — `encode_hashable()` conditionally includes `message`:**

`TransferMessagePayload.encode_hashable()` in `near/omni-types/src/lib.rs` branches on whether the `message` field is empty:

```rust
pub fn encode_hashable(&self) -> Result<Vec<u8>, String> {
    if self.message.is_empty() {
        borsh::to_vec(&TransferMessagePayloadV1::from(self.clone())).map_err(stringify)
    } else {
        borsh::to_vec(self).map_err(stringify)   // ← includes message bytes
    }
}
``` [1](#0-0) 

`TransferMessagePayload` has a `message: Vec<u8>` field that `TransferMessagePayloadV1` does not: [2](#0-1) 

In `sign_transfer`, the `message` is populated from `DestinationChainMsg::DestHexMsg` and then hashed:

```rust

`DestinationChainMsg::destination_msg()` returns `Some(bytes)` only for the `DestHexMsg` variant: [4](#0-3) 

**Solana verification side — `serialize_for_near()` never includes `message`:**

`FinalizeTransferPayload` has no `message` field at all, and `serialize_for_near()` serializes only: prefix, destination\_nonce, transfer\_id, token, amount, recipient, fee\_recipient — stopping there with no message bytes:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
    // ← no message field
}
``` [5](#0-4) 

```rust
fn serialize_for_near(&self, params: Self::AdditionalParams) -> Result<Vec<u8>> {
    // ... prefix, destination_nonce, transfer_id, token, amount, recipient, fee_recipient
    // ← message is never written
}
``` [6](#0-5) 

`SignedPayload::verify_signature` hashes the output of `serialize_for_near` and checks it against the MPC signature: [7](#0-6) 

**Contrast with StarkNet (consistent):**

StarkNet's `to_borsh()` conditionally appends the message when `Some`, matching NEAR's behavior: [8](#0-7) 

Solana is the only destination chain whose payload serializer omits the `message` field entirely.

---

### Impact Explanation

When a user initiates a transfer to Solana with `msg: '{"DestHexMsg":"<hex>"}'`:

1. Funds are locked or burned on the source chain.
2. NEAR's `sign_transfer` computes `keccak256(borsh(TransferMessagePayload { ..., message: [non-empty] }))` and requests an MPC signature over this hash.
3. The relayer submits the signed payload to Solana's `finalize_transfer`.
4. Solana's `verify_signature` computes `keccak256(serialize_for_near(...))` which **does not include the message bytes**.
5. The two hashes differ → `SignatureVerificationFailed` → the instruction reverts.
6. Because the nonce is not marked used (the instruction reverted), the relayer can retry, but every retry will produce the same hash mismatch and fail.
7. The source-chain funds are permanently frozen: locked in the vault or burned with no corresponding mint on Solana.

Impact class: **Critical — Permanent freezing / irrecoverable lock of user funds in the bridge flow.**

---

### Likelihood Explanation

Any unprivileged user can trigger this by calling `init_transfer` (or the equivalent on EVM/StarkNet) targeting a Solana recipient and supplying a non-empty `msg` string of the form `{"DestHexMsg":"<any hex>"}`. The NEAR bridge accepts arbitrary `msg` strings at `init_transfer` time without chain-specific validation. The freeze is deterministic and requires no special timing or race condition.

---

### Recommendation

Add a `message: Option<Vec<u8>>` field to `FinalizeTransferPayload` and serialize it in `serialize_for_near()` to match NEAR's `encode_hashable()` logic:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
    pub message: Option<Vec<u8>>,   // ← add this
}

fn serialize_for_near(&self, params: Self::AdditionalParams) -> Result<Vec<u8>> {
    // ... existing fields ...
    // 7. message (only if Some and non-empty, matching encode_hashable)
    if let Some(msg) = &self.message {
        if !msg.is_empty() {
            msg.serialize(&mut writer)?;
        }
    }
    ...
}
```

Alternatively, if Solana transfers are not intended to carry destination messages, add an explicit validation in NEAR's `sign_transfer` that rejects transfers to `ChainKind::Sol` when `message` is non-empty, preventing the mismatch at the source.

---

### Proof of Concept

1. User calls `init_transfer` on NEAR (or EVM) with recipient `sol:<pubkey>` and `msg: '{"DestHexMsg":"deadbeef"}'`.
2. NEAR stores the transfer; `sign_transfer` is called by a relayer.
3. NEAR computes:
   ```
   hash_near = keccak256(borsh(TransferMessagePayload {
       prefix: 0x00,
       destination_nonce: N,
       transfer_id: { origin_chain, origin_nonce },
       token_address: ...,
       amount: ...,
       recipient: ...,
       fee_recipient: ...,
       message: [0xde, 0xad, 0xbe, 0xef],   // ← included
   }))
   ```
4. MPC signs `hash_near`.
5. Relayer submits `finalize_transfer` to Solana with the signed payload.
6. Solana computes:
   ```
   hash_sol = keccak256(serialize_for_near({
       prefix: 0x00,
       destination_nonce: N,
       transfer_id: ...,
       token: ...,
       amount: ...,
       recipient: ...,
       fee_recipient: ...,
       // message bytes absent
   }))
   ```
7. `hash_near ≠ hash_sol` → `secp256k1_recover` returns a key that does not match `derived_near_bridge_address` → `ErrorCode::SignatureVerificationFailed` → revert.
8. Every retry produces the same failure. Funds are permanently frozen.

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
