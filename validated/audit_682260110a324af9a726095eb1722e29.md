Audit Report

## Title
`addLiquidity` and `removeLiquidity` lack `whenNotPaused` modifier, allowing liquidity operations to bypass pool pause — (File: metric-core/contracts/MetricOmmPool.sol)

## Summary

`MetricOmmPool.swap` is protected by `whenNotPaused`, but `addLiquidity` and `removeLiquidity` carry only `nonReentrant`. When the pool is paused to halt activity during a security incident, any unprivileged caller can still add or remove liquidity directly on the pool or through the `MetricOmmPoolLiquidityAdder` periphery, bypassing the intended freeze. This exposes LPs to direct principal loss (adding into a corrupted pool) and enables an attacker to exit inflated positions (removing before remediation).

## Finding Description

`MetricOmmPool` defines `whenNotPaused` at L174–177 which calls `_checkNotPaused()` and reverts when `pauseLevel != 0`.

`swap` at L217–224 correctly applies `whenNotPaused nonReentrant(PoolActions.SWAP)`.

`addLiquidity` at L182–196 is declared `external nonReentrant(PoolActions.ADD_LIQUIDITY)` with no `whenNotPaused`. `removeLiquidity` at L199–212 is likewise `external nonReentrant(PoolActions.REMOVE_LIQUIDITY)` with no `whenNotPaused`.

The `MetricOmmPoolLiquidityAdder._addLiquidity` at L183–207 calls `IMetricOmmPoolActions(pool).addLiquidity(...)` directly, so the bypass is reachable through the standard user-facing entry points `addLiquidityExactShares` (L56–68) and `addLiquidityWeighted` (L88–116). The pool documentation at L147 states that `pausePool` "Disables swaps (and generally 'active' trading); use for incidents, upgrades, or market conditions," establishing the intent that the pause should halt all active pool operations, not only swaps.

## Impact Explanation

**`addLiquidity` without pause guard:** The pool is paused because its state is unsafe (e.g., stale oracle, corrupted bin accounting). A user with a pending transaction routed through `MetricOmmPoolLiquidityAdder` deposits real tokens into the compromised pool. When the pool is unpaused, arbitrageurs immediately drain value from those newly-added positions at the skewed price bands, causing direct loss of deposited principal with no recourse. This is a direct loss of user principal, meeting the allowed impact gate.

**`removeLiquidity` without pause guard:** If the pool is paused after an exploit has inflated certain positions (e.g., via a swap-path bug), the attacker can call `removeLiquidity` (which only requires `msg.sender == owner` at L206) before further remediation. The attacker exits with inflated proceeds while honest LPs are left with a depleted pool, breaking the solvency invariant that pool balances must cover all LP claims.

## Likelihood Explanation

- The pause mechanism is live and permissioned, reachable through `MetricOmmPoolFactory.pausePool` (admin) and `protocolPausePool` (protocol owner).
- Once paused, **any** unprivileged address can call `addLiquidity` or `removeLiquidity` directly on the pool, or route through `MetricOmmPoolLiquidityAdder`, with no special role or setup required.
- `MetricOmmPoolLiquidityAdder` is the standard periphery entry point for LPs; users with pending mempool transactions at pause time will have their transactions succeed when they should revert.
- The attacker scenario for `removeLiquidity` requires only that the attacker already holds a position (which they would, having exploited the pool), and that the admin pauses before fully remediating.

## Recommendation

Add the `whenNotPaused` modifier to both `addLiquidity` and `removeLiquidity` in `metric-core/contracts/MetricOmmPool.sol`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

This ensures the pause level uniformly halts all state-mutating pool operations, consistent with the documented intent and with the guard already applied to `swap`.

## Proof of Concept

```
1. Admin calls MetricOmmPoolFactory.pausePool(pool)
   → pool.pauseLevel = 1

2. pool.swap(...) → reverts with PoolPaused()   ✓ (guarded)

3. MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, salt, deltas, max0, max1, ext)
   → internally calls pool.addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData)
   → pool.addLiquidity has NO whenNotPaused check (L188)
   → call SUCCEEDS, tokens are pulled from user and deposited into the paused/compromised pool

4. Pool is unpaused after incident resolution.
   Arbitrageurs swap against the pool at the skewed price bands.
   LP who added in step 3 suffers immediate loss of deposited principal.

5. (Attacker path) Attacker holds inflated position from prior exploit.
   pool.removeLiquidity(attacker, salt, deltas, ext)
   → only checks msg.sender == owner (L206), no pause check
   → call SUCCEEDS, attacker exits with inflated proceeds before remediation
   → honest LPs left with depleted pool balances
```
