### Title
Valid Transactions Writing Zero to Storage Are Permanently Unprovable Due to Unconditional Storage-Deletion Rejection in `validate_virtual_os_state_diff` — (File: `crates/starknet_transaction_prover/src/running/committer_utils.rs`)

---

### Summary

The transaction prover's `validate_virtual_os_state_diff` function unconditionally rejects any state diff that contains a storage update whose value is `Felt::ZERO`. Because writing zero to a Starknet storage slot is semantically equivalent to a deletion in the Patricia trie, and because the prover's facts-DB construction inserts dummy nodes instead of real preimages for sibling hashes, the prover cannot handle such updates. The result is that any valid Invoke transaction whose execution writes zero to at least one storage slot is permanently unprovable: the prover returns an authoritative-looking `InvalidStateDiff` error, even though the sequencer itself accepts and executes the transaction without issue.

---

### Finding Description

**Root cause — `validate_virtual_os_state_diff`** [1](#0-0) 

The function iterates over every `(address, key, value)` triple in the committer `StateDiff`. If `value.0 == Felt::ZERO` it immediately returns `ProofProviderError::InvalidStateDiff`. There is no exception for nonces, class hashes, or any other field — only storage values are checked, but the check is absolute: **any** zero storage value is rejected.

**Why the constraint exists**

The companion function `add_dummy_nodes_for_orphan_hashes` explains the design intent: [2](#0-1) 

RPC storage proofs supply sibling hashes but not their preimages. The committer needs those preimages when it must traverse into a sibling subtree — which only happens during a deletion (leaf removal). To avoid fetching full preimages, the prover inserts dummy binary nodes with zero hashes and bans deletions. The ban is implemented by rejecting zero-valued storage writes.

**Call path — zero value reaches the check during every prove request**

1. `VirtualSnosProver::prove_transaction` → `run_and_prove` → `runner.run_virtual_os`
2. Inside `Runner::run_virtual_os`, `virtual_block_executor.execute()` runs the transactions and returns `VirtualBlockExecutionData` containing `state_diff: StateMaps`. [3](#0-2) 

3. `create_virtual_os_hints` calls `storage_proofs_provider.get_storage_proofs(block_number, &execution_data, ...)`.
4. Inside `get_storage_proofs` (called from `storage_proofs.rs`, two call sites), `validate_virtual_os_state_diff` is invoked with the state diff produced in step 2.
5. If any storage slot was written to zero, the function returns `ProofProviderError::InvalidStateDiff("Storage deletion not allowed …")`, which propagates all the way back to the caller of `prove_transaction`.

**Attacker-controlled trigger**

Any unprivileged user can submit an Invoke V3 transaction (with all `max_price_per_unit` fields set to zero as required by `validate_zero_fee_resource_bounds`) whose Cairo logic writes `0` to any storage variable. Examples:

- Resetting a counter: `storage_write(key, 0)`
- Clearing an allowance or balance to zero in an ERC-20
- Any `delete` / `reset` pattern in application contracts

The sequencer's blockifier executes such transactions successfully (zero is a perfectly valid storage value). The state diff produced by `to_state_diff()` will contain the zero entry, and `validate_virtual_os_state_diff` will reject it. [4](#0-3) 

`state_maps_to_committer_state_diff` passes all storage values verbatim, including zeros, into the `StateDiff` that is then validated.

---

### Impact Explanation

The transaction prover is an RPC service. When it returns `ProofProviderError::InvalidStateDiff` for a transaction that the sequencer accepted and executed correctly, it produces an authoritative-looking wrong result: the caller is told the transaction's state diff is invalid, when it is not. The proof facts (`ProofFacts`) are never generated, so the transaction cannot be verified on L1 through the proving path. This matches:

> **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

The error message `"Storage deletion not allowed: try to delete storage at address …, key …"` misleads callers into believing the transaction attempted an illegal operation.

---

### Likelihood Explanation

Writing zero to storage is a routine operation in Starknet contracts (clearing approvals, resetting counters, burning tokens to zero balance). Any user who submits such a transaction for proving will hit this rejection deterministically and repeatedly. No special privileges, no race conditions, no probabilistic conditions — the trigger is purely a function of the transaction's execution output.

---

### Recommendation

1. **Fetch full sibling preimages for affected paths.** When the state diff contains a zero-valued storage update (a deletion), request the full Merkle path preimages from the RPC node so the committer can traverse into the sibling subtree correctly. This removes the need for dummy nodes on deletion paths and lifts the ban.

2. **If full preimage fetching is deferred**, at minimum replace the hard `InvalidStateDiff` error with a distinct, documented error code (e.g., `ProofProviderError::StorageDeletionUnsupported`) so callers can distinguish "transaction is invalid" from "this prover implementation does not yet support storage deletions." The current error message falsely attributes the failure to the transaction.

3. **Add a test** that submits a transaction writing zero to storage and asserts the prover either succeeds or returns the correct `StorageDeletionUnsupported` variant — not `InvalidStateDiff`.

pub(crate) fn validate_virtual_os_state_diff(
    state_diff: &StateDiff,
) -> Result<(), ProofProviderError> {
    // validate no storage deletions.
    for (address, storage_diffs) in &state_diff.storage_updates {
        for (key, value) in storage_diffs {
            if value.0 == Felt::ZERO {
                return Err(ProofProviderError::InvalidStateDiff(format!(
                    "Storage deletion not allowed: try to delete storage at address {address:?}, \
                     key {key:?}"
                )));
            }
        }
    }
