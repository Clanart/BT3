### Title
Legacy EIP-712 `Tx` Type Schema Omits `timeout_height` Present in Signed Message Data, Enabling Signature Bypass for Transaction Expiry - (File: `ethereum/eip712/eip712_legacy.go`)

---

### Summary

In the legacy EIP-712 path, the hardcoded `Tx` type schema deliberately excludes `timeout_height`, yet the Amino JSON bytes passed as the EIP-712 message data include `timeout_height` whenever it is non-zero. Because EIP-712 hashing ignores message fields absent from the type schema, the signed hash never commits to `timeout_height`. A malicious relayer or fee-payer can strip or alter `timeout_height` on a signed transaction without invalidating the user's signature, allowing the transaction to be replayed after the user's intended expiry block.

---

### Finding Description

**Root cause — type schema vs. message data mismatch:**

The `Tx` type schema in `extractMsgTypes` is hardcoded and explicitly omits `timeout_height`: [1](#0-0) 

The comment acknowledges the omission: *"Note timeout_height was removed because it was not getting filled with the legacyTx."*

However, in `VerifySignature` inside the legacy ante handler, `StdSignBytes` is called with the live transaction's `GetTimeoutHeight()` value: [2](#0-1) 

When `timeout_height` is non-zero, the Cosmos SDK's `StdSignDoc` serialises it as `"timeout_height": "<N>"` in the Amino JSON. That JSON is then passed directly to `LegacyWrapTxToTypedData`: [3](#0-2) 

Inside `LegacyWrapTxToTypedData`, the raw JSON is unmarshalled into `txData` (the EIP-712 `Message` map): [4](#0-3) 

The resulting `TypedData` therefore has `timeout_height` in `Message` but **not** in `Types["Tx"]`. Per EIP-712, `apitypes.TypedDataAndHash` encodes only fields declared in the type schema; undeclared fields in the message map are silently ignored: [5](#0-4) 

**Contrast with the non-legacy path:**

`decodeProtobufSignDoc` explicitly rejects any sign-doc with `TimeoutHeight != 0`: [6](#0-5) 

No equivalent guard exists in the legacy path.

**The legacy ante handler does validate `timeout_height` at the Cosmos layer:** [7](#0-6) 

But that check runs *after* EIP-712 signature verification and operates on the (potentially attacker-modified) transaction field, not on the signed hash.

---

### Impact Explanation

Because the EIP-712 hash never covers `timeout_height`, an attacker who can relay or forward a signed legacy EIP-712 transaction can:

1. **Strip the timeout** — change `timeout_height = X` to `0` (no expiry). The user's signature remains valid. The transaction can be submitted at any future block, long after the user intended it to expire.
2. **Extend the timeout** — replace `timeout_height = X` with a larger value, again without invalidating the signature.

This is a direct EIP-712 authorization bypass: the user's signature is accepted for a transaction whose `timeout_height` field the user never authorised. It enables replay of time-limited transactions past their intended expiry, matching the allowed High impact: *"EIP-712 authorization … signer verification bypass enabling replay … or unauthorized account/code mutation."*

---

### Likelihood Explanation

The legacy EIP-712 path (`ExtensionOptionsWeb3Tx`) is specifically designed for **fee-delegation** scenarios where the signer and the submitter are different parties. The user signs the EIP-712 hash off-chain and hands the signed bytes to a relayer or fee-payer who broadcasts the transaction. That intermediary has full control over the Cosmos transaction fields, including `timeout_height`, before submission. A malicious or compromised relayer can trivially exploit this without any special privileges. The only precondition is that the user set a non-zero `timeout_height`, which is a normal and documented Cosmos SDK feature.

---

### Recommendation

Add `timeout_height` to the `Tx` type schema in `extractMsgTypes` so the signed hash commits to it:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    {Name: "timeout_height", Type: "string"},  // add this
},
```

Alternatively, mirror the non-legacy path and reject legacy EIP-712 transactions that carry a non-zero `timeout_height`, preventing the ambiguity entirely.

---

### Proof of Concept

1. User constructs a legacy EIP-712 Cosmos transaction with `timeout_height = 500` (expires at block 500).
2. `StdSignBytes` serialises `"timeout_height":"500"` into the Amino JSON.
3. `LegacyWrapTxToTypedData` places `"timeout_height":"500"` in `TypedData.Message` but the `Tx` type schema has no `timeout_height` entry.
4. `apitypes.TypedDataAndHash` computes the hash ignoring `timeout_height`; the user signs this hash and embeds it in `ExtensionOptionsWeb3Tx.FeePayerSig`.
5. A malicious relayer receives the signed transaction and sets `timeout_height = 0` in the Cosmos transaction body.
6. At block 600 (after the original expiry), the relayer submits the modified transaction.
7. `LegacyEip712SigVerificationDecorator` calls `VerifySignature`:
   - `StdSignBytes` now serialises `"timeout_height":"0"` (or omits it).
   - `LegacyWrapTxToTypedData` produces the same EIP-712 hash as in step 4 (since `timeout_height` is not in the type schema).
   - Signature verification passes.
8. `TxTimeoutHeightDecorator` passes because `timeout_height = 0` means no expiry.
9. The transaction executes at block 600, violating the user's intent that it expire at block 500. [1](#0-0) [2](#0-1) [8](#0-7)
