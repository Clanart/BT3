### Title
Unchecked ERC20 Transfer Return Value in `charge_fee` Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in the StarkNet OS program calls the fee token's `transfer` entry point via `non_reverting_select_execute_entry_point_func` but **completely discards the return value**, never verifying that the ERC20 transfer actually succeeded. This is the direct Cairo analog of the M-02 bug: if the fee token returns `false` (success=false) without reverting, the OS proceeds as if the fee was paid, allowing users to transact for free.

---

### Finding Description

In `charge_fee` (lines 111–165 of `transaction_impls.cairo`), the OS constructs an `ExecutionContext` targeting the fee token's `transfer` selector and calls:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
``` [1](#0-0) 

The return tuple `(retdata_size, retdata, is_deprecated)` from `non_reverting_select_execute_entry_point_func` is **not bound to any variable** — it is silently dropped. The ERC20 `transfer` function returns a boolean success value (`(success: felt)` or `(success: bool)`). The OS never reads or asserts this value.

`non_reverting_select_execute_entry_point_func` only asserts `is_reverted = 0` (i.e., the call did not revert at the VM level):

```cairo
assert is_reverted = 0;
return (retdata_size, retdata, is_deprecated);
``` [2](#0-1) 

This means: if the fee token's `transfer` executes without reverting but returns `false` (a valid ERC20 behavior per EIP-20), the OS accepts the proof and finalizes the block with no fee collected.

**Contrast with how `__validate__` return values are handled.** Every other call site of `non_reverting_select_execute_entry_point_func` captures and asserts the return data:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(
    block_context=block_context, execution_context=validate_execution_context
);
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = VALIDATED;
}
``` [3](#0-2) 

The same pattern is applied for `__validate_declare__`: [4](#0-3) 

And for `__validate_deploy__`: [5](#0-4) 

The `charge_fee` call is the **only** invocation of `non_reverting_select_execute_entry_point_func` that does not inspect `retdata` at all. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

If the fee token contract's `transfer` function returns `false` without reverting (which is valid per the ERC20 standard and is the exact scenario described in M-02), the OS will:

1. Execute the user's transaction.
2. Call `transfer` on the fee token — which silently fails.
3. Finalize the block with the transaction included and committed to state.
4. The sequencer receives zero fee tokens despite processing the transaction.

At scale, this allows any user to submit transactions that are executed and committed on-chain without paying fees, constituting a direct loss of funds to the protocol/sequencer.

---

### Likelihood Explanation

The current StarkNet fee tokens (STRK, ETH) are audited contracts that revert on insufficient balance, making exploitation unlikely under normal conditions. However:

- The fee token address is loaded from `block_context.os_global_context.starknet_os_config.fee_token_address` — a configuration value.
- Any future fee token upgrade or alternative fee token that follows the EIP-20 spec (returning `false` instead of reverting) would immediately expose this path.
- The OS protocol layer should not rely on the fee token's revert behavior as the sole safety mechanism; it must enforce correctness at the OS level. [7](#0-6) 

---

### Recommendation

Capture and assert the return value of the fee token `transfer` call in `charge_fee`, analogous to how `__validate__` return values are checked. The ERC20 `transfer` entry point returns a single boolean; the OS should assert it equals `TRUE` (1):

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = TRUE;  // transfer must return success=true
}
```

This mirrors the existing pattern used for all `__validate__` entry points and closes the unchecked-return-value gap.

---

### Proof of Concept

1. Deploy a fee token contract that implements `transfer(recipient, amount) -> (success: felt)` but returns `0` (false) instead of reverting when the sender has insufficient balance.
2. Submit any V3 transaction (invoke/declare/deploy_account) with `max_fee > 0` from an account with zero fee token balance.
3. The OS calls `charge_fee`, which calls `transfer` on the fee token.
4. The fee token returns `retdata = [0]` (false) without reverting — `is_reverted = 0` passes the `non_reverting_select_execute_entry_point_func` assertion.
5. `charge_fee` returns without checking `retdata`.
6. The transaction is finalized in the block. The sequencer's balance is unchanged. The user paid no fee. [1](#0-0)

func charge_fee{
    range_check_ptr,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,
    outputs: OsCarriedOutputs*,
}(block_context: BlockContext*, tx_execution_context: ExecutionContext*) {
    alloc_locals;

    local tx_info: TxInfo* = tx_execution_context.execution_info.tx_info;
    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    if (max_fee == 0) {
        return ();
    }

    local low_actual_fee;
    %{ LoadActualFee %}
    local calldata: TransferCallData = TransferCallData(
        recipient=block_context.block_info_for_execute.sequencer_address,
        amount=Uint256(low=low_actual_fee, high=0),
    );

    // Verify that the charged amount is not larger than the transaction's max_fee field.
    assert_nn_le(calldata.amount.low, max_fee);

    // TODO(ilya, 01/01/2026): Consider caching the fee_token_class_hash.
    local fee_token_address = block_context.os_global_context.starknet_os_config.fee_token_address;
    let (fee_state_entry: StateEntry*) = dict_read{dict_ptr=contract_state_changes}(
        key=fee_token_address
    );
    let (__fp__, _) = get_fp_and_pc();
    // Use block_info directly from block_context, so that charge_fee will always run in
    // execute-mode rather than validate-mode.
    local execution_context: ExecutionContext = ExecutionContext(
        entry_point_type=ENTRY_POINT_TYPE_EXTERNAL,
        class_hash=fee_state_entry.class_hash,
        calldata_size=TransferCallData.SIZE,
        calldata=&calldata,
        execution_info=new ExecutionInfo(
            block_info=block_context.block_info_for_execute,
            tx_info=tx_info,
            caller_address=tx_info.account_contract_address,
            contract_address=fee_token_address,
            selector=TRANSFER_ENTRY_POINT_SELECTOR,
        ),
        deprecated_tx_info=tx_execution_context.deprecated_tx_info,
    );

    let remaining_gas = DEFAULT_INITIAL_GAS_COST;
    non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
        block_context=block_context, execution_context=&execution_context
    );
    return ();
}
