Let me look more closely at `insert_entry_to_blob` and `add_leaf` to confirm the exact behavior when duplicates are inserted, and check the Python binding exposure.

### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Validation, Silently Corrupting DataLayer Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key guard that `insert()` enforces. When a caller supplies two entries with the same `KeyId` — or a `KeyId` already present in the tree — both leaf nodes are written to the blob at distinct `TreeIndex` positions, while `BlockStatusCache::add_leaf` silently overwrites the `key_to_index` HashMap entry, leaving the cache tracking only one of them. The Merkle root hash is then computed over both leaves, producing a root that does not correspond to any valid key-value set. The Python binding `py_batch_insert` exposes this path directly to callers.

---

### Finding Description

`insert()` enforces two guards before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert()` for **at most the last two items** (only when `leaf_count <= 1`). All remaining items are written directly via `insert_entry_to_blob()` with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    // ... build block ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // no guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When `leaf_count > 1` (the common production case), **every** item in the batch bypasses the guard entirely. [3](#0-2) 

`insert_entry_to_blob` calls `add_leaf`, which calls `HashMap::insert` — a silent overwrite with no error on collision:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
``` [4](#0-3) 

The result: the blob contains **two** leaf nodes for the same `KeyId` at distinct `TreeIndex` positions, but the cache tracks only the last one. The root hash is computed from both leaves, producing a hash that does not correspond to any valid key-value set. [5](#0-4) 

The Python binding `py_batch_insert` passes caller-supplied `keys_values` and `hashes` directly into `batch_insert` with no deduplication: [6](#0-5) 

---

### Impact Explanation

The DataLayer Merkle tree root hash is the commitment that Chia nodes use to verify DataLayer state. A corrupted root means:

1. **Root hash mismatch**: The committed root does not accurately represent the actual key-value set. Any downstream consumer comparing the root against an expected value will see a divergence.
2. **Proof inconsistency**: `get_proof_of_inclusion` for the duplicate key will only locate one of the two leaves (the one in the cache), while the root was computed from both. A verifier checking the proof against the root will find it invalid, or — depending on traversal — may accept a proof for a leaf that is not the canonical one.
3. **Silent corruption**: `batch_insert` returns `Ok(())` with no indication of the problem. `check_integrity` would detect the divergence (`leaf_count != key_to_index_cache_length`), but it is only called on drop or explicitly — after the corrupted root may already have been committed.

This matches the allowed High impact: **DataLayer Merkle blob/delta logic corrupts tree roots via untrusted input**.

---

### Likelihood Explanation

The Python binding `py_batch_insert` is a public API on `MerkleBlob`, exposed in the wheel stub at `wheel/python/chia_rs/datalayer.pyi`. [7](#0-6) 

Any caller — including DataLayer Python code ingesting data from an external feed — can supply a list with duplicate `KeyId` values. The function provides no contract or documentation warning against duplicates, and the failure is entirely silent at the call site. The existing test `test_batch_insert` only uses unique keys, so the gap is untested. [8](#0-7) 

---

### Recommendation

**Short term:** Add a deduplication check inside the `batch_insert` loop before calling `insert_entry_to_blob`. Maintain a local `HashSet<KeyId>` over the batch items and check `block_status_cache.contains_key(key)` for each item, returning `Err(Error::KeyAlreadyPresent())` on collision — mirroring the guard in `insert()`.

**Long term:** Refactor `batch_insert` to reuse the validation logic from `insert()` rather than duplicating (and omitting) it. Consider replacing the `Vec` input with a type that enforces uniqueness at the boundary, or add an invariant test that calls `check_integrity` after every `batch_insert` in the test suite.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Pre-insert 2 leaves so leaf_count > 1 — all batch items now bypass insert()
blob.insert(KeyId(0), ValueId(0), hashlib.sha256(b"seed0").digest())
blob.insert(KeyId(1), ValueId(1), hashlib.sha256(b"seed1").digest())

root_before = blob.get_root_hash()

# batch_insert with duplicate KeyId(2) — no error raised
blob.batch_insert(
    [(KeyId(2), ValueId(2)), (KeyId(2), ValueId(99))],   # same key, two entries
    [hashlib.sha256(b"hash_a").digest(), hashlib.sha256(b"hash_b").digest()],
)

root_after = blob.get_root_hash()
# root_after is computed from a tree with TWO leaf nodes for KeyId(2).
# The cache only knows about one. The root is corrupted.
# blob.check_integrity() will raise IntegrityKeyToIndexCacheLength.
```

The Rust-level analog:

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();
// bring leaf_count to 2
blob.insert(KeyId(0), ValueId(0), &sha256_num(&0), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &sha256_num(&1), InsertLocation::Auto {}).unwrap();

// duplicate KeyId(2) in the batch — both succeed silently
blob.batch_insert(vec![
    ((KeyId(2), ValueId(2)),  sha256_num(&2)),
    ((KeyId(2), ValueId(99)), sha256_num(&99)),
]).unwrap();  // Ok(()) — no error

// blob now has two leaf nodes for KeyId(2); root hash is corrupted
blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheLength(5, 4)
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }
```

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
