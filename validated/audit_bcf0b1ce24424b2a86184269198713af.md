### Title
`nativeFee` ETH Permanently Locked in Bridge Contract with No Distribution or Refund Mechanism — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.initTransfer`, the user-supplied `nativeFee` ETH is subtracted from `msg.value` to compute `extensionValue` (the Wormhole message fee), but the `nativeFee` portion is never forwarded to any relayer, fee recipient, or returned to the caller. It accumulates permanently in the bridge contract with no withdrawal, distribution, or refund path.

---

### Finding Description

`initTransfer` is the primary user-facing entry point for initiating a cross-chain transfer. It accepts a `nativeFee` parameter (user-controlled `uint128`) and computes:

- **ERC20 path** (`tokenAddress != address(0)`):
  ```
  extensionValue = msg.value - nativeFee;
  ```
- **Native ETH path** (`tokenAddress == address(0)`):
  ```
  extensionValue = msg.value - amount - nativeFee;
  ``` [1](#0-0) 

`extensionValue` is then passed to `initTransferExtension`, which in `OmniBridgeWormhole` forwards it to `_wormhole.publishMessage{value: value}(...)` — the Wormhole message fee. [2](#0-1) 

The `nativeFee` amount is encoded into the Wormhole payload so the NEAR bridge knows the intended relayer incentive: [3](#0-2) 

However, the `nativeFee` ETH itself is **never transferred out of the bridge contract**. It is not sent to a fee recipient, not forwarded to Wormhole, and not refunded to `msg.sender`. There is no `withdraw`, `receive`, or `fallback` function in `OmniBridge.sol` that would allow recovery of this ETH. Every call to `initTransfer` with `nativeFee > 0` permanently locks that ETH in the contract. [4](#0-3) 

---

### Impact Explanation

Every unprivileged user who calls `initTransfer` with a non-zero `nativeFee` (which is the normal case — users set `nativeFee` to incentivize relayers on the destination chain) permanently loses that ETH. The ETH accumulates in the bridge contract with no recovery path. This is an irrecoverable lock of user funds matching the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

**Impact: High**

---

### Likelihood Explanation

Any user who wants their transfer relayed promptly must set `nativeFee > 0`. This is the standard usage pattern. The vulnerability is triggered on every such call. No special conditions, privileged access, or race conditions are required.

**Likelihood: High**

---

### Recommendation

The `nativeFee` ETH must be explicitly routed. Options:

1. **Forward `nativeFee` to a designated fee collector address** at the time of `initTransfer`, so relayers can claim it on the EVM side.
2. **Refund `nativeFee` to `msg.sender`** if the protocol's relayer incentive is handled entirely on the NEAR side (making `nativeFee` ETH payment unnecessary on EVM).
3. **Add a strict check** `require(msg.value == extensionValue, "excess ETH")` to prevent users from accidentally locking ETH as `nativeFee` when it serves no on-chain purpose.

---

### Proof of Concept

**Scenario**: Alice bridges 100 USDC from EVM to NEAR, setting `nativeFee = 0.01 ether` to incentivize the relayer.

```
Alice calls:
  initTransfer(
    tokenAddress = USDC,
    amount      = 100e6,
    fee         = 1e6,
    nativeFee   = 0.01 ether,   // Alice expects this to pay the relayer
    recipient   = "alice.near",
    message     = ""
  )
  { value: 0.01 ether + wormhole.messageFee() }
```

Inside `initTransfer`:
```
extensionValue = msg.value - nativeFee
               = (0.01 ether + messageFee) - 0.01 ether
               = messageFee
```

`extensionValue` (= `messageFee`) is sent to Wormhole. [5](#0-4) 

`nativeFee` = `0.01 ether` remains in the bridge contract. No code path distributes or refunds it. [6](#0-5) 

Alice's `0.01 ETH` is permanently locked. The relayer on NEAR is paid from NEAR-side reserves, not from this ETH. The locked ETH is irrecoverable.
