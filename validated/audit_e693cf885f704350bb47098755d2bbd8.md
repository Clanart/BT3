Audit Report

## Title
Unrestricted `refundETH` / `sweepToken` / `unwrapWETH9` allow any caller to drain router-held ETH and tokens — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH`, `sweepToken`, and `unwrapWETH9` carry no caller restriction and operate on the router's entire balance. ETH surplus accumulates in the router whenever a user calls `exactOutputSingle` (or any other `payable` swap entry point) directly with `msg.value > actualAmountIn`, because `pay` wraps only the exact amount owed and leaves the remainder in the contract. Any unprivileged address can immediately call `refundETH()` and claim that surplus, constituting a direct theft of user principal.

## Finding Description

**Root cause — `pay` wraps only the exact owed amount:**

`PeripheryPayments.pay` (lines 73–78) handles the WETH/native branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // wraps exactly `value`
        IERC20(WETH).safeTransfer(recipient, value);
    }
```

When `nativeBalance >= value`, only `value` (the pool-determined exact input) is wrapped and forwarded. Any surplus `nativeBalance - value` is left as raw ETH in the router with no accounting or lock.

**Surplus creation path:**

`exactOutputSingle` is `payable` (line 130). A user performing an exact-output WETH swap does not know the precise `amountIn` in advance; they send `amountInMaximum` as `msg.value`. The pool determines the actual `amountIn ≤ amountInMaximum`. The callback fires `_justPayCallback` → `pay(…, value = actualAmountIn)`. The difference `msg.value − actualAmountIn` is never refunded automatically and sits in the router.

**Drain path:**

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // sends entire balance to any caller
    }
}
```

No `msg.sender` check, no per-user accounting. Any address that calls `refundETH()` in a subsequent transaction receives the full ETH balance of the router.

The same surface applies to `sweepToken(token, 0, attacker)` for any ERC-20 balance that reaches the router (e.g., WETH output routed through the contract in a multi-hop flow with `recipient: address(router)`), and to `unwrapWETH9` which also accepts a caller-controlled `recipient`.

**Why existing guards are insufficient:**

- `receive()` blocks direct ETH pushes (non-WETH senders), but ETH enters legitimately through `payable` swap functions — this guard is irrelevant to the surplus.
- `amountInMaximum` slippage check (line 145) only prevents overpayment *to the pool*; it does not prevent the user from sending excess ETH as `msg.value`.
- The `multicall` + `refundETH()` pattern shown in tests is never enforced by the contract; calling `exactOutputSingle` directly is fully supported and leaves no automatic refund.

## Impact Explanation

Direct loss of user ETH principal. The stranded amount equals `msg.value − actualAmountIn`, which can be as large as `amountInMaximum − actualAmountIn` — an arbitrarily large value chosen by the user as their slippage buffer. `refundETH()` transfers the entire router ETH balance to the attacker in a single call. This is a Critical/High direct loss of user funds meeting Sherlock thresholds.

## Likelihood Explanation

Any user who calls `exactOutputSingle` (or `exactInput` / `exactOutput`) directly with `msg.value > actualAmountIn` — a normal and expected usage pattern for exact-output ETH swaps — creates the exploitable condition. The attacker needs only to monitor the mempool or router balance and call `refundETH()` in the next block. No special privileges, no setup, fully repeatable.

## Recommendation

1. **Auto-refund surplus ETH at the end of every `payable` swap function.** After the swap and slippage check, call `_transferETH(msg.sender, address(this).balance)` if `address(this).balance > 0`.
2. **Restrict `refundETH` to `msg.sender` only** (already the case for ETH destination, but the function should be non-reentrant and ideally called only within a multicall context).
3. **Add per-call ETH accounting** using transient storage: record `msg.value` at entry, compute surplus after `pay`, and refund atomically before returning.
4. **Restrict `sweepToken` and `unwrapWETH9`** to only be callable within a `multicall` context (e.g., via a `onlyMulticall` modifier using transient storage), preventing standalone theft of any residual router balance.

## Proof of Concept

```solidity
// 1. User calls exactOutputSingle with excess ETH
uint256 actualIn = /* pool-determined, e.g. */ 1 ether;
uint256 msgValue = 2 ether; // amountInMaximum sent as msg.value

vm.prank(user);
router.exactOutputSingle{value: msgValue}(ExactOutputSingleParams({
    pool: address(pool),
    tokenIn: address(weth),
    tokenOut: address(token1),
    zeroForOne: true,
    amountOut: amountOut,
    amountInMaximum: uint128(msgValue),
    recipient: user,
    deadline: block.timestamp + 60,
    priceLimitX64: 0,
    extensionData: ""
}));
// Router now holds (msgValue - actualIn) = 1 ether surplus

// 2. Attacker drains it
uint256 attackerBefore = attacker.balance;
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance - attackerBefore, 1 ether); // stolen
assertEq(address(router).balance, 0);
```
