### Title
Empty `check_is_reverted` Leaves Transaction Revert Status Unconstrained, Enabling Sequencer to Silently Drop L1→L2 Messages and Steal Fees — (`File: execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted` in `execution_constraints.cairo` is a named validation function that performs **no validation at all**. The `is_reverted` value consumed by both `execute_invoke_function_transaction` and `execute_l1_handler_transaction` is sourced from an unconstrained hint (`%{ IsReverted %}`). Because the function that is supposed to constrain it is a no-op, a malicious sequencer can freely set `is_reverted = 1` for any transaction, causing the OS to skip execution while still charging fees and incrementing nonces — and, critically, to silently drop L1→L2 messages, permanently locking bridged funds.

---

### Finding Description

**Root cause — `execution_constraints.cairo` lines 20–22:**

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

The function accepts `is_reverted` as an argument and immediately returns. No `assert`, range check, or any other Cairo constraint is applied to the argument.

**Call site — `transaction_impls.cairo` lines 338–358 (invoke path):**

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    tempvar _dummy_return_value: ...;
}
``` [2](#0-1) 

`is_reverted` is a `local` variable whose only source of value is the hint `%{ IsReverted %}`. In Cairo's constraint model, a memory cell written only by a hint and never constrained by an `assert` is a **free variable** — the prover may assign it any field element. `check_is_reverted` was the designated enforcement point, but it is empty.

**Fee is charged regardless — `transaction_impls.cairo` line 361:**

```cairo
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [3](#0-2) 

The nonce is also incremented unconditionally before the `is_reverted` branch (line 311). So when `is_reverted` is forced to `1`, the user's nonce advances and fee is deducted, but the `__execute__` entry point is never called.

**L1 handler path — `transaction_impls.cairo` lines 383–390:**

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [4](#0-3) 

When `is_reverted` is forced to `1` for an L1 handler, the function returns **before** `consume_l1_to_l2_message` is called (line 444). The message is never written to `outputs.messages_to_l2`, so the OS output does not record it. The L1 bridge has already locked the funds; the L2 side never processes them. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / direct loss of funds.**

1. **L1→L2 message suppression (permanent fund loss):** A user deposits ETH or ERC-20 tokens via the StarkNet bridge. The L1 contract locks the funds and emits an L1→L2 message. The sequencer sets `is_reverted = 1` for the corresponding L1 handler. The OS skips `consume_l1_to_l2_message`, so the message is absent from the proven output. The funds are permanently locked in the L1 bridge with no recovery path.

2. **Fee theft on invoke transactions:** The sequencer forces `is_reverted = 1` for any invoke transaction. The nonce is incremented and the fee is charged (both unconditional), but `__execute__` is never called. The user loses the fee and the transaction has no effect. The ZK proof is still valid because `check_is_reverted` adds no constraints.

---

### Likelihood Explanation

The sequencer is the entity that provides all hints to the Cairo VM. No external party can prevent the sequencer from writing any value to the `is_reverted` memory cell. The only protection was supposed to be `check_is_reverted`, which is empty. Any sequencer operator — or an attacker who compromises the sequencer — can exploit this on every block, against any transaction or L1 message, with no on-chain detection.

---

### Recommendation

`check_is_reverted` must enforce that `is_reverted` is consistent with the actual execution outcome. At minimum it must constrain `is_reverted` to `{0, 1}`:

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (is_reverted - 1) = 0;
    return ();
}
```

More robustly, the OS should derive `is_reverted` from the actual entry-point return value (`entry_point_return_values.failure_flag` in `execute_entry_point.cairo`) rather than from a sequencer-supplied hint, eliminating the free-variable attack surface entirely.

---

### Proof of Concept

**Scenario: L1→L2 bridge deposit silently dropped**

1. User calls `deposit(amount=1000 ETH)` on the StarkNet L1 bridge. The bridge locks 1000 ETH and enqueues an L1→L2 message targeting the ERC-20 handler contract on L2.
2. The sequencer includes the L1 handler transaction in a block. In the OS execution, the hint `%{ IsReverted %}` writes `is_reverted = 1` to the local variable.
3. `check_is_reverted(1)` is called — it returns immediately with no assertion.
4. The `if (is_reverted != FALSE)` branch is taken; the function returns before `consume_l1_to_l2_message` executes.
5. The OS output contains no record of the L1→L2 message. The ZK proof is generated and verified successfully.
6. The L1 bridge sees the block finalized. The message is marked as consumed (or times out). The 1000 ETH is permanently locked.

The proof is valid because `check_is_reverted` imposes zero constraints on `is_reverted`, making the prover's choice of `1` indistinguishable from `0` at the proof-verification level.
