Looking at the Solana `finalize_transfer` instruction handler and the `FinalizeTransfer` accounts struct, I need to check whether the `recipient` account is validated against the signed payload.

### Title
Missing Recipient Validation in Solana `finalize_transfer` Allows Redirection of Bridged Tokens to Arbitrary Address — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary
The Solana `finalize_transfer` instruction accepts a MPC-signed `FinalizeTransferPayload` but never verifies that the `recipient` account passed in the instruction matches the intended recipient encoded in the signed payload. An attacker can front-run a legitimate finalization by submitting the same valid signed payload with a different `recipient` account, redirecting the bridged tokens to themselves.

### Finding Description
In `finalize_transfer.rs`, the `FinalizeTransfer` accounts struct declares `recipient` as a fully unchecked account with no constraint linking it to any field in the signed payload:

```rust
/// CHECK: this can be any type of account
pub recipient: UncheckedAccount<'info>,
``` [1](#0-0) 

The `token_account` is derived from `recipient` and `mint`, so tokens are delivered to whichever address the caller supplies as `recipient`:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    associated_token::mint = mint,
    associated_token::authority = recipient,
    ...
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
``` [2](#0-1) 

In the `process` function, the only fields consumed from the signed `data: FinalizeTransferPayload` are `destination_nonce`, `amount`, `fee_recipient`, and `transfer_id`. There is no `data.recipient` reference anywhere — the recipient is never read from the payload and never compared to `self.recipient.key()`:

```rust
pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
    UsedNonces::use_nonce(data.destination_nonce, ...)?;
    // transfers data.amount to self.token_account (derived from self.recipient)
    // self.recipient is never checked against data.*
    ...
    let payload = FinalizeTransferResponse {
        token: self.mint.key(),
        amount: data.amount,
        fee_recipient: data.fee_recipient.unwrap_or_default(),
        transfer_id: data.transfer_id,
    }.serialize_for_near(())?;
``` [3](#0-2) 

The `FinalizeTransferResponse` sent back to NEAR also omits the recipient, so NEAR cannot detect the misdirection.

The same structural gap exists in `finalize_transfer_sol.rs`, where `recipient` is also `UncheckedAccount` with no payload-binding constraint: [4](#0-3) 

This is the direct analog of the external report's vulnerability: a critical operation (token release) is executed without displaying or verifying the parameter (recipient) that determines where value flows, allowing a third party to substitute their own address.

### Impact Explanation
**Critical.** An attacker who observes a valid MPC-signed `FinalizeTransferPayload` on-chain (submitted by a relayer) can immediately resubmit the same payload with their own address as `recipient`. The nonce is consumed by whichever transaction lands first. If the attacker front-runs the relayer, the bridged tokens (native SOL or any SPL token) are minted or transferred to the attacker's token account. The legitimate recipient receives nothing, and the transfer is permanently marked as finalized (nonce consumed), making the loss irrecoverable.

### Likelihood Explanation
**High.** Signed payloads are submitted as public Solana transactions and are visible in the mempool. Front-running on Solana is feasible via priority fees. No special privilege is required — any unprivileged user can call `finalize_transfer`. The attack is straightforward and repeatable for every inbound transfer to Solana.

### Recommendation
Add an explicit constraint in the `FinalizeTransfer` accounts struct that binds `recipient` to the corresponding field in the signed payload. For example, if `FinalizeTransferPayload` contains a `recipient: Pubkey` field:

```rust
/// CHECK: verified against signed payload
#[account(address = data.payload.recipient @ ErrorCode::InvalidRecipient)]
pub recipient: UncheckedAccount<'info>,
```

If the payload currently does not include a recipient field, add one and include it in the MPC-signed data so the signature covers the intended destination address.

### Proof of Concept
1. Alice initiates a NEAR→Solana transfer of 1000 USDC. The NEAR bridge MPC produces a signed `FinalizeTransferPayload` with `destination_nonce = N`, `amount = 1000`, `transfer_id = T`.
2. A relayer broadcasts a Solana transaction calling `finalize_transfer(signed_payload, recipient=Alice, ...)`.
3. Attacker observes the transaction in the mempool, copies the signed payload, and submits `finalize_transfer(signed_payload, recipient=Attacker, ...)` with a higher priority fee.
4. Attacker's transaction lands first. `UsedNonces::use_nonce(N)` succeeds (nonce not yet consumed). 1000 USDC are minted/transferred to `Attacker`'s token account.
5. Relayer's transaction fails with nonce-already-used. Alice receives 0 tokens. The transfer is permanently finalized with the wrong recipient. [5](#0-4)

                        authority: self.authority.to_account_info(),
