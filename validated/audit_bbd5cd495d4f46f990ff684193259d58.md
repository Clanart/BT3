### Title
`BlockStatusCache::move_index` Fails to Update `key_to_index` / `leaf_hash_to_index` After Leaf Node Relocation — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::delete` physically moves a sibling node to `TreeIndex(0)` when the deleted leaf's parent is the root, then calls `BlockStatusCache::move_index` to update the cache. `move_index` only marks the old slot as free in `free_indexes`; it never updates `key_to_index` or `leaf_hash_to_index` to reflect the new physical position. When the moved node is a leaf, both maps retain stale `TreeIndex` values pointing to the now-freed (and reusable) old slot. Any subsequent insert that reclaims that slot causes all cache-driven operations on the moved leaf — proof generation, upsert, key-value lookup — to read from the wrong blob position, corrupting the Merkle tree and enabling forged proofs of inclusion.

---

### Finding Description

`BlockStatusCache` maintains three parallel data structures that must stay in sync:

```
free_indexes:       IndexSet<TreeIndex>          – which blob slots are available
key_to_index:       HashMap<KeyId, TreeIndex>    – leaf key  → blob slot
leaf_hash_to_index: HashMap<Hash, TreeIndex>     – leaf hash → blob slot
``` [1](#0-0) 

The invariant is: for every live leaf at blob slot `S`, both `key_to_index[leaf.key] == S` and `leaf_hash_to_index[leaf.hash] == S`.

**The move path in `delete`**

When a leaf is deleted and its parent is the root (no grandparent), the sibling is relocated to `TreeIndex(0)` to become the new root:

```rust
let destination = TreeIndex(0);
// ...
self.insert_entry_to_blob(destination, &sibling_block)?;   // blob updated ✓
self.block_status_cache
    .move_index(sibling_index, destination)?;               // cache updated ✗
``` [2](#0-1) 

`move_index` only marks the old slot as free:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    // ...
    self.free_indexes.insert(source);   // old slot freed
    Ok(())
    // key_to_index and leaf_hash_to_index are NEVER updated
}
``` [3](#0-2) 

When the sibling is a leaf (the tree had exactly two leaves), after the call:

- Blob at `TreeIndex(0)` → sibling leaf data ✓  
- `key_to_index[sibling.key]` → `sibling_index` (stale, now freed) ✗  
- `leaf_hash_to_index[sibling.hash]` → `sibling_index` (stale, now freed) ✗  

`add_leaf` and `remove_leaf` show the correct pattern — they always update all three structures — but `move_index` was never extended to do the same for the key/hash maps: [4](#0-3) 

**Corruption sequence**

1. Tree: root (internal, index 0), leaf A (index 1), leaf B (index 2).  
2. `delete(A)`: B is moved to index 0. `sibling_index = 2` is freed. `key_to_index[B.key] = 2` (stale).  
3. `insert(C, ...)`: slot 2 is reused for C. `key_to_index[C.key] = 2`.  
4. `get_leaf_by_key(B.key)` → reads slot 2 → returns C's data.  
5. `get_proof_of_inclusion(B.key)` → builds proof from C's slot → forged/invalid proof.  
6. `upsert(B.key, ...)` → removes C from cache, writes B's update to slot 2 → tree corrupted.

`check_integrity` would detect the mismatch (`cached_index != actual_index`), but it is not called automatically on every operation: [5](#0-4) 

---

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

- `get_proof_of_inclusion` generates a proof anchored to the wrong blob slot, producing a proof that does not correspond to the claimed key. A verifier checking the proof against the committed root will reject it, but the DataLayer node itself believes the proof is valid, creating a cross-layer disagreement.  
- `upsert` on the moved leaf reads and removes the wrong cache entry, then writes to the wrong slot, silently corrupting the Merkle tree's physical layout and invalidating the root hash.  
- `get_keys_values` and `get_node_by_hash` return wrong key-value pairs after slot reuse, corrupting any downstream accounting that relies on them. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

The trigger condition — deleting one of exactly two leaves — is a routine DataLayer operation. Any DataLayer store that starts with two entries and has one deleted hits this path. The subsequent insert that reuses the freed slot is equally routine. No special privileges, keys, or network access are required; a normal DataLayer client submitting standard transactions is sufficient.

---

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf. The caller already has the sibling block in hand, so the node type is known. The fix should be applied either inside `move_index` (by passing the node) or at the call site in `delete`:

```rust
// After insert_entry_to_blob(destination, &sibling_block):
if let Node::Leaf(ref leaf) = sibling_block.node {
    // fix up both maps
    if let Some(entry) = self.block_status_cache.key_to_index.get_mut(&leaf.key) {
        *entry = destination;
    }
    if let Some(entry) = self.block_status_cache.leaf_hash_to_index.get_mut(&leaf.hash) {
        *entry = destination;
    }
}
self.block_status_cache.move_index(sibling_index, destination)?;
```

Alternatively, replace `move_index` with a `move_leaf` variant that takes the `LeafNode` and updates all three maps atomically, mirroring the existing `add_leaf` / `remove_leaf` pattern. [8](#0-7)

### Proof of Concept

```rust
#[test]
fn test_move_index_stale_key_to_index() {
    // Build a two-leaf tree: root(internal,0), leaf_a(1), leaf_b(2)
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    blob.check_integrity_on_drop = false;

    let key_a = KeyId(1);
    let key_b = KeyId(2);
    let key_c = KeyId(3);
    let hash_a = sha256_num(&1u64);
    let hash_b = sha256_num(&2u64);
    let hash_c = sha256_num(&3u64);

    blob.insert(key_a, ValueId(10), &hash_a, InsertLocation::Auto {}).unwrap();
    blob.insert(key_b, ValueId(20), &hash_b, InsertLocation::Auto {}).unwrap();

    // Record where B lives before deletion
    let b_index_before = *blob.block_status_cache.get_index_by_key(key_b).unwrap();

    // Delete A: B is moved to TreeIndex(0); b_index_before is freed
    blob.delete(key_a).unwrap();

    // key_to_index[B] still points to the old (now freed) slot
    let b_index_after = *blob.block_status_cache.get_index_by_key(key_b).unwrap();
    assert_eq!(b_index_before, b_index_after,   // stale: should be TreeIndex(0)
        "key_to_index[B] was not updated after move");

    // Insert C: reuses the freed slot (b_index_before)
    blob.insert(key_c, ValueId(30), &hash_c, InsertLocation::Auto {}).unwrap();

    // Now get_leaf_by_key(B) reads C's slot → returns C's data
    // check_integrity detects the corruption
    blob.check_integrity()
        .expect_err("integrity check must fail due to stale key_to_index");
}
```

The test demonstrates that after deleting leaf A from a two-leaf tree, `key_to_index[B.key]` retains the stale old index. `check_integrity` confirms the desynchronization between the blob and the cache. [3](#0-2) [2](#0-1)
