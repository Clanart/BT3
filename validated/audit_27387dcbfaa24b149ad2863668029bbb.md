### Title
`From<CommitmentStateDiff> for ThinStateDiff` Silently Drops `deprecated_declared_classes`, Producing Wrong `state_diff_length`, Wrong `state_diff_commitment`, and Wrong Block Hash for Any Block Containing a Cairo 0 Declaration — (`File: crates/blockifier/src/state/cached_state.rs`)

---

### Summary

`CommitmentStateDiff` has no `deprecated_declared_classes` field. The `From<CommitmentStateDiff> for ThinStateDiff` conversion hard-codes that field to `Vec::new()`. `BlockExecutionArtifacts::new` feeds this truncated `ThinStateDiff` directly into `calculate_block_commitments`, which uses `ThinStateDiff::len()` for the packed `state_diff_length` in `concat_counts` and hashes the diff for `state_diff_commitment`. Both values are therefore wrong for every block that contains a Cairo 0 (`DECLARE v0/v1`) transaction, producing a wrong `concatenated_counts`, a wrong `state_diff_commitment`, a wrong `PartialBlockHashComponents`, a wrong `PartialBlockHash` (the consensus proposal commitment), and ultimately a wrong final block hash.

---

### Finding Description

**Root cause — the silent drop:** [1](#0-0) 

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` only tracks `address_to_class_hash`, `address_to_nonce`, `storage_updates`, and `class_hash_to_compiled_class_hash`. [2](#0-1)  Cairo 0 declared classes carry no compiled class hash and therefore never appear in `class_hash_to_compiled_class_hash`; they belong exclusively in `deprecated_declared_classes`. The conversion silently discards them.

**Call site — block commitment computation:**

`BlockExecutionArtifacts::new` passes this truncated diff to `calculate_block_commitments`: [3](#0-2) 

```rust
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
).await;
```

Inside `calculate_block_commitments`, two values are derived from the truncated diff:

1. **`state_diff_length`** — used in `concat_counts` to build `concatenated_counts`: [4](#0-3) 

```rust
let concatenated_counts = concat_counts(
    transactions_data.len(),
    event_leaf_elements.len(),
    state_diff.len(),   // wrong: deprecated_declared_classes.len() == 0
    l1_da_mode,
);
```

`ThinStateDiff::len()` explicitly counts `deprecated_declared_classes`: [5](#0-4) 

2. **`state_diff_commitment`** — the Poseidon hash of the `ThinStateDiff` itself, computed over a diff that is missing the deprecated class hashes. [6](#0-5) 

Both wrong values are packed into `BlockHeaderCommitments` and stored in `PartialBlockHashComponents`: [7](#0-6) 

**Block hash derivation — wrong inputs propagate:**

`calculate_block_hash` chains `concatenated_counts` and `state_diff_commitment.0.0` directly into the Poseidon hash: [8](#0-7) 

Both fields are wrong, so the final block hash is wrong for any block with a Cairo 0 declaration.

**Proposal commitment — wrong value used in consensus:**

`BlockExecutionArtifacts::commitment()` derives the `PartialBlockHash` (the consensus proposal commitment) from the same corrupted `partial_block_hash_components`: [9](#0-8) 

**P2P sync — wrong `state_diff_length` in block header:**

The P2P sync client reads `state_diff_length` from the stored block header to decide how many state-diff chunks to collect: [10](#0-9) 

If the stored `state_diff_length` is too small (missing the deprecated class count), the sync loop terminates early, and the syncing node stores an incomplete state diff, diverging from the canonical state.

---

### Impact Explanation

For every block that contains at least one Cairo 0 `DECLARE` transaction:

- `state_diff_length` in `concat_counts` is under-counted by the number of deprecated declared classes.
- `state_diff_commitment` is computed over a diff that omits those class hashes.
- Both errors propagate into `concatenated_counts` and `state_diff_commitment` inside `BlockHeaderCommitments`.
- The resulting `PartialBlockHashComponents` and final block hash are wrong.
- The consensus proposal commitment (`PartialBlockHash`) is wrong, potentially causing validators to reject or accept a mismatched proposal.
- Nodes syncing via P2P may store an incomplete state diff, leading to a wrong authoritative state root and divergent execution for subsequent blocks.

This matches: **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"** (wrong block hash, wrong state diff commitment, wrong state root for syncing nodes) and **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** (wrong block hash returned by RPC for any block with a Cairo 0 declaration).

---

### Likelihood Explanation

Cairo 0 (`DECLARE v0/v1`) transactions are still valid in the Starknet protocol and accepted by the gateway. Any user submitting such a transaction triggers the bug. No privileged access is required. The bug is deterministic: every block containing at least one deprecated class declaration produces a wrong block hash.

---

### Recommendation

Add `deprecated_declared_classes` to `CommitmentStateDiff` and populate it during transaction execution (analogous to how `class_hash_to_compiled_class_hash` is populated for Cairo 1 classes). The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then propagate the field instead of hard-coding `Vec::new()`.

Alternatively, if `CommitmentStateDiff` is intentionally kept without this field, the block-commitment path must obtain `deprecated_declared_classes` from a separate source (e.g., the full `StateMaps` or the executed transaction receipts) and merge it into the `ThinStateDiff` before calling `calculate_block_commitments`.

---

### Proof of Concept

1. Submit a Cairo 0 `DECLARE` transaction to the sequencer.
2. The batcher executes it; `CommitmentStateDiff` is produced with no `deprecated_declared_classes`.
3. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())` → `deprecated_declared_classes = []`.
4. `calculate_block_commitments` computes `state_diff_length = N` (missing the 1 deprecated class) and `state_diff_commitment = H(diff_without_deprecated_class)`.
5. `concat_counts(tx_count, event_count, N, da_mode)` produces a wrong `concatenated_counts` felt.
6. `calculate_block_hash(partial_block_hash_components, state_root, parent_hash)` chains the wrong `concatenated_counts` and `state_diff_commitment`, producing `block_hash_wrong ≠ block_hash_correct`.
7. The wrong hash is stored as the block hash and used as `previous_block_hash` for block N+1, permanently diverging the chain's hash chain from the canonical Starknet protocol value.
8. Any node syncing via P2P reads `state_diff_length = N` from the header and stops collecting chunks one entry short, storing an incomplete state diff and computing a wrong state root.
