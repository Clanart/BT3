### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Replay of Meta Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` omits a nonce from the hash it produces. Because the OS also hardcodes `nonce=0` in the synthetic `TxInfo` and explicitly skips nonce enforcement for version-0 transactions, any two meta-transaction invocations with identical `(contract_address, selector, calldata, chain_id)` produce the **same transaction hash**. An unprivileged attacker who can submit an outer transaction can replay a previously observed meta transaction against the target account, potentially draining funds.

---

### Finding Description

`compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

Compare this to `compute_l1_handler_transaction_hash`, which correctly passes the nonce as `additional_data`: [2](#0-1) 

In `execute_meta_tx_v0`, the synthetic `TxInfo` is constructed with `nonce=0` hardcoded: [3](#0-2) 

`check_and_increment_nonce` is never called inside `execute_meta_tx_v0`. Furthermore, `check_and_increment_nonce` explicitly skips all version-0 transactions at the OS level: [4](#0-3) 

The result is that the OS provides **zero replay protection** for meta transactions. The hash presented to the target account's `__execute__` entry point is identical for any two calls with the same `(contract_address, selector, calldata, chain_id)`, and the nonce field visible to the account is always `0`.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A v0 account contract whose `__execute__` function verifies a signature over `tx_info.transaction_hash` (the standard pattern) will accept the same signature on every replay, because the hash never changes for the same payload. An attacker can:

1. Observe a legitimate meta transaction (e.g., a token transfer signed by a v0 account).
2. Deploy a contract that emits a `META_TX_V0` syscall with the identical `(contract_address, selector, calldata, chain_id, signature)`.
3. Submit an outer transaction calling that contract.
4. The OS computes the same `meta_tx_hash`, presents it to the target account's `__execute__`, which accepts the same signature and executes the transfer again.

Repeated replays drain the target account's token balance.

---

### Likelihood Explanation

**High.** The `META_TX_V0` syscall is reachable by any unprivileged transaction sender who can deploy a contract and submit an outer transaction. No privileged role or leaked key is required. The attacker only needs to observe a valid meta transaction on-chain (public information) and re-submit it with the same parameters.

---

### Recommendation

Include a nonce in `compute_meta_tx_v0_hash`, mirroring the pattern used by `compute_l1_handler_transaction_hash`:

```cairo
// Pass the nonce as additional_data, analogous to compute_l1_handler_transaction_hash.
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // <-- add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // <-- include nonce in hash
    );
    return tx_hash;
}
```

The nonce must be sourced from the target account's on-chain state (read from `contract_state_changes`) and incremented by the OS after each successful meta transaction, removing the reliance on the account contract to self-enforce replay protection.

---

### Proof of Concept

**Step 1.** Alice signs a meta transaction: `(contract=ALICE, selector=__execute__, calldata=[transfer BOB 100 tokens], chain_id=SN_MAIN)`. The OS computes:

```
meta_tx_hash = H(INVOKE_PREFIX, 0, ALICE, __execute__, H(calldata), 0, SN_MAIN)
               ^^^ no nonce ^^^
``` [5](#0-4) 

**Step 2.** Attacker deploys `MaliciousContract` with:
```cairo
syscall_ptr.meta_tx_v0(
    contract_address=ALICE,
    selector=__execute__,
    calldata=[transfer BOB 100 tokens],
    signature=ALICE_ORIGINAL_SIGNATURE,
)
```

**Step 3.** Attacker submits an outer v3 invoke transaction calling `MaliciousContract.__execute__`. The OS enters `execute_meta_tx_v0`: [6](#0-5) 

The same `meta_tx_hash` is produced. `nonce=0` is set in the synthetic `TxInfo`. [7](#0-6) 

**Step 4.** Alice's `__execute__` receives `transaction_hash = meta_tx_hash` (identical to the original) and `signature = ALICE_ORIGINAL_SIGNATURE`. Signature verification passes. 100 tokens are transferred to BOB again.

**Step 5.** Attacker repeats Step 3 until Alice's balance is exhausted.
