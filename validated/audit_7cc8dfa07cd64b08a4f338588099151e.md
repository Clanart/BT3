### Title
`BlockStatusCache::move_index` Fails to Update Leaf Key/Hash Mappings After Node Relocation, Corrupting DataLayer Merkle Tree State — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`BlockStatusCache::move_index` only marks the source slot as free in `free_indexes` but never updates `key_to_index` or `leaf_hash_to_index` when a leaf node is physically relocated to a new index in the blob. This leaves the cache in an inconsistent state after any `delete` that triggers the "sibling-becomes-root" path, causing subsequent operations (e.g., `upsert`) to read from and write to the wrong blob slot, silently corrupting the DataLayer Merkle tree root.

### Finding Description

`BlockStatusCache` maintains three parallel data structures that must stay in sync:

- `free_indexes` — set of unoccupied blob slots
- `key_to_index` — maps each `KeyId` to its blob slot
- `leaf_hash_to_index` — maps each leaf `Hash` to its blob slot [1](#0-0) 

`move_index` is called after a node is physically copied from `source` to `destination` in the blob. It correctly marks `source` as free, but it **never updates `key_to_index` or `leaf_hash_to_index`** to reflect the new position:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    // to be called _after_ having written to the destination index
    if self.free_indexes.contains(&source) {
        return Err(Error::MoveSourceIndexNotInUse(source));
    }
    if self.free_indexes.contains(&destination) {
        return Err(Error::MoveDestinationIndexNotInUse(destination));
    }
    self.free_indexes.insert(source);   // ← only this is updated
    Ok(())
}
``` [1](#0-0) 

`move_index` is invoked inside `delete` when the deleted leaf's sibling must be promoted to the root slot (`TreeIndex(0)`):

```rust
let destination = TreeIndex(0);
// ...
self.insert_entry_to_blob(destination, &sibling_block)?;   // blob[0] = sibling data
self.block_status_cache
    .move_index(sibling_index, destination)?;              // only frees sibling_index
``` [2](#0-1) 

When the sibling is a **leaf** (i.e., the tree had exactly 2 leaves before the delete), after this call:

| State | Expected | Actual |
|---|---|---|
| `blob[0]` | sibling leaf data | sibling leaf data ✓ |
| `blob[sibling_index]` | freed/stale | stale sibling data |
| `key_to_index[sibling_key]` | `TreeIndex(0)` | `sibling_index` ✗ |
| `leaf_hash_to_index[sibling_hash]` | `TreeIndex(0)` | `sibling_index` ✗ |
| `free_indexes` | `{sibling_index}` | `{sibling_index}` ✓ |

The cache now points to the wrong (freed) slot for the surviving leaf.

**Exploitation via `upsert`:** Any subsequent `upsert` on the surviving leaf calls `get_leaf_by_key`, which resolves the key through the cache to `sibling_index` (wrong). It then:
1. Calls `remove_leaf`, which removes the key from the cache and adds `sibling_index` back to `free_indexes`.
2. Writes the updated leaf to `blob[sibling_index]` (wrong position).
3. Re-adds the leaf to the cache at `sibling_index`.

After this sequence:
- `blob[0]` (the actual root) still holds the **old, stale** leaf data.
- `blob[sibling_index]` holds the updated data at a non-root position.
- The Merkle root hash is computed from the stale data at index 0. [3](#0-2) 

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

After the corruption:
- The stored Merkle root reflects the **pre-update** leaf value.
- Inclusion proofs for the **new** value are rejected (the root doesn't commit to it).
- Inclusion proofs for the **old** value are accepted (the root still commits to it), even though the value was updated — a forged inclusion proof for stale state.
- Any downstream system relying on the DataLayer root for state verification (e.g., DataLayer sync, offer verification) operates on incorrect committed state.

### Likelihood Explanation

The trigger sequence is straightforward and requires no privileges:
1. Insert exactly 2 key-value pairs into a DataLayer store.
2. Delete one of them.
3. Call `upsert` on the remaining key.

All three operations are exposed via the public Python API (`MerkleBlob.insert`, `MerkleBlob.delete`, `MerkleBlob.upsert`). Any DataLayer client performing a normal update cycle on a small store (or one that has been pruned to 2 entries) will hit this path. [4](#0-3) 

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf. The function should read the node type from the blob (already written to `destination` before `move_index` is called) and, if it is a leaf, remap its key and hash entries from `source` to `destination`:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    // existing checks ...
    self.free_indexes.insert(source);

    // NEW: remap any leaf key/hash that pointed to source → destination
    for val in self.key_to_index.values_mut() {
        if *val == source { *val = destination; }
    }
    for val in self.leaf_hash_to_index.values_mut() {
        if *val == source { *val = destination; }
    }
    Ok(())
}
```

Alternatively, pass the moved node's type/key/hash explicitly so the remapping is O(1).

### Proof of Concept

```
State: empty MerkleBlob

1. insert(KeyA, ValA, HashA)   → blob[0] = LeafA
2. insert(KeyB, ValB, HashB)   → blob[0] = Internal, blob[1] = LeafA, blob[2] = LeafB
   cache: key_to_index = {A→1, B→2}

3. delete(KeyA)
   → sibling = blob[2] (LeafB), sibling_index = 2, destination = 0
   → blob[0] = LeafB  (correct)
   → move_index(2, 0): free_indexes = {2}
   → BUG: key_to_index = {B→2}  (should be {B→0})

4. upsert(KeyB, ValB_new, HashB_new)
   → get_leaf_by_key(B) → index=2 (wrong, from cache)
   → remove_leaf: key_to_index = {}, free_indexes = {2}
   → insert_entry_to_blob(2, updated_LeafB)  ← writes to blob[2], NOT blob[0]
   → blob[0] still = old LeafB data  ← root is stale

5. get_root_hash() → hash of old LeafB  ← wrong
   proof_of_inclusion(KeyB, ValB_new) → REJECTED  ← forged exclusion
   proof_of_inclusion(KeyB, ValB)     → ACCEPTED  ← forged inclusion of stale state
