### Title
Unhandled `ft_transfer` Failure in `fin_transfer_send_tokens_callback` Permanently Locks User Funds - (File: near/omni-bridge/src/lib.rs)

### Summary
When `process_fin_transfer_to_near` dispatches a plain `ft_transfer` (non-deployed token, empty `msg`), the callback `fin_transfer_send_tokens_callback` never inspects the promise result. If `ft_transfer` reverts, the transfer is permanently marked as finalized, locked-token accounting is not restored, and the recipient never receives their funds — with no recovery path.

### Finding Description
`process_fin_transfer_to_near` calls `send_tokens` and chains `fin_transfer_send_tokens_callback` as the continuation. The `is_ft_transfer_call` flag passed to the callback is `!msg.is_empty()`. [1](#0-0) 

When `msg` is empty and the token is not a deployed bridge token, `send_tokens` issues a plain `ft_transfer`: [2](#0-1) 

The callback receives `is_ft_transfer_call = false`. `is_refund_required` then unconditionally returns `false` for this case, **without reading the promise result at all**: [3](#0-2) 

Consequently, `fin_transfer_send_tokens_callback` always takes the success branch regardless of whether `ft_transfer` actually succeeded: [4](#0-3) 

The cleanup branch — which calls `revert_lock_actions`, `remove_fin_transfer`, and emits `FailedFinTransferEvent` — is never reached for a failed plain `ft_transfer`: [5](#0-4) 

### Impact Explanation
When `ft_transfer` fails:

1. **`finalised_transfers` retains the entry** — the transfer ID is permanently marked as finalized, so no replay or retry is possible.
2. **`locked_tokens` is not restored** — `unlock_tokens_if_needed` already decremented the counter in `process_fin_transfer_to_near`; `revert_lock_actions` is never called.
3. **The recipient never receives tokens** — the `ft_transfer` reverted, so no tokens moved.
4. **Fee dispatch also fails silently** — the detached fee `ft_transfer` will also revert, but is ignored.

The user's funds are permanently frozen: burned/locked on the origin chain, accounting decremented on NEAR, but never delivered. This matches the **Critical/High** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows*.

### Likelihood Explanation
The `ft_transfer` path is taken for every finalization of a non-deployed (natively locked) token with an empty `msg`. Failure can occur when:

- The token contract is paused or upgraded between the storage-balance check and the actual transfer.
- A race condition: the recipient calls `storage_unregister` (with zero balance) between the storage-deposit promise and the `ft_transfer` execution.
- The token contract has non-standard behavior (e.g., a blocklist, transfer cap, or reentrancy guard) that causes the transfer to revert.

The bridge explicitly supports arbitrary NEP-141 tokens, making non-standard behavior a realistic concern. The race condition is exploitable by the recipient themselves (self-inflicted) or by a griefing party if the token allows third-party storage removal.

### Recommendation
`is_refund_required` must also inspect the promise result for non-`ft_transfer_call` cases. A failed promise (i.e., `env::promise_result_checked` returns `Err`) should trigger the same cleanup path as a failed `ft_transfer_call`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        // existing logic
    } else {
        // NEW: check if the plain ft_transfer/mint promise itself failed
        env::promise_result_checked(0, 0).is_err()
    }
}
```

Alternatively, add an explicit promise-result check at the top of `fin_transfer_send_tokens_callback` before branching on `is_ft_transfer_call`.

### Proof of Concept

1. User initiates a transfer of a non-deployed token (e.g., USDC locked in the bridge) from ETH → NEAR with an empty `msg`.
2. Relayer submits a valid proof via `fin_transfer`.
3. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` inserts the transfer ID into `finalised_transfers`.
   - `unlock_tokens_if_needed` decrements `locked_tokens[Eth][USDC]` by `amount`.
   - `send_tokens` dispatches `ft_transfer(recipient, amount, None)`.
4. The USDC contract is paused (or the recipient's storage was just removed); `ft_transfer` reverts.
5. `fin_transfer_send_tokens_callback` runs with `is_ft_transfer_call = false`.
6. `is_refund_required(false)` returns `false` — no promise result is read.
7. The else-branch fires: fee dispatch (also fails silently), `FinTransferEvent` logged.
8. **Final state**: `finalised_transfers` contains the ID (no replay possible), `locked_tokens` is under-counted by `amount`, recipient balance unchanged. Funds are permanently lost. [6](#0-5) [7](#0-6) [8](#0-7)
