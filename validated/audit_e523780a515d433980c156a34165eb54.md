### Title
Missing Token Registration Check in `initTransfer` Allows Locking Unregistered ERC20 Tokens With No User-Controlled Recovery — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer` on the EVM bridge accepts any ERC20 `tokenAddress` and locks it in the contract without verifying that the token is registered in the bridge's cross-chain token mapping. When NEAR's `fin_transfer_callback` subsequently panics because the token has no entry in `token_decimals`, no pending transfer is created on NEAR, and the MPC will never sign a release. The user has no self-service recovery path; recovery requires admin intervention (register the token on NEAR) followed by a relayer retrying the proof submission.

---

### Finding Description

**EVM side — no registration guard:**

In `initTransfer`, the token-handling branch falls through to a plain `safeTransferFrom` for any address that is neither a `customMinter` nor an `isBridgeToken`:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

There is no check against `ethToNearToken[tokenAddress]` or any other registration mapping before locking. Any ERC20 that grants the bridge an allowance will be accepted and locked.

**NEAR side — hard panic on unregistered token:**

When a relayer submits the EVM proof to NEAR's `fin_transfer`, the callback immediately looks up the token in `token_decimals`:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [2](#0-1) 

`near_expect` panics with `ERR_TOKEN_DECIMALS_NOT_FOUND`. Because the panic occurs before `finalised_transfers` is updated, the proof is not consumed — but no pending transfer is created in NEAR state either.

**No EVM-side recovery path:**

EVM `finTransfer` (the only function that can release locked ERC20s) requires a valid MPC signature over a `TransferMessagePayload`:

```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

The MPC only signs payloads derived from NEAR state (via `sign_transfer`). Since NEAR never stored a pending transfer for this lock event, the MPC will never produce a valid signature. The CLAUDE.md security invariant explicitly forbids any admin bypass of this gate:

> "No admin function, emergency path, or refactor may bypass this — it is the only authorization gate for finTransfer" [4](#0-3) 

There is no `emergencyWithdraw` or equivalent in `OmniBridge.sol`.

---

### Impact Explanation

From the user's perspective, the locked ERC20 tokens are irrecoverable without external intervention:

- The user cannot call any EVM function to retrieve them (no refund path, no admin escape hatch).
- The MPC will not sign a release because NEAR has no record of the transfer.
- Recovery requires: (1) the DAO/admin registers the token on NEAR, and (2) a trusted relayer resubmits the proof. Neither is guaranteed, and neither is user-controlled.

If the admin declines to register the token (e.g., it is an unsupported or malicious token), the funds are permanently locked. This matches the scoped impact: **permanent freezing / irrecoverable lock of user funds in the bridge**.

---

### Likelihood Explanation

- Any user who holds an ERC20 and grants the bridge an allowance can trigger this.
- No privileged role is required.
- The scenario is reachable by accident (user bridges a token before it is officially registered) or by deliberate action.
- The `logMetadata` flow exists to register tokens, but it is a separate, optional step that is not enforced before `initTransfer`.

---

### Recommendation

Add a registration guard at the top of the `else` branch in `initTransfer` (or at the function entry for all non-native, non-bridge-token paths):

```solidity
require(
    bytes(ethToNearToken[tokenAddress]).length > 0,
    "ERR_TOKEN_NOT_REGISTERED"
);
```

This ensures that only tokens with a confirmed NEAR-side mapping can be locked, matching the invariant that every locked ERC20 must be claimable on the destination chain.

---

### Proof of Concept

1. Deploy or use any ERC20 not registered in `ethToNearToken`, `isBridgeToken`, or `customMinters`.
2. Approve the bridge for `amount` tokens.
3. Call `initTransfer(tokenAddress, amount, 0, 0, "recipient.near", "")`.
4. Observe: `safeTransferFrom` succeeds, tokens are held by the bridge, `InitTransfer` is emitted. [5](#0-4) 
5. Relayer submits the EVM proof to NEAR `fin_transfer`.
6. NEAR `fin_transfer_callback` panics at `near_expect(BridgeError::TokenDecimalsNotFound)` — no pending transfer is stored. [2](#0-1) 
7. Attempt to call EVM `finTransfer` with any payload for this token — reverts with `InvalidSignature` because no MPC signature exists. [3](#0-2) 
8. Assert: tokens remain in the bridge with no user-accessible release path.
