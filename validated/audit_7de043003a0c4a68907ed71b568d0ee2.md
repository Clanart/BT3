### Title
Unconstrained `is_reverted` Hint Enables Sequencer to Charge Fees for Skipped Transaction Execution — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is an empty no-op. The `is_reverted` flag controlling whether a transaction's execute entry point runs is loaded from an unconstrained sequencer-controlled hint. Because no Cairo constraint enforces the correctness of this flag, a sequencer can mark any transaction as reverted — skipping its execution entirely — while still charging the user the full fee. The resulting proof remains cryptographically valid.

---

### Finding Description

In `execute_invoke_function_transaction`, the `is_reverted` flag is loaded from a hint and then passed to `check_is_reverted` before the execute entry point is conditionally called: [1](#0-0) 

The hint load and the conditional branch are:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // stack alignment only — no execution
}
// Charge fee — always runs regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
```

The function `check_is_reverted` that is supposed to validate this flag is: [2](#0-1) 

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

It performs **zero validation**. The `is_reverted` value is a free variable constrained by nothing in the Cairo trace. The sequencer can set it to any felt value.

Fee charging is unconditional: [3](#0-2) 

And `charge_fee` only enforces an upper bound — `actual_fee ≤ max_fee` — with no lower bound and no relationship to whether execution occurred: [4](#0-3) 

The same unconstrained `is_reverted` pattern applies to `execute_deploy_account_transaction` and `execute_l1_handler_transaction` via the same hint mechanism. [5](#0-4) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

A sequencer that sets `is_reverted = TRUE` for a transaction that would have succeeded causes:

1. The `__execute__` entry point to be skipped — the transaction has no on-chain effect.
2. The fee transfer to the sequencer's address to still execute unconditionally.
3. The resulting STARK proof to be fully valid, because `check_is_reverted` imposes no constraint.

The user pays the fee and receives nothing. The sequencer collects the fee and can simultaneously prevent any transaction from executing — a selective, fee-extracting censorship mechanism that is indistinguishable from a legitimate revert in the proof.

The `sequencer_address` is the fee recipient hardcoded in `charge_fee`: [6](#0-5) 

---

### Likelihood Explanation

The sequencer controls all hints provided to the Cairo VM during proof generation. No additional privilege, key leak, or external dependency is required beyond the sequencer's normal role. The OS is the mechanism that is supposed to cryptographically constrain the sequencer's behavior — the empty `check_is_reverted` is a direct failure of that constraint. Any block the sequencer produces can exploit this for any subset of transactions.

---

### Recommendation

`check_is_reverted` must be implemented to derive the revert status from the actual execution trace rather than accepting it as an unconstrained hint. The revert flag should be a deterministic output of the entry point execution (e.g., derived from `entry_point_return_values.failure_flag` as done inside `execute_entry_point`), not a free variable loaded before execution occurs. [7](#0-6) 

The pre-execution `is_reverted` hint pattern should be eliminated. If pre-classification of reverted transactions is needed for performance, the classification must be proven consistent with the actual execution result.

---

### Proof of Concept

1. User submits an invoke transaction (v3) with non-zero resource bounds, so `max_fee > 0`.
2. Sequencer includes the transaction in a block.
3. During proof generation, sequencer sets `%{ IsReverted %}` to return `is_reverted = 1` (TRUE) for this transaction, regardless of whether the transaction would actually revert.
4. The OS Cairo code skips `non_reverting_select_execute_entry_point_func` — no state changes from `__execute__` are applied.
5. `charge_fee` runs unconditionally, transferring `actual_fee` (up to `max_fee`) from the user's account to the sequencer's address.
6. `check_is_reverted(1)` returns immediately with no assertion — the proof is valid.
7. The block is finalized. The user's balance is reduced by the fee; their intended transaction had zero effect.
