### Title
`ProofOfInclusion::valid()` Never Verifies Against an External Root — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate contains a tautological final check that makes the function verify only internal self-consistency of the proof structure, never correctness against an externally-known tree root. An attacker can construct an arbitrary, internally-consistent `ProofOfInclusion` from scratch — with any `node_hash` and any chain of `layers` — and `valid()` will return `true`. This allows forged DataLayer inclusion proofs to pass validation.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()
}
``` [1](#0-0) 

The final check `existing_hash == self.root_hash()` is **tautological**. After the loop, `existing_hash` holds the last `calculated_hash`, which was already asserted to equal `layer.combined_hash` for the last layer. `root_hash()` returns exactly `last.combined_hash`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- same value as existing_hash after the loop
    } else {
        self.node_hash       // <-- same value as existing_hash when layers is empty
    }
}
``` [2](#0-1) 

In both the non-empty and empty layers cases, `existing_hash == self.root_hash()` is always `true` when the loop completes without returning `false`. The function therefore **never compares the computed root against any externally-supplied expected root**. It only verifies that the proof's own fields are mutually consistent.

`ProofOfInclusion` derives `Streamable`, meaning it can be deserialized from arbitrary bytes:

```rust
#[derive(Clone, Debug, std::hash::Hash, Eq, PartialEq, Streamable)]
pub struct ProofOfInclusion {
    pub node_hash: Hash,
    pub layers: Vec<ProofOfInclusionLayer>,
}
``` [3](#0-2) 

An attacker can craft a `ProofOfInclusion` where:
- `node_hash` = SHA256 of any arbitrary data (e.g., a fake key-value pair hash)
- `layers` = a chain of `ProofOfInclusionLayer` entries where each `combined_hash` is computed from the previous hash and an attacker-chosen `other_hash` and `other_hash_side`

Such a proof is internally consistent and `valid()` returns `true`, even though it corresponds to no real DataLayer tree state.

The `get_proof_of_inclusion` function on `MerkleBlob` correctly builds proofs from the actual tree, using `parent.hash` as `combined_hash` at each layer: [4](#0-3) 

But the `valid()` method — the only verification API — does not accept an expected root parameter and cannot enforce that the proof corresponds to any specific committed tree root.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A DataLayer client that receives a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` to verify it will accept any internally-consistent forged proof. The attacker can claim that any arbitrary key-value pair is present in the DataLayer tree, even if it is not. Since DataLayer state is used to back off-chain data commitments anchored to Chia coins, accepting forged inclusion proofs allows an attacker to prove invalid state to any party relying on `valid()` as the verification gate.

### Likelihood Explanation

The `ProofOfInclusion` type is `Streamable` and is exposed via Python bindings. Any DataLayer client that receives proofs over the network and calls `proof.valid()` without separately asserting `proof.root_hash() == known_committed_root` is vulnerable. The API design actively invites this misuse: `valid()` sounds like a complete verification, but it is not. The tautological final check provides false assurance that a root comparison is occurring.

### Recommendation

1. **Add an expected root parameter to `valid()`**:
   ```rust
   pub fn valid(&self, expected_root: &Hash) -> bool {
       // ... existing hash chain check ...
       existing_hash == *expected_root
   }
   ```
   This forces every caller to supply the known-good root and eliminates the tautological check.

2. Alternatively, rename the current function to `is_internally_consistent()` and add a separate `valid(expected_root: &Hash) -> bool` that calls `is_internally_consistent() && self.root_hash() == *expected_root`.

3. Audit all call sites of `proof.valid()` in the Python DataLayer implementation to ensure they also check `proof.root_hash()` against the on-chain committed root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, Side, ValueId, InsertLocation};
use chia_datalayer::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};
use chia_sha2::Sha256;

fn forge_proof() {
    // Attacker picks an arbitrary fake leaf hash
    let fake_node_hash = Hash(/* any 32 bytes */);
    let fake_other_hash = Hash(/* any 32 bytes */);

    // Compute a combined_hash that is internally consistent
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash,
        Side::Right,
        &fake_other_hash,
    );

    let forged = ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Right,
            other_hash: fake_other_hash,
            combined_hash: combined,  // attacker-controlled root
        }],
    };

    // valid() returns true even though this proof was never generated
    // from any real MerkleBlob
    assert!(forged.valid());
}
```

The `valid()` call succeeds because `existing_hash` after the loop equals `combined`, and `root_hash()` also returns `combined` — the tautological check passes unconditionally.
