### Title
Missing Replay Guard in `utxo_fin_transfer_fast` for Near-Destination Fast Transfers Enables Double Settlement — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a UTXO fast transfer targets a Near recipient, `utxo_fin_transfer_fast` removes the fast-transfer entry from `fast_transfers` but **never inserts the UTXO ID into `finalised_utxo_transfers`**. After the first finalization, the UTXO ID is absent from both replay-guard stores. A second submission of the same `UtxoFinTransferMsg` passes both guards and falls through to `add_fin_utxo_transfer`, which succeeds and issues a duplicate token release.

---

### Finding Description

`utxo_fin_transfer` (lib.rs:2456) has two branches:

**Branch A — fast path** (line 2483–2486): if a matching `fast_transfer` entry exists, execution returns immediately via `utxo_fin_transfer_fast`. `add_fin_utxo_transfer` is **never called**. [1](#0-0) 

**Branch B — normal path** (line 2488–2489): only reached when no fast-transfer entry exists; `add_fin_utxo_transfer` inserts the UTXO ID into `finalised_utxo_transfers`. [2](#0-1) 

Inside `utxo_fin_transfer_fast`, when the destination is `ChainKind::Near`, the fast-transfer record is **deleted** (`remove_fast_transfer`), not marked finalised: [3](#0-2) 

For non-Near destinations the entry is only marked finalised (line 2533), so a second call would hit `FastTransferAlreadyFinalised`. But for Near, the entry is gone entirely. [4](#0-3) 

After the first fast-path finalization to Near:
- `fast_transfers` no longer contains the UTXO ID (entry removed).
- `finalised_utxo_transfers` never contained it (never inserted).

A second call with the identical `UtxoFinTransferMsg`:
1. `get_fast_transfer_status` returns `None` → fast-path branch skipped.
2. `add_fin_utxo_transfer` is called and **succeeds** (UTXO ID absent from the set).
3. Tokens are sent again to the recipient — duplicate settlement. [5](#0-4) 

The `remove_fin_utxo_transfer` helper (used in callbacks on failure) also removes from `finalised_utxo_transfers`, confirming that set is the intended authoritative guard — but it is never populated by the fast path. [6](#0-5) 

---

### Impact Explanation

A duplicate settlement mints/releases tokens against the same Bitcoin UTXO twice, creating unbacked supply. This maps directly to the **High** impact category: *cross-chain replay / double-finalization / duplicate settlement enabling double-spend or unbacked supply*.

---

### Likelihood Explanation

The entry point is `ft_on_transfer` → `utxo_fin_transfer`, gated by `sender_id == config.connector` (line 2471–2474). [7](#0-6) 

The connector is a DAO-configured contract. The connector's `verify_deposit` is called by a relayer (shown in tests at line 166–175 of `utxo_fin_transfer.rs`). [8](#0-7) 

If the connector contract does not maintain its own per-UTXO replay guard (the bridge is supposed to be the authoritative guard), any relayer who can call `verify_deposit` twice with the same UTXO proof triggers the double-settlement. The bridge contract itself provides **zero protection** for this case — it fully delegates replay prevention to the connector for the fast-path Near branch, which is an unintended and undocumented assumption.

---

### Recommendation

In `utxo_fin_transfer_fast`, unconditionally insert the UTXO ID into `finalised_utxo_transfers` **before** removing the fast-transfer entry, regardless of destination chain:

```rust
fn utxo_fin_transfer_fast(
    &mut self,
    fast_transfer: FastTransfer,
    fast_transfer_status: FastTransferStatus,
    utxo_fin_transfer_msg: UtxoFinTransferMsg,
) -> PromiseOrPromiseIndexOrValue<U128> {
    require!(
        !fast_transfer_status.finalised,
        BridgeError::FastTransferAlreadyFinalised.as_ref()
    );

    // Always record in the authoritative replay guard first
    let origin_chain = fast_transfer.get_origin_chain();
    self.add_fin_utxo_transfer(&utxo_fin_transfer_msg.get_transfer_id(origin_chain));

    let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
        self.remove_fast_transfer(&fast_transfer.id());
        fast_transfer.amount
    } else {
        ...
    };
    ...
}
```

This ensures `finalised_utxo_transfers.contains(utxo_id)` is true after **every** successful `utxo_fin_transfer` call, regardless of which branch was taken.

---

### Proof of Concept

1. Relayer performs a fast transfer for UTXO `X` to a Near recipient: `fast_transfers[X] = {relayer, finalised: false}`.
2. Connector calls `ft_transfer_call` → `ft_on_transfer` → `utxo_fin_transfer` with UTXO `X`.
3. Fast-path branch taken: `remove_fast_transfer(X)` called. `finalised_utxo_transfers` unchanged. Tokens sent to relayer. ✓
4. Connector calls `ft_transfer_call` → `ft_on_transfer` → `utxo_fin_transfer` again with the same UTXO `X`.
5. `get_fast_transfer_status(X)` → `None` (entry was deleted in step 3).
6. `add_fin_utxo_transfer(X)` → succeeds (X not in `finalised_utxo_transfers`).
7. `utxo_fin_transfer_to_near` called → tokens sent to recipient a second time.

Result: two token releases for one Bitcoin UTXO deposit. [9](#0-8)

    fn utxo_fin_transfer_fast(
        &mut self,
        fast_transfer: FastTransfer,
        fast_transfer_status: FastTransferStatus,
        utxo_fin_transfer_msg: UtxoFinTransferMsg,
    ) -> PromiseOrPromiseIndexOrValue<U128> {
        require!(
            !fast_transfer_status.finalised,
            BridgeError::FastTransferAlreadyFinalised.as_ref()
        );

        let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
            self.remove_fast_transfer(&fast_transfer.id());
            fast_transfer.amount
        } else {
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
            // With transfers to other chain the fee will be claimed after finalization on the destination chain

                fast_transfer
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            )
        };

        self.send_tokens(
            fast_transfer.token_id.clone(),
            fast_transfer_status.relayer,
            amount,
            "",
        )
        .detach();

        env::log_str(
            &OmniBridgeEvent::UtxoTransferEvent {
                token_id: fast_transfer.token_id,
                amount,
                utxo_transfer_message: utxo_fin_transfer_msg,
                new_transfer_id: None,
            }
            .to_log_string(),
        );

        PromiseOrPromiseIndexOrValue::Value(U128(0))
