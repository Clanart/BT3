### Title
Stale `key_to_index` / `leaf_hash_to_index` Cache After Leaf-Sibling Promotion in `MerkleBlob::delete()` — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`BlockStatusCache::move_index()` only updates `free_indexes` when a node is physically relocated from `source` to `destination`, but never updates `key_to_index` or `leaf_hash_to_index`. This is called inside `MerkleBlob::delete()` when the sibling of the deleted leaf is a **leaf node** that must be promoted to the root slot (`TreeIndex(0)`). After the operation the two lookup maps remain stale, pointing to the old (now free) slot. Subsequent proof-of-inclusion generation panics, and after a new insert reuses the freed slot, `get_proof_of_inclusion` silently returns a proof built from the wrong leaf — a forged DataLayer inclusion proof.

---

### Finding Description

`BlockStatusCache` maintains three parallel structures: [1](#0-0) 

```
free_indexes:        IndexSet<TreeIndex>
key_to_index:        HashMap<KeyId,  TreeIndex>
leaf_hash_to_index:  HashMap<Hash,   TreeIndex>
```

When a leaf is deleted and its sibling has no grandparent (i.e., the tree had exactly two leaves), `delete()` physically copies the sibling block to `TreeIndex(0)` and then calls `move_index(sibling_index, destination)`: [2](#0-1) 

`move_index` only marks `source` as free: [3](#0-2) 

It never touches `key_to_index` or `leaf_hash_to_index`. After the call:

- `key_to_index[sibling.key]` → still `sibling_index` (old, now free slot)
- `leaf_hash_to_index[sibling.hash]` → still `sibling_index` (old, now free slot)
- Actual live data → `TreeIndex(0)`

Compare with `add_leaf`, which correctly updates all three structures: [4](#0-3) 

---

### Impact Explanation

**Immediate crash (DoS):** `get_proof_of_inclusion(sibling.key)` reads `key_to_index[sibling.key]` → stale `sibling_index`. It then traverses the lineage from that stale slot. The stale block still carries `parent = Some(TreeIndex(0))`. `TreeIndex(0)` now holds a **leaf** (the promoted sibling), not an internal node. The call to `expect_internal(...)` inside `get_proof_of_inclusion` panics deterministically: [5](#0-4) 

**Forged proof (after next insert):** `sibling_index` is now in `free_indexes`. The next `insert` call reuses it for a new leaf C. At that point `key_to_index[sibling.key]` points to the slot now occupied by leaf C. `get_proof_of_inclusion(sibling.key)` builds a proof using leaf C's hash and lineage — a structurally valid proof for the wrong data, silently accepted by `ProofOfInclusion::valid()`. [6](#0-5) 

`get_node_by_hash` is equally affected via the stale `leaf_hash_to_index`: [7](#0-6) 

Both paths are exposed through Python bindings: [8](#0-7) 

---

### Likelihood Explanation

The trigger is a minimal, common DataLayer sequence: insert two key-value pairs, delete one, then either query a proof or insert a third entry. No privilege is required. Any DataLayer store participant who can issue `insert`/`delete`/`get_proof_of_inclusion` calls reaches this path. The two-leaf tree is the smallest non-trivial tree and is a natural starting state for any new DataLayer store.

---

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf. The caller already has the sibling block in hand at the point of the call. The fix is to add a variant of `move_index` (or extend the existing one) that, when the moved node is a leaf, re-maps `key_to_index[leaf.key]` and `leaf_hash_to_index[leaf.hash]` from `source` to `destination`:

```rust
fn move_leaf_index(&mut self, source: TreeIndex, destination: TreeIndex, leaf: &LeafNode) -> Result<(), Error> {
    self.move_index(source, destination)?;
    // update the two stale maps
    self.key_to_index.insert(leaf.key, destination);
    self.leaf_hash_to_index.insert(leaf.hash, destination);
    Ok(())
}
```

Call this instead of `move_index` in the leaf-sibling branch of `delete()`.

---

### Proof of Concept

```
1. blob = MerkleBlob::new(vec![])
2. blob.insert(KeyId(1), ValueId(1), &hash_A, Auto{})   // leaf A at index 0
3. blob.insert(KeyId(2), ValueId(2), &hash_B, Auto{})
   // insert_second rebuilds: internal@0, leaf_A@1, leaf_B@2
4. blob.delete(KeyId(1))
   // sibling leaf_B (index 2) is copied to index 0
   // move_index(2, 0) marks index 2 as free
   // key_to_index[KeyId(2)] still = TreeIndex(2)  ← BUG
5. blob.get_proof_of_inclusion(KeyId(2))
   // reads stale index 2, follows parent→index 0 (now a leaf), panics in expect_internal
   // OR:
6. blob.insert(KeyId(3), ValueId(3), &hash_C, Auto{})
   // reuses free index 2 for leaf C
7. blob.get_proof_of_inclusion(KeyId(2))
   // key_to_index[KeyId(2)] = TreeIndex(2) = leaf C
   // returns proof for leaf C's hash under key 2 → forged proof
```
