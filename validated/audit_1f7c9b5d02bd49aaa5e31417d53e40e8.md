The key finding is in `crates/blockifier/src/state/state_reader_and_contract_manager.rs` — an asymmetric `is_declared` guard that protects Cairo1 classes from stale cache entries after block reverts, but deliberately skips the same check for Cairo0 (`V0`) classes.

### Title
Stale Cairo0 Class Cache Entry Bypasses Revocation Check, Enabling Execution of Undeclared Class — (`File: crates/blockifier/src/state/state_reader_and_contract_manager.rs`)

### Summary

`StateReaderAndContractManager::get_compiled_from_class_manager` applies an `is_declared` guard to Cairo1 cached classes to prevent execution of classes whose declaration was reverted, but unconditionally skips that guard for Cairo0 (`RunnableCompiledClass::V0`) classes. Because the global `ContractClassManager` cache is never rolled back on block revert, a Cairo0 class that was cached during a proposal that was later aborted or reverted remains in the cache indefinitely. A subsequent block can then execute that class without it being declared in storage, producing a wrong execution result, wrong state diff, and wrong state root.

### Finding Description

In `get_compiled_from_class_manager`:

```rust
// crates/blockifier/src/state/state_reader_and_contract_manager.rs  lines 73-83
match &runnable_class {
    RunnableCompiledClass::V0(_) => {}          // ← no check at all
    _ => {
        // The Cairo1 class is cached; verify it is declared,
        // since existence in the cache does not guarantee that
        // (it might contain a declared class from a reverted block, for example).
        if !self.state_reader.is_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
}
```

The comment explicitly acknowledges the revert-cache race for Cairo1 classes and adds a storage check. The identical risk exists for Cairo0 classes, but the `V0` arm is an empty no-op.

The `FetchCompiledClasses::is_declared` trait is documented as returning `false` for Cairo0 classes by design:

```rust
// crates/blockifier/src/state/state_reader_and_contract_manager.rs  lines 19-21
/// Returns whether the given class hash corresponds to a declared Cairo 1 class.
/// Cairo 0 classes always return `false`.
fn is_declared(&self, class_hash: ClassHash) -> StateResult<bool>;
```

This means there is no existing mechanism to verify that a cached V0 class is still declared in storage. The storage layer does correctly remove Cairo0 classes on revert (`delete_deprecated_declared_classes_block` in `crates/apollo_storage/src/state/mod.rs` lines 873-905), but the in-memory `GlobalContractCache` (an LRU cache shared across proposals) is never invalidated.

**Attack path:**

1. Attacker submits a transaction that declares Cairo0 class `C` and deploys contract `A` with class hash `C` in proposal `P`.
2. During batcher validation of `P`, `get_compiled_from_class_manager` is called, finds a cache miss, fetches `C` from storage, and calls `set_and_compile` — populating the global `ContractClassManager` cache with `C`.
3. Proposal `P` is rejected (consensus failure, bouncer overflow, or explicit abort). The batcher calls `revert_state_diff`, which removes `C` from `deprecated_declared_classes_block_table` in storage. The `CachedState` wrapping the proposal is dropped.
4. The global `ContractClassManager` cache is **not** cleared. `C` remains cached as `CompiledClasses::V0(...)`.
5. In the next proposal `P'`, a transaction calls contract `A` (which was re-deployed in `P'` or was already deployed in an earlier committed block with class hash `C`).
6. `CachedState::get_compiled_class` finds no local entry for `C`, delegates to `StateReaderAndContractManager::get_compiled_class` → `get_compiled_from_class_manager`.
7. The global cache returns `RunnableCompiledClass::V0(C)`. The `V0` arm is a no-op — no `is_declared` check is performed.
8. The transaction executes with the undeclared class, producing a wrong execution result, wrong events, wrong storage writes, and a wrong state diff.

### Impact Explanation

Execution of an undeclared Cairo0 class produces a wrong `CallInfo`, wrong storage updates, wrong events, and a wrong `StateDiff`. These feed directly into `BlockExecutionArtifacts`, `CommitmentStateDiff`, and ultimately `calculate_block_commitments` / `calculate_block_hash`. The resulting block hash and state root are wrong for the accepted input, matching the Critical impact: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

### Likelihood Explanation

The trigger requires a Cairo0 class to be cached during a proposal that is subsequently reverted. Block reverts occur during normal batcher operation (proposal abort, consensus failure, bouncer overflow). The global cache is long-lived and shared across proposals. Any node running with a warm cache after a revert is vulnerable. The attacker controls the class declaration and the follow-up call transaction; no privileged access is required.

### Recommendation

Apply the same `is_declared`-equivalent guard to Cairo0 classes. Because `is_declared` always returns `false` for Cairo0 by design, a separate storage check is needed — for example, checking `get_deprecated_class_definition_block_number` and verifying it is `Some` and `<= current_block_number`. Alternatively, extend `FetchCompiledClasses::is_declared` to cover Cairo0 classes and use it uniformly for both arms:

```rust
match &runnable_class {
    RunnableCompiledClass::V0(_) => {
        // Same revert-cache risk as Cairo1: verify the class is still declared.
        if !self.state_reader.is_cairo0_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
    _ => {
        if !self.state_reader.is_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
}
```

### Proof of Concept

```
1. Submit declare(C: Cairo0) + deploy(A, class_hash=C) in proposal P.
   → get_compiled_from_class_manager cache miss → set_and_compile(C) → global cache holds C.

2. Reject proposal P → revert_state_diff(block_N) removes C from
   deprecated_declared_classes_block_table. CachedState dropped.
   Global ContractClassManager cache: still holds C.

3. Submit proposal P' containing invoke(A, ...).
   → CachedState::get_compiled_class(C): local map empty → delegates to
     StateReaderAndContractManager::get_compiled_class(C)
   → get_compiled_from_class_manager: cache hit → V0 arm → no is_declared check
   → returns RunnableCompiledClass::V0(C)
   → transaction executes with undeclared class C
   → wrong CallInfo / storage writes / events / StateDiff
   → wrong state root committed.
```

    let mut deleted_data = Vec::new();

pub struct GlobalContractCache<T: Clone>(pub Arc<Mutex<ContractLRUCache<T>>>);

impl<T: Clone> GlobalContractCache<T> {
    /// Locks the cache for atomic access. Although conceptually shared, writing to this cache is
    /// only possible for one writer at a time.
    pub fn lock(&self) -> LockedClassCache<'_, T> {
        self.0.lock().expect("Global contract cache is poisoned.")
    }

    pub fn get(&self, class_hash: &ClassHash) -> Option<T> {
        self.lock().cache_get(class_hash).cloned()
    }

    pub fn set(&self, class_hash: ClassHash, contract_class: T) {
        self.lock().cache_set(class_hash, contract_class);
    }

    pub fn clear(&mut self) {
        self.lock().cache_clear();
    }

    pub fn new(cache_size: usize) -> Self {
        Self(Arc::new(Mutex::new(ContractLRUCache::<T>::with_size(cache_size))))
    }
