### Title
`CommitmentStateDiff ŌåÆ ThinStateDiff` Hardcodes `deprecated_declared_classes: Vec::new()`, Producing Wrong `state_diff_commitment` and `concatenated_counts` in Block Hash - (`crates/blockifier/src/state/cached_state.rs`)

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally sets `deprecated_declared_classes: Vec::new()`. When a block contains a Declare V0/V1 (Cairo 0) transaction, the `ThinStateDiff` fed into `calculate_block_commitments` is missing those class hashes. This produces a wrong `state_diff_commitment` and a wrong `state_diff_length` inside `concatenated_counts`, both of which are chained into the final block hash via `PartialBlockHashComponents`. The SNOS computes the state diff commitment over the real set of deprecated declared classes, so the sequencer's commitment diverges from the prover's, breaking proof verification.

### Finding Description

**Root cause ŌĆö hardcoded empty field:**

```rust
// crates/blockifier/src/state/cached_state.rs
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ŌåÉ always empty
        }
    }
}
``` [1](#0-0) 

`CommitmentStateDiff` has no field for deprecated declared classes; they are silently dropped. The TODO comment acknowledges this is a structural gap, not an intentional design choice.

**Propagation into block hash:**

`BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())` and passes the result directly to `calculate_block_commitments`:

```rust
// crates/apollo_batcher/src/block_builder.rs
let transactions_data =
    prepare_txs_hashing_data(&execution_data.execution_infos_and_signatures);
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
``` [2](#0-1) 

`calculate_block_commitments` then computes two values that depend on `deprecated_declared_classes`:

1. **`state_diff_commitment`** ŌĆö via `calculate_state_diff_hash`, which chains `deprecated_declared_classes` into the Poseidon hash:

```rust
// crates/starknet_api/src/block_hash/state_diff_hash.rs
hash_chain = chain_deprecated_declared_classes(
    &state_diff.deprecated_declared_classes,   // always [] from batcher path
    hash_chain
);
``` [3](#0-2) 

2. **`state_diff_length`** ŌĆö via `ThinStateDiff::len()`, which counts `deprecated_declared_classes.len()`:

```rust
// crates/starknet_api/src/state.rs
result += self.deprecated_declared_classes.len();   // always 0 from batcher path
``` [4](#0-3) 

`state_diff_length` is packed into `concatenated_counts` and chained into the block hash:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs
let concatenated_counts = concat_counts(
    transactions_data.len(),
    event_leaf_elements.len(),
    state_diff.len(),   // wrong: missing deprecated declared classes
    l1_da_mode,
);
``` [5](#0-4) 

Both corrupted values are stored in `BlockHeaderCommitments` and then in `PartialBlockHashComponents`, which feeds `calculate_block_hash`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs
.chain(&block_commitments.concatenated_counts)          // wrong
.chain(&block_commitments.state_diff_commitment.0.0)    // wrong
``` [6](#0-5) 

**Trigger ŌĆö Declare V0/V1 is accepted by the blockifier:**

The blockifier's `verify_tx_version` explicitly allows `TransactionVersion::ZERO` and `TransactionVersion::ONE` for Declare transactions:

```rust
// crates/blockifier/src/transaction/account_transaction.rs
Transaction::Declare(_) => {
    vec![
        TransactionVersion::ZERO,
        TransactionVersion::ONE,
        TransactionVersion::TWO,
        TransactionVersion::THREE,
    ]
}
``` [7](#0-6) 

A Declare V0/V1 transaction declares a Cairo 0 (deprecated) class. Its class hash goes into `ThinStateDiff::deprecated_declared_classes` when the full `StateDiff` path is used (e.g., `ThinStateDiff::from_state_diff`), but is silently dropped in the `CommitmentStateDiff ŌåÆ ThinStateDiff` path used by the batcher.

**Contrast with the correct path:**

`ThinStateDiff::from_state_diff` correctly populates the field:

```rust
// crates/starknet_api/src/state.rs
deprecated_declared_classes: diff
    .deprecated_declared_classes
    .keys()
    .copied()
    .collect(),
``` [8](#0-7) 

The batcher uses the `CommitmentStateDiff` path, not `from_state_diff`, so it never populates this field.

### Impact Explanation

A block containing one or more Declare V0/V1 transactions will have:
- A wrong `state_diff_commitment` (missing the deprecated class hashes from the Poseidon chain)
- A wrong `state_diff_length` inside `concatenated_counts` (count is short by the number of deprecated declared classes)
- A wrong final block hash stored in `PartialBlockHashComponents` and propagated to `ProposalCommitment`

The SNOS computes the state diff commitment over the actual state diff (including deprecated declared classes). The sequencer's commitment diverges from the prover's, causing proof verification to fail for any block that includes a deprecated class declaration. This matches the "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input" (Critical) and the commitment construction audit pivot.

### Likelihood Explanation

The blockifier accepts Declare V0/V1 at the execution layer. Any unprivileged user who submits a valid Declare V0 or V1 transaction that passes gateway admission and is included in a block triggers the wrong commitment. The likelihood is medium: deprecated declares are rare in modern Starknet but are not blocked at the execution layer, and the structural gap has been present since the `CommitmentStateDiff` type was introduced (evidenced by the TODO comment).

### Recommendation

Add a `deprecated_declared_classes` field to `CommitmentStateDiff` and populate it during state diff finalization in the blockifier, mirroring how `ThinStateDiff::from_state_diff` handles it. The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then forward this field instead of hardcoding `Vec::new()`.

### Proof of Concept

1. Submit a Declare V0 transaction (Cairo 0 class) through the RPC gateway.
2. The batcher includes it in a block and calls `BlockExecutionArtifacts::new`.
3. `ThinStateDiff::from(commitment_state_diff)` produces a `ThinStateDiff` with `deprecated_declared_classes = []`.
4. `calculate_state_diff_hash` computes `Poseidon("STARKNET_STATE_DIFF0", ..., 0 deprecated classes, ...)` ŌĆö missing the actual class hash.
5. `ThinStateDiff::len()` returns a count that is short by 1 (or more), corrupting `concatenated_counts`.
6. `calculate_block_hash` chains both wrong values, producing a block hash that differs from what the SNOS would compute for the same block.
7. Proof verification fails; the block cannot be settled on L1.
