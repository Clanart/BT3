### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` for V0_14_x blocks, producing a wrong block hash commitment — (`File: crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

The `gas_prices_to_hash` helper that feeds the gas-prices field of every block hash is frozen at the V0_13_4 schema. A developer-acknowledged TODO states that `l2_gas_consumed` and `next_l2_gas_price` must be added to the hash "after 0.14.0". The codebase already carries `StarknetVersion::V0_14_2`, `V0_14_3`, and test fixtures for `0.14.4`, yet the function signature and implementation were never updated. Every block produced at V0_14_x therefore commits a gas-prices hash that is missing two fields, making the sequencer's authoritative block hash diverge from the on-chain specification.

---

### Finding Description

**Root cause — frozen `BlockHashVersion` enum and incomplete `gas_prices_to_hash`**

`BlockHashVersion` has exactly two variants:

```rust
pub enum BlockHashVersion {
    V0_13_2,
    V0_13_4,
}
```

The `TryFrom<StarknetVersion>` implementation maps every version ≥ V0_13_4 — including all V0_14_x releases — to `BlockHashVersion::V0_13_4`:

```rust
} else {
    Ok(Self::V0_13_4)   // catches V0_14_2, V0_14_3, V0_14_4, …
}
```

`gas_prices_to_hash`, called from both `calculate_block_hash` (Rust) and the `GetBlockHashes` OS hint, accepts only three price tokens and produces:

```
Poseidon("STARKNET_GAS_PRICES0",
         l1_gas_price_wei, l1_gas_price_fri,
         l1_data_gas_price_wei, l1_data_gas_price_fri,
         l2_gas_price_wei, l2_gas_price_fri)
```

The function carries an explicit, unresolved TODO:

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> {
```

`l2_gas_consumed` and `next_l2_gas_price` are fully present in the protocol: `BlockHeaderWithoutHash` carries both fields, the P2P protobuf converter deserialises them, the storage layer persists them, and the echonet system-test harness explicitly passes them to the block-hash CLI:

```python
"l2_gas_consumed": fee_market_info["l2_gas_consumed"],
"next_l2_gas_price": fee_market_info["next_l2_gas_price"],
```

Despite being passed to the CLI, the Rust `calculate_block_hash` path never forwards them to `gas_prices_to_hash`, so they are silently dropped from the hash preimage.

**Propagation path**

1. `calculate_block_hash` (called by `CommitmentManager::finalize_commitment_output`) computes the wrong gas-prices sub-hash.
2. The wrong block hash is written to storage and broadcast to peers via `SyncBlock`.
3. The OS hint `get_block_hashes` calls the same `gas_prices_to_hash` with the same frozen schema, so the SNOS proof input also carries the wrong hash.
4. The `PartialBlockHash` used during consensus proposal validation is derived from the same function, so validators accept proposals whose committed hash is wrong.

---

### Impact Explanation

Every block at Starknet version ≥ 0.14.0 receives a block hash whose gas-prices component is computed over a shorter preimage than the protocol specifies. The sequencer stores, broadcasts, and proves this wrong value as the authoritative block hash. Any RPC call returning `block_hash`, any L1 state-update commitment, and any SNOS proof input that references the block hash will carry the incorrect value. This matches:

> **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

and potentially:

> **Critical. Wrong state … storage value … from blockifier/syscall/execution logic for accepted input** (the block hash is stored at the block-hash contract address 0x1 and is readable by Cairo contracts via `get_block_hash`).

---

### Likelihood Explanation

The bug is triggered by ordinary block production at any Starknet version ≥ 0.14.0. No adversarial action is required; the sequencer silently produces the wrong hash for every such block. The codebase already contains `StarknetVersion::V0_14_2`, `V0_14_3`, and a test fixture for `0.14.4`, confirming the affected version range is live.

---

### Recommendation

1. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` parameters to `gas_prices_to_hash`.
2. Add a new `BlockHashVersion::V0_14_0` (or the exact version at which the schema changed) and extend the `TryFrom<StarknetVersion>` match arm accordingly.
3. For V0_14_x, chain both new fields into the Poseidon preimage before `get_poseidon_hash()`.
4. Update the OS hint `get_block_hashes` to pass the same fields so the SNOS proof input stays consistent.
5. Add a regression test analogous to `test_block_hash_regression` that covers a V0_14_x block and asserts the expected hash changes when `l2_gas_consumed` or `next_l2_gas_price` is modified.

#[derive(Clone, Debug, PartialEq, Eq, PartialOrd)]
pub enum BlockHashVersion {
    V0_13_2,
    V0_13_4,
}

impl From<BlockHashVersion> for StarknetVersion {
    fn from(value: BlockHashVersion) -> Self {
        match value {
            BlockHashVersion::V0_13_2 => StarknetVersion::V0_13_2,
            BlockHashVersion::V0_13_4 => StarknetVersion::V0_13_4,
        }
    }
}

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
