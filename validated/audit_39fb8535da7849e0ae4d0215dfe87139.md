### Title
Cached Cairo0 Class Bypasses State Verification After Reorg, Allowing Execution of Reverted Classes — (`crates/blockifier/src/state/state_reader_and_contract_manager.rs`)

---

### Summary

`StateReaderAndContractManager::get_compiled_from_class_manager` checks the global `ContractClassManager` cache first. For Cairo1 classes, it guards against stale cache entries from reverted blocks by calling `state_reader.is_declared(class_hash)`. For Cairo0 (`V0`) classes, this guard is entirely absent. After a reorg that reverts a block containing a Cairo0 class declaration, the class remains in the shared global cache and is returned to callers without any state verification, allowing execution of a class that no longer exists in the committed state.

---

### Finding Description

The structural analog to the external bug is:

| External (AgentToken) | Sequencer |
|---|---|
| `uniswapV2Pair` (immutable cached field) | `ContractClassManager` global LRU cache |
| `_liquidityPools.contains(addr)` (authoritative set) | `state_reader.is_declared(class_hash)` (authoritative state) |
| `removeLiquidityPool` removes from set but not from cached field | `revert_state_diff` removes from state tables but does not clear the cache |
| `isLiquidityPool` returns `true` for removed pool | `get_compiled_from_class_manager` returns the reverted class for a cache hit |

In `get_compiled_from_class_manager`:

```rust
if let Some(runnable_class) =
    self.contract_class_manager.get_runnable(&class_hash, &self.native_classes_whitelist)
{
    match &runnable_class {
        RunnableCompiledClass::V0(_) => {}   // ← NO state verification
        _ => {
            // The Cairo1 class is cached; verify it is declared,
            // since existence in the cache does not guarantee that
            // (it might contain a declared class from a reverted block, for example).
            if !self.state_reader.is_declared(class_hash)? {
                return Err(StateError::UndeclaredClassHash(class_hash));
            }
        }
    }
    return Ok(runnable_class);   // ← V0 returned without any check
}
```

The comment explicitly acknowledges the reorg concern for Cairo1 but the same concern applies to Cairo0 classes. The `is_declared` trait is documented as always returning `false` for Cairo0, so the developers intentionally skipped the check — but provided no alternative verification path for Cairo0. [1](#0-0) 

The `ContractClassManager` (`TrivialClassManager`) holds a global `RawClassCache` (an `Arc<Mutex<SizedCache>>`). It is shared across block proposals and is not cleared when `revert_state_diff` is called. [2](#0-1) 

`revert_state_diff` correctly removes the class from all state tables (`declared_classes`, `declared_classes_block`, `casms`, `stateless_compiled_class_hash_v2`, etc.) but has no mechanism to invalidate the in-process `ContractClassManager` cache. [3](#0-2) 

The `stateless_compiled_class_hash_v2` table is explicitly described as "separate from Papyrus storage; scope and version do not apply on it" and uses `upsert` with no delete path during normal operation — though `delete_compiled_class_hashes_v2` is called during revert. The in-memory `compiled_class_hash_v2_cache` inside `TrivialClassManager` is never cleared on revert. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

After a reorg that reverts block N (which declared a Cairo0 class `C`):

1. `C` is removed from all state tables by `revert_state_diff`.
2. The global `ContractClassManager` cache still holds `C` as `RunnableCompiledClass::V0`.
3. The batcher rebuilds block N′ using the same `ContractClassManager`.
4. A transaction in N′ that calls a contract deployed with class `C` triggers `get_compiled_class(C)`.
5. `get_compiled_from_class_manager` finds `C` in cache as `V0`, skips `is_declared`, and returns it.
6. Execution proceeds with a class that is not declared in the current state.

This produces a wrong execution result, wrong `CommitmentStateDiff`, wrong `ThinStateDiff`, and therefore a wrong `state_diff_commitment` fed into `calculate_block_commitments` and ultimately into `calculate_block_hash`. The block commitment is cryptographically bound to a state diff that includes effects from a class that does not exist in the committed state. [6](#0-5) [7](#0-6) 

**Impact category**: Critical — Wrong state, receipt, or revert result from blockifier/execution logic for accepted input; wrong block commitment derived from incorrect state diff.

---

### Likelihood Explanation

The trigger requires a reorg (revert of a committed block). In the sequencer, `revert_state_diff` is a supported production operation used when consensus fails to finalize a block that was optimistically appended to storage. The batcher then rebuilds the block using the same shared `ContractClassManager`. Cairo0 class declarations are rare but valid on Starknet. An attacker who can influence consensus (e.g., a validator) can deliberately cause a reorg after a Cairo0 class is declared, then submit a transaction using that class in the rebuilt block.

---

### Recommendation

Add a Cairo0-specific declared-class check in `get_compiled_from_class_manager`. Since `is_declared` always returns `false` for Cairo0, a separate verification path is needed — for example, calling `state_reader.get_compiled_class(class_hash)` directly (bypassing the cache) to confirm the class is still reachable from state, or clearing the `ContractClassManager` cache entries for all classes in the reverted `ThinStateDiff` when `revert_state_diff` is called.

```rust
RunnableCompiledClass::V0(_) => {
    // Same reorg concern as Cairo1: verify the class is still declared.
    // is_declared() always returns false for Cairo0, so use a direct state check.
    if self.state_reader.get_compiled_classes(class_hash).is_err() {
        return Err(StateError::UndeclaredClassHash(class_hash));
    }
}
```

Alternatively, propagate revert events to the `ContractClassManager` so it can evict stale entries.

---

### Proof of Concept

1. Deploy a contract using Cairo0 class `C` in block N. The class is cached in `ContractClassManager`.
2. Revert block N via `revert_state_diff(N)`. Class `C` is removed from all state tables.
3. Construct a new `StateReaderAndContractManager` wrapping the post-revert state reader, but sharing the same `ContractClassManager` (as the batcher does).
4. Call `get_compiled_class(C)` on the new `StateReaderAndContractManager`.
5. Observe: `get_compiled_from_class_manager` finds `C` in cache as `V0`, skips `is_declared`, and returns the class — even though `state_reader.get_compiled_classes(C)` would return `UndeclaredClassHash`.
6. Execute a transaction calling a contract deployed with `C`. Execution succeeds, producing a wrong state diff and wrong block commitment. [8](#0-7) [9](#0-8)

    }

pub async fn calculate_block_commitments(
    transactions_data: &[TransactionHashingData],
    state_diff: ThinStateDiff,
    l1_da_mode: L1DataAvailabilityMode,
    starknet_version: &StarknetVersion,
) -> (BlockHeaderCommitments, BlockCommitmentsMeasurements) {
    let transaction_leaf_elements: Vec<TransactionLeafElement> = transactions_data
        .iter()
        .map(|tx_leaf| {
            let mut tx_leaf_element = TransactionLeafElement::from(tx_leaf);
            if starknet_version < &BlockHashVersion::V0_13_4.into()
                && tx_leaf.transaction_signature.0.is_empty()
            {
                tx_leaf_element.transaction_signature =
                    TransactionSignature(vec![Felt::ZERO].into());
            }
            tx_leaf_element
        })
        .collect();

    let event_leaf_elements: Vec<EventLeafElement> = transactions_data
        .iter()
        .flat_map(|transaction_data| {
            transaction_data.transaction_output.events.iter().map(|event| EventLeafElement {
                event: event.clone(),
                transaction_hash: transaction_data.transaction_hash,
            })
        })
        .collect();

    let receipt_elements: Vec<ReceiptElement> =
        transactions_data.iter().map(ReceiptElement::from).collect();

    let concatenated_counts = concat_counts(
        transactions_data.len(),
        event_leaf_elements.len(),
        state_diff.len(),
        l1_da_mode,
    );

    let n_txs = transactions_data.len();
    let n_events = event_leaf_elements.len();
    let state_diff_length = state_diff.len();

    // Spawn tasks for parallel execution; each measures its own duration.
    let transaction_task = spawn_measured_task(move || {
        calculate_transaction_commitment::<Poseidon>(&transaction_leaf_elements)
    });

    let event_task =
        spawn_measured_task(move || calculate_event_commitment::<Poseidon>(&event_leaf_elements));

    let receipt_task =
        spawn_measured_task(move || calculate_receipt_commitment::<Poseidon>(&receipt_elements));

    let state_diff_task = spawn_measured_task(move || calculate_state_diff_hash(&state_diff));

    // Wait for all tasks to complete.
    let (
        (transaction_commitment, transaction_duration),
        (event_commitment, event_duration),
        (receipt_commitment, receipt_duration),
        (state_diff_commitment, state_diff_duration),
    ) = tokio::try_join!(transaction_task, event_task, receipt_task, state_diff_task)
        .expect("Failed to join block commitments tasks.");

    let commitments = BlockHeaderCommitments {
        transaction_commitment,
        event_commitment,
        receipt_commitment,
        state_diff_commitment,
        concatenated_counts,
    };
    let measurements = BlockCommitmentsMeasurements {
        transaction_commitment_duration: transaction_duration,
        event_commitment_duration: event_duration,
        receipt_commitment_duration: receipt_duration,
        state_diff_commitment_duration: state_diff_duration,
        n_txs,
        n_events,
        state_diff_length,
    };
    (commitments, measurements)
}
