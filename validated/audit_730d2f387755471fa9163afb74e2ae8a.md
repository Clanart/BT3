### Title
`set_state_header` accepts a chunk for shard B as the state sync header for shard A due to missing leaf-position check in `verify_path` — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` verifies that the supplied chunk is *a member* of the block's `chunk_headers_root` Merkle tree, but does not verify that it occupies the leaf slot corresponding to the requested `shard_id`. A malicious peer can supply shard B's chunk (with shard B's `prev_state_root`) as the state sync header for shard A, causing the syncing node to install shard B's state root for shard A.

### Finding Description
In `set_state_header` (step 3a), the check is:

```rust
verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
)
```

`verify_path` computes `compute_root_from_path(path, hash(item)) == root`. It does not verify that the path's Left/Right directions correspond to the `shard_index`-th leaf position. A proof for shard B's chunk at position B in the tree will satisfy this check even when `shard_id = A` is passed.

The function `verify_path_with_index` (used correctly in `shards_manager_actor.rs` for chunk parts) calls `verify_path_matches_index` which checks that each path item's direction matches the expected direction for a given leaf index. This function is **not called** in `set_state_header`.

Additionally, `chunk.shard_id()` is never compared to the `shard_id` argument anywhere in `set_state_header`, and `validate_chunk_proofs` does not check shard identity.

The receipt proof validation in step 4 uses the caller-supplied `shard_id` (shard A) to hash receipts, so an attacker can supply correct receipt proofs for shard A (public on-chain data) while substituting shard B's chunk. Step 5 validates `state_root_node` against `chunk_inner.prev_state_root()` — but both are from shard B's chunk and are internally consistent, so this check also passes.

The corrupted header is then stored under `StateHeaderKey(shard_A, sync_hash)` with shard B's `prev_state_root`.

### Impact Explanation
When `set_state_finalize` subsequently reads this header and calls `apply_chunk` with `RuntimeStorageConfig::new(chunk_header.prev_state_root(), true)`, it applies shard A's transactions and receipts against shard B's state trie. The resulting state root is wrong for shard A. The syncing node diverges from the canonical chain state for shard A. [7](#0-6) 

### Likelihood Explanation
Any peer participating in state sync can send a crafted `ShardStateSyncResponseHeader`. The attacker needs: (1) a valid chunk for shard B (available from any full node), (2) shard B's Merkle proof (computable from the block), (3) valid receipt proofs for shard A (public on-chain data). No validator or privileged role is required. The attack is executable by any unprivileged network peer during state sync.

### Recommendation
Replace `verify_path` with `verify_path_with_index` in step 3a (and step 3b for `prev_chunk_proof`), passing the `shard_index` derived from `shard_layout.get_shard_index(shard_id)` and the total number of shards. Additionally, add an explicit check `assert_eq!(chunk.shard_id(), shard_id)` after `validate_chunk_proofs`.

### Proof of Concept
```rust
// In a 2-shard network (shard 0 = A, shard 1 = B):
// 1. Obtain shard B's chunk and its Merkle proof at index 1
let (chunk_headers_root, chunk_proofs) = merklize(&[
    ChunkHashHeight(shard_A_chunk.chunk_hash(), shard_A_chunk.height_included()),
    ChunkHashHeight(shard_B_chunk.chunk_hash(), shard_B_chunk.height_included()),
]);
// chunk_proofs[1] is shard B's proof

// 2. Construct a ShardStateSyncResponseHeaderV2 with shard B's chunk
//    but shard B's proof, plus valid receipt proofs for shard A
let malicious_header = ShardStateSyncResponseHeaderV2 {
    chunk: shard_B_chunk,          // shard B's chunk, shard B's prev_state_root
    chunk_proof: chunk_proofs[1],  // valid proof for position 1 (shard B)
    // ... valid receipt proofs for shard A (public data)
};

// 3. Submit as header for shard A
adapter.set_state_header(shard_A_id, sync_hash, malicious_header.into())
// → Ok(()), header stored under StateHeaderKey(shard_A, sync_hash)
//   with shard B's prev_state_root
```

`verify_path(chunk_headers_root, &chunk_proofs[1], ChunkHashHeight(B_hash, B_height))` returns `true` because shard B's chunk IS in the tree at position 1 — the position check is absent.

        byzantine_assert!(false);
        return Ok(false);

    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();

    // 2b. Checking that chunk transactions are valid

        byzantine_assert!(false);
        return Ok(false);

            byzantine_assert!(false);
            return Ok(false);
        }
    }
