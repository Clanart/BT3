Looking at the `claim_fee` flow in `near/omni-bridge/src/lib.rs`, I can identify a direct analog to the H-01 access control mismatch pattern.

### Title
Fee Permanently Locked When `fee_recipient` Is Not a Trusted Relayer in `claim_fee` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`claim_fee` is gated by `#[trusted_relayer]`, restricting callers to the trusted-relayer set. Its callback then enforces `fee_recipient == predecessor_account_id`. Because `fee_recipient` is an arbitrary `AccountId` set during `sign_transfer` with no constraint that it must be a trusted relayer, the two access-control requirements can never simultaneously be satisfied for a non-relayer `fee_recipient`. The result is a permanent lock of the fee portion of the transfer in `pending_transfers`.

---

### Finding Description

**`claim_fee` (lines 1057–1064)** is decorated with `#[trusted_relayer]`, which blocks any caller that is not in the trusted-relayer set (DAO and `UnrestrictedRelayer` bypass roles are also permitted):

```rust
#[payable]
#[trusted_relayer]
#[pause(except(roles(Role::DAO)))]
pub fn claim_fee(&mut self, #[serializer(borsh)] args: ClaimFeeArgs) -> Promise {
    self.verify_proof(args.chain_kind, args.prover_args).then(
        Self::ext(env::current_account_id())
            .with_attached_deposit(env::attached_deposit())
            .with_static_gas(CLAIM_FEE_CALLBACK_GAS)
            .claim_fee_callback(&env::predecessor_account_id()),   // ← captures caller identity
    )
}
``` [1](#0-0) 

The captured `predecessor_account_id` (the trusted relayer's account) is then checked inside **`claim_fee_callback` (lines 1083–1086)**:

```rust
require!(
    fee_recipient == *predecessor_account_id,
    BridgeError::OnlyFeeRecipientCanClaim.as_ref()
);
``` [2](#0-1) 

The `fee_recipient` value comes from the proof (`fin_transfer.fee_recipient`), which was embedded in the MPC-signed payload during `sign_transfer`. The `sign_transfer` function accepts any `AccountId` as `fee_recipient` with **no validation** that it is a trusted relayer:

```rust
pub fn sign_transfer(
    &mut self,
    transfer_id: TransferId,
    fee_recipient: Option<AccountId>,   // ← any account, no relayer check
    fee: &Option<Fee>,
) -> Promise {
``` [3](#0-2) 

This creates an irreconcilable access-control mismatch:

| Caller type | Can call `claim_fee`? | Passes callback check? |
|---|---|---|
| Non-trusted-relayer account (e.g., treasury, multisig) | **No** — blocked by `#[trusted_relayer]` | N/A |
| Trusted relayer whose account ≠ `fee_recipient` | Yes | **No** — `fee_recipient != predecessor_account_id` |
| DAO (bypass role) whose account ≠ `fee_recipient` | Yes | **No** — same check applies |

The only account that can succeed is one that is simultaneously a trusted relayer **and** equals `fee_recipient`. If `fee_recipient` is any other account, the fee is permanently unclaimable.

A second trigger path: a trusted relayer who set `fee_recipient = their_own_account` is later removed from the trusted-relayer set (by `Role::RelayerManager` or `Role::DAO`). After removal they cannot call `claim_fee`, and no other trusted relayer can satisfy the callback check. The fee is again permanently locked. [4](#0-3) 

---

### Impact Explanation

The fee portion of the transfer amount is permanently frozen inside `pending_transfers`. The `remove_transfer_message` call inside `claim_fee_callback` is reached only **after** the `require!` check passes; if the check fails the function panics and the transfer message is never removed. No admin escape hatch exists to forcibly remove a pending transfer and recover the fee. This satisfies the **Critical/High** impact criterion: *permanent freezing or irrecoverable lock of user or protocol funds in bridge flows*.

---

### Likelihood Explanation

Moderate-to-high. Trusted relayers routinely direct fees to a separate treasury, multisig, or cold wallet — all of which are non-relayer accounts. The `sign_transfer` interface explicitly accepts an optional `fee_recipient` parameter, implying this is an intended use case. Additionally, relayer churn (removal via `RelayerManager`) is a normal operational event that can strand already-signed, not-yet-claimed fees. Neither scenario requires malicious intent.

---

### Recommendation

Remove the `#[trusted_relayer]` guard from `claim_fee`, or introduce a separate permissionless entry point that accepts a proof and sends the fee directly to the `fee_recipient` embedded in the proof without comparing it to `predecessor_account_id`. The identity of the caller is irrelevant to fee correctness; only the proof's authenticity and the `fee_recipient` field matter.

---

### Proof of Concept

1. Trusted relayer calls `sign_transfer(transfer_id, fee_recipient = Some("treasury.near"), ...)`. `treasury.near` is a multisig, not a trusted relayer.
2. MPC signs the payload; the signed message (including `fee_recipient = treasury.near`) is submitted to the destination EVM chain.
3. The EVM bridge emits a `FinTransfer` event with `fee_recipient = treasury.near`.
4. Any account calls `claim_fee` with a proof of that event.
   - If the caller is `treasury.near`: blocked at the `#[trusted_relayer]` gate — **reverts**.
   - If the caller is any trusted relayer `R`: passes the gate, but `claim_fee_callback` checks `"treasury.near" == R` — **reverts**.
5. The transfer message remains in `pending_transfers` indefinitely. The fee is permanently locked.

    #[private]
    #[payable]
    pub fn claim_fee_callback(
        &mut self,
        #[serializer(borsh)] predecessor_account_id: &AccountId,
        #[callback_result]
        #[serializer(borsh)]
        call_result: Result<ProverResult, PromiseError>,
    ) -> PromiseOrValue<()> {
        let Ok(ProverResult::FinTransfer(fin_transfer)) = call_result else {
            env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str())
        };

        let fee_recipient = fin_transfer.fee_recipient.unwrap_or_else(|| {
            env::panic_str(BridgeError::FeeRecipientNotSetOrEmpty.to_string().as_str());
        });

        require!(
            fee_recipient == *predecessor_account_id,
            BridgeError::OnlyFeeRecipientCanClaim.as_ref()
        );
        require!(
            self.factories
                .get(&fin_transfer.emitter_address.get_chain())
                == Some(fin_transfer.emitter_address),
            BridgeError::UnknownFactory.as_ref()
        );

        let transfer_message = self.remove_transfer_message(fin_transfer.transfer_id);

        if let Some(origin_transfer_id) = transfer_message.origin_transfer_id.clone() {
            let mut fast_transfer = FastTransfer::from_transfer(
                transfer_message.clone(),
                self.get_token_id(&transfer_message.token),
            );
            fast_transfer.transfer_id = origin_transfer_id;

            if let Some(fast_transfer_status) = self.get_fast_transfer_status(&fast_transfer.id()) {
                // For fast transfers we need to wait for finalization of the first leg (Origin chain -> Near) before allowing fee claim.
                // This confirms that fast transfer was executed with correct parameters.
                // Othewise malicious relayer can create a fast transfer with arbitrary high fee and claim it here.
                if fast_transfer_status.finalised {
                    self.remove_fast_transfer(&fast_transfer.id());
                } else {
                    env::panic_str(BridgeError::FastTransferNotFinalised.to_string().as_str());
                }
            }
        }

        let token = self.get_token_id(&transfer_message.token);
        let token_address = self
            .get_token_address(transfer_message.get_destination_chain(), token.clone())
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let denormalized_amount = Self::denormalize_amount(
            fin_transfer.amount.0,
            self.token_decimals
                .get(&token_address)
                .near_expect(BridgeError::TokenDecimalsNotFound),
        );
        // Fee includes both the user-specified fee and any dust lost during decimal
        // normalization (see `normalize_amount`). Since `denormalize(normalize(x)) <= x`
        // due to floor division, the difference naturally captures the normalization remainder.
        let fee = transfer_message.amount.0 - denormalized_amount;

        self.send_fee_internal(&transfer_message, fee_recipient, fee)
    }
