### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Breaks EVM Bridge Collateralization - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer()` accepts a caller-supplied `amount` parameter, pulls tokens via `safeTransferFrom`, and then emits an `InitTransfer` event using that same caller-supplied `amount`. For fee-on-transfer ERC20 tokens, the bridge vault receives fewer tokens than `amount`, but the event records the full `amount`. The NEAR-side relayer finalizes the cross-chain transfer for the full `amount`, minting more wrapped tokens on NEAR than are actually backed by EVM-side collateral.

### Finding Description
In `OmniBridge.initTransfer()`, the non-bridge, non-custom-minter ERC20 path (the `else` branch) calls:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied, not verified against actual receipt
);
```

Immediately after, the function emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // same caller-supplied value, not actual received balance
    fee,
    nativeFee,
    recipient,
    message
);
```

No balance-before/balance-after check is performed to determine the actual number of tokens received. For a fee-on-transfer token (e.g., a deflationary ERC20 that deducts 1% on every transfer), if a user calls `initTransfer` with `amount = 1000`, the bridge vault receives only 990 tokens, but the event records 1000. The NEAR bridge relayer reads this event and finalizes the transfer on NEAR for 1000 tokens, minting 1000 wrapped tokens to the recipient. The EVM vault is now undercollateralized by 10 tokens.

### Impact Explanation
Every `initTransfer` call with a fee-on-transfer token creates a collateral deficit in the EVM vault. Over time, or in a single large transfer, the cumulative shortfall means the EVM bridge cannot honor all redemptions from NEAR back to EVM. The last users to bridge back will find the vault insolvent and their funds permanently locked. This directly breaks bridge collateralization and constitutes an irrecoverable lock of user funds.

This matches the allowed High impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation
Any unprivileged user can call `initTransfer` with any ERC20 token that is not a registered bridge token and has no custom minter. The bridge imposes no allowlist on which ERC20 tokens can be deposited via this path. Fee-on-transfer tokens exist on mainnet (e.g., tokens with reflection mechanics, certain stablecoins in specific configurations). A single user bridging such a token is sufficient to trigger the deficit; no coordination or privileged access is required.

### Recommendation
Measure the actual received amount using a balance-before/balance-after pattern and use that value in the event and downstream accounting:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    require(actualReceived > 0, "Zero received");
    amount = uint128(actualReceived); // use actual amount for event and accounting
}
```

### Proof of Concept
1. Deploy or use any ERC20 token with a 1% fee-on-transfer on an EVM chain supported by OmniBridge.
2. Approve OmniBridge to spend 1000 tokens.
3. Call `initTransfer(feeToken, 1000, 0, 0, "alice.near", "")`.
4. OmniBridge vault receives 990 tokens (1% fee deducted by the token contract).
5. `InitTransfer` event is emitted with `amount = 1000`.
6. NEAR relayer picks up the event and calls `fin_transfer` on the NEAR bridge with `amount = 1000`.
7. NEAR bridge mints 1000 wrapped tokens to `alice.near`.
8. EVM vault holds only 990 tokens but 1000 wrapped tokens exist on NEAR.
9. When any user attempts to redeem 1000 wrapped tokens from NEAR back to EVM, the EVM bridge cannot release 1000 tokens — the last 10 tokens are permanently unclaimable, locking user funds. [1](#0-0) [2](#0-1)
