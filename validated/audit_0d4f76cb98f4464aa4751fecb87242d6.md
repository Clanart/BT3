### Title
Missing Declared-Class Validation in `execute_replace_class` Enables Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The OS-level `execute_replace_class` syscall handler does not verify that the supplied new class hash corresponds to a previously declared class. This is the StarkNet protocol analog of the reported `_authorizeUpgrade()` missing-validation bug: a privileged state-mutating operation (class replacement, i.e., contract upgrade) is executed without a required validity check. An unprivileged transaction sender can exploit this to permanently set a contract's class hash to an undeclared value, making the contract permanently uncallable and freezing any funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` processes the `replace_class` syscall. After deducting gas, it reads the current `StateEntry` for the calling contract and writes a new `StateEntry` with the caller-supplied `class_hash`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing check. There is no assertion that `class_hash` exists in `contract_class_changes` (the dictionary of declared classes). The OS commits whatever value is supplied directly into the state. [1](#0-0) 

Compare this to the normal declare path, which enforces `prev_value=0` and `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`: [2](#0-1) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, every subsequent call to that contract will fail at class-lookup time because no valid Sierra/CASM class exists for that hash. The contract becomes permanently uncallable. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are irrecoverably frozen. The state commitment produced by the OS will include the corrupted class hash, making the freeze permanent and provable on L1.

---

### Likelihood Explanation

**Medium.**

The direct attack path requires the attacker to execute `replace_class` in the context of a contract that holds funds. Two realistic scenarios:

1. **Attacker-controlled contract acting as a honeypot**: An attacker deploys a contract that appears to be a legitimate vault or yield strategy. Users deposit funds. The attacker then calls a function that internally invokes `replace_class(invalid_hash)`. The OS accepts the call without validation, the contract is permanently broken, and all deposited funds are frozen.

2. **Vulnerable upgradeable contract**: A legitimate upgradeable contract (e.g., a DEX pool or bridge) exposes an upgrade function that calls `replace_class` with a caller-supplied hash, relying on the OS to reject invalid hashes (a reasonable assumption given the TODO comment implies this check *should* exist). An attacker who can reach that upgrade function (e.g., through a governance attack or a missing access-control check at the contract level) supplies an undeclared hash. The OS does not reject it, and the contract is permanently frozen.

The vulnerability is reachable by any unprivileged transaction sender who can trigger `replace_class` within a contract execution context.

---

### Recommendation

Before committing the new `StateEntry`, assert that `request.class_hash` is present in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for `request.class_hash` and assert the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the protection already present in the declare transaction path and closes the gap acknowledged by the TODO comment.

---

### Proof of Concept

1. Attacker deploys Contract A (a "vault") with a public `freeze()` function that calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is never declared.
2. Users deposit 1000 STRK into Contract A.
3. Attacker sends an invoke transaction calling `freeze()` on Contract A.
4. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`.
5. Line 906–910 of `syscall_impls.cairo` writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes` with no validation.
6. The block is proven and committed to L1 with this state.
7. All subsequent calls to Contract A fail — no class exists for `0xdeadbeef`.
8. The 1000 STRK are permanently frozen. [3](#0-2)
