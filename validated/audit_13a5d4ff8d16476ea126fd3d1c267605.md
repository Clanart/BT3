### Title
Wrong State Diff Commitment and Block Hash Due to Missing `deprecated_declared_classes` in `CommitmentStateDiff` → `ThinStateDiff` Conversion — (File: `crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion in the blockifier unconditionally sets `deprecated_declared_classes: Vec::new()`. This `ThinStateDiff` is the sole input to `calculate_block_commitments` during block production. As a result, any block containing deprecated (Cairo 0) `Declare` transactions will have a wrong `state_diff_commitment` and a wrong `state_diff_length` packed into `concatenated_counts`, both of which are chained into the final block hash. The analog to the external report is exact: a value (the deprecated declared class list) is read from a structure that has already "reset" it to a default (empty), so the commitment is computed over the default rather than the actual data.

---

### Finding Description

**Root cause — `crates/blockifier/src/state/cached_state.rs` lines 690–701:**

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` has no field for deprecated declared classes. When a Cairo 0 `Declare` transaction executes, the blockifier records the class hash in the state but does not write it into `class_hash_to_compiled_class_hash` (that field is only for Cairo 1 classes with a compiled class hash). Consequently, the conversion above always produces `deprecated_declared_classes = []` regardless of how many Cairo 0 classes were declared in the block.

**Propagation path — `crates/apollo_batcher/src/block_builder.rs` lines 160–166:**

```rust
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // ← empty deprecated_declared_classes
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
```

`calculate_block_commitments` calls two functions that both consume `deprecated_declared_classes`:

1. **`calculate_state_diff_hash`** (`crates/starknet_api/src/block_hash/state_diff_hash.rs` lines 30–41) — chains `deprecated_declared_classes` into the Poseidon hash. With an empty list, the hash is `Poseidon(..., 0 /* count */, ...)` instead of `Poseidon(..., N, hash_0, hash_1, ...)`.

2. **`concat_counts`** (`crates/starknet_api/src/block_hash/block_hash_calculator.rs` lines 318–323) — uses `state_diff.len()` which adds `deprecated_declared_classes.len()`. With an empty list, the packed `state_diff_length` field is under-counted by exactly the number of deprecated declared classes in the block.

Both wrong values (`state_diff_commitment` and `concatenated_counts`) are then chained into `calculate_block_hash` (`crates/starknet_api/src/block_hash/block_hash_calculator.rs` lines 245–282), producing a wrong block hash that is stored in `partial_block_hash_components` and ultimately written to storage via `set_global_root_and_block_hash`.

---

### Impact Explanation

For every block that includes at least one deprecated (Cairo 0) `Declare` transaction:

- The `state_diff_commitment` stored in `BlockHeaderCommitments` is wrong — it is computed over a state diff that omits the declared class hashes.
- The `concatenated_counts` field (which encodes `state_diff_length`) is wrong — it under-counts the state diff length.
- The final block hash is wrong — both corrupted fields are chained into `calculate_block_hash`.
- The `ThinStateDiff` written to storage via `append_state_diff` (called from `commit_proposal_and_block`) also has empty `deprecated_declared_classes`, so the stored state diff is itself incomplete.

This matches the impact category: **Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** — specifically a wrong authoritative `state_diff_commitment` and block hash for accepted transactions.

---

### Likelihood Explanation

Deprecated `Declare` transactions (v1/v2) are still valid on Starknet and are submitted by any user deploying a Cairo 0 contract. No special privilege is required. Any such transaction included in a sequenced block triggers the bug. The TODO comment in the code (`// TODO(AlonH): Remove this when the structure of storage diffs changes.`) confirms this is a known structural gap that has not been resolved.

---

### Recommendation

Before calling `calculate_block_commitments`, the sequencer must reconstruct the full `deprecated_declared_classes` list from the execution state. The blockifier's `CachedState` tracks all class hash writes; deprecated declared classes can be identified as class hashes written to the state that have no corresponding entry in `class_hash_to_compiled_class_hash` (i.e., Cairo 0 classes). Alternatively, `CommitmentStateDiff` should be extended with a `deprecated_declared_classes` field populated during transaction execution, mirroring how `class_hash_to_compiled_class_hash` is populated for Cairo 1 classes. The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then use this field instead of `Vec::new()`.

---

### Proof of Concept

The structural proof is direct from the code:

1. `CommitmentStateDiff` has no `deprecated_declared_classes` field. [1](#0-0) 

2. `From<CommitmentStateDiff> for ThinStateDiff` hardcodes `deprecated_declared_classes: Vec::new()`. [2](#0-1) 

3. `BlockExecutionArtifacts::new` passes `ThinStateDiff::from(commitment_state_diff.clone())` directly to `calculate_block_commitments`. [3](#0-2) 

4. `calculate_block_commitments` calls `calculate_state_diff_hash` which chains `deprecated_declared_classes` into the Poseidon hash — with an empty list, the count field is `0` and no class hashes are chained. [4](#0-3) 

5. `calculate_block_commitments` also calls `concat_counts` with `state_diff.len()`, which counts `deprecated_declared_classes.len()` — always `0` here. [5](#0-4) 

6. Both corrupted values are chained into `calculate_block_hash`. [6](#0-5) 

A concrete trigger: submit a deprecated `Declare` transaction (Cairo 0 class) to the sequencer gateway. When the sequencer includes it in a block, `commitment_state_diff.class_hash_to_compiled_class_hash` will not contain the declared class hash (Cairo 0 has no compiled class hash), so `ThinStateDiff::from(commitment_state_diff)` produces `deprecated_declared_classes = []`. The `state_diff_commitment` and `concatenated_counts` in the resulting block hash will differ from the values a correct implementation would produce, and the wrong block hash is written to storage. [7](#0-6)
