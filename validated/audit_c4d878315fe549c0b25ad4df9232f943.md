### Title
Permissionless `register()` Irrevocably Clears Admin Blacklist, Neutralizing the Only Emergency-Stop for Compromised Pools — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase.register()` is a permissionless, fee-gated function that unconditionally clears the admin blacklist for any pool it processes. Because there is also no `unregister()` function, the admin's only two mitigation paths for a compromised pool — blacklisting and de-registration — are both unavailable: de-registration does not exist, and the blacklist can be reset by any caller for 1 wei (the default `registrationFee`).

---

### Finding Description

`OracleBase` exposes two relevant admin controls:

1. **`setBlacklist(address, bool)`** — ADMIN-only; prevents a blacklisted pool from calling `price()`.
2. **`register(bytes32, address, address)`** — permissionless, payable; whitelists a pool for a feed.

The critical flaw is inside `register()`:

```solidity
// OracleBase.sol lines 201-214
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);          // default: 1 wei
    require(pool != address(0));
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;                       // ← clears admin blacklist
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

The NatSpec comment even documents this as intentional: *"Paying also clears any blacklist on the pool."*

There is no corresponding `unregister()` anywhere in the codebase (confirmed by grep across all `.sol` files). Once `registeredPool[feedId][pool]` is set to `true`, it can never be set back to `false` through any admin path. The only mitigation is the blacklist, which is itself defeatable by the permissionless `register()`.

The `price()` function enforces both guards:

```solidity
// OracleBase.sol lines 160-172
function price(bytes32 feedId, address pool) external feedExists(feedId) notBlacklisted returns (...) {
    require(pool != address(0) && IPool(pool).inSwap() == msg.sender, InvalidInSwap());
    require(!blacklisted[pool], Blacklisted(pool));
    require(registeredPool[feedId][pool], NotRegistered(feedId, pool));
    ...
}
```

Both guards are bypassed once `register()` is called: `registeredPool` is already `true` and the blacklist is cleared.

---

### Impact Explanation

When a pool is actively being exploited (e.g., via a price-manipulation or callback-reentrancy attack), the admin's intended response is to call `setBlacklist(pool, true)`. This causes every subsequent `price()` call from that pool's price provider to revert, halting all swaps through the pool.

An attacker (or any third party) can immediately undo this by calling:

```solidity
register(feedId, pool, approvedFactory)
```

with `msg.value = 1 wei` (the default `registrationFee`). The blacklist is cleared atomically inside the same transaction. The pool resumes reading oracle prices, swaps proceed, and the exploit continues draining LP funds.

Because the admin cannot permanently de-register a pool either, there is no durable on-chain mechanism to stop a compromised pool from consuming oracle data. The only remaining option — removing the factory from `approvedFactories` — is a nuclear action that blocks all pools from that factory from registering, not just the compromised one.

---

### Likelihood Explanation

- **Trigger cost**: 1 wei (the hardcoded default `registrationFee`).
- **Caller restriction**: none — any EOA or contract can call `register()`.
- **Preconditions**: the pool must have been created by an approved factory (already satisfied for any legitimate pool being exploited).
- **Attacker motivation**: high — the attacker is already profiting from the exploit and has every incentive to spend 1 wei to continue.

---

### Recommendation

1. **Remove the blacklist-clearing side-effect from `register()`**. A blacklisted pool should not be able to re-register without explicit admin approval.

2. **Add an `unregister()` function** (direct analog to the Bond Protocol fix) so the admin can permanently revoke a pool's oracle access for a specific feed:

```solidity
function unregister(bytes32 feedId, address pool) external onlyRole(ADMIN_ROLE) {
    registeredPool[feedId][pool] = false;
    emit PoolUnregistered(feedId, pool);
}
```

3. **Require admin co-signature to re-register a previously blacklisted pool**, or gate `register()` with a check that reverts if `blacklisted[pool]` is true.

---

### Proof of Concept

```
1. Pool P is registered: registeredPool[feedId][P] = true
2. Pool P is found to be exploitable; admin calls setBlacklist(P, true)
   → price(feedId, P) now reverts with Blacklisted(P); swaps halt
3. Attacker calls register(feedId, P, approvedFactory) with msg.value = 1 wei
   → Inside register(): blacklisted[P] = false (line 208)
   → registeredPool[feedId][P] = true (line 212, already true, no-op)
4. price(feedId, P) succeeds again; swaps through P resume
5. Attacker continues draining LP positions; admin blacklist is permanently ineffective
   as long as the attacker can spend 1 wei per admin action
