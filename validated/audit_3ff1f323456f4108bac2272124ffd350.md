Audit Report

## Title
Router's Residual ETH Consumed for Any Caller's WETH Swap, Draining Prior Users' Funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` contains a WETH branch that unconditionally uses the router's native ETH balance to settle a swap before pulling WETH from the designated payer. Because all four swap entry points are `payable`, ETH can accumulate in the router across transactions. Any attacker who calls a swap with `tokenIn=WETH9` while the router holds residual ETH will have their swap settled using that ETH — paying nothing themselves — permanently stealing funds that belong to prior users.

## Finding Description
In `pay()` (`PeripheryPayments.sol` L73–84), when `token == WETH` and `payer != address(this)`, the function checks `address(this).balance` (the router's own ETH) and, if nonzero, wraps and transfers it to the pool instead of pulling WETH from the payer. There is no verification that this ETH was contributed by the current transaction's `msg.sender`.

ETH enters the router via any `payable` swap call (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). The `receive()` guard (`msg.sender != WETH`) only blocks plain ETH transfers with no calldata; it does not prevent ETH from accumulating via `payable` function calls. A user who sends `1 ETH` via `multicall` for an exact-output WETH swap where only `0.7 ETH` is consumed, and omits `refundETH`, leaves `0.3 ETH` stranded in the router.

An attacker then calls `exactInputSingle` with `tokenIn=WETH9` and `amountIn=0.3 ether`. The pool calls back into `metricOmmSwapCallback`, which dispatches to `_justPayCallback`, which calls `pay(WETH, attacker, pool, 0.3e18)`. Since `payer != address(this)`, the WETH branch fires, finds `nativeBalance >= value`, wraps the router's `0.3 ETH`, and sends WETH to the pool. The attacker receives swap output without spending any WETH or ETH of their own.

## Impact Explanation
Direct loss of user principal. Any ETH stranded in the router (a normal operational state) can be stolen in full by an unprivileged EOA. The victim's ETH is permanently lost with no recovery path. This meets the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
Residual ETH is a routine occurrence: any `multicall{value: X}` where actual input < X and `refundETH` is not appended leaves ETH behind. The attack requires no special permissions, no malicious pool, no non-standard token behavior, and no privileged role. It is executable by any EOA against any factory-registered WETH pool in a single transaction.

## Recommendation
Restrict the ETH-wrapping shortcut to the case where `payer == address(this)` (the router is paying from its own balance mid-path). When `payer != address(this)`, always use `safeTransferFrom` from the payer's WETH balance. A minimal fix:

```solidity
} else if (token == WETH) {
    if (payer == address(this)) {
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

Alternatively, track `msg.value` in a transient slot at swap entry and only allow `pay()` to consume up to that tracked amount of ETH per call.

## Proof of Concept
1. Victim calls `router.multicall{value: 1 ether}([exactOutputSingle(..., amountOut=X, amountInMaximum=1 ether)])` without appending `refundETH`. Actual input is `0.7 ETH`; `0.3 ETH` remains in the router.
2. Attacker calls `router.exactInputSingle(ExactInputSingleParams({tokenIn: WETH, amountIn: 0.3 ether, ...}))` with no ETH sent and no WETH approval.
3. Pool calls `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, attacker, pool, 0.3e18)`.
4. `nativeBalance = 0.3 ether >= value`; router wraps its own ETH and sends WETH to pool.
5. Attacker receives swap output; `address(router).balance == 0`; victim's `0.3 ETH` is gone.
