Looking at the codebase, I need to trace the proof verification path carefully to find the analog.

### Title
`run_proof_verification` Skips Cryptographic Proof Verification When `proof_facts` Hash Is Already Stored, Allowing Invalid Proofs to Be Admitted - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

---

### Summary

`run_proof_verification` checks whether the proof manager already contains an entry for the given `proof_facts` hash. If it does, the function returns early **without verifying the submitted `proof`**. Because `proof_facts` are public (embedded in the transaction hash and observable on-chain), any party can reuse a previously-stored `proof_facts` value while supplying an entirely invalid `proof`, causing both the gateway and consensus flows to admit the transaction without any cryptographic proof check.

---

### Finding Description

In `crates/apollo_transaction_converter/src/transaction_converter.rs`, `run_proof_verification` is the single shared verification gate for both the gateway and consensus paths:

```rust
async fn run_proof_verification(
    proof_facts: ProofFacts,
    proof: Proof,
    proof_manager_client: SharedProofManagerClient,
) -> Result<bool, TransactionConverterError> {
    let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

    if contains_proof {
        return Ok(false);   // ← proof is never checked
    }
    // actual cryptographic verification only reached on first submission
    starknet_proof_verifier::verify_proof(proof_facts, proof)?;
    Ok(true)
}
``` [1](#0-0) 

The guard `contains_proof(proof_facts)` queries the proof manager keyed by `proof_facts.hash()` — a Poseidon hash of the `proof_facts` vector. [2](#0-1) 

`proof_facts` for an Invoke V3 transaction is a public vector of Felt values (`[PROOF_VERSION, VIRTUAL_SNOS, program_hash, VIRTUAL_OS_OUTPUT_VERSION, block_number, block_hash, config_hash, ...]`) that is hashed into the transaction hash and therefore visible to any observer. [3](#0-2) 

The transaction hash includes `proof_facts` but **not** the `proof` blob itself: [4](#0-3) 

When `contains_proof` returns `true`, both the gateway flow (`spawn_proof_verification`) and the consensus flow (`spawn_verify_and_store_proof`) treat the task as complete without error, and the transaction proceeds to the mempool and batcher. [5](#0-4) 

The gateway awaits the verification task and propagates any error as `KnownStarknetErrorCode::InvalidProof` — but because `run_proof_verification` returns `Ok(false)` (not an error) when the cache hits, no rejection occurs. [6](#0-5) 

The blockifier's `validate_proof_facts` validates the semantic fields inside `proof_facts` (program hash, block number, block hash, config hash) but never touches the `proof` blob. [7](#0-6) 

**Analog to `onlyExecutor`:** The original bug checked `timelock.hasRole(EXECUTOR_ROLE, msg.sender)` — a property of the caller — instead of requiring `msg.sender == address(timelock)`. Here, `run_proof_verification` checks a property of the proof_facts (whether its hash is stored) instead of verifying the actual submitted proof. In both cases the check is necessary but not sufficient: holding the role / having a matching hash does not imply the current action is authorized / the current proof is valid.

---

### Impact Explanation

Any party who observes a previously accepted Invoke V3 transaction (on-chain or in the mempool) can extract its `proof_facts` and craft a new transaction with the same `proof_facts` but an arbitrary (invalid) `proof` blob. The gateway and consensus converter will skip `verify_proof` entirely, and the transaction will be admitted to the mempool and sequenced. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The `proof` is the cryptographic attestation that the Virtual OS executed the transaction correctly. Bypassing it means the sequencer accepts a client-side-proving transaction whose claimed pre-execution result has never been verified.

---

### Likelihood Explanation

- `proof_facts` are public: they appear in the transaction hash and are stored in block storage. Any observer can read them from any accepted transaction. [8](#0-7) 
- Multiple transactions can share the same `proof_facts` (same base block number and hash), so a single previously-stored proof unlocks bypass for all subsequent transactions referencing that base block.
- The production gateway config has `allow_client_side_proving: true`, confirming the path is live. [9](#0-8) 
- No privilege is required; any unprivileged user can submit an Invoke V3 transaction.

---

### Recommendation

Replace the early-return bypass with one of the following:

1. **Always verify**: Remove the `contains_proof` short-circuit entirely. Proof verification is a CPU-bound operation already dispatched to `spawn_blocking`; the cost is paid once per unique `(proof_facts, proof)` pair.

2. **Verify-then-cache**: If deduplication is desired for performance, retrieve the stored proof and verify that the submitted proof **matches** the stored one before skipping full re-verification:
   ```rust
   if let Some(stored) = proof_manager_client.get_proof(proof_facts.clone()).await? {
       if stored == proof {
           return Ok(false); // identical proof already verified
       }
   }
   // fall through to full verification
   ```

3. **Bind proof to transaction hash**: Include the `proof` blob (or its hash) in the transaction hash so that a different proof produces a different transaction identity, making reuse structurally impossible.

---

### Proof of Concept

1. **Setup**: User A submits `tx1` with `proof_facts = PF` (base block N) and a valid `proof = P`. `run_proof_verification` calls `verify_proof(PF, P)` → passes → `set_proof(PF, P)` stores the entry.

2. **Attack**: Attacker submits `tx2` with the same `proof_facts = PF` (same base block N, different calldata/nonce/sender) and `proof = P'` where `P'` is arbitrary garbage bytes.

3. **Bypass**: `run_proof_verification` calls `contains_proof(PF)` → returns `true` → returns `Ok(false)` without calling `verify_proof(PF, P')`.

4. **Admission**: The gateway's `await_verification_task_and_extract_proof_data` receives `Ok(())` from the task, extracts `(PF, P')` as proof data, and proceeds to store and sequence `tx2`. [6](#0-5) 

5. **Result**: `tx2` carrying an invalid proof is admitted to the mempool and sequenced. The `proof_facts` fields are validated by the blockifier, but the cryptographic proof `P'` is never checked anywhere in the pipeline.

    /// Used by the consensus flow, where tasks run concurrently with batcher execution and

        &self,
        proof_facts: ProofFacts,
        proof: Proof,
    ) -> VerifyAndStoreProofTask {
        let pmc = self.proof_manager_client.clone();
        let proof_facts_hash = proof_facts.hash();
        tokio::spawn(async move {

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
