### Title
Missing Token/Mint Association Check in `FinalizeTransfer` Enables Nonce Consumption with Wrong Token — (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

The `FinalizeTransfer` and `FinalizeTransferSol` instructions accept a `mint` account and a signed `FinalizeTransferPayload`, but the payload contains **no token/mint address field**, and the `mint` account has **no constraint binding it to the payload**. An unprivileged attacker can front-run a legitimate finalization transaction by substituting a different (worthless) mint, consuming the destination nonce while delivering the wrong token. The legitimate transfer is then permanently unclaimable because the nonce is already marked used.

---

### Finding Description

`FinalizeTransferPayload` is defined as:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
``` [1](#0-0) 

There is no `token` or `mint` field. The NEAR bridge MPC signature covers only these four fields; it does not commit to which SPL token should be transferred or minted on Solana.

The `mint` account in the `FinalizeTransfer` accounts struct carries only one constraint:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [2](#0-1) 

`mint::token_program = token_program` only validates that the mint belongs to the expected token program. There is no constraint of the form `mint.key() == data.payload.token_address` because no such field exists in the payload. The same absence exists in `FinalizeTransferSol`: [3](#0-2) 

The `process` function then unconditionally transfers or mints `data.amount` of whatever `self.mint` was supplied:

```rust
transfer_checked(... from: vault ..., mint: self.mint ..., data.amount ...)?;
// or
mint_to(... mint: self.mint ..., data.amount ...)?;
``` [4](#0-3) 

After the transfer/mint, the nonce is permanently consumed via `UsedNonces::use_nonce`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An attacker who observes a pending `finalize_transfer` transaction (the signed `FinalizeTransferPayload` is public once broadcast) can front-run it with the same valid signed payload but a different `mint`:

- **Bridged-token path**: any mint whose `mint_authority` is the bridge `authority` PDA qualifies (any token previously deployed through the bridge). The attacker mints `amount` of a worthless bridged token to the recipient.
- **Native-token path**: any mint that has a vault PDA qualifies. The attacker drains `amount` of a different native token from its vault to the recipient.

In both cases the destination nonce is marked used. The NEAR bridge has already locked the user's funds and will never re-issue the same nonce. The user's original tokens are permanently frozen on NEAR with no recovery path.

**Matched allowed impact**: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

The signed `FinalizeTransferPayload` is submitted as public instruction data by a relayer. Any on-chain observer can extract it from the mempool and resubmit it with a substituted `mint` in the same block (front-run). No privileged access, leaked key, or colluding party is required. The attacker only needs to know which mints have vaults or carry the bridge authority — both are discoverable on-chain. The attack is straightforward and repeatable for every cross-chain transfer to Solana.

---

### Recommendation

Add the token/mint address to `FinalizeTransferPayload` so the NEAR bridge MPC signature commits to it:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub token: Pubkey,   // add this field
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
```

Then enforce the association in the `FinalizeTransfer` accounts struct:

```rust
#[account(
    mut,
    mint::token_program = token_program,
    constraint = mint.key() == data.payload.token @ ErrorCode::InvalidMint,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

Apply the same fix to `FinalizeTransferSol`. Ensure the NEAR bridge serializes the token address into the payload before signing, and update `serialize_for_near` accordingly.

---

### Proof of Concept

1. Alice calls `init_transfer` on NEAR, locking 1 000 USDC. NEAR bridge signs `FinalizeTransferPayload { destination_nonce: 42, transfer_id: {NEAR, 7}, amount: 1000, fee_recipient: None }`.
2. Relayer broadcasts `finalize_transfer` with `mint = USDC_bridged_mint`.
3. Attacker observes the mempool, extracts the identical signed payload, and submits `finalize_transfer` with `mint = WORTHLESS_bridged_mint` (any mint whose `mint_authority == authority PDA`) at higher priority.
4. Attacker's transaction lands first:
   - `UsedNonces::use_nonce(42, ...)` succeeds — nonce 42 is now consumed.
   - `mint_to(... mint: WORTHLESS_bridged_mint, amount: 1000 ...)` — 1 000 worthless tokens minted to Alice.
5. Relayer's transaction fails: `NonceAlreadyUsed`.
6. Alice's 1 000 USDC remain locked on NEAR forever; no re-finalization is possible because nonce 42 cannot be reused. [7](#0-6) [8](#0-7)
