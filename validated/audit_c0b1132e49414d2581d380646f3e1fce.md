### Title
EIP-712 Legacy Signature Does Not Bind `timeout_height` ŌĆö Timed-Out Tx Can Be Replayed by Stripping the Field - (`ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

---

### Summary

The legacy EIP-712 signing path includes `timeout_height` in the Amino JSON payload (`txBytes`) passed to `LegacyWrapTxToTypedData`, but the EIP-712 typed data schema for the `"Tx"` type explicitly omits `timeout_height`. Under EIP-712, only fields declared in the type schema are hashed. Any field present in the message data but absent from the schema is silently ignored. This means the `FeePayerSig` does not commit to `timeout_height`, and an attacker can strip or alter that field on a signed tx without invalidating the signature.

---

### Finding Description

**Step 1 ŌĆö `timeout_height` enters `txBytes`**

In `VerifySignature`, `StdSignBytes` is called with `tx.GetTimeoutHeight()`:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ŌåÉ included in JSON payload
    ...
)
``` [1](#0-0) 

When `timeout_height > 0`, the Cosmos SDK `StdSignDoc` serialises it as `"timeout_height": "N"` (the field is `omitempty`, so it is absent when zero).

**Step 2 ŌĆö `timeout_height` is absent from the EIP-712 type schema**

`extractMsgTypes` in the legacy path hard-codes the `"Tx"` type without `timeout_height`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
``` [2](#0-1) 

The same omission exists in the non-legacy `createEIP712Types`: [3](#0-2) 

**Step 3 ŌĆö EIP-712 hashing ignores undeclared fields**

`LegacyWrapTxToTypedData` JSON-unmarshals `txBytes` into `txData` (so `txData["timeout_height"]` is present when non-zero), then passes both `txData` and the schema to `apitypes.TypedDataAndHash`. Per the EIP-712 spec, `hashStruct` encodes only fields declared in the type schema; extra fields in the message object are silently dropped. Therefore `sigHash` is identical whether `timeout_height` is `0` or any value `N`. [4](#0-3) 

**Step 4 ŌĆö The legacy handler is still wired into the ante chain**

`newLegacyCosmosAnteHandlerEip712` (marked deprecated but still present) installs `LegacyEip712SigVerificationDecorator` as the signature-verification step for txs carrying `ExtensionOptionsWeb3Tx`: [5](#0-4) 

`authante.NewTxTimeoutHeightDecorator()` (line 46) does enforce the timeout at the chain level, but it reads `timeout_height` from the tx body ŌĆö not from the signed hash. Because the signature does not bind `timeout_height`, an attacker can change the field in the tx body without breaking the signature check. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a broadcast (but unconfirmed) legacy EIP-712 tx that carries `timeout_height=N` can:

1. Copy the tx.
2. Set `timeout_height=0` (or any value > current block height).
3. Resubmit with the original `FeePayerSig` unchanged.

`VerifySignature` will accept the modified tx because the sigHash is identical. The `TxTimeoutHeightDecorator` will no longer reject it (timeout removed). The tx executes after the user's intended expiry window, constituting a replay of a timed-out transaction ŌĆö a direct EIP-712 authorization bypass.

This maps to the allowed High impact: *"EIP-712 authorization bypass enabling replay, forged execution, or unauthorized account/code mutation."*

---

### Likelihood Explanation

- The legacy EIP-712 path is still reachable via `ExtensionOptionsWeb3Tx` txs.
- Broadcast txs are publicly visible in the mempool.
- The modification requires only changing one JSON field; no cryptographic material is needed.
- The attacker needs no special privileges.

---

### Recommendation

Re-add `timeout_height` to the `"Tx"` type schema in both `extractMsgTypes` (legacy) and `createEIP712Types` (non-legacy), and ensure `StdSignBytes` always serialises it (even when zero, to avoid omitempty ambiguity). Alternatively, reject any legacy EIP-712 tx where `tx.GetTimeoutHeight() != 0` until the schema is fixed, consistent with how `legacyDecodeProtobufSignDoc` already rejects non-zero `TimeoutHeight`: [7](#0-6) 

---

### Proof of Concept

```
1. Construct a legacy EIP-712 Cosmos tx T1:
     timeout_height = 100, sequence = S, memo = "x", msgs = [MsgSend ...]
   Sign it ŌåÆ FeePayerSig_A (covers sigHash_A, which does NOT include timeout_height).

2. Wait until block height > 100.
   Submit T1 ŌåÆ rejected by TxTimeoutHeightDecorator (expected).

3. Construct T2 identical to T1 except timeout_height = 0.
   Attach FeePayerSig_A unchanged.

4. Submit T2.
   - VerifySignature recomputes sigHash from txBytes where timeout_height is absent
     (omitempty, value 0) ŌåÆ sigHash_B == sigHash_A.
   - Signature check passes.
   - TxTimeoutHeightDecorator: timeout_height == 0 ŌåÆ no expiry ŌåÆ passes.
   - Tx executes.

Assert: both T1 and T2 pass VerifySignature with the same FeePayerSig_A,
violating the invariant that the signature must bind all fields affecting execution validity.
```
