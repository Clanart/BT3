Audit Report

## Title
Trapped ETH in `MetricOmmPoolLiquidityAdder` is silently consumed to subsidize a subsequent user's WETH liquidity deposit — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function's WETH branch reads `address(this).balance` — the entire contract ETH balance — rather than only ETH the current caller sent. Any ETH left in the contract by a prior user (e.g., via `multicall{value: X}` without a trailing `refundETH()`) is silently wrapped and forwarded to the pool on behalf of the next user who adds WETH liquidity, permanently destroying the prior user's ETH with no recovery path.

## Finding Description
**Step 1 — ETH accumulates.** The `receive()` guard at [1](#0-0)  only blocks bare ETH transfers (no calldata). All payable entry points — `multicall` [2](#0-1) , `addLiquidityExactShares` [3](#0-2) , `addLiquidityWeighted` [4](#0-3) , `refundETH`, `unwrapWETH9`, `sweepToken` — accept `msg.value` without restriction. A user calling `multicall{value: X}([addLiquidityExactShares(...)])` for a non-WETH pool who omits `refundETH()` leaves `X` wei stranded in the contract.

**Step 2 — Trapped ETH is consumed.** In `pay()`, the WETH branch reads `address(this).balance` at [5](#0-4) . This is the full contract balance, not the current caller's contribution. If `nativeBalance > 0`, the contract wraps and forwards the trapped ETH to the pool, then pulls only `value - nativeBalance` from the actual payer via `safeTransferFrom`. The prior user's ETH is gone; the current user receives a discount equal to `nativeBalance`.

**Step 3 — Payer is always the current outer caller.** `_setPayContext` stores `msg.sender` as payer [6](#0-5) , and the callback at [7](#0-6)  calls `pay(token, payer, pool, amount)` with no mechanism to restrict `address(this).balance` to only ETH the current payer sent.

## Impact Explanation
Direct, irreversible loss of user principal (ETH). The prior user permanently loses their stranded ETH with no recovery path. The attacker (user B) needs only to observe the publicly readable contract ETH balance and submit a normal, permissionless WETH liquidity deposit. This meets the Sherlock threshold for direct loss of user funds.

## Likelihood Explanation
The multicall + omitted `refundETH()` pattern is a well-documented footgun in Uniswap-style routers; front-ends, scripts, and integrators routinely omit the refund step. The attacker's action is a standard liquidity deposit requiring no special role, privilege, or setup. The contract's ETH balance is publicly visible on-chain, making the attack trivially detectable and repeatable by any observer.

## Recommendation
Restrict the WETH branch to only ETH the current caller explicitly sent in the current transaction. Track `msg.value` at the outer entry point and pass it through to `pay()` as the maximum native contribution, using only that amount rather than `address(this).balance`. Alternatively, enforce `address(this).balance == msg.value` at the start of each non-multicall entry point, or automatically refund any remaining ETH balance to the payer after each `pay()` call.
