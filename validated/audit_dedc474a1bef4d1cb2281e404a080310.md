### Title
Missing `from` Account Ownership Constraint Allows Delegate to Bridge Victim's Tokens - (File: `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

### Summary

The Solana `InitTransfer` instruction does not enforce that the `from` token account is owned by the `user` signer. Because the SPL token `transfer_checked` and `burn` CPIs accept a delegate as the authority, any attacker who holds a token delegation over a victim's account can call `init_transfer` with the victim's token account as `from` and bridge the victim's tokens to an attacker-controlled NEAR address.

### Finding Description

In `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`, the `InitTransfer` accounts struct declares the `from` field with only mint and token-program constraints:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint. The `user` field is only required to be a `Signer` and a system-program-owned account:

```rust
#[account(
    mut,
    owner = common.system_program.key(),
)]
pub user: Signer<'info>,
``` [2](#0-1) 

In `process()`, the native-token path calls `transfer_checked` with `authority: self.user.to_account_info()`:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),
            authority: self.user.to_account_info(),
            mint: self.mint.to_account_info(),
        },
    ),
    ...
)?;
``` [3](#0-2) 

The bridged-token path calls `burn` with the same pattern:

```rust
burn(
    CpiContext::new(
        self.token_program.to_account_info(),
        Burn {
            mint: self.mint.to_account_info(),
            from: self.from.to_account_info(),
            authority: self.user.to_account_info(),
        },
    ),
    ...
)?;
``` [4](#0-3) 

The SPL Token program accepts `authority` as either the **owner** or an approved **delegate** of the `from` account. Because Anchor does not enforce `token::authority = user` on `from`, the program never verifies that `user` owns `from`. Any signer who holds a token delegation over a victim's account can pass the victim's token account as `from` and the SPL token CPI will succeed.

The Wormhole message is then posted with `self.user.key()` as the sender:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
``` [5](#0-4) 

The attacker controls the `recipient` field inside `InitTransferPayload`, so the bridged tokens arrive at the attacker's NEAR address.

### Impact Explanation

**Critical — Direct theft of native or bridged assets on Solana.**

An attacker who holds a token delegation over a victim's SPL token account can call `init_transfer` with:
- `from` = victim's token account
- `user` = attacker (signer, delegate of victim's account)
- `recipient` = attacker's NEAR address (inside the payload)

The SPL token program accepts the transfer because `user` is a valid delegate. The victim's tokens are locked in the bridge vault (native) or burned (bridged), and the NEAR side mints/releases the equivalent amount to the attacker's address. The victim loses their tokens with no recourse.

### Likelihood Explanation

Token delegations on Solana are used by DEXes, lending protocols, and other DeFi programs. A victim who has approved any program or account as a delegate on their token account — including approvals made for entirely unrelated purposes — is exposed. The attacker only needs to be the delegate, sign the transaction, and supply the victim's token account as `from`. No privileged access, no oracle manipulation, and no cross-chain coordination is required beyond the delegation already granted.

### Recommendation

Add `token::authority = user` to the `from` account constraint in `InitTransfer`:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,   // <-- add this
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This enforces at the Anchor constraint level that `user` must be the **owner** of `from`, not merely a delegate, eliminating the delegation-based attack path.

### Proof of Concept

1. Alice holds 1,000 USDC in `alice_ata` on Solana.
2. Alice previously approved `attacker_pubkey` as a delegate on `alice_ata` for 1,000 USDC (e.g., via a DEX interaction).
3. Attacker constructs an `init_transfer` instruction:
   - `from` = `alice_ata`
   - `user` = `attacker_pubkey` (signs the transaction)
   - `payload.recipient` = `"near:attacker.near"`
   - `payload.amount` = 1,000 USDC
4. Anchor validates: `from.mint == mint` ✓, `from.token_program == token_program` ✓ — no ownership check.
5. `transfer_checked` CPI: SPL token checks `alice_ata.delegate == attacker_pubkey` and `alice_ata.delegated_amount >= 1000` ✓ — transfer succeeds.
6. Wormhole message is posted; NEAR bridge releases 1,000 USDC equivalent to `attacker.near`.
7. Alice's 1,000 USDC are gone.
