Audit Report

## Title
Excess ETH from payable swap/liquidity entry points is permanently stranded and claimable by any caller via permissionless `refundETH()` — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`exactOutputSingle`, `exactOutput`, `addLiquidityExactShares`, and `addLiquidityWeighted` are all `payable` and accept native ETH for WETH-leg payments. The internal `pay()` function wraps only the exact amount required by the pool, leaving any surplus ETH in the contract. `refundETH()` is unrestricted and transfers the full contract ETH balance to `msg.sender`, not to the original payer, so any address can race to claim the stranded ETH in a subsequent transaction.

## Finding Description
When `token == WETH` and `address(this).balance >= value`, `pay()` deposits exactly `value` wei into WETH and transfers it to the pool, leaving any surplus untouched:

```solidity
// PeripheryPayments.sol L74-77
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

For exact-output swaps, the pool determines the actual `amountIn` at execution time; it will be ≤ `amountInMaximum`. The gap `amountInMaximum − amountIn` remains in the contract after `exactOutputSingle` (L130–147) and `exactOutput` (L154–188) return, with no automatic refund. For `addLiquidityWeighted`, the probe-then-scale flow makes the final ETH consumption unpredictable at call time, widening the gap further.

The refund helper is fully permissionless:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // sends to caller, not original payer
    }
}
```

There is no binding between the original ETH sender and the refund recipient. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only applies to plain ETH transfers with no calldata; it does not prevent ETH from being deposited via the `payable` function selectors. Any user who calls a payable entry point directly (not via `multicall` with a trailing `refundETH()` leg) leaves their surplus ETH claimable by any address.

## Impact Explanation
Direct loss of user principal. A user who sends 1 ETH for an exact-output swap that costs 0.8 ETH loses 0.2 ETH to any address that calls `refundETH()` before the user can. The loss is proportional to the over-send and is bounded only by `amountInMaximum − actual amountIn`. This satisfies the contest's "Critical/High/Medium direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The Uniswap v3-style multicall-plus-`refundETH` pattern is non-obvious to integrators and end users. Any direct call to a payable swap or liquidity function with a conservative (over-estimated) ETH value — a common defensive pattern — silently leaves funds at risk. MEV bots routinely monitor for stranded ETH in known router contracts and can extract it atomically in the next block.

## Recommendation
Add an automatic ETH refund at the end of each payable entry point:

```solidity
function exactOutputSingle(ExactOutputSingleParams calldata params)
    external payable returns (uint256 amountIn)
{
    // ... existing logic ...
    _clearExpectedCallbackPool();

    uint256 leftover = address(this).balance;
    if (leftover > 0) _transferETH(msg.sender, leftover);
}
```

Apply the same tail-refund to `exactOutput`, `addLiquidityExactShares`, and `addLiquidityWeighted`. Alternatively, for exact-input single-hop swaps where `amountIn` is caller-specified, enforce `msg.value == amountIn` when `tokenIn == WETH` to eliminate the surplus entirely.

## Proof of Concept
1. Pool is priced such that swapping to receive 1 000 token1 costs 800 wei of WETH.
2. User calls `router.exactOutputSingle{value: 1000}(params)` where `params.amountInMaximum = 1000` and `params.tokenIn = WETH`.
3. Inside the swap callback, `pay()` wraps 800 wei and transfers it to the pool; 200 wei remains in the router.
4. `exactOutputSingle` returns without calling `refundETH()` (direct invocation, not multicall).
5. Attacker calls `router.refundETH()` in the next block; receives 200 wei.
6. User's net ETH loss: 200 wei above the swap cost, with no recourse. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
