### Title
Missing Declared Class Validation in `execute_replace_class` Allows Arbitrary Class Hash State Bypass - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary `class_hash` value from the caller without verifying that the hash corresponds to a previously declared class in the contract class tree. This is a direct analog to the Linea Token Bridge `setCustomContract` bug: just as that function allowed setting arbitrary status codes (including sentinel values) without confirmation, `execute_replace_class` allows setting an arbitrary class hash — including `UNINITIALIZED_CLASS_HASH` (0) or any undeclared hash — without confirmation that the target class exists. The OS itself acknowledges this gap with a `TODO` comment at the exact vulnerable line.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the caller-supplied `class_hash` from the syscall request and immediately writes it into `contract_state_changes` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The same flaw exists in the deprecated path: [2](#0-1) 

The OS maintains two separate trees: `contract_state_changes` (maps contract address → `StateEntry` containing `class_hash`) and `contract_class_changes` (maps `class_hash` → `compiled_class_hash`). The `execute_replace_class` handler updates only the former, with no cross-check against the latter. There is no Cairo-level assertion that `class_hash` exists as a key in `contract_class_changes`.

The `UNINITIALIZED_CLASS_HASH` sentinel is defined as `0`: [3](#0-2) 

And `get_contract_state_hash` treats `class_hash == 0` with `storage_root == 0` and `nonce == 0` as an empty/uninitialized contract: [4](#0-3) 

An attacker can supply any of:
- `class_hash = 0` (`UNINITIALIZED_CLASS_HASH`): reverts the contract to an uninitialized state in the commitment tree.
- `class_hash = <any undeclared felt>`: the contract's state entry now references a class that does not exist in the class tree, making the contract permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to an undeclared value (or `UNINITIALIZED_CLASS_HASH`), every subsequent transaction targeting that contract will fail at class resolution time, because the OS cannot find a compiled class for the stored hash. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible. The state commitment is updated with the invalid class hash and finalized on-chain; there is no recovery path.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, publicly documented StarkNet syscall callable by any deployed contract. An attacker can:

1. Deploy a contract that holds user funds (e.g., a token vault or multi-sig).
2. After attracting deposits, call `replace_class` with `class_hash = 1` (or any undeclared felt).
3. The OS accepts the state transition without validation.
4. All deposited funds are permanently frozen.

No privileged role, leaked key, or operator cooperation is required. The entry path is fully unprivileged: any contract deployer or transaction sender can trigger this.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, the OS must verify that the supplied `class_hash` exists as a key in `contract_class_changes` (i.e., has been previously declared). Specifically:

- Assert `class_hash != UNINITIALIZED_CLASS_HASH` (reject zero).
- Assert that `class_hash` maps to a non-zero `compiled_class_hash` in the class tree (i.e., the class was declared in a prior `declare` transaction).

The existing `TODO` comment at line 898 of `syscall_impls.cairo` already identifies this gap; it must be resolved before deployment. [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` with a legitimate class hash `C_valid`. Users deposit funds.
2. `VaultContract` executes a transaction that calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared felt).
3. `execute_replace_class` in `syscall_impls.cairo` reads `request.class_hash = 0xdeadbeef`, skips any validation (the `TODO` is unimplemented), and calls `dict_update` on `contract_state_changes` setting `VaultContract`'s `class_hash` to `0xdeadbeef`.
4. The block is proven. `compute_contract_state_commitment` hashes the new `StateEntry` with `class_hash=0xdeadbeef` and commits it to the global state root.
5. Any subsequent transaction targeting `VaultContract` fails: the OS looks up `0xdeadbeef` in the compiled class facts and finds nothing. All user funds are permanently frozen. [6](#0-5) [7](#0-6)
