### Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens Up to Max Caps - (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address in `addLiquidityExactShares` and `addLiquidityWeighted` without verifying it against the `MetricOmmPoolFactory` registry. A malicious pool contract can exploit the transient callback context to pull up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 directly from the victim's wallet.

---

### Finding Description

The `MetricOmmPoolLiquidityAdder` stores a pay-context in transient storage keyed to the caller-supplied `pool` address, then calls `pool.addLiquidity(...)`. The callback `metricOmmModifyLiquidityCallback` validates only that `msg.sender == expectedPool` (the same address stored in transient storage) and that the requested amounts do not exceed the user-supplied caps. It does **not** verify that `pool` is a legitimate pool registered in `MetricOmmPoolFactory`.

The code even acknowledges this explicitly in its NatDoc:

> *"The caller is responsible for supplying a legitimate pool address and other non-malicious parameters. This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."*

This self-acknowledged gap is the root cause. A malicious pool contract can:

1. Be passed as the `pool` argument by a tricked user.
2. Inside its `addLiquidity` implementation, immediately call back `metricOmmModifyLiquidityCallback(maxAmount0, maxAmount1, abi.encode(KIND_PAY))`.
3. The callback's two guards both pass:
   - `msg.sender != expectedPool` → `MaliciousPool != MaliciousPool` → **false** (passes).
   - `amount0Delta > max0 || amount1Delta > max1` → `maxAmount0 > maxAmount0` → **false** (passes).
4. `pay(token0, payer, msg.sender, amount0Delta)` and `pay(token1, payer, msg.sender, amount1Delta)` transfer tokens from the victim to the malicious pool.

The attack path is identical in structure to the external M-07 report: a factory/router trusts an unvalidated address as a legitimate counterparty, and that address redirects token flows to the attacker.

---

### Impact Explanation

**Direct loss of user principal.** Any user who has approved `MetricOmmPoolLiquidityAdder` to spend their tokens and is tricked into calling `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address will lose up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 in a single transaction. The loss is bounded only by the user-supplied caps and their token approval amount.

---

### Likelihood Explanation

**Medium.** The attack requires a user to be directed to a malicious pool address — achievable via a compromised front-end, phishing UI, or a malicious integration that presents a fake pool as legitimate. Since `MetricOmmPoolFactory.isPool()` is a public view function that returns `true` only for factory-deployed pools, off-chain tooling could check legitimacy, but the on-chain contract itself provides no such guard. Users relying solely on the contract's own validation are fully exposed.

---

### Recommendation

Validate the `pool` argument against the factory registry before storing the pay-context. Add a factory address to the constructor and check `IMetricOmmPoolFactory(factory).isPool(pool)` at the top of `_addLiquidity`:

```solidity
// In constructor:
address public immutable factory;
constructor(address weth, address _factory) PeripheryPayments(weth) {
    factory = _factory;
}

// In _addLiquidity:
function _addLiquidity(...) internal returns (...) {
    if (!IMetricOmmPoolFactory(factory).isPool(pool)) revert InvalidPool(pool);
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This mirrors the fix recommended in M-07: use the factory's registry (`deployedVaults` / `isPool`) to reject unrecognized addresses before any token-pull authority is granted.

### Proof of Concept

```solidity
contract MaliciousPool {
    address adder;
    constructor(address _adder) { adder = _adder; }

    // Implements IMetricOmmPoolActions.addLiquidity
    function addLiquidity(
        address, uint80, LiquidityDelta calldata,
        bytes calldata, bytes calldata
    ) external returns (uint256, uint256) {
        // Callback with KIND_PAY and full max amounts
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            MAX_AMOUNT_TOKEN0,
            MAX_AMOUNT_TOKEN1,
            abi.encode(uint8(1)) // KIND_PAY = 1
        );
        return (MAX_AMOUNT_TOKEN0, MAX_AMOUNT_TOKEN1);
    }
}

// Attack:
// 1. Victim approves MetricOmmPoolLiquidityAdder for token0 and token1.
// 2. Victim is tricked into calling:
liquidityAdder.addLiquidityExactShares(
    address(maliciousPool),  // <-- unvalidated
    victim,
    0,
    deltas,
    MAX_AMOUNT_TOKEN0,       // attacker drains this much token0
    MAX_AMOUNT_TOKEN1,       // attacker drains this much token1
    ""
);
// Result: token0 and token1 transferred from victim to MaliciousPool.
```
