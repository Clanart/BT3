### Title
Permanent Fund Freeze via `normalize_amount` Rounding to Zero in `sign_transfer` — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR-to-EVM bridge transfer with an amount (minus fee) smaller than the decimal normalization divisor, `normalize_amount` returns 0 due to integer floor division. The `sign_transfer` function then panics with `InvalidAmountToTransfer`. Because tokens are already burned/locked in `init_transfer_internal` before `sign_transfer` is ever called, and no cancel/refund path exists for this case, the user's funds are permanently frozen.

---

### Finding Description

The bridge normalizes token amounts when crossing decimal boundaries. For a token registered with `origin_decimals = 24` and `decimals = 18`, the divisor is `10^6 = 1_000_000`.

**Step 1 — `init_transfer` / `init_transfer_internal`** burns or locks the user's tokens unconditionally: [1](#0-0) 

The only fee guard at this stage is: [2](#0-1) 

This only requires `fee < amount`; it does **not** verify that `normalize_amount(amount - fee) > 0`.

**Step 2 — `sign_transfer`** later calls `normalize_amount` on the net amount: [3](#0-2) 

`normalize_amount` uses floor division: [4](#0-3) 

If `amount_without_fee < 10^diff_decimals`, the result is `0`, and the `require!` at line 482 panics. Every subsequent call to `sign_transfer` for this transfer ID will also panic — there is no alternative finalization path and no cancel/refund function for NEAR-originated transfers.

---

### Impact Explanation

User tokens are burned or locked on NEAR in `init_transfer_internal` but can never be released: `sign_transfer` always reverts, no EVM-side mint ever occurs, and no on-chain cancel mechanism exists. This is **permanent, irrecoverable loss of user funds** — matching the allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` with an amount (net of fee) below the decimal-normalization threshold. For tokens with a 6-decimal gap (e.g., a 24-decimal NEAR token bridging to an 18-decimal EVM token), any net amount below `1_000_000` base units suffices. This is a realistic dust-level amount for many tokens and requires no special privileges.

---

### Recommendation

Add a normalization check inside `init_transfer` (before tokens are burned/locked) to reject transfers whose net amount normalizes to zero:

```rust
let token_address = self.get_token_address(destination_chain, token_id.clone())
    .near_expect(BridgeError::FailedToGetTokenAddress);
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing guard in `sign_transfer` but places it before any state mutation.

---

### Proof of Concept

Assume a token registered with `origin_decimals = 24`, `decimals = 18` (divisor = `1_000_000`).

1. User calls `ft_transfer_call` transferring `amount = 500_000` (base units), `fee = 0`.
2. `init_transfer` passes the fee check (`0 < 500_000`) and calls `init_transfer_internal`.
3. `init_transfer_internal` burns `500_000` tokens from the user. [5](#0-4) 
4. Relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(500_000, decimals)` = `500_000 / 1_000_000` = **0**. [6](#0-5) 
6. `require!(0 > 0, ...)` panics — `sign_transfer` reverts. [7](#0-6) 
7. No EVM mint occurs. No cancel path exists. The `500_000` tokens are permanently burned.
