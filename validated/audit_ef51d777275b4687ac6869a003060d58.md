### Title
Unchecked u128 Overflow in `denormalize_amount` Permanently Freezes Cross-Chain Transfers for Tokens with Large Decimal Differences - (File: near/omni-bridge/src/lib.rs)

### Summary

`denormalize_amount` in the NEAR bridge contract performs `amount * (10_u128.pow(diff_decimals))` without any overflow guard. When a token is registered via `bind_token` with a large `origin_decimals` (e.g. 255) and a normalized `decimals` (capped at 18 by EVM/Starknet/Solana), `diff_decimals` reaches 237. Since `10^39 > u128::MAX ≈ 3.4 × 10^38`, the exponentiation itself overflows, causing every subsequent `fin_transfer_callback` for that token to panic and permanently lock user funds.

### Finding Description

`denormalize_amount` is a private helper called on every inbound cross-chain transfer finalization:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

Both `decimals` and `origin_decimals` are `u8` (0–255). No upper-bound check is applied to `diff_decimals` before the exponentiation or the multiplication.

The `Decimals` struct stores these two fields independently:

```rust
pub struct Decimals {
    pub decimals: u8,
    pub origin_decimals: u8,
}
``` [2](#0-1) 

They are written by `add_token`, which is called from `bind_token_callback` with the values taken directly from the on-chain `DeployTokenMessage`:

```rust
self.add_token(
    &deploy_token.token,
    &deploy_token.token_address,
    deploy_token.decimals,
    deploy_token.origin_decimals,
);
``` [3](#0-2) 

The EVM bridge normalizes `decimals` to `min(18, origin)` but passes the raw `metadata.decimals` as `originDecimals`:

```solidity
uint8 decimals = _normalizeDecimals(metadata.decimals);
// ...
emit BridgeTypes.DeployToken(bridgeTokenProxy, ..., decimals, metadata.decimals);
``` [4](#0-3) 

So after `bind_token` for a NEAR token whose `origin_decimals = 255`, NEAR stores `Decimals { decimals: 18, origin_decimals: 255 }`. Every subsequent call to `fin_transfer_callback` for that token invokes:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
fee: Self::denormalize_fee(&init_transfer.fee, decimals),
``` [5](#0-4) 

`diff_decimals = 255 − 18 = 237`. `10_u128.pow(237)` overflows `u128` (max ≈ 3.4 × 10^38; 10^39 already exceeds it). With NEAR's standard build (overflow checks enabled in WASM), this panics and the callback reverts. The same overflow is triggered in `fast_fin_transfer` and `claim_fee_callback` via the same helpers. [6](#0-5) [7](#0-6) 

### Impact Explanation

Every user who sends this token from EVM (or any chain) to NEAR will have their funds permanently locked in the origin-chain bridge contract. The `fin_transfer_callback` will always panic, so the transfer can never be finalized on NEAR. Because the `Decimals` entry is immutable after `bind_token`, there is no recovery path without a DAO-level contract upgrade. This matches **Critical – permanent freezing / irrecoverable lock of user funds**.

### Likelihood Explanation

Any unprivileged user can:
1. Deploy a NEAR NEP-141 token with `decimals = 255` (no permission required).
2. Call `log_metadata` on the NEAR bridge to initiate cross-chain token registration.
3. The MPC signs the metadata payload containing `decimals = 255`.
4. A relayer submits the signed payload to EVM `deployToken`; EVM normalizes to `decimals = 18`, `originDecimals = 255`, and emits `DeployToken`.
5. A relayer submits the EVM proof to NEAR `bind_token`; NEAR stores `Decimals { decimals: 18, origin_decimals: 255 }`.

From this point, any user who bridges this token from EVM to NEAR triggers the overflow. The attacker does not need any privileged role; the entire setup is permissionless.

### Recommendation

Add an explicit overflow guard in `denormalize_amount`:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    // 10^38 < u128::MAX; 10^39 overflows. Reject any diff >= 39.
    require!(diff_decimals < 39, "ERR_DECIMAL_DIFF_TOO_LARGE");
    amount.checked_mul(10_u128.pow(diff_decimals))
        .near_expect(BridgeError::AmountOverflow)
}
```

Additionally, enforce `origin_decimals >= decimals` and `origin_decimals - decimals < 39` inside `add_token` at registration time, so the invariant is guaranteed before any transfer is ever attempted.

### Proof of Concept

1. Deploy a NEAR NEP-141 token with `ft_metadata()` returning `decimals: 255`.
2. Call `log_metadata` on the NEAR bridge for this token.
3. After MPC signing and EVM `deployToken`, the EVM emits `DeployToken(decimals=18, originDecimals=255)`.
4. Submit the EVM proof to NEAR `bind_token`. NEAR now stores `Decimals { decimals: 18, origin_decimals: 255 }`.
5. Send any nonzero amount of this token from EVM to NEAR (call EVM `initTransfer`).
6. Submit the EVM proof to NEAR `fin_transfer`. `fin_transfer_callback` calls `denormalize_amount(amount, Decimals { decimals: 18, origin_decimals: 255 })`.
7. `diff_decimals = 237`; `10_u128.pow(237)` panics with integer overflow. The callback reverts. The user's EVM-side tokens are burned/locked with no NEAR-side release — funds are permanently frozen.
