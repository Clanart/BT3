### Title
Missing Class Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash corresponds to a declared, existing contract class before committing the state transition. This is an exact structural analog to the external report's vulnerability class: a state-modifying function that applies a transition without validating the precondition that makes the new state valid. Any contract can call `replace_class` with an arbitrary, non-existent class hash, permanently rendering itself uncallable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the replacement class hash directly from the syscall request and writes it into the contract's `StateEntry` without checking whether that class hash has ever been declared:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The developer-acknowledged TODO comment at line 898 explicitly states the missing check: *"Check that there is a declared contract class with the given hash."*

The function updates `contract_state_changes` with the new `class_hash` without consulting `contract_class_changes` (the dictionary mapping class hash → compiled class hash) or any historical declared-class state. There is no `assert` or `dict_read` on `contract_class_changes` for the replacement hash.

Compare this to `deploy_contract`, which enforces `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` and `state_entry.nonce = 0` as preconditions before writing a new state entry — a pattern of validating the current state before transitioning. `execute_replace_class` has no equivalent postcondition check on the new state.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

If a contract calls `replace_class` with a class hash that does not correspond to any declared class (e.g., an arbitrary felt like `0xdeadbeef`), the OS will commit this state transition into the proven block output. After the proof is verified on L1:

- The contract's on-chain `class_hash` is set to the non-existent value.
- All subsequent calls to the contract will fail at class resolution time, because no compiled class exists for that hash.
- Any ERC-20 token balances held by the contract (i.e., balances recorded in the fee token contract's storage under the contract's address) become permanently inaccessible — the contract can no longer execute a `transfer` call to move them out.
- The freeze is irreversible: there is no upgrade path once the class hash is set to a non-existent value, because the contract itself cannot be called to invoke another `replace_class`.

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

1. Attacker deploys a contract (e.g., a fake vault or yield aggregator).
2. Victims deposit ERC-20 tokens into the contract (the balances are recorded in the fee token contract's storage under the vault's address).
3. Attacker calls `replace_class` with an arbitrary non-existent class hash from within the contract.
4. The OS processes this syscall, writes the invalid class hash into `contract_state_changes`, and includes it in the proven block.
5. After L1 verification, the contract is permanently bricked and all deposited funds are frozen.

No privileged role, leaked key, or external dependency is required. The attacker controls the contract code and can trigger this at any time.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the replacement class hash exists as a declared class. This requires a lookup in `contract_class_changes` (for classes declared in the current block) and/or a read from the historical class state. Concretely, add a check analogous to:

```cairo
// Verify the new class hash is a declared class.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

If the class was declared in a prior block (not in `contract_class_changes` for the current block), the OS must also support reading from the committed class state, similar to how `contract_state_changes` reads prior storage via hints. The TODO comment at line 898 already acknowledges this gap.
