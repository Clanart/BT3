### Title
`is_cairo_1_class_declared_at` and `is_class_declared_at` skip sync-state guard, returning authoritative class-declaration answers for unsynced blocks — (`File: crates/apollo_state_sync/src/lib.rs`)

### Summary

`StateSync::is_cairo_1_class_declared_at` and `StateSync::is_class_declared_at` answer class-declaration queries for any caller-supplied `block_number` without first verifying that the node has actually synced to that block. Every sibling state-query function (`get_storage_at`, `get_nonce_at`, `get_class_hash_at`) calls `verify_synced_up_to` and returns `BlockNotFound` for unsynced blocks. The two class-declaration functions omit this guard entirely, so they can return `true` for a block the node has never seen, producing an authoritative-looking wrong value that feeds the Gateway's stateful `Declare`-transaction validation.

### Finding Description

In `crates/apollo_state_sync/src/lib.rs`, the three storage-backed state readers all begin with the same guard:

```rust
// get_storage_at (line 227)
verify_synced_up_to(&txn, block_number)?;

// get_nonce_at (line 247)
verify_synced_up_to(&txn, block_number)?;

// get_class_hash_at (line 268)
verify_synced_up_to(&txn, block_number)?;
``` [1](#0-0) [2](#0-1) [3](#0-2) 

`verify_synced_up_to` checks both the state marker and the body marker and returns `StateSyncError::BlockNotFound` if the node has not yet stored the requested block:

```rust
fn verify_synced_up_to(...) -> Result<(), StateSyncError> {
    if let Some(latest_block_number) = latest_synced_block(txn)? {
        if latest_block_number >= block_number { return Ok(()); }
    }
    Err(StateSyncError::BlockNotFound(block_number))
}
``` [4](#0-3) 

The two class-declaration functions contain **no such guard**:

```rust
// is_cairo_1_class_declared_at (lines 295-310) — no verify_synced_up_to
async fn is_cairo_1_class_declared_at(&self, block_number: BlockNumber, class_hash: ClassHash)
    -> StateSyncResult<bool>
{
    let class_definition_block_number_opt = storage_reader
        .begin_ro_txn()?
        .get_state_reader()?
        .get_class_definition_block_number(&class_hash)?;
    if let Some(class_definition_block_number) = class_definition_block_number_opt {
        Ok(class_definition_block_number <= block_number)   // ← no sync check
    } else {
        Ok(false)
    }
}
``` [5](#0-4) 

`is_class_declared_at` delegates entirely to `is_cairo_1_class_declared_at` and then repeats the same pattern for deprecated (Cairo 0) classes, also without any sync guard: [6](#0-5) 

**Concrete inconsistency:** Suppose the node is synced to block M and class C was declared at block K ≤ M.

| Query | Result |
|---|---|
| `get_storage_at(M+1000, addr, key)` | `Err(BlockNotFound(M+1000))` |
| `get_nonce_at(M+1000, addr)` | `Err(BlockNotFound(M+1000))` |
| `is_cairo_1_class_declared_at(M+1000, C)` | `Ok(true)` ← **wrong** |
| `is_class_declared_at(M+1000, C)` | `Ok(true)` ← **wrong** |

The function answers `true` for a block the node has never processed, while every other state-query function correctly refuses to answer.

### Impact Explanation

The `StateSyncClient` trait documents that `is_cairo_1_class_declared_at` and `is_class_declared_at` are authoritative answers used by the Gateway for stateful `Declare`-transaction validation: [7](#0-6) 

The Gateway's `sync_state_reader` exposes these functions directly to callers: [8](#0-7) 

An RPC caller or the Gateway itself can supply an arbitrary future `block_number`. The function returns `Ok(true)` — an authoritative-looking positive answer — for a block the node has never synced. This matches the impact category: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

A secondary impact is **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"**: if the Gateway queries `is_class_declared_at` with a future block number and receives `true`, it may incorrectly reject a `Declare` transaction as a re-declaration of an already-declared class, even though the node has no knowledge of the state at that future block.

### Likelihood Explanation

The trigger is unprivileged: any caller of the `StateSyncRequest::IsCairo1ClassDeclaredAt` or `StateSyncRequest::IsClassDeclaredAt` RPC path can supply an arbitrary `block_number`. No special role or key is required. The node only needs to have synced at least one block that declared the target class. The inconsistency is permanent until the guard is added.

### Recommendation

Add `verify_synced_up_to(&txn, block_number)?;` at the top of both `is_cairo_1_class_declared_at` and `is_class_declared_at`, exactly as done in `get_storage_at`, `get_nonce_at`, and `get_class_hash_at`. This ensures all state-query functions share the same precondition: the node must have synced to the requested block before returning any answer about it.

### Proof of Concept

1. Start a node and let it sync to block M. Ensure class C (any Cairo 1 class) was declared at some block K ≤ M.
2. Call `StateSyncRequest::GetStorageAt(M + 10_000, some_address, some_key)` → receives `Err(BlockNotFound(M+10000))`.
3. Call `StateSyncRequest::IsCairo1ClassDeclaredAt(M + 10_000, C)` → receives `Ok(true)`.
4. The node has never processed block M+10,000, yet it returns an authoritative positive class-declaration answer for that block, while refusing to answer any other state query for the same block.
