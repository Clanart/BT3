### Title
`max_l2_gas_amount` Cap Not Enforced for Declare Transactions in Gateway Admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator` defines and enforces a `max_l2_gas_amount` cap for `Invoke` and `DeployAccount` transactions, but explicitly skips this cap for `Declare` transactions. This is the direct sequencer analog of the `OmoVault` `supplyCap` bug: a limit is configured and enforced for some entry points but silently bypassed for another.

### Finding Description

In `validate_resource_bounds`, the gateway checks whether a transaction's `l2_gas.max_amount` exceeds the configured `max_l2_gas_amount` (default: 1,210,000,000). However, the check is wrapped in a branch that short-circuits for `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The `max_l2_gas_amount` field is defined in `StatelessTransactionValidatorConfig` with a production default of `1_210_000_000`: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms this bypass: a Declare transaction with `l2_gas.max_amount = 200` passes validation even when the configured limit is `100`: [3](#0-2) 

The cap check is enforced for `Invoke` and `DeployAccount` (the `test_invalid_max_l2_gas_amount` test only runs against those two types): [4](#0-3) 

### Impact Explanation

A caller can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX` (or any value exceeding `max_l2_gas_amount`). The gateway and mempool will admit it without error. The `max_l2_gas_amount` cap exists to prevent transactions from claiming more gas than the block can supply (the block's `sierra_gas` bouncer cap is 5,000,000,000) and to prevent spam. Bypassing it for Declare transactions means the gateway accepts transactions it is configured to reject, violating the admission invariant.

**Matching impact:** *High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

### Likelihood Explanation

Any unprivileged user submitting a `Declare` transaction can set `l2_gas.max_amount` to an arbitrarily large value. No special access or knowledge is required. The bypass is unconditional for all Declare transactions.

### Recommendation

Apply the same `max_l2_gas_amount` cap to `Declare` transactions in `validate_resource_bounds`. Remove the `if let RpcTransaction::Declare(_) = tx { }` short-circuit branch, or add an explicit check inside it:

```rust
// Remove the Declare exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The TODO comment at line 78 already acknowledges this gap. [5](#0-4) 

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`.
2. Submit it to the gateway with `validate_resource_bounds: true` in the config.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never returned for Declare transactions regardless of the `max_amount` value.
4. The same transaction submitted as `Invoke` or `DeployAccount` with the same `max_amount` would be rejected with `StatelessTransactionValidatorError::MaxGasAmountTooHigh`.
