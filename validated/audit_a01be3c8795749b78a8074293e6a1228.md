### Title
`set_state_finalize` and `set_state_finalize_on_height` Never Persist `StateTransitionData`, Breaking State Witness Generation After Catchup - (`File: chain/chain/src/chain_update.rs`)

---

### Summary

After state sync / shard catchup completes, the blocks applied by `set_state_finalize` and `set_state_finalize_on_height` never write a `DBCol::StateTransitionData` entry. When the node later acts as a chunk producer and calls `create_state_witness`, the `collect_state_transition_data` walk-back reads those missing entries and hard-fails. The chunk producer cannot produce a valid state witness for any chunk whose implicit-transition range overlaps the catchup window, permanently breaking its ability to produce chunks for the newly-tracked shard.

---

### Finding Description

**Normal block-processing path** (`process_apply_chunk_result`, `chain_update.rs` lines 113–189):

For both `NewChunk` and `OldChunk` results, the code explicitly extracts `apply_result.proof` and calls `save_state_transition_data` before handing the result to `apply_chunk_postprocessing`:

```rust
if should_save_state_transition_data {
    let apply_result = &mut new_chunk_result.apply_result;
    self.chain_store_update.save_state_transition_data(
        *block_hash, shard_id,
        apply_result.proof.take(),          // proof extracted here
        apply_result.applied_receipts_hash,
        mem::take(&mut apply_result.contract_updates),
    );
}
``` [1](#0-0) 

The same pattern is repeated for `OldChunk`: [2](#0-1) 

**State-sync finalization path** (`set_state_finalize`, lines 519–558):

The main chunk at `height_included` is applied and the result is moved directly into `NewChunkResult` and consumed by `apply_chunk_postprocessing`. `apply_result.proof` is **never extracted**; it is silently dropped. No `save_state_transition_data` call exists anywhere in this function. [3](#0-2) 

**Missing-chunk finalization path** (`set_state_finalize_on_height`, lines 611–651):

Each old chunk between `height_included+1` and `sync_hash−1` is applied. The function saves `trie_changes` and `chunk_extra`, but `apply_result.proof` is again silently dropped:

```rust
self.chain_store_update.save_trie_changes(*block_header.hash(), apply_result.trie_changes);
let mut new_chunk_extra = ChunkExtra::clone(&chunk_extra);
*new_chunk_extra.state_root_mut() = apply_result.new_root;
self.chain_store_update.save_chunk_extra(...);
// apply_result.proof is dropped here — StateTransitionData never written
``` [4](#0-3) 

**Witness generation reads those missing entries** (`get_state_transition`, `state_witness.rs` lines 193–236):

When the chunk producer later calls `create_state_witness` → `collect_state_transition_data`, it walks back through every block between the new chunk and the previous new chunk, calling `get_state_transition` for each implicit transition. That function does a hard lookup in `DBCol::StateTransitionData`:

```rust
let stored_chunk_state_transition_data = self
    .store()
    .get_ser(DBCol::StateTransitionData, &get_block_shard_id(block_hash, shard_id))
    .ok_or_else(|| {
        let message = format!(
            "Missing transition state proof for block {block_hash} and shard {shard_id}"
        );
        ...
        Error::Other(message)
    })?;
``` [5](#0-4) 

Because `set_state_finalize_on_height` never wrote those entries, every block in the catchup window returns `Error::Other("Missing transition state proof …")`, and `create_state_witness` fails.

The entry point that calls both finalization functions is `Chain::set_state_finalize`: [6](#0-5) 

---

### Impact Explanation

A validator that completes shard catchup (the standard path for tracking a new shard in a multi-shard epoch) will be unable to produce `ChunkStateWitness` for the first new chunk of that shard. Without a valid witness, chunk validators cannot endorse the chunk, the chunk cannot be included in a block, and the chunk producer accrues missed-chunk penalties. In a network with frequent resharding, this affects every validator on every epoch boundary where it picks up a new shard.

**Severity: High** — permanent loss of chunk-production capability for the newly-tracked shard until the node re-syncs.

---

### Likelihood Explanation

Catchup is the normal, automatic code path executed by every validator that will track a different shard in the next epoch. With dynamic resharding enabled, this occurs at every epoch boundary for a large fraction of validators. No special attacker action is required; the node triggers the bug itself.

---

### Recommendation

In `set_state_finalize`, extract `apply_result.proof` before passing the result to `apply_chunk_postprocessing`, and call `save_state_transition_data` when the node is a chunk producer for this or the next epoch — mirroring the guard already used in `process_apply_chunk_result` and `block_catch_up_postprocess`:

```rust
// In set_state_finalize, before constructing NewChunkResult:
if should_save_state_transition_data {
    self.chain_store_update.save_state_transition_data(
        *block_header.hash(),
        shard_id,
        apply_result.proof.take(),
        apply_result.applied_receipts_hash,
        mem::take(&mut apply_result.contract_updates),
    );
}
```

Apply the same fix to `set_state_finalize_on_height` for the old-chunk case, saving `apply_result.proof` before it is dropped. The `should_save_state_transition_data` flag should be threaded into both functions from `Chain::set_state_finalize`, which already has access to `self` and can call `should_produce_state_witness_for_this_or_next_epoch`. [7](#0-6) 

---

### Proof of Concept

1. Run a validator node that tracks shard 0 in epoch T.
2. At the start of epoch T, the node begins catchup for shard 1 (which it will track in epoch T+1). `set_state_finalize` applies the chunk at `height_included`; `set_state_finalize_on_height` applies old chunks up to `sync_hash`. Neither writes `DBCol::StateTransitionData`.
3. In epoch T+1, the node is assigned as chunk producer for shard 1. It calls `send_chunk_state_witness_to_chunk_validators` → `create_state_witness` → `collect_state_transition_data`.
4. `collect_state_transition_data` walks back from the new chunk's `prev_block_hash` to `prev_chunk_height_included` (which is `height_included` from the catchup). For every block in that range, `get_state_transition` looks up `DBCol::StateTransitionData` and finds nothing.
5. `create_state_witness` returns `Error::Other("Missing transition state proof for block … and shard …")`.
6. The chunk producer cannot distribute a witness; the chunk is not endorsed; the chunk producer accrues missed-chunk penalties.

    pub fn set_state_finalize(
        &mut self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<(), Error> {
        let shard_state_header = self.state_sync_adapter.get_state_header(shard_id, sync_hash)?;
        let mut height = shard_state_header.chunk_height_included();
        let mut chain_update = self.chain_update();
        let shard_uid = chain_update.set_state_finalize(shard_id, sync_hash, shard_state_header)?;
        chain_update.commit()?;

        // We restored the state on height `shard_state_header.chunk.header.height_included`.
        // Now we should build a chain up to height of `sync_hash` block.
        loop {
            height += 1;

            // Result of successful execution of set_state_finalize_on_height is bool,
            // should we commit and continue or stop.
            if chain_update.set_state_finalize_on_height(height, shard_id, sync_hash)? {
                chain_update.commit()?;

                break;
            }

        if let Some(flat_storage) = flat_storage_manager.get_flat_storage_for_shard(shard_uid) {
            let header = self.get_block_header(&sync_hash)?;
            flat_storage.update_flat_head(header.prev_hash()).unwrap();
        }

    pub fn should_produce_state_witness_for_this_or_next_epoch(
        &self,
        block_header: &BlockHeader,
    ) -> Result<bool, Error> {
        if cfg!(feature = "shadow_chunk_validation") {
            return Ok(true);
        }
        let epoch_id = block_header.epoch_id();
        // Use epoch manager because block is not in DB yet.
        let next_epoch_id =
            self.epoch_manager.get_next_epoch_id_from_prev_block(block_header.prev_hash())?;
        let validator_signer = self.validator_signer.get();

        let validator_id = if let Some(validator_signer) = &validator_signer {
            Some(validator_signer.validator_id())
        } else {
            // If the node shadows another validator, we want it to have all the transition
            // data so it can seamlessly take over as the main validator if required.
            self.shard_tracker.get_shadow_validator_id()
        };

        let Some(account_id) = validator_id else {
            return Ok(false);
        };
        Ok(self.epoch_manager.is_chunk_producer_for_epoch(epoch_id, account_id)?
            || self.epoch_manager.is_chunk_producer_for_epoch(&next_epoch_id, account_id)?)
