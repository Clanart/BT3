### Title
Recipient Blacklisting Causes Permanent Irrecoverable Lock of User Funds in `finTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The EVM `finTransfer` function (and its StarkNet counterpart `fin_transfer`) transfers tokens directly to the MPC-signed recipient address. If the recipient is blacklisted by the token contract (e.g., USDC, USDT), the transfer reverts atomically, the nonce is never consumed, and the call can never succeed. Because the MPC signature is cryptographically bound to that specific recipient and there is no cancel/refund path on the NEAR side, the user's tokens locked or burned during `initTransfer` on NEAR are permanently irrecoverable.

---

### Finding Description

**EVM `finTransfer` flow:** [1](#0-0) 

The nonce is marked used at line 287 **before** the token transfer. However, because Solidity reverts are atomic, if the subsequent `safeTransfer` call reverts (e.g., USDC blacklist check), the entire transaction rolls back — including the nonce write. The nonce is therefore never permanently consumed. [2](#0-1) 

For native (non-bridge) tokens such as USDC, the path taken is `IERC20.safeTransfer(payload.recipient, payload.amount)`. If `payload.recipient` is on the USDC blacklist, this call reverts unconditionally.

The MPC signature covers the recipient address as part of the Borsh-encoded payload: [3](#0-2) 

There is no mechanism to re-sign the payload with a different recipient. The NEAR `sign_transfer` function reads the recipient directly from the stored `TransferMessage`: [4](#0-3) 

The recipient is fixed at `initTransfer` time and cannot be changed.

**NEAR-side token lock is permanent:**

When a user initiates a NEAR→EVM transfer, tokens are locked or burned on NEAR inside `init_transfer_internal`: [5](#0-4) 

After `sign_transfer_callback` emits the signature event (and removes the pending transfer if fee is zero), the NEAR side has no record of the transfer and no cancel/refund function exists: [6](#0-5) 

There is no `cancel_transfer`, `refund_transfer`, or equivalent function anywhere in the NEAR contract. The `pending_transfers` map has no user-accessible removal path.

**StarkNet `fin_transfer` has the identical pattern:** [7](#0-6) 

Nonce is set at line 250, then the ERC-20 `transfer` is called. If it returns `false` or panics (blacklisted recipient), the `assert(success, 'ERR_TRANSFER_FAILED')` reverts the whole transaction including the nonce write. Same permanent lock results.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who initiates a NEAR→EVM (or NEAR→StarkNet) transfer of a blacklist-capable token (USDC, USDT, etc.) to a recipient address that is or becomes blacklisted will have their tokens permanently locked:

- Tokens are burned or locked on NEAR at `initTransfer` time.
- `finTransfer` on EVM/StarkNet always reverts for the blacklisted recipient.
- The nonce is never consumed, so the call can be retried indefinitely — but will always fail.
- No admin or user function exists to cancel the pending transfer and return tokens.
- The funds are irrecoverable without a contract upgrade.

---

### Likelihood Explanation

**Moderate-to-High.**

- USDC and USDT are among the most commonly bridged tokens; both implement address blacklists controlled by their issuers.
- OFAC sanctions and exchange compliance actions blacklist addresses regularly and without advance notice.
- The bridge process is multi-step and asynchronous; a recipient address can be blacklisted in the window between `initTransfer` on NEAR and `finTransfer` on EVM/StarkNet.
- No user action is required to trigger the lock — it occurs passively when the token issuer blacklists the recipient.

---

### Recommendation

Implement a pull-payment (escrow) pattern for `finTransfer`, analogous to the fix applied in the referenced Teller Protocol report:

1. Instead of transferring tokens directly to `payload.recipient`, deposit them into a per-recipient escrow mapping inside the bridge contract.
2. Expose a separate `withdraw(address token)` function that lets the recipient pull their tokens.
3. This way, `finTransfer` succeeds regardless of the recipient's blacklist status (the bridge contract itself is not blacklisted), and the nonce is permanently consumed. The recipient can claim tokens later if/when their blacklist status changes, or the protocol can define a governance-controlled recovery path.

Alternatively, catch transfer failures and store the amount for later withdrawal rather than reverting the entire `finTransfer` call.

---

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR and calls `ft_transfer_call` to the bridge with `recipient = 0xAliceEVM`, initiating a NEAR→EVM transfer. Tokens are locked on NEAR.
2. The relayer calls `sign_transfer` on NEAR; the MPC signs a payload binding `recipient = 0xAliceEVM`. The pending transfer is removed from NEAR state.
3. Before the relayer submits `finTransfer` on EVM, Circle blacklists `0xAliceEVM` (e.g., OFAC compliance).
4. The relayer calls `finTransfer(signatureData, payload)` on EVM. Execution reaches `IERC20(usdc).safeTransfer(0xAliceEVM, 10000e6)`. USDC reverts. The entire transaction reverts, including `completedTransfers[nonce] = true`.
5. Every subsequent attempt to call `finTransfer` with the same (only valid) signature reverts identically.
6. Alice's 10,000 USDC is permanently locked in the NEAR bridge contract with no recovery path.
