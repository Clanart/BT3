### Title
`deployToken` / `deploy_token` Signature Lacks Chain-ID Binding, Enabling Cross-Chain Replay of Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The NEAR MPC signature that authorizes `deployToken` / `deploy_token` across all bridge implementations (EVM, StarkNet, Solana) does **not** include any chain identifier in the signed message. The identical signature is therefore valid on every chain simultaneously. An unprivileged attacker who observes a valid `deployToken` transaction on one chain can replay the same signature on every other chain to deploy the same token there without NEAR MPC authorization for those chains.

---

### Finding Description

Every bridge implementation signs a `MetadataPayload` whose borsh encoding is:

```
PayloadType::Metadata | token_string | name_string | symbol_string | decimals
```

No chain identifier of any kind is included.

### Impact Explanation

A single NEAR MPC signature authorizing deployment of token `X` on Ethereum is byte-for-byte valid on Arbitrum, Base, BNB, Polygon, HyperEVM, StarkNet, and Solana. An attacker can:

1. Observe the signature in a public `deployToken` transaction on chain A.
2. Submit the same `(signatureData, metadata)` tuple to `deployToken` / `deploy_token` on every other chain B, C, D … before the protocol does.
3. Each call passes signature verification and registers the token in that chain's bridge mappings (`nearToEthToken`, `near_to_starknet_token`, etc.).

Consequences:
- **Unauthorized token deployment** on chains the protocol has not yet enabled for that token, bypassing the protocol's controlled rollout.
- The NEAR bridge observes `DeployToken` events from all chains and registers the attacker-triggered token addresses. Subsequent `finTransfer` calls to those chains will mint tokens against those addresses, making the unauthorized deployment permanent and operationally indistinguishable from an authorized one.
- If the attacker front-runs the legitimate deployment on a chain (before the protocol's own relayer submits it), the resulting token contract address will differ from what the protocol would have deployed (address depends on the deploying contract's nonce), causing the NEAR bridge to register a different address than intended, potentially breaking the expected token mapping for that chain.

This matches the allowed impact: **High — signature/MPC verification bypass enabling unauthorized token deployment**.

---

### Likelihood Explanation

- The signature is publicly visible on-chain the moment the first `deployToken` transaction is mined.
- No special privilege, leaked key, or colluding MPC signer is required — any observer can extract and replay it.
- The attacker only needs to call `deployToken` on the target chain with the same calldata before the protocol's relayer does.
- The OmniBridge is deployed on at least six EVM chains plus StarkNet and Solana, giving a wide replay surface.

---

### Recommendation

Include the destination chain identifier in the signed `MetadataPayload` message, mirroring the pattern already used in `finTransfer`. Concretely:

**EVM** — prepend `bytes1(omniBridgeChainId)` to the `borshEncoded` buffer in `deployToken`.

**StarkNet** — add a `chain_id: u8` parameter to `MetadataPayload::to_borsh` and append it (as done in `TransferMessagePayload::to_borsh`).

**Solana** — change `AdditionalParams` for `DeployTokenPayload` from `()` to the chain ID constant and write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the serialized buffer (as done in `FinalizeTransferPayload::serialize_for_near`).

The NEAR side must also include the destination chain in the payload it signs so that the MPC key produces chain-specific signatures.

---

### Proof of Concept

1. NEAR MPC signs `MetadataPayload { prefix: Metadata, token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 }` for Ethereum. The resulting `(signatureData, metadata)` is submitted in a public Ethereum transaction.

2. Attacker copies `signatureData` and `metadata` verbatim and calls `OmniBridge.deployToken(signatureData, metadata)` on the Arbitrum OmniBridge.

3. `ECDSA.recover(keccak256(borshEncoded), signatureData)` returns `nearBridgeDerivedAddress` — the check passes because the borsh encoding is identical (no chain field).

4. A new `BridgeToken` proxy for `usdc.near` is deployed on Arbitrum; `nearToEthToken["usdc.near"]` is set to the attacker-triggered address; a `DeployToken` event is emitted.

5. The NEAR bridge relayer picks up the event and registers the Arbitrum token address. All future `finTransfer` calls to Arbitrum for `usdc.near` will mint against this address — the unauthorized deployment is now permanent.

6. Steps 2–5 can be repeated for Base, BNB, Polygon, HyperEVM, StarkNet, and Solana using the same signature.
