Audit Report

## Title
`_fillBids` subtracts the wrong bin's `lengthE6` when stepping `walkDistE6`, producing wrong `lowerX64`/`upperX64` price bounds for every bid depth level below `curBinIdx` — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

In `_fillBids`, the downward loop reads `lenAbove` from bin `b+1` and subtracts it from `walkDistE6` to compute the lower boundary of bin `b`. The correct step is to subtract the length of bin `b` itself. When adjacent bins have different `lengthE6` values (the standard configuration), every `DepthLevel` below `curBinIdx` receives wrong `lowerX64`/`upperX64` bounds, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` for all bid depth levels. Integrators relying on `getLiquidityDepth` to derive slippage parameters for sells are induced into fund-losing trades.

## Finding Description

`walkDistE6` is initialized to `curBinDistFromProvidedPriceE6`, the lower boundary of `curBinIdx`. To step to the lower boundary of bin `b = curBinIdx - 1`, the code must subtract `len(b)`. Instead, at lines 496–498:

```solidity
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
```

`lenAbove` is the length of bin `b+1`, not bin `b`. The current bin's own `lengthE6` is read at line 500 but never used for the cursor step. The resulting `lowerX64` and `upperX64` at lines 503–506 are computed from the wrong `walkDistE6`, so `mStartX64 = upperX64` and `mEndX64 = lowerX64` passed to `_accumulateBidLevel` are both wrong, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` for every bin below `curBinIdx`.

Contrast with `_fillAsks` at line 441, which correctly advances the cursor using the current bin's own length:

```solidity
ctx.cumDistE6 += int256(uint256(lengthE6));  // lengthE6 of the bin just processed
```

No such equivalent exists in `_fillBids`; the cursor is stepped by the wrong bin's length on every iteration. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation

`getLiquidityDepth` is the primary public lens for integrators to read bid depth and derive slippage parameters for sells. Every `DepthLevel` below `curBinIdx` carries a wrong `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64`. Because the code subtracts `len(curBinIdx)` instead of `len(b)`, the reported bid prices are systematically shifted — the error magnitude is `|len(curBinIdx) - len(b)|` per step and accumulates across deeper bins, up to the full `uint16` range (65535 E6 units) per step. Integrators using these values to set `amountOutMinimum` for sells will either revert (if the inflated price is used directly) or execute at a worse price than expected (if they scale from the wrong reference), constituting a direct fund loss above Sherlock Medium thresholds. This matches the lens/quoter path impact gate: quoted amounts must match actual pool execution closely enough that integrations are not induced into fund-losing trades. [4](#0-3) [5](#0-4) 

## Likelihood Explanation

Any pool with non-uniform bin `lengthE6` values (the standard factory configuration per `BinDataLibrary`) triggers this bug on every call to `getLiquidityDepth` when `curBinIdx` is not at the lowest bin. No privileged action is required — `getLiquidityDepth` is a public `external` function callable by any address. The corrupted output is deterministic and repeatable on every invocation. [6](#0-5) 

## Recommendation

Replace the `lenAbove` read with the current bin's own `lengthE6`, which is already fetched at line 500. Move the cursor step to after the bin state read:

```solidity
// Remove lines 496-498 entirely, then after line 500:
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
walkDistE6 -= int256(uint256(lengthE6));  // subtract current bin's own length
```

This mirrors the correct pattern in `_fillAsks` at line 441. [7](#0-6) 

## Proof of Concept

1. Deploy a pool with `curBinIdx=2` (`lengthE6=100`), bin 1 (`lengthE6=200`), bin 0 (`lengthE6=300`). Set `curBinDistFromProvidedPriceE6 = 500`.
2. Call `getLiquidityDepth(pool, 3)`.
3. For `bids[1]` (bin 1): code computes `walkDistE6 = 500 - 100 = 400`; correct is `500 - 200 = 300`. `lowerX64 = price(400)` instead of `price(300)`; `upperX64 = price(400 + 200) = price(600)` instead of `price(500)`.
4. For `bids[2]` (bin 0): code computes `walkDistE6 = 400 - 200 = 200`; correct is `300 - 300 = 0`. `lowerX64 = price(200)` instead of `price(0)`; `upperX64 = price(200 + 300) = price(500)` instead of `price(300)`.
5. Assert `binAvgExecPriceX64` for bins 1 and 0 is systematically higher than the true pool bid price.
6. Show that an integrator using `bids[1].binAvgExecPriceX64` as the reference for `amountOutMinimum` on a sell executes at a worse price than expected, constituting a direct fund loss.
