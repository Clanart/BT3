### Title
`OmniBridge.finTransfer` Permanently Consumes Destination Nonce When `ICustomMinter.mint` Silently Returns Without Minting — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` irrevocably marks a destination nonce as used **before** calling `ICustomMinter.mint`. The `ICustomMinter` interface declares `mint` as a `void`-returning external function with no enforced return value or balance-change verification. If a custom minter's `mint` implementation silently returns without actually minting tokens — a legitimate design pattern analogous to the CVX operator-change shutdown — the nonce is permanently consumed, the `FinTransfer` event is emitted as if the transfer succeeded, and the recipient receives nothing. Because the nonce cannot be reused, the user's funds are irrecoverably lost.

---

### Finding Description

In `OmniBridge.finTransfer`, the destination nonce is marked used at line 287, before any token movement occurs:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287 — permanent, before mint
``` [1](#0-0) 

Signature verification follows, and then the custom-minter branch executes:

```solidity
} else if (customMinters[payload.tokenAddress] != address(0)) {
    ICustomMinter(customMinters[payload.tokenAddress]).mint(
        payload.tokenAddress,
        payload.recipient,
        payload.amount
    );
``` [2](#0-1) 

The `ICustomMinter` interface declares `mint` as returning nothing:

```solidity
interface ICustomMinter {
    function mint(address token, address to, uint128 amount) external;
    ...
}
``` [3](#0-2) 

There is no balance-before/balance-after check, no return-value assertion, and no post-call verification that `payload.amount` tokens were actually credited to `payload.recipient`. If the custom minter's `mint` silently returns (does not revert) without minting — for example because the minter has been paused, its internal operator/controller was changed, or it entered a shutdown state — the EVM transaction succeeds, `completedTransfers[nonce]` stays `true`, and the `FinTransfer` event is emitted. The recipient receives zero tokens and there is no recovery path.

This is structurally identical to the reported CVX vulnerability: the CVX `mint` function silently returns when `msg.sender != operator`, and the Vault assumed tokens were minted without checking the actual balance delta.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

- Tokens were already locked or burned on the origin chain when the `InitTransfer` was submitted.
- The destination nonce is permanently consumed (`completedTransfers[nonce] = true`), so `finTransfer` cannot be called again for the same transfer.
- The recipient receives no tokens.
- There is no admin escape hatch in `OmniBridge` to reset a consumed nonce or replay a failed mint.

The result is a complete, unrecoverable loss of the bridged asset for the affected user.

---

### Likelihood Explanation

**Low-Medium.** The condition requires:

1. A token registered with a custom minter (`customMinters[tokenAddress] != address(0)`), which is an admin-configured path used for non-standard tokens.
2. The custom minter's `mint` to silently return without minting — a realistic design pattern for minters that implement pause logic, operator/controller gating, or protocol-level shutdown (exactly the CVX pattern).

No attacker action is required to trigger the silent failure; it can arise from a legitimate state transition in the custom minter contract. Once triggered, every subsequent `finTransfer` call for any token using that minter will silently consume nonces and lose funds until the minter is fixed and the bridge is updated.

---

### Recommendation

Check the recipient's token balance before and after the `ICustomMinter.mint` call and revert if the delta does not equal `payload.amount`:

```solidity
} else if (customMinters[payload.tokenAddress] != address(0)) {
    uint256 balanceBefore = IERC20(payload.tokenAddress).balanceOf(payload.recipient);
    ICustomMinter(customMinters[payload.tokenAddress]).mint(
        payload.tokenAddress,
        payload.recipient,
        payload.amount
    );
    uint256 balanceAfter = IERC20(payload.tokenAddress).balanceOf(payload.recipient);
    require(balanceAfter - balanceBefore == payload.amount, "Mint amount mismatch");
}
```

Alternatively, move `completedTransfers[payload.destinationNonce] = true` to **after** all token-delivery logic so that a revert in the mint path does not permanently consume the nonce.

---

### Proof of Concept

1. Admin registers a token with a custom minter via `addCustomToken`. The custom minter implements a `paused` flag; when paused, `mint` returns immediately without minting.
2. A user bridges tokens from NEAR to EVM: tokens are locked/burned on NEAR, an MPC-signed `TransferMessagePayload` is produced.
3. The custom minter is paused (legitimate protocol action, not malicious admin).
4. Anyone calls `OmniBridge.finTransfer` with the valid signed payload.
5. `completedTransfers[payload.destinationNonce]` is set to `true` (line 287).
6. `ICustomMinter.mint` is called; the minter is paused, so it returns without minting.
7. `FinTransfer` event is emitted. Transaction succeeds.
8. Recipient balance: unchanged. Nonce: permanently consumed. User funds: irrecoverably lost. [4](#0-3) [5](#0-4)
