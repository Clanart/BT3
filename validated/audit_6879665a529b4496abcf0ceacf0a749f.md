### Title
Unvalidated Caller-Supplied `fee_recipient` in `sign_transfer` Enables Fee Theft and Permanent Fee Locking - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The `sign_transfer` function accepts a caller-supplied `fee_recipient: Option<AccountId>` parameter that is embedded directly into the MPC-signed `TransferMessagePayload` without any validation against the caller's identity or any stored value. A malicious trusted relayer can set `fee_recipient` to an arbitrary address to steal fees from legitimate relayers, or set it to `None` to permanently lock the fee portion of user deposits in the bridge contract.

### Finding Description

`sign_transfer` in `near/omni-bridge/src/lib.rs` is callable by any trusted relayer (anyone who stakes NEAR via `apply_for_trusted_relayer`). It accepts `fee_recipient: Option<AccountId>` as a free parameter:

```rust
pub fn sign_transfer(
    &mut self,
    transfer_id: TransferId,
    fee_recipient: Option<AccountId>,
    fee: &Option<Fee>,
) -> Promise {
```

The only validation performed is that the fee *amount* matches the stored transfer:

```rust
if let Some(fee) = &fee {
    require!(
        &transfer_message.fee == fee,
        BridgeError::InvalidFee.as_ref()
    );
}
```

There is **no validation** that `fee_recipient` equals the caller or any stored value. The parameter is passed directly into the signed payload:

```rust
let transfer_payload = TransferMessagePayload {
    ...
    fee_recipient,   // ← directly from caller, completely unvalidated
    ...
};
```

The MPC signs this payload. The resulting signature is submitted to EVM `finTransfer`, which verifies the signature and emits a `FinTransfer` event containing the attacker-controlled `feeRecipient`. On NEAR, `claim_fee_callback` enforces:

```rust
let fee_recipient = fin_transfer.fee_recipient.unwrap_or_else(|| {
    env::panic_str(BridgeError::FeeRecipientNotSetOrEmpty.to_string().as_str());
});

require!(
    fee_recipient == *predecessor_account_id,
    BridgeError::OnlyFeeRecipientCanClaim.as_ref()
);
```

**Attack path A — Fee theft**: Attacker sets `fee_recipient = attacker_address`. The MPC signs the payload, the attacker submits it to EVM, the EVM emits `FinTransfer` with `feeRecipient = attacker_address`, and the attacker calls `claim_fee` on NEAR to collect the fee that was meant for the legitimate relayer.

**Attack path B — Permanent fee lock**: Attacker sets `fee_recipient = None`. The EVM encodes this as `\x00` in the `FinTransfer` event. `claim_fee_callback` panics on `unwrap_or_else`, so no one can ever claim the fee. The fee tokens remain permanently locked in the bridge contract. Since `completedTransfers[payload.destinationNonce] = true` is set on EVM before the transfer, no other signature can be submitted for the same nonce, making the lock irrecoverable.

The transfer message is only removed from `pending_transfers`
