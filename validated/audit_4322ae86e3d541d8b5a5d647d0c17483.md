### Title
Unauthenticated BOOTSTRAP Declare Path Bypasses Signature Verification and Fee Payment — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` contains a special "BOOTSTRAP" path intended for system bootstrapping. It skips all authorization checks — signature verification (`__validate_declare__`), nonce management, and fee payment — when `sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and `max_possible_fee == 0`. There is no check that the caller is actually authorized to use this path. Any unprivileged user can craft a declare transaction satisfying these conditions and trigger the bypass.

---

### Finding Description

In `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the following branch is evaluated:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately, skipping:
- `check_and_increment_nonce` (line 779) — no nonce enforcement
- `run_validate` / `__validate_declare__` — no signature verification
- `charge_fee` (line 822) — no fee deduction [2](#0-1) 

The `sender_address` is loaded from hints via `%{ DeclareTxFields %}` and is bound to the transaction hash via `compute_declare_transaction_hash`, which includes `sender_address` as `common_fields.sender_address`. [3](#0-2) 

`'BOOTSTRAP'` is simply the felt encoding of the ASCII string — a specific but publicly known felt value. Any user can submit a declare transaction with:
- `sender_address = felt('BOOTSTRAP')`
- `nonce = 0`
- `version = 3`
- All resource bounds set to 0 (making `max_possible_fee == 0`)

The OS will process this through the BOOTSTRAP path without any ownership or authorization check. This is the direct analog of the original bug: just as `create_stop_order_ticket` sets `user_address` to the caller without verifying the caller owns the `account_id`, here the OS sets `sender_address` from the transaction without verifying the caller is authorized to use the `'BOOTSTRAP'` identity.

Additionally, because `check_and_increment_nonce` is never called in the BOOTSTRAP path, the `'BOOTSTRAP'` address nonce in `contract_state_changes` is never incremented. This means an attacker can submit an unbounded number of BOOTSTRAP declare transactions, all with `nonce = 0`, across multiple blocks. [4](#0-3) 

---

### Impact Explanation

**Permanent freezing of funds** via class hash poisoning:

The `dict_update` in the BOOTSTRAP path stores `class_hash -> compiled_class_hash` with `prev_value=0`, enforcing that a class may only be declared once. [5](#0-4) 

An attacker who knows a target class hash (computable from public Sierra source) can race to declare it with an arbitrary invalid `compiled_class_hash` (e.g., `1`). Once stored, the legitimate declarer's `dict_update` with `prev_value=0` will fail because the slot is already occupied. The class is permanently poisoned: it is declared but maps to a non-existent compiled class. Any contract deployed under that class hash will fail at execution time in `execute_entry_point`, where `find_element` searches for the `compiled_class_hash` in the compiled class facts bundle and finds nothing. [6](#0-5) 

If the poisoned class is a critical system class (fee token, account class, bridge contract), all funds held in contracts of that class are permanently frozen.

**Secondary impact — direct loss of funds** via fee bypass: the attacker declares classes without paying fees, depriving the sequencer/protocol of owed fee revenue.

---

### Likelihood Explanation

The conditions are fully attacker-controlled and require no privileged access:
- `sender_address = 'BOOTSTRAP'` is a publicly known felt value
- `nonce = 0` is always valid for the 'BOOTSTRAP' address (never incremented)
- `version = 3` is the only accepted version
- Zero resource bounds is a valid transaction configuration

The class hash of any publicly known Sierra contract is computable offline. An attacker can front-run a legitimate declare transaction by submitting a BOOTSTRAP declare with the same `class_hash` but an invalid `compiled_class_hash`. The attack is repeatable across blocks due to the missing nonce increment.

---

### Recommendation

Add an explicit authorization check before the BOOTSTRAP branch. The simplest fix is to verify that the `sender_address` is a designated, protocol-controlled bootstrap address stored in `block_context`, rather than accepting any transaction that happens to use the felt encoding of `'BOOTSTRAP'`. Alternatively, remove the BOOTSTRAP path entirely if it is no longer needed for production, or gate it behind a block-number range check (e.g., only valid during genesis blocks).

---

### Proof of Concept

1. Compute the felt encoding of `'BOOTSTRAP'`: `felt = 0x424f4f545354524150` (ASCII bytes of "BOOTSTRAP").
2. Identify a target Sierra class hash `C` (e.g., the fee token class, computable from public source).
3. Craft a declare transaction:
   - `sender_address = 0x424f4f545354524150`
   - `class_hash = C`
   - `compiled_class_hash = 1` (invalid, non-zero)
   - `nonce = 0`
   - `version = 3`
   - All resource bounds = 0
4. Submit to the sequencer. The OS processes it via the BOOTSTRAP path (lines 764–775), skipping `__validate_declare__`, nonce check, and fee charge, and writes `C -> 1` into `contract_class_changes`.
5. The legitimate declarer submits the correct declare transaction for `C` with the real `compiled_class_hash`. The `dict_update` with `prev_value=0` fails — the class is already declared.
6. Any contract deployed under class hash `C` calls `execute_entry_point`, which reads `compiled_class_hash = 1` from `contract_class_changes` and fails to find it in the compiled class facts bundle, causing permanent execution failure and freezing all funds in those contracts. [1](#0-0) [6](#0-5)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
