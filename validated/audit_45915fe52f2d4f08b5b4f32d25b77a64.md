### Title
Missing Class Declaration Validation in `execute_replace_class` Allows Permanent Freezing of Funds — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by the caller has been previously declared on-chain. This is the direct Cairo analog of the EVM "missing approval" pattern: a prerequisite step (class declaration) that is required for the replaced contract to remain callable is never enforced, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` accepts an arbitrary `class_hash` from the calling contract and writes it directly into `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is an explicit acknowledgment that the required check is absent.

When any future transaction attempts to call the replaced contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

Because the undeclared class hash was never inserted into `contract_class_changes`, `dict_read` returns `0`. `find_element` then searches for a compiled class with hash `0`, which does not exist, causing the OS to panic. Any call to the bricked contract is permanently unexecutable, and all funds held by it are frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared hash, the contract becomes permanently uncallable. No recovery path exists: calling `replace_class` again requires executing the contract, which itself fails. Any ETH, tokens, or other assets held by the contract are irreversibly locked.

---

### Likelihood Explanation

**High.** The `replace_class` syscall is available to any deployed contract with no access restriction. An attacker needs only to:

1. Deploy a contract whose code calls `replace_class` with an arbitrary (undeclared) class hash.
2. Invoke that contract once.

No privileged role, leaked key, or external dependency is required. The OS enforces no guard at the protocol level.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the class hash exists in `contract_class_changes` (i.e., has been declared). The check should mirror the existing declaration enforcement used in `execute_declare_transaction`:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This is precisely what the existing TODO comment calls for.

---

### Proof of Concept

**Attack path (unprivileged):**

1. Attacker deploys `AttackerContract` whose `attack()` function issues a `replace_class` syscall with a class hash that was never declared (e.g., `0xdeadbeef`).
2. Attacker calls `attack()`. The OS executes `execute_replace_class`, writes `class_hash=0xdeadbeef` into `contract_state_changes` for `AttackerContract`, and the transaction succeeds.
3. The state is committed. `AttackerContract` now has `class_hash=0xdeadbeef` in the global state.
4. Any subsequent call to `AttackerContract` reaches `execute_entry_point`, which calls `dict_read(contract_class_changes, key=0xdeadbeef)` → returns `0` (never declared). `find_element` searches for compiled class hash `0` → not found → OS panics / call permanently fails.
5. All funds held by `AttackerContract` are permanently frozen.

The root cause is solely the missing declaration check at: [3](#0-2)
