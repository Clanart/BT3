### Title
Unchecked `recipient` Account in Solana `FinalizeTransfer` Allows Token Theft — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary

The Solana `FinalizeTransfer` instruction accepts a caller-supplied `recipient` account that is declared as `UncheckedAccount` with no constraint binding it to the `recipient` field inside the MPC-signed `FinalizeTransferPayload`. Because the token account that receives bridged assets is derived from this unchecked `recipient`, any caller can substitute an arbitrary address and redirect the finalized tokens to themselves while presenting a fully valid MPC signature.

### Finding Description

In `finalize_transfer.rs`, the Anchor accounts struct is:

```rust
#[derive(Accounts)]
#[instruction(data: SignedPayload<FinalizeTransferPayload>)]
pub struct FinalizeTransfer<'info> {
    ...
    /// CHECK: this can be any type of account
    pub recipient: UncheckedAccount<'info>,
    ...
    #[account(
        init_if_needed,
        payer = common.payer,
        associated_token::mint = mint,
        associated_token::authority = recipient,   // ← derived from unchecked recipient
        token::token_program = token_program,
    )]
    pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
    ...
}
``` [1](#0-0) [2](#0-1) 

The `process` function then transfers or mints tokens directly into `self.token_account` (the ATA of `self.recipient`):

```rust
pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
    UsedNonces::use_nonce(data.destination_nonce, ...)?;

    if let Some(vault) = &self.vault {
        transfer_checked(... to: self.token_account.to_account_info() ...)?;
    } else {
        mint_to(... to: self.token_account.to_account_info() ...)?;
    }
    ...
}
``` [3](#0-2) 

At no point is `self.recipient.key()` compared against `data.recipient` (the recipient committed to in the MPC-signed payload). The `#[instruction(data: ...)]` attribute makes `data.payload` available for account constraints, but the `recipient` account carries only `/// CHECK: this can be any type of account` with no binding constraint.

Compare with the EVM side, where the recipient is included in the Borsh-encoded hash that is ECDSA-verified against `nearBridgeDerivedAddress`, making substitution impossible:

```solidity
Borsh.encodeAddress(payload.recipient),
``` [4](#0-3) 

On Solana, no equivalent binding exists.

Additionally, the `FinalizeTransferResponse` posted back to NEAR does not include the actual recipient who received the tokens:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),
    amount: data.amount,
    fee_recipient: data.fee_recipient.unwrap_or_default(),
    transfer_id: data.transfer_id,   // ← no recipient field
}.serialize_for_near(())?;
``` [5](#0-4) 

NEAR therefore cannot detect that the wrong recipient received the tokens.

### Impact Explanation

**Critical — Direct theft of bridged assets on Solana.**

An attacker who submits `finalize_transfer` with a valid MPC-signed `FinalizeTransferPayload` but substitutes their own public key as the `recipient` account will receive the bridged tokens (either released from the vault or freshly minted) instead of the legitimate recipient. The nonce is consumed, so the legitimate recipient can never claim their tokens. The NEAR bridge contract is not informed of the substitution and cannot revert or compensate.

### Likelihood Explanation

**High.** Wormhole VAAs (which carry the signed payload) are publicly observable on-chain once guardian-signed. Any party — not just the original relayer — can call `finalize_transfer` with a valid VAA. The attacker simply passes their own address as `recipient` in the instruction accounts. No privileged access, leaked key, or colluding MPC signer is required. The only race condition is being first to submit the instruction for a given nonce, which is straightforward on Solana given VAA public availability.

### Recommendation

Add an explicit account constraint that binds the `recipient` account to the recipient committed in the signed payload:

```rust
#[account(
    constraint = recipient.key() == data.payload.recipient @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

This mirrors the EVM approach where the recipient is part of the signed hash and cannot be substituted.

### Proof of Concept

1. A legitimate user initiates a transfer from NEAR/EVM to Solana. The MPC signs a `FinalizeTransferPayload` containing `recipient = <victim_pubkey>` and `destination_nonce = N`.
2. The Wormhole VAA carrying this signed payload becomes publicly available.
3. Attacker calls `finalize_transfer` with:
   - `data` = the valid `SignedPayload<FinalizeTransferPayload>` (signature intact, `destination_nonce = N`, `recipient = <victim_pubkey>` inside the payload)
   - `recipient` account = attacker's own Solana public key
4. Anchor derives `token_account` as the ATA of the attacker's key.
5. `process` verifies the nonce (marks N as used), then transfers/mints `data.amount` tokens into the attacker's ATA.
6. The victim's nonce N is now consumed; any subsequent attempt by the legitimate relayer to finalize the transfer fails with `NonceAlreadyUsed`.
7. NEAR receives a `FinalizeTransferResponse` with no recipient field and cannot detect the theft.

impl FinalizeTransfer<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        UsedNonces::use_nonce(
            data.destination_nonce,
            &self.used_nonces,
            &mut self.config,
            self.authority.to_account_info(),
            self.common.payer.to_account_info(),
            &Rent::get()?,
            self.system_program.to_account_info(),
        )?;

        if let Some(vault) = &self.vault {
            // Native version. We have a proof of token registration by vault existence
            transfer_checked(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    TransferChecked {
                        from: vault.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                        mint: self.mint.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
                self.mint.decimals,
            )?;
        } else {
            // Bridged version. May be a fake token with our authority set but it will be ignored on the near side
            require!(
                self.mint.mint_authority.contains(self.authority.key),
                ErrorCode::InvalidBridgedToken
            );

            mint_to(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    MintTo {
                        mint: self.mint.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
            )?;
        }

        let payload = FinalizeTransferResponse {
            token: self.mint.key(),
            amount: data.amount,
            fee_recipient: data.fee_recipient.unwrap_or_default(),
            transfer_id: data.transfer_id,
        }
        .serialize_for_near(())?;

        self.common.post_message(payload)?;

        Ok(())
    }
