### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts an arbitrary, user-supplied class hash and writes it directly into the contract's state entry **without verifying that the class hash corresponds to a declared contract class**. A contract can therefore replace its own class with an undeclared (non-existent) hash, permanently breaking all future calls to that contract and freezing any funds it holds. The OS itself contains a `TODO` comment acknowledging this missing check.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `class_hash` directly from the syscall request and updates `contract_state_changes` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `class_hash` field is taken verbatim from the contract-controlled syscall segment. The OS performs no lookup against `compiled_class_facts` or `deprecated_compiled_class_facts` to confirm the hash is declared. The state update is committed and the transaction succeeds. Any subsequent call to the contract will attempt to dispatch to the now-invalid class hash, find no matching compiled class, and fail permanently.

This is the direct analog of the "Beware Of Malicious Adapters" pattern: just as `Periphery.sol` accepted any adapter address and derived critical state (token approvals) from it without validation, the OS here accepts any class hash from the contract and derives critical persistent state (the contract's class pointer) from it without validation.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class` is executed with an undeclared class hash, the contract's `class_hash` field in `contract_state_changes` is permanently set to an invalid value. Every future call to that contract will fail at class dispatch time because no compiled class exists for the hash. There is no recovery path: the contract cannot call `replace_class` again (because execution of its entry points is impossible), and no OS-level mechanism exists to reset the class hash externally. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

**High.**

- The `replace_class` syscall is a standard, publicly available syscall callable by any contract.
- No privilege, operator role, or special permission is required — any contract deployer can deploy a contract that calls `replace_class` with an arbitrary hash.
- The missing check is explicitly acknowledged by a `TODO` comment in the production code, confirming the developers are aware the guard is absent.
- A single transaction is sufficient to trigger the freeze; no repeated interaction or race condition is needed.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that `class_hash` is present in the block's declared class set. Concretely:

1. Perform a `dict_read` on `contract_class_changes` (or the global compiled-class mapping) for `class_hash` and assert the result is non-zero.
2. Alternatively, enforce at the Sierra/Casm compilation boundary that `replace_class` can only be called with a hash that was declared in the same block or a prior block, mirroring the check already applied to `deploy` class hashes.

This directly resolves the `TODO` at line 898 and closes the analogous gap to the adapter-whitelist recommendation in the external report.

---

### Proof of Concept

1. **Attacker deploys a vault contract** (e.g., a simple ERC-20 wrapper) that holds user funds and exposes a public `drain_class()` entry point.
2. Users deposit tokens into the vault; the vault's storage now holds balances.
3. **Attacker calls `drain_class()`**, which internally issues a `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared hash).
4. The OS executes `execute_replace_class`:
   - Gas is deducted.
   - `contract_state_changes` is updated: `vault_address → StateEntry(class_hash=0xdeadbeef, ...)`.
   - No validation occurs (the `TODO` guard is absent).
   - The transaction succeeds and is committed to state.
5. **Any subsequent call to the vault** (e.g., `withdraw`) reaches the OS entry-point dispatcher, which looks up `class_hash = 0xdeadbeef` in `compiled_class_facts` — finds nothing — and the call reverts with `ENTRYPOINT_NOT_FOUND`.
6. **All user funds are permanently frozen.** The vault can never execute another entry point; no recovery is possible. [2](#0-1)
