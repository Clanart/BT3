### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from the contract request without verifying that the hash corresponds to a previously declared class. This is an explicit, acknowledged missing check (marked with a `TODO` in the source). A contract can replace its own class hash with an undeclared value, causing the contract to become permanently inaccessible and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The `request.class_hash` value is taken directly from the syscall request and written into `contract_state_changes` with no check that the hash exists in `contract_class_changes` (the declared class registry). The `TODO` comment on line 898 explicitly acknowledges this missing validation.

When a contract with an undeclared class hash is subsequently called, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

If `class_hash` is not declared, `dict_read` returns `0`, and `find_element` cannot locate a compiled class for hash `0`. This causes a hard Cairo assertion failure — not a graceful transaction revert — making the contract permanently uncallable.

---

### Impact Explanation

**Critical — Permanent Freezing of Funds.**

Once a contract's class hash is replaced with an undeclared value and the block is finalized:

1. The state change is committed and irreversible (the revert log only applies within the same transaction).
2. Any future transaction targeting the contract will fail at the OS proof level when `find_element` cannot locate the compiled class.
3. All assets (ETH, ERC-20 tokens, NFTs) held by the contract are permanently inaccessible.

The sequencer cannot include transactions calling the affected contract without causing proof failure, so the funds are frozen at the protocol level with no recovery path.

---

### Likelihood Explanation

The `replace_class` syscall is available to any executing contract — no special privilege is required. The attack path is:

1. An unprivileged user deploys a contract (standard operation).
2. The contract's logic calls `replace_class` with an arbitrary felt value that does not correspond to any declared class hash.
3. The OS processes the syscall, writes the undeclared hash to `contract_state_changes`, and commits the block.
4. The contract is now permanently frozen.

This can occur through:
- A deliberately malicious contract designed to self-destruct after receiving funds.
- A buggy contract that passes an incorrect class hash to `replace_class`.
- An attacker exploiting a reentrancy or input-validation flaw in a third-party contract to trigger `replace_class` with a crafted hash.

The missing check is acknowledged in the codebase itself, confirming the developers are aware the validation is absent.

---

### Recommendation

Before committing the `replace_class` state change, verify that `request.class_hash` is present in `contract_class_changes`:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation implicitly performed in `execute_entry_point` and should be enforced at the point of the syscall, not deferred to the next call.

---

### Proof of Concept

1. Deploy contract `A` with a valid declared class hash `X`.
2. Fund contract `A` with tokens.
3. Contract `A` executes `replace_class(Y)` where `Y` is any felt not present in `contract_class_changes` (e.g., `Y = 0xdeadbeef`).
4. `execute_replace_class` writes `StateEntry(class_hash=Y, ...)` to `contract_state_changes` with no validation.
5. Block is finalized; state is committed.
6. Any subsequent invoke targeting contract `A` reaches `execute_entry_point`, calls `dict_read(contract_class_changes, Y)` → returns `0`, then `find_element(..., key=0)` fails with a Cairo assertion.
7. The sequencer cannot include any transaction calling `A`. All funds in `A` are permanently frozen.
