### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key/Hash Validation for Bulk Entries, Enabling Tree-Root Corruption — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that the single-item `insert` path enforces. An untrusted caller supplying a batch that contains repeated `KeyId` or `Hash` values can silently corrupt the Merkle tree's internal structure and produce an incorrect root hash, invalidating all subsequent proofs of inclusion or exclusion.

### Finding Description

`MerkleBlob::insert` explicitly rejects duplicates before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` routes only the **last two** items (popped from the end of the vector when the tree has ≤ 1 leaf) through `self.insert()`, which carries those guards. Every remaining item is written directly via `insert_entry_to_blob` with **no duplicate check**:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no key/hash guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When a duplicate `KeyId` is written, `insert_entry_to_blob` updates `block_status_cache` so the cache's `key_to_index` map now points to the **new** index, leaving the original leaf node as an unreachable orphan in the blob. The parent-chain hashes built during the subsequent tree-linking phase are computed over the new (duplicate) leaf, not the original, so the committed root hash diverges from the true data set. A duplicate `Hash` produces the same structural corruption through `leaf_hash_to_index`.

The Python binding `py_batch_insert` passes caller-supplied lists directly into this path without any pre-filtering:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2) 

The analogy to the external report is direct: just as `distributeRewards()` updates `periodState.distributed` without transferring tokens (allowing a cap to be exhausted without the corresponding resource), `batch_insert` updates the Merkle tree's internal accounting (`block_status_cache`, blob indices, parent-chain hashes) without enforcing the invariant that each key and hash is unique. The result is a committed root that does not faithfully represent the stored data.

### Impact Explanation

A corrupted root hash means every `get_proof_of_inclusion` call for the orphaned original key will either fail or return a proof that does not verify against the stored root. Conversely, a proof constructed against the duplicate leaf will verify against the wrong root. This satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`py_batch_insert` is the primary Python-facing bulk-insert API. Any code path that feeds externally-sourced key/value/hash triples into `batch_insert` without pre-deduplication is vulnerable. Because the single-item `insert` path correctly rejects duplicates, callers may reasonably assume `batch_insert` provides the same guarantee.

### Recommendation

Add the same duplicate guards at the top of the bulk loop in `batch_insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    ...
}
```

Alternatively, pre-deduplicate the input vector before the loop and return an error if any duplicates are detected, keeping the fast path intact.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Insert 4 items; items 0-1 go through the safe `insert` path,
# items 2-3 bypass duplicate checks in the bulk loop.
# Provide a duplicate KeyId for item 2 (same as item 0).
key0, val0 = KeyId(0), ValueId(0)
key1, val1 = KeyId(1), ValueId(1)
key_dup = KeyId(0)   # duplicate of key0
val_dup  = ValueId(99)

h = lambda n: bytes(hashlib.sha256(n.to_bytes(8,'big')).digest())

# batch_insert pops from the END first, so ordering matters:
# index 3 and 2 are popped first (go through safe insert),
# index 1 and 0 go through the unchecked bulk loop.
blob.batch_insert(
    [(key_dup, val_dup), (key1, val1), (key0, val0), (KeyId(2), ValueId(2))],
    [h(99), h(1), h(0), h(2)]
)
blob.calculate_lazy_hashes()

# The tree now contains two leaves with KeyId(0); the root hash is wrong.
# A proof for the original (key0, val0) will fail or verify against a bad root.
proof = blob.get_proof_of_inclusion(key0)
print("proof valid:", proof.valid())   # may return True for the wrong leaf
``` [4](#0-3)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
