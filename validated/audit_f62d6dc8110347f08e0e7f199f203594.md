### Title
Missing Declared Class Validation in `execute_replace_class` Enables Permanent Freezing of Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The `execute_replace_class` function in the StarkNet OS does not validate that the new class hash provided by the caller corresponds to a declared contract class. This allows any contract to replace its class with an arbitrary, undeclared hash, permanently breaking the contract and freezing any funds it holds. The missing check is explicitly acknowledged in the codebase with a TODO comment.

---

### Finding Description

In `syscall_impls.cairo` at line 898, there is an explicit TODO acknowledging the missing validation:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The function reads `class_hash` directly from the attacker-controlled syscall request and writes it into `contract_state_changes` without ever consulting `contract_class_changes` to confirm the hash is declared. The two state dictionaries are entirely decoupled at this point:

- `contract_class_changes` — maps `class_hash → compiled_class_hash` (populated only by `declare` transactions)
- `contract_state_changes` — maps `contract_address → StateEntry` (which embeds `class_hash`)

`execute_replace_class` updates only the second dict. No cross-check against the first is performed.

**Structural analogy to the reported vulnerability:** In the report, a beneficiary calls `beneficiarySwitch` to change their identifier to an arbitrary value, causing the owner's `setBeneficiaries` accounting (shareSum) to become inconsistent and revert. Here, a contract calls `replace_class` to change its class identifier to an arbitrary, undeclared value, causing the OS's class-execution lookup to become inconsistent and fail permanently. In both cases, a mutable identifier is changed to an invalid value without validation, breaking downstream operations that depend on the identifier being stable and valid.

The same missing check exists in the deprecated path at `deprecated_execute_syscalls.cairo` line 307–328, which also performs no validation of the new class hash.

---

### Impact Explanation

**Critical: Permanent freezing of funds.**

Once a contract's `class_hash` in `contract_state_changes` is set to an undeclared value:

1. The state is committed to the global state tree via `compute_contract_state_commitment` in `commitment.cairo`.
2. All subsequent calls to the contract require the OS to locate and execute the class identified by that hash.
3. Since the hash has no entry in `contract_class_changes`, the prover cannot supply a valid compiled class, and execution of the contract is permanently impossible.
4. Any ERC-20 balances, ETH, or other assets held by the contract become permanently inaccessible — there is no recovery path because the contract's own execution entry point is broken.

---

### Likelihood Explanation

**Medium.** The attack is reachable by:

1. **Contract deployer (direct):** A deployer deploys a contract whose logic calls `replace_class` with an undeclared hash (e.g., a hash of `0xdeadbeef` or any felt not present in `contract_class_changes`). The OS accepts this without validation.

2. **Unprivileged transaction sender (indirect):** Many real-world contracts implement on-chain governance or upgrade mechanisms that allow token holders or other unprivileged parties to propose and execute class upgrades. An attacker submits a proposal pointing to an undeclared class hash. If the governance contract executes `replace_class` with that hash, the contract is permanently broken and its funds frozen.

The TODO comment at line 898 confirms this is a known, unresolved gap in the OS's enforcement layer.

---

### Recommendation

Inside `execute_replace_class`, after reading `class_hash` from the request, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the result is non-zero (i.e., the class has been declared). This mirrors the invariant already enforced for `declare` transactions, where `dict_update` with `prev_value=0` ensures a class is declared at most once.

```cairo
// Proposed fix (pseudocode):
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

---

### Proof of Concept

1. Deploy contract **C** that holds an ERC-20 token balance (or any funds).
2. From within **C** (or via a governance call that **C** exposes), invoke the `replace_class` syscall with `class_hash = H`, where `H` is any felt that was never submitted in a `declare` transaction and therefore absent from `contract_class_changes`.
3. `execute_replace_class` in `syscall_impls.cairo` (lines 877–916) updates `contract_state_changes[C].class_hash = H` with no validation.
4. The updated state is committed to the global state tree by `compute_contract_state_commitment` (`commitment.cairo` lines 190–203).
5. In all subsequent blocks, any transaction targeting **C** requires the OS to execute class **H**. Since **H** has no compiled class, the prover cannot construct a valid proof for any call to **C**.
6. All funds held by **C** are permanently frozen with no recovery mechanism.
