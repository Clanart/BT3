### Title
Unconstrained `is_reverted` Hint Allows Prover to Skip Transaction Execution While Charging Full Fees â€” (`File: execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a no-op stub. The `is_reverted` flag controlling whether an invoke transaction's execute entry point runs is loaded from an unconstrained prover hint. Because no Cairo constraint enforces that this flag matches the actual execution outcome, a malicious prover can mark any successful invoke transaction as "reverted," skip its execution entirely, and still charge the user the full fee â€” producing a valid ZK proof of an incorrect state transition.

---

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`, lines 338â€“361), the outer `is_reverted` value is loaded from a hint and then passed to `check_is_reverted`:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(...);
    }
} else {
    // Align the stack ... (execution skipped entirely)
}

// Charge fee â€” always called regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The `check_is_reverted` function that is supposed to constrain this value is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This is a **complete no-op**. There is no `assert`, no range check, and no comparison against any execution result. The value is entirely prover-controlled.

This is structurally distinct from the `is_reverted` flag inside `execute_entry_point`, which is correctly constrained to `entry_point_return_values.failure_flag` from the actual contract execution:

```cairo
local is_reverted = entry_point_return_values.failure_flag;
``` [3](#0-2) 

The outer `is_reverted` in `execute_invoke_function_transaction` gates whether `execute_entry_point` is ever called at all. If it is set to `TRUE`, `execute_entry_point` is never invoked, so the inner constrained `failure_flag` is never produced. The prover can therefore assert `is_reverted = TRUE` for any transaction, skip execution, and the proof remains valid because no constraint is violated.

The `charge_fee` call that follows is unconditional and uses `updated_tx_execution_context`, which is the same context regardless of the `is_reverted` branch taken: [4](#0-3) 

The actual fee amount is loaded from a hint (`%{ LoadActualFee %}`), constrained only to be `<= max_fee`. A malicious prover can set it to `max_fee` while skipping execution.

---

### Impact Explanation

**Direct loss of funds (Critical).**

A malicious prover can, for any invoke transaction submitted by any user:

1. Set `is_reverted = TRUE` via the `%{ IsReverted %}` hint.
2. Skip the `__execute__` entry point entirely â€” no state changes are applied.
3. Still invoke `charge_fee`, deducting up to `max_fee` from the user's ERC-20 balance.
4. Produce a valid ZK proof accepted by the L1 verifier.

The user pays the full fee but receives no execution. Their intended state changes (token transfers, contract interactions, etc.) are silently dropped. Because the proof is valid, the L1 verifier accepts the block, and the user has no on-chain recourse. This is a direct, provably-valid theft of user fee funds.

---

### Likelihood Explanation

The prover must be malicious or compromised. In StarkNet's current architecture the sequencer and prover are centralized, making this a privileged-role attack. However:

- The entire purpose of a ZK proof is to remove the need to trust the prover. This bug breaks that guarantee at the protocol level.
- As StarkNet decentralizes proving, the attack surface widens to any entity that can submit a proof.
- The attack is silent and leaves no on-chain evidence distinguishing it from a legitimately reverted transaction.
- The `check_is_reverted` stub name and its placement in `execution_constraints.cairo` strongly suggest it is an **unimplemented placeholder**, not an intentional design choice.

---

### Recommendation

Implement `check_is_reverted` to actually constrain the `is_reverted` value. The correct approach is one of:

1. **Always run execution**: Remove the outer `is_reverted` hint entirely. Always call `execute_entry_point` and use its constrained `failure_flag` as the authoritative revert signal.
2. **Commit-and-verify**: If skipping execution for pre-known reverts is desired for performance, require the prover to supply a commitment (e.g., a hash of the execution trace) that is verified against a known-revert registry, and add a Cairo `assert` inside `check_is_reverted` that validates this commitment.

The current empty body must not remain in production:

```cairo
// BROKEN â€” provides zero constraint:
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1)

### Proof of Concept

**Attacker**: Malicious prover (controls hint values).  
**Victim**: Any user who submits an invoke transaction.

**Step-by-step**:

1. User submits an invoke transaction (e.g., a token transfer of 100 STRK).
2. Prover sets `%{ IsReverted %}` hint to `is_reverted = 1` (TRUE).
3. In `execute_invoke_function_transaction`, `check_is_reverted(1)` is called â€” returns immediately, no constraint fired.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called; the token transfer never executes.
5. `charge_fee` is called unconditionally; prover sets `%{ LoadActualFee %}` to `max_fee`; the user's fee token balance is debited.
6. The state diff shows: fee token balance decreased, no other changes. This is a consistent state.
7. The ZK proof is generated over this consistent (but incorrect) state â€” all Cairo constraints are satisfied because `check_is_reverted` imposed none.
8. The L1 verifier accepts the proof. The block is finalized. The user's 100 STRK transfer never happened; their fee is gone.
