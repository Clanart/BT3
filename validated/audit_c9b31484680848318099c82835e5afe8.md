Audit Report

## Title
Stale Router ETH from Unclaimed `msg.value` Funds a Subsequent User's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay()` function's WETH branch reads `address(this).balance` with no per-transaction accounting. ETH sent as `msg.value` in any prior `payable` call that was not refunded remains in the router and is silently consumed to fund a subsequent user's WETH payment, causing direct, permanent loss of the original depositor's ETH.

## Finding Description

In `pay()`, when `token == WETH` and `payer != address(this)`, the function reads the router's entire native balance: [1](#0-0) 

`address(this).balance` is a global contract balance with no binding to the current caller or transaction. Any ETH left over from a prior `msg.value` call is indistinguishable from ETH sent in the current call.

The `receive()` guard only blocks *direct* ETH transfers from non-WETH addresses: [2](#0-1) 

It does not prevent ETH from accumulating via `msg.value` on any `payable` entry point. Both `exactInputSingle` and `multicall` are `payable`: [3](#0-2) [4](#0-3) 

`refundETH()` is optional and not enforced: [5](#0-4) 

**Exploit path:**
1. User A calls `exactInputSingle{value: 2 ether}` with `tokenIn = WETH`, `amountIn = 1 ETH`. The callback triggers `pay(WETH, userA, pool, 1e18)`. `nativeBalance = 2 ETH ≥ 1 ETH`, so 1 ETH is wrapped and forwarded. **1 ETH remains stranded in the router.**
2. User A omits `refundETH()`.
3. User B calls `exactInputSingle` (no ETH sent) with `tokenIn = WETH`, `amountIn = 0.5 ETH`. The callback triggers `pay(WETH, userB, pool, 0.5e18)`. `nativeBalance = 1 ETH ≥ 0.5 ETH`, so **User A's ETH** is wrapped and forwarded. User B's WETH `transferFrom` is never called. User A loses 0.5 ETH permanently.

## Impact Explanation

Direct, permanent loss of User A's ETH principal. The ETH is not locked — it is actively transferred to a pool on behalf of an unrelated user. This meets the "direct loss of user principal" impact gate at Critical/High severity.

## Likelihood Explanation

Sending excess `msg.value` is a standard defensive pattern (users pad ETH to ensure swaps succeed at any price). `refundETH()` is optional and not enforced by the router. Any two sequential WETH-input transactions where the first leaves residual ETH trigger the bug. No privileged role or malicious setup is required.

## Recommendation

Track `msg.value` per transaction rather than reading `address(this).balance`. The standard fix (used by Uniswap v3's `PeripheryPayments`) is to pass `msg.value` into the payment context (e.g., via a transient storage slot set at the entry point) and consume only up to that amount in `pay()`. Alternatively, enforce that `address(this).balance == 0` at the start of every non-`multicall` entry point, or require `refundETH()` as the final call in every ETH-involving multicall.

## Proof of Concept

```solidity
// 1. User A sends 2 ETH, swap costs 1 ETH, 1 ETH stranded in router
router.exactInputSingle{value: 2 ether}(paramsA); // tokenIn=WETH, amountIn=1e18
// User A forgets refundETH()
assertEq(address(router).balance, 1 ether);

// 2. User B swaps 0.5 WETH — router uses User A's ETH, never pulls from User B
uint256 balBefore = weth.balanceOf(userB);
router.exactInputSingle(paramsB); // tokenIn=WETH, amountIn=0.5e18, msg.value=0
assertEq(weth.balanceOf(userB), balBefore); // User B's WETH untouched
assertEq(address(router).balance, 0.5 ether); // 0.5 ETH consumed from User A's deposit
```
