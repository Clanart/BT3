### Title
Missing Chain ID, Height, and Round in Precommit Vote Message Digest Enables Cross-Chain and Cross-Height Signature Replay - (`crates/apollo_signature_manager/src/signature_manager.rs`)

### Summary

`build_precommit_vote_message_digest` constructs the ECDSA digest for consensus precommit vote signatures using only a static domain tag (`b"PRECOMMIT_VOTE"`) and the `block_hash`. It omits chain ID, block height, round, and voter address. When the planned signature verification is wired into the consensus vote path (currently a `TODO`), a valid precommit signature produced on one Starknet chain can be replayed verbatim on any other chain where the same block hash appears, or at a different height/round on the same chain.

### Finding Description

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs` constructs the signed payload as:

```
blake2s(b"PRECOMMIT_VOTE" || block_hash_bytes)
``` [1](#0-0) 

The `Vote` struct carries `height`, `round`, `proposal_commitment` (the block hash), `voter`, and `signature`, but none of `height`, `round`, `voter`, or any chain-specific identifier are included in the signed digest. [2](#0-1) 

The signing entry point `sign_precommit_vote` is fully implemented and wired into the live `SignatureManager` component via the `SignatureManagerRequest::SignPrecommitVote` handler: [3](#0-2) [4](#0-3) 

The verification counterpart `verify_precommit_vote_signature` is equally implemented and exported: [5](#0-4) 

The consensus `handle_vote` path currently carries an explicit `// TODO(Asmaa): verify the signature` marker, meaning verification is not yet enforced: [6](#0-5) 

Similarly, `make_self_vote` emits `RawSignature::default()` with a `// TODO(Asmaa): sign the vote` comment: [7](#0-6) 

When both TODOs are resolved and the `SignatureManager` client is called to sign/verify votes, the digest will still be `blake2s(b"PRECOMMIT_VOTE" || block_hash)` — identical across all chains, heights, and rounds that share the same block hash.

### Impact Explanation

Once signature verification is activated, an attacker who observes a valid precommit signature `(r, s)` from validator `V` for block hash `H` on Starknet mainnet can submit the identical `Vote { voter: V, proposal_commitment: H, signature: (r,s) }` on Starknet testnet (or any other deployment) where the same block hash `H` appears. Because the digest does not bind to chain ID, height, or round, `verify_precommit_vote_signature` will return `true` for the replayed vote. A quorum of such replayed precommits would cause `upon_decision` to emit `DecisionReached` for a block the validators never actually voted for on that chain, corrupting the committed block hash and state root. [8](#0-7) 

This maps to: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload** (consensus vote signature logic binds the wrong context, enabling an unauthorized `DecisionReached` event and wrong committed block hash/state root).

### Likelihood Explanation

Starknet is deployed across mainnet, Sepolia testnet, and integration environments. The `ConsensusManager` is designed for multi-chain deployment. Validator key pairs are typically reused across deployments. Block hashes are deterministic from block content; an attacker who can influence block content on one chain (e.g., by submitting transactions) can engineer a collision with a target block hash on another chain. The attack requires only passive observation of a valid precommit on one chain and submission of the same `Vote` message on another — no privileged access is needed.

### Recommendation

Include chain ID, block height, and round in the message digest:

```rust
fn build_precommit_vote_message_digest(
    chain_id: &ChainId,
    height: BlockNumber,
    round: Round,
    block_hash: BlockHash,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(chain_id.as_bytes());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    message.extend_from_slice(&block_hash.to_bytes_be());
    MessageDigest(blake2s_to_felt(&message))
}
```

### Proof of Concept

1. Validator `V` signs a precommit for block hash `H` at height 100, round 0 on Starknet mainnet. The digest is `blake2s(b"PRECOMMIT_VOTE" || H)`.
2. On Starknet Sepolia, block 100 round 0 happens to produce (or is engineered to produce) the same block hash `H`.
3. Attacker submits `Vote { vote_type: Precommit, height: 100, round: 0, proposal_commitment: H, voter: V, signature: (r,s) }` to the Sepolia consensus network.
4. When `verify_precommit_vote_signature(H, (r,s), V.public_key)` is called, it recomputes `blake2s(b"PRECOMMIT_VOTE" || H)` — identical to the mainnet digest — and returns `true`.
5. With enough replayed precommits from the mainnet validator set (whose keys are reused on Sepolia), `upon_decision` fires on Sepolia for a block the Sepolia validators never endorsed, committing a wrong state root. [1](#0-0) [5](#0-4)
