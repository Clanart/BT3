### Title
`Simulator.applyCall` Bypasses `EnableCreate`/`EnableCall` Governance Guards in `eth_simulateV1` — (File: `x/evm/keeper/simulate.go`)

---

### Summary

The `Simulator.applyCall` function, which backs the public `eth_simulateV1` JSON-RPC endpoint, omits the `EnableCreate`/`EnableCall` governance parameter checks that every other EVM execution path enforces. When governance disables contract creation or calls (the Ethermint analog of an "emergency exit"), `eth_simulateV1` still executes those operations successfully, returning results that contradict what actual on-chain execution would produce.

---

### Finding Description

**Guarded paths (correct)**

`ApplyMessageWithConfig` — the path used by real transaction execution, `eth_call`, and `eth_estimateGas` — checks the governance flags at the very top of the function before any EVM state is touched:

```go
// x/evm/keeper/state_transition.go:352-356
if !cfg.Params.EnableCreate && msg.To == nil {
    return nil, errorsmod.Wrap(types.ErrCreateDisabled, "failed to create new contract")
} else if !cfg.Params.EnableCall && msg.To != nil {
    return nil, errorsmod.Wrap(types.ErrCallDisabled, "failed to call contract")
}
``` [1](#0-0) 

The ante handler `ValidateEthBasic` enforces the same check for every `MsgEthereumTx` entering the mempool:

```go
// ante/interfaces/setup.go:116-120
if !enableCreate && tx.To() == nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "failed to create new contract")
} else if !enableCall && tx.To() != nil {
    return errorsmod.Wrap(evmtypes.ErrCallDisabled, "failed to call contract")
}
``` [2](#0-1) 

**Unguarded path (vulnerable)**

`Simulator.applyCall` — the execution engine for `eth_simulateV1` — mirrors `ApplyMessageWithConfig` for intrinsic gas, EIP-7623 floor gas, Shanghai init-code size, nonce, and fee checks, but **never reads `EnableCreate` or `EnableCall`**. It proceeds directly to `evm.Create` or `evm.Call`:

```go
// x/evm/keeper/simulate.go:438-450
if contractCreation {
    ret, _, leftoverGas, vmErr = evm.Create(msg.From, msg.Data, leftoverGas, value)
} else {
    ...
    ret, leftoverGas, vmErr = evm.Call(msg.From, *msg.To, msg.Data, leftoverGas, value)
    ...
}
``` [3](#0-2) 

The `Simulator` struct carries no reference to the EVM `Params` at all — only `chainConfig`, `state`, `validate`, and `budget`: [4](#0-3) 

`SimulateV1` constructs the `Simulator` from `blockCfg` but never passes `blockCfg.Params.EnableCreate` / `blockCfg.Params.EnableCall` into it: [5](#0-4) 

This endpoint is publicly reachable via the gRPC query route registered in the proto definition and via the JSON-RPC backend: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

`EnableCreate` and `EnableCall` are the chain's governance-controlled circuit breakers — the direct analog of the "emergency exit" flag in the referenced report. When governance disables them (e.g., in response to a live exploit draining a contract), every real execution path (`MsgEthereumTx` ante handler, `ApplyMessageWithConfig`, `eth_call`, `eth_estimateGas`) correctly rejects the operation.

`eth_simulateV1` with `validation: true` is explicitly designed to reproduce what on-chain execution would do. Because `applyCall` skips the governance guard, it returns a **successful** simulation result for a contract creation or call that would be **rejected** on-chain. Any automated system — a DApp, a bot, a wallet — that uses `eth_simulateV1` to pre-validate transactions before submission receives incorrect consensus-critical data: it is told the transaction will succeed when it will not. This is a public JSON-RPC/gRPC path feeding incorrect data about the active governance security state into transaction execution decisions.

---

### Likelihood Explanation

- `eth_simulateV1` is an unauthenticated public JSON-RPC endpoint.
- No special privileges, keys, or network position are required.
- Any caller can submit a contract-creation or contract-call payload to `eth_simulateV1` at any time, including when `EnableCreate`/`EnableCall` are disabled.
- The bypass is unconditional — it is not gated on `validate: false`; even `validate: true` (strict mode) does not add the missing check.

---

### Recommendation

Add the `EnableCreate`/`EnableCall` guard at the top of `Simulator.applyCall`, mirroring `ApplyMessageWithConfig`:

```go
// x/evm/keeper/simulate.go — inside applyCall, before any EVM execution
evmParams := sim.keeper.GetParams(ctx)  // or pass Params into Simulator at construction
if !evmParams.EnableCreate && contractCreation {
    return applyCallResult{}, errorsmod.Wrap(types.ErrCreateDisabled, "failed to create new contract")
}
if !evmParams.EnableCall && !contractCreation {
    return applyCallResult{}, errorsmod.Wrap(types.ErrCallDisabled, "failed to call contract")
}
```

Alternatively, store `Params` in the `Simulator` struct (populated from `blockCfg.Params` in `SimulateV1`) and read it in `applyCall` without an extra store round-trip.

---

### Proof of Concept

1. Via governance, set `EnableCreate = false` (or `EnableCall = false`).
2. Submit a `eth_simulateV1` request with a contract-creation call (i.e., `to` field absent):
   ```json
   {
     "blockStateCalls": [{
       "calls": [{"from": "0xABCD...", "data": "0x6080..."}]
     }],
     "validation": true
   }
   ```
3. Observe: `SimulateV1` returns a successful result with a non-zero `contractAddress` in the receipt.
4. Submit the same transaction as a real `MsgEthereumTx` — it is rejected by the ante handler with `ErrCreateDisabled`.

The discrepancy is rooted in `applyCall` (lines 438–439 of `x/evm/keeper/simulate.go`) calling `evm.Create` without the guard present in `ApplyMessageWithConfig` (lines 352–353 of `x/evm/keeper/state_transition.go`). [8](#0-7) [1](#0-0)

func (b *Backend) SimulateV1(opts rpctypes.SimOpts, blockNr rpctypes.BlockNumber) (json.RawMessage, error) {
	if len(opts.BlockStateCalls) == 0 {
		return nil, &rpctypes.InvalidParamsError{Message: "empty input"}
	} else if len(opts.BlockStateCalls) > rpctypes.MaxSimulateBlocks {
		return nil, &rpctypes.ClientLimitExceededError{Message: "too many blocks"}
	}

	header, err := b.HeaderByNumber(blockNr)
	if err != nil {
		return nil, err
	}

	// EthHeaderFromTendermint leaves GasLimit at 0; fill it from
	// consensus params so the simulated blocks inherit the real limit.
	if header.GasLimit == 0 {
		height := blockNr.Int64()
		if height <= 0 {
			if tendermintHeader, err := b.TendermintHeaderByNumber(blockNr); err == nil {
				height = tendermintHeader.Header.Height
			}
		}
		if maxGas, err := rpctypes.BlockMaxGasFromConsensusParams(b.ctx, b.clientCtx, height); err == nil && maxGas > 0 {
			header.GasLimit = uint64(maxGas)
		}
	}

	payload := rpctypes.SimulateV1Args{
		Opts:       opts,
		BaseHeader: header,
	}
	bz, err := json.Marshal(&payload)
	if err != nil {
		return nil, err
	}

	tendermintHeader, err := b.TendermintHeaderByNumber(blockNr)
	if err != nil {
		return nil, err
	}

	req := evmtypes.SimulateV1Request{
		Args:            bz,
		GasCap:          b.RPCGasCap(),
		ProposerAddress: sdk.ConsAddress(tendermintHeader.Header.ProposerAddress),
		ChainId:         b.chainID.Int64(),
	}

	ctx := rpctypes.ContextWithHeight(blockNr.Int64())
	timeout := b.RPCEVMTimeout()

	var cancel context.CancelFunc
	if timeout > 0 {
		ctx, cancel = context.WithTimeout(ctx, timeout)
	} else {
		ctx, cancel = context.WithCancel(ctx)
	}
	defer cancel()

	res, err := b.queryClient.SimulateV1(ctx, &req)
	if err != nil {
		return nil, err
	}

	// Simulation errors are returned via dedicated fields, not packed into Result.
	if res.ErrorCode != 0 {
		return nil, &simulateError{code: int(res.ErrorCode), message: res.ErrorMessage}
	}

	length := len(res.Result)
	if length > int(b.cfg.JSONRPC.ReturnDataLimit) && b.cfg.JSONRPC.ReturnDataLimit != 0 {
		return nil, fmt.Errorf("simulate returned result of length %d exceeding limit %d", length, b.cfg.JSONRPC.ReturnDataLimit)
	}

	return json.RawMessage(res.Result), nil
