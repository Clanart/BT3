### Title
`is_refund_required` Ignores `ft_transfer_call` Panic, Permanently Freezing Funds and Corrupting `locked_tokens` Accounting — (`near/omni-bridge/src/lib.rs`)

### Summary

When `send_tokens` uses `ft_transfer_call` (triggered by a non-empty `msg` field) and the recipient contract's `ft_on_transfer` panics, the NEAR runtime returns the tokens to the bridge but the promise result is `Err`. The `is_refund_required` helper treats this `Err` case as "don't refund" (a success path), so `fin_transfer_send_tokens_callback` never restores `locked_tokens`, never removes the `fin_transfer` record, and incorrectly distributes the fee. The recipient's funds are permanently frozen inside the bridge with no on-chain recovery path.

### Finding Description

The `is_refund_required` function in `near/omni-bridge/src/lib.rs` determines whether a failed `ft_transfer_call` should trigger a state rollback:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                    amount.0 == 0   // refund if recipient returned 0
                } else {
                    false           // unexpected: don't refund
                }
            }
            Err(_) => false,        // ← BUG: promise panic → treated as success
        }
    } else {
        false
    }
}
``` [1](#0-0) 

When `ft_on_transfer` panics, the NEAR runtime rolls back the token transfer (tokens return to the bridge) and sets the promise result to `Err`. `is_refund_required` returns `false` for `Err`, so `fin_transfer_send_tokens_callback` takes the **success branch**:

```rust
} else {
    // Send fee to the fee recipient
    if transfer_message.fee.fee.0 > 0 { ... mint/transfer fee ... }
    env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
}
``` [2](#0-1) 

The **refund branch** (which would have been correct) does three things the success branch skips:

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    self.burn_tokens_if_needed(...);
    self.revert_lock_actions(&lock_actions);          // ← restores locked_tokens
    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner); // ← clears record
    env::log_str(&OmniBridgeEvent::FailedFinTransferEvent { ... });
}
``` [3](#0-2) 

The `lock_actions` vector was populated by `unlock_tokens_if_needed` inside `process_fin_transfer_to_near` **before** `send_tokens` was called:

```rust
let lock_actions = vec![self.unlock_tokens_if_needed(
    transfer_message.get_origin_chain(),
    &token,
    transfer_message.amount.0,
)];
``` [4](#0-3) 

Because `revert_lock_actions` is never called, `locked_tokens` is permanently decremented even though the tokens are back in the bridge. The `fin_transfer` record is never removed, so the transfer ID is stuck in storage and cannot be re-processed.

The `msg` field that triggers `ft_transfer_call` is user-controlled: it originates from the `message` parameter of EVM `initTransfer`, is emitted in the `InitTransfer` event, and is decoded by the NEAR prover into `init_transfer.msg`, which flows directly into `transfer_message.msg` and then into `send_tokens`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message   // ← user-supplied
);
``` [5](#0-4) 

```rust

.then(Self::ext(...).fin_transfer_send_tokens_callback(
    transfer_message, &fee_recipient,
    !msg.is_empty(),   // ← is_ft_transfer_call = true when msg non-empty

``` [6](#0-5) 

The same flaw exists in `resolve_fast_transfer`, where `burn_tokens_if_needed` is called unconditionally even when the promise is `Err`, destroying the relayer's returned tokens:

```rust
pub fn resolve_fast_transfer(...) -> U128 {
    self.burn_tokens_if_needed(token_id.clone(), amount);  // always burns
    if Self::is_refund_required(is_ft_transfer_call) {
        self.remove_fast_transfer(fast_transfer_id);
        amount
    } else {
        U128(0)   // ← Err case: tokens burned, fast_transfer record stays
    }
}

### Impact Explanation

**Permanent fund freeze (Critical):** The recipient never receives the bridged tokens. The tokens are physically present in the bridge contract (returned by the failed `ft_transfer_call`) but the bridge's accounting treats them as already delivered. There is no on-chain function an unprivileged user can call to recover them; only a DAO-privileged `transfer_token_as_dao` call could rescue the funds, and even that would leave `locked_tokens` corrupted.

**Accounting corruption (High):** `locked_tokens` is decremented by `unlock_tokens_if_needed` but never restored. The bridge now under-reports its locked collateral for the affected chain. On subsequent `fin_transfer` calls for the same token/chain pair, `unlock_tokens` may underflow or allow the bridge to sign more outbound transfers than it has collateral for, breaking bridge collateralization.

**Incorrect fee distribution:** The fee is minted/transferred to the fee recipient even though the underlying transfer failed, constituting a misdirection of value.

### Likelihood Explanation

The trigger condition is a non-empty `message` field in `initTransfer` combined with a recipient NEAR contract whose `ft_on_transfer` panics. This is reachable by any EVM user:

- Any user calling `initTransfer` with a non-empty `message` (a documented feature for cross-chain contract calls) is exposed.
- The recipient contract may panic due to: a bug, being paused, rejecting an unexpected message format, or being a contract that does not implement `ft_on_transfer` at all.
- An attacker can deliberately craft a transfer to a contract they control that always panics, freezing the transferred funds and corrupting accounting.

No privileged role is required. The EVM `initTransfer` is a public, payable function.

### Recommendation

In `is_refund_required`, treat a promise `Err` (panic) the same as a returned-zero result — i.e., require a refund:

```rust
Err(_) => true,  // ft_on_transfer panicked → tokens returned → refund required
```

Alternatively, restructure `fin_transfer_send_tokens_callback` to explicitly check `env::promise_result_checked` and always call `revert_lock_actions` + `remove_fin_transfer` when the token transfer did not succeed (either panic or returned-zero). Apply the same fix to `resolve_fast_transfer` to avoid burning tokens on a failed promise.

### Proof of Concept

1. On EVM, call `initTransfer(tokenAddress, amount, fee, 0, "near:panic-contract.near", "trigger_panic")`. The `message` field is non-empty.
2. The NEAR relayer calls `fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_near`.
3. Inside `process_fin_transfer_to_near`: `unlock_tokens_if_needed` decrements `locked_tokens[Eth][token]` by `amount`; `add_fin_transfer` stores the transfer ID.
4. `send_tokens` calls `ft_transfer_call(panic-contract.near, amount, "trigger_panic")`.
5. `panic-contract.near::ft_on_transfer` calls `env::panic_str(...)`. The NEAR runtime rolls back the token transfer; tokens return to the bridge.
6. `fin_transfer_send_tokens_callback` is invoked. `is_refund_required(true)` checks `env::promise_result_checked(0, ...)` → `Err` → returns `false`.
7. The callback takes the `else` branch: mints/transfers `fee` to the fee recipient, logs `FinTransferEvent`.
8. Result: `locked_tokens` is permanently decremented (collateral accounting broken), the `fin_transfer` record is stuck, the recipient has no tokens, and the fee was incorrectly paid out.
