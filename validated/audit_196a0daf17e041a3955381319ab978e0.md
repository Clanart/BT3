### Title
EIP-712 `timeout_height` Field Excluded from Type Schema Allows Silent Mutation Without Signature Invalidation — (`ethereum/eip712/types.go`, `ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

---

### Summary

The `Tx` type schema used in both the legacy and new EIP-712 paths deliberately omits `timeout_height`. However, `StdSignBytes` (called during signature verification) includes `timeout_height` in the JSON message payload when it is non-zero. Because EIP-712 only hashes fields declared in the type schema, `timeout_height` is silently dropped from the hash. An attacker who intercepts a signed tx can freely mutate `timeout_height` — in either direction — without invalidating the signature.

---

### Finding Description

**Root cause — schema omission (both paths):**

In `createEIP712Types`, the `Tx` schema is:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
},
``` [1](#0-0) 

The identical omission exists in the legacy path's `extractMsgTypes`: [2](#0-1) 

**`timeout_height` IS present in the JSON payload:**

In `VerifySignature`, `txBytes` is produced by calling `StdSignBytes` with the **current tx's** `timeout_height`:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← live value from the submitted tx
    legacytx.StdFee{...},
    msgs, tx.GetMemo(),
)
``` [3](#0-2) 

The Cosmos SDK's `StdSignDoc` uses `json:"timeout_height,omitempty"`, so the field appears in the JSON when non-zero. This `txBytes` is then passed directly to `LegacyWrapTxToTypedData`: [4](#0-3) 

`LegacyWrapTxToTypedData` unmarshals `txBytes` into `txData` (the EIP-712 message), but the type schema it builds via `extractMsgTypes` does not include `timeout_height`. Per EIP-712, `TypedDataAndHash` only encodes fields declared in the type schema — any extra fields in the message map are silently ignored.

**The new (non-legacy) Protobuf path has a guard, but the Amino path does not:**

`decodeProtobufSignDoc` explicitly rejects `TimeoutHeight != 0`: [5](#0-4) 

However, `decodeAminoSignDoc` has no such guard and passes `signDocBytes` (which may contain `timeout_height`) directly to `WrapTxToTypedData`: [6](#0-5) 

The active ante handler path (`VerifySignature` → `LegacyWrapTxToTypedData`) has no guard at all.

---

### Impact Explanation

An attacker who obtains a signed EIP-712 Cosmos tx (e.g., as a fee payer or relayer) can:

1. **Strip timeout protection:** User signs with `timeout_height=500` intending the tx to expire. Attacker sets `timeout_height=0` in the tx body. The Cosmos SDK's `TxTimeoutHeightDecorator` sees `0` and skips expiry enforcement. The EIP-712 hash is identical to the original (field absent from schema in both cases). Signature verifies. Tx executes after block 500.

2. **Impose a timeout:** User signs with `timeout_height=0`. Attacker sets `timeout_height=1` to force expiry at block 1 (DoS).

The first scenario is the security-relevant one: the user's explicit expiry intent is nullified. This is an **EIP-712 authorization bypass** — a field the user included in the signed payload is not committed to in the hash, enabling silent field mutation.

---

### Likelihood Explanation

- The legacy EIP-712 ante path is still active production code (marked deprecated but not removed).
- Fee payer / relayer patterns are the primary use case for this path — the fee payer is explicitly a third party who holds the signed tx bytes.
- No special privileges are required; any party who receives a signed tx can perform the mutation.
- The sequence number prevents true replay (the tx can only execute once), but the timeout bypass is still meaningful: a user who signed a tx intending it to expire cannot rely on that protection.

---

### Recommendation

1. **Re-add `timeout_height` to both `Tx` schemas.** The comment "not getting filled with the legacyTx" reflects a past implementation gap, not a reason to exclude it from the hash. If `timeout_height=0` is the default, it should still appear in the schema (as `"string"`) and the JSON should always emit it (remove `omitempty` from `StdSignDoc.TimeoutHeight`, or normalize to `"0"` before hashing).

2. **Mirror the Protobuf guard in the Amino path:** Reject txs with non-zero `timeout_height` in `decodeAminoSignDoc` until the schema is fixed, consistent with the guard already present in `decodeProtobufSignDoc`. [5](#0-4)

payload100 := []byte(`{"account_number":"1","chain_id":"ethermint_9000-1","fee":{"amount":[{"amount":"1000","denom":"aphoton"}],"gas":"21000"},"memo":"","msgs":[{"type":"cosmos-sdk/MsgSend","value":{"amount":[{"amount":"1","denom":"aphoton"}],"from_address":"cosmos1...","to_address":"cosmos1..."}}],"sequence":"0","timeout_height":"100"}`)

td0,   _ := WrapTxToTypedData(9000, payload0)
td100, _ := WrapTxToTypedData(9000, payload100)

hash0,   _, _ := apitypes.TypedDataAndHash(td0)
hash100, _, _ := apitypes.TypedDataAndHash(td100)

The hashes are identical because `timeout_height` is absent from the `Tx` type schema in `createEIP712Types`, so `TypedDataAndHash` ignores it regardless of its presence in the message map. [1](#0-0) [7](#0-6)

func WrapTxToTypedData(
	chainID uint64,
	data []byte,
) (apitypes.TypedData, error) {
	messagePayload, err := createEIP712MessagePayload(data)
	message := messagePayload.message
	if err != nil {
		return apitypes.TypedData{}, err
	}

	types, err := createEIP712Types(messagePayload)
	if err != nil {
		return apitypes.TypedData{}, err
	}

	value, err := ethermint.SafeInt64(chainID)
	if err != nil {
		return apitypes.TypedData{}, err
	}
	domain := createEIP712Domain(value)

	typedData := apitypes.TypedData{
		Types:       types,
		PrimaryType: txField,
		Domain:      domain,
		Message:     message,
	}

	return typedData, nil
