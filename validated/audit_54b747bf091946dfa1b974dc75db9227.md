Audit Report

## Title
`_fillBids` subtracts the length of bin `b+1` instead of bin `b` when computing `walkDistE6`, corrupting bid-side price ranges for all bins below `curBinIdx` when bins have unequal lengths — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

In `_fillBids`, the downward walk loop decrements `walkDistE6` by `lenAbove` — the `lengthE6` of bin `b+1` (the bin just exited) — rather than the `lengthE6` of bin `b` (the bin being entered). This produces wrong `lowerX64`/`upperX64` for every bid bin below `curBinIdx` whenever adjacent bins have different lengths, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` in the returned `DepthLevel` array. Integrators using these inflated bid prices as `priceLimitX64` will receive less token1 than expected or have swaps revert.

## Finding Description

At [1](#0-0)  the loop reads:

```solidity
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
```

`lenAbove` is the length of bin `b+1`, not bin `b`. The correct lower edge of bin `b` is `lower_edge(b+1) - length(b)`, but the code computes `lower_edge(b+1) - length(b+1)`. The actual `lengthE6` of bin `b` is fetched two lines later at [2](#0-1)  but is only used for the `upperX64` offset, not for the `walkDistE6` step.

Because `walkDistE6` is wrong, both `lowerX64` and `upperX64` passed to `_accumulateBidLevel` are wrong for every bin below `curBinIdx`, as seen at [3](#0-2) .

The pool's own swap engine does this correctly: after decrementing `curBinIdxCache`, it subtracts `binState.lengthE6` of the **newly entered** bin at [4](#0-3) .

The ask-side lens also does it correctly, advancing `ctx.cumDistE6` by the current bin's own `lengthE6` at [5](#0-4) .

## Impact Explanation

`getLiquidityDepth` is the primary off-chain read path for bid-side depth. Integrators (aggregators, market-makers, UIs) read `bids[i].binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` to decide whether to submit a sell order and at what `priceLimitX64`. When bins have unequal lengths, the reported bid prices diverge from the prices the pool actually executes at. A seller who reads an inflated bid price and submits a sell order with a matching `priceLimitX64` will either receive less token1 than expected (the pool executes at the true, lower price) or have the swap revert if the true execution price falls below their limit. This matches the "bad-price execution" and "lens/quoter path" impact gates: quoted amounts must match actual pool execution closely enough that integrations are not induced into fund-losing trades.

## Likelihood Explanation

Any pool whose bins were configured with non-uniform `lengthE6` values triggers the bug. The factory only validates that cumulative distances stay within `(-1e6, 1e6)` at [6](#0-5)  — it imposes no constraint that all bins share the same length. A pool with a narrow current bin and wider outer bins (a common liquidity-concentration pattern) will exhibit the error on every call to `getLiquidityDepth`. No privileged access is required; any public pool creation with non-uniform bin lengths suffices.

## Recommendation

Replace the `lenAbove` read with the length of the current bin `b`, which is already fetched at line 500. The corrected loop body:

```solidity
for (int256 b = int256(curBinIdx) - 1; b >= int256(lowCap); b--) {
    int8 binIdx = int8(b);
    (, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) =
        PoolStateLibrary._binState(pool, binIdx);

    walkDistE6 -= int256(uint256(lengthE6));   // subtract length of bin b, not b+1

    uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(uint256(addFeeSellE6), Q64, ONE_E6);
    uint256 lowerX64   = _priceFromMidAndDistE6(midPriceX64, walkDistE6, Math.Rounding.Floor);
    uint256 upperX64   = _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), Math.Rounding.Floor);
    // ... rest unchanged
}
```

This eliminates the redundant `_binState(pool, int8(b + 1))` call and mirrors the pattern used by `_fillAskRow` and the pool's own swap engine.

## Proof of Concept

1. Deploy a pool with three bins: `curBinIdx = 0` with `lengthE6 = 10_000`, bin `-1` with `lengthE6 = 20_000`, bin `-2` with `lengthE6 = 5_000`.
2. Call `getLiquidityDepth(pool, 3)`.
3. For bin `-1`: the code subtracts `length(0) = 10_000` → `walkDistE6 = curBinDist - 10_000`. The correct lower edge is `curBinDist - 20_000`. The reported `lowerX64`/`upperX64` are shifted by `+10_000 E6` units, inflating the reported bid price.
4. For bin `-2`: the code subtracts `length(-1) = 20_000` → `walkDistE6 = curBinDist - 30_000`. The correct lower edge is `curBinDist - 25_000`. The reported price is shifted by `+5_000 E6` units.
5. Manually compute expected prices from raw bin state and compare against `bids[1].binAvgExecPriceX64` and `bids[2].binAvgExecPriceX64` — they will not match.
6. Submit a sell order using the inflated `cumulativeAvgExecPriceX64` as `priceLimitX64`; the swap executes at the true (lower) price, delivering less token1 than the integrator expected.
