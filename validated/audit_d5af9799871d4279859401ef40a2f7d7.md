### Title
Unchecked Return Value of Fee Transfer in `charge_fee` Allows Silent Fee Evasion — (File: execution/transaction_impls.cairo)

---

### Summary

The `charge_fee` function in `transaction_impls.cairo` calls `non_reverting_select_execute_entry_point_func` to execute the ERC20 STRK fee transfer but silently discards the return value. If the fee transfer reverts — for example, because the user's account contract drained its STRK balance during `__execute__` — the OS does not detect the failure and produces a valid proof for a state in which no fee was collected.

---

### Finding Description

In `charge_fee` (lines 160–163 of `transaction_impls.cairo`):

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
```

The return tuple `(retdata_size, retdata, is_deprecated)` is never captured. This is the direct analog to the M-11 pattern: a call whose success indicator is silently dropped. [1](#0-0) 

Every other call site of `non_reverting_select_execute_entry_point_func` in the same file captures and validates the return value:

- **`execute_deploy_account_transaction`** (validate-deploy step, lines 677–683): captures `(retdata_size, retdata, is_deprecated)` and asserts `retdata[0] == VALIDATED`.
- **`execute_declare_transaction`** (validate-declare step, lines 804–811): same pattern. [2](#0-1) [3](#0-2) 

When the inner entry point reverts, `execute_entry_point` calls `handle_revert` to undo all state changes made by that call (lines 309–319 of `execute_entry_point.cairo`). This means the ERC20 token transfer is rolled back. Because `charge_fee` never inspects the return value, it cannot distinguish a successful transfer from a silently reverted one. [4](#0-3) 

The OS proof is therefore valid for a final state in which no fee was deducted from the user.

---

### Impact Explanation

**Direct loss of funds.** The sequencer/protocol does not collect the fees it is owed. Because the OS proof does not enforce that the fee transfer succeeded, a valid STARK proof can be produced for a block in which one or more transactions were executed without any fee being paid. Any L1 verifier or downstream consumer that trusts the OS proof to reflect correct fee accounting is deceived.

---

### Likelihood Explanation

An unprivileged transaction sender controls their own account contract. They can craft an `__execute__` implementation that, as part of normal execution, transfers all STRK out of the account to a second address they control. The sequencer validates the STRK balance *before* execution; after execution the balance is zero. When `charge_fee` subsequently calls the ERC20 `transfer`, the call reverts due to insufficient balance. Because the return value is unchecked, the OS does not flag this as an error. The proof is accepted as valid. The attacker recovers the STRK from the second address and has paid nothing.
