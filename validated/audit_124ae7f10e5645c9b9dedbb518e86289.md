Audit Report

## Title
Unverified Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User-Approved Tokens - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address in `addLiquidityExactShares` and `addLiquidityWeighted` without verifying it against the factory registry. A victim induced to call either entry point with a malicious pool address will have tokens pulled from their wallet up to the `maxAmountToken0`/`maxAmountToken1` caps they supply, because the callback's only caller check compares `msg.sender` against the attacker-controlled address that was stored in transient storage.

## Finding Description
`addLiquidityExactShares` (L56–68) and `addLiquidityWeighted` (L88–116) accept a caller-supplied `pool` with no factory validation and forward it directly to `_addLiquidity`.

`_addLiquidity` (L193) calls `_setPayContext(pool, payer, max0, max1)`, storing the unverified address as the trusted callback counterparty in transient storage, then calls `IMetricOmmPoolActions(pool).addLiquidity(...)` (L194–195).

Inside `metricOmmModifyLiquidityCallback` (L162–164), the guard is:
```solidity
(address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
if (msg.sender != expectedPool) revert InvalidCallbackCaller(...);
```
Because `expectedPool` is the attacker-controlled address, the check trivially passes when the malicious pool re-enters as `msg.sender`.

The callback then queries `IMetricOmmPool(msg.sender).getImmutables()` (L169) — again from the attacker-controlled contract — to obtain `token0`/`token1`, and executes `pay(token0, payer, msg.sender, amount0Delta)` (L173), pulling tokens from the victim to the malicious pool.

The contract's own NatSpec (L19–21) explicitly acknowledges this gap:
> "This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."

By contrast, `MetricOmmSwapRouterBase._requireExpectedCallbackCaller` (L82–85) enforces `FACTORY.isPool(caller)` in addition to the transient-context check, and `_setNextCallbackContext` (L29–32) calls `_requireFactoryPool(pool)` before storing any pool in transient storage. `MetricOmmPoolLiquidityAdder` has no equivalent guard.

## Impact Explanation
A victim who has approved `MetricOmmPoolLiquidityAdder` and is induced to call it with a malicious pool address loses up to `maxAmountToken0 + maxAmountToken1` of any ERC-20 token (including USDC/USDT/WETH) in a single transaction. This is a direct loss of user principal, satisfying the High-impact threshold.

## Likelihood Explanation
The victim must be the transaction sender (since `payer = msg.sender`), so the attacker must induce the victim to call the adder with the malicious pool address — achievable via a phishing UI or malicious front-end. No privileged role is required; deploying a fake pool contract is permissionless and the factory is never consulted. Likelihood is Medium; combined severity is Medium.

## Recommendation
Before storing `pool` in transient context, verify it was deployed by the canonical factory, mirroring the pattern in `MetricOmmSwapRouterBase`:

```solidity
// In _addLiquidity (or at each public entry point):
if (!IMetricOmmPoolFactory(factory).isPool(pool)) revert UnknownPool(pool);
```

`MetricOmmPoolLiquidityAdder` would need to accept and store a `factory` immutable (as `MetricOmmSwapRouterBase` does at L20–25) and call this check before `_setPayContext`.

## Proof of Concept
1. Victim approves `MetricOmmPoolLiquidityAdder` for 1000 USDC.
2. Attacker deploys `MaliciousPool` whose `addLiquidity` re-enters `metricOmmModifyLiquidityCallback(1000e6, 0, abi.encode(uint8(1)))` and whose `getImmutables()` returns USDC as `token0`.
3. Victim is phished into calling:
   ```solidity
   adder.addLiquidityExactShares(
       address(maliciousPool),
       victim,   // owner (victim is msg.sender → payer)
       0, deltas,
       1000e6,   // maxAmountToken0
       0, ""
   );
   ```
4. `_addLiquidity` stores `maliciousPool` as `expectedPool` and calls `maliciousPool.addLiquidity(...)`.
5. `MaliciousPool` fires the callback; `msg.sender == expectedPool` passes; callback reads USDC from `getImmutables()` and executes `pay(USDC, victim, maliciousPool, 1000e6)`.
6. 1000 USDC is transferred from victim to `MaliciousPool` in one transaction.
