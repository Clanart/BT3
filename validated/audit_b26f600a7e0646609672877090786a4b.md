### Title
Permanent Irrecoverable Lock of Source-Chain Tokens When EVM `finTransfer` Recipient Is Blacklisted — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

When a user bridges a native ERC20 token (e.g., USDC) from NEAR to EVM, the source-chain tokens are burned/locked on NEAR before the EVM-side `finTransfer` is executed. If the recipient address becomes blacklisted in the EVM token contract (e.g., USDC's `notBlacklisted` modifier) after the transfer is initiated but before `finTransfer` is called, every `finTransfer` attempt reverts. Because the signed payload encodes the recipient immutably and no recovery path exists on NEAR, the user's tokens are permanently and irrecoverably locked.

---

### Finding Description

**Root cause — `OmniBridge.sol::finTransfer`**

`finTransfer` marks the destination nonce as used before executing the token transfer:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC20(payload.tokenAddress).safeTransfer(             // line 351
    payload.recipient,
    payload.amount
);
```

If `safeTransfer` reverts (e.g., USDC's `notBlacklisted` modifier fires), the entire EVM transaction reverts, including the nonce-marking. The relayer can retry, but every retry will revert for the same reason. The nonce is never consumed, yet the transfer can never succeed. [1](#0-0) [2](#0-1) 

**Why source-chain funds are irrecoverable**

On the NEAR side, `init_transfer_internal` burns (for deployed tokens) or locks (for native tokens) the user's funds atomically with storing the transfer message:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(...);
``` [3](#0-2) 

After `sign_transfer` is called, `sign_transfer_callback` removes the pending transfer message from storage (when `fee.is_zero()`):

```rust
if fee.is_zero() {
    self.remove_transfer_message(message_payload.transfer_id);
}
``` [4](#0-3) 

Once the transfer message is removed, there is no NEAR-side function to cancel the transfer, change the recipient, or reclaim the burned/locked tokens. The MPC signature is over a Borsh-encoded `TransferMessagePayload` that includes the recipient address — it cannot be reused with a different recipient. [5](#0-4) 

There is no `withdrawBlacklistedFunds` or equivalent recovery function anywhere in the EVM or NEAR bridge contracts.

**Same issue exists for ETH delivery:**

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
```

If the recipient is a contract that rejects ETH (e.g., no `receive` function), the same permanent lock occurs. [6](#0-5) 

**Same issue exists on StarkNet `fin_transfer`:**

The StarkNet bridge marks the nonce used before the transfer and asserts success:

```cairo
_set_transfer_finalised(ref self, payload.destination_nonce);
// ...
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');
```

A StarkNet panic reverts the nonce marking, but the source-chain funds are already gone with no recovery path. [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

The user's tokens are burned or locked on NEAR the moment `init_transfer_internal` executes. If the EVM recipient is subsequently blacklisted, every `finTransfer` call reverts. Because:
- The signed payload encodes the recipient immutably (MPC signature cannot be reused for a different address),
- The NEAR pending transfer is removed after signing,
- No cancel/recover function exists,

the funds are permanently and irrecoverably lost. This matches the "Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds" impact class.

---

### Likelihood Explanation

**Medium.** USDC (Circle) actively maintains a blacklist and has blacklisted addresses in the past. The window between a user initiating a NEAR→EVM transfer and the relayer submitting `finTransfer` includes MPC signing latency and relayer processing time — during which a recipient can be blacklisted. The scenario is realistic for any bridge user sending to a third-party address (e.g., "bridge to Bob's address"). The same risk applies to any ERC20 with transfer restrictions (e.g., USDT, regulated tokens).

---

### Recommendation

1. **Try/catch pattern on EVM**: Wrap the token delivery in a try/catch. On failure, store the amount in a per-recipient mapping (`blacklistedFunds[recipient][token] += amount`) and emit a `TransferFailed` event. Add a `withdrawBlacklistedFunds(address token)` function allowing the recipient (once removed from the blacklist) to claim their funds.

2. **Consume the nonce on failure**: Ensure the destination nonce is marked used even when delivery fails, so the failed transfer is recorded and cannot be replayed once a recovery path is added.

3. **NEAR-side cancellation**: Add a `cancel_transfer` function on NEAR that can be triggered (e.g., by the original sender after a timeout) when the EVM side has permanently failed, allowing the burned/locked tokens to be restored.

---

### Proof of Concept

**Scenario (NEAR → EVM, USDC):**

1. Alice holds 10,000 USDC (bridged) on NEAR and calls `ft_transfer_call` to bridge to Bob's EVM address.
2. `init_transfer_internal` burns Alice's USDC on NEAR and stores the transfer message. [8](#0-7) 
3. A trusted relayer calls `sign_transfer`; MPC signs the `TransferMessagePayload` encoding Bob's address. `sign_transfer_callback` removes the pending transfer (fee = 0). [9](#0-8) 
4. Before the relayer submits `finTransfer` on EVM, Circle blacklists Bob's address in the USDC contract.
5. Relayer calls `OmniBridge.finTransfer(signature, payload)` on EVM. `safeTransfer(Bob, amount)` reverts due to USDC's `notBlacklisted` modifier. The entire EVM transaction reverts. [2](#0-1) 
6. Every subsequent `finTransfer` attempt reverts identically. The nonce is never consumed.
7. Alice's 10,000 USDC are permanently burned on NEAR. No recovery function exists. Funds are irrecoverably lost.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
