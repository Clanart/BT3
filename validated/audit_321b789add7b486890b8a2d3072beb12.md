### Title
`batch_insert` Bypasses Duplicate-Key/Hash Guards, Corrupting DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::insert` enforces two uniqueness invariants before writing a leaf. `MerkleBlob::batch_insert` enforces those invariants only for the first two items in the batch; all subsequent items are written directly to the blob without any duplicate-key or duplicate-hash check. An unprivileged caller supplying a crafted batch can silently insert a key that already exists in the tree (or a key that appears twice within the same batch), producing a structurally invalid tree whose root hash no longer faithfully represents the committed key-value set, and from which forged inclusion proofs can be derived.

---

### Finding Description

`MerkleBlob::insert` guards against two invariants before writing:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When the tree already has more than one leaf, the first two items from the incoming batch are routed through `self.insert()` (which runs the checks), but every subsequent item is written directly via `insert_entry_to_blob` with no guard at all:

```rust
// lines 578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← checked
    }
}

for ((key, value), hash) in keys_values_hashes {   // ← ALL remaining items
    let new_leaf_index = self.get_new_index();
    let new_block = Block { node: Node::Leaf(LeafNode { hash, key, value, ... }), ... };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // ← NO duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

This is the exact structural analog of the external report: `insert` is the "safe" registration path (like `registerWallet` with its `isWallet` guard), while `batch_insert`'s fast path is the "unsafe" path (like `registerSubAccount` missing the `isWallet` check). An entity — here a `KeyId` — can therefore occupy two roles simultaneously: it is already a committed leaf in the tree **and** it is silently inserted again as a second leaf, violating the set-membership invariant the Merkle tree is supposed to enforce.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
// lines 1503-1518
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2) 

---

### Impact Explanation

After a duplicate-key batch insert the blob contains two leaf nodes with the same `KeyId`. The `block_status_cache` (a `HashMap<KeyId, TreeIndex>`) will silently overwrite the first entry with the second, so the cache points only to the second leaf, but the first leaf remains in the blob and is wired into the internal-node structure. The root hash computed by `calculate_lazy_hashes` therefore reflects a tree that contains a phantom leaf the cache no longer tracks. Consequences:

1. **Corrupted tree root** — the committed on-chain root hash no longer faithfully represents the actual key-value set.
2. **Forged inclusion proof** — `get_proof_of_inclusion` walks the cache-tracked path to the second leaf and produces a `ProofOfInclusion` whose `valid()` check passes (all intermediate hashes chain correctly), yet the proof attests to a value that was silently overwritten, or to a key that should not be present.
3. **Cross-node state divergence** — two DataLayer nodes that built their trees via different code paths (`insert` vs `batch_insert`) will compute different roots for the same logical dataset, causing consensus divergence on the committed root.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is the primary bulk-loading API and is called by the Python layer for every DataLayer store update involving multiple records. Any caller who supplies a batch containing a key already present in the tree (or a key repeated within the batch) triggers the bug silently — no error is returned. Because the Python binding accepts arbitrary caller-supplied `(KeyId, ValueId, Hash)` tuples, the entry path is fully attacker-controlled and requires no special privilege.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the fast path inside `batch_insert`, before calling `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, pre-validate the entire incoming batch for internal duplicates and duplicates against the existing tree before writing any node, so the tree is never left in a partially-written corrupt state on error.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(bytearray())

# Seed the tree with 2 leaves so leaf_count > 1 before the batch
blob.insert(KeyId(10), ValueId(10), h(10))
blob.insert(KeyId(20), ValueId(20), h(20))

# batch_insert: first two items go through insert() (checked),
# third item bypasses all checks — duplicate key 10 with a different value
blob.batch_insert(
    [(KeyId(30), ValueId(30)), (KeyId(40), ValueId(40)), (KeyId(10), ValueId(999))],
    [h(30), h(40), h(999)],   # different hash, same key
)

blob.calculate_lazy_hashes()

# The tree now has two leaf nodes for KeyId(10).
# get_proof_of_inclusion returns a proof for the *second* leaf (value=999),
# but the root hash was computed over a tree that also contains the first leaf (value=10).
# The proof passes valid() yet attests to corrupted state.
proof = blob.get_proof_of_inclusion(KeyId(10))
assert proof.valid()   # passes — forged proof accepted
