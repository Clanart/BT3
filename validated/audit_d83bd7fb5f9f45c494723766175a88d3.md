### Title
Silent Payout Failure Permanently Locks Relayer Funds After Fast Transfer Finalization â€” (File: `near/omni-bridge/src/lib.rs`)

### Summary

In `process_fin_transfer_to_other_chain`, when a fast-transfer relayer is owed repayment upon finalization of the original cross-chain transfer, the `send_tokens` call is fired with `.detach()`. If that token transfer fails, the fast-transfer record is still permanently marked as finalised and the origin transfer is recorded in `finalised_transfers`. The relayer irrecoverably loses the full amount they fronted, with no retry path.

### Finding Description

`process_fin_transfer_to_other_chain` is invoked from `fin_transfer_callback` when the destination of a finalized transfer is not NEAR. When a fast transfer was previously executed for the same transfer ID, the bridge owes the relayer the amount they fronted. The repayment sequence is: [1](#0-0) 

```rust
let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id()); // (1) permanently marks transfer finalised
``` [2](#0-1) 

```rust
if let Some(relayer) = recipient {
    self.send_tokens(token, relayer, U128(...), "")
        .detach();                                    // (2) fire-and-forget â€” result never checked
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // (3) permanently marks fast transfer finalised
}
```

Step (1) inserts the transfer ID into `finalised_transfers` â€” re-submission of the same proof will panic with `TransferAlreadyFinalised`. Step (3) sets `status.finalised = true` in `fast_transfers` â€” any subsequent attempt to process the fast transfer will panic with `FastTransferAlreadyFinalised`. Between steps (1) and (3), the `send_tokens` promise is detached: its success or failure is never observed.

`send_tokens` for a non-deployed (locked) token calls `ft_transfer`, which requires the recipient to hold a registered storage account for that token. [3](#0-2) 

If the relayer's storage account for the token does not exist or was closed, `ft_transfer` panics inside the token contract, the promise fails, but the bridge contract never sees the failure because `.detach()` was used. Both state records are already committed.

The identical pattern exists in `utxo_fin_transfer_fast`, where `remove_fast_transfer` or `mark_fast_transfer_as_finalised` is called before `send_tokens(...).detach()`: [4](#0-3) 

A secondary instance occurs in `fin_transfer_send_tokens_callback`, where fee payments to the relayer are also detached after the transfer is already finalised: [5](#0-4) 

### Impact Explanation

A relayer who fronted the full transfer amount for a fast transfer permanently loses those tokens. The bridge has already consumed the proof (transfer is in `finalised_transfers`) and closed the fast-transfer record (`finalised = true`). There is no admin function, no retry entry point, and no refund path. This constitutes **permanent, irrecoverable lock of relayer funds** â€” matching the allowed Critical impact class.

### Likelihood Explanation

`ft_transfer` on a NEP-141 token panics if the recipient has no registered storage account. A relayer operating across many tokens may not have pre-registered storage for every token they relay. Additionally, a malicious user who knows a relayer is about to be repaid can front-run by calling `storage_unregister` on the token contract for the relayer's account (if the relayer's balance is zero at that moment), causing the repayment to fail silently. The condition is reachable in normal operation without any privileged access.

### Recommendation

Replace `.detach()` with a proper callback that checks the transfer result. If `send_tokens` fails, the fast-transfer record should be kept open (or the `finalised` flag should not be set) so the repayment can be retried. The state mutation ordering should follow the checks-effects-interactions pattern: only commit the finalised state after confirming the payout succeeded.

### Proof of Concept

1. Relayer R executes a fast transfer for transfer ID T (origin chain â†’ NEAR â†’ destination chain). R fronts 1000 USDC to the recipient on NEAR.
2. R does **not** hold a storage account for USDC on the bridge's token contract (or closes it before step 3).
3. Anyone (including an unprivileged user) calls `fin_transfer` with a valid proof for transfer T.
4. `fin_transfer_callback` â†’ `process_fin_transfer_to_other_chain` runs:
   - `add_fin_transfer(T)` â†’ T is inserted into `finalised_transfers`.
   - `send_tokens(USDC, R, 1000, "").detach()` â†’ `ft_transfer` panics inside the USDC contract because R has no storage; the promise fails silently.
   - `mark_fast_transfer_as_finalised(fast_transfer_id)` â†’ fast transfer record is permanently closed.
5. R attempts to retry: any re-submission of the proof panics with `TransferAlreadyFinalised`. Any attempt to re-process the fast transfer panics with `FastTransferAlreadyFinalised`.
6. R has permanently lost 1000 USDC with no recourse.
