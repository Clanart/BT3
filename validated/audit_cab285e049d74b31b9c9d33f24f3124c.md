### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the embedded chunk's actual shard matches the caller-supplied `shard_id`. Every individual validation step passes for a cross-shard substitution, so a connected peer advertising as a snapshot host can cause a syncing node to install shard A's trie state under shard B's `ShardUId`.

---

### Finding Description

**Entrypoint and missing guard**

`set_state_header` receives three inputs: `shard_id` (the shard being synced), `sync_hash`, and `shard_state_header` (peer-supplied). The function extracts `chunk = shard_state_header.cloned_chunk()` and runs five validation groups. None of them check `chunk.shard_id() == shard_id`. [1](#0-0) 

**Check 1 — `validate_chunk_proofs`**: verifies the chunk's internal hash, tx root, and outgoing-receipts root. It does not inspect `chunk.shard_id()` at all. [2](#0-1) 

**Check 2 — `verify_path` for chunk inclusion**: proves `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in `sync_prev_block_header.chunk_headers_root()`. The Merkle tree contains every shard's chunk header. A valid proof for shard-0's chunk verifies against the same root even when the caller claims `shard_id=1`; the proof encodes only the path direction, not which shard index the leaf occupies. [3](#0-2) 

**Check 3 — prev_chunk inclusion**: same structure, same absence of shard-id binding. [4](#0-3) 

**Check 4 — incoming receipts**: the receipts hash is computed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the *caller-supplied* `shard_id` (line 488). Critically, `to_shard_id` inside `ShardProof` is explicitly ignored with `_` at line 478. An attacker can supply the legitimate on-chain incoming receipts for the *claimed* shard (shard 1), which are publicly available. Those receipts hash correctly under `shard_id=1` and their Merkle proofs against `prev_chunk_outgoing_receipts_root` verify. The terminal check at line 507 (`header.height() != prev_chunk_header.height_included()`) passes whenever both shards have the same `height_included` for their previous chunks — the common case when both shards produce chunks at every block. [5](#0-4) 

**Check 5 — `validate_state_root_node`**: checks the node's hash against `chunk_inner.prev_state_root()`. No shard-id binding. [6](#0-5) 

**Storage**: the header is written under `StateHeaderKey(shard_id, sync_hash)` where `shard_id` is the attacker-controlled envelope value, not the chunk's actual shard. [7](#0-6) 

**Network delivery path**

A peer sends `PeerMessage::VersionedStateResponse(StateResponseInfo::V2 { shard_id: 1, sync_hash, state_response: <shard-0 chunk> })`. The receiving node routes it via `StateResponseReceived`, matches it to the pending request by `PendingPeerRequestKey { shard_id: 1, sync_hash, part_id_or_header: Header }`, and forwards the inner `ShardStateSyncResponse` to `set_state_header(shard_id=1, sync_hash, header)`. The peer-id check at line 97 only verifies the response came from the expected peer — it does not validate the embedded chunk's shard. [8](#0-7) 

**Downstream corruption**

`set_state_part(shard_id=1, ...)` reads the stored header (shard-0's chunk), extracts shard-0's `prev_state_root`, and validates incoming parts against it. Shard-0's parts pass `validate_state_part(shard_id=1, state_root_of_shard_0, ...)` because the cryptographic check is purely hash-based. [9](#0-8) 

`set_state_finalize` then computes `shard_uid` for shard 1 (line 513–514) but calls `apply_chunk` with `chunk_header.prev_state_root()` from shard-0's chunk, installing shard-0's trie nodes and producing a `ChunkExtra` for shard 1 whose `state_root` is shard-0's root. [10](#0-9) 

---

### Impact Explanation

After a successful substitution, the syncing node's shard-1 storage contains shard-0's trie data. The `chunk_extra.state_root` recorded for shard 1 is shard-0's root. Any subsequent block application for shard 1 will fail or produce wrong results, and the node will diverge from the canonical chain. The corrupted state is durable (written to RocksDB via `store_update.commit()`).

---

### Likelihood Explanation

The attacker must be a connected peer selected as the state sync provider. Any node can advertise itself as a snapshot host by signing a `SnapshotHostInfo` with its own peer key — no validator credentials are required. The attack is constructible entirely from public on-chain data (the shard-0 chunk, its Merkle proof, and shard-1's incoming receipts). It succeeds in the common case where both shards produce chunks at every block (equal `height_included` for their previous chunks). The attack does not require any cryptographic break.

---

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be inserted after line 376 (`let chunk = shard_state_header.cloned_chunk();`) and before any other validation. [11](#0-10)

### Proof of Concept

```
Setup: two-shard network; node B is syncing shard 1.

Attacker (connected peer, advertised as snapshot host):
  1. Obtain shard-0's chunk C0 for sync_prev_block (valid, signed by chunk producer).
  2. Obtain the Merkle proof P0 for C0 in sync_prev_block.chunk_headers_root().
  3. Obtain shard-0's prev_chunk PC0 and its Merkle proof PP0.
  4. Obtain the legitimate incoming receipts for shard 1 from sync_hash back to
     PC0.height_included() (public on-chain data).
  5. Obtain shard-0's state_root_node (public).
  6. Construct ShardStateSyncResponseHeader::V2 {
         chunk: C0, chunk_proof: P0,
         prev_chunk_header: PC0, prev_chunk_proof: PP0,
         incoming_receipts_proofs: <shard-1 receipts>,
         root_proofs: <matching proofs>,
         state_root_node: <shard-0 node>,
     }
  7. Send VersionedStateResponse { shard_id: 1, sync_hash, state_response: above }.

Node B:
  - set_state_header(shard_id=1, sync_hash, header) → all 5 checks pass → stored
    under StateHeaderKey(shard_id=1, sync_hash).
  - set_state_part(shard_id=1, ...) → validates shard-0 parts against shard-0 root → OK.
  - apply_state_part(shard_id=1, state_root_of_shard_0, ...) → installs shard-0 trie.
  - set_state_finalize(shard_id=1, ...) → chunk_extra for shard 1 has shard-0's state_root.

Assert: get_chunk_extra(sync_hash, shard_uid_1).state_root != expected_shard_1_state_root.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
