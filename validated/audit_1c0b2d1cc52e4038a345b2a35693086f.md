### Title
`deprecated_declared_classes` Silently Dropped in `CommitmentStateDiff → ThinStateDiff` Conversion Produces Wrong `state_diff_commitment` and Block Hash — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally hardcodes `deprecated_declared_classes: Vec::new()`. Because `BlockExecutionArtifacts::new` feeds this converted `ThinStateDiff` directly into `calculate_block_commitments` → `calculate_state_diff_hash`, any block that contains deprecated (Cairo 0) class declarations will produce a `state_diff_commitment` and a `state_diff_length` that silently omit those classes. Both values are chained into `calculate_block_hash`, so the resulting block hash is also wrong. This is the direct Sequencer analog of M-09: just as `transferFrom` skips the `onERC721Received` callback and silently bypasses required state-changing logic, this conversion skips the `deprecated_declared_classes` field and silently bypasses the required commitment step.

---

### Finding Description

**Root cause — the silent drop:**

```rust
// crates/blockifier/src/state/cached_state.rs  lines 690-701
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` has no `deprecated_declared_classes` field; the struct only carries `address_to_class_hash`, `address_to_nonce`, `storage_updates`, and `class_hash_to_compiled_class_hash`. [1](#0-0) 

**Commitment path that consumes the wrong `ThinStateDiff`:**

`BlockExecutionArtifacts::new` converts `commitment_state_diff` to `ThinStateDiff` and immediately passes it to `calculate_block_commitments`: [2](#0-1) 

`calculate_block_commitments` calls `calculate_state_diff_hash(&state_diff)` in a spawned task: [3](#0-2) 

`calculate_state_diff_hash` explicitly chains `deprecated_declared_classes` into the Poseidon hash: [4](#0-3) 

Because `deprecated_declared_classes` is always `[]`, `chain_deprecated_declared_classes` always chains `[0]` (count = 0, no hashes) regardless of how many Cairo 0 classes were actually declared in the block. [5](#0-4) 

**`state_diff_length` is also wrong:**

`ThinStateDiff::len()` counts `deprecated_declared_classes.len()` as part of the total: [6](#0-5) 

`calculate_block_commitments` uses `state_diff.len()` to build `concatenated_counts`, which is chained into the block hash: [7](#0-6) 

**Block hash is doubly corrupted:**

`calculate_block_hash` chains both `state_diff_commitment` and `concatenated_counts` (which encodes `state_diff_length`): [8](#0-7) 

Both values are wrong when any deprecated class declaration is present.

**The `verify_state_diff_hash` guard does not catch this:**

The committer's optional hash-verification check compares the provided `state_diff_commitment` against `calculate_state_diff_hash` applied to the same already-converted `ThinStateDiff` (also with empty `deprecated_declared_classes`). Both sides of the comparison use the same wrong value, so the check passes silently. [9](#0-8) 

---

### Impact Explanation

For any block that contains a deprecated (Cairo 0) class declaration:

1. `state_diff_commitment` stored in the block header omits the deprecated class hashes — the commitment does not represent the actual state diff.
2. `state_diff_length` in `concatenated_counts` is under-counted by the number of deprecated declared classes.
3. The final `block_hash` is wrong on both counts.
4. Any external verifier (proof system, SNOS, cross-chain bridge) that recomputes the state diff commitment or block hash from the actual state diff will disagree with the sequencer's value, causing proof verification failures or acceptance of a block whose commitment is inconsistent with its contents.

This matches the Critical scope item: **Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** — specifically a wrong authoritative `state_diff_commitment` and block hash produced for accepted transactions.

---

### Likelihood Explanation

The sequencer's `DeprecatedGatewayTransactionV3` only handles V3 (Sierra) declare transactions; there is no V1/V2 (Cairo 0) path in the current gateway. [10](#0-9)  However, the TODO comment in the conversion (`// TODO(AlonH): Remove this when the structure of storage diffs changes.`) confirms this is an acknowledged structural gap, not a deliberate permanent exclusion. [11](#0-10)  If any future gateway or RPC path admits a deprecated declare transaction, or if the sequencer is used in a context where Cairo 0 declarations are still live, the commitment invariant is silently broken with no error or warning.

---

### Recommendation

1. Add a `deprecated_declared_classes` field to `CommitmentStateDiff` (or to `StateMaps`) so that deprecated class declarations are tracked through the execution pipeline.
2. Populate it from the blockifier's state maps when a deprecated declare transaction is executed.
3. Remove the hardcoded `Vec::new()` in `From<CommitmentStateDiff> for ThinStateDiff` and propagate the actual value.
4. Add a test that executes a deprecated declare transaction and asserts that the resulting `state_diff_commitment` matches `calculate_state_diff_hash` applied to a `ThinStateDiff` with the correct non-empty `deprecated_declared_classes`.

---

### Proof of Concept

1. Submit a Cairo 0 (`DECLARE` V1) transaction through any admission path that reaches the blockifier.
2. The blockifier executes it; `StateMaps` does not record the deprecated class hash in any field consumed by `From<StateMaps> for CommitmentStateDiff`.
3. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())` → `deprecated_declared_classes = []`.
4. `calculate_state_diff_hash` hashes `[STARKNET_STATE_DIFF0, ..., 0 /* count=0 */, ...]` — the deprecated class hash is absent.
5. The `state_diff_commitment` written to the block header and chained into `calculate_block_hash` does not include the declared class.
6. An independent verifier that reads the stored `ThinStateDiff` (which also has `deprecated_declared_classes = []` for the same reason) and recomputes the hash will get the same wrong value — the discrepancy is invisible inside the node but will surface when a proof system or cross-chain verifier reconstructs the commitment from the full `StateDiff` (which does carry the deprecated class).

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))

fn chain_deprecated_declared_classes(
    deprecated_declared_classes: &[ClassHash],
    hash_chain: HashChain,
) -> HashChain {
    let mut sorted_deprecated_declared_classes = deprecated_declared_classes.to_vec();
    sorted_deprecated_declared_classes.sort_unstable();
    hash_chain
        .chain(&sorted_deprecated_declared_classes.len().into())
        .chain_iter(sorted_deprecated_declared_classes.iter().map(|class_hash| &class_hash.0))
}
