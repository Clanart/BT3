### Title
Insufficient Amount Validation in `init_transfer` Allows Permanent Lock of User Funds via `normalize_amount` Zero Truncation — (`File: near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR-side transfer to another chain, tokens are burned or locked in `init_transfer_internal` before any decimal normalization check occurs. If the transferred amount (minus fee) is smaller than the decimal scaling divisor (`10^(origin_decimals - decimals)`), `normalize_amount` returns zero, causing every subsequent `sign_transfer` call to permanently revert with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Because no cancel or refund path exists for pending transfers, the user's tokens are irrecoverably locked or burned.

### Finding Description

`normalize_amount` performs floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

In `sign_transfer`, this is applied to `amount_without_fee` and the result is required to be non-zero:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(amount_to_transfer > 0, BridgeError::InvalidAmountToTransfer.as_ref());
``` [2](#0-1) 

However, `init_transfer` only validates `fee.fee < amount` — it does **not** pre-check that `normalize_amount(amount - fee, decimals) > 0`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [3](#0-2) 

After this check passes, `init_transfer_internal` immediately burns (for deployed tokens) or locks (for native tokens) the full amount and stores the transfer message:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [4](#0-3) 

There is no public `cancel_transfer` or user-accessible refund function. `remove_transfer_message` is only called internally (e.g., in `claim_fee_callback` or `sign_transfer_callback` after a successful MPC signature). Since `sign_transfer` reverts before reaching the MPC call when `amount_to_transfer == 0`, the callback is never reached and the transfer message is never removed. [5](#0-4) 

### Impact Explanation

For any token where `origin_decimals > decimals` (e.g., a NEAR-native token with `origin_decimals = 24` bridged to EVM with `decimals = 6`, giving a divisor of `10^18`), any transfer where `amount_without_fee < 10^18` (i.e., less than 1 NEAR-unit) will normalize to zero. The user's tokens are burned or locked at `init_transfer` time and can never be recovered. This constitutes **permanent, irrecoverable loss of user funds** — matching the allowed impact: "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."

The code comment on `normalize_amount` acknowledges that *dust* (the sub-unit remainder after normalization) stays locked, but this is a distinct and more severe case: the **entire** `amount_without_fee` is consumed as "dust" with no recovery path. [6](#0-5) 

### Likelihood Explanation

Any unprivileged user can trigger this by calling `ft_transfer_call` with a small amount. For tokens with large decimal differences (e.g., 18 decimal gap), amounts below the minimum representable unit on the destination chain (e.g., below 1 NEAR for a 24→6 decimal token) will trigger the lock. This is a realistic user mistake — a user sending a "small test transfer" or a dust amount. No privileged access is required; the entry point is the standard NEP-141 `ft_transfer_call` flow.

### Recommendation

Add a pre-check in `init_transfer` (or `init_transfer_internal`) that validates the normalized amount is non-zero **before** burning or locking tokens. For example:

```rust
// Before burning/locking, verify the amount survives normalization
if let Some(token_address) = self.get_token_address(
    transfer_message.get_destination_chain(),
    token_id.clone(),
) {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, enforce a minimum transfer amount at the `init_transfer` entry point based on the registered token decimals.

### Proof of Concept

1. A token is registered with `origin_decimals = 24` (NEAR) and `decimals = 6` (EVM), giving `diff_decimals = 18` and divisor `10^18`.
2. User calls `ft_transfer_call` with `amount = 500_000_000_000_000_000` (0.5 NEAR-units) and `fee = 0`.
3. `init_transfer` passes the `fee < amount` check.
4. `init_transfer_internal` locks 0.5 NEAR-units in the contract (or burns them if a deployed token).
5. Relayer calls `sign_transfer` for this transfer ID.
6. `normalize_amount(500_000_000_000_000_000, {origin_decimals:24, decimals:6}) = 500_000_000_000_000_000 / 10^18 = 0`.
7. `require!(amount_to_transfer > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
8. No cancel function exists; the transfer message remains in `pending_transfers` indefinitely.
9. The 0.5 NEAR-units are permanently locked/burned with no recovery path. [2](#0-1) [1](#0-0) [7](#0-6)
