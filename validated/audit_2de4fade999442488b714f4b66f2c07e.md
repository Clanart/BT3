### Title
`revert_state_diff` Unconditionally Destroys `stateless_compiled_class_hash_v2` Entries for Migrated Classes Without Block-Ownership Check — (`File: crates/apollo_storage/src/state/mod.rs`)

### Summary

`revert_state_diff` calls `delete_compiled_class_hashes_v2` for every class hash present in the reverted block's `class_hash_to_compiled_class_hash` map. When compiled-class-hash migration is active, that map contains class hashes that were **originally declared in earlier blocks** (not the reverted block). The deletion removes their `stateless_compiled_class_hash_v2` entries permanently, leaving the table inconsistent with `declared_classes_block_table`. Any subsequent call to `get_compiled_class_hash_v2` for those classes returns `None`, causing `StateError::MissingCompiledClassHashV2` and wrong execution results.

### Finding Description

`finalize_block` in `crates/blockifier/src/blockifier/transaction_executor.rs` calls `set_compiled_class_hash_migration`, which writes a new v2 compiled class hash for every class that was executed with an old v1 hash. [1](#0-0)  Those migrated class hashes are then inserted into `thin_state_diff.class_hash_to_compiled_class_hash` alongside freshly declared classes. [2](#0-1) 

When `revert_state_diff` is later called for that block, it iterates over `thin_state_diff.class_hash_to_compiled_class_hash.keys()` and passes them to `delete_compiled_class_hashes_v2`:

```rust
delete_compiled_class_hashes_v2(
    &self.txn,
    thin_state_diff.class_hash_to_compiled_class_hash.keys(),
    &compiled_class_hash_v2_table,
)?;
``` [3](#0-2) 

`delete_compiled_class_hashes_v2` deletes every entry unconditionally, with no block-ownership check:

```rust
fn delete_compiled_class_hashes_v2<'env>(
    txn: &'env DbTransaction<'env, RW>,
    class_hashes: impl Iterator<Item = &'env ClassHash>,
    compiled_class_hash_v2_table: &'env StatelessCompiledClassHashV2Table<'env>,
) -> StorageResult<()> {
    for class_hash in class_hashes {
        compiled_class_hash_v2_table.delete(txn, class_hash)?;
    }
    Ok(())
}
``` [4](#0-3) 

Contrast this with `delete_declared_classes_block`, which correctly guards deletion with a block-ownership check:

```rust
if class_block_entry == block_number {
    declared_classes_block_table.delete(txn, class_hash)?;
``` [5](#0-4) 

The `stateless_compiled_class_hash_v2` table is keyed only by `ClassHash` (no block number), so it stores a single authoritative v2 hash per class. [6](#0-5)  After the revert, `declared_classes_block_table` still records the class as declared in its original block, but `stateless_compiled_class_hash_v2` has no entry for it.

The RPC execution state reader falls back to `stateless_compiled_class_hash_v2` when the class manager is unavailable, and returns `StateError::MissingCompiledClassHashV2` on a missing entry: [7](#0-6) 

### Impact Explanation

After reverting a block that contained a compiled-class-hash migration, any execution path that reads `get_executable_class_hash_v2` for the affected class receives `None`. This causes:

- **Wrong execution result / revert** for transactions that invoke the affected class, because the blockifier cannot resolve the v2 compiled class hash needed for execution.
- **Wrong state root and block hash** if the corrupted state is used as the base for the next block.
- **Wrong RPC responses** (`starknet_call`, `starknet_estimateFee`, `starknet_traceTransaction`) when the class manager is not in the execution path.

This matches: *Critical — Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input* and *High — RPC execution, fee estimation, tracing, or simulation returns an authoritative-looking wrong value.*

### Likelihood Explanation

The trigger requires three conditions that can all occur in normal sequencer operation:

1. `enable_casm_hash_migration = true` (a versioned-constants flag).
2. A Cairo 1 class declared with a v1 Poseidon compiled class hash is executed in a later block, triggering migration.
3. That later block is reverted (consensus failure, reorg, or batcher restart).

Conditions 1 and 3 are routine. Condition 2 is satisfied by any user who deploys a class with the old hash scheme and later invokes it. No privileged access is required.

### Recommendation

Mirror the block-ownership guard used by `delete_declared_classes_block`. Before deleting from `stateless_compiled_class_hash_v2`, check `declared_classes_block_table` to confirm the class was **first declared** in the block being reverted:

```rust
fn delete_compiled_class_hashes_v2<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    compiled_class_hash_v2_table: &'env StatelessCompiledClassHashV2Table<'env>,
    block_number: BlockNumber,
) -> StorageResult<()> {
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        if let Some(declared_block) = declared_classes_block_table.get(txn, class_hash)? {
            if declared_block == block_number {
                compiled_class_hash_v2_table.delete(txn, class_hash)?;
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Declare class `C` in block 0 with a v1 Poseidon compiled class hash. `set_executable_class_hash_v2` writes `C → hash_v2` to `stateless_compiled_class_hash_v2`. [8](#0-7) 

2. In block 1, submit an invoke transaction that calls class `C`. With `enable_casm_hash_migration = true`, `finalize_block` detects the v1 hash and calls `set_compiled_class_hash_migration`, inserting `C → hash_v2` into `block_1.state_diff.class_hash_to_compiled_class_hash`. [9](#0-8) 

3. Trigger a revert of block 1 (e.g., consensus failure). `revert_state_diff(BlockNumber(1))` calls `delete_compiled_class_hashes_v2` with `{C}`. The entry `C → hash_v2` is deleted from `stateless_compiled_class_hash_v2`. [3](#0-2) 

4. `declared_classes_block_table` still records `C → block 0` (correctly not deleted, because `0 ≠ 1`). [5](#0-4) 

5. In block 2, any transaction invoking class `C` causes `get_executable_class_hash_v2(C)` to return `None` → `StateError::MissingCompiledClassHashV2(C)`. Execution fails or reverts incorrectly, producing a wrong state root and wrong block hash for block 2.

    fn set_executable_class_hash_v2(
        self,
        class_hash: &ClassHash,
        executable_class_hash_v2: CompiledClassHash,
    ) -> StorageResult<Self> {
        let table = self.open_table(&self.tables.stateless_compiled_class_hash_v2)?;
        table.upsert(&self.txn, class_hash, &executable_class_hash_v2)?;
        Ok(self)

    fn get_compiled_class_hash_v2(
        &self,
        class_hash: ClassHash,
        _compiled_class: &RunnableCompiledClass,
    ) -> StateResult<CompiledClassHash> {
        let maybe_hash =
            if let Some((class_manager_client, run_time_handle)) = &self.class_manager_handle {
                // First, try getting from class manager if available.
                run_time_handle
                    .block_on(class_manager_client.get_executable_class_hash_v2(class_hash))
                    .map_err(|e| StateError::StateReadError(e.to_string()))?
            } else {
                // Fall back to reading from storage.
                self.storage_reader
                    .begin_ro_txn()
                    .map_err(storage_err_to_state_err)?
                    .get_executable_class_hash_v2(&class_hash)
                    .map_err(storage_err_to_state_err)?
            };

        maybe_hash.ok_or(StateError::MissingCompiledClassHashV2(class_hash))
    }
