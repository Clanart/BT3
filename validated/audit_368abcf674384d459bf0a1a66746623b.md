### Title
ERC1155 `safeTransferFrom` Reentrancy Causes `currentOriginNonce` Collision in `InitTransfer` Events — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` increments `currentOriginNonce` before making an external call to an attacker-controlled ERC1155 contract, then reads `currentOriginNonce` again after that call to populate `initTransferExtension` and the `InitTransfer` event. A malicious ERC1155 contract can reenter `initTransfer1155` during `safeTransferFrom`, incrementing the nonce a second time. Both the inner and outer calls then emit `InitTransfer` with the same `originNonce`, while nonce N is permanently skipped. The NEAR side will process one event and reject the other as a duplicate nonce, permanently locking the rejected transfer's tokens in the bridge.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer1155` follows this sequence:

1. `currentOriginNonce += 1` — nonce incremented to N
2. `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` — **external call to attacker-controlled contract**
3. `initTransferExtension(..., currentOriginNonce, ...)` — reads `currentOriginNonce` after the external call
4. `emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...)` — emits `currentOriginNonce` after the external call [1](#0-0) 

The ERC1155 standard requires that `safeTransferFrom` call `onERC1155Received` on the recipient. However, the malicious ERC1155 contract controls its own `safeTransferFrom` implementation and can execute arbitrary code **before** calling `onERC1155Received`. The bridge's `onERC1155Received` guard only checks `operator != address(this)`: [2](#0-1) 

Since the bridge itself is the `operator` (it called `safeTransferFrom`), this check passes in both the outer and inner calls. The guard does **not** prevent reentrancy originating inside the malicious ERC1155's `safeTransferFrom` body.

There is no `nonReentrant` modifier on `initTransfer1155`.

**Reentrant execution trace:**

| Step | `currentOriginNonce` | Action |
|------|---------------------|--------|
| Outer call enters | N-1 → **N** | `currentOriginNonce += 1` |
| Outer call hits `safeTransferFrom` | N | Malicious ERC1155 reenters |
| Inner call enters | N → **N+1** | `currentOriginNonce += 1` |
| Inner call completes | N+1 | Emits `InitTransfer(nonce=N+1, recipient="attacker.near")` |
| Outer call resumes | **N+1** | `currentOriginNonce` is now N+1, not N |
| Outer call completes | N+1 | Emits `InitTransfer(nonce=N+1, recipient="victim.near")` |

Result: nonce N is never emitted; nonce N+1 is emitted twice with different recipients.

In the `OmniBridgeWormhole` variant, `initTransferExtension` publishes a Wormhole message with `originNonce`: [3](#0-2) 

Two Wormhole messages are published with the same `originNonce`. The NEAR side processes the first and rejects the second as a duplicate nonce.

The CLAUDE.md security invariant explicitly states: *"Every `originNonce` is incremented atomically. A nonce must never be reusable"* and *"`InitTransfer` events must contain every field needed to reconstruct the transfer… Fields must not be collapsible (e.g. two different transfers must never produce the same event data)"* — both invariants are violated. [4](#0-3) 

---

### Impact Explanation

**Critical / High.** Two `InitTransfer` events share the same `originNonce`. The NEAR bridge processes the first event it receives and marks that nonce as finalized. The second event is permanently rejected as a duplicate nonce. The tokens from the rejected transfer are locked in the EVM bridge contract forever with no corresponding NEAR-side release — permanent, irrecoverable freezing of user funds.

If the attacker controls the relayer (or submits their Wormhole VAA first), they ensure their inner-call event (with `recipient="attacker.near"`) is processed first. The victim's outer-call event is then rejected. The victim's ERC1155 tokens are permanently locked in the bridge while the attacker receives the corresponding NEAR-side tokens. This constitutes direct theft of bridged assets.

Additionally, nonce N is permanently skipped, breaking the monotonic nonce sequence the NEAR side relies on for ordering and gap detection.

---

### Likelihood Explanation

Any unprivileged user can deploy a malicious ERC1155 contract and call `initTransfer1155`. No admin role, no whitelisting, and no special permissions are required — `initTransfer1155` is a public `external` function. The attacker only needs to hold ERC1155 tokens (which they mint themselves in their malicious contract). The attack is fully self-contained and requires no victim cooperation beyond the victim also using the bridge.

---

### Recommendation

Cache `currentOriginNonce` into a local variable **before** any external call, and use only the local variable in `initTransferExtension` and the emitted event:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce; // cache before external call
    // ...
    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
    // ...
    initTransferExtension(msg.sender, deterministicToken, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, originNonce, ...);
}
```

Apply the same fix to `initTransfer` (ERC20 path), which has the same pattern with `customMinters` external calls. [5](#0-4) 

Additionally, add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuardUpgradeable`) to both `initTransfer` and `initTransfer1155` as defense-in-depth.

### Proof of Concept

```solidity
// Malicious ERC1155 contract
contract MaliciousERC1155 is ERC1155 {
    IOmniBridge bridge;
    bool reentered;

    function safeTransferFrom(
        address from, address to, uint256 id, uint256 amount, bytes memory
    ) public override {
        // Transfer tokens to bridge (so outer call doesn't revert)
        _safeTransferFrom(from, to, id, amount, "");
        
        if (!reentered) {
            reentered = true;
            // Reenter with attacker's recipient — currentOriginNonce is now N
            // After this call, currentOriginNonce becomes N+1
            bridge.initTransfer1155(
                address(this), id, amount, 0, 0, "attacker.near", ""
            );
            // Now currentOriginNonce = N+1
        }
        // Outer call will resume and use currentOriginNonce = N+1 (collision!)
    }
}
```

Attack steps:
1. Deploy `MaliciousERC1155`, mint tokens to attacker
2. Call `bridge.initTransfer1155(maliciousERC1155, tokenId, amount, 0, 0, "victim.near", "")`
3. Outer call: nonce incremented to N; `safeTransferFrom` triggers reentrancy
4. Inner call: nonce incremented to N+1; emits `InitTransfer(nonce=N+1, "attacker.near")`
5. Outer call resumes: emits `InitTransfer(nonce=N+1, "victim.near")`
6. Attacker submits inner-call Wormhole VAA to NEAR first → NEAR mints to "attacker.near"
7. Victim's VAA rejected as duplicate nonce → victim's tokens permanently locked

**File:** evm/CLAUDE.md (L32-36)
```markdown
- **No replay attacks**: Every `destinationNonce` must be checked against `completedTransfers` and marked used before any token transfer. Every `originNonce` is incremented atomically. A nonce must never be reusable
- **Event completeness**: `InitTransfer` and `FinTransfer` events must contain every field needed to reconstruct the transfer. The NEAR side relies solely on these events — any missing or ambiguous field means lost funds or spoofable transfers. Fields must not be collapsible (e.g. two different transfers must never produce the same event data)
- **State before external calls**: Always mutate state (e.g. mark nonce used) before any external call (token transfer, ETH send, custom minter). This is the primary reentrancy defense
- **No token release without signature**: Never mint, transfer, or unlock tokens to a recipient without first verifying a valid MPC signature. No admin function, emergency path, or refactor may bypass this — it is the only authorization gate for finTransfer
- **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. If the token transfer reverts or is skipped, the event must not emit — the NEAR side will treat any emitted event as proof that tokens are held
```
