Audit Report

## Title
Payable swap and liquidity functions silently trap ETH sent for non-WETH operations, allowing any caller to steal it via `refundETH()` - (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol, metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
Every swap entry-point in `MetricOmmSimpleRouter` and every liquidity entry-point in `MetricOmmPoolLiquidityAdder` is marked `payable`, but `PeripheryPayments.pay()` only consumes `address(this).balance` when `token == WETH`. ETH sent alongside a non-WETH call is silently retained by the contract. Because `refundETH()` is an unrestricted `external payable` function that forwards the entire contract balance to `msg.sender`, any third party can immediately drain the trapped ETH.

## Finding Description
`PeripheryPayments.pay()` has three branches: [1](#0-0) 

Only the `token == WETH` branch (lines 73–84) ever touches `address(this).balance`. The `else` branch at line 86 calls `safeTransferFrom` and leaves any ETH in `msg.value` untouched in the contract.

All four router swap functions are `payable` with no guard rejecting non-zero `msg.value` for non-WETH inputs: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

All four liquidity functions in `MetricOmmPoolLiquidityAdder` are likewise `payable`: [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

All four `SelfPermit` functions are `payable` and never consume ETH under any circumstances: [10](#0-9) 

`refundETH()` has no access control and sends the full balance to any caller: [11](#0-10) 

The `receive()` guard only blocks plain ETH transfers (no calldata) from non-WETH addresses: [12](#0-11) 

It does **not** prevent ETH from arriving via `msg.value` in a normal function call, so the guard is entirely ineffective against this attack vector.

## Impact Explanation
Direct, 1:1 loss of user ETH principal. A user who sends ETH alongside a non-WETH swap or liquidity call loses that ETH to the first caller of `refundETH()`. No privileged access is required. This constitutes a direct loss of user principal above Sherlock thresholds and falls squarely within the allowed impact gate (direct loss of user principal, broken payment path).

## Likelihood Explanation
UI bugs, wallet integrations that pre-populate `msg.value`, or copy-paste errors are realistic triggers. Users adding liquidity to non-WETH pools (e.g., USDC/DAI) via `addLiquidityWeighted{value: X}` is a plausible mistake. MEV bots routinely monitor the mempool for unprotected drain patterns; the theft window can be a single block. No privileged access, no special setup, and no non-standard token behavior is required.

## Recommendation
Add a `msg.value == 0` guard to every `payable` function that does not accept ETH as a payment leg:

```solidity
modifier noNativeValue() {
    require(msg.value == 0, "unexpected ETH");
    _;
}
```

Apply `noNativeValue` to: both `addLiquidityExactShares` overloads, both `addLiquidityWeighted` overloads, all four `selfPermit*` functions, and any swap function whose `tokenIn` is statically known to be non-WETH. For functions that conditionally accept ETH (WETH path), add a post-execution check that `address(this).balance` has not increased unexpectedly, or enforce the refund inside the same call. Long term, consider restricting `refundETH()` to be callable only within a `multicall` context so it cannot be weaponized as a standalone drain.

## Proof of Concept

```
1. Alice calls:
   router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
       tokenIn:  USDC,   // ← not WETH
       tokenOut: DAI,
       amountIn: 1_000e6,
       ...
   }));

   Execution path:
   → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, Alice, USDC)
   → pool.swap(...) triggers metricOmmSwapCallback
   → _justPayCallback → pay(USDC, Alice, pool, amount)
   → token != WETH → safeTransferFrom(Alice, pool, amount)   // ETH untouched
   → function returns; 1 ETH remains in router

2. Bob (MEV bot) calls:
   router.refundETH();
   → balance = address(router).balance  // = 1 ETH
   → _transferETH(Bob, 1 ETH)           // Bob receives Alice's ETH

Alice loses 1 ETH. Bob gains 1 ETH. No privileged access required.
```

The identical scenario applies to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares{value: X}(...)` when neither pool token is WETH, and to all four `SelfPermit` functions which never consume ETH under any circumstances.

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
