### Title
Missing `chunk.shard_id()` vs. requested `shard_id` binding in `set_state_header` allows wrong-shard chunk to be stored as state header — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer without ever asserting that the embedded chunk's `shard_id()` matches the `shard_id` parameter. The Merkle inclusion check (`verify_path`) only proves the chunk hash appears somewhere in the block's `chunk_headers_root` tree; it does not verify the tree position corresponds to the requested shard. A malicious peer can supply shard 1's real chunk with its valid position-1 Merkle proof in response to a shard-0 state-sync request, and the function will accept and persist it under `StateHeaderKey(shard_id=0, sync_hash)`.

### Finding Description

**`validate_chunk_proofs`** checks only internal self-consistency of the chunk: [1](#0-0) 

It verifies the chunk hash, `tx_root`, and `receipts_root` — but never reads `chunk.shard_id()` at all. No shard identity check exists.

**`set_state_header`** calls `validate_chunk_proofs` and then does a Merkle inclusion check: [2](#0-1) 

`verify_path` is defined as: [3](#0-2) 

It computes `compute_root_from_path(path, hash(item)) == root`. The path encodes left/right directions that implicitly encode a tree position, but the code never checks that this position equals `shard_layout.get_shard_index(shard_id)`. A proof for position 1 (shard 1) will verify successfully against the same root even when `shard_id=0` is requested, because the root is the same for the whole block.

After both checks pass, the header is stored unconditionally: [4](#0-3) 

There is no `chunk.shard_id() == shard_id` guard anywhere in the function.

**Receipt proof check does not close the gap.** At line 488 the receipts hash is computed with the requested `shard_id`: [5](#0-4) 

The attacker supplies valid on-chain incoming receipts for shard 0 (public data), so this check passes independently of which chunk body is embedded.

**State root node check does not close the gap.** The check validates the attacker-supplied `state_root_node` against `chunk_inner.prev_state_root()`: [6](#0-5) 

Since `chunk_inner` is from the wrong chunk (shard 1), the state root is shard 1's state root. The attacker knows shard 1's state and can supply a valid node for it.

Note that `verify_path_with_index` (which does check position) exists in the codebase but is not used here: [7](#0-6) 

### Impact Explanation

After `set_state_header` succeeds, `StateHeaderKey(shard_id=0, sync_hash)` in `DBCol::StateHeaders` contains shard 1's chunk. Subsequent `set_state_part` calls read this header to obtain `prev_state_root`: [8](#0-7) 

State parts are then validated against shard 1's state root and written to the DB for shard 0. The syncing node's flat storage and `ChunkExtra` for shard 0 are populated with shard 1's trie data, permanently corrupting the node's view of shard 0.

### Likelihood Explanation

Any peer participating in state sync can send a `ShardStateSyncResponseHeader`. No validator or operator privilege is required. The attacker only needs to be a reachable peer during the syncing node's state-sync phase. All required inputs (shard 1's chunk, its Merkle proof, shard 0's incoming receipts, shard 1's state root node) are public on-chain data.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// Guard: chunk must belong to the requested shard.
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the bare `verify_path` call with `verify_path_with_index` (already present in `core/primitives/src/merkle.rs`) using the shard's index in the epoch's shard layout, so the Merkle position is also bound to the requested shard.

### Proof of Concept

1. Stand up a two-shard network. Let shard 0 be the sync target.
2. Intercept or mock the state-sync peer response for `(shard_id=0, sync_hash)`.
3. Replace `chunk` in the `ShardStateSyncResponseHeaderV2` with the real shard-1 chunk for the same block, and replace `chunk_proof` with the real position-1 Merkle proof from that block's `chunk_headers_root`.
4. Keep `incoming_receipts_proofs` and `root_proofs` as valid proofs for shard 0 (copy from a legitimate shard-0 response).
5. Set `state_root_node` to the valid node for shard 1's `prev_state_root`.
6. Call `set_state_header(shard_id=0, sync_hash, tampered_header)`.
7. Assert the call returns `Ok(())` — it will, because no check compares `chunk.shard_id()` (1) against `shard_id` (0).
8. Read `DBCol::StateHeaders` at key `StateHeaderKey(0, sync_hash)` and assert the stored chunk's `shard_id()` is 1, confirming the corruption.
