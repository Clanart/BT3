### Title
`sign_transfer` Rejects Transfers That Passed `init_transfer`, Permanently Locking User Funds - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` validates only that `fee < amount`, but `sign_transfer` additionally requires `normalize_amount(amount - fee) > 0`. For tokens where `origin_decimals > decimals`, the normalization divides by `10^(origin_decimals - decimals)`. A user can construct a transfer that passes `init_transfer` but can never be completed via `sign_transfer`, permanently locking their tokens in the bridge with no recovery path.

### Finding Description

`init_transfer` (NEAR bridge entry point) enforces a single fee check:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This guarantees only that `amount_without_fee > 0`. Tokens are then locked/burned and the transfer is stored in `pending_transfers`.

`sign_transfer` (the only exit path for NEAR-origin transfers) applies a stricter check:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` performs floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

For any token where `origin_decimals > decimals`, if `amount_without_fee < 10^(origin_decimals - decimals)`, then `normalize_amount` returns 0 and `sign_transfer` permanently panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.

`update_transfer_fee` cannot rescue the transfer because it only allows fee *increases* (`fee.fee >= current_fee.fee`), which would make `amount_without_fee` even smaller: [4](#0-3) 

There is no `cancel_transfer` or refund mechanism in the contract. The transfer remains in `pending_transfers` indefinitely with no recovery path.

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Once `init_transfer` succeeds, the user's tokens are burned or held by the bridge. If `sign_transfer` permanently reverts, those tokens are unrecoverable. The amount locked is not limited to dust: for a token with 8 NEAR origin decimals and 0 EVM decimals (divisor = 10^8), a user transferring 1.5 tokens with a fee of 1 token would have `amount_without_fee = 0.5 tokens = 50,000,000 units < 100,000,000`, causing `normalize_amount = 0` and permanently locking 1.5 tokens.

### Likelihood Explanation

**Medium.** The condition requires a token registered with `origin_decimals > decimals` (common for NEAR tokens with 24 decimals bridged to EVM with 18 decimals, or tokens with 8 NEAR decimals bridged to EVM with 0 decimals) and a transfer where `amount - fee` falls below the normalization divisor. A user who sets a fee close to the transfer amount (which `init_transfer` permits) can inadvertently trigger this. No privileged access is required — any bridge user calling `ft_transfer_call` with a valid `InitTransferMsg` can reach this state.

### Recommendation

Add the same `normalize_amount > 0` check inside `init_transfer` (or `init_transfer_internal`) before locking tokens, mirroring the check already present in `sign_transfer`. Alternatively, implement a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have not yet been signed.

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 8`, `decimals = 0` (divisor = 10^8 = 100,000,000).

1. User calls `ft_transfer_call` with `amount = 150_000_000` and `fee = 100_000_000`.
2. `init_transfer` check: `100_000_000 < 150_000_000` → passes. Tokens locked.
3. `amount_without_fee() = 150_000_000 - 100_000_000 = 50_000_000`.
4. `normalize_amount(50_000_000, {decimals: 0, origin_decimals: 8}) = 50_000_000 / 100_000_000 = 0`.
5. Relayer calls `sign_transfer` → panics: `ERR_INVALID_AMOUNT_TO_TRANSFER`.
6. `update_transfer_fee` cannot reduce the fee (only increases allowed).
7. No cancellation function exists. 150,000,000 units (1.5 tokens) are permanently locked. [2](#0-1) [1](#0-0) [5](#0-4) [3](#0-2)
