Audit Report

## Title
Wrong Bin Length Subtracted from `walkDistE6` in `_fillBids` Corrupts All Bid-Side Price Boundaries — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

In `_fillBids`, the downward walk decrements `walkDistE6` by `lenAbove` — the `lengthE6` of bin `b+1` — instead of the `lengthE6` of the current bin `b`. This produces wrong lower/upper price boundaries for every bid-side bin below `curBinIdx`, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` in the `LiquidityDepth` struct returned by `getLiquidityDepth`. Integrations that consume these fields to size slippage guards are induced into fund-losing trades.

## Finding Description

`walkDistE6` is initialized to `curBinDistFromProvidedPriceE6`, the lower edge of `curBinIdx`. The correct invariant for the downward walk is:

```
lower(b) = lower(b+1) - lengthE6(b)
```

The ask-side walk (`_fillAskRow`) implements the analogous invariant correctly — after processing bin `b` it advances `ctx.cumDistE6` by that bin's own `lengthE6`:

```solidity
// line 441 — correct: uses current bin's lengthE6
ctx.cumDistE6 += int256(uint256(lengthE6));
```

The bid-side loop does the opposite. At each step it reads the length of the **bin above** (`b+1`) and subtracts that:

```solidity
// lines 496-498 — wrong: reads b+1's length, not b's
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));

// line 500 — current bin's lengthE6 is available here but not used for the walk
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
```

`lowerX64` and `upperX64` are then computed from the wrong `walkDistE6`, so both marginal price endpoints passed to `_accumulateBidLevel` are wrong. The error accumulates with each hop: if bins have heterogeneous lengths the offset grows unboundedly as the walk descends.

Concrete example with three bins of different lengths:

| Bin | `lengthE6` | Correct `walkDistE6` (lower edge) | Code's `walkDistE6` |
|-----|-----------|----------------------------------|---------------------|
| 0 (cur) | 10 000 | 0 | 0 ✓ |
| −1 | 20 000 | −20 000 | −10 000 ✗ |
| −2 | 15 000 | −35 000 | −30 000 ✗ |

No guard in `getLiquidityDepth` or its callers detects or corrects this; the corrupted `DepthLevel` array is returned as-is.

## Impact Explanation

`getLiquidityDepth` is the primary off-chain depth oracle for the protocol. The `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` fields are consumed by aggregators, front-ends, and market-making bots to estimate execution price for a given sell size and to set `amountOutMinimum` / `priceLimitX64` before calling `exactInput` or `exactInputSingle`. When the bid depth ladder reports systematically shifted prices (wrong bin boundaries → wrong marginal prices → wrong fee-adjusted execution prices), callers who size their slippage guard from this data will accept a worse price than intended — a direct loss of output tokens — or revert unnecessarily. This matches the "Lens/quoter path" Smart Audit Pivot: quoted amounts must match actual pool execution closely enough that integrations are not induced into fund-losing trades.

## Likelihood Explanation

The bug is triggered unconditionally whenever `getLiquidityDepth` is called on any pool that has at least one bin below `curBinIdx` (i.e., any pool with negative-index bins and a non-minimum current bin). No special attacker capability is required; any unprivileged caller of the public `getLiquidityDepth` function receives corrupted data. The error is present in every deployment of `MetricOmmPoolDataProvider` and affects every integration that relies on bid-side depth.

## Recommendation

Replace the `lenAbove` read with the current bin's own `lengthE6`. Move the `walkDistE6` decrement to after the `_binState` call for `binIdx`:

```solidity
for (int256 b = int256(curBinIdx) - 1; b >= int256(lowCap); b--) {
    int8 binIdx = int8(b);
    (, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);

    walkDistE6 -= int256(uint256(lengthE6)); // ← use current bin's length

    uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(uint256(addFeeSellE6), Q64, ONE_E6);
    uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, Math.Rounding.Floor);
    uint256 upperX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), Math.Rounding.Floor);
    // ... rest unchanged
}
```

This mirrors the ask-side pattern exactly and satisfies `lower(b) = lower(b+1) - lengthE6(b)`.
