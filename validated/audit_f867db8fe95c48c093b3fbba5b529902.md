### Title
Missing Lower-Bound Timestamp Validation in `is_block_info_valid` Allows Proposer to Commit Blocks with Arbitrarily Stale Timestamps — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` enforces that a proposed block timestamp is not too far **in the future** (`timestamp <= now + block_timestamp_window_seconds`) but imposes **no lower bound relative to `now`**. The only lower bound is monotonicity against the previous block (`timestamp >= last_block_timestamp`). A malicious proposer can set `init.timestamp` to any value in `[last_block_timestamp, now + window]`, including values arbitrarily far in the past. Honest validators pass the check and vote for the proposal, committing a block with a wrong timestamp to state. This corrupts the `get_block_timestamp` syscall result for every transaction in that block and the block hash/commitment.

---

### Finding Description

In `is_block_info_valid`:

```rust
let now: u64 = clock.unix_now();
let last_block_timestamp =
    block_info_validation.previous_block_info.as_ref().map_or(0, |info| info.timestamp);
if init_proposed.timestamp < last_block_timestamp {          // only monotonicity
    return Err(...);
}
if init_proposed.timestamp > now + block_info_validation.block_timestamp_window_seconds {
    return Err(...);                                          // only future bound
}
``` [1](#0-0) 

The config field `block_timestamp_window_seconds` is documented as *"Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time"*, implying a symmetric window around `now`. [2](#0-1) 

In practice only the upper half is enforced. There is no check of the form `timestamp >= now - block_timestamp_window_seconds`. The same asymmetric pattern is replicated in `try_sync`: [3](#0-2) 

`is_block_info_valid` is called unconditionally for both immediately-processed and previously-queued proposals before the batcher is asked to execute them: [4](#0-3) [5](#0-4) 

The validated `init.timestamp` is then forwarded verbatim to the batcher as `block_info.block_timestamp` and stored in the committed block header: [6](#0-5) 

---

### Impact Explanation

**Wrong `get_block_timestamp` syscall result (Critical — Wrong execution result from syscall/execution logic).**  
Every transaction executed in the block calls `get_block_timestamp` against `block_info.block_timestamp`. If the proposer sets `timestamp = last_block_timestamp + 1` while `now` is hours later, every contract that uses `block_timestamp` for time-based logic (deadline checks, vesting schedules, oracle freshness guards) receives a stale value and may execute incorrectly — e.g., a contract guarded by `assert(block_timestamp >= deadline)` would revert when it should succeed, or vice-versa. [7](#0-6) 

**Wrong block hash / commitment (Critical — Wrong state commitment).**  
The block timestamp is a direct input to `calculate_block_hash` / `PartialBlockHashComponents`. A stale timestamp produces a different partial block hash, which propagates into the proposal commitment that consensus finalises and that the prover must reproduce.

---

### Likelihood Explanation

Any consensus validator that becomes the round proposer can trigger this. The proposer role rotates among all validators, so no special privilege beyond being a validator is required. The manipulation is trivial: set `init.timestamp` to `last_block_timestamp + 1`. Honest validators run `is_block_info_valid`, the check passes, they vote prevote/precommit, and the block is finalised with the stale timestamp. The window of manipulation equals `now - last_block_timestamp - 1` seconds, which grows whenever block production is slow or the network experiences latency.

---

### Recommendation

Add a symmetric lower-bound check inside `is_block_info_valid`, mirroring the existing upper-bound check:

```rust
// Existing upper bound (keep as-is):
if init_proposed.timestamp > now + block_info_validation.block_timestamp_window_seconds {
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}

// Add symmetric lower bound:
if now.saturating_sub(block_info_validation.block_timestamp_window_seconds)
    > init_proposed.timestamp
{
    return Err(ValidateProposalError::InvalidBlockInfo(
        init_proposed.clone(),
        block_info_validation.clone(),
        format!(
            "Timestamp is too old: now={}, block_timestamp_window_seconds={}, proposed={}",
            now,
            block_info_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

Apply the same fix to the `try_sync` timestamp check in `sequencer_consensus_context.rs`. [8](#0-7) 

---

### Proof of Concept

**Setup:** Last committed block has `timestamp = T_prev`. Current wall-clock time is `now = T_prev + 3600` (one hour later, e.g. after a network partition). `block_timestamp_window_seconds = 1`.

**Attack:**
1. Malicious proposer constructs `ProposalInit` with `timestamp = T_prev + 1` (3599 seconds in the past).
2. Proposal is broadcast to validators.
3. Each honest validator calls `is_block_info_valid`:
   - `T_prev + 1 >= T_prev` → **passes** (monotonicity check)
   - `T_prev + 1 <= (T_prev + 3600) + 1` → **passes** (future-window check)
4. Validators call `initiate_validation` → batcher executes all transactions with `block_timestamp = T_prev + 1`.
5. Every `get_block_timestamp` syscall in those transactions returns `T_prev + 1` instead of the correct `~T_prev + 3600`.
6. Consensus reaches decision; block is committed with the stale timestamp and the derived (wrong) block hash. [9](#0-8)
