### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Overstates Locked Collateral and Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` uses the caller-supplied `amount` parameter verbatim in the cross-chain Wormhole message and emitted event, without measuring the actual tokens received. For Fee-on-Transfer (FoT) tokens, the bridge contract receives fewer tokens than `amount`, but the cross-chain message instructs the destination chain to mint or release the full `amount`. This creates unbacked supply on the destination chain and permanently breaks bridge collateralization.

---

### Finding Description

In `OmniBridge.sol::initTransfer()`, the plain ERC20 lock path (for tokens that are neither bridge tokens nor custom-minter tokens) is:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied, not verified against actual receipt
);
```

Immediately after, the same unverified `amount` is forwarded to `initTransferExtension()` and the emitted event:

```solidity
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // <-- not actual received amount
    fee, nativeFee, recipient, message, extensionValue
);

emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // <-- not actual received amount
    fee, nativeFee, recipient, message
);
```

In `OmniBridgeWormhole.sol::initTransferExtension()`, this `amount` is Borsh-encoded into the Wormhole cross-chain message:

```solidity
bytes memory payload = bytes.concat(
    ...
    Borsh.encodeUint128(amount),   // overstated for FoT tokens
    ...
);
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

For a FoT token that charges a percentage fee on every `transferFrom`, the bridge contract receives `amount - token_fee` tokens, but the Wormhole message carries `amount`. The NEAR bridge (or any other destination chain) reads this message and mints or releases the full `amount` to the recipient. The EVM bridge is now undercollateralized by `token_fee` per transfer.

---

### Impact Explanation

Each `initTransfer` call with a FoT token creates a deficit: the EVM bridge holds `amount - token_fee` tokens but the destination chain has minted `amount` tokens. This deficit accumulates linearly with usage. When users bridge tokens back to EVM, the bridge will eventually be unable to release the full amount owed — the last users' funds are permanently frozen and irrecoverable. This directly matches:

- **High**: Balance/accounting corruption that breaks bridge collateralization.
- **Critical**: Permanent freezing or irrecoverable lock of user funds in bridge flows.

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer()` with any ERC20 token address. FoT tokens exist in production (e.g., tokens with configurable transfer fees). No privileged access, leaked keys, or external compromise is required. The attacker does not even need to be malicious — ordinary users bridging FoT tokens will trigger this silently.

---

### Recommendation

Measure the actual balance change and use it as the canonical amount for the cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
uint128 actualReceived = uint128(balanceAfter - balanceBefore);
// Use actualReceived instead of amount in initTransferExtension and emit
```

Additionally, document clearly which token types are supported, and consider rejecting tokens whose actual received amount differs from `amount` if FoT support is not intended.

---

### Proof of Concept

1. Deploy a FoT ERC20 token that charges a 1% fee on every `transferFrom`.
2. Call `OmniBridge.initTransfer(fotToken, 1_000_000, 0, 0, nearRecipient, "")`.
3. Bridge receives `990_000` tokens (1% fee deducted by the token contract).
4. Wormhole message encodes `amount = 1_000_000`.
5. NEAR bridge mints `1_000_000` tokens to the recipient.
6. Repeat 100 times: EVM bridge holds `99_000_000` tokens, but `100_000_000` tokens exist on NEAR.
7. When users bridge back `100_000_000` tokens to EVM, the bridge can only release `99_000_000` — the last `1_000_000` tokens are permanently stuck.
