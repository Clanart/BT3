### Title
`ProofOfInclusion::valid()` Final Root Check Is a Tautology, Enabling Forged DataLayer Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` is the sole public API for verifying a DataLayer Merkle inclusion proof. Its final check — `existing_hash == self.root_hash()` — is always `true` whenever the loop body passes, because both sides resolve to the same `last.combined_hash`. The function therefore only verifies internal self-consistency of the proof object; it never compares the computed root against any external, trusted tree root. An attacker who can supply a `ProofOfInclusion` (possible via the `Streamable` wire format or the `from_py_object` Python binding) can forge a proof for any arbitrary key and have `valid()` return `true`.

### Finding Description

In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`, `valid()` is implemented as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;          // attacker-controlled

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {  // only checks self-consistency
            return false;
        }
        existing_hash = calculated_hash;             // = layer.combined_hash
    }

    existing_hash == self.root_hash()                // ALWAYS TRUE (see below)
}
```

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // same value as `existing_hash` after the loop
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash` holds the last `calculated_hash`. The loop body already asserts `calculated_hash == layer.combined_hash`, so `existing_hash` equals `last.combined_hash`. `self.root_hash()` also returns `last.combined_hash`. The final comparison is therefore a tautology — it can never be `false` once the loop completes. The empty-layers case is equally trivially true: `existing_hash = self.node_hash` and `root_hash() = self.node_hash`.

The struct is `Streamable` (wire-serializable/deserializable) and carries `pyclass(get_all, from_py_object)` Python bindings, so an attacker can construct or transmit an arbitrary `ProofOfInclusion` with any `node_hash` and any internally consistent `layers` vector. No external trusted root is ever consulted.

### Impact Explanation

Any DataLayer client that calls `proof.valid()` as its sole proof-verification step — the pattern shown in every test and the only method the Python API exposes for this purpose — will accept a forged proof for a key that is not present in the tree. This satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion, letting untrusted input prove invalid state.**

### Likelihood Explanation

The Python binding exposes `valid()` as `py_valid()` with no companion "check against trusted root" method. The Rust and Python tests call only `proof.valid()` without separately comparing `proof.root_hash()` to a known-good root. Any DataLayer consumer following the documented pattern is vulnerable. The `Streamable` trait makes it trivial to deliver a crafted proof over the network.

### Recommendation

`valid()` must accept a trusted root hash and compare against it instead of against `self.root_hash()`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare to external trusted root
}
```

The existing `valid()` / `root_hash()` pair should be deprecated or clearly documented as "internal consistency only — callers must separately verify `root_hash()` against a trusted root."

### Proof of Concept

```python
import hashlib
from chia_rs import ProofOfInclusion, ProofOfInclusionLayer, Side

# Attacker wants to forge a proof that key with hash NODE_HASH is in the tree.
node_hash = bytes([0xAA] * 32)
other_hash = bytes([0xBB] * 32)

# Compute a combined_hash that is internally consistent with the layer.
# calculate_internal_hash(node_hash, Left, other_hash) — attacker computes offline.
h = hashlib.sha256(b'\x02' + node_hash + other_hash).digest()  # simplified
combined_hash = h

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Left,
    other_hash=other_hash,
    combined_hash=combined_hash,
)
forged = ProofOfInclusion(node_hash=node_hash, layers=[layer])

# valid() returns True for any internally consistent proof,
# regardless of the actual DataLayer tree root.
assert forged.valid()   # passes — forged inclusion accepted
```
