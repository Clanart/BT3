### Title
`batch_insert` Bypasses `HashAlreadyPresent` Guard for All Entries When Tree Has ≥2 Leaves — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` only calls the validated `insert()` path (which checks `HashAlreadyPresent`) when the tree currently has ≤1 leaves. When the tree already has ≥2 leaves, **every** entry in the batch is written directly via `insert_entry_to_blob` with no duplicate-hash check — neither against existing leaves nor within the batch itself. An attacker who controls the `Hash` parameter can insert multiple leaves sharing the same hash, silently corrupting the tree.

### Finding Description

`batch_insert` has two distinct code paths:

**Path A — tree has ≤1 leaves** (lines 578–584): pops up to 2 entries and calls `self.insert()`, which guards against duplicates: [1](#0-0) 

**Path B — remaining entries, or ALL entries when tree has ≥2 leaves** (lines 587–603): iterates the remaining vector and calls only `insert_entry_to_blob` — no key or hash uniqueness check: [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` and silently overwrites any existing entry for the same hash: [3](#0-2) 

`add_leaf` itself: [4](#0-3) 

The result: two (or more) `LeafNode` blocks with identical `hash` fields exist in the blob, but `leaf_hash_to_index` only tracks the last one written. The earlier leaf becomes unreachable by hash-based lookup.

### Impact Explanation

- **Tree root corruption**: `calculate_lazy_hashes` computes internal node hashes from child hashes. Two leaves with the same hash produce an internal hash that is identical to what a single-leaf subtree would produce, making the root ambiguous.
- **Proof-of-inclusion forgery**: `get_proof_of_inclusion` and `get_node_by_hash` both rely on `leaf_hash_to_index`. With a duplicate hash, one leaf is invisible to hash-based lookup, enabling a proof for one (key, value) pair to be presented as proof for a different pair sharing the same leaf hash.
- **Cache/blob divergence**: `leaf_count` (counted from blob traversal in `check_just_integrity`) will exceed `leaf_hash_to_index_cache_length`, but `check_integrity` is not called automatically after `batch_insert`. [5](#0-4) 

### Likelihood Explanation

The condition `self.block_status_cache.leaf_count() <= 1` means that in any production DataLayer store with ≥2 existing entries, **the entire batch bypasses all uniqueness validation**. The `Hash` field is a caller-supplied parameter independent of `KeyId`/`ValueId`: [6](#0-5) 

Any code path that feeds attacker-controlled hash values into `batch_insert` (e.g., DataLayer sync receiving key-value-hash triples from a peer) is directly exploitable without any cryptographic work.

### Recommendation

Add a duplicate-hash check inside the bulk loop in `batch_insert`, mirroring the guard in `insert()`:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

This should be inserted before `insert_entry_to_blob` is called in the bulk path. [2](#0-1) 

### Proof of Concept

```rust
// Tree already has 2 leaves (so leaf_count > 1, bulk path is taken for all new entries)
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.insert(KeyId(0), ValueId(0), &Hash::from([0u8; 32]), InsertLocation::AsRoot {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &Hash::from([1u8; 32]), InsertLocation::Auto {}).unwrap();

// Now batch_insert with duplicate hash H across entries — all go through bulk path
let dup_hash = Hash::from([0xABu8; 32]);
blob.batch_insert(vec![
    ((KeyId(2), ValueId(2)), dup_hash),
    ((KeyId(3), ValueId(3)), dup_hash),  // same hash, different key
]).unwrap(); // succeeds — no error

// Tree now contains two leaves with hash 0xAB...AB
// check_integrity() will fail with IntegrityLeafHashToIndexCacheLength
// get_node_by_hash(dup_hash) returns only one of the two leaves
assert!(blob.check_integrity().is_err());
```
