### Title
Missing shard-position binding in `set_state_header` allows cross-shard chunk substitution to corrupt `ChunkExtra.state_root` for the wrong shard — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` validates that the supplied chunk is *somewhere* in the block's `chunk_headers_root` merkle tree, but never verifies that the chunk occupies the position that corresponds to the requested `shard_id`. Because `verify_path` is position-agnostic, an unprivileged peer can supply shard X's chunk (with shard X's valid merkle path) inside a `ShardStateSyncResponseHeader` claimed to be for shard Y. All validation checks pass. The header is stored under `StateHeaderKey(shard_id=Y, sync_hash)`. When `set_state_finalize(shard_id=Y)` runs, it derives `shard_uid_Y` from the `shard_id` parameter but applies shard X's chunk body (transactions, receipts) against shard X's state (installed into shard Y's trie slot), writing a `ChunkExtra` with a wrong `state_root` under the `shard_uid_Y` key.

### Finding Description

**Step 1 – Position-agnostic `verify_path`.**

`verify_path` in `core/primitives/src/merkle.rs` only checks that `compute_root_from_path(path, hash(item)) == root`. It does not check which leaf index the item occupies. [1](#0-0) 

The position-aware variant `verify_path_with_index` exists and calls `verify_path_matches_index` first, but it is not used here. [2](#0-1) 

**Step 2 – No `chunk.shard_id() == shard_id` guard in `set_state_header`.**

The grep search for any such equality check in `chain/chain/src/state_sync/adapter.rs` returns zero matches. The only chunk-membership check is:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
``` [3](#0-2) 

Shard X's chunk IS in the `chunk_headers_root` tree (at position X). Providing shard X's chunk with shard X's merkle path makes `verify_path` return `true` even when `shard_id=Y` is requested, because the path reconstructs the same root regardless of which position is claimed.

**Step 3 – `validate_chunk_proofs` does not check shard identity.**

`validate_chunk_proofs` only verifies the chunk's internal hash, transaction root, and outgoing-receipts root. It does not compare `chunk.shard_id()` to any expected value. [4](#0-3) 

**Step 4 – Receipt-proof validation uses the `shard_id` parameter, not `chunk.shard_id()`.**

At line 488 the receipts hash is computed as `ReceiptList(shard_id, receipts)` where `shard_id` is the caller-supplied Y. The attacker can supply shard Y's legitimate incoming receipt proofs (public on-chain data) alongside shard X's chunk, satisfying all receipt-proof checks. [5](#0-4) 

**Step 5 – `set_state_finalize` derives `shard_uid` from the parameter, not from the chunk.**

After the header passes validation and is stored under `StateHeaderKey(shard_id=Y, sync_hash)`, `set_state_finalize` retrieves it and computes:

```rust
let shard_uid =
    shard_id_to_uid(self.epoch_manager.as_ref(), shard_id, block_header.epoch_id())?;
``` [6](#0-5) 

`shard_id` here is Y (the caller-supplied value). `shard_uid` is therefore `shard_uid_Y`. But the chunk body applied is shard X's chunk body (transactions from shard X, `prev_state_root` from shard X's header):

```rust
let apply_result = self.runtime_adapter.apply_chunk(
    RuntimeStorageConfig::new(chunk_header.prev_state_root(), true),
    …
    ApplyChunkShardContext { shard_uid, … },   // shard_uid_Y
    …
    transactions,   // shard X's transactions
)?;
``` [7](#0-6) 

The result is stored via `apply_chunk_postprocessing` keyed by `shard_uid_Y`, writing `ChunkExtra.state_root` = post-execution root of shard X's state under shard Y's key. [8](#0-7) 

### Impact Explanation

The exact corrupted value is `ChunkExtra.state_root` stored in the DB column keyed by `(block_hash, shard_uid_Y)`. This root is derived from shard X's trie (installed into shard Y's trie slot via `apply_state_part(shard_id=Y, state_root=shard_X_prev_root, …)`) after applying shard X's transactions and shard Y's incoming receipts. The node's authoritative record of shard Y's post-state is now wrong. Any subsequent block production or validation for shard Y will use this corrupted root, causing the node to produce blocks that other honest nodes reject, or to accept blocks it should reject.

### Likelihood Explanation

State sync headers are downloaded from peers via `StateSyncDownloader::ensure_shard_header`, which accepts any response from a `StateSyncDownloadSource` (peer or external storage) and forwards it to `set_state_header` for the only validation gate. [9](#0-8) 

No validator or operator privilege is required. Any peer that can respond to a state sync header request can trigger this path. The attack requires knowledge of the on-chain chunk hashes and merkle paths (all public), plus the ability to serve a crafted response.

### Recommendation

In `set_state_header`, after extracting the chunk, add an explicit shard-identity check before the merkle-path verification:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the position-agnostic `verify_path` call with `verify_path_with_index`, supplying the expected shard index derived from `shard_layout.get_shard_index(shard_id)?`, so that the merkle proof is bound to the correct leaf position in the `chunk_headers_root` tree. [3](#0-2) 

Apply the same fix to the `prev_chunk_header` merkle-path check. [10](#0-9) 

### Proof of Concept

In a two-shard `TestEnv`:

1. Produce enough blocks to reach a sync hash.
2. Obtain the legitimate `ShardStateSyncResponseHeader` for shard 0 (contains shard 0's chunk at position 0 in the tree, with its merkle path).
3. Obtain shard 1's legitimate incoming receipt proofs and `state_root_node`.
4. Construct a crafted `ShardStateSyncResponseHeader` for shard 1 by replacing its chunk and chunk_proof with shard 0's chunk and shard 0's merkle path, keeping shard 1's receipt proofs and state_root_node.
5. Call `chain.state_sync_adapter.set_state_header(shard_id=1, sync_hash, crafted_header)` — this succeeds without error.
6. Apply shard 0's state parts with `apply_state_part(shard_id=1, state_root=shard0_prev_root, …)`.
7. Call `chain.set_state_finalize(shard_id=1, sync_hash)`.
8. Read `get_chunk_extra(prev_block_hash, shard_uid_1)` and assert its `state_root` equals shard 0's post-execution root rather than shard 1's legitimate post-execution root.

The assertion will hold, demonstrating that shard 1's `ChunkExtra.state_root` has been overwritten with shard 0's execution result.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
