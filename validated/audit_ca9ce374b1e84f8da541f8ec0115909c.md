### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Complete Fee Bypass - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in the StarkNet OS performs fee computation using Cairo's native felt (field element) arithmetic, which is modular arithmetic over the StarkNet prime P ≈ 2^251. An unprivileged transaction sender can craft V3 transaction resource bounds such that the felt-arithmetic sum wraps around to exactly 0 modulo P. When `max_fee == 0`, `charge_fee` returns immediately without executing any ERC20 transfer, granting the attacker free execution of up to `EXECUTE_MAX_SIERRA_GAS` (1.1 billion) gas units per transaction.

---

### Finding Description

`charge_fee` in `transaction_impls.cairo` first calls `compute_max_possible_fee`:

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
``` [1](#0-0) 

If the result is 0, `charge_fee` returns immediately, skipping the ERC20 transfer entirely:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [2](#0-1) 

The only constraints on resource bound fields come from `pack_resource_bounds` (called during transaction hash computation):
- `max_amount` is bounded to `[0, 2^64 - 1]`
- `max_price_per_unit` is bounded to `[0, P/2)` via `assert_nn` [3](#0-2) 

These bounds are wide enough to allow the felt-arithmetic sum to wrap to exactly 0 mod P. Specifically, a user can set:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | 2 |
| `l1_gas_bounds.max_price_per_unit` | (P−1)/2 |
| `l2_gas_bounds.max_amount` | any large value (e.g., 2^64−1) |
| `l2_gas_bounds.max_price_per_unit` | 0 |
| `tip` | 0 |
| `l1_data_gas_bounds.max_amount` | 1 |
| `l1_data_gas_bounds.max_price_per_unit` | 1 |

Verification: `2 × (P−1)/2 + (2^64−1) × 0 + 1 × 1 = (P−1) + 0 + 1 = P ≡ 0 (mod P)` ✓

All individual constraints pass:
- `max_amount = 2 ≤ 2^64−1` ✓
- `max_price_per_unit = (P−1)/2 < P/2` ✓ (`assert_nn` passes)
- `l2_gas_bounds.max_amount` can be set to any value up to `2^64−1` ✓

The `l2_gas_bounds.max_amount` field is also used directly as the initial gas budget for execution:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [4](#0-3) 

So the attacker simultaneously achieves `max_fee = 0` (no fee charged) and a large gas budget for execution (capped at `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` by `cap_remaining_gas`). [5](#0-4) 

---

### Impact Explanation

**Direct loss of funds (Critical):** The sequencer executes up to 1.1 billion gas units of computation per transaction without receiving any fee payment. The ERC20 `transfer` call to the sequencer address is never made.

**Network shutdown (High):** An attacker can flood the network with free, maximally-expensive transactions. Since each transaction consumes up to `EXECUTE_MAX_SIERRA_GAS` without cost, the attacker can saturate block capacity indefinitely, preventing legitimate transactions from being confirmed.

---

### Likelihood Explanation

The attack requires only that the sequencer's fee pre-validation uses the same felt arithmetic as the OS (which is the expected behavior for a ZK-rollup sequencer that simulates OS execution via the Cairo VM). The crafted resource bounds pass all hash-level validation checks (`pack_resource_bounds`, `hash_fee_fields`), so the transaction hash is valid and the transaction is well-formed. The attacker needs no special privileges — only the ability to submit a V3 transaction.

---

### Recommendation

In `compute_max_possible_fee`, validate that each multiplicative term does not overflow the field prime before summing, or enforce explicit upper bounds on `max_price_per_unit` (e.g., `assert_nn_le(max_price_per_unit, MAX_PRICE_PER_UNIT)` with a practical bound much smaller than P/2). Additionally, add a post-computation check that `max_fee > 0` whenever any resource bound is non-zero, reverting if the result is 0 due to overflow.

---

### Proof of Concept

1. Craft a V3 invoke transaction with:
   - `l1_gas_bounds = {max_amount: 2, max_price_per_unit: (P-1)/2}`
   - `l2_gas_bounds = {max_amount: 1_100_000_000, max_price_per_unit: 0}`
   - `tip = 0`
   - `l1_data_gas_bounds = {max_amount: 1, max_price_per_unit: 1}`

2. Sign and submit the transaction. All hash-level validations pass because `pack_resource_bounds` only checks `max_amount ≤ 2^64−1` and `max_price_per_unit ≥ 0`.

3. During OS execution, `compute_max_possible_fee` computes:
   `2 × (P−1)/2 + 1_100_000_000 × 0 + 1 × 1 = P − 1 + 1 = P ≡ 0 (mod P)`

4. `charge_fee` sees `max_fee == 0` and returns immediately at line 123–125 without executing the ERC20 transfer.

5. The transaction's `__execute__` entry point runs with 1.1 billion gas units of budget at zero cost to the attacker. [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
