Audit Report

## Title
Unguarded `refundETH` and `sweepToken` Allow Any Caller to Drain Stranded ETH/ERC20 from the Router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` transfers the router's full ETH balance to `msg.sender` with no access control, and `sweepToken()` transfers the router's full ERC20 balance to a caller-supplied `recipient` with no restriction. When a user calls `exactInputSingle` or `exactInput` with excess `msg.value`, the `pay()` helper wraps only the required amount and leaves the surplus ETH on the router. Any third party can then call `refundETH()` to steal that surplus.

## Finding Description
`pay()` handles native ETH payment for WETH-denominated swaps: [1](#0-0) 

When `nativeBalance >= value`, exactly `value` wei is wrapped and forwarded; the remainder stays on the router. Neither `exactInputSingle` nor `exactInput` auto-refunds after the swap: [2](#0-1) 

The surplus ETH is then drainable by anyone via the unguarded `refundETH()`: [3](#0-2) 

Similarly, `sweepToken()` transfers the full ERC20 balance to a caller-chosen `recipient`: [4](#0-3) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH transfers to the router — it does not prevent ETH from accumulating via `msg.value` in payable swap functions: [5](#0-4) 

`multicall` uses `delegatecall`, so `msg.sender` is preserved within sub-calls, meaning a user who correctly bundles `[exactInputSingle, refundETH]` is safe — but a user who calls `exactInputSingle` directly (which is `external payable`) is not: [6](#0-5) 

## Impact Explanation
A victim who sends excess ETH with a swap call and omits `refundETH` from their multicall bundle loses the surplus ETH to any attacker who calls `refundETH()` afterward. For `sweepToken`, any ERC20 balance stranded on the router (e.g., intermediate tokens from multi-hop exact-input swaps) can be drained to an attacker-controlled address. This is a direct loss of user principal. Severity is **Medium**: the loss is bounded to the surplus/stranded amount, not the full swap principal, and requires the user to omit the refund call.

## Likelihood Explanation
No special permissions, malicious pool, or oracle manipulation are required — only a public call. Users interacting directly with `exactInputSingle` or `exactInput` without a frontend that auto-bundles `refundETH` will routinely leave ETH on the router. The attacker can poll the router's ETH balance and call at any time; front-running is not even necessary.

## Recommendation
1. **Auto-refund excess ETH** at the end of each top-level swap function (after `_clearExpectedCallbackPool()`) rather than relying on the caller to bundle a separate `refundETH` call.
2. **Restrict `refundETH`** to return only to the original `msg.sender` of the enclosing multicall, tracked via transient storage at swap entry.
3. **Restrict `sweepToken` recipient** to `msg.sender`, or add an `onlySelf` guard so it can only be invoked from within a `multicall` bundle initiated by the token owner.

## Proof of Concept
```solidity
// Victim sends 1 ETH; swap only needs 0.9 ETH.
// pay() wraps 0.9 ETH → WETH → pool; 0.1 ETH stays on router.
router.exactInputSingle{value: 1 ether}(params); // victim tx

// Attacker (separate tx, no special role):
router.refundETH(); // attacker receives 0.1 ETH
// OR
router.sweepToken(address(someToken), 0, attacker); // drains any ERC20 balance
```

A Foundry test can confirm this by: (1) deploying the router with a mock WETH and pool, (2) calling `exactInputSingle{value: 1 ether}` with a swap that consumes only 0.9 ETH, (3) asserting `address(router).balance == 0.1 ether`, (4) calling `router.refundETH()` from a different address, and (5) asserting the attacker received 0.1 ETH.
