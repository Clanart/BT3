### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` and `starknet/src/omni_bridge.cairo::init_transfer()` both pull tokens from the caller using the caller-supplied `amount` parameter and then emit `InitTransfer` with that same `amount` — without measuring the actual balance received. For fee-on-transfer ERC20 tokens, the bridge locks less than `amount` but the event records `amount`. The NEAR prover reads the event's `amount` field verbatim and uses it to release/mint tokens on the destination chain, creating unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer()`, the non-bridge, non-custom-minter ERC20 path is:

```solidity
// OmniBridge.sol lines 406–412
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← caller-supplied, not verified against actual receipt
);
```

Immediately after, the function emits:

```solidity
// OmniBridge.sol lines 427–436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← same caller-supplied value, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
```

No balance snapshot is taken before/after the transfer to verify the actual amount received. For a fee-on-transfer token, the bridge receives `amount - transferFee` but the event records `amount`.

The NEAR-side EVM prover parses this event and maps the `amount` field directly into `InitTransferMessage.amount`:

```rust
// near/omni-types/src/evm/events.rs line 126
amount: near_sdk::json_types::U128(event.data.amount),
```

The NEAR bridge then uses `transfer_message.amount` to unlock/mint tokens to the recipient on the destination chain. The full `amount` is released, but the EVM bridge only holds `amount - transferFee`.

The identical pattern exists on StarkNet:

```cairo
// starknet/src/omni_bridge.cairo lines 303–306
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
// ... emits InitTransfer { amount, ... } — no actual-receipt check
```

---

### Impact Explanation

Each `initTransfer` call with a fee-on-transfer token creates a deficit: the bridge holds `amount - transferFee` but the destination chain releases `amount`. Repeated calls progressively drain the bridge's collateral. Eventually, legitimate users bridging back to EVM (or StarkNet) will find the bridge insolvent — their `finTransfer` calls will fail because the bridge no longer holds enough tokens to cover outstanding obligations. This is a direct break of bridge collateralization and falls under the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

Fee-on-transfer tokens exist in production (USDT with fee enabled, certain rebasing/deflationary tokens). The EVM `initTransfer` path has no on-chain whitelist preventing a fee-on-transfer token from being used — any ERC20 that has been registered in the NEAR token mapping can be bridged. An unprivileged user can trigger this by simply calling `initTransfer` with a registered fee-on-transfer token. No special role or leaked key is required.

---

### Recommendation

Measure the actual received amount using a balance snapshot before and after the `safeTransferFrom` call, and use the delta as the canonical bridged amount in the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "ZeroReceived");
uint128 canonicalAmount = uint128(actualReceived);
// use canonicalAmount in emit and initTransferExtension
```

Apply the same fix to `starknet/src/omni_bridge.cairo::init_transfer()`. Alternatively, maintain an explicit allowlist of supported tokens and reject any token known to have transfer fees.

---

### Proof of Concept

1. A fee-on-transfer ERC20 token `FeeToken` (1% fee on every transfer) is registered in the bridge's token mapping on both EVM and NEAR.
2. Attacker calls `OmniBridge.initTransfer(FeeToken, 1_000_000, 0, 0, "near:attacker.near", "")`.
3. `safeTransferFrom` moves `1_000_000` tokens from attacker; due to the 1% fee, the bridge receives `990_000`.
4. `InitTransfer` event is emitted with `amount = 1_000_000`.
5. NEAR prover reads the event; `InitTransferMessage.amount = 1_000_000`.
6. NEAR bridge releases `1_000_000` tokens (minus any bridge fee) to `attacker.near`.
7. The EVM bridge is now short `10_000` tokens per call. After 100 such calls the bridge is short `1_000_000` tokens.
8. Legitimate users attempting to bridge back to EVM via `finTransfer` will find the bridge unable to pay out, permanently freezing their funds.
