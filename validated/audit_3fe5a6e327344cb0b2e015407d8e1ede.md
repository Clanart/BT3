### Title
Missing Zero Address Validation for `nearBridgeDerivedAddress_` in `initialize` Enables Signature Verification Bypass ŌĆö (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `initialize` function in `OmniBridge.sol` accepts `nearBridgeDerivedAddress_` without any zero-address guard. This address is the sole authority used to verify MPC-signed payloads in both `finTransfer` and `deployToken`. If it is set to `address(0)` at initialization time, an attacker can bypass signature verification to finalize unauthorized transfers (minting tokens to arbitrary recipients) or deploy unauthorized bridge tokens.

---

### Finding Description

`OmniBridge.initialize` stores three caller-supplied parameters with no input validation:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;   // ŌåÉ no zero-address check
    omniBridgeChainId = omniBridgeChainId_;                 // ŌåÉ no zero-value check
    ...
}
``` [1](#0-0) 

`nearBridgeDerivedAddress` is subsequently used as the sole trusted signer in both critical bridge operations:

```solidity
// deployToken
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

```solidity
// finTransfer
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

If `nearBridgeDerivedAddress` is `address(0)`:

- **OpenZeppelin v4 behavior**: `ECDSA.recover` returns `address(0)` for certain malformed or degenerate signatures (e.g., `s = 0`, `v` outside `{27,28}`). An attacker who submits such a crafted signature passes the `!= nearBridgeDerivedAddress` check because both sides equal `address(0)`.
- **OpenZeppelin v5 behavior**: `ECDSA.recover` reverts on any signature that would recover to `address(0)`, meaning every call to `finTransfer` and `deployToken` reverts permanently ŌĆö bricking the bridge entirely.

The same pattern is absent in the StarkNet constructor, where `omni_bridge_derived_address` (an `EthAddress`) and `token_class_hash` (a `ClassHash`) are also stored without zero-value guards:

```cairo
fn constructor(
    ref self: ContractState,
    omni_bridge_derived_address: EthAddress,   // ŌåÉ no zero check
    omni_bridge_chain_id: u8,                  // ŌåÉ no zero check
    token_class_hash: ClassHash,               // ŌåÉ no zero check
    default_admin: ContractAddress,            // ŌåÉ no zero check
    strk_token_address: ContractAddress,       // ŌåÉ no zero check
) {
``` [4](#0-3) 

---

### Impact Explanation

**Critical / High.**

On the EVM side, if `nearBridgeDerivedAddress` is `address(0)`:

1. **(OZ v4)** An attacker submits a crafted `signatureData` to `finTransfer` with a valid `TransferMessagePayload` of their choosing (arbitrary `recipient`, arbitrary `amount`, any `tokenAddress`). `ECDSA.recover` returns `address(0)`, matching `nearBridgeDerivedAddress`. The nonce is marked used and tokens are minted or transferred to the attacker ŌĆö **direct unauthorized mint/release of bridged assets**.

2. **(OZ v5)** Every call to `finTransfer` and `deployToken` reverts unconditionally ŌĆö **permanent freezing of all bridge settlement and token deployment**.

Both outcomes fall within the allowed impact scope:
- *Critical*: Unauthorized mint of bridged assets.
- *Critical*: Permanent freezing / irrecoverable lock of bridge flows.

---

### Likelihood Explanation

**Low-to-Medium.** The deployer must accidentally pass `address(0)` for `nearBridgeDerivedAddress_`. While `setNearBridgeDerivedAddress` (admin-only) exists as a recovery path, it does not prevent the vulnerability window between deployment and correction. Deployment scripts, copy-paste errors, or misconfigured CI pipelines are realistic causes. The original M-02 report was accepted at Medium Risk for the identical root cause pattern. [5](#0-4) 

---

### Recommendation

Add explicit zero-address and zero-value guards in `initialize`:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    require(nearBridgeDerivedAddress_ != address(0), "ERR_ZERO_BRIDGE_ADDRESS");
    require(omniBridgeChainId_ != 0, "ERR_ZERO_CHAIN_ID");
    // tokenImplementationAddress_ may legitimately be set later, but
    // if it must be set at init time, add: require(tokenImplementationAddress_ != address(0), ...)
    ...
}
```

Apply the same pattern to the StarkNet constructor for `omni_bridge_derived_address`, `token_class_hash`, `default_admin`, and `strk_token_address`.

---

### Proof of Concept

**Setup (OZ v4 path):**

1. Deployer calls `initialize(someImpl, address(0), 1)` ŌĆö accidentally passes zero for `nearBridgeDerivedAddress_`.
2. Attacker constructs a `TransferMessagePayload` with:
   - `recipient = attacker_address`
   - `tokenAddress = any_bridge_token`
   - `amount = 1_000_000e18`
   - `destinationNonce = 1` (unused)
3. Attacker computes the borsh-encoded payload hash as the contract would.
4. Attacker submits `signatureData` = any 65-byte value that causes `ECDSA.recover` to return `address(0)` (e.g., `v=27, r=0, s=1` ŌĆö a degenerate signature).
5. `ECDSA.recover(hashed, signatureData)` returns `address(0)`.
6. Check `address(0) != address(0)` ŌåÆ **false** ŌåÆ no revert.
7. `completedTransfers[1] = true`.
8. `IBridgeToken(tokenAddress).mint(attacker_address, 1_000_000e18)` executes.
9. Attacker receives 1,000,000 tokens with no backing deposit on any origin chain. [6](#0-5)
