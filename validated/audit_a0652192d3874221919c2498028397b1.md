### Title
Gateway Skips `__validate__` Entry Point for Invoke Transactions Targeting Undeployed Accounts, Admitting Signature-Less Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway unconditionally skips the `__validate__` entry point execution for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because this check is purely nonce-based and relies on mempool state rather than cryptographic proof of authorization, any unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can submit a signature-less invoke transaction from the victim's address with `nonce = 1` and have it admitted by the gateway without any `__validate__` execution. This is the direct sequencer analog of the external report's "skip signature verification for privileged addresses" pattern: a special-case condition replaces cryptographic authentication with an off-chain/mempool-state heuristic.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes `validate_tx` in the blockifier to return `Ok(None)` without calling the account's `__validate__` entry point at all: [3](#0-2) [4](#0-3) 

**The mempool check is not deploy-account-specific:**

The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but `account_tx_in_pool_or_recent_block` checks for *any* transaction from that address in the pool or any committed block record: [5](#0-4) 

**Exploit path:**

1. Victim broadcasts a `deploy_account` transaction for address `X` (nonce=0). It enters the mempool.
2. Attacker observes the mempool and sees address `X` with a pending `deploy_account`.
3. Attacker constructs an `InvokeTransaction` with `sender_address = X`, `nonce = 1`, arbitrary calldata, and an empty or invalid signature.
4. Attacker submits this to the gateway.
5. Gateway calls `run_pre_validation_checks`:
   - `validate_state_preconditions`: passes — `account_nonce=0`, `tx_nonce=1`, within allowed gap.
   - `validate_by_mempool`: passes — no duplicate hash.
   - `skip_stateful_validations`: `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
6. `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` is **never called**.
7. The gateway accepts and forwards the transaction to the mempool.
8. The attacker's invalid invoke (no valid signature) is now queued at nonce=1 for address `X`. [6](#0-5) 

**What happens at execution time:**

When the batcher executes, it uses `new_for_sequencing` which sets `validate: true`: [7](#0-6) 

So the attacker's transaction will call `__validate__` during execution and revert for standard accounts. However, the transaction was already admitted to the mempool, occupying the victim's nonce=1 slot.

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway's `__validate__` skip is the on-chain authentication gate for Starknet accounts. Bypassing it at admission time means:

- A transaction with no valid signature from address `X` is accepted by the gateway and enters the mempool. This directly matches the "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" impact category.
- The attacker occupies the victim's nonce=1 slot. The victim's legitimate first invoke after deploy is either rejected as a duplicate nonce or must win a fee-escalation race against the attacker's transaction.
- The attacker's transaction will revert during batcher execution (for standard accounts with signature-checking `__validate__`), but the admission bypass and nonce-slot griefing are real and unprivileged.
- For account contracts that do not enforce signature checks in `__validate__` (e.g., multisig contracts with permissive validation, or contracts under development), the attacker's transaction could execute successfully, causing unauthorized state changes.

### Likelihood Explanation

The attack requires only:
1. Observing the public mempool for a `deploy_account` transaction (zero privilege — mempool is public).
2. Constructing an invoke transaction with `nonce=1` from the victim's address (trivial).
3. Submitting it to the gateway before the `deploy_account` is committed (race condition, but the mempool retains addresses even after commit per the test at line 343). [8](#0-7) 

The window is wide: `account_tx_in_pool_or_recent_block` returns `true` even after the deploy_account is committed to a block, meaning the skip condition persists beyond the block boundary.

### Recommendation

Replace the mempool-state heuristic with a cryptographically sound check. Specifically:

1. **Do not skip `__validate__` based solely on nonce and mempool state.** The `__validate__` entry point is the account's own authorization gate; it must be called for every transaction regardless of deployment status.
2. If the UX goal is to allow `deploy_account + invoke` to be submitted atomically, the gateway should instead execute `__validate__` against the *prospective* post-deploy state (i.e., simulate the account class that will be deployed), rather than skipping it entirely.
3. Alternatively, require that the `deploy_account` transaction hash be provided and cryptographically verified as belonging to the same sender before granting the skip.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
