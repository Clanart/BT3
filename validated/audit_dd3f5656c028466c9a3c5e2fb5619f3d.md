### Title
Unhandled Revert in `charge_fee` Fee Transfer Causes Network Halt — (File: `execution/execute_transaction_utils.cairo`)

---

### Summary

The `charge_fee` function in `transaction_impls.cairo` executes the ERC20 fee transfer to the sequencer via `non_reverting_select_execute_entry_point_func`. That wrapper unconditionally asserts `is_reverted = 0` after calling the underlying entry point. If the fee token transfer reverts for any reason, the assertion fails, the Cairo OS program cannot complete, no valid proof can be generated for the block, and the network halts. This is the direct structural analog to the `panic(err)` pattern described in the external report.

---

### Finding Description

`charge_fee` in `transaction_impls.cairo` constructs an ERC20 `transfer` call targeting the sequencer address and dispatches it through `non_reverting_select_execute_entry_point_func`:

```cairo
// transaction_impls.cairo:160-163
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
``` [1](#0-0) 

`non_reverting_select_execute_entry_point_func` is defined as:

```cairo
// execute_transaction_utils.cairo:191-196
let revert_log = init_revert_log();
let (is_reverted, retdata_size, retdata, is_deprecated) = select_execute_entry_point_func{
    revert_log=revert_log
}(block_context=block_context, execution_context=execution_context);
assert is_reverted = 0;   // <-- hard assertion, no error path
return (retdata_size, retdata, is_deprecated);
``` [2](#0-1) 

The `assert is_reverted = 0` is the Cairo equivalent of `panic(err)`: if the assertion is false, the Cairo VM cannot satisfy the constraint, the proof cannot be generated, and the sequencer cannot finalize any block containing that transaction. There is no fallback, no error return, and no graceful degradation.

`charge_fee` is invoked unconditionally (when `max_fee != 0`) for every invoke, deploy-account, and declare transaction: [3](#0-2) [4](#0-3) [5](#0-4) 

The fee token address and sequencer address are taken directly from `block_context`: [6](#0-5) 

---

### Impact Explanation

If the fee token ERC20 `transfer` call reverts — for any reason — the `assert is_reverted = 0` constraint is unsatisfiable. The Cairo OS program cannot produce a valid proof for the block. The sequencer is unable to finalize any block that contains a fee-paying transaction. Because virtually every user transaction carries a non-zero fee, this effectively halts the entire network. This matches the **High: Network not being able to confirm new transactions (total network shutdown)** impact category.

---

### Likelihood Explanation

The fee token contract (STRK ERC20) is an upgradeable Sierra contract. Any of the following realistic conditions causes the transfer to revert and triggers the halt:

1. **Governance/upgrade path**: A governance action upgrades the fee token to an implementation that reverts on `transfer` under any condition (e.g., a paused state, a broken implementation, or a blocked recipient list that includes the sequencer address — directly mirroring the blocked-address scenario in the external report).
2. **Fee token contract bug**: A bug in the fee token contract that causes `transfer` to revert for specific amounts or callers can be triggered by any user who submits a transaction with a fee that hits the buggy code path.
3. **Sequencer address misconfiguration**: If `block_context.block_info_for_execute.sequencer_address` is set to an address that the fee token contract rejects (e.g., address zero, or a blocked address), every fee-paying transaction in every block will fail.

Unlike the L1-handler path (which has a hint-controlled `IsReverted` early-exit at line 387), `charge_fee` has no such escape hatch — the transfer is always attempted and the revert is always fatal. [7](#0-6) 

---

### Recommendation

Mirror the pattern already used for user-transaction execution: check `is_reverted` and handle it gracefully rather than asserting it away. Concretely:

- Change `non_reverting_select_execute_entry_point_func` to return `is_reverted` to its caller, or introduce a separate `charge_fee`-specific wrapper that logs the failure and skips the fee deduction (crediting zero fee) rather than asserting.
- At minimum, add a hint-controlled early-exit in `charge_fee` analogous to the `IsReverted` hint used in `execute_l1_handler_transaction`, so the sequencer can signal a failed fee transfer without causing a proof failure.

---

### Proof of Concept

1. Deploy or upgrade the STRK fee token contract to an implementation whose `transfer` entry point always reverts (e.g., by adding `assert 1 = 0` to the transfer body — achievable via a governance upgrade).
2. Submit any V3 invoke transaction with `max_l2_gas_amount > 0` (non-zero fee).
3. The sequencer calls `charge_fee`, which calls `non_reverting_select_execute_entry_point_func`.
4. `select_execute_entry_point_func` returns `is_reverted = 1`.
5. `assert is_reverted = 0` at `execute_transaction_utils.cairo:195` is unsatisfiable.
6. The Cairo OS program cannot complete; no valid STARK proof is generated for the block.
7. The sequencer cannot advance the chain — **total network shutdown**. [8](#0-7)
