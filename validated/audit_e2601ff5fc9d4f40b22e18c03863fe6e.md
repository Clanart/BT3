### Title
Unconditional ETH Transfer to Recipient in `finTransfer` Permanently Locks Funds When Recipient Is a Contract Without `receive`/`fallback` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` unconditionally executes a low-level ETH call to `payload.recipient` whenever `payload.tokenAddress == address(0)`, even when `payload.amount` is zero. If the recipient is a contract that implements neither `receive()` nor `fallback()`, the call returns `false`, the function reverts with `FailedToSendEther`, and the transfer can never be finalized on the EVM side. Because there is no on-chain recall or refund path for NEAR-originated transfers whose EVM finalization permanently fails, the user's funds are irrecoverably locked.

---

### Finding Description

In `OmniBridge.finTransfer`, the native-ETH delivery branch reads:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The call is made unconditionally regardless of whether `payload.amount` is zero or non-zero. Two distinct failure sub-cases exist:

**Sub-case A — zero amount.** When `payload.amount == 0`, `call{value: 0}("")` is still dispatched. Any contract recipient that has no `receive()` or `fallback()` function will reject this call, causing `FailedToSendEther` to revert the entire transaction.

**Sub-case B — non-zero amount.** When `payload.amount > 0`, the same revert occurs for any contract recipient without `receive()` or `fallback()`.

The nonce guard is set *before* the transfer:

```solidity
completedTransfers[payload.destinationNonce] = true;
``` [2](#0-1) 

Because the entire transaction reverts on `FailedToSendEther`, the `completedTransfers` write is also rolled back, so the nonce is never consumed. This means `finTransfer` can be retried indefinitely — but every retry will revert for the same reason, making finalization permanently impossible.

On the NEAR side, tokens are locked or burned at `init_transfer` time and stored in `pending_transfers`. The NEAR contract's refund logic (`fin_transfer_send_tokens_callback`, `is_refund_required`) only handles NEAR-to-NEAR `ft_transfer_call` failures; there is no mechanism to cancel or refund a NEAR-to-EVM transfer whose EVM-side `finTransfer` is structurally unreachable. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who bridges native ETH (or any asset whose NEAR-side representation maps to `tokenAddress == address(0)` on EVM) to a contract recipient that lacks `receive`/`fallback` will have their funds locked on NEAR forever. No admin function, no retry, and no recall path exists to recover them. The zero-amount sub-case is additionally reachable whenever the MPC signs a payload where the fee equals the full bridged amount, which the NEAR contract does not explicitly prevent.

---

### Likelihood Explanation

**Medium-to-High.** Contract-to-contract bridge interactions are common in DeFi. Many protocol contracts (multisigs, vaults, governance contracts, proxy contracts) do not implement `receive()` or `fallback()`. A user or integrating protocol that specifies such a contract as the EVM recipient of a native-ETH bridge transfer will trigger this path. The zero-amount sub-case is an additional, user-invisible trigger that requires no special attacker action.

---

### Recommendation

1. **Skip the call when `payload.amount == 0`** — add a guard before the low-level call:

```solidity
if (payload.tokenAddress == address(0)) {
    if (payload.amount > 0) {
        (bool success, ) = payload.recipient.call{value: payload.amount}("");
        if (!success) revert FailedToSendEther();
    }
}
```

2. **For non-zero amounts**, consider wrapping the ETH into WETH and transferring the ERC-20 token instead when the raw ETH call fails, or emit a claimable-ETH event that the recipient can pull later.

3. **Add a NEAR-side recall path** for NEAR-to-EVM transfers so that if EVM finalization is provably impossible (e.g., after N failed attempts), the originating user can recover their locked tokens on NEAR.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-3.0-or-later
pragma solidity ^0.8.24;

// A contract with no receive() or fallback()
contract NoReceive {
    uint256 public x;
    function doSomething() external { x = 1; }
}

// In a test:
// 1. Deploy NoReceive → address noReceive
// 2. Obtain a valid MPC signature for a TransferMessagePayload where:
//      payload.tokenAddress = address(0)   // native ETH path
//      payload.recipient    = noReceive
//      payload.amount       = 1 ether      // or 0 for zero-value sub-case
// 3. Fund OmniBridge with 1 ether
// 4. Call omniBridge.finTransfer{value: 1 ether}(sig, payload)
// 5. Transaction reverts with FailedToSendEther()
// 6. completedTransfers[nonce] remains false → retry loop → permanent lock
```

The zero-value sub-case requires `payload.amount = 0` with the same `NoReceive` recipient; the call `noReceive.call{value: 0}("")` returns `false` and triggers the identical revert path. [4](#0-3)
