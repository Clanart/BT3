### Title
Caller Must Be Both Trusted Relayer AND Fee Recipient — Permanent Fee Lock When They Differ - (File: near/omni-bridge/src/lib.rs)

### Summary

`claim_fee` enforces two independent identity checks on the same caller: (1) the `#[trusted_relayer]` macro gate, and (2) an explicit `fee_recipient == predecessor_account_id` check inside `claim_fee_callback`. When the fee recipient embedded in the on-chain proof is a different account from the trusted relayer calling the function, neither account can ever satisfy both checks simultaneously, permanently locking the fee in the bridge contract.

### Finding Description

The `claim_fee` function in `near/omni-bridge/src/lib.rs` is decorated with `#[trusted_relayer]`, which requires `env::predecessor_account_id()` to be an active trusted relayer (or hold a bypass role such as `DAO` or `UnrestrictedRelayer`). [1](#0-0) 

Inside the callback, a second independent check requires the caller to equal the `fee_recipient` field extracted from the proof: [2](#0-1) 

The `fee_recipient` is a free parameter supplied by the relayer during `sign_transfer`: [3](#0-2) 

The relayer can legitimately set `fee_recipient` to any NEAR account — for example, a shared treasury, a multisig, or a fee-splitting contract — that is not itself a trusted relayer. Once that signed payload is submitted and finalized on the destination chain, the resulting proof permanently encodes that non-relayer `fee_recipient`. At that point:

- The **fee_recipient account** cannot call `claim_fee` because it is not a trusted relayer.
- The **calling relayer** cannot call `claim_fee` because `fee_recipient != predecessor_account_id`.
- No other account can satisfy both checks simultaneously.

The pending transfer entry remains in `pending_transfers` indefinitely, and the fee tokens are irrecoverably locked inside the bridge contract.

### Impact Explanation

The fee amount (the relayer reward) is held in the bridge's locked token balance after `fin_transfer` completes. If `claim_fee` can never succeed, those tokens are permanently frozen in the contract with no recovery path available to unprivileged users. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge flows.** [4](#0-3) 

### Likelihood Explanation

The scenario is reachable by any trusted relayer without any privileged operator collusion. A relayer may set `fee_recipient` to a treasury or multisig account for operational reasons (fee aggregation, accounting separation). The protocol places no restriction on what account can be named as `fee_recipient` in `sign_transfer`. Once the destination-chain finalization proof is produced with that `fee_recipient`, the lock is irreversible at the protocol level. The only administrative escape hatch would be for the DAO to grant the fee_recipient the `UnrestrictedRelayer` role, which is an out-of-band intervention not guaranteed to occur. [5](#0-4) 

### Recommendation

Decouple the two identity requirements. One approach: allow any trusted relayer to call `claim_fee` for a proof whose `fee_recipient` is any valid NEAR account, and send the fee directly to the `fee_recipient` regardless of who the caller is. Alternatively, allow the `fee_recipient` account itself to call `claim_fee` without requiring it to be a trusted relayer, since the fee_recipient is already authenticated by the MPC-signed proof embedded in the prover result.

### Proof of Concept

1. Trusted relayer `relayer.near` calls `sign_transfer` with `fee_recipient = Some("treasury.near")`, where `treasury.near` is not a trusted relayer.
2. The MPC-signed payload encodes `fee_recipient = treasury.near`. The destination chain finalizes the transfer and emits a `FinTransfer` event with `fee_recipient = treasury.near`.
3. `relayer.near` calls `claim_fee` with the proof. The `#[trusted_relayer]` gate passes (relayer is trusted), but `claim_fee_callback` panics at `fee_recipient == *predecessor_account_id` → `"treasury.near" != "relayer.near"`.
4. `treasury.near` calls `claim_fee` with the same proof. The `#[trusted_relayer]` gate panics immediately because `treasury.near` is not a trusted relayer.
5. No account can ever claim the fee. The pending transfer entry and its associated fee tokens remain locked in the bridge contract permanently. [6](#0-5)

    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };

        self.verify_proof(args.chain_kind, args.prover_args).then(
            Self::ext(env::current_account_id())

        )
