### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Unbacked Cross-Chain Supply - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer()` does not perform a pre/post balance check when pulling native ERC20 tokens from the caller. For fee-on-transfer tokens, the contract receives fewer tokens than the `amount` parameter specifies, yet the `InitTransfer` event is emitted with the full user-supplied `amount`. The NEAR side processes this event and releases/mints the full `amount`, creating unbacked supply that cannot be redeemed when bridging back.

### Finding Description
In `OmniBridge.sol`, the `initTransfer` function handles native ERC20 tokens (those that are neither bridge tokens nor custom-minter tokens) via:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

After this transfer, the function emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-supplied, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

No balance snapshot is taken before the transfer, and no post-transfer balance check is performed to confirm the contract actually received `amount` tokens. For fee-on-transfer ERC20 tokens (tokens that deduct a percentage on every transfer), `safeTransferFrom` succeeds but the contract receives `amount - fee_deducted`. The emitted event still carries the full `amount`, which is what the NEAR bridge observes to determine how many tokens to release or mint on the destination chain.

### Impact Explanation
**High — Balance/accounting corruption that breaks bridge collateralization.**

The NEAR bridge reads the `InitTransfer` event's `amount` field and releases or mints that quantity of tokens to the recipient on NEAR. The EVM bridge, however, only holds `amount - token_fee`. When those NEAR-side tokens are later bridged back to EVM, the bridge attempts to release the full `amount` but only has `amount - token_fee` in custody. Repeated use of a fee-on-transfer token progressively drains the EVM bridge's reserves, eventually causing legitimate withdrawals to fail (permanent fund lock) or, if the bridge mints bridged tokens on NEAR, creates unbacked supply (unauthorized mint).

### Likelihood Explanation
**Low-Medium.** The bridge is permissionless for native ERC20 tokens — any caller can invoke `initTransfer` with any ERC20 address. Fee-on-transfer tokens are a well-known token class (e.g., deflationary tokens, tokens with redistribution mechanics). No whitelist or token-type guard prevents their use in the `else` branch. An attacker or ordinary user can trigger this discrepancy without any privileged access.

### Recommendation
Record the contract's token balance before and after the `safeTransferFrom` call, and use the actual received amount (post-balance minus pre-balance) as the canonical transfer amount for both the event and the cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "FEE_ON_TRANSFER_NOT_SUPPORTED");
// or use actualReceived as the bridged amount
```

This mirrors the recommendation in the external report: check the post-balance and compare it to the sum of the pre-balance and the requested amount.

### Proof of Concept
1. Deploy or identify a fee-on-transfer ERC20 token `T` that deducts 1% on every transfer.
2. Call `OmniBridge.initTransfer(T, 1000, 0, 0, "recipient.near", "")` with 1000 tokens approved.
3. `safeTransferFrom` moves 1000 from the caller; the token contract deducts 10 (1%) as a fee, so `OmniBridge` receives 990.
4. The `InitTransfer` event is emitted with `amount = 1000`.
5. The NEAR bridge observes the event and releases/mints 1000 tokens to `recipient.near`.
6. The EVM bridge now holds 990 tokens but has a 1000-token liability.
7. Repeating this inflates the NEAR-side supply beyond what the EVM bridge can back, breaking collateralization and eventually causing withdrawal failures for legitimate users. [3](#0-2)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

            amount,
