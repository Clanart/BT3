### Title
Factory Address Overwrite in `add_factory` Permanently Freezes Pending Fee Claims and Corrupts Locked-Token Accounting — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`add_factory` silently overwrites the registered factory address for a chain with no migration mechanism for in-flight transfers. After the overwrite, `claim_fee_callback` permanently rejects every fee-claim proof whose `emitter_address` matches the **old** factory, leaving those transfer messages stranded in `pending_transfers` forever and locking the associated relayer fees inside the bridge contract. The same overwrite also blocks any future `fin_transfer_callback` that carries the old factory's emitter address, freezing the corresponding locked-token accounting.

---

### Finding Description

**Root cause — `add_factory` silently overwrites:**

```rust
#[access_control_any(roles(Role::DAO))]
pub fn add_factory(&mut self, address: OmniAddress) {
    self.factories.insert(&(&address).into(), &address);   // overwrites silently
}
```

`LookupMap::insert` returns the displaced value but the function discards it. There is no guard, no event, and no migration path for transfers that were already in flight under the old factory address.

**Gating check in `claim_fee_callback`:**

```rust
require!(
    self.factories
        .get(&fin_transfer.emitter_address.get_chain())
        == Some(fin_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
```

After `add_factory(0xNEW)` is called, `self.factories.get(&Eth)` returns `0xNEW`. Any `FinTransferMessage` whose `emitter_address` is `0xOLD` will fail this check unconditionally.

**Gating check in `fin_transfer_callback` (same pattern):**

```rust
require!(
    self.factories
        .get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
```

New `fin_transfer` calls carrying the old factory address are also permanently rejected.

**Locked-token accounting set at transfer initiation:**

```rust
// init_transfer_internal
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

`locked_tokens[(EVM, token)]` is incremented when a NEAR → EVM transfer is initiated. It is only decremented inside `process_fin_transfer_to_near` (for the reverse leg) or inside `send_fee_internal` (called from `claim_fee_callback`). If `claim_fee_callback` never succeeds, the counter is never decremented, leaving the accounting permanently inflated.

**No escape hatch:** There is no public function to force-remove a `pending_transfers` entry or to manually decrement `locked_tokens` without DAO intervention via `set_locked_tokens`.

---

### Impact Explanation

| Asset | Effect |
|---|---|
| Relayer fees (NEAR-side token) | Permanently locked inside the bridge contract; `claim_fee_callback` panics before `send_fee_internal` is reached |
| `pending_transfers` entries | Remain in storage forever; no mechanism to purge them |
| `locked_tokens[(EVM, token)]` | Remains inflated; future `unlock_tokens` calls for the reverse leg may succeed against an overstated balance, creating an undercollateralization risk |

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

The DAO has a legitimate operational reason to rotate a factory address (security incident on the old factory contract, protocol upgrade, chain migration). The function `add_factory` is the only mechanism provided for this; there is no `update_factory` or `remove_factory`. Any routine factory rotation silently triggers the freeze for all in-flight transfers that were signed under the old address. The window between factory rotation and fee-claim submission can span hours or days (MPC signing latency, relayer queue depth).

---

### Recommendation

1. **Prevent silent overwrite:** Require that no factory is currently registered for the chain, or emit a warning and require explicit acknowledgement.
2. **Maintain a factory history:** Store a `LookupMap<(ChainKind, OmniAddress), bool>` of ever-valid factories and accept proofs from any historically valid factory.
3. **Provide a migration window:** Before overwriting, allow a grace period during which `claim_fee` still accepts the old factory address.
4. **Add a DAO escape hatch:** Provide a privileged function to forcibly settle or cancel stranded `pending_transfers` entries and correct `locked_tokens`.

---

### Proof of Concept

```
State before:
  factories[Eth] = 0xOLD
  pending_transfers[{Near, nonce=42}] = TransferMessage { fee: 100, ... }
  locked_tokens[(Eth, usdc.near)] = 1000

Step 1: User initiates NEAR → EVM transfer (nonce 42, fee 100 USDC).
        init_transfer_internal → locked_tokens[(Eth, usdc.near)] += 1000

Step 2: Relayer calls sign_transfer → MPC signs with 0xOLD as emitter.

Step 3: Relayer calls finTransfer on EVM (factory = 0xOLD) → tokens minted on EVM.

Step 4: DAO calls add_factory(0xNEW).
        factories[Eth] = 0xNEW   ← 0xOLD is gone

Step 5: Relayer calls claim_fee with proof:
          FinTransferMessage { emitter_address: 0xOLD, transfer_id: {Near, 42}, ... }

Step 6: claim_fee_callback executes:
          self.factories.get(&Eth) == Some(0xNEW)
          fin_transfer.emitter_address == 0xOLD
          0xNEW != 0xOLD  →  panic("ERR_UNKNOWN_FACTORY")

Result:
  - pending_transfers[{Near, 42}] is never removed.
  - 100 USDC fee is permanently locked in the bridge contract.
  - locked_tokens[(Eth, usdc.near)] remains 1000 (never decremented).
  - No recovery path exists without another DAO intervention.

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
