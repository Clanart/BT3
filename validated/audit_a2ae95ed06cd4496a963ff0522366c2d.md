### Title
Blacklisted Recipient Permanently Freezes Bridged Funds in `finTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` pushes tokens directly to `payload.recipient` in a single atomic call. If the recipient address is blacklisted by the token contract (e.g., USDC, USDT), every finalization attempt reverts. Because the recipient is cryptographically bound inside the MPC-signed payload and no cancel/redirect mechanism exists on the NEAR side, the origin-chain funds are permanently frozen with no recovery path.

---

### Finding Description

In `OmniBridge.finTransfer`, the nonce is marked used and the signature is verified, then tokens are pushed directly to `payload.recipient`:

```solidity
// OmniBridge.sol line 287 – nonce consumed (reverts with tx if transfer fails)
completedTransfers[payload.destinationNonce] = true;
...
// line 351-354 – native ERC20 path
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
```

`SafeERC20.safeTransfer` wraps the call and reverts if the underlying `transfer` returns `false` or reverts. USDC and USDT maintain on-chain blacklists; a `transfer` to a blacklisted address reverts unconditionally.

Because the entire transaction reverts, `completedTransfers[nonce]` is also rolled back, so the nonce is not permanently consumed. However, the recipient address is Borsh-encoded into the signed message at line 298:

```solidity
Borsh.encodeAddress(payload.recipient),
```

The MPC signature covers this exact byte sequence. No alternative recipient can be substituted without invalidating the signature, and the NEAR bridge has no mechanism to issue a replacement signature for a different recipient on the same transfer. The NEAR-side tokens are already locked or burned at the point the MPC signature was issued. There is no cancel/refund path visible in the NEAR bridge contract for a transfer whose EVM finalization permanently fails.

The same pattern exists in `starknet/src/omni_bridge.cairo` `fin_transfer` (lines 260–262):

```cairo
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');
```

A blacklisted recipient causes `transfer` to return `false`, the `assert` panics, the transaction reverts, and the same permanent-lock outcome applies.

The ETH path (lines 319–322) has an analogous issue: if `payload.recipient` is a contract that reverts on `receive`, `FailedToSendEther` is raised and the transfer can never be finalized.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

A user who bridges USDC (or any blacklistable token) from NEAR to EVM and whose EVM recipient address is subsequently blacklisted by the token issuer will have their funds permanently frozen:

- Every `finTransfer` call reverts; the nonce is never consumed.
- The signed payload cannot be altered (recipient is inside the signature).
- No new MPC signature can be obtained for a different recipient for the same transfer.
- The NEAR-side tokens are already locked/burned; no refund mechanism exists.

The funds are irrecoverable without a protocol-level upgrade or admin intervention.

---

### Likelihood Explanation

USDC and USDT are among the most commonly bridged tokens. Circle and Tether regularly blacklist addresses (sanctions compliance, hack response). A user whose address is blacklisted after initiating a bridge transfer but before finalization — a window that can span minutes to hours depending on relayer latency — will trigger this condition. No attacker action is required; the blacklisting is performed by the token issuer as a routine compliance action.

---

### Recommendation

Replace the push-payment pattern with a pull-payment (escrow) pattern for the EVM native-ERC20 path:

1. Instead of `safeTransfer(recipient, amount)`, credit `pendingWithdrawals[recipient][token] += amount`.
2. Add a `withdraw(address token)` function that lets the recipient pull their funds.
3. This decouples finalization success from the recipient's token-acceptance state, matching the fix recommended in the reference report.

Alternatively, wrap the transfer in a try/catch and, on failure, store the amount in a claimable mapping so the nonce is consumed and the transfer is considered finalized (preventing replay), while still allowing the recipient to claim via a different mechanism or address.

---

### Proof of Concept

1. Alice holds USDC on NEAR and calls `ft_on_transfer` to bridge 10,000 USDC to her EVM address `0xAlice`. NEAR locks the tokens and requests an MPC signature.
2. The MPC produces a signature over a `TransferMessagePayload` with `recipient = 0xAlice`.
3. Before the relayer submits `finTransfer`, Circle blacklists `0xAlice` (e.g., OFAC sanction).
4. Relayer calls `OmniBridge.finTransfer(sig, payload)`. Execution reaches line 351:
   ```solidity
   IERC20(usdcAddress).safeTransfer(0xAlice, 10000e6);
   ```
   USDC's `transfer` reverts because `0xAlice` is blacklisted. `safeTransfer` propagates the revert. The entire transaction reverts.
5. The relayer retries — same result every time.
6. No alternative payload can be constructed: changing `recipient` to `0xAlice2` invalidates the MPC signature (line 311 check fails).
7. The NEAR bridge has no cancel/refund entry point for this transfer.
8. Alice's 10,000 USDC is permanently locked in the NEAR bridge contract.
