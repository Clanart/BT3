### Title
Missing class-body-to-class-hash binding in p2p deprecated class sync allows a malicious peer to store a wrong class body under a committed class hash — (`crates/apollo_p2p_sync/src/client/class.rs`, `crates/apollo_protobuf/src/converters/class.rs`)

### Summary
A malicious p2p sync peer can send a `protobuf::ClassesResponse` containing a `Cairo0Class` whose program field encodes body_A, wrapped in a `protobuf::Class` with `class_hash = H_B` (a hash that legitimately appears in the committed state diff). The sync client accepts the message because `H_B` is in the state diff, but never verifies `compute_deprecated_class_hash(body_A) == H_B`. Body_A is then stored in the class manager under key `H_B`, permanently corrupting the class storage for that hash.

### Finding Description
The `TryFrom<protobuf::Class> for (ApiContractClass, ClassHash)` conversion extracts the class body and the class hash independently and returns them as a tuple without any binding check. [9](#0-8) 

`parse_data_for_block` validates only that the peer-supplied `class_hash` is present in the state diff's `deprecated_declared_classes` list (a `Vec<ClassHash>` containing only hashes, not bodies). It does not call `compute_deprecated_class_hash` on the received body. [10](#0-9) 

`add_deprecated_class` and `FsClassStorage::set_deprecated_class` store the body verbatim under the supplied `class_id` with no hash recomputation. [7](#0-6) 

### Impact Explanation
After the corrupt write, `get_executable(H_B)` returns body_A. This affects:

- **RPC `starknet_getClass`**: returns body_A for class hash `H_B` — an authoritative-looking wrong class body.
- **Execution**: any contract deployed under class `H_B` will execute body_A's bytecode, producing wrong state transitions, wrong return values, and wrong fee accounting.
- **SNOS/proof**: `load_deprecated_class` does recompute the hash and would detect the mismatch during proof generation, but by then the class storage is already corrupted and the node's RPC and execution paths have already served wrong data. [11](#0-10) 

### Likelihood Explanation
Any node that participates in p2p sync and connects to a malicious peer is vulnerable. The attacker only needs to know a valid `ClassHash` from the committed state diff (publicly observable) and craft a `Cairo0Class` message with a mismatched program. No operator or admin privileges are required.

### Recommendation
After `TryFrom<protobuf::Cairo0Class>` succeeds, compute `compute_deprecated_class_hash(&body)` and compare it to the `class_hash` extracted from the outer `protobuf::Class`. If they differ, return a `BadPeerError` and report the peer. This check should be added in `parse_data_for_block` in `crates/apollo_p2p_sync/src/client/class.rs`, consistent with the existing TODO comment at line 39. [8](#0-7)
