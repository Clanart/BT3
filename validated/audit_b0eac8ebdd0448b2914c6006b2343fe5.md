### Title
Cross-Chain Replay of `deployToken` Metadata Signature Enables Unauthorized Token Deployment on Any EVM Chain — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies a MPC-produced signature over a metadata payload that does **not** include the destination chain identifier (`omniBridgeChainId`). Because all EVM deployments of OmniBridge share the same `nearBridgeDerivedAddress` (derived from the same NEAR MPC key), a valid `deployToken` signature obtained for one EVM chain (e.g., Ethereum) can be replayed verbatim on any other EVM chain (Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract) to deploy the same bridge token there without authorization.

---

### Finding Description

In `OmniBridge.sol::deployToken`, the signed payload is constructed as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The hash covers only `(PayloadType, token, name, symbol, decimals)` — **no `omniBridgeChainId` is included**.

Contrast this with `finTransfer`, which correctly binds the signature to the destination chain by embedding `omniBridgeChainId` twice in the hash:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
    Borsh.encodeUint64(payload.destinationNonce),
    bytes1(payload.originChain),
    Borsh.encodeUint64(payload.originNonce),
    bytes1(omniBridgeChainId),          // <-- chain binding
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),          // <-- chain binding (recipient side)
    ...
``` [2](#0-1) 

The NEAR-side `MetadataPayload` struct that is signed by the MPC signer also contains no chain field:

```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
``` [3](#0-2) 

The only replay guard in `deployToken` is:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [4](#0-3) 

This prevents double-deployment **on the same chain** but does nothing to prevent the same signature from being submitted to a different chain's OmniBridge instance.

---

### Impact Explanation

An attacker who observes a valid `deployToken` signature (published in a NEAR event after `sign_log_metadata`) for chain A can submit it to chain B's OmniBridge contract. The result:

1. A bridge token for the NEAR token ID is deployed on chain B and registered in `isBridgeToken` / `nearToEthToken` / `ethToNearToken`.
2. Users can call `initTransfer` on chain B using this token — their ERC-20 tokens are burned or locked by the bridge.
3. The NEAR bridge has no corresponding `token_id_to_address` entry for chain B (no `bind_token` was ever called for chain B), so `sign_transfer` will panic with `BridgeError::FailedToGetTokenAddress`.
4. The transfer can never be finalized; user funds are **permanently locked** in the bridge.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows**, and also **Proof/signature verification bypass enabling unauthorized token deployment**. [5](#0-4) 

---

### Likelihood Explanation

- All EVM OmniBridge deployments share the same `nearBridgeDerivedAddress` (same NEAR MPC key path `"bridge-1"`).
- `deployToken` signatures are published publicly in NEAR transaction logs.
- The attack requires no privileged access: any unprivileged user can call `deployToken` on any EVM chain with a signature observed from another chain.
- The only precondition is that the target chain's OmniBridge has not yet deployed the same token (the `ERR_TOKEN_EXIST` guard). For tokens deployed on Ethereum but not yet on Arbitrum/Base/BNB/Polygon/etc., the attack is immediately executable.

---

### Recommendation

Include `omniBridgeChainId` in the metadata signature hash, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR-side `MetadataPayload` and `sign_log_metadata` must be updated correspondingly to include the destination chain kind in the signed payload, so that a signature produced for Ethereum cannot be accepted by Arbitrum.

---

### Proof of Concept

1. Token `usdc.near` is legitimately deployed on Ethereum via `deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})`. The signature `sig_eth` is published in the NEAR `LogMetadataEvent`.

2. Attacker calls `deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on the Arbitrum OmniBridge. The `ECDSA.recover` check passes because the hash is identical (no chain ID in the payload). `ERR_TOKEN_EXIST` does not trigger because `nearToEthToken["usdc.near"]` is zero on Arbitrum.

3. A new `BridgeToken` proxy is deployed on Arbitrum and registered as a bridge token for `"usdc.near"`.

4. A user calls `initTransfer(arb_usdc_proxy, 1000e6, ...)` on Arbitrum. Their USDC is burned. An `InitTransfer` event is emitted.

5. A relayer submits the proof to NEAR's `fin_transfer`. Inside `fin_transfer_callback`, `self.get_token_address(ChainKind::Arb, "usdc.near")` returns `None` (no `bind_token` was ever called for Arb), causing a panic at `BridgeError::FailedToGetTokenAddress` — or alternatively the transfer is stored but `sign_transfer` later panics at the same point.

6. The user's 1000 USDC is permanently locked with no path to recovery. [6](#0-5) [5](#0-4)

    function deployToken(
        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

        // slither-disable-next-line reentrancy-no-eth
        address bridgeTokenProxy = address(
            new ERC1967Proxy(
                tokenImplementationAddress,
                abi.encodeWithSelector(
                    BridgeToken.initialize.selector,
                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )
        );

        deployTokenExtension(
            metadata.token,
            bridgeTokenProxy,
            decimals,
            metadata.decimals
        );

        emit BridgeTypes.DeployToken(
            bridgeTokenProxy,
            metadata.token,
            metadata.name,
            metadata.symbol,
            decimals,
            metadata.decimals
        );

        isBridgeToken[address(bridgeTokenProxy)] = true;
        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
        nearToEthToken[metadata.token] = address(bridgeTokenProxy);

        return bridgeTokenProxy;
    }
