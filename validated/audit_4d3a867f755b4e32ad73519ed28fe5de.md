### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Arbitrary Class Hash Assignment — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` does not verify that the new class hash supplied via the `replace_class` syscall corresponds to a class that has actually been declared on-chain. The missing check is explicitly acknowledged by a developer TODO comment. Because the OS is the authoritative proof-level enforcer, this omission means the protocol does not prevent a contract from replacing its own class hash with an arbitrary, undeclared value, permanently breaking the contract's executability and freezing any funds it holds.

---

### Finding Description

The vulnerability class from the external report is a **missing validation check on a critical state-transition operation**: `AddressSet.add()` silently returns `false` instead of reverting, and the caller never checks the return value, allowing an invalid state mutation to proceed. The analog here is structurally identical: the OS performs a critical state mutation (updating a contract's `class_hash`) without checking a prerequisite condition (that the new hash is declared), so the invalid state transition is accepted silently.

In `execute_replace_class`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The function reads `request.class_hash` and immediately writes it into `contract_state_changes` with no lookup into `contract_class_changes` (the declared-classes dictionary). The TODO at line 898 is an explicit, in-code acknowledgment that this check is absent. [2](#0-1) 

For comparison, the normal declare flow in `execute_declare_transaction` enforces `prev_value=0` (class declared at most once) and writes the compiled class hash into `contract_class_changes`:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [3](#0-2) 

`execute_replace_class` performs no equivalent lookup. The OS is the Cairo program that generates the STARK proof accepted by the L1 verifier; it is the sole protocol-level enforcement mechanism. If the OS does not enforce a rule, the rule is not enforced at the proof level, regardless of what the sequencer's off-chain execution engine does.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` in `contract_state_changes` is set to an undeclared value, no compiled class exists for it. Every subsequent call to that contract will fail at class resolution. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible. The state commitment written to L1 reflects the corrupted class hash, making the freeze irreversible without a protocol upgrade.

---

### Likelihood Explanation

The `replace_class` syscall is reachable by any deployed contract; no privileged role is required to invoke it. The OS's missing check means the proof-level protocol does not reject the invalid state transition. An attacker can:

- Deploy a contract that calls `replace_class` with an arbitrary felt value as the class hash.
- If the sequencer's execution engine also lacks this validation (consistent with the OS TODO indicating the check is a future work item across the stack), the transaction executes successfully end-to-end.
- Even if the sequencer's engine rejects it, a sequencer that is compromised or buggy would produce a valid OS proof for the invalid state, which the L1 verifier would accept.

The explicit TODO comment signals that this is a known, unimplemented guard — not a design choice — increasing the likelihood that the gap exists in practice.

---

### Recommendation

Inside `execute_replace_class`, after reading `request.class_hash`, perform a lookup into `contract_class_changes` to confirm the hash has a non-zero compiled class entry before writing the new `StateEntry`. Concretely:

```cairo
let class_hash = request.class_hash;

// Verify the class is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the enforcement already present in the declare transaction path and closes the gap acknowledged by the TODO.

---

### Proof of Concept

1. Attacker deploys contract `A` holding user funds.
2. Contract `A` executes the `replace_class` syscall with `class_hash = 0xdeadbeef` (never declared).
3. `execute_replace_class` in `syscall_impls.cairo` passes the gas check, skips the missing declared-class check, and calls `dict_update` to set `contract_state_changes[A].class_hash = 0xdeadbeef`.
4. The OS generates a valid STARK proof for this block; the L1 verifier accepts it.
5. On-chain state now records contract `A` with `class_hash = 0xdeadbeef`.
6. Any future `call_contract` or `library_call` targeting `A` attempts to resolve compiled class `0xdeadbeef` from `contract_class_changes`, finds zero (undeclared), and fails.
7. All funds in `A`'s storage are permanently frozen. [4](#0-3)

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
