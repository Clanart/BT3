### Title
Implicit Nonce Constraint Violation in Batched EVM CREATE Transactions Causes Silent Contract Deployment Failure and Gas Loss - (File: x/evm/keeper/state_transition.go)

### Summary

Ethermint's Cosmos SDK batch transaction support introduces an implicit ordering/sequencing constraint on batched `MsgEthereumTx` CREATE messages: nested contract creations within one batch message must not collide with the intended deployment addresses of subsequent batch messages. This constraint is not enforced anywhere in the code. When violated, subsequent CREATE transactions in the same batch fail silently at the EVM level (`vmErr` is set), the Cosmos transaction still commits successfully, and the user's gas fees are fully consumed without deploying the intended contract.

### Finding Description

**Root cause — two-phase nonce management with no collision guard:**

**Phase 1 — Ante handler pre-increments all batch nonces upfront.**

`CheckAndSetEthSenderNonce` in `ante/eth.go` iterates over every `MsgEthereumTx` in the batch and increments the account sequence for each one before any EVM execution begins:

```go
for _, msg := range tx.GetMsgs() {
    // ...
    if err := acc.SetSequence(expectedNonce + 1); err != nil { ... }
    ak.SetAccount(ctx, acc)
}
```

For a batch of 3 messages from sender with nonce N, after the ante handler the on-chain sequence is N+3. [1](#0-0) 

**Phase 2 — `ApplyMessageWithConfig` resets and reconciles the nonce per-message.**

For each CREATE message, the code resets the stateDB nonce back to `msg.Nonce`, calls `evm.Create()`, then reconciles nested-create increments:

```go
oldNonce := stateDB.GetNonce(sender)          // post-ante value, e.g. N+3
stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to msg.Nonce, e.g. N
ret, _, leftoverGas, vmErr = evm.Create(...)
afterCreateNonce := stateDB.GetNonce(sender)
nestedCreates := afterCreateNonce - msg.Nonce - 1
stateDB.SetNonce(sender, oldNonce+nestedCreates, ...)
``` [2](#0-1) 

**The collision scenario:**

Consider a batch of two CREATE messages from sender A at nonce N:

| Step | Event | Nonce |
|------|-------|-------|
| Ante handler | Pre-increments both messages | N → N+2 |
| tx0 (msg.Nonce=N) | `evm.Create()` deploys parent (nonce N→N+1) + nested child (N+1→N+2) | N+2 |
| tx0 reconcile | `nestedCreates=1`, sets nonce to `(N+2)+1 = N+3` | N+3 |
| tx1 (msg.Nonce=N+1) | Reset to N+1; `evm.Create()` targets `address(A, N+1)` — **already occupied by tx0's nested child** | collision |
| tx1 result | `vmErr = ErrContractAddressCollision`; nonce increments to N+2; `nestedCreates=0`; final nonce = N+3 | N+3 |

tx1's CREATE fails silently. The Cosmos transaction commits with `code=0`. The user's gas for tx1 is fully consumed. No contract is deployed at the address the user intended.

This is explicitly acknowledged in ADR-003 as a known design limitation:

> "Both transactions attempt to create a contract at an address derived from nonce 1, causing a conflict." [3](#0-2) 

However, the constraint is **never enforced in production code**. The protocol continues to accept and execute such batches, committing the gas deduction while silently discarding the deployment.

The same class of implicit constraint applies to batched EIP-7702 self-authorization + CREATE combinations, where the authorization nonce must be pre-calculated accounting for all prior ante-handler increments and prior auth increments in the same batch — a calculation that is entirely the caller's responsibility with no on-chain validation: [4](#0-3) 

### Impact Explanation

**Allowed impact matched:** *"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

- The Cosmos transaction commits successfully (`code=0`) even though the EVM-level CREATE failed.
- The user's gas fees are fully deducted and not refunded.
- The intended contract is never deployed; any funds sent with the CREATE call (`msg.Value`) are returned, but the gas cost is permanently lost.
- The sender's nonce is advanced beyond what the user expects (N+3 instead of N+2), requiring all subsequent transactions to use the higher nonce or they will be rejected.

In the EIP-7702 self-authorization batch variant, an incorrectly signed authorization nonce causes the authorization to be silently skipped (`applyAuthorization` errors are logged and ignored), meaning the delegation is never set while gas is still consumed: [5](#0-4) [6](#0-5) 

### Likelihood Explanation

Batch `MsgEthereumTx` transactions are a supported, documented Cosmos SDK feature in Ethermint. Any unprivileged user can submit them via `eth_sendRawTransaction` (wrapped in a Cosmos batch) or directly via the Cosmos broadcast endpoint. The protocol imposes no restriction on batch CREATE transactions. A user building a deployment script that batches a factory deployment (which internally deploys children) alongside further deployments will hit this silently. The constraint is documented only in an ADR, not surfaced to callers at the protocol level.

### Recommendation

1. **Enforce the constraint at the ante handler level:** Before accepting a batch of CREATE messages, simulate the nonce consumption of each message (including nested creates) and reject the batch if any two messages would target the same contract address.

2. **Alternatively, reject batch transactions that mix CREATE messages from the same sender**, since Ethereum has no equivalent concept and the behavior cannot be validated against go-ethereum.

3. **At minimum, emit a Cosmos-level event or set a non-zero receipt status** when `vmErr != nil` inside `ApplyTransaction`, so callers can detect the silent failure rather than observing a `code=0` response with no deployed contract. [7](#0-6)

When multiple CREATE transactions are batched, nonce resets can cause address collisions:

| Transaction | Initial Nonce | Contracts Created | Nonces Used | Issue |
|-------------|---------------|-------------------|-------------|-------|

| tx1 (msg.Nonce=1) | 1 (after tx0) | 1 | 1 | Reset to 1, creates at nonce 1 (collision!) |

**Result**: Both transactions attempt to create a contract at an address derived from nonce 1, causing a conflict.
