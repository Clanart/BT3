## Analysis

The claim is traceable through real scoped code. Let me walk the full path.

**How ETH reaches the router:**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` are all `external payable`. [1](#0-0) 

The `receive()` guard blocks direct ETH sends from non-WETH addresses, but it does **not** block `msg.value` arriving via a `payable` function call. [2](#0-1) 

So if a user calls `exactInputSingle({tokenIn: WETH, amountIn: X})` with `msg.value = Y > X`, the callback's `pay()` wraps exactly `X` ETH and sends it to the pool. The remaining `Y - X` ETH is **never refunded automatically** and stays on the router.

**The `pay()` hybrid logic:**

```solidity
} else if (token == WETH) {
  uint256 nativeBalance = address(this).balance;   // ← total router ETH, not msg.value
  if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // ← attacker pays 0 from own wallet
  } else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // ← partial subsidy
  } else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
  }
``` [3](#0-2) 

The critical flaw: `pay()` reads `address(this).balance` — the **total** router ETH balance — not `msg.value` of the current transaction. Any ETH stranded from a prior transaction is indistinguishable from ETH the current caller sent.

**The attack path:**

1. Victim calls `exactInputSingle({tokenIn: WETH, amountIn: X})` with `msg.value = Y > X` (or any payable swap entry) without a subsequent `refundETH()`. `Y - X` ETH is stranded on the router.
2. Attacker calls `exactInputSingle({tokenIn: WETH, amountIn: Z})` where `Z ≤ router.balance`.
3. In `_justPayCallback`, `pay()` is called with `payer = attacker`, `token = WETH`, `value = Z`. [4](#0-3) 
4. `nativeBalance >= Z`, so the router wraps the stranded ETH and sends it to the pool. The attacker's own WETH/ETH balance is untouched. The attacker receives swap output for free.

**Guards that do NOT prevent this:**

- `receive()` blocks direct ETH sends but not `msg.value` on payable calls.
- The callback caller check (`_requireExpectedCallbackCaller`) only validates the pool identity, not the payment source. [5](#0-4) 
- `refundETH` is opt-in and not enforced. [6](#0-5) 

**Severity:**

The root cause is a design flaw (`address(this).balance` vs. `msg.value`), not purely user error. The precondition (excess ETH left on router) is a realistic and common pattern — users routinely send ETH with WETH swaps and rely on `refundETH` in a multicall, but a single-call invocation of any payable swap function with excess ETH silently strands it. The attacker needs no special access, no malicious pool, and no oracle manipulation.

### Title
Stranded router ETH stolen via `pay()` WETH hybrid — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` uses `address(this).balance` (total router ETH) rather than the current transaction's `msg.value` when paying WETH swaps. Any ETH left on the router from a prior transaction is freely consumable by any subsequent caller who specifies `tokenIn = WETH`.

### Finding Description
When `pay(token=WETH, payer=attacker, recipient=pool, value=Z)` is called during a swap callback, the function checks `address(this).balance` and, if nonzero, wraps that ETH and transfers it to the pool on behalf of the attacker. The attacker's own balance is not touched. ETH accumulates on the router whenever a user calls any `payable` swap entry point with `msg.value > amountIn` and omits a `refundETH()` call.

### Impact Explanation
Direct loss of stranded ETH belonging to prior users. The attacker receives swap output without spending their own funds. The victim cannot recover the ETH once the attacker has consumed it (the attacker's swap settles it into the pool).

### Likelihood Explanation
Any user who calls `exactInputSingle` (or any payable swap function) directly with excess ETH — a common pattern when estimating gas or using ETH-to-WETH flows — and omits `refundETH` creates the precondition. Bots monitoring the mempool or router balance can exploit it in the same block.

### Recommendation
Replace `address(this).balance` in `pay()` with a tracked per-call ETH budget. The cleanest fix is to pass `msg.value` down from the entry point and use it as the cap in `pay()`, zeroing it after use. Alternatively, enforce that `refundETH` is always called at the end of any payable multicall by checking `address(this).balance == 0` after execution.

### Proof of Concept
```solidity
// 1. Victim sends excess ETH
router.exactInputSingle{value: 2 ether}(
    ExactInputSingleParams({tokenIn: WETH, amountIn: 1 ether, ...})
);
// Router now holds 1 ETH (stranded)

// 2. Attacker calls with amountIn = 1 ether, msg.value = 0
router.exactInputSingle{value: 0}(
    ExactInputSingleParams({tokenIn: WETH, amountIn: 1 ether, ...})
);
// pay() sees address(this).balance = 1 ether >= 1 ether
// Wraps router's ETH, sends WETH to pool
// Attacker receives swap output; own balance unchanged
// Victim's 1 ETH is gone
```
