Now I have the complete picture of the validation logic. Let me analyze the exact gap in step 4f.

### Title
Missing shard-index binding in `root_proofs` step 4f allows unprivileged peer to substitute cross-shard receipts during state sync, corrupting synced node state — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

In `ChainStateSyncAdapter::set_state_header`, step 4f verifies that `root_proofs[i][j].root` is *a* leaf in the block's outgoing-receipts Merkle tree, but never verifies it is the leaf at position `from_shard_index`. An unprivileged peer can supply shard Y's `prev_outgoing_receipts_root` (with a valid Merkle path) for a receipt proof whose `from_shard_id` is X. This breaks the pigeonhole completeness argument: shard X's receipts are silently dropped, shard Y's receipts are counted twice, the crafted header passes all validation and is persisted, and `set_state_finalize` applies the wrong receipt set, producing a state root that diverges from the canonical chain.

### Finding Description

The block's outgoing-receipts Merkle tree is built over `[chunk_0.prev_outgoing_receipts_root, chunk_1.prev_outgoing_receipts_root, ...]` ordered by shard index. [1](#0-0) 

Step 4f calls:

```rust
if !verify_path(
    *block_header.prev_chunk_outgoing_receipts_root(),
    block_proof,
    root,
) { ... }
``` [2](#0-1) 

`verify_path` only checks that `root` is *some* leaf reachable via `block_proof` from the tree root. It does not check that the leaf's position equals `from_shard_index`. The code comment explicitly relies on a pigeonhole argument to guarantee completeness: [3](#0-2) 

But that argument assumes each distinct `from_shard_id` maps to a unique chunk's outgoing receipts root — an invariant that is never enforced.

**Concrete attack** (block B, `chunks_included = 2`, shards X and Y both sent receipts R_X and R_Y to shard A):

| j | `from_shard_id` | receipts | `proof` | `root` | `block_proof` |
|---|---|---|---|---|---|
| 0 | X | R_Y | proof_Y (in Y's tree) | root_Y | block_proof_Y |
| 1 | Y | R_Y | proof_Y (in Y's tree) | root_Y | block_proof_Y |

- **4c**: `receipt_proofs.len() == 2 == chunks_included` ✓
- **4d**: X ≠ Y, uniqueness passes ✓
- **4e** (j=0,1): `verify_path(root_Y, proof_Y, hash(ReceiptList(shard_id, R_Y)))` ✓
- **4f** (j=0,1): `verify_path(block_root, block_proof_Y, root_Y)` ✓

All checks pass. R_X is never included; R_Y appears twice. [4](#0-3) 

### Impact Explanation

The crafted header is stored verbatim in `DBCol::StateHeaders`. [5](#0-4) 

`set_state_finalize` reads `incoming_receipts_proofs` directly from the stored header with no re-validation against `root_proofs`, collects the receipts, and passes them to `apply_chunk`: [6](#0-5) [7](#0-6) 

There is no post-application check of the resulting state root against the canonical chain's expected value inside `set_state_finalize`. The syncing node therefore stores a state that diverges from the canonical chain, causing it to produce invalid blocks that the network rejects.

### Likelihood Explanation

The attack requires only: (1) an unprivileged peer position — any node can serve state sync responses; (2) knowledge of public on-chain data (outgoing receipts and their Merkle proofs, all publicly available); (3) a target shard that receives receipts from at least two shards in the same block — a routine condition on mainnet. No validator or operator privilege is needed.

### Recommendation

After extracting `from_shard_id` from the `ShardProof`, resolve `from_shard_index` via `prev_shard_layout.get_shard_index(from_shard_id)` and assert:

```rust
let expected_root = block.chunks()[from_shard_index].prev_outgoing_receipts_root();
if root != expected_root { return Err(...); }
```

This binds each `RootProof` to the specific shard it claims to represent, restoring the pigeonhole invariant. The legitimate server-side construction already does this correctly: [8](#0-7) 

The fix must be applied on the *receiving* (validation) side in `set_state_header`.

### Proof of Concept

A cargo test in `chain/chain/src/state_sync/adapter.rs` would:

1. Build a two-shard test environment where both shards send receipts to shard A.
2. Obtain a legitimate `ShardStateSyncResponseHeader` for shard A via `get_state_header`.
3. Duplicate `incoming_receipts_proofs[0][0]` (shard Y's entry) into position `[0][1]`, setting `from_shard_id = X` on the duplicate, and set both `root_proofs[0][0]` and `root_proofs[0][1]` to `(root_Y, block_proof_Y)`.
4. Call `set_state_header` and assert it returns `Err(...)`.
5. Without the fix, the call returns `Ok(())`, demonstrating the bypass and confirming that the corrupted header is persisted.
