### Title
`deprecated_declared_classes` Omitted from `ThinStateDiff` in Batcher Path Produces Wrong `state_diff_commitment` and `state_diff_length` in Block Hash — (File: `crates/blockifier/src/state/cached_state.rs`)

---

### Summary

`From<CommitmentStateDiff> for ThinStateDiff` hard-codes `deprecated_declared_classes: Vec::new()`. The batcher feeds this `ThinStateDiff` directly into `calculate_block_commitments`, which uses it to compute both the `state_diff_commitment` (via `calculate_state_diff_hash`) and the `state_diff_length` packed into `concatenated_counts` (via `concat_counts`). Both values are chained into the final block hash. For every block that contains a deprecated (Cairo 0) class declaration, the sequencer stores and propagates a wrong block hash.

---

### Finding Description

**Root cause — the conversion:**

```rust
// crates/blockifier/src/state/cached_state.rs  lines 756-767
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` has no `deprecated_declared_classes` field because deprecated (Cairo 0) classes do not produce a compiled-class-hash entry in the Patricia trie. The conversion therefore silently drops them.

**Where the wrong `ThinStateDiff` is consumed:**

`BlockExecutionArtifacts::new` calls:

```rust
// crates/apollo_batcher/src/block_builder.rs  lines 170-176
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes == []
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
```

Inside `calculate_block_commitments` (`crates/starknet_api/src/block_hash/block_hash_calculator.rs` lines 318–356):

```rust
let concatenated_counts = concat_counts(
    transactions_data.len(),
    event_leaf_elements.len(),
    state_diff.len(),          // ← ThinStateDiff::len() counts deprecated_declared_classes
    l1_da_mode,
);
// ...
let state_diff_task = spawn_measured_task(move || calculate_state_diff_hash(&state_diff));
```

`ThinStateDiff::len()` (`crates/starknet_api/src/state.rs` lines 111–122) adds `self.deprecated_declared_classes.len()` to the total. Because that slice is always empty, the packed `state_diff_length` is under-counted.

`calculate_state_diff_hash` (`crates/starknet_api/src/block_hash/state_diff_hash.rs` lines 30–42) chains `deprecated_declared_classes` into the Poseidon hash. With an empty slice it chains a zero-length list, producing a different hash than the correct one.

Both the wrong `state_diff_commitment` and the wrong `concatenated_counts` are stored in `BlockHeaderCommitments`, which is then committed into the block hash via `calculate_block_hash` (`crates/starknet_api/src/block_hash/block_hash_calculator.rs` lines 245–282):

```rust
.chain(&block_commitments.concatenated_counts)
.chain(&block_commitments.state_diff_commitment.0.0)
```

**Why existing checks do not catch this:**

The committer's optional `verify_state_diff_hash` check (`crates/apollo_committer/src/committer.rs` lines 267–276) recomputes the hash from the same `ThinStateDiff` that was passed in — which also has empty `deprecated_declared_classes` — so the comparison always passes.

---

### Impact Explanation

Every block that includes a `DECLARE` v0 or v1 transaction (deprecated Cairo 0 class) receives a wrong `state_diff_commitment` and a wrong `state_diff_length` in its `concatenated_counts`. Both fields are chained into the block hash. Consequently:

1. The block hash stored in the sequencer's MDBX storage is wrong.
2. All subsequent block hashes are wrong (each chains the wrong parent hash).
3. Proof facts embedded in SNIP-36 invoke transactions reference the wrong block hash; the gateway accepts them because the stored hash matches the wrong value.
4. The SNOS `get_block_hashes` hint (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/block_hash.cairo` lines 55–82) receives the wrong `header_commitments` and produces a wrong block hash that the prover commits to.
5. An external verifier or L1 contract that independently recomputes the block hash from the actual state diff will disagree, breaking proof soundness.

This matches **"Wrong state … or revert result from blockifier/syscall/execution logic for accepted input"** (Critical) and **"RPC … returns an authoritative-looking wrong value"** (High).

---

### Likelihood Explanation

`DECLARE` v0 and v1 transactions are valid Starknet transaction types. The gateway does not reject them. Any user can submit one. A single such transaction in any block is sufficient to corrupt that block's hash and every subsequent block hash in the chain.

---

### Recommendation

The batcher must supply the actual `deprecated_declared_classes` when constructing the `ThinStateDiff` for commitment calculations. Two complementary fixes:

1. **Track deprecated declarations in `CommitmentStateDiff`**: add a `deprecated_declared_classes: Vec<ClassHash>` field and populate it from the blockifier's `StateMaps::declared_contracts` (filtering entries whose `CompiledClassHash` is absent).

2. **Alternatively, pass them separately into `BlockExecutionArtifacts::new`**: collect deprecated class hashes from the block's execution infos and inject them into the `ThinStateDiff` before calling `calculate_block_commitments`, mirroring the pattern already used in `induced_state_diff` (`crates/apollo_rpc_execution/src/execution_utils.rs` lines 130–145).

The `From<CommitmentStateDiff> for ThinStateDiff` TODO comment (`// TODO(AlonH): Remove this when the structure of storage diffs changes.`) should be resolved as part of this fix.

---

### Proof of Concept

1. Submit a `DECLARE` v1 transaction declaring any Cairo 0 contract class to the sequencer.
2. The sequencer includes it in a block. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff)`, producing a `ThinStateDiff` with `deprecated_declared_classes = []`.
3. `calculate_block_commitments` computes:
   - `state_diff_length = N` (missing the 1 deprecated class entry → should be `N+1`).
   - `state_diff_commitment = H(…, chain_deprecated_declared_classes([]), …)` (wrong hash).
4. `calculate_block_hash` chains both wrong values, producing `block_hash_wrong`.
5. The sequencer stores `block_hash_wrong` and uses it as the parent hash for block `n+1`.
6. An independent verifier recomputes the block hash from the full `ThinStateDiff` (including the deprecated class) and obtains `block_hash_correct ≠ block_hash_wrong`.
7. All proofs referencing `block_hash_wrong` are unsound with respect to the canonical chain.
