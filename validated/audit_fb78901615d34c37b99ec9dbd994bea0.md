### Title
Unauthenticated Requester in Spice Witness/Receipt Distribution Allows Any Peer to Obtain Restricted Consensus Data - (File: chain/client/src/spice/data_distributor_actor.rs)

### Summary

`SpiceDataDistributorActor::handle_partial_data_request` serves state witnesses and receipt proofs to any `AccountId` supplied in the `requester` field of a `SpicePartialDataRequest` network message, without verifying that the requester is a member of the authorized recipient set (chunk validators for witnesses, destination-shard chunk producers for receipt proofs). Any unprivileged peer can obtain restricted consensus data by sending a request with their own `AccountId`.

### Finding Description

`SpicePartialDataRequest` carries two fields: a `data_id` identifying the witness or receipt proof, and a plain `requester: AccountId` that is entirely self-reported by the sending peer. [1](#0-0) 

When a producer node receives this message, `handle_partial_data_request` checks only that **the local node itself** is a producer of the requested data. It never checks whether `requester` is in the authorized recipient set: [2](#0-1) 

The code contains an explicit TODO acknowledging the missing guard:

```
// TODO(spice): Check that requester is one of the recipients and implement a
// lower-priority way for other nodes that aren't validators (e.g. rpc nodes) to get
// data they require.
```

After this unguarded path, the full encoded data (witness or receipt proof) is sent directly to whatever `AccountId` was supplied: [3](#0-2) 

The authorized recipient sets are computed by `recipients_and_producers`: for witnesses, recipients are the ordered chunk validators; for receipt proofs, recipients are the destination-shard chunk producers. Neither set is consulted before serving the data. [4](#0-3) 

A test in the repository explicitly demonstrates and accepts this behavior — a `"not-validator"` account successfully receives a receipt proof: [5](#0-4) 

### Impact Explanation

Any peer that can reach a Spice chunk producer over the network can:

1. Send a `SpicePartialDataRequest` with `requester = <their own AccountId>` for any `(block_hash, shard_id)` or `(block_hash, from_shard_id, to_shard_id)`.
2. Receive the full encoded state witness (containing the complete state transition for that shard) or the full receipt proof, bypassing the intended access control that restricts these to chunk validators and chunk producers respectively.
3. Repeat for every block and shard, exfiltrating all in-flight consensus data.

Additionally, because `requester` is not signature-verified, an attacker can forge any `AccountId` in the field. If they supply a real validator's `AccountId`, the producer routes the response to that validator's network address — enabling misdirection of data delivery and potential amplification of network traffic toward targeted validators.

### Likelihood Explanation

The attack requires only a TCP connection to a Spice chunk producer node, which is a standard peer-to-peer connection. No stake, no validator key, and no privileged role is needed. The `SpicePartialDataRequest` message is a plain borsh-serialized struct with no signature field. The missing check is acknowledged in a TODO comment, confirming the code path is reachable in production.

### Recommendation

In `handle_partial_data_request`, after confirming the local node is a producer, verify that `requester` is a member of the computed `recipients` set before serving data:

```rust
let (recipients, producers) = self.recipients_and_producers(&data_id, &block)?;
if !producers.contains(signer.validator_id()) {
    return Err(Error::Other("we do not produce requested data"));
}
// ADD: enforce recipient allowlist
if !recipients.contains(&requester) {
    tracing::warn!(target: "spice_data_distribution", ?requester, "data request from non-recipient");
    return Ok(());
}
```

Additionally, add a signature field to `SpicePartialDataRequest` (analogous to `SpiceContractCodeRequest`, which already carries a signature verified against the epoch validator set) so the `requester` identity cannot be forged. [6](#0-5) 

### Proof of Concept

The existing test `test_requesting_receipts_when_not_validator` in `chain/client/src/spice/tests/data_distributor_actor.rs` (lines 2106–2156) already demonstrates the issue: a node with `AccountId = "not-validator"` sends a `SpicePartialDataRequest` to a legitimate producer and successfully receives the full encoded receipt proof. No modification to the test is needed — it passes as written, confirming the unauthorized access path is live in production code. [7](#0-6)

    fn recipients_and_producers(
        &self,
        data_id: &SpiceDataIdentifier,
        block: &Block,
    ) -> Result<(HashSet<AccountId>, Vec<AccountId>), Error> {
        let (recipients, producers) = match data_id {
            SpiceDataIdentifier::ReceiptProof { from_shard_id, to_shard_id, block_hash } => {
                debug_assert_eq!(block.hash(), block_hash);
                let epoch_id = block.header().epoch_id();
                let next_block_epoch_id =
                    self.epoch_manager.get_epoch_id_from_prev_block(block_hash)?;
                // TODO(spice-resharding): validate whether from_shard_id and to_shard_id would be
                // correct when resharding.
                let producers = self
                    .epoch_manager
                    .get_epoch_chunk_producers_for_shard(&epoch_id, *from_shard_id)?;
                let recipients = self
                    .epoch_manager
                    .get_epoch_chunk_producers_for_shard(&next_block_epoch_id, *to_shard_id)?;
                (recipients, producers)
            }
            SpiceDataIdentifier::Witness { block_hash, shard_id } => {
                debug_assert_eq!(block.hash(), block_hash);
                let epoch_id = block.header().epoch_id();
                let producers =
                    self.epoch_manager.get_epoch_chunk_producers_for_shard(epoch_id, *shard_id)?;
                let validator_assignments = self.epoch_manager.get_chunk_validator_assignments(
                    epoch_id,
                    *shard_id,
                    block.header().height(),
                )?;
                let recipients = validator_assignments.ordered_chunk_validators();
                (recipients, producers)
            }
        };
        // Since producers would produce the data anyway they shouldn't be in the recipients set.
        let mut recipients_set: HashSet<_> = HashSet::from_iter(recipients.into_iter());
        for account in &producers {
            recipients_set.remove(account);
        }
        Ok((recipients_set, producers))
    }

    fn handle_partial_data_request(
        &mut self,
        SpicePartialDataRequest { data_id, requester }: SpicePartialDataRequest,
    ) -> Result<(), Error> {
        let Some(signer) = self.validator_signer.get() else {
            return Err(Error::Other(
                "without validator signer we cannot handle partial data requests",
            ));
        };

        let block = self.chain_store.get_block(data_id.block_hash())?;
        let (_recipients, producers) = self.recipients_and_producers(&data_id, &block)?;
        if !producers.contains(signer.validator_id()) {
            return Err(Error::Other("we do not produce requested data"));
        }

        let Some(data) = self.get_distribution_data(&data_id, producers.len()) else {
            // TODO(spice): Make sure we send requests for data only after we know it may be
            // available and make this into error.
            tracing::debug!(target:"spice_data_distribution", ?data_id, ?requester, "received request for unknown data");
            return Ok(());
        };

        // lower-priority way for other nodes that aren't validators (e.g. rpc nodes) to get
        // data they require.

#[test]
#[cfg_attr(not(feature = "protocol_feature_spice"), ignore)]
fn test_requesting_receipts_when_not_validator() {
    let (genesis, chain) = setup(2, 1);
    let requester_chain = new_chain(&chain, &genesis);

    let block = latest_block(&chain);
    let receipt_proof = new_test_receipt_proof(&block);
    let mut store_update = chain.chain_store.store().store_update();
    save_receipt_proof(&mut store_update, block.hash(), &receipt_proof);
    store_update.commit();

    let producer = producers_of_receipt_proof(&chain, &block, &receipt_proof).swap_remove(0);
    let data_id = SpiceDataIdentifier::ReceiptProof {
        block_hash: *block.hash(),
        from_shard_id: receipt_proof.1.from_shard_id,
        to_shard_id: receipt_proof.1.to_shard_id,
    };
    let to_shard_id = receipt_proof.1.to_shard_id;

    let (outgoing_sc, mut outgoing_rc) = unbounded_channel();
    let mut actor = new_actor_for_account(outgoing_sc, &chain, &producer);

    let requester = AccountId::from_str("not-validator").unwrap();
    actor.handle(SpicePartialDataRequest { data_id, requester: requester.clone() });
    let (partial_data, recipients) = drain_outgoing_partial_data(&mut outgoing_rc).swap_remove(0);
    assert_eq!(recipients, HashSet::from([requester.clone()]));

    let to_shard_uid = {
        let shard_layout = chain.epoch_manager.get_shard_layout(block.header().epoch_id()).unwrap();
        ShardUId::from_shard_id_and_layout(to_shard_id, &shard_layout)
    };

    let (outgoing_sc, mut outgoing_rc) = unbounded_channel();
    let mut actor = ActorBuilder::new(Some(requester))
        .tracked_shards_config(TrackedShardsConfig::Shards(vec![to_shard_uid]))
        .build(outgoing_sc, &requester_chain);
    actor.handle(SpiceIncomingPartialData { data: partial_data });

    let message = outgoing_rc.try_recv().unwrap();
    assert_matches!(outgoing_rc.try_recv(), Err(TryRecvError::Empty));
    let OutgoingMessage::ExecutorIncomingUnverifiedReceipts(ExecutorIncomingUnverifiedReceipts {
        block_hash: reconstructed_block_hash,
        receipt_proof: reconstructed_receipt_proof,
    }) = message
    else {
        panic!();
    };
    assert_eq!(block.hash(), &reconstructed_block_hash);
    assert_eq!(receipt_proof, reconstructed_receipt_proof);
}
