### Title
Missing Chain-ID Domain Binding in `deploy_token` Signature Hash Enables Cross-Chain Replay of Token Deployment — (Files: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deploy_token` / `deployToken` message hash on every destination chain (EVM, Starknet, Solana) omits the destination chain ID from the signed payload. Because the same NEAR-derived MPC address is the signer on all chains, a valid deployment signature observed on one chain is byte-for-byte valid on every other chain. An unprivileged attacker can replay any legitimate `deployToken` signature to deploy the same NEAR token on an unintended chain, corrupting the bridge's token registry and permanently blocking the legitimate deployment on that chain.

---

### Finding Description

The `fin_transfer` path correctly binds the signed message to a specific destination chain by embedding `omniBridgeChainId` twice in the borsh payload (once for the token address chain prefix, once for the recipient chain prefix). The `deploy_token` path does not.

**EVM** — `OmniBridge.sol` `deployToken()`:
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)          // ← no omniBridgeChainId anywhere
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) revert InvalidSignature();
```

**Starknet** — `bridge_types.cairo` `MetadataPayloadTrait::to_borsh()`:
```cairo
borsh_bytes.append_byte(PayloadType::Metadata.into());
borsh_bytes.append(@borsh::encode_byte_array(self.token));
borsh_bytes.append(@borsh::encode_byte_array(self.name));
borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
borsh_bytes.append_byte(*self.decimals);   // ← no chain_id
```

**Solana** — `deploy_token.rs` `DeployTokenPayload::serialize_for_near()`:
```rust
IncomingMessageType::Metadata.serialize(&mut writer)?;
self.serialize(&mut writer)?;   // token, name, symbol, decimals — no SOLANA_OMNI_BRIDGE_CHAIN_ID
```

Contrast with `finalize_transfer.rs` which writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` at positions 3 and 5, and `OmniBridge.sol` `finTransfer()` which writes `bytes1(omniBridgeChainId)` at lines 294 and 297. The asymmetry is the root cause.

Because all chains verify against the same NEAR MPC-derived Ethereum address (`nearBridgeDerivedAddress` / `omni_bridge_derived_address`), the identical hash produced on chain A is accepted without modification on chain B.

---

### Impact Explanation

**Impact: High** — Proof/signature verification bypass enabling unauthorized token deployment.

1. **Unauthorized token deployment on any chain**: An attacker who observes a legitimate `deployToken` transaction on chain A (e.g., Ethereum EVM) extracts the `(signatureData, MetadataPayload)` tuple and submits it verbatim to chain B (e.g., Starknet or Solana). Chain B's bridge accepts the signature and registers the token as a legitimate bridge token (`isBridgeToken[proxy] = true` / `near_to_starknet_token` mapping written).

2. **Permanent DoS on legitimate deployment**: Once the token is registered on chain B via replay, the guard `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` / `assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED')` permanently blocks the protocol from ever legitimately deploying that token on chain B. The bridge's token registry for that chain is irrecoverably corrupted without an admin intervention.

3. **Potential for subsequent unauthorized minting**: If NEAR's bridge relayer monitors `DeployToken` events from all chains to update its internal state, it may treat the replayed deployment as legitimate and begin signing `fin_transfer` messages for chain B. This would allow minting of unbacked bridge tokens on chain B.

---

### Likelihood Explanation

**Likelihood: High**

- No privileged access is required. The attacker only needs to observe a public on-chain `deployToken` transaction on any one chain.
- The signature and payload are fully visible in calldata.
- The replay requires only a standard contract call on the target chain — no special tooling.
- The protocol is multi-chain by design, so every new token deployment on one chain immediately creates a replayable signature for all other chains.
- The attack is permanent: once replayed, the damage (registry corruption + deployment DoS) cannot be undone without admin action.

---

### Recommendation

Embed the destination chain ID into the `deploy_token` borsh payload in the same way `fin_transfer` already does. Specifically:

- **EVM**: Append `bytes1(omniBridgeChainId)` to `borshEncoded` in `deployToken()` before hashing.
- **Starknet**: Add `borsh_bytes.append_byte(chain_id)` inside `MetadataPayloadTrait::to_borsh()`, passing `omni_bridge_chain_id` as a parameter (mirroring `TransferMessagePayloadTrait::to_borsh(chain_id)`).
- **Solana**: Write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into `DeployTokenPayload::serialize_for_near()` before or after the token fields.

The NEAR side that produces the signature must include the same chain ID field so that signatures are chain-specific.

---

### Proof of Concept

**Setup**: Two EVM-compatible chains are both running `OmniBridge` with the same `nearBridgeDerivedAddress` (e.g., Ethereum mainnet chain ID `omniBridgeChainId = 1` and BNB chain `omniBridgeChainId = 2`).

**Step 1**: NEAR bridge signs and broadcasts a legitimate `deployToken` for `usdc.near` on Ethereum. The transaction is mined; calldata is public.

**Step 2**: Attacker extracts `(signatureData, MetadataPayload{token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` from the Ethereum transaction.

**Step 3**: Attacker calls `deployToken(signatureData, payload)` on the BNB chain `OmniBridge`.

**Step 4**: BNB chain computes:
```
keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)
```
This is byte-for-byte identical to the Ethereum hash. `ECDSA.recover` returns `nearBridgeDerivedAddress`. Signature check passes.

**Step 5**: A new `BridgeToken` proxy is deployed on BNB. `isBridgeToken[proxy] = true`, `nearToEthToken["usdc.near"] = proxy`. The BNB bridge now has an unauthorized, attacker-triggered USDC bridge token registered.

**Step 6**: Any future legitimate attempt by NEAR to deploy `usdc.near` on BNB hits `require(!isBridgeToken[nearToEthToken["usdc.near"]], "ERR_TOKEN_EXIST")` and reverts permanently.

The same replay works from any source chain to any destination chain (EVM → Starknet, EVM → Solana, Starknet → EVM, etc.) because the hash is chain-agnostic.
