### Title
Missing Blocked-Sender Check in Gateway Stateless Validator Allows Any Address to Bypass Admission Control — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate()` function in the Apollo gateway contains an explicit, unimplemented TODO acknowledging that no mechanism exists to reject transactions from blocked/sanctioned sender addresses. Any address the operator intends to block can submit `Invoke` and `DeployAccount` transactions that pass all stateless checks and are admitted to the mempool for sequencing.

### Finding Description

`StatelessTransactionValidator::validate()` is the first admission gate for every incoming `RpcTransaction`. It checks contract address validity, resource bounds, calldata/signature size, DA modes, and proof-facts consistency. However, line 34 contains:

```rust
// TODO(Arni, 1/5/2024): Add a mechanism that validate the sender address is not blocked.
``` [1](#0-0) 

No corresponding field exists in `StatelessTransactionValidatorConfig` and none is serialized in `SerializeConfig`: [2](#0-1) 

The only sender-level admission restriction in the gateway is `check_declare_permissions`, which enforces an *allowlist* of authorized declarers for `Declare` transactions only: [3](#0-2) 

No equivalent *blocklist* check exists for `Invoke` or `DeployAccount` transactions anywhere in the stateless or stateful validator paths: [4](#0-3) 

The `BlockingCheckClient` in `starknet_transaction_prover` is the only other blocking mechanism, but it is scoped exclusively to the client-side proving path and is not invoked during normal gateway admission: [5](#0-4) 

The `echonet` regression tool has a `filter_blocked` helper and a `blocked_senders` config key, but these are non-production replay utilities and have no effect on the live gateway: [6](#0-5) 

### Impact Explanation

A sender address that the operator intends to block can submit `Invoke` and `DeployAccount` transactions through the HTTP gateway. These transactions pass `StatelessTransactionValidator::validate()` without any sender-identity check, proceed through `StatefulTransactionValidator` (which only checks nonce, resource bounds, and runs `__validate__`), and are admitted to the mempool for sequencing. The result is that the gateway's admission control is bypassed for all non-`Declare` transaction types, matching the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

### Likelihood Explanation

The trigger is entirely unprivileged: any user who knows their address is on an operator blocklist simply submits an `Invoke` or `DeployAccount` transaction through the standard JSON-RPC endpoint. No special access, peer position, or privileged role is required. The TODO comment confirms the gap has been known since at least 2024-01-05 and remains unaddressed.

### Recommendation

Add a `blocked_senders: HashSet<ContractAddress>` field to `StatelessTransactionValidatorConfig`. In `StatelessTransactionValidator::validate()`, after `validate_contract_address`, extract the sender address and return a `StatelessTransactionValidatorError::BlockedSender` if it appears in the set. This mirrors the pattern already used for `check_declare_permissions` and closes the gap acknowledged by the TODO.

### Proof of Concept

1. Operator configures a blocklist containing address `0xABCD`.
2. User at `0xABCD` submits `starknet_addInvokeTransaction` with a valid V3 invoke.
3. `StatelessTransactionValidator::validate()` runs: `validate_contract_address` passes (address is in valid range), all size/bounds checks pass, no blocked-sender check is executed.
4. `StatefulTransactionValidator::extract_state_nonce_and_run_validations` runs: nonce is fetched, `run_pre_validation_checks` passes, `run_validate_entry_point` runs `__validate__` on the account contract and passes.
5. Transaction is pushed to the mempool and sequenced into a block — the operator's intended block is silently bypassed. [7](#0-6)
