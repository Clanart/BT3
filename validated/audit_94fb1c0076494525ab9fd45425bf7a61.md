### Title
Missing Recipient Address Validation Before Irreversible Token Burn/Lock in `initTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`, `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

On EVM, StarkNet, and Solana, `initTransfer` irreversibly burns or locks user tokens before performing any validation of the `recipient` address string. If a user supplies an empty, malformed, or otherwise unparseable recipient string, the source-chain tokens are permanently destroyed while the NEAR-side finalization will always fail, leaving the funds irrecoverable.

---

### Finding Description

**EVM (`evm/src/omni-bridge/contracts/OmniBridge.sol`, lines 373–437)**

`initTransfer` accepts `string calldata recipient` with zero validation. The function immediately burns bridge tokens (`BridgeToken(tokenAddress).burn(msg.sender, amount)`) or locks native ERC-20 tokens (`safeTransferFrom` into the contract) before the recipient string is ever inspected. The recipient is only emitted in an event; it is never parsed or checked on-chain.

**StarkNet (`starknet/src/omni_bridge.cairo`, lines 281–331)**

`init_transfer` accepts `recipient: ByteArray` and only asserts `amount > 0` and `fee < amount`. Tokens are burned or transferred to the contract before any recipient validation occurs.

**Solana (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`, lines 72–130)**

`process` validates only `amount > fee`. The `recipient: String` field in `InitTransferPayload` is serialized into the Wormhole message without any format check.

**NEAR-side failure path**

When a relayer submits the proof to NEAR's `fin_transfer_callback`, the prover must parse the recipient string into an `OmniAddress`. `OmniAddress::from_str` (in `near/omni-types/src/lib.rs`, lines 392–411) returns an error for any empty or malformed string (e.g., `""` → treated as `"eth:"` → `H160` parse fails). The callback then panics with `InvalidProofMessage`. Because the source-chain burn/lock is already finalized and irreversible, and because no refund or recovery mechanism exists for this failure mode, the funds are permanently lost.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who provides an empty string, a typo, or a chain-prefix-only string (e.g., `"eth:"`, `""`, `"near:"`) as the recipient will have their tokens burned or locked on the source chain. The NEAR bridge will never be able to finalize the transfer because `OmniAddress::from_str` will always reject the malformed address. There is no on-chain refund path: the EVM/StarkNet/Solana contracts do not store the transfer for later cancellation, and the NEAR contract never records a pending transfer for an unparseable proof. The funds are gone.

---

### Likelihood Explanation

**Medium-High.** Any unprivileged bridge user calling `initTransfer` directly (e.g., via a script, a dApp with a UI bug, or a copy-paste error in the recipient field) can trigger this. The `recipient` parameter is a free-form string on all three source chains. There is no client-side or contract-side guard. A single mistyped or empty recipient field is sufficient.

---

### Recommendation

**Short term:** Add an explicit non-empty and format check for the `recipient` string in `initTransfer` on EVM, StarkNet, and Solana before any token movement occurs. At minimum, reject empty strings and strings that do not contain a `:` separator with a recognized chain prefix.

**Long term:** Implement a two-step commit-reveal or a cancellable pending-transfer pattern so that if NEAR-side finalization fails for any reason (including an unparseable recipient), the source-chain tokens can be reclaimed by the original sender within a timeout window.

---

### Proof of Concept

1. Alice calls `initTransfer` on the EVM bridge with `recipient = ""` and `amount = 1000 USDC`.
2. The EVM contract passes the `fee >= amount` check, then calls `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` — Alice's 1000 USDC are now locked in the bridge contract.
3. The `InitTransfer` event is emitted with `recipient = ""`.
4. A relayer picks up the event and submits the proof to NEAR's `fin_transfer`.
5. NEAR's `fin_transfer_callback` calls `Self::decode_prover_result(0)`, which internally calls `OmniAddress::from_str("")`. Since `split_once(':')` returns `None`, the parser defaults to `chain = "eth"` and `recipient = ""`. Parsing `""` as an `H160` fails.
6. The prover returns an error; `fin_transfer_callback` panics with `BridgeError::InvalidProofMessage`.
7. Alice's 1000 USDC remain locked in the EVM bridge contract forever with no recovery path.
