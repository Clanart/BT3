### Title
`MerkleBlob::batch_insert()` Skips Duplicate-Key Check in Bulk Path, Silently Corrupting Tree Root and Invalidating Proofs - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary

`MerkleBlob::batch_insert()` has a split execution path: the first two items (when the tree has ≤1 leaf) are inserted via the safe `insert()` method that enforces `KeyAlreadyPresent` / `HashAlreadyPresent` guards, but all remaining items in the batch are written directly to the blob via `insert_entry_to_blob()` with **no duplicate-key check**. When a duplicate key reaches the bulk path, `BlockStatusCache::add_leaf()` silently overwrites the cache entry via `HashMap::insert()`, while the old leaf node remains in the blob. The result is a blob/cache split: two leaf nodes share the same `KeyId`, the cache tracks only the last one, and the computed Merkle root is over a structurally invalid tree. Proofs of inclusion derived from this root are forged relative to any honest root, and `check_integrity()` will detect the inconsistency — but only when explicitly called.

### Finding Description

`MerkleBlob::batch_insert()` processes its input vector in two distinct phases:

**Phase 1 — safe path (lines 578–585):** Only entered when `leaf_count <= 1`. Pops the last two items from the vector and delegates to `self.insert()`, which explicitly guards against duplicates:

```rust
// insert() — lines 369–374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

**Phase 2 — bulk path (lines 587–603):** Iterates over all remaining items (or all items when `leaf_count >= 2`) and calls `insert_entry_to_blob()` directly — **no duplicate check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // no guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob()` (lines 1013–1030) writes the block to the blob and then calls `block_status_cache.add_leaf()`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
```

`HashMap::insert` returns the displaced value but the caller discards it. The old leaf node at the displaced index is now orphaned in the blob — it is still reachable by the tree traversal iterators but is no longer tracked by the cache. The tree now contains two leaf nodes with the same `KeyId`.

Two concrete trigger scenarios exist:

1. **Intra-batch duplicate:** The caller passes a batch where the same `KeyId` appears more than once. Both leaves are written into the blob; the cache tracks only the last one.
2. **Cross-call duplicate (tree already has ≥2 leaves):** The caller passes a key that already exists in the tree. Because `leaf_count >= 2`, Phase 1 is skipped entirely, and the duplicate is written unconditionally.

### Impact Explanation

The corrupted tree produces a root hash that does not correspond to any valid DataLayer key-value mapping. Specifically:

- `get_proof_of_inclusion(key)` for the duplicated key returns a proof anchored to the cache-tracked leaf. That proof is structurally valid against the corrupted root, but the root itself is invalid — any peer that independently computes the root from the canonical key-value set will disagree.
- `check_integrity()` will eventually detect the inconsistency via `IntegrityKeyToIndexCacheLength` (leaf count in blob ≠ cache length), but this check is not called automatically after `batch_insert`.
- A DataLayer store whose root is derived from a corrupted `MerkleBlob` will present a root hash that cannot be reproduced by honest nodes, causing inclusion/exclusion proof verification to fail across the network and corrupting committed DataLayer state.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is a public API exposed directly as a Python binding (`py_batch_insert`, line 1503) and is the primary bulk-load path used by the DataLayer Python node. Any caller — including DataLayer store operators processing externally-sourced key-value deltas — can trigger the bug by supplying a batch containing a repeated key or a key already present in the tree. No privilege is required beyond the ability to call `batch_insert`. The Python binding performs only a length-equality check between the keys and hashes vectors; it adds no duplicate-key validation.

### Recommendation

Add duplicate-key and duplicate-hash guards at the start of the bulk loop in `batch_insert`, mirroring the checks already present in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, pre-validate the entire input batch for uniqueness before any mutation, so the tree is never left in a partially-corrupted state on error.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};

fn main() {
    // Tree already has 2 leaves → Phase 1 is skipped entirely
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &sha256_num(i), InsertLocation::Auto {}).unwrap();
    }

    let dup_key   = KeyId(99);
    let dup_hash  = Hash::from([0xABu8; 32]);

    // batch_insert with the same key twice — no error returned
    blob.batch_insert(vec![
        ((dup_key, ValueId(1)), dup_hash),
        ((dup_key, ValueId(2)), Hash::from([0xCDu8; 32])),
    ]).unwrap();  // succeeds silently

    // Cache tracks only the second leaf; blob contains both.
    // check_integrity() now returns IntegrityKeyToIndexCacheLength error,
    // confirming blob/cache divergence and a corrupted root hash.
    blob.check_integrity().unwrap_err();
}
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
