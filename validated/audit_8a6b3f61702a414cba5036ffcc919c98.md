### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key Validation, Corrupting Tree Root and Proof-of-Inclusion State — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key guard that `insert` enforces, allowing an attacker-controlled input to write two leaf nodes with the same `KeyId` into the blob. The in-memory `BlockStatusCache` silently overwrites the first occurrence's index with the second, leaving an orphaned leaf in the tree structure. The resulting root hash is computed over a structurally invalid tree, and `get_proof_of_inclusion` produces a proof anchored to that corrupted root — a direct analog to the Redis state-tracking failure in the external report.

---

**Concrete state after `batch_insert([(K, V1, H1), (K2, V2, H2), (K, V3, H3)])` on a tree with ≥ 2 existing leaves:**

| Step | Blob | Cache (`key_to_index`) |
|---|---|---|
| Write leaf (K, V1, H1) at index A | A present | K → A |
| Write leaf (K2, V2, H2) at index B | B present | K2 → B |
| Write leaf (K, V3, H3) at index C | C present | K → C (A silently lost) |
| Build internal nodes over {A, B, C} | tree has two K-leaves | cache knows only C |

The blob now contains two structurally connected leaves with the same `KeyId`. The root hash is computed over this invalid tree.

### Impact Explanation

The corrupted `MerkleBlob` produces a root hash that encodes a tree with duplicate keys. Any DataLayer store that commits this root on-chain presents a root that:

1. **Cannot be reproduced** by an honest peer reconstructing the tree from the same key-value set (they would reject the duplicate and produce a different root).
2. **Yields structurally valid but semantically forged proofs**: `get_proof_of_inclusion` returns a `ProofOfInclusion` whose `valid()` check passes, yet the proof is anchored to a corrupted root — satisfying the "corrupts tree roots / lets untrusted input prove invalid state" criterion.
3. **Hides data**: the orphaned leaf at index A is reachable via tree traversal but invisible to all cache-based lookups (`get_keys_values`, `get_key_index`, `get_proof_of_inclusion`), creating a persistent cache/blob split analogous to the Redis tracking failure in the external report.

This matches: **High — DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

- `batch_insert` is the primary bulk-load path used by the DataLayer sync layer (called from Python via `py_batch_insert`).
- Any Python caller — including a DataLayer peer sending a crafted delta — that supplies a list containing a repeated `KeyId` triggers the bug silently (no error is returned).
- The condition `leaf_count > 1` is the normal operating state of any non-trivial DataLayer store, so the unguarded loop is the common code path.
- No special privilege is required; the Python binding accepts the input directly.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    ...
}
```

Alternatively, pre-validate the entire input vector for uniqueness before any blob writes, so the operation is fully atomic (all-or-nothing).

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    // Pre-populate with 2 leaves so leaf_count > 1 (bypasses the guarded path)
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i),
            &Hash(Bytes32::new([i as u8; 32])), InsertLocation::Auto {}).unwrap();
    }

    // batch_insert with duplicate KeyId(99) — no error returned
    let dup_key = KeyId(99);
    blob.batch_insert(vec![
        ((dup_key, ValueId(1)), Hash(Bytes32::new([10u8; 32]))),
        ((KeyId(100), ValueId(2)), Hash(Bytes32::new([11u8; 32]))),
        ((dup_key, ValueId(3)), Hash(Bytes32::new([12u8; 32]))),  // duplicate
    ]).unwrap();  // succeeds silently

    blob.calculate_lazy_hashes().unwrap();

    // Cache reports only one entry for dup_key (the last one written)
    let proof = blob.get_proof_of_inclusion(dup_key).unwrap();
    assert!(proof.valid());  // passes — but root is corrupted

    // check_integrity would detect the inconsistency if called:
    // blob.check_integrity().unwrap();  // would fail: two leaves with same key in blob
}
```

The `check_integrity` call (which rebuilds the cache from the blob via `BlockStatusCache::new`) would return `Error::KeyAlreadyPresent` — confirming the blob contains two leaves with the same key while the live cache silently masked the first. [7](#0-6)
