### Title
Detached `swap_migrated_token` Promise Has No Error Handling — Burn Succeeds While Mint Fails, Permanently Destroying User Tokens - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`swap_migrated_token` is invoked via `ft_on_transfer` with a detached promise. The bridge immediately returns `U128(0)` (keeping all transferred tokens), then fires a parallel `burn.and(mint)` with no callback and no error handling. If the `mint` leg panics — which happens whenever the recipient has no storage registered on the new token — the old tokens are burned from the bridge's balance but the user receives nothing. There is no recovery path.

### Finding Description

When a user sends old tokens to the bridge with the `SwapMigratedToken` message, `ft_on_transfer` dispatches `swap_migrated_token` and immediately returns `U128(0)`:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
```

Returning `U128(0)` from `ft_on_transfer` tells the NEP-141 token contract to keep all transferred tokens in the bridge's balance — no refund is possible after this point.

`swap_migrated_token` then builds a combined promise:

```rust
fn swap_migrated_token(&mut self, sender_id: AccountId, old_token: AccountId, amount: U128) -> Promise {
    let new_token = self.migrated_tokens.get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);
    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
    burn.and(mint)
}
```

`burn.and(mint)` schedules two independent receipts. In NEAR, `.and()` is **not atomic**: if one receipt panics, the other is not rolled back. The combined promise is `.detach()`ed, so there is no callback to detect or handle failure.

The `mint` call on `OmniToken` calls `internal_deposit(&account_id, ...)`, which panics if `account_id` has no storage registered on the new token contract. The new token is freshly deployed; users who hold old tokens have had no reason to register storage on it. When `mint` panics:

- Its state changes are rolled back (no tokens minted).
- The `burn` receipt has already executed (or executes independently) and its state changes are **not** rolled back — the bridge's balance of old tokens is reduced.
- The user's old tokens are gone; no new tokens are received.

### Impact Explanation

Any user who sends old tokens to the bridge for migration without first registering storage on the new token contract will have their tokens permanently destroyed. The bridge holds the old tokens (returned `U128(0)`), burns them, and fails to mint replacements. There is no recovery function, no refund path, and no admin escape hatch. This is an irrecoverable loss of user funds in the bridge token flow.

**Impact: Critical — Permanent, irrecoverable destruction of user bridged assets.**

### Likelihood Explanation

The new token is a freshly deployed contract. Users who accumulated old tokens before the migration have no reason to have registered storage on the new token. The `migrate_deployed_token` DAO function only registers storage for the bridge itself on the new token, not for any user. Any user who calls `ft_transfer_call` with `SwapMigratedToken` without first calling `storage_deposit` on the new token will trigger this path. Because the migration flow is designed for ordinary token holders, and because storage registration is a separate, non-obvious prerequisite, this will affect a significant fraction of migrating users.

**Likelihood: High.**

### Recommendation

1. Add a callback after `burn.and(mint)` that checks both results. If `mint` failed, re-credit the user (or re-mint the old tokens) rather than silently losing them.
2. Before burning, verify that `sender_id` has storage registered on the new token. If not, either register it automatically (charging the user) or revert the entire swap so the old tokens are refunded.
3. Do not return `U128(0)` unconditionally from `ft_on_transfer` for `SwapMigratedToken`. Return `amount` (full refund) if the swap cannot be guaranteed to succeed, or only return `U128(0)` after confirming the mint succeeded in a callback.

### Proof of Concept

1. DAO calls `migrate_deployed_token(ChainKind::Eth, old_token, new_token)`.
2. Alice holds 1000 units of `old_token` and has never interacted with `new_token`.
3. Alice calls `ft_transfer_call` on `old_token`, sending 1000 units to the bridge with `msg = "SwapMigratedToken"`.
4. Bridge's `ft_on_transfer` returns `U128(0)` — Alice's 1000 old tokens are now in the bridge's balance with no refund possible.
5. Bridge fires `burn(1000)` on `old_token` (succeeds — bridge has the tokens and is still controller).
6. Bridge fires `mint(alice, 1000, None)` on `new_token` — panics because Alice has no storage registered on `new_token`.
7. Mint state is rolled back; burn state is not. Alice's 1000 old tokens are destroyed; she receives 0 new tokens. [1](#0-0) [2](#0-1) [3](#0-2)

#[near]
impl MintAndBurn for OmniToken {
    #[payable]
    fn mint(
        &mut self,
        account_id: AccountId,
        amount: U128,
        msg: Option<String>,
    ) -> PromiseOrValue<U128> {
        self.assert_controller();

        if let Some(msg) = msg {
            self.token
                .internal_deposit(&env::predecessor_account_id(), amount.into());

            self.ft_transfer_call(account_id, amount, None, msg)
        } else {
            self.token.internal_deposit(&account_id, amount.into());
            PromiseOrValue::Value(amount)
        }
    }

    fn burn(&mut self, amount: U128) {
        self.assert_controller();

        self.token
            .internal_withdraw(&env::predecessor_account_id(), amount.into());
    }
