### Title
`gas_prices_to_hash` Omits `l2_gas_consumed` and `next_l2_gas_price` for Starknet ≥ 0.14.0 Blocks, Producing Wrong Block Hash Commitment - (`File: crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

`gas_prices_to_hash` is called by `calculate_block_hash` to produce the gas-prices field of the Poseidon block hash. For Starknet version ≥ 0.14.0 the specification requires `l2_gas_consumed` and `next_l2_gas_price` to be included in that hash, but the implementation still uses the V0_13_4 formula that omits both fields. The codebase already runs at version 0.14.2 (confirmed by test fixtures), so every block produced at that version carries a structurally wrong block hash commitment.

---

### Finding Description

`gas_prices_to_hash` branches on `block_hash_version >= V0_13_4` and, for that branch, hashes only the six price fields (`l1_gas_price_wei/fri`, `l1_data_gas_price_wei/fri`, `l2_gas_price_wei/fri`):

---

### Impact Explanation

Every block produced at Starknet version ≥ 0.14.0 carries a block hash that is missing two mandatory fields (`l2_gas_consumed`, `next_l2_gas_price`) from the Poseidon preimage. The resulting hash diverges from the canonical Starknet specification. Concretely:

1. **Wrong block hash stored in storage** — the value written to the block-hash contract is incorrect for all 0.14.x blocks.
2. **Wrong `PartialBlockHash` / `ProposalCommitment`** — validators sign a commitment derived from the wrong hash; any external verifier or light client that recomputes the hash independently will disagree.
3. **Proof-facts validation broken** — `validate_proof_block_hash` reads the stored (wrong) hash and compares it against the hash embedded in `SnosProofFacts`; a correctly-generated SNOS proof that uses the canonical hash will be rejected, while a proof built against the wrong hash will be accepted.
4. **SNOS / transaction prover feeds wrong state root chain** — the block hash is chained into the global state root path used by the transaction prover; an incorrect hash propagates into every subsequent proof.

This matches the allowed impact: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"* and *"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution"* (via the proof-facts path).

---

### Likelihood Explanation

The bug is triggered unconditionally for every block at Starknet version ≥ 0.14.0. No special transaction or attacker action is required; the sequencer itself produces the wrong hash during normal block building. The codebase already targets version 0.14.2 in production fixtures, so the condition is permanently satisfied.

---

### Recommendation

Introduce a new `BlockHashVersion::V0_14_0` (or `V0_14_1`) variant and extend `gas_prices_to_hash` to chain `l2_gas_consumed` and `next_l2_gas_price` when `block_hash_version >= V0_14_0`:

```rust
if block_hash_version >= &BlockHashVersion::V0_14_0 {
    vec![
        HashChain::new()
            .chain(&STARKNET_GAS_PRICES0)
            .chain(&l1_gas_price.price_in_wei.0.into())
            .chain(&l1_gas_price.price_in_fri.0.into())
            .chain(&l1_data_gas_price.price_in_wei.0.into())
            .chain(&l1_data_gas_price.price_in_fri.0.into())
            .chain(&l2_gas_price.price_in_wei.0.into())
            .chain(&l2_gas_price.price_in_fri.0.into())
            .chain(&l2_gas_consumed.0.into())          // NEW
            .chain(&next_l2_gas_price.0.into())        // NEW
            .get_poseidon_hash(),
    ]
}
```

`gas_prices_to_hash` must receive `l2_gas_consumed` and `next_l2_gas_price` as additional parameters, sourced from `BlockHeaderWithoutHash`. `PartialBlockHashComponents` should be extended to carry these fields, and `calculate_block_commitments` / `BlockExecutionArtifacts::new` updated accordingly.

---

### Proof of Concept

1. Build a block at `StarknetVersion::V0_14_2` (the current `LATEST`).
2. Call `calculate_block_hash` with the resulting `PartialBlockHashComponents`.
3. Independently compute the canonical hash by including `l2_gas_consumed` and `next_l2_gas_price` in the `STARKNET_GAS_PRICES0` Poseidon preimage per the 0.14.x specification.
4. Observe that the two hashes differ.
5. Store the sequencer-produced hash in the block-hash contract, then submit an Invoke V3 transaction whose `proof_facts` reference that block with the canonical hash — `validate_proof_block_hash` will reject it with `"Block hash mismatch"`, demonstrating that the stored hash is wrong. ```rust
    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
