### Title
Router ETH Residue Theft via Unrestricted `refundETH()` and Total-Balance WETH Payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`exactInput` is `payable` and accepts ETH for WETH-input swaps. The pool callback requests only the exact amount needed, so any `msg.value` surplus is silently stranded on the router. Because `refundETH()` has no access control and `pay()` draws from `address(this).balance` (the router's total ETH, not the caller's `msg.value`), any attacker can steal that stranded ETH — either by calling `refundETH()` directly, or by calling `exactInput` with WETH input and `msg.value = 0` to have the router fund their swap from the victim's residue.

---

### Finding Description

**Step 1 — ETH stranding in `exactInput`.**

`exactInput` is declared `payable` and the `pay()` helper handles WETH input by consuming `address(this).balance`:

```
function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
``` [1](#0-0) 

Inside the callback, `_justPayCallback` calls `pay()` with `value` equal to exactly what the pool requested — not `msg.value`: [2](#0-1) 

`pay()` then reads the router's **total** native balance:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [3](#0-2) 

`address(this).balance` is the **entire** router ETH balance — it includes ETH stranded by prior callers, not just the current `msg.value`. After the pool callback consumes exactly `amountIn`, any `msg.value - amountIn` surplus remains on the router. `exactInput` never refunds it.

**Step 2 — `refundETH()` has no access control.**

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
``` [4](#0-3) 

Any address can call this and receive the router's full ETH balance. There is no check that `msg.sender` is the original depositor, no per-user accounting, and no reentrancy guard.

**Step 3 — `sweepToken` and `unwrapWETH9` are equally unrestricted.**

Both are `public payable` with no caller check: [5](#0-4) 

An attacker can also call `sweepToken(WETH, 0, attacker)` to drain any WETH that landed on the router during intermediate hops, or `unwrapWETH9(0, attacker)` to drain WETH and receive ETH.

**Step 4 — `pay()` lets an attacker's WETH swap be funded by the victim's stranded ETH.**

Because `pay()` uses `address(this).balance` unconditionally, an attacker who calls `exactInput` with WETH as input and `msg.value = 0` will have their payment drawn from whatever ETH is already on the router. The attacker's `payer` is `msg.sender` (attacker), but the router never pulls from the attacker's WETH allowance if `nativeBalance >= value`. The victim's ETH is consumed instead.

---

### Impact Explanation

Direct, unconditional loss of user ETH. Any caller who sends `msg.value > amountIn` (a common defensive pattern) loses the surplus to the first attacker who calls `refundETH()`. The attacker needs no special role, no token approval, and no pool interaction — a single external call suffices. The same ETH can also be silently consumed to subsidize an attacker's own WETH swap, making the theft harder to detect on-chain.

---

### Likelihood Explanation

- `exactInput` is a primary public entrypoint; WETH-input paths are a standard use case.
- Users routinely send a small ETH buffer above `amountIn` to avoid reverts from minor price movement.
- `refundETH()` is trivially callable; no mempool monitoring is required — the attacker can simply poll the router's ETH balance and call `refundETH()` whenever it is non-zero.
- The attack is atomic and requires no capital.

---

### Recommendation

1. **Track per-call ETH separately.** Record `msg.value` in a transient slot at the start of `exactInput` and pass it as the ceiling to `pay()`. Only use native ETH up to that recorded amount.
2. **Auto-refund at the end of `exactInput`.** After the hop loop and slippage check, call `_transferETH(msg.sender, address(this).balance)` (or a transient-tracked remainder) before returning.
3. **Restrict `refundETH()` to the original depositor**, or document clearly that it must always be bundled in the same `multicall` as the swap. If the design intent is multicall-only usage, add a guard that reverts if called outside a multicall context.
4. **Restrict `sweepToken` and `unwrapWETH9`** similarly, or at minimum require `amountMinimum > 0` so callers cannot drain dust left by others.

---

### Proof of Concept

```
// Setup: router has 0 ETH initially.

// Victim calls exactInput:
//   params.amountIn  = 0.5 ETH (WETH path)
//   msg.value        = 1.0 ETH  (common safety buffer)
router.exactInput{value: 1 ether}(params);
// Pool callback fires, pay() wraps 0.5 ETH → sends WETH to pool.
// 0.5 ETH remains on router. exactInput returns without refunding.

// Attacker (any address) calls:
router.refundETH();
// _transferETH(attacker, 0.5 ETH) executes.
// Attacker receives victim's 0.5 ETH. No approval, no role, no capital needed.

// --- Alternative attack (free swap) ---
// Attacker calls exactInput with WETH input, msg.value = 0:
router.exactInput{value: 0}(attackerParams); // amountIn = 0.3 ETH, token[0] = WETH
// pay() sees nativeBalance = 0.5 ETH >= 0.3 ETH
// Wraps 0.3 ETH of victim's residue, sends WETH to pool.
// Attacker receives swap output. Victim loses 0.3 ETH.
```
