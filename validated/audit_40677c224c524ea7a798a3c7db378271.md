### Title
Missing shard-index binding in `set_state_header` `chunk_proof` verification allows cross-shard state installation — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` verifies that the supplied chunk is present somewhere in the block's `chunk_headers_root` merkle tree, but never verifies that it is present at the leaf position that corresponds to the requested `shard_id`. An unprivileged peer can present a legitimately-signed chunk from shard N, together with its correct merkle path, as the state-sync header for shard M (M ≠ N). All five validation steps pass, and the wrong chunk is committed to `StateHeaderKey(M, sync_hash)`. Every subsequent `set_state_part` call then validates parts against shard N's `prev_state_root`, and `set_state_finalize` installs shard N's trie under shard M.

---

### Finding Description

**`verify_path` does not bind leaf position.**

`verify_path` is defined as:

```rust
// core/primitives/src/merkle.rs:113-118
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
```

It only checks that `compute_root_from_path(path, leaf_hash) == root`. It does **not** check which position in the tree the leaf occupies. A position-aware variant, `verify_path_with_index`, exists at line 121 and is used in chunk-part validation, but is **not** used in `set_state_header`.

**`chunk_headers_root` is a positional tree.**

`compute_chunk_headers_root` (block.rs:815-822) calls `merklize` over `ChunkHashHeight` values in shard-index order:

```rust
merklize(&self.iter()
    .map(|chunk| ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()))
    .collect_vec())
```

Leaf 0 is shard 0's chunk; leaf 1 is shard 1's chunk, etc. A valid merkle path for leaf 1 proves shard 1's chunk is in the tree — it does not prove it is at position 0.

**Step 3a uses `verify_path`, not `verify_path_with_index`.**

```rust
// chain/chain/src/state_sync/adapter.rs:394-403
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { ... }
```

The call proves the chunk is *somewhere* in the block. There is no subsequent assertion that `chunk.shard_id() == shard_id` anywhere in the function.

**`validate_chunk_proofs` does not check shard identity.**

```rust
// chain/chain/src/validate.rs:22-67
pub fn validate_chunk_proofs(chunk: &ShardChunk, epoch_manager: &dyn EpochManagerAdapter)
    -> Result<bool, Error>
```

It only verifies the chunk's internal hash, `tx_root`, and `receipts_root`. It does not compare `chunk.shard_id()` against any expected value.

**Step 4 (incoming receipts) uses the caller-supplied `shard_id`, not `chunk.shard_id()`.**

```rust
// adapter.rs:488
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
```

The attacker supplies valid incoming receipts for the *requested* shard (M), which are public on-chain data. These pass independently of which chunk was supplied.

**Step 5 (state root node) validates against `chunk_inner.prev_state_root()`.**

```rust
// adapter.rs:513-523
let chunk_inner = chunk.take_header().take_inner();
self.runtime_adapter.validate_state_root_node(
    shard_state_header.state_root_node(),
    chunk_inner.prev_state_root(),
)
```

`chunk_inner` is shard N's chunk, so this validates shard N's `state_root_node` against shard N's `prev_state_root` — which trivially passes when the attacker supplies shard N's authentic node.

### Impact Explanation

The exact corrupted value is `StateHeaderKey(shard_id=M, sync_hash)` → header whose `chunk.shard_id() = N` and `chunk.prev_state_root() = shard_N_state_root`.

Downstream consequences:

- `set_state_part` (adapter.rs:541-543) reads the stored header and extracts `state_root = chunk.take_header().take_inner().prev_state_root()` — shard N's root. It validates all incoming state parts against shard N's root. Parts for shard N's trie pass; parts for shard M's trie would fail, so the attacker also supplies shard N's parts.
- `set_state_finalize` (chain_update.rs:452-548) applies shard N's chunk for shard M, using shard N's `prev_state_root` as the starting trie. Shard M's `ShardUId` receives shard N's trie data.
- The syncing node operates with the wrong state for shard M for the remainder of the epoch, producing incorrect execution results, failing to serve correct data to its own clients, and potentially being slashed or forking. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

Any unprivileged peer that can respond to a `StateHeaderRequest` can execute this attack. All required data is public:

- Shard N's chunk body and its merkle path at position N in the block — available from any full node or archival node.
- Shard N's prev chunk header and its merkle path — same.
- Valid incoming receipts for shard M for the relevant block range — available from any node that tracked shard M.
- Shard N's `StateRootNode` — available from any node that holds shard N's state.

The attack is most straightforward when both shards have the same `height_included` (the common case when all chunks are included in every block), so the incoming-receipts block range is identical for both shards. The check at line 507 (`header.height() != prev_chunk_header.map_or(0, |h| h.height_included())`) uses shard N's `prev_chunk_header.height_included()`, which matches the block range the attacker provides for shard M's receipts. [10](#0-9) 

---

### Recommendation

Add an explicit shard-identity guard immediately after extracting the chunk, before any other validation:

```rust
// After line 377 in adapter.rs
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Optionally, replace the bare `verify_path` call in step 3a with `verify_path_with_index`, supplying the expected shard index derived from `shard_layout.get_shard_index(shard_id)`, to make the position binding cryptographically explicit rather than relying solely on the `shard_id` field inside the chunk header. ### Proof of Concept

```
1. Set up a two-shard chain (shard 0, shard 1) and advance it past genesis.
2. Choose a sync_hash. Let B_prev = block at sync_hash.prev_hash.
3. Obtain shard 1's chunk from B_prev: chunk1, with chunk1.shard_id() == 1.
4. Compute chunk_headers_root paths for B_prev:
       (root, paths) = merklize([ChunkHashHeight(chunk0.hash, h0),
                                  ChunkHashHeight(chunk1.hash, h1)])
   paths[1] is the valid merkle path for chunk1.
5. Obtain shard 1's prev_chunk_header and its merkle path from B_prev.prev_block.
6. Obtain valid incoming receipts for shard 0 for the block range
   [sync_hash .. prev_chunk_header.height_included].
7. Obtain shard 1's StateRootNode.
8. Construct ShardStateSyncResponseHeader {
       chunk: chunk1,
       chunk_proof: paths[1],
       prev_chunk_header: shard1_prev_chunk_header,
       prev_chunk_proof: shard1_prev_chunk_proof,
       incoming_receipts_proofs: <valid receipts for shard 0>,
       root_proofs: <matching root proofs>,
       state_root_node: shard1_state_root_node,
   }
9. Call set_state_header(shard_id=0, sync_hash, crafted_header).
10. Assert: get_state_header(0, sync_hash).chunk.shard_id() == 1  // wrong shard stored
11. Assert: get_state_header(0, sync_hash).chunk.prev_state_root()
            == shard1_state_root  // shard 1's root will be used for shard 0's parts
```

```rust
pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
```
