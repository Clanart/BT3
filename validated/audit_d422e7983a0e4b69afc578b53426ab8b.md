### Title
Unauthenticated `eth_resend` Allows Any Caller to Re-sign and Broadcast Another Keyring Account's Pending Transaction — (`rpc/backend/call_tx.go`)

---

### Summary

`Resend` accepts a caller-supplied `args.From` and uses it directly as the identity to match against pending transactions and then to sign a new transaction via `SendTransaction`. There is no check that the caller owns or is authorized to act on behalf of `args.From` before the match loop or before signing. Any unprivileged caller who can reach `eth_resend` and knows the fields of another keyring account's pending transaction can trigger re-signing of that transaction with an attacker-chosen gas price/limit.

---

### Finding Description

The `Resend` function in `rpc/backend/call_tx.go` performs the following steps:

1. Calls `SetTxDefaults(args)` — fills in fee defaults but **does not validate that `args.From` is in the keyring**.
2. Builds `matchTx` from the attacker-supplied `args` and computes `wantSigHash = signer.Hash(matchTx)`.
3. Iterates over all pending transactions, recovering `pFrom` from each pending tx's actual ECDSA signature.
4. Matches on `pFrom == *args.From && signer.Hash(pTx) == wantSigHash`.
5. On match, updates `args.GasPrice`/`args.Gas` with attacker-supplied values and calls `b.SendTransaction(args)`. [1](#0-0) 

`SendTransaction` then checks only that `args.From` exists in the keyring: [2](#0-1) 

Since the victim is a keyring account (precondition), this check passes, and `msg.Sign(signer, b.clientCtx.Keyring)` signs the new transaction with the victim's private key: [3](#0-2) 

The attacker satisfies the match condition by:
- Setting `args.From = victim_keyring_address` (so `pFrom == *args.From` is true, since `pFrom` is recovered from victim's own signed pending tx)
- Supplying the same `nonce`, `to`, `value`, `data`, `gas`, `gasPrice` as victim's pending tx (observable via `eth_getPendingTransactions` or mempool), so `signer.Hash(pTx) == wantSigHash`

After the match, `args.GasPrice` is replaced with the attacker's higher value, and `SendTransaction` re-signs and broadcasts the tx using victim's key without victim's consent.

The `eth_resend` RPC entry point passes args directly to the backend with no caller authentication: [4](#0-3) 

---

### Impact Explanation

An unprivileged attacker can cause the node's keyring to sign and broadcast a transaction on behalf of any other keyring account without that account holder's consent. The re-signed transaction carries the attacker-chosen gas price (up to the `RPCTxFeeCap` limit), causing the victim to pay higher fees than intended — an unauthorized EVM-denom balance transfer. Additionally, the victim's original pending tx is effectively replaced by a new tx with a different hash, disrupting the victim's intended transaction flow. The `to`, `value`, and `data` fields cannot be changed (they must match the pending tx for the hash check), so direct fund redirection is not possible, but unauthorized key usage and forced fee escalation are concrete impacts.

---

### Likelihood Explanation

- `eth_resend` is a public JSON-RPC method with no caller authentication at the RPC layer.
- Pending transaction fields are observable via `eth_getPendingTransactions` or by watching the CometBFT mempool.
- The only precondition is that the node keyring contains multiple accounts and at least one has a pending transaction — a normal operating condition for any multi-account node.
- No privileged access, leaked keys, or network-level attack is required.

---

### Recommendation

Add a keyring membership check for `args.From` at the **start** of `Resend`, before the match loop, mirroring what `SendTransaction` does:

```go
// In Resend, immediately after the nonce nil-check:
if args.From == nil {
    return common.Hash{}, fmt.Errorf("missing from address")
}
if _, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.From.Bytes())); err != nil {
    return common.Hash{}, fmt.Errorf("from address not found in keyring: %w", err)
}
```

This alone is insufficient for authentication (any caller can supply any keyring address). The deeper fix is to require that `eth_resend` is only callable by the account that owns `args.From`, which requires RPC-level caller authentication (e.g., comparing against an authenticated session identity). As a minimum viable fix, the keyring check prevents re-signing for addresses not managed by this node.

---

### Proof of Concept

```
Setup:
  - Node keyring has account_A (victim) and account_B (attacker)
  - account_A submits a pending tx: nonce=5, to=0xDEAD, value=1e18, gasPrice=1gwei, gas=21000

Attack:
  attacker calls eth_resend with:
    args.From    = account_A  // victim's address
    args.Nonce   = 5
    args.To      = 0xDEAD
    args.Value   = 1e18
    args.GasPrice = 1gwei     // must match pending tx for hash equality
    args.Gas     = 21000
    gasPrice param = 100gwei  // new higher gas price
    gasLimit param = 21000

Result:
  - Resend match loop: pFrom=account_A == args.From=account_A ✓
                       signer.Hash(pTx) == wantSigHash ✓
  - SendTransaction called with args.From=account_A, gasPrice=100gwei
  - Keyring signs with account_A's private key (no consent)
  - New tx broadcast: account_A pays 100gwei * 21000 = 2.1M gwei instead of 21k gwei
``` [5](#0-4)
