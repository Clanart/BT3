### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS program does not verify that the new class hash supplied by a contract corresponds to a previously declared contract class. This is structurally identical to the external report's vulnerability class: one action (class declaration) is properly constrained, while a related action (class replacement) that must also be constrained is not. An unprivileged contract can replace its own class hash with an arbitrary undeclared value, permanently making the contract non-callable and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash exists in `contract_class_changes` (the mapping of declared class hashes to compiled class hashes):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. The OS developers intended to add this validation but have not done so.

By contrast, `execute_entry_point` — the function that actually dispatches execution to a contract — **does** require the class hash to be present in `contract_class_changes`. It calls `dict_read` on `contract_class_changes` to obtain the compiled class hash, then calls `find_element` which will panic (causing OS execution failure) if the compiled class hash is not found:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

The asymmetry is exact:
- `execute_entry_point` enforces that a class hash must be declared before it can be executed ✓
- `execute_replace_class` does **not** enforce that the replacement class hash must be declared ✗

This is the direct analog of the Wildcat bug: withdrawal is blocked during the fixed term, but APR reduction (the related action) is not.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract calls `replace_class(undeclared_hash)`:

1. The OS writes `contract_state_changes[contract_address].class_hash = undeclared_hash` — this is committed to the proven state.
2. Any subsequent transaction that calls the contract causes `execute_entry_point` to look up `undeclared_hash` in `contract_class_changes`, receiving `0` (the default dict value for an undeclared key).
3. `find_element` with key `0` finds no compiled class and panics, making the contract permanently non-callable at the OS/proof level.
4. All funds held in the contract's storage are permanently inaccessible — no withdrawal, no transfer, no recovery path exists.

---

### Likelihood Explanation

The `replace_class` syscall is available to any executing contract with no caller restriction. An attacker can:

- Deploy a contract that holds user funds (e.g., a vault or escrow) and embed a call to `replace_class(arbitrary_felt)` triggered by a specific condition.
- Or exploit a reentrancy or logic bug in a third-party contract to invoke `replace_class` with an attacker-controlled argument.

The entry path is fully unprivileged: any transaction sender can deploy a contract and invoke this syscall. The missing check is confirmed by the in-code TODO, meaning no defensive logic exists at the OS layer.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that the class hash exists in `contract_class_changes`:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already implicitly enforced by `execute_entry_point` and closes the asymmetry between the two functions.

---

### Proof of Concept

1. Deploy a contract `MaliciousVault` that accepts user deposits and exposes a `drain()` function.
2. `drain()` calls the `replace_class` syscall with `class_hash = 0xDEAD` (an undeclared felt).
3. The OS executes `execute_replace_class`: no validation occurs, `contract_state_changes[MaliciousVault].class_hash = 0xDEAD` is committed.
4. Users attempt to withdraw their funds by calling `MaliciousVault`.
5. `execute_entry_point` reads `contract_class_changes[0xDEAD]` → returns `0`.
6. `find_element(..., key=0)` finds no compiled class → OS panics or the call permanently reverts.
7. All deposited funds are permanently frozen with no recovery path. [3](#0-2)
