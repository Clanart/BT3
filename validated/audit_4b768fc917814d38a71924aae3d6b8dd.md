### Title
`normalize_amount` Floor Division Can Round Transfer Amount to Zero, Permanently Locking User Funds â€” (`near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` locks user tokens into the NEAR bridge contract before validating that the decimal-normalized transfer amount is non-zero. When a token has `origin_decimals > decimals`, any user-supplied amount smaller than `10^(origin_decimals - decimals)` normalizes to zero via floor division. Because `sign_transfer` then reverts with `InvalidAmountToTransfer` and no cancel/refund path exists, the user's tokens are permanently locked.

---

### Finding Description

The NEAR bridge normalizes token amounts when crossing decimal boundaries. The normalization function is:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

When `origin_decimals > decimals` (e.g., a token with 24 origin decimals bridged to a chain that uses 18 decimals, giving `diff_decimals = 6`), any `amount < 10^6` normalizes to `0`.

The `init_transfer` function only checks that `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

It does **not** check that `normalize_amount(amount - fee) > 0`. The tokens are locked immediately when `ft_on_transfer` â†’ `init_transfer` â†’ `init_transfer_internal` stores the transfer message and returns `U128(0)` (no refund to the FT contract).

Later, when a relayer calls `sign_transfer`, the zero-amount check fires and reverts:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

Because `sign_transfer` will always revert for this transfer, and there is no `cancel_transfer` or refund path, the locked tokens are irrecoverable.

The `Decimals` struct stores both values and is set at token registration time:

```rust
pub struct Decimals {
    pub decimals: u8,
    pub origin_decimals: u8,
}
``` [4](#0-3) 

Tokens with large decimal differences are explicitly supported by the protocol (e.g., Solana tokens capped at `MAX_ALLOWED_DECIMALS` with their original decimals stored as `origin_decimals`). [5](#0-4) 

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.** Any user who calls `ft_transfer_call` with an amount (minus fee) below the decimal-normalization threshold for that token will have their tokens permanently locked in the NEAR bridge contract. The `sign_transfer` call will always revert, and no cancel or refund mechanism exists to recover the funds.

---

### Likelihood Explanation

**Low-to-medium.** The condition requires a token registered with `origin_decimals > decimals` (a normal configuration for tokens bridged from high-decimal chains) and a user sending a sub-threshold amount. A user could trigger this accidentally (e.g., sending a "dust" test transfer), or an attacker could socially engineer a victim into doing so. The protocol's own comment acknowledges that dust is locked when `fee = 0`, but does not guard against the case where the *entire* amount is sub-threshold.

---

### Recommendation

Add a zero-amount guard inside `init_transfer` (or `init_transfer_internal`) before locking tokens, mirroring the check already present in `sign_transfer`:

```rust
// After computing the transfer_message, before locking:
let token_address = self.get_token_address(...);
if let Some(decimals) = self.token_decimals.get(&token_address) {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

This ensures tokens are never locked when the normalized transfer amount would be zero, consistent with the guard already enforced in `sign_transfer`.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`). The normalization divisor is `10^6 = 1_000_000`.
2. A user calls `ft_transfer_call` with `amount = 500_000` (below the threshold) and `fee = 0`.
3. `init_transfer` passes the `fee < amount` check (`0 < 500_000`).
4. `init_transfer_internal` stores the transfer message and returns `U128(0)` â€” tokens are locked.
5. Any relayer calls `sign_transfer` for this transfer ID.
6. `normalize_amount(500_000 - 0, Decimals { decimals: 18, origin_decimals: 24 }) = 500_000 / 1_000_000 = 0`.
7. `require!(0 > 0, ...)` panics with `InvalidAmountToTransfer`.
8. No other path exists to recover the 500,000 locked tokens. [6](#0-5) [1](#0-0)
