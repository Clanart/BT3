### Title
`batch_insert` Bypasses Duplicate-Key Guard, Corrupting DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces for every item beyond the first two, allowing an unprivileged caller to silently insert duplicate keys into the Merkle tree. The resulting tree has an incorrect root hash, and any proof of inclusion derived from it is invalid or ambiguous.

---

### Finding Description

`MerkleBlob::insert` enforces two guards before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` (with those guards) only for the first two items when the tree has ≤ 1 existing leaf. All remaining items are written directly to the blob via `insert_entry_to_blob` with **no duplicate check**:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 578-602
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← guarded
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← ALL remaining items
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { key, value, hash, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?; // ← NO duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When the tree already has ≥ 2 leaves the `if` branch is skipped entirely, so **every** item in the batch bypasses the guard.

Two concrete exploitation paths exist:

1. **Intra-batch duplicate**: pass `[(K, V1, H1), (K, V2, H2), ...]` — the same key `K` appears twice in the input. Both entries are written to the blob; the `block_status_cache` is updated for only one of them (or neither, depending on tree size), leaving the tree in an inconsistent state.

2. **Cross-call duplicate**: call `batch_insert` with a key that already exists in the tree. Because the fast path writes directly to the blob without consulting `block_status_cache`, the existing leaf is not detected and a second leaf with the same key is appended.

In both cases the subtree is then grafted onto the existing tree via `insert_subtree_at_key`, producing a structurally connected but semantically corrupt tree whose root hash is computed over duplicate keys. [3](#0-2) 

The Python binding `py_batch_insert` exposes this path directly to untrusted callers: [4](#0-3) 

---

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

- The root hash stored in the committed DataLayer state is computed over a tree that contains duplicate keys. Any proof of inclusion generated from this tree is either ambiguous (two leaves match the same key) or provably wrong when verified against the correct root.
- A client that receives a proof-of-inclusion for key `K` cannot distinguish which of the two duplicate leaves was intended, enabling a server to selectively present either value as "proven."
- The `block_status_cache` diverges from the blob, so subsequent `insert`, `delete`, and `upsert` operations on the same `MerkleBlob` object may panic, silently corrupt further, or produce wrong results.

---

### Likelihood Explanation

The Python binding is the primary DataLayer API surface. Any caller that constructs a batch with repeated keys — whether by mistake or deliberately — triggers the bug. No special privilege is required; the function is `pub` in Rust and exposed as a normal method in Python. The existing test `test_double_insert_fails` only exercises the single-item `insert` path and does not cover `batch_insert` with duplicates. [5](#0-4) 

---

### Recommendation

Add the same duplicate-key and duplicate-hash checks at the top of the fast-path loop in `batch_insert`, mirroring what `insert` does:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing blob write ...
}
```

Alternatively, deduplicate the input vector before processing and validate it against the existing cache. A fuzz target analogous to `merkle_blob_insert.rs` should be added for `batch_insert` to catch regressions. [6](#0-5) 

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so the `if leaf_count <= 1` branch is skipped
    for i in 0i64..2 {
        blob.insert(
            KeyId(i), ValueId(i),
            &Hash(Bytes32::new([i as u8; 32])),
            InsertLocation::Auto {},
        ).unwrap();
    }

    // batch_insert with a duplicate key (KeyId(0) already exists)
    let batch = vec![
        ((KeyId(0), ValueId(99)), Hash(Bytes32::new([0xAA; 32]))), // duplicate!
        ((KeyId(5), ValueId(5)),  Hash(Bytes32::new([5u8; 32]))),
        ((KeyId(6), ValueId(6)),  Hash(Bytes32::new([6u8; 32]))),
    ];
    blob.batch_insert(batch).unwrap(); // succeeds — no error raised

    // Tree now has two leaves with KeyId(0); root hash is corrupt.
    // check_integrity() will detect the inconsistency:
    blob.check_integrity().expect_err("tree is corrupt");
}
```

The call to `batch_insert` succeeds without error despite `KeyId(0)` already being present. The resulting root hash is computed over a tree with two leaves sharing the same key, making any proof of inclusion for `KeyId(0)` ambiguous and unverifiable. [7](#0-6)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
