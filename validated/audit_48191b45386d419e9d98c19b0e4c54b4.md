### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by the caller has been declared in the `contract_class_changes` dictionary. This is the direct analog of the reported "stale state after removal" bug class: just as `removeContract` leaves method policies uncleaned (allowing stale state to persist), `execute_replace_class` writes an unvalidated class hash into contract state (allowing invalid state to persist). A contract can replace its own class hash with any arbitrary, undeclared felt value, rendering itself permanently uncallable and freezing any funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` (lines 877–916) updates a contract's `class_hash` field in `contract_state_changes` without verifying that the new class hash corresponds to any entry in `contract_class_changes` (the class registry). The code even carries an explicit TODO acknowledging this missing check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

By contrast, `execute_declare_transaction` correctly enforces class integrity by requiring `prev_value=0` in the `contract_class_changes` dict update, ensuring a class can only be declared once and that the compiled class hash is non-zero:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

The same missing validation exists in the deprecated syscall path: [3](#0-2) 

The `StateEntry` struct stores `class_hash` as a plain felt with no invariant enforcing it must be declared: [4](#0-3) 

When the OS later processes a call to the affected contract, it must resolve the `class_hash` to a compiled class via `contract_class_changes`. If the hash was never declared, no valid compiled class exists, and no valid proof can be generated for any transaction targeting that contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to an undeclared value:

1. The `contract_state_changes` dict records the new (invalid) class hash as the contract's authoritative class.
2. In any subsequent block, any transaction calling this contract requires the OS to look up a compiled class for the undeclared hash. Since no such class exists in `contract_class_changes`, the sequencer/prover cannot produce a valid proof for that transaction.
3. The contract becomes permanently uncallable. All funds (native ETH, ERC-20 balances, or any other assets) stored in the contract's storage are permanently frozen with no recovery path, because no valid execution can reach the contract's withdrawal or transfer logic.

The freezing is irreversible unless someone can declare a class whose Sierra hash collides with the arbitrary value — computationally infeasible for a random felt.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender through the following realistic paths:

- **Direct self-freeze:** A contract with an upgrade function that accepts a user-supplied class hash (a common pattern) and does not independently validate that the hash is declared. The OS is supposed to enforce this invariant but does not.
- **Malicious deployer:** An attacker deploys a shared pool, escrow, or multi-sig contract, attracts user deposits, then calls `replace_class` with an arbitrary hash, permanently freezing all deposited funds.
- **Constructor-time freeze:** A contract calls `replace_class(arbitrary_hash)` during its own constructor execution. The constructor succeeds (the class hash change only takes effect in state), but the contract is immediately frozen for all future calls.

The TODO comment at line 898 is a developer acknowledgment that this check is intentionally absent from the current implementation, confirming the gap is real and not a false positive.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, perform a lookup in `contract_class_changes` to assert the hash is declared:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the invariant already enforced in `execute_declare_transaction` and closes the state-consistency gap.

---

### Proof of Concept

1. Attacker deploys `VaultContract` (holds user ETH deposits) with an `upgrade(new_class_hash: felt)` function that calls `replace_class(new_class_hash)` without validation.
2. Users deposit funds into `VaultContract`.
3. Attacker calls `upgrade(0xdeadbeef)` — an arbitrary felt with no corresponding declared class.
4. The OS processes `execute_replace_class`: no check against `contract_class_changes` is performed; `VaultContract`'s `class_hash` in `contract_state_changes` is updated to `0xdeadbeef`.
5. In the next block, any user attempting to call `VaultContract.withdraw()` causes the OS to look up the compiled class for `0xdeadbeef`. No such class exists; the sequencer cannot include the transaction in a provable block.
6. All user funds in `VaultContract` are permanently frozen.
