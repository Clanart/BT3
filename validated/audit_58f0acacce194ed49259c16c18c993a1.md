### Title
Cross-Shard State Root Substitution via Unbound Merkle Position in `set_state_header` - (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` verifies that the supplied chunk's hash appears somewhere in `sync_prev_block.chunk_headers_root` via `verify_path`, but never checks that the chunk occupies the shard-layout position for the requested `shard_id`. An unprivileged peer can supply shard B's chunk (with `chunk.shard_id()=B`) as the response for `shard_id=A`, pass every validation gate, and cause the node to store shard B's `prev_state_root` under shard A's `StateHeaderKey`. All subsequent `set_state_part` calls validate and store shard B's trie parts under shard A's `StatePartKey` entries, and `set_state_finalize` applies shard B's state and writes a `ChunkExtra` for shard A whose `state_root` is shard B's root.

---

### Finding Description

**Entrypoint.** Any peer can send `PeerMessage::VersionedStateResponse`, which is forwarded without peer-authentication to `StateResponseReceived` → `state_sync.apply_peer_message` → `StateHeaderValidationRequest` → `ClientActor::handle` → `chain.state_sync_adapter.set_state_header(shard_id, sync_hash, header)`. [1](#0-0) [2](#0-1) 

**The missing shard-position check.** Inside `set_state_header`, step 3a verifies chunk inclusion with:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
``` [3](#0-2) 

`verify_path` is a pure hash-path computation that returns `true` whenever `compute_root_from_path(path, hash(item)) == root`. It does **not** check which leaf index the item occupies. [4](#0-3) 

The `chunk_headers_root` is built by merklizing `ChunkHashHeight` values in shard-index order:

```rust
merklize(&self.iter().map(|chunk|
    ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included())
).collect_vec())
``` [5](#0-4) 

Shard B's chunk sits at leaf index `shard_layout.get_shard_index(B)`. Its merkle proof is valid for that position. `verify_path` accepts it regardless of whether the caller requested shard A or shard B, because position is never checked. The contrast with `verify_path_with_index` (used for erasure-coded chunk parts) makes the omission concrete: [6](#0-5) 

**Remaining validation gates all pass.**

- `validate_chunk_proofs` checks internal chunk consistency (hash, tx root, receipts root) — no `shard_id` vs. request `shard_id` comparison. [7](#0-6) 
- Receipt proofs (step 4) are validated against `shard_id` (the *requested* shard A) using on-chain data the attacker can copy verbatim from the canonical chain. [8](#0-7) 
- `validate_state_root_node` (step 5) checks that the supplied `state_root_node` is consistent with `chunk_inner.prev_state_root()` — shard B's root. The attacker provides a valid node for shard B's trie, which passes. [9](#0-8) 

**Storage.** The header is written under `StateHeaderKey(shard_id=A, sync_hash)` but contains shard B's chunk. [10](#0-9) 

**`set_state_part` propagation.** The stored header is read back; `state_root` is extracted from the embedded chunk's inner:

```rust
let state_root = *chunk.take_header().take_inner().prev_state_root();
``` [11](#0-10) 

Parts are validated against shard B's `state_root`. The attacker supplies shard B's trie parts, which pass `validate_state_part`. They are stored under `StatePartKey(sync_hash, shard_id=A, part_id)`. [12](#0-11) 

**`set_state_finalize` corruption.** `set_state_finalize` reads the stored header and calls `apply_chunk` with `RuntimeStorageConfig::new(chunk_header.prev_state_root(), true)` — shard B's root — for shard A's `shard_uid`. The resulting `ChunkExtra` written for shard A has `state_root = shard_B_prev_state_root`. [13](#0-12) [14](#0-13) 

---

### Impact Explanation

The syncing node permanently installs shard B's state under shard A's identity. Every subsequent block application for shard A reads the wrong trie, producing wrong execution results, wrong outgoing receipts, and wrong `ChunkExtra.state_root`. The node diverges from the canonical chain and cannot recover without a full re-sync.

---

### Likelihood Explanation

Any peer connected to a syncing node can send `VersionedStateResponse`. No validator key or privileged role is required. All inputs (shard B's chunk, merkle proof, state root node, trie parts, receipt proofs) are publicly available on-chain data. The attack requires only that the attacker respond to the syncing node's state request before the legitimate provider does, or that the syncing node accepts the first valid-looking response.

---

### Recommendation

After the `verify_path` check passes, add an explicit shard-position guard:

```rust
let shard_layout = self.epoch_manager.get_shard_layout(sync_block_header.epoch_id())?;
let expected_shard_index = shard_layout.get_shard_index(shard_id)?;
// Verify the chunk is at the correct shard-index position in the merkle tree
if !verify_path_with_index(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
    expected_shard_index as u64,
    shard_layout.num_shards(),
) { … }
```

Alternatively, add a direct equality check: `chunk.shard_id() == shard_id` immediately after extracting the chunk, before any merkle verification.

---

### Proof of Concept

In a two-shard `TestEnv`:

1. Produce several blocks so both shards have non-trivial state.
2. Obtain the legitimate `ShardStateSyncResponseHeaderV2` for `shard_id=1` (shard B) from client 0.
3. Swap the `shard_id` field in the `StateResponseInfoV2` wrapper to `0` (shard A), keeping the embedded chunk (shard 1's chunk) and its merkle proof unchanged.
4. Call `client1.chain.state_sync_adapter.set_state_header(shard_id=0, sync_hash, tampered_header)` — expect `Ok(())`.
5. Obtain shard 1's state parts from client 0 and call `set_state_part(shard_id=0, …)` for each — expect all `Ok(())`.
6. Call `client1.chain.set_state_finalize(shard_id=0, sync_hash)`.
7. Assert `chain_store.get_chunk_extra(prev_hash, shard_uid_0).state_root() == shard_1_prev_state_root` — demonstrating shard A's `ChunkExtra` holds shard B's state root.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
