Looking at the actual code in `set_state_header`, I need to trace all five validation steps and check whether any of them enforce `chunk.shard_id() == shard_id`.

**Step 1-2: `validate_chunk_proofs`** [1](#0-0) 

This checks internal consistency of the chunk body against its own header (hash integrity, tx root, receipts root). It does **not** check `chunk.shard_id()` against any external `shard_id` argument.

**Step 3a: `verify_path` against `chunk_headers_root`** [2](#0-1) 

The `chunk_headers_root` is a merkle tree over all shards' chunks ordered by shard index. The `verify_path` call proves that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in that tree. The merkle path encodes position (left/right directions), so it IS position-specific — but the code **never checks that the encoded position equals `shard_id`**. An attacker supplying shard 0's chunk with shard 0's valid proof (position 0) will pass `verify_path` even when `shard_id=1`. [3](#0-2) 

**Step 3b: prev_chunk proof** — same analysis; no shard_id position check.

**Step 4: Receipt proofs** [4](#0-3) 

Step 4e computes `ReceiptList(shard_id, receipts)` using the **argument** `shard_id` (not `chunk.shard_id()`). An attacker can supply the actual on-chain incoming receipts for shard 1 (public data) with their valid proofs. These pass because the receipts genuinely belong to shard 1.

**Step 5: `validate_state_root_node`** [5](#0-4) 

This checks that `state_root_node`'s hash matches `chunk_inner.prev_state_root()`. Since `chunk_inner` is from shard 0's chunk, and the attacker provides shard 0's `state_root_node`, this passes. No `shard_id` argument is involved.

**Storage** [6](#0-5) 

`StateHeaderKey(shard_id=1, sync_hash)` is written with shard 0's header — containing shard 0's `prev_state_root`.

**Impact on `set_state_part`** [7](#0-6) 

`set_state_part` reads the stored header and extracts `state_root` from it — which is now shard 0's `prev_state_root`. It calls `validate_state_part(shard_id=1, &shard_0_state_root, ...)`. Shard 0's state parts validate correctly against shard 0's state root and get installed as shard 1's state.

The syncing node requests a header for shard 1 from a peer. The peer (unprivileged) returns a crafted `ShardStateSyncResponseHeader` containing:

All five steps pass. The `shard_id` argument is never compared to `chunk.shard_id()` anywhere in the function. [8](#0-7)

### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` from an untrusted peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the embedded chunk belongs to `shard_id`. All five validation steps can be satisfied with a chunk from a different shard.

### Finding Description
In `chain/chain/src/state_sync/adapter.rs`, `set_state_header` performs five checks:

1. `validate_chunk_proofs` — verifies internal chunk consistency, not shard identity.
2. `verify_path` against `sync_prev_block_header.chunk_headers_root()` — proves the chunk is *somewhere* in the block's chunk merkle tree, but does not verify the leaf position equals `shard_id`.
3. `prev_chunk` proof — same gap.
4. Receipt proofs — uses the `shard_id` argument (not `chunk.shard_id()`) to compute `ReceiptList(shard_id, receipts)`, so an attacker can supply the real on-chain receipts for the target shard.
5. `validate_state_root_node` — checks hash consistency against `chunk_inner.prev_state_root()`, which is the attacker-supplied chunk's state root.

After all checks pass, `StateHeaderKey(shard_id, sync_hash)` is written with the wrong chunk, embedding the wrong `prev_state_root`. Subsequent `set_state_part` calls extract this wrong state root and validate parts against it.

### Impact Explanation
A syncing node that accepts a crafted header installs the wrong `prev_state_root` for the target shard. State parts for the wrong shard validate correctly against the wrong root and are applied via `apply_state_part`, corrupting the shard's trie. The node then operates on incorrect state for that shard for the remainder of the epoch.

### Likelihood Explanation
The attack requires the malicious peer to be selected as the state sync source for the victim node, and to possess the state of the source shard (to supply a valid `state_root_node`). Both conditions are achievable by an unprivileged node that tracks any shard. The crafted header uses only public on-chain data.

### Recommendation
Add an explicit shard identity check immediately after `validate_chunk_proofs`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match expected shard_id".into(),
    ));
}
```

Similarly verify `prev_chunk_header.shard_id() == shard_id` when `prev_chunk_header` is `Some`.

### Proof of Concept
In a two-shard test environment:
1. Obtain shard 0's chunk, its merkle proof in `sync_prev_block`, and its `state_root_node`.
2. Obtain shard 1's actual incoming receipts and their `root_proofs` from the chain.
3. Construct a `ShardStateSyncResponseHeaderV2` with shard 0's chunk/proof/state_root_node and shard 1's receipt proofs.
4. Call `set_state_header(shard_id=1, sync_hash, crafted_header)` — assert it returns `Ok(())`.
5. Read back `StateHeaderKey(1, sync_hash)` — assert `chunk_prev_state_root()` equals shard 0's state root, not shard 1's.
6. Call `set_state_part` with shard 0's parts — assert they pass validation and are stored under `StatePartKey(sync_hash, shard_id=1, ...)`.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
