### Title
Receipt Commitment Omits Actual `l2_gas` Consumed, Allowing Unverifiable L2 Gas Values in RPC Receipts — (`crates/starknet_api/src/block_hash/receipt_commitment.rs`)

### Summary

`chain_gas_consumed` in the receipt commitment calculation hardcodes `Felt::ZERO` for the L2 gas position instead of chaining the actual `gas_consumed.l2_gas` value from the `GasVector` struct. Because the receipt commitment (and therefore the block hash) never commits to per-transaction L2 gas consumed, any RPC node can return an arbitrary `l2_gas` value in a transaction receipt and no client can detect the forgery by verifying against the block hash.

### Finding Description

`calculate_receipt_hash` builds the per-receipt leaf that feeds into `ReceiptCommitment`, which is then chained into the block hash via `calculate_block_hash`. The gas-consumed portion of that leaf is assembled by `chain_gas_consumed`:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-90
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)               // ← hardcoded zero, NOT gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

The `GasVector` struct that is passed in carries three fields:

```
l1_gas, l1_data_gas, l2_gas
```

`l1_gas` and `l1_data_gas` are committed. `l2_gas` is silently replaced by a constant zero. The comment acknowledges this: *"L2 gas consumed (In the current RPC: always 0)"*. However, starting with Starknet 0.14.x, `l2_gas` is non-zero and economically significant — the test fixture `block_post_0_14_2.json` shows per-transaction `l2_gas` values of `76014735` and a block-level `l2_gas_consumed` of `988191555`.

The receipt element is constructed from `TransactionOutputForHash`, which already carries the real `gas_consumed.l2_gas`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  lines 99-105
pub struct TransactionOutputForHash {
    pub actual_fee: Fee,
    pub events: Vec<Event>,
    pub execution_status: TransactionExecutionStatus,
    pub gas_consumed: GasVector,   // ← l2_gas is populated here
    pub messages_sent: Vec<MessageToL1>,
}
```

The value is present but discarded at hash time.

### Impact Explanation

The receipt commitment is a Patricia-tree root over per-receipt hashes; it is included verbatim in the block hash:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  lines 263-264
.chain(&block_commitments.receipt_commitment.0)
```

Because `l2_gas` is not part of any receipt leaf, the receipt commitment — and therefore the block hash — is invariant to the L2 gas value reported for any transaction. An RPC node serving `starknet_getTransactionReceipt` can return any `l2_gas` figure (zero, inflated, or deflated) and the value is structurally unverifiable against the committed block hash. Clients relying on the receipt to audit fee breakdowns, estimate future costs, or feed the value into proof-verification logic receive an authoritative-looking but unverifiable number.

### Likelihood Explanation

The bug is present in every block produced after L2 gas became non-zero (Starknet ≥ 0.14.0). Any RPC provider — including a malicious or misconfigured one — can silently misreport `l2_gas` without triggering any commitment mismatch. No privileged access is required; the attacker only needs to serve RPC responses.

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual field value:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())    // was: Felt::ZERO
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

This change must be coordinated with a Starknet version bump (analogous to the `V0_13_4` gate already used elsewhere in `calculate_block_commitments`) so that historical blocks computed under the old rule remain verifiable.

### Proof of Concept

1. Observe that `chain_gas_consumed` chains `Felt::ZERO` at the L2-gas position regardless of the input `GasVector`.
2. Take any post-0.14.0 block where `l2_gas > 0` (e.g., the fixture at `crates/apollo_starknet_client/resources/reader/block_post_0_14_2.json` with `l2_gas: 76014735`).
3. Compute the receipt hash for that transaction with `l2_gas = 76014735` and again with `l2_gas = 0`. Both produce the same Poseidon digest because `chain_gas_consumed` substitutes `Felt::ZERO` in both cases.
4. The receipt commitment and block hash are therefore identical for both inputs, confirming that any RPC-reported `l2_gas` value passes commitment verification. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
