### Title
Native Fee Tokens Permanently Locked in Bridge Contracts With No Withdrawal Mechanism — (`starknet/src/omni_bridge.cairo`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The Starknet and EVM bridge contracts collect native fee tokens from users during transfer initiation (`init_transfer` / `initTransfer`) and hold them in the contract balance. Neither contract implements any mechanism to withdraw or distribute these accumulated fees. Every transfer with a non-zero `native_fee` permanently locks the corresponding STRK (Starknet) or ETH (EVM) in the bridge contract, making them irrecoverable.

---

### Finding Description

**Starknet (`starknet/src/omni_bridge.cairo`, lines 309–314):**

In `init_transfer`, when `native_fee > 0`, the caller's STRK tokens are pulled into the contract:

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
```

The entire public interface of the Starknet contract (`log_metadata`, `deploy_token`, `fin_transfer`, `init_transfer`, `upgrade_token`, `set_pause_flags`, `pause_all`) contains no function to withdraw accumulated STRK fees. The STRK balance grows monotonically with every transfer that includes a native fee and can never be recovered.

**EVM (`evm/src/omni-bridge/contracts/OmniBridge.sol`, lines 386–413):**

In `initTransfer`, the `nativeFee` portion of `msg.value` is retained by the contract:

```solidity
if (tokenAddress == address(0)) {
    extensionValue = msg.value - amount - nativeFee;
} else {
    extensionValue = msg.value - nativeFee;
    ...
}
```

The base `initTransferExtension` reverts if `value != 0`, meaning `extensionValue` must be zero. The `nativeFee` ETH is therefore absorbed by the contract. No `withdrawFee`, `collectFee`, or equivalent function exists in `OmniBridge.sol`. The contract has a bare `receive()` fallback but no ETH withdrawal path.

**Root cause — accounting desynchronisation analogous to H-35:**

In H-35, protocol fee tokens accumulate in the strategy contract mixed with user funds, and the tracked `protocolFees` variable diverges from the actual withdrawable balance, making `collectProtocolFees` revert. Here the analogous failure is symmetric: the bridge contracts accumulate native fee tokens in their balance, but unlike the token-fee path (which is tracked in `TransferMessage.fee` and claimable on NEAR via `claim_fee`), the native fee tokens transferred on-chain to the EVM/Starknet contract are never tracked in any on-chain accounting variable and have no corresponding withdrawal path. The on-chain balance grows; the claimable amount stays zero.

---

### Impact Explanation

Every user who pays a non-zero `native_fee` on Starknet or EVM permanently loses that amount. The STRK/ETH is transferred to the bridge contract and is irrecoverable. Over the lifetime of the bridge, the total locked amount scales linearly with transfer volume. This constitutes a **permanent, irrecoverable lock of user and protocol funds** in the bridge contract.

---

### Likelihood Explanation

The `native_fee` parameter is a first-class field in both `init_transfer` (Starknet) and `initTransfer` (EVM). Relayers routinely require a non-zero native fee to cover gas costs on the destination chain. Any transfer submitted with `native_fee > 0` — which is the normal operating case — triggers the lock. Likelihood is **high**.

---

### Recommendation

Add a privileged `collect_native_fee` / `withdrawNativeFee` function to both the Starknet and EVM bridge contracts that allows the protocol owner or a designated fee recipient to withdraw accumulated native fee tokens. Alternatively, transfer the native fee directly to a fee recipient address at the time of `init_transfer` rather than holding it in the bridge contract, mirroring the pattern used by the NEAR contract's `claim_fee` flow.

---

### Proof of Concept

1. User calls `init_transfer` on Starknet with `native_fee = 1e18` STRK.
2. `IERC20Dispatcher { contract_address: strk_token }.transfer_from(caller, get_contract_address(), 1e18)` executes — 1 STRK enters the contract.
3. The `InitTransfer` event is emitted; the NEAR relayer processes the transfer and claims the fee on NEAR in NEAR tokens (from the user's NEAR storage balance), not from the Starknet contract.
4. The Starknet contract's STRK balance is now 1 STRK higher with no corresponding withdrawal path.
5. Repeat for every transfer: STRK accumulates indefinitely and is permanently locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
