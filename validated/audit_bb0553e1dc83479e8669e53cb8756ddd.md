### Title
`ProofOfInclusion::valid()` Does Not Verify Against a Known Root Hash — Forged DataLayer Inclusion Proofs Accepted - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks the internal self-consistency of the proof chain. Its final assertion `existing_hash == self.root_hash()` is a logical tautology — it is always `true` when the loop completes without returning `false`. The function never compares the computed root against any externally-known, trusted root hash. An attacker who can supply a crafted `ProofOfInclusion` (via the `Streamable` deserialization path exposed through Python/wasm bindings) can forge an inclusion proof for any arbitrary key-value pair and have it accepted as valid.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()     // ← tautology
}
```

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same value as existing_hash after loop
    } else {
        self.node_hash
    }
}
```

After the loop body executes for the last layer:
- `existing_hash` is set to `calculated_hash`
- The guard `calculated_hash != layer.combined_hash` already passed, so `calculated_hash == last.combined_hash`
- Therefore `existing_hash == last.combined_hash`
- `self.root_hash()` returns `last.combined_hash`
- The final check `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash` — always `true`

The same tautology holds for the empty-layers case: `existing_hash = self.node_hash` and `root_hash()` returns `self.node_hash`.

**The function never compares against any externally-supplied, trusted root hash.** There is also no alternative function in the DataLayer crate that accepts a known root and verifies the proof against it (contrast with the consensus `validate_merkle_proof(&proof, item, &root)` in `crates/chia-consensus/src/merkle_tree.rs` which does take an external root).

`ProofOfInclusion` derives `Streamable`, so it is fully deserializable from raw bytes via `from_bytes()` — exposed to Python through the `py-bindings` feature in `wheel/src/api.rs`.

### Impact Explanation

An attacker can construct a `ProofOfInclusion` with:
- `node_hash` = SHA-256 hash of any fabricated leaf (e.g., a fake key-value pair)
- `layers` = any internally-consistent chain (each `combined_hash` = `calculate_internal_hash(prev, side, other)`)

`valid()` returns `true` for this forged proof. Any verifier that calls only `proof.valid()` — without separately asserting `proof.root_hash() == on_chain_committed_root` — will accept the forged proof as genuine inclusion evidence. This lets untrusted input prove invalid DataLayer state, satisfying the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

The `ProofOfInclusion` type is exposed to Python callers via `add_datalayer_submodule` in `wheel/src/api.rs`. Python code receiving a proof over the network can call `proof.valid()` and, misled by the function name, trust the result without the mandatory root-hash cross-check. The DataLayer fuzz target and the Rust/Python integration tests both call `proof.valid()` without checking the root, demonstrating the pattern is easy to replicate incorrectly. The serialization path (`from_bytes`) is fully attacker-controlled.

### Recommendation

1. **Add a root-binding verification method** to `ProofOfInclusion`:
   ```rust
   pub fn verify(&self, expected_root: &Hash) -> bool {
       self.valid_chain() && &self.root_hash() == expected_root
   }
   ```
   where `valid_chain()` contains the current loop logic.

2. **Remove or rename `valid()`** to `is_internally_consistent()` to prevent callers from mistaking it for a full proof verification.

3. **Expose `verify(root)` through the Python bindings** instead of (or in addition to) the bare `valid()` method, so callers cannot accidentally omit the root check.

### Proof of Concept

```rust
use chia_datalayer::{Hash, ProofOfInclusion, ProofOfInclusionLayer, Side};

fn forge_proof(fake_leaf_hash: Hash, fake_sibling: Hash) -> ProofOfInclusion {
    // Compute a single internally-consistent layer
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_leaf_hash, Side::Left, &fake_sibling
    );
    ProofOfInclusion {
        node_hash: fake_leaf_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: fake_sibling,
            combined_hash: combined,   // matches calculated_hash → loop passes
        }],
    }
}

fn main() {
    let fake_leaf = [0xAA; 32];
    let fake_sibling = [0xBB; 32];
    let proof = forge_proof(fake_leaf, fake_sibling);

    // valid() returns true even though this proof has nothing to do
    // with the actual committed DataLayer root
    assert!(proof.valid());   // ← passes; forged proof accepted
    // proof.root_hash() is an attacker-controlled value, not the real root
}
```
