Audit Report

## Title
`addLiquidityWeighted` Reverts with `MaxAmountExceeded` Due to Per-Bin Ceiling Rounding Absorbing Floor-Scaled Share Reduction — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`addLiquidityWeighted` uses a probe-then-pay pattern where the probe measures `need0`/`need1`, then `_scaleWeightsToShares` applies floor division to compute scaled shares, and the pay phase recomputes token amounts using per-bin ceiling division. When `max0 < need0`, the floor-scaled shares `sᵢ` are reduced, but the per-bin `Math.ceilDiv` in the pay phase can round back up to the same scaled total as the probe, causing `amount0Added > max0` and an unconditional `MaxAmountExceeded` revert for a valid user-supplied cap.

## Finding Description

**Probe phase** — `LiquidityLib.addLiquidity` is called with weight shares `wᵢ`. For each non-empty bin:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol:109-110
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
```

The per-bin ceiling values are summed and converted to external units with a second ceiling:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol:141-142
(amount0Added, amount1Added) =
  _deltasScaledToExternal(totalToken0ToAddScaled, totalToken1ToAddScaled, ctx, Math.Rounding.Ceil);
```

The callback reverts with `LiquidityProbe(need0, need1)` where `need0 = ceil(Σᵢ ceil(b₀ᵢ·wᵢ/Tᵢ) / M0)`.

**Scale phase** — `_scaleWeightsToShares` computes the binding scale factor and new shares entirely with floor division:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol:231-240
uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0); // floor
uint256 scaleWad  = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;
...
out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD); // floor
```

When `max0 < need0`, `scaleWad < WAD` and `sᵢ = floor(wᵢ · scaleWad / WAD) < wᵢ`.

**Pay phase** — `LiquidityLib.addLiquidity` is called again with `sᵢ`. The same ceiling division is applied per bin. The callback then enforces:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol:165-167
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
```

**Root cause**: The floor reduction in `sᵢ` can be smaller than one unit of the divisor `Tᵢ`, so `ceil(b₀ᵢ · sᵢ / Tᵢ) = ceil(b₀ᵢ · wᵢ / Tᵢ)` — the ceiling absorbs the entire share reduction. Summed across N bins, `Σᵢ ceil(b₀ᵢ · sᵢ / Tᵢ)` equals `Σᵢ ceil(b₀ᵢ · wᵢ / Tᵢ)`, so `amount0Added = need0 > max0`, triggering the revert.

**Concrete minimal example** (2 bins, `token0ScaleMultiplier M0 = 1`):

| Parameter | Value |
|---|---|
| `b₀₁ = b₀₂` | `1` |
| `T₁ = T₂` | `2` |
| `w₁ = w₂` | `1e18` |
| `max0` | `1e18 − 1` |

- Probe: `ceil(1·1e18/2) = 5e17` per bin → `need0 = 1e18`
- Scale: `scaleWad = floor((1e18−1)·WAD/1e18) = WAD−1`; `s₁ = s₂ = floor(1e18·(WAD−1)/WAD) = 1e18−1`
- Pay: `ceil(1·(1e18−1)/2) = ceil((1e18−1)/2)` — `1e18−1` is odd → rounds up to `5e17`; total = `1e18`
- `amount0Added = 1e18 > max0 = 1e18−1` → **`MaxAmountExceeded` revert**

The user provided a legitimate cap (`max0 = need0 − 1`), the scaling executed correctly, yet the function reverts unconditionally.

## Impact Explanation
`addLiquidityWeighted` is the primary entry point for proportional liquidity provision with slippage caps. When a user's `max0` or `max1` is set to any value strictly below `need0`/`need1` (a normal slippage-tolerance pattern), the function always reverts regardless of how many shares are scaled down, because the per-bin ceiling rounding re-inflates the total to the probe value. This renders the weighted liquidity addition flow unusable for the affected input range — broken core liquidity functionality per the allowed impact gate.

## Likelihood Explanation
Any unprivileged LP calling `addLiquidityWeighted` with a `maxAmountToken0` or `maxAmountToken1` cap set below the probe-measured `need` value triggers the bug. This is the normal usage pattern: users set caps based on acceptable slippage. The condition is reproducible deterministically with standard ERC-20 tokens and any pool where per-bin balances are not exact multiples of total shares. No special privileges, flash loans, or oracle manipulation are required.

## Recommendation
In `_scaleWeightsToShares`, after computing `scaleWad`, apply a one-unit conservative reduction to guarantee the pay-phase ceiling sum stays within `max0`/`max1`. Alternatively, after scaling shares, perform a simulation (or tighter bound computation) of the pay-phase ceiling sum and reduce `scaleWad` iteratively until the computed pay amount is ≤ `max`. A simpler fix is to pass `max0 + N_bins` (where `N_bins` is the number of bins) as the effective cap in the pay-phase callback check, acknowledging the bounded rounding slack, while keeping the user-visible cap for the actual token pull.

## Proof of Concept

```solidity
// Foundry unit test sketch
function test_addLiquidityWeighted_MaxAmountExceededRounding() public {
    // Pool with 2 bins, token0BalanceScaled=1, binTotalShares=2 in each bin
    // token0ScaleMultiplier = 1
    // w = [1e18, 1e18], max0 = 1e18 - 1, max1 = type(uint256).max

    // Probe: need0 = ceil(ceil(1*1e18/2) + ceil(1*1e18/2)) = 1e18
    // Scale: scaleWad = (1e18-1)*WAD/1e18 = WAD-1
    //        s[0] = s[1] = (1e18)*(WAD-1)/WAD = 1e18-1
    // Pay:   ceil(1*(1e18-1)/2) = 5e17 per bin (odd numerator rounds up)
    //        amount0Added = 1e18 > max0 = 1e18-1 → MaxAmountExceeded

    vm.expectRevert(
        abi.encodeWithSelector(
            IMetricOmmPoolLiquidityAdder.MaxAmountExceeded.selector,
            1e18, /* amount0Delta */
            0,    /* amount1Delta */
            1e18 - 1, /* max0 */
            type(uint256).max /* max1 */
        )
    );
    adder.addLiquidityWeighted(
        pool, salt, weightDeltas,
        1e18 - 1,          // maxAmountToken0
        type(uint256).max, // maxAmountToken1
        minBin, minPos, maxBin, maxPos,
        ""
    );
}
```
