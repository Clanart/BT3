### Title
Missing Pause Check on `deploy_token` Allows Token Deployment When Bridge Is Fully Paused — (`solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana `bridge_token_factory` program's `deploy_token` instruction contains no pause guard, and the `ALL_PAUSED` bitmask does not include a deploy-token flag. Even when an admin triggers a full pause, any caller holding a valid pre-signed MPC payload can still deploy new token mints and establish NEAR→Solana token mappings. This is the direct analog of H-09: a critical circuit-breaker operation is silently bypassed for one of the three core bridge actions.

---

### Finding Description

In `solana/programs/bridge_token_factory/src/lib.rs`, the three user-facing instructions are guarded as follows:

- `finalize_transfer` (line 82–85): checks `config.paused & FINALIZE_TRANSFER_PAUSED == 0`
- `init_transfer` (line 125–128): checks `config.paused & INIT_TRANSFER_PAUSED == 0`
- `deploy_token` (lines 66–76): **no pause check whatsoever** [1](#0-0) 

In `constants.rs`, the pause bitmask constants are:

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;       // 0x01
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;   // 0x02
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED; // 0x03
```

There is **no `DEPLOY_TOKEN_PAUSED` constant**, and `ALL_PAUSED` is `0x03`, not `0x07`. [2](#0-1) 

The `pause` instruction sets `config.paused = ALL_PAUSED` (= `0x03`). [3](#0-2) 

This means even a full bridge pause leaves `deploy_token` completely unguarded.

**Contrast with EVM and StarkNet implementations:**

EVM `OmniBridge.sol` defines `PAUSED_DEPLOY_TOKEN = 1 << 2` and guards `deployToken` with `whenNotPaused(PAUSED_DEPLOY_TOKEN)`: [4](#0-3) [5](#0-4) 

StarkNet `omni_bridge.cairo` defines `PAUSE_DEPLOY_TOKEN: u8 = 0x04` and guards `deploy_token` with an explicit assertion: [6](#0-5) [7](#0-6) 

The Solana implementation is the only chain missing this protection.

---

### Impact Explanation

`deploy_token` establishes the authoritative NEAR token ID → Solana SPL mint mapping stored in the program's config. If a bug is discovered in the token deployment logic (e.g., incorrect decimal normalization, salt collision, or a compromised MPC key that signed a malicious `deploy_token` payload), the admin's only recourse is to pause the bridge. Because `deploy_token` ignores the pause flag, any attacker holding a valid pre-signed payload can still:

1. Deploy a token with an incorrect or attacker-influenced mapping while the bridge is paused.
2. Once the bridge is unpaused, subsequent `finalize_transfer` calls for that token ID will mint SPL tokens against the attacker-controlled mint, creating unbacked supply or misdirecting minted tokens.

This breaks bridge collateralization and can result in unauthorized minting of bridged assets — matching the **High** impact category: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

The attack requires a valid MPC-signed `deploy_token` payload. Realistic scenarios where this exists during a pause:

- A payload was signed by MPC but not yet submitted on-chain before the pause was triggered (race condition between signing and pausing).
- A bug is discovered in the deployment flow after MPC has already signed several pending deployment messages.
- A partial MPC key compromise results in a malicious payload being signed before the compromise is detected and the bridge paused.

These are not purely theoretical: the pause mechanism exists precisely to handle such emergency scenarios, and its incompleteness on Solana undermines the entire protection model.

---

### Recommendation

1. Add a `DEPLOY_TOKEN_PAUSED: u8 = 1 << 2` constant to `constants.rs`.
2. Update `ALL_PAUSED` to `INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED`.
3. Add a pause guard at the top of `deploy_token` in `lib.rs`:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
        error::ErrorCode::Paused
    );
    // ...
}
```

This mirrors the existing pattern used by `finalize_transfer` and `init_transfer`, and aligns Solana with the EVM and StarkNet implementations.

---

### Proof of Concept

1. Admin discovers a critical bug in `deploy_token` and calls `pause` → `config.paused = ALL_PAUSED = 0x03`.
2. Attacker holds a valid pre-signed `SignedPayload<DeployTokenPayload>` for token `"exploit.near"`.
3. Attacker calls `deploy_token` with the payload. The function has no pause check, so it proceeds.
4. A new SPL mint is created and the mapping `"exploit.near" → attacker_mint` is written to program state.
5. Admin unpauses the bridge after believing the incident is contained.
6. Attacker (or anyone) calls `finalize_transfer` with a valid MPC-signed payload for `"exploit.near"`, minting tokens against the attacker-controlled mint — creating unbacked supply or stealing funds from the vault. [1](#0-0) [8](#0-7)
