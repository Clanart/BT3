Audit Report

## Title
`addLiquidityWeighted` Lacks Minimum-Shares-Received Guard, Enabling Sandwich Attack on LP Deposits — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`addLiquidityWeighted` uses a probe-then-scale pattern to compute integer shares from a user's token budget. The cursor-bounds check (`_validateBinAndBinPosition`) only prevents the cursor from being *outside* the user-supplied range; it does not prevent an attacker from front-running the transaction and moving the cursor to an extreme position *within* that range. Because the probe's `need0`/`need1` ratio is cursor-position-sensitive, a manipulated cursor causes the scale factor to be computed against a skewed ratio, resulting in the user receiving fewer shares per token spent. No `minSharesReceived` parameter exists to bound this loss.

## Finding Description

`addLiquidityWeighted` executes the following sequence:

1. **Cursor-bounds check** — `_validateBinAndBinPosition` reads the live cursor and reverts if it falls outside `[minimalCurBin, maximalCurBin]`. It does not constrain the cursor to a single point within that range.

2. **Probe call** — `addLiquidity` is called with `KIND_PROBE`; the callback reverts with `LiquidityProbe(need0, need1)`, revealing the token amounts the pool would require at the current cursor position.

3. **Share scaling** — `_scaleWeightsToShares` computes `scaleWad = min(max0/need0, max1/need1)` and multiplies every weight by that factor to produce integer shares.

4. **Paying add** — `_addLiquidity` is called with the scaled shares; the callback enforces `amount0Delta ≤ max0 && amount1Delta ≤ max1` before pulling tokens.

The cursor position directly determines the `need0`/`need1` ratio returned by the probe. In `LiquidityLib.addLiquidity`, for a bin below the cursor only token1 is required; for a bin above the cursor only token0 is required; for the active bin both tokens are required in proportion to `curPosInBin`. An attacker who front-runs and moves the cursor to an extreme bin within the user's allowed range causes the probe to return a skewed ratio. The scale factor is then computed against that skewed ratio, so the user deposits at the wrong price and receives fewer shares than they would at the fair cursor position. The max-amount caps prevent the user from paying *more* than their budget, but they do not guarantee a minimum number of shares received. After the attacker back-runs and restores the cursor, the user's position is worth less than the tokens deposited.

The function signature contains no `minSharesReceived` or equivalent parameter:

```solidity
function addLiquidityWeighted(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata weightDeltas,
    uint256 maxAmountToken0,   // caps spending only
    uint256 maxAmountToken1,   // caps spending only
    int8 minimalCurBin,        // bounds cursor range, not shares
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added)
```

By contrast, every swap entry-point in `MetricOmmSimpleRouter` carries an explicit output bound (`amountOutMinimum` / `amountInMaximum`).

## Impact Explanation

A user who calls `addLiquidityWeighted` with a cursor range wider than a single point can be sandwiched. The attacker shifts the cursor to an extreme bin within the allowed range, skewing the probe's `need0`/`need1` ratio. The scale factor `min(max0/need0, max1/need1)` is computed against the skewed ratio, so the user receives fewer shares than they would at the fair price. After the attacker back-runs and restores the cursor, the user's position is worth less than the tokens deposited. The loss is bounded by the width of the cursor range and the user's max token caps, but it is a direct, unrecoverable loss of deposited principal — qualifying as a direct loss of user funds above the Sherlock Medium threshold.

## Likelihood Explanation

- `addLiquidityWeighted` is the primary convenience entry point for users who do not know the exact share amounts to deposit; it is expected to be called frequently.
- The attack requires only a standard front-run/back-run sandwich, which is trivially executable by any MEV searcher on any EVM chain.
- Users who specify wide cursor bounds (common for general-purpose deposits) are fully exposed; only users who set very tight bounds (e.g., a single bin and position) are effectively protected.
- The NatDoc comment "use slot0 cursor bounds to revert when state has been manipulated" implies the cursor bounds are the intended slippage guard, but they are insufficient for any range wider than a single point.

## Recommendation

Add a `minTotalSharesReceived` parameter to `addLiquidityWeighted`. After the paying add completes, sum the scaled shares across all bins and revert if the total falls below the user-supplied minimum:

```solidity
function addLiquidityWeighted(
    ...
    uint256 minTotalSharesReceived,   // NEW: revert if shares < this
    ...
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    ...
    LiquidityDelta memory scaled = _scaleWeightsToShares(...);
    uint256 totalShares;
    for (uint256 i; i < scaled.shares.length; i++) totalShares += scaled.shares[i];
    if (totalShares < minTotalSharesReceived) revert InsufficientSharesReceived(totalShares, minTotalSharesReceived);
    return _addLiquidity(...);
}
```

Alternatively, tighten the cursor bounds check to a single point (requiring `minimalCurBin == maximalCurBin` and `minimalPosition == maximalPosition`) so the probe result cannot be skewed by cursor movement within the range.

## Proof of Concept

**Setup**: Pool with token0/token1, fair cursor at bin 0. User wants to deposit up to 1000 token0 and 1000 token1 with `minimalCurBin = -5, maximalCurBin = 5`. Weight vector covers bins 3, 4, 5.

1. **Attacker front-runs**: swaps a large amount of token0 → token1, moving the cursor to bin 5 (within the user's allowed range). At bin 5, bins 0–4 are below the cursor and require only token1; bin 5 (active) requires a mix.

2. **User's `addLiquidityWeighted` executes**:
   - `_validateBinAndBinPosition`: cursor is at bin 5, within `[-5, 5]` → passes.
   - Probe returns `need0 = 100, need1 = 900` (skewed toward token1 at manipulated cursor).
   - Scale factor = `min(1000/100, 1000/900) = min(10, 1.11) = 1.11`.
   - User receives `1.11 × weight` shares, depositing ~111 token0 + ~1000 token1.
   - At fair cursor (bin 0), the same budget would have yielded `2 × weight` shares.

3. **Attacker back-runs**: swaps token1 → token0, restoring cursor to bin 0. Attacker profits from the round-trip; the token1 the user deposited into bins 0–4 is consumed by the back-run at the manipulated price.

4. **Result**: The user's position (1.11× shares) is worth less than the tokens deposited (111 token0 + 1000 token1 at fair price). No revert occurs because there is no minimum-shares-received check.

  function _scaleWeightsToShares(LiquidityDelta calldata w, uint256 max0, uint256 max1, uint256 need0, uint256 need1)
    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
    uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;

    uint256 n = w.binIdxs.length;
    out.binIdxs = new int256[](n);
    out.shares = new uint256[](n);
    for (uint256 i; i < n; i++) {
      out.binIdxs[i] = w.binIdxs[i];
      out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
      if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
    }
  }
