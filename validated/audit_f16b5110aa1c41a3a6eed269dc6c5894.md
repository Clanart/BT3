Looking at the reNFT bug pattern: an asset is transferred (action) **before** the rental order's existence is checked (validation), and the ERC721 `onReceived` callback during the transfer lets the attacker create the order, making the subsequent check pass.

I searched for nearcore analogs across: state-sync part storage, chunk reconstruction, witness assembly, the `main_state_transition_cache`, `process_partial_encoded_chunk` ordering, orphan witness handling, and `PartialEncodedStateWitnessTracker`.

**Closest candidate examined — `validate_chunk_state_witness_impl`:**

The `main_state_transition_cache` is written at line 653 **before** the final validation checks at lines 751–777 (`validate_chunk_with_chunk_extra_and_receipts_root` and `validate_chunk_with_encoded_merkle_root`). [1](#0-0) [2](#0-1) 

If the final validation fails, the cache entry (keyed by `(witness_chunk_shard_uid, block_hash)`) persists with potentially incorrect `chunk_extra` / `outgoing_receipts`. A subsequent legitimate witness for the same key would use the poisoned cached result, potentially causing it to fail validation.

**Why this is rejected:**

The `ChunkStateWitness` is assembled exclusively from `PartialEncodedStateWitness` parts, each of which is signature-validated against the chunk producer's key in `validate_partial_encoded_state_witness` before being stored. [3](#0-2) [4](#0-3) 

There is no direct network path for an unsigned `ChunkStateWitness`; the assembled witness reaches the validation actor only through `PartialEncodedStateWitnessTracker::process_update` after all parts pass signature checks. Triggering the cache-before-validation ordering issue therefore requires a **malicious chunk producer** — an explicitly privileged role excluded by the analog standard.

**Other candidates examined and rejected:**

- `process_partial_encoded_chunk`: parts are merged into `encoded_chunks` cache before full signature validation, but the entry is evicted on failure and the window is intra-call with no re-entrant callback mechanism. Impact is ordinary resource/DoS, not a bypassed commitment invariant. [5](#0-4) [6](#0-5) 

- `set_state_part` / `set_shard_state`: validation precedes storage in both paths. [7](#0-6) 

- `processed_witnesses` cache written before chunk validation actor receives the witness: intentional deduplication, no bypassed check.
