### Title
`execute_deploy` Syscall Unconditionally Reports Success (`failure_flag=0`) When Constructor Reverts, Enabling Permanent Fund Freezing - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall in `syscall_impls.cairo` always writes `failure_flag=0` in the response header regardless of whether the constructor execution succeeded or reverted. When a constructor reverts, the state changes (including the class hash assignment for the new address) are rolled back via the revert log, leaving the deployed address with `class_hash=0`. However, the calling contract receives a success response and proceeds as if the deploy succeeded. Any tokens subsequently sent to that address are permanently frozen, as no contract exists there to handle transfers.

---

### Finding Description

In `execute_deploy` at line 539 of `syscall_impls.cairo`:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The response header is unconditionally written with `failure_flag=0`. The TODO comment explicitly acknowledges that failure reporting is not yet implemented.

The asymmetry is structural:

1. `execute_deploy` calls `deploy_contract` using the **caller's** `revert_log` as an implicit argument (inherited from `execute_deploy`'s own implicit arguments, line 452–531).
2. If the constructor reverts, `handle_revert` processes the revert log and rolls back the class hash assignment for the new contract address back to `0`.
3. The contract at the deployed address now has `class_hash=0` — no executable code.
4. Despite this, the OS writes `failure_flag=0` to the syscall response.
5. The calling contract (Sierra/Cairo 1.0) reads `failure_flag=0` and interprets the deploy as successful.
6. The calling contract proceeds to send tokens to the "deployed" address.
7. Those tokens are recorded in the ERC20 contract's storage under that address, but since `class_hash=0`, no contract can be called to transfer them out.

This is the direct analog to the external report: tokens can flow **into** the address (because the caller believes the deploy succeeded) but **cannot flow back out** (because the address has no class hash), due to the missing failure check in the OS response.

Contrast with `execute_deploy_account_transaction` in `transaction_impls.cairo` (lines 642–646), which correctly initializes a **fresh** `revert_log` for the constructor and handles the revert at the transaction level — but the syscall path has no equivalent failure propagation to the caller.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Tokens sent to a contract address where the constructor reverted are unrecoverable under normal conditions:
- The address has `class_hash=0` and no storage.
- No entry point exists to call `transfer` or any other function.
- The ERC20 balance for that address is permanently locked in the fee token contract's storage.

Recovery would require re-deploying a contract at the exact same address (same salt, class hash, constructor calldata) with a constructor that does not revert — a scenario that may be impossible if the original failure was due to the class hash itself or constructor arguments.

---

### Likelihood Explanation

**Medium.** The trigger requires:
1. A contract that calls the `deploy` syscall (available to any unprivileged contract caller).
2. A constructor that can revert (e.g., out-of-gas, failed assertion, invalid initial state).
3. The calling contract sending tokens to the deployed address after the syscall returns.

This is a standard DeFi pattern: deploy a new pool/vault contract and immediately fund it. The calling contract checks `failure_flag` to decide whether to proceed — but since the OS always returns `0`, it always proceeds. No privileged access is required; any user can trigger this by deploying a contract that follows this pattern.

---

### Recommendation

In `execute_deploy`, track whether `deploy_contract` reverted (e.g., by checking the revert log state before and after the call, or by having `deploy_contract` return a revert flag). When the constructor reverts, write `failure_flag=1` and include the failure reason in the response
