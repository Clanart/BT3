### Title
Unchecked Class Hash in `execute_replace_class` Allows Permanent Contract Bricking and OS Proof Failure — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` accepts an arbitrary `class_hash` from the caller's syscall request and writes it directly into the contract state without verifying that the hash corresponds to a previously declared contract class. A developer-acknowledged TODO comment at the exact location confirms this check is missing. An unprivileged contract deployer can exploit this to permanently brick any contract they control (freezing its funds) and, if the bricked contract is subsequently called within any block, cause the StarkNet OS to fail hard during proof generation, halting the network.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the `class_hash` field is read directly from the user-supplied `ReplaceClassRequest` struct and written into `contract_state_changes` without any check that the hash exists in `contract_class_changes` (the declared-class registry):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing validation. After this update, the contract's `class_hash` in `contract_state_changes` is set to an arbitrary, undeclared value.

When any subsequent call targets this contract — whether in the same block or a future block — `execute_entry_point` (`execute_entry_point.cairo`, lines 154–166) performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // undeclared hash → returns 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // key = 0, not present → hard Cairo assertion failure
);
```

`dict_read` on `contract_class_changes` returns `0` for an undeclared class hash (the Cairo dict default). `find_element` then asserts the element exists in the `compiled_class_facts_bundle` array; since no compiled class with hash `0` is present for an honest prover, this is a hard Cairo VM assertion failure — not a graceful revert — causing the entire OS execution to abort and the block proof to be ungenerable.

---

### Impact Explanation

**Critical — Permanent freezing of funds:** Any contract that calls `replace_class` with an undeclared hash is permanently bricked. All assets (tokens, ETH, etc.) held by that contract become irrecoverable, because no entry point can ever be dispatched to it again.

**High — Network halt:** If the sequencer includes any call to the bricked contract in a block (which it will, marking it as reverted), the OS program fails hard at `find_element` during proof generation for that block. No proof can be produced, and the network cannot confirm new transactions until the block is dropped or the issue is resolved at the protocol level.

---

### Likelihood Explanation

The `replace_class` syscall is available to any contract without access control. An attacker needs only to:
1. Deploy a contract (permissionless on StarkNet).
2. Issue a transaction that calls `replace_class` with any felt value that is not a declared class hash.

This is a single, cheap, permissionless transaction. The attacker does not need any privileged role, leaked key, or operator cooperation. The impact is immediate and irreversible.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., it has been declared). Concretely, perform a `dict_read` on `contract_class_changes` with the requested `class_hash` and assert the returned `compiled_class_hash` is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
// Reject if the class has not been declared.
assert_not_zero(compiled_class_hash);
```

This mirrors the validation that `execute_entry_point` implicitly relies on and closes the gap acknowledged by the existing TODO comment.

---

### Proof of Concept

**Step 1 — Attacker deploys a malicious contract** containing:
```cairo
// Inside the contract's __execute__ entry point:
replace_class(class_hash=0xdeadbeef);  // arbitrary undeclared hash
```

**Step 2 — Attacker submits an invoke transaction** calling `__execute__`. The sequencer executes `execute_replace_class`:
- `request.class_hash = 0xdeadbeef` (undeclared).
- No validation is performed (line 898 TODO).
- `contract_state_changes[attacker_contract].class_hash = 0xdeadbeef` is written.
- Transaction succeeds; state is committed.

**Step 3 — Any subsequent call to `attacker_contract`** (e.g., a second transaction, or an inner call from another contract):
- Sequencer tries to execute; fails to find compiled class; marks transaction as reverted; includes it in the block.

**Step 4 — OS processes the block:**
- `execute_entry_point` is called for the reverted call.
- `dict_read(contract_class_changes, key=0xdeadbeef)` → returns `0` (undeclared).
- `find_element(..., key=0)` → Cairo assertion failure: element not found.
- OS program aborts. Block proof cannot be generated. **Network halt.**

**Funds frozen:** Any ERC-20 tokens or ETH held by `attacker_contract` are permanently inaccessible. [1](#0-0) [2](#0-1)
