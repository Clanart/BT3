### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Guard, Allowing Silent Tree Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` omits the duplicate-key and duplicate-hash existence checks that `MerkleBlob::insert` enforces. When the tree already contains two or more leaves, any items beyond the first two in a batch are written directly into the blob without verifying that their `KeyId` or `Hash` is not already present. Supplying a `KeyId` that already exists in the tree silently inserts a second leaf node with the same key, corrupting the Merkle tree structure, producing an incorrect root hash, and invalidating proofs of inclusion — all without returning an error.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicate state at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` calls `self.insert(…)` (with those guards) only for the first two items when `leaf_count() <= 1` (lines 578–585). For every subsequent item — the main loop at lines 587–603 — it calls `self.insert_entry_to_blob(new_leaf_index, &new_block)` directly, bypassing both checks entirely:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no duplicate check
    indexes.push(new_leaf_index);
}
```

This means:
- A `KeyId` already present in the tree can be inserted again as a second leaf node.
- Duplicate `KeyId` values within the batch itself are also silently accepted.
- The `block_status_cache` ends up with two entries for the same key, making the in-memory index inconsistent with the blob.
- The subtree built from the batch is then grafted into the existing tree via `insert_subtree_at_key`, so the corrupted leaves become part of the committed tree structure.
- `calculate_lazy_hashes()` then computes a root hash over the corrupted tree, producing a root that does not correspond to any valid key-set.

The Python binding `py_batch_insert` (lines 1503–1519) exposes this path directly to callers with no additional guard.

---

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

1. **Root hash corruption**: After a duplicate-key batch insert, `calculate_lazy_hashes()` produces a root hash computed over a tree that contains two leaf nodes for the same `KeyId`. This root hash is committed as the authoritative state root.
2. **Proof-of-inclusion forgery / invalidity**: `get_proof_of_inclusion(key)` will resolve to whichever of the two duplicate leaves the cache happens to return. The other duplicate leaf is unreachable by key lookup but still contributes to the root hash, so any proof generated is either for the wrong node or structurally invalid against the committed root.
3. **Proof-of-exclusion bypass**: A key that should be absent can be made to appear present (or vice versa) by inserting a duplicate that shifts the tree structure, causing exclusion proofs to verify incorrectly.
4. **Persistent state corruption**: The corrupted blob is serializable and can be persisted to disk (`to_path`), propagated to peers, and used as the basis for further delta operations, spreading the inconsistency.

---

### Likelihood Explanation

`batch_insert` is a public API exposed through the Python binding `py_batch_insert`. Any caller that constructs a batch containing a `KeyId` already present in the tree — whether by mistake or by deliberate crafting — triggers the corruption. The DataLayer layer above `MerkleBlob` does not appear to enforce uniqueness before calling `batch_insert`. No privileged role or leaked key is required; only the ability to call `batch_insert` with attacker-controlled input.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the main loop of `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing blob-write logic …
}
```

Alternatively, collect all incoming keys into a temporary `HashSet` before the loop and reject the entire batch if any key collides with an existing key or with another key in the same batch.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 3 leaves so leaf_count > 1 (bypasses the guarded path)
blob.insert(KeyId(1), ValueId(1), h(1))
blob.insert(KeyId(2), ValueId(2), h(2))
blob.insert(KeyId(3), ValueId(3), h(3))

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with KeyId(1) already present — no error raised
blob.batch_insert(
    [(KeyId(1), ValueId(99)), (KeyId(4), ValueId(4)), (KeyId(5), ValueId(5))],
    [h(99), h(4), h(5)],
)
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# root_after is computed over a tree with two leaves for KeyId(1)
# proof_of_inclusion for KeyId(1) resolves to one of the two duplicates
proof = blob.get_proof_of_inclusion(KeyId(1))
print("proof valid:", proof.valid())   # may be True against the wrong root
print("root changed:", root_before != root_after)
```

The batch succeeds without error. The resulting root hash is computed over a structurally invalid tree containing two leaf nodes for `KeyId(1)`, and any proof of inclusion generated against it is unreliable. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
