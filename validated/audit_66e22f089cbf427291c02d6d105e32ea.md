### Title
Integer Truncation in `validate_tx_l2_gas_price_within_threshold` Collapses the L2 Gas-Price Admission Floor to Zero — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`validate_tx_l2_gas_price_within_threshold` computes the minimum acceptable L2 gas price as `(min_gas_price_percentage / 100) × previous_block_l2_gas_price` using integer-truncating `.to_integer()`. When the product of the percentage and the previous-block price is less than 100, the computed threshold truncates to **zero**, and the comparison `tx_l2_gas_price.0 < 0` is vacuously false for every `u128` value. The gateway then admits any `AllResources` transaction regardless of its declared L2 gas price, including `max_price_per_unit = 0`.

---

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `validate_tx_l2_gas_price_within_threshold` computes:

```rust
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();          // ← floor-truncation
if tx_l2_gas_price.0 < threshold {
    return Err(...);
}
```

`Ratio::to_integer()` returns the integer part of the rational number, discarding the remainder. When

```
min_gas_price_percentage × previous_block_l2_gas_price < 100
```

the rational value is less than 1, so `.to_integer()` returns **0**. Because `tx_l2_gas_price.0` is a `u128`, the condition `tx_l2_gas_price.0 < 0` is structurally impossible, and the error branch is never reached. Every `AllResources` transaction passes the check unconditionally.

Concrete trigger (non-default but explicitly documented configuration):

| `min_gas_price_percentage` | `previous_block_l2_gas_price` | Intended threshold | Computed threshold |
|---|---|---|---|
| 50 | 1 fri | 0.5 → reject `price = 0` | **0** → accept `price = 0` |
| 10 | 9 fri | 0.9 → reject `price = 0` | **0** → accept `price = 0` |
| 80 | 1 fri | 0.8 → reject `price = 0` | **0** → accept `price = 0` |

The config field is typed `u8` with the documented example "E.g., 80 to require 80% of threshold", confirming that sub-100 values are an intended operational mode. [1](#0-0) 

The `StatefulTransactionValidatorConfig` default sets `min_gas_price_percentage = 100`, which avoids truncation for integer gas prices. However, the field is explicitly configurable and values below 100 are documented as valid. [2](#0-1) 

The function is called from `validate_resource_bounds`, which is guarded by `self.config.validate_resource_bounds` (default `true`), and is invoked during the stateful pre-validation path for every incoming transaction. [3](#0-2) 

---

### Impact Explanation

When the truncation fires, the gateway's L2 gas-price floor is silently disabled. An attacker can submit `AllResources` V3 transactions with `l2_gas.max_price_per_unit = 0`. These transactions:

1. Pass the stateful gas-price check (threshold = 0).
2. Enter the mempool's `FeeTransactionQueue` with `max_l2_gas_price = 0`, landing in the `pending_queue` (below the gas-price threshold) rather than the `priority_queue`.
3. Are never selected for sequencing while the gas price is above zero, but they occupy mempool slots and can be used to exhaust mempool capacity or to front-run a future gas-price drop to zero.

The stateless validator has a separate `min_gas_price` floor (default 8 000 000 000 fri), which provides a partial mitigation when both checks are active. However:
- The stateless check is independently configurable and can be set to 0.
- The two checks serve different purposes (absolute floor vs. market-relative floor); operators who lower `min_gas_price_percentage` to, e.g., 50 expect the stateful check to enforce 50 % of the current market price, not 0. [4](#0-3) 

---

### Likelihood Explanation

The bug is latent under the default configuration (`min_gas_price_percentage = 100`) because `100 × price / 100 = price` with no fractional part. It activates as soon as an operator sets `min_gas_price_percentage` to any value below 100 **and** the previous block's L2 gas price is below `100 / min_gas_price_percentage` fri. During network bootstrap, fee-market adjustment, or testnet operation, L2 gas prices in the single-digit fri range are realistic. No privileged access is required; any user can submit a transaction with `max_price_per_unit = 0`.

---

### Recommendation

Replace `.to_integer()` (floor) with ceiling division to preserve the intended "at least X% of the previous price" semantics:

```rust
// Instead of:
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();

// Use ceiling:
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .ceil()
    .to_integer();
```

`num_rational::Ratio::ceil()` returns the smallest integer ≥ the rational value, so a product of 0.5 becomes 1 rather than 0, and the check correctly rejects `tx_l2_gas_price = 0`.

Alternatively, enforce a minimum threshold of 1 after the computation:

```rust
let threshold = std::cmp::max(
    1_u128,
    (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer(),
);
```

### Proof of Concept

The existing test suite already demonstrates the truncation-to-zero path but marks it as an intentional "disabled" case:

```rust
#[case::gas_price_check_disabled_when_percentage_zero_pass(
    100_u128.try_into().unwrap(),
    0,          // min_gas_price_percentage = 0
    0_u128.into(),
    Ok(()),
)]
```

The following case is **not** tested but reproduces the bug with a non-zero percentage:

```rust
// min_gas_price_percentage = 50, previous price = 1 fri
// Intended threshold = 0.5 → should reject price = 0
// Actual threshold   = 0   → accepts  price = 0  ← BUG
#[case::truncation_bypasses_check(
    1_u128.try_into().unwrap(),   // previous_block_l2_gas_price = 1
    50,                           // min_gas_price_percentage = 50
    0_u128.into(),                // tx_l2_gas_price = 0
    Err(StarknetError { ... }),   // expected: rejection
    // actual result: Ok(())      ← gateway admits the transaction
)]
``` [5](#0-4)
