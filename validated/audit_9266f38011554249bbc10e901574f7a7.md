### Title
Missing Class Existence Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler updates a contract's class hash in the global state without verifying that the new class hash was ever declared on-chain. An unprivileged contract can replace its own class with an arbitrary, undeclared hash, permanently rendering itself unexecutable and freezing any funds it holds.

---

### Finding Description

The `execute_replace_class` function at line 878 of `syscall_impls.cairo` processes the `replace_class` syscall. After deducting gas, it reads `class_hash` from the request and immediately writes it into `contract_state_changes` via `dict_update`: [1](#0-0) 

The critical missing step is acknowledged by the developers themselves in a TODO comment at line 898: [2](#0-1) 

There is no check that `class_hash` exists in `contract_class_changes` — the dictionary that tracks classes declared via `execute_declare_transaction`. Declared classes are registered there via `dict_update` with `prev_value=0`: [3](#0-2) 

The `execute_replace_class` handler never cross-references `contract_class_changes` before committing the new class hash to `contract_state_changes`. This is the direct analog of the external report's vulnerability: just as `backend_url` was written before voting resolved, here `class_hash` is written before the prerequisite condition (class declaration) is verified.

The corrupted state entry is then committed permanently to the Patricia Merkle Tree by `compute_contract_state_commitment`, which hashes the new (invalid) `class_hash` into the contract leaf: [4](#0-3) 

And the global state root is updated to reflect this: [5](#0-4) 

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value and committed to the global state root, every future call to that contract will fail at entry-point dispatch — no valid Sierra/CASM code exists for the class. Any funds (tokens, ETH) held in the contract's storage are permanently frozen with no recovery path. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The attack requires no privileged role. Any contract deployer can:
1. Deploy a contract that holds funds (e.g., a vault or escrow).
2. Call the `replace_class` syscall from within that contract, supplying an arbitrary, never-declared class hash.
3. The OS processes the block, commits the invalid state, and the contract is permanently bricked.

The syscall is reachable from any executing contract. The attacker controls the `class_hash` field in `ReplaceClassRequest` directly.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, verify that `class_hash` is present in `contract_class_changes` with a non-zero compiled class hash. Specifically, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the result is non-zero. This is already identified in the codebase's own TODO at line 898.

---

### Proof of Concept

1. Attacker deploys contract `C` (e.g., a token vault holding user funds). `C`'s class hash is legitimately declared.
2. Attacker triggers a transaction that causes `C` to call `replace_class(class_hash=0xdeadbeef)`, where `0xdeadbeef` has never been declared.
3. `execute_replace_class` deducts gas, reads `class_hash = 0xdeadbeef` from the request, and calls `dict_update` on `contract_state_changes` — setting `C`'s class hash to `0xdeadbeef` with no validation.
4. The OS block execution completes. `compute_contract_state_commitment` hashes `0xdeadbeef` into `C`'s Patricia leaf. `calculate_global_state_root` incorporates this into the final state root. The proof is generated and accepted.
5. All subsequent transactions targeting `C` fail at entry-point dispatch — no class `0xdeadbeef` exists. Funds in `C` are permanently frozen.
