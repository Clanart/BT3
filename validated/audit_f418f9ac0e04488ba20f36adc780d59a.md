Audit Report

## Title
Router's `pay()` WETH path silently consumes stranded ETH to settle any caller's swap, bypassing payer's WETH pull entirely — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` checks the router's native ETH balance before pulling WETH from the designated payer. When `address(this).balance >= value`, it wraps the router's own ETH and forwards it to the pool without touching the payer's WETH allowance or balance at all. Because all four swap entry-points are `external payable` with no guard requiring `msg.value == 0` when `tokenIn != WETH`, ETH sent with a non-WETH swap is silently stranded on the router. Any subsequent WETH swap by any caller drains that stranded ETH as payment, giving the attacker free swap output while the original depositor loses their ETH.

## Finding Description

**Root cause — `pay()` WETH branch:**

In `PeripheryPayments.pay()`, lines 73–84, when `token == WETH` and `payer != address(this)`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } else if (nativeBalance > 0) {
        ...
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

When `nativeBalance >= value`, the payer is completely bypassed. The router wraps its own ETH and pays the pool. The payer needs zero WETH balance and zero allowance.

**ETH stranding mechanism:**

All four swap entry-points are `external payable` with no `msg.value == 0` guard when `tokenIn != WETH`:

- `exactInputSingle` — line 67, `external payable` [2](#0-1) 
- `exactInput` — line 92, `external payable` [3](#0-2) 
- `exactOutputSingle` — line 130, `external payable` [4](#0-3) 
- `exactOutput` — line 154, `external payable` [5](#0-4) 

The `receive()` guard only blocks direct ETH sends from non-WETH addresses; it does **not** prevent ETH arriving via `msg.value` in a payable call: [6](#0-5) 

`refundETH()` sends only to `msg.sender`, so a different caller cannot recover stranded ETH — but they can consume it via a WETH swap: [7](#0-6) 

**Exploit call path:**

`exactInputSingle{value: X}(tokenIn=tokenA)` → pool calls `metricOmmSwapCallback` → `_justPayCallback` → `pay(tokenA, msg.sender, pool, amount)` — the non-WETH branch uses `safeTransferFrom(payer, ...)` for tokenA, swap succeeds, X ETH is stranded. [8](#0-7) 

Then attacker calls `exactInputSingle(tokenIn=WETH, amountIn=X)` → `_justPayCallback` → `pay(WETH, attacker, pool, X)` → `nativeBalance >= X` → router wraps its own ETH, attacker never charged.

## Impact Explanation

Direct loss of user principal. Any ETH stranded on the router (via accidental `msg.value` on a non-WETH swap, or a multicall missing a trailing `refundETH()`) is fully consumable by any unprivileged attacker who calls a WETH swap of equal or smaller size. The attacker receives real swap output tokens while paying nothing. Loss is 100% of the stranded ETH. This meets the Critical/High threshold for any non-trivial ETH amount under Sherlock contest rules.

## Likelihood Explanation

Medium–High. ETH stranding occurs through ordinary user error: sending `msg.value` with a non-WETH swap, or composing a multicall with a WETH swap and omitting `refundETH()`. Both patterns are realistic for wallets and aggregators. An attacker watching the mempool can front-run the victim's transaction or call a WETH swap in the same block. No privileged role is required; any EOA can trigger the exploit.

## Recommendation

1. **Enforce `msg.value == 0` when `tokenIn != WETH`** in each swap entry-point, reverting if ETH is sent with a non-WETH swap.
2. **Restrict the ETH-wrapping path to `payer == address(this)`** only. When the payer is an external EOA and `token == WETH`, always use `safeTransferFrom(payer, ...)` regardless of the router's ETH balance.
3. **Alternatively**, track per-caller ETH deposits in transient storage and only allow each caller's own deposited ETH to be wrapped on their behalf.

## Proof of Concept

```solidity
// Step 1 – strand ETH on the router (victim, accidental)
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    pool:            address(tokenA_tokenB_pool),
    tokenIn:         address(tokenA),   // NOT WETH — non-WETH branch uses safeTransferFrom
    tokenOut:        address(tokenB),
    zeroForOne:      true,
    amountIn:        1_000e18,
    amountOutMinimum: 0,
    recipient:       victim,
    deadline:        block.timestamp + 1,
    priceLimitX64:   0,
    extensionData:   ""
}));
// 1 ETH is now stranded on the router; victim's tokenA was spent normally.

// Step 2 – attacker drains stranded ETH via a WETH swap (zero WETH needed)
router.exactInputSingle(ExactInputSingleParams({
    pool:            address(weth_tokenC_pool),
    tokenIn:         address(WETH),
    tokenOut:        address(tokenC),
    zeroForOne:      true,
    amountIn:        1 ether,
    amountOutMinimum: 0,
    recipient:       attacker,
    deadline:        block.timestamp + 1,
    priceLimitX64:   0,
    extensionData:   ""
}));
// pay(WETH, attacker, pool, 1e18):
//   nativeBalance = 1 ether >= 1e18 → deposit + safeTransfer, attacker never charged.
// Attacker receives tokenC; victim's 1 ETH is gone.
```
