### Title
Gateway Gas Price Threshold Bypassed for Legacy `L1Gas` Transactions, Allowing Admission of Economically Invalid Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`validate_tx_l2_gas_price_within_threshold` in the stateful gateway validator enforces a configurable gas-price floor (production default: 100% of the previous block's L2 gas price) only for `AllResources` (V3 ≥ 0.13.3) transactions. The `L1Gas` (legacy V3 pre-0.13.3) branch is an explicit no-op. During periods of rapid L2 gas-price movement — the sequencer analog of the DeFi "market turbulence" scenario — the gateway admits `L1Gas` transactions whose `max_price_per_unit` is far below the current network price, while simultaneously rejecting economically equivalent `AllResources` transactions. The admitted transactions then fail inside the blockifier's pre-validation (`MaxGasPriceTooLow`), wasting mempool capacity and producing an authoritative-looking wrong admission decision from the gateway RPC layer.

---

### Finding Description

In `validate_resource_bounds` the gateway reads the previous block's STRK L2 gas price and delegates to `validate_tx_l2_gas_price_within_threshold`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L223-243
async fn validate_resource_bounds(...) {
    if self.config.validate_resource_bounds {
        let previous_block_l2_gas_price = self
            .gateway_fixed_block_state_reader
            .get_block_info().await?
            .gas_prices.strk_gas_prices.l2_gas_price;
        self.validate_tx_l2_gas_price_within_threshold(
            executable_tx.resource_bounds(),
            previous_block_l2_gas_price,
        )?;
    }
}
```

Inside that function the `L1Gas` arm is a hard-coded pass-through:

```rust
// L358-390
// TODO(Arni): Consider running this validation for all gas prices.
fn validate_tx_l2_gas_price_within_threshold(...) {
    match tx_resource_bounds {
        ValidResourceBounds::AllResources(tx_resource_bounds) => {
            // threshold = min_gas_price_percentage% * previous_block_l2_gas_price
            if tx_l2_gas_price.0 < threshold { return Err(...); }
        }
        ValidResourceBounds::L1Gas(_) => {
            // No validation required for legacy transactions.   ← hard-coded bypass
        }
    }
    Ok(())
}
```

The production deployment config hard-codes `min_gas_price_percentage = 100`:

```json
// crates/apollo_deployments/resources/app_configs/gateway_config.json L19
"gateway_config.static_config.stateful_tx_validator_config.min_gas_price_percentage": 100,
```

This means the threshold is exactly the previous block's L2 gas price — zero tolerance for deviation. The same value appears in `StatefulTransactionValidatorConfig::default()` and in `config_schema.json`.

The blockifier **does** enforce the L1 gas price for `L1Gas` transactions during execution (`check_resource_bounds` in `account_transaction.rs` L441: `resource_bounds.max_price_per_unit < actual_gas_price.get()` → `MaxGasPriceTooLow`). The gateway therefore creates a split-brain: it tells the caller "transaction accepted" while the blockifier will unconditionally reject it.

**Concrete attack / trigger path:**

1. L2 gas price rises from block N to block N+1 (normal EIP-1559 adjustment, e.g. from 1 000 fri to 1 003 fri).
2. User A submits an `AllResources` transaction with `l2_gas.max_price_per_unit = 1 001` — gateway rejects it (`GAS_PRICE_TOO_LOW`, threshold = 1 003).
3. User B submits an `L1Gas` transaction with `l1_gas.max_price_per_unit = 1` (far below the current L1 gas price) — gateway **accepts** it, mempool admits it.
4. Blockifier pre-validation rejects it with `MaxGasPriceTooLow { resource: L1Gas }`.

No privileged access is required. Any unprivileged user can craft an `L1Gas` transaction.

---

### Impact Explanation

**Matching scope:** *High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

- The gateway RPC returns a success response (transaction hash) for an `L1Gas` transaction that will never be sequenced. This is an authoritative-looking wrong value from the gateway admission layer.
- The mempool holds stale, unexecutable `L1Gas` transactions, consuming capacity that could be used by valid transactions.
- The asymmetry means `AllResources` users are blocked during gas-price spikes while `L1Gas` users bypass the guard entirely, creating an inconsistent and exploitable admission policy.

---

### Likelihood Explanation

- The `L1Gas` bypass is unconditional and requires no special conditions beyond submitting a legacy-format transaction.
- Gas price volatility is a normal operating condition (the EIP-1559 fee market is always active).
- The production config sets `min_gas_price_percentage = 100`, maximising the window during which `AllResources` transactions are rejected while `L1Gas` transactions are not.
- The TODO comment (`// TODO(Arni): Consider running this validation for all gas prices`) confirms the gap is known but unresolved.

---

### Recommendation

Apply the gas-price threshold check to `L1Gas` transactions as well, comparing `l1_gas.max_price_per_unit` against the previous block's L1 gas price (analogous to how `AllResources` checks `l2_gas.max_price_per_unit` against the L2 gas price):

```rust
ValidResourceBounds::L1Gas(l1_bounds) => {
    let l1_gas_price = previous_block_l1_gas_price.get().0;
    let threshold = (Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128)
        * l1_gas_price).to_integer();
    if l1_bounds.max_price_per_unit.0 < threshold {
        return Err(StarknetError { ... });
    }
}
```

`validate_resource_bounds` should also fetch and pass `previous_block_l1_gas_price` alongside the existing `previous_block_l2_gas_price`.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
