### Title
MetadataPayload Signature Lacks Chain-ID Binding, Enabling Cross-Chain Replay to Unauthorized `deployToken` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-types/src/lib.rs`)

---

### Summary

The `MetadataPayload` borsh encoding signed by the NEAR MPC contract contains no chain-id or contract-address discriminator. Because every EVM (and Starknet) deployment of OmniBridge shares the same `nearBridgeDerivedAddress` (derived from the same MPC key and path), a single valid metadata signature obtained from a public NEAR log can be replayed verbatim on any other supported chain to call `deployToken` and register an unauthorized canonical token mapping.

---

### Finding Description

**NEAR signing side** — `near/omni-bridge/src/lib.rs` `log_metadata_callback` constructs a `MetadataPayload` and signs `keccak256(borsh(payload))`:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload)...
);
``` [1](#0-0) 

The `MetadataPayload` struct itself contains only `prefix`, `token`, `name`, `symbol`, `decimals`: [2](#0-1) 

No `chain_id`, no destination contract address, no nonce.

**EVM verification side** — `OmniBridge.deployToken` reconstructs the identical borsh blob and checks the ECDSA signature:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

`omniBridgeChainId` is stored in the contract but is **never included** in the metadata hash. Compare this to `finTransfer`, which correctly embeds `omniBridgeChainId` twice in its borsh payload: [4](#0-3) 

**Starknet side** — `MetadataPayloadImpl::to_borsh` has the same omission: [5](#0-4) 

**Shared derived address** — The deployment script derives `nearBridgeDerivedAddress` from the same MPC root key and path `"bridge-1"` for every chain: [6](#0-5) 

This means `ECDSA.recover(keccak256(borsh(MetadataPayload)), sig) == nearBridgeDerivedAddress` evaluates identically on Ethereum, Base, BSC, and Starknet for the same metadata tuple.

---

### Impact Explanation

An attacker who observes a `LogMetadataEvent` on NEAR (public log, no privilege required) extracts the `SignatureResponse` and calls `deployToken` on any other chain's `OmniBridge` with the same `metadata` struct and signature bytes. The call passes signature verification and:

1. Registers the NEAR token ID → EVM address mapping on the target chain (`nearToEthToken`, `isBridgeToken`).
2. Permanently blocks the protocol from deploying its own canonical token for that NEAR token ID on that chain (`ERR_TOKEN_EXIST` on any subsequent legitimate `deployToken`).
3. In the Wormhole variant, also publishes a `DeployToken` Wormhole message from the attacker-triggered call, potentially confusing downstream Wormhole consumers.

The attacker controls which EVM address becomes the canonical bridge token for a given NEAR asset on any chain, hijacking the canonical asset identity. This matches the allowed High impact: *"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment."*

---

### Likelihood Explanation

- NEAR transaction logs are fully public; any observer can extract a `LogMetadataEvent` and its embedded `SignatureResponse`.
- The MPC-derived address is identical across all EVM deployments by design.
- `deployToken` is an unpermissioned, payable external function callable by anyone.
- No nonce, no replay guard, and no chain discriminator exist in the metadata signature path.

All preconditions are trivially satisfiable by any unprivileged actor.

---

### Recommendation

Include `omniBridgeChainId` (and optionally the OmniBridge contract address) in the borsh-encoded payload that is signed and verified for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
// In deployToken borsh encoding:
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // chain discriminator
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same fix to the NEAR `MetadataPayload` struct and the Starknet `MetadataPayloadImpl::to_borsh`. The NEAR signing side must include the destination chain ID in the payload before requesting the MPC signature.

---

### Proof of Concept

1. On NEAR, call `log_metadata("foo.near")`. Observe the emitted `LogMetadataEvent` containing `SignatureResponse { big_r, s, recovery_id }`.

2. Reconstruct the signature bytes via `SignatureResponse::to_bytes()`.

3. On chain A (e.g., Ethereum), call:
   ```solidity
   omniBridgeA.deployToken(sigBytes, MetadataPayload("foo.near","Foo","FOO",18));
   // succeeds — token deployed on chain A
   ```

4. On chain B (e.g., Base), call with the **identical** `sigBytes` and `metadata`:
   ```solidity
   omniBridgeB.deployToken(sigBytes, MetadataPayload("foo.near","Foo","FOO",18));
   // also succeeds — ECDSA.recover produces the same nearBridgeDerivedAddress
   // foo.near is now registered on chain B without protocol authorization
   ```

5. Any subsequent legitimate `deployToken` call for `foo.near` on chain B reverts with `ERR_TOKEN_EXIST`.

The borsh hash is provably identical on both chains because neither `omniBridgeChainId` nor any contract address is included in the signed payload.
