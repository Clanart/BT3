### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Validation, Corrupting DataLayer Merkle Tree Root and Invalidating Proofs - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `insert` enforces. When the tree already has ≥ 2 leaves, every item in the batch is written directly to the blob without any uniqueness check. When the tree has ≤ 1 leaf, only the last two items are checked; all earlier items in the vector are written unchecked. An attacker who controls the batch input can insert the same key multiple times, producing a blob whose leaf count exceeds the cache's key count, a corrupted Merkle root, and proofs of inclusion that do not match the committed root.

### Finding Description

`MerkleBlob::insert` guards against duplicate keys and hashes at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When `leaf_count > 1`, the entire `if` block is skipped and every item in the caller-supplied vector is written directly via `insert_entry_to_blob` with no uniqueness check:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // checked
    }
}

for ((key, value), hash) in keys_values_hashes {   // ← ALL items when leaf_count > 1
    let new_leaf_index = self.get_new_index();
    // ... build block ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // NO duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which silently overwrites the existing `key_to_index` entry for a duplicate key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

After the call, the blob contains two leaf nodes with the same key, but the cache tracks only the second. The tree structure (parent/child links) is built around both leaves, so the computed root hash incorporates both. `check_integrity` will detect the mismatch between `leaf_count` (from blob traversal) and `key_to_index_cache_length`:

```rust
if leaf_count != key_to_index_cache_length {
    return Err(Error::IntegrityKeyToIndexCacheLength(...));
}
``` [4](#0-3) 

The Python binding `py_batch_insert` is the direct attacker-controlled entry point:

```rust

    hashes: Vec<Hash>,

### Impact Explanation

A caller who supplies a batch containing a repeated `KeyId` (or a `KeyId` already present in the tree) causes:

1. **Corrupted Merkle root** — the root hash is computed over a tree that contains two leaf nodes for the same key. Any downstream consumer of `get_root_hash()` receives a hash that does not correspond to a valid key-value mapping.
2. **Forged / invalid proofs of inclusion** — `get_proof_of_inclusion` resolves the key through the cache (which tracks only the second leaf) and builds a proof path against the corrupted root. The proof will fail `valid()` or will prove a different value than the one the caller believes is committed.
3. **Persistent state corruption** — the blob is serialized to disk via `to_path`. Once written, the corrupted tree is reloaded and the integrity violation persists across restarts.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The Python binding is the primary interface used by the Chia DataLayer service. Any code path that assembles a batch from external store updates and calls `batch_insert` without pre-deduplicating keys triggers the bug. The condition is easy to satisfy accidentally (duplicate keys in a delta) and trivially exploitable intentionally by a store owner who controls the key-value pairs submitted to the DataLayer.

### Recommendation

Add duplicate-key and duplicate-hash checks at the start of the unchecked loop in `batch_insert`, mirroring the guards in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, pre-deduplicate the entire input vector (including the two items handled by the `if` branch) before any insertion begins, and return an error if duplicates are found.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 → entire batch skips checks
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(200), ValueId(200), h(200))

# Batch with a duplicate key (KeyId(1) appears twice)
keys_values = [(KeyId(1), ValueId(1)), (KeyId(2), ValueId(2)), (KeyId(1), ValueId(999))]
hashes      = [h(1),                   h(2),                   h(3)]

# Should raise KeyAlreadyPresentError — but does NOT
blob.batch_insert(keys_values, hashes)

blob.calculate_lazy_hashes()

# check_integrity reveals the corruption
try:
    blob.check_integrity()
    print("CORRUPT: integrity check passed on a corrupted tree")
except Exception as e:
    print(f"Integrity violation detected: {e}")

# Proof of inclusion for KeyId(1) is built against a corrupted root
proof = blob.get_proof_of_inclusion(KeyId(1))
print(f"Proof valid: {proof.valid()}")  # may be False or prove wrong value
