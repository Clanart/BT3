### Title
Unconstrained `is_reverted` Flag in Invoke Transaction Execution Due to No-Op `check_is_reverted` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary
`check_is_reverted` is defined as a completely empty function that performs no validation. The `is_reverted` flag controlling whether `__execute__` runs for invoke transactions is set exclusively by a prover-controlled hint and is never constrained by any Cairo assertion. A malicious prover can set `is_reverted = 1` for any valid invoke transaction, causing the execution step to be silently skipped while the fee is still charged — a direct, provably-valid loss of user funds.

---

### Finding Description

In `execution_constraints.cairo`, the function meant to enforce the revert invariant is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

It accepts the `is_reverted` argument and immediately returns — no `assert`, no range check, no constraint of any kind is applied.

In `transaction_impls.cairo`, `execute_invoke_function_transaction` sets `is_reverted` via a hint and then calls this no-op:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    ...
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

Because `check_is_reverted` enforces nothing, the Cairo constraint system places **zero restriction** on the value of `is_reverted`. The prover can freely set it to `1` (reverted) for any transaction that actually succeeded validation, and the resulting proof will be accepted as valid on L1.

The same unconstrained pattern appears in `execute_l1_handler_transaction`, where `is_reverted` is set by hint and used directly with no call to `check_is_reverted` at all:

```cairo
local is_reverted;
%{ IsReverted %}
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [3](#0-2) 

For L1 handlers, an early return skips `consume_l1_to_l2_message`, leaving the L1-to-L2 message unconsumed and the associated L1 funds locked. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds (invoke transactions):** For any invoke transaction, the prover sets `is_reverted = 1`. The `__execute__` entry point is skipped entirely. `charge_fee` still runs and deducts the full fee from the user's account. The user's intended state changes never occur, but their funds are taken. Because the resulting proof satisfies all Cairo constraints (since `check_is_reverted` imposes none), the L1 verifier accepts it as a valid block.

**Critical — Permanent freezing of funds (L1 handler transactions):** For any L1-to-L2 message transaction, the prover sets `is_reverted = 1`. The function returns before `consume_l1_to_l2_message` is called. The message is never marked consumed in the OS output, so the L1 bridge contract retains the deposited funds with no on-chain mechanism to release them within the proven block.

---

### Likelihood Explanation

The StarkNet ZK-proof security model is explicitly designed so that the prover need not be trusted — Cairo constraints are the sole source of correctness guarantees. `check_is_reverted` is the designated constraint point for the revert flag, but it is empty. Any prover instance (i.e., the sequencer operating the OS) can exploit this with zero additional effort: it simply supplies `is_reverted = 1` in the `IsReverted` hint for targeted transactions. No key compromise, no external dependency, and no network-level attack is required. The attack is silent and produces a proof indistinguishable from an honest one.

---

### Recommendation

`check_is_reverted` must enforce the invariant it is named after. The correct approach is to derive `is_reverted` from a verifiable on-chain condition rather than a bare hint, or to assert that the hint value is consistent with a constrained execution outcome. For example, the function should assert that `is_reverted` is a boolean (`0` or `1`) and, more importantly, that it matches the `failure_flag` returned by the actual entry-point execution — analogous to how `execute_entry_point` derives `is_reverted` directly from `entry_point_return_values.failure_flag` without relying on an unconstrained hint: [5](#0-4) 

At minimum, `check_is_reverted` must contain `assert is_reverted * (1 - is_reverted) = 0` to enforce booleanness, and the broader design must tie the flag to a provable execution outcome rather than a free hint.

---

### Proof of Concept

1. User submits a valid invoke transaction (e.g., a token transfer). `__validate__` passes.
2. The prover, when running the OS Cairo program, supplies `is_reverted = 1` in the `IsReverted` hint at line 339 of `transaction_impls.cairo`.
3. `check_is_reverted(1)` is called — it returns immediately with no assertion.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` (i.e., `__execute__`) is never called.
5. `charge_fee` executes normally, deducting the full fee from the user's account.
6. The OS output records the transaction as reverted with fee charged. No state changes from `__execute__` appear.
7. The Cairo proof is generated and verified on L1. It is valid — no constraint was violated.
8. The user has lost their fee and their transaction had no effect. [1](#0-0) [2](#0-1)
