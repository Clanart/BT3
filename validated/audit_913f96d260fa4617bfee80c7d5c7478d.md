### Title
P2P Peer-Supplied `state_diff_length=0` Header Causes Empty State Diff Stored and Served — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

A malicious p2p peer can send a `SignedBlockHeader` with `state_diff_length = Some(0)` for a block that actually has a non-empty state diff. The header sync accepts this header without any cryptographic verification of the block hash or signature content. The state diff sync then reads `target_state_diff_len = 0` from the stored header, the while loop is never entered, and an empty `ThinStateDiff` is stored. The RPC `get_state_update` serves the empty state diff, and the class sync skips all classes declared in that block.

---

### Finding Description

**Step 1 — Header accepted without cryptographic guard.**

`ALLOWED_SIGNATURES_LENGTH = 1` is defined in `mod.rs`. [1](#0-0) 

`HeaderStreamBuilder::parse_data_for_block` only checks that the block number matches and that `signatures.len() == 1`. There is no cryptographic verification of the signature against the block hash or `state_diff_commitment`, and no block hash recomputation. The existing TODO confirms this guard is absent: [2](#0-1) 

A peer can therefore supply any `state_diff_length` value, including `Some(0)`, and the header is stored verbatim.

**Step 2 — State diff sync reads `target_state_diff_len` from the stored (attacker-controlled) header.**

`StateDiffStreamBuilder::parse_data_for_block` reads `state_diff_length` from the stored header: [3](#0-2) 

**Step 3 — When `target_state_diff_len == 0`, the while loop is never entered.**

The loop condition is `current_state_diff_len < target_state_diff_len`, i.e., `0 < 0 = false`. The loop body is skipped entirely: [4](#0-3) 

**Step 4 — The post-loop equality check also passes.**

`current_state_diff_len != target_state_diff_len` evaluates to `0 != 0 = false`, so no `BadPeer` error is raised: [5](#0-4) 

The function returns `Ok(Some((ThinStateDiff::default(), block_number)))` — an empty state diff — which is written to storage: [6](#0-5) 

**Step 5 — Class sync reads the empty stored state diff and skips all class syncing.**

`ClassStreamBuilder::parse_data_for_block` derives `target_class_len` from the stored state diff. With an empty state diff, `target_class_len = 0`, the while loop is never entered, and no classes are fetched for that block: [7](#0-6) 

**Step 6 — No `state_diff_commitment` verification anywhere in the p2p sync client path.**

Grep confirms zero calls to `verify_block_signature` or any `state_diff_commitment` check in `crates/apollo_p2p_sync/src/client/`. The `verify_block_signature` function exists in `starknet_api` but is never invoked here: [8](#0-7) 

---

### Correction to the Question's Description

The question states the function "immediately returns `Ok(None)`." This is **incorrect**. When `target_state_diff_len == 0`, the function returns `Ok(Some((empty_ThinStateDiff, block_number)))`, not `Ok(None)`. `Ok(None)` would cause the sync to sleep and retry; `Ok(Some(empty))` causes the empty state diff to be **committed to storage immediately**. The end result (empty state diff stored) is the same, but the mechanism differs.

The question also states "the while loop condition `current_state_diff_len < 0`" — the actual condition is `current_state_diff_len < target_state_diff_len`.

---

### Impact Explanation

**Actual impact: High** — not Critical as claimed.

The committer (`apollo_committer`) is **not** part of the p2p sync path. It is used by the sequencer to compute the global root from execution outputs, not from p2p-synced state diffs. The `verify_state_diff_hash: true` default in `CommitterConfig` is irrelevant here: [9](#0-8) 

The actual impacts are:

1. **RPC `starknet_getStateUpdate` returns an authoritative-looking empty state diff** for the affected block — a High-severity wrong value from an RPC endpoint.
2. **Class sync skips all classes declared in the affected block**, leaving the node without those class definitions. Subsequent `starknet_call` / `starknet_estimateFee` calls for transactions using those classes will fail or return wrong results.
3. **The p2p sync server re-serves the corrupted `state_diff_length = 0`** to downstream nodes syncing from this node, propagating the corruption: [10](#0-9) 

---

### Likelihood Explanation

Any p2p peer that can establish a connection and respond to a header query can trigger this. No special privileges are required. The attack requires only sending a `SignedBlockHeader` with `state_diff_length = Some(0)` and exactly one (arbitrary) `BlockSignature`.

---

### Recommendation

1. **Cryptographically verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` using `verify_block_signature` (already implemented in `starknet_api::block`), binding the signature to `poseidon_hash(block_hash, state_diff_commitment)`.
2. **Verify the stored state diff against `state_diff_commitment`** after `append_state_diff` in the state diff sync, using `calculate_state_diff_hash`.
3. **Recompute and verify the block hash** from the stored header fields before accepting a header, resolving the existing TODO at `header.rs:102`.
