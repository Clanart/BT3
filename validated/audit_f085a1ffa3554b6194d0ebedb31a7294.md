### Title
Decimal Normalization Floor Division in `sign_transfer` Permanently Locks User Funds When `amount - fee < 10^(origin_decimals - decimals)` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` accepts any `(amount, fee)` pair satisfying `fee < amount`, but never validates that the normalized net amount `normalize_amount(amount - fee, decimals)` is non-zero. When a user bridges tokens to a chain with fewer decimals and the net amount falls below the minimum representable unit on the destination chain, `sign_transfer` panics with `BridgeError::InvalidAmountToTransfer`. Because no public cancel mechanism exists for pending transfers, the user's locked/burned tokens are permanently irrecoverable.

---

### Finding Description

**Root cause — missing pre-flight check in `init_transfer`:**

`init_transfer` stores the transfer and immediately burns or locks the user's tokens:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

This is the only validation on the `(amount, fee)` pair. No check is made against the destination chain's decimal precision.

**Floor division in `normalize_amount`:**

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For any token where `origin_decimals > decimals`, if `amount - fee < 10^(origin_decimals - decimals)`, the result is `0`.

**Hard revert in `sign_transfer`:**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

When `normalize_amount` returns `0`, `sign_transfer` panics. The transfer message remains in `pending_transfers` forever.

**No cancel path exists:**

A grep for `cancel_transfer`, `withdraw_transfer`, `refund_transfer`, `remove_pending`, and `cancel_pending` in `near/omni-bridge/src/` returns zero matches. `remove_transfer_message` is only called internally from `claim_fee_callback` and `process_fin_transfer_to_near` — neither of which is reachable for a transfer that can never be signed.

**`update_transfer_fee` cannot rescue the transfer:**

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

The fee can only be increased. If `amount - current_fee < 10^diff`, then `current_fee > amount - 10^diff`, so the user cannot raise the fee to `amount - 10^diff` (which would be lower than the current fee). The transfer is irrecoverably stuck.

---

### Impact Explanation

User tokens are burned or locked in `init_transfer_internal` before any decimal-precision check occurs. Once locked, the only exit path is through `sign_transfer` → MPC signing → destination chain finalization → `claim_fee`. All of these are blocked when `normalize_amount(amount - fee) == 0`. There is no DAO-accessible cancel function for pending transfers. The funds are permanently frozen in the bridge contract.

This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

The scenario is reachable by any unprivileged bridge user calling `ft_transfer_call` → `init_transfer`. Concrete triggering conditions:

| Origin decimals | Destination decimals | `diff` | Minimum net amount to avoid lock |
|---|---|---|---|
| 24 (NEAR) | 18 (ETH) | 6 | 10^6 yoctoNEAR ≈ negligible |
| 24 (NEAR) | 9 (Solana) | 15 | 10^15 yoctoNEAR = 0.001 NEAR |
| 24 (NEAR) | 6 (USDC-like) | 18 | 10^18 yoctoNEAR = 1 NEAR |

For tokens with large decimal differences (e.g., 24 → 6), any transfer of less than 1 NEAR-equivalent is permanently locked. A user who sets `fee = amount - 1` (the maximum allowed by the `fee < amount` check) with any `amount < 10^diff` triggers this condition. The `update_transfer_fee` function cannot rescue the transfer because the fee can only be increased, not decreased.

---

### Recommendation

1. **Add a pre-flight normalization check in `init_transfer`** (or in `ft_on_transfer` before tokens are burned/locked). Look up the destination token's `Decimals` and assert `normalize_amount(amount - fee, decimals) > 0` before accepting the transfer.

2. **Add a user-callable cancel function** for pending transfers that have not yet been signed, returning the locked/burned tokens to the sender. This provides a safety valve for any future edge cases.

---

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 24`, `decimals = 6` (e.g., bridging a NEAR-native token to an EVM chain that uses 6 decimals). `diff = 18`, `factor = 10^18`.

**Steps:**

1. User calls `ft_transfer_call` on the token contract, transferring `amount = 5 × 10^17` (0.5 NEAR-equivalent) with `fee = 0` and recipient on the EVM chain.

2. `init_transfer` passes the only guard: `0 < 5 × 10^17` ✓. The transfer message is stored in `pending_transfers`. `init_transfer_internal` burns/locks `5 × 10^17` tokens.

3. A trusted relayer calls `sign_transfer(transfer_id, ...)`.

4. Inside `sign_transfer`:
   - `amount_without_fee() = 5 × 10^17 - 0 = 5 × 10^17`
   - `normalize_amount(5 × 10^17, {origin: 24, dest: 6}) = 5 × 10^17 / 10^18 = 0`
   - `require!(0 > 0, ...)` → **panic: `ERR_INVALID_AMOUNT_TO_TRANSFER`**

5. The transfer remains in `pending_transfers`. No cancel function exists. The user's `5 × 10^17` tokens are permanently locked.

6. The user attempts `update_transfer_fee` to raise the fee to `amount - 10^18 = 5×10^17 - 10^18 < 0` — impossible. No recovery path exists. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
