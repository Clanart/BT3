### Title
State-Sync Header Accepts Cross-Shard Chunk Without Shard-ID Binding Check — (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`set_state_header` validates that a supplied chunk is cryptographically included in the canonical block, but **never verifies that the chunk's own `shard_id` matches the `shard_id` parameter**. A malicious peer can substitute a valid chunk from shard B (plus its genuine merkle proof) when answering a state-header request for shard A. The node accepts the header, stores it under shard A's DB key, and then applies shard B's trie state into shard A's storage — corrupting the syncing node's state for shard A.

---

### Finding Description

`set_state_header` performs five validation steps before persisting the header:

**Step 1-2 — chunk internal validity** (`validate_chunk_proofs`): [1](#0-0) 

This checks the chunk hash, tx-root, and receipts-root. It does **not** check `chunk.shard_id()`.

**Step 3a — chunk is in the block** (`verify_path`): [2](#0-1) 

`verify_path` proves that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` sits somewhere in `chunk_headers_root`. The position is encoded in the merkle proof supplied by the peer. The node never checks that this position corresponds to `shard_id`. A proof for shard B's chunk at index 1 passes even when `shard_id == 0`.

**No shard-id binding check exists anywhere in the function.** The header is then stored: [3](#0-2) 

Downstream, `set_state_part` retrieves the stored header and extracts the state root: [4](#0-3) 

Because the stored header contains shard B's chunk, `state_root` is shard B's `prev_state_root`. State parts for shard B validate correctly against it and are stored under shard A's `StatePartKey`. `apply_state_part` then writes shard B's trie nodes into shard A's `shard_uid`: [5](#0-4) 

`validate_state_part_impl` only checks the trie structure against the state root; it has no shard-id awareness: [6](#0-5) 

---

### Impact Explanation

After the attack completes, shard A's trie in the syncing node's DB contains shard B's account/contract state. Every subsequent block application for shard A starts from the wrong state root, producing wrong `ChunkExtra` values. If the node is a chunk producer or validator for shard A it will sign or endorse chunks with an incorrect post-state-root, causing it to diverge from the honest network. At minimum the node is permanently unable to track shard A without a full re-sync. **Impact: High.**

---

### Likelihood Explanation

State sync peers are selected from the p2p network; any reachable node can volunteer as a source. The attacker needs only:
1. A valid chunk from shard B that is included in the target block (publicly available on-chain).
2. The genuine merkle proof for that chunk's position in `chunk_headers_root` (derivable from the block header).

No privileged key material is required. The attack is fully constructible by an unprivileged network peer. **Likelihood: Medium** (requires the victim to select the attacker as its state-sync peer, which is non-trivial but achievable via eclipse or Sybil positioning).

---

### Recommendation

After `validate_chunk_proofs` succeeds, add an explicit shard-id binding check in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} does not match requested shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

This should be inserted immediately after the `validate_chunk_proofs` call at line 385, before the merkle-path check.

        let partial_state = part.to_partial_state();
        let Ok(partial_state) = part.to_partial_state() else {
            // Deserialization error means we've got the data from malicious peer

            return StatePartValidationResult::Invalid;

            Ok(_) => StatePartValidationResult::Valid,
            // Storage error should not happen
            Err(err) => {
                tracing::error!(target: "state-parts", ?err, "state part storage error");
                StatePartValidationResult::Invalid
            }
        }
    }
