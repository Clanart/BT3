### Title
`batch_insert` in `MerkleBlob` silently accepts duplicate keys, corrupting tree root and `block_status_cache` state — (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that the single-item `insert` path enforces. When a key already present in the tree is included in a batch, the in-memory `block_status_cache` is silently overwritten to point to the new leaf while the old leaf remains physically embedded in the blob and wired into the tree's internal-node structure. The result is a tree whose root hash is computed over two leaves for the same key, a stale `leaf_hash_to_index` entry for the old hash, and an orphaned blob slot that is neither free nor reachable through the cache — a direct analog of the `lockFunds` / `increaseActionLockedBalance` accounting split in the reference report.

### Finding Description

`MerkleBlob::insert` (the single-item path) opens with two mandatory guards:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `self.insert()` for at most the last two items in the input vector, and only when the tree currently holds ≤ 1 leaf. Every other item is written directly through `insert_entry_to_blob` with no duplicate check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1024-1027
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    Node::Internal(..) => self.block_status_cache.add_internal(index),
}
```

`add_leaf` silently overwrites the existing `key_to_index` entry for the duplicate key:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // old index silently dropped
    self.leaf_hash_to_index.insert(leaf.hash, index); // old hash entry NOT removed
}
```

After the overwrite:

| State | Expected | Actual |
|---|---|---|
| `key_to_index[K]` | old_index | new_index (overwritten) |
| `leaf_hash_to_index[old_hash]` | removed | still present (stale) |
| old_index in `free_indexes` | yes (freed) | **no** (orphaned) |
| old leaf in blob | absent | **present**, wired into tree |

The batch then builds internal nodes over the new leaves and splices the resulting subtree into the existing tree via `insert_subtree_at_key`. The old leaf remains a live child of some internal node, so the root hash is computed over **both** the old and new leaf for key K.

### Impact Explanation

**Corrupted tree root.** The root hash returned by `get_root_hash()` / `get_hash_at_index(TreeIndex(0))` is computed over two leaves for the same key. Any downstream consumer that commits this root on-chain or uses it as a trust anchor receives a hash that does not correspond to any valid single-valued mapping.

**Forged proof of inclusion.** `get_proof_of_inclusion(K)` resolves K through `key_to_index` to the new leaf and builds a Merkle path that is valid against the corrupted root. A verifier holding only the root hash will accept this proof as proving `K → new_value`, even though the tree was never legitimately updated.

**Stale `get_node_by_hash`.** `get_node_by_hash(old_hash)` still resolves through the stale `leaf_hash_to_index` entry and returns the old `(key, value)` pair, creating a split-brain between the two lookup paths.

**`check_integrity` failure.** The tree traversal counts more leaves than `key_to_index` has entries, triggering `IntegrityKeyToIndexCacheLength`. Any code path that calls `check_integrity` after a corrupted `batch_insert` will panic or error, potentially halting the DataLayer node.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is a public method exposed through the Python wheel (`datalayer.pyi` line 331). The DataLayer sync protocol receives key-value batches from remote peers. A malicious peer that includes a key already present in the recipient's local store triggers the corruption path with no special privilege — only the ability to participate in the DataLayer sync protocol. The tree must already contain ≥ 2 leaves (the common case for any non-trivial store) for the guard-bypassing loop to execute on all items.

### Recommendation

Add the same duplicate guards that `insert` uses at the top of the bulk loop in `batch_insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, collect all keys and hashes from the batch into a temporary set before any blob writes and reject the entire batch on the first duplicate, preserving atomicity.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib, struct

def h(n):
    return bytes(hashlib.sha256(struct.pack(">q", n)).digest())

blob = MerkleBlob(bytearray())

# Seed the tree with 3 leaves so leaf_count > 1 (guard-bypass path active)
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), h(i))
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with key 0 already present — no error raised
blob.batch_insert([(KeyId(0), ValueId(999))], [h(999)])
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# Root changed silently; proof for key 0 now proves value 999
assert root_before != root_after          # tree root corrupted
proof = blob.get_proof_of_inclusion(KeyId(0))
assert proof.valid()                      # proof valid against wrong root
# check_integrity() will raise IntegrityKeyToIndexCacheLength
blob.check_integrity()                    # raises — confirms state corruption
