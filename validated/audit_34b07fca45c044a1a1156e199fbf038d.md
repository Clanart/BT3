Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain excess ETH left in `MetricOmmPoolLiquidityAdder` after a direct `addLiquidityExactShares` call — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` sends the contract's full ETH balance to `msg.sender` with no access control. `addLiquidityExactShares` is `payable` and `pay()` wraps only the exact amount the pool requests, leaving any surplus ETH in the contract. A user who calls `addLiquidityExactShares` directly with excess ETH loses that surplus to any caller of `refundETH()`.

## Finding Description

`refundETH()` is unconditionally callable by any address:

```solidity
// PeripheryPayments.sol:58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // no caller check
    }
}
```

ETH enters legitimately via `addLiquidityExactShares{value: X}(...)`. Inside `pay()`, only the exact amount the pool requests via callback is wrapped and forwarded; any surplus stays in the contract:

```solidity
// PeripheryPayments.sol:73-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // wraps only `value`, not nativeBalance
        IERC20(WETH).safeTransfer(recipient, value);
```

`_addLiquidity` performs no ETH refund after the pool call returns:

```solidity
// MetricOmmPoolLiquidityAdder.sol:194-200
try IMetricOmmPoolActions(pool)
  .addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData) returns (
  uint256 a0, uint256 a1
) {
    amount0Added = a0;
    amount1Added = a1;
    _clearPayContext();   // no ETH refund
```

The `receive()` guard only blocks bare ETH transfers; it does not prevent ETH from entering via `payable` function calls:

```solidity
// PeripheryPayments.sol:32-34
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
```

The intended safe pattern is `multicall([addLiquidityExactShares(...), refundETH()])`, which is atomic and preserves `msg.sender` via `delegatecall`. The test confirms this is the only tested refund path:

```solidity
// test/MetricOmmPoolLiquidityAdder.native.t.sol:100-105
calls[0] = abi.encodeWithSelector(ADD_LIQUIDITY_EXACT_SHARES_WITH_OWNER, ...);
calls[1] = abi.encodeWithSelector(helper.refundETH.selector);
helper.multicall{value: msgValue}(calls);
```

However, `addLiquidityExactShares` is `external payable` with no enforcement that it must be called through `multicall`. The test at line 83-91 demonstrates a direct call with ETH is a supported usage pattern (`test_mixedNativeAndWeth_exactShares` calls `addLiquidityExactShares{value: nativePart}(...)` directly). A user who sends more ETH than the pool needs leaves the surplus permanently exposed.

## Impact Explanation

Any ETH remaining in the contract after a direct `addLiquidityExactShares` call is immediately claimable by any address via `refundETH()`. The user suffers a direct, complete loss of their unspent native ETH with no recovery path. This is a direct loss of user principal, meeting the Critical/High threshold under Sherlock contest rules.

## Likelihood Explanation

MEV bots routinely monitor the mempool for exactly this pattern. A user who calls `addLiquidityExactShares{value: X}(...)` directly — a natural usage given the function is `payable` and direct calls are demonstrated in tests — and sends more ETH than the pool needs will lose the surplus in the same block. The attack requires no special privileges, no malicious pool, and no non-standard tokens.

## Recommendation

Add an automatic ETH refund at the end of `_addLiquidity` in `MetricOmmPoolLiquidityAdder.sol`:

```solidity
function _addLiquidity(...) internal returns (...) {
    _setPayContext(...);
    try ... {
        ...
        _clearPayContext();
    } catch { ... }
    uint256 leftover = address(this).balance;
    if (leftover > 0) _transferETH(payer, leftover);
}
```

Alternatively, restrict `refundETH()` so it can only be called via `delegatecall` (i.e., from within `multicall`), preventing standalone invocations.

## Proof of Concept

1. WETH pool exists with token0 = WETH, token1 = ERC20.
2. User calls `adder.addLiquidityExactShares{value: 2 ether}(pool, salt, deltas, 2e18, 0, "")` directly (not via `multicall`). Pool only needs 1 ETH worth of WETH.
3. `pay()` wraps and forwards exactly 1 ETH; 1 ETH remains in `adder`.
4. `_addLiquidity` returns with no ETH refund (`_clearPayContext()` only clears transient storage).
5. Attacker calls `adder.refundETH()` in the same or next block.
6. Attacker receives 1 ETH. User's surplus is permanently lost.
