### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash is actually declared on-chain before committing the state change. This is directly analogous to M-09's use of `transferFrom` instead of `safeTransferFrom`: an "unsafe" state-transition path is used that skips a critical receiver-side validity check. Any contract that allows user-influenced `replace_class` calls can have its class replaced with an undeclared hash, permanently rendering it uncallable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` (lines 877–916) writes the caller-supplied `class_hash` directly into `contract_state_changes` with no check that the hash exists in `contract_class_changes` (the declared-class registry). The developers explicitly acknowledge this gap: [1](#0-0) 

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The function signature lacks `contract_class_changes` as an implicit argument, so no lookup is even possible: [2](#0-1) 

The identical omission exists in the deprecated path: [3](#0-2)
