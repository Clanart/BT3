### Title
`finish_withdraw_v2` Lacks Pause Check, Bypassing Bridge Init-Transfer Pause — (`near/omni-bridge/src/lib.rs`)

### Summary
The NEAR omni-bridge contract exposes a public `finish_withdraw_v2` function that initiates cross-chain transfers (NEAR → EVM) without any pause guard. The primary init-transfer entry point `ft_on_transfer` carries a `#[pause]` attribute, but `finish_withdraw_v2` — which is callable by any deployed token contract and produces the same `InitTransferEvent` — has no such guard. When the bridge is paused to stop outbound transfers, users can still trigger `finish_withdraw_v2` through a deployed token contract's `withdraw`-style function, burning their NEAR-side tokens while the bridge is unable to finalize the transfer, resulting in permanent fund loss.

### Finding Description

`ft_on_transfer` is the standard entry point for init transfers and is correctly gated: [1](#0-0) 

```rust
#[pause(except(roles(Role::DAO, Role::UnrestrictedDeposit)))]
pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
```

`finish_withdraw_v2`, however, is a `pub` function with **no** `#[pause]` attribute: [2](#0-1) 

```rust
pub fn finish_withdraw_v2(
    &mut self,
    #[serializer(borsh)] sender_id: &AccountId,
    #[serializer(borsh)] amount: u128,
    #[serializer(borsh)] recipient: String,
) {
    let token_id = env::predecessor_account_id();
    require!(self.is_deployed_token(&token_id),);
    ...
    env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
}
```

The only guard is that `env::predecessor_account_id()` must be a deployed bridge token. Any deployed token contract that exposes a user-callable `withdraw` function (the standard pattern for legacy NEAR bridge tokens) will call `finish_withdraw_v2` on the bridge after burning the user's tokens. Because the bridge contract does not check the pause state inside `finish_withdraw_v2`, the pause on `ft_on_transfer` is fully bypassed.

The downstream finalization path (`sign_transfer`) is also paused: [3](#0-2) 

```rust
#[pause(except(roles(Role::DAO)))]
pub fn sign_transfer(...) -> Promise {
```

This means the transfer is recorded and tokens are burned, but the transfer can never be signed or finalized while the bridge remains paused.

### Impact Explanation

When the bridge is paused (e.g., in response to a security incident):

1. User calls `withdraw` on a deployed bridge token contract.
2. The token contract burns the user's tokens and calls `finish_withdraw_v2` on the bridge.
3. `finish_withdraw_v2` stores the pending transfer and increments `current_origin_nonce` — no pause check occurs.
4. The user's NEAR-side tokens are permanently burned.
5. `sign_transfer` is paused, so no MPC signature is produced and no EVM-side `finTransfer` can be submitted.
6. The user's funds are irrecoverably locked: burned on NEAR, unclaimable on EVM.

This matches the **Critical** allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge flows.*

### Likelihood Explanation

- Any holder of a deployed bridge token can trigger this path by calling the token contract's `withdraw` function — no special role or privilege is required.
- The deployed token contracts are the standard omni-token contracts whose `withdraw` function calls `finish_withdraw_v2` on the bridge.
- The bridge is paused precisely during incidents where allowing new outbound transfers is dangerous; this bypass defeats that protection entirely.

### Recommendation

Add the `#[pause]` attribute to `finish_withdraw_v2`, consistent with `ft_on_transfer`:

```rust
#[pause(except(roles(Role::DAO, Role::UnrestrictedDeposit)))]
pub fn finish_withdraw_v2(
    &mut self,
    ...
) {
```

This ensures that all init-transfer entry points — both the NEP-141 `ft_on_transfer` path and the legacy `finish_withdraw_v2` path — are subject to the same pause enforcement.

### Proof of Concept

1. Admin pauses the bridge (init transfers paused).
2. Attacker holds 1000 `eth.bridge.near` tokens (a deployed bridge token).
3. Attacker calls `eth.bridge.near::withdraw(amount=1000, recipient="0xAttacker")`.
4. `eth.bridge.near` burns 1000 tokens and calls `omni-bridge::finish_withdraw_v2(sender_id=attacker, amount=1000, recipient="0xAttacker")`.
5. `finish_withdraw_v2` executes without reverting — no pause check — stores the pending transfer, emits `InitTransferEvent`.
6. Attacker's 1000 tokens are burned on NEAR.
7. `sign_transfer` is paused; no MPC signature is ever produced.
8. Attacker's 1000 tokens are permanently lost. [2](#0-1) [1](#0-0) [3](#0-2)

    #[allow(clippy::needless_pass_by_value)]
    pub fn finish_withdraw_v2(
        &mut self,
        #[serializer(borsh)] sender_id: &AccountId,
        #[serializer(borsh)] amount: u128,
        #[serializer(borsh)] recipient: String,
    ) {
        let token_id = env::predecessor_account_id();
        require!(self.is_deployed_token(&token_id),);

        self.current_origin_nonce += 1;
        let destination_nonce = self.get_next_destination_nonce(ChainKind::Eth);

        let transfer_message = TransferMessage {
            origin_nonce: self.current_origin_nonce,
            token: OmniAddress::Near(token_id),
            amount: U128(amount),
            recipient: OmniAddress::Eth(
                H160::from_str(&recipient).near_expect(BridgeError::InvalidRecipientAddress),
            ),
            fee: Fee {
                fee: U128(0),
                native_fee: U128(0),
            },
            sender: OmniAddress::Near(sender_id.clone()),
            msg: String::new(),
            destination_nonce,
            origin_transfer_id: None,
        };

        let required_storage_balance =
            self.add_transfer_message(transfer_message.clone(), sender_id.clone());

        self.update_storage_balance(
            env::current_account_id(),
            required_storage_balance,
            NearToken::from_yoctonear(0),
        );

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
    }
