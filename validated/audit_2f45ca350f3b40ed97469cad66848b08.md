### Title
Wrong Shard Layout Used to Derive `prev_state_root` in `get_state_response_part` at Resharding Epoch Boundary — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` resolves the `shard_index` used to read `prev_state_root` from `prev_block.chunks()` using the **sync block's (new epoch's) shard layout**, while `prev_block` belongs to the **previous epoch** whose chunks are indexed by the **old epoch's shard layout**. In `ShardLayoutV2`/`V3`, shard IDs and shard indices are explicitly decoupled after resharding, so the two index spaces diverge. The sibling function `compute_state_response_header` correctly uses `prev_shard_layout` for the same lookup; `get_state_response_part` does not, producing a state part built from the wrong trie root.

---

### Finding Description

`get_state_response_part` in `chain/chain/src/state_sync/adapter.rs` fetches the shard layout for the **sync block's epoch** and uses it to compute `shard_index`:

```rust
// lines 295, 305, 311-316
let epoch_id = block.header().epoch_id();          // sync block = new epoch
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;  // new layout
...
let prev_block = self.chain_store.get_block(header.prev_hash())?;   // old epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;          // index in NEW layout
let state_root = prev_block
    .chunks()
    .get(shard_index)                                                // indexes OLD epoch chunks!
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

The sibling function `compute_state_response_header` performs the identical lookup correctly, explicitly fetching the **previous epoch's** shard layout:

```rust
// lines 93-96
let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;
``` [2](#0-1) 

The shard layout documentation makes the invariant explicit:

```rust
// In V0 & V1 the shard id and shard index are the same.
Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
// In V2 & V3 the shard id and shard index are **not** the same.
Self::V2(v2) => v2.get_shard_index(shard_id),
Self::V3(v3) => v3.get_shard_index(shard_id),
``` [3](#0-2) 

In `ShardLayoutV2`, after a shard split, child shards receive new non-contiguous IDs (e.g., `max_shard_id + 1`, `max_shard_id + 2`), so their index in the new layout differs from their parent's index in the old layout: [4](#0-3) 

**Concrete example** (4 → 5 shards):
- Old layout: shard IDs `[0, 1, 2, 3]`, indices `[0, 1, 2, 3]`
- New layout after splitting shard 3: shard IDs `[0, 1, 2, 4, 5]`, indices `[0, 1, 2, 3, 4]`

A state-sync request for `shard_id=4` (new shard, new-layout index=3) causes `get_state_response_part` to read `prev_block.chunks()[3]`, which is old shard 3's chunk — the parent shard. The `prev_state_root()` returned is the parent's full trie root, not the child shard's committed root. For `shard_id=5` (new-layout index=4), `prev_block.chunks().get(4)` returns `None` (old block has only 4 chunks), and the function returns `Error::InvalidShardId` — silently blocking state sync for that shard.

---

### Impact Explanation

The state part is generated from the wrong trie root. On the receiving side, `set_state_part` validates the part against the `state_root` committed in the `ShardStateSyncResponseHeader` (which was computed correctly by `compute_state_response_header`). The mismatch causes `validate_state_part` to return `Invalid`, and `set_state_part` returns an error: [5](#0-4) 

Every state-sync attempt for every affected shard at a resharding epoch boundary fails. Nodes cannot complete state sync and cannot join the network at that epoch. For the second child shard (out-of-bounds index), the serving node returns an error immediately, making the shard completely unservable via this path.

**Severity: High** — state sync is the only mechanism for nodes to bootstrap at a resharding epoch; a deterministic failure blocks all such nodes.

---

### Likelihood Explanation

The bug is triggered deterministically whenever:
1. A node requests state parts for a shard at a sync hash that is the first block of a resharding epoch (standard state sync scenario), and
2. The network uses `ShardLayoutV2` or `ShardLayoutV3` (both are present in production code and actively used in resharding tests).

Any peer can send a `StateRequestPart` message with a valid `shard_id` from the new epoch and a valid `sync_hash` at the resharding boundary. No privileged role is required. The serving node's `get_state_response_part` is the reachable root cause. [6](#0-5) 

---

### Recommendation

In `get_state_response_part`, mirror the pattern used in `compute_state_response_header`: fetch the **previous block's epoch's shard layout** and use it to compute `shard_index` before indexing into `prev_block.chunks()`:

```rust
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

The existing `shard_layout` (new epoch's) is still needed for the `shard_ids.contains(&shard_id)` validation and should be retained for that purpose.

---

### Proof of Concept

1. Deploy a network with `ShardLayoutV2` that reshards at epoch N (splitting shard 3 into shards 4 and 5).
2. Start a fresh node that needs to state-sync at epoch N's first block (`sync_hash`).
3. The node sends `StateRequestHeader` for `shard_id=4` → `compute_state_response_header` correctly uses `prev_shard_layout`, returns a header with the correct `state_root_node` and `chunk_prev_state_root`.
4. The node sends `StateRequestPart` for `shard_id=4`, `part_id=0` → `get_state_response_part` uses the **new** shard layout (index=3) to read `prev_block.chunks()[3].prev_state_root()` — this is old shard 3's root, which matches the header's root by coincidence for the first child, but the part boundaries computed by `obtain_state_part` with `PartId::new(part_id, num_parts)` will be wrong relative to the child shard's actual key range.
5. For `shard_id=5` (new-layout index=4): `prev_block.chunks().get(4)` returns `None` → `Error::InvalidShardId` → state sync for shard 5 is completely blocked.
6. `set_state_part` on the receiving node calls `validate_state_part` against the committed `state_root`; for any part where the root diverges, validation returns `Invalid` and state sync stalls permanently. [7](#0-6) [3](#0-2)

    pub fn get_state_response_part(
        &mut self,
        shard_id: ShardId,
        part_id: u64,
        sync_hash: CryptoHash,
    ) -> Result<StatePart, Error> {
        let _span = tracing::debug_span!(
            target: "sync",
            "get_state_response_part",
            %shard_id,
            part_id,
            ?sync_hash)
        .entered();
        let block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let header = block.header();
        let epoch_id = block.header().epoch_id();
        // Check cache
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id)).unwrap();
        if let Some(bytes) = self.chain_store.store_ref().get(DBCol::StateParts, &key) {
            metrics::STATE_PART_CACHE_HIT.inc();
            let state_part = StatePart::from_bytes(bytes.to_vec())?;
            return Ok(state_part);
        }
        metrics::STATE_PART_CACHE_MISS.inc();

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_ids = self.epoch_manager.shard_ids(epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let prev_block = self.chain_store.get_block(header.prev_hash())?;
        let shard_index = shard_layout.get_shard_index(shard_id)?;
        let state_root = prev_block
            .chunks()
            .get(shard_index)
            .ok_or(Error::InvalidShardId(shard_id))?
            .prev_state_root();
        let prev_hash = *prev_block.hash();
        let prev_prev_hash = *prev_block.header().prev_hash();
        let state_root_node = self
            .runtime_adapter
            .get_state_root_node(shard_id, &prev_hash, &state_root)
            .log_storage_error("get_state_root_node fail")?;
        let num_parts = get_num_state_parts(state_root_node.memory_usage);
        if part_id >= num_parts {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let current_time = Instant::now();
        let state_part = self
            .runtime_adapter
            .obtain_state_part(
                shard_id,
                &prev_prev_hash,
                &state_root,
                PartId::new(part_id, num_parts),
            )
            .log_storage_error("obtain_state_part fail")?;

        let elapsed_ms = (self.clock.now().signed_duration_since(current_time))
            .whole_milliseconds()
            .max(0) as u128;
        self.requested_state_parts
            .save_state_part_elapsed(&sync_hash, &shard_id, &part_id, elapsed_ms);

        // Cache the part data, but only if the corresponding header is also cached.
        // At epoch boundaries, clear_all_downloaded_parts() deletes all cached headers
        // and parts. Since serving runs on a separate actor, a part request can arrive
        // after the clear and re-create a StatePartKey without its StateHeaderKey,
        // which the storage validator treats as an inconsistency.
        let header_key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
        if self.chain_store.store_ref().exists(DBCol::StateHeaders, &header_key) {
            let mut store_update = self.chain_store.store().store_update();
            let bytes = state_part.to_bytes();
            store_update.set(DBCol::StateParts, &key, &bytes);
            store_update.commit();
        }

        Ok(state_part)
    }
