### Title
`ProofOfInclusion::valid()` Is a Tautology — Forged DataLayer Inclusion Proofs Always Pass - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary
The `ProofOfInclusion::valid()` method contains a logical tautology: the final check `existing_hash == self.root_hash()` is always `true` after the loop completes without returning `false`. The function only verifies internal self-consistency of the proof chain but never validates the proof against any external trusted root hash. An attacker can construct a `ProofOfInclusion` with any arbitrary `node_hash` and a chain of internally consistent hashes that will always pass `valid()`, regardless of the actual tree state.

### Finding Description
In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`, the `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()  // ← always true
}
``` [1](#0-0) 

After the loop, `existing_hash` holds the last `calculated_hash`, which was already verified to equal `layer.combined_hash` for the last layer. The `root_hash()` method returns `last.combined_hash`: [2](#0-1) 

Therefore `existing_hash == self.root_hash()` is always `true` after the loop completes. The function is semantically equivalent to returning `true` unconditionally once the loop passes. It never compares the derived root against any external trusted root hash.

The `valid()` method is the sole validation entry point exposed to Python via `py_valid()`: [3](#0-2) 

### Impact Explanation
An attacker who supplies a `ProofOfInclusion` to any verifier that calls only `proof.valid()` can:

1. Choose any arbitrary `node_hash` (claiming any key is in the tree).
2. Construct a chain of `ProofOfInclusionLayer` entries with internally consistent hashes (each `combined_hash` is the correct hash of the previous hash and the sibling).
3. The forged proof passes `valid()` unconditionally.

This allows an attacker to prove inclusion of any key in any DataLayer tree, enabling false state proofs. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation
The `valid()` method is the primary and only validation function exposed for `ProofOfInclusion` in both Rust and Python. Callers who rely solely on `valid()` — without separately checking `proof.root_hash() == trusted_root` — will accept forged proofs. Since the function name and signature suggest complete validation, this is a realistic mistake. The fuzz target and all tests call `proof.valid()` without any external root comparison: [4](#0-3) [5](#0-4) 

### Recommendation
The `valid()` method should accept a trusted root hash parameter and compare against it instead of against the self-derived `root_hash()`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root
}
```

The current `root_hash()` helper can remain for convenience, but `valid()` must compare against an externally supplied trusted root, not the root embedded in the proof itself.

### Proof of Concept
```rust
use chia_datalayer::{Hash, Side, calculate_internal_hash};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

// Attacker claims key with hash [0x01; 32] is in the tree
let node_hash: Hash = [0x01; 32];
let other_hash: Hash = [0x02; 32];

// Compute a self-consistent combined_hash for the single layer
let combined_hash = calculate_internal_hash(&node_hash, Side::Left, &other_hash);

let forged_proof = ProofOfInclusion {
    node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Left,
        other_hash,
        combined_hash,
    }],
};

// Returns true even though this proof was never generated from any real MerkleBlob
assert!(forged_proof.valid());
// forged_proof.root_hash() == combined_hash, which is attacker-controlled
```

The forged proof passes `valid()` because after the loop `existing_hash == combined_hash` and `self.root_hash()` also returns `combined_hash` — the final equality check is a tautology.
