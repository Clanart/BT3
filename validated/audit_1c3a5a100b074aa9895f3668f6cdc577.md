### Title
Stale Pending Block Fallback Zeroes `l1_data_gas_price` and `l2_gas_price` in RPC Fee Estimation — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When the in-memory pending block is stale (its `parent_block_hash` does not match the latest committed block), `read_pending_data` constructs a `DeprecatedPendingBlock` fallback that intentionally omits `l1_data_gas_price` and `l2_gas_price`. Both fields default to zero. Every subsequent call to `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_traceBlockTransactions` with `block_id = "pending"` during this window executes against a `BlockContext` whose `l1_data_gas_price` and `l2_gas_price` are clamped to `NonzeroGasPrice::MIN` (= 1), producing authoritative-looking but wrong fee estimates and wrong `overall_fee` values.

### Finding Description

**Root cause — `read_pending_data` fallback path:** [1](#0-0) 

When `pending_data.block.parent_block_hash() != latest_header.block_hash`, the function returns a synthetic `DeprecatedPendingBlock`. It copies `l1_gas_price` (wei + fri) from the latest header but leaves everything else as `Default::default()`. `DeprecatedPendingBlock` has no `l1_data_gas_price` or `l2_gas_price` fields at all: [2](#0-1) 

**Accessor methods return zero for `Deprecated` variant:** [3](#0-2) 

**Conversion to execution `PendingData` propagates the zeros:** [4](#0-3) 

**`BlockContext` construction clamps zero to `NonzeroGasPrice::MIN`:** [5](#0-4) 

`NonzeroGasPrice::new(GasPrice(0))` returns `None`, so `.unwrap_or(NonzeroGasPrice::MIN)` silently substitutes `1` for both `l1_data_gas_price` and `l2_gas_price` in wei and fri.

**Fee estimation reads these wrong prices directly into the response:** [6](#0-5) 

The `FeeEstimation` struct returned to the caller contains `l1_data_gas_price = 1`, `l2_gas_price = 1`, and an `overall_fee` computed with those wrong prices.

**Trigger path:**

`starknet_estimateFee` → `read_pending_data` (stale fallback) → `client_pending_data_to_execution_pending_data` → `create_block_context_for_execution` → `exec_estimate_fee` → `tx_execution_output_to_fee_estimation`. [7](#0-6) 

### Impact Explanation

`starknet_estimateFee` and `starknet_simulateTransactions` return an authoritative-looking `FEE_ESTIMATE` object with `l1_data_gas_price` and `l2_gas_price` set to `1` (fri or wei) and an `overall_fee` computed from those values. On mainnet, the actual `l2_gas_price` is on the order of `10^9`–`10^10` fri. A user who trusts the estimate and sets `resource_bounds.l2_gas.max_price_per_unit` accordingly will have their transaction rejected at execution time with `MaxGasPriceTooLow`. Conversely, a user who adds a safety multiplier on top of the wrong estimate may overpay by orders of magnitude. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

The stale-pending-data window occurs on every block transition: the moment a new block is committed to storage, the in-memory `pending_data` (which still points to the previous block's hash) becomes stale until the pending sync loop refreshes it. This is a normal, recurring condition, not an edge case. Any unprivileged caller who issues `starknet_estimateFee` with `block_id = "pending"` during this window receives wrong data. No special privileges or adversarial setup are required.

### Recommendation

In the `read_pending_data` fallback, construct a `PendingBlockOrDeprecated::Current(PendingBlock)` instead of a `DeprecatedPendingBlock`, copying `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode` from the latest committed block header alongside `l1_gas_price`. This ensures all three gas prices are populated correctly even when the pending sync has not yet caught up to the latest block.

```rust
// Fallback: pending data is stale — synthesise from latest committed header
Ok(PendingData {
    block: PendingBlockOrDeprecated::Current(PendingBlock {
        parent_block_hash: latest_header.block_hash,
        l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
        l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price,
        l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,
        l1_da_mode: latest_header.block_header_without_hash.l1_da_mode,
        timestamp: latest_header.block_header_without_hash.timestamp,
        sequencer_address: latest_header.block_header_without_hash.sequencer,
        starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
        ..Default::default()
    }),
    state_update: ClientPendingStateUpdate {
        old_root: latest_header.block_header_without_hash.state_root,
        state_diff: Default::default(),
    },
})
```

### Proof of Concept

1. Commit block N to storage (latest header has `l2_gas_price.price_in_fri = 10_000_000_000`).
2. Do **not** update the in-memory `pending_data` (simulate the sync lag — `pending_data.block.parent_block_hash()` still points to block N-1).
3. Call `starknet_estimateFee([invoke_v3_tx], [], "pending")`.
4. `read_pending_data` detects the hash mismatch and returns a `DeprecatedPendingBlock` fallback.
5. `PendingBlockOrDeprecated::l2_gas_price()` returns `GasPricePerToken { price_in_wei: 0, price_in_fri: 0 }`.
6. `create_block_context_for_execution` clamps `0` → `NonzeroGasPrice::MIN = 1`.
7. The returned `FEE_ESTIMATE` contains `l2_gas_price = 1` and `overall_fee` computed at price `1` instead of `10_000_000_000`.
8. A V3 transaction submitted with `max_price_per_unit` derived from this estimate is rejected by the sequencer with `MaxGasPriceTooLow`. [8](#0-7) [9](#0-8) [10](#0-9)

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
