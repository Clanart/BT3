### Title
`ThinStateDiff::from(CommitmentStateDiff)` Hardcodes Empty `deprecated_declared_classes`, Producing Wrong `state_diff_commitment` and Block Hash for Blocks Containing Deprecated Declare Transactions - (File: crates/blockifier/src/state/cached_state.rs)

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally sets `deprecated_declared_classes: Vec::new()`. When a block contains a deprecated (Cairo 0) Declare transaction, the `ThinStateDiff` fed into `calculate_block_commitments` is missing those class hashes, producing a wrong `state_diff_commitment` and a wrong block hash. The `state_diff_length` used in `concat_counts` is also under-counted, corrupting the same block hash field a second time.

### Finding Description

`CommitmentStateDiff` has no `deprecated_declared_classes` field:

```rust
pub struct CommitmentStateDiff {
    pub address_to_class_hash: IndexMap<ContractAddress, ClassHash>,
    pub address_to_nonce: IndexMap<ContractAddress, Nonce>,
    pub storage_updates: IndexMap<ContractAddress, IndexMap<StorageKey, Felt>>,
    pub class_hash_to_compiled_class_hash: IndexMap<ClassHash, CompiledClassHash>,
}
``` [1](#0-0) 

The `From` impl therefore hardcodes the field to an empty `Vec`:

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            ...
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),
        }
    }
}
``` [2](#0-1) 

`BlockExecutionArtifacts::new` calls this conversion twice: once to feed `calculate_block_commitments` (which computes `state_diff_commitment` and `state_diff_length`), and once via `thin_state_diff()` which is the value committed to storage and returned in `DecisionReachedResponse`:

```rust
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
).await;
``` [3](#0-2) 

```rust
pub fn thin_state_diff(&self) -> ThinStateDiff {
    ThinStateDiff::from(self.commitment_state_diff.clone())  // deprecated_declared_classes = []
}
``` [4](#0-3) 

`calculate_state_diff_hash` explicitly chains `deprecated_declared_classes` into the Poseidon hash:

```rust
hash_chain = chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
``` [5](#0-4) 

`ThinStateDiff::len()` also counts `deprecated_declared_classes.len()`, so `state_diff_length` fed into `concat_counts` is under-counted:

```rust
result += self.deprecated_declared_classes.len();
``` [6](#0-5) 

The `blockifier_reexecution` crate explicitly acknowledges this gap: *"Blocks before v0.14.0 may include deprecated (Cairo 0) declared classes which are not represented in `CommitmentStateDiff`; those blocks skip hash comparison below."* [7](#0-6) 

### Impact Explanation

When a deprecated Declare transaction is included in a sequenced block:

1. `state_diff_commitment` stored in `PartialBlockHashComponents` is wrong — it hashes `deprecated_declared_classes = []` instead of the actual declared class hashes.
2. `state_diff_length` in `concat_counts` is wrong — it is smaller than the true length by the number of deprecated declared classes.
3. Both fields feed `calculate_block_hash`, so the final block hash is wrong.
4. The `ThinStateDiff` written to storage via `append_state_diff` is also missing the deprecated declared classes, corrupting the persistent state diff record.
5. The SNOS/prover, which computes the correct state diff, would produce a proof over a different `state_diff_commitment` than what the sequencer committed, breaking proof verification.

This matches: **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

Deprecated Declare transactions (Cairo 0, v0/v1) remain protocol-valid. The storage, sync, and central-sync layers all contain live handling for `deprecated_declared_classes`. An unprivileged user can submit such a transaction through the public gateway. The trigger requires no special privilege. Likelihood is **Medium** because deprecated Declare transactions are rare in practice on modern Starknet but are not blocked at the protocol level.

### Recommendation

`CommitmentStateDiff` must be extended with a `deprecated_declared_classes: Vec<ClassHash>` field, populated from `StateMaps::declared_contracts` (filtering entries where the class has no compiled class hash, i.e., Cairo 0 classes). The `From<CommitmentStateDiff> for ThinStateDiff` impl should then propagate this field instead of hardcoding `Vec::new()`. The TODO comment at line 698–699 of `cached_state.rs` should be resolved as part of this fix.

### Proof of Concept

1. Submit a deprecated Declare transaction (Cairo 0 class) to the sequencer gateway.
2. The sequencer executes it; `StateMaps::declared_contracts` records `{class_hash: true}`.
3. `CommitmentStateDiff::from(StateMaps)` drops `declared_contracts`; `deprecated_declared_classes` is absent.
4. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff)` → `deprecated_declared_classes = []`.
5. `calculate_state_diff_hash` hashes `[0]` for the deprecated classes segment instead of `[1, class_hash]`.
6. `state_diff_commitment` in `PartialBlockHashComponents` is wrong.
7. `calculate_block_hash` produces a wrong block hash.
8. The SNOS computes the correct `state_diff_commitment` (including the deprecated class), producing a proof that does not match the committed block hash — proof verification fails.
