### Title
`HyperliquedBridgeToken` 3-arg `mint` permanently locks recipient funds at `_systemAddress` when `finTransfer` is called with non-empty `message` — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`OmniBridge.finTransfer` dispatches to the 3-arg `IBridgeToken.mint(address, uint256, bytes)` whenever `payload.message.length > 0` and the token is a registered bridge token. `HyperliquedBridgeToken` overrides this function to first mint tokens to the recipient and then immediately transfer the full minted amount to `_systemAddress` via `_update`. The net effect is that the recipient receives zero tokens and the entire minted amount is permanently parked at `_systemAddress`. Because the `message` field is user-controlled at transfer initiation and is included verbatim in the MPC-signed payload, any unprivileged actor can trigger this path.

---

### Finding Description

**Dispatch logic in `OmniBridge.finTransfer` (lines 337–349):** [1](#0-0) 

When `isBridgeToken[payload.tokenAddress]` is true and `payload.message.length > 0`, the bridge calls:

```solidity
IBridgeToken(payload.tokenAddress).mint(
    payload.recipient,
    payload.amount,
    payload.message   // attacker-controlled bytes
);
```

**`HyperliquedBridgeToken` override (lines 76–83):** [2](#0-1) 

```solidity
function mint(address account, uint256 value, bytes memory) external override onlyOwner {
    _mint(account, value);              // mints `value` to `account`
    _update(account, _systemAddress, value); // immediately moves all of it to _systemAddress
}
```

`_update` is the internal ERC-20 transfer hook (OpenZeppelin `ERC20Upgradeable`). Calling `_update(account, _systemAddress, value)` transfers the entire just-minted balance from `account` to `_systemAddress`. The recipient ends with a net balance change of **zero**; `_systemAddress` absorbs the full amount.

**The `message` field is attacker-controlled and MPC-signed as-is:** [3](#0-2) 

The borsh encoding includes `payload.message` only when non-empty, but the NEAR bridge signs whatever message the originating user provides. Any user initiating a cross-chain transfer from NEAR can supply an arbitrary non-empty `message` string; the MPC cluster signs the resulting payload without inspecting the message content.

---

### Impact Explanation

- Tokens are minted to `payload.recipient` and immediately drained to `_systemAddress` in the same transaction.
- The recipient receives zero tokens despite a valid, fully-verified cross-chain transfer.
- `_systemAddress` is a protocol-internal address with no withdrawal path for arbitrary ERC-20 balances deposited this way; funds are permanently locked.
- The originating NEAR-side tokens have already been burned/locked, so the loss is unrecoverable on both sides.

Impact: **Critical — permanent, irrecoverable lock of user funds in the bridge token flow.**

---

### Likelihood Explanation

- No privileged role is required. Any user who initiates a NEAR→EVM transfer with a non-empty `message` field triggers the path.
- The `message` parameter is a standard, documented feature of `initTransfer` and is expected to be used for cross-chain messaging.
- The MPC cluster signs the payload unconditionally; it does not validate whether the destination token's 3-arg mint is safe to call.
- `HyperliquedBridgeToken` is a production contract in scope, not a mock.

---

### Recommendation

1. **Remove the 3-arg `mint` dispatch from `finTransfer`** entirely, or gate it behind an explicit per-token allowlist that only permits tokens whose 3-arg mint is safe for the bridge finalization path.
2. **Fix `HyperliquedBridgeToken.mint(address, uint256, bytes)`** so it does not call `_update` to `_systemAddress` when invoked by the bridge owner during finalization. The HyperCore parking behavior should be triggered only through the dedicated `coreReceiveWithData` path, not through the bridge's `finTransfer` path.
3. As a defense-in-depth measure, `finTransfer` should validate that calling the 3-arg mint does not alter the recipient's net balance (e.g., check balance before/after), or simply always use the 2-arg mint and handle message delivery separately.

---

### Proof of Concept

```solidity
// Setup
HyperliquedBridgeToken token = new HyperliquedBridgeToken();
token.initialize("T", "T", 18, systemAddress, hyperCoreDeployer);
// Register as bridge token (done by admin at deployment)
bridge.isBridgeToken[address(token)] = true;

// Attacker initiates transfer from NEAR with non-empty message
// NEAR bridge signs: { ..., tokenAddress: token, amount: 1000, recipient: alice, message: "x" }

// Relayer (or attacker) submits finTransfer with the signed payload
bridge.finTransfer(validSignature, {
    tokenAddress: address(token),
    amount: 1000,
    recipient: alice,
    message: "x",   // non-empty — triggers 3-arg mint
    ...
});

// Result:
assert(token.balanceOf(alice) == 0);           // alice gets nothing
assert(token.balanceOf(systemAddress) == 1000); // funds permanently locked
```

The `message` bytes `"x"` are included in the MPC-signed borsh payload, so the signature check passes. The 3-arg mint fires, mints 1000 to `alice`, then `_update(alice, _systemAddress, 1000)` moves all 1000 to `_systemAddress`. Alice's funds are permanently lost.
