### Title
`PropellerUnit` Signature Does Not Bind Channel, Enabling Cross-Channel Replay — (`crates/apollo_propeller/src/signature.rs`)

---

### Summary

The Propeller protocol's `sign_message_id` function signs only the raw `MessageRoot` bytes (the Merkle root of the erasure-coded shards), without binding the `channel` field. Because `channel` is carried in the `PropellerUnit` wire message but is absent from the signed payload, any network peer who receives a valid unit can strip the original channel value, substitute a different channel ID, and rebroadcast the unit. Signature verification passes unchanged. The codebase itself acknowledges the gap with a `// TODO(AndrewL): CRITICAL: protect against replay attacks` comment in the proto definition.

---

### Finding Description

**Root cause — what is signed**

`sign_message_id` constructs the signed message as:

```
<propeller>  ||  message_root_bytes  ||  </propeller>
``` [1](#0-0) 

`verify_message_id_signature` reconstructs the same three-part concatenation and verifies against it: [2](#0-1) 

The `channel` field (`Channel(u32)`) present in every `PropellerUnit` is never included in this signed payload.

**Wire format — channel is a mutable, unsigned field**

The proto definition carries `channel` as a plain `uint32` alongside the signature: [3](#0-2) 

The developer comment on line 35 explicitly flags the replay risk.

**Signing path — channel is ignored**

`create_units_to_publish` calls `sign_message_id` once over the `message_root` and then stamps the same signature onto every shard unit, with the `channel` value attached but not covered: [4](#0-3) 

**Verification path — channel is not re-checked against the signature**

`UnitValidator::verify_signature` calls `verify_message_id_signature` with only `unit.root()` (the `MessageRoot`). The `channel` stored in `self.channel` is asserted equal to `unit.channel()` via a Rust `assert_eq!` (panic, not a validation error), but this assertion is only a consistency check within a single `MessageProcessor` session — it does not prevent a fresh session from being opened for a different channel carrying the same signature: [5](#0-4) 

**Deduplication does not cover cross-channel replay**

The engine's `finalized_messages` `TimeCache` keys on `MessageKey { channel, publisher, root }`. A replayed unit with a different `channel` value produces a distinct key and bypasses the cache entirely: [6](#0-5) [7](#0-6) 

The engine also carries a second TODO acknowledging the gap: [8](#0-7) 

**End-to-end exploit path**

1. Attacker is a normal network peer — no privileged keys required.
2. Publisher P broadcasts a block proposal on channel C1 (e.g., height N, round 0). Each `PropellerUnit` carries `signature = sign(<propeller> || merkle_root || </propeller>)`.
3. Attacker receives one or more of these units.
4. Attacker changes the `channel` field to C2 (e.g., height N, round 1, or height M) and rebroadcasts the units to other peers.
5. Receiving peers find no entry for `{channel: C2, publisher: P, root: merkle_root}` in `finalized_messages`, so they spawn a fresh `MessageProcessor`.
6. `UnitValidator::verify_signature` verifies the signature against `merkle_root` — passes, because `channel` was never signed.
7. Enough shards accumulate; the message is reconstructed and `Event::MessageReceived { publisher: P, message_root, message }` is emitted for channel C2.
8. The consensus orchestrator receives a valid-looking block proposal attributed to channel C2 (a different height or round than the original).

---

### Impact Explanation

The Propeller protocol is the data-dissemination layer for consensus block proposals. Channels map to consensus committee/round contexts. Injecting a reconstructed proposal into the wrong channel causes the consensus layer to evaluate a stale or misattributed block proposal as if it were a fresh one for that channel. Depending on how the orchestrator binds channel identity to block height and round, this can:

- Force consensus peers to waste resources processing a replayed proposal (resource exhaustion / liveness degradation).
- If the orchestrator does not independently re-validate the height/round embedded in the proposal bytes against the channel, cause an incorrect `ProposalCommitment` to be accepted, leading to wrong state root, wrong block hash, or wrong transaction set being committed — matching the "Wrong state … from blockifier/syscall/execution logic" critical impact category.

The `Event::MessageReceived` payload does not carry the channel, so the application layer has no channel context with which to reject the replayed delivery: [9](#0-8) 

---

### Likelihood Explanation

The attacker needs only to be a connected peer — a role available to any participant in the p2p network. No private keys, no governance access, and no cryptographic break are required. The attacker needs to observe a single valid `PropellerUnit` (which is broadcast to all channel members) and retransmit it with a modified `channel` field. The window of opportunity is bounded only by `stale_message_timeout`, which is a configurable duration, not a single block.

---

### Recommendation

Include the `channel` (and ideally the block height / round) in the signed payload so that a signature is bound to exactly one context:

```rust
// In sign_message_id / verify_message_id_signature, replace:
let msg = [SIGNING_PREFIX, &message_id.0, SIGNING_POSTFIX].concat();

// With something like:
let channel_bytes = channel.0.to_le_bytes();
let msg = [SIGNING_PREFIX, &channel_bytes, &message_id.0, SIGNING_POSTFIX].concat();
```

Correspondingly, pass `channel` into both `sign_message_id` and `verify_message_id_signature`, and update `create_units_to_publish` and `UnitValidator::verify_signature` to supply it.

pub fn sign_message_id(
    message_id: &MessageRoot,
    keypair: &Keypair,
) -> Result<Vec<u8>, ShardPublishError> {
    let msg = [SIGNING_PREFIX, &message_id.0, SIGNING_POSTFIX].concat();
    // TODO(AndrewL): Use a transparent error type for this.
    keypair.sign(&msg).map_err(|e| ShardPublishError::SigningFailed(e.to_string()))
}

pub fn verify_message_id_signature(
    message_id: &MessageRoot,
    signature: &[u8],
    public_key: &PublicKey,
) -> Result<(), ShardSignatureVerificationError> {
    if signature.is_empty() {
        return Err(ShardSignatureVerificationError::VerificationFailed);
    }
    let msg = [SIGNING_PREFIX, &message_id.0, SIGNING_POSTFIX].concat();
    let signature_valid = public_key.verify(&msg, signature);
    if signature_valid { Ok(()) } else { Err(ShardSignatureVerificationError::VerificationFailed) }
}

pub fn create_units_to_publish(
    message: Vec<u8>,
    channel: Channel,
    keypair: Keypair,
    num_data_shards: usize,
    num_coding_shards: usize,
) -> Result<Vec<PropellerUnit>, ShardPublishError> {
    let publisher = PeerId::from(keypair.public());

    // The reed-solomon-simd crate requires the shard length to be even.
    let divisor =
        NonZeroUsize::new(2 * num_data_shards).ok_or(ShardPublishError::InvalidDataSize)?;
    let message = pad_message(message, divisor);

    let data_shards = split_data_into_shards(message, num_data_shards)
        .expect("split_data_into_shards on a padded message should always succeed");

    let (all_shards, message_root, merkle_tree) =
        create_coding_shards_and_merkle(data_shards, num_coding_shards)
            .expect("encoding my own data shards should always succeed");

    let signature = signature::sign_message_id(&message_root, &keypair)?;

    let mut messages = Vec::with_capacity(all_shards.len());
    for (index, shard) in all_shards.into_iter().enumerate() {
        let proof = merkle_tree
            .prove(index)
            .expect("index is within bounds of all_shards from which merkle_tree was built");
        let message = PropellerUnit::new(
            channel,
            publisher,
            message_root,
            signature.clone(),
            // TODO(AndrewL): Change ShardIndex to u64 so this narrowing conversion is unnecessary.
            ShardIndex(u32::try_from(index).expect("shard index exceeds u32::MAX")),
            shard,
            proof,
        );
        messages.push(message);
    }

    Ok(messages)
}

    fn verify_signature(
        &mut self,
        unit: &PropellerUnit,
    ) -> Result<(), ShardSignatureVerificationError> {
        if let Some(signature) = &self.verified_signature {
            return if signature == unit.signature() {
                Ok(())
            } else {
                Err(ShardSignatureVerificationError::VerificationFailed)
            };

            unit.signature(),
            &self.publisher_public_key,
        );

        if let Ok(()) = &result {
            self.verified_signature = Some(unit.signature().to_vec());
        }

        result
    }

    pub fn validate_shard(
        &mut self,
        sender: PeerId,
        unit: &PropellerUnit,
    ) -> Result<(), ShardValidationError> {
        // TODO(AndrewL): Think about how to correctly get rid of these assertions
        assert_eq!(self.channel, unit.channel(), "Channel mismatch");
        assert_eq!(self.publisher, unit.publisher(), "Publisher mismatch");
        assert_eq!(self.message_root, unit.root(), "Message root mismatch");

        if self.received_indices.contains(&unit.index()) {
            return Err(ShardValidationError::DuplicateShard);
        }

        self.schedule_manager.validate_origin(sender, unit.publisher(), unit.index())?;
        unit.validate_shard_proof(self.schedule_manager.num_shards())?;

        // add for next time we see this shard
        self.received_indices.insert(unit.index());

        Ok(())
    }
