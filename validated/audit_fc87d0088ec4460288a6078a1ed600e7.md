### Title
Missing Zero-Address Validation for `nearBridgeDerivedAddress` Enables Complete Signature Verification Bypass in `finTransfer` and `deployToken` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.sol`'s `initialize()` function accepts `nearBridgeDerivedAddress_` without a zero-address guard. This address is the sole trusted signer checked in both `finTransfer()` and `deployToken()`. If it is set to `address(0)` at deployment, OpenZeppelin's `ECDSA.recover()` always returns a non-zero address (or reverts), so the inequality `ECDSA.recover(...) != address(0)` is permanently `true`, meaning **any signature from any caller passes the check**. An unprivileged attacker can then mint arbitrary amounts of bridge tokens or release locked assets to any recipient with no authorization.

### Finding Description

`OmniBridge.initialize()` stores `nearBridgeDerivedAddress_` directly without validation:

```solidity
// OmniBridge.sol lines 72-86
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,   // ← no zero-address check
    uint8 omniBridgeChainId_
) public initializer {
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;   // stored as-is
    ...
}
``` [1](#0-0) 

This address is the only authorization gate for two public, permissionless entry points:

**`finTransfer()` (line 311):**
```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

**`deployToken()` (line 151):**
```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

OpenZeppelin's `ECDSA.recover()` reverts when `ecrecover` returns `address(0)` (i.e., on a malformed/zero-recovery signature). For any well-formed ECDSA signature over any message, it returns a **non-zero** address. Therefore, when `nearBridgeDerivedAddress == address(0)`:

- `ECDSA.recover(hashed, signatureData)` → some non-zero address `X`
- `X != address(0)` → **always `true`**
- `revert InvalidSignature()` → **never reached**

The attacker supplies any valid ECDSA signature (e.g., self-signed) and any payload they choose. The check passes unconditionally.

By contrast, `tokenImplementationAddress` already has an explicit zero-address guard at line 139:
```solidity
if (tokenImplementationAddress == address(0)) {
    revert TokenImplementationNotSet();
}
``` [4](#0-3) 

`nearBridgeDerivedAddress` has no equivalent protection. The setter `setNearBridgeDerivedAddress()` also lacks a zero-address check: [5](#0-4) 

### Impact Explanation

If `nearBridgeDerivedAddress` is `address(0)` at deployment:

- **`finTransfer()`**: Attacker constructs any `TransferMessagePayload` (arbitrary `tokenAddress`, `amount`, `recipient`) with a fresh self-signed ECDSA signature. The signature check passes. The contract then mints bridge tokens (`IBridgeToken.mint`), releases locked ERC-20 (`safeTransfer`), or sends native ETH to the attacker-chosen recipient. This is **direct unauthorized mint/release of bridged assets** — Critical impact.
- **`deployToken()`**: Attacker deploys arbitrary bridge token contracts mapped to any NEAR token ID, corrupting the token registry and enabling follow-on minting attacks.

### Likelihood Explanation

The `initialize()` function is called once at proxy deployment. A deployment script error, copy-paste mistake, or misconfigured environment variable passing `address(0)` for `nearBridgeDerivedAddress_` is a realistic operational risk, especially given that no on-chain guard prevents it. The contract provides no post-deployment detection mechanism. Once deployed with a zero address, the bridge is permanently open to unauthorized finalization until an admin calls `setNearBridgeDerivedAddress()` — but only if the misconfiguration is noticed.

### Recommendation

Add a zero-address require in `initialize()` and in `setNearBridgeDerivedAddress()`:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    require(nearBridgeDerivedAddress_ != address(0), "ERR_ZERO_NEAR_BRIDGE_ADDRESS");
    require(tokenImplementationAddress_ != address(0), "ERR_ZERO_TOKEN_IMPL");
    ...
}

function setNearBridgeDerivedAddress(
    address nearBridgeDerivedAddress_
) external onlyRole(DEFAULT_ADMIN_ROLE) {
    require(nearBridgeDerivedAddress_ != address(0), "ERR_ZERO_NEAR_BRIDGE_ADDRESS");
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;
}
```

### Proof of Concept

1. Deploy `OmniBridge` proxy, calling `initialize(validImpl, address(0), chainId)`.
2. Attacker generates a fresh private key `k` and computes `sig = sign(k, keccak256(borshEncode(payload)))` for any desired `TransferMessagePayload` (e.g., `tokenAddress = someLockedToken`, `amount = totalLocked`, `recipient = attacker`).
3. Attacker calls `finTransfer(sig, payload)`.
4. Inside `finTransfer`: `ECDSA.recover(hashed, sig)` returns the attacker's address (non-zero). `attackerAddress != address(0)` is `true`. The `revert InvalidSignature()` branch is skipped.
5. The contract executes the transfer: locked tokens are sent to the attacker, or bridge tokens are minted to the attacker.
6. Repeat with fresh nonces for each additional drain.
