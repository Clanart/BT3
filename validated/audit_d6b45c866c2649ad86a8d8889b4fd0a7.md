### Title
Reverted L1 Handler Transaction Skips `consume_l1_to_l2_message`, Omitting Message from OS Output — (`execution/transaction_impls.cairo`)

---

### Summary

In `execute_l1_handler_transaction`, when the hint `%{ IsReverted %}` marks a transaction as reverted, the function returns immediately without calling `consume_l1_to_l2_message`. This means the consumed L1→L2 message is never written into `outputs.messages_to_l2`. The OS proof therefore does not include the message as consumed, so the L1 verifier contract will not remove it from the on-chain message queue. The message remains claimable/re-includable on L1 while the associated L2 state changes are discarded, creating a permanent inconsistency that can freeze or duplicate funds.

---

### Finding Description

`execute_l1_handler_transaction` begins with an early-return guard:

```cairo
%{ StartTx %}
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();          // ← early return
}
``` [1](#0-0) 

When `is_reverted != FALSE` the function exits before reaching the only call to `consume_l1_to_l2_message`:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [2](#0-1) 

`consume_l1_to_l2_message` is the sole site that appends a `MessageToL2Header` entry to `outputs.messages_to_l2`:

```cairo
assert [outputs.messages_to_l2] = MessageToL2Header(
    from_address=[execution_context.calldata],
    to_address=execution_info.contract_address,
    nonce=nonce,
    selector=execution_info.selector,
    payload_size=payload_size,
);
``` [3](#0-2) 

`outputs.messages_to_l2` is the `OsCarriedOutputs.messages_to_l2` pointer that is serialized verbatim into the OS proof output by `serialize_messages`:

```cairo
let messages_to_l2_segment_size = (
    final_carried_outputs.messages_to_l2 - initial_carried_outputs.messages_to_l2
);
serialize_word(messages_to_l2_segment_size);
relocate_segment(src_ptr=initial_carried_outputs.messages_to_l2, dest_ptr=output_ptr);
``` [4](#0-3) 

The L1 StarkNet core contract reads this segment from the proof to determine which L1→L2 messages were consumed and removes them from the on-chain queue. If a message is absent from the segment, the L1 contract never removes it.

The existing TODO comment in the same function acknowledges the incompleteness of reverted L1 handler handling:

```cairo
// TODO(Yoni): currently, the contract state is not fetched for reverted L1 handlers.
//   Once block hash is supported, we should fetch the contract state for them as well.
``` [5](#0-4) 

This confirms the reverted-L1-handler path is intentionally simplified, but the omission of `consume_l1_to_l2_message` is an unintended consequence that breaks the protocol invariant.

---

### Impact Explanation

**Critical — Permanent freezing of funds / Direct loss of funds.**

In the StarkNet protocol, an L1→L2 message is considered consumed the moment the L1 handler is included in a proven block, regardless of whether the handler's execution succeeds or reverts. The L1 message queue entry must be removed from the L1 contract in either case.

When the OS omits the message from its output for a reverted handler:

1. The L1 verifier does not remove the message from the queue.
2. The message remains permanently in the queue but its nonce has been used on L2 (or the L2 state is otherwise inconsistent), so it cannot be legitimately re-executed.
3. Any ETH or ERC-20 tokens locked on L1 as part of the message (e.g., a deposit bridging ETH to L2) are permanently frozen — the L1 contract holds the funds, the message is never cleared, and no valid proof can ever consume it again.

Alternatively, if the sequencer can craft a scenario where the same message is re-included in a later block (because the L1 contract still considers it unconsumed), double-execution of the handler becomes possible, leading to direct duplication of bridged funds on L2.

---

### Likelihood Explanation

L1 handler transactions can legitimately revert whenever:
- The target L2 contract does not implement the expected handler selector.
- The handler's execution runs out of gas.
- The handler's Cairo logic explicitly reverts (e.g., a failed assertion).

All of these are reachable by any unprivileged L1/L2 message sender simply by sending a message to a contract whose handler is known to fail, or by sending a message with a payload that causes the handler to revert. The `%{ IsReverted %}` hint is set by the sequencer based on actual execution outcome, so it is triggered by real on-chain conditions, not by privileged operator action.

---

### Recommendation

Move `consume_l1_to_l2_message` (and the surrounding context setup required to call it) **before** the `is_reverted` early-return guard, so that the message is always recorded in `outputs.messages_to_l2` regardless of execution outcome. The execution and state-change steps can still be skipped for reverted transactions, but the message consumption must be unconditional:

```diff
  %{ StartTx %}
+ // Always set up execution context and consume the L1→L2 message,
+ // regardless of revert status, to maintain the protocol invariant.
+ local entry_point_selector;
+ %{ TxEntryPointSelector %}
+ let (local tx_execution_context: ExecutionContext*) = get_invoke_tx_execution_context(...);
+ ...
+ consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);

  local is_reverted;
  %{ IsReverted %}
  if (is_reverted != FALSE) {
      %{ EndTx %}
      return ();
  }

- // (existing execution path below, minus consume_l1_to_l2_message)
+ // (existing execution path below)
```

### Proof of Concept

1. An L1 user calls the StarkNet L1 core contract's `sendMessageToL2`, locking 1 ETH and targeting an L2 contract whose `handle_deposit` entry point always reverts (e.g., `assert 1 = 0`).
2. The sequencer includes the resulting L1 handler transaction in a block. Because the handler reverts, `%{ IsReverted %}` is set to a non-zero value.
3. `execute_l1_handler_transaction` hits the early-return at line 387–390 and returns without calling `consume_l1_to_l2_message`.
4. `outputs.messages_to_l2` is not advanced; the message is absent from the OS proof output.
5. The L1 verifier processes the proof and does not remove the message from the on-chain queue.
6. The 1 ETH remains locked in the L1 contract with no valid mechanism to release it — the message cannot be re-proven (the L2 nonce/state is already past it) and cannot be cancelled (StarkNet L1→L2 messages are not cancellable once included in a proven block). The funds are permanently frozen.
