### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS omits a critical validation: it does not verify that the replacement class hash is actually declared before permanently writing it to the contract state. This is structurally analogous to the reported "Insecure Workflow Whitelisting" class — a state change that should be gated by an authorization check is instead applied unconditionally and persists permanently. An attacker who controls a contract can exploit this to permanently freeze any funds held by that contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function (lines 877–916) contains an explicit TODO acknowledging the missing check:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `class_hash` value comes directly from `request.class_hash` — fully attacker-controlled — and is written unconditionally into `contract_state_changes` with no check against `contract_class_changes` or `compiled_class_facts_bundle`. The change is then committed to the global state trie via `state_update` in `os.cairo`.

The intended invariant (enforced elsewhere in the OS) is that a contract's class hash must correspond to a declared, compiled class. In `execute_entry_point`, the OS resolves the class hash through two lookups:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
```

If the class hash set via `replace_class` is undeclared, `dict_read` returns 0 (the dict default), and `find_element` — which asserts the element exists — will fail to locate a compiled class for hash `0`. Every future call to the contract becomes unprovable or reverts, permanently locking any funds the contract holds.

The vulnerability class is a **state-transition bypass**: a permanent, irreversible state mutation is applied without the authorization check (declared class requirement) that is supposed to gate it. This is directly analogous to the reported pattern where a permission is written to state and never cleaned up.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class` is called with an undeclared class hash, the contract's class hash in the global state trie is permanently set to an invalid value. All subsequent entry-point calls to the contract fail at the OS level. There is no recovery path: the contract cannot call `replace_class` again (its own entry points are unreachable), and no external actor can reset the class hash. Any ERC-20 tokens, ETH, or other assets held by the contract are permanently inaccessible.

---

### Likelihood Explanation

The attack requires only that the adversary:
1. Deploy a contract whose code includes a call to `replace_class` (an unprivileged, permissionless action on StarkNet).
2. Attract user funds to the contract (e.g., by presenting it as a vault, DEX pool, or other DeFi primitive).
3. Invoke the function that triggers `replace_class` with an undeclared class hash.

No privileged role, leaked key, or trusted operator is required. The `replace_class` syscall is available to any contract. The missing OS-level validation is the necessary vulnerable step: without it, the sequencer/prover cannot reject the state transition, and the broken class hash is committed to the canonical state.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, validate that `class_hash` exists as a key in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for the requested `class_hash` and assert the returned compiled class hash is non-zero. This mirrors the check already implicitly required by `execute_entry_point` and closes the gap acknowledged by the existing TODO comment.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts deposits and exposes an `owner_exit()` function. The `owner_exit()` function calls the `replace_class` syscall with a felt value that has never been declared as a class hash (e.g., `0xdead`).
2. Users deposit funds into `MaliciousVault`, trusting its published source code.
3. Attacker calls `owner_exit()`. The OS executes `execute_replace_class`:
   - `request.class_hash = 0xdead` (undeclared).
   - No validation is performed (the TODO check is absent).
   - `dict_update` writes `class_hash=0xdead` into `contract_state_changes`.
4. `state_update` commits the change; the global state trie now records `MaliciousVault.class_hash = 0xdead`.
5. Any subsequent transaction calling `MaliciousVault` reaches `execute_entry_point`:
   - `dict_read(contract_class_changes, 0xdead)` → returns `0` (undeclared).
   - `find_element(..., key=0)` → no compiled class with hash `0` exists → assertion failure / unprovable execution.
6. All user funds in `MaliciousVault` are permanently frozen with no recovery path.
