### Title
Committed Nonce Expiry in `MempoolState::commit` Allows Replay of Previously-Executed Transactions at the Gateway/Mempool Admission Layer - (File: crates/apollo_mempool/src/mempool.rs)

### Summary

The mempool's `MempoolState` deliberately removes committed nonce records after `committed_nonce_retention_block_count` blocks. Once the record is gone, the mempool falls back to the caller-supplied `incoming_account_nonce` for replay-protection. A caller who supplies a stale (low) `incoming_account_nonce` can therefore re-submit a transaction whose nonce was already executed and committed, and the mempool will accept it — classifying it as a valid future transaction rather than a replay.

### Finding Description

`MempoolState` tracks committed nonces in a bounded sliding window (`CommitHistory`) of size `committed_nonce_retention_block_count` (default 100, production config also 100). [1](#0-0) 

Each call to `MempoolState::commit` pushes the new block's nonces into the deque and pops the oldest entry. If the popped entry's nonce equals the current `committed` value for that address, the address is **removed** from `self.committed`: [2](#0-1) 

After removal, `resolve_nonce` finds no entry in `staged` or `committed` and falls back to `incoming_account_nonce` — the value supplied by the caller at submission time: [3](#0-2) 

`validate_incoming_tx` then compares `tx_nonce` against this caller-controlled fallback: [4](#0-3) 

The test `committed_account_nonce_cleanup` explicitly documents and confirms this behavior: after the retention window expires, a transaction with nonce 0 that was previously rejected as `NonceTooOld` is accepted: [5](#0-4) 

The gateway's stateful validator reads the on-chain nonce from state sync and passes it as `account_nonce`, but the mempool's own `validate_incoming_tx` / `validate_tx` path uses `incoming_account_nonce` from the `ValidationArgs`, which is the value the gateway obtained from the chain state at the time of the call — not a value the mempool independently verifies against its own committed history once that history has expired. [6](#0-5) 

The gateway's `validate_nonce` only checks that the tx nonce is within `[account_nonce, account_nonce + max_allowed_nonce_gap]`: [7](#0-6) 

If the on-chain state has already advanced (nonce is now N) but the mempool's committed record for that address has expired, a submission with `tx_nonce = old_nonce` and `account_nonce = old_nonce` (a stale value the gateway might return for a contract that has since been re-deployed or whose nonce the gateway reads from a lagging state snapshot) passes both the gateway check and the mempool check.

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid (already-executed) transaction before sequencing.**

A transaction with a nonce that was already committed and executed can be re-admitted to the mempool after the retention window expires. This transaction will then be handed to the batcher/blockifier for execution. The blockifier's `handle_nonce` with `strict_nonce_check = true` (used during actual block execution) will reject it at execution time: [8](#0-7) 

However, the transaction is accepted at the RPC/gateway layer and enters the mempool, which constitutes the "Mempool/gateway/RPC admission accepts invalid transactions" impact. The transaction wastes mempool capacity, propagates over P2P, and may cause misleading RPC responses (e.g., `starknet_getTransactionStatus` showing `RECEIVED`).

### Likelihood Explanation

The default `committed_nonce_retention_block_count` is 100 blocks in both the config schema and production deployment configs: [9](#0-8) [10](#0-9) 

After 100 blocks (~a few minutes on Starknet), any account that has not transacted again will have its committed nonce record evicted. An attacker who previously submitted a transaction for address A at nonce N can, after 100 blocks, re-submit the same transaction with `account_nonce = N` and have it accepted by the mempool. The gateway reads the on-chain nonce from state sync; if the attacker supplies a stale `account_nonce` value (or if the gateway's state reader is slightly behind), the gateway-level check also passes.

### Recommendation

1. **Do not evict committed nonces from the mempool's in-memory state** unless the account has submitted a newer transaction that supersedes the old nonce. The current eviction logic is designed to bound memory, but it creates a replay window.
2. **Alternatively**, when `resolve_nonce` falls back to `incoming_account_nonce` (i.e., no committed or staged record exists), the mempool should query the authoritative on-chain nonce rather than trusting the caller-supplied value.
3. **At minimum**, document the replay window explicitly and ensure the gateway always reads the nonce from the latest committed state, not a potentially stale snapshot, before passing it to the mempool.

### Proof of Concept

The existing test `committed_account_nonce_cleanup` in `crates/apollo_mempool/src/fee_mempool_test.rs` already demonstrates the exact behavior: [11](#0-10) 

Step-by-step:
1. Account `0x0` executes a transaction; `commit_block` is called with nonce `1` for `0x0`.
2. Mempool records `committed[0x0] = 1`. A tx with nonce `0` is rejected as `NonceTooOld`.
3. After `committed_nonce_retention_block_count` (2 in the test, 100 in production) empty blocks, `committed[0x0]` is removed.
4. A new submission with `tx_nonce = 0, account_nonce = 0` is now **accepted** by the mempool — the same nonce that was already executed on-chain.

struct CommitHistory {
    commits: VecDeque<AddressToNonce>,
}

impl CommitHistory {
    fn new(capacity: usize) -> Self {
        CommitHistory { commits: std::iter::repeat_n(AddressToNonce::new(), capacity).collect() }
    }

    fn push(&mut self, commit: AddressToNonce) -> AddressToNonce {
        let removed_commit = self.commits.pop_front();
        self.commits.push_back(commit);
        removed_commit.expect("Commit history should be initialized with capacity.")
    }
}

    fn handle_nonce(
        state: &mut dyn State,
        tx_info: &TransactionInfo,
        strict: bool,
    ) -> TransactionPreValidationResult<()> {
        if tx_info.is_v0() {
            return Ok(());
        }

        let address = tx_info.sender_address();
        let account_nonce = state.get_nonce_at(address)?;
        let incoming_tx_nonce = tx_info.nonce();
        let valid_nonce = if strict {
            account_nonce == incoming_tx_nonce
        } else {
            account_nonce <= incoming_tx_nonce
        };
        if valid_nonce {
            return Ok(state.increment_nonce(address)?);
        }
        Err(TransactionPreValidationError::InvalidNonce {
            address,
            account_nonce,
            incoming_tx_nonce,
        })
    }
