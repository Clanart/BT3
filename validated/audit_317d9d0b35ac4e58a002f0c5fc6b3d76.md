### Title
Missing Destination Chain Factory Validation in `init_transfer` Allows Permanent Fund Lock - (File: `near/omni-bridge/src/lib.rs`)

### Summary
The NEAR bridge's `init_transfer` function only checks that the destination chain is not `ChainKind::Near`, but does not verify that a factory is registered for the destination chain or that the token is deployed there. A user can lock tokens targeting a syntactically valid but unsupported `ChainKind` (e.g., `Fogo`, `Abs`, `HyperEvm` if not yet deployed), causing the transfer to be permanently stuck in `pending_transfers` with no user-accessible cancel path.

### Finding Description

`init_transfer` in `near/omni-bridge/src/lib.rs` performs only one destination-chain check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

After this check passes, the transfer is immediately stored in `pending_transfers` and a `destination_nonce` is consumed:

```rust
let destination_nonce =
    self.get_next_destination_nonce(init_transfer_msg.get_destination_chain());
``` [2](#0-1) 

The `ChainKind` enum contains 13 variants — `Eth`, `Near`, `Sol`, `Arb`, `Base`, `Bnb`, `Btc`, `Zcash`, `Pol`, `HyperEvm`, `Strk`, `Abs`, `Fogo` — all of which are accepted by the JSON deserializer: [3](#0-2) 

Not all of these chains necessarily have a registered factory in `self.factories` or a deployed token in `self.token_id_to_address`. When a relayer later calls `sign_transfer` for such a transfer, it panics unconditionally:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [4](#0-3) 

There is no `cancel_transfer` or user-accessible refund function in the contract. The transfer remains in `pending_transfers` indefinitely and the user's tokens are irrecoverably locked.

The `Contract` struct stores `factories: LookupMap<ChainKind, OmniAddress>` and `token_id_to_address: LookupMap<(ChainKind, AccountId), OmniAddress>`, but neither is consulted at `init_transfer` time: [5](#0-4) 

### Impact Explanation

A user who initiates a NEAR-side transfer to a `ChainKind` that has no registered factory (e.g., `Fogo` or `Abs` if those chains are not yet live) will have their tokens permanently locked in the NEAR bridge contract. The `pending_transfers` entry cannot be cancelled by the user, and `sign_transfer` will always panic for that entry. This constitutes an irrecoverable lock of user funds matching the **Critical** impact class: "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."

### Likelihood Explanation

The `ChainKind` enum is part of the public type system and its variants are valid JSON inputs. A user who mistypes a chain name, uses a chain that was recently added to the enum but not yet deployed, or is confused about which chains are live will trigger this path. The `ft_transfer_call` entry point is fully permissionless. No privileged access is required. Likelihood is **Medium**: accidental misuse is realistic, and the enum already contains chains (`Fogo`, `Abs`, `HyperEvm`) that may not have factories registered on all deployments.

### Recommendation

Add a factory-existence check at the start of `init_transfer`, before any state mutation or nonce consumption:

```rust
require!(
    self.factories.contains_key(&init_transfer_msg.recipient.get_chain()),
    BridgeError::InvalidRecipientChain.as_ref()
);
```

Optionally also verify that the token is deployed on the destination chain (`self.token_id_to_address.contains_key(&(chain, token_id))`), mirroring the validation already performed in `sign_transfer` but enforced eagerly so no funds are locked before the check.

### Proof of Concept

1. Deploy the NEAR bridge with factories registered only for `Eth` and `Sol`.
2. Call `ft_transfer_call` on any NEP-141 token with `msg` containing:
   ```json
   {"recipient": "fogo:2xNweLHLqbS9YpP3UyaPrxKqgqoC6yPBFyuLxA8qtgr4", "fee": "0", "native_token_fee": "0"}
   ```
3. Observe that `init_transfer` succeeds: tokens are deducted from the user, a `TransferMessage` is inserted into `pending_transfers`, and a `destination_nonce` for `ChainKind::Fogo` is incremented.
4. Attempt to call `sign_transfer` for the resulting `TransferId`. The call panics with `ERR_FAILED_TO_GET_TOKEN_ADDRESS`.
5. Confirm there is no `cancel_transfer` or equivalent function. The user's tokens are permanently locked. [6](#0-5) [7](#0-6)
