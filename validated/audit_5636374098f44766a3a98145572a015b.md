### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS program updates a contract's class hash to an attacker-supplied value without verifying that the new class hash corresponds to a previously declared class. This is structurally identical to the reported "Missing Receipt Token Balance Check" pattern: a state-transition proceeds without an ownership/existence proof on the resource being referenced. A malicious contract can set its own class hash to an arbitrary undeclared value, permanently breaking the contract and freezing any funds held within it.

---

### Finding Description

In `execute_replace_class`, the OS reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no check that the hash exists in the declared-class registry (`contract_class_changes`). The code itself contains an explicit acknowledgment of this gap:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

By contrast, the `execute_declare_transaction` flow enforces that `compiled_class_hash` is non-zero before writing to `contract_class_changes`, and `dict_read` returns `0` for any key that was never declared:

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

When `execute_entry_point` later tries to execute a contract whose class hash was set to an undeclared value, it performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// compiled_class_hash == 0 for undeclared class
let (compiled_class_fact: CompiledClassFact*) = find_element(
    ...
    key=compiled_class_hash,  // key = 0, not found → assertion failure
);
``` [3](#0-2) 

`find_element` raises a Cairo assertion failure when the key is absent, making it impossible to generate a valid proof for any block that includes a call to the corrupted contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and that state is committed to the chain, every subsequent attempt to call the contract causes proof generation to fail at the `find_element` call. The contract becomes permanently uncallable. Any ERC-20 tokens, ETH, STRK, or other assets held in the contract's storage are irrecoverably frozen, with no upgrade or recovery path available.

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

1. Deploy a contract that accepts user deposits (e.g., a vault or DeFi pool).
2. After accumulating user funds, the contract calls the `replace_class` syscall with an arbitrary felt value that was never declared (e.g., `0xdead`).
3. The OS writes the undeclared class hash into `contract_state_changes` and commits it to state.
4. All future calls to the contract fail at proof generation; funds are permanently frozen.

No privileged role, leaked key, or external dependency is required. The only prerequisite is the ability to deploy a contract and call `replace_class` from within it, which is available to any network participant.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., has a non-zero compiled class hash). The check should mirror the validation already present in `execute_entry_point`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This directly closes the gap acknowledged by the existing TODO comment at line 898. [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts token deposits and exposes a `drain()` entry point.
2. Legitimate users deposit funds; `MaliciousVault` accumulates a balance.
3. Attacker calls `drain()`, which internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (never declared).
4. The OS executes `execute_replace_class`: gas check passes, `dict_update` writes `class_hash=0xdeadbeef` into `contract_state_changes`. No declared-class check is performed.
5. The block is proven and the corrupted state is committed on-chain.
6. In the next block, any user attempts to call `MaliciousVault` to withdraw funds.
7. `execute_entry_point` calls `dict_read{dict_ptr=contract_class_changes}(key=0xdeadbeef)` → returns `0`.
8. `find_element(..., key=0)` fails with a Cairo assertion — proof generation aborts.
9. The contract is permanently uncallable; all deposited funds are frozen with no recovery mechanism.
