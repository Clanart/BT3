## Full Trace

**Step 1 — Solana `deploy_token` (the divergence origin)** [1](#0-0) 

`origin_decimals` is captured from the raw payload before clamping; `metadata.decimals` is then clamped to `MAX_ALLOWED_DECIMALS=9`. The `DeployTokenResponse` posts both values verbatim via Wormhole.

**Step 2 — Signature gate** [2](#0-1) 

`deploy_token` requires a valid MPC signature over the `DeployTokenPayload`. This signature is produced by the NEAR bridge's `log_metadata` flow, which reads `ft_metadata` from the NEAR token contract. An attacker who deploys a NEAR token with `decimals=255` in its `ft_metadata` can legitimately trigger `log_metadata` and receive a valid MPC-signed payload with `decimals=255`. No key compromise is required.

**Step 3 — NEAR `bind_token_callback` stores the divergent pair** [3](#0-2) [4](#0-3) 

`add_token` is called with `decimals=9` and `origin_decimals=255`, storing `Decimals { decimals: 9, origin_decimals: 255 }` in `token_decimals` keyed by the Solana mint address. There is no validation that `origin_decimals >= decimals` or that the exponent is bounded.

**Step 4 — Arithmetic functions compute `diff_decimals = 246`** [5](#0-4) 

Both `denormalize_amount` and `normalize_amount` compute:
```rust
let diff_decimals: u32 = (255u8 - 9u8) as u32; // = 246
```

Then evaluate `10_u128.pow(246)`. Since `u128::MAX ≈ 3.4 × 10^38` and `10^246 = 2^246 × 5^246`, and `2^128 | 2^246`, the value `10^246 mod 2^128 = 0`. Rust's `u128::pow` wraps to **0** in release mode.

Consequences:
- **`normalize_amount`** (used in `sign_transfer`, NEAR→Solana path): `amount / 0` → **panic** → every outbound transfer for this token is permanently blocked.
- **`denormalize_amount`** (used in `fin_transfer_callback`, Solana→NEAR path): `amount * 0 = 0` → recipient is minted **0 tokens** regardless of the actual transfer amount. [6](#0-5) [7](#0-6)

### Title
Unbounded `origin_decimals` in Solana `DeployTokenResponse` causes `10^246` u128 overflow, permanently freezing NEAR→Solana transfers and silently zeroing Solana→NEAR transfer amounts — (`solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs`)

### Summary
When a NEAR token with `ft_metadata.decimals=255` is deployed to Solana, the Solana bridge clamps the on-chain mint to 9 decimals but posts `origin_decimals=255` in the Wormhole `DeployTokenResponse`. NEAR's `bind_token` stores `Decimals { decimals: 9, origin_decimals: 255 }` without validation. Both `normalize_amount` and `denormalize_amount` then compute `10_u128.pow(246)`, which wraps to 0 in Rust release mode, causing division-by-zero panics on outbound transfers and silent zero-minting on inbound transfers.

### Finding Description
In `initialize_token_metadata`, `origin_decimals` is saved from the raw payload before clamping:

```rust
let origin_decimals = metadata.decimals;           // 255
metadata.decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, metadata.decimals); // 9
// ...
DeployTokenResponse { decimals: metadata.decimals, origin_decimals }
// posts: decimals=9, origin_decimals=255
```

NEAR's `bind_token_callback` stores this pair verbatim. The scaling functions then compute `diff_decimals = 246` and call `10_u128.pow(246)`. Since `10^246 = 2^246 × 5^246` and `2^128 | 2^246`, the result modulo `2^128` is exactly 0. Division by 0 panics; multiplication by 0 silently returns 0.

### Impact Explanation
- **NEAR→Solana** (`sign_transfer`): `normalize_amount` divides by 0 → panic → all outbound transfers for this token are permanently bricked.
- **Solana→NEAR** (`fin_transfer_callback`): `denormalize_amount` multiplies by 0 → recipient receives 0 tokens → funds sent from Solana are permanently lost.

This is both a permanent fund freeze and a silent fund loss, satisfying Critical impact.

### Likelihood Explanation
The precondition is fully unprivileged: any user can deploy a NEAR NEP-141 token with `decimals=255` in its `ft_metadata` (the standard imposes no upper bound), call `log_metadata` to obtain a legitimate MPC-signed payload, call `deploy_token` on Solana, and call `bind_token` on NEAR. No key compromise, no colluding validators, no privileged access required.

### Recommendation
1. In `initialize_token_metadata`, validate `origin_decimals <= MAX_ALLOWED_DECIMALS` (or equivalently, reject payloads where `origin_decimals > decimals` after clamping) before posting the `DeployTokenResponse`.
2. In `normalize_amount` and `denormalize_amount`, add a checked/saturating exponentiation guard and assert `origin_decimals >= decimals` before computing the exponent.
3. Consider capping `origin_decimals` to a safe maximum (e.g., 38, since `10^38 < u128::MAX`) in the `add_token` path.

### Proof of Concept
```rust
// Rust playground verification
fn main() {
    let diff: u32 = (255u8 - 9u8) as u32; // 246
    let base: u128 = 10_u128.pow(diff);    // wraps to 0 in release mode
    println!("{}", base);                  // prints 0
    // normalize_amount: amount / 0 → panic
    // denormalize_amount: amount * 0 = 0
}
```

Call sequence:
1. Deploy NEAR token with `ft_metadata.decimals = 255`
2. Call `log_metadata` on NEAR bridge → MPC signs payload with `decimals=255`
3. Call `deploy_token` on Solana with signed payload → mint created with `decimals=9`, Wormhole posts `{decimals:9, origin_decimals:255}`
4. Call `bind_token` on NEAR with Wormhole proof → stores `Decimals{decimals:9, origin_decimals:255}`
5. Any `sign_transfer` for this token panics; any `fin_transfer` delivers 0 tokens

    fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount * (10_u128.pow(diff_decimals))
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
