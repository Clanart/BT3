Audit Report

## Title
Unguarded `refundETH` / `sweepToken` / `unwrapWETH9` drain router-held ETH and token residue left by excess-ETH callers â€” (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` deposits only the exact `value` of ETH needed when `nativeBalance >= value`, leaving any surplus on the router with no attribution. Because `refundETH`, `sweepToken`, and `unwrapWETH9` carry no caller check and operate on the router's entire balance, any third party can immediately claim that surplus. A user who calls `exactInputSingle` (or any other payable entry point) directly with `msg.value > amountIn` and without a same-transaction `refundETH` call loses the difference to the first caller who observes the stranded balance.

## Finding Description

**Root cause â€” ETH stranding in `pay`:** [1](#0-0) 

When `nativeBalance >= value`, exactly `value` wei is wrapped and forwarded; `nativeBalance âˆ’ value` remains on the router with no record of who sent it.

**Root cause â€” unguarded drain helpers:** [2](#0-1) 

`refundETH` sends the router's **entire** ETH balance to `msg.sender`. No check ties the caller to the address that originally deposited the funds. [3](#0-2) 

`unwrapWETH9` and `sweepToken` similarly drain the router's full WETH / ERC-20 balance to a **caller-chosen** `recipient`.

**Payable entry point:** [4](#0-3) 

`exactInputSingle` is `payable`, so a user legitimately sends ETH to pay for a WETH-input swap. The `receive()` guard only blocks plain ETH transfers; it does not prevent ETH from arriving via a payable function call. [5](#0-4) 

**Exploit flow:**

1. Victim calls `exactInputSingle{value: 2 ether}(params)` where `params.amountIn = 1_000` and `params.tokenIn = WETH`, without wrapping in a multicall that includes `refundETH`.
2. The pool callback triggers `_justPayCallback` â†’ `pay(WETH, victim, pool, 1_000)`.
3. `nativeBalance (2 ether) >= value (1_000)`: only 1_000 wei is deposited as WETH and forwarded; `2 ether âˆ’ 1_000` remains on the router.
4. Attacker calls `refundETH()` in a separate transaction and receives the entire stranded balance.

The test suite confirms this stranding is real and that the intended mitigation is a same-transaction `refundETH` call inside `multicall`: [6](#0-5) 

The same surface exists in `MetricOmmPoolLiquidityAdder`, which inherits `PeripheryPayments` and exposes the same `refundETH` / `sweepToken` / `unwrapWETH9` helpers: [7](#0-6) 

**Existing guards are insufficient:** The `receive()` guard prevents unsolicited ETH pushes but does nothing to prevent ETH from arriving via payable function calls, and nothing prevents a third party from calling `refundETH` after the fact.

## Impact Explanation
Direct loss of user ETH principal. Any user who calls a payable router entry point with `msg.value > amountIn` and omits the `refundETH` step loses the surplus to the first observer. The loss is unbounded (up to the full `msg.value` sent) and repeatable across every WETH-input swap or liquidity-add transaction. This meets the Sherlock High/Medium threshold for direct loss of user principal.

## Likelihood Explanation
The pattern is easy to trigger accidentally: a user calling `exactInputSingle` directly (not via a frontend that constructs the multicall) with a round-number ETH value will strand the difference. MEV bots routinely monitor for non-zero router balances and can back-run any such transaction in the same block. No special privileges, approvals, or setup are required for the attacker.

## Recommendation
Add a `msg.sender`-binding check to `refundETH`, `sweepToken`, and `unwrapWETH9`, or restrict them to be callable only from within a `multicall` context (e.g., via a reentrancy-style lock that is set only during `multicall` execution). Alternatively, record the depositor address in transient storage when ETH arrives via a payable entry point and enforce that only that address can call the refund helpers in the same transaction.

## Proof of Concept

```solidity
// Foundry test sketch
function test_strandedEthDrainedByAttacker() public {
    uint128 amountIn = 1_000;
    uint256 msgValue = 2 ether; // user sends excess ETH

    // Victim calls exactInputSingle directly (no multicall + refundETH)
    vm.prank(victim);
    router.exactInputSingle{value: msgValue}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // Router now holds msgValue - amountIn ETH
    assertGt(address(router).balance, 0, "ETH stranded on router");

    // Attacker drains it
    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();
    assertGt(attacker.balance, attackerBefore, "attacker drained stranded ETH");
    assertEq(address(router).balance, 0, "router emptied");
}
```
