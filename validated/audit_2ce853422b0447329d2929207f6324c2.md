### Title
Unauthenticated Requester Identity in `SpicePartialDataRequest` Allows Any Peer to Obtain Witness and Receipt-Proof Data for Arbitrary Shards — (File: `chain/client/src/spice/data_distributor_actor.rs`)

---

### Summary

`handle_partial_data_request` in the SPICE data-distribution path serves full `SpiceChunkStateWitness` and `ReceiptProof` data to any `AccountId` supplied in the `requester` field of a `SpicePartialDataRequest` message, without verifying that the requester is an authorized recipient for the requested shard/block. The `requester` field carries no signature and is never cross-checked against the epoch's recipient set. Any peer that can deliver a `SpicePartialDataRequest` to a producer node can name itself (or any other account) as the `requester` and receive the full encoded witness or receipt-proof data for any `(block_hash, shard_id)` pair the producer holds.

---

### Finding Description

`SpicePartialDataRequest` is defined as a plain struct with no signature field:

```rust
pub struct SpicePartialDataRequest {
    pub data_id: SpiceDataIdentifier,
    pub requester: AccountId,
}
``` [1](#0-0) 

`handle_partial_data_request` receives this message, verifies only that **the serving node itself** is a producer of the requested data, then unconditionally sends the full encoded data to whatever `AccountId` appears in `requester`:

```rust
fn handle_partial_data_request(
    &mut self,
    SpicePartialDataRequest { data_id, requester }: SpicePartialDataRequest,
) -> Result<(), Error> {
    ...
    let (_recipients, producers) = self.recipients_and_producers(&data_id, &block)?;
    if !producers.contains(signer.validator_id()) {
        return Err(Error::Other("we do not produce requested data"));
    }
    ...
    // TODO(spice): Check that requester is one of the recipients and implement a
    // lower-priority way for other nodes that aren't validators (e.g. rpc nodes) to get
    // data they require.

    let recipients = HashSet::from([requester]);
    self.network_adapter.send(PeerManagerMessageRequest::NetworkRequests(
        NetworkRequests::SpicePartialData {
            partial_data: SpicePartialData::new(data_id, data.commitment, data.parts, &signer),
            recipients,
        },
    ));
    Ok(())
}
``` [2](#0-1) 

The TODO comment at lines 1023–1025 is an explicit in-code acknowledgement that the recipient authorization check is absent. [3](#0-2) 

For witness requests the producer additionally sends `SpiceChunkContractAccesses` (the set of contract code hashes accessed during chunk execution) to the same unauthenticated `requester`:

```rust
if let SpiceDataIdentifier::Witness { block_hash, shard_id } = &data_id {
    let chunk_id = SpiceChunkId { block_hash: *block_hash, shard_id: *shard_id };
    let accesses = get_contract_accesses(...).expect("...");
    let accesses_msg = SpiceChunkContractAccesses::new(chunk_id, accesses, &signer);
    self.network_adapter.send(PeerManagerMessageRequest::NetworkRequests(
        NetworkRequests::SpiceChunkContractAccesses(vec![requester.clone()], accesses_msg),
    ));
}
``` [4](#0-3) 

The broken invariant is: `recipients_and_producers` computes the authorized recipient set for a given `SpiceDataIdentifier`, but the result (`_recipients`) is discarded and never compared against `requester`. [5](#0-4) 

The existing test `test_requesting_receipts_when_not_validator` explicitly demonstrates and accepts this behavior — a `"not-validator"` account successfully receives full receipt-proof data from a producer: [6](#0-5) 

---

### Impact Explanation

**Broken authorization invariant — unauthorized data exfiltration across shard boundaries.**

The SPICE distribution protocol is designed so that only the epoch-assigned chunk validators for a given `(block_hash, shard_id)` receive the corresponding `SpiceChunkStateWitness`, and only the shard-tracking nodes receive the corresponding `ReceiptProof`. The missing recipient check completely removes this boundary.

Concrete consequences:

1. **Cross-shard witness exfiltration**: A validator assigned to shard A can request and receive the full `SpiceChunkStateWitness` for shard B — including the complete state-transition recording, contract accesses, and all receipts — without being an authorized recipient for shard B.

2. **Receipt-proof exfiltration**: Any peer can obtain `ReceiptProof` data for any `(from_shard_id, to_shard_id)` pair by supplying an arbitrary `data_id` in the request.

3. **Requester field forgery / data redirection**: Because `requester` is an unauthenticated `AccountId`, an attacker can set it to any account. The producer will route the full encoded data (and contract-accesses message) to that account via `send_message_to_account`, effectively redirecting sensitive data to an arbitrary destination.

4. **Contract-access disclosure**: For every witness request, the producer also sends `SpiceChunkContractAccesses` to the unauthenticated `requester`, disclosing which contracts were called during chunk execution.

---

### Likelihood Explanation

The `SpicePartialDataRequest` message is delivered over T1 (validator-to-validator) connections, so the attacker must be a known validator in the current or next epoch. However, being a validator for **any** shard is sufficient — there is no per-shard gating on who may send the request. The attack requires only:

1. Establishing a T1 connection to a producer node (normal validator operation).
2. Sending a `SpicePartialDataRequest` with an arbitrary `data_id` and `requester`.

No cryptographic material, no privileged key, and no special timing is required. The code path is exercised in normal operation and the missing check is explicitly noted as a TODO.

---

### Recommendation

1. **Authenticate the requester**: Add a signature field to `SpicePartialDataRequest` (analogous to how `SpiceContractCodeRequest` carries a signature verified at lines 1114–1124). Verify the signature against the requester's public key before serving any data.

2. **Enforce recipient authorization**: After verifying the signature, check that `requester` is contained in the `recipients` set returned by `recipients_and_producers`. Reject requests from accounts outside that set (or implement a separate, explicitly lower-privilege path for non-recipient nodes).

3. **Remove the TODO and replace with enforcement**: The comment at lines 1023–1025 should become a hard check, not a deferred note.

---

#[cfg_attr(not(feature = "protocol_feature_spice"), ignore)]
fn test_requesting_receipts_when_not_validator() {
    let (genesis, chain) = setup(2, 1);
    let requester_chain = new_chain(&chain, &genesis);

    let block = latest_block(&chain);
    let receipt_proof = new_test_receipt_proof(&block);
    let mut store_update = chain.chain_store.store().store_update();
    save_receipt_proof(&mut store_update, block.hash(), &receipt_proof);
    store_update.commit();

    let producer = producers_of_receipt_proof(&chain, &block, &receipt_proof).swap_remove(0);
    let data_id = SpiceDataIdentifier::ReceiptProof {
        block_hash: *block.hash(),
        from_shard_id: receipt_proof.1.from_shard_id,
        to_shard_id: receipt_proof.1.to_shard_id,
    };
    let to_shard_id = receipt_proof.1.to_shard_id;

    let (outgoing_sc, mut outgoing_rc) = unbounded_channel();
    let mut actor = new_actor_for_account(outgoing_sc, &chain, &producer);

    let requester = AccountId::from_str("not-validator").unwrap();
    actor.handle(SpicePartialDataRequest { data_id, requester: requester.clone() });
    let (partial_data, recipients) = drain_outgoing_partial_data(&mut outgoing_rc).swap_remove(0);
    assert_eq!(recipients, HashSet::from([requester.clone()]));

    let to_shard_uid = {
        let shard_layout = chain.epoch_manager.get_shard_layout(block.header().epoch_id()).unwrap();
        ShardUId::from_shard_id_and_layout(to_shard_id, &shard_layout)
    };

    let (outgoing_sc, mut outgoing_rc) = unbounded_channel();
    let mut actor = ActorBuilder::new(Some(requester))
        .tracked_shards_config(TrackedShardsConfig::Shards(vec![to_shard_uid]))
        .build(outgoing_sc, &requester_chain);

    let message = outgoing_rc.try_recv().unwrap();
    assert_matches!(outgoing_rc.try_recv(), Err(TryRecvError::Empty));
    let OutgoingMessage::ExecutorIncomingUnverifiedReceipts(ExecutorIncomingUnverifiedReceipts {
        block_hash: reconstructed_block_hash,
        receipt_proof: reconstructed_receipt_proof,
    }) = message
    else {
        panic!();
    };
    assert_eq!(block.hash(), &reconstructed_block_hash);
    assert_eq!(receipt_proof, reconstructed_receipt_proof);
}
