### Title
DOS of `DeployToken` via Pre-Created Mint PDA — (`File: solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs`)

---

### Summary

The `DeployToken` instruction initializes a wrapped mint account using `init` with a deterministic PDA derived from the NEAR token address. Because the PDA address is publicly computable and `init` requires the account to have zero lamports, an attacker can permanently block deployment of any bridged token on Solana by sending a single lamport to the mint PDA address before the instruction is executed.

---

### Finding Description

In `deploy_token.rs`, the `mint` account is declared with the `init` constraint:

```rust
#[account(
    init,
    payer = common.payer,
    seeds = [WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes().as_ref()],
    bump,
    mint::decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, data.payload.decimals),
    mint::authority = authority,
)]
pub mint: Box<Account<'info, Mint>>,
``` [1](#0-0) 

The PDA is derived from `[WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes()]`, where `data.payload.token` is the NEAR token address — a value that is publicly known and deterministically computable by anyone.

Anchor's `init` constraint internally calls the system program's `create_account` CPI. The system program rejects `create_account` if the target account already has any lamports (i.e., already exists on-chain). An attacker can therefore:

1. Compute the mint PDA for any NEAR token: `find_program_address([WRAPPED_MINT_SEED, token.to_hashed_bytes()], bridge_program_id)`.
2. Send 1 lamport to that address via a standard SOL transfer. This creates a system-owned account at the PDA address.
3. Every subsequent call to `deploy_token` for that token will fail with an "account already exists" error from the system program.

Because the NEAR token address is public (visible in the NEAR token registry and in pending bridge transactions), an attacker can pre-compute and pre-fund the mint PDAs for all tokens of interest — or even all possible tokens — before any deployment is attempted.

---

### Impact Explanation

`deploy_token` is the instruction that creates the Solana-side wrapped mint for a NEAR token. Without a successful `deploy_token`, the corresponding `finalize_transfer` instruction cannot mint tokens to the recipient (the mint account does not exist). Users who have already locked tokens on the NEAR side expecting to receive bridged tokens on Solana will have their funds permanently frozen with no on-chain recovery path on the Solana side.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge token flows.** [2](#0-1) 

---

### Likelihood Explanation

- The NEAR token address used to derive the mint PDA is fully public.
- The attacker does not need to interact with the bridge program at all — a plain SOL transfer suffices.
- A single lamport per token is sufficient to block deployment indefinitely.
- The attack can be performed proactively for all known NEAR tokens before any bridge activity occurs, or reactively by front-running a specific `deploy_token` transaction observed in the mempool.
- There is no on-chain mechanism to "un-block" the PDA once it has lamports, because the bridge program cannot close a system-owned account it did not create.

Likelihood: **Medium–High**.

---

### Recommendation

Replace `init` with `init_if_needed` on the `mint` account, consistent with how other accounts in this codebase handle the same pattern (e.g., `vault` in `log_metadata.rs` and `token_account` in `finalize_transfer.rs`):

```diff
 #[account(
-    init,
+    init_if_needed,
     payer = common.payer,
     seeds = [WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes().as_ref()],
     bump,
     mint::decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, data.payload.decimals),
     mint::authority = authority,
 )]
 pub mint: Box<Account<'info, Mint>>,
``` [3](#0-2) [4](#0-3) 

Additionally, add a post-initialization check to verify that the mint's `authority` and `decimals` match the expected values from the payload, to prevent a scenario where a pre-existing mint with different parameters is silently accepted.

---

### Proof of Concept

```
1. Observe (or pre-compute) the mint PDA for NEAR token "usdc.near":
   mint_pda = find_program_address(
       [WRAPPED_MINT_SEED, "usdc.near".to_hashed_bytes()],
       bridge_program_id
   )

2. Send 1 lamport to mint_pda via any Solana wallet:
   system_program::transfer(attacker, mint_pda, 1)

3. The bridge's MPC now signs and submits a DeployToken payload for "usdc.near".

4. The deploy_token instruction fails:
   Error: "0x0" (system program create_account: account already exists)

5. finalize_transfer for any "usdc.near" transfer also fails because the mint
   account does not exist.

6. User funds locked on NEAR have no corresponding Solana mint to receive them.
``` [1](#0-0)
