### Title
Cross-User ETH Consumption in `PeripheryPayments.pay` WETH Branch Causes Direct Loss of Trapped ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay` uses `address(this).balance` — the contract's **entire** native ETH balance — when deciding how much ETH to wrap for a WETH payment. Because multiple users can leave ETH in the contract across separate transactions (via any `payable` entry point), a subsequent caller's WETH payment is silently subsidized by a prior user's unrefunded ETH, consuming that ETH without the prior user's consent.

### Finding Description

The `pay()` function contains the following WETH branch: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire contract balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);  // ← payer only charged remainder
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` is not scoped to the current caller. ETH enters the contract legitimately via any `payable` entry point: [2](#0-1) [3](#0-2) 

`multicall`, `addLiquidityExactShares`, `addLiquidityWeighted`, `refundETH`, `unwrapWETH9`, and `sweepToken` are all `payable`. The `receive()` guard only blocks *direct* ETH sends from non-WETH addresses: [4](#0-3) 

It does **not** prevent ETH from accumulating via `msg.value` on any of the above functions. If a user sends excess ETH with `addLiquidityExactShares` (e.g., overestimates the WETH needed) and does not bundle a `refundETH` call in the same multicall, the surplus ETH sits in the contract until explicitly reclaimed.

### Impact Explanation

**Direct loss of user principal.** When user B subsequently calls `addLiquidityExactShares` for a WETH pool:

1. The pool's `metricOmmModifyLiquidityCallback` fires, calling `pay(WETH, userB, pool, amount)`.
2. `nativeBalance = address(this).balance` picks up user A's unrefunded ETH (call it `Y`).
3. The contract wraps `Y` of user A's ETH and transfers it to the pool.
4. Only `amount - Y` is pulled from user B via `safeTransferFrom`.
5. User A's `Y` ETH is permanently consumed for user B's liquidity deposit — user A receives nothing in return.

The pool receives the correct total WETH, so no pool invariant is broken, but **user A suffers a direct, unrecoverable loss of `Y` ETH**.

A secondary (simpler) path exists: anyone can call `refundETH()` to drain the entire contract ETH balance to themselves, since it sends `address(this).balance` to `msg.sender` with no access control: [5](#0-4) 

### Likelihood Explanation

The standard usage pattern (bundle `addLiquidityExactShares` + `refundETH` in a single `multicall`) is safe. However:

- Users calling `addLiquidityExactShares` directly (not via multicall) with `msg.value` and sending more ETH than the pool requests will leave a residual balance.
- Front-runners monitoring the mempool can observe the unrefunded ETH and immediately call `addLiquidityExactShares` (or simply `refundETH`) to consume it.
- No special permissions, malicious pools, or non-standard tokens are required.

### Recommendation

Track per-caller ETH contributions using transient storage (analogous to the existing `_setPayContext` / `_loadPayContext` pattern), and in `pay()` only consume up to the amount the current `payer` deposited. Alternatively, restrict the WETH branch to only use `address(this).balance` when `payer == address(this)` (i.e., mid-path), and always use `safeTransferFrom` for external payers regardless of the contract's ETH balance, leaving ETH management entirely to explicit `unwrapWETH9` / `refundETH` calls.

### Proof of Concept

```solidity
// Foundry test sketch
function test_crossUserETHConsumption() public {
    // User A sends 1 ETH with addLiquidityExactShares for a WETH pool
    // Pool only needs 0.6 ETH worth of WETH; 0.4 ETH excess stays in adder
    vm.prank(userA);
    adder.addLiquidityExactShares{value: 1 ether}(pool, salt, deltas, 1e18, 0, "");
    // 0.4 ETH now sits in adder (userA forgot refundETH)

    uint256 userBWETHBefore = weth.balanceOf(userB);

    // User B calls addLiquidityExactShares for same WETH pool needing 1 ETH
    vm.prank(userB);
    adder.addLiquidityExactShares(pool, salt, deltas2, 1e18, 0, "");

    uint256 userBWETHAfter = weth.balanceOf(userB);

    // User B was only charged 0.6 WETH (1 - 0.4), not 1 WETH
    assertEq(userBWETHBefore - userBWETHAfter, 0.6 ether);
    // User A's 0.4 ETH is gone — consumed for user B's deposit
    assertEq(address(adder).balance, 0);
}
```
