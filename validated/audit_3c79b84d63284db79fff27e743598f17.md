### Title
`ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Timelock Bypass via Integer Wrapping in `check_time_locks` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false` (the legacy code path), relative-timelock conditions (`ASSERT_HEIGHT_RELATIVE`, `ASSERT_SECONDS_RELATIVE`, `ASSERT_BEFORE_HEIGHT_RELATIVE`, `ASSERT_BEFORE_SECONDS_RELATIVE`) use `wrapping_add` instead of `saturating_add`. An unprivileged attacker who crafts a CLVM puzzle emitting a near-maximum relative value (e.g., `ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF`) causes the addition to wrap to a small integer, making the timelock check pass immediately — bypassing a lock that should be effectively permanent.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` boolean for all four relative-timelock conditions:

```
ASSERT_HEIGHT_RELATIVE (nowrap=false path):
  prev_transaction_block_height < unspent.confirmed_block_index.wrapping_add(height_relative)
``` [1](#0-0) 

When `nowrap=false` and `confirmed_block_index + height_relative` overflows `u32`, the wrapped result is a small number. The comparison `prev_height < small_number` evaluates to `false`, so no error is returned — the spend is accepted even though the timelock has not elapsed.

The same wrapping flaw applies to `ASSERT_SECONDS_RELATIVE` (u64 overflow): [2](#0-1) 

And the inverse direction for `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` — wrapping causes a spend that should be valid (deadline far in the future) to be incorrectly rejected, or vice versa: [3](#0-2) 

The repository's own test suite explicitly documents and confirms the divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [4](#0-3) 

The Python-level test confirms the same for both height and seconds: [5](#0-4) 

The function is exported as a first-class Python binding callable by chia-blockchain with any `nowrap` value: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Impact class: High — timelock condition validation bypass enables unauthorized spend acceptance.**

A CLVM puzzle that emits `(ASSERT_HEIGHT_RELATIVE . 0xFFFFFFFF)` is intended to be unspendable for ~4.3 billion blocks (effectively permanent). Under `nowrap=false`:

- `confirmed_block_index.wrapping_add(0xFFFF_FFFF)` wraps to `confirmed_block_index - 1`
- `prev_height < (confirmed_block_index - 1)` is `false` for any reasonable current height
- The check returns `Ok(())` — the spend is accepted immediately

This allows a coin whose puzzle enforces a very long relative timelock to be spent at any block height, defeating the puzzle's security invariant. Any coin whose puzzle relies on a large `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value for security (e.g., vaults, time-locked custody, DeFi contracts) is vulnerable.

Additionally, nodes running with `nowrap=false` and nodes running with `nowrap=true` will reach different consensus outcomes for the same spend bundle, constituting a **consensus divergence** — a Critical-class impact if both modes are simultaneously active on the network.

---

### Likelihood Explanation

- The attacker-controlled entry path is fully unprivileged: any user can submit a spend bundle containing a CLVM puzzle that outputs `ASSERT_HEIGHT_RELATIVE 0xFFFFFFFF`.
- The `nowrap=false` path is a live, exported code path in the Python binding (`py_check_time_locks`), callable by chia-blockchain's mempool and block validation logic.
- The wrapping behavior is not gated behind any hard-fork height check within chia_rs itself — the caller decides.
- If chia-blockchain passes `nowrap=False` for any block height range (e.g., pre-hard-fork blocks), all spends with large relative timelocks in that range are affected.

---

### Recommendation

1. **Remove the `nowrap=false` branch entirely** or make `nowrap=true` (saturating arithmetic) the only supported behavior. The wrapping path has no safe use case: a puzzle author who writes `ASSERT_HEIGHT_RELATIVE 0xFFFFFFFF` intends a permanent lock, not an immediate unlock.
2. If backward compatibility with historical blocks requires the wrapping behavior, gate it strictly by block height inside `check_time_locks` itself (using `constants.hard_fork_height` or equivalent), rather than delegating the choice to the Python caller.
3. Add a consensus-layer assertion that `height_relative` and `seconds_relative` values that would overflow are treated as "impossible" constraints (similar to how `ImpossibleHeightAbsoluteConstraints` is enforced in `validate_conditions`).

---

### Proof of Concept

**Craft a CLVM puzzle that emits `ASSERT_HEIGHT_RELATIVE 0xFFFFFFFF`:**

```python
# Coin confirmed at block 10.
# ASSERT_HEIGHT_RELATIVE = 0xFFFFFFFF should require block ~4,294,967,305 to spend.
# With nowrap=False: 10 + 0xFFFFFFFF wraps to 9.
# Check: current_height (e.g., 15) < 9 → False → no error → spend accepted.

from chia_rs import check_time_locks, SpendBundleConditions, CoinRecord
from chia.types.blockchain_format.sized_bytes import bytes32
from chia.util.ints import uint32, uint64

coin_id = bytes32(b'\x03' * 32)
coin_record = CoinRecord(
    coin=...,
    confirmed_block_index=uint32(10),
    spent_block_index=uint32(0),
    coinbase=False,
    timestamp=uint64(1000),
)
conds = make_test_conds(height_relative=0xFFFF_FFFF)  # near-max u32

# nowrap=True (correct): returns error code 13 (ASSERT_HEIGHT_RELATIVE_FAILED)
result_safe = check_time_locks({coin_id: coin_record}, conds, uint32(15), uint64(2000), True)
assert result_safe == 13  # correctly rejected

# nowrap=False (vulnerable): returns None (spend accepted — timelock bypassed)
result_vuln = check_time_locks({coin_id: coin_record}, conds, uint32(15), uint64(2000), False)
assert result_vuln is None  # BUG: timelock silently bypassed
``` [8](#0-7)
