### Title
`HlBridgeToken` 3-arg `mint` silently parks all tokens at `_systemAddress` when `finTransfer` is called with any non-empty `message` — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`OmniBridge.finTransfer` dispatches to the 3-arg `IBridgeToken.mint(recipient, amount, message)` whenever `payload.message.length > 0`. `HlBridgeToken` overrides that function to **ignore the message bytes entirely** and unconditionally move all minted tokens from `recipient` to `_systemAddress`. Because `coreReceiveWithData` is the only release valve for `_systemAddress`'s balance, and it can only be triggered by the HyperLiquid system address in response to HyperCore events (not NEAR-originated transfers), any NEAR→HyperEVM transfer that carries a non-empty `message` field results in the recipient receiving zero tokens and the full amount being stranded at `_systemAddress` with no on-chain recovery path.

---

### Finding Description

**`OmniBridge.finTransfer` mint dispatch** (`OmniBridge.sol` lines 337–349):

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient, payload.amount, payload.message   // 3-arg path
        );
    }
}
``` [1](#0-0) 

**`HlBridgeToken.mint` override** (`HlBridgeToken.sol` lines 76–83):

```solidity
function mint(
    address account,
    uint256 value,
    bytes memory          // ← unnamed, completely ignored
) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);   // tokens leave recipient immediately
}
``` [2](#0-1) 

The `bytes memory` parameter is unnamed and never read. Regardless of what the message contains, the function mints `value` to `account` and then immediately transfers the entire balance from `account` to `_systemAddress`. The recipient ends with a zero balance.

The only mechanism to release tokens from `_systemAddress` is `coreReceiveWithData`, which is gated to `msg.sender == _systemAddress`:

```solidity
if (msg.sender != _systemAddress) revert NotSystemAddress();
``` [3](#0-2) 

That callback is invoked by the HyperLiquid system address only when a HyperCore user triggers `sendToEvmWithData`. A NEAR-originated `finTransfer` creates no corresponding HyperCore event, so the system address has no reason to call `coreReceiveWithData`, and the tokens remain stranded.

---

### Impact Explanation

Any NEAR→HyperEVM transfer targeting an `HlBridgeToken` with a non-empty `message` field results in:

- `recipient.balanceOf == 0` after `finTransfer` completes
- `_systemAddress.balanceOf == amount` with no automatic release trigger
- The nonce is consumed (`completedTransfers[nonce] = true`), so the transfer cannot be replayed

This is a **critical permanent fund lock**: bridged assets are irreversibly stranded at `_systemAddress` with no on-chain recovery path available to the user or the bridge protocol. [4](#0-3) 

---

### Likelihood Explanation

The trigger is a non-empty `message` field in a NEAR-originated transfer to an `HlBridgeToken`. This is a normal, user-accessible bridge parameter — the NEAR bridge contract accepts arbitrary `message` strings in `initTransfer`. Any user who attaches a message (e.g., for cross-chain DeFi routing, memo fields, or protocol integrations) while targeting an HlBridgeToken will silently lose their funds. No privileged access, no key compromise, and no MPC collusion is required: the MPC signs any valid source-chain event, including those with non-empty messages. [5](#0-4) 

---

### Recommendation

**Option A (preferred):** Override `mint(address, uint256, bytes)` in `HlBridgeToken` to revert unconditionally, forcing all NEAR→HyperEVM finalization through the 2-arg path (which correctly delivers tokens to the recipient). The 3-arg path should only be reachable from the HyperCore accounting flow, not from `finTransfer`.

**Option B:** In `OmniBridge.finTransfer`, detect `HlBridgeToken` (e.g., via an interface check or a separate registry flag) and always call the 2-arg mint, ignoring the message field for that token type.

**Option C:** If the 3-arg path is intentionally reserved for HyperCore-originated mints, add an explicit guard in `HlBridgeToken.mint(address, uint256, bytes)` that only allows calls originating from a HyperCore-specific context (e.g., a flag set by `coreReceiveWithData`), and reverts otherwise.

---

### Proof of Concept

```solidity
// 1. User on NEAR calls initTransfer targeting hlToken with message = "hello"
//    → MPC observes the event and signs a TransferMessagePayload with message = "hello"

// 2. Relayer submits finTransfer:
TransferMessagePayload memory payload = TransferMessagePayload({
    tokenAddress: address(hlToken),
    recipient:    victim,
    amount:       1000e18,
    message:      "hello",   // non-empty → triggers 3-arg mint
    ...
});
omniBridge.finTransfer(mpcSignature, payload);

// 3. Inside HlBridgeToken.mint(victim, 1000e18, "hello"):
//    _mint(victim, 1000e18)              → victim.balance = 1000e18
//    _update(victim, _systemAddress, 1000e18) → victim.balance = 0, systemAddress.balance = 1000e18

// 4. Assert:
assert(hlToken.balanceOf(victim) == 0);
assert(hlToken.balanceOf(_systemAddress) == 1000e18);
// No coreReceiveWithData is ever called → tokens permanently stranded
