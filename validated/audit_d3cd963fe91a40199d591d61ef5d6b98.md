### Title
Pending Transfers with Sub-Minimum Amounts Are Permanently Locked Due to Missing Normalization Check in `init_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts any amount where `fee < amount`, but `sign_transfer` later rejects the transfer if `normalize_amount(amount - fee) == 0`. Because tokens are already burned or locked before `sign_transfer` is called, and no cancel/refund path exists for pending transfers, any transfer whose net amount is below the decimal-normalization threshold results in a permanent, irrecoverable loss of user funds.

### Finding Description

The NEAR bridge contract stores a `Decimals` struct per token address containing `decimals` (destination-chain precision) and `origin_decimals` (NEAR-side precision). When a NEAR-origin token is bridged to a lower-precision chain (e.g., NEAR 24-decimal token → Solana 9-decimal token), `normalize_amount` divides by `10^(origin_decimals - decimals)`:

```rust
// near/omni-bridge/src/lib.rs:2784-2787
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For a 24→9 decimal pair, any `amount - fee < 10^15` normalizes to 0.

`init_transfer` only validates `fee < amount`:

```rust
// near/omni-bridge/src/lib.rs:554-557
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

It then immediately burns or locks the tokens and stores the transfer in `pending_transfers`:

```rust
// near/omni-bridge/src/lib.rs:1850-1857
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

Later, when the relayer calls `sign_transfer`, the normalization check fires:

```rust
// near/omni-bridge/src/lib.rs:475-485
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This panics unconditionally for every future call to `sign_transfer` for this transfer ID. There is no public `cancel_transfer` or user-accessible refund path. `sign_transfer_callback` only removes the pending entry if the MPC signing succeeds and `fee.is_zero()`, but `sign_transfer` panics before MPC is ever called:

```rust
// near/omni-bridge/src/lib.rs:655-658
if let Ok(signature) = call_result {
    if fee.is_zero() {
        self.remove_transfer_message(message_payload.transfer_id);
    }
```

`claim_fee` requires a proof from the destination chain that the transfer was finalized there — impossible since the MPC signature was never produced. No other public function removes a pending transfer.

### Impact Explanation

- **For deployed (bridged) tokens**: tokens are burned in `burn_tokens_if_needed` at transfer initiation. They are permanently destroyed with no recovery path.
- **For native NEAR tokens**: tokens are held in the bridge contract and the `locked_tokens` counter is incremented. The tokens are permanently frozen; no user-callable function can release them.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

Any user bridging a NEAR-origin token to a lower-precision destination chain with a net amount below the normalization threshold triggers this. For a 24→9 decimal pair (NEAR token to Solana), the threshold is `10^15` base units — a realistic small transfer. The user receives no warning at `init_transfer` time; the failure only surfaces later when the relayer attempts `sign_transfer`. The user has no recourse.

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before burning/locking tokens:

```rust
let token_address = self.get_token_address(destination_chain, token_id.clone())
    .expect("token address not found");
let decimals = self.token_decimals.get(&token_address)
    .expect("decimals not found");
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().expect("invalid fee"),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the check already present in `sign_transfer` and prevents tokens from being burned/locked for transfers that can never be finalized.

### Proof of Concept

1. Token `foo.near` is registered with `origin_decimals = 24`, `decimals = 9` (NEAR → Solana pair).
2. User calls `ft_transfer_call` on `foo.near` with `amount = 500_000_000_000_000` (5×10¹⁴, below the 10¹⁵ threshold), `fee = 0`, recipient = Solana address.
3. `init_transfer` passes the `fee < amount` check, burns 5×10¹⁴ tokens, stores the transfer in `pending_transfers`.
4. Relayer calls `sign_transfer` for this `transfer_id`.
5. `normalize_amount(500_000_000_000_000, {decimals:9, origin_decimals:24})` = `500_000_000_000_000 / 10^15` = `0`.
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Every subsequent call to `sign_transfer` for this transfer ID panics identically.
8. The 5×10¹⁴ tokens are permanently burned. The entry remains in `pending_transfers` forever with no user-accessible escape hatch. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
