### Title
Missing Duplicate-Key Validation in `batch_insert()` Allows DataLayer Merkle Tree Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert()` contains a fast-path that writes leaf nodes directly to the blob without performing the duplicate-key or duplicate-hash checks that `MerkleBlob::insert()` enforces. An unprivileged caller who supplies a batch containing a key already present in the tree (or two identical keys within the same batch) bypasses the `KeyAlreadyPresent` / `HashAlreadyPresent` guards, corrupting the tree structure and producing an incorrect Merkle root hash.

### Finding Description

`insert()` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` uses `insert()` only for the first two items when the tree has ≤ 1 existing leaves. All remaining items — and **all** items when the tree already has ≥ 2 leaves — are written directly via `insert_entry_to_blob()` with no key-existence or hash-existence check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

The bootstrap branch that does call `insert()` pops items from the **end** of the input vector, so items at the front of a batch with ≥ 3 entries always take the unchecked path even when the tree starts empty: [3](#0-2) 

When a duplicate key is written, `insert_entry_to_blob` updates `block_status_cache` (via `add_leaf`), overwriting the cache entry for that key. The original leaf node remains in the blob but is no longer reachable through the cache. The Merkle root is then computed over a tree that contains both the orphaned leaf and the new duplicate leaf, producing a root hash that does not correspond to any consistent key-value mapping. [4](#0-3) 

The Python binding `py_batch_insert` exposes this path directly to callers: [5](#0-4) 

### Impact Explanation
A corrupted tree produces an incorrect root hash. Any `ProofOfInclusion` generated afterward is anchored to that wrong root. The `valid()` method on `ProofOfInclusion` only checks internal hash consistency within the proof object itself — it does not re-verify the root against an external trusted value: [6](#0-5) 

This means a proof generated from a corrupted tree will pass `valid()` while attesting to state that was never legitimately committed. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation
`batch_insert` is the primary bulk-insertion API exposed to Python and used during DataLayer synchronization. Any peer-supplied delta that includes a key already present in the local tree, or that repeats a key within the same batch, triggers the bug. No special privilege is required — only the ability to supply input to `batch_insert`.

### Recommendation
Add the same duplicate-key and duplicate-hash guards to the fast path in `batch_insert()` that `insert()` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Alternatively, add a pre-pass that validates all keys and hashes in the batch before any writes occur.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate with 3 leaves so leaf_count > 1
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
}

let root_before = blob.get_root_hash();

// batch_insert with a key that already exists (KeyId(0))
// All items take the unchecked fast path because leaf_count > 1
blob.batch_insert(vec![
    ((KeyId(0), ValueId(999)), sha256_num(&999i64)),  // duplicate key — no error raised
    ((KeyId(10), ValueId(10)), sha256_num(&10i64)),
    ((KeyId(11), ValueId(11)), sha256_num(&11i64)),
]).unwrap();

// Tree is now corrupted: two leaves with KeyId(0) exist in the blob,
// block_status_cache points only to the second one, root hash is wrong.
assert!(blob.check_integrity().is_err());  // integrity violation detected
``` [7](#0-6)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
