### Title
Missing Access Control on `migrate_deployed_token` Allows Any User to Corrupt Token Mappings and Permanently Freeze Bridge Funds — (`File: near/omni-bridge/src/lib.rs`)

### Summary

`migrate_deployed_token` in the NEAR omni-bridge contract carries no access-control guard. Any unprivileged caller can invoke it with a legitimate deployed token as `old_token` and an attacker-controlled account as `new_token`, permanently corrupting the bridge's bidirectional token-address mappings. Once corrupted, all in-flight and future cross-chain transfers for the affected token fail irrecoverably, freezing user funds on both the EVM and NEAR sides.

### Finding Description

`migrate_deployed_token` is a `#[payable]` public function with no role check: [1](#0-0) 

The only barrier is a small storage deposit (`NEP141_DEPOSIT`, a fraction of a NEAR token): [2](#0-1) 

The function then unconditionally:

1. Removes `old_token` from `deployed_tokens` and `deployed_tokens_v2`.
2. Inserts `new_token` (attacker-controlled) into both sets.
3. Removes `(origin_chain, old_token)` from `token_id_to_address` and inserts `(origin_chain, new_token)`.
4. Overwrites `token_address_to_id[origin_address]` with `new_token`.
5. Writes `migrated_tokens[old_token] = new_token`. [3](#0-2) 

Compare this with every other state-mutating admin operation in the same file, all of which carry `#[access_control_any(roles(Role::DAO, ...))]`: [4](#0-3) [5](#0-4) [6](#0-5) 

`migrate_deployed_token` is the sole exception.

### Impact Explanation

**EVM → NEAR direction (permanent fund freeze):**

`fin_transfer_callback` resolves the NEAR token to mint via `get_token_id(&transfer_message.token)`, which reads `token_address_to_id`: [7](#0-6) 

After the attack, this lookup returns `attacker_token`. The bridge then calls `mint` on `attacker_token`, over which it has no authority. The mint fails. Because `add_fin_transfer` is called before `send_tokens`, the finalization record is written first; if the callback reverts the record, every subsequent retry hits the same corrupted mapping and fails identically. The user's EVM tokens remain locked in the EVM bridge contract with no recovery path. [8](#0-7) 

**NEAR → EVM direction (fund freeze + broken swap):**

`legitimate_token` is removed from `deployed_tokens`. `burn_tokens_if_needed` silently skips the burn (token no longer recognized as deployed), so tokens sent to the bridge via `ft_on_transfer` accumulate in the contract with no corresponding cross-chain message. Additionally, `swap_migrated_token` — triggered when a user sends `legitimate_token` — attempts to burn it and mint `attacker_token`, which also fails: [9](#0-8) 

**Token-mapping accounting corruption:**

`token_address_to_id` and `token_id_to_address` are the authoritative registries used by every bridge flow. Corrupting them breaks bridge collateralization for the affected token across all chains.

### Likelihood Explanation

The attack requires only:
- Knowledge of any deployed token's NEAR account ID and its origin chain (both publicly readable from contract state).
- A deposit of `NEP141_DEPOSIT` (≈ 0.00125 NEAR, negligible cost).
- A single transaction.

No special role, leaked key, or colluding party is needed. The function is callable by any NEAR account.

### Recommendation

Add a DAO-only access control guard matching the pattern used by every other administrative function in the contract:

```rust
#[access_control_any(roles(Role::DAO))]
#[payable]
pub fn migrate_deployed_token(
    &mut self,
    origin_chain: ChainKind,
    old_token: AccountId,
    new_token: AccountId,
) {
```

Additionally, consider requiring that `locked_tokens` for `old_token` across all chains is zero before allowing migration, analogous to the external report's recommendation to verify no open positions remain before switching a critical contract reference.

### Proof of Concept

1. Attacker identifies a deployed token, e.g., `usdc.near`, with `origin_chain = ChainKind::Eth` and EVM address `0xA0b8...`.
2. Attacker deploys a worthless NEAR token contract at `fake.attacker.near`.
3. Attacker calls:
   ```
   migrate_deployed_token(
       origin_chain = ChainKind::Eth,
       old_token    = "usdc.near",
       new_token    = "fake.attacker.near"
   )
   ```
   with `NEP141_DEPOSIT` attached.
4. State after call:
   - `token_address_to_id[Eth(0xA0b8...)] = "fake.attacker.near"`
   - `token_id_to_address[(Eth, "fake.attacker.near")] = Eth(0xA0b8...)`
   - `deployed_tokens` contains `"fake.attacker.near"`, not `"usdc.near"`
   - `migrated_tokens["usdc.near"] = "fake.attacker.near"`
5. Any user who subsequently bridges USDC from Ethereum to NEAR triggers `fin_transfer_callback`, which resolves the token to `"fake.attacker.near"` and attempts to mint it. The bridge has no mint authority → mint fails → EVM USDC permanently locked.
6. Any user who sends `usdc.near` to the bridge for a NEAR→EVM transfer finds the token no longer recognized as deployed; tokens accumulate in the bridge with no cross-chain event. [10](#0-9)

    #[payable]
    pub fn migrate_deployed_token(
        &mut self,
        origin_chain: ChainKind,
        old_token: AccountId,
        new_token: AccountId,
    ) {
        require!(
            env::attached_deposit() >= NEP141_DEPOSIT,
            BridgeError::NotEnoughAttachedDeposit.as_ref()
        );

        require!(
            self.deployed_tokens.remove(&old_token),
            BridgeError::OldTokenNotDeployed.as_ref(),
        );
        require!(
            self.deployed_tokens.insert(&new_token),
            BridgeError::TokenExists.as_ref()
        );
        self.deployed_tokens_v2.remove(&old_token);
        self.deployed_tokens_v2.insert(&new_token, &origin_chain);

        let origin_address = self
            .token_id_to_address
            .remove(&(origin_chain, old_token.clone()))
            .near_expect(BridgeError::FailedToGetTokenAddress);

        require!(
            self.token_id_to_address
                .insert(&(origin_chain, new_token.clone()), &origin_address)
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );

        self.token_address_to_id
            .insert(&origin_address, &new_token)
            .near_expect(BridgeError::ExpectedToOverwriteTokenAddress);

        require!(
            self.migrated_tokens
                .insert(&old_token, &new_token)
                .is_none(),
            BridgeError::TokenAlreadyMigrated.as_ref()
        );

        ext_token::ext(new_token.clone())
            .with_static_gas(STORAGE_DEPOSIT_GAS)
            .with_attached_deposit(NEP141_DEPOSIT)
            .storage_deposit(&env::current_account_id(), Some(true))
            .detach();

        env::log_str(
            &OmniBridgeEvent::MigrateTokenEvent {
                old_token_id: old_token,
                new_token_id: new_token,
            }
            .to_log_string(),
        );
    }

    fn swap_migrated_token(
        &mut self,
        sender_id: AccountId,
        old_token: AccountId,
        amount: U128,
    ) -> Promise {
        let new_token = self
            .migrated_tokens
            .get(&old_token)
            .near_expect(BridgeError::TokenNotMigrated);

        let burn = ext_token::ext(old_token).burn(amount);
        let mint = ext_token::ext(new_token).mint(sender_id, amount, None);

        burn.and(mint)
    }
