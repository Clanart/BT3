### Title
Push Payment in `finTransfer` Causes Irrecoverable Lock of NEAR-Side Funds When EVM Recipient Reverts - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::finTransfer` uses a push-payment pattern to deliver assets to `payload.recipient`. If the recipient is unable to receive the tokens (e.g., a contract that rejects ETH, a USDC-blacklisted address, or an ERC-1155 receiver that reverts), every `finTransfer` call for that transfer reverts. Because the NEAR-side bridge has no public cancel/refund path for a pending transfer, the user's locked or burned NEAR-side tokens become permanently irrecoverable.

---

### Finding Description

**Root cause — push payment in `finTransfer`:**

`OmniBridge.sol::finTransfer` marks the destination nonce as used, then immediately pushes assets to `payload.recipient` in three branches:

1. **Native ETH** — a low-level `call{value}` that reverts the whole transaction if the recipient rejects ETH.
2. **ERC-1155** — `safeTransferFrom` which invokes `onERC1155Received` on the recipient; a reverting hook causes the whole transaction to revert.
3. **ERC-20 (e.g., USDC)** — `safeTransfer` which reverts if the recipient is blacklisted by the token contract. [1](#0-0) [2](#0-1) 

Because the nonce write at line 287 and the asset push are in the same transaction, a revert in the push also rolls back the nonce. The nonce is therefore never consumed, and the relayer can retry — but every retry will revert for the same reason. The transfer is permanently unfinalisable on the EVM side.

**No cancel path on NEAR:**

When a user initiates a NEAR→EVM transfer, `init_transfer_internal` locks or burns the user's tokens and stores the transfer in `pending_transfers`. [3](#0-2) 

`remove_transfer_message` (which releases the pending-transfer record and refunds storage) is only called from `sign_transfer_callback` (after a zero-fee signing) and `claim_fee_callback` (after EVM-side finalization proof is submitted). [4](#0-3) 

There is no public `cancel_transfer` or user-callable refund function anywhere in `near/omni-bridge/src/lib.rs`. A transfer stuck in `pending_transfers` with no EVM-side finalization proof can never be removed by the user.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds** — matching the allowed Critical impact class.

A user whose NEAR-side tokens have been locked or burned (in `init_transfer_internal`) and whose EVM-side recipient address permanently rejects the push payment will lose those tokens forever. There is no on-chain escape hatch.

---

### Likelihood Explanation

**Moderate.** Two realistic triggers exist:

1. **USDC blacklist** — Circle actively blacklists addresses. A user whose EVM address is blacklisted after initiating a NEAR→EVM USDC transfer will have their NEAR-side tokens permanently locked, because every `safeTransfer` to that address will revert.
2. **ETH-rejecting contract recipient** — A user who specifies a smart-contract address that has no `receive()` / `fallback()` (or one that explicitly reverts) as the EVM recipient of a native-ETH transfer will trigger the same permanent lock.

Both scenarios require only normal, unprivileged user actions (initiating a bridge transfer). No privileged role or external compromise is needed.

---

### Recommendation

1. **Separate nonce consumption from asset delivery.** Mark `completedTransfers[payload.destinationNonce] = true` unconditionally (before any external call), and store the claimable amount in a per-recipient mapping. Let recipients pull their funds in a separate `claimTransfer` call. This is the Pull-over-Push pattern.

2. **Alternatively, wrap the push in a try/catch and emit a `TransferFailed` event** so that a NEAR-side proof of failure can be submitted to unlock/re-mint the user's tokens on NEAR.

3. **Add a public `cancel_transfer` function on NEAR** that accepts a proof of EVM-side permanent failure (or a timeout) and returns the locked/burned tokens to the original sender.

---

### Proof of Concept

**Scenario: USDC blacklist**

1. Alice holds USDC on NEAR and calls `ft_transfer_call` to the bridge with an `InitTransferMsg` specifying her EVM address as recipient.
2. `init_transfer_internal` burns Alice's NEAR-side USDC and stores the transfer in `pending_transfers`. [5](#0-4) 
3. Before the relayer acts, Circle blacklists Alice's EVM address.
4. The relayer calls `OmniBridge::finTransfer` with the MPC-signed payload.
5. `completedTransfers[nonce] = true` is written, then `IERC20(usdc).safeTransfer(alice_evm, amount)` reverts because Alice is blacklisted. [6](#0-5) 
6. The entire transaction reverts; the nonce is not consumed.
7. Every subsequent retry reverts identically.
8. Alice's NEAR-side USDC is permanently burned with no recovery path. [4](#0-3)

        storage_owner: AccountId,

            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(

            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

                &token_id,
                transfer_message.amount.0,
            );

            return transfer_message.amount;

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
