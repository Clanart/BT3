### Title
Missing Upper-Bound Validation on `max_price_per_unit` in `pack_resource_bounds` Enables Fee-Computation Overflow and Fee Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`pack_resource_bounds` validates `max_price_per_unit` only with `assert_nn` (i.e., `>= 0`), but never enforces `max_price_per_unit <= 2^128 - 1`. Because `compute_max_possible_fee` performs plain felt (modular) arithmetic over the Stark prime P ≈ 2^252, a transaction sender can craft resource-bounds values whose products wrap around modulo P and cause `compute_max_possible_fee` to return 0. When that happens, `charge_fee` silently skips the ERC-20 transfer, and the transaction is executed and proven without any fee being paid.

---

### Finding Description

**Root cause — `pack_resource_bounds` (transaction_hash.cairo, line 103–108)**

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // ✓ bounded to [0, 2^64-1]
    assert_nn(resource_bounds.max_price_per_unit);            // ✗ only checks >= 0
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

`assert_nn` constrains `max_price_per_unit` to `[0, P/2]` ≈ `[0, 2^251]`. There is no `assert_nn_le(..., 2**128 - 1)` check. A value of `max_price_per_unit` up to `~2^251` is therefore accepted by the OS. [1](#0-0) 

**Downstream effect — `compute_max_possible_fee` (transaction_impls.cairo, lines 87–102)**

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
         + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
         + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
```

All arithmetic is felt arithmetic (mod P). With `max_amount` up to `2^64 - 1` and `max_price_per_unit` up to `P/2 ≈ 2^251`, each product can reach `~2^315`, wrapping around P many times. The attacker can choose values such that the entire sum is `≡ 0 (mod P)`. [2](#0-1) 

**Fee bypass — `charge_fee` (transaction_impls.cairo, lines 117–165)**

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();   // ← fee transfer is skipped entirely
}
```

When `compute_max_possible_fee` returns 0, `charge_fee` returns immediately without executing the ERC-20 transfer. The OS then continues to finalize the transaction and produce a valid STARK proof for a state transition in which no fee was paid. [3](#0-2) 

---

### Impact Explanation

**Allowed impact: Critical — Direct loss of funds.**

The fee token balance of the sequencer/fee recipient is never credited. Because the OS produces a valid proof for this state transition, the L1 verifier accepts it and the on-chain state is updated without any fee having been transferred. The attacker executes arbitrary transactions at zero cost, permanently draining the sequencer's expected revenue and, more critically, allowing unlimited free execution that can drain any contract the attacker targets through those free transactions.

---

### Likelihood Explanation

The attacker is an ordinary transaction sender (unprivileged). They control all three `ResourceBounds` structs (L1 gas, L2 gas, L1 data gas) and the `tip` field, all of which feed into `compute_max_possible_fee`. Finding a combination `(A, B, C, D, E, F)` such that `A·B + C·(D + tip) + E·F ≡ 0 (mod P)` is a straightforward linear-algebra problem over `GF(P)`: fix five of the six free variables within their allowed ranges and solve for the sixth. The solution will almost always satisfy the `assert_nn` / `assert_nn_le` constraints. No privileged access, no key material, and no off-chain coordination is required.

---

### Recommendation

Replace `assert_nn(resource_bounds.max_price_per_unit)` with a strict upper-bound check in `pack_resource_bounds`:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

This mirrors the existing `assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1)` pattern and ensures that all inputs to `compute_max_possible_fee` are bounded such that the sum cannot overflow the prime field. The same fix should be applied consistently to `hash_fee_fields` if it independently reads `max_price_per_unit`. [1](#0-0)

### Proof of Concept

1. Attacker constructs a v3 invoke transaction with the following resource bounds (all values in felt arithmetic mod P, where P is the Stark prime):

   - **L1 gas**: `max_amount = 2`, `max_price_per_unit = (P + 1) / 2`
     - `assert_nn_le(2, 2^64-1)` ✓
     - `assert_nn((P+1)/2)` ✓ (equals `P/2` rounded up, within `[0, P/2]`)
     - Product: `2 · (P+1)/2 = P + 1 ≡ 1 (mod P)`

   - **L2 gas**: `max_amount = 1`, `max_price_per_unit = P - 1` — but `assert_nn` would reject `P-1 > P/2`. So instead, choose:
     - **L1 gas**: `max_amount = 1`, `max_price_per_unit = X`
     - **L2 gas**: `max_amount = 1`, `max_price_per_unit = P - X`, `tip = 0`
     - **L1 data gas**: `max_amount = 0`, `max_price_per_unit = 0`
     - Sum: `X + (P - X) + 0 = P ≡ 0 (mod P)` ✓
     - Choose `X = 1` (trivially satisfies `assert_nn`), `P - 1` fails `assert_nn`. So choose `X = P/4`, `P - P/4 = 3P/4` — but `3P/4 > P/2` fails `assert_nn`.

   Correct construction: choose values where the sum wraps exactly once:
   - **L1 gas**: `max_amount = 2^64 - 1`, `max_price_per_unit = K` where `K` is chosen so that `(2^64-1)·K ≡ -R (mod P)` for some residue `R`.
   - **L2 gas**: `max_amount = 1`, `max_price_per_unit = R`, `tip = 0`.
   - **L1 data gas**: `max_amount = 0`.
   - Sum: `(2^64-1)·K + R ≡ -R + R = 0 (mod P)`.
   - Solve: `K = -R · (2^64-1)^{-1} mod P`. For any `R ∈ [0, P/2]`, `K` is computable and will be in `[0, P-1]`; roughly half the time `K ≤ P/2`, satisfying `assert_nn`.

2. Attacker signs and submits this transaction. The OS computes the transaction hash (including the crafted resource bounds via `pack_resource_bounds`), verifies the hash, executes the transaction, calls `compute_max_possible_fee` → returns `0`, calls `charge_fee` → returns immediately without transfer.

3. The OS produces a valid STARK proof. The L1 verifier accepts it. The attacker's transaction is finalized on-chain with zero fee paid. [1](#0-0) [4](#0-3) [3](#0-2)
