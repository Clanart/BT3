### Title
Missing `__gap` in `BridgeToken` Base Upgradeable Contract Enables Storage Collision in `HyperliquedBridgeToken` — (File: `evm/src/omni-bridge/contracts/BridgeToken.sol`)

### Summary
`BridgeToken` is a UUPS-upgradeable base contract that is directly inherited by `HyperliquedBridgeToken`. `BridgeToken` declares three sequential storage variables (`_name`, `_symbol`, `_decimals`) but contains no `__gap` reserve. `HyperliquedBridgeToken` appends `_systemAddress` immediately after those slots. If `BridgeToken` is ever upgraded to add new state variables — a routine protocol operation — the `_systemAddress` slot in every deployed `HyperliquedBridgeToken` proxy will be silently overwritten, corrupting the only access-control guard on the `coreReceiveWithData` mint path.

### Finding Description
`BridgeToken` is declared as a UUPS-upgradeable contract and is used as the concrete base class for `HyperliquedBridgeToken`. [1](#0-0) 

It declares three storage variables and ends without any `__gap`: [2](#0-1) 

`HyperliquedBridgeToken` inherits `BridgeToken` and immediately appends `_systemAddress` as the next sequential storage slot: [3](#0-2) 

`_systemAddress` is the sole access-control guard for the `coreReceiveWithData` entry point: [4](#0-3) 

By contrast, `OmniBridge` — the other upgradeable base — correctly reserves a gap: [5](#0-4) 

`BridgeToken` has no analogous reserve. The `grep` search across all in-scope EVM Solidity files confirms `__gap` appears only in `OmniBridge.sol`.

### Impact Explanation
When the protocol team upgrades `BridgeToken` to add any new state variable (e.g., a fee flag, a metadata field, a role mapping), the Solidity storage layout shifts every child-contract slot downward. In every already-deployed `HyperliquedBridgeToken` proxy the raw bytes previously holding `_systemAddress` will now be interpreted as the new `BridgeToken` variable, and `_systemAddress` will read from the next slot — which holds unrelated or zero data.

Two concrete outcomes:

1. **Unauthorized mint (Critical):** If `_systemAddress` reads as `address(0)` after the collision, the guard `if (msg.sender != _systemAddress)` passes for any non-zero caller, allowing any address to invoke `coreReceiveWithData` and trigger the `_update` / `initTransfer` mint paths without restriction.

2. **Permanent freeze (Critical):** If `_systemAddress` reads as a non-zero garbage value, the real HyperLiquid system address will always fail the check, permanently bricking the HyperCore→HyperEVM bridge path and locking all in-flight HyperCore-side balances. [6](#0-5) 

### Likelihood Explanation
`BridgeToken` is an actively evolving contract in a live multi-chain bridge. The protocol already upgraded `OmniBridge` multiple times (evidenced by the `.openzeppelin/` manifests across Arbitrum, Base, Polygon, BSC, and custom networks). Adding metadata, fee, or role fields to `BridgeToken` is a foreseeable routine upgrade. The admin performing the upgrade would have no indication that the child contract's storage is being silently corrupted — the upgrade transaction itself succeeds without error.

### Recommendation
Add a `__gap` array at the end of `BridgeToken`, sized so that the total number of storage slots consumed by `BridgeToken` and its OpenZeppelin bases equals a fixed round number (e.g., 50):

```solidity
// evm/src/omni-bridge/contracts/BridgeToken.sol
uint256[47] private __gap; // adjust size to fill to slot boundary
```

The exact size must be calculated by counting all slots consumed by `Initializable`, `UUPSUpgradeable`, `ERC20Upgradeable`, `Ownable2StepUpgradeable`, and `BridgeToken`'s own three variables, then subtracting from the chosen boundary.

### Proof of Concept

**Current storage layout of a deployed `HyperliquedBridgeToken` proxy (simplified):**

| Slot | Variable | Contract |
|------|----------|----------|
| … | OZ base slots | OZ bases |
| N | `_name` | `BridgeToken` |
| N+1 | `_symbol` | `BridgeToken` |
| N+2 | `_decimals` | `BridgeToken` |
| N+3 | `_systemAddress` | `HyperliquedBridgeToken` |

**After upgrading `BridgeToken` to add one new variable `_newField`:**

| Slot | Variable | Contract |
|------|----------|----------|
| … | OZ base slots | OZ bases |
| N | `_name` | `BridgeToken` |
| N+1 | `_symbol` | `BridgeToken` |
| N+2 | `_decimals` | `BridgeToken` |
| N+3 | `_newField` | `BridgeToken` ← **overwrites old `_systemAddress` bytes** |
| N+4 | `_systemAddress` | `HyperliquedBridgeToken` ← **reads garbage / zero** |

With `_systemAddress == address(0)`, any caller passes the `msg.sender != _systemAddress` check at line 114 of `HlBridgeToken.sol`, and can call `coreReceiveWithData` to mint arbitrary token amounts to any address via the `ACTION_TRANSFER` or `ACTION_INIT_TRANSFER` dispatch paths. [7](#0-6)

        bytes32 /*destinationRecipient*/,

        bytes calldata data
    ) external override {
        if (msg.sender != _systemAddress) revert NotSystemAddress();
        if (data.length == 0) revert EmptyActionData();

        uint8 action = uint8(data[0]);
        bytes calldata tail = data[1:];

        if (action == ACTION_TRANSFER) {

                address(this),
                amount128,
                fee,
                0,
                recipient,
                message
            );
        } else {
            revert UnknownAction(action);
        }

        emit CoreReceived(from, action, amount, data);
    }
