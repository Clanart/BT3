### Title
Missing Recipient Validation in `FinalizeTransferSol` Allows SOL to Be Redirected to Attacker-Controlled Address — (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer_sol.rs`)

### Summary

In the Solana bridge's `FinalizeTransferSol` instruction, the `recipient` account is declared as an unchecked, unconstrained account. The `FinalizeTransferPayload` struct contains no `recipient` field, and the `process` function transfers SOL directly to `self.recipient` without validating it against the signed payload. An unprivileged user who observes a valid signed payload (publicly available via Wormhole) can front-run the legitimate finalizer and substitute an attacker-controlled `recipient`, permanently diverting the bridged SOL.

### Finding Description

In `finalize_transfer_sol.rs`, the `FinalizeTransferSol` accounts struct declares `recipient` as:

```rust
/// CHECK: this can be any type of account
#[account(mut)]
pub recipient: UncheckedAccount<'info>,
``` [1](#0-0) 

There is no Anchor constraint tying this account to any value in the signed payload. The `FinalizeTransferPayload` struct (used as the inner type of `SignedPayload<FinalizeTransferPayload>`) contains only `destination_nonce`, `transfer_id`, `amount`, and `fee_recipient` — **no `recipient` field**. The `process` function then unconditionally transfers SOL to whatever account was passed as `recipient`:

```rust
transfer(
    CpiContext::new_with_signer(
        self.common.system_program.to_account_info(),
        Transfer {
            from: self.sol_vault.to_account_info(),
            to: self.recipient.to_account_info(),   // ← unchecked, unvalidated
        },
        &[&[SOL_VAULT_SEED, &[self.config.bumps.sol_vault]]],
    ),
    data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
)?;
``` [2](#0-1) 

Because `FinalizeTransferPayload` has no `recipient` field, the MPC/NEAR-bridge signature cannot bind to a specific recipient. The nonce is consumed and the SOL is released to whichever account the caller supplies.

**Contrast with `FinalizeTransfer` (SPL-token path):** there, the recipient is implicitly validated through the Anchor ATA constraint:

```rust
#[account(
    init_if_needed,
    associated_token::mint = mint,
    associated_token::authority = recipient,   // ← recipient bound to ATA
    ...
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
``` [3](#0-2) 

No equivalent constraint exists in `FinalizeTransferSol`.

### Impact Explanation

An attacker who supplies a different `recipient` account causes the SOL vault to transfer the full bridged amount to the attacker. The destination nonce is simultaneously marked used, so the legitimate finalizer's subsequent call reverts with `NonceAlreadyUsed`. The intended recipient receives nothing and has no recourse — the funds are permanently lost to them. This matches **Critical: Permanent freezing / irrecoverable lock of user funds in bridge vault flows**.

### Likelihood Explanation

Wormhole VAAs (the signed payloads) are publicly observable on-chain the moment they are posted. Any unprivileged observer can extract a valid `SignedPayload<FinalizeTransferPayload>` and replay it with a substituted `recipient` before the legitimate relayer finalizes. No privileged access, leaked key, or colluding party is required — only the ability to submit a Solana transaction faster than the original relayer.

### Recommendation

Add an explicit recipient field to `FinalizeTransferPayload` so the MPC signature binds to the intended recipient, and add an Anchor constraint in `FinalizeTransferSol` that enforces the passed `recipient` matches the value in the verified payload. For example:

```rust
#[account(
    mut,
    constraint = recipient.key() == data.payload.recipient
        @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

Alternatively, mirror the SPL-token path by deriving the recipient from the signed payload inside `process` and asserting `self.recipient.key() == data.recipient` before executing the transfer.

### Proof of Concept

1. NEAR bridge MPC signs a `FinalizeTransferPayload` for nonce N, amount A, intended recipient `Alice`.
2. The VAA is published on Wormhole and is publicly readable.
3. Attacker constructs a `finalize_transfer_sol` instruction using the same valid `SignedPayload` but passes `Attacker` as the `recipient` account.
4. Anchor validates the signature over `{destination_nonce, transfer_id, amount, fee_recipient}` — all correct — and proceeds.
5. `process` calls `transfer(sol_vault → Attacker, A)`.
6. Nonce N is marked used; Alice's legitimate finalization reverts.
7. Attacker receives `A` lamports; Alice's bridged SOL is permanently lost. [4](#0-3)

impl FinalizeTransferSol<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        UsedNonces::use_nonce(
            data.destination_nonce,
            &self.used_nonces,
            &mut self.config,
            self.authority.to_account_info(),
            self.common.payer.to_account_info(),
            &Rent::get()?,
            self.system_program.to_account_info(),
        )?;

        transfer(
            CpiContext::new_with_signer(
                self.common.system_program.to_account_info(),
                Transfer {
                    from: self.sol_vault.to_account_info(),
                    to: self.recipient.to_account_info(),
                },
                &[&[SOL_VAULT_SEED, &[self.config.bumps.sol_vault]]],
            ),
            data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
        )?;

        let payload = FinalizeTransferResponse {
            token: Pubkey::default(),
            amount: data.amount,
            fee_recipient: data.fee_recipient.unwrap_or_default(),
            transfer_id: data.transfer_id,
        }
        .serialize_for_near(())?;

        self.common.post_message(payload)?;

        Ok(())
    }
