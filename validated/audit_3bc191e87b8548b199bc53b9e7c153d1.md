Audit Report

## Title
Residual ETH in Router Consumed by Subsequent Caller's WETH Swap, Causing Direct ETH Loss - (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function uses `address(this).balance` — the router's total native ETH balance — rather than only the ETH sent by the current caller. Any ETH left in the router from a prior caller (e.g., excess `msg.value` not yet refunded) is silently consumed to fund a subsequent caller's WETH swap, causing the prior caller to lose their ETH with no recourse.

## Finding Description
In `PeripheryPayments.sol` lines 73–84, when `token == WETH` and `payer != address(this)`, the function reads `uint256 nativeBalance = address(this).balance` and uses it unconditionally:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

`address(this).balance` is the router's aggregate ETH, not the current `msg.value`. ETH enters the router via any `payable` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). If a caller sends `msg.value > amountIn` and does not atomically call `refundETH()`, the surplus ETH persists in the router across transactions.

**Exploit path:**
1. Alice calls `exactInputSingle{value: 1.1 ether}(tokenIn=WETH, amountIn=1 ether, ...)`. The swap callback calls `pay(WETH, Alice, pool, 1 ether)`. `nativeBalance = 1.1 ether >= 1 ether`, so exactly 1 ETH is wrapped and sent. 0.1 ETH remains in the router. Alice does not call `refundETH()` (or a front-runner sandwiches her before she can).
2. Bob calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=0.1 ether, ...)`. The callback calls `pay(WETH, Bob, pool, 0.1 ether)`. `nativeBalance = 0.1 ether >= 0.1 ether`, so Alice's residual ETH is wrapped and sent to the pool on Bob's behalf. Bob's swap succeeds without Bob spending any ETH or WETH. Alice's 0.1 ETH is permanently lost.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only prevents arbitrary ETH pushes; it does not prevent ETH accumulation via `msg.value` on payable swap functions. No per-caller accounting exists anywhere in the payment path.

## Impact Explanation
Direct loss of user principal (ETH). A victim who sends excess ETH with a WETH swap and fails to atomically refund loses that ETH to any subsequent WETH swap caller. This satisfies the "Critical/High/Medium direct loss of user principal" gate. Severity is **High**: the stolen amount is bounded only by the victim's excess `msg.value`, and the attack requires zero privilege and zero capital from the attacker.

## Likelihood Explanation
Any unprivileged EOA or contract can trigger this by calling any WETH-input swap function with `msg.value = 0` (or any value less than `amountIn`) whenever residual ETH exists in the router. Residual ETH arises naturally whenever a user sends `msg.value > amountIn` without an atomic `refundETH()` call — a common pattern when users estimate gas or overshoot slippage. A front-runner can monitor the mempool for transactions that leave ETH in the router and immediately follow with a zero-cost WETH swap to drain it. The attack is repeatable, permissionless, and requires no special setup.

## Recommendation
Track only the ETH contributed by the current call. Replace the global `address(this).balance` read with `msg.value` (passed into `pay` as a parameter or read at the entry point and threaded through). Alternatively, record the router's ETH balance at the start of each top-level call and compute the caller's contribution as `address(this).balance - balanceBefore`. Ensure `refundETH()` is called (or enforced) at the end of every payable entry point, or make the entry points non-payable when `tokenIn != WETH`.

## Proof of Concept
```solidity
// Foundry test sketch
function test_residualEthStolenByNextCaller() public {
    // Alice sends 1.1 ETH for a 1 ETH WETH swap, leaves 0.1 ETH in router
    vm.deal(alice, 1.1 ether);
    vm.prank(alice);
    router.exactInputSingle{value: 1.1 ether}(
        ExactInputSingleParams({tokenIn: WETH, ..., amountIn: 1 ether, ...})
    );
    // Alice does NOT call refundETH; 0.1 ETH sits in router

    assertEq(address(router).balance, 0.1 ether);

    // Bob calls with 0 ETH, amountIn = 0.1 ETH (tokenIn = WETH)
    vm.deal(bob, 0);
    vm.prank(bob);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({tokenIn: WETH, ..., amountIn: 0.1 ether, ...})
    );

    // Bob's swap succeeds; Alice's 0.1 ETH is gone
    assertEq(address(router).balance, 0);
    // Bob received swap output funded entirely by Alice's ETH
}
```
