Audit Report

## Title
Permissionless `refundETH` Drains Stranded ETH from Non-WETH `multicall{value}` Calls — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`multicall` is `public payable`, so ETH sent with it bypasses the `receive()` guard and enters the contract. When `addLiquidityExactShares` is called for a pool whose tokens are both non-WETH ERC20s, `pay()` takes the `safeTransferFrom` branch and never consumes `address(this).balance`. The stranded ETH is then claimable by any caller via the permissionless `refundETH()`, which unconditionally forwards the full contract balance to `msg.sender`.

## Finding Description

**Step 1 — ETH enters via `multicall`.**

`multicall` is declared `public payable`: [1](#0-0) 

Because the call carries calldata, the `receive()` fallback is never triggered. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only fires on plain ETH transfers with no calldata: [2](#0-1) 

**Step 2 — For a non-WETH pool, `pay()` never consumes the ETH.**

In `metricOmmModifyLiquidityCallback`, `pay()` is called with `token0`/`token1` from the pool's immutables. When neither token matches `WETH`, the `else` branch executes: [3](#0-2) 

`safeTransferFrom` pulls ERC20 tokens from the payer. `address(this).balance` is never read or decremented. The ETH sent with the multicall remains in the contract.

**Step 3 — `refundETH()` is permissionless.**

There is no check that `msg.sender` is the original depositor: [4](#0-3) 

Any EOA or contract can call `refundETH()` and receive the full `address(this).balance`.

## Impact Explanation

Direct loss of user ETH principal. The user's ETH is transferred to the attacker. The liquidity operation itself succeeds (ERC20 tokens are pulled correctly via `safeTransferFrom`), so the user receives LP shares but loses the ETH they attached to the call. This meets the "direct loss of user principal" threshold at Critical/High severity.

## Likelihood Explanation

The `multicall{value}` pattern is the documented ETH-input path for WETH pools. A user who copies this pattern for a non-WETH pool — or uses a frontend/SDK that does not gate ETH attachment on pool token type — will strand ETH. An attacker monitoring the mempool can front-run the user's own `refundETH` call, or simply call it after the multicall settles. No special privileges are required; any unprivileged EOA can execute the drain.

## Recommendation

In `refundETH()`, restrict the recipient to the address that initiated the current multicall. One approach: at the start of `multicall`, record `msg.sender` in transient storage and enforce it as the sole recipient in `refundETH()` for the duration of the call. Alternatively, accept a caller-supplied `recipient` address in `refundETH()` and validate it against the stored initiator. A simpler short-term fix is to change `refundETH` to send only to a hardcoded `msg.sender` check against transient storage set at multicall entry.

## Proof of Concept

```solidity
// 1. User calls (no refundETH in the batch, pool has two non-WETH ERC20 tokens):
adder.multicall{value: 1 ether}([
    abi.encodeCall(adder.addLiquidityExactShares, (erc20Pool, owner, salt, deltas, max0, max1, ""))
]);
// adder.balance == 1 ether; user's ERC20 LP position is created correctly

// 2. Attacker calls:
adder.refundETH();
// attacker receives 1 ether; user's ETH is gone
```

The `pay()` call in `metricOmmModifyLiquidityCallback` takes the `safeTransferFrom` path for both tokens, leaving `address(this).balance` untouched, and `refundETH()` forwards it to the attacker with no access check. [5](#0-4)
