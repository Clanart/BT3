### Title
Unbounded `AnteCache` Growth via Valid CheckTx Submissions Causes Node Memory Exhaustion - (File: `ante/cache/antecache.go`)

### Summary

The `AnteCache` used by Ethermint's ante handler to track pending mempool nonces is explicitly unbounded when initialized with `maxTx == 0`. The `Set` method only evicts entries when `maxTx > 0`; when `maxTx == 0`, every successful CheckTx permanently inserts a `(address, nonce)` entry with no eviction path. An attacker controlling many funded accounts can flood the node with valid Ethereum transactions, growing the cache without limit until the process is OOM-killed.

### Finding Description

`AnteCache` in `ante/cache/antecache.go` documents three modes:

```
// - if maxTx == 0, there is no cap on the number of transactions in the cache
// - if maxTx > 0, the cache will cap the number of transactions it stores,
// - if maxTx < 0, the cache is a no-op cache.
``` [1](#0-0) 

The `Set` method only triggers LRU eviction under the `maxTx > 0` branch:

```go
if c.maxTx > 0 && len(c.cache) >= c.maxTx {
    // evict oldest entry
}
elem := c.order.PushBack(key)
c.cache[key] = elem
``` [2](#0-1) 

When `maxTx == 0`, every call to `Set` unconditionally appends to both the `map` and the `list.List` with no upper bound.

Cache entries are written in `newEthAnteHandler` only after the **full** ante chain succeeds during CheckTx:

```go
if finalCtx.IsCheckTx() && !finalCtx.IsReCheckTx() {
    for _, entry := range pendingNonces {
        options.AnteCache.Set(entry.Address, entry.Nonce)
    }
}
``` [3](#0-2) 

Entries are only **removed** inside `CheckAndSetEthSenderNonce` during DeliverTx or a ReCheckTx nonce mismatch:

```go
} else if ex {
    // delete in case of deliver tx
    nonceCache.Delete(fromStr, txNonce)
}
``` [4](#0-3) 

There is no hook that removes a cache entry when a transaction is **evicted from the mempool** without going through DeliverTx. Consequently, if the attacker's transactions are displaced from the mempool by higher-priority transactions, the `(address, nonce)` entries remain in the cache permanently.

**Production initialization path**: `EthermintApp.setAnteHandler` passes `mempoolMaxTxs` directly to `NewAnteCache`:

```go
AnteCache: cache.NewAnteCache(mempoolMaxTxs),
``` [5](#0-4) 

`setupMempoolAndProposalHandlers` applies a floor of `DefaultMaxTxs = 3000` when the flag is `<= 0`:

```go
if maxTxs <= 0 {
    maxTxs = srvconfig.DefaultMaxTxs
}
``` [6](#0-5) 

However, this floor is **not** applied inside `setAnteHandler`. If an operator sets `--mempool.max-txs=0` (intending an unlimited mempool), the SDK mempool is silently capped at 3000, but the `AnteCache` receives `maxTx = 0` and becomes unbounded. The discrepancy is invisible to the operator.

`DefaultMaxTxs = 3000` is defined in `server/config/config.go`: [7](#0-6) 

### Impact Explanation

With an unbounded `AnteCache`, an attacker who controls N funded Ethereum accounts can submit N valid transactions (each with the correct nonce and sufficient fees) through the public `eth_sendRawTransaction` / CometBFT `broadcast_tx` endpoint. Each transaction that passes CheckTx inserts one `TxNonce` struct plus a `list.Element` (~100 bytes) into the cache. If the attacker's transactions are continuously displaced from the mempool before block inclusion, the entries are never cleaned up. Sustained flooding grows the cache until the Go runtime triggers an OOM kill, crashing the validator node. If enough validators are targeted simultaneously, the chain halts.

### Likelihood Explanation

The attack requires:
1. The node to be configured with `--mempool.max-txs=0` (or the equivalent app-options value), which is a plausible operator mistake given the misleading semantics (the mempool itself is still bounded at 3000 due to the floor check, but the cache is not).
2. Many funded accounts — each account contributes one cache entry per pending nonce. At ~100 bytes per entry, exhausting 8 GB requires ~80 million entries, which is expensive but feasible for a well-resourced attacker targeting a high-value chain.

The `simulation_test.go` explicitly instantiates `NewAnteCache(0)`, confirming the unbounded path is exercised and not dead code: [8](#0-7) 

### Recommendation

1. **Apply the same floor check in `setAnteHandler`** that `setupMempoolAndProposalHandlers` already applies: if `mempoolMaxTxs <= 0`, substitute `srvconfig.DefaultMaxTxs` before constructing the `AnteCache`.
2. **Add a mempool-eviction hook** that calls `AnteCache.Delete` whenever a transaction is removed from the mempool without going through DeliverTx, preventing stale entries from accumulating even in the bounded case.
3. **Document the `maxTx == 0` semantics** as explicitly dangerous and not suitable for production nodes.

### Proof of Concept

1. Start a node with `--mempool.max-txs=0`.
2. Generate N Ethereum key pairs and fund each with enough native token to cover gas fees.
3. For each address `addr_i` (current nonce = 0), broadcast a valid `MsgEthereumTx` with nonce 0 via `eth_sendRawTransaction`.
4. Immediately broadcast a higher-fee transaction from a different address to displace `addr_i`'s transaction from the mempool (preventing DeliverTx cleanup).
5. Observe `AnteCache.Size()` growing monotonically. Repeat until the node process is OOM-killed.

The cache entry for `(addr_i, 0)` is inserted at step 3 (CheckTx succeeds) and is never removed because the transaction is evicted from the mempool in step 4 without going through DeliverTx or a ReCheckTx nonce-mismatch path.
