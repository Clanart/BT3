### Title
`estimate_minimal_gas_vector` Omits Compiled-Class-Hash DA Cost for Declare Transactions, Allowing Under-Resourced Transactions Through Pre-Validation — (`crates/blockifier/src/fee/gas_usage.rs`)

### Summary

`estimate_minimal_gas_vector` hard-codes `n_compiled_class_hash_updates: 0` for `Declare` transactions even though every V2/V3 Declare execution writes exactly one compiled-class-hash entry to the DA segment. The resulting underestimate is used verbatim in `check_fee_bounds` (pre-validation), so a user can craft a Declare transaction whose resource bounds are below the true minimum, pass gateway/mempool admission, and force the sequencer to execute a transaction that will revert — paying less than the actual DA cost.

### Finding Description

In `estimate_minimal_gas_vector`:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,   // ← wrong; should be 1
    n_modified_contracts: 1,
},
``` [1](#0-0) 

`get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts to the DA segment, so setting the field to 0 silently drops 2 felts from the segment length fed into `get_da_gas_cost`. [2](#0-1) 

`get_da_gas_cost` converts that length to either blob gas (KZG DA mode) or L1 gas (calldata DA mode): [3](#0-2) 

The resulting `GasVector` is returned as the "minimal gas vector" and immediately used in `check_fee_bounds` to gate admission: [4](#0-3) 

Meanwhile, the actual execution of a V2/V3 Declare transaction always produces `n_compiled_class_hash_updates: 1`, as confirmed by the test helper that models real execution state changes: [5](#0-4) 

The gap between the admitted minimum and the true minimum is:

| DA mode | Underestimation |
|---|---|
| KZG (blob) | `2 × DATA_GAS_PER_FIELD_ELEMENT` = **64 l1_data_gas units** |
| Calldata | `2 × SHARP_GAS_PER_DA_WORD` ≈ **1 102 l1_gas units** | [6](#0-5) 

### Impact Explanation

A user submits a Declare V3 transaction with `l1_data_gas.max_amount` set to exactly the (underestimated) minimal value. `check_fee_bounds` passes. During execution the actual DA gas exceeds the resource bound, the transaction reverts, and the user pays only up to their (too-low) bound. The sequencer bears the execution cost without full compensation. This is a direct economic impact on sequencer revenue for every Declare transaction whose sender deliberately (or accidentally) sets bounds at the underestimated floor.

Fits: **Critical — Incorrect fee/gas/resource accounting with economic impact.**

### Likelihood Explanation

Every Declare V2/V3 transaction is affected. Any user who queries `starknet_estimateFee` (which executes the transaction and returns the correct gas) and then submits with bounds equal to the pre-validation floor will trigger the discrepancy. A malicious user can exploit this deliberately by setting bounds to the underestimated minimum.

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for the `Declare` arm in `estimate_minimal_gas_vector`, consistent with the existing TODO comment and with how `declare_expected_state_changes_count` models actual execution:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1,  // fix: always 1 for V2/V3 Declare
    n_modified_contracts: 1,
},
``` [1](#0-0) 

### Proof of Concept

1. Query `starknet_estimateFee` for a Declare V3 transaction in KZG DA mode; note the returned `data_gas_consumed` (call it `G`).
2. Compute the underestimated floor: `G_floor = G - 64` (the 2-felt compiled-class-hash DA cost).
3. Submit the Declare transaction with `l1_data_gas: { max_amount: G_floor, max_price_per_unit: <current price> }`.
4. `check_fee_bounds` computes `minimal_gas_amount_vector.l1_data_gas = G_floor` (because `n_compiled_class_hash_updates = 0`) and passes the check.
5. During execution the actual `l1_data_gas` consumed equals `G > G_floor`; the transaction reverts with `MaxGasAmountExceeded`.
6. The user is charged only `G_floor × price` instead of `G × price`; the sequencer is short-changed by `64 × price` per such transaction. [7](#0-6)

pub fn get_da_gas_cost(state_changes_count: &StateChangesCount, use_kzg_da: bool) -> GasVector {
    let onchain_data_segment_length = get_onchain_data_segment_length(state_changes_count);

    let (l1_gas, blob_gas) = if use_kzg_da {
        (
            0_u8.into(),
            u64_from_usize(
                onchain_data_segment_length * eth_gas_constants::DATA_GAS_PER_FIELD_ELEMENT,
            )
            .into(),
        )
    } else {
        // TODO(Yoni, 1/5/2024): count the exact amount of nonzero bytes for each DA entry.
        let naive_cost = onchain_data_segment_length * eth_gas_constants::SHARP_GAS_PER_DA_WORD;

        // For each modified contract, the expected non-zeros bytes in the second word are:
        // 1 bytes for class hash flag; 2 for number of storage updates (up to 64K);
        // 3 for nonce update (up to 16M).
        let modified_contract_cost = eth_gas_constants::get_calldata_word_cost(1 + 2 + 3);
        let modified_contract_discount =
            eth_gas_constants::GAS_PER_MEMORY_WORD - modified_contract_cost;
        let mut discount = state_changes_count.n_modified_contracts * modified_contract_discount;

        // Up to balance of 8*(10**10) ETH.
        let fee_balance_value_cost = eth_gas_constants::get_calldata_word_cost(12);
        discount += eth_gas_constants::GAS_PER_MEMORY_WORD - fee_balance_value_cost;

        // Cost must be non-negative after discount.
        let gas = naive_cost.saturating_sub(discount);

        (u64_from_usize(gas).into(), 0_u8.into())
    };

    GasVector { l1_gas, l1_data_gas: blob_gas, ..Default::default() }
}

pub const GAS_PER_LOG_TOPIC: usize = 375;
