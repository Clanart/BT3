### Title
Unchecked `recipient` Account in `FinalizeTransfer` and `FinalizeTransferSol` Allows Any Caller to Redirect Bridged Funds to an Arbitrary Address - (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`, `finalize_transfer_sol.rs`)

---

### Summary

Both `FinalizeTransfer` and `FinalizeTransferSol` Anchor instruction structs declare the `recipient` account as a completely unchecked, caller-supplied account with no constraint binding it to the recipient encoded in the signed `FinalizeTransferPayload`. Any caller who possesses a valid MPC-signed payload (intended for User B) can invoke the instruction while substituting their own address as `recipient`, causing the vault or mint authority to transfer tokens/SOL to the attacker instead of the legitimate recipient.

---

### Finding Description

In `finalize_transfer.rs`, the `FinalizeTransfer` account struct is declared as:

```rust
#[derive(Accounts)]
#[instruction(data: SignedPayload<FinalizeTransferPayload>)]
pub struct FinalizeTransfer<'info> {
    ...
    /// CHECK: this can be any type of account
    pub recipient: UncheckedAccount<'info>,

    #[account(
        init_if_needed,
        payer = common.payer,
        associated_token::mint = mint,
        associated_token::authority = recipient,   // ATA derived from caller-supplied recipient
        token::token_program = token_program,
    )]
    pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
    ...
}
``` [1](#0-0) 

The `#[instruction(data: SignedPayload<FinalizeTransferPayload>)]` attribute makes the signed payload available for use in Anchor account constraints, but **no constraint is placed on `recipient`** to verify it matches `data.payload.recipient`. The `token_account` is derived as the ATA of the caller-supplied `recipient`, and the `process` function transfers tokens directly into it:

```rust
transfer_checked(
    CpiContext::new_with_signer(
        self.token_program.to_account_info(),
        TransferChecked {
            from: vault.to_account_info(),
            to: self.token_account.to_account_info(),   // ATA of attacker-supplied recipient
            authority: self.authority.to_account_info(),
            mint: self.mint.to_account_info(),
        },
        &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
    ),
    data.amount...,
    self.mint.decimals,
)?;
``` [2](#0-1) 

The same flaw exists in `FinalizeTransferSol`, where native SOL is transferred directly to the unchecked `recipient`:

```rust
/// CHECK: this can be any type of account
#[account(mut)]
pub recipient: UncheckedAccount<'info>,
``` [3](#0-2) 

```rust
transfer(
    CpiContext::new_with_signer(
        self.common.system_program.to_account_info(),
        Transfer {
            from: self.sol_vault.to_account_info(),
            to: self.recipient.to_account_info(),   // attacker-supplied
        },
        &[&[SOL_VAULT_SEED, &[self.config.bumps.sol_vault]]],
    ),
    data.amount...,
)?;
``` [4](#0-3) 

The shared `authority` PDA (seeds `[AUTHORITY_SEED]`) signs for all vault transfers and mints on behalf of all users, exactly mirroring the shared vault authority described in the reference report. Because the `recipient` account is never validated against the payload, the authority will sign a transfer to any address the caller provides. [5](#0-4) 

By contrast, the EVM `finTransfer` in `OmniBridge.sol` correctly includes `payload.recipient` inside the Borsh-encoded, ECDSA-signed blob, so the recipient is cryptographically bound and cannot be substituted without invalidating the signature. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct theft of bridged assets on Solana.**

An attacker who observes a valid `SignedPayload<FinalizeTransferPayload>` (emitted as a Wormhole message or visible in the mempool/RPC) for any victim's inbound transfer can call `finalize_transfer` or `finalize_transfer_sol` first, supplying their own Solana address as `recipient`. The shared authority PDA will transfer the full token amount (or SOL) to the attacker's ATA/account. The victim receives nothing. The nonce is consumed, so the legitimate finalization cannot be replayed. The attacker suffers no loss.

---

### Likelihood Explanation

**High.** Signed payloads are broadcast publicly via Wormhole VAAs or observable on-chain. Any unprivileged Solana account can call these instructions. No special role, leaked key, or colluding party is required — only the ability to observe a pending finalization and submit the transaction first (trivial front-running on Solana given its public mempool).

---

### Recommendation

Add an Anchor account constraint on `recipient` in both `FinalizeTransfer` and `FinalizeTransferSol` that enforces the account address matches the recipient encoded in the signed payload. For example:

```rust
/// CHECK: validated against signed payload
#[account(address = data.payload.recipient.to_pubkey() @ ErrorCode::InvalidRecipient)]
pub recipient: UncheckedAccount<'info>,
```

This mirrors how the EVM bridge cryptographically binds the recipient inside the signed message, ensuring the authority PDA only releases funds to the address the NEAR MPC actually authorized.

---

### Proof of Concept

1. User B initiates a transfer from NEAR to Solana, specifying their Solana wallet `PK_B` as recipient.
2. NEAR MPC signs a `FinalizeTransferPayload` covering `destination_nonce`, `amount`, `token`, and `recipient = PK_B`. The signed VAA is published via Wormhole.
3. Attacker (User A) observes the VAA before User B submits the finalization.
4. User A calls `finalize_transfer` (or `finalize_transfer_sol`) passing:
   - `data` = the valid `SignedPayload` intended for User B (signature is valid, nonce unused)
   - `recipient` = User A's own Solana public key `PK_A`
   - `token_account` = User A's ATA for the bridged token
5. Anchor validates the signature on `data` (passes — payload is unmodified), marks the nonce used, and the shared `authority` PDA transfers the full token amount to User A's ATA.
6. User B's finalization attempt reverts with `NonceAlreadyUsed`. User A has stolen the entire bridged amount.
