### Title
Incorrect Gas Accumulation in `eth_feeHistory` Reward Percentile Calculation — (`File: rpc/backend/utils.go`)

### Summary

In `processBlock`, the total Cosmos-tx-level `GasUsed` is attributed to **each individual Ethereum message** inside that tx when building the `sortGasAndReward` slice. For a Cosmos tx carrying N `MsgEthereumTx` messages, the same `txGasUsed` value is inserted N times, inflating `sumGasUsed` by a factor of N. The downstream percentile loop then crosses its `thresholdGasUsed` prematurely, selecting lower-reward transactions for higher percentile buckets and returning systematically wrong `eth_feeHistory` reward data to every caller.

### Finding Description

`processBlock` in `rpc/backend/utils.go` reads the Cosmos-level result gas once per Cosmos tx:

```go
txGasUsed, err := ethermint.SafeUint64(eachTendermintTxResult.GasUsed)
```

Then, for every Ethereum message inside that tx, it appends the **same** `txGasUsed` to the sorter:

```go
for _, msg := range tx.GetMsgs() {
    ethMsg, ok := msg.(*evmtypes.MsgEthereumTx)
    ...
    sorter = append(sorter, txGasAndReward{gasUsed: txGasUsed, reward: reward})
}
``` [1](#0-0) 

`eachTendermintTxResult.GasUsed` is the **aggregate** gas consumed by the whole Cosmos tx (all messages combined). Assigning it to each individual message means a tx with N Ethereum messages contributes `N × txGasUsed` to `sumGasUsed` instead of `1 × txGasUsed`.

The percentile loop then uses this inflated accumulator:

```go
sumGasUsed := sorter[0].gasUsed
for i, p := range rewardPercentiles {
    thresholdGasUsed := uint64(blockGasUsed * p / 100)
    for sumGasUsed < thresholdGasUsed && txIndex < ethTxCount-1 {
        txIndex++
        sumGasUsed += sorter[txIndex].gasUsed
    }
    targetOneFeeHistory.Reward[i] = sorter[txIndex].reward
}
``` [2](#0-1) 

`thresholdGasUsed` is derived from the real block gas (`blockGasUsed`), but `sumGasUsed` grows N× faster than it should. The threshold is crossed after fewer transactions, so lower-reward transactions are selected for higher percentile slots. The result is that `eth_feeHistory` returns reward values that are systematically too low for every percentile when multi-message Cosmos txs are present.

This is structurally identical to the Vader `TwapOracle` bug: two quantities that must be combined **per-element** (`gasUsed` and `reward` for each individual message) are instead accumulated with the wrong granularity — the total tx gas is summed once per message rather than once per tx.

### Impact Explanation

`eth_feeHistory` is the primary source of `maxPriorityFeePerGas` estimates for EIP-1559-compatible wallets (MetaMask, WalletConnect, etc.). Systematically underreported reward percentiles cause wallets to set priority fees below what is actually needed for timely inclusion. Transactions submitted with these underestimated fees may be stuck in the mempool or rejected by nodes enforcing `MinGasPrice`. This is a public JSON-RPC path returning incorrect data that directly influences transaction construction and submission. [3](#0-2) 

### Likelihood Explanation

Any Cosmos tx wrapping more than one `MsgEthereumTx` triggers the bug. The ante handler explicitly supports multi-message Ethereum txs (it iterates over all messages and accumulates `gasWanted`), so such txs are valid and reachable by any unprivileged user. [4](#0-3) 

No special privilege is required: a user simply broadcasts a Cosmos tx containing two or more `MsgEthereumTx` messages. The more messages per tx, the larger the inflation factor and the more distorted the fee history.

### Recommendation

Replace the single `txGasUsed` value (total Cosmos tx gas) with the per-message gas. The per-message gas is emitted as `AttributeKeyTxGasUsed` in the tx events and is already parsed by `ParsedTxs` / `fillTxAttribute` in `rpc/types/events.go`. [5](#0-4) 

Parse the block's tx events into `ParsedTxs` and use `parsedTx.GasUsed` for each individual Ethereum message instead of `eachTendermintTxResult.GasUsed`. Alternatively, if per-message gas is unavailable, divide `txGasUsed` equally among the N messages in the tx as an approximation, though the event-based approach is more accurate.

### Proof of Concept

1. Construct a Cosmos tx containing two `MsgEthereumTx` messages: msg₁ with gas limit 100 000 and msg₂ with gas limit 200 000. Both pay a tip of 1 gwei.
2. Submit the tx. `eachTendermintTxResult.GasUsed` will be ~300 000 (the combined gas).
3. `processBlock` appends `{gasUsed: 300000, reward: 1gwei}` **twice** to the sorter.
4. `sumGasUsed` starts at 300 000 instead of the correct 100 000 for the first message.
5. For a block with `blockGasUsed = 400 000`, the 50th-percentile threshold is 200 000. With correct accounting, `sumGasUsed` would need to advance past msg₁ (100 000) and into msg₂ before crossing 200 000. With the bug, `sumGasUsed` starts at 300 000 and immediately exceeds the threshold, so the 50th-percentile reward is taken from `sorter[0]` (the lowest-reward tx) rather than the correct position.
6. Call `eth_feeHistory` and observe that the returned reward percentiles are lower than the actual tips paid. [6](#0-5)
