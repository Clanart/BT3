### Title
Orphaned Non-EVM Bank Balances Persist After SELFDESTRUCT, Enabling Theft via CREATE2 Redeploy - (File: x/evm/statedb/statedb.go)

### Summary

`StateDB.Commit()` only burns the EVM-native denom balance when processing a self-destructed account. Non-EVM-native Cosmos bank tokens (IBC tokens, CosmWasm bridge tokens) held by the destroyed address are explicitly left as orphaned bank balances. An attacker who controls a CREATE2 factory can exploit this to steal those tokens by redeploying a contract at the same address after self-destructing it.

### Finding Description

In `StateDB.Commit()`, when an account is marked `selfDestructed`, the code burns only the EVM denom balance and then calls `DeleteAccount` (which removes the auth account and EVM storage). The code explicitly acknowledges the gap: [1](#0-0) 

`DeleteAccount` in the keeper removes the auth account and clears EVM storage, but never touches the bank module balances for non-EVM denoms: [2](#0-1) 

The `BankKeeper` interface exposes `GetBalance` per-denom and `SendCoins`/`BurnCoins`, but `DeleteAccount` never calls any of them for non-EVM denoms: [3](#0-2) 

After `DeleteAccount`, the auth account is gone but the Cosmos bank balances for IBC tokens remain at `sdk.AccAddress(addr.Bytes())`. Because CREATE2 address derivation is deterministic, an attacker who controls the factory and salt can redeploy a new contract at the identical address. The new contract's Cosmos address is the same, so it inherits the orphaned bank balances and can drain them via a precompile or native action.

The existing fix for the EVM denom (post-selfdestruct balance burn) confirms the developers understand the pattern, but the fix is incomplete: [4](#0-3) 

### Impact Explanation

An attacker who controls a CREATE2 factory can:
1. Deploy a contract that accepts IBC token deposits from users.
2. Accumulate IBC tokens at the contract's Cosmos address.
3. Call SELFDESTRUCT on the contract (valid under EIP-6780 if created in the same tx, or via a stored selfdestruct path).
4. Redeploy a new contract at the same CREATE2 address in a subsequent transaction.
5. Call a bank precompile or `ExecuteNativeAction` from the new contract to transfer the orphaned IBC tokens to the attacker.

This constitutes unauthorized theft of Cosmos bank funds through Ethermint's stateDB commit logic, matching the Critical allowed impact: *"Unauthorized theft... of... Cosmos bank funds through Ethermint transaction execution or stateDB/native action logic."*

### Likelihood Explanation

The attack requires:
- A contract that holds non-EVM-native bank tokens (IBC tokens are routinely sent to contract addresses on Cosmos chains).
- The attacker controlling the CREATE2 factory (or being the original deployer).
- The ability to trigger SELFDESTRUCT (either the contract exposes it, or the attacker is the deployer).

The most realistic scenario is a malicious deployer who presents a legitimate-looking vault/bridge contract, accumulates IBC tokens from users, then self-destructs and redeploys to drain them. This is a reachable, unprivileged EVM transaction path.

### Recommendation

In `StateDB.Commit()`, before calling `DeleteAccount`, enumerate and burn **all** bank balances held by the self-destructed address, not just the EVM denom. The `BankKeeper` should expose a method to iterate all coins held by an address (e.g., `GetAllBalances`), and each non-zero balance should be sent to the module account and burned, analogous to how the EVM denom is handled at lines 816–820.

### Proof of Concept

```
Tx 1 (attacker):
  factory.attackInOneTx(salt, value=0):
    CREATE2 → childAddr
    childAddr.selfdestruct()
    // IBC tokens previously sent to childAddr remain in bank

Tx 2 (attacker):
  factory.redeployChild(salt):
    CREATE2 → childAddr  // same address, new auth account
    childAddr.callPrecompile(bank.transfer(childAddr, attacker, ibcDenom, amount))
    // orphaned IBC tokens transferred to attacker
```

The existing integration test `SelfDestructExploitFactory.redeployChild` in `tests/integration_tests/hardhat/contracts/SelfDestructExploit.sol` demonstrates the redeploy-and-drain pattern for the EVM denom (now fixed). The same pattern applies to IBC denoms, which remain unaddressed. [5](#0-4)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account

// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
