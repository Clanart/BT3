### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Fund Freezing - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by the caller corresponds to a previously declared class. This is an exact structural analog to the reported "missing registration check" vulnerability: just as the original contract allowed deposits/spins without confirming registration, the OS allows a contract to replace its class with any arbitrary felt value without confirming that class has been declared. A contract that does so becomes permanently uncallable, freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 877–916) processes the `replace_class` syscall. After deducting gas, it reads `class_hash` from the request and directly writes it into the contract's `StateEntry` with no validation against the `contract_class_changes` dictionary:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developer-authored TODO comment at line 898 explicitly acknowledges this missing check. The `contract_class_changes` dictionary — which maps declared class hashes to their compiled class hashes — is never consulted.

The consequence becomes clear in `execute_entry_point.cairo`. When any contract is subsequently called, the OS resolves its class by reading from `contract_class_changes`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [2](#0-1) 

If the class hash was never declared, `dict_read` returns the default value `0`. The OS then calls `find_element` searching for a compiled class with hash `0`:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

Since no compiled class with hash `0` exists, every future call to the contract reverts. The contract is permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract successfully calls `replace_class` with an undeclared class hash and the transaction is committed to state, the contract's `StateEntry.class_hash` is permanently set to an invalid value. No future transaction can execute the contract's code. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are irrecoverably frozen. There is no recovery path: the contract cannot call `replace_class` again because it cannot be called at all.

---

### Likelihood Explanation

- Any deployed contract can issue the `replace_class` syscall — no privileged role is required.
- The attacker only needs to deploy a contract whose code calls `replace_class` with an arbitrary felt (e.g., `0xdeadbeef`).
- The missing check is self-documented in the source with a TODO, confirming it is a known, unimplemented guard.
- The attack is deterministic and requires a single successful transaction.

---

### Recommendation

Before writing the new `class_hash` into the contract's `StateEntry`, verify that it exists in `contract_class_changes` (i.e., has a non-zero compiled class hash mapping). Concretely, add a `dict_read` on `contract_class_changes` keyed by `request.class_hash` and assert the result is non-zero before proceeding with the `dict_update`. This mirrors the prerequisite check that `execute_entry_point` already relies on implicitly.

---

### Proof of Concept

1. **Deploy** a contract whose constructor or any external entry point issues `replace_class(class_hash=0xdeadbeef)` where `0xdeadbeef` is never declared.
2. **Invoke** the contract. The OS calls `execute_replace_class`.
3. `execute_replace_class` deducts gas, reads `class_hash = 0xdeadbeef` from the request, and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes` — **no check against `contract_class_changes`**. [4](#0-3) 
4. The transaction succeeds. The state is committed with `class_hash=0xdeadbeef`.
5. **In any subsequent transaction**, call the contract.
6. `execute_entry_point` calls `dict_read{dict_ptr=contract_class_changes}(key=0xdeadbeef)` → returns `0` (undeclared). [2](#0-1) 
7. `find_element(..., key=0)` finds no matching compiled class → the call reverts with `ERROR_ENTRY_POINT_NOT_FOUND` or panics. [5](#0-4) 
8. The contract is permanently uncallable. All funds in its storage are frozen with no recovery mechanism.
