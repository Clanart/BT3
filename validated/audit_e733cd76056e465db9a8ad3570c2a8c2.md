### Title
Unvalidated WEI Gas Prices in `is_block_info_valid` Allow Proposer to Commit Wrong Block Hash and ETH-Denominated Fees - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

`is_block_info_valid` validates the proposed L1 gas prices in FRI but explicitly discards the locally computed WEI prices, accepting the proposer's `l1_gas_price_wei` / `l1_data_gas_price_wei` without any bounds check. A malicious proposer can set arbitrary WEI values that pass all validation, causing `convert_to_sn_api_block_info` to derive a wrong `eth_to_fri_rate`, a wrong `l2_gas_price_wei`, and ultimately a wrong `gas_prices_hash` inside the block hash commitment — the exact "wrong value dimension" pattern of M-03.

### Finding Description

In `is_block_info_valid`, the validator calls `get_l1_prices_in_fri_and_wei`, which returns both FRI and WEI prices. The WEI component is explicitly discarded via the underscore binding `_l1_gas_prices_wei`. Only the FRI prices are checked against the proposed values via `within_margin`:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;
// ...
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, ...)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, ...))
``` [1](#0-0) 

The proposed WEI prices then flow unchecked into `convert_to_sn_api_block_info`:

1. `l1_gas_price_wei` and `l1_data_gas_price_wei` are used directly as `eth_gas_prices` in `BlockInfo`.
2. `eth_to_fri_rate` is derived as `l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei` from the proposed (unvalidated) WEI value.
3. `l2_gas_price_wei` is derived as `l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)` using that wrong rate. [2](#0-1) [3](#0-2) 

All three values are committed into the block hash via `gas_prices_to_hash`, which for Starknet ≥ 0.13.3 hashes the full tuple `(gas_price_wei, gas_price_fri, data_gas_price_wei, data_gas_price_fri, l2_gas_price_wei, l2_gas_price_fri)`: [4](#0-3) 

`PartialBlockHashComponents::new` pulls these values directly from `BlockInfo.gas_prices`: [5](#0-4) 

The `ProposalInit` struct carries both WEI and FRI fields, and the comment in the code acknowledges the WEI prices are kept "to use with L1 transactions" — confirming they are live execution inputs, not decorative: [6](#0-5) 

### Impact Explanation

A malicious proposer sets `l1_gas_price_wei` to `locally_computed_wei / k` (for any integer `k > 1`) while keeping `l1_gas_price_fri` within the allowed margin. The validation passes. Then:

- `eth_to_fri_rate` becomes `k × correct_rate`, so `l2_gas_price_wei` becomes `correct_l2_wei / k`.
- `eth_gas_prices.l1_gas_price` is `locally_computed_wei / k` instead of the correct value.
- `gas_prices_hash` in the block hash commitment is wrong for all six price fields.

Concrete corrupted values:
- **Wrong block hash** — the `gas_prices_hash` component of `calculate_block_hash` is wrong, producing a block hash that diverges from what honest nodes would compute and from what the SNOS/proof verifier expects.
- **Wrong ETH-denominated fees** — deprecated (v1/v2) transactions use `eth_gas_prices` for fee calculation; with a halved `l1_gas_price_wei`, users pay half the correct ETH fee, corrupting fee-token balance state.
- **Wrong `l2_gas_price_wei`** — stored in the block header and included in the block hash, this derived value is wrong by factor `k`.

This matches the "Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" and "Critical. Wrong state … from blockifier/syscall/execution logic for accepted input" impact categories.

### Likelihood Explanation

Any consensus validator whose turn it is to propose can trigger this. No special privilege beyond being a validator is required. The manipulation is invisible to other validators because the only check they perform on WEI prices is the zero-check warning in `convert_to_sn_api_block_info` (line 290–297), which does not reject the proposal. The attack requires only crafting a `ProposalInit` with a manipulated WEI field.

### Recommendation

In `is_block_info_valid`, use the already-computed `_l1_gas_prices_wei` to validate the proposed WEI prices with the same `within_margin` check applied to FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;
// existing FRI checks ...
if !(within_margin(init_proposed.l1_gas_price_wei, l1_gas_prices_wei.l1_gas_price, margin)
    && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(...);
}
```

This mirrors the existing FRI validation and closes the asymmetric check that is the root cause.

### Proof of Concept

1. Honest validator computes: `l1_gas_price_fri = 1_000_000`, `l1_gas_price_wei = 1_000`.
2. Malicious proposer crafts `ProposalInit` with `l1_gas_price_fri = 1_000_000` (within margin ✓), `l1_gas_price_wei = 500` (half the correct value, unchecked ✓).
3. `is_block_info_valid` passes — only FRI prices are checked.
4. `convert_to_sn_api_block_info` computes `eth_to_fri_rate = 1_000_000 × 10^18 / 500 = 2 × 10^21` (double the correct rate).
5. `l2_gas_price_wei = l2_gas_price_fri / (2 × 10^21)` — half the correct value.
6. `gas_prices_to_hash` hashes `(500, 1_000_000, ...)` instead of `(1_000, 1_000_000, ...)`.
7. Block hash commitment is wrong; ETH-denominated fees for all deprecated transactions in this block are halved; the wrong `l2_gas_price_wei` is stored in the block header and propagated to the next block's `previous_block_info`.
