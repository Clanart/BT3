### Title
Unvalidated `mint` Account in `finalize_transfer` Allows Token Substitution, Causing Permanent Fund Freeze and Misdirected Value — (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

The Solana `FinalizeTransfer` instruction accepts a caller-supplied `mint` account that is never validated against the signed payload from NEAR. Because `FinalizeTransferPayload` contains no token/mint field, any caller who possesses a valid signed payload can substitute an arbitrary registered mint, causing the wrong token to be released on Solana while the correct token remains permanently locked on NEAR.

---

### Finding Description

**Vulnerability class:** Token-mapping / asset-accounting bug — unvalidated token identifier in a finalization path.

**Root cause — `FinalizeTransferPayload` carries no token field:** [1](#0-0) 

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
```

The signed payload that NEAR's MPC authorizes contains only `destination_nonce`, `transfer_id`, `amount`, and `fee_recipient`. There is **no `token` / `mint` field**.

**Root cause — `mint` account constraint is unchecked against the payload:** [2](#0-1) 

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

The only constraint on `mint` is that it belongs to the specified `token_program`. There is no `constraint = mint.key() == data.payload.token_address` or equivalent, because `FinalizeTransferPayload` has no such field to compare against.

**Execution paths that release tokens using the unchecked mint:**

*Native path (vault exists):* [3](#0-2) 

*Bridged path (mint_authority == authority):* [4](#0-3) 

In both paths the program uses `self.mint` — the caller-supplied account — to determine which token to release or mint. The `FinalizeTransferResponse` sent back to NEAR also uses `self.mint.key()` as the authoritative token: [5](#0-4) 

**Contrast with NEAR's `sign_transfer`, which does include the token address in the signed payload:** [6](#0-5) 

NEAR signs a `TransferMessagePayload` that includes `token_address`, but this field is never carried into `FinalizeTransferPayload` and therefore never verified on the Solana side.

---

### Impact Explanation

An attacker who observes a valid signed payload (emitted as an on-chain event on NEAR after `sign_transfer`) can:

1. Submit `finalize_transfer` with the correct signed payload but with `mint` set to a **different registered token** (one that has a vault or whose `mint_authority` is the bridge authority).
2. The program marks `destination_nonce` as used, permanently consuming it.
3. The recipient receives the substituted token (e.g., a low-value SPL token) instead of the intended token (e.g., USDC).
4. The intended tokens remain **permanently locked on NEAR** — the nonce is spent, so the legitimate relayer can never finalize the correct transfer.

This satisfies two allowed impact categories:
- **Critical:** Permanent freezing / irrecoverable lock of user funds.
- **High:** Token-mapping / accounting corruption that breaks bridge collateralization and misdirects value.

---

### Likelihood Explanation

- The signed payload is broadcast publicly as a NEAR event after `sign_transfer` succeeds.
- Any unprivileged actor can call `finalize_transfer` on Solana — no special role is required.
- The attacker only needs to front-run or race the legitimate relayer, which is straightforward on Solana given its public mempool visibility.
- Every cross-chain transfer from NEAR to Solana is a potential target.

---

### Recommendation

Add the token/mint address to `FinalizeTransferPayload` and enforce a PDA or key constraint in the `FinalizeTransfer` accounts struct:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub token: Pubkey,   // ADD: the expected mint
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
```

Then constrain the `mint` account:

```rust
#[account(
    mut,
    constraint = mint.key() == data.payload.token @ ErrorCode::InvalidMint,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

NEAR's `sign_transfer` already includes `token_address` in the payload it signs; the fix is to propagate that field through to `FinalizeTransferPayload` and verify it on the Solana side.

---

### Proof of Concept

1. Alice initiates a transfer of 10,000 USDC from NEAR to Solana. NEAR's `sign_transfer` emits a `SignTransferEvent` containing the MPC-signed `TransferMessagePayload` (includes `token_address = USDC_solana_mint`, `destination_nonce = N`, `amount = 10000`).

2. The relayer serializes this into a `SignedPayload<FinalizeTransferPayload>` where `FinalizeTransferPayload = { destination_nonce: N, transfer_id: ..., amount: 10000, fee_recipient: ... }` — **no token field**.

3. Attacker observes the signed payload on-chain and calls `finalize_transfer` before the legitimate relayer, passing:
   - `data` = the valid signed payload (unchanged)
   - `mint` = `BONK_mint` (a registered low-value token with a vault)
   - `vault` = the BONK vault PDA

4. The program passes all checks (valid signature, nonce not yet used, vault exists for BONK), transfers 10,000 BONK from the BONK vault to Alice, marks nonce `N` as used, and posts a `FinalizeTransferResponse { token: BONK_mint, amount: 10000, ... }` back to NEAR.

5. The legitimate relayer's subsequent attempt to finalize with USDC fails: `ERR_NONCE_ALREADY_USED`.

6. Alice's 10,000 USDC is permanently locked on NEAR. Alice received 10,000 BONK (worth a fraction of the USDC). The BONK vault is drained by the amount. Bridge accounting is corrupted.
