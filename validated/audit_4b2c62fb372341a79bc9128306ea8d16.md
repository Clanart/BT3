### Title
Cross-Chain Replay of `deployToken`/`deploy_token` Signature Due to Missing Chain ID in Metadata Message Hash — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The Borsh-encoded message signed by the NEAR MPC for `deployToken`/`deploy_token` does not include any destination chain identifier. The message format is byte-for-byte identical across EVM, Starknet, and Solana. Because the same NEAR MPC-derived key (`nearBridgeDerivedAddress`) is used on all chains, a valid `deployToken` signature obtained from one chain can be replayed on any other chain to deploy the same token there without authorization. This permanently blocks the legitimate deployment of that token on the target chain.

---

### Finding Description

**EVM** (`OmniBridge.sol`, `deployToken`):

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),  // = 0x01
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

**Starknet** (`bridge_types.cairo`, `MetadataPayloadTrait::to_borsh`):

```cairo

    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [2](#0-1) 

**Solana** (`deploy_token.rs`, `DeployTokenPayload::serialize_for_near`):

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
    IncomingMessageType::Metadata.serialize(&mut writer)?;  // = 0x01
    self.serialize(&mut writer)?; // token, name, symbol, decimals
    ...
}
``` [3](#0-2) 

All three produce the identical byte sequence: `[0x01 || token || name || symbol || decimals]`. No chain ID, no contract address, no nonce is included. The `IncomingMessageType::Metadata` variant is `1` (second variant of a 0-indexed enum), matching `PayloadType::Metadata = 1` on EVM and Starknet. [4](#0-3) 

Contrast this with `finTransfer`/`fin_transfer`, which correctly binds the message to the destination chain by including `omniBridgeChainId` twice in the Borsh encoding:

```solidity
bytes1(omniBridgeChainId),   // destination chain for token address
...
bytes1(omniBridgeChainId),   // destination chain for recipient
``` [5](#0-4) 

The Starknet `fin_transfer` similarly passes `chain_id` into `to_borsh(self.omni_bridge_chain_id.read())`: [6](#0-5) 

The `deployToken` path has no equivalent binding.

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (e.g., EVM) can extract the signature and replay it on chain B (e.g., Starknet or Solana) with the same `MetadataPayload`. The signature verification passes because the hash is identical and the same MPC-derived key is used everywhere.

On Starknet, `deploy_token` sets `near_to_starknet_token[token_id_hash] = contract_address` and asserts `ERR_TOKEN_ALREADY_DEPLOYED` on any subsequent attempt: [7](#0-6) 

There is no admin override or re-deployment path on Starknet. The token mapping is permanently poisoned with the attacker-deployed contract address. NEAR's legitimate deployment attempt for that token on Starknet will always revert. Users can never bridge that token to Starknet — a permanent, irrecoverable lock of that token's bridging capability on the chain.

On EVM, the `ERR_TOKEN_EXIST` guard similarly blocks re-deployment: [8](#0-7) 

An admin can partially recover via `addCustomToken` (which overwrites `nearToEthToken`), but this requires privileged intervention and the attacker-deployed proxy token persists.

---

### Likelihood Explanation

The attack requires no privileged access. Any public bridge user can:
1. Watch the mempool or block explorer on any chain for a `deployToken` transaction.
2. Extract the `signatureData` and `MetadataPayload` from the calldata.
3. Submit the same call to any other chain's bridge contract.

The only precondition is that the target token has not yet been deployed on the target chain, which is the normal state before NEAR initiates deployment. The attacker races NEAR's own deployment transaction.

---

### Recommendation

Include the destination chain identifier in the `deployToken` message hash, mirroring the pattern already used in `finTransfer`. Specifically:

- **EVM**: Append `bytes1(omniBridgeChainId)` to the `borshEncoded` buffer before hashing in `deployToken`.
- **Starknet**: Pass `self.omni_bridge_chain_id.read()` into `MetadataPayloadTrait::to_borsh` and append it to the Borsh bytes, matching the pattern in `TransferMessagePayloadTrait::to_borsh`.
- **Solana**: Write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the serialized buffer in `DeployTokenPayload::serialize_for_near`, as is already done in `DeployTokenResponse::serialize_for_near` (line 47). [9](#0-8) 

The NEAR MPC signing logic must also be updated to include the destination chain ID when constructing the `MetadataPayload` hash.

---

### Proof of Concept

1. NEAR MPC signs a `MetadataPayload` for token `"usdc.near"` (name `"USD Coin"`, symbol `"USDC"`, decimals `6`) destined for EVM. The signed hash is `keccak256([0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06])`.

2. Attacker observes the EVM `deployToken` transaction and extracts `signatureData`.

3. Attacker calls `deploy_token(signature, MetadataPayload { token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 })` on the Starknet bridge.

4. Starknet's `_verify_borsh_signature` computes `keccak256([0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06])` — identical hash — and verifies against `omni_bridge_derived_address`. Verification passes.

5. `near_to_starknet_token[keccak("usdc.near")]` is set to the attacker-triggered deployment address.

6. When NEAR later legitimately attempts to deploy `"usdc.near"` on Starknet, `assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED')` reverts. The token is permanently undeployable on Starknet via the normal bridge flow.
