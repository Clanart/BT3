### Title
Partial Token Return from `ft_transfer_call` Permanently Locks Tokens in NEAR Bridge — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a recipient contract partially uses tokens sent via `ft_transfer_call` (returning some but not all tokens), the returned tokens are permanently locked in the NEAR bridge contract. The `is_refund_required` helper only handles the binary case — all tokens used or all tokens returned — and silently discards the partial-return case, leaving the returned tokens irrecoverable.

---

### Finding Description

The `send_tokens` function dispatches `ft_transfer_call` (instead of `ft_transfer`) whenever the `msg` field is non-empty: [1](#0-0) 

Under NEP-141, the recipient's `ft_on_transfer` may return a non-zero value indicating unused tokens. The token contract then refunds those tokens back to the bridge contract. The bridge's callback logic relies on `is_refund_required` to decide whether to propagate a refund: [2](#0-1) 

`is_refund_required` reads the promise result of `ft_transfer_call`, which in NEAR is the amount **used** (= sent − returned). It returns `true` only when `amount.0 == 0`, i.e., when **zero** tokens were used (all returned). For any partial return — where the recipient uses some tokens and returns the rest — `amount.0 > 0`, so `is_refund_required` returns `false`.

In `resolve_fast_transfer` (the callback for the fast-transfer path): [3](#0-2) 

When `is_refund_required` is `false`, the function returns `U128(0)`, signalling to the relayer's `ft_transfer_call` that all tokens were consumed. The N returned tokens now sit in the bridge contract with no accounting entry and no withdrawal path.

The same pattern applies in `fin_transfer_send_tokens_callback` (the callback for the standard `fin_transfer` path), which also delegates to `is_refund_required` and takes no action on partial returns.

The `locked_tokens` counter for the origin chain was already decremented by the full transfer amount at the time the transfer was finalised: [4](#0-3) 

So after a partial return the bridge's on-chain token balance exceeds what `locked_tokens` tracks, and the surplus N tokens have no owner and no recovery function.

The integration test in `near/omni-tests/src/fin_transfer.rs` explicitly demonstrates this outcome — when the recipient returns 1 token, `expected_locker_balance: 1` confirms the token is stranded in the bridge: [5](#0-4) 

---

### Impact Explanation

Tokens returned by a recipient contract after a partial `ft_on_transfer` are permanently locked inside the NEAR bridge contract. There is no admin withdrawal, no sweep function, and no re-crediting of the sender or the `locked_tokens` ledger. This constitutes an irrecoverable lock of user or protocol funds in the bridge vault flow, matching the **Critical** impact class: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

The trigger conditions are:

1. A cross-chain transfer is initiated with a non-empty `msg` field (user-controlled on the origin chain).
2. The NEAR recipient is a contract whose `ft_on_transfer` returns a non-zero amount (e.g., a DeFi protocol that rejects part of the deposit, or a contract that enforces a minimum deposit size).

Both conditions are realistic in production. Any bridge user can supply a `msg` when initiating a transfer, and the NEAR ecosystem has many token-receiver contracts that partially consume tokens. No privileged role is required.

---

### Recommendation

1. **Track the partial return**: In `fin_transfer_send_tokens_callback` and `resolve_fast_transfer`, read the actual used amount from the promise result and compute `returned = sent - used`. Handle `returned > 0` explicitly.
2. **Burn or re-lock returned tokens**: For deployed (bridged) tokens, burn the returned amount. For locked (native) tokens, re-credit the `locked_tokens` counter and either return the tokens to the original sender or hold them in a claimable escrow.
3. **Extend `is_refund_required`** to return the actual returned amount rather than a boolean, so callers can act on partial returns instead of silently ignoring them.

---

### Proof of Concept

**Setup**: Deploy a NEAR token receiver contract whose `ft_on_transfer` always returns `U128(1)` (1 token unused).

**Steps**:
1. On EVM, call `initTransfer` locking 1000 tokens with `msg = <encoded receiver message>` and recipient = the malicious NEAR contract.
2. A relayer submits the proof via `fin_transfer` on NEAR.
3. `fin_transfer_callback` calls `process_fin_transfer_to_near`, which calls `send_tokens` → `ft_transfer_call(recipient, 1000, msg)`.
4. The recipient's `ft_on_transfer` returns `U128(1)`. The token contract refunds 1 token to the bridge.
5. `fin_transfer_send_tokens_callback` is invoked. `is_refund_required` reads the used amount = 999 ≠ 0, returns `false`. No action is taken on the 1 returned token.
6. The bridge's `locked_tokens[Eth][token]` was already decremented by 1000. The bridge now holds 1 extra token with no accounting entry and no recovery path — permanently locked.

This is confirmed by the existing integration test expectation `expected_locker_balance: 1` when `return_value: U128(1)`. [5](#0-4) [2](#0-1) [3](#0-2) [1](#0-0)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0

                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
