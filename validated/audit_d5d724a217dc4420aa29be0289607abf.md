### Title
Committed Finalization State Not Reverted When `send_tokens` Fails for Non-`ft_transfer_call` Cases — (`File: near/omni-bridge/src/lib.rs`)

### Summary

In `process_fin_transfer_to_near`, critical state mutations (marking the transfer as finalized, decrementing `locked_tokens`, deducting storage balance) are committed synchronously inside `fin_transfer_callback` before the asynchronous token delivery. If the subsequent `send_tokens` call fails — for example, when `near_withdraw` on the wNEAR contract fails — the callback `fin_transfer_send_tokens_callback` is invoked with a failed promise result but **never reverts the committed state**, because `is_refund_required` unconditionally returns `false` for non-`ft_transfer_call` paths. The transfer is permanently finalized, the user never receives tokens, and the locked-token accounting is permanently corrupted.

---

### Finding Description

**Step 1 — State committed synchronously in `fin_transfer_callback`**

`fin_transfer_callback` calls `process_fin_transfer_to_near`, which synchronously:
- Calls `add_fin_transfer` → inserts the transfer ID into `finalised_transfers` (replay guard, cannot be re-submitted)
- Calls `unlock_tokens_if_needed` → decrements `locked_tokens[(origin_chain, token)]`
- Calls `update_storage_balance` → deducts storage from the relayer's balance

These state changes are committed when `fin_transfer_callback` returns its Promise. They cannot be rolled back by a later panic in a child callback. [1](#0-0) [2](#0-1) 

**Step 2 — `send_tokens` for wNEAR creates a two-step async chain**

For wNEAR (token == `wnear_account_id`, empty `msg`), `send_tokens` returns the chain:

```
near_withdraw  →  near_withdraw_callback  →  fin_transfer_send_tokens_callback
``` [3](#0-2) 

**Step 3 — `near_withdraw_callback` panics on failure**

If `near_withdraw` fails (e.g., bridge's wNEAR FT balance is insufficient), `near_withdraw_callback` calls `env::panic_str`. In NEAR's async model, this panic reverts only the state changes inside `near_withdraw_callback` itself (there are none). The committed state from Step 1 is unaffected. The next callback in the chain, `fin_transfer_send_tokens_callback`, is still invoked with a failed promise result. [4](#0-3) 

**Step 4 — `fin_transfer_send_tokens_callback` never reverts for non-`ft_transfer_call` paths**

`fin_transfer_send_tokens_callback` delegates to `is_refund_required(is_ft_transfer_call)`. For wNEAR and plain `ft_transfer` paths, `msg` is empty, so `is_ft_transfer_call = false`. `is_refund_required(false)` unconditionally returns `false` without inspecting the promise result at all. [5](#0-4) [6](#0-5) 

The revert path (`revert_lock_actions`, `remove_fin_transfer`) is therefore never reached. The else branch executes fee-sending logic and logs a success event, even though the user received nothing. [7](#0-6) 

The same desync applies to plain `ft_transfer` (non-wNEAR, non-deployed token, empty `msg`): if `ft_transfer` fails, `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false` and still does not revert. [8](#0-7) 

---

### Impact Explanation

- **Permanent freezing of user funds**: The transfer ID is in `finalised_transfers`; the same proof cannot be re-submitted. The user's bridged NEAR (or other non-deployed token) is irrecoverably lost.
- **Locked-token accounting corruption**: `locked_tokens[(origin_chain, token)]` is permanently decremented even though the tokens were never delivered. This breaks bridge collateralization for subsequent transfers of the same token.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

`near_withdraw` fails if the bridge contract's wNEAR FT balance in the wNEAR contract is less than the requested amount. This can occur when:

1. A prior bug or race condition causes `locked_tokens` to diverge from the actual FT balance held by the bridge.
2. The wNEAR contract itself reverts (e.g., paused, storage issue, or the bridge's storage registration lapses).
3. An attacker exploits another accounting flaw to drain the bridge's wNEAR balance below the tracked `locked_tokens` value, then triggers a large finalization to hit this path.

The same applies to plain `ft_transfer` for any non-deployed token whose actual balance falls below `locked_tokens`. Likelihood is **Medium**.

---

### Recommendation

`fin_transfer_send_tokens_callback` must inspect the promise result regardless of `is_ft_transfer_call`. Add a check for the failed promise result and, on failure, call `revert_lock_actions` and `remove_fin_transfer` to restore state:

```rust
// Pseudocode fix
if Self::is_refund_required(is_ft_transfer_call)
    || (env::promise_results_count() > 0
        && env::promise_result_checked(0, usize::MAX).is_err())
{
    // revert path
    self.revert_lock_actions(&lock_actions);
    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
    ...
}
```

For the wNEAR path specifically, `near_withdraw_callback` should avoid panicking and instead return a value indicating failure, so that `fin_transfer_send_tokens_callback` can detect and handle it cleanly.

---

### Proof of Concept

1. User bridges 100 wNEAR from Ethereum → NEAR. The bridge's wNEAR FT balance is 99 (e.g., due to a prior accounting discrepancy).
2. Relayer calls `fin_transfer` with a valid proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_near` runs synchronously:
   - `add_fin_transfer` inserts the transfer ID into `finalised_transfers`. [9](#0-8) 
   - `unlock_tokens_if_needed(Eth, wNEAR, 100)` decrements `locked_tokens[(Eth, wNEAR)]` by 100. [10](#0-9) 
4. `send_tokens` schedules `near_withdraw(100)` on the wNEAR contract.
5. `near_withdraw` fails (bridge only has 99 wNEAR).
6. `near_withdraw_callback` is called with a failed result → panics. [11](#0-10) 
7. `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false`.
8. `is_refund_required(false)` returns `false`. No revert. [12](#0-11) 
9. Result: transfer permanently finalized, `locked_tokens` permanently decremented by 100, user receives 0 NEAR.

impl Contract {
    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
    }
