### Title
`batch_insert` Skips Duplicate-Key/Hash Validation, Corrupting DataLayer Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. Supplying a batch that contains a key or hash already present in the tree (or a duplicate within the batch itself) silently writes corrupt leaf nodes into the blob, producing an incorrect Merkle root and invalidating all subsequent proofs of inclusion/exclusion.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) opens with two mandatory pre-condition checks:

```rust
// blob.rs lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a completely different code path for all items beyond the first two (or for every item when the tree already has ≥ 2 leaves). It calls `insert_entry_to_blob` directly, with no key-presence or hash-presence check:

```rust
// blob.rs lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` unconditionally writes the block to the blob and calls `block_status_cache.add_leaf`, which overwrites the cache entry for that key with the new index:

```rust
// blob.rs lines 1024-1027
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    Node::Internal(..) => self.block_status_cache.add_internal(index),
}
``` [3](#0-2) 

The result is that the blob now contains two physical leaf nodes for the same key, while the cache points only to the second one. The tree structure (parent/child links) is built over both leaves, so the computed internal-node hashes — and ultimately the root hash — incorporate the phantom duplicate leaf.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
// blob.rs lines 1503-1519
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

---

### Impact Explanation

The Merkle root stored after a corrupt `batch_insert` does not represent the actual key-value set. Consequences:

1. **Forged inclusion proofs** — a proof path can be constructed through the phantom duplicate leaf, making `get_proof_of_inclusion` return a structurally valid proof for a key/value pair that was never legitimately inserted, or for a stale value that was superseded.
2. **Corrupted tree root committed on-chain** — DataLayer stores the Merkle root on the Chia blockchain. A corrupt root means the on-chain commitment no longer matches the actual data, breaking the integrity guarantee that DataLayer provides to all consumers of that store.
3. **`check_integrity` divergence** — `check_just_integrity` compares `leaf_count` (physical leaves in the blob) against `key_to_index_cache_length` (unique keys in the cache). After a duplicate insert these differ, causing `IntegrityKeyToIndexCacheLength` — a committed state corruption detectable only after the fact. [5](#0-4) 

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is the primary bulk-load path used during DataLayer delta synchronization. Any node that receives a delta from an untrusted peer and feeds it into `batch_insert` without pre-deduplication is vulnerable. The Python API surface (`py_batch_insert`) makes this reachable from application-layer code without any special privilege. No key material, governance role, or network-level attack is required — only the ability to supply a list of `(KeyId, ValueId, Hash)` tuples that contains a duplicate.

---

### Recommendation

Add the same pre-condition guards at the top of the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, deduplicate the input vector before processing and return an error if duplicates are detected, consistent with the single-insert contract.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    // Start with a tree that already has 2 leaves so batch_insert
    // skips the `insert()` path entirely.
    let mut blob = MerkleBlob::new(Vec::new()).unwrap();
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i),
                    &Hash(Bytes32::new([i as u8; 32])),
                    InsertLocation::Auto {}).unwrap();
    }

    // Now batch-insert a key that already exists (KeyId(0)).
    // No error is returned — the duplicate is silently accepted.
    let dup_key = KeyId(0);
    let dup_hash = Hash(Bytes32::new([99u8; 32]));
    blob.batch_insert(vec![((dup_key, ValueId(99)), dup_hash)]).unwrap();

    // The blob now has 3 physical leaf nodes but only 2 unique keys.
    // check_integrity() will report IntegrityKeyToIndexCacheLength,
    // confirming the tree is corrupt.
    blob.check_integrity().unwrap_err(); // panics / errors here
}
```

The root hash computed after this call incorporates the phantom leaf, so any proof of inclusion derived from it is based on a corrupted commitment.
