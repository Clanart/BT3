### Title
`ProofOfInclusion::valid()` Does Not Verify Against a Trusted Root Hash — Forged Proofs Always Pass - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary
`ProofOfInclusion::valid()` in the DataLayer crate only checks internal self-consistency of the proof chain. It never compares the computed root against any externally-supplied trusted root hash. An attacker can construct a completely fabricated `ProofOfInclusion` — with an arbitrary `node_hash` representing a key that does not exist in the real tree — and `valid()` will return `true`, allowing forged inclusion proofs to pass verification.

### Finding Description

The `ProofOfInclusion` struct holds a `node_hash` (the leaf being proven) and a `Vec<ProofOfInclusionLayer>`, where each layer carries `other_hash_side`, `other_hash`, and `combined_hash`. [1](#0-0) 

The `root_hash()` helper derives the root entirely from the proof itself: [2](#0-1) 

The `valid()` method then:
1. Walks each layer, computing `calculated_hash = calculate_internal_hash(existing_hash, layer.other_hash_side, layer.other_hash)`.
2. Returns `false` if `calculated_hash != layer.combined_hash`.
3. After the loop, checks `existing_hash == self.root_hash()`. [3](#0-2) 

The final check `existing_hash == self.root_hash()` is a **tautology**. After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `last.combined_hash` inside the loop. `self.root_hash()` returns exactly `last.combined_hash`. So the final comparison is always `last.combined_hash == last.combined_hash` — unconditionally `true`.

**No external trusted root is ever consulted.** An attacker can craft a `ProofOfInclusion` where:
- `node_hash` is any arbitrary 32-byte value (e.g., a key-value pair that does not exist in the real tree).
- Each `ProofOfInclusionLayer` is constructed so that `combined_hash = calculate_internal_hash(prev_hash, chosen_side, chosen_other_hash)` — a trivially satisfiable constraint.

Such a proof passes `valid()` with `true` despite proving inclusion of a non-existent entry.

The struct is `Streamable` and fully exposed via Python bindings: [4](#0-3) [5](#0-4) 

This means any Python DataLayer client that receives a `ProofOfInclusion` over the network and calls `proof.valid()` to verify it is fully vulnerable to forged proofs.

By contrast, the `chia-consensus` `MerkleSet` proof system correctly validates against an external root: [6](#0-5) 

The DataLayer `ProofOfInclusion` has no equivalent parameter.

### Impact Explanation

Any DataLayer peer or client that receives a `ProofOfInclusion` from an untrusted source and calls `proof.valid()` to decide whether a key-value pair is committed in the DataLayer tree will accept a completely fabricated proof. This allows an attacker to prove inclusion of arbitrary key-value pairs that do not exist in the actual tree, enabling invalid state to be accepted as proven. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` (deserializable from bytes), exposed via Python bindings, and the `valid()` method is the sole public API for verifying a proof. Any DataLayer application that receives proofs from peers and calls `valid()` — the natural and documented usage — is affected. The attacker only needs to construct a self-consistent hash chain, which requires no cryptographic secrets and is trivially computable.

### Recommendation

`valid()` must accept a trusted root hash as a parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == *trusted_root  // compare against externally-supplied trusted root
}
```

The no-argument `valid()` should either be removed or clearly documented as an internal-consistency-only check that must never be used for security decisions. All call sites in the Python bindings and Rust API should be updated to require the caller to supply the trusted root obtained from a verified on-chain commitment.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from chia_rs import calculate_internal_hash  # or equivalent

# Fabricate a leaf hash for a key that does NOT exist in any real tree
fake_node_hash = bytes([0xAB] * 32)

# Build a single-layer proof: combined_hash = H(fake_node_hash, side, other_hash)
other_hash = bytes([0xCD] * 32)
side = 0  # Left
combined_hash = calculate_internal_hash(fake_node_hash, side, other_hash)

layer = ProofOfInclusionLayer(
    other_hash_side=side,
    other_hash=other_hash,
    combined_hash=combined_hash,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True despite the proof being entirely fabricated
assert forged_proof.valid() == True
# root_hash() returns the attacker-chosen combined_hash, not any real tree root
print(forged_proof.root_hash())  # attacker-controlled value
```

The proof passes `valid()` with no connection to any real `MerkleBlob` tree state.
