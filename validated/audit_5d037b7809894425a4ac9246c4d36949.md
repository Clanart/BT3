### Title
`ProofOfInclusion::valid()` Never Verifies Against an External Trusted Root — Forged DataLayer Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only verifies internal chain consistency of the proof's own fields. The final equality check is a tautology: after the loop, `existing_hash` is always equal to `self.root_hash()` by construction, so the function never validates the proof against any external trusted root. An attacker can craft a `ProofOfInclusion` for any arbitrary key-value pair, with a fabricated `node_hash` and a self-consistent chain of layers, and `valid()` will return `true`. Any verifier that relies solely on `proof.valid()` to accept a DataLayer inclusion proof from an untrusted source will accept forged state.

### Finding Description

`ProofOfInclusion` is a `Streamable` type exposed via Python bindings, meaning it can be serialized, transmitted over the network, and deserialized by a peer. The `valid()` method is the only verification interface provided:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs

pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // root is taken FROM the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()      // TAUTOLOGY: self.root_hash() == last.combined_hash == existing_hash
}
```

**The tautology:** After each loop iteration, `existing_hash` is set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. After the final iteration, `existing_hash` equals the last `layer.combined_hash`. `self.root_hash()` returns exactly `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` once the loop completes without an early return. The function never compares against any externally supplied, trusted root hash.

**Exploit path:**

1. Attacker wants to prove that key `K` maps to value `V` in a DataLayer store whose actual root hash is `R`.
2. Attacker constructs a `ProofOfInclusion` with:
   - `node_hash` = SHA256 of any fabricated leaf for `(K, V)`
   - A single layer: `other_hash` = any 32-byte value, `other_hash_side` = Left, `combined_hash` = `calculate_internal_hash(node_hash, Left, other_hash)` (attacker computes this themselves)
3. Attacker serializes and sends this `ProofOfInclusion` to a verifier.
4. Verifier calls `proof.valid()` → returns `true`.
5. Verifier accepts the forged inclusion proof.

The `ProofOfInclusion` struct is `Streamable` and fully constructible from Python via `ProofOfInclusion(node_hash=..., layers=[ProofOfInclusionLayer(...)])`, making this trivially reachable from any unprivileged network input.

### Impact Explanation

**High.** DataLayer Merkle proof logic accepts forged inclusion proofs, letting untrusted input prove invalid state. Any consumer that calls `proof.valid()` as the sole check — which is the only verification method the API exposes — will accept a proof for any key-value pair the attacker chooses, regardless of the actual tree state. This enables an attacker to convince a DataLayer verifier that arbitrary data is committed in a store when it is not.

### Likelihood Explanation

**High.** The `valid()` method is the only verification method on `ProofOfInclusion`. There is no `verify(root: Hash)` method. The struct is `Streamable` and exposed via Python bindings with a `from_bytes` constructor. Any DataLayer peer that receives a proof from an untrusted source and calls `proof.valid()` is vulnerable. The exploit requires no privileged access, no keys, and no special network position — only the ability to send a serialized `ProofOfInclusion` to a verifier.

### Recommendation

Replace the tautological final check with a comparison against a caller-supplied trusted root:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // compare against externally trusted root
}
```

Deprecate or remove the current `valid()` method, or make it explicitly document that it only checks internal consistency and must not be used as a security check. Update the Python binding to expose `valid_against_root(root: bytes32) -> bool`.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Fabricate a leaf hash for any key-value pair
fake_node_hash = bytes(range(32))  # attacker-chosen

# Compute a consistent layer (attacker controls all values)
other_hash = bytes([0xAB] * 32)
# calculate_internal_hash(fake_node_hash, Left=0, other_hash)
combined = hashlib.sha256(b"\x01" + fake_node_hash + other_hash).digest()  # simplified

layer = ProofOfInclusionLayer(
    other_hash_side=0,           # Left
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True despite no connection to any real tree root
assert forged_proof.valid() == True
# root_hash() returns the attacker-controlled combined_hash, not any real tree root
assert forged_proof.root_hash() == combined
```
