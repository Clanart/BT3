### Title
Queued Future-Round Proposal Timestamp Validated Against Dequeue-Time Clock, Bypassing `block_timestamp_window_seconds` and Corrupting Block Hash Commitment - (File: crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs)

### Summary

When a `ProposalInit` arrives for a future consensus round, it is stored in `queued_proposals` with no timestamp validation. When the round later advances and the proposal is dequeued, `is_block_info_valid` evaluates the timestamp against `clock.unix_now()` at dequeue time. A proposer can therefore embed a `block_timestamp` that exceeds `now_at_queue + block_timestamp_window_seconds` — a value that would have been rejected immediately — and have it accepted once wall-clock time catches up. The accepted timestamp is then committed into `PartialBlockHashComponents`, the block hash, and the execution `BlockInfo`, producing a wrong authoritative block hash and wrong `get_block_timestamp()` results for all transactions in that block.

### Finding Description

In `validate_proposal` in `sequencer_consensus_context.rs`, when `init.round > self.current_round`, the proposal is inserted into `self.queued_proposals` with **no timestamp check**: [1](#0-0) 

When `set_height_and_round` later advances to that round, it dequeues the proposal and calls `validate_current_round_proposal`, which eventually calls `is_block_info_valid`. Inside `is_block_info_valid`, the upper-bound check uses `clock.unix_now()` evaluated at **dequeue time**, not at queue time: [2](#0-1) 

A proposer who sets `timestamp = T_queue + block_timestamp_window_seconds + Δ` (Δ > 0) for a future round will have the proposal queued. After Δ seconds pass (when the round advances), `now_dequeue ≥ T_queue + Δ`, so the check becomes:

```
T_queue + window + Δ  ≤  (T_queue + Δ) + window   ✓ passes
```

The dequeue path that performs this late check is: [3](#0-2) 

The accepted `init.timestamp` then flows into three critical paths:

1. **Block hash commitment** — `convert_to_sn_api_block_info` maps `init.timestamp` to `BlockInfo::block_timestamp`, which is captured in `PartialBlockHashComponents::timestamp` and hashed into the block hash: [4](#0-3) [5](#0-4) [6](#0-5) 

2. **Execution context** — `BlockInfo::block_timestamp` is the value returned to contracts by the `get_block_timestamp` syscall, affecting all time-dependent contract logic in the block.

3. **L1 gas price calculation** — `is_block_info_valid` passes `init_proposed.timestamp` to `get_l1_prices_in_fri_and_wei`, so a future timestamp causes the L1 gas price lookup to use a wrong time reference: [7](#0-6) 

The block header written to state sync also carries this timestamp directly: [8](#0-7) 

### Impact Explanation

- **Critical — Wrong execution result**: Every transaction in the block that calls `get_block_timestamp()` receives a value that is `block_timestamp_window_seconds + Δ` seconds ahead of the real wall clock, corrupting time-dependent contract logic (e.g., deadline checks, vesting schedules, oracle freshness guards).
- **Critical — Wrong block hash committed to storage and proofs**: `block_timestamp` is a direct input to `calculate_block_hash`. A manipulated timestamp produces a wrong `BlockHash` stored in storage and fed to the SNOS/proof pipeline, breaking the chain of block hash commitments.
- **Critical — Incorrect L1 gas price / fee effect**: The L1 gas price is looked up using the manipulated future timestamp, producing wrong fee calculations with direct economic impact.

### Likelihood Explanation

The proposer must be the designated proposer for a future round. In Tendermint-style consensus, proposals for future rounds are unusual but explicitly handled by the code (the queue exists precisely for this case). The attack requires:
1. Being the designated proposer for round R+1.
2. Sending a `ProposalInit` for round R+1 with `timestamp = now + block_timestamp_window_seconds + Δ`.
3. Waiting Δ seconds for the round to advance (which happens naturally on timeout).

No special privileges beyond being a scheduled proposer are required. The window `block_timestamp_window_seconds` is a static config value, so the attacker can compute the exact Δ needed.

### Recommendation

Capture the wall-clock time at **queue time** and store it alongside the queued proposal. When the proposal is dequeued, use the stored queue-time `now` (not the current `now`) for the upper-bound timestamp check. Alternatively, perform the full `is_block_info_valid` check at queue time and reject proposals with out-of-window timestamps immediately, before inserting them into `queued_proposals`.

### Proof of Concept

1. Consensus is at height H, round 0. Current wall-clock time is `T`. `block_timestamp_window_seconds = W`.
2. The malicious proposer (designated for round 1) sends `ProposalInit { height: H, round: 1, timestamp: T + W + 60, ... }`.
3. `validate_proposal` sees `round (1) > current_round (0)` → inserts into `queued_proposals` with no timestamp check. [1](#0-0) 
4. Round 0 times out after 60 seconds. `set_height_and_round(H, 1)` is called at wall-clock time `T + 60`.
5. The proposal is dequeued. `is_block_info_valid` runs with `now = T + 60`:
   - Check: `T + W + 60 ≤ (T + 60) + W` → `T + W + 60 ≤ T + W + 60` → **passes**. [9](#0-8) 
6. The block is committed with `block_timestamp = T + W + 60` — `W + 60` seconds ahead of the real time at queue.
7. The block hash includes this manipulated timestamp: [6](#0-5) 
8. All contracts in this block calling `get_block_timestamp()` receive `T + W + 60` instead of `≈ T + 60`, corrupting time-dependent logic. The wrong block hash propagates to storage, state sync, and the proof pipeline.
