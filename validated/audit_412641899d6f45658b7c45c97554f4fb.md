The vulnerability is real. Here is the complete analysis:

---

### Title
Excess ETH Sent to `exactInputSingle` Is Permanently Stranded and Claimable by Any Caller via `refundETH` — (`metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`exactInputSingle` (and all other payable swap entry points) accept arbitrary `msg.value` but never auto-refund excess ETH. The `pay()` WETH branch wraps exactly `amountIn` ETH and leaves the remainder in the router. Because `refundETH()` is permissionless and sends `address(this).balance` to `msg.sender`, any subsequent caller can drain the stranded ETH.

---

### Finding Description

**Step 1 — ETH enters the router.**

`exactInputSingle` is `payable` and stores `msg.sender` as the payer in transient storage. It does not record `msg.value` or enforce that `msg.value == amountIn`. [1](#0-0) 

**Step 2 — `pay()` wraps exactly `value` ETH, leaving the rest.**

When `token == WETH` and `nativeBalance >= value`, the WETH branch wraps exactly `value` wei and transfers WETH to the pool. The surplus `nativeBalance - value` ETH is never touched. [2](#0-1) 

**Step 3 — `refundETH()` is permissionless and sends to `msg.sender`.**

`refundETH` transfers the entire native balance of the router to whoever calls it — not to the original depositor. [3](#0-2) 

**Step 4 — No auto-refund exists in any swap entry point.**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all end with `_clearExpectedCallbackPool()` and return — no ETH refund is issued. [4](#0-3) 

The codebase's own test file documents that the intended safe pattern is `multicall{value}([exactInputSingle, refundETH])`: [5](#0-4) 

However, `exactInputSingle` is also directly callable with `msg.value` (as shown in `test_mixedNativeAndWeth_exactInputSingle_wethForToken`), and there is no guard preventing a user from sending excess ETH in a direct call. [6](#0-5) 

---

### Impact Explanation

A user who calls `exactInputSingle{value: 200}(amountIn: 100, tokenIn: WETH)` directly (not via multicall) loses 100 ETH permanently. The stranded ETH sits in the router until any address calls `refundETH()`, at which point the caller receives the full balance. This is direct, irreversible loss of user principal with no privileged access required.

---

### Likelihood Explanation

The router exposes `exactInputSingle` as a standard `payable` external function. Users and integrators who are accustomed to Uniswap v3-style "send ETH, get tokens" UX will naturally call it directly with `msg.value > amountIn` (e.g., to avoid a separate approval). The multicall+refundETH requirement is only documented in a test-file comment, not enforced on-chain. Likelihood is medium-to-high given the intuitive but incorrect usage pattern.

---

### Recommendation

Add an auto-refund at the end of every payable swap entry point:

```solidity
function exactInputSingle(ExactInputSingleParams calldata params)
    external payable returns (uint256 amountOut)
{
    // ... existing logic ...
    _clearExpectedCallbackPool();

    // Auto-refund any unused ETH to the caller
    uint256 remaining = address(this).balance;
    if (remaining > 0) _transferETH(msg.sender, remaining);
}
```

Apply the same pattern to `exactInput`, `exactOutputSingle`, and `exactOutput`. Alternatively, restrict `refundETH` to only refund a tracked per-sender balance rather than the entire contract balance.

// Foundry test (pseudo-code, adapt to SimpleRouterTestBase)
function test_excessEthStranded_claimedByAttacker() public {
    uint128 amountIn = 100;
    uint256 msgValue = 200; // 2x the needed amount

    uint256 attackerEthBefore = attacker.balance;

    // User swaps WETH->token1 but sends 2x ETH, no refundETH call
    vm.prank(swapper);
    router.exactInputSingle{value: msgValue}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: swapper,
            deadline: block.timestamp + 1000,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // 100 ETH is now stranded in the router
    assertEq(address(router).balance, 100, "excess ETH stranded");

    // Attacker claims it
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance - attackerEthBefore, 100, "attacker stole excess ETH");
    assertEq(address(router).balance, 0, "router drained");
}
