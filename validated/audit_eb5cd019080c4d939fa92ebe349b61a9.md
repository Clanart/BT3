### Title
Missing Chain ID in `deployToken` Signed Payload Enables Cross-Chain Signature Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` and the `deploy_token` function in `omni_bridge.cairo` verify a NEAR MPC signature over a Borsh-encoded metadata payload that **does not include the destination chain ID**. In contrast, `finTransfer` / `fin_transfer` on both chains correctly binds the signature to the destination chain by embedding `omniBridgeChainId` twice in the signed payload. This asymmetry means a valid `deployToken` signature obtained for one chain (e.g., Ethereum) can be submitted verbatim to any other chain sharing the same `nearBridgeDerivedAddress` (e.g., Arbitrum, Base, BNB, Starknet), causing an unauthorized token deployment on that chain.

---

### Finding Description

**EVM — `OmniBridge.sol::deployToken`**

The signed payload is constructed as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

No `omniBridgeChainId` appears anywhere in this payload. The function is `external` with no caller restriction beyond the pause flag.

**Compare with `finTransfer`**, which correctly embeds `omniBridgeChainId` twice:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
    Borsh.encodeUint64(payload.destinationNonce),
    bytes1(payload.originChain),
    Borsh.encodeUint64(payload.originNonce),
    bytes1(omniBridgeChainId),          // ← chain-bound
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),          // ← chain-bound again
    Borsh.encodeAddress(payload.recipient),
    ...
``` [2](#0-1) 

**Starknet — `bridge_types.cairo::MetadataPayloadTrait::to_borsh`**

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    let mut borsh_bytes: ByteArray = Default::default();
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [3](#0-2) 

No chain ID. The `deploy_token` entry point calls `_verify_borsh_signature(ref self, @payload.to_borsh(), signature)` — the hash is computed solely from this chain-ID-free payload. [4](#0-3) 

**Compare with `TransferMessagePayloadTrait::to_borsh`**, which takes `chain_id: u8` and embeds it twice: [5](#0-4) 

The byte-level layout of the `deployToken` signed payload is identical across EVM and Starknet:

```
0x01 | borsh(token) | borsh(name) | borsh(symbol) | decimals
```

Both chains verify against an Ethereum-style ECDSA address (`nearBridgeDerivedAddress` / `omni_bridge_derived_address`). A signature produced for Ethereum is therefore cryptographically valid on Arbitrum, Base, BNB, Starknet, and any other chain sharing the same derived address.

**Solana** has the same missing chain ID in `DeployTokenPayload::serialize_for_near`, but uses raw 64-byte secp256k1 public key comparison rather than an Ethereum address, so cross-replay between EVM/Starknet and Solana is not directly possible with the same signature bytes. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` signature on chain A can submit it to chain B's `OmniBridge` contract. Because the signature covers only `(prefix, token, name, symbol, decimals)` — identical on every chain — `ECDSA.recover` / `verify_eth_signature` succeeds. The effects are:

1. **Unauthorized token deployment**: A bridged token is deployed on chain B without the NEAR bridge ever authorizing it for that chain.
2. **Permanent blocking of legitimate deployment**: The `ERR_TOKEN_EXIST` guard (`require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")`) prevents the NEAR bridge from ever deploying the same token on chain B through the normal flow. The token registry on chain B is permanently poisoned.
3. **NEAR-side registry corruption**: The NEAR bridge's EVM prover processes `DeployToken` events from the authorized bridge contract. It would register the attacker-triggered deployment as the canonical token address for chain B, causing all future `finTransfer` operations on chain B to reference this address.

This matches the allowed impact: **High — Proof/signature verification bypass enabling unauthorized token deployment**. [7](#0-6) 

---

### Likelihood Explanation

- `deployToken` is `external` with no caller restriction beyond the pause flag — any address can call it.
- Signatures are submitted in plaintext as calldata on the source chain and are trivially observable from any block explorer.
- The Omni Bridge is deployed on multiple EVM chains (Ethereum, Arbitrum, Base, BNB, etc.) sharing the same `nearBridgeDerivedAddress`. Every legitimate `deployToken` call on any one of these chains immediately produces a replayable signature for all others.
- No special knowledge, capital, or privileged access is required.

---

### Recommendation

Embed `omniBridgeChainId` in the `deployToken` signed payload, mirroring the pattern already used in `finTransfer`:

**EVM (`OmniBridge.sol`):**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet (`bridge_types.cairo`):**
Change `MetadataPayloadTrait::to_borsh` to accept `chain_id: u8` and append it after the prefix, matching `TransferMessagePayloadTrait::to_borsh`.

The NEAR bridge's signing logic must be updated in lockstep to include the destination chain ID when producing `deployToken` signatures.

---

### Proof of Concept

1. The NEAR bridge signs a `deployToken` payload for token `"usdc.near"` on Ethereum (`omniBridgeChainId = 0`). The relayer submits it; the `DeployToken` event is emitted on Ethereum.

2. An attacker copies the `signatureData` bytes from the Ethereum transaction calldata.

3. The attacker calls `OmniBridge.deployToken(signatureData, metadata)` on Arbitrum (`omniBridgeChainId = 3`) with the identical `metadata` struct.

4. `ECDSA.recover(keccak256(borshEncoded), signatureData)` returns `nearBridgeDerivedAddress` — the payload is byte-for-byte identical to what was signed for Ethereum, so the signature is valid.

5. A `BridgeToken` proxy for `"usdc.near"` is deployed on Arbitrum. `nearToEthToken["usdc.near"]` is set to this address. `DeployToken` is emitted.

6. When the NEAR bridge later attempts to legitimately deploy `"usdc.near"` on Arbitrum, the call reverts with `ERR_TOKEN_EXIST`. The Arbitrum token registry is permanently locked to the attacker-triggered address.
