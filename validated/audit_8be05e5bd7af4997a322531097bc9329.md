### Title
No User-Initiated Cancellation for Pending Transfers Causes Permanent Fund Lock - (`near/omni-bridge/src/lib.rs`)

### Summary

Once a user initiates a NEAR-side bridge transfer via `ft_on_transfer` → `init_transfer_internal`, their tokens are immediately locked or burned and recorded in `pending_transfers`. The only functions that can remove a transfer from `pending_transfers` — `sign_transfer` and `claim_fee` — are both gated behind the `#[trusted_relayer]` macro. No user-callable cancellation or self-rescue path exists. If the relayer set becomes unavailable (e.g., relayer infrastructure goes offline, the relayer set is rotated and old transfers are abandoned, or the destination chain becomes permanently unreachable), the user's funds are irrecoverably locked with no on-chain recovery mechanism.

### Finding Description

`init_transfer_internal` is the terminal step of every NEAR-originated transfer:

```rust
// near/omni-bridge/src/lib.rs:1829-1865
fn init_transfer_internal(
    &mut self,
    transfer_message: TransferMessage,
    storage_owner: AccountId,
) -> U128 {
    let required_storage_balance = self
        .add_transfer_message(transfer_message.clone(), storage_owner.clone())  // inserted into pending_transfers
        ...
    if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
        self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);  // burned (bridged tokens)
        self.lock_tokens_if_needed(                                              // locked (native tokens)
            transfer_message.get_destination_chain(),
            &token_id,
            transfer_message.amount.0,
        );
    }
    ...
    U128(0)  // 0 returned to ft_transfer_call → tokens are NOT refunded
}
```

After this point, the only two code paths that remove the entry from `pending_transfers` are:

1. `sign_transfer` → `sign_transfer_callback` (removes only when `fee.is_zero()`): [1](#0-0) [2](#0-1) 

2. `claim_fee` → `claim_fee_callback` → `remove_transfer_message`: [3](#0-2) 

Both are decorated with `#[trusted_relayer]`, meaning only accounts in the protocol's trusted relayer set can call them: [1](#0-0) [4](#0-3) 

There is no public `cancel_transfer`, `refund_transfer`, or equivalent function anywhere in the contract. The `storage_unregister` function with `force=true` can remove the storage balance entry but does not recover locked tokens or remove the `pending_transfers` entry: [5](#0-4) 

The `pending_transfers` map and `locked_tokens` map are entirely separate; removing a storage balance entry does not unlock or return the bridged assets: [6](#0-5) 

### Impact Explanation

A user who calls `ft_transfer_call` to initiate a bridge transfer has their tokens immediately consumed (burned for bridged tokens, locked for native tokens) and a `pending_transfers` entry created. If no trusted relayer ever calls `sign_transfer` or `claim_fee` for that transfer ID, the tokens remain locked in the contract indefinitely with no on-chain path for the user to recover them. This constitutes a permanent, irrecoverable lock of user funds in the bridge vault flow, matching the Critical impact class: "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."

### Likelihood Explanation

Realistic triggering scenarios include:
- The relayer set is rotated and old in-flight transfers are not processed before the old relayer keys are decommissioned.
- The destination chain (e.g., a new EVM chain) becomes permanently unreachable or its factory address is removed, making `sign_transfer` revert on the MPC call or `claim_fee` revert on proof verification — yet the NEAR-side tokens remain locked.
- A relayer infrastructure outage during which the user has no self-help path.

None of these require malicious behavior; they are operational failure modes. The user has no recourse other than waiting for DAO intervention via a contract upgrade.

### Recommendation

Add a user-callable `cancel_transfer` function that:
1. Verifies the caller is the original `sender` recorded in the `TransferMessageStorage`.
2. Optionally enforces a minimum waiting period (e.g., 7 days) to prevent griefing of relayers who are actively processing the transfer.
3. Removes the entry from `pending_transfers` (with storage refund).
4. Re-mints burned tokens or unlocks locked tokens back to the sender.
5. Emits a `CancelTransferEvent` for off-chain indexers.

Additionally, expose a view function `is_transfer_pending(transfer_id)` so users and tooling can check whether a transfer is still awaiting relayer action.

### Proof of Concept

1. User calls `ft_transfer_call` on a NEP-141 token with `msg` encoding an `InitTransferMsg` targeting an EVM recipient.
2. `ft_on_transfer` → `init_transfer` → `init_transfer_internal` executes: tokens are burned/locked, `pending_transfers[TransferId { origin_chain: Near, origin_nonce: N }]` is inserted, `ft_transfer_call` returns `U128(0)` (tokens consumed).
3. The relayer set goes offline or is rotated without processing nonce `N`.
4. User attempts to recover funds: no public function accepts their call. `sign_transfer` and `claim_fee` both revert with "Relayer is not active" or equivalent for non-relayer callers.
5. `get_transfer_message(TransferId { origin_chain: Near, origin_nonce: N })` still returns the transfer indefinitely — confirming the entry and the locked tokens are permanently stranded. [7](#0-6) [8](#0-7) [9](#0-8)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }

    #[payable]
    pub fn storage_unregister(&mut self, force: Option<bool>) -> bool {
        assert_one_yocto();
        let account_id = env::predecessor_account_id();
        let Some(storage) = self.storage_balance_of(&account_id) else {
            return false;
        };

        if !force.unwrap_or_default() {
            require!(
                storage.total.saturating_sub(storage.available)
                    == self.required_balance_for_account(),
                BridgeError::StoragePendingTransfers.as_ref()
            );
        }

        self.accounts_balances.remove(&account_id);

        let refund = self
            .required_balance_for_account()
            .saturating_add(storage.available);
        Promise::new(account_id).transfer(refund).detach();
        true
    }
