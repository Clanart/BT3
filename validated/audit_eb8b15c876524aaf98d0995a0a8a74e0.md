### Title
Cross-Chain Replay of MPC-Signed MetadataPayload Enables Unauthorized Token Deployment on Any EVM Chain — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-types/src/lib.rs`)

---

### Summary

The `MetadataPayload` borsh encoding signed by the NEAR MPC signer contains no chain-id or contract-address discriminator. Because the same MPC key (and thus the same `nearBridgeDerivedAddress`) is used across all EVM deployments, a valid signature observed from a `LogMetadataEvent` on NEAR can be replayed verbatim against `deployToken` on any other EVM chain, bypassing the intended chain-scoped authorization.

---

### Finding Description

**NEAR signing side** — `log_metadata_callback` in `near/omni-bridge/src/lib.rs` constructs a `MetadataPayload` and signs its borsh encoding:

```
MetadataPayload { prefix, token, name, symbol, decimals }
```

No chain-id, no destination contract address, no nonce is included. [1](#0-0) [2](#0-1) 

The resulting `SignatureResponse` is emitted publicly in a `LogMetadataEvent` NEAR log. [3](#0-2) 

**EVM verification side** — `deployToken` in `OmniBridge.sol` reconstructs the same borsh blob and checks the ECDSA signer:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [4](#0-3) 

No `omniBridgeChainId` is included in the hash. The only other guard is `!isBridgeToken[nearToEthToken[metadata.token]]` (token must not already exist on that chain). [5](#0-4) 

**Contrast with `finTransfer`** — the transfer payload encoding correctly binds to the chain by embedding `omniBridgeChainId` twice (for token address and recipient address):

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
...
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [6](#0-5) 

This chain-id binding is entirely absent from the metadata path.

**Starknet is equally affected** — `MetadataPayload.to_borsh()` on Starknet encodes the same chain-agnostic fields: [7](#0-6) 

**Solana is equally affected** — `DeployTokenPayload.serialize_for_near()` encodes `IncomingMessageType::Metadata` followed by token/name/symbol/decimals, no chain discriminator: [8](#0-7) 

---

### Impact Explanation

An attacker who observes a `LogMetadataEvent` on NEAR (containing a valid `SignatureResponse` for a token on chain A) can immediately call `deployToken` on chain B's `OmniBridge` with the identical metadata and signature. Because:

1. The borsh-encoded hash is identical on every chain (no chain-id in the payload).
2. `nearBridgeDerivedAddress` is the same on every EVM chain (it is derived from the same MPC key path with no per-chain derivation).
3. The only guard (`ERR_TOKEN_EXIST`) passes if the token has not yet been deployed on chain B.

The attacker successfully deploys a canonical bridge token on chain B before the protocol intends to. Once registered in `nearToEthToken`, the legitimate deployment is permanently blocked (`ERR_TOKEN_EXIST`). All future `finTransfer` calls on chain B for that NEAR token will mint into this attacker-triggered deployment. While the attacker does not control the deployed `BridgeToken` contract itself, they control the timing and chain-selection of canonical token deployment, hijacking the asset identity across chains.

This matches the allowed impact: **High — signature verification bypass enabling unauthorized token deployment**.

---

### Likelihood Explanation

The attack requires no privilege. The `LogMetadataEvent` is a public NEAR log. Any observer can extract the `SignatureResponse`, construct the call, and submit it to any EVM chain's `deployToken`. The attack is a single transaction with no preconditions beyond observing the NEAR log. It is front-runnable and repeatable across every EVM chain the bridge is deployed on.

---

### Recommendation

Include `omniBridgeChainId` (and optionally the bridge contract address) in the `MetadataPayload` borsh encoding on the NEAR signing side, mirroring the pattern already used in `TransferMessagePayload`. On the EVM side, include `omniBridgeChainId` in the reconstructed borsh blob before hashing, exactly as `finTransfer` does. Apply the same fix to the Starknet and Solana `deploy_token` verification paths.

---

### Proof of Concept

1. Token `wrap.near` has its metadata signed on NEAR for deployment on Ethereum (chain A). The `LogMetadataEvent` is emitted with a valid `SignatureResponse`.

2. Attacker observes the event and extracts `(signature, metadata)`.

3. Attacker calls `deployToken(signatureData, metadata)` on Arbitrum's (chain B) `OmniBridge`.

4. Chain B reconstructs:
   ```
   keccak256(borsh([Metadata, "wrap.near", "Wrapped NEAR", "wNEAR", 24]))
   ```
   This is byte-for-byte identical to what NEAR signed.

5. `ECDSA.recover(hash, sig) == nearBridgeDerivedAddress` — passes.

6. `nearToEthToken["wrap.near"] == address(0)` on chain B — passes.

7. A new `BridgeToken` proxy is deployed and registered as the canonical `wrap.near` token on Arbitrum, before the protocol intended it, permanently blocking legitimate deployment.
