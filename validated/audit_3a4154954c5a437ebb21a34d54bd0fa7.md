Audit Report

## Title
Unprivileged caller can drain all WETH held by the router via `unwrapWETH9(0, attacker)` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` is `public payable` with no access control and accepts a caller-supplied `recipient`. Passing `amountMinimum = 0` trivially bypasses the only guard (`balanceWETH < amountMinimum` is always `false` for a `uint256`), allowing any caller to withdraw the router's entire WETH balance as ETH to an arbitrary address. `sweepToken` is identically vulnerable for any ERC-20 residue.

## Finding Description [1](#0-0) 

`unwrapWETH9` reads the router's full WETH balance at line 38, checks `balanceWETH < amountMinimum` at line 39, then unconditionally withdraws and forwards the full amount to `recipient` at lines 42–43. There is no `msg.sender` check, no ownership mapping, and no per-user accounting anywhere in the function or its parent contract.

With `amountMinimum = 0`, the guard evaluates to `uint256 < 0`, which is permanently `false` in Solidity. The attacker disables the only protection with their own input. `sweepToken` at lines 48–55 has the identical structure and is equally exploitable for any ERC-20 token stranded on the router. [2](#0-1) 

## Impact Explanation
Any WETH that lands on the router — regardless of which user deposited it — can be immediately claimed by an unprivileged attacker. The attacker receives the full ETH value; the legitimate user receives nothing. This is a direct, unconditional loss of user principal with no protocol-side mitigation, meeting the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
WETH reaches the router in the normal intended flow: a user routes a token→WETH swap with `recipient = address(router)` as step 1 of a multicall, then calls `unwrapWETH9` as step 2. If the user submits step 1 as a standalone transaction and step 2 separately (a pattern common in manual wallet batching), or directly transfers WETH to the router before calling `unwrapWETH9`, an attacker monitoring the mempool can front-run step 2 with `unwrapWETH9(0, attacker)` and drain the full balance. The attack requires zero privileges, zero capital, and a single call.

## Recommendation
Restrict `unwrapWETH9` and `sweepToken` to `msg.sender` as the only permitted recipient, or enforce `require(recipient == msg.sender, "recipient must be caller")`. Alternatively, track per-user WETH credits inside the router and only allow each user to unwrap their own credited amount.

## Proof of Concept
1. Alice calls `exactInputSingle(tokenIn=token1, tokenOut=WETH, recipient=address(router))` in a standalone transaction. The router now holds `N` WETH.
2. Before Alice calls `unwrapWETH9(N, alice)`, Bob calls `unwrapWETH9(0, bob)`.
3. Guard at line 39: `N < 0` → `false` → no revert.
4. Line 42: `IWETH9(WETH).withdraw(N)` executes; router receives `N` ETH.
5. Line 43: `_transferETH(bob, N)` executes; Bob receives `N` ETH.
6. Alice's WETH is gone; router WETH balance is zero. [1](#0-0)

  function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);

    if (balanceToken > 0) {
      IERC20(token).safeTransfer(recipient, balanceToken);
    }
