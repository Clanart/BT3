### Title
`execute_deploy` Syscall Unconditionally Reports Success Regardless of Constructor Revert — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in `syscall_impls.cairo` always writes `failure_flag=0` in the response header, regardless of whether the deployed contract's constructor actually reverted. This is a direct analog to the reported Sense Finance bug: just as `_swapPTsForTarget` unconditionally attempted to redeem without checking the restricted flag, `execute_deploy` unconditionally reports success without checking the constructor's revert status.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–539), after calling `deploy_contract`, the OS writes the syscall response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): consider sharing code with call_contract_helper.
let response_header = cast(syscall_ptr, ResponseHeader*);
let syscall_ptr = syscall_ptr + ResponseHeader.SIZE;

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The developer TODO comment explicitly acknowledges that failure cases are not yet supported. Compare this to `contract_call_helper` in the same file, which correctly propagates the `is_reverted` flag into the response:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_entry_point` function in `execute_entry_point.cairo` correctly handles the reverted case by calling `handle_revert` to undo state changes via the revert log — meaning the constructor's storage writes and class hash assignment **are** rolled back on revert. However, the response written back to the calling contract still says `failure_flag=0`. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a Cairo 1 contract calls the `deploy` syscall and the constructor reverts:

1. The OS correctly processes the revert log, undoing the constructor's state changes (storage writes, class hash assignment via `CHANGE_CLASS_ENTRY`).
2. The deployed contract address ends up with no class hash (i.e., no contract code exists at that address).
3. The OS writes `failure_flag=0` to the syscall response — the calling contract's Cairo code sees a successful deploy.
4. The calling contract proceeds to send funds (e.g., via `transfer` or `call_contract`) to the "deployed" address, believing the contract is live and initialized.
5. Since no contract exists at that address (class hash was reverted), the funds are permanently frozen with no mechanism for recovery.

This matches the **Permanent freezing of funds** impact category.

---

### Likelihood Explanation

**Medium.** The attacker-controlled entry path is:

- An unprivileged user deploys a contract whose constructor can revert (e.g., due to an out-of-gas condition, a failed assertion, or a deliberate revert triggered by attacker-controlled calldata).
- A victim contract (e.g., a factory or vault) calls `deploy` and then immediately sends funds to the returned `contract_address` based on the `failure_flag=0` response.
- This pattern — deploy-then-fund — is standard in DeFi factory contracts.

No privileged access is required. The attacker only needs to craft a class whose constructor reverts under predictable conditions and induce a victim factory contract to deploy it.

---

### Recommendation

Mirror the pattern used in `contract_call_helper`. After calling `deploy_contract`, capture the `is_reverted` return value and propagate it into the response header:

```cairo
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

If the constructor reverted, the response should carry `failure_flag=1` and include the revert reason in `constructor_retdata`, so the calling contract can branch correctly.

---

### Proof of Concept

1. Declare a Cairo 1 class whose constructor unconditionally reverts: `fn constructor(...) { panic!(...); }`.
2. Deploy a victim factory contract that calls `deploy(class_hash, ...)` and then immediately calls `transfer(deployed_address, amount)` on an ERC-20, trusting the `failure_flag=0` response.
3. Submit an invoke transaction calling the factory's deploy-and-fund function with the reverting class hash.
4. The OS processes the block: the constructor reverts → state is rolled back → `failure_flag=0` is written → the factory transfers `amount` tokens to the dead address.
5. Observe that `amount` tokens are permanently frozen at the address with no contract code. [4](#0-3)
