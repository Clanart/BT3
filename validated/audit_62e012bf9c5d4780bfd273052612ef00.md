Let me examine the exact execution order in `finTransfer` and the nonce-consumption logic carefully.

### Title
ERC1155 `finTransfer` Permanently Locks Funds When Recipient Contract Reverts `onERC1155Received` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`finTransfer` uses `IERC1155.safeTransferFrom` to deliver ERC1155 tokens to `payload.recipient`. If the recipient is a contract whose `onERC1155Received` always reverts, the entire transaction reverts. Because the `destinationNonce` is set to `true` before the transfer but is rolled back on revert, the nonce is never consumed. The recipient address is cryptographically bound in the MPC signature and cannot be changed. There is no alternative delivery path and no admin rescue function. The result is that the ERC1155 tokens held by the bridge are permanently undeliverable, and the corresponding NEAR-side locked tokens are irrecoverable.

---

### Finding Description

In `finTransfer`, the nonce is marked used at line 287 before the actual token delivery:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC1155(multiToken.tokenAddress).safeTransferFrom(    // line 324
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
``` [1](#0-0) 

`IERC1155.safeTransferFrom` mandatorily calls `onERC1155Received` on the recipient if it is a contract (ERC-1155 standard requirement). If that callback reverts, the entire EVM transaction reverts atomically, rolling back the `completedTransfers` write. The nonce is therefore never consumed.

The `payload.recipient` is Borsh-encoded into the signed hash at line 298:

```solidity
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

This means the recipient is cryptographically fixed by the MPC signature. No caller can substitute a different recipient without producing an `InvalidSignature` revert at line 311. [3](#0-2) 

There is no admin rescue function, no `rescueERC1155`, no redirect-recipient mechanism, and no way to re-issue a signature for a different recipient once the NEAR-side transfer has been committed. The contract has only `pause`, `pauseAll`, and `upgradeToken`, none of which can recover stuck ERC1155 transfers. [4](#0-3) 

On the NEAR side, `process_fin_transfer_to_other_chain` stores the transfer message and locks/burns the tokens. There is no cancel or refund path for a transfer whose EVM-side finalization permanently fails. [5](#0-4) 

---

### Impact Explanation

Any user who initiates an ERC1155 bridge transfer from NEAR to EVM with a recipient address that is a contract not implementing `IERC1155Receiver` (or one that deliberately reverts) will have their tokens permanently frozen:

- The ERC1155 tokens remain in the bridge contract on EVM, undeliverable.
- The NEAR-side tokens are burned/locked with no refund path.
- Every retry of `finTransfer` with the same signed payload reverts identically.
- No privileged actor can redirect or recover the transfer without a contract upgrade.

This matches **Critical: Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows**.

---

### Likelihood Explanation

The scenario is reachable by any unprivileged user:

1. The user controls the recipient field when initiating the transfer on NEAR.
2. Specifying a contract address that does not implement `IERC1155Receiver` (e.g., a plain multisig, a DAO contract, or a deliberately crafted reverting receiver) is a realistic mistake or intentional act.
3. The MPC will sign any valid transfer payload; it has no knowledge of whether the EVM recipient can accept ERC1155 tokens.
4. `logMetadata1155` is permissionless, so any ERC1155 token can be registered. [6](#0-5) 

---

### Recommendation

Replace `safeTransferFrom` with a two-step pull pattern for ERC1155 deliveries:

1. Use `IERC1155.safeTransferFrom` but wrap it in a try/catch (Solidity `try`/`catch` on external calls). On failure, store the pending amount in a claimable mapping keyed by `(destinationNonce, recipient)` and mark the nonce consumed.
2. Add a `claimERC1155(uint64 destinationNonce)` function that lets the recipient (or an admin-designated address) pull the tokens later.

Alternatively, use `transferFrom` (non-safe) if the ERC1155 token supports it, bypassing the mandatory callback. However, this is non-standard and token-dependent.

At minimum, add an admin `rescueERC1155(uint64 destinationNonce, address newRecipient)` function gated by `DEFAULT_ADMIN_ROLE` to redirect permanently stuck transfers.

---

    fn process_fin_transfer_to_other_chain(
        &mut self,
        predecessor_account_id: AccountId,
        transfer_message: TransferMessage,
    ) {
        let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
        let token = self.get_token_id(&transfer_message.token);

        if transfer_message.recipient.is_utxo_chain() {
            let btc_account_id =
                self.get_utxo_chain_token(transfer_message.get_destination_chain());
            require!(
                token == btc_account_id,
                BridgeError::NativeTokenRequiredForChain.as_ref()
            );
        }

        self.unlock_tokens_if_needed(
            transfer_message.get_origin_chain(),
            &token,
            transfer_message.amount.0,
        );
        self.lock_tokens_if_needed(
            transfer_message.get_destination_chain(),
            &token,
            transfer_message.fee.fee.into(),
        );

        let fast_transfer = FastTransfer::from_transfer(transfer_message.clone(), token.clone());
        let recipient = if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
            require!(
                !status.finalised,
                BridgeError::FastTransferAlreadyFinalised.as_ref()
            );
            Some(status.relayer)
        } else {
            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token,
                transfer_message
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            );

            None
        };

        // If fast transfer happened, send tokens to the relayer that executed fast transfer
        if let Some(relayer) = recipient {
            self.send_tokens(
                token,
                relayer,
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
                "",
            )
            .detach();
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
        } else {
            required_balance = self
                .add_transfer_message(transfer_message.clone(), predecessor_account_id.clone())
                .saturating_add(required_balance);
        }

        self.update_storage_balance(
            predecessor_account_id,
            required_balance,
            env::attached_deposit(),
        );

        env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
    }
