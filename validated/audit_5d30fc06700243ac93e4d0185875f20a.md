### Title
`get_class(Pending, H)` bypasses parent-hash staleness guard, leaking uncommitted class body from a prior pending block — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`starknet_getClass` with `block_id=Pending` reads directly from `self.pending_classes` without the parent-hash staleness check that every other pending-data path applies via `read_pending_data`. After the node advances from block P1 to P2, `pending_classes` is not cleared atomically; it is cleared only when the feeder gateway later returns new pending data with `parent_hash=P2`. During that window — which can span multiple polling cycles — any unprivileged RPC client can retrieve the full class body of a class that was only ever in the pending state for P1 and was never committed to any block.

---

### Finding Description

**`get_class` handler — no staleness check on `pending_classes`** [1](#0-0) 

When `block_id=Pending`, the handler acquires a read lock on `self.pending_classes` and returns the class body immediately if found. There is no check that `pending_classes` is anchored to the current latest block.

**`read_pending_data` — the staleness guard that `get_class` skips** [2](#0-1) 

Every other pending-data consumer (`get_block`, `get_nonce`, `get_class_hash_at`, `estimate_fee`, etc.) calls `read_pending_data`, which compares `pending_data.block.parent_block_hash()` against the latest committed block hash and returns an empty pending state when they differ. `get_class` never calls this function.

**`pending_classes` is not cleared when a new block is found — only when the feeder gateway confirms the new parent** [3](#0-2) 

When `get_pending_data` detects that the feeder gateway's pending data no longer matches `latest_block_hash`, it returns `PendingSyncFinished` **without touching `pending_classes`**. [4](#0-3) 

`pending_classes.clear()` is only called when the feeder gateway subsequently returns pending data whose `parent_hash` differs from the currently stored pending data's `parent_hash`. Until that happens — potentially across many polling cycles — `pending_classes` retains every class that was fetched for the P1 pending block.

**Race window**

```
Storage advances: latest = P2 (parent = P1)
pending_data.parent_block_hash = P1  ← stale, read_pending_data returns empty
pending_classes = { H: <class body> } ← stale, NOT cleared yet

RPC call: starknet_getClass(Pending, H)
  → reads pending_classes directly
  → finds H
  → returns class body  ← WRONG: H was never committed
```

The window lasts until the feeder gateway returns a pending block with `parent_hash=P2` and `get_pending_data` executes the `clear()` branch.

---

### Impact Explanation

An unprivileged RPC client receives the full Sierra or Cairo 0 class body for a class hash that:
- was only ever declared in a pending block anchored to the now-superseded block P1, and
- was never included in any committed block.

The response is indistinguishable from a legitimate `starknet_getClass` result. Downstream consumers (wallets, indexers, SDKs, fee estimators) that cache or act on this response will operate on uncommitted, potentially discarded class definitions. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The race window is bounded by the feeder gateway polling interval and block time. On mainnet, block times are on the order of seconds to tens of seconds, and the polling interval is configurable. Any client that queries `starknet_getClass(Pending, H)` immediately after a new block is committed — a common pattern for wallets and explorers watching for class declarations — will reliably hit this window.

---

### Recommendation

Apply the same staleness guard to `pending_classes` that `read_pending_data` applies to `pending_data`. The simplest fix is to call `read_pending_data` first inside `get_class` when `block_id=Pending`, and only consult `pending_classes` if the returned pending data is fresh (i.e., `read_pending_data` did not return an empty fallback). Alternatively, store the anchor block hash inside `PendingClasses` and check it atomically in `get_class` before returning any class.

---

### Proof of Concept

```rust
// 1. Write block P1 to storage (block_hash = H1).
// 2. Populate pending_classes with class C under parent H1.
// 3. Write block P2 to storage (parent = H1, block_hash = H2).
//    Storage now reports latest = P2; pending_data.parent still = H1.
// 4. Call get_class(Pending, C).
//    Expected: CLASS_HASH_NOT_FOUND (C was never committed).
//    Actual:   full class body of C is returned.
//
// The test below demonstrates the gap:

#[tokio::test]
async fn get_class_pending_stale_pending_classes() {
    let pending_classes = get_test_pending_classes();
    let (module, mut storage_writer) =
        get_test_rpc_server_and_storage_writer_from_params::<JsonRpcServerImpl>(
            None, None, None, Some(pending_classes.clone()), None,
        );

    // Commit block 0 (P1, hash = 0x1).
    let p1_hash = BlockHash(felt!("0x1"));
    let header0 = BlockHeader {
        block_hash: p1_hash,
        block_header_without_hash: BlockHeaderWithoutHash {
            block_number: BlockNumber(0),
            ..Default::default()
        },
        ..Default::default()
    };
    storage_writer.begin_rw_txn().unwrap()
        .append_header(BlockNumber(0), &header0).unwrap()
        .append_state_diff(BlockNumber(0), ThinStateDiff::default()).unwrap()
        .commit().unwrap();

    // Add class C to pending_classes (anchored to P1).
    let class_hash = ClassHash(felt!("0xdeadbeef"));
    let class = ApiContractClass::ContractClass(
        SierraContractClass::get_test_instance(&mut get_rng()),
    );
    pending_classes.write().await.add_class(class_hash, class);

    // Commit block 1 (P2, parent = P1). Storage now reports latest = P2.
    let p2_hash = BlockHash(felt!("0x2"));
    let header1 = BlockHeader {
        block_hash: p2_hash,
        block_header_without_hash: BlockHeaderWithoutHash {
            block_number: BlockNumber(1),
            parent_hash: p1_hash,
            ..Default::default()
        },
        ..Default::default()
    };
    storage_writer.begin_rw_txn().unwrap()
        .append_header(BlockNumber(1), &header1).unwrap()
        .append_state_diff(BlockNumber(1), ThinStateDiff::default()).unwrap()
        .commit().unwrap();

    // pending_data is NOT updated (still has parent = P1, which read_pending_data
    // would detect as stale). pending_classes still has class C.

    // This call should return CLASS_HASH_NOT_FOUND because C was never committed
    // and the pending state is stale. Instead it returns the class body.
    let result = module
        .call::<_, GatewayContractClass>(
            "starknet_V0_8_getClass",
            (BlockId::Tag(Tag::Pending), class_hash),
        )
        .await;

    // BUG: result is Ok(class body) instead of Err(CLASS_HASH_NOT_FOUND).
    assert!(result.is_err(), "expected CLASS_HASH_NOT_FOUND for stale pending class");
}
```
