### Title
Missing `multiTokens` Mapping Write in `initTransfer1155` Causes ERC1155 Fund Lock When `logMetadata1155` Is Not Called First — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` accepts and locks ERC1155 tokens into the bridge without writing to the `multiTokens` mapping. `finTransfer` relies exclusively on that mapping to identify and release ERC1155 tokens. If a user calls `initTransfer1155` without a prior `logMetadata1155` call, the ERC1155 tokens are locked in the bridge and every subsequent `finTransfer` attempt for that transfer will revert, freezing user funds until a separate permissionless `logMetadata1155` call is made.

---

### Finding Description

`deriveDeterministicAddress` computes a pseudo-address from `(tokenAddress, tokenId)` by taking the **first 20 bytes** of `keccak256(abi.encodePacked(tokenAddress, tokenId))`:

```solidity
function deriveDeterministicAddress(
    address tokenAddress,
    uint256 tokenId
) public pure returns (address) {
    return address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))));
}
``` [1](#0-0) 

`logMetadata1155` is the only function that writes to `multiTokens[deterministicToken]`:

```solidity
MultiTokenInfo storage multiToken = multiTokens[deterministicToken];
if (multiToken.tokenAddress == address(0)) {
    multiToken.tokenAddress = tokenAddress;
    multiToken.tokenId = tokenId;
}
``` [2](#0-1) 

`initTransfer1155` computes the same `deterministicToken` address and emits `InitTransfer` with it, but **never writes to `multiTokens`**:

```solidity
address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
// ... no multiTokens write ...
emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, ...);
``` [3](#0-2) 

`finTransfer` uses `multiTokens[payload.tokenAddress]` to decide how to release funds. If the mapping is empty, it falls through to the ERC20 branch and calls `IERC20(deterministicToken).safeTransfer(...)`, which reverts because `deterministicToken` is not a real contract:

```solidity
MultiTokenInfo memory multiToken = multiTokens[payload.tokenAddress];
// ...
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(...);
// ...
} else {
    IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
}
``` [4](#0-3) 

---

### Impact Explanation

Any user who calls `initTransfer1155` without a prior `logMetadata1155` for the same `(tokenAddress, tokenId)` pair will have their ERC1155 tokens locked in the bridge. Every `finTransfer` call for that transfer will revert (the nonce is not consumed on revert since the whole transaction rolls back), freezing the user's assets. The freeze persists until someone calls the permissionless `logMetadata1155` for that pair. If the user is unaware of this requirement, funds can remain locked indefinitely. This is a token-mapping desync that breaks bridge settlement for ERC1155 assets.

---

### Likelihood Explanation

`initTransfer1155` is a public, permissionless function with no guard requiring `logMetadata1155` to have been called first. There is no on-chain check, no revert, and no documentation in the function itself. Any user interacting with the ERC1155 bridge path without reading the full protocol flow will trigger this. The NEAR-side indexer will process the `InitTransfer` event normally, sign a `finTransfer` payload, and the relayer will submit it — only to have it silently fail at the EVM side.

---

### Recommendation

Add a check in `initTransfer1155` that requires `multiTokens[deterministicToken].tokenAddress != address(0)` before accepting the ERC1155 transfer, or have `initTransfer1155` itself write to `multiTokens` if the slot is empty (mirroring the logic in `logMetadata1155`).

---

### Proof of Concept

1. Deploy `OmniBridge` with a valid `nearBridgeDerivedAddress`.
2. Deploy any ERC1155 contract and mint token ID `7` to `user`.
3. `user` approves the bridge and calls `initTransfer1155(erc1155, 7, 1, 0, 0, "recipient.near", "")` — **without** calling `logMetadata1155` first.
4. ERC1155 token is transferred to the bridge. `InitTransfer` is emitted with `deterministicToken = deriveDeterministicAddress(erc1155, 7)`.
5. NEAR side processes the event and a relayer calls `finTransfer(sig, payload)` where `payload.tokenAddress = deterministicToken`.
6. Inside `finTransfer`: `multiTokens[deterministicToken].tokenAddress == address(0)` → falls to `IERC20(deterministicToken).safeTransfer(...)` → reverts (no code at `deterministicToken`).
7. The ERC1155 token remains locked in the bridge. The nonce is not consumed. The user's funds are frozen until `logMetadata1155(erc1155, 7)` is called by anyone. [5](#0-4) [4](#0-3)
