### Title
Decimal Mismatch Between StarkNet `_normalizeDecimals` Clamping and NEAR `bind_token` Storage Causes `denormalize_amount` u128 Overflow, Permanently Locking Bridged Funds — (`starknet/src/omni_bridge.cairo`, `near/omni-bridge/src/lib.rs`)

---

### Summary

StarkNet's `_normalizeDecimals` silently clamps any `decimals > 18` to 18 while still emitting the original value as `origin_decimals` in the `DeployToken` event. NEAR's `bind_token_callback` stores this inconsistent pair verbatim. When any user subsequently bridges tokens from StarkNet to NEAR, `fin_transfer_callback` calls `denormalize_amount` which computes `10_u128.pow(origin_decimals - decimals)` = `10^237`, overflowing u128. Because `overflow-checks = true` is set in `near/Cargo.toml`, this panics unconditionally, permanently preventing settlement of every transfer for that token.

---

### Finding Description

**Step 1 — StarkNet `deploy_token` clamps but leaks `origin_decimals`** [1](#0-0) 

`_normalizeDecimals(255)` returns 18. The token is deployed with 18 decimals, but the `DeployToken` event is emitted with `decimals=18` and `origin_decimals=255`: [2](#0-1) 

**Step 2 — NEAR `bind_token_callback` stores the inconsistent pair without validation**

The StarkNet prover parses both fields from the event: [3](#0-2) 

`bind_token_callback` passes them directly to `add_token`: [4](#0-3) 

`add_token` stores `Decimals { decimals: 18, origin_decimals: 255 }` keyed by the StarkNet token address — no bounds check: [5](#0-4) 

**Step 3 — `fin_transfer_callback` overflows on every StarkNet→NEAR transfer** [6](#0-5) 

`denormalize_amount` computes: [7](#0-6) 

With `origin_decimals=255`, `decimals=18`: `diff_decimals = 237`. `10_u128.pow(237)` vastly exceeds `u128::MAX` (~3.4 × 10^38). Because the NEAR workspace profile mandates: [8](#0-7) 

…the overflow panics unconditionally. The callback reverts, the transfer is never settled, and the tokens burned/locked on StarkNet are irrecoverable.

**Note on threshold:** The overflow triggers for any `origin_decimals - decimals >= 39` (since 10^39 > u128::MAX). After clamping to 18, any source token with `decimals >= 57` is sufficient. `decimals=255` is the worst case but not the only case.

---

### Impact Explanation

Every call to `fin_transfer_callback` for the affected token panics. Tokens burned on StarkNet during `init_transfer` are permanently unrecoverable. This satisfies the "irrecoverable lock of user funds" critical impact criterion.

---

### Likelihood Explanation

The precondition is a valid MPC signature for a `MetadataPayload` with `decimals >= 57`. This is reachable without privilege: an attacker deploys a NEAR-compatible token contract whose `decimals()` returns 255, calls `log_metadata` on StarkNet (permissionless), the NEAR bridge processes the `LogMetadata` event and requests an MPC signature for the payload as-is (no decimals validation), and the relayer uses that signature to call `deploy_token` on StarkNet. The MPC service has no guard against high decimal values. The entire path is unprivileged and publicly callable.

---

### Recommendation

1. **Reject high-decimal payloads at the MPC signing boundary on NEAR** — before requesting an MPC signature, validate that `metadata.decimals <= MAX_ALLOWED_DECIMALS` (e.g., 24 or 18).
2. **Validate in `bind_token_callback`** — assert `deploy_token.origin_decimals <= some_safe_max` and `deploy_token.origin_decimals >= deploy_token.decimals` before calling `add_token`.
3. **Use checked arithmetic in `denormalize_amount`** — replace `amount * (10_u128.pow(diff_decimals))` with `checked_pow` and `checked_mul`, panicking with a descriptive error rather than an opaque overflow.
4. **Align StarkNet's `_normalizeDecimals` with NEAR's storage** — either reject `decimals > 18` outright in `deploy_token`, or cap `origin_decimals` in the emitted event to the same clamped value so NEAR never stores a divergent pair.

Call sequence:
1. Attacker deploys token with `decimals()=255` on any chain; calls `log_metadata` (permissionless).
2. NEAR MPC signs `MetadataPayload { decimals: 255, … }`.
3. Relayer calls `deploy_token` on StarkNet → succeeds, emits `DeployToken { decimals: 18, origin_decimals: 255 }`.
4. Relayer calls `bind_token` on NEAR with StarkNet proof → `Decimals { decimals: 18, origin_decimals: 255 }` stored.
5. Any user calls `init_transfer` on StarkNet for this token.
6. Relayer calls `fin_transfer` on NEAR → `fin_transfer_callback` → `denormalize_amount(X, {18, 255})` → **panic** → transfer permanently fails, funds locked.
