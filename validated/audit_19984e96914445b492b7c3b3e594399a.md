### Title
Missing shard_id binding in `set_state_header` Merkle proof allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` verifies that the supplied chunk is *somewhere* in the `chunk_headers_root` Merkle tree but never checks that the chunk's position in that tree corresponds to the requested `shard_id`. An unprivileged peer acting as a state-sync provider can supply shard B's chunk (with shard B's `prev_state_root`) together with shard B's valid Merkle path while the node is syncing shard A. The function accepts the header and stores it under `StateHeaderKey(shard_id=A, sync_hash)`. Every subsequent `set_state_part` and `set_state_finalize` call for shard A then operates against shard B's committed state root, corrupting shard A's state.

---

### Finding Description

**Entrypoint**: `set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The Merkle membership check at lines 394–403 is:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
``` [1](#0-0) 

`verify_path` is defined as:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [2](#0-1) 

It only checks that the supplied `(chunk_hash, height_included)` pair, combined with the supplied path, reproduces the root. It does **not** check the leaf's position (index) in the tree. A position-aware variant, `verify_path_with_index`, exists in the same file but is not used here: [3](#0-2) 

The `chunk_headers_root` Merkle tree is built by indexing each shard's chunk at its shard index. Shard B's chunk hash sits at index B; its Merkle path is valid against the root. If an attacker supplies shard B's chunk together with shard B's path, `verify_path` returns `true` regardless of the requested `shard_id`. There is no subsequent guard of the form `chunk.shard_id() == shard_id` anywhere in `set_state_header`. [4](#0-3) 

The same positional blindness applies to the `prev_chunk_header` proof at lines 416–425. [5](#0-4) 

The receipt-proof loop (step 4) hashes receipts as `ReceiptList(shard_id, receipts)` using the *requested* shard A id, not the chunk's shard id. An attacker can satisfy this by supplying the correct public incoming-receipt data for shard A, which is observable on-chain. [6](#0-5) 

Step 5 validates the `state_root_node` against `chunk_inner.prev_state_root()`, which is shard B's state root. A state-sync provider has access to shard B's trie and can supply a valid node. [7](#0-6) 

After all checks pass, the header is stored under the *requested* shard A key:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [8](#0-7) 

---

### Impact Explanation

`set_state_part` reads the stored header and extracts `prev_state_root` from the chunk it contains:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
``` [9](#0-8) 

All parts are validated against shard B's `state_root`. If the attacker also supplies shard B's state parts (which they possess as a state-sync provider), every part passes validation and is stored for shard A.

`set_state_finalize` then applies the chunk using `chunk_header.prev_state_root()` (shard B's root) under shard A's `shard_uid`: [10](#0-9) 

The node installs shard B's entire state as shard A's state. All subsequent chunk execution for shard A operates on the wrong trie root, producing wrong outcomes, wrong receipts, and wrong validator proposals — a complete and silent corruption of shard A's committed state.

---

### Likelihood Explanation

Any peer that the syncing node contacts for state sync can exploit this. State-sync providers are not required to be validators; the protocol accepts responses from any reachable peer. The attacker needs:

1. Knowledge of shard B's chunk hash and Merkle path (public, derivable from block headers).
2. The correct incoming receipts for shard A (public, derivable from block headers).
3. A valid `state_root_node` for shard B's state root (available to any node tracking shard B).

No privileged key material is required.

---

### Recommendation

Replace the plain `verify_path` call with `verify_path_with_index`, binding the proof to the shard's position in the Merkle tree:

```rust
verify_path_with_index(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
    prev_shard_index as u64,
    sync_prev_block.chunks().len() as u64,
)
```

Additionally, add an explicit guard immediately after `validate_chunk_proofs`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other("set_shard_state failed: chunk shard_id mismatch".into()));
}
```

Apply the same fix to the `prev_chunk_header` proof.

### Proof of Concept

```rust
// Integration test sketch (production code path only):
// 1. Build a two-shard chain; obtain sync_hash.
// 2. Retrieve shard 1's ShardStateSyncResponseHeader (chunk, chunk_proof, prev_chunk, prev_chunk_proof).
// 3. Collect the correct incoming receipts for shard 0 and build valid root_proofs against
//    the block's prev_chunk_outgoing_receipts_root.
// 4. Construct a ShardStateSyncResponseHeader with:
//      chunk        = shard 1's chunk
//      chunk_proof  = shard 1's Merkle path (valid against chunk_headers_root)
//      incoming_receipts_proofs = correct receipts for shard 0
//      state_root_node = valid node for shard 1's prev_state_root
// 5. Call chain_adapter.set_state_header(shard_id=0, sync_hash, crafted_header).
// 6. Assert Ok(()) is returned.
// 7. Read back StateHeaderKey(0, sync_hash) and assert stored chunk.shard_id() == 1
//    and stored chunk.prev_state_root() == shard_1_state_root  ≠  shard_0_state_root.
```

The test confirms that `set_state_header` accepts the cross-shard header and that the stored `prev_state_root` is shard 1's root, not shard 0's, violating the invariant that `StateHeaderKey(A, h).chunk.prev_state_root()` equals shard A's committed state root.
