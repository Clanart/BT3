### Title
`FeeEstimation` RPC Response Omits `l2_gas_consumed` While `overall_fee` Silently Includes L2 Gas Cost — (File: `crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but deliberately omits `l2_gas_consumed`. The `overall_fee` field is populated from `receipt.fee`, which **does** include L2 gas costs. However, both the struct's own doc-comment and the OpenRPC specification describe `overall_fee` as equalling only `gas_consumed * gas_price + data_gas_consumed * data_gas_price` — an expression that excludes L2 gas entirely. Any caller who reconstructs the fee from the returned components will obtain a value that is systematically lower than `overall_fee` for every V3 transaction that consumes L2 gas, and the discrepancy is invisible because the missing quantity is never surfaced.

---

### Finding Description

**Root cause — struct definition:** [1](#0-0) 

```rust
pub struct FeeEstimation {
    pub gas_consumed: Felt,          // L1 gas
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,     // L1 data gas
    pub l1_data_gas_price: GasPrice,
    // TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of
    // l1_gas_price only is close enough (as there are roundings) to the fee
    // of both l1_gas_price and l2_gas_price.
    pub l2_gas_price: GasPrice,      // price present, amount absent
    /// The total amount of fee. This is equal to:
    /// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
    pub overall_fee: Fee,            // actually includes l2_gas cost
    pub unit: PriceUnit,
}
```

The inline TODO at line 104 is the team's own acknowledgement that `l2_gas_consumed` is missing and that the invariant stated in the doc-comment has not been verified.

**Root cause — construction:** [2](#0-1) 

`tx_execution_output_to_fee_estimation` reads `gas_vector.l1_gas` and `gas_vector.l1_data_gas` for the two "consumed" fields, but reads `receipt.fee` for `overall_fee`. `receipt.fee` is produced by `get_fee_by_gas_vector`, which sums all three resources: [3](#0-2) 

```
// l1_gas=10, l1_data_gas=20, l2_gas=30, tip=3
// expected_fee_eth = 10 + 3*20 + (5+3)*30   ← l2_gas term is non-zero
```

So `overall_fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * l2_gas_price + tip * l2_gas`, but the struct only returns the first two terms' quantities.

**OpenRPC spec repeats the wrong invariant:** [4](#0-3) 

```json
"overall_fee": {
  "description": "…equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price"
}
```

**Transaction prover fixture confirms the field is expected:** [5](#0-4) 

The prover's recorded RPC response includes `"l2_gas_consumed": "0xb56b6"` alongside `"l2_gas_price"` and `"overall_fee"`. The sequencer never emits that field, so any consumer that relies on it to reconstruct or verify the fee silently receives zero.

**Prover documentation points users at the missing field:** [6](#0-5) 

> `l2_gas.max_amount`: … Set this to the value returned by `starknet_estimateFee`

`starknet_estimateFee` does not return `l2_gas_consumed`, so this instruction is impossible to follow precisely.

---

### Impact Explanation

**Scope match:** "High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction receives:

| Field | Correct? | Verifiable from response? |
|---|---|---|
| `overall_fee` | Yes (includes L2 gas) | No — `l2_gas_consumed` absent |
| `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` | Partial | Yes, but ≠ `overall_fee` |

A caller who reconstructs the fee from the returned components obtains a value that is **lower** than `overall_fee` by exactly `l2_gas_consumed * l2_gas_price`. The spec and doc-comment both assert equality, so the discrepancy is authoritative-looking and silent. Callers who use the reconstructed value to set `max_fee` or `max_price_per_unit` bounds will under-provision and have their transactions rejected at execution time.

---

### Likelihood Explanation

Every V3 transaction on a post-0.13.3 network consumes L2 gas. The `l2_gas_price` is non-zero in all production blocks. The discrepancy therefore affects every `starknet_estimateFee` / `starknet_simulateTransactions` call for any V3 invoke, declare, or deploy-account transaction. No special conditions are required; the inconsistency is structural and permanent until `l2_gas_consumed` is added.

---

### Recommendation

1. Add `pub l2_gas_consumed: Felt` to `FeeEstimation` in `crates/apollo_rpc_execution/src/objects.rs`.
2. Populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`.
3. Update the doc-comment on `overall_fee` to read: *"equals `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * l2_gas_price`"*.
4. Update the OpenRPC schema description at `crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` line 3650 to match.
5. Add `l2_gas_consumed` to the `FEE_ESTIMATE` required fields list in the schema.

### Proof of Concept

```
# 1. Submit a V3 invoke transaction to starknet_estimateFee
# 2. Receive FeeEstimation:
#      gas_consumed        = 0x0
#      l1_gas_price        = 0xe8d4a51000
#      data_gas_consumed   = 0x80
#      l1_data_gas_price   = 0x3e8
#      l2_gas_price        = 0x1dcd65000
#      overall_fee         = 0x151eb86f3ed400
#      (l2_gas_consumed is absent)

# 3. Reconstruct fee per the spec:
reconstructed = 0x0 * 0xe8d4a51000 + 0x80 * 0x3e8
             = 0 + 128 * 1000
             = 128_000  (0x1f400)

# 4. Compare:
overall_fee   = 0x151eb86f3ed400  (≈ 6.0 × 10^15 fri)
reconstructed = 0x1f400           (≈ 1.3 × 10^5 fri)

# Discrepancy = l2_gas_consumed * l2_gas_price
#             = 0xb56b6 * 0x1dcd65000
#             ≈ 0x151eb86f1f0000  (matches the gap)
```

The fixture values are taken directly from the transaction prover's own RPC record at `crates/starknet_transaction_prover/resources/rpc_records/test_simulate_and_get_initial_reads.json` lines 92–100, which already contains the `l2_gas_consumed` the sequencer omits.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
