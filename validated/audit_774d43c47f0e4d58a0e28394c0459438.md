Audit Report

## Title
Stranded Router ETH Consumed by Subsequent Callers' WETH Payments — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` with no per-caller ownership tracking. ETH left in the router by any prior payable call is silently consumed to subsidise a later caller's WETH payment, causing the original depositor to lose ETH with no swap output. The exploiting caller receives a free or discounted swap funded by the victim's ETH.

## Finding Description
`pay()` at lines 73–84 reads `nativeBalance = address(this).balance` and, when `token == WETH`, uses that balance to wrap ETH before falling back to `transferFrom`. [1](#0-0) 

Every public swap entry point — `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), `exactOutput` (L154), and `multicall` (L39) — is `payable`. [2](#0-1) [3](#0-2) 

The `receive()` guard only blocks direct ETH sends; it cannot block ETH attached to payable function calls. [4](#0-3) 

If a caller sends excess ETH and omits `refundETH`, the surplus persists in the router across transactions. A subsequent caller whose `pay()` hits the `nativeBalance >= value` branch receives a fully free WETH payment, or the `nativeBalance > 0 && nativeBalance < value` branch receives a partial subsidy — in both cases consuming ETH they have no entitlement to. The original depositor loses ETH with no swap output.

Cross-transaction exploit path:
1. User A calls `exactInputSingle{value: 1 ether}` with `amountIn = 0.5 ether`, no `refundETH`. `pay()` wraps 0.5 ETH; 0.5 ETH remains stranded.
2. Attacker calls `exactInputSingle{value: 0}` with `amountIn = 1 ether`. `pay()` sees `nativeBalance = 0.5`, `value = 1` → hits the partial branch: wraps user A's 0.5 ETH, sends WETH to pool, then pulls only 0.5 WETH from attacker via `transferFrom`. Attacker pays half price; user A loses 0.5 ETH with no swap output.

## Impact Explanation
Direct loss of user principal: the original ETH sender loses ETH without receiving any swap output. The exploiting caller receives a free or discounted WETH payment funded by the victim's ETH. This is a direct loss of user principal satisfying the Critical/High/Medium gate.

## Likelihood Explanation
Medium. ETH stranding requires a user to send excess ETH without `refundETH` — a common pattern in DeFi multicall usage. Once stranded, any attacker monitoring `address(router).balance` can immediately exploit it in the next block with a zero-ETH swap call. The attack is repeatable and requires no special privileges.

## Recommendation
Track per-call ETH entitlement in transient storage: record `msg.value` at the start of each payable entry point and allow `pay()` to consume only up to that recorded amount. Alternatively, enforce that `pay()` only wraps ETH equal to `msg.value` of the current top-level call, and revert if `address(this).balance` exceeds that amount at entry (indicating stranded ETH from a prior call).

## Proof of Concept
```solidity
// 1. User A sends excess ETH, no refundETH
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    tokenIn: WETH, amountIn: 0.5 ether, ...
}));
// Router now holds 0.5 ETH (stranded)

// 2. Attacker sends 0 ETH, amountIn = 1 ETH
// pay() sees nativeBalance=0.5, value=1 → hits partial branch:
//   wraps 0.5 ETH (user A's), transferFrom attacker only 0.5 WETH
// Attacker pays 50% of required input; user A loses 0.5 ETH.
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn: WETH, amountIn: 1 ether, ...
}));
assertEq(address(router).balance, 0);       // stranded ETH consumed
assertEq(userA_ethLost, 0.5 ether);         // victim lost principal
```
