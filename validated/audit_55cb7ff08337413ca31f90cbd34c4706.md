### Title
Transfers Where `amount - fee` Falls Below Decimal-Normalization Threshold Are Permanently Frozen — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR bridge contract accepts and locks user tokens for a cross-chain transfer even when the net transfer amount (`amount - fee`) is too small to survive decimal normalization to the destination chain. Because `sign_transfer` enforces `amount_to_transfer > 0` **after** normalization, any transfer whose net amount normalizes to zero can never be signed, yet there is no cancellation path. The user's tokens are permanently frozen in `pending_transfers`.

---

### Finding Description

**Root cause — `init_transfer` (lines 523–619, `near/omni-bridge/src/lib.rs`)**

When a user initiates a cross-chain transfer via `ft_on_transfer` → `init_transfer`, the only fee-related validation is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This allows `fee = amount - 1`, leaving a net amount of 1 NEAR-unit. The tokens are immediately locked or burned on the source chain.

**Normalization check — `sign_transfer` (lines 447–521, `near/omni-bridge/src/lib.rs`)**

When a relayer later calls `sign_transfer`, the net amount is normalized to the destination chain's decimal precision:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

The codebase itself documents that `normalize_amount` uses floor division:

> "Since `denormalize(normalize(x)) <= x` due to floor division, the difference naturally captures the normalization remainder." [3](#0-2) 

For any token where NEAR stores more decimals than the destination chain (e.g., 24 NEAR decimals vs. 6 EVM decimals, normalization factor = 10¹⁸), any `amount_without_fee < 10¹⁸` normalizes to zero. The `require!` then panics with `InvalidAmountToTransfer`, and the relayer call reverts.

**No cancellation path exists**

`remove_transfer_message` is called in only two places:
- `sign_transfer_callback` — only when `fee.is_zero()` (line 657).
- `claim_fee_callback` — requires a proof of finalization on the destination chain (line 1094), which is impossible if the transfer was never signed. [4](#0-3) [5](#0-4) 

There is no admin cancel or user-initiated refund function. The transfer entry in `pending_transfers` and the locked/burned tokens are irrecoverable.

---

### Impact Explanation

A user whose `amount - fee` falls below the normalization threshold has their tokens permanently frozen:

- On NEAR: the NEP-141 tokens are transferred into the bridge contract and never returned.
- On EVM/Solana/StarkNet: no finalization ever occurs because `sign_transfer` always reverts.
- The `pending_transfers` entry persists indefinitely with no removal path.

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user:

1. **Accidental**: A user bridging a token with a large decimal gap (e.g., 24 NEAR vs. 6 EVM decimals, normalization unit = 10¹⁸ ≈ 1 NEAR) sets a fee that leaves a sub-unit remainder. This is easy to do when specifying fees in raw integer units.
2. **Via `update_transfer_fee`**: After initiating a transfer, the sender can raise their own fee (line 405–409) up to `amount - 1`, pushing the net amount below the normalization threshold on an already-locked transfer. [6](#0-5) 

The decimal gap between NEAR (24 decimals) and common EVM tokens (6–8 decimals) makes the normalization unit large enough that real-value amounts can fall below it.

---

### Recommendation

Add a normalization-aware minimum check at `init_transfer` time, before tokens are locked:

```rust
// After computing transfer_message, before locking tokens:
let token_address = self.get_token_address(
    init_transfer_msg.get_destination_chain(),
    token_id.clone(),
).near_expect(BridgeError::TokenNotFound);

let decimals = self.token_decimals
    .get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);

let net_amount = transfer_message
    .amount_without_fee()
    .near_expect(BridgeError::InvalidFee);

require!(
    Self::normalize_amount(net_amount, decimals) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Alternatively, add a DAO-accessible emergency cancel function that refunds the sender and removes the stuck `pending_transfers` entry.

---

### Proof of Concept

**Setup**: Token with 24 NEAR decimals, 6 EVM decimals. Normalization factor = 10¹⁸.

1. User calls `ft_transfer_call` on the NEP-141 token contract, transferring `amount = 2 × 10¹⁸` to the bridge with `msg = InitTransfer { fee: 10¹⁸ + 1, ... }`.
2. `init_transfer` passes the check `fee (10¹⁸+1) < amount (2×10¹⁸)`. Tokens are locked in the bridge.
3. `amount_without_fee() = 10¹⁸ - 1`.
4. `normalize_amount(10¹⁸ - 1, 6) = (10¹⁸ - 1) / 10¹⁸ = 0` (floor division).
5. Relayer calls `sign_transfer`; it panics: `require!(0 > 0, ...)` → `InvalidAmountToTransfer`.
6. No MPC signature is produced. No finalization occurs on EVM.
7. `pending_transfers` retains the entry. `fee.is_zero()` is false, so `sign_transfer_callback` never removes it. `claim_fee_callback` requires a destination-chain proof that can never exist.
8. `2 × 10¹⁸` NEAR-units are permanently frozen. [2](#0-1) [1](#0-0)
