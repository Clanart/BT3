Looking at the external report's vulnerability class — **asset accounting desync where tokens are collected in a contract but the expected distribution step never executes, leaving funds permanently stuck** — I need to find an analog in the Omni Bridge production files.

The most direct analog is in the StarkNet bridge's `init_transfer` fee handling.

---

### Title
Native Fee (STRK) Permanently Locked in StarkNet Bridge Contract — (`starknet/src/omni_bridge.cairo`)

### Summary

The `init_transfer` function in the StarkNet bridge contract collects `native_fee` in STRK tokens from users by transferring them into the contract itself. However, the contract exposes no function to withdraw or distribute these accumulated STRK tokens, causing every STRK native fee ever paid to be permanently and irrecoverably locked.

### Finding Description

In `omni_bridge.cairo`, the `init_transfer` function accepts a `native_fee` parameter. When `native_fee > 0`, it explicitly pulls STRK tokens from the caller into the contract's own address: [1](#0-0) 

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
```

The complete public interface of the contract is: [2](#0-1) 

The functions are: `log_metadata`, `deploy_token`, `fin_transfer`, `init_transfer`, `upgrade_token`, `set_pause_flags`, `pause_all`, `get_token_address`, `is_bridge_token`, `is_transfer_finalised`, and `upgrade`. **None of these provide any mechanism to withdraw or distribute the accumulated STRK native fees.** There is no `withdraw_fees`, `claim_fees`, or equivalent function.

This contrasts with the NEAR side of the bridge, which imports `ClaimFeeArgs` and has a dedicated fee-claiming flow: [3](#0-2) 

The StarkNet contract simply has no equivalent.

### Impact Explanation

Every STRK token paid as `native_fee` during `init_transfer` on StarkNet is permanently locked in the bridge contract with no recovery path. This constitutes an **irrecoverable lock of user and protocol funds** in the bridge flow. Additionally, relayers who process StarkNet → NEAR transfers cannot receive their STRK fee compensation, breaking the economic incentive for relayers to service this chain direction.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

High. Any user who pays a non-zero `native_fee` when calling `init_transfer` on the StarkNet bridge triggers this loss. No special conditions, privileged access, or attacker coordination is required — it is a normal user action on the public interface.

### Recommendation

Add a fee withdrawal function to the StarkNet bridge contract, restricted to an authorized role (e.g., `DEFAULT_ADMIN_ROLE` or a designated fee recipient), analogous to the `ClaimFeeArgs`-based fee claiming mechanism on the NEAR side. For example:

```cairo
fn withdraw_native_fees(ref self: ContractState, recipient: ContractAddress, amount: u128) {
    self.accesscontrol.assert_only_role(DEFAULT_ADMIN_ROLE);
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer(recipient, amount.into());
    assert(success, 'ERR_FEE_WITHDRAWAL_FAILED');
}
```

### Proof of Concept

1. User calls `init_transfer(token_address, 1000, 10, 50, "near:recipient.near", "")` on the StarkNet bridge, paying 50 STRK as `native_fee`.
2. The contract executes `transfer_from(user, bridge_contract, 50 STRK)` — 50 STRK now sits in the bridge contract.
3. The `InitTransfer` event is emitted with `native_fee: 50`, signaling to relayers that a fee is available.
4. A relayer processes the transfer on NEAR, expecting to claim the 50 STRK fee.
5. No function exists on the StarkNet contract to transfer those 50 STRK to the relayer or anyone else.
6. The 50 STRK is permanently locked. Repeated across all users paying native fees, the total locked amount grows unboundedly. [4](#0-3)

            ref self: ContractState,
            token_address: ContractAddress,
            amount: u128,
            fee: u128,
            native_fee: u128,

            message: ByteArray,
        ) {
            assert(!_is_paused(@self, PAUSE_INIT_TRANSFER), 'ERR_INIT_TRANSFER_PAUSED');

            assert(amount > 0, 'ERR_ZERO_AMOUNT');
            assert(fee < amount, 'ERR_INVALID_FEE');

            let origin_nonce = self.current_origin_nonce.read() + 1;
            self.current_origin_nonce.write(origin_nonce);

            let caller = get_caller_address();

            if self.is_bridge_token(token_address) {
                IBridgeTokenDispatcher { contract_address: token_address }
                    .burn(caller, amount.into());
            } else {
                let success = IERC20Dispatcher { contract_address: token_address }
                    .transfer_from(caller, get_contract_address(), amount.into());
                assert(success, 'ERR_TRANSFER_FROM_FAILED');

            if native_fee > 0 {
                let native_token = self.strk_token_address.read();
                let success = IERC20Dispatcher { contract_address: native_token }
                    .transfer_from(caller, get_contract_address(), native_fee.into());
                assert(success, 'ERR_FEE_TRANSFER_FAILED');
            }

            self
                .emit(
                    Event::InitTransfer(

                            sender: caller,
                            token_address,
                            origin_nonce,
                            amount,
                            fee,
                            native_fee,
                            recipient,
                            message,
                        },
                    ),
