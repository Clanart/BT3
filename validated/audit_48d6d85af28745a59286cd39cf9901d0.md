### Title
Periphery contracts accept arbitrary pool addresses without factory validation, enabling malicious pool to drain user-approved tokens — (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol, metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

---

### Summary

`MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` accept caller-supplied `pool` addresses and interact with them (swap callbacks, liquidity callbacks) without verifying that the address was deployed by the legitimate `MetricOmmPoolFactory`. A malicious actor can deploy a fake contract implementing `IMetricOmmPoolActions`/`IMetricOmmPool`, advertise it as a legitimate pool, and use the periphery's own callback payment logic to drain tokens from any user who interacts with it through the router or liquidity adder.

---

### Finding Description

**Router path (`MetricOmmSimpleRouter`):**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all accept a `pool` (or `pools[]`) parameter directly from the caller and immediately call `IMetricOmmPoolActions(pool).swap(...)` without any factory membership check. [1](#0-0) 

Before calling `swap`, the router stores the caller-supplied pool address as the *expected* callback caller in transient storage: [2](#0-1) 

The callback guard `_requireExpectedCallbackCaller(msg.sender)` then accepts any address that matches what was stored — including the malicious pool itself: [3](#0-2) 

Inside `_justPayCallback`, the router pays the positive delta amount of `tokenIn` from the original user (`_getPayer()`) to `msg.sender` (the malicious pool): [4](#0-3) 

**Liquidity adder path (`MetricOmmPoolLiquidityAdder`):**

`addLiquidityExactShares` and `addLiquidityWeighted` accept an arbitrary `pool` address. The NatSpec comment explicitly acknowledges the missing guard: [5](#0-4) 

`_addLiquidity` stores the caller-supplied pool as the expected callback caller, then calls `pool.addLiquidity(...)`: [6](#0-5) 

In `metricOmmModifyLiquidityCallback`, the guard only checks that `msg.sender == expectedPool` — which is the malicious pool — and then pulls tokens from the user up to `max0`/`max1`: [7](#0-6) 

**The factory already exposes the necessary check** (`isPool`) but neither periphery contract calls it: [8](#0-7) 

The router already holds a `factory` reference passed at construction: [9](#0-8) 

---

### Impact Explanation

A user who interacts with a malicious pool through the router loses their `tokenIn` up to the amount they have approved to the router. The malicious pool's `swap()` implementation calls back `metricOmmSwapCallback` with an inflated positive delta (e.g., `amount1Delta = user_approval`), causing the router to transfer that amount from the user to the malicious pool. The malicious pool then returns fabricated swap deltas that satisfy the `amountOutMinimum` slippage check, while sending zero actual output tokens to the recipient. The transaction succeeds and the user's funds are gone.

For the liquidity adder, the malicious pool's `addLiquidity()` calls back `metricOmmModifyLiquidityCallback` requesting up to `maxAmountToken0`/`maxAmountToken1`, which the adder pays from the user without receiving any LP position in return.

**Severity: Medium** — direct loss of user principal; requires social engineering or UI-level deception to supply the malicious pool address, but the on-chain exploit path is fully reachable once the user is induced to call the periphery with the attacker's pool.

---

### Likelihood Explanation

- Deploying a malicious contract implementing `IMetricOmmPoolActions` is trivial and permissionless.
- Users routinely interact with pools via front-ends that resolve pool addresses; a phishing UI, a compromised integration, or a misleading pool listing is sufficient to supply the malicious address.
- Users commonly grant large or unlimited ERC-20 approvals to router contracts, maximising the drained amount.
- The factory's `isPool` registry is already populated and queryable; the missing check is a single call away.

---

### Recommendation

In `MetricOmmSimpleRouter`, add a factory validation helper and call it for every pool address before setting the callback context:

```solidity
function _requireLegitimatePool(address pool) internal view {
    if (!IMetricOmmPoolFactory(factory).isPool(pool)) revert NotARegisteredPool(pool);
}
```

Call `_requireLegitimatePool(pool)` at the top of `exactInputSingle`, `exactOutputSingle`, and inside the loop of `exactInput`/`exactOutput` before each `_setNextCallbackContext`.

In `MetricOmmPoolLiquidityAdder`, either accept a `factory` address at construction and apply the same check, or require callers to pass a factory reference and validate it before `_setPayContext`.

### Proof of Concept

```solidity
contract MaliciousPool {
    address router;
    address victim;
    address tokenIn;
    uint256 drainAmount;

    // Implements IMetricOmmPoolActions.swap
    function swap(
        address, bool, int128, uint128, bytes calldata, bytes calldata
    ) external returns (int128, int128) {
        // Call back the router: positive amount1Delta triggers payment
        IMetricOmmSwapCallback(router).metricOmmSwapCallback(
            0,
            int256(drainAmount), // positive → router pays this from victim
            ""
        );
        // Return fabricated deltas that satisfy amountOutMinimum = 1
        return (0, -1);
    }
}

// Attack:
// 1. Attacker deploys MaliciousPool, sets router/victim/tokenIn/drainAmount
// 2. Victim approves router for tokenIn (e.g., 10_000 USDC)
// 3. Victim calls router.exactInputSingle({
//        pool: MaliciousPool,
//        tokenIn: USDC,
//        amountIn: 10_000,
//        amountOutMinimum: 1,
//        recipient: victim, ...
//    })
// 4. Router sets callback context: expectedPool = MaliciousPool, payer = victim
// 5. Router calls MaliciousPool.swap(...)
// 6. MaliciousPool calls back metricOmmSwapCallback(0, 10_000, "")
// 7. _requireExpectedCallbackCaller passes (msg.sender == MaliciousPool == expectedPool)
// 8. Router pays 10_000 USDC from victim to MaliciousPool
// 9. MaliciousPool.swap returns (0, -1) → amountOut = 1 ≥ amountOutMinimum = 1 → passes
// 10. Victim loses 10_000 USDC, receives 0 tokens
```
