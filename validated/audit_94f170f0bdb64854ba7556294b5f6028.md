### Title
Missing Deadline on Liquidity-Add Operations Allows Stale-Price Execution and Immediate LP Principal Loss - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

---

### Summary

`MetricOmmPoolLiquidityAdder` exposes `addLiquidityExactShares` and `addLiquidityWeighted` with no deadline parameter and no deadline check. A pending transaction can sit in the mempool while the oracle price moves, then execute at a price the LP never agreed to, causing immediate impermanent loss on the deposited principal.

---

### Finding Description

`MetricOmmSimpleRouter` correctly guards every swap entry point with `_checkDeadline`: [1](#0-0) 

All four router functions call it: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

`MetricOmmPoolLiquidityAdder` has no equivalent. Neither overload of `addLiquidityExactShares` nor either overload of `addLiquidityWeighted` accepts or checks a deadline: [6](#0-5) [7](#0-6) 

`addLiquidityWeighted` has cursor-bounds validation: [8](#0-7) 

However, `_validateBinAndBinPosition` reads only `curBinIdx` and `curPosInBin` from slot0. It does **not** read `curBinDistFromProvidedPriceE6`, which is the field that tracks how far the pool cursor has drifted from the live oracle price: [9](#0-8) 

Slot0 layout confirms `curBinDistFromProvidedPriceE6` is a separate field that the cursor-bounds check ignores: [10](#0-9) 

Therefore, if the oracle price moves while no swaps have occurred (so `curBinIdx`/`curPosInBin` are unchanged), the cursor-bounds check passes and the LP's transaction executes at the stale price.

`addLiquidityExactShares` has no cursor-bounds check at all — only max-token caps — so it is fully unprotected.

---

### Impact Explanation

Metric OMM is an oracle-based AMM. The pool fetches a live bid/ask from the price provider on every swap. When an LP adds liquidity to a bin at oracle price P, and the oracle price subsequently moves to P′ > P before the transaction is mined:

1. The LP's token0 liquidity sits in a bin priced below the new oracle mid.
2. The first swap after the LP's add sells token0 out of that bin at price P (below market P′).
3. The LP receives token1 at the old price, losing `(P′ − P) × amount_sold` of value.

This is a direct reduction of the LP's deposited principal, bounded by `maxAmountToken0`/`maxAmountToken1` caps but otherwise proportional to the oracle price move and the liquidity depth. On volatile oracle feeds or congested networks where transactions can be delayed by minutes, the loss can be material.

`addLiquidityExactShares` is fully exposed (no cursor bounds, no deadline).
`addLiquidityWeighted` is partially exposed: cursor bounds protect against bin-cursor drift caused by prior swaps, but not against oracle price movement that has not yet been reflected in the cursor.

---

### Likelihood Explanation

- Any LP transaction submitted with a low gas price on a congested network can be delayed.
- Oracle prices in Metric OMM are pushed off-chain and can move independently of on-chain swap activity, so `curBinIdx` can remain unchanged while the oracle price shifts significantly.
- No privileged access is required; any LP calling either function is exposed.
- The trigger is passive: the attacker need only wait for the LP's pending transaction and a favorable oracle update.

---

### Recommendation

Add a `uint256 deadline` parameter to all four `addLiquidityExactShares` and `addLiquidityWeighted` overloads and check it at entry, mirroring the router pattern:

```solidity
// In MetricOmmPoolLiquidityAdder (or a shared base)
function _checkDeadline(uint256 deadline) internal view {
    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
}
```

Call `_checkDeadline(deadline)` as the first statement in each public liquidity function, before any pool interaction.

---

### Proof of Concept

1. Alice calls `addLiquidityExactShares` targeting bin 4 (the active bin at oracle price P = 2000 USDC/ETH) with `maxAmountToken0 = 1 ETH`, `maxAmountToken1 = 2000 USDC`. Transaction is submitted but not yet mined.
2. The oracle price provider pushes a new price P′ = 2100 USDC/ETH. No swap has occurred, so `curBinIdx` and `curPosInBin` are unchanged.
3. Alice's transaction is mined. No deadline check exists, so it proceeds. Alice's 1 ETH is deposited into bin 4 at the old price range anchored around 2000.
4. Bob immediately calls `exactInputSingle` selling USDC for ETH. The pool fetches the oracle price (2100) and executes the swap, buying Alice's ETH out of bin 4 at the bin's price (≈2000), paying Alice ≈2000 USDC per ETH.
5. Alice has effectively sold 1 ETH for 2000 USDC when the market price was 2100 USDC — a direct loss of ≈100 USDC (≈5%) of principal with no recourse. [6](#0-5) [8](#0-7)

    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
