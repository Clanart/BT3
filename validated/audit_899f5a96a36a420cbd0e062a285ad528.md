### Title
Missing Duplicate-Key Check in `batch_insert` Corrupts DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for every item beyond the first two. An untrusted caller can supply a batch containing a key that already exists in the tree (or a key repeated within the batch itself), silently inserting a second leaf node with the same `KeyId` into the blob. The in-memory cache is silently overwritten, the orphaned duplicate leaf remains in the raw blob, and the computed Merkle root no longer reflects the true committed state. Proofs of inclusion derived from the corrupted tree are therefore invalid.

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `self.insert()` (with those guards) only for the first two items when the tree has ≤1 existing leaf (lines 578–585). All remaining items are written directly to the blob via `insert_entry_to_blob` with **no existence check** (lines 587–603):

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` (lines 1013–1030) calls `block_status_cache.add_leaf`, which calls `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index` (lines 191–192). `HashMap::insert` silently overwrites any existing entry for the same key. The old leaf node at its original blob index is now orphaned — it is still present in the raw bytes but unreachable through the cache — while the cache points to the new duplicate leaf. The subtree built from the batch (lines 607–654) incorporates the duplicate leaf's hash into the root, producing a root hash that does not match the canonical tree.

When the tree already has ≥2 leaves, **every** item in the batch bypasses the duplicate check, not just items beyond the first two.

### Impact Explanation

The corrupted Merkle root is the committed state for a DataLayer store. Any `get_proof_of_inclusion` call after a corrupted `batch_insert` traverses the tree using the cache, which now points to the wrong (duplicate) leaf index. The proof path is built from a tree whose root hash was computed over a blob containing a phantom duplicate leaf. A verifier checking the proof against the stored root will accept it even though the tree's actual key-value mapping is inconsistent. This satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`batch_insert` is exposed directly as a Python binding (`py_batch_insert`, lines 1503–1519) and is the primary bulk-insertion API used by the DataLayer. Any caller — including one constructing a delta sync or replaying a blob — that passes a batch containing a key already present in the tree (or a repeated key within the batch) triggers the bug silently, with no error returned.

### Recommendation

Before the fast-path loop in `batch_insert` (line 587), add the same guards present in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Alternatively, pre-validate the entire input batch for duplicates (both within the batch and against the existing tree) before writing any nodes.

### Proof of Concept

```
Tree starts with 2 existing leaves (leaf_count >= 2).
Call batch_insert([(dup_key, new_value)], [new_hash])
  → leaf_count > 1, so the `if self.block_status_cache.leaf_count() <= 1` branch is skipped entirely.
  → The loop at line 587 runs for dup_key without any contains_key check.
  → insert_entry_to_blob writes a new LeafNode(dup_key, new_value, new_hash) at a fresh index.
  → add_leaf silently overwrites key_to_index[dup_key] = new_index.
  → The old leaf at old_index is now orphaned in the blob.
  → The subtree root hash is computed over new_hash, not the original leaf's hash.
  → get_proof_of_inclusion(dup_key) returns a proof path built from the new (duplicate) leaf,
    but the root hash stored in the tree was computed over a blob containing both the old and
    new leaf, making the proof inconsistent with the canonical committed state.
```

            Err(Error::UnmatchedKeysAndValues(
