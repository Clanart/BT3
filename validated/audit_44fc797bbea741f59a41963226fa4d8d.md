### Title
Asymmetric Staleness Enforcement Between L1 Gas Price Provider and ETH/STRK Oracle Silently Commits Stale Conversion Rate into FRI Gas Price Block Commitments - (File: crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs)

---

### Summary

The sequencer computes block gas prices from two independent data sources: the `L1GasPriceProvider` (L1 base fee and blob fee in wei) and the `EthToStrkOracleClient` (ETH/STRK conversion rate). The L1 gas price path enforces a hard `max_time_gap_seconds` (900 s) staleness guard and returns `StaleL1GasPricesError` when the data is too old. The ETH/STRK oracle path has **no equivalent staleness check**: when the current-bucket query is not yet resolved, it silently falls back to the previous quantized-bucket rate cached in the LRU, which can be up to `2 × lag_interval_seconds` (1 800 s in production) old. Both proposer and validator call the same oracle with the same `lag_interval_seconds`, so they independently derive the same stale rate, the `within_margin` cross-check passes, and the wrong FRI gas prices are committed into the block hash and used for all fee calculations in that block.

---

### Finding Description

**Two data sources, one without a staleness guard**

`get_l1_prices_in_fri_and_wei_and_conversion_rate` fires both calls in parallel:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:147-150
let (eth_to_fri_rate, price_info) = tokio::join!(
    l1_gas_price_provider_client.get_eth_to_fri_rate(timestamp),
    l1_gas_price_provider_client.get_price_info(BlockTimestamp(timestamp))
);
``` [1](#0-0) 

**Path 1 – L1 gas price: staleness is enforced**

`get_price_info` compares the requested block timestamp against the newest scraped L1 sample:

```rust
// crates/apollo_l1_gas_price/src/l1_gas_price_provider.rs:119-124
if timestamp.0 > (*last_timestamp + self.config.max_time_gap_seconds) {
    return Err(L1GasPriceProviderError::StaleL1GasPricesError { ... });
}
``` [2](#0-1) 

`max_time_gap_seconds` defaults to 900 s in both the code and the production deployment config. [3](#0-2) [4](#0-3) 

**Path 2 – ETH/STRK oracle: no staleness check**

`eth_to_fri_rate` quantizes the timestamp and, when the current-bucket query is still in flight, silently returns the previous bucket's cached rate:

```rust
// crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs:196-227
let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
    .checked_div(self.config.lag_interval_seconds)
    .expect("lag_interval_seconds should be non-zero");
...
if !handle.is_finished() {
    if let Some(rate) = cache.get(&(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)) {
        return Ok(*rate);   // ← stale rate returned as Ok, no age check
    }
    return Err(EthToStrkOracleClientError::QueryNotReadyError(timestamp));
}
``` [5](#0-4) 

`NUMBER_OF_TIMESTAMPS_BACK = 1`. With `lag_interval_seconds = 900` (production), the fallback rate was fetched for `adjusted_timestamp = (quantized_timestamp − 1) × 900`. Relative to the current block timestamp `T`, this rate can be up to `2 × 900 = 1 800 s` old. There is no check analogous to `max_time_gap_seconds` before returning it.

**Stale rate flows into block hash and fee calculations**

The stale `eth_to_fri_rate` is used to convert wei prices to FRI:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:170-174
let l1_gas_prices_fri_result =
    L1PricesInFri::convert_from_wei(&prices_in_wei, eth_to_fri_rate);
if let Ok(prices_in_fri) = l1_gas_prices_fri_result {
    return (prices_in_fri, prices_in_wei, eth_to_fri_rate);
}
``` [6](#0-5) 

The resulting `l1_gas_price_fri` and `l1_data_gas_price_fri` are placed in `ProposalInit`, passed to `convert_to_sn_api_block_info`, and fed into `BlockInfo::gas_prices`. Those prices are then chained into the block hash via `gas_prices_to_hash` inside `calculate_block_hash`: [7](#0-6) 

They are also used by the blockifier for every fee calculation in the block.

**Why the `within_margin` cross-check does not catch it**

During proposal validation, `is_block_info_valid` calls `get_l1_prices_in_fri_and_wei` with the same `timestamp` and compares the result against the proposer's values:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:302-318
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, ...))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
``` [8](#0-7) 

Because every validator independently calls the same `EthToStrkOracleClient` with the same `lag_interval_seconds`, they all fall back to the same previous-bucket rate during an oracle slowdown. Both sides of `within_margin` are computed from the same stale rate, so the check passes and the wrong FRI prices are accepted into the committed block.

---

### Impact Explanation

Wrong `l1_gas_price_fri` / `l1_data_gas_price_fri` values are committed into the block hash and used for every fee calculation in the block. This produces incorrect fee charges, incorrect STRK balance changes, and therefore an incorrect state root — matching the impact category **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** and **"Wrong state … or revert result from blockifier/syscall/execution logic for accepted input"**.

The magnitude of the error is proportional to the percentage change in the ETH/STRK rate between the stale bucket and the current one. During periods of high STRK price volatility, a 1 800-second-old rate can differ by tens of percent from the live rate.

---

### Likelihood Explanation

The trigger is any period during which the external oracle API responds slowly (not necessarily down — a response latency exceeding `query_timeout_sec` = 10 s is sufficient to leave the current-bucket query unresolved at block-proposal time). This is a realistic operational condition. No privileged access is required; the condition arises automatically from normal network or API latency.

---

### Recommendation

Add a maximum-age check to the fallback path in `EthToStrkOracleClient::eth_to_fri_rate`, mirroring the `max_time_gap_seconds` guard in `L1GasPriceProvider::get_price_info`. Concretely:

1. Store the wall-clock time at which each quantized-bucket rate was cached.
2. Before returning the `quantized_timestamp − 1` fallback, verify that `now − cached_at ≤ max_oracle_rate_age_seconds` (a new field in `EthToStrkOracleConfig`, defaulting to e.g. `2 × lag_interval_seconds`).
3. If the cached rate is too old, return `QueryNotReadyError` instead of the stale value, allowing the orchestrator's fallback chain (`previous_block_info` → defaults) to engage.

This mirrors the pattern already used for L1 gas prices and eliminates the asymmetry between the two data sources.

---

### Proof of Concept

1. Production config: `lag_interval_seconds = 900`, `query_timeout_sec = 10`.
2. The oracle API becomes slow (responses take > 10 s) at wall time `W`.
3. At block-proposal time `T > W + 900` (a new quantized bucket has started), `eth_to_fri_rate(T)` is called.
4. `quantized_timestamp = (T − 900) / 900` — a new bucket; no cache entry exists.
5. A new query is spawned but is not yet finished (oracle still slow).
6. The code checks `cache.get(&(quantized_timestamp − 1))` — this entry exists from before the slowdown, with a rate that was valid for timestamps up to `T − 900`.
7. `Ok(stale_rate)` is returned silently. The stale rate is up to 1 800 s old.
8. `get_l1_prices_in_fri_and_wei_and_conversion_rate` treats both `eth_to_fri_rate` and `price_info` as `Ok` and computes FRI prices using `stale_rate`.
9. Every validator independently repeats steps 3–8 with the same `stale_rate`, so `within_margin` passes.
10. The block is committed with wrong `l1_gas_price_fri` / `l1_data_gas_price_fri` in its hash, and all transaction fees in the block are computed from the wrong rate.

```rust
    async fn eth_to_fri_rate(&self, timestamp: u64) -> Result<u128, EthToStrkOracleClientError> {
        const NUMBER_OF_TIMESTAMPS_BACK: u64 = 1;
        let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
            .checked_div(self.config.lag_interval_seconds)
            .expect("lag_interval_seconds should be non-zero");

        let mut cache = self.cached_prices.lock().unwrap();

        if let Some(rate) = cache.get(&quantized_timestamp) {
            debug!("Cached conversion rate for timestamp {timestamp} is {rate}");
            return Ok(*rate);
        }

        // Check if there is a query already sent out for this timestamp, if not, start one.
        let mut queries = self.queries.lock().unwrap();
        let handle = queries
            .get_or_insert_mut(quantized_timestamp, || self.spawn_query(quantized_timestamp));
        // If the query is not finished, return an error.
        if !handle.is_finished() {
            debug!("Query not yet resolved: timestamp={timestamp}");
            // If the previous quantized timestamp is in the cache, use it.
            if let Some(rate) = cache.get(&(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)) {
                debug!(
                    "Query not yet resolved: timestamp={timestamp}, using previous rate {rate} \
                     from quantized timestamp={}",
                    (quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)
                        * self.config.lag_interval_seconds
                );
                return Ok(*rate);
            }
            // If not, return a query not ready error.
            return Err(EthToStrkOracleClientError::QueryNotReadyError(timestamp));
        }
