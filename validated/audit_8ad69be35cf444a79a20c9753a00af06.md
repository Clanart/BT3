### Title
Blocklisted Recipient Causes Permanent Irrecoverable Fund Lock in `finTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` uses a push-based token delivery model: it unconditionally pushes tokens to `payload.recipient` via `safeTransfer`. For tokens with admin-controlled address blocklists (USDC, USDT), any transfer to a blocklisted address reverts. Because the recipient address is cryptographically committed inside the MPC-signed payload, and because the NEAR side has no mechanism to cancel or re-route a transfer whose EVM finalization permanently fails, the bridged funds are irrecoverably locked on NEAR.

---

### Finding Description

In `OmniBridge.finTransfer`, after verifying the MPC signature, the contract unconditionally pushes tokens to `payload.recipient`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol, lines 350-354
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
```

The recipient address is part of the Borsh-encoded, MPC-signed payload:

```solidity
// lines 297-298
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
```

If `payload.tokenAddress` is a token with an admin-controlled blocklist (e.g., USDC, USDT) and `payload.recipient` is on that blocklist, `safeTransfer` will always revert. Because the recipient is committed in the MPC signature, no alternative recipient can be substituted without a new MPC signature over a new transfer message.

On the NEAR side, tokens are locked/burned at `init_transfer` time. There is no on-chain mechanism in the NEAR bridge to cancel a pending NEAR→EVM transfer and refund the sender when EVM finalization permanently fails. The `fin_transfer_send_tokens_callback` refund path only handles failed NEAR-side `ft_transfer_call` callbacks — it has no analogue for failed EVM-side finalizations.

The `completedTransfers[payload.destinationNonce] = true` write at line 287 is rolled back on revert, so the nonce is not permanently consumed. However, this provides no relief: every retry with the same signed payload will revert for the same reason, and no alternative payload can be constructed without a new MPC signature.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who initiates a NEAR→EVM bridge transfer of a blocklist-capable token (USDC, USDT) to an EVM recipient address that is on the token's blocklist will have their tokens permanently locked on NEAR. The EVM `finTransfer` call will revert on every attempt. There is no cancel, re-route, or refund path in the NEAR bridge contract for this scenario. The funds are unrecoverable without an admin contract upgrade.

---

### Likelihood Explanation

**Low.** Two conditions must coincide: (1) the bridged token must implement an admin-controlled blocklist (USDC and USDT both do, and both are high-value bridge targets), and (2) the intended EVM recipient must be on that blocklist at the time of finalization. The second condition can arise naturally (e.g., a sanctioned address, a user whose address was later blocked) or can be induced by a malicious actor who controls the recipient address and arranges to be blocklisted after initiating the transfer.

---

### Recommendation

Adopt the Pull-over-Push pattern for the native ERC20 path in `finTransfer`. Instead of calling `safeTransfer` directly to `payload.recipient`, credit the amount to a claimable balance mapping and let the recipient withdraw separately. This decouples finalization (nonce consumption, event emission) from token delivery, so a blocklisted recipient cannot prevent the nonce from being permanently settled and cannot lock funds in the bridge.

Alternatively, catch the revert from `safeTransfer` and store the unclaimed amount in a rescue mapping that the recipient (or an admin) can later withdraw once the blocklist condition is resolved.

---

### Proof of Concept

1. Alice holds USDC on NEAR (the NEAR-side wrapped representation).
2. Alice calls `init_transfer` on the NEAR bridge, specifying her EVM address `0xAlice` as recipient. NEAR-side USDC is locked/burned.
3. The NEAR MPC co-signs a `TransferMessagePayload` with `tokenAddress = USDC_EVM`, `recipient = 0xAlice`, `amount = X`.
4. Between step 2 and step 5, Circle's admin adds `0xAlice` to the USDC blocklist (e.g., due to a sanctions update).
5. A relayer calls `OmniBridge.finTransfer(signature, payload)`.
6. Execution reaches line 351: `IERC20(USDC_EVM).safeTransfer(0xAlice, X)`. USDC's `transfer` reverts because `0xAlice` is blocklisted.
7. The entire transaction reverts. `completedTransfers[nonce]` is rolled back to `false`.
8. Every subsequent retry with the same signed payload reverts identically.
9. No new MPC signature can be obtained for a different recipient without a new NEAR-side transfer (which requires new tokens).
