### Title
Gateway `validate_proof_block_number` Uses Incremented Block Number While `validate_proof_block_hash` Reads Stale State, Causing Valid Proof-Facts Transactions to Be Rejected - (`File: crates/blockifier/src/transaction/account_transaction.rs`)

### Summary

The gateway's stateful validator increments the block number by one (`unchecked_next()`) when building the block context for pre-validation, so that `validate_proof_block_number` permits `proof_block_number = N − STORED_BLOCK_HASH_BUFFER + 1`. However, `validate_proof_block_hash` reads the block-hash contract from the **current** state (block N), where that slot is still zero. The hash check therefore always fails for the one block that the number check just unlocked, causing the gateway to reject a transaction that would succeed when actually executed in block N+1 (after `pre_process_block` writes the missing hash).

### Finding Description

**Root cause — inconsistent block-number horizon in `validate_proof_facts`**

`run_validate_entry_point` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` builds the block context with:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // N → N+1
``` [1](#0-0) 

The state reader, however, is still anchored to block N (the latest synced block). The resulting `BlockContext` (block_number = N+1) is passed into `perform_validations`, which calls `perform_pre_validation_stage` → `validate_proof_facts`.

Inside `validate_proof_facts`, two sub-checks run back-to-back:

**Check 1 — block-number bound** (`validate_proof_block_number`):

```rust
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER).ok_or_else(|| { … })?;
if proof_block_number > max_allowed { return Err(…); }
``` [2](#0-1) 

With `current_block_number = N+1` and `STORED_BLOCK_HASH_BUFFER = 10`, `max_allowed = N − 9`. A proof referencing block `N − 9` **passes** this check.

**Check 2 — block-hash match** (`validate_proof_block_hash`):

```rust
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;
if stored_block_hash != proof_block_hash { return Err(…); }
``` [3](#0-2) 

`state` is the reader for block N. The block-hash contract at block N only holds hashes for blocks 0 … N−10 (written by `pre_process_block` when block N was processed). The slot for block `N − 9` is **zero**. The check therefore always fails with either "Proof block hash is zero" or "Block hash mismatch".

**What happens at actual execution time (block N+1)**

`pre_process_block` runs at the start of block N+1 and writes the hash of block `N+1 − 10 = N − 9`:

```rust
let should_block_hash_be_provided =
    next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
// …
state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
``` [4](#0-3) 

After this write the hash for block `N − 9` is present, and the OS-level `check_proof_facts` constraint `assert_nn_le(base_block_number, current_block_number − STORED_BLOCK_HASH_BUFFER)` is satisfied: [5](#0-4) 

The transaction would succeed in block N+1, but the gateway has already rejected it.

**Analogy to the seed bug**

The original veRWA bug: `require(toLocked.end >= fromLocked.end)` used a single ordering check that did not account for the self-delegation special case, forcing users into a costly workaround. Here, `validate_proof_block_number` uses the *next-block* horizon (`N+1`) while `validate_proof_block_hash` uses the *current-block* state (`N`). The number check opens a one-block window (`proof_block_number = N − STORED_BLOCK_HASH_BUFFER + 1`) that the hash check immediately closes, making that window permanently unreachable at the gateway.

### Impact Explanation

Any user who submits an Invoke V3 transaction with `proof_facts` referencing block `N − STORED_BLOCK_HASH_BUFFER + 1` (the highest block number the number-check permits) receives a spurious `InvalidProofFacts` rejection from the gateway. The transaction is valid and would be accepted by the blockifier in block N+1. This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

Any unprivileged user submitting a client-side-proving Invoke V3 transaction can trigger this. The user naturally picks the most recent allowed block for their proof (to minimise staleness). The gateway's block-number check explicitly advertises `N − STORED_BLOCK_HASH_BUFFER + 1` as the maximum, so users will attempt it. No special setup is required beyond having `allow_client_side_proving = true` in the gateway config.

### Recommendation

In `validate_proof_block_number`, derive `max_allowed` from the **actual current block** (the block whose state is readable), not the simulated next block:

```rust
// Use block_context.block_info.block_number - 1 (the last finalised block)
// so the number bound and the hash-storage bound are consistent.
let effective_current = current_block_number.0.saturating_sub(1);
let max_allowed = effective_current.checked_sub(STORED_BLOCK_HASH_BUFFER)…;
```

Alternatively, simulate the `pre_process_block` hash write in the gateway's cached state before calling `validate_proof_facts`, so the hash for block `N − STORED_BLOCK_HASH_BUFFER + 1` is present when the hash check runs.

### Proof of Concept

1. Let the latest synced block be N = 20, `STORED_BLOCK_HASH_BUFFER = 10`.
2. Submit an Invoke V3 transaction with `proof_facts` containing `block_number = 11` (`= N − STORED_BLOCK_HASH_BUFFER + 1`) and the correct hash of block 11.
3. Gateway builds block context with `block_number = 21`.
4. `validate_proof_block_number(11, 21)` → `max_allowed = 11` → **passes**.
5. `validate_proof_block_hash(hash_of_11, 11, state_at_N=20)` → `stored = 0` (block 11's hash is written only when block 21 is processed) → **fails** with "Block hash mismatch".
6. Gateway returns `InvalidProofFacts` and drops the transaction.
7. If the same transaction were submitted after block 21 is finalised (state now at N=21), `pre_process_block` has written `hash(11)` into the contract, and the identical transaction would be accepted. [6](#0-5) [7](#0-6) [8](#0-7)

    fn validate_proof_block_number(
        proof_block_number: u64,
        current_block_number: BlockNumber,
    ) -> TransactionPreValidationResult<()> {
        // Proof block must be old enough to have a stored block hash.
        // Stored block hashes are guaranteed only up to: current - STORED_BLOCK_HASH_BUFFER.
        let max_allowed =
            current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER).ok_or_else(|| {
                TransactionPreValidationError::InvalidProofFacts(format!(
                    "The current block number {current_block_number} is below the required \
                     block-hash retention buffer: {STORED_BLOCK_HASH_BUFFER}."
                ))
            })?;

        if proof_block_number > max_allowed {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "The proof block number {proof_block_number} is too recent. The maximum allowed \
                 block number is {max_allowed}."
            )));
        }

        Ok(())
    }

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
