### Title
Stale Gas Refund Counter in `Simulator.applyCall` Causes Incorrect Gas Accounting Across Multi-Call Simulations - (File: `x/evm/keeper/simulate.go`)

---

### Summary

The `Simulator` in Ethermint reuses a single shared `*statedb.StateDB` (`sim.state`) across all calls within a simulated block. The `StateDB.Prepare()` call between transactions resets the access list and transient storage, but **never resets the `refund` counter**. Gas refunds accumulated by call N persist into call N+1's refund calculation, causing `eth_simulateV1` to report systematically incorrect (under-counted) `gasUsed` values for any call that follows a call with SSTORE refunds.

---

### Finding Description

The `Simulator` struct holds a single `sim.state *statedb.StateDB` that is created once and shared across every call in every simulated block: [1](#0-0) 

Inside `processBlock`, for each call in the block, `applyCall` is invoked on this shared state: [2](#0-1) 

Inside `applyCall`, `sim.state.Prepare()` is called to reset per-transaction state: [3](#0-2) 

`StateDB.Prepare()` resets the access list and transient storage, but **does not reset `s.refund`**: [4](#0-3) 

The `refund` field is a plain `uint64` on the `StateDB` struct: [5](#0-4) 

After the EVM executes, `applyCall` reads the accumulated refund counter to compute the gas refund for the current call: [6](#0-5) 

`GasToRefund` uses `availableRefund` as a cap: [7](#0-6) 

Because `sim.state.GetRefund()` returns the **cumulative** refund from all previous calls in the block (not just the current call), call N+1 can claim refunds that were earned by call N. The `Finalise` call between transactions does not help — it only removes self-destructed accounts from `stateObjects`: [8](#0-7) 

By contrast, the production `ApplyMessageWithConfig` path creates a **fresh** `StateDB` for every transaction, so the refund counter always starts at zero: [9](#0-8) 

---

### Impact Explanation

The `eth_simulateV1` JSON-RPC endpoint (backed by `Simulator`) returns incorrect `gasUsed` values for any call that follows a call with SSTORE refunds. Concretely:

- Call N clears a storage slot → adds up to 4,800 gas to `sim.state.refund` (EIP-3529 SSTORE clear refund).
- Call N+1 executes with `gasUsed = G` and generates zero refund of its own.
  - **Correct**: `GasToRefund(0, G, 5)` = 0 → reported `gasUsed = G`.
  - **Buggy**: `GasToRefund(4800, G, 5)` = min(G/5, 4800) → reported `gasUsed = G − min(G/5, 4800)`.

The simulation reports call N+1 as consuming significantly less gas than it actually would in real execution. A user or tool relying on `eth_simulateV1` for gas estimation will derive an insufficient gas limit, causing the real on-chain transaction to fail with out-of-gas. This is an incorrect gas accounting result from a public JSON-RPC simulation path.

---

### Likelihood Explanation

Any `eth_simulateV1` request containing multiple calls where an earlier call clears EVM storage (a common DeFi pattern — e.g., token approvals, swap completions, vault withdrawals) will trigger this bug. The endpoint is publicly reachable with no authentication. No privileged role is required. The attacker-controlled entry path is simply crafting a multi-call `eth_simulateV1` request where call 1 generates SSTORE refunds and call 2 is the target whose gas cost the attacker wants to understate.

---

### Recommendation

Reset `sim.state.refund` to zero at the start of each `applyCall` invocation, mirroring the behavior of `ApplyMessageWithConfig` which creates a fresh `StateDB` per transaction. The simplest fix is to add a `ResetRefund()` method to `StateDB` and call it inside `applyCall` after `sim.state.Prepare(...)`:

```go
// In StateDB:
func (s *StateDB) ResetRefund() {
    s.refund = 0
}

// In applyCall, after Prepare:
sim.state.Prepare(rules, msg.From, evm.Context.Coinbase, msg.To, activePrecompiles, msg.AccessList)
sim.state.ResetRefund() // clear stale refund from previous calls
```

---

### Proof of Concept

1. Deploy a contract with a storage slot set to a non-zero value (e.g., slot 0 = 1).
2. Submit an `eth_simulateV1` request with two calls in one block:
   - **Call 1**: Call the contract to clear slot 0 (SSTORE 0 → 0). This generates a 4,800-gas refund in `sim.state.refund`.
   - **Call 2**: A simple ETH transfer (intrinsic gas = 21,000, no refund of its own).
3. Observe that the simulation reports `gasUsed` for Call 2 as `21000 − min(21000/5, 4800) = 21000 − 4200 = 16800` instead of the correct `21000`.
4. Submit a real transaction equivalent to Call 2 with `gasLimit = 16800` — it will fail with out-of-gas, proving the simulation result was incorrect. [10](#0-9) [4](#0-3)

func (s *StateDB) Prepare(
	rules params.Rules,
	sender,
	coinbase common.Address,
	dst *common.Address,
	precompiles []common.Address,
	list ethtypes.AccessList,
) {
	if rules.IsBerlin {
		// Clear out any leftover from previous executions
		al := newAccessList()
		s.accessList = al

		al.AddAddress(sender)
		if dst != nil {
			al.AddAddress(*dst)
			// If it's a create-tx, the destination will be added inside evm.create
		}
		for _, addr := range precompiles {
			al.AddAddress(addr)
		}
		for _, el := range list {
			al.AddAddress(el.Address)
			for _, key := range el.StorageKeys {
				al.AddSlot(el.Address, key)
			}
		}
		if rules.IsShanghai { // EIP-3651: warm coinbase
			al.AddAddress(coinbase)
		}
	}
	// Reset transient storage at the beginning of transaction execution
	s.transientStorage = newTransientStorage()
}

func GasToRefund(availableRefund, gasConsumed, refundQuotient uint64) uint64 {
	// Apply refund counter
	refund := gasConsumed / refundQuotient
	if refund > availableRefund {
		return availableRefund
	}
	return refund
}
