### Title
`MerkleBlob::batch_insert` Silently Accepts Duplicate Keys, Corrupting the DataLayer Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key guard that `MerkleBlob::insert` enforces. When a batch containing a repeated `KeyId` is supplied (and the tree already holds ≥ 2 leaves), both leaf nodes are written into the blob without error. The in-memory cache silently overwrites the first entry with the second, leaving a "ghost" leaf in the blob that is unreachable through the cache but still contributes to every ancestor hash up to the root. The resulting root hash is permanently wrong, and `check_integrity()` will subsequently fail with `IntegrityKeyToIndexCacheIndex`.

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (lines 570–657) uses `self.insert()` only for the first two items when the tree has ≤ 1 existing leaf (lines 578–585). For every subsequent item it calls `insert_entry_to_blob` directly, with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {   // line 587
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // no key check
    indexes.push(new_leaf_index);
}
```

`BlockStatusCache::add_leaf` (lines 188–192) uses `HashMap::insert`, which silently overwrites the previous index for the same key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

After the batch completes, the blob contains two leaf nodes sharing the same `KeyId`. The cache tracks only the last one. The subtree built over all collected indexes (lines 607–648) incorporates both leaves, so every internal-node hash up to the root is computed over a tree that contains a phantom leaf. The committed root hash is therefore incorrect.

The Python binding `py_batch_insert` (lines 1503–1519) passes the caller-supplied list directly to `batch_insert` with no additional deduplication:

```rust
self.batch_insert(zip(keys_values, hashes).collect())?;
```

### Impact Explanation

The corrupted root hash is the authoritative commitment that the chia-blockchain stores on-chain for a DataLayer store. Once committed:

- `check_integrity()` raises `IntegrityKeyToIndexCacheIndex` because the blob contains a leaf whose key maps to a different index than the cache records.
- `get_proof_of_inclusion` for the duplicate key generates a proof anchored to the wrong root, so the proof fails external verification.
- The ghost leaf is permanently embedded in the blob; subsequent `calculate_lazy_hashes()` propagates the wrong hash upward, making the corruption persistent across serialization/deserialization cycles.
- Any external verifier comparing the on-chain root against an independently computed root will see a mismatch, effectively making the entire DataLayer store's state unverifiable — matching the "corrupts tree roots" criterion.

### Likelihood Explanation

`batch_insert` is the primary bulk-load path and is called directly from the Python binding exposed to all DataLayer store operators. A caller that accidentally or intentionally supplies a list with a repeated `KeyId` (e.g., due to a retry, a merge of two lists that share a key, or a crafted API call) will silently corrupt the tree. No privilege beyond the ability to call `MerkleBlob.batch_insert` is required. The `insert` path correctly rejects duplicates, so the discrepancy between the two paths is a latent trap for any caller that switches from sequential inserts to batch inserts.

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path in `batch_insert` before writing each leaf to the blob:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob call
}
```

Alternatively, pre-deduplicate the input vector (keeping the last entry per key, consistent with `upsert` semantics) and document the chosen policy. Add a regression test that passes a batch with a repeated key to `batch_insert` and asserts `KeyAlreadyPresent` is returned.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so batch_insert skips the guarded path
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i),
            &Hash(Bytes32::new([i as u8; 32])),
            InsertLocation::Auto {}).unwrap();
    }

    // Batch with a duplicate key (KeyId(99) appears twice)
    let batch = vec![
        ((KeyId(99), ValueId(1)), Hash(Bytes32::new([99u8; 32]))),
        ((KeyId(99), ValueId(2)), Hash(Bytes32::new([100u8; 32]))), // duplicate key
        ((KeyId(100), ValueId(3)), Hash(Bytes32::new([101u8; 32]))),
    ];

    // Succeeds — no error raised despite duplicate key
    blob.batch_insert(batch).unwrap();
    blob.calculate_lazy_hashes().unwrap();

    // Tree is now corrupt: check_integrity() will fail
    blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheIndex
}
```
