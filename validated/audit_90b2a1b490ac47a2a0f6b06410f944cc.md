### Title
Wrong Block Timestamp Propagated into Block Hash Commitment via Silent `get_proposal_timestamp` Fallback in Echonet Mode — (`crates/apollo_consensus_orchestrator/src/build_proposal.rs`)

### Summary

In Echonet (replay) mode the sequencer is supposed to stamp each block with the **original** mainnet block timestamp fetched from the mempool's FIFO queue. `get_proposal_timestamp()` silently falls back to `clock.unix_now()` (current wall-clock time) when `batcher.get_timestamp()` returns an error, and silently accepts `0` when the FIFO queue has never held a transaction. Either wrong value is embedded in `ProposalInit.timestamp`, flows through `BlockInfo.block_timestamp` → `PartialBlockHashComponents.timestamp` → `PartialBlockHash` (the `ProposalCommitment`), and is committed to storage and state-sync as the authoritative block timestamp. The block hash commitment is therefore wrong, and every `get_block_timestamp()` syscall executed inside that block returns the wrong value.

### Finding Description

**Root cause — `get_proposal_timestamp` accepts any `Ok` value without range validation** [1](#0-0) 

```rust
async fn get_proposal_timestamp(override_timestamp: bool, batcher: &dyn BatcherClient, clock: &dyn Clock) -> u64 {
    if override_timestamp {
        match batcher.get_timestamp().await {
            Ok(timestamp) => return timestamp,          // ← accepts 0 or any stale value
            Err(err) => {
                warn!("Failed to get timestamp from batcher, falling back to clock time: {err:?}");
            }
        }
    }
    clock.unix_now()   // ← wrong for Echonet: returns current wall-clock, not original block ts
}
```

`override_timestamp` is set to `true` exactly when `BehaviorMode::Echonet` is active: [2](#0-1) 

**Path 1 — empty FIFO queue returns `Ok(0)`**

`batcher.get_timestamp()` delegates to `mempool.get_timestamp()` → `FifoTransactionQueue::resolve_timestamp()`: [3](#0-2) 

When the queue has never held a transaction `last_returned_timestamp` is `None`, so `resolve_timestamp()` returns `0`. This is returned as `Ok(0)` all the way up to `get_proposal_timestamp`, which treats it as a valid timestamp and returns `0`. The test explicitly documents this: [4](#0-3) 

**Path 2 — batcher error falls back to wall-clock time**

If `batcher.get_timestamp()` returns `Err`, the function falls back to `clock.unix_now()`. In Echonet mode this is semantically wrong: the current wall-clock time is not the original block timestamp that Echonet is designed to replay.

**Propagation of the wrong timestamp into the block hash commitment**

The wrong timestamp from `get_proposal_timestamp` is placed in `ProposalInit.timestamp`: [5](#0-4) 

`convert_to_sn_api_block_info(&init)` converts it to `BlockInfo.block_timestamp`. `BlockExecutionArtifacts::new()` then calls: [6](#0-5) 

`PartialBlockHashComponents::new` stores the wrong timestamp: [7](#0-6) 

`calculate_block_hash` chains it directly into the Poseidon hash: [8](#0-7) 

The resulting `PartialBlockHash` is the `ProposalCommitment` that consensus agrees on. The same wrong timestamp is written to state-sync: [9](#0-8) 

And the `get_block_timestamp()` syscall inside the blockifier returns `block_info.block_timestamp`, so every contract executed in this block receives the wrong value.

### Impact Explanation

**Critical — Wrong state/receipt/execution result from blockifier/syscall logic for accepted input.**

- `get_block_timestamp()` syscall returns `0` (or current wall-clock time) instead of the correct original block timestamp. Any contract that uses the block timestamp for time-locked logic, expiry checks, or oracle validation will behave incorrectly.
- The `PartialBlockHash` / `ProposalCommitment` is computed with the wrong timestamp, so the block hash commitment stored in storage and propagated to state-sync is wrong.
- The `StorageBlockHeader.timestamp` written to the MDBX database is wrong, corrupting the authoritative header record for that block.

### Likelihood Explanation

**Path 1 (zero timestamp):** Occurs at sequencer startup in Echonet mode before any transaction has been fetched from the recorder and inserted into the FIFO queue. The first `build_proposal` call at height 0 will receive `Ok(0)` from the batcher. For height 0 the validator timestamp window check (`timestamp >= last_block_timestamp`) passes because `last_block_timestamp = 0`, so the proposal is not rejected.

**Path 2 (wall-clock fallback):** Occurs whenever the batcher-to-mempool RPC call fails transiently (network hiccup, mempool restart). The fallback is silent (only a `warn!` log) and produces a valid-looking Unix timestamp that passes all validator range checks, so the wrong block is accepted by the network.

### Recommendation

1. In `get_proposal_timestamp`, validate that the returned timestamp is non-zero and within a reasonable range before accepting it:
   ```rust
   Ok(timestamp) if timestamp > 0 => return timestamp,
   Ok(zero_ts) => {
       warn!("Batcher returned zero timestamp in Echonet mode; aborting proposal");
       // propagate as an error so the proposal is not built
   }
   ```
2. When `batcher.get_timestamp()` fails in Echonet mode, **abort the proposal** rather than falling back to `clock.unix_now()`. The fallback defeats the invariant that Echonet blocks carry the original mainnet timestamp.
3. Add a `BuildProposalError::InvalidTimestamp` variant so the orchestrator can surface this as a proposal failure rather than silently committing a wrong block.

### Proof of Concept

1. Deploy the sequencer in `BehaviorMode::Echonet`.
2. Start the sequencer without pre-populating the FIFO queue (no transactions in the recorder yet, or recorder unreachable).
3. Consensus triggers `build_proposal` for height 0.
4. `get_proposal_timestamp(override_timestamp=true, ...)` calls `batcher.get_timestamp()`.
5. `Batcher::get_timestamp()` → `mempool.get_timestamp()` → `FifoTransactionQueue::resolve_timestamp()` returns `0` (queue empty, `last_returned_timestamp = None`).
6. `get_proposal_timestamp` receives `Ok(0)` and returns `0`.
7. `ProposalInit { timestamp: 0, ... }` is broadcast; `convert_to_sn_api_block_info` produces `BlockInfo { block_timestamp: BlockTimestamp(0), ... }`.
8. `BlockExecutionArtifacts::new` calls `PartialBlockHashComponents::new(&block_info, ...)` → `timestamp: BlockTimestamp(0)`.
9. `PartialBlockHash::from_partial_block_hash_components(...)` → `ProposalCommitment` computed with timestamp `0`.
10. Consensus reaches decision; `update_state_sync_with_new_block` stores `block_header_without_hash.timestamp = BlockTimestamp(0)`.
11. Any contract calling `get_block_timestamp()` in this block receives `0`; the block hash in storage is wrong.

```rust
    fn resolve_timestamp(&mut self) -> UnixTimestamp {
        // If queue is non-empty, use front tx timestamp and persist it as current threshold.
        if let Some(timestamp) = self.queue.front().map(|tx| tx.timestamp) {
            self.last_returned_timestamp = Some(timestamp);
            return timestamp;
        }
        // If queue is empty, reuse last returned threshold.
        // If no threshold was ever returned, default to 0.
        self.last_returned_timestamp.unwrap_or(0)
    }
```
