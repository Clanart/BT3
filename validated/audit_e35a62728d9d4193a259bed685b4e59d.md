### Title
Missing Zero-Value Validation for `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in `syscall_impls.cairo` (and its deprecated counterpart in `deprecated_execute_syscalls.cairo`) accepts a `class_hash` of zero without any validation. Setting a contract's class hash to zero is equivalent to marking it as `UNINITIALIZED_CLASS_HASH`, permanently making the contract's entry points unreachable and freezing any funds it holds. The codebase itself acknowledges the missing check via a TODO comment at the exact location of the vulnerability.

---

### Finding Description

In `execute_replace_class` within `syscall_impls.cairo`, the new class hash is read directly from the syscall request and written into the contract state without any zero-value guard:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges that the OS does not verify the provided hash corresponds to any declared class. The value `0` is the `UNINITIALIZED_CLASS_HASH` sentinel used by `deploy_contract.cairo` to assert a contract slot is empty before deployment:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
``` [2](#0-1) 

The same missing validation exists in the deprecated syscall path:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    // No zero-check.
    ...
    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
``` [3](#0-2) 

After `replace_class(0)` is accepted, any subsequent call to the contract enters `execute_entry_point`, which performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // key = 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,           // searches for hash of 0
);
``` [4](#0-3) 

`find_element` is a hint-based Cairo primitive that panics if the key is absent. Since no compiled class fact with hash `0` will exist in the bundle, every subsequent call to the contract causes an OS-level panic, making the contract permanently unreachable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class(0)` — whether due to a bug in its upgrade logic or because an attacker supplies `0` as the class hash argument to a publicly callable upgrade function — will have its class hash permanently set to the uninitialized sentinel. All entry points become unreachable. Any ETH, ERC-20 tokens, or other assets held by the contract are permanently frozen with no recovery path, because the OS provides no mechanism to restore a class hash once it has been overwritten.

---

### Likelihood Explanation

Upgradeable contracts are a common pattern on StarkNet. A contract that exposes an `upgrade(new_class_hash: felt)` function and passes the argument directly to the `replace_class` syscall — without its own zero-check — is vulnerable. An attacker who can call that function (e.g., if access control is absent or bypassed) can supply `class_hash = 0` and permanently brick the contract. The OS is the last line of defense and provides none. The TODO comment at line 898 confirms the development team is aware the check is missing.

---

### Recommendation

Add an explicit zero-value assertion immediately after reading `class_hash` in both syscall handlers:

**`syscall_impls.cairo` — `execute_replace_class`:**
```cairo
let class_hash = request.class_hash;
assert_not_zero(class_hash);  // Prevent setting UNINITIALIZED_CLASS_HASH
```

**`deprecated_execute_syscalls.cairo` — `execute_replace_class`:**
```cairo
let class_hash = syscall_ptr.class_hash;
assert_not_zero(class_hash);  // Prevent setting UNINITIALIZED_CLASS_HASH
```

Ideally, also resolve the TODO by verifying that `class_hash` corresponds to a previously declared class in `contract_class_changes`.

---

### Proof of Concept

1. Deploy an upgradeable contract with the following logic:
   ```cairo
   @external
   func upgrade(new_class_hash: felt) {
       replace_class(new_class_hash);  // no local zero-check
       return ();
   }
   ```
2. Fund the contract with tokens (e.g., transfer ERC-20 tokens to it).
3. Call `upgrade(0)` from any address permitted to call the function.
4. The OS `execute_replace_class` handler accepts `class_hash = 0` without error and writes `StateEntry(class_hash=0, ...)` into `contract_state_changes`.
5. Any subsequent call to the contract (including a withdrawal) reaches `execute_entry_point`, which calls `find_element(..., key=0)` on the compiled class facts bundle.
6. `find_element` panics because no compiled class with hash `0` exists.
7. The contract is permanently bricked; all funds are frozen with no recovery path.
