### Title
Irrecoverable Loss of Fast-Transfer Relayer Funds Due to Detached `send_tokens` with No Failure Callback in `process_fin_transfer_to_other_chain` - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain`, when a fast transfer was previously performed and the original `fin_transfer` is being finalized, the bridge calls `send_tokens(...).detach()` to repay the fast-transfer relayer, then immediately calls `mark_fast_transfer_as_finalised()`. Because the promise is detached, there is no failure callback. If `send_tokens` fails, the fast-transfer relayer permanently loses the tokens they fronted, with no retry or recovery path possible.

---

### Finding Description

When `fin_transfer_callback` processes a proof for a transfer whose recipient is on a non-NEAR chain, it calls `process_fin_transfer_to_other_chain`. Inside that function, if a fast transfer was previously recorded for this transfer, the code repays the relayer:

```rust
// near/omni-bridge/src/lib.rs lines 2028-2040
if let Some(relayer) = recipient {
    self.send_tokens(
        token,
        relayer,
        U128(transfer_message.amount_without_fee()...),
        "",
    )
    .detach();                                          // ← no callback
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // ← state committed
}
``` [1](#0-0) 

Before this block executes, `add_fin_transfer` has already inserted the transfer ID into `finalised_transfers`: [2](#0-1) 

And `add_fin_transfer` panics with `TransferAlreadyFinalised` if the same ID is submitted again: [3](#0-2) 

`mark_fast_transfer_as_finalised` sets `finalised = true` on the fast transfer record: [4](#0-3) 

Any subsequent call to `process_fin_transfer_to_other_chain` for the same transfer panics at `add_fin_transfer`. Any subsequent call to `process_fin_transfer_to_near` for the same fast transfer panics at `FastTransferAlreadyFinalised`. There is no mechanism to retry the relayer repayment.

The same pattern exists in `utxo_fin_transfer_fast`, where `send_tokens(...).detach()` is called after `remove_fast_transfer` or `mark_fast_transfer_as_finalised`, again with no failure callback: [5](#0-4) 

Contrast this with `process_fin_transfer_to_near`, which correctly chains a `fin_transfer_send_tokens_callback` that reverts lock actions and removes the fin transfer record on failure: [6](#0-5) 

And `fin_transfer_send_tokens_callback` properly handles the failure case: [7](#0-6) 

---

### Impact Explanation

The fast-transfer relayer fronted their own tokens to the user on NEAR. When `fin_transfer` is called to finalize the original cross-chain transfer, the bridge is supposed to repay the relayer from the unlocked/minted tokens. If `send_tokens` fails (e.g., the token contract is paused, the relayer's storage registration is missing, or the `mint` call on a deployed token reverts), the relayer's repayment is permanently lost:

- The transfer ID is in `finalised_transfers` — `fin_transfer` cannot be retried.
- The fast transfer is marked `finalised = true` — the fast transfer record cannot be used again.
- The relayer's fronted tokens are irrecoverably locked in the bridge contract with no withdrawal path.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

`send_tokens` for a deployed (bridged) token calls `ext_token::ext(token).mint(...)`: [8](#0-7) 

The bridge token's `mint` function is guarded by `whenNotPaused`: [9](#0-8) 

If the token is paused at the moment `fin_transfer` is processed, `send_tokens` fails silently (detached promise). For non-deployed tokens, `ft_transfer` requires the recipient to have storage registered; if the relayer's storage was removed or the token contract is paused, the transfer fails. These are realistic operational conditions, not purely theoretical.

---

### Recommendation

Replace the detached `send_tokens` call with a chained callback that handles failure, analogous to how `process_fin_transfer_to_near` handles it. The callback should:

1. On success: keep the fast transfer as finalised.
2. On failure: un-mark the fast transfer as finalised (or store a claimable balance for the relayer) so the repayment can be retried.

Alternatively, store the owed amount in a claimable mapping before calling `send_tokens`, so the relayer can pull their funds even if the push fails.

---

### Proof of Concept

1. Relayer R performs a fast transfer for transfer ID `T` (origin: Ethereum, recipient: Solana) by calling `ft_transfer_call` with a `FastFinTransferMsg`. Tokens are fronted to the user on NEAR. Fast transfer record is stored with `finalised = false`.

2. A trusted relayer calls `fin_transfer` with a valid proof for transfer `T`. `fin_transfer_callback` routes to `process_fin_transfer_to_other_chain` because the recipient is on Solana.

3. Inside `process_fin_transfer_to_other_chain`:
   - `add_fin_transfer` inserts `T` into `finalised_transfers`.
   - `unlock_tokens_if_needed` adjusts locked token accounting.
   - Fast transfer status is found for `T`; `recipient = Some(R)`.
   - `send_tokens(token, R, amount, "").detach()` is called — the token contract is currently paused, so this promise will fail silently.
   - `mark_fast_transfer_as_finalised(&fast_transfer.id())` sets `finalised = true`.

4. The `send_tokens` promise fails. No callback fires. R never receives their tokens.

5. Any retry of `fin_transfer` for `T` panics with `TransferAlreadyFinalised`. Any attempt to use the fast transfer record panics with `FastTransferAlreadyFinalised`. R's fronted tokens are permanently lost. [10](#0-9)

    fn process_fin_transfer_to_other_chain(
        &mut self,
        predecessor_account_id: AccountId,
        transfer_message: TransferMessage,
    ) {
        let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
        let token = self.get_token_id(&transfer_message.token);

        if transfer_message.recipient.is_utxo_chain() {
            let btc_account_id =
                self.get_utxo_chain_token(transfer_message.get_destination_chain());
            require!(
                token == btc_account_id,
                BridgeError::NativeTokenRequiredForChain.as_ref()
            );
        }

        self.unlock_tokens_if_needed(
            transfer_message.get_origin_chain(),
            &token,
            transfer_message.amount.0,
        );
        self.lock_tokens_if_needed(
            transfer_message.get_destination_chain(),
            &token,
            transfer_message.fee.fee.into(),
        );

        let fast_transfer = FastTransfer::from_transfer(transfer_message.clone(), token.clone());
        let recipient = if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
            require!(
                !status.finalised,
                BridgeError::FastTransferAlreadyFinalised.as_ref()
            );
            Some(status.relayer)
        } else {
            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token,
                transfer_message
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            );

            None
        };

        // If fast transfer happened, send tokens to the relayer that executed fast transfer
        if let Some(relayer) = recipient {
            self.send_tokens(
                token,
                relayer,
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
                "",
            )
            .detach();
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
        } else {
            required_balance = self
                .add_transfer_message(transfer_message.clone(), predecessor_account_id.clone())
                .saturating_add(required_balance);
        }

        self.update_storage_balance(
            predecessor_account_id,
            required_balance,
            env::attached_deposit(),
        );

        env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
    }
