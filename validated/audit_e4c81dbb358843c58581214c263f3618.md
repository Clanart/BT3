### Title
Stale `BlockStatusCache` After Leaf-to-Root Move in `delete()` Corrupts DataLayer Merkle Proofs - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
When `MerkleBlob::delete()` reduces a two-leaf tree to a single leaf, the surviving sibling is physically relocated to `TreeIndex(0)`. The `move_index` cache helper that records this relocation only updates `free_indexes`; it never updates `key_to_index` or `leaf_hash_to_index`. The stale cache entries cause every subsequent key-lookup, value-read, and proof-of-inclusion for the surviving leaf to operate on the wrong (now-free) block slot. A subsequent insert that reuses that free slot makes two distinct keys share one cache entry, producing silently wrong key-value associations and forged inclusion proofs.

### Finding Description

`MerkleBlob::delete()` has three cases. When the deleted leaf's parent is the root (no grandparent), the sibling is moved to `TreeIndex(0)`:

```

    sibling_block.node.set_parent(Parent(None));

        for child_index in [node.left, node.right] {

    self.block_status_cache
        .move_index(sibling_index, destination)?;   // ← only updates free_indexes
    return Ok(());

```

`move_index` is:

```
// lines 210-224
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    if self.free_indexes.contains(&source) {
        return Err(Error::MoveSourceIndexNotInUse(source));
    }
    if self.free_indexes.contains(&destination) {
        return Err(Error::MoveDestinationIndexNotInUse(destination));
    }
    self.free_indexes.insert(source);   // marks old slot as free
    Ok(())                              // key_to_index and leaf_hash_to_index NOT updated
}
```

After the call:
- `key_to_index[sibling_key]` still equals `sibling_index` (the old, now-free slot).
- `leaf_hash_to_index[sibling_hash]` still equals `sibling_index`.
- `sibling_index` is in `free_indexes` and eligible for reuse.

`get_leaf_by_key` and `get_proof_of_inclusion` both resolve a key through `key_to_index`:

`get_keys_values` iterates `key_to_index` and reads each index from the blob, so it too returns wrong values once `sibling_index` is overwritten.

The `Drop`-based integrity check that would catch this is compiled only under `#[cfg(any(test, debug_assertions))]`, and `check_integrity_on_drop` defaults to `cfg!(test)` in `MerkleBlob::new`, so it is **silent in production release builds**.

### Impact Explanation

**Concrete corruption sequence (two-leaf tree):**

1. Insert leaf A and leaf B → tree has root (internal) at index 0, A at index 1, B at index 2.
2. `delete(A)` → B is moved to index 0; `key_to_index[B] = 2` (stale); index 2 added to `free_indexes`.
3. `insert(C, ...)` → `get_new_index()` returns index 2 (free); C is written there; `key_to_index[C] = 2`.
4. Now `key_to_index[B] = 2 = key_to_index[C]`: two keys share one slot.
5. `get_keys_values()` returns B's entry with C's value.
6. `get_proof_of_inclusion(B)` generates a proof for C's data under B's key — a structurally valid-looking but semantically forged inclusion proof.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The trigger is a two-leaf `MerkleBlob` followed by a deletion — a routine DataLayer operation. Any DataLayer store that shrinks to two entries and then deletes one will silently corrupt its cache. The Python binding `MerkleBlob.delete()` is the direct entry point; no privilege is required.

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf. The simplest fix is to add a leaf-aware variant or extend `move_index` to accept the optional leaf node:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex,
              leaf: Option<&LeafNode>) -> Result<(), Error> {
    // existing free_indexes checks ...
    self.free_indexes.insert(source);
    if let Some(leaf) = leaf {
        self.key_to_index.insert(leaf.key, destination);
        self.leaf_hash_to_index.insert(leaf.hash, destination);
    }
    Ok(())
}
```

Call site in `delete()` should pass `Some(&sibling_leaf)` when the sibling is a leaf node.

### Proof of Concept

```rust
// Reproduces the stale-cache corruption in a 2-leaf tree
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.check_integrity_on_drop = false; // disable so we can observe the corruption

let hash_a = sha256_num(&1i64);
let hash_b = sha256_num(&2i64);
let hash_c = sha256_num(&3i64);

blob.insert(KeyId(1), ValueId(10), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(20), &hash_b, InsertLocation::Auto {}).unwrap();

// Delete key 1 → key 2 (sibling) is moved to TreeIndex(0)
// but key_to_index[KeyId(2)] still points to the old free slot
blob.delete(KeyId(1)).unwrap();

// Insert key 3 → reuses the free slot that key 2's cache entry still points to
blob.insert(KeyId(3), ValueId(30), &hash_c, InsertLocation::Auto {}).unwrap();

// key_to_index[KeyId(2)] == key_to_index[KeyId(3)] → same slot
// get_keys_values() returns KeyId(2) → ValueId(30) (wrong, should be 20)
let kv = blob.get_keys_values().unwrap();
assert_eq!(kv[&KeyId(2)], ValueId(20)); // FAILS: returns ValueId(30)

// get_proof_of_inclusion(KeyId(2)) generates proof for KeyId(3)'s data
// check_integrity() would also fail here
blob.check_integrity().unwrap(); // FAILS: IntegrityKeyToIndexCacheIndex
```
