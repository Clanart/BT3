### The Missing Guard

The `chunk_headers_root` Merkle tree is built from all shards in shard-index order:

```rust
pub fn compute_chunk_headers_root(&self) -> (CryptoHash, Vec<MerklePath>) {
    merklize(&self.iter()
        .map(|chunk| ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))
        .collect_vec())
}
``` [3](#0-2) 

A `verify_path` call with shard 0's chunk header and shard 0's Merkle proof (position 0) will succeed against the block's `chunk_headers_root` — the proof is valid for position 0. The code never asserts that the position corresponds to the target `shard_id`.

### Attack Construction

Suppose the syncing node is syncing **shard 1** (H1 = last height shard 1 was included), and shard 0 was last included at H0 < H1.

The attacker provides:
- `chunk` = real shard 1 chunk with valid Merkle proof for position 1 ✓
- `prev_chunk_header` = real shard 0 chunk header with valid Merkle proof for position 0 ✓ (no shard_id cross-check)
- `incoming_receipts_proofs` = valid receipt chain from `sync_hash` back to **H0** (not H1)

Step 4g passes because the chain ends at `prev_chunk_header.height_included()` = H0. All receipt Merkle proofs verify against real on-chain `prev_chunk_outgoing_receipts_root` values — the attacker uses real chain data.

### Effect in `set_state_finalize`

```rust
for incoming_receipt_proof in &incoming_receipts_proofs {
    let ReceiptProofResponse(hash, _) = incoming_receipt_proof;
    let block_header = self.chain_store_update.get_block_header(hash)?;
    if block_header.height() <= chunk.height_included() {  // <= H1
        receipt_proof_responses.push(incoming_receipt_proof.clone());

``` [4](#0-3) 

Because the receipt chain now extends from `sync_hash` back to H0 (instead of H1), blocks in the range [H0, H1) have heights ≤ H1 and pass the filter. These extra receipts are applied to shard 1's state — receipts that should not be applied at this point. The resulting state root diverges from the canonical value, and the node will fail to validate subsequent chunks.

### Constraint Check

- The handler is `StateHeaderValidationRequest` → `set_state_header`, reachable from the network.
- All Merkle proofs use real on-chain data; no forgery is required.

### Title
Missing `prev_chunk_header.shard_id()` cross-check in `set_state_header` allows wrong receipt height range during state finalization — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` verifies that `prev_chunk_header` is included in the block's Merkle tree but never checks that its `shard_id` matches the main chunk's shard. An unprivileged peer can supply a `prev_chunk_header` from a different shard with a valid Merkle proof, causing the receipt chain validation to accept a wrong lower-bound height, and `set_state_finalize` to apply extra (or missing) receipts, producing a wrong state root.

### Finding Description
In `set_state_header` (step 3b), `verify_path` confirms that `prev_chunk_header` is a leaf in `prev_block_header.chunk_headers_root`, but the leaf value is `ChunkHashHeight(hash, height_included)` — it does not encode `shard_id`. The code never asserts `prev_chunk_header.shard_id() == chunk.shard_id()`. Step 4g then anchors the receipt chain at `prev_chunk_header.height_included()`, which is the wrong shard's height. In `set_state_finalize`, the filter `block_header.height() <= chunk.height_included()` admits all receipts from the extended range, corrupting the applied receipt set.

### Impact Explanation
The syncing node installs a state root that differs from the canonical value. It subsequently fails to validate any chunk that references this state root, rendering it unable to participate in consensus or track the shard correctly. The corruption is silent — no error is returned from `set_state_finalize`.

### Likelihood Explanation
Any peer serving state sync responses can trigger this. State sync is used by every node joining the network or catching up after downtime. The attacker needs only to serve a crafted `ShardStateSyncResponseHeader` with real on-chain Merkle proofs — no key material or validator privileges are required.

### Recommendation
Add an explicit shard ID check after step 3b:
```rust
if let Some(prev_chunk_header) = &prev_chunk_header {
    if prev_chunk_header.shard_id() != chunk.shard_id() {
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk_header shard_id does not match chunk shard_id".into()
        ));
    }
}
```
Similarly, verify `chunk.shard_id() == shard_id` (the parameter) after step 1-2.

### Proof of Concept
1. Node A is syncing shard 1. Shard 1's prev chunk was last included at H1=100; shard 0's at H0=90.
2. Attacker constructs `ShardStateSyncResponseHeader::V2` with:
   - `chunk` = real shard 1 chunk (valid `chunk_proof` for position 1)
   - `prev_chunk_header` = real shard 0 chunk header (valid `prev_chunk_proof` for position 0)
   - `incoming_receipts_proofs` = valid receipt chain from `sync_hash` back to height 90
3. Call `set_state_header(shard_id=1, sync_hash, crafted_header)` — all checks pass.
4. Call `set_state_finalize(shard_id=1, sync_hash)` — receipts from blocks 90–100 are included.
5. Assert the resulting `ChunkExtra` state root differs from the canonical value.
