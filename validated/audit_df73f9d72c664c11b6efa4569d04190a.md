### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary `class_hash` from the caller without verifying that the hash corresponds to a previously declared contract class. This is the direct analog of the external report's pattern: a critical role-bearing value (the new class hash) is accepted without validating it is a valid, usable entity. A contract can replace its own class with an undeclared hash, permanently bricking itself and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` with no validation:

```cairo
// syscall_impls.cairo lines 896-913
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-acknowledged TODO at line 898 confirms the missing check. The `class_hash` field is written directly into the state without any lookup against `contract_class_changes` or the `compiled_class_facts_bundle`.

When any subsequent transaction attempts to call this contract, `execute_entry_point` performs:

```cairo
// execute_entry_point.cairo lines 154-166
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

`dict_read` on an undeclared class hash returns `0` (the default). `find_element` with key `0` will then fail to locate a matching `CompiledClassFact`, causing an OS-level panic. The sequencer cannot include any future call to this contract in a provable block. The contract — and all funds it holds — are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash (including `0` or any arbitrary felt) will have its `StateEntry.class_hash` permanently set to an invalid value. All future calls to that contract will cause the OS prover to abort. The contract's storage and token balances become permanently inaccessible. There is no recovery path: the state change is committed on-chain and no subsequent transaction can undo it (since calling the contract itself is what fails).

---

### Likelihood Explanation

**Medium-High.** The `replace_class` syscall is a standard, documented StarkNet syscall callable by any deployed contract. An attacker can:
- Deploy a contract that deliberately calls `replace_class(0)` or `replace_class(arbitrary_undeclared_hash)` to self-brick after receiving funds.
- A buggy contract could accidentally pass an incorrect hash.
- The TODO comment confirms the development team is aware the check is absent.

No privileged access, leaked keys, or external dependencies are required. Any unprivileged contract deployer can trigger this.

---

### Recommendation

Before writing the new `class_hash` into the `StateEntry`, verify that it exists in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Specifically, perform a `dict_read` on `contract_class_changes` for the requested `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the validation already performed implicitly in `execute_entry_point` but must be enforced eagerly at the point of `replace_class` to prevent invalid state from being committed.

---

### Proof of Concept

1. Attacker deploys `VictimVault` — a contract that holds ETH/STRK and exposes a `drain()` function.
2. Users deposit funds into `VictimVault`.
3. Attacker calls a function on `VictimVault` that internally invokes `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. The OS executes `execute_replace_class`: no validation occurs; `VictimVault`'s `StateEntry.class_hash` is set to `0xdeadbeef`. The transaction succeeds and is committed.
5. Any future transaction calling `VictimVault` reaches `execute_entry_point`, which calls `dict_read` on `contract_class_changes` for key `0xdeadbeef`, receives `0`, then calls `find_element(..., key=0)` — which panics because no compiled class with hash `0` exists.
6. The sequencer cannot include any call to `VictimVault` in a provable block. All deposited funds are permanently frozen.
