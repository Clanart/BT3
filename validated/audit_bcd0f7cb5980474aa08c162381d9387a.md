### Title
Silent Failure in Proof Manager Storage Allows Sequencing of Transactions Without Stored Proofs - (File: crates/apollo_gateway/src/gateway.rs)

---

### Summary

In `GenericGateway::store_proof_and_spawn_archiving`, when `store_proof_in_proof_manager` returns an error, the error is only logged — it is never propagated. Execution continues, the archive-writer task is spawned, and `Some((handle, tx_hash))` is returned. The caller `add_tx_inner` then unconditionally adds the transaction to the mempool. The result is that a transaction carrying `ProofFacts` can be sequenced into a block while its proof is absent from the proof manager, breaking the invariant that every sequenced client-side-proving transaction has a retrievable proof.

---

### Finding Description

`store_proof_and_spawn_archiving` in `crates/apollo_gateway/src/gateway.rs` (lines 241–282) calls `store_proof_in_proof_manager` and pattern-matches the result:

```rust

    Ok(proof_manager_store_duration) => { /* record latency */ }
    Err(e) => {
        error!("Failed to set proof in proof manager: {}", e);
        // ← no return, no abort
    }
}
// execution falls through unconditionally
let handle = tokio::spawn(async move { ... });
Some((handle, tx_hash))          // ← returned even on storage failure
``` [1](#0-0) 

The caller `add_tx_inner` receives the handle and proceeds:

```rust
let proof_archive_handle =
    self.store_proof_and_spawn_archiving(proof_data, internal_tx.tx_hash).await;
// ...
let mempool_client_result = self.mempool_client.add_tx(add_tx_args).await;
// transaction is in the mempool regardless of proof storage outcome
``` [2](#0-1) 

The proof is verified earlier during `convert_rpc_tx_to_internal_and_executable_txs` (via `spawn_proof_verification` → `run_proof_verification`), but verification and storage are decoupled. A successful verification does not guarantee a successful store. [3](#0-2) 

The `ProofManager` stores proofs on the local filesystem keyed by `proof_facts.hash()`. If `set_proof` fails (I/O error, disk full, transient unavailability), the proof is absent from both the in-memory LRU cache and the persistent directory. [4](#0-3) 

---

### Impact Explanation

The broken invariant is: **every sequenced transaction carrying `ProofFacts` must have its proof retrievable from the proof manager**.

When the invariant is violated:

1. The transaction is sequenced into a block.
2. The transaction prover later calls `get_proof(proof_facts)` and receives `None` (proof not found).
3. Depending on how the prover handles a missing proof, it either:
   - **Fails to generate a block proof** — a DoS on the proving pipeline for that block, or
   - **Skips proof verification for the missing entry** — producing a wrong proof output / wrong SNOS value for the block.

Either outcome breaks the proof-facts commitment invariant and matches the allowed impact scope: **"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"** (wrong SNOS/proof output) and **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"** (gateway admits a transaction whose proof cannot be retrieved by the prover).

---

### Likelihood Explanation

The proof manager is a local filesystem component (`FsProofStorage`). Storage failures can occur due to:

- Transient I/O errors or filesystem pressure during normal operation.
- A proof manager process restart or upgrade window.
- Disk exhaustion (resource exhaustion reachable without privilege).

Any of these conditions, combined with a user submitting a transaction with `ProofFacts` at that moment, triggers the silent failure. No privileged access is required.

---

### Recommendation

Propagate the error from `store_proof_in_proof_manager` and abort the transaction if proof storage fails. The corrected logic should be:

```rust
self.transaction_converter
    .store_proof_in_proof_manager(proof_facts.clone(), proof.clone())
    .await
    .map_err(|e| {
        error!("Failed to set proof in proof manager: {}", e);
        GatewayError::ProofStorageFailure(e)  // propagate, do not swallow
    })?;
```

This mirrors the fix applied in the Moonwell Finance case: add return-value validation so that a storage failure causes the overall operation to abort rather than silently proceeding.

---

### Proof of Concept

**Step 1.** The proof manager (`FsProofStorage`) becomes temporarily unavailable (e.g., I/O error, disk full, or process restart).

**Step 2.** A user submits an `InvokeV3` transaction carrying non-empty `ProofFacts` and a valid `Proof` to the gateway.

**Step 3.** The gateway verifies the proof successfully via `run_proof_verification` → `verify_proof`. [5](#0-4) 

**Step 4.** `store_proof_and_spawn_archiving` calls `store_proof_in_proof_manager`, which returns `Err(...)`. The error is logged but not propagated. [6](#0-5) 

**Step 5.** The function returns `Some((handle, tx_hash))` as if storage succeeded. [7](#0-6) 

**Step 6.** `add_tx_inner` adds the transaction to the mempool. The transaction is sequenced into a block. [8](#0-7) 

**Step 7.** The transaction prover calls `proof_manager.get_proof(proof_facts)`. The proof is absent from both the LRU cache and the filesystem. `get_proof` returns `Ok(None)`. [9](#0-8) 

**Step 8.** The transaction prover either fails to generate a proof for the block (DoS on proving) or proceeds without verifying the proof (wrong SNOS/proof output), breaking the proof-facts commitment invariant for that block.

    /// Runs proof verification: checks if the proof already exists, and if not, verifies it.
    /// Returns `true` if verification was performed, `false` if skipped (proof already stored).
    /// This is the shared verification logic used by both gateway and consensus flows.
    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    pub async fn set_proof(
        &self,
        proof_facts: ProofFacts,
        proof: Proof,
    ) -> Result<(), FsProofStorageError> {
        if self.contains_proof(proof_facts.clone()).await? {
            return Ok(());
        }
        let facts_hash = proof_facts.hash();
        self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
        self.cache.insert(facts_hash, proof);
        Ok(())
    }

    pub async fn get_proof(
        &self,
        proof_facts: ProofFacts,
    ) -> Result<Option<Proof>, FsProofStorageError> {
        let facts_hash = proof_facts.hash();
        // Check cache first.
        if let Some(proof) = self.cache.get(&facts_hash) {
            return Ok(Some(proof));
        }
        // Fallback to filesystem.
        let proof = self.proof_storage.get_proof(facts_hash).await?;
        if let Some(proof) = &proof {
            self.cache.insert(facts_hash, proof.clone());
        }
        Ok(proof)
    }
