### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key/Hash Validation, Silently Corrupting the DataLayer Merkle Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When the tree already contains ≥ 2 leaves, every entry in the batch is written directly to the blob via `insert_entry_to_blob` without any uniqueness check. Supplying a batch that contains two entries sharing the same `KeyId` (or the same `Hash`) silently produces a tree blob with two leaf nodes for the same key, while the `block_status_cache` retains only the last-written index for that key. The resulting Merkle root is computed over the corrupted structure and diverges from the canonical single-key root, enabling forged inclusion proofs and committed-state corruption.

---

### Finding Description

**`insert()` enforces uniqueness; `batch_insert()` does not (for trees with ≥ 2 leaves)**

`MerkleBlob::insert` guards against both duplicate keys and duplicate leaf hashes before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` takes a completely different code path for trees that already hold ≥ 2 leaves. It skips the guard entirely and calls `insert_entry_to_blob` directly for every item in the batch:

```rust
// lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        ...
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs a plain `HashMap::insert` that **silently overwrites** the previous entry for a duplicate key:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index); // overwrites silently
}
```

**What happens with a duplicate key in the batch**

Both leaf nodes are written into the blob and wired into the tree structure by the subsequent pairing loop (lines 607-648). The `block_status_cache` ends up pointing only to the *last*-written leaf for that key. The root hash is computed over a tree that contains two leaves for the same key — a state that is structurally invalid and that `insert()` would have rejected.

The partial guard that exists (lines 578-585) only applies when `leaf_count() <= 1`, and even then it only protects the last two items popped from the vector (via `pop()`), leaving the remaining front items unguarded.

**`check_integrity` would detect the corruption — but is not called automatically**

`check_integrity` compares the leaf count from a full blob traversal against `key_to_index_cache_length`. With two blob leaves sharing one cache entry these counts diverge and the check fails. However, `check_integrity` is only called on drop when `check_integrity_on_drop` is `true` (a debug flag), and is never called automatically after `batch_insert`.

**Python binding exposes the path directly**

```rust
// lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,

    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

Any Python caller — including the chia-blockchain DataLayer node — can pass a list with repeated `KeyId` values and receive `Ok(())`.

---

### Impact Explanation

A caller who supplies a batch containing a repeated `KeyId` (or repeated `Hash`) causes `MerkleBlob` to commit a structurally invalid tree to its blob. The root hash derived from this tree differs from the canonical root that would result from a single-key tree. Downstream DataLayer operations that rely on `get_root_hash()`, `get_proof_of_inclusion()`, or delta synchronisation will operate against a corrupted root:

- **Forged inclusion proof**: `get_proof_of_inclusion(key)` generates a proof anchored to the last-written duplicate leaf. That proof is internally consistent with the corrupted root but proves a state that was never legitimately committed.
- **Corrupted tree root**: Any DataLayer singleton that commits this root on-chain records an incorrect state hash, permanently diverging from the intended data.
- **Cross-node disagreement**: Nodes that reconstruct the tree from the blob will compute the same corrupted root, while nodes that built the tree via sequential `insert()` calls will compute the correct root — causing consensus divergence over DataLayer state.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

The Python binding `batch_insert` is part of the public `MerkleBlob` API and is called by the chia-blockchain DataLayer implementation whenever a batch of key-value pairs is committed. Any code path that does not deduplicate its input before calling `batch_insert` — including paths driven by external data submitted to a DataLayer store — can trigger the corruption. No special privilege is required; the attacker only needs to influence the set of keys submitted in a single batch operation.

---

### Recommendation

Add the same uniqueness checks to `batch_insert` that `insert` already enforces. The simplest correct fix is to check each incoming key and hash against the cache before writing to the blob:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob call
}
```

Alternatively, collect all keys and hashes into a `HashSet` before the loop and reject the entire batch if any duplicate is found. The partial guard for the `leaf_count() <= 1` branch (which uses `pop()` and therefore processes items in reverse order) should also be reviewed for ordering correctness.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(bytearray())

# Pre-populate with 2 leaves so batch_insert takes the unguarded path
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(200), ValueId(200), h(200))

# Batch with a duplicate KeyId(300) — two different hashes, same key
blob.batch_insert(
    [(KeyId(300), ValueId(300)), (KeyId(400), ValueId(400)), (KeyId(300), ValueId(999))],
    [h(300), h(400), h(999)],
)
# Returns Ok(()) — no error raised

blob.calculate_lazy_hashes()
root = blob.get_root_hash()

# The tree now contains two leaf nodes for KeyId(300).
# check_integrity() will raise because leaf_count (5) != cache key count (4).
try:
    blob.check_integrity()
except Exception as e:
    print(f"Corruption confirmed: {e}")

# Proof of inclusion is generated for the last-written KeyId(300) leaf,
# but the root was computed over both — the proof is anchored to a corrupted root.
proof = blob.get_proof_of_inclusion(KeyId(300))
print(f"proof.valid() = {proof.valid()}")  # True, but root is wrong

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
