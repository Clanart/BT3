### Title
Unvalidated `l1_gas_price_wei` in `ProposalInit` Allows Malicious Proposer to Commit Wrong Gas Prices and Block Hash — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_block_info_valid` validates the FRI-denominated L1 gas prices from a received `ProposalInit` against the local oracle, but silently discards the oracle-computed WEI prices and never checks the proposer-supplied `l1_gas_price_wei` / `l1_data_gas_price_wei`. A consensus proposer can therefore send valid FRI prices (within the accepted margin) alongside arbitrary WEI prices. Those unvalidated WEI values flow directly into `convert_to_sn_api_block_info`, which (a) uses them verbatim as the ETH gas prices for execution and (b) derives `l2_gas_price_wei` from the ratio `l1_gas_price_fri / l1_gas_price_wei`. Both the raw WEI prices and the derived `l2_gas_price_wei` are then committed into the block hash via `gas_prices_to_hash`, producing a wrong, accepted block hash and wrong ETH-denominated fees.

### Finding Description

**Step 1 – Validation gap.**

`is_block_info_valid` calls `get_l1_prices_in_fri_and_wei` to obtain the locally-computed oracle prices, but immediately discards the WEI half:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;
```

Only the FRI prices are checked against the proposer's values (within a configurable margin). The proposer-supplied `init.l1_gas_price_wei` and `init.l1_data_gas_price_wei` are never compared to anything. [1](#0-0) 

**Step 2 – Unvalidated WEI prices enter execution.**

`convert_to_sn_api_block_info` takes the accepted `ProposalInit` and builds the `BlockInfo` that is handed to the batcher. It uses `init.l1_gas_price_wei` and `init.l1_data_gas_price_wei` verbatim as the ETH gas prices:

```rust
let l1_gas_price_wei  = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
``` [2](#0-1) 

**Step 3 – `l2_gas_price_wei` is derived from the tainted ratio.**

`calculate_eth_to_fri_rate` reconstructs the ETH/STRK rate as `l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei`. Because `l1_gas_price_wei` is attacker-controlled, the reconstructed rate is arbitrary. `l2_gas_price_wei` is then derived from this tainted rate:

```rust
let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
let l2_gas_price_wei = NonzeroGasPrice::new(
    init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?
)?;
``` [3](#0-2) [4](#0-3) 

**Step 4 – Wrong values committed to block hash.**

`gas_prices_to_hash` (for Starknet ≥ 0.13.4) hashes all six gas price fields including `l1_gas_price.price_in_wei` and `l2_gas_price.price_in_wei`. Both are now wrong:

```rust
.chain(&l1_gas_price.price_in_wei.0.into())   // attacker-controlled
...
.chain(&l2_gas_price.price_in_wei.0.into())   // derived from attacker-controlled ratio
``` [5](#0-4) 

`PartialBlockHashComponents::new` pulls these values directly from the `BlockInfo` that was built from the tainted `ProposalInit`: [6](#0-5) 

### Impact Explanation

1. **Wrong block hash committed.** The block hash includes `l1_gas_price_wei` and the derived `l2_gas_price_wei`. A proposer who inflates or deflates `l1_gas_price_wei` by any factor produces a block hash that differs from the honest value, yet is accepted by all validators because the FRI prices pass the margin check.

2. **Wrong ETH-denominated fees for L1 handler transactions.** `BlockInfo.eth_gas_prices.l1_gas_price` is set to the unvalidated `l1_gas_price_wei`. L1 handler transactions pay fees in ETH; a manipulated WEI price causes over- or under-charging, directly affecting balances and receipts stored in state.

3. **Wrong `l2_gas_price_wei` propagated to state sync and storage.** `update_state_sync_with_new_block` writes `l2_gas_price` (including the WEI component) into `BlockHeaderWithoutHash`, which is persisted and served by the RPC layer. [7](#0-6) 

### Likelihood Explanation

Any consensus validator that is selected as proposer in a given round can trigger this without any special privilege beyond normal participation. The FRI prices must stay within the oracle's margin (a loose percentage check), while the WEI prices are completely unconstrained. The attack requires no external dependency failure and is reachable in every block where the attacker is proposer.

### Recommendation

In `is_block_info_valid`, validate the proposer-supplied WEI prices against the locally-computed oracle WEI prices using the same `within_margin` check already applied to the FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;
// existing FRI checks ...
if !(within_margin(init_proposed.l1_gas_price_wei,     l1_gas_prices_wei.l1_gas_price,      margin)
  && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

This ensures the WEI prices are consistent with the oracle's ETH/STRK rate, preventing the tainted ratio from propagating into `l2_gas_price_wei` and the block hash.

### Proof of Concept

1. Attacker is selected as proposer for block N.
2. Oracle reports `eth_to_fri_rate = R`, L1 base fee = `P_wei`. Honest FRI price = `P_fri = P_wei * R / 10^18`.
3. Attacker constructs `ProposalInit` with:
   - `l1_gas_price_fri = P_fri` (passes `within_margin` check)
   - `l1_gas_price_wei = P_wei * 1000` (1000× inflated, never checked)
4. Validator's `is_block_info_valid` accepts: FRI prices are within margin; WEI prices are discarded.
5. `convert_to_sn_api_block_info` reconstructs `eth_to_fri_rate = P_fri * 10^18 / (P_wei * 1000) = R / 1000`.
6. `l2_gas_price_wei = l2_gas_price_fri * 10^18 / (R/1000) = honest_l2_gas_price_wei * 1000`.
7. `gas_prices_to_hash` commits `l1_gas_price_wei = P_wei * 1000` and `l2_gas_price_wei = honest * 1000` into the block hash.
8. All validators reach consensus on this wrong block hash; L1 handler fees are 1000× inflated.
