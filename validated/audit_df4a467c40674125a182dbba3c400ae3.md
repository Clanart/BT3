### Title
`ProofOfInclusion::valid()` Contains a Tautological Final Check That Accepts Forged Proofs Against Any Root - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` is the primary API for verifying DataLayer Merkle inclusion proofs. Its final correctness check — `existing_hash == self.root_hash()` — is a tautology: after the loop, `existing_hash` is always equal to `self.root_hash()` by construction. The function therefore only verifies internal chain consistency, never that the proof terminates at the actual committed tree root. An unprivileged attacker can construct a `ProofOfInclusion` (via the `Streamable` deserializer or the Python constructor) that `valid()` accepts as `true` while proving inclusion of an arbitrary key in an arbitrary, attacker-chosen subtree — not the on-chain committed root.

### Finding Description

**Root cause — `valid()` in `proof_of_inclusion.rs`:**

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived entirely from self.layers
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;   // ← existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true if loop completes
}
```

After the loop body, `existing_hash` holds the last `calculated_hash`, which was accepted only because `calculated_hash == layer.combined_hash`. Therefore `existing_hash` equals `last_layer.combined_hash`. `self.root_hash()` also returns `last_layer.combined_hash`. The final comparison is an identity, not a security check. [1](#0-0) 

**What is missing:** `valid()` accepts no external root parameter and never compares the computed chain against any externally-committed root hash. The correct check would be `existing_hash == expected_root` where `expected_root` is the on-chain committed value.

**Attacker-controlled entry path:** `ProofOfInclusion` derives `Streamable`, so it can be deserialized from arbitrary bytes. It is also directly constructible from Python via the exposed `__new__` constructor:

```python
ProofOfInclusion(node_hash=bytes32(...), layers=[...])
``` [2](#0-1) 

The `valid()` method is exposed to Python as `py_valid()` and is the sole verification primitive callers are expected to invoke. [3](#0-2) 

### Impact Explanation

Any verifier that calls `proof.valid()` without separately checking `proof.root_hash() == on_chain_committed_root` will accept a forged proof. An attacker can:

1. Choose any `node_hash` (representing a key-value pair not in the real tree).
2. Build a chain of `ProofOfInclusionLayer` values that are internally consistent (each `combined_hash = internal_hash(prev, side, other_hash)`).
3. Submit the resulting `ProofOfInclusion`; `valid()` returns `true`.

This lets untrusted input prove invalid DataLayer state — a key-value pair is "proven" to be in the tree when it is not. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion or lets untrusted input prove invalid state.**

### Likelihood Explanation

The `valid()` method is the only verification primitive exposed. Its name and signature give no indication that callers must also check `root_hash()` against an external value. The fuzz target and all tests call only `proof.valid()` without any external root comparison, confirming the intended usage pattern. [4](#0-3) [5](#0-4) 

Any downstream DataLayer client that follows the documented API and calls only `proof.valid()` is exploitable without any privileged access.

### Recommendation

`valid()` must accept the expected root hash as a parameter and compare against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against externally-committed root
}
```

Remove or deprecate the no-argument `valid()` to prevent misuse. Update the Python binding accordingly. The `root_hash()` helper can remain as a convenience accessor but must not be used as the security anchor.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker-chosen leaf hash (not in any real tree)
fake_node_hash = bytes(range(32))

# Build one internally-consistent layer
other_hash = bytes([0xAB] * 32)
# Simulate calculate_internal_hash(fake_node_hash, Side::Right, other_hash)
# (exact byte layout depends on calculate_internal_hash; attacker computes it)
combined = hashlib.sha256(b"\x01" + fake_node_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=1,          # Side::Right
    other_hash=bytes32(other_hash),
    combined_hash=bytes32(combined),
)

forged_proof = ProofOfInclusion(
    node_hash=bytes32(fake_node_hash),
    layers=[layer],
)

# valid() returns True even though fake_node_hash is not in any real tree
assert forged_proof.valid() == True

# root_hash() returns the attacker-chosen combined value, not the on-chain root
print(forged_proof.root_hash())  # attacker-controlled bytes
```

The `valid()` call returns `True` for a proof that was never generated from a real `MerkleBlob`, proving that the check provides no binding to any committed tree state.
