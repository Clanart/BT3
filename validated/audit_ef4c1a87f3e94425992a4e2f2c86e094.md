Let me look at the exact threading model and the `shard_layout` acquisition in `process_tx_internal` to confirm the race window.

### Title
Post-Reshard Stale-ShardUId Insertion Race in `ShardedTransactionPool` — (`chain/chunks/src/client.rs`, `chain/client/src/rpc_handler.rs`)

---

### Summary

An unprivileged RPC client can submit a transaction T for account A at an epoch boundary such that `process_tx_internal` computes `shard_uid` from the **old** shard layout outside the pool mutex, then inserts T under that stale key **after** `reshard()` has already completed under the same mutex. T ends up permanently stored in `tx_pools[old_shard_uid]`, a pool that no chunk producer will ever drain, causing T to be silently dropped.

---

### Finding Description

**Entrypoint and target symbol:** `RpcHandlerActor::process_tx_internal` (`chain/client/src/rpc_handler.rs`) → `ShardedTransactionPool::insert_transaction` (`chain/chunks/src/client.rs:65`). The resharding counterpart is `ShardedTransactionPool::reshard` (`chain/chunks/src/client.rs:131`), called from `on_block_accepted_with_optional_chunk_produce` (`chain/client/src/client.rs:1901`).

**The race window:**

In `process_tx_internal`, the shard layout and derived `shard_uid` are computed **before** the pool mutex is acquired:

```
line 177: let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)?;
line 216: let shard_uid = shard_layout.account_id_to_shard_uid(signer_id);
          // ... validation work, no lock held ...
line 279: let mut pool = self.tx_pool.lock();
line 280: pool.insert_transaction(shard_uid, validated_tx)
``` [1](#0-0) [2](#0-1) [3](#0-2) 

Meanwhile, on the Client thread, at the epoch boundary:

```
line 1900: let mut guarded_pool = self.chunk_producer.sharded_tx_pool.lock();
line 1901: guarded_pool.reshard(&old_shard_layout, &new_shard_layout);
           // lock released
``` [4](#0-3) 

`reshard()` drains every old-shard pool and re-inserts all transactions under new `ShardUId` keys: [5](#0-4) 

**Concrete race sequence:**

1. RPC thread reads `head`, derives `epoch_id` → old epoch, computes `shard_uid = old_shard_uid` at line 216. No lock held yet.
2. Client thread acquires the mutex, calls `reshard()`: drains `tx_pools[old_shard_uid]`, re-inserts all existing transactions under `new_shard_uid`. Releases mutex.
3. RPC thread acquires the mutex at line 279, calls `insert_transaction(old_shard_uid, T)`. `pool_for_shard` uses `HashMap::entry().or_insert_with()`, so it either finds the now-empty old pool or creates a fresh one, and inserts T there. [6](#0-5) 

**Exact corrupted value:** `tx_pools[old_shard_uid]` contains T; `tx_pools[new_shard_uid]` does not. The chunk producer for `new_shard_uid` calls `get_pool_iterator(new_shard_uid)` and never sees T. The old shard no longer exists after resharding, so no chunk producer ever drains `tx_pools[old_shard_uid]`.

The codebase itself acknowledges this gap with a TODO directly above the reshard call:

```rust
// TODO make sure transactions don't get added for the old shard
// layout after the pool resharding
``` [7](#0-6) 

---

### Impact Explanation

T is permanently stranded in a dead pool entry keyed by `old_shard_uid`. No chunk producer will ever call `get_pool_iterator(old_shard_uid)` after the epoch boundary, so T is never included in any chunk. From the user's perspective the transaction is silently dropped with no error response. The corrupted value — the `ShardUId` key under which T is stored in `tx_pools` — survives indefinitely because nothing cleans up stale shard entries after resharding.

---

### Likelihood Explanation

The race requires a transaction to arrive at the exact epoch boundary block. Epoch boundaries are predictable (fixed epoch length), so a motivated client can time submissions deliberately. `RpcHandlerActor` is explicitly multi-threaded (`spawn_multithread_actor`), making the interleaving physically possible on any multi-core validator node. The window between line 216 and line 279 spans several DB reads (`validate_tx`, `can_verify_and_charge_tx`), giving the Client thread ample time to complete `reshard()`.

---

### Recommendation

Compute `shard_uid` **inside** the pool mutex, or re-derive it from the current shard layout at the moment of insertion. Alternatively, after `reshard()` completes, record the new layout version in `ShardedTransactionPool` and have `insert_transaction` remap any incoming `old_shard_uid` to the correct `new_shard_uid` before inserting. The existing TODO at `client.rs:1890` already tracks this gap.

---

### Proof of Concept

A test-loop test with a 2-shard resharding scenario:

1. Configure a genesis with 2 shards and a known epoch length.
2. Advance the chain to one block before the epoch boundary.
3. On the RPC handler thread, submit a transaction for an account whose shard assignment changes at the boundary; pause the thread after `shard_uid` is computed (line 216) but before the mutex is acquired (line 279) — achievable with a `tokio::task::yield_now()` or a condvar injected in test mode.
4. On the main thread, process the epoch-boundary block, triggering `reshard()`.
5. Resume the RPC thread; it acquires the mutex and inserts T under `old_shard_uid`.
6. Assert: `pool.tx_pools.contains_key(&old_shard_uid)` is true and `pool.tx_pools[&old_shard_uid].len() == 1`, while `pool.tx_pools[&new_shard_uid]` does not contain T's hash.
7. Advance one more block and assert T does not appear in any produced chunk.

    pub fn reshard(&mut self, old_shard_layout: &ShardLayout, new_shard_layout: &ShardLayout) {
        tracing::debug!(
            target: "resharding",
            old_shard_layout_version = old_shard_layout.version(),
            new_shard_layout_version = new_shard_layout.version(),
            "resharding the transaction pool"
        );
        debug_assert!(old_shard_layout != new_shard_layout);

        let mut validated_txs = vec![];

        for old_shard_uid in old_shard_layout.shard_uids() {
            if let Some(mut iter) = self.get_pool_iterator(old_shard_uid) {
                while let Some(group) = iter.next() {
                    while let Some(validated_tx) = group.next() {
                        validated_txs.push(validated_tx);
                    }
                }
            }
        }

        for validated_tx in validated_txs {
            let signer_id = validated_tx.signer_id();
            let new_shard_uid = new_shard_layout.account_id_to_shard_uid(signer_id);
            self.insert_transaction(new_shard_uid, validated_tx);
        }
    }
