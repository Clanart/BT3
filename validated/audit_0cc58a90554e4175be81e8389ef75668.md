### Title
EIP-712 `Tx` Type Schema Omits `timeout_height`, Enabling Signature Reuse Across Arbitrary Expiry Windows — (File: `ethereum/eip712/eip712_legacy.go`)

---

### Summary

Both the legacy and current EIP-712 `Tx` type schemas in Ethermint intentionally omit the `timeout_height` field. Because EIP-712 only hashes fields declared in the type schema, a user's `timeout_height` constraint is never covered by their signature. An attacker who intercepts a signed legacy-EIP-712 Cosmos transaction can strip or replace `timeout_height` without invalidating the signature, allowing the transaction to be submitted and executed after the user's intended expiry block.

---

### Finding Description

**Root cause — type schema omission:**

In `ethereum/eip712/eip712_legacy.go`, `extractMsgTypes` hardcodes the `Tx` EIP-712 type schema and explicitly comments out `timeout_height`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
```

The identical omission exists in the current (non-legacy) schema in `ethereum/eip712/types.go`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "sequence", Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
},
``` [1](#0-0) [2](#0-1) 

**Sign-bytes construction includes `timeout_height`:**

In `ante/cosmos/eip712.go`, `VerifySignature` reconstructs the Amino sign bytes from the live transaction, passing `tx.GetTimeoutHeight()`:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),          // ← included in JSON when non-zero
    legacytx.StdFee{...},
    msgs, tx.GetMemo(),
)
``` [3](#0-2) 

The Cosmos SDK `StdSignBytes` emits `"timeout_height": "<N>"` in the Amino JSON whenever the value is non-zero (the struct field carries `omitempty`). That JSON is then passed to `LegacyWrapTxToTypedData`, which builds the EIP-712 typed data. Because `timeout_height` is absent from the `Tx` type definition, the EIP-712 `hashStruct` algorithm ignores it entirely — the field is present in `typedData.Message` but is never encoded into the hash. [4](#0-3) 

**Consequence — hash is identical for any `timeout_height`:**

The EIP-712 hash `H` produced for a transaction with `timeout_height = 100` is byte-for-byte identical to the hash produced for the same transaction with `timeout_height = 0`. A signature `S` over `H` therefore authorises execution at any block height.

The ante handler chain for the legacy EIP-712 path is:

```
NewTxTimeoutHeightDecorator  →  NewLegacyEip712SigVerificationDecorator  →  ...
``` [5](#0-4) 

`NewTxTimeoutHeightDecorator` enforces the `timeout_height` from the **submitted** transaction. If the attacker zeroes it, the decorator passes unconditionally. `LegacyEip712SigVerificationDecorator` then recomputes the sign bytes from the modified transaction (with `timeout_height = 0`), derives the same EIP-712 hash, and the original signature verifies successfully.

---

### Impact Explanation

A user who signs a Cosmos EIP-712 transaction with `timeout_height = X` intends the transaction to be invalid after block `X`. Because `timeout_height` is excluded from the EIP-712 hash, the signature is equally valid for `timeout_height = 0`. An attacker can:

1. Intercept the signed transaction from the mempool before block `X`.
2. Mutate `timeout_height` to `0` in the serialised transaction.
3. Withhold the transaction until after block `X` (when the user believes it has safely expired).
4. Submit the mutated transaction; both `TxTimeoutHeightDecorator` and `LegacyEip712SigVerificationDecorator` pass.
5. The transaction executes — transferring funds, delegating stake, voting, etc. — against the user's intent.

This is an EIP-712 signer-verification bypass enabling forged execution: the user's signature is accepted for a transaction whose expiry parameter differs from what was signed.

---

### Likelihood Explanation

**Medium.** The legacy EIP-712 ante handler (`newLegacyCosmosAnteHandlerEip712`) is still present and wired in the codebase. Any wallet or dapp that constructs a Cosmos EIP-712 transaction with a non-zero `timeout_height` (a standard user-safety measure) is exposed. The attacker only needs mempool read access — no privileged role, no key material, no governance action. The mutation is a single integer field change in a publicly broadcast transaction.

---

### Recommendation

**Short term:** Add `timeout_height` back to the `Tx` type definition in both `ethereum/eip712/eip712_legacy.go` (`extractMsgTypes`) and `ethereum/eip712/types.go` (`createEIP712Types`):

```go
{Name: "timeout_height", Type: "string"},
```

This ensures the field is covered by the EIP-712 hash and any mutation invalidates the signature.

**Alternative:** Reject Amino sign docs that carry a non-zero `timeout_height` in `decodeAminoSignDoc` (mirroring the existing guard in `decodeProtobufSignDoc`), so that the unsupported field cannot be silently stripped. [6](#0-5)

### Proof of Concept

```
1. User builds a MsgSend with timeout_height = 500 and signs it via the
   legacy EIP-712 path (ExtensionOptionsWeb3Tx present).

2. Signing path:
   - StdSignBytes produces JSON: {..., "timeout_height": "500", ...}
   - LegacyWrapTxToTypedData builds TypedData; Tx schema has no timeout_height
   - TypedDataAndHash computes H (timeout_height absent from hash input)
   - User's ethsecp256k1 key signs H → signature S

3. Attacker observes the broadcast transaction in the mempool.
   Attacker sets TxBody.TimeoutHeight = 0 in the Cosmos Tx protobuf.

4. Block 500 passes. User believes the transaction has expired.

5. Attacker submits the mutated transaction at block 600.

6. Ante handler execution:
   a. NewTxTimeoutHeightDecorator: timeout_height == 0 → no expiry check → PASS
   b. LegacyEip712SigVerificationDecorator.VerifySignature:
      - StdSignBytes(..., tx.GetTimeoutHeight()=0, ...) → JSON without timeout_height
      - LegacyWrapTxToTypedData → same Tx schema (no timeout_height)
      - TypedDataAndHash → same hash H
      - ethcrypto.VerifySignature(pubKey, H, S) → TRUE → PASS

7. MsgSend executes; user's funds are transferred at block 600,
   150 blocks after the user believed the transaction had safely expired.
``` [3](#0-2) [1](#0-0) [7](#0-6)
