### Title
Token-2022 TransferFee Extension Causes Vault Undercollateralization vs. Wormhole Message Amount — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

`InitTransfer::process` calls `transfer_checked` with `payload.amount` as the gross transfer amount, then immediately serializes that same `payload.amount` into the Wormhole message via `serialize_for_near`. When the mint has a Token-2022 `TransferFee` extension, the SPL Token-2022 program withholds the fee from the **destination** (vault), so the vault receives only `payload.amount - fee`. The Wormhole message, however, encodes the full `payload.amount`. NEAR then mints or unlocks `payload.amount` tokens against a vault that holds only `payload.amount - fee`, creating unbacked supply.

---

### Finding Description

**Entrypoint:** `InitTransfer::process` in `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`.

**Step 1 — Vault registration is permissionless and unchecked for TransferFee.**

`log_metadata` creates the vault PDA for any Token-2022 mint whose `mint_authority` is not the bridge authority PDA. There is no check for the `TransferFee` extension: [1](#0-0) 

**Step 2 — `transfer_checked` is called with the gross `payload.amount`.** [2](#0-1) 

Under SPL Token-2022 `TransferFee`, `transfer_checked(amount=X)` debits `X` from the sender but credits only `X - fee` to the destination (vault); the fee is withheld inside the vault's `withheld_amount` field and is **not** part of the vault's spendable `amount`.

**Step 3 — The Wormhole message encodes the full `payload.amount`, not the net amount received.** [3](#0-2) 

`serialize_for_near` writes `self.amount` directly: [4](#0-3) 

**Step 4 — No guard exists.** `SECURITY.md` acknowledges that Token-2022 tokens with *transfer hooks* are unsupported (they fail at runtime), but says nothing about `TransferFee`. `TransferFee` requires no extra accounts and does not cause a runtime failure — it silently reduces the vault credit. [5](#0-4) 

---

### Impact Explanation

Every `init_transfer` call on a `TransferFee`-bearing native Token-2022 mint creates a deficit: vault holds `X - fee`, NEAR mints/unlocks `X`. The deficit compounds with each transfer. Eventually the vault cannot satisfy all redemptions — some users bridging back will find the vault short, permanently locking their NEAR-side tokens or forcing a protocol shortfall. This is a direct bridge collateralization break.

---

### Likelihood Explanation

- `log_metadata` is fully permissionless; any Token-2022 mint (including legitimate tokens with `TransferFee`) can be registered.
- `TransferFee` is a standard, widely-used Token-2022 extension (e.g., many DeFi tokens use it for protocol revenue).
- No privileged role is required. Any user calling `init_transfer` with such a mint triggers the discrepancy.
- The discrepancy is silent — no error is thrown, the transaction succeeds, and the Wormhole VAA is valid.

---

### Recommendation

After `transfer_checked` returns, read back the vault's actual post-transfer balance (or compute the net amount using the mint's `TransferFee` extension parameters) and use that net amount in `serialize_for_near` instead of `payload.amount`. Alternatively, reject mints that have a non-zero `TransferFee` extension during `log_metadata` vault creation, similar to how transfer-hook mints are documented as unsupported.

---

### Proof of Concept

1. Deploy a Token-2022 mint with `TransferFeeConfig { transfer_fee_basis_points: 1000, maximum_fee: u64::MAX, ... }` (10% fee).
2. Call `log_metadata` to register the mint and create the vault PDA.
3. Call `init_transfer` with `payload.amount = 1_000_000`.
4. Observe: vault `amount` = 900_000 (10% withheld); Wormhole payload `amount` field = 1_000_000.
5. NEAR processes the VAA and mints 1_000_000 wrapped tokens.
6. Vault is undercollateralized by 100_000 tokens from a single call; the deficit grows linearly with usage.
