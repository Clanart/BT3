### Title
Cross-Chain Replay of `deployToken` / `deploy_token` Signature Permanently Blocks Legitimate Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`, `starknet/src/bridge_types.cairo`)

---

### Summary

The Borsh-encoded message signed by the NEAR MPC key for `deployToken` / `deploy_token` operations does not include a destination chain identifier. Because the same `nearBridgeDerivedAddress` / `derived_near_bridge_address` is used across all supported chains, a valid signature produced for one chain is cryptographically valid on every other chain. An unprivileged attacker can observe a `deployToken` transaction on chain A, extract the signature, and replay it on chains B, C, … N, permanently preventing the NEAR bridge from ever legitimately deploying that token on those chains.

---

### Finding Description

**Asymmetry between `deployToken` and `finTransfer` message construction**

`finTransfer` / `finalize_transfer` correctly binds its signed message to the destination chain by embedding `omniBridgeChainId` / `SOLANA_OMNI_BRIDGE_CHAIN_ID` in the Borsh payload. `deployToken` / `deploy_token` does not.

### Impact Explanation

Once the NEAR MPC signer produces a `deployToken` signature for token `T` on chain A, the identical signature passes `ECDSA.recover(...) != nearBridgeDerivedAddress` / `secp256k1_recover` / `verify_eth_signature` on every other chain that shares the same derived address, because the signed bytes are identical.

An attacker who replays the signature on chain B causes:

1. The bridge token for `T` to be deployed on chain B without NEAR's authorization.
2. The `nearToEthToken[metadata.token]` / `near_to_starknet_token` mapping to be permanently populated on chain B.
3. Any future legitimate attempt by NEAR to deploy `T` on chain B to revert with `ERR_TOKEN_EXIST` (EVM), `ERR_TOKEN_ALREADY_DEPLOYED` (StarkNet), or the equivalent Solana check — **permanently**.

This permanently freezes the bridge's ability to support token `T` on chain B through the normal flow, matching the **"Permanent freezing, irrecoverable lock, or unclaimable settlement"** impact class. [9](#0-8) [10](#0-9) 

---

### Likelihood Explanation

- All `deployToken` transactions are public on-chain. Any observer can extract the `signatureData` / `signature` field from a confirmed transaction on chain A.
- No privileged access, leaked key, or colluding MPC signer is required — the attacker only needs to submit a standard public transaction on chain B.
- The Omni Bridge supports many EVM-compatible chains (Ethereum, Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract) plus Solana and StarkNet. Every new token deployment on any one chain immediately creates a replayable signature valid on all others.
- The attack is cheap (one transaction per target chain) and irreversible.

---

### Recommendation

Include the destination chain identifier in the Borsh-encoded metadata payload that is signed by the MPC key, mirroring the pattern already used in `finTransfer` / `finalize_transfer`.

**EVM**: Add `bytes1(omniBridgeChainId)` to the `borshEncoded` concatenation in `deployToken`.

**Solana**: Write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the `BufWriter` in `DeployTokenPayload::serialize_for_near`, as is done in `FinalizeTransferPayload::serialize_for_near`.

**StarkNet**: Add a `chain_id: u8` parameter to `MetadataPayload::to_borsh` and append it to the byte array, then pass `self.omni_bridge_chain_id.read()` at the call site in `deploy_token`.

The NEAR side that produces the signature must also include the target chain ID in the payload it submits to the MPC signer.

---

### Proof of Concept

1. NEAR MPC signer produces a `deployToken` signature `σ` for token `"usdc.near"` targeting Ethereum (`omniBridgeChainId = 0`). The signed bytes are:
   ```
   [0x01] ++ borsh("usdc.near") ++ borsh("USD Coin") ++ borsh("USDC") ++ [0x06]
   ```
2. Attacker observes the Ethereum `deployToken` transaction and extracts `σ`.
3. Attacker calls `deployToken(σ, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum, Base, BNB, Polygon, StarkNet, and Solana.
4. On each chain, `ECDSA.recover(keccak256(borshEncoded), σ) == nearBridgeDerivedAddress` succeeds because the signed bytes are identical — no chain ID was included.
5. `nearToEthToken["usdc.near"]` is now populated on all chains.
6. NEAR later attempts to legitimately deploy USDC on Arbitrum. The call reverts with `ERR_TOKEN_EXIST`. The token can never be bridged to Arbitrum through the standard flow. [11](#0-10) [12](#0-11) [13](#0-12)
