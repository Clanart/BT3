### Title
Fee-on-Transfer Token Accounting Mismatch Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer()` on EVM (and the analogous `init_transfer()` on StarkNet) uses the caller-specified `amount` in the emitted `InitTransfer` event rather than the actual tokens received by the contract. For fee-on-transfer (FoT) tokens, the bridge locks fewer tokens than the event claims, causing the NEAR side to release/mint more tokens than are backed by the EVM vault — breaking bridge collateralization.

### Finding Description

In `OmniBridge.initTransfer()`, when the token is a plain ERC20 (not a bridge token or custom minter), the contract pulls tokens from the user: [1](#0-0) 

After the transfer, the function emits `InitTransfer` using the caller-supplied `amount` parameter directly: [2](#0-1) 

For a FoT token, `safeTransferFrom(..., amount)` causes the contract to receive `amount - fee_deducted` tokens, but the event broadcasts `amount`. The NEAR bridge reads this event to determine how many tokens to release or mint on the destination chain. The result is that the NEAR side releases `amount` tokens while the EVM vault only holds `amount - fee_deducted`, creating an undercollateralized position that grows with every bridging operation.

The same pattern exists in the StarkNet bridge's `init_transfer`: [3](#0-2) [4](#0-3) 

### Impact Explanation

Every `initTransfer` call with a FoT token causes the bridge to emit a cross-chain message claiming more tokens are locked than actually are. The NEAR side honors the full `amount` from the event. Over repeated transfers, the EVM vault becomes progressively undercollateralized. When users attempt to bridge back (NEAR → EVM), the vault will eventually lack sufficient tokens to fulfill withdrawals — permanently freezing funds for the last users who cannot be paid out. This matches: **High — Balance/fee/accounting corruption that breaks bridge collateralization or misdirects value**, and potentially **Critical — Permanent freezing of user funds** for the last claimants.

### Likelihood Explanation

Any unprivileged bridge user can call `initTransfer` with any ERC20 token address that is not registered as a bridge token or custom minter. There is no whitelist enforced in the contract itself for the "else" branch. A user holding a FoT token (e.g., tokens with transfer taxes, deflationary tokens) can trigger this path directly. The entry point is fully public and requires no special role.

### Recommendation

Use a balance-before/balance-after pattern to compute the actual received amount, and use that value in the emitted event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
uint128 actualReceived = uint128(balanceAfter - balanceBefore);
// use actualReceived instead of amount in the event and downstream logic
```

Apply the same fix to StarkNet's `init_transfer`. Alternatively, document explicitly that FoT/rebasing tokens are not supported and enforce a token allowlist to prevent their use.

### Proof of Concept

1. A FoT token `FOT` charges a 5% fee on every transfer.
2. Attacker (or any user) calls `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
3. `safeTransferFrom` moves 1000 FOT from the user, but the bridge contract receives only 950 FOT (5% fee taken by the token).
4. The contract emits `InitTransfer(..., amount=1000, ...)`.
5. The NEAR bridge relayer reads the event and calls `ft_on_transfer` / releases 1000 FOT-equivalent tokens to `alice.near`.
6. The EVM vault is now short 50 tokens per transfer.
7. After 20 such transfers, the vault is completely drained while NEAR has minted 20,000 tokens against only 19,000 locked — the last ~1,000 tokens worth of NEAR-side claims cannot be honored on EVM withdrawal. [5](#0-4) [2](#0-1)
