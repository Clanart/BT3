### Title
Missing Shard-ID Binding Check in `set_state_header` Allows Cross-Shard State Confusion — (File: `chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` validates a peer-supplied `ShardStateSyncResponseHeader` by checking the chunk's internal proofs and its Merkle inclusion in the block, but never verifies that the chunk inside the header actually belongs to the requested `shard_id`. A malicious peer can supply a structurally valid header whose embedded chunk belongs to a different shard, causing the syncing node to store the wrong state root under the target shard's key and subsequently reconstruct the wrong trie state for that shard.

### Finding Description

`set_state_header(shard_id, sync_hash, shard_state_header)` performs the following checks:

1. `validate_chunk_proofs(&chunk, ...)` — verifies the chunk's internal hash, tx root, and receipts root.
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk is present somewhere in the block's Merkle tree of chunk headers. [1](#0-0) 

Neither check binds the chunk to the caller-supplied `shard_id`. The `chunk_headers_root` is a Merkle tree over **all** shards' chunks ordered by shard index. A proof for shard Y's chunk at position Y passes `verify_path` regardless of whether the request was for shard X. After passing both checks, the header is persisted under the key `StateHeaderKey(shard_id, sync_hash)`: [2](#0-1) 

The missing check is `chunk.shard_id() == shard_id`. This is the exact structural analog of the external report: the governor account is validated in isolation (its `smart_wallet` field is consistent) but is never bound to the locker it is supposed to govern. Here, the chunk is validated in isolation and its block inclusion is verified, but it is never bound to the shard it is supposed to represent.

Downstream, `set_state_part` retrieves the stored header and extracts the state root from the embedded chunk to validate incoming parts: [3](#0-2) 

If the header was poisoned with shard Y's chunk, the state root is shard Y's `prev_state_root`. The attacker then supplies state parts that are cryptographically valid for shard Y's trie. All part validations pass. `set_state_finalize` then applies those parts into shard X's storage slot: [4](#0-3) 

The result is that shard X's trie is overwritten with shard Y's state.

### Impact Explanation

A syncing node (validator catching up after resharding, or a new node) that connects to a malicious peer for state sync will reconstruct the wrong trie for the targeted shard. Consequences:

- The node produces chunks with an incorrect `prev_state_root`, which are rejected by the rest of the network.
- The node's chunk endorsements are invalid, causing it to be kicked out of the validator set and lose staked tokens.
- The node cannot serve correct state to other peers for that shard.

The corruption is silent at the time of `set_state_header`; it only manifests when the node attempts to produce or validate chunks after state sync completes.

### Likelihood Explanation

State sync peers are selected from `SnapshotHostInfo` advertisements received over the gossip network. Any node that advertises a snapshot for the target `sync_hash` and `shard_id` can be selected as a source. An attacker only needs to run a node, advertise a snapshot, and serve a crafted `ShardStateSyncResponseHeader`. No privileged role (validator, chunk producer) is required. The attack is reachable whenever a node performs state sync, which occurs at every epoch boundary for validators assigned to new shards.

### Recommendation

Add an explicit shard-ID binding check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the remediation in the external report: verify the binding between the parent entity (the requested `shard_id`) and the child entity (the chunk embedded in the header), not just the child's internal consistency.

### Proof of Concept

1. Honest node N is syncing shard 0 at `sync_hash` H. It selects attacker peer P as its state source.
2. P serves a `ShardStateSyncResponseHeader` where the embedded `chunk` is the legitimate chunk for **shard 1** at the same block, accompanied by shard 1's valid Merkle proof against `chunk_headers_root`.
3. N calls `set_state_header(shard_id=0, sync_hash=H, header)`. `validate_chunk_proofs` passes (shard 1's chunk is internally valid). `verify_path` passes (shard 1's chunk is genuinely in the block). No check on `chunk.shard_id()` exists. The header is stored under `StateHeaderKey(shard_id=0, H)`.
4. N requests state parts for shard 0. `set_state_part` reads the stored header, extracts shard 1's `prev_state_root`, and validates incoming parts against it. P supplies parts valid for shard 1's trie; all pass.
5. `set_state_finalize` applies shard 1's trie into shard 0's storage slot (`shard_uid` for shard 0).
6. N now has shard 1's state installed as shard 0's state. All subsequent chunk production for shard 0 uses the wrong state root and is rejected by the network.
