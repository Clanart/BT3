Audit Report

## Title
Unbounded `newTimelock` Causes uint32 Overflow in `_afterTimelock()`, Allowing Pool Admin to Bypass All Subsequent Timelocks — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`proposeOracleStopLossTimelock()` and `initialize()` accept any `uint32 timelock` with no upper-bound validation. Setting `timelock = type(uint32).max` causes a silent truncation in `_afterTimelock()`, producing an `executeAfter` value approximately one second in the past. Every subsequent parameter proposal (drawdown, decay, watermarks) becomes immediately executable, defeating the timelock protection that LPs rely on to exit before adverse parameter changes take effect.

## Finding Description

`_afterTimelock()` computes the execution deadline for all pending proposals:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

The addition occurs in `uint256` space (since `block.timestamp` is `uint256`), then is explicitly cast to `uint32`, which silently truncates — this is not a checked overflow. With `timelock = type(uint32).max` (4,294,967,295) and `block.timestamp ≈ 1,753,000,000` (July 2026):

```
1,753,000,000 + 4,294,967,295 = 6,047,967,295
uint32(6,047,967,295) = 6,047,967,295 mod 4,294,967,296 ≈ 1,752,999,999
```

The result is `block.timestamp − 1`, so `_requireElapsed` passes immediately for every subsequent proposal:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

`proposeOracleStopLossTimelock()` has no validation on `newTimelock`:

```solidity
// L78-84
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
}
```

`initialize()` similarly validates `drawdownE6` and `decayPerSecondE8` but not `timelock`:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock: no validation
oracleStopLossConfig[pool] = PoolStopLossConfig({..., timelock: timelock, ...});
```

Once `timelock = type(uint32).max` is active, the pool admin can immediately call `proposeOracleStopLossHighWatermarks(pool, binIdx, type(uint104).max, type(uint104).max)` and `executeOracleStopLossHighWatermarks(pool)` without waiting. With watermarks set to `type(uint104).max`, every subsequent swap produces `metricT0` and `metricT1` far below the floor, causing `_checkAndUpdateWatermarks` to revert with `OracleStopLossTriggered` on every swap direction.

Note: The submitted PoC incorrectly states that setting `drawdown = 0` triggers stop-loss. In fact, line 217 (`if (drawdown == 0) return;`) causes the check to be skipped entirely when drawdown is zero. The correct exploit path uses watermark manipulation (setting watermarks to `type(uint104).max`) or a non-zero drawdown value, both of which are immediately executable after the overflow.

## Impact Explanation

This is an admin-boundary break: the pool admin, who is semi-trusted only within caps and timelocks, bypasses the timelock mechanism entirely. The NatSpec explicitly states "Drawdown and decay changes are timelocked so LPs can react." With the overflow active, the pool admin can atomically set watermarks to `type(uint104).max` and execute immediately, permanently blocking all swaps through `OracleStopLossTriggered`. LPs cannot exit via swaps and must use `removeLiquidity` directly. This constitutes broken core swap functionality and an admin-boundary break, both of which are allowed impacts under the contest gate.

## Likelihood Explanation

Two paths exist. First, a pool initialized with `timelock = type(uint32).max` (which appears to offer maximum LP protection) is exploitable from block 1 — no waiting required. Second, a pool with a legitimate timelock (e.g., 7 days) is exploitable after the current timelock elapses: the pool admin proposes `type(uint32).max`, waits the existing delay, executes, and then all future proposals are immediately executable. The emitted event value (`type(uint32).max`) is not obviously malicious to LP monitors. The pool admin role is distinct from the factory owner and is the party most likely to act adversarially against LPs.

## Recommendation

Add an upper-bound check in both `proposeOracleStopLossTimelock` and `initialize`, mirroring the existing pattern for `drawdownE6` and `decayPerSecondE8`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
    ...
}
```

Apply the same guard in `initialize` before storing the initial timelock value.

## Proof of Concept

```solidity
// 1. Pool initialized with timelock = type(uint32).max (no validation in initialize())
extension.initialize(pool, abi.encode(uint32(500_000), uint32(58), type(uint32).max));

// 2. _afterTimelock() overflows: uint32(block.timestamp + type(uint32).max) ≈ block.timestamp - 1
//    All proposals are immediately executable.

// 3. Set watermarks to max — no wait required
extension.proposeOracleStopLossHighWatermarks(pool, 0, type(uint104).max, type(uint104).max);
extension.executeOracleStopLossHighWatermarks(pool); // passes: executeAfter < block.timestamp

// 4. Every subsequent swap reverts with OracleStopLossTriggered — pool frozen for swaps
pool.swap(...); // reverts: metricT0/metricT1 << type(uint104).max
```
