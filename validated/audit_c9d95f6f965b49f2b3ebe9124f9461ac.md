### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Checks, Silently Corrupting Tree State and Root Hash - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` has a fast-path that skips the duplicate-key and duplicate-hash guards enforced by `MerkleBlob::insert`. When a batch contains a key or hash already present in the tree (or a duplicate within the batch itself), `insert_entry_to_blob` silently overwrites the `BlockStatusCache` HashMap entries, leaving a phantom leaf in the blob that is no longer tracked. The resulting tree root hash is computed over a structurally invalid tree, enabling forged proofs of inclusion and corrupted DataLayer state.

### Finding Description

`MerkleBlob::insert` enforces two guards before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` uses `insert()` (and therefore these guards) only for the last ≤2 items when the tree currently has ≤1 leaf. All remaining items are written through the unguarded fast path:

```rust
for ((key, value), hash) in keys_values_hashes {          // no duplicate check
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which calls `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`. `HashMap::insert` silently overwrites an existing entry and returns the old value — no error is raised:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);  // silent overwrite
}
```

After the overwrite:
- The cache (`key_to_index`, `leaf_hash_to_index`) points to the *new* leaf's index.
- The *old* leaf node is still physically present in the blob and is included in the tree structure and root-hash computation.
- The tree now contains two leaf nodes with the same key/hash, but only one is tracked by the cache.

This is the direct analog of the Solidity bug: an "overwrite" path that appends to a tracking collection without checking for an existing entry, leaving the collection in an inconsistent state that breaks subsequent operations.

### Impact Explanation

**DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

1. **Corrupted root hash**: `calculate_lazy_hashes` traverses the full blob, including the phantom (untracked) old leaf. The computed root hash covers a tree with duplicate keys — an invalid state that no honest tree would produce.

2. **Forged proof of inclusion**: `get_proof_of_inclusion` uses `block_status_cache.get_index_by_key` to locate the leaf, then walks the lineage. Because the cache points to the *new* leaf's index, the proof is built from the new leaf's position in the tree. However, the root hash was computed over a tree that also contains the old leaf, so the proof verifies against a root that does not correspond to any valid single-key tree state.

3. **Subsequent delete corruption**: `delete` calls `get_leaf_by_key` (cache lookup) and then `remove_leaf`. The old phantom leaf is never deleted; it remains in the blob permanently, further skewing all future root-hash computations.

### Likelihood Explanation

The Python binding `py_batch_insert` is the primary public entry point for bulk DataLayer insertions. Any caller that:
- Passes a batch containing a key already present in a tree with ≥2 existing leaves, **or**
- Passes a batch with an internal duplicate key (≥3 items when tree is empty, ≥1 item when tree has ≥2 leaves)

will silently corrupt the tree. No error is returned; the operation appears to succeed. DataLayer sync code that calls `batch_insert` without pre-deduplicating against the existing tree state is a realistic trigger.

### Recommendation

Add duplicate-key and duplicate-hash checks at the start of the fast path in `batch_insert`, mirroring the guards in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Alternatively, deduplicate the input vector before processing, or track seen keys/hashes in a local `HashSet` within `batch_insert` to catch intra-batch duplicates as well.

### Proof of Concept

```rust
#[test]
fn test_batch_insert_duplicate_key_corrupts_tree() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so the fast path is taken for all batch items
    blob.insert(KeyId(0), ValueId(0), &sha256_num(&0i64), InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(1), ValueId(1), &sha256_num(&1i64), InsertLocation::Auto {}).unwrap();

    // Batch contains KeyId(0) which already exists — fast path, no duplicate check
    let batch = vec![
        ((KeyId(0), ValueId(99)), sha256_num(&99i64)), // duplicate key!
        ((KeyId(2), ValueId(2)), sha256_num(&2i64)),
    ];

    // Returns Ok — no error despite duplicate
    blob.batch_insert(batch).unwrap();

    // Tree now has 3 leaf nodes in the blob but cache only tracks 3 keys
    // (KeyId(0) cache entry was silently overwritten to point to the new leaf)
    // The old KeyId(0) leaf is still in the blob and included in root hash computation
    // check_integrity() will fail or produce an incorrect root
    blob.check_integrity().unwrap(); // may panic or return integrity error
}
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }
