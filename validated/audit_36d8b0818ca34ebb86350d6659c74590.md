### Title
`ProofOfInclusion::valid()` Contains a Tautological Root Check, Accepting Forged Inclusion Proofs Without External Root Verification — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` only verifies the internal self-consistency of a proof structure. Its final check — `existing_hash == self.root_hash()` — is a tautology: after the loop, `existing_hash` is always equal to `self.root_hash()` by construction. The function never validates the computed root against an externally trusted root hash. Any attacker who can supply a `ProofOfInclusion` (which is `Streamable` and exposed via Python bindings) can forge a proof for an arbitrary key/value pair that passes `valid()`.

### Finding Description

The `valid()` method in `proof_of_inclusion.rs` is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautology
}
```

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← returns the proof-embedded value
    } else {
        self.node_hash
    }
}
```

**The tautology:** The loop only continues past each layer if `calculated_hash == layer.combined_hash`, so after the loop `existing_hash` equals `last.combined_hash`. `root_hash()` also returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` when the loop completes without returning `false`. The same holds for the empty-layers case: `existing_hash` starts as `self.node_hash`, the loop does not execute, and `root_hash()` returns `self.node_hash`.

The function never accepts an external trusted root as a parameter. It only checks that the proof's own internal hashes are self-consistent — analogous to the Chainlink report's `latestAnswer()` returning a value without checking `answeredInRound >= roundId` or `updatedAt > 0`. In both cases, the validation function omits the check that would bind the returned/computed value to an authoritative external reference.

`ProofOfInclusion` is `Streamable` (deserializable from raw bytes) and is exposed directly via Python bindings:

```rust
#[pyo3(name = "get_proof_of_inclusion")]
pub fn py_get_proof_of_inclusion(
    &self,
    key: KeyId,
) -> PyResult<proof_of_inclusion::ProofOfInclusion> {
    Ok(self.get_proof_of_inclusion(key)?)
}
```

Any caller that deserializes a `ProofOfInclusion` from an untrusted source and calls `.valid()` — without separately asserting `proof.root_hash() == known_trusted_root` — will accept a completely fabricated proof.

### Impact Explanation

An attacker who can supply a `ProofOfInclusion` object (via the Python/wasm boundary, network deserialization, or any API that accepts a `Streamable` proof) can construct a proof for an arbitrary `node_hash` with any set of internally consistent layers. `valid()` returns `true`. Any DataLayer consumer that gates state transitions, access control, or coin-spend authorization on `proof.valid()` alone will accept forged inclusion proofs, allowing the attacker to prove invalid state — matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

### Likelihood Explanation

`ProofOfInclusion` is `Streamable` and fully exposed via Python bindings. The `valid()` method is the sole public API for proof verification. Any Python-layer DataLayer consumer (e.g., chia-blockchain's DataLayer plugin) that receives a proof from a peer and calls `proof.valid()` without a separate root check is vulnerable. The forged proof requires only that the attacker compute a valid SHA-256 chain for their chosen `node_hash` and sibling hashes — trivially achievable.

### Recommendation

`valid()` must accept an externally trusted root hash and compare against it, not against the proof-embedded `combined_hash`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == *trusted_root   // compare against caller-supplied root
}
```

The existing `valid()` (which only checks internal consistency) should be renamed to `is_internally_consistent()` to prevent misuse, and all call sites — including the Python binding — should be updated to pass the known-good tree root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side, calculate_internal_hash};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

// Attacker chooses arbitrary node_hash and sibling
let fake_node_hash: Hash = [0xAA; 32];
let fake_sibling:   Hash = [0xBB; 32];

// Compute a self-consistent combined_hash — no knowledge of the real tree needed
let fake_combined = calculate_internal_hash(&fake_node_hash, Side::Left, &fake_sibling);

let forged = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Left,
        other_hash:      fake_sibling,
        combined_hash:   fake_combined,   // self-consistent
    }],
};

// valid() returns true — no external root was checked
assert!(forged.valid());
// forged.root_hash() == fake_combined, which is attacker-controlled
```

The forged proof passes `valid()` because the tautological final check compares `existing_hash` (= `fake_combined`) to `self.root_hash()` (= `fake_combined`). The real tree root is never consulted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
