### Title
ERC1155 `safeTransferFrom` in `finTransfer` Permanently Locks Bridged Tokens When Recipient Cannot Receive ERC1155 - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` uses `IERC1155.safeTransferFrom()` to deliver ERC1155 tokens to the recipient. Because `safeTransferFrom` on ERC1155 invokes `onERC1155Received` on the recipient contract, any recipient that does not implement `IERC1155Receiver` will cause the call to revert. Since the recipient address is embedded in the MPC-signed payload and cannot be changed, the finalization can never succeed for that recipient, and the ERC1155 tokens are permanently irrecoverable from the bridge.

---

### Finding Description

In `OmniBridge.sol`, `finTransfer()` handles the delivery of ERC1155 tokens as follows:

```solidity
// OmniBridge.sol lines 323–330
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this),
        payload.recipient,
        multiToken.tokenId,
        payload.amount,
        ""
    );
}
``` [1](#0-0) 

The ERC1155 standard mandates that `safeTransferFrom` calls `onERC1155Received` on the recipient if it is a contract. If the recipient contract does not implement `IERC1155Receiver`, the call reverts.

The `payload.recipient` is part of the MPC-signed `TransferMessagePayload`. The signature is verified at line 311:

```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

Because the recipient is cryptographically bound in the signed payload, no relayer or user can substitute a different recipient address. Every subsequent call to `finTransfer` with the same signed payload will revert at the same `safeTransferFrom` line. There is no admin rescue function, no alternative delivery path, and no mechanism to redirect the tokens.

The ERC1155 tokens remain locked inside the `OmniBridge` contract indefinitely.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who bridges ERC1155 tokens from EVM to NEAR and specifies a contract recipient on EVM that does not implement `IERC1155Receiver` (e.g., a multisig wallet, a DAO treasury, a DeFi vault, or any contract deployed before ERC1155 receiver support was common) will have their tokens permanently locked in the bridge. The signed payload cannot be altered, there is no admin recovery path, and the `safeTransferFrom` will revert on every retry.

---

### Likelihood Explanation

**Medium-to-High.** ERC1155 tokens are a supported asset class in the bridge via `initTransfer1155`. Many real-world contract wallets (older Gnosis Safe versions without a fallback handler, DAO treasuries, protocol vaults) do not implement `IERC1155Receiver`. A user bridging ERC1155 tokens back to such a contract recipient — a common pattern in DeFi — will trigger this permanently. No special attacker capability is required; this is reachable by any ordinary bridge user.

---

### Recommendation

Replace `safeTransferFrom` with `transferFrom` when delivering ERC1155 tokens to the recipient in `finTransfer`:

```solidity
// Before (unsafe for non-receiver contracts):
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);

// After (safe for all recipients):
IERC1155(multiToken.tokenAddress).transferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount
);
```

Note: `IERC1155` does not define a plain `transferFrom`; the correct non-safe variant is `safeTransferFrom` with a trusted operator bypass, or the bridge can use a low-level call that skips the receiver callback. Alternatively, the bridge can expose an admin-controlled token rescue function so that stuck tokens can be redirected to a valid recipient.

---

### Proof of Concept

1. User calls `initTransfer1155(tokenAddress, tokenId, amount, ...)` on EVM, specifying a NEAR address as recipient. The ERC1155 tokens are pulled from the user into the bridge via `safeTransferFrom` at line 458. [3](#0-2) 

2. The NEAR bridge mints/unlocks the equivalent tokens for the user on NEAR.

3. The user on NEAR initiates a return transfer, specifying a contract address on EVM as the recipient — for example, a DAO treasury contract that does not implement `IERC1155Receiver`.

4. The MPC signs a `TransferMessagePayload` with `payload.recipient = <DAO treasury address>` and `payload.tokenAddress = <deterministicToken>`.

5. A relayer calls `finTransfer(signatureData, payload)` on EVM. The signature check passes. Execution reaches line 324: [4](#0-3) 

6. `IERC1155.safeTransferFrom` calls `onERC1155Received` on the DAO treasury. The treasury does not implement this interface, so the call reverts. The entire `finTransfer` transaction reverts.

7. Because the recipient is fixed in the MPC-signed payload, every retry of `finTransfer` with the same payload reverts at the same point. The ERC1155 tokens remain permanently locked in the `OmniBridge` contract. The user has no recourse.
