### Title
Rejected L1 Handler Transactions Remain Indefinitely Re-Includable by Malicious Proposer Due to Missing `Rejected` Guard in `is_validatable` - (File: crates/apollo_l1_provider/src/transaction_record.rs)

### Summary

`TransactionRecord::is_validatable()` does not exclude the `Rejected` state. When a block proposal containing an L1 handler transaction is committed with that transaction in the rejected set, the transaction's record state is set to `Rejected` and it is removed from the `proposable_index` (so honest proposers cannot re-propose it). However, because `is_validatable()` returns `true` for `Rejected`, a malicious proposer can include the same L1 handler transaction in a future block proposal and every validator node will call `validate_tx` → receive `ValidationStatus::Validated` → accept the transaction into the block. The transaction is then re-executed by the blockifier, producing a wrong state diff and wrong block commitment.

### Finding Description

`is_validatable` is defined as:

```rust
pub fn is_validatable(&self) -> bool {
    !self.is_committed() && !self.is_cancelled() && !self.is_consumed()
}
``` [1](#0-0) 

It guards only against `Committed`, `CancelledOnL2`, and `Consumed` states. The `Rejected` state is not listed.

`validate_tx` in `TransactionManager` calls `is_validatable()` as its primary gate:

```rust
if !record.is_validatable() {
    match record.state {
        TransactionState::Committed => InvalidValidationStatus::AlreadyIncludedOnL2.into(),
        TransactionState::CancelledOnL2 => InvalidValidationStatus::CancelledOnL2.into(),
        TransactionState::Consumed => InvalidValidationStatus::ConsumedOnL1.into(),
        _ => unreachable!(),   // ← Rejected falls here if it ever reaches this arm
    }
} else if record.try_mark_staged(current_staging_epoch_cloned) {
    ValidationStatus::Validated   // ← Rejected reaches here instead
}
``` [2](#0-1) 

Because `Rejected` passes `is_validatable()`, the `match` arm is never reached for it; instead the record is staged and `Validated` is returned.

The `Rejected` state is set during `commit_txs` when the batcher reports an L1 handler transaction as both consumed and rejected:

```rust
for &tx_hash in rejected_txs {
    self.with_record(tx_hash, |r| r.mark_rejected()).expect(...);
}
``` [3](#0-2) 

`mark_rejected` sets `state = TransactionState::Rejected` and `rejected = true`: [4](#0-3) 

`maintain_indices` then removes the transaction from `proposable_index` because `is_proposable()` only returns `true` for `Pending`: [5](#0-4) [6](#0-5) 

So the rejected transaction is invisible to honest proposers but fully visible to validators. The existing test `validate_rejected_transactions` explicitly confirms this behavior — `tx_hash!(1)` in `Rejected` state returns `ValidationStatus::Validated`: [7](#0-6) 

The batcher constructs the rejected L1 handler set as the intersection of `rejected_tx_hashes` and `consumed_l1_handler_tx_hashes`: [8](#0-7) 

### Impact Explanation

A malicious proposer streams a block proposal that re-includes a previously-rejected L1 handler transaction. Every validator node calls `l1_provider_client.validate(tx_hash, height)`, which internally calls `validate_tx` → `is_validatable()` → `true` → `Validated`. The validator's `ValidateTransactionProvider` then forwards the transaction to the blockifier for execution. The blockifier re-executes the L1 handler, producing a state diff that differs from what the honest network expects. The resulting `ThinStateDiff`, `state_diff_commitment`, and `PartialBlockHashComponents` are all computed over this corrupted execution output, yielding a wrong block commitment that validators sign and commit to storage.

This matches the **Critical** impact: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

### Likelihood Explanation

Any consensus participant that can act as a proposer for a single block can trigger this. No privileged access is required beyond normal proposer rights. The attacker only needs to know the hash of a previously-rejected L1 handler transaction (observable from on-chain L1 events and L2 block history) and include it in their proposal stream. The validator-side acceptance is deterministic and requires no race condition.

### Recommendation

Add `Rejected` to the non-validatable states in `is_validatable`:

```rust
pub fn is_validatable(&self) -> bool {
    !self.is_committed()
        && !self.is_cancelled()
        && !self.is_consumed()
        && !self.is_rejected()   // add this
}
```

and add a corresponding `is_rejected` helper:

```rust
pub fn is_rejected(&self) -> bool {
    matches!(self.state, TransactionState::Rejected)
}
```

Also add `TransactionState::Rejected => InvalidValidationStatus::AlreadyRejectedOnL2.into()` (or a suitable variant) to the `match` arm inside `validate_tx` so the `unreachable!()` macro is not silently bypassed. [1](#0-0) 

### Proof of Concept

1. L1 handler transaction `T` is scraped from L1; state = `Pending`, present in `proposable_index`.
2. Honest proposer includes `T` in block `N`. Blockifier execution reverts (e.g., `OutOfRangeContractAddress`). `T` appears in both `consumed_l1_handler_tx_hashes` and `rejected_tx_hashes`.
3. Batcher calls `commit_block(consumed={T}, rejected={T}, height=N)`. Inside `apply_commit_block`, `T` is partitioned into `rejected_and_consumed`; `commit_txs` calls `mark_rejected(T)` → state = `Rejected`. `maintain_indices` removes `T` from `proposable_index`.
4. For block `N+1`, the malicious proposer streams a proposal that includes `T` in its transaction stream.
5. Each validator's `ValidateTransactionProvider::get_txs` receives `T` and calls `l1_provider_client.validate(T.tx_hash, N+1)`.
6. `validate_tx` evaluates `is_validatable()` for `T`: `!Committed && !CancelledOnL2 && !Consumed` → `true`. `try_mark_staged` succeeds → returns `ValidationStatus::Validated`.
7. `T` is forwarded to the blockifier and re-executed. The resulting state diff and block commitment for block `N+1` are wrong.

The existing unit test at line 502 of `l1_provider_tests.rs` already demonstrates step 6 in isolation — `validate_rejected_transaction` asserts `ValidationStatus::Validated` for a `Rejected` tx, confirming the path is reachable without any special setup. [7](#0-6)
