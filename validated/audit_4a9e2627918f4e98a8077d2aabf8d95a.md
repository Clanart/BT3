### Title
Deploy Syscall Unconditionally Asserts Constructor Success, Panicking on Revert Instead of Returning Failure Response — (`execution/deploy_contract.cairo`)

---

### Summary

The `deploy_contract` function in the StarkNet OS unconditionally asserts that the constructor does not revert (`assert is_reverted = 0`). If a constructor reverts, the OS panics rather than writing a graceful failure response. Because the sequencer (blockifier) handles constructor reverts gracefully and can include such transactions in a block, the resulting block becomes unprovable, halting the network.

---

### Finding Description

In `deploy_contract.cairo`, after invoking the constructor via `select_execute_entry_point_func`, the OS checks the return value `is_reverted` and then unconditionally asserts it is zero:

```cairo
// Invoke the contract constructor.
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);
...
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
return (retdata_size=retdata_size, retdata=retdata);
``` [1](#0-0) 

If `is_reverted == 1` (constructor reverted), the Cairo `assert` fails, causing an OS-level panic. This is not a graceful failure — it is an unrecoverable abort of the entire OS execution.

The caller, `execute_deploy` in `syscall_impls.cairo`, compounds this by hardcoding `failure_flag=0` in the response header with an explicit TODO acknowledging the limitation:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [2](#0-1) 

This is the direct analog to M-07: just as the Kakarot `account_contract.cairo` used a panicking `call_contract` syscall instead of a recoverable one, the StarkNet OS `deploy_contract` uses a panicking `assert` instead of writing a `failure_flag=1` response.

Contrast this with `contract_call_helper`, which correctly handles reverts by writing a failure response:

```cairo
if (is_reverted != FALSE) {
    // Append `ERROR_ENTRY_POINT_FAILED` to the retdata.
    assert retdata[retdata_size] = ERROR_ENTRY_POINT_FAILED;
    ...
}
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The `deploy` syscall path has no equivalent graceful handling.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

The sequencer (blockifier) handles constructor reverts gracefully: it returns a `failure_flag=1` deploy response and may allow the calling transaction to continue or revert cleanly. The sequencer will include such a transaction in a block.

When the OS attempts to prove that block, it reaches `assert is_reverted = 0` in `deploy_contract.cairo` and panics. The block becomes unprovable. Since the sequencer and OS disagree on how to handle this case, any block containing a transaction with a reverting constructor deploy is permanently stuck — it cannot be proven, and the network cannot advance past it.

---

### Likelihood Explanation

**High.** The attack requires no privileged access:

1. An attacker declares a contract class whose constructor always reverts (e.g., `assert 1 = 0`).
2. The attacker submits an invoke transaction that calls the `DEPLOY_SELECTOR` syscall with that class hash.
3. The sequencer executes the transaction: constructor reverts → deploy syscall fails → transaction is included in the block (as a reverted deploy or a reverted transaction, depending on how the caller handles the failure).
4. The OS attempts to prove the block → `assert is_reverted = 0` panics → block is unprovable → network halt.

This is a standard, low-cost transaction that any unprivileged user can submit.

---

### Recommendation

Replace the unconditional `assert is_reverted = 0` in `deploy_contract` with proper failure propagation, mirroring the pattern used in `contract_call_helper`:

- Return `is_reverted` from `deploy_contract` to `execute_deploy`.
- In `execute_deploy`, write `ResponseHeader(gas=remaining_gas, failure_flag=is_reverted)` instead of hardcoding `failure_flag=0`.
- Append `ERROR_ENTRY_POINT_FAILED` to the retdata on revert, consistent with `contract_call_helper`. [4](#0-3) 

---

### Proof of Concept

1. Declare a class with constructor body `assert 1 = 0` (always reverts).
2. Submit an invoke transaction from any account that calls `deploy(class_hash=<above>, ...)`.
3. Sequencer includes the transaction in a block (constructor revert is handled gracefully by the blockifier).
4. OS proving run reaches `deploy_contract.cairo:91` → `assert is_reverted = 0` fails → OS panics → block is unprovable → network halt.
