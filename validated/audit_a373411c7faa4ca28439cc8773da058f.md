### Title
Timelock Validation Bypass via Integer Overflow in `wrapping_add` Path — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary

`check_time_locks` uses `wrapping_add` (instead of `saturating_add`) for relative timelock arithmetic when `nowrap=false`. An attacker can craft a spend with a near-maximum `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` condition value that wraps around to a small integer, causing the timelock check to pass when it should fail — bypassing the timelock entirely.

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic paths for relative timelock validation:

- `nowrap=true` → `saturating_add`: overflow saturates to `u32::MAX` / `u64::MAX`, always failing the check correctly.
- `nowrap=false` → `wrapping_add`: overflow wraps around to a small value, which can be less than the current height/timestamp, causing the check to incorrectly pass.

The four affected checks are:

**`height_relative` (lines 56–68):**
```rust
if nowrap {
    if prev_transaction_block_height < unspent.confirmed_block_index.saturating_add(height_relative) { ... }
} else if prev_transaction_block_height < unspent.confirmed_block_index.wrapping_add(height_relative) { ... }
```

**`seconds_relative` (lines 70–78):**
```rust
if nowrap {
    if timestamp < unspent.timestamp.saturating_add(seconds_relative) { ... }
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) { ... }
```

**`before_height_relative` (lines 79–98)** and **`before_seconds_relative` (lines 100–112)** have the symmetric inverse problem: `wrapping_add` causes a large "before" deadline to wrap to a small value, incorrectly rejecting a valid spend.

The test suite explicitly documents the divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),  // <-- passes when it should fail
)]
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),  // <-- passes when it should fail
)]
``` [1](#0-0) [2](#0-1) 

### Impact Explanation

This is a **timelock validation bypass**. When `nowrap=false`:

- A coin confirmed at block `B` with `ASSERT_HEIGHT_RELATIVE = (u32::MAX - B + k)` for small `k` will have `B.wrapping_add(u32::MAX - B + k) = k - 1`, which is less than any current block height. The timelock check passes immediately, allowing the coin to be spent at any block height regardless of the intended lock period.
- The same applies to `ASSERT_SECONDS_RELATIVE` with a near-`u64::MAX` value.

This matches the **High** impact: *timelock condition validation bypass enables unauthorized spend acceptance*. [3](#0-2) 

### Likelihood Explanation

The `nowrap` parameter is caller-controlled and is exposed directly through the Python binding `py_check_time_locks`. Any full-node code calling `check_time_locks` with `nowrap=false` (the legacy path) is vulnerable. The attacker only needs to submit a spend bundle with a crafted `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` condition value near `u32::MAX` / `u64::MAX`. No privileged access is required — this is a standard unprivileged spend. [4](#0-3) 

### Recommendation

Remove the `wrapping_add` path entirely. The `nowrap=false` branch is semantically incorrect for timelock validation. Replace all four `wrapping_add` calls with `saturating_add` unconditionally, or remove the `nowrap` parameter and always use `saturating_add`. If backward compatibility with old chain state is required, document precisely which blocks were validated with `wrapping_add` and gate the old path on a specific block height rather than a caller-supplied boolean. [5](#0-4) 

### Proof of Concept

Given:
- Coin confirmed at block index `100`
- Current `prev_transaction_block_height = 200`
- Attacker sets `ASSERT_HEIGHT_RELATIVE = 0xffff_ffff` (u32::MAX)

With `nowrap=false`:
```
100u32.wrapping_add(0xffff_ffff) = 99
200 < 99  →  false  →  check passes (timelock bypassed)
```

With `nowrap=true`:
```
100u32.saturating_add(0xffff_ffff) = 0xffff_ffff (u32::MAX)
200 < 0xffff_ffff  →  true  →  check fails (correct)
```

The attacker submits a spend bundle with `ASSERT_HEIGHT_RELATIVE = u32::MAX - confirmed_block_index + k` for any small `k ≥ 0`. The wrapping result is `k`, which is always less than any real block height, so the timelock is bypassed unconditionally. [6](#0-5)
