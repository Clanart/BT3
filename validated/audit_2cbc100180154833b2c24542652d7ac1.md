### Title
Missing Zero-Check on `class_hash` in `execute_replace_class` Allows Permanent Freezal of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall handler accepts a caller-supplied `class_hash` of `0` without validation. A contract can invoke `replace_class(0)`, permanently setting its own class hash to zero in the committed state. Any subsequent call to that contract causes the OS to attempt a `find_element` lookup for compiled class hash `0`, which panics at the prover level because no compiled class with hash `0` exists. All funds held in the affected contract are permanently frozen.

### Finding Description
In `execute_replace_class`, the `class_hash` field is read directly from the syscall request and written into the contract's `StateEntry` without any non-zero assertion:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-inserted TODO comment explicitly acknowledges the missing validation. [1](#0-0) 

When a subsequent transaction calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // = 0
);
// compiled_class_hash = 0 (dict default for missing key)

let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,           // = 0
);
```

`find_element` with key `0` panics at the prover level because no compiled class fact with hash `0` is registered. [2](#0-1) 

### Impact Explanation
**Critical — Permanent freezing of funds.**

Once `replace_class(0)` is committed to state, the contract's `StateEntry.class_hash` is `0`. Every future call to that contract causes an OS-level prover panic (not a graceful revert), making the block unprovable if it includes such a call. The sequencer must permanently exclude all transactions targeting the contract. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are irrecoverably locked with no upgrade or rescue path.

### Likelihood Explanation
The `replace_class` syscall is callable by any deployed contract on itself — no privileged role is required. A malicious contract deployer can intentionally trigger this in a single transaction. The missing check is confirmed by the in-code TODO comment, indicating the developers are aware the validation is absent.

### Recommendation
Add an `assert_not_zero` guard on `class_hash` immediately after reading it from the request, analogous to the pattern already used elsewhere in the same file and in `deploy_contract.cairo`:

```cairo
let class_hash = request.class_hash;
assert_not_zero(class_hash);   // <-- add this
```

Additionally, once the TODO is resolved, verify that `class_hash` corresponds to a previously declared class by checking its presence in `contract_class_changes` before committing the state update.

### Proof of Concept

1. Attacker deploys `MaliciousContract` holding user funds (e.g., acts as a vault).
2. `MaliciousContract` internally calls the `replace_class` syscall with `class_hash = 0`.
3. `execute_replace_class` writes `StateEntry(class_hash=0, ...)` to `contract_state_changes` with no validation. [3](#0-2) 
4. The transaction succeeds and is committed; the contract's class hash is now `0` in the global state.
5. Any future transaction invoking `MaliciousContract` reaches `execute_entry_point`, which calls `dict_read` for key `0` (returning default `0`), then calls `find_element` with key `0`. [2](#0-1) 
6. `find_element` panics — no compiled class fact with hash `0` exists — making the block unprovable.
7. The sequencer permanently excludes all calls to `MaliciousContract`; all funds are frozen with no recovery mechanism.
