Audit Report

## Title
Residual ETH in Router Silently Subsidizes Subsequent WETH Swaps, Stealing Prior User's Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` contains a partial-ETH branch that wraps **all** native ETH currently held by the router when `0 < address(this).balance < value`, without verifying that the ETH belongs to the current caller. Because the router is `payable` at multiple entry points and accumulates ETH from users who omit `refundETH()`, a subsequent WETH swap silently consumes that stranded ETH, permanently transferring it to the pool on behalf of a different user.

## Finding Description

The root cause is in `pay` at lines 73–84: [1](#0-0) 

The middle branch (`nativeBalance > 0 && nativeBalance < value`) wraps **all** native ETH in the router — regardless of who deposited it — and sends it to the pool, then pulls only `value - nativeBalance` from the current payer's WETH allowance. There is no check that `nativeBalance` was contributed by the current caller in the current transaction.

ETH accumulates in the router because both `multicall` and `exactInputSingle` are `payable`: [2](#0-1) [3](#0-2) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on these `payable` entry points: [4](#0-3) 

**Exploit flow:**

1. **User A** calls `multicall{value: 1 ether}([exactInputSingle(tokenIn=WETH, amountIn=0.3 ether)])` without appending `refundETH()`. The swap consumes 0.3 ETH; 0.7 ETH remains stranded in the router.
2. **User B** calls `exactInputSingle(tokenIn=WETH, amountIn=1 ether)` (no ETH sent).
3. The pool callback triggers `pay(WETH, userB, pool, 1 ether)`.
4. `nativeBalance = 0.7 ETH` → middle branch: wraps 0.7 ETH (User A's), sends to pool; pulls only 0.3 WETH from User B.
5. Pool receives 1 ETH (correct), but User A's 0.7 ETH is permanently consumed with no recourse.

## Impact Explanation

User A suffers a direct, irreversible loss of principal (stranded ETH) through a public, unprivileged swap path. User B receives a discount funded by User A's ETH. The pool itself is not insolvent, but User A's funds are unrecoverably redirected. This constitutes a direct loss of user principal meeting the High impact threshold.

## Likelihood Explanation

The pattern of `multicall{value}(swap)` without `refundETH()` is a common integration mistake explicitly anticipated by the test suite: [5](#0-4) 

Any user who sends excess ETH and omits `refundETH()` — or whose transaction reverts after ETH is already in the router — creates the vulnerable state. An attacker can monitor the mempool for stranded ETH or simply be the next WETH swapper. No special privileges are required.

## Recommendation

In the partial-ETH branch, only consume native ETH that was sent by the current caller in the current transaction. Track `msg.value` in transient storage at each `payable` entry point and cap the native ETH consumed in `pay` to that amount. Alternatively, revert if `address(this).balance > msg.value` at entry, ensuring no cross-user ETH leakage is possible.

## Proof of Concept

```
1. User A: router.multicall{value: 1 ether}([exactInputSingle(tokenIn=WETH, amountIn=0.3 ether)])
   → pay() sees nativeBalance=1 ETH >= value=0.3 ETH → wraps 0.3 ETH, sends to pool
   → router.balance = 0.7 ETH (stranded, no refundETH call)

2. User B: router.exactInputSingle(tokenIn=WETH, amountIn=1 ether)
   → pay(WETH, userB, pool, 1 ether)
   → nativeBalance=0.7 ETH > 0, < 1 ETH → middle branch
   → wraps 0.7 ETH (User A's), sends to pool
   → safeTransferFrom(userB, pool, 0.3 WETH)
   → pool receives 1 ETH ✓, but userB paid only 0.3 WETH
   → User A loses 0.7 ETH with no recourse
```
