### Title
Unconstrained `previous_block_hash` in `get_block_hashes` Allows Zero-Parent Block Hash to Produce a Valid Proof — (`File: block_hash.cairo`)

---

### Summary

In `get_block_hashes` (`block_hash.cairo`), the `previous_block_hash` field is populated entirely by a prover-controlled hint with no Cairo-level non-zero assertion for non-genesis blocks. A malicious sequencer/prover can supply `previous_block_hash = 0`, produce a cryptographically valid proof, and have the OS emit an `OsOutputHeader` with `prev_block_hash = 0`. This breaks the hash-chain linkage between consecutive blocks, constituting an unintended chain split.

---

### Finding Description

In `get_block_hashes` (`block_hash.cairo` lines 55–82), `previous_block_hash` is declared as a `local` variable and filled exclusively by the hint `%{ GetBlockHashes %}`:

```cairo
local previous_block_hash;
...
%{ GetBlockHashes %}

let block_hash = calculate_block_hash(
    ...
    previous_block_hash=previous_block_hash,
    ...
);

%{ CheckBlockHashConsistency %}
```

Hints in Cairo are prover-side Python code. They are **not** Cairo constraints and are **not** enforced in the proof. A malicious prover can set `previous_block_hash = 0` and skip or override `CheckBlockHashConsistency` entirely; the resulting proof is still valid because no `assert_not_zero(previous_block_hash)` exists in Cairo.

The computed `new_block_hash` (which includes `previous_block_hash` as an input via `calculate_block_hash`) is then placed into `OsOutputHeader.prev_block_hash` and `OsOutputHeader.new_block_hash` and serialized to the output segment via `serialize_output_header`. The OS developers themselves acknowledge this gap with an explicit comment:

> *"NOTE: both the previous block hash and previous state root are guessed, and the OS does not verify their consistency (unlike the new hash and root). The consumer of the OS output should verify both."*

This is the direct structural analog of the Taiko M-09 bug: just as Taiko's `proposeBlock` omits a `parentMetaHash != 0` guard outside genesis, the StarkNet OS omits an `assert_not_zero(previous_block_hash)` guard outside genesis.

A secondary, compounding issue exists in `write_block_number_to_block_hash_mapping` (`os_utils.cairo` line 66–73): `old_block_hash` is also hint-provided with no Cairo constraint on its value, and the comment explicitly states *"Currently, the block hash mapping is not enforced by the OS."* A malicious sequencer can write an arbitrary (including zero) value into the block-number→block-hash mapping contract, corrupting the data later consumed by `read_block_hash_from_storage` and `check_proof_facts`.

---

### Impact Explanation

**High — Unintended chain split.**

If a sequencer produces a valid proof with `previous_block_hash = 0`:

1. The OS output carries `prev_block_hash = 0` and a `new_block_hash` computed over a zero parent.
2. The L1 verifier, which trusts the OS proof output, records this block as accepted.
3. All subsequent blocks that correctly chain off the *real* previous block hash diverge from the accepted on-chain state.
4. The network is partitioned: nodes following the correct chain disagree with the L1-accepted state, constituting a chain split.

Additionally, because `old_block_hash` in `write_block_number_to_block_hash_mapping` is unconstrained, the block-hash mapping contract can be poisoned with wrong values. `check_proof_facts` in `execution_constraints.cairo` reads from this mapping to validate virtual OS proof facts; corrupted mapping values undermine that validation path.

---

### Likelihood Explanation

The sequencer in StarkNet controls proof generation. The sequencer is the entity that runs the Cairo prover and supplies all hints. Since `previous_block_hash` is entirely hint-provided with no Cairo constraint, a malicious or compromised sequencer can exploit this without any external dependency, leaked key, or privileged access beyond their normal sequencer role. The entry path is direct and requires no special conditions.

---

### Recommendation

Add a Cairo-level non-zero assertion on `previous_block_hash` for all blocks beyond genesis inside `get_block_hashes`, analogous to the mitigation recommended in the Taiko M-09 report:

```cairo
// After %{ GetBlockHashes %}
if (block_info.block_number != 0) {
    assert_not_zero(previous_block_hash);
}
```

Similarly, in `write_block_number_to_block_hash_mapping`, enforce that `old_block_hash` is non-zero and matches the committed block hash output (resolving the existing TODO at line 65).

The `CheckBlockHashConsistency` hint should be replaced or supplemented with a Cairo `assert` that binds `previous_block_hash` to a verifiable on-chain commitment, removing reliance on the consumer to perform a check the OS itself should enforce.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Malicious sequencer prepares a block at `block_number > 0`.
2. During proof generation, the hint `%{ GetBlockHashes %}` is overridden to set `previous_block_hash = 0`.
3. `calculate_block_hash` computes `new_block_hash` with `previous_block_hash = 0` as input — no Cairo assertion fires.
4. `get_block_os_output_header` (`os_utils.cairo` line 132–134) calls `get_block_hashes` and places the result into `OsOutputHeader` with `prev_block_hash = 0`.
5. `serialize_output_header` (`output.cairo` line 165) writes `prev_block_hash = 0` to the proof output segment.
6. The proof verifies successfully on L1; the accepted block is not linked to its true parent.
7. The chain hash-linkage is broken; a split results between nodes tracking the real chain and the L1-accepted state.
