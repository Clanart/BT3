### Title
Tautological Self-Validation in `ProofOfInclusion::valid()` Allows Forged Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` contains a tautological final check that makes it only verify internal self-consistency of the proof structure, never against a trusted external root hash. Any attacker-crafted `ProofOfInclusion` with an arbitrary `node_hash` and internally consistent `combined_hash` chain will pass `valid()`, allowing forged proof of inclusion for any key/value pair.

### Finding Description

The `valid()` method in `ProofOfInclusion` is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← always true
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**The tautology:** After the loop completes without returning `false`, `existing_hash` holds `calculated_hash` from the last iteration. The loop body already asserted `calculated_hash == layer.combined_hash`, so `existing_hash == last.combined_hash`. But `root_hash()` also returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is **unconditionally true** whenever the loop completes.

The function verifies only that each layer's `combined_hash` is consistent with the previous layer's hash — it never compares the final accumulated hash against any externally trusted root. A correct implementation would require a trusted root to be passed in and compared against `existing_hash` at the end.

The struct is `Streamable` (deserializable from untrusted bytes) and is fully exposed via Python bindings with `py_valid` and `py_root_hash`: [3](#0-2) 

The struct carries `#[derive(Streamable)]` and `from_py_object`, meaning it can be constructed entirely from attacker-controlled bytes: [4](#0-3) 

### Impact Explanation

An attacker can craft a `ProofOfInclusion` with:
- An arbitrary `node_hash` (claiming any key/value hash is in the tree)
- A chain of `layers` where each `combined_hash` is computed correctly from the previous hash and a chosen `other_hash`

Calling `valid()` on this forged proof returns `true`. Any DataLayer consumer that uses `proof.valid()` as the sole check for proof validity — without separately verifying `proof.root_hash() == trusted_root` — will accept the forged proof as genuine inclusion evidence.

This matches the allowed impact: **High — DataLayer Merkle proof logic lets untrusted input prove invalid state (forged inclusion).**

### Likelihood Explanation

- `ProofOfInclusion` is `Streamable` and exposed via Python bindings, making it directly constructable from attacker-controlled bytes.
- The method name `valid()` strongly implies it performs complete proof validation, making it likely that callers treat it as sufficient.
- No existing call site within chia_rs passes a trusted root to compare against; the fuzz target and all tests call `proof.valid()` alone: [5](#0-4) 

### Recommendation

Change `valid()` to accept a trusted root hash parameter and compare `existing_hash` against it at the end, rather than against `self.root_hash()`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external trusted root
}
```

Remove or deprecate the current `valid()` / `py_valid()` API to prevent misuse.

### Proof of Concept

```python
from chia_rs import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

# Forge a proof claiming node_hash = FAKE_HASH is in the tree
FAKE_HASH = bytes([0xAA] * 32)
OTHER_HASH = bytes([0xBB] * 32)

# Compute a consistent combined_hash for one layer
def calc_internal(left, right):
    h = hashlib.sha256()
    h.update(b'\x00' * 30 + b'\x00\x02')  # Mid prefix
    h.update(left)
    h.update(right)
    return h.digest()

combined = calc_internal(FAKE_HASH, OTHER_HASH)

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=OTHER_HASH,
    combined_hash=combined,
)
proof = ProofOfInclusion(node_hash=FAKE_HASH, layers=[layer])

# valid() returns True even though this proof was never generated
# from any real MerkleBlob and FAKE_HASH is not in any tree
assert proof.valid()  # passes — forged proof accepted
# proof.root_hash() == combined, but no real tree has this root
```

The final `existing_hash == self.root_hash()` check in `valid()` is always satisfied because both sides are derived from the same `combined_hash` field inside the proof itself. [1](#0-0)
