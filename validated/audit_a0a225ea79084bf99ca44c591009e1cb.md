### Title
Unprotected `new` Initializer Allows Anyone to Seize DAO/Admin Control of the NEAR Omni Bridge - (File: near/omni-bridge/src/lib.rs)

### Summary

The `Contract::new` initializer in `near/omni-bridge/src/lib.rs` contains no caller restriction. Because the deployment script deploys the contract code **without** an init call in a separate step, any observer can front-run the deployer's initialization transaction and become the permanent DAO super-admin of the bridge.

### Finding Description

The NEAR `OmniBridge` contract's `#[init]` function grants the `DEFAULT_ADMIN_ROLE` (super-admin) and `Role::DAO` unconditionally to `env::predecessor_account_id()`:

```rust
// near/omni-bridge/src/lib.rs  lines 285-314
#[init]
pub fn new(mpc_signer: AccountId, wnear_account_id: AccountId) -> Self {
    let mut contract = Self { ... };
    contract.acl_init_super_admin(near_sdk::env::predecessor_account_id());
    contract.acl_grant_role(Role::DAO.into(), near_sdk::env::predecessor_account_id());
    contract
}
``` [1](#0-0) 

There is no check that the caller is the contract's own account or any trusted deployer. The NEAR SDK's `#[init]` attribute only prevents a second call once state is written; it does not restrict who makes the first call.

The deployment script explicitly separates code deployment from initialization:

```bash
# Deploy WITHOUT init call
near contract deploy "$CONTRACT_ID" use-file "$WASM_PATH" \
    without-init-call ...
# Then separately call init
near contract call-function as-transaction "$CONTRACT_ID" "$INIT_FUNCTION" ...
``` [2](#0-1) 

This creates a race-condition window between the two transactions. An attacker monitoring the NEAR chain can detect the `deploy` transaction and immediately submit a `new(attacker_mpc_signer, attacker_wnear)` call, winning the initialization race.

The same pattern exists in `near/token-deployer/src/lib.rs`:

```rust
#[init]
pub fn new(controller: AccountId, dao: AccountId, global_code_hash: Base58CryptoHash) -> Self {
    let mut contract = Self { ... };
    contract.acl_init_super_admin(near_sdk::env::predecessor_account_id());
    contract.acl_grant_role(Role::DAO.into(), dao.clone());
    contract.acl_grant_role(Role::Controller.into(), controller);
    contract.acl_transfer_super_admin(dao);
    contract
}
``` [3](#0-2) 

By contrast, `near/omni-token/src/lib.rs` correctly protects its `new` function by requiring the predecessor to be the parent (deployer) account:

```rust
require!(
    env::predecessor_account_id().as_str() == deployer_account,
    "Only the deployer account can init this contract"
);
``` [4](#0-3) 

The EVM contracts (`OmniBridge.sol`, `BridgeToken.sol`, `ENearProxy.sol`) are **not** affected because they call `_disableInitializers()` in their constructors and use OpenZeppelin's `initializer` modifier, and the NEAR prover contracts (`EvmProver`, `WormholeOmniProverProxy`) are protected by `#[private]`. [5](#0-4) [6](#0-5) 

### Impact Explanation

An attacker who wins the initialization race on `OmniBridge.new()` becomes the permanent DAO super-admin. With that role they can:

1. Call `set_prover` / `add_factory` to register malicious prover and factory accounts that accept fabricated cross-chain proofs.
2. Call `fin_transfer` with fake proofs to mint or release bridged tokens on NEAR without any corresponding lock on the source chain — **direct unauthorized minting of bridged assets**.
3. Upgrade the contract (via `UpgradableCodeDeployer` / `DAO` roles) to arbitrary malicious code, enabling complete fund drainage.
4. Permanently freeze the bridge by pausing all operations, making all locked user funds irrecoverable.

This satisfies the **Critical** impact class: *Direct theft, unauthorized release, or unauthorized mint of native or bridged assets* and *Permanent freezing / irrecoverable lock of user or protocol funds*.

### Likelihood Explanation

NEAR transactions are publicly visible in the mempool. A bot watching for `deploy` transactions targeting the known bridge contract account can immediately submit a competing `new` call. The two-step deployment pattern in the official deployment script guarantees the window exists on every deployment. The attack requires no special privilege, no leaked key, and no colluding party — only the ability to submit a NEAR transaction.

### Recommendation

1. **Combine deploy and init in one transaction.** Use `near contract deploy ... with-init-call` so no window exists between code upload and initialization.
2. **Add a deployer check inside `new`.** Mirror the pattern already used in `OmniToken`:

```rust
#[init]
pub fn new(mpc_signer: AccountId, wnear_account_id: AccountId) -> Self {
    require!(
        env::predecessor_account_id() == env::current_account_id(),
        "Only the contract account can initialize"
    );
    // ... rest of init
}
```

Apply the same fix to `TokenDeployer::new` in `near/token-deployer/src/lib.rs`.

### Proof of Concept

1. Deployer broadcasts: `near contract deploy omni-bridge.near use-file omni_bridge.wasm without-init-call ...`
2. Attacker observes the transaction on-chain and immediately broadcasts:
   `near contract call-function as-transaction omni-bridge.near new json-args '{"mpc_signer":"attacker.near","wnear_account_id":"wrap.near"}' sign-as attacker.near ...`
3. Attacker's `new` call is processed first (or in the same block before the deployer's init call).
4. `acl_init_super_admin(attacker.near)` and `acl_grant_role(DAO, attacker.near)` execute.
5. Deployer's subsequent `new` call panics (`Contract already initialized`).
6. Attacker, now holding `Role::DAO`, registers a malicious prover, submits a fabricated `fin_transfer` proof, and mints arbitrary bridged tokens to their own account — or upgrades the contract to drain all locked funds. [1](#0-0) [3](#0-2)
