### Title
`meta_tx_v0` Signature Replay: Missing Nonce and Block Binding in Hash Allows Indefinite Re-execution of Signed Calldata — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a synthetic version-0 transaction whose hash (`compute_meta_tx_v0_hash`) omits any nonce, block number, or timestamp. The OS also explicitly skips both the `__validate__` entry point and the nonce increment for version-0 transactions. As a result, any meta-tx-v0 signature is permanently valid and can be replayed by any unprivileged caller across any future block, enabling direct loss of funds from victim accounts.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` builds a new `TxInfo` with `version=0` and `nonce=0` hardcoded: [1](#0-0) 

The transaction hash is computed by `compute_meta_tx_v0_hash`, which calls `deprecated_get_transaction_hash` with `additional_data_size=0` — meaning **no nonce is included in the hash**: [2](#0-1) 

The hash covers only `(INVOKE_HASH_PREFIX, version=0, contract_address, entry_point_selector, calldata_hash, max_fee=0, chain_id)`. There is no block number, timestamp, or nonce component.

Two OS-level replay defenses are then both bypassed for version 0:

1. **`run_validate` is skipped** — the `__validate__` entry point (signature verification) is not called for version-0 transactions: [3](#0-2) 

2. **`check_and_increment_nonce` is skipped** — the account nonce is never checked or incremented: [4](#0-3) 

The syscall is invoked via `execute_meta_tx_v0` which is reachable from any contract through the `META_TX_V0_SELECTOR` syscall dispatch: [5](#0-4) 

---

### Impact Explanation

An attacker who observes a valid `(contract_address, calldata, signature)` tuple from any prior meta-tx-v0 execution can replay it in any future block. Because the OS presents `nonce=0` to the target account's `__execute__` every time and never calls `__validate__`, the account contract has no OS-enforced mechanism to distinguish a fresh invocation from a replay. Any value-moving calldata (token transfers, approvals, withdrawals) signed once can be re-executed indefinitely, constituting **direct loss of funds**.

**Impact: Critical — Direct loss of funds.**

---

### Likelihood Explanation

The `meta_tx_v0` syscall is callable by any unprivileged contract deployed on StarkNet. An attacker only needs to:
1. Observe a meta-tx-v0 signature on-chain (it is part of the transaction calldata, fully public).
2. Deploy a contract that issues the same `MetaTxV0Request` with the captured signature.

No privileged access, leaked key, or trusted role is required. The signature is permanently valid because the hash is stateless with respect to block and nonce.

---

### Recommendation

1. **Include a nonce in `compute_meta_tx_v0_hash`**: Pass the target account's current on-chain nonce as `additional_data` (analogous to how `compute_l1_handler_transaction_hash` passes the L1 nonce), and verify + increment it in the OS before execution.
2. **Include a block-number or timestamp expiry field** in the meta-tx hash so that signatures become invalid after a bounded window.
3. **Call `__validate__`** for meta-tx-v0 executions, or enforce that the OS checks the nonce against the account's state entry before dispatching `__execute__`.

---

### Proof of Concept

**Step 1.** Alice signs a meta-tx-v0 authorizing a 1000-token transfer from her account. The signed hash is:
```
H(INVOKE_PREFIX, 0, alice_addr, __execute__, calldata_hash, 0, chain_id)
```
No nonce, no block number.

**Step 2.** The transaction is included in block N. The signature `sig_alice` is visible in the transaction calldata.

**Step 3.** Attacker deploys `ReplayContract` with an `__execute__` that issues:
```cairo
syscall_ptr.meta_tx_v0(
    contract_address = alice_addr,
    selector         = EXECUTE_ENTRY_POINT_SELECTOR,
    calldata         = <original transfer calldata>,
    signature        = sig_alice,   // captured from block N
)
```

**Step 4.** The OS calls `execute_meta_tx_v0`. It recomputes the same hash (no nonce in hash), skips `__validate__`, skips nonce increment, and dispatches `alice_addr.__execute__` with `tx_info.nonce = 0` and `tx_info.transaction_hash = original_hash`.

**Step 5.** Alice's `__execute__` receives a `TxInfo` that looks identical to the original. If the account does not maintain its own storage-based replay counter (most standard account contracts do not for meta-tx paths), the transfer executes again.

**Step 6.** Attacker repeats Step 3–5 in every subsequent block until Alice's balance is drained. [6](#0-5) [7](#0-6)
