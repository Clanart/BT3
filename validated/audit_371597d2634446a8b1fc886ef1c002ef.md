### Title
Integer Overflow via `wrapping_add` in Relative Timelock Validation Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Conditions — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap = false` (the pre-soft-fork consensus path), the relative timelock comparisons for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` use `wrapping_add` instead of `saturating_add`. An attacker-controlled condition value near the type maximum causes the addition to wrap around to a small integer, making the "not yet mature" check evaluate to `false` and allowing the coin to be spent immediately — bypassing the intended timelock entirely.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` flag for every relative condition:

```rust
// lines 55-68
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { ... }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { ... }
}
``` [1](#0-0) 

The same pattern applies to `seconds_relative` (lines 70–78): [2](#0-1) 

When `nowrap = false`, `wrapping_add` is used. For a coin confirmed at block index `B`, if a puzzle outputs `ASSERT_HEIGHT_RELATIVE` with value `V` such that `B + V > u32::MAX`, the sum wraps to a small integer `W`. The guard `prev_height < W` evaluates to `false` (since `prev_height >= B > W`), so no error is returned and the coin is accepted as spendable — even though the intended timelock has not elapsed.

The same overflow applies to `ASSERT_SECONDS_RELATIVE` with `u64` arithmetic: [3](#0-2) 

The tests explicitly confirm this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [4](#0-3) 

The `nowrap` parameter is passed from the Python caller (`py_check_time_locks`) and is exposed as a public API: [5](#0-4) 

The Python type stub confirms the parameter is caller-controlled: [6](#0-5) 

---

### Impact Explanation

**High — Timelock condition validation bypass enables unauthorized spend acceptance.**

A CLVM puzzle that outputs `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a value large enough to cause `wrapping_add` overflow will have its timelock silently bypassed when the block is validated under `nowrap = false`. The coin is accepted as spendable at any block height, regardless of the intended lock period. This matches the allowed High impact: *"timelock... validation bypass enables unauthorized spend acceptance."*

The inverse wrapping case for `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` (lines 79–112) causes the opposite: a valid spend is incorrectly rejected. While this is a correctness defect, it does not enable unauthorized spend acceptance and falls outside the allowed impact scope. [7](#0-6) 

---

### Likelihood Explanation

**Low.** The overflow only triggers for condition values near the type maximum (`u32::MAX ≈ 4.3 × 10⁹` blocks, `u64::MAX ≈ 5.8 × 10¹¹ years`). No legitimate puzzle uses such values. The `nowrap = true` path (saturating arithmetic) was introduced as a soft-fork fix; once that soft-fork activates for all new blocks, the wrapping path is only exercised when re-validating historical blocks. However, the `nowrap = false` path remains live in the codebase and is reachable via the Python binding with an attacker-supplied spend bundle on any node that has not yet activated the soft-fork.

---

### Recommendation

1. **Remove the `wrapping_add` path entirely** once the soft-fork has activated on all supported networks. Replace both branches with `saturating_add` unconditionally, or gate the wrapping path behind a compile-time or runtime constant tied to the activation height rather than a caller-supplied boolean.
2. **Validate the `nowrap` flag at the call site** in the Python full node to ensure it is derived solely from the block height and consensus constants, never from untrusted input.
3. **Add a condition-parsing guard** that rejects `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` values that would overflow when added to any plausible `confirmed_block_index`, treating them as `PositiveOverflow` (already done for values exceeding the type width, but not for values that overflow only in context).

---

### Proof of Concept

Craft a CLVM puzzle that outputs:

```
(ASSERT_HEIGHT_RELATIVE . 0xFFFFFFFF)
```

Create a coin with this puzzle confirmed at block index `B = 1`. Submit a spend bundle at any block height. When `check_time_locks` is called with `nowrap = false`:

```
confirmed_block_index.wrapping_add(height_relative)
= 1u32.wrapping_add(0xFFFF_FFFFu32)
= 0u32
```

The guard `prev_height < 0u32` is always `false`, so no `AssertHeightRelativeFailed` error is returned. The spend is accepted at block 2, 3, or any height — the 4-billion-block timelock is fully bypassed.

The test vector at line 277–281 of `check_time_locks.rs` already demonstrates this outcome (`Ok(())` for the `nowrap = false` branch): [8](#0-7)

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,

        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
