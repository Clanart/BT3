### Title
Fixed Gas Budget for L1 Handler Execution May Permanently Freeze Bridged Funds — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` assigns a hardcoded, fixed gas budget of `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100,000,000` to every L1 handler execution. Critically, the L1→L2 message is **consumed before execution begins**. If a Sierra-gas-mode L1 handler exhausts this budget, the execution fails, state changes are reverted, and the bridged funds are never credited on L2 — while the L1 message is permanently marked as processed and cannot be replayed.

---

### Finding Description

In `execute_l1_handler_transaction`, the execution sequence is:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [1](#0-0) 

The message is consumed (written to `outputs.messages_to_l2`) at line 444, **before** the handler runs at line 446. The gas budget is fixed at:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;
``` [2](#0-1) 

This is 11× less than the gas available to a regular `__execute__` call:

```cairo
const EXECUTE_MAX_SIERRA_GAS = 1100000000;
``` [3](#0-2) 

In `select_execute_entry_point_func`, for Sierra-gas-mode contracts the `remaining_gas` passed in is used directly as the execution budget:

```cairo
if (is_sierra_gas_mode != FALSE) {
    tempvar inner_remaining_gas = remaining_gas;
} else {
    // Run with high enough gas to avoid out-of-gas.
    tempvar inner_remaining_gas = DEFAULT_INITIAL_GAS_COST;
}
``` [4](#0-3) 

So for any Sierra-gas-mode L1 handler, the execution budget is exactly and irrevocably `100,000,000` gas — regardless of the handler's actual complexity.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

L1 handlers are the primary mechanism for L1→L2 asset bridging (e.g., depositing ETH or ERC-20 tokens from Ethereum to StarkNet). The sequence is:

1. `consume_l1_to_l2_message` writes the message to the OS output, marking it as processed on-chain.
2. The handler executes with the fixed 100M gas budget.
3. If the handler runs out of gas, `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0`, meaning the OS proof still succeeds but the handler's state changes (e.g., token minting) are rolled back.
4. The L1 message is already consumed and **cannot be replayed**.
5. The funds sent from L1 are never credited on L2 and cannot be recovered. [5](#0-4) 

---

### Likelihood Explanation

Any Sierra (Cairo 1.0) L1 handler contract whose execution requires more than 100M gas is affected. This is realistic for bridge contracts that perform multiple nested calls, storage writes, and event emissions. For reference, a single `call_contract` costs `91,560` gas and a `storage_write` costs `44,970` gas. [6](#0-5) 

A bridge handler that calls a token contract (91,560), performs several storage writes (4 × 44,970 = 179,880), emits events (3 × 10,000 = 30,000), and has its own logic overhead can easily approach or exceed 100M gas for complex DeFi protocols. The L1 message sender (an unprivileged Ethereum user or contract) triggers this path simply by sending a valid L1→L2 message.

---

### Recommendation

1. **Align the L1 handler gas budget with `EXECUTE_MAX_SIERRA_GAS`** (1,100,000,000) rather than `VALIDATE_MAX_SIERRA_GAS` (100,000,000), since L1 handlers perform execution-equivalent work, not validation-equivalent work.
2. **Alternatively**, allow the L1 message to specify a gas limit (analogous to how V3 transactions specify `resource_bounds`), so the gas budget is not hardcoded at the OS level.
3. **As a safety measure**, consider consuming the L1 message only after successful execution, or providing a cancellation/refund path on L1 for failed handlers.

---

### Proof of Concept

1. Deploy a Sierra (Cairo 1.0) contract on StarkNet with an L1 handler that performs complex logic requiring > 100M gas (e.g., multiple nested `call_contract` syscalls to a token contract, several storage writes, and event emissions).
2. Send an L1→L2 message from Ethereum targeting this handler with a deposit amount.
3. The OS processes the block: `consume_l1_to_l2_message` marks the message as consumed in the output.
4. `non_reverting_select_execute_entry_point_func` runs the handler with `remaining_gas = 100,000,000`.
5. The handler exhausts its gas budget mid-execution; the Sierra VM returns an out-of-gas error.
6. The handler's state changes (e.g., token mint) are reverted.
7. The block is proven and finalized on L1 with the message marked as consumed.
8. The deposited funds are permanently frozen: not credited on L2, and the L1 message cannot be replayed. [1](#0-0) [2](#0-1)
