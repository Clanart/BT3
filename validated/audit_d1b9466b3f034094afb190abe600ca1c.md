### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts a fully user-controlled class hash and writes it directly to contract state without verifying that the class hash is declared (i.e., exists in `contract_class_changes`). This is the direct analog of the GammaProtocol finding: just as the Controller contract makes an external call to an unvalidated callee address, the OS here performs a state-transition to an unvalidated class hash. The result is that any contract can be permanently bricked — and all funds in its storage permanently frozen — by setting its class hash to an undeclared value.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the OS reads the new class hash directly from the user-supplied syscall request and writes it to `contract_state_changes` with no check that the hash corresponds to a declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `class_hash` value originates from `request.class_hash` which is the raw syscall payload — fully attacker-controlled. The same missing validation exists in the deprecated path: [2](#0-1) 

When any subsequent call is made to the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

If `class_hash` is undeclared, `dict_read` returns 0 (no entry), and `find_element` fails to locate a compiled class fact — causing a proof-level failure for any block that includes a call to the bricked contract. The contract becomes permanently uncallable.

The `deploy_contract` function correctly guards against deploying to reserved addresses and verifies `UNINITIALIZED_CLASS_HASH`, but `execute_replace_class` has no equivalent guard: [4](#0-3) 

The `UNINITIALIZED_CLASS_HASH = 0` constant is defined in the state commitment module: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value via `replace_class`, the contract is permanently uncallable at the OS level. Any ERC-20 balances, ETH, or other assets held in the contract's storage are irretrievably locked. There is no recovery path: the OS cannot execute the contract to withdraw funds, and the state cannot be rolled back after the block is finalized.

---

### Likelihood Explanation

**Medium.**

The attacker path requires one of the following:
1. A deployed contract that exposes `replace_class` with an attacker-controlled class hash parameter (e.g., an upgradeable proxy where the upgrade function does not validate the new implementation hash).
2. A contract that calls `replace_class` with a hash derived from attacker-controlled calldata.

Upgradeable contract patterns are common in StarkNet. Any such contract that does not independently validate the new class hash before calling the syscall is exploitable. The OS is the last line of defense and currently provides none.

---

### Recommendation

In `execute_replace_class` (`syscall_impls.cairo`), before writing the new class hash to state, verify that it exists in `contract_class_changes` (i.e., that it has been declared in the current or a prior block). The TODO comment at line 898 already acknowledges this gap:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

The fix should assert that `dict_read{dict_ptr=contract_class_changes}(key=class_hash)` returns a non-zero compiled class hash, analogous to how `deploy_contract` asserts `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` before proceeding. Apply the same fix to the deprecated `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Deploy contract `Victim` holding user funds, with an upgrade function:
   ```
   fn upgrade(new_class_hash: felt252) {
       replace_class_syscall(new_class_hash);  // no validation
   }
   ```
2. Attacker calls `Victim.upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. The OS executes `execute_replace_class`: `contract_state_changes[Victim_address].class_hash = 0xdeadbeef`. No validation occurs. [6](#0-5) 
4. In any subsequent block, any call to `Victim` reaches `execute_entry_point`, which does `dict_read(key=0xdeadbeef)` → returns 0, then `find_element(..., key=0)` → fails to find a compiled class fact. [7](#0-6) 
5. All calls to `Victim` are permanently rejected. Funds in `Victim`'s storage are frozen forever.
