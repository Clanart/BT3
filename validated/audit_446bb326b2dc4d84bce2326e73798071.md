### Title
`ProofOfInclusion::valid()` Is a Tautological Self-Check That Never Validates Against an External Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle library only checks internal consistency of the proof chain. Its final comparison `existing_hash == self.root_hash()` is a logical tautology — it always evaluates to `true` after the loop completes — because `root_hash()` is derived from the proof's own last `combined_hash` field, not from any external trusted root. An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` (representing a fake key-value pair) and internally consistent `layers`, and `valid()` will return `true`. Any verifier that relies solely on `proof.valid()` accepts forged inclusion proofs.

### Finding Description

**Root cause — `valid()` is a tautology:** [1](#0-0) 

The loop invariant is: after each iteration, `existing_hash` is set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. After the last iteration, `existing_hash` therefore equals the last layer's `combined_hash`.

`root_hash()` is: [2](#0-1) 

It returns `last.combined_hash` — the exact same value `existing_hash` holds at loop exit. The final check `existing_hash == self.root_hash()` is therefore always `true` when the loop completes without returning `false`. It adds zero security.

**What `valid()` actually checks:** only that each layer's `combined_hash` equals `calculate_internal_hash(previous_hash, other_hash_side, other_hash)`. It never compares the final root against any externally trusted root.

**Attacker construction:** An attacker builds a `ProofOfInclusion` with:
- `node_hash`: the hash of a fake leaf (e.g., a key-value pair not in the tree)
- `layers`: any chain where each `combined_hash` = `calculate_internal_hash(prev, side, other_hash)` for attacker-chosen `other_hash` values

`valid()` returns `true` for this forged proof.

**Exposure surface:** `ProofOfInclusion` is a public, `Streamable`-serializable type with full Python bindings: [3](#0-2) [4](#0-3) [5](#0-4) 

The Python binding exposes `valid()` and `root_hash()` as separate methods, making it natural for callers to use only `valid()` as the sole check.

**Contrast with the correct MerkleSet proof validation**, which explicitly checks the root against an external trusted value: [6](#0-5) 

The DataLayer `ProofOfInclusion::valid()` has no equivalent external-root parameter.

**Tests and fuzz targets confirm the pattern:** all callers use `proof.valid()` alone, with no separate root comparison: [7](#0-6) [8](#0-7) 

### Impact Explanation

Any DataLayer verifier that calls `proof.valid()` as its sole check accepts forged inclusion proofs. An attacker can prove that an arbitrary key-value pair is present in a DataLayer store when it is not. This maps directly to the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

### Likelihood Explanation

`ProofOfInclusion` is a serializable, Python-exposed type. Any DataLayer client that receives a proof over the network and calls `proof.valid()` is vulnerable. The misleading API name (`valid`) makes it highly likely that callers treat it as a complete validity check. The `root_hash()` method exists but nothing in the API contract or documentation forces callers to compare it against a trusted root.

### Recommendation

Change `valid()` to require an external trusted root parameter:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == *expected_root  // compare against external trusted root
}
```

Alternatively, keep `valid()` but make it accept the expected root as a parameter, and deprecate or remove the zero-argument form. Update all call sites (Python bindings, fuzz targets, tests) to pass the known tree root. This mirrors the correct pattern already used in `validate_merkle_proof` in `merkle_tree.rs`.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from chia_rs.sized_bytes import bytes32
from chia_datalayer import calculate_internal_hash  # or equivalent

# Attacker-chosen values
fake_node_hash = bytes32(b'\xaa' * 32)
other_hash     = bytes32(b'\xbb' * 32)
side           = 0  # Left

# Compute a consistent combined_hash
combined = calculate_internal_hash(fake_node_hash, side, other_hash)

layer = ProofOfInclusionLayer(
    other_hash_side=side,
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though fake_node_hash is not in any real tree
assert forged_proof.valid()          # passes — tautology
assert forged_proof.root_hash() == combined  # root is attacker-controlled
```

The proof passes `valid()` with a completely attacker-controlled `node_hash` and root, proving inclusion of a fake key-value pair in any DataLayer store a verifier trusts.
