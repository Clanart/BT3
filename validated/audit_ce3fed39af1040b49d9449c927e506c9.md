### Title
Zero `fee_actual` Permanently Locks `fee_proposal` at Zero, Corrupting `ProposalCommitment` — (`crates/apollo_consensus_orchestrator/src/dynamic_gas_price/mod.rs`)

---

### Summary

When `fee_actual = GasPrice(0)`, `fee_proposal_bounds` returns `(lower=0, upper=0)`, forcing `compute_fee_proposal` to always return `GasPrice(0)` regardless of the oracle-derived target. This wrong `fee_proposal = 0` is then committed into `ProposalCommitment` via `proposal_commitment_from`. Once `fee_actual` reaches zero, the system is permanently stuck: every subsequent proposal is forced to zero, every validator accepts it, and the SNIP-35 fee mechanism is irreversibly broken.

---

### Finding Description

**Root cause — `fee_proposal_bounds` with zero input:** [1](#0-0) 

When `fee_actual = GasPrice(0)`:
- `upper = 0 * (PPT_DENOMINATOR + margin_ppt) / PPT_DENOMINATOR = 0`
- `lower = 0 * PPT_DENOMINATOR / (PPT_DENOMINATOR + margin_ppt) = 0`

So `compute_fee_proposal` clamps any `fee_target` into `[0, 0]` and always returns `GasPrice(0)`: [2](#0-1) 

This is explicitly documented in the test suite as accepted behavior: [3](#0-2)

```rust
pub(crate) fn fee_proposal_bounds(fee_actual: GasPrice, margin_ppt: u128) -> (u128, u128) {
    let denom = U256::from(PPT_DENOMINATOR);
    let scaled = denom + U256::from(margin_ppt);
    let fee_actual_u256 = U256::from(fee_actual.0);
    let upper = u128::try_from(fee_actual_u256 * scaled / denom).unwrap_or(u128::MAX);
    let lower = u128::try_from(fee_actual_u256 * denom / scaled).unwrap_or(0);
    (lower, upper)
}
```
