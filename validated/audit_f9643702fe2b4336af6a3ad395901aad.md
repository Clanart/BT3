### Title
Permissionless External Call to Unverified Token Address Enables Unbacked Token Minting on NEAR - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol` exposes two permissionless functions — `logMetadata` and `initTransfer` — that make external calls to arbitrary, caller-supplied token addresses without any whitelist or registration check. An attacker can exploit this two-step path to register a malicious ERC20 on NEAR and then mint unbacked NEAR-side bridge tokens by supplying a fake token whose `transferFrom` returns `true` without moving any funds.

---

### Finding Description

**Step 1 — `logMetadata` calls arbitrary address (no access control)**

`logMetadata` is `external payable` with no role restriction. It calls `.name()`, `.symbol()`, and `.decimals()` on any caller-supplied address and emits a `LogMetadata` event:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name    = IERC20Metadata(tokenAddress).name();
    string memory symbol  = IERC20Metadata(tokenAddress).symbol();
    uint8  decimals       = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

The emitted `LogMetadata` event is consumed by NEAR relayers, which call `deploy_token` on the NEAR bridge. `deploy_token_callback` validates only the **emitter address** (the EVM bridge contract itself), not the token address:

```rust
require!(
    self.factories.get(&chain) == Some(metadata.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
self.deploy_token_internal(chain, &metadata.token_address, BasicMetadata { ... }, attached_deposit)
``` [2](#0-1) 

`deploy_token_internal` registers the malicious token address in `token_decimals`, `token_address_to_id`, and `token_id_to_address` on NEAR, making it a fully recognized bridge token.

---

**Step 2 — `initTransfer` calls arbitrary address (no token whitelist)**

`initTransfer` accepts any `tokenAddress`. For tokens that are neither a `customMinter` nor a `isBridgeToken`, it falls through to a raw `safeTransferFrom` call with no registration check:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [3](#0-2) 

A malicious ERC20 that returns `true` from `transferFrom` without moving any tokens passes `safeTransferFrom` (which only checks the return value). The bridge then unconditionally emits `InitTransfer`: [4](#0-3) 

---

**Step 3 — NEAR mints tokens for the fake deposit**

`fin_transfer_callback` on NEAR checks only that the emitter is a registered factory and that `token_decimals` has an entry for the token. Since Step 1 already registered the malicious token, both checks pass:

```rust
require!(
    self.factories.get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [5](#0-4) 

NEAR then mints the full `amount` of the NEAR-side NEP-141 bridge token to the attacker's address, with zero EVM-side collateral.

---

### Impact Explanation

An unprivileged attacker can mint an arbitrary quantity of NEAR-side bridge tokens that are entirely unbacked by EVM-side deposits. This directly violates the bridge's 1:1 collateralization invariant and constitutes an **unauthorized mint of bridged assets**. The attacker can:

- Sell the minted tokens into any liquidity pool that accepts the bridge token.
- Repeat the attack at scale (each `initTransfer` call uses a fresh nonce and emits a fresh provable event).
- Impersonate a legitimate token's name/symbol in the `logMetadata` call to make the minted tokens appear credible to users.

**Impact class**: Critical — unauthorized mint of bridged assets on NEAR.

---

### Likelihood Explanation

Both `logMetadata` and `initTransfer` are fully permissionless public functions callable by any EOA. No privileged role, leaked key, or colluding operator is required. The only cost is gas and a small NEAR storage deposit for `deploy_token`. The attack is straightforward to execute on any EVM chain where the bridge is deployed.

---

### Recommendation

1. **Whitelist tokens in `initTransfer`**: Reject calls where `tokenAddress` is not registered in `ethToNearToken` (or an equivalent registry). This mirrors the post-audit fix in the original report — validate the address before making the external call.

2. **Restrict `logMetadata`**: Add an access-control check (e.g., `DEFAULT_ADMIN_ROLE`) or require the token to already be present in `ethToNearToken` before emitting the `LogMetadata` event. This prevents an attacker from bootstrapping a malicious token's NEAR-side registration.

3. **Apply the same fix to Starknet**: `init_transfer` and `log_metadata` in `starknet/src/omni_bridge.cairo` have the identical pattern — arbitrary `token_address` is called without a whitelist check. [6](#0-5) [7](#0-6)

### Proof of Concept

```
1. Attacker deploys MaliciousERC20 on EVM:
   - name()       → "Legit Token"
   - symbol()     → "LGT"
   - decimals()   → 18
   - transferFrom(from, to, amount) → returns true (no actual transfer)

2. Attacker calls:
   OmniBridge.logMetadata(address(MaliciousERC20))
   → emits LogMetadata(MaliciousERC20, "Legit Token", "LGT", 18)

3. NEAR relayer submits proof of LogMetadata event to NEAR bridge.
   deploy_token_callback validates emitter == registered EVM factory ✓
   deploy_token_internal registers MaliciousERC20 in token_decimals ✓

4. Attacker calls:
   OmniBridge.initTransfer(address(MaliciousERC20), 1_000_000e18, 0, 0, "attacker.near", "")
   → MaliciousERC20.transferFrom(attacker, bridge, 1_000_000e18) returns true (no tokens moved)
   → emits InitTransfer(attacker, MaliciousERC20, nonce, 1_000_000e18, ...)

5. NEAR relayer submits proof of InitTransfer event to NEAR bridge.
   fin_transfer_callback:
     - emitter == registered factory ✓
     - token_decimals[MaliciousERC20] exists ✓
     - mints 1_000_000 NEP-141 "LGT" tokens to attacker.near

Result: Attacker holds 1,000,000 unbacked NEAR-side bridge tokens.
        Zero EVM-side tokens were ever deposited.
```
