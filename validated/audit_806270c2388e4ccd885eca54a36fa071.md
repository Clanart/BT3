### Title
Fee Transfer Return Value Unchecked in `charge_fee` Enables Silent Fee Bypass — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function executes the fee token's `transfer` entry point via `non_reverting_select_execute_entry_point_func` but silently discards all return data. If the fee token contract returns a failure indicator (`false` / `retdata[0] = 0`) without reverting, the OS accepts the block as valid while the sequencer receives no fee. This is the direct Cairo analog of the IERC20.transfer return-value-not-checked vulnerability class.

---

### Finding Description

In `charge_fee` at lines 160–164 of `transaction_impls.cairo`:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
``` [1](#0-0) 

`non_reverting_select_execute_entry_point_func` is declared to return `(retdata_size: felt, retdata: felt*, is_deprecated: felt)`: [2](#0-1) 

All three return values are dropped at the `charge_fee` call site. The only internal guard inside `non_reverting_select_execute_entry_point_func` is `assert is_reverted = 0` — it asserts the entry point did not panic/revert, but says nothing about the content of `retdata`. [3](#0-2) 

By contrast, every other call to `non_reverting_select_execute_entry_point_func` in the same file validates the return data:

- `execute_deploy_account_transaction` (lines 677–684): captures `retdata` and asserts `retdata[0] = VALIDATED`
- `execute_declare_transaction` (lines 804–812): same pattern [4](#0-3) [5](#0-4) 

Therefore, if the fee token's `transfer` entry point executes successfully (`is_reverted = 0`) but returns `false` or an empty array to signal a failed transfer, the OS accepts the execution as valid and the sequencer receives nothing.

---

### Impact Explanation

A user can drain their fee-token balance inside `__execute__` (e.g., by transferring all tokens to a second address they control). When `charge_fee` subsequently calls `transfer(sequencer, actual_fee)` on the fee token, the token contract may return `false` (ERC20 failure indicator) without reverting. Because the OS never inspects `retdata`, the proof is still valid, the block is finalized on L1, and the sequencer receives zero fee for that transaction. This constitutes a **direct loss of funds** for the sequencer: real economic value (STRK/ETH) that should have been transferred is not, yet the state transition is accepted as correct by the verifier.

The `charge_fee` function is invoked for all account transaction types — invoke, deploy-account, and declare — so the vulnerability surface covers the entire fee-charging path. [6](#0-5) 

---

### Likelihood Explanation

The attack requires the fee token's `transfer` to return `false` rather than revert on insufficient balance. The current canonical STRK/ETH token contracts revert on underflow, making the vulnerability latent rather than immediately exploitable. However:

1. The OS places **no constraint** on the fee token's revert-vs-return-false behavior.
2. Any future upgrade of the fee token contract to a standard Cairo 1 ERC20 that returns `false` on failure would immediately open this path.
3. A user who drains their balance inside `__execute__` creates the precondition; the sequencer cannot always detect this before block finalization.
4. The inconsistency with the validate entry-point checks (which do assert `retdata[0] = VALIDATED`) shows this omission is unintentional.

---

### Recommendation

After calling `non_reverting_select_execute_entry_point_func` in `charge_fee`, validate the return data the same way the validate entry points do:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
if (is_deprecated == 0) {
    // ERC20 transfer must return (true,) — retdata[0] = 1.
    assert retdata_size = 1;
    assert retdata[0] = 1;
}
```

This mirrors the pattern already used for `__validate__`, `__validate_deploy__`, and `__validate_declare__` and closes the silent-failure path.

---

### Proof of Concept

1. Deploy an account contract whose `__execute__` transfers all STRK balance to an attacker-controlled address B.
2. Submit an invoke transaction with `max_fee > 0` (so `charge_fee` is not skipped at line 123).
3. OS runs `__validate__` (passes), then `__execute__` (drains balance to B).
4. OS calls `charge_fee` → constructs `TransferCallData(recipient=sequencer, amount=actual_fee)` and calls `transfer` on the fee token.
5. Fee token finds sender balance = 0, returns `(false,)` (`retdata[0] = 0`) without reverting.
6. `non_reverting_select_execute_entry_point_func` returns `(retdata_size=1, retdata=[0], is_deprecated=0)` and asserts `is_reverted = 0` — passes.
7. `charge_fee` discards the return tuple and returns normally.
8. The OS produces a valid proof; the block is accepted on L1.
9. Sequencer balance is unchanged; user paid no fee. [7](#0-6)
