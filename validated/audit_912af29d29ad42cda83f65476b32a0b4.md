### Title
Unvalidated `compiled_class_hash` in Declare Transaction Enables Irreversible Class Corruption and Permanent Fund Freezing — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The StarkNet OS accepts a declare transaction with an arbitrary (non-zero) `compiled_class_hash` without verifying it corresponds to a valid CASM. Because a class can only be declared once (`prev_value=0` is enforced), a malicious class declarer can permanently associate a bogus CASM hash with any Sierra class. Any contract subsequently deployed under that class becomes permanently un-executable, freezing all funds held by those contracts with no recovery path.

---

### Finding Description

In `execute_declare_transaction`, after the Sierra `class_hash` is correctly verified via `finalize_class_hash`, the `compiled_class_hash` (the CASM hash) is accepted with only a non-zero check:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [1](#0-0) 

There is no check that `compiled_class_hash` matches any entry in the `compiled_class_facts_bundle`. The post-execution validation (`validate_compiled_class_facts_post_execution`) only validates CASM facts that were **actually executed** in the block — not those merely declared:

```cairo
validate_compiled_class_facts_post_execution(
    n_compiled_class_facts=compiled_class_facts_bundle.n_compiled_class_facts,
    compiled_class_facts=compiled_class_facts_bundle.compiled_class_facts,
    builtin_costs=compiled_class_facts_bundle.builtin_costs,
);
``` [2](#0-1) 

When a contract using the corrupted class is later invoked, `execute_entry_point` performs a `find_element` lookup on the `compiled_class_facts_bundle` using the stored (bogus) `compiled_class_hash`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [3](#0-2) 

`find_element` **asserts** the element exists. Since the bogus hash matches no real CASM, any block that includes a call to the corrupted contract will produce an invalid proof. The sequencer is therefore permanently unable to include any transaction touching those contracts.

The `prev_value=0` constraint makes this irreversible — the class cannot be re-declared with the correct CASM hash: [1](#0-0) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Contracts deployed under the corrupted class can never be executed. The sequencer cannot include any transaction that calls them (the proof would be unprovable). All ETH/STRK held by those contracts is permanently inaccessible. There is no on-chain recovery mechanism because the class registry entry is immutable after the first declaration.

---

### Likelihood Explanation

Any unprivileged user who submits a declare transaction is a valid attacker. The attacker only needs to:
- Know the Sierra class hash of a target class (publicly derivable from the Sierra source)
- Be the first to declare it (front-running a legitimate declarer)
- Provide any non-zero value as `compiled_class_hash`

This is especially dangerous for widely-used standard classes (e.g., standard account contracts, ERC20 tokens) whose Sierra source is public. An attacker can compute the expected `class_hash` off-chain and race to declare it with a bogus CASM hash before the legitimate deployer.

---

### Recommendation

At declaration time, validate that `compiled_class_hash` is present in the `compiled_class_facts_bundle` (i.e., perform a `search_sorted` or `find_element` lookup against the bundle before writing to `contract_class_changes`). Alternatively, require that the declared class is executed in the same block so that `validate_compiled_class_facts_post_execution` covers it. The `migrate_classes_to_v2_casm_hash` function already demonstrates the correct pattern — it recomputes and asserts the hash before updating:

```cairo
assert compiled_class_fact.hash = casm_hash_v2;
``` [4](#0-3) 

The same validation must be applied at declare time.

---

### Proof of Concept

**Setup**: A popular Sierra account class `C` has a known `class_hash = H` (derivable from public source). It has not yet been declared on-chain.

**Step 1 — Attacker declares with bogus CASM hash**:
Attacker submits a `declare` transaction with:
- Valid Sierra class `C` → `class_hash = H` (passes `finalize_class_hash`)
- `compiled_class_hash = 0xDEAD` (arbitrary non-zero value, passes `assert_not_zero`)

The OS processes this and writes `H → 0xDEAD` into `contract_class_changes`. The proof is valid.

**Step 2 — Legitimate users deploy contracts**:
Users deploy account contracts with `class_hash = H`. The state now maps those contract addresses to class `H`.

**Step 3 — Funds are deposited**:
Users fund their accounts (ETH/STRK transferred to the deployed contract addresses).

**Step 4 — Permanent freeze**:
Any invoke transaction targeting these accounts causes `execute_entry_point` to look up `compiled_class_hash = 0xDEAD` in the `compiled_class_facts_bundle`. No such entry exists. `find_element` fails. The block proof is invalid. The sequencer cannot include any transaction touching these contracts. All funds are permanently frozen.

**Step 5 — No recovery**:
The legitimate declarer attempts to re-declare class `H` with the correct CASM hash. The OS rejects it: `prev_value=0` is violated (the entry already exists as `0xDEAD`). The corruption is permanent. [5](#0-4)
