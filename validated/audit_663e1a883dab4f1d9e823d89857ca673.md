### Title
`timeout_height` Not Committed to in Legacy EIP-712 Hash Enables Expiry Bypass — (`ante/cosmos/eip712.go`, `ethereum/eip712/eip712_legacy.go`)

### Summary

In the legacy EIP-712 ante handler path, `VerifySignature` feeds `tx.GetTimeoutHeight()` into `legacytx.StdSignBytes`, which serializes it as `"timeout_height"` in the Amino JSON message. However, the `Tx` type schema constructed by `LegacyWrapTxToTypedData` explicitly omits `timeout_height`. Because `apitypes.TypedDataAndHash` only hashes fields present in the type schema, the `timeout_height` value is silently dropped from the hash. An attacker can freely mutate `timeout_height` on any signed legacy EIP-712 tx without invalidating the signature.

---

### Finding Description

**Step 1 — `VerifySignature` serializes `timeout_height` into the message JSON**

In `ante/cosmos/eip712.go`, `VerifySignature` calls:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← included in JSON when non-zero
    legacytx.StdFee{...},
    msgs, tx.GetMemo(),
)
``` [1](#0-0) 

`legacytx.StdSignDoc` has `TimeoutHeight string json:"timeout_height,omitempty"`. When `TimeoutHeight > 0`, the field is present in the JSON as `"timeout_height": "100"`.

**Step 2 — `LegacyWrapTxToTypedData` puts the full JSON into `Message` but the `Tx` type schema omits `timeout_height`**

`LegacyWrapTxToTypedData` unmarshals `txBytes` directly into `txData` (the `Message`), then builds `TypedData` with a hardcoded `Tx` type schema that explicitly does not include `timeout_height`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
``` [2](#0-1) 

The `Message` map (`txData`) contains `"timeout_height": "100"`, but the `Types["Tx"]` slice does not list it.

**Step 3 — `TypedDataAndHash` ignores the field**

Per EIP-712 and go-ethereum's `apitypes.TypedDataAndHash`, `encodeData` iterates only over the fields declared in the type schema. Any key present in `Message` but absent from `Types["Tx"]` is silently dropped. The resulting hash is therefore identical for `TimeoutHeight=0` and `TimeoutHeight=100`. [3](#0-2) 

**Step 4 — The guard exists elsewhere but not here**

`legacyDecodeProtobufSignDoc` (used by `PubKey.verifySignatureAsEIP712`) does reject non-zero `TimeoutHeight`:

```go
if body.TimeoutHeight != 0 || ... {
    return apitypes.TypedData{}, errors.New("body contains unsupported fields: TimeoutHeight, ...")
}
``` [4](#0-3) 

But `VerifySignature` in `ante/cosmos/eip712.go` does **not** go through that path. It calls `LegacyWrapTxToTypedData` directly with no guard on `tx.GetTimeoutHeight()`. The protection is absent on the production ante-handler path.

---

### Impact Explanation

An attacker who can observe or intercept a signed legacy EIP-712 Cosmos tx (e.g., from the mempool or a broadcast relay) can:

1. **Strip expiry**: Change `TimeoutHeight=N` → `TimeoutHeight=0`. The signature still verifies. The tx that the user intended to expire at block N now has no deadline and can be submitted at any future block.
2. **Inject expiry**: Change `TimeoutHeight=0` → `TimeoutHeight=N`. The tx is forced to expire at block N, causing a targeted DoS on the user's transaction.

Scenario 1 is the higher-severity case: a user may sign a tx with a short `timeout_height` precisely to limit its validity window (e.g., a time-sensitive trade or authorization). Stripping that deadline allows the tx to be executed at an arbitrarily later time, violating the user's explicit intent and enabling a form of delayed replay.

This matches the High impact category: **EIP-712 authorization field bypass enabling replay / unauthorized execution outside the signer's intended validity window**.

---

### Likelihood Explanation

- The attacker is unprivileged; no special role is required.
- The entry point is public tx submission / mempool observation.
- The modification is trivial: flip one integer field in the tx body before rebroadcasting.
- The `timeout_height` omission is explicitly documented in a code comment, confirming it is a known design choice rather than an accidental gap — but the consequence (hash non-commitment) was not addressed.

---

### Recommendation

Add an explicit guard in `VerifySignature` (or immediately before calling `LegacyWrapTxToTypedData`) that rejects any tx where `tx.GetTimeoutHeight() != 0`, mirroring the guard already present in `legacyDecodeProtobufSignDoc`:

```go
if tx.GetTimeoutHeight() != 0 {
    return errorsmod.Wrap(errortypes.ErrNotSupported,
        "legacy EIP-712 does not support timeout_height; set it to 0")
}
```

Alternatively, add `{Name: "timeout_height", Type: "string"}` back to the `Tx` type schema in `extractMsgTypes` and ensure `StdSignBytes` always serializes it (even as `"0"`) so the field is always committed to in the hash.

---

### Proof of Concept

```
1. Build a legacy EIP-712 Cosmos tx with TimeoutHeight = 100.
2. Call VerifySignature / LegacyWrapTxToTypedData with this tx.
3. Assert that msgTypes["Tx"] contains no entry with Name == "timeout_height".
4. Compute hash_A = TypedDataAndHash(typedData with TimeoutHeight=100).
5. Rebuild with TimeoutHeight = 0; compute hash_B.
6. Assert hash_A == hash_B.
7. Sign hash_A; verify the signature against hash_B → passes.
8. Submit the tx with TimeoutHeight=0 after block 100 → accepted by the chain.
```

The invariant "all fields present in the signed JSON are covered by the type schema" is violated for `timeout_height`, and the field can be freely mutated post-signing.
