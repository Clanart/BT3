### Title
Unbounded `memcpy` in `consume_l1_to_l2_message` Allows L1 Message Senders to Exhaust OS Step Budget Without L2 Gas Bound — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `consume_l1_to_l2_message` function in `transaction_impls.cairo` performs an unbounded `memcpy` of the L1 handler's calldata payload into the OS output segment. The calldata size is only bounded by `SIERRA_ARRAY_LEN_BOUND - 1 = 2^32 - 1` felts. This OS-level copy work executes **before** the L2 gas limit is applied to the user contract, is not bounded by `L1_HANDLER_L2_GAS_MAX_AMOUNT`, and L1 handler transactions pay no L2 fees. An attacker controlling an L1 contract can flood the sequencer with L1 messages carrying maximum-size payloads, causing the OS prover to spend O(payload_size) Cairo steps per transaction outside the gas accounting system, exhausting the block's provable step budget and halting new transaction confirmation.

---

### Finding Description

**Root cause — step 1: Only bound on calldata size is `SIERRA_ARRAY_LEN_BOUND - 1`**

In `get_invoke_tx_execution_context`, the sole OS-enforced upper bound on L1 handler calldata is:

```cairo
assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
``` [1](#0-0) 

`SIERRA_ARRAY_LEN_BOUND` is defined as `4294967296` (2³²): [2](#0-1) 

This permits up to ~4 billion felts (~128 GB) of calldata per L1 handler transaction.

**Root cause — step 2: OS-level `memcpy` runs before the L2 gas limit is applied**

In `execute_l1_handler_transaction`, `consume_l1_to_l2_message` is called **before** `remaining_gas` is set to `L1_HANDLER_L2_GAS_MAX_AMOUNT`:

```cairo
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(...)
``` [3](#0-2) 

**Root cause — step 3: `consume_l1_to_l2_message` performs O(payload_size) `memcpy`**

Inside `consume_l1_to_l2_message`, the entire payload is copied to the OS output segment:

```cairo
tempvar payload_size = execution_context.calldata_size - 1;
...
memcpy(dst=message_payload, src=payload, len=payload_size);
``` [4](#0-3) 

In Cairo, `memcpy` of N felts consumes N Cairo VM steps. These steps are part of the **OS proof**, not the user contract proof, and are therefore entirely outside the `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000` gas cap: [5](#0-4) 

**Root cause — step 4: L1 handler transactions pay no L2 fees**

`execute_l1_handler_transaction` never calls `charge_fee`, unlike `execute_invoke_function_transaction`: [6](#0-5) 

The combination means: an L1 message sender pays only L1 Ethereum gas for calldata, while the StarkNet OS prover bears O(payload_size) Cairo step cost with no L2-side economic deterrent enforced by the protocol code.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

Each L1 handler transaction with a large payload forces the OS prover to spend O(payload_size) Cairo steps in `consume_l1_to_l2_message`. A block's provable step budget is finite. An attacker sending a stream of L1 messages with near-maximum payloads can saturate the OS step budget across consecutive blocks, leaving no room for legitimate invoke, declare, or deploy transactions. Because L1 handler transactions are protocol-mandatory (they represent L1→L2 messages that must be consumed), the sequencer cannot simply drop them without breaking L1↔L2 message integrity.

---

### Likelihood Explanation

**Medium.** The attacker must pay Ethereum calldata gas (16 gas/non-zero byte). However:
- The OS code imposes no tighter limit than `2^32 - 1` felts per payload.
- L1 handler transactions carry no L2 fee, so the L2-side cost is entirely borne by the prover.
- The asymmetry between L1 calldata cost and OS proving cost (Cairo steps are cheap to submit but expensive to prove) mirrors the original report's asymmetry between proposal submission cost and `endProposal()` execution cost.
- A well-funded attacker (e.g., a competing chain or protocol adversary) can sustain this attack at a fraction of the economic damage inflicted.

---

### Recommendation

Add an OS-enforced maximum payload size for L1 handler calldata in `get_invoke_tx_execution_context` (or directly in `consume_l1_to_l2_message`), analogous to the maximum action limit introduced in the referenced PR#199. A reasonable bound (e.g., `MAX_L1_HANDLER_CALLDATA_SIZE = 1000` felts) should be enforced via `assert_nn_le` before the `memcpy` is performed:

```cairo
// In consume_l1_to_l2_message or get_invoke_tx_execution_context:
assert_nn_le(payload_size, MAX_L1_HANDLER_PAYLOAD_SIZE);
```

This mirrors the fix in the external report: capping the number of "actions" (here, payload felts) per "proposal" (here, L1 handler transaction) to make the attack economically infeasible.

---

### Proof of Concept

1. Deploy an L1 contract that calls the StarkNet Core contract's `sendMessageToL2` with a payload array of, e.g., 500,000 felts (≈16 MB of calldata; costs ~8M L1 gas at 16 gas/byte, roughly $400 at current prices).
2. The StarkNet sequencer picks up the L1 message and creates an L1 handler transaction.
3. `execute_l1_handler_transaction` calls `consume_l1_to_l2_message`, which executes `memcpy` of 499,999 felts — consuming ~500,000 OS-level Cairo steps outside the gas accounting system.
4. Repeat with multiple L1 messages per Ethereum block.
5. The OS prover's step budget for each StarkNet block is exhausted by these `memcpy` operations, leaving no capacity for invoke/declare/deploy transactions.
6. The network stops confirming new user transactions for as long as the attacker sustains the L1 message flood.

The calldata size check at line 485 (`assert_nn_le(..., SIERRA_ARRAY_LEN_BOUND - 1)`) does not prevent this because `SIERRA_ARRAY_LEN_BOUND - 1 = 2^32 - 1` is orders of magnitude larger than any practical block step budget. [7](#0-6)

func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    assert_not_zero(execution_context.calldata_size);
    // The payload is the calldata without the from_address argument (which is the first).
    let payload: felt* = execution_context.calldata + 1;
    tempvar payload_size = execution_context.calldata_size - 1;

    tempvar execution_info = execution_context.execution_info;

    // Write the given transaction to the output.
    assert [outputs.messages_to_l2] = MessageToL2Header(
        from_address=[execution_context.calldata],
        to_address=execution_info.contract_address,
        nonce=nonce,
        selector=execution_info.selector,
        payload_size=payload_size,
    );

    let message_payload = cast(outputs.messages_to_l2 + MessageToL2Header.SIZE, felt*);
    memcpy(dst=message_payload, src=payload, len=payload_size);

    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1,
        messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE +
        outputs.messages_to_l2.payload_size,
    );
```
