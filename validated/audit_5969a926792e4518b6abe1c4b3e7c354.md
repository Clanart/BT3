### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by the caller is actually declared in the class tree. This allows any contract to replace its own class hash with an arbitrary, undeclared value. Once committed, the contract becomes permanently uncallable, and any funds it holds are permanently frozen.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` handles the `replace_class` syscall. After deducting gas, it reads the new class hash from the request and immediately writes it into the contract's `StateEntry` — with no check that the class hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. The OS accepts any `felt` value as the new class hash — including values that have never been declared via `declare_transaction`. Once the block is proven and the state is committed with an undeclared class hash, the contract becomes permanently uncallable: the prover cannot supply a valid execution trace for a class that does not exist in the class tree.

This is the direct analog to the external report's vulnerability. Just as the lending protocol allows collateral to be redeemed without accounting for a pending cross-chain borrow (a missing state-invariant check), the StarkNet OS allows `replace_class` to commit an arbitrary class hash without verifying the class is declared (a missing state-invariant check). In both cases, a state transition is permitted that violates a critical protocol invariant, producing an invalid final state.

For contrast, `execute_declare_transaction` correctly enforces the analogous invariant for class declaration by using `prev_value=0` to ensure a class can only be declared once:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

Any contract that calls `replace_class` with an undeclared class hash will have its on-chain class hash permanently set to an invalid value. Future calls to that contract will fail at the class-lookup step because the prover cannot supply bytecode for an undeclared hash. If the contract holds funds (ETH, ERC-20 tokens, or other assets), those funds are permanently frozen with no recovery path. This matches the **Critical: Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The attack is reachable by an unprivileged actor via two paths:

1. **Malicious contract deployer**: Deploys a contract that accepts user deposits and then calls `replace_class` with a random, undeclared hash (e.g., `0xdeadbeef`). After collecting deposits, the deployer triggers the call, permanently freezing all deposited funds.

2. **Unprivileged transaction sender exploiting a legitimate upgradeable contract**: Many DeFi and upgradeable contracts expose an `upgrade(new_class_hash)` function that internally calls `replace_class`. If the application-level function does not validate the new class hash, any user can call it with an undeclared hash, freezing the contract.

The OS-level missing check means the protocol itself does not enforce the invariant, leaving it entirely to application-level code. The TODO comment at line 898 confirms this is a known, unresolved gap in the OS implementation. [3](#0-2) 

---

### Recommendation

In `execute_replace_class`, add an OS-level check that verifies the new class hash is declared before updating the contract state. This requires:

1. Checking `contract_class_changes` for the class hash (covers classes declared in the current block).
2. Using a hint-backed Patricia Merkle Tree lookup to verify the class hash exists in the class tree from previous blocks.

The check must be enforced at the OS level — not left to individual application contracts — to guarantee the protocol invariant holds for all state transitions.

---

### Proof of Concept

1. Attacker deploys `VaultContract` with a legitimate, declared class hash.
2. Users deposit funds into `VaultContract`.
3. Attacker calls a function on `VaultContract` that internally executes `replace_class(0xdeadbeef)`, where `0xdeadbeef` is not a declared class hash.
4. The OS processes the syscall via `execute_replace_class`, updates `VaultContract`'s `StateEntry.class_hash` to `0xdeadbeef`, and commits the state — no validation is performed.
5. In the next block, any call to `VaultContract` fails: the prover cannot supply the class for `0xdeadbeef`, so no valid execution trace can be generated.
6. All funds in `VaultContract` are permanently frozen with no recovery mechanism.

        // Not enough gas to execute the syscall.
        return ();
