### Title
Fee-on-Transfer Token Accounting Mismatch Breaks Bridge Collateralization and Permanently Freezes Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` records the user-supplied `amount` in the emitted event without verifying the actual tokens received. For fee-on-transfer ERC-20 tokens, the bridge vault receives fewer tokens than `amount`, but the NEAR side mints/unlocks the full `amount`. This permanently undercollateralizes the EVM vault, causing the last withdrawer's `finTransfer` to revert and permanently freeze their funds.

---

### Finding Description

In `OmniBridge.initTransfer`, when the token is a plain ERC-20 (not a bridge token, not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the full user-supplied `amount` is emitted in the `InitTransfer` event:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-supplied, not actual received
    ...
);
``` [2](#0-1) 

There is no balance-before / balance-after check. For a fee-on-transfer token (e.g., a token that deducts 1% on every transfer), the bridge vault receives only `amount * 0.99` while the event records `amount`. The NEAR bridge processes this event as proof and credits the full `amount` to the recipient.

On the return path, `finTransfer` attempts to release the full recorded amount:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [3](#0-2) 

Because the vault holds less than the sum of all recorded `amount` values, the last withdrawer's `safeTransfer` reverts. Their funds are permanently locked in the bridge with no recovery path.

---

### Impact Explanation

Each `initTransfer` with a fee-on-transfer token silently drains the vault by the fee amount. After N such deposits, the vault is short by `N * fee`. The first `N-1` withdrawers succeed (consuming the deficit from later depositors' funds), but the final withdrawer's `finTransfer` reverts because the vault balance is insufficient. Their funds are irrecoverably locked in the EVM bridge contract.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows** and **Balance / accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

Fee-on-transfer tokens are a well-known ERC-20 pattern (e.g., USDT in some configurations, STA, REFLECT-style tokens). Any such token that is not a registered `isBridgeToken` and has no `customMinter` falls into the unguarded `safeTransferFrom` branch. An unprivileged user can trigger this by simply calling `initTransfer` with such a token — no special role or key is required.

---

### Recommendation

Record the vault balance before and after the `safeTransferFrom` call and use the difference as the canonical `amount` for the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived instead of amount in the event and downstream logic
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported (e.g., via a token allowlist), and add a runtime check that reverts if `actualReceived < amount`.

---

### Proof of Concept

1. Assume `FOT` is a fee-on-transfer ERC-20 token with a 1% transfer fee. It is not registered as a bridge token (`isBridgeToken[FOT] == false`) and has no custom minter.
2. Alice calls `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
3. The bridge vault receives `990` FOT (1% fee deducted), but the `InitTransfer` event records `amount = 1000`.
4. The NEAR bridge processes the proof and credits Alice with `1000` FOT on NEAR.
5. Alice initiates a return transfer from NEAR, burning `1000` FOT on NEAR.
6. A relayer calls `OmniBridge.finTransfer(...)` with `payload.amount = 1000`.
7. `IERC20(FOT).safeTransfer(alice, 1000)` reverts because the vault only holds `990`.
8. Alice's funds are permanently frozen in the EVM bridge — there is no recovery mechanism.
