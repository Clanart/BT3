### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash corresponds to a previously declared class. An unprivileged contract owner can replace their contract's class with an arbitrary, undeclared hash. The OS accepts this state transition without validation, permanently rendering the contract uncallable and freezing any funds held within it.

### Finding Description

The vulnerability class from the reference report is **state-transition bypass via missing configuration guard**: a component transitions to an "active/valid" state before all required preconditions are satisfied. The analog here is that `execute_replace_class` allows a contract to transition to a new class without verifying the new class is in a valid (declared) state.

In `syscall_impls.cairo`, `execute_replace_class` processes the `replace_class` syscall. After reducing gas, it reads the requested class hash from the syscall request and writes it directly into the contract state with no check that the hash corresponds to a declared class: [1](#0-0) 

The explicit `TODO` at line 898 acknowledges the missing guard:


When a subsequent call is made to the contract, `execute_entry_point` first reads the compiled class hash from `contract_class_changes` using the contract's stored class hash as the key: [2](#0-1) 

Because the replacement class hash was never declared, `dict_read` returns 0 (the default uninitialized value). The subsequent `find_element` call then searches for a compiled class with hash 0, which does not exist, causing a hard Cairo assertion failure. The contract becomes permanently uncallable.

The critical point is that the `replace_class` call itself succeeds and is committed to the global state root — the undeclared class hash is written into the Patricia Merkle Tree via `state_update`: [3](#0-2) 

Once committed to L1, this state is irreversible. The contract's class hash is permanently set to an undeclared value, and no future transaction can successfully execute against it.

This is directly analogous to the zBanc `DynamicLiquidTokenConverter` bug: just as the converter was considered "active" (valid) before all required parameters were set — allowing interaction with a partially configured system — the StarkNet OS considers a `replace_class` operation "valid" without verifying the new class is properly configured (declared). In both cases, the missing guard on a state transition leads to a permanently broken component.

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract holding user funds (e.g., a vault, escrow, or multi-sig) can have its class replaced with an undeclared hash. Once the state is committed to L1, the contract is permanently inaccessible. All funds held in the contract's storage are frozen with no recovery path, since the class hash cannot be changed back (the contract is uncallable).

### Likelihood Explanation

The attack requires only that an unprivileged user:
1. Deploy or control a contract (standard capability)
2. Submit a transaction that calls `replace_class` with an arbitrary undeclared hash

The sequencer's simulation of the `replace_class` call succeeds (the OS imposes no check), so the transaction is included in a block. The proof is generated successfully because the undeclared class hash is only written to state — it is not executed in the same block. The state is committed to L1. In subsequent blocks, any call to the contract fails at the sequencer simulation stage, making the contract permanently inaccessible.

No privileged access, leaked keys, or malicious operator behavior is required.

### Recommendation

Before writing the new class hash to the contract state, verify that it corresponds to a declared class. This can be done by checking that the class hash exists in `contract_class_changes` (i.e., it maps to a non-zero compiled class hash):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the pattern used in `execute_entry_point` when resolving a class for execution, and closes the gap between what the OS enforces and what the protocol requires.

### Proof of Concept

1. Attacker deploys `VaultContract` holding user funds, with a function `backdoor()` that calls `replace_class(0xdeadbeef)`.
2. Attacker submits an invoke transaction calling `backdoor()`.
3. The sequencer simulates the transaction: `execute_replace_class` runs, reads `class_hash = 0xdeadbeef` from the request, and writes it to `contract_state_changes` without any declared-class check. Simulation succeeds.
4. The sequencer includes the transaction in a block. The OS processes it; `execute_replace_class` updates the state. The block is proved and committed to L1.
5. `VaultContract`'s class hash is now `0xdeadbeef` in the global state root.
6. A user submits a withdrawal transaction targeting `VaultContract`. The sequencer simulates it: `execute_entry_point` calls `dict_read` on `contract_class_changes` with key `0xdeadbeef`, gets 0, then `find_element` fails. The sequencer rejects the transaction.
7. All funds in `VaultContract` are permanently frozen. No recovery is possible. [4](#0-3) [5](#0-4)
