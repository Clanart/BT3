Looking at the full call chain and all relevant guards, here is my analysis:

**Call chain:**
1. `parse_data_for_block` receives `FullTransaction { transaction, transaction_output, transaction_hash }` from a P2P peer
2. `transaction_output` (containing `gas_consumed: GasVector { l1_gas, l1_data_gas, l2_gas }`) is pushed directly to `block_body.transaction_outputs` with no validation
3. `write_to_storage` calls `append_body(block_number, block_body)` which serializes and stores the output as-is

**Guard check:**
- The only check in `parse_data_for_block` is that the count of received transactions matches `header.n_transactions` [1](#0-0) 
- There is no check that `transaction_output.gas_consumed` is consistent with the stored header's `receipt_commitment`
- `append_body` performs no commitment verification of any kind [2](#0-1) 
- The TODO comment at line 88 explicitly acknowledges that transaction hashes from untrusted sources are not validated, and the same absence applies to outputs [3](#0-2) 

**Commitment construction:**
`calculate_receipt_hash` chains `gas_consumed.l1_gas` and `gas_consumed.l1_data_gas` directly into the Poseidon hash for each receipt leaf. [4](#0-3)  The `receipt_commitment` stored in the header was computed over the correct values at block production time. [5](#0-4) 

**What the attacker achieves:**
A malicious peer supplies `transaction_output` with zeroed or arbitrary `gas_consumed`. The syncing node stores this in body storage. The stored `gas_consumed` now differs from the value that was hashed into the header's `receipt_commitment`. Any downstream consumer that reads the stored outputs and recomputes `receipt_commitment` (e.g., a proof verifier, another syncing node, or an RPC client) will observe a mismatch or receive wrong gas values.

**Fast rejection check:**
The "bad peer data is rejected/ignored" fast rejection does **not** apply here — the data is not rejected; it is permanently stored. The corrupted value is concrete: the `gas_consumed` field in the stored `TransactionOutput` for a specific block.

### Title
Malicious P2P peer can store attacker-chosen `gas_consumed` in `TransactionOutput`, violating the `receipt_commitment` invariant — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
`TransactionStreamFactory::parse_data_for_block` accepts `transaction_output.gas_consumed` from an untrusted P2P peer and stores it via `append_body` without verifying consistency with the `receipt_commitment` already stored in the block header. This allows a malicious peer to permanently corrupt the stored `gas_consumed` values for any synced block.

### Finding Description
In `parse_data_for_block`, each `FullTransaction` received from the network has its `transaction_output` pushed directly into `block_body.transaction_outputs` and then written to storage via `append_body`. [3](#0-2)  No check is performed to verify that the `gas_consumed` field of the received output is consistent with the `receipt_commitment` stored in the block header. [6](#0-5) 

The `receipt_commitment` in the header is a Poseidon root over per-transaction receipt hashes, each of which commits to `gas_consumed.l1_gas` and `gas_consumed.l1_data_gas`. [7](#0-6)  After `append_body` stores the attacker-supplied output, the stored `gas_consumed` is permanently inconsistent with the committed value in the header.

### Impact Explanation
- **RPC**: `starknet_getTransactionReceipt` reads stored `TransactionOutput` and returns the attacker-chosen `gas_consumed` as an authoritative value to clients. This matches the allowed impact: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*
- **P2P propagation**: The P2P server's `FetchBlockData for FullTransaction` reads stored outputs and re-serves them to other syncing nodes, propagating the corruption. [8](#0-7) 
- **Commitment mismatch**: Any code that reads stored outputs and recomputes `receipt_commitment` will produce a value that differs from the one stored in the header, breaking commitment binding for the affected block.

### Likelihood Explanation
Any peer that can establish a P2P sync connection can trigger this. The syncing node only validates transaction count, not output content. The attack requires no special privileges.

### Recommendation
After collecting all `transaction_output` values for a block, recompute `receipt_commitment` from the received outputs and compare it against `header.receipt_commitment` before calling `append_body`. Reject the peer (via `report_peer`) if the recomputed commitment does not match the stored header value.

### Proof of Concept
1. Store a block header with a `receipt_commitment` computed over `gas_consumed = GasVector { l1_gas: 100, l1_data_gas: 50, l2_gas: 0 }`.
2. Inject a `FullTransaction` via the P2P response manager with `transaction_output.gas_consumed = GasVector::default()` (all zeros).
3. Call `parse_data_for_block` → `write_to_storage` → `append_body`.
4. Read back the stored `TransactionOutput` and recompute `calculate_receipt_commitment`.
5. Assert the recomputed commitment differs from `header.receipt_commitment` — the assertion holds, confirming the stored gas values are wrong.

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }

impl FetchBlockData for FullTransaction {
    async fn fetch_block_data(
        block_number: BlockNumber,
        txn: &StorageTxn<'_, db::RO>,
        _class_manager_client: &mut SharedClassManagerClient,
    ) -> Result<Vec<Self>, P2pSyncServerError> {
        let transactions =
            txn.get_block_transactions(block_number)?.ok_or(P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            })?;
        let transaction_outputs = txn.get_block_transaction_outputs(block_number)?.ok_or(
            P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            },
        )?;
        let transaction_hashes = txn.get_block_transaction_hashes(block_number)?.ok_or(
            P2pSyncServerError::BlockNotFound {
                block_hash_or_number: BlockHashOrNumber::Number(block_number),
            },
        )?;
        let mut result: Vec<FullTransaction> = Vec::new();
        for (transaction, transaction_output, transaction_hash) in transactions
            .into_iter()
            .zip(transaction_outputs.into_iter())
            .zip(transaction_hashes.into_iter())
            .map(|((a, b), c)| (a, b, c))
        {
            result.push(FullTransaction { transaction, transaction_output, transaction_hash });
        }
        Ok(result)
    }
