### Title
Missing Token-Registration Pre-Check in `init_transfer` Permanently Freezes User Funds When Token Is Not Mapped to Destination Chain - (File: near/omni-bridge/src/lib.rs)

### Summary

`init_transfer` in the NEAR omni-bridge contract accepts and irrevocably locks any NEAR-native token for any destination chain without verifying that the token has been registered (deployed/bound) on that chain. The registration check only occurs later in `sign_transfer`, which panics with `FailedToGetTokenAddress` if the mapping is absent. Because no cancel or refund path exists for pending transfers, the user's tokens are permanently frozen.

### Finding Description

The bridge's outbound transfer flow is split across two separate calls:

**Step 1 — `init_transfer` (via `ft_on_transfer`):** [1](#0-0) 

`init_transfer` creates a `TransferMessage`, stores it in `pending_transfers`, and returns `U128(0)` to `ft_on_transfer`, meaning **all transferred tokens are kept by the bridge** with no refund. There is no check that `token_id_to_address[(destination_chain, token_id)]` exists.

**Step 2 — `sign_transfer` (called later by a relayer):** [2](#0-1) 

`sign_transfer` calls `get_token_address`: [3](#0-2) 

which performs a lookup in `token_id_to_address`. If the token has not been deployed/bound on the destination chain (via `deploy_token` or `bind_token`), this returns `None` and the function panics with `ERR_FAILED_TO_GET_TOKEN_ADDRESS`. The transfer message remains in `pending_transfers` forever, and the locked tokens have no recovery path.

There is no `cancel_transfer`, `refund_transfer`, or any user-callable function that removes a pending transfer and returns the locked tokens. `remove_transfer_message` is only invoked inside `sign_transfer_callback` (when MPC signing succeeds and fee is zero) and `claim_fee_callback` — neither of which is reachable if `sign_transfer` itself panics before calling MPC. [4](#0-3) 

### Impact Explanation

Any user who calls `ft_transfer_call` on a NEAR token with an `InitTransfer` message targeting a destination chain where that token has not yet been registered will have their tokens permanently locked in the bridge contract. There is no on-chain mechanism to recover them. This matches the **Critical — permanent freezing / irrecoverable lock of user funds** impact class.

### Likelihood Explanation

The scenario is realistic and reachable by any unprivileged user:

1. A new chain is added to the bridge (e.g., a new `ChainKind` variant is supported).
2. A NEAR token exists but its counterpart has not yet been deployed on the new chain via `deploy_token` / `bind_token`.
3. A user (or a front-end that does not validate registration) calls `ft_transfer_call` targeting the new chain.
4. Tokens are locked; `sign_transfer` will always panic; funds are frozen.

Additionally, the `token_id_to_address` map requires a separate entry per `(ChainKind, AccountId)` pair: [5](#0-4) 

A token registered for Ethereum is **not** automatically registered for Arbitrum, Base, BNB, etc. A user bridging to any chain where the token is absent triggers the freeze.

### Recommendation

Add a pre-check in `init_transfer` (before accepting tokens) that verifies the token is registered for the destination chain:

```rust
fn init_transfer(...) {
    // Verify token is registered for destination chain before locking funds
    let token_address = self.get_token_address(
        init_transfer_msg.get_destination_chain(),
        token_id.clone(),
    );
    require!(
        token_address.is_some(),
        BridgeError::FailedToGetTokenAddress.as_ref()
    );
    // ... rest of init_transfer
}
```

This mirrors the check already present in `sign_transfer` and ensures the bridge rejects (refunds) tokens for unregistered destination chains at the point of entry rather than after irrevocable locking.

### Proof of Concept

1. Token `foo.near` exists on NEAR. It has been deployed on Ethereum (`token_id_to_address[(Eth, foo.near)]` is set) but **not** on Arbitrum.
2. User calls `foo.near::ft_transfer_call(bridge, 1000, InitTransfer { recipient: "arb:0x...", fee: 0, ... })`.
3. Bridge's `ft_on_transfer` → `init_transfer` runs, stores the transfer, returns `U128(0)`. 1000 tokens are now locked.
4. Relayer calls `sign_transfer(transfer_id, ...)`.
5. `sign_transfer` calls `get_token_address(ChainKind::Arb, "foo.near")` → returns `None`.
6. `env::panic_str(BridgeError::FailedToGetTokenAddress)` — transaction reverts, no state change.
7. Steps 4–6 repeat indefinitely. The 1000 tokens remain locked in the bridge with no recovery path. [2](#0-1) [6](#0-5)

    fn init_transfer(
        &mut self,
        sender_id: AccountId,
        signer_id: AccountId,
        token_id: AccountId,
        amount: U128,
        init_transfer_msg: InitTransferMsg,
    ) -> PromiseOrPromiseIndexOrValue<U128> {
        require!(
            init_transfer_msg.recipient.get_chain() != ChainKind::Near,
            BridgeError::InvalidRecipientChain.as_ref()
        );

        self.current_origin_nonce += 1;
        let destination_nonce =
            self.get_next_destination_nonce(init_transfer_msg.get_destination_chain());

        let transfer_message = TransferMessage {
            origin_nonce: self.current_origin_nonce,
            token: OmniAddress::Near(token_id),
            amount,
            recipient: init_transfer_msg.recipient,
            fee: Fee {
                fee: init_transfer_msg.fee,
                native_fee: init_transfer_msg.native_token_fee,
            },
            sender: OmniAddress::Near(sender_id),
            msg: init_transfer_msg.msg.map(String::from).unwrap_or_default(),
            destination_nonce,
            origin_transfer_id: None,
        };
        require!(
            transfer_message.fee.fee < transfer_message.amount,
            BridgeError::InvalidFee.as_ref()
        );
