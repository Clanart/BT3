### Title
Virtual-Only Fee Deduction Never Settles to Real Bank Balance — Fee Collector Receives No Real Tokens from EVM Transactions - (File: `x/evm/keeper/utils.go`, `x/evm/keeper/gas.go`)

### Summary
`DeductFees` uses `SendCoinsFromAccountToModuleVirtual` (transient object-store only) to deduct EVM transaction fees, and `RefundGas` uses `SendCoinsFromModuleToAccountVirtual` for the gas refund. Neither operation touches the real Cosmos bank KV store. The fee collector's real bank balance is never funded by EVM transaction fees, and the user's real bank balance is never reduced by the ante-handler fee deduction. The virtual accounting is never settled into real balances anywhere in the block lifecycle.

### Finding Description

`DeductFees` in `x/evm/keeper/utils.go`:

```go
func DeductFees(bankKeeper types.BankKeeper, ctx sdk.Context, acc sdk.AccountI, fees sdk.Coins) error {
    if !fees.IsValid() { ... }
    if ctx.BlockHeight() > 0 {
        if err := bankKeeper.SendCoinsFromAccountToModuleVirtual(
            ctx, acc.GetAddress(), authtypes.FeeCollectorName, fees); err != nil {
            return errorsmod.Wrap(errortypes.ErrInsufficientFunds, err.Error())
        }
    }
    return nil
}
```

`RefundGas` in `x/evm/keeper/gas.go`:

```go
err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(
    ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
```

Both `SendCoinsFromAccountToModuleVirtual` and `SendCoinsFromModuleToAccountVirtual` operate exclusively on a transient `ObjKVStore` keyed by `addr || txIndex` (confirmed by `testutil/base_test_suite.go` `addVirtualCoins`). They do **not** call the real `bankKeeper.SendCoinsFromAccountToModule` / `SendCoinsFromModuleToAccount` that would write to the persistent bank KV store.

The balance check in `VerifyEthAccount` (`ante/eth.go` line 101) reads the **real** bank balance:

```go
balance := evmKeeper.GetBalance(ctx, from, evmDenom)
if err := keeper.CheckSenderBalance(sdkmath.NewIntFromBigIntMut(balance.ToBig()), ethTx); err != nil { ... }
```

Because `DeductFees` never reduces the real bank balance, the real balance seen by `CheckSenderBalance` for every subsequent EVM tx in the same block is identical to the pre-block balance. A sender whose real balance covers exactly one tx's cost can have multiple EVM txs included in the same block, each passing `CheckSenderBalance`, because the virtual deductions from earlier txs are invisible to the real-balance read.

`EndBlock` in `x/evm/keeper/abci.go` performs no settlement of virtual balances:

```go
func (k *Keeper) EndBlock(ctx sdk.Context) error {
    k.CollectTxBloom(ctx)
    k.RemoveParamsCache(ctx)
    return nil
}
```

The transient object store is discarded at block commit. The fee collector's real balance is never credited with EVM tx fees.

Additionally, `DeductFees` skips even the virtual deduction at `ctx.BlockHeight() == 0`, while `RefundGas` has no such guard — creating a path where `RefundGas` attempts to draw from a fee collector virtual balance that was never populated.

### Impact Explanation

- **Mis-accounted user funds**: The sender's real bank balance is never reduced by the ante-handler fee deduction. Multiple EVM txs from the same sender in one block can each pass `CheckSenderBalance` against the same unmodified real balance, allowing execution of more transactions than the sender's real balance can cover.
- **Fee collector never receives real tokens**: The fee collector's real bank balance is never funded by EVM tx fees. Any downstream Cosmos fee distribution (staking rewards, community pool) that depends on the fee collector's real balance receives nothing from EVM transactions.
- This matches the allowed High impact: "EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."

### Likelihood Explanation

Every EVM transaction on every block goes through `DeductFees` → `SendCoinsFromAccountToModuleVirtual` and `RefundGas` → `SendCoinsFromModuleToAccountVirtual`. No special conditions are required. Any unprivileged user submitting an EVM transaction triggers this path.

### Recommendation

Replace the virtual transfers in `DeductFees` and `RefundGas` with real bank operations:

- `DeductFees`: call `bankKeeper.SendCoinsFromAccountToModule` (real transfer) instead of `SendCoinsFromAccountToModuleVirtual`.
- `RefundGas`: call `bankKeeper.SendCoinsFromModuleToAccount` (real transfer) instead of `SendCoinsFromModuleToAccountVirtual`.

If the virtual mechanism is intentional as a within-block optimization, add an explicit end-of-block settlement step in `EndBlock` that converts accumulated virtual balances into real bank transfers, ensuring the fee collector's real balance is funded and user real balances are properly debited before the object store is discarded.

### Proof of Concept

1. Alice has a real bank balance of exactly `gasLimit * gasPrice` (enough for one EVM tx).
2. Alice submits two EVM txs in the same block, each costing `gasLimit * gasPrice`.
3. Ante handler for tx 1: `CheckSenderBalance` reads real balance → passes. `DeductFees` → `SendCoinsFromAccountToModuleVirtual` → virtual deduction only, real balance unchanged.
4. Ante handler for tx 2: `CheckSenderBalance` reads real balance (still full, unchanged) → passes. `DeductFees` → virtual deduction only.
5. Both txs are included in the block. Alice executed two txs but her real balance was only sufficient for one.
6. At block commit, the transient object store is discarded. The fee collector's real balance received zero tokens from either tx.

// RefundGasWithPrice transfers the leftover gas to sender using the provided gas price.
func (k *Keeper) RefundGasWithPrice(
	ctx sdk.Context,
	msg *core.Message,
	leftoverGas uint64,
	gasPrice *big.Int,
	denom string,
) error {
	if gasPrice == nil {
		gasPrice = new(big.Int)
	}

	// Return EVM tokens for remaining gas, exchanged at the original rate.
	remaining := new(big.Int).Mul(new(big.Int).SetUint64(leftoverGas), gasPrice)

	switch remaining.Sign() {
	case -1:
		// negative refund errors
		return errorsmod.Wrapf(types.ErrInvalidRefund, "refunded amount value cannot be negative %d", remaining.Int64())
	case 1:
		// positive amount refund
		refundedCoins := sdk.Coins{sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(remaining))}

		// refund to sender from the fee collector module account, which is the escrow account in charge of collecting tx fees
		err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
		if err != nil {
			err = errorsmod.Wrapf(errortypes.ErrInsufficientFunds, "fee collector account failed to refund fees: %s", err.Error())
			return errorsmod.Wrapf(err, "failed to refund %d leftover gas (%s)", leftoverGas, refundedCoins.String())
		}
	default:
		// no refund, consume gas and update the tx gas meter
	}

	return nil
}

func addVirtualCoins(store storetypes.ObjKVStore, txIndex int, addr sdk.AccAddress, amt sdk.Coins) {
	key := make([]byte, len(addr)+8)
	copy(key, addr)
	binary.BigEndian.PutUint64(key[len(addr):], uint64(txIndex)) //nolint:gosec // test only

	var coins sdk.Coins
	value := store.Get(key)
	if value != nil {
		coins = value.(sdk.Coins)
	}
	coins = coins.Add(amt...)
	store.Set(key, coins)
}
