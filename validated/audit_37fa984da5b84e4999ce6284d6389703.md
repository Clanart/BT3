### Title
`ProofOfInclusion::valid()` Is a Tautology — No External Root Binding — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks that the proof's internal hash chain is self-consistent. It never compares the computed root against any external, trusted tree root. Because `root_hash()` is derived entirely from the proof's own fields, the final equality check is a mathematical tautology: it always returns `true` for any well-formed `ProofOfInclusion`, regardless of whether the claimed leaf is actually present in the real DataLayer Merkle tree.

### Finding Description

`ProofOfInclusion` is a `Streamable` (deserializable from bytes) struct exposed through both the Rust API and the Python/wasm bindings. [1](#0-0) 

Its `root_hash()` method derives the claimed root entirely from the proof's own last layer: [2](#0-1) 

The `valid()` method walks the layer chain, verifying each `calculated_hash == layer.combined_hash`, then ends with: [3](#0-2) 

After the loop, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `layer.combined_hash` inside the loop. `self.root_hash()` returns that same `last.combined_hash`. Therefore the final comparison `existing_hash == self.root_hash()` is always `true` when the loop completes without returning `false`. The function is a tautology — it never binds the proof to any external trusted root.

**Concrete forgery:** An attacker constructs a `ProofOfInclusion` with:
- `node_hash` = SHA-256 of any arbitrary leaf they wish to "prove"
- One layer: `other_hash` = any 32-byte value, `combined_hash` = `calculate_internal_hash(node_hash, side, other_hash)`

`valid()` returns `true`. `root_hash()` returns the attacker-chosen `combined_hash`, not the real tree root. Any verifier that calls only `proof.valid()` accepts the forged inclusion.

The Python binding exposes `valid()` as the sole verification method with no `valid_against_root(trusted_root)` counterpart: [4](#0-3) 

The fuzz target for proofs of inclusion also only calls `proof.valid()` without checking the root: [5](#0-4) 

### Impact Explanation

Any DataLayer client that receives a `ProofOfInclusion` from an untrusted source (network peer, RPC response, deserialized bytes) and calls `proof.valid()` to verify it will accept a completely forged proof. The attacker can claim that any arbitrary key-value pair is present in any DataLayer store, without possessing the actual tree or any valid leaf. This directly enables forged inclusion proofs and lets untrusted input prove invalid state — matching the allowed High impact for DataLayer Merkle proof logic.

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type, meaning it is routinely serialized and sent over the network between DataLayer peers. The Python API exposes only `valid()` with no root-binding variant, making it the natural and only call a developer would make. Any DataLayer client that does not separately extract `proof.root_hash()` and compare it against a locally-trusted tree root is vulnerable. The forgery requires no cryptographic break — only constructing a consistent hash chain, which is trivial.

### Recommendation

Replace the rootless `valid()` with a method that accepts a trusted external root:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == *trusted_root
}
```

The existing `valid()` should either be removed or clearly documented as an internal-consistency-only check that provides no security guarantee without a separate root comparison. The Python binding should expose `valid_against_root(root: bytes32) -> bool` as the primary verification API.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forged_proof_passes_valid() {
    // Attacker wants to "prove" this arbitrary leaf hash is in the tree
    let fake_leaf_hash: Hash = [0xAA; 32];
    let other_hash: Hash = [0xBB; 32];

    // Compute a consistent combined_hash
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_leaf_hash,
        Side::Left,
        &other_hash,
    );

    let forged = ProofOfInclusion {
        node_hash: fake_leaf_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash,
            combined_hash: combined,
        }],
    };

    // valid() returns true — no real tree involved
    assert!(forged.valid());
    // root_hash() returns attacker-controlled value, not the real tree root
    assert_eq!(forged.root_hash(), combined);
}
