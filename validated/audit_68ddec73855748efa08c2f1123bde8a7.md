### Title
EIP-7702 Durable Authorization Committed to Parent Context Despite PostTxProcessing Failure, Creating Persistent Code Delegation Without Accompanying EVM State — (File: `x/evm/keeper/state_transition.go`)

---

### Summary

In `ApplyTransaction`, when a `SetCodeTx` is processed with registered hooks, the EIP-7702 authorization effects (nonce bump + delegation code) are committed to the parent `ctx` via `commitDurableAuthorization()` even when `PostTxProcessing` fails and the main `tmpCtx` is discarded. This creates a durable, committed state inconsistency: the authority has delegation code installed in the canonical store, but the EVM call's state changes (balance transfers, storage writes) are rolled back.

---

### Finding Description

The execution path is confirmed line-by-line in production code.

**Step 1 — Two separate cache contexts are created** when `k.hooks != nil` and the tx carries authorizations:

```
tmpCtx, commit = ctx.CacheContext()                          // line 181
durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()  // line 188
cfg.DurableSetCodeAuthorizationCtx = &durableAuthorizationCtx             // line 189
``` [1](#0-0) 

Both are forked from the **same parent `ctx`**, not from each other.

**Step 2 — Inside `ApplyMessageWithConfig`, the durable stateDB is committed BEFORE the EVM call:**

```go
durableStateDB := statedb.NewWithParams(*cfg.DurableSetCodeAuthorizationCtx, ...)
k.applyDurableAuthorization(&va.auth, va.authority, durableStateDB)  // nonce bump + SetCode
durableStateDB.Commit()   // committed to durableAuthorizationCtx
``` [2](#0-1) 

Then `evm.Call(...)` runs and succeeds (no `vmErr`), and `stateDB.Commit()` writes EVM state changes into `tmpCtx`. [3](#0-2) 

**Step 3 — Back in `ApplyTransaction`, PostTxProcessing fails:**

```go
if err = k.PostTxProcessing(tmpCtx, msg, receipt); err != nil {
    res.VmError = types.ErrPostTxProcessing.Error()
    // commit() is NOT called → tmpCtx discarded
    // tmpCtxCommitted stays false
}
``` [4](#0-3) 

**Step 4 — The durable authorization is unconditionally committed when `!tmpCtxCommitted`:**

```go
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()   // ← fires here
}
``` [5](#0-4) 

`!tmpCtxCommitted` is `true` in two cases: (a) EVM call failed, (b) **PostTxProcessing failed** — the case in question. The condition does not distinguish between them.

**What `applyDurableAuthorization` writes** is exactly the nonce bump and delegation code:

```go
stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
``` [6](#0-5) 

**Net result**: The authority's nonce is permanently incremented and its code is permanently set to the EIP-7702 delegation designator in the canonical store, while every other state change from the EVM call (balance transfers, storage writes, emitted logs) is rolled back.

---

### Impact Explanation

This is a confirmed **unauthorized account/code mutation**:

1. **Persistent delegation without consent to the failed outcome**: The authority signed an authorization expecting the full transaction to either succeed or fail atomically. Instead, the delegation is installed even though the Cosmos-level outcome is a `VmError` (`ErrPostTxProcessing`). The authority's EOA now permanently executes the delegated contract's code on every future call, a state the authority never agreed to under a failed-tx scenario.

2. **Nonce permanently consumed**: The authority's nonce is bumped in the canonical store. The same authorization tuple can never be replayed, and the authority's account nonce is now out of sync with what the authority would expect after a "failed" transaction.

3. **EVM state inconsistency**: Any storage writes or balance changes that the EVM call made (and that the delegated contract may rely on for correct operation) are absent. A contract delegated to by the authority may behave incorrectly or be exploitable because its expected preconditions are not met.

Impact category: **High — EIP-7702 authorization mutation persisting across a transaction that is otherwise fully reverted.**

---

### Likelihood Explanation

The preconditions are:
- A `PostTxProcessing` hook is registered (standard in production Ethermint deployments using ERC-20/IBC hooks).
- The hook can return an error under attacker-reachable conditions (e.g., a hook that checks ERC-20 transfer events, token balances, or contract state that the EVM call can manipulate to trigger a failure path).
- The attacker controls the transaction sender (trivially true for any user) and can obtain a valid EIP-7702 authorization (either self-signed or from a cooperating/deceived authority).

The hook failure condition is the gating factor. In chains with non-trivial `PostTxProcessing` hooks (e.g., Evmos's ERC-20 module hook), crafting a call that causes the hook to error is a realistic, low-privilege operation. The attacker does not need governance or validator access.

---

### Recommendation

The guard at line 266 should only commit the durable authorization when the **overall transaction succeeded**, not merely when `tmpCtx` was not committed. The correct invariant is:

```go
// Only commit durable authorization if the tx succeeded end-to-end.
if commitDurableAuthorization != nil && tmpCtxCommitted {
    commitDurableAuthorization()
}
```

Alternatively, if EIP-7702 spec compliance requires authorization effects to survive an EVM-level `vmErr` (but not a Cosmos-level hook failure), the condition should explicitly distinguish the two failure modes:

```go
evmFailed := res.VmError != "" && res.VmError != types.ErrPostTxProcessing.Error()
if commitDurableAuthorization != nil && (tmpCtxCommitted || evmFailed) {
    commitDurableAuthorization()
}
```

### Proof of Concept

1. Register a `PostTxProcessing` hook that returns an error when a specific ERC-20 transfer event is present in the receipt logs (a realistic hook pattern).
2. Deploy contract `DelegateTarget` (attacker-controlled).
3. Authority signs an EIP-7702 authorization delegating to `DelegateTarget`.
4. Attacker submits a `SetCodeTx` where:
   - Authorization: authority → `DelegateTarget`
   - EVM call: calls a contract that emits the specific ERC-20 transfer event, triggering the hook failure
5. Observe:
   - `ApplyMessageWithConfig`: `durableStateDB.Commit()` writes nonce+1 and delegation code into `durableAuthorizationCtx` (line 510).
   - `evm.Call` succeeds; `stateDB.Commit()` writes EVM state into `tmpCtx` (line 603).
   - `PostTxProcessing` returns error; `commit()` is not called; `tmpCtx` is discarded.
   - `commitDurableAuthorization()` fires at line 267.
6. Query authority's code: `0xef0100` + `DelegateTarget` address — **present**.
7. Query storage slot written by the EVM call: **absent**.
8. Query authority's nonce: **incremented by 1**.

The authority now permanently executes `DelegateTarget`'s code on every call, with no accompanying EVM state, in a transaction the chain recorded as failed (`VmError: PostTxProcessing`).

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
