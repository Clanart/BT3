### Title
Unbounded `calls` Array in `eth_simulateV1` `SimBlock` Enables Memory Exhaustion DoS - (File: `x/evm/keeper/simulate.go`)

### Summary
The `eth_simulateV1` JSON-RPC endpoint enforces a limit on the number of simulated blocks (`MaxSimulateBlocks = 256`) but imposes **no limit on the number of calls per `SimBlock`**. In `processBlock`, two slices are allocated proportional to `len(block.Calls)` before any gas or count check is performed. An unprivileged caller can send a single request with millions of calls in one block, causing the node to allocate gigabytes of memory before the first gas check fires, potentially triggering OOM and crashing the node process.

### Finding Description

`SimBlock.Calls` is an unbounded `[]json.RawMessage` slice: [1](#0-0) 

In `processBlock`, the very first thing done after applying state overrides is allocating result and transaction slices sized to `len(block.Calls)`: [2](#0-1) 

These allocations occur **before** the per-call loop that invokes `sanitizeCall` (which enforces the block gas limit) and `applyCall` (which enforces the gas budget): [3](#0-2) 

The gas budget check in `sanitizeCall` only fires inside the loop, after the upfront allocation: [4](#0-3) 

The block-level guard (`MaxSimulateBlocks`) and the span check in `sanitizeChain` only bound the number of blocks, not the number of calls within each block: [5](#0-4) [6](#0-5) 

The backend layer also only checks block count, not call count: [7](#0-6) 

`eth_simulateV1` is in the `eth` namespace, which is enabled by default: [8](#0-7) 

### Impact Explanation

A single unauthenticated `eth_simulateV1` HTTP request containing one `SimBlock` with N = 10,000,000 calls causes the server to execute:

```go
callResults  := make([]rpctypes.SimCallResult, 10_000_000)  // ~800 MB
transactions := make([]*ethtypes.Transaction, 10_000_000)   // ~80 MB
```

before any gas check runs. The Go runtime must satisfy this allocation synchronously. If the node process runs out of memory, the OS OOM-killer terminates it. Because Ethermint runs the JSON-RPC server in the same process as the CometBFT consensus engine and the EVM keeper, a process crash takes the validator offline. Coordinated requests against multiple validators can halt the chain (no 2/3 quorum). This maps to the Critical allowed impact: "block-processing path can halt the chain."

Even without a full crash, repeated requests cause GC pressure and latency spikes that degrade block production.

### Likelihood Explanation

- `eth_simulateV1` is enabled by default in the `eth` namespace.
- No authentication is required.
- A single HTTP POST with a large JSON body is sufficient; no on-chain funds or keys are needed.
- The attack is trivially scriptable and requires no knowledge of chain state.
- The `EVMTimeout` context deadline does not prevent the upfront memory allocation, which is synchronous and precedes any EVM execution. [9](#0-8) 

### Recommendation

1. **Add a per-block call count limit** in `processBlock` (or in `sanitizeChain`) before allocating result slices:
   ```go
   const MaxCallsPerBlock = 1000
   if len(block.Calls) > MaxCallsPerBlock {
       return nil, nil, nil, &rpctypes.ClientLimitExceededError{Message: "too many calls in block"}
   }
   ```
2. **Move slice allocation inside the loop** or use lazy append to avoid upfront allocation for the full declared count.
3. **Enforce an HTTP request body size limit** on the JSON-RPC server to bound the raw JSON payload size for `eth_simulateV1`.

### Proof of Concept

```python
import requests, json

# Build a SimBlock with 5,000,000 zero-gas calls
calls = [json.dumps({"from": "0x" + "aa"*20, "to": "0x" + "bb"*20}) for _ in range(5_000_000)]

payload = {
    "jsonrpc": "2.0",
    "method": "eth_simulateV1",
    "params": [
        {"blockStateCalls": [{"calls": calls}]},
        "latest"
    ],
    "id": 1
}

# Single request; server allocates ~400 MB before the first gas check fires.
r = requests.post("http://localhost:8545", json=payload, timeout=60)
print(r.status_code, r.text[:200])
```

The server allocates `len(calls) * sizeof(SimCallResult)` bytes at `x/evm/keeper/simulate.go:178-179` before the gas budget is consulted. With 5 million calls this is ~400 MB per request; concurrent requests multiply the effect. [2](#0-1) [1](#0-0)

func (sim *Simulator) sanitizeCall(call *evmtypes.TransactionArgs, state *statedb.StateDB, header *ethtypes.Header, remainingBlockGas *uint64) error {
	if call.Nonce == nil {
		nonce := state.GetNonce(call.GetFrom())
		call.Nonce = (*hexutil.Uint64)(&nonce)
	}
	// Let the call run wild unless explicitly specified.
	if call.Gas == nil {
		call.Gas = (*hexutil.Uint64)(remainingBlockGas)
	}
	if *remainingBlockGas < uint64(*call.Gas) {
		return &rpctypes.BlockGasLimitReachedError{
			Message: fmt.Sprintf(
				"block gas limit reached: remaining: %d, required: %d",
				*remainingBlockGas,
				*call.Gas,
			),
		}
	}
	// Clamp to the cross-block gas budget.
	gas := sim.budget.Cap(uint64(*call.Gas))
	call.Gas = (*hexutil.Uint64)(&gas)

	return call.CallDefaults(0, header.BaseFee, sim.chainConfig.ChainID)
