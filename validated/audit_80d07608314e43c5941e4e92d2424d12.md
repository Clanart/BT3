### Title
Missing shard-identity check on `prev_chunk_header` in `set_state_header` allows unprivileged peer to corrupt `prev_chunk_height_included` and truncate incoming receipt application — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` verifies that `prev_chunk_header` is a member of `prev_block_header.chunk_headers_root()` via `verify_path`, but never checks that `prev_chunk_header.shard_id() == shard_id`. Because `verify_path` is a pure membership proof with no position/index binding, an attacker can supply a chunk header belonging to a different shard at a different slot in the same merkle tree, pass step 3b, and install a header whose `prev_chunk_height_included` is wrong. This corrupts the lower bound used to collect incoming receipts, causing the syncing node to apply receipts from the wrong height range and arrive at a divergent post-sync state.

### Finding Description

**Production entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called when a syncing node receives a `ShardStateSyncResponseHeader` from any peer.

**Step 3b** (lines 412–425):

``` [1](#0-0) 

`verify_path` is defined as:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [2](#0-1) 

This is a pure hash-chain membership check. It does **not** verify which leaf index (shard slot) the item occupies. A proof for shard X's slot is indistinguishable from a proof for shard Y's slot as far as this function is concerned — both produce the same root if the item hash and sibling hashes are correct.

A grep over `set_state_header` confirms there is **no** `prev_chunk_header.shard_id() == shard_id` guard anywhere in the function.

**Step 4g** (line 507) uses `prev_chunk_header.height_included()` as the lower bound for the receipt chain:

```rust
if header.height() != prev_chunk_header.map_or(0, |h| h.height_included()) {
    return Err(...)
}
``` [3](#0-2) 

**`set_state_finalize`** then applies the receipts stored in the header:

```rust
for incoming_receipt_proof in &incoming_receipts_proofs {
    let ReceiptProofResponse(hash, _) = incoming_receipt_proof;
    let block_header = self.chain_store_update.get_block_header(hash)?;

        receipt_proof_responses.push(incoming_receipt_proof.clone());

}
let receipts = collect_receipts_from_response(&receipt_proof_responses);
``` [4](#0-3) 

The receipts applied to the chunk come entirely from the attacker-supplied `incoming_receipts_proofs`, bounded below by the attacker-controlled `prev_chunk_header.height_included()`.

### Impact Explanation

Let H_Y = the true `height_included` for shard Y's prev chunk, and H_X = `height_included` for shard X's prev chunk in the same block, where H_X > H_Y.

The attacker:
1. Supplies `prev_chunk_header` = shard X's chunk header (H_X > H_Y).
2. Supplies `prev_chunk_proof` = the valid merkle proof for shard X's slot in `prev_block.chunk_headers_root()`. This proof is publicly computable from on-chain data.
3. Supplies `incoming_receipts_proofs` covering only blocks from `sync_hash` down to H_X, omitting blocks at heights H_Y … H_X − 1.
4. Steps 4b–4f pass because all supplied receipt proofs are valid on-chain proofs for `shard_id`.
5. Step 4g passes because the chain terminates at H_X = `prev_chunk_header.height_included()`.

The header is stored. During `set_state_finalize`, receipts from blocks at heights H_Y … H_X − 1 are never applied. The node executes `apply_chunk` with a truncated receipt set, producing a state root that diverges from the canonical chain. The node cannot produce valid chunks and is effectively excluded from consensus.

### Likelihood Explanation

Any peer in the p2p network can respond to a state header request. The attacker needs only:
- Knowledge of the `sync_hash` and `shard_id` being requested (observable from network traffic).
- On-chain data to construct valid chunk headers and merkle proofs (publicly available).
- The ability to respond before or instead of honest peers.

No validator key, node admin access, or trusted storage-service role is required.

### Recommendation

After the `verify_path` check in step 3b, add an explicit shard identity assertion:

```rust
if prev_chunk_header.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: prev_chunk_header belongs to wrong shard".into(),
    ));
}
```

The same guard should be applied to the `chunk` field in step 3a (its `shard_id()` is also never compared to the `shard_id` parameter).

### Proof of Concept

Given a target `shard_id = Y` and a known `sync_hash`:

1. Fetch `prev_block` = block at `chunk.height_included() - 1` from the chain.
2. Let shard X ≠ Y be any shard in `prev_block` where `prev_block.chunks()[X].height_included() > prev_block.chunks()[Y].height_included()`.
3. Compute `(chunk_headers_root, chunk_proofs) = merklize(prev_block.chunks().map(ChunkHashHeight))`.
4. Set `prev_chunk_header = prev_block.chunks()[X]` and `prev_chunk_proof = chunk_proofs[X]`.
5. Collect `incoming_receipts_proofs` for shard Y from `sync_hash` down to `prev_block.chunks()[X].height_included()` (omitting the lower range).
6. Send the crafted `ShardStateSyncResponseHeaderV2` to the syncing node.
7. `verify_path(prev_block_header.chunk_headers_root(), chunk_proofs[X], ChunkHashHeight(chunks[X].chunk_hash(), chunks[X].height_included()))` returns `true` — step 3b passes.
8. Step 4g passes because the receipt chain terminates at `chunks[X].height_included()`.
9. The header is stored; `set_state_finalize` applies a truncated receipt set; the node's post-sync state diverges from canonical.
