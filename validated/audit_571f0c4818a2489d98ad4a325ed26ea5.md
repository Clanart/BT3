### Title
`MerkleBlob::upsert` Missing Hash-Collision Guard Allows Two Leaves to Share the Same Hash — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::upsert` removes the old leaf's hash from the cache before writing the new hash, but **never checks whether the new hash is already held by a different leaf**. This allows a caller to silently create two leaves with identical hashes, corrupting the Merkle tree root and invalidating all subsequent inclusion/exclusion proofs.

### Finding Description

`insert` correctly guards against hash collisions: [1](#0-0) 

`upsert` does not apply the same guard. Its full update sequence is:

1. **`remove_leaf(&leaf)`** — removes the old hash (`H1`) from `leaf_hash_to_index` and the old key from `key_to_index`. [2](#0-1) 

2. **No collision check** — `new_hash` is never tested against `leaf_hash_to_index` for a pre-existing entry belonging to a *different* key. [3](#0-2) 

3. **`insert_entry_to_blob`** calls `add_leaf`, which does an unconditional `HashMap::insert` for the new hash — silently overwriting the existing `H2 → K2_index` mapping if `H2` was already present. [4](#0-3) [5](#0-4) 

**Concrete call sequence:**

```
insert(K1, V1, H1, Auto)   // blob: K1→H1
insert(K2, V2, H2, Auto)   // blob: K1→H1, K2→H2
upsert(K1, V1_new, H2)     // no error returned
```

After step 3:
- The raw blob contains **two leaf nodes both carrying hash `H2`**.
- `leaf_hash_to_index` has only **one** entry for `H2` (pointing to K1's slot), so K2's leaf is now unreachable via the hash index.
- `check_integrity` detects this: `leaf_hash_to_index_cache_length` (1 entry for H2) will be less than `leaf_count` (2 leaves), triggering `IntegrityLeafHashToIndexCacheLength`. [6](#0-5) 

### Impact Explanation

The DataLayer Merkle blob is the authoritative structure for computing tree roots committed on-chain. With two leaves sharing hash `H2`:

- **Tree root is corrupted**: internal node hashes propagate from leaf hashes; a duplicate leaf hash produces a root that does not correspond to the actual key-value set.
- **Inclusion proofs are forgeable**: `get_proof_of_inclusion` resolves a leaf via `leaf_hash_to_index`; after the overwrite, the mapping for `H2` points to K1's slot, so a proof for K2 is silently redirected.
- **Exclusion proofs are invalid**: a verifier checking that a key is absent may receive a structurally valid proof against a corrupted root.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

Any caller of the public `upsert` API who controls both the key being updated and the target hash can trigger this. In the DataLayer, hashes are derived from key-value content; an operator or client that knows `H2` (the hash of any existing leaf) can craft an `upsert` call with that hash. No privileged access is required — `upsert` is a standard DataLayer write operation.

### Recommendation

Add a hash-collision check inside `upsert` **before** (or immediately after) `remove_leaf`, verifying that `new_hash` is not already present in `leaf_hash_to_index` for a different key:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // Guard: new_hash must not already belong to a different leaf
    if *new_hash != leaf.hash && self.block_status_cache.contains_leaf_hash(new_hash) {
        return Err(Error::HashAlreadyPresent());
    }

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    leaf.value = value;
    block.node = Node::Leaf(leaf);
    self.insert_entry_to_blob(leaf_index, &block)?;
    ...
}
```

The check must happen **before** `remove_leaf` so that `H2` is still present in the cache when the test is performed.

### Proof of Concept

```rust
#[test]
fn test_upsert_hash_collision() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    let h1 = Hash::from([1u8; 32]);
    let h2 = Hash::from([2u8; 32]);

    blob.insert(KeyId(1), ValueId(10), &h1, InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(2), ValueId(20), &h2, InsertLocation::Auto {}).unwrap();

    // Upsert K1 to use H2 — should return HashAlreadyPresent, but currently succeeds
    let result = blob.upsert(KeyId(1), ValueId(99), &h2);
    assert!(result.is_err(), "expected HashAlreadyPresent, got Ok");

    // Without the fix, integrity check reveals the corruption:
    blob.check_integrity().expect_err("integrity should fail with duplicate hashes");
}
