### Title
Missing Recipient Validation Before Irreversible Token Burn/Lock in `initTransfer` Causes Permanent Fund Loss - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` and `initTransfer1155` accept a free-form `string calldata recipient` with no validation before burning or locking the caller's tokens. An empty string, a malformed string, or any value that cannot be parsed as a valid `OmniAddress` on the NEAR side causes the tokens to be permanently destroyed with no on-chain recovery path.

### Finding Description
`initTransfer` performs all token movement — burning bridge tokens or locking native ERC-20s — before emitting the `InitTransfer` event, and it never validates the `recipient` parameter:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    uint128 fee,
    uint128 nativeFee,
    string calldata recipient,   // ← no length or format check
    string calldata message
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    if (fee >= amount) { revert InvalidFee(); }
    // tokens burned/locked here ...
    emit BridgeTypes.InitTransfer(..., recipient, message);
}
``` [1](#0-0) 

The NEAR side's `fin_transfer_callback` decodes the prover result and expects the `recipient` field to be a valid `OmniAddress`. If the string is empty or malformed, the prover cannot construct a valid `InitTransferMessage`, and `fin_transfer_callback` panics with `InvalidProofMessage`:

```rust
let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) else {
    env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str())
};
``` [2](#0-1) 

At that point the tokens are already gone from EVM and no NEAR-side state has been written. There is no admin refund function, no reversal path, and no recovery mechanism in the contract.

The same flaw exists in `initTransfer1155`. [3](#0-2) 

The `OmniAddress` type on the NEAR side has an `is_zero()` helper and a strict `FromStr` parser that rejects unknown chain prefixes and malformed addresses, confirming that the EVM contract is the only place where the string could be validated before the irreversible operation. [4](#0-3) 

The Solana SECURITY.md explicitly acknowledges the identical pattern for the Solana `InitTransferPayload` as a known issue ("An invalid recipient causes the transfer to fail on the NEAR side after tokens are locked/burned on Solana. Manual intervention would be needed"), confirming the protocol team is aware of the class but has not addressed it on the EVM side. [5](#0-4) 

The EVM SECURITY.md contains no corresponding acknowledgement for `initTransfer`. [6](#0-5) 

### Impact Explanation
Any user who calls `initTransfer` with `recipient = ""` (empty string) or any string that is not a valid `OmniAddress` (e.g., `"invalid"`, `"0x1234"` without a chain prefix, a mistyped chain name) will have their tokens permanently destroyed. The EVM contract burns or locks the tokens atomically in the same transaction that emits the event. The NEAR side cannot finalize the transfer because the recipient cannot be parsed. No on-chain recovery path exists. This is a permanent, irrecoverable lock of user funds in the bridge flow — matching the Critical impact tier.

### Likelihood Explanation
The `recipient` parameter is a raw string typed by the caller. Any UI bug, copy-paste error, or direct contract interaction with a malformed address string triggers the loss. The call is permissionless and requires no special role. The Solana team already acknowledged the identical class as causing fund loss requiring manual intervention, indicating the scenario is realistic in practice.

### Recommendation
Add a non-empty length guard and, where feasible, a basic format check (e.g., presence of a `:` separator matching a known chain prefix) before any token movement:

```solidity
if (bytes(recipient).length == 0) revert InvalidRecipient();
```

A stricter approach would maintain an on-chain set of supported chain-prefix strings and reject any `recipient` whose prefix is not in that set, mirroring the NEAR-side `ChainKind` enum. [7](#0-6) 

### Proof of Concept
1. Deploy or use the existing `OmniBridge` on any supported EVM chain.
2. Acquire any bridge token (e.g., wrapped NEAR).
3. Call `initTransfer(tokenAddress, amount, 0, 0, "", "")`.
4. The transaction succeeds: `BridgeToken.burn` is called, the user's balance drops to zero, and `InitTransfer` is emitted with `recipient = ""`.
5. A relayer attempts to submit the proof to the NEAR `fin_transfer` endpoint.
6. The NEAR prover cannot construct a valid `OmniAddress` from `""`, so `fin_transfer_callback` panics with `ERR_INVALID_PROOF_MESSAGE`.
7. No NEAR-side state is written. The tokens are gone. No refund function exists.

# Security Notes

## Design Decisions (Non-Issues)

These patterns have been reviewed and confirmed as intentional. Do not flag or "fix" them.

- **Fee-on-transfer tokens not supported**: `initTransfer` emits the requested `amount`, not the actual received balance. Fee-on-transfer and rebasing tokens are intentionally unsupported
- **`logMetadata` and `deployToken` are permissionless**: Anyone can call `logMetadata` for any ERC20, and anyone can submit a valid MPC signature to `deployToken`. This is by design — the bridge is fully permissionless
- **`ENearProxy.burn` uses empty NEAR recipient**: `eNear.transferToNear(amount, "")` is intentional — `transferToNear` is a legacy method used purely as a burn mechanism. The actual NEAR recipient is tracked in the OmniBridge `InitTransfer` event
- **`deployToken` signature has no chain ID**: Metadata signatures are intentionally chain-agnostic — one NEAR-side signature deploys the same token on all EVM chains

## Known Issues

Low-severity items acknowledged but not yet addressed:

- **`addCustomToken` can overwrite existing mappings** (H-01): Admin-only function. No existence check — calling with an already-mapped `nearTokenId` silently overwrites `nearToEthToken`. Accepted as operational risk
- **`pause(flags)` replaces all flags** (H-02): `_pause(flags)` does full replacement, not bitwise OR. Calling `pause(PAUSED_INIT_TRANSFER)` when `PAUSED_FIN_TRANSFER` is set will unpause finTransfer. Use `pauseAll()` for emergencies
- **`BridgeToken.initialize` stores metadata redundantly** (L-01): `__ERC20_init(name_, symbol_)` writes to parent storage that is never read (getters are overridden). Minor gas waste on init
- **`require` strings instead of custom errors** (L-02): Several locations use `require` with string messages instead of custom errors (`OmniBridge.sol:150,204,556`, `SelectivePausableUpgradable.sol:100,107`, `ENearProxy.sol:56,76,86`)
- **`OmniBridgeWormhole` has no `__gap`** (L-04): Three storage variables with no gap array. Safe as a leaf contract but would need a gap if inherited from
- **`PayloadType.ClaimNativeFee` defined but unused** (L-05): Enum value 2 is never referenced. Native fees are recovered via `finTransfer` with `tokenAddress=address(0)`
```
