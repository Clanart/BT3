### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Validation for Bulk Entries, Corrupting DataLayer Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` applies duplicate-key and duplicate-hash guards only to the first two items it processes (via the regular `insert` path). All remaining items in the batch are written directly into the blob without any uniqueness check. An attacker who controls the batch input can therefore insert multiple leaves sharing the same `KeyId` or `Hash`, producing a structurally invalid Merkle tree whose root is computed over duplicate entries. This corrupts the committed DataLayer root and enables ambiguous or forged inclusion proofs.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) enforces two guards before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `self.insert(…)` for only the first two items (when the tree has ≤ 1 existing leaf). Every subsequent item in the batch is written directly via `insert_entry_to_blob` with **no duplicate check**:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        metadata: NodeMetadata { node_type: NodeType::Leaf, dirty: false },
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

Because `block_status_cache` is not consulted, a caller may supply a `Vec` containing two entries with the same `KeyId`. The first two items (popped from the tail) pass through `insert` and are registered in the cache; the duplicate appearing earlier in the `Vec` is then written unconditionally into the blob. The resulting blob contains two `LeafNode`s with identical keys, which:

1. Causes `check_integrity` to detect a mismatch between `leaf_count` and `key_to_index_cache_length` — but only if integrity is explicitly checked.
2. Causes `calculate_lazy_hashes` to hash both duplicate leaves into the tree, producing a root that commits to an invalid set.
3. Makes `get_proof_of_inclusion` return a proof for one of the two leaves non-deterministically, allowing the prover to choose which value to prove.

The Python binding exposes this path directly to caller-controlled data:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

---

### Impact Explanation

A DataLayer store owner (or any code path that feeds untrusted data into `batch_insert`) can commit a Merkle root that encodes duplicate keys with different values. Downstream verifiers calling `get_proof_of_inclusion` receive a proof that is structurally valid against the committed root but proves a value of the attacker's choosing — either the "real" value or the injected duplicate. This satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The Python binding `py_batch_insert` is the primary DataLayer write path. Any caller who supplies a list with a repeated `KeyId` triggers the bug. No special privilege beyond store ownership is required. The regular `insert` path correctly rejects duplicates, so the discrepancy is a latent design gap in the batch path rather than an intentional relaxation.

---

### Recommendation

Add the same duplicate guards at the top of the bulk loop in `batch_insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing write logic ...
    self.block_status_cache.add_leaf(new_leaf_index, LeafNode { key, value, hash, parent: Parent(None) });
}
```

Alternatively, pre-deduplicate and validate the input vector before entering the bulk path, and ensure `block_status_cache` is kept in sync with every write.

### Proof of Concept

```python
from chia_rs import MerkleBlob
from chia_rs.sized_bytes import bytes32
import hashlib

blob = MerkleBlob(bytearray())

# KeyId 0 appears twice with different values and hashes
keys_values = [(0, 100), (1, 200), (0, 999)]   # key 0 duplicated
hashes = [
    bytes32(hashlib.sha256(b"a").digest()),
    bytes32(hashlib.sha256(b"b").digest()),
    bytes32(hashlib.sha256(b"c").digest()),   # different hash, same key
]

# batch_insert processes (1,200) and (0,999) via insert() [tail-pop],
# then writes (0,100) directly — no duplicate check fires.
blob.batch_insert(keys_values, hashes)
blob.calculate_lazy_hashes()

# Two leaves with KeyId=0 now exist; proof picks one arbitrarily.
proof = blob.get_proof_of_inclusion(0)
print("proof valid:", proof.valid())   # True — but root is corrupted
```
