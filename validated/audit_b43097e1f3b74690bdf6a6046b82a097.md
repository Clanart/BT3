### Title
Missing Token Registration Check in `initTransfer` Allows Permanent Freezing of User Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` accepts any arbitrary ERC20 token address without verifying that the token is registered in the bridge's token registry (`ethToNearToken` mapping). A user can lock tokens that have no corresponding NEAR-side registration, causing the NEAR `fin_transfer_callback` to panic and leaving the funds permanently frozen in the EVM bridge contract.

---

### Finding Description

`OmniBridge.initTransfer()` performs three checks on `tokenAddress` — custom minter, bridge token, or plain ERC20 — but none of these branches verify that the token is actually registered in the bridge's cross-chain token registry: [1](#0-0) 

The `else` branch at line 406–412 accepts **any** ERC20 token and transfers it into the bridge contract with no guard:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

The bridge maintains two mappings for registered tokens: [2](#0-1) 

A token is only added to `ethToNearToken` / `nearToEthToken` via `deployToken()` (MPC-signed) or `addCustomToken()` (admin-only). Neither is called during `initTransfer`. There is no `require(bytes(ethToNearToken[tokenAddress]).length > 0)` guard anywhere in `initTransfer`.

On the NEAR side, `fin_transfer_callback` immediately looks up `token_decimals` for the incoming token: [3](#0-2) 

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

Because the unregistered token has no entry in `token_decimals`, this call panics unconditionally. The NEAR transaction reverts, but the ERC20 tokens are already held by the EVM bridge contract with no recovery path.

The same structural gap exists in `initTransfer1155`: there is no check that `deterministicToken` has a corresponding entry in `multiTokens` before accepting the ERC1155 transfer: [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Any ERC20 token transferred via `initTransfer` with an unregistered address is locked inside the EVM bridge contract. The NEAR-side `fin_transfer_callback` panics on `TokenDecimalsNotFound`, so the cross-chain settlement never completes. There is no admin rescue function visible in the contract for arbitrary ERC20 tokens. The funds are irrecoverably frozen.

---

### Likelihood Explanation

**High.** The entry point is a public, unpermissioned function callable by any user. No special role, key, or privileged access is required. A user who mistakenly (or deliberately) calls `initTransfer` with a token that has not yet been deployed/registered through the bridge will trigger this path. The `else` branch is the default for all non-bridge, non-custom-minter ERC20 tokens, which is the common case for native tokens being bridged for the first time before their metadata is registered.

---

### Recommendation

Add a registration check at the top of `initTransfer` (and `initTransfer1155`) before accepting any token:

```solidity
// For ERC20:
require(
    tokenAddress == address(0) ||
    isBridgeToken[tokenAddress] ||
    customMinters[tokenAddress] != address(0) ||
    bytes(ethToNearToken[tokenAddress]).length > 0,
    "ERR_TOKEN_NOT_REGISTERED"
);

// For ERC1155:
require(
    multiTokens[deterministicToken].tokenAddress != address(0),
    "ERR_TOKEN_NOT_REGISTERED"
);
```

This mirrors the pattern already used in `setMetadata`: [5](#0-4) 

---

### Proof of Concept

1. Deploy or obtain any ERC20 token `UNREGISTERED_TOKEN` that is **not** in `ethToNearToken` and is not a bridge token or custom minter token.
2. Approve the `OmniBridge` contract to spend `amount` of `UNREGISTERED_TOKEN`.
3. Call `OmniBridge.initTransfer(UNREGISTERED_TOKEN, amount, 0, 0, "<near_recipient>", "")`.
4. The `else` branch executes: `IERC20(UNREGISTERED_TOKEN).safeTransferFrom(msg.sender, address(this), amount)` — tokens are now held by the bridge.
5. An `InitTransfer` event is emitted with `tokenAddress = UNREGISTERED_TOKEN`.
6. A relayer picks up the event and calls `fin_transfer` on the NEAR bridge contract.
7. `fin_transfer_callback` calls `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` — panics because `UNREGISTERED_TOKEN` has no NEAR-side registration.
8. The NEAR transaction reverts. The EVM bridge holds `amount` of `UNREGISTERED_TOKEN` with no mechanism to release it. Funds are permanently frozen. [6](#0-5)
