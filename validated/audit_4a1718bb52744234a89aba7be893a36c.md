### Title
Reverted L1 Handler Transactions Omit `consume_l1_to_l2_message`, Leaving L1→L2 Messages Permanently Unconsumed — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `execute_l1_handler_transaction`, when the sequencer marks a transaction as reverted (`is_reverted != FALSE`), the OS returns immediately without ever calling `consume_l1_to_l2_message`. This is a direct analog to the LeXscrow bug: one piece of state (the transaction's reverted status) is acted upon, while a coupled piece of state (the L1→L2 message consumption record in `outputs.messages_to_l2`) is left untouched. The L1 StarkNet core contract uses the OS output's `messages_to_l2` list to mark messages as consumed; if a message is absent from that list, it remains permanently pending on L1 with no on-chain mechanism to recover the locked funds.

---

### Finding Description

`execute_l1_handler_transaction` begins with an early-exit guard:

```cairo
%{ StartTx %}
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [1](#0-0) 

When this branch is taken, the function returns before reaching the two critical steps that appear only in the non-reverted path:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
...
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(...);
``` [2](#0-1) 

`consume_l1_to_l2_message` is the sole function that writes a `MessageToL2Header` entry into `outputs.messages_to_l2`:

```cairo
assert [outputs.messages_to_l2] = MessageToL2Header(
    from_address=[execution_context.calldata],
    to_address=execution_info.contract_address,
    nonce=nonce,
    selector=execution_info.selector,
    payload_size=payload_size,
);
``` [3](#0-2) 

The `outputs` implicit argument is the `OsCarriedOutputs` struct that accumulates all L1↔L2 messages across the block:

```cairo
struct OsCarriedOutputs {
    messages_to_l1: MessageToL1Header*,
    messages_to_l2: MessageToL2Header*,
}
``` [4](#0-3) 

This carried output is serialized into the final OS output via `serialize_messages`, which computes the segment size as the pointer difference between the final and initial `messages_to_l2` pointers:

```cairo
let messages_to_l2_segment_size = (
    final_carried_outputs.messages_to_l2 - initial_carried_outputs.messages_to_l2
);
``` [5](#0-4) 

Because the reverted handler never advances `outputs.messages_to_l2`, the message is absent from the OS output. The L1 StarkNet core contract verifies this output to determine which L1→L2 messages were consumed; a message not present in the output is never marked consumed on L1.

A developer-acknowledged TODO in the same function confirms the incomplete handling of reverted L1 handlers:

```cairo
// TODO(Yoni): currently, the contract state is not fetched for reverted L1 handlers.
//   Once block hash is supported, we should fetch the contract state for them as well.
``` [6](#0-5) 

Compare with how invoke transactions handle reverts: the nonce is incremented and fees are charged **regardless** of revert status, because those state transitions must always occur. The L1 message consumption is the exact equivalent for L1 handlers — it must always occur — but it is gated behind the non-reverted path only.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

An L1→L2 message carries value (ETH or STRK locked in the L1 StarkNet bridge contract). The L1 bridge releases or marks funds only when the OS output confirms the message was consumed. If the OS output never includes the message (because the handler was reverted and the OS returned early), the bridge never releases the lock. The message remains in the L1 pending state indefinitely. Because the sequencer will continue to include the reverted handler in subsequent blocks (the message is still pending on L1), and each time the OS will again skip consumption, the message can never be consumed through normal protocol operation. The user's bridged funds are permanently frozen in the L1 contract.

---

### Likelihood Explanation

**High.**

Any unprivileged user can trigger this path:

1. Deploy an L2 contract whose `@l1_handler` entry point unconditionally reverts (e.g., `assert 1 = 0`).
2. Send an L1→L2 message targeting that contract from L1 (a permissionless L1 call).
3. The sequencer picks up the pending message, attempts execution, observes the revert, and sets `is_reverted = TRUE` via the `%{ IsReverted %}` hint.
4. The OS takes the early-exit branch, omits `consume_l1_to_l2_message`, and the message is absent from the block's OS output.
5. The L1 bridge never marks the message consumed; funds are frozen.

No privileged role, leaked key, or operator collusion is required. The attacker controls only the L2 contract code and the L1 message send — both are permissionless protocol inputs.

---

### Recommendation

Move `consume_l1_to_l2_message` (and the minimal setup required to call it: reading the contract address, nonce, and calldata) **before** the `is_reverted` early-exit check, or add a separate early-exit path that still calls `consume_l1_to_l2_message` before returning. This mirrors the pattern used for invoke transactions, where nonce increment and fee charge occur unconditionally regardless of revert status. Specifically, the message header should be written to `outputs.messages_to_l2` for every L1 handler transaction included in a block, reverted or not, so the L1 bridge always marks the message consumed.

---

### Proof of Concept

1. **Attacker deploys** an L2 contract with an `@l1_handler` that always reverts:
   ```
   @l1_handler
   func handle_deposit(from_address: felt, amount: felt) {
       assert 1 = 0;  // always reverts
   }
   ```

2. **Attacker sends** an L1→L2 message from L1 to this contract, locking ETH in the L1 bridge.

3. **Sequencer** picks up the pending message, includes it as an L1 handler transaction, executes it, observes the revert, and sets `is_reverted = 1` in the hint `%{ IsReverted %}`.

4. **OS** enters `execute_l1_handler_transaction`, reads `is_reverted = 1`, and takes the early-exit branch at lines 387–390, returning without calling `consume_l1_to_l2_message`. [7](#0-6) 

5. **OS output** (`outputs.messages_to_l2`) does not contain the message. `serialize_messages` computes a segment size of zero for this message. [5](#0-4) 

6. **L1 bridge** verifies the OS output, finds the message absent from the consumed list, and does not release the lock. The ETH remains frozen in the bridge contract permanently.

7. On every subsequent block, the sequencer re-includes the still-pending message, the handler reverts again, and the cycle repeats — the message is never consumed.
