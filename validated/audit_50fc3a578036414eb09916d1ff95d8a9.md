### Title
Missing Class Existence Validation in `execute_replace_class` Allows Setting Undeclared Class Hash — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS program updates a contract's class hash in the state without verifying that the new class hash corresponds to a declared class. This is structurally analogous to the external report: a state-modifying operation changes a critical accounting field (the contract's class hash) without a required post-operation validity check. A contract can replace its own class with an arbitrary, undeclared hash. Any subsequent call to that contract will cause the OS to fail when looking up the compiled class, permanently freezing all funds held by the contract.

---

### Finding Description

In `execute_replace_class`, the OS reads the requested `class_hash` from the syscall request and immediately writes it into `contract_state_changes` via `dict_update`, with no check that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment explicitly acknowledges the missing check. The OS never consults `contract_class_changes` to confirm that `class_hash` has a corresponding compiled class hash entry before committing the state update.

When a subsequent transaction calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

If `class_hash` was never declared, `dict_read` returns the default value `0`. `find_element` then searches for a compiled class with hash `0`, which does not exist, causing an OS-level assertion failure. Unlike `search_sorted_optimistic` (which returns a success flag), `find_element` is a hard assertion — it panics if the key is absent. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds funds (ERC-20 balances, ETH, NFTs) and calls `replace_class` with an undeclared class hash will have its class hash corrupted in the committed state. All future calls to that contract — including withdrawal or transfer functions — will fail at the OS execution layer because the compiled class cannot be resolved. The funds are permanently inaccessible with no recovery path, since the state transition is already committed and the OS has no mechanism to revert a committed class hash to a valid one without a new valid transaction, which itself cannot execute.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is a standard, permissionless syscall callable by any contract on itself. No privileged role is required. A malicious contract deployer can deploy a contract, fund it (or have users fund it), and then call `replace_class` with an arbitrary felt value. The missing check is explicitly flagged in the source with a TODO, confirming it is a known gap in the OS validation logic. The attack requires only the ability to deploy and call a contract — available to any unprivileged user.

---

### Recommendation

In `execute_replace_class`, before committing the `dict_update`, verify that the requested `class_hash` has a corresponding entry in `contract_class_changes` (i.e., it has been declared). This mirrors the validation that `execute_entry_point` relies on downstream:

```cairo
// Verify the class has been declared.
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This check should be inserted after reading `class_hash` from the request and before the `dict_update` call, resolving the acknowledged TODO at line 898.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts deposits from users and implements a `drain` function.
2. Users deposit funds into `MaliciousVault`, trusting its declared class.
3. Attacker calls `MaliciousVault.__execute__` → internally calls the `replace_class` syscall with `class_hash = 0xDEAD` (an arbitrary, never-declared felt).
4. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xDEAD` from the syscall request.
   - Skips the missing existence check (the TODO).
   - Writes `StateEntry(class_hash=0xDEAD, ...)` into `contract_state_changes` for `MaliciousVault`.
5. Block is finalized. `MaliciousVault`'s on-chain class hash is now `0xDEAD`.
6. Any user attempts to call `MaliciousVault.withdraw()`:
   - OS calls `execute_entry_point` for `MaliciousVault`.
   - `dict_read(contract_class_changes, key=0xDEAD)` → returns `0`.
   - `find_element(..., key=0)` → hard assertion failure; no compiled class with hash `0` exists.
   - OS proof for the block containing this call fails.
7. All funds in `MaliciousVault` are permanently frozen — no transaction can ever successfully execute against the contract again.
