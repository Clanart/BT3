**Conclusion**

The attack requires `incoming_receipts_proofs` to be empty for shard A (no cross-shard receipts in the relevant epoch window), which is a realistic but specific condition. Under that condition, a malicious peer — requiring no special privileges — can send shard A's legitimate header in response to a request for shard B's header, bypass all validation in `set_state_header`, and cause shard A's state to be installed into shard B's trie. ### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows a malicious peer to install shard A's state into shard B's trie — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to shard A while the caller-supplied `shard_id` is B. Because no check compares `chunk.shard_id()` to the caller's `shard_id`, and because the receipt-proof loop is a no-op when `incoming_receipts_proofs` is empty, the header passes all validation and is persisted under `StateHeaderKey(shard_id=B, sync_hash)` carrying shard A's `state_root`. Every downstream step — part validation, part storage, and trie application — then operates on shard A's state root while believing it is working on shard B.

### Finding Description
In `set_state_header` (`chain/chain/src/state_sync/adapter.rs:368–532`), the five validation steps are:

1. `validate_chunk_proofs` — checks the chunk's internal consistency, not its shard identity relative to the caller.
2. `verify_path` against `chunk_headers_root` — a position-independent Merkle proof; shard A's chunk passes because it is legitimately in the block.
3. `prev_chunk` inclusion — same, position-independent.
4. Receipt-proof loop — uses caller-supplied `shard_id=B` to recompute `ReceiptList(B, receipts_A)`, which would mismatch the proof root **only when the list is non-empty**. When `incoming_receipts_proofs` is empty the loop body never executes.
5. `validate_state_root_node` — validates shard A's `state_root_node` against shard A's `prev_state_root`; passes trivially.

None of these steps assert `chunk.shard_id() == shard_id`.

After acceptance, `run_state_sync_for_shard` extracts `state_root = header.chunk_prev_state_root()` (shard A's root) and `num_parts = header.num_state_parts()` (shard A's part count) and passes them to `ensure_shard_part_downloaded_single_attempt(shard_id=B, …, state_root=A_root, …)`.

Inside that function, `validate_state_part_impl` ignores `shard_id` entirely and validates only against `state_root`. A part that is cryptographically valid for shard A's trie passes. It is stored under `StatePartKey(sync_hash, shard_id=B, part_id)` and later applied to shard B's trie via `apply_state_part(shard_id=B, state_root=A_root, …)`.

### Impact Explanation
Shard B's trie is overwritten with shard A's key-value space. Any node performing state sync for shard B that contacts the malicious peer will operate on the wrong state after `set_state_finalize`. This corrupts account balances, contract storage, and all other state for shard B on the affected node.

### Likelihood Explanation
The condition is that shard A has no incoming cross-shard receipts in the epoch window covered by the header (i.e., `incoming_receipts_proofs` is empty). In low-traffic epochs or for shards with no cross-shard activity this is realistic. The attacker needs only to be a reachable peer — no validator or operator privileges are required.

### Recommendation
Add an explicit shard-id check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This single guard closes the mismatch before any other validation runs.

### Proof of Concept
1. Stand up a two-shard test network with two clients.
2. Produce enough blocks to trigger state sync.
3. Intercept the `download_shard_header(shard_id=1, …)` call on the syncing node and return the legitimate header for shard 0 (which has an empty `incoming_receipts_proofs`).
4. Observe that `set_state_header(shard_id=1, sync_hash, header_for_shard_0)` returns `Ok(())`.
5. Observe that all parts for shard 0 pass `validate_state_part` and are stored under `StatePartKey(sync_hash, shard_id=1, part_id)`.
6. After `set_state_finalize`, read shard 1's trie and confirm it contains shard 0's accounts/storage.
