Audit Report

## Title
`unwrapWETH9` Missing Zero-Address Check on `recipient` Allows Permanent ETH Burn - (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` accepts a caller-supplied `recipient` address with no zero-address guard. When `recipient == address(0)`, `_transferETH` executes `address(0).call{value: value}("")`, which the EVM resolves as a successful call (address(0) has no code, so the low-level call returns `(true, "")`), permanently destroying the unwrapped ETH. Any unprivileged caller can invoke `unwrapWETH9(0, address(0))` to burn the router's entire WETH balance.

## Finding Description
`unwrapWETH9` (L37–45) reads the router's full WETH balance, calls `IWETH9(WETH).withdraw(balanceWETH)` converting all WETH to native ETH held by the router, then calls `_transferETH(recipient, balanceWETH)` (L43). `_transferETH` (L90–93) performs a bare low-level call:

```solidity
(bool ok,) = to.call{value: value}("");
if (!ok) revert ETHTransferFailed();
```

When `to == address(0)`, the EVM sends ETH to the zero address. Because address(0) has no deployed code, the call returns `(true, "")`, `ok` is `true`, no revert is triggered, and the ETH is permanently burned. There is no `require(recipient != address(0))` guard anywhere in `unwrapWETH9` or `_transferETH`. The `amountMinimum` check (L39) is trivially bypassed by passing `0`. By contrast, `sweepToken` (L48–55) uses `safeTransfer`, which internally reverts on `to == address(0)`, creating an asymmetry where ERC20 sweeps are protected but ETH unwraps are not.

## Impact Explanation
Direct, permanent loss of user principal. A user who swaps `tokenX → WETH` with `recipient = address(router)` in a standalone transaction (a documented pattern) leaves WETH on the router until they call `unwrapWETH9`. An attacker can front-run that second call with `unwrapWETH9(0, address(0))`, burning the entire WETH balance. The victim's subsequent call sees `balanceWETH == 0` and receives nothing. This constitutes a Critical/High direct loss of user swap output with no recourse.

## Likelihood Explanation
The attack requires no special privilege — any EOA can call `unwrapWETH9(0, address(0))`. The precondition (WETH sitting on the router between two non-atomic transactions) is a documented and tested usage pattern. Any WETH sent directly to the router (e.g., direct transfer) is also permanently at risk. The attack is repeatable and costs only gas.

## Recommendation
Add a zero-address guard at the top of `unwrapWETH9` and `sweepToken`, and optionally in `_transferETH` as defense-in-depth:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    if (recipient == address(0)) revert InvalidRecipient();
    ...
}

function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    if (recipient == address(0)) revert InvalidRecipient();
    ...
}
```

## Proof of Concept
1. Alice calls `exactInputSingle{value: 1 ether}(tokenIn=WETH, tokenOut=token1, ..., recipient=address(router))` — router now holds `X` WETH.
2. Bob observes Alice's pending `unwrapWETH9(X, alice)` in the mempool and front-runs with `router.unwrapWETH9(0, address(0))`.
3. Bob's call: `balanceWETH = X` (passes `amountMinimum=0` check) → `IWETH9(WETH).withdraw(X)` → router holds `X` ETH → `_transferETH(address(0), X)` → `address(0).call{value: X}("")` returns `(true, "")` → `X` ETH permanently burned.
4. Alice's `unwrapWETH9` call now sees `balanceWETH == 0`, receives nothing. Alice loses her entire swap output.
