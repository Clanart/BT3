### Title
Missing `SetCodeTxType` Basefee Guard in `ValidateEthBasic` Allows EIP-7702 Transactions to Bypass Fee-Market Check — (`File: ante/interfaces/setup.go`)

---

### Summary

`ValidateEthBasic` explicitly rejects `DynamicFeeTxType` (type 2) when `baseFee == nil`, but contains no equivalent guard for `SetCodeTxType` (type 4, EIP-7702). Both transaction types are EIP-1559-style (they carry `GasFeeCap`/`GasTipCap`) and both require an active fee market for correct fee accounting. The missing case allows an EIP-7702 transaction to pass the ante handler and reach EVM execution on a chain where the fee market is disabled (`NoBaseFee = true` or London not yet active), while the analogous `DynamicFeeTxType` is correctly blocked.

---

### Finding Description

In `ante/interfaces/setup.go`, `ValidateEthBasic` iterates over every `MsgEthereumTx` in the transaction and performs per-type checks:

```go
// ante/interfaces/setup.go line 122-124
if baseFee == nil && tx.Type() == ethtypes.DynamicFeeTxType {
    return errorsmod.Wrap(ethtypes.ErrTxTypeNotSupported, "dynamic fee tx not supported")
}
```

This guard exists only for `DynamicFeeTxType`. There is no corresponding guard for `SetCodeTxType` (EIP-7702, type 4). [1](#0-0) 

`SetCodeTx` is structurally identical to `DynamicFeeTx` in its fee fields — it carries `GasFeeCap` and `GasTipCap` and its `EffectiveGasPrice` is computed via the same EIP-1559 formula: [2](#0-1) 

When `baseFee == nil` (fee market disabled via `NoBaseFee = true`, or London hardfork not yet active), a `SetCodeTxType` transaction passes `ValidateEthBasic` unchecked. It then proceeds through `CheckEthGasConsume` → `VerifyFee`, where the fee is computed using `GetEffectiveFee(nil)`, which falls back to `GasPrice()` (= `GasFeeCap`) rather than the EIP-1559 effective price. [3](#0-2) 

The transaction then reaches `ApplyMessageWithConfig`, where the EIP-7702 authorization processing block runs unconditionally — there is no `rules.IsPrague` guard around it: [4](#0-3) 

`applyAuthorization` validates and applies the delegation, bumping the authority's nonce and writing delegation code to state, without any check that Prague rules are active: [5](#0-4) 

The durable authorization context is then committed to the Cosmos KV store even if the EVM call itself fails: [6](#0-5) 

---

### Impact Explanation

An attacker can submit a valid EIP-7702 `SetCodeTxType` transaction on a chain where `baseFee == nil` (fee market disabled). The ante handler accepts it while it would reject an equivalent `DynamicFeeTxType`. The EIP-7702 authorization processing then runs without Prague rules being enforced, writing delegation code to arbitrary EOA accounts (`SetCode` + nonce bump) and committing those writes durably to the Cosmos KV store. This constitutes unauthorized account/code mutation — an EOA's code hash and nonce are permanently altered without the fee-market preconditions that the protocol requires for this transaction type.

This matches the **High** allowed impact: *"EIP-7702 authorization … verification bypass enabling … unauthorized account/code mutation."*

---

### Likelihood Explanation

The condition `baseFee == nil` is reachable in two realistic configurations:
1. **`NoBaseFee = true`** in fee market params (a supported operator configuration for chains that want fixed gas prices).
2. **London hardfork not yet activated** on a chain that has Prague scheduled but London still pending (misconfigured or staged rollout).

Any unprivileged user can craft and submit a `SetCodeTxType` transaction via `eth_sendRawTransaction`. No special privileges are required.

---

### Recommendation

Add the missing guard for `SetCodeTxType` immediately after the existing `DynamicFeeTxType` check in `ValidateEthBasic`:

```go
// ante/interfaces/setup.go
if baseFee == nil && (tx.Type() == ethtypes.DynamicFeeTxType ||
    tx.Type() == ethtypes.SetCodeTxType) {
    return errorsmod.Wrap(ethtypes.ErrTxTypeNotSupported,
        "EIP-1559/EIP-7702 tx not supported: fee market not active")
}
```

Additionally, add a `rules.IsPrague` guard in `ApplyMessageWithConfig` around the `msg.SetCodeAuthorizations != nil` block to prevent authorization processing when Prague is not active, mirroring how go-ethereum gates this logic.

---

### Proof of Concept

1. Deploy an Ethermint chain with `NoBaseFee = true` (fee market disabled, `baseFee == nil`).
2. Craft a valid EIP-7702 `SetCodeTxType` transaction with a well-formed authorization list (authority signs over a delegate address).
3. Submit via `eth_sendRawTransaction`.
4. `ValidateEthBasic` passes — the `DynamicFeeTxType` guard at line 122 does not fire for type 4. [1](#0-0) 
5. `CheckEthGasConsume` → `VerifyFee` computes fee using `GasFeeCap` as price (baseFee nil path). [7](#0-6) 
6. `ApplyMessageWithConfig` processes the authorization list, calling `applyAuthorization` for each entry, writing delegation code and bumping the authority nonce in the EVM stateDB. [8](#0-7) 
7. The durable authorization context is committed to the KV store regardless of EVM call outcome. [6](#0-5) 
8. The authority EOA now has delegation code set and its nonce bumped — unauthorized account mutation on a chain that never enabled the Prague hardfork or fee market.

func (msg MsgEthereumTx) GetEffectiveGasPrice(baseFee *big.Int) *big.Int {
	tx := msg.AsTransaction()
	if baseFee == nil {
		return tx.GasPrice()
	}
	// for legacy tx, both gasTipCap and gasFeeCap are gasPrice, the result is equavalent.
	return ethermint.BigMin(new(big.Int).Add(tx.GasTipCap(), baseFee), tx.GasFeeCap())
}

func (k *Keeper) applyAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (common.Address, error) {
	authority, err := k.validateAuthorization(auth, stateDB)
	if err != nil {
		return authority, err
	}

	// If the account already exists in state, refund the new account cost
	// charged in the intrinsic calculation.
	if stateDB.Exist(authority) {
		stateDB.AddRefund(params.CallNewAccountGas - params.TxAuthTupleGas)
	}

	k.setAuthorizationDelegation(auth, authority, stateDB)
	return authority, nil
}
