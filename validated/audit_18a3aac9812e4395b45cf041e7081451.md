### Title
`nativeFee` ETH Is Permanently Locked in `OmniBridge` With No Withdrawal or Forwarding Mechanism — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

When a user calls `initTransfer` with a non-zero `nativeFee`, the corresponding ETH is transferred into the `OmniBridge` contract but is never forwarded to any relayer or fee recipient, and no withdrawal function exists. The `nativeFee` ETH is irrecoverably locked in the contract.

---

### Finding Description

In `OmniBridge.sol::initTransfer`, the user sends `msg.value` that must cover both the bridged `amount` (for native ETH transfers) and the `nativeFee`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol, lines 387–413
if (tokenAddress == address(0)) {
    if (fee != 0) { revert InvalidFee(); }
    extensionValue = msg.value - amount - nativeFee;   // nativeFee silently absorbed
} else {
    extensionValue = msg.value - nativeFee;             // nativeFee silently absorbed
    ...
}
``` [1](#0-0) 

The `nativeFee` portion is subtracted from `msg.value` to compute `extensionValue`, but the ETH itself is never forwarded anywhere. It is passed as a parameter to `initTransferExtension`, but the base implementation ignores it entirely and only asserts `extensionValue == 0`:

```solidity
// lines 492–506
function initTransferExtension(
    address, address, uint64, uint128, uint128,
    uint128 /*nativeFee*/,   // ← received but never used
    string calldata, string calldata,
    uint256 value
) internal virtual {
    if (value != 0) { revert InvalidValue(); }
}
``` [2](#0-1) 

The contract has no `withdraw`, `claimFee`, or any other function that could release accumulated `nativeFee` ETH. The only ETH egress path is `finTransfer`, which sends only `payload.amount` to the recipient and is unrelated to outbound `nativeFee` balances:

```solidity
// lines 317–322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [3](#0-2) 

The contract does expose `receive() external payable {}` (line 574), confirming it can hold ETH, but provides no mechanism to release it. [4](#0-3) 

The same issue applies to `initTransfer1155`:

```solidity
// line 466
uint256 extensionValue = msg.value - nativeFee;
``` [5](#0-4) 

---

### Impact Explanation

Every call to `initTransfer` or `initTransfer1155` with a non-zero `nativeFee` results in that ETH being permanently locked in the contract. The user loses the fee they intended to pay to the relayer, and the relayer receives no on-chain compensation for NEAR-side gas costs. Over time, the cumulative locked ETH grows with each bridging operation that includes a `nativeFee`. This matches the **Critical** allowed impact: *permanent freezing / irrecoverable lock of user funds in bridge flows*.

---

### Likelihood Explanation

The `nativeFee` parameter is a first-class field in the `InitTransfer` event (line 427–436) and in the NEAR-side `TransferMessage` fee structure, indicating it is a designed and expected protocol input — not an edge case. [6](#0-5) 

Any user or relayer infrastructure that sets `nativeFee > 0` to incentivize relayers will trigger the lock. The mainnet EVM bridge (`0xe00c629aFaCCb0510995A2B95560E446A24c85B9` on Ethereum) is the base `OmniBridge.sol` deployment, so real users are exposed.

---

### Recommendation

Override `initTransferExtension` (or modify `initTransfer` directly) to forward the `nativeFee` ETH to the designated fee recipient or relayer at the time of the outbound transfer. Alternatively, accumulate it in a claimable mapping keyed by relayer address and add a `claimNativeFee()` withdrawal function. At minimum, if `nativeFee` is not yet supported in the base contract, add a `require(nativeFee == 0)` guard to prevent silent loss of user funds.

---

### Proof of Concept

1. User calls `initTransfer(USDC, 1000e6, 0, 0.01 ether, "near:alice.near", "")` sending `msg.value = 0.01 ether`.
2. Inside `initTransfer`: `extensionValue = 0.01 ether - 0.01 ether = 0`. USDC is burned. `initTransferExtension` is called with `nativeFee = 0.01 ether` and `value = 0`.
3. Base `initTransferExtension` passes (value == 0). The `0.01 ether` sits in the contract.
4. The `InitTransfer` event is emitted. A relayer picks it up and submits the proof to NEAR, paying NEAR gas out of pocket.
5. No on-chain function exists to send the `0.01 ether` to the relayer. The ETH is permanently locked.
6. Repeat for every bridging user who sets `nativeFee > 0`; the contract balance grows monotonically with no egress.
