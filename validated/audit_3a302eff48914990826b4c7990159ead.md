### Title
`ProofOfInclusion::valid()` Final Root-Hash Check Is a Tautology — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate contains a self-referential final check that is always true when the loop completes, meaning the function never actually verifies the proof against an externally-known root hash. An attacker can craft a `ProofOfInclusion` that passes `valid()` for any claimed leaf inclusion in any claimed tree root.

### Finding Description

`root_hash()` derives the root entirely from the proof's own data:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← proof's own field
    } else {
        self.node_hash
    }
}
```

`valid()` then ends with:

```rust
existing_hash == self.root_hash()
```

After the loop, `existing_hash` holds `calculated_hash` for the last layer. The loop body already asserted `calculated_hash == layer.combined_hash` (returning `false` otherwise), so at loop exit:

```
existing_hash  ==  last.combined_hash  ==  self.root_hash()
```

The final comparison is a tautology — it compares a value against itself. [1](#0-0) 

The correct design requires `valid()` to accept an externally-supplied root hash and compare `existing_hash` against it, not against `self.root_hash()`. The `root_hash()` helper is useful for callers who want to read the claimed root, but using it as the validation target inside `valid()` defeats the purpose of the check entirely.

The `calculate_internal_hash` function used in the loop is straightforward: [2](#0-1) 

An attacker who controls a serialized `ProofOfInclusion` (a `Streamable` type exposed to Python) can:

1. Set `node_hash` to any leaf hash they wish to claim is included.
2. For each layer, freely choose `other_hash` and `other_hash_side`, then compute `combined_hash = calculate_internal_hash(existing_hash, side, other_hash)`.
3. The resulting proof passes `valid()` unconditionally, regardless of the actual DataLayer tree root.

The `ProofOfInclusion` struct is `Streamable` and fully exposed through Python bindings: [3](#0-2) 

The Python binding exposes `valid()` directly: [4](#0-3) 

The existing test `test_proof_of_inclusion_invalid_identified` only tests that corrupting an *intermediate* `combined_hash` is caught (which it is, by the loop body check), but never tests that a fully attacker-crafted proof is rejected: [5](#0-4) 

### Impact Explanation

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, lets untrusted input prove invalid state.**

Any DataLayer client that calls `proof.valid()` as its sole check before trusting a proof of inclusion (e.g., to confirm a key-value pair exists in a committed tree) can be deceived. An attacker can prove inclusion of any key in any tree root without possessing the actual tree data.

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type received over the network from DataLayer peers. The Python node calls `proof.valid()` to verify received proofs. The attacker needs only to craft a well-formed byte sequence — no privileged access, no key material, no chain reorg required. The attack is deterministic and requires only arithmetic on SHA-256 hashes.

### Recommendation

`valid()` must accept an externally-known root hash and compare against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root   // compare against caller-supplied root
}
```

All call sites must supply the root hash from a trusted source (e.g., the committed on-chain root, not the proof itself). The existing `root_hash()` accessor can remain for informational use, but must not be used as the validation target inside `valid()`.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side, proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer}};
use chia_datalayer::blob::calculate_internal_hash;

fn forge_proof(claimed_node_hash: Hash, arbitrary_other: Hash) -> ProofOfInclusion {
    // attacker freely picks other_hash and side
    let combined = calculate_internal_hash(&claimed_node_hash, Side::Right, &arbitrary_other);
    ProofOfInclusion {
        node_hash: claimed_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Right,
            other_hash: arbitrary_other,
            combined_hash: combined,   // attacker computes this to satisfy the loop check
        }],
    }
}

fn main() {
    let fake_leaf  = Hash([0xAA; 32]);
    let junk_other = Hash([0xBB; 32]);
    let proof = forge_proof(fake_leaf, junk_other);

    // valid() returns true even though this proof was never generated
    // from any real MerkleBlob and the "root" is attacker-controlled
    assert!(proof.valid());

    // The "root" the attacker can claim:
    println!("Forged root: {:?}", proof.root_hash());
}
```

The loop check passes because `combined_hash` was computed to match. The final check `existing_hash == self.root_hash()` passes because both sides equal `combined_hash`. No real tree is involved.
