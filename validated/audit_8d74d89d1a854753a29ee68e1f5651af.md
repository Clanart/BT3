### Title
`compute_meta_tx_v0_hash` Omits Nonce, Enabling Unbounded Signature Replay for `meta_tx_v0` Syscall — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall path in the StarkNet OS computes a transaction hash that contains **no nonce and no block-specific data**. Because `check_and_increment_nonce` explicitly skips version-0 transactions, and the OS never increments any counter for meta_tx_v0 executions, a valid signature obtained for any `(contract_address, selector, calldata, chain_id)` tuple can be replayed an unlimited number of times across any block. This is a direct analog of the cross-function signature replay described in the external report.

---

### Finding Description

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
// transaction_hash.cairo lines 294-315
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(...) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no block data
    );
    return tx_hash;
}
```

Compare to `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1, additional_data=&nonce`.

In `execute_meta_tx_v0`, the resulting `TxInfo` is constructed with `nonce=0` and `version=0`:

```cairo
// syscall_impls.cairo lines 343-363
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,   // ← always zero
    ...
);
```

`check_and_increment_nonce` is never called for this path, and even if it were, it explicitly skips version-0 transactions:

```cairo
// execute_transaction_utils.cairo lines 63-67
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
```

The result: the hash `H(INVOKE_PREFIX, 0, contract_address, selector, calldata_hash, 0, chain_id)` is **deterministic and permanent** for any fixed set of inputs. No OS-level mechanism invalidates a used meta_tx_v0 signature.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any account contract that relies on the OS-enforced `transaction_hash` for signature verification (the standard pattern for `__execute__`) will accept the same signature on every replay. An attacker who observes or obtains a single valid meta_tx_v0 signature authorizing a token transfer can replay it in every subsequent block, draining the victim account completely. The OS provides no counter, no used-hash registry, and no block-binding for meta_tx_v0 — replay protection is structurally absent at the protocol level.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is callable by any deployed contract (it is only blocked in virtual OS mode). An attacker needs only to:
1. Observe a legitimate on-chain meta_tx_v0 call (the signature is in the transaction data), or
2. Trick a victim into signing a meta_tx_v0 payload once.

After that, the attacker deploys a contract that calls the `meta_tx_v0` syscall with the captured signature and replays it in every block. No privileged access is required. The entry path is fully reachable by an unprivileged transaction sender.

---

### Recommendation

Include a monotonically-increasing, per-account nonce in `compute_meta_tx_v0_hash` (analogous to how `compute_l1_handler_transaction_hash` passes `additional_data=&nonce`), and increment that nonce in the OS after each successful meta_tx_v0 execution. Alternatively, maintain a used-hash registry in contract state and reject duplicate meta_tx_v0 hashes. Either fix must be enforced at the OS level, not left to individual account contracts.

---

### Proof of Concept

1. Victim account `A` is called via a legitimate `meta_tx_v0` syscall that transfers 100 tokens to address `B`. The outer transaction is visible on-chain; the signature `sig` and calldata `cd` are public.

2. Attacker deploys contract `Evil` containing:
   ```
   fn execute():
       loop:
           syscall meta_tx_v0(
               contract_address = A,
               selector         = __execute__,
               calldata         = cd,   // same transfer calldata
               signature        = sig,  // same signature
           )
   ```

3. Attacker submits a transaction calling `Evil.execute()`.

4. The OS computes `meta_tx_hash = H(INVOKE_PREFIX, 0, A, __execute__, hash(cd), 0, chain_id)` — identical to the original. [1](#0-0) 

5. `execute_meta_tx_v0` sets `nonce=0`, never calls `check_and_increment_nonce`, and dispatches to `A.__execute__` with the replayed signature. [2](#0-1) 

6. `A.__execute__` verifies `sig` against `meta_tx_hash` — verification passes because the hash is identical to the original.

7. 100 tokens are transferred again. Steps 3–7 repeat in every block until `A` is drained.

The nonce-skip in `check_and_increment_nonce` confirms no OS-level counter is ever updated: [3](#0-2) 

The absence of `additional_data` in the meta_tx_v0 hash (vs. the nonce-bound L1-handler hash) is the structural root cause: [4](#0-3)
