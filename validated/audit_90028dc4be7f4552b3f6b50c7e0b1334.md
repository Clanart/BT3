### Title
Integer Division Truncation in `NewDynamicFeeChecker` Truncates `feeCap` to Zero, Enabling Complete Fee Bypass for Cosmos Transactions - (File: `ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` computes the per-gas fee cap via integer division `fee / gas`. When `fee < gas` (which is the normal case whenever `minGasPrice < 1`), the result truncates to zero. With `baseFee = 0` — a reachable state when `minGasPrice.TruncateInt() = 0` — the zero `feeCap` passes the base-fee guard, `EffectiveGasPrice` clamps to zero, and the returned `effectiveFee` is zero. The Cosmos SDK's `DeductFeeDecorator` then deducts nothing from the sender, even though the sender provided non-zero fee coins that satisfied `MinGasPriceDecorator`.

---

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` derives the per-gas fee cap from the Cosmos tx fee fields using integer (`sdkmath.Int`) division:

```go
// ante/evm/fee_checker.go:83
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

`fee` is the total fee amount in the EVM denom (an integer, e.g. in `aevmos`). `gas` is the `uint64` gas limit. `Quo` is truncating integer division. Whenever `fee < gas`, `feeCap` becomes zero.

The only guard against a zero `feeCap` is:

```go
// ante/evm/fee_checker.go:86-88
if feeCap.LT(baseFeeInt) {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, ...)
}
```

This guard is bypassed when `baseFeeInt = 0`. The effective price is then computed as:

```go
// ante/evm/fee_checker.go:91
effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
```

`EffectiveGasPrice` is `min(tipCap + baseFee, feeCap)` = `min(maxPriorityPrice + 0, 0)` = `0`.

```go
// ante/evm/fee_checker.go:94-99
effectiveFee := sdk.Coins{{Denom: denom, Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))}}
```

`effectiveFee = 0 * gas = 0`. The checker returns zero coins to `DeductFeeDecorator`, which deducts nothing.

**When is `baseFee = 0` reachable?**

`x/evm/keeper/keeper.go` returns `big.NewInt(0)` (not nil) when London HF is enabled but the feemarket keeper returns nil:

```go
// x/evm/keeper/keeper.go:314-318
baseFee := k.feeMarketKeeper.GetBaseFee(ctx)
if baseFee == nil {
    // return 0 if feemarket not enabled.
    baseFee = big.NewInt(0)
}
```

More critically, `CalculateBaseFee` bounds the base fee from below by `minGasPrice.TruncateInt()`:

```go
// x/feemarket/keeper/eip1559.go:103-104
minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
```

If `minGasPrice` is a fractional `LegacyDec` value such as `0.5` (the default `DefaultMinGasMultiplier`), then `TruncateInt() = 0`, and the base fee can be driven to zero through sustained under-utilization of blocks. Once `baseFee = 0`, the truncation vulnerability is fully exploitable.

**The split between validation and deduction:**

`MinGasPriceDecorator` validates that `feeCoins >= ceil(minGasPrice * gasLimit)` against the raw `feeCoins` field:

```go
// ante/cosmos/min_gas_price.go:77
fee := gp.Amount.Mul(gasLimit).Ceil().RoundInt()
```

This check passes because the user provides sufficient coins. But `NewDynamicFeeChecker` independently computes `effectiveFee = 0` and returns it to `DeductFeeDecorator`. The two decorators operate on different quantities: one validates the raw coins, the other deducts the computed effective fee. The user's coins are never actually deducted.

---

### Impact Explanation

A Cosmos transaction sender can submit a transaction that passes all ante-handler fee checks yet has zero fees deducted from their account. This is a direct mis-accounting of user funds/fees in the ante handler. The sender retains all fee coins they nominally "paid," effectively executing transactions for free. This matches: **"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."**

---

### Likelihood Explanation

The preconditions are:
1. London HF enabled (standard for any EIP-1559 chain).
2. `minGasPrice` set to a fractional value less than 1 (e.g., `0.5`, which is the `DefaultMinGasMultiplier` value and a common deployment choice).
3. Base fee driven to zero — achievable by submitting empty or under-utilized blocks over time, since `minGasPrice.TruncateInt() = 0` removes the lower bound.
4. `fee < gas` in the Cosmos tx — trivially satisfied whenever `minGasPrice < 1`, since `requiredFee = ceil(minGasPrice * gasLimit) < gasLimit`.

An unprivileged user can trigger this by simply constructing a Cosmos tx with `fee = ceil(minGasPrice * gasLimit)` and `gas = gasLimit`. No special access is required beyond the ability to submit transactions.

---

### Recommendation

Replace the integer division with a ceiling division or use `LegacyDec` arithmetic to preserve sub-unit precision:

```go
// Instead of:
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))

// Use ceiling division so feeCap is never artificially zero:
gasInt := sdkmath.NewIntFromUint64(gas)
feeCap := fee.Add(gasInt.Sub(sdkmath.OneInt())).Quo(gasInt) // ceil(fee/gas)
```

Alternatively, perform the comparison in `LegacyDec` space to avoid truncation:

```go
feeCapDec := sdkmath.LegacyNewDecFromInt(fee).Quo(sdkmath.LegacyNewDecFromInt(sdkmath.NewIntFromUint64(gas)))
```

Additionally, add a lower bound to `minGasPrice` validation to prevent fractional values that truncate to zero when used as the base-fee floor in `CalculateBaseFee`.

---

### Proof of Concept

**Setup**: London HF enabled, `minGasPrice = 0.5 aevmos/gas`, `baseFee` driven to 0 (empty blocks, since `TruncateInt(0.5) = 0`).

**Attack**:
- Attacker constructs a Cosmos tx with `gasLimit = 100_000` and `fee = 50_000 aevmos` (satisfies `MinGasPriceDecorator`: `ceil(0.5 * 100_000) = 50_000`).
- In `NewDynamicFeeChecker`: `feeCap = 50_000 / 100_000 = 0` (integer truncation).
- `baseFeeInt = 0`, so `feeCap (0) >= baseFeeInt (0)` — guard passes.
- `effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = min(MaxInt64, 0) = 0`.
- `effectiveFee = 0 * 100_000 = 0`.
- `DeductFeeDecorator` deducts 0 aevmos from the attacker.
- Attacker's 50_000 aevmos are never deducted; the transaction executes for free. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
