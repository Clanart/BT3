### Title
Cross-Chain Replay of `deployToken` Signature Enables Unauthorized Token Deployment - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deployToken` / `deploy_token` signature hash across EVM, StarkNet, and Solana does **not** include any chain identifier, while the analogous `finTransfer` / `fin_transfer` / `finalize_transfer` hash **does** include the chain ID. A valid `deployToken` signature produced by the NEAR MPC signer for one chain can be replayed verbatim on any other chain that uses the same `nearBridgeDerivedAddress`, deploying an unauthorized bridge token and permanently blocking the legitimate deployment for that token on the target chain.

---

### Finding Description

**EVM — `OmniBridge.sol` `deployToken`:**

The Borsh-encoded hash used for signature verification is:

```
PayloadType.Metadata || token || name || symbol || decimals
```

No chain ID is included. [1](#0-0) 

By contrast, `finTransfer` encodes `omniBridgeChainId` **twice** (once for the token address slot, once for the recipient slot): [2](#0-1) 

**StarkNet — `bridge_types.cairo` `MetadataPayload::to_borsh`:**

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

No chain ID. [3](#0-2) 

`deploy_token` in `omni_bridge.cairo` calls `payload.to_borsh()` (no chain ID argument): [4](#0-3) 

`fin_transfer` calls `payload.to_borsh(self.omni_bridge_chain_id.read())` (chain ID included): [5](#0-4) 

`TransferMessagePayload::to_borsh(chain_id)` encodes `chain_id` twice: [6](#0-5) 

**Solana — `deploy_token.rs` `DeployTokenPayload::serialize_for_near`:**

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    self.serialize(&mut writer)?; // borsh encoding — no chain ID
    ...
}
``` [7](#0-6) 

`FinalizeTransferPayload::serialize_for_near` includes `SOLANA_OMNI_BRIDGE_CHAIN_ID` twice: [8](#0-7) 

**The asymmetry is intentional for `finTransfer` (the CLAUDE.md explicitly notes "Chain ID binding: Destination chain_id encoded in message hash — prevents cross-chain replay") but was not applied to `deployToken`.** [9](#0-8) 

All three destination chains verify the `deployToken` signature against the same type of MPC-derived address (`nearBridgeDerivedAddress` on EVM, `omni_bridge_derived_address` on StarkNet, `derived_near_bridge_address` on Solana). The NEAR bridge uses a single fixed `SIGN_PATH` constant for all MPC signing requests: [10](#0-9) 

If the same derivation path is used for all EVM chains (the default when no chain-specific path is configured), the `nearBridgeDerivedAddress` is identical across chains, making the `deployToken` signature directly replayable.

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on Chain A (e.g., Ethereum) can submit the same `signatureData` and `metadata` payload to the `deployToken` function on Chain B (e.g., Polygon or Base) before the protocol does so legitimately.

Consequences:
1. **Unauthorized bridge token deployed on Chain B** — the token is registered in `nearToEthToken` / `near_to_starknet_token` mappings under attacker-controlled timing.
2. **Permanent blocking of legitimate deployment** — `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED` prevents any subsequent legitimate deployment of the same NEAR token on Chain B.
3. **Token mapping corruption** — the bridge's canonical token address for that NEAR token on Chain B is now set to the attacker-triggered deployment, which may differ from what the protocol intended (e.g., different `tokenImplementationAddress` if the attacker front-runs an upgrade window).

This matches the **High** impact category: cross-chain replay enabling unauthorized token deployment and permanent blocking of legitimate bridge operations for that token on the target chain.

---

### Likelihood Explanation

- The attack requires only a valid `deployToken` transaction on any one chain — fully observable on-chain.
- No privileged access, leaked keys, or colluding MPC signers are required.
- The attacker only needs to call `deployToken` on the target chain with the same arguments before the protocol does.
- The NEAR bridge uses a single `SIGN_PATH` constant for all MPC signing, making the same derived address likely across EVM chains.
- The attack is a simple front-run or replay with zero cost beyond gas.

---

### Recommendation

Include the destination chain ID in the `deployToken` / `deploy_token` Borsh-encoded hash, mirroring the existing pattern used in `finTransfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain ID
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**StarkNet (`bridge_types.cairo`):**
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append_byte(chain_id);   // add chain ID
    ...
}
```

**Solana (`deploy_token.rs`):**
```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;  // add chain ID
    self.serialize(&mut writer)?;
    ...
}
```

The NEAR bridge's MPC signing for `deployToken` must also include the destination chain ID in the payload it signs, so that signatures are chain-scoped.

### Proof of Concept

1. NEAR bridge operator calls `log_metadata` for token `"token.near"` on NEAR.
2. Relayer calls `deployToken` on Ethereum (`omniBridgeChainId = 2`) with valid `signatureData` S and metadata M. Transaction succeeds; token deployed at address `0xAAA`.
3. Attacker observes the Ethereum transaction and extracts `signatureData` S and metadata M.
4. Attacker calls `deployToken` on Polygon (`omniBridgeChainId = 4`) with the **same** `signatureData` S and metadata M.
5. `ECDSA.recover(keccak256(borshEncoded), S)` returns `nearBridgeDerivedAddress` (same on both chains if same derivation path) → signature check passes.
6. Token is deployed on Polygon at address `0xBBB` under attacker-controlled timing.
7. When the legitimate relayer later tries to deploy on Polygon, `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` reverts — the legitimate deployment is permanently blocked. [11](#0-10)
