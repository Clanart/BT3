### Title
Missing Mint-Metadata Binding Check in Token-2022 Third-Party Metadata Path Enables Fake Token Registration with Stolen Identity — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` reads `name`/`symbol` from a Metaplex metadata account but never verifies that the account's embedded `mint` field matches `self.mint.key()`. An unprivileged attacker can create a Token-2022 mint whose `MetadataPointer` extension points to USDC's (or any high-value token's) Metaplex PDA, call `log_metadata`, and cause the bridge to emit a Wormhole VAA pairing the attacker's mint pubkey with USDC's name, symbol, and decimals. NEAR's `deploy_token_callback` accepts this VAA and registers the attacker's mint as a deployed bridge token with USDC's identity, after which the attacker can mint unlimited unbacked "USDC" tokens on NEAR.

---

### Finding Description

**Root cause — `parse_metadata_account` (lines 72–90):** [1](#0-0) 

The function verifies only that the passed-in account key equals `metadata_pointer.metadata_address.0` and that the account is owned by `MetaplexID`. It then deserializes the account and returns `(metadata.name, metadata.symbol)` without ever asserting `metadata.mint == self.mint.key()`.

**Trigger path — Token-2022 "third-party metadata" branch (lines 104–106):** [2](#0-1) 

When `metadata_pointer.metadata_address.0 != self.mint.key()` and `!= Pubkey::default()`, the code calls `parse_metadata_account` with the pointer's address. An attacker sets this pointer to USDC's Metaplex PDA (a deterministic, publicly readable account owned by `MetaplexID`). The check `metadata.key() == address` passes trivially because the attacker supplied that exact address.

**Payload construction (lines 130–136):** [3](#0-2) 

`token` is set to `self.mint.key()` (the attacker's mint), while `name`/`symbol`/`decimals` come from USDC's Metaplex account. The resulting Wormhole VAA carries `{token: attacker_mint, name: "USD Coin", symbol: "USDC", decimals: 6}`.

**No constraint on the `metadata` account in the struct:** [4](#0-3) 

The field is `Option<UncheckedAccount<'info>>` with no PDA derivation or mint-binding constraint.

**NEAR `deploy_token_callback` accepts the VAA without further validation:** [5](#0-4) 

The only guard is that the emitter matches a known factory. Since the VAA was emitted by the legitimate Solana bridge program, this check passes. `deploy_token_internal` is called with the attacker's mint pubkey as `token_address` and USDC's metadata.

**`deploy_token_internal` registers the token permanently:** [6](#0-5) 

The `token_id` is derived from the attacker's mint pubkey prefix, so it is unique and passes the `TokenExists` guard. A new NEP-141 token contract is deployed with name="USD Coin", symbol="USDC", decimals=6, permanently bound to the attacker's mint.

---

### Impact Explanation

The attacker controls the mint authority of their fake Token-2022 mint (the `LogMetadata` constraint `!mint.mint_authority.contains(authority.key)` explicitly requires this). They can therefore:

1. Mint unlimited supply of their fake Solana token.
2. Deposit it into the bridge vault (created for their mint by `log_metadata`).
3. Initiate a Solana→NEAR transfer; NEAR's `finalize_transfer` mints the registered NEP-141 token (named "USD Coin"/"USDC") to any recipient the attacker specifies.

The result is unbounded unauthorized minting of a NEAR token that is indistinguishable by name/symbol/decimals from real USDC, backed by zero real value. Any user or protocol that accepts the token by name/symbol (e.g., a DEX, lending protocol, or wallet) is exposed to loss.

---

### Likelihood Explanation

- Requires no privileged access; any Solana keypair can create a Token-2022 mint and set an arbitrary `MetadataPointer`.
- USDC's Metaplex PDA is a well-known, permanently on-chain, publicly readable account.
- The Wormhole VAA is produced by the legitimate bridge program and will be signed by Wormhole guardians.
- The attack is a single transaction on Solana followed by a standard `deploy_token` call on NEAR.
- No race condition, no oracle dependency, no special timing required.

---

### Recommendation

In `parse_metadata_account`, after deserializing the `MplMetadata` struct, add a binding check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // MISSING CHECK — add this:
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This ensures the Metaplex metadata account actually belongs to the mint being registered, closing the cross-mint metadata hijack.
