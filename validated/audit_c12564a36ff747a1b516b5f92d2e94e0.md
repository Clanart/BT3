### Title
Duplicate Key Bypass in `MerkleBlob::batch_insert` Fast Path Corrupts DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` contains a fast path for inserting three or more leaves that bypasses the duplicate-key guard enforced by the single-item `insert()` function. Supplying a batch with repeated `KeyId` values causes two leaf nodes with the same key to be written into the blob, silently corrupting the Merkle tree root hash and invalidating all subsequent proofs of inclusion/exclusion derived from it.

### Finding Description

`MerkleBlob::insert()` guards against duplicate keys before writing anything to the blob:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` pops the **last two** elements from the input vector and routes them through `insert()` (with the duplicate check). All remaining elements — the first `N-2` entries of the original vector — are written directly to the blob via `insert_entry_to_blob` with **no duplicate-key check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { key, value, hash }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When a duplicate `KeyId` appears in the fast-path portion of the batch:

1. **Two leaf nodes with the same key** are written into the blob at different `TreeIndex` positions.
2. The `block_status_cache` (`key_to_index` HashMap) silently overwrites the first entry with the second, so the cache maps the key to only the last-inserted leaf's index.
3. The internal-node tree is built over **all** inserted leaf indexes (including both duplicates), so the computed root hash incorporates both leaves.
4. The resulting root hash differs from what a correctly-built tree containing only unique keys would produce.

The Python binding `py_batch_insert` performs no deduplication before forwarding to `batch_insert`:

```rust
self.batch_insert(zip(keys_values, hashes).collect())?;
``` [3](#0-2) 

The `BlockStatusCache::add_leaf` function, which is called by `insert_entry_to_blob` for leaf nodes, uses a plain `HashMap::insert` that silently overwrites on collision:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [4](#0-3) 

`check_integrity` would eventually detect the inconsistency (`leaf_count != key_to_index_cache_length`), but it is not called automatically after `batch_insert` in production paths — only in debug/test mode via `check_integrity_on_drop`. [5](#0-4) 

### Impact Explanation

The DataLayer Merkle tree root hash is the authoritative commitment used to generate and verify proofs of inclusion and exclusion. A corrupted root — one computed over a tree containing duplicate leaf nodes — means:

- **Proofs of inclusion** generated by `get_proof_of_inclusion` are anchored to a root that does not correspond to the intended key-value set.
- **Proof verification** (`ProofOfInclusion::valid()`) checks internal hash consistency but does not re-derive the root from scratch; it trusts the `combined_hash` stored in each layer. A proof built from a corrupted tree can appear internally consistent while attesting to a wrong root.
- Any downstream consumer comparing the stored root against an expected root will observe a mismatch, or — if the corrupted root is accepted as authoritative — will accept proofs for a state that was never legitimately committed.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.** [6](#0-5) 

### Likelihood Explanation

The Python binding `py_batch_insert` is the primary public entry point. Any caller — including DataLayer clients submitting batch updates — can supply a list with repeated `KeyId` values. No deduplication is performed at the binding layer. The vulnerability is triggered whenever a batch of three or more items contains a duplicate key among the first `N-2` entries (those routed through the fast path). This is a straightforward programming mistake that requires no special privileges to exploit.

### Recommendation

Add a duplicate-key check at the start of `batch_insert` (or within the fast-path loop) before writing any leaf to the blob:

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Check for duplicates within the batch and against existing keys
    let mut seen = HashSet::new();
    for ((key, _), _) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) || !seen.insert(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
    }
    // ... rest of the function
}
```

Alternatively, route all items through the validated `insert()` path, accepting the performance trade-off, or add the `contains_key` / `contains_leaf_hash` guards inline in the fast-path loop.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
from hashlib import sha256

def make_hash(n: int):
    return sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Build a batch of 4 items where item[0] and item[3] share the same KeyId.
# Items are popped from the END, so item[3] and item[2] go through insert()
# (with duplicate check), while item[0] and item[1] go through the fast path
# (no duplicate check).
dup_key = KeyId(0)
batch_keys_values = [
    (dup_key,   ValueId(99)),   # fast path — duplicate, no check
    (KeyId(1),  ValueId(1)),    # fast path
    (KeyId(2),  ValueId(2)),    # insert() — ok
    (dup_key,   ValueId(0)),    # insert() — ok (first occurrence)
]
batch_hashes = [make_hash(i) for i in range(4)]

# This succeeds silently — no KeyAlreadyPresentError raised
blob.batch_insert(batch_keys_values, batch_hashes)
blob.calculate_lazy_hashes()

# The blob now contains 4 leaf nodes, two with dup_key.
# The root hash is computed over all 4 leaves — it is corrupted.
# check_integrity() will raise IntegrityKeyToIndexCacheLengthError
# because leaf_count (4) != cache_length (3).
blob.check_integrity()  # raises — tree is silently corrupted
