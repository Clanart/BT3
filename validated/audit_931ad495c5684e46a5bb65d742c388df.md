### Title
Missing `logLimit` enforcement in `Filter.Logs` BlockHash path — (`rpc/namespaces/ethereum/eth/filters/filters.go`)

### Summary
`Filter.Logs` enforces `RPCLogsCap` only in the block-range loop. When `BlockHash != nil`, the function calls `blockLogs` and returns all matching logs with no size cap, allowing any unprivileged caller to trigger unbounded memory allocation via `eth_getLogs`.

### Finding Description

`Filter.Logs` takes a `logLimit int` parameter and is called from `GetLogs` with `int(api.backend.RPCLogsCap())`: [1](#0-0) 

Inside `Filter.Logs`, the BlockHash branch (lines 108–137) fetches block logs and returns them directly: [2](#0-1) 

No check against `logLimit` is performed anywhere in this branch. By contrast, the range path explicitly enforces the cap: [3](#0-2) 

`blockLogs` itself has no size guard — it collects all logs from all transactions in the block and returns them unconditionally: [4](#0-3) 

### Impact Explanation

An attacker deploys a contract that emits many events per call, fills a block with such transactions (up to the block gas limit), then calls `eth_getLogs` with `{blockHash: <that block's hash>}`. The node allocates memory proportional to the total number of logs in the block — with no cap — before returning. Since the JSON-RPC server shares the OS process with the CometBFT/consensus engine in Ethermint, an OOM kill terminates the entire node, causing it to miss blocks and drop out of consensus. Coordinated against multiple validators this halts the chain.

`RPCLogsCap` defaults to 10,000: [5](#0-4) 

but is never applied on the BlockHash path, making the configured cap meaningless for that query type.

### Likelihood Explanation

The attack requires only: (1) a funded account to deploy a log-emitting contract and submit transactions, and (2) knowledge of the block hash — both are trivially available to any unprivileged user. No special privileges, governance access, or validator cooperation is needed.

### Recommendation

After the `blockLogs` call in the BlockHash branch, add the same guard used in the range path:

```go
logs, err := f.blockLogs(blockRes, bloom)
if err != nil {
    return nil, err
}
if len(logs) > logLimit {
    return nil, fmt.Errorf("query returned more than %d results", logLimit)
}
```

This mirrors the existing check at lines 204–206 and closes the asymmetry between the two paths.

### Proof of Concept

1. Deploy a contract with a function that emits `RPCLogsCap + 1` events in a single transaction (or spread across multiple transactions in one block).
2. Record the block hash after the block is committed.
3. Call `eth_getLogs({"blockHash": "<recorded hash>"})`.
4. Observe: the node returns all logs with no error, allocating memory proportional to the log count, bypassing the configured `RPCLogsCap`.
5. With a sufficiently large block, the node process OOMs and crashes, halting consensus participation.

func (f *Filter) blockLogs(blockRes *tmrpctypes.ResultBlockResults, bloom ethtypes.Bloom) ([]*ethtypes.Log, error) {
	if !bloomFilter(bloom, f.criteria.Addresses, f.criteria.Topics) {
		return []*ethtypes.Log{}, nil
	}

	logsList, err := backend.GetLogsFromBlockResults(blockRes)
	if err != nil {
		return []*ethtypes.Log{}, errors.Wrapf(err, "failed to fetch logs block number %d", blockRes.Height)
	}

	unfiltered := make([]*ethtypes.Log, 0)
	for _, logs := range logsList {
		unfiltered = append(unfiltered, logs...)
	}

	logs := FilterLogs(unfiltered, nil, nil, f.criteria.Addresses, f.criteria.Topics)
	if len(logs) == 0 {
		return []*ethtypes.Log{}, nil
	}

	return logs, nil
