Audit Report

## Title
Unattributed `address(this).balance` in `pay()` enables theft of stranded native ETH via `refundETH()` or zero-cost WETH swap ‚Äî (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` funds WETH swap callbacks using `address(this).balance` ‚Äî the router's total ETH balance ‚Äî with no per-transaction attribution. When a user calls `exactInputSingle` with a `priceLimitX64` that causes a partial fill, only the actual swap amount is consumed as WETH; the remainder stays on the router unowned. Any subsequent caller can steal this stranded ETH via the unrestricted `refundETH()` function, or consume it for free through their own WETH swap callback.

## Finding Description

`pay()` in `PeripheryPayments.sol` (lines 73‚Äì84) reads `address(this).balance` to determine how much native ETH to wrap and forward to the pool. There is no transient-storage slot or any other mechanism recording how much ETH the current transaction contributed:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // total router ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) { ... }
    else { IERC20(WETH).safeTransferFrom(payer, recipient, value); }
}
```

**Stranding path.** `exactInputSingle` (lines 67‚Äì86 of `MetricOmmSimpleRouter.sol`) is `payable` and accepts a `priceLimitX64`. When the price limit is hit, the pool partially fills: the swap callback is invoked with `actualAmountIn < params.amountIn`. `pay()` deposits only `actualAmountIn` ETH as WETH and transfers it to the pool. The remaining `msg.value ‚àí actualAmountIn` ETH stays on the router. Crucially, `exactInputSingle` does **not** revert on partial fills ‚Äî it only checks `amountOut >= amountOutMinimum` (line 83). (By contrast, `exactInput` explicitly reverts on partial fills at line 115: `if (amountInActual < amount) revert InvalidInputAmountAtHop(...)`.)

The same stranding occurs for `exactOutputSingle` when `msg.value > actualAmountIn`.

**`receive()` does not prevent this.** It only rejects plain ETH transfers from non-WETH addresses (line 32‚Äì34); it does not intercept `msg.value` arriving inside a function call.

**Theft vector 1 ‚Äî `refundETH()`.** Lines 58‚Äì63 of `PeripheryPayments.sol`:
```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) { _transferETH(msg.sender, balance); }
}
```
No access control. Any attacker observing stranded ETH calls `refundETH()` and receives the entire router balance.

**Theft vector 2 ‚Äî zero-cost WETH swap.** An attacker calls `exactInputSingle{value: 0}` for a WETH‚Üítoken swap. Inside the callback, `pay(WETH, attacker, pool, value)` is invoked. Because `address(this).balance` contains the victim's stranded ETH and `nativeBalance >= value`, the router wraps the victim's ETH as WETH and forwards it to the pool. The attacker receives tokens without spending any ETH or WETH.

## Impact Explanation

Direct loss of user principal. A victim who sends ETH for a WETH swap that partially fills loses the unspent ETH to the first caller of `refundETH()`, or to an attacker who executes a zero-cost WETH swap consuming the victim's ETH. The loss equals the stranded amount, bounded only by the victim's `msg.value`. This meets the Critical/High threshold for direct loss of user principal with no privileged access required.

## Likelihood Explanation

Medium. The standard multicall pattern (`[exactInputSingle, refundETH]`) is atomic and leaves no residue. However, any user who calls `exactInputSingle` or `exactOutputSingle` directly with a `priceLimitX64` that causes a partial fill, or who over-estimates `msg.value`, will strand ETH. Attackers can monitor the router's ETH balance in the mempool and front-run the victim's next transaction with a `refundETH()` call, or back-run the stranding transaction in the same block. No privileged access is required.

## Recommendation

Track the ETH contributed by the current call in transient storage at each payable entry-point (e.g., `tstore(SLOT_MSG_VALUE, msg.value)`) and use only that recorded amount inside `pay()` instead of `address(this).balance`. Alternatively, for WETH exact-input swaps, enforce `msg.value == amountIn` and revert on surplus, mirroring the pattern used by Uniswap v3's `SwapRouter` which stores `msg.value` in a transient slot and subtracts from it as ETH is consumed. Additionally, add a post-swap check in `exactInputSingle` that reverts if `actualAmountIn < params.amountIn` and `msg.value > 0`, consistent with the guard already present in `exactInput` at line 115.

## Proof of Concept

```
Setup:
  - WETH/TOKEN pool registered on factory.
  - Alice holds 1000 ETH; Bob holds 0 ETH.

Step 1 ‚Äî Alice strands ETH:
  Alice calls:
    router.exactInputSingle{value: 1000}(ExactInputSingleParams{
        tokenIn: WETH, tokenOut: TOKEN, zeroForOne: true,
        amountIn: 1000,
        priceLimitX64: <tight limit causing partial fill at 600>,
        amountOutMinimum: 0, recipient: alice, ...
    })
  Pool partially fills; callback requests 600 ETH.
  pay(WETH, alice, pool, 600) ‚Üí deposits 600 ETH as WETH, transfers to pool.
  Remaining 400 ETH stays on router (address(router).balance == 400).
  exactInputSingle does NOT revert (amountOut >= 0 == amountOutMinimum).

Step 2a ‚Äî Bob steals via refundETH:
  Bob calls: router.refundETH()
  ‚Üí router sends 400 ETH to Bob. Alice loses 400 ETH.

Step 2b ‚Äî Bob steals via zero-cost WETH swap:
  Bob calls:
    router.exactInputSingle{value: 0}(ExactInputSingleParams{
        tokenIn: WETH, tokenOut: TOKEN, zeroForOne: true,
        amountIn: 300, amountOutMinimum: 0, recipient: bob, ...
    })
  Callback: pay(WETH, bob, pool, 300).
  address(this).balance == 400 >= 300 ‚Üí deposits Alice's 300 ETH as WETH.
  Bob receives TOKEN output without spending any ETH or WETH.
```
