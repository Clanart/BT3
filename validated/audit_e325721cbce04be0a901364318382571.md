### Title
Missing Token/Mint Binding in `FinalizeTransfer` Signed Payload Allows Wrong Token to Be Minted or Drained - (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary
The Solana `FinalizeTransfer` instruction verifies a signed payload from NEAR's MPC but the payload does not include the token (mint) address. The `mint` account passed by the caller has no constraint linking it to the signed payload. An attacker can reuse a valid signed payload for token A and substitute a different bridge-deployed token B as the `mint`, causing the program to mint or transfer token B instead of token A.

### Finding Description
`FinalizeTransferPayload` on the Solana side is defined as:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
``` [1](#0-0) 

There is no `token` or `mint` field. The `AdditionalParams = (Pubkey, Pubkey)` comment labelled `// mint, recipient` is used only for serializing the **outgoing** `FinalizeTransferResponse` back to NEAR — it is not part of the incoming signed payload that NEAR's MPC signs. [2](#0-1) 

In the `FinalizeTransfer` accounts struct, the `mint` account carries only a token-program constraint:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [3](#0-2) 

There is no `constraint` or `seeds` check that ties `mint.key()` to any field in `data.payload`. The vault account is derived from the caller-supplied `mint`:

```rust
seeds = [VAULT_SEED, mint.key().as_ref()],
``` [4](#0-3) 

In `process`, the bridged-token path only checks that the mint's authority is the bridge PDA — it does not verify the mint matches the intended token:

```rust
require!(
    self.mint.mint_authority.contains(self.authority.key),
    ErrorCode::InvalidBridgedToken
);
``` [5](#0-4) 

The `FinalizeTransferResponse` sent back to NEAR reports `self.mint.key()` — the attacker-supplied mint — not the originally intended token. [6](#0-5) 

### Impact Explanation
An attacker who obtains a valid `SignedPayload<FinalizeTransferPayload>` for a transfer of token A (e.g., USDC) can call `finalize_transfer` substituting token B (e.g., USDT) as the `mint`:

- **Bridged token path**: The program mints token B to the recipient without any corresponding lock of token B on NEAR. This is unauthorized minting of bridged assets.
- **Native token path**: The program transfers token B out of its vault using a proof that only authorizes releasing token A. This is direct theft of native assets from the vault.

In both cases the destination nonce for the original transfer is consumed, permanently preventing the legitimate transfer of token A from ever being finalized (irrecoverable lock of user funds on NEAR).

### Likelihood Explanation
Any unprivileged user who can observe a valid signed payload (e.g., by watching the mempool or NEAR events) can execute this attack. No privileged access is required. The attacker only needs to call `finalize_transfer` with the intercepted payload and a different mint address. The attack is straightforward and requires no special tooling beyond a standard Solana transaction.

### Recommendation
Add a constraint in the `FinalizeTransfer` accounts struct that binds the `mint` account to the token recorded in the signed payload. The `FinalizeTransferPayload` should include the expected Solana mint `Pubkey`, and the accounts constraint should enforce:

```rust
#[account(
    mut,
    constraint = mint.key() == data.payload.token @ ErrorCode::InvalidMint,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

Correspondingly, NEAR's MPC must sign the Solana mint address as part of the payload it produces for Solana-bound transfers.

### Proof of Concept

1. Alice initiates a transfer of 1000 USDC from NEAR to Solana. NEAR's MPC produces a `SignedPayload<FinalizeTransferPayload>` with `{destination_nonce: N, transfer_id: T, amount: 1000, fee_recipient: None}`. The USDC mint address is **not** in this payload.
2. Attacker observes the signed payload (e.g., from NEAR event logs or the relayer's pending queue).
3. Attacker calls `finalize_transfer` on Solana with the valid signed payload but passes in the USDT mint (a bridge-deployed token whose `mint_authority` is the bridge PDA) instead of the USDC mint.
4. Solana verifies the signature (valid — covers only nonce/transfer_id/amount/fee_recipient), marks nonce N as used, and mints 1000 USDT to Alice's token account.
5. Alice receives 1000 USDT instead of 1000 USDC. The attacker has obtained 1000 USDT backed by nothing on NEAR.
6. Nonce N is consumed; Alice's original USDC transfer can never be finalized — her 1000 USDC is permanently locked on NEAR.
