### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash without verifying that it corresponds to a declared contract class. An attacker who controls a contract can replace its class with a non-existent class hash, permanently rendering the contract non-callable and freezing any funds it holds. The developers themselves acknowledge this gap with an explicit TODO comment in the code.

---

### Finding Description

In `execute_replace_class`, the OS reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` with no validation that the hash corresponds to a class that has actually been declared on-chain:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO at line 898 explicitly acknowledges the missing check. Once the state is committed with an invalid class hash, any subsequent call to that contract will fail at the OS execution layer because no compiled class exists for the hash. Crucially, the only mechanism to repair a contract's class hash is `replace_class` itself — which requires successfully calling the contract. Since the contract is now permanently uncallable, the class hash can never be corrected, and any funds held by the contract are frozen forever.

This is structurally identical to the M-02 Cally bug: just as Cally accepted `token = address(this)` and produced a vault whose ownership was immediately transferred to the contract itself (making it permanently broken), the StarkNet OS accepts `class_hash = <undeclared value>` and produces a contract whose execution context is permanently broken.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After `replace_class` is called with a non-existent class hash:

- The contract's `StateEntry.class_hash` is set to an undeclared value in `contract_state_changes`.
- Every future invocation of the contract (invoke, L1 handler, or internal call) will fail when the OS attempts to resolve the compiled class for that hash.
- There is no recovery path: `replace_class` can only be called from within the contract, which is now permanently uncallable.
- All token balances, ETH equivalents, or other assets stored in the contract's storage are irrecoverably frozen.

---

### Likelihood Explanation

The entry path is fully accessible to any unprivileged contract deployer:

1. Any user can declare a class and deploy a contract.
2. Any deployed contract can issue the `replace_class` syscall (selector `REPLACE_CLASS_SELECTOR`) from within its own execution.
3. The syscall request carries an attacker-controlled `class_hash` field with no on-chain constraint.

A realistic attack scenario is a malicious vault or escrow contract that solicits user deposits and then calls `replace_class` with a garbage hash to permanently freeze those deposits. The attack requires no privileged access, no leaked keys, and no operator cooperation.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, verify that it exists as a declared class by checking `contract_class_changes` for a non-zero compiled class hash entry:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the mitigation recommended for M-02 (`require(token != address(this))`): a single guard at the entry point of the state-mutating operation that rejects inputs which would produce an irrecoverable state.

---

### Proof of Concept

1. **Attacker declares** class `A` (a valid Cairo contract) and deploys contract `C` using class `A`.
2. **Users deposit funds** into contract `C` (e.g., by calling a `deposit` entry point).
3. **Attacker triggers** a call path inside `C` that issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared hash).
4. **The OS executes** `execute_replace_class`: no validation is performed; `contract_state_changes[C].class_hash` is updated to `0xdeadbeef`. [2](#0-1) 
5. **The block is proven and committed** to L1. The state root now encodes `C` with class hash `0xdeadbeef`.
6. **Any subsequent transaction** targeting `C` (invoke, L1 handler, internal call) fails: the OS cannot resolve a compiled class for `0xdeadbeef`. [3](#0-2) 
7. **Funds are permanently frozen.** No `replace_class` can be issued to repair `C` because `C` itself cannot be called. The only state-change mechanism for `class_hash` is `replace_class`, creating the same irrecoverable circular dependency as M-02.
