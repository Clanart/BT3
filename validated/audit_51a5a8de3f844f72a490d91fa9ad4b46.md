### Title
`ProofOfInclusion::valid()` Never Verifies Against an External Trusted Root — Forged Inclusion Proofs Always Pass - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle library only checks that the proof's internal hash chain is self-consistent. It never compares the computed root against any external, trusted root hash. Because `root_hash()` returns the last `combined_hash` stored inside the proof itself, the final equality check in `valid()` is a tautology that is always `true` after the loop. An attacker who can deliver a serialized `ProofOfInclusion` to any DataLayer client can forge a proof for an arbitrary key-value pair and have it accepted as valid.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← always true after the loop
}
``` [1](#0-0) 

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same field already checked in the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` equals the last `calculated_hash`, which equals the last `layer.combined_hash` (the loop would have returned `false` otherwise). `root_hash()` returns that same `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is unconditionally `true` once the loop exits normally. The function never accepts an external trusted root to compare against.

`ProofOfInclusion` is a `Streamable` type fully exposed through Python bindings — it can be constructed from raw bytes (`from_bytes`), from JSON (`from_json_dict`), or directly via `__new__`: [3](#0-2) [4](#0-3)

### Impact Explanation

Any DataLayer client that receives a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` to verify it will accept a completely fabricated proof. The attacker can claim that any arbitrary key-value pair (represented by any `node_hash`) exists in any DataLayer store, without knowing the actual tree contents or root. This lets untrusted input prove invalid state, matching the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion or lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a Streamable type with Python bindings, designed to be transmitted between DataLayer peers. Any DataLayer node that receives and validates proofs from untrusted sources using only `valid()` is directly exploitable. No special privileges or keys are required — the attacker only needs to send a crafted serialized `ProofOfInclusion`.

### Recommendation

`valid()` must accept an external trusted root hash parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external root, not self.root_hash()
}
```

The existing `valid()` method (which only checks internal consistency) should either be removed or clearly documented as insufficient for security purposes. All callers that use `valid()` to accept proofs from untrusted sources must be updated to use the root-anchored variant.

### Proof of Concept

Construct a forged `ProofOfInclusion` for an arbitrary `node_hash` with a single internally consistent layer:

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob
from chia_rs.datalayer import calculate_internal_hash  # or compute manually

# Arbitrary fake leaf hash (attacker-chosen)
fake_node_hash = bytes([0xAA] * 32)
# Arbitrary sibling hash
fake_sibling   = bytes([0xBB] * 32)

# Compute a combined_hash that is internally consistent:
# combined = internal_hash(fake_node_hash, fake_sibling)
import hashlib
combined = hashlib.sha256(b'\x02' + fake_node_hash + fake_sibling).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=1,        # Side::Right
    other_hash=fake_sibling,
    combined_hash=combined,
)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated from any real tree
assert forged_proof.valid() == True
# root_hash() returns combined — the attacker-controlled value
assert forged_proof.root_hash() == combined
```

The proof passes `valid()` because the loop verifies `calculate_internal_hash(fake_node_hash, Right, fake_sibling) == combined`, which is true by construction, and the final check `existing_hash == self.root_hash()` compares `combined` to `combined` — always true. [1](#0-0) [6](#0-5)
