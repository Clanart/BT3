### Title
Receipt Commitment Hardcodes `Felt::ZERO` for Per-Transaction L2 Gas Consumed, Producing Wrong Block Hash for Post-0.14.2 Blocks — (`File: crates/starknet_api/src/block_hash/receipt_commitment.rs`)

---

### Summary

`chain_gas_consumed` in the receipt-commitment path chains a uniform `Felt::ZERO` for the L2-gas-consumed slot of every transaction leaf, instead of the actual per-transaction `gas_consumed.l2_gas` value. Starting with Starknet 0.14.2, transactions carry non-zero L2 gas (e.g., `76 014 735` units per transaction in the fixture data). The hardcoded zero produces a wrong leaf hash for every such transaction, a wrong `ReceiptCommitment`, and therefore a wrong `BlockHeaderCommitments.receipt_commitment` that propagates into the partial block hash and the final block hash.

---

### Finding Description

**Root cause — `chain_gas_consumed` ignores `gas_consumed.l2_gas`:**

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)               // ← hardcoded; should be gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

The comment reads *"L2 gas consumed (In the current RPC: always 0)"*, which was accurate before Starknet 0.14.2. From 0.14.2 onward, `total_gas_consumed.l2_gas` is non-zero for every L2-gas-priced transaction. The fixture `block_post_0_14_2.json` confirms this: every transaction in that block carries `"l2_gas": 76014735` in `total_gas_consumed`, and the block-level `l2_gas_consumed` is `988 191 555`.

**Data flow from execution to commitment:**

1. `BlockBuilder` calls `prepare_txs_hashing_data`, which calls `info.output_for_hashing()` to populate `TransactionOutputForHash.gas_consumed` (a `GasVector` with `l1_gas`, `l1_data_gas`, `l2_gas`).
2. `calculate_block_commitments` converts each entry to a `ReceiptElement` and calls `calculate_receipt_commitment::<Poseidon>`.
3. `calculate_receipt_hash` calls `chain_gas_consumed`, which silently drops `gas_consumed.l2_gas` and chains `Felt::ZERO`.
4. The resulting `ReceiptCommitment` is stored in `BlockHeaderCommitments.receipt_commitment`.
5. `PartialBlockHashComponents::new` embeds these commitments; `PartialBlockHash::from_partial_block_hash_components` and ultimately `calculate_block_hash` derive the final block hash from them.

**Analog to the EIP-2981 pattern:**

Just as `ERC721Bridgable._setDefaultRoyalty(address(this), _royaltyBps)` applies a single uniform royalty amount to every NFT instead of tracking per-token values, `chain_gas_consumed` applies a single uniform `Felt::ZERO` to every transaction's L2-gas slot instead of using the actual per-transaction value. Both bugs erase per-item information by substituting a global constant.

---

### Impact Explanation

Every transaction in a post-0.14.2 block that consumed non-zero L2 gas produces a wrong receipt leaf hash. The wrong leaves propagate through the Patricia tree to a wrong `ReceiptCommitment`. That wrong commitment is embedded in `BlockHeaderCommitments`, which feeds `PartialBlockHashComponents` and the final block hash. The sequencer therefore commits and broadcasts a block whose `receipt_commitment` and block hash do not match the true execution output.

This matches the Critical impact category: **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**

---

### Likelihood Explanation

The condition is met by any block produced on Starknet 0.14.2 or later that contains at least one transaction with non-zero L2 gas. The fixture `block_post_0_14_2.json` shows that every transaction in such a block has `l2_gas: 76014735`. The sequencer is already handling 0.14.2 blocks (the fixture is a production resource), so the wrong commitment is produced on every such block.

---

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual per-transaction L2 gas value:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // actual per-transaction L2 gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

Remove or update the stale comment. Add a regression test that constructs a `ReceiptElement` with non-zero `l2_gas` and asserts the resulting hash differs from the hash produced with `l2_gas = 0`.

---

### Proof of Concept

**Step 1 — Confirm non-zero L2 gas in production blocks.**

`crates/apollo_starknet_client/resources/reader/block_post_0_14_2.json` shows:
```json
"total_gas_consumed": { "l1_gas": 0, "l1_data_gas": 128, "l2_gas": 76014735 }
```
and block-level `"l2_gas_consumed": 988191555`.

**Step 2 — Trace the wrong value into the commitment.**

`chain_gas_consumed` (line 88) chains `Felt::ZERO` unconditionally. For a transaction with `l2_gas = 76014735`, the leaf hash is:

```
Poseidon(tx_hash, fee, messages_hash, revert_hash,
         0,          ← wrong: should be 76014735
         l1_gas, l1_data_gas)
```

**Step 3 — Propagate to block hash.**

`calculate_receipt_commitment` builds a Patricia tree over the wrong leaves → wrong `ReceiptCommitment` → stored in `BlockHeaderCommitments.receipt_commitment` → embedded in `PartialBlockHashComponents` → used by `calculate_block_hash`.

**Exact corrupted value:** `BlockHeaderCommitments.receipt_commitment` for any block containing a transaction with `gas_consumed.l2_gas > 0`.
