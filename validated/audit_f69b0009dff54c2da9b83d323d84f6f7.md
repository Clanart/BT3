### Title
Insufficient 5 TGas Callback Budget Causes Irrecoverable Fund Lock on BTC Connector Rejection â€” (`near/omni-bridge/src/btc.rs`)

---

### Summary

`submit_transfer_to_btc_connector_callback` is allocated only 5 TGas. Its failure branch must synchronously execute `add_transfer_message` (storage write proportional to `TransferMessage` size) and `update_storage_balance` (storage read + write). Because `remove_transfer_message` is committed **before** the cross-contract call at line 84, a callback OOG panic leaves the transfer permanently deleted with no recovery path.

---

### Finding Description

In `near/omni-bridge/src/btc.rs`: [1](#0-0) 

The constant is 5 TGas. The outer function removes the transfer unconditionally before dispatching the cross-contract call: [2](#0-1) 

The callback's failure branch (BTC connector returns 0 or errors) must re-insert the full `TransferMessage` and update the storage balance: [3](#0-2) 

`TransferMessage` contains a free-form `msg: String` field populated from the user-supplied `InitTransferMsg.msg` at transfer creation time. There is no enforced length cap on this field. The callback receives the full `TransferMessage` as a JSON argument, so deserialization cost scales with `msg` length. `add_transfer_message` then serializes and writes the entire struct to a `LookupMap`, and `update_storage_balance` performs an additional read-modify-write on `accounts_balances`: [4](#0-3) 

**Gas budget breakdown for the failure path:**

| Step | Approximate cost |
|---|---|
| NEAR base function-call overhead | ~2.3 TGas |
| Deserialize large `TransferMessage` arg | ~0.5â€“2 TGas (scales with `msg` size) |
| `add_transfer_message` (serialize + LookupMap insert) | ~1â€“3 TGas |
| `update_storage_balance` (LookupMap read + write) | ~0.5â€“1 TGas |
| **Total** | **~4.3â€“8+ TGas** |

5 TGas is exhausted before the failure path completes for any `TransferMessage` with a `msg` field of even moderate size (~500 bytes+). Comparable callbacks in the same contract use 10 TGas: [5](#0-4)

### Impact Explanation

When the callback panics due to OOG:
- The transfer record was already deleted by `remove_transfer_message` (line 84) â€” this write is committed.
- The callback panic rolls back only the callback's own state changes, not the parent call's committed deletion.
- The `TransferMessage` is gone from `pending_transfers`; the user's locked tokens are gone from the BTC token contract (returned by `ft_transfer_call` refund), but the bridge has no record to re-credit them.
- There is no admin recovery function that can reconstruct an erased transfer.

**Result: irrecoverable lock / permanent loss of user funds.**

---

### Likelihood Explanation

1. A user submits a BTC withdrawal via the public `ft_on_transfer` / `init_transfer` path with a `msg` field padded to ~1 KB (valid JSON or arbitrary bytes â€” the `DestinationChainMsg` parse is optional and silently ignored on failure).
2. A trusted relayer calls `submit_transfer_to_utxo_chain_connector` (normal operation).
3. The BTC connector's `ft_on_transfer` returns 0 for any legitimate reason (UTXO validation failure, dust limit, connector-side error).
4. The callback OOGs; the transfer is permanently erased.

Step 1 is a fully public call. Steps 2â€“3 are normal operational paths. No privileged collusion is required.

---

### Recommendation

Raise `SUBMIT_TRANSFER_TO_BTC_CONNECTOR_CALLBACK_GAS` to at least **15â€“20 TGas**, consistent with the budget used by analogous callbacks (`FAST_TRANSFER_CALLBACK_GAS = 10`, `UTXO_FIN_TRANSFER_CALLBACK_GAS = 10`), and add a maximum length validation on `InitTransferMsg.msg` at transfer-creation time to bound worst-case deserialization cost.

    fn update_storage_balance(
        &mut self,
        account_id: AccountId,
        required_balance: NearToken,
        attached_deposit: NearToken,
    ) {
        self.try_update_storage_balance(account_id, required_balance, attached_deposit)
            .unwrap_or_else(|err| env::panic_str(err.to_string().as_str()));
    }

    fn try_update_storage_balance(
        &mut self,
        account_id: AccountId,
        required_balance: NearToken,
        attached_deposit: NearToken,
    ) -> Result<(), StorageBalanceError> {
        if attached_deposit >= required_balance {
            Self::refund(
                account_id,
                attached_deposit.saturating_sub(required_balance),
            );
            Ok(())
        } else {
            let required_balance = required_balance.saturating_sub(attached_deposit);

            let Some(mut storage_balance) = self.accounts_balances.get(&account_id) else {
                return Err(StorageBalanceError::AccountNotRegistered(account_id));
            };

            if storage_balance.available >= required_balance {
                storage_balance.available =
                    storage_balance.available.saturating_sub(required_balance);
                self.accounts_balances.insert(&account_id, &storage_balance);

                Ok(())
            } else {
                Err(StorageBalanceError::NotEnoughStorage {
                    required: required_balance,
                    available: storage_balance.available,
                })
            }
        }
    }
