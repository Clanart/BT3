### Title
`fin_transfer_send_tokens_callback` Does Not Handle Token Transfer Failure — Permanent Fund Loss on NEAR - (File: near/omni-bridge/src/lib.rs)

### Summary

The NEAR bridge's `fin_transfer_send_tokens_callback` function does not check whether the underlying token transfer (mint or `ft_transfer`) actually succeeded. If the token operation fails after the transfer has already been marked as finalized and locked-token accounting has been updated, the user's funds are permanently irrecoverable: burned on the source chain, never minted on NEAR, and the nonce is consumed so the transfer cannot be retried.

### Finding Description

In `process_fin_transfer_to_near`, the bridge performs three state-mutating operations synchronously before dispatching the async token transfer:

1. `add_fin_transfer` — marks the transfer ID as finalized in `finalised_transfers`
2. `unlock_tokens_if_needed` — decrements `locked_tokens` accounting
3. `send_tokens` — dispatches a cross-contract call to mint or transfer tokens to the recipient [1](#0-0) 

The callback `fin_transfer_send_tokens_callback` is then invoked. Its only recovery branch is gated on `is_refund_required(is_ft_transfer_call)`: [2](#0-1) 

`is_refund_required` returns `true` only when `is_ft_transfer_call` is `true` **and** the promise result deserializes to `U128(0)`. In all other cases — including when the promise outright fails (`Err(_)`) — it returns `false`: [3](#0-2) 

`is_ft_transfer_call` is set to `!msg.is_empty()`. For the common case of a plain transfer (empty `msg`), `is_ft_transfer_call` is `false`, so `is_refund_required` unconditionally returns `false` regardless of whether `send_tokens` succeeded or failed.

Even for non-empty `msg` (ft_transfer_call path), a hard failure of the cross-contract call returns `Err(_)` from `promise_result_checked`, which also returns `false` — no refund is triggered.

The callback therefore always falls through to the `else` branch, emitting `FinTransferEvent` as if the transfer succeeded: [4](#0-3) 

Because `finalised_transfers` already contains the transfer ID, any retry of `fin_transfer` with the same proof will be rejected by `add_fin_transfer`. The user has no recovery path.

### Impact Explanation

For bridged (deployed) tokens: the source-chain tokens were burned during `initTransfer`; on NEAR, `mint` is called but fails (e.g., token contract paused, supply cap reached, controller revoked). The transfer is permanently finalized, the tokens are never minted, and the user's funds are irrecoverably lost. This matches **Critical — Permanent freezing, irrecoverable lock of user funds in bridge flows**.

For native tokens locked in the bridge: if `ft_transfer` fails, the tokens remain physically in the bridge contract but the accounting entry is already decremented and the transfer is finalized. The tokens are stranded with no claim mechanism.

### Likelihood Explanation

The NEAR bridge supports arbitrary NEP-141 tokens. Any of the following realistic conditions causes `send_tokens` to fail:

- The bridged token contract is paused (e.g., emergency security pause by the token controller)
- The token has a per-period mint cap that has been reached (analogous to USDC's `mintingAllowedAmount`)
- The token controller has been revoked or changed between proof submission and callback execution
- The token contract has a bug that causes `mint` to panic under certain amounts

These are not attacker-controlled conditions; they are operational states that can occur independently of the bridge user.

### Recommendation

In `fin_transfer_send_tokens_callback`, explicitly check the promise result for failure regardless of `is_ft_transfer_call`. If the promise failed, revert the finalization and re-lock the tokens:

```rust
pub fn fin_transfer_send_tokens_callback(...) {
    let promise_failed = env::promise_result(0) == PromiseResult::Failed;
    
    if promise_failed || Self::is_refund_required(is_ft_transfer_call) {
        // Revert: re-lock tokens, remove fin_transfer record
        self.revert_lock_actions(&lock_actions);
        self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
        // For deployed tokens that were minted then returned, burn them
        if !promise_failed {
            self.burn_tokens_if_needed(token, ...);
        }
        env::log_str(&OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string());
    } else {
        // existing success path
    }
}
```

Removing the finalization record allows the relayer to resubmit the proof once the token contract is operational again.

### Proof of Concept

1. User bridges 1000 USDC from Ethereum to NEAR via `initTransfer` — tokens are burned on EVM.
2. Relayer calls `fin_transfer` on NEAR with valid proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` commits the transfer ID to `finalised_transfers`. [5](#0-4) 
   - `unlock_tokens_if_needed` decrements `locked_tokens`. [6](#0-5) 
   - `send_tokens` dispatches `mint(recipient, 1000)` to the NEAR USDC token contract.
4. The NEAR USDC token contract is paused → `mint` panics → promise result is `Failed`.
5. `fin_transfer_send_tokens_callback` is called. `is_ft_transfer_call = false` (empty msg). `is_refund_required()` returns `false`. Callback emits `FinTransferEvent`. [2](#0-1) 
6. Relayer retries `fin_transfer` with the same proof → `add_fin_transfer` panics with `TransferAlreadyFinalized`.
7. User's 1000 USDC are permanently lost: burned on Ethereum, never minted on NEAR, no recovery path.

        if is_ft_transfer_call {

                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {

                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.

                        // Unexpected case: don't refund
                        false
                    }
                }

                Err(_) => false,
            }

            // Not ft_transfer_call: don't refund
