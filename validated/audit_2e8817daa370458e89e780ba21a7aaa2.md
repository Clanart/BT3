### Title
EIP-7702 Self-Delegation Authorization Always Silently Fails Due to Ante Handler Nonce Pre-Increment — (`x/evm/keeper/set_code_authorizations.go`)

---

### Summary

The `validateAuthorization` function checks the authority's nonce against `auth.Nonce` by reading `stateDB.GetNonce(authority)`. For self-delegating EIP-7702 (`SetCodeTx`) transactions — where the transaction sender is also the authorization authority — the ante handler has already incremented the sender's nonce in the Cosmos account store before EVM execution begins. The freshly-created `stateDB` reads the committed (post-increment) nonce, causing the nonce equality check to always fail. The authorization is then silently skipped, and the EVM call executes without the intended code delegation ever being applied.

---

### Finding Description

**Step 1 — Ante handler pre-increments the sender's nonce.**

`CheckAndSetEthSenderNonce` reads the current account sequence `N`, verifies `txNonce == N`, increments it to `N+1`, and persists it via `ak.SetAccount(ctx, acc)`: [1](#0-0) 

**Step 2 — A fresh `stateDB` is created from the now-committed state.**

Inside `ApplyMessageWithConfig`, a brand-new `stateDB` is constructed from the same `ctx` that the ante handler already wrote to: [2](#0-1) 

Because the ante handler committed `nonce = N+1` to the store, `stateDB.GetNonce(sender)` returns `N+1`.

**Step 3 — Authorization nonce check fails for self-delegation.**

`validateAuthorization` enforces strict equality:

```go
if have := stateDB.GetNonce(authority); have != auth.Nonce {
    return authority, core.ErrAuthorizationNonceMismatch
}
``` [3](#0-2) 

For a self-delegating transaction, `authority == sender`. The user signed the authorization with `auth.Nonce = N` (the nonce at signing time). But `stateDB.GetNonce(authority)` now returns `N+1`. The check `N+1 != N` is always true → `ErrAuthorizationNonceMismatch` is returned.

**Step 4 — The error is silently swallowed.**

The authorization processing loop in `ApplyMessageWithConfig` explicitly ignores errors and simply skips invalid authorizations:

```go
for _, auth := range msg.SetCodeAuthorizations {
    authority, err := k.applyAuthorization(&auth, stateDB)
    if err != nil {
        k.Logger(ctx).Debug("failed to apply authorization", ...)
        continue   // ← silently skipped
    }
    ...
}
``` [4](#0-3) 

**Step 5 — EVM call proceeds without the delegation.**

After the loop, `evm.Call` is invoked unconditionally: [5](#0-4) 

The sender's account code is never updated. The call executes against the EOA as if no delegation was requested.

**Contrast with contract-creation path.**

For `CREATE` transactions, Ethermint explicitly resets the stateDB nonce to `msg.Nonce` before calling `evm.Create`, precisely to compensate for the ante handler's pre-increment:

```go
stateDB.SetNonce(sender, msg.Nonce, tracing.NonceChangeUnspecified)
ret, _, leftoverGas, vmErr = evm.Create(...)
``` [6](#0-5) 

No equivalent reset exists for the `CALL` path before authorization processing.

---

### Impact Explanation

Every EIP-7702 `SetCodeTx` where the signer of an authorization entry is the same account as the transaction sender (self-delegation) will have that authorization silently dropped. The transaction is accepted, fees are deducted, and the EVM call executes — but the intended code delegation is never written to state. Any logic that depends on the delegation being active (e.g., a factory pattern where the EOA delegates to a contract and immediately calls it) will execute against a bare EOA, producing incorrect or failed outcomes. This maps to the allowed High impact: **EIP-7702 authorization verification is broken, causing incorrect account/code mutation**.

---

### Likelihood Explanation

Self-delegation is the primary EIP-7702 use case (an EOA granting itself smart-contract behavior for a single transaction). Any user or protocol relying on this pattern on Ethermint is affected deterministically — there is no probabilistic element. The entry path is a standard unprivileged `eth_sendRawTransaction` submission.

---

### Recommendation

Before iterating over `msg.SetCodeAuthorizations` in the CALL branch of `ApplyMessageWithConfig`, reset the sender's nonce in the stateDB to `msg.Nonce`, mirroring the existing fix for the CREATE branch:

```go
// Compensate for ante handler pre-increment before validating authorizations.
if msg.SetCodeAuthorizations != nil {
    stateDB.SetNonce(sender, msg.Nonce, tracing.NonceChangeUnspecified)
    for _, auth := range msg.SetCodeAuthorizations {
        ...
    }
    // Restore the incremented nonce so evm.Call sees the correct value.
    stateDB.SetNonce(sender, msg.Nonce+1, tracing.NonceChangeUnspecified)
}
```

Alternatively, adjust `validateAuthorization` to accept `auth.Nonce + 1 == stateDB.GetNonce(authority)` when the authority equals the transaction sender, to account for the pre-increment.

---

### Proof of Concept

1. Alice holds account with Cosmos sequence / EVM nonce `N = 5`.
2. Alice constructs an EIP-7702 `SetCodeTx`:
   - `tx.Nonce = 5`
   - `auth.Nonce = 5`, `auth.Address = MyContract`, authority = Alice (self-delegation)
3. Ante handler runs `CheckAndSetEthSenderNonce`: sequence `5 → 6`, committed to store.
4. `ApplyMessageWithConfig` creates `stateDB`; `stateDB.GetNonce(Alice) = 6`.
5. `validateAuthorization`: `have=6 != auth.Nonce=5` → `ErrAuthorizationNonceMismatch`.
6. Authorization silently skipped; `evm.Call` executes against Alice's bare EOA.
7. Alice's account code is never set to `MyContract`'s delegation prefix; any call that expected delegated behavior silently misbehaves. [7](#0-6) [8](#0-7) [9](#0-8)

func CheckAndSetEthSenderNonce(
	ctx sdk.Context, tx sdk.Tx, ak evmtypes.AccountKeeper, unsafeUnOrderedTx bool, accountGetter AccountGetter, nonceCache *cache.AnteCache,
) ([]cache.TxNonce, error) {
	var pending []cache.TxNonce

	for _, msg := range tx.GetMsgs() {
		msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
		if !ok {
			return nil, errorsmod.Wrapf(errortypes.ErrUnknownRequest, "invalid message type %T, expected %T", msg, (*evmtypes.MsgEthereumTx)(nil))
		}

		tx := msgEthTx.AsTransaction()

		from := msgEthTx.GetFrom()
		acc := accountGetter(from)
		if acc == nil {
			return nil, errorsmod.Wrapf(
				errortypes.ErrUnknownAddress,
				"account %s is nil", common.BytesToAddress(from.Bytes()),
			)
		}
		expectedNonce := acc.GetSequence()
		txNonce := tx.Nonce()
		fromStr := from.String()

		// if flag is set, we bypass nonce all check verification
		if !unsafeUnOrderedTx {
			ex := nonceCache.Exists(fromStr, txNonce)
			// to support tx replacement, we check if the transaction nonce exists in the cache and if yes we skip
			// nonce verification, and we don't set the sequence
			// We allow skip verification only during CheckTx to keep sequence safe during the execution.
			if ctx.IsCheckTx() && !ctx.IsReCheckTx() && ex {
				continue
			}

			// nonce verification, the sequence needs to be in order
			if txNonce != expectedNonce {
				// delete in case of recheck tx
				if ex {
					nonceCache.Delete(fromStr, txNonce)
				}
				return nil, errorsmod.Wrapf(
					errortypes.ErrInvalidSequence,
					"invalid nonce; got %d, expected %d", txNonce, expectedNonce,
				)
			}

			if ctx.IsCheckTx() {
				if !ctx.IsReCheckTx() {
					pending = append(pending, cache.TxNonce{Address: fromStr, Nonce: txNonce})
				}
			} else if ex {
				// delete in case of deliver tx
				nonceCache.Delete(fromStr, txNonce)
			}
		}

		// increase sequence of sender
		if err := acc.SetSequence(expectedNonce + 1); err != nil {
			return nil, errorsmod.Wrapf(err, "failed to set sequence to %d", acc.GetSequence()+1)
		}

		ak.SetAccount(ctx, acc)
	}

func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
	// Verify chain ID is null or equal to current chain ID.
	if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
		return authority, core.ErrAuthorizationWrongChainID
	}
	// Limit nonce to 2^64-1 per EIP-2681.
	if auth.Nonce+1 < auth.Nonce {
		return authority, core.ErrAuthorizationNonceOverflow
	}
	// Validate signature values and recover authority.
	authority, err = auth.Authority()
	if err != nil {
		return authority, fmt.Errorf("%w: %v", core.ErrAuthorizationInvalidSignature, err)
	}
	// Check the authority account
	//  1) doesn't have code or has exisiting delegation
	//  2) matches the auth's nonce
	//
	// Note it is added to the access list even if the authorization is invalid.
	stateDB.AddAddressToAccessList(authority)
	code := stateDB.GetCode(authority)
	if _, ok := types.ParseDelegation(code); len(code) != 0 && !ok {
		return authority, core.ErrAuthorizationDestinationHasCode
	}
	if have := stateDB.GetNonce(authority); have != auth.Nonce {
		return authority, core.ErrAuthorizationNonceMismatch
	}
	return authority, nil
}
