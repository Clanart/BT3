### Title
`assert_nn_le` Range-Check Overflow in `charge_fee` When `compute_max_possible_fee` Exceeds 2^128 — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` returns a felt-arithmetic product of resource-bound fields that can legitimately exceed `2^128`. The immediately following `assert_nn_le(calldata.amount.low, max_fee)` call uses Cairo's range-check builtin, which is bounded to `[0, 2^128)`. When `max_fee > 2^128`, the subtraction `max_fee - calldata.amount.low` overflows the range-check bound, causing an irrecoverable assertion failure inside the OS program. A block containing any such transaction cannot produce a valid STARK proof, halting the network.

---

### Finding Description

**Root cause — `compute_max_possible_fee` (lines 87–102, `transaction_impls.cairo`)**

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
         + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
         + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
```

The only upstream validation of these fields occurs in `pack_resource_bounds` (`transaction_hash.cairo`, lines 103–108), called during hash computation:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount ≤ 2^64 − 1
    assert_nn(resource_bounds.max_price_per_unit);            // max_price_per_unit < 2^128
    ...
}
```

And `tip` is bounded to `2^64 − 1` in `hash_fee_fields` (line 117). With these validated bounds the maximum value of `max_fee` is:

```
max_fee_max ≈ 3 × (2^64 − 1) × (2^128 − 1) ≈ 3 × 2^192
```

This is orders of magnitude larger than `2^128`.

**Failure site — `charge_fee` (line 135, `transaction_impls.cairo`)**

```cairo
assert_nn_le(calldata.amount.low, max_fee);
```

`assert_nn_le(a, b)` from `starkware.cairo.common.math` expands to:
1. `assert_nn(a)` → range-check that `a ∈ [0, 2^128)`
2. `assert_nn(b − a)` → range-check that `b − a ∈ [0, 2^128)`

When `max_fee > 2^128`, the value `max_fee − calldata.amount.low` exceeds `2^128 − 1` (since `calldata.amount.low < 2^128`). The range-check builtin rejects it, causing an irrecoverable OS-program failure. No valid STARK proof can be generated for the block.

**Why the bounds check in `pack_resource_bounds` does not protect `charge_fee`**

`pack_resource_bounds` validates that `max_price_per_unit < 2^128` (not `< 2^64`). The product `max_amount × max_price_per_unit` with `max_amount = 2^64 − 1` and `max_price_per_unit = 2^65` already yields `≈ 2^129 > 2^128`, yet both individual fields pass their respective checks. There is no subsequent guard before `assert_nn_le` is reached.

---

### Impact Explanation

If the sequencer includes a transaction whose resource bounds produce `max_fee > 2^128`, the OS Cairo program aborts at `assert_nn_le` inside `charge_fee`. The STARK proof for that block cannot be generated. The sequencer must discard the block and rebuild it, but if it lacks a matching pre-filter, repeated submission of such transactions causes a sustained inability to finalize blocks — a total network shutdown.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a V3 invoke, deploy-account, or declare transaction with:

- `max_price_per_unit = 2^65` (passes `assert_nn`, i.e., `< 2^128`)
- `max_amount = 2^64 − 1` (passes `assert_nn_le(..., 2^64 − 1)`)

This yields `max_fee ≈ 2^129 > 2^128` for a single resource type alone. The transaction hash computation succeeds (bounds are individually valid), the account signature covers these values, and the sequencer may include the transaction if it does not independently compute and bound `max_fee`. The OS then fails at proof time. The attack requires no privileged access, no key compromise, and no external dependency.

---

### Recommendation

1. **Add an explicit upper bound on `max_fee` inside `charge_fee`** before calling `assert_nn_le`:

```cairo
// Ensure max_fee fits in a uint128 so assert_nn_le is safe.
assert_nn_le(max_fee, MAX_FEE_BOUND);   // MAX_FEE_BOUND = 2^128 − 1
assert_nn_le(calldata.amount.low, max_fee);
```

2. **Alternatively, tighten `pack_resource_bounds`** to bound `max_price_per_unit` to `2^64 − 1` (matching the Starknet spec), which would keep any single product within `2^128` and the three-term sum within `3 × 2^128`, still requiring a separate sum-overflow guard.

3. **Add a matching pre-filter in the sequencer** that rejects any transaction where the computed `max_fee` exceeds `2^128 − 1`, so the OS and the sequencer agree on validity before block construction.

---

### Proof of Concept

**Attacker constructs a V3 invoke transaction with:**

```
resource_bounds[L1_GAS] = { max_amount: 2^64 − 1, max_price_per_unit: 2^65 }
resource_bounds[L2_GAS] = { max_amount: 0, max_price_per_unit: 0 }
resource_bounds[L1_DATA_GAS] = { max_amount: 0, max_price_per_unit: 0 }
tip = 0
```

**Step 1 — Hash computation succeeds:**
- `pack_resource_bounds` for L1_GAS: `assert_nn_le(2^64 − 1, 2^64 − 1)` ✓; `assert_nn(2^65)` ✓ (since `2^65 < 2^128`).

**Step 2 — `compute_max_possible_fee` returns:**
```
max_fee = (2^64 − 1) × 2^65 = 2^129 − 2^65  ≈ 2^129
```

**Step 3 — `charge_fee` reaches line 135:**
```cairo
assert_nn_le(calldata.amount.low, max_fee)
// expands to: assert_nn(max_fee − calldata.amount.low)
// = assert_nn(2^129 − 2^65 − actual_fee)
// 2^129 − 2^65 > 2^128 − 1  →  range-check FAILS
```

**Step 4 — OS program aborts.** The STARK proof for the block cannot be generated. The sequencer cannot finalize the block, causing a network halt for as long as such transactions are submitted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
