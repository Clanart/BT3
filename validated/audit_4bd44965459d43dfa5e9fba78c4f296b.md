### Title
Missing Zero-Address Validation for `payload.recipient` in `finTransfer` Enables Permanent ETH Loss - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `finTransfer` function does not validate that `payload.recipient != address(0)` before dispatching assets. For the native-ETH branch (`tokenAddress == address(0)`), a low-level `.call{value: amount}("")` to `address(0)` succeeds silently in the EVM, permanently destroying the bridged ETH. The originating-chain entry point (`init_transfer` on NEAR) performs no zero-address guard on EVM-destined recipients, so a user can accidentally (or deliberately) route a transfer to the zero address and the full cross-chain flow completes without revert.

---

### Finding Description

**Root cause 1 — NEAR `init_transfer` accepts zero EVM recipient**

`near/omni-bridge/src/lib.rs` `init_transfer` validates only that the destination chain is not NEAR:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
```

No check is made that the inner address value is non-zero. A caller may supply `OmniAddress::Eth(H160::ZERO)` (or any EVM-chain variant: `Arb`, `Base`, `Bnb`, `Pol`, `HyperEvm`, `Abs`) and the transfer message is accepted, stored, and forwarded to the MPC for signing.

**Root cause 2 — EVM `finTransfer` dispatches to `address(0)` without guard**

`OmniBridge.sol` `finTransfer` (lines 279–367) contains no `require(payload.recipient != address(0))` check. For the native-ETH branch:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

A low-level `.call{value: N}("")` to `address(0)` returns `success = true` in the EVM (address(0) has no code; the call is treated as a value transfer to an EOA). The ETH is irrecoverably destroyed and the nonce is marked `completedTransfers[nonce] = true`, preventing any retry.

---

### Impact Explanation

When a user bridges native ETH from EVM → NEAR and then initiates a return transfer from NEAR → EVM with `recipient = OmniAddress::Eth(H160::ZERO)`:

1. The MPC signs the message (it has no independent zero-address policy).
2. `finTransfer` is called on EVM; the nonce is consumed.
3. ETH is sent to `address(0)` and permanently lost.
4. The nonce is marked used; the transfer cannot be re-executed.

This is an irrecoverable loss of user funds in the bridge flow — matching the "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds" impact class.

For bridge-token and standard ERC-20 paths the impact is mitigated: OpenZeppelin's `ERC20Upgradeable._mint` and `SafeERC20.safeTransfer` both revert on `address(0)`, so those branches are self-protecting. The native-ETH branch is the unprotected case.

---

### Likelihood Explanation

Any unprivileged user who holds bridged ETH on NEAR and calls `init_transfer` with a zero EVM recipient triggers this path. No privileged role is required. The mistake is easy to make programmatically (e.g., a front-end bug, an uninitialized address variable, or a scripted relayer passing a default-zero address). Once submitted, the transfer is irreversible.

---

### Recommendation

**On NEAR (`near/omni-bridge/src/lib.rs`):** Extend `init_transfer` to reject zero-valued EVM addresses:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
// Add:
require!(
    init_transfer_msg.recipient != OmniAddress::new_zero(
        init_transfer_msg.recipient.get_chain()
    ).unwrap(),
    BridgeError::InvalidRecipientAddress.as_ref()
);
```

**On EVM (`OmniBridge.sol`):** Add a guard at the top of `finTransfer`:

```solidity
if (payload.recipient == address(0)) revert InvalidRecipient();
```

Both layers should be hardened; defense-in-depth is appropriate here because the MPC layer does not enforce this constraint.

---

### Proof of Concept

1. User holds bridged ETH on NEAR (obtained by bridging ETH from EVM → NEAR).
2. User calls `ft_transfer_call` on the wETH token contract targeting the NEAR bridge with `InitTransferMsg { recipient: OmniAddress::Eth(H160::ZERO), fee: U128(0), native_token_fee: U128(0), msg: None, external_id: None }`.
3. NEAR `init_transfer` passes the only check (`get_chain() != ChainKind::Near`) and emits `InitTransferEvent` with `recipient = 0x0000000000000000000000000000000000000000`.
4. MPC observes the event and signs a `TransferMessagePayload` with `recipient = address(0)` and `tokenAddress = address(0)` (native ETH).
5. Anyone calls `OmniBridge.finTransfer(signatureData, payload)` on EVM.
6. `completedTransfers[payload.destinationNonce] = true` — nonce consumed.
7. `(bool success, ) = address(0).call{value: payload.amount}("")` — returns `true`, ETH destroyed.
8. `FailedToSendEther` is never triggered; the transaction succeeds; ETH is permanently lost.
