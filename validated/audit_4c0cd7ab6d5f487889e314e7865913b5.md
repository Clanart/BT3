### Title
Unchecked Arithmetic on User-Supplied `ResourceBounds` in `compute_max_possible_fee` Enables Fee Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function performs multiplication and addition on user-controlled `ResourceBounds` field elements (`max_amount`, `max_price_per_unit`, `tip`) without any range-check constraints. Because Cairo arithmetic is modulo the Stark field prime P ≈ 2²⁵¹, a transaction sender can craft field-element values that cause the fee sum to wrap around to zero. When `max_fee == 0`, the OS unconditionally skips fee charging, allowing the transaction to execute for free.

---

### Finding Description

`compute_max_possible_fee` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

All six operands (`max_amount` × 3, `max_price_per_unit` × 3, `tip`) are loaded from the hint `LoadCommonTxFields` and written directly into `CommonTxFields` without any `assert_nn_le` or `assert_lt` range-check constraints bounding them to their protocol-specified widths (u64 for `max_amount`, u128 for `max_price_per_unit`). [2](#0-1) 

The only validation applied to the resource bounds in the OS is:

```cairo
assert tx_info.version = 3;
assert n_resource_bounds = 3;
``` [3](#0-2) 

Neither `assert_deprecated_tx_fields_consistency` nor `fill_account_tx_info` adds any bounds check on the individual resource bound values. [4](#0-3) 

The result of `compute_max_possible_fee` is then used in `charge_fee` as follows:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [5](#0-4) 

If the attacker-chosen field elements cause the sum to equal 0 mod P, the OS returns immediately without executing the ERC-20 transfer, and the transaction is proven as valid with zero fee paid.

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

Fee revenue is the economic mechanism that compensates sequencers and funds the protocol. If `compute_max_possible_fee` returns 0 due to field-arithmetic wrap-around, `charge_fee` exits early and no ERC-20 transfer is executed. The transaction is fully executed (state changes committed, L2 gas consumed, L1 messages emitted) while the sequencer collects nothing. A single attacker can drain sequencer fee revenue across arbitrarily many transactions, constituting a direct, permanent loss of funds at the protocol level.

---

### Likelihood Explanation

**Likelihood: High.**

The attacker controls five independent felt-valued inputs (`l1_max_amount`, `l1_max_price`, `l2_max_amount`, `l2_max_price`, `tip`, `l1_data_max_amount`, `l1_data_max_price`) and must satisfy one equation over GF(P). This is trivially solvable: for example, set `l1_gas_bounds.max_amount = P − k` for any small k, and choose `l1_gas_bounds.max_price_per_unit = 1`, then adjust the remaining terms to cancel the residue. The transaction hash commits to these values (the user signs them), so no key leakage or privileged access is required. Any unprivileged transaction sender can construct and submit such a transaction.

---

### Recommendation

Add explicit range-check constraints on all `ResourceBounds` fields before performing arithmetic. Specifically, in `compute_max_possible_fee` (or in `get_account_tx_common_fields` where the values are first loaded), assert:

```cairo
assert_nn_le(l1_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_U128);
assert_nn_le(l2_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l2_gas_bounds.max_price_per_unit, MAX_U128);
assert_nn_le(l1_data_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l1_data_gas_bounds.max_price_per_unit, MAX_U128);
assert_nn_le(tx_info.tip, MAX_U64);
```

With these constraints, the maximum product of any pair is 2⁶⁴ × 2¹²⁸ = 2¹⁹², and the maximum sum of three such products is 3 × 2¹⁹² ≈ 2¹⁹³·⁶, which is far below P ≈ 2²⁵¹, making field-arithmetic overflow impossible.

---

### Proof of Concept

1. Choose field elements satisfying:
   ```
   A * B + C * (D + T) + E * F ≡ 0  (mod P)
   ```
   For example: set `A = P − 1`, `B = 1`, `C = 1`, `D = 1`, `T = 0`, `E = 0`, `F = 0`. Then `A*B = P−1 ≡ −1`, `C*(D+T) = 1`, sum = 0 mod P.

2. Construct a V3 invoke transaction with `resource_bounds` set to these field elements. Sign the transaction hash (which commits to these values).

3. Submit the transaction. The sequencer includes it in a block.

4. The OS executes `compute_max_possible_fee`, which returns 0.

5. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without transferring any fee token.

6. The transaction's `__execute__` runs normally; state changes are committed; the proof is valid. The sequencer receives zero fee. [6](#0-5) [7](#0-6)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
