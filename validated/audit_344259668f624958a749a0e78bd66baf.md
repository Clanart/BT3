### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` calls `safeTransferFrom` for arbitrary ERC20 tokens without checking the before/after balance. The user-supplied `amount` is emitted in the cross-chain message regardless of how many tokens the bridge actually received. For fee-on-transfer tokens, the destination chain will release or mint more tokens than were ever locked, permanently breaking bridge collateralization.

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract executes a plain `safeTransferFrom` and then immediately uses the caller-supplied `amount` in both the emitted event and the cross-chain payload:

```solidity
// OmniBridge.sol lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← actual received amount may be less
    );
}
```

After the transfer, `amount` (the caller-supplied value, not the actual received value) is forwarded unchanged to `initTransferExtension` and to the `InitTransfer` event:

```solidity
// OmniBridge.sol lines 415-436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
```

In `OmniBridgeWormhole`, `initTransferExtension` publishes a Wormhole message encoding the same unchecked `amount`:

```solidity
// OmniBridgeWormhole.sol lines 136
Borsh.encodeUint128(amount),
```

There is no token whitelist in `OmniBridge`. The `else` branch at line 406 is reachable for any ERC20 that is not registered as a bridge token or custom minter, meaning any user can supply a fee-on-transfer token.

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

If a fee-on-transfer token deducts X% on every transfer:
- Bridge receives `amount * (1 - fee%)` tokens
- Cross-chain message claims `amount` tokens were locked
- Destination chain (NEAR or other) releases/mints `amount` tokens
- Bridge is undercollateralized by `amount * fee%` per transfer

Repeated transfers drain the bridge's real reserves while the destination supply grows unbacked. When legitimate users attempt to bridge back, the bridge will be unable to fulfill withdrawals, permanently freezing or misdirecting user funds.

### Likelihood Explanation

**Medium.** No token whitelist exists; any ERC20 is accepted in the `else` branch. Fee-on-transfer tokens (e.g., USDT on some chains, STA, PAXG, reflection tokens) are common. Any unprivileged user can call `initTransfer` with such a token. No special access or leaked key is required.

### Recommendation

Capture the balance before and after the `safeTransferFrom` and use the difference as the actual transferred amount:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    amount = uint128(balanceAfter - balanceBefore);
}
```

Use this corrected `amount` for all downstream accounting, event emission, and cross-chain message encoding.

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FeeToken` that deducts 10% on every transfer.
2. Call `OmniBridge.initTransfer(feeToken, 1000, 0, 0, "near:recipient", "")`.
3. `safeTransferFrom` moves 1000 tokens from the caller; `FeeToken` deducts 100, so the bridge receives 900.
4. The `InitTransfer` event is emitted with `amount = 1000`.
5. The Wormhole message (or NEAR prover event) encodes `amount = 1000`.
6. NEAR bridge processes the message and releases/mints 1000 tokens to the recipient.
7. Bridge holds 900 tokens but has issued a claim for 1000 — 100 tokens of unbacked supply per call.
8. Repeat to drain the bridge's real reserves while destination supply grows unbacked.
