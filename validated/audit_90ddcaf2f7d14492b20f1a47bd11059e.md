### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Replay of Meta-Transaction v0 - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` omits a nonce from the hash preimage, directly analogous to the reported issue where a signature is not bound to a nonce. Because the OS also skips nonce enforcement for version-0 transactions, any attacker who observes a valid meta-tx v0 can replay it an unlimited number of times by invoking the `meta_tx_v0` syscall from their own contract, causing the victim account to re-execute the same calldata.

---

### Finding Description

**Root cause — `compute_meta_tx_v0_hash`** (`transaction_hash/transaction_hash.cairo`, lines 295–315):

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
```

The hash commits to `(prefix, version=0, contract_address, selector, calldata, max_fee=0, chain_id)` — **no nonce**. Compare with `compute_l1_handler_transaction_hash` (same file, lines 220–237), which correctly passes `additional_data_size=1, additional_data=&nonce`.

**Nonce enforcement is also absent for version 0** (`execute_transaction_utils.cairo`, lines 63–67):

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
```

**`execute_meta_tx_v0`** (`syscall_impls.cairo`, lines 286–399) hard-codes `nonce=0` in the synthesised `TxInfo` and calls `contract_call_helper` directly, bypassing `run_validate` and `check_and_increment_nonce`. The syscall is reachable by any contract via the `META_TX_V0_SELECTOR` dispatch in `execute_syscalls.cairo` (line 343–344).

The combined effect: a signature over a meta-tx v0 hash is valid for **every future replay** of the same `(contract_address, selector, calldata, chain_id)` tuple, because the hash never changes.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An account contract that participates in the meta-tx v0 flow (e.g., a smart-wallet or paymaster) verifies the user's signature against the meta-tx hash inside `__execute__`. Because the hash contains no nonce, a signature that was valid once remains valid forever. An attacker can drain the account by replaying a signed transfer or approval an arbitrary number of times.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is publicly accessible to any deployed contract. An attacker only needs to:
1. Observe a single on-chain meta-tx v0 (calldata and signature are public in the transaction trace).
2. Deploy a one-function contract that calls `meta_tx_v0` with the same `(contract_address, selector, calldata, signature)`.
3. Submit that contract's transaction repeatedly.

No privileged access, leaked key, or social engineering is required.

---

### Recommendation

Include the nonce in the meta-tx v0 hash, mirroring the pattern used for L1-handler transactions:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← commit to nonce
    );
    return tx_hash;
}
```

The OS must then enforce that the nonce in the meta-tx request matches and increments the target account's on-chain nonce, analogous to `check_and_increment_nonce` for versioned transactions.

---

### Proof of Concept

1. Alice's smart-wallet signs a meta-tx v0 authorising a transfer of 1000 STRK to Bob. The signed hash is `H(prefix, 0, alice_addr, __execute__, calldata, 0, chain_id)` — no nonce.
2. A relayer submits an outer transaction whose contract calls `meta_tx_v0(contract_address=alice_addr, selector=__execute__, calldata=transfer_calldata, signature=alice_sig)`.
3. The OS executes `execute_meta_tx_v0` (`syscall_impls.cairo:286`): computes the same hash via `compute_meta_tx_v0_hash` (`transaction_hash.cairo:295`), sets `nonce=0`, and calls Alice's `__execute__` directly.
4. Alice's `__execute__` reads `get_execution_info().tx_info` — the hash matches the signature — and executes the transfer. 1000 STRK leaves Alice's account.
5. An attacker deploys `Replayer` with a single function that calls `meta_tx_v0` with the identical parameters observed from step 2.
6. The attacker submits a transaction invoking `Replayer`. The OS recomputes the identical hash (no nonce changes it), Alice's `__execute__` accepts the same signature, and another 1000 STRK is transferred.
7. The attacker repeats step 6 until Alice's balance is zero.
