### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Replay of Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall path computes a transaction hash that deliberately omits any nonce or uniqueness token. The OS also explicitly skips nonce enforcement and `__validate__` for version-0 transactions. This means any valid `(contract_address, calldata, signature)` tuple can be replayed an unlimited number of times by any unprivileged caller, directly analogous to the zAuction global-random-nonce issue in the external report.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
``` [1](#0-0) 

The resulting hash is a pure function of `(chain_id, contract_address, selector, calldata)`. Two executions with identical parameters produce an identical hash.

`execute_meta_tx_v0` in `syscall_impls.cairo` then constructs the `TxInfo` with `nonce=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // ← always zero
    ...
);
``` [2](#0-1) 

The OS-level nonce guard in `check_and_increment_nonce` explicitly returns early for version-0 transactions:

```cairo
// Do not handle nonce for version 0.
if (tx_info.version == 0) {
    return ();
}
``` [3](#0-2) 

Similarly, `run_validate` skips `__validate__` for version-0 transactions, so the account contract's signature-checking entry point is never invoked by the OS:

```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
``` [4](#0-3) 

The only code that runs against the target contract is `contract_call_helper`, which dispatches directly to `__execute__`: [5](#0-4) 

The target contract's `__execute__` receives a `TxInfo` whose `transaction_hash` is nonce-free. Any account contract that verifies the signature against `tx_info.transaction_hash` (the standard pattern) provides no replay protection, because the hash is identical for every replay.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker who observes a legitimately executed `meta_tx_v0` (all data is public on-chain) can:

1. Deploy a contract that calls the `meta_tx_v0` syscall with the same `(contract_address, calldata, signature)`.
2. The OS recomputes the identical hash (no nonce → same inputs → same hash).
3. The target account contract's `__execute__` is invoked with the same hash and signature, which passes signature verification.
4. If the calldata encodes a token transfer or other value-moving operation, it executes again, draining funds from the target account.

Because the outer transaction's nonce only prevents re-submission of the *outer* transaction, it provides no protection against replaying the inner meta-tx in a *different* outer transaction.

---

### Likelihood Explanation

**High.** The attack requires no privileged access:

- Any unprivileged user can deploy a contract and invoke the `meta_tx_v0` syscall.
- All transaction data (calldata, signature) is publicly visible on-chain after the first execution.
- The hash collision is deterministic, not probabilistic — identical inputs always produce the same hash.
- No brute-force or cryptographic break is needed.

The only limiting factor is that the target account contract must not implement its own independent replay guard inside `__execute__`. Standard account contracts that rely on the OS-provided `tx_info.nonce` (which is always 0 here) or the transaction hash alone for replay protection are fully vulnerable.

---

### Recommendation

Include a per-use nonce in `compute_meta_tx_v0_hash` by passing it as `additional_data`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← include nonce in hash
    );
```

The nonce should be sourced from the target contract's on-chain state (analogous to how `check_and_increment_nonce` works for version ≥ 1 transactions) and incremented after each successful meta-tx execution. Alternatively, adopt the same Poseidon-based structured hash used by v3 transactions (`compute_invoke_transaction_hash`) which already includes the nonce via `hash_tx_common_fields`. [6](#0-5) 

---

### Proof of Concept

1. Alice's account at address `A` executes a meta-tx v0 via some outer contract, with `calldata = [transfer(Bob, 100 tokens)]` and a valid ECDSA signature `sig`.
2. The OS computes `H = hash(INVOKE_PREFIX, 0, A, __execute__, calldata, 0, chain_id)` — no nonce.
3. Alice's `__execute__` verifies `sig` against `H` and transfers 100 tokens to Bob. Nonce on Alice's account is NOT incremented (version 0 path).
4. Attacker deploys contract `Evil` that calls `meta_tx_v0(contract_address=A, calldata=[transfer(Bob, 100 tokens)], signature=sig)`.
5. OS recomputes the identical `H` (same inputs). Alice's `__execute__` is called again with the same `(H, sig)`, passes verification, and transfers another 100 tokens.
6. Step 4–5 can be repeated until Alice's account is drained.
