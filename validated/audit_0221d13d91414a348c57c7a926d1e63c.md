### Title
EIP-7702 Authorization Nonce/Code Committed to Durable State Even When EVM Execution Fails Without Hooks — (`x/evm/keeper/state_transition.go`)

### Summary

In `ApplyTransaction`, the durable EIP-7702 authorization context (`commitDurableAuthorization`) is only created when `k.hooks != nil`. When no hooks are registered, the entire EVM execution runs directly against `ctx` (not a `tmpCtx` cache branch), and the `StateDB` commits authorization effects (nonce bump + delegation code) directly into the live store via `stateDB.Commit()` inside `ApplyMessageWithConfig`. This means that when EVM execution fails (reverts with a `vmErr`), the authorization nonce and delegation code are still committed to the KV store — permanently consuming the authorization — while all other EVM state changes are correctly discarded. This is an invalid state transition: the authorization is consumed without the corresponding EVM execution succeeding, which can be exploited to permanently lock an authority's account into a delegation state or to prevent replay-protection from working correctly.

### Finding Description

In `ApplyTransaction` (`x/evm/keeper/state_transition.go`), the cache-context and durable-authorization machinery is only activated when `k.hooks != nil`:

```go
if k.hooks != nil {
    tmpCtx, commit = ctx.CacheContext()
    if len(msg.SetCodeAuthorizations) > 0 {
        durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()
        cfg.DurableSetCodeAuthorizationCtx = &durableAuthorizationCtx
    }
}
```

When `k.hooks == nil`, `tmpCtx = ctx` (no branching), `commit = nil`, and `cfg.DurableSetCodeAuthorizationCtx = nil`.

Inside `ApplyMessageWithConfig`, the authorization processing path is:

1. `applyAuthorization` is called for each authorization in `msg.SetCodeAuthorizations`. This calls `setAuthorizationDelegation`, which calls `stateDB.SetNonce(authority, auth.Nonce+1, ...)` and `stateDB.SetCode(authority, delegation, ...)`. These writes go into the `stateDB`'s in-memory dirty state.

2. The durable replay block at lines 502–513 only runs when `commit && cfg.DurableSetCodeAuthorizationCtx != nil`. When hooks are absent, `cfg.DurableSetCodeAuthorizationCtx` is `nil`, so this block is skipped.

3. `evm.Call(...)` is then executed. If the call reverts (e.g., the target contract has `INVALID` opcode `0xfe`), `vmErr != nil` is set.

4. After the call, `stateDB.Commit()` is called unconditionally (when `commit == true`). The `StateDB.Commit()` method checks `s.err` (set by failed native balance operations), but **does not check `vmErr`**. The EVM revert only rolls back EVM storage changes tracked by the journal's snapshot/revert mechanism. However, the authorization's nonce and code changes were applied to the `stateDB` **before** `evm.Call()` and **outside** any EVM snapshot scope — they are not reverted by the EVM's internal revert mechanism.

5. Therefore, `stateDB.Commit()` writes the authority's bumped nonce and delegation code to the KV store, even though the EVM call failed.

The key asymmetry: when `k.hooks != nil`, the durable authorization context is a separate sibling cache branch that is committed independently via `commitDurableAuthorization()`. The main `tmpCtx` (which contains the authorization effects from the main `stateDB`) is discarded on EVM failure because `commit()` is never called. The durable branch is then committed separately. But when `k.hooks == nil`, there is no `tmpCtx` isolation — the `stateDB` writes directly to `ctx`, and `stateDB.Commit()` flushes everything including the authorization effects.

The EVM journal's `RevertToSnapshot` only reverts in-memory `stateObject` dirty state. But `SetNonce` and `SetCode` in the authorization path are applied to the `stateDB` **before** `evm.Create/Call` is invoked, and no snapshot is taken before authorization processing. Even if a snapshot were taken, the EVM's revert only undoes journal entries — the `stateDB.Commit()` call at the end still flushes all dirty state to the KV store regardless of `vmErr`.

### Impact Explanation

**High — EIP-7702 authorization nonce/code bypass enabling unauthorized account/code mutation.**

An attacker (or any user) can craft an EIP-7702 `SetCode` transaction targeting a contract that is guaranteed to revert (e.g., a contract with `INVALID` opcode). On a chain deployed without EVM hooks (`k.hooks == nil`):

- The authority's nonce is permanently incremented by 1 (consuming the authorization).
- The authority's code is permanently set to the delegation bytecode pointing to the attacker-chosen delegate address.
- All other EVM state changes (value transfers, storage writes) are correctly rolled back.

This means:
1. **Forced delegation**: An attacker who controls the outer transaction signer can force any authority (whose private key signed the authorization) to have their account permanently delegated to an arbitrary contract, even if the EVM execution fails. The authority cannot undo this because the nonce was bumped, making the original authorization unreplayable.
2. **Nonce manipulation**: The authority's nonce is incremented without a corresponding successful transaction, which can desynchronize nonce-based replay protection.
3. **Permanent code installation**: The authority's EOA now has delegation code installed pointing to an arbitrary contract, which changes how all future calls to that address are executed.

This is reachable by any unprivileged user who can submit a type-4 (`SetCode`) transaction to the network.

### Likelihood Explanation

- EIP-7702 (Prague hardfork) is active on the chain.
- The chain is deployed without EVM post-processing hooks (`k.hooks == nil`). This is the default configuration — hooks are an optional extension.
- The attacker needs a valid signed authorization from the victim authority (which the victim may have provided for a legitimate purpose, or the attacker may have obtained via social engineering).
- The attacker constructs a type-4 transaction targeting a reverting contract with the victim's authorization in the `authorizationList`.
- This is a single transaction, fully within the attacker's control.

### Recommendation

The authorization effects (nonce bump + code) must be isolated from the main `stateDB` commit path in the same way as when hooks are present. Specifically:

1. **Always** create a `tmpCtx` cache branch and a `durableAuthorizationCtx` sibling branch when `len(msg.SetCodeAuthorizations) > 0`, regardless of whether `k.hooks != nil`.
2. Apply authorization effects only to the `durableStateDB` (backed by `durableAuthorizationCtx`), not to the main `stateDB`. The main `stateDB` should not see authorization nonce/code changes at all — only the durable branch should.
3. Commit `durableAuthorizationCtx` unconditionally after `ApplyMessageWithConfig` returns (as long as no cosmos-level error occurred), mirroring the existing `commitDurableAuthorization()` call at line 266–268.

Alternatively, apply authorization effects inside an EVM snapshot scope so that a `vmErr` triggers a `RevertToSnapshot` that undoes them before `stateDB.Commit()` is called — but this requires careful ordering since the EVM's own snapshot/revert is managed by go-ethereum internals.

### Proof of Concept

```
Setup:
- Chain with Prague hardfork active, k.hooks == nil (default, no hooks registered)
- authority: EOA with nonce=0, controlled by victim
- delegate: attacker-controlled contract address
- failTarget: contract with bytecode 0xfe (INVALID opcode)

Step 1: Victim signs EIP-7702 authorization:
  auth = SignSetCode(victimKey, {ChainID: chainID, Address: delegate, Nonce: 0})

Step 2: Attacker submits type-4 transaction:
  tx = {
    type: 4 (SetCode),
    to: failTarget,       // guaranteed to revert
    authorizationList: [auth],
    nonce: attackerNonce,
    gas: 100000,
  }

Step 3: ApplyTransaction executes:
  - k.hooks == nil → tmpCtx = ctx, commit = nil, DurableSetCodeAuthorizationCtx = nil
  - applyAuthorization(auth, stateDB) succeeds:
      stateDB.SetNonce(authority, 1)
      stateDB.SetCode(authority, delegation(delegate))
  - evm.Call(attacker, failTarget, ...) → vmErr = ErrExecutionReverted
  - stateDB.Commit() is called:
      - s.err == nil (no native action errors)
      - Flushes authority nonce=1 and code=delegation(delegate) to KV store
      - EVM storage reverts are irrelevant (no storage was written)

Result:
  - authority.nonce = 1 (was 0)
  - authority.code = 0xef0100<delegate_address>
  - All other EVM state: unchanged (correctly reverted)
  - The authorization is permanently consumed; the victim's EOA is now delegated
    to the attacker's contract without any successful EVM execution.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
