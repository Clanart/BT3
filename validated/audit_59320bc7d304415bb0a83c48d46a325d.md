### Title
`removeCustomToken` Deletes EVM Token Mappings Without Draining In-Flight Transfers, Permanently Freezing User Funds on NEAR - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.removeCustomToken()` immediately wipes all token-dispatch mappings (`isBridgeToken`, `customMinters`, `ethToNearToken`, `nearToEthToken`) with no check for in-flight transfers. Any NEAR→EVM transfer whose tokens were already burned/locked on NEAR will have its `finTransfer` call on EVM permanently revert after removal, while the NEAR-side `pending_transfers` entry has no cancellation path. The user's funds are irrecoverably frozen.

---

### Finding Description

**Root cause — EVM side (`OmniBridge.sol`):**

`removeCustomToken` deletes all four mappings in one shot:

```solidity
function removeCustomToken(address tokenAddress) external onlyRole(DEFAULT_ADMIN_ROLE) {
    delete isBridgeToken[tokenAddress];
    delete nearToEthToken[ethToNearToken[tokenAddress]];
    delete ethToNearToken[tokenAddress];
    delete customMinters[tokenAddress];
}
``` [1](#0-0) 

There is no guard that checks whether any `pending_transfers` on NEAR are still awaiting finalization for this token.

**Root cause — EVM dispatch in `finTransfer`:**

`finTransfer` dispatches token delivery through a priority chain:

```solidity
} else if (customMinters[payload.tokenAddress] != address(0)) {
    ICustomMinter(customMinters[payload.tokenAddress]).mint(...);
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(...);
} else {
    IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
}
``` [2](#0-1) 

After `removeCustomToken`, both `customMinters[tokenAddress]` and `isBridgeToken[tokenAddress]` are zero/false. The call falls through to `IERC20.safeTransfer`. For custom tokens (which are minted on demand, not held in the bridge's balance), this call reverts — permanently.

**Root cause — NEAR side (`near/omni-bridge/src/lib.rs`):**

The NEAR bridge stores in-flight transfers in `pending_transfers`:

```rust
pub pending_transfers: LookupMap<TransferId, TransferMessageStorage>,
``` [3](#0-2) 

`pending_transfers` entries are only removed by `remove_transfer_message` (called inside `claim_fee_callback`) or `remove_transfer_message_without_refund` (called on storage failure in `init_transfer_internal`): [4](#0-3) 

There is **no user-callable cancel or refund function** for a pending transfer. A grep for `cancel`, `refund_transfer`, `revert_transfer`, or `withdraw_pending` returns zero matches in the NEAR bridge source. Once `finTransfer` on EVM permanently reverts, the NEAR entry is stuck forever.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For a NEAR→EVM transfer of a custom token:
1. User's tokens are burned on NEAR (via `burn_tokens_if_needed`) or locked, and a `pending_transfer` is written.
2. Admin calls `removeCustomToken` on EVM (legitimate deprecation/migration action).
3. Relayer submits `finTransfer` on EVM → reverts because `customMinters` and `isBridgeToken` are both cleared and the bridge holds no token balance.
4. The NEAR `pending_transfer` can never be removed — no cancel path exists.
5. User's tokens are permanently lost.

This matches the allowed impact: **"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

Token removal is a routine operational action (deprecation, migration to a new contract, security incident response). The window between a user submitting a NEAR→EVM transfer and the relayer finalizing it on EVM is non-zero (block confirmation latency, relayer queue). Any removal during that window silently destroys the in-flight transfer. The admin need not act maliciously — a well-intentioned removal is sufficient to trigger the loss.

---

### Recommendation

Before allowing `removeCustomToken` to clear mappings, the contract should either:

1. **Require zero in-flight transfers**: Maintain a per-token in-flight counter (incremented on `initTransfer`, decremented on `finTransfer`) and revert `removeCustomToken` if the counter is non-zero.
2. **Introduce a "claim-only" mode** (analogous to the original report's recommendation): mark the token as removed so no new `initTransfer` is accepted, but preserve `isBridgeToken`/`customMinters` so existing `finTransfer` calls can still complete.
3. **Add a NEAR-side cancel path**: Allow users to cancel a `pending_transfer` after a timeout, returning their tokens, so that even if EVM finalization is permanently broken the user can recover funds.

---

### Proof of Concept

1. Admin calls `addCustomToken("token.near", tokenAddr, customMinterAddr, 18)` on EVM OmniBridge.
2. User on NEAR calls `ft_on_transfer` → `init_transfer_internal` → tokens burned, `pending_transfers[transferId]` written. [5](#0-4) 
3. Admin calls `removeCustomToken(tokenAddr)` on EVM — clears `isBridgeToken`, `customMinters`, `ethToNearToken`, `nearToEthToken`. [1](#0-0) 
4. Relayer calls `finTransfer` on EVM with the signed payload for `tokenAddr`:
   - `customMinters[tokenAddr]` == `address(0)` → skip
   - `isBridgeToken[tokenAddr]` == `false` → skip
   - Falls to `IERC20(tokenAddr).safeTransfer(recipient, amount)` → **reverts** (bridge holds 0 balance) [6](#0-5) 
5. No retry will ever succeed. The NEAR `pending_transfers` entry for this transfer has no cancellation path. [4](#0-3) 
6. User's tokens are permanently frozen.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        U128(0)

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
