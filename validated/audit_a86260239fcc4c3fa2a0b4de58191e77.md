### Title
`batch_insert` Bypasses Duplicate-Key Validation, Enabling DataLayer Merkle Tree Root Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` silently omits the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When the tree already contains two or more leaves, every key in a batch is written directly to the blob without any existence check. Supplying a key that is already present — or supplying the same key twice within one batch — silently produces a structurally invalid tree whose committed root hash is wrong, breaking all subsequent inclusion/exclusion proofs.

### Finding Description

`MerkleBlob::insert` (the single-item path) performs two explicit pre-checks before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a completely different code path for the bulk portion of the batch:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else {
            return Ok(());
        };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← guarded
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← remaining items
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?; // ← NO duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Two concrete bypass scenarios arise:

1. **Tree already has ≥ 2 leaves** — the `if leaf_count <= 1` branch is skipped entirely; every item in the batch goes straight to `insert_entry_to_blob` with no key or hash check.
2. **Tree has 0–1 leaves, batch has ≥ 3 items** — `pop()` removes the last two items and inserts them via the guarded `insert`; all earlier items in the vector are then processed by the unguarded loop.

In both cases a caller can supply a `KeyId` that already exists in the tree (or repeat the same `KeyId` twice within one batch) and the function returns `Ok(())` while writing a second leaf node for that key into the blob. The `block_status_cache` is not updated for bulk-path items before the subtree is attached, so the cache remains inconsistent with the blob. The root hash subsequently computed over this malformed tree is incorrect.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust

    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,

    …
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}

### Impact Explanation

The DataLayer commits a Merkle root hash on-chain. If `batch_insert` is called with a duplicate key, the tree contains two leaf nodes for the same key. The root hash computed from this malformed tree is wrong. Any proof of inclusion or exclusion generated from the corrupted blob will either fail to verify against the on-chain root or, worse, verify against a root that does not reflect the true data set. This directly matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

**Medium.** The Python binding `py_batch_insert` is part of the public `chia_rs` wheel API consumed by `chia-blockchain`. Any code path that feeds externally-sourced key/value pairs into `batch_insert` without a prior deduplication step is vulnerable. Because the single-item `insert` correctly rejects duplicates, developers may not realise `batch_insert` does not share that guarantee, making accidental or adversarially-induced duplicate injection plausible.

### Recommendation

Add the same pre-checks to the bulk loop in `batch_insert` that `insert` already performs:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing blob-write logic …
}
```

Alternatively, deduplicate the input vector before processing and return an error if any duplicate is detected.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId

# Start with a tree that already has two leaves so leaf_count > 1
blob = MerkleBlob(bytearray())
blob.insert(KeyId(10), ValueId(10), bytes([0xAA]*32))
blob.insert(KeyId(20), ValueId(20), bytes([0xBB]*32))

# Now batch_insert with a key that already exists (KeyId(10))
# The guarded branch is skipped; insert_entry_to_blob is called with no check
blob.batch_insert(
    [(KeyId(10), ValueId(99)), (KeyId(30), ValueId(30))],
    [bytes([0xCC]*32), bytes([0xDD]*32)],
)
# Returns Ok — no error raised.
# Tree now contains two leaf nodes for KeyId(10).
# calculate_lazy_hashes() will produce a root that does not match
# the root of a correctly-built tree containing {10, 20, 30}.
blob.calculate_lazy_hashes()
# Proofs generated from this blob are invalid.

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
