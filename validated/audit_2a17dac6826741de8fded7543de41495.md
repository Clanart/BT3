### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Contract Fund Freeze - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler updates a contract's class hash in `contract_state_changes` without verifying that the new class hash has been declared (i.e., exists in `contract_class_changes`). This is the direct analog of the stale-oracle pattern: just as the TWAP oracle is read without first calling `_update()` on the underlying pool, here a class hash is written without first validating it against the underlying class declaration registry. The result is that any contract can permanently freeze its own funds — and any funds of users who interact with it — by replacing its class with an arbitrary undeclared hash.

---

### Finding Description

In `execute_replace_class` the function accepts any arbitrary `class_hash` from the syscall request and writes it directly into `contract_state_changes` with no check that the hash exists in `contract_class_changes`:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developers themselves flag this with a `TODO` at line 898, confirming the validation is intentionally deferred but not yet implemented.

When `execute_entry_point` is subsequently called for a contract whose class hash was replaced with an undeclared value `X`, the following sequence occurs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // = X, undeclared
);
// compiled_class_hash == 0 (default for uninitialized dict key)

let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,   // key == 0
);
``` [2](#0-1) 

`dict_read` on an uninitialized key returns `0`. `find_element` with `key=0` panics if no compiled class with hash `0` exists in the bundle — which is always the case for a legitimately operating sequencer. Because `find_element` panics (not gracefully reverts), the OS proof itself fails for any block that includes a call to the affected contract. The sequencer is therefore forced to permanently exclude all calls to that contract from every future block.

---

### Impact Explanation

**Permanent freezing of funds.** Once a contract's class hash is set to an undeclared value, no call to that contract can ever be included in a valid block again. Any ERC-20 tokens, ETH, or other assets held by the contract become permanently inaccessible. This satisfies the "Critical — Permanent freezing of funds" impact category.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, permissionless syscall callable by any contract during its own execution. The attacker-controlled entry path is:

1. Deploy or control a contract that holds user funds.
2. Invoke a function on that contract that calls `replace_class(X)` where `X` is any felt value not present in `contract_class_changes`.
3. The OS accepts the syscall with no validation.
4. The contract's class hash is permanently set to `X`.
5. All future calls to the contract fail at the OS level; funds are frozen forever.

This is reachable by any unprivileged transaction sender who can call a contract they control. It also affects shared contracts (multisigs, vaults) where one party can trigger `replace_class`.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, assert that the hash exists in `contract_class_changes`:

```cairo
// Verify the new class hash has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed implicitly in `execute_entry_point` — making it explicit and enforced at the point of replacement rather than at the point of execution.

---

### Proof of Concept

1. Deploy `VaultContract` holding 1000 STRK.
2. `VaultContract.__execute__` calls `replace_class(0xDEADBEEF)` — an arbitrary undeclared felt.
3. `execute_replace_class` writes `class_hash=0xDEADBEEF` into `contract_state_changes` with no validation. [1](#0-0) 
4. Block N is finalized. `VaultContract`'s on-chain class hash is now `0xDEADBEEF`.
5. In block N+1, any user attempts to call `VaultContract.withdraw()`.
6. The OS calls `execute_entry_point` with `execution_context.class_hash = 0xDEADBEEF`. [3](#0-2) 
7. `dict_read(contract_class_changes, 0xDEADBEEF)` returns `0` (never declared).
8. `find_element(compiled_class_facts, key=0)` panics — no compiled class with hash `0` exists. [4](#0-3) 
9. The sequencer cannot include any call to `VaultContract` in any future block.
10. The 1000 STRK are permanently frozen.
