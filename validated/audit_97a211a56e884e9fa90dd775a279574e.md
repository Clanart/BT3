### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` for shard Y but never verifies that the embedded `ShardChunk` actually belongs to shard Y. All five validation steps operate on the chunk's own internal consistency and its position in the block's multi-shard merkle tree — none of them bind the chunk to the requested `shard_id`. A malicious peer can therefore substitute shard X's chunk (with its own valid proofs) into a response for shard Y, causing the node to persist shard X's `prev_state_root` under `StateHeaderKey(Y, sync_hash)`. Every subsequent `set_state_part` and `apply_state_part` call for shard Y then uses shard X's state root, installing shard X's trie data into shard Y's `shard_uid` prefix.

### Finding Description

**Entrypoint and call path:**

An unprivileged peer responds to a `StateRequestHeader` network message. The response travels through `StateSyncDownloadSourcePeer::download_shard_header` → `StateSyncDownloader::ensure_shard_header` → `StateHeaderValidationRequest` → `ClientActor` handler → `ChainStateSyncAdapter::set_state_header`. [1](#0-0) [2](#0-1) 

**The five validation steps and why none catches the shard mismatch:**

| Step | Check | Why it passes for shard X's chunk |
|------|-------|-----------------------------------|
| 1–2 | `validate_chunk_proofs` | Checks internal hash/tx/receipt consistency of the chunk itself; no `shard_id` field compared | 
| 3a | `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(...))` | The block's `chunk_headers_root` is a merkle tree over **all** shards; a valid proof for shard X's position passes |
| 3b | `verify_path(prev_block.chunk_headers_root, prev_chunk_proof, ...)` | Same multi-shard merkle tree; shard X's prev chunk proof is valid |
| 4 | Receipt chain validation | Uses `shard_id` (Y) to hash receipts, but the attacker supplies valid on-chain receipt proofs for shard Y for the height range dictated by shard X's `prev_chunk_header.height_included()` |
| 5 | `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` | Validates shard X's state root node against shard X's `prev_state_root` — trivially passes | [3](#0-2) [4](#0-3) 

**The missing guard:** There is no line of the form `chunk.shard_id() == shard_id` anywhere in `set_state_header`.

**Corrupt write:**

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();  // key = (Y, sync_hash)
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);    // value contains shard X's chunk
``` [5](#0-4) 

**Downstream propagation in `set_state_part`:**

```rust

let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();   // shard X's state root
// validate_state_part called with shard Y's shard_id but shard X's state_root
``` [6](#0-5) 

**Downstream propagation in `apply_state_part`:**

```rust

tries.apply_all(&trie_changes, shard_uid, &mut store_update);            // shard X's trie data
flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
``` [7](#0-6) 

### Impact Explanation

The exact corrupted value is: `prev_state_root` stored under `StateHeaderKey(Y, sync_hash)` is shard X's `prev_state_root`, not shard Y's. All state parts for shard Y are validated against shard X's state root and accepted. `apply_state_part` writes shard X's trie nodes and flat-state entries into shard Y's `shard_uid` prefix. After `set_state_finalize`, the node operates with shard X's state installed as shard Y's state — account balances, contract storage, and all key-value data for shard Y are replaced with shard X's data.

### Likelihood Explanation

The attack requires only that the malicious node be selected as the state-sync provider for the syncing node. State sync providers are chosen from the peer set without authentication of the response content beyond the checks analyzed above. The attacker needs read access to the canonical chain (to construct valid receipt proofs for shard Y) and the ability to serve a modified `ShardStateSyncResponseHeader`. No validator or privileged role is required.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [8](#0-7) 

### Proof of Concept

Build a `TestEnv` with two shards. Call `get_state_response_header(shard_0, sync_hash)` on client[0] to obtain a legitimate header for shard 0. Submit it to client[1] via `set_state_header(shard_1, sync_hash, shard_0_header)`. Assert that:

1. `set_state_header` returns `Ok(())` (no error).
2. `get_state_header(shard_1, sync_hash)` on client[1] deserializes to a header whose `chunk.shard_id() == 0`.
3. `header.chunk_prev_state_root() != shard_1_actual_prev_state_root`.

This directly demonstrates that `StateHeaderKey(1, sync_hash)` stores shard 0's `prev_state_root`, confirming the commitment invariant is violated without any privileged access.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
