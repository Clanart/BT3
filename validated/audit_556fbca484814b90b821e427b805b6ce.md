### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Installation — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the embedded chunk's own shard identity matches the `shard_id` argument. An unprivileged peer can craft a header whose `chunk` belongs to shard 0 but whose `incoming_receipts_proofs` are valid for shard 1, pass every validation gate in the function, and cause the victim to install shard 0's trie state into shard 1's storage.

---

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` performs five validation steps before persisting the header:

**Step 1-2 — `validate_chunk_proofs`** [1](#0-0) 

This function verifies the chunk's internal hash, transaction Merkle root, and outgoing-receipts root. It takes only `(chunk, epoch_manager)` — there is no `shard_id` parameter and no assertion that `chunk.shard_id()` equals anything external.

**Step 3a — Merkle-path check** [2](#0-1) 

`verify_path` proves that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in `sync_prev_block_header.chunk_headers_root()`. The proof encodes a path (left/right directions) from the leaf to the root; it does **not** encode or verify the leaf's ordinal position in the tree. A proof for shard 0's chunk at index 0 verifies correctly against the same root even when the caller claims `shard_id = 1`.

**Step 4 — Receipt-proof check** [3](#0-2) 

This is the only step that uses the `shard_id` argument. It hashes `ReceiptList(shard_id, receipts)` and verifies the result against the supplied proof. An attacker who populates `incoming_receipts_proofs` with the **actual** on-chain incoming receipts for shard 1 (public data) satisfies this check while the `chunk` field still belongs to shard 0.

**Step 5 — State-root-node check** [4](#0-3) 

Validates `state_root_node` against `chunk_inner.prev_state_root()` — the chunk's own state root. No `shard_id` binding.

**Storage** [5](#0-4) 

The header is persisted under `StateHeaderKey(shard_id, sync_hash)` where `shard_id` is the caller's argument, not `chunk.shard_id()`. The chunk embedded in the stored header still carries shard 0's `prev_state_root`.

**Downstream propagation — `set_state_part`** [6](#0-5) 

Reads the stored header back via `get_state_header(shard_id=1, sync_hash)`, extracts `state_root` from the chunk (shard 0's root), validates the supplied part against that root, and stores the part under `StatePartKey(sync_hash, shard_id=1, part_id)`.

**Downstream propagation — `apply_state_part`** [7](#0-6) 

Derives `shard_uid = get_shard_uid_from_epoch_id(shard_id=1, epoch_id)` and calls `tries.apply_all(&trie_changes, shard_uid=1, ...)` and `flat_state_delta.apply_to_flat_state(..., shard_uid=1)`. Shard 0's trie nodes and flat-state delta are written into shard 1's column-space.

---

### Impact Explanation

A victim node completing state sync for shard 1 ends up with shard 0's trie committed to `DBCol::State` and `DBCol::FlatState` under shard 1's `ShardUId`. When the node subsequently tries to apply blocks, the state root in shard 1's chunk header will not match the installed trie, causing every block application for shard 1 to fail. The node cannot recover without a full re-sync.

---

### Likelihood Explanation

The attacker must be a peer from which the victim downloads state headers — achievable by any unprivileged node operator. All data needed to craft the attack (shard 0's chunk, its Merkle proof, and shard 1's incoming-receipt proofs) is public on-chain data. No validator key, chunk-producer key, or privileged storage-service access is required.

---

### Recommendation

Add an explicit shard-identity guard immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// Guard: the chunk must belong to the shard being synced.
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This single check closes the gap before any other validation runs. [8](#0-7)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)

    pub fn set_state_part(

        sync_hash: CryptoHash,

    fn apply_state_part(
        &self,
        shard_id: ShardId,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,
    ) -> Result<(), Error> {
        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);
        let tries = self.get_tries();
        let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;
        let mut store_update = tries.store_update();
        tries.apply_all(&trie_changes, shard_uid, &mut store_update);
        tracing::debug!(target: "chain", %shard_id, values_count = %flat_state_delta.len(), "inserting values to flat storage");
        // TODO: `apply_to_flat_state` inserts values with random writes, which can be time consuming.
        //       Optimize taking into account that flat state values always correspond to a consecutive range of keys.
        flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
        self.precompile_contracts(epoch_id, contract_codes)?;
        store_update.commit();
        Ok(())
