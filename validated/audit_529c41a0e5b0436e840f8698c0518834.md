### Title
`charge_fee` Does Not Verify the Fee Token `transfer` Return Value — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary

The `charge_fee` function in the StarkNet OS calls the fee token's `transfer` entry point via `non_reverting_select_execute_entry_point_func` but **completely discards the return value**. The ERC20 `transfer` function returns a boolean indicating success or failure. If the fee token returns `false` (transfer failed) instead of reverting, the OS silently proceeds as if the fee was collected, allowing transactions to execute without paying fees.

### Finding Description

In `charge_fee` (`transaction_impls.cairo`, lines 160–164), the fee token transfer is executed as:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
```

`non_reverting_select_execute_entry_point_func` returns `(retdata_size: felt, retdata: felt*, is_deprecated: felt)`. The ERC20 `transfer` entry point returns a `bool` (success flag) in `retdata[0]`. This return value is **never captured or checked** in `charge_fee`.

Contrast this with how `run_validate` and `execute_deploy_account_transaction` handle their entry point return values — they explicitly assert the expected return:

```cairo
// run_validate (execute_transaction_utils.cairo, lines 149–156)
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(...);
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = VALIDATED;
}
```

No equivalent check exists for the fee transfer result. `non_reverting_select_execute_entry_point_func` only asserts `is_reverted = 0` (line 195 of `execute_transaction_utils.cairo`), meaning it catches execution reverts but **not a `false` return value from a non-reverting transfer**.

### Impact Explanation

**Critical — Direct loss of funds.**

If the fee token's `transfer` implementation returns `false` on failure (e.g., insufficient balance) rather than reverting — which is valid ERC20 behavior — the OS will:
1. Execute the user's transaction in full.
2. Call `transfer` on the fee token.
3. Receive `false` in `retdata[0]` (transfer failed, no tokens moved).
4. Ignore the result and finalize the block as if the fee was paid.

The sequencer receives no fee. The user's transaction is executed for free. This constitutes a direct, permanent loss of fee revenue and breaks the economic security assumption of the protocol.

### Likelihood Explanation

**Medium.** The current canonical StarkNet fee tokens (STRK, ETH) revert on insufficient balance rather than returning `false`. However:
- The OS code makes no assumption enforcement about this — it is a latent protocol-level bug.
- If the fee token contract is ever upgraded, replaced, or a new fee token is registered that follows the ERC20 standard strictly (returning `false` on failure), the bug becomes immediately exploitable by any transaction sender with insufficient balance.
- Any unprivileged user can trigger this path simply by submitting a transaction with a non-zero resource bounds fee.

### Recommendation

Capture and validate the return value of the fee token transfer in `charge_fee`, analogous to how `run_validate` checks `VALIDATED`:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = 1;  // ERC20 transfer must return true.
}
```

### Proof of Concept

1. Deploy a fee token contract whose `transfer` entry point returns `(false,)` instead of reverting when the caller has insufficient balance.
2. Register this contract as the `fee_token_address` in `starknet_os_config`.
3. Submit any V3 invoke transaction with non-zero resource bounds from an account with zero fee token balance.
4. The OS executes `charge_fee`:
   - `compute_max_possible_fee` returns non-zero → fee path is entered.
   - `assert_nn_le(calldata.amount.low, max_fee)` passes (actual fee ≤ max fee).
   - `non_reverting_select_execute_entry_point_func` calls `transfer`, which returns `false`.
   - `is_reverted = 0` (no revert), so `non_reverting_select_execute_entry_point_func` returns normally.
   - The return value `(retdata_size=1, retdata=[0], is_deprecated=0)` is discarded.
   - `charge_fee` returns without error.
5. The block is finalized. The transaction was executed; no fee was transferred. The sequencer suffers a direct loss of funds.
