Audit Report

## Title
Unguarded `refundETH`, `sweepToken`, and `unwrapWETH9` Allow Any Caller to Drain Router ETH/Token Balances — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH`, `sweepToken`, and `unwrapWETH9` carry no `msg.sender` authorization check. When a user calls `exactInputSingle` (or `exactOutputSingle`) with `tokenIn == WETH` and `msg.value` exceeding the pool-consumed amount, the surplus ETH is left on the router after the swap returns. Any third party can immediately call `refundETH()` and receive that ETH, causing a direct loss of user principal with no recovery path.

## Finding Description
`PeripheryPayments` exposes three public balance-draining helpers with no caller guard:

- `refundETH()` (L58–63): sends `address(this).balance` to `msg.sender` unconditionally.
- `sweepToken()` (L48–55): sends the full ERC-20 balance to a caller-supplied `recipient`.
- `unwrapWETH9()` (L37–45): unwraps the full WETH balance and sends it to a caller-supplied `recipient`.

The `pay` internal function (L69–88), when `token == WETH`, wraps **only the pool-requested `value`** of native ETH and forwards it; any surplus `msg.value` is left as raw ETH on the contract:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // wraps only `value`, not all ETH
    IERC20(WETH).safeTransfer(recipient, value);
}
```

`exactInputSingle` (L67–86) is `external payable` and does not enforce `msg.value == params.amountIn`. It sets the payer to `msg.sender` and calls the pool with the specified `amountIn`; the pool callback fires `pay`, which wraps exactly the pool-requested amount. If `msg.value > amountIn` (or if a `priceLimitX64` causes a partial fill consuming less than `amountIn`), the difference remains on the router after `exactInputSingle` returns. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers — it does not prevent ETH accumulation via `msg.value` in payable function calls.

Because `refundETH()` sends `address(this).balance` to `msg.sender` with no check that the caller is the original depositor, any third party who calls it first receives the stranded ETH. Similarly, `sweepToken` and `unwrapWETH9` accept a caller-supplied `recipient` with no authorization, so any ERC-20 or WETH balance on the router can be redirected to an attacker.

The `multicall` function (L39–44) uses `delegatecall` and is the intended atomic batching mechanism, but there is no enforcement that users must route through `multicall`; `exactInputSingle` is callable directly as a standalone `external` function.

## Impact Explanation
Direct loss of user ETH principal. A user who calls `exactInputSingle` with `msg.value` exceeding the actual pool consumption loses the surplus ETH to the first caller of `refundETH()`. This is a concrete, irreversible loss of user funds meeting the Critical/High direct-loss-of-principal threshold. The same applies to any ERC-20 balance left on the router (e.g., from a failed or partial multi-hop path) via `sweepToken`.

## Likelihood Explanation
- `exactInputSingle` and `exactOutputSingle` are `external payable`; users routinely over-send ETH when the exact consumption depends on pool state or when a price limit causes partial fill.
- The router is a shared singleton; any ETH left between a user's swap call and their `refundETH` call is visible on-chain and trivially front-runnable by MEV bots.
- No special role or permission is required; the attack is permissionless and requires only a single external call.
- The attack is repeatable against every user who calls the router directly without multicall batching.

## Recommendation
Restrict `refundETH`, `sweepToken`, and `unwrapWETH9` so that only the address that initiated the current multicall can invoke them (e.g., require `msg.sender == address(this)` enforced via a `onlyViaMulticall` modifier using a transient storage flag set at multicall entry), or record the depositor in transient storage at swap entry and validate it in each cleanup function. Alternatively, enforce that `msg.value` is fully consumed within the same atomic call by reverting if `address(this).balance > 0` after the swap completes.

## Proof of Concept
1. User calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18, priceLimitX64: <non-zero limit>, ...})` directly (not via multicall) with `msg.value = 2e18`.
2. Pool callback fires; `pay(WETH, msg.sender, pool, 1e18)` wraps exactly `1e18` ETH into WETH and sends it to the pool. The remaining `1e18` ETH stays at `address(router)`.
3. `exactInputSingle` returns successfully (output meets `amountOutMinimum`).
4. Attacker observes the confirmed transaction, calls `router.refundETH()`.
5. `refundETH` executes: `address(this).balance == 1e18 > 0`, so `_transferETH(msg.sender /*attacker*/, 1e18)` runs.
6. Attacker receives 1 ETH; user's subsequent `refundETH` call finds zero balance and returns nothing.

Foundry test sketch:
```solidity
function test_frontrun_refundETH() public {
    // user swaps with excess ETH
    vm.prank(user);
    router.exactInputSingle{value: 2e18}(params); // amountIn = 1e18
    // attacker front-runs refund
    vm.prank(attacker);
    router.refundETH();
    assertEq(attacker.balance, 1e18);
    assertEq(address(router).balance, 0);
}
```
