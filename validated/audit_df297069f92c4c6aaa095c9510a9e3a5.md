### Title
`uint32 wormholeNonce` Overflow Permanently Freezes All Bridge Operations - (File: `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole` stores `wormholeNonce` as `uint32`. In Solidity ≥ 0.8, arithmetic overflow reverts by default. Once `wormholeNonce` reaches `type(uint32).max` (4,294,967,295), every subsequent call to `initTransfer`, `finTransfer`, `deployToken`, or `logMetadata` will revert permanently, freezing the entire Wormhole bridge path and locking any funds already held by the contract.

---

### Finding Description

`OmniBridgeWormhole.sol` declares:

```solidity
uint32 public wormholeNonce;
``` [1](#0-0) 

This counter is unconditionally incremented with `wormholeNonce++` inside four internal extension hooks:

- `deployTokenExtension` — line 69
- `logMetadataExtension` — line 93
- `finTransferExtension` — line 115
- `initTransferExtension` — line 149 [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

Every public bridge action — `initTransfer`, `initTransfer1155`, `finTransfer`, `deployToken`, `logMetadata`, `logMetadata1155` — calls one of these hooks, so every bridge action increments `wormholeNonce`. Because Solidity 0.8+ uses checked arithmetic by default, the expression `wormholeNonce++` will revert with an overflow panic once the counter reaches `4,294,967,295`. There is no `unchecked` block, no reset mechanism, and no admin escape hatch.

The Wormhole `publishMessage` interface itself accepts `uint32 nonce` purely as a grouping hint; the actual uniqueness guarantee comes from the sequence number returned by `publishMessage`. Wrapping the counter would be semantically safe for Wormhole, but the Solidity overflow check prevents it. [6](#0-5) 

---

### Impact Explanation

Once `wormholeNonce == type(uint32).max`, every call to `initTransfer` or `finTransfer` reverts before any token movement or state change can complete. Concretely:

- **Funds already locked** in the bridge from pending `initTransfer` operations (ERC-20 tokens transferred to the contract, or native ETH held) cannot be released because `finTransfer` also reverts.
- **No new transfers** can be initiated or finalized.
- The bridge is permanently bricked on the Wormhole path with no on-chain recovery path, matching the allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.* [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

`uint32` overflows at ~4.3 billion increments. At sustained high throughput (e.g., 1,000 tx/s across all four hooks), the ceiling is reached in roughly 50 days. At more typical bridge volumes (10–100 tx/s), the window is years. The likelihood is low but non-zero for a long-lived, high-volume deployment. Critically, the overflow is **deterministic and irreversible** once reached — no operator action can unblock it without a contract upgrade.

---

### Recommendation

Replace the bare `wormholeNonce++` increments with an `unchecked` block to allow safe wrapping (the Wormhole nonce is not a uniqueness guarantee):

```solidity
unchecked { wormholeNonce++; }
```

Alternatively, widen the storage variable to `uint256` and cast to `uint32` only at the call site:

```solidity
uint256 public wormholeNonce;
// ...
_wormhole.publishMessage{value: ...}(uint32(wormholeNonce), payload, _consistencyLevel);
unchecked { wormholeNonce++; }
```

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` on a test network.
2. Call `initTransfer` (or any bridge action) `type(uint32).max` times, incrementing `wormholeNonce` to `4,294,967,295`.
3. Call `initTransfer` one more time. The transaction reverts with a Solidity overflow panic inside `wormholeNonce++`.
4. All subsequent `initTransfer`, `finTransfer`, `deployToken`, and `logMetadata` calls revert identically.
5. Any ERC-20 tokens previously transferred to the bridge contract in step 2 are permanently locked with no release path. [9](#0-8)
