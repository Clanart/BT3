### Title
Unconstrained `is_reverted` Hint Allows Sequencer to Skip `__execute__` While Charging Fees, Breaking Validate-Execute Atomicity - (File: execution/execution_constraints.cairo)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is completely empty — it imposes zero constraints on the `is_reverted` hint loaded in `execute_invoke_function_transaction`. A malicious sequencer can freely set `is_reverted = 1` for any invoke transaction, causing the `__execute__` entry point to be skipped while the nonce is still incremented and fees are still charged. This breaks the atomicity of the validate → execute → charge_fee sequence and allows a provably-valid but semantically-invalid state transition to be committed to L1.

---

### Finding Description

In `transaction_impls.cairo`, `execute_invoke_function_transaction` loads `is_reverted` from a sequencer-supplied hint and calls `check_is_reverted` to validate it before deciding whether to run `__execute__`:

```cairo
// transaction_impls.cairo lines 338-358
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    tempvar range_check_ptr = range_check_ptr;
    tempvar remaining_gas = remaining_gas;
    tempvar builtin_ptrs = builtin_ptrs;
    tempvar contract_state_changes = contract_state_changes;
    tempvar contract_class_changes = contract_class_changes;
    tempvar outputs = outputs;
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}

// Charge fee — always runs regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The validation function `check_is_reverted` is defined in `execution_constraints.cairo` as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This function is **completely empty**. It adds no Cairo constraint whatsoever on the value of `is_reverted`. In Cairo's constraint model, a `local` variable loaded from a hint (`%{ IsReverted %}`) is only constrained if the surrounding code asserts something about it. Since `check_is_reverted` returns immediately, `is_reverted` is a free variable — the prover (sequencer) can assign it any field element and still produce a valid proof.

The consequence is a broken atomicity guarantee. The three operations that must be atomic for a valid invoke transaction are:

1. **Nonce increment** — `check_and_increment_nonce` runs unconditionally before the `is_reverted` check.
2. **`__execute__` entry point** — runs only if `is_reverted == FALSE`.
3. **Fee charge** — `charge_fee` runs unconditionally after the `is_reverted` branch. [3](#0-2) 

With `is_reverted = 1` (freely set by the sequencer), the nonce is incremented and fees are deducted, but `__execute__` is never called. No user-intended state changes are applied. The resulting state root is provably valid on L1 despite representing an invalid execution.

This is structurally identical to the external report's vulnerability class: operations that must execute atomically (approve + issue, or validate + execute + charge) can be split by a party controlling an execution parameter (gas amount there; `is_reverted` hint here), causing partial state application and direct fund loss.

---

### Impact Explanation

A malicious sequencer exploiting this can:

- Accept a user's signed invoke transaction (e.g., a token transfer, a withdrawal, a DeFi interaction).
- Set `is_reverted = 1` in the hint, skipping `__execute__` entirely.
- Still charge the user the full fee and increment their nonce (making the transaction unreplayable).
- Commit a valid ZK proof of this state to L1.

The user permanently loses their fee payment (**direct loss of funds**). If the transaction was intended to release locked assets (e.g., a withdrawal from an escrow contract), those assets remain permanently inaccessible (**permanent freezing of funds**). Both impacts are in the allowed scope.

---

### Likelihood Explanation

The StarkNet sequencer is currently centralized. As the network decentralizes and the sequencer role opens to additional participants, any sequencer operator can exploit this unconstrained hint. No special cryptographic capability is required — only the ability to run the OS with a modified hint provider that sets `is_reverted = 1` for targeted transactions. The exploit produces a proof that passes L1 verification with no detectable anomaly in the proof itself.

---

### Recommendation

`check_is_reverted` must impose a real constraint. The correct fix is to derive `is_reverted` from the actual execution outcome rather than from an unconstrained pre-execution hint. Specifically:

- Remove the pre-execution `is_reverted` hint and always invoke `non_reverting_select_execute_entry_point_func` (or the reverting variant).
- Derive the reverted/succeeded status from the `failure_flag` returned by `execute_entry_point`, which is constrained by the contract's actual return values.
- If the optimization of skipping execution for known-reverted transactions is required, the hint must be validated post-execution by asserting it equals the actual `failure_flag`.

---

### Proof of Concept

1. User submits a valid invoke transaction (e.g., `transfer(recipient, amount)`) with sufficient gas and a correct signature.
2. Sequencer's hint provider sets `is_reverted = 1` for this transaction via `%{ IsReverted %}`.
3. OS executes `execute_invoke_function_transaction`:
   - `check_and_increment_nonce` runs → nonce incremented.
   - `run_validate` runs → signature verified, gas consumed.
   - `check_is_reverted(1)` → no-op, no constraint added.
   - `is_reverted == FALSE` is false → `__execute__` is skipped.
   - `charge_fee` runs → fee deducted from user's account.
4. `check_is_reverted` adds no constraint, so the Cairo proof is valid.
5. L1 verifier accepts the proof; state root reflects nonce increment and fee deduction but no token transfer.
6. User's fee is lost; the intended transfer never occurred; the nonce prevents replay. [2](#0-1) [4](#0-3)
