### Title
Unchecked Product Overflow in `compute_max_possible_fee` Causes `assert_nn_le` Failure, Halting Block Proof Generation â€” (File: `execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` multiplies user-controlled `max_amount` (bounded to `[0, 2^64)`) by `max_price_per_unit` (bounded only to `[0, 2^128)`) in plain felt arithmetic. Their product can exceed `2^128`. The result is then passed as the second argument to `assert_nn_le(actual_fee, max_fee)` in `charge_fee`, which internally calls `assert_le(actual_fee, max_fee)` â€” a range-check that requires `max_fee - actual_fee âˆˆ [0, 2^128)`. When `max_fee â‰¥ 2^128 + actual_fee`, this range check is unsatisfiable regardless of what the sequencer sets for `actual_fee`, causing the OS proof to be unprovable for any block containing such a transaction.

### Finding Description

```cairo

The individual bounds enforced during transaction-hash computation in `pack_resource_bounds` are:

- `max_amount â‰¤ 2^64 âˆ’ 1` via `assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1)`
- `max_price_per_unit âˆˆ [0, 2^128)` via `assert_nn(resource_bounds.max_price_per_unit)` [2](#0-1) 

The product of these two individually-valid values can reach up to `(2^64 âˆ’ 1) Ã— (2^128 âˆ’ 1) â‰ˆ 2^192`, and the sum of three such products can reach `â‰ˆ 3 Ã— 2^192`. There is **no upper-bound check on the product or the sum**.

The result is then used in `charge_fee`:

```cairo

`assert_nn_le(a, b)` from `starkware.cairo.common.math` is implemented as:
1. `assert_nn(a)` â€” range-checks `a âˆˆ [0, 2^128)`
2. `assert_le(a, b)` â€” range-checks `b âˆ’ a âˆˆ [0, 2^128)`

When `max_fee â‰¥ 2^128`, condition 2 requires `actual_fee > max_fee âˆ’ 2^128`. But `actual_fee` is itself constrained to `[0, 2^128)` by `assert_nn`. If `max_fee â‰¥ 2 Ã— 2^128 = 2^129`, no value of `actual_fee` can satisfy both constraints simultaneously. The range check is **permanently unsatisfiable**, making the proof invalid.

### Impact Explanation

**High â€” Network not being able to confirm new transactions (total network shutdown).**

Any block that includes a transaction where `compute_max_possible_fee` returns a value `â‰¥ 2^129` cannot be proven by the StarkNet OS. The sequencer cannot produce a valid STARK proof for that block. If the sequencer's off-chain mempool validation uses non-modular (e.g., Rust/Python big-integer) arithmetic to compute `max_fee`, it will see a large but "valid" fee cap and include the transaction. The OS Cairo program then fails the range check, the proof is rejected, and the block cannot be finalized â€” halting the network for that block and all subsequent blocks until the transaction is excluded.

---

### Likelihood Explanation

**High.** The attack requires only submitting a standard V3 transaction with:
- `max_amount = 2^64 âˆ’ 1` (maximum allowed)
- `max_price_per_unit = 2^65 + 3` (well within `[0, 2^128)`)

Product = `(2^64 âˆ’ 1) Ã— (2^65 + 3) = 2^129 + 2^64 âˆ’ 3 > 2^128`.

With all three resource bounds set similarly, `max_fee â‰ˆ 3 Ã— 2^129`. No special privileges, leaked keys, or trusted-role access are required. The attacker is an ordinary transaction sender. The individual field values pass all existing range checks (`assert_nn_le` for `max_amount`, `assert_nn` for `max_price_per_unit`, `assert_nn_le` for `tip`), so the transaction is accepted into the mempool.

---

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds` (or in `compute_max_possible_fee`) to ensure the product `max_amount Ã— max_price_per_unit` cannot exceed `2^128 âˆ’ 1`. For example, restrict `max_price_per_unit â‰¤ 2^64 âˆ’ 1` (matching the bit-width of `max_amount`), so the product is at most `(2^64 âˆ’ 1)^2 < 2^128`. Alternatively, replace `assert_nn_le(actual_fee, max_fee)` with a checked comparison that handles values larger than `2^128`. [2](#0-1)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
