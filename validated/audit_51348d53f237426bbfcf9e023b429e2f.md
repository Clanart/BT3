### Title
Fee Validity Check Ignores Decimal Normalization, Enabling Permanent Fund Lock - (`near/omni-bridge/src/lib.rs`)

### Summary

The `init_transfer` and `update_transfer_fee` functions validate that `fee < amount` in origin-chain token units. However, when `sign_transfer` is called, the net amount `(amount - fee)` is normalized to the destination chain's decimal precision via floor division. If the destination chain has fewer decimals than the origin chain, the normalized amount can be 0 even when `fee < amount` passes. This causes `sign_transfer` to always panic with `InvalidAmountToTransfer`, permanently locking the user's tokens with no recovery path.

### Finding Description

**Step 1 — Fee check at `init_transfer` time (origin units only):**

In `near/omni-bridge/src/lib.rs`, the fee validity check is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

This check is performed in raw origin-chain token units. No normalization is applied. [1](#0-0) 

**Step 2 — Same check in `update_transfer_fee` (origin units only, fee can only increase):**

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

Critically, `fee.fee >= current_fee.fee` means the fee can only be **increased**, never decreased. Once a user sets a fee that makes the normalized net amount 0, they cannot undo it. [2](#0-1) 

**Step 3 — Normalization at `sign_transfer` time (destination units):**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` uses floor division: `amount / 10^(origin_decimals - dest_decimals)`. If `(amount - fee) < 10^(origin_decimals - dest_decimals)`, the result is 0 and `sign_transfer` panics. [3](#0-2) 

**Step 4 — `normalize_amount` uses floor division:**

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

**Step 5 — No recovery mechanism exists:**

The transfer message is stored in `pending_transfers` and can only be removed via `claim_fee_callback` (requires proof of finalization on the destination chain) or `remove_fin_transfer` (same). If `sign_transfer` always panics, neither path is reachable, and the tokens are permanently locked. [5](#0-4) 

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Concrete example:
- Token has 24 decimals on NEAR, 6 decimals on EVM (diff = 18, factor = `10^18`)
- User transfers `amount = 2_000_000` (in 24-decimal units)
- User sets `fee = 1_999_999` — passes `fee < amount` check
- Net amount = `1` in 24-decimal units
- `normalize_amount(1, {origin: 24, dest: 6}) = 1 / 10^18 = 0`
- `sign_transfer` panics with `InvalidAmountToTransfer` every time
- User cannot decrease the fee (only increases allowed)
- Tokens are permanently locked in the bridge

### Likelihood Explanation

**Moderate.** Tokens with large decimal differences between NEAR (24 decimals) and EVM destinations (6 or 18 decimals) are common. A user who sets a fee close to the transfer amount — whether accidentally, through a misconfigured UI, or through misunderstanding of decimal normalization — will permanently lose their funds. The `update_transfer_fee` path makes this worse: a user who tries to increase their fee to attract a relayer can inadvertently cross the threshold.

### Recommendation

Add a normalized-amount check in both `init_transfer` and `update_transfer_fee`. At `init_transfer` time, if the destination token's decimals are already registered, validate:

```rust
let decimals = self.token_decimals.get(&token_address);
if let Some(decimals) = decimals {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee().expect("fee < amount"),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

Similarly in `update_transfer_fee`, after updating the fee, validate that the normalized net amount remains > 0 for the destination chain.

Additionally, consider adding a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have not yet been signed, to provide a recovery path for stuck transfers.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (e.g., NEAR → EVM USDC).
2. Call `ft_transfer_call` → `init_transfer` with `amount = 1_000_000` and `fee = 999_999`.
   - Check `fee < amount` → `999_999 < 1_000_000` ✓ passes.
3. Call `sign_transfer` for this transfer.
   - `amount_without_fee() = 1`
   - `normalize_amount(1, {origin: 24, dest: 6}) = 0`
   - `require!(0 > 0, ...)` → panics with `InvalidAmountToTransfer`.
4. Attempt `update_transfer_fee` to decrease fee → fails because `fee.fee >= current_fee.fee` is required.
5. No other recovery path exists. The `1_000_000` tokens are permanently locked in the bridge.
