### Title
Asymmetric `within_margin` Computation Uses Proposer-Controlled Price as Margin Basis, Allowing Gas Price Cap Bypass in Block Commitments — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `within_margin` function used to validate a proposer's L1 gas price computes the allowed tolerance band from `number1` — the **proposer-supplied** price — rather than from `number2`, the **locally computed reference** price. This creates an asymmetric acceptance window: a proposer can embed a gas price up to ≈11.11% above the reference instead of the intended 10%, bypassing the `max_l1_gas_price_wei` cap enforced by `apply_fee_transformations`. The inflated price is committed verbatim into the block header and block hash, causing every transaction in the block to be charged at a higher-than-permitted rate.

---

### Finding Description

**Reference price computation (clamped)**

In `is_block_info_valid`, the validator independently derives a reference gas price by calling `get_l1_prices_in_fri_and_wei`, which internally calls `apply_fee_transformations`:

```rust
price_info.base_fee_per_gas = price_info
    .base_fee_per_gas
    .saturating_add(gas_price_params.l1_gas_tip_wei)
    .clamp(gas_price_params.min_l1_gas_price_wei, gas_price_params.max_l1_gas_price_wei);
``` [1](#0-0) 

The result is a reference price `R` that is guaranteed to lie in `[min, max_l1_gas_price_wei]`.

**Margin check — wrong base**

The validator then calls:

```rust
within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
``` [2](#0-1) 

Inside `within_margin`, the tolerance is computed from `number1` (the **proposed** price), not `number2` (the reference):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← uses proposed, not reference
    number1.0.abs_diff(number2.0) <= margin
}
``` [3](#0-2) 

**Algebraic consequence**

Let `P = proposed`, `R = reference`, `m = 0.10` (10 %).

The check reduces to: `|P − R| ≤ P · m`

For `P > R`:
```
P − R ≤ P · 0.10
P · 0.90 ≤ R
P ≤ R / 0.90 ≈ R · 1.1111
```

The intended upper bound (margin from reference) would be `R · 1.10`. The actual upper bound is `R · 1.1111`, i.e. the proposer can exceed the reference by **≈11.11 %** instead of 10 %, a systematic upward bias of **≈1.11 %** beyond the intended cap.

For `P < R` the window is correspondingly tighter (≈9.09 % below reference instead of 10 %), so the asymmetry always favours the proposer setting a higher price.

**Accepted price enters the block commitment**

The accepted `ProposalInit` (containing the inflated `l1_gas_price_fri`) is passed directly to `convert_to_sn_api_block_info`, which constructs the `BlockInfo` used for execution and the block header: [4](#0-3) 

The block header gas price feeds into the block hash calculation and into every fee charged to users in that block.

---

### Impact Explanation

- **Wrong gas price committed to block header / block hash.** The proposer can embed `l1_gas_price_fri ≈ max_l1_gas_price_wei / 0.9`, which is ≈11.11 % above the reference. Because `apply_fee_transformations` already clamps the reference to `max_l1_gas_price_wei`, the proposer effectively bypasses the operator-configured maximum by ≈1.11 %.
- **Incorrect fee charged to every transaction in the block.** All transactions whose fee is computed against `l1_gas_price_fri` pay ≈1.11 % more than the protocol maximum permits.
- **Systematic, repeatable.** Every round in which the exploiting node is proposer produces a block with an inflated gas price that passes validator checks.

This falls under: *Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.*

---

### Likelihood Explanation

Any node that is selected as proposer in a consensus round can exploit this without any special privilege. The proposer simply sets `l1_gas_price_fri = locally_computed_reference / 0.9` in the `ProposalInit`. All validators run the same asymmetric `within_margin` check and accept the proposal. No external oracle manipulation, flashloan, or MEV infrastructure is required — the bug is purely in the margin arithmetic.

---

### Recommendation

Compute the margin from the **reference** price (`number2`), not the proposer-supplied price (`number1`):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    // Use the locally-computed reference (number2) as the basis, not the proposed value.
    let margin = (number2.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```

This produces a symmetric window `[R · 0.90, R · 1.10]` centred on the reference, matching the intended 10 % tolerance and preventing the proposer from inflating the margin by choosing a higher base.

---

### Proof of Concept

1. Validator computes reference `R = apply_fee_transformations(oracle_price)`, clamped to `max_l1_gas_price_wei`. Suppose `R = 1 000 000 fri`.
2. Proposer sets `l1_gas_price_fri_proposed = R / 0.9 = 1 111 112 fri` in `ProposalInit`.
3. `within_margin(1_111_112, 1_000_000, 10)`:
   - `abs_diff = 111_112`
   - `margin = (1_111_112 × 10) / 100 = 111_111`
   - `111_112 ≤ 111_111` — passes (off by one, within `GAS_PRICE_ABS_DIFF_MARGIN = 1`).
4. `is_block_info_valid` returns `Ok(())`.
5. `convert_to_sn_api_block_info` builds `BlockInfo` with `l1_gas_price_fri = 1_111_112`.
6. Every transaction in the block is charged at `1_111_112 fri` per L1 gas unit — ≈11.11 % above the reference and ≈1.11 % above the operator-configured maximum.
7. The block header and block hash are committed with this inflated gas price. [5](#0-4) [3](#0-2) [6](#0-5)

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
