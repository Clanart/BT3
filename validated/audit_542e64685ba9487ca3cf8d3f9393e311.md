### Title
Permanent Fund Loss via `normalize_amount` Floor-Division Rounding to Zero in `init_transfer` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer_internal` burns or locks user tokens before any decimal-normalization check. If the deposited `amount_without_fee` is smaller than the decimal scaling factor (`10^(origin_decimals − decimals)`), `normalize_amount` returns 0 and `sign_transfer` permanently panics with `InvalidAmountToTransfer`. Because no cancel/refund path exists for the pending transfer, the user's tokens are irrecoverably lost.

---

### Finding Description

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

When `origin_decimals > decimals` (e.g., 24 vs 18, scale = 1 000 000), any `amount_without_fee < 1 000 000` normalizes to 0.

`init_transfer` only validates that `fee.fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

It does **not** check that `normalize_amount(amount_without_fee) > 0`. Execution then proceeds to `init_transfer_internal`, which burns or locks the full token amount before returning:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [3](#0-2) 

Later, when a relayer calls `sign_transfer`, the normalization check fires and panics:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

`sign_transfer` panics without removing the transfer message from `pending_transfers`. No public cancel or refund function exists; `remove_transfer_message` and `remove_transfer_message_without_refund` are internal helpers that do not restore burned/locked tokens. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

For any token registered with `origin_decimals > decimals` (a common configuration for EVM tokens bridged to NEAR), a user who deposits an `amount_without_fee` below the scaling threshold has their tokens permanently burned or locked with no on-chain recovery path. The transfer message sits in `pending_transfers` indefinitely; `sign_transfer` will always revert for it.

This is distinct from the acknowledged "decimal dust" behavior (a small sub-unit remainder). Here the **entire** transferred value is lost, not a rounding remainder.

---

### Likelihood Explanation

**Medium.** The condition is reachable by any unprivileged user calling `ft_transfer_call` → `init_transfer` with a small amount. Tokens with a 6-decimal difference (e.g., 24 vs 18) require only that `amount_without_fee < 1 000 000` base units — a realistic scenario for low-value transfers or tokens with high unit prices. No special privileges, colluding parties, or external dependencies are required.

---

### Recommendation

Add a normalization guard inside `init_transfer` (before tokens are burned/locked), using the destination-chain decimals for the token:

```rust
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
require!(
    Self::normalize_amount(
        transfer_message.amount_without_fee()
            .near_expect(BridgeError::InvalidFee),
        decimals,
    ) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This mirrors the existing guard in `sign_transfer` and ensures the transfer is rejected before any state mutation occurs.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (scale = 1 000 000).
2. Alice calls `ft_transfer_call` with `amount = 500 000` and `fee = 0`.
3. `init_transfer` passes the `fee < amount` check (0 < 500 000). [2](#0-1) 
4. `init_transfer_internal` burns Alice's 500 000 tokens (deployed token) or locks them (native token). [3](#0-2) 
5. The transfer message is stored in `pending_transfers`.
6. A relayer calls `sign_transfer`. `normalize_amount(500 000, {24, 18}) = 500 000 / 1 000 000 = 0`. [1](#0-0) 
7. `require!(amount_to_transfer > 0, ...)` panics. The transfer message is never removed. [6](#0-5) 
8. Alice's 500 000 tokens are permanently burned/locked. No cancel or refund function exists to restore them.

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
