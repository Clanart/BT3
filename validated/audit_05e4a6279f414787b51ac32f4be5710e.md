### Title
Unbacked New Token Minting via Unguarded `burn.and(mint)` Promise Combinator in `swap_migrated_token` — (`near/omni-bridge/src/lib.rs`)

### Summary

`swap_migrated_token` uses `burn.and(mint)` to atomically swap old tokens for new tokens. In NEAR's promise system, `Promise::and()` schedules both sub-promises **independently and in parallel** — the mint is NOT conditional on the burn succeeding. If the burn fails for any reason, the mint still executes, producing unbacked new token supply.

### Finding Description

The function is:

```rust
fn swap_migrated_token(
    &mut self,
    sender_id: AccountId,
    old_token: AccountId,
    amount: U128,
) -> Promise {
    let new_token = self
        .migrated_tokens
        .get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);

    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);

    burn.and(mint)
}
``` [1](#0-0) 

It is called from `ft_on_transfer` with `.detach()`, meaning there is no outer callback to observe or react to the combined result:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
``` [2](#0-1) 

**The critical flaw:** In NEAR's runtime, `Promise::and()` creates a *join* of two independently-dispatched cross-contract calls. Both are scheduled to execute regardless of each other's outcome. This is fundamentally different from `burn.then(callback_that_mints_only_if_burn_succeeded)`. If the burn panics or is rejected by the old token contract, the mint still executes.

Contrast this with the correct pattern used elsewhere in the codebase — `burn_tokens_if_needed` — which explicitly allocates gas and detaches the burn as a fire-and-forget side effect, never pairing it with a dependent mint:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();
    }
}
``` [3](#0-2) 

Additionally, `swap_migrated_token` does **not** call `.with_static_gas()` on either the burn or mint promise, unlike every other token interaction in the file. This means gas is distributed dynamically by the runtime, creating an additional failure surface.

### Impact Explanation

If the burn fails (old token contract panics, gas exhaustion, contract upgrade that rejects burns, etc.):
- The old tokens remain in the bridge's account (not destroyed)
- The mint still executes — the user receives `amount` of new tokens
- New token supply is unbacked: new tokens exist without corresponding old token destruction
- This breaks the core collateralization invariant of the migration mechanism

Impact: **High** — unbacked supply / accounting corruption that breaks bridge collateralization.

### Likelihood Explanation

**Low-to-medium.** The old token is a bridge-deployed token whose controller is the bridge itself (set via `migrate_deployed_token`, a DAO-only function). Under normal operation, the burn should succeed. However:

1. If the old token contract is upgraded post-migration to reject burns (possible via `update_tokens_controller` or a token upgrade), any user holding old tokens can trigger unbacked minting.
2. Gas exhaustion scenarios (user provides minimal gas to `ft_transfer_call`) could cause the burn to fail while the mint, receiving its independent gas share, succeeds.
3. The structural flaw is present regardless — the code is one contract upgrade away from being exploitable by any unprivileged user.

### Recommendation

Replace the parallel `burn.and(mint)` with a sequential callback pattern that gates minting on confirmed burn success:

```rust
fn swap_migrated_token(...) -> Promise {
    let new_token = ...;
    ext_token::ext(old_token)
        .with_static_gas(BURN_TOKEN_GAS)
        .burn(amount)
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(MINT_TOKEN_GAS)
                .swap_migrated_token_callback(new_token, sender_id, amount)
        )
}

#[private]
pub fn swap_migrated_token_callback(
    &mut self,
    #[callback_result] burn_result: Result<(), PromiseError>,
    new_token: AccountId,
    sender_id: AccountId,
    amount: U128,
) {
    require!(burn_result.is_ok(), "Burn failed; mint aborted");
    ext_token::ext(new_token)
        .with_static_gas(MINT_TOKEN_GAS)
        .mint(sender_id, amount, None)
        .detach();
}
```

### Proof of Concept

1. DAO calls `migrate_deployed_token(origin_chain, old_token, new_token)` — sets `migrated_tokens[old_token] = new_token`.
2. DAO upgrades `old_token` contract so its `burn` function always panics.
3. Attacker holds `amount` of `old_token`.
4. Attacker calls `old_token.ft_transfer_call(bridge, amount, SwapMigratedToken)`.
5. Bridge's `ft_on_transfer` calls `swap_migrated_token`, which dispatches `burn.and(mint)` and detaches.
6. `burn` executes on `old_token` → panics → fails.
7. `mint` executes on `new_token` → succeeds → attacker receives `amount` of `new_token`.
8. Old tokens remain in bridge account (not destroyed). New token supply is unbacked.
