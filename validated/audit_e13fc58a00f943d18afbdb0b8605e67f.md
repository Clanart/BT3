### Title
Incorrect Resource Identifier `'L1_DATA'` for L1 Data Gas Causes Transaction Hash Mismatch for All V3 Transactions - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/constants.cairo`)

---

### Summary

In `constants.cairo`, the resource identifier constant `L1_DATA_GAS` is assigned the short-form value `'L1_DATA'` instead of the protocol-correct `'L1_DATA_GAS'`. This identifier is packed into the fee-fields hash during v3 transaction hash computation. Because the OS enforces this incorrect identifier via a Cairo assertion and packs it into the hash, the OS computes a different transaction hash than any conforming client, making every v3 transaction unprocessable.

---

### Finding Description

The vulnerability class from the external report is a **type-identifier mismatch**: a name registered in code does not match the name declared in the serialization/protocol metadata, causing messages to be rejected. The direct analog here is in the StarkNet OS transaction hash computation.

In `constants.cairo`, the three resource-type identifiers are defined as:

```cairo
const L1_GAS     = 'L1_GAS';
const L2_GAS     = 'L2_GAS';
const L1_DATA_GAS = 'L1_DATA';   // ← should be 'L1_DATA_GAS'
``` [1](#0-0) 

The naming pattern for `L1_GAS` and `L2_GAS` is self-consistent: the variable name matches the string literal. `L1_DATA_GAS` breaks this pattern — the variable name implies the value should be `'L1_DATA_GAS'`, but the actual value is the truncated `'L1_DATA'`.

This constant is consumed in two critical places inside `hash_fee_fields` in `transaction_hash.cairo`:

**1. Assertion that enforces the identifier:**
```cairo
let l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];
assert l1_data_gas_bounds.resource = L1_DATA_GAS;   // enforces 'L1_DATA'
assert data_to_hash[3] = pack_resource_bounds(l1_data_gas_bounds);
``` [2](#0-1) 

**2. Packing the identifier into the hash:**
```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    ...
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
``` [3](#0-2) 

The packed value — including the resource identifier — is then fed into `poseidon_hash_many` as part of the fee-fields hash, which is itself included in the full transaction hash via `hash_tx_common_fields`. [4](#0-3) 

The fee-fields hash is then committed into the full transaction hash for every v3 transaction type (invoke, declare, deploy_account): [5](#0-4) 

The OS also hard-enforces that exactly 3 resource bounds are present for every v3 transaction:

```cairo
with_attr error_message("Invalid number of resource bounds: {n_resource_bounds}.") {
    assert n_resource_bounds = 3;
}
``` [6](#0-5) 

This means L1 data gas resource bounds are mandatory for every v3 transaction, and the mismatch affects all of them.

---

### Impact Explanation

The mismatch produces two mutually exclusive failure modes, both fatal:

**Path A — Client sends the protocol-correct `'L1_DATA_GAS'`:**
The Cairo assertion `assert l1_data_gas_bounds.resource = L1_DATA_GAS` fails (since `'L1_DATA_GAS' ≠ 'L1_DATA'`). The OS proof cannot be generated. The block cannot be proven and finalized.

**Path B — Prover adapts to supply `'L1_DATA'` to satisfy the assertion:**
The OS packs `'L1_DATA'` into the fee-fields hash and computes a transaction hash that differs from the hash the client signed (the client used `'L1_DATA_GAS'` per the protocol spec). The signature verification step operates over the client-computed hash, so the OS-computed hash will never match the signed hash. The transaction is rejected.

In both paths, **every v3 transaction** (the only supported non-deprecated transaction type, since `assert common_fields.version = 3` is enforced in all three account transaction hash functions) is unprocessable. [7](#0-6) 

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

Every v3 transaction submitted by any unprivileged user triggers this code path. No special privileges, leaked keys, or adversarial conditions are required. The bug is deterministic and affects 100% of v3 transactions. Likelihood is **certain** once the OS is deployed.

---

### Recommendation

Change the constant in `constants.cairo` to match the protocol-specified resource identifier:

```cairo
// Before (incorrect):
const L1_DATA_GAS = 'L1_DATA';

// After (correct):
const L1_DATA_GAS = 'L1_DATA_GAS';
``` [8](#0-7) 

The same fix must be propagated to `constants_template.txt` so that the autogenerated file is regenerated correctly: [9](#0-8) 

---

### Proof of Concept

1. Construct any valid v3 Invoke transaction. Per the StarkNet protocol, the `resource_bounds` array must contain an entry with `resource = 'L1_DATA_GAS'` (the protocol-specified identifier for L1 data gas).

2. Submit the transaction to the sequencer. The OS program processes it via `execute_invoke_function_transaction` → `compute_invoke_transaction_hash` → `hash_tx_common_fields` → `hash_fee_fields`.

3. Inside `hash_fee_fields`, the assertion `assert l1_data_gas_bounds.resource = L1_DATA_GAS` evaluates as `assert 'L1_DATA_GAS' = 'L1_DATA'`, which is `assert 0x4c315f444154415f474153 = 0x4c315f44415441`. This is false; the Cairo VM raises an assertion error and proof generation fails.

4. Alternatively, if the prover supplies `resource = 'L1_DATA'` to pass the assertion, the OS computes `fee_fields_hash` using `'L1_DATA'` packed in, producing a transaction hash that differs from the client-signed hash. The block proof is invalid.

5. Since `assert n_resource_bounds = 3` is mandatory and L1 data gas is always the third bound, **no v3 transaction can ever be successfully proven**, halting the network.
