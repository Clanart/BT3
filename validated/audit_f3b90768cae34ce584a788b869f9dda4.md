### Title
Storage Write to `CHANGE_CLASS_ENTRY` Key Causes Class Hash Corruption and Missed Storage Revert — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/revert.cairo`)

### Summary
`execute_storage_write` logs any storage key — including the sentinel value `CHANGE_CLASS_ENTRY = 2**251+1` — directly into the revert log without bounds validation. During backward traversal in `revert_contract_changes`, this entry is indistinguishable from a genuine class-change entry, causing the contract's class hash to be overwritten with the old storage value and the storage write to go un-reverted.

### Finding Description

**Root cause 1 — missing key validation in `execute_storage_write`:** [1](#0-0) 

`storage_key` is taken from `request.key` and written verbatim as `RevertLogEntry.selector`. No assertion enforces `storage_key < CHANGE_CONTRACT_ENTRY (2**251)`.

**Root cause 2 — type disambiguation in `revert_contract_changes` relies solely on the selector value:** [2](#0-1) 

When `selector == CHANGE_CLASS_ENTRY`, the function sets `class_hash = revert_log_end[0].value` and recurses — it does **not** write a storage dict entry. A storage write with key `2**251+1` produces a revert log entry that is byte-for-byte identical to a `replace_class` entry.

**Contrast with the legitimate class-change path** (written by `execute_replace_class`): [3](#0-2) 

Both paths produce `RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=<some felt>)`. There is no tag or source marker to distinguish them.

**Sentinel constants:** [4](#0-3) 

`CHANGE_CLASS_ENTRY = 2**251 + 1` is a valid STARK field element (field prime is `2^251 + 17·2^192 + 1`), so it is a legal storage key.

### Impact Explanation

After the reverted transaction:
- The contract's `class_hash` is set to `prev_value` at storage key `2**251+1` — typically `0` if never previously written, making the contract permanently uncallable.
- The storage write to key `2**251+1` is **not** reverted; the post-revert storage state is wrong.

A contract holding funds (ETH, ERC-20 balances) whose class hash is zeroed out becomes permanently inaccessible — **Critical: Permanent freezing of funds**. Additionally, because the OS proof encodes this corrupted state while any independent state-replay node would compute the correct state, divergent state roots can emerge — **High: Unintended chain split**.

### Likelihood Explanation

The attacker only needs to:
1. Declare any Cairo class containing `storage_write(key=2**251+1, value=X)` followed by a panic.
2. Deploy a contract from that class and invoke the function.
3. The transaction reverts; the OS revert path misclassifies the log entry.

No privileged access, leaked keys, or operator cooperation is required. The exploit is fully self-contained and locally testable.

### Recommendation

Add a range check in `execute_storage_write` before writing to the revert log:

```cairo
// In execute_storage_write, after reading storage_key:
assert_lt(storage_key, CHANGE_CONTRACT_ENTRY);  // key must be < 2**251
assert [revert_log] = RevertLogEntry(selector=storage_key, value=prev_value);
```

This mirrors the implicit assumption already present in `revert_contract_changes` and closes the ambiguity at the source.

### Proof of Concept

```

2. Deploy contract A from class C.

   - storage[2**251+1] = 42  (not reverted; should be 0)
```
