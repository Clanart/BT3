### Title
Missing Upper Bound on `max_price_per_unit` in Resource Bounds Causes Felt Arithmetic Overflow, Aborting OS Fee Charging and Invalidating Block Proofs - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo` and `execution/transaction_impls.cairo`)

---

### Summary

The `pack_resource_bounds` function validates `max_amount` with an explicit upper bound (`assert_nn_le(..., 2**64 - 1)`) but only checks `max_price_per_unit` is non-negative (`assert_nn`), leaving no upper bound. When `compute_max_possible_fee` multiplies `max_amount * max_price_per_unit` in felt arithmetic (mod P ≈ 2^251), the result can be a felt value exceeding 2^128. The subsequent `assert_nn_le(actual_fee, max_fee)` call in `charge_fee` then fails unconditionally because Cairo's range-check-based `assert_nn_le` requires both its arguments and their difference to lie in [0, 2^128 − 1]. This aborts the OS execution, invalidates the block proof, and can halt the network.

---

### Finding Description

**Root cause — `pack_resource_bounds` (transaction_hash.cairo:103–108):**

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // bounded ✓
    assert_nn(resource_bounds.max_price_per_unit);            // only ≥ 0, NO upper bound ✗
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

`max_price_per_unit` is a raw felt accepted from the transaction sender. Any value in [0, P−1] passes the `assert_nn` check.

**Propagation — `compute_max_possible_fee` (transaction_impls.cairo:99–101):**

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is felt arithmetic (mod P). If `max_price_per_unit > 2^128`, the product `max_amount * max_price_per_unit` is a felt in (2^128, P−1], making `max_fee` a felt > 2^128.

**Failure point — `charge_fee` (transaction_impls.cairo:135):**

```cairo
assert_nn_le(calldata.amount.low, max_fee);
```

Cairo's `assert_nn_le(a, b)` is implemented via range checks:
- `assert_nn(a)` → requires `a ∈ [0, 2^128 − 1]` ✓ (`actual_fee` is a Uint256 low limb)
- `assert_le(a, b)` → requires `b − a ∈ [0, 2^128 − 1]`

If `max_fee > 2^128`, then `max_fee − actual_fee > 2^128 − 1` for any valid `actual_fee`, and the range check fails unconditionally. There is no way for the sequencer to supply an `actual_fee` that avoids this failure — even `actual_fee = 0` fails because `max_fee` itself is not in the range-check range. The OS execution aborts, the block proof is invalid.

---

### Impact Explanation

If the sequencer includes a transaction carrying `max_price_per_unit > 2^128` for any resource type, the OS execution aborts during `charge_fee`. The block proof cannot be generated. If the sequencer's mempool validation mirrors the OS validation (which it is expected to, since the OS is the authoritative validator), the sequencer has no independent reason to reject such a transaction. A single such transaction in a block causes the entire block to be unprovable, halting the network's ability to confirm new transactions.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a v3 invoke, declare, or deploy-account transaction with `max_price_per_unit` set to any felt value (e.g., 2^200). The transaction hash computation in `hash_fee_fields` / `pack_resource_bounds` does not reject it. The sequencer's off-chain validation is expected to mirror OS-level checks; since the OS imposes no upper bound, the sequencer's mempool is unlikely to either. The attacker only needs to submit one such transaction and have it included in a block.

---

### Recommendation

Add an explicit upper bound on `max_price_per_unit` in `pack_resource_bounds` so that the product `max_amount * max_price_per_unit` is guaranteed to fit within the range-check domain:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // add upper bound
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

With `max_amount ≤ 2^64 − 1` and `max_price_per_unit ≤ 2^128 − 1`, the product `max_amount * max_price_per_unit ≤ (2^64 − 1)(2^128 − 1) < 2^192 < P`, and the sum of three such products is bounded well within the felt field and within the range-check domain.

---

### Proof of Concept

1. Attacker constructs a v3 invoke transaction with:
   - `l1_gas_bounds.max_amount = 1`
   - `l1_gas_bounds.max_price_per_unit = 2^200` (a valid felt, passes `assert_nn`)
   - All other resource bounds set to 0
2. The transaction hash is computed normally; `pack_resource_bounds` accepts `max_price_per_unit = 2^200` (only `assert_nn` is checked).
3. The sequencer includes the transaction in a block (no OS-level upper bound to mirror in mempool validation).
4. During OS execution, `compute_max_possible_fee` computes:
   `max_fee = 1 * 2^200 + 0 + 0 = 2^200` (a felt in (2^128, P−1])
5. `charge_fee` calls `assert_nn_le(actual_fee, 2^200)`. The range check on `2^200 − actual_fee` fails because `2^200 > 2^128 − 1`.
6. OS execution aborts → block proof fails → network cannot confirm new transactions.
