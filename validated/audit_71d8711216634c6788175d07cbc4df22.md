### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Corruption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS Cairo program performs no validation on the caller-supplied `class_hash` value. Specifically, it does not check that the new class hash is non-zero or that it corresponds to a previously declared class. An unprivileged contract can call `replace_class` with `class_hash = 0` (i.e., `UNINITIALIZED_CLASS_HASH`) or any arbitrary undeclared hash, permanently corrupting its own on-chain state and freezing any funds held in its storage.

---

### Finding Description

In `execute_replace_class`, after deducting gas, the function reads `request.class_hash` directly from the syscall request and writes it into `contract_state_changes` without any sanity checks: [1](#0-0) 

The code even contains an explicit acknowledgment of the missing check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [2](#0-1) 

The missing checks are:

1. **No non-zero check**: `class_hash = 0` is accepted. In the OS, `UNINITIALIZED_CLASS_HASH` (imported in `deploy_contract.cairo`) is the sentinel value used to mark a contract as not yet deployed. Setting a live contract's class hash back to this value produces an inconsistent state. [3](#0-2) 

2. **No declared-class check**: Any arbitrary felt value is accepted as the new class hash, even one that has never been declared on-chain and therefore has no corresponding compiled class in the OS's `compiled_class_facts` bundle. [4](#0-3) 

By contrast, `deploy_contract` correctly enforces that the initial class hash is `UNINITIALIZED_CLASS_HASH` before writing a new one, showing the OS does have the concept of class-hash validity — it is simply not applied in `execute_replace_class`. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract calls `replace_class` with `class_hash = 0` or an undeclared hash:

- The corrupted class hash is committed into `contract_state_changes` and ultimately into the global state root via `state_update` → `compute_contract_state_commitment`. [6](#0-5) 

- All future transactions targeting this contract will attempt to execute using the invalid class hash. The OS fetches the class hash from state in `get_invoke_tx_execution_context` and uses it directly as the execution class: [7](#0-6) 

- Because the class hash does not exist in the `compiled_class_facts` bundle, no valid execution path exists for the contract. Any ERC-20 tokens, ETH, or other assets stored in the contract's storage slots become permanently inaccessible.

- The corruption is irreversible: the state root is updated and proven with the invalid class hash, and there is no recovery mechanism in the OS.

---

### Likelihood Explanation

**High.** The `replace_class` syscall is callable by any contract without privilege. A malicious contract deployer can:

1. Deploy a contract that holds user funds (e.g., a vault or token contract).
2. Attract deposits.
3. Call `replace_class(class_hash=0)` from within the contract.
4. The OS accepts the call, commits the corrupted state, and the funds are frozen permanently.

No special role, leaked key, or operator cooperation is required. The attacker only needs to be a contract deployer, which is a public, permissionless action on StarkNet.

---

### Recommendation

Add the following checks inside `execute_replace_class` before writing to `contract_state_changes`:

1. **Non-zero check**: Assert `class_hash != 0` (i.e., `class_hash != UNINITIALIZED_CLASS_HASH`). Return a failure response if violated.
2. **Declared-class check**: Verify that `class_hash` exists in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Return a failure response if the class is not found.

This mirrors the pattern already used in `execute_declare_transaction`, which enforces `assert_not_zero(compiled_class_hash)` before writing a class declaration: [8](#0-7) 

---

### Proof of Concept

1. Deploy a contract `VaultContract` that holds user ERC-20 balances and exposes a `drain()` entry point that internally calls the `replace_class` syscall with `class_hash = 0`.
2. Users deposit tokens into `VaultContract`.
3. Attacker calls `drain()`. The OS processes `execute_replace_class` with `class_hash = 0`, no validation fires, and `contract_state_changes` is updated with `class_hash = 0` for `VaultContract`.
4. `state_update` squashes and commits this change into the global state root. [9](#0-8) 

5. In all subsequent blocks, any transaction targeting `VaultContract` fetches `class_hash = 0` from state, fails to locate a compiled class, and cannot execute. All deposited tokens are permanently frozen with no recovery path.

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;

    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,

    );
