### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt value as a new class hash without verifying that the hash corresponds to a class that has actually been declared on-chain. An unprivileged contract can call this syscall with a hash that has never been declared, permanently replacing its own class pointer with an unresolvable reference. Any funds held by that contract become permanently frozen because no future call to the contract can succeed.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` with no membership check against `contract_class_changes`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-acknowledged TODO comment confirms the check is intentionally absent. The OS commits the state transition to the block without ever consulting `contract_class_changes` to confirm the target class exists.

After this state transition is finalized, every subsequent invocation of the contract will attempt to resolve the undeclared class hash. Because no Sierra/CASM bytecode is registered under that hash, the OS cannot load the class, execution fails unconditionally, and the contract is rendered permanently inert. The `replace_class` syscall can only be issued by the contract itself; once the class pointer is corrupted the contract cannot self-repair.

The analog to the external report is direct: just as IL protection was applied to **all** pools without a whitelist—letting an attacker manufacture a qualifying position at negligible cost—`replace_class` is accepted for **any** class hash without a declared-class whitelist, letting an attacker manufacture a permanently broken contract state at negligible cost.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class` is called with an undeclared hash and the block is proven:

- The contract's class pointer in the committed state root is irrecoverably set to the undeclared hash.
- No future transaction targeting the contract can be included in a valid block, because the OS cannot produce a valid execution trace for an unresolvable class.
- All ERC-20 balances, ETH, or other assets held in the contract's storage are permanently inaccessible.
- There is no protocol-level recovery path; `replace_class` can only be called by the contract itself, which is now non-executable.

---

### Likelihood Explanation

The attack path is fully unprivileged and requires only standard StarkNet capabilities:

1. Deploy any contract (standard `DeployAccount` or `deploy` syscall — no special role required).
2. Attract user funds into the contract (e.g., present it as a token, vault, or DEX).
3. Issue a transaction that calls `replace_class` with an arbitrary felt that has never been declared (e.g., `0xdeadbeef`).
4. The OS accepts the syscall, the sequencer includes the transaction, the block is proven, and the state root is updated.

No privileged key, no operator collusion, and no cryptographic break is required. The only social-engineering step (attracting deposits) is identical to the attack in the reference report, which also required luring liquidity into a manipulated pool.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the hash exists as a key in `contract_class_changes` (i.e., that a compiled class hash has been registered for it via a prior `Declare` transaction). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned value is non-zero before proceeding with the `dict_update`.
