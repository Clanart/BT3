### Title
Stale Stored Token Decimals Enable Unauthorized Minting of Bridged Assets After Origin-Token Decimal Upgrade - (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR OmniBridge records token decimals once at registration time in the `token_decimals` map and provides no mechanism to update them. All cross-chain amount conversions (`normalize_amount` / `denormalize_amount`) rely exclusively on these stored values. If an origin-chain token (e.g., an upgradeable EVM ERC-20) later changes its decimal count, the stored `origin_decimals` becomes stale, causing every subsequent `fin_transfer_callback` to apply the wrong scaling factor and mint an incorrect number of bridged tokens on NEAR.

### Impact Explanation

**Scenario — EVM token increases decimals (e.g., USDC 6 → 18):**

1. USDC is registered: `origin_decimals = 6`, `decimals = 6`, `diff_decimals = 0`.
2. USDC upgrades on Ethereum to 18 decimals. The stored NEAR entry is unchanged.
3. An attacker calls `initTransfer` on the EVM bridge with `amount = 10^18` (1 new USDC).
4. The EVM bridge locks/burns `10^18` raw units.
5. The NEAR `fin_transfer_callback` receives `init_transfer.amount = 10^18` from the verified proof.
6. `denormalize_amount(10^18, Decimals { decimals: 6, origin_decimals: 6 })` = `10^18 × 10^0 = 10^18`.
7. NEAR mints `10^18` units of the 6-decimal bridged token — representing **10^12 tokens** — to the attacker, while the bridge received only 1 USDC worth of value.

**Scenario — EVM token with >18 decimals decreases to 18:**

Stored `origin_decimals = 24`, `decimals = 18`, `diff_decimals = 6`. After the upgrade, 1 token = `10^18` units. The attacker sends `10^18` units; NEAR applies `× 10^6`, minting `10^24` NEAR-side units — a **10^6 over-mint**.

Both scenarios constitute unauthorized minting of bridged assets, directly breaking bridge collateralization.

### Likelihood Explanation

USDC and several other major ERC-20 tokens are deployed behind upgradeable proxies. A decimal change is a known upgrade vector (the external report explicitly cites it). Any such upgrade applied after the token is registered in the bridge immediately activates the vulnerability for all subsequent inbound transfers. No special attacker capability is required beyond submitting a normal `fin_transfer` proof — a path open to any bridge user.

### Recommendation

1. Add a `decimals` field to the stored `Decimals` struct that can be updated by a privileged role (DAO) after verifying the live on-chain value.
2. Provide an admin function `update_token_decimals(token_address, new_origin_decimals)` that re-reads the current decimals from the origin chain (via a cross-chain proof or a trusted oracle) and updates `token_decimals`.
3. Optionally, emit an event and pause inbound transfers for the affected token until the decimals are reconciled.

### Proof of Concept

1. USDC (6 decimals) is registered on NEAR: `token_decimals[Eth(usdc_addr)] = Decimals { decimals: 6, origin_decimals: 6 }`.
2. USDC upgrades on Ethereum to 18 decimals.
3. Attacker calls `OmniBridge.initTransfer(usdc, 10^18, ...)` on Ethereum, locking `10^18` raw units (= 1 new USDC ≈ $1).
4. Relayer submits proof to NEAR `fin_transfer`.
5. `fin_transfer_callback` extracts `amount = 10^18`, calls `denormalize_amount(10^18, {decimals:6, origin_decimals:6})` → `10^18`.
6. Bridge mints `10^18` units of the 6-decimal NEAR bridged-USDC token to the attacker.
7. Attacker holds `10^18 / 10^6 = 10^12` bridged USDC tokens (≈ $1,000,000,000,000) for a $1 input.

    fn add_token(
        &mut self,
        token_id: &AccountId,
        token_address: &OmniAddress,
        decimals: u8,
        origin_decimals: u8,
    ) {
        let chain_kind = token_address.get_chain();
        require!(
            self.token_id_to_address
                .insert(&(chain_kind, token_id.clone()), token_address)
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );
        require!(
            self.token_address_to_id
                .insert(token_address, token_id)
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );
        require!(
            self.token_decimals
                .insert(
                    token_address,
                    &Decimals {
                        decimals,
                        origin_decimals,
                    }
                )
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );
    }

    fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount * (10_u128.pow(diff_decimals))
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
