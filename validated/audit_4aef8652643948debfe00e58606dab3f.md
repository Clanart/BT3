Audit Report

## Title
Pool admin bypasses `maxAdminSpreadFeeE6` cap via uncapped per-bin additional fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`setPoolBinAdditionalFees` forwards caller-supplied `addFeeBuyE6`/`addFeeSellE6` values directly to the pool with no upper-bound validation, while the parallel `setPoolAdminFees` path correctly enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin fees to `type(uint16).max` (65 535, ≈6.5 %) with a single transaction and no timelock, front-run any swap in the affected bin, then reset — charging users materially more than the advertised rate and exceeding the protocol's hard fee ceiling.

## Finding Description

`setPoolAdminFees` enforces the factory-level cap before applying any fee change: [1](#0-0) 

`setPoolBinAdditionalFees` performs no analogous check and passes values straight through: [2](#0-1) 

`setBinAdditionalFees` on the pool stores the values without any cap: [3](#0-2) 

During every swap the per-bin fee is added on top of `params.baseFeeX64` (which already encodes the global spread fee): [4](#0-3) [5](#0-4) [6](#0-5) 

The hard ceiling for the global spread fee is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %): [7](#0-6) 

Because `addFeeBuyE6`/`addFeeSellE6` are `uint16`, the pool admin can set them to 65 535 (≈6.5 %) with no cap check and no timelock, pushing the effective per-bin fee to ≈26.5 % and bypassing the protocol's hard cap. The exploit path is: (1) admin observes victim swap in mempool, (2) admin front-runs with `setPoolBinAdditionalFees(pool, bin, 65535, 0)`, (3) victim swap executes at inflated fee, (4) admin resets fee to 0.

## Impact Explanation

This is a confirmed admin-boundary break: the pool admin exceeds the `maxAdminSpreadFeeE6` cap that the factory owner intended as an absolute limit, via a path that has no analogous guard. The direct consequence is loss of user principal — traders receive materially less output than quoted, with the excess flowing to the admin fee destination. The effective fee can reach ≈26.5 %, far above the 20 % hard ceiling. Slippage guards on `amountOutMinimum` do not protect against this because the swap still executes within the price limit while silently charging a higher fee.

## Likelihood Explanation

The pool admin is a semi-trusted role (EOA or multisig) that requires only two transactions to execute the attack. No special setup, malicious initial configuration, or privileged oracle access is needed. Any admin watching the mempool can execute this pattern repeatedly against any swap in any bin they administer.

## Recommendation

1. Add a cap check in `setPoolBinAdditionalFees`: enforce `addFeeBuyE6 <= maxAdminSpreadFeeE6` and `addFeeSellE6 <= maxAdminSpreadFeeE6` (or a dedicated per-bin cap stored on the factory), mirroring the guard at lines 414–415.
2. Collect accrued fees before updating per-bin fees, mirroring the `collectFees` call already present in `setPoolAdminFees` (lines 418–425), to prevent retroactive fee re-attribution.
3. Introduce a timelock on per-bin fee increases analogous to the oracle rotation timelock, giving users advance notice before a fee hike takes effect.

## Proof of Concept

```solidity
function test_admin_frontRuns_swap_with_binFee() public {
    // Pool is live with addFeeBuyE6 = 0 (advertised to users)
    (,,, uint16 buyFeeBefore,) = PoolStateLibrary._binState(pool, 0);
    assertEq(buyFeeBefore, 0);

    // Admin front-runs Alice's swap: spike per-bin fee to uint16 max (~6.5%)
    vm.prank(admin);
    factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, 0);

    (,,, uint16 buyFeeAfter,) = PoolStateLibrary._binState(pool, 0);
    assertEq(buyFeeAfter, type(uint16).max);

    // Alice's swap executes at inflated fee — receives less token0 than quoted
    vm.prank(alice);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            tokenIn: address(token1), tokenOut: address(token0),
            pool: pool, recipient: alice,
            amountIn: 1_000e18, amountOutMinimum: 0, sqrtPriceLimitX64: 0
        })
    );

    // Admin resets fee to 0 — excess flows to admin fee destination
    vm.prank(admin);
    factory.setPoolBinAdditionalFees(pool, 0, 0, 0);
    // amountOut is materially lower than at fee=0; difference accrued to admin
}
```
