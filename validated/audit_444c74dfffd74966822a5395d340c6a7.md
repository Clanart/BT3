### Title
Unbounded Storage Iteration in `DeleteAccount` During SELFDESTRUCT Executes Outside EVM Gas Accounting, Enabling Chain Halt - (File: x/evm/keeper/statedb.go)

---

### Summary

`DeleteAccount` iterates over every storage slot of a self-destructed contract using `ForEachStorage` and deletes them one-by-one in the Cosmos KV store. This iteration occurs **after** EVM execution, entirely outside EVM gas metering. Because the EVM charges a fixed 5,000 gas for `SELFDESTRUCT` regardless of storage size, an attacker can accumulate an arbitrarily large number of storage slots across many blocks and then trigger deletion in a single cheap transaction, causing block processing to consume unbounded wall-clock time and halting the chain.

---

### Finding Description

**Root cause — `DeleteAccount` in `x/evm/keeper/statedb.go`:** [1](#0-0) 

`DeleteAccount` calls `ForEachStorage`, which opens a KV prefix iterator over the entire storage namespace of the contract and deletes every slot: [2](#0-1) 

`ForEachStorage` itself is an unbounded iterator with no pagination or limit: [3](#0-2) 

**Call path — `statedb.Commit()` in `x/evm/statedb/statedb.go`:**

`DeleteAccount` is invoked from `Commit()` for every self-destructed object, on the consensus path: [4](#0-3) 

`Commit()` is called from `ApplyMessageWithConfig` with `commit=true` during block processing: [5](#0-4) 

**Why gas does not bound the work:**

The EVM charges a fixed 5,000 gas for `SELFDESTRUCT` (post-EIP-3529). There is no per-slot cost for storage clearing during self-destruction. The Cosmos-layer `DeleteAccount` loop runs entirely outside EVM gas metering — it is not subject to the block gas limit. In contrast, go-ethereum abandons the storage trie lazily; Ethermint eagerly iterates and deletes every slot in the KV store.

**Storage accumulation:**

Each `SSTORE` to a new slot costs 20,000 gas. With a 30 M gas block limit, ~1,500 new slots can be written per block. Over 1,000 blocks an attacker accumulates ~1.5 M slots. The subsequent `SELFDESTRUCT` costs 5,000 gas but triggers 1.5 M KV deletions outside gas accounting.

---

### Impact Explanation

The unbounded KV iteration in `DeleteAccount` runs on the consensus-critical block-processing path (`ApplyTransaction` → `ApplyMessageWithConfig` → `stateDB.Commit()`). A sufficiently large storage set causes the block to exceed the CometBFT block-processing timeout. All validators experience the same deterministic slowdown on the same block, causing a consensus failure and chain halt.

This matches the allowed Critical impact: *"Valid unprivileged transaction, RPC submission, or block-processing path can halt the chain, corrupt committed state, or cause deterministic validator consensus failure."*

---

### Likelihood Explanation

- Any unprivileged user can deploy a contract and call `SSTORE` in a loop.
- The cost is real but amortizable: 20,000 gas × N slots spread across N/1,500 blocks.
- The trigger transaction costs only 5,000 gas.
- No privileged role, governance action, or validator collusion is required.
- The attack is deterministic: every validator processes the same `SELFDESTRUCT` transaction and executes the same unbounded loop.

---

### Recommendation

1. **Charge gas proportional to storage size before executing `SELFDESTRUCT`**: Before calling `DeleteAccount`, count the number of storage slots (or use a stored slot-count per account) and deduct gas at the EVM level (e.g., `SSTORE_RESET_GAS` × slot_count). Reject the transaction if gas is insufficient.

2. **Lazy deletion**: Instead of eagerly iterating and deleting all slots in `DeleteAccount`, mark the account as deleted and schedule storage cleanup asynchronously in `EndBlock` with a per-block slot-deletion budget.

3. **Cap slots per contract**: Enforce a maximum number of storage slots per contract address at the `SetState` level, bounding the worst-case deletion cost.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract StorageSpammer {
    // Fill storage: call fill(0, 50000) to write 50,000 slots
    function fill(uint256 start, uint256 count) external {
        for (uint256 i = start; i < start + count; i++) {
            assembly { sstore(i, 1) }
        }
    }
    function boom() external {
        selfdestruct(payable(msg.sender));
    }
}
```

**Attack steps:**

1. Deploy `StorageSpammer`.
2. Over many blocks, call `fill(offset, 50000)` repeatedly (each call costs ~1 B gas for 50,000 slots; spread across blocks). Accumulate, e.g., 2,000,000 storage slots.
3. Call `boom()` — costs 5,000 gas for `SELFDESTRUCT`.
4. `statedb.Commit()` calls `DeleteAccount`, which calls `ForEachStorage` and iterates all 2,000,000 slots, issuing 2,000,000 KV store deletions outside EVM gas accounting.
5. Block processing time grows linearly with slot count; at sufficient scale it exceeds the CometBFT timeout, causing all validators to fail to commit the block and halting the chain. [3](#0-2) [2](#0-1) [4](#0-3)

func (k *Keeper) ForEachStorage(ctx sdk.Context, addr common.Address, cb func(key, value common.Hash) bool) {
	store := ctx.KVStore(k.storeKey)
	prefix := types.AddressStoragePrefix(addr)

	iterator := storetypes.KVStorePrefixIterator(store, prefix)
	defer iterator.Close()

	for ; iterator.Valid(); iterator.Next() {
		key := common.BytesToHash(iterator.Key())
		value := common.BytesToHash(iterator.Value())

		// check if iteration stops
		if !cb(key, value) {
			return
		}
	}
}

func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
