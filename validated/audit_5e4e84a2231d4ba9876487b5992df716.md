### Title
Stale `ChunkExtra` Fields Not Reset in Child Shard After Resharding Split — (`File: chain/chain/src/resharding/manager.rs`)

### Summary

During a shard split in `process_memtrie_resharding_storage_update`, the child shard's `ChunkExtra` is created by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The fields `validator_proposals`, `outcome_root`, `gas_used`, `balance_burnt`, `bandwidth_requests`, and `proposed_split` are carried over verbatim from the parent shard. This is the exact nearcore analog of the "yieldAmount not reset after withdrawal" pattern: a primary field (`state_root`) is correctly updated while secondary committed fields are left stale, breaking the invariant that `ChunkExtra` faithfully represents the result of applying the child shard's chunk.

The child `ChunkExtra` inherits from the parent:

| Field | Expected value for child | Actual value |
|---|---|---|
| `state_root` | child trie root | ✅ correctly set |
| `congestion_info` | child congestion | ✅ correctly set |
| `validator_proposals` | `[]` (no proposals from resharding) | ❌ parent's proposals |
| `outcome_root` | `CryptoHash::default()` | ❌ parent's outcome root |
| `gas_used` | `0` | ❌ parent's gas used |
| `balance_burnt` | `0` | ❌ parent's balance burnt |
| `bandwidth_requests` | child-appropriate | ❌ parent's requests |
| `proposed_split` | `None` | ❌ parent's `proposed_split` |

The `proposed_split` field is the most critical. If the parent shard's last chunk in epoch N+1 computed `proposed_split = Some(boundary)` (because the parent shard's memory exceeded the split threshold near the epoch boundary), that value is cloned into the child's `ChunkExtra`. The chunk producer for the child shard's first chunk in epoch N+2 then reads `chunk_extra.proposed_split()` and embeds it in the chunk header. During state witness validation, validators independently run the runtime for the child shard's first chunk and compute `proposed_split = None` (the cooldown prevents a new split). The mismatch triggers `InvalidChunkHeaderShardSplit`, causing all validators to reject the child shard's first chunk.

The `validator_proposals` field is the second critical field. If the parent shard's last chunk contained stake-change transactions, those `ValidatorStake` entries are cloned into the child's `ChunkExtra`. The chunk producer embeds them in the child shard's first chunk header as `prev_validator_proposals`. The epoch manager then processes these stale proposals as if they originated from the child shard, corrupting the validator set computation for the next epoch.
