### Title
`initTransfer1155` Does Not Set `multiTokens` Mapping, Causing Permanent ERC1155 Token Lock When `logMetadata1155` Is Not Called First — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` locks ERC1155 tokens in the bridge and emits an `InitTransfer` event using a deterministic virtual address, but never writes to the `multiTokens` mapping. The only function that populates `multiTokens` is `logMetadata1155`, which is a separate, optional, permissionless call. If a user calls `initTransfer1155` without a prior `logMetadata1155`, the bridge holds the ERC1155 tokens but `multiTokens[deterministicToken]` remains zero. When `finTransfer` is later called by the relayer, it falls through to the ERC20 branch and calls `IERC20(deterministicToken).safeTransfer(...)` on a codeless hash-derived address. OpenZeppelin v5's `SafeERC20` reverts with `AddressEmptyCode` on codeless targets, causing `finTransfer` to revert. The ERC1155 tokens remain locked with no on-chain recovery path unless `logMetadata1155` is called before the next `finTransfer` attempt.

---

### Finding Description

`deriveDeterministicAddress` produces a 20-byte truncation of `keccak256(abi.encodePacked(tokenAddress, tokenId))`: [1](#0-0) 

`logMetadata1155` is the only function that writes to `multiTokens`: [2](#0-1) 

`initTransfer1155` computes the same deterministic address, transfers the ERC1155 tokens into the bridge, and emits `InitTransfer` — but **never writes to `multiTokens`** and **never checks that `multiTokens[deterministicToken]` is already set**: [3](#0-2) 

`finTransfer` reads `multiTokens[payload.tokenAddress]`. If the entry is zero, it skips the ERC1155 branch and falls through to the final `else` branch: [4](#0-3) 

The final `else` branch calls `IERC20(payload.tokenAddress).safeTransfer(...)`. Because `deterministicToken` is a hash-derived address with no deployed code, OpenZeppelin v5's `SafeERC20` reverts with `AddressEmptyCode`. The entire `finTransfer` transaction reverts, rolling back `completedTransfers[nonce] = true`, so the nonce is not consumed — but the ERC1155 tokens remain locked in the bridge with no withdrawal mechanism.

The only recovery path is for someone to call `logMetadata1155(tokenAddress, tokenId)` before the next `finTransfer` attempt. If the relayer does not retry, or if `logMetadata1155` is never called, the tokens are permanently frozen.

The SECURITY.md confirms `logMetadata` is intentionally permissionless and optional: [5](#0-4) 

The test suite always calls `logMetadata1155` before `initTransfer1155`, masking the missing enforcement: [6](#0-5) 

---

### Impact Explanation

Any ERC1155 tokens transferred via `initTransfer1155` without a prior `logMetadata1155` call are locked in the bridge contract with no on-chain withdrawal or cancellation function. `finTransfer` will revert on every attempt until `logMetadata1155` is called. If the relayer does not retry after the revert, or if `logMetadata1155` is never called, the tokens are irrecoverably frozen. This matches: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

`initTransfer1155` is a public, permissionless function. Any user who calls it directly (e.g., via a script, a frontend that omits the `logMetadata1155` step, or a wallet integration) without first calling `logMetadata1155` will trigger the lock. The protocol provides no on-chain guard, no revert, and no warning. The missing enforcement is invisible to callers. Likelihood is **medium-high** for any ERC1155 integration that does not strictly follow the undocumented two-step flow.

---

### Recommendation

`initTransfer1155` should either:
1. **Require** that `multiTokens[deterministicToken]` is already set (i.e., `logMetadata1155` was called first), reverting with a clear error if not; or
2. **Set** `multiTokens[deterministicToken]` itself when it is not yet populated, mirroring the logic in `logMetadata1155`.

Option 1 is safer because it makes the dependency explicit and prevents silent state divergence.

        uint128 amount,
        uint128 fee,
        uint128 nativeFee,

        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {

        }

            tokenId,

            msg.sender,
            deterministicToken,
            currentOriginNonce,
            amount,
            fee,

            message,

            amount,
            fee,
