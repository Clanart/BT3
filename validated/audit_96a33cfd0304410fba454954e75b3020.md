Audit Report

## Title
Stranded Native ETH on Router Subsidizes Any Subsequent WETH Swap Caller — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function in `PeripheryPayments` uses `address(this).balance` to settle WETH swap obligations without verifying that the native ETH was contributed by the current payer in the current transaction. ETH left on the router after an `exactOutputSingle` or `exactInputSingle` call where `msg.value` exceeds the actual swap input is permanently stranded unless the user explicitly calls `refundETH()`. Any subsequent caller can drain this stranded ETH by calling `exactInputSingle` with `msg.value = 0` and `tokenIn = WETH`, receiving a fully subsidized swap output at the victim's expense.

## Finding Description
**Root cause — `pay` uses the router's full native balance:** [1](#0-0) 

When `nativeBalance >= value`, the function wraps and forwards the router's ETH to the pool with no check that the ETH was sent by `payer` in the current call. There is no per-transaction accounting of which caller contributed which ETH.

**ETH stranding mechanism — `exactOutputSingle` does not refund excess ETH:** [2](#0-1) 

A user sends `msg.value = amountInMaximum`. The swap callback fires and `pay` deposits only the actual `amountIn < amountInMaximum`. The remainder `amountInMaximum − amountIn` stays on the router. There is no automatic refund; the user must separately call `refundETH()` (lines 58–63), which most direct callers omit.

**Attack path — `exactInputSingle` with `msg.value = 0` and `tokenIn = WETH`:** [3](#0-2) 

`exactInputSingle` is `payable` and imposes no `msg.value >= amountIn` check. An attacker calls it with `msg.value = 0`, `tokenIn = WETH`, `amountIn = <stranded amount>`. The swap callback fires, `_justPayCallback` calls `pay` with `token = WETH`, `payer = attacker`, `value = stranded_amount`. Since `address(this).balance >= value`, the function wraps the victim's ETH and forwards WETH to the pool. The attacker receives the full swap output at zero cost.

**Existing guard is insufficient:** [4](#0-3) 

The `receive()` guard prevents arbitrary ETH injection via direct transfer, but the victim's ETH entered legitimately through the `payable` function's `msg.value`, bypassing `receive()` entirely. The guard is irrelevant to this attack vector.

## Impact Explanation
Direct loss of user principal. The victim loses the entire ETH surplus they sent (e.g., `amountInMaximum − amountIn` for an exact-output swap). The attacker receives a fully subsidized swap output. The loss is bounded only by the victim's `amountInMaximum` and is realized in a single subsequent transaction with no special privileges required. This meets the Sherlock threshold for a High/Medium direct loss of user principal.

## Likelihood Explanation
`exactOutputSingle` with native ETH input is a standard UX pattern where users send a safe upper bound and expect change back. Any user who calls `exactOutputSingle` directly without wrapping in a `multicall` that also calls `refundETH()` creates the residue. An attacker can monitor the mempool for such transactions and immediately follow with a zero-cost `exactInputSingle`. No privileged role, special token, or pool manipulation is required.

## Recommendation
Track the ETH contributed by the current payer in the transient callback context. When setting the callback context in `exactInputSingle` / `exactOutputSingle`, store `msg.value` alongside the payer address. In `pay`, cap the native ETH used to `min(nativeBalance, storedMsgValue)` rather than the router's full `address(this).balance`. Alternatively, enforce that `msg.value` is consumed exactly (require `msg.value == amountIn` for WETH-input swaps) and refund any excess atomically before returning.

## Proof of Concept
1. Victim calls `exactOutputSingle` with `tokenIn = WETH`, `amountInMaximum = 1 ETH`, `msg.value = 1 ETH`. Actual `amountIn = 0.8 ETH`. The `pay` callback deposits 0.8 ETH; 0.2 ETH remains on the router.
2. Attacker observes the stranded 0.2 ETH on the router (on-chain or via mempool).
3. Attacker calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 0.2 ETH`, `msg.value = 0`.
4. The swap callback fires; `pay` sees `address(this).balance = 0.2 ETH >= value = 0.2 ETH`, deposits the victim's ETH, and transfers 0.2 WETH to the pool.
5. Attacker receives the full swap output. Victim's 0.2 ETH is gone.

Foundry test outline:
```solidity

// 2. victim.exactOutputSingle{value: 1 ether}(..., amountInMaximum: 1 ether, amountOut: X)
// 3. assertEq(address(router).balance, 0.2 ether); // stranded

// 5. assertEq(address(router).balance, 0); // drained
// 6. assertGt(outputToken.balanceOf(attacker), 0); // attacker profited
```

```text
  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
```
