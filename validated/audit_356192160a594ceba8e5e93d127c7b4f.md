### Title
Non-Standard ERC20 Tokens Without Boolean Return Values Permanently Lock User Funds in StarkNet Bridge - (File: starknet/src/omni_bridge.cairo)

### Summary

The StarkNet `OmniBridge` contract uses `IERC20Dispatcher` to call `transfer` and `transfer_from` on non-bridge tokens, unconditionally expecting a `bool` return value. Old-standard StarkNet ERC20 tokens (e.g., the legacy ETH and STRK token implementations) return `()` (unit type) instead of `bool`. When such a token is involved in a cross-chain transfer, `fin_transfer` will always revert on StarkNet while the user's funds on the source chain have already been burned or locked, causing a permanent, irrecoverable loss.

### Finding Description

In `starknet/src/omni_bridge.cairo`, the `fin_transfer` function handles non-bridge tokens by calling:

```cairo
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');
``` [1](#0-0) 

Similarly, `init_transfer` calls:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
``` [2](#0-1) 

`IERC20Dispatcher` is the OpenZeppelin Cairo dispatcher, which expects `transfer` and `transfer_from` to return a `bool`. In Cairo, if the callee returns `()` (unit/empty return), the dispatcher's ABI decoder will fail to deserialize the return data as `bool` and the call will panic, reverting the entire transaction.

The old StarkNet ERC20 standard (predating SNIP-2/OpenZeppelin v0.7+) does not return a boolean from `transfer` or `transfer_from`. The bridge's own `log_metadata` function already acknowledges this dual-standard reality by explicitly branching on whether `name`/`symbol` return a `felt252` (old) or `ByteArray` (new):

```cairo
let name = if res.len() == 1 {
    // Old standard (felt252)
    ...
} else {
    // New standard (ByteArray)
    ...
};
``` [3](#0-2) 

Yet no equivalent dual-standard handling exists for `transfer` or `transfer_from`. The EVM counterpart correctly uses OpenZeppelin's `SafeERC20` (which handles missing return values via low-level calls), but the StarkNet side has no such protection. [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of user funds.**

The cross-chain flow for a native StarkNet token (one that originated on StarkNet, was bridged to NEAR, and is now being bridged back) is:

1. User burns/locks the NEAR-side representation of the token on NEAR.
2. The MPC network signs a `TransferMessagePayload`.
3. User calls `fin_transfer` on StarkNet to receive the original token.

If the StarkNet token uses the old ERC20 standard (no `bool` return), step 3 will always revert. Because the NEAR-side burn is already committed and irreversible, the user's funds are permanently unclaimable. The `destination_nonce` is not permanently consumed (the revert undoes `_set_transfer_finalised`), but every retry of `fin_transfer` will revert identically — the funds are effectively frozen forever with no recovery path. [5](#0-4) 

### Likelihood Explanation

**Medium.** StarkNet has a well-known split between old-standard tokens (ETH, early STRK deployments, and many community tokens deployed before SNIP-2) and new-standard tokens. The bridge's `log_metadata` already handles this split for metadata fields, confirming the developers are aware old-standard tokens exist and may be registered. Any user who bridges an old-standard StarkNet token to NEAR and attempts to bridge it back will trigger this permanent lock. No special privileges are required — any unprivileged bridge user performing a standard cross-chain transfer is the attacker-controlled entry path.

### Recommendation

Mirror the dual-standard approach already used in `log_metadata`. Use a low-level `call_contract_syscall` for `transfer` and `transfer_from`, then treat a non-reverting call with an empty return as success (analogous to `SafeERC20`'s behavior):

```cairo
// Instead of IERC20Dispatcher.transfer(...), use:
let mut calldata: Array<felt252> = array![];
(payload.recipient, payload.amount).serialize(ref calldata);
let result = syscalls::call_contract_syscall(
    payload.token_address, selector!("transfer"), calldata.span(),
).unwrap_syscall();

// Accept both: empty return (old standard) and bool true (new standard)
if result.len() > 0 {
    let success = *result[0]; // felt252 representation of bool
    assert(success != 0, 'ERR_TRANSFER_FAILED');
}
```

This makes the bridge compatible with both old-standard tokens (no return value) and new-standard tokens (return `bool`), matching the pattern already applied to `name`/`symbol` deserialization.

### Proof of Concept

1. Deploy an old-standard StarkNet ERC20 token `OldToken` whose `transfer` returns `()`.
2. Register `OldToken` with the bridge via `log_metadata` (succeeds because `log_metadata` handles old-standard metadata).
3. Bridge 100 `OldToken` from StarkNet to NEAR: user calls `init_transfer` — this also reverts for `OldToken` (same dispatcher issue at line 304), so the user cannot even initiate the bridge. However, if the token was bridged before the current contract version, or if the token is a wrapped version whose `transfer_from` does return bool but `transfer` does not, the asymmetry creates a one-way trap.
4. Alternatively: a token whose `transfer_from` returns `bool` (allowing `init_transfer` to succeed) but whose `transfer` returns `()` (causing `fin_transfer` to always revert) creates the permanent lock: funds are burned on NEAR, `fin_transfer` on StarkNet always reverts, funds are permanently frozen.
