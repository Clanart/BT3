Audit Report

## Title
Untracked Native ETH Balance in `PeripheryPayments.pay()` Allows Any Caller to Drain Stranded ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses the router's entire `address(this).balance` when wrapping ETH for WETH payments, with no per-depositor accounting. When `exactInputSingle` is called with `msg.value` and a price limit that causes a partial swap, the unconsumed ETH remains on the router unattributed. Any subsequent caller swapping WETH with `msg.value = 0` can have their payment fully or partially covered by the stranded ETH, causing direct loss of the original depositor's principal.

## Finding Description
In `PeripheryPayments.pay()`, when `token == WETH` and `payer != address(this)`, the function reads the contract's total native ETH balance and uses it to wrap-and-transfer before pulling any remainder from the payer's WETH allowance:

```solidity
uint256 nativeBalance = address(this).balance;   // entire contract balance, no per-user cap
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [1](#0-0) 

The transient callback context (`TransientCallbackPool`) stores pool address, callback mode, payer, and token-to-pay, but records no ETH amount contributed by the current `msg.value`. There is no per-depositor ETH cap enforced anywhere in the payment path. [2](#0-1) 

`exactInputSingle` is `payable` and passes a user-supplied `priceLimitX64` to the pool. If the pool hits the price limit before consuming all of `amountIn`, the callback fires with only the consumed delta. The `_justPayCallback` then calls `pay()` with only the consumed amount, leaving the surplus ETH on the router. Critically, unlike `exactInput` (which reverts with `InvalidInputAmountAtHop` if the full input is not consumed), `exactInputSingle` has no such guard — it only checks `amountOut < params.amountOutMinimum`. [3](#0-2) [4](#0-3) 

The `refundETH()` function returns the entire contract balance to `msg.sender`, but it is not called automatically after `exactInputSingle`. A user who calls `exactInputSingle` directly (not via `multicall`) or omits `refundETH()` from their batch leaves the surplus ETH stranded with no owner record. [5](#0-4) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only applies to the fallback; it does not prevent ETH from being sent via `msg.value` on `payable` entry points. [6](#0-5) 

## Impact Explanation
Direct loss of user ETH principal. The stranded ETH is silently wrapped and transferred to a pool on behalf of an unrelated caller. The loss equals the ETH surplus left after a partial WETH swap via `exactInputSingle`. There is no cap on the amount; a user who sends a large `msg.value` with a tight price limit can lose the entire surplus in a single subsequent block. This meets the Critical/High direct loss of user principal threshold under the contest rules.

## Likelihood Explanation
Medium. The precondition — ETH stranded on the router — occurs whenever a user calls `exactInputSingle` with `tokenIn = WETH`, `msg.value > 0`, and a `priceLimitX64` that causes the pool to stop before consuming all of `amountIn`, without a subsequent `refundETH()` call. Once ETH is stranded, exploitation is trivial and permissionless: any caller can observe `address(router).balance > 0` and call `exactInputSingle` with `tokenIn = WETH`, `amountIn ≤ balance`, and `msg.value = 0`. No special role or setup is required.

## Recommendation
Track each depositor's ETH contribution in a dedicated transient storage slot at the start of each `payable` entry point (e.g., store `msg.value` in a `T_MSG_VALUE_SLOT`). In `pay()`, cap the native ETH consumed to `min(nativeBalance, recordedMsgValue)` rather than the full `address(this).balance`. Alternatively, assert after each swap that `address(this).balance` has not decreased beyond what the current call's `msg.value` permits, and revert if excess ETH from a prior depositor would be consumed. A simpler mitigation is to automatically call `refundETH()` at the end of every `payable` swap entry point, though this does not prevent the consumption within the same transaction.

## Proof of Concept
```
1. User A calls exactInputSingle({
       tokenIn: WETH,
       amountIn: 1e18,
       priceLimitX64: <tight limit>,
       amountOutMinimum: 0,
       recipient: userA,
       ...
   }) with msg.value = 1e18.

2. Pool hits the price limit after consuming 0.5e18.
   metricOmmSwapCallback fires with value = 0.5e18.
   _justPayCallback -> pay() reads nativeBalance = 1e18 >= value = 0.5e18.
   Wraps 0.5e18 ETH -> sends to pool.
   Remaining 0.5e18 ETH stays on the router.

3. exactInputSingle returns (amountOut >= 0 passes the minimum check).
   No refundETH() called. router.balance == 0.5e18.

4. User B (attacker) calls exactInputSingle({
       tokenIn: WETH,
       amountIn: 0.5e18,
       amountOutMinimum: 0,
       recipient: userB,
       ...
   }) with msg.value = 0.

5. metricOmmSwapCallback fires with value = 0.5e18.
   pay() reads nativeBalance = 0.5e18 >= value = 0.5e18.
   Wraps User A's 0.5e18 ETH -> sends to pool on behalf of User B.

Result: User B receives 0.5e18 WETH worth of output for free.
        User A permanently loses 0.5e18 ETH.
```
