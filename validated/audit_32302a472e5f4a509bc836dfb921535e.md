### Title
EIP-712 `Tx` Type Definition Omits `timeout_height`, Allowing Signature-Valid Transaction Expiry Bypass — (`ethereum/eip712/types.go`)

### Summary
The EIP-712 `Tx` type schema hardcoded in `ethereum/eip712/types.go` deliberately omits the `timeout_height` field. However, the `txBytes` JSON blob passed into both the legacy and non-legacy `WrapTxToTypedData` paths is produced by `legacytx.StdSignBytes`, which **does** include `timeout_height`. Because EIP-712's `hashStruct` only encodes fields present in the type definition, `timeout_height` is silently excluded from the signed hash. An attacker who intercepts a signed EIP-712 Cosmos transaction can freely mutate `timeout_height` — stripping a user-set expiry — without invalidating the signature.

---

### Finding Description

**Root cause — type definition omits `timeout_height`:**

In `ethereum/eip712/types.go`, the hardcoded `Tx` type schema is:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
},
``` [1](#0-0) 

**`txBytes` always includes `timeout_height`:**

In `ante/cosmos/eip712.go`, `VerifySignature` constructs `txBytes` via `legacytx.StdSignBytes`, explicitly passing `tx.GetTimeoutHeight()`:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← included in JSON
    legacytx.StdFee{Amount: tx.GetFee(), Gas: tx.GetGas()},
    msgs, tx.GetMemo(),
)
``` [2](#0-1) 

This `txBytes` JSON (containing `"timeout_height": "..."`) is then passed to `eip712.LegacyWrapTxToTypedData`: [3](#0-2) 

**Same gap in the non-legacy Amino path:**

`ethereum/eip712/encoding.go`'s `decodeAminoSignDoc` also calls `WrapTxToTypedData(chainID.Uint64(), signDocBytes)` where `signDocBytes` is the full Amino JSON including `timeout_height`: [4](#0-3) 

`createEIP712MessagePayload` parses the full JSON into the `message` map (so `timeout_height` appears in `TypedData.Message`), but `createEIP712Types` only emits the five hardcoded `Tx` fields — `timeout_height` is never added to the type schema: [5](#0-4) 

Per EIP-712, `hashStruct` iterates only over fields declared in the type definition. Any field present in the message but absent from the type is **silently ignored** during hashing. Therefore `timeout_height` is not covered by the signature.

**Contrast: the Protobuf path is protected:**

`decodeProtobufSignDoc` explicitly rejects any transaction where `body.TimeoutHeight != 0`, so the Protobuf EIP-712 path is not affected: [6](#0-5) 

The Amino path has no equivalent guard.

---

### Impact Explanation

A user who signs an EIP-712 Cosmos transaction with `timeout_height = N` intends the transaction to be rejected by the chain if it is not included before block `N`. Because `timeout_height` is excluded from the EIP-712 hash, an attacker who obtains the signed transaction bytes can:

1. Modify `timeout_height` from `N` to `0` (no expiry).
2. Submit the modified transaction after block `N`.
3. The signature verification in `VerifySignature` succeeds — the EIP-712 hash is identical regardless of `timeout_height`.
4. The Cosmos SDK ante handler sees `timeout_height = 0` and does not reject the transaction.
5. The transaction executes with the user's full authorization (correct nonce, chain-id, fee, messages).

This allows execution of a transaction the user believed had expired — for example, a token transfer or contract call the user attempted to cancel by letting the timeout lapse. This is an EIP-712 authorization bypass enabling replay/forged execution of a transaction outside the user's intended validity window.

---

### Likelihood Explanation

- The legacy EIP-712 path (`LegacyEip712SigVerificationDecorator`) is still wired into the ante handler chain and reachable by any unprivileged user submitting a Cosmos tx with a `Web3Tx` extension.
- The non-legacy Amino EIP-712 path (`GetEIP712TypedDataForMsg` → `decodeAminoSignDoc`) is the primary path used by the `ethsecp256k1` public key's `VerifySignature`.
- Any wallet or dApp that sets `timeout_height` on EIP-712 Cosmos transactions (a common pattern for UX-level tx cancellation) produces transactions vulnerable to this mutation.
- The attacker only needs to observe a pending/mempool transaction and resubmit it with `timeout_height` zeroed after the original expiry block passes.

---

### Recommendation

Add `timeout_height` back to the `Tx` type definition in `ethereum/eip712/types.go`:

```go
"Tx": {
    {Name: "account_number",  Type: "string"},
    {Name: "chain_id",        Type: "string"},
    {Name: "fee",             Type: "Fee"},
    {Name: "memo",            Type: "string"},
    {Name: "sequence",        Type: "string"},
    {Name: "timeout_height",  Type: "string"},  // ← restore coverage
},
```

Alternatively, mirror the Protobuf path's approach and reject Amino sign docs that carry a non-zero `timeout_height` until the field is properly covered by the type hash. Either way, the `message` and the `types` must agree on every security-relevant field.

---

### Proof of Concept

1. Construct a Cosmos EIP-712 tx (Amino-encoded) with `timeout_height = 100` and any fund-transferring message (e.g., `MsgSend`).
2. Sign it with an `ethsecp256k1` key via MetaMask or equivalent — the wallet signs the EIP-712 hash, which does **not** include `timeout_height`.
3. Wait until block 101 (past the intended expiry).
4. Intercept the signed tx bytes; set `timeout_height` to `"0"` in the Amino JSON.
5. Recompute `txBytes` with the modified `timeout_height = 0` and re-wrap into a new `TxRaw` (the `FeePayerSig` in `ExtensionOptionsWeb3Tx` is unchanged).
6. Broadcast the modified tx. `VerifySignature` reconstructs `txBytes` using `tx.GetTimeoutHeight()` (now 0), calls `LegacyWrapTxToTypedData`, and computes the same EIP-712 hash as the original — signature verification passes.
7. The Cosmos SDK ante handler sees `timeout_height = 0` and does not apply any expiry check; the transaction executes.

	}

	payload, numPayloadMsgs, err := FlattenPayloadMessages(basicPayload)

		return eip712MessagePayload{}, errorsmod.Wrap(err, "failed to flatten payload JSON messages")
	}

	message, ok := payload.Value().(map[string]interface{})
	if !ok {
		return eip712MessagePayload{}, errorsmod.Wrap(errortypes.ErrInvalidType, "failed to parse JSON as map")
	}

	messagePayload := eip712MessagePayload{
		payload:        payload,
		numPayloadMsgs: numPayloadMsgs,
		message:        message,
	}
