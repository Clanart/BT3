### Title
Unbounded `payload_size` in `execute_send_message_to_l1` Enables Prover OOM/Halt via Fixed-Cost Syscall — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_send_message_to_l1` OS syscall handler charges a **fixed** gas cost (`SEND_MESSAGE_TO_L1_GAS_COST = 14470`) regardless of message payload size, then calls `memcpy` with the fully attacker-controlled `payload_size`. An unprivileged contract can set `payload_end` to an arbitrarily large value, forcing the StarkNet OS prover to execute O(payload\_size) Cairo steps and allocate O(payload\_size) memory cells — with negligible gas cost — causing the prover to OOM or hang and preventing block finalization.

---

### Finding Description

In `execute_send_message_to_l1` (lines 1346–1374 of `syscall_impls.cairo`):

```cairo
// Gas deducted: fixed SEND_MESSAGE_TO_L1_GAS_COST = 14470, regardless of payload size.
let success = reduce_syscall_gas_and_write_response_header(
    total_gas_cost=SEND_MESSAGE_TO_L1_GAS_COST, request_struct_size=SendMessageToL1Request.SIZE
);
...
tempvar payload_start = request.payload_start;
tempvar payload_size = request.payload_end - payload_start;   // ← no bounds check

memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE, src=payload_start, len=payload_size
);
``` [1](#0-0) 

`payload_size` is derived from two user-supplied pointers with **no `assert_nn_le` or any upper-bound check**. The gas deduction (`SEND_MESSAGE_TO_L1_GAS_COST = 14470`) is fixed and does not scale with `payload_size`. [2](#0-1) 

Contrast this with `execute_meta_tx_v0`, where the developers **explicitly** added a Sierra-array-length bound on the signature:

```cairo
assert_nn_le(request.signature_end - request.signature_start, SIERRA_ARRAY_LEN_BOUND - 1);
``` [3](#0-2) 

No equivalent guard exists for `payload_size` in `execute_send_message_to_l1`. The same unbounded `memcpy` pattern also appears in the deprecated syscall path:

```cairo
memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE,
    src=syscall.payload_ptr,
    len=syscall.payload_size,   // ← no bounds check
);
``` [4](#0-3) 

Additionally, `SIERRA_ARRAY_LEN_BOUND = 4294967296` (2³²) is itself a very large value, meaning even the bounded paths allow enormous inputs. [5](#0-4) 

---

### Impact Explanation

The StarkNet OS Cairo program is executed by the **prover** to generate STARK proofs for blocks. Every Cairo step in the OS execution must be traced and proven. `memcpy` in Cairo is a recursive loop — each iteration is one Cairo step. With `payload_size = 2³²` (~4 billion):

- **Step count**: ~4 × 10⁹ Cairo steps for `memcpy` alone.
- **Memory**: Each felt is 32 bytes → 4 × 10⁹ × 32 = **128 GB** of Cairo memory cells written to the OS output segment.

This causes the prover to exhaust physical memory or exceed its time budget, making it unable to produce a proof for the block. Without a proof, the block cannot be finalized and **no new transactions can be confirmed** — a total network shutdown matching the High impact criterion.

---

### Likelihood Explanation

- **Entry path**: Any unprivileged contract can call `send_message_to_l1`. No special role or permission is required.
- **Attacker cost**: The gas cost is fixed at 14470 regardless of payload size. Setting `payload_end = payload_start + 2³²` costs the same as a 1-element payload.
- **No mitigation in scope**: The OS program contains no bounds check on `payload_size`, and the gas model does not scale with it (unlike `execute_keccak`, which correctly charges `KECCAK_GAS_COST + q * KECCAK_ROUND_COST_GAS_COST` proportional to input length). [6](#0-5) 

---

### Recommendation

1. Add an explicit upper-bound check on `payload_size` before calling `memcpy`, analogous to the check in `execute_meta_tx_v0`:
   ```cairo
   assert_nn_le(payload_size, MAX_L1_MESSAGE_PAYLOAD_SIZE);
   ```
2. Make the gas cost scale with `payload_size`, similar to how `execute_keccak` charges per-block and `execute_deploy` charges per calldata element (`DEPLOY_CALLDATA_FACTOR_GAS_COST * constructor_calldata_size`). [7](#0-6)
