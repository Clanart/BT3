Audit Report

## Title
Unrestricted `sweepToken`, `unwrapWETH9`, and `refundETH` Allow Any Caller to Drain Router-Held User Funds — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.refundETH`, `sweepToken`, and `unwrapWETH9` are public/external with no access control and no caller-identity check. The `pay()` function deliberately leaves excess native ETH on the router when `msg.value > amountIn` in the WETH-input path, and the standard WETH-output-to-router pattern leaves WETH on the router between a swap and an unwrap step. Any unprivileged third party can immediately call these helpers to redirect the full router balance to an arbitrary address, causing direct loss of user principal.

## Finding Description
Three balance-draining helpers carry zero caller restrictions:

`refundETH()` (L58–63) sends `address(this).balance` to `msg.sender` unconditionally.
`sweepToken()` (L48–55) transfers `IERC20(token).balanceOf(address(this))` to a caller-supplied `recipient`.
`unwrapWETH9()` (L37–45) withdraws the router's full WETH balance and forwards it to a caller-supplied `recipient`.

**Path A — excess native ETH stranded by `pay()`.**
Inside `pay()` (L73–77), when `token == WETH` and `nativeBalance >= value`, exactly `value` ETH is wrapped and forwarded; the remainder stays on the router:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}   // nativeBalance - value ETH is now stranded
```

`exactInputSingle` is `external payable` (L67), so a user can call it directly with `msg.value > amountIn`. The callback fires `_justPayCallback → pay(WETH, alice, pool, amountIn)`, which wraps only `amountIn` ETH; the surplus stays on the router. Any third party then calls `refundETH()` and receives the entire surplus as `msg.sender`.

**Path B — WETH stranded from swap-output-to-router pattern.**
The standard unwrap flow (confirmed in test `test_multicall_tokenForWeth_thenUnwrapEth`, L135–162) routes swap output to `address(router)` then calls `unwrapWETH9` in the same multicall. If a user splits these into two separate transactions, WETH sits on the router between them. Any third party front-runs the second transaction with `sweepToken(WETH, 0, attacker)` or `unwrapWETH9(0, attacker)` and claims the full swap output.

No existing guard prevents either path: `receive()` (L32–34) only blocks direct ETH sends from non-WETH addresses but does not prevent excess ETH from accumulating via `exactInputSingle{value: X}`. There is no per-depositor accounting, no transient-storage beneficiary record, and no multicall-context enforcement on any of the three helpers.

## Impact Explanation
Direct loss of user principal. In Path A the stolen amount equals `msg.value − amountIn`, which can be the user's entire ETH balance sent to the router. In Path B the stolen amount equals the full swap output routed to `address(router)`. Both losses are immediate, irreversible, and can reach any magnitude the user trades. This meets Sherlock High/Critical thresholds for direct loss of user funds.

## Likelihood Explanation
The attack is fully unprivileged and requires no special setup beyond monitoring the mempool. MEV bots routinely scan for stranded router balances. `exactInputSingle` and `exactOutputSingle` are `external payable`, implying direct ETH-input use is expected, yet the router provides no automatic refund. Any user who calls these functions directly (not via multicall) with excess ETH, or who splits a WETH-output swap and unwrap into two transactions, is immediately exploitable. The attack is repeatable against every such user.

## Recommendation
1. Replace the free-floating `refundETH` with a version that records the depositor address in transient storage at swap entry and enforces it in the cleanup call, so only the originating caller can claim the residue.
2. Remove the arbitrary `recipient` parameter from `sweepToken` and `unwrapWETH9`, replacing it with the transient-storage-recorded beneficiary set at swap entry.
3. Alternatively, auto-refund excess native ETH at the end of each top-level swap function rather than relying on the user to batch a cleanup call.

## Proof of Concept
**Scenario 1 — ETH theft via `refundETH`:**
1. Alice calls `router.exactInputSingle{value: 2 ether}(ExactInputSingleParams{ tokenIn: WETH, amountIn: 1 ether, … })` directly (no multicall).
2. Pool callback fires → `_justPayCallback` → `pay(WETH, alice, pool, 1 ether)`: `nativeBalance = 2 ether ≥ value = 1 ether`; wraps 1 ETH → WETH → pool; **1 ETH remains on router**.
3. Bob calls `router.refundETH()` in a separate transaction.
4. `refundETH` sends `address(this).balance = 1 ether` to `msg.sender` (Bob).
5. Alice loses 1 ETH with no recourse.

**Scenario 2 — WETH theft via `sweepToken`:**
1. Alice calls `router.exactInputSingle(ExactInputSingleParams{ tokenIn: token1, tokenOut: WETH, recipient: address(router), … })` — WETH swap output lands on the router.
2. Alice intends to call `router.unwrapWETH9(0, alice)` in a follow-up transaction.
3. Bob front-runs Alice's second transaction with `router.sweepToken(WETH, 0, bob)`.
4. `sweepToken` transfers the entire WETH balance to Bob.
5. Alice's full swap output is stolen.
