### Title
`deprecated_declared_classes` Silently Dropped from `ThinStateDiff` in Commitment Calculation, Producing Wrong State-Diff Commitment and Block Hash - (`File: crates/blockifier/src/state/cached_state.rs`)

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally sets `deprecated_declared_classes: Vec::new()`. This `ThinStateDiff` is the sole input to `calculate_block_commitments` inside `BlockExecutionArtifacts::new`. Because `ThinStateDiff::len()` counts `deprecated_declared_classes`, any block that contains a Cairo 0 (deprecated) class declaration will have a wrong `state_diff_length`, wrong `concatenated_counts`, and a wrong `state_diff_commitment` baked into `partial_block_hash_components`, ultimately producing a wrong block hash.

### Finding Description

**Root cause — the silent drop:** [1](#0-0) 

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` has no field for deprecated declared classes; they are tracked in `StateMaps::declared_contracts` during execution but are dropped when `StateMaps` is converted to `CommitmentStateDiff`. [2](#0-1) 

**Where the wrong `ThinStateDiff` feeds commitment calculation:** [3](#0-2) 

`calculate_block_commitments` receives this `ThinStateDiff` and calls `state_diff.len()` for both `concatenated_counts` and `state_diff_length`: [4](#0-3) 

`ThinStateDiff::len()` explicitly counts `deprecated_declared_classes`: [5](#0-4) 

`calculate_state_diff_hash` is also called on this same `ThinStateDiff`: [6](#0-5) 

The resulting `BlockHeaderCommitments` (containing the wrong `state_diff_commitment` and wrong `concatenated_counts`) are stored in `partial_block_hash_components`: [7](#0-6) 

The final block hash is derived from these components: [8](#0-7) 

**The same wrong `ThinStateDiff` is also written to batcher storage** via `thin_state_diff()`, so the stored state diff is also missing deprecated declared classes: [9](#0-8) 

**What the correct path looks like** — `ThinStateDiff::from_state_diff` properly populates `deprecated_declared_classes` from `StateDiff`, but this path is never taken during block building: [10](#0-9) 

### Impact Explanation

For any block containing at least one Cairo 0 (deprecated) class declaration:

1. `state_diff_commitment` is computed over a `ThinStateDiff` with empty `deprecated_declared_classes` — the hash is wrong.
2. `concatenated_counts` encodes a `state_diff_length` that is smaller than the true length by the number of deprecated declarations — the block header field is wrong.
3. `partial_block_hash_components` carries both wrong values, so `calculate_block_hash` produces a wrong block hash.
4. The `ThinStateDiff` written to batcher storage is also missing the deprecated declared classes, so `append_state_diff` never updates `deprecated_declared_classes_block_table` for those classes.

This matches the **Critical** impact category: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input* — specifically a wrong authoritative block hash and wrong state diff commitment for accepted blocks.

### Likelihood Explanation

Cairo 0 (deprecated) class declarations are still valid Starknet transactions. Any unprivileged user can submit a `DeclareV1` transaction. Once included in a block, the bug fires deterministically. No special permissions or adversarial peer are required.

### Recommendation

`CommitmentStateDiff` must be extended to carry `deprecated_declared_classes`, populated from `StateMaps::declared_contracts` during the `From<StateMaps>` conversion. The `From<CommitmentStateDiff> for ThinStateDiff` implementation should then map that field instead of hardcoding `Vec::new()`. The TODO comment at the conversion site confirms this is a known gap.

### Proof of Concept

1. Submit a `DeclareV1` (Cairo 0) transaction through the gateway. It is accepted and enters the mempool.
2. The batcher includes it in a block. `BlockExecutionArtifacts::new` is called; `ThinStateDiff::from(commitment_state_diff)` produces a `ThinStateDiff` with `deprecated_declared_classes = []`.
3. `calculate_block_commitments` computes `state_diff.len()` = N (missing the 1 deprecated declaration), so `concatenated_counts` encodes N instead of N+1, and `calculate_state_diff_hash` hashes a diff that omits the declared class hash.
4. `partial_block_hash_components` stores the wrong `state_diff_commitment` and wrong `concatenated_counts`.
5. `calculate_block_hash` produces a block hash that differs from what any correct verifier (prover, L1 contract, syncing node) would compute from the actual block contents.
6. The stored `ThinStateDiff` also has empty `deprecated_declared_classes`, so `deprecated_declared_classes_block_table` is never updated — the class is invisible to state queries.
