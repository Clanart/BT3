### Title
Excess ETH Forwarded to Wormhole in `finTransfer`, `deployToken`, and `logMetadata` Is Permanently Lost — (`evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole` overrides the extension hooks of `OmniBridge` to publish Wormhole messages. In `finTransferExtension`, `deployTokenExtension`, and `logMetadataExtension`, the entire `msg.value` is forwarded verbatim to `_wormhole.publishMessage{value: msg.value}(...)`. The real Wormhole core contract only requires `msg.value >= messageFee()` and does **not** refund any excess. Any ETH sent above the Wormhole fee is permanently transferred to the Wormhole fee-collector and is irrecoverable by the caller or the bridge.

---

### Finding Description

`OmniBridgeWormhole` inherits from `OmniBridge`, which declares `finTransfer`, `deployToken`, and `logMetadata` as `payable` entry points. The Wormhole-specific overrides of the internal extension hooks forward `msg.value` directly:

**`finTransferExtension`** (lines 96–116):
```solidity
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    messagePayload,
    _consistencyLevel
);
```

**`deployTokenExtension`** (lines 48–70):
```solidity
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    payload,
    _consistencyLevel
);
```

**`logMetadataExtension`** (lines 72–94):
```solidity
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    payload,
    _consistencyLevel
);
```

None of these functions enforce `msg.value == _wormhole.messageFee()`. The test stub (`TestWormhole.sol`) does enforce exact equality (`require(msg.value == this.messageFee(), "invalid fee")`), which masks the issue in tests. The production Wormhole core contract accepts any `msg.value >= messageFee()` and retains the surplus in its fee-collector account — it does not refund excess ETH to the caller.

For `initTransferExtension` (lines 118–150), the forwarded value is `extensionValue = msg.value - nativeFee` (ERC20) or `msg.value - amount - nativeFee` (ETH). If the user sends more than `nativeFee + wormhole.messageFee()` (ERC20) or `amount + nativeFee + wormhole.messageFee()` (ETH), the surplus is similarly forwarded to Wormhole and lost.

There is no post-call refund path in any of these functions, and the contract's `receive()` fallback does not help because the ETH never returns to the bridge contract.

---

### Impact Explanation

Any caller of `finTransfer`, `deployToken`, or `logMetadata` who sends `msg.value` exceeding the current Wormhole `messageFee()` permanently loses the difference. The ETH is transferred to the Wormhole fee-collector, which is controlled by Wormhole governance — not the bridge or the caller. There is no admin rescue path in `OmniBridgeWormhole` to recover ETH that has already been forwarded to Wormhole. This constitutes irrecoverable misdirection of user/relayer value, matching the "High — misdirects value" impact class.

---

### Likelihood Explanation

Several realistic scenarios cause over-payment:

1. **Fee-change race**: A caller queries `_wormhole.messageFee()` off-chain, then the Wormhole guardian set votes to raise the fee before the transaction lands. The caller's `msg.value` is now above the new fee — but the new fee is still satisfied, so the call succeeds and the excess is silently consumed.
2. **Front-end rounding**: UIs that add a small buffer to the Wormhole fee to avoid reverts will routinely over-pay.
3. **Programmatic integrators**: Contracts or scripts that pass `msg.value` as a user-supplied parameter without enforcing exact equality will silently drain excess ETH on every call.

The `finTransfer` path is particularly exposed because it is callable by any address (relayers, users, MEV bots) and is the hot path for every inbound cross-chain settlement.

---

### Recommendation

After calling `_wormhole.publishMessage`, compute the actual fee consumed and refund the remainder to `msg.sender`:

```solidity
uint256 fee = _wormhole.messageFee();
_wormhole.publishMessage{value: fee}(wormholeNonce, payload, _consistencyLevel);
uint256 excess = msg.value - fee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "ETH refund failed");
}
```

Apply this pattern in `finTransferExtension`, `deployTokenExtension`, `logMetadataExtension`, and `initTransferExtension` (where the forwarded value is `value` rather than `msg.value`).

---

### Proof of Concept

1. Wormhole `messageFee()` is currently `0.001 ETH` (10 000 wei in the test stub).
2. A relayer calls `finTransfer(sig, payload, {value: 0.01 ETH})` — 10× the required fee — to ensure the transaction does not revert if the fee ticks up slightly.
3. `finTransferExtension` executes `_wormhole.publishMessage{value: 0.01 ETH}(...)`.
4. The real Wormhole core accepts the call (fee satisfied) and retains all `0.01 ETH` in its fee-collector.
5. The relayer's `0.009 ETH` surplus is permanently lost; the bridge contract never held it and has no mechanism to recover it.

        uint128 nativeFee,
