### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` accepts any arbitrary class hash supplied by a contract's `replace_class` syscall without verifying that the hash corresponds to a previously declared contract class. This is the direct StarkNet analog of the XSS report's root cause: **unsanitized user-controlled input embedded into a protocol output without validation**. Here, an attacker-controlled class hash is written into the committed protocol state with no sanitization, permanently corrupting the contract's class pointer and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the new class hash is read directly from the syscall request and written into `contract_state_changes` with no check that the hash was ever declared:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is an explicit in-code acknowledgement that this validation is absent. [2](#0-1) 

The only post-execution class validation that exists is `validate_compiled_class_facts_post_execution`, which only checks classes that were **actually executed** during the block — it does not verify that every class hash stored in the state corresponds to a declared class. [3](#0-2) 

By contrast, `execute_declare_transaction` correctly enforces that a declared class hash is the authentic Poseidon hash of the Sierra class components: [4](#0-3) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value and the block is committed:

1. The OS state permanently records the invalid class hash for that contract address.
2. Any future transaction that calls the contract causes the OS to look up the compiled class for the invalid hash. Because no such class exists in `compiled_class_facts`, the OS execution fails.
3. The contract becomes permanently uncallable.
4. All funds (tokens, ETH) held in the contract's storage are permanently inaccessible — there is no recovery path, because fixing the class hash requires calling the contract, which itself fails.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack requires only an unprivileged contract deployer:

1. Attacker deploys a contract (e.g., a vault, a fake token, or a shared pool) that accepts deposits from other users.
2. After users deposit funds, the attacker triggers the contract to issue a `replace_class` syscall with an arbitrary undeclared hash (e.g., `felt 0x1`).
3. `execute_replace_class` writes the invalid hash into `contract_state_changes` without any validation.
4. The state is committed. The contract is now permanently broken.

No privileged access, leaked keys, or external dependency compromise is required. The entry point is a standard `replace_class` syscall issued by a contract the attacker controls.

---

### Recommendation

In `execute_replace_class`, before updating `contract_state_changes`, verify that `request.class_hash` exists as a key in `contract_class_changes` (declared in the current block) or is present in the committed state's class trie (declared in a prior block). This mirrors the validation already enforced in `execute_declare_transaction` and closes the gap identified by the existing TODO comment.

---

### Proof of Concept

1. Attacker deploys a vault contract that accepts ERC-20 deposits from users.
2. Users deposit funds; the vault's storage now holds balances.
3. Attacker calls a function on the vault that internally issues:
   ```
   replace_class(class_hash=0x1)  // 0x1 is never declared
   ```
4. The OS executes `execute_replace_class`. `request.class_hash = 0x1` is written directly into `contract_state_changes` with no validation. [5](#0-4) 
5. `validate_compiled_class_facts_post_execution` does not catch this because `0x1` was never executed, only stored. [3](#0-2) 
6. The block is proven and committed. The vault's class hash in the state is now `0x1`.
7. Any subsequent transaction targeting the vault fails at the OS level — the compiled class for `0x1` does not exist.
8. All deposited funds are permanently frozen with no recovery mechanism.
