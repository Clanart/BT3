### Title
Unprotected Bootstrap Declare Path Allows Unauthorized Class Hash Registration Without Signature Verification - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS contains a special "bootstrap" code path that completely bypasses signature verification, nonce enforcement, and fee payment. The guard condition relies solely on the user-supplied `sender_address` field equaling the felt literal `'BOOTSTRAP'`. Because no cryptographic check ties this field to any authorized key, any unprivileged transaction sender can craft a declare transaction that satisfies all four conditions and register an arbitrary class hash with an arbitrary compiled class hash, without owning or controlling any account.

---

### Finding Description

Inside `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the OS checks:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the OS:
- Skips `check_and_increment_nonce` (nonce at `'BOOTSTRAP'` is never incremented, so `nonce == 0` is always satisfiable again)
- Skips `run_validate` (no `__validate_declare__` entry point is called, so no signature is verified)
- Skips `charge_fee` (no fee is deducted)
- Directly writes `compiled_class_hash` into `contract_class_changes` [2](#0-1) 

The four conditions an attacker must satisfy are all user-controlled fields in a declare transaction:

| Field | Required value | Attacker control |
|---|---|---|
| `sender_address` | `'BOOTSTRAP'` (felt `0x424f4f545354524150`) | Yes — arbitrary felt |
| `nonce` | `0` | Yes — arbitrary felt |
| `version` | `3` | Yes — arbitrary felt |
| all resource bounds | `0` (so `max_possible_fee == 0`) | Yes — arbitrary |

`'BOOTSTRAP'` is not a special privileged address enforced by any cryptographic primitive; it is simply a felt literal. The normal declare path calls `run_validate` which invokes the account contract's `__validate_declare__` entry point to verify the sender's signature: [3](#0-2) 

The bootstrap path skips this entirely. Because `check_and_increment_nonce` is also skipped, the nonce at address `'BOOTSTRAP'` is never written, so the condition `nonce == 0` remains satisfiable for every subsequent bootstrap declare transaction. An attacker can therefore register an unlimited number of class hashes. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `dict_update` call uses `prev_value=0`, meaning a class hash can only be registered once. An attacker who observes a pending legitimate class declaration (e.g., a new account contract or token contract class) can front-run it by submitting a bootstrap declare transaction for the same `class_hash` but with an attacker-controlled `compiled_class_hash`. The effects are:

1. The legitimate declaration fails at the OS level because `prev_value=0` is violated (the slot is already occupied by the attacker's value).
2. Any contract subsequently deployed under that `class_hash` executes the attacker's malicious compiled class instead of the intended one.
3. A malicious compiled class can implement arbitrary logic — including draining all funds from the deployed contract or from callers.

During a network bootstrapping phase, system-level class hashes (fee token, account contracts) are declared for the first time. Front-running these declarations with malicious compiled class hashes would cause system contracts to execute attacker-controlled code, enabling direct theft of all funds held by those contracts.

---

### Likelihood Explanation

**High.** The conditions are trivially constructable by any user who can submit a transaction to the network. No privileged key, operator cooperation, or external dependency is required. The only prerequisite is the ability to observe pending class declarations (public mempool) and submit a transaction before them — a standard front-running attack. The bootstrap path is explicitly intended for use during network initialization, which is precisely the highest-value window for this attack.

---

### Recommendation

Replace the felt-literal identity check with a cryptographically enforced access control mechanism. Options include:

1. **Remove the bootstrap path entirely** and require all class declarations to go through the standard `__validate_declare__` flow, using a pre-deployed bootstrap account whose key is controlled by the deployer.
2. **Use the TypeScript/template system** (analogous to the suggestion in the external report) to inject a hardcoded authorized address constant at compile time, and additionally enforce that the transaction carries a valid signature from that address — i.e., do not skip `run_validate` even for the bootstrap sender.
3. **Enforce a one-time flag in state**: write a storage slot the first time the bootstrap path is used and assert it is unset before allowing subsequent bootstrap declares, preventing replay even if the nonce is not incremented.

---

### Proof of Concept

An attacker constructs a v3 declare transaction with the following fields:

```
sender_address  = 0x424f4f545354524150   // felt encoding of 'BOOTSTRAP'
nonce           = 0
version         = 3
resource_bounds = [L1_GAS(0,0), L2_GAS(0,0), L1_DATA_GAS(0,0)]  // max_fee == 0
class_hash      = <target class hash being front-run>
compiled_class_hash = <attacker-controlled malicious CASM hash>
```

The OS evaluates the branch at line 764:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
``` [5](#0-4) 

All conditions are true. `compute_max_possible_fee` returns 0. The OS executes:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

The class hash is now permanently associated with the attacker's malicious compiled class. The legitimate declarer's subsequent transaction fails. Any contract deployed under this class hash executes the attacker's code, enabling direct fund theft.

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
```
