### Title
Malicious Recipient Can Remove Finalization Record via `ft_on_transfer` Rejection, Enabling Cross-Chain Transfer Replay — (`near/omni-bridge/src/lib.rs`, `near/omni-token/src/lib.rs`)

---

### Summary

When a cross-chain transfer to NEAR includes a non-empty `msg` field, the bridge mints tokens and delivers them via `ft_transfer_call`, which invokes `ft_on_transfer` on the recipient contract. If the recipient rejects the transfer (returns the full refund amount), `fin_transfer_send_tokens_callback` removes the finalization record from `finalised_transfers`. This allows the same proof to be submitted again, enabling double-finalization and double-spend of bridged assets.

---

### Finding Description

**Step 1 — Finalization marks the nonce used, then calls `send_tokens`:**

In `process_fin_transfer_to_near`, the transfer ID is inserted into `finalised_transfers` via `add_fin_transfer` before any external call is made. This is the sole replay guard. [1](#0-0) 

Then `send_tokens` is called, which — for a deployed (bridged) token with a non-empty `msg` — calls `mint` on the token contract: [2](#0-1) 

**Step 2 — `mint` with `msg` deposits to bridge then calls `ft_transfer_call`:**

In `OmniToken::mint`, when `msg` is `Some`, the tokens are first deposited to `env::predecessor_account_id()` (the bridge contract), then forwarded to the recipient via `ft_transfer_call`: [3](#0-2) 

`ft_transfer_call` invokes `ft_on_transfer` on the recipient contract. Per NEP-141, if the recipient returns the full amount as refund, `ft_resolve_transfer` returns `0` (zero tokens used).

**Step 3 — Callback removes the finalization record on rejection:**

`fin_transfer_send_tokens_callback` checks `is_refund_required`, which returns `true` when `ft_resolve_transfer` returned `0`: [4](#0-3) 

When refund is required, the callback **removes the finalization record**: [5](#0-4) 

`remove_fin_transfer` deletes the transfer ID from `finalised_transfers`. Since `add_fin_transfer` is the only replay guard and it panics only if the ID is already present, removing the record makes the same proof replayable. [6](#0-5) 

**Step 4 — Replay is now possible:**

The same proof can be submitted again via `fin_transfer`. `add_fin_transfer` succeeds (record was removed), tokens are minted again, and if the recipient accepts this time, they receive a second payout backed by the same single cross-chain lock/burn event.

---

### Impact Explanation

**Critical.** A malicious recipient can cause the bridge to remove the finalization record for any `msg`-bearing transfer targeting a NEAR contract. The attacker then replays the same proof to receive a second mint of bridged tokens. The original source-chain lock/burn event is consumed only once, but the NEAR side mints twice — breaking bridge collateralization and enabling direct unauthorized minting of bridged assets.

For native (non-deployed) tokens with a `msg`, the same path applies: `ft_transfer_call` is used, the refund path re-locks tokens and removes the finalization record, and the same proof can be replayed to transfer native tokens a second time.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A cross-chain transfer with a non-empty `msg` field — this is a standard, documented feature of the bridge protocol.
2. The recipient is a NEAR contract controlled by the attacker that implements `ft_on_transfer` to return the full refund amount.
3. The attacker (or any relayer) resubmits the same proof after the finalization record is removed.

No privileged access, leaked keys, or colluding operators are required. The attacker controls the recipient contract and the proof submission. The `msg` field is user-supplied and unrestricted.

---

### Recommendation

Do not remove the finalization record on `ft_on_transfer` rejection. The finalization record is the sole replay guard; once a proof is verified and the transfer is finalized, the record must be permanent.

Instead, on rejection, keep the finalization record intact and either:
- Burn the minted tokens (for deployed tokens) and emit `FailedFinTransferEvent` without removing the record, or
- Require the recipient to be a registered, trusted contract before using `ft_transfer_call` with `msg`.

The `burn_tokens_if_needed` call is already present in the refund path but is `.detach()`-ed (fire-and-forget). Even if the burn succeeds, the removal of the finalization record is the root cause and must be eliminated independently. [7](#0-6) 

---

### Proof of Concept

1. Attacker deploys a malicious NEAR contract `evil.near` implementing `ft_on_transfer` that always returns the full `amount` (refund all).
2. Attacker initiates a transfer from EVM to NEAR: `recipient = evil.near`, `msg = "trigger"`, `amount = 1000`.
3. Relayer submits proof to `fin_transfer`. `process_fin_transfer_to_near` runs:
   - `add_fin_transfer` inserts `(Eth, nonce=N)` into `finalised_transfers`. [1](#0-0) 
   - `send_tokens` → `mint(evil.near, 1000, Some("trigger"))` → bridge gets 1000 tokens, `ft_transfer_call(evil.near, 1000, "trigger")` fires. [3](#0-2) 
4. `evil.near::ft_on_transfer` returns `U128(1000)` (refund all). `ft_resolve_transfer` returns `U128(0)`.
5. `fin_transfer_send_tokens_callback` sees `amount.0 == 0`, calls `remove_fin_transfer((Eth, N))`. [8](#0-7) 
6. Attacker resubmits the same proof. `add_fin_transfer` succeeds (record gone). `mint(evil.near, 1000, Some("trigger"))` fires again.
7. This time `evil.near::ft_on_transfer` returns `U128(0)` (accept all). Attacker receives 1000 tokens.
8. Net result: 1000 tokens minted twice from a single source-chain lock event. Bridge is undercollateralized by 1000 tokens.

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
