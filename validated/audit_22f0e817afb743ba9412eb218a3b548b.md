Audit Report

## Title
Stranded ETH in Router Consumed for Subsequent User's WETH Payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay` function's WETH hybrid branch reads `address(this).balance` — the router's total ETH balance — with no per-user accounting. ETH legitimately stranded in the router by a prior `multicall{value: X}` caller (who omitted `refundETH`) is silently wrapped and forwarded to the pool on behalf of a subsequent WETH-swap user, causing the prior depositor to permanently lose their ETH while the subsequent user's WETH `transferFrom` is skipped entirely.

## Finding Description

`PeripheryPayments.pay` contains a three-branch WETH hybrid at lines 73–84: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` (the router's **total** native ETH) and uses it to wrap WETH before pulling any remainder from `payer`. There is no mechanism to distinguish which caller deposited the ETH.

ETH accumulates in the router because `multicall` is `payable`: [2](#0-1) 

The `receive()` guard only blocks **plain** ETH transfers (no calldata); ETH sent as part of a `multicall{value: X}(...)` call bypasses it entirely: [3](#0-2) 

So a user who calls `multicall{value: 1 ether}([...])` without including `refundETH` leaves ETH stranded. The next user who calls `exactInputSingle` with `tokenIn=WETH` triggers `_justPayCallback`: [4](#0-3) 

This calls `pay(WETH, userB, pool, value)`. If `address(this).balance >= value`, the branch at line 75–77 executes: it wraps the stranded ETH and transfers WETH to the pool — `safeTransferFrom(userB, ...)` is **never called**. [5](#0-4) 

## Impact Explanation

User A loses their ETH permanently — it is wrapped and transferred to the pool on behalf of User B. User B's WETH balance is untouched (receives a free or subsidized swap). This is a direct loss of user principal with no recovery path, meeting the Critical/High threshold for direct loss of funds.

## Likelihood Explanation

The precondition (stranded ETH in the router) is a common operational pattern in Uniswap-style routers: users frequently call `multicall{value: X}` for native-ETH swaps and omit `refundETH`. An opportunistic MEV bot can monitor the mempool for such transactions and immediately follow with `exactInputSingle({tokenIn: WETH, amountIn: X})` to drain the stranded ETH. No privileged access, no malicious pool, and no non-standard token behavior is required.

## Recommendation

Track per-user ETH deposits using transient storage keyed to `msg.sender` (EIP-1153), and restrict the WETH hybrid branch in `pay` to wrap only up to the amount the current caller deposited in the same transaction. Alternatively, restrict the hybrid branch to only activate when `msg.value > 0` in the current call context (passed explicitly through the call chain), and require that `msg.value` exactly covers the native portion.

## Proof of Concept

1. User A calls `router.multicall{value: 1 ether}([swapCallData])` where the inner swap does not consume ETH, and omits `refundETH`. `address(router).balance == 1 ether`.
2. User B calls `router.exactInputSingle({tokenIn: WETH, amountIn: 1e18, ...})` with a valid WETH allowance.
3. The pool calls `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, userB, pool, 1e18)`.
4. `nativeBalance = 1 ether >= 1e18` → branch at line 75 executes: wraps 1 ETH, transfers WETH to pool. `safeTransferFrom(userB, ...)` is never called.
5. User A's 1 ETH is gone. User B's WETH balance is untouched.
