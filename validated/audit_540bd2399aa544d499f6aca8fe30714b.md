### Title
StarkNet `init_transfer` Accepts Unprefixed Recipient Strings That Permanently Lock User Funds — (`starknet/src/omni_bridge.cairo`, `near/omni-types/src/starknet/events.rs`, `near/omni-types/src/lib.rs`)

---

### Summary

The StarkNet `init_transfer` function accepts any `ByteArray` as `recipient` with no format validation. If a user supplies a recipient string without the required `chain:address` prefix (e.g., `"alice.near"` instead of `"near:alice.near"`), the contract burns/locks the user's tokens, emits the event, and the transfer becomes permanently unclaimable: the NEAR-side proof parser will fail to parse the recipient, so NEAR never settles the transfer and there is no refund path.

---

### Finding Description

**Step 1 — StarkNet `init_transfer` has no recipient format guard.**

`starknet/src/omni_bridge.cairo:281-331` accepts `recipient: ByteArray` with only two checks: `amount > 0` and `fee < amount`. No validation of the `chain:address` format is performed. [1](#0-0) 

**Step 2 — Tokens are irreversibly consumed before the event is emitted.**

For bridge-deployed tokens, `burn` is called at line 302; for native tokens, `transfer_from` to the contract is called at line 305. Both happen before the `InitTransfer` event is emitted. There is no refund branch in `init_transfer`. [2](#0-1) 

**Step 3 — `parse_init_transfer` propagates a parse error as a hard failure.**

`near/omni-types/src/starknet/events.rs:61` calls `recipient_str.parse().map_err(stringify)?`. If parsing fails, the entire proof submission returns `Err`, and NEAR never finalises the transfer. [3](#0-2) 

**Step 4 — `OmniAddress::from_str` silently defaults to `eth` chain when no `:` separator is present, then fails on non-hex input.**

`near/omni-types/src/lib.rs:393` uses `split_once(':').unwrap_or(("eth", input))`. A string like `"alice.near"` is treated as `chain="eth"`, `address="alice.near"`, and `H160::from_str("alice.near")` returns `Err(InvalidHex)`. [4](#0-3) 

**Step 5 — No recovery mechanism exists.**

There is no admin-callable refund, no re-submission path with a corrected recipient, and no way to override a failed proof on NEAR. The nonce is consumed on StarkNet and the tokens are gone.

---

### Impact Explanation

Any user who calls `init_transfer` with a recipient string lacking the `chain:` prefix (e.g., `"alice.near"`, `"0xabc..."` without `"eth:"`, or any unsupported chain prefix) will have their tokens permanently burned or locked on StarkNet with no settlement on NEAR. This is an irrecoverable loss of user funds matching the Critical impact category: **permanent freezing / unclaimable settlement of user funds in bridge flows**.

---

### Likelihood Explanation

- The StarkNet contract interface is fully public; any user can call `init_transfer` with an arbitrary `ByteArray` recipient.
- The `chain:address` format is non-obvious and not enforced at the contract level. The test suite itself passes `"recipient.near"` (no prefix) as a recipient to `init_transfer` without triggering any contract-level error, confirming the contract accepts it. [5](#0-4) 
- Users bridging to NEAR are the most likely to make this mistake (writing `alice.near` instead of `near:alice.near`).
- No off-chain validation is enforced at the contract layer; a frontend bug or direct contract call is sufficient to trigger the loss.

---

### Recommendation

Validate the recipient format inside `init_transfer` on StarkNet before consuming tokens. The contract should assert that the `recipient` ByteArray contains a `:` separator and that the prefix matches a known chain identifier. Alternatively, the NEAR-side `parse_init_transfer` could be made more lenient for known-safe fallbacks, but the correct fix is to reject malformed recipients at the source before funds are moved.

---

### Proof of Concept

1. User calls `init_transfer(token_address, 1000, 10, 0, "alice.near", "")` on the StarkNet bridge.
2. Contract passes `amount > 0` and `fee < amount` checks.
3. `burn(caller, 1000)` executes — tokens are gone.
4. `InitTransfer` event is emitted with `recipient = "alice.near"`.
5. Relayer submits the proof to NEAR; `parse_init_transfer` is called.
6. `cursor.read_byte_array()` returns `"alice.near"` (valid UTF-8).
7. `"alice.near".parse::<OmniAddress>()` → `split_once(':')` returns `None` → defaults to `("eth", "alice.near")` → `H160::from_str("alice.near")` → `Err(InvalidHex)`.
8. `parse_init_transfer` returns `Err`; proof submission fails.
9. NEAR never settles; user's 1000 tokens are permanently lost.
