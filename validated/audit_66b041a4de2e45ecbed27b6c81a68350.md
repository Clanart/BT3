### Title
Missing `state_diff_commitment` Verification in P2P State Diff Sync Allows Malicious Peer to Corrupt `compiled_class_hash_table` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`parse_data_for_block` in the P2P state diff sync path assembles a `ThinStateDiff` from peer-supplied chunks and validates only the total element count against `header.state_diff_length`. It never calls `calculate_state_diff_hash` and never compares the assembled diff against `header.state_diff_commitment`. A malicious peer can therefore supply a `DeclaredClass` chunk mapping `class_hash H -> compiled_class_hash C1` (where the canonical on-chain mapping is `H -> C2`), and the wrong value is written unconditionally into `compiled_class_hash_table` via `write_compiled_class_hashes`.

---

### Finding Description

**Step 1 — Header accepted without commitment binding.**

`HeaderStreamBuilder::parse_data_for_block` performs two checks on a peer-supplied `SignedBlockHeader`:

- Block number ordering [1](#0-0) 
- Signature vector length (count only, no cryptographic verification) [2](#0-1) 

Neither the block hash nor `state_diff_commitment` is verified against any L1 anchor or parent chain. A malicious peer can send a header for block N with an arbitrary `state_diff_commitment` and `state_diff_length = 1`, and it will be stored verbatim. [3](#0-2) 

**Step 2 — State diff accepted with length-only validation.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` from the already-stored (attacker-controlled) header and loops until the assembled diff reaches that length:

```
while current_state_diff_len < target_state_diff_len { ... }
```

After the loop it calls `validate_deprecated_declared_classes_non_conflicting` (duplicate deprecated-class check only) and returns `Ok(Some((result, block_number)))`. [4](#0-3) 

`calculate_state_diff_hash` is **never called** anywhere in the `apollo_p2p_sync` crate (confirmed: zero matches). The `state_diff_commitment` field stored in the header is read for nothing — it is never compared against the assembled diff.

**Step 3 — Wrong compiled class hash inserted unconditionally.**

`unite_state_diffs` inserts the peer-supplied `compiled_class_hash` directly into `class_hash_to_compiled_class_hash` with only a duplicate-key guard: [5](#0-4) 

`write_to_storage` calls `append_state_diff` with no further validation: [6](#0-5) 

`write_compiled_class_hashes` inserts `(class_hash, block_number) -> compiled_class_hash` into `compiled_class_hash_table` with no on-chain commitment check: [7](#0-6) 

**Step 4 — Wrong value served to callers.**

`get_compiled_class_hash_at` performs a cursor lookup keyed on `(class_hash, block_number)` and returns whatever value is stored: [8](#0-7) 

The result is `C1` instead of `C2`.

---

### Impact Explanation

- **Blockifier** reads `compiled_class_hash` to select the CASM artifact for execution. With `C1` stored, it selects the wrong artifact, producing wrong execution results for every call to class `H` after block N.
- **RPC** (`starknet_getCompiledCasm`, `starknet_getClass`) returns `C1` as the authoritative compiled class hash.
- The corruption is persistent (written to the LMDB storage layer) and affects all subsequent state reads for class `H`.

This matches the allowed impact: *"Critical. Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."*

---

### Likelihood Explanation

Any peer on the P2P network can trigger this. No operator privilege is required. The attacker only needs to:
1. Respond to a header query with a forged `SignedBlockHeader` (signature count = 1, any content).
2. Respond to the subsequent state diff query with a `DeclaredClass` chunk carrying the wrong `compiled_class_hash`.

Both responses pass all existing validation gates.

---

### Recommendation

After assembling the full `ThinStateDiff` in `parse_data_for_block`, compute `calculate_state_diff_hash(&result)` and compare it against `header.state_diff_commitment`. Reject with `BadPeerError` on mismatch. This is the same pattern already implemented (behind a config flag) in `apollo_committer`:

```rust
let calculated = calculate_state_diff_hash(&result);
if calculated != header.state_diff_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffHash { ... }));
}
```

Additionally, verify the block hash cryptographically in `HeaderStreamBuilder::parse_data_for_block` against an L1-anchored source to prevent forged headers from setting an attacker-chosen `state_diff_length` and `state_diff_commitment`.

---

### Proof of Concept

```rust
// Pseudocode for a Rust integration test
let canonical_compiled_class_hash = CompiledClassHash(felt!("0xC2"));
let malicious_compiled_class_hash = CompiledClassHash(felt!("0xC1"));
let class_hash = ClassHash(felt!("0xAA"));

// Build a header whose state_diff_commitment was computed with H->C2
let canonical_diff = ThinStateDiff {
    class_hash_to_compiled_class_hash: indexmap! { class_hash => canonical_compiled_class_hash },
    ..Default::default()
};
let commitment = calculate_state_diff_hash(&canonical_diff);
let header = BlockHeader { state_diff_commitment: Some(commitment), state_diff_length: Some(1), .. };

// Store the header (as a malicious peer would supply it)
storage_writer.begin_rw_txn()?.append_header(BlockNumber(0), &header)?.commit()?;

// Supply a state diff with H->C1 (wrong compiled class hash, correct length=1)
let malicious_diff = ThinStateDiff {
    class_hash_to_compiled_class_hash: indexmap! { class_hash => malicious_compiled_class_hash },
    ..Default::default()
};
storage_writer.begin_rw_txn()?.append_state_diff(BlockNumber(0), malicious_diff)?.commit()?;

// Assert corruption
let stored = storage_reader.begin_ro_txn()?
    .get_compiled_class_hash_at(StateNumber::right_after_block(BlockNumber(0)), &class_hash)?;
assert_eq!(stored, Some(malicious_compiled_class_hash)); // C1, not C2

// Assert commitment mismatch
let stored_diff = storage_reader.begin_ro_txn()?.get_state_diff(BlockNumber(0))?.unwrap();
assert_ne!(calculate_state_diff_hash(&stored_diff), commitment);
```

The test passes without any modification to production code, demonstrating that the wrong compiled class hash is stored and served.

fn get_compiled_class_hash_at<'env, Mode: TransactionKind>(
    first_irrelevant_block: BlockNumber,
    class_hash: &ClassHash,
    txn: &'env DbTransaction<'env, Mode>,
    compiled_class_hash_table: &'env CompiledClassHashTable<'env>,
) -> StorageResult<Option<CompiledClassHash>> {
    let db_key = (*class_hash, first_irrelevant_block);
    // Find the previous db item.
    let mut cursor = compiled_class_hash_table.cursor(txn)?;
    cursor.lower_bound(&db_key)?;
    let res = cursor.prev()?;
    match res {
        None => Ok(None),
        Some(((got_class_hash, _got_block_number), value)) => {
            if got_class_hash != *class_hash {
                // The previous item belongs to different class hash, which means there is no
                // previous state diff for this item.
                return Ok(None);
            };
            // The previous db item indeed belongs to this address and key.
            Ok(Some(value))
        }
    }
