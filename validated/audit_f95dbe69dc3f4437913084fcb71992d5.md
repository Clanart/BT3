Audit Report

## Title
Unattributed ETH residue on router is claimable by any caller via `refundETH()` ŌĆö (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
When a user calls `exactOutputSingle` (or `exactInputSingle`) with `msg.value` exceeding the actual swap cost, `pay()` deposits only the required `amountIn` as WETH and leaves the surplus native ETH on the router with no record of who deposited it. Because `refundETH()` unconditionally forwards the entire ETH balance to `msg.sender`, any third party can call it in a subsequent transaction to steal that surplus.

## Finding Description
**Root cause 1 ŌĆö `pay()` leaves ETH residue** (`PeripheryPayments.sol` L73-78):
When `token == WETH` and `nativeBalance >= value`, the function wraps exactly `value` wei and transfers it to the pool. The remaining `nativeBalance - value` native ETH is silently left on the router with no per-depositor accounting.

**Root cause 2 ŌĆö `refundETH()` is unrestricted** (`PeripheryPayments.sol` L58-63):
`refundETH()` is `external payable` with no access control. It reads `address(this).balance` and forwards the entire amount to `msg.sender` ŌĆö any caller, not the original depositor.

**Trigger path:**
1. Victim calls `exactOutputSingle{value: 2 ether}(...)` with `amountInMaximum: 2 ether`.
2. The pool determines `amountIn = 1 ether`; the swap callback fires `_justPayCallback` ŌåÆ `pay(WETH, victim, pool, 1 ether)`.
3. `pay()` deposits 1 ether as WETH and transfers it to the pool. The remaining 1 ether stays on the router.
4. `exactOutputSingle` returns without issuing any refund (L130-147 ŌĆö no `refundETH` call).
5. Attacker calls `router.refundETH()` in the next transaction; `address(this).balance == 1 ether` is forwarded to the attacker.

**Existing guards are insufficient:**
- `receive()` (L32-34) only blocks unsolicited ETH pushes from non-WETH addresses; it does not prevent `msg.value` from accumulating inside payable swap functions.
- `amountInMaximum` check (L145) only reverts if the pool charges *more* than the cap; it does not refund the difference when the pool charges *less*.
- `multicall` (L39-44) supports bundling a `refundETH()` call, but nothing enforces that users do so, and the router is stateless between calls.

## Impact Explanation
Direct loss of user principal: any ETH surplus left after an `exactOutputSingle` or `exactInputSingle` call is immediately claimable by an unprivileged attacker. For a user who sets `amountInMaximum` at 2├Ś the expected cost (a common slippage buffer), up to 50 % of committed ETH is stolen. This exceeds the Critical threshold (>20 % loss, >$100 at any realistic ETH price).

## Likelihood Explanation
- `exactOutputSingle` is the canonical "I want exactly X output tokens" flow; users routinely set a generous `amountInMaximum` to avoid slippage reverts.
- No privileged access is required ŌĆö a single `refundETH()` call from any EOA suffices.
- The attack is MEV-bot-friendly: the attacker can observe the victim's swap in the mempool and submit `refundETH()` in the immediately following block (or sandwich it in the same block via bundle).
- The condition recurs on every direct (non-multicall) invocation of any payable swap function with `tokenIn == WETH`.

## Recommendation
1. **Auto-refund inside swap functions**: after `pay()` returns, call `_transferETH(msg.sender, address(this).balance)` (guarded by `balance > 0`) at the end of `exactInputSingle`, `exactOutputSingle`, `exactInput`, and `exactOutput`.
2. **Restrict `refundETH` to the depositor**: track `msg.value` credited to `msg.sender` at each payable entry point and allow only that address to reclaim its share.
3. **Remove caller-chosen `recipient` from `sweepToken` / `unwrapWETH9`**, or gate them so only the address that deposited in the same multicall context can invoke them.

## Proof of Concept
```solidity
// 1. Victim transaction
router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
    pool:            address(wethPool),
    tokenIn:         WETH,
    tokenOut:        token1,
    zeroForOne:      true,
    amountOut:       1_000,
    amountInMaximum: 2 ether,
    recipient:       victim,
    deadline:        block.timestamp + 60,
    priceLimitX64:   0,
    extensionData:   ""
}));
// Pool charges amountIn = 1 ether.
// pay() deposits 1 ether as WETH ŌåÆ pool.
// 1 ether remains on router, unattributed.
// exactOutputSingle returns ŌĆö no refund issued.

// 2. Attacker transaction (next block or same block via MEV)
// address(router).balance == 1 ether
router.refundETH();   // attacker receives 1 ether; victim lost 1 ether
```

Foundry test plan: deploy router with mock WETH and mock pool that returns `amountIn < msg.value`; assert attacker's balance increases by the surplus after calling `refundETH()`, and victim's balance is not restored.
