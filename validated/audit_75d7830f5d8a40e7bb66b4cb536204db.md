Audit Report

## Title
Router `pay()` WETH Branch Silently Consumes Stranded Native ETH, Enabling Free Swaps at Prior Users' Expense — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` uses `address(this).balance` as the ETH source for WETH swaps without any per-caller accounting. ETH sent as `msg.value` to a `payable` entry point (e.g., `exactInputSingle`) but not consumed by the swap remains on the router. Any subsequent WETH-input swap will silently consume that stranded ETH, giving the next caller a free trade at the prior user's expense.

## Finding Description
`PeripheryPayments.pay()` (lines 73–84) handles WETH payments by reading the raw contract ETH balance:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
}
```

The transient storage context set by `_setNextCallbackContext()` records only the pool, callback mode, payer address, and token — it does not record how much ETH the current top-level call contributed via `msg.value`. There is therefore no mechanism to distinguish "ETH deposited by the current caller" from "ETH left by a prior caller."

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) does not prevent ETH from accumulating: `payable` entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) accept `msg.value` directly without routing through `receive()`. If a user sends `msg.value > amountIn` and omits the trailing `refundETH()` call, the surplus ETH sits on the router indefinitely.

Exploit path:
1. Alice calls `exactInputSingle{value: 1000}(amountIn=500, tokenIn=WETH, ...)`. The callback fires `_justPayCallback` → `pay(WETH, Alice, pool, 500)`. `nativeBalance=1000 >= 500`, so 500 ETH is wrapped and sent; 500 ETH remains on the router. Alice does not call `refundETH()`.
2. Bob calls `exactInputSingle{value: 0}(amountIn=500, tokenIn=WETH, ...)`. The callback fires `pay(WETH, Bob, pool, 500)`. `nativeBalance=500 >= 500`, so Alice's remaining 500 ETH is wrapped and forwarded to the pool. Bob receives the full swap output; his `payer` address is never debited.

No existing guard prevents this: the `receive()` check is irrelevant to `msg.value` accumulation, and the transient callback context carries no ETH-credit field.

## Impact Explanation
Direct loss of user principal. Any ETH stranded on the router is permanently transferred to a pool as payment for an arbitrary caller's trade. The victim loses their ETH; the attacker receives full swap output at zero cost. This satisfies the "Critical/High direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The precondition — ETH left on the router — arises naturally when a user sends `msg.value` larger than `amountIn` (common defensive practice to avoid partial-fill reverts) or assembles a `multicall` that omits the trailing `refundETH`. Once ETH is stranded, any observer can exploit it in the very next block with a zero-cost WETH swap requiring no special privileges.

## Recommendation
**Short term:** In the `nativeBalance >= value` branch, verify that the ETH credit attributable to the current call covers `value`. The simplest approach is to store `msg.value` in transient storage at each payable entry point and deduct from that per-call credit rather than from the raw `address(this).balance`.

**Long term:** Enforce that `address(this).balance == 0` at the start of every non-payable entry point, or add an invariant check that reverts when the router holds ETH at callback time and the current call's `msg.value` is zero.

## Proof of Concept
```
Block N:
  Alice calls exactInputSingle{value: 1000}(
      tokenIn=WETH, amountIn=500, recipient=Alice, ...
  )
  → metricOmmSwapCallback → _justPayCallback
  → pay(WETH, Alice, pool, 500)
      nativeBalance=1000 >= 500
      router wraps 500 wei → sends WETH to pool
      500 wei remains on router
  Alice omits refundETH().

Block N+1:
  Bob calls exactInputSingle{value: 0}(
      tokenIn=WETH, amountIn=500, recipient=Bob, ...
  )
  → metricOmmSwapCallback → _justPayCallback
  → pay(WETH, Bob, pool, 500)
      nativeBalance=500 >= 500
      router wraps Alice's 500 wei → sends WETH to pool
      Bob receives TOKEN output; pays nothing
```

Foundry test: deploy router + mock WETH pool; call `exactInputSingle{value:1000}(amountIn:500)` from Alice without `refundETH`; assert `address(router).balance == 500`; call `exactInputSingle{value:0}(amountIn:500)` from Bob; assert Bob received swap output and `address(router).balance == 0` while Alice's balance decreased by 1000 (not 500).
