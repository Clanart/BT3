### Title
`pauseAll()` Blocks `finTransfer` Alongside `initTransfer`, Freezing In-Flight User Funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
The `pauseAll()` function in `OmniBridge.sol` (EVM), `pause()` in Solana's `bridge_token_factory`, and `pause_all()` in StarkNet's `omni_bridge.cairo` each set a combined bitmask that simultaneously blocks both the outbound transfer (`initTransfer`) and the inbound settlement (`finTransfer`). When `finTransfer` is blocked, users who have already locked or burned tokens on the source chain and are waiting for destination-chain finalization have no recourse: they cannot receive their tokens, and no cancel/refund path exists while the bridge is paused. This is the direct analog of the reported "liquidations should not be paused" class: the protective settlement action (`finTransfer`) is blocked alongside the initiating action (`initTransfer`), leaving in-flight user funds frozen.

---

### Finding Description

**EVM — `OmniBridge.sol`**

`pauseAll()` is callable by any account holding `PAUSABLE_ADMIN_ROLE` and always sets all three flags at once:

```solidity
function pauseAll() external onlyRole(PAUSABLE_ADMIN_ROLE) {
    uint256 flags = PAUSED_FIN_TRANSFER |
        PAUSED_INIT_TRANSFER |
        PAUSED_DEPLOY_TOKEN;
    _pause(flags);
}
``` [1](#0-0) 

`finTransfer` carries `whenNotPaused(PAUSED_FIN_TRANSFER)`:

```solidity
function finTransfer(

``` [2](#0-1) 

`initTransfer` carries `whenNotPaused(PAUSED_INIT_TRANSFER)`: [3](#0-2) 

The `SelectivePausableUpgradable` base contract confirms that `_pause(flags)` **replaces** the entire flag word, so calling `pauseAll()` atomically blocks both directions: [4](#0-3) 

**Solana — `bridge_token_factory/src/lib.rs`**

`pause()` sets `ALL_PAUSED = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED`: [5](#0-4) [6](#0-5) 

Both `finalize_transfer` and `finalize_transfer_sol` check `FINALIZE_TRANSFER_PAUSED` and revert when set: [7](#0-6) [8](#0-7) 

**StarkNet — `omni_bridge.cairo`**

`pause_all()` sets `PAUSE_ALL = 0xFF`, blocking both `fin_transfer` and `init_transfer`: [9](#0-8) [10](#0-9) 

`fin_transfer` asserts `!_is_paused(@self, PAUSE_FIN_TRANSFER)`: [11](#0-10) 

**NEAR — partial mitigation but still affects relayers**

On NEAR, `fin_transfer` uses `#[pause(except(roles(Role::DAO)))]`, meaning only DAO can finalize while paused; trusted relayers (the normal path) are blocked: [12](#0-11) 

**The missing cancel/refund path**

There is no `cancel_transfer` or `refund_transfer` function in any of the bridge contracts. `pending_transfers` on NEAR are only removed via `sign_transfer_callback` (zero-fee path) or `claim_fee_callback`, both of which are also paused: [13](#0-12) [14](#0-13)

### Impact Explanation

A user who has already locked or burned tokens on the source chain (NEAR, EVM, Solana, or StarkNet) and whose MPC-signed `finTransfer` payload is ready for submission on the destination chain is completely blocked when `pauseAll()` / `pause()` / `pause_all()` is called on the destination bridge:

- `finTransfer` / `finalize_transfer` / `fin_transfer` reverts — tokens cannot be delivered.
- No cancel or refund path exists while paused — tokens remain locked on the source chain.
- The signed payload has a fixed `recipient` and `destinationNonce`, so the user cannot reroute the transfer.

This constitutes **unclaimable settlement of user funds in bridge flows** — matching the allowed Critical/High impact class "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."

Additionally, on NEAR, `sign_transfer` accepts a caller-supplied `fee_recipient` parameter. When the bridge is unpaused after an extended pause, a front-running bot can call `sign_transfer` on all accumulated `pending_transfers` with the bot's own address as `fee_recipient`, stealing fees from the legitimate relayers who initiated those transfers — analogous to the "front-running bot liquidates before user can repay" scenario in the original report. [15](#0-14) 

---

### Likelihood Explanation

`pauseAll()` on EVM is callable by any `PAUSABLE_ADMIN_ROLE` holder (a lower-privilege role than `DEFAULT_ADMIN_ROLE`), making it a realistic emergency action. The Solana `pause()` is callable by either `admin` or `pausable_admin`. StarkNet `pause_all()` requires `PAUSER_ROLE`. All of these are expected to be exercised during security incidents. The window between a user locking tokens on the source chain and the relayer submitting `finTransfer` on the destination chain is non-zero and can span multiple blocks, making it realistic that a pause occurs mid-flight. The longer the pause, the more in-flight transfers accumulate with no recourse.

---

### Recommendation

1. **Do not include `PAUSED_FIN_TRANSFER` / `FINALIZE_TRANSFER_PAUSED` / `PAUSE_FIN_TRANSFER` in the "pause all" bitmask.** Emergency pauses should block new outbound transfers (`initTransfer`) but must allow inbound settlements (`finTransfer`) to complete, since the source-chain funds are already committed.

2. **Add a cancel/refund mechanism** for `pending_transfers` on NEAR that is callable by the original sender even while the bridge is paused, so users are not left with no recourse.

3. **On NEAR, restrict `fee_recipient` in `sign_transfer`** to the original transfer's designated relayer or the sender, to prevent fee front-running when the bridge is unpaused after an extended pause.

---

### Proof of Concept

**EVM freeze scenario:**

1. Alice calls `ft_transfer_call` on NEAR, locking 1000 USDC. The NEAR bridge records a `pending_transfer` and emits `InitTransferEvent`.
2. A relayer obtains the MPC signature via `sign_transfer` on NEAR and prepares the `finTransfer` calldata for EVM.
3. Admin (or any `PAUSABLE_ADMIN_ROLE` holder) calls `OmniBridge.pauseAll()` on EVM.
4. Relayer submits `finTransfer` → reverts with `"Pausable: paused"`.
5. Alice's 1000 USDC is locked on NEAR. She cannot receive on EVM. She cannot cancel on NEAR (all NEAR bridge functions are also paused). Her funds are frozen for the duration of the pause.

**Solana freeze scenario:**

1. Bob initiates a transfer from NEAR to Solana. Tokens are locked on NEAR.
2. Admin calls `pause()` on the Solana `bridge_token_factory` program, setting `ALL_PAUSED`.
3. Relayer calls `finalize_transfer` → fails with `ErrorCode::Paused` (custom error 6008).
4. Bob's tokens are frozen on NEAR with no recourse. [16](#0-15) 

**Fee front-running on NEAR unpausing:**

1. During a pause, 50 `pending_transfers` accumulate on NEAR (all with non-zero fees).
2. Bridge is unpaused.
3. A bot monitors the mempool and front-runs all `sign_transfer` calls, passing `fee_recipient = bot_account`.
4. Bot collects all 50 fees; legitimate relayers receive nothing. [15](#0-14)
