### Title
Cross-Chain Signature Replay in `deploy_token` — `MetadataPayload` Omits Chain ID - (Files: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `near/omni-bridge/src/lib.rs`)

---

### Summary

The `MetadataPayload` signed by the NEAR MPC for `deploy_token` does not include a destination chain ID. Because the signed bytes are identical across all chains, a valid MPC signature obtained for deploying a token on one chain can be replayed verbatim on any other chain that shares the same `nearBridgeDerivedAddress`, enabling unauthorized token deployment.

---

### Finding Description

**EVM — `OmniBridge.sol::deployToken()`**

The borsh-encoded payload that is hashed and signature-verified is constructed as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

No `omniBridgeChainId` is included. Compare this with `finTransfer()`, which explicitly embeds `omniBridgeChainId` twice in the signed payload:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
...
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

**Starknet — `bridge_types.cairo::MetadataPayload::to_borsh()`**

**NEAR — `lib.rs::log_metadata_callback()`**

The NEAR bridge constructs and signs the `MetadataPayload` without any chain-specific field:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),

);
``` [7](#0-6) 

The signed bytes are therefore chain-agnostic and identical for every chain that supports the same token.

---

### Impact Explanation

All EVM chains (Ethereum, Base, Arbitrum, Polygon, BNB) and Starknet share the same `nearBridgeDerivedAddress` / `omni_bridge_derived_address`. Because the `MetadataPayload` signature does not commit to a destination chain, a single MPC signature is valid on every one of these chains simultaneously.

An attacker who observes a valid `deploy_token` transaction on chain A can immediately replay the same `(signatureData, metadata)` call on chain B, C, D, etc. The token is deployed on those chains without the NEAR bridge having explicitly authorized deployment there. This constitutes **unauthorized token deployment** — a High-severity impact under the allowed scope ("Proof, signature, MPC … verification bypass enabling … token deployment").

Concrete downstream harms:
- A token is deployed on a chain before the NEAR bridge has registered the corresponding factory/accounting for that chain. Users who bridge to that chain may have their funds permanently locked because the NEAR hub will not finalize inbound transfers for an unregistered chain.
- The deployment order is attacker-controlled, which can race ahead of legitimate deployments and interfere with the bridge's token-mapping invariants.

---

### Likelihood Explanation

- The `deploy_token` call is fully permissionless; any address can submit it.
- A valid signature is publicly observable on-chain the moment the first legitimate `deploy_token` succeeds on any chain.
- The attacker needs zero special access: copy `signatureData` and `metadata` from chain A's calldata and submit to chain B.
- Multiple EVM chains and Starknet are all live targets sharing the same signer address.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` borsh encoding, mirroring the pattern already used in `TransferMessagePayload`:

**EVM (`OmniBridge.sol`):**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet (`bridge_types.cairo`):**
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    ...
    borsh_bytes.append_byte(chain_id);   // bind to this chain
    ...
}
```

**NEAR (`lib.rs`):** Add a `destination_chain` field to `MetadataPayload` and populate it before signing, so each chain's deployment requires a distinct MPC signature.

---

### Proof of Concept

1. NEAR MPC signs `keccak256([0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06])` for an Ethereum `deploy_token` call.
2. The relayer submits `deployToken(sig, metadata)` on Ethereum — succeeds, USDC bridge token deployed at address `T_eth`.
3. Attacker copies `sig` and `metadata` from the Ethereum transaction.
4. Attacker calls `deployToken(sig, metadata)` on the Base `OmniBridge` contract (different chain, same `nearBridgeDerivedAddress`).
5. `ECDSA.recover(keccak256(borshEncoded), sig) == nearBridgeDerivedAddress` — passes, because the payload bytes are identical.
6. `isBridgeToken[nearToEthToken["usdc.near"]]` is `false` on Base — passes.
7. USDC bridge token is deployed on Base at address `T_base` without any NEAR MPC authorization for Base.
8. If the NEAR bridge later registers Base for USDC transfers, `T_base` is the canonical token — but it was deployed by an unauthorized actor, and any users who bridged to Base before the NEAR hub was ready have their funds unclaimable.
