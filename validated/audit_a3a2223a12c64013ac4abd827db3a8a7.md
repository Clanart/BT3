### Title
Duplicate Storage Keys in Pending State Diff Cause Wrong Value Returned by RPC `starknet_getStorageAt` - (File: `crates/apollo_rpc_execution/src/execution_utils.rs`)

---

### Summary

`PendingData.storage_diffs` stores per-contract storage entries as `Vec<StorageEntry>` with no uniqueness constraint on keys. The `get_storage_at` function resolves pending storage using `.find()`, which stops at the **first** matching key. If the pending state diff received from the feeder gateway contains duplicate storage keys for the same contract, the second entry is silently unreachable, and the RPC returns an authoritative-looking wrong value.

---

### Finding Description

`PendingData` in `crates/apollo_rpc_execution/src/objects.rs` stores pending storage diffs as:

```rust
pub storage_diffs: IndexMap<ContractAddress, Vec<StorageEntry>>,
```

The `Vec<StorageEntry>` carries no uniqueness invariant on `StorageEntry.key`. The code even acknowledges this with a TODO:

```rust
// TODO(shahak): Consider indexing by address and key.
```

When `starknet_getStorageAt` is called with `block_id = pending`, the RPC handler extracts `state_diff.storage_diffs` from the pending data and passes it to `execution_utils::get_storage_at`:

```rust
if let Some(StorageEntry { key: _, value }) = storage_entries
    .iter()
    .find(|StorageEntry { key: other_key, value: _ }| key == *other_key)
{
    return Ok(*value);
}
```

`.find()` returns the **first** match and stops. If the feeder gateway sends a pending state diff where the same storage key appears twice for the same contract (e.g., two incremental updates to the same slot within a pending block), the second entry — which represents the authoritative latest value — is never reached. The RPC returns the stale first value.

This is the direct structural analog of the original bug: just as `InvoiceManager::creditTokensToInvoice()` uses a linear search with `break` that stops at the first token match, `get_storage_at` uses `.find()` that stops at the first key match, making any subsequent duplicate entry unreachable.

The same `get_storage_at` function is also called from `ExecutionStateReader::get_storage_at` (the blockifier state reader used during pending-block simulation and fee estimation), so the wrong value propagates into execution as well.

---

### Impact Explanation

Matches: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

- `starknet_getStorageAt` with `block_id = pending` returns the wrong storage value.
- `starknet_estimateFee`, `starknet_simulateTransactions`, and `starknet_call` against the pending block execute with the wrong storage value, producing wrong fee estimates, wrong traces, and wrong call results.
- The response is indistinguishable from a correct one — no error is raised.

---

### Likelihood Explanation

The feeder gateway (central sync source) can emit pending state diffs with duplicate storage keys. The `ClientStateDiff.storage_diffs` type is `IndexMap<ContractAddress, Vec<StorageEntry>>` — a `Vec` with no key-uniqueness enforcement. No deduplication or conflict check is applied when ingesting pending data from the central sync path (unlike the P2P sync path, which explicitly calls `unite_state_diffs` and returns `BadPeerError::ConflictingStateDiffParts` on duplicate keys). A feeder gateway that accumulates multiple writes to the same slot within a pending block and emits them as separate list entries would trigger this silently.

---

### Recommendation

- Change `PendingData.storage_diffs` from `IndexMap<ContractAddress, Vec<StorageEntry>>` to `IndexMap<ContractAddress, IndexMap<StorageKey, Felt>>`, enforcing key uniqueness at the type level (resolving the existing TODO).
- Alternatively, deduplicate (keeping the last occurrence) when converting `ClientStateDiff.storage_diffs` into `PendingData.storage_diffs` in `client_pending_data_to_execution_pending_data`.
- At minimum, change `.find()` to scan the entire list and return the **last** matching entry, so that the most-recently-written value wins.

---

### Proof of Concept

1. The feeder gateway emits a pending state diff with a duplicate storage key:
   ```
   storage_diffs: {
     contract_A: [
       StorageEntry { key: K, value: V1 },   // first write
       StorageEntry { key: K, value: V2 },   // second write (authoritative)
     ]
   }
   ```
2. Central sync ingests this into `PendingData.storage_diffs` without deduplication. [1](#0-0) 
3. RPC receives `starknet_getStorageAt(contract_A, K, "pending")`. The handler extracts `state_diff.storage_diffs` and passes it to `execution_utils::get_storage_at`. [2](#0-1) 
4. `get_storage_at` calls `.find()` on the `Vec<StorageEntry>`, which matches `K` at index 0 and returns `V1` immediately. [3](#0-2) 
5. The entry at index 1 (`V2`) is never examined. The RPC returns `V1` — an authoritative-looking wrong value.
6. The same wrong value is used by `ExecutionStateReader::get_storage_at` during pending-block fee estimation and simulation. [4](#0-3) 

The `PendingData.storage_diffs` field carries the TODO acknowledging the structural gap: [5](#0-4) 

The P2P sync path correctly rejects duplicate storage keys via `unite_state_diffs`, but the central sync / pending data path has no equivalent guard. [6](#0-5)
