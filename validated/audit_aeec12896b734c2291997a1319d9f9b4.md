### Title
EVM-Level Gas Refund Silently Discarded by `MinGasMultiplier` Floor Override — (`File: x/evm/keeper/state_transition.go`)

### Summary

In `ApplyMessageWithConfig`, the EVM-level gas refund (earned by clearing storage slots via SSTORE) is computed and added to `leftoverGas`, but is then silently discarded when the `MinGasMultiplier` floor overrides `gasUsed`. The final `leftoverGas` returned to `ApplyTransaction` for the actual bank refund is derived from the floored `gasUsed`, not from the post-refund `temporaryGasUsed`. The earned EVM refund is absorbed by the fee collector rather than returned to the sender.

### Finding Description

The gas accounting sequence in `ApplyMessageWithConfig` is:

**Step 1 — EVM refund computed and added:**
```go
temporaryGasUsed := msg.GasLimit - leftoverGas
refund := GasToRefund(stateDB.GetRefund(), temporaryGasUsed, refundQuotient)
leftoverGas += refund                          // EVM refund credited
temporaryGasUsed = msg.GasLimit - leftoverGas  // now reflects refund
``` [1](#0-0) 

**Step 2 — Minimum gas floor applied, `leftoverGas` reset:**
```go
gasUsed = sdkmath.LegacyMaxDec(minimumGasUsed, sdkmath.LegacyNewDec(tempGasUsed)).TruncateInt().Uint64()
leftoverGas = msg.GasLimit - gasUsed   // EVM refund silently discarded here
``` [2](#0-1) 

**Step 3 — Bank refund uses the floored value:**
```go
leftoverGas := msg.GasLimit - res.GasUsed   // res.GasUsed = floored gasUsed
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
``` [3](#0-2) 

When `minimumGasUsed > temporaryGasUsed` (post-EVM-refund), the `leftoverGas` reset at Step 2 is smaller than the value that included the EVM refund. The EVM refund that was added in Step 1 is never forwarded to the bank-level `RefundGas` call. The fee collector retains the difference.

The `RefundGasWithPrice` function sends coins from the fee collector to the sender: [4](#0-3) 

Since `leftoverGas` is under-counted, the sender receives fewer coins back than they earned.

### Impact Explanation

Any user who:
1. Submits a transaction that clears storage slots (earning EVM-level gas refunds via SSTORE opcodes), and
2. Sets a gas limit significantly higher than actual gas consumed, such that `gasLimit × MinGasMultiplier > actualGasUsed − evmRefund`

will not receive the gas refund they earned. The earned refund amount (`evmRefund × gasPrice`) remains in the fee collector instead of being returned to the sender. This is a direct mis-accounting of valid user funds.

**Concrete example** (with `MinGasMultiplier = 0.5`):
- `gasLimit = 1,000,000`, `gasPrice = 1`
- Ante handler deducts `1,000,000` from sender
- EVM execution uses `100,000` gas; SSTORE clearing earns `20,000` gas refund
- `temporaryGasUsed = 80,000`
- `minimumGasUsed = 500,000`
- `gasUsed = max(500,000, 80,000) = 500,000`
- Sender receives back `500,000` (not `920,000`)
- The `20,000`-gas EVM refund is absorbed by the floor — sender loses `20,000 × gasPrice`

### Likelihood Explanation

- `MinGasMultiplier` is a governance-controlled `feemarket` parameter, non-zero in standard deployments (default 0.5).
- SSTORE-clearing operations are routine in DeFi (e.g., zeroing balances, deleting mappings).
- No special privileges are required; any unprivileged EVM transaction triggers this path.
- The condition `minimumGasUsed > temporaryGasUsed` is easily satisfied by any user who over-estimates gas and clears storage.

### Recommendation

Apply the `MinGasMultiplier` floor to the raw (pre-EVM-refund) gas used, then add the EVM refund on top, so the refund is always returned regardless of the floor:

```go
rawGasUsed := msg.GasLimit - leftoverGasBeforeRefund
gasUsed = max(minimumGasUsed, rawGasUsed)
// Apply EVM refund after the floor
gasUsed = gasUsed - min(refund, gasUsed - minimumGasUsed.TruncateInt().Uint64())
```

Alternatively, ensure the EVM refund is always forwarded to `RefundGas` independently of the `gasUsed` floor, so the sender always receives `refund × gasPrice` back when they earned it.

### Proof of Concept

1. Deploy a contract that writes to storage (e.g., `SSTORE slot 1 = 1`).
2. Call a function that clears that slot (`SSTORE slot 1 = 0`), earning the EVM refund.
3. Set `gasLimit` high enough that `gasLimit × MinGasMultiplier > actualGasUsed − evmRefund`.
4. Observe that the sender's post-transaction balance equals `startBalance − gasLimit × gasPrice + (gasLimit − minimumGasUsed) × gasPrice`, not `startBalance − gasLimit × gasPrice + (gasLimit − temporaryGasUsed) × gasPrice`.
5. The difference `(minimumGasUsed − temporaryGasUsed) × gasPrice` remains in the fee collector, confirming the EVM refund was not returned.

The integration test at `tests/integration_tests/test_tracers.py` already validates the `min_gas_multiplier` charge formula but does not test the interaction with EVM-level SSTORE refunds, leaving this case uncovered. [5](#0-4)
