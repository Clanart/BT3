### Title
Silent Drop of Cross-Chain Message in `BridgeToken.mint` Causes Tokens Trapped in Contract Recipients - (File: `evm/src/omni-bridge/contracts/BridgeToken.sol`)

---

### Summary

`BridgeToken.mint(address, uint256, bytes)` silently discards the `message` parameter without invoking any callback on the recipient. When `OmniBridge.finTransfer` delivers a cross-chain transfer that carries a non-empty `message` to a bridge token, the tokens are minted to the recipient contract but the intended callback/action is never triggered. A smart-contract recipient that relies on the message to process the incoming tokens (e.g., a DeFi vault or router) will receive tokens it cannot act on, permanently trapping them.

---

### Finding Description

`OmniBridge.finTransfer` dispatches to the 3-argument `mint` overload whenever `payload.message.length > 0` and the token is a bridge token:

```solidity
// OmniBridge.sol lines 337-349
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount
        );
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount,
            payload.message   // <-- message passed here
        );
    }
}
```

The base `BridgeToken` implementation of that overload is:

```solidity
// BridgeToken.sol lines 54-60
function mint(
    address account,
    uint256 value,
    bytes memory          // <-- unnamed, never read
) external virtual onlyOwner {
    _mint(account, value);
}
```

The `bytes memory` parameter is unnamed and completely ignored. `_mint` credits the tokens to `account` with no notification to that address. No ERC-1363 `onTransferReceived`, no custom callback, no forwarding of the message bytes to the recipient â€” the message is silently dropped.

This is the direct analog of the reported ERC-721 issue: just as `_mint` skips `onERC721Received`, `BridgeToken._mint` skips any recipient notification, making the `message` field of the cross-chain transfer a no-op for all bridge tokens that use the base implementation.

---

### Impact Explanation

The Omni Bridge explicitly supports a `message` field in cross-chain transfers to enable DeFi composability (e.g., "bridge tokens and immediately deposit into vault X"). When the destination is a smart contract that:

1. Implements a handler expecting to be called with the message bytes (e.g., a router, vault, or aggregator), and
2. Has no independent mechanism to recover tokens it did not expect to receive,

the tokens are minted to the contract but the handler is never invoked. The contract holds tokens it cannot process, and the user has no recourse â€” the bridge considers the transfer finalised (`completedTransfers[payload.destinationNonce] = true` is set before the mint). This constitutes a **permanent, irrecoverable lock of user funds** in the recipient contract.

---

### Likelihood Explanation

The `message` field is a first-class protocol feature advertised for cross-chain DeFi composability. Any integrator building a contract recipient that expects a callback (the natural pattern for "receive tokens + instructions") will encounter this silently. The attacker-controlled entry path is fully unprivileged: any user can call `finTransfer` with a valid MPC signature and a non-empty `message` targeting a contract recipient. No special role or key is required beyond a valid proof/signature from the MPC network.

---

### Recommendation

- **Short term:** Override `mint(address, uint256, bytes)` in `BridgeToken` to forward the message bytes to the recipient via a standardised callback (e.g., ERC-1363 `onTransferReceived`, or a custom `IBridgeTokenReceiver.onBridgeTokenReceived(address, uint256, bytes)` interface), guarded by a check that the recipient is a contract. Fall back to plain `_mint` if the recipient does not implement the interface (matching the `_safeMint` pattern).
- **Long term:** Audit all `IBridgeToken` implementations (including `HlBridgeToken`) to ensure the `message` parameter is consistently handled and that the protocol's composability guarantees are actually enforced end-to-end.

---

### Proof of Concept

1. Alice deploys a DeFi vault contract on an EVM chain. The vault implements a function `onBridgeReceive(uint256 amount, bytes calldata data)` and expects to be called after receiving bridge tokens so it can credit Alice's internal balance.
2. Alice initiates a cross-chain transfer from NEAR to EVM with `message = abi.encode("deposit_for_alice")` and `recipient = address(vault)`.
3. A relayer calls `OmniBridge.finTransfer` with the MPC-signed payload. `completedTransfers[nonce]` is set to `true`.
4. `OmniBridge` calls `IBridgeToken(token).mint(vault, amount, message)`.
5. `BridgeToken.mint` executes `_mint(vault, amount)` and returns â€” the `message` bytes are never read.
6. The vault's `onBridgeReceive` is never called. The vault holds `amount` tokens with no record of Alice's deposit.
7. Alice's tokens are permanently locked in the vault with no recovery path; the bridge marks the transfer as complete and will reject any replay attempt. [1](#0-0) [2](#0-1) [3](#0-2)
