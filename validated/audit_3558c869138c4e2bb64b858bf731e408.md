### Title
Transaction Prover Unconditionally Rejects Valid Storage-Zero Writes, Blocking Proof Generation — (`crates/starknet_transaction_prover/src/running/committer_utils.rs`)

### Summary

`validate_virtual_os_state_diff` in the virtual SNOS prover pipeline rejects any state diff that contains a storage update whose value is `Felt::ZERO`. Writing zero to a storage slot is a valid Starknet operation (it clears/resets the slot to its default). Any transaction that performs such an operation cannot be proven via the transaction prover RPC, even though the transaction itself is accepted and executed correctly by the sequencer. This is a direct structural analog to the original `require(amount - claimable != 0)` bug: a guard that blocks a valid boundary-value operation.

### Finding Description

In `validate_virtual_os_state_diff`:

```rust
for (address, storage_diffs) in &state_diff.storage_updates {
    for (key, value) in storage_diffs {
        if value.0 == Felt::ZERO {
            return Err(ProofProviderError::InvalidStateDiff(format!(
                "Storage deletion not allowed: try to delete storage at address {address:?}, \
                 key {key:?}"
            )));
        }
    }
}
``` [1](#0-0) 

This check is called unconditionally from `create_commitment_infos_with_state_changes` before the Patricia Merkle commitment is computed:

```rust
let committer_state_diff = state_maps_to_committer_state_diff(state_diff.clone());
validate_virtual_os_state_diff(&committer_state_diff)?;
``` [2](#0-1) 

The rationale given in the code is that dummy binary nodes (with zero hashes) are inserted for orphan sibling hashes, and this dummy-node strategy only works correctly when no trie deletions occur. However, the check conflates two distinct concepts:

1. **Trie deletion** — removing a leaf from the Patricia trie (requires traversal of sibling nodes that may lack preimages).
2. **Storage write-to-zero** — writing `0x0` as a value to a storage slot. This is a valid Starknet state transition; the slot is not deleted from the trie, it is updated to hold the value zero.

The dummy-node comment confirms the developers are aware of the deletion limitation:

> "Since we don't allow deletions, we insert dummy binary nodes (with zero hashes) to satisfy the committer's traversal requirements without requiring full preimages." [3](#0-2) 

But the guard is applied to any storage value equal to zero, not only to actual trie deletions. A storage write of `Felt::ZERO` does not require sibling traversal in the same way a deletion does; it is an update to an existing leaf. The check is therefore overly broad.

### Impact Explanation

Any user who submits an `InvokeV3` transaction to the `prove_transaction` RPC endpoint where the transaction's execution writes `0x0` to any storage slot will receive a `ProofProviderError::InvalidStateDiff` error. The transaction itself is valid, accepted by the sequencer, and included in a block with the correct state diff. Only the proof generation path is broken.

Concretely:
- The `ProveTransactionResult` (containing `proof`, `proof_facts`, and `l2_to_l1_messages`) is never returned.
- The user cannot embed proof facts in a subsequent transaction to leverage client-side proving.
- The transaction prover RPC returns an authoritative-looking error for a provably valid transaction.

This matches: **High — RPC execution returns an authoritative-looking wrong value** (the prover claims the transaction is unprovable when it is not). [4](#0-3) 

### Likelihood Explanation

Writing zero to a storage slot is a routine Starknet operation. Common examples include:
- Resetting a counter or allowance to zero.
- Clearing a mapping entry (e.g., `balances[user] = 0` after a full withdrawal).
- Any contract that uses zero as a sentinel "deleted" value.

Any unprivileged user can trigger this by submitting such a transaction to the prover. No special privileges or adversarial setup are required.

### Recommendation

Distinguish between a true Patricia trie deletion (removing a leaf) and a storage write whose value happens to be zero. The dummy-node strategy is only needed for the former. The check should be:

- If the storage key did not previously exist in the trie and the new value is zero, that is a no-op and can be skipped.
- If the storage key previously existed and the new value is zero, that is a valid update (write zero) and should be allowed through the commitment pipeline with full preimage support, or the dummy-node strategy should be extended to handle this case.

At minimum, document the limitation clearly in the RPC error message so users understand why their transaction cannot be proven and what workaround exists.

### Proof of Concept

1. Deploy a contract with a storage variable `x` initialized to `1`.
2. Submit an `InvokeV3` transaction that calls `x.write(0)` (sets `x` to zero).
3. The sequencer accepts and executes the transaction; the state diff contains `storage_updates = {contract: {key: 0x0}}`.
4. Call `prove_transaction(block_id, tx)` on the transaction prover RPC.
5. Inside `run_and_prove` → `runner.run_virtual_os` → `create_commitment_infos_with_state_changes`, `validate_virtual_os_state_diff` is called.
6. The check `if value.0 == Felt::ZERO` fires and returns `ProofProviderError::InvalidStateDiff("Storage deletion not allowed...")`.
7. The prover returns an error; no `ProofFacts` are produced for a transaction that was validly executed and committed. [5](#0-4) [6](#0-5)

/// Validates that the committer state diff contains only allowed state transitions.
///
/// This function enforces the following constraints:
/// * **No Storage Deletions:** Storage entries cannot be updated to `Felt::ZERO`.
/// * **No Class Declarations:** The `class_hash_to_compiled_class_hash` map must be empty.
/// * **No Contract Deployments:** The `address_to_class_hash` map must be empty.
pub(crate) fn validate_virtual_os_state_diff(
    state_diff: &StateDiff,
) -> Result<(), ProofProviderError> {
    // validate no storage deletions.
    for (address, storage_diffs) in &state_diff.storage_updates {
        for (key, value) in storage_diffs {
            if value.0 == Felt::ZERO {
                return Err(ProofProviderError::InvalidStateDiff(format!(
                    "Storage deletion not allowed: try to delete storage at address {address:?}, \
                     key {key:?}"
                )));
            }
        }
    }
    // validate no contract deployments (or replaced classes).
    if !state_diff.address_to_class_hash.is_empty() {
        return Err(ProofProviderError::InvalidStateDiff(format!(
            "Contract deployments not allowed: try to deploy contracts(address to class hash): \
             {0:?}",
            state_diff.address_to_class_hash
        )));
    }
    // validate no contract declarations (or compiled class hash updates).
    if !state_diff.class_hash_to_compiled_class_hash.is_empty() {
        return Err(ProofProviderError::InvalidStateDiff(format!(
            "Contract declarations not allowed: try to declare classes(class hash to compiled \
             class hash): {0:?}",
            state_diff.class_hash_to_compiled_class_hash
        )));
    }
    Ok(())
}

    pub async fn prove_transaction(
        &self,
        block_id: BlockId,
        transaction: RpcTransaction,
    ) -> Result<ProveTransactionResult, VirtualSnosProverError> {
        let start_time = Instant::now();

        // Validate block_id is not pending.
        if matches!(block_id, BlockId::Pending) {
            return Err(VirtualSnosProverError::ValidationError(
                "Pending blocks are not supported; only finalized blocks can be proven."
                    .to_string(),
            ));
        }

        let invoke_v3 = extract_rpc_invoke_tx(transaction.clone())?;
        validate_transaction_input(&invoke_v3, self.validate_zero_fee_fields)?;
        let invoke_tx = InvokeTransaction::V3(invoke_v3.into());

        let result = match &self.blocking_check_client {
            None => self.run_and_prove(block_id, vec![invoke_tx]).await?,
            Some(client) => {
                self.prove_with_blocking_check(client, block_id, transaction, invoke_tx).await?
            }
        };

        info!(total_duration_ms = %start_time.elapsed().as_millis(), "prove_transaction completed");
        Ok(result)
    }
