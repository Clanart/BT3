### Title
`MerkleBlob::upsert` Missing Duplicate-Hash Guard Enables Merkle Tree Corruption and Forged Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::insert` explicitly rejects a new leaf whose hash already exists in the tree. `MerkleBlob::upsert` — the only other write path that changes a leaf's hash — omits this guard entirely. Supplying a hash that is already held by a *different* key silently overwrites the `leaf_hash_to_index` cache entry for that hash, places two leaves with identical hashes into the tree, and makes every subsequent `ProofOfInclusion` for either leaf cross-valid for the other, constituting a forged inclusion proof.

---

### Finding Description

`insert` guards against hash collisions before touching the tree:

```rust
// blob.rs lines 372-374
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`upsert` has no equivalent guard. Its update path is:

```rust
// blob.rs lines 792-810
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;   // insert path IS guarded
        return Ok(());
    };

    self.block_status_cache.remove_leaf(&leaf)?;   // removes old hash from cache
    leaf.hash.clone_from(new_hash);                // sets new (potentially colliding) hash
    leaf.value = value;
    block.node = Node::Leaf(leaf);
    self.insert_entry_to_blob(leaf_index, &block)?; // calls add_leaf → silently overwrites cache
    ...
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which calls `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index` without any collision check:

```rust
// blob.rs lines 188-192
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);  // silently overwrites existing entry
}
```

After `upsert(key_A, value_A, hash_B)` where `hash_B` is already the stored hash of `key_B`:

| State | Before | After |
|---|---|---|
| `leaf_hash_to_index[hash_B]` | → index of key_B | → index of key_A (overwritten) |
| Blob leaf at key_B's index | hash = hash_B | hash = hash_B (unchanged) |
| Blob leaf at key_A's index | hash = hash_A | hash = hash_B (duplicate) |

Two leaves now carry the same hash. `check_integrity` would detect this via the `leaf_hash_to_index` length mismatch, but `upsert` never calls it, and the caller is not required to.

---

### Impact Explanation

`ProofOfInclusion::valid()` reconstructs the root by hashing upward from `node_hash` through the layer chain:

```rust
// proof_of_inclusion.rs lines 40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
```

`node_hash` is the leaf's stored hash. Because both key_A and key_B now store `hash_B`, the proof generated for key_A (`node_hash = hash_B`, path through key_A's subtree) is structurally identical to a proof for key_B. A verifier calling `proof.valid()` cannot distinguish them: the proof is accepted as proving inclusion of *either* key. This is a forged inclusion proof — an untrusted party can present a valid proof for a key-value pair that was never legitimately committed to the tree.

Additionally, `get_node_by_hash(hash_B)` now returns key_A's `(KeyId, ValueId)` instead of key_B's, corrupting any lookup that relies on the hash-to-node mapping.

**Matched allowed impact:** *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

---

### Likelihood Explanation

The `upsert` Python binding (`MerkleBlob.upsert(key, value, new_hash)`) is the primary DataLayer write path for updating existing records. The hash argument is caller-supplied. Any code path — including automated sync, delta application, or a Python consumer — that passes a hash already present in the tree for a different key silently triggers the corruption. No cryptographic break is required; the attacker only needs to supply a hash value that matches an existing leaf's hash, which is trivially achievable if the attacker controls the data being upserted (e.g., by choosing a value whose hash equals a known existing leaf hash). Likelihood is **Low** in practice (requires knowledge of existing leaf hashes and control over the upserted hash), but the missing guard is a direct, reachable code path with no defensive backstop.

---

### Recommendation

Add the same duplicate-hash guard to `upsert` that `insert` already enforces. After `remove_leaf` removes the old hash, check whether `new_hash` is already present for a *different* key before writing:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // Guard: reject if new_hash already belongs to a different leaf
    if self.block_status_cache.contains_leaf_hash(new_hash) {
        return Err(Error::HashAlreadyPresent());
    }

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    ...
}
```

Apply the same guard to the bulk path in `batch_insert`, which also bypasses duplicate-hash checks for all items beyond the first two.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Insert key_A with hash_A
let hash_a = Hash(Bytes32::new([0xAA; 32]));
blob.insert(KeyId(1), ValueId(1), &hash_a, InsertLocation::Auto {}).unwrap();

// Insert key_B with hash_B
let hash_b = Hash(Bytes32::new([0xBB; 32]));
blob.insert(KeyId(2), ValueId(2), &hash_b, InsertLocation::Auto {}).unwrap();

blob.calculate_lazy_hashes().unwrap();

// Upsert key_A with hash_B (hash already owned by key_B) — no error returned
blob.upsert(KeyId(1), ValueId(99), &hash_b).unwrap();

blob.calculate_lazy_hashes().unwrap();

// Proof for key_A uses node_hash = hash_B
let proof_a = blob.get_proof_of_inclusion(KeyId(1)).unwrap();
assert!(proof_a.valid());                    // passes
assert_eq!(proof_a.node_hash, hash_b);      // same hash as key_B

// Proof for key_B also uses node_hash = hash_B — structurally identical root path
let proof_b = blob.get_proof_of_inclusion(KeyId(2)).unwrap();
assert!(proof_b.valid());                    // passes
assert_eq!(proof_b.node_hash, hash_b);      // same node_hash

// get_node_by_hash now returns key_A's data for hash_B — key_B's mapping is lost
let (returned_key, _) = blob.get_node_by_hash(hash_b).unwrap();
assert_eq!(returned_key, KeyId(1));          // key_B (KeyId(2)) is silently displaced

// check_integrity reveals the corruption
blob.check_integrity().expect_err("leaf_hash_to_index length mismatch");
