### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash corresponds to a declared contract class before committing the state change. This is the direct analog of the ERC721 `_mint()` vs `_safeMint()` vulnerability class: an operation is performed on a target without first verifying the target can handle it. Any contract can replace its own class with an arbitrary, undeclared class hash, permanently making itself uncallable and freezing any funds held within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested new class hash from the syscall request and immediately writes it into `contract_state_changes` without checking whether that class hash has been declared in the protocol:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The code itself contains an explicit acknowledgment of the missing check at line 898:

> `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.`

The OS never verifies that `request.class_hash` exists in `contract_class_changes` (the dictionary of declared class hashes → compiled class hashes). Any arbitrary `felt` value is accepted and committed as the contract's new class hash. [1](#0-0) 

The analogous missing check in `deploy_contract.cairo` does verify that the contract address is uninitialized before deploying, but `execute_replace_class` has no equivalent guard for the new class hash. [2](#0-1) 

---

### Impact Explanation

When a contract's class hash is set to an undeclared value, the OS has no compiled class to dispatch execution to. Every subsequent call to that contract — including calls needed to withdraw or transfer funds — will fail at the class lookup stage. There is no recovery path: the state change is committed to the Patricia Merkle Tree and proven on-chain. Any ERC-20 tokens, ETH, or STRK held in the contract's storage are permanently frozen.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract on itself. A malicious contract deployer can:
1. Deploy a contract that accepts user deposits (e.g., a yield vault or token bridge).
2. Attract user funds.
3. Call `replace_class` with an arbitrary felt (e.g., `1`) that is not a declared class hash.
4. The OS accepts the call without validation and commits the invalid class hash.
5. All subsequent calls to the contract revert; funds are permanently frozen.

Additionally, any legitimate upgradeable contract whose upgrade logic contains a bug (e.g., passes the wrong argument) can accidentally brick itself, with no OS-level safety net to catch the mistake. The explicit TODO comment confirms the development team is aware this check is absent.

---

### Recommendation

Before committing the new class hash in `execute_replace_class`, verify that `request.class_hash` exists as a key in `contract_class_changes` with a non-zero compiled class hash value. This mirrors the `_safeMint()` pattern: validate the target can handle the operation before performing it.

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
``` [3](#0-2) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` which accepts STRK deposits and exposes an `upgrade(new_class_hash)` function that calls `replace_class`.
2. Users deposit STRK into `VaultContract`; the contract accumulates funds in its storage.
3. Attacker calls `upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. `execute_replace_class` in `syscall_impls.cairo` (line 896–910) reads `class_hash = 0xdeadbeef`, skips the missing existence check (line 898 TODO), and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. The state update is committed and proven. `VaultContract`'s class hash is now `0xdeadbeef`.
6. Any user attempting to call `withdraw()` on `VaultContract` causes the OS to look up the compiled class for `0xdeadbeef`, find nothing, and revert.
7. All deposited STRK is permanently frozen with no recovery mechanism. [4](#0-3)
