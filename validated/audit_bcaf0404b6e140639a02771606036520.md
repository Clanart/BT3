### Title
Fixed Gas Cost for Variable-Length Payload in `execute_send_message_to_l1` Enables Unbounded OS-Level Work — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

`execute_send_message_to_l1` charges a flat, fixed gas cost (`SEND_MESSAGE_TO_L1_GAS_COST = 14470`) regardless of the message payload size, then unconditionally performs a `memcpy` of an attacker-controlled number of felts into the OS output segment. The OS-level `memcpy` work is O(payload_size) in Cairo steps, but zero additional gas is charged per payload element. An unprivileged contract can therefore force the StarkNet OS prover to execute a disproportionately large number of steps while paying only a fixed fee, potentially exhausting the prover's step budget and halting block confirmation.

---

### Finding Description

In `execute_send_message_to_l1`:

```cairo
func execute_send_message_to_l1{range_check_ptr, syscall_ptr: felt*, outputs: OsCarriedOutputs*}(
    contract_address: felt
) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, SendMessageToL1Request*);
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=SEND_MESSAGE_TO_L1_GAS_COST,   // <-- flat 14470, no per-element cost
        request_struct_size=SendMessageToL1Request.SIZE
    );
    if (success == FALSE) { return (); }

    tempvar payload_start = request.payload_start;
    tempvar payload_size = request.payload_end - payload_start;  // attacker-controlled

    assert [outputs.messages_to_l1] = MessageToL1Header(..., payload_size=payload_size);
    memcpy(                                            // <-- O(payload_size) OS steps, uncharged
        dst=outputs.messages_to_l1 + MessageToL1Header.SIZE,
        src=payload_start,
        len=payload_size
    );
    ...
}
```

The gas constant is:

```
const SEND_MESSAGE_TO_L1_GAS_COST = 14470;   // flat, no payload factor
```

Compare this to `execute_deploy`, which correctly scales gas with data size:

```cairo
let specific_base_gas_cost = DEPLOY_GAS_COST + DEPLOY_CALLDATA_FACTOR_GAS_COST *
    constructor_calldata_size;
```

and `execute_meta_tx_v0`, which similarly uses `META_TX_V0_CALLDATA_FACTOR_GAS_COST * calldata_size`. No analogous per-element factor exists for `send_message_to_l1` payloads.

Additionally, unlike calldata (which is checked with `assert_nn_le(calldata_size, SIERRA_ARRAY_LEN_BOUND - 1)` in transaction processing), the message payload has **no upper-bound assertion** inside `execute_send_message_to_l1` before the `memcpy`.

The same structural flaw exists in the deprecated path in `deprecated_execute_syscalls.cairo`:

```cairo
memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE,
    src=syscall.payload_ptr,
    len=syscall.payload_size,   // attacker-controlled, no gas scaling
);
```

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

A contract's gas budget bounds how many Sierra steps it can execute, and therefore how many felts it can write to the syscall segment. With `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` and `STEP_GAS_COST = 100`, a contract can write up to ~11,000,000 felts as a payload. The OS-level `memcpy` of that payload costs approximately 3× that many Cairo VM steps (~33M steps) — work that is entirely outside the contract's gas budget and not accounted for by the flat `SEND_MESSAGE_TO_L1_GAS_COST`. If the prover's per-block step capacity is exceeded, the block cannot be proven and the network halts. Even below that threshold, an attacker can systematically fill blocks with transactions that each carry maximum-size payloads, forcing the sequencer/prover to do far more work than the fee collected, degrading throughput to zero.

---

### Likelihood Explanation

Any unprivileged contract can invoke the `send_message_to_l1` syscall with an arbitrarily large payload bounded only by its gas budget. No special privilege, key, or operator cooperation is required. The attacker pays only the flat `SEND_MESSAGE_TO_L1_GAS_COST` (14470 gas) per call, making repeated exploitation cheap. The attack is deterministic and reproducible.

---

### Recommendation

Introduce a per-element gas factor for the payload, mirroring the pattern already used for `deploy` and `meta_tx_v0`:

```cairo
// In constants.cairo
const SEND_MESSAGE_TO_L1_PAYLOAD_FACTOR_GAS_COST = <calibrated_value>;

// In execute_send_message_to_l1
let payload_size = request.payload_end - request.payload_start;
let specific_gas_cost = SEND_MESSAGE_TO_L1_GAS_COST +
    SEND_MESSAGE_TO_L1_PAYLOAD_FACTOR_GAS_COST * payload_size;
let success = reduce_syscall_gas_and_write_response_header(
    total_gas_cost=specific_gas_cost,
    request_struct_size=SendMessageToL1Request.SIZE
);
```

Additionally, add an explicit upper-bound assertion on `payload_size` (e.g., `assert_nn_le(payload_size, SIERRA_ARRAY_LEN_BOUND - 1)`) before the `memcpy`, consistent with how calldata sizes are validated elsewhere.

---

### Proof of Concept

1. Deploy a contract whose `__execute__` function calls `send_message_to_l1` with `payload_end - payload_start = N` (maximized within the gas budget, e.g., N ≈ 10,000,000 felts).
2. Submit the transaction. The contract pays `SEND_MESSAGE_TO_L1_GAS_COST = 14470` gas for the syscall.
3. The OS executes `execute_send_message_to_l1`, which calls `memcpy(..., len=N)` — approximately 30,000,000 Cairo VM steps of OS-level work, uncharged.
4. If N is large enough to push the block's total OS step count past the prover's capacity, the block cannot be proven. Repeated across multiple transactions in a block, the attacker can reliably prevent block finalization.
