### Title
Case-Sensitive String Comparison in `OmniAddress::Btc`/`OmniAddress::Zcash` Allows Duplicate Token Registration for the Same UTXO Address - (File: near/omni-types/src/lib.rs)

---

### Summary

`OmniAddress::Btc` and `OmniAddress::Zcash` wrap a raw `String` (`UTXOChainAddress = String`) and derive `PartialEq`/`Hash` via the standard string comparison, which is byte-exact and case-sensitive. Bitcoin and Zcash addresses are case-sensitive in their canonical encoding, but the bridge accepts any UTF-8 string without normalization. A caller can register the same physical UTXO address under two different string representations (e.g., mixed-case variants of a legacy Base58Check address, or two structurally distinct but semantically equivalent Zcash unified address encodings) and bypass the `add_token` uniqueness guard, creating two separate token mappings for the same underlying asset. This breaks bridge collateralization accounting.

---

### Finding Description

`UTXOChainAddress` is defined as a plain `String`:

```rust
pub type UTXOChainAddress = String;
```

`OmniAddress` derives `PartialEq` and `Hash` from the standard library, which for `String` performs byte-exact comparison:

```rust
#[near(serializers=[borsh])]
#[derive(Debug, Clone, Hash, PartialEq, Eq)]
pub enum OmniAddress {
    ...
    Btc(UTXOChainAddress),
    Zcash(UTXOChainAddress),
    ...
}
``` [1](#0-0) 

The `FromStr` implementation for `OmniAddress` accepts the raw string without any normalization:

```rust
"btc"   => Ok(Self::Btc(recipient.to_string())),
"zcash" => Ok(Self::Zcash(recipient.to_string())),
``` [2](#0-1) 

The `add_token` function in the NEAR bridge contract uses `OmniAddress` as a key in `token_address_to_id` and `token_decimals` maps. The uniqueness check relies entirely on the `PartialEq`/`Hash` of `OmniAddress`:

```rust
fn add_token(&mut self, token_id: &AccountId, token_address: &OmniAddress, ...) {
    require!(
        self.token_address_to_id.insert(token_address, token_id).is_none(),
        BridgeError::TokenExists.as_ref()
    );
    ...
}
``` [3](#0-2) 

Because `OmniAddress::Btc("BC1QXXX".to_string())` and `OmniAddress::Btc("bc1qxxx".to_string())` are byte-distinct strings, they hash to different slots and pass the `.is_none()` guard independently. The same physical Bitcoin/Zcash address can therefore be registered twice under two different `OmniAddress` keys, creating two separate NEAR token accounts that both claim to represent the same on-chain asset.

The `deploy_token_internal` function derives the NEAR token account ID from `token_address.get_token_prefix()`, which for `Btc`/`Zcash` calls `self.encode('-', true)` — a direct string embed of the raw address:

```rust
_ => self.encode('-', true),
``` [4](#0-3) 

Two differently-cased strings produce two different NEAR account IDs, so both `deployed_tokens.insert` calls succeed and two separate NEP-141 token contracts are deployed for the same UTXO asset.

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

Once two NEAR token accounts exist for the same Bitcoin/Zcash address:

1. The UTXO connector credits incoming BTC/ZEC transfers to whichever token account matches the string it received. The other token account has zero backing but can still be used in `ft_on_transfer` to initiate outbound transfers.
2. A user holding tokens from the "shadow" account can call `ft_on_transfer` → `init_transfer` → `submit_transfer_to_utxo_chain_connector`, causing the bridge to attempt to send real BTC/ZEC that was never deposited against that token account.
3. This directly breaks the locked-token accounting (`locked_tokens` map is keyed by `(ChainKind, AccountId)`) and can drain the UTXO connector's funds.

---

### Likelihood Explanation

**Medium.** Bitcoin legacy addresses (Base58Check) are case-sensitive and mixed-case variants are syntactically valid strings. A relayer or any caller of `deploy_token` who controls the `LogMetadata` event content (or who can submit a valid proof with a differently-cased address string) can trigger this. The bridge explicitly notes it does not validate UTXO address format:

> "The bridge doesn't validate Zcash address format — `OmniAddress::Zcash(String)` accepts anything" [5](#0-4) 

The `new_from_slice` path for `Btc`/`Zcash` also performs no normalization:

```rust
ChainKind::Btc => Ok(Self::Btc(
    String::from_utf8(address.to_vec())
        .map_err(|e| format!("Invalid BTC address: {e}"))?,
)),
``` [6](#0-5) 

---

### Recommendation

Normalize `Btc` and `Zcash` address strings to a canonical form before storing or comparing them. For Bech32/Bech32m addresses (Bitcoin SegWit, Zcash Sapling/Orchard), the canonical form is lowercase. For legacy Base58Check addresses, the string is already case-sensitive by design, but the bridge should enforce a single accepted encoding (e.g., reject addresses that do not match a known format). At minimum, apply `.to_lowercase()` on parse for `Btc` and `Zcash` variants in `FromStr` and `new_from_slice`, consistent with how EVM addresses are normalized by `H160`'s own parser.

---

### Proof of Concept

1. A relayer emits two `LogMetadata` events for the same BTC token, one with address `"bc1qabcdef..."` and one with `"BC1QABCDEF..."`.
2. Both pass `OmniAddress::from_str` without error.
3. Both are submitted to `deploy_token` on the NEAR bridge. Each call reaches `add_token` with a distinct `OmniAddress::Btc(...)` key, so both `.is_none()` checks pass.
4. Two NEAR token accounts are created: `bc1qabcdef-....omnidep.near` and `bc1qabcdef-....omnidep.near` (or two distinct account IDs if the prefix differs by case).
5. The attacker acquires tokens from the "shadow" account (e.g., via a fast-transfer relayer that uses the uppercase variant) and calls `ft_on_transfer` → `submit_transfer_to_utxo_chain_connector` to withdraw real BTC against unbacked supply. [7](#0-6) [8](#0-7)

        decimals: u8,

                        decimals,
                        origin_decimals,
