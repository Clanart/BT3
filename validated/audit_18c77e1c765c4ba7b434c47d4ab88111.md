### Title
`ChunkExtra.proposed_split` Stale After Old-Chunk Processing Breaks Chunk Validation Near Epoch Boundaries — (`chain/chain/src/chain_update.rs`, `core/primitives/src/types.rs`)

---

### Summary

When a chunk is missing (old chunk) near an epoch boundary and `ProtocolFeature::DynamicResharding` is active, `ChunkExtra.proposed_split` is silently carried over from the previous block without recomputation. The next new chunk's header carries a freshly computed `proposed_split` (based on the actual post-state of the missing-chunk block), which will not equal the stale value stored in `ChunkExtra`. The equality check in `validate_chunk_with_chunk_extra_and_receipts_root` then rejects the honest chunk with `InvalidChunkHeaderShardSplit`, producing a liveness failure at the epoch boundary.

---

### Finding Description

**The commitment invariant.** Under dynamic resharding, `ChunkExtra.proposed_split` is the authoritative record of the shard-split proposal computed when applying a chunk. The next chunk header must carry an identical value; `validate_chunk_with_chunk_extra_and_receipts_root` enforces this:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [1](#0-0) 

**How `proposed_split` is set for new chunks.** During `apply_transactions` (new chunk path), `NightshadeRuntime::compute_proposed_split` is called with `is_new_chunk: true`. It checks `is_next_block_possibly_last_in_epoch`, runs `find_trie_split` against the live trie, and stores the result in `ChunkExtra` and in the chunk header. The value is therefore always consistent with the actual post-state of that block.

**How `proposed_split` is set for old (missing) chunks — the bug.** For an old chunk, `process_apply_chunk_result` calls `next_for_old_chunk`, which clones the previous `ChunkExtra` and only replaces `state_root`:

```rust
let old_extra = self.chain_store_update.get_chunk_extra(prev_hash, &shard_uid)?;
let new_extra = old_extra.next_for_old_chunk(apply_result.new_root);
``` [2](#0-1) 

```rust
pub fn next_for_old_chunk(&self, state_root: StateRoot) -> Self {
    let mut new_extra = self.clone();
    *new_extra.state_root_mut() = state_root;
    new_extra   // proposed_split is silently inherited
}
``` [3](#0-2) 

The same pattern appears in `set_state_finalize_on_height` (used during state-sync catch-up):

```rust
let mut new_chunk_extra = ChunkExtra::clone(&chunk_extra);
*new_chunk_extra.state_root_mut() = apply_result.new_root;
self.chain_store_update.save_chunk_extra(block_header.hash(), &shard_uid, new_chunk_extra.into());
``` [4](#0-3) 

`proposed_split` is **never recomputed** for old chunks. Because old-chunk application passes `is_new_chunk: false` to the runtime, `compute_proposed_split` is never invoked, and the `OldChunkResult` carries no `proposed_split` field. [5](#0-4) 

**The mismatch scenario.**

| Block | Chunk type | `ChunkExtra.proposed_split` |
|-------|-----------|----------------------------|
| H−1 | new chunk | `None` (not near epoch end) |
| H | **old (missing) chunk** | `None` — carried over from H−1 |
| H+1 | new chunk (last of epoch) | chunk producer calls `compute_proposed_split` on state-at-H; memory usage ≥ threshold → `Some(TrieSplit{boundary_account, left_mem, right_mem})` |

Validation at H+1: `chunk_header.proposed_split() = Some(…)` vs `ChunkExtra[H].proposed_split = None` → **`InvalidChunkHeaderShardSplit`** → chunk rejected.

The reverse direction also applies: if H−1 had `proposed_split = Some(split_X)` (computed near a prior epoch boundary), H carries it forward stale, but the chunk producer at H+1 recomputes based on the actual state at H (which has changed due to validator-account updates applied during the old-chunk pass). Even if the `boundary_account` is identical, the `left_memory`/`right_memory` fields in `TrieSplit` will differ because validator-account state changes alter trie memory usage. The equality check fails.

---

### Impact Explanation

An honest chunk producer near an epoch boundary produces a chunk header whose `proposed_split` is computed from the real post-state of block H. The local node's `ChunkExtra[H].proposed_split` is stale (from H−1). `validate_chunk_with_chunk_extra_and_receipts_root` rejects the chunk with `InvalidChunkHeaderShardSplit`. If the missing-chunk condition persists across multiple consecutive heights near the epoch boundary, no new chunk can be accepted, stalling the chain at the epoch transition. This is a **liveness failure** scoped to the epoch-boundary window when `DynamicResharding` is active.

---

### Likelihood Explanation

Three conditions must coincide:
1. `ProtocolFeature::DynamicResharding` is enabled (protocol version ≥ 153 per the codebase).
2. A chunk is missing at a height where `is_next_block_possibly_last_in_epoch` returns `true` for the *following* block — a window of a few blocks near each epoch end.
3. The shard's trie memory usage is at or above `memory_usage_threshold`.

Condition 1 is a scheduled mainnet upgrade. Condition 2 occurs naturally under any network partition or slow chunk producer. Condition 3 is the explicit trigger for dynamic resharding and is expected to be true for the shard being monitored for splitting. All three conditions are reachable in production without privileged access.

---

### Recommendation

In both `process_apply_chunk_result` (old-chunk branch) and `set_state_finalize_on_height`, after computing `apply_result.new_root`, recompute `proposed_split` using the same `compute_proposed_split` logic applied to the new state root, and store the result in the new `ChunkExtra` before saving it. Alternatively, explicitly reset `proposed_split` to `None` for old-chunk blocks (since no new-chunk application occurred), and ensure the chunk producer at the next height also produces `None` for old-chunk predecessor blocks — but this requires a coordinated protocol rule. The cleaner fix is to recompute `proposed_split` from the actual post-state whenever `ChunkExtra` is written for an old-chunk block.

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` in a test network with a shard whose memory usage exceeds `memory_usage_threshold`.
2. Configure `epoch_length = 10`. At height 8 (two blocks before epoch end), arrange for the chunk to be missing (e.g., drop the chunk producer's network messages).
3. At height 9 (last block of epoch), observe that `is_next_block_possibly_last_in_epoch` returns `true`. The chunk producer calls `compute_proposed_split` on the state at height 8 and produces `proposed_split = Some(TrieSplit{…})` in the chunk header.
4. The validating node reads `ChunkExtra[8].proposed_split`, which was set by `next_for_old_chunk` and equals `ChunkExtra[7].proposed_split = None`.
5. `validate_chunk_with_chunk_extra_and_receipts_root` compares `Some(…) != None` and returns `Err(InvalidChunkHeaderShardSplit)`.
6. The chunk at height 9 is rejected; the epoch cannot close. [6](#0-5) [7](#0-6)

/// Validate that all next chunk information matches previous chunk extra.
pub fn validate_chunk_with_chunk_extra_and_receipts_root(
    prev_chunk_extra: &ChunkExtra,
    chunk_header: &ShardChunkHeader,
    outgoing_receipts_root: &CryptoHash,
) -> Result<(), Error> {
    if *prev_chunk_extra.state_root() != chunk_header.prev_state_root() {
        return Err(Error::InvalidStateRoot);
    }

    if prev_chunk_extra.outcome_root() != chunk_header.prev_outcome_root() {
        return Err(Error::InvalidOutcomesProof);
    }

    let chunk_extra_proposals = prev_chunk_extra.validator_proposals();
    let chunk_header_proposals = chunk_header.prev_validator_proposals();
    if chunk_header_proposals.len() != chunk_extra_proposals.len()
        || !chunk_extra_proposals.eq(chunk_header_proposals)
    {
        return Err(Error::InvalidValidatorProposals);
    }

    if prev_chunk_extra.gas_limit() != chunk_header.gas_limit() {
        return Err(Error::InvalidGasLimit);
    }

    if prev_chunk_extra.gas_used() != chunk_header.prev_gas_used() {
        return Err(Error::InvalidGasUsed);
    }

    if prev_chunk_extra.balance_burnt() != chunk_header.prev_balance_burnt() {
        return Err(Error::InvalidBalanceBurnt);
    }

    if outgoing_receipts_root != chunk_header.prev_outgoing_receipts_root() {
        return Err(Error::InvalidReceiptsProof);
    }

    validate_congestion_info(prev_chunk_extra.congestion_info(), chunk_header.congestion_info())?;
    validate_bandwidth_requests(
        prev_chunk_extra.bandwidth_requests(),
        chunk_header.bandwidth_requests(),
    )?;

    if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
        DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
        return Err(Error::InvalidChunkHeaderShardSplit(format!(
            "header has {:?}, expected {:?} (prev block hash: {:?} height created: {:?})",
            chunk_header.proposed_split(),
            prev_chunk_extra.proposed_split(),
            chunk_header.prev_block_hash(),
            chunk_header.height_created(),
        )));
    }

    Ok(())
}

/// Applies shard update corresponding to missing chunk.
/// (logunov) From what I know, the state update may include only validator
/// accounts update on epoch start.
pub fn apply_old_chunk(
    apply_reason: ApplyChunkReason,
    parent_span: &tracing::Span,
    data: OldChunkData,
    shard_context: ShardContext,
    runtime: &dyn RuntimeAdapter,
    memtrie_pin: MaybePinnedMemtrieRoot,
) -> Result<OldChunkResult, Error> {
    let OldChunkData { prev_chunk_extra, block, storage_context } = data;
    let shard_uid = shard_context.shard_uid;
    let shard_id = shard_uid.shard_id();
    let _span = tracing::debug_span!(
        target: "chain",
        parent: parent_span,
        "apply_old_chunk",
        height = block.height,
        %shard_id,
        block_type = ?block.block_type,
        ?apply_reason,
        tag_block_production = true)
    .entered();

    let storage_config = RuntimeStorageConfig {
        state_root: *prev_chunk_extra.state_root(),
        use_flat_storage: true,
        source: storage_context.storage_data_source,
        state_patch: storage_context.state_patch,
    };
    match runtime.apply_chunk(
        storage_config,
        apply_reason,
        ApplyChunkShardContext {
            shard_uid,
            last_validator_proposals: prev_chunk_extra.validator_proposals(),
            gas_limit: prev_chunk_extra.gas_limit(),
            is_new_chunk: false,
            on_post_state_ready: None,
            memtrie_pin,
        },
        block,
        &[],
        SignedValidPeriodTransactions::empty(),
    ) {
        Ok(apply_result) => Ok(OldChunkResult { shard_uid: shard_context.shard_uid, apply_result }),
        Err(err) => Err(err),
    }
