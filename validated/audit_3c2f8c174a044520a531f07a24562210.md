Audit Report

## Title
Router Address Substitution in `SwapAllowlistExtension.beforeSwap` Breaks Per-User Allowlist for All Router-Mediated Swaps ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives the pool's immediate caller (`msg.sender` of `pool.swap`) as its `sender` argument. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, silently blocking every individually-allowlisted user who routes through the router.

## Finding Description

**Step 1 ŌĆö Router calls `pool.swap` as `msg.sender`**

In `MetricOmmSimpleRouter.exactInputSingle`, the original user address is stored only as the transient payer context; the pool call itself is issued by the router contract: [1](#0-0) 

The pool therefore sees `msg.sender == router`.

**Step 2 ŌĆö Pool forwards `msg.sender` (router) as `sender` to `_beforeSwap`** [2](#0-1) 

**Step 3 ŌĆö `ExtensionCalling._beforeSwap` encodes that router address and dispatches to the extension** [3](#0-2) 

**Step 4 ŌĆö Extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`** [4](#0-3) 

Here `msg.sender` = pool, `sender` = router. The lookup `allowedSwapper[pool][router]` is `false` unless the admin explicitly allowlisted the router, so the call reverts with `NotAllowedToSwap` even though `allowedSwapper[pool][user]` is `true`.

The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` ŌĆö all router entry points call `pool.swap` directly. [5](#0-4) 

## Impact Explanation

Any pool that deploys `SwapAllowlistExtension` with per-user allowlisting and expects users to route through `MetricOmmSimpleRouter` is immediately broken: every allowlisted user's swap reverts. The only workaround ŌĆö allowlisting the router address ŌĆö grants swap access to all users routing through the router, completely defeating the per-user access control. This constitutes broken core swap functionality (unusable swap flow) for the router path, meeting the contest's allowed impact threshold.

## Likelihood Explanation

No attacker action is required. The miscorrelation is structural and triggered by normal usage: any allowlisted user calling `exactInputSingle` (or any other router entry point) on a pool with `SwapAllowlistExtension` will be blocked. The condition is met on every such call.

## Recommendation

The pool should expose the original swap initiator to extensions separately from the immediate caller. The cleanest fix is to add a dedicated `swapInitiator` parameter to `pool.swap` (or encode it in `extensionData`) so the extension can check the true originating user. Alternatively, `SwapAllowlistExtension.beforeSwap` could accept a user address from `extensionData` accompanied by a signature, but that requires off-chain coordination. The root fix belongs in the pool's interface: extensions must receive both the immediate caller and the original initiator.
