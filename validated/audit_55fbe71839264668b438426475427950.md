### Title
Unvalidated Class Hash in `replace_class` Syscall Enables Permanent Freezing of L1 Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function does not validate that the new class hash supplied to the `replace_class` syscall corresponds to a declared contract class. A malicious contract deployer can replace a contract's class with an undeclared hash, making subsequent L1 handler executions impossible to include in a provable block. The sequencer is forced to mark those L1 handlers as reverted, which prevents `consume_l1_to_l2_message` from being called, leaving L1 funds permanently frozen in the bridge.

---

### Finding Description

**Root cause — `execute_replace_class` (syscall_impls.cairo lines 896–915):**

The function writes an arbitrary `class_hash` into `contract_state_changes` with no check that the hash was ever declared:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment is the codebase's own acknowledgement that this check is absent.

**Downstream failure — `execute_entry_point` (execute_entry_point.cairo lines 154–166):**

When the contract is subsequently called, the OS reads the (now-invalid) class hash from state and attempts to resolve it:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash          // undeclared hash → returns 0
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,                  // key = 0, not present → find_element panics
);
``` [2](#0-1) 

`find_element` is not a graceful failure path — it causes the Cairo OS execution to abort, making the block unprovable. The sequencer therefore cannot include any call to the broken contract in a valid block.

**L1 handler forced-revert path — `execute_l1_handler_transaction` (transaction_impls.cairo lines 383–390):**

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [3](#0-2) 

When `is_reverted` is TRUE the function returns **before** `consume_l1_to_l2_message` is reached (line 444). The L1→L2 message is therefore never consumed. [4](#0-3) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

1. A malicious contract (e.g., a fake bridge) deploys with an L1 handler.
2. The contract calls `replace_class` with an undeclared class hash.
3. Victims send L1→L2 messages to the contract, locking ETH/tokens in the L1 bridge.
4. The sequencer cannot execute the L1 handler (OS would abort); it is forced to mark it `is_reverted = TRUE`.
5. `consume_l1_to_l2_message` is never called; the L1 message remains unconsumed.
6. Victims' funds are permanently frozen in the L1 bridge contract.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Deploying a contract is an unprivileged operation on StarkNet.
- The `replace_class` syscall is callable by any contract on itself — no privileged role is required.
- The attack requires social engineering (presenting a malicious contract as a legitimate bridge), which is realistic given the prevalence of bridge exploits.
- The missing validation is self-documented via the TODO comment, indicating it is a known gap, not an edge case.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., that it was previously declared via a `declare` transaction). This is exactly what the existing TODO comment calls for. A `dict_read` on `contract_class_changes` with the new `class_hash` as key, followed by an `assert_not_zero` on the returned compiled class hash, would close the gap.

---

### Proof of Concept

1. **Deploy** a contract `MaliciousBridge` that implements an L1 handler (`handle_deposit`) and exposes a public function `poison()`.
2. **Call** `poison()` — inside, the contract issues `replace_class(class_hash=0xdeadbeef)` where `0xdeadbeef` is never declared. `execute_replace_class` accepts this without validation.
3. **Victim** calls the L1 bridge contract and sends 1 ETH, triggering an L1→L2 message to `MaliciousBridge.handle_deposit`.
4. **Sequencer** attempts to include the L1 handler. `execute_entry_point` calls `dict_read(contract_class_changes, 0xdeadbeef)` → returns 0. `find_element(..., key=0)` aborts the OS. The sequencer is forced to set `is_reverted = TRUE`.
5. **Result**: `consume_l1_to_l2_message` is never called. The L1 message stays unconsumed. The victim's 1 ETH is permanently locked in the L1 bridge.
