### Title
Felt Arithmetic Wrap-Around in `compute_max_possible_fee` Allows Zero-Fee Transaction Execution - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the fee cap for V3 transactions using unchecked felt (modular) arithmetic. Because `max_price_per_unit` is only validated to be non-negative (up to `(P-1)/2 ≈ 2^250`) while `max_amount` is bounded to `[0, 2^64-1]`, their product can silently wrap around the Cairo field prime `P`. An attacker can craft resource bounds whose products sum to exactly `0 mod P`, causing the OS to skip fee collection entirely and process the transaction for free.

---

### Finding Description

`compute_max_possible_fee` in `transaction_impls.cairo` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

All arithmetic here is felt arithmetic — modular reduction modulo the Cairo field prime `P = 2^251 + 17·2^192 + 1`. There are no range checks on the intermediate products or the final sum.

The only validation of `max_price_per_unit` occurs in `pack_resource_bounds` during transaction hash computation:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn` only enforces `max_price_per_unit ∈ [0, (P-1)/2]`. It does **not** bound the product `max_amount * max_price_per_unit`. With `max_amount` up to `2^64 - 1` and `max_price_per_unit` up to `(P-1)/2 ≈ 2^250`, the product can reach `~2^314`, which wraps modulo `P` to an arbitrary felt value.

`charge_fee` then uses the result directly:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

If `max_fee` wraps to `0`, the function returns immediately — no ERC-20 transfer is executed, no fee is charged, and the transaction is processed for free.

---

### Impact Explanation

**Critical — Direct loss of funds / High — Network halt.**

The sequencer's off-chain mempool logic computes fees as regular (non-modular) integers and sees a large `max_fee`, so it accepts and includes the transaction. The OS, however, computes the same expression as felt arithmetic and gets `0`, then skips fee collection. The sequencer processes the transaction but receives zero fee tokens. An attacker can repeat this indefinitely, flooding the network with zero-cost transactions, draining sequencer revenue, and causing a network halt through transaction spam.

---

### Likelihood Explanation

Any unprivileged user who can submit a V3 transaction can exploit this. The crafted resource bounds pass all existing validation checks (`assert_nn_le`, `assert_nn`, transaction hash verification). No special role, key, or operator access is required. The attack is deterministic and requires only knowledge of the Cairo field prime.

---

### Recommendation

Add explicit upper-bound range checks on `max_price_per_unit` in `pack_resource_bounds` (e.g., `assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1)`) so that the product `max_amount * max_price_per_unit` cannot exceed `(2^64 - 1) * (2^128 - 1) < 2^192 << P`. Additionally, add an overflow guard in `compute_max_possible_fee` itself — either by asserting the result is non-negative after each addition, or by computing the fee using `Uint256` arithmetic instead of bare felt arithmetic.

---

### Proof of Concept

Let `P = 2^251 + 17·2^192 + 1` (Cairo field prime).

Craft the following resource bounds for a V3 transaction:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `2` |
| `l1_gas_bounds.max_price_per_unit` | `(P - 1) / 2` |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas_bounds.max_amount` | `0` |
| `l1_data_gas_bounds.max_price_per_unit` | `0` |

**Validation checks all pass:**
- `assert_nn_le(2, 2^64 - 1)` ✓
- `assert_nn((P-1)/2)` ✓ — exactly at the non-negative boundary
- `assert_nn_le(1, 2^64 - 1)` ✓
- `assert_nn(1)` ✓
- `assert_nn_le(0, 2^64 - 1)` ✓
- `assert_nn(0)` ✓

**Fee computation in the OS (felt arithmetic):**

```
Term1 = 2 * (P-1)/2  =  P - 1  ≡  -1  (mod P)
Term2 = 1 * (1 + 0)  =  1
Term3 = 0 * 0        =  0

Sum = (P - 1) + 1 + 0 = P ≡ 0 (mod P)
```

`compute_max_possible_fee` returns `0`. `charge_fee` hits the `if (max_fee == 0) { return (); }` branch and exits without executing any ERC-20 transfer. The transaction is fully executed with zero fee paid. [4](#0-3) [5](#0-4) [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
