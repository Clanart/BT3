### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Validation, Enabling DataLayer Tree Root Corruption - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the `KeyAlreadyPresent` and `HashAlreadyPresent` guards that `MerkleBlob::insert` enforces for all but the first two items in a batch. An untrusted caller supplying a batch that contains a key already present in the tree (or duplicate keys within the batch itself) silently writes orphaned leaf nodes into the blob, producing an incorrect Merkle root hash. Any proof of inclusion or exclusion derived from that root is then forged relative to the actual committed state.

### Finding Description

`MerkleBlob::insert` enforces two uniqueness invariants before writing any node:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` calls `self.insert(…)` only for the first two items it pops from the input vector (the bootstrap case when the tree has ≤ 1 existing leaf). Every subsequent item is written directly to the blob via `insert_entry_to_blob` with no uniqueness check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { …, node: Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no duplicate guard
    indexes.push(new_leaf_index);
}
```

When a duplicate key is written this way the `BlockStatusCache` HashMap silently overwrites the old `key → index` mapping, leaving the original leaf node unreachable from the root but still physically present in the blob. The subtree built from the new leaves is then grafted onto the existing tree via `insert_subtree_at_key`, producing a root hash that does not reflect the true committed state. The same flaw applies to duplicate hashes within a single batch call.

The Python binding `py_batch_insert` exposes this path directly to any caller:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    …
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

### Impact Explanation

The DataLayer Merkle root hash is the commitment used to generate and verify proofs of inclusion and exclusion. After a `batch_insert` call that contains a key already present in the tree, the root hash is computed over a structurally corrupt tree (one that contains an orphaned leaf). Any `ProofOfInclusion` generated against this corrupt root either:

- Proves a key/value pair that is no longer the canonical entry for that key (the orphaned leaf), or
- Proves a key/value pair at a position that does not match what a clean tree would commit to.

A verifier checking the proof against the corrupt root will accept it, while a verifier holding the pre-corruption root will reject it. This is a forged-state proof scenario matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion API and is called during DataLayer delta synchronisation. A DataLayer peer that sends a delta containing a key already committed in the local tree will trigger this path. The Python binding is directly reachable from application code without any privilege requirement. The flaw is structural: the fast path was written for performance and the duplicate guards were simply never added to it.

### Recommendation

Add the same uniqueness checks to the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing write logic …
}
```

Alternatively, deduplicate the input vector against the existing cache before entering the bulk loop, and reject the call if any collision is detected.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Seed the tree with 3 leaves so batch_insert uses the fast (unchecked) path
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), h(i))

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with a key (KeyId(0)) already present in the tree
# The fast path (leaf_count > 1) skips the KeyAlreadyPresent guard
blob.batch_insert(
    [(KeyId(0), ValueId(99)), (KeyId(10), ValueId(10))],
    [h(99), h(10)],
)
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# Root hash has changed even though KeyId(0) was already committed
assert root_before != root_after, "tree root silently mutated by duplicate key"

# The tree now contains an orphaned leaf for KeyId(0) at the old index,
# and a new leaf at a different index; check_integrity() will report
# IntegrityLeafHashToIndexCacheLength or a cache/blob mismatch.
