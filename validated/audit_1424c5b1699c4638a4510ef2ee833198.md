### Title
Unvalidated `class_hash` in `replace_class` Syscall Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler accepts any arbitrary felt value as the new `class_hash` without verifying it corresponds to a previously declared class. This is the direct StarkNet analog of the Across V2 bug where a relayer can specify an invalid `repaymentChainId` that is never validated against an accepted set. A contract (or any contract it calls) can invoke `replace_class` with a non-existent class hash, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` in `syscall_impls.cairo`, the user-supplied `class_hash` from the syscall request is read and immediately written into `contract_state_changes` with no check that the hash exists in the `contract_class_changes` dictionary (i.e., that it was ever declared):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges this missing validation. The same flaw exists in the deprecated path `execute_replace_class` in `deprecated_execute_syscalls.cairo` (lines 307–329), which also performs no such check.

By contrast, the `execute_declare_transaction` flow correctly enforces that a class hash is the result of a valid Sierra class hash computation before writing it to `contract_class_changes`. The `replace_class` syscall skips this entire validation step.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an arbitrary invalid value (one with no corresponding compiled class in the OS's `compiled_class_facts` bundle), every future attempt to execute any entry point on that contract will fail at class resolution time. The contract becomes permanently non-executable. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irrecoverably frozen, as there is no mechanism to recover from an invalid class hash assignment.

---

### Likelihood Explanation

The syscall is directly callable by any deployed contract. A malicious contract can call `replace_class(0xdeadbeef)` (or any arbitrary felt) on itself or on a contract it controls. The attack requires no privileged access, no leaked keys, and no operator cooperation — only the ability to deploy and execute a contract, which is available to any unprivileged user.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, verify that the supplied `class_hash` exists as a key in `contract_class_changes` (i.e., it was declared in the current or a prior block). This is exactly what the TODO comment at line 898 calls for. The check should be added to both `execute_replace_class` in `syscall_impls.cairo` and the deprecated variant in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Attacker deploys `VaultContract` holding user funds (e.g., an ERC-20 balance in storage).
2. `VaultContract` calls the `replace_class` syscall with `class_hash = 0x1337` (an arbitrary felt with no declared class).
3. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0x1337` from the request. [1](#0-0) 
   - Skips the missing validation (acknowledged by the TODO). [2](#0-1) 
   - Writes `StateEntry(class_hash=0x1337, ...)` into `contract_state_changes`. [3](#0-2) 
4. In any subsequent block, any call to `VaultContract` attempts to resolve class hash `0x1337` in the compiled class facts bundle — it is not found.
5. Execution fails permanently. All funds in `VaultContract`'s storage are frozen.

The deprecated path has the identical missing check: [4](#0-3) 

For comparison, `execute_declare_transaction` correctly validates the class hash before committing it: [5](#0-4)
