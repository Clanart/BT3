### Title
Token-2022 MetadataPointer `mint` Field Not Verified in `parse_metadata_account` — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` validates that the passed metadata account's address matches the pointer stored in the Token-2022 `MetadataPointer` extension and that the account is owned by `MetaplexID`, but **never checks that `metadata.mint == self.mint.key()`**. An unprivileged attacker can create a Token-2022 mint whose `MetadataPointer` points to a legitimate Metaplex metadata account belonging to a different (victim) token, then call `log_metadata` to emit a Wormhole VAA that maps the attacker's mint address to the victim token's name and symbol. Relaying this VAA to NEAR's `deploy_token` deploys a NEAR token with the victim's identity but backed by the attacker's mint.

---

### Finding Description

In `process()`, when the token program is Token-2022, the code reads the `MetadataPointer` extension from the attacker-controlled mint: [1](#0-0) 

If the pointer address differs from the mint's own key (i.e., it is a "third-party" metadata account), `parse_metadata_account` is called with that pointer address: [2](#0-1) 

The two checks performed are:
1. `metadata.key() == address` — the passed account matches the pointer value.
2. `metadata.owner == &MetaplexID` — the account is owned by Metaplex.

There is **no check** that `metadata.mint == self.mint.key()`. The Metaplex `MetadataAccount` layout stores the associated mint at a fixed offset (bytes 1–33, after the 1-byte key discriminator), as confirmed by the test helper: [3](#0-2) 

The deserialized `MplMetadata` struct exposes a `mint` field that is never compared against `self.mint.key()`.

The resulting payload encodes the attacker's mint as `token` but the victim's name/symbol: [4](#0-3) 

On NEAR, `deploy_token_callback` only verifies the emitter is the known Solana factory — which it is, since the attacker used the legitimate bridge program — and then calls `deploy_token_internal` with the spoofed metadata: [5](#0-4) 

---

### Impact Explanation

A NEAR bridged token is deployed with:
- `token_address` = `M_attacker` (attacker's Solana mint)
- `name` / `symbol` = victim token's name and symbol

This creates a NEAR token that is visually indistinguishable from the victim token but is backed by the attacker's worthless mint. Users bridging from NEAR to Solana using this token receive `M_attacker` tokens, not the victim's tokens. This constitutes identity hijacking and enables phishing-based theft of bridged value. If the victim token has not yet been registered on NEAR, the attacker can front-run its legitimate registration entirely.

---

### Likelihood Explanation

- Creating a Token-2022 mint with an arbitrary `MetadataPointer` is permissionless on Solana.
- Calling `log_metadata` is a public, unpermissioned instruction.
- No privileged access, leaked keys, or colluding parties are required.
- The attack is locally testable end-to-end.

---

### Recommendation

After deserializing the Metaplex metadata account in `parse_metadata_account`, add a binding check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This ensures the Metaplex metadata account actually belongs to the mint being registered.

---

### Proof of Concept

1. Identify victim Token-2022 (or classic SPL) mint `M_victim` with a deployed Metaplex metadata PDA `Meta_victim = PDA([METADATA_SEED, MetaplexID, M_victim], MetaplexID)`.
2. Create a new Token-2022 mint `M_attacker` with a `MetadataPointer` extension set to `Meta_victim`.
3. Call `log_metadata` on the Solana bridge program with `mint = M_attacker`, `metadata = Meta_victim`.
4. `parse_metadata_account` passes both checks (address match, owner == MetaplexID) and returns `M_victim`'s name/symbol.
5. The emitted Wormhole VAA contains `token = M_attacker`, `name = <victim name>`, `symbol = <victim symbol>`.
6. Relay the VAA to NEAR `deploy_token`.
7. `deploy_token_callback` verifies the emitter is the known Solana factory ✓ and calls `deploy_token_internal`.
8. Assert: NEAR deploys a token with `M_victim`'s name/symbol mapped to `M_attacker`'s Solana address.
