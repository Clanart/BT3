### Title
Missing Class Hash Validation in `execute_replace_class` Enables Contract Re-deployment with Inherited Stale Storage — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`, `deploy_contract.cairo`)

---

### Summary

The `execute_replace_class` syscall handler performs no validation that the supplied `class_hash` is a declared class. A malicious contract can therefore call `replace_class(UNINITIALIZED_CLASS_HASH)` to reset its own `class_hash` to the sentinel "uninitialized" value while leaving `nonce = 0`. Because `deploy_contract` only guards re-deployment with `assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH` and `assert state_entry.nonce = 0`, and because it unconditionally inherits the existing `storage_ptr` into the new `StateEntry`, the same contract address can be re-deployed and will silently inherit all prior storage — the exact "struct-with-mapping not cleared on deletion" pattern from the external report.

---

### Finding Description

**Root cause 1 — missing validation in `execute_replace_class`**

In `syscall_impls.cairo` the new-syscall implementation of `replace_class` reads the caller-supplied `class_hash` and writes it directly into the state without any check:

```cairo
let class_hash = request.class_hash;
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
``` [1](#0-0) 

The inline TODO explicitly acknowledges the missing guard. The deprecated-syscall path in `deprecated_execute_syscalls.cairo` has the same defect with no TODO at all:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    ...
    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
``` [2](#0-1) 

Because `class_hash` is an unconstrained felt, passing `0` (i.e. `UNINITIALIZED_CLASS_HASH`) is syntactically valid and accepted.

**Root cause 2 — `deploy_contract` inherits `storage_ptr` without clearing it**

`deploy_contract` guards re-deployment with only two assertions and then copies the existing `storage_ptr` verbatim into the new `StateEntry`:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
assert state_entry.nonce = 0;

tempvar new_state_entry = new StateEntry(
    class_hash=constructor_execution_context.class_hash,
    storage_ptr=state_entry.storage_ptr,   // ← stale storage inherited
    nonce=0,
);
``` [3](#0-2) 

`StateEntry.storage_ptr` is a pointer to a `DictAccess` segment — the contract's full key-value storage. It is the direct analog of the "mapping inside the struct" from the external report. When the outer struct (`StateEntry`) is "deleted" by resetting `class_hash` to `UNINITIALIZED_CLASS_HASH`, the inner mapping (`storage_ptr`) is never zeroed.

**Combined effect**

After `replace_class(0)` the state entry for address X satisfies both guards (`class_hash == UNINITIALIZED_CLASS_HASH`, `nonce == 0`) while retaining all prior storage writes. A subsequent `deploy_contract` call to the same address passes both assertions and produces a new `StateEntry` that points to the old storage segment. The squashing logic in `squash_state_changes_inner` then faithfully squashes the inherited storage into the final state diff:

```cairo
local prev_state: StateEntry* = cast(state_changes.prev_value, StateEntry*);
local new_state: StateEntry* = cast(state_changes.new_value, StateEntry*);
...
let (local squashed_storage_ptr_end) = squash_dict(
    dict_accesses=prev_state.storage_ptr,
    dict_accesses_end=new_state.storage_ptr,
    squashed_dict=squashed_storage_ptr,
);
``` [4](#0-3) 

The stale storage is therefore committed to the global state root as if it were legitimate initial state for the re-deployed contract.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious contract deployer attracts user deposits (token balances, escrow entries, LP shares) into a contract at address X. After accumulating user funds in storage, the deployer calls `replace_class(0)` to reset the contract's class hash to `UNINITIALIZED_CLASS_HASH` while preserving all storage. The deployer then re-deploys an upgraded (or identical) contract at the same address. The re-deployed contract inherits the full prior storage, including all user balances. The deployer can then invoke withdrawal or transfer functions against those inherited balances to drain user funds. Because the OS commits the inherited storage to the state root without any anomaly, the theft is indistinguishable from a legitimate state transition at the proof level.

---

### Likelihood Explanation

**Medium.** The attacker role is an unprivileged contract deployer — no privileged key or operator access is required. The only prerequisite is that users interact with the malicious contract before the reset. This is realistic in any DeFi or token-contract scenario. The missing validation is a single-line omission explicitly flagged by a TODO in the production code, confirming it is a known gap rather than an intentional design choice.

---

### Recommendation

1. **Immediate**: In both `execute_replace_class` implementations, assert that the supplied `class_hash` is non-zero and present in `contract_class_changes` (i.e., has been declared in the current or a prior block). This closes the `replace_class(0)` vector.

2. **Defence-in-depth**: In `deploy_contract`, add an assertion that `state_entry.storage_ptr` points to an empty segment (no prior writes) before inheriting it. This prevents stale storage from being silently adopted even if another reset path is discovered.

---

### Proof of Concept

```
Block N:
  Tx 1 (Deploy):  deploy MaliciousEscrow at address X
                  (class_hash=C, salt=S, calldata=[])
                  → StateEntry{class_hash=C, nonce=0, storage={}}

  Tx 2..N (User): users call deposit() on X
                  → StateEntry{class_hash=C, nonce=0,
                               storage={user_A: 1000, user_B: 500}}

  Tx N+1 (Attacker): calls reset() on X, which internally calls
                     replace_class(class_hash=0)
                     → execute_replace_class writes:
                       StateEntry{class_hash=0, nonce=0,
                                  storage={user_A: 1000, user_B: 500}}
                     No validation → accepted by OS.

Block N+1:
  Tx 1 (Deploy):  deploy MaliciousEscrow v2 at address X
                  (same class_hash=C, salt=S, calldata=[])
                  deploy_contract checks:
                    assert state_entry.class_hash == UNINITIALIZED_CLASS_HASH (0) ✓
                    assert state_entry.nonce == 0 ✓
                  new_state_entry = StateEntry{
                    class_hash=C,
                    storage_ptr=<old storage with user_A:1000, user_B:500>,
                    nonce=0
                  }
                  → stale storage committed to state root.

  Tx 2 (Attacker): calls withdraw(user_A, 1000) on re-deployed X
                   → drains user_A's funds from inherited storage.
```
