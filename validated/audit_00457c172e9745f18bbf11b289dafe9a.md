### Title
Unvalidated Arbitrary Token Address in `logMetadata` Enables Decimal-Corrupted Token Deployment Across Chains — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-bridge/src/lib.rs`)

---

### Summary

The `logMetadata` function on EVM and the `log_metadata` function on NEAR both accept any arbitrary token address with no access control and no validation of the token's properties. An unprivileged attacker can supply a malicious token contract that returns extreme decimal values (e.g., 255), triggering the full cross-chain token deployment pipeline. The resulting token is registered in NEAR's bridge with corrupted decimal settings, causing the bridge's normalization math to produce zero or overflowing amounts for any user who subsequently bridges that token, permanently misdirecting their funds.

---

### Finding Description

**Root cause — EVM side (`OmniBridge.sol:224-232`):**

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name  = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals       = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
```

There is no `onlyRole`, no `isBridgeToken` check, and no `_normalizeDecimals` call. Any caller may pass any `tokenAddress`. The raw `uint8 decimals` (0–255) is forwarded verbatim into the Wormhole message via `logMetadataExtension`. [1](#0-0) 

**Root cause — NEAR side (`lib.rs:316-327`):**

```rust
#[pause(except(roles(Role::DAO)))]
pub fn log_metadata(&self, token_id: &AccountId) -> Promise {
    ext_token::ext(token_id.clone())
        .with_static_gas(LOG_METADATA_GAS)
        .ft_metadata()
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(LOG_METADATA_CALLBACK_GAS)
                .with_attached_deposit(env::attached_deposit())
                .log_metadata_callback(token_id),
        )
}
```

No access control beyond the global pause. Any NEAR account ID is accepted; `ft_metadata()` is called on it and the result is signed by MPC without any decimal bounds check. [2](#0-1) 

**Propagation — NEAR `deploy_token_callback` (`lib.rs:1147-1174`):**

The callback verifies only that the emitter is a registered factory, then passes `metadata.decimals` directly to `deploy_token_internal` with no capping:

```rust
self.deploy_token_internal(
    chain,
    &metadata.token_address,
    BasicMetadata {
        name: metadata.name,
        symbol: metadata.symbol,
        decimals: metadata.decimals,   // ← raw, uncapped
    },
    attached_deposit,
)
``` [3](#0-2) 

**Storage — `deploy_token_internal` (`lib.rs:2413-2419`):**

Both `decimals` and `origin_decimals` are set to the same raw value:

```rust
self.add_token(
    &token_id,
    token_address,
    metadata.decimals,        // decimals
    metadata.decimals,        // origin_decimals — same raw value
);
``` [4](#0-3) 

**Contrast with `deployToken` on EVM (`OmniBridge.sol:159`):**

`deployToken` (the downstream consumer of the signed metadata) *does* normalize decimals:

```solidity
uint8 decimals = _normalizeDecimals(metadata.decimals);
``` [5](#0-4) 

This creates a cross-chain mismatch: the EVM-deployed token gets `decimals = 18` (capped), while the NEAR-registered entry retains `decimals = 255`. The normalization functions `normalize_amount` and `denormalize_amount` use `10^(24 - origin_decimals)`. With `origin_decimals = 255`, the exponent is `24 - 255 = -231`, which in integer arithmetic rounds every bridged amount to **zero** (funds lost) or causes overflow on the return path.

---

### Impact Explanation

**Impact: High — Balance/decimal/accounting corruption that misdirects value.**

Any user who bridges a token deployed via a malicious `logMetadata` call will have their funds permanently lost:

- **EVM → NEAR direction**: `normalize_amount(amount, {decimals:255, origin_decimals:255})` computes `amount × 10^(24−255)`, which integer-divides to **0** for any realistic amount. The user's EVM tokens are locked/burned; they receive nothing on NEAR.
- **NEAR → EVM direction**: the inverse computation overflows, producing an astronomically large value that the EVM token contract cannot mint, causing the transaction to revert and leaving the user's NEAR tokens locked in the bridge.

The EVM/NEAR decimal mismatch (EVM token: 18 decimals; NEAR entry: 255) is permanent once the token is registered, because `add_token` enforces uniqueness and there is no admin path to correct a deployed token's decimal entry without a DAO migration. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- `logMetadata` on EVM requires no role, no stake, and no prior token registration. Any EOA can call it with a freshly deployed malicious ERC20.
- The NEAR `log_metadata` similarly requires no role.
- The relayer infrastructure automatically picks up `LogMetadata` events and submits proofs to NEAR, so the attacker does not need to operate a relayer.
- The only cost is deploying a cheap malicious ERC20 and paying gas for `logMetadata`.
- The attack is most effective against tokens that have not yet been bridged; it cannot overwrite existing registered tokens due to the `TokenExists` guard in `add_token`.

---

### Recommendation

1. **EVM `logMetadata`**: Add a decimal bounds check before emitting, consistent with `_normalizeDecimals` used in `deployToken`:
   ```solidity
   function logMetadata(address tokenAddress) external payable {
       uint8 rawDecimals = IERC20Metadata(tokenAddress).decimals();
       require(rawDecimals <= 18, "ERR_INVALID_DECIMALS");
       // ...
   }
   ```
2. **NEAR `log_metadata_callback`**: Validate `metadata.decimals` before signing:
   ```rust
   require!(metadata.decimals <= 24, BridgeError::InvalidMetadata.as_ref());
   ```
3. **`deploy_token_internal`**: Apply a decimal cap before calling `add_token`, mirroring the EVM normalization logic.
4. **Optional access control**: Restrict `logMetadata` / `log_metadata` to a `MetadataManager` role or require the token to pass a basic sanity check (e.g., non-zero total supply, non-zero address).

### Proof of Concept

**Step 1 — Deploy malicious ERC20 on EVM:**
```solidity
contract MaliciousToken {
    function name()     external pure returns (string memory) { return "USD Coin"; }
    function symbol()   external pure returns (string memory) { return "USDC"; }
    function decimals() external pure returns (uint8)         { return 255; }
}
```

**Step 2 — Call `logMetadata` (no access control):**
```solidity
omniBridge.logMetadata{value: wormholeFee}(address(maliciousToken));
```

**Step 3 — Relayer automatically submits the Wormhole VAA to NEAR `deploy_token`.**

**Step 4 — NEAR registers the token with `decimals = 255, origin_decimals = 255`.**

**Step 5 — Victim bridges 1,000 units of this token from EVM to NEAR:**
- NEAR calls `normalize_amount(1000, {decimals:255, origin_decimals:255})`
- Computes `1000 × 10^(24−255)` → integer result = **0**
- Victim receives 0 tokens on NEAR; EVM tokens are burned/locked permanently.
