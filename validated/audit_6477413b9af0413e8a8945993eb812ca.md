### Title
Gateway L2 Gas Price Admission Check Uses Stale Previous-Block Price, Not Next-Block Execution Price — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` gates admission on the *previous* committed block's L2 gas price, but every admitted transaction executes in the *next* block whose L2 gas price is computed by the EIP-1559 `calculate_next_base_gas_price` formula. The gap between the two prices is the direct sequencer analog of the Market protocol's missing slippage bound: a value is read at admission time, the underlying state (gas price) changes before execution, and the check is therefore wrong in both directions.

---

### Finding Description

In `validate_resource_bounds`, the gateway fetches the last committed block's `strk_gas_prices.l2_gas_price` and computes a threshold:

```
threshold = (min_gas_price_percentage / 100) * previous_block_l2_gas_price
```

A transaction is admitted if and only if `tx.max_price_per_unit >= threshold`.

The code itself carries an unresolved acknowledgement of the problem:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

The actual execution block is built by the batcher/orchestrator using `calculate_next_l2_gas_price_for_fin`, which applies the EIP-1559 formula to the previous block's price and gas usage:

```rust
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    ...
) -> GasPrice {
    ...
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, min_gas_price)
}
``` [2](#0-1) 

The batcher's `perform_pre_validation_stage` then calls `check_fee_bounds`, which compares `max_price_per_unit` against the *next* block's gas price (the one actually in the `BlockContext`):

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(
        ResourceBoundsError::MaxGasPriceTooLow { ... }
    );
}
``` [3](#0-2) 

The gateway also runs `run_validate_entry_point` with a `BlockContext` built from the previous block's gas prices (only the block number is incremented):

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [4](#0-3) 

So the gateway's entire stateful validation — both the threshold check and the blockifier `__validate__` execution — runs against the wrong gas price.

---

### Impact Explanation

**Accepts invalid transactions (High — Mempool/gateway admission):** When the network is congested, the next block's L2 gas price rises above the previous block's price. A transaction with `max_price_per_unit` that satisfies `threshold * P_prev` but is below `P_next` passes the gateway and enters the mempool, yet will be rejected by the batcher's `check_fee_bounds` with `MaxGasPriceTooLow`. The mempool accumulates transactions that can never be sequenced.

**Rejects valid transactions (High — Mempool/gateway admission):** When congestion falls, `P_next < P_prev`. A transaction with `max_price_per_unit` between `threshold * P_prev` and `P_prev` fails the gateway threshold even though it would satisfy the batcher's check at `P_next`. Valid transactions are incorrectly denied admission.

---

### Likelihood Explanation

The EIP-1559 formula can move the price by up to `price / gas_price_max_change_denominator` per block: [5](#0-4) 

With a denominator of 8 this is ±12.5 % per block. Combined with a `min_gas_price_percentage` below 100 % (the default), the exploitable window between the gateway threshold and the actual next-block price is wide. Any unprivileged user can craft a transaction whose `max_price_per_unit` falls in that window and trigger either direction of the mismatch deterministically.

---

### Recommendation

Replace `previous_block_l2_gas_price` in `validate_resource_bounds` with the computed next-block L2 gas price. The orchestrator already exposes `calculate_next_l2_gas_price_for_fin`; the gateway should call it (or read `next_l2_gas_price` from the block header once it is stored there, as the TODO notes) so that the admission check and the execution check operate on the same price.

---

### Proof of Concept

1. Read the current committed block's L2 gas price `P_prev` and gas usage `G`.
2. Compute `P_next = calculate_next_base_gas_price(P_prev, G, gas_target, min_price)`. If the block was above the gas target, `P_next > P_prev`.
3. Let `pct = min_gas_price_percentage / 100`. Choose `max_price_per_unit = pct * P_prev + 1` (just above the gateway threshold).
4. If `P_next > pct * P_prev + 1`, submit the transaction. The gateway admits it (`max_price_per_unit >= threshold`). The batcher rejects it (`max_price_per_unit < P_next`). The transaction occupies a mempool slot indefinitely.
5. For the reverse: choose `max_price_per_unit = pct * P_prev - 1` when `P_next < pct * P_prev - 1`. The gateway rejects a transaction that the batcher would have accepted.

pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],

    if let Some(override_value) = override_l2_gas_price_fri {

            current_l2_gas_price.0

    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, min_gas_price)
}
