### Title
Router's `pay()` WETH path silently consumes stranded ETH to settle any caller's swap, bypassing payer's WETH pull entirely — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` checks the router's native ETH balance before pulling WETH from the designated payer. When the router holds ETH ≥ the owed amount, it wraps that ETH and forwards it to the pool without touching the payer's WETH allowance at all. Because every swap entry-point is `payable` with no enforcement that `msg.value == 0` when `tokenIn != WETH`, ETH can be stranded on the router. Any subsequent WETH swap drains that stranded ETH as payment, giving the attacker free swap output while the original depositor loses their ETH.

---

### Finding Description

In `PeripheryPayments.pay()` (lines 73–84), the WETH branch is:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `nativeBalance >= value` the payer is **completely bypassed**: the router wraps its own ETH and pays the pool. The payer needs zero WETH balance and zero allowance.

ETH reaches the router and stays there because every public swap function is `external payable` with no guard requiring `msg.value == 0` when `tokenIn != WETH`:

- `exactInputSingle` — `external payable`, no `msg.value` check
- `exactInput` — `external payable`, no `msg.value` check
- `exactOutputSingle` — `external payable`, no `msg.value` check
- `exactOutput` — `external payable`, no `msg.value` check

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH sends; it does **not** prevent ETH arriving via `msg.value` in a payable call. Once stranded, `refundETH()` sends only to `msg.sender`, so a different caller cannot recover it — but they can consume it via a WETH swap.

---

### Impact Explanation

**Direct loss of user principal.** A victim who accidentally sends ETH with a non-WETH swap (or who sends excess ETH in a multicall without a trailing `refundETH`) loses that ETH to any attacker who immediately calls a WETH swap of equal or smaller size. The attacker receives real swap output tokens while paying nothing. Loss is 100 % of the stranded ETH, bounded only by the amount the victim left on the router. Exceeds both the Critical (>20 %, >$100) and High (>1 %, >$10) thresholds for any non-trivial ETH amount.

---

### Likelihood Explanation

**Medium–High.** The trigger requires ETH to be stranded on the router, which happens through ordinary user error:

1. A user calls `exactInputSingle{value: X}(tokenIn = tokenA)` where `tokenA != WETH` — the swap succeeds, X ETH is silently stranded.
2. A user batches a WETH swap in `multicall{value: X}` but omits the trailing `refundETH()` call — partial ETH remains.

Both patterns are realistic for wallets and aggregators that compose multicalls. An attacker watching the mempool can front-run the victim's transaction or simply call a WETH swap in the same block. No privileged role is required; any EOA can trigger the exploit.

---

### Recommendation

1. **Enforce `msg.value == 0` when `tokenIn != WETH`** in each swap entry-point, or add a single guard in `_justPayCallback` / `_exactOutputIterateCallback` that reverts if `address(this).balance > 0` and `tokenToPay != WETH`.
2. **Restrict the ETH-wrapping path to `payer == address(this)`** (mid-path hops) only. When the payer is an external EOA and `token == WETH`, always use `safeTransferFrom(payer, ...)` regardless of the router's ETH balance.
3. **Alternatively**, track per-caller ETH deposits in transient storage and only allow each caller's own deposited ETH to be wrapped on their behalf.

---

router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    pool:            address(tokenA_tokenB_pool),

    tokenOut:        address(tokenB),
    zeroForOne:      true,
    amountIn:        1_000e18,          // tokenA, pulled via safeTransferFrom
    amountOutMinimum: 0,
    recipient:       victim,
    deadline:        block.timestamp + 1,

    extensionData:   ""

    extensionData:   ""
}));
