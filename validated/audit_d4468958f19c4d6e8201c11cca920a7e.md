Audit Report

## Title
Cross-User ETH Consumption in `PeripheryPayments.pay` WETH Branch Causes Direct Loss of Trapped ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses `address(this).balance` — the contract's entire native ETH balance — when deciding how much ETH to wrap for a WETH payment. Because multiple users can leave ETH in the contract across separate transactions via any `payable` entry point, a subsequent caller's WETH payment is silently subsidized by a prior user's unrefunded ETH, consuming that ETH without the prior user's consent. A secondary path allows any caller to drain the full ETH balance via the unguarded `refundETH()`.

## Finding Description

The `pay()` function at [1](#0-0)  reads `address(this).balance` without any per-caller scoping. When `token == WETH` and `payer != address(this)`, it wraps however much ETH the contract holds before pulling the remainder from the payer via `safeTransferFrom`.

ETH enters the contract legitimately via `msg.value` on any `payable` entry point. The `receive()` guard at [2](#0-1)  only blocks *direct* ETH sends from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on `addLiquidityExactShares` [3](#0-2) , `multicall` [4](#0-3) , `unwrapWETH9`, `sweepToken`, or `refundETH`.

**Exploit flow:**
1. User A calls `addLiquidityExactShares{value: 1 ether}(...)` for a WETH pool that only needs 0.6 ETH. `pay()` wraps exactly 0.6 ETH; 0.4 ETH remains in the contract.
2. User A omits a `refundETH` call (or calls the function directly, not via multicall). The 0.4 ETH sits in the contract.
3. User B calls `addLiquidityExactShares(...)` for a WETH pool needing 1 ETH. `metricOmmModifyLiquidityCallback` fires and calls `pay(WETH, userB, pool, 1e18)`.
4. `nativeBalance = address(this).balance` = 0.4 ETH (user A's residual). The `else if (nativeBalance > 0)` branch at [5](#0-4)  wraps user A's 0.4 ETH and pulls only 0.6 ETH from user B.
5. User A's 0.4 ETH is permanently consumed for user B's deposit with no compensation.

**Secondary path:** `refundETH()` at [6](#0-5)  has no access control and sends `address(this).balance` to `msg.sender`. Any caller can drain all accumulated ETH from the contract at any time.

## Impact Explanation

Direct, unrecoverable loss of user principal. User A's ETH is consumed for user B's liquidity deposit; user A receives no LP shares, no WETH, and no refund. The pool receives the correct total WETH so no pool invariant is broken, but the loss to user A is real and permanent. The `refundETH()` path enables outright theft of any ETH balance by an arbitrary third party. Both paths meet the Critical/High direct loss of user principal threshold.

## Likelihood Explanation

The standard multicall pattern (`addLiquidityExactShares` + `refundETH` in one multicall) is safe. However, any user who calls `addLiquidityExactShares` directly with excess `msg.value` — a natural usage given the function is `payable` — leaves residual ETH. A front-runner monitoring the mempool can observe the unrefunded ETH and immediately call `refundETH()` to steal it, or simply wait for the next WETH payment to consume it. No special permissions, malicious pools, or non-standard tokens are required.

## Recommendation

Track per-caller ETH contributions using transient storage (analogous to the existing `_setPayContext` / `_loadPayContext` pattern). In `pay()`, only consume up to the amount the current `payer` deposited in the current transaction. Alternatively, restrict the WETH branch to only use `address(this).balance` when `payer == address(this)` (mid-path), and always use `safeTransferFrom` for external payers regardless of the contract's ETH balance, leaving ETH management entirely to explicit `unwrapWETH9` / `refundETH` calls. Additionally, add access control to `refundETH()` so only the original depositor (tracked via transient storage) can reclaim their ETH.

## Proof of Concept

```solidity
// Foundry test sketch
function test_crossUserETHConsumption() public {
    // User A sends 1 ETH; pool only needs 0.6 ETH; 0.4 ETH stays in adder
    vm.prank(userA);
    adder.addLiquidityExactShares{value: 1 ether}(pool, salt, deltas, 1e18, 0, "");
    assertEq(address(adder).balance, 0.4 ether);

    // User B calls addLiquidityExactShares for same WETH pool needing 1 ETH
    uint256 userBWETHBefore = weth.balanceOf(userB);
    vm.prank(userB);
    adder.addLiquidityExactShares(pool, salt, deltas2, 1e18, 0, "");

    // User B was only charged 0.6 WETH (1 - 0.4), not 1 WETH
    assertEq(userBWETHBefore - weth.balanceOf(userB), 0.6 ether);
    // User A's 0.4 ETH is gone
    assertEq(address(adder).balance, 0);

    // Alternative: attacker simply calls refundETH() to steal user A's ETH
    vm.prank(attacker);
    adder.refundETH(); // drains 0.4 ETH to attacker with no checks
}
```
