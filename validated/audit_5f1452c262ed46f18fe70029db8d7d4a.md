### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Unbacked Token Minting on NEAR — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol:initTransfer` calls `safeTransferFrom(msg.sender, address(this), amount)` for native ERC20 tokens, then emits `InitTransfer` with the caller-supplied `amount` — without verifying the actual balance change. For fee-on-transfer tokens, the bridge receives `amount - fee_on_transfer` but records `amount` in the event. The NEAR bridge consumes this event to mint or unlock `amount` tokens, creating unbacked supply and breaking the 1:1 collateralization invariant.

---

### Finding Description

In `OmniBridge.sol:initTransfer`, the non-bridge, non-custom-minter ERC20 path is:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // requested amount, not actual received
    );
}
```

followed immediately by:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ← uses input amount, not actual balance delta
    fee,
    nativeFee,
    recipient,
    message
);
```

`safeTransferFrom` does not return the actual transferred amount; it only reverts on failure. The bridge never measures the balance before and after the call. For a fee-on-transfer ERC20 token, the contract receives `amount - transfer_fee` but the `InitTransfer` event records `amount`. The NEAR bridge's `fin_transfer` flow reads `amount` from the proven event and uses it to unlock or mint that many tokens to the recipient.

The analog to the Lido report is exact: Lido called `withdrawRewards(maxAmount)`, ignored the returned actual amount, and used `maxAmount` for its buffered-ETH counter. Here, `initTransfer` calls `safeTransferFrom(…, amount)`, cannot observe the actual received amount, and uses `amount` for the cross-chain accounting record (the emitted event).

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

After a successful `initTransfer` with a fee-on-transfer token:

- EVM bridge holds: `amount - transfer_fee` tokens
- NEAR bridge mints/unlocks: `amount` tokens

The difference (`transfer_fee`) is unbacked. An attacker who then bridges those NEAR tokens back to EVM via `fin_transfer` will drain `amount` tokens from the EVM bridge, but the bridge only holds `amount - transfer_fee`. Repeated cycles drain the bridge's reserves for that token, or cause the final redemption to fail for legitimate users (permanent lock of their funds).

This matches the allowed impact: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

**Medium.** For the exploit to complete end-to-end:

1. A fee-on-transfer ERC20 token must be registered on the NEAR bridge (i.e., have an entry in `token_address_to_id`). Registration flows through `log_metadata` → MPC signing of the metadata payload. The MPC signers sign name/symbol/decimals — they have no visibility into the token's transfer mechanics and cannot distinguish a fee-on-transfer token from a standard one.
2. Any unprivileged user can then call `initTransfer` with that token address and any `amount`.
3. The NEAR relayer submits the proof and `fin_transfer` mints the inflated amount.

Fee-on-transfer tokens are a well-established ERC20 pattern (e.g., USDT on some chains, STA, REFLECT-style tokens). If any such token is ever registered — intentionally or by oversight — the accounting divergence is immediately exploitable by any bridge user.

---

### Recommendation

Measure the actual balance delta around `safeTransferFrom` and use that value in the emitted event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint128 actualReceived = uint128(
    IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore
);
// replace `amount` with `actualReceived` in the emit and downstream logic
```

This mirrors the Lido fix: capture the return value (or balance delta) and use the actual received quantity for all accounting, not the caller-supplied maximum.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FoT` that deducts 10% on every transfer.
2. Register `FoT` with the Omni Bridge (via `log_metadata` + MPC signing of metadata).
3. Attacker holds 1000 `FoT` on EVM. Calls:
   ```solidity
   omniBridge.initTransfer(FoT, 1000, 0, 0, nearRecipient, "");
   ```
4. Bridge receives 900 `FoT` (10% fee deducted). Event emits `amount = 1000`.
5. NEAR relayer submits proof; NEAR bridge mints 1000 wrapped-`FoT` to `nearRecipient`.
6. Attacker bridges 1000 wrapped-`FoT` back to EVM via `fin_transfer`.
7. EVM bridge attempts to release 1000 `FoT` but only holds 900 → either the attacker drains other depositors' funds or the transaction reverts, permanently locking legitimate users' tokens.

The root cause is at: [1](#0-0) 

where the actual received amount is never captured, and at: [2](#0-1) 

where the unverified caller-supplied `amount` is recorded as the canonical transfer amount in the cross-chain event.
