### Title
Unauthenticated `CommitBlockRequest` on `RemoteCommitterServer` Injects Arbitrary State Diff and Corrupts Global State Root — (`File: crates/apollo_committer/src/committer.rs`, `crates/apollo_infra/src/component_server/remote_component_server.rs`)

---

### Summary

The `RemoteComponentServer` that fronts the committer component accepts `CommitBlockRequest` and `RevertBlockRequest` messages from any TCP peer without any caller authentication. In all distributed and hybrid production deployments the committer binds to `0.0.0.0`. Any actor with network access to that port can inject an arbitrary `ThinStateDiff` at the current committed height, causing the committer to compute and permanently store a wrong Patricia-Merkle-Tree global root and a wrong `StateDiffCommitment`. That wrong root is returned to the batcher and written into the block header, breaking the state-root commitment invariant.

---

### Finding Description

**Root cause — no authentication in `RemoteComponentServer`**

`remote_component_server_handler` deserialises every inbound HTTP body and forwards it directly to the local component client. There is no IP allowlist, no shared secret, no mutual-TLS, and no caller-identity field anywhere in the handler: [1](#0-0) 

The handler logs the peer address but performs zero authorisation before dispatching the request.

**Committer exposes `CommitBlock` and `RevertBlock` over this unauthenticated channel**

`RemoteCommitterServer` is a type alias for `RemoteComponentServer<CommitterRequest, CommitterResponse>`: [2](#0-1) 

Both `CommitterRequest::CommitBlock` and `CommitterRequest::RevertBlock` are dispatched without any additional identity check.

**Production deployments bind the committer to `0.0.0.0`**

Every distributed and hybrid deployment config sets `remote_server_config.bind_ip: "0.0.0.0"`: [3](#0-2) [4](#0-3) 

**`commit_block_inner` writes attacker-controlled data as the authoritative global root**

When `height == self.offset` (the normal commit path), the function:
1. Accepts any `ThinStateDiff` supplied by the caller.
2. Computes the Patricia-Merkle-Tree root from that diff.
3. Writes `ForestMetadataType::StateRoot`, `ForestMetadataType::StateDiffHash`, and `ForestMetadataType::CommitmentOffset` to the forest storage.
4. Returns the computed `global_root` to the caller. [5](#0-4) 

**`verify_state_diff_hash: true` does not block the attack**

The flag only rejects a `CommitBlockRequest` where `state_diff_commitment` is `Some(X)` and `X ≠ hash(state_diff)`. An attacker who sets `state_diff_commitment: None` bypasses this check entirely — the commitment is then silently computed from the attacker-supplied diff: [6](#0-5) 

The production default is `verify_state_diff_hash: true`: [7](#0-6) 

---

### Impact Explanation

An attacker who can reach the committer's TCP port sends:

```
CommitBlockRequest {
    state_diff: <arbitrary ThinStateDiff>,
    state_diff_commitment: None,   // bypasses verify_state_diff_hash
    height: <current committer offset>,
}
```

The committer:
- Computes a wrong Patricia-Merkle-Tree root from the injected diff.
- Stores it as `ForestMetadataType::StateRoot(height)` — the authoritative global root for that block.
- Stores the matching (but wrong) `StateDiffHash`.
- Advances `CommitmentOffset` to `height + 1`.

The batcher subsequently reads this wrong `global_root` from `CommitBlockResponse` and passes it to `set_global_root_and_block_hash`, embedding the wrong root in the block header. Any subsequent proof generation or state-sync that reads the forest storage receives the corrupted root. This directly satisfies: **Critical — Wrong state … storage value … from blockifier/syscall/execution logic for accepted input.**

Additionally, when the legitimate batcher then sends its own `CommitBlockRequest` for the same height, the committer detects `height < self.offset` and compares the legitimate `StateDiffCommitment` against the attacker-written one. They differ, so the batcher receives `CommitterError::InvalidStateDiffCommitment` and cannot commit the real block.

---

### Likelihood Explanation

In distributed and hybrid Kubernetes deployments the committer is a `ClusterIP` service. Any pod co-located in the same cluster namespace — including a compromised sidecar, a misconfigured workload, or a pod in a shared tenant cluster — can reach the committer's port. The committer is not a P2P gossip peer; it is an internal RPC server that is architecturally expected to receive only batcher-originated requests. The absence of any application-layer authentication means the only barrier is network topology, which is not a security guarantee.

---

### Recommendation

1. **Add mutual-TLS or a shared-secret header** to `RemoteComponentServer` so that only the authorised batcher client can connect.
2. Alternatively, restrict the committer's `bind_ip` to the loopback or a specific internal CIDR and enforce Kubernetes `NetworkPolicy` rules that allow only the batcher pod to reach the committer port.
3. As a defence-in-depth measure, require `state_diff_commitment` to always be `Some(...)` (never `None`) in production, so the hash check is always enforced.

---

### Proof of Concept

```
# Assume committer is reachable at COMMITTER_HOST:COMMITTER_PORT
# Current committer offset is N (discoverable from logs or block explorers)

# Craft a CommitBlockRequest with an empty (but valid) ThinStateDiff
# and state_diff_commitment = None, height = N

POST http://COMMITTER_HOST:COMMITTER_PORT/
Content-Type: application/octet-stream
Body: bincode-serialised CommitterRequest::CommitBlock(CommitBlockRequest {
    state_diff: ThinStateDiff::default(),   // all-empty diff
    state_diff_commitment: None,            // bypasses verify_state_diff_hash
    height: BlockNumber(N),
})

# Expected response: CommitterResponse::CommitBlock(Ok(CommitBlockResponse { global_root: <wrong_root> }))
# Effect: forest storage now contains wrong_root as the state root for block N,
#         and CommitmentOffset is advanced to N+1.
# The legitimate batcher's subsequent CommitBlock(N) will fail with
# CommitterError::InvalidStateDiffCommitment, and the block header will
# carry the attacker-injected global root.
```
