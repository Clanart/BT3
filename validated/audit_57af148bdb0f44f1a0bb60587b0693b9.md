### Title
Reverted-Execution DA Gas Included in `recommended_fee` for `AllResources` Bounds Causes Incorrect Fee Overcharge and Inconsistent Receipt/State-Diff — (`crates/blockifier/src/fee/fee_checks.rs`)

---

### Summary

When a V3 transaction with `AllResources` resource bounds executes successfully but then fails the post-execution resource-bounds check (e.g., L2 gas overdraft), the `recommended_fee` charged on revert is computed using the L1DataGas from the **full execution state diff** (validate + execute). However, the execution state changes are immediately aborted afterward, so the committed state diff contains only validation-phase storage writes. The user is charged for DA gas that corresponds to reverted storage writes that never appear in the committed state diff, producing an incorrect fee, an inconsistent `(fee, gas)` pair in the final receipt, and a wrong fee-token balance update in the block's state diff.

---

### Finding Description

**Root cause — `FeeCheckReport::from_fee_check_error` in `fee_checks.rs`:** [1](#0-0) 

When `MaxGasAmountExceeded` fires for an `AllResources` transaction, the recommended fee is:

```
gas_for_fee_charge.l1_data_gas = min(max_l1_data_gas_bound, actual_gas.l1_data_gas)
```

The `actual_gas` argument is `tx_receipt.gas`, taken directly from the receipt that was built over the **squashed validate+execute state diff**: [2](#0-1) 

That receipt's `l1_data_gas` is derived from `StateResources::new(state_changes, ...)` where `state_changes` includes every storage write made during execution: [3](#0-2) 

Immediately after computing `recommended_fee`, the execution state is aborted: [4](#0-3) 

The final receipt is then assembled as:

```rust
let tx_receipt = TransactionReceipt {
    fee: post_execution_report.recommended_fee(),  // includes execution DA gas
    ..get_revert_receipt()                          // gas field: validation DA gas only
};
```

`get_revert_receipt()` uses only `validate_state_cache.to_state_diff()` — no execution storage writes: [5](#0-4) 

**Result:** `receipt.fee` is computed from execution DA gas (reverted), while `receipt.gas.l1_data_gas` reflects only validation DA gas. The invariant `fee == get_fee_by_gas_vector(receipt.gas)` is broken. The fee-token balance deduction in the committed state diff is larger than the actual DA cost of the committed state changes.

---

### Impact Explanation

The fee token balance update written to the state diff (and therefore to the Patricia trie, block commitment, and receipt commitment) is inflated by the DA cost of storage writes that were reverted. Concretely:

- A user who writes N storage slots during execution and then hits an L2 gas overdraft is charged `min(max_l1_data_gas, DA_gas_for_N_writes)` in L1DataGas, even though none of those N writes appear in the committed state diff.
- The sequencer collects more fee than the actual on-chain DA cost.
- The `TransactionReceipt` stored in the block has `fee ≠ get_fee_by_gas_vector(gas)`, corrupting the receipt commitment that feeds into `calculate_block_commitments` and ultimately the block hash.

This matches: **Critical — Incorrect fee/balance with economic impact.**

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a V3 `AllResources` invoke transaction that:
1. Writes to storage during `__execute__` (generating non-zero L1DataGas in the execution state diff), and
2. Consumes more L2 gas (or L1 gas) than the declared `max_amount` bound.

Both conditions are reachable without any special privilege. The overcharge scales with the number of storage writes in the reverted execution.

---

### Recommendation

In `FeeCheckReport::from_fee_check_error`, for the `AllResources` + `MaxGasAmountExceeded` branch, the `l1_data_gas` component of `gas_for_fee_charge` must be sourced from the **revert receipt** (validation-only state diff), not from the full execution receipt. Concretely, the caller (`PostExecutionReport::new`) should pass the revert-receipt gas vector separately, or `from_fee_check_error` should receive the validation-only DA gas so that:

```rust
l1_data_gas: std::cmp::min(
    all_resource_bounds.l1_data_gas.max_amount,
    revert_gas.l1_data_gas,   // validation-only DA gas, not execution DA gas
),
```

This ensures `fee == get_fee_by_gas_vector(receipt.gas)` holds for the committed receipt, and the fee-token balance update in the state diff matches the actual DA cost of committed state changes.

---

### Proof of Concept

1. Deploy a contract with a function that writes to 100 distinct storage keys.
2. Submit a V3 `AllResources` invoke transaction with:
   - `l2_gas.max_amount` set to exactly the gas consumed by `__validate__` plus ~10% of the execution gas (tight enough to trigger a post-execution L2 overdraft after execution completes).
   - `l1_data_gas.max_amount` set generously (e.g., 10 000).
3. Execution runs to completion, writes 100 storage slots (generating ~6 400 L1DataGas with KZG DA), then fails the post-execution L2 gas check.
4. Observe: `receipt.fee` includes `min(10000, 6400) * l1_data_gas_price` of DA cost, but `receipt.gas.l1_data_gas` is near zero (only validation-phase DA gas). The fee-token balance of the sender is reduced by the full inflated fee, while the committed state diff contains no trace of the 100 storage writes. [6](#0-5) [7](#0-6) [8](#0-7)
