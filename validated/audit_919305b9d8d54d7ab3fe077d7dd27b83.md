### Title
Reentrancy via ERC777 `tokensToSend` Hook Causes `currentOriginNonce` Desync, Permanently Locking Bridged Tokens - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` increments the storage variable `currentOriginNonce` at the top of the function but reads it again from storage when passing it to `initTransferExtension` and the `InitTransfer` event at the bottom. An ERC777 token's `tokensToSend` hook fires on the sender mid-execution, allowing a reentrant call to increment `currentOriginNonce` a second time. When the outer call resumes, it publishes the wrong (inner call's) nonce to Wormhole/NEAR, permanently orphaning the outer call's nonce and locking the deposited tokens in the bridge forever.

---

### Finding Description

In `OmniBridge.initTransfer`:

```
currentOriginNonce += 1;          // line 381 — increments storage to N
...
IERC20(tokenAddress).safeTransferFrom(   // line 407 — ERC777 fires tokensToSend on sender
    msg.sender, address(this), amount
);
...
initTransferExtension(
    msg.sender, tokenAddress,
    currentOriginNonce,            // line 418 — reads storage again (may now be N+1)
    ...
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress,
    currentOriginNonce,            // line 427 — reads storage again (may now be N+1)
    ...
);
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`currentOriginNonce` is never cached into a local variable. The value passed to `initTransferExtension` and the event is whatever the storage slot holds at the time those lines execute — not the value it held when the function entered.

In `OmniBridgeWormhole.initTransferExtension`, the nonce parameter is Borsh-encoded directly into the Wormhole payload that NEAR uses to identify and settle the transfer: [5](#0-4) 

There is no `ReentrancyGuard` on `initTransfer` or `initTransfer1155`. [6](#0-5) 

---

### Impact Explanation

**Permanent freezing of user funds (Critical).**

Execution trace with ERC777 token (nonce starts at 0):

| Step | Who | `currentOriginNonce` (storage) | Action |
|---|---|---|---|
| 1 | Outer call enters | 0 → **1** | `currentOriginNonce += 1` |
| 2 | `safeTransferFrom` fires `tokensToSend` on attacker | 1 | Attacker reenters `initTransfer` |
| 3 | Inner call enters | 1 → **2** | `currentOriginNonce += 1` |
| 4 | Inner call's `safeTransferFrom` completes | 2 | No further reentry |
| 5 | Inner call: `initTransferExtension(... nonce=2 ...)` | 2 | Wormhole message published: **nonce 2** |
| 6 | Inner call: `emit InitTransfer(... nonce=2 ...)` | 2 | Event logged: **nonce 2** |
| 7 | Outer call resumes | 2 | `safeTransferFrom` completed |
| 8 | Outer call: `initTransferExtension(... nonce=currentOriginNonce=2 ...)` | 2 | Wormhole message published: **nonce 2** (duplicate) |
| 9 | Outer call: `emit InitTransfer(... nonce=2 ...)` | 2 | Event logged: **nonce 2** (duplicate) |

**Nonce 1 is never published to Wormhole or NEAR.** The outer call's tokens (transferred to the bridge via `safeTransferFrom` at step 1) are permanently locked: NEAR never receives a settlement message for nonce 1, so it never releases the corresponding assets on the NEAR side. The tokens sit in the bridge contract with no recovery path.

Nonce 2 is published twice; NEAR deduplicates it and processes only one settlement, so the inner call's transfer completes normally. The attacker loses nothing from the inner call but causes the outer call's funds to be irrecoverably frozen.

---

### Likelihood Explanation

**Medium.** ERC777 tokens implement `IERC20` and are accepted by `safeTransferFrom`. Any ERC777 token that is not a `isBridgeToken` (i.e., a native/locked token rather than a mint/burn token) follows the `safeTransferFrom` path. The `tokensToSend` hook is registered by the token holder in the ERC1820 registry and fires before the balance update, giving the attacker a clean reentry window. No privileged role is required — any token holder of a compatible ERC777 token can trigger this. The same structural issue exists in `initTransfer1155` if the ERC1155 token has a malicious `_afterTokenTransfer` hook, though that path is less common. [7](#0-6) 

---

### Recommendation

Cache `currentOriginNonce` into a local variable immediately after incrementing it, and use only the local variable for all subsequent reads within the function:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce;  // cache locally
    ...
    initTransferExtension(msg.sender, tokenAddress, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
}
```

Apply the same fix to `initTransfer1155`. Additionally, add OpenZeppelin's `ReentrancyGuardUpgradeable` and the `nonReentrant` modifier to both `initTransfer` and `initTransfer1155` as defense-in-depth.

---

### Proof of Concept

1. Attacker deploys a malicious ERC777 token `MalToken` and registers a `tokensToSend` hook via ERC1820 that calls `OmniBridge.initTransfer(MalToken, amount, ...)` when triggered.
2. Attacker calls `OmniBridge.initTransfer(MalToken, amount, ...)` (outer call). `currentOriginNonce` becomes 1.
3. `safeTransferFrom(attacker, bridge, amount)` fires the `tokensToSend` hook on the attacker.
4. Hook reenters `initTransfer`. `currentOriginNonce` becomes 2. Inner call's `safeTransferFrom` completes. `initTransferExtension` publishes Wormhole message with `originNonce=2`. Event emitted with `originNonce=2`. Inner call returns.
5. Outer call resumes. Its `safeTransferFrom` completes. `initTransferExtension` reads `currentOriginNonce=2` from storage and publishes a second Wormhole message with `originNonce=2`. Event emitted with `originNonce=2`.
6. NEAR receives two Wormhole messages both claiming `originNonce=2`. It deduplicates and processes only one. `originNonce=1` is never received.
7. The `amount` of `MalToken` deposited by the outer call is permanently locked in the bridge with no recovery mechanism. [8](#0-7) [5](#0-4)
