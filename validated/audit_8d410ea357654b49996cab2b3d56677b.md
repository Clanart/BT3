### Title
Stale MPC-Signed Metadata Signature Enables Token Deployment with Incorrect Decimals - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`, `solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs`)

### Summary
The `deployToken` / `deploy_token` / `fin_transfer` (deploy path) functions across EVM, StarkNet, and Solana accept MPC-signed `MetadataPayload` messages that contain no nonce and no expiry timestamp. Because old MPC signatures are permanently valid and publicly visible on the NEAR blockchain, an attacker can use a stale signature carrying outdated token metadata — most critically, wrong `decimals` — to deploy a bridge token with incorrect decimal precision before the correct signature is used.

### Finding Description

The `log_metadata` → `log_metadata_callback` → `sign_log_metadata_callback` flow on NEAR fetches the current token metadata and asks the MPC signer to sign a `MetadataPayload`:

```rust
// near/omni-bridge/src/lib.rs:341-350
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
```

The resulting signature is emitted as a public NEAR event:

```rust
// near/omni-bridge/src/lib.rs:376-383
env::log_str(
    &OmniBridgeEvent::LogMetadataEvent {
        signature,
        metadata_payload,
    }
    .to_log_string(),
);
```

The `MetadataPayload` type contains only `prefix`, `token`, `name`, `symbol`, and `decimals` — **no nonce, no deadline**:

```rust
// near/omni-types/src/lib.rs:696-702
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
```

On EVM, `deployToken` verifies this signature and the only replay guard is whether the token has already been deployed:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol:142-153
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
```

StarkNet has the identical structure:

```cairo
// starknet/src/omni_bridge.cairo:202-209
fn deploy_token(ref self: ContractState, signature: Signature, payload: MetadataPayload) {
    assert(!_is_paused(@self, PAUSE_DEPLOY_TOKEN), 'ERR_DEPLOY_TOKEN_PAUSED');
    _verify_borsh_signature(ref self, @payload.to_borsh(), signature);
    let token_id_hash = compute_keccak_byte_array(@payload.token);
    let existing_token = self.near_to_starknet_token.read(token_id_hash);
    assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED');
```

On Solana, the PDA `init` constraint enforces uniqueness, but the same stale-signature attack applies before first deployment:

```rust
// solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs:46-52
#[account(
    init,
    payer = common.payer,
    seeds = [WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes().as_ref()],
    bump,
    mint::decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, data.payload.decimals),
    mint::authority = authority,
)]
```

### Impact Explanation

If a NEAR token's `decimals` field changes (e.g., from 6 to 18) after an MPC signature was already produced and emitted, an attacker can submit the old signature to any destination chain before the correct one is used. The bridge token is then deployed with the wrong decimal precision. Because NEAR's bridge normalizes transfer amounts using the stored `origin_decimals` vs `decimals` ratio:

```rust
// near/omni-bridge/src/lib.rs:2784-2786
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

A 12-decimal mismatch (e.g., 6 vs 18) causes a 10^12 factor error in every subsequent transfer, permanently breaking bridge collateralization for that token. The deployed token cannot be replaced because the "already deployed" guard prevents redeployment.

**Impact class:** High — Balance/decimal/accounting corruption that breaks bridge collateralization.

### Likelihood Explanation

- All MPC-signed metadata events are permanently and publicly visible on the NEAR blockchain. Any observer can extract old signatures.
- NEAR NEP-141 tokens can update their metadata (name, symbol, decimals) at any time via their own contract logic.
- The window of vulnerability is any time between the first `log_metadata` call and the actual `deployToken` call on the destination chain — which can be hours or days in practice.
- The attacker needs only to call `deployToken` with the old signature before the legitimate relayer uses the new one. No privileged access is required.

### Recommendation

Include a `deadline` (block timestamp expiry) in the `MetadataPayload` that is signed by the MPC, and enforce it on each destination chain:

**NEAR (near/omni-types/src/lib.rs):**
```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
    pub deadline: u64, // Unix timestamp
}
```

**EVM (evm/src/omni-bridge/contracts/OmniBridge.sol):**
```solidity
struct MetadataPayload {
    string token;
    string name;
    string symbol;
    uint8 decimals;
    uint64 deadline;
}

// In deployToken():
if (block.timestamp > metadata.deadline) revert SignatureExpired();
```

Apply the same pattern to StarkNet (`starknet/src/bridge_types.cairo`) and Solana (`solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`).

### Proof of Concept

1. NEAR token `usdc.near` is created with `decimals = 6`.
2. A relayer calls `log_metadata("usdc.near")` on NEAR. The MPC signs `MetadataPayload{token:"usdc.near", decimals:6}`. The signature `SIG_OLD` is emitted as a public NEAR event.
3. The token deployer updates `usdc.near` metadata to `decimals = 18`.
4. A relayer calls `log_metadata("usdc.near")` again. The MPC signs `MetadataPayload{token:"usdc.near", decimals:18}`. Signature `SIG_NEW` is emitted.
5. An attacker reads `SIG_OLD` from NEAR blockchain history and calls `deployToken(SIG_OLD, {token:"usdc.near", decimals:6})` on the EVM bridge before the legitimate relayer submits `SIG_NEW`.
6. The EVM bridge deploys a token with `decimals = 6`. The `ERR_TOKEN_EXIST` guard now prevents any correction.
7. All subsequent `finTransfer` calls for `usdc.near` → EVM will normalize amounts using `decimals=6` instead of `decimals=18`, causing a 10^12 factor error in every user's received balance. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8)

pub struct MetadataPayload {
    pub token: ByteArray,
    pub name: ByteArray,
    pub symbol: ByteArray,
    pub decimals: u8,
}

#[generate_trait]
pub impl MetadataPayloadImpl of MetadataPayloadTrait {
    fn to_borsh(self: @MetadataPayload) -> ByteArray {
        let mut borsh_bytes: ByteArray = Default::default();
        borsh_bytes.append_byte(PayloadType::Metadata.into());
        borsh_bytes.append(@borsh::encode_byte_array(self.token));
        borsh_bytes.append(@borsh::encode_byte_array(self.name));
        borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
        borsh_bytes.append_byte(*self.decimals);
        borsh_bytes
    }
