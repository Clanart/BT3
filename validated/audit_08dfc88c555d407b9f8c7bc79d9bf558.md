### Title
Missing Duplicate Key/Hash Validation in `MerkleBlob::batch_insert` Bulk Path Corrupts DataLayer Merkle Tree Root - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary
`MerkleBlob::batch_insert` enforces duplicate-key and duplicate-hash checks only for the first two items (via the single-item `insert` path), but silently bypasses those checks for all remaining items in the bulk path. Supplying duplicate `KeyId` or `Hash` values in a batch corrupts the Merkle tree blob, produces ghost leaf nodes that are invisible to the `BlockStatusCache`, and causes the computed root hash to diverge from the hash that any honest proof-of-inclusion would verify against.

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` routes the first two items through `self.insert()` (which carries those guards), but then enters a fast bulk path for all remaining items:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

There is **no** call to `block_status_cache.contains_key(key)` or `block_status_cache.contains_leaf_hash(hash)` before `insert_entry_to_blob`. Consequently:

1. A duplicate `KeyId` within the batch, or a `KeyId` already present in the tree, is written as a second leaf node into the blob without error.
2. `insert_entry_to_blob` updates the cache's `key_to_index` map, so the second write silently overwrites the first entry's index in the cache — the first leaf node becomes a "ghost": it exists in the blob and participates in the tree structure (and therefore in hash computation) but is unreachable through the cache.
3. The root hash is computed over a tree that includes the ghost leaf, so it differs from the root hash that would result from a correctly deduplicated tree.
4. `get_proof_of_inclusion` uses the cache to locate leaves, so it generates proofs anchored to the corrupted root — proofs that are internally self-consistent but reflect a state that was never legitimately committed.

The Python binding `py_batch_insert` passes caller-supplied lists directly into `batch_insert` with no pre-deduplication:

```rust
self.batch_insert(zip(keys_values, hashes).collect())?;
``` [3](#0-2) 

The contrast with `insert` is explicit in the existing test suite, which confirms `insert` rejects duplicates but has no analogous test for `batch_insert` with duplicate keys: [4](#0-3) 

### Impact Explanation

Corrupting the Merkle blob root hash means:

- **Forged inclusion proofs**: a ghost leaf with key K and hash H is part of the tree structure; an attacker who knows the ghost leaf's position can construct a proof-of-inclusion for (K, H) that `ProofOfInclusion::valid()` accepts, even though the DataLayer store's canonical view (via the cache) does not contain that leaf.
- **Root hash mismatch**: the root hash stored on-chain diverges from the root hash that would be computed from the cache-visible leaves, breaking the integrity guarantee that the on-chain commitment reflects the actual store contents.
- **Exclusion proof invalidity**: exclusion proofs generated against the corrupted root are invalid against any honest reconstruction of the tree.

This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.* [5](#0-4) 

### Likelihood Explanation

`batch_insert` is the primary high-throughput insertion API and is called by the Python DataLayer sync code via `py_batch_insert`. Any caller that does not pre-deduplicate its input — including legitimate DataLayer store update code that processes externally sourced key-value pairs — can trigger the corruption. The Python binding accepts arbitrary caller-supplied lists with no validation beyond length matching. [3](#0-2) 

### Recommendation

Add the same duplicate guards to the bulk path of `batch_insert` that `insert` already enforces. Before writing each leaf in the loop at lines 587–603, check:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, pre-validate the entire input vector for internal duplicates before entering the bulk path, and then check each entry against the existing cache. A corresponding test for `batch_insert` with duplicate keys (analogous to `test_double_insert_fails`) should be added.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate to ensure leaf_count > 1 so the bulk path is taken
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
}

let dup_key = KeyId(99);
let hash_a = sha256_num(&99i64);
let hash_b = sha256_num(&100i64);

// Both entries share the same KeyId — batch_insert accepts this silently
blob.batch_insert(vec![
    ((dup_key, ValueId(1)), hash_a),
    ((dup_key, ValueId(2)), hash_b),
]).unwrap(); // <-- should return Err(KeyAlreadyPresent) but does not

// The blob now contains two leaf nodes with KeyId(99).
// The cache tracks only one; the other is a ghost that inflates the root hash.
blob.calculate_lazy_hashes().unwrap();
// check_integrity() will detect the cache/blob mismatch:
blob.check_integrity().expect_err("ghost leaf causes integrity failure");
