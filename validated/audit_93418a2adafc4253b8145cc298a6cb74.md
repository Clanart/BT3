### Title
State sync `set_state_header` does not verify that the embedded chunk belongs to the requested shard — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a caller-supplied `shard_id` without ever asserting that the chunk embedded in the header actually belongs to that shard. A malicious peer can supply shard X's chunk (with a valid Merkle proof for shard X's position in the block) while claiming the header is for shard Y. Every existing check passes because the chunk hash genuinely appears in the block's Merkle tree. The header is stored under shard Y's key, and `set_state_finalize` subsequently applies shard X's state root and trie data to shard Y's storage, permanently corrupting the syncing node's state for shard Y.

### Finding Description

In `set_state_header` the function performs four checks before persisting the header:

1. `validate_chunk_proofs(&chunk, ...)` — validates the chunk's internal Merkle proofs (tx root, receipt root, etc.). Does not compare `chunk.shard_id()` to the `shard_id` parameter.
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk hash appears in the block's chunk-headers Merkle tree **at the position encoded in the attacker-supplied proof**. The position in the proof corresponds to shard X's index, not shard Y's. `verify_path` does not check that this position equals the index of the requested `shard_id`.
3. Receipt-proof and root-proof verification — checks consistency with block headers, but uses the chunk's own shard identity internally; does not cross-check against the `shard_id` parameter.
4. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the state root node against the chunk's `prev_state_root`. Correct for shard X, irrelevant to shard Y.

After all checks pass, the header is stored under `StateHeaderKey(shard_id=Y, sync_hash)`. [1](#0-0) 

When `Chain::set_state_finalize` is later called for shard Y, it retrieves this header via `get_state_header(shard_id=Y, sync_hash)`, obtains shard X's chunk, and calls `chain_update.set_state_finalize(shard_id=Y, ...)`. Inside that function, `shard_uid` is derived from the `shard_id` **parameter** (Y), not from `chunk.shard_id()` (X): [2](#0-1) 

Shard X's state root and trie data are then written into shard Y's column family and flat storage. The `validate_state_part` call in `set_state_part` also uses the state root extracted from the stored header (shard X's root), so valid parts for shard X pass validation and are committed under shard Y's key: [3](#0-2) 

The `validate_state_part_impl` implementation ignores `shard_id` entirely — it is accepted as a parameter only for metrics: [4](#0-3) 

### Impact Explanation

After state sync completes, shard Y's trie and flat storage contain shard X's key-value pairs. Every subsequent block the node processes for shard Y produces a state root that diverges from the canonical chain. The node cannot validate chunks, cannot produce valid chunk state witnesses, and is effectively excluded from consensus. All accounts and contracts assigned to shard Y on this node are inaccessible or incorrect. Because the corruption is written to persistent storage, a restart does not recover the node.

### Likelihood Explanation

Any network participant can broadcast a `SnapshotHostInfo` advertisement claiming to hold a snapshot for epoch E. The syncing node's peer-selection logic (`select_host_for_header` / `select_host_for_part`) will route requests to that peer. No validator key, chunk-producer role, or storage-operator privilege is required. The attacker only needs a full node with access to the canonical chain state (to supply valid parts for shard X), which is publicly available. The attack is deterministic and requires no timing luck. [5](#0-4) 

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: verify the chunk actually belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match \
         requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the existing `shard_ids.contains(&shard_id)` guard in `get_state_response_part` and closes the identity gap between the storage key and the chunk's actual shard binding. [6](#0-5) 

### Proof of Concept

1. Attacker runs a full node synced to epoch E and broadcasts `SnapshotHostInfo` for epoch E.
2. Victim node begins state sync for shard Y; peer selection routes the header request to the attacker.
3. Attacker responds with a `ShardStateSyncResponseHeader` whose `chunk` field is shard X's chunk and whose `chunk_proof` is the valid Merkle path for shard X's position (index X) in `sync_prev_block.chunk_headers_root`.
4. `set_state_header` calls `verify_path(chunk_headers_root, chunk_proof_for_X, ChunkHashHeight(shard_X_hash, ...))` — this succeeds because shard X's chunk hash IS in the tree at position X.
5. No check compares `chunk.shard_id()` (X) to `shard_id` (Y). The header is stored under `StateHeaderKey(shard_id=Y, sync_hash)`.
6. Attacker supplies valid state parts for shard X. `set_state_part` extracts shard X's `prev_state_root` from the stored header and validates each part against it — all pass.
7. `set_state_finalize(shard_id=Y, sync_hash)` retrieves the header for shard Y, finds shard X's chunk, computes `shard_uid` from Y, and applies shard X's trie changes and flat-state delta to shard Y's storage columns.
8. Victim node now holds shard X's state under shard Y's identity. All block processing for shard Y produces wrong state roots; the node diverges from the network. [7](#0-6) [8](#0-7)

    fn validate_state_part_impl(
        &self,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
    ) -> StatePartValidationResult {
        let partial_state = part.to_partial_state();
        let Ok(partial_state) = part.to_partial_state() else {
            // Deserialization error means we've got the data from malicious peer
            tracing::error!(target: "state-parts", ?partial_state, "state part deserialization error");
            return StatePartValidationResult::Invalid;
        };
        match Trie::validate_state_part(state_root, part_id, partial_state) {
            Ok(_) => StatePartValidationResult::Valid,
            // Storage error should not happen
            Err(err) => {
                tracing::error!(target: "state-parts", ?err, "state part storage error");
                StatePartValidationResult::Invalid
            }
        }
    }
