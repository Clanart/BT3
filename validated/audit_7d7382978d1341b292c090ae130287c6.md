### Title
Unchecked Felt Arithmetic in `compute_max_possible_fee` Enables Fee Bypass via Modular Overflow — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function performs felt-field multiplication of `ResourceBounds` fields (`max_amount`, `max_price_per_unit`) without enforcing that these values are within their protocol-specified ranges (u64 and u128 respectively). Because Cairo 0 arithmetic is modular over the STARK prime P, a transaction sender can craft resource-bounds values whose product wraps to zero modulo P, causing `charge_fee` to skip fee collection entirely. This is the direct analog of the original report's "minimum acceptable amount hardcoded to zero" pattern: here, the minimum fee bound is effectively zero because no lower-bound or range constraint is enforced on the inputs to the fee computation.

---

### Finding Description

In `transaction_impls.cairo`, `compute_max_possible_fee` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

All arithmetic is felt arithmetic (mod P ≈ 2²⁵¹). The `ResourceBounds` fields are loaded from the hint `%{ LoadCommonTxFields %}` and used directly with no range assertions: [2](#0-1) 

The result is then used in `charge_fee`:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);

if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

If `max_fee == 0`, the function returns immediately — no ERC-20 transfer is executed, no fee is charged. The only downstream check is:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [4](#0-3) 

When `max_fee = 0`, this forces `actual_fee = 0`, confirming the complete fee bypass.

**Concrete overflow example:**

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | P − 1 (a valid felt, outside u64 range) |
| `l1_gas_bounds.max_price_per_unit` | 1 |
| `l2_gas_bounds.max_amount` | 1 |
| `l2_gas_bounds.max_price_per_unit` | 0 |
| `tx_info.tip` | 1 |
| `l1_data_gas_bounds.max_amount` | 0 |

Computation: `(P−1)·1 + 1·(0+1) + 0 = (P−1) + 1 = P ≡ 0 (mod P)`.

Result: `max_fee = 0` → fee charging skipped.

The transaction hash commits to these felt values, so the user signs this specific crafted transaction. The OS Cairo code has no assertion preventing `max_amount` from exceeding 2⁶⁴−1 or `max_price_per_unit` from exceeding 2¹²⁸−1.

---

### Impact Explanation

**Direct loss of funds.** The sequencer's off-chain fee validation typically uses bounded integer arithmetic (u64 × u128), computing a large, legitimate-looking `max_fee`. It includes the transaction expecting to collect fees. When the prover runs the OS Cairo program, it computes `max_fee = 0` (felt overflow) and must set `low_actual_fee = 0` to satisfy `assert_nn_le(0, 0)`. The resulting proof is valid, but the sequencer collects zero fee tokens for a transaction it executed. The attacker's transaction executes for free, and the sequencer suffers a direct loss of the fee revenue that should have been transferred.

---

### Likelihood Explanation

**Medium.** The crafted felt values are outside the u64/u128 ranges specified by the StarkNet protocol. A sequencer with strict off-chain type validation would reject such transactions at the gateway. However, if the sequencer's fee pre-check uses standard integer arithmetic without explicit felt-overflow detection, the discrepancy between off-chain computation and on-chain felt arithmetic creates the exploitable window. The attack requires no privileged access — only the ability to submit a signed transaction with crafted `ResourceBounds` fields.

---

### Recommendation

Add explicit range assertions on `ResourceBounds` fields before performing fee arithmetic. Specifically, in `compute_max_possible_fee` (or in `get_account_tx_common_fields` where the fields are first loaded), assert that each `max_amount` fits within u64 and each `max_price_per_unit` fits within u128:

```cairo
assert_nn_le(l1_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_U128);
// ... similarly for l2 and l1_data bounds
```

This mirrors the pattern already used for `signature_len`: [5](#0-4) 

With bounded inputs, the product `max_amount * max_price_per_unit ≤ 2⁶⁴ · 2¹²⁸ = 2¹⁹²`, which is well below P ≈ 2²⁵¹, eliminating the overflow.

---

### Proof of Concept

1. Construct a V3 transaction with:
   - `l1_gas_bounds = ResourceBounds(max_amount=P−1, max_price_per_unit=1)`
   - `l2_gas_bounds = ResourceBounds(max_amount=1, max_price_per_unit=0)`, `tip=1`
   - `l1_data_gas_bounds = ResourceBounds(max_amount=0, max_price_per_unit=0)`
2. Compute the Poseidon transaction hash over these felt values and sign it with a valid account key.
3. Submit the transaction to the sequencer. The sequencer's off-chain code computes `max_fee ≈ (2⁶⁴−1) · 1 + 1 · 1 + 0` (bounded arithmetic) — a non-zero value — and includes the transaction.
4. The prover runs `compute_max_possible_fee` in Cairo felt arithmetic: `(P−1)·1 + 1·(0+1) + 0·0 = P ≡ 0 (mod P)`.
5. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without executing the ERC-20 transfer.
6. The proof is valid. The transaction executed. Zero fee was charged. [6](#0-5) [7](#0-6)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
