### Title
Asymmetric Pause Bypass Allows `UnrestrictedDeposit` Users to Lock Tokens Without Ability to Complete Transfer - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `ft_on_transfer` function (which initiates outbound NEAR → foreign transfers) has a pause bypass for `Role::UnrestrictedDeposit`, but `sign_transfer` (which is required to complete the outbound transfer by obtaining the MPC signature) only bypasses the pause for `Role::DAO`. This asymmetry means a user with `UnrestrictedDeposit` can lock their NEAR tokens and create a pending transfer during a pause, but the relayer cannot call `sign_transfer` to release those tokens on the destination chain. The user's funds are irrecoverably stuck until DAO intervention.

---

### Finding Description

In `near/omni-bridge/src/lib.rs`, the two functions that form the outbound transfer pipeline carry different pause-bypass roles:

**Entry point — `ft_on_transfer` (initiates transfer, locks tokens):**

```rust
#[pause(except(roles(Role::DAO, Role::UnrestrictedDeposit)))]
pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
``` [1](#0-0) 

**Completion point — `sign_transfer` (produces MPC signature to release tokens on foreign chain):**

```rust
#[payable]
#[trusted_relayer]
#[pause(except(roles(Role::DAO)))]
pub fn sign_transfer(
    &mut self,
    transfer_id: TransferId,
    ...
``` [2](#0-1) 

The outbound transfer lifecycle is:
1. User calls `ft_on_transfer` → `init_transfer` → tokens locked on NEAR, `TransferMessage` stored in `pending_transfers`.
2. Relayer calls `sign_transfer` → MPC signature obtained → `SignTransferEvent` emitted.
3. Relayer submits signature to foreign chain → `finTransfer` on foreign chain releases tokens to recipient.

When the bridge is paused, step 1 is still accessible to `UnrestrictedDeposit` users, but step 2 is blocked for all callers except `Role::DAO`. The relayer — the only actor that calls `sign_transfer` in normal operation — cannot proceed. The user's tokens are locked in the bridge with no path to release them on the destination chain.

The `init_transfer_internal` path stores the transfer message and emits no refund: [3](#0-2) 

There is no cancel or refund mechanism visible in the contract that would allow the user to recover their locked tokens without DAO action.

---

### Impact Explanation

A user with `Role::UnrestrictedDeposit` who initiates an outbound transfer while the bridge is paused (or whose transfer is in-flight when the bridge is paused) will have their tokens permanently locked on NEAR until the DAO either:
- Unpauses the bridge, or
- Manually calls `sign_transfer` on behalf of the user.

This constitutes **irrecoverable lock of user funds in bridge flows** — matching the Critical/High allowed impact scope. The user has no self-service recovery path; they are entirely dependent on DAO intervention.

---

### Likelihood Explanation

The `UnrestrictedDeposit` role exists precisely to allow certain users to bridge even during a pause. This implies the protocol intends these users to be able to complete transfers during a pause. However, the code does not honor that intent for the completion step. Any time the bridge is paused and a user with `UnrestrictedDeposit` initiates a transfer — or any time the bridge is paused after a user has already initiated a transfer — the funds become stuck. This is a realistic operational scenario (e.g., emergency pause triggered mid-transfer).

---

### Recommendation

Either:
1. **Remove the `UnrestrictedDeposit` bypass from `ft_on_transfer`** so that no user can initiate a transfer that cannot be completed, or
2. **Add `Role::UnrestrictedDeposit` (or `Role::UnrestrictedRelayer`) to the pause bypass of `sign_transfer`** so that the relayer can complete transfers initiated by `UnrestrictedDeposit` users even during a pause:

```rust
#[pause(except(roles(Role::DAO, Role::UnrestrictedRelayer)))]
pub fn sign_transfer(...) -> Promise {
```

Option 1 is safer because it prevents the one-way-door scenario entirely.

---

### Proof of Concept

1. Bridge is paused (or becomes paused after step 2).
2. User with `Role::UnrestrictedDeposit` calls `ft_transfer_call` on a NEAR token, routing to `ft_on_transfer` with an `InitTransfer` message. The `#[pause(except(roles(Role::DAO, Role::UnrestrictedDeposit)))]` guard passes. Tokens are locked; `TransferMessage` is stored in `pending_transfers`.
3. Relayer attempts to call `sign_transfer(transfer_id, ...)` to obtain the MPC signature needed to release tokens on the destination chain.
4. The `#[pause(except(roles(Role::DAO)))]` guard on `sign_transfer` rejects the relayer call — only `Role::DAO` can proceed.
5. The user's tokens remain locked in `pending_transfers` with no self-service recovery. The destination chain never receives a `SignTransferEvent` and never releases tokens to the recipient. [4](#0-3)

    #[payable]
    #[trusted_relayer]
    #[pause(except(roles(Role::DAO)))]
    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };

        let payload = near_sdk::env::keccak256_array(
            transfer_payload
                .encode_hashable()
                .near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_TRANSFER_CALLBACK_GAS)
                    .sign_transfer_callback(transfer_payload, &transfer_message.fee),
            )
    }
