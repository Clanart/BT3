### Title
`get_state_response_part` Uses New-Epoch Shard Layout to Index Old-Epoch Block Chunks, Producing Wrong State Root After Resharding - (File: chain/chain/src/state_sync/adapter.rs)

### Summary

`get_state_response_part` resolves the `shard_index` for a carried-over shard using the **new epoch's** shard layout, then uses that index to read `prev_state_root` from the **previous epoch's** block chunks. After a resharding event, shard IDs and shard indices are decoupled (V2/V3 layouts maintain an explicit `id_to_index_map`), so the same shard ID maps to a different index in the old and new layouts. The function therefore reads the wrong chunk's `prev_state_root`, generates a state part from that wrong root, and serves it to syncing nodes. The companion function `compute_state_response_header` correctly uses `prev_shard_layout` for the same lookup, so the header commits the correct root while the part is built from a different root — breaking the reconstruction invariant and causing every served part to fail the requester's `validate_state_part` check.

### Finding Description

In `get_state_response_part`:

```rust
// line 295 — epoch_id is the SYNC block's epoch (new epoch)
let epoch_id = block.header().epoch_id();
// line 305 — shard_layout is the NEW epoch's layout
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
// line 310 — prev_block is the LAST block of the OLD epoch
let prev_block = self.chain_store.get_block(header.prev_hash())?;
// line 311 — shard_index from NEW layout
let shard_index = shard_layout.get_shard_index(shard_id)?;
// line 312-316 — indexes OLD block's chunks with NEW index → wrong state_root
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

The sibling function `compute_state_response_header` performs the same lookup correctly:

```rust

let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let chunk_header = chunks.get(prev_shard_index)...;
``` [2](#0-1) 

In V2 and V3 shard layouts, shard ID and shard index are explicitly decoupled via `id_to_index_map`: [3](#0-2) [4](#0-3) 

The mainnet epoch-78 config confirms that after shard 0 split into shards 10 and 11, shard 1's index shifted from 1 to 2, shard 4's index shifted from 2 to 7, etc.: [5](#0-4) 

So for shard 1 after that resharding, `get_state_response_part` reads `prev_block.chunks()[2].prev_state_root()` (shard 4's root) instead of `prev_block.chunks()[1].prev_state_root()` (shard 1's correct root). The state part is then generated from shard 4's trie root via `obtain_state_part`: [6](#0-5) 

`obtain_state_part_impl` derives `shard_uid` from `prev_prev_hash` (old epoch), so the trie lookup targets the correct column family but the wrong root hash — either failing with a storage error or, if the hash collides, silently returning wrong trie nodes. [7](#0-6) 

### Impact Explanation

Any node serving state parts for a carried-over shard whose index shifted during resharding will produce parts whose embedded trie root does not match the root committed in the `ShardStateSyncResponseHeader`. The requesting node's `validate_state_part` call will reject every such part: [8](#0-7) 

State sync for those shards loops indefinitely, blocking the syncing node from advancing past the resharding epoch. New validators, nodes recovering from downtime, and nodes reassigned to a previously-untracked carried-over shard are all unable to complete state sync after a resharding event.

### Likelihood Explanation

The condition is: (a) a resharding has occurred (shard layout changed at an epoch boundary), and (b) a node requests state parts for a carried-over shard whose `id_to_index_map` entry differs between the old and new layouts. Both conditions are satisfied on mainnet after every resharding event. Any node that was not tracking a carried-over shard before resharding and needs to begin tracking it afterward will trigger this path. The `StateRequestPart` message is sent by any syncing node — no privileged role is required.

### Recommendation

In `get_state_response_part`, mirror the pattern used in `compute_state_response_header`: derive `prev_shard_layout` from the previous block's epoch and use `prev_shard_layout.get_shard_index(shard_id)` to index into `prev_block.chunks()`:

```rust
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

Add a test that exercises `get_state_response_part` across a resharding boundary for a carried-over shard whose index changed, asserting that the returned part validates against the header's committed root.

### Proof of Concept

1. Run a network through a resharding event (e.g., shard 0 splits into shards 10 and 11 as on mainnet epoch 78).
2. After resharding, bring up a new node that needs to track shard 1 (a carried-over shard whose index changed from 1 to 2).
3. The new node sends `StateRequestPart { shard_id: 1, sync_hash: <first block of new epoch>, part_id: 0 }` to a peer.
4. The peer's `get_state_response_part` computes `shard_index = new_layout.get_shard_index(1) = 2`, reads `prev_block.chunks()[2].prev_state_root()` (shard 4's root), and generates a part from that root.
5. `compute_state_response_header` for shard 1 correctly uses `prev_shard_layout.get_shard_index(1) = 1`, committing shard 1's actual root in the header.
6. The new node calls `validate_state_part(shard_id=1, state_root=<header root>, part_id, part)` — the part was built from a different root, so validation fails.
7. State sync for shard 1 never completes; the node cannot process blocks for the new epoch.
