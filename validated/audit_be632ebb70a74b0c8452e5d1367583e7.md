### Title
EIP-712 Legacy Signature Does Not Commit to `timeout_height`, Enabling Replay of Expired Transactions ‚Äî (`ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

---

### Summary

The legacy EIP-712 `Tx` type schema explicitly omits `timeout_height` from its type definition. Because EIP-712 only hashes fields present in the type schema, the computed `sigHash` is identical regardless of the `timeout_height` value in the transaction. An unprivileged attacker who observes a signed legacy EIP-712 transaction with a non-zero `timeout_height` can strip or zero out that field and replay the transaction after the intended expiry block, bypassing the user's validity-window intent.

---

### Finding Description

**Root cause ‚Äî type schema omission:**

In `extractMsgTypes`, the `"Tx"` type is defined without `timeout_height`: [1](#0-0) 

The comment explicitly acknowledges the omission. Because EIP-712 (`apitypes.TypedDataAndHash`) only encodes fields listed in the type definition, any `timeout_height` value present in the message payload is silently ignored during hash computation.

**`txBytes` does include `timeout_height`:**

In `VerifySignature`, `legacytx.StdSignBytes` is called with `tx.GetTimeoutHeight()`: [2](#0-1) 

The resulting JSON (`txBytes`) contains `"timeout_height": "100"` when non-zero. This JSON is passed as `data` to `LegacyWrapTxToTypedData`, which JSON-unmarshals it into `txData` and sets it as the EIP-712 `Message`: [3](#0-2) 

But since `timeout_height` is absent from the `"Tx"` type, `TypedDataAndHash` ignores it: [4](#0-3) 

**The ante handler is active (not dead code):**

`newLegacyCosmosAnteHandlerEip712` is wired into `evmd/ante/ante.go` and includes `NewLegacyEip712SigVerificationDecorator` at position 13 in the decorator chain: [5](#0-4) 

`authante.NewTxTimeoutHeightDecorator()` runs at position 5 (line 46), before signature verification. It rejects a tx only when `timeout_height > 0 AND current_block_height > timeout_height`. If the attacker sets `timeout_height=0`, this decorator is a no-op.

---

### Impact Explanation

A user signs a legacy EIP-712 tx with `timeout_height=100` to ensure it cannot execute after block 100 (e.g., a time-sensitive delegation or swap). The tx is broadcast but not included before block 100. An attacker:

1. Copies the tx from the mempool (public data).
2. Replaces `timeout_height=100` with `timeout_height=0` in the tx body.
3. Submits the modified tx after block 100.
4. `TxTimeoutHeightDecorator` passes (no timeout on `timeout_height=0`).
5. `VerifySignature` recomputes `sigHash` ‚Äî identical to the original because `timeout_height` is not in the type schema.
6. Signature verification passes; the tx executes.

The user's explicit expiry intent is completely nullified. This is a **High** severity EIP-712 authorization bypass enabling replay of transactions the signer intended to expire.

---

### Likelihood Explanation

- The `LegacyEip712SigVerificationDecorator` path is reachable by any unprivileged user submitting a tx with `ExtensionOptionsWeb3Tx`.
- Mempool transactions are publicly observable on most networks.
- The attacker needs only to modify one field in the tx body ‚Äî no key material, no privileged access.
- The omission is intentional per the comment, meaning no accidental guard exists.

---

### Recommendation

Add `timeout_height` back to the `"Tx"` type definition in `extractMsgTypes`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    {Name: "timeout_height", Type: "string"},  // must be committed to
},
```

Ensure `legacytx.StdSignBytes` always serializes `timeout_height` (even when 0, e.g., as `"0"`) so the field is consistently present in the message payload and hashed. Alternatively, reject legacy EIP-712 txs with non-zero `timeout_height` in `LegacyValidatePayloadMessages` until the type schema is fixed.

---

### Proof of Concept

```go
// Differential test: two txBytes differing only in timeout_height
// produce identical sigHash via LegacyWrapTxToTypedData.

txBytes0 := legacytx.StdSignBytes("chain_9000-1", 1, 0, 0,   fee, msgs, "")
txBytes1 := legacytx.StdSignBytes("chain_9000-1", 1, 0, 100, fee, msgs, "")

td0, _ := eip712.LegacyWrapTxToTypedData(cdc, 9000, msgs[0], txBytes0, feeDelegation)
td1, _ := eip712.LegacyWrapTxToTypedData(cdc, 9000, msgs[0], txBytes1, feeDelegation)

hash0, _, _ := apitypes.TypedDataAndHash(td0)
hash1, _, _ := apitypes.TypedDataAndHash(td1)

assert.Equal(t, hash0, hash1) // passes ‚Äî timeout_height is not committed to
```

A signature produced over `hash0` (timeout_height=0) is accepted by `VerifySignature` for a tx carrying `timeout_height=100`, and vice versa ‚Äî the attacker can zero out `timeout_height` on any signed tx and replay it indefinitely past its intended expiry.
