### Title
Non-Cumulative Balance Check in Batched EVM Tx Ante Handler Allows Over-Commitment of Fees Beyond Actual Balance - (File: ante/eth.go)

### Summary

In `VerifyEthAccount` (`ante/eth.go`) and `CheckEthGasConsume` (`ante/eth.go`), the sender balance check and fee deduction loop over all `MsgEthereumTx` messages in a batched Cosmos tx independently. Each iteration reads the **same unmodified KV-store balance** because fee deductions are recorded only in the transient `ObjectStore` (virtual path via `SendCoinsFromAccountToModuleVirtual`), never in the KV store during ante handling. A sender whose balance covers one message's cost but not the sum of all messages' costs passes every per-message check, allowing the ante handler to accept and execute a batched tx whose total committed fees exceed the sender's actual balance.

### Finding Description

**Step 1 – Non-cumulative balance check in `VerifyEthAccount`**

`VerifyEthAccount` iterates over every `MsgEthereumTx` in the tx and, for each one, reads the sender's KV-store balance fresh and checks it against that single message's `tx.Cost()` (value + gasLimit × gasPrice):

```go
// ante/eth.go:76-106
for _, msg := range tx.GetMsgs() {
    ...
    balance := evmKeeper.GetBalance(ctx, from, evmDenom)   // reads KV store each time
    if err := keeper.CheckSenderBalance(sdkmath.NewIntFromBigIntMut(balance.ToBig()), ethTx); err != nil {
        return errorsmod.Wrap(err, "failed to check sender balance")
    }
}
```

`evmKeeper.GetBalance` delegates to `k.bankKeeper.GetBalance(ctx, addr, denom).Amount.BigInt()` — a pure KV-store read. No running total is maintained. For a batched tx with N messages each costing X, the check passes as long as `balance ≥ X`, even when `N × X > balance`.

**Step 2 – Virtual fee deductions do not update the KV-store balance**

`CheckEthGasConsume` deducts fees for each message via `DeductTxCostsFromUserBalance` → `DeductFees` → `SendCoinsFromAccountToModuleVirtual`:

```go
// x/evm/keeper/utils.go:207-216
func DeductFees(bankKeeper types.BankKeeper, ctx sdk.Context, acc sdk.AccountI, fees sdk.Coins) error {
    if ctx.BlockHeight() > 0 {
        if err := bankKeeper.SendCoinsFromAccountToModuleVirtual(ctx, acc.GetAddress(), authtypes.FeeCollectorName, fees); err != nil {
            return errorsmod.Wrap(errortypes.ErrInsufficientFunds, err.Error())
        }
    }
    return nil
}
```

The `Virtual` suffix denotes the ObjectStore path. Virtual coins are stored in the transient `ObjectStore` keyed by `addr + txIndex`, not in the KV bank store. The KV balance is therefore **unchanged** after each virtual deduction. The next iteration of the `CheckEthGasConsume` loop reads the same original KV balance, so the second (and every subsequent) message's deduction sees the full pre-tx balance.

**Step 3 – Gas refunds are also virtual**

After EVM execution, `RefundGasWithPrice` issues refunds via `SendCoinsFromModuleToAccountVirtual`:

```go
// x/evm/keeper/gas.go:78
err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
```

The fee collector's virtual balance is funded only by the virtual deductions. If total virtual deductions exceed the sender's real balance, the fee collector holds unbacked virtual credits.

**Concrete scenario**

- Sender balance (KV store): 100 aevmos
- Batched tx: 2 × `MsgEthereumTx`, each with `gasLimit × gasPrice = 60 aevmos`, value = 0

| Step | Action | KV balance | Virtual deducted |
|------|--------|-----------|-----------------|
| `VerifyEthAccount` msg1 | `GetBalance` = 100, cost = 60 → pass | 100 | 0 |
| `VerifyEthAccount` msg2 | `GetBalance` = 100, cost = 60 → pass | 100 | 0 |
| `CheckEthGasConsume` msg1 | `SendCoinsFromAccountToModuleVirtual(60)` | 100 (unchanged) | 60 |
| `CheckEthGasConsume` msg2 | `SendCoinsFromAccountToModuleVirtual(60)` — KV still 100 → passes | 100 (unchanged) | 120 |

Total committed: 120 aevmos. Actual balance: 100 aevmos. The ante handler accepts the tx.

### Impact Explanation

This is a **High** impact finding matching: *"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

A sender can submit a batched Cosmos tx containing N `MsgEthereumTx` messages whose combined fee cost exceeds their balance, with each individual message passing the balance check. The ante handler accepts the tx, EVM execution proceeds for all N messages, and the fee collector accumulates virtual credits that are not fully backed by the sender's real balance. Depending on how the virtual-to-KV settlement is handled at end-of-block, this results in either: (a) validators receiving fewer fees than owed (mis-accounting of user funds/fees), or (b) a settlement error that could corrupt committed state.

### Likelihood Explanation

Batched EVM transactions are a supported Cosmos SDK feature in Ethermint (documented in ADR-003 and tested in `tests/integration_tests/test_batch.py`). Any unprivileged user can construct and broadcast a batched tx via the standard `broadcast_tx_json` RPC endpoint. No special privileges, governance actions, or validator cooperation are required. The attacker only needs to know their balance and set each message's fee to exactly their balance, then batch two or more such messages.

### Recommendation

1. **Cumulative balance check in `VerifyEthAccount`**: Maintain a running total of committed costs per sender across all messages in the loop and check `balance ≥ totalCost` rather than `balance ≥ singleCost`.

2. **Cumulative fee deduction guard in `CheckEthGasConsume`**: After each virtual deduction, track the total virtually deducted per sender and reject if it exceeds the KV balance.

3. **Alternatively**: Enforce a single-message-per-tx constraint for `MsgEthereumTx` batches (as ADR-003 already recommends discouraging batched EVM txs), which would eliminate the multi-message attack surface entirely.

```go
func CheckSenderBalance(
	balance sdkmath.Int,
	tx *ethtypes.Transaction,
) error {
	cost := tx.Cost()

	if cost.Sign() < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInvalidCoins,
			"tx cost (%s) is negative and invalid", cost,
		)
	}

	if balance.IsNegative() || balance.BigInt().Cmp(cost) < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInsufficientFunds,
			"sender balance < tx cost (%s < %s)", balance, tx.Cost(),
		)
	}
	return nil
```

```go
func addVirtualCoins(store storetypes.ObjKVStore, txIndex int, addr sdk.AccAddress, amt sdk.Coins) {
	key := make([]byte, len(addr)+8)
	copy(key, addr)
	binary.BigEndian.PutUint64(key[len(addr):], uint64(txIndex)) //nolint:gosec // test only

	var coins sdk.Coins
	value := store.Get(key)
	if value != nil {
		coins = value.(sdk.Coins)
	}
	coins = coins.Add(amt...)
	store.Set(key, coins)
}
```
