### Title
`NextBaseFee()` and `processBlock()` Use `gasUsed` Instead of Consensus-Layer `gasWanted` for Base Fee Prediction, Causing Systematic RPC Underestimation — (`rpc/backend/chain_info.go`, `rpc/backend/utils.go`)

---

### Summary

The JSON-RPC `eth_baseFee` endpoint and `eth_feeHistory` `NextBaseFee` field compute the predicted next-block base fee using raw `gasUsed` from block results. The consensus layer (`CalculateBaseFee`) was deliberately changed (v0.16.0) to use an adjusted `gasWanted` value instead. The RPC layer was never updated to match. This structural divergence causes the RPC to systematically underestimate the actual base fee that consensus will enforce, feeding incorrect consensus-critical data to every wallet and client that relies on `eth_baseFee`.

---

### Finding Description

**Consensus-layer base fee calculation** (`x/feemarket/keeper/eip1559.go`):

```go
parentGasUsed := k.GetBlockGasWanted(ctx)   // line 57 — uses adjusted gasWanted
```

`GetBlockGasWanted` returns the value stored by `EndBlock`:

```go
// x/feemarket/keeper/abci.go lines 72-74
minGasMultiplier := k.GetParams(ctx).MinGasMultiplier
limitedGasWanted := sdkmath.LegacyNewDec(gw).Mul(minGasMultiplier)
gasWanted = sdkmath.LegacyMaxDec(limitedGasWanted, sdkmath.LegacyNewDec(gasUsed)).TruncateInt().Uint64()
```

So the stored value is `max(gasWanted × MinGasMultiplier, gasUsed)`, which is always `≥ gasUsed` and can be substantially larger when blocks are under-utilized.

**RPC-layer base fee prediction** (`rpc/backend/chain_info.go`):

```go
gasUsed, err := computeGasUsed(blockRes)   // line 151 — uses raw gasUsed
...
header := ethtypes.Header{
    GasUsed: gasUsed,   // line 168
}
nextBaseFee, err := CalcBaseFee(cfg, &header, feeParams.Params)
```

`computeGasUsed` sums `TxsResults[i].GasUsed` from the block result — the actual EVM gas consumed, not the adjusted `gasWanted`.

The same pattern appears in `processBlock` (`rpc/backend/utils.go` line 201):

```go
header.GasUsed = uint64(gasUsed)   // gasUsed from ethBlock["gasUsed"]
nextBaseFee, err := CalcBaseFee(cfg, &header, params.Params)
```

`CalcBaseFee` in `rpc/backend/utils.go` is a standalone reimplementation of the EIP-1559 formula that takes `parent.GasUsed` as input — it has no knowledge of the `MinGasMultiplier` adjustment that `EndBlock` applies before storing `blockGasWanted`.

**Concrete divergence example** (default `MinGasMultiplier = 0.5`, `ElasticityMultiplier = 2`):

| Parameter | Value |
|---|---|
| Block gas limit | 10,000,000 |
| Gas target | 5,000,000 |
| Actual `gasUsed` | 500,000 |
| Stored `gasWanted` | `max(0.5 × 10,000,000, 500,000)` = **5,000,000** |

- **Consensus** sees `gasWanted = 5,000,000 = gasTarget` → base fee **unchanged**
- **RPC** sees `gasUsed = 500,000 < gasTarget` → base fee **decreases** by `~baseFee × 4,500,000 / 5,000,000 / 8`

With `baseFee = 1,000,000,000` (1 Gwei), the RPC would predict a next base fee of approximately `887,500,000`, while consensus keeps it at `1,000,000,000`. Any wallet that sets `maxFeePerGas` based on the RPC prediction will have its transaction rejected.

---

### Impact Explanation

`eth_baseFee` (via `NextBaseFee()`) and `eth_feeHistory` (via `processBlock()`) are the primary sources wallets and dApps use to construct EIP-1559 transactions. When these endpoints systematically underestimate the actual base fee enforced by consensus, every transaction built on that estimate with `maxFeePerGas = predictedBaseFee + tip` will be rejected at the mempool or block inclusion stage with `insufficient gas prices`. This is a public JSON-RPC path feeding incorrect consensus-critical data directly into transaction execution, matching the allowed High impact.

---

### Likelihood Explanation

The discrepancy is active whenever `gasUsed < gasWanted × MinGasMultiplier`. With the default `MinGasMultiplier = 0.5`, this occurs whenever actual EVM gas consumption is less than 50% of the declared gas limit — a routine condition on any chain that is not consistently running at full capacity. The divergence grows larger as blocks become emptier, making it worst precisely when the chain is under low load and users most expect fee estimates to be reliable.

---

### Recommendation

`NextBaseFee()` in `rpc/backend/chain_info.go` and `processBlock()` in `rpc/backend/utils.go` must replicate the `EndBlock` adjustment before calling `CalcBaseFee`. Specifically, after computing `gasUsed` from block results, apply:

```go
// Replicate x/feemarket/keeper/abci.go EndBlock adjustment
limitedGasWanted := gasWanted * minGasMultiplier   // gasWanted from block header
adjustedGas := max(limitedGasWanted, gasUsed)
header.GasUsed = adjustedGas
```

Alternatively, query the stored `blockGasWanted` value directly from the feemarket keeper state at the relevant height instead of recomputing from raw block results.

### Proof of Concept

1. Consensus path: `BeginBlock` → `CalculateBaseFee` reads `GetBlockGasWanted` (adjusted value stored by `EndBlock`). [1](#0-0) 

2. `EndBlock` stores `max(gasWanted × MinGasMultiplier, gasUsed)`, not raw `gasUsed`. [2](#0-1) 

3. RPC `NextBaseFee()` calls `computeGasUsed` (raw gasUsed) and passes it directly to `CalcBaseFee`. [3](#0-2) 

4. `processBlock()` (used by `eth_feeHistory`) does the same. [4](#0-3) 

5. `CalcBaseFee` in the RPC layer uses `parent.GasUsed` with no MinGasMultiplier correction. [5](#0-4) 

6. `eth_baseFee` is served directly from `NextBaseFee()`. [6](#0-5)

	// If the parent gasUsed is the same as the target, the baseFee remains unchanged.
	if parent.GasUsed == parentGasTarget {
		return new(big.Int).Set(parent.BaseFee), nil
	}

	var (
		num   = new(big.Int)
		denom = new(big.Int)
	)

	if parent.GasUsed > parentGasTarget {
		// If the parent block used more gas than its target, the baseFee should increase.
		// max(1, parentBaseFee * gasUsedDelta / parentGasTarget / baseFeeChangeDenominator)
		num.SetUint64(parent.GasUsed - parentGasTarget)
		num.Mul(num, parent.BaseFee)
		num.Div(num, denom.SetUint64(parentGasTarget))
		num.Div(num, denom.SetUint64(uint64(p.BaseFeeChangeDenominator)))
		baseFeeDelta := ethermint.BigMax(num, common.Big1)
		return num.Add(parent.BaseFee, baseFeeDelta), nil
	}

	// Otherwise if the parent block used less gas than its target, the baseFee should decrease.
	// max(0, parentBaseFee * gasUsedDelta / parentGasTarget / baseFeeChangeDenominator)
	num.SetUint64(parentGasTarget - parent.GasUsed)
	num.Mul(num, parent.BaseFee)
	num.Div(num, denom.SetUint64(parentGasTarget))
	num.Div(num, denom.SetUint64(uint64(p.BaseFeeChangeDenominator)))
	baseFee := num.Sub(parent.BaseFee, num)
	minGasPrice := p.MinGasPrice.TruncateInt().BigInt()
	return ethermint.BigMax(baseFee, minGasPrice), nil
}
