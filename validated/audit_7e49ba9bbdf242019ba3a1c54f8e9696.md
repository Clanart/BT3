### Title
EIP-712 Chain-ID Verification Bypass via Silent `uint64` Truncation of `*big.Int` Chain ID — (`File: ante/cosmos/eip712.go`)

### Summary
The Legacy EIP-712 ante handler compares the attacker-supplied `TypedDataChainID` (`uint64`) against the chain's actual chain ID by calling `signerChainID.Uint64()` on an arbitrary-precision `*big.Int`. Go's `big.Int.Uint64()` silently returns only the low-order 64 bits when the value exceeds `math.MaxUint64`, with no error or overflow signal. For any Ethermint chain whose numeric chain ID exceeds 2⁶⁴, this truncation makes the chain-ID guard bypassable, allowing a signature produced for a different chain (whose ID equals the low 64 bits of the real chain ID) to be accepted as valid.

### Finding Description

`ParseChainID` in `types/chain_id.go` returns a `*big.Int` with no upper-bound constraint beyond the 48-character string limit. The regex `[1-9][0-9]*` permits numeric values up to ~10³⁸, far exceeding `uint64` max (~1.8×10¹⁹). [1](#0-0) [2](#0-1) 

In `VerifySignature` (the EIP-712 ante handler), the parsed chain ID is compared to the attacker-controlled `extOpt.TypedDataChainID` field:

```go
signerChainID, err := ethermint.ParseChainID(signerData.ChainID)
// ...
if extOpt.TypedDataChainID != signerChainID.Uint64() {   // ← truncation here
    return errorsmod.Wrap(errortypes.ErrInvalidChainID, "invalid chain-id")
}
``` [3](#0-2) 

`signerChainID.Uint64()` silently returns `signerChainID mod 2⁶⁴`. If the chain's actual ID is `N = 2⁶⁴ + K`, the guard reduces to `extOpt.TypedDataChainID != K`. An attacker sets `TypedDataChainID = K` and the check passes. The typed data is then constructed with chain ID `K`:

```go
typedData, err := eip712.LegacyWrapTxToTypedData(
    ethermintCodec, extOpt.TypedDataChainID,   // ← K, not N
    msgs[0], txBytes, feeDelegation)
``` [4](#0-3) 

A signature produced for a chain with ID `K` is therefore accepted on a chain with ID `2⁶⁴ + K`. The same truncation occurs in the non-legacy EIP-712 encoding path: [5](#0-4) [6](#0-5) 

`ExtensionOptionsWeb3Tx.TypedDataChainID` is `uint64` in the proto definition, so the attacker can supply any 64-bit value without further validation: [7](#0-6) [8](#0-7) 

### Impact Explanation

An attacker who holds a valid EIP-712 Cosmos transaction signature for chain B (ID = `K`) can replay it verbatim on chain A (ID = `2⁶⁴ + K`). The replayed transaction carries the signer's full authority: it can transfer funds, execute arbitrary Cosmos messages (e.g., `MsgSend`, governance votes, staking operations), or mutate any state the signer is authorized to change. Because the ante handler accepts the signature, the transaction commits normally. This matches the allowed High impact: *EIP-712 chain-id verification bypass enabling replay and forged execution*.

### Likelihood Explanation

The precondition is a chain whose numeric EIP-155 ID exceeds `math.MaxUint64` (≈1.8×10¹⁹). Ethermint's chain-ID format (`[a-z]+_[1-9][0-9]*-[1-9][0-9]*`, max 48 chars) syntactically permits IDs up to ~10³⁸, so the condition is reachable within the protocol. No currently deployed Ethermint chain is known to use such a large ID, making exploitation unlikely on existing networks. However, the code contains no guard preventing deployment with such an ID, and the bug is latent for any future chain that does. Likelihood is **Low** but non-zero.

### Recommendation

Replace the silent-truncation comparison with an explicit overflow check before calling `Uint64()`:

```go
if !signerChainID.IsUint64() || extOpt.TypedDataChainID != signerChainID.Uint64() {
    return errorsmod.Wrap(errortypes.ErrInvalidChainID, "invalid chain-id")
}
```

Apply the same guard in `ethereum/eip712/encoding.go` wherever `chainID.Uint64()` is called after `ParseChainID`. Alternatively, store and compare chain IDs as `*big.Int` throughout the EIP-712 pipeline, eliminating the narrowing conversion entirely.

### Proof of Concept

1. Deploy an Ethermint chain with Cosmos chain ID `"mychain_18446744073709560616-1"` (numeric part = 2⁶⁴ + 10000 = 18446744073709560616 > `math.MaxUint64`).
2. On a second chain with ID `"otherchain_10000-1"` (numeric part = 10000), obtain a valid signed EIP-712 Cosmos transaction `T` from victim account `V`.
3. Construct a `MsgEthereumTx`-wrapping Cosmos tx targeting chain A, copying `T`'s `FeePayerSig` and setting `ExtensionOptionsWeb3Tx.TypedDataChainID = 10000`.
4. Submit to chain A. In `VerifySignature`: `ParseChainID("mychain_18446744073709560616-1")` returns `big.Int(18446744073709560616)`; `.Uint64()` returns `10000`; the guard `10000 != 10000` is **false**, so no error is returned.
5. The typed data is built with chain ID `10000` (matching what `V` signed on chain B), `TypedDataAndHash` produces the same hash, `VerifySignature` succeeds, and the transaction executes under `V`'s authority on chain A. [3](#0-2) [2](#0-1)

func ParseChainID(chainID string) (*big.Int, error) {
	chainID = strings.TrimSpace(chainID)
	if len(chainID) > 48 {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "chain-id '%s' cannot exceed 48 chars", chainID)
	}

	matches := ethermintChainID.FindStringSubmatch(chainID)
	if matches == nil || len(matches) != 4 || matches[1] == "" {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "%s: %v", chainID, matches)
	}

	// verify that the chain-id entered is a base 10 integer
	chainIDInt, ok := new(big.Int).SetString(matches[2], 10)
	if !ok {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "epoch %s must be base-10 integer format", matches[2])
	}

	return chainIDInt, nil
