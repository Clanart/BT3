### Title
Child `ChunkExtra.proposed_split` Inherits Parent's Non-None Value After Shard Split, Breaking Chunk Header Validation — (`File: chain/chain/src/resharding/manager.rs`)

---

### Summary

During a shard split in `process_memtrie_resharding_storage_update`, the child `ChunkExtra` is created by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field is never reset to `None`. Because the parent's `ChunkExtra` at the resharding block carries `proposed_split = Some(TrieSplit)` (the split that triggered resharding), every child shard inherits this non-`None` value. The first chunk header produced for each child shard in the new epoch computes `proposed_split = None` (because `can_reshard` returns `false` due to `min_epochs_between_resharding > 0`). The invariant check in `validate_chunk_with_chunk_extra_and_receipts_root` then rejects every child-shard chunk with `InvalidChunkHeaderShardSplit`, making it impossible to advance the chain on any child shard after a dynamic shard split.

---

### Finding Description

In `process_memtrie_resharding_storage_update`, the child `ChunkExtra` is constructed as:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

Only `state_root` and `congestion_info` are updated. `proposed_split` is never touched, so it retains whatever value the parent held.

The parent shard's `ChunkExtra` at the resharding block (the last block of epoch N) has `proposed_split = Some(TrieSplit {...})` because `compute_proposed_split` fires when `is_next_block_possibly_last_in_epoch` is true and the shard is above the memory threshold. [2](#0-1) 

The child `ChunkExtra` is then saved under the child `ShardUId` at the resharding block hash: [3](#0-2) 

In epoch N+1, when the first chunk of a child shard is produced, `compute_proposed_split` returns `None` because `can_reshard` returns `false` (the cooldown `min_epochs_between_resharding > 0` has not elapsed). The chunk header therefore carries `proposed_split = None`.

`validate_chunk_with_chunk_extra_and_receipts_root` then compares:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [4](#0-3) 

`prev_chunk_extra.proposed_split()` = `Some(TrieSplit)` (inherited from parent)  
`chunk_header.proposed_split()` = `None` (freshly computed)  
→ mismatch → `InvalidChunkHeaderShardSplit` → chunk rejected.

This check is called from `validate_chunk_with_chunk_extra` during `apply_chunks_preprocessing` for every new chunk on a child shard: [5](#0-4) 

The `ChunkExtraV5` struct that holds `proposed_split`: [6](#0-5) 

The dynamic resharding architecture document explicitly lists this as an open TODO:

> `chain/chain/src/resharding/manager.rs:249` — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field). [7](#0-6) 

---

### Impact Explanation

Every chunk produced for a child shard in the first epoch after a dynamic shard split is rejected by `validate_chunk_with_chunk_extra_and_receipts_root`. No child-shard chunk can be included in any block. The chain cannot advance on those shards, causing a complete stall of all child shards immediately after resharding. Because resharding is a normal protocol operation (not an attack), this is triggered automatically whenever the dynamic resharding threshold is crossed.

**Severity: High** — complete chain stall on all child shards after any dynamic shard split.

---

### Likelihood Explanation

The bug fires deterministically on the first block of epoch N+1 after any dynamic shard split, provided `ProtocolFeature::DynamicResharding` is enabled. No adversarial action is required; the protocol itself triggers it. The only mitigation in the current code is that `DynamicResharding` must be enabled by protocol version upgrade, so the window is bounded to deployments where that feature flag is active.

---

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra`, explicitly reset `proposed_split` to `None` on the child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Reset proposed_split: child shards start fresh; the parent's split
// value must not propagate or the first chunk header validation fails.
*child_chunk_extra.proposed_split_mut() = None;
```

This is the direct analog of calling `touch()` on the newly created order in the GMX bug: the commitment field must be explicitly initialized to its correct post-creation value rather than inherited from the object it was cloned from.

---

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` (protocol version that activates it).
2. Run a network with a shard whose trie memory usage exceeds `split_threshold`.
3. At the last block of epoch N, `compute_proposed_split` returns `Some(TrieSplit)` for the parent shard; this is stored in `ChunkExtra`.
4. `process_memtrie_resharding_storage_update` clones the parent `ChunkExtra` into both child `ChunkExtra` records, each carrying `proposed_split = Some(TrieSplit)`.
5. At the first block of epoch N+1, the chunk producer for each child shard calls `compute_proposed_split`, which returns `None` (cooldown active).
6. The chunk header is signed with `proposed_split = None`.
7. `validate_chunk_with_chunk_extra` compares `Some(TrieSplit) != None` and returns `Err(InvalidChunkHeaderShardSplit)`.
8. The chunk is rejected; no child-shard chunk can ever be included; both child shards are permanently stalled.

    fn compute_proposed_split(
        &self,
        shard_trie: &Trie,
        shard_id: ShardId,
        epoch_id: &EpochId,
        protocol_version: ProtocolVersion,
        epoch_config: &EpochConfig,
        height: BlockHeight,
        prev_block_hash: &CryptoHash,
    ) -> Result<Option<TrieSplit>, Error> {
        if !ProtocolFeature::DynamicResharding.enabled(protocol_version) {
            return Ok(None);
        }

        let Some(config) = epoch_config.dynamic_resharding_config() else {
            return Ok(None);
        };

        if !self.epoch_manager.is_next_block_possibly_last_in_epoch(height, prev_block_hash)? {
            return Ok(None);
        }

        if !self.epoch_manager.can_reshard(prev_block_hash, config.min_epochs_between_resharding)? {
            return Ok(None);
        }

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
        match check_dynamic_resharding(shard_trie, shard_id, shard_layout, config) {
            Err(FindSplitError::Storage(err)) => Err(err)?,
            Err(err) => {
                DYNAMIC_RESHARDING_FIND_SPLIT_ERRORS
                    .with_label_values(&[&shard_uid.to_string()])
                    .inc();
                tracing::error!(target: "runtime", ?shard_id, ?err, "dynamic resharding check failed");
                Ok(None)
            }
            Ok(split) => Ok(split),
        }
