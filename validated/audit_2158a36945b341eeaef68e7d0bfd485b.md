### Title
`batch_insert` Bypasses Duplicate-Key and Duplicate-Hash Validation for All Items Beyond the First Two, Corrupting DataLayer Merkle Tree Roots and Enabling Forged Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` applies the full duplicate-key and duplicate-hash validation only to the last two items in the input batch (via `insert()`), while silently writing all remaining items directly to the blob via `insert_entry_to_blob()` with no uniqueness checks. An untrusted caller supplying a batch of three or more entries that contains a repeated key or hash bypasses the invariant enforced by `insert()`, producing a structurally corrupt Merkle tree whose root hash is wrong and whose inclusion proofs are invalid.

### Finding Description

`MerkleBlob::insert()` enforces two critical invariants before writing a leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When the tree already has two or more leaves, it skips the `insert()` call entirely for all items except the last two (which are `pop()`-ed and routed through `insert()`). Every other item in the batch is written directly via `insert_entry_to_blob()`:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // validated
    }
}

for ((key, value), hash) in keys_values_hashes {   // all remaining items
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // NO duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` unconditionally writes the block and updates the cache:

```rust
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    ...
}
``` [3](#0-2) 

`add_leaf` calls `key_to_index.insert(leaf.key, index)`, which silently overwrites the cache entry for a duplicate key, leaving the old leaf node orphaned in the blob but still wired into the internal-node tree structure. [4](#0-3) 

The subsequent subtree-linking phase (`insert_subtree_at_key`) then attaches the entire batch subtree — which may contain duplicate leaves — to the existing tree, producing internal nodes whose stored hashes are computed from the corrupted leaf set. [5](#0-4) 

The Python binding `py_batch_insert` exposes this path directly to callers with no additional filtering: [6](#0-5) 

### Impact Explanation

After a corrupt `batch_insert`:

1. `calculate_lazy_hashes()` propagates hashes upward through the corrupted structure, producing a root hash that does not correspond to any valid key-value set.
2. `get_proof_of_inclusion()` generates a proof whose `combined_hash` fields are derived from the corrupted internal nodes. `proof.valid()` returns `true` because the proof is internally self-consistent — but the root it proves against is wrong. [7](#0-6) 
3. A verifier comparing the proof's root hash against a known-good root will see a mismatch, but a verifier who trusts the tree's own root (e.g., after accepting the blob from an untrusted source) will accept a proof for a key-value pair that was never legitimately inserted.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

### Likelihood Explanation

The Python binding is the primary DataLayer API surface. Any caller that constructs a batch of three or more entries — including one with a repeated key or hash — triggers the bug. The condition is reachable with ordinary, unprivileged input; no special privileges or key material are required. The fuzz target for `batch_insert` does not test for duplicate keys within a single batch call, so this path is not covered by existing fuzz infrastructure. [8](#0-7) 

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the fast path inside `batch_insert` before calling `insert_entry_to_blob`. Concretely, before writing each leaf in the bulk loop, check:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, route all items through `insert()` and rely on the existing `InsertLocation::Auto` path, accepting the performance trade-off, or pre-validate the entire input vector for uniqueness before entering the fast path.

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate so leaf_count > 1, ensuring the fast path is taken
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i),
        &Hash(Bytes32::new([i as u8; 32])),
        chia_datalayer::InsertLocation::Auto {}).unwrap();
}

// Batch with a duplicate key (KeyId(0) already exists)
let dup_key = KeyId(0);
let batch = vec![
    ((dup_key, ValueId(99)), Hash(Bytes32::new([0xAA; 32]))),
    ((KeyId(100), ValueId(100)), Hash(Bytes32::new([0xBB; 32]))),
    ((KeyId(101), ValueId(101)), Hash(Bytes32::new([0xCC; 32]))),
];
// Should return Err(KeyAlreadyPresent) but succeeds silently
blob.batch_insert(batch).unwrap();

// Tree now has two leaf nodes for KeyId(0); root hash is corrupted
blob.calculate_lazy_hashes().unwrap();
let proof = blob.get_proof_of_inclusion(dup_key).unwrap();
// proof.valid() == true, but root hash does not match any legitimate state
assert!(proof.valid());
```

The duplicate key bypasses the `KeyAlreadyPresent` guard because `batch_insert` never calls `block_status_cache.contains_key()` for items in the bulk loop. The resulting tree root is computed from a blob containing two leaves for `KeyId(0)`, making every subsequent inclusion proof untrustworthy.
