### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS program accepts any arbitrary class hash as the replacement class without verifying that the hash corresponds to a previously declared class. This is an explicitly acknowledged missing check (marked with a `TODO` comment). A contract deployer can exploit this to replace a contract's class hash with an undeclared value, rendering the contract permanently inaccessible and freezing all funds held within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The OS unconditionally writes the caller-supplied `class_hash` into the contract's `StateEntry` with no check against the `contract_class_changes` dictionary (which tracks declared classes). Compare this to the `execute_declare_transaction` path, which enforces `prev_value=0` to guarantee a class is declared exactly once before it can be used:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

The `StateEntry` struct stores `class_hash` as the sole pointer to the contract's executable code: [3](#0-2) 

Once `class_hash` is set to an undeclared value, every subsequent call to the contract will fail to resolve any entry point, because the class does not exist in the class tree. The contract's storage and balance remain in the state trie but are permanently unreachable.

The analog to the Linea bridge bug is direct:

| Linea Bridge | StarkNet OS |
|---|---|
| No check of token's native status on the remote layer | No check that replacement `class_hash` is declared |
| Both tokens get `NATIVE_STATUS` simultaneously | Contract's `class_hash` is set to a non-existent class |
| Bridging completes with wrong token semantics | All future calls to the contract fail permanently |
| Funds transferred to wrong recipient / minted incorrectly | Funds locked in contract forever |

---

### Impact Explanation

**Impact: Critical — Permanent Freezing of Funds.**

Any contract that calls `replace_class` with an undeclared class hash transitions into a permanently broken state. Its `class_hash` field in the global state trie points to a class that does not exist in the class commitment tree. No transaction can successfully execute against the contract thereafter. All ETH, ERC-20 tokens, or other assets held in the contract's storage are permanently inaccessible. The state commitment is updated and finalized by the OS, making the freeze irreversible without a protocol-level upgrade.

---

### Likelihood Explanation

**Likelihood: Medium.**

The `replace_class` syscall is a standard, documented StarkNet syscall available to any deployed contract. A contract deployer (an unprivileged actor) can:

1. Deploy a contract (via `deploy_account` or the `deploy` syscall in `execute_deploy`) that accepts user deposits and implements an upgrade mechanism using `replace_class`.
2. Call `replace_class` supplying a felt value that has never been passed through `execute_declare_transaction` — i.e., a hash with no corresponding entry in `contract_class_changes`.
3. The OS writes the invalid `class_hash` into the contract's `StateEntry` without any cross-check against the class tree.
4. The contract is permanently frozen.

The attack surface is further widened by any contract that exposes `replace_class` to external callers without access control, allowing a third-party attacker (not just the deployer) to trigger the freeze on a contract holding other users' funds.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, the OS must verify that the hash exists in `contract_class_changes` (i.e., it was previously declared). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `request.class_hash` and assert the returned `compiled_class_hash` is non-zero (`!= UNINITIALIZED_CLASS_HASH`). This mirrors the invariant already enforced in `execute_declare_transaction` and closes the state-consistency gap.

---

### Proof of Concept

1. Declare a legitimate class `C` with hash `H_C` via `execute_declare_transaction`. This writes `(H_C → compiled_hash)` into `contract_class_changes`.
2. Deploy a contract `X` at address `A` using class `H_C`. The OS verifies `state_entry.class_hash == UNINITIALIZED_CLASS_HASH` before writing. [4](#0-3) 

3. Contract `X` calls the `replace_class` syscall with `class_hash = 0xDEADBEEF` — a felt that was never declared.
4. `execute_replace_class` reads `request.class_hash = 0xDEADBEEF`, skips any existence check (the TODO line), and calls `dict_update` writing `StateEntry(class_hash=0xDEADBEEF, ...)` for address `A`. [5](#0-4) 

5. The block is proven and finalized. The global state root now commits to contract `A` having `class_hash = 0xDEADBEEF`.
6. Any subsequent invoke, L1 handler, or call-contract targeting `A` will attempt to resolve entry points against a class that does not exist in the class tree. All executions fail. All funds in `A`'s storage are permanently frozen.
