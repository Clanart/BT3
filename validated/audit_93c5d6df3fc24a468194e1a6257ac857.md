### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts an arbitrary class hash from the caller and updates the contract's on-chain state entry without verifying that the new class hash corresponds to a declared contract class. An explicit TODO in the code acknowledges this missing check. A malicious contract deployer can exploit this to permanently freeze all funds held by a contract by replacing its class with an undeclared hash, after which no call to the contract can ever succeed.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 878–916) reads `request.class_hash` directly from the syscall request and writes it into `contract_state_changes` via `dict_update` with no validation that the hash is present in the block's compiled class facts:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The compiled class facts available to the OS are fixed at block-execution time inside `CompiledClassFactsBundle` (referenced from `OsGlobalContext`): [2](#0-1) 

When any subsequent call is made to a contract whose class hash is not present in those compiled class facts, the OS cannot resolve the class and the call fails unconditionally. Because the OS itself does not enforce that `replace_class` targets a declared hash, a provably valid block can be produced that permanently sets a contract's class hash to an undeclared value.

This is directly analogous to the external report's vulnerability class: **a critical parameter (class hash) is accepted from caller-controlled input and written into persistent state without being validated against the authoritative registry** (compiled class facts), allowing an attacker to commit the contract to an irrecoverable state.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, every future call to that contract (including withdrawals, transfers, or any fund-recovery function) will fail at the OS level. The contract's storage and any token balances it holds become permanently inaccessible. The state transition is committed to the proven block output and cannot be reversed without a protocol-level upgrade.

---

### Likelihood Explanation

**Medium.** Deploying a contract on StarkNet is a permissionless action available to any unprivileged user. An attacker can:

1. Deploy a contract that accepts user deposits and exposes a `replace_class` backdoor (e.g., gated by a time-lock or a hidden admin key).
2. Attract user funds.
3. Call the backdoor, which issues `replace_class` with an arbitrary undeclared hash.
4. The OS processes the syscall, updates the state entry, and produces a valid proof — no sequencer collusion required.

The only social-engineering barrier is convincing users to deposit into the contract. The OS-level missing check is the necessary vulnerable step; without it, the syscall would revert.

---

### Recommendation

Inside `execute_replace_class`, before writing the new state entry, verify that `class_hash` is present in `contract_class_changes` (i.e., has a non-zero compiled class hash mapping). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the result is non-zero, mirroring the check already performed implicitly during `execute_declare_transaction`: [3](#0-2) 

This ensures `replace_class` can only target classes that have been declared and whose compiled class hash is known to the OS.

---

### Proof of Concept

1. **Attacker deploys** a contract `MaliciousVault` that:
   - Accepts ERC-20 deposits from users.
   - Exposes an owner-only `drain_class()` function that calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared hash).

2. **Users deposit funds** into `MaliciousVault`, trusting its published source code.

3. **Attacker calls** `drain_class()`. The OS executes `execute_replace_class`:
   - `request.class_hash = 0xdeadbeef` is written into `contract_state_changes` for `MaliciousVault`.
   - No validation is performed against compiled class facts.
   - The revert log records the old class hash, but the transaction is not reverted.

4. **Block is proven.** The state transition is committed: `MaliciousVault.class_hash = 0xdeadbeef`.

5. **All subsequent calls to `MaliciousVault` fail** — including any withdrawal function — because the OS cannot resolve class `0xdeadbeef` from `CompiledClassFactsBundle`.

6. **All deposited funds are permanently frozen.** [4](#0-3)

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }
