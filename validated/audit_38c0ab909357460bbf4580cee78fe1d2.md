### Title
State-witness receipt-proof shard-ID mismatch at resharding boundary causes permanent witness rejection — (File: chain/chain/src/stateless_validation/state_witness.rs)

---

### Summary

At a shard-split resharding boundary, `collect_source_receipt_proofs` collects `ReceiptProof` objects whose Merkle hashes were committed against the **parent** shard ID. The downstream validator (`validate_source_receipt_proofs`) re-hashes those receipts using the **child** shard ID. The two hashes are structurally different, so `verify_path` always fails, and every state witness produced for the first chunk of a child shard after resharding is permanently rejected by all validators.

---

### Finding Description

**Analog mapping.** The external bug's pattern is: a value is transformed through a conditional path (self-lending pod unwrap → borrow token), but the downstream consumer always uses the original type (`PAIRED_LP_TOKEN`), causing a type mismatch. In nearcore the same pattern appears as a shard-ID mismatch: the receipt proof is committed against the parent shard ID, but the verifier always uses the child shard ID.

**Root cause — producer side.**

`collect_source_receipt_proofs` calls `get_incoming_receipts_for_shard` with `ReceiptFilter::All`: [1](#0-0) 

Inside `get_incoming_receipts_for_shard`, when the loop crosses the epoch boundary where the shard layout changed, `current_shard_id` is silently replaced with the **parent** shard ID: [2](#0-1) 

The `ReceiptProof` objects fetched from the DB were stored when the parent shard still existed. Their Merkle proof leaf was committed as:

```
hash(ReceiptList(parent_shard_id, receipts_for_parent))
```

These proofs are inserted verbatim into the state witness `source_receipt_proofs` map, keyed by chunk hash: [3](#0-2) 

The codebase itself acknowledges this is broken: [4](#0-3) 

**Root cause — validator side.**

`pre_validate_chunk_state_witness` calls `validate_source_receipt_proofs` with `last_chunk_shard_id`, which is the **child** shard ID: [5](#0-4) 

The verifier recomputes the leaf hash as:

```
hash(ReceiptList(child_shard_id, receipts_for_child))
```

Because `child_shard_id ≠ parent_shard_id`, and because the receipts for the child are a strict subset of the parent's receipts, both the shard-ID field and the receipt list differ from what was committed. `verify_path` returns `false` for every proof, and the witness is rejected with `InvalidChunkStateWitness`.

**Exact corrupted value.** The `CryptoHash` leaf in the Merkle proof stored in `source_receipt_proofs` is `hash(ReceiptList(parent_shard_id, full_parent_receipts))`. The validator computes `hash(ReceiptList(child_shard_id, filtered_child_receipts))`. These two hashes are always different at a resharding boundary.

---

### Impact Explanation

**Severity: High.**

Every state witness produced by the chunk producer for the first new chunk of a child shard after a shard split is permanently invalid from the validators' perspective. No validator can endorse the chunk. Without endorsements, block producers cannot include the chunk in a block, stalling the affected shard indefinitely until the node software is patched and restarted. The failure is deterministic and affects all validators simultaneously.

---

### Likelihood Explanation

**Likelihood: High.**

The condition triggers at every shard-split resharding boundary when stateless validation (chunk state witnesses) is active. Resharding has occurred on mainnet and is expected to recur. The TODO comment at line 260 of `state_witness.rs` confirms the developers are aware the code is broken at this boundary. No existing guard prevents the mismatch from reaching `verify_path`.

---

### Recommendation

As the TODO itself prescribes: collect the **full** parent-shard proof (all receipts, parent shard ID) from `get_incoming_receipts_for_shard`, include it in the witness verbatim so the Merkle proof remains valid, and only **after** the validator has verified the proof against the committed Merkle root should it filter out the receipts that belong to the sibling child shard. Concretely:

1. In `collect_source_receipt_proofs`, pass the parent shard ID (not the child shard ID) as the target when crossing the resharding boundary, and store the unfiltered proof.
2. In `validate_source_receipt_proofs`, verify the proof using the parent shard ID, then filter the verified receipt list to retain only receipts whose `receiver_shard_id` maps to the child shard being validated.

---

### Proof of Concept

1. Run a two-node testnet with stateless validation enabled and a forced shard split scheduled for epoch E+2.
2. At the first block of epoch E+2, the chunk producer for the newly created child shard calls `create_state_witness` → `collect_source_receipt_proofs`.
3. `get_incoming_receipts_for_shard` crosses the epoch boundary, switches `current_shard_id` to the parent shard ID, and returns `ReceiptProof` objects whose leaf hashes embed the parent shard ID.
4. The witness is distributed to chunk validators.
5. Each validator calls `pre_validate_chunk_state_witness` → `validate_source_receipt_proofs` with the child shard ID.
6. `verify_path` fails for every proof because `hash(ReceiptList(child_shard_id, …)) ≠ hash(ReceiptList(parent_shard_id, …))`.
7. All validators return `InvalidChunkStateWitness`; no endorsement is produced; the child shard stalls.

    /// boundaries, because it removes the receipts that target the other half of a split shard,
    /// which makes the proof invalid. We need to collect the original proof and later, after verification,
    /// filter it to remove the receipts that were meant for the other half of the split shard.
    fn collect_source_receipt_proofs(
        &self,
        epoch_manager: &dyn EpochManagerAdapter,
        prev_block_header: &BlockHeader,
        prev_chunk_header: &ShardChunkHeader,
    ) -> Result<HashMap<ChunkHash, ReceiptProof>, Error> {
        if prev_chunk_header.is_genesis() {
            // State witness which proves the execution of the first chunk in the blockchain
            // doesn't have any source receipts.
            return Ok(HashMap::new());
        }

        // Find the first block that included `prev_chunk`.
        // Incoming receipts were generated by the blocks before this one.
        let mut cur_block;
        let prev_chunk_original_block: &BlockHeader = {
            if prev_chunk_header.is_new_chunk(prev_block_header.height()) {
                prev_block_header
            } else {
                cur_block = self.get_block_header(prev_block_header.prev_hash())?;
                loop {
                    if prev_chunk_header.is_new_chunk(cur_block.height()) {
                        break &cur_block;
                    }
                    cur_block = self.get_block_header(cur_block.prev_hash())?;
                }
            }
        };

        // Get the last block that contained `prev_prev_chunk` (the chunk before `prev_chunk`).
        // We are interested in all incoming receipts that weren't handled by `prev_prev_chunk`.
        let prev_prev_chunk_block = self.get_block(prev_chunk_original_block.prev_hash())?;
        // Find the header of the chunk before `prev_chunk`
        let prev_prev_chunk_header = epoch_manager
            .get_prev_chunk_header(&prev_prev_chunk_block, prev_chunk_header.shard_id())?;

        // Fetch all incoming receipts for `prev_chunk`.
        // They will be between `prev_prev_chunk.height_included` (first block containing `prev_prev_chunk`)
        // and `prev_chunk_original_block`
        let shard_layout = epoch_manager
            .get_shard_layout_from_prev_block(prev_chunk_original_block.prev_hash())?;
        let incoming_receipt_proofs = get_incoming_receipts_for_shard(
            &self,
            epoch_manager,
            prev_chunk_header.shard_id(),
            &shard_layout,
            *prev_chunk_original_block.hash(),
            prev_prev_chunk_header.height_included(),
            ReceiptFilter::All,
        )?;
