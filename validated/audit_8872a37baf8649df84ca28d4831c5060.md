### Title
Meta-Transaction Replay via Missing Nonce in `compute_meta_tx_v0_hash` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a version-0 transaction hash that omits the nonce entirely. Because the hash is a pure function of `(contract_address, selector, calldata, chain_id)`, any attacker who observes a valid meta-transaction on-chain can replay the identical signature in a fresh outer transaction, causing the target account's `__execute__` to run again with no protocol-level defence. This enables direct, repeated theft of funds from any account that accepts meta-transactions.

---

### Finding Description

**Vulnerability class**: Fee/accounting attribution bug → transaction replay (no sender-unique binding in the signed message).

**Root cause — two cooperating defects:**

**1. `compute_meta_tx_v0_hash` omits the nonce (`transaction_hash.cairo` lines 295–315)**

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // ← nonce slot is empty
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

Compare with `compute_l1_handler_transaction_hash`, which correctly passes the nonce as `additional_data`: [2](#0-1) 

**2. `execute_meta_tx_v0` hardcodes `nonce=0` and never calls `check_and_increment_nonce` (`syscall_impls.cairo` lines 343–363)**

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    account_contract_address=contract_address,
    ...
    transaction_hash=meta_tx_hash,
    ...
    nonce=0,          // ← always zero, never incremented
    ...
);
``` [3](#0-2) 

No `check_and_increment_nonce` is invoked anywhere in `execute_meta_tx_v0`, unlike every other account transaction type (`execute_invoke_function_transaction`, `execute_deploy_account_transaction`, `execute_declare_transaction`), all of which call it. [4](#0-3) 

The result: the OS-level hash `H(INVOKE_PREFIX, 0, contract_address, __execute__, calldata_hash, 0, chain_id)` is **identical** for every replay of the same calldata. The target contract receives `tx_info.nonce = 0` on every invocation and has no protocol-supplied counter to distinguish executions.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A standard account contract's `__execute__` verifies the caller's signature against `tx_info.transaction_hash`. Because that hash is nonce-free and the `nonce` field is always `0`, a valid `(signature, calldata)` pair authorises the same action an unlimited number of times. An attacker who copies the `MetaTxV0Request` fields from any confirmed transaction can drain the victim's account by replaying the signed transfer/approval in successive outer transactions, each with the attacker's own (valid) nonce.

---

### Likelihood Explanation

**High.** The attacker's entry path requires only:
1. Observing a confirmed block containing a `meta_tx_v0` syscall (public on-chain data).
2. Extracting `contract_address`, `selector`, `calldata`, and `signature` from the call.
3. Submitting a new invoke transaction that calls the same relayer contract (or any contract that issues `meta_tx_v0`) with the copied fields.

No privileged access, leaked key, or off-chain coordination is required. The attacker is an ordinary unprivileged transaction sender.

---

### Recommendation

Include the nonce in `compute_meta_tx_v0_hash` via `additional_data`, mirroring `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,
    );
    return tx_hash;
}
```

Additionally, `execute_meta_tx_v0` must:
- Accept a caller-supplied nonce from the `MetaTxV0Request`.
- Call `check_and_increment_nonce` against the target contract's on-chain nonce before execution.

---

### Proof of Concept

1. **Setup**: Deploy account contract `A` at address `0xACC`. `A.__execute__` verifies a Stark signature over `tx_info.transaction_hash` and transfers tokens to `calldata[0]`.

2. **Victim action**: User signs a meta-tx authorising transfer of 100 STRK to `0xATTACKER`. A relayer submits outer transaction `T1` containing `meta_tx_v0(contract_address=0xACC, selector=__execute__, calldata=[0xATTACKER, 100], signature=[r,s])`. `T1` succeeds; 100 STRK transferred.

3. **Replay**: Attacker copies `(contract_address, selector, calldata, signature)` from `T1`. Submits new outer transaction `T2` (attacker's own nonce) with identical `MetaTxV0Request`.

4. **OS behaviour**: `execute_meta_tx_v0` recomputes `meta_tx_hash = H(INVOKE_PREFIX, 0, 0xACC, __execute__, calldata_hash, 0, chain_id)` — **identical** to `T1`'s hash. Sets `new_tx_info.nonce = 0`. Calls `A.__execute__` with the same `tx_info`.

5. **Account behaviour**: `A.__execute__` verifies `(r, s)` against the same hash — **passes**. Transfers another 100 STRK to `0xATTACKER`.

6. **Result**: Attacker repeats until `A`'s balance is zero. Direct loss of all funds held by `A`.
