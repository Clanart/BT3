### Title
Integer Wrapping in Relative Timelock Validation Allows Timelock Bypass — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary
When `check_time_locks` is called with `nowrap=false`, relative height and seconds timelock comparisons use `wrapping_add` instead of `saturating_add`. An attacker-controlled spend carrying `ASSERT_HEIGHT_RELATIVE(0xffff_ffff)` or `ASSERT_SECONDS_RELATIVE(u64::MAX)` causes the addition to wrap around to a value smaller than the current block height/timestamp, making the timelock check pass immediately instead of failing. This is the direct analog of the external report's missing bounds check: just as an unchecked fee field lets anyone drain the minter's Algo, an unchecked overflow in the timelock arithmetic lets anyone bypass a coin's relative timelock and spend it before the intended maturity.

### Finding Description
`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` accepts a `nowrap: bool` parameter. When `nowrap=false`, the relative-height and relative-seconds checks use `wrapping_add`:

```rust
// lines 64-68
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

```rust
// lines 75-77
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

If `confirmed_block_index = 100` and `height_relative = 0xffff_ffff`, then `100u32.wrapping_add(0xffff_ffff) = 99`. The guard `prev_transaction_block_height < 99` is false at any height ≥ 99, so the check silently passes — the spend is accepted as if the ~4-billion-block timelock had already elapsed. The test suite explicitly documents this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),  // nowrap=true
    Ok(()),                                                           // nowrap=false ← bypass
)]
```

The same wrapping flaw applies to `seconds_relative` (u64 wrapping) and, in the opposite direction, to `before_height_relative` / `before_seconds_relative` (where wrapping causes a valid spend to be incorrectly rejected).

The function is exposed as a Python binding (`py_check_time_locks`) and is the sole gate enforcing relative timelocks for spend bundles submitted to the mempool.

### Impact Explanation
This is a **High** impact finding matching the allowed scope: *"timelock… validation bypass enables unauthorized spend acceptance."*

Any coin whose puzzle enforces only `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a large value (near u32::MAX / u64::MAX) can be spent immediately when the validator uses `nowrap=false`. The attacker does not need any key material beyond what is required by the puzzle itself; the bypass is purely arithmetic. Coins locked for vesting, escrow, or delayed-reveal purposes are fully unprotected under this code path.

### Likelihood Explanation
The `nowrap` parameter is caller-controlled from Python. The Python full node passes `nowrap=False` for pre-fork block validation. If any mempool or block-validation call site passes `nowrap=False` for current-height blocks (which the Rust library cannot prevent), the bypass is immediately reachable by any unprivileged user who can submit a spend bundle. The test suite confirms the wrapping path is live and produces the wrong result.

### Recommendation
Replace all four `wrapping_add` calls in `check_time_locks` with `saturating_add` unconditionally, and remove the `nowrap` branch entirely. The `nowrap=false` path has no safe use: for historical blocks the chain state is already committed, so re-validating with wrapping arithmetic serves no purpose and only introduces risk. If backward-compatible re-validation of old blocks is required, it should be handled at a higher layer, not by exposing an unsafe arithmetic mode in the consensus-critical timelock gate.

### Proof of Concept
Attacker steps (pseudocode):

1. Construct a puzzle that outputs `(ASSERT_HEIGHT_RELATIVE 0xffffffff)` and requires no signature.
2. Fund a coin with this puzzle hash.
3. At any block height ≥ 1 (i.e., immediately after confirmation), submit a spend bundle for the coin.
4. The full node calls `check_time_locks(..., nowrap=False)`.
5. `confirmed_block_index.wrapping_add(0xffff_ffff)` wraps to `confirmed_block_index - 1`.
6. `prev_transaction_block_height < (confirmed_block_index - 1)` is false → no error returned.
7. The spend is accepted; the ~4-billion-block timelock is completely bypassed.

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,

        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
