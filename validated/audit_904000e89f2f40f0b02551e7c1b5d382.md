### Title
Unsigned `ContractCodeResponse::V1` Accepted Without Sender Identity Check, Allowing Any Peer to Corrupt Witness Assembly Cache — (File: `chain/client/src/stateless_validation/validate.rs`)

---

### Summary

Before `ProtocolFeature::SignedContractCodeResponse` (protocol version 85) is active, `validate_contract_code_response` performs no sender-identity or signature check on `ContractCodeResponse::V1` messages. Any network peer — not just a legitimate chunk producer — can send a `ContractCodeResponse::V1` carrying arbitrary contract bytes for any valid `ChunkProductionKey`. The receiving chunk-validator stores those bytes unconditionally into its `PartialEncodedStateWitnessTracker` cache. When the witness is later assembled, the injected bytes are merged into the witness's `base_state` and used during chunk-state-witness validation, causing the validator to compute a wrong post-state root, fail validation, and withhold its endorsement for an otherwise-valid chunk.

---

### Finding Description

`validate_contract_code_response` in `chain/client/src/stateless_validation/validate.rs` (lines 369–382) gates the signature check behind a protocol-version flag:

```rust
pub fn validate_contract_code_response(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
    store: &Store,
) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);   // ← only height/epoch window
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)
}
``` [1](#0-0) 

When the feature is **not** yet active, the only gate is `validate_chunk_relevant`, which merely checks that the chunk height falls within the `(final_head, head + MAX_HEIGHTS_AHEAD]` window. No sender identity, no chunk-producer membership check, and no cryptographic signature is required.

`ContractCodeResponse::V1` carries no `responder` field at all — `responder()` returns `None` for V1 — so there is nothing to verify even if the caller tried:

```rust
pub fn responder(&self) -> Option<&AccountId> {
    match self {
        Self::V1(_) => None,          // ← no identity in V1
        Self::V2(v2) => Some(&v2.inner.responder),
    }
}
``` [2](#0-1) 

The handler in `PartialWitnessActor` then stores whatever bytes the response contains:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    if !validate_contract_code_response(...).is_relevant() { return Ok(()); }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
``` [3](#0-2) 

`store_accessed_contract_codes` is called with `create_if_not_exists: false`, meaning it only writes when a cache entry already exists (i.e., after a `ChunkContractAccesses` message has been received). An attacker who races the legitimate chunk-producer response — or who sends the fake response immediately after observing the `ChunkContractAccesses` broadcast — can win the race and install wrong bytes into `AccessedContractsState::Received`. [4](#0-3) 

When `try_finalize` fires, the corrupted bytes are returned alongside the decoded witness and passed to `validate_chunk_state_witness`, where they are merged into the witness's `base_state` via `merge_contracts`. The re-execution then uses the wrong contract code, produces a wrong post-state root, and the validator returns `Err(InvalidChunkStateWitness(...))` instead of sending a `ChunkEndorsement`.

---

### Impact Explanation

A chunk-state witness requires endorsements from a threshold of assigned chunk validators before the block producer can include the chunk. If an attacker corrupts the `accessed_contracts` cache entry for a specific `ChunkProductionKey` on enough validators, none of them will endorse the chunk, stalling shard block production. Because the `ChunkProductionKey` is predictable (epoch, shard, height), the attacker can target the exact set of validators assigned to a shard for a given height and suppress endorsements for every chunk in that epoch window.

**Impact: High** — targeted, deterministic liveness failure for a shard; no stake or validator role required by the attacker.

---

### Likelihood Explanation

The attack requires:
1. A network peer connection to the targeted validator (any node on the NEAR P2P network qualifies).
2. Knowledge of the `ChunkProductionKey` (epoch, shard, height) — fully public from block headers.
3. Timing the fake `ContractCodeResponse::V1` to arrive after `ChunkContractAccesses` but before the legitimate response — a narrow but achievable race on a well-connected adversary node.

The window exists on every chunk produced before protocol version 85 is activated on mainnet.

---

### Recommendation

1. **Immediate**: In `validate_contract_code_response`, unconditionally verify that the `responder` is a current-epoch chunk producer for the shard, regardless of protocol version. The `ContractCodeResponse::V1` variant carries no identity, so V1 messages should be rejected outright (or the check should be applied to the network-layer sender identity if available).
2. **Structural**: Accelerate the rollout of `ProtocolFeature::SignedContractCodeResponse` (protocol version 85), which introduces `ContractCodeResponseV2` with a mandatory chunk-producer signature, matching the pattern already used by `ChunkContractAccesses` and `ContractCodeRequest`.
3. **Defense-in-depth**: In `store_accessed_contract_codes`, verify that each received `CodeBytes` hashes to one of the `CodeHash` values in the `Requested { missing }` set before transitioning to `Received`, so that even a response that passes the outer gate cannot inject bytes whose hash does not match the expected contract hashes.

---

### Proof of Concept

```
1. Observe a ChunkContractAccesses broadcast for (epoch_id=E, shard_id=S, height=H).
   This tells the network that validators for (E,S,H) are waiting for contract code.

2. Construct a ContractCodeResponse::V1:
     next_chunk = ChunkProductionKey { epoch_id: E, shard_id: S, height_created: H }
     compressed_contracts = <compress(arbitrary_wrong_bytes)>
   No signature field exists in V1; the message is valid on the wire.

3. Send this message directly to each chunk validator assigned to (E,S,H)
   before the legitimate chunk producer's response arrives.

4. validate_contract_code_response passes (height is in window, no sig check pre-v85).
   store_accessed_contract_codes writes the wrong bytes into the cache entry
   created by the earlier ChunkContractAccesses handling.

5. When enough witness parts arrive, try_finalize fires, merges the wrong bytes
   into the witness base_state, and validate_chunk_state_witness fails with a
   post-state-root mismatch.  No ChunkEndorsement is sent.

6. Repeat for every height in the epoch to suppress all endorsements for shard S.
```
