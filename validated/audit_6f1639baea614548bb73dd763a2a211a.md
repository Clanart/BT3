### Title
Timelock Bypass via Integer Wrapping in `check_time_locks` When `nowrap=False` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` uses `wrapping_add` for relative timelock arithmetic when the `nowrap` parameter is `false`. An unprivileged spender can craft an `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` condition whose value, when added to the coin's `confirmed_block_index` / `timestamp`, wraps around to a small integer, making the timelock appear already satisfied at any block height. This allows spending a coin before its intended relative timelock has elapsed.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` boolean:

- `nowrap=true` → `saturating_add` (correct: overflow clamps to `u32::MAX` / `u64::MAX`, timelock is never spuriously satisfied)
- `nowrap=false` → `wrapping_add` (unsafe: overflow wraps to a small value, timelock check passes immediately)

For `ASSERT_HEIGHT_RELATIVE` (lines 55–68):

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

If `confirmed_block_index = 100` and `height_relative = 0xFFFF_FFF6` (u32::MAX − 9), then `wrapping_add` yields `100 + 0xFFFF_FFF6 = 0x1_0000_0060 mod 2^32 = 0x60 = 96`. The check becomes `prev_height < 96`, which is `false` at any height ≥ 96 — the timelock is bypassed regardless of how large the intended delay was.

The same wrapping flaw applies to `ASSERT_SECONDS_RELATIVE` (lines 70–78) using `u64::wrapping_add`.

The test suite explicitly documents and confirms this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
```

The function is exposed to Python via `py_check_time_locks` (lines 118–141), where `nowrap` is a caller-supplied parameter. The Python node passes `nowrap=false` for pre-hard-fork blocks, meaning any coin confirmed before the hard fork height is subject to this bypass.

---

### Impact Explanation

An attacker who controls a coin's puzzle (or solution) can embed an `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` condition with a crafted large value. When the node evaluates that spend under `nowrap=false` (legacy block rules), the wrapping arithmetic makes the timelock appear satisfied at any block height. The attacker can spend the coin immediately, bypassing the intended relative timelock. This constitutes an unauthorized spend acceptance — a High-severity timelock validation bypass.

---

### Likelihood Explanation

The `nowrap=false` path is the active code path for all blocks below the hard-fork activation height. Any coin confirmed in that range whose puzzle includes a crafted `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value is exploitable. The attacker needs only to control the puzzle or solution of their own coin; no privileged access, key compromise, or network-level attack is required. The crafted value is a simple arithmetic calculation from the coin's known `confirmed_block_index`.

---

### Recommendation

1. Remove the `nowrap=false` branch entirely from `check_time_locks`. The `saturating_add` path is the correct semantic for all timelock arithmetic: a relative delay that overflows the integer type should never be treated as already elapsed.
2. If backward compatibility with pre-hard-fork consensus is required, document the exact block height at which `nowrap` transitions and add a consensus-level guard that rejects any spend with `height_relative > u32::MAX - confirmed_block_index` under the legacy rules, rather than silently wrapping.
3. Audit all callers of `py_check_time_locks` in the Python node to confirm `nowrap=true` is passed for all blocks at or above the hard-fork height.

---

### Proof of Concept

Given a coin confirmed at block height `H` (e.g., `H = 100`), craft a puzzle that emits:

```
(ASSERT_HEIGHT_RELATIVE . (u32::MAX - H + 1))
; = (ASSERT_HEIGHT_RELATIVE . 0xFFFF_FF9C)  for H=100
```

Under `nowrap=false`:
```
confirmed_block_index.wrapping_add(height_relative)
= 100u32.wrapping_add(0xFFFF_FF9C)
= 0x1_0000_003C mod 2^32
= 0x3C = 60
```

The check `prev_height < 60` is `false` at any height ≥ 60, so `check_time_locks` returns `Ok(())` — the spend is accepted at block 60, even though the puzzle intended a delay of ~4 billion blocks.

The same construction applies to `ASSERT_SECONDS_RELATIVE` using `u64::MAX - timestamp + 1` as the crafted value. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
