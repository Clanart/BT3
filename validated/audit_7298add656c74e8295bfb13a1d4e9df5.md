### Title
Permissionless `logMetadata`/`log_metadata` Trusts Attacker-Controlled Token Contracts, Enabling Unbacked Bridge Token Minting - (File: evm/src/omni-bridge/contracts/OmniBridge.sol, near/omni-bridge/src/lib.rs)

---

### Summary

Both `OmniBridge.logMetadata()` on EVM and `Contract.log_metadata()` on NEAR are permissionless entry points that call `name()`, `symbol()`, `decimals()` (or `ft_metadata()`) on a **user-supplied contract address** without any authenticity or ownership verification. A malicious contract can return arbitrary values from these functions. The returned metadata is then used to register a new token mapping in the bridge and deploy a bridge token on the destination chain, enabling an attacker to mint unbacked bridged assets.

---

### Finding Description

**EVM path â€” `OmniBridge.logMetadata`:**

`logMetadata` is a public, payable, access-control-free function:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
```

`tokenAddress` is entirely attacker-controlled. The bridge calls `name()`, `symbol()`, and `decimals()` on it and unconditionally trusts the return values. In `OmniBridgeWormhole`, `logMetadataExtension` publishes a Wormhole VAA containing those attacker-controlled values. The emitter address of the VAA is the OmniBridge contract itself â€” a registered factory on NEAR.

**NEAR path â€” `Contract.log_metadata`:**

```rust
#[pause(except(roles(Role::DAO)))]
pub fn log_metadata(&self, token_id: &AccountId) -> Promise {
    ext_token::ext(token_id.clone())
        .with_static_gas(LOG_METADATA_GAS)
        .ft_metadata()
        .then(
            Self::ext(env::current_account_id())
                ...
                .log_metadata_callback(token_id),
        )
}
```

`token_id` is entirely attacker-controlled. The bridge calls `ft_metadata()` on it and in `log_metadata_callback` constructs a `MetadataPayload` from the returned values, then requests the MPC signer to sign it:

```rust
ext_signer::ext(self.mpc_signer.clone())
    ...
    .sign(SignRequest { payload, path: SIGN_PATH.to_owned(), key_version: 0 })
```

The MPC has no way to distinguish a legitimate token from a malicious one. The resulting signature is emitted as `LogMetadataEvent` and is valid for use in EVM `deployToken`.

### Impact Explanation

**Full EVM â†’ NEAR attack path:**

1. Attacker deploys a malicious ERC-20 contract that returns arbitrary `name()`, `symbol()`, `decimals()` values and allows the attacker to mint unlimited tokens.
2. Attacker calls `logMetadata(malicious_erc20)` on `OmniBridge`. The bridge emits a `LogMetadata` event (in Wormhole variant: a VAA) with attacker-controlled metadata and `tokenAddress = malicious_erc20`. The emitter is the OmniBridge â€” a known factory.
3. Attacker submits proof of this event to NEAR `deploy_token()`. `deploy_token_callback` verifies the emitter is a known factory âś“ and calls `deploy_token_internal`, deploying a NEAR omni-token for `malicious_erc20` with attacker-chosen name/symbol/decimals.
4. Attacker mints arbitrary amounts of `malicious_erc20` to themselves.
5. Attacker calls `initTransfer` on EVM with `malicious_erc20` â€” the bridge locks the tokens and emits `InitTransfer`.
6. Attacker submits proof to NEAR `fin_transfer`. `fin_transfer_callback` verifies the emitter is a known factory âś“ and finds the token registered in `token_decimals` âś“ (registered in step 3). NEAR mints omni-tokens to the attacker.

Result: The attacker receives NEAR omni-tokens backed by worthless self-minted ERC-20. The bridge's collateralization is broken â€” it holds worthless tokens while having minted real omni-tokens that can be sold to unsuspecting users.

**Full NEAR â†’ EVM attack path (via MPC signing):**

1. Attacker deploys a malicious NEAR NEP-141 contract with arbitrary minting and arbitrary `ft_metadata()` return values.
2. Attacker calls `log_metadata(malicious_contract)` on NEAR bridge â†’ MPC signs the metadata payload â†’ `LogMetadataEvent` emitted with a valid MPC signature.
3. Attacker calls `deployToken` on EVM with the MPC signature. EVM verifies `ECDSA.recover(hashed, signatureData) == nearBridgeDerivedAddress` âś“. EVM deploys a bridge token mapped to `malicious_contract` and emits `DeployToken`.
4. Attacker calls `bind_token` on NEAR with proof of the `DeployToken` event. `bind_token_callback` verifies the emitter is a known factory âś“ and registers the token mapping.
5. Attacker mints arbitrary malicious NEAR tokens â†’ `ft_transfer_call` to NEAR bridge â†’ `InitTransfer` emitted â†’ relayer calls `sign_transfer` â†’ MPC signs â†’ attacker calls `finTransfer` on EVM â†’ EVM bridge mints bridge tokens to attacker.

Result: Unauthorized mint of EVM bridge tokens backed by worthless self-minted NEAR tokens.

---

### Likelihood Explanation

- Both entry points are fully permissionless and callable by any unprivileged user.
- No special role, key, or privileged access is required.
- Deploying a malicious ERC-20 or NEAR NEP-141 contract is trivial.
- The attack requires only standard on-chain interactions (no off-chain coordination beyond waiting for proof finality).
- Likelihood: **High**.

---

### Recommendation

1. **Require token ownership/deployer verification**: Before accepting a token address in `logMetadata` / `log_metadata`, verify that `msg.sender` (EVM) or `env::predecessor_account_id()` (NEAR) is the owner or deployer of the token contract. This is the direct analog to the oracle deployer check in the external report.
2. **Alternatively, implement an allowlist**: Maintain a governance-controlled allowlist of approved token addresses that can be registered via `logMetadata` / `log_metadata`.
3. **At minimum, document and surface the risk in the UI**: If permissionlessness is intentional (as noted in the CLAUDE.md), clearly display a warning in the bridge UI when a token was registered via an unverified `logMetadata` call, so users can distinguish legitimate bridge tokens from attacker-registered ones.

### Proof of Concept

**EVM path (simplified):**

```solidity
// Attacker deploys this contract
contract MaliciousToken {
    function name() external pure returns (string memory) { return "USD Coin"; }
    function symbol() external pure returns (string memory) { return "USDC"; }

    function transferFrom(address, address, uint256) external pure returns (bool) { return true; }

// Step 1: Register malicious token
omniBridge.logMetadata(address(maliciousToken)); // permissionless, no check

// Step 2: Submit Wormhole VAA proof to NEAR deploy_token() â†’ NEAR omni-token deployed

// Step 3: Mint and bridge

maliciousToken.approve(address(omniBridge), 1_000_000e6);
omniBridge.initTransfer(address(maliciousToken), 1_000_000e6, ...);

// Step 4: Submit InitTransfer proof to NEAR fin_transfer() â†’ 1,000,000 NEAR omni-tokens minted to attacker
```

    #[private]
    #[result_serializer(borsh)]
    pub fn log_metadata_callback(
        &self,
        #[callback] metadata: FungibleTokenMetadata,
        token_id: &AccountId,
    ) -> Promise {
        require!(
            !metadata.name.is_empty() && !metadata.symbol.is_empty(),
            BridgeError::InvalidMetadata.as_ref()
        );

        let metadata_payload = MetadataPayload {
            prefix: PayloadType::Metadata,
            token: token_id.to_string(),
            name: metadata.name,
            symbol: metadata.symbol,
            decimals: metadata.decimals,
        };

        let payload = near_sdk::env::keccak256_array(
            borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_LOG_METADATA_CALLBACK_GAS)
                    .sign_log_metadata_callback(metadata_payload),
            )
    }

    function logMetadataExtension(
        address tokenAddress,
        string memory name,
        string memory symbol,
        uint8 decimals
    ) internal override {
        bytes memory payload = bytes.concat(
            bytes1(uint8(MessageType.LogMetadata)),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(tokenAddress),
            Borsh.encodeString(name),
            Borsh.encodeString(symbol),
            bytes1(decimals)
        );
        // slither-disable-next-line reentrancy-eth
        _wormhole.publishMessage{value: msg.value}(
            wormholeNonce,
            payload,
            _consistencyLevel
        );

        wormholeNonce++;
    }
