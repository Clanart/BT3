The `receive()` guard at line 32-34 only blocks plain ETH transfers (no calldata). ETH sent via `payable` function calls (`addLiquidityExactShares{value: X}(...)`) bypasses `receive()` entirely and lands on the contract. The vulnerability is confirmed.

Audit Report

## Title
Unattributed `address(this).balance` in `PeripheryPayments.pay()` WETH branch allows stranded ETH from one user to fund another user's WETH liquidity payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads the contract's total native ETH balance without per-caller attribution when settling WETH payments. Because all liquidity entry points are `payable`, ETH sent with a non-WETH pool call is never consumed and sits on the contract. Any subsequent WETH-pool liquidity call silently consumes that stranded ETH, causing a direct, unrecoverable cross-user loss of principal.

## Finding Description

The `pay()` internal function's WETH branch reads `address(this).balance` globally: [1](#0-0) 

There is no per-caller ETH accounting. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers with no calldata — it does **not** prevent ETH from being deposited via `msg.value` in `payable` function calls: [2](#0-1) 

All three liquidity entry points are `payable`: [3](#0-2) [4](#0-3) 

When User A calls `addLiquidityExactShares{value: X}` for a non-WETH pool, `pay()` takes the ERC20 branch for both tokens and the X ETH is never consumed. It remains on the contract until `refundETH()` is called: [5](#0-4) 

`refundETH()` sends the **entire** contract balance to `msg.sender` — whichever caller reaches it first wins. If User B's WETH-pool callback fires first, `pay()` consumes User A's ETH: [6](#0-5) 

The transient pay context stores `payer` for cap enforcement but does not restrict which ETH `pay()` may consume — the ETH attribution gap is entirely in `PeripheryPayments.pay()`. [7](#0-6) 

## Impact Explanation

Direct, cross-user loss of native ETH principal with no recovery path. Once User B's WETH-pool call consumes User A's stranded ETH, `refundETH()` returns 0 to User A. The lost amount equals `min(stranded ETH, WETH amount requested by pool)`, which can be up to the full value User A deposited. This is a Critical/High direct loss of user principal meeting Sherlock thresholds.

## Likelihood Explanation

The Uniswap-style router UX pattern of sending ETH with a liquidity call and relying on a separate `refundETH()` step is standard and expected. Any user who:
- Sends ETH with `addLiquidityExactShares` for a non-WETH pool,
- Sends excess ETH with a WETH pool call, or
- Uses `multicall` with ETH and omits `refundETH` as the last step

...leaves ETH on the contract. A MEV bot observing the mempool can immediately follow with a WETH-pool liquidity call to consume it. No special privileges are required; any unprivileged caller can trigger this.

## Recommendation

Track per-caller ETH deposits in transient storage alongside the existing pay context. In `pay()`'s WETH branch, only consume ETH attributed to the current `payer` (stored at `_setPayContext` time as `msg.value`). Alternatively, pass `msg.value` explicitly through the call stack and consume only that amount rather than reading `address(this).balance`.

## Proof of Concept

1. User A calls `addLiquidityExactShares{value: 1 ether}(nonWethPool, ...)`. Pool requests ERC20 tokens only; `pay()` calls `safeTransferFrom` for both. 1 ETH remains on the contract.
2. Attacker calls `addLiquidityExactShares(wethPool, ...)` (no ETH sent) for a pool where `token0 = WETH` and the pool requests `amount0Delta = 1e18`.
3. `metricOmmModifyLiquidityCallback` loads `payer = attacker`, calls `pay(WETH, attacker, wethPool, 1e18)`.
4. `nativeBalance = 1 ether >= 1e18` → `IWETH9.deposit{value: 1 ether}()` + `safeTransfer(wethPool, 1e18)`. Attacker's WETH allowance is not touched.
5. User A's 1 ETH is gone. `refundETH()` now returns 0 to User A.
