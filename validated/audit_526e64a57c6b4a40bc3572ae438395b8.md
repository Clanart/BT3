### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary `class_hash` value and updates the contract's state entry without verifying that the class hash is actually declared in the class registry. The code itself contains an explicit TODO acknowledging this missing check. An attacker who can trigger `replace_class` on a target contract with an attacker-controlled class hash can permanently render that contract non-functional, freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without any validation against the `contract_class_changes` dictionary (the declared class registry): [1](#0-0) 

The critical lines are:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `contract_class_changes` dict is available as an implicit argument to `execute_replace_class` but is **never consulted** to verify that `class_hash` is a declared class. The TODO comment at line 898 is an explicit developer acknowledgment that this validation is absent. [2](#0-1) 

For contrast, the normal `execute_declare_transaction` path enforces `prev_value=0` to prevent re-declaration and verifies the Sierra class hash via `finalize_class_hash`: [3](#0-2) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `StateEntry.class_hash` is set to an undeclared value, every subsequent call to that contract will fail at the class-lookup stage (the OS cannot find compiled code for the unknown hash). The contract's storage and token balances remain intact in the state trie but become permanently inaccessible — a classic permanent fund freeze. Because the `revert_log` entry is written *after* the `dict_update`, a successful `replace_class` with an invalid hash is committed to state and cannot be undone by the caller. [4](#0-3) 

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who can cause a target contract to execute `replace_class` with an attacker-supplied class hash. Realistic entry paths include:

1. **Upgradeable contracts with insufficient access control** — a contract exposes an upgrade function that accepts the new class hash as a calldata parameter. If the contract does not validate the hash itself (relying on the OS to do so), the attacker passes an undeclared hash.
2. **Contracts that delegate upgrade authority to a role that the attacker holds** — e.g., a multisig where the attacker controls enough signers.

The OS is the protocol-level enforcement layer. Contract authors have a reasonable expectation that the OS will reject `replace_class` calls with undeclared hashes, just as it rejects re-declarations via `prev_value=0`. The missing check breaks this expectation and shifts the entire burden of validation onto every individual contract.

---

### Recommendation

Inside `execute_replace_class`, before writing the new `StateEntry`, verify that `class_hash` exists in `contract_class_changes`:

```cairo
// Verify the class is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the invariant already enforced during `execute_declare_transaction` and closes the gap the TODO comment identifies.

---

### Proof of Concept

1. Deploy an upgradeable contract `Victim` that holds token funds and exposes:
   ```cairo
   fn upgrade(new_class_hash: ClassHash) {
       replace_class_syscall(new_class_hash).unwrap();
   }
   ```
   (No access control, or attacker holds the upgrade role.)

2. Attacker calls `Victim.upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared.

3. The OS executes `execute_replace_class`:
   - `request.class_hash = 0xdeadbeef`
   - The TODO check is absent; no lookup in `contract_class_changes` occurs.
   - `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` for `Victim`'s address.
   - The syscall succeeds; the transaction is committed.

4. Any subsequent call to `Victim` (e.g., to withdraw funds) reaches the class-lookup step in the OS, finds no compiled class for `0xdeadbeef`, and fails.

5. All funds held by `Victim` are permanently frozen. [2](#0-1)
