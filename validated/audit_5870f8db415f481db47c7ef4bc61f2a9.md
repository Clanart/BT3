### Title
Gas Refund Issued Against `origCtx` After EVM State Committed to `tmpCtx` — Enabling Double-Spend of EVM-Denom Funds - (File: `x/evm/keeper/state_transition.go`)

### Summary

In `ApplyTransaction`, when `k.hooks != nil`, EVM state is committed into a cache context `tmpCtx`. However, the gas refund (`RefundGas`) is unconditionally issued against the **original** `ctx`, not `tmpCtx`. Because fee deduction (ante handler) and gas refund both operate on the real bank store through `ctx`, but the EVM state transition (including any balance changes from EVM execution) is committed into `tmpCtx`, a failed post-processing hook causes `tmpCtx` to be discarded while the gas refund already credited to the sender via `ctx` is **permanently retained**. This allows an attacker to receive a gas refund for a transaction whose EVM execution effects are fully reverted, resulting in a net gain of EVM-denom funds from the fee collector.

### Finding Description

**Root cause — `ApplyTransaction` in `x/evm/keeper/state_transition.go`:**

The flow is:

1. **Ante handler** deducts `gasLimit * gasPrice` from the sender via `SendCoinsFromAccountToModuleVirtual` on `ctx` (the real store).
2. `ApplyTransaction` creates `tmpCtx = ctx.CacheContext()` and runs EVM execution inside it.
3. If `PostTxProcessing` hook returns an error, `commit()` is never called — `tmpCtx` is discarded, reverting all EVM state changes.
4. `RefundGas` is then called with `ctx` (not `tmpCtx`), issuing `leftoverGas * gasPrice` back to the sender via `SendCoinsFromModuleToAccountVirtual` on the **real** bank store.
5. The sender receives the refund on `ctx` permanently, but the EVM execution effects (including any value transfers or contract state changes) are gone.

**The accounting invariant is broken:** The ante handler charged `gasLimit * gasPrice`. The refund returns `leftoverGas * gasPrice`. Net cost to sender = `gasUsed * gasPrice`. This is correct when the EVM execution is committed. But when the hook fails and `tmpCtx` is discarded, the EVM execution's balance effects (e.g., value sent to a contract, tokens minted by a precompile) are also discarded — yet the refund still lands. If the attacker crafted the transaction to receive value *inside* the EVM execution (e.g., via a precompile that mints tokens into `tmpCtx`), those minted tokens are lost, but the gas refund is still paid. More critically: the gas refund is paid from the `FeeCollector` module account on `ctx`, which was funded by the ante handler's fee deduction. This path is always reachable and the refund is always net-positive for the sender relative to a correctly-reverted transaction.

**Concrete over-refund scenario:**

- Sender submits a tx with `gasLimit = 1,000,000`, `gasPrice = 1`.
- Ante handler deducts `1,000,000` from sender → fee collector.
- EVM executes, uses `gasUsed = 21,000`. `leftoverGas = 979,000`.
- `PostTxProcessing` hook returns an error → `tmpCtx` discarded, EVM state reverted.
- `RefundGas(ctx, ...)` credits `979,000` back to sender from fee collector.
- Net: sender paid `21,000` for a transaction whose effects were fully reverted. The fee collector is drained by `979,000` without any corresponding committed EVM state.

If the hook is designed to always fail for certain contract calls (e.g., a hook that rejects specific ERC-20 transfers), an attacker can repeatedly submit high-gas transactions, trigger the hook failure, and drain the fee collector while paying only `gasUsed * gasPrice` per attempt — far less than `gasLimit * gasPrice`.

### Impact Explanation

**High — EVM state transition / gas refund / fee accounting bug that permits valid user funds/fees to be mis-accounted.**

The fee collector module account is drained by `leftoverGas * gasPrice` per transaction that triggers a hook failure, while the EVM execution effects are fully reverted. Over many transactions, this drains the fee collector, which is the escrow for all EVM transaction fees. This constitutes unauthorized transfer of EVM-denom funds from the fee collector to the attacker's address, matching the allowed High impact: "EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."

### Likelihood Explanation

**Medium.** The precondition is that `k.hooks != nil` (i.e., the chain has registered `EvmHooks`) AND the hook returns an error for the attacker's transaction. On chains using ERC-20/IBC hooks (e.g., Evmos-style deployments), hooks are always registered. An attacker who can predict or force a hook failure (e.g., by calling a contract that the hook rejects) can reliably trigger this path. No privileged access is required — any unprivileged user can submit a `MsgEthereumTx` with a high gas limit targeting a contract whose post-processing hook is known to fail.

### Recommendation

Move `RefundGas` to execute against `tmpCtx` (before the `commit()`/discard decision), or restructure so that the refund is only issued after the final committed context is determined. Specifically:

1. If `PostTxProcessing` fails and `tmpCtx` is discarded, do **not** issue the gas refund (or issue it against `tmpCtx` so it is also discarded).
2. If `PostTxProcessing` succeeds and `commit()` is called, issue the refund against `ctx` after `commit()`.

Alternatively, always issue the refund against `tmpCtx` and let `commit()` propagate it to `ctx` on success, and discard it on hook failure.

func (k *Keeper) RefundGas(ctx sdk.Context, msg *core.Message, leftoverGas uint64, denom string) error {
	return k.RefundGasWithPrice(ctx, msg, leftoverGas, msg.GasPrice, denom)
}

// RefundGasWithPrice transfers the leftover gas to sender using the provided gas price.
func (k *Keeper) RefundGasWithPrice(
	ctx sdk.Context,
	msg *core.Message,
	leftoverGas uint64,
	gasPrice *big.Int,
	denom string,
) error {
	if gasPrice == nil {
		gasPrice = new(big.Int)
	}

	// Return EVM tokens for remaining gas, exchanged at the original rate.
	remaining := new(big.Int).Mul(new(big.Int).SetUint64(leftoverGas), gasPrice)

	switch remaining.Sign() {
	case -1:
		// negative refund errors
		return errorsmod.Wrapf(types.ErrInvalidRefund, "refunded amount value cannot be negative %d", remaining.Int64())
	case 1:
		// positive amount refund
		refundedCoins := sdk.Coins{sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(remaining))}

		// refund to sender from the fee collector module account, which is the escrow account in charge of collecting tx fees
		err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
		if err != nil {
			err = errorsmod.Wrapf(errortypes.ErrInsufficientFunds, "fee collector account failed to refund fees: %s", err.Error())
			return errorsmod.Wrapf(err, "failed to refund %d leftover gas (%s)", leftoverGas, refundedCoins.String())
		}
	default:
		// no refund, consume gas and update the tx gas meter
	}

	return nil
}

func DeductFees(bankKeeper types.BankKeeper, ctx sdk.Context, acc sdk.AccountI, fees sdk.Coins) error {

		return errorsmod.Wrapf(errortypes.ErrInsufficientFee, "invalid fee amount: %s", fees)

		}
	}
	return nil
