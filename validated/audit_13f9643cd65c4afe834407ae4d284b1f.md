### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows cross-shard state-root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`StateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer for `shard_id=X` but never asserts that the embedded `ShardChunk` actually belongs to shard X. A malicious peer can embed shard Y's chunk (with a valid Merkle path against `chunk_headers_root`) and pass all validation steps, causing the node to store shard Y's `prev_state_root` under `DBCol::StateHeaders[StateHeaderKey(X, sync_hash)]`. Subsequent `set_state_part` and `apply_state_part` calls then reconstruct shard X's trie from shard Y's state data.

### Finding Description

`set_state_header` performs five validation steps:

1. **`validate_chunk_proofs(&chunk, ...)`** — checks header hash self-consistency, tx Merkle root, and outgoing-receipt Merkle root. No `shard_id` field check. [1](#0-0) 

2. **`verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk_hash, height_included))`** — verifies the chunk hash appears somewhere in the block's chunk-header Merkle tree. The path encodes position implicitly, but the code never checks that the position corresponds to shard X's index. [2](#0-1) 

3. **`prev_chunk` inclusion check** — same structure; no shard_id binding. [3](#0-2) 

4. **Receipt proof checks (steps 4a–4g)** — use the `shard_id` parameter (X) to compute `receipts_hash = hash_borsh(ReceiptList(X, receipts))`, but `root` and `block_proof` are attacker-supplied. The only trusted anchor is `block_header.prev_chunk_outgoing_receipts_root()`. An attacker can supply the legitimate incoming-receipt proofs for shard X (obtained by observing the network) independently of which chunk is embedded. [4](#0-3) 

5. **`validate_state_root_node`** — validates the state root node against `chunk_inner.prev_state_root()` (shard Y's root). Passes trivially for a valid shard Y header. [5](#0-4) 

After all checks pass, the header is committed: [6](#0-5) 

`set_state_part` then reads this header, extracts `state_root` from shard Y's chunk, and validates incoming parts against shard Y's state root while writing them under shard X's `StatePartKey`: [7](#0-6) 

`apply_state_part` (called from `chain/client/src/sync/state/shard.rs`) writes trie changes to the `shard_uid` derived from `shard_id=X` and `epoch_id`, but the trie data comes from shard Y's state parts: [8](#0-7) 

`set_state_finalize` then calls `apply_chunk` with `chunk_header.prev_state_root()` (shard Y's root) on `shard_uid` for shard X, producing and persisting a wrong `ChunkExtra` for shard X: [9](#0-8) 

### Impact Explanation

The exact corrupted values are:

| Store entry | Expected | Actual (after attack) |
|---|---|---|
| `DBCol::StateHeaders[StateHeaderKey(X, sync_hash)]` | Shard X's `ShardStateSyncResponseHeader` | Shard Y's header |
| `DBCol::StateParts[StatePartKey(sync_hash, X, *)]` | Parts of shard X's trie | Parts of shard Y's trie |
| Shard X's trie (keyed by shard X's `ShardUId`) | Shard X's account/contract state | Shard Y's account/contract state |
| `ChunkExtra` for shard X at `height_included` | Correct post-state root for shard X | Post-execution root of shard Y's transactions on shard Y's state |

The node ends up tracking the wrong state root for shard X. Any subsequent block processing, chunk validation, or state queries for shard X will use the wrong trie, causing the node to diverge from consensus and be unable to participate in block production or validation for shard X.

### Likelihood Explanation

An unprivileged peer connected to the syncing node can respond to state sync header requests. NEAR's state sync protocol does not authenticate the source of `ShardStateSyncResponseHeader` messages beyond basic network connectivity. The attacker needs only to:
1. Be a connected peer of the syncing node.
2. Respond to the state sync header request for shard X with a crafted header containing shard Y's chunk.
3. Supply valid state parts for shard Y (obtainable from any legitimate state sync provider on the network).

No validator, chunk producer, or operator privileges are required.

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD: reject cross-shard substitution
if chunk.shard_id() != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
``` [10](#0-9) 

Similarly add the same guard for `prev_chunk_header.shard_id()` after it is extracted.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
