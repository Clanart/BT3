Audit Report

## Title
Any caller can drain all ETH held by the router via `refundETH()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` unconditionally transfers the router's entire ETH balance to `msg.sender` with no access control. ETH accumulates on the router whenever a user sends more `msg.value` than the swap requires through any payable entry point, because `pay()` wraps only the exact required amount and leaves the rest. A subsequent caller can invoke `refundETH()` to steal that stranded ETH.

## Finding Description
`PeripheryPayments.refundETH()` (L58–63) sends `address(this).balance` to `msg.sender` unconditionally:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

ETH accumulates via the `pay()` function (L73–77): when `token == WETH` and `nativeBalance >= value`, exactly `value` ETH is wrapped and forwarded; any excess `msg.value` beyond `value` remains on the router after the transaction ends.

The `receive()` guard (L32–34) only blocks *direct* ETH transfers (no calldata) from non-WETH addresses. It does not prevent ETH accumulation through payable function calls. All four swap entry points — `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), `exactOutput` (L154) — are `payable` and accept arbitrary `msg.value`. `multicall` (L39–44) is also `payable` and performs no automatic refund.

**Concrete attack path:**
1. User A calls `exactInputSingle{value: 2 ether}(...)` with `tokenIn = WETH`, `amountIn = 1 ether`.
2. The pool callback fires; `pay()` wraps exactly 1 ETH and sends WETH to the pool. 1 ETH remains on the router.
3. Transaction ends. 1 ETH is stranded on the router.
4. Attacker calls `refundETH()` in a separate transaction. `address(this).balance == 1 ETH` → transferred to attacker.

No privileged role, no malicious pool, and no non-standard token is required.

## Impact Explanation
Direct loss of user ETH principal. The attacker receives ETH that belongs to a legitimate user. Impact is bounded only by the amount of excess ETH sent in the victim's transaction, which can be arbitrarily large. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation
Any user who calls a payable swap function directly (not via multicall, or via multicall without appending `refundETH()`) with excess `msg.value` is vulnerable. This is a common usage pattern for ETH→token swaps where the exact input amount may not be known precisely in advance. A bot monitoring the mempool can back-run the victim's transaction to call `refundETH()` immediately after, making exploitation trivially repeatable.

## Recommendation
Restrict `refundETH()` to only refund ETH that was sent by `msg.sender` in the current call context. Options include:
- Use transient storage to record per-caller ETH contributions at each payable entry point and only refund the recorded amount in `refundETH()`.
- Automatically refund excess ETH at the end of each swap entry point rather than relying on the caller to include a separate `refundETH()` step.
- Track `msg.value` at the start of each payable call and refund the unused portion before returning.

## Proof of Concept
```solidity
// User A swaps ETH→Token, sends 2 ETH but only 1 ETH is needed
router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
    tokenIn: WETH, amountIn: 1 ether, ...
}));
// 1 ETH now sits on the router

// Attacker in a separate tx:
router.refundETH(); // receives 1 ETH belonging to User A
assertEq(attacker.balance, 1 ether); // passes
```
