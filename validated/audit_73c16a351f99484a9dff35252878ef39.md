### Title
Cross-Chain Replay of `deploy_token` Signatures via Missing Chain ID in `MetadataPayload` Borsh Encoding — (Files: `near/omni-bridge/src/lib.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` borsh encoding used to produce MPC signatures for `deploy_token` / `deployToken` across all destination chains (EVM, Starknet, Solana) contains no destination chain ID. Unlike `TransferMessagePayload`, which binds to a specific chain by including `chain_id` twice in the borsh hash, the metadata signature is fully chain-agnostic. Because the signature is emitted publicly as a NEAR log event, any observer can replay it to any chain. An attacker can deploy a bridge token on an unintended chain before the legitimate relayer, causing the legitimate deployment to revert with `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`. For chains that do not post a Wormhole confirmation back to NEAR (EVM, Starknet), NEAR never learns the deployed token address, permanently preventing users from bridging that token to those chains.

---

### Impact Explanation

**Impact: High — Cross-chain replay enabling unauthorized token deployment and permanent bridge DoS for affected tokens.**

1. Attacker observes the publicly emitted `LogMetadataEvent` on NEAR containing the MPC signature over `MetadataPayload` for token `foo.near`.
2. Attacker replays the identical `(signature, payload)` to EVM and Starknet `deploy_token` before the legitimate relayer.
3. Both chains accept the signature (it is cryptographically valid — no chain ID mismatch possible).
4. The token is deployed on EVM and Starknet by the attacker.
5. When the legitimate relayer subsequently attempts deployment on EVM or Starknet, it reverts: `ERR_TOKEN_EXIST` (EVM) / `ERR_TOKEN_ALREADY_DEPLOYED` (Starknet).
6. Neither EVM nor Starknet posts a Wormhole confirmation back to NEAR upon `deployToken` — NEAR never learns the deployed token address from the attacker-triggered deployment.
7. NEAR's token address registry for `foo.near` on EVM and Starknet remains empty. All subsequent `fin_transfer` calls for `foo.near` to EVM or Starknet fail because NEAR cannot route them to the correct token contract.
8. Result: **permanent inability to bridge `foo.near` to EVM and Starknet** for all users.

For Solana, the attacker's `deploy_token` call does post a `DeployTokenResponse` via Wormhole back to NEAR, so NEAR registers the Solana token correctly — the Solana path is less impacted. EVM and Starknet are the primary affected chains.

---

### Likelihood Explanation

**Likelihood: High.**

- The MPC signature is emitted as a public NEAR log event (`LogMetadataEvent`) — no privileged access or key material is needed.
- The attacker only needs to monitor NEAR events and submit a standard transaction to EVM/Starknet `deploy_token` with the observed `(signature, payload)`.
- No MEV infrastructure, no chain reorganization, no colluding validators required.
- The attack is profitable for any adversary wishing to permanently disable bridging of a competitor token or any token of value.
- The attack window is the time between NEAR emitting the `LogMetadataEvent` and the legitimate relayer submitting the deployment transaction — typically seconds to minutes.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` borsh encoding, mirroring the existing protection in `TransferMessagePayload`. On the NEAR signing side, pass the target `ChainKind` into `log_metadata_callback` and encode it into the payload before hashing. On each destination chain, encode the local `omniBridgeChainId` / `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the borsh hash before signature verification — exactly as `finTransfer` / `fin_transfer` already does.

---

### Proof of Concept

1. Call `log_metadata("foo.near")` on the NEAR bridge. NEAR fetches metadata, signs `MetadataPayload{Metadata, "foo.near", "Foo", "FOO", 18}`, and emits `LogMetadataEvent{signature, payload}` publicly.

2. Attacker reads the event and extracts `(signature, payload)`.

3. Attacker calls `OmniBridge.deployToken(signature, {token:"foo.near", name:"Foo", symbol:"FOO", decimals:18})` on EVM. The borsh hash matches exactly — `ECDSA.recover` returns `nearBridgeDerivedAddress`. Token is deployed. `isBridgeToken[proxy] = true`, `nearToEthToken["foo.near"] = proxy`. [4](#0-3) 

4. Attacker calls `deploy_token(signature, {token:"foo.near", name:"Foo", symbol:"FOO", decimals:18})` on Starknet. The borsh hash matches — `verify_eth_signature` passes. Token is deployed. `near_to_starknet_token[hash("foo.near")] = contract_address`. [10](#0-9) 

5. Legitimate relayer attempts `deployToken` on EVM → reverts `ERR_TOKEN_EXIST`. Attempts `deploy_token` on Starknet → reverts `ERR_TOKEN_ALREADY_DEPLOYED`.

6. NEAR's `token_id_to_address` for `(EVM, "foo.near")` and `(Starknet, "foo.near")` is never populated. All `fin_transfer` calls for `foo.near` to EVM or Starknet are permanently broken. [11](#0-10)
