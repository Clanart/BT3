### Title
Wrong Shard Layout Used to Index `prev_block.chunks()` in `get_state_response_part`, Producing Incorrect `prev_state_root` After Resharding - (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`get_state_response_part` resolves `shard_index` from the **new epoch's** shard layout and uses it to index into `prev_block.chunks()`, which is a block from the **previous epoch** and is therefore indexed by the **old epoch's** shard layout. After a resharding event, the shard_index of a surviving shard can shift (because the split inserts two new shards at the split position, pushing all subsequent shards up by one). The function therefore reads `prev_state_root` from the wrong chunk, generates state parts against that wrong root, and serves them to syncing peers. The receiving peer validates those parts against the correct root committed in the header (computed by `compute_state_response_header`, which correctly uses the previous epoch's layout), causing validation to fail and state sync to be permanently broken for every shard whose index shifted.

---

### Finding Description

**Buggy function** — `get_state_response_part` (`chain/chain/src/state_sync/adapter.rs`, lines 277–358):

```rust
let epoch_id = block.header().epoch_id();          // NEW epoch
let shard_layout = self.epoch_manager
    .get_shard_layout(epoch_id)?;                  // NEW epoch's layout
...
let prev_block = self.chain_store
    .get_block(header.prev_hash())?;               // PREV epoch's last block
let shard_index = shard_layout
    .get_shard_index(shard_id)?;                   // index in NEW layout ← BUG
let state_root = prev_block
    .chunks()
    .get(shard_index)                              // reads PREV block with NEW index
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

**Correct sibling function** — `compute_state_response_header` (same file, lines 93–101) explicitly fetches the **previous** epoch's layout and uses it to index into `sync_prev_block.chunks()`:

```rust
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager
    .get_shard_layout(&prev_epoch_id)?;            // PREV epoch's layout ← correct
let prev_shard_index = prev_shard_layout
    .get_shard_index(shard_id)?;                   // index in PREV layout ← correct
let chunk_header = chunks
    .get(prev_shard_index)...;
``` [2](#0-1) 

**Why shard_index shifts after resharding** — `ShardLayoutV2::derive` and `ShardLayoutV3::derive_impl` both use `Vec::splice` to replace the parent shard with two child shards at the same position. Every shard whose boundary-account position is *after* the split point gets its `shard_index` incremented by 1 in the new layout:

```rust
// V2 derive: parent at new_boundary_account_index is replaced by [max+1, max+2]
let parent_shard_id = shard_ids
    .splice(new_boundary_account_index..new_boundary_account_index + 1,
            new_shards.clone())
    .collect_vec();
``` [3](#0-2) 

**Concrete example:**

| Epoch | shard_ids | shard_indexes |
|-------|-----------|---------------|
| Old   | [3, 4, 7] | {3→0, 4→1, 7→2} |
| New (shard 3 split into 5,6) | [5, 6, 4, 7] | {5→0, 6→1, 4→2, 7→3} |

For `shard_id=4` in `get_state_response_part`:
- `shard_layout.get_shard_index(4)` → **2** (new layout)
- `prev_block.chunks().get(2)` → **shard 7's chunk** (old layout) → **wrong `prev_state_root`**

For `shard_id=7`:
- `shard_layout.get_shard_index(7)` → **3** (new layout)
- `prev_block.chunks().get(3)` → **out of bounds** → `Error::InvalidShardId`

**Production call path** — `StateRequestActor::handle(StateRequestPart)` calls `get_state_response_part` directly: [4](#0-3) 

The actor is spawned in production at node startup: [5](#0-4) 

**Validation that catches the mismatch** — `set_state_part` on the receiving node validates each part against the `prev_state_root` from the header (which `compute_state_response_header` computed correctly using the old layout): [6](#0-5) 

The parts generated from the wrong root fail this check, causing state sync to abort.

---

### Impact Explanation

Any node attempting state sync at an epoch boundary where resharding occurred will be permanently unable to complete sync for shards whose `shard_index` shifted. The serving peer returns parts keyed to the wrong `prev_state_root`; the receiving peer's `validate_state_part` rejects them; the syncing node retries indefinitely and never advances past the sync phase. This blocks new validators and RPC nodes from joining the network after a resharding event — a High-severity availability failure for the state-sync reconstruction invariant.

---

### Likelihood Explanation

Resharding is a planned, protocol-level event on NEAR mainnet. Every node that falls behind and needs to state-sync across a resharding epoch boundary will hit this path. The `StateRequestActor` is always running in production and handles `StateRequestPart` messages from any peer without privilege checks, so no special role is required to trigger the bug.

---

### Recommendation

In `get_state_response_part`, derive `shard_index` from the **previous** epoch's shard layout (matching the approach in `compute_state_response_header`). For child shards that do not exist in the previous layout, map the child `shard_id` to its parent via `shard_layout.get_parent_shard_id(shard_id)` before looking up the index:

```rust
let prev_block = self.chain_store.get_block(header.prev_hash())?;
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;

// If shard_id is a new child shard, resolve to its parent in the old layout.
let lookup_shard_id = if prev_shard_layout.shard_ids().any(|id| id == shard_id) {
    shard_id
} else {
    shard_layout.get_parent_shard_id(shard_id)?
};
let shard_index = prev_shard_layout.get_shard_index(lookup_shard_id)?;
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

### Proof of Concept

1. Configure a two-shard network (shards 3 and 4, `shard_index` 0 and 1).
2. Trigger a resharding that splits shard 3 into children 5 and 6. Post-resharding layout: `[5, 6, 4]` with `shard_index` `{5→0, 6→1, 4→2}`.
3. Let a new node fall behind so it must state-sync at the resharding epoch boundary.
4. The new node sends `StateRequestPart { shard_id: 4, sync_hash, part_id: 0 }` to a peer.
5. The peer's `StateRequestActor` calls `get_state_response_part(4, 0, sync_hash)`.
6. Inside, `shard_layout.get_shard_index(4)` returns **2** (new layout).
7. `prev_block.chunks().get(2)` returns the chunk for old shard 7 (or panics out-of-bounds if there are only 2 old shards), yielding the wrong `prev_state_root`.
8. The peer generates and returns parts against this wrong root.
9. The new node's `set_state_part` calls `validate_state_part` with the correct root from the header — validation fails.
10. State sync loops forever; the node never joins the network.
