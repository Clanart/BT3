### Title
Unchecked Constructor Failure in `execute_deploy` Syscall Always Reports Success — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall implementation in the StarkNet OS unconditionally writes `failure_flag=0` in the response header regardless of whether the deployed contract's constructor succeeded or failed. This is a direct analog to the "unsafe ERC20 operation" class: a critical sub-operation (the constructor) can fail silently, and the caller receives a success signal, leading to interaction with an uninitialized or partially-initialized contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function calls `deploy_contract` and then writes the syscall response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `failure_flag` field in `ResponseHeader` is the mechanism by which the OS communicates syscall failure back to the calling contract. By hardcoding it to `0` (success), the OS discards any constructor revert information. The developer-inserted TODO comment at line 538 explicitly acknowledges this gap: `"TODO(Yoni, 1/1/2026): support failures."` This is not a theoretical edge case — it is a documented, unimplemented code path.

Contrast this with `contract_call_helper`, which correctly propagates the `is_reverted` flag:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` function does not follow this pattern. The return value of `deploy_contract` — which includes constructor revert data — is structurally discarded at the response-writing step.

---

### Impact Explanation

**Critical — Direct loss of funds / Permanent freezing of funds.**

When a contract uses the `deploy` syscall to deploy a child contract (e.g., a vault, escrow, or token contract), it relies on `failure_flag` in the response to determine whether the constructor ran successfully. Because `failure_flag` is always `0`, the calling contract cannot distinguish between:

- A successfully deployed and initialized contract, and
- A deployed contract whose constructor reverted, leaving it in an uninitialized or partially-initialized state.

A concrete loss-of-funds scenario:

1. Contract A deploys Contract B (a vault) via the `deploy` syscall, expecting the constructor to set `owner = caller`.
2. The constructor of B fails (e.g., due to an out-of-gas condition, a failed inner call, or attacker-influenced calldata).
3. The OS writes `failure_flag=0` — Contract A sees a successful deploy and proceeds.
4. Contract A transfers user funds to Contract B's address.
5. Contract B has no initialized `owner`, so funds are permanently locked (or accessible to an unintended party).

This results in either **permanent freezing of funds** or **direct loss of funds**, both of which are in-scope critical impacts.

---

### Likelihood Explanation

The `deploy` syscall is a standard, publicly accessible operation reachable by any unprivileged transaction sender. Any contract that:
- Deploys a child contract and then interacts with it (transfers funds, grants roles, etc.), and
- Has a constructor that can fail under any condition (out-of-gas, inner call failure, invalid state)

is vulnerable. The attacker does not need any privileged access. They only need to craft a transaction that triggers a constructor failure — for example, by providing calldata that causes the constructor to run out of gas or to call a failing inner contract. The TODO comment confirms this is a known, unresolved gap in the production code path.

---

### Recommendation

Propagate the constructor's revert status through `deploy_contract`'s return value and write it into the `ResponseHeader.failure_flag` field, mirroring the pattern used in `contract_call_helper`:

```cairo
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

Additionally, when `is_reverted != 0`, the deploy should not register the contract address in state (or should revert the registration), so the caller cannot interact with an uninitialized contract.

---

### Proof of Concept

1. Deploy a factory contract on StarkNet that:
   - Calls the `deploy` syscall with a class whose constructor always reverts.
   - Reads `failure_flag` from the syscall response.
   - Asserts `failure_flag == 0` (this assertion will pass, demonstrating the silent failure).
   - Transfers funds to the deployed address.

2. The OS, executing `execute_deploy` at lines 527–553 of `syscall_impls.cairo`, will:
   - Call `deploy_contract`, which internally runs the constructor and observes the revert.
   - Write `ResponseHeader(gas=remaining_gas, failure_flag=0)` unconditionally (line 539).
   - Return the deployed contract address to the caller as if the deploy succeeded.

3. The factory contract receives `failure_flag=0`, proceeds to transfer funds to the address, and the funds are locked in an uninitialized contract. [3](#0-2)

    revert_log: RevertLogEntry*,
    outputs: OsCarriedOutputs*,

    let request = cast(syscall_ptr + RequestHeader.SIZE, DeployRequest*);

    let specific_base_gas_cost = DEPLOY_GAS_COST + DEPLOY_CALLDATA_FACTOR_GAS_COST *
        constructor_calldata_size;

        specific_base_gas_cost=specific_base_gas_cost, request_struct_size=DeployRequest.SIZE

        return ();

    tempvar deploy_from_zero = request.deploy_from_zero;
    assert deploy_from_zero * (deploy_from_zero - 1) = 0;

    let deployer_address = (1 - deploy_from_zero) * caller_address;

    let selectable_builtins = &builtin_ptrs.selectable;
    let hash_ptr = selectable_builtins.pedersen;
    with hash_ptr {
        let (contract_address) = get_contract_address(
            salt=request.contract_address_salt,
            class_hash=request.class_hash,
            constructor_calldata_size=constructor_calldata_size,
            constructor_calldata=constructor_calldata_start,
            deployer_address=deployer_address,
        );
    }
    tempvar builtin_ptrs = new BuiltinPointers(
        selectable=SelectableBuiltins(
            pedersen=hash_ptr,
            range_check=selectable_builtins.range_check,

            bitwise=selectable_builtins.bitwise,
            ec_op=selectable_builtins.ec_op,
            poseidon=selectable_builtins.poseidon,
            segment_arena=selectable_builtins.segment_arena,
            range_check96=selectable_builtins.range_check96,
            add_mod=selectable_builtins.add_mod,

        calldata=constructor_calldata_start,
        execution_info=new ExecutionInfo(
            block_info=caller_execution_info.block_info,
            tx_info=caller_execution_info.tx_info,

        deprecated_tx_info=caller_execution_context.deprecated_tx_info,
    );

    with remaining_gas {
        let (retdata_size, retdata) = deploy_contract(
            block_context=block_context, constructor_execution_context=constructor_execution_context
        );
    }

    // TODO(Yoni, 1/1/2026): consider sharing code with call_contract_helper.

    let syscall_ptr = syscall_ptr + ResponseHeader.SIZE;

    // Write the response header.

        constructor_retdata_start=retdata,
        constructor_retdata_end=retdata + retdata_size,
