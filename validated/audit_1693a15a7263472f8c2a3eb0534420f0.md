### Title
Starknet OmniBridge `bridge_token_class_hash` Has No Update Mechanism, Causing Permanent Stale-Class Deployment After BridgeToken Upgrades - (File: starknet/src/omni_bridge.cairo)

### Summary
The Starknet `OmniBridge` stores `bridge_token_class_hash` at construction time and provides no function to update it. When the admin upgrades existing deployed tokens via `upgrade_token()` to a new `ClassHash`, the bridge's stored `bridge_token_class_hash` remains pointing to the old class. Every subsequent `deploy_token()` call deploys new bridged tokens using the stale, pre-upgrade class hash. This is the direct analog of the vault bytecode mismatch: the class hash used for new token deployments diverges from the reviewed/patched class, meaning new tokens do not receive the security guarantees of the updated `BridgeToken` implementation.

### Finding Description
`OmniBridge` stores `bridge_token_class_hash: ClassHash` in contract storage, written once in the constructor:

```cairo
self.bridge_token_class_hash.write(token_class_hash);
```

The public interface exposes no setter for this field. The `upgrade_token()` function upgrades individual already-deployed tokens to a caller-supplied `new_class_hash`, but it does not update `bridge_token_class_hash`. The `upgrade()` function upgrades the bridge contract itself but also does not touch `bridge_token_class_hash`. Every call to `deploy_token()` reads the stale stored value:

```cairo
let (contract_address, _) = deploy_syscall(
    self.bridge_token_class_hash.read(), salt, constructor_calldata.span(), false,
)
.unwrap_syscall();
```

After any `upgrade_token()` call that migrates existing tokens to a patched class, all future `deploy_token()` calls will instantiate the old, pre-patch class. The EVM `OmniBridge` has the same structural flaw: `tokenImplementationAddress` is set in `initialize()` and there is no `setTokenImplementationAddress()` function, so new ERC1967 proxies always point to the original implementation even after `upgradeToken()` has been used to migrate existing tokens.

### Impact Explanation
`deploy_token()` is callable by any submitter who presents a valid NEAR-derived signature. Every new bridged token deployed after a `BridgeToken` class upgrade will be instantiated from the old class. If the old class contained a vulnerability that the upgrade was intended to fix (e.g., an unauthorized `mint` path, missing access control on `burn`, or a reentrancy vector), every newly deployed token inherits that vulnerability. An attacker can exploit the vulnerability in any such newly deployed token to mint unbacked supply or drain locked collateral, breaking bridge collateralization. This maps to the allowed High impact: "Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."

### Likelihood Explanation
The bridge is explicitly designed to be upgradeable: `upgrade_token()` exists precisely to patch deployed tokens. Any security fix to `BridgeToken` will be applied to existing tokens via `upgrade_token()`, but the stored `bridge_token_class_hash` will silently remain stale. The next `deploy_token()` call — which any relayer or user with a valid NEAR signature can trigger — will instantiate the unpatched class. This is a realistic, near-certain operational scenario whenever a BridgeToken security patch is deployed.

### Recommendation
- **Short term:** Add a `set_bridge_token_class_hash(new_class_hash: ClassHash)` function restricted to `DEFAULT_ADMIN_ROLE` so the stored class hash can be updated atomically when a new `BridgeToken` class is declared. Similarly, add `setTokenImplementationAddress()` to the EVM `OmniBridge`.
- **Long term:** Document the required operational checklist: whenever `BridgeToken` is upgraded, `bridge_token_class_hash` (Starknet) and `tokenImplementationAddress` (EVM) must be updated before any new `deploy_token()` call is processed. Consider emitting an event when the class hash diverges from the last-upgraded token class to make staleness observable on-chain.

### Proof of Concept

1. Admin declares a patched `BridgeToken` class on Starknet (e.g., fixing an unauthorized-mint bug). Class hash = `NEW_HASH`.
2. Admin calls `upgrade_token(existing_token_address, NEW_HASH)` for all currently deployed tokens. They are now running the patched class.
3. `bridge_token_class_hash` in storage still holds `OLD_HASH` — no function exists to change it.
4. A relayer submits a valid `deploy_token()` call for a new NEAR token. The bridge executes:
   ```cairo
   deploy_syscall(self.bridge_token_class_hash.read(), ...)  // deploys OLD_HASH
   ```
5. The newly deployed token runs the pre-patch class. If the old class had an unauthorized-mint vulnerability, an attacker calls the vulnerable entry point on the new token, mints unbacked supply, and bridges it back to NEAR, draining collateral. build:
	scarb build

	scarb test

declare-token:
	@mkdir -p $(MAKEFILE_DIR)/target
	sncast declare \
		--contract-name BridgeToken \
		--network $(NETWORK) \
		2>&1 | tee /dev/stderr \
		| grep -oE 'class.hash[: ]+0x[0-9a-fA-F]+' | grep -oE '0x[0-9a-fA-F]+' \
		> $(BRIDGE_TOKEN_CLASS_HASH_FILE)
	@echo "BridgeToken class hash saved to $(BRIDGE_TOKEN_CLASS_HASH_FILE)"

declare-bridge:
	sncast declare \
		--contract-name OmniBridge \
		--network $(NETWORK)

deploy-bridge:
	@class_hash=$$(cat $(BRIDGE_TOKEN_CLASS_HASH_FILE)); \
	set -x; \
	sncast deploy \
		--contract-name OmniBridge \
		--network $(NETWORK) \
		--arguments "$(OMNI_BRIDGE_DERIVED_ADDRESS), $(OMNI_BRIDGE_CHAIN_ID), $$class_hash, $(DEFAULT_ADMIN), $(STRK_TOKEN_ADDRESS)"
