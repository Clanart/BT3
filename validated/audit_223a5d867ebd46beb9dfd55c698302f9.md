### Title
`ProofOfInclusion::valid()` Does Not Verify Against a Trusted Root — Forged DataLayer Inclusion Proofs Pass Validation - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only checks internal self-consistency of the proof structure. Its final comparison is a tautology: it compares `existing_hash` against `self.root_hash()`, but `root_hash()` returns `self.layers.last().combined_hash`, which is exactly the value `existing_hash` was assigned in the last loop iteration. The method never accepts a trusted external root and cannot bind the proof to any particular committed tree state. Because `ProofOfInclusion` is `Streamable` and fully exposed through Python bindings, an unprivileged attacker can craft arbitrary bytes that deserialize to a forged proof that passes `valid()` while proving inclusion of data that is not in the actual DataLayer tree.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← tautology
}
``` [1](#0-0) 

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same value as existing_hash after the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which the loop body already asserted equals `layer.combined_hash`. `root_hash()` returns that same `combined_hash`. Therefore `existing_hash == self.root_hash()` is unconditionally `true` whenever the loop body does not short-circuit. The final guard provides zero additional security.

The method checks only that each layer's `combined_hash` is arithmetically consistent with the previous hash and the sibling hash — i.e., the proof is internally self-consistent. It does **not** check that the computed root matches any externally trusted, committed tree root. An attacker can freely choose any `node_hash` and construct a chain of consistent `ProofOfInclusionLayer` values leading to any root they wish; `valid()` will return `true`.

`ProofOfInclusion` is a `Streamable` type with full Python-binding exposure: [3](#0-2) [4](#0-3) 

This means any Python caller can receive bytes from an untrusted source, call `ProofOfInclusion.from_bytes()`, and then call `.valid()` — and the result will be `True` for any internally consistent forged proof.

### Impact Explanation

An attacker who can deliver a serialized `ProofOfInclusion` to a DataLayer node or client can prove inclusion of arbitrary key-value pairs that do not exist in the actual committed tree. Because `valid()` is the only verification method on the type and its name implies complete validation, callers will trust its result. This allows untrusted input to prove invalid DataLayer state, matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` network/storage object. Any DataLayer peer exchange, RPC response, or persisted blob that carries a `ProofOfInclusion` is an attacker-controlled entry point. The method name `valid()` strongly implies complete proof verification; callers have no reason to additionally check `proof.root_hash()` against a separately trusted root. The fuzz target itself only calls `proof.valid()` without an external root check, demonstrating the expected usage pattern. [5](#0-4) 

### Recommendation

`valid()` must accept a trusted root parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against externally supplied root
}
```

The existing `valid()` method (which only checks internal consistency) should either be removed or clearly renamed to `is_internally_consistent()` with documentation that it does not constitute proof verification. All callers — including the Python binding and the fuzz target — must be updated to supply the trusted root obtained from a locally committed `MerkleBlob::get_root_hash()`.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer

# Attacker-chosen leaf hash (not in any real tree)
fake_leaf = bytes([0xAA] * 32)
# Attacker-chosen sibling hash
fake_sibling = bytes([0xBB] * 32)

# Compute what combined_hash must be for internal consistency
import hashlib
# (simplified; use actual calculate_internal_hash logic)
combined = hashlib.sha256(fake_leaf + fake_sibling).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=1,          # sibling is on the right
    other_hash=fake_sibling,
    combined_hash=combined,     # attacker sets this to match
)

forged_proof = ProofOfInclusion(node_hash=fake_leaf, layers=[layer])

# valid() returns True for a completely fabricated proof
assert forged_proof.valid()   # passes — no trusted root is checked
print("Forged proof accepted:", forged_proof.root_hash().hex())
```

The forged proof passes `valid()` and claims `fake_leaf` is included in a tree whose root is `combined`. No actual `MerkleBlob` containing `fake_leaf` exists.
