Audit Report

## Title
Missing Factory Pool Validation in `metricOmmModifyLiquidityCallback` Allows Malicious Pool to Drain User Tokens - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
`MetricOmmPoolLiquidityAdder` stores a caller-supplied pool address in transient context and later validates only that `msg.sender` matches that stored address during the modify-liquidity callback. Because no factory registration check is performed, an attacker can deploy a fake pool, trick a victim into calling `addLiquidityExactShares` or `addLiquidityWeighted` with that address, and drain any ERC-20 token the victim has approved to the adder, up to the caller-supplied `maxAmountToken0`/`maxAmountToken1` caps.

## Finding Description
`_addLiquidity` stores the caller-supplied `pool` in transient context with no factory check:

```solidity
// MetricOmmPoolLiquidityAdder.sol L193–195
_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
try IMetricOmmPoolActions(pool)
    .addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData)
```

The callback then validates only `msg.sender == expectedPool`:

```solidity
// L162–167
(address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
if (expectedPool == address(0)) revert CallbackContextNotActive();
if (msg.sender != expectedPool) revert InvalidCallbackCaller(msg.sender, expectedPool);
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
```

After passing, token addresses are read from the caller itself:

```solidity
// L169–177
PoolImmutables memory imm = IMetricOmmPool(msg.sender).getImmutables();
address token0 = imm.token0;
address token1 = imm.token1;
if (amount0Delta > 0) { pay(token0, payer, msg.sender, amount0Delta); }
if (amount1Delta > 0) { pay(token1, payer, msg.sender, amount1Delta); }
```

`pay` resolves to:

```solidity
// PeripheryPayments.sol L85–87
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value);
}
```

A malicious pool controls `token0`/`token1` returned by `getImmutables()`, sets `amount0Delta = max0`, and receives `safeTransferFrom(victim, fakePool, max0)` of any token it names. The contract has no `FACTORY` immutable at all — confirmed by the absence of any `FACTORY` or `isPool` reference in the file. The correct pattern exists in `MetricOmmSwapRouterBase._requireExpectedCallbackCaller` (lines 82–85) but is absent here.

## Impact Explanation
A victim who has approved `MetricOmmPoolLiquidityAdder` (required for any liquidity operation) and who calls either public entry point with an attacker-supplied pool address suffers direct, irreversible loss of ERC-20 principal up to the `maxAmountToken0`/`maxAmountToken1` values they supply. The attacker receives the tokens directly. This is a direct loss of user principal, meeting the Critical/High allowed impact gate.

## Likelihood Explanation
The attack requires the victim to call the adder with a malicious pool address. This is achievable via a compromised frontend (which constructs the calldata), a phishing UI, or a malicious integration contract that passes an unverified pool address. Any user who has already approved the adder — a prerequisite for normal use — is at risk until they revoke the approval. The NatDoc at lines 19–21 explicitly acknowledges the gap, confirming the developers were aware; the omission of a `FACTORY` reference entirely (unlike every other router contract) makes this a persistent, non-self-correcting exposure.

## Recommendation
Add a factory pool check inside `_addLiquidity` before storing the pool in transient context, mirroring `MetricOmmSwapRouterBase._requireFactoryPool`:

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool); // add this
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires adding an `IMetricOmmPoolFactory internal immutable FACTORY` to `MetricOmmPoolLiquidityAdder` and passing the factory address in the constructor, consistent with `MetricOmmSwapRouterBase`.

## Proof of Concept
1. Deploy `MaliciousPool` with `getImmutables()` returning `token0 = victimToken` and `addLiquidity()` calling back `metricOmmModifyLiquidityCallback(max0, 0, abi.encode(KIND_PAY))`.
2. Victim approves `MetricOmmPoolLiquidityAdder` for `victimToken` (normal prerequisite).
3. Victim (or attacker on victim's behalf via phishing) calls `addLiquidityExactShares(address(fakePool), victim, 0, delta, 1000e18, 0, "")`.
4. `_addLiquidity` sets transient context with `fakePool` as expected pool and calls `fakePool.addLiquidity(...)`.
5. `fakePool.addLiquidity` calls back `metricOmmModifyLiquidityCallback(1000e18, 0, abi.encode(KIND_PAY))`.
6. Callback passes `msg.sender == expectedPool` check; reads `token0 = victimToken` from `fakePool.getImmutables()`; executes `safeTransferFrom(victim, fakePool, 1000e18)`.
7. Victim loses 1000e18 of `victimToken`; attacker's pool receives it.
