### Title
Missing Duplicate Key Validation in `batch_insert` Corrupts MerkleBlob Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` applies duplicate-key and duplicate-hash guards only to the last two items in the input batch (via the single-item `insert` path). All remaining items are written directly into the blob with `insert_entry_to_blob`, bypassing the `block_status_cache` existence checks entirely. Supplying a batch that contains a repeated `KeyId` — or a key already present in the tree — silently inserts a second leaf node for that key, corrupting the tree structure, producing a wrong root hash, and enabling forged inclusion/exclusion proofs.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) correctly guards against duplicates:

```rust
// blob.rs lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `self.insert()` (with its guards) only for the last two items in the vector, and only when the tree currently has ≤ 1 leaf:

```rust
// blob.rs lines 578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else {
            return Ok(());
        };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

Every other item in the batch is written directly via `insert_entry_to_blob` with no existence check:

```rust
// blob.rs lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

When a duplicate `KeyId` appears in the batch (or matches a key already in the tree), two leaf nodes with the same key are written into the blob. The `block_status_cache` key-to-index map is overwritten for the second occurrence, making the first leaf permanently unreachable. The internal-node hash chain built over these leaves produces an incorrect root hash. `check_integrity` will subsequently detect the inconsistency, but it is not called automatically after `batch_insert`.

The Python binding `py_batch_insert` exposes this path directly to callers with arbitrary `keys_values` and `hashes` vectors:

```rust
// blob.rs lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

The single-insert path correctly rejects duplicates and is tested: [5](#0-4) 

But no equivalent test or guard exists for the bulk path.

---

### Impact Explanation

A corrupted Merkle tree root means:

- **Forged inclusion proofs**: a leaf that was never legitimately inserted can be "proven" present if the root is wrong.
- **Forged exclusion proofs**: a leaf that is present may appear absent.
- **Persistent state corruption**: once `batch_insert` returns `Ok(())` with a duplicate, the blob is silently invalid; all subsequent proofs derived from it are untrustworthy.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`py_batch_insert` is a public Python API. Any DataLayer client that assembles a batch from user-supplied or network-supplied key/value pairs without pre-deduplicating them can trigger this path. The attacker does not need any privileged role — they only need to supply a list with a repeated key. The single-insert path's `KeyAlreadyPresent` error gives a false sense of safety that the bulk path shares the same protection.

---

### Recommendation

Add a duplicate check at the start of the bulk loop in `batch_insert`, mirroring the single-insert guards:

```rust
for ((key, value), hash) in keys_values_hashes {
+   if self.block_status_cache.contains_key(key) {
+       return Err(Error::KeyAlreadyPresent());
+   }
+   if self.block_status_cache.contains_leaf_hash(hash) {
+       return Err(Error::HashAlreadyPresent());
+   }
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
+   // update block_status_cache for key and hash here, as insert() does
    indexes.push(new_leaf_index);
}
```

Alternatively, pre-deduplicate the input vector before processing, and verify no key in the batch already exists in the tree.

### Proof of Concept

```rust
#[test]
fn test_batch_insert_duplicate_key_corrupts_tree() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so batch_insert skips the guarded path
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
    }

    // Batch contains a duplicate key (KeyId(0) already in tree)
    let batch = vec![
        ((KeyId(10), ValueId(10)), sha256_num(&10i64)),
        ((KeyId(11), ValueId(11)), sha256_num(&11i64)),
        ((KeyId(0),  ValueId(99)), sha256_num(&99i64)),  // duplicate!
    ];

    // Expected: Err(KeyAlreadyPresent). Actual: Ok(()) — tree is now corrupt.
    blob.batch_insert(batch).unwrap();

    // check_integrity reveals the corruption
    blob.check_integrity().expect_err("tree should be corrupt");
}
