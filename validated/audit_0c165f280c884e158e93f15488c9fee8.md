### Title
Cross-Chain Metadata Signature Replay: EVM `deployToken` Signature Accepted by Solana `deploy_token` Due to Identical Chain-Agnostic Borsh Encoding â€” (`solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

### Summary

`DeployTokenPayload::serialize_for_near` on Solana produces a byte-for-byte identical encoding to the one verified by `OmniBridge.deployToken` on EVM. Neither encoding includes a destination chain ID. Because both chains verify against the same MPC key using raw keccak256 + secp256k1, a valid MPC signature obtained from an EVM `deployToken` call is unconditionally accepted by Solana's `deploy_token`, and vice versa.

---

### Finding Description

**EVM encoding** (`OmniBridge.sol:142-148`):

```
bytes1(PayloadType.Metadata=1)
|| Borsh.encodeString(token)    // u32_LE(len) || UTF-8
|| Borsh.encodeString(name)
|| Borsh.encodeString(symbol)
|| bytes1(decimals)
```

`keccak256(borshEncoded)` is verified via `ECDSA.recover(...) == nearBridgeDerivedAddress` (raw hash, no Ethereum prefix). [1](#0-0) 

**Solana encoding** (`deploy_token.rs:19-26`):

```
IncomingMessageType::Metadata  // Borsh enum discriminant = 1
|| borsh(token)                // u32_LE(len) || UTF-8
|| borsh(name)
|| borsh(symbol)
|| borsh(decimals)             // u8 = single byte
```

`keccak::hash(serialized)` is verified via `secp256k1_recover(...) == derived_near_bridge_address` (raw hash, no prefix). [2](#0-1) [3](#0-2) 

**Byte-for-byte identity proof:**

| Field | EVM | Solana |
|---|---|---|
| Type byte | `uint8(PayloadType.Metadata)` = `0x01` | `IncomingMessageType::Metadata` (index 1) = `0x01` |
| String encoding | `encodeUint32(len) \|\| bytes` (LE swap) | Borsh `String`: `u32_LE(len) \|\| bytes` |
| Decimals | `bytes1(decimals)` | Borsh `u8` |

`IncomingMessageType` has `{InitTransfer=0, Metadata=1}`.
Both produce `0x01`. [4](#0-3) [5](#0-4) 

`Borsh.encodeString` on EVM calls `encodeUint32` which byte-swaps to little-endian â€” identical to Borsh's native `u32` LE encoding used on Solana. [6](#0-5) 

Both chains use the same MPC public key (`nearBridgeDerivedAddress` / `derived_near_bridge_address`) and raw keccak256 without any chain-specific domain separator or prefix. [7](#0-6) [8](#0-7) 

There is no nonce or replay guard on `deploy_token` (unlike `finalize_transfer` which has `destination_nonce`). [9](#0-8)

### Impact Explanation

An attacker holding a valid MPC signature from an EVM `deployToken` call for token T (name N, symbol S, decimals D) can:

1. Submit the identical 65-byte signature to Solana `deploy_token` with the same `DeployTokenPayload`.
2. `verify_signature` passes because the keccak hash is identical and the recovered key equals `derived_near_bridge_address`.
3. A canonical wrapped SPL mint is created for token T under the bridge's authority PDA.
4. Solana posts a Wormhole `DeployTokenResponse` back to NEAR containing the new mint address.
5. NEAR processes this Wormhole message and registers the Solana mint as the canonical wrapped token for T on Solana. [11](#0-10) 

Once the unauthorized mint is registered on NEAR, legitimate `finalize_transfer` calls routed to Solana for token T will mint against this attacker-triggered mint. The attacker did not need any Solana-specific MPC authorization â€” the MPC never signed for Solana deployment.

---

### Likelihood Explanation

The attack requires only:
- A publicly observable EVM `deployToken` transaction (the signature is in calldata, fully public).
- The ability to call Solana `deploy_token` (permissionless, any relayer).

No privileged access, no key compromise, no colluding MPC signers. The signature is already on-chain on EVM. Any observer can extract it and replay it on Solana before or after the legitimate Solana deployment occurs.

---

### Recommendation

Include a destination chain ID in the signed payload for `DeployTokenPayload`. For example, prepend `SOLANA_OMNI_BRIDGE_CHAIN_ID` to the serialized bytes in `serialize_for_near`:

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?; // <-- add chain ID
    self.serialize(&mut writer)?;
    writer.into_inner().map_err(|_| error!(ErrorCode::InvalidArgs))
}
```

Apply the same fix symmetrically on EVM (`OmniBridge.deployToken`) and StarkNet (`MetadataPayload::to_borsh`), and update the MPC signing logic on NEAR to include the destination chain ID in the payload it signs. [2](#0-1)

    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
