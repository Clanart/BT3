### Title
Missing Bounds Check Before Felt Underflow in `compute_deploy_account_transaction_hash` Enables Network Halt - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_deploy_account_transaction_hash` performs `calldata_size - 2` and dereferences `&calldata[2]` without first asserting `calldata_size >= 2`. In Cairo's felt arithmetic, if a user submits a `DeployAccount` transaction with `calldata_size = 0` or `calldata_size = 1`, the subtraction wraps around to `P - 2` (≈ 2²⁵¹), which is passed as the `data_length` argument to `poseidon_hash_update_with_nested_hash`. This causes the Cairo VM to attempt reading a near-infinite number of uninitialized memory cells, crashing OS-level block execution and preventing valid proof generation.

---

### Finding Description

In `compute_deploy_account_transaction_hash` (line 254):

```cairo
poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
poseidon_hash_update(data_ptr=calldata, data_length=2);
```

The function assumes the DeployAccount calldata layout is:
- `calldata[0]` = class hash
- `calldata[1]` = contract address salt
- `calldata[2..]` = constructor calldata

There is **no assertion** that `calldata_size >= 2` before the subtraction. In Cairo, all arithmetic is over the prime field `F_P` (P ≈ 2²⁵¹ + 17·2¹⁹² + 1). When `calldata_size = 0`:

```
calldata_size - 2  ≡  P - 2  (mod P)
```

This enormous felt value is passed as `data_length` to `poseidon_hash_update_with_nested_hash`, which iterates over that many memory cells. Since `calldata[2]` is uninitialized (the array has 0 elements), the Cairo VM immediately encounters an uninitialized memory read, causing a fatal VM-level error — not a graceful transaction revert.

The analogous bug class to the reported `Buffer` library issue is identical: **missing bounds check before a length-driven memory traversal**, where an underflowing length causes out-of-bounds memory access. [1](#0-0) 

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

A fatal error inside the StarkNet OS Cairo program (as opposed to inside a contract) is not recoverable as a transaction revert. The OS processes all transactions in a block atomically to produce a single proof. If the OS VM crashes mid-block due to an uninitialized memory read triggered by a malformed `DeployAccount` transaction, the sequencer cannot finalize the block or generate a valid STARK proof. The block must be abandoned. If the sequencer retries with the same mempool contents (which still contains the malicious transaction), the cycle repeats, halting block production.

---

### Likelihood Explanation

**Medium.** Any unprivileged user can submit a `DeployAccount` transaction. The transaction format is public and user-controlled. Setting `calldata_size = 0` (or `1`) is trivially achievable via the public RPC/gateway. No privileged access, leaked key, or operator cooperation is required. The only mitigation would be off-chain gateway validation rejecting such transactions before they reach the OS — but the OS itself is the authoritative validator and must not rely on upstream filtering for soundness.

---

### Recommendation

Add an explicit range check before the subtraction and pointer offset:

```cairo
// Assert calldata contains at least class_hash and salt.
assert_nn_le(2, calldata_size);  // calldata_size >= 2
poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
poseidon_hash_update(data_ptr=calldata, data_length=2);
```

This mirrors the pattern already used elsewhere in the OS, e.g., `assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1)` in `pack_resource_bounds`. [2](#0-1) 

---

### Proof of Concept

1. Craft a `DeployAccount` transaction (v3) with `calldata_size = 0` and an empty `calldata` array.
2. Submit it via the public StarkNet gateway/RPC endpoint.
3. The sequencer includes it in a block and runs the OS Cairo program.
4. The OS calls `compute_deploy_account_transaction_hash` with `calldata_size=0, calldata=[]`.
5. Line 254 evaluates: `data_length = 0 - 2 = P - 2`; `data_ptr = &calldata[2]` (uninitialized).
6. `poseidon_hash_update_with_nested_hash` attempts to read `calldata[2]` — uninitialized memory.
7. The Cairo VM raises a fatal memory error; the OS cannot complete block execution.
8. No valid STARK proof can be generated; the block is abandoned.
9. Repeat from step 2 — the network cannot advance. [1](#0-0)

func compute_deploy_account_transaction_hash{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    common_fields: CommonTxFields*, calldata_size: felt, calldata: felt*
) -> felt {
    alloc_locals;

    with_attr error_message("Invalid transaction version: {version}.") {
        assert common_fields.version = 3;
    }

    let hash_state: PoseidonHashState = poseidon_hash_init();
    with hash_state {
        hash_tx_common_fields(common_fields=common_fields);
        // Hash and add the constructor calldata to the hash state.
        poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
        // Add the class hash and the contract address salt to the hash state.
        poseidon_hash_update(data_ptr=calldata, data_length=2);
    }

    let transaction_hash = poseidon_hash_finalize(hash_state=hash_state);
    return transaction_hash;
}
