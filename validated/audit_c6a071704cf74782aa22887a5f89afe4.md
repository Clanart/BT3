### Title
Unregistered Token in `init_transfer` Causes Permanent Fund Lock with No Recovery Path — (`starknet/src/omni_bridge.cairo`)

### Summary

`init_transfer` on StarkNet accepts any ERC20 `token_address` without verifying it is registered in the bridge's token mappings. Tokens are locked (via `transfer_from`) into the bridge contract and an `InitTransfer` event is emitted. When a relayer submits the resulting proof to NEAR's `fin_transfer_callback`, the callback panics at `TokenDecimalsNotFound` because the token has no registered decimals entry. No refund path exists on the StarkNet side, so the user's funds are permanently locked.

---

### Finding Description

**Step 1 — Missing token registration guard in `init_transfer`**

`init_transfer` performs only three checks before accepting tokens: [1](#0-0) 

```cairo
assert(!_is_paused(@self, PAUSE_INIT_TRANSFER), 'ERR_INIT_TRANSFER_PAUSED');
assert(amount > 0, 'ERR_ZERO_AMOUNT');
assert(fee < amount, 'ERR_INVALID_FEE');
...
if self.is_bridge_token(token_address) {
    IBridgeTokenDispatcher { contract_address: token_address }.burn(caller, amount.into());
} else {
    let success = IERC20Dispatcher { contract_address: token_address }
        .transfer_from(caller, get_contract_address(), amount.into());
    assert(success, 'ERR_TRANSFER_FROM_FAILED');
}
```

There is no check that `token_address` is registered in `starknet_to_near_token` or that NEAR has a corresponding `token_decimals` entry. `is_bridge_token` only distinguishes burn-path from lock-path; it does not gate entry: [2](#0-1) 

Any ERC20 that returns `true` from `transfer_from` will succeed, lock tokens in the bridge, and emit `InitTransfer`.

**Step 2 — NEAR `fin_transfer_callback` panics on unregistered token**

When a relayer submits the proof, `fin_transfer_callback` first validates the factory (passes, because the emitter is the legitimate StarkNet bridge contract), then immediately panics if the token has no registered decimals: [3](#0-2) 

```rust
require!(
    self.factories.get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);

let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);  // ← panics here
```

The NEAR transaction reverts entirely. No transfer record is created on NEAR.

**Step 3 — No refund path on StarkNet**

The StarkNet contract has no function to release tokens locked by a failed `init_transfer`. There is no callback from NEAR back to StarkNet, no admin rescue function, and no user-callable refund. The `InitTransfer` event's `origin_nonce` is consumed and cannot be reused. The tokens are permanently locked in the bridge contract.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

Any user who calls `init_transfer` with a valid ERC20 that has no corresponding NEAR `token_decimals` entry loses their tokens permanently. The StarkNet bridge holds the tokens with no protocol-level mechanism to return them. Admin intervention (registering decimals on NEAR post-hoc and having a relayer resubmit) is the only theoretical recovery, but it is not guaranteed and is not part of the protocol design.

---

### Likelihood Explanation

**Medium.** The scenario requires a user to call `init_transfer` with a token that is not registered on NEAR. This can happen:
- Accidentally, when a user bridges a token before the `deploy_token` / `log_metadata` flow has been completed on NEAR.
- Intentionally by an attacker who wants to grief themselves or demonstrate the missing guard.

The `fin_transfer` call on NEAR is `#[trusted_relayer]`-gated, so a careful relayer might not submit the proof. However, this does not help the user: whether the relayer submits (NEAR panics) or does not submit (tokens stuck indefinitely), the outcome is the same permanent lock. The missing guard is the root cause in both branches.

---

### Recommendation

Add a whitelist check in `init_transfer` that requires `token_address` to be either a bridge token (already in `starknet_to_near_token`) or an explicitly allowlisted native token:

```cairo
assert(
    self.is_bridge_token(token_address) || self.is_allowed_native_token(token_address),
    'ERR_TOKEN_NOT_REGISTERED'
);
```

Alternatively, require that `starknet_to_near_token.read(token_address).len() > 0` for all tokens before accepting the transfer, rejecting any token that has not been registered via the `deploy_token` flow.

---

### Proof of Concept

1. Deploy any ERC20 on StarkNet that is **not** registered via `deploy_token` (i.e., absent from `starknet_to_near_token`).
2. Approve the StarkNet bridge to spend tokens.
3. Call `init_transfer(unregistered_erc20, amount, fee, 0, "victim.near", "")`.
   - `is_bridge_token` returns `false` → `transfer_from` executes → tokens locked → `InitTransfer` event emitted.
4. Submit the resulting StarkNet proof to NEAR's `fin_transfer`.
5. Observe `fin_transfer_callback` panic: `ERR_TOKEN_DECIMALS_NOT_FOUND` at `near/omni-bridge/src/lib.rs:718`.
6. Confirm: NEAR transaction reverted, no transfer stored, tokens remain locked in StarkNet bridge with no recovery path. [4](#0-3) [5](#0-4)

            let origin_nonce = self.current_origin_nonce.read() + 1;
            self.current_origin_nonce.write(origin_nonce);

            let caller = get_caller_address();

            if self.is_bridge_token(token_address) {
                IBridgeTokenDispatcher { contract_address: token_address }
                    .burn(caller, amount.into());

                let native_token = self.strk_token_address.read();
                let success = IERC20Dispatcher { contract_address: native_token }

                assert(success, 'ERR_FEE_TRANSFER_FAILED');
