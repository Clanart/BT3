### Title
`CheckEthGasConsume` Overflow Guard Broken by uint64 Underflow — Block Gas Limit Bypass - (`ante/eth.go`)

---

### Summary

The overflow guard in `CheckEthGasConsume` uses the expression `math.MaxInt64 - gasLimit` where `gasLimit` is `uint64`. When `gasLimit > math.MaxInt64`, this subtraction itself **underflows as uint64**, making the guard permanently false. An attacker can submit a batch of two `MsgEthereumTx` messages each with `gasLimit = math.MaxUint64/2 + 1`, causing `gasWanted` to wrap to `0` after accumulation, bypassing the block gas limit check entirely.

---

### Finding Description

The guard in `CheckEthGasConsume` is:

```go
// ante/eth.go lines 151-155
gasLimit := msgEthTx.GetGas()          // uint64
if gasWanted > math.MaxInt64-gasLimit { // BROKEN
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow", gasWanted, gasLimit)
}
gasWanted += gasLimit
``` [1](#0-0) 

In Go, `math.MaxInt64` is an untyped constant. In the expression `math.MaxInt64 - gasLimit` where `gasLimit` is `uint64`, the constant is implicitly converted to `uint64`. When `gasLimit > uint64(math.MaxInt64)` (i.e., the high bit is set), the subtraction **underflows**:

```
uint64(9223372036854775807) - uint64(9223372036854775808) = 18446744073709551615  (math.MaxUint64)
```

So `gasWanted > math.MaxUint64` is always `false` — the guard never fires for any `gasLimit` with the high bit set.

**Overflow trace with two messages, each `gasLimit = math.MaxUint64/2 + 1 = 9223372036854775808`:**

| Iteration | Check `gasWanted > MaxInt64 - gasLimit` | Result | `gasWanted` after `+=` |
|---|---|---|---|
| msg 1 | `0 > 18446744073709551615` | false (no error) | `9223372036854775808` |
| msg 2 | `9223372036854775808 > 18446744073709551615` | false (no error) | `0` (uint64 wrap) |

`gasWanted` wraps to `0`. The subsequent block gas limit check:

```go
// ante/eth.go lines 156-163
if gasWanted > blockGasLimit {
    return ctx, errorsmod.Wrapf(errortypes.ErrOutOfGas, ...)
}
``` [2](#0-1) 

…passes trivially (`0 > blockGasLimit` is false). The context is then set with a gas meter whose limit is `0`:

```go
// ante/eth.go lines 196-198
newCtx := ctx.
    WithGasMeter(ethermint.NewInfiniteGasMeterWithLimit(gasWanted)). // limit = 0
    WithPriority(minPriority)
``` [3](#0-2) 

The `infiniteGasMeterWithLimit` with `limit=0` is still fully infinite — `IsPastLimit()` always returns `false`, `IsOutOfGas()` always returns `false`, and `GasRemaining()` always returns `math.MaxUint64` — so execution is not blocked by the meter: [4](#0-3) 

---

### Impact Explanation

An attacker can submit a Cosmos tx wrapping two `MsgEthereumTx` messages each with `gasLimit = math.MaxUint64/2 + 1`. The accumulated `gasWanted` wraps to `0`, bypassing the block gas limit check. The tx is accepted and the EVM executes both messages with their actual (enormous) gas limits, while the Cosmos layer records `gasWanted = 0` — meaning the tx consumes **zero** of the block's gas budget. This is a block gas limit bypass that permits invalid transactions to commit EVM state, matching the High impact category: *"ante handler bug that permits invalid transactions to commit."*

---

### Likelihood Explanation

**Conditional on fee cost.** The fee check at line 170 still runs per-message:

```go
fees, err := keeper.VerifyFee(msgEthTx, evmDenom, baseFee, rules, ctx.IsCheckTx())
``` [5](#0-4) 

With `gasLimit ≈ 9.2 × 10^18` and any non-zero `effectiveGasPrice`, the required fee is astronomically large. However:
- If `baseFee = 0` (fee market disabled, or chain configured with zero min gas price) **and** the attacker sets `gasPrice = 0`, fees are `0` regardless of `gasLimit`. The attack is then fully free and unrestricted.
- Chains with fee market enabled and `baseFee > 0` are protected by the fee barrier in practice.

---

### Recommendation

Replace the broken guard with a proper pre-addition overflow check that does not itself underflow:

```go
// Correct: check addition overflow without subtracting from MaxInt64
if gasLimit > math.MaxInt64 || gasWanted > math.MaxInt64-gasLimit {
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow int64", gasWanted, gasLimit)
}
```

Or use the existing `addUint64Overflow` helper already present in `types/gasmeter.go`:

```go
var overflow bool
gasWanted, overflow = addUint64Overflow(gasWanted, gasLimit)
if overflow || gasWanted > uint64(math.MaxInt64) {
    return ctx, fmt.Errorf("gasWanted overflow")
}
``` [6](#0-5)

### Proof of Concept

```go
// Two MsgEthereumTx, each with Gas = math.MaxUint64/2 + 1
gasLimit := uint64(math.MaxUint64/2 + 1) // = 9223372036854775808

// Iteration 1: gasWanted = 0
guard1 := uint64(0) > uint64(math.MaxInt64)-gasLimit
// uint64(math.MaxInt64) - gasLimit = 9223372036854775807 - 9223372036854775808 = 18446744073709551615
// guard1 = false → no error
gasWanted := uint64(0) + gasLimit // = 9223372036854775808

// Iteration 2: gasWanted = 9223372036854775808
guard2 := gasWanted > uint64(math.MaxInt64)-gasLimit
// = 9223372036854775808 > 18446744073709551615 = false → no error
gasWanted += gasLimit // = 9223372036854775808 + 9223372036854775808 = 0 (uint64 wrap)

// gasWanted = 0 → passes blockGasLimit check
// Gas meter set with limit 0 → infinite meter, execution proceeds
```

The overflow check at line 152 is the root cause: it uses `math.MaxInt64 - gasLimit` as a `uint64` subtraction, which underflows for any `gasLimit > math.MaxInt64`, making the guard permanently ineffective for the exact values needed to trigger the wrap. [7](#0-6)
