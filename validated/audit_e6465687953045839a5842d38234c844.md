### Title
Missing Cross-Category Class Hash Exclusivity Check in P2P State Diff Validation Allows Corrupted State Diff Storage - (File: crates/apollo_p2p_sync/src/client/state_diff.rs)

### Summary

The P2P sync state diff validation checks for intra-category duplicates (duplicate `DeclaredClass` or duplicate `DeprecatedDeclaredClass` chunks) but never checks that the same class hash cannot appear in **both** `class_hash_to_compiled_class_hash` (Sierra/Cairo 1) and `deprecated_declared_classes` (Cairo 0) simultaneously. A malicious peer can exploit this to inject a `ThinStateDiff` that violates the stated invariant, producing a `StateDiffCommitment` that diverges from the one committed in the block header, and causing `starknet_getStateUpdate` to return an authoritative-looking wrong value.

### Finding Description

`starknet_api/src/state.rs` carries an explicit, unenforced invariant comment on `StateDiff`:

```
// Invariant: Class hashes of declared_classes and deprecated_declared_classes are exclusive.
// TODO(yair): Enforce this invariant.
``` [1](#0-0) 

The P2P sync path in `unite_state_diffs` handles the three chunk types as follows:

- **`DeclaredClass`**: inserts into `class_hash_to_compiled_class_hash`; returns `ConflictingStateDiffParts` if the key already exists (intra-category duplicate check).
- **`DeprecatedDeclaredClass`**: unconditionally pushes to `deprecated_declared_classes` with no cross-category check. [2](#0-1) 

After all chunks are collected, `validate_deprecated_declared_classes_non_conflicting` only checks for duplicates **within** `deprecated_declared_classes`: [3](#0-2) 

No check verifies that a class hash present in `class_hash_to_compiled_class_hash` is absent from `deprecated_declared_classes`. After validation, the state diff is written directly to storage without computing `calculate_state_diff_hash` and comparing it against the `state_diff_commitment` stored in the block header: [4](#0-3) 

The `calculate_state_diff_hash` function chains both categories independently, so a class hash `X` present in both categories is counted twice — once in `chain_declared_classes` and once in `chain_deprecated_declared_classes` — producing a `StateDiffCommitment` that differs from the legitimate one committed in the block header: [5](#0-4) 

Similarly, `ThinStateDiff::len()` counts both `class_hash_to_compiled_class_hash.len()` and `deprecated_declared_classes.len()` independently, so `X` inflates the length by 1 extra unit: [6](#0-5) 

### Impact Explanation

The corrupted `ThinStateDiff` is persisted to the MDBX storage and subsequently served by `starknet_getStateUpdate` as an authoritative state update. The stored state diff does not match the `state_diff_commitment` in the block header. Any downstream consumer (RPC client, proof verifier, re-execution tool) that recomputes the state diff hash will observe a mismatch. Additionally, the wrong class-type classification (a class appearing as both Sierra and deprecated) can cause wrong results in `get_compiled_class_hash` lookups that branch on whether a class is in `class_hash_to_compiled_class_hash`.

Impact: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The attack requires only a malicious P2P peer. Any peer participating in the `StateDiff` protocol can craft the two-chunk sequence. No privileged access is required. The node must be in P2P sync mode, which is a standard production deployment path.

### Recommendation

1. In `validate_deprecated_declared_classes_non_conflicting` (or at the end of `parse_data_for_block`), add a cross-category check:
   ```rust
   let deprecated_set: HashSet<_> = state_diff.deprecated_declared_classes.iter().collect();
   if state_diff.class_hash_to_compiled_class_hash.keys().any(|h| deprecated_set.contains(h)) {
       return Err(BadPeerError::ConflictingStateDiffParts);
   }
   ```
2. After assembling the final `ThinStateDiff`, compute `calculate_state_diff_hash(&result)` and compare it against the `state_diff_commitment` stored in the block header, rejecting the peer if they differ.

### Proof of Concept

1. A block header with `state_diff_length = 2` and `state_diff_commitment = H_legitimate` is stored (received from a prior header sync).
2. A malicious peer responds to the state diff query with:
   - Chunk 1: `DeclaredClass { class_hash: X, compiled_class_hash: Y }` — `len()` = 1
   - Chunk 2: `DeprecatedDeclaredClass { class_hash: X }` — `len()` = 1
3. `current_state_diff_len` reaches `target_state_diff_len = 2`; the loop exits.
4. `validate_deprecated_declared_classes_non_conflicting` passes: `deprecated_declared_classes = [X]` has no duplicates.
5. The resulting `ThinStateDiff { class_hash_to_compiled_class_hash: {X: Y}, deprecated_declared_classes: [X] }` is written to storage.
6. `starknet_getStateUpdate` for this block returns the corrupted diff.
7. Computing `calculate_state_diff_hash` on the stored diff yields `H_corrupted ≠ H_legitimate`, demonstrating the commitment invariant is broken. [7](#0-6)
