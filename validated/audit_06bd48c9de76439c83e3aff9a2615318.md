### Title
Missing `shard_id` Consistency Check in `set_state_header` Allows Peer-Induced State Corruption — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer without verifying that the chunk embedded in the header belongs to the requested `shard_id`. A malicious peer can supply shard A's chunk (with a valid Merkle proof for shard A's position in the block) in response to a state-sync request for shard B. All existing checks pass, the header is stored under `StateHeaderKey(shard_id=B, sync_hash)`, and subsequent state-part application writes shard A's trie data into shard B's storage, permanently corrupting the node's state for shard B.

---

### Finding Description

During state sync, a syncing node calls `set_state_header(shard_id, sync_hash, shard_state_header)` with data received from a peer. The function performs five validation steps: [1](#0-0) 

1. **`validate_chunk_proofs`** – verifies the chunk's internal hash, tx root, and receipts root.
2. **`verify_path`** – verifies the chunk is included in `sync_prev_block_header.chunk_headers_root()` via the supplied Merkle proof. [2](#0-1) 

3. Incoming-receipts proof chain validation.
4. **`validate_state_root_node`** – verifies the `state_root_node` matches `chunk.prev_state_root()`. [3](#0-2) 

**None of these steps check that `chunk.shard_id() == shard_id`.**

A malicious peer can respond with shard A's chunk and shard A's Merkle proof (both valid for shard A's position in the block). Steps 1 and 2 pass because shard A's chunk is genuinely in the block. Steps 3 and 4 pass because the peer can supply valid receipts for shard B and a valid `state_root_node` for shard A's state root.

The header is then stored at `StateHeaderKey(shard_id=B, sync_hash)` but contains shard A's `prev_state_root`: [4](#0-3) 

When `set_state_part` is subsequently called for shard B, it reads this header, extracts shard A's state root, and validates incoming parts against it: [5](#0-4) 

A malicious peer supplies shard A's state parts (valid for shard A's state root), which pass `validate_state_part` and are stored under `StatePartKey(sync_hash, shard_id=B, part_id)`. Finally, `apply_state_part` writes shard A's trie nodes into shard B's `ShardUId` column in RocksDB: [6](#0-5) 

And `set_state_finalize` applies shard A's chunk transactions to shard B's storage: [7](#0-6) 

The `shard_uid` used for storage is derived from the `shard_id` parameter (shard B), not from the chunk's actual shard: [8](#0-7) 

The broken invariant is: the `ShardStateSyncResponseHeader` stored at `StateHeaderKey(shard_id, sync_hash)` must contain the chunk and state root for `shard_id`. No code enforces this.

---
