Audit Report

## Title
Unverified Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` stores a caller-supplied `pool` address verbatim in transient context without verifying it against any factory allowlist. The callback `metricOmmModifyLiquidityCallback` authenticates the caller only by comparing `msg.sender` to that unverified stored address, then unconditionally trusts `msg.sender` to supply token addresses via `getImmutables()`. A malicious pool can therefore drain up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 from any victim who has approved this contract.

## Finding Description
All four public entry points (`addLiquidityExactShares` ×2, `addLiquidityWeighted` ×2) funnel into `_addLiquidity`, which calls `_setPayContext(pool, ...)` storing the caller-supplied address verbatim: [1](#0-0) 

`_addLiquidity` then calls `IMetricOmmPoolActions(pool).addLiquidity(...)` on the unverified address: [2](#0-1) 

Inside `metricOmmModifyLiquidityCallback`, the sole authenticity check is: [3](#0-2) 

Since `expectedPool` is the attacker-controlled address, this check passes trivially. The callback then calls `getImmutables()` on `msg.sender` (the malicious pool) to obtain token addresses, and pays those tokens from the victim to `msg.sender`: [4](#0-3) 

The contract's own NatSpec explicitly acknowledges this gap: [5](#0-4) 

The attacker controls both the token addresses (via `getImmutables()`) and the amounts (via the `amount0Delta`/`amount1Delta` arguments passed into the callback), bounded only by the victim's own `maxAmountToken0`/`maxAmountToken1` caps.

## Impact Explanation
Direct loss of user principal. A victim who has approved `MetricOmmPoolLiquidityAdder` for any ERC-20 (e.g., USDC, WETH) loses up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 in a single transaction. Tokens are transferred directly to the attacker's malicious pool contract with no recovery path. This meets the Critical/High threshold for direct loss of user principal under the Metric OMM allowed impact gate.

## Likelihood Explanation
The attack requires the victim to have approved `MetricOmmPoolLiquidityAdder` (a normal prerequisite for any LP action) and to be induced to call one of the four entry points with a malicious pool address. This is achievable via a spoofed front-end, phishing link, or malicious aggregator integration. No privileged role is required on the attacker's side; the malicious pool is a trivially deployable contract. The attack is repeatable against any victim who has granted approval.

## Recommendation
Verify the supplied `pool` address against `MetricOmmPoolFactory` before storing it in the transient pay context, for example in `_addLiquidity` or at each public entry point:

```solidity
require(IMetricOmmPoolFactory(factory).isPool(pool), "Unknown pool");
```

This closes the gap without restricting permissionless LP flows for legitimate factory-registered pools.

## Proof of Concept
1. Victim approves `MetricOmmPoolLiquidityAdder` for USDC and WETH.
2. Attacker deploys `MaliciousPool` implementing `addLiquidity` and `getImmutables` returning USDC/WETH.
3. Victim is induced to call `addLiquidityExactShares(maliciousPool, victim, 0, deltas, MAX0, MAX1, "")`.
4. `_addLiquidity` stores `maliciousPool` as `expectedPool` in transient storage, then calls `maliciousPool.addLiquidity(...)`.
5. `MaliciousPool.addLiquidity` calls back `metricOmmModifyLiquidityCallback(MAX0, MAX1, abi.encode(KIND_PAY))`.
6. Callback checks `msg.sender == expectedPool` → passes (both are `maliciousPool`).
7. Callback calls `maliciousPool.getImmutables()` → returns `token0=USDC, token1=WETH`.
8. `pay(USDC, victim, maliciousPool, MAX0)` and `pay(WETH, victim, maliciousPool, MAX1)` execute.
9. Attacker receives MAX0 USDC + MAX1 WETH from victim's wallet.
