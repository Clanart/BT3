### Title
Undocumented Bootstrap Exemption Bypasses Signature Verification, Nonce Enforcement, and Fee Payment in Declare Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `execute_declare_transaction`, a special-case "bootstrap" path is triggered solely by matching a magic felt literal (`'BOOTSTRAP'`), a zero nonce, version 3, and zero resource bounds. These criteria are not cryptographically enforced. Any unprivileged transaction sender can craft a declare transaction satisfying all four conditions, causing the OS to skip `__validate_declare__`, skip `check_and_increment_nonce`, and skip `charge_fee` — declaring an arbitrary class hash into the global state with no fee payment and no signature verification.

---

### Finding Description

Inside `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the OS performs the following check:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately — before:

- `check_and_increment_nonce` (line 779), which enforces replay protection
- `non_reverting_select_execute_entry_point_func` for `__validate_declare__` (line 804), which verifies the sender's signature
- `charge_fee` (line 822), which deducts STRK from the sender [2](#0-1) 

The four triggering conditions are:

| Condition | Enforcement |
|---|---|
| `sender_address == 'BOOTSTRAP'` | None — any sender can set this field |
| `tx_info.nonce == 0` | None — any sender can set nonce=0 |
| `tx_info.version == 3` | None — any sender can set version=3 |
| `max_possible_fee == 0` | None — any sender can set all resource bounds to 0 |

`compute_max_possible_fee` returns 0 when all three resource bound `max_amount` fields are zero: [3](#0-2) 

`sender_address` is loaded from the hint `%{ DeclareTxFields %}` — it is a user-supplied field of the transaction, not a protocol-enforced identity. There is no on-chain check that the address `'BOOTSTRAP'` corresponds to a deployed account contract or that the sender is authorized to use this path.

The `'BOOTSTRAP'` string is a Cairo short-string felt literal. Its numeric value (`0x424f4f545354524150`) is a valid felt that any user can place in the `sender_address` field of a declare transaction.

---

### Impact Explanation

**Direct loss of funds (Critical).**

1. **Fee bypass**: STRK tokens that must be paid as transaction fees are never collected. `charge_fee` is skipped entirely. An attacker can declare an unbounded number of distinct class hashes (each with a unique `compiled_class_hash`) at zero cost, draining the protocol of fee revenue that is owed.

2. **Unauthorized class declaration without signature**: Because `__validate_declare__` is skipped, no cryptographic proof of account ownership is required. The attacker does not need to control any account contract. Arbitrary class hashes are written into `contract_class_changes` and committed to the global state root, which is verified on L1. Malicious classes (e.g., backdoored account contracts or ERC-20 implementations) can be registered in the canonical class registry without authorization, enabling downstream fund theft when users deploy from those classes.

3. **Nonce not incremented**: Because `check_and_increment_nonce` is skipped, the nonce at address `'BOOTSTRAP'` remains 0 indefinitely. The attacker can submit an unlimited stream of bootstrap declare transactions, each with `nonce=0`, without nonce conflicts. The only per-class guard is `prev_value=0` in `dict_update`, which prevents re-declaring the same class hash but does not limit the total number of distinct classes declared. [4](#0-3) 

---

### Likelihood Explanation

The four triggering conditions are entirely attacker-controlled fields in a standard declare transaction. No privileged key, leaked secret, or operator cooperation is required to craft the transaction. The OS Cairo code — which is the authoritative state-transition function proven on L1 — contains no cryptographic guard preventing this path from being triggered by any sender. A sequencer that does not apply an out-of-band filter (which is not enforced by the protocol) will include such a transaction, produce a valid STARK proof, and have it accepted by the L1 verifier.

---

### Recommendation

1. **Remove the magic-string bootstrap path from the production OS code.** Bootstrapping should be handled through a separate, explicitly privileged mechanism (e.g., a genesis block with a distinct proof mode) rather than a runtime felt comparison.

2. If the bootstrap path must remain, **enforce it cryptographically**: require a signature from a well-known, protocol-defined bootstrap key, and verify it inside the Cairo code rather than relying on a felt literal match.

3. **Document the selection criteria** for any fee-exempt or validation-exempt sender in the OS specification, analogous to the recommendation in the BWGToken report, so that sequencer implementations can apply consistent filtering.

---

### Proof of Concept

An attacker constructs and submits the following declare transaction to the sequencer mempool:

```
sender_address  = 0x424f4f545354524150  // felt encoding of 'BOOTSTRAP'
nonce           = 0
version         = 3
resource_bounds = [
    { resource: L1_GAS,      max_amount: 0, max_price_per_unit: 0 },
    { resource: L2_GAS,      max_amount: 0, max_price_per_unit: 0 },
    { resource: L1_DATA_GAS, max_amount: 0, max_price_per_unit: 0 },
]
class_hash      = <hash of attacker-chosen malicious Sierra class>
compiled_class_hash = <corresponding CASM hash>
signature       = []  // empty — never checked
```

When the sequencer includes this transaction and the OS processes it:

1. `compute_declare_transaction_hash` computes a valid hash over these fields. [5](#0-4) 
2. `fill_account_tx_info` populates `tx_info` with `max_fee=0` and the zero resource bounds. [6](#0-5) 
3. The bootstrap condition at line 764 evaluates to true. `compute_max_possible_fee` returns 0. [7](#0-6) 
4. `dict_update` writes `compiled_class_hash` into `contract_class_changes` for the attacker's class hash. No fee is charged, no signature is verified, no nonce is incremented. [4](#0-3) 
5. The OS returns a valid proof. L1 accepts it. The malicious class is now part of the canonical StarkNet state.

The attacker repeats with a different `class_hash` and `compiled_class_hash` (still `nonce=0`) to register additional malicious classes at zero cost.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
