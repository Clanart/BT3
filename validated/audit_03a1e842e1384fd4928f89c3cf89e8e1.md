### Title
Unchecked Parallel Burn-and-Mint in `swap_migrated_token` Enables Unbacked New-Token Minting - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`swap_migrated_token` issues a NEAR joint promise `burn.and(mint)` with no result callback. Because NEAR executes both sub-promises independently, a failure of the `burn` leg does not prevent the `mint` leg from completing. Any user who triggers the swap path while the burn reverts receives freshly minted new tokens without the corresponding old tokens being destroyed, inflating the new token's total supply beyond what is collateralized.

---

### Finding Description

After a DAO calls `migrate_deployed_token`, users are expected to exchange their old-token balances for new-token balances by sending old tokens to the bridge with the `SwapMigratedToken` message. The handler in `ft_on_transfer` immediately returns `U128(0)` (all tokens consumed by the bridge) and fires `swap_migrated_token` as a detached promise:

```rust
// near/omni-bridge/src/lib.rs  lines 275-279
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
```

Inside `swap_migrated_token` the burn and mint are composed with `Promise::and`:

```rust
// near/omni-bridge/src/lib.rs  lines 2749-2752
let burn = ext_token::ext(old_token).burn(amount);
let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
burn.and(mint)
```

`Promise::and` in the NEAR SDK schedules both sub-promises for independent execution. Neither sub-promise is conditional on the other succeeding. If `burn` panics or reverts for any reason, `mint` still executes and new tokens are credited to `sender_id`.

The `burn` implementation on `OmniToken` withdraws from `env::predecessor_account_id()` (the bridge contract) and asserts the caller is the controller:

```rust
// near/omni-token/src/lib.rs  lines 146-151
fn burn(&mut self, amount: U128) {
    self.assert_controller();
    self.token
        .internal_withdraw(&env::predecessor_account_id(), amount.into());
}
```

Conditions under which `burn` can revert while `mint` still succeeds:

- The old token contract is upgraded (by DAO or deployer) between the `ft_on_transfer` receipt and the async `burn` dispatch, removing or altering the `burn` entry point.
- The old token contract is a non-standard or partially-compatible token whose `burn` method panics for any reason (e.g., storage exhaustion, changed controller).
- Any cross-contract panic in the old token's execution environment.

Because `ft_on_transfer` already returned `U128(0)`, the old token contract records the bridge as having received the full `amount`. The bridge holds those old tokens permanently (no refund path exists), while the user receives `amount` of new tokens that are entirely unbacked.

---

### Impact Explanation

Every successful `swap_migrated_token` call where `burn` fails but `mint` succeeds mints `amount` new tokens with no corresponding destruction of old tokens. The new token's `total_supply` grows beyond the amount that was legitimately migrated, breaking the 1:1 collateralization invariant of the bridge. Repeated exploitation inflates the new token supply arbitrarily, enabling the attacker to redeem unbacked bridged assets on the origin chain (EVM, Solana, StarkNet) or sell them on secondary markets.

This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

The entry path is fully unprivileged: any holder of old tokens can call `ft_transfer_call` on the old token contract targeting the bridge with `"SwapMigratedToken"` as the message. The precondition (a successful `migrate_deployed_token` call) is a normal protocol operation. The burn failure condition is low-probability in a healthy deployment but is not guarded against at all in the code. A single upgrade of the old token contract, a storage panic, or any revert in the old token's execution makes the vulnerability immediately exploitable by any user who races to call the swap before the bridge operator notices.

---

### Recommendation

Replace the parallel `burn.and(mint)` with a sequential callback pattern that only mints if the burn succeeded:

```rust
fn swap_migrated_token(
    &mut self,
    sender_id: AccountId,
    old_token: AccountId,
    amount: U128,
) -> Promise {
    let new_token = self
        .migrated_tokens
        .get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);

    ext_token::ext(old_token)
        .burn(amount)
        .then(
            Self::ext(env::current_account_id())
                .swap_migrated_token_callback(sender_id, new_token, amount),
        )
}

#[private]
pub fn swap_migrated_token_callback(
    &mut self,
    sender_id: AccountId,
    new_token: AccountId,
    amount: U128,
) {
    require!(
        env::promise_results_count() == 1
            && matches!(env::promise_result(0), PromiseResult::Successful(_)),
        "Burn failed; aborting mint"
    );
    ext_token::ext(new_token).mint(sender_id, amount, None);
}
```

This ensures the mint is only issued after the burn is confirmed successful.

### Proof of Concept

1. DAO calls `migrate_deployed_token(origin_chain, old_token, new_token)`.
   - `migrated_tokens[old_token] = new_token` is recorded.
   - `deployed_tokens` now contains `new_token`, not `old_token`.

2. Attacker holds `X` units of `old_token`.

3. Between the `migrate_deployed_token` transaction and the attacker's swap, the old token contract is upgraded (or panics for any reason during `burn`).

4. Attacker calls `old_token.ft_transfer_call(bridge, X, "\"SwapMigratedToken\"")`.
   - `ft_on_transfer` returns `U128(0)` — bridge is recorded as holding `X` old tokens.
   - `swap_migrated_token` is detached.

5. NEAR runtime executes `burn.and(mint)`:
   - `old_token.burn(X)` → **reverts** (e.g., method not found after upgrade).
   - `new_token.mint(attacker, X, None)` → **succeeds** (independent sub-promise).

6. Result:
   - Attacker receives `X` new tokens.
   - Bridge holds `X` old tokens (not burned, no refund path).
   - `new_token.total_supply` is inflated by `X` with no backing. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
