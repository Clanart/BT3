### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Unbounded Signature Replay - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that contains no nonce and explicitly skips nonce enforcement for version-0 transactions. Any signature produced for a `meta_tx_v0` call is therefore valid for unlimited replays, because the hash over `(prefix, version=0, contract_address, selector, calldata, max_fee=0, chain_id)` is permanently fixed for a given set of inputs. An unprivileged relayer or attacker who observes a single valid `meta_tx_v0` signature can re-submit it indefinitely, draining the target account.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash/transaction_hash.cairo` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce committed
    );
``` [1](#0-0) 

The resulting `meta_tx_hash` is then placed into a synthetic `TxInfo` with `nonce=0` hardcoded:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // ← always zero
    ...
);
``` [2](#0-1) 

The OS nonce-enforcement function `check_and_increment_nonce` explicitly returns early for version-0 transactions without reading or updating any on-chain nonce:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

Furthermore, `execute_meta_tx_v0` calls `contract_call_helper` directly, which invokes `select_execute_entry_point_func` without ever calling `run_validate`. The OS therefore never independently verifies the signature; it is passed raw into `tx_info.signature_start/end` for the target contract's `__execute__` to consume. [4](#0-3) 

Because the hash is a pure function of `(prefix, 0, contract_address, selector, calldata, 0, chain_id)` with no time-varying component, the hash — and therefore the valid signature set — is identical across every invocation with the same inputs.

---

### Impact Explanation

The `meta_tx_v0` syscall is the StarkNet mechanism for gasless/sponsored transactions: a user signs a version-0 style transaction hash and a relayer submits it. Once a user has signed a `meta_tx_v0` authorizing, for example, an ERC-20 transfer of amount X to address Y, that signature is permanently valid. Any party who observes the signature (e.g., from a public mempool, a past block, or a relayer's logs) can replay it an unlimited number of times, repeatedly executing the same call against the target account contract. For a token-transfer calldata this directly and repeatedly drains the user's balance — **direct loss of funds (Critical)**.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is a first-class protocol feature with its own gas schedule (`META_TX_V0_GAS_COST`), indicating it is intended for production use. [5](#0-4) 

Any relayer, MEV searcher, or on-chain observer who captures a single valid `meta_tx_v0` signature from a transaction or event log can immediately replay it. No privileged access, leaked key, or social engineering is required — only the ability to call a contract that issues the `meta_tx_v0` syscall with the captured signature. The attack is fully permissionless and deterministic.

---

### Recommendation

Include a per-account, monotonically incrementing nonce in the `meta_tx_v0` hash, analogous to how standard v3 transactions commit to a nonce via `CommonTxFields.nonce`. Concretely:

1. Add a `nonce` parameter to `compute_meta_tx_v0_hash` and pass it as `additional_data` (size=1) to `deprecated_get_transaction_hash`.
2. In `execute_meta_tx_v0`, read the current nonce from `contract_state_changes` for `contract_address`, pass it to the hash function, and then increment it (do not rely on `check_and_increment_nonce`'s version-0 early-return path — either fix that guard or call a dedicated helper).
3. Set `new_tx_info.nonce` to the actual consumed nonce rather than the hardcoded `0`.

This mirrors the fix recommended in the referenced report (EIP-712 domain separation / per-use nonce) and closes the replay window entirely.

---

### Proof of Concept

**Setup:** Deploy an ERC-20-compatible account contract `A` at address `addr_A`. User signs a `meta_tx_v0` authorizing `transfer(recipient=attacker, amount=1000)`. The signature `sig` is submitted once by a legitimate relayer.

**Replay steps (all permissionless):**

1. Attacker deploys or calls any contract `R` that issues the `meta_tx_v0` syscall.
2. `R` calls `meta_tx_v0` with:
   - `contract_address = addr_A`
   - `selector = EXECUTE_ENTRY_POINT_SELECTOR`
   - `calldata = [transfer, attacker, 1000]`
   - `signature = sig` (copied from the first legitimate transaction)
3. The OS computes `meta_tx_hash = hash(INVOKE_PREFIX, 0, addr_A, EXECUTE_SELECTOR, calldata, 0, chain_id)` — identical to the original.
4. `A.__execute__` receives `tx_info` with `transaction_hash = meta_tx_hash` and `signature = sig`. Standard ECDSA validation passes because the hash and signature are unchanged.
5. The transfer executes, sending 1000 tokens to the attacker.
6. Steps 2–5 repeat until `A`'s balance is zero. No nonce is ever checked or incremented by the OS.
