### Title
Borsh `Option` Encoding Mismatch in `OmniBridgeWormhole.finTransferExtension` Permanently Freezes Locked NEAR Tokens - (File: evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol)

### Summary
`OmniBridgeWormhole.finTransferExtension` serializes `feeRecipient` as a plain Borsh string in the Wormhole VAA payload, but the NEAR side's `FinTransferMessage.fee_recipient` is typed `Option<AccountId>` and therefore expects Borsh Option encoding (`\x00` for None, `\x01` + string for Some). This systematic encoding mismatch causes the NEAR Wormhole prover to fail to parse every Wormhole VAA produced by `finTransfer`, making `claim_fee` permanently uncallable and leaving the originating user's locked NEAR tokens irrecoverably frozen in the bridge.

### Finding Description

**Root cause — `OmniBridgeWormhole.finTransferExtension`:**

```solidity
// evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol  lines 96-116
function finTransferExtension(
    BridgeTypes.TransferMessagePayload memory payload
) internal override {
    bytes memory messagePayload = bytes.concat(
        bytes1(uint8(MessageType.FinTransfer)),
        bytes1(payload.originChain),
        Borsh.encodeUint64(payload.originNonce),
        bytes1(omniBridgeChainId),
        Borsh.encodeAddress(payload.tokenAddress),
        Borsh.encodeUint128(payload.amount),
        Borsh.encodeString(payload.feeRecipient)   // ← plain string, no Option prefix
    );
    _wormhole.publishMessage{value: msg.value}(wormholeNonce, messagePayload, _consistencyLevel);
    wormholeNonce++;
}
``` [1](#0-0) 

`Borsh.encodeString(s)` produces `[len_le32][utf8_bytes]`. For an empty string it produces `\x00\x00\x00\x00` (4 bytes); for `"alice.near"` it produces `\x0a\x00\x00\x00alice.near`.

**The NEAR side expects Borsh `Option<AccountId>` encoding:**

The `FinTransferMessage` struct on NEAR declares `fee_recipient` as `Option<AccountId>`:

```rust
// near/omni-types/src/prover_result.rs  lines 22-27
pub struct FinTransferMessage {
    pub transfer_id: TransferId,
    pub fee_recipient: Option<AccountId>,
    pub amount: U128,
    pub emitter_address: OmniAddress,
}
``` [2](#0-1) 

Borsh `Option<T>` encoding is: `\x00` for `None`, `\x01` + `T` for `Some(T)`. The EVM `finTransfer` signature-verification path already uses this exact encoding, as confirmed by the inline comment:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 299-304
bytes(payload.feeRecipient).length == 0 // None or Some(String) in rust
    ? bytes("\x00")
    : bytes.concat(
        bytes("\x01"),
        Borsh.encodeString(payload.feeRecipient)
    ),
``` [3](#0-2) 

The comment `// None or Some(String) in rust` is the developer's own acknowledgement that the NEAR side uses Option encoding — yet `finTransferExtension` omits the Option prefix entirely.

**Concrete byte-level mismatch:**

| Scenario | Wormhole VAA bytes | NEAR expects |
|---|---|---|
| `feeRecipient = ""` | `\x00\x00\x00\x00` (4 bytes) | `\x00` (1 byte, None) |
|
