### Title
Missing Coinbase Maturity Check in `check_time_locks` Allows Premature Reward Coin Spending - (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
The `check_time_locks` function, which is the Rust-side time-lock validation exported as a Python binding, receives `CoinRecord` structs that carry a `coinbase: bool` field marking farming reward coins. The function never inspects this field and never raises `CoinbaseNotYetSpendable`, meaning a spend bundle that includes an immature reward coin passes all Rust-side time-lock validation without restriction. The error code exists in the codebase but is dead — it is defined but never emitted.

### Finding Description

`CoinRecord` carries a `coinbase` field that distinguishes farming reward coins from ordinary coins: [1](#0-0) 

`check_time_locks` receives a `HashMap<Bytes32, CoinRecord>` for every coin being removed, iterates over each spend, and performs birth-height, birth-seconds, height-relative, seconds-relative, before-height-relative, and before-seconds-relative checks — but never reads `unspent.coinbase`: [2](#0-1) 

The error code `CoinbaseNotYetSpendable` is defined and mapped to wire code 22, but is never raised anywhere in the consensus crate: [3](#0-2) 

A grep across all `crates/chia-consensus/src/*.rs` files confirms `coinbase` appears only once — in the test helper `dummy_coin_record`, where it is hardcoded to `false`: [4](#0-3) 

The function is exported as a `#[pyfunction]` and is the primary Rust-side gate for time-lock validation called by the Python mempool manager: [5](#0-4) 

### Impact Explanation

Coinbase/reward coins in Chia have a maturity period before they may be spent. If the Python mempool manager delegates coinbase maturity enforcement to `check_time_locks` (which is the natural expectation given the function's purpose and the presence of `CoinbaseNotYetSpendable`), an attacker who controls a farming reward coin can include it in a spend bundle before it matures. The spend bundle passes all Rust-side time-lock validation and may be accepted into the mempool and included in a block, constituting an unauthorized spend of a reward coin — a timelock validation bypass enabling unauthorized spend acceptance.

**Impact class:** High — timelock validation bypass enables unauthorized spend acceptance of reward coins.

### Likelihood Explanation

The `coinbase` field is present in the data structure passed to the function, the error code for the missing check already exists in the codebase, and the function is the sole Rust-side time-lock validator. Any caller that relies on `check_time_locks` as the complete maturity gate is exposed. Likelihood depends on whether the Python layer independently re-checks coinbase maturity; if it does not, exploitation requires only constructing a spend bundle referencing an immature reward coin.

### Recommendation

Inside the per-spend loop in `check_time_locks`, add a maturity check for coinbase coins:

```rust
if unspent.coinbase {
    // enforce the coinbase freeze period
    if prev_transaction_block_height
        < unspent.confirmed_block_index.saturating_add(constants.coinbase_freeze_period)
    {
        return Err(ValidationErr::Err(ErrorCode::CoinbaseNotYetSpendable));
    }
}
```

The `CoinbaseNotYetSpendable` error code is already defined and mapped; it only needs to be wired to an actual check.

### Proof of Concept

1. Farm a block and obtain the resulting coinbase reward `CoinRecord` with `coinbase = true` and `confirmed_block_index = H`.
2. Construct a `SpendBundle` spending that reward coin at block height `H` (before maturity).
3. Call `check_time_locks(removal_coin_records, bundle_conds, H, timestamp, true)`.
4. Observe `Ok(())` — no `CoinbaseNotYetSpendable` error is raised, because lines 40–113 of `check_time_locks.rs` never read `unspent.coinbase`.
5. The spend bundle passes Rust-side validation and can be forwarded to block inclusion.

#[cfg(feature = "py-bindings")]
#[pyfunction]
#[pyo3(name = "check_time_locks")]
#[allow(clippy::needless_pass_by_value)] // pyo3 prefers pass_by_value
pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
