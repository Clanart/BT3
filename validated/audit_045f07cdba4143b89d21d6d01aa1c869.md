### Title
Password Parameter Silently Discarded in `personal_sign` and `personal_sendTransaction` — Unauthenticated Key Use via JSON-RPC - (File: `rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

The Ethermint JSON-RPC `personal` namespace accepts a `password` parameter in `personal_sign` and `personal_sendTransaction` (as required by the Ethereum personal API specification) but silently discards it. Both methods sign with the node's keyring key for the requested address without ever verifying the supplied password. Any caller who can reach the JSON-RPC endpoint and whose target address exists in the node keyring can obtain a valid ECDSA signature or broadcast a signed Ethereum transaction on behalf of that address — with no password required.

---

### Finding Description

The Ethereum `personal_*` API specification requires that `personal_sign` and `personal_sendTransaction` authenticate the caller by verifying the supplied password against the encrypted key before signing. Ethermint's implementation ignores the password entirely.

**`personal_sign`** — `rpc/namespaces/ethereum/personal/api.go`, line 145:

```go
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)
}
```

The third parameter (password) is named `_ string` — it is received and immediately discarded. The call proceeds unconditionally to `api.backend.Sign(addr, data)`.

**`personal_sendTransaction`** — same file, line 131:

```go
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)
}
```

Again, the password parameter is `_ string` — discarded. `api.backend.SendTransaction(args)` signs and broadcasts the transaction using the keyring key for `args.From` without any password check.

The backend implementations confirm no password is ever used:

- `rpc/backend/sign_tx.go` `Sign()` (line 129–148): calls `b.clientCtx.Keyring.SignByAddress(from, ...)` directly.
- `rpc/backend/sign_tx.go` `SendTransaction()` (line 38–126): calls `msg.Sign(signer, b.clientCtx.Keyring)` directly.

The `personal_sign` docstring in the same file even states *"The key used to calculate the signature is decrypted with the given password"* — but the implementation never does this.

The analog to the original report is exact: sensitive cryptographic material (the node's private keys) is used without the protection mechanism (password/encryption) that the API contract promises. The keys are stored in the keyring (which may be the `test` backend — plaintext on disk — as shown in multiple startup scripts), and the password gate that should protect their use is absent.

---

### Impact Explanation

**High — Ethereum transaction signer verification bypass enabling forged execution / unauthorized account mutation.**

Any caller who can reach the JSON-RPC port (default `0.0.0.0:8545`) and knows (or can enumerate via `eth_accounts` / `personal_listAccounts`) an address whose key is in the node keyring can:

1. Call `personal_sendTransaction` with `from` set to that address and any `to`/`value` — the node will sign and broadcast a valid EIP-155 Ethereum transaction transferring funds from the victim address. This is a direct, unauthorized transfer of EVM-denom funds.
2. Call `personal_sign` with arbitrary data — the node will produce a valid ECDSA signature under the victim's key, which can be used for EIP-712 permit approvals, off-chain authorization schemes, or other signature-gated operations.

The `personal` namespace is explicitly included in the recommended startup flags (`--json-rpc.api eth,txpool,personal,net,debug,web3`) in `init.sh` and `tests/solidity/init-test-node.sh`, and the `test` keyring backend (keys stored unencrypted on disk) is the default used in all provided startup scripts.

---

### Likelihood Explanation

**High.** The `personal` namespace is enabled by default in the documented startup configuration. The JSON-RPC port is bound to `0.0.0.0` in typical deployments. The `test` keyring backend (no password, plaintext disk storage) is the default in all provided scripts. Any process or user that can reach port 8545 — including any co-located process, container sibling, or network peer if the port is exposed — can exploit this without any credentials. The attack requires only knowledge of an address in the keyring, which is trivially obtained via `eth_accounts`.

---

### Recommendation

1. **Short term:** In `personal_sign` and `personal_sendTransaction`, validate the supplied password against the keyring before signing. If the keyring backend does not support per-call password verification (the `test` backend does not), reject the call with an explicit error rather than proceeding unauthenticated.
2. **Long term:** Disable the `personal` namespace by default in production configurations, or gate it behind an explicit `--rpc.unsafe` flag (similar to how `--rpc.unsafe` is already used for other sensitive endpoints). Enforce use of the `os` or `file` keyring backend (never `test`) in any deployment that exposes the `personal` namespace.

---

### Proof of Concept

With a node running with `--json-rpc.api eth,personal` and `--keyring-backend test`:

```bash
# 1. Enumerate addresses in the keyring (no auth required)
curl -s -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_accounts","params":[],"id":1}'
# Returns: {"result":["0xVICTIM_ADDRESS",...]}

# 2. Send a transaction FROM the victim address TO attacker — no password needed
curl -s -X POST http://localhost:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{
      "from":"0xVICTIM_ADDRESS",
      "to":"0xATTACKER_ADDRESS",
      "value":"0xDE0B6B3A7640000"
    }, "wrong_password_or_empty"],
    "id":1
  }'
# Returns a valid tx hash — funds transferred despite wrong/empty password
```

The password `"wrong_password_or_empty"` is accepted without error because it is never checked.

func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}

// Sign signs the provided data using the private key of address via Geth's signature standard.
func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}

**File:** x/evm/types/msg.go (L237-257)
```go
func (msg *MsgEthereumTx) Sign(ethSigner ethtypes.Signer, keyringSigner keyring.Signer) error {
	from := msg.GetFrom()
	if from.Empty() {
		return fmt.Errorf("sender address not defined for message")
	}

	tx := msg.AsTransaction()
	txHash := ethSigner.Hash(tx)

	sig, _, err := keyringSigner.SignByAddress(from, txHash.Bytes(), signing.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		return err
	}

	tx, err = tx.WithSignature(ethSigner, sig)
	if err != nil {
		return err
	}

	msg.Raw.Transaction = tx
	return nil
```
