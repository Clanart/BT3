### Title
Proof Verification Permanently Bypassed for Any `proof_facts` Hash Already in `ProofManager` — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`run_proof_verification` skips ZK proof verification whenever `contains_proof(proof_facts)` returns `true`, keyed only on `proof_facts.hash()`. Once a valid `(proof_facts, proof)` pair is stored, any subsequent transaction carrying the **same** `proof_facts` but a completely **different, invalid** proof bypasses `verify_proof` entirely and is admitted by the gateway and consensus flows. This is the direct sequencer analog of the Hyperlane `retry` bug: a stored entry is never invalidated after first use, so the same "already-verified" marker can be exploited an unlimited number of times.

---

### Finding Description

In `run_proof_verification` the first thing done is:

```rust
let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
if contains_proof {
    return Ok(false);   // verification skipped
}
``` [1](#0-0) 

`contains_proof` resolves to a filesystem/cache lookup keyed on `proof_facts.hash()` — a Poseidon hash of the proof-facts vector:

```rust
pub async fn contains_proof(&self, proof_facts: ProofFacts) -> Result<bool, FsProofStorageError> {
    let facts_hash = proof_facts.hash();
    if self.cache.contains(&facts_hash) { return Ok(true); }
    self.proof_storage.contains_proof(facts_hash).await
}
``` [2](#0-1) 

The check is purely on the **hash of the proof facts**, not on the `(proof_facts, proof)` pair. Once `hash(F) → P1` is written to the proof manager, every future call with the same `proof_facts` F — regardless of what `proof` is supplied — returns `true` and skips `starknet_proof_verifier::verify_proof`. [3](#0-2) 

Both the **gateway flow** (`spawn_proof_verification` → `store_proof_and_spawn_archiving`) and the **consensus flow** (`spawn_verify_and_store_proof`) share this path: [4](#0-3) [5](#0-4) 

The gateway calls `store_proof_in_proof_manager` only after stateful validation passes, so the invalid proof is never stored and never verified: [6](#0-5) 

The `FsProofStorage` persistence layer confirms the key is the hash alone: [7](#0-6) 

---

### Impact Explanation

An attacker who has submitted **one** valid client-side-proving `InvokeV3` transaction (valid `proof_facts` F, valid proof P1) can subsequently submit an unlimited number of transactions carrying the same `proof_facts` F but an entirely invalid proof (e.g., all-zero bytes). Each such transaction passes `contains_proof` → skips `verify_proof` → passes blockifier `validate_proof_facts` (which only checks program hash, block hash, block number, config hash — not the proof itself) → is admitted to the mempool.

The sequencer therefore accepts transactions bearing false ZK attestations. In the client-side proving / privacy-pool model this breaks the correctness guarantee: a user can claim to have proven their action is valid without possessing a valid proof.

Matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The precondition is trivially met: the attacker submits one legitimate transaction to seed the proof manager, then replays the same `proof_facts` with arbitrary invalid proofs indefinitely. No privileged access, no malicious peer, no hash collision is required. The `proof_facts` for any given block are public (block hash and block number are on-chain).

---

### Recommendation

The `contains_proof` shortcut must also verify that the **submitted proof matches the stored proof**, or the skip must be removed entirely and every `(proof_facts, proof)` pair verified independently. A minimal fix is to compare the incoming proof against the stored proof before returning early:

```rust
if let Some(stored_proof) = proof_manager_client.get_proof(proof_facts.clone()).await? {
    if stored_proof == proof {
        return Ok(false); // identical pair already verified
    }
    // proof differs — fall through and re-verify
}
```

Alternatively, store the proof only after verification and never skip verification for a new proof, even if the proof_facts hash is known.

---

### Proof of Concept

1. Attacker submits `InvokeV3` **Tx1** with valid `proof_facts` F (referencing block B, correct `block_hash`, `program_hash`, `config_hash`) and valid proof P1.
2. Gateway calls `run_proof_verification(F, P1, pmc)` → `contains_proof(F)` = `false` → `verify_proof(F, P1)` succeeds → `hash(F) → P1` written to `FsProofStorage`.
3. Attacker submits **Tx2** with the same `proof_facts` F but proof P2 = `Proof::from(vec![0u8; 1])` (invalid).
4. Gateway calls `run_proof_verification(F, P2, pmc)` → `contains_proof(F)` = `true` → returns `Ok(false)` immediately; `verify_proof(F, P2)` is **never called**.
5. Blockifier `validate_proof_facts` checks only the fields inside F (program hash, block hash, block number, config hash) — all valid — and returns `Ok(())`.
6. Tx2 is forwarded to the mempool and sequenced with an unverified, invalid proof.

Steps 3–6 can be repeated with any proof payload, unlimited times, for any `proof_facts` F that was previously seeded in step 1. [8](#0-7) [9](#0-8) [10](#0-9)

    /// Runs proof verification: checks if the proof already exists, and if not, verifies it.
    /// Returns `true` if verification was performed, `false` if skipped (proof already stored).
    /// This is the shared verification logic used by both gateway and consensus flows.
    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
