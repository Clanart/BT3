### Title
Missing Validation of Declared Class Hash in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by a contract corresponds to a previously declared class. This allows any contract — including one deployed by an unprivileged user — to replace its own class hash with an arbitrary, undeclared value. Once replaced, the contract becomes permanently non-executable, and any funds held within it are permanently frozen.

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts the caller-supplied `class_hash` from the syscall request and writes it directly into `contract_state_changes` without verifying that a compiled class with that hash has been declared:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges this missing check. The OS developers recognize it as an OS-level responsibility, not a contract-level one.

When a subsequent call is made to the contract with the invalid class hash, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

`find_element` is not a safe search — it asserts the element exists. If the class hash is undeclared, no valid proof can be generated for any call to that contract. The sequencer cannot include such calls, making the contract permanently non-executable. Since `replace_class` can only be invoked from within the contract itself, and the contract can no longer be called, the state is irreversible.

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any funds (tokens, ETH, or other assets) held in a contract whose class hash has been replaced with an undeclared value are permanently inaccessible. No withdrawal, transfer, or recovery is possible because:
1. The contract cannot be called (no valid proof can be generated).
2. The only mechanism to restore the class hash (`replace_class`) requires calling the contract, which is impossible.
3. There is no OS-level recovery path.

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer or caller:

- **Scenario A (self-inflicted, affecting depositors):** An attacker deploys a contract that appears to be a legitimate vault or token. Users deposit funds. The attacker then calls a function that invokes `replace_class` with an undeclared hash (e.g., `0xdeadbeef`). All deposited funds are permanently frozen.
- **Scenario B (vulnerable contract):** A legitimate contract with a bug that allows an attacker to supply an arbitrary class hash to `replace_class` can be exploited to freeze funds held by that contract.

The `replace_class` syscall is available to any Cairo 1 contract. No privileged role is required to call it. The OS provides no guard against an invalid hash.

### Recommendation

In `execute_replace_class`, before writing the new class hash to `contract_state_changes`, validate that the hash exists in `contract_class_changes` (i.e., it was previously declared in this block or in a prior block). Specifically, perform a `dict_read` on `contract_class_changes` with the new `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// Validate that the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the validation already performed implicitly in `execute_entry_point` and makes the invariant explicit and enforced at the point of mutation.

### Proof of Concept

1. Declare a Cairo 1 contract class `MaliciousVault` with a public `freeze_self()` function that calls `replace_class(0xdeadbeef_undeclared_hash)`.
2. Deploy `MaliciousVault`. Users deposit funds.
3. Call `freeze_self()`. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`, which is not in `contract_class_changes`. The syscall succeeds — no validation is performed.
4. Attempt any subsequent call to `MaliciousVault`. `execute_entry_point` reads `compiled_class_hash = 0` (undeclared), then `find_element` fails to locate a compiled class, making proof generation impossible.
5. The sequencer cannot include any transaction targeting `MaliciousVault`. All deposited funds are permanently frozen.
