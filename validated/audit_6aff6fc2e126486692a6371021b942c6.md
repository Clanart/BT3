### Title
Replay Attack on `execute_meta_tx_v0` Due to Missing Nonce in Hash and No Nonce Increment — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall implementation computes a transaction hash that does not include any nonce, and the OS never increments the target contract's nonce after executing a meta-transaction. This allows any unprivileged relayer to replay the same `(contract_address, calldata, signature)` tuple an unlimited number of times, causing repeated execution of the target account's `__execute__` entry point — directly analogous to the CommunityRound.sol replay attack.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` constructs a version-0 transaction context for the target contract and calls its `__execute__` entry point. The hash it signs over is computed by `compute_meta_tx_v0_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // <-- NO NONCE
        additional_data=cast(0, felt*),
    );
``` [1](#0-0) 

The `additional_data_size=0` means no nonce is committed to in the signed hash. The resulting `new_tx_info` is constructed with a hardcoded `nonce=0` and `version=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // <-- hardcoded, never incremented
    ...
);
``` [2](#0-1) 

The OS's `check_and_increment_nonce` function explicitly skips all version-0 transactions:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

Critically, `execute_meta_tx_v0` never calls `check_and_increment_nonce` at all — the target contract's nonce in `contract_state_changes` is never touched. The function goes directly from computing the hash to calling `contract_call_helper`: [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `meta_tx_v0` syscall is the StarkNet gasless-relay primitive: a user signs a meta-transaction authorizing a specific `(contract_address, calldata)` action (e.g., a token transfer via `__execute__`). Because the signed hash contains no nonce and the OS never increments the target contract's nonce, the identical `(contract_address, calldata, signature)` triple produces the same valid hash on every invocation. Any party that observes a successfully relayed meta-tx can re-submit it in a new outer invoke transaction, causing the target account's `__execute__` to fire again — draining funds with each replay.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Observing a valid meta-tx on-chain (public information).
2. Submitting a new outer invoke transaction that calls any relayer contract (or a purpose-built attacker contract) with the same `contract_address`, `calldata`, and `signature`.

No privileged access, leaked keys, or trusted-role compromise is needed. The attacker is an ordinary unprivileged transaction sender.

---

### Recommendation

Include the target contract's current nonce in the meta-tx hash (as `additional_data`) and increment the target contract's nonce in `contract_state_changes` after each successful `meta_tx_v0` execution — mirroring exactly what `check_and_increment_nonce` does for regular account transactions. Specifically:

1. In `compute_meta_tx_v0_hash`: pass `additional_data_size=1` and `additional_data=&nonce` (the current nonce of `contract_address`).
2. In `execute_meta_tx_v0`: read the current nonce from `contract_state_changes` for `contract_address`, include it in the hash, and write back an incremented nonce — regardless of the version-0 designation. [5](#0-4) 

---

### Proof of Concept

**Setup**: Alice's account contract at address `A` holds 1000 tokens. Alice signs a meta-tx authorizing `calldata = [transfer(Bob, 100)]` and gives the signature to Relayer R.

**Step 1 — Legitimate relay**: R submits an outer invoke transaction. R's relayer contract calls `meta_tx_v0(contract_address=A, selector=__execute__, calldata=[transfer(Bob,100)], signature=sig)`. The OS computes `hash = H(invoke, 0, A, __execute__, calldata, 0, chain_id)` — identical every time. Alice's `__execute__` runs, 100 tokens transfer to Bob. R's outer tx nonce increments; A's nonce does not.

**Step 2 — Replay**: Attacker (or R again) submits a new outer invoke transaction with a fresh nonce, calling `meta_tx_v0` with the exact same parameters. The OS recomputes the identical hash. Alice's `__execute__` runs again, transferring another 100 tokens to Bob (or to the attacker if calldata is crafted accordingly).

**Step 3 — Drain**: Repeat until Alice's balance is zero. Each replay costs only the outer transaction's gas fee, paid by the attacker, not Alice.

The root cause is confirmed at:
- `compute_meta_tx_v0_hash` — `additional_data_size=0` omits nonce from hash [5](#0-4) 
- `execute_meta_tx_v0` — `nonce=0` hardcoded, no nonce increment on target contract [6](#0-5) 
- `check_and_increment_nonce` — version-0 bypass, and not called at all in meta-tx path [3](#0-2)
