### Title
Griefing Attack: Pre-Creating Deterministic NEAR Token Account Permanently Blocks Token Deployment - (`near/token-deployer/src/lib.rs`, `near/omni-bridge/src/lib.rs`)

---

### Summary

An unprivileged attacker can permanently prevent any specific token from being deployed on NEAR by pre-creating the deterministic NEAR account that the bridge's token deployer would use. Because `TokenDeployer::deploy_token` unconditionally calls `Promise::new(account_id).create_account()`, and because NEAR's `create_account` action fails if the account already exists, the deployer call always fails for that token. The bridge rolls back its internal state on failure, but the attacker retains control of the account, making the DoS permanent. Users who have already initiated cross-chain transfers from EVM (or other chains) to NEAR for that token have their funds permanently locked in the source-chain bridge with no path to finalization.

---

### Finding Description

**Root cause — deterministic, publicly computable token account ID:**

In `deploy_token_internal`, the NEAR account ID for the bridged token is derived deterministically from the source-chain token address:

```rust
let prefix = token_address.get_token_prefix();
let token_id: AccountId = format!("{prefix}.{deployer}").parse()...;
``` [1](#0-0) 

`get_token_prefix` is fully deterministic and public:

```rust
pub fn get_token_prefix(&self) -> String {
    match self {
        Self::Eth(address) => address.to_string()[2..].to_string(),  // e.g. "23ddd3e3..."
        ...
    }
}
``` [2](#0-1) 

**Root cause — unconditional `create_account` in the deployer:**

`TokenDeployer::deploy_token` always calls `create_account()` with no check for prior existence:

```rust
pub fn deploy_token(&mut self, account_id: AccountId, metadata: &BasicMetadata) -> Promise {
    Promise::new(account_id)
        .create_account()
        .transfer(env::attached_deposit())
        .use_global_contract(self.global_code_hash)
        .function_call("new".to_string(), ...)
}
``` [3](#0-2) 

On NEAR, `create_account` fails if the account already exists. Creating a NEAR account at any sub-account address is permissionless and costs only ~0.1 NEAR.

**State rollback on failure — but the account remains blocked:**

`deploy_token_by_deployer_callback` correctly rolls back the bridge's internal maps when the deployer call fails:

```rust
} else {
    self.deployed_tokens.remove(&token_id);
    self.deployed_tokens_v2.remove(&token_id);
    self.token_id_to_address.remove(&(token_address.get_chain(), token_id));
    self.token_address_to_id.remove(token_address);
    self.token_decimals.remove(token_address);
    PromiseOrValue::Value(())
}
``` [4](#0-3) 

The bridge state is cleaned up, but the attacker still controls the NEAR account at `{prefix}.{deployer}`. Every subsequent `deploy_token` call for that token will hit the same `create_account` failure. The token is permanently undeployable on NEAR.

---

### Impact Explanation

**Permanent freezing of user funds (Critical):**

The EVM `initTransfer` does not require the destination NEAR token to already be deployed. A user can lock EVM tokens in the bridge targeting a NEAR recipient before the NEAR token contract exists. If the NEAR token can never be deployed (because the attacker holds the account), `fin_transfer` on NEAR can never succeed for that token, and the user's EVM-side locked tokens have no recovery path. This constitutes an irrecoverable lock of user funds in the bridge vault flow.

**Permanent denial of service for token deployment:**

Even absent in-flight transfers, the attack permanently removes a specific token from the bridge's supported set on NEAR. Any token whose source-chain address is known in advance (i.e., all tokens) is vulnerable before its NEAR deployment is executed.

---

### Likelihood Explanation

- **Permissionless entry:** Creating a NEAR sub-account costs ~0.1 NEAR and requires no special access.
- **Publicly computable target:** The account ID `{get_token_prefix(token_address)}.{deployer}` is fully deterministic from on-chain data visible to anyone monitoring the source chain.
- **Timing window:** The attacker only needs to act between the source-chain `LogMetadata` event emission and the first `deploy_token` call on NEAR — a window that can be hours or days.
- **Permanent effect:** The attacker need only create the account once; they retain control indefinitely.

---

### Recommendation

Replace the unconditional `create_account()` in `TokenDeployer::deploy_token` with a check-or-skip pattern. If the account already exists and is already initialized as a valid omni-token contract (verifiable by calling `ft_metadata` or checking the contract hash), the deployer should skip account creation and proceed directly to the `new` initialization call (or treat it as already deployed). Additionally, `deploy_token_internal` in the bridge should verify that the account does not already exist before registering the token in its maps, or alternatively use a nonce/salt in the account ID derivation that is controlled by the bridge rather than being purely address-derived.

---

### Proof of Concept

1. Observe a `LogMetadata` event on Ethereum for token `0x23ddd3e3692d1861ed57ede224608875809e127f`.
2. Compute the NEAR account ID: `23ddd3e3692d1861ed57ede224608875809e127f.omnidep.near` (using `get_token_prefix` logic). [5](#0-4) 
3. Call `near account create-account` to create `23ddd3e3692d1861ed57ede224608875809e127f.omnidep.near` with ~0.1 NEAR. This is permissionless.
4. A legitimate user initiates a transfer from EVM to NEAR for this token, locking EVM tokens in `OmniBridge`.
5. Anyone calls `deploy_token` on the NEAR bridge with a valid `LogMetadata` proof.
6. `deploy_token_internal` registers the token in its maps, then calls `ext_deployer::deploy_token`. [6](#0-5) 
7. `TokenDeployer::deploy_token` calls `Promise::new(account_id).create_account()` — this fails because the account already exists. [7](#0-6) 
8. `deploy_token_by_deployer_callback` fires the `else` branch, removing the token from all bridge maps. [4](#0-3) 
9. The token is unregistered in the bridge. The attacker still holds the account. Step 5–8 repeats identically on every retry.
10. The user's EVM-locked tokens have no finalization path on NEAR and are permanently frozen.
