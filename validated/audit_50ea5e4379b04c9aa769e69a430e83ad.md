### Title
`execute_deploy` Syscall Unconditionally Reports Success on Constructor Failure, Enabling Permanent Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` (success) in the response header, regardless of whether the deployed contract's constructor actually succeeded or failed. A TODO comment in the code explicitly acknowledges this: `// TODO(Yoni, 1/1/2026): support failures.` This is the direct analog of the MasterChef "missing emergency mode" class: a missing error-recovery path that causes the protocol to silently swallow a failure state, leaving funds permanently inaccessible.

---

### Finding Description

In `execute_deploy` (the syscall implementation, not the transaction-level handler), after calling `deploy_contract`, the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `failure_flag` field in `ResponseHeader` is the mechanism by which the OS communicates syscall failure back to the calling Sierra contract. When `failure_flag=1`, the calling contract's execution is expected to handle the failure (e.g., revert, skip fund transfer, etc.). When `failure_flag=0`, the calling contract proceeds as if the deployment succeeded.

By contrast, `contract_call_helper` — used for `call_contract` and `library_call` — correctly propagates the `is_reverted` flag:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` function does call `deploy_contract` with `remaining_gas` and receives `(retdata_size, retdata)`, but the return value carries no failure indicator that is forwarded to the caller: [3](#0-2) 

The `revert_log` implicit argument is present in `execute_deploy`, meaning constructor-induced storage changes are tracked for potential reversion. However, even if the constructor's storage writes are reverted via the revert log, the calling contract is never informed of the failure — it always sees `failure_flag=0`.

---

### Impact Explanation

**Impact: Critical — Permanent Freezing of Funds.**

A calling contract that uses the `deploy` syscall to deploy a fund-holding sub-contract will proceed with its post-deploy logic (e.g., transferring tokens to the newly deployed address, recording it as a vault, granting it allowances) even when the constructor failed. Concretely:

1. Contract A calls `deploy` syscall targeting class C whose constructor fails (e.g., out-of-gas, assertion failure, or revert).
2. The OS reverts the constructor's storage writes via the revert log, leaving the deployed address with `class_hash=0` (no code) or an uninitialized state.
3. The OS writes `failure_flag=0` to the response — Contract A reads success.
4. Contract A transfers ERC-20 tokens or ETH to the "deployed" address.
5. The address has no executable code (or broken initialization). The funds are permanently locked with no recovery path.

This matches the "permanent freezing of funds" impact category exactly.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any Sierra contract that calls the `deploy` syscall and subsequently transfers value to the returned address is vulnerable. This is a common DeFi pattern (factory contracts, vault deployers, proxy deployers).
- The constructor failure can be triggered by an unprivileged caller by providing a gas amount that is sufficient to pass the `reduce_syscall_base_gas` check but insufficient for the constructor to complete — a gas-griefing vector.
- The `deploy` syscall is a standard, publicly accessible syscall available to all contracts. No privileged role is required.
- The TODO comment confirms this is a known, unresolved gap in the implementation.

---

### Recommendation

Propagate the constructor's failure status through the `deploy` syscall response, mirroring the pattern used in `contract_call_helper`:

1. Modify `deploy_contract` to return an `is_reverted` flag alongside `retdata_size` and `retdata`.
2. In `execute_deploy`, replace the hardcoded `failure_flag=0` with the actual `is_reverted` value returned by `deploy_contract`.
3. Ensure that when `is_reverted=1`, the deployed contract address is not written into the `DeployResponse` (or is written as zero), so calling contracts cannot mistakenly use it.

---

### Proof of Concept

**Setup**: Declare a class `VaultFactory` (Sierra) that:
- Calls `deploy(class_hash=BrokenVault, ...)` where `BrokenVault`'s constructor always reverts.
- On receiving the deploy response, reads `failure_flag` — which the OS writes as `0`.
- Calls `transfer(token, deployed_address, amount)` to fund the "vault."

**Execution trace**:
1. User submits an invoke transaction calling `VaultFactory.__execute__`.
2. OS processes `execute_deploy` syscall: calls `deploy_contract`, constructor reverts, revert log undoes storage writes, `class_hash` at `deployed_address` remains `0`.
3. OS writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` — hardcoded.
4. `VaultFactory` reads success, calls ERC-20 `transfer` to `deployed_address`.
5. ERC-20 balance at `deployed_address` is updated in state.
6. `deployed_address` has `class_hash=0` — no contract code. No `__execute__` entry point exists. Funds are permanently frozen.
