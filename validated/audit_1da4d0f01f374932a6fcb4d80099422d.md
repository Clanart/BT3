### Title
Hardcoded Static Gas Limit for L1 Handler Execution Causes Permanent Freezing of L1→L2 Message Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/constants.cairo`)

---

### Summary

`L1_HANDLER_L2_GAS_MAX_AMOUNT` is a compile-time constant (100,000,000 gas units) used as the sole, non-negotiable gas budget for every L1 handler execution. Unlike regular invoke transactions — which derive their gas budget from user-specified `resource_bounds` — L1 handlers carry no resource bounds at all. Any L1 handler whose execution requires more than 100M gas can never be successfully processed. The sequencer is forced to mark such transactions as reverted, which causes the L1→L2 message to remain permanently unconsumed, freezing any funds associated with it.

---

### Finding Description

**Root cause — hardcoded constant:**

In `constants.cairo` line 19, the gas budget for all L1 handler executions is fixed at compile time:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;
``` [1](#0-0) 

**Contrast with regular transactions:**

For invoke, deploy-account, and declare transactions, the gas budget is derived dynamically from the user's submitted `resource_bounds`:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [2](#0-1) 

L1 handlers have no such mechanism. Their `TxInfo` is constructed with null resource bounds:

```cairo
resource_bounds_start=cast(0, ResourceBounds*),
resource_bounds_end=cast(0, ResourceBounds*),
``` [3](#0-2) 

**Static gas assignment in `execute_l1_handler_transaction`:**

The hardcoded constant is applied unconditionally to every L1 handler:

```cairo
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [4](#0-3) 

**The revert-or-proof-fail dilemma:**

`non_reverting_select_execute_entry_point_func` internally asserts `is_reverted = 0`. If the L1 handler runs out of gas and reverts, this assertion fails, making the entire block proof invalid. The sequencer therefore has no choice but to pre-mark any gas-exceeding L1 handler as reverted via the `IsReverted` hint.

When `is_reverted != FALSE`, the function returns early at lines 387–390, **before** `consume_l1_to_l2_message` is called at line 444:

```cairo
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();   // ← returns here; message is never consumed
}
// ...
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);  // line 444
``` [5](#0-4) [6](#0-5) 

The L1→L2 message is never written to the OS output, so it is never acknowledged as processed. It remains permanently pending on L1 with no mechanism for retry or cancellation at the OS level.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any L1→L2 message whose handler requires more than 100M L2 gas units can never be executed. The L1 message fee (ETH paid on L1) is unrecoverable, and any L2-side action the message was intended to trigger (e.g., token minting, bridge credit) never occurs. The funds are permanently frozen. The static constant cannot be adjusted without a full protocol upgrade and OS re-deployment.

For reference, `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` for regular transactions — 11× higher than the L1 handler limit — confirming that complex operations routinely require more than 100M gas. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** An unprivileged L1 message sender triggers this by sending a message to any deployed StarkNet contract whose `@l1_handler` function performs gas-intensive work. At `STORAGE_WRITE_GAS_COST = 44,970` gas per write, only ~2,226 storage writes exhaust the 100M budget. DeFi bridge contracts, NFT minting handlers, or any handler that iterates over attacker-supplied calldata arrays are realistic targets. The attacker controls the calldata length, directly influencing gas consumption. [8](#0-7) 

---

### Recommendation

1. Allow L1 messages to carry an explicit L2 gas limit (paid for by the L1 fee), analogous to how invoke transactions carry `resource_bounds[L2_GAS_INDEX].max_amount`.
2. At minimum, align `L1_HANDLER_L2_GAS_MAX_AMOUNT` with `EXECUTE_MAX_SIERRA_GAS` (1,100,000,000) to reduce the gap between L1 handler and regular transaction gas budgets.
3. Introduce a per-block configurable parameter for the L1 handler gas cap so it can be adjusted via governance without a full OS upgrade.

---

### Proof of Concept

1. Deploy a StarkNet contract with an `@l1_handler` that iterates over its calldata and performs one `STORAGE_WRITE` per element (44,970 gas each).
2. From L1, send a message to this contract with a calldata array of length ≥ 2,227 (≥ 100,000,890 gas required).
3. The sequencer simulates the handler, finds it exceeds `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100,000,000`.
4. To avoid a proof-breaking assertion failure in `non_reverting_select_execute_entry_point_func`, the sequencer sets `IsReverted = TRUE`.
5. `execute_l1_handler_transaction` returns at line 389, before `consume_l1_to_l2_message` at line 444.
6. The L1→L2 message is never consumed. The ETH fee paid on L1 is unrecoverable. Any tokens or credits the handler was supposed to mint on L2 are never created. Funds are permanently frozen.
