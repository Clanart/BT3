### Title
Excess ETH Permanently Stuck in Wormhole Contract When Overpaying for `publishMessage` Fee - (File: `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole` forwards the entire `msg.value` to `_wormhole.publishMessage{value: msg.value}(...)` in three extension functions (`deployTokenExtension`, `logMetadataExtension`, `finTransferExtension`). Wormhole's `publishMessage` only consumes exactly `messageFee()` worth of ETH. Any ETH sent beyond that amount is permanently stuck in the Wormhole contract with no refund mechanism.

---

### Finding Description

`OmniBridgeWormhole` overrides three internal extension hooks and in each one passes the full `msg.value` to Wormhole's `publishMessage`:

**`deployTokenExtension`** — called from the `payable` function `deployToken`: [1](#0-0) 

**`logMetadataExtension`** — called from the `payable` functions `logMetadata` and `logMetadata1155`: [2](#0-1) 

**`finTransferExtension`** — called from the `payable` function `finTransfer`: [3](#0-2) 

The Wormhole interface exposes `messageFee()` to query the exact required fee: [4](#0-3) 

Wormhole's `publishMessage` only consumes exactly `messageFee()` ETH. Any ETH sent above that amount is retained by the Wormhole contract and is not refunded to the caller. None of the three affected functions contain a refund path or compute the exact fee before forwarding.

By contrast, `initTransferExtension` correctly avoids this problem — it receives a pre-computed `value` (i.e., `msg.value - nativeFee`) and forwards only that: [5](#0-4) 

The parent `initTransfer` computes `extensionValue = msg.value - nativeFee` before passing it down: [6](#0-5) 

This demonstrates the protocol already knows the correct pattern — it is simply not applied to the other three extension functions.

---

### Impact Explanation

Any ETH sent in excess of `_wormhole.messageFee()` when calling `deployToken`, `logMetadata`, `logMetadata1155`, or `finTransfer` is permanently locked in the Wormhole contract. There is no sweep, refund, or recovery mechanism. This constitutes an **irrecoverable lock of user funds** matching the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

- `messageFee()` can change between the time a user queries it off-chain and the time their transaction is mined, causing unintentional overpayment.
- Users and integrators commonly send a small ETH buffer above the expected fee to avoid reverts, which is standard practice.
- `deployToken`, `logMetadata`, and `logMetadata1155` are callable by any unprivileged user (no role check on `logMetadata`/`logMetadata1155`; `deployToken` is open to anyone with a valid MPC signature).
- `finTransfer` is callable by any relayer or user submitting a cross-chain proof.

---

### Recommendation

Replace `msg.value` with `_wormhole.messageFee()` in the three affected extension functions, and refund any excess ETH to `msg.sender`:

```solidity
// In deployTokenExtension, logMetadataExtension, finTransferExtension:
uint256 fee = _wormhole.messageFee();
_wormhole.publishMessage{value: fee}(wormholeNonce, payload, _consistencyLevel);
wormholeNonce++;

// Refund excess
uint256 excess = msg.value - fee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "ETH refund failed");
}
```

This mirrors the correct pattern already used in `initTransferExtension`.

---

### Proof of Concept

1. `_wormhole.messageFee()` returns `0.001 ETH`.
2. A user calls `logMetadata(tokenAddress)` and sends `0.002 ETH` as a buffer.
3. `logMetadataExtension` executes `_wormhole.publishMessage{value: 0.002 ETH}(...)`.
4. Wormhole consumes `0.001 ETH` for the fee; the remaining `0.001 ETH` is retained by the Wormhole contract.
5. No refund is issued. The `0.001 ETH` excess is permanently lost.

The same scenario applies identically to `deployToken` and `finTransfer`.
