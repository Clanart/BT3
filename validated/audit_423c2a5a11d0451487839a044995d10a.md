### Title
`batch_insert` Bypasses Duplicate-Key/Hash Guards, Enabling DataLayer Merkle Tree Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces. An attacker who can supply duplicate `KeyId` or `Hash` values through the DataLayer sync path can silently corrupt the Merkle tree blob and its in-memory cache, producing an incorrect root hash and invalidating all subsequent proofs of inclusion.

---

### Finding Description

`MerkleBlob::insert` enforces two guards before writing any leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` is the performance-optimised bulk path. It calls `self.insert()` (with guards) for at most the last two items when the tree has ≤ 1 existing leaves, then writes every remaining item directly through `insert_entry_to_blob` — with **no duplicate check at all**:

```rust
// lines 578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // ← guarded
    }
}
for ((key, value), hash) in keys_values_hashes {          // ← ALL remaining items
    let new_leaf_index = self.get_new_index();
    // ... builds Block directly ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?; // ← NO duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When `leaf_count > 1`, **every** item in the batch bypasses the guards. `insert_entry_to_blob` calls `block_status_cache.add_leaf`, which silently overwrites the cache entry for a duplicate key:

```rust
// lines 188-192
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index); // overwrites silently
}
``` [3](#0-2) 

The result is a split state: the blob contains two physical leaf nodes for the same key, but the cache tracks only the last one. The orphaned node is still wired into the tree's parent-child graph, so the root hash computed from the tree is wrong, and `check_integrity` will detect a mismatch between `leaf_count` and `key_to_index_cache_length`. [4](#0-3) 

The Python binding exposes `batch_insert` directly:

```python
def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...
``` [5](#0-4) 

---

### Impact Explanation

A corrupted `MerkleBlob` produces an incorrect root hash. Any `ProofOfInclusion` generated after the corruption will fail to verify against the true root, and a proof for a key that was inserted twice will traverse the wrong lineage. This directly matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

---

### Likelihood Explanation

The DataLayer sync path receives `(KeyId, ValueId, Hash)` tuples from external peers and feeds them to `batch_insert`. A malicious or faulty peer that sends a batch containing a repeated `KeyId` or `Hash` will trigger the corruption silently — no error is returned, and the caller has no indication that the tree is now invalid. The Python binding makes this reachable from any code that calls `MerkleBlob.batch_insert` with attacker-influenced data.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards at the top of `batch_insert`, mirroring `insert`:

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Guard: reject duplicates within the batch and against existing leaves
    for ((key, _value), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // ... rest of existing logic
```

Alternatively, deduplicate the input vector before processing, or promote the per-item check into the fast path loop so that each item is validated before `insert_entry_to_blob` is called.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};

fn main() {
    let mut blob = MerkleBlob::new(Vec::new()).unwrap();

    // Pre-populate with 2 leaves so leaf_count > 1 → ALL batch items bypass guards
    blob.insert(KeyId(1), ValueId(1), &[0x01u8; 32], InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(2), ValueId(2), &[0x02u8; 32], InsertLocation::Auto {}).unwrap();

    // batch_insert with a duplicate KeyId(3) — no error returned
    let dup_hash_a: Hash = [0x03u8; 32];
    let dup_hash_b: Hash = [0x04u8; 32];
    blob.batch_insert(vec![
        ((KeyId(3), ValueId(3)), dup_hash_a),
        ((KeyId(3), ValueId(99)), dup_hash_b), // same key, different value/hash
    ]).unwrap(); // succeeds — no KeyAlreadyPresent error

    // Tree is now corrupted: blob has two leaf nodes for KeyId(3),
    // cache tracks only the last one, root hash is wrong.
    blob.check_integrity().unwrap_err(); // IntegrityLeafHashToIndexCacheLength or similar
}
```
