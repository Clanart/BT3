### Title
Empty `check_is_reverted` Leaves `is_reverted` Flag Unconstrained, Enabling Prover to Skip Execution While Charging Fees — (File: `execution/execution_constraints.cairo`)

---

### Summary

The function `check_is_reverted` in `execution_constraints.cairo` is a complete no-op. It accepts the hint-supplied `is_reverted` flag but imposes zero proof constraints on it. In `execute_invoke_function_transaction`, this flag gates whether the transaction's `__execute__` entry point is called at all — yet because the check is empty, a malicious prover can freely set `is_reverted = TRUE` for any invoke transaction, skip its execution, and still charge the user fees, while producing a proof that the L1 verifier accepts as valid.

---

### Finding Description

**Root cause — the empty constraint function:** [1](#0-0) 

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

The function signature implies it should validate the prover-supplied `is_reverted` value, but the body is empty — no assertion, no range check, no consistency check against the actual execution outcome.

**How `is_reverted` is used in `execute_invoke_function_transaction`:** [2](#0-1) 

```cairo
local is_reverted;
%{ IsReverted %}                      // ← hint: prover supplies any felt
check_is_reverted(is_reverted);       // ← no-op; imposes no constraint
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(...);
    }
} else {
    // Execution is skipped entirely.
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}

// Fee is charged regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
```

The `is_reverted` value is loaded from a hint (`%{ IsReverted %}`) and is the sole determinant of whether `__execute__` is called. Because `check_is_reverted` does nothing, the Cairo proof system places **no algebraic constraint** on this value. The prover can set it to any non-zero felt for any transaction.

**Contrast with `execute_l1_handler_transaction`**, which also reads `is_reverted` from a hint with no constraint: [3](#0-2) 

The same unconstrained pattern appears there, though L1 handlers do not charge fees, so the financial impact is different.

**The fee is always charged** regardless of the revert flag: [4](#0-3) 

`charge_fee` executes an ERC-20 transfer from the user's account to the sequencer unconditionally after the `if/else` block.

---

### Impact Explanation

**Direct loss of funds (Critical).**

A malicious prover (sequencer) can, for any invoke transaction:

1. Set `is_reverted = TRUE` in the hint.
2. Skip the `__execute__` call entirely — no state changes are applied.
3. Still execute `charge_fee`, deducting the actual fee from the user's account.
4. Produce a proof that is **cryptographically valid** because `check_is_reverted` imposes no constraint.
5. Submit the proof to L1; the verifier accepts it.

The user pays fees for a transaction whose effects were silently discarded. Because the proof is valid, there is no on-chain mechanism to detect or dispute this. The sequencer can selectively apply this to any or all invoke transactions in a block, draining user balances while producing no observable execution.

---

### Likelihood Explanation

The StarkNet ZK-proof model is explicitly designed so that the sequencer does **not** need to be trusted — the proof is supposed to enforce correctness. The `check_is_reverted` no-op is a soundness gap: it is the one place where the proof system should bind the hint-supplied revert flag to the actual execution outcome but does not. Any sequencer operator who discovers this gap can exploit it silently and at scale, with no special key material or external dependency required beyond operating the sequencer.

---

### Recommendation

Implement `check_is_reverted` to enforce that the hint-supplied `is_reverted` value is consistent with the actual execution result. Concretely, the function should assert that `is_reverted` is a boolean (0 or 1) and, more importantly, that it matches the `failure_flag` returned by the entry-point execution. One approach is to restructure the flow so that `is_reverted` is derived deterministically from the entry-point return values rather than being supplied as a free hint before execution.

---

### Proof of Concept

1. User submits an invoke transaction with a valid signature and sufficient gas.
2. Malicious sequencer sets `is_reverted = 1` in the `%{ IsReverted %}` hint for that transaction.
3. `check_is_reverted(1)` is called — returns immediately, no constraint added.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
5. `charge_fee` executes: an ERC-20 transfer moves the fee from the user to the sequencer.
6. The OS generates a proof over this execution trace. The proof is valid because no constraint was violated.
7. The proof is submitted to and accepted by the L1 verifier.
8. The resulting state diff shows the fee deducted from the user but none of the intended `__execute__` state changes applied.
9. The user has lost their fee with no recourse, and the transaction's intended effects are permanently absent from the chain.
