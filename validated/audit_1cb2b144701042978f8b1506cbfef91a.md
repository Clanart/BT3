### Title
`OmniBridge.logMetadata()` Reverts for ERC20 Tokens with Non-String `name()`/`symbol()`, Permanently Blocking Their Bridging - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.logMetadata()` is the public, unpermissioned entry point that must be called to register an EVM token for bridging to NEAR. It calls `IERC20Metadata(tokenAddress).name()`, `.symbol()`, and `.decimals()` with no try-catch or fallback. For tokens like MKR that return `bytes32` instead of `string` from `name()`/`symbol()`, the ABI decode will revert unconditionally. Because `initTransfer()` has no corresponding registration guard, a user can lock tokens in the bridge before discovering that `logMetadata()` can never succeed, resulting in permanently frozen funds.

---

### Finding Description

`logMetadata` is defined as:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name    = IERC20Metadata(tokenAddress).name();    // line 225
    string memory symbol  = IERC20Metadata(tokenAddress).symbol();  // line 226
    uint8  decimals       = IERC20Metadata(tokenAddress).decimals(); // line 227
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

`IERC20Metadata` is the OpenZeppelin interface that expects `name()` and `symbol()` to return `string`. Tokens such as MKR (`0x9f8f72aa9304c8b593d555f12ef6589cc3a579a2`) return `bytes32` from both functions. When Solidity's ABI decoder tries to decode a `bytes32` return value as a `string`, it reverts because the length prefix is missing. There is no try-catch, no fallback, and no alternative path.

`initTransfer`, on the other hand, has no registration check:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [2](#0-1) 

A user can call `initTransfer` with MKR, successfully transferring tokens into the bridge contract, while `logMetadata` for MKR will always revert. The NEAR side never receives the `LogMetadata` event required to deploy the corresponding NEAR token, so the `InitTransfer` event cannot be finalized. The tokens are permanently locked with no rescue path in the contract.

---

### Impact Explanation

**Critical — Permanent freezing of user funds in the bridge vault.**

Any user who calls `initTransfer` with a `bytes32`-metadata token (MKR being the canonical example) will have their tokens transferred into `OmniBridge` with no possibility of recovery:

- `logMetadata` always reverts for such tokens → NEAR never deploys the corresponding token.
- `initTransfer` succeeds and locks the tokens → `InitTransfer` event is emitted but can never be finalized on NEAR.
- There is no `rescue`, `withdraw`, or admin recovery function in `OmniBridge` for locked ERC-20 balances. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** MKR is one of the most liquid DeFi tokens and is explicitly listed as compatible with Uniswap V3 (the same compatibility criterion cited in the analogous Teller report). Any user who attempts to bridge MKR or any other `bytes32`-metadata token will trigger this path. The `initTransfer` function imposes no prerequisite check, so the fund-freeze scenario requires only two sequential user transactions with no privileged access.

---

### Recommendation

Replace the direct `IERC20Metadata` calls in `logMetadata` with try-catch fallbacks, mirroring the pattern already used in the StarkNet bridge for the analogous `felt252`/`ByteArray` dual-standard case:

```solidity
function _safeTokenName(address token) internal view returns (string memory) {
    try IERC20Metadata(token).name() returns (string memory n) {
        return n;
    } catch {
        return "";
    }
}

function _safeTokenSymbol(address token) internal view returns (string memory) {
    try IERC20Metadata(token).symbol() returns (string memory s) {
        return s;
    } catch {
        return "";
    }
}

function _safeTokenDecimals(address token) internal view returns (uint8) {
    try IERC20Metadata(token).decimals() returns (uint8 d) {
        return d;
    } catch {
        return 18; // safe default
    }
}
```

Apply the same pattern to `addCustomToken` (lines 99–101), which has the same flaw but is admin-gated. [4](#0-3) 

The StarkNet bridge already handles the analogous dual-standard problem correctly by branching on return-data length: [5](#0-4) 

---

### Proof of Concept

1. Deploy or reference MKR on a network where `OmniBridge` is live.
2. Call `logMetadata(MKR_ADDRESS)` → transaction reverts with ABI decode error because MKR's `name()` returns `bytes32`.
3. Call `initTransfer(MKR_ADDRESS, 1e18, 0, 0, "alice.near", "")` with sufficient MKR approval → transaction **succeeds**, transferring 1 MKR into `OmniBridge`.
4. Observe that no `LogMetadata` event was ever emitted for MKR → NEAR side has no corresponding token → `InitTransfer` event cannot be finalized → 1 MKR is permanently locked.
