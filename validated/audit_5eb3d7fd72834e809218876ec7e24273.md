### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay for `meta_tx_v0` Syscall — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function omits any nonce or replay-prevention field from the signed hash. Combined with the OS hardcoding `nonce=0` in the resulting `TxInfo`, a valid meta-tx v0 signature can be replayed an unlimited number of times by any unprivileged contract caller, potentially draining funds from any account contract that accepts meta-tx v0 calls.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // <-- no nonce
        additional_data=cast(0, felt*),  // <-- no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The resulting hash covers only: `INVOKE_HASH_PREFIX || version=0 || contract_address || selector || H(calldata) || max_fee=0 || chain_id`. There is no nonce, block number, timestamp, or caller address.

In `execute_meta_tx_v0` (`syscall_impls.cairo`), the OS then constructs the `TxInfo` with `nonce=0` hardcoded:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    account_contract_address=contract_address,
    ...
    transaction_hash=meta_tx_hash,
    nonce=0,   // hardcoded — no replay protection
    ...
);
``` [2](#0-1) 

Additionally, `check_and_increment_nonce` explicitly skips nonce enforcement for version-0 transactions:

```cairo
// Do not handle nonce for version 0.
if (tx_info.version == 0) {
    return ();
}
``` [3](#0-2) 

And `run_validate` skips `__validate__` entirely for version-0 transactions:

```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
``` [4](#0-3) 

The combination means: the OS computes a deterministic, nonce-free hash; the target contract's `__validate__` is never called; and the nonce in `TxInfo` is always 0. A target account contract that signs a meta-tx v0 payload cannot prevent that signature from being replayed, because the hash it signed will be identical on every replay.

---

### Impact Explanation

**Direct loss of funds (Critical).**

Any account contract that accepts meta-tx v0 calls (e.g., a gasless-transaction relayer pattern) signs over `H(prefix, 0, contract_address, selector, H(calldata), 0, chain_id)`. Because this hash is static for a fixed `(contract_address, calldata)` pair, an attacker who observes one valid meta-tx v0 on-chain can replay it indefinitely. If the `__execute__` entry point of the target contract transfers tokens or ETH based on the calldata (e.g., a transfer call), the attacker can drain the contract's funds by replaying the same signed payload repeatedly through the `meta_tx_v0` syscall from their own deployed contract.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is callable by any contract executing on StarkNet — no privileged role is required. The attacker only needs to:
1. Observe a legitimate `meta_tx_v0` invocation on-chain (public mempool/block data).
2. Deploy a contract that calls `meta_tx_v0` with the same `contract_address`, `calldata`, and `signature`.

The OS will compute the identical hash, the target contract's `__validate__` is skipped, and `__execute__` runs again. This is a fully deterministic, permissionless attack path.

---

### Recommendation

Include a nonce (or block-number-bound replay token) in `compute_meta_tx_v0_hash` via the `additional_data` parameter of `deprecated_get_transaction_hash`. The nonce should be sourced from the target contract's on-chain state and incremented after each successful meta-tx v0 execution, analogous to how `check_and_increment_nonce` works for version ≥ 1 transactions. Additionally, the caller contract's address should be bound into the hash to prevent cross-caller replay.

---

### Proof of Concept

1. Alice's account contract signs a meta-tx v0 payload: `(contract_address=Alice, selector=__execute__, calldata=[transfer Bob 100 tokens])`. The OS computes hash `H = H(INVOKE_HASH_PREFIX, 0, Alice, __execute__, H(calldata), 0, chain_id)`.

2. Alice submits a transaction that calls `meta_tx_v0` with this signature. Bob receives 100 tokens. Alice's account balance decreases.

3. Attacker Eve deploys `EvilContract` with a single function that calls the `meta_tx_v0` syscall using the same `contract_address=Alice`, same `calldata`, and same `signature` observed from step 2.

4. Eve calls `EvilContract`. The OS recomputes `H` — identical to step 1 because `compute_meta_tx_v0_hash` takes no nonce. `__validate__` is skipped (version 0). Alice's `__execute__` runs again, transferring another 100 tokens to Bob.

5. Eve repeats step 4 until Alice's account is drained.

The root cause is in `compute_meta_tx_v0_hash` at `transaction_hash.cairo` lines 294–315 (`additional_data_size=0`) and confirmed by `syscall_impls.cairo` lines 351 (`nonce=0`) and `execute_transaction_utils.cairo` lines 63–67 (nonce check skipped for version 0). [5](#0-4) [6](#0-5) [3](#0-2)
