### Title
Missing Validation of `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the supplied `class_hash` is non-zero or corresponds to a previously declared class. An unprivileged contract can call `replace_class(0)` (or any undeclared class hash), permanently making itself uncallable and freezing any funds held within it. This is the direct analog of the `maxLeverageFactor = 0` bug: a critical protocol value is accepted without a lower-bound or existence check.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The inline TODO comment explicitly acknowledges the missing check. The same omission exists in the deprecated path in `deprecated_execute_syscalls.cairo`.

When a subsequent call is made to a contract whose `class_hash` is 0 (or any undeclared hash), `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // == 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // == 0, not in the bundle
);
```

`find_element` is a Cairo primitive that causes a prover-level failure (not a graceful revert) when the key is absent. The contract becomes permanently uncallable at the OS level.

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

Any ERC-20 tokens, ETH, or other assets held in a contract's storage become permanently inaccessible once its `class_hash` is replaced with 0 or any undeclared hash. The state transition is committed to the OS output and cannot be reversed. No recovery path exists because the contract can no longer execute any entry point (including any hypothetical recovery function).

---

### Likelihood Explanation

**High.** The `replace_class` syscall is callable by any contract without privilege. A malicious actor can:
1. Deploy a contract that holds user funds (e.g., a vault or escrow).
2. Invoke `replace_class(0)` from within that contract.
3. The OS accepts the call, commits `class_hash = 0` to state, and the contract is permanently bricked.

No operator key, leaked secret, or network-level attack is required. The entry path is a standard user-submitted transaction.

---

### Recommendation

Add a non-zero check and a declared-class existence check before updating the state entry in `execute_replace_class`:

1. **Non-zero check** (analogous to the `_maxLeverageFactor > 0` recommendation): assert `class_hash != 0`.
2. **Declared-class check**: verify that `class_hash` maps to a known compiled class hash in `contract_class_changes` (i.e., `dict_read` returns a non-zero value). The TODO comment at line 898 already identifies this requirement.

Apply the same fix to the deprecated path in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Deploy contract `Vault` that accepts deposits and exposes a `replace_class` call path.
2. Users deposit funds into `Vault`.
3. Attacker (or malicious `Vault` logic) calls the `replace_class` syscall with `class_hash = 0`.
4. The OS executes `execute_replace_class` — no validation fires, `dict_update` commits `StateEntry(class_hash=0, ...)` for `Vault`'s address.
5. All subsequent calls to `Vault` reach `execute_entry_point`, which calls `dict_read` for key `0` on `contract_class_changes` (returns 0), then calls `find_element` with key `0` — element not found, prover failure.
6. `Vault` is permanently uncallable; all deposited funds are frozen.
