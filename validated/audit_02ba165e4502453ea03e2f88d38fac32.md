### Title
Missing Fund Recovery Mechanism for Stuck ETH and ERC20 Tokens Enables Permanent Irrecoverable Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol` accepts ETH via an open `receive()` function and holds native ERC20 tokens in custody for bridge flows, but provides no admin recovery function to retrieve accidentally sent or force-injected funds. Any ETH or ERC20 tokens that arrive at the contract outside of normal bridge flows are permanently irrecoverable, with no `farm()` or equivalent sweep mechanism.

---

### Finding Description

The EVM `OmniBridge` contract has two surfaces through which funds can become permanently stuck:

**1. ETH — open `receive()` with no recovery path**

The contract declares an unrestricted ETH receiver: [1](#0-0) 

This is intentional for the native-ETH bridge flow: `initTransfer` with `tokenAddress == address(0)` deposits ETH into the contract via `msg.value`, and `finTransfer` later releases it to the recipient: [2](#0-1) 

However, because `receive()` is completely open, an attacker can force-inject arbitrary ETH into the contract using an intermediate contract that calls `selfdestruct(omniBridgeAddress)`. The injected ETH accumulates in the contract's balance with no way for any admin to retrieve it. There is no `farm()`, `sweep()`, or `rescueETH()` function anywhere in the contract.

**2. ERC20 tokens — custody without recovery**

For non-bridge, non-custom ERC20 tokens, `initTransfer` pulls tokens into the contract: [3](#0-2) 

A user who accidentally calls `token.transfer(omniBridgeAddress, amount)` directly (instead of going through `initTransfer`) deposits tokens into the contract with no corresponding bridge record. Because there is no recovery function, those tokens are permanently locked. The same applies to any ERC20 token sent directly to the contract address by mistake.

**3. No recovery function exists**

The entire contract has no admin-callable function to retrieve stuck ETH or ERC20 tokens. The only admin functions are `pause`, `upgradeToken`, `setNearBridgeDerivedAddress`, and `_authorizeUpgrade`: [4](#0-3) 

This contrasts with the pattern described in the external report, where a `farm()` function was added to allow the owner to retrieve stray balances.

---

### Impact Explanation

Any ETH or ERC20 tokens that arrive at `OmniBridge` outside of a legitimate `initTransfer` call are permanently irrecoverable. This includes:

- ETH force-injected via `selfdestruct` (no user action required beyond deploying an intermediate contract)
- ERC20 tokens sent directly to the contract address by a user who misaddresses a transfer

The funds are locked in the contract forever with no admin escape hatch. This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock of user or protocol funds in bridge flows.**

---

### Likelihood Explanation

- **User error (ERC20)**: Sending tokens directly to a contract address instead of calling `initTransfer` is a well-documented and common user mistake, especially for users interacting with the contract directly rather than through a frontend.
- **Selfdestruct injection (ETH)**: Requires deploying a small intermediate contract and self-destructing it targeting `OmniBridge`. This is a low-cost, permissionless operation available to any attacker.

Neither path requires any privileged access, leaked keys, or colluding parties.

---

### Recommendation

Add an admin-only recovery function analogous to the `farm()` function described in the external report:

```solidity
function rescueETH(address payable to) external onlyRole(DEFAULT_ADMIN_ROLE) {
    (bool success, ) = to.call{value: address(this).balance}("");
    require(success, "ETH rescue failed");
}

function rescueERC20(address token, address to) external onlyRole(DEFAULT_ADMIN_ROLE) {
    // Optionally restrict to tokens not actively held for bridge flows
    IERC20(token).safeTransfer(to, IERC20(token).balanceOf(address(this)));
}
```

For the ERC20 rescue, consider restricting it to tokens that are not currently expected to be held (i.e., not tokens with pending `finTransfer` payloads), to avoid the admin accidentally sweeping legitimate bridge collateral.

---

### Proof of Concept

**ETH force-injection:**

```solidity
contract ForceETH {
    constructor(address payable target) payable {
        selfdestruct(target); // forces ETH into OmniBridge.receive()
    }
}
// Deploy with 1 ETH:
new ForceETH{value: 1 ether}(omniBridgeAddress);
// Result: 1 ETH is now permanently stuck in OmniBridge with no recovery path.
```

**ERC20 direct send:**

```solidity
// User mistakenly calls transfer instead of approve+initTransfer:
IERC20(usdc).transfer(omniBridgeAddress, 1000e6);
// Result: 1000 USDC is now permanently stuck in OmniBridge.
// No admin function exists to retrieve it.
```

Both scenarios are confirmed by the absence of any recovery function in `OmniBridge.sol`. [1](#0-0) [4](#0-3)
