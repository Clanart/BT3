### Title
Missing Chain ID in `deployToken` Signature Domain Enables Cross-Chain Replay of NEAR MPC Signatures — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `near/omni-types/src/lib.rs`)

---

### Summary

The `deployToken` message hash on every destination chain (EVM, Starknet, Solana) omits a chain identifier from the signed payload. Because NEAR MPC signs the same chain-agnostic borsh blob and the same derived address is used across all chains, a valid `deployToken` signature obtained for one chain can be replayed verbatim on any other supported chain. This is the direct analog of the BeaconKit "active fork version" flaw: a domain-separation value that should bind the signature to a specific context is absent, making the signature universally valid across contexts.

---

### Finding Description

**NEAR signing side** (`near/omni-types/src/lib.rs`, `near/omni-bridge/src/lib.rs`):

`log_metadata_callback` constructs a `MetadataPayload` and signs `keccak256(borsh(MetadataPayload))`:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
```

No chain ID is included. [1](#0-0) 

**EVM verification** (`evm/src/omni-bridge/contracts/OmniBridge.sol`):

```solidity

    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),

bytes32 hashed = keccak256(borshEncoded);

    revert InvalidSignature();

```

No `omniBridgeChainId` is included. [2](#0-1) 

**Starknet verification** (`starknet/src/bridge_types.cairo`):

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    let mut borsh_bytes: ByteArray = Default::default();
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

No `omni_bridge_chain_id` is included. [3](#0-2) 

**Contrast with `finTransfer`**, which correctly binds the signature to the chain:

EVM `finTransfer` includes `omniBridgeChainId` twice (for token chain and recipient chain): [4](#0-3) 

Starknet `fin_transfer` passes `self.omni_bridge_chain_id.read()` into `to_borsh`: [5](#0-4) 

The `deployToken` path is the only message type that lacks this binding. The NEAR MPC signing path (`SIGN_PATH = "bridge-1"`) is a single fixed derivation path, producing the same `nearBridgeDerivedAddress` / `omni_bridge_derived_address` on every chain. [6](#0-5) 

---

### Impact Explanation

**High** — Proof/signature verification bypass enabling unauthorized token deployment.

An attacker who observes a `LogMetadataEvent` on NEAR (which contains the MPC signature) can submit that exact signature to any other supported chain before the legitimate relayer does. On the target chain, `deployToken` will:

1. Accept the signature as valid (same borsh payload, same signer address).
2. Deploy the bridge token contract and register it in `nearToEthToken` / `near_to_starknet_token`.
3. Set `isBridgeToken[proxy] = true`.

After this replay, the NEAR bridge has no record of the token address on the target chain (it was never registered via the normal `fin_transfer` flow). When the legitimate relayer later attempts to deploy the same token on that chain, it reverts with `ERR_TOKEN_EXIST`. The token is now permanently unregisterable through the normal protocol flow on that chain, blocking all future bridging of that asset to that destination. Users who attempt to bridge to the target chain will have their transfers stuck on NEAR with no valid destination token address.

---

### Likelihood Explanation

**Medium** — The `LogMetadataEvent` containing the MPC signature is emitted publicly on NEAR and is observable by any party. An attacker only needs to:

1. Monitor NEAR for `LogMetadataEvent` emissions.
2. Identify a target chain where the token has not yet been deployed.
3. Submit the signature to the target chain's bridge contract before the legitimate relayer.

No special privileges are required. The attack is a simple front-run of a publicly broadcast signature. The window is bounded only by block confirmation times across chains.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` borsh encoding on both the NEAR signing side and all destination-chain verification sides, mirroring the pattern already used in `finTransfer`/`fin_transfer`. Specifically:

- Add a `chain_id: u8` field to `MetadataPayload` on NEAR and encode it in `borsh::to_vec`.
- Add `bytes1(omniBridgeChainId)` to the EVM `deployToken` borsh blob.
- Pass and encode `omni_bridge_chain_id` in Starknet's `MetadataPayload::to_borsh()`.
- Add `SOLANA_OMNI_BRIDGE_CHAIN_ID` to Solana's `DeployTokenPayload::serialize_for_near`.

---

### Proof of Concept

1. NEAR emits `LogMetadataEvent` for token `"usdc.near"` with MPC signature `S` targeting EVM chain A (Ethereum). The event is public.

2. Attacker extracts `S` and the `MetadataPayload` from the NEAR event log.

3. Attacker calls `deployToken(S, {token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6})` on EVM chain B (Arbitrum) before the legitimate relayer.

4. EVM chain B computes `keccak256(borsh(MetadataPayload))` — identical to what NEAR signed, because no chain ID is included. `ECDSA.recover` returns `nearBridgeDerivedAddress`. Signature accepted.

5. Token `"usdc.near"` is deployed on Arbitrum and registered in `nearToEthToken`. [7](#0-6) 

6. NEAR's bridge has no record of the Arbitrum token address. When the legitimate relayer later calls `deployToken` on Arbitrum, it reverts: `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")`. [8](#0-7) 

7. `"usdc.near"` is permanently undeployable on Arbitrum through the normal protocol flow. All user transfers targeting Arbitrum for this token are permanently stuck.
