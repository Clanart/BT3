### Title
Cross-Chain Replay of Metadata Signature Enables Unauthorized Token Deployment on Unintended EVM Chains — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-bridge/src/lib.rs`)

---

### Summary

The `deployToken` metadata digest does not include the destination chain identifier (`omniBridgeChainId`). Because the NEAR MPC signer uses a single fixed signing path for all chains, the derived signer address (`nearBridgeDerivedAddress`) is identical across every EVM deployment. A valid metadata signature produced for one EVM chain (e.g., Ethereum) can therefore be replayed verbatim on any other EVM chain (Arbitrum, Base, BNB, Polygon, HyperEvm, Abstract) to deploy the same token there without authorization.

---

### Finding Description

**Root cause — missing chain ID in the metadata digest**

In `OmniBridge.sol`, `deployToken` constructs its digest as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The field `omniBridgeChainId` — which uniquely identifies the destination chain — is **absent** from this digest. Compare this to `finTransfer`, which correctly embeds `omniBridgeChainId` twice (once for the token chain, once for the recipient chain):

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

**Root cause — single MPC signing path for all chains**

On the NEAR side, `log_metadata_callback` constructs the `MetadataPayload` with no chain field and signs it via the MPC signer using the hardcoded path `"bridge-1"`:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
ext_signer::ext(self.mpc_signer.clone())
    .sign(SignRequest { payload, path: SIGN_PATH.to_owned(), key_version: 0 })
``` [3](#0-2) 

`SIGN_PATH` is the constant `"bridge-1"` used for every chain: [4](#0-3) 

Because the MPC key derivation path is identical for all EVM chains, the resulting `nearBridgeDerivedAddress` is the same on Ethereum, Arbitrum, Base, BNB, Polygon, HyperEvm, and Abstract. A signature that passes `ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress` on Ethereum will pass the identical check on every other EVM chain.

**The `MetadataPayload` struct itself has no chain field:**

```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
``` [5](#0-4) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` call on Ethereum (signature is public on-chain) can immediately replay it on Arbitrum, Base, BNB, Polygon, HyperEvm, or Abstract before the NEAR bridge team performs the official deployment there. Consequences:

1. **Unauthorized token deployment**: The token is deployed on a chain the NEAR bridge has not yet registered it for. The NEAR bridge's `token_address_to_id` / `token_id_to_address` mappings do not contain the new chain's address, so the NEAR bridge cannot process transfers involving it.

2. **Permanent front-running / DoS of legitimate deployment**: Once the attacker's replay succeeds, `nearToEthToken[metadata.token]` is populated on the target chain. Any subsequent legitimate `deployToken` call for the same NEAR token ID reverts with `"ERR_TOKEN_EXIST"`. [6](#0-5) 

3. **Permanent freezing of user funds**: Because `isBridgeToken[attackerDeployedToken]` is `true` on the target chain, a user who calls `initTransfer` with that token address will have their tokens burned by the bridge contract. However, since the NEAR bridge has no record of this token on that chain, no relayer will process the transfer and no funds will be released on NEAR — resulting in an irrecoverable loss. [7](#0-6) 

This matches the allowed impact: **High — signature verification bypass enabling unauthorized token deployment**, and potentially **Critical — permanent freezing of user funds**.

---

### Likelihood Explanation

- The metadata signature is emitted as a public on-chain event the moment any token is deployed on any EVM chain.
- The attacker needs no privileged access, no leaked key, and no colluding party — only the ability to submit a transaction on another EVM chain.
- The Omni Bridge is already deployed across multiple EVM chains sharing the same MPC-derived signer, making the preconditions immediately satisfied for every token deployed on any one of them.
- Likelihood is **High**.

---

### Recommendation

Include `omniBridgeChainId` in the metadata digest, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Correspondingly, add a `destination_chain` field to `MetadataPayload` on the NEAR side and populate it before signing, so the MPC signature is chain-specific.

---

### Proof of Concept

1. Token `"usdc.near"` is officially deployed on Ethereum via `deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})`. The transaction is public; `sig_eth` is visible on-chain.

2. Attacker submits the identical call to the Arbitrum `OmniBridge` contract:
   ```
   deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})
   ```
   The digest is byte-for-byte identical (no chain ID), `nearBridgeDerivedAddress` is the same on both chains, so `ECDSA.recover` succeeds and the token is deployed.

3. The NEAR bridge has no Arbitrum address registered for `"usdc.near"`. When the NEAR bridge team later tries to officially deploy `"usdc.near"` on Arbitrum, the call reverts with `"ERR_TOKEN_EXIST"`.

4. A user, seeing `isBridgeToken[attackerToken] == true` on Arbitrum, calls `initTransfer(attackerToken, 1000e6, ...)`. The bridge burns 1000 USDC from the user. No relayer processes the transfer. The 1000 USDC is permanently lost.
