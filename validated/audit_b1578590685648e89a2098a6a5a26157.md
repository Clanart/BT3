### Title
Silent Swallowing of Failed `ft_transfer`/`mint` in `fin_transfer_send_tokens_callback` Causes Permanent Token Lock on NEAR - (`near/omni-bridge/src/lib.rs`)

---

### Summary

When finalizing an inbound cross-chain transfer to a NEAR recipient, the NEAR bridge contract calls `send_tokens` and then `fin_transfer_send_tokens_callback`. The callback delegates failure detection to `is_refund_required`, but that helper **only inspects the promise result when `is_ft_transfer_call` is `true`** (i.e., when a non-empty `msg` triggers `ft_transfer_call`). For the common case of a plain `ft_transfer` or `mint` (empty `msg`), `is_refund_required` unconditionally returns `false`, meaning a failed token delivery is silently treated as a success. Because `add_fin_transfer` has already inserted the transfer ID into the `finalised_transfers` set before `send_tokens` is dispatched, the nonce is permanently consumed and the transfer cannot be retried. The recipient never receives the tokens, and there is no recovery path.

---

### Finding Description

**Vulnerable call chain:**

1. `fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_near`
2. Inside `process_fin_transfer_to_near`, `add_fin_transfer` is called first, permanently recording the transfer ID in `finalised_transfers`.
3. `send_tokens` is then dispatched. For a non-deployed token with an empty `msg`, it issues `ft_transfer(recipient, amount, None)`. For a deployed token with an empty `msg`, it issues `mint(recipient, amount, None)`.
4. The `.then(...)` callback is `fin_transfer_send_tokens_callback`, called with `is_ft_transfer_call = !msg.is_empty()` — which is `false` for the common empty-message case.

**`is_refund_required` logic:**

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => { ... }
            Err(_) => false,   // ← also does NOT refund on ft_transfer_call failure
        }
    } else {
        false   // ← always false; promise result is never inspected
    }
}
```

When `is_ft_transfer_call` is `false`, the function returns `false` without ever calling `env::promise_result_checked`. A failed `ft_transfer` or `mint` (promise result = `Err`) is indistinguishable from a successful one.

**`fin_transfer_send_tokens_callback` consequence:**

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    // burn tokens, revert lock actions, remove fin transfer  ← never reached
} else {
    // send fee, emit FinTransferEvent  ← always reached, even on failure
}
```

The transfer is logged as successful. The `finalised_transfers` entry is never removed. The source-chain tokens were already burned or locked. No recovery function exists.

**`add_fin_transfer` permanently consumes the nonce:**

```rust
fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
    require!(
        self.finalised_transfers.insert(transfer_id),
        BridgeError::TransferAlreadyFinalised.as_ref()
    );
    ...
}
```

Any subsequent attempt to call `fin_transfer` with the same proof panics with `TransferAlreadyFinalised`.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of bridged assets.**

- Tokens are burned (deployed token) or locked (native token) on the source chain at `init_transfer` time.
- On NEAR, `add_fin_transfer` permanently marks the transfer as done.
- The recipient never receives the tokens.
- No admin function, no `withdrawBlacklistedFunds`, no cancellation path exists.
- The loss is total and irreversible for every affected transfer.

---

### Likelihood Explanation

**Medium.**

USDC is a first-class bridged asset in Omni Bridge. Circle's USDC contract on NEAR implements a blacklist (`notBlacklisted` modifier on `ft_transfer`). A recipient who is not blacklisted at the time the user initiates the transfer on the source chain can be blacklisted before the relayer finalizes it on NEAR. Cross-chain finalization is not instantaneous; the window is minutes to hours depending on the source chain and relayer latency. The same risk applies to any NEP-141 token that implements transfer restrictions.

---

### Recommendation

1. **Check the promise result unconditionally.** In `is_refund_required`, inspect `env::promise_result_checked(0, ...)` regardless of `is_ft_transfer_call`. A failed `ft_transfer` or `mint` (promise result `Err`) must trigger the refund/recovery branch.

2. **Implement a recovery path.** When delivery fails, do not remove the `finalised_transfers` entry (to prevent replay), but store the undeliverable amount in a per-recipient mapping. Add a permissionless `withdraw_failed_transfer` function that lets the original sender or recipient claim the stuck funds once the restriction is lifted.

3. **Mirror the fix on `ft_transfer_call` failures.** `is_refund_required` also returns `false` on `Err` for the `ft_transfer_call` branch (line 1798), which has the same silent-failure problem.

---

### Proof of Concept

**Scenario: USDC bridged from EVM to NEAR, recipient blacklisted mid-flight.**

1. Alice holds USDC on Ethereum and calls `initTransfer` on the EVM `OmniBridge`, specifying Bob's NEAR account as recipient. Tokens are locked in the EVM vault.
2. Bob is not blacklisted at this moment.
3. A relayer picks up the Wormhole VAA / proof and calls `fin_transfer` on the NEAR bridge contract.
4. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` inserts `(origin_chain=Eth, origin_nonce=N)` into `finalised_transfers`. [1](#0-0) 
   - `send_tokens` dispatches `ext_token::ext(usdc).ft_transfer(bob, amount, None)`. [2](#0-1) 
5. Between steps 1 and 4, Circle blacklists Bob on the NEAR USDC contract. `ft_transfer` panics with `"Blacklistable: account is blacklisted"`. The promise result is `Err`.
6. `fin_transfer_send_tokens_callback` is invoked with `is_ft_transfer_call = false` (msg was empty).
7. `is_refund_required(false)` returns `false` without reading the promise result. [3](#0-2) 
8. The `else` branch executes: fee is sent, `FinTransferEvent` is emitted. [4](#0-3) 
9. Any retry of `fin_transfer` with the same proof panics: `TransferAlreadyFinalised`.
10. Alice's USDC is permanently locked in the EVM vault; Bob never receives anything on NEAR.

**Relevant code locations:**

- `process_fin_transfer_to_near` — `add_fin_transfer` called before `send_tokens`: [5](#0-4) 
- `send_tokens` — `ft_transfer` path (non-deployed token, empty msg): [2](#0-1) 
- `send_tokens` — `mint` path (deployed token, empty msg): [6](#0-5) 
- `fin_transfer_send_tokens_callback` — no failure check when `is_ft_transfer_call=false`: [7](#0-6) 
- `is_refund_required` — unconditional `false` for non-`ft_transfer_call`: [8](#0-7) 
- `add_fin_transfer` — permanent nonce consumption: [1](#0-0)
