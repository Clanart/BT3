### Title
Missing `parent_hash` Chain Validation in P2P Header Sync Allows Malicious Peer to Corrupt Stored Block Hash Chain — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary

`HeaderStreamBuilder::parse_data_for_block` contains an explicit, unimplemented TODO for the parent-hash chain check. A malicious p2p peer can supply a `SignedBlockHeader` for block N with a correct `block_number` but an arbitrary `parent_hash`, and the value will be accepted and durably stored without error, breaking the invariant `BlockHeader[N].parent_hash == BlockHeader[N-1].block_hash`.

### Finding Description

**Entrypoint — `parse_data_for_block`** (`header.rs` lines 82–123):

The function performs exactly two checks on the incoming `SignedBlockHeader`:

1. `block_number` matches the expected value (lines 104–113).
2. `signatures.len() == ALLOWED_SIGNATURES_LENGTH` — length only, no cryptographic verification (lines 115–118).

Lines 102–103 contain the explicit acknowledgement that the parent-hash check is absent:

```rust
// TODO(shahak): Check that parent_hash is the same as the previous block's hash
// and handle reverts.
``` [1](#0-0) 

**Storage layer — `append_header`** (`apollo_storage/src/header.rs` lines 294–340):

`append_header` only enforces sequential ordering via `update_marker` (the stored marker must equal `block_number`). It then writes the peer-supplied `parent_hash` verbatim into `StorageBlockHeader.parent_hash` with no cross-check against the previously stored `block_hash` of block N-1. [2](#0-1) [3](#0-2) 

**`write_to_storage`** (`header.rs` lines 28–71) is the bridge: it calls `append_header` directly with the peer-provided `BlockHeader`, introducing no additional validation. [4](#0-3) 

### Impact Explanation

The `parent_hash` field is a committed value: it is part of the block header that the sequencer signs and that block-hash calculation chains. Once a wrong `parent_hash` is durably stored for block N, every consumer that reads `get_block_header(N)` — including RPC endpoints — receives an authoritative-looking wrong value. The stored chain is permanently inconsistent (`stored[N].parent_hash ≠ stored[N-1].block_hash`) until a revert or re-sync.

**Impact: High** — RPC returns an authoritative-looking wrong committed value (`parent_hash`) for any block the attacker targets.

### Likelihood Explanation

Any p2p peer the node connects to can trigger this. No special privilege is required. The peer only needs to send a well-formed `SignedBlockHeader` (correct `block_number`, signature vector of length 1) with an arbitrary `parent_hash`. The TODO comment confirms the check was intentionally deferred, not accidentally omitted.

### Recommendation

In `parse_data_for_block`, before returning `Ok(Some(signed_block_header))`, read the stored `block_hash` of `block_number - 1` from `storage_reader` and compare it to `signed_block_header.block_header.block_header_without_hash.parent_hash`. Return `ParseDataError::BadPeer` on mismatch. The `storage_reader` parameter is already available in the function signature (currently named `_storage_reader`, indicating it was reserved for exactly this purpose). [5](#0-4) 

### Proof of Concept

Concrete Rust test sketch (production storage, no mocks):

```rust
// 1. Store block 0 with block_hash = H0.
writer.begin_rw_txn().unwrap()
    .append_header(BlockNumber(0), &BlockHeader {
        block_hash: BlockHash(Felt::from(0xH0)),
        ..Default::default()
    }).unwrap().commit().unwrap();

// 2. Construct a SignedBlockHeader for block 1 with parent_hash = Felt::ZERO (≠ H0).
let bad_header = SignedBlockHeader {
    block_header: BlockHeader {
        block_hash: BlockHash(Felt::from(1u64)),
        block_header_without_hash: BlockHeaderWithoutHash {
            block_number: BlockNumber(1),
            parent_hash: BlockHash(Felt::ZERO), // wrong: should be H0
            ..Default::default()
        },
        ..Default::default()
    },
    signatures: vec![BlockSignature::default()], // length == ALLOWED_SIGNATURES_LENGTH
};

// 3. write_to_storage accepts and commits it.
Box::new(bad_header).write_to_storage(&mut writer, &mut class_mgr).await.unwrap();

// 4. Assert the wrong parent_hash is now stored.
let stored = reader.begin_ro_txn().unwrap()
    .get_block_header(BlockNumber(1)).unwrap().unwrap();
assert_eq!(stored.block_header_without_hash.parent_hash, BlockHash(Felt::ZERO));
// Invariant violated: stored[1].parent_hash (ZERO) ≠ stored[0].block_hash (H0)
```

The test passes without error, confirming the invariant is not enforced anywhere in the production path.

```rust
fn update_marker<'env>(
    txn: &DbTransaction<'env, RW>,
    markers_table: &'env MarkersTable<'env>,
    block_number: BlockNumber,
) -> StorageResult<()> {
    // Make sure marker is consistent.
    let header_marker = markers_table.get(txn, &MarkerKind::Header)?.unwrap_or_default();
    if header_marker != block_number {
        return Err(StorageError::MarkerMismatch { expected: header_marker, found: block_number });
    };

    // Advance marker.
    markers_table.upsert(txn, &MarkerKind::Header, &block_number.unchecked_next())?;
    Ok(())
}
