### Title
Unvalidated `l1_gas_price_wei` / `l1_data_gas_price_wei` in `ProposalInit` Allows Proposer to Corrupt `eth_to_fri_rate` and `l2_gas_price_wei` in Executed Block Info — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates only the FRI-denominated L1 gas prices (`l1_gas_price_fri`, `l1_data_gas_price_fri`) within a percentage margin. The corresponding wei prices (`l1_gas_price_wei`, `l1_data_gas_price_wei`) carried in `ProposalInit` are accepted without any bounds or consistency check. Because `convert_to_sn_api_block_info` derives `eth_to_fri_rate` from the ratio of these two proposer-supplied values, a malicious proposer can set `l1_gas_price_wei` to an arbitrary nonzero value while keeping `l1_gas_price_fri` within the allowed margin. This silently corrupts `eth_to_fri_rate`, which in turn corrupts `l2_gas_price_wei` (and the direct wei prices) in the `BlockInfo` fed to the batcher. Every validator re-derives the same wrong `BlockInfo` from the same `ProposalInit`, so the corrupted block is accepted by consensus and committed with wrong ETH-denominated gas prices and a wrong block hash.

---

### Finding Description

**Step 1 — Validation gap.**

`is_block_info_valid` fetches the locally computed FRI prices and checks them within `l1_gas_price_margin_percent`:

```rust
// validate_proposal.rs lines 286-319
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;
...
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, ...)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, ...))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

`l1_gas_price_wei` and `l1_data_gas_price_wei` from `init_proposed` are never compared against any locally computed value. The discarded `_l1_gas_prices_wei` return value makes this explicit. [1](#0-0) 

**Step 2 — Corrupted rate derivation.**

`convert_to_sn_api_block_info` (called by both proposer and validator via `initiate_build` / `initiate_validation`) derives `eth_to_fri_rate` entirely from the proposer-supplied `l1_gas_price_fri` and `l1_gas_price_wei`:

```rust
// utils.rs lines 304-314
let previous_block_info = PreviousBlockInfo::from(init);   // uses init.l1_gas_price_wei
let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
let l2_gas_price_wei = NonzeroGasPrice::new(
    init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?
)?;
```

`calculate_eth_to_fri_rate` computes `l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei`. If `l1_gas_price_wei` is set to an arbitrary value, the rate is wrong, and `l2_gas_price_wei` is wrong. [2](#0-1) [3](#0-2) 

**Step 3 — Wrong `BlockInfo` fed to batcher.**

The resulting `BlockInfo` carries wrong `eth_gas_prices.l1_gas_price`, `eth_gas_prices.l1_data_gas_price`, and `eth_gas_prices.l2_gas_price`. This is passed to `propose_block` / `validate_block`: [4](#0-3) [5](#0-4) 

**Step 4 — Wrong gas prices enter block hash.**

`PartialBlockHashComponents::new` captures `l1_gas_price_per_token()`, `l1_data_gas_price_per_token()`, and `l2_gas_price_per_token()` from the corrupted `BlockInfo`. These are chained into `calculate_block_hash` via `gas_prices_to_hash`, so the committed block hash reflects the manipulated wei prices. [6](#0-5) 

**Step 5 — `ProposalInit` carries both wei and FRI prices; only FRI is guarded.**

The `ProposalInit` struct explicitly carries `l1_gas_price_wei` and `l1_data_gas_price_wei` alongside their FRI counterparts. The design comment states these are included "to allow more informative validations," but the validation in `is_block_info_valid` never uses them. [7](#0-6) 

---

### Impact Explanation

A malicious proposer (any validator in its proposer turn) sets `l1_gas_price_fri` within the allowed margin while setting `l1_gas_price_wei` to an extreme value (e.g., 1 or `u128::MAX / WEI_PER_ETH`). Every validating node re-derives the same corrupted `eth_to_fri_rate` and `l2_gas_price_wei` from the same `ProposalInit`, so all validators accept the block. The committed block contains:

- Wrong `eth_gas_prices.l1_gas_price` / `l1_data_gas_price` → wrong fees for L1 handler transactions paying in ETH.
- Wrong `eth_gas_prices.l2_gas_price` (`l2_gas_price_wei`) → wrong fees for all V3 transactions whose `max_price_per_unit` is denominated in ETH.
- Wrong `gas_prices_hash` in the block hash → the on-chain block hash commitment is incorrect.

This matches **Critical: Incorrect fee, gas, resource accounting, or balance with economic impact** and **Critical: Wrong receipt/revert result from blockifier execution logic for accepted input**.

---

### Likelihood Explanation

Any validator that reaches its proposer slot can trigger this without any external oracle manipulation or network-level attack. The trigger is a single crafted `ProposalInit` message with a manipulated `l1_gas_price_wei` field. All other validators will accept the proposal because the only enforced check (`within_margin` on FRI prices) passes.

---

### Recommendation

In `is_block_info_valid`, after computing `l1_gas_prices_fri` and the corresponding `_l1_gas_prices_wei` (currently discarded), add a consistency check that the proposed wei prices are within the same margin of the locally computed wei prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;
...
if !(within_margin(init_proposed.l1_gas_price_wei,
                   l1_gas_prices_wei.l1_gas_price,
                   l1_gas_price_margin_percent)
    && within_margin(init_proposed.l1_data_gas_price_wei,
                     l1_gas_prices_wei.l1_data_gas_price,
                     l1_gas_price_margin_percent))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

Alternatively, derive `l2_gas_price_wei` from the locally validated conversion rate rather than from the proposer-supplied wei prices, so the block info is always internally consistent with the oracle.

---

### Proof of Concept

1. Validator A is the proposer for block N.
2. A queries the oracle: `l1_gas_price_fri = 1_000_000`, `l1_gas_price_wei = 500` (normal values).
3. A crafts `ProposalInit` with `l1_gas_price_fri = 1_000_000` (within margin) but `l1_gas_price_wei = 1` (manipulated).
4. `calculate_eth_to_fri_rate` computes `1_000_000 * 10^18 / 1 = 10^24` instead of `10^21`.
5. `l2_gas_price_wei = l2_gas_price_fri * 10^18 / 10^24 = l2_gas_price_fri / 10^6` — effectively zero for typical FRI prices.
6. Validator B calls `is_block_info_valid`: `l1_gas_price_fri = 1_000_000` is within margin → passes. Wei prices are not checked → passes.
7. Validator B calls `convert_to_sn_api_block_info` with the same `ProposalInit` → derives the same corrupted `l2_gas_price_wei`.
8. Batcher executes all V3 transactions with `l2_gas_price_wei ≈ 0` → ETH-denominated L2 gas fees are near-zero.
9. Block is committed; block hash encodes the manipulated gas prices.

fn calculate_eth_to_fri_rate(block_info: &PreviousBlockInfo) -> Result<u128, StarknetApiError> {
    let eth_to_fri_rate = block_info
        .l1_prices_fri
        .l1_gas_price
        .0
        .checked_mul(WEI_PER_ETH)
        .ok_or_else(|| {
            StarknetApiError::GasPriceConversionError(format!(
                "Gas price in Fri should be small enough to multiply by WEI_PER_ETH. Previous \
                 block info: {:?}",
                block_info
            ))
        })?
        .checked_div(block_info.l1_prices_wei.l1_gas_price.0)
        .ok_or_else(|| {
            StarknetApiError::GasPriceConversionError(format!(
                "Gas price in Wei should be non-zero. Previous block info: {:?}",
                block_info
            ))
        })?;
    if eth_to_fri_rate == 0 {
        return Err(StarknetApiError::GasPriceConversionError(format!(
            "Eth to fri rate is zero. Previous block info: {:?}",
            block_info
        )));
    }
    Ok(eth_to_fri_rate)

pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,

    Ok(BlockHash(

/// This message must be sent first when proposing a new block.
/// This struct differs from `BlockInfo` in `starknet_api` because we send L1 gas prices in ETH and
/// include the ETH to STRK conversion rate. This allows for more informative validations, as we can
/// distinguish whether an issue comes from the L1 price reading or the conversion rate instead of
/// comparing after multiplication.
#[derive(Clone, Debug, PartialEq)]
pub struct ProposalInit {
    /// The height of the consensus (block number).
    pub height: BlockNumber,
    /// The current round of the consensus.
    pub round: Round,
    /// The last round that was valid.
    pub valid_round: Option<Round>,
    /// Address of the one who proposed the block in consensus.
    pub proposer: ContractAddress,
    /// Block timestamp.
    pub timestamp: u64,
    /// Address of the one who builds/sequences the block.
    pub builder: ContractAddress,
    /// L1 data availability mode.
    pub l1_da_mode: L1DataAvailabilityMode,
    /// L2 gas price in FRI.
    pub l2_gas_price_fri: GasPrice,
    /// L1 gas price in FRI.
    pub l1_gas_price_fri: GasPrice,

    /// Version constant commitment.
    pub version_constant_commitment: StarkHash,
}
