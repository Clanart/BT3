### Title
Permanent Freeze of User Funds When wNEAR Withdrawal Fails During `fin_transfer` — (File: `near/omni-bridge/src/lib.rs`)

### Summary
When a cross-chain transfer targets a NEAR recipient and the bridged token is wNEAR, the bridge finalizes the transfer (commits it to `finalised_transfers` and decrements `locked_tokens`) in one NEAR transaction, then calls `near_withdraw` on the wNEAR contract in a subsequent transaction. If `near_withdraw` fails, `near_withdraw_callback` panics unconditionally. Because the finalization state was already committed in the prior transaction, it cannot be reverted. The transfer is permanently marked as finalized, the user never receives their NEAR tokens, and no retry path exists.

### Finding Description

**Step 1 — Finalization state committed before token delivery.**

`process_fin_transfer_to_near` is called synchronously inside `fin_transfer_callback`. Both `add_fin_transfer` (which inserts the transfer ID into `finalised_transfers`) and `unlock_tokens_if_needed` execute and commit in the same `fin_transfer_callback` transaction before any cross-contract call is made: [1](#0-0) 

**Step 2 — `send_tokens` for wNEAR creates a two-step async chain.**

When the token is wNEAR and `msg` is empty, `send_tokens` issues `near_withdraw` on the wNEAR contract, then chains `near_withdraw_callback`: [2](#0-1) 

**Step 3 — `near_withdraw_callback` panics unconditionally on failure.**

If `near_withdraw` fails for any reason (wNEAR contract paused, insufficient balance, upgrade bug), `near_withdraw_callback` calls `env::panic_str` with no recovery path: [3](#0-2) 

**Step 4 — `fin_transfer_send_tokens_callback` does not revert finalization for the wNEAR path.**

`fin_transfer_send_tokens_callback` is chained after `send_tokens`. Its refund logic only triggers when `is_ft_transfer_call = true` (i.e., `msg` is non-empty). For the wNEAR path `msg` is always empty, so `is_refund_required` returns `false` and lock actions are never reverted: [4](#0-3) 

Even if lock actions were reverted, the transfer ID is already in `finalised_transfers` (inserted by `add_fin_transfer`), so `fin_transfer` cannot be called again for the same transfer: [5](#0-4) 

### Impact Explanation

The user's NEAR tokens (held as wNEAR in the bridge) are permanently locked. The transfer is irrevocably marked as finalized. There is no built-in recovery path: the user cannot retry `fin_transfer` (replay protection blocks it), and no separate claim function exists for this failure mode. This matches the **Critical — Permanent freezing, irrecoverable lock of user funds** impact class.

### Likelihood Explanation

The trigger requires `near_withdraw` on the wNEAR contract to fail. Realistic scenarios include:
- The wNEAR contract owner pauses the contract during a security incident.
- A wNEAR contract upgrade introduces a regression.
- The bridge's wNEAR balance is drained by a separate bug, leaving it unable to cover the withdrawal.

These are exotic but not impossible, matching the same "exotic but realistic" framing used in the original report. Critically, once the condition occurs, every in-flight wNEAR `fin_transfer` targeting a NEAR recipient is permanently frozen with no operator-level recovery short of a contract upgrade.

### Recommendation

Move `add_fin_transfer` (and `unlock_tokens_if_needed`) to execute only after successful token delivery, or add a recovery branch in `fin_transfer_send_tokens_callback` that removes the transfer from `finalised_transfers` when the wNEAR withdrawal fails, allowing the relayer to retry. Analogous to the referenced fix, the bridge should continue without blocking: if wNEAR withdrawal fails, either keep the wNEAR claimable by the user in a separate step, or remove the finalization record so the transfer can be re-submitted.

### Proof of Concept

1. User initiates a transfer of wNEAR from Ethereum to NEAR (recipient = NEAR account).
2. Relayer calls `fin_transfer` with a valid proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_near` runs:
   - `add_fin_transfer` inserts transfer ID into `finalised_transfers`. ✓ committed.
   - `unlock_tokens_if_needed` decrements `locked_tokens`. ✓ committed.
   - `send_tokens` schedules `near_withdraw` on wNEAR contract.
4. Between step 3 and step 5, the wNEAR contract is paused (or any failure condition occurs).
5. `near_withdraw` fails → `near_withdraw_callback` is invoked with `PromiseResult::Failed` → panics with `NearWithdrawFailed`.
6. `fin_transfer_send_tokens_callback` is invoked with `PromiseResult::Failed`. `is_refund_required(false)` → `false`. No state is reverted.
7. Result: transfer ID permanently in `finalised_transfers`; user's NEAR tokens permanently locked in bridge as wNEAR; no retry possible. [6](#0-5)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
    }
