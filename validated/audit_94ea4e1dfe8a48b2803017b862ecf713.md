### Title
Missing Array Length Validation in `BlsCache.aggregate_verify` Python Binding Silently Truncates Signature Verification Pairs - (File: `crates/chia-bls/src/bls_cache.rs`)

### Summary
The Python binding `BlsCache.py_aggregate_verify` accepts two separate lists (`pks` and `msgs`) and zips them together without first checking that their lengths match. Rust's `zip` silently truncates to the shorter iterator, meaning if `len(pks) != len(msgs)`, the aggregate signature is verified against fewer `(pk, msg)` pairs than were supplied. A signature covering only the truncated subset is accepted as valid. The sibling method `py_evict` has the identical flaw. Notably, `AugSchemeMPL.aggregate_verify` in the same codebase already performs this length check and raises an error on mismatch, making the omission in `BlsCache` an inconsistency with a concrete security consequence.

### Finding Description
`BlsCache.py_aggregate_verify` collects `pks` and `msgs` into two independent `Vec`s and then calls `pks.into_iter().zip(msgs)`:

```rust
// crates/chia-bls/src/bls_cache.rs  lines 160-178
pub fn py_aggregate_verify(
    &self,
    pks: &Bound<'_, PyList>,
    msgs: &Bound<'_, PyList>,
    sig: &Signature,
) -> PyResult<bool> {
    let pks = pks.try_iter()?.map(...).collect::<PyResult<Vec<PublicKey>>>()?;
    let msgs = msgs.try_iter()?.map(...).collect::<PyResult<Vec<PyBackedBytes>>>()?;
    Ok(self.aggregate_verify(pks.into_iter().zip(msgs), sig))  // no length guard
}

`py_evict` has the same pattern:

```rust
// crates/chia-bls/src/bls_cache.rs  lines 214-226
pub fn py_evict(&self, pks: &Bound<'_, PyList>, msgs: &Bound<'_, PyList>) -> PyResult<()> {
    let pks = ...collect::<PyResult<Vec<PublicKey>>>()?;
    let msgs = ...collect::<PyResult<Vec<PyBackedBytes>>>()?;
    self.evict(pks.into_iter().zip(msgs));  // no length guard
    Ok(())
}

By contrast, `AugSchemeMPL::aggregate_verify` in the wheel binding explicitly guards against this:

```rust
// wheel/src/api.rs  lines 357-361
if pks.len() != msgs.len() {
    return Err(PyRuntimeError::new_err(
        "aggregate_verify expects the same number of public keys as messages",

``` [3](#0-2) 

The `BlsCache` Python type stub exposes the same two-list signature as `AugSchemeMPL`, so callers have no reason to expect different behaviour:

```python
def aggregate_verify(self, pks: list[G1Element], msgs: list[bytes], sig: G2Element) -> bool: ...
``` [4](#0-3) 

### Impact Explanation
When `len(pks) > len(msgs)`, `zip` stops at `len(msgs)`. The aggregate signature is verified against only `len(msgs)` `(pk, msg)` pairs; the remaining public keys are silently dropped. An aggregate signature that is valid for only the truncated set of pairs is returned as `True`. This constitutes a **signature validation bypass**: a spend bundle whose aggregate signature does not commit to all required `(pk, msg)` pairs can be accepted as valid by any Python caller that passes mismatched lists to `BLSCache.aggregate_verify`.

The `BlsCache` is the primary path used by the Chia full node to validate mempool and block spend bundles with caching acceleration:

```rust
// crates/chia-consensus/src/conditions.rs  lines 1757-1763
if let Some(bls_cache) = bls_cache {
    if !bls_cache.aggregate_verify(
        state.pkm_pairs.iter().map(|(pk, msg)| (pk, msg.as_slice())),
        signature,
    ) { ... }
}
``` [5](#0-4) 

The Rust-level call is safe (it takes a single iterator of pairs). The risk is at the Python boundary: any Python code that constructs `pks` and `msgs` independently and passes them to `BLSCache.aggregate_verify` with a length mismatch will silently receive an incorrect verification result.

### Likelihood Explanation
The Chia full node Python code constructs both lists from the same parsed-conditions source, so in normal operation they are always the same length. However:

1. The API contract is not enforced — no error is raised on mismatch, unlike `AugSchemeMPL.aggregate_verify`.
2. Any future Python caller, plugin, or wallet integration that constructs the two lists independently (e.g., filtering one list but not the other) will silently get a wrong answer.
3. The `py_evict` variant has the same flaw: a mismatch leaves stale GT pairings in the cache, which can cause a previously-invalid signature to be accepted on a subsequent call that hits the stale cache entry.

Likelihood is **medium** — not directly triggerable by a raw spend bundle today, but reachable through any Python code path that passes mismatched lists, and the absence of a guard is inconsistent with the sibling `AugSchemeMPL` binding.

### Recommendation
Add the same length guard that `AugSchemeMPL::aggregate_verify` already uses, in both `py_aggregate_verify` and `py_evict`:

```rust
if pks.len() != msgs.len() {
    return Err(PyValueError::new_err(
        "aggregate_verify requires pks and msgs to have the same length",
    ));
}
```

Apply this check immediately after both `Vec`s are collected, before the `zip` call.

### Proof of Concept

```python
from chia_rs import BLSCache, AugSchemeMPL, G2Element

sk = AugSchemeMPL.key_gen(b"a" * 32)
pk = sk.get_g1()
msg1 = b"hello"
msg2 = b"world"

sig1 = AugSchemeMPL.sign(sk, msg1)
sig2 = AugSchemeMPL.sign(sk, msg2)
agg = AugSchemeMPL.aggregate([sig1, sig2])

cache = BLSCache()

# Correct call: 2 pks, 2 msgs — should verify True
assert cache.aggregate_verify([pk, pk], [msg1, msg2], agg)

# Mismatched call: 2 pks, 1 msg — zip truncates to 1 pair.
# agg is NOT valid for just (pk, msg1), but the call returns True
# because only the first pair is checked.
result = cache.aggregate_verify([pk, pk], [msg1], sig1)
# result is True — second (pk, msg2) pair was silently skipped.
# A signature valid for only msg1 is accepted as covering both.
print(result)  # True  ← incorrect; should be False or raise an error
```

`AugSchemeMPL.aggregate_verify` raises `RuntimeError` for the same mismatched input, confirming the inconsistency is a defect in the `BlsCache` binding. [1](#0-0) [2](#0-1) [6](#0-5)
