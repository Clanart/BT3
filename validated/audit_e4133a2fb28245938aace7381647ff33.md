### Title
Integer Truncation in `NewDynamicFeeChecker` Allows Zero-Fee Cosmos Transaction Execution When `baseFee = 0` - (File: `ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` computes the per-gas fee cap via integer division `feeCap = fee / gas`. When `fee < gas`, this truncates to zero. If the chain's `baseFee` is also zero (reachable when `MinGasPrice` is a fractional `LegacyDec` value, because `CalculateBaseFee` uses `MinGasPrice.TruncateInt()` as its lower bound), the `feeCap < baseFeeInt` guard passes, `effectivePrice` resolves to zero, and `effectiveFee = 0` is returned to `DeductFeeDecorator`. The decorator then deducts nothing from the sender, allowing Cosmos transactions to execute with zero fee payment despite satisfying the `MinGasPriceDecorator` check.

---

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` derives the per-gas fee cap from the Cosmos tx fee fields using integer (floor) division:

```go
// ante/evm/fee_checker.go:83
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

`sdkmath.Int.Quo` performs truncating integer division. When `fee < gas` (e.g., `fee = 20999`, `gas = 21000`), `feeCap` truncates to `0`.

The guard that follows:

```go
// ante/evm/fee_checker.go:86-88
if feeCap.LT(baseFeeInt) {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, ...)
}
```

only rejects the tx when `feeCap < baseFeeInt`. When `baseFee = 0`, the condition `0 < 0` is false, so the tx is accepted.

The effective price is then:

```go
// ante/evm/fee_checker.go:91
effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
// = min(0 + maxPriorityPrice, 0) = 0
```

And the effective fee:

```go
// ante/evm/fee_checker.go:94-99
effectiveFee := sdk.Coins{{Denom: denom, Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))}}
// = 0 * gas = 0
```

`DeductFeeDecorator` receives `effectiveFee = 0` and skips deduction:

```go
// ante/evm/nativefee.go:110-114
if !fee.IsZero() {
    err := evmkeeper.DeductFees(dfd.bankKeeper, ctx, deductFeesFromAcc, fee)
    ...
}
```

**How `baseFee = 0` is reached:** `CalculateBaseFee` in `x/feemarket/keeper/eip1559.go` bounds the base fee from below using:

```go
// x/feemarket/keeper/eip1559.go:103-104
minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
```

If `MinGasPrice` is a fractional `LegacyDec` value such as `0.5`, `TruncateInt()` returns `0`, so the base fee lower bound is `0`. Under low chain activity the base fee decreases to `0`. Meanwhile, `MinGasPriceDecorator` uses the untruncated `MinGasPrice = 0.5` for its check (using `Ceil()`), so it correctly requires a non-zero fee from the user — but `NewDynamicFeeChecker` then truncates that fee back to zero effective price.

---

### Impact Explanation

An unprivileged user can submit valid Cosmos transactions (e.g., `MsgSend`, `MsgDelegate`, any non-EVM message) and pay **zero fees** despite the `MinGasPriceDecorator` check passing. The fee tokens the user nominally attached are never deducted from their account. This constitutes a fee mis-accounting bug in the ante handler: the chain's fee revenue is silently zeroed out for all Cosmos transactions when `baseFee = 0`, and the `MinGasPrice` enforcement is effectively bypassed at the deduction step.

This matches the allowed High impact: *"ante handler bug that permits valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

The condition requires:
1. `MinGasPrice` set to a fractional value (e.g., `0.5 aevmos`) via governance — a realistic configuration on low-fee chains.
2. `baseFee` reaching `0` — guaranteed when `MinGasPrice.TruncateInt() = 0` and the chain has any period of low block utilization.
3. Attacker submits a Cosmos tx with `fee < gas` — trivially crafted; the minimum intrinsic gas for a Cosmos tx is typically 21,000+, so setting `fee = gas - 1` satisfies the condition.

Both conditions are reachable in production without any privileged access.

---

### Recommendation

Replace the truncating integer division with a check that preserves the full precision of the fee-per-gas comparison. Specifically, validate that `fee >= baseFee * gas` before computing `feeCap`, or compute `feeCap` using `LegacyDec` arithmetic with rounding:

```go
// Use ceiling division so feeCap is never under-reported
feeDec := sdkmath.LegacyNewDecFromInt(fee)
gasDec := sdkmath.LegacyNewDecFromInt(sdkmath.NewIntFromUint64(gas))
feeCap := feeDec.Quo(gasDec).Ceil().TruncateInt()
```

Alternatively, perform the minimum-fee check directly in integer space: `fee >= baseFee * gas`, and set `effectiveFee = min(fee, effectivePrice * gas)` without going through a truncated per-gas price.

---

### Proof of Concept

**Setup:**
- Chain params: `MinGasPrice = 0.5`, `BaseFee = 0` (reached after low-activity blocks since `TruncateInt(0.5) = 0`).
- Attacker account has balance.

**Transaction:**
- Cosmos `MsgSend` with `gas = 21000`, `fee = 10500 aevmos` (satisfies `MinGasPriceDecorator`: `ceil(0.5 * 21000) = 10500`).

**Trace through ante handler:**

1. `MinGasPriceDecorator`: `feeCoins (10500) >= requiredFees (10500)` → passes. [1](#0-0) 

2. `NewDynamicFeeChecker`:
   - `feeCap = 10500 / 21000 = 0` (integer truncation).
   - `baseFeeInt = 0`.
   - `0 < 0` → false, guard passes.
   - `effectivePrice = min(0 + MaxInt64, 0) = 0`.
   - `effectiveFee = 0 * 21000 = 0`. [2](#0-1) 

3. `DeductFeeDecorator`: `effectiveFee.IsZero()` → skips deduction. Attacker pays **0 fees**. [3](#0-2) 

4. Root cause of `baseFee = 0`: `MinGasPrice.TruncateInt() = 0` used as lower bound in `CalculateBaseFee`. [4](#0-3)
