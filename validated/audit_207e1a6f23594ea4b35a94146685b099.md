### Title
`rejected_txs` silently dropped during catch-up causes rejected L1 handler transactions to be permanently marked Committed — (`crates/apollo_l1_provider/src/l1_provider.rs`)

### Summary
`commit_block` drops the `rejected_txs` argument before forwarding to `accept_commit_while_catching_up`. Both the `Equal` branch and the backlog replay loop call `apply_commit_block(..., Default::default())`. As a result, every L1 handler transaction that was included in a block but failed execution during catch-up is marked `Committed` (not `Rejected`), permanently removing it from the proposable set and causing the corresponding L1 message to be skipped from all future state diffs.

### Finding Description
`commit_block` signature accepts `rejected_txs: IndexSet<TransactionHash>` but the catch-up routing at line 315 passes only `committed_txs` and `height` to `accept_commit_while_catching_up`. The function signature of `accept_commit_while_catching_up` does not even accept a `rejected_txs` parameter. Inside that function, both the `Equal` branch (line 427) and the backlog loop (line 464) call `apply_commit_block(txs, Default::default())`. `apply_commit_block` partitions `consumed_txs` by membership in `rejected_txs`; with an empty set, all txs land in the committed bucket. `commit_txs` then calls `mark_committed()` on every one of them, setting `state = Committed` and `committed = true`. The `mark_committed` guard only checks `!self.committed`, not `!self.rejected`, so it does not panic. The tx record is now permanently `Committed`.

### Impact Explanation
A `Committed` record fails `is_validatable()` (`!is_committed()` is false), so it is never returned by `get_txs` or accepted by `validate_tx`. The L1 handler transaction — representing a concrete L1→L2 message — will never be re-proposed. The L1 message is permanently absent from every future state diff, breaking the L1→L2 message guarantee.

### Likelihood Explanation
The catch-up path is exercised on every node restart when the batcher is ahead of the provider. Any block committed during catch-up that contains a rejected L1 handler tx (execution failure, bouncer rejection, etc.) triggers the bug. No special privileges or adversarial input are required; normal batcher operation suffices.

### Recommendation
Pass `rejected_txs` through the entire catch-up path:

1. Add `rejected_txs: IndexSet<TransactionHash>` to `accept_commit_while_catching_up`.
2. Pass it from `commit_block` at line 315.
3. In the `Equal` branch, call `apply_commit_block(committed_txs, rejected_txs)` instead of `apply_commit_block(committed_txs, Default::default())`.
4. Store `rejected_txs` alongside `committed_txs` in `CommitBlockBacklog` and pass it through the backlog replay loop at line 464.

### Proof of Concept
```rust
// Rust pseudo-test (unit level, no async needed for the Equal branch)
let mut provider = L1ProviderContentBuilder::new()
    .with_txs([l1_handler(1)])          // tx1 is Pending
    .with_height(BlockNumber(3))
    .with_state(ProviderState::CatchingUp)
    .build_into_l1_provider();

// Batcher commits block 3 with tx1 consumed but rejected (execution failed).
let consumed = [tx_hash!(1)].into();
let rejected = [tx_hash!(1)].into();
provider.commit_block(consumed, rejected, BlockNumber(3)).unwrap();

// BUG: tx1 is now Committed, not Rejected.
let snap = provider.tx_manager.snapshot();
assert!(snap.rejected.contains(&tx_hash!(1)));   // FAILS — it's in committed
assert!(!snap.committed.contains(&tx_hash!(1))); // FAILS — it's committed

// Consequence: get_txs returns nothing; L1 message is permanently skipped.
```
