Audit Report

## Title
Excess Native ETH Sent to Payable Swap Functions Is Not Automatically Refunded and Can Be Stolen by Any Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function in `PeripheryPayments.sol` wraps only the exact `value` owed when `token == WETH` and `nativeBalance >= value`, leaving any excess `msg.value` silently stranded in the router. `refundETH()` is a permissionless `external` function that sends the entire contract ETH balance to `msg.sender`, allowing any third party to steal the stranded ETH. A compounding path exists where pre-existing stranded ETH is silently consumed to subsidize a subsequent user's swap.

## Finding Description
**Root cause — `pay` in `PeripheryPayments.sol` lines 73–84:**

When `nativeBalance >= value`, the function executes:
```solidity
IWETH9(WETH).deposit{value: value}();
IERC20(WETH).safeTransfer(recipient, value);
```
Only `value` wei is wrapped; the remainder (`nativeBalance - value`) stays in the contract with no refund issued. None of the four swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) issue an automatic refund after execution.

**Theft vector — `refundETH()` lines 58–63:**
```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```
No access control, no record of the original depositor. Any address calling this after a victim's swap receives the full stranded balance.

**Compounding theft — partial-payment branch lines 78–81:**
If stranded ETH remains when a second user's WETH swap triggers `pay` with `0 < nativeBalance < value`, the function wraps all stranded ETH and credits it toward the second user's obligation, then pulls only the shortfall from the second user's ERC-20 allowance. The first user's ETH is permanently transferred to the pool on behalf of a stranger.

**Existing guards are insufficient:** The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH sends; ETH arriving via `msg.value` in payable functions bypasses it entirely. The intended safe pattern (`multicall([swap, refundETH])`) is shown in the test suite but is not enforced at the contract level.

## Impact Explanation
Direct loss of user ETH principal. For exact-output swaps (`exactOutputSingle`, `exactOutput`), over-sending ETH is the structurally required usage pattern because the exact input amount is unknown before pool execution. Any user who calls these functions directly without appending `refundETH()` in a `multicall` bundle will strand ETH that is immediately extractable by any caller. The compounding path additionally drains stranded ETH silently into a subsequent user's swap with no revert or event. This constitutes a Critical/High direct loss of user principal above Sherlock thresholds.

## Likelihood Explanation
Moderate-to-high. Exact-output swaps structurally require the caller to send more ETH than will be consumed, making the over-send the normal usage pattern rather than an exceptional one. Any user calling `exactOutputSingle` or `exactOutput` directly — rather than through a carefully constructed `multicall` bundle — will strand ETH. MEV searchers routinely monitor mempools for permissionless refund functions holding ETH and will extract it within the same block.

## Recommendation
1. **Automatic refund at swap exit:** After each top-level swap function completes, check `address(this).balance` and transfer any non-zero amount back to `msg.sender` before returning.
2. **Restrict `refundETH` to the original payer:** Record the initiating `msg.sender` in transient storage at the start of each top-level call and require `msg.sender == originalPayer` inside `refundETH()`, or make `refundETH` internal and call it automatically.
3. **Guard the partial-payment branch:** The `else if (nativeBalance > 0)` branch should only consume native balance deposited in the current call (tracked via transient storage), not any pre-existing balance left by a prior user.
