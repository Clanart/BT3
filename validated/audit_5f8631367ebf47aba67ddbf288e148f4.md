### Title
Missing Zero-Rate Validation in `resolve_query` Bypasses Fallback Mechanism, Producing Wrong L1 Gas Prices in Block Commitment - (File: crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs)

### Summary

The `resolve_query` function in `EthToStrkOracleClient` does not validate that the parsed ETH-to-STRK conversion rate is non-zero before returning `Ok(rate)`. Because `GasPrice::wei_to_fri(0)` silently returns `Ok(GasPrice(0))` rather than an error, a zero rate from the oracle causes `L1PricesInFri::convert_from_wei` to succeed with zero FRI prices. This success result bypasses the three-tier fallback mechanism (live oracle → previous block info → configured defaults), so the sequencer proceeds with zero L1 gas prices in FRI instead of using safe fallback values. The zero FRI prices propagate into `ProposalInit` and ultimately into the block hash commitment path, producing an incorrect `gas_prices_hash` component in `calculate_block_hash`.

### Finding Description

**Root cause — `resolve_query` accepts a zero rate:**

```rust
// crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs  lines 166-167
let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)
    .expect("Failed to parse price as u128");
// No check: if rate == 0 { return Err(...) }
Ok(rate)   // returns Ok(0) when oracle sends {"price":"0x0","decimals":18}
``` [1](#0-0) 

**`wei_to_fri(0)` returns `Ok(GasPrice(0))` — no error, no fallback trigger:**

```rust
// crates/starknet_api/src/block.rs  lines 424-437
pub fn wei_to_fri(self, eth_to_fri_rate: u128) -> Result<GasPrice, StarknetApiError> {
    Ok(self
        .checked_mul_u128(eth_to_fri_rate)   // any_price * 0 = 0  → Some(GasPrice(0))
        .ok_or_else(|| ...)?
        .checked_div(WEI_PER_ETH)            // 0 / WEI_PER_ETH = 0 → Some(GasPrice(0))
        .expect("WEI_PER_ETH must be non-zero"))
}
``` [2](#0-1) 

Contrast with `fri_to_wei`, which explicitly rejects a zero rate: [3](#0-2) 

**Fallback mechanism is bypassed:**

`get_l1_prices_in_fri_and_wei_and_conversion_rate` only falls back to previous-block-info or configured defaults when `convert_from_wei` returns `Err`. Because `wei_to_fri(0)` returns `Ok`, `convert_from_wei` returns `Ok(L1PricesInFri { l1_gas_price: GasPrice(0), l1_data_gas_price: GasPrice(0) })`, and the function returns immediately with zero FRI prices and `eth_to_fri_rate = 0`: [4](#0-3) 

**Zero FRI prices enter `ProposalInit` and the block hash path:**

`PartialBlockHashComponents::new` reads gas prices directly from `BlockInfo`, which is constructed from `ProposalInit`. `calculate_block_hash` chains `gas_prices_to_hash(l1_gas_price, l1_data_gas_price, l2_gas_price, ...)` into the Poseidon hash: [5](#0-4) [6](#0-5) 

**`convert_to_sn_api_block_info` logs a warning but does not prevent the zero prices from reaching the commitment path in the proposer flow:** [7](#0-6) 

The warning is emitted, then `NonzeroGasPrice::new(GasPrice(0))` returns an error that aborts block execution — but only after the zero prices have already been broadcast in `ProposalInit` to all validators. Validators that also receive `Ok(0)` from their oracle will compute the same zero FRI prices and accept the proposal as valid per `is_block_info_valid`. [8](#0-7) 

### Impact Explanation

1. **Wrong L1 gas price in block commitment**: When the oracle returns `{"price":"0x0","decimals":18}`, the sequencer computes `l1_gas_price_fri = 0` and `l1_data_gas_price_fri = 0`. These zero values are chained into `gas_prices_to_hash` and therefore into `calculate_block_hash`, producing a block hash that differs from the correct value (which would use fallback prices). This is a wrong `gas_prices_hash` component in the block commitment.

2. **Fallback invariant broken**: The three-tier fallback (live oracle → previous block → defaults) is designed to ensure the sequencer always uses a valid, non-zero conversion rate. A zero rate from the oracle bypasses all three tiers because `wei_to_fri(0)` returns `Ok` rather than `Err`, violating the invariant that fallback is triggered on invalid oracle data.

3. **Incorrect fee accounting**: Transactions in a block built with zero FRI L1 gas prices would be charged zero FRI for L1 gas consumption, constituting an incorrect fee/resource-accounting effect with direct economic impact.

### Likelihood Explanation

The oracle is an external HTTP endpoint. A transient misconfiguration, API bug, or a response of `{"price":"0x0","decimals":18}` (which passes all existing checks — the field is present, parseable as hex, and `decimals == 18`) is sufficient to trigger the issue. No privileged access is required beyond the ability to influence the oracle response. The existing test suite has no test for `wei_to_fri(0)` returning an error (unlike `fri_to_wei(0)`, which is explicitly tested), confirming the gap. [9](#0-8) 

### Recommendation

1. **In `resolve_query`**: After parsing `rate`, add:
   ```rust
   if rate == 0 {
       return Err(EthToStrkOracleClientError::InvalidRateError(
           "ETH-to-STRK rate must be non-zero".to_string()
       ));
   }
   ```
   This mirrors the existing `decimals` validation and ensures a zero rate is treated as an oracle failure, triggering the fallback mechanism.

2. **In `GasPrice::wei_to_fri`**: Add an explicit zero-rate guard analogous to `fri_to_wei`:
   ```rust
   if eth_to_fri_rate == 0 {
       return Err(StarknetApiError::GasPriceConversionError(
           "ETH to FRI rate must be non-zero".to_string()
       ));
   }
   ```

3. **In `get_l1_prices_in_fri_and_wei_and_conversion_rate`**: Add an explicit check `if eth_to_fri_rate == 0 { /* treat as error, fall through to fallback */ }` before calling `convert_from_wei`, as defense-in-depth.

### Proof of Concept

1. Configure the oracle endpoint to return `{"price":"0x0","decimals":18}`.
2. The sequencer calls `eth_to_fri_rate(timestamp)` → `resolve_query` returns `Ok(0)`.
3. `get_l1_prices_in_fri_and_wei_and_conversion_rate` receives `Ok(0)` and `Ok(price_info)`.
4. `apply_fee_transformations` clamps WEI prices to configured minimums (non-zero).
5. `L1PricesInFri::convert_from_wei(&prices_in_wei, 0)` calls `wei_to_fri(0)` → `Ok(GasPrice(0))` for both L1 gas and L1 data gas.
6. The function returns `(L1PricesInFri { l1_gas_price: GasPrice(0), l1_data_gas_price: GasPrice(0) }, prices_in_wei, 0)` — fallback is never reached.
7. `ProposalInit` is constructed with `l1_gas_price_fri = 0`, `l1_data_gas_price_fri = 0`.
8. `PartialBlockHashComponents::new` reads these zero FRI prices; `gas_prices_to_hash` chains `0` into the Poseidon hash, producing a wrong `gas_prices_hash` in `calculate_block_hash`.
9. Validators that also receive `Ok(0)` from their oracle compute the same zero FRI prices and accept the proposal via `is_block_info_valid` (prices match within margin).
10. Block execution then fails at `NonzeroGasPrice::new(GasPrice(0))`, but the wrong commitment has already been broadcast and agreed upon by consensus.

/// Poseidon (
///     block_hash_constant, block_number, global_state_root, sequencer_address,
///     block_timestamp, concat_counts, state_diff_hash, transaction_commitment,
///     event_commitment, receipt_commitment, gas_prices, starknet_version, 0, parent_block_hash

pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}
