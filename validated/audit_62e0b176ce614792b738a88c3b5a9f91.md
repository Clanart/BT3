### Title
Stale Cached `feemarketParams` in Cosmos Ante Handler Causes Incorrect Fee Validation Across All Blocks - (`evmd/ante/handler_options.go`)

---

### Summary

`newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712` fetch `feemarketParams` (including `BaseFee` and `MinGasPrice`) once at handler construction time and pass them by pointer into `NewDynamicFeeChecker` and `NewMinGasPriceDecorator`. Because `feemarketParams.BaseFee` is recalculated and persisted to the KV store every block by `CalculateBaseFee`, the cached snapshot becomes stale immediately after the first block. Every subsequent Cosmos (non-Ethereum) transaction processed by these handlers is validated against the wrong base fee, allowing fee underpayment when the base fee rises, or causing spurious rejection when it falls.

---

### Finding Description

In `evmd/ante/handler_options.go`, both `newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712` snapshot module parameters at construction time:

```go

    ...

    ...
    cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams),
```

The returned `sdk.AnteHandler` closure captures `feemarketParams` (a local variable on the stack) by pointer. Both decorators then read from this frozen snapshot on every transaction invocation:

**`NewDynamicFeeChecker`** reads `feemarketParams.GetBaseFee()` via `types.GetBaseFee`:

```go
// ante/evm/fee_checker.go:56

```

```go

    ...
    baseFee := feemarketParams.GetBaseFee()   // reads the stale snapshot
    ...
}
```

**`MinGasPriceDecorator`** reads `mpd.feemarketParams.MinGasPrice` directly:

```go

```

Meanwhile, the Ethereum-tx path (`newEthAnteHandler`) correctly fetches fresh params on every invocation:


The base fee is recalculated and written to the KV store on every block by `CalculateBaseFee`:

The stale `feemarketParams.BaseFee` in the Cosmos ante handler is never refreshed from the KV store after construction.

---

### Impact Explanation

**Outcome 1 – Fee underpayment (base fee has risen):** An attacker submits a Cosmos tx with a fee just above the stale (lower) base fee but below the actual current base fee. `NewDynamicFeeChecker` accepts it because it compares against the stale value. The tx commits with fees that are insufficient relative to the current fee market, mis-accounting user funds and fee-collector revenue. This matches: *"ante handler bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

**Outcome 2 – Spurious rejection (base fee has fallen):** Legitimate Cosmos txs with fees sufficient for the current (lower) base fee are rejected because the stale (higher) base fee is used as the threshold. This is a reachable denial-of-service for honest users.

**Outcome 3 – MinGasPrice governance bypass:** If governance raises `MinGasPrice`, the cached value in `MinGasPriceDecorator` remains at the old lower value, allowing txs that should be rejected to pass.

---

### Likelihood Explanation

- The base fee changes every block; the divergence between the cached and live value grows monotonically with block count.
- Any unprivileged user submitting a standard Cosmos tx (bank send, IBC, governance vote, etc.) through the Cosmos ante handler path triggers this code.
- No special privileges, validator access, or key compromise are required.
- The attacker only needs to observe the current on-chain base fee (public RPC) and the stale cached value (derivable from the block at which the handler was constructed) to craft a fee that passes the stale check but is below the live threshold.

---

### Recommendation

Replace the construction-time parameter snapshot in `newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712` with live per-invocation fetches, mirroring the pattern already used in `newEthAnteHandler`. Specifically:

1. Remove the `ctx sdk.Context` parameter from `newCosmosAnteHandler` (it is only used to snapshot params).
2. Inside the returned closure, call `options.FeeMarketKeeper.GetParams(ctx)` and `options.EvmKeeper.GetParams(ctx)` on each invocation before constructing the fee checker logic, or pass a keeper reference into `MinGasPriceDecorator` and `NewDynamicFeeChecker` so they can fetch live params.

---

### Proof of Concept

1. At block N, `newCosmosAnteHandler` is constructed. `feemarketParams.BaseFee = 100`.
2. Blocks N+1 … N+K are full; `CalculateBaseFee` raises the live base fee to `200` in the KV store.
3. Attacker submits a Cosmos bank-send tx with `feeCap = 150` (fee = 150 × gas).
4. `NewDynamicFeeChecker` evaluates `feeCap (150) >= baseFee (100 stale)` → **passes**.
5. The tx is included and committed; the fee collector receives `150 × gas` instead of the required `200 × gas`.
6. The live base fee in the KV store was never consulted; the stale snapshot was the only check.
