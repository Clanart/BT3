### Title
Unchecked `uint64` Underflow in `nestedCreates` Nonce Reconciliation Corrupts Sender Nonce When `evm.Create()` Returns Without Incrementing Nonce - (File: x/evm/keeper/state_transition.go)

---

### Summary

In `ApplyMessageWithConfig`, the nonce reconciliation formula `nestedCreates := afterCreateNonce - msg.Nonce - 1` performs an unchecked `uint64` subtraction. When `evm.Create()` returns early — specifically when the EVM-level `CanTransfer` balance check fails — the nonce is **not** incremented, leaving `afterCreateNonce == msg.Nonce`. The subtraction silently wraps to `MaxUint64` in Go's unsigned arithmetic, and the subsequent `stateDB.SetNonce(sender, oldNonce+MaxUint64)` wraps the committed nonce to `oldNonce - 1`. This permanently corrupts the sender's on-chain sequence number, enabling replay of a previously-executed transaction.

---

### Finding Description

`ApplyMessageWithConfig` implements a reset-and-reconcile pattern to handle Ethermint's Cosmos-specific ante handler pre-increment of the sender nonce: [1](#0-0) 

The logic:
1. Saves `oldNonce = stateDB.GetNonce(sender)` — the ante-handler-incremented value.
2. Resets the nonce to `msg.Nonce` before calling `evm.Create()`.
3. After `evm.Create()` returns, computes `nestedCreates := afterCreateNonce - msg.Nonce - 1`.
4. Sets the final nonce to `oldNonce + nestedCreates`.

The formula at step 3 **assumes `evm.Create()` always increments the nonce by at least 1**. In go-ethereum's `evm.Create()`, the nonce increment occurs **after** the `CanTransfer` balance check. If the balance check fails (returning `ErrInsufficientBalance`), the function returns immediately without touching the nonce. This leaves `afterCreateNonce == msg.Nonce`, so:

```
nestedCreates = msg.Nonce - msg.Nonce - 1
             = 0 - 1  (uint64)
             = 18446744073709551615  (MaxUint64, silent wrap-around)
```

Then:
```
stateDB.SetNonce(sender, oldNonce + MaxUint64)
      = stateDB.SetNonce(sender, oldNonce - 1)  (mod 2^64)
```

The corrupted nonce is committed to the KVStore at the end of `ApplyMessageWithConfig`: [2](#0-1) 

**Trigger path — batched Cosmos transaction:**

The ante handler's `CheckEthCanTransfer` and `VerifyEthAccount` check each message's value **independently** against the pre-execution balance, not cumulatively: [3](#0-2) [4](#0-3) 

`CheckAndSetEthSenderNonce` pre-increments the nonce for **all** messages in the batch upfront: [5](#0-4) 

A batched Cosmos transaction containing:
- **msg0**: CALL with `value = V0` (drains most of the balance)
- **msg1**: CREATE with `value = V1 > 0`

passes the ante handler if `balance >= V0` and `balance >= V1` independently. After msg0 executes and transfers `V0` out, msg1's `evm.Create()` finds `balance - V0 < V1` and returns `ErrInsufficientBalance` without incrementing the nonce. The underflow then corrupts the committed nonce.

---

### Impact Explanation

The sender's nonce is committed to state as `oldNonce - 1` instead of `oldNonce`. Since `oldNonce` was the post-ante-handler value (original nonce + number of messages in the batch), the account's sequence number is permanently set lower than expected. This allows the attacker to submit a new transaction with nonce `oldNonce - 1` — a nonce that was already consumed — effectively bypassing Ethereum's replay-protection invariant. The attacker can use this to re-execute a previously-authorized transaction (e.g., a value transfer or contract call) or to inject a different transaction at a nonce that was already spent.

This matches the allowed High impact: **Ethereum transaction nonce verification bypass enabling replay or forged execution**.

---

### Likelihood Explanation

Requires crafting a batched Cosmos EVM transaction (a Cosmos-SDK-specific feature, discouraged by ADR-003 but not prohibited) where a prior CALL message drains the sender's balance below the value required by a subsequent CREATE message. The attacker must control the sender account. The ante handler's per-message independent balance checks make this feasible without triggering a pre-execution rejection. Likelihood: **Low-Medium** — requires deliberate construction but no privileged access.

---

### Recommendation

Guard the `nestedCreates` subtraction against underflow. Before computing it, verify that `evm.Create()` actually incremented the nonce:

```go
afterCreateNonce := stateDB.GetNonce(sender)
var nestedCreates uint64
if afterCreateNonce > msg.Nonce {
    // evm.Create() incremented nonce at least once (normal path)
    nestedCreates = afterCreateNonce - msg.Nonce - 1
} else {
    // evm.Create() returned early without incrementing nonce (e.g., ErrInsufficientBalance)
    // Restore nonce to oldNonce; no nested creates occurred.
    nestedCreates = 0
}
stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
```

### Proof of Concept

**Setup:** Sender has balance `B = 100 wei`, on-chain nonce `N = 5`.

**Batch transaction (2 messages):**
- msg0: CALL to any address, `value = 90 wei`, `nonce = 5`
- msg1: CREATE (contract deployment), `value = 20 wei`, `nonce = 6`

**Ante handler execution:**
- `VerifyEthAccount`: checks `100 >= cost(msg0)` ✓ and `100 >= cost(msg1)` ✓ (independent, pre-execution balance)
- `CheckEthCanTransfer`: checks `100 >= 90` ✓ and `100 >= 20` ✓ (independent)
- `CheckAndSetEthSenderNonce`: increments nonce to `N+2 = 7`; `oldNonce` for msg1 = `7`

**Execution of msg0:** transfers 90 wei out → balance = 10 wei.

**Execution of msg1 (`ApplyMessageWithConfig`):**
```
oldNonce = stateDB.GetNonce(sender) = 7
stateDB.SetNonce(sender, msg.Nonce=6)
evm.Create(): CanTransfer(10 wei, 20 wei) → false → returns ErrInsufficientBalance
              nonce NOT incremented
afterCreateNonce = 6
nestedCreates = 6 - 6 - 1 = MaxUint64  (uint64 underflow)
stateDB.SetNonce(sender, 7 + MaxUint64) = stateDB.SetNonce(sender, 6)
```

**Committed state:** sender nonce = `6` instead of `7`.

**Replay:** Attacker now submits a new transaction with `nonce = 6`. The chain accepts it (expected nonce is `6`), allowing re-execution at a nonce that was already spent by msg1.
