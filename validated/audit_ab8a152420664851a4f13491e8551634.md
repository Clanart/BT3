### Title
Double-Finalization via `remove_fin_transfer` After `ft_transfer_call` Rejection - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a NEAR-destined `fin_transfer` uses `ft_transfer_call` (non-empty `msg`) and the recipient contract rejects the transfer, `fin_transfer_send_tokens_callback` removes the transfer ID from `finalised_transfers`. This allows the same MPC-signed proof to be submitted a second time, minting or releasing tokens a second time for a single cross-chain transfer — a direct double-spend.

---

### Finding Description

The `process_fin_transfer_to_near` function adds the transfer ID to `finalised_transfers` at the very start as replay protection: [1](#0-0) 

```rust
let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
```

`add_fin_transfer` enforces uniqueness — it panics with `TransferAlreadyFinalised` if the ID is already present: [2](#0-1) 

When `transfer_message.msg` is non-empty, `send_tokens` issues an `ft_transfer_call`. The callback `fin_transfer_send_tokens_callback` checks `is_refund_required`: [3](#0-2) 

`is_refund_required` returns `true` when the recipient contract's `ft_on_transfer` returns `U128(0)` — meaning it rejected the transfer and returned all tokens: [4](#0-3) 

On this path, the callback calls `remove_fin_transfer`, which **unconditionally deletes the transfer ID from `finalised_transfers`**: [5](#0-4) 

After removal, the same `(origin_chain, origin_nonce)` transfer ID is no longer in `finalised_transfers`. A subsequent call to `add_fin_transfer` with the same ID will succeed, allowing the entire `fin_transfer` flow to execute again.

---

### Impact Explanation

**Critical — double-spend / unauthorized mint of bridged assets.**

For deployed bridge tokens: the bridge mints tokens to the recipient on the first submission, the recipient rejects them (returned to bridge, burned), finalization record is deleted, then the bridge mints again on the second submission — the attacker receives tokens for a single cross-chain proof.

For native (non-deployed) tokens: `burn_tokens_if_needed` is a no-op, so the returned tokens remain in the bridge's balance. On the second submission the bridge transfers from its locked balance again, draining it by the transfer amount for a single proof. [6](#0-5) 

---

### Likelihood Explanation

**High.** The attacker needs only:
1. A NEAR contract they control as the transfer recipient.
2. A cross-chain transfer initiated with a non-empty `msg` field (fully user-controlled in `initTransfer` on EVM/Solana/StarkNet).
3. The MPC-signed proof (obtained normally through the bridge protocol).

No privileged access, leaked keys, or colluding validators are required. The `msg` field is part of the signed payload and is set by the user at transfer initiation time. [7](#0-6) 

---

### Recommendation

Do not remove a transfer from `finalised_transfers` on `ft_transfer_call` failure. The finalization record must be permanent once written. Instead, if the recipient rejects the transfer, record a separate "claimable refund" entry for the original sender rather than erasing the replay-protection record. The `finalised_transfers` set must be append-only after a transfer is first finalized.

---

### Proof of Concept

1. Attacker deploys a NEAR contract `attacker.near` whose `ft_on_transfer` returns the full received amount (rejects) on the first call, and returns `0` (accepts) on the second call.
2. On EVM, attacker calls `initTransfer(token, amount, fee, nativeFee, "attacker.near", "trigger_reject")` — the non-empty `message` field ensures `ft_transfer_call` will be used on NEAR.
3. MPC signs the transfer; attacker obtains the signed payload.
4. Attacker calls `fin_transfer` on NEAR with the signed payload. `process_fin_transfer_to_near` runs: `add_fin_transfer` inserts the ID, `send_tokens` mints tokens to `attacker.near` via `ft_transfer_call`.
5. `attacker.near::ft_on_transfer` returns the full amount → tokens returned to bridge → `fin_transfer_send_tokens_callback` calls `burn_tokens_if_needed` (burns for deployed tokens) and `remove_fin_transfer` (deletes ID from `finalised_transfers`).
6. Attacker calls `fin_transfer` again with the **same** signed payload. `add_fin_transfer` succeeds (ID was removed). `send_tokens` mints tokens again. `attacker.near::ft_on_transfer` now returns `0` → tokens kept.
7. Attacker has received tokens for a single cross-chain transfer proof. For deployed tokens, the bridge's total supply is inflated by `amount`. For native tokens, the bridge's locked balance is drained by `amount` beyond what was deposited. [8](#0-7) [9](#0-8)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {

                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {

            BridgeError::TransferAlreadyFinalised.as_ref()

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
