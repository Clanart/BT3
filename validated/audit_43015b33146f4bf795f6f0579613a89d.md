### Title
`wrapping_add` Integer Overflow in `check_time_locks` Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Timelocks When `nowrap=false` — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false` (the legacy pre-softfork path), relative timelock conditions (`ASSERT_HEIGHT_RELATIVE`, `ASSERT_SECONDS_RELATIVE`) are evaluated using `wrapping_add`. A crafted condition value near the type maximum causes the sum to wrap to a small number, making the timelock check pass immediately regardless of the actual block height or timestamp. This is a direct arithmetic-overflow analog to the external report's underflow-causes-wrong-validation pattern.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` accepts a `nowrap: bool` parameter that selects between `saturating_add` (safe) and `wrapping_add` (unsafe) for relative timelock arithmetic.

For `ASSERT_HEIGHT_RELATIVE` when `nowrap=false`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

For `ASSERT_SECONDS_RELATIVE` when `nowrap=false`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

**Overflow scenario for `ASSERT_HEIGHT_RELATIVE`:**

- `confirmed_block_index = 100`, `height_relative = 0xffff_ffff` (u32::MAX)
- `100u32.wrapping_add(0xffff_ffff) = 99` (wraps)
- Guard: `prev_height (200) < 99` → `false` → **`Ok(())`** — spend accepted
- Correct result: the coin cannot be spent for ~4 billion more blocks → **`Err(AssertHeightRelativeFailed)`**

The project's own test suite explicitly documents and asserts this divergent behavior:

```rust
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [3](#0-2) 

The same overflow pattern applies to `ASSERT_SECONDS_RELATIVE` with `u64::MAX`:

```rust
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),
)]
``` [4](#0-3) 

The inverse overflow also affects `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE` in the `nowrap=false` path: a large value wraps to a small deadline, causing valid spends to be incorrectly rejected (DoS). [5](#0-4) 

The function is exported as a Python binding and the `nowrap` flag is caller-controlled: [6](#0-5) 

---

### Impact Explanation

**High — Timelock validation bypass enables unauthorized spend acceptance.**

A coin locked with `ASSERT_HEIGHT_RELATIVE(N)` or `ASSERT_SECONDS_RELATIVE(N)` where `confirmed_block_index + N` overflows the integer type will have its timelock silently bypassed when the full node validates using `nowrap=false`. The spend is accepted as consensus-valid at any block height, regardless of the intended lock duration. This matches the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

For puzzles that accept the relative timelock value from the spender's solution (a common pattern in vesting, escrow, or DeFi-style contracts), an attacker who controls the solution can supply `height_relative = 0xffff_ffff` or `seconds_relative = u64::MAX`, causing the wrapping bypass on any node still running the `nowrap=false` code path.

---

### Likelihood Explanation

The `nowrap=false` path is the legacy pre-softfork behavior. The Python binding exposes `nowrap` as a direct parameter, meaning any full node calling `check_time_locks` with `nowrap=False` (e.g., when validating blocks before the softfork activation height) is vulnerable. The overflow values (`u32::MAX` for height, `u64::MAX` for seconds) are valid CLVM integers that pass all upstream sanitization checks in `parse_amount` / `sanitize_uint`. [7](#0-6) 

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for both `height_relative` and `seconds_relative` checks, eliminating the `nowrap` branch for these two conditions. The `nowrap=false` path was introduced for backward compatibility, but the wrapping behavior produces consensus-incorrect results and should not be preserved. Alternatively, validate at condition-parse time that `confirmed_block_index + height_relative` cannot overflow (reject with `AssertHeightRelativeFailed` if it would), mirroring the fix pattern from the external report: add an explicit pre-check before the arithmetic.

```diff
- } else if prev_transaction_block_height
-     < unspent.confirmed_block_index.wrapping_add(height_relative)
+ } else if prev_transaction_block_height
+     < unspent.confirmed_block_index.saturating_add(height_relative)
  {
      return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
  }
```

---

### Proof of Concept

1. Craft a puzzle that outputs `(ASSERT_HEIGHT_RELATIVE . 0xffffffff)` (or accepts the value from the solution).
2. Submit a spend bundle for this coin at any block height.
3. Call `check_time_locks(..., nowrap=False)`.
4. Observe: `confirmed_block_index.wrapping_add(0xffffffff)` wraps to `confirmed_block_index - 1`.
5. The guard `prev_height < (confirmed_block_index - 1)` evaluates to `false` for any `prev_height >= confirmed_block_index - 1`.
6. The spend is accepted as valid despite the timelock not being satisfied.

The test suite at lines 276–281 and 301–306 of `check_time_locks.rs` already encodes this exact scenario and confirms the `Ok(())` return under `nowrap=false`. [8](#0-7)

#[pyfunction]

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
