### Title
`ProofOfInclusion::valid()` Contains a Tautological Final Check, Allowing Forged Inclusion Proofs to Pass Validation — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` is the primary API for verifying DataLayer Merkle inclusion proofs and is exposed directly to Python consumers via PyO3 bindings. The function's final correctness check — `existing_hash == self.root_hash()` — is a logical tautology: after the loop, `existing_hash` is always equal to `self.root_hash()` by construction. As a result, `valid()` only verifies internal chain consistency and never anchors the proof to any external trusted root. An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key is present) and an arbitrary fake root, and `valid()` will return `true`.

### Finding Description

**Root cause — tautological final check in `valid()`:**

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs

pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← derived entirely from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← TAUTOLOGY: both sides are last.combined_hash
}
```

After the loop body executes without returning `false`, `existing_hash` holds the last `calculated_hash`, which the loop already verified equals `last_layer.combined_hash`. `self.root_hash()` returns exactly `last_layer.combined_hash`. Therefore the final expression is `last_layer.combined_hash == last_layer.combined_hash`, which is unconditionally `true`.

**What `valid()` actually checks:** only that each layer's `combined_hash` is the correct SHA-256 hash of the previous hash combined with `other_hash`. It does **not** check that the resulting root matches any externally trusted value.

**Attacker-controlled entry path:**

`ProofOfInclusion` derives `Streamable` and is exposed as a `pyclass` with `from_py_object`, meaning it can be deserialized from arbitrary bytes supplied by an untrusted peer or API caller:

```rust
#[cfg_attr(
    feature = "py-bindings",
    pyclass(get_all, from_py_object),
    derive(PyJsonDict, PyStreamable)
)]
pub struct ProofOfInclusion {
    pub node_hash: Hash,
    pub layers: Vec<ProofOfInclusionLayer>,
}
```

A Python consumer calls `proof.valid()` via the binding:

```rust
#[pyo3(name = "valid")]
pub fn py_valid(&self) -> bool {
    self.valid()
}
```

**Forge construction:** An attacker builds a `ProofOfInclusion` where:
- `node_hash` = any hash they wish to claim is in the tree (e.g., a key they do not own)
- `layers[0].other_hash` = any value; `layers[0].combined_hash` = `calculate_internal_hash(node_hash, side, other_hash)` (correctly computed)
- Additional layers follow the same pattern, each `combined_hash` correctly derived from the previous

The final `combined_hash` (the fake root) is whatever the attacker chose. `valid()` returns `true`. The attacker then presents this proof to any verifier that calls only `proof.valid()` without separately comparing `proof.root_hash()` to a known-good tree root.

### Impact Explanation

**Impact category:** High — DataLayer Merkle proof logic lets untrusted input prove invalid state.

Any DataLayer client or protocol layer that calls `proof.valid()` as the sole inclusion check can be deceived into accepting a forged proof of inclusion for a key that is not present in the actual Merkle tree. Because `ProofOfInclusion` is `Streamable` and fully constructible from untrusted bytes, an unprivileged attacker with network access can supply a crafted proof object. The attacker can assert the presence of arbitrary key-value pairs in a DataLayer store whose committed root they do not control, enabling forged state acceptance.

### Likelihood Explanation

The `valid()` function name strongly implies complete proof validation. No documentation or inline comment warns callers that they must additionally compare `proof.root_hash()` against a trusted external root. The Python type stub exposes `valid() -> bool` and `root_hash() -> bytes32` as separate, independent methods with no guidance on required call order. Any DataLayer integration that follows the natural API usage pattern — calling `proof.valid()` and treating `True` as "proof accepted" — is vulnerable. The `Streamable` deserialization path means the attack requires only the ability to send a crafted serialized `ProofOfInclusion` to a consumer.

### Recommendation

**Short term:** Add a `trusted_root: Hash` parameter to `valid()` (and `py_valid()`) and replace the tautological final check with a comparison against the caller-supplied trusted root:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // anchor to external trusted root
}
```

**Long term:** Audit all DataLayer proof verification call sites to ensure they supply a root hash obtained from a trusted, authenticated source (e.g., a committed on-chain root), not from the proof object itself.

### Proof of Concept

```python
from chia_rs.datalayer import (
    ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob, KeyId, ValueId
)
import hashlib

# Attacker wants to forge a proof that key_id=999 is in some tree.
# They don't have access to the real tree.

FAKE_NODE_HASH = bytes([0xAA] * 32)   # claimed leaf hash for key 999
FAKE_OTHER_HASH = bytes([0xBB] * 32)  # arbitrary sibling hash

# Compute combined_hash exactly as calculate_internal_hash does:
# internal_hash(left, right) = SHA256(b"\x02" + left + right)
def internal_hash(left: bytes, right: bytes) -> bytes:
    return hashlib.sha256(b"\x02" + left + right).digest()

# other_hash_side = Right (1) means: combined = internal_hash(existing, other)
combined = internal_hash(FAKE_NODE_HASH, FAKE_OTHER_HASH)

layer = ProofOfInclusionLayer(
    other_hash_side=1,           # Side::Right
    other_hash=FAKE_OTHER_HASH,
    combined_hash=combined,      # correctly computed → loop won't return False
)

forged_proof = ProofOfInclusion(
    node_hash=FAKE_NODE_HASH,
    layers=[layer],
)

# valid() returns True despite this proof being entirely fabricated
assert forged_proof.valid(), "Expected True — tautological check passes"

# The "root" is the attacker-chosen combined hash, not any real tree root
print("Forged root:", forged_proof.root_hash().hex())
# A verifier calling only proof.valid() accepts this as a valid inclusion proof.
```
