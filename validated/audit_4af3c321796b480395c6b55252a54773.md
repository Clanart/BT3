### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (`execution/transaction_impls.cairo`)

### Summary

The `compute_max_possible_fee` function in `transaction_impls.cairo` performs unchecked felt-field arithmetic on user-supplied resource bound values. Because Cairo felt arithmetic is modular (mod the Stark prime P ≈ 2²⁵¹), a malicious transaction sender can craft `max_amount` and `max_price_per_unit` values whose products wrap around to zero, causing `charge_fee` to skip fee collection entirely and allowing the attacker to execute transactions at zero cost.

### Finding Description

`compute_max_possible_fee` computes the maximum chargeable fee as a sum of felt-field products:

```cairo
// transaction_impls.cairo L99-101
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

No range checks are applied to the individual `max_amount` or `max_price_per_unit` fields before this multiplication. All arithmetic is modular over the Stark prime P. An attacker can choose values such that the entire sum ≡ 0 (mod P). A concrete example:

- `l2_gas_bounds.max_amount = (P+1)/2`, `l2_gas_bounds.max_price_per_unit = 2` → contribution = `(P+1)/2 * 2 = P+1 ≡ 1 (mod P)`
- `l1_gas_bounds.max_amount = (P-1)/2`, `l1_gas_bounds.max_price_per_unit = 2` → contribution = `(P-1)/2 * 2 = P-1 ≡ P-1 (mod P)`
- `l1_data_gas_bounds.max_amount = 0` → contribution = 0
- Total: `1 + (P-1) + 0 = P ≡ 0 (mod P)`

The result `max_fee = 0` is then tested in `charge_fee`:

```cairo
// transaction_impls.cairo L121-125
let max_fee = compute_max_possible_fee(tx_info=tx_info);

if (max_fee == 0) {
    return ();
}
``` [2](#0-1) 

When `max_fee == 0`, `charge_fee` returns immediately without executing the ERC-20 transfer to the sequencer. The attacker's L2 gas budget is `(P+1)/2 ≈ 2²⁵⁰`, which is far above `EXECUTE_MAX_SIERRA_GAS`, so `cap_remaining_gas` still grants the attacker the full allowed execution gas:

```cairo
// transaction_impls.cairo L322-323
let initial_user_gas_bound = get_initial_user_gas_bound(common_tx_fields=common_tx_fields);
let remaining_gas = initial_user_gas_bound;
``` [3](#0-2) 

The same `charge_fee` call path is present in `execute_deploy_account_transaction` (line 687) and `execute_declare_transaction` (line 822), making all V3 transaction types vulnerable. [4](#0-3) 

### Impact Explanation

**Critical — Direct loss of funds.** The sequencer/validators perform real computational work (executing Cairo contracts, generating proofs) but receive zero fee compensation. At scale, any number of attackers can flood the network with zero-cost transactions, draining sequencer resources without payment. Because the OS proof is valid under these crafted inputs, the block is accepted on-chain and the fee bypass is permanent and irrefutable.

### Likelihood Explanation

**High.** The attack requires only:
1. Computing a pair of felt values whose products sum to P (trivial modular arithmetic, no privileged access needed).
2. Signing and submitting a standard V3 invoke/deploy-account/declare transaction with those resource bounds.

No special role, leaked key, or network-level capability is required. Any unprivileged user can perform this.

### Recommendation

Apply explicit range checks to `max_amount` and `max_price_per_unit` before performing the fee multiplication. Specifically, enforce that each field fits within a safe integer range (e.g., `assert_nn_le(max_amount, MAX_RESOURCE_BOUND)` and `assert_nn_le(max_price_per_unit, MAX_PRICE_BOUND)`) so that no product can overflow the felt field. Alternatively, use `Uint256` arithmetic for the fee computation to make overflow impossible.

### Proof of Concept

1. Choose Stark prime P = 2²⁵¹ + 17·2¹⁹² + 1.
2. Set resource bounds for a V3 invoke transaction:
   - `l2_gas_bounds = { max_amount: (P+1)/2, max_price_per_unit: 2 }`
   - `l1_gas_bounds = { max_amount: (P-1)/2, max_price_per_unit: 2 }`
   - `l1_data_gas_bounds = { max_amount: 0, max_price_per_unit: 0 }`
   - `tip = 0`
3. `compute_max_possible_fee` returns `(P+1)/2·2 + (P-1)/2·2 + 0 = (P+1) + (P-1) = 2P ≡ 0 (mod P)`.
4. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without transferring any fee.
5. The transaction executes with up to `EXECUTE_MAX_SIERRA_GAS` gas, the proof is valid, and the block is accepted — with zero fee paid to the sequencer.
