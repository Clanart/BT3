### Title
Missing Nonce in `meta_tx_v0` Hash Construction Enables Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` constructs the hash used for signature verification in the `meta_tx_v0` syscall with `additional_data_size=0`, meaning **no nonce is included**. Because the hash is purely a function of `(INVOKE_HASH_PREFIX, 0, contract_address, entry_point_selector, calldata, 0, chain_id)`, any signature produced for a given `(contract_address, calldata)` pair is valid forever and can be replayed by any unprivileged actor who observed it on-chain.

---

### Finding Description

`compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

The `deprecated_get_transaction_hash` function appends `additional_data` to the hash state at the end: [2](#0-1) 

With `additional_data_size=0`, nothing is appended — the nonce is entirely absent from the hash.

Compare this to `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1` and `additional_data=&nonce`: [3](#0-2) 

In `execute_meta_tx_v0`, the `new_tx_info` is constructed with `nonce=0` hardcoded, and **`check_and_increment_nonce` is never called**: [4](#0-3) 

The resulting `meta_tx_hash` is placed into `new_tx_info.transaction_hash` and the target account's `__execute__` entry point is invoked with the caller-supplied `signature`: [5](#0-4) 

Because the hash is deterministic over `(contract_address, calldata, chain_id)` with no nonce, the same signature is valid for every future invocation with the same parameters.

This is the direct analog of the KintoID bug: the nonce field is present in the struct (`nonce=0`) but is never incorporated into the signed hash, and no nonce is incremented after use.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any account contract that relies on `meta_tx_v0` to authorize financial operations (e.g., ERC-20 transfers, vault withdrawals, session-key-delegated spending) is permanently exposed after the first use. An attacker who observes a `meta_tx_v0` execution on-chain can extract the `(contract_address, calldata, signature)` triple from the transaction trace and replay it an unlimited number of times by deploying a contract that issues the same `meta_tx_v0` syscall. The OS will compute the identical hash, the victim account's `__execute__` will accept the replayed signature, and the financial operation will execute again — draining the account.

---

### Likelihood Explanation

`meta_tx_v0` is a first-class OS syscall reachable by any deployed contract. The attacker requires no privileged role: they only need to observe a prior `meta_tx_v0` execution (the signature is part of the on-chain syscall trace) and deploy a contract that re-issues the syscall with the same parameters. The `selector` is forced to `EXECUTE_ENTRY_POINT_SELECTOR` by the OS, so the replay targets the victim's `__execute__` directly. No key compromise, Sybil attack, or operator collusion is required.

---

### Recommendation

Include the account contract's current nonce in the `meta_tx_v0` hash, mirroring how `compute_l1_handler_transaction_hash` handles it:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // <-- add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,   // <-- was 0
        additional_data=&nonce,   // <-- was cast(0, felt*)
    );
    return tx_hash;
}
```

Additionally, `execute_meta_tx_v0` must read the current nonce from `contract_state_changes` for `contract_address`, pass it to the hash function, and increment it after the call — exactly as `check_and_increment_nonce` does for regular account transactions. [6](#0-5) 

---

### Proof of Concept

1. **Victim** calls contract `C` which issues `meta_tx_v0(contract_address=V, calldata=[transfer(attacker, 1000)], signature=SIG)`. The OS computes `H = H(INVOKE_PREFIX, 0, V, EXECUTE_SELECTOR, H(calldata), 0, chain_id)`. `V.__execute__` verifies `SIG` against `H` and executes the transfer.

2. **Attacker** reads `(V, calldata, SIG)` from the on-chain trace.

3. **Attacker** deploys contract `Evil` containing:
   ```
   meta_tx_v0(contract_address=V, selector=EXECUTE_SELECTOR,
              calldata=[transfer(attacker, 1000)], signature=SIG)
   ```

4. **Attacker** calls `Evil`. The OS computes the identical `H` (no nonce in hash, `nonce=0` hardcoded in `new_tx_info`). `V.__execute__` receives the same `SIG` against the same `H`, verifies successfully, and executes the transfer again.

5. Step 3–4 repeats until `V`'s balance is drained.

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
