### Title
`set_state_header` never asserts `chunk.shard_id() == shard_id`, allowing a peer to substitute a cross-shard chunk and corrupt the syncing node's trie - (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the embedded chunk's own `shard_id` field equals the caller-supplied `shard_id` parameter. A malicious peer can respond to a state-sync request for shard Y by embedding shard X's chunk (with shard X's valid merkle proof). All existing checks pass. The header is persisted under shard Y's key. Subsequent `set_state_part` and `apply_state_part` calls use shard X's `prev_state_root` as the authoritative root for shard Y, writing shard X's trie data into shard Y's `ShardUId`-keyed trie storage and applying shard X's transactions/receipts as shard Y's chunk in `set_state_finalize`.

---

### Finding Description

**The missing identity binding.** `set_state_header` runs five validation groups before persisting the header:

1. **`validate_chunk_proofs`** (line 380) — verifies the chunk's internal hash, `tx_root`, and `prev_outgoing_receipts_root`. It does **not** compare `chunk.shard_id()` against the `shard_id` parameter. [1](#0-0) 

2. **`verify_path` for chunk inclusion** (lines 394–403) — verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's `chunk_headers_root` merkle tree. It uses the standard `verify_path` (not `verify_path_with_index`), which only checks that the item is reachable from the root via the supplied path; it does **not** check which shard-index position the path encodes. Supplying shard X's chunk with shard X's path (position X) passes this check even when `shard_id = Y`. [2](#0-1) 

3. **`verify_path` for `prev_chunk`** (lines 416–425) — same position-agnostic check for the previous chunk.

4. **Receipt-proof chain** (lines 438–510) — verifies receipt proofs against `shard_id` (the parameter), but this check is independent of the chunk's own `shard_id` field.

5. **`validate_state_root_node`** (lines 514–522) — verifies the state root node against `chunk_inner.prev_state_root()`, which is shard X's root when the chunk is from shard X.

After all five checks the header is stored under `StateHeaderKey(shard_id=Y, sync_hash)`: [3](#0-2) 

**Downstream propagation.** `set_state_part` retrieves the stored header for shard Y, extracts `prev_state_root` from the embedded chunk (shard X's root), and validates incoming state parts against it: [4](#0-3) 

`apply_state_part` then derives `shard_uid` from the `shard_id=Y` parameter and writes shard X's trie nodes into shard Y's trie column: [5](#0-4) 

`set_state_finalize` similarly derives `shard_uid` from `shard_id=Y` and applies shard X's transactions and receipts as shard Y's chunk: [6](#0-5) 

---

### Impact Explanation

The syncing node's trie for shard Y is overwritten with shard X's trie data. After `set_state_finalize` the node holds a `ChunkExtra` for shard Y whose `state_root` is shard X's root. Every subsequent block application for shard Y computes a state root that diverges from the canonical chain, causing the node to:

- Produce invalid chunks for shard Y (if it is a chunk producer), which are rejected by other validators and may trigger slashing or kickout.
- Fail to validate any block that includes shard Y's chunk, breaking consensus participation.
- Require a full re-sync to recover.

The corruption is silent at the time of state sync; no error is surfaced until the node attempts to apply the first post-sync block.

**Severity: High** — permanent local state corruption for the targeted shard, with validator-level consequences if the node holds a chunk-producer seat.

---

### Likelihood Explanation

The attack requires an unprivileged peer that:

1. Knows a valid chunk from shard X in the sync block — this is public on-chain data.
2. Holds the merkle proof for shard X's position in that block — also public.
3. Can serve state parts whose trie nodes hash to shard X's `prev_state_root` — available to any node that tracks shard X.

Any node tracking shard X that is connected to the syncing node as a state-sync peer satisfies all three conditions. State sync peers are chosen from the general peer set; no privileged role is required.

---

### Recommendation

Add an explicit identity-binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind the chunk's own shard_id to the caller-supplied shard_id
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same guard to `prev_chunk_header` if present:

```rust
if let Some(ref prev) = prev_chunk_header {
    if prev.shard_id() != shard_id {
        byzantine_assert!(false);
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk shard_id does not match requested shard_id".into(),
        ));
    }
}
```

Additionally, replace the position-agnostic `verify_path` call with `verify_path_with_index` (already used in `validate_part`) so that the merkle proof is also bound to the correct shard index, providing defence-in-depth.

```

  - sync_hash H is the first block of a new epoch.
  - Block B = prev(H) contains chunks for shard X (index iX) and shard Y (index iY).

  - Attacker tracks shard X and holds its full state.

Steps:
1. Syncing node enters state sync for shard Y, requests header from attacker.

2. Attacker constructs a ShardStateSyncResponseHeader where:
     .chunk          = shard X's ShardChunk (shard_id = X, prev_state_root = rootX)
     .chunk_proof    = merkle path for position iX in B.chunk_headers_root
     .state_root_node = valid StateRootNode for rootX
     .incoming_receipts_proofs = [] (empty, passes check 4g if prev_chunk absent)

3. Syncing node calls set_state_header(shard_id=Y, sync_hash=H, attacker_header).
     validate_chunk_proofs(shard_X_chunk)  → Ok(true)   [no shard_id check]
     verify_path(B.chunk_headers_root,
                 shard_X_proof,
                 ChunkHashHeight(shard_X_hash, shard_X_height_included))
                                           → true        [position iX is valid]
     validate_state_root_node(rootX_node, rootX) → Valid

4. Attacker serves state parts P_0 … P_n whose trie nodes reconstruct rootX.

5. For each part:
     set_state_part(shard_id=Y, sync_hash=H, part_id=i, part=P_i)
       get_state_header(Y, H) → header with rootX
       validate_state_part(Y, rootX, i, P_i) → Valid
       stored under StatePartKey(H, Y, i)

6. apply_state_part(shard_id=Y, state_root=rootX, …, epoch_id)
     shard_uid = get_shard_uid_from_epoch_id(Y, epoch_id)   ← shard Y's UID
     Trie::apply_state_part(rootX, …)                        ← shard X's nodes
     tries.apply_all(trie_changes, shard_Y_uid, …)           ← written to shard Y

7. set_state_finalize(shard_id=Y, sync_hash=H)
     shard_uid = shard_id_to_uid(Y, epoch_id)
     apply_chunk(shard_Y_uid, shard_X_transactions, shard_X_receipts, …)
     ChunkExtra for shard Y now has state_root = rootX

Result: shard Y's trie contains shard X's state. The node diverges from the
canonical chain on the first block it processes for shard Y.
```

    chunk: &ShardChunk,

    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());

        if &receipts_root != outgoing_receipts_root {
