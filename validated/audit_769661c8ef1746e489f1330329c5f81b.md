### Title
Felt-Field Overflow in `compute_max_possible_fee` Allows Fee Bypass — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the fee ceiling using unchecked felt-field arithmetic. Because `max_price_per_unit` is only constrained to `[0, PRIME/2)` (a ~250-bit range) while `max_amount` is constrained to `[0, 2^64 - 1]`, their product can exceed the field prime and wrap around modulo PRIME. A user can craft resource-bound values such that the entire sum reduces to exactly `0 mod PRIME`. When `max_fee == 0`, `charge_fee` returns immediately without executing any ERC-20 transfer, so the transaction is processed with zero fee paid.

---

### Finding Description

`compute_max_possible_fee` in `transaction_impls.cairo` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

All arithmetic is in the Cairo felt field (modulo PRIME ≈ 2^251). The only constraints enforced on the inputs come from `pack_resource_bounds`, called during hash computation on the same `ResourceBounds` struct:

- `max_amount` ∈ `[0, 2^64 - 1]` via `assert_nn_le`
- `max_price_per_unit` ∈ `[0, PRIME/2)` via `assert_nn` (a ~250-bit range)
- `tip` ∈ `[0, 2^64 - 1]` via `assert_nn_le` [2](#0-1) 

A single product `max_amount * max_price_per_unit` can reach `(2^64 - 1) * (PRIME/2 - 1) ≈ 2^314`, far exceeding PRIME, so it wraps around. The result of the three-term sum is therefore an arbitrary felt value, including `0`.

In `charge_fee`, if `max_fee == 0` the function returns immediately, skipping the ERC-20 transfer entirely:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

There is no range check on the return value of `compute_max_possible_fee` to ensure it is non-zero or within a meaningful bound.

---

### Impact Explanation

An unprivileged transaction sender can execute invoke, declare, or deploy-account transactions without paying any fee. The sequencer processes the transaction (state changes are committed, the block proof is valid) but receives zero fee tokens. This is a **direct loss of funds** for the sequencer/protocol on every such transaction. Because the OS proof is valid, the on-chain verifier accepts the block, making the loss permanent and unrecoverable.

---

### Likelihood Explanation

The attack requires only basic modular arithmetic. Given any target sum `S` such that `S ≡ 0 (mod PRIME)`, the attacker solves for `max_price_per_unit = S_i * max_amount_i^{-1} mod PRIME` for each resource slot. Since `max_amount` has 2^64 choices and roughly half yield a `max_price_per_unit` in `[0, PRIME/2)`, valid inputs are abundant and trivially computable offline. No special privileges, leaked keys, or network access beyond submitting a transaction are required.

---

### Recommendation

After computing `max_fee`, add an explicit range check that the result is within a safe integer range (e.g., `[0, 2^128 - 1]`) before using it as a fee ceiling. Alternatively, decompose the fee computation into 128-bit-safe arithmetic using `Uint256` or multi-limb representations, ensuring no intermediate product can overflow the felt field. The simplest fix is to add `assert_nn_le(max_fee, MAX_FEE_BOUND)` immediately after the return of `compute_max_possible_fee`, where `MAX_FEE_BOUND` is a protocol-defined constant that fits within 128 bits.

---

### Proof of Concept

**Goal:** craft a V3 transaction whose resource bounds make `compute_max_possible_fee` return `0 mod PRIME`.

Let PRIME = 2^251 + 17·2^192 + 1.

1. Choose `l1_gas.max_amount = 1`, `l2_gas.max_amount = 1`, `l1_data_gas.max_amount = 0`, `tip = 0`.
2. The fee sum reduces to: `P1 + P2 ≡ 0 (mod PRIME)`, i.e., `P2 = PRIME - P1`.
3. Choose `P1 = 1`. Then `P2 = PRIME - 1`. But `PRIME - 1 > PRIME/2`, so `assert_nn(P2)` fails.
4. Instead, use `l1_gas.max_amount = 2`, `l2_gas.max_amount = 1`, `l1_data_gas.max_amount = 1`, `tip = 0`. The sum is `2*P1 + P2 + P3 ≡ 0 (mod PRIME)`. Choose `P1 = (PRIME - 1)/4` (approximately), `P2 = (PRIME - 1)/4`, `P3 = (PRIME - 1)/2`. All three are in `[0, PRIME/2)` and their weighted sum is `≡ 0 (mod PRIME)`.
5. Sign the transaction with these resource bounds. The transaction hash commits to the packed bounds (which pass all range checks). The OS computes `max_fee = 0` and skips fee charging. The transaction executes and state changes are finalized with no fee paid. [4](#0-3) [5](#0-4) [6](#0-5)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
