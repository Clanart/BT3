Audit Report

## Title
Excess native ETH sent to `exactOutputSingle`/`exactOutput` is silently trapped in the router and claimable by any caller via `refundETH()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
When a user calls `exactOutputSingle` (or `exactOutput`) with `tokenIn = WETH` and sends more native ETH than the pool consumes, `PeripheryPayments.pay()` wraps only the exact amount owed to the pool and leaves the remainder as raw ETH inside the router. Neither swap function performs a post-swap ETH balance check or automatic refund. Because `refundETH()` is permissionless and transfers to `msg.sender`, any third party can drain the trapped surplus in a follow-up call, causing direct loss of user principal.

## Finding Description
`PeripheryPayments.pay()` handles the WETH-as-native-ETH path at lines 73–84. When `address(this).balance >= value`, it wraps exactly `value` wei and forwards it to the pool; the delta `balance − value` remains as unaccounted native ETH in the router with no automatic return path.

`exactOutputSingle` (lines 130–147) sets the callback context, calls the pool, validates `amountOut` and `amountInMaximum`, then calls `_clearExpectedCallbackPool()` and returns — with no check on the remaining ETH balance. The same omission applies to `exactOutput` (lines 154–188).

The only recovery mechanism is `refundETH()` (lines 58–63), which is `external` with no access control and sends `address(this).balance` to `msg.sender`. An attacker who observes the victim's transaction in the mempool can back-run it with a standalone `refundETH()` call and collect the surplus.

The `receive()` guard (line 32–34) that restricts ETH acceptance to WETH does not mitigate this: `msg.value` attached to a `payable` function call is credited directly to the contract balance without invoking `receive()`.

## Impact Explanation
Direct loss of user principal. A user who calls `exactOutputSingle{value: X}` where `X > amountIn` (the amount the pool actually charges) permanently loses `X − amountIn` ETH unless they include a `refundETH()` call in the same `multicall`. A MEV bot or any observer can back-run the swap transaction with a standalone `refundETH()` call and collect the surplus. This meets the Sherlock threshold for a High-severity direct loss of user funds.

## Likelihood Explanation
For exact-output swaps the caller cannot know `amountIn` in advance; sending `msg.value = amountInMaximum` as a safety buffer is the natural and documented pattern (`amountInMaximum` is the slippage guard in `ExactOutputSingleParams`). Users who call `exactOutputSingle` directly (not via `multicall`) will routinely leave ETH in the router. The existing test suite covers the `multicall + refundETH` pattern only for `exactInputSingle`; no analogous test guards the exact-output path, confirming the gap is not a deliberate design choice.

## Recommendation
At the end of `exactOutputSingle` and `exactOutput`, automatically refund any remaining native ETH balance to `msg.sender`:

```solidity
// after _clearExpectedCallbackPool()
uint256 residual = address(this).balance;
if (residual > 0) _transferETH(msg.sender, residual);
```

Alternatively, enforce that `msg.value` equals the actual `amountIn` returned by the pool, reverting if the user over-sent. Either approach eliminates the window in which a third party can claim the surplus via `refundETH()`.

## Proof of Concept
1. User calls `exactOutputSingle{value: 2 ether}(...)` with `tokenIn = WETH`, `amountOut = X`, `amountInMaximum = 2 ether`.
2. Pool determines `amountIn = 1 ether` and invokes `metricOmmSwapCallback`.
3. `_justPayCallback` → `pay(WETH, user, pool, 1 ether)`: `nativeBalance (2 ether) >= value (1 ether)`, so 1 ether is wrapped and sent to the pool.
4. 1 ether remains as native ETH in the router; `exactOutputSingle` returns `amountIn = 1 ether` with no refund.
5. Attacker calls `refundETH()` in a subsequent transaction; `_transferETH(msg.sender, 1 ether)` sends the trapped ETH to the attacker.
