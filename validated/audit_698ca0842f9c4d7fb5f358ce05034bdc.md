### Title
`MerkleBlob::get_root_hash` Returns Stale Hash When Dirty Nodes Are Pending — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::py_get_root_hash` (the Python-exposed `get_root_hash()`) reads the stored root hash directly from the blob without first flushing pending ("dirty") internal-node hashes. After any `upsert()` or `delete()` operation, ancestor nodes are marked dirty but their hashes are not recalculated until `calculate_lazy_hashes()` is explicitly called. Because `get_root_hash()` never calls `calculate_lazy_hashes()` or checks the dirty flag, it silently returns the pre-mutation root hash, committing a stale tree root to the DataLayer chain state.

### Finding Description

`MerkleBlob` uses a lazy-hash design: when a leaf is updated via `upsert()`, the leaf's new hash is written immediately, but every ancestor internal node is only *marked dirty* via `mark_lineage_as_dirty()` — their stored hashes are not recalculated. [1](#0-0) 

Hash recalculation is deferred to an explicit `calculate_lazy_hashes()` call, which iterates dirty nodes in leaf-first order and recomputes each internal hash from its children. [2](#0-1) 

The Python binding `py_get_root_hash()` simply delegates to `py_get_hash_at_index(TreeIndex(0))`, which reads the raw stored bytes at index 0 with no dirty-node check and no call to `calculate_lazy_hashes()`. [3](#0-2) 

By contrast, `get_proof_of_inclusion()` *does* guard against dirty nodes — it returns `Err(Error::Dirty(...))` if any ancestor is dirty. [4](#0-3) 

The Python-facing API exposes `get_root_hash()` and `calculate_lazy_hashes()` as independent, unordered calls with no enforcement that the latter precedes the former. [5](#0-4) 

### Impact Explanation

If Python DataLayer code calls `upsert()` or `delete()` and then calls `get_root_hash()` before `calculate_lazy_hashes()`, the returned hash is the pre-mutation root. Any root hash committed to the chain from this stale value does not reflect the actual tree state. This corrupts the committed DataLayer tree root: inclusion proofs for the new state will fail against the committed root, while proofs for the old (superseded) state will still verify — allowing forged or invalid state to be proven against the on-chain root.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The vulnerability is latent in the API contract. Any Python caller that omits `calculate_lazy_hashes()` before `get_root_hash()` — a plausible omission given they are separate, unordered methods — triggers the bug. The `get_proof_of_inclusion()` guard demonstrates the codebase is aware of the dirty-node hazard, but `get_root_hash()` lacks the same protection.

### Recommendation

Inside `py_get_root_hash()` (or the underlying `get_hash_at_index` when called for the root), either:

1. Call `calculate_lazy_hashes()` before reading the root hash, or
2. Check whether the root block's `metadata.dirty` flag is set and return an error (analogous to the guard already present in `get_proof_of_inclusion()`).

```rust
#[pyo3(name = "get_root_hash")]
pub fn py_get_root_hash(&mut self) -> PyResult<Option<Hash>> {
    self.calculate_lazy_hashes()?;   // flush pending hashes first
    self.py_get_hash_at_index(TreeIndex(0))
}
```

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

h1 = hashlib.sha256(b"leaf1").digest()
blob.insert(KeyId(1), ValueId(1), h1)
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# Update the leaf — marks ancestors dirty, does NOT recalculate hashes
h2 = hashlib.sha256(b"leaf2").digest()
blob.upsert(KeyId(1), ValueId(2), h2)

# get_root_hash() returns the stale pre-upsert root
stale_root = blob.get_root_hash()
assert stale_root == root_before  # BUG: returns old root, not the updated one

blob.calculate_lazy_hashes()
correct_root = blob.get_root_hash()
assert correct_root != root_before  # correct root differs after lazy hash flush
```

    pub fn calculate_lazy_hashes(&mut self) -> Result<(), Error> {
        // OPT: yeah, storing the whole set of blocks via collect is not great
        for item in LeftChildFirstIterator::new_with_block_predicate(
            &self.blob,
            None,
            Some(|block: &Block| block.metadata.dirty),
        )
        .collect::<Vec<_>>()
        {
            let (index, mut block) = item?;
            assert!(block.metadata.dirty);

            let Node::Internal(ref leaf) = block.node else {
                panic!("leaves should not be dirty")
            };
            // OPT: obviously inefficient to re-get/deserialize these blocks inside
            //      an iteration that's already doing that
            let left_hash = self.get_hash(leaf.left)?;
            let right_hash = self.get_hash(leaf.right)?;
            block.update_hash(&left_hash, &right_hash);
            self.insert_entry_to_blob(index, &block)?;
        }

        Ok(())
    }
