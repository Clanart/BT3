Audit Report

## Title
Missing factory validation in `MetricOmmPoolLiquidityAdder` allows a malicious pool to drain user tokens up to caller-supplied caps — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address in every `addLiquidity*` entry point and stores it as the authorised callback caller without verifying it against the factory registry. Because the callback guard only checks `msg.sender == expectedPool`, an attacker-deployed contract that implements the pool interface passes the check unconditionally. The malicious pool then controls which tokens are pulled and from whom, enabling it to drain up to `maxAmountToken0 + maxAmountToken1` of any ERC-20 tokens the victim has approved to the adder.

## Finding Description

**No factory reference in the contract.** The constructor signature is `constructor(address weth) PeripheryPayments(weth)` — no factory address is stored, so no on-chain registry check is possible. [1](#0-0) 

**NatSpec acknowledges the gap explicitly.** The contract's own documentation states that it does not verify the pool against the factory and that a malicious pool can request token pulls up to the caller-provided caps. [2](#0-1) 

**Entry points forward the caller-supplied pool address without validation.** Both overloads of `addLiquidityExactShares` and `addLiquidityWeighted` pass the raw `pool` argument directly into `_addLiquidity`, which calls `_setPayContext(pool, ...)` to register it as the authorised callback caller. [3](#0-2) [4](#0-3) 

**Callback guard is trivially bypassed.** `metricOmmModifyLiquidityCallback` checks only `msg.sender != expectedPool`. Because `expectedPool` was set to the attacker-controlled address, the check passes unconditionally for the malicious pool. [5](#0-4) 

**Token resolution trusts the malicious pool.** After the guard, the callback calls `IMetricOmmPool(msg.sender).getImmutables()` to determine which tokens to pull. The malicious pool controls this return value and can name any ERC-20 the victim has approved. [6](#0-5) 

**Payment transfers tokens from victim to malicious pool.** `pay(token, payer, msg.sender, amount)` resolves to `safeTransferFrom(payer, recipient, value)`, moving tokens directly from the victim's wallet to the malicious pool, bounded only by the victim's stated caps. [7](#0-6) 

**The factory already exposes the required check.** `IMetricOmmPoolFactory.isPool(address pool)` returns `true` iff `poolToIdx[pool] != 0`, i.e., the pool was deployed and registered by the canonical factory. This check is unused by the adder. [8](#0-7) 

## Impact Explanation

A victim who has approved `MetricOmmPoolLiquidityAdder` (a prerequisite for any legitimate liquidity addition) and is induced to call any `addLiquidity*` function with an attacker-controlled pool address loses up to `maxAmountToken0 + maxAmountToken1` worth of arbitrary ERC-20 tokens in a single transaction. The tokens are transferred directly to the malicious pool with no recovery path. This constitutes a **direct loss of user principal**, meeting the Critical/High threshold under the Metric OMM allowed impact gate. [9](#0-8) 

## Likelihood Explanation

Any unprivileged attacker can deploy a malicious pool contract implementing `IMetricOmmPoolActions.addLiquidity` and `IMetricOmmPool.getImmutables`. The victim only needs to be directed to a spoofed or compromised frontend that substitutes the legitimate pool address with the attacker's. Because `MetricOmmPoolLiquidityAdder` is a shared periphery contract that all LPs must approve before adding liquidity, a single prior approval is sufficient. The attack requires no special role, no governance action, no flash loan, and no non-standard token behaviour. It is repeatable across any victim who has approved the adder. [2](#0-1) 

## Recommendation

Store the canonical factory address in the adder's constructor and add a registry check in `_addLiquidity` (and in the probe path of `addLiquidityWeighted`) before calling `_setPayContext`:

```solidity
require(IMetricOmmPoolFactory(factory).isPool(pool), "Unknown pool");
```

`IMetricOmmPoolFactory.isPool` is already implemented and returns `true` only for pools deployed through the canonical factory, making this a minimal, non-breaking fix. [10](#0-9) 

## Proof of Concept

```solidity
contract MaliciousPool {
    address victim; address adder; address token;
    uint256 constant MAX_AMOUNT = 1_000_000e18;

    constructor(address _victim, address _adder, address _token) {
        victim = _victim; adder = _adder; token = _token;
    }

    // Implements IMetricOmmPoolActions.addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta calldata,
                          bytes calldata callbackData, bytes calldata)
        external returns (uint256, uint256)
    {
        // callbackData already encodes KIND_PAY (1) — forwarded from _addLiquidity
        IMetricOmmPoolLiquidityAdder(adder)
            .metricOmmModifyLiquidityCallback(MAX_AMOUNT, 0, callbackData);
        return (MAX_AMOUNT, 0);
    }

    // Implements IMetricOmmPool.getImmutables
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token; // any token victim approved
    }
}

// Victim (or attacker on victim's behalf) calls:
adder.addLiquidityExactShares(
    address(maliciousPool), victim, 0, deltas,
    MAX_AMOUNT, 0, ""
);
// Result: MAX_AMOUNT of `token` transferred from victim → maliciousPool
```

Execution path: `addLiquidityExactShares` → `_addLiquidity` → `_setPayContext(maliciousPool, victim, MAX_AMOUNT, 0)` → `IMetricOmmPoolActions(maliciousPool).addLiquidity(...)` → malicious pool calls `metricOmmModifyLiquidityCallback(MAX_AMOUNT, 0, callbackData)` → guard passes (`msg.sender == expectedPool == maliciousPool`) → `getImmutables()` returns attacker-chosen token → `pay(token, victim, maliciousPool, MAX_AMOUNT)` → `safeTransferFrom(victim, maliciousPool, MAX_AMOUNT)`. [11](#0-10)
