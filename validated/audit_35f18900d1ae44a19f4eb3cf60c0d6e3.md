### Title
`batch_insert` Bypasses Duplicate-Key and Duplicate-Hash Validation, Corrupting DataLayer Merkle Tree Roots - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash precondition checks that `MerkleBlob::insert` enforces for all items beyond the first two. Supplying a batch that contains a key or hash already present in the tree (or duplicated within the batch itself) silently writes corrupt leaf nodes into the blob, producing an internally inconsistent tree whose root hash cannot be verified by any proof of inclusion.

### Finding Description

`MerkleBlob::insert` enforces two critical guards before writing any leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` has two code paths. When the tree has ≤ 1 existing leaf it pops up to two items and routes them through `self.insert(...)`, which does run the guards: [2](#0-1) 

All remaining items (the fast path, line 587 onward) are written directly via `insert_entry_to_blob` with **no** `contains_key` or `contains_leaf_hash` check: [3](#0-2) 

`insert_entry_to_blob` updates `block_status_cache` (it calls `add_leaf` internally, as evidenced by `insert_second` calling `clear()` then `insert_entry_to_blob` and expecting the cache to be repopulated). When a duplicate key `K` is written a second time:

1. The cache entry `K → index_old` is silently overwritten with `K → index_new`.
2. `index_old` remains as an orphaned leaf in the blob, invisible to the cache.
3. The subtree built from `indexes` (lines 607–654) incorporates **both** `index_old` and `index_new`, so the computed root hash reflects both leaves.
4. `get_proof_of_inclusion(K)` finds only `index_new` via the cache and builds a proof whose `root_hash()` does not match the actual tree root.
5. `proof.valid()` returns `false` for a legitimately inserted key, and any external verifier that trusts the stored root will reject valid state or accept forged state. [4](#0-3) 

The same corruption occurs when the same `Hash` value appears twice in the batch, because `leaf_hash_to_index` is also overwritten without error.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A corrupted root hash means:
- Proofs of inclusion generated after the corrupt batch will fail `valid()` checks, breaking DataLayer state verification.
- A node that accepted the corrupt batch will disagree with peers about the Merkle root, causing DataLayer consensus divergence between nodes.
- An attacker who controls the content of a DataLayer store can craft a batch that makes a syncing peer's tree root diverge from the canonical root, enabling forged inclusion/exclusion proofs against that peer.

### Likelihood Explanation

`batch_insert` is exposed directly through the Python binding `py_batch_insert`: [5](#0-4) 

Any Python caller (including DataLayer sync code that processes externally-supplied key-value deltas) can pass a list containing a duplicate key or a key already present in the tree. The only existing guard in `py_batch_insert` checks that `keys_values.len() == hashes.len()`; it does not check for duplicates. The trigger requires no special privilege — only the ability to supply a batch to a DataLayer node, which is the normal DataLayer sync scenario.

### Recommendation

Add the same precondition checks to the fast path of `batch_insert` that `insert` already enforces:

```rust
// In the fast-path loop (line 587):
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing write logic
}
```

Additionally, validate that no key or hash appears more than once within the batch itself before beginning any writes, to prevent intra-batch duplicates.

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate with 2 leaves so the fast path is taken for subsequent items
let h0 = Hash(Bytes32::new([0u8; 32]));
let h1 = Hash(Bytes32::new([1u8; 32]));
blob.insert(KeyId(0), ValueId(0), &h0, chia_datalayer::InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &h1, chia_datalayer::InsertLocation::Auto {}).unwrap();

// batch_insert fast path: key=2 appears twice with different hashes
let h2a = Hash(Bytes32::new([2u8; 32]));
let h2b = Hash(Bytes32::new([3u8; 32]));
let batch = vec![
    ((KeyId(2), ValueId(10)), h2a),
    ((KeyId(2), ValueId(20)), h2b),  // duplicate key — no error raised
];
blob.batch_insert(batch).unwrap();  // succeeds silently

blob.calculate_lazy_hashes().unwrap();

// Proof for key=2 is now invalid because the root reflects both orphaned leaves
let proof = blob.get_proof_of_inclusion(KeyId(2)).unwrap();
assert!(!proof.valid());  // tree root is corrupted
```
