### Title
Native Fee ETH Permanently Locked in OmniBridge With No Withdrawal Mechanism - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
In `OmniBridge.sol`, the `initTransfer` and `initTransfer1155` functions accept a `nativeFee` ETH payment from users initiating cross-chain transfers. This ETH accumulates in the contract balance but there is no function to withdraw or distribute it. The ETH is permanently locked.

### Finding Description

`initTransfer` is `payable` and requires the caller to send `msg.value` covering both the transfer `amount` (for native ETH transfers) and the `nativeFee`: [1](#0-0) 

For ERC20 transfers, `msg.value` equals `nativeFee` (plus any `extensionValue`): [2](#0-1) 

The `extensionValue` (everything beyond `nativeFee`) is passed to `initTransferExtension`, which in the base contract reverts if it is non-zero: [3](#0-2) 

This means the user must send exactly `nativeFee` ETH (for ERC20 transfers). That ETH is credited to the contract's balance and never forwarded to any recipient. The `finTransfer` function only sends `payload.amount` to the transfer recipient and has no logic to distribute the accumulated `nativeFee` ETH to any fee recipient: [4](#0-3) 

The contract has a bare `receive()` fallback confirming it can hold ETH, but no corresponding `withdraw` or `rescue` function exists anywhere in the contract: [5](#0-4) 

The README explicitly states that native chain tokens (ETH) are a valid fee payment method: [6](#0-5) 

On the NEAR side, the relayer's `native_fee` reward is minted as a wrapped token — it is not sourced from the locked ETH in the EVM contract. So the ETH paid as `nativeFee` on EVM has no corresponding outflow path and accumulates permanently.

### Impact Explanation

Every ERC20 `initTransfer` call with a non-zero `nativeFee` deposits ETH into the OmniBridge contract that can never be recovered. Over time, as users pay native fees to incentivize relayers, the total locked ETH grows without bound. There is no admin sweep, no fee-recipient forwarding, and no upgrade path that does not require a privileged key. This constitutes an irrecoverable lock of user-paid funds in the bridge vault flow, matching the allowed impact: **Critical — Permanent freezing, irrecoverable lock of user or protocol funds in bridge or vault flows**.

### Likelihood Explanation

The `nativeFee` parameter is a documented, first-class feature of the bridge protocol. Any user who pays a native fee to incentivize a relayer (the normal, expected usage) triggers this lock. Likelihood is **High** — it occurs on every standard ERC20 transfer with a non-zero native fee.

### Recommendation

Add a privileged withdrawal function (e.g., callable by `DEFAULT_ADMIN_ROLE`) that allows accumulated `nativeFee` ETH to be swept to a designated fee recipient:

```solidity
function withdrawNativeFees(address payable recipient, uint256 amount)
    external
    onlyRole(DEFAULT_ADMIN_ROLE)
{
    (bool success, ) = recipient.call{value: amount}("");
    if (!success) revert FailedToSendEther();
}
```

Alternatively, forward `nativeFee` directly to a fee recipient address at the time of `initTransfer`, similar to how the NEAR side mints the native fee to the relayer.

### Proof of Concept

1. User calls `initTransfer(tokenAddress=USDC, amount=1000e6, fee=10e6, nativeFee=0.01 ether, recipient="alice.near", message="")` sending `msg.value = 0.01 ether`.
2. Inside `initTransfer`: `extensionValue = msg.value - nativeFee = 0`. The base `initTransferExtension` does not revert (value == 0). The 0.01 ETH is now in the contract balance.
3. The relayer finalizes the transfer on NEAR and receives their fee as a minted wrapped token — the ETH in the EVM contract is untouched.
4. No function in `OmniBridge.sol` allows anyone to withdraw this ETH. It is permanently locked.
5. Repeat for every ERC20 bridge transfer with `nativeFee > 0`; the locked ETH grows monotonically with no recovery path.
