### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Unbacked Token Minting on NEAR — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `initTransfer` function in `OmniBridge.sol` records the user-supplied `amount` parameter directly in the `InitTransfer` event without verifying the actual amount of tokens received by the contract. For fee-on-transfer ERC20 tokens, the contract receives fewer tokens than `amount`, but the NEAR bridge mints the full `amount` of wrapped tokens. This is the direct analog of the external report's "outdated exchange rate" class: a stale/unverified value is used for accounting instead of the actual post-operation state.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native (non-bridge) ERC20 tokens by pulling `amount` from the caller and locking it in the contract:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // user-supplied, not verified post-transfer
    );
}
``` [1](#0-0) 

Immediately after, the same user-supplied `amount` is emitted in the `InitTransfer` event:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← not the actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The NEAR bridge's `fin_transfer_callback` reads the `InitTransfer` event proof and uses the `amount` field from the event to mint wrapped tokens on NEAR (after decimal denormalization):

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

For a fee-on-transfer ERC20 token, the actual amount received by the EVM contract is `amount - fee_deducted`, but the event records `amount`. NEAR mints `amount` wrapped tokens, creating supply that is not backed by locked collateral.

The root cause is structurally identical to the external report: a pre-operation or user-supplied value (`amount`) is used for accounting instead of the actual post-operation state (real balance delta), analogous to using `exchangeRateStored()` instead of `exchangeRateCurrent()`.

---

### Impact Explanation

**Critical/High — Balance and accounting corruption that breaks bridge collateralization.**

- The EVM vault holds `amount - transfer_fee` tokens.
- NEAR mints `amount` wrapped tokens.
- The bridge is undercollateralized by `transfer_fee` per such transfer.
- When users attempt to bridge back the full minted supply, the EVM contract cannot release all tokens — some users' funds are permanently unclaimable (irrecoverable lock of user funds).
- An attacker who deploys a fee-on-transfer token, logs its metadata (permissionless via `logMetadata`), and repeatedly initiates transfers can systematically drain the collateral backing of other users' positions.

---

### Likelihood Explanation

`logMetadata` is fully permissionless — any caller can register any ERC20 token:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [4](#0-3) 

An unprivileged attacker can deploy a fee-on-transfer ERC20, call `logMetadata`, wait for NEAR to register it, and then exploit `initTransfer`. No privileged access is required. The entry path is fully attacker-controlled.

---

### Recommendation

Measure the actual received amount using a balance-before/after pattern, and use the delta in the emitted event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(uint128(actualReceived) == actualReceived, "amount overflow");
amount = uint128(actualReceived);
```

Then use `amount` (now the actual received amount) in the `InitTransfer` event and all downstream accounting.

---

### Proof of Concept

1. Attacker deploys `FeeToken` — an ERC20 that deducts 10% on every `transferFrom`.
2. Attacker calls `logMetadata(feeTokenAddress)` on `OmniBridge.sol`. This is permissionless.
3. NEAR relayer picks up the `LogMetadata` event and registers `FeeToken`.
4. Attacker calls `initTransfer(feeTokenAddress, 1000, 0, 0, nearRecipient, "")`.
   - `safeTransferFrom` moves 1000 tokens from attacker, but `FeeToken` deducts 10%, so the bridge receives **900** tokens.
   - The `InitTransfer` event records `amount = 1000`.
5. NEAR relayer submits proof; `fin_transfer_callback` mints **1000** wrapped `FeeToken` on NEAR.
6. Bridge is now undercollateralized: 1000 NEAR-side tokens backed by only 900 EVM-side tokens.
7. Attacker bridges back 1000 tokens; the EVM contract attempts to release 1000 but only holds 900 — the last 100 tokens worth of redemptions by other users permanently fail. [5](#0-4) [6](#0-5)
