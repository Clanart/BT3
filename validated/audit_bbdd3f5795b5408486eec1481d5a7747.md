### Title
`batch_insert` Fast Path Skips Duplicate-Key Guard, Enabling DataLayer Tree Corruption — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` contains a conditional bootstrap block that calls the guarded `insert()` only when `leaf_count <= 1`. When the tree already has two or more leaves, the entire bootstrap block is skipped and **all** batch items are written directly to the blob without any duplicate-key check. This allows a caller to insert a `KeyId` that already exists in the tree, violating the invariant that each key appears at most once and corrupting the `BlockStatusCache::key_to_index` map.

---

### Finding Description

`insert()` enforces uniqueness at lines 369–371:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` calls `insert()` only inside the `leaf_count <= 1` guard (lines 578–585). When `leaf_count > 1`, execution falls straight through to the fast-path loop at lines 587–603, which writes every item directly via `insert_entry_to_blob()` with **no** `contains_key` check:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    // pops up to 2 items and calls self.insert() (guarded)
}

for ((key, value), hash) in keys_values_hashes {   // ← no duplicate check
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob()` unconditionally calls `add_leaf()`: [3](#0-2) 

`add_leaf()` silently overwrites the existing `key_to_index` entry for the duplicate key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);   // ← overwrites, no error
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [4](#0-3) 

After the call:
- The blob contains **two** leaf nodes sharing the same `KeyId` (at `OLD_INDEX` and `NEW_INDEX`).
- `key_to_index` maps the key to `NEW_INDEX` only; the old leaf at `OLD_INDEX` is still physically present in the blob but is now orphaned from the cache.
- The Merkle root is computed over a tree that includes both leaves, producing a root that commits to an invalid state.

---

### Impact Explanation

Two concrete consequences follow:

1. **Proof-of-inclusion forgery.** `get_proof_of_inclusion(K)` uses the cache to locate the leaf, so it generates a proof for `NEW_LEAF`. But `OLD_LEAF` is also reachable in the tree and contributes to the root. An attacker who controls the batch can choose the value stored in `NEW_LEAF`, effectively proving an arbitrary value for key `K` against the committed root. [5](#0-4) 

2. **Persistent state corruption.** `get_keys_values()` iterates `key_to_index` and returns only the value from `NEW_LEAF`; the old value is silently lost. Any downstream consumer that trusts the DataLayer root to reflect a canonical key→value mapping is operating on forged state. [6](#0-5) 

`check_integrity()` would detect the corruption (the leaf count in the blob exceeds `key_to_index.len()`, and the old leaf's cached index mismatches), but only if explicitly called after the fact — it is not called inside `batch_insert`. [7](#0-6) 

---

### Likelihood Explanation

Any caller that can supply the `Vec` argument to `batch_insert` can trigger this. The precondition (`leaf_count > 1`) is trivially satisfied after two normal `insert()` calls. No privileged access, key material, or network-level capability is required — only the ability to call the public API with a repeated `KeyId`.

---

### Recommendation

Add a duplicate-key check at the top of the fast-path loop in `batch_insert`, mirroring the guard in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Also add the same check for items that remain in the vec after the bootstrap block in the `leaf_count <= 1` branch, since those also bypass `insert()`.

### Proof of Concept

```rust
use chia_datalayer::merkle::blob::{MerkleBlob, InsertLocation};
use chia_datalayer::{Hash, KeyId, ValueId};

let mut blob = MerkleBlob::new(vec![]).unwrap();
let h1 = Hash::from([1u8; 32]);
let h2 = Hash::from([2u8; 32]);
let h3 = Hash::from([3u8; 32]);

// Bring leaf_count to 2 so the fast path is taken for all batch items
blob.insert(KeyId(10), ValueId(1), &h1, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(20), ValueId(2), &h2, InsertLocation::Auto {}).unwrap();

// batch_insert with a KeyId that already exists — no error returned
blob.batch_insert(vec![((KeyId(10), ValueId(99)), h3)]).unwrap();

// Tree now has two leaves with KeyId(10); integrity is broken
assert!(blob.check_integrity().is_err());

// get_keys_values returns only one entry for KeyId(10) (the duplicate),
// confirming the old leaf is orphaned
let kv = blob.get_keys_values().unwrap();
assert_eq!(kv.len(), 2); // only 2 unique keys tracked, but 3 leaves in blob
```

        key: KeyId,

            .block_status_cache
            .get_index_by_key(key)
            .ok_or(Error::UnknownKey(key))?;

        let node = self
            .get_node(index)?

        let mut layers: Vec<proof_of_inclusion::ProofOfInclusionLayer> = Vec::new();
        let mut parents_iter = parents.iter();
        // first in the lineage is the index itself, second is the first parent
        parents_iter.next();
        for (next_index, block) in parents_iter {
            if block.metadata.dirty {
                return Err(Error::Dirty(*next_index));
            }
            let parent = block
                .node

                other_hash: sibling.hash(),
