### Title
Hardcoded `failure_flag=0` in `execute_deploy` Syscall Ignores Constructor Failure — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` (success) into the syscall response regardless of whether the deployed contract's constructor actually succeeded. This is a direct analog to the EVM boolean-return assumption bug: just as the EVM bug assumes `transferFrom` always returns a truthy boolean, the OS here assumes `deploy` always succeeds. Any calling contract that inspects the `failure_flag` field to branch on deploy success/failure will be given incorrect information, potentially leading to funds being sent to an address where no contract was actually deployed.

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function calls `deploy_contract` and then unconditionally writes a success response:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The `failure_flag` field is hardcoded to `0` (success). The TODO comment explicitly acknowledges that failure handling is not yet implemented. The `deploy_contract` call returns `(retdata_size, retdata)` but no `is_reverted` indicator — and even if the constructor execution fails internally (e.g., out-of-gas, explicit panic), the OS writes `failure_flag=0` to the response buffer that the calling contract reads.

This is the exact same class of bug as the EVM report: an incorrect return-value assumption. The EVM contract assumed `transferFrom` always returns `true`; the OS here assumes `deploy` always returns success. [1](#0-0) 

### Impact Explanation

A Cairo 1 contract that calls the `deploy` syscall and then reads `response.failure_flag` to decide whether to proceed will always see `0` (success), even when the constructor failed and the contract was not actually deployed (or was deployed in a reverted/inconsistent state). If the calling contract subsequently transfers tokens or other assets to the `contract_address` returned in the `DeployResponse`, those assets are sent to an address with no functioning contract — permanently locking them. This constitutes **direct loss of funds** (Critical).

### Likelihood Explanation

Any unprivileged user can trigger this path by:
1. Deploying or calling a contract that uses the `deploy` syscall.
2. Supplying constructor calldata or gas conditions that cause the constructor to fail.
3. The OS reports success; the calling contract proceeds to send funds to the returned address.

No privileged role is required. The `deploy` syscall is a standard, publicly accessible syscall available to all Cairo 1 contracts.

### Recommendation

The `execute_deploy` handler must propagate the actual success/failure status of `deploy_contract` into the `failure_flag` field of the `ResponseHeader`. The `deploy_contract` function's return signature should be extended to include an `is_reverted` flag (analogous to how `contract_call_helper` in the same file handles `is_reverted`), and the response should be written as:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

If the constructor is reverted, the `DeployResponse` should reflect failure and the calling contract must not be given a valid `contract_address` to act upon.

### Proof of Concept

1. Attacker writes a Cairo 1 contract `Victim` that:
   - Calls `deploy(failing_class_hash, salt, calldata, false)`
   - Reads `response.failure_flag` — sees `0` (success)
   - Calls `token.transfer(response.contract_address, amount)`
2. The constructor of `failing_class_hash` consumes all gas and fails.
3. The OS writes `failure_flag=0` unconditionally at line 539 of `syscall_impls.cairo`.
4. `Victim` transfers `amount` tokens to `response.contract_address`.
5. No contract exists at that address (constructor was reverted); tokens are permanently locked. [2](#0-1)
