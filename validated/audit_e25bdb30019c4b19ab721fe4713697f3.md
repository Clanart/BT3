### Title
`locked_tokens` Accounting Understatement in `fast_fin_transfer_to_other_chain` Permanently Freezes Return Transfers - (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fast_fin_transfer_to_other_chain` locks only `amount_without_fee` on the destination chain, omitting the fee portion. However, `send_fee_internal` (called from `claim_fee_callback`) unconditionally unlocks `fee + dust` from that same destination chain. After claim, `locked_tokens[destination]` is understated by `fee.fee`. Any subsequent transfer of the received tokens back from the destination chain to NEAR panics with `ERR_INSUFFICIENT_LOCKED_TOKENS`, permanently freezing those funds.

---

### Finding Description

**Normal transfer path** (`process_fin_transfer_to_other_chain`, lines 1997–2022):

```
unlock origin:      amount.0
lock destination:   fee.fee          ← fee locked separately
lock destination:   amount_without_fee
─────────────────────────────────────
total locked[dest] = fee.fee + amount_without_fee = amount.0
```

After `claim_fee` → `send_fee_internal` unlocks `fee.fee + dust`:

```
remaining locked[dest] = amount.0 − (fee.fee + dust) = amount_without_fee − dust  ✓
```

This correctly represents the tokens actually on the destination chain.

**Fast transfer path** (`fast_fin_transfer_to_other_chain`, lines 914–973):

```rust
self.burn_tokens_if_needed(fast_transfer.token_id.clone(), amount_without_fee.into());
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,   // ← fee.fee is NEVER locked
);
``` [1](#0-0) 

Total locked on destination = `amount_without_fee` only (fee portion missing).

Then `claim_fee_callback` computes:

```rust
let denormalized_amount = Self::denormalize_amount(fin_transfer.amount.0, decimals);
let fee = transfer_message.amount.0 - denormalized_amount;  // = fee.fee + dust
self.send_fee_internal(&transfer_message, fee_recipient, fee)
``` [2](#0-1) 

Inside `send_fee_internal`:

```rust
self.unlock_tokens_if_needed(transfer_message.get_destination_chain(), &token, token_fee);
``` [3](#0-2) 

This unlocks `fee.fee + dust` from destination. After unlock:

```
locked[dest] = amount_without_fee − (fee.fee + dust) = amount_without_fee − fee.fee − dust
```

But the actual tokens on the destination chain = `normalize_amount(amount_without_fee)` ≈ `amount_without_fee − dust`.

**Understatement = `fee.fee`.**

When the recipient later initiates a return transfer from the destination chain to NEAR, `process_fin_transfer_to_near` calls:

```rust
let lock_actions = vec![self.unlock_tokens_if_needed(
    transfer_message.get_origin_chain(),   // = destination chain of fast transfer
    &token,
    transfer_message.amount.0,            // = amount_without_fee − dust
)];
``` [4](#0-3) 

`unlock_tokens` enforces:

```rust
require!(
    available >= amount,
    TokenLockError::InsufficientLockedTokens.as_ref()
);
``` [5](#0-4) 

Since `available = amount_without_fee − fee.fee − dust < amount_without_fee − dust = amount`, this panics and the return transfer is permanently blocked.

---

### Impact Explanation

Any user whose tokens were delivered via a fast transfer to a non-NEAR chain (e.g., ETH, SOL, StarkNet) with a non-zero fee cannot transfer those tokens back to NEAR. The `locked_tokens` counter is permanently understated by `fee.fee`, causing every return `fin_transfer` to revert. Funds are irrecoverably frozen in the destination chain contract.

This matches: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Fast transfers to non-NEAR chains with non-zero fees are a core, advertised protocol feature. Every such transfer followed by a `claim_fee` call silently corrupts `locked_tokens`. No malicious intent is required; normal relayer operation triggers the bug. The only precondition is `fee.fee > 0`, which is the standard case.

---

### Recommendation

In `fast_fin_transfer_to_other_chain`, lock the fee portion on the destination chain in addition to `amount_without_fee`, mirroring `process_fin_transfer_to_other_chain`:

```rust
// After existing burn + lock of amount_without_fee:
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    fast_transfer.fee.fee.0,   // ← add this
);
```

This ensures `locked_tokens[dest] = fee.fee + amount_without_fee`, so that after `claim_fee` unlocks `fee.fee + dust`, the remaining `amount_without_fee − dust` correctly represents the tokens on the destination chain and return transfers succeed.

---

### Proof of Concept

1. Trusted relayer calls `fast_fin_transfer` (via `ft_transfer_call`) with:
   - `recipient = OmniAddress::Sol(...)` (non-NEAR)
   - `amount = 1_000_000`, `fee.fee = 10_000`
   - `amount_without_fee = 990_000`

2. `fast_fin_transfer_to_other_chain` executes:
   - `lock_tokens_if_needed(Sol, token, 990_000)` → `locked[Sol] = 990_000`
   - Fee (`10_000`) is **not** locked.

3. Relayer signs and executes transfer on Solana. Recipient receives `normalize(990_000)` tokens.

4. Relayer calls `claim_fee` with Solana proof:
   - `token_fee = 1_000_000 − denormalize(normalize(990_000)) = 10_000 + dust`
   - `unlock_tokens_if_needed(Sol, token, 10_000 + dust)` → `locked[Sol] = 990_000 − 10_000 − dust = 979_999...`

5. Recipient initiates return transfer on Solana for `normalize(990_000)` tokens.

6. Relayer calls `fin_transfer` on NEAR with Solana proof:
   - `transfer_message.amount.0 = denormalize(normalize(990_000)) = 990_000 − dust`
   - `unlock_tokens_if_needed(Sol, token, 990_000 − dust)` → **panics**: `available (979_999) < amount (989_999)`

7. Return transfer permanently reverts. Recipient's funds are frozen on Solana.
