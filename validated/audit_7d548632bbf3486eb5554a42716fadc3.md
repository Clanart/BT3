### Title
Missing `chunk.shard_id()` == `shard_id` guard in `set_state_header` allows cross-shard state-root substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for `(shard_id=X, sync_hash=H)` whose embedded `ShardChunk` belongs to shard Y (Y ≠ X). Neither `validate_chunk_proofs` nor the position-independent `verify_path` call verifies that `chunk.shard_id() == shard_id`. Any network peer that is selected as a state-sync responder can exploit this to store `StateHeaderKey(X, H)` whose `chunk.prev_state_root()` is shard Y's state root, causing all subsequent state-part validation and trie installation for shard X to operate against shard Y's trie.

---

### Finding Description

**Entrypoint**: `set_state_header` at [1](#0-0) 

The function receives `shard_id` (the requested shard, X) and `shard_state_header` (attacker-controlled). It extracts the embedded chunk: [2](#0-1) 

**Check 1 — `validate_chunk_proofs`**: Verifies header hash self-consistency, `tx_root`, and `outgoing_receipts_root`. It does **not** check `chunk.shard_id()` against the requested `shard_id`: [3](#0-2) 

**Check 2 — `verify_path`**: Verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's `chunk_headers_root` Merkle tree: [4](#0-3) 

`verify_path` is purely position-independent — it only checks `compute_root_from_path(path, item_hash) == root`: [5](#0-4) 

The `chunk_headers_root` is a Merkle root over **all** shards' chunks in the block. Shard Y's chunk at position Y has a valid proof against the same root. Providing shard Y's chunk with shard Y's proof passes this check even when `shard_id=X` was requested. There is no `verify_path_with_index` call (which does check position) — only the position-blind `verify_path`.

The legitimate server builds the proof at `prev_shard_index` (position of X), but the attacker substitutes shard Y's chunk with shard Y's proof (position Y), and the check passes identically.

**Storage**: The corrupted header is stored under `StateHeaderKey(X, H)`: [6](#0-5) 

**Downstream propagation**:

`set_state_part` reads the stored header and validates all state parts against `chunk.prev_state_root()` — now shard Y's state root: [7](#0-6) 

`set_state_finalize` → `chain_update.set_state_finalize` applies the chunk starting from `chunk_header.prev_state_root()` — shard Y's state root — for shard X's `shard_uid`: [8](#0-7) 

The result is shard X's `ChunkExtra` being populated with shard Y's trie state.

**Attacker role**: The state-sync downloader uses `StateSyncDownloadSourcePeer`, which downloads from any network peer: [9](#0-8) 

Any peer selected from `highest_height_peers` can serve a crafted response. No validator, chunk-producer, or operator privilege is required.

---

### Impact Explanation

The exact corrupted value is `chunk.prev_state_root()` stored in `StateHeaderKey(X, H)` — shard Y's state root instead of shard X's. All state parts for shard X are validated against shard Y's state root, and `set_state_finalize` installs shard Y's trie as shard X's canonical state. The node then applies subsequent blocks for shard X starting from the wrong trie, producing wrong execution results, wrong `ChunkExtra` state roots, and wrong outgoing receipts for shard X for the remainder of the epoch.

---

### Likelihood Explanation

Exploiting this requires a peer to be selected as the state-sync responder for the victim node. In the P2P state-sync protocol any peer with sufficient height can be selected. A full node (tracking all shards) has all the data needed: shard Y's chunk, shard Y's Merkle proof, shard X's incoming receipt proofs (needed for step 4), and shard Y's `state_root_node`. The attack is constructive and requires no cryptographic forgery.

---

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, replace the position-blind `verify_path` call with `verify_path_with_index` (which already exists in `core/primitives/src/merkle.rs`) to bind the chunk to its expected shard index position in the Merkle tree: [10](#0-9)

    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);

    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid

        byzantine_assert!(false);
        return Ok(false);
