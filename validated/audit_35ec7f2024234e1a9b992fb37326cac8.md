### Title
Unauthenticated `RemoteProofManagerServer` `SetProof` Endpoint Allows Bypassing Cryptographic Proof Verification, Causing Invalid Transactions to Be Accepted — (`File: crates/apollo_proof_manager/src/communication.rs`)

---

### Summary

The `ProofManager` exposes a `SetProof` RPC endpoint via `RemoteProofManagerServer` bound to `0.0.0.0` with no caller authentication. Any network-reachable client can pre-populate the proof storage with an arbitrary `(ProofFacts, Proof)` pair without cryptographic verification. When a subsequent transaction carrying matching `proof_facts` arrives at the gateway, `run_proof_verification` detects the pre-stored entry via `contains_proof` and **skips** `verify_proof` entirely, allowing a transaction backed by an invalid proof to pass gateway admission and enter the block.

---

### Finding Description

**Root cause — no authentication on `SetProof`:**

`communication.rs` dispatches every incoming `ProofManagerRequest::SetProof` directly to `ProofManager::set_proof` with no identity check on the caller:

```rust
ProofManagerRequest::SetProof(proof_facts, proof) =>
    ProofManagerResponse::SetProof(
        self.set_proof(proof_facts, proof).await ...
    ),
```

`ProofManager::set_proof` stores the proof keyed by `proof_facts.hash()` after only checking whether the key already exists — it never calls `verify_proof`:

```rust
pub async fn set_proof(&self, proof_facts: ProofFacts, proof: Proof) -> ... {
    if self.contains_proof(proof_facts.clone()).await? { return Ok(()); }
    let facts_hash = proof_facts.hash();
    self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
    self.cache.insert(facts_hash, proof);
    Ok(())
}
```

The remote server is configured to bind on `0.0.0.0` in the production distributed deployment:

```json
"components.proof_manager.remote_server_config.bind_ip": "0.0.0.0"
```

**Broken invariant — `run_proof_verification` skips verification when proof is pre-stored:**

`TransactionConverter::run_proof_verification` (the shared path for both gateway and consensus flows) short-circuits on a cache/storage hit:

```rust
let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
if contains_proof {
    return Ok(false);   // ← cryptographic verify_proof is never called
}
// Only reached when proof is NOT already stored:
starknet_proof_verifier::verify_proof(proof_facts, proof)?;
```

An attacker who calls `SetProof` directly on the `RemoteProofManagerServer` before submitting a transaction causes `contains_proof` to return `true`, permanently suppressing `verify_proof` for that `proof_facts` hash.

**Downstream acceptance path:**

The blockifier's `validate_proof_facts` (called from `perform_pre_validation_stage`) checks only the *content* of the proof facts — program hash membership in `allowed_virtual_os_program_hashes`, block number recency, stored block hash equality, and config hash — but never touches the cryptographic proof object. A transaction whose proof facts pass those content checks but whose proof is cryptographically invalid will therefore clear every sequencer-side gate and be included in the block.

---

### Impact Explanation

An attacker pre-populates the proof manager with `(valid_proof_facts, garbage_proof)`. Any subsequent Invoke V3 transaction embedding those `proof_facts` is accepted by the gateway (proof verification skipped), passes blockifier pre-validation (content checks pass), is executed, and its state changes are committed. The block commitment, state diff, and SNOS input all reflect a transaction that was never backed by a valid proof. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The `RemoteProofManagerServer` binds to `0.0.0.0` in the production distributed deployment with no TLS and no authentication layer in the `RemoteComponentServer` infrastructure. Any host that can reach the proof manager's TCP port — including a compromised co-tenant in the same cluster or an external host if the port is inadvertently exposed — can issue `SetProof` calls. The attack requires only a single unauthenticated RPC call followed by a normal gateway transaction submission.

---

### Recommendation

1. **Authenticate callers on `RemoteProofManagerServer`**: Restrict `SetProof` to known internal clients (e.g., mutual TLS, network policy, or an explicit allowlist of caller identities).
2. **Verify before storing**: Move `verify_proof` into `ProofManager::set_proof` so that no proof is persisted without passing cryptographic verification, regardless of the call path.
3. **Remove the skip-if-stored shortcut** or make it conditional on the stored proof having been verified (e.g., store a verified flag alongside the proof).

---

### Proof of Concept

1. Attacker connects directly to the `RemoteProofManagerServer` port (bound `0.0.0.0`).
2. Attacker sends `ProofManagerRequest::SetProof(valid_proof_facts, garbage_proof)`.
   - `ProofManager::set_proof` stores `garbage_proof` under `valid_proof_facts.hash()` without calling `verify_proof`.
3. Attacker submits an Invoke V3 transaction to the HTTP gateway with `proof_facts = valid_proof_facts` and `proof = garbage_proof`.
4. Gateway calls `convert_rpc_tx_to_internal_rpc_tx` → `spawn_proof_verification` → `run_proof_verification`:
   - `contains_proof(valid_proof_facts)` → `true` (pre-populated in step 2).
   - `verify_proof` is **not called**; function returns `Ok(false)`.
5. Gateway calls `perform_pre_validation_stage` in the blockifier:
   - `validate_proof_facts` checks program hash ∈ `allowed_virtual_os_program_hashes` ✓, block number ✓, block hash ✓, config hash ✓.
   - All content checks pass; the invalid proof is never inspected.
6. Transaction is admitted to the mempool, included in the next block, executed, and its state changes committed — with no valid cryptographic proof ever verified.

        proof_facts: ProofFacts,

        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

            return Ok(false);

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {

        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
