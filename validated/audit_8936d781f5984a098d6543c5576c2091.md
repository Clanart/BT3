Looking at the `set_state_header` function in `chain/chain/src/state_sync/adapter.rs`, I found a direct analog to the Vader H-06 bug: the function receives a `shard_id` parameter but never validates it against `chunk.shard_id()` from the peer-supplied header, while using both identities for different critical operations.

### Title
Missing `chunk.shard_id()` Validation Against `shard_id` Parameter in `set_state_header` Enables Cross-Shard State Corruption - (File: chain/chain/src/state_sync/adapter.rs)

### Summary

`set_state_header` accepts a `shard_id` parameter (the shard being synced) and a peer-supplied `ShardStateSyncResponseHeader` containing a chunk. The function uses `shard_id` for receipt-hash computation and storage keying, but uses `chunk.shard_id()` (implicitly, via `chunk_inner.prev_state_root()`) for state-root validation — without ever asserting the two are equal. A malicious peer can supply shard Y's chunk inside a header claimed to be for shard X. Every individual check passes, the header is stored, and subsequent `set_state_part` / `apply_state_part` calls write shard Y's trie data into shard X's storage slot, permanently corrupting the syncing node's state for shard X.
