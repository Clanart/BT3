### Title
Missing Storage Gap in `BridgeToken` Corrupts `_systemAddress` in `HyperliquedBridgeToken` on Upgrade — (File: evm/src/omni-bridge/contracts/BridgeToken.sol)

---

### Summary

`BridgeToken` is an upgradeable UUPS parent contract that declares three sequential storage variables (`_name`, `_symbol`, `_decimals`) but provides **no `__gap` reservation**. `HyperliquedBridgeToken` inherits from `BridgeToken` and appends `_systemAddress` immediately after `_decimals` in the storage layout. If `BridgeToken` is ever upgraded to introduce a new state variable, `_systemAddress` is displaced to a different slot while the proxy storage retains the old layout, silently reading `address(0)` for `_systemAddress` in all subsequent calls. This permanently breaks the HyperCore→HyperEVM bridge path and corrupts the HyperCore spot-balance accounting.

---

### Finding Description

**Storage layout of a deployed `HyperliquedBridgeToken` proxy (current):**

All OZ v5 base contracts (`Initializable`, `UUPSUpgradeable`, `ERC20Upgradeable`, `Ownable2StepUpgradeable`) use EIP-7201 namespaced storage and occupy no sequential slots. Sequential storage therefore begins at slot 0 with `BridgeToken`'s own variables:

| Slot | Variable | Source |
|------|----------|--------|
| 0 | `_name` (string) | `BridgeToken` |
| 1 | `_symbol` (string) | `BridgeToken` |
| 2 | `_decimals` (uint8, 1 B) + `_systemAddress` (address, 20 B) — packed | `BridgeToken` + `HyperliquedBridgeToken` |

`_decimals` and `_systemAddress` are packed together in slot 2 because 1 + 20 = 21 bytes < 32 bytes. [1](#0-0) [2](#0-1) 

**`BridgeToken` has no `__gap`:**

The contract ends after `_decimals` with no gap variable, unlike `OmniBridge` which correctly reserves `uint256[49] private __gap`. [3](#0-2) 

**Storage layout after upgrading `BridgeToken` to add one new variable (e.g., `uint256 _newVar`):**

| Slot | Variable | Source |
|------|----------|--------|
| 0 | `_name` | `BridgeToken` v2 |
| 1 | `_symbol` | `BridgeToken` v2 |
| 2 | `_decimals` (uint8) | `BridgeToken` v2 |
| 3 | `_newVar` (uint256) | `BridgeToken` v2 — **NEW** |
| 4 | `_systemAddress` (address) | `HyperliquedBridgeToken` — **SHIFTED** |

The proxy's physical storage still holds `_systemAddress` packed into slot 2. The new implementation reads `_systemAddress` from slot 4, which is zero. Every subsequent read of `_systemAddress` returns `address(0)`.

Even adding a smaller type (e.g., `uint8` or `address`) that packs with `_decimals` in slot 2 pushes `_systemAddress` from slot 2 to slot 3, producing the same `address(0)` read.

---

### Impact Explanation

Two critical consequences follow from `_systemAddress` silently becoming `address(0)`:

**1. Permanent freeze of the HyperCore→HyperEVM bridge path.**

`coreReceiveWithData` guards all HyperCore callbacks with:

```solidity
if (msg.sender != _systemAddress) revert NotSystemAddress();
```

With `_systemAddress == address(0)`, `msg.sender` can never equal `address(0)`, so every call reverts. No HyperCore user can ever release tokens to HyperEVM or initiate a cross-chain transfer via this path again. The lock is irrecoverable without a further upgrade that re-initializes `_systemAddress` — but re-initialization is blocked by `initializer` guards. [4](#0-3) 

**2. Silent token burn / accounting corruption on the 3-arg `mint` path.**

The 3-arg `mint` (used for HyperCore-bound tokens) calls:

```solidity
_update(account, _systemAddress, value);
```

With `_systemAddress == address(0)`, this transfers minted tokens to the zero address (burning them), while the HyperCore spot-balance ledger still credits the user. This breaks bridge collateralization: HyperCore balances are unbacked by any EVM-side token supply. [5](#0-4) 

Both impacts fall within the allowed scope:
- *Permanent freezing / irrecoverable lock of user funds in bridge flows.*
- *Balance / accounting corruption that breaks bridge collateralization.*

---

### Likelihood Explanation

`BridgeToken` is explicitly designed to be upgraded via `OmniBridge.upgradeToken`. Adding new fields to `BridgeToken` (e.g., a fee-on-transfer flag, a blacklist mapping, a permit nonce) is a routine and expected upgrade operation. The missing `__gap` provides no compile-time or runtime warning; the storage corruption is silent and only manifests at runtime after the upgrade is already live. A developer upgrading `BridgeToken` without auditing the full inheritance tree of every deployed proxy (including `HyperliquedBridgeToken`) will trigger this silently. [6](#0-5) 

---

### Recommendation

Add a `__gap` at the end of `BridgeToken`'s storage variable block and reduce its size by one each time a new variable is introduced:

```solidity
// BridgeToken.sol — after _decimals
uint256[47] private __gap; // 50 slots minus 3 used (_name, _symbol, _decimals)
```

This mirrors the pattern already applied in `OmniBridge`: [3](#0-2) 

Additionally, consider using the `openzeppelin-contracts-upgradeable` package exclusively (already done for base contracts) and running `hardhat-upgrades` storage layout checks (`upgrades.validateUpgrade`) in CI to catch layout violations before deployment.

### Proof of Concept

**Setup:**
1. Deploy a `HyperliquedBridgeToken` proxy via `ERC1967Proxy`. After `initialize`, the proxy storage has `_systemAddress` = `0xSYSTEM` packed into slot 2 alongside `_decimals`.

**Trigger (routine admin upgrade):**
2. Developer adds `uint256 public transferCount;` to `BridgeToken` after `_decimals` and deploys `BridgeToken` v2.
3. Admin calls `OmniBridge.upgradeToken(hlTokenProxy, bridgeTokenV2Impl)` — a normal, non-malicious operation.

**Corruption:**
4. The new `HyperliquedBridgeToken` implementation (compiled against `BridgeToken` v2) now reads `_systemAddress` from slot 4. Slot 4 in the proxy storage is `0x0000...0000`.
5. `_systemAddress` is now `address(0)`.

**Exploit path A — permanent freeze:**
6. Any HyperCore user triggers `sendToEvmWithData`, causing the HL system to call `coreReceiveWithData` on the token. The call reverts with `NotSystemAddress()`. All HyperCore→HyperEVM settlements are permanently frozen.

**Exploit path B — accounting corruption:**
7. `OmniBridge.finTransfer` calls the 3-arg `mint(recipient, amount, message)`. Inside, `_update(recipient, address(0), amount)` burns the freshly minted tokens. The recipient receives nothing; HyperCore still shows a balance. The bridge is undercollateralized by `amount` per call. [7](#0-6) [8](#0-7) [9](#0-8)
