### Title
Missing `finalised_utxo_transfers` insertion in `utxo_fin_transfer_fast` (Near branch) enables double-settlement of the same UTXO — (`near/omni-bridge/src/lib.rs`)

### Summary
When `utxo_fin_transfer_fast` handles a fast-transfer whose destination is `ChainKind::Near`, it removes the fast-transfer record from `fast_transfers` but never inserts the UTXO identifier into `finalised_utxo_transfers`. A subsequent `utxo_fin_transfer` call with the same `utxo_id` finds no fast-transfer entry and no finalization record, passes `add_fin_utxo_transfer` without error, and mints a second batch of tokens to the recipient — violating the at-most-once settlement invariant.

### Finding Description

In `utxo_fin_transfer_fast`, the Near-destination branch and the other-chain branch are asymmetric:

```
// other-chain branch — leaves a permanent tombstone
self.mark_fast_transfer_as_finalised(&fast_transfer.id());   // finalised=true stays in fast_transfers

// Near branch — deletes the record entirely, writes nothing to finalised_utxo_transfers
self.remove_fast_transfer(&fast_transfer.id());              // entry gone
// ← add_fin_utxo_transfer is NEVER called here
``` [1](#0-0) 

After the Near branch executes, the bridge's two deduplication stores are both empty for this UTXO:

- `fast_transfers` — entry deleted by `remove_fast_transfer`
- `finalised_utxo_transfers` — never written

A second invocation of `utxo_fin_transfer` with the same `utxo_id` reaches the normal (non-fast) path:

```rust
// get_fast_transfer_status returns None → skips utxo_fin_transfer_fast
let required_storage_balance =
    self.add_fin_utxo_transfer(&utxo_fin_transfer_msg.get_transfer_id(origin_chain));
// ↑ succeeds — set was never populated for this id
``` [2](#0-1) 

It then calls `utxo_fin_transfer_to_near`, which mints `amount` tokens to the recipient a second time. [3](#0-2) 

The connector-identity guard (`sender_id == config.connector`) is the only access control on `utxo_fin_transfer`, but the connector's `verify_deposit` is a **public, permissionless function** that any account can call with arbitrary parameters and no internal deduplication: [4](#0-3) 

The `add_fin_utxo_transfer` guard that is supposed to prevent double-finalization: [5](#0-4) 

…is only reached on the non-fast path. The fast-path Near branch bypasses it entirely and leaves no tombstone.

### Impact Explanation

An attacker who performed (or observed) a fast transfer to a Near recipient can call `verify_deposit` on the connector a second time with the same `utxo_id`. The bridge mints a full second batch of tokens to the recipient. The connector's token balance is drained twice for one Bitcoin UTXO, creating unbacked circulating supply. This is a direct, on-chain double-spend of bridged BTC tokens.

### Likelihood Explanation

The attack requires two steps both executable by an unprivileged account:
1. Initiate or observe a fast transfer to a Near recipient (public `ft_transfer_call`).
2. Call `verify_deposit` on the connector twice with the same `UtxoFinTransferMsg` (public function, no deduplication in the connector).

No privileged role, leaked key, or off-chain coordination is needed. The connector check is satisfied automatically because `verify_deposit` is the legitimate call path. The only prerequisite is that a fast transfer to a Near address has already been executed, which is the normal operating mode of the bridge.

### Recommendation

In `utxo_fin_transfer_fast`, when the destination is `ChainKind::Near`, insert the UTXO transfer ID into `finalised_utxo_transfers` **before** removing the fast-transfer entry, mirroring the deduplication that the normal (non-fast) path performs:

```rust
let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
+   self.add_fin_utxo_transfer(
+       &utxo_fin_transfer_msg.get_transfer_id(origin_chain),
+   );
    self.remove_fast_transfer(&fast_transfer.id());
    fast_transfer.amount
} else {
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());
    ...
};
```

This ensures that after the fast-path completes, the UTXO identifier is permanently recorded in `finalised_utxo_transfers`, and any subsequent call with the same id will panic with `ERR_UTXO_TRANSFER_ALREADY_FINALISED`.

### Proof of Concept

State-based reproduction (no cryptographic proof required):

1. Relayer calls `ft_transfer_call` on the token contract with `BridgeOnTransferMsg::FastFinTransfer` for `utxo_id=X`, `recipient=OmniAddress::Near(alice)`. Bridge records entry in `fast_transfers`; Alice receives `amount - fee` tokens.
2. Relayer calls `verify_deposit(amount, utxo_msg{utxo_id=X, recipient=Near(alice)})` on the connector. Connector calls `ft_transfer_call` → bridge's `utxo_fin_transfer` → `utxo_fin_transfer_fast` (Near branch) → `remove_fast_transfer` (entry deleted, `finalised_utxo_transfers` untouched) → relayer reimbursed.
3. Relayer calls `verify_deposit(amount, utxo_msg{utxo_id=X, recipient=Near(alice)})` on the connector **again**. Bridge's `utxo_fin_transfer` finds no fast-transfer entry, calls `add_fin_utxo_transfer` (succeeds), calls `utxo_fin_transfer_to_near` → Alice receives `amount` tokens a second time.

Assert: Alice's balance increased by `(amount - fee) + amount`; connector's token balance decreased by `2 × amount` for a single UTXO deposit.
