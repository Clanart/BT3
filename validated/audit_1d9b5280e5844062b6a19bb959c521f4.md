### Title
`replace_class` Syscall Accepts Undeclared Class Hash Without Verification, Enabling Permanent Contract Freezing â€” (`execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not verify that the new class hash provided to the `replace_class` syscall corresponds to a previously declared contract class. This is a direct state-transition bypass: the state update (class hash replacement) is committed to the protocol without the prerequisite condition (class declaration) being enforced. Any contract can replace its class hash with an arbitrary undeclared value, permanently breaking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` inside `syscall_impls.cairo`, the OS reads the requested class hash from the syscall request and immediately writes it into `contract_state_changes` with no check that the class hash exists in the declared class tree: [1](#0-0) 

The critical missing check is explicitly acknowledged by the developers:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `contract_class_changes` dict â€” which tracks declared classes within the block â€” is never consulted. The OS has access to it as an implicit argument throughout syscall execution (visible in `execute_syscalls`'s signature), but `execute_replace_class` does not receive or check it: [2](#0-1) 

Compare this to `execute_declare_transaction`, which correctly enforces `prev_value=0` to prevent double-declaration and writes into `contract_class_changes`: [3](#0-2) 

The `replace_class` path has no symmetric check reading from `contract_class_changes` to confirm the target class exists.

**Analog to M-19**: In M-19, `receiveFunds()` (state update) executes before the actual funds arrive (prerequisite), because two coupled operations can be separated. Here, `replace_class` (state update: class hash written to state tree) executes without the prerequisite (class declaration) being verified. Both are state-transition bypasses where the OS/protocol commits a state change without confirming the dependency that makes that state valid.

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value and the block is proven and accepted on L1:

- All future calls to that contract will fail at class lookup â€” the Sierra/CASM class for the new hash does not exist in the class tree.
- Any ERC-20 tokens, ETH (fee token), or other assets held in the contract's storage become permanently inaccessible.
- The state commitment (`compute_contract_state_commitment`) will faithfully commit the broken class hash into the global state root, making the damage irreversible on-chain. [4](#0-3) 

This matches: **Critical â€” Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is reachable by any unprivileged contract deployer or caller:

1. An attacker deploys a contract (or exploits an existing upgradeable proxy).
2. The contract calls `replace_class` with an arbitrary felt value that has never been declared.
3. The OS processes the block, the proof is generated, and the state is finalized on L1 with the broken class hash committed.

No privileged role, leaked key, or external dependency is required. The entry path goes through `execute_syscalls` â†’ `execute_replace_class`: [5](#0-4) 

A realistic scenario: a proxy contract calls `replace_class` targeting a class hash that was declared in a *later* transaction in the same block (or not at all). The OS accepts the upgrade unconditionally, the class is never resolvable, and the proxy â€” potentially holding user funds â€” is permanently frozen.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the class hash exists in the declared class set. Concretely, `execute_replace_class` should receive `contract_class_changes` as an implicit argument and perform a lookup (e.g., `dict_read`) to confirm the target class hash has a non-zero compiled class hash entry. If the lookup fails, write a failure response instead of updating state.

---

### Proof of Concept

1. Attacker deploys contract `C` holding 1000 STRK tokens.
2. Attacker sends an invoke transaction calling a function in `C` that executes `replace_class(0xdeadbeef)` â€” an arbitrary undeclared hash.
3. The OS calls `execute_replace_class`; the TODO check is absent; `contract_state_changes` is updated: `C.class_hash = 0xdeadbeef`.
4. The block is proven. `compute_contract_state_commitment` commits the new state root including `C`'s broken class hash.
5. The L1 core contract accepts the state update.
6. Any subsequent call to `C` (e.g., to withdraw the 1000 STRK) fails: the class `0xdeadbeef` does not exist in the class tree.
7. The 1000 STRK are permanently frozen. [1](#0-0)
