### Title
Unbounded `Predecessors` Iteration in `TraceTx` gRPC Handler Enables Node Resource-Exhaustion DoS — (`x/evm/keeper/grpc_query.go`)

### Summary

The `TraceTx` gRPC handler iterates over the caller-supplied `req.Predecessors` slice without any count or cumulative-gas limit, executing every predecessor transaction through the full EVM state-transition engine (`ApplyMessageWithConfig`). Because the gRPC server is bound to `0.0.0.0:9900` by default and requires no authentication, any unprivileged network peer can submit a single `QueryTraceTxRequest` carrying thousands of high-gas predecessors, forcing the node to perform unbounded EVM computation and exhausting CPU/memory until the process crashes or becomes unresponsive.

### Finding Description

**Root cause — no predecessor count or gas cap in `TraceTx`**

`x/evm/keeper/grpc_query.go` lines 542–563:

```go
for i, tx := range req.Predecessors {          // ← no len() guard
    ethTx := tx.AsTransaction()
    msg, err := core.TransactionToMessage(ethTx, signer, cfg.BaseFee)
    if err != nil { ... continue }
    rsp, err := k.ApplyMessageWithConfig(ctx, msg, cfg, true)  // full EVM exec
    if err != nil { ... continue }
    cfg.TxConfig.LogIndex += uint(len(rsp.Logs))
}
```

`req.Predecessors` is a `repeated MsgEthereumTx` protobuf field with no server-side size constraint. Each element is executed via `ApplyMessageWithConfig`, which runs the complete EVM state transition including storage reads/writes, contract calls, and gas accounting. The gas limit for each predecessor comes directly from the attacker-supplied transaction field; unlike `EstimateGas` and `EthCall`, `TraceTx` never applies `queryMaxGasLimit`:

```go
// EstimateGas / EthCall — gas cap IS enforced:
gasCap := req.GasCap
if k.queryMaxGasLimit != GasNoLimit {
    if gasCap == 0 { gasCap = k.queryMaxGasLimit }
    else if k.queryMaxGasLimit < gasCap { gasCap = k.queryMaxGasLimit }
}
// TraceTx — NO equivalent cap; msg.GasLimit comes straight from the tx
```

**Attacker-controlled entry path**

The gRPC server default binding is `0.0.0.0:9900` (publicly reachable):

```go
// server/config/config.go
DefaultGRPCAddress = "0.0.0.0:9900"
```

The `TraceTx` RPC is declared as a public query method in `proto/ethermint/evm/v1/query.proto` and requires no authentication. An attacker constructs a `QueryTraceTxRequest` with:

- `Predecessors`: N = 10 000 entries, each a valid `MsgEthereumTx` with `Gas = 25 000 000` (the default `DefaultGasCap`) and a data payload that exercises expensive opcodes (SSTORE loops, etc.)
- `Msg`: any valid transaction (even a trivial transfer)

The node will execute N × 25 M gas worth of EVM computation in a single synchronous gRPC call, consuming all available CPU cores and gigabytes of heap before the call returns or the process is killed by the OS OOM killer.

**Contrast with the JSON-RPC path**

When `debug_traceTransaction` is called via the JSON-RPC backend (`rpc/backend/tracing.go` lines 64–78), predecessors are populated from the actual on-chain block:

```go
for _, txBz := range blk.Block.Txs[:transaction.TxIndex] {
    ...
    predecessors = append(predecessors, ethMsg)
}
```

This is bounded by the real block size. The vulnerability is that the underlying gRPC endpoint accepts the `Predecessors` field directly from any caller without applying the same bound.

**`TraceBlock` is similarly affected**

`TraceBlock` (`x/evm/keeper/grpc_query.go` lines 622–643) iterates over `req.Txs` with no count limit, executing each via `prepareTrace` → `ApplyMessageWithConfig`. A direct gRPC call with thousands of `Txs` entries triggers the same unbounded execution.

### Impact Explanation

An unprivileged attacker sends one crafted gRPC request to port 9900. The node's goroutine handling that request enters an unbounded loop of full EVM executions. Because `ApplyMessageWithConfig` is CPU- and memory-intensive (stateDB snapshots, KV store reads, EVM interpreter loop), a request with 10 000 predecessors at 25 M gas each can consume hundreds of CPU-seconds and gigabytes of memory in a single call. The node process crashes (OOM) or becomes unresponsive. Since the gRPC server is bound to `0.0.0.0` by default, validator nodes are reachable. A crashed or stalled validator fails to sign blocks, and if enough validators are targeted simultaneously, the chain halts.

This maps to the allowed impact: **"Valid unprivileged RPC submission can halt the chain"** and **"Public gRPC path exposes a reachable route to chain halt"**.

### Likelihood Explanation

- The gRPC endpoint is publicly reachable by default (`0.0.0.0:9900`).
- No authentication or rate-limiting is applied to `TraceTx`.
- The attack requires only a single gRPC call; no on-chain funds, keys, or governance access are needed.
- The `QueryTraceTxRequest` protobuf message is well-documented and trivially constructable with any gRPC client.
- The attack is cheap: the attacker pays nothing; the victim node bears all computation cost.

### Recommendation

1. **Add a predecessor count cap** in `TraceTx` before the loop:
   ```go
   const maxPredecessors = 1000
   if len(req.Predecessors) > maxPredecessors {
       return nil, status.Errorf(codes.InvalidArgument,
           "too many predecessors: %d > %d", len(req.Predecessors), maxPredecessors)
   }
   ```
2. **Apply `queryMaxGasLimit` to each predecessor** the same way `EstimateGas`/`EthCall` do, capping `msg.GasLimit` before calling `ApplyMessageWithConfig`.
3. **Add a cumulative gas budget** across all predecessors (analogous to `GasBudget` in the `eth_simulateV1` simulator).
4. **Apply the gRPC server timeout** (`EVMTimeout`) to `TraceTx` and `TraceBlock` handlers, not only to `eth_call`.
5. **Bind the gRPC server to `127.0.0.1` by default** or document that public exposure requires explicit operator opt-in.
6. Apply the same fixes to `TraceBlock`'s `req.Txs` iteration.

### Proof of Concept

```
# Attacker machine (no keys, no funds required)
grpcurl -plaintext -d '{
  "msg": { ... any valid MsgEthereumTx ... },
  "predecessors": [
    { "gas": "25000000", "data": "<SSTORE-loop bytecode>" },
    ... repeated 10000 times ...
  ],
  "block_number": 1
}' <validator-ip>:9900 ethermint.evm.v1.Query/TraceTx
```

Expected outcome: the validator node's CPU spikes to 100 %, memory grows until OOM, and the process is killed or becomes unresponsive. Block production stalls on that validator. Repeating the call against a quorum of validators halts the chain.
