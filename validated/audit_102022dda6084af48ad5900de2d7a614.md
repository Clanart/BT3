### Title
`gasWanted` uint64 Underflow in `CheckEthGasConsume` Bypasses Block Gas Limit Check — (`ante/eth.go`)

---

### Summary

The overflow guard in `CheckEthGasConsume` uses `math.MaxInt64 - gasLimit` in unsigned 64-bit arithmetic. When `gasLimit > math.MaxInt64`, the subtraction underflows (wraps to a large uint64), making the guard permanently FALSE. A two-message batch transaction can exploit this to wrap `gasWanted` to zero, silently bypassing the block gas limit check.

---

### Finding Description

The guard at `ante/eth.go` line 152:

```go
if gasWanted > math.MaxInt64-gasLimit {
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow", ...)
}
gasWanted += gasLimit
```

is evaluated entirely in `uint64` arithmetic. When `gasLimit > math.MaxInt64` (= 9223372036854775807), the expression `math.MaxInt64 - gasLimit` underflows:

```
9223372036854775807 - gasLimit  →  wraps to a large uint64 near MaxUint64
```

So `gasWanted > <large_uint64>` is always `false`, and the guard never fires.

**Two-message wrap-to-zero attack:**

| Step | gasLimit | gasWanted after | Check result |
|------|----------|-----------------|--------------|
| msg1 | `blockGasLimit` (e.g. 40 M) | 40 M | block-gas check: 40M ≤ 40M ✓ |
| msg2 | `MaxUint64 − 40M + 1` | `40M + (MaxUint64 − 40M + 1)` = **0** (wrap) | block-gas check: 0 ≤ 40M ✓ |

After the loop `gasWanted = 0`, and the context is set with `NewInfiniteGasMeterWithLimit(0)`.

A secondary, identical overflow exists in `ValidateEthBasic` (`ante/interfaces/setup.go` line 112):

```go
txGasLimit += msgEthTx.GetGas()   // no overflow guard
```

`txGasLimit` also wraps to 0, so the attacker simply sets `authInfo.Fee.GasLimit = 0` to satisfy the equality check at line 149. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

When the attack succeeds:

1. **Block gas limit bypass** — the tx is admitted to the mempool and committed to a block even though its true aggregate gas limit (`blockGasLimit + MaxUint64 − blockGasLimit + 1 ≈ MaxUint64`) vastly exceeds the block gas limit. The EVM executes msg2 with a gas limit of `≈ MaxUint64`; actual gas consumed is bounded only by the computation, not the block limit.

2. **Fee market manipulation** — `gasWanted = 0` is propagated into `ctx.BlockGasWanted()`, which `EndBlock` uses to recalculate `baseFee`. Reporting zero gas wanted for a block drives the base fee downward, distorting the EIP-1559 fee market. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

**Precondition — fee constraint is the primary blocker.**

`VerifyFee` computes `feeAmt = effectivePrice × gasLimit`. For msg2 with `gasLimit ≈ MaxUint64`:

- If `baseFee > 0`: `effectivePrice ≥ baseFee > 0`, so `feeAmt` is astronomically large and `DeductTxCostsFromUserBalance` fails — attack is blocked.
- If `baseFee = 0` **and** `gasPrice = 0`: `feeAmt = 0`, the deduction succeeds, and the attack goes through. [5](#0-4) 

The attack is therefore **only reachable when the fee market is disabled or `baseFee` has not yet risen above zero** (e.g., genesis, test chains, or chains that deliberately disable EIP-1559). On chains with an active fee market the fee for msg2 is unpayable, making exploitation infeasible in practice.

---

### Recommendation

Replace the broken guard with a correct uint64 overflow check:

```go
// Correct: check addition overflow before performing it
if gasLimit > math.MaxUint64-gasWanted {
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow uint64", gasWanted, gasLimit)
}
gasWanted += gasLimit
if gasWanted > math.MaxInt64 {
    return ctx, fmt.Errorf("gasWanted(%d) exceeds MaxInt64", gasWanted)
}
```

Apply the same fix to `txGasLimit` accumulation in `ValidateEthBasic`. [6](#0-5) [7](#0-6)
