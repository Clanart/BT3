### Title
Missing Recipient Validation in `init_transfer_sol` Causes Permanent SOL Lock — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer_sol.rs`, `src/state/message/init_transfer.rs`)

---

### Summary

`InitTransferSol::process` accepts any arbitrary string as `recipient` with no format or validity check. An unprivileged user who supplies an invalid recipient string (empty, too long, or containing invalid characters) causes SOL to be transferred into the `sol_vault` PDA and a Wormhole VAA to be emitted. When NEAR's prover attempts to parse the VAA, it fails to construct a valid `OmniAddress` from the malformed string and panics. Because the Solana program has no cancel or refund instruction, the SOL is permanently locked in `sol_vault` with no recovery path.

---

### Finding Description

**Step 1 — No validation on Solana.**

`InitTransferPayload.recipient` is a plain `String`: [1](#0-0) 

`InitTransferSol::process` only checks `fee == 0` and `amount > 0`; the `recipient` field is never inspected: [2](#0-1) 

SOL (`amount + native_fee`) is unconditionally transferred to `sol_vault`, and the raw recipient string is serialized into the Wormhole message: [3](#0-2) 

**Step 2 — NEAR prover parses the recipient string.**

On the NEAR side, `ParsedVAA::try_into::<InitTransferMessage>` calls:

```rust
recipient: transfer.recipient.parse().map_err(stringify)?,
``` [4](#0-3) 

This invokes `OmniAddress::from_str`, which for an empty string defaults to parsing `""` as an EVM hex address (fails), and for `"near:<invalid>"` tries to construct a NEAR `AccountId` (fails for empty, >64 chars, or invalid characters): [5](#0-4) 

**Step 3 — NEAR bridge panics; finalization is impossible.**

`fin_transfer_callback` panics with `InvalidProofMessage` if the prover returns any error: [6](#0-5) 

**Step 4 — No recovery path on Solana.**

The only instruction that releases SOL from `sol_vault` is `finalize_transfer_sol`, which requires a valid MPC-signed `FinalizeTransferPayload` originating from a successful NEAR finalization: [7](#0-6) 

There is no cancel, timeout, or admin-drain instruction in the Solana program. The SOL is permanently locked.

---

### Impact Explanation

Any user who calls `init_transfer_sol` with a recipient string that is not a parseable `OmniAddress` (e.g., `""`, `"near:"`, `"near:a!b"`, a 65-character NEAR account ID) permanently loses their SOL. The `sol_vault` PDA accumulates these funds with no mechanism to return them. This matches the Critical impact category: **irrecoverable lock of user funds in a bridge vault flow**.

---

### Likelihood Explanation

The call is fully permissionless — any wallet can invoke `init_transfer_sol` with an arbitrary payload. A user who mistypes a recipient, uses a UI that passes a raw string without the `chain:address` prefix, or deliberately crafts a malformed string triggers the lock. No privileged access, key compromise, or external dependency failure is required.

---

### Recommendation

Validate `recipient` in `InitTransferSol::process` (and symmetrically in `InitTransfer::process`) before accepting funds. At minimum:

1. Require the string to be non-empty.
2. Require it to parse as a valid `OmniAddress` using the same `FromStr` logic used on the NEAR side (or an equivalent Rust implementation on Solana).
3. Reject the transaction with a clear error code (`ErrorCode::InvalidRecipient`) if validation fails, so the user's SOL is never moved.

Alternatively, add a protocol-level refund/cancel instruction on Solana that allows the original sender to reclaim SOL for a transfer whose origin nonce was never finalized on NEAR within a timeout window.

---

### Proof of Concept

```
1. User calls init_transfer_sol with:
     amount    = 1_000_000 lamports
     recipient = ""          // or "near:", or "near:a!b", or 65-char string
     fee       = 0
     native_fee = 0

2. Solana program:
   - Passes require!(fee == 0) ✓
   - Passes require!(amount > 0) ✓
   - Transfers 1_000_000 lamports from user → sol_vault PDA ✓
   - Posts Wormhole message with recipient = "" ✓

3. Wormhole guardians sign the VAA.

4. Relayer submits VAA to NEAR fin_transfer.

5. NEAR wormhole-omni-prover-proxy calls:
     transfer.recipient.parse()  // OmniAddress::from_str("")
     // "" has no ':', defaults to ("eth", "")
     // "".parse::<H160>() → Err("ERR_INVALID_HEX")
     // TryInto returns Err(...)

6. fin_transfer_callback receives prover error:
     env::panic_str("InvalidProofMessage")  // NEAR tx fails

7. No FinalizeTransferPayload is ever signed by NEAR MPC.

8. finalize_transfer_sol can never be called.

9. 1_000_000 lamports remain permanently locked in sol_vault.
   assert!(sol_vault.lamports increased by 1_000_000)
   assert!(no finalize_transfer_sol can succeed for this nonce)
```

        require!(payload.fee == 0, ErrorCode::InvalidFee);

            self.user.key(),
            Pubkey::default(),
