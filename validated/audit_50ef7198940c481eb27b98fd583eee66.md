Audit Report

## Title
Unpartitioned `address(this).balance` in `pay` allows attacker to consume stranded ETH via zero-cost WETH swap — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay` helper in `PeripheryPayments` uses the router's entire `address(this).balance` when settling a WETH-input payment, with no per-call or per-user accounting. ETH left in the router from a prior user's transaction (e.g., excess ETH sent with a WETH swap before `refundETH` is called) is indistinguishable from ETH the current caller contributed, allowing an attacker to submit a WETH swap with `msg.value = 0` and have the victim's stranded ETH settle it at zero cost.

## Finding Description

When `token == WETH` and `payer != address(this)`, `pay` reads `address(this).balance` and, if it covers `value`, wraps and forwards it to the pool without pulling anything from `payer`:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } else if (nativeBalance > 0) {
        ...
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

The `receive()` guard only blocks bare ETH transfers (no calldata); it does not block ETH attached to any `payable` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, etc.). [2](#0-1) 

`refundETH` returns the entire balance to `msg.sender` but is never called automatically at the end of a swap entry point, so excess ETH persists in the router across transactions: [3](#0-2) 

The transient storage (`TransientCallbackPool`) tracks the expected callback pool, callback mode, payer address, and token-to-pay, but records **no per-call ETH budget**. There is no slot that records how much ETH `msg.sender` contributed in the current transaction, so `pay` has no way to limit itself to the current caller's contribution. [4](#0-3) 

The callback path that invokes `pay` is `_justPayCallback`, called from `metricOmmSwapCallback` for both `exactInputSingle` and `exactOutputSingle`: [5](#0-4) 

**Exploit path:**
1. User A calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18})` with `msg.value = 1.5e18`. The callback wraps 1 ETH; 0.5 ETH remains in the router.
2. User A submits `refundETH()` to recover the surplus.
3. Attacker front-runs with `exactInputSingle({tokenIn: WETH, amountIn: 0.5e18, msg.value: 0})`. The callback fires; `pay(WETH, attacker, pool, 0.5e18)` reads `address(this).balance == 0.5e18`, wraps User A's ETH, and sends WETH to the pool. Attacker pays nothing.
4. User A's `refundETH()` executes: `address(this).balance == 0`. User A loses 0.5 ETH.

## Impact Explanation

Direct loss of user principal (native ETH). The victim loses the full amount of ETH stranded in the router; the attacker receives a complete WETH swap at zero cost. The loss is bounded only by the stranded amount, which equals the excess ETH the victim sent — an amount entirely under the victim's (and attacker's) control. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation

ETH stranding is a normal operational pattern: users routinely overpay ETH with WETH swaps and rely on a subsequent `refundETH` call. Any window between the swap transaction and the `refundETH` transaction — including mempool latency, failed multicall legs, or user error — leaves ETH exploitable. An attacker needs only to monitor the router's ETH balance on-chain and submit a WETH swap with `msg.value = 0` before the victim's `refundETH` is mined. No special privileges, tokens, or setup are required.

## Recommendation

Track per-call ETH contributions in a dedicated transient slot (EIP-1153, already used via `TransientCallbackPool`). At each `payable` entry point, store `msg.value` in a new `T_MSG_VALUE_SLOT`. In `pay`, cap the native balance consumed to `min(address(this).balance, tload(T_MSG_VALUE_SLOT))` and deduct from the slot as ETH is used. Clear the slot at the end of each entry point alongside `_clearExpectedCallbackPool()`. This ensures ETH contributed by one caller cannot be consumed by a different caller's callback.

## Proof of Concept

```solidity
// Foundry fork test outline
function test_strandedEthStolenByAttacker() public {
    // 1. User A swaps WETH exact-input, overpays ETH
    vm.prank(userA);
    router.exactInputSingle{value: 1.5e18}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 1e18, ...})
    );
    assertEq(address(router).balance, 0.5e18); // 0.5 ETH stranded

    // 2. Attacker front-runs refundETH with a zero-cost WETH swap
    vm.prank(attacker);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 0.5e18, ...})
    );
    // Attacker's swap settles using userA's 0.5 ETH; attacker pays nothing

    assertEq(address(router).balance, 0);

    // 3. User A's refundETH returns nothing
    vm.prank(userA);
    router.refundETH();
    assertEq(userA.balance, initialBalance - 1.5e18); // full 0.5 ETH lost
}
```
