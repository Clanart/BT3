### Title
Relative Timelock Bypass via Integer Wrapping in `check_time_locks` (`nowrap=false` Path) — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false` (the pre-hard-fork consensus path), relative timelock arithmetic uses `wrapping_add` instead of `saturating_add`. An unprivileged attacker can craft a CLVM spend condition with a large `height_relative` or `seconds_relative` value that causes integer overflow, making the timelock check silently pass when it should reject the spend. This is a direct analog to the external report's missing boundary check: just as `GenesisGroup` continued accepting purchases past `maxGenesisPrice`, `check_time_locks` continues accepting spends past their relative timelock expiry when the sum overflows.

---

### Finding Description

In `check_time_locks`, two separate branches exist depending on the `nowrap` flag:

**`nowrap=true` (post-hard-fork, correct):**
```rust
unspent.confirmed_block_index.saturating_add(height_relative)
```

**`nowrap=false` (pre-hard-fork, vulnerable):**
```rust
unspent.confirmed_block_index.wrapping_add(height_relative)
```

For `ASSERT_HEIGHT_RELATIVE`, the check is:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

If `confirmed_block_index + height_relative` overflows `u32`, the result wraps to a small value (e.g., 0 or 9). The comparison `prev_height < small_value` then evaluates to `false`, so **no error is returned** and the spend is accepted — even though the timelock has not expired.

The same flaw applies to `seconds_relative` (u64 wrapping):

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

The codebase's own test suite explicitly documents this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- spend incorrectly passes when nowrap=false
)]
```

The attacker-controlled entry path is the `height_relative` / `seconds_relative` value embedded in a CLVM spend condition, parsed by `parse_args` via `sanitize_uint` with `max_size=4` (u32) or `max_size=8` (u64). Both are fully attacker-controlled.

---

### Impact Explanation

This is a **timelock validation bypass** enabling unauthorized spend acceptance. A coin protected by `ASSERT_HEIGHT_RELATIVE N` (or `ASSERT_SECONDS_RELATIVE N`) can be spent immediately — before the required number of blocks or seconds have elapsed — by choosing `N` such that `confirmed_block_index + N` overflows the integer type. The spend is accepted by consensus as valid, even though the coin's own timelock condition has not been satisfied. This matches the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

---

### Likelihood Explanation

The `nowrap=false` path is active for all blocks validated under pre-hard-fork consensus rules. The Python binding exposes `check_time_locks` with `nowrap` as a caller-supplied boolean, and the full node passes `nowrap=False` for pre-hard-fork heights. Any node that has not yet crossed the hard-fork activation height, or that validates historical blocks, uses this path. The attacker only needs to submit a spend bundle with a crafted `height_relative` or `seconds_relative` value — no privileged access, leaked keys, or network-level attack is required.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally in all four relative timelock checks, eliminating the `nowrap` branch entirely for the arithmetic step. The `nowrap=true` path already demonstrates the correct approach. The `nowrap` flag (if still needed for other semantic reasons) should not gate overflow safety.

---

### Proof of Concept

**Setup:** Coin confirmed at block height `H = 100`. Attacker crafts a spend with:

```
ASSERT_HEIGHT_RELATIVE = 0xffff_ff9c   (= 2^32 - 100 = 4_294_967_196)
```

**Arithmetic (nowrap=false):**
```
100u32.wrapping_add(4_294_967_196u32)
= (100 + 4_294_967_196) mod 2^32
= 4_294_967_296 mod 2^32
= 0
```

**Check:**
```
prev_transaction_block_height (e.g., 200) < 0  →  false  →  no error
```

The spend is accepted at block 200, even though the intended timelock requires waiting until block `100 + 4_294_967_196` — effectively never under correct arithmetic. The codebase's own test vector confirms this outcome: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
