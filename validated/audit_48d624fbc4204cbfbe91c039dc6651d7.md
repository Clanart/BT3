### Title
Missing Class Hash Validation in `execute_replace_class` Allows Arbitrary State Corruption - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The OS-level handler for the `replace_class` syscall does not validate that the caller-supplied class hash refers to a previously declared class. An unprivileged contract can invoke `replace_class` with an arbitrary, undeclared class hash. The OS accepts and commits this value to state without any whitelist or existence check, directly analogous to the MarginSwap finding where caller-supplied contract addresses were accepted without validation.

---

### Finding Description

In `execute_replace_class`, the new `class_hash` is read directly from the syscall request and written into `contract_state_changes` with no check that the hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
``` [1](#0-0) 

The `TODO` comment is an explicit acknowledgment by the developers that this validation is absent. The `class_hash` field originates entirely from the contract's syscall segment, which is written by the executing contract and is therefore fully attacker-controlled.

When a subsequent transaction calls the contract whose class hash has been replaced with an undeclared value, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [2](#0-1) 

Since the class hash was never declared, `dict_read` returns `0` (the default for an uninitialized entry). The OS then calls:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

`find_element` asserts the element exists. If no compiled class with hash `0` is present in the bundle, the OS panics and cannot produce a valid proof for the block.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

Once a contract's class hash is replaced with an undeclared value and committed to state, every subsequent block that includes a transaction targeting that contract will cause the OS to fail during proof generation. Because the corrupted state entry is committed to the Patricia Merkle Tree and propagated to L1, the sequencer cannot simply omit the contract — the state is permanently poisoned. Any block touching the affected contract is unprovable, halting finality for those transactions. If the affected contract is a widely-used one (e.g., a token or bridge), this constitutes a network-level denial of service.

Additionally, if the contract holds funds, those funds are permanently frozen because no valid execution of the contract can be proven.

---

### Likelihood Explanation

The OS is the authoritative protocol validator. The `replace_class` syscall is callable by any deployed contract — no privileged role is required. A user deploys a contract containing a `replace_class(arbitrary_felt)` call, submits a transaction invoking it, and the OS processes the syscall without any existence check. The explicit `TODO` comment confirms this is a known, unimplemented guard. The sequencer's blockifier may independently validate class hashes, but the OS is the security boundary for proof correctness; a discrepancy between blockifier and OS validation is itself a protocol-level gap.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, the OS must verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` with `class_hash` as the key and assert the returned `compiled_class_hash` is non-zero before proceeding with the state update. This mirrors the check already performed in `execute_entry_point` when resolving a class for execution.

---

### Proof of Concept

1. Deploy contract `Attacker` containing:
   ```
   syscall replace_class(class_hash=0xdeadbeef)  // 0xdeadbeef is never declared
   ```
2. Submit an invoke transaction calling `Attacker.__execute__`.
3. The OS processes `execute_replace_class`: `class_hash = 0xdeadbeef` is written to `contract_state_changes` for `Attacker`'s address with no validation. [1](#0-0) 
4. The block is proven successfully (the replace itself does not trigger the failure).
5. Submit a second transaction calling `Attacker` (e.g., a transfer to it).
6. The OS calls `execute_entry_point` for `Attacker`. `dict_read(contract_class_changes, 0xdeadbeef)` returns `0`. `find_element(..., key=0)` fails to locate a compiled class and panics. [4](#0-3) 
7. The OS cannot generate a proof for the block. The block is permanently unprovable. Network finality halts for any block containing a transaction to `Attacker`.
