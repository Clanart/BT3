### Title
Missing hash-uniqueness guard in `MerkleBlob::upsert` allows duplicate leaf hashes, corrupting the DataLayer Merkle tree — (`crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::upsert` removes the existing leaf from the cache before checking whether the supplied `new_hash` is already used by a different leaf. This allows an attacker who controls one key (K1) to assign another key's hash (H2) to it, silently overwriting the `leaf_hash_to_index` cache entry for K2 and producing a tree with two leaves sharing the same hash.

### Finding Description
In `upsert` (line 798), `block_status_cache.remove_leaf(&leaf)` is called unconditionally before any uniqueness check on `new_hash`. After removal, the old hash (H1) is gone from the cache, but H2 (belonging to K2) is still present. The code then calls `insert_entry_to_blob` (line 803), which internally calls `add_leaf` (line 1025), which does `leaf_hash_to_index.insert(leaf.hash, index)` — a plain `HashMap::insert` that silently overwrites the existing H2→K2 mapping with H2→K1. No `HashAlreadyPresent` error is ever raised.

`insert` correctly guards against this with an explicit `contains_leaf_hash` check before any mutation. `upsert` has no such guard.

### Impact Explanation
- **`leaf_hash_to_index` cache corruption**: K2's entry is permanently lost; `get_node_by_hash(H2)` returns K1's data.
- **Duplicate leaf hashes in the blob**: Both K1 and K2 now store hash H2, making the Merkle root computation incorrect.
- **DataLayer tree root mismatch**: Any downstream consumer comparing the committed root against a recomputed root will observe a divergence.
- **`check_integrity()` failure**: The length invariant `leaf_count == leaf_hash_to_index.len()` is violated (one entry was overwritten, so the map has one fewer entry than leaves).

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic … corrupts tree roots … or lets untrusted input prove invalid state."*

### Likelihood Explanation
Any caller with write access to a DataLayer store who owns key K1 can trigger this by calling `upsert(K1, any_value, H2)` where H2 is the known hash of any other key K2. No privileged access is required beyond the ability to call `upsert`, which is the normal update API.

### Recommendation
Add a hash-uniqueness check in `upsert` **before** calling `remove_leaf`. The check must account for the case where the key is being updated to the same hash (which should be allowed):

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // Guard: reject if new_hash is already used by a DIFFERENT key
    if &leaf.hash != new_hash && self.block_status_cache.contains_leaf_hash(new_hash) {
        return Err(Error::HashAlreadyPresent());
    }

    self.block_status_cache.remove_leaf(&leaf)?;
    // ... rest unchanged
}
```
