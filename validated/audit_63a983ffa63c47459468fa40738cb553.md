### Title
`nativeFee` ETH Permanently Locked in OmniBridge Contract with No Relayer Claim or User Refund Mechanism — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

When a user calls `initTransfer` on the EVM `OmniBridge` contract with a non-zero `nativeFee`, the corresponding ETH is transferred into the contract and has no on-chain path to reach a relayer or be returned to the user. The contract exposes no `withdrawNativeFee`, `claimNativeFee`, or equivalent function. The `nativeFee` value is only emitted in the `InitTransfer` event as a signal to the NEAR-side relayer; the ETH itself is irrecoverably locked in the EVM contract.

---

### Finding Description

In `OmniBridge.sol` `initTransfer` (lines 373–437), the caller sends `msg.value` that must equal `nativeFee` for ERC-20 transfers (or `amount + nativeFee` for native ETH transfers):

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 392-393
extensionValue = msg.value - nativeFee;
// ...
initTransferExtension(..., extensionValue);
```

The base `initTransferExtension` (lines 492–506) reverts if `extensionValue != 0`, so for ERC-20 transfers `msg.value` must equal exactly `nativeFee`. That ETH remains in the contract. The only downstream use of `nativeFee` is its inclusion in the emitted event:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
```

The NEAR relayer reads this event off-chain and uses the `nativeFee` value as a priority signal, but the ETH itself never leaves the EVM contract. No function in `OmniBridge.sol` allows the relayer, the user, or the admin to withdraw accumulated `nativeFee` ETH. The contract does have `receive() external payable {}` (line 574), confirming it can hold ETH, but no corresponding withdrawal path exists.

The analog on the NEAR side (`fee.native_fee`) is deducted from the user's storage balance and stored in the `TransferMessage`; `claim_fee` on NEAR handles only the token-denominated fee (`fee.fee`), not the EVM-side ETH. There is therefore no cross-chain mechanism that routes the EVM `nativeFee` ETH to the relayer either.

---

### Impact Explanation

Every user who calls `initTransfer` with `nativeFee > 0` permanently loses that ETH. The funds accumulate in the OmniBridge contract with no on-chain recovery path for the user, the relayer, or the protocol. This constitutes an irrecoverable lock of user funds in a bridge flow. The incentive mechanism is also broken: the relayer is supposed to be compensated by `nativeFee`, but receives nothing on the EVM side, while the user pays and loses the fee regardless of whether the transfer is ever processed.

Impact classification: **High** — fee/accounting corruption that permanently misdirects user value in the bridge flow.

---

### Likelihood Explanation

The `nativeFee` parameter is a first-class field in the public `initTransfer` interface and is documented as the mechanism for users to incentivize relayers. Any user who follows the standard bridge flow and sets `nativeFee > 0` (which is the expected behavior for timely processing) is immediately and silently affected. No special attacker capability is required; the loss occurs for every ordinary bridge user who pays a native fee.

---

### Recommendation

1. Implement a `claimNativeFee` function that allows the relayer who submitted the corresponding proof on NEAR to withdraw the accumulated `nativeFee` ETH on EVM, keyed by origin nonce.
2. Alternatively, if `nativeFee` is intended only as an off-chain signal and not as an on-chain payment, revert or refund any non-zero `msg.value` that exceeds the token transfer amount, so users are not silently charged.
3. Add a guarded `rescueNativeFee` admin function as a short-term mitigation to prevent indefinite accumulation.

---

### Proof of Concept

1. Alice calls `initTransfer(usdcAddress, 1000e6, 0, 1e18, "near:alice.near", "")` with `msg.value = 1e18` (1 ETH as `nativeFee`).
2. `OmniBridge` burns/locks 1000 USDC from Alice.
3. `extensionValue = 1e18 - 1e18 = 0`; `initTransferExtension` passes.
4. `InitTransfer` event is emitted with `nativeFee = 1e18`.
5. The NEAR relayer observes the event, submits the proof to NEAR `fin_transfer`, and calls `sign_transfer` to release tokens on the destination chain.
6. Alice's 1 ETH remains in `OmniBridge` indefinitely. The relayer receives only the NEAR-side token fee via `claim_fee`. Alice has no function to call to recover her 1 ETH. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
