### Title
Non-Final Finality Levels in MPC Prover Enable Reorganization-Based Unauthorized Token Minting on NEAR - (File: near/omni-prover/mpc-omni-prover/src/lib.rs)

---

### Summary

The `mpc-omni-prover` is hardcoded at initialization to use `EvmFinality::Latest` for the Abstract chain and `StarknetFinality::AcceptedOnL2` for Starknet. Both are explicitly pre-final states that can be reorganized before settlement on L1. If a source-chain reorganization removes an `initTransfer` transaction after the MPC network has already signed off on it and the NEAR bridge has already credited tokens, the attacker retains minted/unlocked assets on NEAR while the source-chain lock or burn no longer exists. The NEAR bridge has no mechanism to reverse a finalized transfer.

---

### Finding Description

In `MpcOmniProver::init()`, the prover is initialized with two non-final finality levels:

```rust
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Latest));
finalities.insert(
    ChainKind::Strk,
    MpcFinality::Starknet(StarknetFinality::AcceptedOnL2),
);
``` [1](#0-0) 

`EvmFinality::Latest` is the tip of the chain — not safe, not finalized. `StarknetFinality::AcceptedOnL2` is the Starknet sequencer's acceptance, which is explicitly pre-final: Starknet documentation and the SDK's own `StarknetFinality` enum distinguish `AcceptedOnL2` from `AcceptedOnL1` (L1-settled). The `request_matches_finality` check enforces that submitted proofs must carry exactly the configured finality tag, so the prover will accept and forward proofs at these pre-final levels:

```rust
fn request_matches_finality(request: &ForeignChainRpcRequest, finality: &MpcFinality) -> bool {
    match (request, finality) {
        (
            ForeignChainRpcRequest::Ethereum(args) | ForeignChainRpcRequest::Abstract(args),
            MpcFinality::Evm(finality),
        ) => args.finality == *finality,
        (ForeignChainRpcRequest::Starknet(args), MpcFinality::Starknet(finality)) => {
            args.finality == *finality
        }
        _ => false,
    }
}
``` [2](#0-1) 

Once the MPC network signs the payload and the NEAR bridge's `fin_transfer_callback` runs, the transfer ID is permanently inserted into `finalised_transfers`:

```rust
fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
    let storage_usage = env::storage_usage();
    require!(
        self.finalised_transfers.insert(transfer_id),
        BridgeError::TransferAlreadyFinalised.as_ref()
    );
    ...
}
``` [3](#0-2) 

There is no path to remove an entry from `finalised_transfers` after a source-chain reorganization. The `remove_fin_transfer` helper exists only for internal storage-refund bookkeeping in specific callback failure paths, not for reorganization recovery. [4](#0-3) 

The `set_finality` function is `#[private]` (only callable by the contract itself via DAO governance), so the default non-final levels are the live production configuration unless explicitly changed. [5](#0-4)

### Impact Explanation

An attacker who initiates an `initTransfer` on Starknet or Abstract chain receives tokens minted or unlocked on NEAR after MPC verification at a pre-final block. If the source chain subsequently reorganizes that block away — removing the burn or lock — the NEAR-side credit is permanent and irrecoverable. The bridge's collateralization is broken: NEAR holds minted supply with no corresponding locked collateral on the source chain. This is a direct unauthorized mint of bridged assets.

Impact classification: **Critical** — direct unauthorized mint of bridged assets on NEAR.

---

### Likelihood Explanation

**Starknet (`AcceptedOnL2`):** The Starknet sequencer is centralized and can reorganize L2 blocks before they are proven and settled on L1. `AcceptedOnL2` is explicitly documented as a pre-final state. A relayer racing to submit a proof immediately after a transaction reaches `AcceptedOnL2` status — before L1 settlement — creates a window where a sequencer reorganization (due to bugs, upgrades, or deliberate sequencer behavior) removes the source transaction. An attacker monitoring mempool and sequencer state can time submissions to exploit this window.

**Abstract chain (`EvmFinality::Latest`):** Abstract is a ZK rollup on ZKsync. `Latest` is the sequencer tip, not L1-finalized. Before the ZK proof is submitted and verified on L1, the sequencer can reorganize L2 blocks.

Likelihood: **Medium** — requires a source-chain reorganization event, which is not attacker-controlled in the general case, but is a realistic operational risk for both chains and can be timed by a sophisticated attacker monitoring sequencer state.

---

### Recommendation

1. **Short term:** Change the configured finality for Starknet to `StarknetFinality::AcceptedOnL1` and for Abstract chain to `EvmFinality::Finalized` or `EvmFinality::Safe`. These are the only finality levels that provide reorganization resistance.

2. **Long term:** Add a configurable confirmation delay or finality-level enforcement at the bridge level, independent of the prover, so that even if a prover is misconfigured, the bridge can enforce a minimum finality requirement before crediting tokens.

---

### Proof of Concept

1. Attacker calls `initTransfer` on Starknet, burning bridged tokens.
2. Transaction reaches `AcceptedOnL2` status (sequencer accepted, not yet L1-proven).
3. Relayer (or attacker acting as relayer) immediately calls NEAR bridge `fin_transfer` with a `MpcVerifyProofArgs` carrying `StarknetFinality::AcceptedOnL2`.
4. `mpc-omni-prover::verify_proof` passes the finality check (`request_matches_finality` returns `true` for `AcceptedOnL2` against the configured `AcceptedOnL2`). [6](#0-5) 
5. MPC network signs the payload; `verify_callback` confirms `SHA-256(borsh(sign_payload)) == response.payload_hash` and returns `ProverResult::InitTransfer`. [7](#0-6) 
6. NEAR bridge `fin_transfer_callback` calls `process_fin_transfer_to_near`, which calls `add_fin_transfer` (permanently marks transfer as finalized) and mints/unlocks tokens to the attacker. [8](#0-7) 
7. Starknet sequencer reorganizes the L2 block containing the `initTransfer`. The burn on Starknet is undone; the attacker's Starknet tokens are restored.
8. Attacker now holds tokens on both Starknet and NEAR. The NEAR bridge's `finalised_transfers` entry is permanent and cannot be reversed. [9](#0-8)

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
