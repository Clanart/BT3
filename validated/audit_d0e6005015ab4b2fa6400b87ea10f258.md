Audit Report

## Title
Unprotected `unwrapWETH9` allows any caller to drain all stranded WETH from the router as ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` is a `public payable` function with no access control that sweeps the router's entire WETH balance and sends it to a caller-supplied `recipient`. Any WETH that reaches the router outside of an atomic multicall is permanently stealable by any third party. The router's swap functions accept `recipient: address(router)` as a valid parameter with no guard, making the stranding precondition reachable by ordinary users.

## Finding Description
The function at `PeripheryPayments.sol` L37–45 performs no `msg.sender` check:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
``` [1](#0-0) 

`exactInputSingle` passes `params.recipient` directly to the pool's `swap` call with no restriction against `address(router)`: [2](#0-1) 

A user who calls `exactInputSingle` (or `exactInput`/`exactOutputSingle`) with `recipient: address(router)` as a standalone transaction — not inside a multicall — leaves WETH on the router across a transaction boundary. The `receive()` guard at L32–34 only blocks direct native ETH transfers from non-WETH addresses; it does not prevent ERC-20 WETH from being transferred to the router. [3](#0-2) 

Once WETH is stranded, any attacker calls `router.unwrapWETH9(0, attacker)`. With `amountMinimum = 0`, the only guard is bypassed, and `_transferETH(attacker, balanceWETH)` sends the full balance to the attacker.

The payment test suite confirms the stranding scenario directly — it transfers WETH to the router and calls `unwrapWETH9` from an arbitrary caller with an arbitrary recipient: [4](#0-3) 

By contrast, `refundETH` correctly hardcodes `msg.sender` as the recipient: [5](#0-4) 

## Impact Explanation
Direct theft of 100% of stranded user principal. The victim's WETH output is converted to ETH and transferred to the attacker; the victim receives zero. This is a direct loss of user principal meeting the Sherlock High threshold.

## Likelihood Explanation
The stranding precondition is reachable through normal router usage: any user who calls a swap function with `recipient: address(router)` outside of a multicall strands WETH. The router's public interface does not prevent or warn against this. WETH can also be stranded by a direct ERC-20 `transfer` to the router. An attacker can monitor the mempool or poll the router's WETH balance and call `unwrapWETH9(0, attacker)` at any time with no special privileges.

## Recommendation
Restrict `unwrapWETH9` and `sweepToken` so that only `msg.sender` can be the recipient — i.e., replace the `address recipient` parameter with `msg.sender` internally, or add `require(recipient == msg.sender)`. This matches the pattern already used correctly in `refundETH`. Alternatively, use transient storage to track per-caller WETH attribution and only allow each caller to withdraw their own balance.

## Proof of Concept
1. **Strand WETH**: Victim calls `router.exactInputSingle(...)` with `recipient: address(router)` as a standalone transaction. The pool swap completes; WETH lands on the router.
2. **Steal**: Attacker calls `router.unwrapWETH9(0, attacker)` from any EOA.
3. **Result**: `IWETH9(WETH).withdraw(balanceWETH)` converts all router WETH to ETH; `_transferETH(attacker, balanceWETH)` sends it to the attacker. Victim receives zero.

A Foundry test can reproduce this by: (a) pranking a victim to call `exactInputSingle` with `recipient: address(router)`, (b) asserting WETH balance on router is nonzero, (c) pranking an attacker to call `unwrapWETH9(0, attacker)`, and (d) asserting attacker received ETH and router WETH balance is zero.
