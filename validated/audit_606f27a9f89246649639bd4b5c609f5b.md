### Title
Revert Log Sentinel Value Collision with Attacker-Controlled Storage Keys Causes Incorrect State Reversion - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/revert.cairo`)

---

### Summary

The revert log in the StarkNet OS uses two sentinel constants — `CHANGE_CONTRACT_ENTRY` (`2^251`) and `CHANGE_CLASS_ENTRY` (`2^251 + 1`) — as special `selector` values to distinguish contract-boundary and class-change entries from storage-write entries in the same log structure. However, no validation prevents a contract from writing to storage keys equal to these sentinel values. If a storage write to key `2^251` or `2^251 + 1` is included in a reverted transaction, the revert log parser misidentifies the storage entry as a structural sentinel, causing the revert to execute incorrectly and leaving corrupted state on-chain.

---

### Finding Description

In `revert.cairo`, the `RevertLogEntry` struct uses a single `selector` field to encode three distinct entry types: [1](#0-0) 

```
const CONTRACT_ADDRESS_UPPER_BOUND = 2 ** 251;
const CHANGE_CONTRACT_ENTRY = CONTRACT_ADDRESS_UPPER_BOUND;   // sentinel 1
const CHANGE_CLASS_ENTRY = CHANGE_CONTRACT_ENTRY + 1;         // sentinel 2
```

The parser `revert_contract_changes` distinguishes entry types by comparing `selector` against these two sentinels, falling through to treat any other value as a storage key: [2](#0-1) 

Storage write entries are appended in `execute_storage_write` (new syscall path): [3](#0-2) 

And in the deprecated syscall path: [4](#0-3) 

In both cases, `storage_key` / `storage_address` is taken directly from the user-supplied request with **no upper-bound check** to ensure it is less than `CHANGE_CONTRACT_ENTRY` (`2^251`). The STARK field prime is `P = 2^251 + 17·2^192 + 1`, so `2^251` and `2^251 + 1` are valid felt values and valid storage keys.

**Collision scenario:**

If a contract writes to storage key `K = 2^251` (= `CHANGE_CONTRACT_ENTRY`) and the transaction is subsequently reverted, the revert log contains:

```
RevertLogEntry { selector: 2^251, value: prev_storage_value }
```

When `revert_contract_changes` processes this entry backwards, it hits:

```cairo
if (selector == CHANGE_CONTRACT_ENTRY) {
    return ();   // ← exits immediately, treating this as a contract boundary
}
``` [5](#0-4) 

Control returns to `handle_revert`, which reads `revert_log_end[0].value` as the **next contract address** to revert: [6](#0-5) 

The `value` field here is `prev_storage_value` — an attacker-controlled felt — so `handle_revert` now attempts to revert changes for a completely wrong contract address. The actual storage write at key `2^251` is **never reverted**, and all preceding revert log entries (for the real contract) are skipped or misattributed.

Similarly, a write to key `2^251 + 1` (= `CHANGE_CLASS_ENTRY`) causes the entry to be misread as a class-hash change, corrupting `class_hash` with the old storage value and skipping the storage revert. [7](#0-6) 

---

### Impact Explanation

A reverted transaction can produce **permanent, non-reverted storage mutations**. Any storage slot at key `2^251` or `2^251 + 1` written during a transaction that is later reverted will retain its written value. If such a slot controls token balances, allowances, ownership, or other financial state, this directly enables:

- **Direct loss of funds**: an attacker can write a fraudulent balance/allowance to a sentinel-colliding key, trigger a revert (e.g., via out-of-gas), and the fraudulent value persists, enabling theft.
- **Permanent freezing of funds**: corrupted class hash or storage state can render a contract permanently inoperable.

---

### Likelihood Explanation

Any unprivileged transaction sender can deploy a contract that issues a `storage_write` syscall with key `2^251`. The attacker then deliberately causes the transaction to revert (e.g., by consuming all gas after the write, or by calling a function that explicitly reverts). No privileged access, leaked keys, or external dependencies are required. The attack is fully self-contained and deterministic.

---

### Recommendation

Add an explicit upper-bound assertion on the storage key before appending to the revert log in both `execute_storage_write` (in `syscall_impls.cairo`) and `execute_storage_write` (in `deprecated_execute_syscalls.cairo`):

```cairo
// Ensure storage_key does not collide with revert log sentinels.
assert_lt(storage_key, CHANGE_CONTRACT_ENTRY);
```

Alternatively, choose sentinel values that are provably unreachable as storage keys (e.g., values ≥ the field prime, which are impossible as felt values), or use a tagged union encoding that does not rely on value-range separation.

---

### Proof of Concept

1. Deploy a Cairo contract with an entry point that:
   - Calls `storage_write(key=2^251, value=<attacker_chosen_value>)`
   - Then reverts (e.g., calls `assert 0 = 1` or exhausts gas)
2. Submit the transaction to the sequencer.
3. The OS processes the transaction, appends `RevertLogEntry { selector: 2^251, value: old_value }` to the revert log, then invokes `handle_revert`.
4. `revert_contract_changes` reads `selector == CHANGE_CONTRACT_ENTRY` and returns immediately without reverting the storage write.
5. `handle_revert` reads `old_value` as the next contract address and attempts (incorrectly) to revert that address's changes.
6. After the block is finalized, the storage slot at key `2^251` in the attacker's contract retains `<attacker_chosen_value>` despite the transaction being marked as reverted.
