### Title
Unconstrained `is_reverted` Hint Allows Prover to Bypass Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`, `execution/transaction_impls.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a complete no-op. The `is_reverted` flag controlling whether a transaction's execute phase runs is set entirely by an unconstrained hint. A malicious prover (sequencer) can freely set `is_reverted = 1` for any valid invoke transaction, skipping its execution while still charging the user's fee — producing a cryptographically valid proof for an incorrect state transition.

---

### Finding Description

In `execution_constraints.cairo`, the function that is supposed to validate the revert status does nothing:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

In `execute_invoke_function_transaction`, `is_reverted` is populated exclusively by a hint and then passed to this no-op:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // stack alignment only — no execution
}

// Charge fee regardless.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

Because `check_is_reverted` imposes zero Cairo constraints on `is_reverted`, the value is entirely prover-controlled. In Cairo's ZK proof model, an unconstrained local variable can be set to any felt by the prover without invalidating the proof. The execute phase is therefore skippable at will, while `charge_fee` always runs unconditionally after the branch. [3](#0-2) 

For **L1 handler transactions**, the situation is even more direct — `check_is_reverted` is not even called; `is_reverted` is used raw from the hint to gate the entire transaction body including `consume_l1_to_l2_message`:

```cairo
local is_reverted;
%{ IsReverted %}
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [4](#0-3) 

If `is_reverted = 1` is set for an L1 handler, `consume_l1_to_l2_message` is never called, so the L1→L2 message is not written to `outputs.messages_to_l2`. The L1 funds associated with that message are never credited on L2. [5](#0-4) 

---

### Impact Explanation

**For invoke transactions:** A malicious sequencer sets `is_reverted = 1` for any valid invoke transaction. The execute phase (`__execute__`) is skipped; the user's intended state changes never occur. `charge_fee` still runs and deducts the fee from the user's account. The resulting proof is cryptographically valid and accepted by the L1 verifier. This is **direct loss of funds** (fees paid for zero execution).

**For L1 handler transactions:** A malicious sequencer sets `is_reverted = 1` for an L1 handler. The L1→L2 message is not consumed in the OS output. The L1 deposit (ETH/ERC-20 bridged via the message) is never credited on L2. Since the message is not in the output, the L1 core contract does not mark it consumed, but the sequencer controls whether it is ever re-processed. This is **permanent freezing of funds**.

Both impacts are within the allowed scope:
- Critical: Direct loss of funds (invoke fee theft).
- Critical: Permanent freezing of funds (L1 bridge deposits silently dropped).

---

### Likelihood Explanation

The sequencer is the entity that provides all hints to the Cairo OS program. No external key compromise or third-party dependency is required. The sequencer already controls hint values; the OS is specifically designed to constrain the sequencer via Cairo assertions. The absence of any constraint in `check_is_reverted` means this constraint is simply missing — the sequencer can exploit it on any block, for any transaction, without detection, because the proof remains valid.

---

### Recommendation

Replace the no-op `check_is_reverted` with a real constraint. The correct pattern is to derive `is_reverted` from the actual execution result (the `failure_flag` returned by `execute_entry_point`) rather than from a free hint, or to add an assertion that the hint-provided value matches the execution outcome. At minimum:

```cairo
func check_is_reverted(is_reverted: felt) {
    // Constrain is_reverted to be a boolean (0 or 1).
    assert is_reverted * (1 - is_reverted) = 0;
    // Additional: assert consistency with actual execution result.
}
```

For L1 handler transactions, apply the same pattern — either remove the hint-based gate entirely or add a verifiable constraint before branching on `is_reverted`.

---

### Proof of Concept

1. User submits an invoke transaction transferring 100 STRK to address `0xABCD`.
2. Malicious sequencer includes the transaction in a block.
3. Sequencer provides hint `IsReverted → 1` for this transaction.
4. `check_is_reverted(1)` executes and returns immediately (no constraint).
5. The `if (is_reverted == FALSE)` branch is not taken; `__execute__` is never called.
6. `charge_fee` runs unconditionally; the user's account is debited the full fee.
7. The 100 STRK transfer never executes; `0xABCD` receives nothing.
8. The OS generates a valid STARK proof; the L1 verifier accepts it.
9. The state root on L1 reflects the fee deduction but not the transfer.
10. For an L1 handler variant: replace step 1 with a bridge deposit; the deposit is silently dropped and the funds are permanently frozen. [1](#0-0) [2](#0-1) [4](#0-3)

func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    assert_not_zero(execution_context.calldata_size);
    // The payload is the calldata without the from_address argument (which is the first).
    let payload: felt* = execution_context.calldata + 1;
    tempvar payload_size = execution_context.calldata_size - 1;

    tempvar execution_info = execution_context.execution_info;

    // Write the given transaction to the output.
    assert [outputs.messages_to_l2] = MessageToL2Header(
        from_address=[execution_context.calldata],
        to_address=execution_info.contract_address,
        nonce=nonce,
        selector=execution_info.selector,
        payload_size=payload_size,
    );

    let message_payload = cast(outputs.messages_to_l2 + MessageToL2Header.SIZE, felt*);
    memcpy(dst=message_payload, src=payload, len=payload_size);

    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1,
        messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE +
        outputs.messages_to_l2.payload_size,
    );
    return ();
