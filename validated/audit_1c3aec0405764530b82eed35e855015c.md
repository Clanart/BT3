### Title
EVM Gas Fee Bypass via `MsgExec`-Wrapped `MsgEthereumTx` When `AuthzLimiterDecorator.disabledMsgs` Is Empty — (`ante/cosmos/reject_msgs.go`, `ante/cosmos/authz.go`, `x/evm/keeper/msg_server.go`)

### Summary

`RejectMessagesDecorator` only inspects top-level transaction messages. `AuthzLimiterDecorator` is a complete no-op when its `disabledMsgs` set is empty. The EVM `MsgServer.EthereumTx` calls `ApplyTransaction` directly with no fee deduction. Together, these three facts allow an unprivileged attacker to wrap a `MsgEthereumTx` inside `MsgExec`, bypass all EVM-specific ante handlers, and execute EVM state transitions without paying EVM gas fees.

### Finding Description

**Guard 1 — `RejectMessagesDecorator`** iterates only `tx.GetMsgs()`:

```go
for _, msg := range tx.GetMsgs() {
    if _, ok := msg.(*evmtypes.MsgEthereumTx); ok {
        return ctx, errorsmod.Wrapf(...)
    }
}
``` [1](#0-0) 

When the top-level message is `MsgExec`, `tx.GetMsgs()` returns only the `MsgExec` wrapper. The nested `MsgEthereumTx` is invisible to this check.

**Guard 2 — `AuthzLimiterDecorator`** has an explicit early-exit when `disabledMsgs` is empty:

```go
func (ald AuthzLimiterDecorator) AnteHandle(...) (...) {
    if len(ald.disabledMsgs) == 0 {
        return next(ctx, tx, simulate)   // ← complete bypass
    }
    ...
}
``` [2](#0-1) 

Any deployment that initializes `NewAuthzLimiterDecorator([]string{})` (empty slice) causes this decorator to be a no-op, passing all messages through unconditionally.

**Guard 3 — EVM fee deduction** lives exclusively in the EVM ante handler path (`CheckEthGasConsume`), which calls `DeductTxCostsFromUserBalance`:

```go
err = evmKeeper.DeductTxCostsFromUserBalance(ctx, fees, common.BytesToAddress(msgEthTx.From))
``` [3](#0-2) 

This ante handler only runs for transactions that enter through the EVM-specific ante handler chain (those carrying `ExtensionOptionsEthereumTx`). A Cosmos tx wrapping `MsgExec` never triggers it.

**The MsgServer** calls `ApplyTransaction` directly with no fee deduction of its own:

```go
func (k *Keeper) EthereumTx(goCtx context.Context, msg *types.MsgEthereumTx) (*types.MsgEthereumTxResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    tx := msg.AsTransaction()
    ...
    evmResult, err := k.ApplyTransaction(ctx, msg)
``` [4](#0-3) 

There is no fee deduction, no balance check, and no nonce enforcement inside `EthereumTx` itself.

### Impact Explanation

An attacker submits a Cosmos tx: `[MsgExec([MsgEthereumTx])]`.

- `RejectMessagesDecorator` passes — sees only `MsgExec` at the top level.
- `AuthzLimiterDecorator` passes — no-op when `disabledMsgs` is empty.
- Cosmos `DeductFeeDecorator` runs — deducts a small Cosmos fee, not EVM gas.
- `MsgExec` handler dispatches to `x/evm MsgServer.EthereumTx`.
- `ApplyTransaction` executes and **commits EVM state changes**.
- The sender's EVM balance is **never debited for gas**.

This violates the invariant that EVM gas must be pre-paid before state transition. The attacker can execute arbitrary EVM calls (including value transfers and contract interactions) paying only the negligible Cosmos tx fee instead of the full EVM gas cost. This constitutes mis-accounting of user funds/fees and permits invalid transactions to commit state.

### Likelihood Explanation

The `AuthzLimiterDecorator` early-exit at line 51 is the critical enabler. Any chain that initializes the decorator with an empty disabled-msgs list — which is a natural default when no explicit configuration is provided — is immediately exploitable. The `NewAuthzLimiterDecorator` constructor accepts an arbitrary `[]string` slice with no validation or required entries. [5](#0-4) 

### Recommendation

1. **`RejectMessagesDecorator`** must recursively unwrap `MsgExec` and reject any nested `MsgEthereumTx`, mirroring the recursive logic already present in `AuthzLimiterDecorator.checkDisabledMsgs`.
2. **`AuthzLimiterDecorator`** must not short-circuit when `disabledMsgs` is empty; it should always recurse into `MsgExec` to check for `MsgEthereumTx` regardless of the disabled list.
3. **`MsgServer.EthereumTx`** should assert that the calling context originated from the EVM ante handler path (e.g., via a context flag set by `CheckEthGasConsume`) and reject execution otherwise.

### Proof of Concept

1. Deploy a chain with `NewAuthzLimiterDecorator([]string{})`.
2. Grant authz from account A to account B for `MsgEthereumTx`.
3. Account B submits: Cosmos tx → `[MsgExec([MsgEthereumTx{from: A, to: C, value: X, gas: Y}])]`.
4. Observe: EVM state transition commits (value transferred, contract called), but account A's EVM balance is not debited for `Y * gasPrice`. Only the Cosmos tx fee is charged.
