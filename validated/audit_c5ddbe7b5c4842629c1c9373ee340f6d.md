### Title
Unconditional Token Burn Before Transfer-Success Check in `resolve_fast_transfer` Causes Permanent Loss of Relayer Funds — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `near/omni-bridge/src/lib.rs`, the `resolve_fast_transfer` callback calls `burn_tokens_if_needed` with the full `amount` **unconditionally**, before checking whether the downstream `send_tokens` call (which uses `ft_transfer_call`) actually succeeded. When the fast transfer to a NEAR recipient fails, the bridge simultaneously burns the relayer's tokens and signals a full refund to the token contract. Because both operations target the same bridge balance, one must fail — and if the burn executes first, the relayer permanently loses their tokens.

This is a direct analog to the reported Swapnet bug: a sub-call processes fewer tokens than requested, but the caller removes the full amount from the account regardless.

---

### Finding Description

The fast-transfer-to-NEAR flow is:

1. Relayer calls `ft_transfer_call` on a bridge (deployed) token, sending `amount_without_fee` tokens to the bridge.
2. Bridge's `ft_on_transfer` processes the fast transfer and calls `send_tokens` to forward tokens to the recipient via `ft_transfer_call`.
3. `resolve_fast_transfer` is called as the callback after `send_tokens`.

Inside `resolve_fast_transfer`:

```rust
pub fn resolve_fast_transfer(
    &mut self,
    token_id: &AccountId,
    fast_transfer_id: &FastTransferId,
    amount: U128,
    is_ft_transfer_call: bool,
) -> U128 {
    // Burn the tokens to ensure the locked tokens are not double-minted
    self.burn_tokens_if_needed(token_id.clone(), amount);   // ← UNCONDITIONAL

    if Self::is_refund_required(is_ft_transfer_call) {
        self.remove_fast_transfer(fast_transfer_id);
        amount                                               // ← signals full refund
    } else {
        U128(0)
    }
}

`burn_tokens_if_needed` fires a **detached** (fire-and-forget) cross-contract call to burn `amount` from the bridge's token balance:

```rust

    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();          // ← no callback, no error handling
    }
}

`is_refund_required` checks the result of the `send_tokens` promise. When `is_ft_transfer_call` is true (i.e., `fast_transfer.msg` is non-empty) and the recipient's `ft_on_transfer` returns 0 used tokens (failure/panic), `is_refund_required` returns `true`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                    amount.0 == 0   // used == 0 → transfer failed → refund required
                } else { false }
            }
            Err(_) => false,
        }
    } else { false }
}
``` [3](#0-2) 

When the transfer fails, `resolve_fast_transfer` therefore:
1. Schedules a detached burn of `amount` from the bridge's token balance.
2. Returns `amount` — which the token contract's `ft_resolve_transfer` interprets as "refund `amount` to the relayer", also drawing from the bridge's token balance.

Both operations target the same bridge balance of exactly `amount`. One will succeed and one will fail. In NEAR's receipt scheduling, the detached burn receipt is enqueued during the execution of `resolve_fast_transfer`; the `ft_resolve_transfer` refund receipt is enqueued by the token contract after receiving the return value. If the burn receipt is processed first, the bridge balance drops to zero, and the subsequent refund transfer panics — the relayer permanently loses their tokens.

The `is_ft_transfer_call` flag is set to `!fast_transfer.msg.is_empty()`:

```rust
.resolve_fast_transfer(
    &fast_transfer.token_id,
    &fast_transfer.id(),
    amount_without_fee,
    !fast_transfer.msg.is_empty(),   // ← true when msg is present
),
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent, irrecoverable loss of relayer funds.**

When a fast transfer with a non-empty `msg` to a NEAR recipient fails (recipient panics or returns unused tokens), the relayer's bridge tokens are burned by the detached promise before the token contract can execute the refund. The relayer's `amount_without_fee` tokens are destroyed with no recovery path. The bridge's token supply is reduced without a corresponding cross-chain settlement, breaking collateralization.

---

### Likelihood Explanation

**Medium.** The trigger conditions are:
- A fast transfer must carry a non-empty `msg` field (so `is_ft_transfer_call = true`).
- The recipient's `ft_on_transfer` must fail (panic, insufficient storage, or return the full amount unused).

Both conditions are reachable by any unprivileged bridge user who submits a fast transfer with a message to a recipient contract that rejects the transfer. A malicious recipient can reliably trigger this to grief relayers.

---

### Recommendation

Move `burn_tokens_if_needed` inside the success branch only. Tokens should only be burned when the transfer to the recipient actually succeeded:

```rust
pub fn resolve_fast_transfer(
    &mut self,
    token_id: &AccountId,
    fast_transfer_id: &FastTransferId,
    amount: U128,
    is_ft_transfer_call: bool,
) -> U128 {
    if Self::is_refund_required(is_ft_transfer_call) {
        // Transfer failed: do NOT burn; refund the relayer
        self.remove_fast_transfer(fast_transfer_id);
        amount
    } else {
        // Transfer succeeded: burn to prevent double-minting
        self.burn_tokens_if_needed(token_id.clone(), amount);
        U128(0)
    }
}
```

### Proof of Concept

1. Relayer calls `ft_transfer_call` on a deployed bridge token (e.g., `usdc.bridge.near`), sending 1000 tokens to the bridge with a `FastFinTransferMsg` that has a non-empty `msg` field targeting a NEAR recipient.
2. Bridge's `ft_on_transfer` processes the fast transfer and calls `send_tokens` → `ft_transfer_call` on the token contract to forward 1000 tokens to the recipient.
3. The recipient's `ft_on_transfer` panics (e.g., due to insufficient storage deposit or a malicious contract).
4. `resolve_fast_transfer` is called:
   - `burn_tokens_if_needed` enqueues a detached receipt to burn 1000 tokens from the bridge's balance.
   - `is_refund_required` returns `true` (used amount == 0).
   - `resolve_fast_transfer` returns `U128(1000)`.
5. NEAR processes the detached burn receipt first: bridge balance goes from 1000 → 0.
6. Token contract's `ft_resolve_transfer` attempts to transfer 1000 tokens from bridge (balance 0) to relayer → panics.
7. Relayer's 1000 tokens are permanently destroyed. The bridge's token supply is reduced by 1000 with no cross-chain settlement.
