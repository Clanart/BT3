### Title
Missing Nonce/Deadline in `compute_meta_tx_v0_hash` Enables Indefinite Replay of Signed Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that contains no nonce, no deadline, and no block-number bound. Any signature produced for a `meta_tx_v0` call is therefore valid forever and can be replayed by any party who has observed it, leading to direct loss of funds from the targeted account.

---

### Finding Description

`compute_meta_tx_v0_hash` is the hash function used to authenticate `meta_tx_v0` syscall requests. It delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),
    );
``` [1](#0-0) 

The `additional_data` slot is the only place a nonce or deadline could be injected into the Pedersen hash chain. With `additional_data_size=0`, the hash commits only to `(prefix, version=0, contract_address, selector, calldata_hash, max_fee=0, chain_id)` — no ordering or time-bounding field whatsoever.

Inside `execute_meta_tx_v0`, the `TxInfo` struct is constructed with `nonce=0` hardcoded:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
``` [2](#0-1) 

The OS never calls `check_and_increment_nonce` for a `meta_tx_v0` execution path; that function is only invoked for top-level account transactions (invoke, deploy_account, declare). [3](#0-2) 

The called contract's `__execute__` entry point receives a `TxInfo` whose `transaction_hash` is this nonce-free hash. Any signature-verification logic inside `__execute__` that checks against `transaction_hash` will accept the same signature on every future replay, because the hash never changes.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A `meta_tx_v0` call is typically used to authorize an action (e.g., an ERC-20 `transfer`) on behalf of an account. Because the hash commits to no nonce and no deadline, an attacker who observes a single valid `meta_tx_v0` call (e.g., from the mempool or from on-chain history) can re-submit the identical syscall request in any future block, indefinitely. Each replay executes the same authorized action — draining the victim's token balance one transfer at a time, or repeating any other state-changing operation the original call performed.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is reachable by any unprivileged contract caller via the `META_TX_V0_SELECTOR` dispatch path in `execute_syscalls.cairo`. [4](#0-3) 

Any contract that uses `meta_tx_v0` to authorize fund movements is immediately vulnerable. An attacker needs only to observe one successful call (publicly visible on-chain) and re-submit it. No privileged access, key compromise, or network-level attack is required.

---

### Recommendation

Include a monotonically-increasing, per-account nonce (or a user-supplied deadline block number) in the `additional_data` array passed to `deprecated_get_transaction_hash` inside `compute_meta_tx_v0_hash`. The OS must then enforce that this nonce matches and increments the account's on-chain nonce, exactly as `check_and_increment_nonce` does for standard account transactions. [1](#0-0) 

---

### Proof of Concept

1. Alice's contract calls `meta_tx_v0` to transfer 100 STRK to Bob, producing a signed payload `(contract_address, __execute__, calldata, chain_id)`.
2. The OS computes `H = Pedersen(prefix, 0, contract_address, __execute__, H(calldata), 0, chain_id)` — no nonce, no deadline.
3. Alice's `__execute__` verifies the signature against `H` and executes the transfer. The call succeeds.
4. An attacker observes the `MetaTxV0Request` (signature + calldata) from the transaction trace.
5. In block N+1, the attacker submits an invoke transaction whose `__execute__` issues the identical `meta_tx_v0` syscall with the same signature and calldata.
6. The OS recomputes the same `H` (nothing has changed), Alice's `__execute__` verifies the same signature, and another 100 STRK is transferred.
7. Steps 5–6 repeat until Alice's balance is exhausted.

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
