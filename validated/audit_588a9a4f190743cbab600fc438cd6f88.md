### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Unbounded Signature Replay — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function omits a nonce from the hash commitment. Because the hash that the target account contract's `__execute__` verifies its signature against does not include any replay-protection field, any unprivileged attacker who observes a valid `meta_tx_v0` syscall can re-submit it an unlimited number of times, draining the victim account's funds.

---

### Finding Description

`execute_meta_tx_v0` is the OS handler for the `meta_tx_v0` syscall. It computes a transaction hash and presents it to the target account contract as `tx_info.transaction_hash`, which the account's `__execute__` uses to verify the caller-supplied signature.

The hash is computed by `compute_meta_tx_v0_hash`:

```cairo
// transaction_hash.cairo:295-315
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // <-- no nonce
        additional_data=cast(0, felt*),  // <-- no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The hash covers only `(prefix, version=0, contract_address, selector, calldata, max_fee=0, chain_id)`. No nonce is committed to.

In `execute_meta_tx_v0`, the resulting `TxInfo` is constructed with `nonce=0` hardcoded:

```cairo
// syscall_impls.cairo:343-363
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    transaction_hash=meta_tx_hash,
    nonce=0,          // hardcoded
    ...
);
``` [2](#0-1) 

And `check_and_increment_nonce` explicitly skips version-0 transactions:

```cairo
// execute_transaction_utils.cairo:65-67
if (tx_info.version == 0) {
    return ();
}
``` [3](#0-2) 

The result: the tuple `(contract_address, selector, calldata, chain_id, signature)` is the complete, permanent authorization credential. Once a valid signature over this tuple exists on-chain, it is valid forever and for any submitter.

This is the direct structural analog to the reported Shardeum bug: in that case, a signing node committed to a subset of the payload fields (cert fields) without rejecting extra fields (apoptosis fields), so the signature was reusable for a different purpose. Here, the OS commits to a subset of the transaction fields (omitting the nonce), so the signature is reusable for unlimited replays.

---

### Impact Explanation

**Direct loss of funds (Critical).**

Any operation authorized via `meta_tx_v0` — token transfers, approvals, state mutations — can be replayed by any observer. An attacker who sees a single legitimate `meta_tx_v0` call can drain the victim account by replaying the same signed calldata until the account balance is exhausted or the state change is no longer meaningful.

---

### Likelihood Explanation

`meta_tx_v0` is a general-purpose syscall callable from any deployed contract. An attacker needs only to:
1. Observe one on-chain transaction that triggered a `meta_tx_v0` syscall (public data).
2. Deploy a contract that re-issues the same syscall with the captured `(contract_address, selector, calldata, signature)`.

No privileged access, leaked keys, or Sybil attack is required. The attacker is a standard unprivileged transaction sender.

---

### Recommendation

Include a nonce in the meta-tx v0 hash commitment, analogous to how `deprecated_get_transaction_hash` uses `additional_data` for nonces in other transaction types. The OS must also track and increment (or invalidate) that nonce per `(contract_address)` to prevent replay.

```cairo
// Pass the account's current nonce as additional_data
deprecated_get_transaction_hash(
    ...
    additional_data_size=1,
    additional_data=&nonce,
);
```

And enforce nonce increment in `check_and_increment_nonce` for meta-tx v0 (remove the version-0 early return, or handle it separately).

---

### Proof of Concept

1. Victim account `A` holds tokens. A legitimate caller invokes `meta_tx_v0` targeting `A` with calldata for `transfer(attacker, amount)`, providing a valid signature `sig` from `A`.
2. The OS computes `H = hash(A, __execute__, calldata, chain_id)` (no nonce). `A.__execute__` verifies `sig` against `H` and executes the transfer.
3. Attacker deploys contract `M`. `M.__execute__` calls `meta_tx_v0(contract_address=A, selector=__execute__, calldata=same_calldata, signature=sig)`.
4. The OS recomputes the identical `H` (same inputs, no nonce), presents it to `A.__execute__`, which again verifies `sig` as valid and executes another transfer.
5. Attacker repeats step 3–4 until `A`'s balance is zero.

The root cause is entirely within `compute_meta_tx_v0_hash` at [1](#0-0)  and the nonce bypass in `execute_meta_tx_v0` at [4](#0-3) .
