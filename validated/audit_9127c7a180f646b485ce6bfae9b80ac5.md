### Title
Hardcoded Virtual OS Program Hash Check in `check_proof_facts` Can Cause Total Network Halt — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The `check_proof_facts` function enforces a hard Cairo `assert` against a single hardcoded allowed virtual OS program hash. If the virtual OS program is upgraded (its hash changes) and `constants.cairo` is not simultaneously updated, any invoke transaction that carries `proof_facts` will trigger an OS-level assertion failure — not a transaction-level revert — causing the entire block proof to be unprovable and halting the network.

---

### Finding Description

`check_proof_facts` in `execution_constraints.cairo` is called unconditionally for every invoke transaction that supplies a non-empty `proof_facts` field:

```cairo
// execution_constraints.cairo, lines 34–81
func check_proof_facts{range_check_ptr, contract_state_changes: DictAccess*}(
    proof_facts_size: felt,
    proof_facts: felt*,
    current_block_number: felt,
    virtual_os_config_hash: felt,
) {
    if (proof_facts_size == 0) {
        return ();
    }
    ...
    let proof_header = cast(proof_facts, ProofHeader*);
    assert is_program_hash_allowed(proof_header.program_hash) = TRUE;   // ← hard OS assert
    ...
}
``` [1](#0-0) 

`is_program_hash_allowed` checks against exactly **one** compile-time constant:

```cairo
// execution_constraints.cairo, lines 25–31
func is_program_hash_allowed(program_hash: felt) -> felt {
    static_assert ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_LEN == 1;
    if (program_hash == ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0) {
        return TRUE;
    }
    return FALSE;
}
``` [2](#0-1) 

That constant is baked into `constants.cairo`:

```cairo
// constants.cairo, lines 68–71
const ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0 = (
    0x03e98c2d7703b03a7edb73ed7f075f97f1dcbaa8f717cdf6e1a57bf058265473
);
const ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_LEN = 1;
``` [3](#0-2) 

The call site in `execute_invoke_function_transaction` passes user-supplied `proof_facts` directly into this check **before** any transaction-level revert boundary:

```cairo
// transaction_impls.cairo, lines 313–318
check_proof_facts(
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
    current_block_number=block_context.block_info_for_execute.block_number,
    virtual_os_config_hash=block_context.os_global_context.virtual_os_config_hash,
);
``` [4](#0-3) 

In Cairo, an `assert` that fails inside OS code (not inside a contract call) is **not** a transaction revert — it is an unsatisfied constraint that makes the entire proof invalid. There is no surrounding try/catch or revert boundary here; the failure propagates to the top-level proof, making the block unprovable.

---

### Impact Explanation

**High — Total network shutdown.**

If the assertion on line 48 of `execution_constraints.cairo` fails for any transaction in a block, the STARK proof for that block cannot be generated. The sequencer cannot finalize the block, and no subsequent blocks can be built on top of it. This is a complete halt of transaction confirmation on the network until the OS is redeployed with an updated hash list. [5](#0-4) 

---

### Likelihood Explanation

The virtual OS program (`ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0`) is a separately deployed component. StarkNet protocol upgrades routinely redeploy the virtual OS. If `constants.cairo` is not updated atomically with the virtual OS redeployment — even for a single block — any in-flight invoke transaction carrying `proof_facts` will trigger the failing assertion. The window of exposure is the gap between the virtual OS upgrade and the OS program redeployment. This is structurally identical to the Stargate router redeployment scenario described in the reference report: an external dependency address/hash changes, and the hardcoded check causes a critical function to fail.

Additionally, an unprivileged user who learns that the virtual OS hash has changed (e.g., by observing an upgrade transaction on-chain) can immediately submit invoke transactions with `proof_facts` referencing the new hash. If the sequencer includes any such transaction before `constants.cairo` is updated, the block proof fails.

---

### Recommendation

1. **Replace the hard `assert` with a transaction-level failure.** The OS should write a failure response and revert the transaction (not the proof) when `proof_facts` contain an unrecognized program hash. This mirrors how other invalid inputs (e.g., out-of-gas, invalid argument) are handled via `write_failure_response` in `syscall_impls.cairo`.

2. **Support a dynamic or multi-entry allowed hash list.** Rather than a single compile-time constant, store allowed hashes in a way that can be extended without redeploying the OS binary (e.g., read from a privileged storage slot, or allow a list of N hashes).

3. **Enforce atomic upgrade coordination.** Document and enforce that any virtual OS program upgrade must be accompanied by a simultaneous OS redeployment with the updated hash, with zero blocks processed in between.

---

### Proof of Concept

1. The virtual OS program is upgraded; its new hash is `H_new ≠ ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0`.
2. An unprivileged user submits an invoke transaction with `proof_facts` whose `ProofHeader.program_hash = H_new`.
3. The sequencer (which knows `H_new` is the current valid hash) includes the transaction in a block.
4. The OS executes `execute_invoke_function_transaction` → `check_proof_facts` → `is_program_hash_allowed(H_new)` → returns `FALSE`.
5. `assert is_program_hash_allowed(H_new) = TRUE` is unsatisfied.
6. The STARK proof for the block cannot be generated.
7. The block is permanently unprovable; the network halts until the OS is redeployed. [6](#0-5) [3](#0-2) [7](#0-6)

    proof_facts_size: felt,
