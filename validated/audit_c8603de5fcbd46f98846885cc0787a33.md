### Title
Reentrancy in `initTransfer` / `initTransfer1155` Causes Nonce Collision and Permanent Fund Lock — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer` and `initTransfer1155` in `OmniBridge.sol` lack a reentrancy guard. Both functions increment the shared storage variable `currentOriginNonce` at the top of the function, then make external calls to attacker-influenced token contracts, and finally read `currentOriginNonce` again from storage when emitting the `InitTransfer` event and calling `initTransferExtension`. A re-entrant call during the external token transfer modifies `currentOriginNonce` before the outer call reads it, causing the outer call to emit the wrong nonce. This produces a duplicate nonce on-chain: the re-entrant call's nonce is emitted twice and the outer call's original nonce is never emitted. NEAR rejects the duplicate, permanently locking the outer call's deposited tokens in the bridge.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer` increments `currentOriginNonce` at line 381 but does **not** capture it into a local variable. The function then makes external calls to the token contract (lines 395–412), and only afterward reads `currentOriginNonce` from storage again at lines 418 and 430 to pass to `initTransferExtension` and `emit BridgeTypes.InitTransfer`. [1](#0-0) [2](#0-1) [3](#0-2) 

Because `currentOriginNonce` is a storage variable read **after** the external call, any re-entrant invocation of `initTransfer` that runs during the external call will increment `currentOriginNonce` again. When the outer call resumes, it reads the already-incremented value and emits the re-entrant call's nonce a second time.

The same pattern exists in `initTransfer1155`, where `IERC1155.safeTransferFrom` is the external call vector: [4](#0-3) [5](#0-4) [6](#0-5) 

Neither function carries a `nonReentrant` modifier; the only guard is `whenNotPaused(PAUSED_INIT_TRANSFER)`.

---

### Impact Explanation

**Nonce collision / permanent fund lock (Critical):**

Let `currentOriginNonce` be `N` before the attack.

| Step | Action | `currentOriginNonce` |
|------|--------|----------------------|
| Outer call enters `initTransfer` | `+= 1` | N+1 |
| Outer call reaches `safeTransferFrom` | external call to malicious token | N+1 |
| Re-entrant call enters `initTransfer` | `+= 1` | N+2 |
| Re-entrant `safeTransferFrom` | malicious token returns `true`, no real transfer | N+2 |
| Re-entrant `initTransferExtension` + `emit InitTransfer` | uses `currentOriginNonce` = **N+2** | N+2 |
| Re-entrant call returns | | N+2 |
| Outer `initTransferExtension` + `emit InitTransfer` | reads `currentOriginNonce` = **N+2** (wrong!) | N+2 |

Result:
- Nonce **N+1** is never emitted — the outer call's real token deposit has no valid nonce on NEAR.
- Nonce **N+2** is emitted **twice** — NEAR processes the first (re-entrant, unbacked) emission and rejects the second as a duplicate.
- The outer call's tokens are **permanently locked** in the bridge with no path to finalization.
- If the re-entrant call's `safeTransferFrom` returns `true` without transferring tokens (trivially achievable with a custom ERC20), NEAR releases tokens for a deposit that never happened — **unauthorized mint / direct theft**.

This matches two allowed impacts:
- **Critical**: Permanent, irrecoverable lock of user funds in the bridge.
- **Critical**: Unauthorized mint of bridged assets on NEAR.

---

### Likelihood Explanation

The external call at lines 407–411 is made to an arbitrary `tokenAddress` supplied by the caller. Any ERC20 with a `transferFrom` callback (ERC777 via ERC1820, or a custom token) can trigger re-entry. The attacker only needs to:
1. Deploy or control a token with a `transferFrom` hook.
2. Have that token recognized on NEAR (via the standard `deployToken` flow, which requires an MPC signature but is a normal protocol operation for new tokens).
3. Call `initTransfer` with that token.

No privileged role, leaked key, or colluding operator is required. The entry point is fully public.

---

### Recommendation

1. **Capture `currentOriginNonce` into a local variable immediately after incrementing it**, before any external call, and use that local variable in `initTransferExtension` and `emit`:

```solidity
currentOriginNonce += 1;
uint64 nonce = currentOriginNonce; // capture before any external call
// ... external calls ...
initTransferExtension(..., nonce, ...);
emit BridgeTypes.InitTransfer(..., nonce, ...);
```

2. **Add a `nonReentrant` modifier** (OpenZeppelin `ReentrancyGuardUpgradeable`) to both `initTransfer` and `initTransfer1155` as defense-in-depth.

Apply the same fix to `initTransfer1155`.

```
          // re-entrant: currentOriginNonce becomes N+2
          // safeTransferFrom inside re-entrant call returns true, no tokens moved
          // emit InitTransfer(nonce=N+2, amount=1_000_000e18)
      return true  // outer call's transferFrom also returns true, transfers 1 token

Attacker calls:
  OmniBridge.initTransfer(MaliciousERC20, 1, 0, 0, "attacker.near", "")
  // currentOriginNonce: N -> N+1
  // safeTransferFrom triggers re-entry (see above)
  // after re-entry: currentOriginNonce = N+2
  // outer emit InitTransfer(nonce=N+2, amount=1)  <-- DUPLICATE

NEAR side:
  - Sees InitTransfer(nonce=N+2, amount=1_000_000e18) -> processes, mints 1_000_000 tokens to attacker.near
  - Sees InitTransfer(nonce=N+2, amount=1)            -> rejects as duplicate nonce
  - Nonce N+1 never seen -> 1 token locked in bridge forever
```
