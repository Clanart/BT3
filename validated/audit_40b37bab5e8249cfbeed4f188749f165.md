### Title
Missing Validation of Declared Class Hash in `execute_replace_class` Enables Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a caller corresponds to a previously declared class. This mirrors the original report's vulnerability class — an irreversible state transition with no validity guard — and allows any contract to permanently replace its own class with an undeclared hash, making the contract permanently non-executable and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS accepts any arbitrary `felt` value as the new class hash and writes it directly into the contract state without checking whether that hash exists in the class commitment tree:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The acknowledged TODO at line 898 confirms this check is intentionally absent from the current implementation. The class hash written into the state is never cross-referenced against `contract_class_changes` (the class declaration registry), which enforces `prev_value=0` on declaration to guarantee uniqueness:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) [3](#0-2) 

Once the state is committed with an undeclared class hash, the contract is permanently non-executable:

- Future calls read the contract's class hash from state and attempt to resolve the class. The class does not exist in the class tree, so execution fails unconditionally.
- The contract cannot self-recover by calling `replace_class` again because it cannot execute at all.
- There is no admin or OS-level mechanism to reset a contract's class hash externally.

The `replace_class` syscall is dispatched from `execute_syscalls` using the currently executing contract's address, so the attacker controls the target contract:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash (e.g., `0`, or any arbitrary felt) will have its class hash permanently set to an invalid value. All funds held in that contract's storage become permanently inaccessible. The state commitment machinery in `compute_contract_state_commitment` and `hash_contract_state_changes` will faithfully commit the invalid class hash into the Patricia Merkle Tree, making the freeze provably permanent on-chain. [5](#0-4) 

---

### Likelihood Explanation

**Medium.**

The attack path is directly reachable by any unprivileged transaction sender:

1. An attacker deploys a contract (e.g., a fake vault or DeFi protocol) that accepts user deposits.
2. Users deposit funds into the contract.
3. The attacker calls a function in the contract that issues `replace_class(0)` (or any undeclared hash).
4. The OS processes the syscall, writes the invalid class hash into the contract state, and commits it.
5. All subsequent calls to the contract fail; user funds are permanently frozen.

No privileged role, leaked key, or external dependency is required. The only prerequisite is deploying a contract and executing a transaction — both are standard unprivileged operations on StarkNet.

---

### Recommendation

Before writing the new class hash into the contract state, `execute_replace_class` must verify that the supplied class hash exists in the declared class registry. Concretely, a `dict_read` on `contract_class_changes` (or the equivalent committed class tree lookup) should be performed, and the syscall should return a failure response if the class hash resolves to `UNINITIALIZED_CLASS_HASH` (`0`):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the guard already present in `execute_declare_transaction` (`assert_not_zero(compiled_class_hash)`) and closes the irreversible-state-transition gap.

### Proof of Concept

```
1. Attacker deploys ContractA (a fake vault) — users deposit 1000 STRK into it.
2. Attacker sends InvokeTx calling ContractA::drain(), which internally calls:
       replace_class(class_hash=0)   // 0 is never declared
3. OS executes execute_replace_class:
   - Reads ContractA's StateEntry (class_hash=valid, storage=funds, nonce=N)
   - Writes new StateEntry(class_hash=0, storage=funds, nonce=N) — no validation
   - dict_update commits the change
4. Block is proven and committed. ContractA's class hash in the Patricia tree = 0.
5. Any user attempts to withdraw: the OS reads class_hash=0, finds no class,
   execution fails unconditionally.
6. 1000 STRK are permanently frozen with no recovery path.
```
