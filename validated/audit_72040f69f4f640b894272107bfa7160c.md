### Title
Deploy Syscall Always Reports Success Regardless of Constructor Outcome — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally asserts `failure_flag=0` in the response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This mirrors the PufferVault vulnerability class exactly: an operation that can partially fail is assumed to always succeed, causing the caller to receive incorrect accounting information and potentially act on it in ways that permanently freeze funds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function calls `deploy_contract` (which supports constructor reverts via the `revert_log` implicit argument), but then unconditionally writes a success response:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `assert` is a ZK proof constraint. It forces the proof to encode `failure_flag=0` in the syscall response unconditionally. The calling contract reads this response and always observes a successful deployment, even when the constructor reverted and all constructor-side storage writes were rolled back via the revert log.

By contrast, `execute_call_contract` and `contract_call_helper` correctly propagate `is_reverted` into the response:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `deploy_contract` call inside `execute_deploy` does accept a `revert_log` implicit argument (the same mechanism used in `execute_deploy_account_transaction`), confirming that constructor reverts are structurally possible and handled at the state level — but the outcome is never surfaced to the calling contract. [3](#0-2) 

---

### Impact Explanation

When a constructor reverts:

1. The revert log rolls back all constructor storage writes. The deployed contract's storage is left uninitialized (no owner set, no balances initialized, no access controls configured).
2. The class hash is still registered at the computed contract address (the contract "exists").
3. The calling contract receives `failure_flag=0` and the contract address, and has no way to detect the failure.
4. If the calling contract subsequently transfers tokens or ETH to the deployed address (a common pattern: deploy-then-fund), those assets enter a contract whose invariants were never established by the constructor.
5. Because the contract's storage is uninitialized, withdrawal or recovery functions may be permanently broken, causing **permanent freezing of funds**.

This matches the allowed impact: **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

- Any unprivileged user can invoke a contract that uses the `deploy` syscall.
- The user controls the `constructor_calldata` passed to the deploy syscall (directly or indirectly via the outer contract's interface).
- A constructor revert can be triggered by supplying calldata that violates a constructor precondition (e.g., zero address, out-of-range value, failed assertion).
- Contracts that follow the deploy-then-fund pattern (deploy a vault/escrow/pool, then deposit assets) are directly exploitable: the attacker causes the constructor to revert, the caller funds the uninitialized contract, and the funds are frozen.
- No privileged access, leaked keys, or external dependency compromise is required.

---

### Recommendation

Propagate the constructor's revert status into the deploy response, consistent with how `contract_call_helper` handles call reverts:

```cairo
// Capture whether the constructor reverted.
let (is_constructor_reverted, retdata_size, retdata) = deploy_contract_with_revert_status(...);

assert [response_header] = ResponseHeader(
    gas=remaining_gas,
    failure_flag=is_constructor_reverted  // not hardcoded 0
);
```

Until this is fixed, contracts using the `deploy` syscall must not assume deployment succeeded and must not transfer assets to the returned address without an independent verification step.

---

### Proof of Concept

**Scenario:**

1. Attacker calls `VaultFactory.create_vault(bad_calldata)` where `bad_calldata` causes the `Vault` constructor to revert (e.g., `assert owner != 0` fails because attacker passes `owner=0`).
2. The OS executes the constructor, the revert log rolls back all storage writes. The `Vault` contract address is computed and the class hash is registered, but storage is empty.
3. The OS writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` — the constraint is satisfied.
4. `VaultFactory` reads `failure_flag=0`, concludes deployment succeeded, and executes `token.transfer(vault_address, deposit_amount)`.
5. Tokens arrive at `vault_address`. The `Vault` contract's `owner` slot is `0` (constructor never ran to completion). The `withdraw` function checks `assert caller == owner` → always fails since `owner == 0`.
6. Tokens are permanently frozen.

**Root cause line:** [4](#0-3) 

The `// TODO(Yoni, 1/1/2026): support failures.` comment immediately above the hardcoded `failure_flag=0` confirms this is an acknowledged gap in the implementation, not an intentional design choice.
