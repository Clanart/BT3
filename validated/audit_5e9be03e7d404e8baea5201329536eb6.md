### Title
Unbounded `Predecessors` Iteration in `Keeper.TraceTx` Enables gRPC Resource Exhaustion DoS - (File: x/evm/keeper/grpc_query.go)

### Summary

`Keeper.TraceTx` iterates over a caller-supplied `req.Predecessors` slice without any count limit, executing a full `ApplyMessageWithConfig` EVM state transition for every entry. An unprivileged caller can submit a `QueryTraceTxRequest` directly to the public gRPC endpoint with an arbitrarily large `Predecessors` array, causing unbounded CPU and memory consumption on the node.

### Finding Description

`Keeper.TraceTx` in `x/evm/keeper/grpc_query.go` accepts a `QueryTraceTxRequest` whose `Predecessors` field is a `repeated MsgEthereumTx` — a protobuf-repeated field with no enforced upper bound. The handler loops over every entry and calls `ApplyMessageWithConfig` (a full EVM execution that commits to a cache context) for each one:

```go
for i, tx := range req.Predecessors {          // no length cap
    ...
    rsp, err := k.ApplyMessageWithConfig(ctx, msg, cfg, true)  // full EVM run per entry
    ...
}
``` [1](#0-0) 

The `Predecessors` field is defined in the protobuf schema as an unbounded repeated field: [2](#0-1) 

No predecessor count cap, no per-request timeout specific to `TraceTx`, and no `queryMaxGasLimit` enforcement are applied to the predecessor loop. The `queryMaxGasLimit` guard is only present in `EthCall` and `EstimateGas`, not here. [3](#0-2) 

The gRPC endpoint `ethermint.evm.v1.Query/TraceTx` is publicly reachable and also exposed via the REST gateway at `/ethermint/evm/v1/trace_tx`. The JSON-RPC `debug_traceTransaction` backend path builds predecessors from actual block contents (bounded), but the raw gRPC endpoint accepts the `Predecessors` array directly from the caller with no such bound. [4](#0-3) 

The same structural issue exists in `Keeper.TraceBlock`, which iterates over an unbounded `req.Txs` array: [5](#0-4) 

### Impact Explanation

Each predecessor entry triggers a full EVM state transition (`ApplyMessageWithConfig` with `commit=true`). An attacker submitting N predecessors forces N sequential EVM executions. With N in the thousands and each predecessor containing a gas-heavy contract call, the node's CPU and memory are exhausted for the duration of the request. If the targeted node is a validator, sustained exhaustion causes missed block proposals and prevote/precommit timeouts, which can lead to consensus failure or validator jailing. This matches the allowed High impact: a public gRPC tracing path that exposes a reachable route to chain halt.

### Likelihood Explanation

The gRPC port (default 9090) is commonly exposed on full nodes and some validator setups. No authentication is required to call `ethermint.evm.v1.Query/TraceTx`. The attacker only needs to craft a valid `QueryTraceTxRequest` protobuf message with a large `Predecessors` array — each entry can be a minimal but valid `MsgEthereumTx`. This requires no privileged access, no keys, and no on-chain funds.

### Recommendation

1. **Enforce a hard cap on `len(req.Predecessors)`** in `Keeper.TraceTx` before entering the loop. A reasonable limit mirrors the maximum number of transactions per block (e.g., reject if `len(req.Predecessors) > maxPredecessors`).
2. **Apply the same cap to `len(req.Txs)`** in `Keeper.TraceBlock`.
3. **Apply `queryMaxGasLimit`** to each predecessor message before calling `ApplyMessageWithConfig`, consistent with how `EthCall` and `EstimateGas` enforce it.
4. **Add a per-request deadline** (context timeout) for all trace gRPC handlers, separate from the general EVM timeout.

### Proof of Concept

An attacker constructs and sends the following gRPC request directly to port 9090:

```go
req := &evmtypes.QueryTraceTxRequest{
    Msg: validEthMsg,                          // any valid MsgEthereumTx
    Predecessors: make([]*evmtypes.MsgEthereumTx, 10_000), // 10k predecessors
    BlockNumber: latestHeight,
    ChainId:     9000,
}
// populate each predecessor with a gas-heavy contract call tx
for i := range req.Predecessors {
    req.Predecessors[i] = buildHeavyEthMsg()   // e.g., max-gas ERC-20 transfer
}
client.TraceTx(ctx, req)  // blocks node for minutes; no auth required
```

`Keeper.TraceTx` enters the loop at line 542 and calls `ApplyMessageWithConfig` 10,000 times sequentially with no escape hatch, exhausting the node's CPU and goroutine budget for the duration. [6](#0-5)
