### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Bricking via Undeclared Class Hash — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The StarkNet OS `execute_replace_class` syscall handler does not validate that the supplied `class_hash` corresponds to a declared contract class before committing the state update. An unprivileged contract can call `replace_class` with an arbitrary or zero class hash. Because `replace_class` is the only mechanism to change a contract's class, and it can only be invoked from within the contract itself, a contract whose class is set to an undeclared hash becomes permanently unexecutable — permanently freezing any funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing check. The same omission exists in the deprecated path at `deprecated_execute_syscalls.cairo` lines 307–329, which has no TODO at all — it silently accepts any class hash.

The `library_call` syscall (the StarkNet analog of EVM `delegatecall`) executes arbitrary class code in the context of the *caller's* contract address. A malicious class that issues `replace_class(0)` during a `library_call` will replace the *victim caller's* class hash with `0` — an undeclared hash. After the block is proven and committed, every future call to the victim contract will fail at class resolution. Because `replace_class` can only be invoked from within the contract, there is no recovery path: the contract is permanently bricked and all funds it holds are permanently frozen.

This is the direct analog of the HyVM `callcode` + `selfdestruct` pattern: an unrestricted dangerous operation (here, `replace_class` with an undeclared hash) that can be triggered through a delegatecall-equivalent (`library_call`) to permanently destroy the usability of a contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds token balances (account contracts, DeFi vaults, multisigs) can be permanently bricked by setting its class hash to an undeclared value. Once the OS state transition is proven and the new state root is committed on L1, the contract's class hash is `0` (or another undeclared value) in the canonical state. No subsequent transaction can execute against it, and no recovery mechanism exists within the protocol. All funds stored in the contract's storage or token balances associated with its address are permanently inaccessible.

---

### Likelihood Explanation

**Medium.**

The attack requires a victim contract to invoke `library_call` with an attacker-controlled class hash. This is a realistic scenario: contracts that support pluggable logic, proxy patterns, or user-supplied class hashes for library calls are common in StarkNet. The attacker only needs to publish a malicious declared class containing a `replace_class(0)` call and convince (or trick) a target contract into calling it via `library_call`. No privileged access, leaked keys, or operator cooperation is required — only a standard unprivileged transaction.

---

### Recommendation

In `execute_replace_class` (both the new-syscall path in `syscall_impls.cairo` and the deprecated path in `deprecated_execute_syscalls.cairo`), add a check against `contract_class_changes` to verify that the requested `class_hash` has a non-zero compiled class hash entry (i.e., it has been declared) before committing the state update. If the class hash is not declared, write a failure response and return without modifying state. This is exactly what the existing TODO comment at line 898 calls for.

---

### Proof of Concept

1. **Attacker** declares a malicious Cairo class `MaliciousLib` whose sole logic is:
   ```
   replace_class(class_hash=0)
   ```
2. **Victim** is a contract holding user funds that supports `library_call` with a user-supplied class hash (e.g., a proxy or plugin pattern).
3. **Attacker** submits a transaction calling the victim contract with `MaliciousLib`'s class hash as the library target.
4. The victim contract executes `library_call(class_hash=MaliciousLib, selector=..., ...)`.
5. Inside `execute_library_call`, the execution context is constructed with `contract_address = victim_address` and `class_hash = MaliciousLib`.
6. `MaliciousLib` issues `replace_class(0)`.
7. `execute_replace_class` is called with `contract_address = victim_address` and `class_hash = 0`.
8. **No validation occurs** (line 898 TODO is unimplemented). The OS writes `StateEntry(class_hash=0, ...)` for the victim address into `contract_state_changes`.
9. The block is proven and the new state root — containing `victim_address → class_hash=0` — is committed to L1.
10. All subsequent calls to the victim contract fail at class resolution. Funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
