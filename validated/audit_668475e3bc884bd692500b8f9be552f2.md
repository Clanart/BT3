### Title
Sequencer ECDSA Signing Private Key Stored Unprotected in Memory and Exposed via `Debug` Derive on `LocalKeyStore` — (`File: crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

The `LocalKeyStore` struct, which holds the sequencer's ECDSA private key used to sign BFT precommit votes and peer-identity challenges, derives `Debug` without redacting the `private_key` field. The enclosing `SignatureManager<KS>` also derives `Debug` and exposes the keystore via a `pub` field. Any debug-level log, panic message, or `{:?}` format call that touches either type prints the raw private key as a hex `Felt`. Compounding this, the production entry point `create_signature_manager()` unconditionally calls `LocalKeyStore::new_for_testing()`, which hardcodes a publicly known private key in source code. An attacker who can read process memory, access debug logs, or simply read the repository obtains the key and can forge precommit votes over arbitrary block hashes, binding the wrong block commitment to a consensus decision.

---

### Finding Description

**Root cause 1 — `Debug` derive leaks the private key through logs/panics**

`LocalKeyStore` is declared with `#[derive(Clone, Copy, Debug)]`: [1](#0-0) 

`PrivateKey` itself also derives `Debug` (and `Serialize`/`Deserialize`), printing the raw inner `Felt`: [2](#0-1) 

`SignatureManager<KS>` derives `Debug` and exposes the keystore as a `pub` field: [3](#0-2) 

`LocalKeyStoreSignatureManager` (the concrete production type) also derives `Debug`: [4](#0-3) 

Any `tracing` span, `{:?}` format, or panic that captures one of these types will emit the private key in plaintext to the log stream. The `tracing` subscriber configured for the sequencer writes to stdout/files with `with_file(true)` and `with_line_number(true)`: [5](#0-4) 

**Root cause 2 — Production code uses a hardcoded, publicly known private key**

The public production factory `create_signature_manager()` (no `#[cfg(test)]` guard) calls `SignatureManager::new()`, which calls `LocalKeyStore::new_for_testing()`: [6](#0-5) 

`new_for_testing()` hardcodes the private key `0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133` directly in source: [7](#0-6) 

The `_new` constructor that accepts a caller-supplied key exists but is dead code (prefixed `_`): [8](#0-7) 

**Root cause 3 — `pub keystore` field allows direct key extraction without `get_key()`**

Because `keystore` is `pub`, any code holding a `&SignatureManager<LocalKeyStore>` can call `sm.keystore.get_key().await` directly, bypassing any future access-control wrapper: [9](#0-8) 

---

### Impact Explanation

The private key is used in two signing paths:

1. `sign_precommit_vote(block_hash)` — signs the BFT precommit vote that commits the sequencer to a specific `BlockHash`: [10](#0-9) 

2. `sign_identification(peer_id, challenge)` — authenticates the sequencer node on the P2P layer: [11](#0-10) 

An attacker who obtains the private key can forge precommit votes over any `BlockHash`, causing the consensus layer to accept a commitment to a wrong block hash/state root. Because the hardcoded key is shared across all nodes that use `create_signature_manager()`, a single key compromise allows forging votes attributed to every such node simultaneously, trivially reaching the 2/3+ quorum threshold needed for a decision.

This matches the allowed impact: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

- **Trivially known key**: The private key `0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133` is hardcoded in a public source file. No memory dump is required.
- **`Debug` exposure**: Any debug-level log line that formats the `SignatureManager` or `LocalKeyStore` (e.g., during startup, error handling, or test output) prints the key in plaintext.
- **`pub keystore` field**: Any component that receives a `SignatureManager` reference can extract the key programmatically without privilege escalation.
- **No `#[cfg(test)]` guard**: `create_signature_manager()` and `LocalKeyStoreSignatureManager::new()` are unconditionally compiled into production binaries.

---

### Recommendation

1. **Redact `Debug` output**: Implement a custom `Debug` for `LocalKeyStore` that prints `private_key: [REDACTED]` instead of the raw value.
2. **Remove `pub` from `keystore`**: Change `pub keystore: KS` to `keystore: KS` in `SignatureManager<KS>` to prevent direct field access.
3. **Remove the hardcoded key from production**: Gate `new_for_testing()` behind `#[cfg(any(test, feature = "testing"))]` and implement a proper `new(private_key: PrivateKey)` constructor for production use, loading the key from a secrets manager or environment variable.
4. **Implement `Zeroize`**: Implement the `zeroize::Zeroize` trait for `PrivateKey` and `LocalKeyStore` so the key material is cleared from memory when dropped.
5. **Audit `Serialize`/`Deserialize` on `PrivateKey`**: The derived `Serialize` on `PrivateKey` means the key can be serialized into any log sink or RPC response that serializes a containing struct.
