### Title
Missing `shard_id` Binding in `set_state_header` Allows Cross-Shard State Substitution — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and validates it against `sync_hash`, but **never checks that the chunk embedded in the header belongs to the requested `shard_id`**. A malicious peer can supply a header whose `chunk` field is for a different shard (with a valid merkle proof for that shard's position in the block) while providing the legitimate receipt proofs for the requested shard. All five validation steps pass, the header is stored under the wrong shard's key, and subsequent state-part downloads and application install the wrong shard's trie into the syncing node's storage.

---

### Finding Description

`set_state_header` performs five checks:

1. `validate_chunk_proofs` — internal chunk consistency (encoded merkle root, signature)
2. `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — proves the chunk is somewhere in the block
3. `verify_path` for `prev_chunk_header` — proves the previous chunk is in the block
4. Receipt-proof loop — validates incoming receipts using the caller-supplied `shard_id`
5. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the state root node against the chunk's own `prev_state_root`

**What is never checked**: `chunk.shard_id() == shard_id`.

The merkle proof in step 2 proves the chunk is at *some* position in the block's `chunk_headers_root`; it does not prove the position corresponds to the requested `shard_id`. The chunk's internal `shard_id` field (part of `ShardChunkHeaderInner`) is committed to in the chunk hash, but the code never compares it to the `shard_id` argument.

**Attack construction**:

A malicious peer responding to a state-header request for `(shard_id_A, sync_hash)` sends:
- `chunk` = the genuine chunk for `shard_id_B` (different shard), with its valid merkle proof for `shard_id_B`'s position in the block
- `incoming_receipts_proofs` = the genuine incoming receipts for `shard_id_A` (with valid proofs against the block's `prev_chunk_outgoing_receipts_root`)
- `state_root_node` = the genuine state root node for `shard_id_B`'s trie

Step 2 passes: the chunk for `shard_id_B` is genuinely in the block.
Step 4 passes: the receipt proofs use `shard_id_A` (the caller-supplied value) and are valid for `shard_id_A`.
Step 5 passes: the state root node matches `chunk.prev_state_root()`, which is `shard_id_B`'s state root.

The header is stored under `StateHeaderKey(shard_id_A, sync_hash)`.

When `set_state_part` is subsequently called, it retrieves this header, extracts `shard_id_B`'s `prev_state_root`, and validates incoming parts against it. A malicious peer serving state parts for `shard_id_B` passes `validate_state_part` (which only checks the trie proof against the stored state root). `apply_state_part` then writes `shard_id_B`'s trie nodes into `shard_uid_A`'s storage column. ### Impact Explanation

After the attack completes:

- The syncing node's storage for `shard_id_A` contains `shard_id_B`'s trie, rooted at `shard_id_B`'s state root.
- `set_state_finalize` applies the chunk (which is `shard_id_B`'s chunk) to `shard_id_A`'s storage, producing a `ChunkExtra` with `shard_id_B`'s post-state root stored under `shard_uid_A`.
- Every subsequent block application for `shard_id_A` reads the wrong state, producing incorrect execution results, wrong outgoing receipts, and invalid chunk state witnesses.
- If the node is a chunk producer or validator for `shard_id_A`, it emits witnesses that all other validators reject, causing missed blocks and potential slashing.
- The corrupted state persists until the node re-syncs from scratch. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

Any peer in the network can respond to a `StateRequestHeader` message. The attack requires:

1. Constructing a header with a chunk from a different shard — trivially available from the block.
2. Providing valid receipt proofs for the *requested* shard — also available from the block.
3. Serving matching state parts for the substituted shard — available from any node tracking that shard.

No privileged role is required. The attack is feasible for any network participant who can intercept or respond to state sync requests, which is the normal peer-to-peer state sync model. [6](#0-5) [7](#0-6) 

---

### Recommendation

Add an explicit shard binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the existing `shard_id` membership check already present in `get_state_response_part` and `compute_state_response_header`, and closes the identity-assumption gap analogous to the external report. [8](#0-7) [9](#0-8)

### Proof of Concept

```
1. Honest node N begins state sync for epoch E, requesting state header for
   (shard_id = 0, sync_hash = H).

2. Malicious peer M intercepts the StateRequestHeader message.

3. M constructs a ShardStateSyncResponseHeader where:
     - chunk          = block[H-1].chunks[shard_index_1]   // shard 1's chunk
     - chunk_proof    = merkle_path_for_shard_1_in_block   // valid proof for shard 1
     - incoming_receipts_proofs = genuine receipts for shard 0 from block H
     - root_proofs    = genuine root proofs for shard 0 from block H
     - state_root_node = state_root_node for shard 1's trie

4. N calls set_state_header(shard_id=0, sync_hash=H, malicious_header):
     - validate_chunk_proofs(chunk_1)          → OK (chunk 1 is internally valid)
     - verify_path(chunk_headers_root, proof_1, ChunkHashHeight(chunk_1.hash, ...))
                                               → OK (chunk 1 is in the block)
     - receipt proof loop with shard_id=0      → OK (receipts are genuinely for shard 0)
     - validate_state_root_node(node_1, chunk_1.prev_state_root)
                                               → OK (node matches shard 1's root)
     - chunk.shard_id() == shard_id?           → NOT CHECKED
     - Header stored under StateHeaderKey(0, H) with shard 1's state root.

5. N requests state parts for (shard_id=0, sync_hash=H, part_id=0..N).
   M serves state parts for shard 1.
   validate_state_part(shard_id=0, shard_1_state_root, part_id, part_for_shard_1) → OK.
   Parts stored under StatePartKey(H, 0, part_id).

6. apply_state_part writes shard 1's trie nodes into shard_uid_0's storage.
   set_state_finalize applies shard 1's chunk to shard 0's storage.
   Node N now has shard 1's state in shard 0's slot.
```

    pub fn set_state_finalize(
        &mut self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
        shard_state_header: ShardStateSyncResponseHeader,
    ) -> Result<ShardUId, Error> {
        let _span =
            tracing::debug_span!(target: "sync", "chain_update_set_state_finalize", %shard_id, ?sync_hash).entered();
        let (chunk, incoming_receipts_proofs) = match shard_state_header {
            ShardStateSyncResponseHeader::V1(shard_state_header) => (
                ShardChunk::V1(shard_state_header.chunk),
                shard_state_header.incoming_receipts_proofs,
            ),
            ShardStateSyncResponseHeader::V2(shard_state_header) => {
                (shard_state_header.chunk, shard_state_header.incoming_receipts_proofs)
            }
        };

        // Note that block headers are already synced and can be taken
        // from store on disk.
        let block_header = get_block_header_on_chain_by_height(
            &self.chain_store_update.chain_store(),
            &sync_hash,
            chunk.height_included(),
        )?;

        // Getting actual incoming receipts.
        let mut receipt_proof_responses: Vec<ReceiptProofResponse> = vec![];
        for incoming_receipt_proof in &incoming_receipts_proofs {
            let ReceiptProofResponse(hash, _) = incoming_receipt_proof;
            let block_header = self.chain_store_update.get_block_header(hash)?;
            if block_header.height() <= chunk.height_included() {
                receipt_proof_responses.push(incoming_receipt_proof.clone());
            }
        }
        let receipts = collect_receipts_from_response(&receipt_proof_responses);
        let is_genesis = block_header.height() == self.chain_store_update.get_genesis_height();
        let prev_block_header = (!is_genesis)
            .then(|| self.chain_store_update.get_block_header(block_header.prev_hash()))
            .transpose()?;

        // Prev block header should be present during state sync, since headers have been synced at
        // this point, except for genesis.
        let gas_price = if let Some(prev_block_header) = &prev_block_header {
            prev_block_header.next_gas_price()
        } else {
            block_header.next_gas_price()
        };

        let chunk_header = chunk.cloned_header();
        let gas_limit = chunk_header.gas_limit();
        let block = self.chain_store_update.get_block(block_header.hash())?;
        let transactions = chunk.to_transactions().to_vec();
        let transaction_validity = if let Some(prev_block_header) = prev_block_header {
            self.chain_store_update
                .chain_store()
                .compute_transaction_validity(&prev_block_header, &chunk)
        } else {
            vec![true; transactions.len()]
        };
        let transactions = SignedValidPeriodTransactions::new(transactions, transaction_validity);
        let shard_uid =
            shard_id_to_uid(self.epoch_manager.as_ref(), shard_id, block_header.epoch_id())?;

            .runtime_adapter

                gas_limit,

                congestion_info: block.block_congestion_info(),
                bandwidth_requests: block.block_bandwidth_requests(),
            },
            &receipts,
            transactions,
        )?;

async fn apply_state_part(
    store: Store,
    runtime: Arc<dyn RuntimeAdapter>,
    computation_task_tracker: TaskTracker,
    cancel: CancellationToken,
    sync_hash: CryptoHash,
    shard_id: ShardId,
    part_id: u64,
    num_parts: u64,
    state_root: CryptoHash,
    epoch_id: EpochId,
) -> Result<StatePartApplyResult, near_chain::Error> {
    let key = StatePartKey(sync_hash, shard_id, part_id);
    let key_bytes = borsh::to_vec(&key).unwrap();
    let already_applied = store.exists(DBCol::StatePartsApplied, &key_bytes);
    if already_applied {
        tracing::debug!(target: "sync", ?key, "state part already applied, skipping");
        return Ok(StatePartApplyResult::AlreadyApplied);
    }
    return_if_cancelled!(cancel);
    let handle =
        computation_task_tracker.get_handle(&format!("shard {} part {}", shard_id, part_id)).await;
    return_if_cancelled!(cancel);
    handle.set_status("Loading part data from store");
    let bytes = store
        .get(DBCol::StateParts, &key_bytes)
        .ok_or_else(|| {
            near_chain::Error::DBNotFoundErr(format!(
                "No state part {} for shard {}",
                part_id, shard_id
            ))
        })?
        .to_vec();
    let state_part = StatePart::from_bytes(bytes)?;
    handle.set_status("Applying part data to runtime");
    runtime.apply_state_part(
        shard_id,
        &state_root,
        PartId { idx: part_id, total: num_parts },
        &state_part,
        &epoch_id,
    )?;
    tracing::debug!(target: "sync", ?key, "applied state part");

    // Mark part as applied.
    let mut store_update = store.store_update();
    store_update.set_ser(DBCol::StatePartsApplied, &key_bytes, &true);
    store_update.commit();

    Ok(StatePartApplyResult::Applied)
