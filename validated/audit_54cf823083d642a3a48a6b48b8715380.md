### Title
`ExecutionStateReader::get_compiled_class_hash` Missing State-Number Boundary Check Returns Future-Block Compiled Class Hash — (`crates/apollo_rpc_execution/src/state_reader.rs`)

---

### Summary

`ExecutionStateReader::get_compiled_class_hash` calls `get_class_definition_block_number` to find the block where a class was declared, then unconditionally fetches and returns the compiled class hash from that block's state diff — without ever checking whether that block is visible at the reader's `state_number`. Every sibling function that performs the same lookup (`get_class_definition_at`, `get_contract_class`, `get_compiled_class_hash_at`) applies an explicit `is_before` guard; this one does not. The existing test suite confirms the wrong behavior at line 186 of `state_reader_test.rs`.

---

### Finding Description

`ExecutionStateReader::get_compiled_class_hash` is implemented as follows: [1](#0-0) 

After obtaining `block_number` from `get_class_definition_block_number`, the code immediately fetches the state diff at that block and returns the compiled class hash. There is no guard of the form:

```rust
if self.state_number.is_before(block_number) {
    return Ok(CompiledClassHash::default());
}
```

Every other function that performs the same lookup applies this guard:

- `get_class_definition_at` (storage layer): [2](#0-1) 

- `get_contract_class` (execution utils): [3](#0-2) 

- `get_compiled_class_hash_at` (storage layer) uses `state_number.block_after()` as the exclusive upper bound: [4](#0-3) 

`get_class_definition_block_number` itself is a raw table lookup with no state-number filtering: [5](#0-4) 

The existing test suite **confirms the wrong behavior** at line 186. The reader has `state_number = right_after_block(0)`, `class_hash0` is declared in block 1, yet the assertion expects the block-1 compiled class hash to be returned: [6](#0-5) 

Note that `get_compiled_class` (line 181–185) correctly returns `UndeclaredClassHash` for the same reader and class hash — demonstrating the inconsistency between the two sibling methods on the same struct.

---

### Impact Explanation

`ExecutionStateReader` is the `StateReader` implementation passed to blockifier's `CachedState` for all RPC-driven execution paths: `starknet_simulateTransactions`, `starknet_traceTransaction`, `starknet_estimateFee`, and historical re-execution. [7](#0-6) 

Blockifier calls `get_compiled_class_hash` to determine whether a class is already declared before processing a declare transaction. When re-executing block N (state_number = `right_before_block(N)`) that contains a declare of class C:

1. `get_compiled_class_hash(C)` returns the compiled class hash from block N's state diff (wrong — should return `CompiledClassHash::default()`).
2. Blockifier observes a non-default value and treats C as already declared.
3. The declare transaction is incorrectly rejected during re-execution.
4. The RPC response returns a wrong execution result (reverted/failed instead of succeeded) for a transaction that was originally accepted and committed.

This is a concrete wrong committed value served authoritatively to callers of simulation, tracing, and fee-estimation endpoints.

---

### Likelihood Explanation

The trigger condition is straightforward: any RPC call that re-executes or simulates a block containing a Cairo 1 declare transaction. This is a normal, unprivileged operation. The class hash is attacker-controlled (any declared class hash suffices). No special privileges, storage access, or peer manipulation are required.

---

### Recommendation

Add the missing boundary check immediately after obtaining `block_number`:

```rust
let Some(block_number) = maybe_block_number else {
    return Ok(CompiledClassHash::default());
};

// Guard: class declared in block_number is not yet visible at self.state_number.
if self.state_number.is_before(block_number) {
    return Ok(CompiledClassHash::default());
}
```

This mirrors the guard already present in `get_class_definition_at` and `get_contract_class`. The test at line 186 of `state_reader_test.rs` must be updated to assert `CompiledClassHash::default()` for a reader whose `state_number` precedes the declaring block.

---

### Proof of Concept

The existing test already demonstrates the bug. With `state_number = right_after_block(0)` and `class_hash0` declared in block 1:

```
state_reader0.get_compiled_class(class_hash0)
  → Err(UndeclaredClassHash)          // correct: boundary check present

state_reader0.get_compiled_class_hash(class_hash0)
  → Ok(compiled_class_hash0)          // WRONG: boundary check absent
``` [6](#0-5) 

A minimal Rust unit test to prove the boundary violation:

```rust
// class_hash declared in block 1
// ExecutionStateReader with state_number = right_before_block(1)
let state_number = StateNumber::right_before_block(BlockNumber(1)).unwrap();
let reader = ExecutionStateReader { storage_reader, state_number, ... };
// Should return CompiledClassHash::default() — class not yet visible
assert_eq!(reader.get_compiled_class_hash(class_hash).unwrap(), CompiledClassHash::default());
// Currently returns the block-1 compiled_class_hash instead, proving the missing guard.
```
