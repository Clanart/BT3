### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in `transaction_impls.cairo` computes the maximum chargeable fee using unchecked felt arithmetic. Because Cairo felt operations are implicitly reduced modulo the STARK prime P (~2²⁵¹), a user can craft `ResourceBounds` values whose mathematical sum equals exactly P (or 2P), causing the felt result to be 0. The `charge_fee` function then short-circuits on `max_fee == 0` and skips fee collection entirely, allowing the transaction to execute for free.

---

### Finding Description

In `charge_fee`, the OS first computes the maximum fee:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [1](#0-0) 

`compute_max_possible_fee` performs the following computation entirely in felt arithmetic — no range checks, no overflow guards:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [2](#0-1) 

Each term is a felt product, so each is in `[0, P)`. The sum of three such terms is in `[0, 3P)`. In felt arithmetic the result is taken mod P. If the mathematical sum equals exactly P or 2P, the felt result is 0.

When `max_fee == 0`, `charge_fee` returns immediately — the `LoadActualFee` hint is never reached, `assert_nn_le` is never executed, and no ERC-20 transfer is initiated. The transaction executes with zero fee paid. [3](#0-2) 

This is structurally identical to the reported Sandclock bug: a fee-related parameter (here, the computed `max_fee`) is effectively ignored because the arithmetic that produces it is not validated for overflow, causing the fee enforcement gate to be silently bypassed.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The sequencer is entitled to collect fees for every executed transaction. When `max_fee` overflows to 0, the OS skips the ERC-20 transfer to the sequencer address entirely. The sequencer receives no compensation for the computational resources consumed. At scale, an attacker can submit a stream of crafted transactions that execute arbitrary logic (state writes, contract calls, etc.) at zero cost, draining sequencer revenue and enabling unbounded free computation against the network.

---

### Likelihood Explanation

**High.**

The Rust-based sequencer (blockifier) validates fee bounds using standard integer arithmetic (u128/u256), not felt arithmetic. A transaction whose resource bounds overflow to 0 mod P will appear to have a large, valid `max_fee` in the Rust validator (mathematical sum ≈ P ≈ 2²⁵¹) and will be accepted into a block. The Cairo OS then recomputes `max_fee` in felt arithmetic, obtains 0, and skips the fee charge. No privileged access is required — any transaction sender can craft the necessary `ResourceBounds` values. The transaction hash commits to the crafted values but does not prevent their inclusion.

---

### Recommendation

Add explicit range checks on each `ResourceBounds` field (`max_amount`, `max_price_per_unit`) to enforce that they fit within a safe sub-field range (e.g., `[0, 2⁶⁴)` or `[0, 2¹²⁸)`), ensuring that the sum of all fee terms cannot reach P. Alternatively, perform the fee computation using checked multi-limb arithmetic (analogous to `Uint256`) rather than raw felt operations, so overflow is impossible by construction.

---

### Proof of Concept

Choose the following `ResourceBounds` for a V3 Invoke transaction:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `P − 1` (a valid felt) |
| `l1_gas_bounds.max_price_per_unit` | `1` |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `l1_data_gas_bounds.max_amount` | `0` |
| `l1_data_gas_bounds.max_price_per_unit` | `0` |
| `tip` | `0` |

Felt arithmetic in `compute_max_possible_fee`:

```
term1 = (P−1) * 1 = P−1
term2 = 1 * (1 + 0) = 1
term3 = 0 * 0 = 0
sum   = (P−1) + 1 + 0 = P ≡ 0  (mod P)
```

The Rust sequencer computes the mathematical sum ≈ P ≈ 2²⁵¹ using u256, sees a large non-zero `max_fee`, and includes the transaction in a block. The Cairo OS evaluates `compute_max_possible_fee` → `0`, hits the early-return branch in `charge_fee`, and executes the transaction without transferring any fee to the sequencer. [4](#0-3)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    local low_actual_fee;

    let (__fp__, _) = get_fp_and_pc();

    // execute-mode rather than validate-mode.

    non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
        block_context=block_context, execution_context=&execution_context
    );
    return ();
}
