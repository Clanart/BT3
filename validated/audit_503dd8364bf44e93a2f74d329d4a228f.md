### Title
Missing Nonce in `meta_tx_v0` Transaction Hash Enables Unlimited Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes a transaction hash with `additional_data_size=0`, meaning **no nonce is included** in the signed message. Combined with the OS explicitly skipping nonce enforcement for version-0 transactions, any signature produced over a `meta_tx_v0` hash is valid forever and can be replayed by any party who observes it.

---

### Finding Description

`compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

This means the hash is a pure function of `(INVOKE_PREFIX, 0, contract_address, selector, calldata, 0, chain_id)` — no nonce, no block number, no uniqueness token of any kind.

In `execute_meta_tx_v0`, the resulting `new_tx_info` is constructed with `nonce=0`: [2](#0-1) 

The OS-level nonce guard, `check_and_increment_nonce`, explicitly returns early for version 0: [3](#0-2) 

`run_validate` is also skipped for version 0: [4](#0-3) 

`execute_meta_tx_v0` enforces that `selector == EXECUTE_ENTRY_POINT_SELECTOR` and then calls `contract_call_helper` directly — no `__validate__` step, no nonce increment, no replay tracking: [5](#0-4) [6](#0-5) 

The target contract's `__execute__` entry point receives `tx_info.transaction_hash` (the nonce-free hash) and `tx_info.signature_start/end`. If the contract verifies the signature against this hash (the standard pattern for account contracts), the same signature is valid in every future block.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who authorizes a `meta_tx_v0` call (e.g., a token transfer) produces a signature over a hash that is permanently valid. Any party who observes the signature — from on-chain calldata, mempool, or block history — can replay the identical `meta_tx_v0` syscall in any subsequent transaction, causing the target contract's `__execute__` to re-execute the same transfer repeatedly. There is no OS-enforced bound on how many times the same hash/signature pair can be used.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is a public protocol feature callable by any contract. Any contract that uses it with a user-supplied signature (the intended use case, given the signature field in `MetaTxV0Request`) is immediately vulnerable. An attacker only needs to observe one successful execution — the signature and calldata are visible on-chain — and can replay it in any future block at zero additional cost.

---

### Recommendation

Include a unique, monotonically increasing nonce in `compute_meta_tx_v0_hash` — analogous to how `compute_l1_handler_transaction_hash` passes `additional_data_size=1, additional_data=&nonce`. The OS must then enforce uniqueness of that nonce (e.g., by reading and incrementing the target contract's nonce in `contract_state_changes`, as `check_and_increment_nonce` does for v1/v2/v3 transactions). Without this, every `meta_tx_v0` signature is a permanent, replayable authorization.

---

### Proof of Concept

1. User U signs a `meta_tx_v0` message authorizing transfer of 100 tokens from contract `C` to attacker `A`. The hash is `H = Pedersen(INVOKE_PREFIX, 0, C, __execute__, calldata, 0, chain_id)`.
2. U submits a transaction that calls a relay contract `R`, which invokes `meta_tx_v0(contract_address=C, selector=__execute__, calldata=..., signature=sig_U)`.
3. The OS computes `meta_tx_hash = H`, sets `tx_info.transaction_hash = H`, `tx_info.nonce = 0`, and calls `C.__execute__`.
4. `C.__execute__` verifies `sig_U` against `H` — valid — and transfers 100 tokens to `A`.
5. In the next block, attacker `A` submits a new transaction calling `R` (or any contract) with the same `meta_tx_v0` parameters and `sig_U`.
6. The OS again computes `H` (identical, no nonce), calls `C.__execute__`, which again verifies `sig_U` against `H` — still valid — and transfers another 100 tokens.
7. Step 5–6 repeats until `C`'s balance is drained.
