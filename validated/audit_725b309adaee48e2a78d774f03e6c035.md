### Title
Unconstrained `is_reverted` Flag Allows Prover to Skip Invoke Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is an empty stub that performs no validation whatsoever. The `is_reverted` flag — which controls whether the execute step of an invoke transaction runs — is set exclusively by an unverified hint and is never constrained by any Cairo assertion. A malicious prover can freely set `is_reverted = 1` for any invoke transaction, causing execution to be skipped while fees are still charged, producing a valid STARK proof for an incorrect state transition.

---

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`, lines 338–358), the `is_reverted` flag is populated by a hint and then passed to `check_is_reverted` before branching:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    // Align the stack ...
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}
``` [1](#0-0) 

The function `check_is_reverted` in `execution_constraints.cairo` is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This function contains **no assertions, no range checks, and no constraints of any kind**. It is a direct analog to the external report's `_Ownable_init()` never being called: just as the missing initializer left the owner as `address(0)` making all `onlyOwner` guards meaningless, the empty `check_is_reverted` leaves `is_reverted` completely unconstrained in the STARK proof.

In Cairo, hints (`%{ ... %}`) are **not part of the proof**. They are witness values provided by the prover. The STARK proof only verifies that Cairo assertions hold. Since `check_is_reverted` asserts nothing, the prover can assign any value to `is_reverted` without invalidating the proof.

The same pattern appears in `execute_l1_handler_transaction`, where `is_reverted` is used directly from a hint with **no call to `check_is_reverted` at all**:

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [3](#0-2) 

After the `if (is_reverted == FALSE)` branch, `charge_fee` is called unconditionally regardless of the revert status:

```cairo
// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious sequencer/prover can:

1. **For invoke transactions**: Set `is_reverted = 1` for any transaction that should succeed. The execute step (`non_reverting_select_execute_entry_point_func`) is skipped — no state changes from execution are applied — but `charge_fee` still runs, deducting fees from the user. The resulting STARK proof is valid because no Cairo assertion is violated. Users pay fees for transactions that are never executed.

2. **For L1 handler transactions**: Set `is_reverted = 1` for any L1-to-L2 message handler. The handler is not executed and `consume_l1_to_l2_message` is never called, meaning the message is not recorded in the OS output. Funds sent via L1-to-L2 messages are permanently frozen: consumed on L1 (the block is accepted) but never processed on L2.

The STARK proof produced for such a block is cryptographically valid — the verifier cannot distinguish it from a correctly-executed block — because the constraint that was supposed to enforce correctness of `is_reverted` is simply absent.

---

### Likelihood Explanation

The attack requires a malicious sequencer, which is the entity that provides hints to the OS. In the current StarkNet architecture the sequencer is trusted, but the entire purpose of the STARK proof system is to make the sequencer's behavior verifiable and trustless. This vulnerability directly undermines that guarantee: the proof system accepts blocks where the sequencer arbitrarily skips transaction execution. The entry path is any invoke or L1-handler transaction submitted by any unprivileged user.

---

### Recommendation

The `check_is_reverted` function must enforce that `is_reverted` is consistent with the actual execution outcome. At minimum it should assert that `is_reverted` is a boolean:

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (is_reverted - 1) = 0;
    return ();
}
```

More robustly, the OS should derive `is_reverted` from the actual execution result rather than accepting it as an unconstrained hint. The value should be constrained against the entry-point return values (specifically `entry_point_return_values.failure_flag` in `execute_entry_point.cairo`) so that the proof enforces the correct relationship between execution outcome and the revert flag. [5](#0-4) 

---

### Proof of Concept

1. Sequencer constructs a block containing an invoke transaction `T` that transfers 100 STRK from user `A` to user `B`.
2. When executing the OS, the sequencer provides hint `%{ IsReverted %}` → `is_reverted = 1`.
3. `check_is_reverted(1)` is called — returns immediately with no assertion.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called; the 100 STRK transfer never happens.
5. `charge_fee` is called unconditionally; user `A` is charged the full fee.
6. The OS produces a state commitment reflecting the state without the transfer.
7. The STARK proof is generated and verified successfully — no Cairo assertion was violated.
8. On-chain: the block is accepted, user `A` has paid fees, user `B` has received nothing, and the proof is indistinguishable from a correctly-executed block. [2](#0-1) [6](#0-5)
