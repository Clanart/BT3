### Title
`batch_insert` Silently Overwrites Duplicate Keys Without Freeing Old Indexes, Corrupting MerkleBlob Tree Root and Proof State — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `insert` enforces for all items beyond the first two. When a caller supplies duplicate `KeyId` or `Hash` values in a batch of three or more entries, `insert_entry_to_blob` → `add_leaf` silently overwrites the `key_to_index` and `leaf_hash_to_index` cache entries without freeing the old index. The old leaf node remains physically present in the blob and is wired into the tree's parent/child structure, but the `BlockStatusCache` no longer tracks it correctly. The result is a permanently corrupted tree: the root hash includes the orphaned duplicate leaf, `get_proof_of_inclusion` generates proofs against the wrong tree state, and `leaf_hash_to_index` retains a stale entry pointing to the orphaned node.

---

### Finding Description

**Normal `insert` path (safe):**

`insert` checks both guards before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

**`batch_insert` bulk path (unsafe):**

When `leaf_count > 1` and the batch has more than two items, the first `N-2` entries are inserted through the bulk path. No duplicate check is performed:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` unconditionally calls `add_leaf`:

```rust
// lines 1024-1027
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    Node::Internal(..) => self.block_status_cache.add_internal(index),
}
``` [3](#0-2) 

`add_leaf` uses `HashMap::insert`, which silently overwrites any existing entry and discards the old `TreeIndex`:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // old index silently lost
    self.leaf_hash_to_index.insert(leaf.hash, index);   // old hash entry silently lost
}
``` [4](#0-3) 

**What happens to the old index:**

`remove_leaf` (the correct removal path) adds the old index to `free_indexes`:

```rust
// lines 199-208
fn remove_leaf(&mut self, node: &LeafNode) -> Result<(), Error> {
    let Some(index) = self.key_to_index.remove(&node.key) else { ... };
    self.leaf_hash_to_index.remove(&node.hash);
    self.free_indexes.insert(index);
    Ok(())
}
``` [5](#0-4) 

`batch_insert` never calls `remove_leaf` for the displaced entry. The old index is therefore:
- **Not** in `free_indexes` (not reclaimable)
- **Not** in `key_to_index` (unreachable by key lookup)
- **Still** physically present in the blob with valid parent/child pointers
- **Still** reachable via `leaf_hash_to_index` if the old hash differs from the new hash (stale pointer)

The tree's parent/child structure includes the orphaned node, so `calculate_lazy_hashes` and `get_root_hash` incorporate it into the root hash computation, producing a root that diverges from any externally expected value. [6](#0-5) 

The Python binding exposes `batch_insert` directly to callers with no additional validation:

```python

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Concrete consequences:

1. **Corrupted root hash.** The orphaned duplicate leaf is wired into the tree and participates in `internal_hash` computation up to the root. The committed root hash no longer represents the intended key-value set.

2. **Stale `leaf_hash_to_index` entry.** If the duplicate entries carry different hashes, `get_node_by_hash(old_hash)` resolves to the orphaned node. Any delta-sync or proof logic that uses this lookup operates on a phantom node.

3. **Invalid proof-of-inclusion.** `get_proof_of_inclusion(key)` resolves the key to the *new* index via `key_to_index`, builds a Merkle path from that position, and returns a proof. But the root hash was computed over a tree that also contains the orphaned old leaf. The proof path is inconsistent with the root, so verification fails — or, if the orphaned node happens to be on the same path, the proof silently covers the wrong data.

4. **`check_integrity` would detect this** (it verifies that every leaf in the blob is in the cache and vice versa), but `check_integrity_on_drop` is only enabled in `#[cfg(test)]` builds; production code runs with `check_integrity_on_drop: cfg!(test)` which is `false` in release. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

`batch_insert` is a public API exposed through both the Rust crate and the Python wheel. Any caller that constructs a batch without pre-deduplicating keys — whether due to a bug in the caller or deliberate adversarial input — triggers the corruption. The DataLayer server calls this function when applying updates received from the network. An attacker who can submit a DataLayer update containing duplicate keys in a batch of three or more entries can corrupt the Merkle tree of any store they have write access to. [10](#0-9) 

---

### Recommendation

Add duplicate-key and duplicate-hash guards at the top of the bulk insertion loop in `batch_insert`, mirroring the checks in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    let new_leaf_index = self.get_new_index();
    // ... rest unchanged
}
```

Alternatively, pre-validate the entire input vector for duplicates before any blob mutation begins, so the function is atomic (all-or-nothing).

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so leaf_count > 1 (bulk path is taken for item 0)
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &Hash(Bytes32::new([i as u8; 32])),
                    InsertLocation::Auto {}).unwrap();
    }

    let dup_key   = KeyId(99);
    let hash_a    = Hash(Bytes32::new([0xAA; 32]));
    let hash_b    = Hash(Bytes32::new([0xBB; 32]));

    // batch of 3: item 0 (dup_key/hash_a) goes through bulk path (no guard),
    // items 1-2 go through insert() (popped from end).
    blob.batch_insert(vec![
        ((dup_key, ValueId(1)), hash_a),   // bulk path — no duplicate check
        ((dup_key, ValueId(2)), hash_b),   // insert() — would catch dup, but key not yet in cache
        ((KeyId(100), ValueId(3)), Hash(Bytes32::new([0xCC; 32]))),
    ]).unwrap();

    // Both hash_a and hash_b are now in leaf_hash_to_index.
    // The blob contains two leaf nodes with key=99.
    // Root hash is computed over both — corrupted tree.
    // check_integrity() would panic here in a test build.
    let root = blob.get_root_hash().unwrap();
    println!("Corrupted root: {:?}", root);

    // get_proof_of_inclusion returns proof for the *second* dup leaf,
    // but root hash includes the orphaned first leaf — proof is inconsistent.
    let proof = blob.get_proof_of_inclusion(dup_key).unwrap();
    println!("Proof valid against corrupted root: {}", proof.valid());
}

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
