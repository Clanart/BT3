### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. Any contract can call `replace_class` with an arbitrary, undeclared class hash, permanently rendering itself uncallable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts any felt value as the new class hash and writes it directly into the contract's `StateEntry` without checking whether that hash exists in the declared class registry:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;
    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}
    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
``` [1](#0-0)

---

### Impact Explanation

Once a contract's `StateEntry.class_hash` is set to an undeclared value (e.g., a random felt `0xdeadbeef`), the OS cannot resolve a compiled class for it in any subsequent block. The `validate_compiled_class_facts_post_execution` step requires the prover to supply a valid CASM entry for every class hash used during execution; no such entry can exist for an undeclared hash. Every future call to the contract will therefore fail at the proof level, making the contract permanently uncallable. Any ERC-20 balances, ETH, or protocol-level assets stored in the contract's storage are irrecoverably frozen.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack surface is broad:

1. Any deployed contract can issue the `replace_class` syscall — no special privilege is required.
2. The sequencer's pre-execution simulation does not fail on `replace_class` itself (the syscall succeeds; the breakage only manifests on the next call to the contract).
3. A malicious contract owner of any multi-user protocol (DEX, lending pool, bridge escrow) can trigger this in a single transaction, freezing all depositor funds.
4. A buggy upgrade script that passes the wrong class hash achieves the same result accidentally.

**Likelihood: High** — the entry path is a standard user-submitted transaction; no privileged role is needed.

---

### Recommendation

Before writing the new `StateEntry`, assert that `class_hash` is present in `contract_class_changes` (i.e., was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the returned compiled class hash is non-zero, mirroring the check already enforced in `execute_declare_transaction`:

```cairo
// Ensure the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

The same fix must be applied to `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. **Deploy** a multi-user vault contract `V` that holds user deposits.
2. **Submit** an invoke transaction from the vault owner that calls `replace_class(class_hash=0xdeadbeef)` where `0xdeadbeef` is never declared.
3. The OS executes `execute_replace_class`: no validation fires; `V`'s `StateEntry.class_hash` is updated to `0xdeadbeef` and the state diff is committed.
4. **In any subsequent block**, a user submits a `withdraw` call to `V`. The OS must resolve the compiled class for `0xdeadbeef`. No compiled class fact exists; the prover cannot satisfy `validate_compiled_class_facts_post_execution`. The transaction is unprovable and reverts.
5. All user funds inside `V` are permanently inaccessible — no withdrawal, no migration, no recovery path exists at the protocol level.

        // Not enough gas to execute the syscall.
        return ();
