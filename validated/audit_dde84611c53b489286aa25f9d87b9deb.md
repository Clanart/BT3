Audit Report

## Title
Stranded native ETH on router is silently consumed by any subsequent WETH swap ŌĆö (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` helper in `PeripheryPayments` uses `address(this).balance` ŌĆö the router's total native ETH balance ŌĆö when settling a WETH leg, with no attribution to the current caller's `msg.value`. ETH left on the router after a partial-fill `exactInputSingle` (price-limit hit) or after a multicall that omits `refundETH` is indistinguishable from any subsequent caller's ETH, and will be silently consumed by the next WETH swap, causing the original depositor to lose principal with no on-chain recourse.

## Finding Description

**Root cause ŌĆö `pay` WETH branch:**

`pay` reads `address(this).balance` as the available native ETH for any WETH payment: [1](#0-0) 

This is the aggregate router balance, not the ETH contributed by the current call. Any ETH left from a prior transaction is spent first, before pulling WETH from the payer.

**How ETH becomes stranded ŌĆö path 1 (partial fill in `exactInputSingle`):**

`exactInputSingle` passes `priceLimitX64` to the pool and, when the price limit is hit, the pool executes a partial fill and calls `metricOmmSwapCallback` with the actual (smaller) deltas. The callback pays only the actual delta: [2](#0-1) 

`exactInputSingle` has **no check** that `amountInActual == params.amountIn`. It only checks the output minimum: [3](#0-2) 

Contrast with `exactInput`, which reverts if the actual input is less than requested: [4](#0-3) 

The transaction succeeds, the excess ETH (`params.amountIn ŌłÆ actualAmountIn`) stays on the router, and no revert returns it to the caller.

**How ETH becomes stranded ŌĆö path 2 (excess `msg.value` in multicall):**

The standard multicall-with-ETH pattern relies on `refundETH` in the same batch. If a user omits it, surplus ETH is permanently stranded. `refundETH` sends to `msg.sender`: [5](#0-4) 

The victim cannot reclaim it after the fact without a race condition.

**Exploit ŌĆö attacker consumes stranded ETH:**

Any subsequent caller who calls `exactInputSingle` with `tokenIn = WETH` and `amountIn Ōēż router.balance` (sending zero `msg.value`) will have `pay` see `address(this).balance >= value`, deposit the victim's ETH as WETH, and transfer it to the pool on the attacker's behalf. The attacker receives full swap output at zero cost.

## Impact Explanation

Direct loss of user principal. The victim's stranded ETH is deposited as WETH and transferred to a pool on behalf of an attacker's swap. The attacker receives the full swap output without spending any of their own ETH. The victim receives nothing in return for the consumed ETH. This satisfies the Critical/High/Medium direct loss of user principal gate.

## Likelihood Explanation

Medium. Two realistic triggers exist: (1) any user who calls `exactInputSingle{value: X}` with a non-trivial `priceLimitX64` and receives a partial fill leaves `X ŌłÆ actual` ETH on the router ŌĆö the pool's oracle-anchored bin model makes partial fills routine when the price limit is tighter than the current spread; (2) any user who follows the standard multicall-with-ETH pattern but omits `refundETH` leaves excess ETH on the router. The attacker requires no special privilege: a single `exactInputSingle` call with `tokenIn = WETH` and `amountIn Ōēż router.balance` suffices.

## Recommendation

**Short term:** Snapshot `address(this).balance` before each payable entry point and cap native ETH usage in `pay` to that snapshot minus any ETH already spent in the current call sequence. Alternatively, track a `_nativeDeposited` transient variable that is incremented by `msg.value` at entry and decremented as ETH is consumed, and use that as the cap.

**Long term:** Enforce that `exactInputSingle` reverts when the actual input amount is less than `params.amountIn` (matching the guard already present in `exactInput` at line 115), so partial fills always revert and return ETH to the caller rather than leaving it stranded.

## Proof of Concept

```
Setup:
  - Pool: WETH / token1, oracle bid/ask set so that a price limit of P
    causes a partial fill of 60% of any given amountIn.

Step 1 ŌĆö Victim strands ETH:
  Alice calls:
    router.exactInputSingle{value: 1000}(
      pool, WETH, token1, zeroForOne=true,
      amountIn=1000, amountOutMinimum=0,
      priceLimitX64=P,   // causes 40% partial fill
      ...
    )
  Pool executes 600 units, _justPayCallback pays 600 from address(this).balance.
  Transaction succeeds (amountOut >= 0 = amountOutMinimum).
  Router now holds 400 wei of ETH.
  Alice has no on-chain mechanism to reclaim it atomically.

Step 2 ŌĆö Attacker steals:
  Bob calls (no msg.value):
    router.exactInputSingle(
      pool, WETH, token1, zeroForOne=true,
      amountIn=400, amountOutMinimum=0,
      priceLimitX64=0,   // open limit, full fill
      ...
    )
  pay() sees address(this).balance = 400 >= 400.
  Deposits Alice's 400 wei as WETH, transfers to pool.
  Bob receives token1 output. Alice's 400 wei is gone.

Result:
  Alice loses 400 wei of ETH (40% of her deposit).
  Bob gains a full swap at zero cost.
```
