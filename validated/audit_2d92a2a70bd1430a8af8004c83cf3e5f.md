### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by a contract is actually declared on-chain before committing the state update. This is directly analogous to the external report's `transferFrom` vs `safeTransferFrom` issue: just as `transferFrom` blindly sends a token to a receiver without checking it can handle it, `execute_replace_class` blindly writes any class hash to a contract's state without checking it exists. The result is that any contract can permanently brick itself (and freeze all funds it holds) by replacing its class with an undeclared hash — and the OS will accept and commit this state transition.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), after deducting gas, the OS reads `request.class_hash` and immediately writes it into `contract_state_changes` via `dict_update` with no check that the class hash is present in `contract_class_changes` (the declared class registry):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 is an explicit developer acknowledgment that this validation is missing. The OS commits the new class hash into the contract state tree (via `compute_contract_state_commitment` in `state/commitment.cairo`) without any cross-reference to `contract_class_changes`. Once committed on-chain, any subsequent transaction that calls the bricked contract will fail at the class-lookup stage (the prover cannot supply a class that was never declared), making the contract permanently uncallable and all funds it holds permanently frozen.

The syscall is reachable by any unprivileged contract via the `REPLACE_CLASS_SELECTOR` dispatch path in `execute_syscalls.cairo` (lines 195–203):

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
    ...
}
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value and the block is proven and finalized:

- The contract state tree leaf is updated to reference a class hash that has no corresponding entry in the class commitment tree.
- Every future call to that contract requires the OS to resolve the class hash to a compiled class. Since the class is not declared, no valid proof can be generated for any transaction touching that contract.
- All token balances, ETH, STRK, or any other assets held in the contract's storage become permanently inaccessible — there is no recovery path at the protocol level.

This matches the external report's impact: just as an ERC721 token sent to an unprepared contract is permanently locked, funds in a contract whose class is replaced with an undeclared hash are permanently frozen.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is available to any deployed contract. A malicious actor can:

1. Deploy a contract that holds user funds (e.g., a fake vault or pool).
2. Attract deposits.
3. Call `replace_class` with an arbitrary undeclared felt value (e.g., `0xdeadbeef`).
4. The OS accepts the call, the block is proven, and the funds are permanently frozen.

Additionally, legitimate contracts with upgrade mechanisms that do not independently validate the new class hash (e.g., DAO-governed upgrades) are vulnerable to governance attacks where a malicious proposal supplies an undeclared class hash. The missing OS-level guard means there is no safety net even if the contract-level logic is flawed.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, add a lookup into `contract_class_changes` to assert the class hash is declared (i.e., its compiled class hash is non-zero):

```cairo
let class_hash = request.class_hash;

// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}

local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
...
```

This mirrors the `safeTransferFrom` fix in the external report: validate that the target (class) can handle the operation before committing the state change.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts deposits of STRK/ETH from users.
2. Users deposit funds; `MaliciousVault` accumulates balances in its storage.
3. Attacker calls an internal function on `MaliciousVault` that issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary undeclared felt).
4. `execute_replace_class` in `syscall_impls.cairo` (lines 896–910) accepts the call without validation and writes `class_hash=0xdeadbeef` into `contract_state_changes`.
5. The block is proven. `compute_contract_state_commitment` in `state/commitment.cairo` (lines 76–111) commits the new contract state leaf — which now references the undeclared class hash — into the state tree.
6. In any subsequent block, any transaction attempting to call `MaliciousVault` causes the OS to look up class `0xdeadbeef` in `contract_class_changes`. The class does not exist; the prover cannot supply it; no valid proof can be generated.
7. All user funds stored in `MaliciousVault`'s storage are permanently frozen with no recovery path.
