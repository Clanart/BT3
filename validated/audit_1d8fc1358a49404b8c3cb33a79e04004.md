### Title
`eth_estimateGas` / `eth_call` silently succeed for transactions that will fail with `ErrStateConflict` during real block execution — (`x/evm/keeper/state_transition.go`)

### Summary
`eth_estimateGas` and `eth_call` call `ApplyMessageWithConfig` with `commit=false`, which skips `stateDB.Commit()` entirely. The `ErrStateConflict` detection lives exclusively inside `Commit()`. Any transaction that triggers a storage-slot conflict between EVM-dirty state and a nested native action will appear to succeed in simulation but fail with `status=0` during real execution, consuming the user's full gas fees for a no-op transaction.

### Finding Description

`ApplyMessageWithConfig` in `x/evm/keeper/state_transition.go` gates the entire `stateDB.Commit()` call — including `ErrStateConflict` detection — behind the `commit` flag:

```go
if commit {
    if err := stateDB.Commit(); err != nil {
        if errors.Is(err, statedb.ErrStateConflict) {
            return &types.EVMResult{
                GasUsed: gasUsed,
                VmError: statedb.ErrStateConflict.Error(),
                ...
            }, nil
        }
        return nil, errorsmod.Wrap(err, "failed to commit stateDB")
    }
}
```

The `EstimateGas` gRPC handler passes `commit=false`:

```go
rsp, err = k.ApplyMessageWithConfig(ctx, msg, cfg, false)
```

The code itself documents the gap explicitly:

```
// Note: estimateGas and eth_call do not hit this path because commit is
// false for simulations, so they will succeed even when a real execution
// would produce a state conflict.
```

`ErrStateConflict` is raised in `statedb.Commit()` when a storage slot is written by both the EVM (dirty storage) and a nested native action via `ExecuteNativeAction`. The conflict check reads the live store value and compares it against both `origin` and `dirty`:

```go
store := s.keeper.GetState(s.ctx, obj.Address(), key)
if store != origin && store != dirty {
    return fmt.Errorf("%w: ...", ErrStateConflict, ...)
}
```

Because `Commit()` is never called during simulation, this check is completely invisible to `eth_estimateGas` and `eth_call`. The simulation returns a successful result with a valid gas estimate, while real execution will return `VmError = "state conflict"` and `status=0`.

### Impact Explanation

Any chain that registers a custom precompile (via `customContractFns`) that writes to a contract's storage slot — a supported and documented pattern — can produce transactions where:

1. `eth_estimateGas` returns a valid gas estimate (simulation succeeds, no conflict detected)
2. The user submits the transaction with that estimate
3. Real execution detects `ErrStateConflict` in `Commit()` and returns `VmError = "state conflict"` (status=0)
4. The transaction is included in the block with status=0, consuming the user's full gas fees

This is a mis-accounting of user funds: gas fees are charged for a transaction that produces no committed state change. It maps to the allowed impact: **"EVM state transition... bug that permits... valid user funds/fees to be mis-accounted."**

Additionally, `eth_call` returning success when real execution would fail misleads callers about the outcome of their transaction, mapping to: **"Public JSON-RPC... path feeds incorrect consensus-critical data into transaction execution."**

### Likelihood Explanation

The scenario requires a deployed custom precompile that writes to a contract's storage slot. This is a supported feature of the keeper (`customContractFns`). Any chain using such precompiles is affected. The user's entry path is a standard `eth_estimateGas` JSON-RPC call — no special privileges required on the user side. The precompile deployment is a normal chain-operator action, not an attack prerequisite.

### Recommendation

Run `stateDB.Commit()` in a dry-run mode during simulation to detect `ErrStateConflict` before returning the gas estimate. Alternatively, surface the conflict as a simulation failure so callers receive an accurate result. At minimum, document this divergence in the JSON-RPC API so integrators are aware that `eth_estimateGas` success does not guarantee execution success for transactions involving native-action precompiles.

### Proof of Concept

1. Chain operators register a custom precompile `P` that, when called, writes value `V` to storage slot `K` of contract `C` via `ExecuteNativeAction`.
2. Contract `C` also writes to slot `K` in the same call frame (e.g., as part of its normal logic).
3. User calls `eth_estimateGas` for a transaction invoking `C` → simulation returns success with a gas estimate (no `Commit()` called, no conflict detected).
4. User submits the transaction with the estimated gas.
5. During block execution, `stateDB.Commit()` detects `store != origin && store != dirty` for slot `K` and returns `ErrStateConflict`.
6. `ApplyMessageWithConfig` returns `EVMResult{VmError: "state conflict"}` (status=0).
7. Transaction is included in the block with status=0; user's gas fees are fully consumed. [1](#0-0) [2](#0-1) [3](#0-2)
