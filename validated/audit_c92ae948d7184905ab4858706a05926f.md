### Title
Rejected L1 Handler Transactions Silently Treated as Committed During Catch-Up, Permanently Removing Re-Proposable Transactions - (`crates/apollo_l1_provider/src/l1_provider.rs`)

### Summary

During the L1 provider's catch-up phase, the `rejected_txs` parameter passed to `commit_block` is silently dropped. Rejected L1 handler transactions — those that failed execution and should remain in `Pending` state for re-proposal — are instead marked as `Committed`, permanently removing them from the proposable set. This is the sequencer analog of the external bug where a function always returns `true` regardless of actual outcome: here, every L1 handler transaction included in a block during catch-up is unconditionally treated as successfully committed, even if it was rejected.

### Finding Description

`L1Provider::commit_block` accepts both `committed_txs` and `rejected_txs`:

``` [1](#0-0) 

When the provider is in `CatchingUp` state, it immediately forwards to `accept_commit_while_catching_up`, **dropping `rejected_txs` entirely**:

```rust
if self.state.is_catching_up() {
    return self.accept_commit_while_catching_up(committed_txs, height);

``` [2](#0-1) 

`accept_commit_while_catching_up` only accepts `committed_txs`:

```rust

    &mut self,
    committed_txs: IndexSet<TransactionHash>,
    new_height: BlockNumber,

``` [3](#0-2) 

It then calls `apply_commit_block` with an empty `Default::default()` for `rejected_txs` — both for the current height and for every backlogged block:

```rust
Equal => self.apply_commit_block(committed_txs, Default::default()),
// ...
for committed_block in backlog {
    self.apply_commit_block(committed_block.committed_txs, Default::default());
}
``` [4](#0-3) 

The `CommitBlockBacklog` struct also has no field for `rejected_txs`, so the information is structurally unrecoverable once backlogged: [5](#0-4) 

In `apply_commit_block`, the partition that separates rejected-and-consumed from committed transactions uses the `rejected_txs` set:

```rust
let (rejected_and_consumed, committed_txs): (Vec<_>, Vec<_>) =
    consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));
self.tx_manager.commit_txs(&committed_txs, &rejected_and_consumed);
``` [6](#0-5) 

With an empty `rejected_txs`, every consumed L1 handler transaction — including those that failed execution — is passed to `commit_txs` as committed. The transaction manager then transitions them from `Pending` to `Committed`, making them ineligible for future re-proposal.

The batcher correctly computes and passes `rejected_l1_handler_tx_hashes` to the L1 provider:

```rust
let rejected_l1_handler_tx_hashes = rejected_tx_hashes
    .iter()
    .copied()
    .filter(|tx_hash| consumed_l1_handler_tx_hashes.contains(tx_hash))
    .collect();
self.l1_provider_client
    .commit_block(consumed_l1_handler_tx_hashes, rejected_l1_handler_tx_hashes, height)
    .await;
``` [7](#0-6) 

The information is available at the call site but is structurally discarded by the catch-up path. A `TODO` comment in the code acknowledges this gap:

### Impact Explanation

Rejected L1 handler transactions are those that failed execution (returned `Err` from the executor, placed in `rejected_tx_hashes`). Per the protocol, they should remain in `Pending` state and be re-proposed in a subsequent block. During catch-up — which occurs at node startup or whenever the provider falls behind the batcher — any such rejected L1 handler transactions are permanently marked `Committed`. They are removed from the proposable index and will never be re-proposed. The corresponding L1 messages are effectively silenced on L2 without successful execution, breaking the L1→L2 message delivery guarantee.

This matches the allowed impact: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.** Valid L1 handler transactions that should be retried are permanently excluded from future block proposals.

### Likelihood Explanation

Catch-up is triggered at every node startup when the provider's height lags behind the batcher, and also during any restart or resync scenario. Any block during the catch-up window that contains a rejected L1 handler transaction will trigger the incorrect state transition. The window is bounded by the catch-up duration, but in practice every node restart traverses this path.

### Recommendation

Pass `rejected_txs` through the entire catch-up path:

1. Add `rejected_txs: IndexSet<TransactionHash>` to `accept_commit_while_catching_up`'s signature and forward it from `commit_block`.
2. Add a `rejected_txs` field to `CommitBlockBacklog` and populate it in `add_commit_block_to_backlog`.
3. Pass the stored `rejected_txs` when draining the backlog in `apply_commit_block` calls.

### Proof of Concept

1. Start a sequencer node from scratch (catch-up is triggered at startup).
2. Arrange for an L1 handler transaction to be included in a block during the catch-up window but have its execution fail (e.g., the target contract reverts).
3. The batcher calls `L1Provider::commit_block(consumed={tx_hash}, rejected={tx_hash}, height)`.
4. Because `state.is_catching_up()` is true, `accept_commit_while_catching_up(committed_txs={tx_hash}, height)` is called; `rejected_txs` is dropped.
5. `apply_commit_block({tx_hash}, {})` is called; the partition produces `rejected_and_consumed=[]`, `committed_txs=[tx_hash]`.
6. `tx_manager.commit_txs([tx_hash], [])` marks the transaction as `Committed`.
7. The transaction is removed from the proposable index and never re-proposed, despite having failed execution.

        committed_txs: IndexSet<TransactionHash>,

    ) {
        assert!(
            self.commit_block_backlog
                .last()
                .is_none_or(|commit_block| commit_block.height.unchecked_next() == height),
            "Heights should be sequential."
        );

        debug!("Adding future commit-block to backlog at height: {height}");
        self.commit_block_backlog
            .push(CommitBlockBacklog { height, committed_txs: committed_txs.clone() });
