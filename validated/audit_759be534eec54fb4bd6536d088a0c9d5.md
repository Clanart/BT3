### Title
StarkNet `fin_transfer` silently drops the `message` field, making message-dependent cross-chain operations impossible on StarkNet — (`starknet/src/omni_bridge.cairo`)

---

### Summary

The StarkNet `OmniBridge.fin_transfer` implementation receives a `TransferMessagePayload` that includes a `message` field, but never forwards it to the bridge token's `mint` call. The EVM counterpart explicitly forwards the message to enable callback-driven DeFi interactions on the destination chain. On StarkNet, any transfer carrying a non-empty `message` silently discards it, making it impossible to trigger message-dependent actions on StarkNet — an exact structural analog to the reported missing `liquidateOnHost` endpoint.

---

### Finding Description

In `starknet/src/omni_bridge.cairo`, the `fin_transfer` function mints bridge tokens without forwarding the `message` field:

```cairo
// starknet/src/omni_bridge.cairo  fin_transfer (lines ~256-263)
if self.is_bridge_token(payload.token_address) {
    IBridgeTokenDispatcher { contract_address: payload.token_address }
        .mint(payload.recipient, payload.amount.into());   // message silently dropped
} else {
    let success = IERC20Dispatcher { contract_address: payload.token_address }
        .transfer(payload.recipient, payload.amount.into());
    assert(success, 'ERR_TRANSFER_FAILED');
}
```

The `message` field is present in `TransferMessagePayload` and is even emitted in the `FinTransfer` event, confirming it is part of the intended protocol payload:

```cairo
// starknet/src/omni_bridge.cairo  FinTransfer event emission (lines ~265-278)
Event::FinTransfer(FinTransfer {
    ...
    message: payload.message,   // present in event but never acted upon
})
```

The `IBridgeToken` interface defined inside the same file only exposes a two-argument `mint`:

```cairo
// starknet/src/omni_bridge.cairo  IBridgeToken interface (lines ~413-417)
#[starknet::interface]
trait IBridgeToken<TContractState> {
    fn mint(ref self: TContractState, recipient: ContractAddress, amount: u256);
    fn burn(ref self: TContractState, account: ContractAddress, amount: u256);
}
```

Compare this to the EVM implementation, which explicitly branches on message presence and forwards it:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  finTransfer (lines ~337-349)
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient, payload.amount, payload.message   // forwarded
        );
    }
}
```

The asymmetry is complete:

| Chain | `init_transfer` accepts `message` | `fin_transfer` forwards `message` |
|---|---|---|
| EVM | Yes | Yes |
| StarkNet | Yes | **No** |
| NEAR | Yes | Yes (via `send_tokens` / `ft_transfer_call`) |

A user or protocol can embed a `message` in a transfer originating from any chain targeting StarkNet. The NEAR bridge signs and relays the full payload including the message. When `fin_transfer` is called on StarkNet, the message is verified as part of the Borsh-encoded signature check but then discarded before the token action, making it impossible to trigger any callback-driven operation on StarkNet.

---

### Impact Explanation

Any cross-chain use case that relies on the `message` field to drive a post-mint action on StarkNet is permanently broken. Concretely:

- A DeFi protocol on StarkNet that expects to receive bridged tokens via a `mint(recipient, amount, message)` callback (e.g., to auto-deposit into a vault, execute a swap, or repay a loan) will receive the tokens silently minted to its address with no callback. If the contract has no independent recovery path for unexpectedly received tokens, those funds are permanently frozen.
- This is not a theoretical edge case: the EVM bridge already supports this pattern, so protocols designed to work with the EVM bridge's message feature will be broken when the destination is StarkNet.

This matches the **Critical** allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

The `message` field is a first-class parameter of `init_transfer` on every chain including StarkNet itself. Any relayer or user who initiates a transfer to a StarkNet contract address with a non-empty `message` will trigger this silent discard. The likelihood is **Medium**: it requires a protocol integration that uses the message callback pattern, but such integrations are the primary motivation for the `message` field's existence.

---

### Recommendation

1. Extend the `IBridgeToken` interface in `starknet/src/omni_bridge.cairo` to include a three-argument `mint`:
   ```cairo
   fn mint(ref self: TContractState, recipient: ContractAddress, amount: u256, message: ByteArray);
   ```
2. In `fin_transfer`, branch on message presence and forward it, mirroring the EVM logic:
   ```cairo
   if payload.message.len() == 0 {
       IBridgeTokenDispatcher { contract_address: payload.token_address }
           .mint(payload.recipient, payload.amount.into());
   } else {
       IBridgeTokenDispatcher { contract_address: payload.token_address }
           .mint(payload.recipient, payload.amount.into(), payload.message.clone());
   }
   ```
3. Update `starknet/src/bridge_token.cairo` to implement the extended `mint` interface and forward the message to the recipient contract via a low-level call when the recipient is a contract.

### Proof of Concept

1. Deploy a StarkNet contract `VaultProxy` that implements `mint(recipient, amount, message)` and uses `message` to route tokens into a vault. Register it as a custom recipient for a bridged token.
2. From Ethereum, call `OmniBridge.initTransfer(token, amount, fee, nativeFee, "VaultProxy_address", encoded_deposit_instruction)`.
3. The NEAR bridge signs the full payload including `message = encoded_deposit_instruction`.
4. A relayer calls `OmniBridge.fin_transfer(signature, payload)` on StarkNet.
5. Observe: `IBridgeTokenDispatcher.mint(VaultProxy_address, amount)` is called — the two-argument variant. `VaultProxy` receives tokens but no deposit instruction. The tokens sit idle in `VaultProxy` with no recovery mechanism.
