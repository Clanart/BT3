### Title
`merge_storage_proofs` Reorders `contracts_storage_proofs` But Leaves `contract_leaves_data` in Chunk Order, Feeding Wrong Class Hashes and Nonces into OS Input - (`crates/starknet_transaction_prover/src/running/storage_proofs.rs`)

### Summary

When the transaction prover fetches Patricia storage proofs for a block that requires more keys than `MAX_KEYS_PER_REQUEST`, the query is split into chunks and the partial `RpcStorageProof` responses are merged. The merge function explicitly reconstructs `contracts_storage_proofs` in the original query order using an index map, but performs only a raw `extend` on `contract_leaves_data`. The function's own doc-comment claims both positional arrays are "reconstructed in the original query's order," but only one of them is. The other is left in chunk-emission order. Downstream consumers zip `contract_leaves_data` against `query.contract_addresses` by position, so any mismatch silently assigns the wrong class hash and nonce to each contract address, and the wrong storage root to each contract's commitment info.

### Finding Description

`merge_storage_proofs` in `crates/starknet_transaction_prover/src/running/storage_proofs.rs` builds an `addr_to_idx` map from `original_query.contract_storage_keys` and uses it to scatter each chunk's `contracts_storage_proofs` entries back into the correct slot:

```rust
for (csk, storage_proof) in
    chunk_query.contract_storage_keys.iter().zip(proof.contracts_storage_proofs)
{
    let idx = addr_to_idx
        .get(&csk.contract_address)
        .expect("chunk address not in original query");
    contracts_storage_proofs[*idx].extend(storage_proof);
}
``` [1](#0-0) 

`contract_leaves_data`, however, is simply appended in chunk order with no reordering:

```rust
contracts_proof.contract_leaves_data.extend(proof.contracts_proof.contract_leaves_data);
``` [2](#0-1) 

The function's own doc-comment explicitly states that *both* positional arrays are reconstructed in the original query's order:

> "Positional data (`contract_leaves_data`, `contracts_storage_proofs`) is reconstructed in the original query's order." [3](#0-2) 

After the merge, two consumers zip `contract_leaves_data` against `query.contract_addresses` by position:

**1. `get_storage_proofs` — class hash and nonce assignment:**
```rust
for (leaf, addr) in
    rpc_proof.contracts_proof.contract_leaves_data.iter().zip(&query.contract_addresses)
{
    extended_initial_reads.class_hashes.insert(*addr, ClassHash(leaf.class_hash));
    extended_initial_reads.nonces.insert(*addr, Nonce(leaf.nonce));
}
``` [4](#0-3) 

**2. `build_storage_commitment_infos` — storage root per contract:**
```rust
let addr_to_leaf: HashMap<ContractAddress, _> = query
    .contract_addresses
    .iter()
    .zip(&rpc_proof.contracts_proof.contract_leaves_data)
    .map(|(addr, leaf)| (*addr, leaf))
    .collect();
```
