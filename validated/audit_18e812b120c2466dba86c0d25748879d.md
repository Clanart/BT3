### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary class hash from the caller without verifying that the hash corresponds to a declared contract class. This allows any contract to permanently corrupt its own class hash entry in the state, making the contract permanently uncallable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall and updates `contract_state_changes` with the caller-supplied class hash — with no validation that the hash is declared:

```cairo
// execute_replace_class (syscall_impls.cairo, lines 878–916)
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment explicitly acknowledges the missing check. The `class_hash` field is taken directly from the syscall request (fully attacker-controlled) and written into the persistent state dictionary without any membership check against `contract_class_changes`.

When `execute_entry_point` is subsequently called for the affected contract, it reads the stored class hash and looks up the compiled class:

```cairo
// execute_entry_point (execute_entry_point.cairo, lines 154–166)
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);

let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the class hash is undeclared, `dict_read` returns `0` (the default for an uninitialized dict entry). `find_element` then searches for a compiled class with hash `0`. Since no such entry exists in the compiled class facts bundle, `find_element` fails — the OS cannot generate a valid proof for any block containing a call to the affected contract. The contract becomes permanently uncallable.

The revert log entry added by `execute_replace_class` only protects against intra-transaction reverts:

```cairo
assert [revert_log] = RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=state_entry.class_hash);
``` [3](#0-2) 

If the transaction containing the `replace_class` call succeeds (does not revert), the state change is committed permanently.

The `replace_class` syscall is available to all contracts with no access restriction, as seen in the syscall dispatch loop:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
    ...
}
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

An attacker deploys or controls a contract that holds user funds (e.g., a vault or token contract). By calling `replace_class` with any undeclared class hash (e.g., `1` or any random felt not in `contract_class_changes`), the attacker permanently corrupts the contract's class hash in the committed state. All subsequent calls to the contract cause the OS to fail proof generation (`find_element` cannot locate the compiled class). The sequencer is forced to exclude all transactions targeting the contract. Any funds held by the contract are permanently inaccessible — there is no recovery path because the state commitment is final and no valid transaction can execute against the contract.

---

### Likelihood Explanation

**High.** The `replace_class` syscall is callable by any contract with no privilege requirement. The attacker only needs to:
1. Deploy a contract (or control an existing one) that holds user funds.
2. Submit a transaction that calls `replace_class` with any felt value not corresponding to a declared class.
3. The transaction succeeds (no revert), the state is committed, and the contract is permanently frozen.

No special role, leaked key, or privileged access is required. The entry path is a standard unprivileged transaction sender.

---

### Recommendation

Implement the acknowledged TODO: before updating `contract_state_changes`, verify that the provided `class_hash` has a non-zero entry in `contract_class_changes` (i.e., it has been declared):

```cairo
// In execute_replace_class, after reading class_hash from request:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed implicitly in `execute_entry_point` and makes the OS self-contained with respect to class hash validity.

---

### Proof of Concept

1. Attacker deploys `VaultContract` (holds user ETH/token deposits). Users deposit `100e18` tokens.
2. Attacker submits a transaction calling `VaultContract::freeze()`, which internally issues the syscall `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. The OS processes `execute_replace_class`: reads `class_hash = 0xdeadbeef` from the request, skips validation (TODO), and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`. The transaction succeeds.
4. The state commitment is computed with `VaultContract`'s class hash set to `0xdeadbeef`.
5. Any subsequent transaction calling `VaultContract` reaches `execute_entry_point`, which calls `dict_read(key=0xdeadbeef)` on `contract_class_changes` → returns `0`. Then `find_element(..., key=0)` fails — no compiled class with hash `0` exists. The OS cannot generate a proof.
6. The sequencer cannot include any transaction targeting `VaultContract`. All `100e18` tokens are permanently frozen.
