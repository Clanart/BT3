### Title
Unchecked Constructor Revert in `deploy` Syscall Causes OS Panic and Network Halt - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deploy_contract.cairo`)

---

### Summary

The `deploy_contract` function unconditionally asserts `is_reverted = 0` after invoking a constructor. When called from the new (non-deprecated) `execute_deploy` syscall, a reverting constructor causes the entire OS Cairo execution to panic. Because the blockifier (sequencer) handles constructor reverts gracefully — returning a failure response to the caller contract — a discrepancy exists: the sequencer can include a block containing a transaction where a deploy syscall's constructor reverts, but the OS cannot prove that block, causing a permanent network halt.

---

### Finding Description

In `deploy_contract.cairo`, after invoking the constructor via `select_execute_entry_point_func`, the return value `is_reverted` is unconditionally asserted to be zero:

```cairo
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

This function is called from `execute_deploy` in `syscall_impls.cairo`, which is the **new** (non-deprecated) `deploy` syscall handler. The `execute_deploy` function also hardcodes `failure_flag=0` in the response header, with an explicit TODO acknowledging that failure support is missing:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [2](#0-1) 

The call chain is:

```
execute_deploy (syscall_impls.cairo:452)
  └─> deploy_contract (deploy_contract.cairo:29)
        └─> select_execute_entry_point_func → is_reverted=1 (constructor reverts)
              └─> assert is_reverted = 0  ← PANIC
``` [3](#0-2) 

The blockifier (Rust-based sequencer simulation) handles constructor reverts gracefully by returning a failure response to the calling contract. A caller contract can catch this failure and allow the outer transaction to succeed. The sequencer then includes this transaction in a block. When the prover runs the OS Cairo program against this block, the `assert is_reverted = 0` fires, the proof fails, and the block can never be finalized.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

If an attacker crafts a transaction that the blockifier accepts (outer transaction succeeds, inner deploy constructor reverts and is caught by the caller) but the OS cannot prove, the sequencer will commit a block that is permanently unprovable. No subsequent blocks can be finalized until the issue is resolved at the protocol level, constituting a total network halt.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. Declaring a contract class whose constructor always reverts — permissionless on StarkNet.
2. Deploying a "wrapper" contract that calls the `deploy` syscall and handles the failure response — permissionless.
3. Submitting an invoke transaction to the wrapper — permissionless.

No privileged access, leaked keys, or third-party compromise is required. The only uncertainty is whether the blockifier's current simulation of the `deploy` syscall returns a failure response to the caller (allowing the outer transaction to succeed) or also panics. Given that the blockifier is a separate Rust implementation designed to handle reverts gracefully, and given the explicit TODO in the OS code acknowledging missing failure support, the discrepancy is realistic.

---

### Recommendation

In `deploy_contract.cairo`, remove the unconditional `assert is_reverted = 0` and instead propagate the `is_reverted` flag back to `execute_deploy`. In `execute_deploy` (`syscall_impls.cairo`), set `failure_flag=is_reverted` in the `ResponseHeader` and handle the revert log rollback correctly, consistent with how `contract_call_helper` handles reverts for `call_contract`. [4](#0-3) 

---

### Proof of Concept

1. **Declare** a contract class `RevertingConstructor` whose `constructor` entry point always panics/reverts.
2. **Deploy** a `Wrapper` contract with an `__execute__` that calls:
   ```
   deploy(class_hash=RevertingConstructor, ...)  // constructor will revert
   // catch the failure_flag=1 in the response and return normally
   ```
3. **Submit** an invoke transaction targeting `Wrapper.__execute__`.
4. **Blockifier simulation**: `deploy` syscall returns `failure_flag=1` to `Wrapper`; `Wrapper` returns normally; outer transaction is marked as **succeeded** and included in the block.
5. **OS proof attempt**: `deploy_contract` calls `select_execute_entry_point_func`, receives `is_reverted=1`, hits `assert is_reverted = 0` at `deploy_contract.cairo:91` → Cairo assertion failure → proof is invalid.
6. **Result**: The committed block is permanently unprovable. Network halts. [5](#0-4) [6](#0-5)
