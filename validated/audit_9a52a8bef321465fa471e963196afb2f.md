Looking at the code carefully, I need to trace the full validation path in `set_state_header`, the `verify_path` semantics, and whether the receipt proof check constitutes a real guard.

**Key code facts:**

`set_state_header` performs these checks: [1](#0-0) 

1. `validate_chunk_proofs` — checks internal chunk consistency (hash, tx root, receipts root). **No `chunk.shard_id() == shard_id` check.**
2. `verify_path` against `chunk_headers_root` — checks the chunk is *somewhere* in the block's Merkle tree.
3. Receipt proof loop — computes `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the **caller's** `shard_id`.
4. `validate_state_root_node` — checks the node is consistent with `chunk.prev_state_root()`.
5. Stores under `StateHeaderKey(shard_id, sync_hash)` — using the **caller's** `shard_id`. [2](#0-1) 

**`verify_path` is position-agnostic:** [3](#0-2) 

`verify_path` only checks that `compute_root_from_path(path, hash) == root`. It does **not** verify the leaf's index/position in the tree. A valid Merkle proof for shard 0's chunk at position 0 passes `verify_path` even when the caller claims `shard_id=1`.

**The receipt proof check is a partial guard, not a complete one:** [4](#0-3) 

The check at line 488 uses the caller's `shard_id` to compute `receipts_hash`. If the attacker submits shard 0's complete header for shard 1, the receipt proofs (computed for `ReceiptList(0, receipts)`) would fail this check. **However**, the attacker can craft a mixed header: shard 0's chunk + shard 1's receipt proofs. Since the receipt proofs are validated against `block_header.prev_chunk_outgoing_receipts_root()` (a block-level root, not shard-specific), and the hash is computed with the caller's `shard_id=1`, shard 1's receipt proofs pass all checks.

**The check at line 506-510** requires the receipt proof chain to end at `prev_chunk_header.height_included()`. In a healthy network where all shards produce chunks at every block, shard 0 and shard 1 have the same `prev_chunk.height_included`, so this check passes with the mixed header.

**`set_state_part` then uses the wrong `state_root`:** [5](#0-4) 

It retrieves the stored header for `shard_id=1`, extracts `state_root` from shard 0's chunk's `prev_state_root()`, and validates parts against it. Shard 0's parts pass. They are stored under `StatePartKey(sync_hash, 1, part_id)`.

**The attack path from peer to `set_state_header` is confirmed:** [6](#0-5) [7](#0-6) 

The `shard_id` in `StateHeaderValidationRequest` comes from the download request, not from the header itself. A malicious peer responding to a shard 1 request with a crafted header causes `set_state_header(shard_id=1, sync_hash, crafted_header)` to be called.

**However**, the attack requires the attacker to craft a *mixed* `ShardStateSyncResponseHeader` — combining shard 0's chunk/prev_chunk/state_root_node with shard 1's receipt proofs and root proofs — and requires that both shards have the same `prev_chunk.height_included`. While technically feasible (all data is publicly available on-chain, any node can be a peer, and the struct is freely constructable), this is a non-trivial multi-component forgery, not a simple replay.

The audit boundary requires the "exact corrupted value" to be concrete. Here it is: `state_root = chunk_0.prev_state_root()` (a specific on-chain `CryptoHash`) is used to validate and install state under `shard_uid(1)`. That is concrete.

### Title
Missing `chunk.shard_id()` binding in `set_state_header` allows cross-shard state installation — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` stores a peer-supplied `ShardStateSyncResponseHeader` under `StateHeaderKey(caller_shard_id, sync_hash)` without asserting that the embedded chunk's `shard_id` equals the caller's `shard_id`. A malicious peer can craft a mixed header (shard A's chunk + shard B's receipt proofs) that passes all existing checks, causing `set_state_part` and `apply_state_part` to validate and install shard A's state under shard B's `ShardUId`.

### Finding Description
In `chain/chain/src/state_sync/adapter.rs`, `set_state_header` (lines 368–531) performs five validation steps before persisting the header:

1. `validate_chunk_proofs` — internal chunk consistency only; no shard_id binding.
2. `verify_path` against `sync_prev_block_header.chunk_headers_root()` — position-agnostic membership proof; shard 0's chunk with its own valid proof passes even when `shard_id=1` is claimed.
3. Receipt proof loop — uses caller's `shard_id` for `ReceiptList(shard_id, receipts)` hash, but an attacker can supply shard B's receipt proofs (which were computed for `ReceiptList(B, receipts)`) alongside shard A's chunk.
4. `validate_state_root_node` — checks consistency with `chunk.prev_state_root()`; passes with shard A's node.
5. Stores under `StateHeaderKey(shard_id, sync_hash)` — using caller's `shard_id`, not the chunk's.

The missing guard is a single equality check: `chunk.shard_id() == shard_id`.

Downstream, `set_state_part` (lines 534–560) retrieves the stored header for `shard_id=1`, extracts `state_root = chunk_0.prev_state_root()`, and validates parts against it. Shard A's parts pass. `apply_state_part` then installs shard A's trie data under shard B's `ShardUId`. [8](#0-7) 

### Impact Explanation
The exact corrupted value is `state_root = shard_A.chunk.prev_state_root()` (a specific on-chain `CryptoHash`) used to validate and install state under `shard_uid(B)`. The resulting trie is cryptographically valid for shard A but installed as shard B's state. After `set_state_finalize`, the node operates with shard B's account/contract state replaced by shard A's, producing wrong execution results, wrong state roots in produced chunks, and potential slashing or fork. [9](#0-8) 

### Likelihood Explanation
The attacker must be a network peer (any node qualifies), must have access to on-chain chunk and receipt proof data (publicly available), and must craft a mixed `ShardStateSyncResponseHeader`. The crafting is non-trivial but requires no privileged access. The condition that both shards share the same `prev_chunk.height_included` holds in any healthy multi-shard network where all shards produce chunks at every block.

### Recommendation
Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [10](#0-9) 

### Proof of Concept
1. In a 2-shard testnet, obtain `header_0 = get_state_response_header(shard_id=0, sync_hash)` and `header_1 = get_state_response_header(shard_id=1, sync_hash)`.
2. Craft `mixed_header = ShardStateSyncResponseHeader { chunk: header_0.chunk, chunk_proof: header_0.chunk_proof, prev_chunk_header: header_0.prev_chunk_header, prev_chunk_proof: header_0.prev_chunk_proof, incoming_receipts_proofs: header_1.incoming_receipts_proofs, root_proofs: header_1.root_proofs, state_root_node: header_0.state_root_node }`.
3. Call `set_state_header(shard_id=1, sync_hash, mixed_header)` — all checks pass.
4. Call `set_state_part(shard_id=1, sync_hash, part_id, shard_0_part)` for each part of shard 0 — validation uses `shard_0.prev_state_root`, parts pass.
5. Call `apply_state_part(shard_id=1, shard_0.prev_state_root, ...)` — shard 0's trie data is installed under `shard_uid(1)`.
6. Assert `shard_uid(1)`'s trie contains shard 0's accounts/contracts.
