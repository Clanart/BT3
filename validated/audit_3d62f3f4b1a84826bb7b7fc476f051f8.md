### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds â€” (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt value as a new class hash without verifying it corresponds to a previously declared contract class. This is an explicit, acknowledged missing data-validation step. Any contract â€” deployed by an unprivileged user â€” that calls `replace_class` with an undeclared class hash will have its on-chain class hash permanently set to an invalid value, making the contract permanently unexecutable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` processes the `replace_class` syscall:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `class_hash` field is taken directly from the caller-supplied `ReplaceClassRequest` and written into `contract_state_changes` with no check that `class_hash` exists in the set of declared classes (`contract_class_changes`). The TODO comment at line 898 is an explicit developer acknowledgment that this validation is absent.

The declared-class registry is maintained separately in `contract_class_changes` (a dict from class hash â†’ compiled class hash). The OS validates compiled class facts post-execution in `validate_compiled_class_facts_post_execution` (called from `os.cairo`). If a contract's class hash is set to a value that was never declared, the prover cannot supply a valid compiled class entry for it, so every future call to that contract will be unprovable and will fail â€” but the state mutation (the invalid class hash) is already committed to the state trie.

---

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

Once `contract_state_changes` is squashed and committed via `state_update` in `os.cairo` (line 230), the invalid class hash becomes the canonical on-chain state for that contract address. Because:

1. The contract's class hash is now an undeclared value.
2. Any future invocation requires the prover to supply a compiled class for that hash â€” which does not exist.
3. There is no recovery path: the contract cannot execute `replace_class` again (it cannot execute anything), and there is no protocol-level admin override.

All ERC-20 balances, ETH, or other assets held in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract from any external entry point. No privileged role is required. Realistic triggering paths include:

- A contract implementing an upgrade mechanism that passes an incorrect (e.g., not-yet-declared or mistyped) class hash â€” a common pattern in upgradeable contracts.
- An attacker who controls a contract deliberately calling `replace_class(0x1)` to self-freeze after draining funds from a shared pool, preventing post-hoc forensic recovery.
- A reentrancy scenario where an attacker's contract, called mid-execution by a victim contract, calls `replace_class` on itself with an invalid hash to lock state before the outer call reverts (exploiting the revert log to selectively undo storage writes but not the class hash change, depending on revert semantics).

The missing check is unconditional â€” every call to `replace_class` with an undeclared hash succeeds at the OS level.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the hash exists as a key in `contract_class_changes` (i.e., it has a non-zero compiled class hash entry). Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the returned value is non-zero, mirroring the pattern used in `execute_declare_transaction` where `assert_not_zero(compiled_class_hash)` is enforced before updating `contract_class_changes`.

### Proof of Concept

1. Attacker deploys contract `C` holding user funds (e.g., acts as a simple vault).
2. Attacker sends an `invoke` transaction calling an entry point on `C` that internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (a felt never passed through `execute_declare_transaction`).
3. `execute_replace_class` in `syscall_impls.cairo` (lines 878â€“916) accepts the request, skips the missing declared-class check, and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
4. `state_update` in `os.cairo` (line 230) squashes and commits this entry to the state trie.
5. Any subsequent call to `C` requires the prover to supply a compiled class for `0xdeadbeef`. No such class exists. The transaction is unprovable and fails.
