### Title
Unbounded EVM Gas in gRPC `EthCall`/`TraceCall` Queries When `queryMaxGasLimit` Is Unset - (File: x/evm/keeper/grpc_query.go)

### Summary
When the keeper's `queryMaxGasLimit` field is left at its default value of `GasNoLimit` (0), a caller who sends a gRPC `EthCall` or `TraceCall` request with `GasCap = 0` causes `args.ToMessage(0, baseFee)` to be invoked with no gas cap. Inside `ToMessage`, a zero `globalGasCap` causes the message gas limit to be set to `math.MaxUint64 / 2`, allowing the EVM to execute contract bytecode with effectively unbounded gas. An attacker can exploit this to make a validator node hang indefinitely while processing a single crafted query, threatening chain liveness.

### Finding Description

`GasNoLimit` is defined as `0` and is the zero-value default for `queryMaxGasLimit`: [1](#0-0) [2](#0-1) 

In `EthCall`, the gas-cap enforcement block is guarded by `k.queryMaxGasLimit != GasNoLimit`. When `queryMaxGasLimit` is 0 (the default), the entire enforcement block is skipped and `gasCap` is passed through unchanged from `req.GasCap`: [3](#0-2) 

If the caller sends `req.GasCap = 0` (the protobuf default when the field is omitted), `gasCap` remains 0 and is forwarded to `args.ToMessage(0, cfg.BaseFee)`. Inside `ToMessage` (in `x/evm/types/tx_args.go`), a zero `globalGasCap` causes the gas limit to be set to `math.MaxUint64 / 2` when `args.Gas` is nil — effectively unbounded execution.

The identical pattern exists in `TraceCall`: [4](#0-3) 

The JSON-RPC backend is **not** affected because `DoCall` and `EstimateGas` always populate `GasCap` with `b.RPCGasCap()` (default 25 M): [5](#0-4) 

However, the gRPC query endpoint is publicly reachable (Cosmos SDK exposes it on port 9090 by default) and has no such protection when `queryMaxGasLimit` is not explicitly configured.

### Impact Explanation

An attacker deploys a contract containing an infinite loop or a maximally expensive opcode sequence, then sends a single gRPC `EthCall` request with `gas_cap = 0` and no `gas` field in the call args. The node executes the contract with `GasLimit ≈ math.MaxUint64 / 2`, consuming all available CPU for an arbitrarily long time. A validator node rendered unresponsive by this query cannot sign or propose blocks, causing it to miss rounds. If multiple validators are targeted simultaneously, the chain can halt below the 2/3 liveness threshold.

**Impact class:** High — public gRPC path exposes a reachable route to chain halt.

### Likelihood Explanation

- The gRPC query port is publicly accessible on every full node and validator by default.
- `queryMaxGasLimit` defaults to `GasNoLimit = 0`; operators must explicitly opt in to set a limit.
- Omitting `gas_cap` from a protobuf `EthCallRequest` is the natural default (zero value), so the attack requires no special knowledge.
- No authentication, stake, or privileged access is required.

### Recommendation

1. Change the default behavior so that `gasCap = 0` in the request is treated as "use the configured cap" rather than "no cap," regardless of `queryMaxGasLimit`.
2. In `EthCall` and `TraceCall`, add a fallback: if `gasCap == 0` after the `queryMaxGasLimit` enforcement block, reject the request or substitute a safe default (e.g., the block gas limit or a hard-coded maximum).
3. Document that `queryMaxGasLimit` must be set to a non-zero value in production deployments, and consider making a non-zero default mandatory at keeper construction time.

### Proof of Concept

```
# 1. Deploy a contract with an infinite loop (JUMPDEST PUSH1 0x00 JUMP = 0x5b 0x60 0x00 0x56)
# 2. Send a gRPC EthCall with gas_cap omitted (defaults to 0):

grpcurl -plaintext -d '{
  "args": "{\"from\":\"0x...\",\"to\":\"0x<infinite-loop-contract>\"}",
  "gas_cap": 0,
  "chain_id": 9000
}' localhost:9090 ethermint.evm.v1.Query/EthCall

# Result: node executes EVM with GasLimit = math.MaxUint64/2,
# consuming 100% CPU indefinitely, blocking block processing.
```
