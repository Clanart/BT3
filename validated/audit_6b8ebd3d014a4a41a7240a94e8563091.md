### Title
OS `get_block_hashes` Accepts Unconstrained `header_commitments` and `gas_prices_hash` via Hints, Producing a Block Hash That Does Not Commit to Transaction, Event, Receipt, or State-Diff Data - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/block_hash.cairo`)

---

### Summary

The `get_block_hashes` function in the Starknet OS Cairo program sets `header_commitments` (containing `transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`, and `packed_lengths`) and `gas_prices_hash` exclusively via the `%{ GetBlockHashes %}` hint, with **no Cairo constraints** (assert statements) verifying these values against the actual block data. The OS itself documents this gap: *"Currently, the header commitments and gas prices are not computed by the OS."* As a result, the block hash written to the block hash table contract storage does not cryptographically commit to the actual transactions, events, receipts, or state diff of the block. The OS flow tests confirm this by always passing `BlockHeaderCommitments::default()` (all fields zero) as `block_hash_commitments`.

This is the direct sequencer analog of the external bug: just as `AaveProvider.exchangeRate` always returns `1` and `BetaProvider.exchangeRate` always returns `0` (causing reward calculations to be wrong), the OS always uses zero/default `header_commitments` (causing the block hash to not commit to real block data).

---

### Finding Description

**Root cause — unconstrained hint inputs in `block_hash.cairo`:**

In `get_block_hashes`, five commitment fields and `gas_prices_hash` are declared as locals and populated solely via a hint:

```cairo
// Currently, the header commitments and gas prices are not computed by the OS.
// TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
local header_commitments: BlockHeaderCommitments*;
local gas_prices_hash;
local starknet_version;

%{ GetBlockHashes %}

let block_hash = calculate_block_hash(
    block_info=block_info,
    header_commitments=header_commitments,
    gas_prices_hash=gas_prices_hash,
    state_root=state_root,
    previous_block_hash=previous_block_hash,
    starknet_version=starknet_version,
);

%{ CheckBlockHashConsistency %}
``` [1](#0-0) 

There are **no `assert` statements** between the hint and the `calculate_block_hash` call that bind `header_commitments` or `gas_prices_hash` to the actual Merkle roots of the block's transactions, events, receipts, or state diff. In Cairo 0, hints are prover-side execution only; they create no verifiable constraints. `%{ CheckBlockHashConsistency %}` is also a hint — it cannot be enforced by the verifier.

**What `calculate_block_hash` commits to:**

```
Poseidon(
  BLOCK_HASH_VERSION, block_number,
  state_root,          ← constrained (Patricia trie)
  sequencer, timestamp,
  header_commitments.packed_lengths,        ← UNCONSTRAINED
  header_commitments.state_diff_commitment, ← UNCONSTRAINED
  header_commitments.transaction_commitment,← UNCONSTRAINED
  header_commitments.event_commitment,      ← UNCONSTRAINED
  header_commitments.receipt_commitment,    ← UNCONSTRAINED
  gas_prices_hash,                          ← UNCONSTRAINED
  starknet_version, 0,
  previous_block_hash  ← constrained (chained from previous block)
)
``` [2](#0-1) 

Only `state_root` and `previous_block_hash` are constrained. The five `BlockHeaderCommitments` fields and `gas_prices_hash` are free hint inputs.

**Hint implementation reads from `OsBlockInput.block_hash_commitments`:**

The `GetBlockHashes` hint in Rust reads `header_commitments` directly from `os_input.block_hash_commitments` without any verification that these match the actual computed Merkle roots:

```rust
let commitments = &os_input.block_hash_commitments;
ctx.insert_values_to_fields(
    header_commitments_ptr,
    CairoStruct::BlockHeaderCommitments,
    &[
        ("transaction_commitment", commitments.transaction_commitment.0.into()),
        ("event_commitment", commitments.event_commitment.0.into()),
        ...
    ],
)?;
``` [3](#0-2) 

**OS flow tests confirm zeros are always used:**

The OS flow test manager always sets `block_hash_commitments = BlockHeaderCommitments::default()` (all fields zero), with the comment *"TODO(Nimrod): Remove dummy block hashes once the OS verifies them."* The `new_block_hash` is then computed from these zeros and passed into `OsBlockInput`:

```rust
// TODO(Nimrod): Remove dummy block hashes once the OS verifies them.
let block_hash_commitments = BlockHeaderCommitments::default();
let new_block_hash = calculate_block_hash(
    &PartialBlockHashComponents::new(&block_info, block_hash_commitments.clone()),
    block_hash_state_root,
    prev_block_hash,
).unwrap();
// ...
let os_block_input = OsBlockInput {
    // ...
    block_hash_commitments,
    new_block_hash,
    // ...
};
``` [4](#0-3) 

Because `new_block_hash` is pre-computed from zeros and passed as input, and the OS computes the same hash from the same zeros, `CheckBlockHashConsistency` passes — but the resulting hash commits to nothing about the actual block content.

---

### Impact Explanation

The block hash written to the block hash table contract storage is a **wrong storage value** — it does not commit to the actual transaction commitments, event commitments, receipt commitments, state diff commitment, or gas prices of the block. This breaks the fundamental security invariant that the block hash is a cryptographic commitment to all block data.

Concrete consequences:

1. **Block hash table stores wrong values.** The `WriteOldBlockToStorage` hint writes the OS-computed block hash (with zero commitments) to the block hash table contract. Any contract or proof system that reads this hash and expects it to commit to transaction/event/receipt/state-diff data receives a wrong authoritative value.

2. **`check_proof_facts` validation is weakened.** The `check_proof_facts` function in `execution_constraints.cairo` reads the block hash from storage to validate virtual OS proofs. If the stored block hash was computed with zero commitments, a proof that references the correct (real-commitment) block hash would fail validation, while a proof referencing the wrong (zero-commitment) hash would pass. [6](#0-5) 

3. **RPC returns wrong block hash.** The block hash returned by `starknet_getBlockWithTxHashes` and related RPC endpoints is the wrong value — it does not commit to the actual block content.

This matches the allowed impact: *"Wrong state... storage value... from blockifier/syscall/execution logic for accepted input"* and *"RPC execution... returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

This is a **systematic, always-occurring** issue — not a rare edge case. Every block produced by the OS has a block hash that does not commit to its transaction, event, receipt, or state-diff data. The broken invariant is present in every OS run, triggered by normal block production without any special user action. The Cairo code comment and the test TODO both confirm this is the current production behavior.

---

### Recommendation

Implement Cairo constraints in `get_block_hashes` that verify `header_commitments` against the actual computed Merkle roots of the block's transactions, events, receipts, and state diff. Specifically:

1. Compute `transaction_commitment`, `event_commitment`, `receipt_commitment`, and `state_diff_commitment` inside the OS Cairo code (or pass them as verified outputs from the transaction execution loop).
2. Add `assert` statements binding the hint-provided `header_commitments` fields to these computed values before calling `calculate_block_hash`.
3. Remove the `BlockHeaderCommitments::default()` usage in the OS flow test manager and replace it with real commitments computed from the actual block data.

Until this is fixed, the block hash does not provide the expected security guarantees and cannot be used as a reliable commitment to block content.

pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}

```rust
pub(crate) fn get_block_hashes<S: StateReader>(
    hint_processor: &mut SnosHintProcessor<'_, S>,
    mut ctx: HintContext<'_>,
) -> OsHintResult {
    let os_input = &hint_processor.get_current_execution_helper()?.os_block_input;
    let block_info = &os_input.block_info;
    let gas_prices = &block_info.gas_prices;
    let commitments = &os_input.block_hash_commitments;

    ctx.insert_value(Ids::PreviousBlockHash, os_input.prev_block_hash.0)?;

    let header_commitments_ptr = ctx.vm.add_memory_segment();
    ctx.insert_values_to_fields(
        header_commitments_ptr,
        CairoStruct::BlockHeaderCommitments,
        &[
            ("transaction_commitment", commitments.transaction_commitment.0.into()),
            ("event_commitment", commitments.event_commitment.0.into()),
            ("receipt_commitment", commitments.receipt_commitment.0.into()),
            ("state_diff_commitment", commitments.state_diff_commitment.0.0.into()),
            ("packed_lengths", commitments.concatenated_counts.into()),
        ],
    )?;
    ctx.insert_value(Ids::HeaderCommitments, header_commitments_ptr)?;

    // The on-chain block hash commits to the exact ASCII encoding of the version string, which
    // may be newer than the latest known `StarknetVersion` variant.
    let starknet_version_string = os_input
        .starknet_version_override
        .clone()
        .unwrap_or_else(|| block_info.starknet_version.to_string());
    ctx.insert_value(Ids::StarknetVersion, ascii_as_felt(&starknet_version_string)?)?;

    let [gas_prices_hash]: [Felt; 1] = gas_prices_to_hash(
        &gas_prices.l1_gas_price_per_token(),
        &gas_prices.l1_data_gas_price_per_token(),
        &gas_prices.l2_gas_price_per_token(),
        &block_info.starknet_version.try_into()?,
    )
    .try_into()
    .map_err(|_| {
        OsHintError::UnsupportedStarknetVersionForBlockHash(block_info.starknet_version)
    })?;

    ctx.insert_value(Ids::GasPricesHash, gas_prices_hash)?;

    Ok(())
}
