### Title
Fee Validation Gap Allows Normalized Net Amount to Become Zero, Permanently Freezing User Funds - (File: near/omni-bridge/src/lib.rs)

### Summary

Both `init_transfer` and `update_transfer_fee` validate the fee only against the raw token amount (`fee.fee < amount`), but `sign_transfer` applies `normalize_amount` (floor division) to the net amount before checking it is non-zero. When the raw net amount (`amount - fee`) is positive but smaller than the decimal normalization divisor, `normalize_amount` floors it to zero, causing `sign_transfer` to permanently revert with `InvalidAmountToTransfer`. Since tokens are already locked or burned at `init_transfer` time and no cancellation mechanism exists, the user's funds are irrecoverably frozen.

### Finding Description

**Step 1 — Tokens are locked/burned at `init_transfer` time.**

In `init_transfer_internal`, the contract burns or locks the full token amount before the transfer is stored in `pending_transfers`:

```rust
// near/omni-bridge/src/lib.rs:1851-1857
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

**Step 2 — Fee validation uses only the raw amount.**

`init_transfer` checks:
```rust
// near/omni-bridge/src/lib.rs:554-557
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

`update_transfer_fee` checks:
```rust
// near/omni-bridge/src/lib.rs:399-402
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

Neither check validates that `normalize_amount(amount - fee) > 0`.

**Step 3 — `sign_transfer` normalizes the net amount and requires it to be non-zero.**

```rust
// near/omni-bridge/src/lib.rs:475-485
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

**Step 4 — `normalize_amount` uses floor division.**

```rust
// near/omni-bridge/src/lib.rs:2784-2787
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For a token with `origin_decimals = 24` and `decimals = 18`, the divisor is `10^6`. Any net amount less than `1_000_000` normalizes to zero.

**Step 5 — No cancellation or recovery path exists.**

A search for `cancel_transfer`, `withdraw_transfer`, `refund_transfer`, or `remove_pending` in `near/omni-bridge/src/` returns no matches. The `storage_unregister` function blocks unregistration while pending transfers exist (unless `force=true`), but even forced unregistration does not recover the locked/burned tokens.

### Impact Explanation

Once a transfer is in the stuck state, `sign_transfer` will always revert with `ERR_INVALID_AMOUNT_TO_TRANSFER`. The tokens were already burned or locked at `init_transfer` time. There is no public function to cancel the pending transfer or recover the funds. The user's assets are permanently frozen in the bridge — matching the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

The condition is reachable in two independent ways:

1. **At `init_transfer` time**: A user bridging a token with a large decimal gap (e.g., 24 → 18, divisor = 10^6) who sets a fee such that `amount - fee < 10^6` will immediately create an unresolvable transfer. This can happen accidentally when a user sets a high fee to attract relayers.

2. **Via `update_transfer_fee`**: A sender who has an existing pending transfer can call `update_transfer_fee` to raise their fee to `amount - 1`, which passes the raw check but leaves a net amount of 1, which normalizes to 0. Fees can only be increased (line 400: `fee.fee >= current_fee.fee`), so the state cannot be self-corrected.

Tokens with large decimal gaps (e.g., NEAR-native tokens at 24 decimals bridging to EVM at 18 decimals) are a standard use case for this bridge, making the condition realistic.

### Recommendation

Add a post-normalization check in both `init_transfer` and `update_transfer_fee` to ensure the net amount after fee deduction is non-zero when normalized to the destination chain's decimals:

```rust
// After computing the fee, look up decimals for the destination token and assert:
let token_address = self.get_token_address(destination_chain, token_id);
if let Some(decimals) = self.token_decimals.get(&token_address) {
    let normalized = Self::normalize_amount(amount_without_fee, decimals);
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

Alternatively, enforce a minimum `amount - fee` floor at the protocol level that accounts for the maximum possible decimal normalization divisor.

### Proof of Concept

Consider a NEAR-native token registered with `origin_decimals = 24` and `decimals = 18` (divisor = `10^6`):

1. User calls `ft_transfer_call` → `init_transfer` with `amount = 1_000_000`, `fee = 999_999`.
   - Check passes: `999_999 < 1_000_000` ✓
   - Tokens are burned/locked for the full `1_000_000`.
   - Transfer stored in `pending_transfers`.

2. Relayer calls `sign_transfer`:
   - `amount_without_fee() = 1_000_000 - 999_999 = 1`
   - `normalize_amount(1, {decimals: 18, origin_decimals: 24}) = 1 / 1_000_000 = 0`
   - `require!(0 > 0, ...)` → **panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`**

3. Every subsequent call to `sign_transfer` for this transfer ID panics identically.

4. No `cancel_transfer` exists. The `1_000_000` raw token units are permanently locked/burned.

The same outcome is reachable via `update_transfer_fee` on an existing transfer with a smaller initial fee, by raising the fee to `amount - 1`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
