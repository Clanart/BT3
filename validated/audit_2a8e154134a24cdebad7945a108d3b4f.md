### Title
Malicious P2P Peer Can Inject Arbitrary Events into Storage via Unvalidated `TransactionOutput` in `TransactionStreamFactory::parse_data_for_block` — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

The P2P sync client accepts `FullTransaction` messages from untrusted peers and stores the peer-supplied `TransactionOutput` (including all events) directly into the events table and transaction output file without ever validating the events against the `event_commitment` stored in the block header. A malicious peer that responds to a transaction query can inject arbitrary events for any block, corrupting the node's storage and causing `starknet_getEvents` to return attacker-chosen events.

### Finding Description

`TransactionStreamFactory::parse_data_for_block` reads `n_transactions` from the stored header and then collects exactly that many `FullTransaction` messages from the network: [1](#0-0) 

The only guard is a count check (`current_transaction_len < target_transaction_len`). The `transaction_output` field — which contains the events — is pushed directly into `block_body.transaction_outputs` with no validation: [2](#0-1) 

A developer TODO at line 88 explicitly acknowledges that peer-supplied transaction hashes are not validated; there is no analogous guard for event content at all.

`write_to_storage` then calls `append_body` unconditionally: [3](#0-2) 

`append_body` calls `write_transactions`, which calls `write_events`: [4](#0-3) 

`write_events` iterates the events from the peer-supplied `tx_output` and writes them to the events table with no commitment check: [5](#0-4) 

The block header's `event_commitment` field — which is a cryptographic commitment to the correct event set — is stored during header sync but is **never consulted** during transaction/event storage. There is no post-storage validation step anywhere in the sync pipeline that recomputes the event commitment and compares it to the header value.

### Impact Explanation

After a malicious peer responds to a transaction query with `FullTransaction` messages carrying forged `TransactionOutput` events:

1. The forged events are persisted in the events table and the transaction output file.
2. `starknet_getEvents` RPC reads directly from this storage and returns the attacker-chosen events as authoritative data for the block.
3. The `event_commitment` stored in the block header is cryptographically inconsistent with the stored events, but no running code detects or surfaces this inconsistency.

This satisfies the **High** allowed impact: *RPC returns an authoritative-looking wrong value*.

### Likelihood Explanation

Any node running P2P sync (`TransactionStreamFactory`) that connects to a malicious peer is vulnerable. The attacker only needs to be a peer that the syncing node queries for transactions — no special privileges are required. The attack is silent: the node logs a successful sync and advances its body marker normally.

### Recommendation

After collecting all `FullTransaction` messages for a block, recompute the event commitment over the collected events and compare it to `header.event_commitment` before calling `write_to_storage`. Reject the peer (via `report_peer`) and retry if the commitment does not match. This is the same pattern already used for `state_diff_length` validation in `StateDiffStreamBuilder::parse_data_for_block`. [6](#0-5)

fn write_events<'env>(
    tx_output: &TransactionOutput,
    txn: &DbTransaction<'env, RW>,
    events_table: &'env EventsTable<'env>,
    transaction_index: TransactionIndex,
) -> StorageResult<()> {
    let mut contract_addresses_set = HashSet::new();

    for event in tx_output.events().iter() {
        contract_addresses_set.insert(event.from_address);
    }

    for contract_address in contract_addresses_set {
        let key = (contract_address, transaction_index);
        // Here, we use the function assumption; the append will fail if an older transaction_index
        // is a table.
        events_table.append_greater_sub_key(txn, &key, &NoValue)?;
    }
    Ok(())
}
