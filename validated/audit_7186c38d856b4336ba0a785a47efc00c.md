### Title
Parallel Unguarded `burn.and(mint)` in `swap_migrated_token` Causes Permanent Token Loss or Unbacked Mint — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`swap_migrated_token` issues a `burn.and(mint)` joint promise that is immediately `.detach()`ed with no callback. In NEAR's async model, `Promise::and()` schedules both sub-promises **independently and in parallel**; the failure of one does not prevent the other from executing. Because `ft_on_transfer` also returns `U128(0)` (keeping the old tokens in the bridge's balance), two critical failure modes exist:

1. **`mint` fails, `burn` succeeds** → old tokens are permanently destroyed; user receives nothing (irrecoverable loss).
2. **`burn` fails, `mint` succeeds** → new tokens are minted without old tokens being destroyed (unbacked supply inflation).

---

### Finding Description

`migrate_deployed_token` (DAO-only) registers an `(old_token → new_token)` mapping in `migrated_tokens` and removes `old_token` from `deployed_tokens`. [1](#0-0) 

After migration, any holder of `old_token` can call `ft_transfer_call` on `old_token` targeting the bridge with the `SwapMigratedToken` message. The bridge's `ft_on_transfer` handler immediately returns `U128(0)` (keeping all transferred tokens) and `.detach()`s the swap promise: [2](#0-1) 

`swap_migrated_token` then constructs: [3](#0-2) 

`burn.and(mint)` in NEAR SDK creates a **joint promise** where both sub-promises are scheduled to the runtime independently. Neither sub-promise's execution is conditioned on the other's success. Because the joint promise is `.detach()`ed, there is no callback to detect or recover from partial failure.

The `burn` call targets `OmniToken::burn`, which calls `internal_withdraw` from the bridge's balance in the old token contract: [4](#0-3) 

The `mint` call targets `OmniToken::mint`, which calls `internal_deposit` into the user's account in the new token contract: [5](#0-4) 

`internal_deposit` panics if the recipient account is not registered (no storage deposit) in the new token contract — a standard NEAR FT requirement. When this panic occurs, the `mint` sub-promise fails. However, the `burn` sub-promise has already been scheduled and executes regardless, permanently withdrawing the old tokens from the bridge's balance. The bridge received those tokens via `ft_on_transfer` (returned `U128(0)`), so the burn succeeds and the tokens are gone. The user receives nothing and has no recourse.

---

### Impact Explanation

**Scenario A — `mint` fails, `burn` succeeds (permanent irrecoverable loss):**
The user's old tokens are burned from the bridge's balance. No new tokens are minted. The user's funds are permanently destroyed with no recovery path. This matches **Critical: Permanent freezing / irrecoverable loss of user funds in bridge token flows**.

**Scenario B — `burn` fails, `mint` succeeds (unbacked supply inflation):**
New tokens are minted to the user while the bridge retains the old tokens (not burned). The new token's circulating supply is inflated beyond what is backed by burned old tokens. This breaks bridge collateralization and matches **High: Balance / accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

Scenario A is reachable by any unprivileged token holder. The NEAR FT standard requires explicit `storage_deposit` registration before receiving tokens. A user who holds `old_token` and initiates a swap without first registering in `new_token` will trigger this path. There is no pre-swap registration check in the bridge, no user-facing warning, and no recovery mechanism. The likelihood is **moderate to high** given that token migrations are infrequent events where users may not be aware of the registration requirement for the new contract.

Scenario B requires the `burn` to fail while `mint` succeeds. The burn targets the bridge's own balance (which it holds after `ft_on_transfer`), so failure requires an external condition such as the old token being paused. This is lower likelihood but still reachable.

---

### Recommendation

Replace the parallel `burn.and(mint)` with a sequential chain `burn.then(mint)` and add a callback that checks the mint result. If the mint fails, the callback should re-mint the old tokens back to the user (or transfer them back), preventing permanent loss. Example structure:

```rust
ext_token::ext(old_token)
    .burn(amount)
    .then(
        Self::ext(env::current_account_id())
            .swap_migrated_token_callback(sender_id, new_token, amount, old_token_for_refund)
    )
```

The callback should inspect the burn result and, only on success, call `mint` on the new token. If `mint` subsequently fails, a further callback should refund the user's old tokens.

Additionally, consider verifying that `sender_id` has a storage deposit in `new_token` before initiating the burn, reverting the entire operation if not.

---

### Proof of Concept

1. DAO calls `migrate_deployed_token(Eth, old_token, new_token)`. `old_token` is removed from `deployed_tokens`; `migrated_tokens[old_token] = new_token`.
2. User holds 1000 units of `old_token` but has **not** called `storage_deposit` on `new_token`.
3. User calls `old_token.ft_transfer_call(bridge, 1000, '{"SwapMigratedToken": null}')`.
4. Bridge's `ft_on_transfer` is invoked with `token_id = old_token`, `amount = 1000`, `sender_id = user`. It returns `U128(0)` — bridge now holds 1000 `old_token`.
5. `swap_migrated_token(user, old_token, 1000)` is called. `burn.and(mint)` is created and `.detach()`ed.
6. NEAR runtime schedules both sub-promises in parallel:
   - `old_token.burn(1000)` — bridge's balance in `old_token` is 1000 (just received). `assert_controller()` passes (bridge is controller). `internal_withdraw(bridge, 1000)` succeeds. **1000 old tokens destroyed.**
   - `new_token.mint(user, 1000, None)` — `internal_deposit(user, 1000)` panics: user has no storage registration. **Mint fails.**
7. No callback exists to detect the failure or refund the user.
8. User has lost 1000 `old_token` permanently and received 0 `new_token`. [2](#0-1) [3](#0-2) [4](#0-3)

    fn swap_migrated_token(
        &mut self,
        sender_id: AccountId,
        old_token: AccountId,
        amount: U128,
    ) -> Promise {

            .migrated_tokens
