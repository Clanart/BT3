### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Allows Unbacked Supply Minting on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` emits the user-supplied `amount` in the `InitTransfer` event without verifying the actual tokens received from `safeTransferFrom`. For fee-on-transfer ERC20 tokens in the native-lock path, the bridge holds fewer tokens than the event claims, while the NEAR bridge mints the full event `amount`, permanently breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer`, when `tokenAddress` is a plain ERC20 (not a bridge token and not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // user-supplied
);
``` [1](#0-0) 

Immediately after, the contract emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // same user-supplied value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

No balance-before/balance-after check is performed. For a fee-on-transfer ERC20 token, `safeTransferFrom` delivers `amount - transferFee` to the bridge, but the event records `amount`. The NEAR bridge consumes this event proof in `fin_transfer_callback`, which reads the `amount` field from the decoded proof and mints or releases exactly that value to the recipient:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

The NEAR side has no way to detect the discrepancy — it trusts the event payload entirely. Every such transfer permanently inflates the NEAR-side supply relative to the EVM-side collateral by exactly `transferFee` tokens.

---

### Impact Explanation

Each `initTransfer` call with a fee-on-transfer token causes the EVM bridge to hold `amount - transferFee` while the NEAR bridge mints `amount`. The bridge becomes progressively undercollateralized. When NEAR-side holders bridge back, the EVM bridge will eventually be unable to fulfill redemptions, permanently trapping user funds. This matches the allowed impact: **Balance/accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

Any ERC20 token with a built-in transfer fee (e.g., tokens with reflection mechanics, deflationary tokens, or tokens with configurable fees) that is registered with the bridge triggers this path. The attacker needs only to call the public `initTransfer` function — no privileged access is required. The discrepancy accumulates silently with every transfer.

---

### Recommendation

Use a balance-before/balance-after pattern to determine the actual amount received, and use that value in the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived (cast to uint128 with overflow check) in the event and downstream logic
```

Alternatively, explicitly disallow fee-on-transfer tokens by requiring `actualReceived == amount` and reverting otherwise.

---

### Proof of Concept

1. Deploy or use a registered ERC20 token with a 1% transfer fee on EVM.
2. Call `OmniBridge.initTransfer(tokenAddress, 1000e18, 0, 0, nearRecipient, "")`.
3. The bridge receives `990e18` tokens (`safeTransferFrom` deducts 1% fee).
4. The `InitTransfer` event is emitted with `amount = 1000e18`.
5. A relayer submits the proof to the NEAR bridge via `fin_transfer`.
6. `fin_transfer_callback` decodes `amount = 1000e18` from the proof and mints `1000e18` NEAR-side tokens to the recipient.
7. The bridge is now undercollateralized by `10e18` tokens per transfer.
8. Repeating this 100 times drains the entire collateral buffer, making all future EVM redemptions fail and permanently trapping other users' funds. [4](#0-3) [5](#0-4)

        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();

            extensionValue = msg.value - amount - nativeFee;
