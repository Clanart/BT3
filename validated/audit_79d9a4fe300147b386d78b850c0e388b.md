### Title
Relayer Reimbursement Permanently Lost When `send_tokens` Fails in `process_fin_transfer_to_other_chain` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain`, when a proof is finalized for a transfer that had a prior fast transfer, the relayer reimbursement is dispatched via `send_tokens(...).detach()`. The promise result is never checked. If `send_tokens` fails for any reason, the fast transfer is still immediately marked as finalised via `mark_fast_transfer_as_finalised`, permanently closing the only reimbursement window. The relayer's funds are irrecoverably locked in the bridge contract with no retry or recovery path.

---

### Finding Description

When `process_fin_transfer_to_other_chain` is called with a proof for a transfer that had a prior fast transfer, the function:

1. Looks up the fast transfer status and retrieves the stored `relayer` account.
2. Calls `send_tokens(token, relayer, amount_without_fee, "").detach()` to reimburse the relayer.
3. **Immediately** calls `mark_fast_transfer_as_finalised(&fast_transfer.id())` — regardless of whether `send_tokens` succeeds or fails. [1](#0-0) 

The `.detach()` call means the promise result is never observed. If `send_tokens` fails (e.g., the relayer's NEAR account has no storage registered for the token, the token contract is paused, or the bridge has insufficient balance for a native token), the failure is silently swallowed.

After `mark_fast_transfer_as_finalised` sets `finalised = true`: [2](#0-1) 

Any subsequent attempt to re-submit the proof panics at `add_fin_transfer` with `TransferAlreadyFinalised`: [3](#0-2) 

And any attempt to re-trigger the fast-transfer reimbursement path panics with `FastTransferAlreadyFinalised`: [4](#0-3) 

There is no admin function, no retry mechanism, and no fallback claim path for the relayer. The relayer's tokens — deposited into the bridge when they executed the fast transfer — are permanently frozen.

The codebase itself acknowledges this class of problem in the UTXO fast-transfer path with an explicit `TODO`: [5](#0-4) 

The same unresolved issue exists in `process_fin_transfer_to_other_chain` without even a TODO.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of relayer funds** — matching the allowed Critical/High impact class.

The relayer provided liquidity upfront by executing a fast transfer. When the canonical proof is finalized, the bridge is supposed to reimburse the relayer. If `send_tokens` fails and the fast transfer is marked finalised, the relayer's deposited tokens are locked in the bridge contract forever. There is no governance escape hatch, no retry entry point, and no claim function. The loss is permanent.

---

### Likelihood Explanation

`send_tokens` for non-deployed tokens calls `ft_transfer`, which requires the recipient to have storage registered for the token. The `relayer` field in `FastFinTransferMsg` is caller-supplied: [6](#0-5) 

A relayer may legitimately specify a reward-collection account that has not yet registered storage for the specific token being reimbursed. Additionally, any transient failure in the token contract (pause, upgrade, out-of-gas on the token side) during the `send_tokens` call produces the same permanent loss. The proof submission is a public, permissionless call, so any party can trigger finalization at an inopportune moment.

---

### Recommendation

Replace `.detach()` with a proper callback that checks the result of `send_tokens`. If the transfer fails, either:
- Store a pending claimable balance for the relayer (similar to how `fin_transfer_send_tokens_callback` reverts lock actions on failure), or
- Do **not** call `mark_fast_transfer_as_finalised` until the reimbursement is confirmed successful.

The pattern already used in `fin_transfer_send_tokens_callback` — reverting state on failure — should be applied here: [7](#0-6) 

---

### Proof of Concept

1. Relayer calls `fast_fin_transfer` for a cross-chain transfer (e.g., ETH → SOL). The bridge records the fast transfer with `relayer = reward_account` and the relayer's tokens are held by the bridge.
2. `reward_account` has not registered storage for the bridged token on NEAR.
3. Any party (including a griefing attacker) submits the canonical proof via `fin_transfer`.
4. `process_fin_transfer_to_other_chain` is called. It finds the fast transfer status and calls:
   ```
   send_tokens(token, reward_account, amount_without_fee, "").detach();
   mark_fast_transfer_as_finalised(&fast_transfer.id());
   ```
5. `send_tokens` dispatches `ft_transfer(reward_account, amount, None)`. The token contract rejects it because `reward_account` has no storage — the promise fails silently.
6. `mark_fast_transfer_as_finalised` has already executed. `finalised = true` is written to state.
7. The relayer attempts to recover: re-submitting the proof panics with `TransferAlreadyFinalised`; re-triggering the fast-transfer path panics with `FastTransferAlreadyFinalised`.
8. The relayer's tokens are permanently locked in the bridge contract.
