### Title
Missing Upper-Bound Range Check on `max_price_per_unit` Enables Modular-Arithmetic Fee Bypass — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`pack_resource_bounds` enforces only a **lower-bound** (`assert_nn`) on `max_price_per_unit`, leaving it unconstrained up to the field prime `P ≈ 2²⁵¹`. Because `compute_max_possible_fee` performs plain felt arithmetic (modulo `P`) on these values, an attacker can craft resource bounds whose fee sum wraps to exactly `0 (mod P)`. The OS then skips fee collection entirely, constituting a direct loss of funds for the protocol.

---

### Finding Description

**Root cause — `pack_resource_bounds` (transaction_hash.cairo, lines 103–108)**

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // ✓ upper-bounded to u64
    assert_nn(resource_bounds.max_price_per_unit);            // ✗ only lower-bounded (≥ 0)
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

`max_amount` is correctly clamped to `[0, 2⁶⁴ − 1]`. `max_price_per_unit` is only checked to be non-negative; it can legally be any felt in `[0, P − 1]`.

**Downstream sink — `compute_max_possible_fee` (transaction_impls.cairo, lines 87–102)**

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
         + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
         + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
```

All arithmetic is modulo `P`. No range check is applied to the result.

**Fee-skip gate — `charge_fee` (transaction_impls.cairo, lines 111–165)**

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();          // ← fee collection is entirely skipped
}
```

If `max_fee ≡ 0 (mod P)`, the function returns immediately and no ERC-20 transfer is executed.

---

### Impact Explanation

An attacker who successfully triggers `max_fee = 0` executes their transaction without paying any fee. Because the OS is the final proof-level arbiter of correctness, a valid STARK proof can be generated for a block containing such a transaction. The sequencer (and ultimately the fee-token holders) receive zero compensation for the work performed. This is a **direct, provable loss of funds** at the protocol level.

---

### Likelihood Explanation

The attack requires only that the attacker submit a transaction whose resource-bound values satisfy a single modular equation. The attacker controls all three `max_amount` / `max_price_per_unit` pairs and `tip`. Solving for a zero-sum is straightforward:

**Concrete example (all values are valid felts passing every existing check):**

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `1` |
| `l1_gas_bounds.max_price_per_unit` | `P − 1` |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas_bounds.max_amount` | `0` |

Sum: `1·(P−1) + 1·(1+0) + 0 = P − 1 + 1 = P ≡ 0 (mod P)`.

Every individual field passes `assert_nn_le(max_amount, 2⁶⁴−1)` and `assert_nn(max_price_per_unit)`. The transaction hash is computed over these values and is cryptographically bound to them, so the proof is valid. The only barrier is whether the off-chain gateway enforces an upper bound on `max_price_per_unit`; the OS itself does not, making the OS proof unsound with respect to fee collection.

---

### Recommendation

Add an upper-bound range check on `max_price_per_unit` inside `pack_resource_bounds`, consistent with the Starknet specification (u128):

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // ADD THIS
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

With `max_amount < 2⁶⁴` and `max_price_per_unit < 2¹²⁸`, each product is bounded by `2¹⁹² − 1`, and the sum of three such products is bounded by `3 · (2¹⁹² − 1) ≪ P`, making modular wrap-around impossible.

---

### Proof of Concept

1. Attacker constructs a v3 `INVOKE` transaction with the resource bounds shown in the table above.
2. The transaction is signed; the hash commits to `max_price_per_unit = P − 1`.
3. `pack_resource_bounds` is called during hash computation: `assert_nn(P − 1)` passes (P − 1 is a valid non-negative felt).
4. The sequencer includes the transaction in a block.
5. The OS executes `compute_max_possible_fee` → returns `0`.
6. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without transferring any tokens.
7. A valid STARK proof is produced for the block; the attacker's transaction is executed at zero cost. [1](#0-0) [2](#0-1) [3](#0-2)
