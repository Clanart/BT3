### Title
`ProofOfInclusion::valid()` Never Validates Against an External Trusted Root — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs` only verifies the internal hash-chain consistency of the proof structure. It never compares the computed root against any external, caller-supplied trusted root. The final equality check in `valid()` is a tautology — it compares `existing_hash` (the last `calculated_hash` from the loop) against `self.root_hash()` (which returns `last.combined_hash`), but the loop already asserted `calculated_hash == layer.combined_hash` for every layer. An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` and a fabricated but internally consistent `layers` chain, and `valid()` will return `true` for any claimed key-value inclusion.

`ProofOfInclusion` is a `Streamable` struct exposed via Python bindings:

The `valid()` method:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← tautology
}
``` [2](#0-1) 

And `root_hash()`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself
    } else {
        self.node_hash
    }
}
``` [3](#0-2) 

**The tautology**: After the loop, `existing_hash` holds the last `calculated_hash`. The loop already enforced `calculated_hash == layer.combined_hash` for every layer. Therefore `existing_hash` is guaranteed to equal `last.combined_hash`. But `root_hash()` returns exactly `last.combined_hash`. So the final check `existing_hash == self.root_hash()` is always `true` when the loop completes without returning `false`. It catches nothing.

**The missing check**: `valid()` never accepts a caller-supplied trusted root hash. It validates the proof against its own claimed root — a root the attacker fully controls. An attacker constructs:

1. Choose any target `node_hash` (e.g., hash of a key-value pair not in the tree).
2. Build a chain of `ProofOfInclusionLayer` values where each `combined_hash` is computed correctly from the previous step using attacker-chosen `other_hash` values.
3. The resulting `ProofOfInclusion` passes `valid()` unconditionally.

**Edge case — empty layers**: When `layers` is empty, the loop does not execute, `existing_hash` stays as `self.node_hash`, and `root_hash()` also returns `self.node_hash`. The check `existing_hash == self.root_hash()` is `self.node_hash == self.node_hash`, which is always `true`. Any `ProofOfInclusion { node_hash: arbitrary, layers: vec![] }` is considered valid. ### Impact Explanation

Any DataLayer consumer that receives a `ProofOfInclusion` from an untrusted peer and calls only `proof.valid()` (without separately comparing `proof.root_hash()` against a locally-known trusted root) can be deceived into accepting a forged inclusion proof for an arbitrary key-value pair. This allows an attacker to prove that any key-value mapping exists in a DataLayer Merkle tree when it does not, corrupting the integrity guarantees of the DataLayer state. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion or lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is `Streamable`, exposed via Python bindings, and the `valid()` method is the sole documented API for proof verification. Any DataLayer integration that follows the natural API (`proof.valid()`) without adding a separate root-hash comparison is vulnerable. The attacker needs only to craft a serialized `ProofOfInclusion` with an internally consistent but fabricated hash chain — no cryptographic break required.

### Recommendation

`valid()` must accept an external trusted root hash and compare against it instead of `self.root_hash()`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root  // compare against caller-supplied trusted root
}
```

The existing `valid()` (and its Python binding `py_valid()`) should either be removed or deprecated. All call sites must supply the trusted root obtained from a locally-verified source (e.g., the committed on-chain root hash), not from the proof itself.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

# Real tree with one entry
blob = MerkleBlob(bytearray())
real_key = KeyId(1)
real_value = ValueId(1)
real_hash = bytes([0xAA] * 32)
blob.insert(real_key, real_value, real_hash)
blob.calculate_lazy_hashes()

# Attacker wants to forge inclusion of a fake key
fake_node_hash = bytes([0xFF] * 32)

# Build a single-layer proof that is internally consistent
# but proves nothing about the real tree
import hashlib
other_hash = bytes([0x11] * 32)
# combined_hash = sha256(fake_node_hash || other_hash) or similar
# (exact hash function from calculate_internal_hash)
combined = hashlib.sha256(fake_node_hash + other_hash).digest()

forged_proof = ProofOfInclusion(
    node_hash=fake_node_hash,
    layers=[ProofOfInclusionLayer(
        other_hash_side=0,
        other_hash=other_hash,
        combined_hash=combined,
    )]
)

# valid() returns True — no external root check performed
assert forged_proof.valid(), "Forged proof accepted!"
# root_hash() returns the attacker-controlled combined hash
print("Forged root:", forged_proof.root_hash().hex())
```

The forged proof passes `valid()` because the internal chain is consistent and the final check compares `existing_hash` against `self.root_hash()` — both of which are attacker-controlled values derived from the proof itself. [2](#0-1)
