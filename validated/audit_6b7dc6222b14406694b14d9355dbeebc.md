### Title
Unchecked Return Value of Fee Transfer in `charge_fee` Allows Silent Fee Evasion - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in the StarkNet OS executes an ERC-20 `transfer` call via `non_reverting_select_execute_entry_point_func` but **never checks whether the transfer succeeded or failed**. This is a direct analog of the original report's "unchecked return value of `transferFrom`": a user can drain their fee token balance during transaction execution, causing the subsequent fee transfer to fail silently, while the OS commits the transaction's state changes as if the fee was paid.

---

### Finding Description

In `transaction_impls.cairo`, the `charge_fee` function is responsible for deducting the transaction fee from the user's account by calling the fee token contract's `transfer` entry point: [1](#0-0) 

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
```

The function `non_reverting_select_execute_entry_point_func` is explicitly designed **not to panic** on entry point failure — it returns a tuple containing `(retdata_size, retdata, is_deprecated)` and an embedded `failure_flag`. This is confirmed by its usage elsewhere in the same file, where the return value **is** inspected: [2](#0-1) 

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(
    block_context=block_context, execution_context=validate_deploy_execution_context
);
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = VALIDATED;
}
```

In `charge_fee`, however, the return value is **completely discarded**. If the ERC-20 `transfer` call fails (e.g., because the account has insufficient balance), the OS has no mechanism to detect this and proceeds to commit the block as if the fee was successfully collected.

This affects all three account transaction types that call `charge_fee`:

- `execute_invoke_function_transaction` — line 361 [3](#0-2) 
- `execute_deploy_account_transaction` — line 687 [4](#0-3) 
- `execute_declare_transaction` — line 822 [5](#0-4) 

The `charge_fee` function itself: [6](#0-5) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

When the fee transfer fails silently:

1. The transaction's state changes (including the fee token drain performed during `__execute__`) are committed to the state trie.
2. The fee transfer's state changes are internally reverted by `execute_entry_point`'s revert mechanism (since `is_reverted=1`).
3. The OS output and state commitment reflect a fully executed transaction with **zero fee paid**.

The sequencer/protocol does not receive the fee that was contractually owed. At scale, a coordinated set of such transactions results in systematic, provable fee theft embedded in valid OS proofs accepted by L1.

---

### Likelihood Explanation

**High.** The attack requires no privileged access. Any unprivileged transaction sender can:

1. Hold enough fee tokens to pass pre-execution checks.
2. Within `__execute__`, call `transfer` on the fee token contract to move all tokens to a different address.
3. When `charge_fee` runs post-execution, the account balance is zero; the ERC-20 transfer fails.
4. `charge_fee` returns without error; the OS proof is valid and accepted by L1.

This is a deliberate, repeatable, low-cost attack requiring only a standard invoke transaction.

---

### Recommendation

Check the return value of `non_reverting_select_execute_entry_point_func` inside `charge_fee`. If the fee transfer is reverted (i.e., `failure_flag != 0` or `retdata[0] != VALIDATED`), the OS must treat the entire transaction as invalid or must revert the transaction's state changes. Concretely:

```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
// Enforce that the fee transfer succeeded.
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = 1;  // ERC-20 transfer returns TRUE on success.
}
```

This mirrors the pattern already used for `__validate__` and `__validate_deploy__` return value checks in the same file.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker deploys an account contract whose `__execute__` function, when called, immediately calls `transfer` on the fee token contract to drain the account's entire balance to an attacker-controlled address.
2. Attacker submits an invoke transaction (version 3) with `max_l2_gas_amount` set to cover execution and a non-zero `max_price_per_unit`.
3. The OS executes `execute_invoke_function_transaction`:
   - `run_validate` passes (account has balance at validate time).
   - `non_reverting_select_execute_entry_point_func` executes `__execute__`, which drains the fee token balance. State change committed.
   - `charge_fee` calls `non_reverting_select_execute_entry_point_func` for the ERC-20 `transfer`. The transfer fails (balance = 0). `failure_flag = 1` is returned but **ignored**.
   - `charge_fee` returns normally.
4. The OS produces a valid proof. L1 accepts it. The transaction is finalized with zero fee paid.
5. The attacker's drained tokens are in the destination address; the sequencer received nothing.

**Necessary vulnerable code step:** `charge_fee` at line 161–163 of `transaction_impls.cairo` is the sole enforcement point for fee collection in the OS. Its failure to check the return value of the fee transfer is the necessary and sufficient root cause. [1](#0-0)
