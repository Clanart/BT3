Audit Report

## Title
Swap Conservation Violation in `buyToken1InBinSpecifiedIn`: Fallback Override Charges Full Input Without Updating Output — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary
In `SwapMath.buyToken1InBinSpecifiedIn`, the fallback at lines 758-760 sets `totalIn0Scaled = state.amountSpecifiedRemainingScaled` without recomputing `out1Scaled`. The subsequent settlement at lines 783-784 charges the trader the full remaining token0 input while crediting only the token1 output computed for the partial `targetPos` from Stage 2, causing a direct, permanent loss of trader principal. The symmetric defect exists in `buyToken0InBinSpecifiedIn` at lines 620-622.

## Finding Description
The function resolves the partial-fill position through three stages:

**Stage 1 (lines 720-736):** Analytical closed-form solution computes `targetPos`, `out1Scaled`, and `totalIn0Scaled`. [1](#0-0) 

**Stage 2 (lines 738-756):** If `totalIn0Scaled < remaining && targetPos > minFinalBinPos`, a linear-scaling refinement moves `targetPos` further toward `minFinalBinPos` and recomputes both `out1Scaled` and `totalIn0Scaled`. [2](#0-1) 

**Stage 3 (lines 758-760):** If Stage 2 still undershoots, the fallback fires:

```solidity
if (totalIn0Scaled < state.amountSpecifiedRemainingScaled && targetPos > minFinalBinPos) {
    totalIn0Scaled = state.amountSpecifiedRemainingScaled;
}
```

`out1Scaled` is **not updated**. [3](#0-2) 

Settlement then charges `totalIn0Scaled` (= full remaining input) and credits `out1Scaled` (= output for the partial `targetPos` from Stage 2): [4](#0-3) 

The overshoot branch at lines 762-773 correctly handles the symmetric case by proportionally rescaling `out1Scaled` alongside `totalIn0Scaled`:

```solidity
out1Scaled = (out1Scaled * state.amountSpecifiedRemainingScaled) / totalIn0Scaled;
totalIn0Scaled = state.amountSpecifiedRemainingScaled;
``` [5](#0-4) 

The undershoot fallback has no equivalent `out1Scaled` update, creating a structural asymmetry. The identical defect exists in `buyToken0InBinSpecifiedIn`: [6](#0-5) 

**Why Stage 3 is reachable:** Stage 2 uses linear scaling of the position delta. The actual cost function is non-linear — `calculateRequiredToken(out1Scaled, avgPriceX64)` depends on the arithmetic mean of two prices that are themselves non-linear functions of `targetPos`. In bins with wide price ranges (`upperPriceX64 / lowerPriceX64 >> 1`), one linear refinement step is insufficient to close the gap, leaving `totalIn0Scaled < remaining` after Stage 2.

## Impact Explanation
When the fallback fires, `state.amountSpecifiedRemainingScaled -= totalIn0Scaled` consumes the full remaining input (line 783), causing the outer swap loop to exit after this bin. `state.amountCalculatedScaled += out1Scaled` credits only the partial output (line 784). The trader pays `state.amountSpecifiedRemainingScaled` token0 but receives `out1Scaled` token1 where `out1Scaled * avgInvertedPrice < totalIn0Scaled`. The excess token0 accrues to LPs via the bin balance update at lines 780-781. This is a direct, permanent loss of trader principal — a swap conservation failure meeting the Critical/High threshold. [7](#0-6) 

## Likelihood Explanation
The condition requires: (a) the analytical solution undershoots, AND (b) one linear refinement still undershoots. This occurs in bins where the price curve non-linearity is large relative to the position step — e.g., bins with `upperPriceX64 / lowerPriceX64` ratios significantly above 1 and input amounts landing in the non-linear region of the quadratic cost curve. These are normal, permissionless pool configurations. No privileged setup is required; any unprivileged trader swapping token0→token1 through such a bin can trigger it.

## Recommendation
Replace the fallback at lines 758-760 with a full traversal to `minFinalBinPos`, recomputing `out1Scaled` and `totalIn0Scaled` consistently:

```solidity
if (totalIn0Scaled < state.amountSpecifiedRemainingScaled && targetPos > minFinalBinPos) {
    targetPos = minFinalBinPos;
    out1Scaled = calculateOutputToken1FromBinPosition(binState.token1BalanceScaled, currBinPos, targetPos);
    invertedFinalPriceX64 =
        invertPriceX64(calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Floor));
    avgPriceX64 = calculateArithmeticMean(invertedStartingPriceX64, invertedFinalPriceX64);
    in0WithoutFeeScaled = calculateRequiredToken(out1Scaled, avgPriceX64);
    totalIn0Scaled = grossInputWithBinFeeCeil(in0WithoutFeeScaled, onePlusSellFeeX64);
}
```

Apply the same fix to `buyToken0InBinSpecifiedIn` lines 620-622.

## Proof of Concept
Deploy a pool with `upperPriceX64 / lowerPriceX64 >= 100`, `currBinPos` near `type(uint104).max / 2`. Craft `remaining` to approximately `1.05 * totalIn0Scaled_after_analytical` so Stage 2's single linear refinement still undershoots. Call `buyToken1InBinSpecifiedIn` directly and assert:

```solidity
uint256 charged = remaining - state.amountSpecifiedRemainingScaled;
// Conservation invariant: charged <= out1Scaled * avgInvertedPrice / (1 << 64) + TOLERANCE
// Bug: charged >> out1Scaled * avgInvertedPrice / (1 << 64)  (trader overpaid)
assertLe(charged, out1Scaled * invertedAvgPrice / (1 << 64) + TOLERANCE, "Conservation violated");
```

The assertion fails when Stage 3 fires, confirming the trader overpays while `out1Scaled` remains at the Stage 2 partial value.
