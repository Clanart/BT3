### Title
Missing Validation of `class_hash` in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS processes the `replace_class` syscall without validating that the supplied `class_hash` is non-zero or corresponds to a declared contract class. A contract can call `replace_class(0)` (or any undeclared hash), causing the OS to commit an invalid class hash to state. The contract becomes permanently uncallable, freezing any funds it holds. The missing check is explicitly acknowledged by a developer TODO comment in the source.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the caller-supplied `class_hash` directly from the syscall request and writes it to `contract_state_changes` with no validation:

```cairo
// syscall_impls.cairo, execute_replace_class
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing check. `class_hash` can be `0` or any value not present in `contract_class_changes`.

When any future transaction calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // now 0
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // 0 — not found → OS panics / proof fails
);
```

Because class hash `0` is never declared, `find_element` cannot locate a matching `CompiledClassFact`, making the contract permanently uncallable at the OS level.

The identical missing validation exists in the deprecated path:

```cairo
// deprecated_execute_syscalls.cairo, execute_replace_class
let class_hash = syscall_ptr.class_hash;
// No validation before writing to contract_state_changes
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract deployer (unprivileged role) can:
1. Deploy a contract that accepts token/ETH deposits from other users.
2. Include a callable function that issues `replace_class(0)` on itself.
3. After accumulating user deposits, invoke that function.
4. The OS writes `class_hash = 0` to the committed state without rejection.
5. All subsequent calls to the contract fail at the OS proof layer.
6. Deposited funds are permanently inaccessible — no withdrawal, no upgrade path.

This is a direct, irreversible loss of user funds caused by the OS accepting an invalid state transition.

---

### Likelihood Explanation

The attack requires only deploying a contract and attracting deposits — both are unprivileged, permissionless actions on StarkNet. The OS's missing validation is the necessary enabling condition: without it, `replace_class(0)` would be rejected because class hash `0` is not declared. The attacker has full control over timing and execution. No privileged role, leaked key, or external dependency is required.

---

### Recommendation

Add explicit validation inside `execute_replace_class` (both `syscall_impls.cairo` and `deprecated_execute_syscalls.cairo`) before writing to `contract_state_changes`:

```cairo
// 1. Reject zero class hash.
assert_not_zero(class_hash);

// 2. Confirm the class hash is declared (exists in contract_class_changes).
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This directly resolves the acknowledged TODO and mirrors the validation pattern already applied to other critical parameters throughout the OS (e.g., `assert_not_zero(compiled_class_hash)` in `execute_declare_transaction`).

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract with a `deposit()` entry point and a `freeze()` entry point that calls `replace_class(0)`.
2. Users call `deposit()`, locking funds inside `MaliciousVault`.
3. Attacker calls `freeze()`. The `replace_class` syscall fires with `class_hash = 0`.
4. `execute_replace_class` in `syscall_impls.cairo` (line 896–915) writes `StateEntry(class_hash=0, ...)` to `contract_state_changes` without any check.
5. The OS commits this state. `MaliciousVault`'s on-chain class hash is now `0`.
6. Any future call to `MaliciousVault` reaches `execute_entry_point` (`execute_entry_point.cairo`, line 154–166), which reads `class_hash = 0`, fails to find a matching `CompiledClassFact`, and the OS cannot produce a valid execution — the contract is permanently bricked.
7. All user deposits are frozen with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2)
