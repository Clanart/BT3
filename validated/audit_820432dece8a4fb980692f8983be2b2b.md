### Title
Deploy Syscall Unconditionally Reports Success Regardless of Constructor Failure - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in `syscall_impls.cairo` always writes `failure_flag=0` in the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is the direct StarkNet analog of the external report's vulnerability class: **incomplete failure handling in a sequence of operations**, where a failure in a sub-step is silently swallowed and the caller proceeds as if everything succeeded.

---

### Finding Description

In `execute_deploy`, after calling `deploy_contract` to run the constructor, the response header is written with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns only `(retdata_size, retdata)` — it does not return an `is_reverted` flag:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [2](#0-1) 

Compare this to `contract_call_helper`, which correctly propagates the `is_reverted` flag from `select_execute_entry_point_func` into the response header:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The `execute_deploy` function has no equivalent path. The `failure_flag` is structurally hardcoded to `0` because `deploy_contract`'s return signature does not include a revert indicator, and the TODO comment explicitly acknowledges the missing failure support. [4](#0-3) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

A factory contract (Contract A) that:
1. Calls the `deploy` syscall to deploy Contract B,
2. Receives `failure_flag=0` (always), and
3. Transfers funds to Contract B's address assuming it was properly initialized,

will proceed even when Contract B's constructor reverted. The constructor revert is silently discarded by the OS. Contract B is deployed at the computed address with its class hash set in state, but its storage is in an uninitialized or partially-initialized state (constructor side-effects are reverted via the revert log). Any funds sent to Contract B after a failed constructor may be permanently locked if the contract's withdrawal or recovery logic depended on constructor-set state that was never committed.

This matches the allowed impact: **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attacker-controlled entry path is direct and unprivileged:
- Any user can submit an `InvokeFunction` transaction calling a factory contract that uses the `deploy` syscall.
- The attacker can craft a class whose constructor conditionally reverts (e.g., based on a storage value, block number, or calldata).
- Factory patterns that deploy-and-fund in a single transaction are common in DeFi protocols built on StarkNet.
- No privileged role, leaked key, or external dependency is required.

The condition is realistic: constructor failures can occur due to re-entrancy guards, duplicate deployment attempts, or attacker-controlled constructor calldata.

---

### Recommendation

1. Change `deploy_contract`'s return signature to include an `is_reverted: felt` flag, analogous to `select_execute_entry_point_func`.
2. In `execute_deploy`, propagate the constructor's revert status into the response header:
   ```cairo
   assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
   ```
3. If the constructor reverted, also revert the class hash assignment for the new contract address (i.e., do not commit the deployment to state), consistent with how `handle_revert` works for storage writes.
4. Remove the TODO comment once the fix is in place.

---

### Proof of Concept

**Setup:**
- Deploy a Sierra class `FailingFactory` whose `__execute__` calls the `deploy` syscall with class hash of `FailingChild`.
- `FailingChild`'s constructor always reverts (e.g., `assert 1 = 0`).
- After the `deploy` syscall, `FailingFactory` reads the returned `contract_address` from the response and calls the ERC-20 `transfer` syscall to send tokens to that address.

**Execution path in the OS:**

1. `execute_transactions_inner` dispatches the invoke transaction. [5](#0-4) 

2. `execute_invoke_function_transaction` runs `__execute__` of `FailingFactory`. [6](#0-5) 

3. The `deploy` syscall is dispatched to `execute_deploy`. [7](#0-6) 

4. `deploy_contract` runs `FailingChild`'s constructor, which reverts. The revert log unwinds constructor storage changes. However, `execute_deploy` writes `failure_flag=0` unconditionally. [8](#0-7) 

5. `FailingFactory` reads `failure_flag=0` from the response, concludes deployment succeeded, and transfers tokens to `FailingChild`'s address.

6. `FailingChild` is at the computed address with its class hash set but uninitialized storage. The tokens are now locked in a contract whose withdrawal logic was never initialized by the constructor.

**Result:** Funds are permanently frozen. The attacker (who controls `FailingChild`'s class) can trigger this against any factory contract that trusts the deploy syscall's success response.
