### Title
Orphaned Non-EVM-Native Bank Balances After `SELFDESTRUCT` Enable Fund Recovery via Address Recreation - (File: `x/evm/statedb/statedb.go`)

### Summary
`StateDB.Commit()` only burns the EVM-denom balance of a self-destructed contract. Non-EVM-native token balances (IBC tokens, CosmWasm bridge tokens) held by the destroyed address are not drained and remain as orphaned Cosmos bank balances. An attacker can recover these funds by recreating a contract at the same address via `CREATE2`, analogous to the external report's incomplete `burn()` that leaves a position token alive after funds are withdrawn.

### Finding Description
When a contract self-destructs, `StateDB.Commit()` in `x/evm/statedb/statedb.go` handles the destruction path as follows:

```go
if obj.selfDestructed {
    cosmosAddr := sdk.AccAddress(obj.Address().Bytes())
    cacheCtx, writeCache := s.origCtx.CacheContext()
    // Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
    // bridge) held by the destroyed address are not drained and may remain as
    // orphaned bank balances.
    if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
        coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
        if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
            return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
        }
    }
    if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
        return errorsmod.Wrap(err, "failed to delete account")
    }
    writeCache()
}
``` [1](#0-0) 

Only `s.evmDenom` is queried and burned. `DeleteAccount` then removes the auth account metadata and contract storage, but the Cosmos bank module stores balances by address independently of auth account existence. [2](#0-1) 

After `accountKeeper.RemoveAccount` executes, the bank module KV store still holds any non-EVM-native coin balances (e.g., IBC-transferred ATOM, USDC) at the destroyed address. These balances are not touched by `DeleteAccount` and are not iterated over in `Commit()`. [3](#0-2) 

In Cosmos SDK, bank balances are stored keyed by `(address, denom)` in the bank module's KV store, entirely separate from the auth module's account store. Removing the auth account does not remove bank balances. When a new account is later created at the same address (e.g., via `CREATE2` redeployment), the new account inherits the orphaned bank balances and can freely transfer them.

The code comment at line 813–815 explicitly acknowledges this gap:

> "Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm bridge) held by the destroyed address are not drained and may remain as orphaned bank balances." [4](#0-3) 

This is the direct analog to the external report: the `burn()` (selfdestruct) operation does not fully clean up all associated state — the ERC721 token (here: non-EVM-native bank balances) is left alive after the position is closed.

### Impact Explanation
An attacker can steal Cosmos bank funds (IBC tokens or any non-EVM-native coin) that were held by a self-destructed contract by recreating the contract at the same `CREATE2` address. The bank module balances survive the selfdestruct and are accessible to whoever controls the recreated account. This constitutes unauthorized theft of Cosmos bank funds through Ethermint stateDB logic, matching the Critical impact tier.

### Likelihood Explanation
- IBC tokens can be sent to any Cosmos address, including a predicted `CREATE2` address, before or after deployment.
- The attacker controls the factory contract and the `CREATE2` salt, so address recreation is deterministic and fully attacker-controlled.
- The entire setup (deploy factory → deploy child → receive IBC tokens → selfdestruct child → redeploy child → drain orphaned balance) can be orchestrated across a small number of transactions with no privileged access required.
- Any Ethermint deployment with IBC enabled (the standard production configuration) is affected.

### Recommendation
In `StateDB.Commit()`, when processing a self-destructed account, drain **all** bank balances held by the address — not only the EVM denom. Before calling `DeleteAccount`, iterate over all coin denominations held by `cosmosAddr` using `bankKeeper.GetAllBalances(cacheCtx, cosmosAddr)` and burn each non-zero balance via `SubBalance` / `BurnCoins`. This ensures no orphaned bank balances survive the selfdestruct and cannot be recovered via address recreation.

### Proof of Concept
1. Attacker deploys factory contract `F` on an Ethermint chain with IBC enabled.
2. Factory predicts the `CREATE2` address `C` for salt `S` and init code `I`.
3. Attacker (or any party) IBC-transfers ATOM to address `C` (before or after deployment).
4. In a single transaction (satisfying EIP-6780): factory deploys child `C` via `CREATE2(S, I)`, then calls `C.destroy()`. `C` self-destructs.
5. `StateDB.Commit()` burns `C`'s EVM-denom balance but leaves the ATOM balance at address `C` in the bank module.
6. `DeleteAccount` removes `C`'s auth account and storage; ATOM balance is untouched.
7. In a subsequent transaction: factory calls `CREATE2(S, I)` again, recreating `C` at the same address. A new auth account is registered at `C`.
8. The new `C` (or its controller) calls `bankKeeper.SendCoins` to transfer the orphaned ATOM to the attacker's address.

The orphaned ATOM balance — which should have been permanently locked or burned — is now in the attacker's possession. [5](#0-4) [6](#0-5)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
