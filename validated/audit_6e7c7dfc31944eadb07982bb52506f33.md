### Title
Unprotected `initialize()` on `OmniBridgeWormhole` Proxy Allows Anyone to Seize `DEFAULT_ADMIN_ROLE` and Forge MPC Signatures — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridge.initialize()` is declared `public` rather than `internal`. For `OmniBridgeWormhole` proxies, the intended initialization path is `initializeWormhole()`, which calls `initialize()` internally. Because `initialize()` is `public`, any unprivileged caller can invoke it directly on a freshly deployed proxy before `initializeWormhole()` is called, seizing `DEFAULT_ADMIN_ROLE` and permanently blocking the legitimate initialization. With `DEFAULT_ADMIN_ROLE`, the attacker can overwrite `nearBridgeDerivedAddress` (the MPC-derived ECDSA key used to verify all cross-chain proofs), forge transfer-finalization signatures, and mint arbitrary bridged tokens.

---

### Finding Description

`OmniBridge.initialize()` grants `DEFAULT_ADMIN_ROLE` and `PAUSABLE_ADMIN_ROLE` to `_msgSender()`:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {          // ← public, not internal
    ...
    _grantRole(DEFAULT_ADMIN_ROLE, _msgSender());
    _grantRole(PAUSABLE_ADMIN_ROLE, _msgSender());
}
``` [1](#0-0) 

`OmniBridgeWormhole` extends `OmniBridge` and provides its own initializer `initializeWormhole()`, which is the intended entry point for Wormhole deployments. It calls `initialize()` internally:

```solidity
function initializeWormhole(...) external initializer {
    initialize(tokenImplementationAddress, nearBridgeDerivedAddress, omniBridgeChainId);
    _wormhole = IWormhole(wormholeAddress);
    _consistencyLevel = consistencyLevel;
}
``` [2](#0-1) 

OpenZeppelin's `initializer` modifier sets `_initialized = 1` on the first top-level call and prevents any subsequent call. Because `initialize()` is `public`, an attacker can call it directly on the proxy as a top-level call. This sets `_initialized = 1` with the attacker as admin. Any subsequent call to `initializeWormhole()` then reverts with `InvalidInitialization` because `_initialized` is already non-zero, permanently bricking the legitimate initialization path and leaving the attacker in control.

The implementation contract itself is protected by `_disableInitializers()` in the `OmniBridge` constructor: [3](#0-2) 

However, this only locks the implementation's own storage. The proxy's `_initialized` slot starts at zero and is fully exploitable.

---

### Impact Explanation

Once the attacker holds `DEFAULT_ADMIN_ROLE` they can call `setNearBridgeDerivedAddress(attacker_controlled_address)`: [4](#0-3) 

`nearBridgeDerivedAddress` is the sole ECDSA signer checked in both `deployToken()` and `finTransfer()`: [5](#0-4) [6](#0-5) 

With that key replaced by an attacker-controlled address, the attacker can sign arbitrary `finTransfer` payloads and mint unbacked bridged tokens to any recipient — **unauthorized mint of bridged assets**. They can also upgrade the proxy (`_authorizeUpgrade` requires `DEFAULT_ADMIN_ROLE`) to a malicious implementation, permanently freezing or draining all locked collateral.

Impact category matched: **Critical — Direct unauthorized mint of bridged assets** and **High — MPC/signature verification bypass enabling unauthorized transfer finalization**.

---

### Likelihood Explanation

The standard deployment script uses `upgrades.deployProxy()` with `{ initializer: "initializeWormhole" }`, which atomically deploys the proxy and calls `initializeWormhole` in the same transaction: [7](#0-6) 

This closes the front-running window for that specific script. However:

1. The code itself imposes no enforcement of atomic deployment. Any deployment that separates proxy creation from initialization (e.g., a multisig governance flow, a manual `new ERC1967Proxy(impl, "")` call, or a deployment framework that does not pass init data to the constructor) opens the window.
2. `initialize()` being `public` is a persistent design flaw: it is callable on any `OmniBridgeWormhole` proxy at any point before `initializeWormhole()` completes, regardless of deployment tooling.
3. The attack requires only a single unprivileged transaction with no special permissions, capital, or timing beyond observing the mempool for a proxy deployment without init data.

Likelihood: **Medium** — requires a non-atomic deployment pattern, which is realistic in governance-controlled or multi-step deployments.

---

### Recommendation

Change `OmniBridge.initialize()` visibility from `public` to `internal` so it can only be called from within the contract hierarchy (e.g., from `initializeWormhole()`), not directly by external callers:

```solidity
// Before
function initialize(...) public initializer { ... }

// After
function initialize(...) internal onlyInitializing { ... }
```

Using `onlyInitializing` (instead of `initializer`) for the internal helper is the standard OpenZeppelin pattern for base-contract initializers that are called from a derived initializer. This ensures the only valid top-level initialization entry point for `OmniBridgeWormhole` is `initializeWormhole()`, and for plain `OmniBridge` proxies a separate `public initializer` wrapper should be added.

### Proof of Concept

```solidity
// 1. Deploy OmniBridgeWormhole implementation (constructor calls _disableInitializers — safe)
OmniBridgeWormhole impl = new OmniBridgeWormhole();

// 2. Deploy proxy WITHOUT initialization data (empty bytes)
ERC1967Proxy proxy = new ERC1967Proxy(address(impl), "");
OmniBridgeWormhole bridge = OmniBridgeWormhole(address(proxy));

// 3. Attacker front-runs and calls initialize() directly
//    _msgSender() == attacker → attacker gets DEFAULT_ADMIN_ROLE
bridge.initialize(
    address(someTokenImpl),
    address(attacker),   // nearBridgeDerivedAddress = attacker's key
    uint8(1)
);

// 4. Legitimate deployer's initializeWormhole() now reverts: InvalidInitialization
// bridge.initializeWormhole(...) → REVERTS

// 5. Attacker already has DEFAULT_ADMIN_ROLE; nearBridgeDerivedAddress is attacker's key.
//    Attacker signs an arbitrary finTransfer payload with their private key:
bytes memory fakeSignature = sign(attacker_privkey, payload);
bridge.finTransfer(fakeSignature, payload);  // mints arbitrary tokens to attacker
```

The root cause is `OmniBridge.initialize()` being `public` at [8](#0-7) , which allows any caller to seize admin roles on an `OmniBridgeWormhole` proxy before `initializeWormhole()` is called. [2](#0-1)
