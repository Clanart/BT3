### Title
`ProofOfInclusion::valid()` Validates Against Its Own Internal Root, Not a Trusted External Root — Forged Inclusion Proofs Always Pass — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` derives the expected root hash from the proof itself via `self.root_hash()`, which returns `self.layers.last().combined_hash`. After the verification loop, `existing_hash` is guaranteed to equal that same value by construction, making the final equality check a tautology. The method never accepts a trusted, externally-committed root hash as a parameter. Any attacker who can supply a `ProofOfInclusion` object — including via the Python/WASM bindings — can forge an internally-consistent proof for an arbitrary `node_hash` and have `valid()` return `true`, regardless of what the actual DataLayer tree root is.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs

pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← derived from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← tautology: both sides are layers.last().combined_hash
}
```

After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which was just verified to equal `layer.combined_hash`. `self.root_hash()` returns `self.layers.last().combined_hash` — the identical value. The final check `existing_hash == self.root_hash()` is therefore always `true` when layers are non-empty.

The root cause is the same structural error as the original report: **the validation is applied to the wrong context**. The original bug applied a yield update to the current chain instead of the chain where the collateral is held. Here, `valid()` validates the proof against the root embedded inside the proof itself, rather than against the actual committed DataLayer tree root. In both cases, the "wrong context" (self-supplied vs. externally trusted) makes the check meaningless.

An attacker can construct a forged `ProofOfInclusion` as follows:

1. Choose any arbitrary `node_hash` (the leaf they want to falsely prove is included).
2. Choose any `other_hash` and `other_hash_side`.
3. Compute `combined_hash = calculate_internal_hash(node_hash, other_hash_side, other_hash)`.
4. Build a `ProofOfInclusion { node_hash, layers: [ProofOfInclusionLayer { other_hash_side, other_hash, combined_hash }] }`.
5. Call `proof.valid()` → returns `true`.

This proof is internally consistent but proves nothing about any real DataLayer tree.

The struct is fully serializable via `Streamable` and exposed to Python via `py_valid()` and `from_bytes()`, meaning untrusted serialized input can be deserialized and validated entirely within the attacker's control.

### Impact Explanation

Any consumer of `ProofOfInclusion::valid()` — including the Python `chia_rs` wheel — that relies on it to confirm DataLayer state (e.g., to verify that a key-value pair is committed in a specific store root before acting on it) can be deceived by a forged proof. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

The `ProofOfInclusion` struct is exposed via Python bindings (`py_valid`, `from_bytes`, `from_json_dict`) and is `Streamable`, so it can be constructed from arbitrary bytes by any unprivileged caller. No special privileges, keys, or network access are required. The forged proof requires only a single hash computation.

### Recommendation

`valid()` must accept a trusted root hash as a parameter and compare the computed root against it, rather than against the proof's own embedded `combined_hash`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against externally-supplied root
}
```

All call sites — including the Python binding `py_valid()` — must be updated to pass the known, committed tree root.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Arbitrary leaf hash the attacker wants to "prove" is in the tree
fake_node_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)
other_hash_side = 1  # Right

# Compute a combined_hash that is internally consistent
h = hashlib.sha256()
h.update(b'\x02')          # NodeType::Mid prefix used by calculate_internal_hash
h.update(fake_node_hash)
h.update(other_hash)
combined_hash = h.digest()

layer = ProofOfInclusionLayer(
    other_hash_side=other_hash_side,
    other_hash=other_hash,
    combined_hash=combined_hash,
)
proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated from any real tree
assert proof.valid(), "Forged proof accepted!"
print("root_hash from forged proof:", proof.root_hash().hex())
# This root_hash has nothing to do with any real DataLayer store root
