### Title
Unchecked u128 Multiplication Overflow in `denormalize_amount` Permanently Freezes User Funds During Cross-Chain Transfer Finalization - (File: near/omni-bridge/src/lib.rs)

### Summary

`denormalize_amount` performs an unchecked `amount * 10_u128.pow(diff_decimals)` multiplication. For tokens where `origin_decimals > decimals` (diff ≥ 1) and where a user transfers an amount large enough that the normalized value exceeds `u128::MAX / 10^diff_decimals`, the multiplication overflows u128. Because NEAR contracts compile with `overflow-checks = true`, this causes a panic in `fin_transfer_callback`, permanently freezing the user's funds on the origin chain with no recovery path.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

This function is called inside `fin_transfer_callback` to reconstruct the full-precision amount from the normalized (bridge-decimal) amount carried in the proof:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [2](#0-1) 

It is also called in `fast_fin_transfer` and `claim_fee_callback`: [3](#0-2) [4](#0-3) 

The `Decimals` struct stores both the bridge-side precision and the origin-chain precision: [1](#0-0) 

The EVM `InitTransfer` event carries `amount` as `uint128` (the normalized, bridge-decimal value): [5](#0-4) 

The overflow condition is:

```
amount_normalized * 10^(origin_decimals - decimals) > u128::MAX
```

For a token with `origin_decimals = 24` and `decimals = 6` (`diff = 18`), the threshold is:

```
amount_normalized > u128::MAX / 10^18  ≈  3.4 × 10^20
```

In 6-decimal human units that is ≈ **340 trillion tokens** — within reach for high-supply tokens (e.g., meme tokens with quadrillion-scale supplies). A user who sends `u128::MAX * 10^diff + k` tokens on the origin chain causes the EVM bridge to emit a normalized `amount = u128::MAX` (which fits in `uint128`). NEAR then attempts `u128::MAX * 10`, which overflows.

### Impact Explanation

When `fin_transfer_callback` panics due to the overflow:

1. The NEAR-side transfer is never finalized.
2. The user's tokens are already burned or locked in the origin-chain vault.
3. There is no reversal or rescue mechanism — the origin-chain transaction is irreversible.

Result: **permanent, irrecoverable loss of the user's bridged funds** — matching the "Critical: Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows" impact category.

The same overflow in `fast_fin_transfer` additionally corrupts the `denormalized_amount == amount.0 + denormalized_fee.fee.0` invariant check, potentially causing a different panic path that also freezes funds.

### Likelihood Explanation

- Requires a token registered with `origin_decimals > decimals` (standard for any token whose origin precision exceeds the bridge cap, e.g., a 24-decimal NEAR token bridged to EVM at 6 decimals).
- Requires the user to hold and transfer a very large token amount (hundreds of trillions for diff = 18). This is realistic for high-supply meme tokens or protocol tokens with quadrillion-scale supplies.
- No privileged role is needed; any token holder can initiate the transfer on the origin chain.
- The attacker loses their own tokens in the process, making this a self-harm scenario rather than a theft — but the impact (permanent fund freeze) is still Critical under the allowed scope.

### Recommendation

Replace the bare multiplication with a checked variant and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.checked_pow(diff_decimals)
        .near_expect(BridgeError::AmountOverflow);
    amount.checked_mul(multiplier)
        .near_expect(BridgeError::AmountOverflow)
}
```

Additionally, enforce an upper bound on `origin_decimals - decimals` when registering tokens in `add_token` to prevent extreme multipliers from being stored in the first place.

### Proof of Concept

1. Deploy or register a token with `origin_decimals = 24`, `decimals = 6` (`diff = 18`).
2. On the EVM origin chain, hold `u128::MAX * 10 + 9` tokens (a valid `uint256` balance for a high-supply token).
3. Call `initTransfer` on the EVM bridge. The bridge normalizes: `(u128::MAX * 10 + 9) / 10 = u128::MAX`. The `InitTransfer` event is emitted with `amount = u128::MAX` (fits in `uint128`).
4. A relayer submits the proof to NEAR's `fin_transfer`.
5. `fin_transfer_callback` calls `denormalize_amount(u128::MAX, Decimals { decimals: 6, origin_decimals: 24 })`.
6. Rust evaluates `u128::MAX * 10^18` — overflow. With `overflow-checks = true`, the NEAR runtime panics.
7. The callback reverts. The user's tokens remain burned/locked on EVM with no recovery path. [1](#0-0) [6](#0-5)
