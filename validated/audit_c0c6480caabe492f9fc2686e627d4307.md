### Title
`ProofOfInclusion.valid()` Tautological Root-Hash Check Allows Forged DataLayer Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary
The `ProofOfInclusion::valid()` method in the DataLayer Merkle proof subsystem contains a tautological final check that makes the function verify only internal chain consistency, not that the proof actually leads to the correct tree root. An attacker can craft a `ProofOfInclusion` with any claimed `node_hash` and any claimed root, pass it through the `Streamable` deserialization boundary, and have `valid()` return `true`, forging inclusion of arbitrary key-value pairs in any claimed DataLayer tree state.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()    // always true
}
``` [1](#0-0) 

After the loop body executes for the last layer, `existing_hash` is set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. The final check compares `existing_hash` against `self.root_hash()`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // same value as existing_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

Both sides of the final comparison resolve to `last.combined_hash`, making `existing_hash == self.root_hash()` a **tautology**. The function never compares the computed root against any externally-supplied expected root. The same tautology holds for the empty-layers case: `existing_hash = self.node_hash` and `root_hash() = self.node_hash`.

`ProofOfInclusion` is a `Streamable` struct exposed through the Python wheel: [3](#0-2) 

It can be deserialized from arbitrary bytes via `from_bytes` / `parse_rust` at the Python boundary: [4](#0-3) 

### Impact Explanation

An attacker can construct a `ProofOfInclusion` byte blob where:
- `node_hash` is any 32-byte value (claiming any leaf is present)
- `layers` are internally consistent (each `combined_hash` = `calculate_internal_hash(prev, side, sibling)`) but lead to an attacker-chosen root hash

`valid()` returns `true` for this forged proof. Any DataLayer verifier that calls only `proof.valid()` — without separately asserting `proof.root_hash() == known_good_root` — will accept the forged inclusion proof, allowing an attacker to prove arbitrary key-value state in any claimed DataLayer tree. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `valid()` method name strongly implies complete proof validity. The Python API exposes `valid()` and `root_hash()` as separate methods with no documentation requiring callers to check both. The fuzz target and internal tests call only `proof.valid()`: [5](#0-4) 

Any downstream consumer (Python full node, DataLayer sync code) that follows the same pattern is vulnerable. The `ProofOfInclusion` struct is fully deserializable from untrusted network bytes, providing a direct attacker-controlled entry path.

### Recommendation

`valid()` must accept an expected root hash parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against externally-supplied root
}
```

The current `valid()` method (or its Python binding) should either be removed or clearly documented as an internal-consistency-only check that is insufficient for security-critical verification.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker-chosen leaf hash (claims this key is in the tree)
fake_node_hash = bytes([0xAA] * 32)

# Attacker-chosen sibling hash
fake_sibling   = bytes([0xBB] * 32)

# Compute a consistent combined_hash (attacker controls the root)
combined = hashlib.sha256(fake_node_hash + fake_sibling).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,          # left
    other_hash=fake_sibling,
    combined_hash=combined,     # attacker-chosen root
)

proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True despite the proof being entirely fabricated
assert proof.valid(), "forged proof accepted"
# proof.root_hash() == combined  (attacker-controlled root)
```

The forged proof passes `valid()` because the loop verifies `calculate_internal_hash(fake_node_hash, 0, fake_sibling) == combined`, which is true by construction, and the final check `existing_hash == self.root_hash()` reduces to `combined == combined`.
