### Title
Reentrancy in `initTransfer` via Malicious ERC20 Enables Unbacked Token Minting on NEAR — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` has no reentrancy guard. For native (non-bridge, non-custom-minter) ERC20 tokens it calls `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` — an external call to a fully user-controlled contract — after only incrementing `currentOriginNonce`. A malicious ERC20 can reenter `initTransfer` during `transferFrom`, causing multiple `InitTransfer` events to be emitted with distinct nonces but without any real token backing. The NEAR side treats each event as an independent, authoritative proof of a locked transfer and mints tokens for each one.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer` handles native ERC20 tokens through the final `else` branch:

```solidity
// OmniBridge.sol lines 381, 406-412
currentOriginNonce += 1;          // ← state update
...
} else {
    IERC20(tokenAddress).safeTransferFrom(   // ← external call to user-controlled contract
        msg.sender,
        address(this),
        amount
    );
}
```

There is no `ReentrancyGuard` or `nonReentrant` modifier anywhere in the EVM contracts (confirmed by grep). The CLAUDE.md security invariant "State before external calls" is satisfied for `finTransfer` (nonce marked used before any external call), but for `initTransfer` the nonce is merely a monotonic counter — incrementing it before the external call only guarantees each reentrant invocation receives a fresh nonce; it does **not** prevent multiple `InitTransfer` events from being emitted without corresponding token custody.

The `logMetadata` function is permissionless:

```solidity
// OmniBridge.sol lines 224-232
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
```

An attacker can register any ERC20 — including a malicious one — causing the NEAR side to deploy a corresponding NEAR token and begin accepting `InitTransfer` events for it.

---

### Impact Explanation

**Critical — unauthorized mint of bridged assets on NEAR.**

Each `InitTransfer` event emitted by the EVM bridge is treated by the NEAR side as proof that `amount` tokens are locked on EVM. If N+1 events are emitted (one original + N reentrant) but only 0 or 1 token units are actually held by the bridge, the NEAR side mints N+1 × `amount` tokens to the attacker's NEAR address. The attacker can then bridge those NEAR tokens to any other chain, extracting real value from the protocol's liquidity.

---

### Likelihood Explanation

**Medium.** The attacker needs to:
1. Deploy a malicious ERC20 (trivial, no privilege required).
2. Call the permissionless `logMetadata` to register it (no privilege required).
3. Wait for the NEAR side to deploy the corresponding token (normal protocol operation).
4. Call `initTransfer` with the malicious token.

No privileged keys, MPC cooperation, or Wormhole guardian compromise is required. The entire path is reachable by an unprivileged bridge user.

---

### Recommendation

Add OpenZeppelin's `ReentrancyGuardUpgradeable` to `OmniBridge` and apply `nonReentrant` to `initTransfer`, `initTransfer1155`, and `finTransfer`. This is the standard mitigation referenced in the original report and is consistent with the existing CEI discipline already applied to `finTransfer`'s nonce marking.

### Proof of Concept

```solidity
contract MaliciousERC20 {
    OmniBridge bridge;
    uint8 depth;

    function transferFrom(address, address, uint256 amount) external returns (bool) {
        if (depth < 3) {
            depth++;
            // Reenter initTransfer — gets a fresh nonce each time
            bridge.initTransfer(address(this), uint128(amount), 0, 0, "near:attacker.near", "");
            depth--;
        }
        return true; // lie: no tokens actually moved
    }
    // name/symbol/decimals return valid strings so logMetadata succeeds
}
```

**Attack steps:**
1. Deploy `MaliciousERC20` pointing at the bridge.
2. Call `bridge.logMetadata(address(malicious))` → NEAR deploys a corresponding NEAR token.
3. Call `bridge.initTransfer(address(malicious), 1e18, 0, 0, "near:attacker.near", "")`.
4. `currentOriginNonce` increments to N; `safeTransferFrom` is called on `MaliciousERC20`.
5. `MaliciousERC20.transferFrom` reenters `initTransfer` 3 times → nonces N+1, N+2, N+3 each emit `InitTransfer`.
6. Unwinding: the original call also emits `InitTransfer` at nonce N.
7. Four `InitTransfer` events with distinct nonces, zero real tokens locked.
8. NEAR side mints 4 × 1e18 tokens to `attacker.near`.
