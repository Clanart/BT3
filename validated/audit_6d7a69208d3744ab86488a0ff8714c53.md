### Title
Legacy EIP-712 `timeout_height` Not Covered by Signature — Attacker Can Strip Transaction Expiry - (File: `ante/cosmos/eip712.go`, `ethereum/eip712/eip712_legacy.go`)

### Summary

The legacy EIP-712 signature path (`LegacyEip712SigVerificationDecorator`) includes `timeout_height` in the Amino sign-bytes that are JSON-encoded, but the EIP-712 typed-data schema (`"Tx"` type) deliberately omits `timeout_height`. Because EIP-712 only hashes fields declared in the type schema, the signature does not commit to `timeout_height`. An unprivileged attacker who observes a signed legacy EIP-712 transaction in the mempool can zero out `timeout_height` without invalidating the signature, making a transaction that the signer intended to expire at a specific block valid indefinitely.

### Finding Description

**Step 1 — `timeout_height` is included in the Amino bytes but excluded from the EIP-712 type schema.**

In `ante/cosmos/eip712.go`, `VerifySignature` reconstructs the Amino sign-bytes with the live `timeout_height` value from the transaction:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← included in the JSON
    legacytx.StdFee{...},
    msgs, tx.GetMemo(),
)
``` [1](#0-0) 

These bytes are then passed to `LegacyWrapTxToTypedData`, which JSON-unmarshals them and builds the EIP-712 typed data using `extractMsgTypes`. That function hard-codes the `"Tx"` type **without** `timeout_height`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
``` [2](#0-1) 

The same omission exists in the non-legacy `"Tx"` type schema used by `createEIP712Types`: [3](#0-2) 

**Step 2 — EIP-712 hashing ignores fields absent from the type schema.**

`apitypes.TypedDataAndHash` only hashes fields declared in the type definition. Because `timeout_height` is not in `"Tx"`, it is silently dropped from the hash regardless of its value. The resulting `sigHash` is identical whether `timeout_height` is `0` or `10000`. [4](#0-3) 

**Step 3 — The ante handler chain enforces `timeout_height` from the *live* transaction, not from the signed data.**

`newLegacyCosmosAnteHandlerEip712` places `authante.NewTxTimeoutHeightDecorator()` before `LegacyEip712SigVerificationDecorator`:

```go
authante.NewTxTimeoutHeightDecorator(),
...
cosmos.NewLegacyEip712SigVerificationDecorator(...)
``` [5](#0-4) 

`TxTimeoutHeightDecorator` reads `tx.GetTimeoutHeight()` from the *submitted* transaction. If an attacker zeroes that field, the decorator passes (height 0 = no expiry), and the signature check also passes (hash is unchanged).

**Step 4 — The legacy path is still reachable.**

The ante router dispatches to `newLegacyCosmosAnteHandlerEip712` for any transaction carrying the `ExtensionOptionsWeb3Tx` extension option:

```go
case "/ethermint.types.v1.ExtensionOptionsWeb3Tx":
    anteHandler = newLegacyCosmosAnteHandlerEip712(ctx, options, ...)
``` [6](#0-5) 

### Impact Explanation

A user who signs a legacy EIP-712 Cosmos transaction with `timeout_height = N` (intending it to expire at block N) provides a signature that does not commit to that field. Any party who observes the signed transaction bytes — a relayer, a mempool participant, or anyone the user shared the bytes with — can modify `timeout_height` to `0` in the `TxBody`, rebroadcast the transaction, and have it accepted at any future block height. The user's conditional authorization (valid only until block N) is silently stripped. This is an EIP-712 authorization bypass enabling unauthorized execution of a transaction the signer intended to expire.

### Likelihood Explanation

The legacy EIP-712 path is still active and reachable by any unprivileged sender. Transactions are visible in the mempool. The modification requires only zeroing one protobuf field in `TxBody`; no cryptographic material is needed. Any relayer or mempool observer can perform this attack.

### Recommendation

Include `timeout_height` in the EIP-712 `"Tx"` type schema in both the legacy and non-legacy paths, and populate it from the Amino sign-bytes JSON. Alternatively, reject any legacy EIP-712 transaction whose `timeout_height` is non-zero (mirroring the protection already applied in `decodeProtobufSignDoc` for the Protobuf path):

```go
// ethereum/eip712/eip712_legacy.go — extractMsgTypes
"Tx": {
    {Name: "account_number",  Type: "string"},
    {Name: "chain_id",        Type: "string"},
    {Name: "fee",             Type: "Fee"},
    {Name: "memo",            Type: "string"},
    {Name: "msgs",            Type: "Msg[]"},
    {Name: "sequence",        Type: "string"},
    {Name: "timeout_height",  Type: "string"},  // ← add this
},
```

The same addition is needed in `createEIP712Types` (`ethereum/eip712/types.go`). The `VerifySignature` function in `ante/cosmos/eip712.go` already passes `timeout_height` to `StdSignBytes`, so no change is needed there once the type schema is fixed.
