### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS program updates a contract's class hash in the state without verifying that the new class hash corresponds to a previously declared class. This is an acknowledged missing check (marked with a TODO comment). A malicious or buggy contract can call `replace_class` with an arbitrary, undeclared class hash, permanently rendering the contract uncallable and freezing all funds held within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested new class hash directly from the syscall request and writes it into `contract_state_changes` with no validation that the hash exists in the set of declared compiled classes:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The OS's compiled class facts are guessed at the start of block execution in `get_os_global_context` via `guess_compiled_class_facts()` and validated post-execution via `validate_compiled_class_facts_post_execution`. [2](#0-1) 

If a contract's class hash is set to a value that has no corresponding compiled class, the sequencer cannot include any transaction that calls that contract in any future block — because it cannot provide a valid compiled class fact for the hash, and the proof would be invalid. The contract becomes permanently uncallable.

This is structurally identical to the H-4 pattern: a check that should gate a state mutation (verifying the class is declared, analogous to verifying `validSignerCount <= maxSigners`) is absent from the syscall handler, allowing the state to enter an invalid configuration that causes all future interactions to fail.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value:
- The sequencer cannot produce a valid OS proof for any block that includes a call to that contract.
- The sequencer is forced to exclude all transactions targeting that contract indefinitely.
- Any ERC-20 balances, NFTs, or other assets held in the contract's storage are permanently inaccessible.
- There is no recovery path: `maxSigners` in H-4 was immutable; here, the class hash is written into the Merkle state tree and there is no OS-level mechanism to forcibly overwrite it without a valid transaction from the contract itself — which is impossible if the class is invalid.

---

### Likelihood Explanation

**Medium.**

The attack requires deploying a contract that calls `replace_class` with an arbitrary felt value (e.g., `0x1`). This is a standard user action — any contract deployer can do it. A realistic attack scenario:

1. Attacker deploys a contract that accepts token deposits from users.
2. After accumulating funds, the contract calls `replace_class(0x1)` (or any undeclared hash).
3. The contract's class hash in the state is now `0x1`, which has no compiled class.
4. All future calls to the contract revert at the proof level; funds are frozen.

In a non-malicious scenario, a buggy contract could accidentally call `replace_class` with a zero or garbage hash, achieving the same outcome.

---

### Recommendation

Add a check inside `execute_replace_class` that verifies the requested `class_hash` exists in `contract_class_changes` (i.e., has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for the given `class_hash` and assert the result is non-zero before updating the contract state. This mirrors the pattern used in `execute_declare_transaction`, which enforces `assert_not_zero(compiled_class_hash)` and uses `prev_value=0` to guarantee uniqueness. [3](#0-2) 

---

### Proof of Concept

1. Deploy a contract `VaultWithTrap` that:
   - Accepts ETH/token deposits via a `deposit()` entry point.
   - Exposes a `trap()` entry point that calls `replace_class(0xdeadbeef)`.

2. Users deposit funds into `VaultWithTrap`.

3. Attacker (or the contract itself, triggered by a condition) calls `trap()`.

4. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`. No validation is performed. The state is updated: `VaultWithTrap.class_hash = 0xdeadbeef`.

5. In the next block, any transaction targeting `VaultWithTrap` requires the sequencer to supply a compiled class for `0xdeadbeef`. No such class exists.

6. The sequencer cannot include any call to `VaultWithTrap` in any provable block. All deposited funds are permanently frozen.
