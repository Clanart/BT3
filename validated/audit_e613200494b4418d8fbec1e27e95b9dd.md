### Title
`starknet_estimateFee` Returns `FeeEstimation` With Missing `l2_gas_consumed` Field, Making `overall_fee` Irreconcilable From Breakdown - (`File: crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field correctly includes L2 gas costs (it is taken directly from `receipt.fee`, which accounts for all three gas dimensions), but the breakdown fields (`gas_consumed`, `data_gas_consumed`) cover only L1 gas and L1 data gas. The OpenRPC spec and the struct's own doc-comment both describe `overall_fee` as equalling `gas_consumed * gas_price + data_gas_consumed * data_gas_price`, which is arithmetically wrong for any V3 transaction that consumes L2 gas. Any caller who reconstructs the fee from the breakdown fields receives a value that is lower than the authoritative `overall_fee`, and the `virtual_snos_prover` documentation explicitly instructs users to read `l2_gas.max_amount` from `starknet_estimateFee`, a value the endpoint does not return.

### Finding Description

`tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs` builds the `FeeEstimation` response:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                   // price present …
    overall_fee: tx_execution_output.execution_info.receipt.fee, // … but l2_gas_consumed absent
    unit: tx_execution_output.price_unit,
})
``` [1](#0-0) 

`receipt.fee` is computed by `get_fee_by_gas_vector`, which sums all three gas dimensions:

```
fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)
``` [2](#0-1) 

So `overall_fee` is correct, but the breakdown is incomplete. The struct's own comment acknowledges this with a TODO:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
``` [3](#0-2) 

The OpenRPC schema reinforces the wrong invariant by describing `overall_fee` as:

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price" [4](#0-3) 

The `virtual_snos_prover` documentation compounds the problem by instructing callers to use the value returned by `starknet_estimateFee` as `l2_gas.max_amount`, a value the endpoint never returns: [5](#0-4) 

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction that consumes L2 gas receives an authoritative-looking `FeeEstimation` where:

1. `overall_fee` is correct but cannot be reconstructed from the exposed breakdown fields.
2. The description of `overall_fee` in both the struct comment and the OpenRPC spec is arithmetically wrong.
3. `l2_gas_consumed` — the value the prover documentation says to read from this endpoint — is absent.

A caller who trusts the breakdown formula (`gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`) to set resource bounds will compute a value strictly lower than the actual fee, causing their transaction to revert with `MaxGasAmountExceeded` or `InsufficientFeeTokenBalance` at execution time. This matches the allowed impact: **High — RPC fee estimation returns an authoritative-looking wrong value**.

### Likelihood Explanation

Every V3 transaction on a post-0.13.3 network (i.e., `AllResources` mode) consumes L2 gas. The `FeeEstimation` struct is returned by both `starknet_estimateFee` and `starknet_simulateTransactions`. Any unprivileged caller who follows the documented formula or the prover's own instructions will encounter the discrepancy.

### Recommendation

Add `l2_gas_consumed: gas_vector.l2_gas.0.into()` to the `FeeEstimation` struct and to `tx_execution_output_to_fee_estimation`. Update the struct doc-comment and the OpenRPC `FEE_ESTIMATE` schema description to include the L2 gas term. Remove or correct the `virtual_snos_prover` documentation that refers to reading `l2_gas.max_amount` from `starknet_estimateFee` until the field is present.

### Proof of Concept

1. Submit a V3 invoke transaction to `starknet_estimateFee` on a network running post-0.13.3 versioned constants.
2. Observe the returned `FeeEstimation`: `l2_gas_price` is non-zero, `overall_fee` is non-zero, but `l2_gas_consumed` is absent.
3. Compute `reconstructed_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`.
4. Observe `reconstructed_fee < overall_fee` by exactly `l2_gas_consumed * l2_gas_price`.
5. Submit the same transaction with `l2_gas.max_amount` set to `reconstructed_fee / l2_gas_price`; the transaction reverts with `MaxGasAmountExceeded` even though the estimate appeared sufficient. [6](#0-5) [7](#0-6)

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
