### Title
Uninitialized `locked_tokens` Entry Silently Bypasses Collateralization Check on First Use — (`near/omni-bridge/src/token_lock.rs`)

### Summary
Both `lock_tokens` and `unlock_tokens` in the NEAR bridge silently return `LockAction::Unchanged` when no entry exists in the `locked_tokens` map for a given `(chain_kind, token_id)` pair. Because `locked_tokens` is never auto-initialized when a token is registered, the `ERR_INSUFFICIENT_LOCKED_TOKENS` guard is completely bypassed for any token/chain pair that has not been explicitly seeded by an admin via `set_locked_tokens`. This breaks the bridge's collateralization accounting invariant for those pairs.

### Finding Description

The only way to create an entry in `locked_tokens` is through the privileged `set_locked_tokens` function, which requires `Role::DAO` or `Role::TokenLockController`:

```rust
#[access_control_any(roles(Role::DAO, Role::TokenLockController))]
pub fn set_locked_tokens(&mut self, args: Vec<SetLockedTokenArgs>) { ... }
``` [3](#0-2) 

There is no automatic initialization of a `locked_tokens` entry when a new NEAR-native token is registered or when the first `init_transfer` for a new token/chain pair is processed. The `lock_tokens_if_needed` and `unlock_tokens_if_needed` wrappers called throughout the bridge flow both delegate to these silent-no-op functions: [4](#0-3) 

These wrappers are called in every critical transfer path:

- **`init_transfer_internal`** (outbound lock): `lock_tokens_if_needed(destination_chain, token_id, amount)` — silently skipped if entry absent. [5](#0-4) 

- **`process_fin_transfer_to_near`** (inbound unlock): `unlock_tokens_if_needed(origin_chain, token, amount)` — silently skipped, bypassing the collateral check. [6](#0-5) 

- **`process_fin_transfer_to_other_chain`** (cross-chain rebalance): both unlock on origin and lock on destination are silently skipped. [7](#0-6) 

The existing unit test `test_fin_transfer_callback_near_fails_without_locked_tokens` only covers the case where the entry **exists** but holds an insufficient value (seeded with `&1`). It does **not** cover the case where the entry is entirely absent — which is the default state for any newly supported token/chain pair: [8](#0-7) 

### Impact Explanation

The `locked_tokens` map is the bridge's on-chain collateralization ledger for NEAR-native tokens bridged to non-NEAR chains. It is supposed to enforce that the bridge never releases more tokens than were locked. When the entry is absent:

1. **Outbound (init_transfer)**: Tokens are accepted and locked in the bridge contract's actual token balance, but no accounting entry is updated. The bridge's collateralization ledger diverges from reality from the very first transfer.

2. **Inbound (fin_transfer)**: The `ERR_INSUFFICIENT_LOCKED_TOKENS` guard — the last line of defense against over-release — is completely bypassed. The bridge releases tokens to the recipient without verifying that any collateral was ever recorded.

3. **Error-path revert**: `revert_lock_actions` is a no-op for `LockAction::Unchanged`, so failed fin-transfers also do not restore the ledger. [9](#0-8) 

The net result is that the bridge's collateralization invariant — "tokens released on NEAR ≤ tokens locked on the source chain" — is unenforceable at the contract level for any token/chain pair that was never seeded. This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**.

### Likelihood Explanation

The `locked_tokens` map has no automatic initialization path. Any NEAR-native token added to the bridge (via `token_address_to_id` / `token_id_to_address` registration) is immediately usable for `init_transfer` and `fin_transfer` without requiring a prior `set_locked_tokens` call. There is no guard in `init_transfer_internal` or `fin_transfer_callback` that checks whether the entry exists before proceeding. An operator who registers a new token but omits the `set_locked_tokens` step — a realistic operational oversight — leaves the entire collateralization layer non-functional for that token. Any bridge user transacting with that token then operates in an untracked state.

### Recommendation

Initialize the `locked_tokens` entry to `0` automatically when a new token/chain pair is first encountered in `lock_tokens`, rather than silently returning `Unchanged`:

```rust
fn lock_tokens(&mut self, chain_kind: ChainKind, token_id: &AccountId, amount: u128) -> LockAction {
    let key = (chain_kind, token_id.clone());
    let current_amount = self.locked_tokens.get(&key).unwrap_or(0); // auto-init to 0
    let new_amount = current_amount
        .checked_add(amount)
        .near_expect(TokenLockError::LockedTokensOverflow);
    self.locked_tokens.insert(&key, &new_amount);
    LockAction::Locked { chain_kind, token_id: token_id.clone(), amount }
}
```

For `unlock_tokens`, the absence of an entry should be treated as a zero balance and trigger the `ERR_INSUFFICIENT_LOCKED_TOKENS` panic rather than silently returning `Unchanged`:

```rust
fn unlock_tokens(&mut self, chain_kind: ChainKind, token_id: &AccountId, amount: u128) -> LockAction {
    let key = (chain_kind, token_id.clone());
    let available = self.locked_tokens.get(&key).unwrap_or(0); // treat absent as 0
    require!(available >= amount, TokenLockError::InsufficientLockedTokens.as_ref());
    ...
}
```

Alternatively, add an explicit check in `init_transfer_internal` and `fin_transfer_callback` that panics if the `locked_tokens` entry is absent for a non-origin-chain transfer, forcing operators to initialize it before the token can be bridged.

### Proof of Concept

1. Admin registers NEAR-native token `usdc.near` for bridging to ETH, but **does not** call `set_locked_tokens({chain_kind: Eth, token_id: "usdc.near", amount: 0})`.
2. User calls `ft_transfer_call` → `ft_on_transfer` → `init_transfer_internal`. Inside, `lock_tokens_if_needed(Eth, "usdc.near", 1000)` is called. Since `locked_tokens.get(&(Eth, "usdc.near"))` returns `None`, `lock_tokens` returns `Unchanged`. No entry is created. Tokens are held in the bridge contract's actual balance, but the ledger records nothing.
3. A trusted relayer submits `fin_transfer` with a valid proof for a 1000 USDC transfer from ETH to NEAR. Inside `process_fin_transfer_to_near`, `unlock_tokens_if_needed(Eth, "usdc.near", 1000)` is called. Since `locked_tokens.get(&(Eth, "usdc.near"))` returns `None`, `unlock_tokens` returns `Unchanged` — the `ERR_INSUFFICIENT_LOCKED_TOKENS` check is never reached. Tokens are released to the recipient.
4. The bridge's collateralization ledger remains at `None` (untracked) for this token/chain pair for all future transfers. Any subsequent `fin_transfer` for any amount also bypasses the guard, making the ledger permanently non-functional for this pair. [10](#0-9) [11](#0-10) [6](#0-5)

    fn lock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(current_amount) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        let new_amount = current_amount
            .checked_add(amount)
            .near_expect(TokenLockError::LockedTokensOverflow);

        self.locked_tokens.insert(&key, &new_amount);

        LockAction::Locked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
