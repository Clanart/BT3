### Title
Missing Declared-Class Validation in `replace_class` Syscall vs. Strict Validation at Declaration — (File: `execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` function in `syscall_impls.cairo` applies no validation that the new class hash corresponds to a previously declared class, while `execute_declare_transaction` in `transaction_impls.cairo` enforces strict validation (non-zero compiled class hash, valid Sierra class hash, single-declaration enforcement). This inconsistency is the direct analog of the external report's pattern: stricter bounds at initialization than at update. Any unprivileged contract can call `replace_class` with class hash `0` or any undeclared hash, permanently bricking itself and freezing any funds it holds.

---

### Finding Description

**At declaration time**, `execute_declare_transaction` enforces three hard constraints:

1. The class hash must be a valid Sierra class hash, verified by `finalize_class_hash`: [1](#0-0) 

2. The compiled class hash must be non-zero: [2](#0-1) 

3. A class can only be declared once (`prev_value=0` in `dict_update`): [3](#0-2) 

**At update time**, `execute_replace_class` applies none of these constraints. After reducing gas, it reads the current state entry and unconditionally writes whatever `request.class_hash` is — including `0` or any undeclared hash — into the contract's state. The only acknowledgment of the missing check is a deferred TODO comment: [4](#0-3) 

The critical lines are:

```cairo
let class_hash = request.class_hash;
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
...
tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(...);
```

No `assert_not_zero(class_hash)`, no lookup into `contract_class_changes` to confirm the hash is declared.

**Downstream failure path**: When any subsequent call targets the bricked contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // = 0 or undeclared
);
let (compiled_class_fact: CompiledClassFact*) = find_element(
    ...
    key=compiled_class_hash,           // = 0 (default dict value)
);
```

`find_element` requires the element to exist; if `compiled_class_hash = 0` is not in the compiled class facts bundle, the hint cannot provide a valid index and the OS proof fails. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract that calls `replace_class(0)` has its class hash set to `0` in the global state. All future calls to that contract fail at the OS level because no compiled class with hash `0` exists. Any ERC-20 tokens, ETH, or other assets held by the contract become permanently inaccessible. The state transition is committed and irreversible.

---

### Likelihood Explanation

The attack requires only deploying a contract and invoking a function that calls the `replace_class` syscall — both are permissionless operations available to any unprivileged user. The `execute_replace_class` function is reachable from any Cairo 1 contract via the standard syscall interface. No privileged role, leaked key, or external dependency is required. The TODO comment confirms the missing check is a known gap in the current OS code, not a deliberate design choice.

---

### Recommendation

**Short term:** Add an explicit check in `execute_replace_class` that the new `class_hash` is non-zero and that it exists in `contract_class_changes` with a non-zero compiled class hash, mirroring the `assert_not_zero(compiled_class_hash)` guard already present in `execute_declare_transaction`.

**Long term:** Remove the TODO comment and add OS-level tests that attempt `replace_class` with `0` and with an undeclared hash, asserting that both are rejected.

---

### Proof of Concept

1. Deploy a contract `MaliciousVault` that accepts user deposits and exposes a `drain()` function.
2. Users deposit funds into `MaliciousVault`.
3. The attacker calls `drain()`, which internally calls `replace_class(0)` via the `REPLACE_CLASS` syscall.
4. The OS accepts the call — `execute_replace_class` writes `class_hash=0` into the contract's `StateEntry` with no validation.
5. All subsequent calls to `MaliciousVault` reach `execute_entry_point`, which performs `dict_read(key=0)` → `compiled_class_hash=0`, then `find_element(key=0)` → proof failure / permanent revert.
6. Users' deposited funds are permanently frozen; no withdrawal function can ever execute successfully.

func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

    // Reduce gas.
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );

    assert [revert_log] = RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=state_entry.class_hash);
    let revert_log = &revert_log[1];

    return ();
}
