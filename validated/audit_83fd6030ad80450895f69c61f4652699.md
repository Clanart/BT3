### Title
`ProofOfInclusion::valid()` Final Root-Hash Check Is a Tautology — Forged Proofs Always Pass — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` is the sole API for verifying DataLayer Merkle inclusion proofs. Its final correctness check compares `existing_hash` against `self.root_hash()`, but `root_hash()` is derived entirely from the proof's own fields — not from any external trusted root. After the loop, `existing_hash` is always equal to `self.root_hash()` by construction, making the final comparison a tautology. Any attacker who can supply a `ProofOfInclusion` (via the Python or Rust API) can forge a self-consistent proof for any arbitrary `node_hash` and have `valid()` return `true`.

### Finding Description

**Root cause — `valid()` in `proof_of_inclusion.rs`:**

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs

pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- derived from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()   // TAUTOLOGY: both sides == last.combined_hash
}
```

After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `layer.combined_hash` in the same iteration. `self.root_hash()` returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` — it compares the proof's own field against itself.

**Exploit path:**

An attacker constructs a `ProofOfInclusion` from scratch (or via `from_bytes()` / `from_bytes_unchecked()`, both exposed in the Python binding):

1. Choose any arbitrary `node_hash` (e.g., claiming key `K` maps to value `V`).
2. Choose any `other_hash` and `other_hash_side` for each layer.
3. Compute each `combined_hash = calculate_internal_hash(prev_hash, side, other_hash)` — this is public, deterministic arithmetic.
4. Call `proof.valid()` → returns `true`.

The proof is internally self-consistent but anchored to no real tree root. The `valid()` call provides no assurance that the claimed `node_hash` is actually present in any committed tree.

### Impact Explanation

This is a **High** severity finding matching the allowed impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

Any consumer of the `ProofOfInclusion` API — including Python DataLayer clients — that calls `proof.valid()` and trusts the result is vulnerable to accepting forged state. An attacker can prove that any key/value pair is included in any DataLayer store without possessing a valid tree or root commitment. This enables:

- Forged inclusion proofs for arbitrary key/value pairs.
- Bypassing DataLayer state integrity checks in any system that relies on `valid()` as its verification gate.

### Likelihood Explanation

The `valid()` method is the only verification primitive exposed for `ProofOfInclusion`. It is exposed as a Python binding (`py_valid()`), is used in the fuzz target as the correctness oracle, and is the method callers are expected to use. Any external verifier that receives a `ProofOfInclusion` over the network or from an untrusted source and calls `valid()` is immediately exploitable. No privileged access is required — only the ability to supply a crafted `ProofOfInclusion` struct.

### Recommendation

`valid()` must accept an externally-supplied trusted root hash and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // compare against EXTERNAL root, not self.root_hash()
}
```

The existing `valid()` method (and its Python binding `py_valid()`) should either be removed or deprecated, since it provides a false sense of security.

### Proof of Concept

**Tautology proof (logical):**

After the loop body executes for the last layer without returning `false`:
- `existing_hash` = `calculated_hash` = `layer.combined_hash` (last layer)
- `self.root_hash()` = `self.layers.last().combined_hash` = same value

Therefore `existing_hash == self.root_hash()` ≡ `last.combined_hash == last.combined_hash` ≡ always `true`.

**Forged proof construction:**

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

# Claim any node_hash we want
fake_node_hash = bytes([0xAA] * 32)
fake_other_hash = bytes([0xBB] * 32)

# Compute combined_hash using the same calculate_internal_hash logic
# (left || right, sorted by side)
combined = hashlib.sha256(fake_node_hash + fake_other_hash).digest()  # simplified

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=fake_other_hash,
    combined_hash=combined,   # self-consistent
)
proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])
assert proof.valid()   # returns True — forged proof accepted
```

The exact hash function used is `calculate_internal_hash` in `blob.rs`, which is public and deterministic, making the forgery fully constructible by any attacker.
