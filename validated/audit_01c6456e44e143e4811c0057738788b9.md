### Title
`validate_virtual_os_state_diff` Applies "No Storage Deletions" Constraint Uniformly to All Storage Updates, Blocking Valid Transaction Proofs — (File: `crates/starknet_transaction_prover/src/running/committer_utils.rs`)

---

### Summary

`validate_virtual_os_state_diff` enforces three constraints on the virtual-OS state diff before committing it to the Patricia trie and generating proof commitment infos. One of those constraints — "No Storage Deletions" — rejects **any** storage update whose value is `Felt::ZERO`. This check is applied unconditionally to every storage entry in the diff, including legitimate zero-value writes produced by ordinary invoke transactions (e.g., resetting a counter, clearing a flag, draining a balance to zero). Because the Starknet OS `execute_storage_write` syscall places no restriction on the written value, and the committer's Patricia-trie implementation handles zero-value leaves correctly, the constraint is overly broad: it blocks proof generation for a class of valid transactions that do not violate any real OS invariant.

---

### Finding Description

`validate_virtual_os_state_diff` is called inside `create_commitment_infos_with_state_changes` immediately after converting the blockifier `StateMaps` to the committer `StateDiff`:

```rust

```

The validation function itself:

```rust
// crates/starknet_transaction_prover/src/running/committer_utils.rs  (lines 112-143)
pub(crate) fn validate_virtual_os_state_diff(state_diff: &StateDiff) -> Result<(), ProofProviderError> {
    // validate no storage deletions.
    for (address, storage_diffs) in &state_diff.storage_updates {
        for (key, value) in storage_diffs {
            if value.0 == Felt::ZERO {          // ← applied to ALL storage updates
                return Err(ProofProviderError::InvalidStateDiff(...));
            }
        }
    }
    // validate no contract deployments (or replaced classes).
    if !state_diff.address_to_class_hash.is_empty() { ... }
    // validate no contract declarations (or compiled class hash updates).
    if !state_diff.class_hash_to_compiled_class_hash.is_empty() { ... }
    Ok(())
}
```

The other two constraints ("no deployments", "no declarations") are correctly scoped to operations that the virtual OS genuinely cannot perform (the virtual block executor only accepts `InvokeTransaction`s). The "no storage deletions" constraint, however, is applied uniformly to **every** `(address, key, value)` triple regardless of whether the zero write originates from a class-deployment side-effect or from a plain user-contract storage reset.

The Starknet OS Cairo implementation of `execute_storage_write` places no restriction on the written value:

```cairo
// crates/apollo_starknet_os_program/src/cairo/.../syscall_impls.cairo  (lines 670-672)
assert [storage_ptr] = DictAccess(
    key=storage_key, prev_value=prev_value, new_value=request.value   // value may be zero
);
```

The committer's `actual_storage_updates` likewise passes zero values through without filtering:

```rust
// crates/starknet_committer/src/block_committer/input.rs  (lines 279-282)
inner_updates.iter().map(|(key, value)| (key.into(), *value)).collect()
```

So the Patricia-trie commit path can handle zero-value leaves (leaf deletion), but `validate_virtual_os_state_diff` aborts before `commit_state_diff` is ever reached, producing a `ProofProviderError::InvalidStateDiff` for any transaction that writes zero to storage.

The analog to the Size M-10 bug is exact:

| Size `withdraw()` | Sequencer `validate_virtual_os_state_diff` |
|---|---|
| `validateUserIsNotBelowOpeningLimitBorrowCR` applied after **both** collateral-token and borrow-token withdrawals | "No storage deletions" applied to **all** storage updates, including legitimate zero-value writes |
| Borrow-token withdrawal does not affect collateral ratio → check is irrelevant | Zero-value write by a user contract does not violate any OS invariant → check is irrelevant |
| User cannot withdraw already-borrowed funds | Prover cannot generate a proof for a valid invoke transaction |

### Impact Explanation

Any invoke transaction whose execution writes `Felt::ZERO` to at least one storage slot will cause `create_commitment_infos_with_state_changes` to return `Err(ProofProviderError::InvalidStateDiff(...))`. This propagates up through the transaction prover's proof-generation pipeline, making it impossible to produce a valid proof for that transaction. Because the transaction prover is the authoritative service for client-side proving (proof-facts transactions), the affected transaction cannot be submitted with proof facts and is effectively rejected before sequencing — matching the **High** impact tier: *"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."*

---

### Likelihood Explanation

Writing zero to a storage slot is a routine Starknet operation: resetting a counter, clearing an approval, draining a token balance to zero, or any contract that uses storage as a boolean flag. The likelihood that a real invoke transaction in the client-side proving path produces at least one zero-value storage write is high.

---

### Recommendation

Remove the "no storage deletions" constraint from `validate_virtual_os_state_diff`, or scope it only to the specific sub-class of storage updates that the virtual OS genuinely cannot handle (if any). The other two constraints ("no deployments", "no declarations") remain correct because the virtual block executor only accepts invoke transactions and will never produce those state-diff entries.
