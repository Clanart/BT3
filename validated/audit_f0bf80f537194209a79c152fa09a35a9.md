### Title
`get_class(pending)` Bypasses Parent-Hash Freshness Guard Present in All Other Pending-State RPC Paths — (`File: crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

Every pending-state RPC accessor (`get_storage_at`, `get_nonce`, `estimate_fee`, `simulate_transactions`, `get_state_update`, `trace_block_transactions`) routes through `read_pending_data`, which validates that the cached pending block's `parent_block_hash` still equals the latest committed block hash before returning any pending-state data. `get_class` with `BlockId::Tag(Tag::Pending)` reads `pending_classes` directly, skipping this guard entirely. During the race window between a new block being committed to storage and the pending-sync task clearing `pending_classes`, `get_class(pending)` returns a class that was declared in a now-superseded pending block — a class that does not exist in either the committed state or the actual current pending state.

---

### Finding Description

**The freshness guard — `read_pending_data`**

`read_pending_data` (lines 1553–1595) enforces the invariant:

```rust
if pending_data.block.parent_block_hash() == latest_header.block_hash {
    Ok((*pending_data).clone())
} else {
    // Return empty pending data — no state diff, no transactions
    Ok(PendingData { state_update: ClientPendingStateUpdate { state_diff: Default::default(), .. }, .. })
}
```

When the pending block's parent hash no longer matches the latest committed block, it returns an empty pending state, effectively treating the pending layer as non-existent. [1](#0-0) 

**Functions that use the guard (consistent)**

- `get_storage_at`: calls `read_pending_data` before reading `storage_diffs`. [2](#0-1) 
- `get_nonce`: calls `read_pending_data` before reading `nonces`. [3](#0-2) 
- `estimate_fee`: calls `read_pending_data` to build `maybe_pending_data`. [4](#0-3) 
- `simulate_transactions`: same pattern. [5](#0-4) 

**`get_class` — the missing guard**

```rust
async fn get_class(&self, block_id: BlockId, class_hash: ClassHash) -> RpcResult<GatewayContractClass> {
    let block_id = if let BlockId::Tag(Tag::Pending) = block_id {
        let maybe_class = &self.pending_classes.read().await.get_class(class_hash);
        if let Some(class) = maybe_class {
            return class.clone().try_into().map_err(internal_server_error);
        } else {
            BlockId::Tag(Tag::Latest)
        }
    } else { block_id };
    // ... storage lookup
```

`self.pending_classes` is read directly with no call to `read_pending_data` and no check of `parent_block_hash`. [6](#0-5) 

**The `pending_classes` clearing mechanism and the race window**

`pending_classes` is cleared in the pending-sync loop only when the sync task detects a parent-hash change:

```rust
if current_pending_parent_hash != new_pending_parent_hash {
    pending_classes.write().await.clear();
}
``` [7](#0-6) 

This clearing is asynchronous and runs in a separate task. Between the moment a new block is committed to storage (advancing `latest_header`) and the moment the pending-sync task wakes, detects the parent-hash mismatch, and calls `pending_classes.clear()`, there is a window during which:

- `read_pending_data` already returns **empty** pending state (parent hash mismatch detected synchronously against storage)
- `pending_classes` still holds classes from the **superseded** pending block

During this window, `get_class(pending, stale_class_hash)` returns a class that:
1. Does not exist in the committed state (it was only in the pending block that was just finalized or replaced)
2. Does not exist in the actual current pending state (the new pending block has not yet declared it)

The caller receives an authoritative-looking `GatewayContractClass` response for a class that is not part of any valid chain state at the time of the call.

---

### Impact Explanation

`get_class(pending)` returns a wrong, authoritative-looking value from the RPC pending view. A client querying the class definition of a contract declared in a now-superseded pending block will receive a non-empty class body instead of `CLASS_HASH_NOT_FOUND`. This is a direct instance of: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

Downstream effects include:
- A client that uses the returned class to construct a `declare` or `invoke` transaction against the pending state will build on a phantom class, leading to transaction failure or incorrect fee estimation.
- `get_class_at(pending, address)` calls `get_class_hash_at` (which uses `read_pending_data` and would return the committed class hash) and then calls `get_class(block_id, class_hash)` — the two sub-calls now operate on inconsistent pending states, potentially returning a class body that does not match the class hash returned for the same address.

---

### Likelihood Explanation

The race window is bounded by the polling interval of the pending-sync task. In a live node this is typically on the order of seconds. Any RPC call to `get_class(pending)` that arrives after a new block is committed but before the sync task clears `pending_classes` will observe the stale value. This is a normal operating condition, not an adversarial one — no special privileges or network position are required.

---

### Recommendation

Apply the same parent-hash freshness check to `get_class` that all other pending-state accessors use. The simplest fix is to call `read_pending_data` and only consult `pending_classes` when the pending data is fresh:

```rust
async fn get_class(&self, block_id: BlockId, class_hash: ClassHash) -> RpcResult<GatewayContractClass> {
    let block_id = if let BlockId::Tag(Tag::Pending) = block_id {
        let txn = self.storage_reader.begin_ro_txn().map_err(internal_server_error)?;
        // Validate pending data is fresh before consulting pending_classes.
        let pending_data = read_pending_data(&self.pending_data, &txn).await?;
        // Only use pending_classes if the pending block is actually current.
        if pending_data.block.parent_block_hash()
            == txn.get_block_header(txn.get_header_marker()?.prev().unwrap())
                .map_err(internal_server_error)?.block_hash
        {
            let maybe_class = &self.pending_classes.read().await.get_class(class_hash);
            if let Some(class) = maybe_class {
                return class.clone().try_into().map_err(internal_server_error);
            }
        }
        BlockId::Tag(Tag::Latest)
    } else { block_id };
    // ... storage lookup unchanged
```

Alternatively, `pending_classes` could be cleared atomically with the `pending_data` update (under the same lock) rather than in a separate async task, eliminating the race window entirely.

---

### Proof of Concept

1. Node is at committed block N with hash `H_N`. Pending block has `parent_block_hash = H_N`. A class `C` is declared in the pending block; `pending_classes` contains `C`.
2. Block N+1 is committed to storage. `latest_header.block_hash` is now `H_{N+1}`. The pending-sync task has not yet run.
3. Client calls `starknet_getClass(block_id="pending", class_hash=C)`.
4. `get_class` reads `self.pending_classes` directly — no parent-hash check — finds `C`, and returns its full class body.
5. Simultaneously, `starknet_getNonce(block_id="pending", contract_address=X)` calls `read_pending_data`, detects `parent_block_hash H_N ≠ H_{N+1}`, and returns the committed nonce (empty pending overlay).
6. The two RPC calls return data from inconsistent states: `get_class` reflects the old pending block; `get_nonce` reflects the committed state. `C` does not exist in either the committed state or the new pending block. [6](#0-5) [1](#0-0) [8](#0-7)

async fn read_pending_data<Mode: TransactionKind>(
    pending_data: &Arc<RwLock<PendingData>>,
    txn: &StorageTxn<'_, Mode>,
) -> RpcResult<PendingData> {
    let latest_header = match get_latest_block_number(txn)? {
        Some(latest_block_number) => get_block_header_by_number(txn, latest_block_number)?,
        None => starknet_api::block::BlockHeader {
            // TODO(Shahak): Consider adding genesis hash to the config to support chains that have
            // different genesis hash.
            block_header_without_hash: BlockHeaderWithoutHash {
                parent_hash: BlockHash::GENESIS_PARENT_HASH,
                ..Default::default()
            },
            ..Default::default()
        },
    };
    let pending_data = &pending_data.read().await;
    if pending_data.block.parent_block_hash() == latest_header.block_hash {
        Ok((*pending_data).clone())
    } else {
        Ok(PendingData {
            block: PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock {
                parent_block_hash: latest_header.block_hash,
                eth_l1_gas_price: latest_header.block_header_without_hash.l1_gas_price.price_in_wei,
                strk_l1_gas_price: latest_header
                    .block_header_without_hash
                    .l1_gas_price
                    .price_in_fri,
                timestamp: latest_header.block_header_without_hash.timestamp,
                sequencer_address: latest_header.block_header_without_hash.sequencer,
                starknet_version: latest_header
                    .block_header_without_hash
                    .starknet_version
                    .to_string(),
                ..Default::default()
            }),
            state_update: ClientPendingStateUpdate {
                old_root: latest_header.block_header_without_hash.state_root,
                state_diff: Default::default(),
            },
        })
    }
}

```rust
async fn get_pending_data<TPendingSource: PendingSourceTrait + Sync + Send + 'static>(
    latest_block_hash: BlockHash,
    pending_source: Arc<TPendingSource>,
    pending_data: Arc<RwLock<PendingData>>,
    pending_classes: Arc<RwLock<PendingClasses>>,
    sleep_duration: Duration,
) -> Result<PendingSyncTaskResult, StateSyncError> {
    tokio::time::sleep(sleep_duration).await;

    let new_pending_data = pending_source.get_pending_data().await?;

    // In Starknet, if there's no pending block then the latest block is returned. We prefer to
    // treat this case as if the pending block is an empty block on top of the latest block.
    // We distinguish this case by looking if the block_hash field is present.
    let new_pending_parent_hash =
        new_pending_data.block.block_hash().unwrap_or(new_pending_data.block.parent_block_hash());
    if new_pending_parent_hash != latest_block_hash {
        // TODO(shahak): If block_hash is present, consider writing the pending data here so that
        // the pending data will be available until the node syncs on the new block.
        debug!("A new block was found. Stopping pending sync.");
        return Ok(PendingSyncTaskResult::PendingSyncFinished);
    };

    let (current_pending_num_transactions, current_pending_parent_hash) = {
        let pending_block = &pending_data.read().await.block;
        (
            pending_block.transactions().len(),
            pending_block.block_hash().unwrap_or(pending_block.parent_block_hash()),
        )
    };
    let is_new_pending_data_more_advanced = current_pending_parent_hash != new_pending_parent_hash
        || new_pending_data.block.transactions().len() > current_pending_num_transactions;
    if is_new_pending_data_more_advanced {
        debug!("Received new pending data.");
        trace!("Pending data: {new_pending_data:#?}.");
        if current_pending_parent_hash != new_pending_parent_hash {
            pending_classes.write().await.clear();
        }
        *pending_data.write().await = new_pending_data;
        Ok(PendingSyncTaskResult::DownloadedNewPendingData)
    } else {
        debug!("Pending block wasn't updated. Waiting for pending block to be updated.");
        Ok(PendingSyncTaskResult::DownloadedOldPendingData)
    }
}
