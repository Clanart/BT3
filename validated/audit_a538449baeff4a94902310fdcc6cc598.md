### Title
Missing Declared-Class Validation in `replace_class` Syscall Allows Permanent Freezing of Contract Funds - (File: execution/syscall_impls.cairo)

### Summary
The `execute_replace_class` function in the StarkNet OS accepts an arbitrary `class_hash` value from the syscall request and commits it directly to contract state without verifying that the hash corresponds to a previously declared class. A malicious contract can exploit this to set its own class hash to an undeclared value, permanently bricking the contract and freezing any funds it holds.

### Finding Description
In `execute_replace_class`, the `class_hash` field from the syscall request is read and written directly into `contract_state_changes` with no validation that the hash exists in the declared-class tree:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The in-code TODO comment at line 898 explicitly acknowledges the missing check. The `class_hash` is a prover/user-controlled felt value that is committed to the global state root without any membership proof against the contract-class Patricia tree. This is structurally identical to the external report's pattern: an unvalidated witness value is committed to a cryptographic commitment, and downstream operations that assume the committed value is well-formed then fail. [1](#0-0) 

The `execute_declare_transaction` path, by contrast, does validate the class hash via `finalize_class_hash`, confirming that the OS knows how to enforce this property but simply omits it in the `replace_class` path. [2](#0-1) 

Once an invalid class hash is committed to state, the contract-state hash is recomputed via `get_contract_state_hash` and folded into the global state root, making the invalid class hash a permanent part of the canonical chain state. [3](#0-2) 

### Impact Explanation
**Critical — Permanent freezing of funds.**

After a successful `replace_class` call with an undeclared hash, the contract's class hash in state points to a class definition that does not exist in the class tree. Any subsequent transaction that attempts to call into that contract will fail at the OS hint level when the prover tries to load the class definition (the hint has no preimage to provide). The sequencer will reject all future calls to the contract. Any ERC-20 tokens, ETH, or other assets held by the contract become permanently inaccessible.

### Likelihood Explanation
**High.** The `replace_class` syscall is callable by any deployed contract with no privileged role required. A malicious contract author can:
1. Deploy a contract that accepts user deposits (e.g., a fake yield vault or escrow).
2. After accumulating victim funds, issue a single `replace_class` syscall with `class_hash = 0x1` (or any undeclared felt).
3. The OS processes the block, commits the invalid class hash to the global state root, and the contract is permanently bricked.

No leaked keys, operator collusion, or network-level attack is required. The attacker only needs to be a standard contract deployer.

### Recommendation
Before committing the new class hash to `contract_state_changes`, verify that `class_hash` exists as a key in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, replace the TODO with a `dict_read` against `contract_class_changes` and assert the returned compiled class hash is non-zero, mirroring the enforcement already present in `execute_declare_transaction`.

### Proof of Concept

1. **Deploy malicious vault contract** — The contract exposes a `deposit` entry point that accepts funds and stores them, and an `exit_scam` entry point that calls `replace_class(class_hash=0xdead)`.

2. **Victims deposit funds** — Users call `deposit`; the contract accumulates balances.

3. **Attacker triggers `replace_class`** — The attacker calls `exit_scam`. Inside `execute_replace_class`, the OS reads `class_hash = 0xdead` from the request and executes:
   ```cairo
   // No validation performed.
   dict_update{dict_ptr=contract_state_changes}(
       key=contract_address,
       prev_value=cast(state_entry, felt),
       new_value=cast(new StateEntry(class_hash=0xdead, ...), felt),
   );
   ```
   The transaction succeeds; the block is proven; the global state root now encodes `class_hash=0xdead` for the vault contract.

4. **Funds frozen** — Any subsequent `withdraw` call targets a contract whose class hash `0xdead` has no entry in the class tree. The OS prover cannot load the class definition; the sequencer rejects all calls to the contract. Victim funds are permanently frozen. [4](#0-3)

func get_contract_state_hash{hash_ptr: HashBuiltin*}(
    class_hash: felt, storage_root: felt, nonce: felt
) -> (hash: felt) {
    const CONTRACT_STATE_HASH_VERSION = 0;
    if (class_hash == UNINITIALIZED_CLASS_HASH) {
        if (storage_root == 0) {
            if (nonce == 0) {
                return (hash=0);
            }
        }
    }

    // Set res = H(H(class_hash, storage_root), nonce).
    let (hash_value) = hash2(class_hash, storage_root);
    let (hash_value) = hash2(hash_value, nonce);

    // Return H(hash_value, CONTRACT_STATE_HASH_VERSION). CONTRACT_STATE_HASH_VERSION must be in the
    // outermost hash to guarantee unique "decoding".
    let (hash) = hash2(hash_value, CONTRACT_STATE_HASH_VERSION);
    return (hash=hash);
}
