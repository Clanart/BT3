### Title
`blocking_check_fail_open` Defaults to `true`, Bypassing the Transaction Proving Security Gate on Any Service Disruption - (File: crates/starknet_transaction_prover/src/config.rs)

---

### Summary

`ProverConfig::blocking_check_fail_open` and `RawServiceConfig::blocking_check_fail_open` both default to `true`. When an operator deploys the transaction prover with a `blocking_check_url` configured (the external compliance/policy gate), any network error, timeout, non-10000 error code, or malformed response from that service causes the prover to silently proceed and return a valid proof — exactly as if the check had passed. The security gate defaults to open, not closed.

---

### Finding Description

The `starknet_transaction_prover` crate exposes a `starknet_proveTransaction` RPC endpoint. Before generating a proof, it optionally consults an external `starknet_checkTransaction` service. The result of that service call drives the `allow` flag in `prove_with_blocking_check`:

```
Blocked      → allow = false  (abort proof)
Allowed      → allow = true   (proceed)
Inconclusive → allow = client.fail_open
Timeout      → allow = client.fail_open
```

`Inconclusive` is returned for all of: network errors, HTTP failures, body-read failures, JSON parse failures, and any non-10000 error code. This covers the entire failure surface of the external service.

The `fail_open` field is set to `true` in both default constructors:

`crates/starknet_transaction_prover/src/config.rs` — `ProverConfig::default()`: [1](#0-0) 

`crates/starknet_transaction_prover/src/server/config.rs` — `RawServiceConfig::default()`: [2](#0-1) 

The `prove_with_blocking_check` logic that consumes this flag: [3](#0-2) 

The `BlockingCheckClient::check_transaction` method returns `Inconclusive` on every failure path (network error, body-read error, parse error, non-10000 error code): [4](#0-3) 

The `blocking_check_url` field defaults to `None`, meaning the blocking check is opt-in. However, once an operator sets `blocking_check_url` to enforce a policy (e.g., compliance screening, sanctions enforcement), the `fail_open: true` default silently undermines that policy on any service disruption.

---

### Impact Explanation

When the blocking check service is unavailable — due to network partition, service restart, proxy failure, or an attacker DoS-ing the check endpoint — every call to `starknet_proveTransaction` returns `Inconclusive`. With `fail_open = true`, the prover proceeds unconditionally and returns a cryptographically valid `ProveTransactionResult` containing `proof_facts` and `proof`. This proof can then be embedded in an invoke transaction and submitted to the sequencer, where `validate_proof_facts` will accept it because the proof is cryptographically sound.

The result: the transaction prover's RPC admission path generates and returns proofs for transactions that the operator explicitly configured the blocking check to reject. This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

---

### Likelihood Explanation

The trigger requires no special privilege. Any caller of `starknet_proveTransaction` benefits automatically whenever the blocking check service is unavailable. The `Inconclusive` path is reached by: transient network errors, service restarts, proxy misconfiguration, or a targeted DoS against the check endpoint. The default `blocking_check_timeout_millis: 2000` means even a 2-second delay in the check service causes the timeout path to fire with `fail_open = true`. [5](#0-4) 

---

### Recommendation

Change the default for `blocking_check_fail_open` to `false` in both `ProverConfig::default()` and `RawServiceConfig::default()`. An operator who explicitly configures a `blocking_check_url` intends to enforce a policy; the safe default when that policy service is unreachable is to deny proving, not to allow it.

```rust
// crates/starknet_transaction_prover/src/config.rs
impl Default for ProverConfig {
    fn default() -> Self {
        Self {
            // ...
            blocking_check_fail_open: false, // Fail-close by default: deny when check is inconclusive.
        }
    }
}
```

```rust
// crates/starknet_transaction_prover/src/server/config.rs
impl Default for RawServiceConfig {
    fn default() -> Self {
        Self {
            // ...
            blocking_check_fail_open: false,
        }
    }
}
```

Operators who require fail-open behavior (e.g., during initial rollout) must explicitly set `blocking_check_fail_open = true` in their configuration, making the permissive choice deliberate and auditable.

---

### Proof of Concept

1. Deploy the transaction prover with a `blocking_check_url` pointing to a real check service (e.g., `https://check.example.com`). Leave all other config at defaults (`blocking_check_fail_open` is `true`).
2. Take the check service offline (stop it, block its port, or simply let it time out within 2000 ms).
3. Submit any transaction — including one the check service would have returned error code 10000 for — to `starknet_proveTransaction`.
4. `check_transaction` returns `Inconclusive` (network error path at line 96–99 of `blocking_check.rs`).
5. `prove_with_blocking_check` evaluates `client.fail_open` → `true` → `allow = true`.
6. The prover proceeds to `run_and_prove`, executes the virtual OS, and returns a valid `ProveTransactionResult` with `proof_facts` and `proof`.
7. The caller embeds the proof in an invoke transaction and submits it to the sequencer. `validate_proof_facts` accepts it because the proof is cryptographically valid.

The blocked transaction has been proven and admitted, bypassing the operator's intended policy gate. [6](#0-5) [7](#0-6)
