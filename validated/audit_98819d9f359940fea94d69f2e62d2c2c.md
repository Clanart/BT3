### Title
`MerkleBlob::batch_insert()` Bypasses Duplicate-Key/Hash Guards, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert()` contains a fast-path loop that directly calls `insert_entry_to_blob()` for all items beyond the first two, completely skipping the duplicate-key and duplicate-hash existence checks that `insert()` enforces. Supplying a batch containing a key or hash already present in the tree silently overwrites the `BlockStatusCache` mapping, leaving a phantom duplicate leaf in the blob. The resulting Merkle root is computed over a structurally corrupt tree, invalidating all subsequent inclusion/exclusion proofs.

### Finding Description

`MerkleBlob::insert()` guards against re-insertion with two explicit checks before touching the blob:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` uses a two-phase strategy. When the tree already has ≥ 2 leaves (`leaf_count > 1`), the bootstrap branch is skipped entirely and the fast-path loop runs directly:

```rust
// blob.rs:587-603  — NO duplicate check here
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob()` unconditionally calls `block_status_cache.add_leaf()`:

```rust
// blob.rs:1024-1026
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    ...
}
``` [3](#0-2) 

`add_leaf()` calls `HashMap::insert`, which silently overwrites any existing entry for the same key or hash:

```rust
// blob.rs:188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);   // overwrites silently
}
``` [4](#0-3) 

After the overwrite, the cache points to the new duplicate leaf's index, but the original leaf node remains physically present in the blob at its old index. The tree now contains two leaf nodes sharing the same `KeyId` (or `Hash`), but the cache tracks only one. The subsequent subtree-linking phase (`insert_subtree_at_key`) and `calculate_lazy_hashes()` then compute a Merkle root over this corrupt structure.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
// blob.rs:1503-1519
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

### Impact Explanation

This is a **High** impact finding matching: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

Concrete consequences after a duplicate-key batch insert:

1. **Corrupted Merkle root**: `calculate_lazy_hashes()` hashes over a tree containing two leaves with the same key. The resulting root hash does not correspond to any valid key-value set.
2. **Forged/invalid inclusion proofs**: `get_proof_of_inclusion(key)` resolves the key through the cache to the *new* (duplicate) leaf's index. The proof path is built from that index, not the original leaf. A verifier checking the proof against the pre-corruption root will reject it; a verifier using the post-corruption root will accept a proof for the wrong leaf.
3. **Phantom leaf in blob**: The original leaf at its old index is now unreachable through the cache but still present in the blob bytes. Delta reconstruction and blob serialization will include it, causing cross-node disagreement on tree state.
4. **`check_integrity()` would detect this**, but it is only called in test builds (`check_integrity_on_drop: cfg!(test)`), not in production. [6](#0-5) 

### Likelihood Explanation

Any caller of `batch_insert` (Rust or Python via `py_batch_insert`) who supplies a batch containing a key already present in the tree, or two identical keys within the same batch, triggers the bug. No privilege is required. The DataLayer is designed to accept external key-value updates, making this a realistic untrusted-input path. The bug is also latent for same-batch duplicates even on a fresh tree once the batch size exceeds 2.

### Recommendation

Add the same existence checks at the top of the fast-path loop in `batch_insert()` that `insert()` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing code
}
```

Alternatively, refactor `batch_insert` to route every item through `insert()` for the guard logic, then apply the bulk tree-linking optimization separately.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    // 1. Build a tree with 3 leaves so leaf_count > 1 (bypasses bootstrap branch)
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    for i in 0i64..3 {
        blob.insert(KeyId(i), ValueId(i), &sha256_num(i), InsertLocation::Auto {}).unwrap();
    }

    // 2. batch_insert a batch where KeyId(0) already exists in the tree
    //    The fast-path loop (line 587) runs without any contains_key() check.
    let duplicate_key = KeyId(0);
    let new_hash = Hash(Bytes32::new([0xAB; 32]));
    blob.batch_insert(vec![((duplicate_key, ValueId(99)), new_hash)]).unwrap();
    // No error returned — duplicate silently accepted.

    // 3. The cache now maps KeyId(0) → new index; old leaf still in blob.
    //    calculate_lazy_hashes produces a root over a corrupt tree.
    blob.calculate_lazy_hashes().unwrap();

    // 4. check_integrity() (only active in test builds) would now fail with
    //    IntegrityKeyToIndexCacheLength or IntegrityKeyToIndexCacheIndex.
    //    In production, the corrupt root is silently committed.
    //    get_proof_of_inclusion(KeyId(0)) returns a proof for the wrong leaf.
}
