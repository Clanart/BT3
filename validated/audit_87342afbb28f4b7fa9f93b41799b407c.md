### Title
Unchecked Multiplication Overflow in `denormalize_amount` Causes Permanent Fund Loss - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The `denormalize_amount` function in the NEAR bridge contract performs `amount * (10_u128.pow(diff_decimals))` without any overflow guard. For tokens with a significant decimal difference between origin and NEAR representations, this multiplication silently wraps (or panics/traps) in WASM release builds, causing user funds to be permanently locked on the origin chain while the NEAR-side credit is either wrong or never issued.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

Neither the exponentiation `10_u128.pow(diff_decimals)` nor the final multiplication uses `checked_mul`, `saturating_mul`, or any other overflow-safe primitive. In Rust, integer overflow in release mode wraps silently (two's complement). NEAR WASM contracts are compiled in release mode; if `overflow-checks` is not explicitly set in `Cargo.toml`, the wrap is silent. If overflow checks are enabled, the WASM trap causes the callback to fail and revert, but the proof is not consumed — the transfer is permanently unfinalisable.

The function is called in three critical paths:

1. **`fin_transfer_callback`** (line 725) — processes a verified proof of an `InitTransfer` event from any origin chain. The `init_transfer.amount` comes directly from the on-chain event and is a full `u128`. [2](#0-1) 

2. **`fast_fin_transfer`** (line 770–772) — called by a trusted relayer, but the overflow still corrupts the amount comparison and stored value. [3](#0-2) 

3. **`claim_fee_callback`** (line 1122–1127) — denormalizes the proof amount to compute the fee. [4](#0-3) 

The `Decimals` struct stores both values as `u8`:

```rust
pub struct Decimals {
    pub decimals: u8,
    pub origin_decimals: u8,
}
``` [5](#0-4) 

For a token where `origin_decimals - decimals = D`, the overflow threshold for `amount` is `u128::MAX / 10^D`:

| D  | Threshold (raw units) | Threshold (18-dec tokens) |
|----|-----------------------|--------------------------|
| 6  | ≈ 3.4 × 10^32        | ≈ 3.4 × 10^14            |
| 18 | ≈ 3.4 × 10^20        | ≈ 340                    |
| 24 | ≈ 3.4 × 10^14        | ≈ 0.00034                |

For `D ≥ 39`, `10_u128.pow(D)` itself overflows u128, producing a completely wrong multiplier before the outer multiplication even runs.

The existing unit tests only exercise `u64::MAX` as the input amount, which does not trigger the overflow even at `D = 15`:

```rust
Contract::denormalize_amount(
    u64::MAX.into(),
    Decimals { decimals: 9, origin_decimals: 24 }
) == u128::from(u64::MAX) * 1_000_000_000_000_000_u128
``` [6](#0-5) 

No test covers `u128::MAX` as input with a nonzero `D`, leaving the overflow path untested.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

When a user initiates a transfer on the origin chain with an amount that causes overflow in `denormalize_amount`:

- **Wrap path**: The NEAR bridge credits the user with a drastically wrong (much smaller) amount. The origin-chain tokens are already locked or burned. The difference is irrecoverably lost, breaking bridge collateralization.
- **Trap path**: The `fin_transfer_callback` transaction reverts. The proof is not consumed, but since the amount is fixed by the origin-chain event, every retry with the same proof will trap again. The user's funds are permanently unclaimable.

In both cases the bridge's locked-token accounting diverges from the actual NEAR-side supply, satisfying the "accounting corruption that breaks bridge collateralization" impact criterion.

### Likelihood Explanation

Likelihood is **medium**. It requires a token registration where `origin_decimals - decimals ≥ 18`. This is realistic for:

- NEAR-native tokens (24 decimals) bridged to EVM chains (18 decimals) → D = 6 (high threshold, low risk)
- NEAR-native tokens (24 decimals) bridged to Solana (capped at 9 decimals via `MAX_ALLOWED_DECIMALS`) → D = 15
- Any token where the deployer or DAO registers with a large decimal gap

The EVM `InitTransfer` event uses `uint128 amount`, so the full u128 range is reachable by any user who sends a sufficiently large amount on the origin chain. [7](#0-6) 

### Recommendation

Replace the unchecked multiplication with a checked variant and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.checked_pow(diff_decimals)
        .expect("decimal multiplier overflows u128");
    amount.checked_mul(multiplier)
        .expect("denormalized amount overflows u128")
}
```

Using `checked_pow` and `checked_mul` ensures the contract panics with a clear error message rather than silently wrapping, and the transaction reverts cleanly. Additionally, add a validation in `add_token` that `origin_decimals >= decimals` and that `origin_decimals - decimals` does not exceed 38 (the maximum safe exponent for u128).

### Proof of Concept

```rust
// In Rust release mode (no overflow-checks), this silently wraps:
let decimals = Decimals { decimals: 6, origin_decimals: 24 }; // D = 18
let large_amount: u128 = u128::MAX / 10_u128.pow(17); // just above threshold
// large_amount ≈ 34_028_236_692_093_846_346
let result = Contract::denormalize_amount(large_amount, decimals);
// Expected: large_amount * 10^18 ≈ 3.4 × 10^37 (fits in u128)
// But large_amount * 10^18 for a slightly larger value wraps to near-zero

// Concrete overflow:
let overflow_amount: u128 = u128::MAX / 10_u128.pow(18) + 1;
// overflow_amount * 10^18 overflows u128 → wraps to a small value
let wrapped = Contract::denormalize_amount(overflow_amount, decimals);
assert!(wrapped < overflow_amount); // user receives far less than sent
```

A user sending `overflow_amount` tokens on the origin chain (a 6-decimal chain) would have their tokens locked/burned on that chain, while the NEAR bridge credits them with `wrapped` — a tiny fraction of the correct value. The difference is permanently lost.
