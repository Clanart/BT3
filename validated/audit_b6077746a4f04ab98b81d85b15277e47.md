### Title
Excess native ETH sent to payable swap functions is permanently trapped and front-runnable via `refundETH` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

The `pay()` helper in `PeripheryPayments` consumes exactly the required WETH-equivalent ETH from `address(this).balance`, but none of the payable swap entry-points (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) automatically return the remainder. Any `msg.value` above the actual swap cost is silently retained by the router. The only recovery path, `refundETH()`, is a separate call that sends the **entire** router ETH balance to `msg.sender` — meaning a front-runner who calls it first steals the trapped ETH.

---

### Finding Description

`PeripheryPayments.pay()` handles native-ETH-backed WETH payments as follows: [1](#0-0) 

When `nativeBalance >= value`, it wraps exactly `value` wei and forwards it to the pool. The surplus `address(this).balance - value` is left in the contract with no automatic sweep.

Every public swap entry-point is `payable` and calls `pay()` indirectly through the callback, but none of them call `refundETH()` before returning: [2](#0-1) 

The only recovery mechanism is the standalone `refundETH()`: [3](#0-2) 

This function sends the **full** router ETH balance to `msg.sender` — not to the original depositor. Any caller who submits this transaction before the victim does receives the trapped ETH.

The protocol's own test confirms that excess ETH is only returned when the user explicitly batches `refundETH()` inside a `multicall`: [4](#0-3) 

A user who calls `exactOutputSingle` directly (without multicall) — a natural usage pattern for an exact-output swap — has no atomic way to reclaim the surplus.

---

### Impact Explanation

**Severity: Medium — direct loss of user principal.**

A user calling `exactOutputSingle{value: amountInMaximum}(...)` with native ETH as `tokenIn` will have `amountInMaximum - actualAmountIn` ETH trapped in the router after the swap. A front-runner monitoring the mempool can call `refundETH()` in the same block and receive all of that ETH. The victim loses the surplus permanently with no on-chain recourse once the front-runner's transaction is confirmed.

---

### Likelihood Explanation

**Low-to-Medium.** The scenario requires:
1. The user calls a payable swap function directly (not via `multicall`) with `tokenIn = WETH` and sends more ETH than the actual swap cost — a natural pattern for exact-output swaps where the precise input is unknown.
2. A front-runner detects the stranded ETH and calls `refundETH()`.

Condition 1 is a realistic user mistake (or deliberate over-provisioning for slippage). Condition 2 is trivially automatable by MEV bots watching for non-zero `address(router).balance`.

---

### Recommendation

Add an automatic ETH refund at the end of each payable swap function, or enforce that `msg.value` equals the exact amount consumed:

**Option A (preferred — matches external report recommendation):** After each swap, refund any remaining native balance to `msg.sender`:

```solidity
// at the end of exactOutputSingle, exactInputSingle, exactInput, exactOutput
uint256 remaining = address(this).balance;
if (remaining > 0) _transferETH(msg.sender, remaining);
```

**Option B:** Require `msg.value` to equal exactly the amount consumed (stricter, breaks mixed native+WETH flows).

**Option C:** Document clearly that all native-ETH calls MUST be wrapped in `multicall` with a trailing `refundETH()`, and add a guard that reverts if `msg.value > 0` and the caller is not `address(this)` (i.e., not inside a delegatecall from `multicall`).

---

```solidity
// Step 2: Attacker front-runs victim's refundETH call
// refundETH() sends address(this).balance to msg.sender (attacker), not victim

// Attacker receives 0.2 ETH; victim's surplus is permanently lost
```

```text
  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
```
