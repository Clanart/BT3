### Title
`get_state_response_part` uses new-epoch shard layout to index into previous-epoch block chunks, producing state parts with a mismatched state root at resharding boundaries — (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`ChainStateSyncAdapter::get_state_response_part` derives `shard_index` from the **sync block's (new-epoch) shard layout** and uses it to index into `prev_block.chunks()`. At resharding boundaries `prev_block` belongs to the **old epoch** whose shard layout has different shard indices. The sibling function `compute_state_response_header` correctly uses `prev_shard_layout` for the same indexing step. The mismatch means the state root committed in the header and the state root used to generate the parts diverge, breaking the state-part/header reconstruction invariant.

---

### Finding Description

`compute_state_response_header` correctly fetches both layouts and uses the **previous epoch's** index to access the previous block's chunks:

```rust
// compute_state_response_header — correct
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let chunk_header = sync_prev_block.chunks().get(prev_shard_index)...;
``` [1](#0-0) 

`get_state_response_part` uses only the **new epoch's** layout to compute the index, then applies it to the previous block's chunks:

```rust
// get_state_response_part — wrong
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;  // new epoch
...
let prev_block = self.chain_store.get_block(header.prev_hash())?;   // old epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;          // index in NEW layout
let state_root = prev_block.chunks().get(shard_index)               // applied to OLD block
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [2](#0-1) 

When a shard is split, `derive_impl` splices two children into the `shard_ids` array at the split position, shifting every subsequent shard's index by +1:

```rust
shard_ids.splice(new_boundary_idx..new_boundary_idx + 1, new_shards.clone())
``` [3](#0-2) 

Concretely, if the old layout has shards `[0, 1, 2]` (indices 0–2) and shard 0 is split, the new layout has `[3, 4, 1, 2]` (indices 0–3). Shard 1 now has index 2 and shard 2 has index 3 in the new layout.

**For shard 1 (unchanged, index shifted from 1 → 2):**
- `compute_state_response_header` uses `prev_shard_index = 1` → reads the correct chunk → commits the correct `state_root` in the header.
- `get_state_response_part` uses `shard_index = 2` → reads shard 2's chunk → extracts shard 2's `state_root` → generates parts from the **wrong state root**.

**For shard 2 (unchanged, index shifted from 2 → 3):**
- `get_state_response_part` uses `shard_index = 3` → `prev_block.chunks().get(3)` is `None` (old block has only 3 chunks) → returns `Error::InvalidShardId`.

The exact corrupted value is the `state_root` extracted at lines 312–316: it is either the wrong shard's state root (causing parts that fail header validation) or triggers an immediate error.

---

### Impact Explanation

Any peer node can request state parts for a shard at a resharding sync hash. For every unchanged shard whose index shifts in the new layout, the serving node either:

1. Generates parts from the wrong state root — the receiving node validates them against the header's (correct) state root and rejects them, making state sync permanently fail for that shard.
2. Returns `InvalidShardId` immediately, also failing state sync.

Nodes that need to state-sync those shards after resharding cannot complete synchronization and cannot track those shards in the new epoch. With dynamic resharding enabled, this affects every resharding event.

---

### Likelihood Explanation

The bug is latent during static resharding (rare protocol-version upgrades) and becomes regularly reachable once `ProtocolFeature::DynamicResharding` is enabled. Any peer can trigger it by requesting state parts for a valid shard at the first block of a post-resharding epoch — no privileged role is required.

---

### Recommendation

In `get_state_response_part`, mirror the pattern used in `compute_state_response_header`: fetch the previous epoch's shard layout and use it to compute `shard_index` before accessing `prev_block.chunks()`.

```rust
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let state_root = prev_block.chunks().get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

---

### Proof of Concept

1. Run a network that undergoes resharding (static or dynamic): old layout `[shard 0, shard 1, shard 2]`, new layout `[shard 3, shard 4, shard 1, shard 2]` (shard 0 split).
2. Let `sync_hash` = first block of the new epoch.
3. A syncing peer calls `get_state_response_header(shard_id=1, sync_hash)` — succeeds, header commits `state_root = S1` (shard 1's correct root, read via `prev_shard_index=1`).
4. The same peer calls `get_state_response_part(shard_id=1, part_id=0, sync_hash)`:
   - `shard_layout` = new epoch layout → `shard_index = 2` (shard 1's index in new layout).
   - `prev_block.chunks().get(2).prev_state_root()` = shard 2's state root `S2 ≠ S1`.
   - Parts are generated from `S2`.
5. The peer validates received parts against `S1` from the header → validation fails → state sync for shard 1 is permanently broken after resharding. [4](#0-3) [5](#0-4)

    /// Computes ShardStateSyncResponseHeader.
    pub fn compute_state_response_header(
        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<ShardStateSyncResponseHeader, Error> {
        // Consistency rules:
        // 1. Everything prefixed with `sync_` indicates new epoch, for which we are syncing.
        // 1a. `sync_prev` means the last of the prev epoch.
        // 2. Empty prefix means the height where chunk was applied last time in the prev epoch.
        //    Let's call it `current`.
        // 2a. `prev_` means we're working with height before current.
        // 3. In inner loops we use all prefixes with no relation to the context described above.
        let sync_block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let sync_block_header = sync_block.header();
        let sync_block_epoch_id = sync_block_header.epoch_id();
        let shard_ids = self.epoch_manager.shard_ids(sync_block_epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }

        // The chunk was applied at height `chunk_header.height_included`.
        // Getting the `current` state.
        // TODO(current_epoch_state_sync): check that the sync block is what we would expect. So, either the first
        // block of an epoch, or the first block where there have been two new chunks in the epoch
        let sync_prev_block = self.chain_store.get_block(sync_block_header.prev_hash())?;

        let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
        let prev_epoch_id = sync_prev_block.header().epoch_id();
        let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
        let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;

        // Chunk header here is the same chunk header as at the `current` height.
        let sync_prev_hash = sync_prev_block.hash();
        let chunks = sync_prev_block.chunks();
        let chunk_header = chunks.get(prev_shard_index).ok_or(Error::InvalidShardId(shard_id))?;

    pub fn get_state_response_part(
        &mut self,
        shard_id: ShardId,
        part_id: u64,
        sync_hash: CryptoHash,
    ) -> Result<StatePart, Error> {
        let _span = tracing::debug_span!(
            target: "sync",
            "get_state_response_part",
            %shard_id,
            part_id,
            ?sync_hash)
        .entered();
        let block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let header = block.header();
        let epoch_id = block.header().epoch_id();
        // Check cache
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id)).unwrap();
        if let Some(bytes) = self.chain_store.store_ref().get(DBCol::StateParts, &key) {
            metrics::STATE_PART_CACHE_HIT.inc();
            let state_part = StatePart::from_bytes(bytes.to_vec())?;
            return Ok(state_part);
        }
        metrics::STATE_PART_CACHE_MISS.inc();

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_ids = self.epoch_manager.shard_ids(epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let prev_block = self.chain_store.get_block(header.prev_hash())?;
        let shard_index = shard_layout.get_shard_index(shard_id)?;
        let state_root = prev_block
            .chunks()
            .get(shard_index)
            .ok_or(Error::InvalidShardId(shard_id))?
            .prev_state_root();
        let prev_hash = *prev_block.hash();
        let prev_prev_hash = *prev_block.header().prev_hash();
        let state_root_node = self
            .runtime_adapter
            .get_state_root_node(shard_id, &prev_hash, &state_root)
            .log_storage_error("get_state_root_node fail")?;
        let num_parts = get_num_state_parts(state_root_node.memory_usage);
        if part_id >= num_parts {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let current_time = Instant::now();
        let state_part = self
            .runtime_adapter
            .obtain_state_part(
                shard_id,
                &prev_prev_hash,
                &state_root,
                PartId::new(part_id, num_parts),
            )
            .log_storage_error("obtain_state_part fail")?;

        let elapsed_ms = (self.clock.now().signed_duration_since(current_time))
            .whole_milliseconds()
            .max(0) as u128;
        self.requested_state_parts
            .save_state_part_elapsed(&sync_hash, &shard_id, &part_id, elapsed_ms);

        // Cache the part data, but only if the corresponding header is also cached.
        // At epoch boundaries, clear_all_downloaded_parts() deletes all cached headers
        // and parts. Since serving runs on a separate actor, a part request can arrive
        // after the clear and re-create a StatePartKey without its StateHeaderKey,
        // which the storage validator treats as an inconsistency.
        let header_key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
        if self.chain_store.store_ref().exists(DBCol::StateHeaders, &header_key) {
            let mut store_update = self.chain_store.store().store_update();
            let bytes = state_part.to_bytes();
            store_update.set(DBCol::StateParts, &key, &bytes);
            store_update.commit();
        }

        Ok(state_part)
    }
