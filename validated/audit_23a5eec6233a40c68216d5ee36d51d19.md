### Title
Unvalidated `class_hash` in `execute_replace_class` Enables Instant Malicious Class Substitution and Fund Theft — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The StarkNet OS `execute_replace_class` syscall handler accepts any arbitrary `class_hash` value from a contract without verifying that the hash corresponds to a previously declared contract class. This is explicitly acknowledged by a TODO comment in the production code. A malicious contract deployer can exploit this to instantly swap a contract's class to a pre-declared drain class — or to an undeclared hash — with no delay, no timelock, and no protocol-level guard, resulting in direct loss of or permanent freezing of user funds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 877–916) processes the `replace_class` syscall. After deducting gas, it reads `request.class_hash` and immediately writes it into the contract's `StateEntry` via `dict_update`, with no check that the supplied hash corresponds to any declared class in `contract_class_changes`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 is a developer acknowledgment that this validation is **intentionally absent** from the current production code. The same omission exists in the deprecated path at `deprecated_execute_syscalls.cairo` lines 307–329, which also performs no class hash validation.

The `replace_class` syscall is callable by any executing contract — no privileged role is required. The OS enforces no delay, no timelock, and no check against the `contract_class_changes` dictionary before committing the new class hash to state.

---

### Impact Explanation

Two concrete critical impacts follow from this missing validation:

**Scenario A — Direct Loss of Funds (Critical):**
1. Attacker deploys a legitimate-looking contract (e.g., a high-APR staking vault).
2. Attacker separately submits a `declare` transaction for a malicious class containing a fund-drain entry point. This is accepted by the protocol because class declaration is permissionless.
3. Users deposit funds into the staking contract.
4. Attacker calls `replace_class(malicious_class_hash)` from within the contract. The OS writes the new class hash to state immediately, with no validation and no delay.
5. Attacker calls the drain entry point. The contract now executes the malicious class, transferring all user funds to the attacker.

**Scenario B — Permanent Freezing of Funds (Critical):**
1. Same setup as above, but in step 4 the attacker calls `replace_class(0)` or any undeclared felt value.
2. The OS writes `class_hash = 0` (or an undeclared hash) to the contract's `StateEntry`.
3. All future calls to the contract fail because no class exists for that hash.
4. Funds stored in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

- **Permissionless entry path**: Any account can deploy a contract and call `replace_class` from within it. No privileged key or operator role is required.
- **Class declaration is permissionless**: The malicious class can be declared by anyone in a prior transaction.
- **No protocol guard**: The OS imposes no delay, no timelock, and no declared-class check before committing the replacement. The TODO comment confirms this is a known, unresolved gap.
- **Realistic attack surface**: Staking contracts, vaults, and any contract holding user funds are natural targets. The attack requires only that users trust the contract's initial behavior before the class is swapped.

---

### Recommendation

In `execute_replace_class` (`syscall_impls.cairo`), before writing the new `StateEntry`, verify that `class_hash` exists as a key in `contract_class_changes` (i.e., it has been declared in the current or a prior block). This is exactly what the existing TODO comment calls for. The same fix must be applied to the deprecated path in `deprecated_execute_syscalls.cairo`.

Additionally, consider whether a protocol-level delay (analogous to a timelock) should be enforced between a `replace_class` call and the point at which the new class becomes active, giving users time to withdraw funds before a class substitution takes effect.
