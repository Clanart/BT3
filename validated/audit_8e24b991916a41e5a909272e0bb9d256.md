### Title
`OmniBridge.initTransfer1155()` Does Not Check That the ERC1155 Token Is Registered in `multiTokens`, Causing Permanent Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer1155()` accepts any ERC1155 `(tokenAddress, tokenId)` pair and locks the tokens in the bridge without verifying that the pair has been registered via `logMetadata1155()`. When an unregistered pair is used, the locked ERC1155 tokens become permanently irrecoverable.

### Finding Description

The ERC1155 bridge flow requires two steps:

1. **Registration**: `logMetadata1155(tokenAddress, tokenId)` must be called first. It computes `deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId)` and writes `multiTokens[deterministicToken] = {tokenAddress, tokenId}`. It also emits a `LogMetadata` event that triggers NEAR-side token registration (decimals, address mapping).

2. **Transfer initiation**: `initTransfer1155(tokenAddress, tokenId, ...)` is called by the user to lock tokens and emit `InitTransfer`.

`logMetadata1155()` is the only function that populates `multiTokens`: [1](#0-0) 

However, `initTransfer1155()` does **not** check whether `multiTokens[deterministicToken]` has been set before accepting and locking the user's tokens: [2](#0-1) 

The function computes `deterministicToken`, calls `IERC1155.safeTransferFrom` to pull tokens into the bridge, and emits `InitTransfer` — all without any `require(multiTokens[deterministicToken].tokenAddress != address(0))` guard.

Compare this to `finTransfer()`, which correctly uses `multiTokens` to dispatch the ERC1155 release: [3](#0-2) 

If `multiTokens[deterministicToken]` is zero (unregistered), `finTransfer` falls through to the ERC20 path and calls `IERC20(deterministicToken).safeTransfer(...)`, which will revert because `deterministicToken` is a hash-derived address with no ERC20 code. There is no recovery path.

On the NEAR side, `fin_transfer_callback` looks up `token_decimals` for the token. Since `logMetadata1155()` was never called, the NEAR-side registration never happened, and the callback panics with `TokenDecimalsNotFound`: [4](#0-3) 

No MPC signature for a `FinTransfer` back to EVM is ever produced. The ERC1155 tokens are permanently locked.

### Impact Explanation

Any user who calls `initTransfer1155()` for an unregistered `(tokenAddress, tokenId)` pair permanently loses their ERC1155 tokens. The tokens are locked in the EVM bridge contract with no release path: the NEAR side panics and never issues a `FinTransfer` signature, and the EVM `finTransfer` cannot release ERC1155 tokens without a valid `multiTokens` entry. This is an irrecoverable fund lock matching the allowed critical impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

### Likelihood Explanation

The entry path is fully unprivileged — any token holder can call `initTransfer1155()` directly. There is no on-chain enforcement of the `logMetadata1155()` → `initTransfer1155()` ordering. A user or integrator who is unaware of the required pre-registration step, or who calls `initTransfer1155()` for a token that was never registered (e.g., a new ERC1155 collection), will trigger this path. The function accepts any ERC1155-compliant contract without restriction.

### Recommendation

Add a registration check at the start of `initTransfer1155()`:

```solidity
function initTransfer1155(
    address tokenAddress,
    uint256 tokenId,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
+   require(
+       multiTokens[deterministicToken].tokenAddress != address(0),
+       "ERC1155 token not registered"
+   );
    currentOriginNonce += 1;
    ...
}
```

This mirrors the pattern already used in `logMetadata1155()` where the mapping is checked for consistency, and ensures that only tokens with a valid NEAR-side registration can be bridged.

### Proof of Concept

1. Deploy any ERC1155 contract and mint tokens to `attacker`.
2. Call `bridge.initTransfer1155(erc1155Address, tokenId, amount, 0, 0, "victim.near", "")` **without** first calling `logMetadata1155(erc1155Address, tokenId)`.
3. `IERC1155(erc1155Address).safeTransferFrom(attacker, bridge, tokenId, amount, "")` succeeds — tokens are now in the bridge.
4. `InitTransfer` is emitted with `deterministicToken` as the token address.
5. NEAR relayer submits the proof; `fin_transfer_callback` calls `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` — panics. No `FinTransfer` signature is issued.
6. On EVM, `finTransfer` cannot be called with a valid signature. Even if called, `multiTokens[deterministicToken].tokenAddress == address(0)`, so it falls to `IERC20(deterministicToken).safeTransfer(...)` which reverts (no code at that address).
7. ERC1155 tokens are permanently locked in the bridge.
