### Title
Precision Loss in `increased_enough` Allows Fee-Escalation Bypass for Small Tip/Gas-Price Values - (File: crates/apollo_mempool/src/mempool.rs)

### Summary

The `increased_enough` function in the mempool uses integer division to compute the required fee escalation threshold. When `existing_value * fee_escalation_percentage < 100`, the division truncates to zero, making the threshold equal to `existing_value` itself. A replacement transaction carrying the exact same tip or `max_l2_gas_price` as the existing transaction is then accepted, bypassing the mandatory escalation requirement entirely.

### Finding Description

`increased_enough` computes the minimum acceptable incoming value as:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);
    let Some(escalation_qualified_value) = existing_value
        .checked_mul(percentage)
        .map(|v| v / 100)                                    // ← truncating division
        .and_then(|increase| existing_value.checked_add(increase))
    else { return false; };
    incoming_value >= escalation_qualified_value
}
``` [1](#0-0) 

The production default is `fee_escalation_percentage = 10`. [2](#0-1) 

When `existing_value ∈ [1, 9]`:

```
existing_value * 10 = 10..90
10..90 / 100 = 0   (integer truncation)
escalation_qualified_value = existing_value + 0 = existing_value
```

The check degenerates to `incoming_value >= existing_value`, so a replacement transaction with the **identical** tip or `max_l2_gas_price` passes. The code comment acknowledges precision loss but the "multiply first" mitigation only helps when `existing_value ≥ 10`; it does not fix the case where the product itself is still below 100. [3](#0-2) 

`should_replace_tx` requires **both** tip and `max_l2_gas_price` to pass `increased_enough`. If either field is in the vulnerable range, the corresponding dimension of the escalation check is nullified. [4](#0-3) 

### Impact Explanation

The mempool's fee-escalation mechanism is the sole guard against transaction-replacement spam: without it, any account can evict its own pending transaction from the pool at zero additional cost, repeatedly refreshing its position or front-running other transactions. When the guard is bypassed, the mempool accepts replacement transactions that should be rejected, matching the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Concretely, an attacker can:
1. Submit a transaction with `tip = 1` (or any value 1–9) and a small `max_l2_gas_price`.
2. Repeatedly replace it with an identical transaction, each time evicting the previous entry from the pool without paying any escalation premium.
3. This can be used to keep a nonce slot occupied indefinitely at minimal cost, or to perform low-cost front-running by refreshing transaction ordering.

### Likelihood Explanation

- `fee_escalation_percentage = 10` is the shipped default in both `MempoolStaticConfig::default()` and the production deployment config.
- The `Tip` field is a `u64` and can legitimately be 0 or any small integer; the test suite itself exercises `tip: 0` and `tip: 90` with small `max_l2_gas_price` values.
- No external privilege is required: any user submitting a standard V3 invoke transaction can trigger this path.
- The vulnerable range (`existing_value < 10`) is narrow but reachable whenever a sender deliberately sets a low tip to minimize fees. [5](#0-4) 

### Recommendation

Replace the floor-division increase with a ceiling-division, mirroring the OpenZeppelin `Math.ceilDiv` fix recommended in the original report:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);
    // Ceiling division: (existing_value * percentage + 99) / 100
    let increase = existing_value
        .checked_mul(percentage)
        .and_then(|v| v.checked_add(99))
        .map(|v| v / 100);
    let Some(escalation_qualified_value) = increase
        .and_then(|inc| existing_value.checked_add(inc))
    else { return false; };
    incoming_value >= escalation_qualified_value
}
```

This ensures that for any `existing_value ≥ 1` and any non-zero `percentage`, the required increase is at least 1, so a replacement with the same value is always rejected.

    fn handle_fee_escalation(
        &mut self,
        incoming_tx_reference: TransactionReference,
        validation_only: bool,
    ) -> MempoolResult<()> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(());
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(());
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        if validation_only {
            return Ok(());
        }

        debug!("{existing_tx_reference} will be replaced by {incoming_tx_reference}.");

        self.tx_queue.remove_txs(&[existing_tx_reference]);
        self.tx_pool
            .remove(existing_tx_reference.tx_hash)
            .expect("Transaction hash from pool must exist.");

        Ok(())
    }

            [existing_tx, incoming_tx].map(|tx| u128::from(tx.tip.0));

    fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
        let percentage = u128::from(self.config.static_config.fee_escalation_percentage);

        // Note: To reduce precision loss, we first multiply by the percentage and then divide by
        // 100. This could cause an overflow and an automatic rejection of the transaction, but the

            .checked_mul(percentage)
            .map(|v| v / 100)
            .and_then(|increase| existing_value.checked_add(increase))
        else {
            // Overflow occurred during calculation; reject the transaction.
            return false;
        };

        incoming_value >= escalation_qualified_value
    }
