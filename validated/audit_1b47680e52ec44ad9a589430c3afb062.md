### Title
Missing Declared-Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. Any contract can replace its own class hash with an arbitrary, undeclared felt value. Once committed to state, the contract becomes permanently inaccessible because the OS cannot resolve the invalid class hash to a compiled class, permanently freezing any funds the contract holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the new `class_hash` from the syscall request and directly updates the contract's `StateEntry` in `contract_state_changes` with no validation that the hash is declared. The code itself contains an explicit TODO acknowledging the missing check: [1](#0-0) 

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The unvalidated `class_hash` is permanently committed to the contract's `StateEntry`. When any subsequent transaction calls this contract, `execute_entry_point` performs the following chain:

1. Reads the contract's class hash from `contract_state_changes` (now the arbitrary value).
2. Calls `dict_read` on `contract_class_changes` to resolve it to a compiled class hash — returns `0` since the arbitrary hash was never declared.
3. Calls `find_element` on `compiled_class_facts` with key `0` — no compiled class fact exists for this key. [2](#0-1) 

`find_element` is a hint-enforced assertion; if no matching element exists, the block is unprovable. The sequencer is forced to exclude every transaction targeting the bricked contract, making its funds permanently inaccessible.

This is the direct analog of the `addStrategy()` missing access control issue: just as `addStrategy()` lacked a check that the caller was the owner (allowing anyone to set the strategy mapping to any address), `execute_replace_class` lacks a check that the new class hash is declared (allowing any contract to set its class hash to any arbitrary value), corrupting critical protocol state with no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract that replaces its class hash with an undeclared value becomes permanently inaccessible. All ERC-20 tokens or other assets held by that contract are frozen with no recovery mechanism. The state update is committed to the global state root via `state_update` and `compute_contract_state_commitment`; there is no OS-level mechanism to revert a committed class hash replacement. [3](#0-2) 

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract without restriction. A malicious contract author can design a contract that:

1. Accepts user deposits.
2. After collecting funds, calls `replace_class` with an arbitrary undeclared hash (e.g., `felt 1`).
3. Permanently bricks the contract, freezing all deposited funds.

The TODO comment in the production code confirms the developers are aware of the missing check, indicating it is a known, unresolved gap in the current implementation. No special privilege, leaked key, or network-level attack is required — only the ability to deploy and call a contract.

---

### Recommendation

Before updating the contract's `StateEntry`, `execute_replace_class` must verify that `request.class_hash` corresponds to a declared class. Concretely, it should assert that a non-zero compiled class hash mapping exists for `request.class_hash` in `contract_class_changes` (i.e., the class was previously declared via `execute_declare_transaction`). This is precisely the check noted in the existing TODO comment and should be enforced as a hard assertion, not a hint.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts user deposits and exposes a `drain()` entry point.
2. Users deposit funds into `MaliciousVault`.
3. Attacker calls `drain()`, which internally issues the `replace_class` syscall with `class_hash = 1` (an arbitrary, undeclared value).
4. The OS executes `execute_replace_class`: no declared-class check is performed; `dict_update` commits `class_hash=1` to `MaliciousVault`'s `StateEntry`.
5. Any subsequent transaction targeting `MaliciousVault` causes `execute_entry_point` to call `find_element` with a resolved compiled class hash of `0` (since `1` is not in `contract_class_changes`).
6. `find_element` cannot locate a matching `CompiledClassFact`; the block is unprovable.
7. The sequencer cannot include any transaction targeting `MaliciousVault`. All deposited funds are permanently frozen. [4](#0-3)

        // Not enough gas to execute the syscall.
