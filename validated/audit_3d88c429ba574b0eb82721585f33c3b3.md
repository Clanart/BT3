### Title
Native Fee (STRK) Permanently Locked in Starknet OmniBridge — No Withdrawal Mechanism - (File: starknet/src/omni_bridge.cairo)

### Summary
Every `init_transfer` call on the Starknet OmniBridge that includes a non-zero `native_fee` transfers STRK tokens from the caller into the contract itself. The contract exposes no function to withdraw or distribute these accumulated STRK tokens to relayers or any other party, making them permanently irrecoverable.

### Finding Description
In `init_transfer`, when `native_fee > 0`, the contract pulls STRK tokens from the caller directly into `get_contract_address()`:

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
``` [1](#0-0) 

The complete public interface of `IOmniBridge` is: `log_metadata`, `deploy_token`, `fin_transfer`, `init_transfer`, `upgrade_token`, `set_pause_flags`, `pause_all`, `get_token_address`, `is_bridge_token`, `is_transfer_finalised`. [2](#0-1) 

None of these functions withdraw or distribute the accumulated STRK native fees. The `fin_transfer` function only transfers `payload.amount` to `payload.recipient` and does not pay any native fee to `fee_recipient`. [3](#0-2) 

The same pattern exists on EVM: `nativeFee` is retained in the `OmniBridge` contract as ETH (via `msg.value`) with no withdrawal function, while `finTransfer` only pays `payload.amount` to `payload.recipient`. [4](#0-3) 

### Impact Explanation
Every `init_transfer` with `native_fee > 0` on Starknet permanently locks STRK tokens in the contract. Relayers who are supposed to receive these fees as cross-chain transfer incentives can never claim them. The locked funds are user-paid protocol value that is irrecoverably frozen. This matches: **Critical — Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows.**

### Likelihood Explanation
Any unprivileged user calling `init_transfer` with a non-zero `native_fee` triggers the lock. The fee field is a standard user-facing parameter documented in the bridge API. Every such call permanently loses the fee. Likelihood is **High** — the code path is exercised by normal bridge usage.

### Recommendation
Add an admin-gated withdrawal function to the Starknet contract (and EVM contract) that transfers accumulated native fees to a designated fee recipient or treasury:

```cairo
fn withdraw_native_fees(ref self: ContractState, recipient: ContractAddress, amount: u128) {
    self.accesscontrol.assert_only_role(DEFAULT_ADMIN_ROLE);
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer(recipient, amount.into());
    assert(success, 'ERR_WITHDRAW_FAILED');
}
```

Alternatively, redesign `init_transfer` to not hold the `native_fee` in the contract at all — instead, route it directly to a pre-configured relayer fee recipient at transfer time.

### Proof of Concept
1. User calls `init_transfer` on Starknet with `native_fee = 1_000_000` (1 STRK).
2. Contract executes `transfer_from(caller, get_contract_address(), 1_000_000)` — STRK enters the contract. [1](#0-0) 
3. Relayer calls `fin_transfer` to complete the cross-chain transfer. `fin_transfer` mints/transfers `payload.amount` to `payload.recipient` only — no STRK is paid out. [5](#0-4) 
4. No function in `IOmniBridge` can move the STRK out of the contract. [2](#0-1) 
5. The 1 STRK is permanently locked. Repeating across all bridge users compounds the loss indefinitely.

            _verify_borsh_signature(

            );

            if self.is_bridge_token(payload.token_address) {
                IBridgeTokenDispatcher { contract_address: payload.token_address }

                    .transfer(payload.recipient, payload.amount.into());

                        FinTransfer {
