Audit Report

## Title
Unattributed `address(this).balance` in `pay()` WETH branch allows stranded ETH from one user to fund another user's WETH liquidity payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` in its WETH branch without attributing that ETH to the current payer. Any ETH sent with a `payable` entry point for a non-WETH pool is never consumed and remains on the contract. A subsequent WETH-pool liquidity call by any other user silently consumes that stranded ETH, causing a direct, unrecoverable loss of principal for the original depositor.

## Finding Description
The WETH branch of `pay()` reads the contract's total native balance:

```solidity
uint256 nativeBalance = address(this).balance;   // total, unattributed
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) { ... }
``` [1](#0-0) 

The `receive()` guard only blocks plain ETH transfers (no calldata):

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

It does **not** intercept ETH sent via `payable` function calls. When User A calls `addLiquidityExactShares{value: X}(nonWethPool, ...)`, the ETH arrives as `msg.value` and the `receive()` function is never invoked. Both `pay()` calls in the callback take the `safeTransferFrom` branch (neither token is WETH), so the ETH is never consumed and sits on the contract. [3](#0-2) 

`refundETH()` sends the entire balance to `msg.sender` with no per-caller accounting: [4](#0-3) 

The callback correctly loads `payer` from transient storage, but `pay()` ignores `payer` for ETH attribution — it consumes whatever is in `address(this).balance` regardless of who deposited it: [5](#0-4) 

## Impact Explanation
Direct, cross-user loss of ETH principal. User A's ETH is permanently transferred to the pool on behalf of User B, with no recovery path once consumed. User B's WETH allowance is not touched. This meets the Critical/High threshold for direct loss of user principal under Sherlock contest rules.

## Likelihood Explanation
The Uniswap-style router UX of sending ETH with a liquidity call and relying on a separate `refundETH()` step is standard. Any user who sends ETH with a non-WETH pool call, sends excess ETH with a WETH pool call, or uses `multicall` without appending `refundETH()` as the last step leaves ETH on the contract. A MEV bot observing the mempool can immediately follow with a WETH-pool liquidity call to consume it. No special privileges are required; any unprivileged caller can trigger this.

## Recommendation
Track per-caller ETH deposits in transient storage alongside the existing pay context. In `_addLiquidity`, store `msg.value` keyed to `payer` in a transient slot. In `pay()`'s WETH branch, consume only the ETH attributed to the current `payer` rather than `address(this).balance`. Alternatively, pass `msg.value` explicitly through the call stack and use only that value within the same call frame, never reading the total contract balance.

## Proof of Concept
1. User A calls `addLiquidityExactShares{value: 1 ether}(nonWethPool, ...)`. Pool requests ERC20 tokens only; both `pay()` calls take `safeTransferFrom`. 1 ETH remains on the contract.
2. Attacker calls `addLiquidityExactShares(wethPool, ...)` (no ETH sent) for a pool where `token0 = WETH` and the pool requests `amount0Delta = 1e18`.
3. `metricOmmModifyLiquidityCallback` loads `payer = attacker`, calls `pay(WETH, attacker, wethPool, 1e18)`.
4. `nativeBalance = 1 ether >= 1e18` → `IWETH9.deposit{value: 1 ether}()` + `safeTransfer(wethPool, 1e18)`. Attacker's WETH allowance is not touched.
5. User A's 1 ETH is gone. `refundETH()` now returns 0.
