Audit Report

## Title
`sweepToken` and `unwrapWETH9` accept arbitrary `recipient` with no access control, enabling front-run theft of stranded router balances — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.sweepToken` and `PeripheryPayments.unwrapWETH9` are `public payable` functions that transfer the router's entire token or WETH balance to a caller-supplied `recipient` address with no `msg.sender` restriction. Any tokens or ETH that accumulate in `MetricOmmSimpleRouter` — from accidental direct sends or surplus ETH left by a `multicall` that omits `refundETH` — can be stolen by an unprivileged attacker, including by front-running a legitimate recovery attempt.

## Finding Description
Both functions are confirmed in the production code at `metric-periphery/contracts/base/PeripheryPayments.sol`:

```solidity
// Lines 37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}

// Lines 48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
```

Neither function checks `msg.sender` or restricts `recipient`. The contrast with `refundETH` (lines 58–63), which correctly hardcodes `msg.sender` as the destination, confirms this is an inconsistency rather than intentional design.

**Token accumulation paths:**

1. **Accidental direct sends** — users or integrators mistakenly transfer ERC-20 tokens (USDC, USDT) directly to the router address.
2. **Surplus ETH from native WETH payment** — the `pay` helper (lines 73–84) wraps only the required ETH amount (`value`); any `msg.value` surplus remains as raw ETH in the contract. If a user's `multicall` omits the `refundETH()` step, the ETH is stranded and sweepable via `unwrapWETH9` after wrapping.

**Front-run exploit path:**
1. Victim accidentally sends 10,000 USDC to `MetricOmmSimpleRouter`.
2. Victim broadcasts `sweepToken(USDC, 0, victim)`.
3. Attacker observes the pending transaction in the mempool.
4. Attacker submits `sweepToken(USDC, 0, attacker)` with higher gas.
5. Attacker's transaction mines first; attacker receives 10,000 USDC.
6. Victim's transaction executes: `balanceToken = 0`, `amountMinimum = 0` → no revert, no transfer. Victim loses all funds silently.

## Impact Explanation
**Severity: Medium.** Any tokens stranded in `MetricOmmSimpleRouter` — from accidental sends or surplus ETH — can be permanently stolen by an unprivileged attacker. The loss is direct user principal (USDC/USDT are explicitly in scope). The victim's recovery transaction silently succeeds with zero transfer when front-run, providing no on-chain indication of the theft. This meets the Sherlock medium threshold for direct loss of user principal with no privileged precondition.

## Likelihood Explanation
- Accidental token sends to router addresses are a common and well-documented user error, especially for high-value stablecoins (USDC/USDT).
- The attack requires zero setup: any EOA can call `sweepToken` or `unwrapWETH9` at any time with `amountMinimum = 0`.
- Front-running is trivially achievable on Ethereum mainnet and Base via standard mempool monitoring.
- The attack is repeatable for every new stranded balance.

## Recommendation
Restrict `recipient` to `msg.sender` in both functions, matching the behavior of `refundETH()`:

```solidity
function sweepToken(address token, uint256 amountMinimum) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(msg.sender, balanceToken);
    }
}

function unwrapWETH9(uint256 amountMinimum) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

Update `IPeripheryPayments` and all call sites accordingly.

## Proof of Concept
```solidity
// Foundry fork test
function test_sweepToken_frontrun() public {
    address usdc = 0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48;
    address router = address(metricOmmSimpleRouter);
    address victim = makeAddr("victim");
    address attacker = makeAddr("attacker");

    // Victim accidentally sends 10,000 USDC to router
    deal(usdc, router, 10_000e6);

    // Attacker front-runs victim's recovery call
    vm.prank(attacker);
    metricOmmSimpleRouter.sweepToken(usdc, 0, attacker);

    assertEq(IERC20(usdc).balanceOf(attacker), 10_000e6);
    assertEq(IERC20(usdc).balanceOf(victim), 0);

    // Victim's recovery call silently succeeds with zero transfer
    vm.prank(victim);
    metricOmmSimpleRouter.sweepToken(usdc, 0, victim); // no revert, no transfer
    assertEq(IERC20(usdc).balanceOf(victim), 0);
}
```
