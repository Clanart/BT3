### Title
Stale-Pending Fallback Silently Zeroes `l1_data_gas_price` and `l2_gas_price` in RPC Fee Estimation — (`File: crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When the in-memory pending block is stale (its `parent_block_hash` does not match the latest committed block hash), `read_pending_data` synthesises a `DeprecatedPendingBlock` fallback. That deprecated type carries no `l1_data_gas_price` or `l2_gas_price` fields; both accessors unconditionally return `GasPricePerToken::default()` (all-zero). Every RPC call that requests `BlockId::Tag(Tag::Pending)` — `starknet_estimateFee`, `starknet_simulateTransactions`, `starknet_traceTransaction` — then builds a `BlockContext` with `NonzeroGasPrice::MIN` for those two dimensions, producing authoritative-looking but materially wrong fee estimates.

---

### Finding Description

**Root cause — `read_pending_data` fallback path** [1](#0-0) 

When `pending_data.block.parent_block_hash() != latest_header.block_hash`, the function constructs a `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock { … })`. Only `eth_l1_gas_price` and `strk_l1_gas_price` are populated from the committed header; every other field is `..Default::default()`.

**Zero-price accessors on the deprecated variant** [2](#0-1) 

`l1_data_gas_price()` and `l2_gas_price()` both return `GasPricePerToken::default()` (price_in_wei = 0, price_in_fri = 0) for the `Deprecated` arm. The comments acknowledge this is intentional for *old* blocks, but the fallback path applies it to *current* blocks whenever pending data lags.

**Zero prices flow into fee-estimation block context** [3](#0-2) 

`exec_block_context` reads `pending_data.l1_data_gas_price` and `pending_data.l2_gas_price` directly. When both are zero, `NonzeroGasPrice::new(…).unwrap_or(NonzeroGasPrice::MIN)` substitutes the protocol minimum, not the real market price.

**Callers that expose this to unprivileged users** [4](#0-3) [5](#0-4) [6](#0-5) 

`estimate_fee`, `simulate_transactions`, and `estimate_message_fee` all call `read_pending_data` when `block_id == BlockId::Tag(Tag::Pending)` and pass the result straight to execution.

---

### Impact Explanation

Any caller of `starknet_estimateFee` / `starknet_simulateTransactions` / `starknet_traceTransaction` with `block_id = "pending"` during the stale window receives fee estimates computed with `l1_data_gas_price = NonzeroGasPrice::MIN` and `l2_gas_price = NonzeroGasPrice::MIN`. For V3 transactions (which pay separately for L2 gas and blob data), the returned `overall_fee`, `data_gas_consumed × data_gas_price`, and `l2_gas_price` fields are all wrong. Wallets and dApps that use these estimates to set `max_amount` / `max_price_per_unit` resource bounds will submit transactions that either revert (bounds too low once the real block is sequenced) or are rejected at the gateway. The RPC response carries no indication that fallback prices were used, so the wrong value is authoritative-looking.

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The stale condition is reached whenever the committed-block pointer in storage advances past the `parent_block_hash` stored in the in-memory `PendingData`. This happens:

- **Normally** during every block transition: the batcher writes the new block to storage before the orchestrator refreshes the pending-data pointer, creating a brief but real window.
- **Persistently** on read-only / sync-only nodes that never produce blocks and therefore never update `pending_data`.
- **Deliberately** by any client that simply queries `pending` immediately after observing a new block number via `starknet_blockNumber`.

No special privilege is required; a standard JSON-RPC call suffices.

---

### Recommendation

Replace the `DeprecatedPendingBlock` fallback with a `PendingBlock` (the current variant) that copies all three gas-price fields from the latest committed header:

```rust
// In read_pending_data, else branch:
block: PendingBlockOrDeprecated::Current(PendingBlock {
    parent_block_hash: latest_header.block_hash,
    l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
    l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price,
    l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,
    l1_da_mode: latest_header.block_header_without_hash.l1_da_mode,
    timestamp: latest_header.block_header_without_hash.timestamp,
    sequencer_address: latest_header.block_header_without_hash.sequencer,
    starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
    ..Default::default()
}),
```

Using the committed header's prices is analogous to the Ditto recommendation of using `getSavedOrSpotOraclePrice` instead of the cached price: it is still a fallback, but it uses the most recently confirmed value for all price dimensions rather than silently dropping two of them to zero.

---

### Proof of Concept

1. Wait for (or trigger) a block commit so that `storage_reader` reports block N but `pending_data.block.parent_block_hash()` still points to block N-1's hash.
2. Call `starknet_estimateFee` with any V3 Invoke transaction and `block_id = "pending"`.
3. Observe that the returned `data_gas_consumed × l1_data_gas_price` component equals `data_gas_consumed × NonzeroGasPrice::MIN` (≈ 1 fri) instead of the real blob-gas price (typically thousands of fri).
4. Submit the transaction with the returned resource bounds; it will revert or be rejected once the sequencer applies the real gas prices.

The corrupted values are `l1_data_gas_price` and `l2_gas_price` in the `FeeEstimation` response, both set to `NonzeroGasPrice::MIN` instead of the values from the latest committed block header.
