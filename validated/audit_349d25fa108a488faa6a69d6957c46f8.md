### Title
Missing Staleness Validation in ETH/STRK Oracle Response Causes Wrong Gas Prices in Block Commitments - (File: `crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs`)

---

### Summary

`resolve_query` in `EthToStrkOracleClient` validates only `price` and `decimals` from the oracle HTTP response. It never checks the `timestamp` field that the oracle returns alongside the rate. This is directly asymmetric with `get_price_info` in `L1GasPriceProvider`, which has an explicit `max_time_gap_seconds` staleness guard. A stale or timestamp-mismatched ETH/STRK rate flows unchecked into `l1_gas_price_fri` / `l1_data_gas_price_fri` in the block header, corrupting the block hash commitment.

---

### Finding Description

**Root cause — `resolve_query` ignores the oracle's returned timestamp**

`resolve_query` parses the oracle JSON body and accepts the rate as long as `price` is a valid hex string and `decimals == 18`:

```rust
fn resolve_query(body: String) -> Result<u128, EthToStrkOracleClientError> {
    let price = match json.get("price").and_then(|v| v.as_str()) { ... };
    let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)...;
    let decimals = match json.get("decimals").and_then(|v| v.as_u64()) { ... };
    if decimals != ETH_TO_STRK_QUANTIZATION { return Err(...); }
    Ok(rate)   // ← timestamp field in the response is never read
}
``` [1](#0-0) 

The oracle is queried with a specific `adjusted_timestamp` appended as a query parameter:

```rust
url.query_pairs_mut().append_pair("timestamp", &adjusted_timestamp.to_string());
``` [2](#0-1) 

The oracle API returns a `timestamp` field in its response (confirmed by the integration-test mock):

```rust
json!({ "timestamp": query.timestamp, "price": price, "decimals": ETH_TO_STRK_QUANTIZATION })
``` [3](#0-2) 

`resolve_query` never reads `json.get("timestamp")`, so the sequencer cannot detect whether the oracle answered for the correct time period or returned a stale cached value.

**Asymmetry with the L1 gas price staleness check**

`get_price_info` in `L1GasPriceProvider` has an explicit guard:

```rust
if timestamp.0 > (*last_timestamp + self.config.max_time_gap_seconds) {
    return Err(L1GasPriceProviderError::StaleL1GasPricesError { ... });
}
``` [4](#0-3) 

No equivalent guard exists for `eth_to_fri_rate`. The LRU cache in `EthToStrkOracleClient` has no TTL; once a rate is stored under a `quantized_timestamp` key it is returned indefinitely until evicted by capacity pressure. [5](#0-4) 

**Propagation path**

`get_l1_prices_in_fri_and_wei_and_conversion_rate` in the orchestrator calls both `get_price_info` and `get_eth_to_fri_rate` in parallel and multiplies the results:

```rust
let (eth_to_fri_rate, price_info) = tokio::join!(
    l1_gas_price_provider_client.get_eth_to_fri_rate(timestamp),
    l1_gas_price_provider_client.get_price_info(BlockTimestamp(timestamp))
);
// ...
let l1_gas_prices_fri_result = L1PricesInFri::convert_from_wei(&prices_in_wei, eth_to_fri_rate);
``` [6](#0-5) 

A stale `eth_to_fri_rate` silently corrupts `l1_gas_price_fri` and `l1_data_gas_price_fri`. These values are embedded in the `ConsensusBlockInfo` / `ProposalPart::BlockInfo` sent to the consensus manager and ultimately written into the block header. The block hash computation explicitly chains gas prices:

> `calculate_block_hash` chains … gas prices, Starknet version, zero field, and parent hash.

A wrong rate therefore produces a wrong block hash commitment for every block produced while the stale rate is in effect.

---

### Impact Explanation

**Impact: Critical — Incorrect fee/gas/resource accounting with economic impact; wrong block hash commitment.**

- `l1_gas_price_fri` and `l1_data_gas_price_fri` in the block header are computed from the stale rate.
- The block hash, which chains gas prices, is wrong.
- All fee estimates, bouncer resource accounting, and transaction fee deductions during that period use the wrong conversion rate.
- Validators that independently compute the expected gas price (via `is_block_info_valid`) may accept or reject proposals based on a stale oracle value, depending on whether they share the same stale cache state. [7](#0-6) 

---

### Likelihood Explanation

**Likelihood: Medium.**

The oracle is an external HTTP service (`lag_interval_seconds = 900 s` in production). Scenarios that trigger the bug without any attacker:

1. The oracle backend serves a stale cached response for the requested timestamp (common in price-feed APIs under load).
2. The oracle API has a bug and returns the rate for the previous interval instead of the requested one.
3. The oracle is temporarily unreachable; the `EthToStrkOracleClient` falls back to `quantized_timestamp - 1` from its LRU cache with no bound on how old that entry is. [8](#0-7) 

The production `lag_interval_seconds` is 900 s, so a one-bucket fallback already introduces a 15-minute-old rate. There is no upper bound on how many blocks can be committed with the stale rate before the oracle recovers.

---

### Recommendation

1. **Validate the returned timestamp in `resolve_query`**: parse `json.get("timestamp")` and reject the response if it does not match the requested `adjusted_timestamp` (within a configurable tolerance).

2. **Add a staleness guard in `eth_to_fri_rate`** analogous to `max_time_gap_seconds` in `get_price_info`: if `current_wall_time > adjusted_timestamp + max_oracle_staleness_seconds`, return an error instead of a cached rate.

3. **Bound the LRU fallback**: `NUMBER_OF_TIMESTAMPS_BACK` is a hard-coded `1`; make it configurable and add a wall-clock check so that a cached rate older than `N * lag_interval_seconds` is not silently reused. [9](#0-8) 

---

### Proof of Concept

```
1. Oracle API is queried with adjusted_timestamp = T (e.g. T = 1_000_000 s).
2. Oracle backend is slow / has a stale cache; it returns:
       { "price": "0x<rate_for_T-900>", "decimals": 18, "timestamp": T-900 }
3. resolve_query reads only "price" and "decimals" → returns rate_for_T-900 as if it were rate_for_T.
4. EthToStrkOracleClient caches rate_for_T-900 under quantized_timestamp(T).
5. get_l1_prices_in_fri_and_wei_and_conversion_rate multiplies L1 Wei prices by rate_for_T-900.
6. ConsensusBlockInfo is built with wrong l1_gas_price_fri / l1_data_gas_price_fri.
7. calculate_block_hash chains the wrong gas prices → wrong block hash commitment for every
   block produced until the oracle cache entry is evicted or the oracle recovers.
8. All transaction fees charged during this window are computed from the wrong rate.
``` [1](#0-0) [10](#0-9)

fn resolve_query(body: String) -> Result<u128, EthToStrkOracleClientError> {
    let Ok(json): Result<serde_json::Value, _> = serde_json::from_str(&body) else {
        return Err(EthToStrkOracleClientError::ParseError(format!(
            "Failed to parse JSON: {body}"
        )));
    };
    // Extract price from API response. Also returns MissingFieldError if value is not a string.
    let price = match json.get("price").and_then(|v| v.as_str()) {
        Some(price) => price,
        None => {
            return Err(EthToStrkOracleClientError::MissingFieldError("price".to_string(), body));
        }
    };
    let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)
        .expect("Failed to parse price as u128");
    // Extract decimals from API response. Also returns MissingFieldError if value is not a number.
    let decimals = match json.get("decimals").and_then(|v| v.as_u64()) {
        Some(decimals) => decimals,
        None => {
            return Err(EthToStrkOracleClientError::MissingFieldError(
                "decimals".to_string(),
                body,
            ));
        }
    };
    if decimals != ETH_TO_STRK_QUANTIZATION {
        return Err(EthToStrkOracleClientError::InvalidDecimalsError(
            ETH_TO_STRK_QUANTIZATION,
            decimals,
        ));
    }
    ETH_TO_STRK_SUCCESS_COUNT.increment(1);
    set_unix_now_seconds(&ETH_TO_STRK_LAST_SUCCESS_TIMESTAMP_SECONDS);
    ETH_TO_STRK_RATE.set_lossy(rate);
    Ok(rate)
}

    async fn eth_to_fri_rate(&self, timestamp: u64) -> Result<u128, EthToStrkOracleClientError> {
        const NUMBER_OF_TIMESTAMPS_BACK: u64 = 1;
        let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
            .checked_div(self.config.lag_interval_seconds)
            .expect("lag_interval_seconds should be non-zero");

        let mut cache = self.cached_prices.lock().unwrap();

        if let Some(rate) = cache.get(&quantized_timestamp) {
            debug!("Cached conversion rate for timestamp {timestamp} is {rate}");
            return Ok(*rate);
        }

        // Check if there is a query already sent out for this timestamp, if not, start one.
        let mut queries = self.queries.lock().unwrap();
        let handle = queries
            .get_or_insert_mut(quantized_timestamp, || self.spawn_query(quantized_timestamp));
        // If the query is not finished, return an error.
        if !handle.is_finished() {
            debug!("Query not yet resolved: timestamp={timestamp}");
            // If the previous quantized timestamp is in the cache, use it.
            if let Some(rate) = cache.get(&(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)) {
                debug!(
                    "Query not yet resolved: timestamp={timestamp}, using previous rate {rate} \
                     from quantized timestamp={}",
                    (quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)
                        * self.config.lag_interval_seconds
                );
                return Ok(*rate);
            }
            // If not, return a query not ready error.
            return Err(EthToStrkOracleClientError::QueryNotReadyError(timestamp));
        }
