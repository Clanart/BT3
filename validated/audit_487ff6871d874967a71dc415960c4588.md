### Title
Duplicate Keys in `MerkleBlob::batch_insert` Bypass Uniqueness Validation, Corrupting Tree Root and Invalidating DataLayer Proofs - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that the single-entry `insert` function enforces. Passing a `keys_values_hashes` vector containing repeated `KeyId` values silently writes multiple leaf nodes with the same key into the blob, overwrites the `block_status_cache` mapping so only the last duplicate is tracked, and produces a Merkle root that does not correspond to any valid key-value mapping. Proofs of inclusion generated from this corrupted tree are inconsistent with the committed root, and `check_integrity` will report a cache-length mismatch — but only if explicitly called after the fact.

### Finding Description

`MerkleBlob::insert` guards against duplicates at the very start:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` only for the last two elements of the input vector (when the tree has ≤ 1 existing leaf). All remaining elements are written directly via `insert_entry_to_blob` with **no uniqueness check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` unconditionally calls `block_status_cache.add_leaf(index, leaf)` for every leaf block written: [3](#0-2) 

When a duplicate `KeyId` appears, `add_leaf` overwrites the `key_to_index` entry for that key with the new index, but the first physical leaf node remains in the blob at its original index. The result is:

1. **Two physical leaf nodes** with the same `KeyId` exist in the blob at different indexes.
2. **`key_to_index` cache** points only to the last duplicate's index.
3. **The Merkle root** is computed over both leaves (including the phantom duplicate), so it does not represent any valid key-value store state.
4. **`check_integrity`** will detect `leaf_count != key_to_index_cache_length` and return `IntegrityKeyToIndexCacheLength`, but this check is not called automatically after `batch_insert`. [4](#0-3) 

The Python binding `py_batch_insert` only validates that the lengths of the two input lists match; it performs no key-uniqueness check and does not call `check_integrity`: [5](#0-4) 

### Impact Explanation

A DataLayer store operator (or any caller of the Python/wasm binding) who passes a `keys_values` list containing a repeated `KeyId` will silently obtain a `MerkleBlob` whose root hash is computed over a phantom duplicate leaf. Any `get_proof_of_inclusion` call for that key will generate a proof anchored to the last duplicate's tree position, but the root was computed over both leaves. Depending on tree shape, the proof may verify correctly against the corrupted root while representing a state that never existed (wrong value for the key), or it may fail to verify at all. Either outcome corrupts DataLayer state: other nodes verifying proofs against the committed root will reach incorrect inclusion/exclusion conclusions, satisfying the allowed impact "DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."

### Likelihood Explanation

The Python binding is the primary public API for DataLayer operations. Any caller constructing a batch insert from external data (e.g., syncing a DataLayer store from a peer) can trigger this path without any privilege. The single-entry `insert` path is protected; only `batch_insert` is vulnerable. The bug is silent — no error is returned — so it will not be caught unless the caller explicitly invokes `check_integrity`.

### Recommendation

Add a duplicate-key (and duplicate-hash) check at the start of `batch_insert`, mirroring the guards in `insert`. The simplest correct fix is to collect all incoming keys into a `HashSet` before writing any nodes, and return `Error::KeyAlreadyPresent()` if any key appears more than once or already exists in `block_status_cache`. Alternatively, route all elements through the existing `insert` function (accepting the performance cost) or add the cache lookup inside the fast-path loop.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Pre-populate so leaf_count > 1, bypassing the insert() path entirely
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), bytes(range(i, i+32)))

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with a duplicate KeyId (key=99 appears twice)
dup_key = KeyId(99)
kv_ids  = [(dup_key, ValueId(100)), (dup_key, ValueId(200))]
hashes  = [bytes(range(10, 42)), bytes(range(20, 52))]

# No error raised — duplicate silently accepted
blob.batch_insert(kv_ids, hashes)
blob.calculate_lazy_hashes()

root_after = blob.get_root_hash()
# root_after is computed over 5 physical leaves (3 original + 2 duplicates of key=99)
# but key_to_index only tracks 4 unique keys → integrity failure
try:
    blob.check_integrity()
except Exception as e:
    print(f"Integrity failure: {e}")
    # IntegrityKeyToIndexCacheLength: found 5 leaves but cache length is 4

# Proof generated for key=99 is anchored to the last duplicate's position,
# but the root was computed over both → proof may be inconsistent
proof = blob.get_proof_of_inclusion(dup_key)
print(f"Proof valid against corrupted root: {proof.valid()}")

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
