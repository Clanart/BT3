Audit Report

## Title
Excess native ETH stranded by `pay()` is stealable by any caller via unguarded `refundETH()` — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary

When a user calls `exactOutputSingle` or `exactInputSingle` with `msg.value` exceeding the actual swap input required, `pay()` deposits only the exact required amount as WETH and silently leaves the remainder on the router. Because `refundETH()` is an unguarded `external` function that forwards the **entire** router ETH balance to `msg.sender`, any attacker can steal the stranded ETH in a subsequent transaction, causing a direct loss of user principal.

## Finding Description

**Root cause — `pay()` deposits only `value`, not `nativeBalance`:**

In `PeripheryPayments.sol`, when `token == WETH` and `nativeBalance >= value`, the function deposits exactly `value` ETH as WETH and transfers it to the pool. The surplus `nativeBalance - value` is left on the router with no attribution and no automatic refund. [1](#0-0) 

**Root cause — `refundETH()` has zero access control:**

`refundETH()` is `external` with no caller check. It sends `address(this).balance` — the entire router ETH balance — to `msg.sender`, regardless of who originally deposited it. [2](#0-1) 

**Trigger path — `exactOutputSingle` is `payable` and performs no post-swap refund:**

`exactOutputSingle` is `payable` and accepts `msg.value`. After the pool callback settles (paying only the actual `amountIn`), the function calls `_clearExpectedCallbackPool()` and returns — no ETH refund is issued. [3](#0-2) 

The callback `_justPayCallback` calls `pay()` with the pool-determined `amountIn`, not with `msg.value`: [4](#0-3) 

**Existing guards are insufficient:**

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks unsolicited ETH pushes via the fallback; it does not apply when ETH arrives as `msg.value` in a named `payable` function call. The `amountInMaximum` check in `exactOutputSingle` (line 145) only reverts if `amountIn > amountInMaximum`; it does not refund the difference when `amountIn < amountInMaximum`. [5](#0-4) 

**The safe multicall pattern is demonstrated in tests but not enforced:**

The test suite shows the intended pattern — wrapping the swap and `refundETH()` together in a `multicall` — but the swap functions are independently callable and `payable`, so any direct caller bypasses this safety net. [6](#0-5) 

## Impact Explanation

Direct loss of user principal. A user who calls `exactOutputSingle{value: X}` where `X > actualAmountIn` permanently loses `X - actualAmountIn` ETH to any attacker who calls `refundETH()` in the next block. For exact-output swaps, sending `msg.value = amountInMaximum` is the standard and expected pattern because the precise input is not known in advance. The loss is proportional to the user's ETH buffer and is unrecoverable. This meets the Sherlock High/Medium threshold for direct loss of user principal.

## Likelihood Explanation

Medium-High. Exact-output swaps with a native ETH buffer are a standard DeFi UX pattern. Any user or integrator calling `exactOutputSingle` or `exactOutput` directly (outside a multicall) will strand ETH whenever `actualAmountIn < amountInMaximum`. An attacker needs only to monitor the mempool for transactions that leave a non-zero ETH balance on the router and call `refundETH()` in the next block. No special privileges, approvals, or setup are required.

## Recommendation

Auto-refund any remaining native ETH balance to `msg.sender` at the end of `exactOutputSingle`, `exactOutput`, and `exactInputSingle` when `tokenIn == WETH`:

```solidity
// After _clearExpectedCallbackPool():
uint256 remaining = address(this).balance;
if (remaining > 0) _transferETH(msg.sender, remaining);
```

Alternatively, track the original depositor in transient storage and restrict `refundETH()` to only refund to that address. Do not rely solely on the multicall pattern as a safety net, since the swap functions are independently callable and `payable`.

## Proof of Concept

1. User calls `exactOutputSingle{value: 2 ETH}` with `tokenIn = WETH`, `amountInMaximum = 2 ETH`; the pool requires only `1 ETH` as actual input.
2. Inside `metricOmmSwapCallback`, `_justPayCallback` calls `pay(WETH, user, pool, 1 ETH)`.
3. `pay()` observes `nativeBalance = 2 ETH >= value = 1 ETH`, deposits 1 ETH as WETH, and sends it to the pool. The remaining 1 ETH stays on the router.
4. `exactOutputSingle` returns; no refund is issued. Router holds 1 ETH.
5. Attacker calls `refundETH()` in the next transaction. `refundETH()` sends `address(this).balance = 1 ETH` to the attacker.
6. User paid 2 ETH but received only the swap output worth 1 ETH of input. Net loss: 1 ETH.

A Foundry test can reproduce this by: (a) calling `exactOutputSingle{value: amountInMaximum}` directly (not via multicall) with `amountInMaximum > actualAmountIn`, (b) asserting `address(router).balance > 0` after the swap, (c) pranking an attacker address and calling `router.refundETH()`, and (d) asserting the attacker received the stranded ETH.

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
