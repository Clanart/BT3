Audit Report

## Title
Unguarded `refundETH()` / `sweepToken()` / `unwrapWETH9()` Allow Any Caller to Steal Router-Held ETH and Tokens Stranded by the `pay()` WETH Partial-Wrap Path — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()`, `sweepToken()`, and `unwrapWETH9()` are public functions with no access control that sweep the router's entire ETH or token balance to any caller-chosen address. The `pay()` function's WETH branch deliberately wraps only `value` wei when `nativeBalance >= value`, leaving the surplus `nativeBalance - value` on the router with no attribution to the original sender. Any attacker can call `refundETH()` in a follow-up transaction to steal that stranded ETH before the victim can reclaim it.

## Finding Description
`pay()` in `PeripheryPayments.sol` handles native ETH as a WETH substitute:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // wraps only `value`, not nativeBalance
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) { ... }
    else { ... }
}
``` [1](#0-0) 

When `nativeBalance >= value`, only `value` wei is wrapped and forwarded to the pool. The surplus `nativeBalance - value` remains on the router with no attribution to the original sender.

`refundETH()` is callable by anyone with no ownership check:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // no caller check
    }
}
``` [2](#0-1) 

`sweepToken()` and `unwrapWETH9()` share the same pattern — they sweep the full router balance to any caller-supplied `recipient` with no ownership check: [3](#0-2) 

`exactInputSingle` is `payable` and passes `msg.sender` as payer and `params.amountIn` as `value` to `pay()`. If a user calls it directly (outside a multicall) with `msg.value > params.amountIn`, the surplus is silently stranded on the router: [4](#0-3) 

The intended safe usage is to bundle `exactInputSingle` + `refundETH` inside a single `multicall`, which is atomic. The test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` demonstrates this pattern: [5](#0-4) 

However, `exactInputSingle` is `payable` and the `pay()` WETH branch silently accepts any `msg.value`, so a user who calls `exactInputSingle` directly with excess ETH (outside a multicall) strands the surplus on the router with no automatic refund and no on-chain signal.

`MetricOmmPoolLiquidityAdder` inherits the same `PeripheryPayments` base and exposes the same three public helpers, creating an identical surface: [6](#0-5) 

## Impact Explanation
**Medium — direct loss of user principal.** A victim who calls `exactInputSingle{value: V}` with `amountIn < V` loses `V - amountIn` ETH to any attacker who calls `refundETH()` before the victim does. The same loss applies to ERC-20 tokens or WETH stranded on the router via `sweepToken` or `unwrapWETH9`. The loss is bounded by the surplus sent in a single transaction but is fully extractable with zero cost to the attacker.

## Likelihood Explanation
**Medium.** The `exactInputSingle` and `exactInput` functions are `payable` and the `pay()` WETH branch silently consumes any native balance, giving users no on-chain signal that excess ETH will be stranded. Users familiar with Uniswap v3 patterns may send excess ETH expecting automatic refund. Front-running the victim's own `refundETH` call is trivial on any mempool-visible chain.

## Recommendation
1. **Auto-refund excess ETH** at the end of `exactInputSingle` / `exactOutputSingle` when `tokenIn == WETH`, mirroring Uniswap v3's approach of calling `refundETH` internally after the swap.
2. **Restrict `refundETH` to the multicall initiator** — store the initiating address in transient storage at the start of each top-level `multicall` call and require it in `refundETH`.
3. **Restrict `sweepToken` and `unwrapWETH9`** similarly, or require that `recipient == msg.sender` when called outside a multicall context.
4. **Document clearly** that calling any swap function with `msg.value > amountIn` outside a `multicall` + `refundETH` bundle will strand the surplus.

## Proof of Concept
```
Setup:
  - pool: legitimate WETH/token1 pool registered on factory
  - victim: EOA with 1 ether

Step 1 — Victim calls exactInputSingle directly (not via multicall):
  router.exactInputSingle{value: 1 ether}(ExactInputSingleParams{
      pool:             pool,
      tokenIn:          WETH,
      zeroForOne:       true,
      amountIn:         1_000,          // only 1000 wei needed
      amountOutMinimum: 0,
      recipient:        victim,
      ...
  });

  Internally, pay() sees nativeBalance = 1 ether >= value = 1000:
    WETH.deposit{value: 1000}();
    WETH.transfer(pool, 1000);
    // 1 ether - 1000 wei remains on router — no revert, no refund

Step 2 — Attacker (separate tx, same block or next):
  router.refundETH();
  // _transferETH(msg.sender = attacker, address(this).balance = ~1 ether)
  // Attacker receives ~1 ether; victim's surplus is gone.
```
