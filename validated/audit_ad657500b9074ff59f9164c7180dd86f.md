### Title
`ProofOfInclusion.valid()` Validates Only Internal Self-Consistency, Not Against a Committed Root — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` derives its own root from the proof's last layer (`combined_hash`) and then compares the computed chain against that same self-supplied value. It never accepts an external, committed root hash as a parameter. Any caller that relies solely on `proof.valid()` — as the entire public API, tests, fuzz targets, and Python bindings encourage — will accept a completely fabricated proof for any arbitrary `node_hash`.

### Finding Description

`root_hash()` returns the `combined_hash` field of the last layer in the proof itself:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← sourced entirely from the proof object
    } else {
        self.node_hash
    }
}
```

`valid()` then checks that each layer's `calculated_hash` equals `layer.combined_hash`, and after the loop compares `existing_hash` (which was just set to the last `calculated_hash`, which was already asserted equal to the last `layer.combined_hash`) against `self.root_hash()` (which is also `last.combined_hash`):

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;   // ← now equals layer.combined_hash
    }
    existing_hash == self.root_hash()      // ← tautologically true when layers exist
}
```

After the loop, `existing_hash` equals the last `layer.combined_hash` (proven by the loop invariant). `self.root_hash()` also returns the last `layer.combined_hash`. The final comparison is therefore always `true` whenever the proof has at least one layer. The function only verifies that the hash chain is internally self-consistent; it never anchors the chain to any externally committed tree root.

An attacker can construct a `ProofOfInclusion` for an arbitrary `node_hash` H (e.g., a leaf they do not own) by:
1. Choosing any `other_hash` and `other_hash_side`.
2. Computing `combined_hash = calculate_internal_hash(H, side, other_hash)`.
3. Repeating for as many layers as desired.

The resulting proof passes `valid()` unconditionally, regardless of what the actual DataLayer tree root is.

The misleading API is reinforced throughout the codebase. The fuzz target calls only `proof.valid()` with no root check:

```rust

assert!(proof.valid());
```

The Rust unit tests do the same:

```rust

```

The Python integration tests do the same:

```python

```

The Python type stub exposes `valid()` and `root_hash()` as independent methods with no documentation requiring callers to also check `root_hash()` against a known value. Any downstream consumer (chia-blockchain, DataLayer sync logic, wallet code) that calls `proof.valid()` as the sole gate will accept forged proofs.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A remote, unprivileged attacker who can submit a `ProofOfInclusion` object — via the Python binding, the Streamable wire format, or JSON — can prove membership of any arbitrary key/hash in any DataLayer store without possessing a valid proof. This enables:
- Falsely proving a key-value pair is committed in a DataLayer singleton when it is not.
- Bypassing DataLayer state verification in any protocol that relies on `proof.valid()` as the sole check.

### Likelihood Explanation

The API design actively encourages the vulnerable pattern. The method is named `valid()`, implying completeness. Every test, fuzz target, and code example in the repository calls `valid()` alone. Any downstream Python or Rust consumer following these examples will be vulnerable. The likelihood that at least one production consumer of the Python wheel calls `proof.valid()` without separately checking `proof.root_hash() == known_root` is high.

### Recommendation

`valid()` must accept the expected root hash as a mandatory parameter and compare the computed chain against it, not against the proof's own `combined_hash`:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // anchor to external committed root
}
```

Alternatively, keep `root_hash()` but rename `valid()` to `is_internally_consistent()` and add a separate `verify(expected_root: &Hash) -> bool` that performs the complete check. Update all call sites, the Python binding, and the fuzz target accordingly.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side, ProofOfInclusion, ProofOfInclusionLayer};
use chia_datalayer::calculate_internal_hash;

// Attacker wants to forge a proof that node_hash = FAKE_LEAF is in the tree.
let fake_leaf: Hash = [0xAA; 32];
let other_hash: Hash = [0xBB; 32];
let side = Side::Left;

// Compute a consistent combined_hash from attacker-chosen inputs.
let combined = calculate_internal_hash(&fake_leaf, side, &other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: fake_leaf,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: side,
        other_hash,
        combined_hash: combined,  // self-consistent, but not the real tree root
    }],
};

// valid() returns true — proof is accepted despite being completely fabricated.
assert!(forged_proof.valid());
// root_hash() returns `combined`, not the real tree root — but callers never check.
