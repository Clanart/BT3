### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the new class hash supplied by a contract corresponds to a previously declared class. The code itself acknowledges this gap with a `TODO` comment. An unprivileged contract deployer or caller can exploit this to permanently freeze funds held in any contract whose upgrade path accepts user-controlled input, by replacing the contract's class with an undeclared (non-existent) hash.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall as follows:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The function accepts any arbitrary felt as `class_hash` and writes it directly into the contract's `StateEntry` in `contract_state_changes` without checking whether that hash exists in `contract_class_changes`. The `dict_update` call at line 906 unconditionally commits the new (potentially invalid) class hash to the state. [2](#0-1) 

By contrast, the `execute_declare_transaction` path in `transaction_impls.cairo` enforces `prev_value=0` to prevent double-declaration and calls `finalize_class_hash` to verify the class hash pre-image:

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [3](#0-2) 

No equivalent guard exists in `execute_replace_class`. The OS-level invariant — that a contract's `class_hash` field must always reference a declared class — is not enforced at the syscall layer.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, every subsequent call to that contract will fail at the entry-point dispatch stage because the OS cannot locate the class bytecode. There is no recovery path: the contract cannot call `replace_class` again (since execution of any entry point fails), and no other mechanism exists to restore the class hash. Any ERC-20 balance, vault deposit, or other asset held in the contract's storage becomes permanently inaccessible.

This is the direct analog of the external report's "Rug Contract's Holdings" scenario: instead of an admin upgrading to a malicious implementation, the OS's missing validation allows a contract's class to be silently corrupted, achieving the same permanent loss of user funds.

---

### Likelihood Explanation

The `replace_class` syscall is the standard StarkNet upgrade primitive. Any contract that exposes an upgrade entry point with insufficient access control — or that accepts a user-supplied class hash argument — is directly exploitable. The attacker's entry path requires only:

1. Identifying or deploying a contract that calls `replace_class` with attacker-influenced input (e.g., a contract whose `upgrade` function does not restrict the caller or validate the hash).
2. Submitting a transaction that passes an undeclared felt as the new class hash.

The OS performs no cross-check against `contract_class_changes`, so the invalid hash is committed to state without any proof-level rejection. The acknowledged `TODO` comment confirms the development team is aware the check is absent.

---

### Recommendation

Before committing the new class hash in `execute_replace_class`, verify that the hash exists as a key in `contract_class_changes` (i.e., that a `dict_read` on `contract_class_changes` for `class_hash` returns a non-zero compiled class hash). This mirrors the invariant already enforced during `execute_declare_transaction` and closes the gap identified in the TODO comment.

---

### Proof of Concept

1. **Attacker deploys** a contract `VaultProxy` whose `upgrade(new_class: felt)` entry point calls `replace_class(new_class)` without validating the caller or the hash.
2. **Victim deposits** funds into `VaultProxy` (e.g., via an ERC-20 `transfer` that credits the contract's storage).
3. **Attacker calls** `VaultProxy.upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. The OS executes `execute_replace_class`:
   - Gas is deducted.
   - `GetContractAddressStateEntry` hint fetches the current `StateEntry`.
   - A new `StateEntry` with `class_hash=0xdeadbeef` is written via `dict_update` — **no existence check performed**.
5. The block is proven and the state transition is accepted.
6. Any subsequent call to `VaultProxy` (e.g., `withdraw`) fails at class dispatch because `0xdeadbeef` has no associated bytecode.
7. **All funds in `VaultProxy` are permanently frozen.** [4](#0-3)
