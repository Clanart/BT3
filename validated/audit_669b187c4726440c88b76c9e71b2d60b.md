Looking at the `batch_insert` function in the DataLayer `MerkleBlob` for the analog to the "identifier reuse / registry bypass" vulnerability class.

### Title
`batch_insert` Bypasses Key-Uniqueness Registry, Enabling Duplicate-Key Injection and Corrupted Merkle Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `insert` enforces, for every item in the batch when the tree already holds ≥ 2 leaves. An unprivileged caller (via the Python binding `py_batch_insert`) can supply a batch containing a key that already exists in the tree, or two identical keys within the same batch. The result is multiple `LeafNode` entries with the same `KeyId` written into the blob, a silently-overwritten `key_to_index` cache entry, an orphaned leaf that is invisible to the cache but still participates in hash computation, and a Merkle root that does not represent any valid key-value mapping. Proofs of inclusion generated against this corrupted root are structurally valid yet prove a false state.

---

### Finding Description

**Guard in `insert` (the safe path)**

`insert` enforces two uniqueness invariants before touching the blob:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

**Guard bypass in `batch_insert`**

`batch_insert` calls `insert` (with its guards) only for the last two items popped from the input vector, and only when the tree currently has ≤ 1 leaves. Every other item is written directly via `insert_entry_to_blob`, which performs no uniqueness check:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // guarded
    }
}
for ((key, value), hash) in keys_values_hashes {
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // NO guard
    ...
}
``` [2](#0-1) 

**Silent overwrite in `add_leaf`**

`insert_entry_to_blob` calls `add_leaf`, which uses `HashMap::insert` — a silent overwrite:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // overwrites previous entry
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

**Consequence: blob/cache divergence**

After a duplicate-key batch:
- The blob contains two `LeafNode` blocks with the same `KeyId` at distinct tree indexes.
- `key_to_index` retains only the last index; the earlier leaf is orphaned — present in the blob, absent from the cache.
- `calculate_lazy_hashes` traverses the full blob tree (including the orphaned leaf) and propagates hashes upward, so the root hash is computed over a tree that contains the phantom leaf.
- `check_integrity` would detect the divergence (`leaf_count` from blob > `key_to_index` cache length → `IntegrityKeyToIndexCacheLength` error), but it is never called automatically after `batch_insert`. [4](#0-3) 

**Python binding — direct attacker entry point**

`py_batch_insert` accepts arbitrary `keys_values` and `hashes` lists from Python with no additional validation beyond length matching:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

---

### Impact Explanation

The corrupted root hash does not correspond to any valid key-value mapping. Any downstream consumer that:
1. Calls `get_root_hash()` and commits it on-chain, or
2. Calls `get_proof_of_inclusion(key)` and serves the resulting proof to a verifier,

will produce a proof that is structurally valid against the corrupted root but proves a false state. A verifier comparing the committed root against an independently computed root will see a mismatch, causing DataLayer state disagreement between nodes. An attacker who controls the committed root (e.g., a DataLayer publisher who controls what gets batch-inserted) can craft a root that makes a deleted or never-inserted key appear provably present, or makes a present key appear absent via a forged exclusion.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion path used in the DataLayer sync workflow and is directly exposed as a Python binding callable by any code with access to a `MerkleBlob` object. The DataLayer protocol accepts data from external publishers; a malicious publisher can submit a batch containing a repeated key. No privilege escalation or key compromise is required — only the ability to supply a list of `(KeyId, ValueId, Hash)` tuples. The condition that triggers the bypass (tree has ≥ 2 leaves) is the normal operating state for any non-trivial DataLayer store.

---

### Recommendation

Add the same uniqueness guards at the top of `batch_insert` that `insert` already enforces, before any item is written to the blob:

1. Collect all keys and hashes from the input vector.
2. Check each key against `block_status_cache.contains_key(key)` and each hash against `block_status_cache.contains_leaf_hash(hash)`.
3. Also check for intra-batch duplicates (two entries with the same key or hash within the same call).
4. Return `Err(Error::KeyAlreadyPresent())` or `Err(Error::HashAlreadyPresent())` on any violation, before any blob mutation occurs.

Alternatively, call `check_integrity()` after every `batch_insert` as a defense-in-depth measure, though this is O(n) and should not replace the pre-check.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
from chia_rs.sized_bytes import bytes32
import hashlib

def h(n: int) -> bytes32:
    return bytes32(hashlib.sha256(n.to_bytes(8, "big")).digest())

blob = MerkleBlob(blob=bytearray())

# Seed the tree with 3 leaves so leaf_count > 1 (bypasses the guarded path)
blob.batch_insert(
    [(KeyId(1), ValueId(1)), (KeyId(2), ValueId(2)), (KeyId(3), ValueId(3))],
    [h(1), h(2), h(3)],
)
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# Now batch_insert with a duplicate of KeyId(1) — bypasses KeyAlreadyPresent check
blob.batch_insert(
    [(KeyId(1), ValueId(99))],   # KeyId(1) already exists
    [h(99)],
)
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# root_after != root_before, yet get_keys_values() shows only one entry for KeyId(1)
# The blob contains two LeafNodes for KeyId(1); the root is computed over both.
# A proof of inclusion for KeyId(1) is valid against root_after,
# but root_after does not represent a clean key-value mapping.
assert root_before != root_after, "root corrupted by phantom leaf"
proof = blob.get_proof_of_inclusion(KeyId(1))
assert proof.valid(), "proof valid against corrupted root"
# check_integrity() would raise IntegrityKeyToIndexCacheLength here
```
