### Title
Unbounded `waiting_for_block` accumulation in `SpiceChunkValidatorActor` allows memory exhaustion via pre-validation witness/contract-accesses flood — (`File: chain/client/src/spice/chunk_validator_actor.rs`)

---

### Summary

`SpiceChunkValidatorActor` stores incoming `SpiceChunkStateWitness` and `SpiceChunkContractAccesses` messages in a plain, unbounded `HashMap` (`waiting_for_block`) before performing any size, distance-from-head, or signature validation. An attacker who can deliver messages to the actor — including via the network path for contract-accesses, which are stored before signature verification — can grow this map without bound, exhausting validator memory and crashing the node.

---

### Finding Description

`SpiceChunkValidatorActor` holds:

```rust
waiting_for_block:
    HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>,
``` [1](#0-0) 

This map is initialized as an empty `HashMap::new()` with no capacity bound. [2](#0-1) 

**Path 1 — witness deferral (no validation before storage):**

When `process_chunk_state_witness` is called and the referenced block is absent from the store, the full witness is pushed into `waiting_for_block` with zero validation — no size check, no distance-from-head check, no signature check:

```rust
// TODO(spice): Implement additional checks (size limit, distance to head)
// before adding witness to `waiting_for_block`. See non-spice handle_orphan_witness().
self.waiting_for_block.entry(chunk_id.block_hash).or_default().0.push(witness);
``` [3](#0-2) 

**Path 2 — contract-accesses deferral (stored before signature verification):**

In `handle_spice_contract_accesses`, when `get_epoch_id` fails because the block is not yet in the store, the `SpiceChunkContractAccesses` message is pushed into `waiting_for_block` before any signature check:

```rust
// TODO(spice): Implement additional checks (size limit, distance to head)
// before adding accesses to `waiting_for_block`. See non-spice handle_orphan_witness().
self.waiting_for_block.entry(chunk_id.block_hash).or_default().1.push(accesses);
``` [4](#0-3) 

Signature verification only occurs after the epoch is resolved (lines 528–541), which never happens for a fabricated block hash. [5](#0-4) 

**Contrast with the non-Spice path:**

The non-Spice orphan witness pool uses a bounded `LruCache` with a configurable capacity, and the code comment in the Spice path explicitly references this as the model to follow:

```rust
OrphanStateWitnessPool {
    witness_cache: LruCache::new(NonZeroUsize::new(cache_capacity).unwrap()),
}
``` [6](#0-5) 

The `partial_chunk_data` LruCache in the same actor is also bounded (`MAX_PENDING_CHUNKS = 24`), but `waiting_for_block` is not: [7](#0-6) 

---

### Impact Explanation

A validator node running the `protocol_feature_spice` build can be crashed via OOM. Each `SpiceChunkStateWitness` can be up to `MAX_UNCOMPRESSED_STATE_WITNESS_SIZE` (64 MiB) and each `SpiceChunkContractAccesses` entry adds further heap. Because `waiting_for_block` is never evicted (entries are only removed when the matching block arrives, which never happens for fabricated hashes), the map grows monotonically until the process is killed by the OS. A crashed validator loses its endorsement slot for the current epoch, degrading consensus liveness.

---

### Likelihood Explanation

For the contract-accesses path (Path 2), the message is stored **before** signature verification, meaning any peer that can deliver a `SpiceChunkContractAccessesMessage` to the actor with an unknown `block_hash` triggers the accumulation. The TODO comment in the production code explicitly acknowledges the missing guard. For the witness path (Path 1), the witness reaches the actor after Reed-Solomon reconstruction by the data distributor, which requires a threshold of producer-signed parts; however, the same missing size/distance guard applies once a valid witness arrives for a slow or withheld block.

---

### Recommendation

1. Apply a per-block-hash entry count cap and a total byte budget to `waiting_for_block`, mirroring the `OrphanStateWitnessPool` LruCache pattern.
2. For contract-accesses, perform at minimum a distance-from-head check and a message-count-per-block-hash cap **before** deferring into `waiting_for_block`, so that signature verification is not the only gate.
3. Resolve the two `TODO(spice)` comments at lines 266 and 522 of `chain/client/src/spice/chunk_validator_actor.rs`.

---

### Proof of Concept

```
1. Build nearcore with `protocol_feature_spice`.
2. Connect a peer to a Spice validator node.
3. Send a stream of SpiceChunkContractAccessesMessage messages, each referencing
   a distinct, fabricated block_hash not present in the validator's store.
   Each message passes the MAX_CONTRACTS_PER_REQUEST check (line 501) with a
   small contract list, so it is not dropped there.
4. Because get_epoch_id returns EpochError::MissingBlock for each fabricated hash,
   each accesses object is pushed into waiting_for_block without signature check.
5. Repeat until the validator process is killed by OOM.
``` [8](#0-7)
