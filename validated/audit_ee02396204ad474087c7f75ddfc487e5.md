### Title
Fee Computation Arithmetic Overflow via Unbounded `max_price_per_unit` Bypasses Fee Collection - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo` and `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

An unprivileged V3 transaction sender can craft resource bounds with a `max_price_per_unit` value that causes `compute_max_possible_fee` to overflow the Stark field prime to zero. When `max_fee == 0`, `charge_fee` returns early without executing any ERC-20 transfer, allowing the attacker to execute transactions with zero fee payment. This is the direct analog of the Launchpad negative-value bug: just as a negative input inverted the token flow there, an unbounded field-element input here inverts the fee accounting — the user pays nothing instead of paying the computed fee.

---

### Finding Description

**Vulnerability class**: Fee/accounting bug — arithmetic overflow in fee computation due to missing upper-bound validation on `max_price_per_unit`.

In `pack_resource_bounds` (`transaction_hash.cairo`, lines 103–107), `max_amount` is correctly bounded to `[0, 2^64 - 1]` via `assert_nn_le`, but `max_price_per_unit` is only checked to be non-negative via `assert_nn` — no upper bound is enforced: [1](#0-0) 

In Cairo, `assert_nn(x)` constrains `x` to `[0, (P−1)/2]` where P is the Stark prime (~2^251). This means `max_price_per_unit` can be as large as ~2^250.

`compute_max_possible_fee` (`transaction_impls.cairo`, lines 87–102) then computes:

```
l1_gas.max_amount * l1_gas.max_price_per_unit
+ l2_gas.max_amount * (l2_gas.max_price_per_unit + tip)
+ l1_data_gas.max_amount * l1_data_gas.max_price_per_unit
```

All arithmetic is modular over P with no overflow guard: [2](#0-1) 

An attacker can choose `max_price_per_unit` values such that the sum of the three products ≡ 0 (mod P). This is a simple linear equation over the felt field — with six free parameters (three `max_amount` values and three `max_price_per_unit` values), solutions are trivially constructible.

When `compute_max_possible_fee` returns 0, `charge_fee` exits immediately at the early-return guard: [3](#0-2) 

No ERC-20 transfer to the sequencer is executed. The transaction is fully processed — validate and execute entry points run — but zero fee is collected.

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

The sequencer/protocol receives zero fee for a fully executed transaction. An attacker can:
1. Execute arbitrary transactions (invoke, declare, deploy-account) at zero cost.
2. Spam the network with zero-cost transactions, exhausting sequencer resources and causing a network halt (High impact, secondary).

The fee token ERC-20 transfer that should debit the user's account is never called, so the user's balance is never reduced. The sequencer's balance is never credited. This is a direct, permanent accounting loss.

---

### Likelihood Explanation

**Likelihood: High.**

- The attacker is any V3 transaction sender — no privilege required.
- Crafting the overflow requires solving one linear congruence mod P: trivial arithmetic, no cryptographic difficulty.
- The transaction passes all hash-integrity checks because `pack_resource_bounds` only enforces `assert_nn` (not an upper bound) on `max_price_per_unit`, so the crafted values are accepted as valid by the hash verification step.
- No special tooling is needed beyond a standard StarkNet transaction submission client.

---

### Recommendation

In `pack_resource_bounds`, add an explicit upper-bound check on `max_price_per_unit` to constrain it to a safe range (e.g., `[0, 2^128 − 1]`):

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // ADD THIS
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
```

With `max_amount ≤ 2^64 − 1` and `max_price_per_unit ≤ 2^128 − 1`, the maximum product per term is `(2^64 − 1) × (2^128 − 1) < 2^192`, and the sum of three such terms is `< 3 × 2^192 ≪ P`, making overflow impossible.

Additionally, add a post-computation `assert_nn` on the result of `compute_max_possible_fee` to catch any future overflow paths.

### Proof of Concept

**Goal**: Craft a V3 transaction where `compute_max_possible_fee` returns 0.

Let P = Stark prime = 3618502788666131213697322783095070105623107215331596699973092056135872020481.

**Step 1**: Choose `l2_gas.max_amount = 0`, `l1_data_gas.max_amount = 0` (zeroing two terms).

**Step 2**: The fee reduces to `l1_gas.max_amount * l1_gas.max_price_per_unit mod P`.

**Step 3**: Set `l1_gas.max_amount = 1` and `l1_gas.max_price_per_unit = P − 1`. But `P − 1 > (P−1)/2`, so `assert_nn` would reject it.

**Step 4** (correct approach): Set `l1_gas.max_amount = 2` and `l1_gas.max_price_per_unit = (P+1)/2`. Then `2 * (P+1)/2 = P + 1 ≡ 1 mod P`. Now set `l1_data_gas.max_amount = 1`, `l1_data_gas.max_price_per_unit = P − 1`... 

More directly: choose `l1_gas.max_amount = 2`, `l1_gas.max_price_per_unit = (P−1)/2`. Product = `2 * (P−1)/2 = P − 1 ≡ −1 mod P`. Set `l1_data_gas.max_amount = 1`, `l1_data_gas.max_price_per_unit = 1`. Sum = `−1 + 1 = 0 mod P`. Both `max_price_per_unit` values pass `assert_nn` (they are in `[0, (P−1)/2]`).

**Result**: `compute_max_possible_fee` returns 0. `charge_fee` hits the guard at line 123 and returns without charging any fee. The transaction executes fully at zero cost. [4](#0-3) [5](#0-4) [6](#0-5)

```text
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
