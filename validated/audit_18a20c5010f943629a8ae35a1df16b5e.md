### Title
Unvalidated Destination Chain in `init_transfer` Causes Permanent Token Loss — (`File: near/omni-bridge/src/lib.rs`)

### Summary
The NEAR bridge's `init_transfer` function accepts any `ChainKind` (except `Near`) as a destination without verifying that the token is registered for that chain. Tokens are burned or locked before any such check, and no cancel/refund path exists. If the destination chain has no token registration, the transfer is permanently stuck in `pending_transfers` and the tokens are irrecoverably lost.

### Finding Description

`init_transfer` in `near/omni-bridge/src/lib.rs` performs only one destination-chain check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

Any of the other 12 `ChainKind` variants (Eth, Sol, Arb, Base, Bnb, Btc, Zcash, Pol, HyperEvm, Strk, Abs, Fogo) is accepted unconditionally. The function then calls `init_transfer_internal`, which:

1. Burns deployed/bridged tokens via `burn_tokens_if_needed` (a fire-and-forget detached promise — no rollback):

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
``` [2](#0-1) 

2. Calls `lock_tokens_if_needed` for the destination chain. If the `(chain_kind, token_id)` pair has no entry in `locked_tokens`, `lock_tokens` silently returns `LockAction::Unchanged` — no accounting is recorded:

```rust
let Some(current_amount) = self.locked_tokens.get(&key) else {
    return LockAction::Unchanged;
};
``` [3](#0-2) 

3. Stores the transfer in `pending_transfers` and returns `U128(0)` — signalling to the NEP-141 token contract that all tokens were consumed (no refund): [4](#0-3) 

Later, when a relayer calls `sign_transfer`, it calls `get_token_address(destination_chain, token_id)`. If the token is not registered for that chain, the call panics:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [5](#0-4) 

`sign_transfer` is `#[trusted_relayer]`-gated, so the user cannot call it themselves. There is no `cancel_transfer` or user-accessible refund function. The transfer is permanently stuck in `pending_transfers`.

### Impact Explanation

- **Deployed/bridged tokens** (e.g., a token bridged from Ethereum to NEAR): `burn_tokens_if_needed` destroys them via a detached async burn. The tokens are gone with no recovery path.
- **Native NEAR tokens**: locked inside the bridge contract with no mechanism to release them.

Both outcomes are **irrecoverable permanent loss** of user funds, matching the allowed impact: *"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

The `ChainKind` enum has 13 variants. A token is typically registered on only a subset of chains (e.g., a token may exist on Eth but not yet on Fogo, HyperEvm, or Abs). A user who specifies a valid but unregistered destination chain — due to a frontend bug, stale UI, or manual API call — triggers this path. No special privilege is required; any token holder calling `ft_transfer_call` can reach it.

### Recommendation

In `init_transfer`, before burning or locking any tokens, validate that:
1. The token is registered for the destination chain (`token_id_to_address` has an entry for `(destination_chain, token_id)`).
2. A factory is registered for the destination chain (`factories.get(&destination_chain)` is `Some`).

If either check fails, revert immediately (return `transfer_message.amount` to refund the caller) before any state mutation or token destruction occurs.

### Proof of Concept

1. Token `foo.near` is a deployed bridge token registered only for `ChainKind::Eth`.
2. User calls `ft_transfer_call` on `foo.near` with:
   ```json
   { "InitTransfer": { "recipient": "sol:2xNweLHLqbS9YpP3UyaPrxKqgqoC6yPBFyuLxA8qtgr4", "fee": "0", "native_token_fee": "0" } }
   ```
3. `ft_on_transfer` → `init_transfer` passes the `!= Near` check.
4. `init_transfer_internal` calls `burn_tokens_if_needed("foo.near", amount)` — tokens are burned (detached promise, no rollback).
5. `lock_tokens_if_needed(ChainKind::Sol, "foo.near", amount)` — `locked_tokens` has no `(Sol, foo.near)` entry → silently returns `Unchanged`.
6. Transfer stored in `pending_transfers`; `ft_on_transfer` returns `U128(0)` — no refund.
7. Relayer calls `sign_transfer` → `get_token_address(ChainKind::Sol, "foo.near")` returns `None` → panics with `ERR_FAILED_TO_GET_TOKEN_ADDRESS`.
8. Transfer is permanently stuck; burned tokens are unrecoverable. [6](#0-5) [7](#0-6) [8](#0-7)
