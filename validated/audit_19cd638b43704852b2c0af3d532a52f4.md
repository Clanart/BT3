### Title
L1→L2 Message Not Consumed When L1 Handler Is Reverted, Causing Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `execute_l1_handler_transaction`, when the sequencer marks an L1 handler transaction as reverted (`is_reverted != FALSE`), the function returns early **without** calling `consume_l1_to_l2_message`. The L1→L2 message is therefore never written to the OS output and is never marked as consumed on L1. Any funds locked in the L1 messaging contract remain frozen until the sender manually cancels the message after a protocol-enforced timeout — or permanently if the sender never does so.

---

### Finding Description

`execute_l1_handler_transaction` (lines 374–452) begins by loading the sequencer-supplied `is_reverted` hint:

```cairo
%{ StartTx %}
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();          // ← early exit, no message consumption
}
``` [1](#0-0) 

Only when `is_reverted == FALSE` does execution continue to:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [2](#0-1) 

`non_reverting_select_execute_entry_point_func` hard-asserts that the handler did **not** revert:

```cairo
let (is_reverted, retdata_size, retdata, is_deprecated) = select_execute_entry_point_func{...}(...);
assert is_reverted = 0;
``` [3](#0-2) 

This creates an inescapable constraint: if the L1 handler execution would revert (e.g., `ERROR_ENTRY_POINT_NOT_FOUND` returned from `get_entry_point`, or the contract logic panics), the sequencer **cannot** include the transaction with `is_reverted = FALSE` — the proof would be unsound. The only valid option is `is_reverted = TRUE`. But that path returns before `consume_l1_to_l2_message` is ever called, so the message is absent from the OS output and the L1 contract never receives proof of consumption.

Additionally, `check_is_reverted` — the function that is supposed to validate the hint — is a complete no-op:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [4](#0-3) 

There is no Cairo-level constraint that bounds what value the sequencer may supply for `is_reverted` on an L1 handler, making the early-exit path fully reachable.

The entry-point lookup that produces `is_reverted = 1` inside `execute_entry_point` is:

```cairo
if (success == 0) {
    %{ ExitCall %}
    let (retdata: felt*) = alloc();
    assert retdata[0] = ERROR_ENTRY_POINT_NOT_FOUND;
    return (is_reverted=1, retdata_size=1, retdata=retdata);
}
``` [5](#0-4) 

---

### Impact Explanation

When an L1→L2 message targets a contract whose handler reverts (or does not exist), the OS proof is generated without that message in the consumed-messages output. The L1 StarkNet core contract never receives proof that the message was processed, so it remains in the pending state. The ETH or ERC-20 tokens deposited alongside the message are locked in the L1 contract. Recovery requires the original L1 sender to invoke the multi-step cancellation flow and wait out the protocol-mandated delay (currently five days). If the sender is unaware of this mechanism or the sending address is a contract without cancellation logic, the funds are permanently frozen.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

Any unprivileged L1 account can trigger this condition by sending a message to:

1. An L2 address that has no deployed contract (class hash = 0, no L1 handler entry point).
2. An L2 contract whose `@l1_handler` always panics or runs out of gas.
3. An L2 contract that was upgraded after the message was sent, removing the targeted selector.

All three scenarios are reachable without any privileged access. Scenario 1 in particular requires only a single L1 transaction to a valid-looking but undeployed address.

---

### Recommendation

`consume_l1_to_l2_message` must be called **unconditionally**, before the `is_reverted` branch, so that the message is always recorded in the OS output regardless of handler outcome. The reverted path should still skip state-changing execution, but the message consumption must not be gated on execution success:

```cairo
// Always consume the message first.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);

if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}

// Execute only if not reverted.
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(...);
```

This mirrors the fix described in the external report: rather than allowing the message to become undeliverable, the protocol should always acknowledge receipt and handle the failure gracefully.

---

### Proof of Concept

1. **L1 attacker** calls the StarkNet L1 core contract's `sendMessageToL2`, targeting an L2 address with no deployed contract, attaching 1 ETH as the message fee/value.
2. The sequencer picks up the pending L1→L2 message and attempts to build an `L1_HANDLER` transaction.
3. Simulation shows the handler would fail with `ERROR_ENTRY_POINT_NOT_FOUND` (no L1 handler entry point exists). The sequencer sets `is_reverted = TRUE` in the hint.
4. The OS executes `execute_l1_handler_transaction`. At line 387, `is_reverted != FALSE` is true; the function returns at line 389.
5. `consume_l1_to_l2_message` (line 444) is **never reached**. The message is absent from `outputs.messages_to_l2`.
6. The proven OS output contains no record of this message being consumed.
7. The L1 core contract, upon verifying the proof, finds no consumption record and leaves the message in the pending queue.
8. The 1 ETH remains locked. The attacker has permanently frozen the sender's funds with a single L1 transaction. [6](#0-5)
