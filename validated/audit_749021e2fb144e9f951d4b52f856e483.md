### Title
`compute_meta_tx_v0_hash` Omits Nonce, Enabling Unlimited Signature Replay — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` constructs the signed hash for a `meta_tx_v0` syscall without committing to any nonce. Combined with the OS explicitly skipping nonce enforcement for version-0 transactions, an attacker who observes a valid meta-tx signature on-chain can replay it an unlimited number of times against the same target contract, executing the same calldata repeatedly with no authorization from the account owner.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

The resulting hash is purely a function of `(INVOKE_PREFIX, 0, contract_address, selector, calldata_hash, 0, chain_id)`. No nonce is committed to. Compare this with `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1, additional_data=&nonce` to bind the hash to a specific nonce value. [2](#0-1) 

In `execute_meta_tx_v0`, the `new_tx_info` is constructed with `nonce=0` hardcoded: [3](#0-2) 

The OS nonce enforcement function `check_and_increment_nonce` explicitly returns early for any version-0 transaction, so no nonce is ever checked or incremented for the meta-tx: [4](#0-3) 

The three defects compose into a complete replay path:
1. Hash has no nonce → same inputs always produce the same hash.
2. `new_tx_info.nonce` is hardcoded to 0 → no state is advanced.
3. Nonce enforcement is skipped for version 0 → no on-chain guard exists.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any `meta_tx_v0` whose calldata encodes a financial operation (e.g., ERC-20 `transfer`, withdrawal, approval) can be replayed by an unprivileged attacker indefinitely. Because the hash is deterministic over `(contract_address, selector, calldata, chain_id)` and the signature is valid for that hash forever, the attacker needs only to copy the signature from a previously included transaction and re-submit it inside any new outer transaction. The target account's funds can be fully drained in successive replays.

---

### Likelihood Explanation

**High.** Every `meta_tx_v0` syscall and its accompanying signature are visible in the public transaction trace. No privileged access, leaked key, or special network position is required. Any observer of the chain can extract the `(calldata, signature)` pair and replay it immediately. The only precondition is that the target contract has been used with `meta_tx_v0` at least once.

---

### Recommendation

Bind the meta-tx hash to a per-account nonce, mirroring the pattern used by `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // <-- add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=1,   // <-- commit to nonce
        additional_data=&nonce,
    );
    return tx_hash;
}
```

In `execute_meta_tx_v0`, read the current account nonce from `state_entry`, pass it to `compute_meta_tx_v0_hash`, and then increment it in `contract_state_changes` — exactly as `check_and_increment_nonce` does for version ≥ 1 transactions.

---

### Proof of Concept

1. Alice submits an outer invoke transaction that internally calls the `meta_tx_v0` syscall targeting `contract_address=C`, `selector=__execute__`, `calldata=[transfer(Bob, 100 STRK)]`, with a valid ECDSA signature `σ` over `H(INVOKE_PREFIX, 0, C, __execute__, calldata_hash, 0, chain_id)`.

2. The transaction is included in a block. The signature `σ` and calldata are now public.

3. Attacker Eve constructs a new outer invoke transaction that calls `meta_tx_v0` with the identical `contract_address`, `selector`, `calldata`, and `σ`.

4. The OS recomputes `meta_tx_hash = H(INVOKE_PREFIX, 0, C, __execute__, calldata_hash, 0, chain_id)` — identical to step 1 because no nonce is included. [5](#0-4) 

5. The account contract's `__validate__` verifies `σ` against `meta_tx_hash` — it passes, because the hash is identical.

6. `__execute__` runs, transferring another 100 STRK to Bob (or to Eve if she modifies the calldata — but the signature would then fail; she must keep calldata identical). Eve repeats until Alice's balance is zero.

The root cause — `additional_data_size=0` in `compute_meta_tx_v0_hash` — is the necessary and sufficient condition for this attack. [6](#0-5)
