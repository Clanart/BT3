### Title
Stale `feemarketParams` Captured at Ante Handler Construction Causes Cosmos Tx Fee Validation to Use Outdated Base Fee - (File: `evmd/ante/handler_options.go`)

### Summary
`newCosmosAnteHandler` reads `feemarketParams` and `evmParams` from the store exactly once at construction time and closes over them. Every subsequent Cosmos transaction is validated against those frozen values. Because the EIP-1559 base fee is recalculated and written to the store on every `BeginBlock`, the fee-market parameters used by `NewDynamicFeeChecker` and `MinGasPriceDecorator` diverge from the live chain state immediately after the ante handler is built, allowing Cosmos-SDK transactions to be accepted and committed with fees that are below the current base fee.

### Finding Description

`newCosmosAnteHandler` in `evmd/ante/handler_options.go` captures fee-market state at construction time:

```go
func newCosmosAnteHandler(ctx sdk.Context, options HandlerOptions, ...) sdk.AnteHandler {
    evmParams      := options.EvmKeeper.GetParams(ctx)          // snapshot
    feemarketParams := options.FeeMarketKeeper.GetParams(ctx)   // snapshot
    ...
    txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)
    ...
    cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams),
``` [1](#0-0) 

Both `NewDynamicFeeChecker` and `MinGasPriceDecorator` receive a pointer to the local `feemarketParams` variable. That variable is never updated after construction.

Inside `NewDynamicFeeChecker`, the base fee is derived exclusively from the captured struct:

```go
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
``` [2](#0-1) 

`types.GetBaseFee` receives `feemarketParams` (not a live `ctx` store read), so it can only return `feemarketParams.BaseFee` — the value frozen at construction.

`MinGasPriceDecorator` similarly reads only the captured field:

```go
minGasPrice := mpd.feemarketParams.MinGasPrice
``` [3](#0-2) 

By contrast, the Ethereum ante handler (`newEthAnteHandler`) calls `EVMBlockConfig(ctx, ...)` on **every** transaction, fetching a fresh base fee from the store each time:

```go
blockCfg, err := options.EvmKeeper.EVMBlockConfig(ctx, options.EvmKeeper.ChainID())
...
baseFee := blockCfg.BaseFee
``` [4](#0-3) 

The base fee is updated every block in `BeginBlock`:

```go
func (k *Keeper) BeginBlock(ctx sdk.Context) error {
    baseFee := k.CalculateBaseFee(ctx)
    ...
    k.SetBaseFee(ctx, baseFee)
``` [5](#0-4) 

`SetBaseFee` writes the new value into the persistent KV store via `SetParams`:

```go
func (k Keeper) SetBaseFee(ctx sdk.Context, baseFee *big.Int) {
    params := k.GetParams(ctx)
    params.BaseFee = ethermint.SaturatedNewInt(baseFee)
    err := k.SetParams(ctx, params)
``` [6](#0-5) 

The store is updated, but the `feemarketParams` pointer held by the Cosmos ante handler decorators still points to the stale snapshot.

### Impact Explanation

Every Cosmos-SDK transaction (non-EVM) that goes through `newCosmosAnteHandler` has its fee validated against the base fee that was current when the node started (or when the ante handler was last rebuilt). As the chain runs and the EIP-1559 base fee rises due to block congestion, the gap between the stale validation value and the live value grows. An unprivileged user can submit Cosmos transactions with a `feeCap` that satisfies the stale (lower) base fee but is below the current base fee. The `DeductFeeDecorator` then deducts the effective fee computed from the stale base fee, committing the transaction with under-paid fees. This is a fee mis-accounting bug in the ante handler path that permits invalid transactions (by current fee-market rules) to commit.

Additionally, if `MinGasPrice` is raised via governance, the old (lower) value continues to be enforced for Cosmos transactions, bypassing the governance-mandated floor.

### Likelihood Explanation

The base fee changes every block. With default parameters (`BaseFeeChangeDenominator = 8`, `ElasticityMultiplier = 2`), the base fee can increase by up to 12.5 % per block under full load. After even a modest period of chain activity the stale value diverges materially. Any user who observes the discrepancy can exploit it by submitting Cosmos transactions with fees calibrated to the stale base fee. No special privileges are required.

### Recommendation

Fetch `feemarketParams` and `evmParams` from the store inside the returned `sdk.AnteHandler` closure (i.e., per-transaction), exactly as `newEthAnteHandler` does with `EVMBlockConfig`. Replace the one-time snapshot with a live store read:

```go
func newCosmosAnteHandler(options HandlerOptions, ...) sdk.AnteHandler {
    return func(ctx sdk.Context, tx sdk.Tx, simulate bool) (sdk.Context, error) {
        evmParams       := options.EvmKeeper.GetParams(ctx)
        feemarketParams := options.FeeMarketKeeper.GetParams(ctx)
        ...
    }
}
```

### Proof of Concept

1. Node starts; `newCosmosAnteHandler` is called once. Suppose the base fee at that moment is **1 Gwei**.
2. The chain runs under high load for N blocks. `BeginBlock` raises the live base fee to **2 Gwei** in the KV store.
3. An attacker submits a Cosmos SDK transaction (e.g., a `MsgSend`) with `feeCap = 1.5 Gwei`.
4. `NewDynamicFeeChecker` evaluates `feeCap (1.5) >= baseFee (1, stale)` → **passes**.
5. `DeductFeeDecorator` deducts `effectiveFee = 1.5 Gwei * gas` from the sender.
6. The transaction commits. The correct fee at the current base fee would have required at least `2 Gwei * gas`. The attacker paid 25 % less than required by the live fee market.
7. Ethereum transactions submitted in the same block correctly enforce the 2 Gwei base fee via `EVMBlockConfig`, demonstrating the asymmetry.
