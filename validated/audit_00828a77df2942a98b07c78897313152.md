Audit Report

## Title
Unprivileged `refundETH()` allows any caller to steal leftover ETH from users who send excess `msg.value` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` transfers the router's entire ETH balance to `msg.sender` with no access control. When a user calls a payable swap function (e.g., `exactInputSingle`) with `msg.value > amountIn` and `tokenIn = WETH`, the `pay()` helper wraps only `amountIn` worth of ETH and leaves the surplus in the router. Any attacker can then call `refundETH()` in a subsequent transaction to drain that surplus.

## Finding Description
`refundETH()` is an unrestricted external function that sends the entire contract ETH balance to `msg.sender`: [1](#0-0) 

The `pay()` function, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH and leaves the remainder in the contract with no automatic refund: [2](#0-1) 

`exactInputSingle` is `payable` and accepts ETH directly (bypassing the `receive()` guard which only blocks non-WETH direct ETH sends), and does not auto-refund surplus after `_clearExpectedCallbackPool()`: [3](#0-2) 

The router has no per-user ETH accounting. The intended safe pattern — wrapping `exactInputSingle` + `refundETH()` in a single `multicall` — is demonstrated in tests but is not enforced at the contract level: [4](#0-3) 

Attack path:
1. User calls `exactInputSingle{value: 1 ether}` with `tokenIn=WETH`, `amountIn=0.5 ether` (no `refundETH()` in the same multicall).
2. `pay()` wraps exactly 0.5 ETH → sends WETH to pool. 0.5 ETH remains in the router.
3. Attacker calls `refundETH()` in a separate tx → receives 0.5 ETH.

No privileged role, no malicious pool, no non-standard ERC20 behavior needed.

## Impact Explanation
Direct loss of user ETH principal. Any ETH sent as `msg.value` in excess of `amountIn` is immediately claimable by any third party via `refundETH()`. This meets the Critical/High direct loss of user principal threshold under Sherlock contest rules.

## Likelihood Explanation
Moderate-to-high. Users interacting with the router directly (not via a frontend that constructs the correct multicall), or users who miscalculate `amountIn` vs `msg.value`, will leave ETH stranded. MEV bots routinely monitor for exactly this pattern. The Uniswap v3 periphery has the same design, and real-world exploits of this pattern have occurred.

## Recommendation
Either:
1. Track the depositor per call (e.g., transient storage mapping `address → uint256 ethDeposited`) and restrict `refundETH()` to return only the caller's own deposited amount, or
2. Auto-refund surplus ETH at the end of each payable swap function (after `_clearExpectedCallbackPool()`), eliminating the need for a separate `refundETH()` call.

## Proof of Concept
```solidity
// 1. User sends 1 ether but only swaps 0.5 ether worth of WETH
router.exactInputSingle{value: 1 ether}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        tokenIn: WETH,
        tokenOut: token1,
        zeroForOne: true,
        amountIn: 0.5 ether,   // only 0.5 ETH consumed by pay()
        amountOutMinimum: 0,
        recipient: user,
        deadline: block.timestamp + 60,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// router.balance == 0.5 ether at this point

// 2. Attacker calls refundETH() in a separate tx
vm.prank(attacker);
router.refundETH();

// 3. Attacker receives 0.5 ether; user's ETH is gone
assertEq(attacker.balance, 0.5 ether);
assertEq(address(router).balance, 0);
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
