### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary felt value as the new class hash without verifying it corresponds to a declared contract class. This allows any contract deployer to permanently brick a contract by replacing its class with an undeclared hash, freezing all funds held within it. The OS itself contains a TODO comment acknowledging this missing check.

---

### Finding Description

In `execute_replace_class`, the OS reads the requested class hash directly from the syscall request and writes it into the contract state without any validation: [1](#0-0) 

The critical missing check is explicitly acknowledged by the developers: [2](#0-1) 

The `class_hash` field comes directly from `request.class_hash`, which is fully attacker-controlled contract calldata. No assertion is made that this hash exists in `contract_class_changes` (the declared class registry) or in `compiled_class_facts_bundle`.

When a subsequent call is made to the bricked contract, `execute_entry_point` performs: [3](#0-2) 

`find_element` panics if the compiled class hash is not found in the bundle. Because the sequencer controls OS input, it must exclude all transactions targeting the bricked contract to avoid OS-level failure — permanently making the contract uncallable and its funds inaccessible.

This is structurally analogous to the Shelter M-19 bug: just as the Shelter `client` could reset `activated[_token]` to prevent user withdrawals, a StarkNet contract deployer can overwrite the contract's class hash with an invalid value to permanently prevent any interaction with the contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 balances, ETH, or other assets held in a contract whose class hash has been replaced with an undeclared value are permanently inaccessible. The contract cannot be called (the OS panics on `find_element`), and there is no recovery path — the state change is committed to the Merkle tree via `dict_update` in `execute_replace_class`. [4](#0-3) 

---

### Likelihood Explanation

The attack is directly reachable by any unprivileged contract deployer:

1. The attacker deploys a contract implementing a token vault or DeFi interface (class A).
2. Users deposit funds into the vault.
3. The attacker's contract calls `replace_class` with an arbitrary invalid felt (e.g., `0xDEAD`).
4. The OS writes the invalid class hash into the contract's `StateEntry` with no validation.
5. All subsequent calls to the vault revert at the OS level; funds are permanently frozen.

No privileged role, leaked key, or trusted operator is required. The attacker only needs to deploy a contract and attract user deposits — a standard DeFi attack surface.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to state, validate that it corresponds to a declared class. Specifically, perform a `dict_read` on `contract_class_changes` for the requested `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// After: let class_hash = request.class_hash;
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed implicitly in `execute_entry_point` when looking up the compiled class, but enforces it at the point of the `replace_class` syscall so the state is never corrupted.

---

### Proof of Concept

**Step 1**: Attacker deploys `VaultContract` (class hash `0xABC`) implementing `deposit()` and `withdraw()`.

**Step 2**: Users call `deposit()`, transferring tokens into `VaultContract`. The contract's storage now holds user balances.

**Step 3**: Attacker sends a transaction calling a hidden `brick()` function in `VaultContract`, which internally issues:
```
replace_class(class_hash=0xDEAD)  // 0xDEAD is not a declared class
```

**Step 4**: `execute_replace_class` in the OS writes `StateEntry(class_hash=0xDEAD, ...)` for `VaultContract`'s address with no validation.

**Step 5**: Any user calling `withdraw()` triggers `execute_entry_point`, which calls:
```cairo
dict_read{dict_ptr=contract_class_changes}(key=0xDEAD)  // returns 0
find_element(..., key=0)  // panics: element not found
```

**Step 6**: The sequencer must exclude all transactions targeting `VaultContract`. User funds are permanently frozen.

The root cause — the missing declared-class validation in `execute_replace_class` — is confirmed by the explicit TODO at line 898 of `syscall_impls.cairo`. [5](#0-4)

func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

    // Reduce gas.
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );

    assert [revert_log] = RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=state_entry.class_hash);
    let revert_log = &revert_log[1];

    return ();
}
