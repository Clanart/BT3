### Title
Missing Minimum Transfer Amount Validation Causes Permanent Fund Lock via Normalization-to-Zero — (File: near/omni-bridge/src/lib.rs)

### Summary
The NEAR bridge's `init_transfer` function accepts any positive token amount without enforcing a minimum that accounts for cross-chain decimal normalization. When a user sends an amount smaller than the decimal-scaling factor between NEAR and the destination chain, the normalized amount rounds to zero. The subsequent `sign_transfer` call then permanently fails, leaving the user's tokens irrecoverably locked or burned in the NEAR bridge contract with no on-chain recovery path.

### Finding Description
`init_transfer` in `near/omni-bridge/src/lib.rs` validates only that `fee < amount`: [1](#0-0) 

No minimum amount floor is enforced. The transfer message is stored and the tokens are locked/burned immediately: [2](#0-1) 

Later, when a relayer calls `sign_transfer`, the stored amount is passed through `normalize_amount` to convert it to the destination chain's decimal representation: [3](#0-2) 

If the NEAR-side token has more decimals than the destination chain token (e.g., 24 vs. 6), any amount below `10^(24−6) = 10^18` units normalizes to zero via integer division. The `require!(amount_to_transfer > 0, ...)` guard then panics, the transaction reverts, and the transfer message remains permanently in `pending_transfers` with no built-in cancellation or refund path.

The entry point is the public `ft_on_transfer` callback, which any NEP-141 token contract invokes when a user calls `ft_transfer_call`: [4](#0-3) 

### Impact Explanation
Tokens are locked (native tokens) or burned (deployed/bridged tokens) inside `init_transfer_internal` before any normalization check occurs: [5](#0-4) 

Because `sign_transfer` is the only path to complete or cancel a pending transfer, and it will always revert for a zero-normalized amount, the funds are permanently unclaimable. This matches the Critical impact class: **permanent freezing / irrecoverable lock of user funds in the bridge flow**.

### Likelihood Explanation
The condition is reachable by any unprivileged user who calls `ft_transfer_call` with a dust amount. For tokens where the NEAR representation uses significantly more decimals than the destination chain (a common configuration — e.g., 24 NEAR decimals vs. 6 EVM decimals), the threshold below which amounts normalize to zero is `10^18` base units, which is 1 full NEAR-side token. A user sending less than 1 full token unit triggers the lock. No special role or privileged access is required.

### Recommendation
Add a minimum-amount guard inside `init_transfer` (or inside `init_transfer_internal`) that rejects transfers whose normalized amount would be zero. Concretely, look up the destination-chain decimals at initiation time and require:

```
normalize_amount(amount - fee, decimals) > 0
```

before locking or burning tokens. This mirrors the guard already present in `sign_transfer` but moves it to the point where the user can still receive a refund.

### Proof of Concept
1. Token `foo.near` is registered with 24 decimals on NEAR and 6 decimals on Ethereum.
2. User calls `ft_transfer_call` on `foo.near` with `amount = 1` (1 yocto-foo) and `msg` encoding an `InitTransfer` to an Ethereum recipient with `fee = 0`.
3. `ft_on_transfer` → `init_transfer` succeeds: `fee (0) < amount (1)` passes; 1 yocto-foo is burned; transfer message is stored.
4. Relayer calls `sign_transfer` for the new `TransferId`.
5. `normalize_amount(1, {near: 24, eth: 6})` = `1 / 10^18` = `0` (integer division).
6. `require!(0 > 0, ...)` panics → transaction reverts.
7. Transfer message remains in `pending_transfers` forever; 1 yocto-foo is permanently destroyed; no refund is possible. [6](#0-5) [3](#0-2)
