### Title
Fast-Transfer Relayer Funds Permanently Lost When `send_tokens` Fails After State Is Already Committed — (`near/omni-bridge/src/lib.rs`)

### Summary

In `process_fin_transfer_to_other_chain` and `utxo_fin_transfer_fast`, the bridge commits all state changes (marking the transfer as finalized, removing or finalizing the fast-transfer record) **before** the token transfer to the relayer is dispatched, and that token transfer is fired with `.detach()` — no callback, no failure handling. If the `send_tokens` call fails, the relayer's fronted tokens are permanently unrecoverable and the transfer ID is irrevocably consumed.

### Finding Description

**Instance 1 — `process_fin_transfer_to_other_chain`**

When a fast-transfer relayer has already fronted tokens for a cross-chain transfer and the canonical `fin_transfer` proof arrives, the bridge executes the following sequence:

1. `self.add_fin_transfer(...)` — inserts the transfer ID into `finalised_transfers` (replay guard, irreversible).
2. Detects the existing fast-transfer record and calls `self.send_tokens(token, relayer, amount, "").detach()` — fire-and-forget, no callback.
3. Immediately calls `self.mark_fast_transfer_as_finalised(...)` — fast-transfer record is now permanently finalized. [1](#0-0) [2](#0-1) 

If `send_tokens` fails (e.g., `ft_transfer` panics because the relayer account lacks storage on the token contract, or the token contract is paused), the relayer receives nothing. Because `finalised_transfers` already contains the transfer ID, no retry of `fin_transfer` is possible. Because `fast_transfers` is marked `finalised: true`, no other recovery path exists.

**Instance 2 — `utxo_fin_transfer_fast`**

The same pattern appears in the UTXO fast-transfer settlement path. The fast-transfer record is removed or marked finalised **before** `send_tokens(...).detach()` is called. The codebase even contains an explicit acknowledgment of the unresolved problem:

```rust
if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
    // TODO: check how to deal with failed send_tokens
    return self.utxo_fin_transfer_fast(fast_transfer, status, utxo_fin_transfer_msg);
}
``` [3](#0-2) [4](#0-3) 

In both instances the root cause is identical to the external report: **state is committed before the downstream operation is confirmed, and no bounce/failure handler exists to roll back the state**.

### Impact Explanation

A fast-transfer relayer who fronted tokens for a user transfer loses their entire principal permanently if the `send_tokens` call fails after state has been committed. The `finalised_transfers` set prevents any re-submission of the proof, and the `fast_transfers` record is already removed or marked finalised, so no protocol-level retry or refund path exists. This constitutes an **irrecoverable lock of relayer funds** in the bridge contract.

### Likelihood Explanation

`send_tokens` dispatches either `ft_transfer` or `mint` on an external token contract. Failure conditions include: the relayer account not having a storage deposit on the token contract at the time of settlement, the token contract being paused or upgraded between fast-transfer initiation and canonical finalization, or an out-of-gas condition in the token contract. These are realistic operational conditions, not theoretical-only scenarios. The UTXO path even carries a developer TODO acknowledging the unhandled failure case.

### Recommendation

- **Short term:** Replace `.detach()` with a chained callback (analogous to `resolve_fast_transfer` used in the NEAR-recipient fast-transfer path) that checks the result of `send_tokens`. On failure, revert the fast-transfer finalization state and remove the transfer ID from `finalised_transfers` so the relayer can retry.
- **Long term:** Establish a uniform invariant: no bridge state that prevents re-execution (replay guards, finalization flags) is committed until the token delivery to the recipient/relayer is confirmed successful. Document the expected behavior when token delivery fails after proof verification.

### Proof of Concept

1. Relayer R calls `ft_transfer_call` on the token contract with a `FastFinTransferMsg`, fronting tokens for a user transfer from Ethereum to Solana. `add_fast_transfer` records R as the relayer.
2. The canonical `fin_transfer` proof arrives. `process_fin_transfer_to_other_chain` is called.
3. At line 1985, `add_fin_transfer` inserts the transfer ID into `finalised_transfers`.
4. At lines 2029–2039, `send_tokens(token, R, amount, "").detach()` is called. Suppose R's account has no storage deposit on the token contract — `ft_transfer` panics inside the token contract.
5. Because the call is detached, the panic is silently ignored. The bridge transaction succeeds.
6. At line 2040, `mark_fast_transfer_as_finalised` sets `finalised: true` for R's fast-transfer record.
7. R attempts to re-submit the proof: `add_fin_transfer` panics with `ERR_TRANSFER_ALREADY_FINALISED`.
8. R's fronted tokens remain locked in the bridge contract with no recovery mechanism. [5](#0-4) [6](#0-5)
