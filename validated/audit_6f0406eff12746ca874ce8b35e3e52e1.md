### Title
Integer Overflow in Relative Timelock Arithmetic Enables Timelock Bypass When `nowrap=false` — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false`, the relative timelock checks (`ASSERT_SECONDS_RELATIVE`, `ASSERT_HEIGHT_RELATIVE`) use `wrapping_add` for the deadline arithmetic. An attacker-controlled spend with a crafted overflow value causes the computed deadline to wrap around to a small integer, making the `<` guard evaluate to `false` and allowing the spend to pass validation even though the timelock has not expired.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` has two code paths controlled by the `nowrap` boolean:

- `nowrap=true` → `saturating_add` (correct: overflow clamps to `u64::MAX`/`u32::MAX`, timelock always fails)
- `nowrap=false` → `wrapping_add` (incorrect: overflow wraps to a small value, timelock silently passes)

For `ASSERT_SECONDS_RELATIVE`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

If `unspent.timestamp = T` and `seconds_relative = u64::MAX - T + 1` (or any value that causes `T + seconds_relative` to overflow), then `wrapping_add` produces a value ≤ `T`. The current `timestamp` (which is ≥ `T`) is then **not** less than the wrapped result, so no error is returned and the spend is accepted — despite the timelock not having expired.

The same flaw applies to `ASSERT_HEIGHT_RELATIVE` via `u32::wrapping_add`. [1](#0-0) 

The inverse overflow also affects `ASSERT_BEFORE_SECONDS_RELATIVE` and `ASSERT_BEFORE_HEIGHT_RELATIVE`: wrapping to a small value causes the `>=` check to fire a **false rejection** of a valid spend. [2](#0-1) 

The `nowrap` flag is a caller-supplied parameter exposed directly through the Python binding `py_check_time_locks`, meaning the `nowrap=false` path is a live production code path, not dead code. [3](#0-2) 

---

### Impact Explanation

This is a **High** impact finding matching the allowed scope:

> *Signature, aggregate signature, synthetic key, puzzle type, condition, timelock, or coin-id validation bypass enables unauthorized spend acceptance or replay.*

A coin protected by `ASSERT_SECONDS_RELATIVE N` (where `N` is chosen to overflow) can be spent **before** the relative timelock has elapsed. Any puzzle that relies on `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` for time-locked custody (vaults, escrows, clawback coins, DID recovery, etc.) is vulnerable to premature spend when the node evaluates with `nowrap=false`.

---

### Likelihood Explanation

- The `nowrap=false` path is the **legacy pre-softfork behavior** retained for backward compatibility with older blocks. Nodes processing historical blocks or running without the softfork flag active will use this path.
- The attacker input is entirely unprivileged: the overflow value is embedded in the CLVM solution's condition list, which any coin spender controls.
- The overflow threshold is deterministic and trivially computable from the coin's `confirmed_block_index` / `timestamp`, both of which are public on-chain data.
- The codebase's own tests explicitly document and confirm the bypass behavior: [4](#0-3) [5](#0-4) 

---

### Recommendation

Replace `wrapping_add` with `saturating_add` in both the `seconds_relative` and `height_relative` branches of `check_time_locks`, unconditionally. The `nowrap` flag was introduced to distinguish old from new behavior, but the wrapping behavior is never semantically correct for a timelock: a timelock with a value that overflows the deadline type should always be treated as unsatisfiable (i.e., reject the spend), not silently bypassed.

Alternatively, validate at condition-parse time (in `conditions.rs`) that `coin_timestamp + seconds_relative` does not overflow, and reject the spend bundle with `AssertSecondsRelativeFailed` if it does, before `check_time_locks` is ever reached.

---

### Proof of Concept

Given a coin confirmed at timestamp `T = 1000`:

1. Attacker crafts a spend with `ASSERT_SECONDS_RELATIVE = 0xFFFF_FFFF_FFFF_FC17` (i.e., `u64::MAX - 999`).
2. `wrapping_add(1000, 0xFFFF_FFFF_FFFF_FC17)` = `0` (wraps to zero).
3. `check_time_locks` evaluates: `current_timestamp < 0` → `false` → **no error, spend accepted**.
4. The actual required wait is `~584 million years`; the spend is accepted immediately.

This is confirmed by the existing test fixture: [6](#0-5) 

The `nowrap=False` column returns `None` (no error) for all overflow cases, confirming the bypass is reachable and deterministic. [7](#0-6)
