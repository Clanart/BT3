Audit Report

## Title
`addLiquidityWeighted` unconditionally reverts with `SharesRoundedToZero` for small depositors — (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary

`_scaleWeightsToShares` computes a WAD-scaled multiplier from `min(max0/need0, max1/need1)` and applies it to each bin's weight. When a user's token budget is small relative to the pool's probed depth, the multiplier `scaleWad` falls below `WAD / w.shares[i]`, causing `Math.mulDiv(w.shares[i], scaleWad, WAD)` to truncate to zero. The unconditional `revert SharesRoundedToZero()` guard at line 241 then fires, making `addLiquidityWeighted` permanently unusable for any depositor whose budget is below the per-share threshold of the current pool state.

## Finding Description

The two-phase flow in `addLiquidityWeighted` (lines 88–116) first probes the pool to obtain `need0`/`need1`, then calls `_scaleWeightsToShares` (lines 226–243) to convert weight deltas into integer share counts:

```solidity
// lines 231-241
uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;
...
out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
```

When `max0 ≪ need0`, `scaleWad` is much smaller than `WAD`. For any bin where `w.shares[i] * scaleWad < WAD`, integer division truncates `out.shares[i]` to zero and the revert fires unconditionally. There is no fallback: the guard does not skip the bin, clamp to 1, or allow partial fills. The alternative `addLiquidityExactShares` requires the caller to supply exact share counts, which is precisely what `addLiquidityWeighted` is designed to compute — so it is not a viable substitute for proportional depositors.

Concrete example confirming the path:
- `maxAmountToken0 = 1e6`, `need0 = 1e12` → `scaleWad = Math.mulDiv(1e6, 1e18, 1e12) = 1e12`
- `w.shares[0] = 1` → `out.shares[0] = Math.mulDiv(1, 1e12, 1e18) = 0` → `revert SharesRoundedToZero()`

The condition `w.shares[i] * scaleWad < WAD` simplifies to `maxAmount / need < 1 / w.shares[i]`, which is easily satisfied in any pool with significant existing liquidity.

## Impact Explanation

`addLiquidityWeighted` is the primary interface for proportional liquidity provision. Any depositor whose token budget is below the per-share threshold of the current pool depth cannot use this function. Because `need0`/`need1` are determined dynamically by the probe (they depend on live pool depth and cursor position), users cannot predict in advance whether their call will revert. This constitutes an unusable core liquidity flow, matching the allowed impact criterion "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."

## Likelihood Explanation

The condition is unprivileged and requires no malicious setup. In any pool with significant existing liquidity, `need` values are large, raising the threshold. Any retail depositor providing amounts that are a small fraction of total pool depth will trigger the revert. The condition is reachable on every call to `addLiquidityWeighted` where `maxAmount / need < 1 / w.shares[i]`, and is repeatable across all callers and pool states with sufficient depth.

## Recommendation

Replace the unconditional revert with a skip or clamp:

**Option A – skip zero-share bins** (requires filtering before the paying `addLiquidity` call if the pool rejects zero-share entries):
```solidity
if (w.shares[i] != 0 && out.shares[i] == 0) {
    out.binIdxs[i] = w.binIdxs[i];
    out.shares[i] = 0;
    continue;
}
```

**Option B – clamp to minimum 1 share:**
```solidity
if (w.shares[i] != 0 && out.shares[i] == 0) {
    out.shares[i] = 1;
}
```

Option B preserves the user's intent (deposit into every specified bin) at the cost of slightly over-depositing in truncated bins. Option A silently skips bins that cannot receive shares, which may surprise callers expecting all bins to be filled.

## Proof of Concept

1. Deploy a pool with sufficient token0 depth so that a unit-weight probe returns `need0 = 1e12` (e.g., 1 million USDC at 6 decimals).
2. Call `addLiquidityWeighted` with `maxAmountToken0 = 1e6`, `maxAmountToken1 = type(uint256).max`, and a single-bin `weightDeltas` with `shares[0] = 1`.
3. Phase 1 (probe): `addLiquidity` with `KIND_PROBE` reverts with `LiquidityProbe(1e12, 0)`.
4. `_scaleWeightsToShares` computes `scaleWad0 = Math.mulDiv(1e6, 1e18, 1e12) = 1e12`; `scaleWad1 = type(uint256).max`; `scaleWad = 1e12`.
5. Loop: `out.shares[0] = Math.mulDiv(1, 1e12, 1e18) = 0`.
6. Guard fires: `revert SharesRoundedToZero()`.
7. Transaction reverts despite the user supplying a non-zero token budget and a non-zero weight. [1](#0-0) [2](#0-1)

```text
  function _scaleWeightsToShares(LiquidityDelta calldata w, uint256 max0, uint256 max1, uint256 need0, uint256 need1)
    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
    uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;

    uint256 n = w.binIdxs.length;
    out.binIdxs = new int256[](n);
    out.shares = new uint256[](n);
    for (uint256 i; i < n; i++) {
      out.binIdxs[i] = w.binIdxs[i];
      out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
      if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
    }
  }
```
