### Title
Missing Duplicate-Key Guard in `MerkleBlob::batch_insert()` Silently Corrupts DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert()` omits the duplicate-key and duplicate-hash checks that `MerkleBlob::insert()` enforces. When the tree already contains ≥2 leaves, any key (or hash) that already exists in the tree — or that appears more than once within the batch itself — is silently written as a second leaf node. The `BlockStatusCache` overwrites its index pointer for that key without error, leaving the old leaf orphaned in the blob. The resulting tree has two physical leaf nodes for the same key, producing a corrupted Merkle root and invalidating proofs of inclusion/exclusion.

### Finding Description

`MerkleBlob::insert()` enforces two guards before writing:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` only calls `insert()` (and therefore only runs those guards) for the first two items when the tree has ≤1 leaves. For every subsequent item — the entire fast path when the tree already has ≥2 leaves — it calls `insert_entry_to_blob()` directly, with no existence check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no key/hash guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob()` calls `block_status_cache.add_leaf()`, which unconditionally overwrites the `key_to_index` and `leaf_hash_to_index` maps:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
``` [3](#0-2) 

After the call the blob contains two physical leaf nodes for the same key. The cache points only to the newer one; the older leaf is unreachable via the cache but is still traversed by `calculate_lazy_hashes()` and all iterators, so it is included in the computed Merkle root. The root hash therefore reflects a tree with a phantom duplicate leaf, while `get_proof_of_inclusion()` generates proofs against the single cache-tracked leaf — a structural mismatch.

`check_integrity()` would detect this (it verifies cache ↔ blob consistency), but it is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`), not in production. [4](#0-3) 

The Python binding `py_batch_insert` passes caller-supplied data directly to `batch_insert()` with no deduplication:

```rust
// lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

### Impact Explanation

The DataLayer Merkle root is committed on-chain. A corrupted root — one computed over a tree containing a phantom duplicate leaf — is permanently wrong. Any proof of inclusion generated from that tree is verified against a root that does not correspond to the logical key-value set. This allows:

- **Forged inclusion proofs**: a key can appear to be proven present at a position that does not match the committed root.
- **Corrupted tree roots**: the on-chain root hash diverges from the true root of the intended key-value set.
- **Cross-subscriber state disagreement**: two subscribers syncing the same DataLayer store may compute different roots depending on whether they used `insert()` or `batch_insert()`, causing consensus-level disagreement about DataLayer state.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

DataLayer store creation is permissionless. Any party can publish a store whose sync data contains duplicate keys. When a subscriber calls `batch_insert()` via the Python binding to sync that data, the corruption occurs automatically. No privileged access, leaked keys, or chain reorg is required. The attacker only needs to be a DataLayer store owner — a role open to anyone.

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the fast path of `batch_insert()` that `insert()` already enforces. Before the loop over `keys_values_hashes`, check each incoming key and hash against `block_status_cache.contains_key()` and `block_status_cache.contains_leaf_hash()`, returning `Err(Error::KeyAlreadyPresent())` or `Err(Error::HashAlreadyPresent())` on collision. Alternatively, track seen keys and hashes within the batch itself to catch intra-batch duplicates.

### Proof of Concept

```rust
// Tree already has 2 leaves (fast path active).
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.insert(KeyId(1), ValueId(1), &hash1, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(2), &hash2, InsertLocation::Auto {}).unwrap();

// batch_insert with a key that already exists — no error returned.
blob.batch_insert(vec![((KeyId(1), ValueId(99)), hash3)]).unwrap();

// blob now contains TWO leaf nodes for KeyId(1).
// cache points only to the new one; old leaf is still traversed by calculate_lazy_hashes.
blob.calculate_lazy_hashes().unwrap();

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
