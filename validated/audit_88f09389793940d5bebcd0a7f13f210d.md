### Title
Precision Loss in `NewDynamicFeeChecker` Causes Systematic Fee Under-Deduction via Integer Div-Then-Mul - (File: ante/evm/fee_checker.go)

### Summary

`NewDynamicFeeChecker` in `ante/evm/fee_checker.go` computes the Cosmos-tx effective fee using an intermediate integer division `feeCap = fee / gas` followed by a multiplication `effectiveFee = effectivePrice * gas`. Because `sdkmath.Int.Quo` truncates toward zero, the round-trip `floor(fee/gas) * gas` is always ≤ `fee`, causing the fee collector to receive systematically less than the user submitted. This is the direct Ethermint analog of the FluxToken `(bpt * veMul / veMax) * veMax` precision-loss pattern.

### Finding Description

Inside `NewDynamicFeeChecker` the following sequence executes on every Cosmos transaction that uses the EIP-1559 dynamic-fee path (both `CheckTx` and `DeliverTx`):

```go
// ante/evm/fee_checker.go  lines 79-99
gas     := feeTx.GetGas()
fee     := feeCoins.AmountOf(denom)                          // total fee submitted by user

feeCap  := fee.Quo(sdkmath.NewIntFromUint64(gas))            // ① integer division – truncates

// feeCap is now floor(fee / gas), losing up to (gas-1) atoms

effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(),
                            feeCap.BigInt(),                  // ② truncated value used here
                            maxPriorityPrice.BigInt()))

effectiveFee := sdk.Coins{{
    Denom:  denom,
    Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)), // ③ multiply back by gas
}}
```

The `effectiveFee` returned here is passed directly to `authante.NewDeductFeeDecorator`, which deducts it from the sender's account.

**Arithmetic identity of the bug:**

```
effectiveFee = floor(fee / gas) * gas
             ≤ fee
```

The discrepancy `fee − effectiveFee` is in the range `[0, gas−1]` atoms and is never deducted. For a transaction with `gas = 10_000_000` the maximum per-tx shortfall is `9_999_999` atoms of the EVM denom.

The pattern is structurally identical to the reported FluxToken bug:
- FluxToken: `(bpt * veMul / veMax) * veMax` — divide by `veMax`, multiply by `veMax`
- Ethermint: `(fee / gas) * gas` — divide by `gas`, multiply by `gas`

Both introduce an unnecessary intermediate integer division that discards the remainder before the inverse multiplication.

### Impact Explanation

`NewDynamicFeeChecker` is wired into the Cosmos ante-handler chain for every non-EVM Cosmos transaction when `DynamicFeeChecker = true`:

```go
// evmd/ante/handler_options.go  line 201
ante.NewDeductFeeDecorator(options.AccountKeeper, options.BankKeeper,
    options.FeegrantKeeper, txFeeChecker)
```

The `effectiveFee` it returns is the amount actually deducted. Because `effectiveFee ≤ fee`, the fee collector module account receives less than the user submitted. The shortfall `fee − effectiveFee` remains in the sender's account, constituting a systematic mis-accounting of fees in the ante handler. This matches the allowed High impact: *"ante handler… bug that permits… valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

The condition is triggered whenever `fee mod gas ≠ 0`, which is the common case for any user who specifies a total fee amount that is not an exact multiple of the gas limit. This is reachable by any unprivileged user submitting a standard Cosmos transaction on a chain with `DynamicFeeChecker` enabled and London hardfork active. No special privileges or unusual conditions are required.

### Recommendation

Eliminate the intermediate division. Compute `feeCap` only for the baseFee comparison, and return the original `fee` (capped to the effective price) rather than reconstructing it via `effectivePrice * gas`:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // keep for baseFee check only

if feeCap.LT(baseFeeInt) {
    return nil, 0, errorsmod.Wrapf(...)
}

effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))

// Use the original fee if it does not exceed effectivePrice * gas,
// otherwise cap to effectivePrice * gas.
maxEffectiveFee := effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))
actualFee := fee
if actualFee.GT(maxEffectiveFee) {
    actualFee = maxEffectiveFee
}
effectiveFee := sdk.Coins{{Denom: denom, Amount: actualFee}}
```

This preserves the full precision of the user-submitted fee and eliminates the truncation loss.

### Proof of Concept

**Concrete numbers:**

```
gas      = 21_000
baseFee  = 1_000_000_000   (1 Gwei)
fee      = 21_000_000_020_999   (user submits this)

// Current code:
feeCap        = 21_000_000_020_999 / 21_000  = 1_000_000_000   (truncated, remainder 20_999 lost)
effectivePrice = min(baseFee + tip, feeCap)  = 1_000_000_000
effectiveFee  = 1_000_000_000 * 21_000       = 21_000_000_000_000

// Shortfall deducted from fee collector:
fee - effectiveFee = 21_000_000_020_999 - 21_000_000_000_000 = 20_999 atoms
```

The 20 999 atoms remain in the sender's account instead of being forwarded to the fee collector. Across high-throughput chains this accumulates continuously across every affected transaction. [1](#0-0) [2](#0-1) [3](#0-2)
