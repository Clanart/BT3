### Title
Missing shard-identity binding check in `set_state_header` allows a malicious peer to substitute a cross-shard state header during state sync — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under the key `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the chunk embedded in the header actually belongs to the requested `shard_id`. A malicious peer can respond to a state-header request for shard Y with a fully valid header for shard X. Every existing check passes, the wrong header is committed to `DBCol::StateHeaders`, and all downstream state-part validation and state-finalization steps then operate against shard X's state root while believing they are reconstructing shard Y's state.

---

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` performs five categories of validation on the peer-supplied `shard_state_header`:

1. **Chunk internal proofs** — `validate_chunk_proofs(&chunk, …)` (line 380)
2. **Merkle inclusion of the chunk in the block** — `verify_path(sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` (lines 394–403)
3. **Merkle inclusion of the previous chunk** (lines 412–436)
4. **Incoming-receipt proof chain** (lines 438–510)
5. **State-root-node validity** (lines 512–523)

None of these checks compares `chunk.shard_id()` against the `shard_id` parameter that was passed in.

The Merkle-inclusion check (step 2) uses `verify_path`, whose implementation is:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
```

`verify_path` is **position-agnostic**: it only checks that `compute_root_from_path(proof, hash_of_chunk) == chunk_headers_root`. It does not verify which shard index (position) the chunk occupies in the tree. A malicious peer that supplies shard X's chunk together with shard X's legitimate Merkle proof will satisfy this check regardless of which `shard_id` the caller requested.

After all checks pass, the header is stored unconditionally:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

The key uses the caller-supplied `shard_id` (shard Y), but the value is shard X's header. Every subsequent step reads this poisoned entry:

- **`set_state_part`** retrieves the header for shard Y, extracts `state_root` from shard X's chunk, and validates incoming state parts against shard X's state root. Legitimate shard-Y parts fail; shard-X parts supplied by the attacker pass.
- **`set_state_finalize`** reads the same poisoned header, calls `apply_chunk` with shard Y's `shard_uid` but shard X's `prev_state_root`, and writes the resulting (wrong) trie changes into shard Y's storage slot.

The missing guard is:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

`ShardChunk::shard_id()` is already available (line 1121 of `core/primitives/src/sharding.rs`), so the fix is a one-line addition.

---

### Impact Explanation

A syncing node that accepts the poisoned header will:

1. Store shard X's state root under shard Y's `StateHeaderKey`.
2. Accept shard X's state parts as valid for shard Y (they verify against shard X's root).
3. Apply shard X's full trie state into shard Y's storage slot during `set_state_finalize`.
4. Emerge from state sync believing shard Y is fully reconstructed, but holding shard X's state.
5. Produce or validate chunks for shard Y using the wrong state root; every such chunk is rejected by honest validators.
6. Become permanently unable to participate in consensus for shard Y without a full re-sync.

The node does not detect the mismatch automatically because the state-part validation and finalization steps all succeed (against the wrong root). The corruption is silent until the node attempts to produce or endorse a chunk.

**Severity: High** — silent, persistent state corruption for the targeted shard; requires a full re-sync to recover.

---

### Likelihood Explanation

State sync is triggered whenever a node joins the network, falls behind by more than one epoch, or is assigned to a new shard after resharding. During sync, the node requests state headers from peers it is connected to. Any single malicious peer in the node's peer set can supply the wrong header. No special privilege (validator, chunk producer, storage operator) is required — an ordinary network peer suffices. The attack is also applicable in a front-running scenario where a peer races the legitimate response.

---

### Recommendation

Add a shard-identity binding check immediately after extracting the chunk from the header in `set_state_header`, before any other validation:

```rust
// In set_state_header, after:
let chunk = shard_state_header.cloned_chunk();

// Add:
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

The same guard should be added in `set_state_finalize` for defence-in-depth, since it also receives a `shard_state_header` and uses the embedded chunk's state root without verifying the shard identity.

---

### Proof of Concept

**Setup**: a 2-shard network (shard 0, shard 1). A syncing node N is performing state sync for shard 1 at `sync_hash = H`. Attacker A is a peer of N and holds a valid `ShardStateSyncResponseHeader` for shard 0 at the same `sync_hash`.

**Steps**:

1. N sends `StateRequestHeader { shard_id: 1, sync_hash: H }` to A.
2. A responds with the shard-0 header (chunk from shard 0, valid Merkle proof for shard 0's position in the block).
3. N calls `set_state_header(shard_id=1, sync_hash=H, header_for_shard_0)`.
4. `validate_chunk_proofs(&chunk_0, …)` — passes; shard 0's internal proofs are valid.
5. `verify_path(chunk_headers_root, proof_for_shard_0, ChunkHashHeight(hash_0, height_0))` — passes; shard 0's chunk IS in the block at position 0.
6. No check that `chunk_0.shard_id() == 1`.
7. `StateHeaderKey(shard_id=1, sync_hash=H)` is written to `DBCol::StateHeaders` with shard 0's header as the value.
8. N requests state parts for shard 1. A supplies shard 0's state parts.
9. `set_state_part(shard_id=1, …)` retrieves the poisoned header, extracts shard 0's `state_root`, validates shard 0's parts against it — all pass.
10. `set_state_finalize(shard_id=1, …)` applies shard 0's trie into shard 1's storage slot.
11. N believes shard 1 is synced. In reality it holds shard 0's state under shard 1's `ShardUId`. Every chunk N produces or validates for shard 1 carries the wrong state root and is rejected by the network.

**Exact corrupted DB entry**: `DBCol::StateHeaders` at key `borsh(StateHeaderKey(ShardId(1), H))` contains `ShardStateSyncResponseHeader` whose embedded chunk has `shard_id = 0`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
