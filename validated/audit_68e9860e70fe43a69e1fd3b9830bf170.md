### Title
Missing `chunk.shard_id() == shard_id` binding in `set_state_header` allows cross-shard state-root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the embedded chunk belongs to `shard_id`. Every validation gate in the function is satisfiable by a crafted header that carries shard 0's chunk (with its valid position-0 Merkle proof) while the caller requests shard 1. The result is that `StateHeaderKey(1, sync_hash)` is committed to `DBCol::StateHeaders` with shard 0's `prev_state_root`, poisoning every downstream consumer (`set_state_part`, `set_state_finalize`).

---

### Finding Description

**Entry point.** A syncing node calls `StateSyncDownloader::ensure_shard_header` for `shard_id=1`. The downloader fetches a `ShardStateSyncResponseHeader` from a peer and forwards it to `ClientActor` via `StateHeaderValidationRequest { shard_id: 1, sync_hash, header }`. [1](#0-0) [2](#0-1) 

`ClientActor` calls `chain.state_sync_adapter.set_state_header(1, sync_hash, header)`. [3](#0-2) 

**The five validation gates and why each passes.**

*Gate 1 — `validate_chunk_proofs`.* Checks the chunk's internal hash consistency, transaction root, and outgoing-receipts root. It never reads `shard_id` and never compares `chunk.shard_id()` against the caller's `shard_id` parameter. [4](#0-3) 

*Gate 2 — `verify_path` for the chunk.* Reconstructs the Merkle root from `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` and the supplied proof. It confirms the chunk is *somewhere* in `sync_prev_block`'s chunk Merkle tree, but it does **not** check which leaf index (i.e., which shard position) the proof encodes. Shard 0's chunk at position 0 with its position-0 proof satisfies this check even when `shard_id=1`. [5](#0-4) 

*Gate 3 — `verify_path` for `prev_chunk`.* Identical structural issue; the prev-chunk proof is also position-agnostic. [6](#0-5) 

*Gate 4 — incoming-receipts proof loop.* The receipt hash is computed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` — using the **caller's** `shard_id=1`, not `chunk.shard_id()`. An attacker supplies the real on-chain receipt proofs for shard 1 (public data). Those proofs verify correctly against the block headers the local node already holds. The terminal height check at 4g compares against `prev_chunk_header.height_included()` (shard 0's prev-chunk height); in a healthy network where all shards include chunks at every height, this equals shard 1's prev-chunk height, so the check passes. [7](#0-6) 

*Gate 5 — `validate_state_root_node`.* Validates the supplied `state_root_node` against `chunk_inner.prev_state_root()` — shard 0's state root. The attacker, as a full node tracking shard 0, can produce a valid node for shard 0's root. [8](#0-7) 

**Storage.** After all gates pass, the header is written under `StateHeaderKey(shard_id=1, sync_hash)` — keyed by the caller's `shard_id`, not by `chunk.shard_id()`. [9](#0-8) 

**Downstream corruption.**

`set_state_part` reads the stored header, extracts `prev_state_root()` from shard 0's chunk, and validates incoming parts against that root. Parts valid for shard 0's trie are accepted and stored under `StatePartKey(sync_hash, 1, idx)`. [10](#0-9) 

`set_state_finalize` retrieves the poisoned header via `get_state_header(1, sync_hash)`, derives `shard_uid` for shard 1, then calls `apply_chunk` with `chunk_header.prev_state_root()` — shard 0's root — under shard 1's `ShardUId`. This installs shard 0's trie data as shard 1's committed state. [11](#0-10) [12](#0-11) 

---

### Impact Explanation

A syncing node that accepts the crafted header will:
1. Store shard 0's `prev_state_root` as shard 1's committed state root in `DBCol::StateHeaders`.
2. Accept and store state parts valid for shard 0 under shard 1's `StatePartKey` entries.
3. In `set_state_finalize`, attempt to apply shard 0's trie as shard 1's state, producing an incorrect `ChunkExtra` for shard 1 and corrupting flat storage for shard 1's `ShardUId`.

The node will diverge from the canonical chain on every subsequent block that touches shard 1, rendering it unable to participate correctly in consensus or serve correct data.

---

### Likelihood Explanation

The attack requires the adversary to be reachable as a state-sync peer (Tier3 network) — no validator or operator privilege is needed. The crafted header is assembled entirely from public on-chain data (shard 0's chunk, its Merkle proof, shard 1's receipt proofs). The precondition that shard 0 and shard 1 share the same `height_included` holds in any healthy epoch where all shards produce chunks at every height, which is the normal operating condition.

---

### Recommendation

Add an explicit shard-identity check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, verify that the Merkle proof position corresponds to the expected shard index by computing `shard_layout.get_shard_index(shard_id)?` and confirming the proof encodes that leaf index, rather than accepting any proof that reconstructs the root. [13](#0-12)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
