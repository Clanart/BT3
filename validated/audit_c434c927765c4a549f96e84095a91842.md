### Title
Dust-Amount Transfer Permanently Locks User Funds Due to Missing Pre-Validation of Normalized Amount — (`near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts and locks user tokens for any amount satisfying `fee < amount`, but `sign_transfer` later rejects the transfer with `InvalidAmountToTransfer` if the decimal-normalized net amount rounds down to zero. Because there is no cancel or refund path for a pending transfer, the user's tokens are permanently frozen in the bridge.

### Finding Description

The NEAR bridge's `init_transfer` function validates only that `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

It then stores the transfer in `pending_transfers` and the user's tokens are locked in the contract (via `ft_transfer_call`).

Later, when a relayer calls `sign_transfer`, the bridge normalizes the net amount to the destination chain's decimal precision using floor division:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` is:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For a NEAR-native token with 24 origin decimals bridged to an EVM chain where it is represented with 18 decimals, `diff_decimals = 6` and the divisor is `10^6 = 1_000_000`. Any net amount (amount minus fee) below `1_000_000` yocto-units normalizes to zero, causing `sign_transfer` to panic unconditionally.

There is no user-callable cancel or refund function. `remove_transfer_message` is only invoked inside `sign_transfer_callback` (when the fee is zero and MPC signing succeeds) and `claim_fee_callback` (after destination-chain finalization). Neither path is reachable when `sign_transfer` itself panics before reaching the MPC call. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Once `init_transfer` succeeds, the user's tokens are held by the bridge contract. If the net amount is below the decimal-normalization threshold, every subsequent `sign_transfer` call panics. The transfer record stays in `pending_transfers` indefinitely and the tokens can never be recovered. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

**Medium-High.** The condition is triggered by any user who sends a net amount smaller than `10^(origin_decimals − dest_decimals)` of the bridged token. For NEAR (24 decimals) → EVM (18 decimals) the threshold is 1 µNEAR (1,000,000 yoctoNEAR ≈ $0.000001). A user sending, for example, 500,000 yoctoNEAR with zero fee satisfies `fee < amount` at `init_transfer` time but produces a normalized amount of 0 at `sign_transfer` time. The condition is reachable by any unprivileged bridge user with no special knowledge.

### Recommendation

Add the normalization check at `init_transfer` time, before tokens are locked, so the transaction reverts cleanly and the user retains their funds:

```rust
// After computing transfer_message, before storing it:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, enforce a protocol-level minimum transfer amount per token that accounts for the decimal gap.

### Proof of Concept

1. Register a NEAR-native token with `origin_decimals = 24` and `decimals = 18` (standard NEAR → EVM setup).
2. User calls `ft_transfer_call` with `amount = 500_000` (yocto-units) and `fee = 0` in the `InitTransferMsg`.
3. `init_transfer` passes: `fee (0) < amount (500_000)` ✓. Transfer stored; tokens locked.
4. Relayer calls `sign_transfer` for the resulting `transfer_id`.
5. Inside `sign_transfer`: `amount_without_fee() = 500_000`; `normalize_amount(500_000, {decimals:18, origin_decimals:24}) = 500_000 / 1_000_000 = 0`.
6. `require!(0 > 0, ...)` panics with `InvalidAmountToTransfer`.
7. No MPC signing occurs; transfer remains in `pending_transfers`; user's 500,000 yocto-units are permanently locked with no recovery path. [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
