### Title
Plain `ft_transfer` Failure in `fin_transfer_send_tokens_callback` Permanently Locks User Funds — (File: near/omni-bridge/src/lib.rs)

---

### Summary

When `fin_transfer` completes for a NEAR-recipient transfer with an empty `msg` field (the plain `ft_transfer` / `mint` code path), if the underlying NEP-141 `ft_transfer` or `mint` cross-contract call fails, the bridge incorrectly treats the failure as a success. The transfer is permanently marked as finalised, the locked-token accounting is not reverted, and the user's funds are irrecoverably locked inside the bridge contract with no recovery path.

---

### Finding Description

`process_fin_transfer_to_near` dispatches token delivery via `send_tokens`, then chains `fin_transfer_send_tokens_callback` as the continuation: [1](#0-0) 

`send_tokens` selects the delivery method based on `msg`:

- `msg.is_empty()` + non-deployed token → `ft_transfer` (plain NEP-141 transfer)
- `msg.is_empty()` + deployed token → `mint` (no `ft_transfer_call`)
- `!msg.is_empty()` → `ft_transfer_call` or `mint` with message [2](#0-1) 

The `is_ft_transfer_call` flag passed to the callback is `!msg.is_empty()`: [3](#0-2) 

Inside `fin_transfer_send_tokens_callback`, the refund/revert decision is delegated to `is_refund_required`: [4](#0-3) 

`is_refund_required` **only inspects the promise result when `is_ft_transfer_call = true`**. When `is_ft_transfer_call = false` it unconditionally returns `false`, regardless of whether the underlying call succeeded or failed: [5](#0-4) 

This means: if `ft_transfer` (or `mint`) fails and `msg` was empty, the callback always takes the **success branch**, emits `FinTransferEvent`, and skips the three critical recovery operations:

1. `revert_lock_actions` — restores the `locked_tokens` counter
2. `remove_fin_transfer` — removes the transfer from `finalised_transfers`, allowing retry
3. `FailedFinTransferEvent` log [6](#0-5) 

Before `send_tokens` is even called, `process_fin_transfer_to_near` has already:
- Inserted the transfer into `finalised_transfers` via `add_fin_transfer` (replay protection consumed)
- Decremented `locked_tokens` via `unlock_tokens_if_needed` [7](#0-6) 

When the callback takes the success path despite a failed `ft_transfer`:
- The nonce in `finalised_transfers` is permanently consumed — the transfer **cannot be retried**
- The `locked_tokens` counter is permanently decremented — bridge accounting is corrupted
- For non-deployed tokens: the tokens remain physically in the bridge contract but are unclaimable
- For deployed tokens: the source-chain tokens were burned/locked but destination tokens were never minted

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

For non-deployed tokens (e.g., USDC, USDT locked in the bridge): the tokens remain in the bridge contract but the accounting says they are unlocked, the nonce is consumed, and there is no admin function that can remove a nonce from `finalised_transfers` without introducing a replay vulnerability. The funds are permanently frozen.

For deployed (bridge-minted) tokens: the user's source-chain assets were burned or locked, the destination `mint` failed silently, and the transfer is finalised — the user suffers a total, unrecoverable loss of the bridged amount.

---

### Likelihood Explanation

**Medium.** The trigger is a failing NEP-141 `ft_transfer` or `mint` call. Realistic causes include:

- A token issuer (e.g., Circle for USDC, Tether for USDT) pausing transfers via governance — a documented, exercised capability
- A bridge-deployed token contract being paused via the bridge's own pause mechanism
- A token contract upgrade that temporarily breaks the `ft_transfer` interface
- Insufficient gas allocated to `FT_TRANSFER_GAS` for a token with non-standard transfer logic

None of these require attacker action; they are normal operational events. Any in-flight `fin_transfer` for a non-deployed token at the moment of a pause will trigger the bug.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, check the promise result **regardless** of `is_ft_transfer_call`. A plain `ft_transfer` can also fail, and the bridge must handle that symmetrically:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    // Check promise result for ALL paths, not only ft_transfer_call
    match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
        Err(_) => true,  // promise failed → always refund
        Ok(value) => {
            if is_ft_transfer_call {
                near_sdk::serde_json::from_slice::<U128>(&value)
                    .map(|a| a.0 == 0)
                    .unwrap_or(false)
            } else {
                false  // ft_transfer succeeded (promise Ok) → no refund needed
            }
        }
    }
}
```

This ensures that a failed `ft_transfer` triggers `revert_lock_actions`, `remove_fin_transfer`, and `FailedFinTransferEvent`, restoring the bridge to a consistent state and allowing the relayer to retry.

---

### Proof of Concept

**Scenario: USDC paused on NEAR mid-flight**

1. Alice initiates `initTransfer` on Ethereum, locking 1000 USDC in the EVM bridge. The EVM bridge emits `InitTransfer`.
2. USDC governance pauses all NEAR-side transfers (a real, exercised capability).
3. A trusted relayer calls `fin_transfer` on the NEAR bridge with Alice's proof and an empty `msg`.
4. `fin_transfer_callback` verifies the proof, calls `process_fin_transfer_to_near`:
   - `add_fin_transfer` marks the nonce as used in `finalised_transfers` [8](#0-7) 
   - `unlock_tokens_if_needed` decrements `locked_tokens` for USDC [9](#0-8) 
5. `send_tokens` dispatches `ft_transfer(alice, 1000, None)` to the USDC contract. The call **fails** because USDC is paused.
6. `fin_transfer_send_tokens_callback` is invoked with `is_ft_transfer_call = false`.
7. `is_refund_required(false)` returns `false` unconditionally. [10](#0-9) 
8. The callback takes the **success branch**: emits `FinTransferEvent`, skips `revert_lock_actions` and `remove_fin_transfer`. [11](#0-10) 
9. Alice never receives her 1000 USDC. The nonce is consumed. The `locked_tokens` counter is wrong. Even after USDC unpauses, the transfer cannot be retried — the nonce is permanently in `finalised_transfers`. Alice's 1000 USDC are irrecoverably locked in the bridge.
