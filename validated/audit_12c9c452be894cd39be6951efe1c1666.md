### Title
Incomplete Self-Destruct Cleanup Leaves Non-EVM-Native (IBC/CosmWasm) Token Balances Orphaned, Enabling Theft via CREATE2 Recreation — (`x/evm/statedb/statedb.go`)

---

### Summary

When a contract self-destructs, `StateDB.Commit()` only burns the EVM-native denom balance of the destroyed account. Non-EVM-native Cosmos bank tokens (IBC tokens, CosmWasm bridge tokens) held by the same address are silently skipped and left as orphaned bank balances. An attacker can then redeploy a contract to the same address via `CREATE2` and access those stranded tokens through bank precompiles or native actions, constituting unauthorized theft of Cosmos bank funds.

---

### Finding Description

In `x/evm/statedb/statedb.go`, the `Commit()` function iterates over dirty state objects and handles self-destructed accounts at lines 800–825:

```go
if obj.selfDestructed {
    cosmosAddr := sdk.AccAddress(obj.Address().Bytes())
    cacheCtx, writeCache := s.origCtx.CacheContext()
    // Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
    // bridge) held by the destroyed address are not drained and may remain as
    // orphaned bank balances.
    if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
        coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
        if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
            return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
        }
    }
    if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
        return errorsmod.Wrap(err, "failed to delete account")
    }
    writeCache()
}
``` [1](#0-0) 

The code explicitly calls `s.keeper.GetBalance(..., s.evmDenom)` — only the single EVM denom is queried and burned. The `DeleteAccount` call removes auth metadata and EVM storage, but **never touches the Cosmos bank module balances for other denominations**. Any IBC vouchers (e.g., `ibc/...` denoms) or other Cosmos-native tokens held by the contract address remain in the bank module after the account is deleted.

The analog to the DYAD report is exact: just as `VaultManagerV2::liquidate` iterated only over non-Kerosene vaults and skipped Kerosene vaults, `Commit()` here iterates only over the EVM denom and skips all other bank denominations.

---

### Impact Explanation

After self-destruct:

1. The EVM account is deleted (`DeleteAccount`).
2. The EVM denom balance is burned.
3. **All other Cosmos bank token balances remain at the now-deleted address.**

An attacker who controls the original contract (or can trigger its self-destruct) can subsequently deploy a new contract to the **same address** via `CREATE2`. The new contract, once deployed, is a live Cosmos account at that address and can invoke bank precompiles or `ExecuteNativeAction`-based transfers to drain the orphaned IBC tokens. This constitutes unauthorized theft of Cosmos bank funds through stateDB native action logic, matching the Critical allowed impact.

Even without CREATE2 recreation, the orphaned balances represent a permanent accounting inconsistency: the bank module holds tokens attributed to a non-existent account, which can never be recovered or burned, effectively destroying those funds.

---

### Likelihood Explanation

The attack path is fully unprivileged:

1. Attacker deploys a contract via `CREATE` (standard, no privilege needed).
2. Users or the attacker itself sends IBC tokens to the contract address (IBC transfers to EVM contract addresses are a supported use case in Cosmos-EVM chains).
3. Attacker calls a `SELFDESTRUCT` opcode in the contract (pre-Cancun) or deploys a contract that self-destructs in its constructor (EIP-6780 path via `SelfDestruct6780` / `newContract == true`).
4. `Commit()` burns only the EVM denom; IBC tokens remain.
5. Attacker redeploys to the same address via `CREATE2` and drains the orphaned tokens.

The `SelfDestruct6780` path in the StateDB confirms both pre- and post-Cancun self-destruct are handled, and the incomplete cleanup applies to both: [2](#0-1) 

The developer comment at line 813 explicitly acknowledges the gap ("may remain as orphaned bank balances"), confirming the code path is reachable and the omission is not guarded elsewhere. [3](#0-2) 

---

### Recommendation

In `Commit()`, after burning the EVM denom balance and before calling `DeleteAccount`, enumerate **all** bank balances held by the self-destructed address and burn or redistribute them:

```go
// Drain all bank balances, not just the EVM denom.
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrapf(err, "failed to burn post-selfdestruct balance for denom %s", coin.Denom)
    }
}
```

This mirrors the fix recommended in the DYAD report: iterate over all asset types (all bank denominations), not just the one the code already handles.

---

### Proof of Concept

1. Deploy contract `C` at address `A` using `CREATE2` with a known salt.
2. Via an IBC transfer, send 1000 `ibc/ATOM` to address `A` (valid Cosmos bank operation).
3. Call `C.selfDestruct()` — triggers `SELFDESTRUCT` opcode.
4. `StateDB.Commit()` burns `C`'s EVM denom balance and calls `DeleteAccount(A)`. The 1000 `ibc/ATOM` remain in the bank module at address `A`.
5. Redeploy a new contract `C'` to address `A` using the same `CREATE2` salt.
6. `C'` calls a bank precompile (or `ExecuteNativeAction`) to transfer the 1000 `ibc/ATOM` to the attacker's address.
7. Attacker receives 1000 `ibc/ATOM` that belonged to the original depositors.
