### Title
Non-EVM-denom bank balances orphaned on self-destruct, recoverable via CREATE2 redeployment — (`x/evm/statedb/statedb.go`)

---

### Summary

`StateDB.Commit()` explicitly burns only the EVM denom on self-destruct. Non-EVM-denom bank balances (IBC tokens, CosmWasm bridge tokens) held at the destroyed address are left as orphaned bank balances. Because `DeleteAccount` removes only the auth account and contract storage — never the bank module balance — those tokens persist at the address and are fully recoverable by redeploying a contract at the same address via CREATE2.

### Finding Description

In `StateDB.Commit()`, the self-destructed branch: [1](#0-0) 

burns only the EVM denom: [2](#0-1) 

The code comment itself documents the gap: *"Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm bridge) held by the destroyed address are not drained and may remain as orphaned bank balances."*

`DeleteAccount` in the keeper confirms this: it clears contract storage and removes the auth account, but explicitly does **not** touch bank balances: [3](#0-2) 

The comment at line 194 reads: *"NOTE: balance should be cleared separately."* The EVM denom is cleared separately; non-EVM denoms are never cleared.

In Cosmos SDK, bank balances are stored independently of auth accounts. After `DeleteAccount`, the IBC token balance at the address persists in the bank module's KV store. When a new contract is deployed at the same address (CREATE2 with identical deployer/salt/initcode), a new auth account is created, and the pre-existing bank balance is immediately accessible to the new contract.

The `Keeper.GetBalance` method reads directly from `bankKeeper.GetBalance`, which is denom-agnostic: [4](#0-3) 

Ethermint supports custom precompiles (bank precompile at `0x64`, IBC relayer precompile, etc.) that allow contracts to call `ExecuteNativeAction` to transfer arbitrary Cosmos bank denoms: [5](#0-4) 

A redeployed contract at the same address can call these precompiles to drain the orphaned IBC tokens.

---

### Impact Explanation

This is a **burn bypass for Cosmos bank funds** through stateDB commit logic. Non-EVM-denom tokens held at a self-destructed contract address are never burned or redistributed. They persist as orphaned bank balances and are recoverable by any party who can redeploy a contract at the same address via CREATE2. If users sent IBC tokens to the contract (e.g., as part of a DeFi protocol), those tokens can be stolen by the original deployer redeploying at the same address and calling a bank/IBC precompile to drain them.

This matches the allowed Critical impact: *"Unauthorized theft, mint, burn bypass, or balance transfer of EVM-denom or Cosmos bank funds through Ethermint transaction execution or stateDB/native action logic."*

---

### Likelihood Explanation

**Moderate.** The preconditions are:
1. IBC (or other non-EVM-denom) tokens must be held at a contract address — realistic on any IBC-enabled Ethermint chain.
2. The contract must be self-destructable — possible for any contract with a `SELFDESTRUCT` opcode path. Under EIP-6780, this is limited to contracts created and destroyed in the same transaction, but pre-EIP-6780 chains are unrestricted.
3. The attacker must control the CREATE2 deployer to redeploy at the same address — achievable by the original contract deployer.
4. A bank or IBC precompile must be registered — common in production Ethermint deployments (Cronos, Evmos, etc.).

The scenario is realistic for a malicious or compromised DeFi protocol that accepts IBC tokens and has a self-destruct path.

---

### Recommendation

In the self-destructed branch of `StateDB.Commit()`, after burning the EVM denom, enumerate **all** bank balances at the destroyed address and burn them before calling `DeleteAccount`. The `BankKeeper` interface already exposes `GetAllBalances` (or equivalent iteration) in standard Cosmos SDK. Alternatively, use `bankKeeper.SpendableCoins` to collect all denoms and burn each one via `SendCoinsFromAccountToModule` + `BurnCoins`, or send them to a community pool / fee collector rather than burning if the chain policy requires it.

func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,

## Example: Cronos bank precompile

The Cronos **bank precompile** exposes an ERC-20-style interface for **native tokens** at a fixed address. Each EVM contract address that calls the precompile is treated as the “token” address: the Cosmos denom is `evm/<caller_hex>`. This section summarizes how it is implemented and how it would be integrated (Cronos currently registers Relayer and ICA in `customContractFns`; the bank precompile is implemented in the same package but can be added the same way).

### Purpose and methods

- **Address**: `0x64` (`common.BytesToAddress([]byte{100})`).
- **Methods** (Solidity ABI): `mint(recipient, amount)`, `burn(recipient, amount)`, `balanceOf(token, account)`, `transfer(sender, recipient, amount)`.
- **Denom**: `EVMDenom(contract.Caller())` = `"evm/" + caller.Hex()` — the caller of the precompile is the “token” (e.g. a deployed contract that mints/burns its own denom).

### ABI and gas table (`init`)

The ABI comes from generated bindings (`bank.BankModuleMetaData.ABI` in `x/cronos/events/bindings/cosmos/precompile/bank`). In `init()` the package parses this ABI and fills a map from 4-byte method ID to gas:

- `mint` / `burn`: 200_000
- `balanceOf`: 10_000
- `transfer`: 150_000

### Contract struct and factory

```go
type BankContract struct {
	bankKeeper  types.BankKeeper
	cdc         codec.Codec
	kvGasConfig storetypes.GasConfig
}

func NewBankContract(bankKeeper types.BankKeeper, cdc codec.Codec, kvGasConfig storetypes.GasConfig) vm.PrecompiledContract {
	return &BankContract{bankKeeper, cdc, kvGasConfig}
}
```

`Address()` returns the fixed address; `RequiredGas(input)` uses `input[:4]` as method ID and returns the mapped gas plus a base cost (`len(input) * kvGasConfig.WriteCostPerByte`).

### Run: ExtStateDB and method dispatch

1. **Cast StateDB** – `stateDB := evm.StateDB.(ExtStateDB)` (from `precompiles/interface.go`).
2. **Dispatch by method** – `method, _ := bankABI.MethodById(contract.Input[:4])`, then `method.Inputs.Unpack(contract.Input[4:])`, and a `switch method.Name`.
3. **Read-only (`balanceOf`)** – Uses `stateDB.Context()` only:  
   `bc.bankKeeper.GetBalance(stateDB.Context(), account, EVMDenom(token))`, then `method.Outputs.Pack(balance)`.
4. **State-changing (`mint`, `burn`, `transfer`)** – Returns an error if `readonly` is true. Then runs all Cosmos logic inside **`stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error { ... })`**:
   - **mint**: `IsSendEnabledCoins` → `MintCoins(module, ...)` → `SendCoinsFromModuleToAccount`.
   - **burn**: `SendCoinsFromAccountToModule` → `BurnCoins(module, ...)`.
   - **transfer**: `IsSendEnabledCoins` → `SendCoins(from, to, coins)`.
   - Uses `EVMDenom(contract.Caller())` as the coin denom and checks `bankKeeper.BlockedAddr` for recipients.
5. **Return** – `method.Outputs.Pack(true)` for mutating methods, or the balance for `balanceOf`.

So: reads go through `Context()`; writes go through `ExecuteNativeAction` so that any failure (or EVM revert) reverts Cosmos state as well.
