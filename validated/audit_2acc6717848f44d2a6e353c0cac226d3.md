### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::initTransfer()` assumes that calling `safeTransferFrom(msg.sender, address(this), amount)` always deposits exactly `amount` tokens into the bridge. For fee-on-transfer ERC20 tokens, the bridge receives `amount - transfer_fee` but emits an `InitTransfer` event crediting the full `amount`. NEAR processes the event at face value and authorizes release of `amount` tokens on the destination chain, creating an unbacked surplus that permanently undercollateralizes the bridge for that token.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles three token cases. For native ERC20 tokens (not bridge tokens, not custom minters), the fallback path is:

```solidity
// OmniBridge.sol lines 406–411
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Immediately after, regardless of which branch was taken, the function emits:

```solidity
// OmniBridge.sol lines 427–436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-supplied, not balance-checked
    fee,
    nativeFee,
    recipient,
    message
);
```

For a fee-on-transfer token (e.g., tokens with a built-in deflationary mechanism), `safeTransferFrom` succeeds but the bridge only receives `amount - token_transfer_fee`. The emitted event still carries the full `amount`. The NEAR bridge's `fin_transfer_callback` reads this event value directly:

```rust
// near/omni-bridge/src/lib.rs lines 722–725
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
```

NEAR then authorizes a `sign_transfer` for the full `amount`, and the destination chain releases `amount` tokens. The bridge's actual reserve for that token is `amount - token_transfer_fee`, so each such transfer drains the bridge by `token_transfer_fee`. Repeated transfers accumulate a deficit until the bridge cannot honor legitimate withdrawals.

---

### Impact Explanation

This is a **balance/accounting corruption that breaks bridge collateralization**. The bridge's on-chain token reserve for any fee-on-transfer ERC20 is systematically less than the sum of all pending/authorized transfers. Later users attempting to finalize transfers will find the bridge insolvent for that token — their funds are permanently locked or unclaimable. This matches the allowed impact: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 address — there is no whitelist check in the EVM contract before the `safeTransferFrom`. Fee-on-transfer tokens are a well-known token class (e.g., tokens with redistribution mechanics, deflationary tokens). A single user bridging such a token is sufficient to trigger the accounting mismatch. The mismatch compounds with every subsequent transfer of the same token.

---

### Recommendation

Use a balance-before / balance-after pattern to measure the actual amount received, and use that measured amount in the event and downstream logic:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    amount = uint128(balanceAfter - balanceBefore); // use actual received amount
}
```

The corrected `amount` must then propagate into both `initTransferExtension(...)` and the `InitTransfer` event so that NEAR authorizes only what the bridge actually holds.

---

### Proof of Concept

1. A fee-on-transfer ERC20 token `FOT` is registered on the bridge (has a NEAR token mapping and decimals entry).
2. Attacker (or any user) calls `initTransfer(FOT, 1000, 0, 0, "near:recipient", "")`.
3. `FOT.safeTransferFrom` succeeds; bridge receives `950` tokens (5% transfer fee).
4. `InitTransfer` event is emitted with `amount = 1000`.
5. NEAR relayer submits proof; `fin_transfer_callback` creates a `TransferMessage` for `1000` tokens.
6. `sign_transfer` is called; MPC signs a payload authorizing release of `1000` tokens on the destination chain.
7. Destination chain releases `1000` tokens to the recipient.
8. Bridge's actual reserve for `FOT` is `950`; it has authorized `1000` — a `50`-token deficit per transfer.
9. After 20 such transfers the bridge holds `0` `FOT` but has authorized `1000` tokens in outstanding transfers, permanently freezing those users' funds. [1](#0-0) [2](#0-1) [3](#0-2)
