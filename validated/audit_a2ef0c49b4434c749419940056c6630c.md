### Title
Missing `chunk.shard_id() == shard_id` Binding in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a caller-supplied `shard_id` but never verifies that the embedded chunk actually belongs to that shard. The `verify_path` call used to prove chunk inclusion is a position-independent Merkle membership proof: it confirms the chunk is *somewhere* in the block's `chunk_headers_root`, not that it occupies the slot for `shard_id`. A malicious snapshot host can craft a mixed header — chunk body from shard Y, receipt proofs from a legitimate shard X response — that passes every validation step and causes `StateHeaderKey(shard_id=X, sync_hash)` to be persisted with shard Y's `prev_state_root`. All subsequent `set_state_part` and `apply_state_part` calls then install shard Y's trie under shard X's `ShardUId`, silently corrupting the victim node's state for shard X.

### Finding Description

**Entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five validation groups:

**Guard 1 — `validate_chunk_proofs`** (lines 380–385): [1](#0-0) 

`validate_chunk_proofs` in `chain/chain/src/validate.rs` checks only internal chunk consistency: header hash, `tx_root`, and `outgoing_receipts_root`. [2](#0-1) 

It contains **no check** that `chunk.shard_id() == shard_id`. A fully valid chunk from shard Y passes this guard.

**Guard 2 — `verify_path` for chunk inclusion** (lines 394–403): [3](#0-2) 

`verify_path` is defined as: [4](#0-3) 

This is a pure Merkle membership proof: `compute_root_from_path(path, hash(item)) == root`. It proves the item is *included somewhere* in the tree. It does **not** verify the leaf's position/index. Compare with `verify_path_with_index` (lines 121–129), which additionally calls `verify_path_matches_index` to enforce position — but that function is not used here. [5](#0-4) 

Because the `chunk_headers_root` is a Merkle root over all shards' chunks, a chunk from shard Y with its own valid Merkle path passes this check even when `shard_id=X` is supplied.

**Guard 3 — `verify_path` for prev_chunk** (lines 416–425): Same position-independent proof, same bypass. [6](#0-5) 

**Guard 4 — Receipt proof validation** (lines 438–510):

The receipt hash at line 488 is computed as: [7](#0-6) 

`ReceiptList(shard_id, receipts)` uses the *parameter* `shard_id` (X), not the chunk's shard_id. If the attacker supplies the `incoming_receipts_proofs` and `root_proofs` from a **legitimate shard X header** (publicly obtainable from any snapshot host), these proofs were built for `to_shard_id=X`, so `ReceiptList(X, receipts)` matches the Merkle leaf and all receipt checks pass. Note also that `to_shard_id` is explicitly ignored in the destructuring at line 478: [8](#0-7) 

The step-4g height check (line 507) requires `shard_X_prev_chunk.height_included() == shard_Y_prev_chunk.height_included()`. In a healthy network where all chunks are included at every block, both values equal `sync_prev_block.height − 1`, so this check passes. [9](#0-8) 

**Guard 5 — `validate_state_root_node`** (lines 513–523): Validates the attacker-supplied `state_root_node` against `chunk_inner.prev_state_root()`. Since both come from shard Y's legitimate header, this passes. [10](#0-9) 

**Storage** (lines 526–529): The header is stored under `StateHeaderKey(shard_id=X, sync_hash)` but contains shard Y's chunk with shard Y's `prev_state_root`. [11](#0-10) 

**Downstream propagation**:

`set_state_part` reads the stored header and uses `chunk.take_header().take_inner().prev_state_root()` — which is now shard Y's state root — to validate incoming state parts: [12](#0-11) 

`set_state_finalize` then applies shard Y's chunk under shard X's `ShardUId` (derived from the `shard_id` parameter, not the chunk's shard_id): [13](#0-12) 

### Impact Explanation

The victim node installs shard Y's trie state under shard X's `ShardUId`. After `set_state_finalize`, the node believes it has shard X's state but actually holds shard Y's. Any chunk production or validation for shard X will use the wrong state root, causing the node to produce invalid chunks that are rejected by the rest of the network. The node is effectively ejected from participation in shard X without any visible error during state sync.

### Likelihood Explanation

The attacker must be a registered snapshot host (or MITM a state sync connection). Becoming a snapshot host requires running a full node and advertising state availability — no validator privileges are needed. Legitimate state sync data for both shards is publicly accessible from any full node. The attack requires both shards to have the same `prev_chunk.height_included()`, which is the common case in a healthy network. The attack is therefore practically reachable by any operator of a full node.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check to `prev_chunk_header.shard_id()`. Additionally, consider replacing the position-independent `verify_path` call with `verify_path_with_index`, supplying the shard's index in the block's chunk list as the expected position, to make the Merkle proof position-binding.

### Proof of Concept

A `cargo test` using `TestEnv` with two shards:

1. Build a chain with two shards (shard 0 and shard 1) and advance to a `sync_hash`.
2. Obtain a legitimate `ShardStateSyncResponseHeader` for shard 1 (chunk, prev_chunk, chunk_proof, prev_chunk_proof, state_root_node).
3. Obtain a legitimate `ShardStateSyncResponseHeader` for shard 0 (incoming_receipts_proofs, root_proofs).
4. Construct a mixed `ShardStateSyncResponseHeaderV2` with shard 1's chunk/prev_chunk/proofs/state_root_node and shard 0's receipt proofs.
5. Call `chain.state_sync_adapter.set_state_header(ShardId::new(0), sync_hash, mixed_header)`.
6. Assert the call returns `Ok(())`.
7. Read back the stored header via `get_state_header(ShardId::new(0), sync_hash)` and assert `header.chunk_prev_state_root() == shard_1_prev_state_root`, confirming shard 1's state root is stored under shard 0's key.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}

/// Verify merkle path for given item and corresponding path.
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))

pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
