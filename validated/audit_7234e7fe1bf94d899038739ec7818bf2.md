### Title
Unauthenticated `eth_resend` Allows Any Caller to Replace a Victim's Pending Transaction via Keyring Re-signing — (`rpc/backend/call_tx.go`)

### Summary
`Resend` matches a pending transaction purely by signing hash and sender address, with no check that the RPC caller owns `args.From`. When the node holds the victim's key (the `eth_sendTransaction` deployment model), any caller who knows the victim's pending transaction parameters can trigger a re-sign and re-broadcast with attacker-chosen gas price/limit.

### Finding Description

`Resend` builds a `matchTx` from the caller-supplied `args`, then iterates the pending pool:

```go
wantSigHash := signer.Hash(matchTx)          // computed from caller-supplied fields
pFrom, err := ethtypes.Sender(signer, pTx)

if pFrom == *args.From && signer.Hash(pTx) == wantSigHash {
    // Match. Re-sign and send the transaction.
    ...
    return b.SendTransaction(args)
}
``` [1](#0-0) 

`signer.Hash` is computed from the transaction fields (nonce, gasPrice, gas, to, value, data, chainID) — **not** from the ECDSA signature. An attacker who observes the victim's pending transaction in the mempool knows all those fields and can construct `args` that produces an identical `wantSigHash`.

`SendTransaction` then checks only whether the key exists in the node keyring:

```go
_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
if err != nil {
    return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; ...")
}
// ...
if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil { ... }
``` [2](#0-1) 

There is no check that the **caller** of `eth_resend` is the owner of `args.From`. The keyring check is a capability check on the node, not an authorization check on the requester.

### Impact Explanation

An attacker who can reach the JSON-RPC endpoint and knows the victim's pending transaction parameters (all observable from the mempool) can:

1. Force the victim's pending transaction to be re-submitted with an arbitrarily high gas price, draining the victim's balance in fees.
2. Alter the gas limit, potentially causing the transaction to fail (if lowered) or succeed with unexpected resource consumption.
3. Repeatedly bump the gas price in a loop, continuously increasing the victim's fee cost.

The attacker **cannot** change `to`, `value`, or `data` because those fields are part of `signer.Hash`; changing them would break the match at line 102. The impact is therefore fee/gas manipulation, not fund redirection.

This fits: *"High. EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits … valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

**Precondition:** The node must hold the victim's key in its keyring — i.e., the victim uses `eth_sendTransaction` rather than `eth_sendRawTransaction`. This is the standard model for custodial or developer nodes. The RPC endpoint need not be internet-facing; any co-tenant or process with network access to the RPC port suffices.

**Attacker knowledge required:** Victim address (public), pending tx nonce/to/value/gasPrice/gas (all visible in the mempool via `eth_getTransactionByHash` or `txpool_content`). No private key material is needed.

The combination is realistic for any Ethermint node that exposes `eth_sendTransaction` without IP-level access control.

### Recommendation

Add a caller-identity check in `Resend` before the pending-pool scan. Since the JSON-RPC layer has no session authentication, the practical mitigations are:

1. **Restrict `eth_sendTransaction` / `eth_resend` to localhost or an authenticated reverse proxy** — the same recommendation geth makes for its personal namespace.
2. **Add an explicit `args.From` allowlist check** against a set of addresses the current RPC session is authorized to act on, if a session/auth layer is introduced.
3. **Document the invariant** that `eth_resend` must only be called by the key owner, and enforce it at the API layer rather than relying on network-level controls alone.

```go
func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}
