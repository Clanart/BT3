### Title
Unconstrained `shard_id` Binding in `set_state_header` Allows Malicious Peer to Substitute a Wrong-Shard State Header — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a caller-supplied `shard_id` and a `ShardStateSyncResponseHeader` from a peer, but never verifies that the chunk embedded in the header actually belongs to `shard_id`. A malicious peer can supply a fully-valid header for shard X while the syncing node believes it is receiving the header for shard Y. The header passes all existing checks and is persisted under the wrong shard's key. Every subsequent state-part validation and application then operates against shard X's state root, silently writing shard X's trie into shard Y's storage.

---

### Finding Description

`set_state_header` performs five validation steps on the received `ShardStateSyncResponseHeader`:

1. `validate_chunk_proofs` — verifies the chunk's internal signature/proofs.
2. `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk_hash, height_included))` — proves the chunk is *somewhere* in the block's chunk-headers Merkle tree.
3. `verify_path` for `prev_chunk_header` — same, for the previous chunk.
4. Receipt-proof chain validation — uses the caller-supplied `shard_id` to compute `ReceiptList(shard_id, receipts)` hashes.
5. `validate_state_root_node` — verifies the `state_root_node` matches `chunk.prev_state_root()`.

**The missing check**: at no point is `chunk.shard_id()` compared to the `shard_id` parameter. The Merkle proof in step 2 proves the chunk is at *some* position in the block, but the position is not constrained to the shard index that corresponds to `shard_id`. A chunk for shard 0 with its valid position-0 Merkle proof will pass the `verify_path` check even when `shard_id = 1`.

After all checks pass, the header is stored under `StateHeaderKey(shard_id, sync_hash)` — the wrong key — and the wrong `state_root` is committed as the authoritative root for `shard_id`. [1](#0-0) 

The receipt-proof loop (step 4) uses the caller-supplied `shard_id` to compute `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))`. When there are no incoming receipts (empty `incoming_receipts_proofs`), the loop body never executes and this check is entirely skipped, making the attack unconditional in that common case. [2](#0-1) 

The header is then stored under the wrong shard's key: [3](#0-2) 

`set_state_part` later retrieves this header by `(shard_id, sync_hash)` and extracts `state_root` from the embedded chunk — which is shard X's root — and validates all incoming parts against it: [4](#0-3) 

`apply_state_part` then writes those trie nodes into shard Y's `ShardUId` column: [5](#0-4) 

---

### Impact Explanation

A syncing node that accepts a header from a malicious peer will:

1. Store shard X's `state_root` as the authoritative root for shard Y.
2. Accept and apply all of shard X's state parts (they validate correctly against shard X's root).
3. Write shard X's entire trie into shard Y's flat storage and trie columns.
4. Call `set_state_finalize` for shard Y, which applies shard X's chunk transactions and receipts on top of shard X's state — producing a completely wrong `ChunkExtra` for shard Y.

The node's shard Y state is permanently corrupted. It will produce invalid chunks for shard Y, fail chunk-state-witness validation, and diverge from the canonical chain. Recovery requires a full re-sync.

---

### Likelihood Explanation

State sync is triggered whenever a node falls behind by more than one epoch. During sync, the node requests headers from its peer set. Any peer in that set — including an unprivileged node that has simply joined the network — can serve a crafted response. No validator key or privileged role is required. The attack is silent: all existing checks pass, no `byzantine_assert!` fires, and no error is returned.

The attack is easiest when the target shard has no incoming cross-shard receipts (the receipt-proof loop is empty), which is common for low-traffic shards.

---

### Recommendation

After extracting the chunk from the header, assert that its `shard_id` matches the requested `shard_id` before any further validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// Bind the header's chunk to the requested shard.
if chunk.shard_id() != shard_id {
    return Err(Error::Other(format!(
        "set_state_header: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, after resolving the shard index from `shard_id`, verify that the Merkle proof position corresponds to that shard index (i.e., the proof path encodes the correct leaf index), so that a chunk for shard X cannot be proven at shard Y's position. [6](#0-5) 

---

### Proof of Concept

Suppose a network has two shards (shard 0 and shard 1). A syncing node S requests the state header for shard 1 from a malicious peer M.

1. M constructs a `ShardStateSyncResponseHeader` containing:
   - `chunk`: the real chunk for shard 0 at the sync block (obtained from the chain).
   - `chunk_proof`: the valid Merkle proof for shard 0's chunk at position 0 in `sync_prev_block.chunk_headers_root`.
   - `state_root_node`: the real `StateRootNode` for shard 0.
   - `incoming_receipts_proofs`: empty (or crafted to match shard 0's receipts).

2. M sends this header to S in response to S's `StateRequestHeader { shard_id: 1, sync_hash }`.

3. S calls `set_state_header(shard_id=1, sync_hash, header_for_shard_0)`:
   - `validate_chunk_proofs` passes (shard 0's chunk is internally valid).
   - `verify_path(chunk_headers_root, proof_for_position_0, ChunkHashHeight(chunk_0_hash, height))` passes (shard 0's chunk IS in the block at position 0).
   - Receipt loop is empty → skipped.
   - `validate_state_root_node(state_root_node, chunk_0.prev_state_root())` passes.
   - Header stored under `StateHeaderKey(shard_id=1, sync_hash)` with shard 0's `state_root`.

4. S requests and validates state parts for shard 1. `set_state_part` retrieves the stored header, extracts shard 0's `state_root`, and validates each part against it. Shard 0's parts pass.

5. `apply_state_part` writes shard 0's trie into shard 1's `ShardUId` storage.

6. `set_state_finalize(shard_id=1, sync_hash)` applies shard 0's chunk on top of shard 0's state, producing a corrupted `ChunkExtra` for shard 1.

Node S now has shard 0's state stored as shard 1's state and will diverge from the canonical chain on the next block it processes for shard 1.

    pub fn set_state_part(
        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,

        let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
        let chunk = shard_state_header.take_chunk();

        let bytes = part.to_bytes();
        store_update.set(DBCol::StateParts, &key, &bytes);

    fn apply_state_part(
        &self,
        shard_id: ShardId,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,
    ) -> Result<(), Error> {
        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);
        let tries = self.get_tries();
        let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;
        let mut store_update = tries.store_update();
        tries.apply_all(&trie_changes, shard_uid, &mut store_update);
        tracing::debug!(target: "chain", %shard_id, values_count = %flat_state_delta.len(), "inserting values to flat storage");
        // TODO: `apply_to_flat_state` inserts values with random writes, which can be time consuming.
        //       Optimize taking into account that flat state values always correspond to a consecutive range of keys.
        flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
        self.precompile_contracts(epoch_id, contract_codes)?;
        store_update.commit();
