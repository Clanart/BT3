### Title
Missing Declared-Class Validation in `execute_replace_class` Allows `contract_state_changes` / `contract_class_changes` Desync Leading to Permanent Fund Freeze - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS updates `contract_state_changes` with an arbitrary caller-supplied class hash without verifying that the hash has a corresponding entry in `contract_class_changes`. This creates a desync between the two state-tracking dictionaries — the exact structural analog of the MultiSig `replaceOwner` off-by-one bug, where one data structure is updated while the other is not. Any contract can call `replace_class` with an undeclared class hash, permanently corrupting its own state entry. Any future block containing a call to that contract becomes unprovable, permanently freezing any funds held by the contract.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), the OS writes the caller-supplied `class_hash` directly into `contract_state_changes` without checking whether that hash has a corresponding entry in `contract_class_changes`: [1](#0-0) 

The developer-inserted TODO comment at line 898 explicitly acknowledges the missing check:

```cairo
let class_hash = request.class_hash;
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

The identical gap exists in the deprecated syscall path for Cairo 0 contracts: [2](#0-1) 

The two dictionaries that must remain in sync are:

| Dictionary | Tracks |
|---|---|
| `contract_state_changes` | Per-contract state: `class_hash`, `storage_ptr`, `nonce` |
| `contract_class_changes` | Declared class hashes → compiled class hashes |

The invariant is: every `class_hash` stored in any `StateEntry` inside `contract_state_changes` **must** have a corresponding entry in `contract_class_changes`. `execute_replace_class` breaks this invariant.

When `execute_entry_point` subsequently processes a call to the affected contract, it reads the class hash from `contract_state_changes` and performs a mandatory lookup in `contract_class_changes`: [3](#0-2) 

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // ← undeclared hash
)
// compiled_class_hash == 0 (default dict value)

let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // ← find_element(key=0) → FAILS
);
```

`find_element` is a non-optional search; if no compiled class with hash 0 exists (which is always the case — compiled class hashes are Poseidon hashes of class content), the Cairo VM traps and the block proof cannot be generated.

---

### Impact Explanation

**Attack sequence:**

1. **Block N**: Contract A calls `replace_class(undeclared_hash)`. The OS accepts it (no constraint). Block N is proven and committed on-chain. Contract A's `StateEntry.class_hash` is now `undeclared_hash`.

2. **Block N+1 onward**: Any transaction calling Contract A causes `execute_entry_point` to look up `undeclared_hash` in `contract_class_changes`, receive 0, and then `find_element(key=0)` fails. Block N+1 cannot be proven.

3. The sequencer must permanently exclude all calls to Contract A from every future block. Any funds held by Contract A are **permanently frozen** — no transfer, withdrawal, or upgrade is possible.

**Impact classification**: **Critical — Permanent freezing of funds.**

If Contract A is a widely-used token contract, bridge, or vault, the impact extends to a **High — Network not being able to confirm new transactions** for any block that must interact with it.

---

### Likelihood Explanation

The attacker-controlled entry path is direct and requires no privileged role:

- Any deployed contract can issue the `replace_class` syscall with an arbitrary felt value as the new class hash.
- The OS imposes no constraint on the value — the TODO comment confirms the check is intentionally absent.
- The attack completes in a single transaction (the `replace_class` call itself); the contract need not be called again in the same block for the state to be committed.
- The presence of the TODO comment (dated 2026) indicates the gap is known and unfixed in the current production code.

Whether the blockifier independently enforces this check is outside the OS scope, but the TODO's existence implies the OS is intended to be the enforcement layer and currently is not.

---

### Recommendation

In `execute_replace_class` (both `syscall_impls.cairo` and `deprecated_execute_syscalls.cairo`), add a constraint that verifies the new class hash has a non-zero entry in `contract_class_changes` before writing to `contract_state_changes`:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
assert_not_zero(compiled_class_hash);
```

This enforces the invariant that every class hash stored in `contract_state_changes` is always resolvable via `contract_class_changes`, eliminating the desync.

---

### Proof of Concept

1. Deploy Contract A (valid class hash `H_valid`, declared in `contract_class_changes`).
2. Contract A executes `replace_class(H_fake)` where `H_fake` is any felt not present in `contract_class_changes`.
3. `execute_replace_class` writes `StateEntry(class_hash=H_fake, ...)` into `contract_state_changes` for Contract A's address. No constraint fires. Block is proven and committed.
4. In Block N+1, any party submits a call to Contract A.
5. `execute_entry_point` reads `H_fake` from `contract_state_changes`, calls `dict_read{contract_class_changes}(key=H_fake)` → returns 0.
6. `find_element(..., key=0)` fails — no compiled class with hash 0 exists.
7. Block N+1 proof generation fails. The sequencer must permanently exclude Contract A from all future blocks.
8. All funds in Contract A are permanently frozen.
