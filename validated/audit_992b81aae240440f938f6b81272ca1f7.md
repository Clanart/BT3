### Title
Gateway Admits Transactions Using Stale Previous-Block L2 Gas Price Instead of Next-Block Price, Causing Execution-Time Rejection — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `validate_resource_bounds` and `run_validate_entry_point` both build their fee-bounds check against the **previous block's** L2 gas price. The actual execution block uses the **next block's** L2 gas price, computed by the EIP-1559 fee market. A transaction whose `max_price_per_unit` falls between `P_prev × threshold%` and `P_next` passes every gateway check but is rejected by the blockifier at sequencing time. The discrepancy is explicitly acknowledged by a TODO comment in the source.

---

### Finding Description

**Step 1 — Gateway admission check uses stale price.**

`validate_resource_bounds` reads the previous block's L2 gas price and applies a configurable percentage threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is `P_prev × min_gas_price_percentage / 100`. With the default `min_gas_price_percentage = 100`, the gate passes any transaction where `max_price_per_unit ≥ P_prev`. [2](#0-1) 

**Step 2 — The blockifier validation inside the gateway also uses the stale price.**

`run_validate_entry_point` increments only the block number; the gas prices in `block_info` remain those of the previous block:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
// ...
blockifier_validator.validate(account_tx)
``` [3](#0-2) 

Inside `validate`, `perform_pre_validation_stage` calls `check_fee_bounds`, which compares `resource_bounds.max_price_per_unit` against `block_info.gas_prices` — still `P_prev`. [4](#0-3) 

**Step 3 — Actual execution uses the next block's L2 gas price.**

The consensus orchestrator computes `next_l2_gas_price` via the EIP-1559 formula before building each block proposal. The maximum per-block increase is bounded by `gas_price_max_change_denominator` but can be up to ~12.5% above `P_prev` when the block is full:

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice { ... }
``` [5](#0-4) 

This `next_l2_gas_price` is stored in the block header and is what the batcher's block context uses at execution time. [6](#0-5) 

**Step 4 — The gap.**

| Check | Gas price used | Admission result |
|---|---|---|
| `validate_resource_bounds` (gateway) | `P_prev × threshold%` | passes if `max_price ≥ P_prev × threshold%` |
| `check_fee_bounds` (blockifier, gateway) | `P_prev` | passes if `max_price ≥ P_prev` |
| `check_fee_bounds` (blockifier, execution) | `P_next` | **fails** if `max_price < P_next` |

A transaction with `max_price_per_unit ∈ [P_prev, P_next)` clears both gateway checks but is rejected by `check_fee_bounds` at sequencing time with `MaxGasPriceTooLow`. [7](#0-6) 

---

### Impact Explanation

The gateway's admission invariant — that admitted transactions will not fail the blockifier's pre-validation fee check — is broken. Any transaction whose `max_price_per_unit` is between `P_prev` and `P_next` is accepted by the gateway, enters the mempool, is forwarded to the batcher, and is then silently dropped at execution time. The user receives no fee-related rejection at submission time; the transaction simply never appears in a block. This matches **High: Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**.

---

### Likelihood Explanation

The EIP-1559 price increase fires every block in which `gas_used > gas_target`. On a busy network this is the common case. With a full block the price rises by `price / gas_price_max_change_denominator` per block. Any user who sets `max_price_per_unit` exactly at `P_prev` (the value the gateway accepts) will have their transaction rejected at execution whenever the next block's price is higher — which is the normal operating condition under load.

---

### Recommendation

Replace the `previous_block_l2_gas_price` read in `validate_resource_bounds` with the pre-computed `next_l2_gas_price` that is already stored in the block header (field `BlockHeaderWithoutHash::next_l2_gas_price`). The same fix must be applied to the `block_info` constructed in `run_validate_entry_point`: the gas prices must reflect `P_next`, not `P_prev`. The TODO comment at line 202 already identifies this exact correction. [8](#0-7) 

---

### Proof of Concept

1. Block N finalises with `l2_gas_price = 1_000_000_000 fri` and `gas_used > gas_target` (full block).
2. The EIP-1559 formula sets `next_l2_gas_price = 1_003_003_003 fri` (~0.3% increase, one step of `MIN_GAS_PRICE_INCREASE_DENOMINATOR = 333`).
3. User calls `starknet_estimateFee` against the pending block; the pending block still advertises `l2_gas_price = 1_000_000_000 fri`.
4. User submits an `INVOKE v3` transaction with `l2_gas.max_price_per_unit = 1_000_000_000`.
5. Gateway `validate_resource_bounds`: threshold = `1_000_000_000 × 100% = 1_000_000_000` → **passes**.
6. Gateway `run_validate_entry_point` → blockifier `check_fee_bounds` with `P_prev = 1_000_000_000` → **passes**.
7. Transaction enters mempool and is forwarded to the batcher.
8. Batcher builds block N+1 with `l2_gas_price = 1_003_003_003`.
9. Blockifier `perform_pre_validation_stage` → `check_fee_bounds`: `max_price_per_unit (1_000_000_000) < actual_gas_price (1_003_003_003)` → **`MaxGasPriceTooLow` error** → transaction rejected.
10. The transaction never appears in any block; the user observes silent failure. [9](#0-8) [7](#0-6)

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants =
        orchestrator_versioned_constants::VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
