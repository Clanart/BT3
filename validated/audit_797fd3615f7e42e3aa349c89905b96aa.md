### Title
Unauthenticated Class Declaration via BOOTSTRAP Sender Address Bypass - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary

The `execute_declare_transaction` function in the StarkNet OS contains a special-case branch that, when a declare transaction's `sender_address` equals the felt literal `'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and all resource bounds are zero, **completely skips signature validation, nonce enforcement, and fee charging**. All four conditions are fully attacker-controllable transaction fields. This is a direct analog to the external report's state-transition bypass: just as `leave()` + `migrateToVault()` bypassed the lock by resetting state through a special migration path, here a specially-crafted declare transaction bypasses the authentication path through a special bootstrap branch.

### Finding Description

In `execute_declare_transaction`, lines 764–776:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
```

When this branch is taken, the function returns immediately after writing to `contract_class_changes`. The following steps that occur for every normal declare transaction are **entirely skipped**:

1. `check_and_increment_nonce` — no nonce check or increment for the 'BOOTSTRAP' address
2. `non_reverting_select_execute_entry_point_func` (the `__validate_declare__` entry point) — no signature verification
3. `charge_fee` — no fee deducted

All four triggering conditions are fields the transaction sender controls directly:
- `sender_address` is a felt in the transaction; the felt encoding of `'BOOTSTRAP'` is `0x424f4f545354524150`
- `nonce` is a transaction field
- `version` is a transaction field
- Resource bounds (all set to `max_amount = 0`) make `compute_max_possible_fee` return 0

No deployed contract at address `'BOOTSTRAP'` is required; the early-return path never reads the contract state for that address.

### Impact Explanation

**Direct loss of funds (Critical).**

The `dict_update` call enforces `prev_value=0`, meaning a class hash can only be declared once. An attacker who observes a Sierra class published off-chain (e.g., in a public repository) can compute its class hash `H` (the OS verifies this via `finalize_class_hash`) and submit a BOOTSTRAP declare transaction pairing `H` with an **attacker-chosen `compiled_class_hash`** pointing to malicious CASM — before the legitimate developer's declare transaction is processed.

Consequences:
- The legitimate developer's declare transaction will fail (`prev_value=0` is violated).
- Any contract subsequently deployed with class hash `H` will execute the malicious CASM.
- Funds held by or interacting with those contracts can be stolen.

Additionally, because the nonce of `'BOOTSTRAP'` is never incremented, the attacker can repeat this for arbitrarily many class hashes in the same block (each with a different `class_hash_ptr`), enabling a systematic front-running campaign against all pending class declarations.

### Likelihood Explanation

The four triggering conditions are all standard transaction fields with no cryptographic barrier. The attacker does not need:
- A deployed account at `'BOOTSTRAP'`
- Any private key
- Any funds

The only prerequisite is that the sequencer includes the transaction. Because the OS itself treats the transaction as valid (the BOOTSTRAP branch is the OS's own acceptance criterion), a sequencer that mirrors OS validation logic will accept it. The attack is realistic whenever a new Sierra class is published publicly before its on-chain declaration.

### Recommendation

Remove the BOOTSTRAP special-case branch entirely, or gate it behind a sequencer-level allowlist enforced outside the OS (e.g., a dedicated genesis transaction type with its own hash prefix and a hardcoded class whitelist). The OS should never skip `__validate_declare__` based solely on attacker-supplied transaction fields.

### Proof of Concept

Attack sequence:

1. Developer publishes Sierra source for a new ERC-20 token; its class hash is `H`.
2. Attacker computes `H` from the Sierra components (same computation as `finalize_class_hash`).
3. Attacker crafts a declare transaction:
   - `sender_address = 0x424f4f545354524150` (`'BOOTSTRAP'` as felt)
   - `nonce = 0`
   - `version = 3`
   - `compiled_class_hash = MALICIOUS_CASM_HASH`
   - All `ResourceBounds.max_amount = 0`
4. Attacker submits to sequencer; OS executes the BOOTSTRAP branch, writes `H → MALICIOUS_CASM_HASH` to `contract_class_changes`, returns without calling `__validate_declare__` or `charge_fee`.
5. Developer's declare transaction arrives; `dict_update` with `prev_value=0` panics because `H` is already declared.
6. Any user who deploys a contract with class hash `H` runs the malicious CASM; attacker drains their funds.
