Audit Report

## Title
Permissionless `refundETH()` allows any caller to steal excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` is an unrestricted `external` function that unconditionally transfers the router's entire native ETH balance to `msg.sender`. When a user sends `msg.value > amountIn` for a WETH-input swap, `pay()` deposits only the exact swap amount as WETH and leaves the remainder on the router. Any subsequent caller — including an MEV bot in the same block — can call `refundETH()` and receive that excess ETH, causing a direct loss of user principal.

## Finding Description

`refundETH()` has no access control: [1](#0-0) 

`pay()`, when `token == WETH` and `nativeBalance >= value`, deposits exactly `value` ETH as WETH and leaves `nativeBalance - value` on the router untouched: [2](#0-1) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable swap entry points: [3](#0-2) 

`multicall()` records no transient state about the original caller, so there is no mechanism inside `refundETH()` to restrict it to the depositing address: [4](#0-3) 

The intended safe pattern — composing `refundETH` as the last call in the same `multicall` — is demonstrated in tests but is not enforced at the contract level: [5](#0-4) 

Any user who calls `exactInputSingle` (or any other payable swap entry point) directly with `msg.value > amountIn`, or who omits `refundETH` from their multicall, leaves ETH stranded on the router that is immediately claimable by any EOA or contract.

## Impact Explanation

Direct, unconditional loss of user principal (native ETH). No oracle manipulation, privileged role, or pool state dependency is required. The attacker receives the full excess ETH amount. This is a High-severity direct loss of user funds meeting Sherlock thresholds.

## Likelihood Explanation

Users sending native ETH for WETH swaps must manually compose a multicall that includes `refundETH` as the final step. Omitting it — or calling `exactInputSingle` directly with excess `msg.value` — leaves ETH on the router. MEV bots routinely monitor for unprotected ETH balances on router contracts and can backrun the victim's transaction in the same block. The attack is permissionless, repeatable, and requires no special setup.

## Recommendation

Remove the cross-transaction theft vector by one of:
1. Recording `msg.sender` in transient storage at `multicall` entry and checking it inside `refundETH`, so only the initiating caller can claim the refund.
2. Accepting an explicit `recipient` parameter in `refundETH` that the caller must supply (matching Uniswap v4's approach), combined with auto-refunding excess ETH at the end of each swap entry point.
3. Auto-refunding `address(this).balance - msg.value` (i.e., any pre-existing balance) at the end of each payable swap function, eliminating cross-transaction residue entirely.

## Proof of Concept

```
1. User calls:
   router.exactInputSingle{value: 1 ether}(
       ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ETH, ...
       })
   );
   // pay() deposits exactly 0.5 ETH as WETH; 0.5 ETH remains on router.

2. Attacker (next tx, same block via backrun):
   router.refundETH();
   // refundETH sends address(this).balance (0.5 ETH) to attacker.

3. Assert: attacker.balance increased by 0.5 ETH; user lost 0.5 ETH.
```
