### Title
Missing Upper-Bound Staleness Check on Proof Facts `base_block_number` Enables Stale-State L2→L1 Message Injection - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_proof_facts` enforces only a **lower-bound** on `base_block_number` (proof must not be too recent), but imposes **no upper-bound** (proof can be arbitrarily old). An unprivileged invoke-transaction sender can anchor proof facts to any historical block, replaying a virtual OS execution that was valid at that stale state, and inject L2→L1 messages derived from balances that no longer exist into the current block's output.

---

### Finding Description

In `check_proof_facts` the sole block-number constraint is:

```cairo
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
``` [1](#0-0) 

This asserts `base_block_number ≤ current_block_number − 10`, preventing a proof that is too recent. There is no symmetric assertion of the form `base_block_number ≥ current_block_number − MAX_AGE`.

The block hash contract accumulates hashes for **every** processed block: `write_block_number_to_block_hash_mapping` writes the hash of `current_block_number − STORED_BLOCK_HASH_BUFFER` on every block, so the mapping grows unboundedly. [2](#0-1) 

Consequently, `read_block_hash_from_storage` will successfully verify the block hash for any block ever processed, no matter how old. [3](#0-2) 

The `VirtualOsOutputHeader` embedded in the proof facts carries `n_l2_to_l1_messages` — the count of L2→L1 message hashes produced by the virtual OS run at `base_block_number`. [4](#0-3) 

These messages are accepted and included in the current block's output without any check that the state they were derived from is still consistent with the current chain state.

The call site in `execute_invoke_function_transaction` passes attacker-supplied `proof_facts` directly into `check_proof_facts`: [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The virtual OS run executes a transaction against the state at `base_block_number` and produces L2→L1 messages (e.g., bridge withdrawal messages) reflecting balances at that historical state. Because there is no upper-bound staleness check, an attacker who:

1. Held funds at block N,
2. Spent those funds on L2 between block N and the current block M,

can still generate a valid ZK proof anchored at block N showing those funds were sent to L1. The OS accepts the proof (all existing checks pass), and the resulting L2→L1 withdrawal message is committed into block M's output. The attacker then claims the funds on L1 — funds they no longer hold on L2. This is a classic double-spend via stale-state replay.

---

### Likelihood Explanation

The attacker is an ordinary invoke-transaction sender — no privileged role required. The `proof_facts` field is attacker-controlled calldata hashed into the transaction hash.
