### Title
`MerkleBlob::batch_insert` Bypasses Per-Item Duplicate Key/Hash Validation, Corrupting DataLayer Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert()` uses a two-phase strategy: a slow path that calls the fully-validated `insert()` for the first two items (when the tree has ≤1 existing leaf), and a fast bulk path that writes all remaining leaf nodes directly to the blob via `insert_entry_to_blob()` without checking for duplicate keys or duplicate hashes. An unprivileged caller can supply a batch containing a duplicate `KeyId` or `Hash`, silently corrupting the Merkle tree structure and its root hash, which invalidates all subsequent DataLayer proofs of inclusion.

### Finding Description

`MerkleBlob::insert()` enforces two invariants before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` calls `insert()` (with full validation) only for the last two items popped from the input vector when the tree currently has ≤1 leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items (the entire batch when the tree already has ≥2 leaves, or all but the last two otherwise) are written directly to the blob with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

Because `insert_entry_to_blob` updates `block_status_cache` via `add_leaf`, which uses `HashMap::insert` (silent overwrite), a duplicate key in the batch causes:

1. Two physical leaf nodes with the same `KeyId` written to the blob.
2. The cache's `key_to_index` map silently overwritten to point only to the last duplicate.
3. The cache's `leaf_hash_to_index` map similarly corrupted if the hash is also duplicated.

The `BlockStatusCache.add_leaf` method that performs the silent overwrite:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
``` [4](#0-3) 

The Python binding `py_batch_insert` is the direct unprivileged entry point:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

The `ProofOfInclusion` validity check (`proof.valid()`) only verifies internal hash consistency of the proof object itself — it does not detect that the underlying tree was built from a corrupted blob:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(...);
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [6](#0-5) 

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A corrupted tree root is committed on-chain. The "orphaned" duplicate leaf (the one the cache no longer tracks) is invisible to `get_proof_of_inclusion` and `get_keys_values`, yet it physically participates in the internal-node hash computation. This means:

- The committed root hash does not correspond to any valid, consistent key-value mapping.
- Proofs of inclusion generated from the corrupted tree will pass `proof.valid()` but prove membership in a state that was never legitimately committed.
- Proofs of exclusion for the orphaned key will incorrectly claim the key is absent even though it is present in the blob.

### Likelihood Explanation

The entry point is the public Python API `MerkleBlob.batch_insert(keys_values, hashes)`, callable by any DataLayer client. No privilege is required. The attacker only needs to include a repeated `KeyId` anywhere in the batch when the tree already has ≥2 leaves (the common production case). The `check_integrity()` function would catch this in test mode (since it is enabled via `check_integrity_on_drop: cfg!(test)`), but it is not called automatically in production after `batch_insert`. [7](#0-6) 

### Recommendation

Add duplicate-key and duplicate-hash checks at the start of the bulk path in `batch_insert`, mirroring the guards in `insert()`. Before writing any leaf in the fast bulk loop, verify:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Additionally, track keys and hashes seen within the current batch itself (using a local `HashSet`) to catch intra-batch duplicates before any blob writes occur.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so the fast path is skipped entirely
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(200), ValueId(200), h(200))

# Now batch_insert with a duplicate KeyId(100) — no error is raised
dup_key = KeyId(100)
blob.batch_insert(
    [(dup_key, ValueId(999)), (KeyId(300), ValueId(300))],
    [h(999), h(300)],
)

blob.calculate_lazy_hashes()

# The cache silently overwrites: KeyId(100) now maps to the new leaf
# but the old leaf node is still physically in the blob, corrupting the root
root = blob.get_root_hash()

# get_proof_of_inclusion returns a "valid" proof for KeyId(100) → ValueId(999)
# but the root hash is computed over a tree that also contains the orphaned
# KeyId(100) → ValueId(100) leaf, making the committed root invalid
proof = blob.get_proof_of_inclusion(dup_key)
assert proof.valid()  # passes, but root is corrupted
```
