### Title
`initTransfer1155` Accepts No-Code Token Address, Enabling Fraudulent Cross-Chain Transfer Events — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155` calls `IERC1155(tokenAddress).safeTransferFrom(...)` as a raw interface call — not through OpenZeppelin's `SafeERC20` wrapper — without verifying that `tokenAddress` has deployed code. In Solidity, a direct interface call to an address with no code succeeds silently (the EVM CALL opcode returns success with empty return data). This allows an attacker to emit a legitimate-looking `InitTransfer` event for a not-yet-deployed ERC1155 token, which can later be submitted as proof to the NEAR bridge to claim tokens without ever depositing any ERC1155 assets.

---

### Finding Description

`OmniBridge.sol` uses `using SafeERC20 for IERC20` for all ERC-20 transfers, which delegates to OpenZeppelin's `SafeERC20.safeTransferFrom`. That wrapper internally calls `Address.functionCall`, which checks `isContract` (code size > 0) before proceeding — so the ERC-20 path is protected.

However, `initTransfer1155` uses a **direct interface cast** for the ERC-1155 transfer:

```solidity
IERC1155(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    tokenId,
    amount,
    ""
);
```

This is not routed through `SafeERC20` and carries no code-existence check. [1](#0-0) 

If `tokenAddress` has no deployed code, the EVM CALL opcode returns `(success=true, returndata="")`. Because `IERC1155.safeTransferFrom` has no return value, the Solidity compiler emits no return-data check, and execution continues normally. The function then derives a `deterministicToken` address and emits a fully valid `BridgeTypes.InitTransfer` event: [2](#0-1) 

The ERC-20 path in `initTransfer` is protected because it uses `SafeERC20`: [3](#0-2) 

The ERC-1155 path has no equivalent protection.

---

### Impact Explanation

The NEAR bridge's `fin_transfer` → `fin_transfer_callback` processes proofs of `InitTransfer` events emitted by the registered EVM factory. It verifies the emitter address against `self.factories`, looks up `token_decimals` for the token, and then mints or releases tokens to the recipient: [4](#0-3) 

If the `deterministicToken` derived from the no-code address is registered on NEAR (i.e., `token_decimals` contains an entry for it), the NEAR bridge will accept the fraudulent proof and mint or release tokens to the attacker's chosen recipient — without any ERC-1155 tokens ever having been deposited on EVM.

**Impact class**: Critical — unauthorized mint of bridged assets on NEAR.

---

### Likelihood Explanation

The attack requires the `deterministicToken` (derived from the attacker-supplied `tokenAddress` and `tokenId`) to be registered on NEAR. This is achievable via the following front-running scenario:

1. An ERC-1155 token is about to be deployed at a predictable address (e.g., via CREATE2 or a known deployer nonce).
2. The attacker calls `initTransfer1155` with the not-yet-deployed address before the token is live. The no-code `safeTransferFrom` succeeds silently; `InitTransfer` is emitted on-chain.
3. The ERC-1155 token is deployed and subsequently registered on NEAR via the normal `deploy_token` flow.
4. The attacker submits the proof of the earlier fraudulent `InitTransfer` event to the NEAR bridge.
5. NEAR verifies the proof (the event is genuine), finds the token registered, and mints tokens to the attacker's recipient.

CREATE2-based deployments make the target address fully predictable, making step 1 straightforward. The window between event emission and token registration can be arbitrarily long — the attacker can wait indefinitely.

---

### Recommendation

Add an explicit code-existence check for `tokenAddress` before the ERC-1155 transfer in `initTransfer1155`, mirroring the protection that `SafeERC20` provides for ERC-20 tokens:

```solidity
require(tokenAddress.code.length > 0, "ERR_TOKEN_NO_CODE");
```

Or use OpenZeppelin's `Address.isContract(tokenAddress)` check. This should be placed before the `IERC1155(tokenAddress).safeTransferFrom(...)` call. [5](#0-4)
