### Title
Cairo 0 Class Cache Bypass: Reverted V0 Class Served Without Declaration Check on Cache Hit — (`crates/blockifier/src/state/state_reader_and_contract_manager.rs`)

### Summary

`get_compiled_from_class_manager` explicitly guards Cairo 1 cache hits with an `is_declared` check to handle reverted blocks, but the V0 branch is an intentional no-op. Because `SyncStateReader::is_declared` is defined to return `false` for Cairo 0 classes (by contract), no equivalent guard exists for V0. The cache-miss path for Cairo 0 correctly calls `is_class_declared_at` (which checks deprecated class storage), while the cache-hit path performs zero declaration checks. After a block revert, a stale V0 entry in the `GlobalContractCache` is served unconditionally.

### Finding Description

**Cache-hit path (incorrect):**

In `get_compiled_from_class_manager`, when the cache returns a `RunnableCompiledClass::V0`:

```rust
match &runnable_class {
    RunnableCompiledClass::V0(_) => {}   // ← no check, falls through
    _ => {
        if !self.state_reader.is_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
}
return Ok(runnable_class);   // ← stale V0 returned
``` [1](#0-0) 

The comment at lines 76–78 explicitly acknowledges the revert scenario for Cairo 1 but provides no equivalent protection for Cairo 0.

**Cache-miss path (correct):**

On a cache miss, `SyncStateReader::get_compiled_classes` is called, which routes through `get_contract_class_from_client`:

```rust
let is_class_declared = self.runtime
    .block_on(self.state_sync_client.is_class_declared_at(self.block_number, class_hash))
    ...;
if !is_class_declared {
    return Err(StateError::UndeclaredClassHash(class_hash));
}
``` [2](#0-1) 

`is_class_declared_at` in `StateSync` checks **both** Cairo 1 (`get_class_definition_block_number`) and Cairo 0 (`get_deprecated_class_definition_block_number`): [3](#0-2) 

After a revert, `get_deprecated_class_definition_block_number` returns `None`, so `is_class_declared = false` → `UndeclaredClassHash` is correctly returned.

**Why `is_declared` cannot fix this:**

`SyncStateReader::is_declared` calls only `is_cairo_1_class_declared_at`, and the `FetchCompiledClasses` trait docstring explicitly states "Cairo 0 classes always return `false`": [4](#0-3) [5](#0-4) 

So even if the V0 branch called `is_declared`, it would always return `false` and always reject — which is why the branch is empty. The design gap is that there is no Cairo-0-aware declaration check callable from the cache-hit path.

**The RPC execution path has the same acknowledged gap:**

```rust

    Ok(RunnableCompiledClass::V0(deprecated_contract_class.try_into()?))
}
``` [6](#0-5) 

This confirms the issue is known and unresolved across both the gateway and RPC execution paths.

### Impact Explanation

The concrete corrupted value is `RunnableCompiledClass::V0` from a reverted block, served to execution or simulation when the class is no longer declared. This produces:

- **Gateway/RPC simulation path:** A transaction that calls `deploy_syscall` with a reverted Cairo 0 class hash H will succeed in simulation (cache hit → V0 returned, deploy proceeds), causing the gateway to admit a transaction that should be rejected with `UndeclaredClassHash`. This matches the allowed High impact: "Mempool/gateway/RPC admission accepts invalid transactions."
- **Block execution path:** If the class remains in cache during block execution, the deploy succeeds with the reverted class code, producing wrong state, wrong receipts, and wrong events. This matches the Critical impact: "Wrong compiled class selected for execution."
- **Inconsistency between nodes:** A node with a warm cache executes differently from a node with a cold cache for the same block, breaking determinism.

### Likelihood Explanation

The precondition is a block revert (reorg). Reverts are not directly triggered by an unprivileged user, but they are normal operational events in any sequencer. The `GlobalContractCache` is a process-wide LRU cache with no revert-aware invalidation for V0 entries. Once a Cairo 0 class is cached, it persists until evicted by LRU pressure. An attacker who has previously declared a Cairo 0 class (a permissionless operation) and whose block was reverted can reliably exploit this on the next request for that class hash, as long as the entry has not been evicted.

### Recommendation

Introduce a Cairo-0-aware declaration check callable from the cache-hit path. The `FetchCompiledClasses` trait should expose a method like `is_v0_declared(class_hash) -> StateResult<bool>` backed by `is_class_declared_at` (which already checks deprecated class storage). The V0 branch in `get_compiled_from_class_manager` should call this method and return `UndeclaredClassHash` on failure, mirroring the existing Cairo 1 guard.

Alternatively, the `GlobalContractCache` should be invalidated for all V0 entries on block revert, though this is architecturally harder given the cache is shared across block boundaries by design.

### Proof of Concept

**Scenario (gateway path):**

1. User submits a `declare` transaction for Cairo 0 class H at block N. The gateway processes it; `get_compiled_classes(H)` is called (cache miss), `is_class_declared_at` returns `true`, V0 is fetched and stored in `GlobalContractCache`.
2. Block N is reverted. `get_deprecated_class_definition_block_number(H)` now returns `None` in storage.
3. User submits a transaction at block N+1 that calls `deploy_syscall(H, ...)`. The gateway simulates it via `StateReaderAndContractManager::get_compiled_class(H)`.
4. Cache hit → `RunnableCompiledClass::V0(_)` branch → no check → `Ok(V0)` returned. The deploy succeeds in simulation.
5. Gateway admits the transaction.
6. During block execution: if H is still in cache → deploy succeeds with reverted class (wrong state committed); if H was evicted → `UndeclaredClassHash` → transaction reverts (inconsistent outcome vs. simulation).

**Divergence proof (unit test sketch):**

```rust

```

The existing test suite already covers the Cairo 1 revert case (`cached_but_verification_failed_after_reorg_scenario`) but has no equivalent test for Cairo 0 after revert, confirming the gap is untested. [7](#0-6)
