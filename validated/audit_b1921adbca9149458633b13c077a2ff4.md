### Title
Unchecked `burn_tokens_if_needed` Promise Result Allows Unbacked Cross-Chain Transfer Finalization - (File: near/omni-bridge/src/lib.rs)

### Summary
In `init_transfer_internal`, the bridge calls `burn_tokens_if_needed` using `.detach()`, which fires the burn promise and immediately discards the result. If the burn fails silently, the transfer message is still recorded and `InitTransferEvent` is still emitted, causing relayers to finalize the cross-chain transfer and release original assets on the destination chain — while the bridge tokens that should have been destroyed remain unburned in the bridge contract.

### Finding Description

`init_transfer_internal` is the core function that records a NEAR-side outbound transfer. For deployed (bridge) tokens, it must burn the bridge tokens to maintain collateralization parity with the origin chain. The burn is dispatched as a detached promise:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result never checked
    }
}
``` [1](#0-0) 

This is called from `init_transfer_internal` before the `InitTransferEvent` is emitted:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(...);
// ...
env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
U128(0)
``` [2](#0-1) 

In NEAR, `.detach()` schedules the cross-contract call but does **not** attach a callback and does **not** propagate failure back to the caller. The current function's state changes — `add_transfer_message`, `try_update_storage_balance`, `lock_tokens_if_needed`, and the `InitTransferEvent` log — are all committed regardless of whether the burn succeeds or fails.

If the burn promise fails (e.g., `BURN_TOKEN_GAS` is insufficient for the token contract's actual burn logic, or the token contract panics), the bridge:
1. Records the transfer as initiated.
2. Emits `InitTransferEvent` that relayers observe.
3. Relayers finalize the transfer on the destination chain, releasing original assets.
4. The bridge tokens that should have been destroyed remain in the bridge contract.

There is no callback, no revert path, and no retry mechanism.

### Impact Explanation

This breaks bridge collateralization. Bridge tokens (representing wrapped assets) are supposed to be burned 1:1 when original assets are released on the destination chain. If the burn silently fails:
- Bridge tokens remain in the bridge contract (unburned).
- Original assets are released on the destination chain.
- The total supply of bridge tokens exceeds the locked original assets, creating unbacked supply.

This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation

**Low.** The burn failure requires the `BURN_TOKEN_GAS` constant to be insufficient for the token contract's `burn` execution, or the token contract to panic for an unexpected reason. Under normal operating conditions the burn succeeds. However, the absence of any result check means any failure mode — including future token contract upgrades that increase gas consumption — silently corrupts the accounting with no on-chain indication.

### Recommendation

Replace the detached call with a chained callback that checks the burn result and reverts the transfer state if the burn failed:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) -> Option<Promise> {
    if self.is_deployed_token(&token) {
        Some(
            ext_token::ext(token)
                .with_static_gas(BURN_TOKEN_GAS)
                .burn(amount)
                .then(
                    Self::ext(env::current_account_id())
                        .with_static_gas(BURN_CALLBACK_GAS)
                        .on_burn_callback(transfer_id),
                )
        )
    } else {
        None
    }
}
```

The callback `on_burn_callback` should check `env::promise_result(0)` and, if the burn failed, call `remove_transfer_message_without_refund` and revert the lock action so the transfer is not processed by relayers.

### Proof of Concept

1. User holds bridge tokens for a deployed token (e.g., `wETH.bridge.near`).
2. User calls `ft_transfer_call` on the bridge token contract, transferring tokens to the bridge with a transfer message targeting an EVM chain.
3. The bridge's `ft_on_transfer` → `init_transfer_internal` is invoked.
4. `burn_tokens_if_needed` fires the burn with `.detach()`.
5. The burn promise fails (e.g., gas exhaustion due to `BURN_TOKEN_GAS` being too low for the token's burn logic).
6. `init_transfer_internal` returns `U128(0)` (success), emitting `InitTransferEvent`.
7. A relayer observes the event and calls `fin_transfer` on the EVM `OmniBridge`, which calls `safeTransfer` to release the original ERC-20 tokens to the recipient.
8. The bridge tokens are never burned; they sit in the bridge contract. The original ERC-20 tokens have been released. Bridge collateralization is broken. [1](#0-0) [3](#0-2) [4](#0-3)
