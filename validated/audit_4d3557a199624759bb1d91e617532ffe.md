### Title
Wrong Epoch's Shard Layout Used to Index `prev_block.chunks()` in `get_state_response_part`, Producing Wrong State Root for Child Shards at Resharding Boundary — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` resolves the shard index using the **sync block's (new epoch's) shard layout** and then uses that index to read `prev_state_root()` from `prev_block.chunks()`, which belongs to the **previous epoch** and is indexed by the **old shard layout**. At a resharding boundary the two layouts differ, so the wrong chunk — and thus the wrong `state_root` — is selected. This is the direct nearcore analog of the external report's "market-level checkpoint used instead of position-level checkpoint" bug.

---

### Finding Description

`compute_state_response_header` (the header path) correctly distinguishes the two layouts:

```rust
// compute_state_response_header — CORRECT
let shard_layout     = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
let prev_epoch_id    = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index  = prev_shard_layout.get_shard_index(shard_id)?;   // ← prev-epoch layout
let chunk_header = sync_prev_block.chunks().get(prev_shard_index)…;
```

`get_state_response_part` (the part-serving path) does **not**:

```rust
// get_state_response_part — BUG
let epoch_id    = block.header().epoch_id();                             // sync block = new epoch
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;      // new-epoch layout
…
let prev_block  = self.chain_store.get_block(header.prev_hash())?;       // old-epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;              // ← new-epoch layout ← BUG
let state_root  = prev_block.chunks()
    .get(shard_index)                                                    // indexes old block with new index
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();                                                  // ← wrong state root
``` [1](#0-0) [2](#0-1) 

At a resharding boundary the sync block is the first block of the new epoch; `prev_block` is the last block of the old epoch. The old epoch has N shards; the new epoch has N+1 shards (one parent split into two children). The child shard IDs (max+1, max+2) exist only in the new layout. `shard_layout.get_shard_index(child_shard_id)` returns the child's ordinal in the **new** layout (e.g., index 3 or 4 for a 5-shard layout). `prev_block.chunks()` has only N entries (old layout). Two outcomes:

| Child shard index in new layout | `prev_block.chunks().get(index)` | Result |
|---|---|---|
| `< N` (within old bounds) | Returns the **parent** shard's chunk | Wrong `state_root` silently used |
| `>= N` (out of bounds) | Returns `None` | `Error::InvalidShardId` |

The silent case is the exact analog: the part is generated against the parent shard's pre-resharding state root, not the child shard's post-resharding state root committed in the header.

The header path (`compute_state_response_header`) commits the correct state root via `prev_shard_layout`. The part path uses a different, wrong root. The receiving node validates each part against the header's state root:

```rust
// chain/client/src/sync/state/downloader.rs
if matches!(
    runtime_adapter.validate_state_part(shard_id, &state_root, …, &part),
    StatePartValidationResult::Valid
) { … } else {
    return Err(near_chain::Error::Other("Part data failed validation".to_owned()));
}
``` [3](#0-2) 

Validation fails because the part was built against the wrong root.

---

### Impact Explanation

Any node that needs to sync a child shard after resharding via P2P state sync will receive parts that fail validation. The node cannot reconstruct the child shard's state. Validators that must produce chunks for a newly assigned child shard cannot do so without a valid state, stalling chunk production for that shard. In deployments without external state sync (S3/GCS), there is no fallback, making this a **High** liveness impact.

---

### Likelihood Explanation

The bug is triggered at every resharding epoch boundary when any node requests state parts for a child shard via P2P. With dynamic resharding enabled, resharding boundaries occur automatically based on trie memory thresholds. The code path is unconditionally reachable by any unprivileged node sending a `StateRequestPart` message.

---

### Recommendation

Mirror the layout-selection logic from `compute_state_response_header` into `get_state_response_part`:

```rust
// Replace:
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
…
let shard_index = shard_layout.get_shard_index(shard_id)?;

// With:
let prev_epoch_id    = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
// For child shards at a resharding boundary, resolve to the parent shard ID first.
let shard_index = prev_shard_layout.get_shard_index(shard_id)
    .or_else(|_| {
        let parent_id = shard_layout.get_parent_shard_id(shard_id)?;
        prev_shard_layout.get_shard_index(parent_id)
    })?;
``` [4](#0-3) 

---

### Proof of Concept

1. Network undergoes resharding: old layout has 4 shards `[0,1,2,3]`; new layout has 5 shards `[0,1,2,4,5]` (shard 3 split into children 4 and 5).
2. `sync_hash` = first block of new epoch (epoch boundary block).
3. A syncing node sends `StateRequestPart { shard_id: 4, sync_hash, part_id: 0 }`.
4. Serving node enters `get_state_response_part`:
   - `epoch_id` = new epoch; `shard_layout` = new layout (5 shards).
   - `shard_ids` check passes (shard 4 is in new layout).
   - `prev_block` = last block of old epoch (4 chunks, indexed 0–3).
   - `shard_index = new_layout.get_shard_index(4)` = **3** (child 4 is at index 3 in new layout).
   - `prev_block.chunks().get(3)` = chunk for old shard 3 (the **parent**).
   - `state_root` = parent shard's pre-resharding state root. ← **wrong**
5. `obtain_state_part` generates a part against the parent's state root.
6. Receiving node fetches the header via `get_state_response_header`, which correctly uses `prev_shard_layout` and commits the child's post-resharding state root.
7. `validate_state_part(shard_id=4, &child_state_root, …, &part)` → **Invalid** (part was built against parent root).
8. State sync for child shard 4 fails; the node cannot produce chunks for shard 4. [5](#0-4) [6](#0-5) [7](#0-6)

        // 1. Everything prefixed with `sync_` indicates new epoch, for which we are syncing.
        // 1a. `sync_prev` means the last of the prev epoch.
        // 2. Empty prefix means the height where chunk was applied last time in the prev epoch.

        // 2a. `prev_` means we're working with height before current.
        // 3. In inner loops we use all prefixes with no relation to the context described above.

            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let sync_block_header = sync_block.header();
        let sync_block_epoch_id = sync_block_header.epoch_id();
        let shard_ids = self.epoch_manager.shard_ids(sync_block_epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }

            return Ok(state_part);
