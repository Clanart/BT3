### Title
`starknet_version` Dropped from Pending Data Causes Wrong `VersionedConstants` in Pending-Block Fee Estimation and Simulation - (File: `crates/apollo_rpc_execution/src/lib.rs`)

### Summary

The `starknet_version` field present in the pending block's wire data is silently dropped during conversion to `ExecutionPendingData`. When `create_block_context` later tries to resolve the version for a pending block, it queries storage for a block number that does not yet exist, receives `None`, and silently falls back to `StarknetVersion::LATEST`. Every `estimate_fee`, `simulate_transactions`, and `call` request against `block_id = "pending"` therefore executes under the wrong `VersionedConstants` whenever the pending block's declared version differs from the hardcoded `LATEST` — most critically during protocol upgrades.

### Finding Description

**Step 1 — `starknet_version` is stripped during conversion.**

`client_pending_data_to_execution_pending_data` in `crates/apollo_rpc/src/pending.rs` converts the feeder-gateway `ClientPendingData` (which carries `block.starknet_version()`) into `ExecutionPendingData`. The `ExecutionPendingData` struct defined in `crates/apollo_rpc_execution/src/objects.rs` has no `starknet_version` field: [1](#0-0) [2](#0-1) 

The conversion copies `timestamp`, `l1_gas_price`, `l1_data_gas_price`, `l2_gas_price`, `l1_da_mode`, and `sequencer` — but never `starknet_version`.

**Step 2 — `create_block_context` queries storage with the pending block number.**

Inside `create_block_context` in `crates/apollo_rpc_execution/src/lib.rs`, when `maybe_pending_data` is `Some`, `block_number` is set to `block_context_number.unchecked_next()` — the pending block number that has not yet been committed to storage. The function then calls:

```rust
let starknet_version = storage_reader
    .begin_ro_txn()?
    .get_starknet_version(block_number)?   // block_number = pending, not in storage
    .unwrap_or(StarknetVersion::LATEST);   // always falls back here
``` [3](#0-2) 

Because the pending block number is never in storage, `get_starknet_version` returns `None` and the fallback is unconditionally `StarknetVersion::LATEST`.

**Step 3 — Wrong `VersionedConstants` are selected.**

The resolved (always-`LATEST`) version is immediately used to select the execution constants:

```rust
let versioned_constants = VersionedConstants::get(&starknet_version)?;
``` [4](#0-3) 

`VersionedConstants` controls step gas costs, builtin gas costs, `enable_casm_hash_migration`, `block_casm_hash_v1_declares`, `enable_reverts`, `invoke_tx_max_n_steps`, `validate_max_n_steps`, event limits, and more. [5](#0-4) 

**Step 4 — The pending block's actual version is available but ignored.**

`PendingBlockOrDeprecated::starknet_version()` is a public method on the client-side type and is used correctly when building the `PendingBlockHeader` for `get_block`: [6](#0-5) [7](#0-6) 

It is only the execution path (`estimate_fee`, `simulate_transactions`, `call`) that loses the version.

### Impact Explanation

Any call to `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_call` with `block_id = "pending"` uses `VersionedConstants::LATEST` instead of the constants matching the pending block's declared `starknet_version`. During a protocol upgrade window — when the feeder gateway serves a pending block whose `starknet_version` string corresponds to the previous protocol version while the node binary already embeds a newer `LATEST` — the returned fee estimates, gas vectors, revert decisions, and execution traces are computed under the wrong rule set. A transaction that would succeed under the old constants may be reported as reverted (or vice versa), and fee estimates may be systematically wrong. This is an authoritative-looking wrong value returned by the RPC layer.

**Matching impact:** *High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

### Likelihood Explanation

The divergence occurs at every protocol upgrade. The pending block's `starknet_version` string is set by the sequencer/feeder gateway and lags the node binary's `LATEST` constant during the upgrade window. No special attacker capability is required — any user calling `estimate_fee` or `simulate_transactions` against `"pending"` during an upgrade window is affected. The bug is structural and always present; it is only dormant when the pending block's version happens to equal `LATEST`.

### Recommendation

Add `starknet_version: StarknetVersion` to `ExecutionPendingData` in `crates/apollo_rpc_execution/src/objects.rs`. Populate it in `client_pending_data_to_execution_pending_data` by parsing `client_pending_data.block.starknet_version()`. In `create_block_context`, when `maybe_pending_data` is `Some`, read `starknet_version` directly from the pending data instead of querying storage:

```rust
Some(pending_data) => {
    // ...existing fields...
    starknet_version = pending_data.starknet_version;  // use pending block's version
}
None => {
    // ...existing storage lookup...
}
```

This mirrors how all other pending-block fields (gas prices, timestamp, sequencer) are already handled — taken directly from the pending data rather than from storage.

### Proof of Concept

1. Start a node synced to block N whose `starknet_version` is `V0_14_3`.
2. The feeder gateway returns a pending block with `"starknet_version": "0.14.3"` (i.e., `StarknetVersion::V0_14_3`).
3. The node binary has `StarknetVersion::LATEST = V0_14_4` (a newer version with different `VersionedConstants`, e.g., different `execute_max_sierra_gas` or `blake` builtin cost).
4. Call `starknet_estimateFee` with `block_id = "pending"`.
5. Observe: `create_block_context` calls `get_starknet_version(block_N+1)` → `None` → falls back to `V0_14_4`.
6. `VersionedConstants::get(V0_14_4)` is used instead of `VersionedConstants::get(V0_14_3)`.
7. The returned fee estimate uses `V0_14_4` gas costs and feature flags, not `V0_14_3`.
8. A transaction sized to fit within `V0_14_3`'s `execute_max_sierra_gas` limit may be reported as out-of-gas under `V0_14_4`'s different limit, or vice versa — producing an authoritative-looking wrong result. [8](#0-7) [2](#0-1)
