### Title
Fire-and-Forget `send_tokens` in Fast-Transfer Relayer Reimbursement Causes Irrecoverable Fund Lock — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain`, when a fast-transfer relayer is owed reimbursement, the bridge calls `send_tokens(...).detach()` — a fire-and-forget promise with no callback. State mutations (transfer finalized, locked-token accounting updated, fast-transfer marked finalized) are committed **before** the token transfer, and any failure of the token transfer is silently discarded. The relayer permanently loses their fronted funds with no recovery path.

---

### Finding Description

The call path is:

`fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_other_chain`

Inside `process_fin_transfer_to_other_chain`, when a fast-transfer relayer exists for the incoming proof:

1. **`add_fin_transfer`** inserts the transfer ID into `finalised_transfers` — this is a one-way door; re-submission is permanently blocked.
2. **`unlock_tokens_if_needed`** decrements the locked-token accounting.
3. **`send_tokens(...).detach()`** fires the token transfer to the relayer with no callback.
4. **`mark_fast_transfer_as_finalised`** marks the fast transfer as done. [1](#0-0) [2](#0-1) [3](#0-2) 

If `send_tokens` fails (gas exhaustion, token contract panic, storage not registered), all four state mutations remain committed, but the relayer receives nothing. Because the transfer ID is already in `finalised_transfers`, `add_fin_transfer` will panic with `BridgeError::TransferAlreadyFinalised` on any retry, making the loss permanent. [4](#0-3) 

The same pattern exists in `utxo_fin_transfer_fast`, where `send_tokens(...).detach()` is used after `mark_fast_transfer_as_finalised`: [5](#0-4) 

Contrast this with the correct pattern used in `process_fin_transfer_to_near`, where `send_tokens` is chained with a proper `fin_transfer_send_tokens_callback` that reverts state on failure: [6](#0-5) [7](#0-6) 

The gas calculation inside `send_tokens` compounds the risk: it always subtracts `SEND_TOKENS_CALLBACK_GAS` from the available gas budget even when called with `.detach()` (no callback), leaving less gas for the actual token transfer than intended. The code even has a `TODO` acknowledging this: [8](#0-7) 

---

### Impact Explanation

A relayer who performed a fast transfer (fronting their own tokens to the user on the destination chain) calls `fin_transfer` with the origin-chain proof to reclaim their funds. If `send_tokens` fails silently:

- For **native tokens**: the tokens remain locked in the bridge contract forever (transfer is finalized, no re-entry possible).
- For **deployed/bridged tokens**: the `mint` call never executes; the relayer's fronted capital is unrecoverable.

This is an **irrecoverable lock / unclaimable settlement of relayer funds** in the bridge flow.

---

### Likelihood Explanation

`send_tokens` can fail due to:

1. **Gas exhaustion**: The gas budget is reduced by `SEND_TOKENS_CALLBACK_GAS` even when no callback exists, leaving the token transfer under-gassed in a gas-tight execution context.
2. **Token contract panic**: If the token contract is paused, upgraded, or encounters an internal error at the moment of the call.
3. **Storage not registered**: If the relayer account does not have storage registered on the token contract (possible for newly onboarded relayers or after token migration).

Any of these conditions — which can occur without attacker involvement — triggers the permanent fund loss.

---

### Recommendation

Replace the `.detach()` pattern with a proper callback that reverts state on failure, mirroring the existing `fin_transfer_send_tokens_callback` pattern used for NEAR-recipient transfers. Specifically:

- In `process_fin_transfer_to_other_chain`: chain `send_tokens` with a callback that calls `remove_fin_transfer`, reverts `lock_actions`, and un-marks the fast transfer if the token send fails.
- In `utxo_fin_transfer_fast`: apply the same callback pattern.
- Remove the unconditional subtraction of `SEND_TOKENS_CALLBACK_GAS` from the gas budget when `send_tokens` is used without a callback.

---

### Proof of Concept

1. Relayer A performs a fast transfer: fronts 1000 USDC to user on Solana.
2. Origin-chain proof arrives; trusted relayer calls `fin_transfer` with the proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_other_chain` executes:
   - `add_fin_transfer` marks transfer as finalized.
   - `unlock_tokens_if_needed` decrements locked balance.
   - `send_tokens(token, relayer_A, 1000, "").detach()` fires.
   - `mark_fast_transfer_as_finalised` marks fast transfer done.
4. The token contract's `ft_transfer` runs out of gas (or panics). The promise fails silently — `.detach()` means no callback observes the failure.
5. Relayer A's 1000 USDC is permanently locked in the bridge. Any retry of `fin_transfer` panics with `ERR_TRANSFER_ALREADY_FINALISED`. [9](#0-8)

    fn process_fin_transfer_to_other_chain(
        &mut self,
        predecessor_account_id: AccountId,
        transfer_message: TransferMessage,
    ) {
        let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
        let token = self.get_token_id(&transfer_message.token);

        if transfer_message.recipient.is_utxo_chain() {
            let btc_account_id =
                self.get_utxo_chain_token(transfer_message.get_destination_chain());
            require!(
                token == btc_account_id,
                BridgeError::NativeTokenRequiredForChain.as_ref()
            );
        }

        self.unlock_tokens_if_needed(
            transfer_message.get_origin_chain(),
            &token,
            transfer_message.amount.0,
        );
        self.lock_tokens_if_needed(
            transfer_message.get_destination_chain(),
            &token,
            transfer_message.fee.fee.into(),
        );

        let fast_transfer = FastTransfer::from_transfer(transfer_message.clone(), token.clone());
        let recipient = if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
            require!(
                !status.finalised,
                BridgeError::FastTransferAlreadyFinalised.as_ref()
            );
            Some(status.relayer)
        } else {
            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token,
                transfer_message
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            );

            None
        };

        // If fast transfer happened, send tokens to the relayer that executed fast transfer
        if let Some(relayer) = recipient {
            self.send_tokens(
                token,
                relayer,
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
                "",
            )
            .detach();
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
        } else {
            required_balance = self
                .add_transfer_message(transfer_message.clone(), predecessor_account_id.clone())
                .saturating_add(required_balance);
        }

        self.update_storage_balance(
            predecessor_account_id,
            required_balance,
            env::attached_deposit(),
        );

        env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
    }
