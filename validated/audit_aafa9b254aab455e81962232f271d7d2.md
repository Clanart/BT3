### Title
Attacker-Controlled `sender_address` Bypasses `__validate__` Signature Check via `skip_stateful_validations`, Admitting Invalid Transactions to Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function uses the transaction's `sender_address` field — which is fully attacker-controlled — to look up whether a deploy-account transaction exists in the mempool for that address, and if so, skips the `__validate__` entry-point execution entirely. No check is made that the submitter is actually authorized to act on behalf of that `sender_address`. An attacker who observes any victim's `deploy_account` transaction in the mempool can craft an invoke transaction with `sender_address = victim`, `nonce = 1`, and an invalid/attacker-owned signature, have it admitted to the mempool without signature verification, and use fee escalation to displace the victim's legitimate nonce-1 invoke.

---

### Finding Description

The `skip_stateful_validations` function in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements a UX feature: when a user sends `deploy_account` + `invoke(nonce=1)` together, the gateway skips `__validate__` for the invoke because the account does not yet exist on-chain. [1](#0-0) 

The skip condition is:

1. Transaction is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in committed state).
4. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [2](#0-1) 

All four inputs are derived from the transaction itself or from the mempool state keyed by the transaction's own `sender_address`. There is no check that the submitter owns the private key for `sender_address`. When `skip_validate = true`, `run_validate_entry_point` is called with `ExecutionFlags { validate: false }`, so `__validate__` is never executed at the gateway. [3](#0-2) 

The `validate_by_mempool` call that precedes this check only validates nonce range and duplicate tx-hash; it does not verify the signature. [4](#0-3) 

The mempool's `add_tx` path then applies fee-escalation logic: if the attacker's transaction carries a higher tip than the victim's existing nonce-1 invoke, the victim's transaction is evicted and replaced. [5](#0-4) 

This is the direct sequencer analog of H-21: in H-21 the `from` parameter is accepted without `msg.sender == from`, allowing anyone to act on behalf of any approved address. Here, `sender_address` is accepted without a signature check, allowing anyone to act on behalf of any address that has a pending deploy-account.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions.**

An attacker can:

1. Admit a cryptographically invalid invoke transaction (wrong or attacker-owned signature) into the mempool under any victim's `sender_address`.
2. Use fee escalation to evict the victim's legitimate nonce-1 invoke, permanently dropping it from the mempool.
3. When the batcher executes the block, the attacker's transaction fails `__validate__` and is rejected/reverted, but the victim's legitimate transaction is already gone.

The result is targeted transaction censorship: the victim's `deploy_account` executes successfully, but their paired invoke is silently dropped, requiring the victim to resubmit.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker only needs to monitor the public mempool for `deploy_account` transactions (trivially observable via the gateway or P2P gossip).
- No privileged access, special role, or prior on-chain state is required.
- The attacker must submit before the victim's `deploy_account` is committed to a block (a window of one block time, typically seconds to minutes).
- The attack is cheap: one crafted invoke transaction with a higher tip than the victim's.

---

### Recommendation

Bind the skip-validate shortcut to the transaction's cryptographic identity rather than to the caller-supplied `sender_address` alone. Concretely, before granting the skip, verify that the transaction's signature is structurally valid for the `sender_address` using the class hash of the pending deploy-account (which is known from the mempool entry). Alternatively, restrict the skip to cases where the gateway can confirm the deploy-account and the invoke share the same submitter session (e.g., submitted in the same batch or signed with the same key), rather than relying solely on the address lookup.

At minimum, add a check that no existing transaction with the same `(sender_address, nonce)` is already in the mempool before granting the skip, to prevent fee-escalation displacement of legitimate transactions.

---

### Proof of Concept

**Setup:**
- Victim generates a new keypair, computes `victim_address`.
- Victim submits `deploy_account(victim_address)` and `invoke(sender=victim_address, nonce=1, calldata=X, tip=100)` to the gateway.
- Both are admitted; the mempool now contains both transactions.

**Attack:**
1. Attacker observes `victim_address` in the mempool via `account_tx_in_pool_or_recent_block`.
2. Attacker submits `invoke(sender=victim_address, nonce=1, calldata=Y, tip=200, signature=attacker_sig)`.
3. Gateway evaluates `skip_stateful_validations`:
   - `nonce == 1` ✓
   - `account_nonce == 0` (victim not yet deployed) ✓
   - `account_tx_in_pool_or_recent_block(victim_address) == true` ✓
   - Returns `skip_validate = true`.
4. `run_validate_entry_point` is called with `validate = false`; `__validate__` is never run.
5. Transaction passes all gateway checks and is forwarded to `mempool.add_tx`.
6. Mempool fee-escalation: attacker's tip (200) > victim's tip (100) → victim's nonce-1 invoke is evicted.
7. Batcher builds block: `deploy_account` executes (victim's account is deployed), then attacker's invoke executes → `__validate__` fails (wrong signature) → transaction reverted/rejected.
8. Victim's legitimate nonce-1 invoke is permanently lost from the mempool. [6](#0-5) [7](#0-6)

        block_info.block_number = block_info.block_number.unchecked_next();

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn add_tx(&mut self, args: AddTransactionArgs) -> MempoolResult<()> {
        // First remove old transactions from the pool.
        let mut account_nonce_updates = self.remove_expired_txs();
        if !self.is_fifo() {
            self.add_ready_declares();
        }

        let tx_reference = TransactionReference::new(&args.tx);
        self.add_tx_validations(tx_reference, &args.tx, args.account_state.nonce)
            .inspect_err(|err| self.log_add_tx_error(err, &args))?;

        MEMPOOL_TRANSACTIONS_RECEIVED.increment(
            1,
            &[(LABEL_NAME_TX_TYPE, InternalRpcTransactionLabelValue::from(&args.tx.tx).into())],
        );

        // May override a removed queued nonce with the received account nonce or the account's
        // state nonce.
        account_nonce_updates.insert(
            args.account_state.address,
            self.state.resolve_nonce(args.account_state.address, args.account_state.nonce),
        );

        let should_delay_declare =
            matches!(&args.tx.tx, InternalRpcTransactionWithoutTxHash::Declare(_))
                && !self.is_fifo();
        if should_delay_declare {
            self.delayed_declares.push_back(self.clock.now(), args);
        } else {
            self.add_tx_inner(args);
        }

        self.update_state_metrics();
        self.update_accounts_with_gap(account_nonce_updates);
        Ok(())
