### Title
Fee-on-Transfer Token Undercollateralization in `initTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` does not measure the actual token amount received by the bridge when pulling ERC20 tokens via `safeTransferFrom`. For fee-on-transfer tokens (not in `isBridgeToken`, not in `customMinters`), the bridge receives `amount − transferFee` but encodes the full caller-supplied `amount` in the Wormhole message. NEAR then stores and signs for the inflated `amount`, creating a persistent undercollateralization: the bridge holds less than what NEAR believes is locked, allowing the first user to bridge back to drain tokens that belong to later depositors.

---

### Finding Description

**EVM side — no balance-delta check**

In `OmniBridge.initTransfer`, the `else` branch for ordinary ERC20 tokens is:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // requested amount, not actual received
);
``` [1](#0-0) 

Immediately after, the unmodified `amount` parameter is forwarded to `initTransferExtension`:

```solidity
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // <-- inflated, not actual received
    fee, nativeFee, recipient, message, extensionValue
);
``` [2](#0-1) 

There is no `balanceBefore`/`balanceAfter` guard. For a 1%-fee-on-transfer token, `safeTransferFrom(sender, bridge, 1000)` delivers 990 to the bridge, but `amount = 1000` is what gets published.

**Wormhole message encodes the inflated amount**

`OmniBridgeWormhole.initTransferExtension` serialises the caller-supplied `amount` directly into the VAA payload:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) [4](#0-3) 

**NEAR side — stores and signs the inflated amount**

`fin_transfer_callback` on NEAR reads the amount directly from the proof and stores it as the locked balance:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [5](#0-4) 

`sign_transfer` then normalises and signs `amount_without_fee()` — derived from the inflated stored amount — for the destination chain:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee()...
    decimals,
);
``` [6](#0-5) 

**EVM `finTransfer` — transfers from actual bridge balance**

When the signed payload is presented back to the EVM bridge, `finTransfer` calls:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [7](#0-6) 

The bridge's actual balance is `amount − transferFee` per deposit, but the signed payload authorises `amount`. The gap is the fee-on-transfer amount silently absorbed on each deposit.

---

### Impact Explanation

With two depositors of the same fee-on-transfer token (1% fee, each depositing 1 000):

| Step | Bridge balance | NEAR wrapped supply |
|---|---|---|
| User A deposits 1 000 | 990 | 1 000 |
| User B deposits 1 000 | 1 980 | 2 000 |
| User A bridges back 1 000 | 980 | 1 000 |
| User B bridges back 1 000 | **REVERTS** (only 980 available) | 1 000 (stuck) |

User A recovers 1 000 despite having net-deposited only 990 — the 10-token shortfall is taken from User B's locked collateral. User B holds 1 000 wrapped tokens that can never be fully redeemed. This is a direct break of the 1:1 collateralisation invariant and constitutes misdirection of value between depositors.

The severity scales with the fee rate and the number of depositors: a 10% fee token with ten depositors leaves the last depositor unable to redeem 10% of their position.

---

### Likelihood Explanation

- `logMetadata` is fully permissionless; any ERC20 can be registered.
- `deploy_token` on NEAR is permissionless once metadata is logged.
- `initTransfer` is permissionless for any registered token.
- Fee-on-transfer tokens are a well-established ERC20 pattern (e.g., USDT on some chains, reflection tokens, tax tokens).
- An attacker can deploy a token with a hidden or configurable transfer fee, register it, seed liquidity to attract other depositors, then bridge back first to extract the undercollateralised surplus.

---

### Recommendation

Use a balance-delta pattern for the `else` (plain ERC20) branch in `initTransfer`:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "fee-on-transfer token not supported");
```

Alternatively, pass `actualReceived` (cast to `uint128`) to `initTransferExtension` instead of `amount`, so the Wormhole message always reflects what the bridge actually holds. [8](#0-7) 

---

### Proof of Concept

1. Deploy `FeeToken` — a standard ERC20 that deducts 1% on every `transferFrom` (i.e., `transferFrom(from, to, 1000)` moves 990 to `to`).
2. Call `logMetadata(feeTokenAddress)` on `OmniBridgeWormhole` — permissionless.
3. On NEAR, call `deploy_token` with the resulting Wormhole VAA — permissionless.
4. **Victim** calls `initTransfer(feeTokenAddress, 10_000, 0, 0, nearRecipient, "")`:
   - Bridge receives 9 900 `FeeToken`.
   - Wormhole VAA encodes `amount = 10_000`.
5. **Attacker** calls `initTransfer(feeTokenAddress, 1_000, 0, 0, nearRecipient, "")`:
   - Bridge receives 990 `FeeToken`.
   - Wormhole VAA encodes `amount = 1_000`.
6. On NEAR, relay both VAAs via `fin_transfer`. NEAR mints 10 000 + 1 000 = 11 000 wrapped tokens. Bridge holds 10 890.
7. Attacker calls `sign_transfer` for their 1 000 wrapped tokens → MPC signs payload for 1 000.
8. Attacker calls `finTransfer` on EVM → bridge transfers 1 000 `FeeToken` to attacker. Bridge now holds 9 890.
9. Victim calls `sign_transfer` for their 10 000 wrapped tokens → MPC signs payload for 10 000.
10. Victim calls `finTransfer` on EVM → `safeTransfer(victim, 10_000)` **reverts** — bridge only holds 9 890.

Assert: `bridge.balanceOf(feeToken) == 9_890 < 10_000` — victim's position is permanently under-redeemable by 110 tokens (the accumulated fee-on-transfer shortfall). The attacker recovered 1 000 despite net-depositing only 990, extracting 10 tokens from the victim's locked collateral.
