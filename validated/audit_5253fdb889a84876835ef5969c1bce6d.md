Looking at the full code path, I need to verify:
1. Whether `set_state_header` checks `chunk.shard_id() == shard_id`
2. Whether `set_state_finalize` uses `shard_id` vs `chunk.shard_id()` to derive `shard_uid`
3. Whether the Merkle proof check implicitly enforces shard identity

**Finding 1 — `set_state_header` has no shard_id consistency check**

`set_state_header` performs these validations:
- `validate_chunk_proofs` — checks chunk hash, tx_root, receipts_root. No `shard_id` check. [1](#0-0) 
- `verify_path` against `chunk_headers_root` — verifies the chunk is at the position encoded by the attacker-supplied `chunk_proof`. The function does **not** verify that the proof position corresponds to the `shard_id` parameter. [2](#0-1) 
- Receipt proof validation — uses the `shard_id` parameter for receipt hashing, but this only validates incoming receipts for the target shard, not the chunk's own shard identity. [3](#0-2) 
- State root node validation — no `shard_id` check. [4](#0-3) 

There is **no line** in `set_state_header` that asserts `chunk.shard_id() == shard_id`. The header is stored under `StateHeaderKey(shard_id, sync_hash)` regardless of what `chunk.shard_id()` returns. [5](#0-4) 

**Finding 2 — `set_state_finalize` derives `shard_uid` from the parameter, not from the chunk**

In `ChainUpdate::set_state_finalize`:
```rust
let shard_uid =
    shard_id_to_uid(self.epoch_manager.as_ref(), shard_id, block_header.epoch_id())?;
``` [6](#0-5) 

`shard_uid` is derived from the caller-supplied `shard_id`, not from `chunk.shard_id()`. The chunk's transactions and receipts are then applied under this `shard_uid`. [7](#0-6) 

**Finding 3 — The Merkle proof check does not bind the chunk to `shard_id`**

The `verify_path` call proves that the chunk hash is at the position encoded by the attacker-supplied `chunk_proof`. An attacker holding chunk A (legitimately in the block at position A) can supply the valid Merkle proof for position A while calling `set_state_header(shard_id=B, ...)`. The proof is valid (chunk A is at position A), and the function never checks that position A corresponds to shard B. [8](#0-7) 

**Finding 4 — The Merkle tree construction confirms position-to-shard binding is never checked**

`compute_state_response_header` builds the Merkle tree over chunks ordered by shard index and selects the proof at `prev_shard_index`. The validator in `set_state_header` never reconstructs this index from `shard_id` to cross-check the proof position. [9](#0-8) 

**Attack chain feasibility**

An unprivileged peer acting as a state sync responder (any connected peer can respond to state sync requests — no privileged role required) can:
1. Provide a `ShardStateSyncResponseHeader` for shard B containing chunk A's body, with the valid Merkle proof for chunk A's position in the block.
2. Provide state parts valid for chunk A's `prev_state_root` — `set_state_part` validates parts against the state root in the stored header, which is now chunk A's root. [10](#0-9) 
3. `set_state_finalize` then calls `apply_chunk` with `shard_uid` for shard B but chunk A's transactions/receipts, and `apply_chunk_postprocessing` writes `chunk_extra` for shard B reflecting shard A's execution. [11](#0-10) 

**However — the practical exploitability is blocked by a storage-level guard**

`apply_chunk` opens the trie for shard B using `shard_uid` (shard B) and `prev_state_root` from chunk A's header. [12](#0-11) 

For this to succeed without a `StorageError::MissingTrieValue`, the attacker must also have installed state parts that write chunk A's state root into shard B's trie namespace. `apply_state_part` is called with the `shard_id` parameter (shard B) and chunk A's state root — so it would write shard A's trie data into shard B's storage slot. This is a second, independent step that the attacker controls as the state sync provider.

**Conclusion on the exact corrupted value**

The question requires a concrete corrupted value. The corrupted value would be `chunk_extra.state_root` for shard B = result of executing shard A's transactions against shard A's state, stored under shard B's `shard_uid`. This is concrete and follows directly from the missing `chunk.shard_id() == shard_id` guard.

The attack chain is fully attacker-controlled (header + state parts), requires no privileged role, and the missing guard is confirmed absent in production code.

### Title
Missing shard_id consistency check in `set_state_header` allows unprivileged peer to corrupt `chunk_extra` for wrong shard via `set_state_finalize` — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` stores a `ShardStateSyncResponseHeader` under `StateHeaderKey(shard_id, sync_hash)` without verifying that the embedded chunk's `shard_id()` matches the `shard_id` parameter. A malicious state sync peer can supply a header for shard B containing chunk A's body with a valid Merkle proof, causing `set_state_finalize` to apply chunk A's transactions under shard B's `shard_uid` and write a corrupted `chunk_extra.state_root` for shard B.

### Finding Description
`set_state_header` in `chain/chain/src/state_sync/adapter.rs` (lines 368–532) performs five validation steps — chunk proof integrity, Merkle inclusion in the block, prev-chunk inclusion, receipt proof validity, and state root node validity — but none of them assert `chunk.shard_id() == shard_id`. The Merkle inclusion check (`verify_path` against `chunk_headers_root`) verifies only that the chunk hash is at the position encoded by the attacker-supplied proof; it does not verify that this position corresponds to the `shard_id` parameter. The header is then stored under `StateHeaderKey(shard_id, sync_hash)`.

`ChainUpdate::set_state_finalize` (`chain/chain/src/chain_update.rs`, line 513) derives `shard_uid` exclusively from the caller-supplied `shard_id`, not from `chunk.shard_id()`. It then calls `apply_chunk` with this `shard_uid` and the chunk's transactions/receipts, and passes the result to `apply_chunk_postprocessing`, which writes `chunk_extra` keyed by `shard_uid`.

### Impact Explanation
A syncing node that accepts a crafted state sync response will execute shard A's transactions against shard A's state but record the result as shard B's `chunk_extra.state_root`. All subsequent block processing for shard B on this node starts from a corrupted state root, causing the node to diverge from the canonical chain and produce or accept invalid blocks for shard B.

### Likelihood Explanation
Any connected peer can respond to state sync requests. The attacker needs to win the race to respond before a legitimate provider, which is feasible on a network with few peers or via eclipse. The crafted header uses only publicly available on-chain data (chunk body, Merkle proof, state parts).

### Recommendation
In `set_state_header`, after extracting the chunk, add an explicit guard:
```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
