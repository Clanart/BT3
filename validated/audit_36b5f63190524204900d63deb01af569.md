### Title
Missing Contract Class Existence Validation in `execute_replace_class` Allows Permanent Freezing of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not verify that the new class hash supplied to the `replace_class` syscall corresponds to a declared contract class. This is the direct StarkNet analog of the EVM "lack of contract existence check" bug: just as the EVM silently succeeds when calling a nonexistent contract, the StarkNet OS silently accepts a `replace_class` call pointing to a nonexistent class hash, committing an invalid state transition. Any funds held by the affected contract are permanently frozen.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS reads the requested new `class_hash` from the syscall request and immediately writes it into the contract's `StateEntry` via `dict_update`, with no check that the hash is actually declared in the system:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. The OS commits whatever `class_hash` the contract provides, even if it is `0` or any arbitrary undeclared felt value.

Once this invalid class hash is committed to state, any future call to the contract reaches `execute_entry_point`, which performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // resolves to the invalid hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,           // no matching entry exists
);
``` [2](#0-1) 

`find_element` panics if the key is absent. The sequencer's blockifier will therefore reject every subsequent transaction that touches the contract, making the contract permanently inaccessible and all funds inside it permanently frozen.

The contrast with `execute_call_contract`, which also reads `state_entry.class_hash` without a zero-check, is instructive: that path at least fails loudly at execution time. `execute_replace_class` is worse because it silently persists the invalid hash into committed state before any failure can occur. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After `replace_class` is accepted with a non-existent class hash, the contract's `StateEntry.class_hash` is committed to the Merkle state root. No future transaction can successfully execute against the contract: every attempt panics inside `find_element` and is dropped by the sequencer. ETH, ERC-20 tokens, or any other assets held in the contract's storage become permanently inaccessible with no recovery path.

---

### Likelihood Explanation

Any contract that exposes an upgrade path (e.g., a proxy, a DAO-governed vault, a multisig) and does not independently validate the new class hash before calling `replace_class` is exploitable. An attacker who can trigger the upgrade call — through a governance attack, a reentrancy, a missing access-control check, or simply by deploying a malicious contract — can supply an undeclared class hash. The OS provides no backstop. The likelihood is **medium**: the attack requires influencing a contract's `replace_class` call, but the OS's failure to validate is the necessary and sufficient condition for the permanent damage.

---

### Recommendation

Inside `execute_replace_class`, before writing the new `class_hash` to `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the returned compiled-class hash is non-zero. This mirrors the check already implicitly performed in `execute_entry_point` but must be enforced eagerly at the point of the syscall, not lazily at the point of the next call.

---

### Proof of Concept

1. **Attacker deploys** a contract `Vault` that holds user funds and exposes an `upgrade(new_class_hash)` function that calls `replace_class(new_class_hash)` without validating the argument.
2. **Users deposit** tokens into `Vault`.
3. **Attacker calls** `upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. **OS executes** `execute_replace_class`: reads `class_hash = 0xdeadbeef` from the request, skips the missing existence check, and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. **State root is updated** with `Vault.class_hash = 0xdeadbeef`.
6. **Any subsequent call** to `Vault` reaches `execute_entry_point`, which calls `find_element(..., key=compiled_class_hash_of(0xdeadbeef))`. No such entry exists; `find_element` panics; the sequencer drops the transaction.
7. **All user funds** in `Vault` are permanently frozen with no recovery mechanism. [4](#0-3)

    alloc_locals;

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE

    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;

    );

        key=contract_address,
