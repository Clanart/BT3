### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by the caller has been declared in the protocol. Any deployed contract can call `replace_class` with an arbitrary, undeclared class hash, permanently bricking itself and freezing any funds it holds. The codebase even contains a developer TODO comment explicitly acknowledging this missing check.

---

### Finding Description

In `execute_replace_class`, the function reads `request.class_hash` from the attacker-controlled syscall request and directly updates the contract's state entry with this new class hash — with no validation that the class hash has ever been declared via a `declare` transaction (i.e., that it exists in `contract_class_changes`).

The relevant code:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The developer TODO on line 898 explicitly acknowledges the missing check. Compare this to `execute_declare_transaction`, which enforces `prev_value=0` to ensure a class is only declared once and validates the class hash pre-image via `finalize_class_hash`. [2](#0-1)

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

After `execute_replace_class` writes an undeclared class hash into the contract's `StateEntry`, every subsequent call to that contract will fail at the class-lookup stage — the OS cannot find a compiled class for the undeclared hash. There is no recovery path: the contract's class hash is committed to state, and no entry point can be invoked to move funds out. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

**High.**

The attack requires only:
1. Deploying a contract (permissionless on StarkNet).
2. That contract calling the `replace_class` syscall with an arbitrary felt value as the class hash.

No privileged role, leaked key, or special condition is required. The `REPLACE_CLASS_SELECTOR` path is unconditionally reachable from any executing contract context. The attacker fully controls `request.class_hash` — it is read directly from the syscall segment written by the executing contract.

---

### Recommendation

Add a validation step in `execute_replace_class` to confirm that the provided class hash has a non-zero compiled class hash entry in `contract_class_changes` before updating the contract state:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // Verify the class hash has been declared.
    let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
    assert_not_zero(compiled_class_hash);

    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}
    ...
}
```

This mirrors the pattern used in `execute_declare_transaction`, which enforces `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`. [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `VictimContract` — a contract that holds user funds (e.g., acts as a vault or ERC-20 holder). This is a normal, permissionless operation.
2. `VictimContract` (or an attacker-controlled contract that calls into it) executes a transaction that internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary felt never declared on-chain).
3. The OS dispatches to `execute_replace_class` via `REPLACE_CLASS_SELECTOR`.
4. `execute_replace_class` reads `request.class_hash = 0xdeadbeef`, skips any declared-class validation (the TODO confirms none exists), and calls `dict_update` to write `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. The block is proven and committed. `VictimContract`'s on-chain class hash is now `0xdeadbeef`.
6. Any future `call_contract` or `invoke` targeting `VictimContract` fails: the OS cannot resolve `0xdeadbeef` to a compiled class, so no entry point can execute.
7. All funds in `VictimContract`'s storage are permanently frozen with no recovery path. [5](#0-4)
