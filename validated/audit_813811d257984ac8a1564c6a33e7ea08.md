### Title
`migrate_deployed_token` Fails to Migrate `locked_tokens` Entries, Permanently Disabling Cross-Chain Collateralization Guard - (File: near/omni-bridge/src/lib.rs)

### Summary
`migrate_deployed_token` updates five token-identity mappings but silently omits migrating `locked_tokens` entries from `old_token` to `new_token`. After a legitimate DAO migration, every subsequent `lock_tokens_if_needed` and `unlock_tokens_if_needed` call for the new token silently returns `LockAction::Unchanged`, permanently bypassing the bridge's collateralization guard for that token.

### Finding Description
`migrate_deployed_token` (lines 1606–1664) updates:

- `deployed_tokens` (remove old / insert new)
- `deployed_tokens_v2` (remove old / insert new)
- `token_id_to_address` (remove old / insert new)
- `token_address_to_id` (overwrite with new)
- `migrated_tokens` (record old→new) [1](#0-0) 

It does **not** touch `locked_tokens`, which is keyed by `(ChainKind, AccountId)` where `AccountId` is the NEAR token account ID. [2](#0-1) 

After migration:
- `locked_tokens[(chain, old_token)]` remains with stale values (orphaned, can never be decremented)
- `locked_tokens[(chain, new_token)]` does not exist

Both `lock_tokens` and `unlock_tokens` in `token_lock.rs` guard on key existence with an early-return `LockAction::Unchanged` when the key is absent: [3](#0-2) [4](#0-3) 

The `unlock_tokens` path also contains the critical collateralization assertion: [5](#0-4) 

Because `(chain, new_token)` is absent, `unlock_tokens_if_needed` returns `Unchanged` without ever reaching this `require!`. The guard is permanently disabled.

`unlock_tokens_if_needed` is called in `process_fin_transfer_to_other_chain` to verify that enough tokens are locked on the origin chain before releasing them: [6](#0-5) 

`lock_tokens_if_needed` is called in the same function to record tokens locked on the destination chain: [7](#0-6) 

Both calls silently no-op for `new_token` after migration.

### Impact Explanation
`locked_tokens` is the bridge's cross-chain collateralization guard for deployed tokens. It tracks how many tokens are locked on each non-origin chain. After migration the guard is permanently disabled for the migrated token:

1. **Unlock bypass:** `unlock_tokens_if_needed` no longer enforces `available >= amount`. Any user holding a valid cross-chain proof can finalize a transfer from a non-origin chain even if zero tokens were ever locked there, allowing redemption of unbacked supply.
2. **Lock accounting lost:** `lock_tokens_if_needed` no longer records new locks. The bridge loses all visibility into how many tokens are outstanding on each chain, making future collateralization checks meaningless even if `locked_tokens` is later re-initialized via `set_locked_tokens`.

This is a balance/accounting corruption that breaks bridge collateralization — matching the **High** impact tier.

### Likelihood Explanation
`migrate_deployed_token` is a documented, expected DAO operation (e.g., upgrading a deployed token contract to a new account). It is not a malicious action; the bug is in the function's incomplete state update. Once called, the broken state is permanent and affects every subsequent transfer of the migrated token. Any user who submits a valid cross-chain proof after migration can exploit the bypassed guard.

### Recommendation
Inside `migrate_deployed_token`, after updating the identity mappings, iterate over all `ChainKind` variants and re-key any existing `locked_tokens` entries from `old_token` to `new_token`:

```rust
for chain_kind in ChainKind::all_variants() {
    let key_old = (chain_kind, old_token.clone());
    if let Some(amount) = self.locked_tokens.remove(&key_old) {
        self.locked_tokens.insert(&(chain_kind, new_token.clone()), &amount);
    }
}
```

This preserves the accumulated lock accounting across the migration and ensures `lock_tokens_if_needed` / `unlock_tokens_if_needed` continue to function correctly for the new token ID.

### Proof of Concept
1. Pre-state: `locked_tokens[(Sol, eth.old.near)] = 5000` (5000 tokens locked on Solana for the ETH-bridged token).
2. DAO calls `migrate_deployed_token(Eth, eth.old.near, eth.new.near)`.
3. Post-state: `locked_tokens[(Sol, eth.old.near)] = 5000` (orphaned); `locked_tokens[(Sol, eth.new.near)]` absent.
4. Attacker obtains a valid Solana proof for a transfer of 10 000 tokens (more than were ever locked).
5. Attacker calls `fin_transfer` with that proof. `process_fin_transfer_to_other_chain` calls `unlock_tokens_if_needed(Sol, eth.new.near, 10_000)`.
6. `unlock_tokens` looks up `(Sol, eth.new.near)` → key absent → returns `Unchanged` without the `require!(available >= amount)` check.
7. Bridge proceeds to mint/release 10 000 tokens on NEAR despite only 5 000 ever being locked on Solana.
8. Collateralization invariant is broken; unbacked supply is created. [8](#0-7) [9](#0-8) [10](#0-9)

    pub fn migrate_deployed_token(
        &mut self,
        origin_chain: ChainKind,
        old_token: AccountId,
        new_token: AccountId,
    ) {
        require!(
            env::attached_deposit() >= NEP141_DEPOSIT,
            BridgeError::NotEnoughAttachedDeposit.as_ref()
        );

        require!(
            self.deployed_tokens.remove(&old_token),
            BridgeError::OldTokenNotDeployed.as_ref(),
        );
        require!(
            self.deployed_tokens.insert(&new_token),
            BridgeError::TokenExists.as_ref()
        );
        self.deployed_tokens_v2.remove(&old_token);
        self.deployed_tokens_v2.insert(&new_token, &origin_chain);

        let origin_address = self
            .token_id_to_address
            .remove(&(origin_chain, old_token.clone()))
            .near_expect(BridgeError::FailedToGetTokenAddress);

        require!(
            self.token_id_to_address
                .insert(&(origin_chain, new_token.clone()), &origin_address)
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );

        self.token_address_to_id
            .insert(&origin_address, &new_token)
            .near_expect(BridgeError::ExpectedToOverwriteTokenAddress);

        require!(
            self.migrated_tokens
                .insert(&old_token, &new_token)
                .is_none(),
            BridgeError::TokenAlreadyMigrated.as_ref()
        );

        ext_token::ext(new_token.clone())
            .with_static_gas(STORAGE_DEPOSIT_GAS)
            .with_attached_deposit(NEP141_DEPOSIT)
            .storage_deposit(&env::current_account_id(), Some(true))
            .detach();

        env::log_str(
            &OmniBridgeEvent::MigrateTokenEvent {
                old_token_id: old_token,
                new_token_id: new_token,
            }
            .to_log_string(),
        );
    }

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
