### Title
Fee-on-Transfer Token Accounting Mismatch Breaks Bridge Collateralization - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` and `OmniBridgeWormhole.initTransferExtension` record the user-supplied `amount` in the cross-chain Wormhole message without verifying the actual tokens received by the vault. For fee-on-transfer ERC20 tokens, the vault receives `amount - fee` while NEAR is instructed to release the full `amount`, permanently undercollateralizing the EVM vault.

### Finding Description
In `OmniBridge.initTransfer`, the native-token branch performs:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // user-supplied, not verified post-transfer
);
``` [1](#0-0) 

The function then passes the original `amount` (not the actual received balance) to `initTransferExtension`: [2](#0-1) 

`OmniBridgeWormhole.initTransferExtension` encodes this unverified `amount` directly into the Wormhole cross-chain message: [3](#0-2) 

No pre/post balance check is performed anywhere between the `safeTransferFrom` call and the Wormhole message publication. The same pattern exists in `starknet/src/omni_bridge.cairo` `init_transfer`, which calls `transfer_from` and then emits the unverified `amount` in the `InitTransfer` event used for cross-chain settlement: [4](#0-3) 

### Impact Explanation
For every `initTransfer` call with a fee-on-transfer token (e.g., STA, PAXG, or future USDC/USDT with fees enabled):

- EVM vault receives `amount - transferFee` tokens.
- NEAR receives a Wormhole message claiming `amount` tokens were locked.
- NEAR releases `amount` tokens to the recipient (or mints `amount` bridged tokens).
- The vault is undercollateralized by `transferFee` per transaction.

Repeated bridging accumulates the deficit. When users bridge back from NEAR to EVM, `finTransfer` attempts to release the full recorded `amount` from the vault: [5](#0-4) 

Once the vault balance is exhausted, all subsequent `finTransfer` calls revert, permanently freezing funds for later users. This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**, and **permanent freezing of user funds in bridge vault flows**.

### Likelihood Explanation
Any unprivileged user can call `initTransfer` with any ERC20 token address — there is no whitelist or token validation. Fee-on-transfer tokens are a well-known ERC20 variant (STA, PAXG, and others). The attacker does not need any privileged role; they only need to hold a fee-on-transfer token and call the public `initTransfer` function. The deficit accumulates with every such call, making exploitation straightforward and repeatable.

### Recommendation
1. **Pre/post balance check**: Record the vault's token balance before and after `safeTransferFrom`, and use the delta as the actual bridged amount:
   ```solidity
   uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
   uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
   // use actualReceived instead of amount in the Wormhole message
   ```
2. **Token whitelist**: Maintain an admin-controlled whitelist of permitted ERC20 tokens and revert `initTransfer` for unlisted tokens.
3. Apply the same fix to `starknet/src/omni_bridge.cairo` `init_transfer`.

### Proof of Concept
1. Deploy or use an existing fee-on-transfer ERC20 token `FOT` with a 10% transfer fee.
2. Call `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
3. Inside `initTransfer`, `safeTransferFrom(msg.sender, address(this), 1000)` executes — vault receives only 900 FOT.
4. `initTransferExtension` is called with `amount = 1000`; the Wormhole message encodes `amount = 1000`. [6](#0-5) 
5. NEAR processes the Wormhole VAA and releases 1000 FOT-equivalent tokens to `alice.near`.
6. Vault holds 900 FOT but the protocol owes 1000 FOT — a 100-token deficit per call.
7. After 9 such calls, the vault holds 8100 FOT but owes 9000 FOT. The 10th `finTransfer` for 1000 FOT reverts, permanently freezing the last user's funds.
