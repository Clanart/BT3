### Title
Missing Class Hash Declaration Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by the caller is actually declared in the system. This is an unimplemented validation step explicitly acknowledged by a TODO in the code. Any contract can call `replace_class` with an arbitrary, undeclared class hash; the OS will accept it, update the contract's class hash in state, and produce a valid proof — permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function handles the `replace_class` syscall. It reads the current `StateEntry` for the calling contract, constructs a new `StateEntry` with the caller-supplied `class_hash`, and writes it into `contract_state_changes` via `dict_update`. The function does **not** verify that the new class hash exists in `contract_class_changes` (the dictionary tracking declared classes).

The missing check is explicitly acknowledged at line 898:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `contract_class_changes` dict is the OS-level record of all declared class hashes (mapping class hash → compiled class hash). The `execute_replace_class` function has access to `contract_state_changes` but does not cross-reference `contract_class_changes` to confirm the new class hash is present. No other guard in the function performs this check. [2](#0-1) 

This is structurally identical to the reported `mintBatch()` analog: a critical code path (class replacement) invokes a semantic operation (class hash validation) that is not implemented anywhere in the OS, leaving the invariant unenforced at proof time.

---

### Impact Explanation

After `replace_class` is processed with an undeclared hash:

1. The contract's `StateEntry.class_hash` is updated to the arbitrary value in `contract_state_changes`.
2. The OS produces a valid proof over this state transition — the L1 verifier accepts it.
3. Any subsequent transaction calling this contract causes the OS to look up the compiled class for the undeclared hash. No entry exists in the compiled class facts bundle, causing OS execution to fail for every future call.
4. All assets (ERC-20 balances, ETH, NFTs) held by the contract are permanently inaccessible.

This matches the **Critical — Permanent freezing of funds** impact.

---

### Likelihood Explanation

The attack requires only that a contract call the `replace_class` syscall with an arbitrary felt value. Any unprivileged user can:

- Deploy a contract (no special role required).
- Have that contract call `replace_class` with an undeclared hash.
- The OS, lacking the validation, accepts the syscall and updates state.

Because the TODO is in the OS (the authoritative proof layer), the blockifier's off-chain simulation is also expected to be inconsistent with a correct check, meaning the sequencer would not reject the transaction before inclusion. The attack is low-complexity and requires no leaked keys, no malicious operator, and no third-party compromise.

---

### Recommendation

Implement the missing check inside `execute_replace_class` before writing the new `StateEntry`. Specifically, after reading `class_hash` from the request, perform a `dict_read` on `contract_class_changes` to assert that an entry for `class_hash` exists (i.e., the class was previously declared). If no entry exists, write a failure response and return, consistent with how other syscalls handle invalid arguments (e.g., `write_failure_response` with `ERROR_INVALID_ARGUMENT`).

---

### Proof of Concept

1. Attacker deploys `MaliciousContract` containing logic: `replace_class(class_hash=0xdeadbeef)`.
2. Attacker submits an invoke transaction calling `MaliciousContract.__execute__`.
3. The OS dispatches `REPLACE_CLASS_SELECTOR` to `execute_replace_class` in `syscall_impls.cairo`.
4. `execute_replace_class` reads `class_hash = 0xdeadbeef` from the syscall request.
5. **No check** is performed against `contract_class_changes` — `0xdeadbeef` is not declared.
6. `dict_update` writes `new StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
7. The OS produces a valid proof; the L1 verifier accepts the new state root.
8. Any future transaction targeting `MaliciousContract` causes the OS to look up compiled class `0xdeadbeef`, finds nothing, and fails — permanently freezing any funds held by the contract. [1](#0-0)
