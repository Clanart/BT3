Looking at the external report's vulnerability class — **an accounting function that doesn't reflect an actual constraint, leading to incorrect/uncapped values** — I'll map this to the Omni Bridge codebase.

The analog is in the decimal-conversion accounting layer of the NEAR bridge.

---

### Title
Unchecked `u128` Overflow in `denormalize_amount` Permanently Freezes Bridged Funds — (File: `near/omni-bridge/src/lib.rs`)

### Summary
`denormalize_amount` performs an unchecked multiplication that can overflow `u128` when converting EVM-side token amounts to NEAR-side amounts. This causes `fin_transfer_callback` to panic, permanently freezing user funds: the EVM tokens are already burned/locked, but the NEAR bridge can never finalize the transfer.

### Finding Description

`denormalize_amount` at `near/omni-bridge/src/lib.rs` lines 2776–2778 is:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

The multiplication `amount * 10^diff_decimals` is unchecked. It does not account for the constraint that the result must fit in `u128`. This is directly analogous to the EIP-4626 `maxMint()` returning `type(uint256).max` without accounting for `maxSupply`: the function returns a value that ignores a real arithmetic bound.

This function is called in three critical paths:

**1. `fin_transfer_callback` (line 725)** — converts the EVM-side proof amount to NEAR decimals before releasing tokens:
```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [2](#0-1) 

**2. `fast_fin_transfer` (line 771)** — same conversion for fast-path transfers:
```rust
let denormalized_amount = Self::denormalize_amount(fast_fin_transfer_msg.amount.0, decimals);
``` [3](#0-2) 

**3. `claim_fee_callback` (line 1122)** — converts the destination-chain proof amount back to NEAR decimals to compute the fee:
```rust
let denormalized_amount = Self::denormalize_amount(
    fin_transfer.amount.0,
    self.token_decimals.get(&token_address)...
);
let fee = transfer_message.amount.0 - denormalized_amount;
``` [4](#0-3) 

**Overflow threshold:** For a token with 18 EVM decimals and 24 NEAR decimals (`diff_decimals = 6`), the overflow occurs when `amount > u128::MAX / 10^6 ≈ 3.4 × 10^32` in 18-decimal units — corresponding to approximately **340 trillion actual tokens**. Tokens such as SHIB (~589 trillion supply) and PEPE (~420 trillion supply) have total supplies that exceed this threshold.

**What happens on overflow:** NEAR contracts are compiled with `overflow-checks = true` by default (Rust's standard for WASM targets in production). The multiplication panics. Since `denormalize_amount` is called *before* `add_fin_transfer` marks the transfer as finalized, the transfer ID is never recorded in `finalised_transfers`. However, the EVM-side tokens are already burned/locked. The relayer cannot retry (same overflow every time). There is no emergency recovery path — `set_locked_tokens` adjusts accounting but cannot release the stuck tokens. [5](#0-4) 

### Impact Explanation
**Permanent, irrecoverable freeze of user funds.** A user who bridges a large-supply token (e.g., SHIB, PEPE) from EVM to NEAR in a single transaction exceeding the overflow threshold loses their tokens permanently: the EVM side burns them, and the NEAR side can never finalize the transfer. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation
**Low-to-medium.** The bridge is permissionless — any ERC-20 token can be bridged. Meme tokens with hundreds-of-trillions supplies (SHIB: 589T, PEPE: 420T) already exceed the overflow threshold for a 6-decimal difference. A single whale bridging a large position, or a protocol moving liquidity, could trigger this. The entry path requires no privileged access: any user can call `initTransfer` on EVM.

### Recommendation
Replace the unchecked multiplication with `checked_mul` and propagate a meaningful error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount
        .checked_mul(10_u128.pow(diff_decimals))
        .near_expect(BridgeError::AmountOverflow)
}
```

Additionally, add a pre-validation in `fin_transfer_callback` that rejects proofs whose denormalized amount would overflow, so the transfer can be cleanly rejected rather than permanently stuck.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`) on the NEAR bridge via `bind_token`.
2. On EVM, call `initTransfer` with `amount = (u128::MAX / 10^6) + 1` (≈ 3.4 × 10^32 in 18-decimal units). The EVM bridge burns the tokens successfully.
3. A relayer calls `fin_transfer` on NEAR with the valid proof.
4. `fin_transfer_callback` calls `denormalize_amount(amount, decimals)`.
5. `amount * 10^6` overflows `u128` → panic.
6. The transfer ID is never inserted into `finalised_transfers`; the relayer retries and panics again indefinitely.
7. The user's EVM tokens are permanently lost with no recovery path. [6](#0-5) [7](#0-6)
