### Title
Missing Class Hash Existence Check in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by the caller is actually declared in the contract class trie. This is directly analogous to the referral-mint bug: just as that bug allowed a user to reference a future, non-existent token ID to gain accounting benefits, here a contract can reference a non-existent (or not-yet-declared) class hash to replace its own class. The OS accepts and commits this invalid state transition. Any funds held by the affected contract are permanently frozen because no entry point can ever be dispatched against an undeclared class.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` processes the `replace_class` syscall:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment at line 898 is an explicit, self-acknowledged admission that the OS **does not** verify `class_hash` exists in `contract_class_changes` before writing it into the contract's state entry. The new class hash is taken directly from the syscall request and committed to state without any existence check. [1](#0-0) 

Compare this to `deploy_contract.cairo`, which **does** enforce an existence pre-condition before writing state — it asserts `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` to confirm the target address is fresh: [2](#0-1) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once the OS commits a `StateEntry` whose `class_hash` points to an undeclared class, every future call to that contract address will fail at class-lookup time: the OS cannot dispatch any entry point because the class bytecode does not exist. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible. The state commitment (Merkle root) will include this poisoned entry, and no subsequent transaction can recover the funds because the contract's `__execute__` and withdrawal entry points are unreachable.

---

### Likelihood Explanation

The attack requires only that a contract (deployed by any unprivileged user) issues a `replace_class` syscall with an arbitrary felt value as the class hash. No privileged role, leaked key, or external dependency is needed. The OS is the authoritative state-transition function; if it accepts the syscall, the invalid state is committed regardless of what the sequencer's off-chain validation does. A malicious sequencer can trivially include such a transaction, or a malicious contract can self-trigger it after accumulating user deposits.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the hash is present in `contract_class_changes` (i.e., it has been declared in the current block or in a prior committed state). This is exactly the check the TODO comment calls for:

```cairo
// Verify the class hash is declared.
let (class_entry) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(class_entry);  // class must exist
```

This mirrors the pattern used in `deploy_contract` where the pre-existing state of the target address is validated before any write.

---

### Proof of Concept

1. Attacker deploys contract `C` (any class) that accepts ETH/token deposits and exposes a `drain_to_undeclared` entry point.
2. Victims deposit funds into `C`; `C.storage` now holds balances.
3. Attacker calls `drain_to_undeclared` on `C`, which internally issues:
   ```
   replace_class(class_hash=0xdeadbeef)   // arbitrary undeclared felt
   ```
4. The OS executes `execute_replace_class`:
   - Reads `request.class_hash = 0xdeadbeef` (line 896).
   - Skips the missing existence check (line 898 TODO).
   - Writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes` (lines 902–910).
5. Block is proven and committed. The state root now contains `C → class_hash=0xdeadbeef`.
6. Any subsequent `call_contract` or `invoke` targeting `C` causes the OS to look up class `0xdeadbeef` in the class trie — it does not exist — execution halts.
7. All funds in `C` are permanently frozen with no recovery path. [3](#0-2)
