### Title
Unchecked u128 Multiplication in `denormalize_amount` Causes Panic and Permanent Fund Lock - (File: near/omni-bridge/src/lib.rs)

### Summary

`denormalize_amount` performs a bare `u128` multiplication with no overflow guard. Because `near/Cargo.toml` sets `overflow-checks = true`, any overflow panics and aborts the transaction. When triggered inside `fin_transfer_callback`, the user's tokens are already burned/locked on the source chain but the NEAR finalization permanently fails, irrecoverably locking funds.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← no checked_mul, no saturation
}
``` [1](#0-0) 

The workspace profile explicitly enables overflow trapping:

```toml
[profile.release]
overflow-checks = true
``` [2](#0-1) 

`denormalize_amount` is called in three production paths:

**Path 1 — `fin_transfer_callback`** (line 725): converts the amount from a verified cross-chain proof back to NEAR-native decimals before minting/releasing tokens. [3](#0-2) 

**Path 2 — `fast_fin_transfer`** (line 771): same conversion for fast-path relayer transfers. [4](#0-3) 

**Path 3 — `claim_fee_callback`** (line 1122): converts the finalized amount to compute the claimable fee. [5](#0-4) 

`denormalize_fee` also delegates to `denormalize_amount` for the fee component: [6](#0-5) 

The `Decimals` struct stores two `u8` fields: [7](#0-6) 

`diff_decimals = origin_decimals − decimals`. For a token registered with `origin_decimals = 24` (NEAR) and `decimals = 4` (EVM), `diff_decimals = 20`. Then:

```
10_u128.pow(20) = 10^20
u64::MAX * 10^20 ≈ 1.84 × 10^39  >  u128::MAX ≈ 3.40 × 10^38  → PANIC
```

The existing unit test for `denormalize_amount` only exercises `diff_decimals` values of 6 and 15, which do not overflow, leaving the overflow case untested: [8](#0-7) 

The EVM `InitTransfer` event carries `uint128 amount`, so any value up to `u128::MAX` is a valid on-chain input: [9](#0-8) 

### Impact Explanation

When `fin_transfer_callback` panics:

1. The NEAR transaction aborts — no tokens are minted or released on NEAR.
2. The source-chain tokens were already burned/locked by `initTransfer` on EVM/Solana/StarkNet before the proof was submitted.
3. There is no automatic refund path on the source chain for a failed NEAR finalization.
4. The transfer ID is not marked as finalised, but re-submitting the same proof triggers the same panic again.

Result: **permanent, irrecoverable lock of the user's bridged assets** — matching the "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds" impact class.

### Likelihood Explanation

- The overflow threshold depends on `diff_decimals`. For `diff_decimals = 20` (e.g., 24-decimal NEAR token bridged to a 4-decimal EVM representation), any amount exceeding `u128::MAX / 10^20 ≈ 3.4 × 10^18` in source-chain units triggers the panic.
- A user sending `u64::MAX` (≈ 1.84 × 10^19) tokens in that configuration hits the overflow.
- For `diff_decimals ≥ 39`, even `10_u128.pow(diff_decimals)` itself overflows before the multiplication, so any non-zero amount panics.
- The attacker does not need any privileged role — only the ability to call `initTransfer` on the source chain with a sufficiently large amount.
- The Solana side explicitly guards against this (test `finalize_transfer_amount_overflow` expects `ErrorCode` 6010), confirming the NEAR side is the unprotected path. [10](#0-9) 

### Recommendation

Replace the bare multiplication in `denormalize_amount` (and the subtraction in both `denormalize_amount` and `normalize_amount`) with checked arithmetic, returning an error on overflow:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Option<u128> {
    let diff = decimals.origin_decimals.checked_sub(decimals.decimals)?;
    let factor = 10_u128.checked_pow(diff.into())?;
    amount.checked_mul(factor)
}
```

Call sites should propagate the error and abort with a descriptive `BridgeError` rather than panicking, so the transaction fails cleanly without permanently locking source-chain funds (or, better, validate the amount on the source chain before locking).

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 4` via `bind_token` / `add_deployed_tokens`.
2. On EVM, call `initTransfer` with `amount = u64::MAX` (≈ 1.84 × 10^19 in 4-decimal units). Tokens are burned/locked.
3. Submit the resulting proof to NEAR's `fin_transfer`.
4. `fin_transfer_callback` calls `denormalize_amount(u64::MAX as u128, Decimals { decimals: 4, origin_decimals: 24 })`.
5. `diff_decimals = 20`; `u64::MAX * 10^20 ≈ 1.84 × 10^39 > u128::MAX`.
6. With `overflow-checks = true`, Rust panics → NEAR transaction aborts.
7. Source-chain tokens remain burned/locked with no recovery path. [1](#0-0) [11](#0-10)
