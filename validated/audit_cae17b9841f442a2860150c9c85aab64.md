### Title
Missing Chain ID in `deployToken`/`deploy_token` Signature Enables Cross-Chain Replay of Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`)

---

### Summary

The `deployToken` function on EVM and the `deploy_token` function on Starknet both verify a NEAR MPC-produced signature over a Borsh-encoded metadata payload. Neither encoding includes the destination chain ID. By contrast, `finTransfer`/`fin_transfer` on both chains explicitly binds `omniBridgeChainId` into the signed message. A single MPC-signed `deployToken` payload is therefore valid on every EVM deployment and on Starknet simultaneously, allowing an unprivileged attacker to replay a token-deployment signature from one chain onto any other chain where the same `nearBridgeDerivedAddress` is configured.

---

### Finding Description

**EVM — `OmniBridge.sol::deployToken`**

The Borsh-encoded message that is hashed and verified against the MPC signature is constructed as:

```
PayloadType::Metadata | token | name | symbol | decimals
```

No `omniBridgeChainId` byte is included. [1](#0-0) 

Compare with `finTransfer`, which encodes `omniBridgeChainId` twice (once for the destination chain field and once for the recipient chain field): [2](#0-1) 

**Starknet — `bridge_types.cairo::MetadataPayloadTrait::to_borsh`**

The same omission exists in the Starknet implementation. `MetadataPayload::to_borsh()` encodes only `PayloadType::Metadata | token | name | symbol | decimals`: [3](#0-2) 

`TransferMessagePayload::to_borsh(chain_id)` correctly accepts and embeds `chain_id` at two positions: [4](#0-3) 

The Starknet `deploy_token` entry point calls `_verify_borsh_signature` with the chain-ID-free encoding: [5](#0-4) 

While `fin_transfer` correctly passes `self.omni_bridge_chain_id.read()` into the borsh encoder: [6](#0-5) 

The `omniBridgeChainId` / `omni_bridge_chain_id` storage variable is set per-deployment and is unique per chain: [7](#0-6) 

---

### Impact Explanation

A single MPC-signed `deployToken` payload for chain A is byte-for-byte valid on chain B because the signed digest is identical. An attacker who observes a `deployToken` transaction on chain A can:

1. Extract the `(signatureData, metadata)` arguments.
2. Submit the identical call to the bridge contract on chain B.
3. The signature check passes (`ECDSA.recover(keccak256(borshEncoded), sig) == nearBridgeDerivedAddress`), because `borshEncoded` is the same on both chains.
4. A bridge token is deployed on chain B without NEAR MPC ever authorizing a deployment on chain B.
5. The `nearToEthToken` / `near_to_starknet_token` mapping on chain B is now populated.
6. The attacker can submit the chain-B `DeployToken` event as a proof to NEAR, causing NEAR to register the token as legitimately deployed on chain B.

If the legitimate deployment for chain B has not yet occurred, the attacker's replay permanently occupies the token slot (`ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED` on any subsequent legitimate call), blocking the authorized deployment. If the NEAR bridge subsequently tries to deploy the token on chain B through the normal flow, it will fail, leaving the token in an inconsistent state across chains.

This matches the allowed HIGH impact: **"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment."**

---

### Likelihood Explanation

- All `deployToken` transactions are public on-chain; extracting `(signatureData, metadata)` requires only a block explorer.
- The attacker needs no special role, no tokens, and no privileged access — only the ability to call a public function.
- The Omni Bridge is deployed on multiple EVM chains (Ethereum, Arbitrum, Base, etc.) and on Starknet, all sharing the same MPC key derivation path, making the replay surface wide.
- The attack is front-runnable: the attacker can monitor the mempool of chain A and submit the replay to chain B before the legitimate deployment arrives.

---

### Recommendation

Include the destination chain ID in the Borsh-encoded metadata payload that is signed by the MPC, mirroring the pattern already used in `finTransfer`/`fin_transfer`.

**EVM** — add `bytes1(omniBridgeChainId)` to the `deployToken` borsh encoding:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet** — change `MetadataPayloadTrait::to_borsh` to accept a `chain_id: u8` parameter and append it, and update `deploy_token` to pass `self.omni_bridge_chain_id.read()`.

The NEAR MPC signing path must also include the target chain ID in the payload it signs for `deployToken`, so that signatures are chain-scoped from the moment of issuance.

---

### Proof of Concept

1. Deploy the Omni Bridge on two EVM chains, chain A (`omniBridgeChainId = 2`) and chain B (`omniBridgeChainId = 3`), both using the same `nearBridgeDerivedAddress`.
2. On chain A, call `deployToken(sig, {token:"foo.near", name:"Foo", symbol:"FOO", decimals:18})`. The MPC produces `sig` over `keccak256(borsh(Metadata | "foo.near" | "Foo" | "FOO" | 18))`.
3. On chain B, call `deployToken(sig, {token:"foo.near", name:"Foo", symbol:"FOO", decimals:18})` with the identical `sig`.
4. The borsh encoding on chain B is identical (no chain ID is included), so `ECDSA.recover` returns `nearBridgeDerivedAddress`, the check passes, and the token is deployed on chain B.
5. The legitimate `deployToken` call for chain B now reverts with `ERR_TOKEN_EXIST`. [8](#0-7)
