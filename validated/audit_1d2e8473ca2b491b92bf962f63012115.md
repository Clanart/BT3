### Title
RPC Simulation and Fee Estimation Always Use `StarknetVersion::LATEST` Versioned Constants for Pending Block, Diverging from Actual Sequencer Execution - (`File: crates/apollo_rpc_execution/src/lib.rs`)

### Summary

When `starknet_estimateFee` or `starknet_simulateTransactions` is called with `block_id = "pending"`, the `create_block_context` function queries storage for the Starknet version of the pending block number. Because the pending block has not yet been committed to storage, `get_starknet_version` always returns `None`, and the code unconditionally falls back to `StarknetVersion::LATEST`. This causes the simulation to execute with the latest known versioned constants (including `validate_block_number_rounding`, `validate_timestamp_rounding`, `validate_max_sierra_gas`, and all syscall gas costs) rather than the version the pending block will actually be sequenced with. The result is an authoritative-looking but wrong fee estimate and simulation trace.

### Finding Description

In `create_block_context` (`crates/apollo_rpc_execution/src/lib.rs`), when `maybe_pending_data` is `Some`, the pending block number is computed as `block_context_number.unchecked_next()`: [1](#0-0) 

This `block_number` (the pending block) is then used to look up the Starknet version from storage: [2](#0-1) 

`get_starknet_version` returns `None` for any block number at or beyond the header marker (i.e., any block not yet committed): [3](#0-2) 

The `.unwrap_or(StarknetVersion::LATEST)` fallback therefore always fires for pending block simulation. The `ExecutionPendingData` struct passed in from the RPC layer does not carry a `starknet_version` field — it only carries `timestamp`, gas prices, `sequencer`, `l1_da_mode`, and state diff fields: [4](#0-3) 

So the pending block's actual Starknet version (present in the feeder-gateway/p2p pending block header) is silently discarded, and LATEST is always substituted.

The versioned constants selected from LATEST govern:

- `validate_block_number_rounding` and `validate_timestamp_rounding` — the values seen by `__validate__` via `get_block_number` / `get_block_timestamp` syscalls: [5](#0-4) 

- `validate_max_sierra_gas` — the gas cap for the `__validate__` entry point
- All syscall step costs used in fee calculation

These constants differ materially across versions. For example, `validate_block_number_rounding` is `1` in v0.13.0 (no rounding) but `100` in v0.13.2+, and `validate_timestamp_rounding` is `1` in v0.13.0 but `3600` in v0.13.2+: [6](#0-5) [7](#0-6) 

### Impact Explanation

**Impact: High** — `starknet_estimateFee` and `starknet_simulateTransactions` with `block_id = "pending"` return authoritative-looking wrong values.

**Concrete wrong values produced:**

1. **Wrong fee/gas estimate**: Syscall step costs and gas prices differ between versions. A fee estimate computed with LATEST constants will be wrong if the pending block is sequenced with an older version's constants.

2. **Wrong `__validate__` block-info values**: If simulation uses LATEST rounding constants (`block_number_rounding=100`, `timestamp_rounding=3600`) but the actual block is sequenced with v0.13.0 constants (rounding=1), the rounded `block_number` and `block_timestamp` seen by `__validate__` in simulation differ from those seen during actual sequencing. An account contract that checks `block_number % 100 == 0` in `__validate__` will pass simulation but fail actual execution (or vice versa) — the direct sequencer analog of M-01.

3. **Wrong `validate_max_sierra_gas` cap**: v0.13.0 uses `10_000_000_000` while v0.13.4+ uses `100_000_000`. A transaction consuming between 100M and 10B gas in `__validate__` will pass simulation (with old constants) but be rejected during actual sequencing (with new constants), or vice versa.

This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

**Likelihood: Medium.** This is triggered whenever:
- A client calls `starknet_estimateFee` or `starknet_simulateTransactions` with `block_id = "pending"` (a common pattern for pre-submission fee estimation)
- The node's binary version (LATEST) differs from the version the pending block will actually be sequenced with

This occurs during protocol upgrade windows when the sequencer has upgraded but sync nodes have not (or vice versa), and during any period where the pending block's declared `starknet_version` differs from the node's LATEST. The trigger requires no special privileges — any unprivileged RPC caller can reach this path.

### Recommendation

1. Add `starknet_version: StarknetVersion` to `ExecutionPendingData` (in `crates/apollo_rpc_execution/src/objects.rs`) and populate it from the pending block header's version string when constructing `ExecutionPendingData` in the RPC layer.

2. In `create_block_context`, when `maybe_pending_data` is `Some`, use `pending_data.starknet_version` directly instead of querying storage for the (non-existent) pending block number:

```rust
// Instead of:
let starknet_version = storage_reader
    .begin_ro_txn()?
    .get_starknet_version(block_number)?
    .unwrap_or(StarknetVersion::LATEST);

// Use:
let starknet_version = match maybe_pending_data {
    Some(pending_data) => pending_data.starknet_version,
    None => storage_reader
        .begin_ro_txn()?
        .get_starknet_version(block_number)?
        .unwrap_or(StarknetVersion::LATEST),
};
```

3. In `read_pending_data` (RPC layer), when the pending block is stale and a synthetic fallback is constructed, use the latest committed block's `starknet_version` for the fallback pending data.

### Proof of Concept

1. Deploy an account contract whose `__validate__` checks `get_block_number() % 100 == 0` (i.e., it expects the rounded value from LATEST versioned constants).

2. On a node where the latest committed block has `starknet_version = "0.13.0"` (rounding=1) but the node binary is at LATEST (rounding=100), call `starknet_simulateTransactions` with `block_id = "pending"` and a transaction from this account.

3. **Simulation result**: `create_block_context` uses LATEST constants (rounding=100). `block_info_for_validate` rounds `block_number` to the nearest 100. `__validate__` sees `block_number % 100 == 0` → passes. Simulation reports success.

4. **Actual sequencing**: The sequencer (running v0.13.0) uses rounding=1. `block_info_for_validate` does not round. `__validate__` sees the actual `block_number` which is not a multiple of 100 → fails. Transaction is rejected.

The simulation returned a success trace and a non-zero fee estimate for a transaction that will actually fail validation — the direct sequencer analog of M-01's simulation/execution divergence. [8](#0-7)

fn create_block_context(
    cached_state: &mut CachedState<ExecutionStateReader>,
    block_context_number: BlockNumber,
    chain_id: ChainId,
    storage_reader: &StorageReader,
    maybe_pending_data: Option<&PendingData>,
    execution_config: &ExecutionConfig,
    // TODO(shahak): Remove this once we stop supporting rpc v0.6.
    override_kzg_da_to_false: bool,
) -> ExecutionResult<BlockContext> {
    let (
        block_number,
        block_timestamp,
        l1_gas_price,
        l1_data_gas_price,
        l2_gas_price,
        sequencer_address,
        l1_da_mode,
    ) = match maybe_pending_data {
        Some(pending_data) => (
            block_context_number.unchecked_next(),
            pending_data.timestamp,
            pending_data.l1_gas_price,
            pending_data.l1_data_gas_price,
            pending_data.l2_gas_price,
            pending_data.sequencer,
            pending_data.l1_da_mode,
        ),
        None => {
            let header = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_context_number)?
                .expect("Should have block header.")
                .block_header_without_hash;
            (
                header.block_number,
                header.timestamp,
                header.l1_gas_price,
                header.l1_data_gas_price,
                header.l2_gas_price,
                header.sequencer,
                header.l1_da_mode,
            )
        }
    };
    let ten_blocks_ago = get_10_blocks_ago(&block_context_number, cached_state)?;

    let use_kzg_da = if override_kzg_da_to_false { false } else { l1_da_mode.is_use_kzg_da() };
    let starknet_version = storage_reader
        .begin_ro_txn()?
        .get_starknet_version(block_number)?
        .unwrap_or(StarknetVersion::LATEST);
    let block_info = BlockInfo {
        block_timestamp,
        sequencer_address: sequencer_address.0,
        use_kzg_da,
        block_number,
        // TODO(yair): What to do about blocks pre 0.13.1 where the data gas price were 0?
        gas_prices: GasPrices {
            eth_gas_prices: GasPriceVector {
                l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_wei)
                    .unwrap_or(NonzeroGasPrice::MIN),
                l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_wei)
                    .unwrap_or(NonzeroGasPrice::MIN),
                l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_wei)
                    .unwrap_or(NonzeroGasPrice::MIN),
            },
            strk_gas_prices: GasPriceVector {
                l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_fri)
                    .unwrap_or(NonzeroGasPrice::MIN),
                l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_fri)
                    .unwrap_or(NonzeroGasPrice::MIN),
                l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_fri)
                    .unwrap_or(NonzeroGasPrice::MIN),
            },
        },
        starknet_version,
    };
    let chain_info = ChainInfo {
        chain_id,
        fee_token_addresses: FeeTokenAddresses {
            strk_fee_token_address: execution_config.strk_fee_contract_address,
            eth_fee_token_address: execution_config.eth_fee_contract_address,
        },
        is_l3: false,
    };
    let versioned_constants = VersionedConstants::get(&starknet_version)?;

    let block_context = BlockContext::new(
        block_info,
        chain_info,
        versioned_constants.clone(),
        BouncerConfig::max(),
    );
    let next_block_number = block_context.block_info().block_number;

    pre_process_block(
        cached_state,
        ten_blocks_ago,
        next_block_number,
        &versioned_constants.os_constants,
    )?;
    Ok(block_context)
}

    pub fn block_info_for_validate(&self) -> BlockInfo {
        let block_number = self.block_info.block_number.0;
        let block_timestamp = self.block_info.block_timestamp.0;
        // Round down to the nearest multiple of validate_block_number_rounding.
        let validate_block_number_rounding =
            self.versioned_constants.get_validate_block_number_rounding();
        let rounded_block_number =
            (block_number / validate_block_number_rounding) * validate_block_number_rounding;
        // Round down to the nearest multiple of validate_timestamp_rounding.
        let validate_timestamp_rounding =
            self.versioned_constants.get_validate_timestamp_rounding();
        let rounded_timestamp =
            (block_timestamp / validate_timestamp_rounding) * validate_timestamp_rounding;
        BlockInfo {
            block_number: BlockNumber(rounded_block_number),
            block_timestamp: BlockTimestamp(rounded_timestamp),
            sequencer_address: 0_u128.into(),
            // TODO(Yoni): consider setting here trivial prices if and when this field is exposed.

            use_kzg_da: self.block_info.use_kzg_da,
            starknet_version: self.block_info.starknet_version,
        }
    }
