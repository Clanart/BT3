### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash provided by a contract is actually declared in `contract_class_changes`. This is an "Invalid Validation" bug — the OS accepts any arbitrary felt value as a replacement class hash without checking its existence. Any contract that calls `replace_class` with an undeclared or zero class hash will have its class permanently replaced with an invalid value, making the contract permanently uncallable and freezing all funds it holds.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the OS reads the requested class hash from the syscall request and immediately writes it into `contract_state_changes` without verifying that the hash corresponds to a declared class in `contract_class_changes`. The code itself acknowledges this gap with an explicit TODO:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `REPLACE_CLASS_SELECTOR` branch in `execute_syscalls` dispatches to this function for any contract, with no caller restriction:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
``` [2](#0-1) 

Compare this to `execute_call_contract`, which does enforce a protocol-level invariant by rejecting calls to `EXECUTE_ENTRY_POINT_SELECTOR`:

```cairo
if (request.selector == EXECUTE_ENTRY_POINT_SELECTOR) {
    write_failure_response(remaining_gas=remaining_gas, failure_felt=ERROR_INVALID_ARGUMENT);
    return ();
}
``` [3](#0-2) 

No analogous guard exists for `replace_class`. The OS enforces other protocol invariants (e.g., class declared only once via `prev_value=0` in `execute_declare_transaction`) but fails to enforce that a replacement class hash must be declared before use. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash in `contract_state_changes` is set to an undeclared value (e.g., `0x1`, `0xdeadbeef`, or any hash not present in `contract_class_changes`), every subsequent call to that contract will fail at entry point resolution with `ENTRYPOINT_NOT_FOUND` because the OS cannot locate the class. The contract becomes permanently uncallable. Any ERC-20 tokens, ETH, STRK, or other assets held in the contract's storage are permanently frozen with no recovery path, since the contract can no longer execute any function — including any recovery or migration logic.

---

### Likelihood Explanation

**High.** The `replace_class` syscall is a standard upgrade primitive available to every Cairo contract. Contracts with user-facing upgrade mechanisms (multisigs, DAOs, proxy patterns, or any contract accepting a class hash as a parameter) are directly exploitable. An attacker who can call such a contract's upgrade function — or who deploys a contract that calls `replace_class` with an invalid hash in a constructor or public function — can trigger permanent fund freezing. No privileged access is required; any unprivileged transaction sender can reach this code path.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, add a lookup into `contract_class_changes` to verify the hash is declared:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant already enforced in `execute_declare_transaction` (where `prev_value=0` ensures a class is declared at most once) and closes the gap identified by the existing TODO comment.

---

### Proof of Concept

1. Deploy `VictimVault` — a contract that holds user funds and exposes a public `upgrade(new_class_hash: felt)` function that calls `replace_class(new_class_hash)`.
2. Users deposit funds into `VictimVault`.
3. Attacker calls `VictimVault.upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. The OS executes `execute_replace_class`, skips the missing existence check, and writes `class_hash=0xdeadbeef` into `contract_state_changes` for `VictimVault`'s address.
5. All subsequent calls to `VictimVault` (withdraw, transfer, etc.) fail with `ENTRYPOINT_NOT_FOUND` because no class with hash `0xdeadbeef` exists.
6. All deposited funds are permanently frozen with no recovery path.
