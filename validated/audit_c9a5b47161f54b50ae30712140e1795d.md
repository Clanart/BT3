### Title
Missing `MintCloseAuthority` Extension Check Allows Decimal Manipulation in Token Registration - (File: `solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary

The `log_metadata` instruction in the Solana bridge program does not validate whether the Token-2022 `MintCloseAuthority` extension is enabled on the registered mint. An attacker can exploit this to register a native token with artificially high decimals, close the mint, reinitialize it at the same address with lower decimals, and leave the NEAR bridge holding a permanently stale decimal record. All subsequent cross-chain amount normalization and denormalization for that token will be corrupted.

### Finding Description

`log_metadata` is the Solana-side entry point for registering a native SPL/Token-2022 token with the NEAR bridge. It reads `self.mint.decimals` at call time and posts it via Wormhole to NEAR, where it is stored in the `token_decimals: LookupMap<OmniAddress, Decimals>` map and used for all future amount conversions.

The `LogMetadata` account constraint is:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

This only verifies that the bridge authority is **not** the mint authority (distinguishing native from bridged tokens). There is **no check** for the `MintCloseAuthority` extension. A grep across all production Solana files confirms `MintCloseAuthority` is referenced only in test helpers, never in production instruction logic.

The Token-2022 `MintCloseAuthority` extension permits the designated close authority to close (reclaim lamports from) a mint account when its supply is zero. After closure, the on-chain address is free and a new mint can be initialized there with entirely different parameters — including a different `decimals` value.

The vault PDA is seeded by `[VAULT_SEED, mint.key()]`. Because the mint's public key does not change across close-and-reinitialize, the vault account survives and remains usable with the new mint, giving the attacker a fully functional token registration backed by wrong decimal metadata on NEAR.

**Attack steps:**

1. Attacker creates a Token-2022 mint at address `M` with `MintCloseAuthority` extension, `decimals = 18`, assigning close authority to themselves.
2. Attacker calls `log_metadata` on Solana. The instruction posts a Wormhole message containing `decimals: 18` for mint `M`. NEAR processes the VAA and stores `token_decimals[Sol(M)] = 18`.
3. With supply still at 0, attacker invokes the `MintCloseAuthority` CPI to close mint `M`, reclaiming lamports.
4. Attacker reinitializes a new mint at address `M` with `decimals = 6`. The vault PDA still exists and is valid for the new mint.
5. NEAR's `token_decimals` map permanently records `18` for `Sol(M)`, while the live mint has `6`.

### Impact Explanation

The NEAR bridge uses `token_decimals` in two critical paths:

**`sign_transfer`** (NEAR → Solana direction):
```rust
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee(),
    decimals,
);
```

**`fin_transfer_callback`** (Solana → NEAR direction):
```rust
let decimals = self.token_decimals.get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let transfer_message = TransferMessage {
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    fee: Self::denormalize_fee(&init_transfer.fee, decimals),
    ...
};
```

With stale decimals (`18` stored, `6` actual):

- **Solana → NEAR**: A user locks `1e6` raw units (1 token at 6 decimals) in the vault. NEAR denormalizes using `18` decimals, producing an amount `10^12` times smaller than the true value. The user's NEAR-side credit is effectively zeroed — funds are permanently locked in the vault with no recoverable NEAR-side balance.
- **NEAR → Solana**: NEAR normalizes using `18` decimals, producing a signed payload amount `10^12` times smaller than intended. The Solana `finalize_transfer` instruction releases a negligible amount from the vault, while the full locked amount remains trapped.

Either direction results in permanent loss or irrecoverable lock of user funds, satisfying the **Critical** impact tier (permanent freezing / irrecoverable lock of user funds in bridge flows) and the **High** tier (decimal/accounting corruption that breaks bridge collateralization).

### Likelihood Explanation

**Low.** The attacker must:
- Deploy a Token-2022 mint with `MintCloseAuthority` (permissionless, costs only rent).
- Call `log_metadata` (permissionless public instruction).
- Close the mint while supply is 0 (trivially satisfied before any user deposits).
- Redeploy at the same address with different decimals.

No privileged role, leaked key, or colluding party is required. The entire sequence is executable by any unprivileged actor. The barrier is purely operational (multi-step setup), not cryptographic.

### Recommendation

In the `LogMetadata` account validation or in `LogMetadata::process`, add an explicit check that the `MintCloseAuthority` extension is **not** present on the mint. For Token-2022 mints, unpack the extension state and reject registration if `MintCloseAuthority` is enabled:

```rust
// Inside LogMetadata::process, for token_2022 mints:
let mint_data = self.mint.to_account_info().try_borrow_data()?;
let mint_with_ext = StateWithExtensions::<spl_token_2022::state::Mint>::unpack(&mint_data)?;
require!(
    mint_with_ext.get_extension::<MintCloseAuthority>().is_err(),
    ErrorCode::UnsupportedMintExtension
);
```

Alternatively, require that the `MintCloseAuthority` is explicitly set to `None` (i.e., the extension exists but the authority field is `COption::None`), which permanently disables the close capability.
