### Title
Missing `shard_id` Binding Validation in `set_state_header` Allows Cross-Shard State Substitution During State Sync - (File: chain/chain/src/state_sync/adapter.rs)

### Summary

`StateSyncAdapter::set_state_header` accepts a caller-supplied `shard_id` and a `ShardStateSyncResponseHeader` from a network peer, but never verifies that the chunk embedded in the header actually belongs to the requested shard. A malicious peer can supply a cryptographically valid header for shard X in response to a request for shard Y. All existing checks pass, the header is stored under shard Y's DB key, and subsequent state-part validation and application use shard X's state root — permanently corrupting the syncing node's state for shard Y.

### Finding Description

`set_state_header` performs five validation steps before committing to `DBCol::StateHeaders`:

1. `validate_chunk_proofs` — verifies the chunk's internal consistency (tx root, receipt root, etc.)
2. `verify_path(sync_prev_block_header.chunk_headers_root(), chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk is included in the block's chunk-headers Merkle tree
3. Receipt-proof chain validation
4. `validate_state_root_node` — verifies the state root node matches the chunk's `prev_state_root`

None of these steps check that `chunk.shard_id() == shard_id`. The Merkle tree at step 2 covers **all** shards' chunks in the block; a chunk from shard X has a perfectly valid Merkle proof in that tree regardless of which shard was requested. After all checks pass, the header is committed:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

`shard_id` here is the caller-supplied value, not the chunk's actual shard. The stored header now contains shard X's `prev_state_root` under shard Y's key. [1](#0-0) 

When `set_state_part` is later called for shard Y, it retrieves this header and extracts the state root:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// validates part against shard X's state_root
self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part)
``` [2](#0-1) 

State parts for shard X pass `validate_state_part` because they are cryptographically consistent with shard X's state root. They are then applied to shard Y via `apply_state_part`, writing shard X's trie data into shard Y's `ShardUId`-prefixed storage namespace. [3](#0-2) 

### Impact Explanation

**Severity: High**

A syncing node that accepts a cross-shard substituted header will:

- Apply shard X's full trie state (accounts, contracts, storage) into shard Y's storage prefix
- Produce incorrect `ChunkExtra` state roots for shard Y after `set_state_finalize`
- Generate invalid chunk state witnesses and endorsements for shard Y
- Diverge permanently from the canonical chain for shard Y

The corruption is committed to RocksDB and survives restarts. Recovery requires a full re-sync. [4](#0-3) 

### Likelihood Explanation

**Likelihood: Medium**

Any peer on the network can respond to state sync header requests — this is not a privileged role. The attacker only needs to:

1. Be a node that tracks shard X
2. Respond to the syncing node's `StateRequestHeader` for shard Y with a valid header for shard X (which the attacker legitimately possesses)
3. Respond to subsequent `StateRequestPart` messages with valid parts for shard X

The syncing node's `validate_sync_hash` check only verifies the `sync_hash` belongs to a known recent epoch; it does not constrain which shard's header a peer may return. [5](#0-4) 

### Recommendation

Add an explicit shard-id binding check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: reject headers whose embedded chunk belongs to a different shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: header chunk shard_id {:?} does not match \
         requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the pattern used in `validate_receipt_proof` where `from_shard_id` and `to_shard_id` are explicitly checked against expected values before any cryptographic verification. [6](#0-5) 

### Proof of Concept

1. Node B is syncing shard 1 from the network at `sync_hash = H`.
2. Malicious node M tracks shard 0 and has its full state header for `sync_hash = H`.
3. M intercepts B's `StateRequestHeader { shard_id: 1, sync_hash: H }` and responds with shard 0's `ShardStateSyncResponseHeader`.
4. B calls `set_state_header(shard_id=1, sync_hash=H, header_for_shard_0)`.
5. `validate_chunk_proofs` passes — shard 0's chunk is internally consistent.
6. `verify_path(block.chunk_headers_root, proof, ChunkHashHeight(shard0_chunk_hash, height))` passes — shard 0's chunk is legitimately in the block.
7. Receipt proofs and `validate_state_root_node` pass — all reference shard 0's data.
8. Header stored: `StateHeaderKey(shard_id=1, sync_hash=H)` → shard 0's header (containing shard 0's `prev_state_root`).
9. B requests state parts for shard 1; M provides valid parts for shard 0.
10. `set_state_part` validates each part against shard 0's state root — all pass.
11. `apply_state_part` writes shard 0's trie into shard 1's storage prefix.
12. `set_state_finalize` computes `ChunkExtra` for shard 1 using shard 0's transactions and receipts.
13. Node B's shard 1 state is now shard 0's state; B diverges from the canonical chain. [7](#0-6)

async fn apply_state_part(
    store: Store,
    runtime: Arc<dyn RuntimeAdapter>,
    computation_task_tracker: TaskTracker,
    cancel: CancellationToken,
    sync_hash: CryptoHash,
    shard_id: ShardId,
    part_id: u64,
    num_parts: u64,
    state_root: CryptoHash,
    epoch_id: EpochId,
) -> Result<StatePartApplyResult, near_chain::Error> {
    let key = StatePartKey(sync_hash, shard_id, part_id);
    let key_bytes = borsh::to_vec(&key).unwrap();
    let already_applied = store.exists(DBCol::StatePartsApplied, &key_bytes);
    if already_applied {
        tracing::debug!(target: "sync", ?key, "state part already applied, skipping");
        return Ok(StatePartApplyResult::AlreadyApplied);
    }
    return_if_cancelled!(cancel);
    let handle =
        computation_task_tracker.get_handle(&format!("shard {} part {}", shard_id, part_id)).await;
    return_if_cancelled!(cancel);
    handle.set_status("Loading part data from store");
    let bytes = store
        .get(DBCol::StateParts, &key_bytes)
        .ok_or_else(|| {
            near_chain::Error::DBNotFoundErr(format!(
                "No state part {} for shard {}",
                part_id, shard_id
            ))
        })?
        .to_vec();
    let state_part = StatePart::from_bytes(bytes)?;
    handle.set_status("Applying part data to runtime");
    runtime.apply_state_part(
        shard_id,
        &state_root,
        PartId { idx: part_id, total: num_parts },
        &state_part,
        &epoch_id,
    )?;
    tracing::debug!(target: "sync", ?key, "applied state part");

    // Mark part as applied.
    let mut store_update = store.store_update();
    store_update.set_ser(DBCol::StatePartsApplied, &key_bytes, &true);
    store_update.commit();

    Ok(StatePartApplyResult::Applied)

    fn validate_sync_hash(&self, sync_hash: &CryptoHash) -> SyncHashValidationResult {
        match self.is_sync_hash_from_known_recent_epoch(sync_hash) {
            Ok(true) => {}
            Ok(false) => {
                tracing::info!(
                    target: "sync",
                    ?sync_hash,
                    "sync_hash didn't pass validation; belongs to an unknown epoch"
                );
                return SyncHashValidationResult::Rejected;
            }
            Err(err) => {
                tracing::warn!(target: "sync", ?err, "failed to check sync_hash epoch");
                return SyncHashValidationResult::Rejected;
            }
        }

        let good_sync_hash = match self.get_sync_hash(sync_hash) {
            Ok(sync_hash) => sync_hash,
            Err(err) => {
                tracing::debug!(target: "sync", ?err, "failed to get sync_hash for state request");
                return SyncHashValidationResult::Rejected;
            }
        };

        if good_sync_hash.as_ref() == Some(sync_hash) {
            SyncHashValidationResult::Valid
        } else {
            tracing::warn!(
                target: "sync",
                "sync_hash didn't pass validation; possible divergence in sync hash computation"
            );
            SyncHashValidationResult::Rejected
        }

pub fn validate_receipt_proof(
    receipt_proof: &ReceiptProof,
    from_chunk: &ShardChunkHeader,
    target_chunk_shard_id: ShardId,
    outgoing_receipts_root: CryptoHash,
) -> Result<(), Error> {
    // Validate that from_shard_id is correct. The receipts must match the outgoing receipt root
    // for this shard, so it's impossible to fake it.
    if receipt_proof.1.from_shard_id != from_chunk.shard_id() {
        return Err(Error::InvalidChunkStateWitness(format!(
            "Receipt proof for chunk {:?} is from shard {}, expected shard {}",
            from_chunk.chunk_hash(),
            receipt_proof.1.from_shard_id,
            from_chunk.shard_id(),
        )));
    }
    // Validate that to_shard_id is correct. to_shard_id is also encoded in the merkle tree,
    // so it's impossible to fake it.
    if receipt_proof.1.to_shard_id != target_chunk_shard_id {
        return Err(Error::InvalidChunkStateWitness(format!(
            "Receipt proof for chunk {:?} is for shard {}, expected shard {}",
            from_chunk.chunk_hash(),
            receipt_proof.1.to_shard_id,
            target_chunk_shard_id
        )));
    }
    // Verify that (receipts, to_shard_id) belongs to the merkle tree of outgoing receipts.
    if !receipt_proof.verify_against_receipt_root(outgoing_receipts_root) {
        return Err(Error::InvalidChunkStateWitness(format!(
            "Receipt proof for chunk {:?} has invalid merkle path, doesn't match outgoing receipts root",
            from_chunk.chunk_hash()
        )));
    }
    Ok(())
