### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Contract Bricking - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` function in the StarkNet OS unconditionally updates a contract's class hash in `contract_state_changes` without verifying that the new class hash is actually declared in `contract_class_changes`. This is an exact analog to the external report's pattern: a prerequisite check is skipped, yet state is committed as if it succeeded. Any contract can call `replace_class` with an arbitrary undeclared class hash, permanently bricking itself and freezing any funds it holds.

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` performs a gas check and then immediately writes the caller-supplied `class_hash` into `contract_state_changes`:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment at line 898 explicitly acknowledges the missing check. There is no lookup into `contract_class_changes` to confirm that `class_hash` was ever declared. The state update at lines 902–910 proceeds unconditionally. [1](#0-0) 

Compare this to the external report's pattern: `dropBoost` can return `false` (prerequisite fails), but the caller does not check the return value and proceeds to update state (`validator = _newValidator`). Here, the prerequisite is "the class hash must be declared," but the OS never checks it and proceeds to commit the new class hash to state.

The syscall dispatcher in `execute_syscalls.cairo` routes `REPLACE_CLASS_SELECTOR` directly to `execute_replace_class` with no additional validation layer: [2](#0-1) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, every future call to that contract will fail at class resolution time (the class does not exist in `contract_class_changes`). The contract becomes permanently non-callable. Any ERC-20 tokens, ETH, or STRK held in that contract's storage are irrecoverably frozen. There is no upgrade path because `replace_class` itself requires a callable contract to execute.

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is callable by any contract's own execution logic — no privileged role is required. A malicious contract can intentionally supply an undeclared class hash to brick itself after receiving funds (e.g., as part of a rug or griefing attack). Additionally, a buggy contract could accidentally pass an incorrect hash. The attack requires only a standard `invoke` transaction from an unprivileged sender.

### Recommendation

Before committing the new class hash to `contract_state_changes`, verify that it exists in `contract_class_changes`:

```cairo
// Check that the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the check already performed implicitly during `deploy_contract` (which reads the class from `contract_class_changes` before invoking the constructor) and closes the gap acknowledged by the existing TODO. [3](#0-2) 

### Proof of Concept

1. Deploy a contract `VictimVault` that holds user funds and exposes a `replace_class(new_hash)` external function that calls the `replace_class` syscall.
2. Users deposit funds into `VictimVault`.
3. An attacker (or the contract itself, via a malicious upgrade path) sends an `invoke` transaction calling `replace_class` with `new_hash = 0xdeadbeef` (an undeclared class hash).
4. The OS executes `execute_replace_class`:
   - Gas check passes.
   - `class_hash = 0xdeadbeef` is read from the request.
   - **No check** against `contract_class_changes` is performed (line 898 TODO).
   - `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. The block is proven and the state root is updated with `VictimVault.class_hash = 0xdeadbeef`.
6. Any subsequent call to `VictimVault` fails: the OS cannot resolve class `0xdeadbeef` from `contract_class_changes`.
7. All funds in `VictimVault` are permanently frozen with no recovery mechanism. [4](#0-3)
