### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the caller-supplied class hash corresponds to a previously declared contract class. This is a direct state-transition bypass: the OS accepts a class replacement with an arbitrary, undeclared class hash, permanently rendering the target contract non-executable and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), after deducting gas and reading the current `StateEntry`, the function unconditionally writes the new class hash into `contract_state_changes` without checking whether that class hash exists in `contract_class_changes` (i.e., has been declared via a `declare` transaction):

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The in-code `TODO` comment at line 898 explicitly acknowledges the missing check. The `contract_class_changes` dictionary is the OS-level record of all classes declared within the current block (populated by `execute_declare_transaction` via `dict_update{dict_ptr=contract_class_changes}(key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash)`). No cross-reference between the replacement class hash and this dictionary is performed.

This is structurally identical to the Putty Finance vulnerability: just as `cancel()` did not check whether an order was already filled before proceeding, `execute_replace_class` does not check whether the new class hash is declared before committing the state update. Both are missing state-validation guards that allow an invalid state transition to be proven and finalized.

---

### Impact Explanation

Once the OS proves a block in which a contract's class hash is set to an undeclared value, that state is committed on-chain. Any subsequent call to the affected contract will attempt to resolve the class hash against the compiled class facts; because no compiled class exists for the undeclared hash, execution will unconditionally fail. The contract becomes permanently non-executable. All ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently frozen with no recovery path.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is available to any Cairo contract without privilege restrictions. Realistic trigger paths include:

1. A contract that exposes an upgrade mechanism accepting a caller-supplied class hash (e.g., a proxy or upgradeable vault) — an unprivileged attacker passes an undeclared hash.
2. A contract with a logic bug that allows `replace_class` to be called with attacker-controlled input.
3. A contract owner who mistakenly supplies an undeclared hash (e.g., a hash computed off-chain before the corresponding `declare` transaction is confirmed).

In all cases the attacker entry point is an unprivileged transaction sender or contract deployer. No sequencer compromise or privileged key is required; the OS itself accepts the invalid state transition.

---

### Recommendation

Before committing the new class hash, verify it exists in `contract_class_changes` with a non-zero compiled class hash value. Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the result is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the guard already present in `execute_declare_transaction` (`prev_value=0` enforces a class may be declared only once) and closes the analogous gap in `execute_replace_class`.

---

### Proof of Concept

1. Deploy contract **Vault** holding user funds; its class is `class_A` (declared).
2. **Vault** exposes an `upgrade(new_class_hash)` entry point that calls the `replace_class` syscall with the supplied hash.
3. Attacker calls `upgrade(0xDEAD…)` where `0xDEAD…` is never declared.
4. `execute_replace_class` reads gas, reads `state_entry`, and writes `new_state_entry` with `class_hash = 0xDEAD…` — **no declared-class check is performed**.
5. The OS produces a valid proof; the state root is updated on L1 with **Vault**'s class set to `0xDEAD…`.
6. All subsequent calls to **Vault** fail at class resolution; all funds are permanently frozen. [1](#0-0) [2](#0-1)
