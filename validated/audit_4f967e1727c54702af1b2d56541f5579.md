### Title
Missing Zero-Check on `class_hash` Parameter in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo` and `deprecated_execute_syscalls.cairo`)

---

### Summary

Both the new and deprecated `execute_replace_class` syscall implementations accept a caller-supplied `class_hash` and write it directly into the contract's `StateEntry` without checking that it is non-zero. Because `UNINITIALIZED_CLASS_HASH = 0`, a contract that calls `replace_class(0)` permanently sets its own class hash to the uninitialized sentinel value, making itself permanently non-callable and freezing any funds it holds.

---

### Finding Description

`UNINITIALIZED_CLASS_HASH` is defined as the constant `0`: [1](#0-0) 

In `execute_replace_class` (new syscall path), the `class_hash` field from the request is read and stored directly into the contract's `StateEntry` with no zero-check: [2](#0-1) 

The developer-left TODO at line 898 acknowledges that class existence is not yet validated, but there is also no guard against `class_hash = 0`: [3](#0-2) 

The same flaw exists in the deprecated path: [4](#0-3) 

After the write, any future call to the contract reads `state_entry.class_hash` (now `0`) and builds an `ExecutionContext` with that value: [5](#0-4) 

No class with hash `0` exists, so all subsequent entry-point dispatches into that contract fail permanently. The contract's storage (and any ERC-20 balances or other assets it holds) become permanently inaccessible.

Additionally, `deploy_contract` enforces `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` before deploying, meaning a contract whose class hash was reset to `0` could be re-deployed over, corrupting its storage state: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class(class_hash=0)` — whether by a malicious actor who controls a contract, or by a buggy contract — permanently loses the ability to execute any entry point. All assets (tokens, NFTs, etc.) stored in that contract's storage become permanently frozen with no recovery path, because the OS state transition is committed to the Merkle tree and there is no upgrade or rescue mechanism once the class hash is set to `0`.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any deployed contract without any privileged role. A malicious contract deployer can deploy a contract whose logic calls `replace_class(0)` on itself or on any contract it controls, immediately and irreversibly freezing it. The attack requires only a standard invoke transaction — no special access, no operator cooperation, no external dependency.

---

### Recommendation

Add an `assert_not_zero(class_hash)` guard in both `execute_replace_class` implementations immediately after reading `class_hash` from the request, before any state update is performed:

```cairo
// In syscall_impls.cairo execute_replace_class:
let class_hash = request.class_hash;
assert_not_zero(class_hash);  // <-- add this

// In deprecated_execute_syscalls.cairo execute_replace_class:
let class_hash = syscall_ptr.class_hash;
assert_not_zero(class_hash);  // <-- add this
```

This mirrors the existing pattern used in `execute_declare_transaction`, which already guards `compiled_class_hash` with `assert_not_zero` before writing it to state: [7](#0-6) 

---

### Proof of Concept

1. Attacker deploys a contract `MaliciousContract` whose `__execute__` entry point calls the `replace_class` syscall with `class_hash = 0`.
2. Attacker sends an invoke transaction targeting `MaliciousContract.__execute__`.
3. The OS processes the `replace_class` syscall via `execute_replace_class` in `syscall_impls.cairo`. No zero-check is performed. `contract_state_changes` is updated: `StateEntry.class_hash = 0`.
4. The block is proven and the new state root (with `class_hash=0` for `MaliciousContract`) is committed on-chain.
5. Any subsequent call to `MaliciousContract` reads `class_hash=0` from state, builds an `ExecutionContext` with `class_hash=0`, and fails to find any entry point — the call reverts permanently.
6. All funds held in `MaliciousContract`'s storage are permanently frozen.
