### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key/Hash Guards, Corrupting Tree Root and Enabling Forged Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::insert` enforces uniqueness of both `KeyId` and leaf `Hash` before writing to the blob. `MerkleBlob::batch_insert` enforces those same guards only for the first two items it processes (via `self.insert()`), then switches to a fast path that writes leaf nodes directly to the blob via `insert_entry_to_blob` with **no duplicate checks**. An attacker who controls the batch input can silently insert a duplicate key or hash, leaving two leaf nodes with the same key in the blob while the in-memory cache (`block_status_cache`) is silently overwritten to point only to the newer one. The result is a permanently diverged blob/cache state: the Merkle root is computed over a tree that contains a ghost leaf, proofs of inclusion are generated against the wrong leaf position, and the tree fails integrity invariants that are only checked on explicit `check_integrity()` calls (disabled in production).

### Finding Description

**`MerkleBlob::insert` — the guarded path** [1](#0-0) 

Both guards (`contains_key` and `contains_leaf_hash`) must pass before any write occurs.

**`MerkleBlob::batch_insert` — the fast path that skips the guards** [2](#0-1) 

When `leaf_count <= 1`, the last two items popped from the vector go through `self.insert()` (guarded). All remaining items — and **all** items when `leaf_count > 1` — go through the loop at line 587 which calls `insert_entry_to_blob` directly, with no key or hash uniqueness check.

**`insert_entry_to_blob` silently overwrites the cache** [3](#0-2) 

It delegates to `add_leaf`, which performs a plain `HashMap::insert` — silently overwriting any existing entry for the same key or hash: [4](#0-3) 

**Concrete corruption sequence**

1. Tree already has ≥ 2 leaves (so `leaf_count > 1`).
2. Caller invokes `batch_insert` with a batch that includes `(key_existing, value_new, hash_new)` where `key_existing` is already present.
3. The fast-path loop writes a new `LeafNode { key: key_existing, … }` at a fresh blob index.
4. `add_leaf` overwrites `key_to_index[key_existing]` to point to the new index; the old leaf node remains in the blob at its original index, still connected to the tree via parent pointers.
5. The batch then builds internal nodes over the new subtree and grafts it in via `insert_subtree_at_key`.
6. The blob now contains **two** leaf nodes with `key_existing`; the cache knows only the newer one.
7. `get_leaf_by_key(key_existing)` returns the new leaf; proofs of inclusion are generated for the new position.
8. The Merkle root is computed over a tree that includes both leaves, so the root hash is wrong relative to any external commitment.
9. `check_integrity_on_drop` is `false` in production (`cfg!(test)` only), so the corruption is never detected. [5](#0-4) 

**Python binding exposes the fast path to untrusted input** [6](#0-5) 

`py_batch_insert` accepts caller-supplied `keys_values` and `hashes` lists with no pre-deduplication, forwarding them directly to `batch_insert`.

### Impact Explanation

A caller who controls the batch input can:

- **Corrupt the Merkle root**: the tree contains a ghost leaf that is included in all ancestor hash computations, producing a root that does not match any honest commitment.
- **Forge inclusion proofs**: `get_proof_of_inclusion` operates on the cache-visible leaf; the proof path is valid against the corrupted root but proves a different (attacker-chosen) value for the key.
- **Cause exclusion-proof failures**: a key that should be absent appears present (or vice versa) depending on which leaf the cache resolves.

This matches the allowed impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

- The Python binding `py_batch_insert` is a public API surface reachable from any Python caller that drives the DataLayer.
- No privilege is required: the attacker only needs to supply a list of `(KeyId, ValueId)` pairs and hashes.
- The bug is triggered whenever a batch contains a key already present in the tree (or a duplicate within the batch itself when the batch has ≥ 3 items).
- No error is returned; the operation silently succeeds, making the corruption invisible to callers.

### Recommendation

Add the same duplicate guards to the fast path of `batch_insert` before writing each leaf:

```rust
for ((key, value), hash) in keys_values_hashes {
    // Add these two guards:
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic ...
}
```

Alternatively, pre-deduplicate the input vector and cross-check against the existing cache before entering the fast path, consistent with how `insert` handles these invariants.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Insert 3 leaves so leaf_count > 1 (fast path is always taken in batch_insert)
    for i in 0i64..3 {
        blob.insert(KeyId(i), ValueId(i), &sha256(i), InsertLocation::Auto {}).unwrap();
    }

    // batch_insert with a duplicate of KeyId(0) — no error returned
    blob.batch_insert(vec![
        ((KeyId(0), ValueId(99)), sha256(99)),  // duplicate key!
        ((KeyId(10), ValueId(10)), sha256(10)),
        ((KeyId(11), ValueId(11)), sha256(11)),
    ]).unwrap();  // succeeds silently

    // Cache now maps KeyId(0) → new index; old leaf still in blob
    // check_integrity() would fail (leaf_count != cache length), but is not called in production
    // Merkle root is computed over a tree with a ghost leaf
    // proof_of_inclusion for KeyId(0) uses the new leaf position, not the original
}
```

```rust
    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
```

```rust
    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }
```
