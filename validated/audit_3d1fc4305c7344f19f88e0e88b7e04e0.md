### Title
`estimate_minimal_gas_vector` Omits Compiled-Class-Hash DA Cost for Declare Transactions, Admitting Transactions with Insufficient Resource Bounds — (File: crates/blockifier/src/fee/gas_usage.rs)

---

### Summary

`estimate_minimal_gas_vector` hard-codes `n_compiled_class_hash_updates: 0` for `Declare` transactions despite an explicit in-code TODO acknowledging it should be `1`. Because this function drives the pre-validation fee gate (`check_fee_bounds`), the gateway/mempool will admit any `Declare` transaction whose L2-gas bound falls in the gap between the underestimated minimum and the true minimum, i.e., transactions that are actually under-resourced pass admission and are only caught—and charged—at post-execution.

---

### Finding Description

`estimate_minimal_gas_vector` builds a `StateChangesCount` for each transaction type to derive the minimum DA gas and OS-step cost before execution:

```rust
// crates/blockifier/src/fee/gas_usage.rs  lines 163-169
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,
    n_modified_contracts: 1,
},
```

Every `Declare` transaction writes the `class_hash → compiled_class_hash` mapping to the state. That is exactly one compiled-class-hash update, contributing two felts to the on-chain data segment. With `gas_per_data_felt = 5120` (all versioned-constants files from v0.13.1 onward), the missing entry underestimates the DA component by **2 × 5120 = 10,240 L2 gas units** (or the equivalent L1 gas in non-KZG mode).

The underestimated vector is fed directly into `check_fee_bounds`, which is called unconditionally from `perform_pre_validation_stage` whenever `charge_fee` is true:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 370-378
fn check_fee_bounds(&self, tx_context: &TransactionContext) -> TransactionPreValidationResult<()> {
    let minimal_gas_amount_vector = estimate_minimal_gas_vector(
        &tx_context.block_context,
        self,
        &tx_context.get_gas_vector_computation_mode(),
    );
    ...
    if minimal_gas_amount > &resource_bounds.max_amount {
        // reject
    }
```

Because the minimum is 10,240 L2 gas units too low, any `Declare` transaction whose `l2_gas.max_amount` sits in the window `[actual_min − 10240, actual_min)` passes the admission gate, enters the mempool, is sequenced, and then fails post-execution with an `Insufficient max L2Gas` revert. The user is charged the maximum fee (snapped to their declared bounds) and must resubmit.

The same underestimated vector also propagates into the KZG-DA OS-step count:

```rust
// crates/blockifier/src/fee/gas_usage.rs  lines 186-190
let data_segment_length = get_onchain_data_segment_length(&state_changes_by_account_tx);
let os_steps_for_type = versioned_constants
    .os_resources_for_tx_type(&tx.tx_type(), tx.extended_calldata_length())
    .n_steps
    + versioned_constants.os_kzg_da_resources(data_segment_length).n_steps;
```

so the L2-gas component of the minimum is doubly underestimated when `use_kzg_da = true`.

---

### Impact Explanation

The gateway/mempool admits `Declare` transactions whose resource bounds are genuinely insufficient to cover the compiled-class-hash DA cost. These transactions:

1. Pass `check_fee_bounds` (underestimated minimum < user's bound).
2. Are sequenced into a block.
3. Revert post-execution with `Insufficient max L2Gas`.
4. Charge the user the maximum fee (their declared bound × price) rather than the actual cost.

This is a direct match for **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**: the admission gate accepts `Declare` transactions that carry provably insufficient resource bounds.

---

### Likelihood Explanation

The underestimation window is fixed at 10,240 L2 gas units (or larger under KZG DA due to the OS-step delta). Any user or tool that computes the minimum bound by calling `estimate_minimal_gas_vector` directly (as done in tests and in the gateway's own pre-validation path) and then submits a `Declare` transaction with exactly that bound will land in the affected window. The window is wide enough to be hit by off-by-one tooling or by users who tightly pack their resource bounds.

---

### Recommendation

Change the `Declare` arm of `estimate_minimal_gas_vector` to reflect the actual state changes:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1,   // was 0; Declare always writes compiled_class_hash
    n_modified_contracts: 1,
},
```

This is already acknowledged by the in-code TODO. Fixing it closes the admission gap and aligns the pre-validation minimum with the actual post-execution cost.

---

### Proof of Concept

```rust
// Add to crates/blockifier/src/fee/gas_usage_test.rs
#[test]
fn test_declare_minimal_gas_underestimates_compiled_class_hash_da() {
    let mut block_context = BlockContext::create_for_testing();
    block_context.block_info.use_kzg_da = true;

    // Build a minimal Declare V3 transaction.
    let contract = FeatureContract::Empty(CairoVersion::Cairo1(RunnableCairo1::Casm));
    let class_info = calculate_class_info_for_testing(contract.get_class());
    let declare_tx = AccountTransaction::new_with_default_flags(
        executable_declare_tx(
            declare_tx_args! {
                class_hash: contract.get_class_hash(),
                compiled_class_hash: contract.get_compiled_class_hash(&HashVersion::V2),
                resource_bounds: default_all_resource_bounds(),
            },
            class_info,
        )
    );

    // Minimum as currently computed (n_compiled_class_hash_updates = 0).
    let underestimated_min = estimate_minimal_gas_vector(
        &block_context, &declare_tx, &GasVectorComputationMode::All
    );

    // Correct minimum (n_compiled_class_hash_updates = 1).
    let correct_da_delta = get_da_gas_cost(
        &StateChangesCount {
            n_storage_updates: 0,
            n_class_hash_updates: 0,
            n_compiled_class_hash_updates: 1,
            n_modified_contracts: 0,
        },
        true, // use_kzg_da
    );
    let correct_min_l2 = underestimated_min.l2_gas.0 + correct_da_delta.l1_data_gas.0; // simplified

    // A resource bound in the gap passes check_fee_bounds but fails post-execution.
    let gap_bound = GasAmount(underestimated_min.l2_gas.0 + 1);
    assert!(gap_bound.0 < correct_min_l2,
        "gap_bound should be below the correct minimum");

    // Submit with gap_bound as l2_gas.max_amount — pre-validation passes,
    // post-execution reverts with Insufficient max L2Gas.
    let mut state = /* ... test state ... */;
    let gap_tx = AccountTransaction::new_with_default_flags(
        executable_declare_tx(
            declare_tx_args! {
                class_hash: contract.get_class_hash(),
                compiled_class_hash: contract.get_compiled_class_hash(&HashVersion::V2),
                resource_bounds: create_all_resource_bounds(
                    GasAmount(0), GasPrice(1),
                    gap_bound, block_context.block_info.gas_prices
                        .strk_gas_prices.l2_gas_price.get(),
                    GasAmount(0), GasPrice(1),
                ),
            },
            class_info_clone,
        )
    );
    let result = gap_tx.execute(&mut state, &block_context).unwrap();
    // Transaction was admitted (no pre-validation error) but reverted post-execution.
    assert!(result.is_reverted());
    assert!(result.revert_error.unwrap().to_string().contains("Insufficient max L2Gas"));
}
```

The test demonstrates that a `Declare` transaction with `l2_gas.max_amount` set to exactly `estimate_minimal_gas_vector(...).l2_gas + 1` passes `check_fee_bounds` (admission) but is reverted post-execution, confirming the admission gate accepts a transaction with provably insufficient resource bounds.
