Audit Report

## Title
Swap Conservation Violation in `buyToken0InBinSpecifiedIn`: Fallback Overcharges Trader Without Updating Output — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary
In `buyToken0InBinSpecifiedIn`, when both the analytical pass (lines 581–596) and the proportional-scaling refinement (lines 598–618) undershoot — i.e., the recomputed `totalIn1Scaled` remains below `state.amountSpecifiedRemainingScaled` while `targetPos < maxFinalBinPos` — the fallback at lines 620–622 forcibly sets `totalIn1Scaled = state.amountSpecifiedRemainingScaled` without updating `out0Scaled`. Settlement then consumes the full remaining input but credits only the token0 output corresponding to the undershot `targetPos`, violating the swap conservation invariant and causing direct loss of trader principal.

## Finding Description

The function enters the `else` branch at line 579 only after confirming (lines 566–578) that consuming the full bin up to `maxFinalBinPos` requires **more** than `remaining`: [1](#0-0) 

**Pass 1 (analytical, lines 581–596):** Computes `targetPos` via a closed-form approximation, then derives `out0Scaled` and `totalIn1Scaled`. Due to bin non-linearity, this can undershoot (`totalIn1Scaled < remaining`). [2](#0-1) 

**Pass 2 (proportional scaling, lines 598–618):** If still undershooting and `targetPos < maxFinalBinPos`, scales `targetPos` proportionally upward. When `scaledTarget > maxFinalBinPos`, it clamps to `maxFinalBinPos` (line 606) — after which the recomputed `totalIn1Scaled` exceeds `remaining` (guaranteed by the initial check), so the scale-down block at line 624 correctly handles it. But when `scaledTarget <= maxFinalBinPos`, `targetPos = scaledTarget` and `out0Scaled`/`totalIn1Scaled` are recomputed — and can still undershoot due to curve non-linearity. [3](#0-2) 

**Fallback (lines 620–622):** If after Pass 2 the undershoot condition still holds, the code sets `totalIn1Scaled = remaining` but **leaves `out0Scaled` unchanged** at the value for the undershot `targetPos`: [4](#0-3) 

After line 621, `totalIn1Scaled == remaining`, so the scale-down block at line 624 (`if (totalIn1Scaled > remaining)`) is skipped entirely. Settlement then proceeds: [5](#0-4) 

- `state.amountSpecifiedRemainingScaled -= totalIn1Scaled` → full input consumed
- `state.amountCalculatedScaled += out0Scaled` → only the undershot token0 output credited

The pool receives `remaining` token1 but releases only `out0Scaled` for `targetPos < maxFinalBinPos`. The correct output for paying `remaining` corresponds to a position strictly above `targetPos` (since `remaining > totalIn1Scaled` at `targetPos`). The difference is silently absorbed by the pool.

The existing guards are insufficient: the initial full-bin check (lines 566–578) only handles the case where `remaining` covers the entire bin; the two-pass refinement handles the common case; but the fallback is the only handler for the double-undershoot case, and it is incorrect.

## Impact Explanation

This is a **swap conservation failure** causing **direct loss of trader principal**. The trader submits an exact-input token1→token0 swap and pays the full `remaining` amount, but receives fewer token0 than the bin curve permits for that payment. The pool's token1 balance is inflated by the excess input, which is not owed to LPs or the protocol under the correct curve math. The magnitude of the loss scales with the degree of undershoot, which grows with bin non-linearity (large `lengthE6`, extreme price ratios, or high fee settings). This meets the Sherlock critical/high threshold for direct loss of user principal via a broken core swap path.

## Likelihood Explanation

The fallback at line 620 fires when: (1) the analytical solution undershoots (known to occur with non-linear bin curves), and (2) the proportional scaling at lines 598–618 also undershoots — specifically when `scaledTarget <= maxFinalBinPos` (the line 608–609 path is taken) but the recomputed `totalIn1Scaled` for `scaledTarget` is still below `remaining`. This is reachable through the normal public swap path (`_swapToken1ForToken0SpecifiedInput` → `buyToken0InBinSpecifiedIn`) with no privileged access required. [6](#0-5) 

Any trader submitting an exact-input token1→token0 swap against a bin with sufficient non-linearity can trigger it. No special role or setup is required beyond choosing swap parameters that exercise a non-linear bin.

## Recommendation

Replace the fallback at lines 620–622 with a clamp to `maxFinalBinPos` and a full recomputation of `out0Scaled`, mirroring the logic at lines 605–606:

```solidity
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    targetPos = maxFinalBinPos;
    out0Scaled = calculateOutputToken0FromBinPosition(binState.token0BalanceScaled, currBinPos, targetPos);
    finalPriceX64 = calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Ceil);
    avgPriceX64 = calculateArithmeticMean(startingPriceX64, finalPriceX64);
    in1WithoutFeeScaled = calculateRequiredToken(out0Scaled, avgPriceX64);
    totalIn1Scaled = grossInputWithBinFeeCeil(in1WithoutFeeScaled, onePlusBuyFeeX64);
}
```

After this, `totalIn1Scaled` for `maxFinalBinPos` is guaranteed to exceed `remaining` (by the initial check at lines 566–578), so the scale-down block at line 624 will correctly proportionally reduce both `out0Scaled` and `totalIn1Scaled`, ensuring the trader receives fair value.

## Proof of Concept

1. Deploy a pool with a bin configured for high non-linearity (large spread, extreme price ratio).
2. Submit an exact-input token1→token0 swap where `remaining` is chosen such that:
   - The full-bin check (lines 566–578) fails (consuming `maxFinalBinPos` requires more than `remaining`).
   - The analytical solution at lines 581–596 gives `totalIn1Scaled < remaining`.
   - The proportional scaling at lines 598–618 gives `scaledTarget <= maxFinalBinPos` but the recomputed `totalIn1Scaled` is still below `remaining`.
3. Observe that after the swap: `consumed == remaining` (full input taken) but `out0` corresponds to `targetPos < maxFinalBinPos` (undershot output).
4. Verify the conservation invariant: `out0 * avgPrice < consumed << 64`, confirming the trader was overcharged.

The Foundry property test sketch in the submission correctly captures this: when `consumed == remainingIn && finalPos < maxFinalBinPos`, the assertion `out0 * avgPrice >= consumed << 64` fails, demonstrating the conservation violation.
