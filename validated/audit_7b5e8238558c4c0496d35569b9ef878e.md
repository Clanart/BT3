### Title
Unauthenticated `SetProof` on `RemoteProofManagerServer` Allows Pre-Poisoning of Proof Cache, Permanently Bypassing Proof Verification for Client-Side-Proving Transactions - (File: `crates/apollo_proof_manager/src/communication.rs`)

---

### Summary

The `ProofManager`'s `handle_request` handler for `ProofManagerRequest::SetProof` stores any caller-supplied `(ProofFacts, Proof)` pair directly to persistent storage without verifying the proof. The `RemoteProofManagerServer` binds to `0.0.0.0` with no authentication. The `run_proof_verification` function in the transaction converter permanently skips cryptographic verification for any `proof_facts` whose hash is already present in the store. An attacker who pre-populates the store with an invalid proof for a chosen `proof_facts` hash causes every subsequent gateway or consensus submission carrying those `proof_facts` to bypass verification entirely, allowing an invalid client-side-proving transaction to be admitted to the mempool and sequenced.

---

### Impact Explanation

An attacker who can reach the `RemoteProofManagerServer` TCP port calls `SetProof(F, P_invalid)` for a chosen `proof_facts` value `F`. Because `ProofManager::set_proof` writes without verification, `P_invalid` is durably stored under `hash(F)`. From that point forward, any transaction submitted to the gateway or consensus layer that carries `proof_facts = F` will have `run_proof_verification` return `Ok(false)` (skipped), and the transaction will be admitted to the mempool and sequenced without any cryptographic proof check. This satisfies:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The stored invalid proof also permanently poisons the cache entry: `ProofManager::set_proof` will never overwrite an existing entry, so a legitimate proof for the same `proof_facts` hash can never displace the attacker's entry. [9](#0-8) 

---

### Likelihood Explanation

The `RemoteProofManagerServer` binds to `0.0.0.0` in both the `distributed/proof_manager.json` and `hybrid/replacer_core.json` deployment configs with no TLS, no token, and no IP allowlist in the server code. Any process that can open a TCP connection to the proof manager port — including an attacker on the same network segment or a misconfigured Kubernetes pod — can issue `SetProof` without credentials. The attack requires a single unauthenticated TCP request before the legitimate proof is stored.

---

### Recommendation

**Short term:**
- Add proof verification inside `ProofManager::handle_request` for `SetProof` before writing to storage, mirroring the check already present in `run_proof_verification`:
  ```rust
  ProofManagerRequest::SetProof(proof_facts, proof) => {
      starknet_proof_verifier::verify_proof(proof_facts.clone(), proof.clone())?;
      self.set_proof(proof_facts, proof).await ...
  }
  ```
- Alternatively, restrict `SetProof` to internal (local channel) callers only and remove it from the `RemoteProofManagerServer` dispatch table, since external callers have no legitimate reason to write proofs directly.

**Long term:**
- Add mutual TLS or a shared-secret header to `RemoteComponentServer` for all write-capable endpoints.
- Audit all other `RemoteComponentServer`-exposed write operations for similar missing authorization checks.

### Proof of Concept

1. Attacker connects to the `RemoteProofManagerServer` TCP port (bound to `0.0.0.0`).
2. Attacker serializes and sends `ProofManagerRequest::SetProof(target_proof_facts, garbage_bytes)`.
3. `ProofManager::set_proof` checks `contains_proof(target_proof_facts)` → `false`, writes `garbage_bytes` to `FsProofStorage` under `hash(target_proof_facts)`.
4. Attacker (or any user) submits an `InvokeV3` transaction to the gateway with `proof_facts = target_proof_facts` and any `proof` field.
5. `run_proof_verification` calls `contains_proof(target_proof_facts)` → `true`, returns `Ok(false)`.
6. Gateway proceeds past proof verification, calls `store_proof_and_spawn_archiving`, and forwards the transaction to the mempool.
7. Transaction is sequenced and executed without any valid proof having been verified. [10](#0-9) [11](#0-10) [12](#0-11)
