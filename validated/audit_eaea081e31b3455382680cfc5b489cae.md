### Title
Missing `state_diff_commitment` Hash Validation in P2P State Diff Sync Allows Malicious Peer to Store Wrong Committed State — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
`StateDiffStreamBuilder::parse_data_for_block` validates only that the sum of `StateDiffChunk::len()` values equals `target_state_diff_len` from the stored header. It never computes `calculate_state_diff_hash` on the assembled `ThinStateDiff` and never compares it against the header's `state_diff_commitment`. A malicious P2P peer can therefore deliver chunks whose lengths sum correctly but whose content is entirely attacker-chosen, and the node will store that wrong state diff unconditionally.

### Finding Description
The full data flow is:

```
peer → StateDiffChunk stream
  → parse_data_for_block          [validates: length only]
  → unite_state_diffs             [validates: no duplicate keys]
  → validate_deprecated_declared_classes_non_conflicting
  → write_to_storage
      → append_state_diff(block_number, thin_state_diff)   [no hash check]
``` [1](#0-0) 

The function reads `state_diff_length` from the stored header: [2](#0-1) 

After the loop, only length equality is checked; `state_diff_commitment` from the header is never read or compared. [3](#0-2) 

`write_to_storage` then persists the attacker-supplied diff directly: [4](#0-3) 

The header sync path also stores headers without block-hash verification (there is an explicit TODO for parent-hash checking): [5](#0-4) 

This means the attacker controls both the header (setting any `state_diff_commitment` and `state_diff_length = N`) and the N-length state diff chunks (with arbitrary content). The `state_diff_commitment` field in the stored header is never used as a guard anywhere in the P2P sync client path.

The committer's `verify_state_diff_hash` guard exists only in the sequencer/proposer path: [6](#0-5) 

It is never invoked by the P2P sync client.

`append_state_diff` writes the corrupted diff into all indexed tables — `deployed_contracts`, `nonces`, `contract_storage`, `compiled_class_hash`, `deprecated_declared_classes_block`: [7](#0-6) 

### Impact Explanation
Any node running the P2P sync client (full node or sequencer catching up) that connects to a malicious peer will store attacker-chosen storage values, nonces, and class hashes for any block. This corrupts the Patricia trie state used for RPC queries, proof generation, and — if the node is a sequencer — the global root fed into subsequent block hashes. The stored `ThinStateDiff` will not match the `state_diff_commitment` in the header, violating the invariant the system is designed to enforce.

### Likelihood Explanation
Any peer that can establish a P2P connection to the target node can exploit this. No special privilege is required. The attacker only needs to serve a header with a chosen `state_diff_length = N` and then send N length-units of crafted `StateDiffChunk`s. The existing conflict checks (`ConflictingStateDiffParts`) do not prevent this because the attacker sends non-conflicting but wrong values.

### Recommendation
After assembling the full `ThinStateDiff`, compute `calculate_state_diff_hash(&result)` and compare it against the `state_diff_commitment` stored in the header. Return `ParseDataError::BadPeer` if they differ. The function and hash are already available in the codebase: [8](#0-7) 

### Proof of Concept
1. Stand up a malicious P2P peer.
2. When the target node queries state diffs for block B, respond with a `SignedBlockHeader` where `state_diff_length = 1` and `state_diff_commitment = <any value>`.
3. Respond to the state diff query with a single `StateDiffChunk::ContractDiff` containing one attacker-chosen storage key/value (length = 1).
4. After sync, read the stored state diff from the node's storage and assert that `calculate_state_diff_hash(&stored_diff) != header.state_diff_commitment`. The assertion will hold, confirming the invariant is broken and the wrong value is stored.
