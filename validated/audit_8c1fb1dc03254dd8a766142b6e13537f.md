### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Bridge Collateralization Drain ŌĆö (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without verifying the actual token balance received by the contract. For fee-on-transfer ERC20 tokens the bridge receives `amount ŌłÆ fee_taken`, but the event (and therefore the NEAR-side proof) records the full `amount`. The NEAR bridge credits the full `amount` to the user, creating unbacked supply. When the user bridges back, the EVM bridge releases the full credited amount, draining tokens that were deposited by other honest users.

---

### Finding Description

In `initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-controlled
);
``` [1](#0-0) 

No balance snapshot is taken before or after the transfer. Immediately after, the unmodified `amount` parameter is emitted:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ŌåÉ user-supplied, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The NEAR bridge's `fin_transfer_callback` decodes the proof of this event and uses `init_transfer.amount` verbatim to construct the `TransferMessage` that is credited to the recipient:

```rust
let transfer_message = TransferMessage {
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

There is no token whitelist and no restriction on which ERC20 can be passed as `tokenAddress`. [4](#0-3) 

When the user later bridges back from NEAR, `finTransfer` on EVM releases the full credited amount via `safeTransfer`:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [5](#0-4) 

Because the bridge holds fewer tokens than the NEAR-side supply represents, this transfer is satisfied from tokens deposited by other users.

---

### Impact Explanation

**High ŌĆö Bridge collateralization is broken; direct theft of other users' locked tokens.**

A user who bridges a fee-on-transfer token receives more NEAR-side tokens than were actually locked on EVM. When they redeem, the shortfall is covered by tokens deposited by other honest users. Those users subsequently cannot withdraw their full balance; their `safeTransfer` either reverts (permanent lock) or succeeds by consuming a third party's deposit (direct theft). Both outcomes match the allowed impact categories:

- *Direct theft of bridged assets* (Critical)
- *Permanent freezing / irrecoverable lock of user funds* (Critical)
- *Balance / accounting corruption that breaks bridge collateralization* (High)

---

### Likelihood Explanation

**Medium.** Fee-on-transfer tokens are a well-known ERC20 variant present on mainnet (e.g., STA, PAXG in certain configurations, tokens with configurable transfer taxes). There is no token whitelist in `initTransfer`. Any unprivileged user can call the function with any ERC20 address. The attacker needs only to hold a fee-on-transfer token and initiate a bridge transfer; no privileged access, leaked key, or colluding party is required.

---

### Recommendation

Record the contract's token balance before and after `safeTransferFrom` and use the delta as the canonical bridged amount in the emitted event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived in the event and downstream accounting
```

This ensures the NEAR-side credit never exceeds the EVM-side lock.

---

### Proof of Concept

1. A fee-on-transfer token `FOT` charges a 1 % transfer fee.
2. **Bob** (honest) calls `initTransfer(FOT, 100, 0, 0, ...)`. Bridge receives 99 FOT. Event records `amount = 100`. NEAR credits Bob 100 FOT.
3. **Alice** (attacker) calls `initTransfer(FOT, 100, 0, 0, ...)`. Bridge receives 99 FOT. Event records `amount = 100`. NEAR credits Alice 100 FOT.
4. Bridge EVM balance: 198 FOT. NEAR-side supply: 200 FOT. Shortfall: 2 FOT.
5. Alice bridges back 100 FOT from NEAR. `finTransfer` releases 100 FOT to Alice. Bridge balance: 98 FOT.
6. Bob bridges back 100 FOT from NEAR. `finTransfer` attempts to release 100 FOT but bridge only holds 98 FOT ŌĆö Bob's withdrawal reverts (permanent lock) or, if other deposits exist, drains a third party's tokens (direct theft).

Alice has recovered 100 FOT having only locked 99, at the expense of Bob's 2 FOT shortfall.
