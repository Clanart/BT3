### Title
Missing Declared-Class Existence Check in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the supplied `class_hash` corresponds to a previously declared class before committing it to contract state. This is the direct Cairo analog of the `StaticHyVM` bug: just as that contract called `delegatecall` to a potentially non-existent target without checking for code, the OS here writes an arbitrary, potentially undeclared class hash into a contract's `StateEntry` without checking for class existence. Any contract can trigger this via an unprivileged `replace_class` syscall, permanently freezing itself and any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts the caller-supplied `class_hash` from the syscall request and immediately writes it into `contract_state_changes` with no validation that the hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}

Once the undeclared `class_hash` is committed to state, any subsequent transaction that calls the affected contract reaches `execute_entry_point`, which performs:

```cairo

...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,

``` [3](#0-2) 

For an undeclared class hash, `dict_read` returns `0` (the uninitialized default). `find_element` then searches for a `CompiledClassFact` with `hash == 0`, which does not exist. `find_element` is not a soft search — it asserts the element is present and panics if it is not. This hard failure propagates up, making the contract permanently uncallable at the OS level.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract that holds ERC-20 tokens, ETH, or any other asset and calls `replace_class` with an undeclared hash becomes permanently uncallable. No `transfer`, `withdraw`, or any other entry point can ever be executed again, because every call attempt causes the OS to hard-fail at `find_element`. The funds locked in that contract's storage are irrecoverably frozen. This matches the highest-severity allowed impact.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, permissionless operation available to any deployed contract. No privileged role is required. A malicious contract author can deliberately call `replace_class` with a fabricated hash to self-destruct their contract after receiving funds from victims (e.g., in a rug-pull scenario). Additionally, a buggy contract could accidentally supply a wrong hash. The missing check is confirmed by the in-code `TODO` comment, meaning the developers themselves identified this gap.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (the class declaration dict). Concretely, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the result is non-zero (i.e., a compiled class hash has been registered for it). This mirrors the check that `execute_entry_point` itself relies on downstream, and closes the gap identified in the `TODO` comment.

---

### Proof of Concept

1. Attacker deploys a contract `VaultContract` that accepts deposits and stores balances.
2. Users deposit funds into `VaultContract`.
3. Attacker calls `VaultContract` with a transaction that invokes the `replace_class` syscall, supplying `class_hash = 0xdeadbeef` (a hash that has never been declared on-chain).
4. The OS `execute_replace_class` handler accepts this without validation and commits `StateEntry(class_hash=0xdeadbeef, ...)` to `contract_state_changes`.
5. The state is finalized; `VaultContract`'s on-chain class hash is now `0xdeadbeef`.
6. Any user who subsequently calls `VaultContract` triggers `execute_entry_point`, which calls `dict_read(key=0xdeadbeef)` on `contract_class_changes`, receiving `compiled_class_hash = 0`.
7. `find_element` searches for `CompiledClassFact` with `hash == 0` — none exists — and the Cairo VM panics.
8. All funds in `VaultContract` are permanently frozen. [4](#0-3) [5](#0-4)
