### Title
Missing Destination Chain Validation in `init_transfer` Allows Permanent Token Lock - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR bridge contract's `init_transfer` function accepts any valid `ChainKind` as a destination (except `Near`) without verifying that the destination chain has a registered factory, a registered prover, or that the transferred token has a mapping on that chain. Tokens are burned or locked before any such check is possible, and there is no cancel/refund path for pending transfers. A user who specifies a destination chain for which the token is not registered causes their funds to be permanently frozen.

---

### Finding Description

`init_transfer` in `near/omni-bridge/src/lib.rs` is the NEAR-side entry point for outbound transfers. It is called from `ft_on_transfer` when a user sends tokens to the bridge contract via `ft_transfer_call`. The only destination-chain guard present is:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

No check is made that:
- `self.factories` contains an entry for the destination `ChainKind`
- `self.provers` contains an entry for the destination `ChainKind`
- `self.token_id_to_address` contains a mapping `(destination_chain, token_id)` for the token being transferred

After passing this single guard, `init_transfer_internal` is called, which immediately burns bridge tokens or locks native tokens and stores the transfer in `pending_transfers`:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [2](#0-1) 

The only way to remove a transfer from `pending_transfers` is through `sign_transfer` (which calls `remove_transfer_message` on success) or `claim_fee`. `sign_transfer` calls `get_token_address(destination_chain, token_id)`, which returns `None` if the token is not registered for that chain, causing an immediate panic:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [3](#0-2) 

There is no `cancel_transfer` or user-accessible refund path. The transfer remains in `pending_transfers` indefinitely while the tokens are already gone.

The `ChainKind` enum defines 13 variants: [4](#0-3) 

Not all tokens are registered on all chains. For example, a token registered only on `Eth` and `Sol` has no `token_id_to_address` entry for `ChainKind::Fogo`, `ChainKind::Strk`, `ChainKind::HyperEvm`, etc. A user who specifies any such chain as the recipient's chain will lose their tokens permanently.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

- For bridge-deployed tokens: `burn_tokens_if_needed` destroys the tokens on-chain. They cannot be re-minted without a valid `fin_transfer` proof from the destination chain, which will never exist.
- For native (locked) tokens: `lock_tokens_if_needed` increments the locked balance. The tokens remain locked in the bridge contract with no mechanism to release them.

In both cases the transfer record is stuck in `pending_transfers` forever, and the user has no recourse.

---

### Likelihood Explanation

**High.** The bridge supports 13 `ChainKind` variants. Token registration is per-chain and per-token; new chains are added over time. Any user who mistakenly (or deliberately) specifies a destination chain for which their token has no registered address triggers the freeze. The `OmniAddress` type accepts all 13 chain prefixes at the JSON/string level, so no off-chain tooling prevents the call from reaching the contract. [5](#0-4) 

---

### Recommendation

Before burning or locking tokens, `init_transfer` should verify that the destination chain is operational for the given token. At minimum, add a check that `self.token_id_to_address` contains an entry for `(destination_chain, token_id)`:

```rust
require!(
    self.get_token_address(
        init_transfer_msg.get_destination_chain(),
        token_id.clone(),
    ).is_some(),
    BridgeError::TokenNotFound.as_ref()
);
```

Optionally also verify that `self.factories` and `self.provers` have entries for the destination chain. This mirrors the factory whitelist check already present in `fin_transfer_callback` and `deploy_token_callback`: [6](#0-5) 

---

### Proof of Concept

1. Suppose token `usdc.near` is registered only for `ChainKind::Eth` (i.e., `token_id_to_address[(Eth, "usdc.near")] = Eth(0xA0b...)` exists, but no entry exists for `ChainKind::Fogo`).

2. Attacker (or mistaken user) calls:
   ```json
   ft_transfer_call(
     receiver_id: "omni-bridge.near",
     amount: "1000000",
     msg: '{"InitTransfer":{"recipient":"fogo:BXss9YNCX2p6VPf2Em54pHXkXnC2FPBeZgbB9fY1cuBR","fee":"0","native_token_fee":"0"}}'
   )
   ```

3. `ft_on_transfer` → `init_transfer` passes the only guard (`recipient.get_chain() != Near`).

4. `init_transfer_internal` is called:
   - `burn_tokens_if_needed("usdc.near", 1000000)` — tokens destroyed if bridge token.
   - `lock_tokens_if_needed(ChainKind::Fogo, "usdc.near", 1000000)` — tokens locked if native.
   - Transfer stored in `pending_transfers`.

5. Any subsequent call to `sign_transfer` for this transfer panics with `ERR_FAILED_TO_GET_TOKEN_ADDRESS` because `token_id_to_address[(Fogo, "usdc.near")]` does not exist.

6. No cancel path exists. Tokens are permanently lost. [7](#0-6) [8](#0-7)
