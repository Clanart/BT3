### Title
`get_class(pending, …)` bypasses the `read_pending_data` staleness guard, returning a stale class body from a superseded pending epoch — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`starknet_getClass` with `block_id = pending` reads directly from the `pending_classes` `RwLock` without first calling `read_pending_data`. Every other pending-aware RPC handler calls `read_pending_data`, which rejects stale pending data by comparing `pending_data.block.parent_block_hash()` against `latest_header.block_hash`. `get_class` skips this check entirely, so during the window between a new block being finalized and the sync task clearing `pending_classes`, any class that was declared in the superseded pending epoch is returned verbatim to an unprivileged caller.

---

### Finding Description

**The staleness guard in `read_pending_data`:**

`read_pending_data` returns a synthetic empty `PendingData` whenever the stored pending block's parent hash does not match the latest finalized block hash: [1](#0-0) 

**`get_class` bypasses this guard entirely:**

When `block_id` is `Tag::Pending`, `get_class` reads `self.pending_classes` directly and returns immediately if the class is found — `read_pending_data` is never called: [2](#0-1) 

**When does `pending_classes` become stale?**

In `pending_sync.rs`, when the sync task detects that a new block has been finalized (`new_pending_parent_hash != latest_block_hash`), it returns `PendingSyncFinished` immediately — it does **not** clear `pending_classes` or update `pending_data` at that point: [3](#0-2) 

`pending_classes` is only cleared in the **next** sync cycle, and only when the pending data itself changes parent hash: [4](#0-3) 

**The race window:**

```
1. Block N finalized → latest_header.block_hash = hash(N)
2. pending_data.parent_block_hash = hash(N-1)   [stale, not yet updated]
3. pending_classes contains classes declared in the pending block on top of N-1
4. Sync task detects new block → returns PendingSyncFinished
   (pending_classes NOT cleared yet)
5. ── race window ──
   RPC: get_class(pending, class_hash)
        reads pending_classes → finds stale class → returns it
6. New sync cycle eventually clears pending_classes
```

During step 5, `read_pending_data` would have returned a synthetic empty `PendingData` (because `parent_block_hash != latest_header.block_hash`), correctly signalling that the pending epoch is superseded. But `get_class` never calls it.

---

### Impact Explanation

An unprivileged RPC caller receives an authoritative-looking class body for `block_id = pending` that belongs to a superseded pending epoch. The class may not be declared in the current pending state diff at all. This is a concrete wrong value (the exact class body from the previous epoch) served under the `pending` tag.

Impact: **High** — "RPC pending view returns an authoritative-looking wrong value."

---

### Likelihood Explanation

The race window exists on every block finalization event and persists until the next pending-sync cycle completes. The window duration is bounded by the pending-sync polling interval (`sleep_duration`). Any caller who queries `starknet_getClass(pending, class_hash)` for a class that was declared in the previous pending epoch but not in the current one will receive the stale body during this window.

---

### Recommendation

Before reading `self.pending_classes` in `get_class`, call `read_pending_data` (or an equivalent staleness check) and skip the `pending_classes` lookup if the pending data is stale. Concretely, mirror the pattern used by `get_nonce` and `maybe_get_class_hash_at`:

```rust
async fn get_class(&self, block_id: BlockId, class_hash: ClassHash) -> RpcResult<GatewayContractClass> {
    let block_id = if let BlockId::Tag(Tag::Pending) = block_id {
        let txn = self.storage_reader.begin_ro_txn().map_err(internal_server_error)?;
        // Staleness check: only consult pending_classes if pending data is current.
        let pending_data = read_pending_data(&self.pending_data, &txn).await?;
        if !pending_data.state_update.state_diff.declared_classes.is_empty()
            || !pending_data.state_update.state_diff.old_declared_contracts.is_empty()
        {
            let maybe_class = self.pending_classes.read().await.get_class(class_hash);
            if let Some(class) = maybe_class {
                return class.try_into().map_err(internal_server_error);
            }
        }
        BlockId::Tag(Tag::Latest)
    } else {
        block_id
    };
    // ... storage fallback unchanged
}
```

Alternatively, `pending_classes` could be cleared atomically with `pending_data` when `PendingSyncFinished` is returned, so the shared state is never inconsistent from the RPC layer's perspective.
