### Title
`ProofOfInclusion::valid()` Does Not Verify Root Against a Trusted Committed Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

The DataLayer `ProofOfInclusion::valid()` method only verifies internal hash-chain consistency within the proof itself. It never compares the computed root against any externally-committed, trusted tree root. An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key/value is in the tree), compute internally-consistent `combined_hash` values for each layer, and `valid()` will return `true` — even though the proof's root does not match the actual committed DataLayer tree root.

### Finding Description

`ProofOfInclusion::valid()` is implemented as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The final check `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash` — a tautology. Because the loop already enforces `calculated_hash == layer.combined_hash` and then sets `existing_hash = calculated_hash`, by the time the loop ends, `existing_hash` is always equal to `last.combined_hash`, which is exactly what `root_hash()` returns. The final equality check is therefore always `true` when the loop completes without returning `false`.

**The result:** `valid()` is a purely self-referential check. It verifies only that the proof's own hash chain is internally consistent — it never compares the computed root against any externally-committed, trusted tree root.

Contrast this with the consensus-layer `validate_merkle_proof` in `merkle_tree.rs`, which correctly performs the missing check:

```rust
pub fn validate_merkle_proof(proof: &[u8], item: &[u8; 32], root: &[u8; 32]) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root {
        return Err(SetError);
    }
    Ok(tree.generate_proof(item)?.0)
}
``` [3](#0-2) 

The DataLayer `ProofOfInclusion::valid()` is missing the equivalent `proof.root_hash() == trusted_committed_root` check entirely.

`ProofOfInclusion` is a `Streamable` type exposed to Python via the `py-bindings` feature, meaning it can be deserialized from untrusted network bytes and its `valid()` method called directly by Python consumers: [4](#0-3) [5](#0-4) 

### Impact Explanation

An attacker who can supply a serialized `ProofOfInclusion` to any code path that calls `proof.valid()` as its sole acceptance check can forge inclusion of any arbitrary `node_hash` (key/value pair) in the DataLayer tree. The attacker constructs a proof with:
- `node_hash` = any hash they wish to claim is included
- One or more `ProofOfInclusionLayer` entries with `other_hash` chosen freely and `combined_hash` computed as `calculate_internal_hash(node_hash, side, other_hash)`

This proof passes `valid()` unconditionally, allowing the attacker to prove invalid state — that a key/value is present in the DataLayer tree when it is not. This matches the allowed High impact: **DataLayer Merkle proof logic accepts forged inclusion, letting untrusted input prove invalid state.**

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` and exposed to Python. Any Python consumer that deserializes a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` without separately comparing `proof.root_hash()` to a known-good committed root is vulnerable. The method name `valid()` strongly implies it performs a complete validity check, making it likely that callers rely on it exclusively. The attacker-controlled entry path is: craft a `ProofOfInclusion` bytes, send to a DataLayer node, node deserializes and calls `valid()`.

### Recommendation

Add a trusted root parameter to `valid()` (or a separate `valid_against_root(trusted_root: &Hash) -> bool` method) that compares the computed root against the externally-committed tree root:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    // Compare against the externally-committed root, not the proof's own field
    &existing_hash == trusted_root
}
```

Deprecate or remove the current `valid()` method, or make it clearly document that it does not verify against any committed root and must not be used as a sole acceptance check.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from chia_rs import calculate_internal_hash  # or equivalent

# Attacker wants to forge inclusion of arbitrary_node_hash
arbitrary_node_hash = bytes([0xAA] * 32)
other_hash = bytes([0xBB] * 32)
side = 0  # Left

# Compute a combined_hash that makes the layer internally consistent
combined_hash = calculate_internal_hash(arbitrary_node_hash, side, other_hash)

layer = ProofOfInclusionLayer(
    other_hash_side=side,
    other_hash=other_hash,
    combined_hash=combined_hash,
)

forged_proof = ProofOfInclusion(node_hash=arbitrary_node_hash, layers=[layer])

# valid() returns True even though arbitrary_node_hash is not in any real tree
assert forged_proof.valid() == True
# root_hash() returns the attacker-controlled combined_hash, not the real tree root
assert forged_proof.root_hash() == combined_hash
```

The proof passes `valid()` with a root that does not correspond to any real committed DataLayer tree, allowing forged state to be accepted by any caller that uses `valid()` as its sole check.
