### Title
Rounding-to-Zero in `normalize_amount` Permanently Locks/Burns User Tokens During `init_transfer` - (`near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` locks or burns user tokens before any decimal-normalization check is performed. When a user bridges a token whose NEAR-side decimals exceed the destination-chain decimals (e.g., 24 → 6, diff = 18), any transfer where `amount_without_fee < 10^diff_decimals` normalizes to zero. The downstream `sign_transfer` guard rejects the zero amount, but by then the tokens are already irrecoverably locked or burned, with no cancel path available to the user.

---

### Finding Description

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

When `amount < 10^diff_decimals`, the result is `0`.

`sign_transfer` does guard against this:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

However, this guard fires **after** `init_transfer_internal` has already locked or burned the tokens:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [3](#0-2) 

`init_transfer` itself only checks `fee < amount`; it never looks up the token's decimal configuration and never verifies that `normalize_amount(amount - fee, decimals) > 0`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [4](#0-3) 

Because `sign_transfer` is restricted to trusted relayers (`#[trusted_relayer]`), the user cannot bypass the zero-amount guard themselves. No public cancel or refund function exists for pending transfers, so the transfer record sits in `pending_transfers` indefinitely while the underlying tokens are gone. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

For deployed tokens (tokens that originated on another chain and were minted on NEAR), `burn_tokens_if_needed` destroys the tokens at `init_transfer_internal` time. The supply is reduced and the tokens cannot be recovered even by the DAO. For native NEAR tokens, `lock_tokens_if_needed` locks them in the bridge; while a DAO intervention via `transfer_token_as_dao` is theoretically possible, no automated or user-accessible recovery path exists.

---

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user. For a token with `origin_decimals = 24` (NEAR) and `decimals = 6` (EVM destination), `diff = 18`, so any `amount_without_fee < 10^18` (i.e., less than 1 full NEAR-side unit) triggers the bug. A user who sets a fee close to their transfer amount, or who simply sends a dust amount, will silently lose their tokens. No special privilege or coordination is required.

---

### Recommendation

Add a decimal-normalization check inside `init_transfer` (before tokens are locked or burned), using the already-registered `token_decimals` map:

```rust
// After resolving token_address and decimals:
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing guard in `sign_transfer` but places it at the entry point, before any irreversible state change.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 6` (diff = 18).
2. User calls `ft_transfer_call` on the NEAR token contract, transferring `amount = 10^17` (0.1 NEAR-side units) with `fee = 0`.
3. `init_transfer` passes the `fee < amount` check.
4. `init_transfer_internal` calls `burn_tokens_if_needed` — tokens are burned.
5. Transfer is stored in `pending_transfers`.
6. Relayer calls `sign_transfer`; `normalize_amount(10^17, {24, 6}) = 10^17 / 10^18 = 0`.
7. `require!(amount_to_transfer > 0, ...)` panics — transfer can never be signed.
8. No cancel function exists; user's `10^17` tokens are permanently destroyed. [1](#0-0) [6](#0-5) [2](#0-1)
