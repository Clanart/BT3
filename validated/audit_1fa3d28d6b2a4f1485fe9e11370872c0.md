### Title
Nonce-Free `meta_tx_v0` Hash Enables Unbounded Signature Replay — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`, `transaction_hash/transaction_hash.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a version-0 transaction context with a hardcoded `nonce=0` and computes a hash that contains **no nonce field**. Because `check_and_increment_nonce` explicitly skips nonce enforcement for `version == 0`, any meta-tx-v0 signature is permanently valid and can be replayed by any unprivileged caller across any future block, leading to direct loss of funds.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` builds a synthetic `TxInfo` with `version=0` and `nonce=0`: [1](#0-0) 

The hash for this synthetic transaction is computed by `compute_meta_tx_v0_hash`, which calls `deprecated_get_transaction_hash` with `additional_data_size=0` — meaning **no nonce is committed to in the hash**: [2](#0-1) 

The hash covers only: `(INVOKE_HASH_PREFIX, version=0, contract_address, selector, calldata, max_fee=0, chain_id)`. There is no block number, no nonce, no timestamp.

`check_and_increment_nonce` then explicitly skips all nonce enforcement for version-0 transactions: [3](#0-2) 

Similarly, `run_validate` skips the `__validate__` entry point for version-0: [4](#0-3) 

The result: a meta-tx-v0 signature is a **permanent, unconditional authorization** to call `__execute__` on the target contract with the given calldata. Once the signature appears on-chain (in any block), it can be replayed by any party in any subsequent block with zero additional authorization.

---

### Impact Explanation

**Critical — Direct loss of funds.**

If a victim's account contract has ever been the target of a `meta_tx_v0` call (e.g., to execute a token transfer, approve a spender, or bridge funds), the signature for that call is permanently replayable. An attacker who observes the signature on-chain can:

1. Deploy or reuse a contract that issues the `meta_tx_v0` syscall.
2. Pass the victim's previously-seen `(contract_address, selector, calldata, signature)` to the syscall.
3. The OS executes `__execute__` on the victim's contract with the original calldata — again, and again, indefinitely.

This can drain the victim's token balances, re-execute approvals, or re-trigger bridge operations, all without the victim's consent and without any on-chain state that would block the replay.

---

### Likelihood Explanation

**High.** Every `meta_tx_v0` invocation that reaches the chain permanently exposes the victim's signature. The attacker needs only:
- Read the signature from a past block (fully public).
- Submit a new outer transaction calling `meta_tx_v0` with that signature.

No privileged access, leaked key, or social engineering is required. The attack is permissionless and deterministic.

---

### Recommendation

Include a nonce in the `compute_meta_tx_v0_hash` computation. Specifically:

1. Add a `nonce` parameter to `compute_meta_tx_v0_hash` and pass it as part of `additional_data` to `deprecated_get_transaction_hash` (set `additional_data_size=1`, `additional_data=[nonce]`).
2. In `execute_meta_tx_v0`, read the current nonce from `contract_state_changes` for `contract_address`, include it in the hash, and increment it after successful execution — analogous to how `check_and_increment_nonce` works for version ≥ 1 transactions.
3. Alternatively, bind the meta-tx-v0 signature to the outer transaction's hash (include `old_tx_info.transaction_hash` in the meta-tx hash), so each meta-tx-v0 invocation is unique to its enclosing transaction.

---

### Proof of Concept

**Setup:**
- Victim's account contract at address `V` previously received a `meta_tx_v0` call with calldata `C` (e.g., `transfer(attacker, 1000_STRK)`). The signature `S` is visible on-chain.

**Attack:**
1. Attacker deploys contract `A` with an `__execute__` function that issues:
   ```
   meta_tx_v0(contract_address=V, selector=EXECUTE_SELECTOR, calldata=C, signature=S)
   ```
2. Attacker submits an invoke transaction calling `A.__execute__`.
3. The OS calls `execute_meta_tx_v0`:
   - Computes `hash = H(INVOKE_PREFIX, 0, V, EXECUTE_SELECTOR, C, 0, chain_id)` — identical to the original hash.
   - Sets `nonce=0`, skips `check_and_increment_nonce` (version 0).
   - Calls `V.__execute__(C)` with signature `S` — which the victim's account validates successfully (same hash, same signature).
4. The transfer executes again. Repeat indefinitely.

The root cause is the absence of any nonce or binding context in the meta-tx-v0 hash: [5](#0-4)
