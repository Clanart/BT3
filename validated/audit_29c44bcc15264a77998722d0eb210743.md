### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts any arbitrary `class_hash` value from the syscall request without verifying that the hash corresponds to a declared contract class. This is explicitly acknowledged by a TODO comment in the code. As a result, any contract can permanently replace its own class hash with `0` or any undeclared hash, rendering the contract permanently unexecutable and freezing all funds held within it. The OS will commit this invalid state transition into the proven block output.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the OS reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment explicitly acknowledges the missing check: `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.`

No `assert_not_zero` or membership check against `contract_class_changes` is performed. The OS will accept `class_hash = 0` or any arbitrary undeclared felt value and commit it to the proven state.

Compare this to `execute_deploy` in `syscall_impls.cairo` (line 452+) and `deploy_contract.cairo` (line 53), where the OS enforces `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` before deployment — a structural validation that is entirely absent from `replace_class`.

Also compare to `execute_declare_transaction` in `transaction_impls.cairo` (line 816), which enforces `assert_not_zero(compiled_class_hash)` before writing a class declaration. No equivalent guard exists for `replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to `0` or an undeclared hash via `replace_class`, the OS has no valid compiled class to dispatch execution to. Every subsequent call to that contract — including calls to withdraw or transfer funds — will fail at the class lookup stage. The state transition is committed to the proven block output and cannot be reversed. Any ERC-20 tokens, ETH, STRK, or other assets held in the contract's storage are permanently inaccessible.

This is not a revert: the state change is finalized on-chain as part of the proven OS output, making the fund freeze irreversible at the protocol level.

---

### Likelihood Explanation

**High.** The attack path requires only:
1. Deploying a contract (permissionless on StarkNet).
2. Calling any function in that contract that invokes the `replace_class` syscall with `class_hash = 0`.

No privileged role, leaked key, or operator cooperation is required. The sequencer will execute the transaction, the OS will process the `replace_class` syscall without rejecting it, and the resulting state (class hash = 0) will be committed to the proven block. Any unprivileged user can trigger this against any contract they control or any contract that exposes an externally callable path to `replace_class`.

A realistic scenario: a multisig wallet, DAO treasury, or DeFi vault contract that includes an upgrade mechanism (calling `replace_class`) can be permanently bricked if an attacker can manipulate the upgrade path — or if the contract owner themselves calls `replace_class(0)` by mistake — with no protocol-level safeguard to prevent it.

---

### Recommendation

Add a validation check in `execute_replace_class` to ensure the provided `class_hash` is both non-zero and corresponds to a previously declared class. Specifically:

1. **Non-zero check**: Add `assert_not_zero(class_hash)` before writing the new `StateEntry`.
2. **Declared class check**: Verify that `class_hash` exists in `contract_class_changes` (current block declarations) or in the global class state (prior declared classes), analogous to how `execute_declare_transaction` enforces `assert_not_zero(compiled_class_hash)` before committing a class declaration.

---

### Proof of Concept

1. Deploy a contract `VaultContract` that holds 1000 STRK and exposes an `upgrade(new_class_hash: felt)` function that calls `replace_class(new_class_hash)`.
2. Submit an invoke transaction calling `VaultContract.upgrade(0)`.
3. The sequencer executes the transaction. `execute_replace_class` is called with `class_hash = 0`. No validation occurs.
4. `dict_update` writes `StateEntry(class_hash=0, ...)` for `VaultContract`'s address into `contract_state_changes`.
5. The OS commits this state to the proven block output. The on-chain state now records `VaultContract`'s class hash as `0`.
6. Any subsequent call to `VaultContract` (e.g., to withdraw the 1000 STRK) fails: the OS reads `class_hash = 0`, finds no compiled class, and the execution aborts.
7. The 1000 STRK are permanently frozen.
