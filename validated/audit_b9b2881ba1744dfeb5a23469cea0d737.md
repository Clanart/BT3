Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` is an unrestricted `external` function that unconditionally transfers the router's entire ETH balance to `msg.sender`. When a user sends `msg.value > amountIn` on a WETH-input swap, `pay()` wraps only the required amount and leaves the remainder as raw ETH on the contract. Because `multicall` performs no automatic refund, any user who omits `refundETH()` from their batch loses the excess ETH to the next caller of `refundETH()`.

## Finding Description

`refundETH()` carries zero access control: [1](#0-0) 

It sends `address(this).balance` to `msg.sender` unconditionally. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable entry points: [2](#0-1) 

When `token == WETH` and `nativeBalance >= value`, `pay()` wraps exactly `value` and leaves `nativeBalance - value` as raw ETH on the contract: [3](#0-2) 

`multicall` performs no automatic ETH refund after executing the batch: [4](#0-3) 

If the user omits `refundETH()` from their multicall, the leftover ETH sits on the router until any external caller claims it via `refundETH()`.

## Impact Explanation

Direct, unconditional loss of user ETH principal. The attacker requires no special role, no pool setup, and no token approval — only a single public call to `refundETH()`. The stolen amount equals `msg.value − amountIn` for every victim transaction that omits the refund step. This meets the "Critical/High direct loss of user principal" threshold.

## Likelihood Explanation

`refundETH()` is a well-known optional cleanup step; users and integrators routinely omit it, especially in single-function calls rather than multicalls. The attack is a simple back-run: watch the mempool for `exactInputSingle` / `exactInput` calls with `msg.value > amountIn` that do not include `refundETH`, then submit `refundETH()` immediately after. No capital, no permissions, and no setup are required. The attack is repeatable for every such victim transaction.

## Recommendation

Either (1) restrict `refundETH()` so it can only be called from within an active `multicall` context (e.g., via a `msg.sender == address(this)` guard enforced by `delegatecall`), (2) automatically refund excess ETH at the end of every payable entry point, or (3) record per-caller deposited amounts in transient storage and only refund the recorded depositor.

## Proof of Concept

```
1. User calls:
   router.multicall{value: 1 ether}([
       abi.encodeCall(router.exactInputSingle, (
           ExactInputSingleParams({
               tokenIn: WETH,
               amountIn: 0.5 ether,
               ...
           })
       ))
       // refundETH() intentionally omitted
   ]);

2. pay() wraps 0.5 ETH → WETH → pool.
   router.balance == 0.5 ETH.

3. Attacker (separate tx):
   router.refundETH();
   // _transferETH(attacker, 0.5 ether) executes.
   // attacker.balance += 0.5 ETH; user loses 0.5 ETH.

assert(attacker.balance_after − attacker.balance_before == 0.5 ether); // passes
```
