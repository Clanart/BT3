### Title
`batch_insert` Bypasses Duplicate Key/Hash Validation for Bulk Entries, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` applies duplicate-key and duplicate-hash guards only to the last two items in a batch (via the guarded `insert()` path), while all remaining items are written directly to the blob via `insert_entry_to_blob()` without any uniqueness check. An untrusted caller supplying a crafted batch with duplicate keys or hashes causes the `BlockStatusCache` to silently overwrite its index mappings, leaving orphaned leaf nodes in the blob. The resulting tree has two leaves sharing the same key, the root hash is computed over both, and every subsequent proof of inclusion is invalid — corrupting the DataLayer Merkle state.

---

### Finding Description

`MerkleBlob::insert()` enforces two invariants before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` has two distinct code paths depending on the current leaf count:

**Path A (leaf_count ≤ 1):** The last two items are popped from the vector and inserted via `self.insert()`, which runs the duplicate checks above.

**Path B (remaining items):** All other items (positions 0..n-3 in the original vector) are written directly via `insert_entry_to_blob()`, which calls `add_leaf()` on the cache without any duplicate check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `add_leaf`, which uses `HashMap::insert` — silently overwriting any existing cache entry for the same key or hash:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index); // silent overwrite
}
``` [3](#0-2) 

When a duplicate key is inserted via Path B:
1. The cache's `key_to_index` is overwritten to point to the new leaf's index.
2. The original leaf remains in the blob at its old index, still connected to its parent in the tree.
3. The tree now contains two leaves with the same key; the root hash is computed over both.
4. `check_integrity()` detects the mismatch (`leaf_count` from traversal > `key_to_index_cache_length`) and fails.
5. `get_proof_of_inclusion(key)` returns a proof for the new leaf only, which does not match the actual root hash (which includes the orphaned old leaf). [4](#0-3) 

The Python binding `py_batch_insert` exposes this path directly to untrusted callers:

```rust
// crates/chia-datalayer/src/merkle/blob.rs:1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [5](#0-4) 

---

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A crafted `batch_insert` call with ≥ 3 entries where any of the first `n-2` entries share a key or hash with a later entry (or with each other) silently corrupts the `MerkleBlob`. The resulting tree root hash is computed over a structurally inconsistent tree. All proofs of inclusion generated from the corrupted blob are invalid — they do not verify against the actual root. A DataLayer node that accepts such a batch will produce and propagate incorrect Merkle proofs, allowing an attacker to make a node assert false inclusion or exclusion of DataLayer state.

---

### Likelihood Explanation

The `py_batch_insert` Python binding is the primary interface used by the Chia DataLayer during sync operations. Any code path that feeds externally-sourced key-value-hash triples into `batch_insert` without pre-deduplication is vulnerable. The attacker only needs to supply a list of ≥ 3 entries with a repeated key or hash; no privileged access is required. The bug is silent — no error is returned, and the corruption is only detectable via `check_integrity()`, which is not called automatically after `batch_insert`.

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the top of the fast-path loop in `batch_insert`, mirroring the guards in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, pre-deduplicate the input vector before processing, or refactor `batch_insert` to always route through `insert()` for the uniqueness check, accepting the small performance cost.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import os

blob = MerkleBlob(bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 after the bootstrap phase
kv_ids = [(KeyId(i), ValueId(i)) for i in range(5)]
hashes = [os.urandom(32) for _ in range(5)]

# Introduce a duplicate: entry 0 has the same key as entry 3
# Entry 3 is popped first (via insert() with check), entry 0 bypasses the check
kv_ids[0] = kv_ids[3]  # duplicate key

# This should raise KeyAlreadyPresent but does not
blob.batch_insert(kv_ids, hashes)

blob.calculate_lazy_hashes()

# check_integrity() will now fail because the blob has 5 leaf nodes
# but key_to_index only has 4 entries (the duplicate overwrote one)
blob.check_integrity()  # raises IntegrityKeyToIndexCacheLength or similar

# Proof of inclusion for the duplicated key is invalid against the actual root
proof = blob.get_proof_of_inclusion(kv_ids[3])
assert not proof.valid()  # proof does not match the corrupted root

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
