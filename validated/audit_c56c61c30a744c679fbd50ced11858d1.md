### Title
Permanent Fund Freeze via Blacklisted EVM Recipient in `finTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

A user initiating a NEAR → EVM bridge transfer can specify a USDC-blacklisted EVM address as the recipient. Once tokens are burned/locked on NEAR and the MPC signs the payload embedding that recipient, the EVM `finTransfer` will permanently revert on every attempt. Because no cancel or re-route mechanism exists on the NEAR side, the user's funds are irrecoverably frozen.

---

### Finding Description

**Entry path — NEAR `init_transfer`:**

When a user calls `ft_on_transfer` on the NEAR bridge with an `InitTransferMsg` whose `recipient` is an EVM address (e.g., a USDC-blacklisted address), `init_transfer_internal` immediately burns or locks the tokens and stores the `TransferMessage` in `pending_transfers`: [1](#0-0) 

The `destination_nonce` is also incremented atomically at this point and cannot be rolled back.

**Signing — NEAR `sign_transfer`:**

When a relayer calls `sign_transfer`, the recipient is read directly from the stored `TransferMessage` and embedded verbatim into the `TransferMessagePayload` that the MPC signs: [2](#0-1) 

There is no parameter to override or redirect the recipient. The signed payload is now permanently bound to the blacklisted address.

**Finalization — EVM `finTransfer`:**

The relayer submits the signed payload to the EVM bridge. The nonce is marked used before the transfer: [3](#0-2) 

Then, for a native ERC-20 token such as USDC, `safeTransfer` is called to the recipient: [4](#0-3) 

USDC's `transfer` reverts when the recipient is blacklisted. Because the entire transaction reverts, the EVM nonce is not permanently consumed — but the relayer can never make this call succeed. Every retry reverts identically.

**No recovery path on NEAR:**

There is no `cancel_transfer` or re-route function in the NEAR bridge contract. The `remove_transfer_message` helper is only called internally on storage-check failures, never exposed as a user-callable escape hatch. The `pending_transfers` entry and the burned/locked tokens remain in limbo indefinitely. [5](#0-4) 

---

### Impact Explanation

The user's tokens are burned/locked on NEAR at transfer initiation and can never be recovered. The EVM finalization permanently fails because the signed payload's recipient field is immutable and points to a blacklisted address. This constitutes an **irrecoverable lock of user funds** in the bridge flow, matching the Critical/High allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

Additionally, the `locked_tokens` accounting counter on NEAR is permanently inflated for the affected token/chain pair, corrupting bridge collateralization bookkeeping.

---

### Likelihood Explanation

USDC is an officially supported token on the bridge (Arbitrum USDC is explicitly referenced in the audit context). USDC maintains an on-chain blacklist. A user can accidentally or deliberately specify a blacklisted address as the EVM recipient — there is no validation of the recipient address against any blacklist at transfer initiation time. The scenario requires only a standard unprivileged bridge user action (`ft_transfer_call` / `initTransfer`) with a blacklisted address as the recipient field.

---

### Recommendation

1. **Add a `cancel_transfer` function on NEAR** that allows the original sender (or a DAO role after a timeout) to cancel a pending transfer, reverse the burn/lock, and refund the user — callable only when the transfer has not yet been finalized on the destination chain.

2. **Alternatively, allow recipient re-routing**: permit the original sender to update the recipient in `pending_transfers` before `sign_transfer` is called, so the MPC can sign a corrected payload.

3. **Validate recipient at initiation**: for EVM destinations, consider rejecting the zero address or other obviously invalid addresses at `init_transfer` time. Full blacklist validation is not feasible on-chain, but a cancel path is the essential safety net.

---

### Proof of Concept

1. Alice holds 1000 USDC-equivalent bridged tokens on NEAR.
2. Alice calls `ft_transfer_call` on the NEAR bridge with:
   ```json
   { "recipient": "eth:0x0E6b8E34dC115a2848F585851AF23D99D09b8463", "fee": "0", ... }
   ```
   (`0x0E6b8E34...` is a known USDC-blacklisted address on Arbitrum.)
3. `init_transfer_internal` burns Alice's 1000 tokens and stores the `TransferMessage` in `pending_transfers`. `destination_nonce` is incremented.
4. Relayer calls `sign_transfer` → MPC signs payload with `recipient = 0x0E6b8E34...`.
5. Relayer calls `OmniBridge.finTransfer` on Arbitrum EVM:
   - `completedTransfers[nonce] = true` (line 287)
   - Signature verified (line 311)
   - `IERC20(usdcAddress).safeTransfer(0x0E6b8E34..., amount)` → **reverts** with `"Blacklistable: account is blacklisted"`
   - Entire tx reverts; nonce not consumed on EVM.
6. Every subsequent `finTransfer` attempt reverts identically.
7. Alice's 1000 tokens are permanently burned on NEAR with no recovery path.

        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
