### Title
Sub-threshold Transfer Amount Causes Permanent Fund Lock in `sign_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts any transfer where `fee < amount`, but `sign_transfer` additionally requires `normalize(amount - fee) > 0`. For tokens with large decimal differences (e.g., NEAR 24 decimals â†’ EVM 6 decimals, diff = 18), any transfer where `amount - fee < 10^(origin_decimals - decimals)` passes `init_transfer` validation but permanently fails in `sign_transfer`. There is no cancel or refund path, so the user's tokens are irrecoverably locked in the bridge contract.

### Finding Description

`init_transfer` stores a `TransferMessage` after validating only that `fee.fee < amount`: [1](#0-0) 

Later, when a relayer calls `sign_transfer`, the code normalizes the net amount and enforces a strict positivity check: [2](#0-1) 

`normalize_amount` performs floor division by `10^(origin_decimals - decimals)`: [3](#0-2) 

For a token registered with `origin_decimals = 24` and `decimals = 6` (diff = 18), the divisor is `10^18`. Any transfer where `amount - fee < 10^18` yields `normalize_amount(...) = 0`, causing `sign_transfer` to revert with `ERR_INVALID_AMOUNT_TO_TRANSFER` on every call.

The only post-initiation mutation available is `update_transfer_fee`, which enforces `fee.fee >= current_fee.fee`: [4](#0-3) 

Increasing the fee only reduces `amount - fee` further, making the situation worse. `remove_transfer_message` is only called internally from `sign_transfer_callback` (when fee is zero and signing succeeds) and `claim_fee_callback`; it is never exposed as a user-callable escape hatch: [5](#0-4) 

There is no cancel or user-initiated refund function anywhere in the contract.

### Impact Explanation

A user whose transfer is stuck in `pending_transfers` has their tokens permanently locked in the bridge contract with no recovery path. This matches the **Critical** impact class: *permanent freezing / irrecoverable lock of user funds in bridge flows*.

### Likelihood Explanation

The scenario is reachable by any unprivileged user who calls `ft_transfer_call` with a small token amount. Tokens bridging between chains with large decimal differences (NEAR 24 â†’ EVM 6, diff = 18; NEAR 24 â†’ Solana 9, diff = 15) have normalization thresholds of `10^18` and `10^15` respectively. A user transferring fewer than 1 unit of the destination-chain token (e.g., less than 1 USDC = `10^6` EVM units, which is `10^6 * 10^18 = 10^24` yoctoNEAR â€” but for a token with 24 NEAR decimals and 6 EVM decimals, any amount below `10^18` yoctoNEAR = 1 microNEAR triggers the bug) will be permanently locked. This is a realistic user mistake, especially for dust amounts or when a user misunderstands the decimal scaling.

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before storing the transfer:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(amount_to_transfer > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the check already present in `sign_transfer` and ensures the transfer is rejected at initiation time (while the user's tokens can still be returned via the `ft_transfer_call` refund mechanism) rather than after the tokens are already locked.

### Proof of Concept

1. A token is registered with `origin_decimals = 24` (NEAR) and `decimals = 6` (EVM), so `diff_decimals = 18` and the normalization factor is `10^18`.
2. User calls `ft_transfer_call` with `amount = 5 * 10^17` (0.5 microNEAR worth of the token) and `fee = 0`, targeting an EVM recipient.
3. `init_transfer` checks `fee (0) < amount (5 * 10^17)` â€” passes. Transfer is stored in `pending_transfers`. User's tokens are now held by the bridge contract.
4. Relayer calls `sign_transfer` for this transfer.
5. `amount_without_fee() = 5 * 10^17 - 0 = 5 * 10^17`.
6. `normalize_amount(5 * 10^17, {decimals: 6, origin_decimals: 24}) = 5 * 10^17 / 10^18 = 0`.
7. `require!(0 > 0, ...)` â€” panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Every future call to `sign_transfer` for this transfer ID will also fail.
8. User attempts `update_transfer_fee` to increase fee â€” this only reduces `amount - fee`, keeping `normalize_amount` at 0.
9. No cancel function exists. Tokens are permanently locked. [2](#0-1) [1](#0-0) [3](#0-2)
