### Title
Cache-Poisoning via Unbound `inner.chunk_hash` / `metadata` in `ChunkEndorsementV2` Silently Drops a Validator's Endorsement — (`chain/client/src/stateless_validation/chunk_endorsement.rs`, `core/primitives/src/stateless_validation/chunk_endorsement.rs`)

---

### Summary

`ChunkEndorsementV2` carries two independently-signed blobs — `inner` (containing `chunk_hash`) and `metadata` (containing `shard_id`, `epoch_id`, `height_created`, `account_id`) — with no cryptographic binding between them. `ChunkEndorsementV2::verify()` checks each signature in isolation. An unprivileged peer that has observed two legitimate endorsements from the same validator V can splice them into a single message whose `inner.chunk_hash` refers to chunk X while `metadata` refers to chunk Y. The spliced message passes every check in `validate_chunk_endorsement`, is stored in the cache under Y's `ChunkProductionKey` with X's `chunk_hash`, and the deduplication guard then silently drops V's real endorsement for Y when it arrives. `collect_chunk_endorsements` subsequently filters the poisoned entry out, so V's stake is never counted toward chunk Y's endorsement threshold.

### Finding Description

**Structure of `ChunkEndorsementV2`** [1](#0-0) 

The struct has four fields: `inner` (holds `chunk_hash`), `signature` (sig over `inner`), `metadata` (holds `shard_id`, `epoch_id`, `height_created`, `account_id`), `metadata_signature` (sig over `metadata`). `verify()` checks each signature independently:

```rust
self.signature.verify(&inner, public_key)
    && self.metadata_signature.verify(&metadata, public_key)
```

There is no check that `inner.chunk_hash` is consistent with any field in `metadata`.

**Validation path**

`validate_chunk_endorsement` calls `validate_chunk_relevant_as_validator` using `endorsement.chunk_production_key()`, which reads exclusively from `metadata`: [2](#0-1) 

Then it calls `validate_chunk_endorsement_signature`, which calls `endorsement.verify()` — the independent two-signature check above: [3](#0-2) 

Neither step checks that `inner.chunk_hash` belongs to the chunk identified by `metadata`.

**Cache insertion**

After validation passes, the entry is stored under Y's `ChunkProductionKey` (from `metadata`) but with X's `chunk_hash` (from `inner`): [4](#0-3) 

**Deduplication guard drops the legitimate endorsement**

When V's real endorsement for Y arrives later, the guard fires and returns early: [5](#0-4) 

**Collection filters out the poisoned entry**

`collect_chunk_endorsements` filters cached entries by `chunk_hash == chunk_header.chunk_hash()`. The poisoned entry has X's hash, so it is excluded and V's stake is not counted: [6](#0-5)

### Impact Explanation

A block-producer node that has been fed enough spliced endorsements may fail to reach the endorsement-stake threshold for chunk Y. Depending on protocol rules, this can cause the block producer to stall waiting for endorsements, or to produce a block that omits chunk Y. The attack is targeted at specific (validator, chunk) pairs and can be repeated across multiple validators to suppress an entire chunk's endorsement set at a targeted node.

---

### Likelihood Explanation

**What the attacker needs:**
1. V's legitimate endorsement for chunk X — any chunk V has previously endorsed, observable from P2P gossip.
2. V's legitimate endorsement for chunk Y — the target chunk, also broadcast over P2P.
3. The ability to relay the spliced message to the target node before V's real endorsement for Y arrives.

All three are achievable by an unprivileged peer that is well-connected in the gossip network. The attacker does not need any validator key material; it only needs to observe and recombine existing signed messages. The race window is the propagation delay between V's endorsement broadcast and the target node's receipt of it — a realistic window for a peer sitting between V and the target.

---

### Recommendation

Add a cross-check inside `ChunkEndorsementV2::verify()` (or in `validate_chunk_endorsement_signature`) that asserts `inner.chunk_hash` is consistent with `metadata`. The simplest approach is to include `chunk_hash` inside `ChunkEndorsementMetadata` and sign it there as well, so that a single combined signature covers both the routing key and the committed hash. Alternatively, include the `ChunkProductionKey` fields inside `ChunkEndorsementInnerV1` so that the inner signature already binds the chunk hash to the shard/epoch/height triple, making splicing impossible.

### Proof of Concept

```
1. Run a local nearcore network with at least one chunk validator V assigned to two shards.
2. Capture V's endorsement E_X for chunk X (shard 0, height H):
       inner_X = {chunk_hash: X_hash}, sig_X = sign(inner_X)
       metadata_X = {shard_id:0, epoch_id:E, height:H, account_id:V}, meta_sig_X = sign(metadata_X)
3. Capture V's endorsement E_Y for chunk Y (shard 1, height H):
       inner_Y = {chunk_hash: Y_hash}, sig_Y = sign(inner_Y)
       metadata_Y = {shard_id:1, epoch_id:E, height:H, account_id:V}, meta_sig_Y = sign(metadata_Y)
4. Construct spliced endorsement E_spliced:
       inner = inner_X, signature = sig_X   (chunk X's hash, validly signed)
       metadata = metadata_Y, metadata_signature = meta_sig_Y  (chunk Y's routing key, validly signed)
5. Feed E_spliced into ChunkEndorsementTracker::process_chunk_endorsement on the target node
   BEFORE E_Y arrives.
6. Assert:
   a. process_chunk_endorsement returns Ok(()) (no error).
   b. The cache entry for (shard_id=1, epoch_id=E, height=H) under account V contains X_hash.
   c. When E_Y subsequently arrives, process_chunk_endorsement returns Ok(()) but does NOT
      update the cache (deduplication guard fires).
   d. collect_chunk_endorsements for chunk Y returns zero valid endorsements for V
      (X_hash ≠ Y_hash filter).
```

#[derive(Debug, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize, ProtocolSchema)]
pub struct ChunkEndorsementV2 {
    // This is the part of the chunk endorsement that signed and included in the block header
    inner: ChunkEndorsementInnerV1,
    // This is the signature of the inner field, to be included in the block header
    signature: Signature,
    // This consists of the metadata for chunk endorsement used in validation
    metadata: ChunkEndorsementMetadata,
    // Metadata signature is used to validate that the metadata is produced by the expected validator
    metadata_signature: Signature,
}

impl ChunkEndorsementV2 {
    fn verify(&self, public_key: &PublicKey) -> bool {
        let inner = borsh::to_vec(&self.inner).unwrap();
        let metadata = borsh::to_vec(&self.metadata).unwrap();
        self.signature.verify(&inner, public_key)
            && self.metadata_signature.verify(&metadata, public_key)
    }
}
