### Title
Missing Access Control on `add_prover` Allows Any User to Register a Malicious Prover, Bypassing All Proof Verification - (File: near/omni-bridge/src/lib.rs)

### Summary
The `add_prover` function in the NEAR omni-bridge contract has no access control, while its counterpart `remove_prover` is correctly restricted to `Role::DAO`. Any unprivileged account can register an arbitrary contract as the prover for any chain. Because every proof-gated bridge operation (`fin_transfer`, `deploy_token`, `bind_token`, `claim_fee`) resolves its prover from the `self.provers` map at call time, an attacker who inserts a malicious prover can make the bridge accept fabricated proof data and finalize unauthorized transfers, mint unbacked bridged tokens, or drain locked native assets.

### Finding Description

`add_prover` is a public, mutable contract method with no access-control decorator: [1](#0-0) 

Its symmetric counterpart is correctly guarded: [2](#0-1) 

The `provers` map (`UnorderedMap<ChainKind, AccountId>`) is the sole source of truth for which contract is trusted to verify cross-chain proofs for each chain: [3](#0-2) 

Every proof-gated entry point calls `self.verify_proof(chain_kind, prover_args)`, which dispatches to the account stored in `self.provers` for the given chain. For example, `fin_transfer`: [4](#0-3) 

`deploy_token` and `bind_token` follow the same pattern: [5](#0-4) [6](#0-5) 

`claim_fee` is similarly proof-gated: [7](#0-6) 

Because `add_prover` is callable by anyone, an attacker can overwrite the legitimate prover for any chain with a contract they control. Their malicious prover can return any `ProverResult` variant (`InitTransfer`, `LogMetadata`, `DeployToken`, `FinTransfer`) for arbitrary inputs, making the bridge treat fabricated data as verified cross-chain events.

### Impact Explanation

**Critical.** An attacker who registers a malicious prover for, say, `ChainKind::Eth` can:

1. Call `fin_transfer` with a fabricated `InitTransferMessage` claiming any amount of any token was locked on Ethereum. The bridge will mint or release that amount to an attacker-controlled NEAR address.
2. Call `deploy_token` with a fabricated `LogMetadataMessage` to register an arbitrary token mapping, corrupting the bridge's token accounting.
3. Call `claim_fee` with a fabricated `FinTransferMessage` to drain pending transfer balances.

This constitutes **direct unauthorized mint and release of bridged assets** and **permanent accounting corruption** across all chains served by the NEAR bridge contract.

### Likelihood Explanation

The function is public, requires no deposit, no role, and no special account. Any NEAR account can call it in a single transaction. The only prerequisite is knowing the contract address, which is public. The attack is immediately executable on mainnet by any user.

### Recommendation

Add the same DAO-only access control to `add_prover` that already protects `remove_prover`:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn add_prover(&mut self, chain: ChainKind, account_id: AccountId) {
    self.provers.insert(&chain, &account_id);
}
```

### Proof of Concept

1. Attacker deploys `MaliciousProver` on NEAR. Its `verify_proof` method ignores input and returns a `ProverResult::InitTransfer` claiming 1,000,000 USDC was locked on Ethereum, with `recipient = OmniAddress::Near(attacker.near)`.
2. Attacker calls `omni_bridge.add_prover(ChainKind::Eth, "malicious-prover.near")`. No access check fires; `self.provers[Eth]` is now the attacker's contract.
3. Attacker calls `omni_bridge.fin_transfer(FinTransferArgs { chain_kind: Eth, storage_deposit_actions: [...], prover_args: <any bytes> })`.
4. The bridge calls `malicious-prover.near.verify_proof(...)`, which returns the fabricated `InitTransfer` result.
5. `fin_transfer_callback` processes the result: it finds the USDC token mapping, mints 1,000,000 USDC to `attacker.near`, and emits a `FinTransferEvent`.
6. Attacker repeats until the bridge's locked token supply is exhausted or the deployed token supply is inflated beyond backing.

    #[pause(except(roles(Role::DAO)))]
    pub fn fin_transfer(&mut self, #[serializer(borsh)] args: FinTransferArgs) -> Promise {
        require!(
            args.storage_deposit_actions.len() <= 3,
            BridgeError::InvalidStorageAccountsLen.as_ref()
        );
        let mut main_promise = self.verify_proof(args.chain_kind, args.prover_args);

        let mut attached_deposit = env::attached_deposit();

        for action in &args.storage_deposit_actions {
            main_promise =
                main_promise.and(Self::check_or_pay_ft_storage(action, &mut attached_deposit));
        }

        main_promise.then(
            Self::ext(env::current_account_id())
                .with_attached_deposit(attached_deposit)
                .with_static_gas(FIN_TRANSFER_CALLBACK_GAS)
                .fin_transfer_callback(
                    &args.storage_deposit_actions,
                    env::predecessor_account_id(),
                ),
        )
    }
