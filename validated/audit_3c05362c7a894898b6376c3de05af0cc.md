Audit Report

## Title
Cross-User ETH Consumption in `PeripheryPayments.pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses the contract's entire native ETH balance when satisfying a WETH payment, with no per-user accounting. ETH left on the contract by a prior LP caller (from an overpaid `payable` liquidity add) is silently consumed by the next caller's payment, causing the prior LP to lose their ETH principal while the subsequent caller adds liquidity for free.

## Finding Description

`addLiquidityExactShares` is `payable` but performs no automatic ETH refund. [1](#0-0) 

When LP-A calls `addLiquidityExactShares{value: 5 ether}` for a WETH pool that only needs 2 ETH, the callback invokes `pay(WETH, LP-A, pool, 2e18)`. The branch at line 75 wraps exactly 2 ETH and sends WETH to the pool. The remaining 3 ETH stays on the contract until `refundETH()` is called in a separate transaction. [2](#0-1) 

When LP-B subsequently calls `addLiquidityExactShares` (no ETH sent) for a WETH pool needing 3 WETH, the callback invokes `pay(WETH, LP-B, pool, 3e18)`. At line 75, `nativeBalance = 3 ETH >= value = 3e18`, so the function wraps LP-A's 3 ETH and transfers WETH to the pool — without pulling any WETH from LP-B. LP-B's WETH allowance is never touched. [3](#0-2) 

The `receive()` guard (only accepting ETH from WETH) does not prevent this: it only applies to plain ETH transfers, not ETH sent with a function call via `msg.value`. [4](#0-3) 

The transient `PayContextAlreadyActive` guard only prevents reentrancy within a single transaction; it does not prevent LP-B from calling in a separate transaction after LP-A's transaction completes. [5](#0-4) 

The callback correctly passes `payer = msg.sender` (LP-B), but `pay()` ignores `payer` entirely when `nativeBalance >= value`, consuming contract-held ETH from any prior depositor. [6](#0-5) 

## Impact Explanation

LP-A suffers a direct loss of ETH principal equal to the overpaid amount. LP-B adds liquidity for free, paying nothing. The loss is bounded only by how much ETH LP-A overpaid. No privileged role is required — LP-B is any ordinary caller. This constitutes a direct loss of user principal, meeting Critical/High severity thresholds under Sherlock contest rules.

## Likelihood Explanation

Any user who sends more ETH than the pool needs (a common pattern when the exact amount is unknown at call time) leaves ETH on the contract. A frontrunner or any concurrent user can immediately consume it. The window is open from the end of LP-A's transaction until `refundETH()` is called in a separate transaction. The attack requires no special permissions, no malicious setup, and is repeatable.

## Recommendation

Track per-user ETH deposits using a transient slot keyed to `msg.sender` and credit only that amount in `pay()`. Alternatively, wrap `msg.value` into WETH immediately on entry (before any pool call) and treat it as an ERC-20 balance, eliminating the native-ETH branch in `pay()` for cross-user safety. At minimum, enforce that the native ETH branch in `pay()` is only used when `nativeBalance` does not exceed the ETH credited to the current `payer` in the current transaction.

// Foundry test sketch
function test_crossUserEthConsumption() public {
    // LP-A adds liquidity to a WETH/token pool, overpaying by 3 ETH
    vm.deal(lpA, 5 ether);
    vm.prank(lpA);
    adder.addLiquidityExactShares{value: 5 ether}(
        pool, lpA, 0, deltas_needing_2eth, 5 ether, 0, ""
    );
    // 3 ETH remains on adder contract — no automatic refund
    assertEq(address(adder).balance, 3 ether);

    // LP-B adds liquidity to a WETH pool needing 3 WETH, sends no ETH
    // LP-B has WETH approval but pay() never pulls it
    vm.prank(lpB);
    adder.addLiquidityExactShares(
        pool, lpB, 0, deltas_needing_3eth, 3 ether, 0, ""
    );

    // LP-A's 3 ETH is gone, LP-B paid nothing
    assertEq(address(adder).balance, 0);
    assertEq(weth.balanceOf(lpB), initialWethB); // LP-B's WETH untouched
}
