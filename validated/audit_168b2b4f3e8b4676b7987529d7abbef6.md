### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Contract Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by the caller is a declared class before committing the state update. This is a direct analog to the reported bug: a critical state-changing operation proceeds without the guard check that is required for protocol correctness. A malicious contract can replace its own class hash with an arbitrary, undeclared felt value. The OS accepts this without complaint, permanently rendering the contract uncallable and freezing any funds held in its storage.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 878–916) reads the requested class hash from the syscall request and immediately writes it into `contract_state_changes` with no check that the hash corresponds to a declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The explicit `TODO` comment at line 898 acknowledges the missing guard. The analogous check that *should* be performed is: `contract_class_changes[class_hash] != 0` (i.e., the class was previously declared via a Declare transaction).

Compare this with `execute_entry_point.cairo` (lines 154–166), which is the code path that runs when the contract is *subsequently called*:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

If `class_hash` was never declared, `dict_read` returns the default value `0`. `find_element` with key `0` will then fail to locate a compiled class, causing proof generation to abort for any future call to that contract. The contract becomes permanently uncallable at the OS/proof level — not merely reverting, but making valid proof generation impossible for any transaction targeting it.

This is structurally identical to the reported bug: the check that *should* gate the operation (`getBorrowingEnabled()` / "is this class declared?") is performed downstream (at call time / at deposit time) but is absent at the point where the irreversible state mutation occurs.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value:

- Any subsequent call to the contract causes `execute_entry_point` to fail at `find_element` (no compiled class for hash `0`), making proof generation impossible for that call.
- The sequencer is forced to exclude all future calls to the contract from every block.
- All ERC-20 balances, NFTs, or other assets held in the contract's storage are permanently inaccessible — no withdrawal, no rescue, no upgrade path.
- The state corruption is committed on-chain via a valid proof (the block containing the `replace_class` call is provable because the contract is not called again in that same block).

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

1. Attacker deploys a contract that appears to be a legitimate vault, DEX, or bridge, attracting user deposits.
2. After accumulating victim funds, the attacker sends a transaction that triggers `replace_class` with an arbitrary undeclared felt (e.g., `0xdeadbeef`).
3. The OS processes this without any validation — the `TODO` comment confirms the check is intentionally absent.
4. The block containing this transaction is provable (the contract is not called again in the same block).
5. In all subsequent blocks, the contract is permanently uncallable at the proof level.
6. Victim funds are frozen forever.

No privileged role, leaked key, or malicious sequencer is required. The attacker only needs to deploy a contract and send a standard transaction.

---

### Recommendation

In `execute_replace_class`, before writing the new `StateEntry`, add a validation that the supplied `class_hash` is present in `contract_class_changes` with a non-zero compiled class hash:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the check already implicitly performed in `execute_entry_point` and closes the gap between where the state mutation occurs and where the validity constraint is enforced.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` contract; victims deposit tokens.
2. Attacker sends an invoke transaction calling `MaliciousVault.__execute__`, which internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (never declared).
3. `execute_replace_class` in `syscall_impls.cairo` (lines 896–914) writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes` with no validation.
4. The block is proven successfully (no call to `MaliciousVault` occurs after the replace in the same block).
5. In block N+1, a victim attempts to withdraw. The sequencer runs `execute_entry_point`:
   - `dict_read(contract_class_changes, key=0xdeadbeef)` → returns `0` (undeclared).
   - `find_element(..., key=0)` → fails; no compiled class with hash `0` exists.
   - Proof generation for this call is impossible.
6. The sequencer must drop all calls to `MaliciousVault` from every future block. Victim funds are permanently frozen.
