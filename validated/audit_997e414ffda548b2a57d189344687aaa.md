### Title
Native ETH Delivery to Non-Payable Contract Recipient Permanently Locks User Funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge::finTransfer` attempts a low-level ETH call to `payload.recipient` when finalizing native-ETH bridge transfers. If the recipient is a contract without a payable fallback/receive function, this call always fails and reverts. Because the NEAR-side tokens are already locked/burned at `initTransfer` time and no NEAR-side refund mechanism exists for EVM-side delivery failures, the user's funds are permanently frozen.

### Finding Description
In `OmniBridge.sol`, `finTransfer` handles native ETH transfers (`payload.tokenAddress == address(0)`) with:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287 — set before transfer

// ...signature verification...

if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();           // line 322
}
``` [1](#0-0) 

The `payload.recipient` is an EVM address freely chosen by the user when calling `initTransfer` on NEAR. There is no pre-validation that this address can receive ETH. If the user specifies a contract without a payable fallback (e.g., a Gnosis Safe, DAO treasury, or any non-payable contract), the low-level call returns `false`, causing `finTransfer` to always revert with `FailedToSendEther`.

Because the whole transaction reverts, `completedTransfers[destinationNonce]` is also reverted, so the nonce is never consumed. However, the NEAR-side tokens were already locked or burned during `initTransfer`:

```rust
// NEAR side: tokens locked/burned before EVM finalization
fn init_transfer_internal(&mut self, transfer_message: TransferMessage, ...) -> U128 {
    // ...
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(...);
}
``` [2](#0-1) 

There is no callback or cross-chain message from the EVM side back to NEAR to trigger a refund when `finTransfer` permanently fails. Every subsequent relay attempt will revert for the same reason. No admin rescue function for stuck ETH exists in `OmniBridge.sol`.

The analogous pattern from the external report: just as `registerWithAvaxFee` lacked validation that the sender could receive AVAX, `finTransfer` lacks validation that the EVM recipient can receive ETH before the NEAR-side tokens are irrevocably committed.

### Impact Explanation
- **Critical — Permanent freezing of user funds.** The user's NEAR-side tokens (locked or burned) can never be recovered. The ETH equivalent remains in the bridge contract indefinitely. No recovery path exists for the user or protocol.
- The ETH held in the bridge is also effectively frozen since the nonce is never consumed and every relay attempt reverts.

### Likelihood Explanation
- Many legitimate EVM addresses cannot receive ETH via plain low-level calls: Gnosis Safe multisigs, DAO contracts, protocol treasuries, smart contract wallets, and any contract without a `receive()` or `fallback() payable` function.
- Users bridging to a DAO treasury or multisig — a common real-world use case — will trigger this condition.
- The NEAR-side `initTransfer` provides no warning about EVM-side recipient compatibility.
- No off-chain tooling is required; any user can trigger this by specifying a non-payable contract address as recipient.

### Recommendation
1. **Pre-validate on EVM before NEAR commitment**: Add a view-level check or require the recipient to be an EOA (via `payload.recipient.code.length == 0`) when `tokenAddress == address(0)`, or require the recipient to explicitly register as ETH-capable.
2. **Add a rescue/redirect mechanism**: Allow the original sender (proven via MPC signature) to redirect a stuck transfer to a different recipient if the initial finalization has failed N times.
3. **Wrap ETH as WETH for contract recipients**: If `payload.recipient` is a contract, deposit into WETH and transfer WETH instead of raw ETH, eliminating the payable-fallback requirement.
