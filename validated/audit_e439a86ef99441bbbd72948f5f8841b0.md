### Title
Unguarded `refundETH()` allows any caller to steal excess ETH left on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.refundETH()` transfers the router's entire ETH balance to `msg.sender` with no access control and no per-depositor accounting. Any attacker who calls it in a separate transaction after a victim's multicall left excess ETH on the router receives those funds.

### Finding Description
`refundETH()` is declared `external payable` and unconditionally sends `address(this).balance` to `msg.sender`:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
``` [1](#0-0) 

The `pay()` function, when `token == WETH` and the router holds enough native ETH, wraps exactly the required amount and leaves any surplus on the contract:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [2](#0-1) 

There is no per-user ETH ledger, no `msg.sender` check in `refundETH`, and no restriction to the originating multicall context. The router's `receive()` guard (rejects non-WETH senders) does not prevent ETH from accumulating via `multicall{value: ...}`. [3](#0-2) 

### Impact Explanation
Direct loss of user principal. Any ETH a user sends in excess of their swap's `amountIn` — a common pattern when the exact cost is unknown — is immediately claimable by any third party who calls `refundETH()` before the user does. The stolen amount equals `msg.value - amountIn` per affected transaction, with no upper bound.

### Likelihood Explanation
Medium. The intended usage pattern (include `refundETH` as the last call in a multicall) is not enforced on-chain. Users who send a round ETH value for a sub-round swap, or who construct multicalls programmatically and omit the refund step, will leave ETH on the router. MEV bots monitoring the mempool can front-run or back-run the victim's transaction to claim the balance atomically.

### Recommendation
Track the depositor within the multicall context and restrict `refundETH` to that address, or record per-caller ETH deposits in transient storage at `multicall` entry and clear them on exit. A minimal fix is to add a `recipient` parameter to `refundETH` and require callers to pass their own address, combined with a transient-storage deposit ledger so only the depositor's balance is refundable. Alternatively, auto-refund excess ETH at the end of each swap entry point rather than relying on a separate user-initiated call.
