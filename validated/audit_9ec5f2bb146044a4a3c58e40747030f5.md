### Title
Cross-Chain Replay of `deployToken` Signatures Due to Missing Chain ID in Signed Message — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

### Summary
The Borsh-encoded message that NEAR MPC signs for `deployToken`/`deploy_token` operations does not include the destination chain ID. Because the same MPC-derived key (`nearBridgeDerivedAddress`) is shared across all supported chains (Ethereum, Arbitrum, Base, BNB, Polygon, Starknet, Solana, etc.), a valid `deployToken` signature captured from one chain can be replayed verbatim on any other chain to deploy the same bridge token without a new MPC authorization. This is a domain-separation flaw directly analogous to the AES-CTR malleability issue: the signed payload lacks a binding to its intended target, allowing an attacker to redirect it.

### Finding Description

**Root cause — EVM (`OmniBridge.sol`)**

`deployToken` constructs its Borsh payload as:

```
PayloadType::Metadata | encodeString(token) | encodeString(name) | encodeString(symbol) | bytes1(decimals)
``` [1](#0-0) 

No `omniBridgeChainId` is included. Contrast this with `finTransfer`, which embeds `omniBridgeChainId` twice in its payload: [2](#0-1) 

The same `nearBridgeDerivedAddress` is used as the verification key on every EVM chain: [3](#0-2) 

**Root cause — Starknet (`bridge_types.cairo`)**

`MetadataPayload::to_borsh()` encodes only `[PayloadType | token | name | symbol | decimals]` with no chain ID: [4](#0-3) 

`TransferMessagePayload::to_borsh(chain_id)` correctly takes and embeds `chain_id`: [5](#0-4) 

The same `omni_bridge_derived_address` is used for signature verification in `_verify_borsh_signature`: [6](#0-5) 

**Root cause — Solana (`deploy_token.rs`)**

`DeployTokenPayload::serialize_for_near` encodes only `[IncomingMessageType::Metadata | token | name | symbol | decimals]` with no chain ID: [7](#0-6) 

**Attack path**

1. Attacker observes a legitimate `deployToken(signatureData, metadata)` transaction on Ethereum (publicly visible on-chain).
2. Attacker submits the identical `(signatureData, metadata)` call to the Arbitrum, Base, BNB, Polygon, Starknet, or Solana bridge contract.
3. Each target chain's signature verification passes because the signed bytes are identical — no chain ID distinguishes them.
4. The token is deployed on every target chain where it has not yet been registered (`ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED` only blocks a second deployment on the *same* chain).
5. NEAR's provers process the resulting `DeployToken` events from each chain and register the attacker-triggered token addresses as the canonical bridge tokens for those chains.

### Impact Explanation

An unprivileged attacker can cause unauthorized token deployment on any chain that shares the same `nearBridgeDerivedAddress`. Concretely:

- A token intended only for Ethereum can be force-deployed on Arbitrum, Base, BNB, Polygon, Starknet, and Solana simultaneously.
- If the attacker deploys before the legitimate operator does, NEAR registers the attacker-triggered contract address as the canonical bridge token for that chain. The legitimate deployment then permanently fails with `ERR_TOKEN_EXIST`.
- Users bridging to the affected chain receive tokens at the attacker-triggered address. While the token contract itself is a standard BridgeToken (the bridge contract holds mint authority), the address is not the one the operator planned, breaking any off-chain tooling, front-ends, or integrations that rely on a known address.
- This matches the allowed High impact: **"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment."**

### Likelihood Explanation

- No privileged access is required. Any observer of on-chain transactions can extract the `signatureData` and `metadata` from a `deployToken` call.
- The replay requires only a standard contract call on the target chain — no special tooling.
- The bridge is live across at least seven chains (`ChainKind` enum: Eth, Arb, Base, Bnb, Pol, HyperEvm, Abs for EVM; plus Strk and Sol), multiplying the replay surface. [8](#0-7) 

### Recommendation

Include the destination chain ID in the Borsh-encoded `deployToken` message, exactly as `finTransfer` already does. For EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD THIS
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `MetadataPayload::to_borsh()` in `starknet/src/bridge_types.cairo` (pass and embed `chain_id` as `TransferMessagePayload::to_borsh` already does) and to `DeployTokenPayload::serialize_for_near` in `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs` (embed `SOLANA_OMNI_BRIDGE_CHAIN_ID`). The NEAR side must update its signing logic to include the target chain ID in the payload it submits to MPC.

### Proof of Concept

1. Deploy the EVM bridge on Ethereum (chain ID byte `0x00`) and Arbitrum (chain ID byte `0x03`).
2. Call `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Ethereum. Capture `sig` from the transaction calldata.
3. Call `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum with the identical `sig`.
4. Observe: the Arbitrum call succeeds — `ECDSA.recover(keccak256(borshEncoded), sig) == nearBridgeDerivedAddress` because `borshEncoded` is byte-for-byte identical on both chains.
5. `DeployToken` event is emitted on Arbitrum; NEAR registers the Arbitrum token address as the canonical USDC bridge token.
6. Any subsequent legitimate `deployToken` call for `usdc.near` on Arbitrum reverts with `ERR_TOKEN_EXIST`.
