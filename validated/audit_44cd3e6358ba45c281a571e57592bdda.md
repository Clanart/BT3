### Title
`replace_class` Syscall Accepts Undeclared Class Hashes Without Validation, Enabling Permanent Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not verify that the new class hash provided by a contract corresponds to a previously declared class. Any contract can replace its own class hash with an arbitrary, undeclared value. The OS accepts and commits this state change immediately, with no validation against the `contract_class_changes` dict. Subsequent calls to the affected contract will fail permanently because the class does not exist in the class trie, freezing any funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The explicit `TODO` comment at line 898 acknowledges the missing check. The function receives `contract_class_changes` as an implicit argument in the broader syscall dispatch context (via `execute_syscalls`), but `execute_replace_class` itself does not have access to `contract_class_changes` and performs **no lookup** to confirm the requested `class_hash` was ever declared via a `declare` transaction.

This is directly analogous to the oracle report's root cause: a new entity (class hash) becomes effective in the protocol state immediately, with no validation that it is legitimate.

The `execute_declare_transaction` function in `transaction_impls.cairo` is the only path that legitimately writes to `contract_class_changes`:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

But `execute_replace_class` only writes to `contract_state_changes`, and never cross-checks against `contract_class_changes`. There is no enforcement that `class_hash` in the replace request maps to any entry in the class trie.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract calls `replace_class` with an undeclared class hash:

1. The OS updates `contract_state_changes` for that contract address with the invalid class hash. The transaction succeeds and is committed to the block.
2. All future calls to the contract (including calls to withdraw or transfer funds) require the OS to load and execute the contract's class. Since the class hash is not in the class trie, execution cannot proceed.
3. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — there is no recovery path once the state is committed.

This affects not only the contract owner but any third-party users who have deposited funds into the contract.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any deployed contract without any privileged role. The attacker-controlled entry path is:

1. An unprivileged user deploys a contract (e.g., a shared vault or escrow).
2. The contract (or a malicious upgrade path within it) calls `replace_class` with a fabricated, never-declared class hash.
3. The OS processes the syscall, updates the contract's class hash in `contract_state_changes`, and commits the block.
4. The contract is permanently bricked.

The attack requires no leaked keys, no operator collusion, no 51% attack, and no external dependency. It is reachable by any contract deployer or any contract that exposes an upgrade function without proper access control. The missing OS-level check means even a well-intentioned contract with a buggy upgrade path can trigger this irreversibly.

---

### Recommendation

In `execute_replace_class`, add `contract_class_changes: DictAccess*` as an implicit argument and perform a `dict_read` to verify that the requested `class_hash` has a non-zero compiled class hash entry (i.e., it was previously declared). Reject the syscall (write a failure response) if the class hash is not found in `contract_class_changes`. This mirrors the existing pattern used in `execute_declare_transaction`, which enforces `prev_value=0` to prevent double-declaration.

---

### Proof of Concept

1. Deploy contract `VaultA` holding user funds.
2. From within `VaultA`, invoke the `replace_class` syscall with `class_hash = 0xdeadbeef` (never declared).
3. Observe: `execute_replace_class` in `syscall_impls.cairo` L896–L910 performs no lookup against `contract_class_changes`. It directly calls `dict_update` on `contract_state_changes` with the invalid hash.
4. The block is proven and committed. `VaultA`'s state entry now has `class_hash = 0xdeadbeef`.
5. Any subsequent call to `VaultA` fails at class resolution — the class trie has no entry for `0xdeadbeef`.
6. All funds in `VaultA` are permanently frozen.
