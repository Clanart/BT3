The vulnerability claim is valid. Here is the full analysis:

---

### Title
Unprotected `refundETH()` allows any caller to steal excess ETH left in `MetricOmmPoolLiquidityAdder` after a WETH-pool liquidity deposit — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` is a public, access-control-free function that sends the contract's entire native ETH balance to `msg.sender`. Because `addLiquidityExactShares` is `payable` and the `pay()` helper only wraps the exact amount of ETH needed (leaving any excess in the contract), any attacker can call `refundETH()` in a subsequent transaction and steal the unspent ETH.

---

### Finding Description

`PeripheryPayments.refundETH()` has no caller restriction: [1](#0-0) 

It unconditionally sends `address(this).balance` to `msg.sender`.

The `pay()` helper, when settling a WETH-pool deposit, wraps only the exact amount needed and transfers it to the pool — any ETH the user sent above that amount remains in the contract: [2](#0-1) 

`addLiquidityExactShares` is `payable` and makes no automatic refund after the pool call returns: [3](#0-2) 

`_addLiquidity` also performs no ETH sweep after the pool call completes: [4](#0-3) 

The `receive()` guard only blocks direct ETH sends from non-WETH addresses; it does **not** prevent ETH from accumulating via `payable` function calls: [5](#0-4) 

---

### Impact Explanation

A user who sends `N` ETH to `addLiquidityExactShares` for a WETH pool where only `M < N` ETH is consumed loses `N - M` ETH. Any address that calls `refundETH()` before the user does receives the full residual balance. This is a **direct, complete loss of user principal** with no protocol-side protection.

---

### Likelihood Explanation

- Sending a small ETH buffer above the exact required amount is standard practice to avoid reverts from minor price movement.
- The attack requires only a single public call with no special privileges.
- A bot watching the mempool can front-run the user's own `refundETH()` call, or simply call it first if the user does not bundle the refund in the same `multicall`.

---

### Recommendation

Either:

1. **Auto-refund in `_addLiquidity`**: after the pool call returns, push any remaining `address(this).balance` back to `payer`.
2. **Restrict `refundETH` to a tracked recipient**: record the depositor in transient storage alongside the pay context and enforce `msg.sender == depositor` in `refundETH()`.
3. **Require users to bundle via `multicall`**: document and enforce that `addLiquidityExactShares` must always be called inside a `multicall` that ends with `refundETH()`, and add a reentrancy-safe check that no ETH remains after the outer call.

Option 1 is the simplest and eliminates the race condition entirely.

---

### Proof of Concept

```
1. Pool token0 = WETH, token1 = USDC.
2. User calls addLiquidityExactShares{value: 2 ether}(...) where the pool
   only needs 1 ether of WETH.
3. pay() wraps 1 ether → WETH → pool. 1 ether remains in the contract.
4. addLiquidityExactShares returns successfully.
5. Attacker calls refundETH() → receives 1 ether.
6. User calls refundETH() → receives nothing (balance is 0).
```
