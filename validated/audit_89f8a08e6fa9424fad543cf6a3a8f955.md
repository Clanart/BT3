### Title
Ante Handler Balance Check Ignores In-Flight Value Transfers, Admitting Transactions That Fail During DeliverTx — (File: `ante/eth.go`)

---

### Summary

`VerifyEthAccount` and `CheckEthCanTransfer` in `ante/eth.go` read the sender's balance from the **check-state store**, which reflects fee deductions from prior successful `CheckTx` calls but **never reflects value transfers** (those only execute during `DeliverTx`). As a result, a second transaction from the same sender can pass `CheckTx` even when the sender's actual post-execution balance will be insufficient to cover its value transfer, causing the transaction to fail during `DeliverTx` and wasting the sender's gas fees.

---

### Finding Description

The Ethereum ante handler pipeline in `newEthAnteHandler` (`evmd/ante/handler_options.go`) runs these steps in order during `CheckTx`:

1. **`VerifyEthAccount`** — reads `balance` from the check-state store and asserts `balance >= tx.Cost()` (where `Cost = value + gasLimit * gasPrice`).
2. **`CheckEthCanTransfer`** — reads `balance` again and asserts `balance >= tx.Value()`.
3. **`CheckEthGasConsume`** — calls `DeductTxCostsFromUserBalance`, which **writes** `gasLimit * gasPrice` back to the check-state store. [1](#0-0) [2](#0-1) [3](#0-2) 

Because step 3 only deducts the **fee** (not the value), the check-state balance after `Tx1`'s `CheckTx` is:

```
check_balance = B - G1          # fee deducted, value V1 NOT deducted
```

When `Tx2` arrives, `VerifyEthAccount` sees `check_balance = B - G1` and asserts:

```
(B - G1) >= V2 + G2             # passes if B >= G1 + V2 + G2
```

But the actual balance available to `Tx2` during `DeliverTx` (after `Tx1` has fully executed) is:

```
deliver_balance = B - G1 - V1 - G2   # after Tx1's fee + value + Tx2's fee
```

For `Tx2`'s EVM execution to succeed, we need `B - G1 - V1 - G2 >= V2`. The `CheckTx` gate only enforces `B >= G1 + V2 + G2`, which is **weaker by `V1`**.

The `AnteCache` tracks nonces (not balances), so it provides no protection here. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

When `Tx2` is included in a block, the `DeliverTx` ante handler deducts `G2` upfront (in `CheckEthGasConsume`). The EVM then attempts to transfer `V2` but finds insufficient balance and reverts. The gas consumed up to the revert point (at minimum the intrinsic gas, 21 000 gas for a simple transfer) is **permanently lost**. This is a mis-accounting of valid user funds/fees: the mempool admitted a transaction that was guaranteed to fail, and the sender bears the cost.

This matches the allowed High impact: *"ante handler, mempool, or proposal handling bug that permits … valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

Any unprivileged user who submits two sequential ETH-value-transfer transactions whose combined value exceeds `balance - fees` triggers this path. No special privileges, governance access, or validator collusion is required. The entry path is a standard `eth_sendRawTransaction` RPC call. The scenario is realistic for wallets or bots that pipeline multiple transfers without waiting for block confirmation.

---

### Recommendation

Track the **cumulative committed value** of in-flight transactions per sender in the check state, analogous to how `CheckEthGasConsume` already tracks and deducts fees. Concretely, after `CheckEthGasConsume` deducts the fee, also deduct the transaction's `value` from the check-state balance (or maintain a separate "pending value" counter per address in the `AnteCache`). This mirrors go-ethereum's txpool behavior, which maintains a per-sender "pending balance" that accounts for both fees and value of all queued transactions.

---

### Proof of Concept

**Setup:** Sender balance `B = G1 + V + G2` (e.g., `B = 21000 + 1000 + 21000 = 43000` wei, `V = 1000`, `G1 = G2 = 21000`).

1. **Tx1** (`nonce=N`, `value=V=1000`, `gasLimit=21000`, `gasPrice=1`): `cost = 22000`. `B=43000 >= 22000` ✓. `CheckTx` passes. Check-state balance becomes `43000 - 21000 = 22000`.

2. **Tx2** (`nonce=N+1`, `value=V=1000`, `gasLimit=21000`, `gasPrice=1`): `cost = 22000`. Check-state balance `22000 >= 22000` ✓. `CheckTx` passes. Both transactions are in the mempool.

3. **DeliverTx Tx1:** Deliver-state balance starts at `B=43000`. Fee `21000` deducted → `22000`. EVM transfers `1000` → `21000`.

4. **DeliverTx Tx2:** Balance `21000`. Fee `21000` deducted → `0`. EVM attempts to transfer `1000`. Balance `0 < 1000`. **EVM reverts.** Intrinsic gas (21000) is consumed. Sender loses `G2 = 21000` wei in gas fees for a transaction that was guaranteed to fail at `CheckTx` admission time. [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8)

func newEthAnteHandler(options HandlerOptions) sdk.AnteHandler {
	return func(ctx sdk.Context, tx sdk.Tx, simulate bool) (sdk.Context, error) {
		blockCfg, err := options.EvmKeeper.EVMBlockConfig(ctx, options.EvmKeeper.ChainID())
		if err != nil {
			return ctx, errorsmod.Wrap(errortypes.ErrLogic, err.Error())
		}
		evmParams := &blockCfg.Params
		evmDenom := evmParams.EvmDenom
		feemarketParams := &blockCfg.FeeMarketParams
		baseFee := blockCfg.BaseFee
		rules := blockCfg.Rules

		// all transactions must implement FeeTx
		_, ok := tx.(sdk.FeeTx)
		if !ok {
			return ctx, errorsmod.Wrapf(errortypes.ErrInvalidType, "invalid transaction type %T, expected sdk.FeeTx", tx)
		}

		// We need to setup an empty gas config so that the gas is consistent with Ethereum.
		ctx, err = interfaces.SetupEthContext(ctx)
		if err != nil {
			return ctx, err
		}

		if err := cosmos.CheckEthMempoolFee(ctx, tx, simulate, baseFee, evmDenom); err != nil {
			return ctx, err
		}

		if err := cosmos.CheckEthMinGasPrice(tx, feemarketParams.MinGasPrice, baseFee); err != nil {
			return ctx, err
		}

		if err := interfaces.ValidateEthBasic(ctx, tx, evmParams, baseFee); err != nil {
			return ctx, err
		}

		ethSigner := ethtypes.MakeSigner(blockCfg.ChainConfig, blockCfg.BlockNumber, blockCfg.BlockTime)
		if err := evmante.VerifyEthSig(tx, ethSigner); err != nil {
			return ctx, err
		}

		// AccountGetter cache the account objects during the ante handler execution,
		// it's safe because there's no store branching in the ante handlers.
		accountGetter := evmante.NewCachedAccountGetter(ctx, options.AccountKeeper)

		if err := evmante.VerifyEthAccount(ctx, tx, options.EvmKeeper, evmDenom, accountGetter, rules); err != nil {
			return ctx, err
		}

		if err := evmante.CheckEthCanTransfer(ctx, tx, baseFee, rules, options.EvmKeeper, evmParams); err != nil {
			return ctx, err
		}

		ctx, err = evmante.CheckEthGasConsume(
			ctx, tx, rules, options.EvmKeeper,
			baseFee, evmDenom,
		)
		if err != nil {
			return ctx, err
		}

// CheckSenderBalance validates that the tx cost value is positive and that the
// sender has enough funds to pay for the fees and value of the transaction.
func CheckSenderBalance(
	balance sdkmath.Int,
	tx *ethtypes.Transaction,
) error {
	cost := tx.Cost()

	if cost.Sign() < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInvalidCoins,
			"tx cost (%s) is negative and invalid", cost,
		)
	}

	if balance.IsNegative() || balance.BigInt().Cmp(cost) < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInsufficientFunds,
			"sender balance < tx cost (%s < %s)", balance, tx.Cost(),
		)
	}
	return nil
}
