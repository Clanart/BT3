### Title
Missing Declared Class Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall updates a contract's class hash in `contract_state_changes` without verifying that the new class hash is actually declared in `contract_class_changes`. This creates an inconsistency between the contract-state component and the class-registry component — a direct structural analog to the external report's pattern of a setter updating one reference store while dependent components are left inconsistent. Any unprivileged contract can exploit this to permanently brick itself (and freeze any funds it holds) by supplying an undeclared class hash.

---

### Finding Description

**Root cause — `execute_replace_class` (`syscall_impls.cairo`, line 878):**

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The function accepts `request.class_hash` verbatim and writes it into `contract_state_changes`. There is **no check** that this class hash exists in `contract_class_changes` (the class registry). The TODO comment at line 898 explicitly acknowledges this missing validation.

**Dependent component — `execute_entry_point` (`execute_entry_point.cairo`, lines 154–166):**

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

When the contract is next invoked, `execute_entry_point` reads the class hash from `contract_state_changes` (now the attacker-supplied undeclared hash), then performs `dict_read` on `contract_class_changes`. Because the hash was never declared, the dict returns `0`. `find_element` then asserts that a `CompiledClassFact` with `hash == 0` exists in `compiled_class_facts_bundle` — an assertion that will fail, making the contract permanently unexecutable.

**The two-component inconsistency (direct analog to the report):**

| Component | State after malicious `replace_class` |
|---|---|
| `contract_state_changes` | Updated with undeclared `class_hash` |
| `contract_class_changes` | No entry for that `class_hash` (compiled hash = 0) |
| `compiled_class_facts_bundle` | No fact with hash 0 |

`execute_entry_point` bridges all three; the inconsistency between the first and the latter two causes an irrecoverable OS-level assertion failure for every future call to the contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A malicious contract deployer can:
1. Deploy a contract that accepts user deposits (e.g., a vault or DeFi primitive).
2. After accumulating user funds, call `replace_class` with any felt value that is not a declared class hash.
3. The `replace_class` transaction succeeds — the OS does not validate the hash.
4. Every subsequent call to the contract causes `find_element` to fail inside the OS, making the contract permanently unexecutable.
5. All funds held by the contract are permanently frozen with no recovery path.

---

### Likelihood Explanation

`replace_class` is an unprivileged syscall callable by any contract from its own execution context. No special role, key, or operator access is required. The OS explicitly skips the validation (acknowledged by the TODO). The sequencer's off-chain simulation may also skip this check if it defers to OS semantics. Any contract deployer — an entirely unprivileged actor — can trigger this path.

---

### Recommendation

Inside `execute_replace_class`, before writing to `contract_state_changes`, add a `dict_read` on `contract_class_changes` for the requested `class_hash` and assert the returned compiled class hash is non-zero (i.e., the class is declared). This is already noted as a TODO at line 898 and should be promoted to a hard requirement.

### Proof of Concept

1. Deploy contract `Vault` that:
   - Accepts ERC-20 deposits from users.
   - Exposes a privileged `brick()` entry point that calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
2. Users deposit funds; `Vault` accumulates a balance.
3. Attacker calls `brick()`. The OS executes `execute_replace_class`:
   - `contract_state_changes[vault_address].class_hash` ← `0xdeadbeef`
   - No validation against `contract_class_changes`. Transaction succeeds.
4. Any subsequent `call_contract` or `invoke` targeting `Vault` reaches `execute_entry_point`:
   - `dict_read(contract_class_changes, 0xdeadbeef)` → `0` (undeclared).
   - `find_element(..., key=0)` → assertion failure; OS cannot produce a valid proof for any block containing this call.
5. `Vault` is permanently unexecutable. All deposited funds are frozen with no withdrawal path.
