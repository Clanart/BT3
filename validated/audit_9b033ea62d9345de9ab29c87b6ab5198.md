Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain stranded ETH from the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` transfers the router's entire native ETH balance to `msg.sender` with no access control or depositor verification. ETH sent as `msg.value` with a `multicall{value: X}(...)` call bypasses the `receive()` guard and can remain on the router after the multicall completes if the user omits a `refundETH` sub-call. Any attacker can then call `refundETH()` in a separate transaction to steal that stranded ETH.

## Finding Description

`refundETH()` unconditionally sends the full contract ETH balance to the caller: [1](#0-0) 

The `receive()` guard only blocks plain ETH transfers (no calldata), not ETH attached to a function call via `msg.value`: [2](#0-1) 

When a user calls `multicall{value: 1 ETH}(...)`, the ETH arrives through the `multicall` function's `payable` modifier — `receive()` is never triggered. Inside the multicall, `pay()` wraps only the exact `amountIn` into WETH: [3](#0-2) 

Any ETH above the swap's exact input remains as native ETH on the router after the multicall returns. If the user did not include `refundETH` as a subsequent sub-call in the same multicall, that ETH is stranded and unprotected. The test suite confirms this is the expected pattern — `refundETH` must be bundled in the same multicall to recover excess ETH: [4](#0-3) 

The payments test further confirms that `refundETH` sends to any arbitrary `msg.sender` with no restriction: [5](#0-4) 

## Impact Explanation

Direct loss of user ETH principal. Any attacker or MEV bot observing the mempool or post-transaction state can call `refundETH()` and receive all stranded ETH. The victim receives nothing back. The loss equals the excess ETH the user sent above the swap's actual input amount, which can be arbitrarily large.

## Likelihood Explanation

The ETH-input multicall pattern is the documented and tested way to swap native ETH through this router. Users routinely send a round-number ETH value for a swap that consumes less, relying on `refundETH` to recover the rest. Omitting `refundETH` from the multicall is a realistic user or integrator mistake, and MEV bots actively monitor for stranded ETH on routers. The attack requires no special privileges and is repeatable.

## Recommendation

Record `msg.sender` in transient storage at the start of `multicall` and enforce in `refundETH` that only that address can call it outside of a multicall context. A simpler partial fix is to accept an explicit `recipient` parameter, though this does not fully close the window against a griefing attacker passing the victim's address:

```solidity
function refundETH(address recipient) external payable {
    uint256 balance = address(this).balance;
    if (balance > 0) _transferETH(recipient, balance);
}
```

The strongest fix is transient-storage-based depositor tracking at multicall entry.

## Proof of Concept

1. User calls `router.multicall{value: 1 ether}([exactInputSingle(tokenIn=WETH, amountIn=0.5 ether, ...)])` — no `refundETH` sub-call included.
2. `pay()` wraps exactly 0.5 ETH into WETH and sends it to the pool. The remaining 0.5 ETH stays on the router as native balance.
3. Multicall returns. Router holds 0.5 ETH.
4. Attacker calls `router.refundETH()` in a separate transaction.
5. `refundETH` executes `_transferETH(msg.sender, 0.5 ether)` — attacker receives 0.5 ETH.
6. User's 0.5 ETH is permanently lost.

The existing test `test_refundETH_sendsBalanceToCaller` at `metric-periphery/test/MetricOmmSimpleRouter.payments.t.sol` lines 74–85 already demonstrates that `refundETH` sends to any arbitrary `msg.sender` with no restriction, confirming the attack path.
