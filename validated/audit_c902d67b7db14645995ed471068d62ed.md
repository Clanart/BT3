### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization - (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

Both the EVM and StarkNet `initTransfer` entry points record the caller-supplied `amount` in the emitted `InitTransfer` event without verifying the actual number of tokens received by the bridge. For fee-on-transfer (FoT) or deflationary ERC-20 tokens, the bridge escrows fewer tokens than the event claims, but the NEAR bridge's finalization logic trusts the event amount and releases/mints the full claimed amount on the destination chain. This permanently breaks bridge collateralization.

---

### Finding Description

**EVM — `OmniBridge.sol` `initTransfer`:**

The non-bridge-token path calls `safeTransferFrom` with the caller-supplied `amount` and then immediately emits the `InitTransfer` event using that same `amount` parameter, with no before/after balance check:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol lines 407–436
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← caller-supplied; actual received may be less
);
// ...
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // ← still the original, uninspected value
    fee,
    nativeFee,
    recipient,
    message
);
```

The same flaw exists in the `customMinters` path (lines 395–403): `safeTransferFrom` to the custom minter uses `amount`, and the event still records `amount`.

**StarkNet — `omni_bridge.cairo` `init_transfer`:**

The non-bridge-token path has the identical pattern — `transfer_from` with the caller-supplied `amount`, then the `InitTransfer` event is emitted with that same `amount`:

```cairo
// starknet/src/omni_bridge.cairo lines 304–330
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
// ...
self.emit(Event::InitTransfer(InitTransfer {
    ...
    amount,   // ← original caller-supplied value, not actual received
    ...
}));
```

**NEAR finalization trusts the event amount:**

The NEAR bridge's `fin_transfer_callback` decodes the proof of the `InitTransfer` event and directly uses `init_transfer.amount` to construct the `TransferMessage` that drives token release or minting on the destination chain:

```rust
// near/omni-bridge/src/lib.rs lines 722–725
let transfer_message = TransferMessage {
    origin_nonce: init_transfer.origin_nonce,
    token: init_transfer.token,
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    // ↑ amount comes directly from the event, no independent verification
    ...
};
```

This `transfer_message.amount` is then used in `process_fin_transfer_to_near` (line 1957–1966) to call `send_tokens` and release the full claimed amount to the recipient.

---

### Impact Explanation

For every `initTransfer` call with a FoT token (e.g., a token that deducts 10% on every transfer):

- The bridge escrows 90 tokens on the source chain.
- The `InitTransfer` event records `amount = 100`.
- The NEAR bridge (or any other destination) releases/mints 100 tokens to the recipient.
- The bridge is now undercollateralized by 10 tokens per transfer.

Repeated transfers drain the bridge's reserves. When later users attempt to bridge back from NEAR to EVM, the bridge cannot fulfill the full withdrawal — their funds are permanently frozen/irrecoverable. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization"** and **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."**

---

### Likelihood Explanation

The bridge is permissionless — any user can call `initTransfer` with any ERC-20 token address. FoT tokens are a well-known, deployed token class (e.g., STA, REFLECT, various rebase tokens). No privileged access is required. The attacker simply calls `initTransfer` with a FoT token they control or that exists on-chain. The discrepancy accumulates silently with every such transfer until the bridge's reserves are exhausted.

---

### Recommendation

Apply a before/after balance check in `initTransfer` for the non-bridge-token path on both EVM and StarkNet, and use the actual received amount in the emitted event:

**EVM (`OmniBridge.sol`):**
```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived (cast to uint128) in initTransferExtension and emit
```

**StarkNet (`omni_bridge.cairo`):**
```cairo
let balance_before = IERC20Dispatcher { contract_address: token_address }
    .balance_of(get_contract_address());
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
let balance_after = IERC20Dispatcher { contract_address: token_address }
    .balance_of(get_contract_address());
let actual_received: u128 = (balance_after - balance_before).try_into().unwrap();
// Use actual_received in the emitted event
```

Alternatively, maintain an explicit token whitelist that excludes FoT and rebasing tokens.

---

### Proof of Concept

1. Deploy or use an existing FoT ERC-20 token `FOT` on an EVM chain supported by Omni Bridge, where every transfer deducts 10%.
2. Call `OmniBridge.initTransfer(FOT, 100, 0, 0, "<NEAR_recipient>", "")`.
   - Bridge receives 90 `FOT` (10% fee deducted).
   - `InitTransfer` event is emitted with `amount = 100`.
3. A relayer submits the proof of this event to the NEAR bridge via `fin_transfer`.
4. `fin_transfer_callback` reads `init_transfer.amount = 100`, denormalizes it, and calls `send_tokens` to release 100 `FOT`-equivalent tokens to the NEAR recipient.
5. The bridge is now undercollateralized by 10 tokens.
6. Repeat step 2–5 ten times: bridge holds 900 `FOT` but has issued claims for 1000.
7. The 10th user who tries to bridge back 100 tokens from NEAR to EVM finds the bridge cannot fulfill the withdrawal — their funds are permanently locked.
