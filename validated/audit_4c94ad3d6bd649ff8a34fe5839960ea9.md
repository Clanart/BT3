### Title
Non-EVM-native Cosmos bank tokens held by self-destructed EVM contracts are permanently orphaned — (File: x/evm/statedb/statedb.go)

### Summary
When an EVM contract that holds non-EVM-native Cosmos bank tokens (e.g., IBC tokens, CosmWasm bridge tokens) is self-destructed, `StateDB.Commit()` only burns the EVM denom balance. All other Cosmos bank balances at the destroyed address are left as orphaned bank entries with no owning account and no recovery path, permanently locking those funds.

### Finding Description
In `StateDB.Commit()`, the self-destruct handling path explicitly limits its burn to the EVM denom only:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
writeCache()
``` [1](#0-0) 

After `DeleteAccount` removes the `EthAccount` from the auth module, the destroyed address has no auth account. Any non-EVM-native tokens (IBC denoms, CosmWasm bridge tokens, etc.) that were held by the contract remain in the Cosmos bank module at that address but are permanently inaccessible because:

1. There is no auth account to sign a `MsgSend` to move them.
2. The EVM contract no longer exists, so no contract function can transfer them.
3. EVM contract addresses are derived from deployer+nonce or CREATE2 parameters — there is no corresponding private key that could sign a Cosmos transaction.
4. No recovery or sweep mechanism exists anywhere in the EVM module.

The `keeper.DeleteAccount` function confirms it only removes auth metadata and storage, never touching bank balances: [2](#0-1) 

The `SubBalance` / `AddBalance` path used for the EVM denom operates only on the EVM denom: [3](#0-2) 

### Impact Explanation
**High.** This is an EVM state transition bug that causes valid user funds to be permanently mis-accounted. Any non-EVM-native Cosmos bank token (IBC atom, USDC via IBC, CosmWasm bridge tokens) held by a self-destructed EVM contract is permanently frozen. The tokens exist in the bank module's supply accounting but can never be moved, transferred, or burned. This is a direct financial loss for users whose contracts hold such tokens at the time of self-destruction.

### Likelihood Explanation
**High.** Cross-chain DeFi contracts routinely receive IBC tokens. The scenario is straightforward:
1. User deploys an EVM contract.
2. IBC tokens are transferred to the contract address via `MsgSend` or IBC transfer (both valid Cosmos operations that work on any address).
3. The contract self-destructs (intentionally, via a bug, or via an attacker who exploits a vulnerability in the contract's self-destruct logic).
4. All non-EVM-native tokens are permanently orphaned.

No privileged role is required. Any unprivileged Ethereum transaction that triggers `SELFDESTRUCT` on a contract holding IBC tokens reaches this path.

### Recommendation
Before calling `DeleteAccount`, iterate over all bank balances at the destroyed address and either burn them (if the module has burn permission for those denoms) or transfer them to a designated recovery address (e.g., the community pool or a governance-controlled address). A minimal fix:

```go
// Drain ALL bank balances, not just the EVM denom.
allBalances := s.bankKeeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if coin.IsPositive() {
        // Transfer to community pool or burn if permitted.
        // At minimum, prevent orphaning by moving to a recoverable module account.
    }
}
```

### Proof of Concept
1. Deploy an EVM contract `C` at address `X` via a standard `eth_sendRawTransaction`.
2. Send IBC-transferred ATOM to address `X` via a Cosmos `MsgSend` (or IBC `MsgTransfer` with receiver = `X`).
3. Call a `selfdestruct` function on `C` via another `eth_sendRawTransaction`.
4. `StateDB.Commit()` burns only the EVM denom balance at `X` and calls `DeleteAccount(X)`.
5. Query `bank.GetAllBalances(X)` — the IBC ATOM balance is still present.
6. Attempt any transaction to move the ATOM from `X` — all fail because there is no auth account at `X` to sign, and the contract no longer exists.
7. The IBC ATOM is permanently locked. The code comment at line 813–815 of `statedb.go` explicitly acknowledges this outcome. [4](#0-3)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
