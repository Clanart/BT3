### Title
`FakeProver` Placeholder Accepts Any Proof, Enabling Unauthorized eNear Minting When `eNear.finaliseNearToEthTransfer` Is Not Paused - (File: `evm/src/eNear/contracts/FakeProver.sol`)

---

### Summary

`FakeProver` is deployed as the production proof verifier for the `eNear` ERC-20 contract on Ethereum. It unconditionally returns `true` for any proof input. The entire security model relies on `eNear.finaliseNearToEthTransfer` being paused at the `eNear` contract level — a deployment assumption that is not enforced anywhere in the `ENearProxy` code. If `eNear.finaliseNearToEthTransfer` is not paused (e.g., during the deployment window, or due to a deployment error), any unprivileged user can call it directly with crafted proof data, bypass all real verification, and mint arbitrary eNear tokens to any address.

---

### Finding Description

**Root cause — `FakeProver` is the live prover for `eNear`:**

`FakeProver` implements `INearProver` and unconditionally returns `true` for every call, ignoring both `proofData` and `blockHeight`:

```solidity
// evm/src/eNear/contracts/FakeProver.sol
contract FakeProver is INearProver {
    function proveOutcome(bytes calldata, uint64) external pure returns (bool) {
        return true;
    }
}
```

Per the README and deployment scripts, `FakeProver` is set as the prover inside the `eNear` contract itself (storage slot 5), replacing the original `NearProver`. This is the production configuration on mainnet (`0x85F17Cf997934a597031b2E18a9aB6ebD4B9f6a4`).

**Security model relies entirely on a deployment-time pause:**

The README states:
> "We will pause the `finaliseNearToEthTransfer` and `transferToNear` functions, and only `eNearProxy`, as the admin, will have the ability to call these functions."

`ENearProxy.mint()` calls `eNear.finaliseNearToEthTransfer` directly (bypassing the proxy's own `finaliseNearToEthTransfer` which has a real prover check), and the admin bypass of the pause is confirmed by the test suite:

```solidity
// evm/src/eNear/contracts/ENearProxy.sol lines 51-73
function mint(address token, address to, uint128 amount) public onlyRole(MINTER_ROLE) {
    // ...builds fakeProofData...
    currentReceiptId += 1;
    eNear.finaliseNearToEthTransfer(fakeProofData, 0);  // admin bypasses eNear pause
}
```

**No on-chain enforcement that `eNear` is paused:**

`ENearProxy` contains no assertion, no check, and no invariant that verifies `eNear.finaliseNearToEthTransfer` is paused before operating. The pause is a deployment assumption only.

**Attack path — direct call to `eNear.finaliseNearToEthTransfer`:**

If `eNear.finaliseNearToEthTransfer` is not paused (deployment window, deployment error, or any future state where the pause is not set), any unprivileged caller can:

1. Craft `proofData` in the Borsh format expected by `eNear.finaliseNearToEthTransfer` (the format is fully documented in `ENearProxy.mint()` lines 58–69, which constructs it explicitly).
2. Call `eNear.finaliseNearToEthTransfer(craftedProofData, 0)` directly.
3. `eNear` calls `FakeProver.proveOutcome(craftedProofData, 0)` → returns `true`.
4. `eNear` mints tokens to the attacker-specified address.

The `ENearProxy.finaliseNearToEthTransfer` function (which does call a real prover) is completely bypassed because the attacker calls `eNear` directly.

---

### Impact Explanation

**Critical — Direct unauthorized mint of native bridged assets (eNear/NEAR on Ethereum).**

An attacker who calls `eNear.finaliseNearToEthTransfer` directly while it is unpaused can mint an arbitrary quantity of eNear tokens to any address with no backing NEAR locked on the NEAR side. This breaks bridge collateralization entirely: eNear supply becomes unbacked, existing holders are diluted, and the attacker can immediately sell or bridge the minted tokens.

---

### Likelihood Explanation

**Medium.** The attack is reachable in two concrete scenarios:

1. **Deployment race window**: Between the moment `FakeProver` is set as `eNear`'s prover and the moment `eNear.finaliseNearToEthTransfer` is paused, there is a window where `eNear` is live with `FakeProver` and unpaused. A front-running attacker monitoring the mempool can exploit this window.

2. **Deployment error**: If the deployment procedure is not followed in the correct order (pause before or simultaneously with `FakeProver` installation), `eNear` is permanently vulnerable until corrected. The `ENearProxy` code provides no safety net.

The proof format is fully public (reconstructible from `ENearProxy.mint()` lines 58–69), so no reverse engineering is required.

---

### Recommendation

1. **Enforce the pause invariant on-chain**: Add a check in `ENearProxy.mint()` (and any other function that calls `eNear.finaliseNearToEthTransfer`) that asserts `eNear` is paused for `finaliseNearToEthTransfer` before proceeding. This converts a deployment assumption into a code-enforced invariant.

2. **Atomic deployment**: Pause `eNear.finaliseNearToEthTransfer` in the same transaction that sets `FakeProver` as the prover, eliminating the race window.

3. **Document the invariant explicitly in `ENearProxy`**: Add a `require` or `assert` that `eNear.paused(PAUSE_FINALISE_FROM_NEAR)` is true at the start of `mint()`, so any future upgrade or deployment error is caught immediately.
