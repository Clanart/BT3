### Title
Missing Class Hash Declaration Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not validate that the replacement class hash corresponds to a previously declared contract class. This is directly analogous to the external report's proxy-admin mismatch: in both cases, the authority-checking step that should gate an upgrade/replacement operation is absent, allowing the operation to succeed while silently leaving the target in a broken, unrecoverable state. Here, a contract can be "upgraded" to an undeclared class hash, permanently freezing it and any funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` processes the `replace_class` syscall, which allows a contract to swap its own class hash (i.e., upgrade its implementation). The function reads the requested new class hash from the syscall request and writes it directly into `contract_state_changes` with no check that the hash was ever declared:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment on line 898 is an explicit developer acknowledgment that this validation is missing.

The downstream consequence is fatal. When any future transaction attempts to call the contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
```

Because the undeclared class hash has no entry in `contract_class_changes`, `dict_read` returns `0`. The subsequent `find_element` call then searches for a compiled class with hash `0`:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

No such compiled class exists, so `find_element` fails. The OS cannot produce a valid proof for any block containing a call to the frozen contract. The sequencer is forced to permanently exclude all transactions targeting that contract. The contract — and all funds inside it — are irrecoverably frozen.

**Analogy to the external report:**

| External Report | This Finding |
|---|---|
| `proxyAdmin` passed as `initialOwner` to `TransparentUpgradeableProxy` | `class_hash` from `ReplaceClassRequest` written directly to state |
| Proxy internally creates a *new* `ProxyAdmin`; the passed address is not the actual admin | OS is supposed to be the authority validating declarations; that check is absent |
| `proxyAdmin.upgradeAndCall()` silently fails — proxy is stuck, unupgradeable | `replace_class(undeclared_hash)` silently succeeds — contract is stuck, uncallable |
| Impact: proxy permanently unupgradeable | Impact: contract permanently frozen, funds locked |

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract whose class hash is replaced with an undeclared hash becomes permanently uncallable. All ERC-20 balances, NFT ownership records, or protocol-controlled funds stored in that contract's storage are irrecoverably locked. The state commitment is finalized on-chain; there is no rollback mechanism.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, documented upgrade mechanism. Any contract can invoke it. The missing validation is reachable by:

1. **A malicious contract deployer** who deploys a contract that calls `replace_class(arbitrary_undeclared_hash)` — either in a callable function or triggered by a third party — to permanently freeze the contract and any deposited funds.
2. **An attacker exploiting a vulnerable legitimate contract** — if a DeFi protocol or wallet contract exposes a path (e.g., via a misconfigured upgrade function or reentrancy) that allows an external caller to supply the class hash argument, the attacker can freeze the protocol contract and all user funds within it.

The OS's missing validation is the necessary and sufficient condition for both scenarios. If the OS enforced that `class_hash` must be declared (i.e., present in `contract_class_changes`), the `replace_class` call would fail safely before any state mutation.

---

### Recommendation

Add a declaration check inside `execute_replace_class` before writing the new class hash to state. Specifically, verify that `class_hash` has a non-zero entry in `contract_class_changes` (meaning it was previously declared via a `declare` transaction):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed implicitly in `execute_entry_point` — making it explicit and enforced at the point of replacement rather than discovered only at the next call.

---

### Proof of Concept

1. Deploy contract `VictimVault` holding user funds, with an upgrade function:
   ```
   fn upgrade(new_class_hash: felt252) {
       replace_class_syscall(new_class_hash);  // no access control for PoC
   }
   ```
2. Attacker calls `VictimVault.upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared.
3. The OS executes `execute_replace_class`: no validation occurs; `contract_state_changes` is updated with `class_hash = 0xdeadbeef`.
4. Block is proven and committed. `VictimVault`'s on-chain class hash is now `0xdeadbeef`.
5. Any subsequent transaction targeting `VictimVault` causes `execute_entry_point` to call `dict_read(key=0xdeadbeef)` → returns `0` → `find_element(key=0)` fails → OS cannot prove the block.
6. The sequencer permanently excludes `VictimVault` from all future blocks. All funds are frozen.
