### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to shard A while the function's `shard_id` parameter names shard B. Because `verify_path` is position-agnostic and no guard compares `chunk.shard_id()` to the `shard_id` argument, the header is stored under `StateHeaderKey(shard_B, sync_hash)`. Subsequent `set_state_part` calls for shard B then extract shard A's `prev_state_root` from that header and install shard A's trie as shard B's state.

### Finding Description

`set_state_header` performs five validation steps:

**Step 1-2** — `validate_chunk_proofs` checks the chunk's internal hash, transaction root, and outgoing-receipts root. It never compares `chunk.shard_id()` to the caller-supplied `shard_id`. [1](#0-0) 

**Step 3a** — `verify_path` checks that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is somewhere in `sync_prev_block_header.chunk_headers_root()`. The function is purely position-agnostic: it reconstructs the root from the supplied path and item hash and compares; it does not verify which leaf index (i.e., which shard slot) the chunk occupies. [2](#0-1) [3](#0-2) 

**Step 4** — Receipt-proof validation hashes receipts with the caller-supplied `shard_id` (B). If the attacker supplies shard B's actual on-chain receipt proofs (publicly available) alongside shard A's chunk, this check passes. [4](#0-3) 

**Step 5** — `validate_state_root_node` checks the `state_root_node` against `chunk_inner.prev_state_root()`. Both values come from shard A's chunk, so they are internally consistent and the check passes. [5](#0-4) 

**Storage** — The header is then written under `StateHeaderKey(shard_B, sync_hash)`. [6](#0-5) 

**`set_state_part` consequence** — When state parts for shard B are applied, `set_state_part` reads the stored header, extracts `prev_state_root` from the embedded chunk (shard A's root), and validates every part against that root. The trie installed for shard B is therefore shard A's trie. [7](#0-6) 

### Impact Explanation

A syncing node that accepts the crafted header installs shard A's state as shard B's state. The exact corrupted value is: `shard_B.installed_state_root = shard_A.chunk.prev_state_root`. All subsequent transaction execution, receipt routing, and chunk production for shard B on that node operates against the wrong trie, producing divergent results from the rest of the network.

### Likelihood Explanation

State sync downloads headers from peers via `StateSyncDownloader::ensure_shard_header`, which forwards the peer-supplied header verbatim to `StateHeaderValidationRequest { shard_id, sync_hash, header }`. [8](#0-7) 

Any peer reachable by the syncing node can serve a crafted header. No validator or operator privilege is required. The attack is feasible whenever a node enters state sync (epoch boundary, cold start, or explicit sync trigger).

### Recommendation

Add an explicit shard-identity guard immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be inserted after line 376 (after `cloned_chunk()`) and before `validate_chunk_proofs`. [9](#0-8) 

### Proof of Concept

In a two-shard `TestEnv`:

1. Obtain the legitimate `ShardStateSyncResponseHeader` for shard 0 (`header_A`) via `get_state_response_header(shard_0, sync_hash)`.
2. Obtain shard 1's on-chain receipt proofs and substitute them into `header_A`'s `incoming_receipts_proofs` / `root_proofs` fields (or use `header_A` directly if no receipts exist for either shard in the epoch).
3. Call `set_state_header(shard_1, sync_hash, crafted_header)` on the syncing client.
4. Assert it returns `Ok(())` — demonstrating the missing shard-identity check.
5. Read back `DBCol::StateHeaders` under `StateHeaderKey(shard_1, sync_hash)` and assert the embedded chunk's `shard_id` is 0, not 1.
6. Call `set_state_part` for shard 1 with parts from shard 0's trie and assert they are accepted, confirming shard 1's state root is now shard 0's `prev_state_root`.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
