### Title
Fee Bypass via Felt Arithmetic Overflow in `compute_max_possible_fee` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function performs fee computation using Cairo felt arithmetic (modular arithmetic over the Stark prime P = 2²⁵¹ + 17·2¹⁹² + 1). No range checks are applied to the individual `ResourceBounds` fields. An attacker can craft a V3 transaction whose resource bounds values sum to exactly P (≡ 0 mod P), causing `charge_fee` to treat `max_fee` as zero and return without collecting any fee — while the transaction still has non-zero L2 gas available for execution.

---

### Finding Description

In `charge_fee`, the entire fee-collection path is gated on a single zero-check:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
```

`compute_max_possible_fee` returns a raw felt:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is in the felt field (mod P). The function asserts only `version = 3` and `n_resource_bounds = 3`; it applies **no range constraints** to `max_amount` or `max_price_per_unit`. Because these values are user-supplied (loaded from the transaction via hints and committed to by the transaction hash), an attacker can freely choose felt values that make the sum ≡ 0 (mod P).

The execution gas budget is drawn from a completely separate field:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
```

`l2_gas_bounds.max_amount` is independent of the fee sum. An attacker can set it to a legitimate non-zero value (e.g., 1,000,000) while still making `compute_max_possible_fee` return 0.

**Concrete example:**

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `1` |
| `l1_gas_bounds.max_price_per_unit` | `P − 1` |
| `l2_gas_bounds.max_amount` | `1,000,000` (execution gas) |
| `l2_gas_bounds.max_price_per_unit` | `0` |
| `l1_data_gas_bounds.max_amount` | `1` |
| `l1_data_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |

Fee sum = `1·(P−1) + 1,000,000·0 + 1·1 = P − 1 + 1 = P ≡ 0 (mod P)`.

`max_fee == 0` → `charge_fee` returns immediately. The transaction executes with 1,000,000 L2 gas and pays **zero fee**.

---

### Impact Explanation

**Direct loss of funds (Critical):** The sequencer collects no fee for the transaction. The attacker receives full execution for free, stealing the fee revenue that should flow to the sequencer/protocol.

**Network halt (High):** Because execution is free, an attacker can flood the network with zero-cost transactions at any volume. The sequencer must process each one (validate → execute → attempt fee charge → skip), consuming real computational resources with no economic disincentive. This can saturate block capacity and halt the network's ability to confirm new transactions.

---

### Likelihood Explanation

The attack requires only:
1. Deploying or controlling any standard account contract (no privilege required).
2. Constructing a V3 transaction with the crafted resource bounds above.
3. Signing it with the account's private key (the attacker controls their own account).

No leaked keys, no operator collusion, no Sybil attack, and no third-party dependency is needed. The math is trivial: choose `l1_gas_bounds.max_price_per_unit = P − (remaining_sum)` to force the total to P. The OS itself is the necessary vulnerable step — it performs no range validation on resource bounds before the fee computation.

---

### Recommendation

Apply `assert_nn_le` (range-check) constraints to each `max_amount` and `max_price_per_unit` field in `ResourceBounds` before they are used in `compute_max_possible_fee`, bounding them to `[0, 2¹²⁸ − 1]`. This ensures the product of any two bounds fits within a 256-bit integer and the sum of three such products cannot reach P. Alternatively, perform the fee computation in `Uint256` arithmetic with explicit overflow detection, mirroring the pattern already used for `calldata.amount` in `TransferCallData`.

---

### Proof of Concept

1. Attacker deploys (or reuses) any account contract on StarkNet.
2. Attacker constructs a V3 `INVOKE_FUNCTION` transaction with:
   - `l1_gas_bounds = {max_amount: 1, max_price_per_unit: P − 1}`
   - `l2_gas_bounds = {max_amount: 1_000_000, max_price_per_unit: 0}`
   - `l1_data_gas_bounds = {max_amount: 1, max_price_per_unit: 1}`
   - `tip = 0`
3. Attacker signs the transaction (the hash commits to these exact bounds).
4. Transaction is submitted; the sequencer includes it in a block.
5. The OS executes `execute_invoke_function_transaction`:
   - `get_initial_user_gas_bound` returns `1_000_000` → transaction has gas to run.
   - `__validate__` and `__execute__` run normally.
   - `charge_fee` calls `compute_max_possible_fee` → returns `P ≡ 0 (felt)`.
   - `if (max_fee == 0) { return (); }` fires → **no fee transfer is executed**.
6. The block proof is generated and accepted on L1. The attacker has executed a transaction at zero cost.

Repeating step 2–6 in a loop constitutes a free-of-charge transaction spam attack capable of halting the network. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
