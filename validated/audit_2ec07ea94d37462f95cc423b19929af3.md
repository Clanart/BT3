### Title
Stale Nonce After Nested CREATEs in `applyCall` CALL Branch Causes Incorrect `eth_simulateV1` Results - (File: `x/evm/keeper/simulate.go`)

### Summary
In `Simulator.applyCall`, after `evm.Call()` returns, the sender's nonce is unconditionally reset to `msg.Nonce+1`. When the CALL triggers nested CREATE operations (e.g., via EIP-7702 delegation), those CREATEs increment the sender's nonce inside the EVM. The unconditional overwrite at line 450 discards all nested-CREATE nonce increments, leaving the shared simulation `StateDB` with a stale, too-low nonce. Every subsequent call in the same `eth_simulateV1` block that reads or validates the sender's nonce then operates on incorrect state.

### Finding Description

`Simulator.applyCall` in `x/evm/keeper/simulate.go` manages nonces manually (no ante handler runs in simulation). Before execution it resets the sender's nonce to `msg.Nonce`:

```go
// line 432
sim.state.SetNonce(msg.From, msg.Nonce, tracing.NonceChangeUnspecified)
```

For the CALL branch, after `evm.Call()` returns, it unconditionally writes `msg.Nonce+1`:

```go
// lines 449-450
ret, leftoverGas, vmErr = evm.Call(msg.From, *msg.To, msg.Data, leftoverGas, value)
sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
```

`evm.Call()` itself does not increment the sender's nonce. However, if the called code performs CREATE operations (e.g., an EIP-7702 delegated EOA whose delegated contract calls `CREATE`), each nested CREATE increments the sender's nonce inside the EVM. After `evm.Call()` returns, the sender's nonce in `sim.state` is `msg.Nonce + 1 + nestedCreates`. Line 450 then overwrites it with `msg.Nonce+1`, silently discarding all nested-CREATE increments.

The production path in `ApplyMessageWithConfig` (`x/evm/keeper/state_transition.go`) correctly handles this for the CREATE branch:

```go
// lines 469-482
oldNonce := stateDB.GetNonce(sender)
stateDB.SetNonce(sender, msg.Nonce, tracing.NonceChangeUnspecified)
ret, _, leftoverGas, vmErr = evm.Create(...)
afterCreateNonce := stateDB.GetNonce(sender)
nestedCreates := afterCreateNonce - msg.Nonce - 1
stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
```

No equivalent reconciliation exists in `applyCall` for the CALL branch.

The `Simulator` uses a single shared `sim.state` (`*statedb.StateDB`) across all calls within a simulated block. The stale nonce persists into every subsequent call processed by `processBlock`.

### Impact Explanation

`eth_simulateV1` is a public JSON-RPC endpoint. The shared `sim.state` nonce is consumed in two ways for subsequent calls in the same block:

1. **Validation mode** (`sim.validate = true`): `applyCall` checks `sim.state.GetNonce(sender)` against `msg.Nonce` (lines 364–372). With the nonce stuck at `msg.Nonce+1` instead of the correct `msg.Nonce+1+nestedCreates`, the next call from the same sender (which correctly uses `msg.Nonce+1+nestedCreates`) is rejected with `ErrNonceTooHigh`, even though it is valid.

2. **Non-validation mode**: `sanitizeCall` auto-fills missing nonces from `state.GetNonce()` (line 511). Subsequent calls receive the wrong nonce, causing contract addresses derived via `CREATE` to be computed from the wrong nonce value.

Both paths return incorrect data to the caller from a public JSON-RPC endpoint. The nonce is consensus-critical: it determines contract addresses and replay protection. Simulation results are used by wallets, relayers, and developer tooling to predict on-chain outcomes; incorrect results mislead callers about which addresses will be created and whether subsequent transactions will succeed.

### Likelihood Explanation

The trigger requires a CALL transaction where the called code performs at least one nested CREATE on the sender's behalf. This is the standard EIP-7702 factory pattern (an EOA delegates to a factory contract and calls it to deploy child contracts). EIP-7702 is active on Prague-fork chains. The scenario is realistic and reachable by any unprivileged user submitting a multi-call `eth_simulateV1` request.

### Recommendation

After `evm.Call()` returns in the CALL branch of `applyCall`, read back the actual nonce from `sim.state` and use it instead of the fixed `msg.Nonce+1`. Mirror the reconciliation logic already present in `ApplyMessageWithConfig`'s CREATE branch:

```go
ret, leftoverGas, vmErr = evm.Call(msg.From, *msg.To, msg.Data, leftoverGas, value)
// Preserve any nonce increments from nested CREATEs triggered by the call.
actualNonce := sim.state.GetNonce(msg.From)
if actualNonce < msg.Nonce+1 {
    // evm.Call does not increment nonce itself; ensure at least msg.Nonce+1.
    sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
}
// If actualNonce > msg.Nonce+1, nested CREATEs already set the correct value; leave it.
```

### Proof of Concept

1. Deploy a contract `Factory` that calls `CREATE` once when invoked.
2. Set up an EIP-7702 delegation from EOA `A` to `Factory`.
3. Submit `eth_simulateV1` with two calls in one block, both from `A`, validation mode enabled:
   - Call 0: `{from: A, to: A, nonce: N}` — triggers `Factory.create()`, incrementing `A`'s nonce to `N+2` inside the EVM.
   - Call 1: `{from: A, to: someContract, nonce: N+2}` — the correct next nonce.
4. Observe: Call 1 is rejected with `ErrNonceTooHigh` because `sim.state.GetNonce(A)` returns `N+1` (set by line 450) instead of `N+2`.
5. Without the nested CREATE (Call 0 does not deploy), Call 1 with `nonce: N+1` succeeds — confirming the bug is specific to the nested-CREATE path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
