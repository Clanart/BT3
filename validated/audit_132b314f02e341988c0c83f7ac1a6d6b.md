Looking at the vulnerability class from the report — **unauthorized one-time setter / initialization race condition** — I need to find an analogous pattern in the StarkNet OS scope: a state-setting operation callable by an unprivileged user, executable only once per target, that permanently corrupts critical protocol state.

After reading the key execution files, I found a direct analog.

---

### Title
Unauthorized Bootstrap Declare Path Bypasses All Signature Verification, Enabling Permanent Class Hash Corruption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` contains a special "bootstrap" code path that allows any unprivileged transaction sender to permanently declare a class hash with an arbitrary `compiled_class_hash`, with zero signature verification, zero nonce enforcement, and zero fee charging. The sole "access control" is a felt-literal equality check (`sender_address == 'BOOTSTRAP'`) that any user can satisfy by setting their transaction's sender address field to that value. This is a direct analog of the `setGauge` pattern: a one-time, unauthenticated state setter that permanently corrupts a critical mapping.

---

### Finding Description

In `execute_declare_transaction`, lines 764–776:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        // Declare the class hash and skip the rest of the transaction.
        // Note that prev_value=0 enforces that a class may be declared only once.
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
```

This path:

1. **Skips `run_validate`** — no `__validate_declare__` entry point is called, so no signature is verified.
2. **Skips `check_and_increment_nonce`** — no nonce enforcement.
3. **Skips `charge_fee`** — no fee is charged.
4. **Writes `compiled_class_hash` directly into `contract_class_changes`** with `prev_value=0`, permanently binding the class hash to the attacker-chosen compiled class hash.

Every condition in the guard is fully attacker-controlled:

| Condition | Attacker control |
|---|---|
| `sender_address == 'BOOTSTRAP'` | Set `sender_address` field to the felt literal `'BOOTSTRAP'` in the tx |
| `tx_info.nonce == 0` | Set nonce to 0 |
| `tx_info.version == 3` | Set version to 3 |
| `max_possible_fee == 0` | Set all resource bounds amounts/prices to 0 |

There is no cryptographic check that the submitter is authorized to use this path. The `class_hash` is verified to be a valid Sierra class hash (via `finalize_class_hash`), but the `compiled_class_hash` is accepted as-is — there is no check that it corresponds to the correct CASM compilation of the Sierra class.

The `dict_update` with `prev_value=0` enforces that the class may be declared only once. This is the exact "one-time setter" property from the original bug: once an attacker writes a malicious `compiled_class_hash` for a given `class_hash`, the entry is permanently locked and no legitimate declare for that class hash can succeed.

---

### Impact Explanation

**Critical. Direct loss of funds.**

Attack chain:

1. Attacker crafts a malicious CASM (e.g., one that drains storage or redirects transfers) and computes its hash: `malicious_compiled_class_hash`.
2. Attacker selects a target Sierra class hash (e.g., a widely-used ERC20 or account class) and provides valid `ContractClassComponentHashes` so `finalize_class_hash` passes.
3. Attacker submits a declare transaction with `sender_address = 'BOOTSTRAP'`, `nonce = 0`, `version = 3`, all resource bounds zero, `class_hash = target_class_hash`, `compiled_class_hash = malicious_compiled_class_hash`.
4. The OS processes this via the bootstrap path, skipping all validation, and writes `target_class_hash → malicious_compiled_class_hash` into `contract_class_changes`.
5. The entry is permanent: any subsequent legitimate declare for `target_class_hash` fails because `prev_value` is no longer 0.
6. Any contract deployed with `target_class_hash` executes the malicious CASM, enabling direct theft of funds.

If the targeted class hash belongs to a critical system contract (fee token, account class), this additionally causes **High. Network not being able to confirm new transactions**.

---

### Likelihood Explanation

- The magic value `'BOOTSTRAP'` is visible in the public source code; no secret knowledge is required.
- All four guard conditions are trivially satisfiable by any transaction submitter.
- The attack requires only submitting a single declare transaction to the sequencer. Even if the sequencer's mempool rejects it (e.g., because no contract is deployed at address `'BOOTSTRAP'`), a sequencer that does not enforce this check — or a future sequencer implementation — would include it, and the OS proof would be valid.
- The OS itself performs no cryptographic check; the vulnerability is in the Cairo program that generates the proof, not in off-chain mempool filtering.
- The attack is front-running-capable: an attacker monitoring pending declare transactions can race to declare the same class hash first with a malicious compiled class hash.

---

### Recommendation

The bootstrap path must be protected by a cryptographic mechanism rather than a felt-literal equality check:

1. **Require a signature from a designated bootstrap key** embedded in the OS config (analogous to introducing an `owner` role in the Bribe constructor).
2. **Restrict the bootstrap path to a specific block number range** (e.g., only block 0), so it cannot be triggered after the system is live.
3. **Remove the bootstrap path entirely** and use a separate, out-of-band mechanism (e.g., a genesis state) to pre-populate the class hash mapping.

---

### Proof of Concept

```
1. Attacker creates malicious_casm (CASM that steals funds).
2. malicious_compiled_class_hash = hash(malicious_casm).
3. Attacker creates any valid Sierra class; target_class_hash = finalize_class_hash(component_hashes).
4. Attacker submits declare tx:
     sender_address    = 'BOOTSTRAP'   // felt literal, no deployed contract needed
     nonce             = 0
     version           = 3
     resource_bounds   = [0, 0, 0]     // max_possible_fee == 0
     class_hash        = target_class_hash
     compiled_class_hash = malicious_compiled_class_hash
5. OS enters bootstrap branch (lines 764-776), skips run_validate, check_and_increment_nonce,
   charge_fee; writes target_class_hash → malicious_compiled_class_hash into contract_class_changes.
6. dict_update prev_value=0 succeeds (class not yet declared).
7. Any future legitimate declare for target_class_hash fails: prev_value != 0.
8. Contracts deployed with target_class_hash execute malicious_casm → funds stolen.
```

This is the direct StarkNet analog of the `setGauge` front-running bug: an unauthenticated one-time setter that permanently corrupts a critical protocol mapping, with no recovery path short of a protocol upgrade.
