### Title
`execute_deploy` Syscall Always Reports Success Even When Constructor Reverts — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` in the response header after calling `deploy_contract`, regardless of whether the constructor execution succeeded or reverted. Calling contracts receive a success signal even when the deployment failed, cannot detect the failure, and may commit subsequent state changes (including fund transfers) to a contract address that has no deployed code — resulting in permanent loss of funds.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 534–555), after invoking `deploy_contract`, the response header is hardcoded to always indicate success:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `TODO` comment explicitly acknowledges that failure propagation is not implemented. The `deploy_contract` call that precedes this line can return normally even when the constructor reverts (the revert log mechanism unwinds the constructor's storage and class-hash changes), but the OS never reflects this outcome in the syscall response.

The `ResponseHeader.failure_flag` field is the only mechanism by which a calling contract can determine whether a syscall succeeded or failed. Since the OS always writes `failure_flag=0` here, the calling contract's execution — as verified by the OS — always proceeds on the assumption that deployment succeeded.

By contrast, `contract_call_helper` (the analogous path for `call_contract` and `library_call`) correctly propagates the revert status:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` path has no equivalent check.

---

### Impact Explanation

**Critical — Direct loss of funds.**

When a constructor reverts, the revert log mechanism (`handle_revert`) undoes the constructor's storage writes and the class-hash assignment for the new contract address. The deployed address ends up with `class_hash=0` — effectively no code.

However, because the calling contract sees `failure_flag=0`, it proceeds as if deployment succeeded. Any subsequent actions the calling contract takes based on this assumption — storing the address as a trusted counterparty, transferring tokens to it, granting it allowances — are committed to the global state. Funds sent to an address with `class_hash=0` are permanently frozen: no entry point exists to withdraw them, and no revert occurs to undo the transfer.

This is structurally identical to the reference report: a critical operation (constructor execution) silently fails and its effects are undone, but the surrounding transaction still succeeds and commits fund-moving state changes.

---

### Likelihood Explanation

**High.** The `deploy` syscall is a standard, unprivileged operation available to any contract. Constructor failures are a normal occurrence: they can be triggered by invalid arguments, arithmetic errors, out-of-gas conditions, or deliberate revert calls. Any protocol that deploys sub-contracts (factory patterns, proxy deployments, token launches) and acts on the deploy result without an independent verification step is vulnerable. The attacker-controlled entry path is simply submitting a transaction that calls a contract which uses the `deploy` syscall with a class whose constructor can be made to revert.

---

### Recommendation

Propagate the actual constructor outcome to `failure_flag` in the response header, mirroring the pattern used in `contract_call_helper`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

`deploy_contract` must be updated to return an `is_reverted` flag (analogous to `select_execute_entry_point_func`), and the response header must be written only after that flag is known. If the constructor reverted, the response should carry `failure_flag=1` and the constructor retdata should contain the revert reason, consistent with how `call_contract` failures are reported.

---

### Proof of Concept

1. Attacker or user submits a transaction invoking contract **A** (a factory).
2. Contract **A** calls the `deploy` syscall with class hash **C**, whose constructor unconditionally reverts (or reverts under a triggerable condition).
3. The OS executes the constructor, detects the revert, and unwinds the constructor's state changes via `handle_revert`. The new contract address ends up with `class_hash=0`.
4. The OS writes `failure_flag=0` in the `DeployResponse` (hardcoded).
5. Contract **A** reads `failure_flag=0`, concludes deployment succeeded, and executes its post-deploy logic — e.g., transferring tokens to the newly "deployed" address or recording it as a trusted vault.
6. These state changes are committed. The target address has no code.
7. Tokens transferred to the address are permanently frozen; no entry point exists to recover them. [3](#0-2)
