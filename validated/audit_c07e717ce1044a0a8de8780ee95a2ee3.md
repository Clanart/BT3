### Title
`ProofOfInclusion::valid()` Uses Self-Referential Root Instead of Trusted External Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` verifies only the internal consistency of a Merkle proof but never checks the computed root against any externally trusted root hash. The final comparison is a tautology: after the loop, `existing_hash` is always equal to `self.root_hash()` by construction. An attacker can forge a structurally valid `ProofOfInclusion` for any arbitrary `node_hash` without knowing the real tree root.

### Finding Description

The `valid()` method in `ProofOfInclusion` is the sole validation entry point exposed to callers (including the Python binding `py_valid()`):

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()      // TAUTOLOGY
}
```

After the loop body executes, `existing_hash` has been set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. Therefore `existing_hash` is always equal to the last `layer.combined_hash`. The `root_hash()` method returns exactly that same value:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // same value as existing_hash after loop
    } else {
        self.node_hash
    }
}
```

The final predicate `existing_hash == self.root_hash()` is always `true` when the loop completes without returning `false`. The function therefore only verifies that each layer's `combined_hash` is correctly derived from the previous hash and `other_hash` — it never anchors the proof to any externally trusted root.

The analog to the external report is direct: just as `isDeviatingWithBpsCheck` uses the wrong reference point (the larger value instead of the mean), `valid()` uses the wrong reference point (the proof's own last `combined_hash` instead of a caller-supplied trusted root), making the check self-referential and incomplete.

**Forged proof construction** (no tree knowledge required):

1. Choose any target `node_hash` (e.g., `SHA256(fake_key || fake_value)`).
2. For each layer, freely choose `other_hash` and `other_hash_side`; compute `combined_hash = calculate_internal_hash(prev, side, other_hash)`.
3. The resulting `ProofOfInclusion` passes `valid()` unconditionally.
4. `root_hash()` returns the attacker-chosen final `combined_hash`.

The empty-layers edge case is equally broken: a `ProofOfInclusion { node_hash: X, layers: [] }` always returns `true` from `valid()` and reports `root_hash() == X`, "proving" that `X` is the sole element of a tree.

### Impact Explanation

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion, letting untrusted input prove invalid state.**

Any DataLayer client that calls `proof.valid()` (or the Python binding `py_valid()`) without separately asserting `proof.root_hash() == known_trusted_root` will accept a forged proof of inclusion for any key-value pair. This enables an attacker to:

- Convince a DataLayer verifier that an arbitrary key-value mapping exists in a tree it does not belong to.
- Corrupt application state that depends on DataLayer inclusion proofs for authorization or data integrity decisions.

### Likelihood Explanation

The Python binding exposes `py_valid()` as the sole named validation method. The internal tests call only `proof_of_inclusion.valid()` without any root-hash cross-check, establishing a usage pattern that omits the missing guard. Any downstream Python or Rust caller following this pattern is vulnerable. The attack requires only the ability to send a crafted `ProofOfInclusion` object to a verifier — no privileged access, no key material, no tree knowledge.

### Recommendation

`valid()` must accept a trusted root hash as a parameter and compare against it instead of `self.root_hash()`:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // anchor to external trusted root
}
```

All call sites — including `py_valid()` — must be updated to supply the trusted root obtained from a verified source (e.g., the committed `MerkleBlob` root).

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side, ProofOfInclusion, ProofOfInclusionLayer};
use chia_sha2::Sha256;

fn fake_hash(seed: u8) -> Hash { [seed; 32] }

fn main() {
    // Attacker wants to "prove" node_hash = [0xAA; 32] is in some tree.
    let node_hash = fake_hash(0xAA);

    // Build one internally-consistent layer with arbitrary other_hash.
    let other_hash = fake_hash(0xBB);
    let combined = calculate_internal_hash(&node_hash, Side::Left, &other_hash);

    let proof = ProofOfInclusion {
        node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash,
            combined_hash: combined,
        }],
    };

    // valid() returns true — no real tree involved.
    assert!(proof.valid());
    // root_hash() returns the attacker-chosen combined hash.
    assert_eq!(proof.root_hash(), combined);
}
