### Title
Non-Deterministic Bridge Token Deployment via `create` Enables Reorg-Based Token Mapping Corruption and Unbacked Mint - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.deployToken` deploys bridge token proxies using the `new` keyword (EVM `create` opcode), whose address derivation depends on the OmniBridge contract address and its nonce. In a chain reorganization where two valid `deployToken` transactions are reordered, the NEAR-to-EVM token address mapping is corrupted: a `finTransfer` signed by the MPC against the pre-reorg token address mints the wrong token to the recipient, inflating one token's supply without backing while the user's locked NEAR-side assets are never redeemed.

### Finding Description

In `OmniBridge.deployToken` (lines 162â€“172), the bridge token proxy is deployed with:

```solidity
address bridgeTokenProxy = address(
    new ERC1967Proxy(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

`new ERC1967Proxy(...)` compiles to the `create` opcode. The resulting address is `keccak256(rlp(OmniBridge_address, nonce))`, where `nonce` is the OmniBridge contract's deployment nonce â€” not a function of the NEAR token ID. There is no `create2` with a deterministic salt tied to `metadata.token`.

If two valid `deployToken` transactions (for NEAR tokens X and Y) are both in the mempool and a reorg reorders them, token X lands at a different EVM address than the one the MPC observed when it signed subsequent `finTransfer` messages. Because `finTransfer` verifies the MPC signature over the full payload (including `payload.tokenAddress`) and then dispatches mint/transfer based on `isBridgeToken[payload.tokenAddress]`, a pre-reorg-signed payload referencing the old address now resolves to the wrong token contract.

### Impact Explanation

**Token mapping corruption that breaks bridge collateralization and misdirects value.**

Concrete outcome:

1. Pre-reorg: Token X deployed at address A. User locks 1 000 token X on NEAR. MPC signs `finTransfer(tokenAddress=A, amount=1000, recipient=user, destinationNonce=5)`. The `finTransfer` executes in block N; `completedTransfers[5] = true`.
2. Reorg reverts block N (and possibly the `deployToken` block).
3. In the reorged chain the attacker frontruns `deployToken(tokenX)` with a valid `deployToken(tokenY)` transaction they held in reserve â†’ token Y lands at address A (nonce N), token X lands at address B (nonce N+1). `nearToEthToken` is updated correctly for both.
4. The pre-reorg `finTransfer` payload (signed by MPC, `tokenAddress=A`) is replayed: `completedTransfers[5]` is false (reorged out), `isBridgeToken[A]` is true (token Y), so **1 000 token Y is minted to the user** instead of token X.
5. Result: token Y supply is inflated without any NEAR-side lock; the user's locked token X on NEAR is never redeemed on EVM; bridge collateralization is broken for both tokens.

### Likelihood Explanation

Low-to-medium. Requires:
- A reorg deep enough to revert the `deployToken` block (common on Polygon/Optimism where 100+ block reorgs have occurred).
- Two valid MPC-signed `deployToken` transactions simultaneously in the mempool (realistic during active token onboarding periods).
- The MPC having signed at least one `finTransfer` referencing the pre-reorg address before the reorg (possible if the MPC's finality threshold is shorter than the reorg depth).

The bridge is intended for Ethereum but the code is chain-agnostic and is already deployed or planned for L2s where reorgs are significantly more frequent.

### Recommendation

Replace `new ERC1967Proxy(...)` with a `create2` deployment using a salt derived deterministically from the NEAR token identifier:

```solidity
bytes32 salt = keccak256(abi.encodePacked(metadata.token));
address bridgeTokenProxy = address(
    new ERC1967Proxy{salt: salt}(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

This makes the EVM bridge token address a pure function of the NEAR token ID, independent of nonce ordering, eliminating the reorg attack surface. The existing `"ERR_TOKEN_EXIST"` guard already prevents double-deployment of the same NEAR token, so `create2` collision is safe.
