### Title
Unbounded `recipient` and `message` Strings in EVM `initTransfer` Enable Permanent Irrecoverable Loss of User Funds via Oversized Wormhole Payload — (`evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridge.initTransfer` and `OmniBridge.initTransfer1155` accept `recipient` and `message` as unbounded `string calldata` with no length validation. In the `OmniBridgeWormhole` extension, both strings are Borsh-encoded and published as a Wormhole message **after** the user's tokens have already been burned or locked. If the resulting payload exceeds the Wormhole guardian network's attestation size limit, no VAA is ever produced, the NEAR bridge never receives the transfer, and the user's funds are permanently and irrecoverably lost. No on-chain recovery path exists.

---

### Finding Description

`OmniBridge.initTransfer` accepts two unbounded string parameters:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    uint128 fee,
    uint128 nativeFee,
    string calldata recipient,   // ← no length check
    string calldata message      // ← no length check
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
``` [1](#0-0) 

Tokens are burned or locked **before** `initTransferExtension` is called: [2](#0-1) 

`OmniBridgeWormhole.initTransferExtension` then Borsh-encodes both strings into a single payload and publishes it to Wormhole:

```solidity
bytes memory payload = bytes.concat(
    ...
    Borsh.encodeString(recipient),   // ← arbitrary length
    Borsh.encodeString(message)      // ← arbitrary length
);
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [3](#0-2) 

`Borsh.encodeString` prepends a 4-byte little-endian length prefix and appends the raw bytes with no upper bound: [4](#0-3) 

Wormhole's `publishMessage` is a fire-and-forget event emission at the contract level; the size limit is enforced **off-chain** by the guardian network. If the payload exceeds the guardian attestation threshold, no VAA is produced. Because the token burn/lock already succeeded on-chain, there is no revert and no refund mechanism. The same unbounded issue exists in `initTransfer1155`: [5](#0-4) 

The Solana `InitTransferPayload` has the same structural gap — `recipient: String` and `message: String` with no length constraint — though Solana's 1232-byte transaction size limit provides a natural (but undocumented) guard: [6](#0-5) 

By contrast, the NEAR-side `InitTransferMsg` correctly caps the hook payload with `BoundedString<MAX_INIT_TRANSFER_MSG_LEN>` (2048 bytes) and `external_id` with `BoundedString<MAX_EXTERNAL_ID_LEN>` (64 bytes), but the `recipient` field (`OmniAddress`) is unbounded for UTXO chain variants (`Btc(String)`, `Zcash(String)`): [7](#0-6) 

The repository's own test file `zcash_stale_transfer_poc.rs` explicitly demonstrates that a 500-character Zcash unified address is accepted without rejection: [8](#0-7) 

---

### Impact Explanation

A user who calls `initTransfer` with a `recipient` or `message` string large enough to exceed the Wormhole guardian attestation limit will have their tokens permanently burned or locked on the EVM chain with no corresponding mint or release on NEAR. There is no on-chain refund, no retry, and no admin recovery path for unattested Wormhole messages. This constitutes **permanent irrecoverable loss of user funds in a bridge vault flow**, matching the Critical/High allowed impact class.

---

### Likelihood Explanation

The entry point is fully public and requires no privileged role. Any token holder can call `initTransfer`. The risk is elevated for:

- Users bridging to UTXO chains (Zcash unified addresses can legitimately exceed 300 characters).
- Integrators or SDK users who pass a destination-chain hook payload in `message` without knowing the Wormhole size ceiling.
- Malicious actors deliberately constructing an oversized call to permanently destroy their own tokens (e.g., to manipulate supply accounting or grief the protocol).

No special knowledge beyond the public ABI is required.

---

### Recommendation

Add explicit length guards at the top of `initTransfer` and `initTransfer1155` before any token movement occurs:

```solidity
uint256 constant MAX_RECIPIENT_LEN = 256;
uint256 constant MAX_MESSAGE_LEN   = 2048;

function initTransfer(..., string calldata recipient, string calldata message) external payable {
    require(bytes(recipient).length <= MAX_RECIPIENT_LEN, "recipient too long");
    require(bytes(message).length   <= MAX_MESSAGE_LEN,   "message too long");
    ...
}
```

Apply the same guard to `initTransfer1155`. Mirror the `BoundedString` pattern already used on the NEAR side for `msg` and `external_id`. [9](#0-8) 

---

### Proof of Concept

1. Attacker holds 1000 USDC bridged tokens on an EVM chain where `OmniBridgeWormhole` is deployed.
2. Attacker calls:
   ```solidity
   omniBridgeWormhole.initTransfer(
       usdcTokenAddress,
       1000e6,          // amount
       0,               // fee
       0,               // nativeFee
       string(new bytes(50_000)),  // 50 KB recipient — far above any Wormhole guardian limit
       ""
   );
   ```
3. `BridgeToken(usdcTokenAddress).burn(msg.sender, 1000e6)` executes — tokens are destroyed.
4. `_wormhole.publishMessage(...)` emits `LogMessagePublished` with a 50 KB payload.
5. Wormhole guardians observe the event but decline to sign a VAA for an oversized payload.
6. No VAA is ever produced; the NEAR bridge never calls `fin_transfer`.
7. The 1000 USDC are permanently gone with no recovery path. [10](#0-9) [11](#0-10)

    function initTransferExtension(
        address sender,
        address tokenAddress,
        uint64 originNonce,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message,

        bytes memory payload = bytes.concat(

            Borsh.encodeString(message)

        _wormhole.publishMessage{value: value}(
            wormholeNonce,
            payload,
            _consistencyLevel
