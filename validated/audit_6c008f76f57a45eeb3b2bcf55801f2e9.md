### Title
`verify_proof` Does Not Validate the Proof Variant Field, Allowing Non-SNOS Proof Facts to Pass Gateway Admission — (`crates/starknet_proof_verifier/src/proof_verifier.rs`)

---

### Summary

`verify_proof` validates `PROOF_VERSION` (index 0 of proof facts) but silently skips the variant field (index 1). Because `reconstruct_output_preimage` also drops the variant before passing the preimage to the circuit verifier, any variant value produces a cryptographically valid result. The variant is only checked later, inside the blockifier's `perform_pre_validation_stage`, which runs during block building — not at gateway admission. When `allow_client_side_proving` is enabled, an attacker can submit a transaction whose proof verifies at the gateway but whose variant is rejected by the blockifier, admitting invalid transactions into the mempool.

---

### Finding Description

**Proof facts layout** (from the Cairo constant and Rust code):

```
[PROOF_VERSION, variant, program_hash, ...task_output]
```

`verify_proof` checks only index 0:

```rust
// crates/starknet_proof_verifier/src/proof_verifier.rs  L128-L136
let expected_proof_version = PROOF_VERSION;
let actual_first = proof_facts.0.first().copied().unwrap_or_default();
if actual_first != expected_proof_version {
    return Err(VerifyProofError::InvalidProofVersion { ... });
}
``` [1](#0-0) 

`reconstruct_output_preimage` then drops both index 0 and index 1 before building the preimage fed to the circuit:

```rust
// L113-L118
// Skip PROOF_VERSION (index 0) and variant (index 1).
let task_content = &proof_facts.0[2..];
let output_size = Felt::from(...);
Ok([Felt::ONE, output_size].into_iter().chain(task_content.iter().copied()).collect())
``` [2](#0-1) 

Because the variant is absent from the output preimage, `verify_recursive_circuit` is blind to it. A proof generated for `VIRTUAL_SNOS` is equally valid for any other variant byte.

The only place the variant is actually enforced is `ProofFactsVariant::try_from` in `starknet_api`:

```rust
// crates/starknet_api/src/transaction/fields.rs  L682-L689
if *variant_marker != VIRTUAL_SNOS {
    return Err(StarknetApiError::InvalidProofFacts(format!(
        "Non-SNOS proofs are not currently supported. ..."
    )));
}
``` [3](#0-2) 

This is called from `validate_proof_facts` inside `perform_pre_validation_stage` — during block building, not during gateway admission.

The gateway's admission path calls only `run_proof_verification` → `verify_proof`:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  L411-L415
tokio::task::spawn_blocking(move || {
    starknet_proof_verifier::verify_proof(proof_facts, proof)
})
.await
.expect("proof verification task panicked")?;
``` [4](#0-3) 

None of the gateway's stateless checks (`validate_client_side_proving_allowed`, `validate_proof_facts_and_proof_consistency`, `validate_tx_size`) inspect the variant field. [5](#0-4) 

**Attack path (when `allow_client_side_proving = true`):**

1. Obtain any existing valid `(proof_facts, proof)` pair where `proof_facts[1] == VIRTUAL_SNOS`.
2. Construct new proof facts with `proof_facts[1]` replaced by an arbitrary felt (e.g., `0x0`).
3. Compute the new transaction hash — `get_invoke_transaction_v3_hash` hashes the full proof facts vector, so the hash changes.
4. Sign the new hash with the attacker's key.
5. Submit. The gateway calls `verify_proof`: `PROOF_VERSION` matches, the circuit verifier reconstructs the preimage from `proof_facts[2..]` (unchanged), and the proof verifies.
6. The transaction enters the mempool.
7. During block building, `perform_pre_validation_stage` calls `ProofFactsVariant::try_from`, finds `variant != VIRTUAL_SNOS`, and drops the transaction.

The transaction hash calculation that binds the variant:

```rust
// crates/starknet_api/src/transaction_hash.rs  L399-L403
if !transaction.proof_facts().0.is_empty() {
    let proof_facts_hash =
        HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
    hash_chain = hash_chain.chain(&proof_facts_hash);
}
``` [6](#0-5) 

---

### Impact Explanation

When client-side proving is enabled, the gateway admits transactions whose proof facts carry an unsupported or malformed variant. These transactions consume mempool slots and force the batcher to run `perform_pre_validation_stage` before discarding them. Because the attacker only needs to re-sign (not re-prove), the cost per invalid transaction is negligible, enabling sustained mempool pollution and batcher resource exhaustion. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

- Requires `allow_client_side_proving = true` (a production configuration for client-side proving deployments).
- Requires possession of any previously valid proof — reuse is sufficient; no new proving work is needed.
- Signing cost is trivial.
- No privileged access required.

---

### Recommendation

Add a variant check inside `verify_proof` immediately after the `PROOF_VERSION` check:

```rust
// After validating PROOF_VERSION at index 0, validate variant at index 1.
let actual_variant = proof_facts.0.get(1).copied().unwrap_or_default();
if actual_variant != VIRTUAL_SNOS {
    return Err(VerifyProofError::InvalidProofVariant {
        expected: VIRTUAL_SNOS,
        actual: actual_variant,
    });
}
```

Alternatively, call `ProofFactsVariant::try_from` at the start of `verify_proof` and reject anything that is not `ProofFactsVariant::Snos`. This mirrors the pattern already used in the blockifier and ensures the gateway and the blockifier enforce the same invariant on the variant field.

### Proof of Concept

```
1. Let (pf_valid, proof) be a valid client-side proving transaction where
   pf_valid = [PROOF_VERSION, VIRTUAL_SNOS, program_hash, ...output].

2. Construct pf_bad = [PROOF_VERSION, 0x0, program_hash, ...output].
   (variant changed; everything else identical)

3. Compute tx_hash_bad = poseidon(INVOKE || version || sender ||
       tip_resource_bounds_hash || ... || poseidon(pf_bad)).

4. Sign tx_hash_bad with the attacker's private key → sig_bad.

5. Submit InvokeV3 { proof_facts: pf_bad, proof: proof, signature: sig_bad, ... }.

6. Gateway:
   - validate_client_side_proving_allowed: proof_facts non-empty, allowed → OK
   - validate_proof_facts_and_proof_consistency: both present → OK
   - verify_proof:
       * pf_bad[0] == PROOF_VERSION → OK
       * reconstruct_output_preimage uses pf_bad[2..] == pf_valid[2..] → same preimage
       * verify_recursive_circuit(proof, preimage) → OK (proof was generated for this preimage)
   → Transaction admitted to mempool.

7. Batcher calls perform_pre_validation

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }

                has_proof_facts,
