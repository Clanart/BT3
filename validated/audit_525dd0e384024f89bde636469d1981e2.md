The vulnerability is real. Let me trace the exact path.

**`refundETH()` access control:** The function is `external payable` with zero access control — it unconditionally sends `address(this).balance` to `msg.sender`. [1](#0-0) 

**ETH stranding mechanism:** The `pay()` function, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH and leaves any excess in the router. [2](#0-1) 

**Payable entry points:** `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, and `multicall` are all `payable`. A user calling `exactOutputSingle{value: X}` where `X > actual amountIn` leaves `X - amountIn` ETH stranded in the router after the transaction. [3](#0-2) 

**`receive()` does NOT prevent stranding:** It only blocks direct ETH pushes from non-WETH addresses. ETH entering via `msg.value` on payable functions is unaffected. [4](#0-3) 

**Correction on the "mid-multicall" framing:** The claim that an attacker can interleave `refundETH()` *mid-multicall* in a separate transaction is incorrect — `multicall` uses `delegatecall` in a single atomic transaction, so no external call can interleave. However, the **cross-transaction** attack is fully valid.

### Title
Unguarded `refundETH()` allows any caller to steal ETH stranded in the router between transactions — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.refundETH()` has no access control and sends the router's entire ETH balance to `msg.sender`. ETH is routinely stranded in the router when users call payable swap functions (especially `exactOutputSingle` / `exactOutput`) with `msg.value` exceeding the actual swap cost. Any attacker can call `refundETH()` in a subsequent transaction to steal that ETH.

### Finding Description
`pay()` wraps only the exact `value` of ETH required by the pool callback:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // only `value`, not nativeBalance
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [5](#0-4) 

Any `msg.value` above `value` remains in the router after the transaction. For exact-output swaps, the caller cannot know the precise `amountIn` before execution, so sending excess ETH is the normal pattern. The intended recovery mechanism is to append `refundETH()` as the last call in a `multicall`. But `refundETH()` itself is unrestricted:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // sends to ANY caller
    }
}
``` [1](#0-0) 

An attacker monitoring the mempool can back-run any swap transaction that leaves ETH in the router and call `refundETH()` to drain it.

### Impact Explanation
Direct theft of user ETH. Any ETH stranded in the router after a swap (excess `msg.value`) is immediately claimable by any address. The router is a shared contract; stranded ETH from one user is visible to all.

### Likelihood Explanation
`exactOutputSingle` and `exactOutput` are the primary affected paths — callers routinely over-provision ETH because the exact input is unknown before execution. The test suite itself demonstrates this pattern (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth` sends `2 ether` for a `1_000` unit swap). [6](#0-5) 

### Recommendation
Restrict `refundETH()` so it can only be called from within a `multicall` context (e.g., check that `msg.sender == address(this)` when not called directly), or track the original `msg.sender` of the outermost `multicall` in transient storage and only allow that address to receive the refund.

### Proof of Concept
```solidity
// 1. User A calls exactOutputSingle with excess ETH
router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
    ...,
    amountOut: someAmount,
    amountInMaximum: 2 ether,   // actual cost = 1 ether
    tokenIn: address(weth),
    ...
}));
// After tx: router holds 1 ether of stranded ETH

// 2. Attacker (separate tx) drains it
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance, 1 ether); // attacker stole user A's ETH
```
