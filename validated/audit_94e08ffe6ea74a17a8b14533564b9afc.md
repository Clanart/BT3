### Title
Missing Class Existence Check in `execute_replace_class` Allows Permanent Contract Bricking - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS program accepts an arbitrary `class_hash` from a user-supplied syscall request and commits it to the contract state without verifying that the class has been declared. This is the direct analog of the Ladle `delegatecall` bug: just as Solidity's `delegatecall` silently succeeds against a non-existent contract, the OS silently accepts a `replace_class` to a non-existent class hash, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the OS reads `class_hash` directly from the syscall request and immediately writes it into `contract_state_changes` via `dict_update`, with no check that the class hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-inserted TODO at line 898 explicitly acknowledges the missing check. [1](#0-0) 

After this state update is committed, any subsequent call to the contract enters `execute_entry_point`, which reads the (now invalid) `class_hash` from `contract_class_changes` and calls `find_element` to locate the compiled class in `compiled_class_facts_bundle`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

Because the class hash is undeclared, `compiled_class_hash` resolves to 0 (the default dict value), and `find_element` cannot locate it in the bundle. The contract becomes permanently uncallable. The same missing check exists in the deprecated path (`deprecated_execute_syscalls.cairo`, `execute_replace_class`, lines 307–329), which also writes an arbitrary class hash into state without validation. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds ERC-20 token balances, ETH, STRK, or other assets can be permanently bricked by calling `replace_class` with an undeclared class hash. Once the state is committed, no entry point of the contract can ever execute again. All assets stored in the contract's storage (token balances, NFT ownership records, vault deposits, etc.) become permanently inaccessible. There is no recovery path because the class hash update is irreversible once the block is proven and finalized.

---

### Likelihood Explanation

**Medium-High.** The attacker-controlled entry path requires only the ability to deploy a contract or to call a contract that exposes `replace_class` to an external caller. No privileged role is needed. A malicious contract deployer can intentionally deploy a contract that immediately calls `replace_class` with `class_hash = 1` (or any undeclared felt), bricking itself and any funds sent to it. Additionally, a buggy or exploited contract that exposes an unguarded `replace_class` call can be triggered by any unprivileged transaction sender. The TODO comment confirms the developers are aware the check is absent.

---

### Recommendation

Before committing the `dict_update` in `execute_replace_class`, verify that `class_hash` exists in `contract_class_changes` (i.e., has a non-zero compiled class hash mapping). Concretely, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the result is non-zero before proceeding. This mirrors the check already implicitly performed in `execute_entry_point` — it should be enforced eagerly at the syscall boundary so the invalid state is never written.

---

### Proof of Concept

1. Attacker deploys contract `VaultBrick` that holds user funds and exposes:
   ```
   fn brick_self() {
       replace_class(class_hash: 0xdeadbeef);  // undeclared hash
   }
   ```
2. Users deposit funds into `VaultBrick`.
3. Attacker (or anyone) calls `brick_self()`.
4. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`.
5. No existence check is performed; `dict_update` writes `class_hash=0xdeadbeef` into `contract_state_changes`. [4](#0-3) 
6. Block is proven and finalized. The state commitment now records `VaultBrick.class_hash = 0xdeadbeef`.
7. Any subsequent call to `VaultBrick` reaches `execute_entry_point`, which reads `compiled_class_hash = 0` (no mapping for `0xdeadbeef`), and `find_element` fails to locate the compiled class. [5](#0-4) 
8. All user funds in `VaultBrick` are permanently frozen with no recovery path.
