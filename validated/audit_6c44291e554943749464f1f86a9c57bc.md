### Title
Fixed Gas Limit for `ft_transfer` in `send_tokens` Can Permanently Lock User Funds — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `send_tokens` helper in the Omni Bridge NEAR contract uses a hardcoded `FT_TRANSFER_GAS = 5 TGas` when calling `ft_transfer` on non-deployed (locked/native) tokens. If the token's transfer logic consumes more than 5 TGas, the cross-contract call fails. Because the transfer is already recorded as finalized before `send_tokens` executes, the user cannot replay or recover the transfer, resulting in permanently locked funds.

---

### Finding Description

In `near/omni-bridge/src/lib.rs`, the constant `FT_TRANSFER_GAS` is defined as a fixed 5 TGas: [1](#0-0) 

Inside `send_tokens`, when the token is neither a deployed bridge token nor wNEAR, and no `msg` is attached, the code takes the following branch: [2](#0-1) 

This is the only branch in `send_tokens` that does **not** dynamically compute available gas. Every other branch either:
- Computes `ft_transfer_call_gas` from remaining prepaid gas (lines 2063–2067), or
- Guards with `MIN_FT_TRANSFER_CALL_GAS` (lines 2089–2091, 2108–2110).

The `ft_transfer` branch silently uses the fixed 5 TGas constant with no floor check and no dynamic adjustment. [3](#0-2) 

The `fin_transfer` entry point chains proof verification with `fin_transfer_callback` at `FIN_TRANSFER_CALLBACK_GAS = 250 TGas`: [4](#0-3) 

Inside `fin_transfer_callback`, the transfer ID is added to `finalised_transfers` (replay-prevention) before `send_tokens` is dispatched. Once that state write is committed, the transfer cannot be re-submitted. If the subsequent `ft_transfer` call panics or runs out of gas, `send_tokens_callback` is invoked but the finalization record already exists, so no retry path is available to the user.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

Any NEP-141 token whose `ft_transfer` implementation requires more than 5 TGas (e.g., tokens with transfer hooks, fee-on-transfer logic, rebasing mechanics, or non-trivial bookkeeping) will fail silently at the cross-contract call level. The bridge marks the transfer finalized, the relayer collects its fee, and the bridged tokens remain locked inside the bridge contract forever with no on-chain recovery path.

---

### Likelihood Explanation

**Medium.** The Omni Bridge is explicitly designed to support arbitrary NEP-141 tokens. The NEP-141 standard does not cap the gas a `ft_transfer` implementation may consume. Tokens with hooks or complex accounting are common in DeFi. A user who bridges such a token to NEAR via the standard `fin_transfer` flow will reliably trigger this condition without any malicious intent.

---

### Recommendation

Apply the same dynamic gas computation already used for `ft_transfer_call` to the `ft_transfer` branch:

```rust
} else if msg.is_empty() {
    let ft_transfer_gas = env::prepaid_gas()
        .saturating_sub(env::used_gas())
        .saturating_sub(SEND_TOKENS_CALLBACK_GAS)
        .min(FT_TRANSFER_GAS_MAX);   // cap at a reasonable upper bound

    require!(
        ft_transfer_gas >= MIN_FT_TRANSFER_GAS,
        BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_gas).as_ref()
    );

    ext_token::ext(token)
        .with_attached_deposit(ONE_YOCTO)
        .with_static_gas(ft_transfer_gas)
        .ft_transfer(recipient, amount, None)
```

Alternatively, expose a user-configurable gas parameter in `FinTransferArgs` for the final token delivery step, mirroring the fix applied in the referenced upstream pull request.

---

### Proof of Concept

1. A user initiates a bridge transfer of a NEP-141 token whose `ft_transfer` implementation requires ~20 TGas (e.g., a token with a transfer hook that updates an index).
2. A relayer submits the proof by calling `fin_transfer` on the NEAR bridge contract.
3. `fin_transfer_callback` executes: the transfer ID is inserted into `finalised_transfers` (replay prevention committed to state).
4. `send_tokens` is called; it dispatches `ft_transfer` with `FT_TRANSFER_GAS = 5 TGas`.
5. The token contract's `ft_transfer` runs out of gas and panics.
6. `send_tokens_callback` is invoked; it observes a failed promise result but cannot re-submit the transfer because `finalised_transfers` already contains the ID.
7. The relayer's fee is paid; the recipient receives nothing; the tokens are permanently locked in the bridge contract.
8. No `retryMessageExecution`-equivalent exists in the Omni Bridge NEAR contract to recover from this state. [1](#0-0) [2](#0-1) [5](#0-4)
