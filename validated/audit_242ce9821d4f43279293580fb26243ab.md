Audit Report

## Title
Excess ETH stranded on `MetricOmmSimpleRouter` after a WETH swap is claimable by any caller via `refundETH()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
When a user sends more ETH than `amountIn` to a payable swap function (e.g., `exactInputSingle`), `pay()` wraps only the exact `amountIn` into WETH and leaves the remainder as raw ETH on the router. Because `refundETH()` has no access control and unconditionally transfers the full ETH balance to `msg.sender`, any third party can call it in a subsequent transaction to drain the stranded ETH. The user permanently loses the excess.

## Finding Description
**Root cause — `pay()` wraps only `value`, leaving excess ETH on the contract:** [1](#0-0) 

When `token == WETH` and `nativeBalance >= value`, exactly `value` wei is deposited into WETH and forwarded to the pool. Any `nativeBalance - value` remainder stays as raw ETH on the router with no further handling.

**`refundETH()` has no access control and sends to `msg.sender`:** [2](#0-1) 

The function is `external payable` with no `onlyOwner`, no deadline, and no recipient parameter — it unconditionally transfers the full ETH balance to whoever calls it.

**`multicall()` performs no post-loop ETH sweep:** [3](#0-2) 

The loop delegates each call and returns. Any ETH not consumed by the swap steps persists on the contract after the multicall transaction completes.

**`receive()` does not prevent ETH accumulation via payable calls:** [4](#0-3) 

The `NotWETH` guard only blocks plain ETH transfers; it does not affect `msg.value` attached to payable function calls like `exactInputSingle` or `multicall`.

**Exploit path:**
1. User calls `multicall([exactInputSingle_calldata]){value: 2 ether}` with `params.amountIn = 1 ether`, `params.tokenIn = WETH`.
2. Inside `_justPayCallback`, `pay()` is called with `value = 1 ether`. Since `nativeBalance (2e18) >= value (1e18)`, it deposits 1 ether into WETH and transfers it to the pool. The remaining 1 ether stays on the router.
3. The multicall returns. 1 ether is stranded on the contract.
4. An attacker (separate EOA, separate tx) calls `MetricOmmSimpleRouter.refundETH()`. `address(this).balance == 1 ether`, so `_transferETH(msg.sender, 1 ether)` sends the attacker 1 ether.

Note that `unwrapWETH9` and `sweepToken` — the analogous helpers — both accept an explicit `recipient` parameter, making this omission in `refundETH()` inconsistent with the rest of the contract's design. [5](#0-4) 

## Impact Explanation
Direct loss of user principal. Any ETH sent in excess of `amountIn` during a WETH swap that is not followed by a `refundETH` call in the same multicall is permanently claimable by an arbitrary third party. The loss is proportional to the excess ETH sent and requires no attacker preconditions beyond calling a public function. This meets the Sherlock Medium threshold for direct loss of user funds.

## Likelihood Explanation
Sending slightly more ETH than `amountIn` is a standard user pattern for slippage tolerance. Omitting `refundETH` from a multicall is a realistic mistake, especially for integrators or front-ends constructing calldata programmatically. MEV bots routinely monitor contract ETH balances and can call `refundETH()` in the same block as the victim's transaction, making exploitation immediate and repeatable with no preconditions.

## Recommendation
Change `refundETH()` to accept an explicit `recipient` parameter, consistent with `unwrapWETH9` and `sweepToken`:

```solidity
function refundETH(address recipient) external payable {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(recipient, balance);
    }
}
```

This ensures the caller must explicitly name the beneficiary, preventing a third party from redirecting stranded ETH to themselves.

## Proof of Concept
```
1. User calls multicall([exactInputSingle_calldata]){value: 2 ether}
   - params.tokenIn = WETH, params.amountIn = 1 ether
   - pay() branch: nativeBalance(2e18) >= value(1e18)
     → deposit 1 ether into WETH, transfer WETH to pool
     → 1 ether remains as ETH on the router
   - multicall returns; 1 ether is stranded on the contract

2. Attacker (separate EOA, separate tx) calls:
   MetricOmmSimpleRouter.refundETH()
   - address(this).balance == 1 ether
   - _transferETH(msg.sender, 1 ether) → attacker receives 1 ether

3. Assert: attacker.balance increased by 1 ether; user's excess ETH is gone.
```
