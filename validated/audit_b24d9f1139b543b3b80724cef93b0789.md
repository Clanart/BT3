### Title
EIP-7702 Authorization Chain-ID Zero Wildcard Enables Cross-Chain/Cross-Fork Replay of EOA Code-Delegation Signatures - (File: x/evm/keeper/set_code_authorizations.go)

### Summary
Ethermint's `validateAuthorization` function explicitly permits EIP-7702 `SetCodeAuthorization` tuples whose `ChainID` field is zero. A zero chain ID is treated as a wildcard that matches any chain. An authorization signed with `ChainID = 0` is valid on every Ethermint deployment (and on every Ethereum-compatible chain), so a single signed authorization can be replayed by any third party on any chain where the authority account's nonce still matches, forcing arbitrary code delegation onto the victim's EOA without their per-chain consent.

### Finding Description

`validateAuthorization` in `x/evm/keeper/set_code_authorizations.go` contains the following chain-ID check:

```go
// Verify chain ID is null or equal to current chain ID.
if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

The guard is only triggered when `auth.ChainID` is **non-zero** and **mismatches** the current chain. When `auth.ChainID` is zero the entire check is skipped and the authorization is accepted unconditionally on any chain.

This is the direct analog of the Term Finance bug: the authentication token (here, the EIP-7702 authorization signature) does not bind to a specific chain, so it can be replayed on any chain where the nonce still matches.

The same permissive logic is replicated in the gRPC access-list helper `getAccessListExcludes` in `x/evm/keeper/grpc_query.go`:

```go
invalidChainID := !auth.ChainID.IsZero() && auth.ChainID.CmpBig(cfg.ChainConfig.ChainID) != 0
```

The behavior is intentional per EIP-7702 (chain-ID 0 is a spec-defined wildcard), but it creates a concrete cross-chain replay surface that is unique to Ethermint's multi-chain deployment model.

**Attack scenario:**

1. Alice signs an EIP-7702 authorization with `ChainID = 0` and `Nonce = N` on Ethermint chain A (e.g., to delegate to a trusted contract she controls).
2. The authorization is broadcast and consumed on chain A; Alice's nonce advances to N+1.
3. Ethermint is also deployed as chain B (a testnet, a fork, or a sister chain) where Alice's account nonce is still N (she has never transacted there).
4. An attacker constructs a new EIP-7702 `SetCodeTx` on chain B, embedding Alice's original signed authorization (same `ChainID = 0`, same `Nonce = N`, same `R/S/V`), but pointing `auth.Address` to an **attacker-controlled contract**.
5. `validateAuthorization` on chain B passes: `ChainID.IsZero()` is true → chain-ID check is skipped; nonce matches; signature is valid.
6. Alice's EOA on chain B now has its code set to the attacker's delegation bytecode. Any subsequent call to Alice's address on chain B executes the attacker's contract in Alice's context, giving the attacker full access to Alice's funds and storage on chain B.

The same replay is possible across Ethermint forks, testnets, or any EVM-compatible chain that accepts chain-ID-0 authorizations.

### Impact Explanation

This is a **High** impact finding matching the allowed scope:

> *Ethereum transaction, EIP-712/EIP-7702 authorization, nonce, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation.*

A successful replay installs attacker-controlled delegation code on a victim's EOA. Any call to that EOA then executes the attacker's contract in the victim's context, enabling:
- Theft of all EVM-denom funds held by the victim's EOA on the target chain.
- Arbitrary storage writes under the victim's address.
- Impersonation of the victim in any protocol that trusts the victim's address.

### Likelihood Explanation

- **Realistic precondition:** The victim must have signed a chain-ID-0 authorization at some point. EIP-7702 wallets and tooling (including the integration test helper `generate_signed_auth` in `tests/integration_tests/eip7702.py`) default to using the current chain's ID, but chain-ID-0 is a valid and documented option that users or libraries may choose for "universal" authorizations.
- **Attacker effort:** Zero privileged access required. The attacker only needs the victim's signed authorization bytes (observable on-chain from chain A) and a matching nonce on chain B.
- **Multi-chain exposure:** Ethermint is explicitly designed to be deployed as multiple independent chains (testnets, mainnets, forks). The nonce divergence between chains is the normal state for most accounts.

### Recommendation

1. **Reject chain-ID-0 authorizations at the protocol level.** Change `validateAuthorization` to require a non-zero chain ID that matches the current chain:

```go
// Require chain ID to be present and match the current chain.
if auth.ChainID.IsZero() || auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

2. If chain-ID-0 wildcard support is intentionally desired for compatibility with the EIP-7702 spec, document the cross-chain replay risk prominently and advise users never to sign chain-ID-0 authorizations.

3. Apply the same fix to `getAccessListExcludes` in `x/evm/keeper/grpc_query.go` for consistency.

	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
