### Title
Missing Validation of `class_hash` in `execute_replace_class` Allows Permanent Contract Bricking - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary `class_hash` value from the caller without verifying it corresponds to a previously declared class. A contract can call `replace_class` with an undeclared or zero class hash, permanently setting its own class to an invalid value. Any subsequent call to that contract will fail at the OS level, permanently freezing all funds held by the contract. The missing check is explicitly acknowledged by a developer TODO comment in the code.

### Finding Description
In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

There are two missing sanity checks:
1. No check that `class_hash != 0`.
2. No check that `class_hash` exists in `contract_class_changes` (i.e., has been declared via a `declare` transaction).

By contrast, the `execute_declare_transaction` flow in `transaction_impls.cairo` enforces `assert_not_zero(compiled_class_hash)` and uses `prev_value=0` to ensure a class is declared only once, establishing a registry of valid class hashes. [2](#0-1) 

The `replace_class` syscall bypasses this registry entirely.

### Impact Explanation
When a contract's `class_hash` is set to an undeclared value (e.g., `0` or any arbitrary felt), the OS cannot resolve the class during subsequent entry point execution. All future calls to the contract will fail at the class lookup stage. Since the state update is committed on-chain, the contract is permanently bricked. Any ERC-20 tokens, ETH, or STRK held in the contract's storage become permanently inaccessible — a **Critical: Permanent freezing of funds**.

### Likelihood Explanation
The `replace_class` syscall is callable by any contract from within its own execution context. An unprivileged user can:
1. Deploy a contract (class declarer / contract deployer role — no privilege required).
2. Send funds to the contract.
3. Invoke a function that calls `replace_class(0)` or `replace_class(<undeclared_hash>)`.

The OS accepts the syscall, commits the invalid class hash to state, and the contract is permanently bricked. No privileged role, leaked key, or operator cooperation is required. The developer TODO comment confirms the check is known to be absent, not intentionally omitted.

### Recommendation
Before committing the new class hash in `execute_replace_class`, add the following sanity checks:

1. **Non-zero check**: Assert `class_hash != 0`.
2. **Declared class check**: Verify that `class_hash` exists in `contract_class_changes` (i.e., its compiled class hash is non-zero), analogous to how `execute_declare_transaction` uses `prev_value=0` to enforce uniqueness.

This mirrors the pattern already used in `execute_declare_transaction` and resolves the acknowledged TODO.

### Proof of Concept

1. Attacker declares and deploys a contract `VictimContract` that holds funds and exposes a function `brick_self()` which calls the `replace_class` syscall with `class_hash = 0`.
2. Users deposit funds into `VictimContract` (or the attacker self-funds it).
3. Attacker submits an invoke transaction calling `brick_self()`.
4. The OS executes `execute_replace_class`:
   - `class_hash = 0` is read from the request. [3](#0-2) 
   - No validation is performed (the TODO is unimplemented). [4](#0-3) 
   - `StateEntry(class_hash=0, ...)` is written to `contract_state_changes`. [5](#0-4) 
5. The block is proven and committed. `VictimContract`'s class hash is now `0` on-chain.
6. Any subsequent call to `VictimContract` fails — the OS cannot find a compiled class for hash `0`.
7. All funds in `VictimContract` are permanently frozen with no recovery path.
