Audit Report

## Title
Stranded native ETH on the router is silently consumed to subsidize a later user's WETH input payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` with no attribution to the current caller or transaction. Any ETH left on the router from a prior user who sent excess `msg.value` without calling `refundETH()` is silently consumed to partially fund a subsequent user's WETH input, causing direct loss of the prior user's principal and giving the subsequent user an unearned discount.

## Finding Description
In `pay()` at `PeripheryPayments.sol` L73–84, when `token == WETH` and `payer != address(this)`, the function reads `nativeBalance = address(this).balance` — the router's total ETH balance — with no check that this ETH was deposited by the current `payer` in the current transaction:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

**Stranding path:** `exactInputSingle` is `external payable`. A user calls `exactInputSingle{value: X}(...)` with WETH as `tokenIn` and `amountIn < X`. `pay()` fires with `value = amountIn`; since `nativeBalance (= X) >= amountIn`, it deposits only `amountIn` and the remaining `X - amountIn` ETH stays on the router. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH transfers — it does not apply to ETH arriving as `msg.value` in a payable function call. Without an explicit `refundETH()` call, the surplus is permanently stranded.

**Exploitation path:** An attacker calls `exactInputSingle(...)` (zero `msg.value`) with WETH as `tokenIn` and `amountIn = V`. `pay()` sees `nativeBalance = stranded ETH > 0` and `< V`, enters the partial-native branch, wraps and forwards the victim's ETH to the pool, then pulls only `V - nativeBalance` WETH from the attacker. The attacker pays less WETH than the swap requires; the victim's ETH is consumed.

No existing guard prevents this: the transient callback context stores only the pool, mode, payer address, and token — it does not record how much ETH the current transaction deposited, and `pay()` never consults `msg.value`.

## Impact Explanation
Direct loss of user principal: the victim permanently loses the ETH they sent to the router in excess of their `amountIn`. The attacker receives a WETH-input discount equal to the stranded amount. This satisfies the contest's "direct loss of user principal" threshold at Medium severity.

## Likelihood Explanation
Any user who calls a payable swap entry point with excess ETH and omits `refundETH()` creates the precondition. The test suite itself demonstrates the pattern (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth` shows `refundETH()` is a separate, optional step). An attacker can monitor the router's ETH balance on-chain and immediately follow with a WETH swap to drain it. The condition is repeatable and requires no privileged access.

## Recommendation
Track how much ETH the current transaction deposited — e.g., record `msg.value` in transient storage at the top of each payable entry point — and cap the native ETH consumed in `pay()` to that recorded amount. Alternatively, pass the caller-supplied ETH amount explicitly through the call stack rather than reading the ambient `address(this).balance`. The simplest safe fix: only consume native ETH when `payer == msg.sender` and limit consumption to `min(nativeBalance, msg.value_remaining)` tracked per-transaction in transient storage.

## Proof of Concept
```solidity
// Step 1: victim strands ETH on the router
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    tokenIn: address(weth),
    amountIn: 0.4 ether,   // only 0.4 ETH consumed; 0.6 ETH stranded
    // ... no refundETH() call
}));
// address(router).balance == 0.6 ether

// Step 2: attacker exploits stranded ETH
uint256 wethBefore = weth.balanceOf(attacker);
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: address(weth),
    amountIn: 1 ether,     // pay() sees nativeBalance=0.6 > 0, < 1 ETH
    // ...
}));
// attacker's WETH spent == 0.4 ether (not 1 ether)
// victim's 0.6 ETH is gone
assert(wethBefore - weth.balanceOf(attacker) == 0.4 ether);
assert(address(router).balance == 0);
```
