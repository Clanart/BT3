### Title
Missing Upper Bound on `max_price_per_unit` in `pack_resource_bounds` Enables Transaction Hash Collisions — (File: `transaction_hash/transaction_hash.cairo`)

---

### Summary

`pack_resource_bounds` in `transaction_hash.cairo` enforces only a non-negativity check (`assert_nn`) on `max_price_per_unit`, with no upper bound of `2^128 - 1`. Because the packing formula places `max_price_per_unit` in the low 128 bits of a 256-bit layout, any value `≥ 2^128` bleeds into the `max_amount` field, producing a packed felt identical to a different `(max_amount, max_price_per_unit)` pair. This is the direct Cairo analog of the `wExp(x)` missing strict upper bound: the guard is present but not tight enough, allowing the arithmetic to silently produce a colliding result.

---

### Finding Description

`pack_resource_bounds` at line 103–108 of `transaction_hash/transaction_hash.cairo`:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // ✓ bounded
    assert_nn(resource_bounds.max_price_per_unit);            // ✗ only ≥ 0, NOT < 2^128
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
``` [1](#0-0) 

The intended bit layout of the packed felt is:

| bits | field |
|------|-------|
| [0, 127] | `max_price_per_unit` |
| [128, 191] | `max_amount` |
| [192, …] | `resource` |

This layout is only a bijection when `max_price_per_unit < 2^128`. When `max_price_per_unit ≥ 2^128`, the low bits overflow into the `max_amount` region. Concretely, for any valid transaction T1 with `max_amount = A ≥ 1` and `max_price_per_unit = P < 2^128`, a second transaction T2 with `max_amount = A − 1` and `max_price_per_unit = P + 2^128` produces an **identical** packed felt:

```
T1: (resource * 2^64 + A)     * 2^128 + P
T2: (resource * 2^64 + A - 1) * 2^128 + (P + 2^128)
  = (resource * 2^64 + A - 1) * 2^128 + P + 2^128
  = (resource * 2^64 + A)     * 2^128 + P   ← same as T1
```

Because `hash_fee_fields` feeds this packed felt directly into `poseidon_hash_many` and the result propagates through `hash_tx_common_fields` into the final transaction hash, T1 and T2 share the **same transaction hash**. [2](#0-1) 

The `assert_nn` check only verifies `max_price_per_unit ∈ [0, P_stark/2]` (where `P_stark ≈ 2^251.58`), so values up to `≈ 2^250` are accepted. The collision construction above requires only `P + 2^128 < P_stark/2`, which holds for any realistic `P`.

---

### Impact Explanation

**Critical — Direct loss of funds / invalid transaction acceptance.**

An attacker who observes a user's signed v3 transaction T1 (broadcast to the mempool) can construct T2 with:
- `max_amount = A − 1` (one less gas unit allowed)
- `max_price_per_unit = P + 2^128` (astronomically high price cap)

T2 carries the **same hash** as T1, so the user's ECDSA/Stark signature over T1 is cryptographically valid for T2. The attacker submits T2 to the sequencer. If T2 is included:

1. The user's transaction executes with `max_amount` reduced by 1. If the transaction requires exactly `A` gas units, it reverts with out-of-gas — **permanent denial of the user's intended state transition**.
2. The `max_price_per_unit` field in the accepted transaction is `P + 2^128`. Depending on how the fee-charging logic uses this field, the user could be charged at the inflated cap rather than the actual network price, **draining the account**.

Both outcomes satisfy the allowed impact scope (direct loss of funds; network inability to confirm transactions for the targeted account).

---

### Likelihood Explanation

- StarkNet mempools are observable by any network participant.
- Constructing T2 from T1 requires only arithmetic on public transaction fields — no key material, no privileged access.
- The attacker only needs to submit T2 before T1 is sequenced, which is feasible in any public mempool environment.
- The `assert_nn` guard is present and looks correct at a glance, making this easy to miss in review — exactly the same class of subtle bound error as the `wExp` report.

---

### Recommendation

Replace the `assert_nn` on `max_price_per_unit` with a strict upper-bound check, mirroring the existing check on `max_amount`:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

This ensures the packing is injective and the transaction hash uniquely identifies the fee parameters. Document the invariant: the packed felt encodes `max_price_per_unit` in bits [0,127], `max_amount` in bits [128,191], and `resource` in bits [192+], which requires all three components to be bounded to their respective bit widths.

---

### Proof of Concept

Given a legitimate signed transaction T1:
- `resource = L1_GAS` (constant, ~`0x4c315f474153`)
- `max_amount = 1000` (fits in 64 bits ✓)
- `max_price_per_unit = 5 * 10^9` (5 Gwei, fits in 128 bits ✓)

Packed T1:
```
(L1_GAS * 2^64 + 1000) * 2^128 + 5_000_000_000
```

Attacker constructs T2 (passes all current OS checks):
- `max_amount = 999`
- `max_price_per_unit = 5_000_000_000 + 2^128`  ← passes `assert_nn`, fails no other check

Packed T2:
```
(L1_GAS * 2^64 + 999) * 2^128 + (5_000_000_000 + 2^128)
= (L1_GAS * 2^64 + 999) * 2^128 + 5_000_000_000 + 2^128
= (L1_GAS * 2^64 + 1000) * 2^128 + 5_000_000_000   ← identical to T1
```

`hash_fee_fields` receives the same packed felt for both transactions → same `fee_fields_hash` → same final transaction hash → T1's signature is valid for T2. The OS at `pack_resource_bounds` (line 105) accepts T2 without error. [3](#0-2)

func hash_fee_fields{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    tip: felt, resource_bounds: ResourceBounds*, n_resource_bounds: felt

    alloc_locals;

    let (local data_to_hash: felt*) = alloc();
    assert data_to_hash[0] = tip;
    assert_nn_le(tip, 2 ** 64 - 1);

    static_assert L1_GAS_INDEX == 0;
    static_assert L2_GAS_INDEX == 1;
    static_assert L1_DATA_GAS_INDEX == 2;

    with_attr error_message("Invalid number of resource bounds: {n_resource_bounds}.") {
        assert n_resource_bounds = 3;
    }

    // L1 gas.
    let l1_gas_bounds = resource_bounds[L1_GAS_INDEX];
    assert l1_gas_bounds.resource = L1_GAS;
    assert data_to_hash[1] = pack_resource_bounds(l1_gas_bounds);

    // L2 gas.
    let l2_gas_bounds = resource_bounds[L2_GAS_INDEX];
    assert l2_gas_bounds.resource = L2_GAS;
    assert data_to_hash[2] = pack_resource_bounds(l2_gas_bounds);

    // L1 data gas.
    let l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];
    assert l1_data_gas_bounds.resource = L1_DATA_GAS;
    assert data_to_hash[3] = pack_resource_bounds(l1_data_gas_bounds);

    let (hash) = poseidon_hash_many(n=n_resource_bounds + 1, elements=data_to_hash);
