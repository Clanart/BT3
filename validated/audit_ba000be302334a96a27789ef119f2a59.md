### Title
Missing Validation of New Class Hash in `execute_replace_class` Syscall - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` function in the StarkNet OS accepts an arbitrary `class_hash` from a contract's syscall request and writes it directly to the contract state without verifying that the class hash has been declared. This is the direct analog of the "Lack of Validation in Event Listeners" pattern: an externally-supplied value is consumed and used to mutate internal state without sanitization. The OS itself contains a `TODO` comment acknowledging this missing check. A contract that replaces its class with an undeclared hash becomes permanently uncallable, freezing any funds it holds.

### Finding Description
In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the new `class_hash` is read directly from the syscall request struct and written into `contract_state_changes` with no assertion that the hash exists in `contract_class_changes` (the declared-class registry):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `TODO` comment is the team's own acknowledgment that this validation is absent. Compare this with `execute_deploy` (lines 479–481), which does validate a boolean flag with `assert deploy_from_zero * (deploy_from_zero - 1) = 0`, and with `execute_declare_transaction` (line 816), which asserts `assert_not_zero(compiled_class_hash)` before writing. No equivalent guard exists for `replace_class`.

The `execute_syscalls` dispatcher (line 195–203 of `execute_syscalls.cairo`) routes any syscall with `selector == REPLACE_CLASS_SELECTOR` directly to `execute_replace_class` with no pre-check on the payload.

### Impact Explanation
**Critical — Permanent freezing of funds.**

If a contract's class hash is set to a value that does not exist in the declared-class registry, every future entry-point call to that contract will fail at class resolution. Any ERC-20 tokens, ETH, or other assets held in that contract's storage become permanently inaccessible. Because the OS generates a valid STARK proof for this state transition (the missing check means the constraint system does not reject it), the corrupted state is committed on-chain and cannot be rolled back.

### Likelihood Explanation
**Medium.**

The attack surface is any contract that exposes `replace_class` with caller-controlled input — the common upgradeable-proxy pattern. A user submits a transaction calling such a contract's upgrade function with an undeclared class hash. The sequencer executes the transaction; the blockifier may or may not catch it depending on its own validation layer. Crucially, the OS — which is the authoritative enforcement layer for the proof — does not enforce the constraint, so even if the blockifier is bypassed or has a gap, the OS will produce a valid proof for the invalid state. No privileged role is required beyond being a normal transaction sender interacting with an upgradeable contract.

### Recommendation
Before writing the new class hash to `contract_state_changes`, assert that it is present in `contract_class_changes` (i.e., that it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled-class hash is non-zero, mirroring the check already done in `execute_declare_transaction`:

```cairo
// Verify the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This closes the gap acknowledged by the existing `TODO` comment.

### Proof of Concept

1. Declare class `A` (valid, compiled hash `H_A`).
2. Deploy contract `C` using class `A`; `C` holds user funds and exposes `upgrade(new_class_hash)` which calls `replace_class(new_class_hash)`.
3. Attacker calls `C.upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared.
4. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xdeadbeef` from the request — no validation performed.
   - Writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. State is committed; a valid STARK proof is produced.
6. All subsequent calls to `C` fail at class resolution — funds are permanently frozen.
