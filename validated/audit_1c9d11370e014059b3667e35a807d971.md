### Title
Integer Truncation in `NewDynamicFeeChecker` Allows Zero-Fee Cosmos Transactions When `NoBaseFee = true` - (File: `ante/evm/fee_checker.go`)

### Summary

`NewDynamicFeeChecker` computes `feeCap` via integer division `fee / gas`. When the submitted fee is smaller than the gas limit, this truncates to zero. If the chain's `NoBaseFee` parameter is `true` (base fee forced to 0), the zero `feeCap` passes the `feeCap >= baseFee` guard, the effective price resolves to zero, and the returned `effectiveFee` has a zero amount — causing `DeductFeeDecorator` to skip fee deduction entirely. A Cosmos (non-EVM) transaction executes with no fee paid.

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` computes the per-gas fee cap using integer (truncating) division:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // line 83
```

`sdkmath.Int.Quo` performs truncating integer division. If the total fee attached to the transaction is less than the gas limit (e.g., `fee = 1`, `gas = 1_000_000`), `feeCap` truncates to `0`.

The guard that follows is:

```go
if feeCap.LT(baseFeeInt) {                          // line 86
    return nil, 0, errorsmod.Wrapf(...)
}
```

When `NoBaseFee = true`, `types.GetBaseFee` returns `big.NewInt(0)` (London HF is still active; the nil branch is only taken when London HF is disabled). Therefore `baseFeeInt = 0`, and `0 < 0` is `false` — the check passes silently.

The effective price is then:

```go
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
// = min(0 + maxPriorityPrice, 0) = 0
```

And the effective fee:

```go
effectiveFee := sdk.Coins{{Denom: denom, Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))}}
// Amount = 0 * gas = 0
```

This zero-amount `sdk.Coins` slice is returned to `DeductFeeDecorator.checkDeductFee` in `ante/evm/nativefee.go`:

```go
if !fee.IsZero() {                                  // line 110
    err := evmkeeper.DeductFees(dfd.bankKeeper, ctx, deductFeesFromAcc, fee)
```

`sdk.Coins{{Denom: denom, Amount: sdkmath.ZeroInt()}}.IsZero()` returns `true`, so `DeductFees` is never called. The transaction is processed with zero fee deducted from the sender.

The `MinGasPriceDecorator` (which runs earlier in `newCosmosAnteHandler`) short-circuits immediately when `MinGasPrice = 0` (the default):

```go
if minGasPrice.IsZero() || simulate {               // line 57
    return next(ctx, tx, simulate)
}
```

No other decorator in the Cosmos ante chain catches this case.

### Impact Explanation

Any unprivileged user can submit a Cosmos (non-EVM) transaction with a fee of `1 aphoton` and a gas limit of any value `> 1`. The ante handler accepts and processes the transaction without deducting any fee from the sender's balance. The fee collector module receives nothing. This is a direct mis-accounting of user fees: the sender retains funds they were contractually required to pay, and the fee distribution mechanism is bypassed. Under sustained use, this enables free-of-cost transaction spam against the chain.

This matches the allowed High impact: *"ante handler bug that permits valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

The conditions required are:
1. `NoBaseFee = true` — explicitly documented as a supported configuration ("needed for 0 price calls") and used in production deployments that need zero-price EVM calls.
2. `MinGasPrice = 0` — the default value per `DefaultMinGasPrice = sdkmath.LegacyZeroDec()`.
3. `DynamicFeeChecker = true` in `HandlerOptions` — the intended configuration for chains using EIP-1559 fee market for Cosmos txs.
4. Attacker submits a Cosmos tx with `fee < gas` — trivially achievable by any user.

All four conditions are simultaneously satisfiable in a standard Ethermint deployment with `NoBaseFee = true`. No privileged access is required.

### Recommendation

Replace the truncating integer division with a ceiling division or a pre-check that rejects any transaction whose total fee is less than `baseFee * gas`:

```go
// Before computing feeCap, reject if fee is insufficient to cover baseFee * gas
requiredFee := baseFeeInt.Mul(sdkmath.NewIntFromUint64(gas))
if fee.LT(requiredFee) {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee,
        "fee too low: got %s, required at least %s", fee, requiredFee)
}
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

Additionally, add a post-computation guard analogous to the Lido recommendation — reject if `effectiveFee` is zero when a non-zero base fee or min gas price is expected:

```go
if effectiveFee.IsZero() && !baseFeeInt.IsZero() {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, "effective fee rounds to zero")
}
```

### Proof of Concept

**Setup**: Chain with `NoBaseFee = true`, `MinGasPrice = 0` (defaults), `DynamicFeeChecker = true`.

**Attack**:
1. Attacker constructs a Cosmos tx (e.g., `MsgSend`) with `fee = sdk.NewCoins(sdk.NewCoin("aphoton", sdkmath.NewInt(1)))` and `gas = 1_000_000`.
2. Tx enters `newCosmosAnteHandler`.
3. `MinGasPriceDecorator`: `MinGasPrice = 0` → short-circuits, passes.
4. `DeductFeeDecorator` calls `NewDynamicFeeChecker`:
   - `baseFee = types.GetBaseFee(...) = big.NewInt(0)` (NoBaseFee=true, London HF active)
   - `fee = 1`, `gas = 1_000_000`
   - `feeCap = 1 / 1_000_000 = 0` (integer truncation)
   - `feeCap (0) < baseFeeInt (0)` → `false` → passes guard
   - `effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = min(MaxInt64, 0) = 0`
   - `effectiveFee = sdk.Coins{{Denom: "aphoton", Amount: 0}}`
5. Back in `checkDeductFee`: `fee.IsZero() = true` → `DeductFees` not called.
6. Tx executes. Sender's balance unchanged. Fee collector receives 0. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
