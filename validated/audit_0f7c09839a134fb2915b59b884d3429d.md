### Title
Missing Class Hash Existence Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash corresponds to a declared contract class. A malicious contract can call `replace_class` with an arbitrary, undeclared class hash. The OS accepts and commits this state transition without validation. Once committed, all future calls to the affected contract fail because the OS cannot resolve the class, permanently freezing any funds held by the contract.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), the function updates a contract's `class_hash` in `contract_state_changes` with no check that the new hash exists in `contract_class_changes` (the declared class registry):

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developer-acknowledged TODO at line 898 confirms the check is absent. Critically, the function's implicit argument list does not include `contract_class_changes`:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
``` [2](#0-1) 

The outer `execute_syscalls` function does carry `contract_class_changes` as an implicit argument but does not thread it into `execute_replace_class`:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
    ...
}
``` [3](#0-2) 

Because the OS is a Cairo proof system, hints (like `%{ GetContractAddressStateEntry %}`) are prover-supplied and not verified by the L1 verifier. Only Cairo constraints are verified. Since no Cairo constraint checks that `class_hash` is a key in `contract_class_changes`, the L1 verifier accepts any class hash, including one that was never declared.

For comparison, `execute_declare_transaction` correctly enforces that a class can only be declared once via `prev_value=0` in `dict_update`:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [4](#0-3) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

Once a contract's `class_hash` is set to an undeclared hash and this state is committed to L1 via the OS proof:

1. Every future call to the contract causes the OS to look up the class hash in its compiled class facts bundle and fail to find it.
2. All calls revert permanently — there is no recovery path.
3. Any ERC-20 tokens, ETH-equivalent assets, or other value held in the contract's storage are permanently inaccessible.
4. The state is final on L1; no upgrade or governance action can undo a committed OS state root.

This matches the allowed impact: **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The attack requires only an unprivileged contract deployer:

1. Attacker deploys a contract (e.g., a fake vault or token) that accepts user deposits.
2. Users deposit funds.
3. Attacker sends a transaction that causes the contract to invoke the `replace_class` syscall with an arbitrary felt value that is not a declared class hash.
4. The OS processes the block, accepts the state transition (no validation), and generates a valid proof.
5. The L1 verifier accepts the proof; the broken class hash is now the canonical on-chain state.
6. All subsequent calls to the contract fail; funds are frozen.

No privileged access, leaked keys, or external dependency compromise is required. The `replace_class` syscall is reachable by any contract during normal execution, as shown in `execute_syscalls`: [3](#0-2) 

---

### Recommendation

Pass `contract_class_changes` as an implicit argument to `execute_replace_class` and add a Cairo-level assertion that the requested `class_hash` exists as a key in `contract_class_changes` (i.e., its value is non-zero). This mirrors the existing enforcement in `execute_declare_transaction` where `prev_value=0` guarantees a class is declared exactly once before use.

---

### Proof of Concept

**State before attack:**
- Contract `C` holds user funds (e.g., 1000 tokens in storage).
- Class hash `H_valid` is declared in `contract_class_changes`.
- `C.class_hash = H_valid`.

**Attack transaction:**
- Attacker sends a transaction to `C` that triggers `replace_class(H_fake)` where `H_fake` is any felt not present in `contract_class_changes`.

**OS processing (`execute_replace_class`):**
```cairo
// No check: is H_fake in contract_class_changes? → MISSING
tempvar new_state_entry = new StateEntry(
    class_hash=H_fake,          // undeclared
    storage_ptr=state_entry.storage_ptr,
    nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),  // committed
);
``` [5](#0-4) 

**State after attack:**
- `C.class_hash = H_fake` (committed to L1 via OS proof).
- Any call to `C` → OS cannot resolve `H_fake` → execution fails.
- User funds in `C`'s storage are permanently frozen.
- No mechanism exists to restore `C.class_hash` to a valid value.
