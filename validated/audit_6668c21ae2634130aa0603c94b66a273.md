### Title
`get_state_response_part` Uses New-Epoch Shard Layout to Index Old-Epoch Block Chunks, Causing State Sync Failure at Resharding Boundaries — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::get_state_response_part` derives the chunk array index from the **new** epoch's shard layout, then uses that index to read `prev_state_root` from the **previous** epoch's block. `compute_state_response_header` does the opposite correctly: it uses the previous epoch's layout. At any resharding boundary where a shard split inserts new child shards before an unchanged shard (shifting its index), the two functions compute different shard indices, breaking the state-root commitment embedded in the header and causing all part requests for the affected shard to be rejected.

---

### Finding Description

**Header path — correct:**

`compute_state_response_header` explicitly fetches the previous epoch's layout and uses it to index into `sync_prev_block.chunks()`:

```
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;   // ← old layout ✓
let chunk_header = sync_prev_block.chunks().get(prev_shard_index)...;
``` [1](#0-0) 

The `state_root_node` (and therefore `num_parts`) embedded in the returned header is derived from the correct chunk. [2](#0-1) 

**Part-serving path — wrong:**

`get_state_response_part` fetches the **new** epoch's layout and uses it to index into `prev_block.chunks()`, which is the same old-epoch block:

```
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;  // new epoch ✗
...
let prev_block = self.chain_store.get_block(header.prev_hash())?;   // old-epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;          // new-epoch index ✗
let state_root = prev_block.chunks().get(shard_index)               // wrong slot ✗
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [3](#0-2) 

The `num_parts` is then derived from this wrong `state_root_node`: [4](#0-3) 

**The invariant that breaks:**

`ShardStateSyncResponseHeader.state_root_node` (and its derived `num_parts = get_num_state_parts(memory_usage)`) is the committed "conversion rate" between part ordinals and trie boundaries. [5](#0-4) 

`set_state_part` validates every received part against the state root extracted from that header: [6](#0-5) 

When `get_state_response_part` reads a different `state_root` than the one committed in the header, the part it generates either fails `validate_state_part` or — if the new-epoch index exceeds the old block's chunk count — the function returns `Error::InvalidShardId` before even generating a part.

**Concrete resharding scenario (V3 / dynamic resharding):**

| Epoch | Shard layout | Chunk indices |
|-------|-------------|---------------|
| N (old) | [shard 0, shard 1, shard 2] | 0, 1, 2 |
| N+2 (new, after shard 1 → shards 5, 6) | [shard 0, shard 5, shard 6, shard 2] | 0, 1, 2, 3 |

Shard 2 is unchanged but its index shifts from 2 → 3. When a node syncing shard 2 at the N+2 boundary requests a part:

- `compute_state_response_header` → `prev_shard_index = 2` → correct `state_root` for shard 2 → header committed.
- `get_state_response_part` → `shard_index = 3` → `prev_block.chunks().get(3)` → `None` → `Error::InvalidShardId(shard_id)`.

Every part request for shard 2 is rejected; state sync is permanently blocked for that shard at that boundary.

---

### Impact Explanation

Any node that was not previously tracking an unchanged shard whose index shifts at a resharding boundary cannot complete state sync for that shard. It receives a valid header (correct state root, correct `num_parts`) but every subsequent part request returns `InvalidShardId`. The node cannot apply the shard's state, cannot produce or validate chunks for it in the new epoch, and falls out of the validator set for that shard. With multiple shards shifting indices in a single resharding event, multiple shards are simultaneously unsynchable. **Impact: High.**

---

### Likelihood Explanation

The bug fires on every dynamic-resharding epoch boundary where at least one unchanged shard's index shifts — a normal consequence of inserting child shards at non-terminal positions in the boundary-account ordering. It requires no privileged role: any node performing ordinary state sync triggers it. **Likelihood: Medium** (gated on `ProtocolFeature::DynamicResharding` being active, but unconditional once active).

---

### Recommendation

In `get_state_response_part`, replace the new-epoch layout lookup with the previous epoch's layout — mirroring what `compute_state_response_header` already does correctly:

```rust
// current (wrong):
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
...
let shard_index = shard_layout.get_shard_index(shard_id)?;

// fix:
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
```

This ensures the `state_root` extracted from `prev_block.chunks()[shard_index]` is identical to the one committed in the header, restoring the `num_parts` invariant across both code paths.

---

### Proof of Concept

1. Configure a network with dynamic resharding enabled (`ProtocolFeature::DynamicResharding`) and 3 shards: [shard 0, shard 1, shard 2].
2. Trigger a split of shard 1 → shards 5, 6. The new layout is [shard 0, shard 5, shard 6, shard 2] with indices [0, 1, 2, 3].
3. Start a fresh node that must sync shard 2 at the resharding boundary (`sync_hash` = first block of new epoch).
4. Node sends `StateRequestHeader { shard_id: 2, sync_hash }` → serving node calls `compute_state_response_header` → `prev_shard_index = 2` → correct `state_root` → header returned successfully.
5. Node sends `StateRequestPart { shard_id: 2, sync_hash, part_id: 0 }` → serving node calls `get_state_response_part` → `shard_index = shard_layout.get_shard_index(2) = 3` → `prev_block.chunks().get(3) = None` → `Error::InvalidShardId(2)` returned.
6. All part requests fail; the node cannot reconstruct shard 2's state and cannot participate in the new epoch for that shard.
