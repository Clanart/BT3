### Title
`deployToken` Uses `CREATE` Opcode for Bridge Token Proxy Deployment, Enabling Reorg-Based Token-Mapping Corruption and Unauthorized Minting â€” (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken` deploys the `ERC1967Proxy` for each bridged NEAR token using the `new` keyword (EVM `CREATE` opcode), whose address derivation depends solely on the `OmniBridge` contract's nonce. On reorg-susceptible chains where `OmniBridge` is deployed (Polygon, Optimism, Arbitrum), a block reorg can cause a different NEAR token's proxy to occupy the address that was previously mapped to the original token. This corrupts the `nearToEthToken` / `isBridgeToken` mappings and enables unauthorized minting of bridged assets via `finTransfer`.

---

### Finding Description

In `OmniBridge.deployToken`, the bridge token proxy is deployed at line 162:

```solidity
address bridgeTokenProxy = address(
    new ERC1967Proxy(          // <-- CREATE opcode; address = f(OmniBridge, nonce)
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

The resulting address depends only on `OmniBridge`'s account nonce, not on any token-specific data such as `metadata.token` (the NEAR token ID). After deployment, the mappings are updated:

```solidity
isBridgeToken[address(bridgeTokenProxy)] = true;
ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
nearToEthToken[metadata.token] = address(bridgeTokenProxy);
```

`deployToken` is callable by any unprivileged user who holds a valid MPC-signed `MetadataPayload`. Multiple such signatures can be in flight simultaneously (e.g., two different NEAR tokens both awaiting deployment).

**Reorg attack path:**

1. The MPC network signs payloads for two NEAR tokens: `token-a.near` and `token-b.near`.
2. Alice submits `deployToken` for `token-a.near`. At nonce N, the proxy lands at address **X**. Mappings: `nearToEthToken["token-a.near"] = X`, `isBridgeToken[X] = true`.
3. The MPC observes address X and signs a `finTransfer` payload with `tokenAddress = X` for a user bridging `token-a.near` tokens.
4. A block reorg occurs (Polygon reorgs of >1.5 min are documented). Alice's transaction is evicted.
5. Bob submits `deployToken` for `token-b.near` first in the reorged chain. At nonce N (same), his proxy lands at address **X**. Mappings: `nearToEthToken["token-b.near"] = X`, `isBridgeToken[X] = true`.
6. Alice's `deployToken` re-executes at nonce N+1, deploying `token-a.near` to address **Y**.
7. The MPC-signed `finTransfer` with `tokenAddress = X` is now submitted. `isBridgeToken[X]` is `true` (for Bob's token). The `finTransfer` branch at line 337â€“349 executes `IBridgeToken(X).mint(recipient, amount)` â€” minting Bob's token (`token-b.near`) to the user who bridged `token-a.near`.

---

### Impact Explanation

**Critical / High.** Two concrete impacts:

1. **Unauthorized mint of bridged assets**: A `finTransfer` signed by the MPC for `token-a.near` (with `tokenAddress = X`) mints Bob's `token-b.near` proxy tokens to the recipient. The MPC signature is valid, but the token at address X is now a different asset. This is an unbacked mint of `token-b.near` supply.

2. **Token-mapping corruption breaking bridge collateralization**: After the reorg, `nearToEthToken["token-a.near"] = Y` while off-chain relayers and the MPC may still reference X. Subsequent `finTransfer` calls signed with `tokenAddress = X` mint the wrong token; calls signed with `tokenAddress = Y` may fail or mint to an uninitialized proxy if Y's deployment was also reorged. This permanently breaks the 1:1 collateral backing of the affected bridged tokens.

---

### Likelihood Explanation

**Moderate.** `OmniBridge` is explicitly deployed on Polygon, Optimism, and Arbitrum â€” all chains with documented reorg exposure. Polygon has publicly visible reorgs exceeding 1.5 minutes. The attack requires only: (a) a concurrent valid MPC signature for a second token (realistic during any multi-token deployment wave), and (b) a reorg window long enough for a competing `deployToken` to be submitted (achievable with scripted automation). No privileged access is required.

---

### Recommendation

Replace `new ERC1967Proxy(...)` with a `CREATE2` deployment using a salt derived from the NEAR token identifier, ensuring the proxy address is deterministically and uniquely bound to the specific NEAR token regardless of nonce:

```solidity
bytes32 salt = keccak256(abi.encodePacked(metadata.token));
address bridgeTokenProxy = address(
    new ERC1967Proxy{salt: salt}(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

This makes the proxy address a pure function of the NEAR token ID, eliminating nonce-based address ambiguity across reorgs.
