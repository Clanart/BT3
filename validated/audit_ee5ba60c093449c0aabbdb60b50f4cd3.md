### Title
`ProofOfInclusion::valid()` Is a Tautological Self-Check That Never Verifies Against a Trusted Root — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only verifies internal self-consistency of the proof struct. Its final comparison is a tautology: it compares `existing_hash` (which was just set to `layer.combined_hash` in the last loop iteration) against `self.root_hash()` (which returns `last.combined_hash`). These are always equal when the loop completes without error. The method never checks the computed root against any externally trusted root hash. An attacker can therefore construct a fully self-consistent `ProofOfInclusion` for any arbitrary key-value pair and any arbitrary tree, and `valid()` will return `true`.

---

### Finding Description

In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`, the two relevant methods are:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- derived from the proof itself, not from a trusted source
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // <-- after last iteration, equals last.combined_hash
    }

    existing_hash == self.root_hash()  // <-- always true: last.combined_hash == last.combined_hash
}
```

After the loop, `existing_hash` holds the last `calculated_hash`, which was already verified to equal `layer.combined_hash` for the last layer. `self.root_hash()` returns `last.combined_hash`. The final comparison is therefore `last.combined_hash == last.combined_hash` — a tautology that is always `true`.

The method performs no comparison against any externally supplied, trusted root hash. It only verifies that the proof's own internal hash chain is self-consistent.

The `ProofOfInclusion` struct is fully attacker-controlled: it is a `Streamable` type exposed via Python bindings (`py_valid()`) and can be deserialized from untrusted bytes. An attacker can craft a `ProofOfInclusion` with:
- `node_hash` = hash of any target key-value pair they wish to falsely prove included
- Arbitrary `other_hash` values at each layer
- `combined_hash` values correctly computed from those arbitrary inputs

Such a proof will pass `valid()` while proving membership in a completely fabricated tree whose root has nothing to do with the actual DataLayer tree root.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Any Python or Rust consumer that receives a `ProofOfInclusion` from an untrusted source and calls `.valid()` to authorize a decision (e.g., confirming a key-value pair is committed in a DataLayer store) will accept a forged proof. The attacker can prove that any key maps to any value in any tree, without possessing the actual tree data or the real root hash.

---

### Likelihood Explanation

The `ProofOfInclusion` type is a `Streamable` struct exposed directly through the Python wheel API (`py_valid()` in the `pymethods` block). It is the primary API surface for DataLayer proof verification. Any off-chain verifier that relies on `proof.valid()` alone — without separately comparing `proof.root_hash()` against a trusted on-chain root — is fully exploitable by an unprivileged attacker who can supply crafted proof bytes. The fuzz target and all existing tests only call `proof.valid()` on proofs generated by the library itself, so the tautology has never been caught.

---

### Recommendation

`valid()` must accept a trusted root hash as a parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root  // compare against externally trusted root
}
```

The no-argument `valid()` method should either be removed or clearly documented as only checking internal self-consistency (not authenticity). The Python binding should expose `valid_against_root(trusted_root: bytes32)` instead of the current `valid()`.

# (using the same calculate_internal_hash logic the library uses)
# The exact hash function doesn't matter — the attacker just needs to
# compute it the same way the library does.
# combined_hash = calculate_internal_hash(fake_node_hash, side, fake_other_hash)
# ... (attacker computes this using the public library) ...

# Step 4: construct the forged proof
forged_proof = ProofOfInclusion(
    node_hash=fake_node_hash,
    layers=[
        ProofOfInclusionLayer(
            other_hash_side=0,
            other_hash=fake_other_hash,
            combined_hash=computed_combined_hash,  # correctly computed
        )
    ]
)

# Step 5: valid() returns True for a completely fabricated proof
assert forged_proof.valid()  # passes — tautological check
# forged_proof.root_hash() == computed_combined_hash (attacker-controlled)
# This root has nothing to do with any real DataLayer tree.
