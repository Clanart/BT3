### Title
Unvalidated `msg.value` in `payable` bridge functions causes permanent ETH loss for callers — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer`, `OmniBridge.logMetadata`, and `OmniBridge.logMetadata1155` are all declared `external payable` but contain no check that `msg.value == 0` when the operation does not require native ETH. Any ETH accidentally sent by a caller is permanently locked in the contract (base variant) or silently consumed by Wormhole beyond the required fee (Wormhole variant), with no recovery path.

---

### Finding Description

Three publicly callable functions in `OmniBridge.sol` are marked `payable` but do not validate `msg.value` for the code paths that handle non-native tokens:

**1. `finTransfer` (lines 279–367)**

```solidity
function finTransfer(
    bytes calldata signatureData,
    BridgeTypes.TransferMessagePayload calldata payload
) external payable whenNotPaused(PAUSED_FIN_TRANSFER) {
```

When `payload.tokenAddress != address(0)` (i.e., ERC20, ERC1155, or bridge token), the function mints or transfers the ERC20/bridge token to the recipient and never touches `msg.value`. In the base `OmniBridge`, `finTransferExtension` is a no-op, so any ETH sent is silently absorbed into the contract's ETH balance. In `OmniBridgeWormhole`, `finTransferExtension` forwards the entire `msg.value` to Wormhole:

```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, messagePayload, _consistencyLevel);
```

Wormhole only requires `msg.value >= messageFee()` and does not refund excess. Any ETH above the fee is permanently lost.

**2. `logMetadata` (lines 224–232) and `logMetadata1155` (lines 234–270)**

Both are `external payable` with no access control. They accept any `msg.value` without validation. In the base contract, `logMetadataExtension` is a no-op; in `OmniBridgeWormhole`, `msg.value` is forwarded wholesale to Wormhole's `publishMessage`. Excess ETH is consumed.

**Root cause:** None of these functions contain a `require(msg.value == 0, ...)` guard for the non-native-ETH code paths, nor do they validate `msg.value == wormhole.messageFee()` in the Wormhole variant.

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user ETH funds in bridge flows.**

- In the base `OmniBridge`: ETH sent with `finTransfer` for an ERC20 token is absorbed into the contract's ETH pool. That pool is drawn upon for future native-ETH `finTransfer` calls (`payload.tokenAddress == address(0)`), meaning the accidentally sent ETH is effectively redistributed to unrelated recipients. The original sender has no recovery path.
- In `OmniBridgeWormhole`: ETH above `messageFee()` is consumed by Wormhole and unrecoverable.
- There is no admin withdrawal function for accidentally sent ETH; the only ETH egress is the native-token payout path in `finTransfer`.

---

### Likelihood Explanation

**Medium.** `finTransfer` is callable by any relayer or user who holds a valid MPC signature. `logMetadata` and `logMetadata1155` have no access control at all — any address can call them. In the Wormhole deployment, callers must supply ETH for the Wormhole fee; a caller who slightly overestimates the fee (e.g., uses a stale `messageFee()` value or adds a buffer) permanently loses the excess. This is a realistic mistake for integrators and front-end users.

---

### Recommendation

Add a `msg.value` guard in each non-native-ETH code path. For the base `OmniBridge`:

```solidity
// In finTransfer, before the non-ETH branches:
if (payload.tokenAddress != address(0)) {
    require(msg.value == 0, "OB: wrong msg.value");
}
```

For `OmniBridgeWormhole`, validate the exact fee:

```solidity
require(msg.value == _wormhole.messageFee(), "OBW: wrong msg.value");
```

Apply the same pattern to `logMetadata`, `logMetadata1155`, and `deployToken`.

---

### Proof of Concept

1. Deploy `OmniBridge` (base variant) with a registered ERC20 bridge token.
2. Obtain a valid MPC signature for a `TransferMessagePayload` where `tokenAddress` is the ERC20 address.
3. Call `finTransfer{value: 1 ether}(signatureData, payload)`.
4. The ERC20 is minted/transferred to `payload.recipient` successfully.
5. The contract's ETH balance increases by 1 ETH; the caller's ETH is gone.
6. No function exists for the caller to reclaim the 1 ETH.
