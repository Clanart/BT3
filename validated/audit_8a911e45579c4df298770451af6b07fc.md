### Title
`MetadataPayload` Hash Missing Chain ID Enables Cross-Chain Replay of `deployToken` Signature - (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` hash used to authorize `deployToken`/`deploy_token` across EVM, Starknet, and Solana does not include the destination chain ID or contract address. In contrast, `TransferMessagePayload` (used in `finTransfer`/`fin_transfer`) explicitly encodes the chain ID twice. Because the NEAR MPC signer uses the same derived key across all destination chains, a valid `deployToken` signature obtained for one chain can be replayed verbatim on any other chain, deploying an unauthorized token contract and permanently blocking the legitimate deployment of that token on the target chain.

---

### Finding Description

**NEAR signing side** — `log_metadata_callback` in `near/omni-bridge/src/lib.rs` constructs and signs a `MetadataPayload` that contains only `{prefix, token, name, symbol, decimals}`:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
```

No destination chain ID is included. [1](#0-0) 

**EVM verification** — `deployToken` in `OmniBridge.sol` reconstructs the hash identically, with no `omniBridgeChainId`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

Compare this to `finTransfer`, which encodes `omniBridgeChainId` twice (once for the token address, once for the recipient address), providing chain binding: [3](#0-2) 

**Starknet verification** — `MetadataPayload.to_borsh()` in `bridge_types.cairo` also omits chain ID:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
}
``` [4](#0-3) 

While `TransferMessagePayload.to_borsh(chain_id)` explicitly takes and encodes `chain_id` twice: [5](#0-4) 

**Solana verification** — `DeployTokenPayload.serialize_for_near()` in `deploy_token.rs` serializes only the metadata fields with no chain ID or program address:

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    self.serialize(&mut writer)?; // borsh encoding
}
``` [6](#0-5) 

While `FinalizeTransferPayload.serialize_for_near()` writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` twice: [7](#0-6) 

The `nearBridgeDerivedAddress` / `omni_bridge_derived_address` is the same Ethereum-style address derived from the NEAR MPC key across all destination chains, so the same raw signature is cryptographically valid on every chain. [8](#0-7) 

---

### Impact Explanation

**High — Permanent freezing of token deployment on target chains.**

Once an attacker replays a valid `deployToken` signature on a chain it was not intended for, the token is registered in `nearToEthToken` / `near_to_starknet_token` with an attacker-triggered contract address. The duplicate-deployment guard (`ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`) then permanently prevents the NEAR bridge from ever deploying the legitimate, bridge-authorized version of that token on the target chain. [9](#0-8) 

This irrecoverably blocks all future bridging of the affected token to the target chain, freezing the bridge's token deployment capability for that (token, chain) pair.

---

### Likelihood Explanation

**Medium-High.** The attack requires only:
1. Observing a `deployToken` transaction on any public chain (all data is on-chain).
2. Submitting the same `(signatureData, metadata)` tuple to `deployToken` on a different chain before the legitimate deployment occurs.

No privileged access, leaked keys, or colluding parties are needed. The attacker is an unprivileged bridge user who can call `deployToken` on any chain. The window of opportunity exists between the NEAR bridge signing the metadata and the legitimate relayer submitting the deployment on each chain.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` hash on all chains, mirroring the pattern already used in `TransferMessagePayload`/`finTransfer`.

**EVM** — add `omniBridgeChainId` to the borsh-encoded hash in `deployToken`:
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    ...
);
```

**Starknet** — change `MetadataPayload.to_borsh()` to accept and encode `chain_id`, matching `TransferMessagePayload.to_borsh(chain_id)`.

**Solana** — add `SOLANA_OMNI_BRIDGE_CHAIN_ID` to `DeployTokenPayload.serialize_for_near()`, matching `FinalizeTransferPayload`.

**NEAR signing** — update `log_metadata_callback` to include the target chain ID in the `MetadataPayload` before hashing and signing, so each chain receives a chain-specific signature.

---

### Proof of Concept

1. NEAR bridge calls `log_metadata` for token `"usdc.near"`. The MPC signer produces signature `S` over `keccak256(borsh({Metadata, "usdc.near", "USD Coin", "USDC", 6}))`. This signature is broadcast on-chain (e.g., via a `sign_log_metadata_callback` event on Ethereum).

2. Attacker observes `S` and the payload on Ethereum. Before the legitimate relayer submits `deployToken` on Arbitrum, the attacker calls:
   ```solidity
   arbitrumOmniBridge.deployToken(S, {token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6});
   ```
   Because `omniBridgeChainId` is not in the hash, `ECDSA.recover(hashed, S) == nearBridgeDerivedAddress` passes on Arbitrum.

3. `"usdc.near"` is now registered in Arbitrum's `nearToEthToken` mapping pointing to the attacker-triggered contract address.

4. When the legitimate relayer later attempts to deploy `"usdc.near"` on Arbitrum, the call reverts with `ERR_TOKEN_EXIST`. The token is permanently undeployable on Arbitrum through the bridge, freezing all future NEAR→Arbitrum transfers of `"usdc.near"`.
