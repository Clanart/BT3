### Title
Missing Chain ID in `deployToken` Signature Hash Enables Cross-Chain Replay of Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The MPC-signed message hashed for `deployToken`/`deploy_token` across all three bridge implementations (EVM, StarkNet, Solana) omits the destination chain ID. Because the same NEAR MPC key (`nearBridgeDerivedAddress`) is shared across every deployed chain, a valid `deployToken` signature produced for one chain is cryptographically valid on every other chain. An unprivileged attacker can replay a legitimate `deployToken` signature on any other OmniBridge deployment, causing unauthorized token deployment without NEAR's per-chain authorization.

---

### Finding Description

**Asymmetry between `deployToken` and `finTransfer` message construction:**

`finTransfer` in `OmniBridge.sol` encodes `omniBridgeChainId` twice in the borsh payload (for the token address and recipient fields), binding the signature to a specific chain:

```solidity
bytes1(omniBridgeChainId),   // line 294 — destination chain for token
...
bytes1(omniBridgeChainId),   // line 297 — destination chain for recipient
``` [1](#0-0) 

`deployToken` in the same contract encodes only the token metadata — no chain ID:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

The same omission is present on StarkNet. `MetadataPayloadTrait::to_borsh()` takes no chain ID parameter and encodes only token metadata:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    ...
}
``` [3](#0-2) 

Contrast with `TransferMessagePayloadTrait::to_borsh(chain_id: u8)`, which explicitly receives and encodes the chain ID:

```cairo
fn to_borsh(self: @TransferMessagePayload, chain_id: u8) -> ByteArray {
    ...
    borsh_bytes.append_byte(chain_id);
``` [4](#0-3) 

The StarkNet `deploy_token` call passes no chain ID to `to_borsh()`:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(), signature);
``` [5](#0-4) 

While `fin_transfer` correctly passes the stored chain ID:

```cairo
_verify_borsh_signature(
    ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature,
);
``` [6](#0-5) 

On Solana, `DeployTokenPayload::serialize_for_near` serializes only the metadata type prefix and the payload fields — no `SOLANA_OMNI_BRIDGE_CHAIN_ID`:

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    self.serialize(&mut writer)?;  // token, name, symbol, decimals only
``` [7](#0-6) 

Whereas `FinalizeTransferPayload::serialize_for_near` writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the buffer:

```rust
writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;
params.0.serialize(&mut writer)?;
``` [8](#0-7)

### Impact Explanation

The NEAR MPC key is a single key whose derived Ethereum address (`nearBridgeDerivedAddress`) is configured identically on every EVM OmniBridge deployment. Because the `deployToken` hash contains no chain-binding field, a signature produced by NEAR's MPC for chain A is byte-for-byte valid on chain B.

An attacker can:
1. Observe a legitimate `deployToken` call on chain A (e.g., Ethereum).
2. Replay the identical `(signatureData, metadata)` calldata on chain B (e.g., BSC or any other OmniBridge deployment).
3. The `ECDSA.recover` check passes because the hash is identical and the signer key is the same.
4. A bridge token is deployed on chain B without NEAR's per-chain authorization.

Consequences:
- **Unauthorized token deployment** on chains NEAR did not authorize, satisfying the "High" impact category ("signature bypass enabling unauthorized token deployment").
- **Permanent griefing of legitimate deployment**: once the attacker's replay sets `nearToEthToken[metadata.token]` on chain B, the `ERR_TOKEN_EXIST` guard prevents NEAR from ever deploying the token legitimately on that chain via `deployToken`. The admin would need to use the privileged `addCustomToken` path to recover.
- **Token-mapping corruption**: the attacker-deployed token is registered in the bridge's canonical mapping (`nearToEthToken`, `isBridgeToken`), so any subsequent legitimate `finTransfer` for that NEAR token on chain B will mint into the replayed deployment — a deployment NEAR never authorized for that chain. [9](#0-8) 

---

### Likelihood Explanation

- All `deployToken` calls are public on-chain transactions; the signature and payload are fully visible in calldata.
- The attacker needs no special access — only the ability to submit a transaction on another EVM chain.
- The NEAR MPC key is shared across all chains by design, so the recovered signer address is identical everywhere.
- The attack is executable immediately after any legitimate `deployToken` is confirmed on any chain.

---

### Recommendation

Include the destination chain ID in the `deployToken` borsh-encoded payload, mirroring the pattern already used in `finTransfer`. For EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `MetadataPayloadTrait::to_borsh()` in `starknet/src/bridge_types.cairo` (add `chain_id: u8` parameter, encode it) and to `DeployTokenPayload::serialize_for_near` in `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs` (write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the buffer). The NEAR MPC signing logic must be updated in lockstep to include the chain ID when producing `deployToken` signatures.

---

### Proof of Concept

1. NEAR's MPC signs a `deployToken` message for NEAR token `"usdc.near"` targeting Ethereum (`omniBridgeChainId = 1`). The signed hash is `keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)`.
2. The attacker observes the Ethereum `deployToken` transaction and extracts `(signatureData, metadata)`.
3. The attacker calls `OmniBridge.deployToken(signatureData, metadata)` on BSC (`omniBridgeChainId = 2`).
4. BSC's `OmniBridge` computes the identical hash (no chain ID in the payload) and `ECDSA.recover` returns `nearBridgeDerivedAddress` — the check passes.
5. A `BridgeToken` proxy for `"usdc.near"` is deployed on BSC and registered in `nearToEthToken["usdc.near"]`.
6. NEAR never authorized a `deployToken` for BSC; the token now exists on BSC without NEAR's per-chain consent, and any future legitimate attempt to deploy it on BSC via `deployToken` will revert with `ERR_TOKEN_EXIST`. [2](#0-1) [3](#0-2) [7](#0-6)
