### Title
Missing Pause Check in Solana `deploy_token` Allows Token Deployment During Emergency Pause — (`solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana `bridge_token_factory` program's `deploy_token` instruction lacks a pause guard, while every other state-mutating instruction (`init_transfer`, `init_transfer_sol`, `finalize_transfer`, `finalize_transfer_sol`) is correctly gated. Furthermore, the `ALL_PAUSED` constant does not include a `DEPLOY_TOKEN_PAUSED` bit at all, so even a full emergency pause cannot block token deployment on Solana. This is inconsistent with the EVM and Starknet implementations, which both define and enforce a dedicated deploy-token pause flag.

---

### Finding Description

**Solana `deploy_token` — no pause check:** [1](#0-0) 

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

No pause flag is checked before executing. Compare to `finalize_transfer` and `init_transfer`, which both gate on their respective pause bits: [2](#0-1) 

```rust
pub fn finalize_transfer(...) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0,
        error::ErrorCode::Paused
    );
    ...
}
``` [3](#0-2) 

**`ALL_PAUSED` does not include a deploy-token bit:** [4](#0-3) 

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED;
```

There is no `DEPLOY_TOKEN_PAUSED` constant. Calling `pause()` (which sets `ALL_PAUSED`) leaves `deploy_token` fully operational.

**Contrast with EVM — `deployToken` is correctly gated:** [5](#0-4) 

```solidity
function deployToken(...) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
```

**Contrast with Starknet — `deploy_token` is correctly gated:** [6](#0-5) 

```cairo
fn deploy_token(ref self: ContractState, signature: Signature, payload: MetadataPayload) {
    assert(!_is_paused(@self, PAUSE_DEPLOY_TOKEN), 'ERR_DEPLOY_TOKEN_PAUSED');
```

**Contrast with NEAR — `deploy_token` is correctly gated:** [7](#0-6) 

```rust
#[pause(except(roles(Role::DAO)))]
pub fn deploy_token(&mut self, #[serializer(borsh)] args: DeployTokenArgs) -> Promise {
```

---

### Impact Explanation

When the bridge admin triggers an emergency pause on Solana (e.g., due to a compromised MPC signing key, a bridge exploit, or a token accounting bug), the intent is to halt all bridge operations. However, any relayer holding a valid `DeployTokenPayload` signature — legitimately obtained from the NEAR bridge before the pause — can still call `deploy_token` and register a new SPL token mint on Solana.

Concrete consequences:
1. **Unauthorized token deployment during a security incident**: A new token mint is created and registered in the program's state, establishing a token mapping that the admin cannot prevent. This undermines the emergency stop mechanism.
2. **Persistent state corruption**: Once a token is deployed (mint PDA created, metadata initialized), it cannot be easily undone. The token mapping persists after the bridge is unpaused, potentially interfering with the incident response.
3. **Escalation with a compromised signing key**: If the pause was triggered because the MPC-derived signing key is compromised, an attacker in possession of that key can deploy arbitrary token mints on Solana even during the pause, pre-positioning for unauthorized `finalize_transfer` calls once the bridge is unpaused.

This matches the allowed impact: **"High. Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized transfer finalization, token deployment, or message execution."**

---

### Likelihood Explanation

- Any relayer with a pending `DeployTokenPayload` signature (obtained before the pause) can trigger this immediately after a pause is set.
- No special privileges are required beyond possessing a valid signed payload — a normal bridge participant role.
- The window of exploitation is the entire duration of the pause, which can be extended during a security incident.
- The inconsistency with EVM and Starknet implementations makes this an oversight rather than an intentional design choice.

---

### Recommendation

1. Add a `DEPLOY_TOKEN_PAUSED` constant:

```rust
pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
```

2. Add the pause guard to `deploy_token` in `lib.rs`:

```rust
pub fn deploy_token(ctx: Context<DeployToken>, data: SignedPayload<DeployTokenPayload>) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
        error::ErrorCode::Paused
    );
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

### Proof of Concept

1. Admin detects a security incident and calls `pause()` on the Solana `bridge_token_factory`, setting `config.paused = ALL_PAUSED` (`0x03`).
2. A relayer holds a valid `SignedPayload<DeployTokenPayload>` for a new token (signature obtained from the NEAR bridge before the pause).
3. The relayer calls `deploy_token` with this payload. The instruction executes successfully because `deploy_token` performs no pause check.
4. A new SPL token mint PDA is created and registered in the program state, despite the bridge being in a full emergency pause.
5. The admin's emergency stop has failed to halt token deployment, contrary to the behavior on EVM and Starknet. [1](#0-0) [4](#0-3)
