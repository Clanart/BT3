### Title
Empty `check_is_reverted()` Performs No Validation, Allowing Unconstrained Hint to Skip Transaction Execution While Charging Fees — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted()` in `execution_constraints.cairo` has a completely empty body. The `is_reverted` flag controlling whether an invoke transaction's execute step runs is set exclusively by a prover-controlled hint and is never constrained by the Cairo program. A malicious sequencer/prover can set `is_reverted = TRUE` for any non-reverted invoke transaction, skip its execution, still charge the fee, and produce a cryptographically valid proof that the L1 verifier accepts — resulting in direct loss of user funds.

---

### Finding Description

In `execute_invoke_function_transaction`, the `is_reverted` local variable is populated entirely by the hint `%{ IsReverted %}` and then passed to `check_is_reverted`:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // Align the stack — execution is skipped entirely.
    ...
}
// Fee is charged regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The function `check_is_reverted` that is supposed to constrain this hint-provided value is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

The function body is **completely empty**. It accepts `is_reverted` as an argument and immediately returns, imposing zero constraints on the value. In Cairo's proof system, any local variable populated by a hint must be constrained by the program logic to be sound. Because `check_is_reverted` does nothing, the prover is free to assign any value to `is_reverted` and still produce a valid proof.

This is the direct analog of the reported `safeTransferMany()` bug: a function whose name implies a safety check (`check_is_reverted`) silently omits the check entirely, just as `safeTransferMany()` called `_transfer()` instead of `_safeTransfer()`.

The fee is charged unconditionally after the branch: [3](#0-2) 

So when `is_reverted` is forged to `TRUE`, the user's `__execute__` entry point is skipped, but `charge_fee` still runs and deducts the fee from the user's account.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious sequencer can:
1. Accept a valid invoke transaction (signature verified by `run_validate`).
2. Set `is_reverted = TRUE` via the `%{ IsReverted %}` hint for a transaction that did not actually revert.
3. Skip the `non_reverting_select_execute_entry_point_func` call (the `__execute__` step).
4. Still invoke `charge_fee`, deducting the full fee from the user's account.
5. Produce a proof that is cryptographically valid because `check_is_reverted` imposes no constraint on `is_reverted`.
6. Submit this proof to the L1 verifier, which accepts it.

The result: users pay fees for transactions that were never executed. The state transition committed on L1 reflects fee deduction and nonce increment but no execution effects — a provably incorrect state accepted as valid.

---

### Likelihood Explanation

The sequencer controls all hints passed to the Cairo OS program. Since `check_is_reverted` is empty, the sequencer can exploit this on any invoke transaction at will, with no additional preconditions. The only requirement is that the sequencer is willing to act maliciously. The StarkNet OS is specifically designed to make sequencer behavior verifiable and prevent exactly this class of manipulation; the empty function defeats that guarantee entirely.

---

### Recommendation

Implement `check_is_reverted` to actually constrain the `is_reverted` value against the verifiable execution outcome. At minimum, the function should assert that `is_reverted` is a boolean (0 or 1) and, more importantly, should verify it against the actual failure flag returned by the entry point execution. The analogous correct pattern is already used in `non_reverting_select_execute_entry_point_func`:

```cairo
assert is_reverted = 0;
``` [4](#0-3) 

For the reverted case, the constraint should tie `is_reverted` to the `failure_flag` returned by `execute_entry_point` / `select_execute_entry_point_func`, so the prover cannot forge it.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. User submits a valid invoke transaction (version 3, valid signature, sufficient gas).
2. Malicious sequencer includes it in a block and runs the OS.
3. In `execute_invoke_function_transaction`, after `run_validate` succeeds, the sequencer sets the hint `%{ IsReverted %}` to return `is_reverted = 1` (TRUE).
4. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
5. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
6. `charge_fee` executes normally, deducting the fee.
7. The Cairo execution trace is consistent with the program (no constraint was violated).
8. A valid STARK proof is generated and submitted to L1.
9. L1 verifier accepts the proof; the state root reflecting "fee charged, nonce incremented, no execution" is finalized.

The user has lost their fee with no corresponding execution. The sequencer can repeat this for any or all invoke transactions in a block. [2](#0-1) [5](#0-4)
