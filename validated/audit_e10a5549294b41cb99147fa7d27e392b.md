### Title
Proposer-Controlled `l1_gas_price_wei` Bypasses Validator Check, Corrupting Block Hash and ETH Fee Calculation — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `is_block_info_valid` function validates only the FRI-denominated L1 gas prices from the proposer's `ProposalInit`, silently discarding the computed WEI prices. A malicious proposer can set arbitrary `l1_gas_price_wei` / `l1_data_gas_price_wei` values that pass all validator checks, get committed into the block hash via `PartialBlockHashComponents`, and are used for ETH-denominated fee calculation inside the blockifier.

### Finding Description

In `validate_proposal.rs`, `is_block_info_valid` calls `get_l1_prices_in_fri_and_wei` to compute the expected L1 gas prices, but only validates the FRI half:

```rust
// validate_proposal.rs:286-319
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
// ...
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, ...))
``` [1](#0-0) 

The computed WEI prices (`_l1_gas_prices_wei`) are explicitly discarded. The proposer's `l1_gas_price_wei` and `l1_data_gas_price_wei` from `ProposalInit` are never compared against the validator's own computed WEI prices. [2](#0-1) 

These unvalidated WEI prices flow through three critical paths:

**Path 1 — Blockifier execution (ETH fee calculation):**
`initiate_validation` calls `convert_to_sn_api_block_info(init)`, which directly uses the proposer's WEI prices to populate `BlockInfo.gas_prices.eth_gas_prices`:

```rust
// utils.rs:301-302
let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
``` [3](#0-2) 

This `BlockInfo` is passed to the batcher via `ValidateBlockInput.block_info`, and the blockifier uses `eth_gas_prices` for all ETH-denominated fee calculations.

**Path 2 — Block hash commitment:**
`BlockExecutionArtifacts::new` calls `PartialBlockHashComponents::new(&block_info, ...)`, which captures `block_info.gas_prices.l1_gas_price_per_token()` (both wei and fri) into the partial hash:

```rust
// block_hash_calculator.rs:228-234
pub fn new(block_info: &BlockInfo, header_commitments: BlockHeaderCommitments) -> Self {
    Self {
        l1_gas_price: block_info.gas_prices.l1_gas_price_per_token(),
        l1_data_gas_price: block_info.gas_prices.l1_data_gas_price_per_token(),
        ...
    }
}
``` [4](#0-3) 

`calculate_block_hash` then chains both wei and fri prices into the Poseidon hash via `gas_prices_to_hash`: [5](#0-4) 

A manipulated WEI price therefore produces a different `PartialBlockHash` and a different final `BlockHash`.

**Path 3 — Cascading fallback to next block:**
`PreviousBlockInfo::from(init)` stores the proposer's WEI prices verbatim:

```rust
// utils.rs:99-112
impl From<&ProposalInit> for PreviousBlockInfo {
    fn from(init: &ProposalInit) -> Self {
        Self {
            l1_prices_wei: L1PricesInWei {
                l1_gas_price: init.l1_gas_price_wei,
                l1_data_gas_price: init.l1_data_gas_price_wei,
            },
            ...
        }
    }
}
``` [6](#0-5) 

When the L1 gas price provider is unavailable for the next block, `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls back to `previous_block_info`, propagating the corrupted WEI price forward. [7](#0-6) 

### Impact Explanation

**Incorrect fee / gas with economic impact (Critical):** All ETH-denominated transaction fees in the block are computed using the attacker-controlled WEI price. Setting `l1_gas_price_wei = u128::MAX` causes every ETH-fee transaction to be charged the maximum possible fee or to revert due to fee overflow, directly harming users.

**Wrong block hash committed (Critical):** The WEI price is hashed into `PartialBlockHashComponents` and ultimately into the final `BlockHash`. The block hash stored in the block-hash contract (used by proof-facts validation in `validate_proof_block_hash`) will be wrong, breaking the proof-facts invariant for any transaction that references this block. [8](#0-7) 

**Wrong RPC fee estimation (High):** `starknet_estimateFee` and `starknet_simulateTransactions` use the stored block header's WEI gas price for pending-block execution context, returning authoritative-looking wrong values. [9](#0-8) 

### Likelihood Explanation

Any consensus participant who becomes a proposer in a given round can trigger this. In a decentralized sequencer, every validator rotates through the proposer role. The attack requires only that the proposer set `l1_gas_price_wei` to an arbitrary value while keeping `l1_gas_price_fri` within the `l1_gas_price_margin_percent` tolerance. No special privilege beyond being a proposer is needed. Validators accept the proposal because `is_block_info_valid` never checks the WEI price.

### Recommendation

In `is_block_info_valid`, validate the WEI prices with the same `within_margin` check applied to the FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

// Existing FRI check:
if !(within_margin(init_proposed.l1_gas_price_fri, l1_gas_prices_fri.l1_gas_price, margin)
    && within_margin(init_proposed.l1_data_gas_price_fri, l1_gas_prices_fri.l1_data_gas_price, margin))
{ ... }

// Add WEI check:
if !(within_margin(init_proposed.l1_gas_price_wei, l1_gas_prices_wei.l1_gas_price, margin)
    && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{ return Err(ValidateProposalError::InvalidBlockInfo(...)); }
```

### Proof of Concept

1. Malicious proposer constructs `ProposalInit` with `l1_gas_price_fri` within the valid margin but `l1_gas_price_wei = u128::MAX`.
2. Validator calls `is_block_info_valid` — only FRI price is checked; WEI price is never compared. Validation passes.
3. `initiate_validation` calls `convert_to_sn_api_block_info(init)` → `BlockInfo.gas_prices.eth_gas_prices.l1_gas_price = u128::MAX`.
4. Batcher executes all transactions with ETH gas price = `u128::MAX`; every ETH-fee transaction is charged the maximum fee or reverts.
5. `BlockExecutionArtifacts::new` builds `PartialBlockHashComponents` with `l1_gas_price.price_in_wei = u128::MAX`.
6. `calculate_block_hash` hashes the corrupted WEI price into the block hash; the committed block hash diverges from what any honest node would compute.
7. The corrupted WEI price is stored in `PreviousBlockInfo` and used as fallback for the next block if the L1 gas price provider is temporarily unavailable, cascading the corruption.

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
