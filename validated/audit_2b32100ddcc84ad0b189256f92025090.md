### Title
Missing Chain-Domain Binding in `deployToken` Signature Allows Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` and `deploy_token` in `omni_bridge.cairo` verify an MPC-produced ECDSA signature over a `MetadataPayload` that contains only `[PayloadType::Metadata, token, name, symbol, decimals]`. No chain identifier (`omniBridgeChainId`, EVM `block.chainid`, or contract address) is included in the signed data. Because the same MPC-derived key (`nearBridgeDerivedAddress`) is shared across all EVM deployments of the bridge, a valid `deployToken` signature produced for one chain is cryptographically valid on every other chain where the bridge is deployed. An unprivileged attacker can replay the signature to deploy a token on any chain the bridge operator has not yet activated, permanently blocking legitimate deployment on that chain.

---

### Finding Description

**EVM — `OmniBridge.sol::deployToken`**

The signed hash is constructed as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

No `omniBridgeChainId`, no `block.chainid`, and no contract address appear in the hash.

**Contrast with `finTransfer`**, which correctly binds the payload to the destination chain by embedding `omniBridgeChainId` twice:

```solidity
bytes1(omniBridgeChainId),   // token chain
...
bytes1(omniBridgeChainId),   // recipient chain
``` [2](#0-1) 

**Starknet — `omni_bridge.cairo::deploy_token`**

`MetadataPayload::to_borsh()` serializes only `[PayloadType::Metadata, token, name, symbol, decimals]` — no chain ID:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [3](#0-2) 

`deploy_token` passes this chain-free encoding directly to `_verify_borsh_signature`:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(), signature);
``` [4](#0-3) 

Whereas `fin_transfer` correctly passes `chain_id`:

```cairo
_verify_borsh_signature(
    ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature,
);
``` [5](#0-4) 

**NEAR signing side** — `MetadataPayload` is defined without any chain field:

```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
``` [6](#0-5) 

The NEAR bridge signs this payload with a single fixed `SIGN_PATH` and `key_version: 0`, producing one MPC-derived address used as `nearBridgeDerivedAddress` on all EVM chains. [7](#0-6) 

---

### Impact Explanation

**High — Proof/signature verification bypass enabling unauthorized token deployment.**

1. **Unauthorized token deployment on unintended chains.** The bridge operator signs a `MetadataPayload` for token T on chain A (e.g., Ethereum). An attacker observes the signature and submits it to the OmniBridge on chain B (e.g., Arbitrum, Base, or any other EVM chain sharing the same `nearBridgeDerivedAddress`). The signature passes verification because the hash is chain-agnostic.

2. **Permanent DoS on legitimate deployment.** After the replay, `isBridgeToken[nearToEthToken[metadata.token]]` is set to `true` on chain B. Any subsequent legitimate `deployToken` call for the same NEAR token ID on chain B reverts with `ERR_TOKEN_EXIST`, permanently blocking the bridge operator from deploying the token with correct parameters (e.g., updated decimals, correct implementation address). [8](#0-7) 

3. **Token mapping corruption.** The attacker-deployed token on chain B is registered in `nearToEthToken` and `ethToNearToken`. The NEAR bridge will pick up the `DeployToken` event and record this address. If the bridge operator later deploys the bridge on chain B with a different token implementation or different decimals normalization, the mapping is already poisoned.

---

### Likelihood Explanation

**High.** The Omni Bridge is explicitly designed to be deployed on multiple EVM chains simultaneously (Ethereum, Arbitrum, Base, Abstract, etc.). The `nearBridgeDerivedAddress` is derived from the same NEAR MPC key across all deployments. Every `deployToken` transaction on any chain is publicly visible on-chain. Any unprivileged observer can extract the `signatureData` and `MetadataPayload` from a confirmed transaction on chain A and immediately submit them to the bridge on chain B. No special access, leaked key, or privileged role is required.

---

### Recommendation

Include `omniBridgeChainId` (and optionally `address(this)`) in the `MetadataPayload` hash, mirroring the pattern already used in `finTransfer`:

**EVM (`OmniBridge.sol`):**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet (`bridge_types.cairo`):**
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
+   borsh_bytes.append_byte(chain_id);
    ...
}
```

**NEAR (`omni-types/src/lib.rs`):**
```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
+   pub chain_id: u8,   // destination chain
    pub token: String,
    ...
}
```

The NEAR bridge must populate `chain_id` with the destination chain's `omniBridgeChainId` before signing, and the EVM/Starknet verifiers must include it in the hash.

### Proof of Concept

1. Bridge operator calls `log_metadata` on NEAR for token `usdc.near`, targeting Ethereum (chain ID `0`). NEAR MPC signs `keccak256(borsh([Metadata, "usdc.near", "USD Coin", "USDC", 6]))`.

2. The signed `deployToken(signatureData, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` transaction is confirmed on Ethereum mainnet.

3. Attacker extracts `signatureData` from the Ethereum transaction calldata.

4. Attacker submits the identical call to the OmniBridge deployed on Arbitrum (chain ID `1` in bridge terms), which shares the same `nearBridgeDerivedAddress`.

5. `OmniBridge.deployToken` on Arbitrum computes `keccak256(borsh([Metadata, "usdc.near", "USD Coin", "USDC", 6]))` — identical hash — and `ECDSA.recover` returns `nearBridgeDerivedAddress`. Signature passes.

6. A new `BridgeToken` proxy is deployed on Arbitrum for `usdc.near`. `isBridgeToken`, `nearToEthToken`, and `ethToNearToken` are set.

7. When the bridge operator later tries to legitimately deploy `usdc.near` on Arbitrum, the call reverts with `ERR_TOKEN_EXIST`. [9](#0-8) [10](#0-9) [11](#0-10)
