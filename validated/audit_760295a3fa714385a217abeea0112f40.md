### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt value as the new class hash — including `0` (`UNINITIALIZED_CLASS_HASH`) and any undeclared class hash — without enforcing that the replacement target is a legitimately declared class. This is the direct analog of M-06: a critical protocol value (a contract's class hash) that should only be set to valid declared classes can instead be set to an invalid sentinel value, permanently disabling the contract and freezing any funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` reads `request.class_hash` directly from the syscall pointer and writes it into the contract's `StateEntry` with no Cairo-level assertion validating the value:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 is the developers' own acknowledgment that this check is absent. There is no `assert_not_zero(class_hash)` and no lookup into `contract_class_changes` to confirm the hash is declared.

`UNINITIALIZED_CLASS_HASH` is defined as the constant `0` in `commitment.cairo`:

```cairo
const UNINITIALIZED_CLASS_HASH = 0;
```

`deploy_contract.cairo` uses this same sentinel to enforce the "deploy only once" invariant:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
assert state_entry.nonce = 0;
```

Because `execute_replace_class` imposes no lower bound on the replacement value, a contract can issue `replace_class(0)`, writing `UNINITIALIZED_CLASS_HASH` back into its own `StateEntry`. After this write, the contract's class hash is `0` — a value that is never present in the declared-class tree — making the contract permanently non-callable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to `0` (or any undeclared hash), every future call to that contract will fail at class resolution: the OS will attempt to look up the class for hash `0`, find nothing, and the call will revert. Any ERC-20 balances, liquidity positions, or other assets whose ownership is tracked inside that contract's storage become permanently inaccessible. Because `UNINITIALIZED_CLASS_HASH = 0` is a reserved sentinel that can never be the output of a valid Sierra class hash computation, the freeze is irreversible.

---

### Likelihood Explanation

Any contract deployer — an unprivileged role on StarkNet — can deploy a contract whose code calls `replace_class(0)` (or `replace_class(<undeclared_hash>)`) in any externally callable function. The attacker does not need a leaked key, operator access, or any privileged role. The entry path is:

1. Attacker deploys a contract that exposes a public function calling the `replace_class` syscall with `class_hash = 0`.
2. Users interact with the contract and deposit funds (e.g., the contract is a vault, AMM, or bridge).
3. Attacker calls the function; the OS processes it through `execute_replace_class` with no validation.
4. The contract's class hash becomes `0`; all future calls revert.
5. Funds are permanently frozen.

The OS is a necessary step in this path: it is the only layer that could enforce the declared-class invariant, and it currently does not.

---

### Recommendation

Add a Cairo-level assertion inside `execute_replace_class` that the replacement class hash is both non-zero and present in the `contract_class_changes` dictionary (i.e., has been declared in the current or a prior block):

```cairo
// Reject UNINITIALIZED_CLASS_HASH.
assert_not_zero(class_hash);

// Verify the class has been declared (compiled_class_hash != 0).
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the existing "declare only once" guard in `execute_declare_transaction` (`prev_value=0` enforces a class may be declared only once) and the "deploy only once" guard in `deploy_contract` (`assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH`).

---

### Proof of Concept

**Step 1 — Attacker deploys a malicious vault contract (class `C1`) at address `X`.**
The contract exposes `deposit()` and `freeze()`. `freeze()` issues `replace_class(0)`.

**Step 2 — Users call `deposit()`, sending tokens to `X`.**
The ERC-20 contract records `balances[X] = N`.

**Step 3 — Attacker calls `freeze()`.**
The OS dispatches the `replace_class` syscall to `execute_replace_class`:

```
execute_replace_class(contract_address=X)
  class_hash = 0                          // from request, no validation
  new_state_entry = StateEntry(
      class_hash=0,                       // UNINITIALIZED_CLASS_HASH
      storage_ptr=state_entry.storage_ptr,
      nonce=state_entry.nonce,
  )
  dict_update(key=X, ..., new_value=new_state_entry)
```

No Cairo assertion fires. The state transition is accepted and committed.

**Step 4 — Any subsequent call to `X` fails.**
The OS reads `state_entry.class_hash = 0`, finds no class, and reverts. The `N` tokens credited to `X` in the ERC-20 contract are permanently inaccessible. [1](#0-0) [2](#0-1) [3](#0-2)
