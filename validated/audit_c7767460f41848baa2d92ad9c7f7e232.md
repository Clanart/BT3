### Title
Fee-on-Transfer / Malicious ERC20 Inflates Bridge Event Amount, Breaking Collateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter directly in the `InitTransfer` event without verifying the actual balance delta received by the contract. A malicious or fee-on-transfer ERC20 token can cause the bridge to emit an event claiming more tokens were deposited than were actually received, causing the NEAR side to mint an inflated supply of wrapped tokens against an undercollateralized EVM vault.

---

### Finding Description

In `OmniBridge.initTransfer`, for any ERC20 token that is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // declared by caller
);
``` [1](#0-0) 

Immediately after, without checking the actual balance delta, the contract emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // still the caller-declared value
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

`SafeERC20.safeTransferFrom` only verifies that `transferFrom` returns `true`; it does not verify that `balanceOf(address(this))` increased by `amount`. A malicious ERC20 whose `transferFrom` silently transfers fewer tokens (e.g., transfers 1 wei regardless of the declared value, or applies a hidden fee) will still cause the event to be emitted with the full caller-declared `amount`.

There is no whitelist gate on the `else` branch — any ERC20 address that is not already registered as `isBridgeToken` or `customMinters` reaches this path: [3](#0-2) 

`logMetadata` is also permissionless, allowing any caller to register an arbitrary ERC20 on the NEAR side: [4](#0-3) 

Once the token is registered on NEAR (via the `log_metadata_callback` → MPC signing flow), the NEAR bridge's `fin_transfer_callback` reads the `amount` field from the proven EVM event and mints that many wrapped tokens to the recipient: [5](#0-4) 

The NEAR side has no mechanism to independently verify how many tokens the EVM vault actually received; it trusts the event's `amount` field entirely.

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

An attacker who deposits 1 wei of a malicious ERC20 but declares `amount = 1,000,000e18` causes the NEAR bridge to mint 1,000,000e18 wrapped tokens. The EVM vault holds only 1 wei. When legitimate holders of the wrapped token attempt to bridge back, the vault is undercollateralized and cannot release the declared amounts, resulting in permanent freezing of user funds for all but the first redeemer. In the worst case the attacker immediately sells the inflated wrapped tokens on NEAR DEXes, extracting real value from liquidity providers.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a malicious ERC20 (trivial, no privileged access).
2. Calling the permissionless `logMetadata` on EVM to register the token on NEAR.
3. Calling `initTransfer` with an inflated `amount`.

No privileged role, leaked key, or colluding operator is needed. The entire path is reachable by any unprivileged EOA.

---

### Recommendation

Record the vault's ERC20 balance before and after `safeTransferFrom` and use the delta as the authoritative deposited amount, consistent with the MonoX fix:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "Zero transfer");
amount = uint128(actualReceived); // use actual delta for event and extension
```

Apply the same pattern to `initTransfer1155` if ERC1155 tokens with non-standard hooks are in scope.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MaliciousERC20 is ERC20 {
    constructor() ERC20("Evil", "EVIL") {
        _mint(msg.sender, 1e18);
    }

    // Always transfers only 1 wei regardless of declared value
    function transferFrom(address from, address to, uint256 /*value*/) public override returns (bool) {
        _transfer(from, to, 1);
        return true;
    }
}
```

Attack sequence:

1. Attacker deploys `MaliciousERC20` and mints 1e18 tokens to themselves.
2. Attacker calls `OmniBridge.logMetadata(maliciousToken)` — permissionless, registers the token on NEAR via MPC-signed metadata event.
3. Attacker approves `OmniBridge` for `1e30` tokens (allowance check passes because `transferFrom` only moves 1 wei).
4. Attacker calls `OmniBridge.initTransfer(maliciousToken, 1e30, 0, 0, nearRecipient, "")`.
   - `safeTransferFrom` calls `MaliciousERC20.transferFrom`, which moves 1 wei and returns `true` — no revert.
   - `emit InitTransfer(..., amount=1e30, ...)` is emitted.
5. Relayer submits proof to NEAR `fin_transfer_callback`.
   - NEAR reads `amount = 1e30` from the proven event.
   - NEAR mints 1e30 wrapped EVIL tokens to the attacker's NEAR account.
6. EVM vault holds 1 wei; NEAR supply is 1e30 — bridge is maximally undercollateralized.
7. Any subsequent user bridging back wrapped EVIL tokens finds the vault empty; their funds are permanently locked. [6](#0-5) [7](#0-6)
