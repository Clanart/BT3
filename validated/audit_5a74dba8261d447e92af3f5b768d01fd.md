### Title
Integer Wrapping in `check_time_locks` Relative Timelock Arithmetic Enables Timelock Bypass and Consensus Divergence — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` evaluates relative timelock conditions (`ASSERT_HEIGHT_RELATIVE`, `ASSERT_SECONDS_RELATIVE`, `ASSERT_BEFORE_HEIGHT_RELATIVE`, `ASSERT_BEFORE_SECONDS_RELATIVE`) using either `saturating_add` (`nowrap=true`) or `wrapping_add` (`nowrap=false`). When `nowrap=false` is active, an unprivileged spender can supply a max-value relative timelock argument (e.g., `height_relative = 0xffff_ffff`) that wraps the addition to a small number, causing the timelock check to produce the opposite result from what the puzzle author intended. This is the direct analog of the external report's "arithmetic bounds not checked before a critical operation" class.

---

### Finding Description

In `check_time_locks`, the relative timelock checks branch on the `nowrap` boolean:

```rust
// ASSERT_HEIGHT_RELATIVE path
if nowrap {
    if prev_transaction_block_height
        < unspent.confirmed_block_index.saturating_add(height_relative)
    { return Err(...AssertHeightRelativeFailed); }
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{ return Err(...AssertHeightRelativeFailed); }
``` [1](#0-0) 

The same dual-path pattern applies to `seconds_relative`, `before_height_relative`, and `before_seconds_relative`: [2](#0-1) 

The `nowrap` flag is **not** derived from `get_flags_for_height_and_constants` and is not part of `ConsensusFlags`; it is passed as a separate caller-controlled boolean: [3](#0-2) 

The Python binding exposes `nowrap` directly to the full-node layer: [4](#0-3) 

The condition parser accepts `height_relative` up to `u32::MAX` (`0xffff_ffff`) as a valid, non-overflowing CLVM atom (5 bytes with a leading zero): [5](#0-4) 

The tests explicitly document the divergent outcomes for max-value inputs:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
height_relative=0xffff_ffff  nowrap=true  → Err  (correctly blocked)
height_relative=0xffff_ffff  nowrap=false → Ok   (timelock bypassed!)

// 200 >= 100 + 0xffff_ffff -> Ok with nowrap (saturates), Err without (wraps)
before_height_relative=0xffff_ffff  nowrap=true  → Ok   (no-op, correct)
before_height_relative=0xffff_ffff  nowrap=false → Err  (coin permanently locked!)
``` [6](#0-5) 

The Python test suite confirms the same divergence: [7](#0-6) 

---

### Impact Explanation

**High — Timelock validation bypass (under `nowrap=false`):**

With `confirmed_block_index = C` and `height_relative = 0xffff_ffff`:

```
C.wrapping_add(0xffff_ffff) = C - 1  (mod 2^32)
```

The check `prev_height < C - 1` is false for any `prev_height >= C - 1`, so the condition passes immediately. A coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE(0xffff_ffff)` — intending to lock it for ~4 billion blocks — is spendable at the very next block. This is a direct timelock validation bypass enabling unauthorized spend acceptance.

**Critical — Consensus divergence (during soft-fork transition):**

Because `nowrap` is not tied to any block height in `get_flags_for_height_and_constants`, nodes running different software versions (or the same version with different `nowrap` settings) will reach opposite validity verdicts for a spend bundle containing `height_relative=0xffff_ffff`. A node with `nowrap=false` accepts it; a node with `nowrap=true` rejects it. This is deterministic consensus divergence triggered by a valid, unprivileged spend bundle.

---

### Likelihood Explanation

The attacker-controlled entry path is fully unprivileged: any user can craft a CLVM puzzle that emits `ASSERT_HEIGHT_RELATIVE` with argument `0xffff_ffff` (encoded as the 5-byte atom `0x00 0xff 0xff 0xff 0xff`, which passes `sanitize_uint` with `max_size=4`). No special keys, governance access, or validator collusion is required. The divergence is deterministic and reproducible.

---

### Recommendation

1. **Remove the `wrapping_add` path entirely.** The `nowrap=false` branch has no safe use case; `saturating_add` is the only semantically correct behavior for relative timelock arithmetic.
2. **Tie `nowrap` activation to a consensus-enforced block height** via `get_flags_for_height_and_constants` and `ConsensusFlags`, so all nodes transition atomically at the same height.
3. **Reject `height_relative` or `seconds_relative` values that would cause `confirmed + relative` to overflow even under saturating arithmetic** (i.e., values where `confirmed + relative >= u32::MAX` / `u64::MAX`) as an additional defense-in-depth measure.

---

### Proof of Concept

```python

# Under nowrap=False, confirmed_height=10, prev_height=15:
#   10.wrapping_add(0xffff_ffff) = 9
#   15 < 9  →  False  →  Ok()  (timelock bypassed)
#
# Under nowrap=True:
#   10.saturating_add(0xffff_ffff) = u32::MAX
#   15 < u32::MAX  →  True  →  Err(AssertHeightRelativeFailed)

from chia_rs import check_time_locks, SpendBundleConditions, SpendConditions, CoinRecord

coin_record = CoinRecord(coin, confirmed_block_index=10, spent_block_index=0,
                         coinbase=False, timestamp=10000)
conds = make_test_conds(height_relative=0xFFFF_FFFF)

# nowrap=False (legacy): returns None (accepted) — timelock bypassed
result_legacy = check_time_locks({coin.name(): coin_record}, conds,
                                  prev_transaction_block_height=15,
                                  timestamp=10150, nowrap=False)
assert result_legacy is None  # passes — WRONG

# nowrap=True (new): returns 13 (AssertHeightRelativeFailed) — correctly blocked
result_new = check_time_locks({coin.name(): coin_record}, conds,
                               prev_transaction_block_height=15,
                               timestamp=10150, nowrap=True)
assert result_new == 13  # fails — CORRECT

pub fn get_flags_for_height_and_constants(
    prev_tx_height: u32,
    constants: &ConsensusConstants,
) -> ConsensusFlags {
    //  the hard-fork initiated with 2.0. To activate June 2024
    //  * costs are ascribed to some unknown condition codes, to allow for
    // soft-forking in new conditions with cost
    //  * a new condition, SOFTFORK, is added which takes a first parameter to
    //    specify its cost. This allows soft-forks similar to the softfork
    //    operator
    //  * BLS operators introduced in the soft-fork (behind the softfork
    //    guard) are made available outside of the guard.
    //  * division with negative numbers are allowed, and round toward
    //    negative infinity
    //  * AGG_SIG_* conditions are allowed to have unknown additional
    //    arguments
    //  * Allow the block generator to be serialized with the improved clvm
    //   serialization format (with back-references)

    // The soft fork initiated with 2.5.0. The activation date is still TBD.
    // Adds a new keccak256 operator under the softfork guard with extension 1.
    // This operator can be hard forked in later, but is not included in a hard fork yet.

    // In hard fork 2, we enable the keccak operator outside the softfork guard
    let mut flags = ConsensusFlags::empty();
    if prev_tx_height >= constants.hard_fork2_height {
        flags |= ConsensusFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD
            | ConsensusFlags::COST_CONDITIONS
            | ConsensusFlags::ENABLE_SECP_OPS
            | ConsensusFlags::RELAXED_BLS;
    }

    if prev_tx_height >= constants.soft_fork8_height {
        flags |= ConsensusFlags::DISABLE_OP;
    }

    if prev_tx_height >= constants.soft_fork9_height {
        flags |= ConsensusFlags::SIMPLE_GENERATOR
            | ConsensusFlags::CANONICAL_INTS
            | ConsensusFlags::LIMIT_SPENDS;
    }
    flags
}
