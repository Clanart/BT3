### Title
Zero-Output Transfer Permanently Locks/Burns User Tokens Due to Missing Pre-Normalization Check - (File: near/omni-bridge/src/lib.rs)

### Summary

`init_transfer` accepts and irrevocably locks or burns a user's tokens before any check that the decimal-normalized transfer amount is non-zero. When a user bridges a small amount whose normalized value floors to zero, their tokens are permanently lost: burned (for deployed tokens) or locked forever (for native tokens), with no cancellation or recovery path.

### Finding Description

The bridge's `normalize_amount` function performs floor division to convert a NEAR-side token amount to the destination chain's decimal precision:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

For a token with `origin_decimals = 24` and `decimals = 6` (a 18-decimal difference), any amount below `10^18` normalizes to zero.

The only guard against a zero normalized amount lives in `sign_transfer`, called by a relayer **after** the user's tokens have already been consumed:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

But by the time `sign_transfer` is called, `init_transfer_internal` has already:
1. Burned the tokens (for deployed/bridged tokens) via `burn_tokens_if_needed`
2. Locked the tokens (for native tokens) via `lock_tokens_if_needed`
3. Stored the transfer message in `pending_transfers` [3](#0-2) 

`init_transfer` itself only validates `fee.fee < amount`, which passes even when `amount` is tiny: [4](#0-3) 

When `sign_transfer` subsequently panics with `InvalidAmountToTransfer`, the entire transaction reverts. No MPC signing is requested, no callback fires, and the transfer message remains permanently in `pending_transfers`. There is no public `cancel_transfer` function that would allow the user to reclaim their tokens. The tokens are irrecoverably lost.

### Impact Explanation

- **Deployed (bridged) tokens**: Tokens are burned on NEAR via `burn_tokens_if_needed`. They are permanently destroyed. The user receives nothing on the destination chain.
- **Native tokens**: Tokens are locked in the bridge contract via `lock_tokens_if_needed`. The pending transfer can never be signed (normalized amount = 0), so the tokens are permanently frozen in the bridge with no recovery path.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

Any unprivileged user can trigger this by calling `ft_transfer_call` on a token whose `origin_decimals` significantly exceeds `decimals` (e.g., a 24-decimal NEAR token bridging to a 6-decimal EVM representation). The user only needs to send an amount smaller than `10^(origin_decimals - decimals)`. This is a realistic scenario for tokens with large decimal differences, and the user has no way to know in advance that their amount will normalize to zero — there is no on-chain pre-check or simulation available.

### Recommendation

Add a zero-output guard inside `init_transfer`, before tokens are locked or burned. Compute the normalized amount at deposit time and revert if it is zero:

```rust
// In init_transfer, after building transfer_message and before init_transfer_internal:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the existing guard in `sign_transfer` but places it at the correct point — before any irreversible state change.

### Proof of Concept

**Setup**: Token registered with `origin_decimals = 24`, `decimals = 6` (18-decimal difference). Destination chain is Ethereum.

**Steps**:

1. Alice calls `ft_transfer_call` on the token contract with `amount = 999` and a bridge `InitTransferMsg` targeting her Ethereum address.

2. `ft_on_transfer` → `init_transfer` is called. The only validation is:
   ```
   fee.fee (= 0) < amount (= 999)  ✓
   ``` [4](#0-3) 

3. `init_transfer_internal` is called. For a deployed token, `burn_tokens_if_needed` burns 999 tokens. `U128(0)` is returned — the NEP-141 standard interprets this as "all tokens consumed." [3](#0-2) 

4. A relayer calls `sign_transfer` for Alice's transfer. `normalize_amount` is computed:
   ```
   999 / 10^18 = 0
   ```
   The `require!(amount_to_transfer > 0, ...)` check panics. The transaction reverts. [2](#0-1) 

5. Alice's 999 tokens are permanently burned on NEAR. The transfer message sits in `pending_transfers` forever. Alice receives nothing on Ethereum. There is no cancel function to recover her funds.
