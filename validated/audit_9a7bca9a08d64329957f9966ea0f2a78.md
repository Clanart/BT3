### Title
Missing `from` Token Account Owner Verification Allows Delegate to Steal User Tokens via Bridge Transfer - (File: `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` Anchor account struct in the Solana bridge does not enforce that the `user` signer is the **owner** of the `from` token account — only that `user` has SPL token authority (owner **or delegate**). Any account that has been approved as a delegate for a victim's token account can call `init_transfer`, bridge the victim's tokens to an attacker-controlled address on another chain, and steal them.

---

### Finding Description

In `InitTransfer`, the `from` account is declared with only `token::mint` and `token::token_program` constraints:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint, meaning the bridge does **not** verify that `user` owns `from`. The `user` field is only required to be a `Signer`:

```rust
#[account(
    mut,
    owner = common.system_program.key(),
)]
pub user: Signer<'info>,
``` [2](#0-1) 

In `process()`, the `transfer_checked` (native token path) and `burn` (bridged token path) CPIs both pass `self.user` as the authority:

```rust
TransferChecked {
    from: self.from.to_account_info(),
    to: vault.to_account_info(),
    authority: self.user.to_account_info(),
    mint: self.mint.to_account_info(),
},
``` [3](#0-2) 

```rust
Burn {
    mint: self.mint.to_account_info(),
    from: self.from.to_account_info(),
    authority: self.user.to_account_info(),
},
``` [4](#0-3) 

The SPL token program accepts `user` as authority if `user` is either the **owner** or an **approved delegate** of `from`. The bridge does not distinguish between these two cases. The Wormhole message records `self.user.key()` as the sender — the attacker's key — not the victim's:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
``` [5](#0-4) 

---

### Impact Explanation

An attacker who holds a delegate approval over a victim's SPL token account (for any amount) can:

1. Call `init_transfer` with `from = victim_token_account`, `user = attacker` (signer), and `recipient = attacker_address_on_near_or_evm`.
2. The SPL token program accepts the CPI because `attacker` is a valid delegate.
3. The victim's tokens are locked in the vault or burned, and a Wormhole message is emitted crediting the attacker as sender with the attacker's chosen recipient.
4. The attacker finalizes the transfer on the destination chain and receives the victim's tokens.

This is **direct theft of user assets** across chains — a Critical impact under the allowed scope.

---

### Likelihood Explanation

SPL token delegation is a standard, widely-used primitive. Users routinely approve DEXes, lending protocols, and other DeFi programs as delegates for their token accounts. An attacker monitoring on-chain state can enumerate all `Approve` instructions for any token and immediately exploit any active delegation by calling `init_transfer` before the delegation is revoked. No privileged access, leaked keys, or colluding parties are required — only a valid delegate approval, which is a normal user action.

---

### Recommendation

Add the `token::authority = user` constraint to the `from` account declaration in `InitTransfer`:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,   // <-- enforce owner, not just delegate
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This ensures only the actual owner of the token account can initiate a bridge transfer, rejecting delegates.

---

### Proof of Concept

1. Alice holds 1000 USDC in her Solana token account `alice_ata`. She has previously approved `bob_wallet` as a delegate for 500 USDC (e.g., for a DEX swap that was never executed).
2. Bob constructs a `init_transfer` transaction:
   - `from` = `alice_ata`
   - `user` = `bob_wallet` (Bob signs)
   - `recipient` = Bob's NEAR address
   - `amount` = 500 (within the delegated amount)
3. Anchor validates: `from.mint == mint` ✓, `from.token_program == token_program` ✓. No owner check.
4. The `transfer_checked` CPI succeeds: SPL token program sees `bob_wallet` is a valid delegate for `alice_ata` with sufficient allowance.
5. 500 USDC are moved from `alice_ata` to the vault. A Wormhole message is emitted with sender = `bob_wallet`, recipient = Bob's NEAR address.
6. Bob finalizes on NEAR and receives 500 USDC worth of bridged tokens. Alice's funds are permanently stolen. [6](#0-5)
