### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key/Hash Guards, Corrupting Tree Root and Enabling Forged Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash existence checks that `MerkleBlob::insert` enforces for all items beyond the first two. Supplying a batch that contains a repeated `KeyId` or `Hash` silently writes two leaf nodes for the same key into the blob, corrupts the `BlockStatusCache`, and produces an incorrect Merkle root — directly enabling forged DataLayer inclusion/exclusion proofs.

### Finding Description

`MerkleBlob::insert` guards against duplicate state at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `self.insert()` (with those guards) only for the first two items when the tree has ≤ 1 existing leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items are written directly via `insert_entry_to_blob` — no duplicate check whatsoever:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` — silently overwriting the existing cache entry for the duplicate key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index); // overwrites silently
}
``` [4](#0-3) 

The result mirrors the original report exactly:

| State | Outcome |
|---|---|
| Blob | Two physical leaf nodes for the same `KeyId` |
| `key_to_index` cache | Points only to the *last* inserted index |
| First (orphaned) leaf | Exists in blob, unreachable via cache |
| Merkle root | Computed over *both* leaves — incorrect |
| `get_leaf_by_key` | Returns wrong leaf for the key |
| `check_integrity` | Fails: `leaf_count` (blob) ≠ `key_to_index_cache_length` (HashMap) | [5](#0-4) 

### Impact Explanation

A corrupted Merkle root means every subsequent proof of inclusion or exclusion generated from that `MerkleBlob` is computed against a tree that contains phantom (duplicate) leaves. An attacker who controls the batch input can:

1. Force a specific root hash that differs from the honest root.
2. Obtain a valid-looking proof of inclusion for a key/value pair that was never legitimately inserted (the orphaned leaf is still physically present in the blob and can be walked by a proof generator that traverses raw blob bytes rather than the cache).
3. Cause `check_integrity` failures that halt DataLayer operations on the affected store.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is a public API exposed directly through the Python wheel bindings: [6](#0-5) 

Any Python caller — including DataLayer client code that aggregates pending key-value updates before committing — can pass a `Vec` containing duplicate `KeyId`s. No privilege is required; the caller need only supply a list with a repeated key. The bug is also reachable from `deltas.rs` and `proof_of_inclusion.rs` which call `batch_insert` internally. [7](#0-6) 

### Recommendation

Add the same existence guards at the top of the `batch_insert` loop body that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob call
}
```

Additionally, `add_leaf` should assert (or return an error) when `HashMap::insert` returns `Some(...)`, signalling an unexpected overwrite, to make future regressions immediately visible.

### Proof of Concept

```rust
// Tree starts empty (leaf_count == 0, so first two items go through insert())
let mut blob = MerkleBlob::new(Vec::new()).unwrap();

let key   = KeyId(1);
let hash1 = Hash(Bytes32::new([0xAA; 32]));
let hash2 = Hash(Bytes32::new([0xBB; 32]));
let hash3 = Hash(Bytes32::new([0xCC; 32]));

// Batch of 3: items 0 and 1 are popped first (via pop()) and go through insert().
// Item 2 (key=1, hash=hash1) goes through the unchecked loop.
// key=1 was already inserted as item 0 (hash=hash1 via pop order).
blob.batch_insert(vec![
    ((key,       ValueId(10)), hash1),  // duplicate key — processed in unchecked loop
    ((KeyId(2),  ValueId(20)), hash2),  // goes through insert() (popped 2nd)
    ((key,       ValueId(30)), hash3),  // goes through insert() (popped 1st)
]).unwrap(); // succeeds — no error raised

// Two leaves for KeyId(1) now exist in the blob.
// Cache points only to the last-written index.
// Root hash is computed over both leaves → incorrect committed root.
blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheLength
