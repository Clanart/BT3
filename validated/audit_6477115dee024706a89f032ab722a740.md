After a thorough trace of the full `fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_near` → `fin_transfer_send_tokens_callback` chain, I found a concrete issue.

---

### Title
Replay of Finalized Inbound Transfer via `remove_fin_transfer` After Failed `ft_transfer_call` — (`near/omni-bridge/src/lib.rs`)

### Summary

When `fin_transfer` delivers tokens to a NEAR recipient using `ft_transfer_call` (i.e., the transfer has a non-empty `msg`), and the recipient contract's `ft_on_transfer` returns the full amount (signaling "unused"), the callback `fin_transfer_send_tokens_callback` removes the transfer from `finalised_transfers`. This erasure of the replay-protection record allows the same source-chain proof to be submitted and accepted a second time, delivering tokens twice from a single on-chain event.

### Finding Description

**Step 1 — Replay guard is set at the start of `process_fin_transfer_to_near`:**

`add_fin_transfer` inserts the `TransferId` into `finalised_transfers` and panics if it is already present. [1](#0-0) 

**Step 2 — Token delivery via `ft_transfer_call` when `msg` is non-empty:**

`send_tokens` dispatches `ft_transfer_call` when `msg` is non-empty, which calls `ft_on_transfer` on the recipient contract. [2](#0-1) 

**Step 3 — `is_refund_required` returns `true` when the recipient returns the full amount:**

If `ft_on_transfer` returns `0` (used amount), `is_refund_required` returns `true`, triggering the refund branch. [3](#0-2) 

**Step 4 — `remove_fin_transfer` erases the replay-protection record:**

In the refund branch of `fin_transfer_send_tokens_callback`, `remove_fin_transfer` removes the `TransferId` from `finalised_transfers`, restoring the state as if the transfer never happened. [4](#0-3) [5](#0-4) 

**Step 5 — The same proof can now be submitted again:**

Because `finalised_transfers` no longer contains the `TransferId`, a second call to `fin_transfer` with the identical proof passes `add_fin_transfer` and delivers tokens a second time.

### Impact Explanation

- **Non-deployed (locked) tokens**: The bridge holds locked tokens from the source chain. First delivery: tokens sent to recipient, returned, lock state restored, finalization record erased. Second delivery: tokens sent again and accepted. Result: tokens delivered twice from one source-chain event — a direct double-spend from the bridge's locked balance.
- **Deployed (minted) tokens**: First delivery: tokens minted, returned, burned, finalization record erased. Second delivery: tokens minted again and accepted. Result: unbacked supply minted — a double-mint.

### Likelihood Explanation

The attack requires:
1. The attacker initiates a transfer on the source chain with their own contract as recipient and a non-empty `msg` (permissionless on EVM/Solana/StarkNet).
2. A trusted relayer submits `fin_transfer` with the proof. The attacker's contract returns all tokens on the first call.
3. The relayer observes the delivery as "failed" (the `FailedFinTransferEvent` is emitted) and retries — standard relayer behavior.
4. On the second submission, the attacker's contract accepts the tokens.

The attacker does not need to be a trusted relayer. They only need to control the recipient contract and rely on a legitimate relayer retrying after an apparent delivery failure.

### Recommendation

Do **not** remove the `TransferId` from `finalised_transfers` on a failed `ft_transfer_call`. The finalization record must be permanent once set. Instead, handle the refund by storing a claimable balance for the original sender, or by marking the transfer as "failed-but-finalized" in a separate set. The invariant must be: once a `TransferId` is inserted into `finalised_transfers`, it is never removed.

### Proof of Concept

```
1. Attacker deploys MaliciousRecipient on NEAR with:
     ft_on_transfer(call_count == 1) → return full amount (refund)
     ft_on_transfer(call_count == 2) → return 0 (accept)

2. Attacker initiates transfer on Ethereum:
     initTransfer(token, amount, recipient=MaliciousRecipient, msg="trigger")

3. Relayer calls fin_transfer(proof) → fin_transfer_callback:
     - add_fin_transfer(transfer_id) → inserts into finalised_transfers ✓
     - send_tokens → ft_transfer_call → MaliciousRecipient.ft_on_transfer returns amount
     - fin_transfer_send_tokens_callback: is_refund_required=true
       → revert_lock_actions (re-locks tokens)
       → remove_fin_transfer(transfer_id) ← ERASES REPLAY GUARD
       → emits FailedFinTransferEvent

4. Relayer retries fin_transfer(same proof):
     - add_fin_transfer(transfer_id) → succeeds (record was removed)
     - send_tokens → ft_transfer_call → MaliciousRecipient.ft_on_transfer returns 0
     - fin_transfer_send_tokens_callback: is_refund_required=false
       → tokens remain at MaliciousRecipient
       → emits FinTransferEvent

Result: tokens delivered twice from one source-chain event.

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {

                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.

                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
