### Title
Fee-on-Transfer Token in `initTransfer` Causes Unbacked Minting on Destination Chain - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without accounting for transfer fees deducted by fee-on-transfer ERC-20 tokens. The bridge actually receives fewer tokens than `amount`, but the event — which is the authoritative proof consumed by the NEAR prover — records the full `amount`. The destination chain then mints or releases `amount` tokens, creating unbacked supply and draining the bridge's reserves.

---

### Finding Description

In `OmniBridge.initTransfer` (lines 373–437), when `tokenAddress` is a standard (non-bridge, non-custom-minter) ERC-20, the bridge executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← requested amount
);
```

For a fee-on-transfer token, the bridge receives `amount - transfer_fee` tokens. Immediately after, the function emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← full requested amount, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
```

The NEAR prover parses this `InitTransfer` event log and constructs an `InitTransferMessage` with `amount` as the bridged value. The NEAR bridge then mints or releases `amount` tokens to the recipient on the destination chain. Because the EVM bridge only holds `amount - transfer_fee`, the bridge is undercollateralized by exactly `transfer_fee` per such transaction.

There is no balance-before/balance-after check anywhere in `initTransfer` to detect the discrepancy.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token causes the bridge to emit a proof for more tokens than it actually holds. The destination chain mints or releases the full `amount`, while the EVM bridge's actual reserve is `amount - transfer_fee`. Repeated calls accumulate a growing shortfall. When legitimate users later attempt to withdraw their tokens back to EVM, the bridge cannot honor all redemptions — constituting a direct, permanent loss of user funds and a break in bridge collateralization.

This matches the allowed impact: **High — Balance/accounting corruption that breaks bridge collateralization or misdirects value**, and potentially **Critical — Unauthorized mint of bridged assets**.

---

### Likelihood Explanation

Fee-on-transfer tokens (e.g., USDT on some chains, PAXG, STA, and many DeFi tokens) are widely deployed ERC-20s. Any unprivileged user can call `initTransfer` with such a token. No special role, leaked key, or privileged access is required. The bridge does not whitelist tokens before they can be used in `initTransfer` (token registration is separate from transfer initiation), so this is reachable by any token holder.

---

### Recommendation

Measure the bridge's actual token balance change around the `safeTransferFrom` call and use the measured delta — not the caller-supplied `amount` — in the emitted event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived (cast to uint128) in the event and downstream logic
```

Alternatively, explicitly disallow fee-on-transfer tokens via a token registry check, or document and enforce that only non-rebasing, non-fee-on-transfer tokens may be bridged.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC-20 token `FOT` that deducts 1% on every transfer.
2. Call `OmniBridge.initTransfer(FOT, 1000, 0, 0, "recipient.near", "")`.
3. Inside `initTransfer`, `safeTransferFrom` moves `1000` tokens from the caller; the bridge receives `990` (1% fee deducted).
4. The `InitTransfer` event is emitted with `amount = 1000`.
5. A relayer submits this event as proof to the NEAR bridge via `fin_transfer`.
6. The NEAR bridge mints `1000` FOT-equivalent tokens to `recipient.near`.
7. The EVM bridge holds only `990` FOT, but has issued claims for `1000` — a `10`-token shortfall per call.
8. Repeating this 100 times creates a `1000`-token shortfall; the bridge cannot honor all withdrawals.
