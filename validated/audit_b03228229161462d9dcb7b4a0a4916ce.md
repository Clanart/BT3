Audit Report

## Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens Up to Max Caps - (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address in `addLiquidityExactShares` and `addLiquidityWeighted` and stores it as the trusted callback authority in transient storage without verifying it against `MetricOmmPoolFactory`. A malicious pool contract passed as the `pool` argument can immediately re-enter `metricOmmModifyLiquidityCallback` during its `addLiquidity` execution, pass both callback guards, and cause `pay()` to pull up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 directly from the victim's wallet.

## Finding Description

In `_addLiquidity` (line 193), `_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1)` stores the caller-supplied `pool` address as the sole trusted callback authority in transient storage with no factory registry check. [1](#0-0) 

The callback `metricOmmModifyLiquidityCallback` enforces only two guards:
1. `msg.sender != expectedPool` — passes trivially when `msg.sender` is the malicious pool (it is the stored `expectedPool`).
2. `amount0Delta > max0 || amount1Delta > max1` — passes when the malicious pool requests exactly the user-supplied caps. [2](#0-1) 

After both guards pass, the callback calls `IMetricOmmPool(msg.sender).getImmutables()` to obtain token addresses — fully attacker-controlled — then calls `pay(token0, payer, msg.sender, amount0Delta)`. [3](#0-2) 

`pay()` resolves to `IERC20(token).safeTransferFrom(payer, recipient, value)` when `payer != address(this)`, transferring tokens directly from the victim to the malicious pool. [4](#0-3) 

`IMetricOmmPoolFactory.isPool()` exists as a public view function and would reject unregistered addresses, but `MetricOmmPoolLiquidityAdder` has no factory reference and never calls it. The contract's own NatDoc explicitly acknowledges this gap. [5](#0-4) 

## Impact Explanation

Direct loss of user principal. Any user who has approved `MetricOmmPoolLiquidityAdder` and is tricked into calling `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 in a single transaction. Loss is bounded only by the user-supplied caps and their ERC-20 approval amount. This matches the allowed impact gate: Critical/High direct loss of user principal.

## Likelihood Explanation

Medium. The attack requires directing a victim to a malicious pool address — achievable via a compromised front-end, phishing UI, or a malicious integration presenting a fake pool as legitimate. The on-chain contract provides no guard; `MetricOmmPoolFactory.isPool()` is available off-chain but not enforced on-chain. [6](#0-5) 

## Recommendation

Add an immutable `factory` address to the constructor and validate `pool` against `IMetricOmmPoolFactory(factory).isPool(pool)` at the top of `_addLiquidity` before calling `_setPayContext`. This mirrors the pattern already available in the factory interface. [7](#0-6) 

## Proof of Concept

```solidity
contract MaliciousPool {
    address adder;
    address token0;
    address token1;
    uint256 MAX0;
    uint256 MAX1;

    constructor(address _adder, address _t0, address _t1, uint256 m0, uint256 m1) {
        adder = _adder; token0 = _t0; token1 = _t1; MAX0 = m0; MAX1 = m1;
    }

    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            MAX0, MAX1, abi.encode(uint8(1)) // KIND_PAY = 1
        );
        return (MAX0, MAX1);
    }

    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token0;
        imm.token1 = token1;
    }
}

// Attack:
// 1. Victim approves MetricOmmPoolLiquidityAdder for token0 and token1.
// 2. Victim is tricked into calling:
liquidityAdder.addLiquidityExactShares(
    address(maliciousPool), victim, 0, deltas,
    MAX_AMOUNT_TOKEN0, MAX_AMOUNT_TOKEN1, ""
);
// Result: safeTransferFrom(victim, maliciousPool, MAX_AMOUNT_TOKEN0) and
//         safeTransferFrom(victim, maliciousPool, MAX_AMOUNT_TOKEN1) execute.
```

The callback at lines 162–178 passes both guards and executes `pay(token0, payer, msg.sender, amount0Delta)` where `msg.sender` is the malicious pool, draining the victim's tokens. [8](#0-7)
