Audit Report

## Title
Unrestricted `recipient` in `unwrapWETH9` and `sweepToken` Allows Any Caller to Redirect Router-Held Balances — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.unwrapWETH9` and `sweepToken` are `public payable` functions that transfer the router's entire WETH/token balance to a caller-supplied `recipient` with no authorization check. Any unprivileged caller can supply their own address as `recipient` and drain the full router balance. The vulnerability window opens whenever a router balance persists across transaction boundaries, which is a realistic condition given that `exactInputSingle` and `exactInput` are standalone `external payable` functions that accept `recipient = address(router)` as a valid parameter.

## Finding Description
Both functions sweep the router's entire balance to an unchecked `recipient`:

`PeripheryPayments.sol` L37–45 (`unwrapWETH9`): transfers all WETH held by the router to `recipient` after withdrawing to ETH, with no check that `msg.sender == recipient` or that `msg.sender` deposited the balance.

`PeripheryPayments.sol` L48–55 (`sweepToken`): calls `safeTransfer(recipient, balanceToken)` for the full router balance of any token, again with no authorization check.

The intended safe usage is atomic multicall: `exactInputSingle` with `recipient = address(router)` followed immediately by `unwrapWETH9` in the same `multicall` call (demonstrated in `test_multicall_tokenForWeth_thenUnwrapEth`, L135–162 of the test file). Within a single multicall the funds are safe because `multicall` uses `delegatecall` and reverts atomically.

The vulnerability opens when:
1. A user calls `exactInputSingle` (or `exactInput`) with `recipient = address(router)` as a **standalone transaction** — a valid call since `exactInputSingle` is `external payable` with no multicall requirement.
2. Before the user's follow-up `unwrapWETH9` transaction, an attacker calls `unwrapWETH9(0, attacker)` or `sweepToken(token, 0, attacker)`.
3. No guard exists to reject this: neither function checks `msg.sender`, verifies deposit origin, or restricts `recipient`.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH sends; it does not protect WETH (ERC20) balances held between transactions.

## Impact Explanation
Direct loss of user principal. An attacker who observes a non-zero router WETH or token balance (via mempool or on-chain state) can call `unwrapWETH9(0, attacker)` or `sweepToken(token, 0, attacker)` and receive 100% of the router's balance. The victim loses their entire deposited amount with no recovery path. This matches the allowed impact gate: "Critical/High/Medium direct loss of user principal."

## Likelihood Explanation
Medium. The multicall pattern is safe, but:
1. `exactInputSingle` and `exactInput` are standalone `external payable` functions — nothing prevents a user from calling them without multicall.
2. The test suite explicitly demonstrates `recipient: address(router)` as the correct intermediate step for WETH unwrapping, which users may replicate outside of multicall.
3. MEV bots routinely monitor for profitable router balances and can atomically front-run any follow-up sweep transaction.
4. No NatSpec or interface-level warning exists to discourage non-multicall usage.

## Recommendation
Add a `msg.sender == recipient` requirement in both functions, or remove the `recipient` parameter and always send to `msg.sender`:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    require(recipient == msg.sender, "recipient must be caller");
    // ...
}

function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    require(recipient == msg.sender, "recipient must be caller");
    // ...
}
```

If third-party recipients are required (e.g., for integrators), add an explicit per-call authorization mechanism rather than accepting an unchecked address.

## Proof of Concept

```solidity
function testSweepToken_UnauthorizedRecipient() public {
    address alice = address(0xA11CE);
    address bob   = address(0xB0B);

    // Alice swaps token1 → WETH, routing output to the router (standalone tx, not multicall)
    vm.startPrank(alice);
    token1.approve(address(router), 3_000);
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          address(token1),
        tokenOut:         address(weth),
        zeroForOne:       false,
        amountIn:         3_000,
        amountOutMinimum: 0,
        recipient:        address(router),   // Alice intends to unwrap in a follow-up tx
        deadline:         block.timestamp + 1,
        priceLimitX64:    type(uint128).max,
        extensionData:    ""
    }));
    vm.stopPrank();

    uint256 routerWeth = weth.balanceOf(address(router));
    assertGt(routerWeth, 0, "router holds Alice's WETH");

    // Bob front-runs Alice's unwrapWETH9 and redirects funds to himself
    uint256 bobEthBefore = bob.balance;
    vm.prank(bob);
    router.unwrapWETH9(0, bob);   // no authorization check; bob steals Alice's WETH

    assertEq(weth.balanceOf(address(router)), 0, "router drained");
    assertGt(bob.balance, bobEthBefore, "bob received Alice's ETH");
}
```
