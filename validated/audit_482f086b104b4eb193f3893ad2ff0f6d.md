Audit Report

## Title
Missing Sequencer Uptime Check in `PriceProviderL2` and `ProtectedPriceProviderL2` Allows Stale-Price Swaps After Sequencer Restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly L2-targeted contracts (they carry `FUTURE_TOLERANCE` for sequencer clock skew) deployed on Arbitrum, Base, Avalanche, BSC, and Berachain. Both supply bid/ask prices to every pool swap via `getBidAndAskPrice()`, but neither checks whether the L2 sequencer is live before trusting oracle data. The only freshness gate is a timestamp-based `_isStale()` check with `MAX_TIME_DELTA` up to 7 days. After a sequencer outage shorter than `MAX_TIME_DELTA`, the pre-outage oracle price passes the staleness check and reaches pool swaps before keepers can push a fresh update, enabling a trader to extract value from LPs at a stale price.

## Finding Description

`PriceProviderL2._getBidAndAskPrice()` reads from the offchain oracle and applies only a timestamp staleness check:

```solidity
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
``` [1](#0-0) 

`_isStale` returns `true` only when `(block.timestamp - refTime) > MAX_TIME_DELTA`. `MAX_TIME_DELTA` is bounded at construction to the range `(0, 7 days]`: [2](#0-1) 

If the sequencer was down for a duration shorter than `MAX_TIME_DELTA`, the pre-outage `refTime` still satisfies the staleness check when the sequencer restarts. The public `getBidAndAskPrice()` then returns the stale bid/ask to the pool: [3](#0-2) 

`ProtectedPriceProviderL2` has the identical gap — `_computeBidAsk` applies the same `_isStale` with no sequencer liveness gate: [4](#0-3) 

The `FUTURE_TOLERANCE` field (present in both L2 contracts, absent from the L1 `PriceProvider`) confirms the authors are aware of L2 sequencer timing issues, yet no sequencer uptime feed is consulted: [5](#0-4) 

The registry ABI for `ChainlinkVerifierL2` confirms the protocol authors implemented `sequencerUptimeFeed` and `GRACE_PERIOD` for the Chainlink path, making the omission in the Pyth/compressed-oracle path an asymmetric gap rather than a design choice. No source file for `ChainlinkVerifierL2` exists in the repository; the pattern is evidenced only through the registry ABI.

`getBidAndAskPrice()` is called directly by `MetricOmmPool` during swap execution (8 call sites confirmed in `metric-core/contracts/MetricOmmPool.sol`), so the stale price reaches swap settlement without any intervening guard.

## Impact Explanation

During the sequencer-restart window — before keepers push a fresh Pyth/compressed-oracle update — `getBidAndAskPrice()` returns a stale pre-outage bid/ask to the pool. Any unprivileged trader can immediately submit a swap at the stale price. If the asset price dropped during the outage, the pool quotes a bid above true market value, allowing the trader to sell at an inflated rate — a direct loss of LP principal. This is a "Bad-price execution: stale bid/ask quote reaches a pool swap" impact, meeting the allowed impact gate. Severity is High given the direct, quantifiable LP fund loss with no privilege requirement.

## Likelihood Explanation

L2 sequencer outages are documented historical events on Arbitrum and other supported chains. The attack requires no special privilege — any user can submit a swap transaction immediately after sequencer restart. The window lasts until keepers push a fresh oracle update (seconds to minutes). With `MAX_TIME_DELTA` up to 7 days, even a multi-hour outage leaves prices appearing fresh. The `PriceProviderFactoryL2.createPriceProvider()` is permissionless, meaning pools using this provider are widely deployable. [6](#0-5) 

## Recommendation

Add a sequencer uptime check to `getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`, mirroring the pattern already used in `ChainlinkVerifierL2`:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _isSequencerActive() internal view returns (bool) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer == 1) return false; // sequencer is down
    if (block.timestamp - startedAt < GRACE_PERIOD) return false;
    return true;
}

function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    if (!_isSequencerActive()) revert SequencerDown();
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
```

On chains without a sequencer uptime feed, deploy the non-L2 variants (`PriceProvider`, `ProtectedPriceProvider`).

## Proof of Concept

1. Pool on Arbitrum is configured with `PriceProviderL2`, `MAX_TIME_DELTA = 3600` (1 hour).
2. At `t=0`, ETH/USDC price is 3000. Oracle `refTime = t=0`.
3. At `t=1`, Arbitrum sequencer goes down. No new oracle updates can be pushed.
4. Off-chain, ETH price drops to 2000 USDC.
5. At `t=1800` (30 min later), sequencer restarts. `block.timestamp - refTime = 1800 < MAX_TIME_DELTA = 3600` → `_isStale` returns `false`, staleness check passes.
6. Attacker immediately calls `swap(zeroForOne=false, ...)` — sells USDC to buy ETH at the stale 3000 USDC/ETH ask price.
7. Pool pays out ETH valued at 3000 USDC when true market value is 2000 USDC — LP suffers a ~33% loss on the trade.
8. Keepers push a fresh oracle update seconds later, closing the window — but the attacker's transaction executes first.

Foundry fork test: fork Arbitrum mainnet, deploy `PriceProviderL2` with a mock oracle returning a stale `refTime`, `warp` block timestamp to `refTime + MAX_TIME_DELTA - 1`, call `getBidAndAskPrice()` and assert it returns the stale price without reverting. [7](#0-6)
