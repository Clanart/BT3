### Title
Missing Failure Handling in `execute_deploy` Syscall Causes Permanently Unprovable Blocks — (File: `execution/syscall_impls.cairo`)

### Summary
The `execute_deploy` function in the StarkNet OS always writes `failure_flag=0` in the syscall response header, even when the deployed contract's constructor reverts. This creates an irreconcilable divergence between the sequencer's recorded execution trace and the OS's re-execution, making any block containing a `deploy` syscall with a reverting constructor permanently unprovable — a network halt.

### Finding Description

In `execute_deploy`, after calling `deploy_contract`, the response header is unconditionally written with `failure_flag=0`: [1](#0-0) 

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
...
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The TODO comment explicitly acknowledges that failure handling is absent. Compare this to `contract_call_helper`, the analogous function for `call_contract`, which correctly propagates the revert status: [2](#0-1) 

```cairo
if (is_reverted != FALSE) {
    assert retdata[retdata_size] = ERROR_ENTRY_POINT_FAILED;
    tempvar retdata_size = retdata_size + 1;
} ...
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

The `deploy_contract` function accepts `revert_log` as an implicit argument and is capable of returning normally even when the constructor reverts (it returns `(retdata_size, retdata)` — the revert data). When the constructor reverts:

1. The **sequencer** records `failure_flag=1` in the deploy syscall response; the caller contract sees failure and takes the failure execution branch.
2. The **OS** re-executes the same block: `deploy_contract` is called, the constructor reverts, but the OS unconditionally writes `failure_flag=0`.
3. The caller contract in the OS sees `failure_flag=0` (success) and takes the **success** execution branch.
4. The OS execution trace diverges from the sequencer's trace — the proof is invalid.
5. The block is permanently unprovable. [3](#0-2) 

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

The StarkNet OS proof is required to finalize blocks on L1. A block that cannot be proven cannot be finalized. Because the divergence is deterministic (the OS always writes `failure_flag=0`), every attempt to re-prove the block will fail identically. The block is permanently stuck. No subsequent blocks can be finalized until the issue is patched and the block is re-processed.

### Likelihood Explanation

**High.** Any unprivileged user can trigger this condition:

- Deploy a class whose constructor always reverts (e.g., `assert 1 = 0`).
- Deploy a caller contract that invokes the `deploy` syscall with that class.
- Submit an invoke transaction targeting the caller contract.

The sequencer will include the transaction (it pays fees regardless of the inner deploy failure). The block then becomes permanently unprovable. No special privileges, operator access, or leaked keys are required.

### Recommendation

Mirror the pattern used in `contract_call_helper`: capture whether `deploy_contract` returned a reverted result and propagate it to `failure_flag`:

```cairo
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

`deploy_contract` would need to be updated to return `is_reverted` alongside `retdata_size` and `retdata`, consistent with how `select_execute_entry_point_func` already returns `is_reverted`.

### Proof of Concept

1. Declare a Sierra class whose constructor body contains `assert(false, 'always reverts')`.
2. Deploy a caller contract whose `__execute__` calls the `deploy` syscall with that class hash.
3. Submit an invoke transaction (V3) targeting the caller contract.
4. The sequencer executes: constructor reverts → `failure_flag=1` recorded → caller contract takes failure branch → transaction included in block.
5. The OS re-executes the block:
   - `execute_deploy` is reached.
   - `deploy_contract` is called; constructor reverts; revert data returned.
   - OS writes `failure_flag=0` (hardcoded).
   - Caller contract in OS sees success → takes success branch → execution diverges from sequencer trace.
6. The Cairo proof is invalid — the block is permanently unprovable.
7. Network halt: no blocks can be finalized on L1 until the OS is patched.
