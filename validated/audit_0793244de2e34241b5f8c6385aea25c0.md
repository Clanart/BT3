### Title
Raw Ethereum Private Key Transmitted in Plaintext Over Unauthenticated HTTP JSON-RPC (`personal_importRawKey`) - (File: `rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

Ethermint's `personal_importRawKey` JSON-RPC method accepts a raw hex-encoded Ethereum private key as a plaintext parameter over HTTP. The JSON-RPC server defaults to plain HTTP with no TLS and no authentication. Any attacker who can observe or intercept the HTTP traffic (e.g., via a local network sniffer, a compromised proxy, or local process inspection of the socket) can steal the private key and drain the associated Ethereum account.

---

### Finding Description

`PrivateAccountAPI.ImportRawKey` in `rpc/namespaces/ethereum/personal/api.go` accepts the raw private key as a string parameter and forwards it directly to `Backend.ImportRawKey`:

```go
// rpc/namespaces/ethereum/personal/api.go:74-76
func (api *PrivateAccountAPI) ImportRawKey(privkey, password string) (common.Address, error) {
    api.logger.Debug("personal_importRawKey")
    return api.backend.ImportRawKey(privkey, password)
}
``` [1](#0-0) 

The backend receives the raw key and decodes it directly from the hex string passed over the wire:

```go

    ...

The JSON-RPC HTTP server is started in `server/json_rpc.go` and only uses TLS when both `CertificatePath` and `KeyPath` are non-empty:

```go
// server/json_rpc.go:123-129

        errCh <- httpSrv.Serve(ln)   // plain HTTP

``` [3](#0-2) 

`DefaultTLSConfig()` returns empty strings for both paths, so TLS is **off by default**: [4](#0-3) 

The `personal` namespace is registered with `Public: false` but no authentication layer is added:

```go

        Version:   apiVersion,

``` [5](#0-4) 

The `personal` namespace is not in the default API list (`GetDefaultAPINamespaces` returns `["eth","net","web3"]`), but it is a documented, available namespace that operators commonly enable via `--json-rpc.api eth,txpool,personal,net,debug,web3`: [6](#0-5) 

The entire JSON-RPC request body — including the raw private key — is sent as plaintext JSON over HTTP when TLS is not configured.

---

### Impact Explanation

When an operator enables the `personal` namespace and TLS is not configured (the default), the raw Ethereum private key is transmitted in plaintext in the HTTP POST body of every `personal_importRawKey` call. An attacker who can observe this traffic obtains the full private key and can immediately sign and broadcast transactions to drain the associated Ethereum account of all EVM-denom funds. This is a reachable route to unauthorized theft of EVM-denom funds.

---

### Likelihood Explanation

- The `personal` namespace is a documented, named namespace in `GetAPINamespaces()` and is commonly enabled in operator guides and startup scripts.
- TLS is **not configured by default** — `DefaultTLSConfig()` returns empty cert/key paths, so the server starts in plain HTTP mode unless the operator explicitly provides TLS credentials.
- An attacker on the same network segment (LAN, cloud VPC, shared hosting) can passively capture HTTP traffic. A compromised reverse proxy or load balancer in front of the node achieves the same result.
- No authentication is required to call `personal_importRawKey`; the only barrier is network reachability to the JSON-RPC port.

---

### Recommendation

1. **Short term (direct analog to the Peggo fix):** Do not accept raw private keys over the JSON-RPC interface at all. Remove or disable `personal_importRawKey` from the HTTP JSON-RPC surface. Require keys to be imported via the local keyring CLI (`unsafe-import-eth-key`) which at least limits exposure to the local process table rather than the network.
2. **Medium term:** Enforce TLS for the JSON-RPC server whenever the `personal` namespace is enabled. Add a startup-time validation that rejects `personal` in `json-rpc.api` unless `tls.certificate-path` and `tls.key-path` are both set.
3. **Long term:** Deprecate the `personal` namespace entirely (as go-ethereum has done) and replace key-management operations with keyring-backed signing that never exposes raw key material over any RPC channel.

---

### Proof of Concept

```bash
# 1. Start ethermintd with personal namespace enabled and no TLS (default)
ethermintd start \
  --json-rpc.api eth,txpool,personal,net,debug,web3 \
  --keyring-backend test

# 2. On the same network, capture HTTP traffic on the JSON-RPC port (default 8545)
tcpdump -i eth0 -A 'tcp port 8545' &

# 3. Operator imports their private key via personal_importRawKey
curl -X POST http://<node-ip>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"personal_importRawKey",
       "params":["<hex-private-key>","mypassword"],"id":1}'

# 4. Attacker reads the captured packet and extracts the raw private key from
#    the plaintext JSON body: {"params":["<hex-private-key>","mypassword"]}

# 5. Attacker uses the stolen key to sign and broadcast a transfer transaction,
#    draining all EVM-denom funds from the account.
```

The private key appears verbatim in the HTTP request body captured by `tcpdump`, `Wireshark`, or any network tap positioned between the client and the node.
