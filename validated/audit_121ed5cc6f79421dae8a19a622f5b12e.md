### Title
`execute_deploy` Syscall Hardcodes `failure_flag=0`, Silently Masking Constructor Failures — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in `syscall_impls.cairo` unconditionally writes `failure_flag=0` (success) into the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is the direct StarkNet OS analog of the ERC20 `approve()` unchecked-return-value class: a critical sub-operation's failure indicator is discarded, execution continues as if success occurred, and callers make irreversible fund-routing decisions on a false premise.

---

### Finding Description

In `execute_deploy`, after calling `deploy_contract`, the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns only `(retdata_size, retdata)` — no `is_reverted` flag is surfaced. The TODO comment at line 538 explicitly acknowledges that failure propagation is unimplemented. [2](#0-1) 

Contrast this with the correct pattern used in `contract_call_helper`, which properly reads `is_reverted` from `select_execute_entry_point_func` and propagates it into the response header:

```cairo
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(...)
...
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The `execute_deploy` path never performs this check. The `failure_flag` field of `ResponseHeader` is the sole mechanism by which the calling contract learns whether a sub-call succeeded or failed. When it is hardcoded to `0`, the calling contract has no way to detect a constructor revert.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The attack scenario proceeds as follows:

1. A factory contract (Contract A) calls the `deploy` syscall to deploy Contract B.
2. Contract B's constructor reverts (e.g., due to an attacker-controlled argument or a reachable failure path).
3. The OS executes `handle_revert`, rolling back Contract B's state changes — the contract is not registered at the computed address.
4. Despite the revert, the OS writes `failure_flag=0` into the response, returning the computed contract address to Contract A as if deployment succeeded.
5. Contract A, trusting the success response, routes funds (e.g., an initial deposit, a token transfer, or an ETH-equivalent) to the returned address.
6. That address has no deployed contract (state was reverted). The funds are permanently locked with no recovery path.

This matches the allowed impact: **Critical — Permanent freezing of funds** and **Critical — Direct loss of funds**. [4](#0-3) 

---

### Likelihood Explanation

The entry path requires no privileged role. Any unprivileged transaction sender can:

- Declare a contract class whose constructor contains a reachable revert path (e.g., conditioned on a constructor argument).
- Interact with any factory contract on the network that uses the `deploy` syscall and subsequently routes funds to the deployed address.

Factory patterns (token deployers, proxy factories, vault deployers) are standard and widespread on StarkNet. The constructor-revert condition can be triggered by the attacker by supplying crafted constructor calldata through the factory's public interface. No leaked keys, operator access, or network-level attack is required. [5](#0-4) 

---

### Recommendation

Replace the hardcoded `failure_flag=0` with the actual constructor execution result. `deploy_contract` must be updated to return an `is_reverted` flag (mirroring `execute_entry_point`'s return signature), and `execute_deploy` must propagate it:

```cairo
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(
        block_context=block_context,
        constructor_execution_context=constructor_execution_context,
    );
}

// Propagate constructor failure to the calling contract.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This mirrors the already-correct pattern in `contract_call_helper`. [6](#0-5) 

---

### Proof of Concept

**Setup:**

1. Declare a class `MaliciousChild` whose constructor reverts when `constructor_calldata[0] == 0xdead`.
2. Deploy a factory contract `Factory` that:
   - Accepts a class hash and constructor calldata.
   - Calls `deploy(class_hash, calldata)`.
   - On the returned address, immediately calls `transfer(address, amount)` to send funds.

**Execution:**

```
User → Factory.deploy_and_fund(MaliciousChild_hash, [0xdead], amount=1000 STRK)
  └─ Factory calls syscall: deploy(MaliciousChild_hash, [0xdead])
       └─ OS: MaliciousChild constructor reverts
       └─ OS: handle_revert() — state rolled back, address not registered
       └─ OS: ResponseHeader(gas=..., failure_flag=0)  ← BUG: always 0
  └─ Factory receives: success=true, contract_address=0xABCD
  └─ Factory calls: STRK.transfer(0xABCD, 1000)
       └─ 0xABCD has no contract — 1000 STRK permanently locked
```

**Expected (correct) behavior:** `failure_flag=1` is returned; Factory detects failure and does not route funds.

**Actual behavior:** `failure_flag=0` is returned; Factory routes funds to an unregistered address; funds are permanently frozen. [7](#0-6)
