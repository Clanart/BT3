### Title
Integer Division Truncation in `NewDynamicFeeChecker` Causes Zero Effective Fee Deduction for Cosmos Txs - (File: `ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` derives the per-gas price (`feeCap`) for Cosmos txs via integer division `fee / gas`. When `fee < gas`, Go's truncating integer division yields `feeCap = 0`. Combined with `baseFee = 0` (reachable because `CalculateBaseFee` floors the base fee at `MinGasPrice.TruncateInt()`, which is 0 for any fractional `MinGasPrice`), the EIP-1559 effective-price formula returns 0, so `DeductFeeDecorator` deducts nothing from the sender. A Cosmos tx that satisfies the `MinGasPriceDecorator` check (which uses decimal arithmetic) is therefore executed for free.

---

### Finding Description

**Root cause — `ante/evm/fee_checker.go` line 83:**

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

`sdkmath.Int.Quo` is truncating integer division. Whenever the total fee amount (in the EVM denom) is smaller than the gas limit, `feeCap` is truncated to 0. [1](#0-0) 

`feeCap = 0` passes the baseFee guard at line 86 (`0 < 0` is false) when `baseFeeInt = 0`, and is then forwarded to `EffectiveGasPrice`:

```go
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
effectiveFee := sdk.Coins{{Denom: denom, Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))}}
``` [2](#0-1) 

With `baseFee = 0` and `feeCap = 0`, `EffectiveGasPrice` computes `tip = min(maxPriorityPrice, feeCap − baseFee) = min(MaxInt64, 0) = 0`, so `effectivePrice = 0` and `effectiveFee = 0 × gas = 0`. `DeductFeeDecorator` deducts exactly this returned value — zero — from the sender.

**Why `baseFee = 0` is reachable:**

`CalculateBaseFee` floors the base fee at `params.MinGasPrice.TruncateInt()`:

```go
minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
``` [3](#0-2) 

`MinGasPrice` is a `LegacyDec` (decimal). The default value is `0`, and the parameter validation only requires it to be non-negative — fractional values such as `0.5` are explicitly valid: [4](#0-3) [5](#0-4) 

`TruncateInt()` of any value in `[0, 1)` is `0`, so the base fee lower bound is `0`. Under sustained low block utilisation the base fee decreases to `0` and stays there.

**The inconsistency with `MinGasPriceDecorator`:**

`MinGasPriceDecorator` enforces the minimum fee using decimal arithmetic:

```go
fee := gp.Amount.Mul(gasLimit).Ceil().RoundInt()
``` [6](#0-5) 

For `MinGasPrice = 0.5` and `gas = 21 000`, the required fee is `ceil(0.5 × 21 000) = 10 500`. A tx providing exactly `10 500` passes this check. But `NewDynamicFeeChecker` then computes `feeCap = 10 500 / 21 000 = 0` (integer truncation), and with `baseFee = 0`, returns `effectiveFee = 0`. The sender is charged nothing.

---

### Impact Explanation

This is a **High** severity fee market / ante handler accounting bug. A Cosmos tx that legitimately satisfies the `MinGasPriceDecorator` gate is executed with zero fee deduction. Repeated exploitation allows an attacker to spam the chain with free Cosmos transactions, exhausting block gas and degrading liveness, and to avoid paying fees that are owed to validators/fee collectors. This matches the allowed impact: *"fee market, ante handler … bug that permits … valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

- `MinGasPrice` defaults to `0`, so `baseFee` can reach `0` on any chain with low utilisation — no special configuration required.
- When `MinGasPrice` is set to a fractional value (e.g., `0.5`, a common operator choice to set a soft floor without a hard integer minimum), the truncation gap is exposed immediately once `baseFee` reaches `0`.
- The attacker-controlled input is simply a Cosmos tx with `fee < gas` in the EVM denom — no privileged access, no key compromise, no validator collusion required.
- The condition `fee < gas` is trivially satisfied: for `gas = 21 000`, any fee below `21 000 aevmos` (≈ 0.000021 token) triggers the truncation.

---

### Recommendation

Replace the truncating integer division with a comparison that preserves the decimal relationship, or use `LegacyDec` arithmetic throughout:

```go
// Instead of:
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))

// Use decimal division and compare against baseFee as a Dec:
feeCapDec := sdkmath.LegacyNewDecFromInt(fee).Quo(sdkmath.LegacyNewDecFromInt(sdkmath.NewIntFromUint64(gas)))
baseFeeDecInt := sdkmath.NewIntFromBigInt(baseFee)
if feeCapDec.LT(sdkmath.LegacyNewDecFromInt(baseFeeDecInt)) {
    return nil, 0, errorsmod.Wrapf(...)
}
```

Additionally, `CalculateBaseFee` should use `MinGasPrice` directly (without `TruncateInt`) when computing the lower bound, or the parameter validation should enforce `MinGasPrice >= 1` when EIP-1559 is enabled, to prevent the base fee from reaching `0` while a non-zero minimum gas price is intended.

---

### Proof of Concept

**Setup:** Chain with `MinGasPrice = 0.5`, `NoBaseFee = false`, London hardfork enabled. After several empty blocks, `baseFee` decreases to `0` (because `TruncateInt(0.5) = 0`).

**Attack:**
1. Attacker constructs a Cosmos tx with `gas = 21 000` and `fee = 10 500 aevmos` (satisfies `MinGasPriceDecorator`: `10 500 >= ceil(0.5 × 21 000) = 10 500`).
2. `NewDynamicFeeChecker` executes:
   - `feeCap = 10 500 / 21 000 = 0` (integer truncation)
   - `baseFeeInt = 0`
   - `feeCap.LT(baseFeeInt)` → `0 < 0` → **false** (check passes)
   - `effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = 0`
   - `effectiveFee = 0 × 21 000 = 0`
3. `DeductFeeDecorator` deducts `0` from the attacker's balance.
4. Tx executes successfully; attacker pays **zero fees** despite providing `10 500 aevmos` in the tx fee field (which is never deducted).
5. Repeat indefinitely for free Cosmos tx execution.
