### Title
Epoch-Blind `descendant_of_tracked_shard_cache` Causes Stale Shard-Tracking Decisions After Resharding — (`chain/epoch-manager/src/shard_tracker.rs`)

---

### Summary

`ShardTracker::check_if_descendant_of_tracked_shard` caches its result keyed only by `ShardId`, but the correct answer depends on both `ShardId` **and** `EpochId`. After resharding, the same `ShardId` maps to a different `ShardUId` (different layout version). A cache entry written in the pre-resharding epoch is returned verbatim in the post-resharding epoch, producing the wrong tracking decision. The sibling cache `tracked_accounts_shard_cache` correctly uses `EpochId` as its key; `descendant_of_tracked_shard_cache` does not.

---

### Finding Description

`ShardTracker` holds two caches:

```
tracked_accounts_shard_cache:       SyncLruCache<EpochId, BitMask>   // keyed by epoch ✓
descendant_of_tracked_shard_cache:  Mutex<HashMap<ShardId, bool>>    // keyed by shard only ✗
``` [1](#0-0) 

`check_if_descendant_of_tracked_shard` receives `shard_id` **and** `epoch_id`, but the cache lookup and insert use only `shard_id`:

```rust
if let Some(is_tracked) = self.descendant_of_tracked_shard_cache.lock().get(&shard_id) {
    return Ok(*is_tracked);   // epoch_id is ignored
}
// ...
self.descendant_of_tracked_shard_cache.lock().insert(shard_id, is_tracked);
``` [2](#0-1) 

The underlying implementation `check_if_descendant_of_tracked_shard_impl` uses `epoch_id` to resolve the shard layout and walk the ancestry chain, so its result is epoch-dependent: [3](#0-2) 

The test suite itself proves that the same `ShardId` carries a **different** `ShardUId` version after resharding:

```rust
// We expect the shard layout version to change in this test.
assert_ne!(non_parent_shard_new_uid.version, not_parent_shard_uid.version);
``` [4](#0-3) 

**Concrete stale-cache scenario** (exercised by the test at lines 905–914, but only with a fresh `ShardTracker` per call — production reuses the same instance across epochs):

| Step | `shard_id` | `epoch_id` | Correct answer | Cache state |
|------|-----------|-----------|----------------|-------------|
| Pre-resharding query | `Y` | epoch N | `false` — `ShardUId{v=0,Y}` is an *ancestor* of the configured `ShardUId{v=1,Y}`, not a descendant | `{Y → false}` written |
| Post-resharding query | `Y` | epoch N+1 | `true` — `ShardUId{v=1,Y}` **is** the configured shard | `{Y → false}` returned ← **wrong** | [5](#0-4) 

The call path that reaches the cache in production is:

```
cares_about_shard / will_care_about_shard
  → tracks_shard (resolves epoch_id from prev_hash)
    → tracks_shard_at_epoch
      → check_if_descendant_of_tracked_shard   ← stale cache hit
``` [6](#0-5) 

---

### Impact Explanation

A node configured with `TrackedShardsConfig::Shards` (used by archival nodes, sharded-RPC nodes, and some validators — confirmed by production call-sites in `nearcore/src/config.rs`, `neard/src/cli.rs`, `chain/indexer/src/lib.rs`) will, after resharding, silently stop tracking a shard it is supposed to track. Consequences:

- **State sync skipped**: `should_catch_up_shard` / `get_shards_to_state_sync` return wrong results, so the node never initiates state sync for the newly-versioned shard.
- **Chunk validation skipped**: the node does not request or validate chunks for the shard.
- **RPC correctness broken**: state queries for accounts in the untracked shard return stale or missing data.
- **Validator correctness broken**: a validator node that fails to track its assigned shard cannot produce or endorse chunks for it, risking slashing.

The false-negative direction (tracking suppressed when it should be active) is the dangerous path; the false-positive direction (tracking enabled when it should not be) wastes resources but does not corrupt state.

---

### Likelihood Explanation

The bug triggers on every resharding event for any node using `TrackedShardsConfig::Shards` that:
1. Queries `check_if_descendant_of_tracked_shard` for a "non-parent" shard (one that is not split but whose `ShardUId` version increments) **before** the resharding epoch boundary, and
2. Continues running across that boundary without restarting (which would clear the in-memory cache).

Long-running archival and RPC nodes are the primary victims. The test at lines 905–914 already encodes the exact scenario but masks the bug by constructing a fresh `ShardTracker` for each assertion.

---

### Recommendation

Change the cache key from `ShardId` to `(ShardId, EpochId)`:

```rust
// Before
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<ShardId, bool>>>,

// After
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<(ShardId, EpochId), bool>>>,
```

Update the lookup and insert in `check_if_descendant_of_tracked_shard` accordingly:

```rust
let cache_key = (shard_id, *epoch_id);
if let Some(&is_tracked) = self.descendant_of_tracked_shard_cache.lock().get(&cache_key) {
    return Ok(is_tracked);
}
// ...
self.descendant_of_tracked_shard_cache.lock().insert(cache_key, is_tracked);
```

Alternatively, key by `ShardUId` (computed from `shard_id` + the epoch's shard layout), which is already the unique identifier used everywhere else in the ancestry logic.

---

### Proof of Concept

1. Start a node with `TrackedShardsConfig::Shards([non_parent_shard_new_uid])` where `non_parent_shard_new_uid = ShardUId{version=1, shard_id=Y}` (the post-resharding identity of a non-split shard).
2. In the pre-resharding epoch (version 0), any call to `cares_about_shard` or `will_care_about_shard` for `shard_id=Y` reaches `check_if_descendant_of_tracked_shard(Y, [ShardUId{v=1,Y}], epoch_N)`. `ShardUId{v=0,Y}` is an ancestor of `ShardUId{v=1,Y}`, not a descendant, so the result is `false`. The cache stores `{Y → false}`.
3. After resharding (epoch N+1), the same call for `shard_id=Y` hits the cache and returns `false` immediately, bypassing `check_if_descendant_of_tracked_shard_impl`. The correct answer is `true` because `ShardUId{v=1,Y}` is now exactly the configured shard.
4. `tracks_shard_at_epoch` returns `false`, `should_catch_up_shard` returns `false`, and the node never initiates state sync or chunk tracking for shard Y in epoch N+1.

The existing test at `shard_tracker.rs:917–934` demonstrates the correct expected behavior but avoids the bug by constructing a new `ShardTracker` (empty cache) before each epoch query. Rewriting that test to reuse a single `ShardTracker` instance across both epoch queries would reproduce the failure. [7](#0-6) [8](#0-7) [9](#0-8)

```rust
fn check_if_descendant_of_tracked_shard_impl(
    shard_id: ShardId,
    tracked_shards: &Vec<ShardUId>,
    epoch_id: &EpochId,
    epoch_manager: &Arc<dyn EpochManagerAdapter>,
) -> Result<bool, EpochError> {
    let tracked_shards: HashSet<ShardUId> = tracked_shards.into_iter().cloned().collect();
    let protocol_version = epoch_manager.get_epoch_protocol_version(epoch_id)?;
    let shard_layout = epoch_manager.get_shard_layout(&epoch_id)?;

    // `ShardLayoutV3` stores all ancestor shards, no need to iterate through protocol versions
    if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
        let ancestors = HashSet::from_iter(ancestors);
        return Ok(!ancestors.is_disjoint(&tracked_shards));
    }

    let mut shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        // We explicitly track `shard_id` (the shard is a descendant of itself).
        return Ok(true);
    }

    // `shard_uid` does not belong to `tracked_shards`, but it might be a descendant of one.
    // Iterate through consecutive pairs of historical shard layouts (newest to oldest) to trace
    // the ancestry. Each pair represents a resharding transition.
    let layout_history = epoch_manager.get_shard_layout_history(protocol_version, None);
    for window in layout_history.windows(2) {
        let current_layout = &window[0];
        let prev_layout = &window[1];
        let Some(parent_shard_id) = current_layout.try_get_parent_shard_id(shard_uid.shard_id())?
        else {
            debug_assert!(
                false,
                "Parent shard is missing for shard {} in shard layout {:?}",
                shard_uid, current_layout,
            );
            return Ok(false);
        };
        shard_uid = ShardUId::from_shard_id_and_layout(parent_shard_id, &prev_layout);
        if tracked_shards.contains(&shard_uid) {
            return Ok(true);
        }
    }
    Ok(false)
