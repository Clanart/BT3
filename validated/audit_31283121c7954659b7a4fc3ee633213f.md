### Title
Permissionless `refundETH()` allows any caller to steal excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`refundETH()` has no access control and sends the router's entire ETH balance to `msg.sender`. Because `pay()` consumes only exactly `amountIn` native ETH when `msg.value > amountIn`, any excess ETH is silently left on the router after a swap. A third party calling `refundETH()` in a subsequent transaction steals that ETH.

### Finding Description

`refundETH()` is unconditionally callable by any address: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH and leaves the remainder untouched: [2](#0-1) 

`exactInputSingle` is `payable` and stores `msg.sender` as payer, but performs no refund of excess ETH before returning: [3](#0-2) 

The codebase's own test confirms this: it explicitly requires `refundETH()` as a second multicall entry to recover unused ETH, and asserts the router is empty only after that second call: [4](#0-3) 

If a user omits `refundETH()` from their multicall (or calls `exactInputSingle` directly with `msg.value > amountIn`), the excess ETH sits on the router unguarded until any attacker calls `refundETH()`.

### Impact Explanation

Direct, permanent loss of user ETH. The attacker receives the full excess balance; the victim has no recovery path because `refundETH()` sends to `msg.sender` with no record of the original depositor.

### Likelihood Explanation

- Any user who sends `msg.value > amountIn` without bundling `refundETH()` in the same multicall is vulnerable.
- Integrators building on top of the router (wallets, aggregators) may omit the refund step.
- MEV bots can monitor the mempool for transactions that leave ETH on the router and back-run them atomically.
- The `receive()` guard (rejects non-WETH senders) gives a false sense of security — it does not prevent ETH accumulation via `msg.value` in payable functions. [5](#0-4) 

### Recommendation

Add a `msg.sender`-binding mechanism: record the original caller at multicall entry (e.g., in transient storage) and restrict `refundETH()` to that address only. Alternatively, auto-refund excess ETH at the end of each swap function rather than relying on the caller to include a separate `refundETH()` step.

### Proof of Concept

```solidity
// Foundry integration test (pseudocode)
function test_attacker_steals_excess_eth() public {
    uint128 amountIn = 90;
    uint256 msgValue = 100;

    // Victim calls exactInputSingle via multicall with only 1 call (no refundETH)
    bytes[] memory calls = new bytes[](1);
    calls[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        ExactInputSingleParams({
            tokenIn: address(weth), amountIn: amountIn,
            /* ... other params ... */
        })
    );
    vm.prank(victim);
    router.multicall{value: msgValue}(calls);
    // 10 ETH now sits on the router

    // Attacker calls refundETH() in a separate transaction
    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance - attackerBefore, 10); // attacker receives 10 ETH
    assertEq(address(router).balance, 0);
    // victim cannot recover the 10 ETH
}
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
