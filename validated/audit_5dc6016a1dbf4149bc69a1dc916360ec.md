### Title
L1 Handler Transactions Bypass Dynamic L2 Fee Mechanism via Hardcoded Gas and No `charge_fee` Call — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` never calls `charge_fee` and hardcodes a fixed `L1_HANDLER_L2_GAS_MAX_AMOUNT` gas budget. Every other transaction type (invoke, deploy\_account, declare) calls `charge_fee` and uses user-supplied resource bounds that scale with L2 congestion. An attacker who controls an L1 message sender can flood L2 blocks with L1 handler transactions at a fixed L1 cost, bypassing the dynamic L2 fee mechanism entirely and crowding out all other transactions.

---

### Finding Description

In `execute_l1_handler_transaction`, the `TxInfo` struct is populated with `max_fee=0`, empty `resource_bounds_start`/`resource_bounds_end`, and `tip=0`:

```cairo
assert [tx_info] = TxInfo(
    version=L1_HANDLER_VERSION,
    ...
    max_fee=0,
    ...
    resource_bounds_start=cast(0, ResourceBounds*),
    resource_bounds_end=cast(0, ResourceBounds*),
    tip=0,
    ...
);
``` [1](#0-0) 

The execution gas budget is then set to the hardcoded constant `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000`:

```cairo
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(...);
``` [2](#0-1) 

The constant is defined as:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;
``` [3](#0-2) 

Critically, the function ends without ever calling `charge_fee`. Compare this to every other transaction type:

- `execute_invoke_function_transaction` → calls `charge_fee` at line 361
- `execute_deploy_account_transaction` → calls `charge_fee` at line 687
- `execute_declare_transaction` → calls `charge_fee` at line 822 [4](#0-3) 

`charge_fee` itself computes the maximum possible fee from the user-supplied resource bounds and enforces payment via an ERC-20 transfer. Because `resource_bounds` are empty for L1 handlers, `compute_max_possible_fee` would return 0, and `charge_fee` would short-circuit at the `if (max_fee == 0) { return (); }` guard — confirming that even if `charge_fee` were called, no fee would be collected. ### Impact Explanation

Each L1 handler transaction consumes up to 100,000,000 L2 gas units at zero L2 fee cost. During L2 congestion, the dynamic fee mechanism raises the price of `resource_bounds` for invoke/declare/deploy\_account transactions, making them progressively more expensive. L1 handler transactions are entirely immune to this mechanism: their cost is fixed at the L1 (Ethereum) gas price paid when the message was submitted, which does not scale with L2 congestion.

An attacker can therefore flood L2 blocks with L1 handler transactions at a fixed and predictable L1 cost, consuming the entire block gas budget and preventing all regular L2 transactions from being included. This matches the **High** impact: **Network not being able to confirm new transactions (total network shutdown)**.

---

### Likelihood Explanation

The attacker-controlled entry path is an L1/L2 message sender — explicitly listed as a valid unprivileged entry point. No privileged role, leaked key, or trusted operator is required. The attacker only needs to call the StarkNet Core contract on Ethereum to enqueue L1→L2 messages. The attack is most economically rational during periods of L2 congestion (e.g., a DeFi liquidation cascade), precisely when the damage is greatest: L2 fees for regular transactions spike while L1 handler spam cost remains constant.

---

### Recommendation

1. **Require resource bounds for L1 handler transactions.** Introduce a minimum L2 gas price commitment for L1 handlers, enforced at the OS level, so their effective cost scales with L2 congestion.
2. **Enforce a per-block cap on L1 handler gas consumption** separate from the general block gas limit, preventing L1 handler transactions from monopolizing block space.
3. **Charge an L2 fee for L1 handlers** by collecting the fee from the L1-paid amount (already escrowed in the StarkNet Core contract) and enforcing it in `execute_l1_handler_transaction` via a call to `charge_fee` or an equivalent mechanism.

---

### Proof of Concept

1. Attacker deploys a contract on L2 with an `@l1_handler` entry point that performs a maximally gas-consuming computation (e.g., repeated storage writes up to `L1_HANDLER_L2_GAS_MAX_AMOUNT`).
2. Attacker calls the StarkNet Core contract on L1 repeatedly, sending L1→L2 messages targeting that contract. Each message costs only L1 gas (fixed, independent of L2 congestion).
3. The sequencer includes these L1 handler transactions in L2 blocks. Each consumes up to 100,000,000 L2 gas at zero L2 fee.
4. During a period of L2 congestion (e.g., a large price move triggering DeFi liquidations), regular L2 transactions must bid increasingly high `resource_bounds` to be included. The attacker's L1 handler transactions face no such bidding pressure.
5. The attacker's messages fill every block, preventing liquidation transactions, collateral top-ups, and withdrawals from landing — potentially causing DeFi protocols to accumulate bad debt or freeze user funds.

func execute_l1_handler_transaction{
    range_check_ptr,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,
    outputs: OsCarriedOutputs*,
}(block_context: BlockContext*) {
    alloc_locals;

    %{ StartTx %}
    local is_reverted;
    %{ IsReverted %}
    // Skip the execution step for reverted transaction.
    if (is_reverted != FALSE) {
        %{ EndTx %}
        return ();
    }

    // TODO(Yoni): currently, the contract state is not fetched for reverted L1 handlers.
    //   Once block hash is supported, we should fetch the contract state for them as well.
    local entry_point_selector;
    %{ TxEntryPointSelector %}
    let (local tx_execution_context: ExecutionContext*) = get_invoke_tx_execution_context(
        block_context=block_context,
        entry_point_type=ENTRY_POINT_TYPE_L1_HANDLER,
        entry_point_selector=entry_point_selector,
    );
    local tx_execution_info: ExecutionInfo* = tx_execution_context.execution_info;

    local nonce;
    %{ LoadTxNonceL1Handler %}
    local chain_id = block_context.os_global_context.starknet_os_config.chain_id;

    let pedersen_ptr = builtin_ptrs.selectable.pedersen;
    with pedersen_ptr {
        let transaction_hash = compute_l1_handler_transaction_hash(
            execution_context=tx_execution_context, chain_id=chain_id, nonce=nonce
        );
    }
    update_pedersen_in_builtin_ptrs(pedersen_ptr=pedersen_ptr);

    %{ AssertTransactionHash %}

    // Write the transaction info and complete the ExecutionInfo struct.
    tempvar tx_info = tx_execution_info.tx_info;
    assert [tx_info] = TxInfo(
        version=L1_HANDLER_VERSION,
        account_contract_address=tx_execution_info.contract_address,
        max_fee=0,
        signature_start=cast(0, felt*),
        signature_end=cast(0, felt*),
        transaction_hash=transaction_hash,
        chain_id=chain_id,
        nonce=nonce,
        resource_bounds_start=cast(0, ResourceBounds*),
        resource_bounds_end=cast(0, ResourceBounds*),
        tip=0,
        paymaster_data_start=cast(0, felt*),
        paymaster_data_end=cast(0, felt*),
        nonce_data_availability_mode=0,
        fee_data_availability_mode=0,
        account_deployment_data_start=cast(0, felt*),
        account_deployment_data_end=cast(0, felt*),
        proof_facts_start=cast(0, felt*),
        proof_facts_end=cast(0, felt*),
    );
    fill_deprecated_tx_info(tx_info=tx_info, dst=tx_execution_context.deprecated_tx_info);
    assert_deprecated_tx_fields_consistency(tx_info=tx_info);

    // Consume L1-to-L2 message.
    consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
    let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
    non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
        block_context=block_context, execution_context=tx_execution_context
    );

    %{ EndTx %}
    return ();
}
