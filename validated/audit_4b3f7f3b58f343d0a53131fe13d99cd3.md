### Title
Missing Class Hash Declaration Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the new class hash supplied by a contract corresponds to a previously declared contract class. This is the direct analog of the ERC4626 compliance gap: just as `BaseRewardPool4626` claimed to implement a standard interface while omitting required enforcement, the OS claims to enforce the `replace_class` protocol invariant while omitting the required class-existence check. Any unprivileged contract deployer can exploit this to permanently freeze funds held by a contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. It reads the requested new `class_hash` directly from the syscall request struct and writes it into the contract's `StateEntry` in `contract_state_changes` without any validation that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing check. No Cairo assertion, range check, or dictionary lookup against `contract_class_changes` is performed to confirm that `class_hash` was ever declared via a `declare` transaction. The OS proof system therefore accepts and commits any arbitrary felt as a contract's new class hash.

Contrast this with `execute_declare_transaction` in `transaction_impls.cairo`, which does enforce class hash correctness:

```cairo
let expected_class_hash = finalize_class_hash(
    contract_class_component_hashes=contract_class_component_hashes
);
with_attr error_message("Invalid class hash pre-image.") {
    assert [class_hash_ptr] = expected_class_hash;
}
```

The `replace_class` path has no equivalent enforcement.

---

### Impact Explanation

Once a contract's `StateEntry.class_hash` is committed to the Patricia-Merkle state tree with an undeclared hash value, every subsequent call to that contract will fail at class lookup time inside the OS execution engine. Because the state commitment is final and the class does not exist in the compiled class facts bundle, the contract becomes permanently non-executable. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irreversibly frozen. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract from within its own execution context — no privileged role is required. An unprivileged user can:

1. Deploy a contract that holds or will receive funds.
2. Have that contract invoke `replace_class` with a felt value that has never been declared (e.g., any random felt not present in `contract_class_changes`).
3. The OS accepts the syscall, the block is proven, and the state is committed.

This is reachable via a normal `invoke` transaction from any externally-owned account. No operator key, sequencer collusion, or network-level attack is needed. The attack surface also includes buggy upgradeable contracts that accidentally pass an incorrect hash to `replace_class`.

---

### Recommendation

Inside `execute_replace_class`, before updating `contract_state_changes`, add a Cairo-level assertion that `class_hash` exists as a key in `contract_class_changes` (i.e., was declared in the current or a prior block). Concretely:

- Perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the returned compiled class hash is non-zero.
- Alternatively, maintain a separate declared-classes set and assert membership before accepting the syscall.

This mirrors the enforcement already present in `execute_declare_transaction` and closes the compliance gap.

---

### Proof of Concept

1. Deploy contract `VaultA` that accepts deposits and stores a balance in its own storage.
2. Users deposit funds into `VaultA`.
3. `VaultA` (or an attacker who controls it) calls the `replace_class` syscall with `class_hash = 0xdeadbeef` — a felt never declared on-chain.
4. The OS executes `execute_replace_class`: no validation is performed; `contract_state_changes` is updated with `class_hash = 0xdeadbeef`.
5. The block is proven and the state root is updated. `VaultA`'s `StateEntry.class_hash` is now `0xdeadbeef`.
6. Any subsequent `invoke` targeting `VaultA` causes the OS to look up class `0xdeadbeef` in the compiled class facts bundle — it does not exist — and the transaction fails unconditionally.
7. All funds in `VaultA`'s storage are permanently inaccessible.
