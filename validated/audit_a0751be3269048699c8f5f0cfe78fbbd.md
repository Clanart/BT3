### Title
Missing Duplicate-Key/Hash State Check in `batch_insert` Bypasses Invariants Enforced by `insert` — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` omits the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When the tree already contains more than one leaf, the bulk path writes leaf nodes directly to the blob via `insert_entry_to_blob` without consulting `block_status_cache`, allowing an untrusted caller to silently insert duplicate keys or hashes. The resulting tree has an incorrect root hash and produces invalid or forgeable proofs of inclusion.

### Finding Description

`MerkleBlob::insert` (the single-item path) enforces two state checks before any mutation:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` calls `self.insert(…)` (with those checks) only for the first two items when the tree has ≤ 1 existing leaf. For every subsequent item it falls through to the bulk path:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) … };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` writes raw bytes to the blob; it does not consult or update `block_status_cache`. There is no guard equivalent to the two checks above anywhere in this loop. A caller that supplies a `Vec` containing a repeated `KeyId` or a repeated `Hash` will have both copies written into the blob without error.

The Python binding `py_batch_insert` (line 1503) passes the caller-supplied `keys_values` and `hashes` vectors directly to `batch_insert` after only a length-equality check, so the missing validation is reachable from Python with no privilege.

### Impact Explanation

After a duplicate-key batch insert:

* The `block_status_cache` maps the key to exactly one of the two leaf indices; the other leaf is "orphaned" in the blob but still participates in hash computation.
* `calculate_lazy_hashes` traverses all blob nodes, so the root hash is computed over both leaves — it reflects a tree that is structurally different from what the cache believes exists.
* `get_proof_of_inclusion` returns a path for the cached leaf only; that path does not match the actual root hash, making the proof invalid.
* Conversely, an attacker who controls the batch input can arrange for a key to appear twice with different values/hashes, causing the committed root to cover a value that `get_proof_of_inclusion` will never return — effectively hiding one value behind a root that attests to it.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

`batch_insert` is the primary bulk-load API for DataLayer synchronization. It is exposed directly through the Python wheel (`py_batch_insert`) and is called whenever a DataLayer store is populated from a peer or from application code. Any caller — including an untrusted peer supplying delta data — can supply a vector with repeated keys or hashes. No privilege, key material, or network-level access is required beyond the ability to call the Python API.

### Recommendation

Add the same duplicate-key and duplicate-hash guards at the top of `batch_insert`, before the bulk loop, mirroring what `insert` does:

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
+   // Validate all inputs before mutating any state
+   for ((key, _), hash) in &keys_values_hashes {
+       if self.block_status_cache.contains_key(*key) {
+           return Err(Error::KeyAlreadyPresent());
+       }
+       if self.block_status_cache.contains_leaf_hash(hash) {
+           return Err(Error::HashAlreadyPresent());
+       }
+   }
+   // Also check for duplicates within the batch itself
+   // (a HashSet over keys and hashes)
    …
```

Alternatively, route all items through the validated `insert` path, accepting the performance trade-off, or add a pre-pass that deduplicates and validates the input vector before any blob mutation.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Pre-populate so leaf_count > 1 (bypasses the guarded path)
for i in range(3):
    h = hashlib.sha256(i.to_bytes(8, "big")).digest()
    blob.insert(KeyId(i), ValueId(i), h)

# Now batch_insert with a duplicate key — no error raised
dup_key = KeyId(99)
h1 = hashlib.sha256(b"value_a").digest()
h2 = hashlib.sha256(b"value_b").digest()
blob.batch_insert(
    [(dup_key, ValueId(0xAA)), (dup_key, ValueId(0xBB))],
    [h1, h2],
)

blob.calculate_lazy_hashes()
root = blob.get_root_hash()

# get_proof_of_inclusion returns a proof for one leaf only;
# that proof is invalid against the root that covers both leaves.
proof = blob.get_proof_of_inclusion(dup_key)
assert not proof.valid(), "proof should be invalid due to tree corruption"
