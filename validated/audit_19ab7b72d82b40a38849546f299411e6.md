### Title
Pending state diff published to shared RPC view before corresponding class bodies are downloaded, causing `starknet_getClass(pending, …)` to return `CLASS_HASH_NOT_FOUND` for a class that the authoritative pending state diff already declares - (`crates/apollo_central_sync/src/pending_sync.rs`)

---

### Summary

`sync_pending_data` writes the new `PendingData` (which includes the state diff listing freshly declared class hashes) into the shared `Arc<RwLock<PendingData>>` **before** the corresponding class bodies are fetched into `Arc<RwLock<PendingClasses>>`. Any concurrent RPC call that reads `pending_data` to confirm a class is declared, then reads `pending_classes` to retrieve the class body, will observe an internally inconsistent pending view: the state diff says the class exists, but `get_class(pending, hash)` returns `CLASS_HASH_NOT_FOUND`.

---

### Finding Description

In `get_pending_data` the ordering is:

```
// 1. Clear old classes when the parent block changes
pending_classes.write().await.clear();          // line 180

// 2. Publish the new state diff (which lists new declared_classes)
*pending_data.write().await = new_pending_data; // line 182

// 3. Return DownloadedNewPendingData
``` [1](#0-0) 

Back in `sync_pending_data`, after receiving `DownloadedNewPendingData`, the code reads the declared class hashes from the now-visible `pending_data` and **spawns async download tasks** for each class: [2](#0-1) 

The class bodies are written into `pending_classes` only when each download task completes: [3](#0-2) 

Between step 2 (state diff published) and the completion of all download tasks, the two shared locks are inconsistent: `pending_data.state_update.state_diff.declared_classes` lists class hash `H`, but `pending_classes` does not yet contain `H`.

The RPC `starknet_getClass` handler reads **only** `pending_classes` for the pending case:

```rust
let maybe_class = &self.pending_classes.read().await.get_class(class_hash);
if let Some(class) = maybe_class {
    return class.clone().try_into()…;
} else {
    BlockId::Tag(Tag::Latest)   // falls through to finalized storage
}
``` [4](#0-3) 

If the class is not yet in `pending_classes` (download still in flight), the handler silently falls through to the finalized storage. If the class was declared only in the pending block (not yet finalized), the finalized storage also lacks it, and the handler returns `CLASS_HASH_NOT_FOUND`.

Meanwhile `starknet_getStateUpdate(pending)` reads `pending_data` and correctly reports the class as declared: [5](#0-4) 

The two endpoints therefore return contradictory authoritative answers about the same pending block.

---

### Impact Explanation

This is a direct analog of the original cross-chain "use before receive" bug: the sequencer publishes the pending state diff (the "commitment" that a class exists) before the class body has been received from the feeder gateway. Any client or tooling that:

1. Polls `starknet_getStateUpdate(pending)` to detect a new class declaration, then
2. Immediately calls `starknet_getClass(pending, hash)` to retrieve the class body

will receive `CLASS_HASH_NOT_FOUND` from step 2 even though step 1 confirmed the class is declared. This is an authoritative-looking wrong value from the pending view, matching the **High** impact category: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

Concretely, a wallet or SDK that uses the pending class body to estimate fees or simulate a `deploy_account` transaction will fail with a spurious error, potentially causing the user to believe their declare transaction was dropped.

---

### Likelihood Explanation

The race window is open for the entire round-trip time of `central_source.get_class(class_hash)` (an HTTP call to the feeder gateway). This is typically hundreds of milliseconds to seconds. The pending block is polled continuously, so the window recurs on every new pending block that contains a class declaration. Any client that reacts to `starknet_getStateUpdate(pending)` events will reliably hit this window.

---

### Recommendation

Delay publishing `pending_data` until all class bodies for the new state diff have been downloaded, or atomically publish both `pending_data` and `pending_classes` together. One concrete approach: collect all class bodies first, then acquire both write locks and update them in a single critical section before returning `DownloadedNewPendingData`. Alternatively, the `get_class` RPC handler for the pending case should cross-check `pending_data.state_update.state_diff.declared_classes` and return a retriable "pending class not yet available" error rather than silently falling through to finalized storage and returning `CLASS_HASH_NOT_FOUND`.

---

### Proof of Concept

1. A pending block arrives containing a `declare` transaction for class `H`.
2. `get_pending_data` clears `pending_classes` (line 180) and writes the new `pending_data` with `declared_classes = [H]` (line 182).
3. A download task for `H` is spawned but has not yet completed.
4. Client A calls `starknet_getStateUpdate(pending)` → response includes `declared_classes: [{class_hash: H, …}]`. ✓
5. Client A immediately calls `starknet_getClass(pending, H)` → `pending_classes` is empty → falls through to finalized storage → `H` not there → returns `CLASS_HASH_NOT_FOUND`. ✗
6. The download task completes; subsequent calls to `starknet_getClass(pending, H)` succeed.

The inconsistency window is bounded only by feeder-gateway latency and is reproducible on every pending block containing a class declaration. [6](#0-5) [7](#0-6)
