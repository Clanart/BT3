### Title
Missing Shard-ID Binding Check in `set_state_header` Allows Cross-Shard State Substitution — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a network peer and validates it against a caller-supplied `shard_id`, but **never asserts that the chunk embedded in the header actually belongs to that shard**. A malicious snapshot host can serve a fully valid header for shard Y in response to a request for shard X. Every existing check passes, the header is stored under the key for shard X, and subsequent state-part application writes shard Y's trie into shard X's storage, permanently corrupting the syncing node's state.

---

### Finding Description

`set_state_header` performs five validation steps:

1. **Chunk internal proofs** — `validate_chunk_proofs(&chunk, …)` verifies the chunk's own body/header consistency.
2. **Merkle inclusion** — `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk_hash, height_included))` proves the chunk is *somewhere* in the block's chunk-header Merkle tree.
3. **Prev-chunk inclusion** — same Merkle check for the previous chunk.
4. **Incoming-receipt proofs** — iterates `incoming_receipts_proofs`; if the list is empty the loop body never executes.
5. **State-root-node validity** — `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())`.

None of these steps compare `chunk.shard_id()` to the caller-supplied `shard_id`. The Merkle check at step 2 proves only that the chunk hash appears *somewhere* in the block's `chunk_headers_root`; it does not prove the chunk occupies the slot for `shard_id`. After all checks pass, the header is persisted under `StateHeaderKey(shard_id, sync_hash)` — keyed by the *requested* shard, not the chunk's actual shard. [1](#0-0) 

The missing guard is:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other("set_shard_state failed: chunk shard_id mismatch".into()));
}
```

No such line exists anywhere in the function. [2](#0-1) 

The downstream `set_state_part` then reads the stored header, extracts `prev_state_root()` from the (wrong) chunk, and validates incoming parts against that root: [3](#0-2) 

`run_state_sync_for_shard` derives `shard_uid` from the *requested* `shard_id` and calls `apply_state_part` with that uid, writing the attacker-chosen trie nodes into the requested shard's storage: [4](#0-3) 

`set_state_finalize` then applies the wrong chunk's transactions and receipts against the now-corrupted trie, completing the substitution: [5](#0-4) 

---

### Impact Explanation

**Critical.** After state sync completes, the node holds shard Y's state under shard X's `ShardUId`. Every subsequent chunk execution for shard X reads and writes the wrong trie, producing incorrect state roots, invalid receipts, and divergent block hashes. The node cannot participate correctly in consensus and will be unable to produce or validate blocks for shard X. Because state sync is the mechanism by which nodes join the network at epoch boundaries, a single malicious snapshot host can silently corrupt every syncing node that contacts it.

---

### Likelihood Explanation

State sync is triggered at every epoch boundary for nodes that are catching up or joining. The syncing node selects snapshot hosts from the peer network; any peer advertising itself as a snapshot host can serve headers. The attack requires no special privilege — only the ability to respond to `StateHeaderRequest` messages. The attacker needs to supply a legitimately signed chunk for shard Y (which is trivially available from the chain) together with its valid Merkle proof. The only partial mitigation is the incoming-receipts check at step 4, but it is bypassed whenever `incoming_receipts_proofs` is empty, which is a normal condition when no cross-shard receipts were sent to the target shard in the relevant window. [6](#0-5) 

---

### Recommendation

Immediately after extracting the chunk from the header, assert shard identity:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind the chunk to the requested shard before any other check
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, verify that `prev_chunk_header.shard_id() == shard_id` when `prev_chunk_header` is `Some`.

---

### Proof of Concept

**Setup**: 4-shard network. Syncing node N requests the state header for `shard_id = 0` at `sync_hash = H`.

**Attack**:

1. Malicious snapshot host M holds the block at `H`. It has valid chunks for shards 0–3.
2. M responds to N's `StateHeaderRequest { shard_id: 0, sync_hash: H }` with a `ShardStateSyncResponseHeader` containing:
   - `chunk` = the legitimate chunk for **shard 1** (signed by the real chunk producer, so `validate_chunk_proofs` passes).
   - `chunk_proof` = the Merkle path for shard 1's chunk in the block's `chunk_headers_root` (passes `verify_path`).
   - `incoming_receipts_proofs = []` (empty — skips the receipt-hash check entirely).
   - `state_root_node` = the valid state root node for shard 1 (passes `validate_state_root_node`).
3. N calls `set_state_header(shard_id=0, sync_hash=H, header_for_shard_1)`. All five checks pass. The header is stored under `StateHeaderKey(shard_id=0, H)`.
4. N downloads state parts for shard 1 (valid against shard 1's `prev_state_root`). `set_state_part` validates each part against shard 1's root — all pass.
5. `apply_state_part` is called with `shard_uid` for shard 0 but shard 1's trie nodes. Shard 0's trie now contains shard 1's state.
6. `set_state_finalize` applies shard 1's chunk (transactions, receipts) against shard 0's trie. Shard 0's `ChunkExtra` is written with shard 1's resulting state root.
7. Node N is now permanently corrupted: shard 0 holds shard 1's state. [7](#0-6) [3](#0-2) [8](#0-7)

    pub fn set_state_finalize(
        &mut self,
        shard_id: ShardId,
        sync_hash: CryptoHash,

            tracing::debug_span!(target: "sync", "chain_update_set_state_finalize", %shard_id, ?sync_hash).entered();

            ShardStateSyncResponseHeader::V1(shard_state_header) => (
                ShardChunk::V1(shard_state_header.chunk),
                shard_state_header.incoming_receipts_proofs,
            ),
            ShardStateSyncResponseHeader::V2(shard_state_header) => {
                (shard_state_header.chunk, shard_state_header.incoming_receipts_proofs)
            }

        // from store on disk.

            &self.chain_store_update.chain_store(),
            &sync_hash,
            chunk.height_included(),
        )?;

        // Getting actual incoming receipts.
        let mut receipt_proof_responses: Vec<ReceiptProofResponse> = vec![];
        for incoming_receipt_proof in &incoming_receipts_proofs {
            let ReceiptProofResponse(hash, _) = incoming_receipt_proof;
            let block_header = self.chain_store_update.get_block_header(hash)?;
            if block_header.height() <= chunk.height_included() {
                receipt_proof_responses.push(incoming_receipt_proof.clone());
            }
        }

        let is_genesis = block_header.height() == self.chain_store_update.get_genesis_height();

            block_header.next_gas_price()

                .chain_store()
                .compute_transaction_validity(&prev_block_header, &chunk)

            vec![true; transactions.len()]

        let transactions = SignedValidPeriodTransactions::new(transactions, transaction_validity);
