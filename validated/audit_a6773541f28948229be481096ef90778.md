### Title
Permanent Fund Lock When Sub-Threshold Amount Normalizes to Zero Before Signing — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer_internal` burns or locks the user's full token amount and returns `U128(0)` (consuming all tokens via the NEP-141 callback), but performs no check that the amount will survive decimal normalization. The zero-amount guard exists only in `sign_transfer`, which is called later by a relayer. When the amount-minus-fee normalizes to zero, `sign_transfer` permanently reverts, and no cancel or refund path exists. The user's tokens are irrecoverably locked or burned.

---

### Finding Description

`normalize_amount` performs floor division to convert a NEAR-side token amount to the destination-chain representation:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

For a token whose NEAR representation has more decimals than the destination chain (e.g., `origin_decimals = 24`, `decimals = 18`), any amount strictly less than `10^6` normalizes to zero.

`init_transfer` only validates `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

It does **not** check whether `normalize_amount(amount_without_fee(), decimals) > 0`. Execution then proceeds to `init_transfer_internal`, which burns or locks the full amount and returns `U128(0)`, signalling to the NEP-141 runtime that all tokens are consumed:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
...
U128(0)
``` [3](#0-2) 

The only zero-amount guard is in `sign_transfer`, called later by a relayer:

```rust
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

Because `sign_transfer` will always revert for this transfer, and no `cancel_transfer` / `refund_transfer` function exists anywhere in `near/omni-bridge/src/`, the stored `TransferMessage` can never be completed or removed by the user. The tokens are permanently locked or burned.

The code comment on `normalize_amount` acknowledges that *dust remainders* stay locked when `fee = 0` and defers to `SECURITY.md`. However, the scenario here is distinct: the **entire** transferred amount is lost, not a sub-unit remainder. A user sending `amount = 999_999` (NEAR units) with `fee = 0` for a token with `origin_decimals = 24, decimals = 18` loses all 999,999 units — not a rounding residual.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

Any user who initiates a NEAR→EVM (or NEAR→other-chain) transfer with an amount below the decimal-normalization threshold (`< 10^(origin_decimals − decimals)`) will have their tokens burned or locked with no recovery path. The transfer message is stored on-chain but can never be signed or cancelled. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Low-to-medium.** The threshold amounts are small in absolute terms (e.g., `< 10^6` yocto-units for a 24/18 decimal pair). However:

- The bridge accepts arbitrary user-supplied amounts with no minimum enforced at `init_transfer` time.
- A user unfamiliar with decimal normalization can easily send a sub-threshold amount.
- The failure is silent at initiation time — the `InitTransferEvent` is emitted normally, giving no indication the transfer is unsignable.
- The loss is permanent and unrecoverable without a protocol-level intervention.

---

### Recommendation

Add a normalization check inside `init_transfer` (or at the start of `init_transfer_internal`) before tokens are consumed. Retrieve the destination-chain decimals for the token and assert that `normalize_amount(amount_without_fee(), decimals) > 0`. If the check fails, return the full amount to the caller (i.e., return `transfer_message.amount` from `init_transfer_internal` before burning/locking), which causes the NEP-141 callback to refund the sender.

Alternatively, add a `cancel_transfer` entry point that allows the original sender to reclaim locked/burned tokens for transfers that have never been signed, providing a recovery path for stuck transfers regardless of cause.

---

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 24` (NEAR), `decimals = 18` (EVM). Normalization divisor = `10^6`.

1. User calls `ft_transfer_call` on the NEAR token contract with `amount = 500_000` and `fee = 0`, targeting the bridge with an EVM recipient.
2. `init_transfer` passes the `fee < amount` check (`0 < 500_000`). [2](#0-1) 
3. `init_transfer_internal` burns/locks `500_000` tokens and returns `U128(0)` — tokens are consumed. [3](#0-2) 
4. `InitTransferEvent` is emitted; the transfer appears valid.
5. Relayer calls `sign_transfer`. `normalize_amount(500_000, {origin: 24, dest: 18}) = 500_000 / 10^6 = 0`. [1](#0-0) 
6. `require!(amount_to_transfer > 0)` panics. [4](#0-3) 
7. No cancel function exists. The 500,000 tokens are permanently lost.
