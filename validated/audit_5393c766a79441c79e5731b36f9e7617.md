### Title
`ProofOfInclusion::valid()` Final Root Check Is Tautologically True, Enabling Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

The `ProofOfInclusion::valid()` method in the DataLayer Merkle proof module only verifies internal consistency of the proof structure. Its final check — `existing_hash == self.root_hash()` — is tautologically true after the loop completes, meaning the function never verifies the proof against any external trusted root. Any attacker who can supply a `ProofOfInclusion` to a caller that relies solely on `valid()` can forge an inclusion proof for an arbitrary key-value pair.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← always true after the loop
}
``` [1](#0-0) 

The loop invariant guarantees that on every iteration, `existing_hash` is set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. After the loop, `existing_hash` therefore holds the last `layer.combined_hash`. The `root_hash()` method returns exactly that same value:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same value existing_hash already holds
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The final comparison `existing_hash == self.root_hash()` is therefore always `true` once the loop exits without returning `false`. The function never compares the computed root against any externally-supplied, trusted tree root.

`ProofOfInclusion` derives `Streamable`, making it serializable and deserializable from untrusted bytes: [3](#0-2) 

An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` (representing a fake leaf) and a chain of layers whose `combined_hash` values are computed consistently using `calculate_internal_hash`. This proof will pass `valid()` while proving inclusion in a fabricated tree root that does not correspond to any on-chain commitment.

The edge case where `layers` is empty is equally broken: `root_hash()` returns `self.node_hash`, and `existing_hash` is also `self.node_hash` (never modified), so the check reduces to `self.node_hash == self.node_hash`, which is always true. An empty proof with any `node_hash` passes `valid()`.

### Impact Explanation

Any DataLayer client that calls `proof.valid()` without separately asserting `proof.root_hash() == on_chain_committed_root` can be convinced by a malicious DataLayer node (or any network peer) that an arbitrary key-value pair is committed to the chain. This allows untrusted input to prove invalid state — forged inclusion — against the DataLayer Merkle tree, matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type exposed through Python bindings, designed to be transmitted between nodes. The `valid()` method is the sole self-contained validation API on the struct; it accepts no external root parameter, so callers who follow the natural API surface will call only `valid()`. The Python test suite calls `proof_of_inclusion.valid()` without any separate root check, establishing the pattern: [4](#0-3) 

Any downstream consumer that mirrors this pattern against an untrusted proof source is vulnerable.

### Recommendation

1. Add an external root parameter to `valid()` (or a companion method `valid_against_root(root: &Hash) -> bool`) that compares the computed root against a caller-supplied trusted root, rather than against the proof's own `combined_hash`.
2. Alternatively, rename the current method to `internally_consistent()` to make clear it does not authenticate the proof against any committed root, and require callers to separately assert `proof.root_hash() == trusted_root`.
3. Update all call sites — including Python bindings and tests — to pass and verify the on-chain committed root.

### Proof of Concept

```rust
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};
use chia_datalayer::{Hash, Side};

fn forge_proof(fake_node_hash: Hash, fake_sibling: Hash) -> ProofOfInclusion {
    // Compute a combined_hash that is internally consistent
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash,
        Side::Left,
        &fake_sibling,
    );
    ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: fake_sibling,
            combined_hash: combined,   // attacker-controlled root
        }],
    }
}

fn main() {
    let fake_node_hash = [0xAA; 32];
    let fake_sibling   = [0xBB; 32];
    let proof = forge_proof(fake_node_hash, fake_sibling);

    // valid() returns true for a completely fabricated proof
    assert!(proof.valid());
    // proof.root_hash() is the attacker-chosen combined_hash, not any on-chain root
}
```

The forged proof passes `valid()` because the tautological final check never compares against a trusted external root. A DataLayer client relying solely on `valid()` would accept this as proof that `fake_node_hash` is included in the DataLayer tree.
