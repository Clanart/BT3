### Title
Missing Slippage Protection When Removing Liquidity - (File: metric-core/contracts/MetricOmmPool.sol)

### Summary

`MetricOmmPool.removeLiquidity()` accepts no minimum-amount-out parameters. LPs cannot bound the token amounts they receive, leaving every withdrawal vulnerable to front-running that shifts the bin cursor before the transaction lands.

### Finding Description

`MetricOmmPool.removeLiquidity()` signature is:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
```

There are no `minAmount0Out` / `minAmount1Out` guards anywhere in the call chain. After the ownership check the pool delegates directly to `LiquidityLib.removeLiquidity`, which computes each token's share proportionally from the bin's current `token0BalanceScaled` / `token1BalanceScaled` and transfers them out:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
...
IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
```

The bin's token composition is entirely determined by the current oracle-driven cursor position. Bins above the cursor hold only token0; bins below hold only token1; only the active bin holds both. A swap that moves the cursor from bin N to bin N-1 (or N+1) before the LP's `removeLiquidity` lands converts the LP's mixed-token bin into a single-token bin. The LP receives 0 of one token with no on-chain recourse.

The periphery `MetricOmmPoolLiquidityAdder` provides `maxAmountToken0` / `maxAmountToken1` caps for *adding* liquidity, but **there is no periphery wrapper for removing liquidity at all** — a `grep` across all `metric-periphery/**/*.sol` files returns zero matches for `removeLiquidity`. Users must call the core pool directly and have no mechanism to enforce a minimum.

### Impact Explanation

An LP who holds shares in the active bin (which holds both tokens) submits `removeLiquidity` expecting a mixed payout. An attacker (or natural oracle movement) shifts the cursor one bin before the transaction executes. The LP's bin now holds only one token; `amount0Removed` or `amount1Removed` is 0. The LP receives the full value in a single token they may not want, and must swap back at the cost of the protocol's bid/ask spread. On chains with public mempools this is a straightforward sandwich: shift cursor in, let LP withdraw single-sided, shift cursor back.

### Likelihood Explanation

Any LP removing liquidity from the active bin is exposed. The attack requires only a swap large enough to move the cursor one bin, which is a normal pool operation available to any unprivileged address. On EVM chains with public mempools (Ethereum mainnet, most L2s) front-running is routine. Natural oracle price movements between block submission and inclusion produce the same outcome without any attacker.

### Recommendation

Add `minAmount0Out` and `minAmount1Out` to `MetricOmmPool.removeLiquidity` and revert if the returned amounts fall below them:

```diff
 function removeLiquidity(
     address owner,
     uint80 salt,
     LiquidityDelta calldata deltas,
+    uint256 minAmount0Out,
+    uint256 minAmount1Out,
     bytes calldata extensionData
 ) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
   returns (uint256 amount0Removed, uint256 amount1Removed)
 {
     ...
     (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
+    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out)
+        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Out, minAmount1Out);
     _afterRemoveLiquidity(...);
 }
```

Additionally, add a `removeLiquidity` wrapper to `MetricOmmPoolLiquidityAdder` that forwards these bounds, mirroring the `maxAmountToken0` / `maxAmountToken1` pattern already present for `addLiquidityExactShares`.

### Proof of Concept

1. Pool is deployed with bins `[-1, 0, 1]`; oracle price places cursor at bin 0 (active bin holds both tokens).
2. LP calls `pool.addLiquidity(owner, salt, deltas_bin0, ...)` depositing 100 token0 + 100 token1 into bin 0.
3. LP submits `pool.removeLiquidity(owner, salt, deltas_bin0, "")` expecting ≈100 token0 + ≈100 token1.
4. Attacker front-runs with a `zeroForOne` swap large enough to exhaust bin 0 and move the cursor to bin -1. Bin 0 is now above the cursor and holds only token0.
5. LP's `removeLiquidity` executes: `binState.token1BalanceScaled == 0`, so `amount1Removed = 0`. LP receives only token0.
6. Attacker back-runs with the reverse swap, restoring the cursor.
7. LP received 0 token1 with no revert, no recourse.
