### Title
`deprecated_declared_classes` Silently Dropped in `CommitmentStateDiff → ThinStateDiff` Conversion Produces Wrong State Diff Commitment and Block Hash — (File: `crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally sets `deprecated_declared_classes: Vec::new()`. When the batcher computes the state diff commitment for a block that contains Cairo 0 (deprecated) class declarations, those declarations are silently excluded. The resulting `state_diff_commitment` and `state_diff_length` (packed into `concatenated_counts`) are both wrong, producing an incorrect block hash that diverges from what any independent verifier would compute.

---

### Finding Description

**Root cause — the silent drop:**

In `crates/blockifier/src/state/cached_state.rs`, `CommitmentStateDiff` has no `deprecated_declared_classes` field (it is not tracked in `StateMaps`), and the conversion to `ThinStateDiff` hard-codes the field to empty:

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),
        }
    }
}
``` [1](#0-0) 

**Commitment path that consumes the wrong `ThinStateDiff`:**

`BlockExecutionArtifacts::new()` calls `ThinStateDiff::from(commitment_state_diff.clone())` and passes the result directly to `calculate_block_commitments`:

```rust
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),   // deprecated_declared_classes always []
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
``` [2](#0-1) 

The same conversion is used in `thin_state_diff()`, which is the `ThinStateDiff` written to storage and returned in `DecisionReachedResponse`:

```rust
pub fn thin_state_diff(&self) -> ThinStateDiff {
    ThinStateDiff::from(self.commitment_state_diff.clone())
}
``` [3](#0-2) 

**Two commitment fields that become wrong:**

1. **`state_diff_commitment`** — `calculate_state_diff_hash` chains `deprecated_declared_classes` into the Poseidon hash:

```rust
hash_chain = chain_deprecated_declared_classes(
    &state_diff.deprecated_declared_classes, hash_chain);
``` [4](#0-3) 

With an empty list the hash is `Poseidon("STARKNET_STATE_DIFF0", …, 0, …)` instead of the correct value that includes the declared class hashes.

2. **`state_diff_length` inside `concatenated_counts`** — `ThinStateDiff::len()` counts `deprecated_declared_classes.len()`:

```rust
result += self.deprecated_declared_classes.len();
``` [5](#0-4) 

`calculate_block_commitments` passes `state_diff.len()` as `state_diff_length` to `concat_counts`, which packs it into the single felt `concatenated_counts` that enters the block hash: [6](#0-5) 

**Block hash chain:**

`calculate_block_hash` chains both `concatenated_counts` (position 6) and `state_diff_commitment` (position 7):

```rust
.chain(&block_commitments.concatenated_counts)
.chain(&block_commitments.state_diff_commitment.0.0)
``` [7](#0-6) 

Both fields are wrong whenever a block contains at least one Cairo 0 Declare transaction, so the final block hash diverges from the value any independent verifier (SNOS, L1 verifier, sync peer) would compute from the same inputs.

**Analog to the external "locked ETH" bug:**

The external bug: a contract is `payable` (can receive ETH) but has no withdrawal path, so ETH is permanently trapped.

The sequencer analog: a Cairo 0 Declare transaction is accepted and executed (the class is stored in the class manager), but the declaration has no path into `CommitmentStateDiff` → `ThinStateDiff` → `state_diff_commitment`. The class declaration is permanently excluded from the commitment — it is "received" but can never be "withdrawn" into the proof-visible state.

---

### Impact Explanation

- **Wrong `state_diff_commitment`** stored in `PartialBlockHashComponents` and propagated to the block header.
- **Wrong `state_diff_length`** in `packed_lengths` / `concatenated_counts`, making the block hash differ from any independent recomputation.
- **Wrong block hash** stored in the database and used as `previous_block_hash` for the next block, breaking the hash chain.
- Any SNOS proof, L1 settlement check, or P2P sync peer that recomputes the block hash from the actual state diff (which includes the deprecated class declaration) will reject the block.

Impact category: **Critical — Wrong state/commitment value from execution logic for accepted input.**

---

### Likelihood Explanation

Cairo 0 (`Declare` v0/v1) transactions are valid Starknet transactions. The `starknet_api` crate models them, the blockifier executes them, and no explicit gateway rejection of deprecated Declare transactions was found in the codebase. Any block that includes a single Cairo 0 class declaration triggers the wrong commitment. The TODO comment (`// TODO(AlonH): Remove this when the structure of storage diffs changes.`) confirms the gap is known but unresolved.

---

### Recommendation

Add `deprecated_declared_classes: Vec<ClassHash>` to `CommitmentStateDiff` and populate it from the blockifier's execution results (e.g., by tracking Cairo 0 class declarations in `StateMaps` or by passing them through `BlockExecutionSummary`). The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then propagate the field instead of hard-coding `Vec::new()`.

---

### Proof of Concept

1. Submit a Cairo 0 (`Declare` v1) transaction to the Apollo sequencer gateway.
2. The blockifier executes it; the class is stored in the class manager.
3. `BlockExecutionArtifacts::new()` calls `ThinStateDiff::from(commitment_state_diff)`, producing a `ThinStateDiff` with `deprecated_declared_classes = []`.
4. `calculate_block_commitments` computes `state_diff_commitment = Poseidon("STARKNET_STATE_DIFF0", …, 0 deprecated classes, …)` — missing the declared class hash.
5. `state_diff_length` in `concat_counts` is `N` instead of `N + 1`.
6. The block hash stored in the database is `H_wrong`.
7. An independent verifier fetches the stored `ThinStateDiff` (also missing the class, since it came from the same conversion), recomputes the hash, and gets `H_wrong` — consistent internally but diverging from the Starknet protocol's expected value that includes the deprecated class declaration.
8. If the verifier instead reconstructs the state diff from on-chain data (which includes the class declaration), it computes `H_correct ≠ H_wrong`, causing proof/settlement failure.
