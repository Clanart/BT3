### Title
`denormalize_amount` u128 Multiplication Overflow Permanently Freezes User Funds on EVM — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`denormalize_amount` performs an unchecked `u128 * u128` multiplication. With `overflow-checks = true` in the workspace `Cargo.toml`, any overflow panics and reverts the transaction. When this panic occurs inside `fin_transfer_callback`, the user's tokens are already locked in the EVM bridge contract but the NEAR-side proof is never consumed, making the funds permanently irrecoverable.

---

### Finding Description

`denormalize_amount` converts a token amount from the destination-chain unit representation to the NEAR unit representation by multiplying by `10^(origin_decimals − decimals)`: [1](#0-0) 

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked u128 multiplication
}
```

`origin_decimals` is the NEAR token's decimal precision (e.g. 24) and `decimals` is the source-chain token's decimal precision (e.g. 18 for an EVM token), so `diff_decimals` can be 6 or more. The workspace enforces `overflow-checks = true`: [2](#0-1) 

`fin_transfer_callback` calls `denormalize_amount` directly on the raw amount extracted from the on-chain proof before any state is written:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

The EVM `initTransfer` function accepts `uint128 amount` with no upper-bound guard beyond `fee < amount`: [4](#0-3) 

The EVM event also encodes the amount as `uint128`: [5](#0-4) 

`Decimals` is stored per token address via `add_token`, populated from the `DeployToken`/`BindToken` proof: [6](#0-5) 

The same overflow is reachable in `fast_fin_transfer` and `claim_fee_callback`: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

When `amount * 10^diff_decimals > u128::MAX`, the Rust runtime panics and reverts the entire `fin_transfer_callback` transaction. Because the panic occurs before any state mutation (no entry is written to `finalised_transfers` or `pending_transfers`), the proof is not consumed. However, the user's tokens were already transferred into the EVM bridge contract during `initTransfer`. Every subsequent relay attempt for the same proof will produce the same panic. The user's funds are permanently locked in the EVM bridge with no recovery path.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

For a token registered with `origin_decimals = 24` (NEAR) and `decimals = 18` (EVM), `diff_decimals = 6`. The maximum safe EVM amount is `u128::MAX / 10^6 ≈ 3.4 × 10^32`. The EVM bridge accepts any `uint128` up to `u128::MAX ≈ 3.4 × 10^38`, so any amount in the range `(3.4 × 10^32, 3.4 × 10^38]` triggers the overflow. For tokens with `diff_decimals = 18` (e.g. `origin_decimals = 24`, `decimals = 6` like USDC-equivalent), the safe ceiling drops to `≈ 3.4 × 10^20` EVM units. High-supply, low-value tokens (meme tokens, wrapped satoshis, etc.) can realistically reach these amounts. No privileged access is required — any user can call `initTransfer` on EVM with a large amount.

---

### Recommendation

Replace the bare multiplication in `denormalize_amount` with a checked or saturating variant and return an error instead of panicking:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Result<u128, ContractError> {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.checked_pow(diff_decimals)
        .ok_or(ContractError::AmountOverflow)?;
    amount.checked_mul(multiplier)
        .ok_or(ContractError::AmountOverflow)
}
```

All callers (`fin_transfer_callback`, `fast_fin_transfer`, `claim_fee_callback`, `denormalize_fee`) should propagate the error and reject the transfer gracefully rather than panicking, so the proof can be marked as processed and the user is informed of the failure without permanently locking funds.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. A user calls `initTransfer` on EVM with `amount = u128::MAX / 10^6 + 1` (≈ `3.4 × 10^32 + 1`). The EVM bridge accepts this (it is a valid `uint128`), locks the tokens, and emits `InitTransfer`.
3. A relayer submits the proof to NEAR `fin_transfer`.
4. `fin_transfer_callback` is invoked. It calls `denormalize_amount(3.4e32 + 1, Decimals { decimals: 18, origin_decimals: 24 })`.
5. The multiplication `(3.4e32 + 1) * 10^6` overflows `u128`. With `overflow-checks = true`, the runtime panics.
6. The transaction reverts. No state is written. The proof is not consumed.
7. Every subsequent relay attempt for the same proof panics identically.
8. The user's EVM tokens are permanently locked in the bridge contract. [1](#0-0) [9](#0-8) [10](#0-9)
