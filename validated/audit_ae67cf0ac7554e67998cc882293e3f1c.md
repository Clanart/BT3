Looking at the production Solana code to verify the claim precisely.

### Title
Missing Pause Guard on `log_metadata` Allows Vault Creation and Wormhole Emission During Full Pause — (`solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana bridge program enforces pause checks on `init_transfer`, `init_transfer_sol`, `finalize_transfer`, and `finalize_transfer_sol`, but the `log_metadata` instruction has no pause guard at all. Any unprivileged user can call it while `config.paused == ALL_PAUSED`, causing a vault PDA to be initialized (`init_if_needed`) and a Wormhole message to be emitted to NEAR.

---

### Finding Description

`ALL_PAUSED` is defined as `INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED` (bits 0 and 1). [1](#0-0) 

Every transfer-related instruction checks the relevant bit before proceeding: [2](#0-1) [3](#0-2) [4](#0-3) 

`log_metadata` has no such check — it unconditionally delegates to `process()`: [5](#0-4) 

Inside `process()`, two irreversible side-effects occur regardless of pause state:

1. **Vault PDA is created** via `init_if_needed`: [6](#0-5) 

2. **Wormhole message is posted** carrying the token's mint address, name, symbol, and decimals: [7](#0-6) 

By contrast, the NEAR-side `log_metadata` is correctly gated with `#[pause(except(roles(Role::DAO)))]`: [8](#0-7) 

---

### Impact Explanation

When the bridge is fully paused, the operator's intent is to halt all cross-chain state mutations. `log_metadata` violates this invariant in two concrete ways:

- **Vault account creation**: a new on-chain account is allocated and owned by the bridge authority for a mint that was never registered before the pause. This is a permanent on-chain state change.
- **Wormhole message emission**: a `LogMetadata` payload is broadcast to NEAR. A relayer can deliver it to NEAR's Wormhole prover, causing NEAR to register the Solana-native token and make it eligible for inbound transfers — while `finalize_transfer` on Solana remains blocked. This creates an asymmetric bridge state: NEAR accepts the token as bridgeable, but Solana cannot finalize any incoming transfers for it until unpaused.

The attacker controls the SPL mint (name, symbol, decimals), so the registered metadata on NEAR reflects attacker-chosen values. While NEAR's `ft_on_transfer` is also paused for regular users, the token registration itself is permanent and persists after unpause, potentially with misleading metadata.

---

### Likelihood Explanation

- Preconditions are minimal: bridge is paused (an expected operational state), attacker holds any native SPL mint not previously registered.
- No privileged access, no leaked keys, no oracle manipulation required.
- The call is permissionless and costs only the Wormhole fee and vault rent.
- Reproducible in a mollusk unit test without any network access.

---

### Recommendation

Add a pause guard to `log_metadata` consistent with the other instructions. Since there is no dedicated `LOG_METADATA_PAUSED` bit, the simplest correct fix is to block it under `ALL_PAUSED` (or introduce a dedicated bit):

```rust
pub fn log_metadata(ctx: Context<LogMetadata>) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused == ALL_UNPAUSED,
        error::ErrorCode::Paused
    );
    msg!("Logging metadata");
    ctx.accounts.process()?;
    Ok(())
}
```

Alternatively, add a dedicated `LOG_METADATA_PAUSED` bit so operators can pause metadata registration independently of transfers.

---

### Proof of Concept

```rust
// mollusk test (pseudocode)
let mut config = default_config();
config.paused = ALL_PAUSED; // 0b11

// vault does not exist yet
assert!(!vault_exists(&program_id, &fresh_mint));

// unprivileged payer calls log_metadata
let result = run_log_metadata_with_config(config, fresh_mint);

// Both side-effects succeed despite full pause:
assert!(result.is_ok());
assert!(vault_exists(&program_id, &fresh_mint));   // vault created
assert!(wormhole_message_emitted(&result));         // Wormhole CPI invoked
```

The vault PDA is derived from `[VAULT_SEED, mint.key()]` and created with `init_if_needed` [6](#0-5) 
and the Wormhole CPI is invoked unconditionally at the end of `process()`. [9](#0-8)
