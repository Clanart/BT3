Audit Report

## Title
Unattributed `address(this).balance` in `pay` allows stranded ETH from prior callers to fund subsequent WETH-input swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` reads `address(this).balance` to determine how much native ETH to wrap as WETH, with no attribution to the current caller. When a user sends `msg.value > amountIn` on a WETH-input swap, the excess ETH is left on the router with no automatic refund. Any subsequent WETH-input swap caller then has that stranded ETH silently applied to their pool payment, receiving a fully-funded swap at zero cost while the original sender loses their excess ETH permanently.

## Finding Description

In `PeripheryPayments.pay`, the WETH branch reads the router's total native balance:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // total router ETH, not caller's ETH
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        ...
    }
}
``` [1](#0-0) 

There is no per-caller accounting. The `receive()` guard only blocks direct ETH transfers from non-WETH addresses: [2](#0-1) 

It does **not** block excess `msg.value` arriving through a legitimate `external payable` swap call. All four swap entry points are `external payable`: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

**Exploit path:**

1. User A calls `exactInputSingle{value: 2_000}(params{ tokenIn: WETH, amountIn: 1_000 })`. Inside `pay`, `nativeBalance = 2_000 >= value = 1_000`, so only 1_000 ETH is deposited and forwarded. The remaining 1_000 ETH stays on the router. No automatic `refundETH` is triggered.
2. User B calls `exactInputSingle{value: 0}(params{ tokenIn: WETH, amountIn: 1_000 })`. Inside `pay`, `nativeBalance = 1_000 >= value = 1_000`, so User A's stranded ETH is deposited and forwarded to the pool. User B's swap is fully funded at zero cost.

`refundETH` exists but is optional and not called automatically: [7](#0-6) 

The mixed native/WETH pattern (`msg.value < amountIn`) is explicitly tested and documented as supported, making it natural for users to also accidentally send `msg.value > amountIn`: [8](#0-7) 

## Impact Explanation

Direct loss of user ETH principal. User A's excess ETH is silently redirected to fund User B's swap. User A receives no tokens for the stolen portion; User B receives their full swap output without spending any ETH or WETH. The loss is bounded only by the excess `msg.value` User A sent. This constitutes a direct loss of user funds above Sherlock thresholds and matches the "direct loss of user principal" allowed impact.

## Likelihood Explanation

Medium. The trigger requires a user to call a swap function directly (not via `multicall`) with `msg.value > amountIn` for a WETH-input swap. The mixed native/WETH flow is explicitly tested and documented as a supported pattern, making overshoot a realistic user mistake. Once ETH is stranded, any observer of chain state can exploit it immediately with a zero-cost follow-up call. The exploit is repeatable and requires no special privileges.

## Recommendation

Track the ETH balance delta attributable to the current call. Snapshot `address(this).balance - msg.value` at entry to each swap function and use only `msg.value` (or the delta) inside `pay` rather than the total `address(this).balance`. Alternatively, enforce `address(this).balance == msg.value` at the start of each non-multicall entry point, reverting if prior ETH is present. A simpler mitigation is to automatically call `refundETH` at the end of each swap entry point, but this does not prevent the root cause — it only limits the window of exploitation.

## Proof of Concept

```solidity
// Step 1 – User A strands 1_000 ETH on the router
router.exactInputSingle{value: 2_000}(ExactInputSingleParams({
    tokenIn:  WETH,
    amountIn: 1_000,   // pay() uses 1_000 ETH; 1_000 ETH left on router
    ...
}));
// router.balance == 1_000 (User A's ETH, unreclaimed)

// Step 2 – User B pays nothing, gets a fully-funded swap
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn:  WETH,
    amountIn: 1_000,   // pay() reads address(this).balance == 1_000 (User A's!)
    ...                // deposits and forwards User A's ETH; pulls 0 from User B
}));
// Result: User A loses 1_000 ETH; User B's swap is fully funded at zero cost.
```

A Foundry test can reproduce this by running two sequential `exactInputSingle` calls in the same test, asserting that User A's balance decreases by 2_000 and User B's WETH/ETH balance is unchanged, while User B still receives swap output tokens.
