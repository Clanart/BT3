### Title
Validator Accepts Unverified `l1_gas_price_wei` / `l1_data_gas_price_wei` from Proposer, Enabling Manipulation of ETH-Denominated Fee Calculations and Derived `eth_to_fri_rate` - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

---

### Summary

`is_block_info_valid` validates only the FRI-denominated L1 gas prices (`l1_gas_price_fri`, `l1_data_gas_price_fri`) against the oracle, but never validates the WEI-denominated counterparts (`l1_gas_price_wei`, `l1_data_gas_price_wei`) that the proposer also broadcasts in `ProposalInit`. Because `convert_to_sn_api_block_info` derives the `eth_to_fri_rate` exclusively from the ratio of the proposer's claimed FRI and WEI prices, a malicious proposer can fix the FRI price within the accepted margin while freely choosing any WEI price, producing an arbitrarily wrong `eth_to_fri_rate`. That rate is then used to compute `l2_gas_price_wei` and to populate `eth_gas_prices` in the `BlockInfo` fed to the blockifier. Every validator re-executes the block using the same unverified WEI prices, so the manipulated `BlockInfo` is accepted by consensus and committed to storage with wrong ETH-denominated fee parameters.

---

### Finding Description

**Validation gap in `is_block_info_valid`**

`is_block_info_valid` calls `get_l1_prices_in_fri_and_wei`, which returns both FRI and WEI prices from the oracle, but immediately discards the WEI result:

```rust
// validate_proposal.rs:286-292
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
```

Only the FRI prices are then compared against the proposer's values:

```rust
// validate_proposal.rs:302-307
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
```

`init_proposed.l1_gas_price_wei` and `init_proposed.l1_data_gas_price_wei` are never compared against `_l1_gas_prices_wei`.

**Downstream use of the unverified WEI prices**

`convert_to_sn_api_block_info` (called by both proposer in `initiate_build` and validator in `initiate_validation`) constructs the `BlockInfo` directly from the proposer's `ProposalInit`:

```rust
// utils.rs:301-305
let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
...
let previous_block_info = PreviousBlockInfo::from(init);
let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
```

`calculate_eth_to_fri_rate` computes:

```
eth_to_fri_rate = l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei
```

This rate is then used to derive `l2_gas_price_wei`:

```rust
// utils.rs:307
let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)?;
```

All three values — `l1_gas_price_wei`, `l1_data_gas_price_wei`, and the derived `l2_gas_price_wei` — are placed into `eth_gas_prices` of the `BlockInfo` that the blockifier uses for every ETH-denominated fee calculation in the block.

**Analog to the SponsorVault spot-price bug**

The SponsorVault bug used a manipulable DEX spot price instead of an oracle price to determine how many native tokens to spend. Here, the sequencer uses the proposer's self-reported WEI price (analogous to the spot price) instead of the oracle's WEI price to derive the ETH/STRK conversion rate and the ETH-denominated L2 gas price. In both cases, the attacker controls the "price" input to a critical calculation while the system only validates a different denomination of the same quantity.

---

### Impact Explanation

**Incorrect fee with economic impact (Critical scope item).**

A malicious proposer who keeps `l1_gas_price_fri` within the accepted margin but sets `l1_gas_price_wei` to an extreme value causes:

1. **Wrong `eth_to_fri_rate`** — all ETH↔STRK conversions inside the block use a fabricated rate.
2. **Wrong `l2_gas_price_wei`** — ETH-denominated L2 gas fees charged to every transaction in the block are computed from the fabricated rate.
3. **Wrong `eth_gas_prices.l1_gas_price` and `eth_gas_prices.l1_data_gas_price`** — ETH-denominated L1 and data gas fees are set to the proposer's arbitrary WEI values.
4. **Committed to storage** — because every validator re-executes using the same `ProposalInit`, all validators compute the same (wrong) block hash and state root; the manipulated block is accepted by consensus and written to the MDBX storage with wrong fee parameters.

Setting `l1_gas_price_wei` to 1 makes `eth_to_fri_rate` ≈ `l1_gas_price_fri × 10^18`, collapsing `l2_gas_price_wei` to near zero and making ETH-denominated L2 gas essentially free. Setting it to `u128::MAX` has the opposite effect, making ETH-denominated fees astronomically large and causing transactions to fail or users to overpay.

---

### Likelihood Explanation

In a decentralized validator set any elected proposer can trigger this. The manipulation requires only setting two fields in `ProposalInit` to non-zero values that pass the nonzero check in `convert_to_sn_api_block_info`; no special cryptographic capability is needed. The FRI price constraint is the only guard, and it is satisfied by keeping `l1_gas_price_fri` within the configured margin while freely choosing `l1_gas_price_wei`.

---

### Recommendation

In `is_block_info_valid`, after computing `_l1_gas_prices_wei`, compare the proposer's WEI prices against the oracle's WEI prices using the same `within_margin` check already applied to FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

// existing FRI checks ...

if !(within_margin(init_proposed.l1_gas_price_wei,    l1_gas_prices_wei.l1_gas_price,      margin)
  && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

This mirrors the existing FRI validation and ensures the proposer cannot decouple the WEI price from the oracle-verified FRI price to fabricate an arbitrary `eth_to_fri_rate`.

---

### Proof of Concept

1. Malicious proposer is elected for height N.
2. Oracle reports `l1_gas_price_wei = 10_000_000_000` (10 Gwei) and `eth_to_fri_rate = 3_000_000` (example).
3. Proposer sets in `ProposalInit`:
   - `l1_gas_price_fri` = oracle FRI value (passes `within_margin` check).
   - `l1_gas_price_wei` = **1** (not validated).
4. `is_block_info_valid` passes — only FRI prices are checked. [1](#0-0) 
5. `convert_to_sn_api_block_info` is called with the proposer's `ProposalInit`:
   - `eth_to_fri_rate = l1_gas_price_fri × 10^18 / 1` — inflated by factor of 10^10.
   - `l2_gas_price_wei = l2_gas_price_fri × 10^18 / eth_to_fri_rate` — deflated by same factor, near zero. [2](#0-1) 
6. `calculate_eth_to_fri_rate` divides by the proposer's `l1_gas_price_wei = 1`: [3](#0-2) 
7. Every validator calls `initiate_validation` → `convert_to_sn_api_block_info` with the same `ProposalInit`, producing the same fabricated `BlockInfo`. [4](#0-3) 
8. The blockifier executes all transactions in the block using `eth_gas_prices.l2_gas_price ≈ 0`, making ETH-denominated L2 gas free. Fee transfers in the state reflect this wrong price.
9. Consensus reaches agreement; the block with wrong ETH-denominated fee parameters is committed to storage.

        .l1_prices_fri
        .l1_gas_price
        .0
