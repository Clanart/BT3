### Title
Stale `feemarketParams` Snapshot in `newCosmosAnteHandler` Bypasses Governance-Updated Fee Enforcement for Cosmos Transactions - (File: `evmd/ante/handler_options.go`)

### Summary

`newCosmosAnteHandler` reads `feemarketParams` once at app-startup time and passes a pointer to that fixed local copy into both `MinGasPriceDecorator` and `NewDynamicFeeChecker`. Any subsequent governance update to `MinGasPrice` (or the natural per-block drift of `BaseFee`) is never reflected in those decorators. Cosmos transactions can therefore be submitted with fees below the live on-chain minimum and will be accepted, bypassing the fee-market enforcement that EVM transactions correctly enforce.

### Finding Description

In `evmd/ante/handler_options.go`, `newCosmosAnteHandler` is called **once** at app-startup from `setAnteHandler` → `ante.NewAnteHandler`:

```go
// evmd/ante/handler_options.go:178-198
func newCosmosAnteHandler(ctx sdk.Context, options HandlerOptions, ...) sdk.AnteHandler {
    evmParams     := options.EvmKeeper.GetParams(ctx)          // snapshot at startup
    feemarketParams := options.FeeMarketKeeper.GetParams(ctx)  // snapshot at startup
    ...
    txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)
    ...
    cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams),
```

`feemarketParams` is a **value-type local variable**; `&feemarketParams` is a pointer to that heap-escaped copy. It is never updated again. Both decorators read from this frozen snapshot for every subsequent transaction:

- `MinGasPriceDecorator.AnteHandle` reads `mpd.feemarketParams.MinGasPrice` (line 54) — the startup value — instead of calling `mpd.feesKeeper.GetParams(ctx).MinGasPrice`.
- `NewDynamicFeeChecker` calls `types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)` (line 56) using the snapshot `feemarketParams.BaseFee` — the genesis/startup base fee — instead of the live per-block value.

By contrast, `newEthAnteHandler` (the EVM path) correctly re-reads params on every invocation:

```go
// evmd/ante/handler_options.go:88-95
blockCfg, err := options.EvmKeeper.EVMBlockConfig(ctx, options.EvmKeeper.ChainID())
evmParams        := &blockCfg.Params
feemarketParams  := &blockCfg.FeeMarketParams
baseFee          := blockCfg.BaseFee
```

The asymmetry is the root cause: EVM transactions get live params; Cosmos transactions get a frozen snapshot.

`MinGasPriceDecorator` even stores `feesKeeper interfaces.FeeMarketKeeper` in its struct — the live keeper is present but never consulted for the actual check.

### Impact Explanation

**High** — ante handler bug that permits invalid Cosmos transactions (with fees below the governance-updated `MinGasPrice` or below the current `BaseFee`) to commit.

1. **MinGasPrice bypass**: After governance raises `MinGasPrice` (a common spam-prevention action), the `MinGasPriceDecorator` for Cosmos transactions still enforces the old, lower value. Attackers can flood the chain with Cosmos transactions paying only the old minimum, bypassing the updated spam floor and mis-accounting fees relative to the live protocol parameters.

2. **BaseFee bypass via `NewDynamicFeeChecker`**: The `feemarketParams.BaseFee` snapshot is the genesis value (1 gwei). As the chain runs and `BeginBlock` updates the base fee upward under load, `NewDynamicFeeChecker` for Cosmos transactions continues to compute `effectiveFee` against the stale genesis base fee. A Cosmos transaction whose `feeCap / gas` is above the genesis base fee but below the current live base fee will pass the fee checker and be accepted, even though it would be rejected for an equivalent EVM transaction.

### Likelihood Explanation

**Medium-High.** The trigger conditions are routine:
- Governance `MsgUpdateParams` raising `MinGasPrice` is a standard operational action on any live Ethermint chain.
- The base fee drifts upward naturally whenever blocks are above the gas target — a normal condition under any non-trivial load.

No special privileges are required. Any unprivileged user can submit a Cosmos transaction (e.g., `MsgSend`, `MsgDelegate`, IBC relayer messages) with a fee between the stale and live minimum. The bypass is silent — no error is returned.

### Recommendation

**Short term**: In `MinGasPriceDecorator.AnteHandle`, replace the snapshot read with a live keeper call:

```go
// ante/cosmos/min_gas_price.go:54
// BEFORE (stale):
minGasPrice := mpd.feemarketParams.MinGasPrice
// AFTER (live):
minGasPrice := mpd.feesKeeper.GetParams(ctx).MinGasPrice
```

Similarly, in `NewDynamicFeeChecker`, read `feemarketParams` from the keeper inside the returned closure rather than closing over the startup snapshot:

```go
// ante/evm/fee_checker.go:56
// BEFORE (stale):
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
// AFTER (live): read from keeper inside the closure
liveParams := feemarketKeeper.GetParams(ctx)
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, &liveParams)
```

**Long term**: Align `newCosmosAnteHandler` with `newEthAnteHandler` by removing the `ctx sdk.Context` construction-time parameter and reading all fee-market state inside the returned closure, mirroring the `EVMBlockConfig` pattern used for EVM transactions.

### Proof of Concept

1. Chain starts with `MinGasPrice = 0` (default). `newCosmosAnteHandler` is called once; `feemarketParams.MinGasPrice = 0` is frozen.

2. Governance submits and passes `MsgUpdateParams` setting `MinGasPrice = 1_000_000_000` (1 gwei). The KV store is updated; `FeeMarketKeeper.GetParams(ctx).MinGasPrice` now returns `1_000_000_000`.

3. Attacker submits a `MsgSend` Cosmos transaction with `fee = 0` (or any value below 1 gwei × gasLimit).

4. `MinGasPriceDecorator.AnteHandle` reads `mpd.feemarketParams.MinGasPrice` → `0` (the frozen snapshot). The check `if minGasPrice.IsZero()` short-circuits and the decorator passes immediately (line 57 of `ante/cosmos/min_gas_price.go`).

5. The transaction commits with zero fee, bypassing the governance-mandated minimum.

6. An equivalent EVM transaction at the same gas price would be correctly rejected by `newEthAnteHandler`, which reads live params via `EVMBlockConfig`.
