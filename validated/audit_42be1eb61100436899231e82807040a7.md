### Title
`execute_deploy` Syscall Always Reports Success Regardless of Constructor Outcome — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` in the response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. A calling contract that relies on `failure_flag` to determine deployment success will always see "success," even when the constructor failed and its state changes were rolled back. This allows a calling contract to proceed with incorrect assumptions — for example, transferring funds to a contract address that was never actually initialized — resulting in permanent loss of funds.

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–555), after calling `deploy_contract`, the response header is written with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call itself participates in the revert mechanism (it carries the `revert_log` implicit argument), meaning that when a constructor reverts, `handle_revert` is invoked inside `execute_entry_point` to undo all state changes — including the class hash assignment for the new contract address:

```cairo
if (is_reverted != FALSE) {
    handle_revert(
        contract_address=execution_context.execution_info.contract_address,
        revert_log_end=revert_log,
    );
    ...
    return (is_reverted=is_reverted, ...);
}
``` [2](#0-1) 

So the OS correctly reverts the internal state when a constructor fails, but it **does not propagate this failure to the calling contract**. The calling contract receives `failure_flag=0` and a valid-looking `contract_address` in the `DeployResponse`, and has no way to distinguish a successful deployment from a failed one.

Contrast this with `contract_call_helper`, which correctly propagates failure:

```cairo
if (is_reverted != FALSE) {
    // Append `ERROR_ENTRY_POINT_FAILED` to the retdata.
    assert retdata[retdata_size] = ERROR_ENTRY_POINT_FAILED;
    ...
}
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The `execute_deploy` path has no equivalent check — the `TODO` comment is the only acknowledgment of this gap.

### Impact Explanation

**Critical — Direct loss of funds.**

A calling contract that uses the `deploy` syscall and then transfers tokens or ETH to the returned `contract_address` (trusting `failure_flag=0` as confirmation of successful deployment) will send funds to an address whose class hash was reverted. That address has no deployed class, so any subsequent call to it will fail with `ERROR_ENTRY_POINT_NOT_FOUND`. The funds are permanently locked with no recourse, because the OS proof will attest to a state where the address has no class hash.

### Likelihood Explanation

This is reachable by any unprivileged user who:
1. Deploys or calls a contract that internally uses the `deploy` syscall.
2. The inner constructor reverts — this can happen due to out-of-gas (attacker controls gas bounds via `resource_bounds`), or due to any constructor-level assertion failure.
3. The calling contract proceeds to transfer funds to the returned address, trusting the always-`0` `failure_flag`.

The `TODO` comment explicitly acknowledges this is unimplemented, confirming the code path is live and the failure case is not handled.

### Recommendation

Propagate the constructor's revert status to the `failure_flag` in the `DeployResponse`, mirroring the pattern used in `contract_call_helper`:

```cairo
// Replace:
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);

// With (after capturing is_reverted from deploy_contract):
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

`deploy_contract` should be updated to return `is_reverted` so the caller can use it. When `is_reverted=1`, the `DeployResponse` should contain an error retdata (e.g., `ERROR_ENTRY_POINT_FAILED`) rather than a `contract_address`, consistent with how `contract_call_helper` handles reverted calls.

### Proof of Concept

1. User submits an invoke transaction calling contract `A`.
2. Contract `A` executes the `deploy` syscall targeting class `B`, whose constructor always reverts (e.g., `assert 1 = 0`).
3. The OS executes `execute_deploy`:
   - `deploy_contract` runs the constructor, which reverts.
   - `handle_revert` undoes the class hash assignment for the new address.
   - **`failure_flag=0` is written unconditionally.**
4. Contract `A` receives `DeployResponse(contract_address=X, failure_flag=0)`.
5. Contract `A` calls `transfer(recipient=X, amount=100)` on an ERC-20 token, believing `X` is a live contract.
6. The ERC-20 transfer succeeds (it only updates storage, no call to `X`).
7. The proven state has `X` with no class hash and 100 tokens in its balance.
8. Any future attempt to call `X` fails — funds are permanently frozen. [4](#0-3)
