### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. A contract can replace its own class hash with any arbitrary felt value. Once committed to state, future calls to that contract will fail at the OS level because no compiled class can be found for the invalid hash, permanently freezing any funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any validation that the hash corresponds to a declared class:

```cairo
func execute_replace_class{
    ...
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. The OS unconditionally commits the new class hash to state.

When any subsequent transaction attempts to call the now-bricked contract, `execute_entry_point` performs:

1. A `dict_read` on `contract_class_changes` for the (invalid) class hash — returning `0` for an undeclared hash.
2. A `find_element` search through `compiled_class_facts` for compiled class hash `0`. [2](#0-1) 

If no compiled class with that hash exists, `find_element` raises an assertion failure inside the OS, making the block unprovable. The sequencer therefore cannot include any transaction that calls the bricked contract, leaving its funds permanently inaccessible.

---

### Impact Explanation

**Permanent freezing of funds.** Once a contract's class hash is replaced with an undeclared value:

- The state change is final and committed to the Merkle tree.
- Every future call to the contract causes an OS-level assertion failure in `find_element`, so the sequencer cannot include such transactions in any provable block.
- Any ERC-20 balances, ETH, STRK, or other assets stored in the contract's storage are permanently inaccessible — no `transfer`, `withdraw`, or other entry point can ever be executed again.

This matches the allowed impact: **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract without privilege. Realistic paths include:

1. **Buggy upgrade logic**: A contract with an on-chain upgrade mechanism (e.g., a multisig or DAO-controlled proxy) passes an incorrect class hash due to an off-chain computation error or a front-running attack that substitutes the hash before the transaction lands.
2. **Malicious contract**: An attacker deploys a contract that deliberately calls `replace_class` with `0` or any arbitrary felt, then self-destructs its upgrade path, permanently bricking the contract and any funds it holds.
3. **Accidental misuse**: A developer passes a Sierra class hash instead of the compiled (CASM) class hash, or passes a hash that was never declared on-chain.

Because the OS imposes no guard, any of these paths succeeds silently at the protocol level.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., it has been declared in the current or a prior block):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);  // 0 means undeclared.
```

This mirrors the check already performed implicitly in `execute_entry_point` when resolving a class hash to its compiled form, but applied eagerly at the point of `replace_class` so that invalid replacements are rejected rather than silently committed.

---

### Proof of Concept

1. Declare a valid class `C` and deploy contract `V` (a vault holding 1000 STRK) using class `C`.
2. `V` calls `replace_class(class_hash=0xdeadbeef)` — an undeclared hash.
3. The OS executes `execute_replace_class`: no validation occurs; `contract_state_changes[V.address].class_hash` is set to `0xdeadbeef`.
4. The block is proven successfully (the `replace_class` call itself succeeds).
5. Any subsequent transaction invoking `V.withdraw(...)` causes `execute_entry_point` to call `find_element(..., key=0)` (since `dict_read` on `contract_class_changes` for `0xdeadbeef` returns `0`). `find_element` raises an assertion failure; the block cannot be proven.
6. The sequencer permanently excludes all calls to `V`. The 1000 STRK are frozen forever. [3](#0-2) [4](#0-3)
