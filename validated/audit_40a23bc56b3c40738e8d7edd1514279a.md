### Title
Non-EVM Cosmos Bank Token Balances Permanently Orphaned After Contract SelfDestruct — (`x/evm/statedb/statedb.go`)

### Summary

When an EVM contract self-destructs, Ethermint's `StateDB.Commit()` only burns the EVM-native denom balance. Any non-EVM Cosmos bank tokens (IBC tokens, CosmWasm bridge tokens, etc.) held by the contract address are never drained. Because `DeleteAccount` simultaneously removes the account from the auth module and erases all contract code and storage, those bank balances become permanently inaccessible orphaned funds — an exact structural analog to the Coded Estate burn-with-active-rentals bug.

---

### Finding Description

**Root cause — `x/evm/statedb/statedb.go` `Commit()`, lines 800–825**

When a self-destructed object is flushed to the keeper, the commit loop explicitly limits its cleanup to the EVM denom:

```go

// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.

**`DeleteAccount` removes the account entirely — `x/evm/keeper/statedb.go`, lines 195–222**

`DeleteAccount` clears EVM storage and removes the auth-module account record. Its own comment acknowledges that balance handling is out of scope ("NOTE: balance should be cleared separately"), but that note only ever refers to the EVM denom. No code path enumerates or drains non-EVM bank balances before the account is removed. [2](#0-1) 

After `DeleteAccount` returns:
1. The auth-module account is gone — no signer can authorize a `MsgSend` from that address.
2. All contract code and storage are deleted — the contract can never be called again.
3. Any IBC / bridge tokens still recorded in the bank module at that address have no owner and no recovery path.

---

### Impact Explanation

Any Cosmos bank tokens (IBC USDC, ATOM, staking derivatives, CosmWasm-bridged assets, etc.) deposited into a contract that subsequently self-destructs are permanently frozen. Unlike the EVM denom, they are never burned, never redistributed, and never refunded. The bank module retains the balance entry at the now-deleted address indefinitely, but no transaction can move those coins. This is a permanent, irreversible loss of user funds — matching the "valid user funds to be mis-accounted" High impact category.

---

### Likelihood Explanation

Cosmos EVM chains routinely hold IBC tokens in EVM contracts (DEX pools, lending protocols, bridges). The SELFDESTRUCT opcode is reachable by any contract deployer — an entirely unprivileged role. A malicious deployer can:

1. Deploy a contract that accepts IBC token deposits.
2. Attract user deposits.
3. Call SELFDESTRUCT (directly or via a backdoor).
4. All deposited IBC tokens are permanently orphaned.

Even without malicious intent, any legitimate contract that holds IBC tokens and is later decommissioned via SELFDESTRUCT silently destroys those balances. No governance action, no privileged role, and no network-level attack is required.

---

### Recommendation

Before calling `DeleteAccount`, enumerate all bank-module balances held by the contract address and either burn them (matching the EVM-denom treatment) or send them to a designated recovery address. The `BankKeeper.GetAllBalances` / `SpendableCoins` query provides the full multi-denom balance. The drain should be wrapped in the same `CacheContext` already used for the EVM-denom burn so that a partial failure rolls back atomically.

---

### Proof of Concept

1. Deploy a Solidity contract that accepts IBC-USDC via a `receive()` fallback and exposes a `destroy()` function calling `selfdestruct(payable(owner))`.
2. Have a user send 1000 IBC-USDC to the contract via a Cosmos `MsgSend` (the bank module records the balance at the contract's Cosmos address).
3. The deployer calls `destroy()`.
4. `StateDB.Commit()` burns only the EVM denom balance (zero in this case); `DeleteAccount` removes the auth account.
5. Query `bank.Balance(contractAddr, "ibc/...")` — it still returns 1000. Query `auth.Account(contractAddr)` — it returns not-found. The 1000 IBC-USDC are permanently orphaned with no recovery path. [3](#0-2) [4](#0-3)

```go
// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
