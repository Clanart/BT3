### Title
Batch `MsgEthereumTx` Balance Check Does Not Account for Cumulative Cost Across Messages — (`File: ante/eth.go`)

### Summary

`VerifyEthAccount` and `CheckEthCanTransfer` in `ante/eth.go` validate each `MsgEthereumTx` in a batch transaction independently against the sender's **original, unmodified balance**. Neither function accumulates the cost of prior messages before checking the next one. This is the direct structural analog of the reported `depositCap` bug: a cap/limit is checked per-operation rather than against the running total, allowing the cumulative obligation to exceed the available balance.

### Finding Description

`VerifyEthAccount` iterates over every message in the batch and calls `CheckSenderBalance` with the same on-chain balance for each message:

```go
// ante/eth.go lines 76–105
for _, msg := range tx.GetMsgs() {
    // ...
    balance := evmKeeper.GetBalance(ctx, from, evmDenom)   // same balance every iteration
    if err := keeper.CheckSenderBalance(sdkmath.NewIntFromBigIntMut(balance.ToBig()), ethTx); err != nil {
        return errorsmod.Wrap(err, "failed to check sender balance")
    }
}
```

`CheckSenderBalance` only compares the current message's cost against the balance:

```go
// x/evm/keeper/utils.go lines 184–203
func CheckSenderBalance(balance sdkmath.Int, tx *ethtypes.Transaction) error {
    cost := tx.Cost()   // gasLimit*gasPrice + value for THIS message only
    if balance.IsNegative() || balance.BigInt().Cmp(cost) < 0 {
        return errorsmod.Wrapf(errortypes.ErrInsufficientFunds, ...)
    }
    return nil
}
```

`CheckEthCanTransfer` has the same pattern — it reads the same unmodified balance for every message and checks only `balance >= value_of_this_message`:

```go
// ante/eth.go lines 213–250
for _, msg := range tx.GetMsgs() {
    // ...
    if value.Sign() > 0 && !canTransfer(ctx, evmKeeper, evmParams.EvmDenom, from, value) {
        return errorsmod.Wrapf(errortypes.ErrInsufficientFunds, ...)
    }
}
```

`CheckEthGasConsume`, which runs after both of the above, **does** deduct fees sequentially (so total fees are bounded), but it only deducts the **fee** component (`gasLimit × effectiveGasPrice`), not the **value** component. The value transfers happen later during EVM execution.

### Impact Explanation

Consider a sender with balance `B = 200,000` submitting a batch of two messages, each with `value = 90,000` and `fee = 21,000` (cost per message = 111,000; total cost = 222,000 > 200,000):

| Step | Check | Result |
|---|---|---|
| `VerifyEthAccount` msg1 | `200,000 >= 111,000` | ✓ pass |
| `VerifyEthAccount` msg2 | `200,000 >= 111,000` | ✓ pass (balance not updated) |
| `CheckEthCanTransfer` msg1 | `200,000 >= 90,000` | ✓ pass |
| `CheckEthCanTransfer` msg2 | `200,000 >= 90,000` | ✓ pass (balance not updated) |
| `CheckEthGasConsume` msg1 | deduct 21,000 → balance 179,000 | ✓ |
| `CheckEthGasConsume` msg2 | deduct 21,000 → balance 158,000 | ✓ |
| EVM exec msg1 | transfer 90,000 → balance 68,000 | ✓ |
| EVM exec msg2 | transfer 90,000, balance 68,000 < 90,000 | ✗ EVM revert |

The ante handler admits the batch. Both fees (42,000 total) are permanently deducted. msg2's value transfer fails at EVM execution. The sender loses 21,000 in fees for a transaction the ante handler should have rejected. This is a direct mis-accounting of user funds/fees through an invalid transaction that the ante handler permitted to commit.

This matches the allowed High impact: *"ante handler … bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

Batch `MsgEthereumTx` transactions are a supported Cosmos SDK feature in Ethermint (see `evmd/ante/handler_options.go` and `tests/integration_tests/test_batch.py`). Any unprivileged user can construct a batch transaction via `eth_sendRawTransaction` or the Cosmos broadcast endpoint. The condition is triggered whenever the sender's balance is sufficient for each individual message but not for all messages combined — a natural edge case for any user sending multiple value transfers in a single batch.

### Recommendation

`VerifyEthAccount` should accumulate the total cost across all messages from the same sender before comparing against the balance:

```go
totalCost := new(big.Int)
for _, msg := range tx.GetMsgs() {
    ethTx := msgEthTx.AsTransaction()
    totalCost.Add(totalCost, ethTx.Cost())
}
if balance.BigInt().Cmp(totalCost) < 0 {
    return errorsmod.Wrapf(errortypes.ErrInsufficientFunds, ...)
}
```

Similarly, `CheckEthCanTransfer` should accumulate total value across all messages before checking against the balance (after accounting for fees already deducted by `CheckEthGasConsume`).

func CheckSenderBalance(
	balance sdkmath.Int,
	tx *ethtypes.Transaction,
) error {
	cost := tx.Cost()

	if cost.Sign() < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInvalidCoins,
			"tx cost (%s) is negative and invalid", cost,
		)
	}

	if balance.IsNegative() || balance.BigInt().Cmp(cost) < 0 {
		return errorsmod.Wrapf(
			errortypes.ErrInsufficientFunds,
			"sender balance < tx cost (%s < %s)", balance, tx.Cost(),
		)
	}
	return nil
