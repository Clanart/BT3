### Title
Deploy Syscall Always Reports Success Regardless of Constructor Revert - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` in its response, even when the deployed contract's constructor reverts. Any caller contract that relies on this flag to determine deployment success will proceed with incorrect assumptions, potentially sending funds to an address where no contract was actually deployed, permanently freezing those funds.

### Finding Description
In `execute_deploy` (syscall_impls.cairo), after calling `deploy_contract`, the OS writes a hardcoded success response:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` function is invoked with `revert_log` as an implicit argument (inherited from `execute_deploy`'s own implicit arguments), meaning the constructor can revert and the revert log will undo its state changes. However, `deploy_contract` only returns `(retdata_size, retdata)` — no failure flag — so `execute_deploy` has no mechanism to detect a constructor revert and always reports `failure_flag=0` to the calling contract. [2](#0-1) 

Compare this with the correct pattern used in `contract_call_helper`, which properly propagates `is_reverted` into the response header: [3](#0-2) 

The `execute_deploy_account_transaction` path in `transaction_impls.cairo` also calls `deploy_contract` with a `revert_log`, confirming that constructor reverts are a real, handled scenario in other code paths — just not in the syscall response: [4](#0-3) 

### Impact Explanation
When a constructor reverts:
1. The revert log undoes the constructor's state changes — the contract is **not** actually deployed at the returned address.
2. The OS still writes `failure_flag=0` and returns the computed `contract_address` to the caller.
3. The caller contract, seeing a success response, may transfer tokens or ETH-equivalent assets to the returned address.
4. Since no contract exists at that address (deployment was reverted), those funds are permanently frozen with no mechanism for recovery.

This matches **Critical — Permanent freezing of funds** from the allowed impact scope.

### Likelihood Explanation
The `deploy` syscall is a standard, permissionless StarkNet syscall callable by any contract. An attacker can:
- Declare a class whose constructor unconditionally reverts (a valid, permissionless operation via a declare transaction).
- Interact with any protocol that uses the `deploy` syscall and passes an attacker-controlled `class_hash` parameter.
- The protocol's contract will receive `failure_flag=0`, believe deployment succeeded, and proceed to fund the address.

No privileged access, leaked keys, or operator cooperation is required. The entry point is fully reachable by an unprivileged transaction sender.

### Recommendation
`deploy_contract` should be modified to return an `is_reverted` flag (analogous to `select_execute_entry_point_func`). `execute_deploy` must then propagate this flag into the `ResponseHeader`:

```cairo
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This mirrors the pattern already used correctly in `contract_call_helper`. [5](#0-4) 

### Proof of Concept
1. Attacker declares class `C` whose constructor body is `assert 1 = 0` (always reverts).
2. Victim protocol contract `P` exposes a function `fund_new_contract(class_hash)` that:
   - Calls `deploy(class_hash, ...)` syscall.
   - Reads `response.failure_flag` — sees `0` (success, due to the bug).
   - Transfers `N` tokens to `response.contract_address`.
3. Attacker calls `P.fund_new_contract(C)`.
4. OS executes constructor of `C`, which reverts; revert log undoes state changes; no contract exists at the returned address.
5. OS writes `failure_flag=0` to the syscall response.
6. `P` transfers `N` tokens to the address — permanently frozen, as no contract can call `transfer` to recover them. [6](#0-5)
