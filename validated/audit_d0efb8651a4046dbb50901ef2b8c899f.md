## Analysis

### Step 1: Production entrypoint

The flow is: peer → `PeerMessage::VersionedStateResponse` → `StateResponseReceived` → `StateSyncDownloader::ensure_shard_header` → `StateHeaderValidationRequest` → `set_state_header(shard_id=Y, sync_hash, header)`.

Any unauthenticated peer can serve a `VersionedStateResponse` message. [1](#0-0) 

### Step 2: The missing shard binding check

`set_state_header` extracts the chunk from the header but **never checks that `chunk.shard_id() == shard_id`**. The `shard_id` parameter is used only to construct the storage key: [2](#0-1) 

### Step 3: `verify_path` is position-agnostic

Step 3a verifies the chunk is in the block's `chunk_headers_root`: [3](#0-2) 

The `verify_path` implementation only checks `compute_root_from_path(path, hash(item)) == root`. It does **not** check which position (shard index) the item occupies in the tree: [4](#0-3) 

A valid Merkle proof for chunk X at position X will pass `verify_path` even when the caller claims `shard_id=Y`. The position-aware variant `verify_path_with_index` is not used here (it is used elsewhere, e.g. in chunk part validation). [5](#0-4) 

### Step 4: `validate_state_root_node` is shard-agnostic

Step 5 validates the state root node against `chunk_inner.prev_state_root()` — which is shard X's root, since the chunk is for shard X: [6](#0-5) 

The implementation only checks `hash(state_root_node.data) == state_root` with no shard binding: [7](#0-6) 

### Step 5: Downstream corruption in `set_state_part`

After the corrupted header is stored, `set_state_part` reads it back and uses `chunk.prev_state_root()` — now shard X's root — as the authoritative root for shard Y: [8](#0-7) 

### Step 6: Receipt proof check is not a blocker

Step 4 verifies incoming receipt proofs for `shard_id` (Y). These are real on-chain proofs an attacker can collect from the network. The check does not bind the receipts to the chunk's `shard_id`. [9](#0-8)

### Title
Missing shard-ID binding in `set_state_header` allows unauthenticated peer to install wrong shard's state root — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` for `shard_id=Y` that internally contains a chunk for shard X. Because `verify_path` is position-agnostic and `validate_state_root_node` carries no shard binding, all five validation steps pass. The header is committed to `DBCol::StateHeaders` under `StateHeaderKey(Y, sync_hash)` with shard X's `prev_state_root` as the authoritative root for shard Y. Every subsequent `set_state_part` call for shard Y then validates trie parts against shard X's root.

### Finding Description
In `chain/chain/src/state_sync/adapter.rs`, `set_state_header` never asserts `chunk.shard_id() == shard_id`. Step 3a calls `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk_X.hash, chunk_X.height_included))`. Because `verify_path` is a pure membership proof (no index check), a valid proof for chunk X at position X satisfies the check even when the caller passes `shard_id=Y`. Step 5 calls `validate_state_root_node(state_root_node_X, chunk_X.prev_state_root())`, which passes because the node is internally consistent with shard X's root. The header is then written under `StateHeaderKey(Y, sync_hash)` with `chunk_prev_state_root = state_root_X`.

### Impact Explanation
The exact corrupted value is `DBCol::StateHeaders[StateHeaderKey(Y, sync_hash)].chunk_prev_state_root = state_root_X`. All `set_state_part(Y, ...)` calls read this header and validate trie parts against `state_root_X`. A cooperating attacker who also serves valid parts for shard X's trie causes `set_state_finalize` to install shard X's trie under shard Y's `ShardUId`, corrupting shard Y's committed state permanently. Even without part cooperation, the stored corrupted header causes all honest parts (for shard Y's root) to fail validation, blocking state sync for shard Y indefinitely.

### Likelihood Explanation
Any peer reachable by a syncing node can serve `VersionedStateResponse`. The attacker needs: (1) a real chunk for shard X with its Merkle proof (observable from the network), (2) real incoming receipt proofs for shard Y (observable from the network), and (3) a valid `StateRootNode` for shard X's root (computable from the trie). No validator or operator privilege is required.

### Recommendation
Add an explicit shard-ID binding check immediately after extracting the chunk in `set_state_header`:
```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```
Additionally, consider replacing the position-agnostic `verify_path` call in step 3a with `verify_path_with_index`, passing the expected shard index derived from `shard_id`, to enforce positional binding in the Merkle proof.

### Proof of Concept
1. Obtain chunk X (for shard 0) and its Merkle proof from the block at `sync_prev_hash`.
2. Obtain valid incoming receipt proofs for shard 1 from the same block range.
3. Construct `ShardStateSyncResponseHeaderV2 { chunk: chunk_X, chunk_proof: proof_X, state_root_node: node_for_state_root_X, incoming_receipts_proofs: receipts_for_shard_1, ... }`.
4. Call `set_state_header(shard_id=1, sync_hash, crafted_header)`.
5. Assert the call returns `Ok(())`.
6. Read back `DBCol::StateHeaders[StateHeaderKey(1, sync_hash)]` and assert `chunk_prev_state_root() == state_root_X` (shard 0's root, not shard 1's root).

    fn validate_state_root_node(
        &self,
        state_root_node: &StateRootNode,
        state_root: &StateRoot,
    ) -> StateRootNodeValidationResult {
        if state_root == &Trie::EMPTY_ROOT {
            return if state_root_node == &StateRootNode::empty() {
                StateRootNodeValidationResult::Valid
            } else {
                StateRootNodeValidationResult::Invalid
            };
        }
        if hash(&state_root_node.data) != *state_root {
            return StateRootNodeValidationResult::Invalid;
        }
        match Trie::get_memory_usage_from_serialized(&state_root_node.data) {
            Ok(memory_usage) => {
                if memory_usage == state_root_node.memory_usage {
                    StateRootNodeValidationResult::Valid
                } else {
                    StateRootNodeValidationResult::Invalid
                }
            }
            Err(_) => StateRootNodeValidationResult::Invalid, // Invalid state_root_node
        }
