### Title
Replay Attack via Nonce-less `meta_tx_v0` Hash Enables Direct Loss of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes the meta-transaction hash without including a nonce (`additional_data_size=0`), and `execute_meta_tx_v0` hardcodes `nonce=0` in the constructed `TxInfo`. The OS's own nonce-check logic explicitly skips version-0 transactions. As a result, any meta_tx_v0 signature observed on-chain can be replayed an unlimited number of times by an unprivileged attacker, causing repeated unauthorized invocations of the target account's `__execute__` entry point and direct loss of funds.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` is a syscall callable by any contract during execution. It constructs a new `TxInfo` with `version=0` and `nonce=0` (hardcoded), then calls the target account's `__execute__` entry point with a caller-supplied signature: [1](#0-0) 

The transaction hash for this call is computed by `compute_meta_tx_v0_hash`, which delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)` — meaning **no nonce is committed to the hash**: [2](#0-1) 

The OS's `check_and_increment_nonce` function explicitly skips nonce enforcement for version-0 transactions: [3](#0-2) 

This is the direct analog of the `MinimalForwarder` issue in the reference report: just as `MinimalForwarder` signed requests had no expiry and no replay protection, `meta_tx_v0` signed requests carry no nonce and no expiry. The hash is fully determined by `(contract_address, selector, calldata, chain_id)` — all of which are static and observable on-chain.

---

### Impact Explanation

An attacker who observes a valid `meta_tx_v0` signature in any on-chain transaction can replay it by submitting a new outer (v3) transaction that calls the same relayer/forwarder contract with the identical `(contract_address, selector, calldata, signature)` tuple. Each replay causes the target account's `__execute__` to be invoked with the same calldata and a cryptographically valid signature (the hash is identical). If the calldata encodes a token transfer or any fund-moving operation, the attacker can drain the account. This is **direct loss of funds** (Critical).

---

### Likelihood Explanation

The `meta_tx_v0` syscall is reachable by any unprivileged transaction sender: any contract can issue it, and any outer v3 transaction can trigger such a contract. All `meta_tx_v0` signatures are visible in calldata on-chain. Standard v0 account contracts do not implement their own nonce protection for version-0 calls (the OS itself skips it). No privileged role, leaked key, or operator cooperation is required — the attacker only needs to have seen one valid `meta_tx_v0` signature.

---

### Recommendation

Include a nonce in `compute_meta_tx_v0_hash` by passing it as `additional_data` (as is done for `compute_l1_handler_transaction_hash`), and enforce nonce checking and incrementing for `meta_tx_v0` calls in the OS. Alternatively, replace the v0 hash scheme with a Poseidon-based versioned hash that commits to a monotonically increasing nonce, preventing all replay.

---

### Proof of Concept

1. User signs a `meta_tx_v0` authorizing a transfer of 100 STRK to address `X`. The signature is submitted to a relayer contract, which calls `execute_meta_tx_v0`. The transfer executes successfully.

2. The attacker observes the relayer's transaction on-chain and extracts `(contract_address, selector, calldata, signature)`.

3. The attacker submits a new outer v3 transaction calling the same relayer contract with the same `meta_tx_v0` parameters.

4. The OS calls `compute_meta_tx_v0_hash` with the same inputs — producing the **identical hash** — because no nonce is included:
   ```
   H(INVOKE_HASH_PREFIX, 0, contract_address, selector, H(calldata), 0, chain_id)
   ```

5. The target account's `__execute__` is invoked with the same signature and calldata. The signature verifies against the same hash. The transfer executes again.

6. Steps 3–5 repeat until the account is drained.
