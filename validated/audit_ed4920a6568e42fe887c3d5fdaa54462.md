### Title
Deploy Syscall Hardcodes `failure_flag=0`, Silently Ignoring Constructor Revert - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
In `execute_deploy`, after invoking `deploy_contract` (which runs the constructor), the OS unconditionally writes `failure_flag=0` into the syscall response header regardless of whether the constructor actually succeeded or reverted. The calling contract receives a success response even when the constructor failed, and cannot distinguish a broken deployment from a valid one.

### Finding Description
In `syscall_impls.cairo`, the `execute_deploy` function calls `deploy_contract` to run the constructor of a newly deployed contract:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns `(retdata_size, retdata)` but **no revert/failure indicator is propagated**. The `failure_flag` field in the `ResponseHeader` is hardcoded to `0` (success) unconditionally. The developer-acknowledged TODO comment confirms this is a known gap: constructor failures are not yet supported.

Compare this to how `contract_call_helper` correctly propagates the `is_reverted` flag from `select_execute_entry_point_func`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` path has no equivalent check.

### Impact Explanation
**Critical — Permanent freezing of funds / Direct loss of funds.**

A calling contract that uses the `deploy` syscall relies on `failure_flag` to decide whether the deployment succeeded. Because `failure_flag` is always `0`:

1. The calling contract believes the deployment succeeded.
2. It may transfer tokens or assets to the returned `contract_address`.
3. If the constructor actually reverted, the contract at that address is in an undefined/uninitialized state with no ability to withdraw received funds.
4. Those funds are permanently frozen at the address.

This is a direct, provable loss of user funds triggered by a normal, unprivileged `deploy` syscall.

### Likelihood Explanation
**High.** Any unprivileged contract can issue a `deploy` syscall. A constructor that reverts (due to a bug, an assertion, or deliberate design) is a common occurrence. Any contract that checks `failure_flag` to gate a subsequent fund transfer will be deceived into sending funds to a broken deployment. No special privileges, leaked keys, or operator cooperation are required.

### Recommendation
Propagate the constructor's revert status into the response header, mirroring the pattern used in `contract_call_helper`:

```diff
- // TODO(Yoni, 1/1/2026): support failures.
- assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
+ let (is_reverted, retdata_size, retdata) = deploy_contract(...);
+ assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

`deploy_contract` must be updated to return an `is_reverted` flag (analogous to `select_execute_entry_point_func`), and the `DeployResponse` retdata must reflect the constructor's actual return data or revert reason.

### Proof of Concept

1. Deploy a contract `A` whose `__execute__` calls the `deploy` syscall targeting class `B`.
2. Class `B`'s constructor unconditionally panics/reverts (e.g., `assert 1 = 0`).
3. Contract `A` reads `failure_flag` from the syscall response — it is `0` (success).
4. Contract `A` calls `transfer(deployed_address, amount)` on an ERC-20 token, believing the deployment succeeded.
5. The OS processes the block. The proof is valid because the OS never asserted the constructor succeeded.
6. The ERC-20 tokens are now at `deployed_address`, which has no functioning code to withdraw them — **permanently frozen**. [3](#0-2)
