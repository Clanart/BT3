### Title
Unauthorized Bootstrap Declare Path Allows Any Unprivileged User to Register Arbitrary Class Hash Mappings Without Signature Validation — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS contains a special "bootstrap" code path intended for sequencer-only use during system initialization. However, the guard condition relies solely on the felt value of `sender_address` equaling the literal `'BOOTSTRAP'`, with no cryptographic or privileged-role check. Any unprivileged user can craft a declare transaction satisfying all four conditions, bypassing `__validate_declare__`, nonce increment, and fee charging, and directly writing an attacker-controlled `compiled_class_hash` into the `contract_class_changes` state dictionary.

---

### Finding Description

In `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the OS checks:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

All four conditions are trivially satisfiable by any external actor:

| Condition | How attacker satisfies it |
|---|---|
| `sender_address == 'BOOTSTRAP'` | Set `sender_address` to the felt encoding of the ASCII string `"BOOTSTRAP"` — no key ownership required |
| `tx_info.nonce == 0` | Submit with nonce field = 0 |
| `tx_info.version == 3` | Set version = 3 |
| `max_possible_fee == 0` | Set all three resource bounds (`l1_gas`, `l2_gas`, `l1_data_gas`) max amounts to 0 |

When all conditions are met, the function:
1. **Skips `__validate_declare__`** — no signature verification is performed.
2. **Skips `check_and_increment_nonce`** — the nonce is never consumed, so the same nonce=0 can be reused across multiple bootstrap-path transactions in different blocks.
3. **Skips `charge_fee`** — no economic cost to the attacker.
4. **Directly writes `compiled_class_hash` into `contract_class_changes`** — the mapping `class_hash → compiled_class_hash` is committed to state. [2](#0-1) 

The normal declare path (lines 779–824) enforces all three protections. The bootstrap path silently skips all of them with no privileged-role enforcement.

The `class_hash` itself is validated to be a genuine Sierra class hash via `finalize_class_hash`: [3](#0-2) 

However, the `compiled_class_hash` field — the CASM hash that the OS will use to actually execute the contract — is **not** independently validated against the Sierra class. The attacker supplies it freely.

The `dict_update` with `prev_value=0` enforces that each `class_hash` can only be declared once: [4](#0-3) 

This means an attacker who front-runs a legitimate class declaration permanently poisons the `class_hash → compiled_class_hash` mapping for that class. The legitimate declaration will subsequently fail because `prev_value` is no longer 0.

The entry path is fully public: `execute_declare_transaction` is called from `execute_transactions_inner` for every `DECLARE`-type transaction in a block, with no prior access gate: [5](#0-4) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

An attacker front-runs a legitimate class declaration by submitting a bootstrap-path declare transaction in the same or an earlier block, mapping the target `class_hash` to an attacker-controlled `compiled_class_hash` (i.e., a malicious CASM program). Because `prev_value=0` is enforced, the legitimate declaration is permanently blocked.

Any contract subsequently deployed with that `class_hash` will execute the malicious CASM. The malicious CASM can implement arbitrary logic — including draining all token balances held by the contract or by callers — resulting in direct, irreversible loss of user funds at the protocol level.

---

### Likelihood Explanation

The attack requires only the ability to submit a standard declare transaction (a public, permissionless protocol operation). The four conditions are trivially constructable with no privileged access, no leaked key, and no operator cooperation. The attacker needs only to observe the mempool or anticipate which class hashes will be declared (e.g., by monitoring Sierra compilation pipelines or public class registries) and submit a bootstrap-path transaction first. This is a realistic, low-cost attack.

---

### Recommendation

Replace the felt-literal identity check with a cryptographically enforced privileged-address check. The `sender_address == 'BOOTSTRAP'` condition provides no security guarantee because any user can set their sender address to that felt value. Options:

1. **Remove the bootstrap path entirely** if it is no longer needed for production.
2. **Enforce a privileged address**: replace `sender_address == 'BOOTSTRAP'` with a check against a hardcoded or OS-config-supplied privileged sequencer address that is controlled by a known key, and require a valid signature even on the bootstrap path.
3. **Require a valid nonce from a real account**: do not skip `check_and_increment_nonce` and `run_validate` for any declare transaction, regardless of sender address.

```diff
- if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
+ // Bootstrap path removed; all declare transactions must pass __validate_declare__.
```

### Proof of Concept

An attacker submits a `DECLARE` transaction with the following fields:

```
tx_type          = 'DECLARE'
tx_version       = 3
sender_address   = 'BOOTSTRAP'   // felt encoding of ASCII "BOOTSTRAP"
nonce            = 0
resource_bounds  = { l1_gas: (max_amount=0, max_price=0),
                     l2_gas: (max_amount=0, max_price=0),
                     l1_data_gas: (max_amount=0, max_price=0) }
class_hash       = <target legitimate class hash>
compiled_class_hash = <attacker-controlled malicious CASM hash>
signature        = []   // empty — __validate_declare__ is never called
```

The OS evaluates the guard at line 764:
- `sender_address == 'BOOTSTRAP'` → **true**
- `tx_info.nonce == 0` → **true**
- `tx_info.version == 3` → **true**
- `compute_max_possible_fee(tx_info) == 0` → **true** (all bounds are 0)

Execution enters the bootstrap branch, calls `dict_update` with `prev_value=0, new_value=<malicious compiled_class_hash>`, and returns. No signature is checked, no nonce is consumed, no fee is paid.

The legitimate class declaration subsequently fails because `prev_value` is no longer 0. All contracts deployed with `class_hash` now execute the malicious CASM, enabling arbitrary fund theft.
