### Title
`removeCustomToken()` Permanently Freezes User ERC20 Funds via Silent `initTransfer()` Fallthrough — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`removeCustomToken()` clears the `isBridgeToken` and `customMinters` mappings for a token. After removal, any user who calls `initTransfer()` with that token address silently falls into the generic ERC20 lock branch, permanently trapping their tokens in the bridge contract with no recovery path, while NEAR still processes the cross-chain event and releases NEAR-side tokens — creating unbacked supply and irrecoverable EVM-side fund loss.

---

### Finding Description

`removeCustomToken()` performs four deletions:

```solidity
// OmniBridge.sol lines 120–127
function removeCustomToken(address tokenAddress) external onlyRole(DEFAULT_ADMIN_ROLE) {
    delete isBridgeToken[tokenAddress];
    delete nearToEthToken[ethToNearToken[tokenAddress]];
    delete ethToNearToken[tokenAddress];
    delete customMinters[tokenAddress];
}
``` [1](#0-0) 

`initTransfer()` dispatches token handling through a cascading if-else with no guard against removed tokens:

```solidity
// OmniBridge.sol lines 394–412
if (customMinters[tokenAddress] != address(0)) {
    // burn via custom minter
} else if (isBridgeToken[tokenAddress]) {
    // burn via BridgeToken
} else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount); // ← silent lock
}
``` [2](#0-1) 

After `removeCustomToken()` is called, both `customMinters[tokenAddress]` and `isBridgeToken[tokenAddress]` are `false`/`address(0)`. The `else` branch executes unconditionally: the user's ERC20 tokens are transferred into the bridge contract. Execution then continues to emit `InitTransfer`:

```solidity
// OmniBridge.sol lines 427–436
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce, amount, fee, nativeFee, recipient, message
);
``` [3](#0-2) 

The NEAR bridge observes this event and processes it as a valid transfer, releasing NEAR-side tokens to the specified recipient. The ERC20 tokens are now held by the bridge contract with no accounting entry and no recovery function anywhere in `OmniBridge.sol`. [4](#0-3) 

---

### Impact Explanation

Two simultaneous harms occur:

1. **Permanent EVM-side fund freeze**: The user's ERC20 tokens are transferred to `address(this)` (the bridge contract). There is no `rescue`, `sweep`, or token-withdrawal function in `OmniBridge.sol`. The tokens are irrecoverable.

2. **Unbacked NEAR-side supply**: Because `InitTransfer` is emitted and NEAR processes it normally, NEAR tokens are minted/released to the recipient. The EVM-side collateral backing those NEAR tokens no longer exists in any redeemable form, breaking bridge collateralization.

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows"** and **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

`removeCustomToken()` is a legitimate admin operation — it is not a malicious or hypothetical action. Once called (e.g., to delist a deprecated or compromised token), any user who:
- holds the removed custom token on EVM, and
- calls `initTransfer()` before learning of the removal (e.g., via a pending transaction, a front-end that has not yet updated, or a bot)

will permanently lose their funds. No on-chain warning or revert protects them. The `initTransfer()` call succeeds silently.

---

### Recommendation

1. Add a registry check at the top of `initTransfer()` that reverts if the token is not a recognized bridge token (i.e., neither a custom-minter token nor a `isBridgeToken` token nor a native-asset path):

```solidity
require(
    tokenAddress == address(0) ||
    customMinters[tokenAddress] != address(0) ||
    isBridgeToken[tokenAddress] ||
    multiTokens[tokenAddress].tokenAddress != address(0),
    "ERR_UNSUPPORTED_TOKEN"
);
```

2. Alternatively, disable `removeCustomToken()` for tokens that have a non-zero circulating supply (i.e., check `IERC20(tokenAddress).totalSupply() == 0` before allowing removal), mirroring the recommendation in the external report.

3. Add a token-rescue function (admin-only) to recover ERC20 tokens that were accidentally locked via the fallthrough path.

---

### Proof of Concept

1. Admin calls `addCustomToken("near-token.near", tokenAddr, minterAddr, 18)`.
   - `isBridgeToken[tokenAddr] = true`, `customMinters[tokenAddr] = minterAddr`.

2. Users bridge tokens; some hold `tokenAddr` on EVM.

3. Admin calls `removeCustomToken(tokenAddr)`.
   - `isBridgeToken[tokenAddr]` deleted → `false`.
   - `customMinters[tokenAddr]` deleted → `address(0)`.

4. User (unaware of removal) calls `initTransfer(tokenAddr, 1000e18, 0, 0, "alice.near", "")`.

5. In `initTransfer()`:
   - `customMinters[tokenAddr] == address(0)` → skip.
   - `isBridgeToken[tokenAddr] == false` → skip.
   - **`else` branch**: `IERC20(tokenAddr).safeTransferFrom(user, address(this), 1000e18)` — 1000 tokens locked in bridge.

6. `emit InitTransfer(...)` fires. NEAR relayer picks it up and releases 1000 NEAR-side tokens to `alice.near`.

7. User's 1000 ERC20 tokens are permanently locked in `OmniBridge`. No recovery function exists. Bridge collateral is now unbacked by that amount. [1](#0-0) [2](#0-1)
