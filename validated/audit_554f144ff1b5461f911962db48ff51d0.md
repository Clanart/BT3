### Title
EIP-1559 Base Fee Calculated Using Gas Limits Instead of Actual Gas Used — (`File: x/feemarket/keeper/eip1559.go`)

### Summary
`CalculateBaseFee` retrieves `parentGasUsed` via `k.GetBlockGasWanted(ctx)`, which in Cosmos SDK terminology returns the sum of gas **limits** (gas "wanted") declared by transactions, not the actual gas **consumed**. The EIP-1559 adjustment formula requires actual gas used. This causes the base fee to be driven by inflated gas-limit values, allowing any unprivileged user to manipulate the base fee upward by submitting transactions with high gas limits but negligible actual consumption.

### Finding Description

In `CalculateBaseFee`, the code comment explicitly states the intent:

> "get the block gas used and the base fee values for the parent block"

But the actual call on the very next line is:

```go
parentGasUsed := k.GetBlockGasWanted(ctx)
``` [1](#0-0) 

In Cosmos SDK, "gas wanted" is the gas limit declared in the transaction's `fee` field — the upper bound a sender is willing to spend. "Gas used" is the actual gas consumed during execution. These are distinct values. `GetBlockGasWanted` accumulates the former; the EIP-1559 spec requires the latter.

The base fee adjustment formula used in the function is:

```
if parent_gas_used > parent_gas_target:
    base_fee += delta
else:
    base_fee -= delta
``` [2](#0-1) 

When `parentGasUsed` is the sum of gas limits rather than actual consumption, the formula compares gas limits against the target. A single transaction with `gasLimit = blockGasLimit` but actual usage of 21,000 gas makes `parentGasUsed` appear equal to the full block capacity, triggering a maximum base fee increase even though the block was nearly empty.

The naming mismatch — the comment says "gas used", the call says `GetBlockGasWanted` — is the direct analog to the external report's wrong-field-path access pattern.

### Impact Explanation

This is a fee market accounting bug. The base fee is systematically over-estimated whenever transactions declare gas limits exceeding their actual consumption, which is the common case for any non-trivial transaction. An attacker can deliberately submit transactions with gas limits at or near the block gas limit but with minimal actual gas usage (e.g., a simple ETH transfer) to drive the base fee upward for all subsequent transactions. Valid user funds are mis-accounted: users pay higher fees than actual network demand warrants, and the base fee never decreases as aggressively as the EIP-1559 spec requires when actual usage is below target.

This matches the allowed High impact: **"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits … valid user funds/fees to be mis-accounted."**

### Likelihood Explanation

High. Any unprivileged user can submit a transaction with a gas limit far exceeding actual gas usage — this is a normal, unrestricted operation requiring no special privileges, governance access, or validator collusion. The effect is deterministic, cumulative across every block, and requires only standard transaction submission via the public mempool or JSON-RPC.

### Recommendation

Replace `GetBlockGasWanted` with the function that returns actual gas consumed in the block (e.g., `GetBlockGasUsed` or reading from the block gas meter's `GasConsumed()` value), consistent with the EIP-1559 specification and the code comment that describes the intent as "block gas used."

```go
// Before (wrong field):
parentGasUsed := k.GetBlockGasWanted(ctx)

// After (correct field):
parentGasUsed := k.GetBlockGasUsed(ctx)  // actual gas consumed
```

### Proof of Concept

1. Submit a transaction with `gasLimit = blockGasLimit` (e.g., 10,000,000) containing a simple ETH transfer (actual gas used ≈ 21,000).
2. Observe that `parentGasUsed` in `CalculateBaseFee` equals `blockGasLimit` (10,000,000) rather than 21,000.
3. The formula computes `parentGasUsed (10,000,000) > parentGasTarget (5,000,000)`, triggering a maximum base fee increase.
4. Actual block utilization was ~0.02%, yet the base fee increases as if the block was at 100% capacity.
5. Repeating this across multiple blocks drives the base fee to its maximum, pricing out legitimate users. [3](#0-2) [4](#0-3)

	gasLimit := big.NewInt(consParams.Block.MaxGas)
	// CONTRACT: ElasticityMultiplier cannot be 0 as it's checked in the params
	// validation
	parentGasTargetBig := new(big.Int).Div(gasLimit, new(big.Int).SetUint64(uint64(params.ElasticityMultiplier)))
	if !parentGasTargetBig.IsUint64() {
		return nil
	}

	parentGasTarget := parentGasTargetBig.Uint64()
	baseFeeChangeDenominator := new(big.Int).SetUint64(uint64(params.BaseFeeChangeDenominator))

	// If the parent gasUsed is the same as the target, the baseFee remains
	// unchanged.
	if parentGasUsed == parentGasTarget {
		return new(big.Int).Set(parentBaseFee)
	}

	if parentGasUsed > parentGasTarget {
		// If the parent block used more gas than its target, the baseFee should
		// increase.
		gasUsedDelta := new(big.Int).SetUint64(parentGasUsed - parentGasTarget)
		x := new(big.Int).Mul(parentBaseFee, gasUsedDelta)
		y := x.Div(x, parentGasTargetBig)
		baseFeeDelta := ethermint.BigMax(
			x.Div(y, baseFeeChangeDenominator),
			common.Big1,
		)

		return x.Add(parentBaseFee, baseFeeDelta)
	}

	// Otherwise if the parent block used less gas than its target, the baseFee
	// should decrease.
	gasUsedDelta := new(big.Int).SetUint64(parentGasTarget - parentGasUsed)
	x := new(big.Int).Mul(parentBaseFee, gasUsedDelta)
	y := x.Div(x, parentGasTargetBig)
	baseFeeDelta := x.Div(y, baseFeeChangeDenominator)

	// Set global min gas price as lower bound of the base fee, transactions below
	// the min gas price don't even reach the mempool.
	minGasPrice := params.MinGasPrice.TruncateInt().BigInt()
	return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
