### Title
`ft_transfer_call` Panic Leaves Transfer Permanently Finalized With Funds Frozen in Bridge — (`near/omni-bridge/src/lib.rs`)

### Summary

In the NEAR bridge contract, `process_fin_transfer_to_near` permanently marks a transfer as finalized in `finalised_transfers` **before** the token delivery to the recipient succeeds. When the delivery uses `ft_transfer_call` (i.e., the transfer carries a non-empty `msg`) and the recipient contract's `ft_on_transfer` panics, the callback `fin_transfer_send_tokens_callback` receives a failed promise result (`Err`). The helper `is_refund_required` explicitly treats a failed promise as "don't refund", so no state rollback occurs. The transfer ID remains permanently in `finalised_transfers`, the user cannot re-submit, and the returned tokens are stranded in the bridge contract with no permissionless recovery path.

### Finding Description

**Step 1 — Finalization before delivery.**
`process_fin_transfer_to_near` calls `add_fin_transfer` at line 1875, which inserts the `TransferId` into `finalised_transfers` and panics with `ERR_TRANSFER_ALREADY_FINALISED` on any future attempt to use the same ID. [1](#0-0) 

**Step 2 — Token delivery via `ft_transfer_call`.**
`send_tokens` (line 2056) selects the delivery path based on the token type and whether `msg` is non-empty. For a deployed (bridge) token with a `msg`, it calls `mint(recipient, amount, Some(msg))` on the token contract, which internally calls `ft_transfer_call` on the recipient. For a native locked token with a `msg`, it calls `ft_transfer_call` directly. [2](#0-1) 

**Step 3 — Callback ignores failed promise.**
`fin_transfer_send_tokens_callback` calls `is_refund_required(is_ft_transfer_call)`. When the promise result is `Err` (recipient's `ft_on_transfer` panicked), `is_refund_required` explicitly returns `false` with the comment "Unexpected case: don't refund". [3](#0-2) 

**Step 4 — No rollback, wrong branch taken.**
Because `is_refund_required` returned `false`, the `else` branch executes: it mints/transfers the fee to the fee recipient and emits `FinTransferEvent` — as if the transfer succeeded — while the user's principal tokens were returned to the bridge contract by NEAR's atomic `ft_transfer_call` rollback. [4](#0-3) 

`remove_fin_transfer` is never called, so the `TransferId` stays in `finalised_transfers` forever. [5](#0-4) 

### Impact Explanation

The user's principal tokens (amount minus fee) are returned to the bridge contract by NEAR's atomic rollback of the failed `ft_transfer_call`, but the transfer ID is permanently consumed. The user cannot re-submit the transfer (it would revert with `ERR_TRANSFER_ALREADY_FINALISED`). The fee is additionally minted/transferred to the relayer even though delivery failed, creating an accounting discrepancy. The only recovery path is a DAO call to `transfer_token_as_dao`, which requires trusted operator intervention and knowledge of the affected user — there is no permissionless recovery.

This matches the allowed impact: **Critical — Permanent freezing / irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

The `msg` field is user-controlled and is passed through from the origin-chain `InitTransfer` event. Any transfer that includes a non-empty `msg` (e.g., to trigger a DeFi action on the recipient contract) is vulnerable. The recipient contract's `ft_on_transfer` can panic due to:
- A bug in the recipient contract
- Intentional panic by a malicious recipient (self-inflicted griefing)
- Out-of-gas in the recipient's `ft_on_transfer` (gas forwarding is bounded by `FT_TRANSFER_CALL_GAS`)

The `msg` field is a documented, supported feature of the bridge (used for cross-chain DeFi composability), so this code path is regularly exercised in production.

### Recommendation

In `is_refund_required`, treat a failed promise (`Err`) the same as a zero-return refund when `is_ft_transfer_call` is `true`:

```rust
Err(_) => true,  // promise panicked → tokens returned → must revert finalization
```

Alternatively, restructure `fin_transfer_send_tokens_callback` to explicitly call `remove_fin_transfer` and `revert_lock_actions` whenever the promise result is `Err` and `is_ft_transfer_call` is `true`, mirroring the existing refund path. This ensures that a failed delivery always rolls back the finalization state, allowing the relayer to re-submit the transfer.

### Proof of Concept

1. On EVM, call `initTransfer` with a non-empty `message` field targeting a NEAR recipient contract `R`.
2. The relayer submits `fin_transfer` on NEAR with the corresponding proof and `storage_deposit_actions`.
3. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` inserts `TransferId{origin_chain: Eth, origin_nonce: N}` into `finalised_transfers`. [1](#0-0) 
   - `send_tokens` calls `ft_transfer_call(R, amount, msg)` (or `mint(R, amount, Some(msg))`). [6](#0-5) 
4. Contract `R`'s `ft_on_transfer` panics. NEAR atomically returns the tokens to the bridge contract.
5. `fin_transfer_send_tokens_callback` is called. `env::promise_result_checked(0, …)` returns `Err`. `is_refund_required` returns `false`. [7](#0-6) 
6. The `else` branch fires: fee is minted to the relayer, `FinTransferEvent` is emitted. `remove_fin_transfer` is never called.
7. The user's tokens sit in the bridge contract. Any re-submission of the same proof reverts with `ERR_TRANSFER_ALREADY_FINALISED`. [8](#0-7)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
