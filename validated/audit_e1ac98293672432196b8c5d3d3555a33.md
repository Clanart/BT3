### Title
Non-EVM-native (IBC/Cosmos) Token Balances Permanently Frozen After EVM Contract Self-Destruct — (`x/evm/statedb/statedb.go`)

---

### Summary

When an EVM contract self-destructs, `StateDB.Commit()` only burns the EVM-native denom balance before calling `DeleteAccount`. Non-EVM-native tokens (IBC tokens, CosmWasm bridge tokens, etc.) held by the contract address are never drained. After `DeleteAccount` removes the auth account record, those bank balances remain under an address that has no auth account and no private key, making them permanently inaccessible — a direct analog to the stETH-frozen-in-wstETH pattern.

---

### Finding Description

In `x/evm/statedb/statedb.go`, the `Commit()` function processes self-destructed accounts. It burns only the EVM denom balance and then calls `DeleteAccount`, which removes the auth account record and clears EVM storage. The code itself acknowledges the gap:

```go

// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.

`DeleteAccount` in `x/evm/keeper/statedb.go` clears EVM storage and removes the auth account via `accountKeeper.RemoveAccount`, but never touches the Cosmos bank module balances for non-EVM denoms:

```go
// clear storage
k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
    k.SetState(ctx, addr, key, nil)
    return true
})
// remove auth account
k.accountKeeper.RemoveAccount(ctx, acct)
``` [2](#0-1) 

After `RemoveAccount`, the Cosmos bank module still holds any non-EVM-native token balances under that address. Because EVM contract addresses have no private key (they are derived from deployer address + nonce or CREATE2 salt), no one can ever sign a transaction to move those tokens. They are permanently frozen.

---

### Impact Explanation

Any non-EVM-native tokens (IBC-transferred assets, CosmWasm bridge tokens, or any Cosmos SDK denom other than the EVM denom) held by an EVM contract at the time of self-destruct become permanently inaccessible. The bank module retains the balance entry, but the auth account is gone and no private key exists for a contract address. This constitutes permanent, irrecoverable loss of Cosmos bank funds through Ethermint stateDB/native action logic — matching the **High** impact category: *"EVM state transition … bug that permits … valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

The attack path is realistic and requires no privileged access:

1. **Pre-Cancun chains**: Any contract with a `SELFDESTRUCT` opcode can be triggered by its deployer or via a reentrancy/exploit path.
2. **Post-Cancun (EIP-6780) chains**: An attacker uses CREATE2 to pre-compute a deterministic contract address, sends IBC tokens to that address before deployment, then deploys and self-destructs the contract in the same transaction (EIP-6780 permits SELFDESTRUCT within the creation transaction).
3. **Legitimate DeFi contracts**: Any IBC-token-holding DeFi contract (vault, AMM, bridge) that is exploited into a SELFDESTRUCT path loses all non-EVM-native holdings permanently.

No governance, validator, or privileged role is required. A single unprivileged EVM transaction is sufficient.

---

### Recommendation

Before calling `DeleteAccount`, enumerate **all** bank module balances for the contract address across every denom and either burn them (send to module then burn) or transfer them to a designated recovery/community-pool address. A minimal fix:

```go
// Drain ALL denoms, not just evmDenom, before removing the account.
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if coin.Denom == s.evmDenom {
        continue // already handled above
    }
    if err := s.keeper.BurnOrRecoverCoin(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to drain non-evm balance on selfdestruct")
    }
}
```

Alternatively, expose a governance-controlled recovery function that can sweep orphaned bank balances from deleted contract addresses.

---

### Proof of Concept

```
1. Attacker pre-computes CREATE2 address A for a malicious vault contract.
2. Victim (or attacker) sends 1000 ATOM (IBC denom) to address A via a Cosmos bank send.
3. Attacker deploys the vault contract to address A via CREATE2.
4. In the same deployment transaction, the constructor calls SELFDESTRUCT (EIP-6780 compliant).
5. StateDB.Commit() runs:
   a. Burns EVM denom balance at A (0, since none was sent).
   b. Calls DeleteAccount(A) → removes auth account, clears EVM storage.
   c. Does NOT touch the 1000 ATOM bank balance.
6. Post-commit: bank module shows 1000 ATOM at address A.
   Auth account for A: deleted.
   Private key for A: does not exist (contract address).
7. 1000 ATOM are permanently frozen with no recovery path.
```

```go
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
