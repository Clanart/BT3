### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by a contract corresponds to a previously declared class. Any contract can replace its own class with an arbitrary undeclared felt value, permanently rendering itself non-functional and freezing all funds held within it. This is the direct structural analog to the NFT report: just as `mintNFT()` lacked an `onlyOwner` guard allowing anyone to mint, `execute_replace_class` lacks a "class must be declared" guard allowing any contract to corrupt its own class pointer irreversibly.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` (lines 878–916) reads the new class hash directly from the syscall request and writes it unconditionally into `contract_state_changes` with no check that the hash exists in the set of declared classes.

The code itself acknowledges the missing check with an explicit TODO:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The OS never cross-references `class_hash` against `contract_class_changes` (classes declared in the current block) or the global `compiled_class_facts` bundle loaded at startup. Any felt value — including one that has never been declared — is accepted and committed to state. [2](#0-1) 

Compare this to the class-declaration path in `execute_declare_transaction`, which enforces `prev_value=0` and verifies the Sierra class hash pre-image before writing to `contract_class_changes`: [3](#0-2) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

When a contract's class hash is set to an undeclared value, the OS cannot resolve any entry point for that class during future executions. The contract becomes permanently non-functional. Every subsequent call to the contract will fail at class-lookup time, and because the corrupted class hash is committed to the Merkle state root, there is no recovery path. Any ERC-20 tokens, ETH, or other assets stored in the contract's storage slots are permanently frozen. This matches the **Critical: Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The attack requires no privileged access, no leaked keys, and no trusted-role cooperation. The attacker-controlled entry path is:

1. Deploy a contract with a legitimate class hash (standard, permissionless operation).
2. Attract user deposits (e.g., present the contract as a token vault or DeFi protocol).
3. Call any function in the contract that issues a `replace_class` syscall with an arbitrary undeclared felt (e.g., `0x1`).
4. The OS executes `execute_replace_class` and commits the invalid class hash to state without validation.
5. All future calls to the contract fail; all deposited funds are permanently frozen.

The syscall is reachable from any contract execution context, triggered by any unprivileged transaction sender. The TODO comment confirms the developers are aware the check is absent.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, add a validation step in `execute_replace_class` that verifies the requested `class_hash` exists in either:

- `contract_class_changes` (classes declared within the current block), or
- The pre-loaded `compiled_class_facts` bundle in `OsGlobalContext`. [4](#0-3) 

If the hash is not found in either set, the syscall should return a failure response (analogous to how `execute_call_contract` returns `ERROR_INVALID_ARGUMENT` for forbidden selectors). [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys **Contract A** (e.g., a token vault) with a valid, declared class hash `H_valid`.
2. Users deposit funds into Contract A; assets accumulate in its storage.
3. Attacker calls a function in Contract A that issues `replace_class(0x1)`, where `0x1` is not a declared class hash.
4. The OS reaches `execute_replace_class` in `syscall_impls.cairo` lines 878–916.
5. `class_hash = 0x1` is read from the request. The TODO-guarded validation is absent; no check is performed.
6. `dict_update` writes `StateEntry(class_hash=0x1, ...)` for Contract A's address into `contract_state_changes`.
7. The block is proven and the state root is updated; Contract A's class hash is now `0x1` on-chain.
8. Every subsequent invoke targeting Contract A fails at class-lookup time — no compiled class for `0x1` exists.
9. All user funds in Contract A are permanently frozen with no recovery mechanism. [6](#0-5)
