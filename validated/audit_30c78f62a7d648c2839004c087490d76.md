### Title
Unvalidated `l1_gas_price_wei` / `l1_data_gas_price_wei` in `ProposalInit` Allows Malicious Proposer to Corrupt `eth_to_fri_rate` and `l2_gas_price_wei` Used in Block Execution - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid()` validates the FRI-denominated L1 gas prices from a proposer's `ProposalInit` against the validator's own oracle, but explicitly discards the computed WEI prices without comparing them against the proposed `l1_gas_price_wei` / `l1_data_gas_price_wei`. Those unvalidated WEI values are then consumed by `convert_to_sn_api_block_info()` to derive the `eth_to_fri_rate`, which in turn determines `l2_gas_price_wei` for every transaction in the block. A malicious proposer can set the WEI prices to any value while keeping the FRI prices within the accepted margin, silently corrupting the conversion rate and the ETH-denominated gas prices that the blockifier uses for fee accounting.

---

### Finding Description

In `is_block_info_valid()`, the validator fetches its own view of the L1 prices:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
```

The underscore-prefixed `_l1_gas_prices_wei` is computed but immediately dropped. Only the FRI prices are compared against the proposal:

```rust
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
```

`init_proposed.l1_gas_price_wei` and `init_proposed.l1_data_gas_price_wei` are never checked.

After validation passes, `initiate_validation()` calls `convert_to_sn_api_block_info(init)`, which:

1. Accepts the unvalidated WEI prices directly as `NonzeroGasPrice`:
   ```rust
   let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
   let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
   ```

2. Derives `eth_to_fri_rate` from the proposer-controlled WEI/FRI ratio:
   ```rust
   let previous_block_info = PreviousBlockInfo::from(init);
   let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
   ```
   where `calculate_eth_to_fri_rate` computes `l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei`.

3. Uses that corrupted rate to derive `l2_gas_price_wei`:
   ```rust
   let l2_gas_price_wei = NonzeroGasPrice::new(
       init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?
   )?;
   ```

4. Passes all three corrupted WEI prices into the `BlockInfo` handed to the batcher:
   ```rust
   eth_gas_prices: GasPriceVector {
       l1_gas_price: l1_gas_price_wei,
       l1_data_gas_price: l1_data_gas_price_wei,
       l2_gas_price: l2_gas_price_wei,
   },
   ```

---

### Impact Explanation

The `eth_gas_prices` vector is the authoritative ETH-denominated price used by the blockifier for every transaction that pays fees in ETH (L1 handler transactions and any account that selects ETH as the fee token). A proposer who sets `l1_gas_price_wei = 1` while keeping `l1_gas_price_fri` within the accepted margin causes:

- `eth_to_fri_rate = l1_gas_price_fri × 10^18 / 1` — astronomically large.
- `l2_gas_price_wei = l2_gas_price_fri / eth_to_fri_rate × 10^18` — rounds to zero, causing `NonzeroGasPrice::new` to return an error and abort the block, or, if the rounding lands on 1, charges near-zero ETH fees for L2 gas.
- Conversely, setting `l1_gas_price_wei` to a very large value makes `eth_to_fri_rate` near zero, causing `fri_to_wei` to overflow and abort.

In the non-aborting range, the blockifier executes the entire block with wrong ETH gas prices, producing incorrect fee charges, incorrect refunds, and incorrect balance changes for every ETH-fee-paying transaction — a direct economic impact on users and the protocol.

This maps to: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

The attack requires a node that is the elected proposer for a slot — a privileged but regularly occurring role in a decentralised validator set. Any validator that wins a proposal round can craft a `ProposalInit` with an arbitrary `l1_gas_price_wei`. The FRI prices need only stay within the oracle margin (a few percent), which is trivially achievable. No external oracle manipulation or network-level attack is required.

---

### Recommendation

In `is_block_info_valid()`, compare the proposed WEI prices against the validator's own oracle output instead of discarding `_l1_gas_prices_wei`. Apply the same `within_margin` check used for FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

// existing FRI checks ...

if !(within_margin(init_proposed.l1_gas_price_wei,
                   l1_gas_prices_wei.l1_gas_price,
                   l1_gas_price_margin_percent)
    && within_margin(init_proposed.l1_data_gas_price_wei,
                     l1_gas_prices_wei.l1_data_gas_price,
                     l1_gas_price_margin_percent))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

Alternatively, derive `l1_gas_price_wei` deterministically from the validated FRI price and the validated conversion rate inside `convert_to_sn_api_block_info`, rather than trusting the proposer-supplied value.

---

### Proof of Concept

1. A malicious proposer wins a proposal slot.
2. It constructs `ProposalInit` with:
   - `l1_gas_price_fri` = oracle value (passes `within_margin`).
   - `l1_gas_price_wei` = `1` (one wei — not validated).
   - `l1_data_gas_price_fri` = oracle value (passes `within_margin`).
   - `l1_data_gas_price_wei` = `1`.
3. `is_block_info_valid()` passes: FRI prices are within margin; WEI prices are never checked. [1](#0-0) 
4. `convert_to_sn_api_block_info()` accepts `l1_gas_price_wei = 1` and computes:
   `eth_to_fri_rate = l1_gas_price_fri × 10^18 / 1`. [2](#0-1) 
5. `l2_gas_price_wei = l2_gas_price_fri × 10^18 / eth_to_fri_rate` rounds to 0 or 1.
6. The batcher receives a `BlockInfo` with `eth_gas_prices.l2_gas_price = 1 wei`, charging near-zero ETH fees for all L2 gas in every transaction of the block. [3](#0-2) 
7. The `ProposalInit` struct confirms both WEI fields are present and proposer-controlled: [4](#0-3) 
8. The discarded `_l1_gas_prices_wei` confirms the validator never uses its own oracle WEI output for comparison: [1](#0-0)
