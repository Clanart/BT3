### Title
`deployToken` Signature Lacks Chain Binding, Enabling Cross-Chain Replay to Unauthorized Token Deployment — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` computes a message digest that contains no chain identifier (`block.chainid`, `omniBridgeChainId`, or `address(this)`). The identical Borsh-serialized message format is also used by the Solana `DeployTokenPayload`. Because the same NEAR MPC key signs for all chains, a single valid `deployToken` signature is replayable on every other EVM deployment of OmniBridge and on the Solana bridge program, enabling unauthorized token deployment on chains the NEAR bridge never authorized.

---

### Finding Description

**EVM `deployToken` message digest (no chain binding):**

```solidity
// OmniBridge.sol lines 142–149
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),  // 0x01
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The digest binds only to token metadata. It does not include `omniBridgeChainId`, EVM `block.chainid`, or `address(this)`.

**Contrast with `finTransfer`, which does include `omniBridgeChainId`:**

```solidity
// OmniBridge.sol lines 289–309
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
    Borsh.encodeUint64(payload.destinationNonce),
    bytes1(payload.originChain),
    Borsh.encodeUint64(payload.originNonce),
    bytes1(omniBridgeChainId),          // ← chain bound
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),          // ← chain bound (recipient side)
    ...
``` [2](#0-1) 

`finTransfer` is protected; `deployToken` is not.

**Solana `DeployTokenPayload` uses the byte-identical format:**

```rust
// deploy_token.rs lines 19–26
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
    IncomingMessageType::Metadata.serialize(&mut writer)?;  // 0x01
    self.serialize(&mut writer)?;  // token, name, symbol, decimals — Borsh
    ...
}
``` [3](#0-2) 

`IncomingMessageType::Metadata` is enum variant 1 (`0x01`), matching `BridgeTypes.PayloadType.Metadata` on EVM. Both hash with keccak256. Both verify against the same NEAR MPC-derived address (`nearBridgeDerivedAddress` on EVM, `derived_near_bridge_address` on Solana). [4](#0-3) 

The serialized bytes are therefore identical for the same token metadata across EVM and Solana, making signatures fully cross-replayable.

---

### Impact Explanation

An attacker who observes a valid `deployToken(signatureData, metadata)` call on chain A (e.g., Ethereum) can:

1. **Replay on any other EVM OmniBridge deployment** (Polygon, Arbitrum, Base, etc.) — the same `signatureData` passes `ECDSA.recover(...) == nearBridgeDerivedAddress` on every chain because the digest is chain-agnostic.
2. **Replay on Solana** — the Borsh-encoded payload and keccak256 hash are byte-identical; `secp256k1_recover` in `SignedPayload::verify_signature` will accept the same signature.

Consequences of a successful replay:
- A bridge token is deployed on the target chain and registered in `isBridgeToken`, `nearToEthToken`, and `ethToNearToken` (EVM) or as a wrapped mint (Solana) without the NEAR bridge having authorized that chain.
- The NEAR bridge observes the resulting `DeployToken` event and may register the token address for that chain, enabling subsequent `finTransfer` calls to mint tokens on the attacker-chosen chain.
- On EVM, if the legitimate deployment later arrives, it reverts with `ERR_TOKEN_EXIST`, permanently blocking the authorized deployment at the correct address.

This matches the allowed impact: **"High — Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment."**

---

### Likelihood Explanation

- OmniBridge is explicitly designed for multi-chain deployment (Ethereum, Polygon, Arbitrum, Solana, StarkNet, etc.).
- All `deployToken` calls are public on-chain; the signature and metadata are visible in calldata the moment the transaction is included.
- No privileged access is required — any unprivileged observer can copy `signatureData` and `metadata` from chain A's transaction and submit them to chain B.
- The attacker only needs to act before the legitimate deployment on the target chain, which is straightforward given that deployments are sequenced and observable.

---

### Recommendation

Include a chain-specific identifier in the `deployToken` message digest, mirroring the pattern already used in `finTransfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Solana (`DeployTokenPayload::serialize_for_near`):**
```rust
IncomingMessageType::Metadata.serialize(&mut writer)?;
writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;  // add chain binding
self.serialize(&mut writer)?;
```

The NEAR MPC signer must include the target chain ID when producing the signature, so that a signature for chain A is cryptographically invalid on chain B.

### Proof of Concept

1. NEAR MPC signs a `deployToken` payload for Ethereum (`omniBridgeChainId = 2`):
   ```
   keccak256( 0x01 || borsh("wrap.near") || borsh("Wrapped NEAR") || borsh("wNEAR") || 0x18 )
   ```
2. The transaction is submitted on Ethereum and succeeds. `signatureData` is visible in calldata.
3. Attacker copies `signatureData` and the `MetadataPayload` struct.
4. Attacker calls `OmniBridge.deployToken(signatureData, metadata)` on Polygon (`omniBridgeChainId = 3`).
5. The digest computed on Polygon is **identical** (no chain field), so `ECDSA.recover` returns `nearBridgeDerivedAddress` and the check passes.
6. A new `BridgeToken` proxy is deployed on Polygon; `isBridgeToken`, `nearToEthToken`, `ethToNearToken` are set.
7. The NEAR bridge observes the `DeployToken` event on Polygon and registers the token address for Polygon.
8. The legitimate Polygon deployment (when submitted) reverts with `ERR_TOKEN_EXIST`, and the NEAR bridge is now bound to the attacker-front-run token address.
9. The same `signatureData` can additionally be submitted to Solana's `deploy_token` instruction, where `SignedPayload::verify_signature` hashes the byte-identical Borsh payload with keccak256 and recovers the same signer. [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8)

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
