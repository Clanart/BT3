### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not validate that the new class hash supplied to the `replace_class` syscall corresponds to a declared contract class. An unprivileged contract deployer can exploit this to permanently freeze funds held in a contract by replacing its class hash with an arbitrary, undeclared value, rendering the contract permanently uncallable.

---

### Finding Description

The vulnerability class from the reference report is a **state-transition bypass due to a missing post-flight check**: `HatsSignerGateBase.checkAfterExecution()` does not compare the owners list before and after execution, allowing malicious actors to commit unauthorized state changes. The direct analog in this codebase is `execute_replace_class` committing a class hash state transition without verifying the new value is valid (declared).

In `syscall_impls.cairo`, the `execute_replace_class` function (lines 878–916) processes the `replace_class` syscall by writing the caller-supplied class hash directly into `contract_state_changes` with no validation:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The explicit `TODO` comment at line 898 acknowledges the missing check. The OS commits the state transition with the undeclared class hash.

In subsequent blocks, when `execute_entry_point` processes a call to the affected contract, it performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
```

For an undeclared class hash, `dict_read` returns `0` (the default for entries never written). The OS then calls:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,  // 0 — not present
);
```

`find_element` panics when the key is absent. The contract becomes permanently uncallable: every future transaction targeting it fails at the OS proving layer, and all funds held within it are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 tokens, ETH, or other assets held in a contract whose class hash has been replaced with an undeclared value are irrecoverably locked. There is no upgrade path, no recovery mechanism, and no way to withdraw funds once the class hash is set to an undeclared value and the state is committed on-chain.

---

### Likelihood Explanation

The attack is fully reachable by an unprivileged contract deployer:

1. Attacker deploys a contract `C` that accepts user deposits and exposes a `freeze()` function that internally calls `replace_class(undeclared_hash)`.
2. Users deposit funds into `C` (attracted by yield, governance, or other incentives).
3. Attacker calls `freeze()` on `C`.
4. The OS processes `replace_class(undeclared_hash)` without validation (the missing check at line 898).
5. The block is proved and the state is committed with `C`'s class hash set to `undeclared_hash`.
6. All future calls to `C` fail at the OS level; deposited funds are permanently frozen.

No privileged role, leaked key, or network-level attack is required. The attacker only needs to deploy a contract and submit transactions — both are standard unprivileged operations.

---

### Recommendation

Before writing the new class hash into `contract_state_changes` in `execute_replace_class`, verify that the supplied class hash exists in `contract_class_changes` (i.e., it has been declared). This is the pre/post-flight comparison analogous to what the reference report recommends for safe owners:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This check should be performed before the `dict_update` call, ensuring the OS enforces the invariant that a contract's class hash always refers to a declared class.

### Proof of Concept

**Step 1 — Attacker deploys malicious contract `C`:**
```
C has:
  - deposit() → accepts funds from users
  - freeze()  → calls replace_class(0xdeadbeef_undeclared)
```

**Step 2 — Users deposit funds into `C`.**

**Step 3 — Attacker calls `C.freeze()`:**
- The `replace_class` syscall is issued with `class_hash = 0xdeadbeef_undeclared`.
- `execute_replace_class` in `syscall_impls.cairo` line 906 writes `new_state_entry` with `class_hash=0xdeadbeef_undeclared` into `contract_state_changes` — **no validation performed** (line 898 TODO).
- The revert log records the old class hash (line 912), but the transaction is not reverted.
- The block is proved; state is committed.

**Step 4 — Any future call to `C` in a subsequent block:**
- `execute_entry_point` (`execute_entry_point.cairo` line 154) reads `contract_class_changes` for key `0xdeadbeef_undeclared` → returns `0`.
- `find_element` (`execute_entry_point.cairo` line 161) searches for compiled class with hash `0` → **panics / fails**.
- The transaction fails; funds are permanently inaccessible. [1](#0-0) [2](#0-1)
