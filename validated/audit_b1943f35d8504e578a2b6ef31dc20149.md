### Title
Missing Declared-Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by the caller is a declared class. This creates a state desynchronization between `contract_state_changes` (which records the new, arbitrary class hash) and `contract_class_changes` (which only contains legitimately declared classes). Any contract can set its own class hash to an arbitrary undeclared value, permanently bricking itself and freezing any funds it holds — and the OS proof system accepts the resulting state as valid.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` updates a contract's class hash in `contract_state_changes` without checking that the new class hash exists in `contract_class_changes`. The code even contains an explicit developer TODO acknowledging the missing check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

This is the direct analog of the external report's vulnerability class: in the Volt protocol, `CoreRef._vcon` is cached at initialization and never updated when `Core.vcon` changes, causing two components to hold desynchronized state. Here, `contract_state_changes` holds a class hash that does not exist in `contract_class_changes`, creating an inconsistency that the OS proof system accepts as valid because no cross-check is performed.

The `OsGlobalContext` and `compiled_class_facts_bundle` are fixed at the start of OS execution:

```cairo
let (n_compiled_class_facts, compiled_class_facts, builtin_costs) = guess_compiled_class_facts();
``` [2](#0-1) 

The post-execution validation only checks that the guessed compiled class facts are internally consistent — it does not verify that every class hash present in `contract_state_changes` corresponds to a declared entry in `contract_class_changes`:

```cairo
validate_compiled_class_facts_post_execution(
    n_compiled_class_facts=compiled_class_facts_bundle.n_compiled_class_facts,
    compiled_class_facts=compiled_class_facts_bundle.compiled_class_facts,
    builtin_costs=compiled_class_facts_bundle.builtin_costs,
);
``` [3](#0-2) 

There is therefore no mechanism — neither inline in `execute_replace_class` nor post-execution — that enforces the invariant: *every class hash stored in `contract_state_changes` must be a declared class*.

---

### Impact Explanation

When a contract calls `replace_class` with an undeclared class hash:

1. `contract_state_changes` is updated with the arbitrary hash — the OS proof is still valid because no check is performed.
2. The L1 verifier accepts the resulting state transition.
3. The contract's on-chain class hash is now an undeclared value.
4. Any future call to this contract requires the OS to locate the compiled class for that hash in `compiled_class_facts_bundle`. Because the hash was never declared, no compiled class exists for it.
5. The contract can never be successfully called again in any future block.
6. All funds held by the contract are **permanently frozen**.

This matches **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The `replace_class` syscall is reachable by any contract via the standard syscall dispatch in `execute_syscalls`:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
``` [4](#0-3) 

No privileged role is required. A realistic attack path:

1. Attacker deploys a contract that accepts user deposits (e.g., a DeFi vault).
2. Users deposit funds.
3. Attacker calls `replace_class` with an arbitrary undeclared felt (e.g., `0xdeadbeef`). The transaction succeeds; the revert log records the old class hash but is never triggered.
4. The OS writes the invalid class hash into `contract_state_changes`, generates a valid proof, and the L1 verifier accepts it.
5. All user funds are permanently frozen.

---

### Recommendation

Inside `execute_replace_class`, before updating `contract_state_changes`, add a lookup into `contract_class_changes` to verify that `class_hash` has been declared in the current block or in a prior block's state. Concretely, the OS should assert that `class_hash` maps to a non-zero compiled class hash in `contract_class_changes`, mirroring the check already enforced for `execute_declare_transaction`:

```cairo
// Proposed fix (pseudocode):
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This ensures the invariant that every class hash stored in `contract_state_changes` corresponds to a declared, compiled class — eliminating the state desynchronization.

---

### Proof of Concept

1. Attacker deploys `VaultContract` (holds user ETH/STRK).
2. Users deposit 1000 STRK into `VaultContract`.
3. Attacker sends an invoke transaction calling `replace_class(class_hash=0xdeadbeef)` from `VaultContract`.
4. OS executes `execute_replace_class`: no validation against `contract_class_changes`; `contract_state_changes[VaultContract].class_hash` is set to `0xdeadbeef`.
5. Block proof is generated and verified on L1. State root now reflects `VaultContract.class_hash = 0xdeadbeef`.
6. In the next block, any call to `VaultContract` causes the OS to search `compiled_class_facts_bundle` for `0xdeadbeef` — it is absent. The block cannot include any call to `VaultContract`.
7. 1000 STRK are permanently frozen.
