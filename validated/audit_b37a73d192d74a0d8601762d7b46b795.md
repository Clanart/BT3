### Title
EIP-3607 Sender Validation Completely Bypassed for Regular Contracts in Prague Mode - (File: `ante/eth.go`)

### Summary

In `VerifyEthAccount`, the check that rejects contract addresses as transaction senders (EIP-3607) is entirely disabled when `rules.IsPrague` is true. Unlike go-ethereum, which still enforces EIP-3607 in Prague mode but carves out an exception only for EIP-7702 delegated accounts, Ethermint skips the check unconditionally. A regular contract address (with a known private key, e.g., via CREATE2 pre-image) can send transactions in Prague mode, bypassing the EIP-3607 protection.

### Finding Description

In `ante/eth.go`, `VerifyEthAccount` guards against contract-address senders:

```go
if !rules.IsPrague {
    if acct.IsContract() {
        fromAddr := common.BytesToAddress(from)
        return errorsmod.Wrapf(errortypes.ErrInvalidType,
            "the sender is not EOA: address %s, codeHash <%s>", fromAddr, acct.CodeHash)
    }
}
``` [1](#0-0) 

When `rules.IsPrague` is `true`, the entire `IsContract()` block is skipped. This means **any** account with code — including regular deployed contracts, not just EIP-7702 delegated accounts — passes the sender check.

The correct behavior (matching go-ethereum's `stateTransition.go`) is to still enforce EIP-3607 in Prague mode, but exempt only accounts whose code is an EIP-7702 delegation pointer (`ParseDelegation` succeeds). Ethermint's condition is inverted: it should be:

```go
if acct.IsContract() {
    if rules.IsPrague {
        // Allow only EIP-7702 delegated accounts
        code := evmKeeper.GetCode(ctx, from)
        if _, ok := ethtypes.ParseDelegation(code); !ok {
            return errorsmod.Wrapf(errortypes.ErrInvalidType, ...)
        }
    } else {
        return errorsmod.Wrapf(errortypes.ErrInvalidType, ...)
    }
}
```

The analog to the Livepeer bug is exact: Livepeer's `_handleVotesOverrides` subtracts a tally without checking `delegate is transcoder`; Ethermint's `VerifyEthAccount` skips the EIP-3607 rejection without checking `account is EIP-7702 delegated`. In both cases, an action (tally subtraction / sender admission) is performed without verifying the prerequisite condition.

### Impact Explanation

An attacker who controls a private key `K` whose derived address `A` has contract code deployed (achievable via CREATE2 with a pre-computed salt) can submit valid Ethereum transactions signed with `K` from address `A` in Prague mode. The ante handler passes `VerifyEthAccount` without error, the transaction is admitted to the block, and the EVM executes it with `msg.sender = A`. This:

- Bypasses EIP-3607, allowing a contract address to act as a transaction originator.
- Allows the attacker to drain `A`'s balance, increment its nonce, and execute arbitrary EVM calls in `A`'s context.
- Constitutes an unauthorized account/code mutation path reachable via an unprivileged transaction submission.

This matches the allowed High impact: *"Ethereum transaction … signer verification bypass enabling … forged execution, or unauthorized account/code mutation."*

### Likelihood Explanation

The attack requires the attacker to:
1. Generate a private key `K` and compute address `A`.
2. Deploy a regular contract to `A` via CREATE2 (requires controlling a deployer contract and choosing the right salt — computationally feasible).
3. Fund `A`.
4. Submit a transaction signed with `K` once Prague fork is active.

This is a realistic, deliberate attack scenario. No privileged role, validator collusion, or network-level capability is required. The entry path is a standard `eth_sendRawTransaction` RPC call.

### Recommendation

Replace the blanket `!rules.IsPrague` guard with a check that still enforces EIP-3607 in Prague mode but exempts EIP-7702 delegated accounts:

```go
if acct.IsContract() {
    allowSender := false
    if rules.IsPrague {
        // EIP-7702: delegated accounts (code = 0xef0100 ++ addr) may send txs
        code := evmKeeper.GetCode(ctx, from)
        _, isDelegation := ethtypes.ParseDelegation(code)
        allowSender = isDelegation
    }
    if !allowSender {
        fromAddr := common.BytesToAddress(from)
        return errorsmod.Wrapf(errortypes.ErrInvalidType,
            "the sender is not EOA: address %s, codeHash <%s>", fromAddr, acct.CodeHash)
    }
}
```

This mirrors go-ethereum's behavior and closes the EIP-3607 bypass.

### Proof of Concept

1. Generate private key `K`; derive address `A`.
2. Deploy a regular contract (non-delegation bytecode) to `A` via CREATE2.
3. Fund `A` with enough EVM denom to cover fees.
4. Activate Prague fork on the chain.
5. Construct and sign a `MsgEthereumTx` with `From = A`, signed by `K`.
6. Submit via `eth_sendRawTransaction`.
7. Observe: `VerifyEthAccount` passes (line 93 guard skips the `IsContract` check), the transaction is included in a block, and EVM execution proceeds with `msg.sender = A` — a regular contract address — as the originator. [2](#0-1) [3](#0-2) [4](#0-3)
