### Title
Missing Chain ID in `deployToken` Signature Allows Cross-Chain Replay of Token Deployment — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` constructs a Borsh-encoded message for signature verification that contains **no chain identifier**. This makes a valid `deployToken` signature issued for one EVM chain (e.g., Ethereum mainnet) fully replayable on every other EVM chain where `OmniBridge` is deployed (Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract). By contrast, the `finTransfer` function in the same contract correctly binds its signed message to the chain via `omniBridgeChainId`. The inconsistency is the root cause.

---

### Finding Description

`deployToken` builds its signed payload as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The encoded blob contains only the payload type tag, the NEAR token ID string, name, symbol, and decimals. There is no `omniBridgeChainId`, no EVM `block.chainid`, and no contract address. The hash — and therefore the signature — is **identical across all deployed instances of `OmniBridge`** for the same NEAR token.

Compare this with `finTransfer`, which explicitly encodes `omniBridgeChainId` twice in its signed message:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

This proves the protocol design intends chain-bound signatures; `deployToken` simply omits the binding.

`deployToken` is a public, permissionless function — any caller can supply a `signatureData` and `metadata`: [3](#0-2) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` call on any one chain (e.g., Ethereum mainnet) can extract the `signatureData` and `metadata` from the transaction and replay the call on every other chain where `OmniBridge` is deployed. The result on each replayed chain:

1. A `BridgeToken` proxy is deployed with the correct metadata.
2. `nearToEthToken[metadata.token]` is permanently set to the attacker-triggered proxy address.
3. `isBridgeToken[proxy]` is set to `true`.

Because `deployToken` guards against re-deployment with `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")`, once the attacker has replayed the deployment, **no legitimate re-deployment of that token is possible on those chains**. Any subsequent `finTransfer` messages from the NEAR bridge that target those chains will mint into the attacker-triggered proxy — which, while functionally correct, was deployed without authorization from the bridge operator for those chains. This constitutes a **signature verification bypass enabling unauthorized token deployment** across all supported EVM chains.

**Severity: High** — matches the allowed impact "Proof, signature bypass enabling unauthorized token deployment."

---

### Likelihood Explanation

- `deployToken` is public and requires no privileged role.
- The attacker only needs to monitor any one chain for a `deployToken` transaction and copy the calldata to other chains.
- All target chains are live EVM networks with the same `OmniBridge` codebase deployed.
- No special knowledge, capital, or timing is required beyond observing a single on-chain transaction.

---

### Recommendation

Include `omniBridgeChainId` (and optionally `address(this)`) in the Borsh-encoded message for `deployToken`, mirroring the pattern already used in `finTransfer`:

```diff
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR bridge's signing logic must be updated in parallel to include the chain ID when producing `deployToken` signatures.

---

### Proof of Concept

1. NEAR bridge issues a `deployToken` signature for NEAR token `"usdc.near"` targeting Ethereum mainnet (`omniBridgeChainId = 0`). The signed hash is `keccak256(0x00 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)`.
2. Attacker observes the Ethereum `deployToken` transaction and extracts `signatureData` and `metadata`.
3. Attacker calls `deployToken(signatureData, metadata)` on the Arbitrum `OmniBridge` (`omniBridgeChainId = 3`). The hash computed on Arbitrum is **identical** because `omniBridgeChainId` is not part of the signed message.
4. `ECDSA.recover` returns `nearBridgeDerivedAddress` — the check passes.
5. A `BridgeToken` proxy for `"usdc.near"` is deployed on Arbitrum without NEAR bridge authorization.
6. `nearToEthToken["usdc.near"]` on Arbitrum is now permanently set; any future legitimate deployment attempt reverts with `"ERR_TOKEN_EXIST"`. [4](#0-3)
