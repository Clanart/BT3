### Title
Password Authentication Bypass in `personal_sign` and `personal_sendTransaction` — Signing Executes Without Credential Verification - (File: `rpc/namespaces/ethereum/personal/api.go`, `rpc/backend/sign_tx.go`)

---

### Summary

The `personal_sign` and `personal_sendTransaction` JSON-RPC methods accept a `password` parameter that is supposed to decrypt and authenticate the caller's right to use a key before signing. In Ethermint's implementation, both password parameters are silently discarded with blank identifiers (`_ string`). Any caller with access to the `personal` JSON-RPC namespace can sign arbitrary data or submit transactions from any account held in the node's keyring without supplying the correct password.

---

### Finding Description

The Ethereum `personal_` namespace API is designed so that signing operations require the caller to supply the password that protects the key. This is the primary access-control mechanism for the `personal_sign` and `personal_sendTransaction` methods.

In `rpc/namespaces/ethereum/personal/api.go`, both methods receive the password as a parameter but immediately discard it:

```go
// personal_sendTransaction — password is the third argument, discarded as `_`
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)
}

// personal_sign — password is the fourth argument, discarded as `_`
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)
}
```

The docstring for `Sign` explicitly states: *"The key used to calculate the signature is decrypted with the given password"* — but the password is never forwarded to the backend. [1](#0-0) 

The backend `Sign` function in `rpc/backend/sign_tx.go` calls `b.clientCtx.Keyring.SignByAddress(from, ...)` directly, with no password check whatsoever: [2](#0-1) 

Similarly, `SendTransaction` in the backend signs and broadcasts the transaction using the keyring directly, with no password verification: [3](#0-2) 

The complementary `personal_unlockAccount` method, which in Geth would be the prerequisite for signing, is also a no-op that always returns `false, nil` with a TODO comment: [4](#0-3) 

This means the entire password-based authentication layer for the `personal_` namespace is absent. The keyring is always treated as unlocked for any caller who can reach these endpoints.

---

### Impact Explanation

Any caller with network access to the `personal` JSON-RPC namespace can:

1. Call `personal_sign(data, <node_account_address>, "")` with an empty or arbitrary password and receive a valid ECDSA signature over attacker-controlled data, signed by the node's key.
2. Call `personal_sendTransaction({from: <node_account_address>, to: <attacker_address>, value: ...}, "")` with any password and have the transaction signed and broadcast, transferring funds from the node-managed account to an arbitrary destination.

This constitutes a **signer verification bypass enabling unauthorized account mutation and fund transfer** from any account whose key is held in the node's keyring.

The `personal` namespace is registered with `Public: false` in `rpc/apis.go` (line 141), which in the go-ethereum RPC framework restricts it from HTTP/WS by default. However, operators who explicitly enable it (e.g., via IPC or by including `"personal"` in the API list) are exposed to this bypass with no mitigation. [5](#0-4) 

---

### Likelihood Explanation

The `personal` namespace must be explicitly enabled by the node operator. However, many operators enable it for compatibility with Ethereum tooling (MetaMask, Hardhat, etc.). Once enabled, the attack requires only a single unauthenticated JSON-RPC call with any password value (including empty string). No special privileges, keys, or chain-level access are needed beyond reaching the RPC endpoint.

---

### Recommendation

1. **Forward the password to the keyring**: The `password` parameter in `Sign` and `SendTransaction` must be passed to the keyring's decrypt/sign operation rather than discarded. The Cosmos keyring's `ExportPrivKeyArmor` / `UnarmorDecryptPrivKey` pattern (as used in `client/export.go`) shows the correct approach for password-gated key access.
2. **Implement `UnlockAccount`**: The `personal_unlockAccount` stub should either be implemented with a time-bounded unlock mechanism or return an explicit error indicating the method is unsupported, rather than silently returning `false, nil` which misleads callers about the security state.
3. **Restrict the `personal` namespace**: Document clearly that enabling the `personal` namespace without password enforcement exposes signing to any RPC caller, and consider disabling it by default until the password check is implemented.

---

### Proof of Concept

```bash

# Attacker calls personal_sign with empty password — succeeds unconditionally:

  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0","method":"personal_sign",
    "params":["0xdeadbeef","0xABCD...1234",""],
    "id":1

# Attacker drains funds:

    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{"from":"0xABCD...1234","to":"0xAttacker","value":"0x...","gas":"0x5208"},""],
    "id":2
  }'
# Transaction is signed and broadcast without any password verification
```

```go
// SendTransaction sends transaction based on received args using Node's key to sign it
func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}
```

```go
// Sign signs the provided data using the private key of address via Geth's signature standard.
func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
```
