### Title
Finalization Record Removed on `ft_transfer_call` Rejection Enables Cross-Chain Replay / Double-Mint — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a `fin_transfer` to a NEAR recipient uses a `msg` (triggering `ft_transfer_call`), the bridge marks the transfer as finalized in `finalised_transfers` **before** the token send. If the recipient contract rejects the tokens (returns 0 from `ft_on_transfer`), `fin_transfer_send_tokens_callback` removes the finalization record via `remove_fin_transfer`. Because the bridge's only replay guard is `finalised_transfers`, the same source-chain proof can now be submitted again, causing a second minting or unlocking of tokens for a single source-chain event.

---

### Finding Description

**Step 1 — Finalization is recorded eagerly.**

`process_fin_transfer_to_near` calls `add_fin_transfer` at line 1875, inserting the `TransferId` into `finalised_transfers` before any token transfer occurs. [1](#0-0) 

**Step 2 — Tokens are sent via `ft_transfer_call`.**

When `msg` is non-empty, `send_tokens` issues an `ft_transfer_call` (or `mint` with msg for deployed tokens). The callback `fin_transfer_send_tokens_callback` is scheduled. [2](#0-1) 

**Step 3 — Callback removes the finalization record on rejection.**

`fin_transfer_send_tokens_callback` calls `is_refund_required`. When the recipient contract returns 0 (no tokens used / rejected), `is_refund_required` returns `true`. The callback then burns minted tokens, reverts lock actions, and — critically — calls `remove_fin_transfer`, which **deletes** the `TransferId` from `finalised_transfers`. [3](#0-2) [4](#0-3) 

**Step 4 — The same proof can now be re-submitted.**

`fin_transfer` verifies the proof via an external prover contract. The bridge's own replay guard is `finalised_transfers`. After `remove_fin_transfer`, `add_fin_transfer` will succeed again for the same `TransferId`, allowing the full finalization flow to execute a second time. [5](#0-4) 

---

### Impact Explanation

- **Deployed (bridged) tokens**: On the first attempt, minted tokens are burned back. On the second attempt, new tokens are minted again. The source chain locked/burned only once, so the second mint creates unbacked supply — **unauthorized mint of bridged assets**.
- **Native locked tokens**: On the first attempt, the unlock is reverted. On the second attempt, tokens are unlocked again from the bridge vault — **unauthorized release of locked assets**.

Both outcomes are Critical under the allowed impact scope.

---

### Likelihood Explanation

The attacker only needs to be the **recipient** of a cross-chain transfer that includes a `msg` field. The attacker deploys a receiver contract that rejects the first `ft_transfer_call` (returns 0 from `ft_on_transfer`). After the finalization record is removed, a trusted relayer — acting in good faith, believing the first attempt failed — re-submits the same proof. The attacker's contract then accepts the tokens on the second call. No privileged access, key compromise, or colluding MPC signers are required. Any user who can receive a bridged transfer with a message can trigger this.

---

### Recommendation

Do **not** remove the finalization record on `ft_transfer_call` rejection. Instead, keep the `TransferId` in `finalised_transfers` permanently. If a retry mechanism is needed, implement it as a separate "claim" flow that does not re-verify the source-chain proof and cannot be triggered by re-submitting the original proof. Alternatively, track a separate "pending delivery" state distinct from the finalization guard so that the replay protection is never cleared.

---

### Proof of Concept

```
1. Alice initiates a transfer from ETH → NEAR for 1000 tokens, recipient = malicious.near, msg = "trigger_call".
2. Relayer calls fin_transfer with the proof.
3. process_fin_transfer_to_near:
   - add_fin_transfer(TransferId{Eth, nonce=1}) → inserted into finalised_transfers
   - mint(malicious.near, 1000, Some("trigger_call")) is called
4. malicious.near::ft_on_transfer returns U128(0) (rejects all tokens).
5. fin_transfer_send_tokens_callback:
   - is_refund_required = true (amount returned == 0)
   - burn_tokens_if_needed(1000)  ← minted tokens burned
   - remove_fin_transfer(TransferId{Eth, nonce=1}) ← RECORD DELETED
6. Relayer (or attacker) calls fin_transfer again with the same proof.
7. Prover verifies the same ETH event again (stateless prover).
8. process_fin_transfer_to_near:
   - add_fin_transfer(TransferId{Eth, nonce=1}) → succeeds (record was deleted)
   - mint(malicious.near, 1000, Some("trigger_call")) is called
9. malicious.near::ft_on_transfer returns U128(1000) (accepts all tokens).
10. fin_transfer_send_tokens_callback: is_refund_required = false → success.
Result: 1000 tokens minted on NEAR for a single ETH lock event. Unbacked supply created.
```
