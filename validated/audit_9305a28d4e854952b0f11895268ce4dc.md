### Title
ERC777 `tokensToSend` Reentrancy Causes Nonce Collision and Permanent Token Lock in `initTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` increments `currentOriginNonce` before the external token transfer but reads `currentOriginNonce` again from storage when emitting `InitTransfer`. An ERC777 token's `tokensToSend` hook fires before the transfer completes, allowing a reentrant call to increment the nonce a second time. The outer call's emit then reads the already-incremented value, producing two `InitTransfer` events with the same nonce while nonce N+1 is never emitted. The NEAR side rejects the duplicate as a replay, permanently locking the outer call's tokens in the bridge.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer` follows this sequence:

```
currentOriginNonce += 1;                          // line 381 — nonce = N+1
...
IERC20(tokenAddress).safeTransferFrom(            // line 407 — external call
    msg.sender, address(this), amount
);
...
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress,
    currentOriginNonce,                           // line 430 — reads storage AFTER external call
    amount, fee, nativeFee, recipient, message
);
```

The nonce is incremented before the external call (correct), but the `emit` reads `currentOriginNonce` from storage **after** the external call. There is no `nonReentrant` guard on `initTransfer`.

ERC777 tokens are backward-compatible with ERC20 and can be used as native (non-bridge, non-custom-minter) tokens locked via `safeTransferFrom`. ERC777's `tokensToSend` hook fires on the sender **before** the transfer settles. An attacker who has registered a `tokensToSend` hook can reenter `initTransfer` during the hook:

**Reentrant execution trace** (starting from `currentOriginNonce = N`):

| Step | Action | `currentOriginNonce` |
|------|--------|----------------------|
| 1 | Outer `initTransfer` called | N+1 |
| 2 | Outer `safeTransferFrom` → `tokensToSend` fires | N+1 |
| 3 | Inner `initTransfer` called (reentrant) | N+2 |
| 4 | Inner `safeTransferFrom` completes, 1000 tokens locked | N+2 |
| 5 | Inner `emit InitTransfer(..., N+2, ...)` | N+2 ✓ |
| 6 | Outer `safeTransferFrom` completes, 1000 more tokens locked | N+2 |
| 7 | Outer `emit InitTransfer(..., N+2, ...)` | N+2 ✗ (should be N+1) |

**Outcome:**
- 2000 tokens are locked in the bridge
- Two `InitTransfer` events are emitted with nonce N+2
- Nonce N+1 is never emitted
- NEAR processes the first N+2 event (releases 1000 tokens)
- NEAR rejects the second N+2 event as a replay
- 1000 tokens are permanently locked with no valid event to claim them

The same flaw exists in `initTransfer1155` (line 448, 483) and propagates into `OmniBridgeWormhole`'s `initTransferExtension`, which also receives `currentOriginNonce` at call time after the external call (line 418), so the Wormhole message also carries the wrong nonce.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.** The outer call's tokens are locked in the bridge contract with no corresponding valid `InitTransfer` event on the correct nonce. The NEAR side has no mechanism to process a nonce that was never emitted, and the bridge has no emergency withdrawal path for locked native tokens. The locked funds are unrecoverable.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

- ERC777 tokens are ERC20-compatible and can be used as native tokens locked by the bridge (the `else` branch at line 406–412 applies to any token not registered as a bridge token or custom minter).
- The attacker only needs to deploy or control an ERC777 token with a registered `tokensToSend` hook — a standard ERC777 feature requiring no privileged access.
- No admin cooperation, MPC compromise, or external oracle is required.
- The entry path is fully unprivileged: any user can call `initTransfer` with any ERC20-compatible token.

---

### Recommendation

Capture the nonce in a local variable immediately after incrementing it, and use that local variable in all subsequent reads (emit, `initTransferExtension`). Additionally, add a `nonReentrant` guard to `initTransfer` and `initTransfer1155`.

```solidity
function initTransfer(...) external payable nonReentrant whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce;  // capture locally
    ...
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...
    initTransferExtension(msg.sender, tokenAddress, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
}
```

### Proof of Concept

**Setup:**
- Deploy an ERC777 token `ERC777Tok` with a `tokensToSend` hook registered for the attacker's address.
- The token is not registered as a bridge token or custom minter, so it falls into the `safeTransferFrom` path.
- `currentOriginNonce = N` initially.

**Attack:**

```solidity
// Attacker's ERC777 hook
function tokensToSend(address, address, address, uint256, bytes memory, bytes memory) external {
    if (!reentered) {
        reentered = true;
        // Reenter initTransfer — nonce becomes N+2
        OmniBridge.initTransfer(address(ERC777Tok), 1000, 0, 0, "attacker.near", "");
    }
}

// Attacker initiates
OmniBridge.initTransfer(address(ERC777Tok), 1000, 0, 0, "victim.near", "");
```

**Result:**
- Inner call emits `InitTransfer(nonce=N+2, amount=1000, recipient="attacker.near")` ✓
- Outer call emits `InitTransfer(nonce=N+2, amount=1000, recipient="victim.near")` ✗ (duplicate nonce)
- NEAR processes nonce N+2 once; the second event is rejected as replay
- 1000 tokens locked under nonce N+1 (never emitted) are permanently frozen [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
