### Title
Wrapping vs. Saturating Arithmetic Divergence in Relative Timelock Validation Enables Timelock Bypass or Consensus Split - (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

`check_time_locks` accepts a caller-controlled `nowrap: bool` parameter that switches relative timelock arithmetic between `saturating_add` (`nowrap=true`) and `wrapping_add` (`nowrap=false`). For the same spend bundle containing a near-maximum relative timelock value (e.g., `ASSERT_SECONDS_RELATIVE = u64::MAX`), the two modes produce **opposite accept/reject decisions**. With wrapping arithmetic, `ASSERT_SECONDS_RELATIVE = u64::MAX` is silently bypassed (coin spendable immediately); with saturating arithmetic, the same condition makes the coin permanently unspendable. Any node or software version that passes a different `nowrap` value than its peers will diverge on spend validity for crafted inputs.

### Finding Description

In `check_time_locks`, the relative timelock checks branch on `nowrap`:

```rust
if let Some(seconds_relative) = spend.seconds_relative {
    if nowrap {
        if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
            return Err(...AssertSecondsRelativeFailed);
        }
    } else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
        return Err(...AssertSecondsRelativeFailed);
    }
}
```

The same branching pattern applies to `height_relative`, `before_height_relative`, and `before_seconds_relative`.

The test suite explicitly documents the divergence:

| Condition | Value | `nowrap=true` (saturating) | `nowrap=false` (wrapping) |
|---|---|---|---|
| `ASSERT_SECONDS_RELATIVE` | `u64::MAX` | **Err** (coin locked forever) | **Ok** (timelock bypassed) |
| `ASSERT_HEIGHT_RELATIVE` | `u32::MAX` | **Err** (coin locked forever) | **Ok** (timelock bypassed) |
| `ASSERT_BEFORE_SECONDS_RELATIVE` | `u64::MAX` | **Ok** (no restriction) | **Err** (coin locked forever) |
| `ASSERT_BEFORE_HEIGHT_RELATIVE` | `u32::MAX` | **Ok** (no restriction) | **Err** (coin locked forever) |

The condition parsing in `conditions.rs` accepts `u64::MAX` as a valid `ASSERT_SECONDS_RELATIVE` argument (it is exactly 8 bytes, within the `sanitize_uint` limit of 8 bytes), so an attacker can craft a spend bundle with this value and submit it to the network.

### Impact Explanation

**Timelock bypass (High):** Under `nowrap=false`, a spend bundle with `ASSERT_SECONDS_RELATIVE = u64::MAX` passes validation immediately because `coin_timestamp.wrapping_add(u64::MAX) = coin_timestamp - 1`, and `current_timestamp >= coin_timestamp - 1` is always true. A coin that was intended to be locked for ~584 billion years becomes spendable at any block.

**Consensus divergence (Critical):** If the Python full node passes `nowrap=True` for some code paths and `nowrap=False` for others (e.g., during a version upgrade or due to a flag-passing bug), nodes running different versions will reach opposite conclusions about whether the same spend bundle is valid. One set of nodes accepts the block; the other rejects it. This is a deterministic chain split triggered by a valid, unprivileged spend bundle.

### Likelihood Explanation

The `nowrap` parameter is exposed directly through the Python binding `py_check_time_locks` and is caller-controlled. The Python full node (chia-blockchain) determines what value to pass. Any inconsistency in how `nowrap` is set across node versions — including during a soft/hard fork transition — produces divergence for spends with near-max relative timelock values. The crafted input (`ASSERT_SECONDS_RELATIVE = u64::MAX`) is trivially constructable by any coin owner.

### Recommendation

1. Remove the `nowrap` parameter and commit to a single arithmetic mode. Saturating arithmetic (`nowrap=true`) is the safer choice: it never silently bypasses a timelock.
2. If the wrapping-to-saturating transition is a planned consensus upgrade, gate it on a block height flag (like `ConsensusFlags`) rather than a free boolean, so all nodes switch atomically at the same height.
3. Add a parse-time rejection in `conditions.rs` for `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` values that would overflow when added to any plausible coin timestamp/height, eliminating the ambiguous input class entirely.

### Proof of Concept

**Timelock bypass with `nowrap=false`:**

A coin is created with puzzle that emits `(ASSERT_SECONDS_RELATIVE . 0xffffffffffffffff)`. The condition is parsed as `seconds_relative = Some(u64::MAX)` (valid 8-byte value, passes `sanitize_uint`). At spend time, `check_time_locks` is called with `nowrap=false`:

```
coin_timestamp = 1_000_000   (coin confirmed at Unix time 1,000,000)
current_timestamp = 1_000_001
seconds_relative = u64::MAX

wrapping_add: 1_000_000 + u64::MAX = 999_999  (wraps)
check: 1_000_001 < 999_999  → false → spend ACCEPTED
```

The coin is spendable one second after creation despite a nominal `u64::MAX`-second timelock.

**Consensus divergence:**

Node A (`nowrap=true`): `saturating_add(1_000_000, u64::MAX) = u64::MAX`. `1_000_001 < u64::MAX` → **Err** → block rejected.
Node B (`nowrap=false`): `wrapping_add(1_000_000, u64::MAX) = 999_999`. `1_000_001 < 999_999` → **Ok** → block accepted.

Both nodes see the same spend bundle. They reach opposite conclusions. Chain splits.
