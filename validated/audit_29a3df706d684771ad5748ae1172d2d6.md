### Title
Unbounded Predecessor Execution Loop in `TraceTx` Enables Resource Exhaustion via Public gRPC - (File: x/evm/keeper/grpc_query.go)

### Summary
`TraceTx` iterates over a caller-supplied `req.Predecessors` slice with no size cap, executing each predecessor as a full EVM transaction via `ApplyMessageWithConfig`. An unprivileged caller can submit a single gRPC/JSON-RPC request containing thousands of max-gas predecessor transactions, exhausting CPU and memory on the node.

### Finding Description
In `TraceTx`, the predecessor replay loop has no upper-bound guard:

```go
for i, tx := range req.Predecessors {
    ...
    rsp, err := k.ApplyMessageWithConfig(ctx, msg, cfg, true)  // commit=true
    ...
}
```

`req.Predecessors` is a `[]*MsgEthereumTx` field in the protobuf request, entirely attacker-controlled. There is no validation of `len(req.Predecessors)` before the loop begins. Each iteration performs a full EVM state transition (including storage writes, code execution, and stateDB commit) with `commit=true`.

The analogous `TraceBlock` loop over `req.Txs` has the same structure:

```go
for i, tx := range req.Txs {
    ...
    traceResult, logIndex, err := k.prepareTrace(ctx, cfg, msg, req.TraceConfig, true)
    ...
}
```

Again, `len(req.Txs)` is never bounded before iteration.

No `MaxPredecessors` or equivalent constant exists anywhere in the codebase.

### Impact Explanation
Each predecessor invokes `ApplyMessageWithConfig` — a full EVM execution including stateDB reads/writes, gas accounting, and potential contract deployment. An attacker submitting N predecessors each at the block gas limit forces N sequential full-block-equivalent EVM executions in a single RPC call. This causes:

- **OOM crash** on nodes with large N (e.g., 10,000 predecessors × max-gas contract deployments)
- **CPU starvation** blocking all other RPC and consensus processing on the same process

If the validator process also serves the gRPC/JSON-RPC endpoint (a common deployment pattern), a single malicious request can crash or stall the validator, causing it to miss blocks and potentially halting the chain if enough validators are targeted. This maps to the Critical impact: "block-processing path can halt the chain."

### Likelihood Explanation
The `debug_traceTransaction` JSON-RPC method and the `QueryTraceTx` gRPC method are publicly accessible on any node that enables the JSON-RPC or gRPC server. No authentication is required. The attacker only needs to craft a `QueryTraceTxRequest` with a large `Predecessors` array containing high-gas transactions (e.g., contract deployments at the block gas limit). This is trivially constructable with any gRPC client or via the `debug_traceTransaction` JSON-RPC wrapper.

### Recommendation
Add a hard cap on `len(req.Predecessors)` and `len(req.Txs)` before entering the loop. For example:

```go
const maxTracePredecessors = 1000
if len(req.Predecessors) > maxTracePredecessors {
    return nil, status.Errorf(codes.InvalidArgument,
        "too many predecessors: %d > %d", len(req.Predecessors), maxTracePredecessors)
}
```

Apply the same guard in `TraceBlock` for `req.Txs`. Additionally, consider enforcing a cumulative gas budget across all predecessor executions so that even within the limit, total work is bounded.

### Proof of Concept

1. Build a `QueryTraceTxRequest` with `Predecessors` set to 5,000 `MsgEthereumTx` entries, each being a contract-creation transaction at the block gas limit (e.g., 30M gas).
2. Submit via gRPC to the `eth.evm.v1.Query/TraceTx` endpoint on a live node.
3. The node enters the loop at line 542, executing `ApplyMessageWithConfig` 5,000 times with `commit=true`.
4. Each call allocates stateDB objects, executes EVM bytecode, and writes storage — the node's memory and CPU spike until OOM or timeout.
5. If the node is a validator, it stops producing/signing blocks during this period.

func (k Keeper) TraceBlock(c context.Context, req *types.QueryTraceBlockRequest) (*types.QueryTraceBlockResponse, error) {
	if req == nil {
		return nil, status.Error(codes.InvalidArgument, "empty request")
	}

	if req.TraceConfig != nil && req.TraceConfig.Limit < 0 {
		return nil, status.Errorf(codes.InvalidArgument, "output limit cannot be negative, got %d", req.TraceConfig.Limit)
	}

	// get the context of block beginning
	contextHeight := req.BlockNumber
	if contextHeight < 1 {
		// 0 is a special value in `ContextWithHeight`
		contextHeight = 1
	}

	ctx := sdk.UnwrapSDKContext(c)
	ctx = ctx.WithBlockHeight(contextHeight)
	ctx = ctx.WithBlockTime(req.BlockTime)
	ctx = ctx.WithHeaderHash(common.Hex2Bytes(req.BlockHash))
	ctx = ctx.WithProposer(GetProposerAddress(ctx, req.ProposerAddress))
	chainID, err := getChainID(ctx, req.ChainId)
	if err != nil {
		return nil, status.Error(codes.InvalidArgument, err.Error())
	}

	cfg, err := k.EVMConfig(ctx, chainID, common.Hash{})
	if err != nil {
		return nil, status.Error(codes.Internal, "failed to load evm config")
	}
	cfg.TraceReplay = req.TraceConfig.GetTraceReplay()
	signer := ethtypes.MakeSigner(cfg.ChainConfig, big.NewInt(ctx.BlockHeight()), uint64(ctx.BlockTime().Unix())) //#nosec G115
	txsLength := len(req.Txs)
	results := make([]*types.TxTraceResult, 0, txsLength)

	for i, tx := range req.Txs {
		result := types.TxTraceResult{}
		ethTx := tx.AsTransaction()
		cfg.TxConfig.TxHash = ethTx.Hash()
		result.TxHash = ethTx.Hash()
		cfg.TxConfig.TxIndex, err = ethermint.SafeUint(i)
		if err != nil {
			return nil, err
		}
		msg, err := core.TransactionToMessage(ethTx, signer, cfg.BaseFee)
		if err != nil {
			result.Error = status.Error(codes.Internal, err.Error()).Error()
		} else {
			traceResult, logIndex, err := k.prepareTrace(ctx, cfg, msg, req.TraceConfig, true)
			if err != nil {
				result.Error = err.Error()
			} else {
				cfg.TxConfig.LogIndex = logIndex
				result.Result = traceResult
			}
		}
		results = append(results, &result)
	}
