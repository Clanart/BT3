### Title
Missing Duplicate-Key/Hash Validation in `batch_insert()` for Items Beyond the Bootstrap Phase — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert()` applies the full uniqueness guard (`KeyAlreadyPresent`, `HashAlreadyPresent`) only to the first one or two items that bootstrap the tree. All remaining items in the batch are written directly to the blob via `insert_entry_to_blob()` with no duplicate-key or duplicate-hash check, allowing an attacker to silently insert multiple leaf nodes that share the same `KeyId` or `Hash`. The resulting tree is structurally corrupt: the `block_status_cache` can only track one index per key, so the second (and any further) duplicate leaf is invisible to proof generation, yet its hash is folded into the Merkle root. This lets an adversary forge inclusion or exclusion proofs against a root that was computed from a tree the verifier cannot reconstruct.

### Finding Description

`MerkleBlob::insert()` enforces two invariants before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` calls `insert()` (with those guards) only for the first one or two items when the tree has ≤ 1 existing leaf:

```rust
// lines 578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else {
            return Ok(());
        };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← guarded
    }
}
``` [2](#0-1) 

Every subsequent item in the same call goes through a bare `insert_entry_to_blob()` with **no** duplicate check:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        node: Node::Leaf(LeafNode { hash, key, value, .. }),
        ..
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // ← no guard
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

The Python binding `py_batch_insert` exposes this path directly to callers:

```python
merkle_blob.batch_insert(keys_values, hashes)
``` [4](#0-3) 

Because `block_status_cache` is a `HashMap<KeyId, TreeIndex>`, inserting a second leaf with the same `KeyId` silently overwrites the cache entry for that key. The first duplicate leaf node remains in the blob but is now unreachable through the cache. The Merkle root is computed from both leaves (their hashes are folded into internal nodes), but `get_proof_of_inclusion()` can only generate a path to the one the cache knows about. The root therefore commits to data that no valid proof can cover, enabling forged inclusion/exclusion claims.

`check_just_integrity()` would detect this after the fact:

```rust
let cached_index = self.block_status_cache.get_index_by_key(node.key)
    .ok_or(Error::IntegrityKeyNotInCache(node.key))?;
if *cached_index != index {
    return Err(Error::IntegrityKeyToIndexCacheIndex(...));
}
``` [5](#0-4) 

However, `check_integrity()` is not called automatically on every `batch_insert()`; it is an opt-in diagnostic. A caller that omits it will silently operate on a corrupt tree.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A corrupt tree root is published (e.g., committed on-chain as a DataLayer store root). A verifier who trusts that root and calls `confirm_included_already_hashed` or `confirm_not_included_already_hashed` against it receives answers that are inconsistent with the actual key-value mapping, because the root was computed from a blob containing a ghost leaf that no proof path can reach. [6](#0-5) 

### Likelihood Explanation

The Python binding is the primary public surface. Any Python caller that constructs a `batch_insert` list with a repeated `KeyId` (or repeated `Hash`) triggers the bug. The first two items are safe; the third onward is not. Because the function returns `Ok(())` with no error, the caller has no signal that the tree is corrupt. Likelihood is **medium**: it requires a caller to supply duplicate keys in a batch, which may happen accidentally (e.g., upsert-style logic that does not pre-deduplicate) or deliberately.

### Recommendation

Apply the same duplicate-key and duplicate-hash guards inside the bulk loop of `batch_insert()`, either by:

1. Calling `self.insert()` for every item (simplest, but loses the performance optimisation), or
2. Checking `self.block_status_cache.contains_key(key)` and `self.block_status_cache.contains_leaf_hash(&hash)` at the top of the `for ((key, value), hash) in keys_values_hashes` loop and returning `Err(Error::KeyAlreadyPresent())` / `Err(Error::HashAlreadyPresent())` on violation.

Additionally, consider calling `check_integrity()` (or a lightweight variant) at the end of `batch_insert()` in debug builds to catch regressions early.

### Proof of Concept

```python
from chia_rs import MerkleBlob  # wheel/python/chia_rs/datalayer.pyi

blob = MerkleBlob(bytearray())

# Seed the tree with 2 items so the bootstrap path is taken for them
seed_kvs  = [(1, 1), (2, 2)]
seed_hash = [b'\x01'*32, b'\x02'*32]
blob.batch_insert(seed_kvs, seed_hash)

# Now batch-insert 3 more items where key 10 appears TWICE.
# Items beyond the first two skip the KeyAlreadyPresent guard.
dup_kvs   = [(10, 100), (11, 110), (10, 999)]   # key 10 duplicated
dup_hashes = [b'\x0a'*32, b'\x0b'*32, b'\xff'*32]
blob.batch_insert(dup_kvs, dup_hashes)           # returns Ok — no error raised

blob.calculate_lazy_hashes()
root = blob.get_root_hash()

# The root commits to 5 leaf nodes, but the cache only knows 4 keys.
# Proof for key 10 covers only one of the two leaves; the other is a ghost
# that contributes to the root but is unprovable.
proof = blob.get_proof_of_inclusion(10)
print("proof valid:", proof.valid())   # True — but root is corrupt
# blob.check_integrity() would raise IntegrityKeyToIndexCacheIndex here
```
