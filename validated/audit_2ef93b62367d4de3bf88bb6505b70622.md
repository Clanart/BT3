### Title
`StateDB.Commit()` Does Not Drain Non-EVM-Native Token Balances on SELFDESTRUCT, Enabling Orphaned-Balance Recovery via CREATE2 Address Reuse - (File: x/evm/statedb/statedb.go)

### Summary

When a contract self-destructs, `StateDB.Commit()` only burns the EVM-denom (native) balance. Any non-EVM-native tokens (IBC, CosmWasm-bridge, or custom bank-module coins) held by the destroyed address are explicitly left as orphaned bank balances. Because Cosmos SDK bank balances are keyed by address rather than by account object, those tokens survive the `accountKeeper.RemoveAccount()` call. An attacker who controls the CREATE2 salt can redeploy a contract at the identical address in a subsequent transaction, and the new account inherits the orphaned bank balance. On chains that expose a bank precompile (or any native-action path callable from EVM), the attacker can then drain those tokens.

---

### Finding Description

`Commit()` in `x/evm/statedb/statedb.go` handles self-destructed objects as follows:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
``` [1](#0-0) 

`DeleteAccount()` in the keeper removes the auth account and EVM storage, but its own doc-comment explicitly states the balance is the caller's responsibility:

```go

``` [2](#0-1) 

The caller (`Commit()`) only satisfies that contract for the single EVM denom. Every other coin denomination held by the contract address in the Cosmos bank module is silently skipped. After `RemoveAccount` executes, the auth record is gone but the bank-module entries for those coins remain, keyed to the same raw address bytes.

---

### Impact Explanation

Cosmos SDK bank balances survive account deletion because they are stored under the raw address key, independent of the auth-module account object. When a new account is created at the same address (via CREATE2 redeploy), the bank module immediately reports those orphaned balances as belonging to the new account. On any Ethermint-based chain that exposes a bank precompile or `ExecuteNativeAction` path callable from EVM (the codebase documents exactly such a precompile pattern), the redeployed contract can call `transfer` / `SendCoins` to drain the orphaned coins to an attacker-controlled address. This constitutes unauthorized theft of Cosmos bank funds through Ethermint stateDB/native-action logic, matching the **Critical/High** allowed impact tier.

---

### Likelihood Explanation

The preconditions are realistic on any IBC-enabled Ethermint chain:

1. A contract receives IBC or other non-EVM-native tokens (common in DeFi bridges and vaults).
2. The contract was deployed via CREATE2 (standard factory pattern), so the attacker knows the salt.
3. The attacker triggers SELFDESTRUCT (either as the contract owner or by exploiting a vulnerability in the contract itself).
4. The attacker redeploys at the same CREATE2 address in the next transaction.
5. The chain exposes a bank precompile or native-action hook (documented as a supported extension in this codebase).

Steps 1–4 are entirely attacker-controlled once the contract is deployed. Step 5 is a chain-configuration dependency, making this **Medium** likelihood overall.

---

### Recommendation

Before calling `DeleteAccount`, enumerate and burn **all** bank-module balances held by the destroyed address, not only the EVM denom. The safest approach is to iterate `bankKeeper.GetAllBalances(ctx, cosmosAddr)` and burn each coin:

```go
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-EVM balance")
    }
}
```

This mirrors the invariant enforced for the EVM denom and closes the orphaned-balance window for all coin types.

---

### Proof of Concept

1. **Deploy factory** via a regular CREATE tx. Factory deploys child contracts via CREATE2 with a known salt.
2. **Send IBC tokens** (e.g., `ibc/...` denom) to the predicted child address before deployment (or after, via a bridge transfer).
3. **Deploy child** via CREATE2 from the factory. Child receives the IBC tokens.
4. **Trigger SELFDESTRUCT** on the child (e.g., call `child.destroy()`). `StateDB.Commit()` burns only the EVM-denom balance; IBC tokens remain in the bank module at the child's address.
5. **Redeploy child** at the same CREATE2 address. The new account object is created; `bankKeeper.GetAllBalances` for that address returns the orphaned IBC tokens.
6. **Call bank precompile** (or `ExecuteNativeAction`) from the redeployed child to `SendCoins` the IBC tokens to the attacker's EOA.

The existing integration-test contract `SelfDestructExploitFactory` already demonstrates steps 3–5 for the EVM denom; the same pattern applies to non-EVM-native tokens once the EVM-denom burn fix is in place. [3](#0-2) [4](#0-3)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account

// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)
