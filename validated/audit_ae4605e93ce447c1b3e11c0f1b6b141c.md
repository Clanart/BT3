### Title
Hardcoded `L1_HANDLER_L2_GAS_MAX_AMOUNT` Gas Limit Permanently Freezes L1→L2 Messages for Gas-Intensive Handlers - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` assigns a hardcoded gas budget of `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000` to every L1 handler execution, with no mechanism for the L1 message sender to specify a higher limit. Unlike account transactions (invoke, deploy_account, declare), which derive their gas budget from user-supplied `resource_bounds`, L1 handlers always receive exactly this fixed amount. If a legitimate L1 handler requires more gas than this constant, the sequencer can never include the transaction without making the block unprovable, permanently freezing any L1 funds associated with the pending message.

---

### Finding Description

In `constants.cairo`, the L1 handler gas cap is a compile-time constant:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;
``` [1](#0-0) 

In `transaction_impls.cairo`, `execute_l1_handler_transaction` unconditionally sets `remaining_gas` to this constant before calling the handler:

```cairo
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [2](#0-1) 

The `TxInfo` written for L1 handlers explicitly sets `resource_bounds_start` and `resource_bounds_end` to null, confirming there is no user-supplied gas bound:

```cairo
resource_bounds_start=cast(0, ResourceBounds*),
resource_bounds_end=cast(0, ResourceBounds*),
``` [3](#0-2) 

By contrast, account transactions call `get_initial_user_gas_bound`, which reads the user-specified `max_amount` from `resource_bounds[L2_GAS_INDEX]`:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [4](#0-3) 

`non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` after execution:

```cairo
assert is_reverted = 0;
``` [5](#0-4) 

If the handler runs out of gas, `is_reverted = 1`, this assertion fails, and the entire block proof is invalid. The sequencer therefore **cannot** include any L1 handler that exceeds the gas cap — it must leave the message pending indefinitely.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

L1→L2 messages are sent by locking assets on L1 (e.g., ETH or ERC-20 tokens in a bridge contract). If the corresponding L2 handler always exceeds `L1_HANDLER_L2_GAS_MAX_AMOUNT`, the sequencer can never include the transaction without making the block unprovable. The message remains in the pending queue forever, and the locked L1 assets are permanently inaccessible. This is structurally identical to H-9: a hardcoded parameter that is appropriate for simple cases but causes permanent failure for complex ones.

---

### Likelihood Explanation

`L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000` gas. With `STEP_GAS_COST = 100`, this equals 1 million VM steps. [6](#0-5) 

A storage write costs `STORAGE_WRITE_GAS_COST = 44_970` gas, so roughly 2,222 storage writes exhaust the budget. [7](#0-6) 

A complex bridge or DeFi L1 handler that iterates over user positions, updates multiple storage slots, or performs cryptographic operations (e.g., Keccak at `KECCAK_BUILTIN_GAS_COST = 136_189` per invocation) can realistically exceed this limit. [8](#0-7) 

Additionally, state growth over time (more storage slots, larger tries) increases per-operation gas costs, meaning a handler that fits within the limit at deployment may exceed it later — exactly the "liquidity dries up after deposit" scenario from H-9.

---

### Recommendation

Mirror the account-transaction pattern: allow the L1 message sender (or the L1 contract) to specify a gas limit that is included in the L1→L2 message payload and verified on-chain. The OS should read this value and use it as `remaining_gas` instead of the hardcoded constant. A minimum and maximum bound can still be enforced to prevent abuse, but the value should not be a single compile-time constant applied uniformly to all handlers.

---

### Proof of Concept

1. Deploy an L2 contract whose `@l1_handler` function performs ~2,300 storage writes (or equivalent gas-intensive operations), requiring slightly more than `100_000_000` gas.
2. From L1, call the StarkNet core contract to send an L1→L2 message targeting that handler (e.g., a bridge deposit locking 10 ETH on L1).
3. The sequencer attempts to execute the handler. The execution exhausts `L1_HANDLER_L2_GAS_MAX_AMOUNT` gas, causing `is_reverted = 1`.
4. `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` — the assertion fails, making the block unprovable if the transaction is included.
5. The sequencer is forced to exclude the transaction permanently. The L1→L2 message stays pending forever.
6. The 10 ETH locked on L1 is permanently inaccessible — permanent freezing of funds.
