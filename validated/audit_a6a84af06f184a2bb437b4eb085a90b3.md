### Title
Fake Block Hash Stored for Internally-Produced Blocks in P2P Sync Client — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

When the P2P sync client converts an internally-produced `SyncBlock` (from the batcher/consensus path) into a `SignedBlockHeader` for persistent storage, it substitutes the real block hash with a synthetic placeholder derived from the block number (`block_number.0` cast to a `StarkHash`), and stores a zero default signature. This fake header is then written to the header table and served authoritatively to RPC callers and P2P peers. The pattern is a direct analog of C-06: instead of using the authenticated/computed value (the real block hash), the system silently substitutes an unauthenticated, predictable value from the input data.

---

### Finding Description

`HeaderStreamBuilder::convert_sync_block_to_block_data` is the function responsible for converting a `SyncBlock` received from the batcher/consensus into a `SignedBlockHeader` that is written to storage. The function is invoked from `BlockDataStreamBuilder::get_internal_block_at` whenever the P2P sync client detects that the batcher has already produced the block internally (i.e., the node is a proposer or validator in the current consensus round).

At line 144 of `header.rs`, the `block_hash` field is set to:

```rust
block_hash: BlockHash(StarkHash::from(block_number.0)),
```

This is not the actual Poseidon block hash computed from the block's commitments, gas prices, state root, and parent hash (as defined in `calculate_block_hash` in `block_hash.cairo`). It is simply the block number integer reinterpreted as a field element. Additionally, the consensus signature is set to:

```rust
signatures: vec![BlockSignature::default()],
```

which is a zero signature.

The resulting `SignedBlockHeader` is then written to the MDBX header table via `write_to_storage` → `append_header` + `append_block_signature`. This is the same table that the RPC layer reads from when answering `starknet_getBlockWithTxHashes`, `starknet_getBlockHashAndNumber`, and related calls, and the same table that the P2P sync server reads from when serving `SignedBlockHeader` responses to syncing peers.

The `SyncBlock` type itself is explicitly documented as trusted and unverified:

> "Blocks that came from the state sync are trusted. Therefore, SyncBlock doesn't contain data needed for verifying the block."

The `block_hash` field is never present in `SyncBlock`; it is synthesized inside `convert_sync_block_to_block_data`. There is no check that the synthesized value matches any externally-computed or consensus-agreed hash. The two open TODOs in the same function confirm that additional fields (`n_transactions`, `state_diff_length`) are also unverified against `concatenated_counts`.

---

### Impact Explanation

**Wrong block hash stored and served (High — RPC returns authoritative-looking wrong value):**

Every sequencer node that participates in consensus will store `block_hash = block_number` for every block it produces or validates via the internal path. The RPC layer reads this field directly from the header table. Any client calling `starknet_getBlockWithTxHashes`, `starknet_getBlockHashAndNumber`, or `starknet_getBlockWithReceipts` for a block produced via the internal path will receive a block hash of `0x<block_number>` instead of the real Poseidon hash. This is an authoritative-looking wrong value returned by the RPC.

**Wrong block hash propagated to syncing peers (High — P2P sync propagates wrong value):**

The P2P sync server's `FetchBlockData for SignedBlockHeader` reads the stored header (including the fake `block_hash`) and sends it to syncing peers. A peer that receives this `SignedBlockHeader` will store the fake hash as the authoritative block hash for that block number. The fake hash then propagates transitively to all nodes that sync from the affected node.

**Zero block signature stored:**

`BlockSignature::default()` (all-zero) is stored for every internally-produced block. Any downstream consumer that verifies the stored block signature against the stored block hash will be operating on a (fake hash, zero signature) pair, which is meaningless for security purposes.

---

### Likelihood Explanation

This code path is exercised on every sequencer node that participates in consensus. The `get_internal_block_at` function is called unconditionally whenever the batcher has already produced the block before the P2P sync client requests it. In a live network with multiple sequencer nodes, every proposer node will trigger this path for every block it proposes. The likelihood is therefore **certain** for any deployment where the P2P sync client and the batcher run together (the standard topology).

---

### Recommendation

The `SyncBlock` type should be extended to carry the actual computed block hash (or the full set of inputs needed to compute it). `convert_sync_block_to_block_data` must compute or verify the real block hash before constructing the `SignedBlockHeader`. Until `SyncBlock` carries the real hash, the function should not write a fake hash to storage; it should either defer header storage until the real hash is available or omit the `block_hash` field and mark the header as pending.

The open TODO at line 140–141 (`// TODO(Shahak): Verify n_transactions and state_diff_length match values in concatenated_counts`) should also be resolved to prevent additional field mismatches.

---

### Proof of Concept

1. A sequencer node proposes block `N` via consensus. The batcher produces a `SyncBlock` with `block_header_without_hash.block_number = N` and valid `block_header_commitments`.

2. The P2P sync client receives this `SyncBlock` via the internal channel. `get_internal_block_at` calls `HeaderStreamBuilder::convert_sync_block_to_block_data(N, sync_block)`.

3. Inside `convert_sync_block_to_block_data`, line 144 executes:
   ```rust
   block_hash: BlockHash(StarkHash::from(block_number.0)),
   // = BlockHash(StarkHash::from(N))  ← fake, not the Poseidon block hash
   ```
   and line 155:
   ```rust
   signatures: vec![BlockSignature::default()],  // ← zero signature
   ```

4. `write_to_storage` calls `append_header(N, &fake_header)` and `append_block_signature(N, &zero_sig)`, persisting both to the MDBX header table.

5. An RPC client calls `starknet_getBlockHashAndNumber`. The RPC layer reads `header.block_hash` from the header table and returns `0x000...00N` (the block number as a felt) instead of the real Poseidon hash.

6. A syncing peer queries the P2P sync server for block `N`'s header. The server reads the stored `SignedBlockHeader` (fake hash + zero signature) and sends it. func calculate_block_hash{poseidon_ptr: PoseidonBuiltin*}(
    block_info: BlockInfo*,
    header_commitments: BlockHeaderCommitments*,
    gas_prices_hash: felt,
    state_root: felt,
    previous_block_hash: felt,
    starknet_version: felt,
) -> felt {
    static_assert BlockInfo.SIZE == 3;
    static_assert BlockHeaderCommitments.SIZE == 5;

    let hash_state = hash_init();
    with hash_state {
        hash_update_single(BLOCK_HASH_VERSION);
        hash_update_single(block_info.block_number);
        hash_update_single(state_root);
        hash_update_single(block_info.sequencer_address);
        hash_update_single(block_info.block_timestamp);
        hash_update_single(header_commitments.packed_lengths);
        hash_update_single(header_commitments.state_diff_commitment);
        hash_update_single(header_commitments.transaction_commitment);
        hash_update_single(header_commitments.event_commitment);
        hash_update_single(header_commitments.receipt_commitment);
        hash_update_single(gas_prices_hash);
        hash_update_single(starknet_version);
        hash_update_single(0);
        hash_update_single(previous_block_hash);
    }

    let block_hash = hash_finalize(hash_state=hash_state);
    return block_hash;
}
