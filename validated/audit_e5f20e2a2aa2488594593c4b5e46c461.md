### Title
Integer Wrapping in `check_time_locks` Relative Timelock Arithmetic Enables Timelock Bypass - (File: crates/chia-consensus/src/check_time_locks.rs)

### Summary

When `check_time_locks` is called with `nowrap=false`, the relative timelock checks for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` use `wrapping_add` instead of `saturating_add`. An attacker-controlled CLVM program can emit a condition with a large `height_relative` (up to `u32::MAX`) or `seconds_relative` (up to `u64::MAX`) value that wraps the addition around to a small integer, causing the timelock guard to pass trivially regardless of the actual block height or timestamp.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` boolean for all four relative timelock conditions:

```rust
// ASSERT_HEIGHT_RELATIVE (nowrap=false path)
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}

// ASSERT_SECONDS_RELATIVE (nowrap=false path)
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

When `nowrap=false`, `wrapping_add` is used. If an attacker supplies `height_relative = u32::MAX - confirmed_block_index + 1`, the sum wraps to `0`. The guard then evaluates `prev_transaction_block_height < 0`, which is always `false` for any `u32`, so no error is returned and the timelock is bypassed. The same arithmetic applies to `seconds_relative` with `u64` wrapping.

The `height_relative` and `seconds_relative` values originate from CLVM condition output, parsed by `sanitize_uint` which accepts the full `u32`/`u64` range:

The `check_time_locks` Python binding exposes `nowrap` as a caller-controlled parameter: [3](#0-2) 

The `nowrap=false` wrapping behavior is explicitly documented in the test suite, confirming it is a reachable production code path:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- nowrap=false: timelock passes when it should fail
)]
``` [4](#0-3) 

The same wrapping bypass applies to `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE` in the opposite direction: wrapping to a small value causes the `>=` guard to fire spuriously, permanently locking coins that should be spendable. [5](#0-4) 

### Impact Explanation

This is a **High** impact finding matching the allowed scope: *"condition, timelock, or coin-id validation bypass enables unauthorized spend acceptance."*

When `nowrap=false` is active, any unprivileged spender can craft a CLVM puzzle or solution that emits `ASSERT_HEIGHT_RELATIVE(u32::MAX)` or `ASSERT_SECONDS_RELATIVE(u64::MAX)`. The wrapping arithmetic reduces the effective deadline to a value already in the past, so the spend is accepted at any block height or timestamp. Coins protected by relative timelocks (vesting contracts, HTLCs, time-locked custody) can be drained immediately.

### Likelihood Explanation

The `nowrap` parameter is passed from the Python full node layer. The function explicitly supports `nowrap=false` as a production mode (not test-only), and the Python binding exposes it as a first-class parameter. The attacker entry path is fully unprivileged: submit a spend bundle whose CLVM program outputs a condition with a near-maximal `height_relative` or `seconds_relative` value. No special keys, governance access, or network position is required. Exploitability is conditional on the full node passing `nowrap=false` for the block being validated (e.g., before a soft-fork activation height).

### Recommendation

1. **Remove the `nowrap=false` wrapping path entirely** once the soft-fork that mandates `saturating_add` has activated on all supported networks. Retain only the `saturating_add` branch.
2. Until removal, add an explicit range check before the `wrapping_add` path: reject any `height_relative` or `seconds_relative` value that would cause `confirmed_block_index + height_relative` to exceed `u32::MAX` (or the timestamp equivalent), returning `AssertHeightRelativeFailed` rather than silently wrapping.
3. Add a consensus-layer assertion that `nowrap=true` is always set for any block height above the soft-fork activation point, so the wrapping path cannot be reached for new blocks.

### Proof of Concept

Given a coin confirmed at block height `100`:

```
height_relative = u32::MAX - 100 + 1 = 0xFFFF_FF9C

confirmed_block_index.wrapping_add(height_relative)
  = 100u32.wrapping_add(0xFFFF_FF9Cu32)
  = (100 + 4294967196) mod 2^32
  = 4294967296 mod 2^32
  = 0

Guard: prev_transaction_block_height < 0  →  always false
Result: no error returned, timelock bypassed at any block height
```

The same construction applies to `seconds_relative` using `u64::MAX - timestamp + 1`. The `sanitize_uint` parser accepts these values without error since they fit within the 4-byte (height) and 8-byte (seconds) limits. [6](#0-5) [7](#0-6) [8](#0-7)
