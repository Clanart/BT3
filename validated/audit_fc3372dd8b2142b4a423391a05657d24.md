### Title
Partial `ft_on_transfer` Refund Causes Permanent Token Freeze and Accounting Desync — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When `process_fin_transfer_to_near` delivers tokens via `ft_transfer_call` (triggered by a non-empty `msg`), the bridge's `is_refund_required` check only triggers a full revert when the recipient uses **zero** tokens. If the recipient's `ft_on_transfer` returns a **partial** refund (non-zero but less than the full amount), the bridge incorrectly treats the transfer as fully successful: `locked_tokens` stays decremented by the full amount, the fee is paid out, the transfer is marked finalized, and the partially-refunded tokens are permanently frozen in the bridge contract with no recovery path.

---

### Finding Description

In `process_fin_transfer_to_near`, when `transfer_message.msg` is non-empty, `send_tokens` dispatches an `ft_transfer_call` to the recipient contract. The callback `fin_transfer_send_tokens_callback` then calls `is_refund_required` to decide whether to revert state.

`is_refund_required` reads the result of `ft_resolve_transfer`, which returns the **amount used** (i.e., `amount_sent − amount_refunded`):

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                    amount.0 == 0   // ← only reverts on FULL rejection
                } else { false }
            }
            Err(_) => false,
        }
    } else { false }
}
``` [1](#0-0) 

The function returns `true` (revert) only when `amount_used == 0`. A **partial** refund (e.g., recipient returns 50 out of 90 tokens) yields `amount_used = 40`, so `is_refund_required` returns `false`.

In `fin_transfer_send_tokens_callback`, the `false` branch:
1. Does **not** call `revert_lock_actions` → `locked_tokens` stays decremented by the full `transfer_message.amount`
2. Mints/transfers the fee to `fee_recipient`
3. Emits `FinTransferEvent` (transfer is considered finalized)
4. Does **not** burn or track the 50 refunded tokens now sitting in the bridge contract [2](#0-1) 

The `send_tokens` path that triggers this for non-deployed tokens with a non-empty message:

```rust
ext_token::ext(token)
    .ft_transfer_call(recipient, amount, None, msg.to_string())
``` [3](#0-2) 

And the `process_fin_transfer_to_near` call that sets `is_ft_transfer_call = !msg.is_empty()`:

```rust
.fin_transfer_send_tokens_callback(
    transfer_message,
    &fee_recipient,
    !msg.is_empty(),   // ← is_ft_transfer_call
    predecessor_account_id,
    lock_actions,
)
``` [4](#0-3) 

The `unlock_tokens_if_needed` call that decrements `locked_tokens` by the full amount before the token send:

```rust
let lock_actions = vec![self.unlock_tokens_if_needed(
    transfer_message.get_origin_chain(),
    &token,
    transfer_message.amount.0,   // ← full amount, including fee
)];
``` [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of user funds.** The partially-refunded tokens accumulate in the bridge contract's token balance. For non-deployed (native) tokens, these are real locked assets with no withdrawal function. For deployed (bridge-minted) tokens, these are minted tokens that are stuck and can never be burned or redeemed. The bridge has no `withdraw` or `recover` function for arbitrary token balances; the only recourse is a DAO-initiated contract upgrade.

Additionally, `locked_tokens` is understated by the refunded amount, creating a persistent accounting desync between the bridge's actual token holdings and its tracked collateral. This breaks the bridge collateralization invariant: `locked_tokens[chain][token]` no longer accurately reflects the tokens backing cross-chain supply.

---

### Likelihood Explanation

Any user who initiates a bridge transfer with a non-empty `msg` field (used for DeFi integrations such as DEX swaps, lending deposits, or yield aggregators on NEAR) is exposed. NEAR DeFi contracts commonly return partial refunds from `ft_on_transfer` when slippage limits are hit, liquidity is insufficient, or position caps are reached. This is a standard NEP-141 pattern. The attacker-controlled entry point is the `msg` field in the origin-chain `InitTransfer` call, which is fully user-controlled and unprivileged.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, read the actual amount used from the `ft_resolve_transfer` result and compare it against `amount_without_fee`. If the used amount is less than `amount_without_fee` (partial refund), treat it as a failure: call `revert_lock_actions`, burn any minted tokens, and remove the finalized transfer record so the user can re-initiate. Alternatively, emit a recoverable event and allow the user to claim the stuck partial refund.

---

### Proof of Concept

1. User on ETH initiates `InitTransfer` with `amount = 100`, `fee = 10`, `msg = "swap:..."` targeting a NEAR DEX contract as recipient.
2. Relayer calls `fin_transfer` on NEAR bridge; `fin_transfer_callback` runs, unlocking 100 from `locked_tokens[ETH][token]`.
3. `process_fin_transfer_to_near` calls `send_tokens(token, dex_contract, 90, "swap:...")` → `ft_transfer_call`.
4. DEX's `ft_on_transfer` hits a slippage limit and returns `50` (refund 50 tokens back to bridge).
5. `ft_resolve_transfer` returns `40` (amount used = 90 − 50).
6. `is_refund_required` evaluates `40 == 0` → `false`.
7. `fin_transfer_send_tokens_callback` enters the success branch: mints 10 tokens to fee_recipient, emits `FinTransferEvent`.
8. `revert_lock_actions` is never called; `locked_tokens[ETH][token]` remains decremented by 100.
9. The 50 refunded tokens sit in the bridge contract's balance with no recovery mechanism.
10. The transfer is marked finalized; it cannot be re-submitted. The user's 50 tokens are permanently frozen.
