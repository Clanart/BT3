**Step 1 — ETH enters the router via `multicall{value: X}`**

`multicall` is `public payable`. The `receive()` guard only blocks direct `.call{value}("")` transfers from non-WETH addresses; it does **not** block `msg.value` arriving through a normal function call. So a user can legitimately do:

```
router.multicall{value: 2 ETH}([exactInputSingle(amountIn=1 ETH, tokenIn=WETH)])
```

**Step 2 — `pay()` wraps only the exact amount needed**

Inside `pay()` when `token == WETH` and `nativeBalance >= value`:

```solidity
IWETH9(WETH).deposit{value: value}();   // wraps exactly 1 ETH
IERC20(WETH).safeTransfer(recipient, value);
// 1 ETH remains on the router
``` [1](#0-0) 

The excess `nativeBalance - value` ETH is **not consumed** and stays on the router after the multicall completes.

**Step 3 — `refundETH()` has zero access control**

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
``` [2](#0-1) 

Any EOA or contract can call this in a **separate transaction** after the victim's multicall settles. `msg.sender` is the attacker, not the original depositor.

**Step 4 — Does the invariant hold?**

The intended invariant is: excess ETH is returned only to the original depositor. This holds **only if** the user includes `refundETH()` as the last leg of their own multicall (where `delegatecall` preserves `msg.sender`). There is no enforcement of this. If the user omits `refundETH()`, the ETH is stranded and claimable by anyone.

### Title
Unguarded `refundETH()` allows any caller to drain ETH stranded on the router after a partial-value multicall — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`refundETH()` unconditionally transfers the router's entire ETH balance to `msg.sender` with no access control. When a user sends `msg.value > amountIn` in a `multicall` without appending a `refundETH()` leg, the excess ETH remains on the router. Any third party can then call `refundETH()` in a subsequent transaction and receive those funds.

### Finding Description
`PeripheryPayments.pay()` wraps exactly `value` wei when `nativeBalance >= value`, leaving `nativeBalance - value` ETH on the contract. [1](#0-0) 

`refundETH()` sends `address(this).balance` to `msg.sender` with no check that the caller is the original depositor. [2](#0-1) 

`multicall` uses `delegatecall`, so `msg.sender` is the original user only while the multicall is executing. Once the multicall transaction is mined, any subsequent caller of `refundETH()` becomes the new `msg.sender` and receives all stranded ETH. [3](#0-2) 

The `receive()` guard does not prevent this because it only blocks direct `.call{value}` transfers, not `msg.value` arriving through `multicall`. [4](#0-3) 

### Impact Explanation
Direct loss of user ETH. A victim who sends `msg.value = X` but whose swap consumes only `Y < X` loses `X - Y` ETH to any attacker who calls `refundETH()` before the victim can recover it. The loss is bounded by the excess ETH sent per transaction but is fully extractable with zero capital requirement by the attacker.

### Likelihood Explanation
Moderate. Native-ETH swap flows are explicitly documented and tested in the codebase. Users or integrators who send a round-number `msg.value` (e.g., to cover slippage) without appending `refundETH()` create the vulnerable state. MEV bots monitoring the mempool can detect and exploit this in the same block.

### Recommendation
Add a caller check in `refundETH()` that restricts it to the address that initiated the current multicall, or track the depositor in transient storage at `multicall` entry and enforce it in `refundETH()`. Alternatively, automatically sweep excess ETH back to `msg.sender` at the end of `multicall` rather than relying on the caller to append a `refundETH()` leg.

        calls[0] = abi.encodeWithSelector(
            router.exactInputSingle.selector,
            IMetricOmmSimpleRouter.ExactInputSingleParams({
