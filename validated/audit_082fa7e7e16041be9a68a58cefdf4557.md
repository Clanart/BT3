### Title
Timelock Bypass via Integer Overflow in `check_time_locks` with Legacy Wrapping Arithmetic — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary
When `nowrap=false` (legacy consensus mode), `check_time_locks` uses `wrapping_add` for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` enforcement. A CLVM puzzle emitting these conditions with near-maximum values (e.g., `u32::MAX` for height, `u64::MAX` for seconds) causes the addition to wrap around to a value smaller than the coin's birth height/timestamp, making the timelock check trivially pass. A coin intended to be locked for billions of blocks or seconds can be spent immediately.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between saturating and wrapping arithmetic for relative timelock enforcement: [1](#0-0) 

When `nowrap=false`, the height-relative check is:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [2](#0-1) 

For a coin confirmed at block index `B` with `ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF` (`u32::MAX`):
- `B.wrapping_add(0xFFFF_FFFF)` = `B - 1` (wraps around)
- The check `prev_height < B - 1` passes for any `prev_height >= B - 1`
- The coin is spendable immediately — the ~4-billion-block timelock is completely bypassed

The same overflow applies to `seconds_relative` with `u64::MAX`: [3](#0-2) 

The test suite explicitly documents and confirms this behavior: [4](#0-3) 

The Python-level test also confirms the divergence between `nowrap=True` and `nowrap=False`: [5](#0-4) 

The Python binding exposes `nowrap` as a caller-controlled parameter: [6](#0-5) 

---

### Impact Explanation

Any CLVM puzzle that emits `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` or `ASSERT_SECONDS_RELATIVE 0xFFFF_FFFF_FFFF_FFFF` is intended to enforce a multi-billion-block or multi-billion-year timelock. Under `nowrap=false`, the wrapping arithmetic reduces the effective threshold to `confirmed_block_index - 1` (or `coin_timestamp - 1`), which is satisfied immediately. This is a **timelock validation bypass enabling unauthorized spend acceptance** — matching the High impact category. Any coin whose puzzle relies on a large relative timelock for security (e.g., vesting contracts, time-locked multisigs, or protocol-enforced delays) is vulnerable.

---

### Likelihood Explanation

The `nowrap` flag is a soft-fork parameter passed by the Python full node based on block height. While `nowrap=true` (saturating) is the correct post-fork behavior, `nowrap=false` (wrapping) remains active for pre-fork blocks and on any node that has not yet applied the soft fork. The attacker entry path requires only:
1. Crafting a CLVM puzzle that emits `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` (or `ASSERT_SECONDS_RELATIVE 0xFFFF_FFFF_FFFF_FFFF`)
2. Submitting a spend bundle to a node running with `nowrap=false`

No privileged access, key compromise, or network-level attack is required. The attacker controls the puzzle entirely.

---

### Recommendation

1. **Activate the soft fork** (`nowrap=true`) at the earliest safe block height to eliminate the wrapping path from live consensus validation.
2. **Reject near-maximum relative timelock values at parse time** in `conditions.rs` — values of `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` that would wrap under any arithmetic mode should be treated as invalid conditions, analogous to how negative values are already clamped to `None`.
3. **Remove the `nowrap=false` code path** once the soft fork is universally active, to eliminate the dead but dangerous branch.

---

### Proof of Concept

The existing test case in `check_time_locks.rs` is itself the proof of concept:

```
Coin confirmed at height 100.
Current height: 200.
ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF (u32::MAX)

nowrap=true:  100.saturating_add(0xFFFF_FFFF) = u32::MAX
              200 < u32::MAX → Err (correctly rejected)

nowrap=false: 100.wrapping_add(0xFFFF_FFFF) = 99
              200 < 99 → false → Ok (timelock bypassed)
``` [7](#0-6) 

A spend bundle containing a coin with puzzle output `(ASSERT_HEIGHT_RELATIVE . 0xFFFF_FFFF)` submitted to a node using `nowrap=false` will be accepted at any block height ≥ 99, regardless of the coin's confirmation height, completely defeating the intended ~4-billion-block delay.
