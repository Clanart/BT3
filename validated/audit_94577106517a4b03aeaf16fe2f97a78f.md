**Can ETH be stranded cross-transaction?**

The `receive()` guard blocks direct ETH pushes from non-WETH addresses: [2](#0-1) 

However, `msg.value` in any `payable` function bypasses `receive()` entirely and lands directly in `address(this).balance`. The `multicall` is `payable`: [3](#0-2) 

`exactInputSingle` is also `payable`: [4](#0-3) 

If User-A calls `multicall{value: 2 ETH}([exactInputSingle(amountIn=1 ETH, tokenIn=WETH)])` **without** a trailing `refundETH()` call, the 1 ETH residue persists on the router after the transaction succeeds. `Address.functionDelegateCall` propagates reverts, so a revert would return the ETH — but a successful multicall that simply over-sends leaves the residue stranded.

**The exploit path is real:**

1. User-A sends `multicall{value: 2 ETH}([exactInputSingle(amountIn=1 ETH, tokenIn=WETH)])` — no `refundETH()` step. 1 ETH is stranded.
2. User-B calls `exactInputSingle(tokenIn=WETH, amountIn=1.5 ETH)` with `msg.value=0`.
3. Inside `pay(WETH, userB, pool, 1.5 ETH)`: `nativeBalance = 1 ETH`, which is `> 0` and `< 1.5 ETH`.
4. The hybrid branch deposits User-A's 1 ETH as WETH → pool, then pulls only 0.5 ETH worth of WETH from User-B.
5. User-A loses 1 ETH; User-B's swap is subsidized by 1 ETH.

**Are there any guards that break this path?**

- `refundETH()` is public and sends the full balance to `msg.sender` — but it only helps if User-A calls it in the same multicall. There is no automatic refund.
- The transient callback context (`_setNextCallbackContext`, `_clearExpectedCallbackPool`) binds the pool and payer correctly for the current swap, but does not restrict which ETH the `pay` function may consume.
- The `receive()` guard is irrelevant here because the ETH arrived via `msg.value`, not a direct push.

No guard preserves the ETH ownership invariant across transactions.

### Title
Unattributed native ETH balance in `pay()` hybrid branch allows any WETH swap to consume prior user's stranded `msg.value` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses `address(this).balance` without any per-user attribution. When the router holds residual ETH from a previous caller's `msg.value`, a subsequent WETH swap by any user triggers the hybrid branch and silently consumes that residual ETH to fund the new swap.

### Finding Description
`pay` contains a WETH hybrid branch:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} ...
```

The branch is designed for the pattern where a user sends `msg.value` in the same transaction to cover a WETH input. However, `address(this).balance` is a global contract balance with no per-caller accounting. Any ETH stranded from a prior transaction (e.g., a user who over-sent `msg.value` and omitted `refundETH()`) is indistinguishable from the current caller's intended payment. The next WETH swap will consume it.

### Impact Explanation
Direct loss of user principal. User-A's stranded ETH is transferred to a pool on behalf of User-B. User-A receives nothing; User-B's swap input cost is reduced by the stolen amount. The loss equals the stranded ETH balance, which can be up to the full `msg.value` of User-A's transaction.

### Likelihood Explanation
Medium. The precondition is that ETH is stranded on the router. This occurs whenever a user (or a frontend on their behalf) calls a `payable` router function with excess `msg.value` and omits `refundETH()`. This is a realistic integration mistake — the pattern is documented as opt-in, not enforced. Once ETH is stranded, any subsequent WETH swap by any user triggers the theft automatically with no special setup.

### Recommendation
Track per-transaction ETH attribution. One approach: record `msg.value` in transient storage at the start of each top-level call and deduct from it inside `pay`, reverting if the deduction would exceed the recorded value. Alternatively, remove the hybrid branch entirely and require callers to wrap ETH to WETH before calling the router (eliminating the ambiguity at the cost of one extra step).
