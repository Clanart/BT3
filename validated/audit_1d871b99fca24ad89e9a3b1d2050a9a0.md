### Title
Wrong State Diff Commitment and Block Hash Due to Unconditionally Empty `deprecated_declared_classes` in `CommitmentStateDiff → ThinStateDiff` Conversion - (File: `crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion in the blockifier unconditionally sets `deprecated_declared_classes: Vec::new()`. This `ThinStateDiff` is the one fed directly into `calculate_block_commitments` (and thus `calculate_state_diff_hash`) during block proposal construction. Any block that contains deprecated (Cairo 0) class declarations will have those classes silently dropped from the state diff commitment, producing a wrong `state_diff_commitment`, a wrong `concatenated_counts` (wrong `state_diff_length`), and therefore a wrong `PartialBlockHash` / final block hash. The sequencer will accept and commit this block with the corrupted commitment.

---

### Finding Description

`CommitmentStateDiff` (in `crates/blockifier/src/state/cached_state.rs`) has four fields: `address_to_class_hash`, `address_to_nonce`, `storage_updates`, and `class_hash_to_compiled_class_hash`. It has **no** `deprecated_declared_classes` field. [1](#0-0) 

The `From<CommitmentStateDiff> for ThinStateDiff` implementation hard-codes `deprecated_declared_classes: Vec::new()`: [2](#0-1) 

`BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())` and passes the result directly to `calculate_block_commitments`: [3](#0-2) 

`calculate_block_commitments` uses `state_diff.len()` for `concatenated_counts` and calls `calculate_state_diff_hash(&state_diff)`: [4](#0-3) 

`ThinStateDiff::len()` counts `deprecated_declared_classes.len()` as part of the state diff length: [5](#0-4) 

`calculate_state_diff_hash` chains `deprecated_declared_classes` into the Poseidon hash: [6](#0-5) 

The resulting `state_diff_commitment` and `concatenated_counts` are stored in `BlockHeaderCommitments`, which is embedded in `PartialBlockHashComponents` and used to compute the `PartialBlockHash` (the consensus proposal commitment) and ultimately the final block hash: [7](#0-6) 

The `thin_state_diff()` method on `BlockExecutionArtifacts` also calls the same lossy conversion, so the `ThinStateDiff` written to storage via `commit_proposal` is also missing `deprecated_declared_classes`: [8](#0-7) 

The `ThinStateDiff` stored in the MDBX `state_diffs` table (via `append_state_diff`) is the canonical source for RPC `starknet_getStateUpdate`, P2P sync, and proof inputs. All of these will serve a state diff that is missing the deprecated declared classes. [9](#0-8) 

The comment in the code acknowledges this is a known structural issue but does not guard against it producing wrong commitments: [10](#0-9) 

---

### Impact Explanation

**Wrong state_diff_commitment in block hash (Critical):** For any block containing a deprecated (Cairo 0) `declare` transaction, the `state_diff_commitment` field of `BlockHeaderCommitments` will be computed over a `ThinStateDiff` with an empty `deprecated_declared_classes`, while the correct value must include those class hashes. This produces a wrong `StateDiffCommitment` value.

**Wrong concatenated_counts / state_diff_length (Critical):** `state_diff.len()` is used to compute `concatenated_counts` packed into the block hash. With `deprecated_declared_classes` empty, `len()` is under-counted by the number of deprecated declared classes, producing a wrong `concatenated_counts` field.

**Wrong PartialBlockHash / final block hash (Critical):** Both wrong values feed into `calculate_block_hash`, producing a wrong block hash that the sequencer commits and propagates. Validators that independently compute the correct hash will disagree.

**Wrong state diff stored and served (High):** The `ThinStateDiff` written to storage and returned by `starknet_getStateUpdate` RPC, P2P sync, and proof inputs is missing `deprecated_declared_classes`, causing downstream consumers (provers, sync nodes, RPC clients) to receive an authoritative-looking but incorrect state diff.

---

### Likelihood Explanation

Deprecated (Cairo 0) class declarations are a standard Starknet transaction type (`DECLARE` v0/v1). Any block produced by the sequencer that includes such a transaction triggers this bug. The conversion is on the hot path for every block proposal. The bug is silent — no error is raised, the wrong value is simply used.

---

### Recommendation

`CommitmentStateDiff` must be extended to carry `deprecated_declared_classes`, or the `From<CommitmentStateDiff> for ThinStateDiff` conversion must be replaced with a conversion that receives the deprecated declared classes separately (as `ThinStateDiff::from_state_diff` does for the full `StateDiff`). The `BlockExecutionSummary` / `BlockExecutionArtifacts` pipeline must propagate deprecated declared class hashes from the blockifier execution output through to the commitment calculation.

The existing comment `// TODO(AlonH): Remove this when the structure of storage diffs changes.` confirms this is a known structural gap, but it is not guarded against producing wrong commitments in the meantime.

---

### Proof of Concept

1. Submit a `DECLARE` v0 (Cairo 0) transaction to the sequencer.
2. The blockifier executes it and produces a `CommitmentStateDiff` with `class_hash_to_compiled_class_hash` empty (Cairo 0 has no compiled class hash) and the class hash present only in the deprecated declared classes — but `CommitmentStateDiff` has no such field.
3. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())`, which sets `deprecated_declared_classes: Vec::new()`.
4. `calculate_block_commitments` computes `state_diff.len()` = N (missing the deprecated class count) and `calculate_state_diff_hash` hashes a diff without the deprecated class.
5. The resulting `state_diff_commitment` and `concatenated_counts` are wrong.
6. `PartialBlockHash::from_partial_block_hash_components` and `calculate_block_hash` produce a wrong block hash.
7. The wrong block hash is committed to storage and broadcast to consensus peers.
8. A correct verifier recomputing the hash from the stored `ThinStateDiff` (which also has `deprecated_declared_classes: Vec::new()` due to the same conversion at line 200) will agree with the wrong hash — masking the bug from internal consistency checks but producing a hash that diverges from the Starknet protocol specification for blocks containing Cairo 0 declarations.
