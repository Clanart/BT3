### Title
Unchecked `u64` Subtraction in `eth_to_fri_rate` Causes Underflow on Small Timestamps, Silently Corrupting the L1 Gas Price Committed to the Block Hash — (File: `crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs`)

---

### Summary

`EthToStrkOracleClient::eth_to_fri_rate` performs two bare `u64` subtractions without first validating that the operands are large enough. A malicious proposer can craft a `ProposalInit` with a timestamp smaller than `lag_interval_seconds`, causing the validator's oracle call to silently produce a garbage `quantized_timestamp`. The oracle query then fails, the validator silently falls back to stale/default gas prices, and the proposer's hand-crafted gas prices — which need only be within the configured margin of those stale defaults — are accepted and committed into the block hash. No `overflow-checks = true` is present in the workspace, so in release builds the subtraction wraps rather than panics.

---

### Finding Description

**Defect 1 — line 198** (`eth_to_strk_oracle.rs`):

```rust
let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
    .checked_div(self.config.lag_interval_seconds)
    .expect("lag_interval_seconds should be non-zero");
```

`timestamp` and `lag_interval_seconds` are both `u64`. There is no guard that `timestamp >= lag_interval_seconds`. When `timestamp < lag_interval_seconds` the subtraction wraps in release mode to `u64::MAX - (lag_interval_seconds - timestamp - 1)`, producing a nonsensical `quantized_timestamp` (≈ `u64::MAX`).

**Defect 2 — line 217** (`eth_to_strk_oracle.rs`):

```rust
if let Some(rate) = cache.get(&(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)) {
```

`NUMBER_OF_TIMESTAMPS_BACK = 1`. When `quantized_timestamp == 0` (which occurs when `timestamp == lag_interval_seconds` exactly, e.g. `timestamp = 1` with the default `lag_interval_seconds = 1`) this second subtraction also wraps to `u64::MAX`.

**How the validator is reached**

`is_block_info_valid` in `validate_proposal.rs` passes `init_proposed.timestamp` directly to `get_l1_prices_in_fri_and_wei`:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← attacker-controlled
    ...
).await;
```

That function calls `L1GasPriceProvider::eth_to_fri_rate(timestamp)` → `EthToStrkOracleClient::eth_to_fri_rate(timestamp)`, hitting the defective subtraction.

**Timestamp validation does not prevent the trigger**

The only timestamp checks in `is_block_info_valid` are:

```rust
if init_proposed.timestamp < last_block_timestamp { ... }   // ≥ previous block
if init_proposed.timestamp > now + window { ... }           // not too far future
```

For the genesis block `last_block_timestamp = 0`, so `timestamp = 0` passes both checks. With `lag_interval_seconds = 1` (the shipped default), `timestamp = 0` causes Defect 1; `timestamp = 1` causes Defect 2. With any larger `lag_interval_seconds` the window of vulnerable timestamps grows proportionally.

**Consequence of the wrap**

The wrapped `quantized_timestamp ≈ u64::MAX` causes `spawn_query` to fire an oracle HTTP request with `adjusted_timestamp = u64::MAX * lag_interval_seconds` (itself wrapping). The oracle returns an error. `get_l1_prices_in_fri_and_wei_and_conversion_rate` then falls through to the previous-block-info or hard-coded default fallback:

```rust
let default_l1_gas_prices_fri =
    L1PricesInFri::convert_from_wei(&default_l1_gas_prices_wei, DEFAULT_ETH_TO_FRI_RATE)
        .expect("Default values should be convertible between wei and fri.");
```

The validator now compares the proposer's gas prices against these stale/default values using `within_margin`. The proposer only needs to stay within `l1_gas_price_margin_percent` of the fallback, regardless of the actual live L1 gas price.

**Gas prices flow into the block hash**

The accepted `ProposalInit` gas prices are stored in `PartialBlockHashComponents` via `PartialBlockHashComponents::new` and then hashed by `calculate_block_hash`:

```rust
.chain_iter(
    gas_prices_to_hash(
        &partial_block_hash_components.l1_gas_price,
        &partial_block_hash_components.l1_data_gas_price,
        &partial_block_hash_components.l2_gas_price,
        &block_hash_version,
    ).iter(),
)
```

Wrong gas prices are therefore permanently committed into the block hash and used for all fee calculations in the block.

---

### Impact Explanation

A malicious proposer can commit an L1 gas price of their choosing (within the margin of stale defaults) to the block hash. Every transaction in that block is executed and charged fees based on the wrong gas price. This constitutes an incorrect fee/gas/resource-accounting effect with direct economic impact, matching the **Critical** impact tier: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

In debug builds the wrap becomes a panic, crashing the validator node (DoS).

---

### Likelihood Explanation

The default `lag_interval_seconds = 1` means only `timestamp = 0` triggers Defect 1 in normal operation; `timestamp = 1` triggers Defect 2. Both are reachable for the genesis block or the first block after genesis. Any operator who increases `lag_interval_seconds` (e.g. to 60 s, as shown in the documentation diagrams) widens the vulnerable timestamp range to cover the first minute of chain history. A Byzantine proposer who controls the first proposal can exploit this deterministically.

---

### Recommendation

Add an explicit guard before the subtraction in `eth_to_fri_rate`:

```rust
if timestamp < self.config.lag_interval_seconds {
    return Err(EthToStrkOracleClientError::InvalidTimestampError(
        timestamp,
        self.config.lag_interval_seconds,
    ));
}
let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
    .checked_div(self.config.lag_interval_seconds)
    .expect("lag_interval_seconds should be non-zero");
```

Similarly guard the second subtraction:

```rust
if quantized_timestamp < NUMBER_OF_TIMESTAMPS_BACK {
    return Err(EthToStrkOracleClientError::QueryNotReadyError(timestamp));
}
if let Some(rate) = cache.get(&(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK)) { ... }
```

Optionally, add a corresponding lower-bound check in `is_block_info_valid` that rejects any proposed timestamp smaller than `lag_interval_seconds`.

### Proof of Concept

1. Configure a sequencer with `lag_interval_seconds = 60`.
2. As the proposer for block 0, broadcast a `ProposalInit` with `timestamp = 0` and `l1_gas_price_fri = DEFAULT_MIN_GAS_PRICE` (within margin of the hard-coded default).
3. The validator calls `eth_to_fri_rate(0)`.
4. Line 198 computes `(0u64 - 60u64)` → wraps to `18446744073709551556` in release mode.
5. `spawn_query(18446744073709551556)` fires; the oracle rejects the nonsensical timestamp.
6. `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls back to `DEFAULT_ETH_TO_FRI_RATE`.
7. `within_margin` passes because the proposer's price equals the default.
8. The proposal is accepted; `DEFAULT_MIN_GAS_PRICE` is committed into the block hash via `calculate_block_hash`, regardless of the actual live L1 gas price. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
