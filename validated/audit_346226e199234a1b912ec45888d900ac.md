### Title
Missing Zero-Check on `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deprecated_execute_syscalls.cairo` and `syscall_impls.cairo`)

---

### Summary

The OS-level `execute_replace_class` handler (both deprecated and current variants) accepts a caller-supplied `class_hash` and writes it directly into the contract's `StateEntry` without asserting it is non-zero. Because `0` is the canonical `UNINITIALIZED_CLASS_HASH` sentinel, any contract that invokes `replace_class(0)` — whether by bug or by attacker-controlled calldata — permanently sets its own class hash to the uninitialized value. The contract becomes permanently uncallable, and any funds it holds are frozen forever with no recovery path.

---

### Finding Description

`execute_replace_class` in `deprecated_execute_syscalls.cairo` reads the caller-supplied `class_hash` directly from the syscall pointer and writes it into `contract_state_changes` with no validation:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;          // ← attacker-controlled, no check

    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash,                        // ← written unconditionally
        storage_ptr=state_entry.storage_ptr,
        nonce=state_entry.nonce,
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The same pattern exists in the non-deprecated `execute_replace_class` imported into `execute_syscalls.cairo`: [2](#0-1) 

By contrast, the OS **does** enforce a non-zero check on `compiled_class_hash` during declare transactions:

```cairo
assert_not_zero(compiled_class_hash);
``` [3](#0-2) 

And `deploy_contract` explicitly asserts that a freshly deployed contract starts with `UNINITIALIZED_CLASS_HASH` (i.e., `0`): [4](#0-3) 

This confirms that `0` is the protocol-defined "no class" sentinel. Writing it back via `replace_class` is semantically equivalent to undeploying the contract, but the OS does not guard against it.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to `0` via `replace_class(0)`, every subsequent call to that contract will fail at class resolution: the OS will look up class hash `0`, find no compiled class, and revert. The contract's storage (including any ERC-20 balances, LP positions, or escrowed assets) becomes permanently inaccessible. There is no upgrade path because `replace_class` itself requires the contract to be callable. The state change is committed to the Starknet state commitment and cannot be undone.

---

### Likelihood Explanation

Any deployed contract can issue the `replace_class` syscall. Two realistic paths exist:

1. **Attacker-controlled calldata forwarded to `replace_class`**: A contract that reads a class hash from user-supplied calldata and passes it to `replace_class` (e.g., a proxy upgrade pattern) can be exploited by an unprivileged sender passing `0`.
2. **Malicious contract rug-pull**: An attacker deploys a contract that accepts user deposits, then calls `replace_class(0)` in a subsequent transaction to freeze all deposited funds. No privileged access is required.

Both paths require only a standard `invoke` transaction from an unprivileged account.

---

### Recommendation

Add an `assert_not_zero` guard on `class_hash` before writing it to state, mirroring the existing check on `compiled_class_hash` in declare transactions:

```cairo
from starkware.cairo.common.math import assert_not_zero

func execute_replace_class{...}(contract_address, syscall_ptr: ReplaceClass*) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    with_attr error_message("replace_class: class_hash must not be zero.") {
        assert_not_zero(class_hash);
    }
    ...
}
```

Apply the same fix to both the deprecated variant in `deprecated_execute_syscalls.cairo` and the current variant in `syscall_impls.cairo`.

---

### Proof of Concept

1. Attacker deploys contract `A` that holds user funds and exposes an `upgrade(class_hash: felt)` entry point which calls `replace_class(class_hash)`.
2. Users deposit funds into contract `A`.
3. Attacker sends an `invoke` transaction calling `A.upgrade(0)`.
4. The OS executes `execute_replace_class` with `class_hash = 0`, writing `StateEntry(class_hash=0, ...)` into `contract_state_changes`.
5. The new state is committed to the Starknet state trie.
6. All subsequent calls to contract `A` fail at class resolution (class hash `0` has no compiled class).
7. All funds in contract `A`'s storage are permanently frozen with no recovery mechanism.
