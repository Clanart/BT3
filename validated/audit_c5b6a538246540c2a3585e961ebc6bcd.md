### Title
Detached Burn Promise in `burn_tokens_if_needed` Allows Unbacked Token Minting on Destination Chain — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `burn_tokens_if_needed` function in the NEAR bridge contract fires the cross-contract `burn` call with `.detach()`, meaning the result is never observed. If the burn fails for any reason (e.g., insufficient gas, token contract panic), the `InitTransferEvent` is still emitted and the destination chain will mint tokens without the corresponding NEAR-side bridged tokens being destroyed. This inflates total supply and breaks bridge collateralization.

---

### Finding Description

In `near/omni-bridge/src/lib.rs`, the internal helper `burn_tokens_if_needed` is:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result is never checked
    }
}
``` [1](#0-0) 

`BURN_TOKEN_GAS` is only `3 TGas`: [2](#0-1) 

This function is called from `init_transfer_internal`, which proceeds unconditionally to emit `InitTransferEvent` and return `U128(0)` (meaning "all tokens consumed") regardless of whether the burn succeeds:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(...);
// ...
env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
U128(0)
``` [3](#0-2) 

The NEP-141 `ft_transfer_call` flow means the tokens are already in the bridge's custody when `init_transfer_internal` runs. The bridge returns `U128(0)` to the token contract, confirming it consumed all tokens. If the detached burn then fails silently, the bridge retains the tokens it should have destroyed, while the `InitTransferEvent` has already been emitted and will be relayed to the destination chain to mint an equivalent amount.

---

### Impact Explanation

**Critical — Direct unauthorized mint of bridged assets / bridge collateralization broken.**

Concrete outcome:
1. User holds 100 `token.near` (a bridge-deployed token) on NEAR.
2. User calls `ft_transfer_call` → bridge receives 100 tokens, calls `burn_tokens_if_needed` with `.detach()`.
3. Burn fails silently (e.g., 3 TGas exhausted by a non-trivial token contract).
4. `InitTransferEvent` is emitted; bridge returns `U128(0)` to the token contract (tokens not refunded).
5. Destination chain (EVM/Solana/StarkNet) relayer observes the event and finalizes the transfer, minting 100 tokens for the recipient.
6. Result: 100 tokens exist in the bridge on NEAR (should have been burned) **and** 100 new tokens exist on the destination chain. Total supply is now 200 instead of 100 — the bridge is undercollateralized by 100 tokens.

This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization."**

---

### Likelihood Explanation

**Medium.** The `BURN_TOKEN_GAS` budget of 3 TGas is extremely tight. Any bridged token whose `burn` implementation does more than a trivial balance decrement (e.g., emits events, performs storage operations, calls into another contract) can exceed this budget. Because `deployToken` is permissionless (any valid MPC-signed metadata can deploy a token), a variety of token implementations may be registered. A burn failure due to gas exhaustion is a realistic, non-adversarial scenario. An attacker who can predict or induce a gas-exhaustion condition (e.g., by deploying a token with a slightly heavier `burn` body) can reliably trigger this.

---

### Recommendation

Replace the detached burn with a chained callback that verifies success before the `InitTransferEvent` is emitted. The event and the `U128(0)` return should only be committed after the burn promise resolves successfully. If the burn fails, the callback should return the full token amount to the caller (refund) and remove the pending transfer message.

Alternatively, increase `BURN_TOKEN_GAS` to a safe margin and add a `.then(...)` callback that checks `env::promise_result_checked(0, ...)` and panics (causing a full rollback) on failure — analogous to the existing `is_refund_required` pattern already used in the codebase. [4](#0-3) 

---

### Proof of Concept

1. Deploy a bridged token on NEAR whose `burn` function consumes slightly more than 3 TGas (e.g., emits a log or performs an extra storage read).
2. Acquire 1000 units of this token on NEAR.
3. Call `ft_transfer_call(bridge_account, "1000", transfer_msg)` targeting any supported destination chain.
4. The bridge's `ft_on_transfer` calls `init_transfer_internal` → `burn_tokens_if_needed` fires a detached burn with 3 TGas → burn runs out of gas and fails silently.
5. `init_transfer_internal` returns `U128(0)` — the token contract does not refund the 1000 tokens.
6. `InitTransferEvent` is emitted on-chain.
7. A relayer picks up the event and calls `finTransfer` on the destination chain, minting 1000 tokens for the attacker.
8. The bridge now holds 1000 tokens on NEAR (not burned) and 1000 new tokens exist on the destination chain — total supply inflated by 1000.
