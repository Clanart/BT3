Looking at the full validation chain in `set_state_header` and `set_state_part`, I need to determine whether the shard-binding check is actually enforced.

### Title
Missing Shard-ID Binding in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from an unprivileged peer without ever verifying that the embedded chunk belongs to the requested `shard_id`. Because `verify_path` is position-blind and `validate_state_part` ignores `shard_id` for actual trie validation, an attacker can substitute shard B's chunk into a header for shard A. The stored header then causes `set_state_part` to validate and persist state parts against shard B's `state_root`, ultimately installing shard B's trie data into shard A's storage.

---

### Finding Description

**Entrypoint:** `StateSyncAdapter::set_state_header` at `chain/chain/src/state_sync/adapter.rs:368`.

The function performs five checks before storing the header:

**Check 1–2 (`validate_chunk_proofs`):** Verifies the chunk's internal hash, tx root, and receipts root. [1](#0-0) 
It does **not** verify `chunk.shard_id() == shard_id`. A legitimately signed chunk from shard B passes this check unchanged.

**Check 3a (Merkle inclusion):** Verifies the chunk is in the block's `chunk_headers_root` via `verify_path`. [2](#0-1) 

`verify_path` is purely position-blind — it only checks that the item hashes to the root via the supplied path, with no index/position constraint: [3](#0-2) 

The `chunk_headers_root` is a Merkle tree over `ChunkHashHeight` values ordered by shard index: [4](#0-3) 

Shard B's chunk is a legitimate leaf in this tree. Its own Merkle path proves membership without revealing which shard index it occupies. The check passes.

**Check 3b (prev_chunk Merkle inclusion):** Same position-blind `verify_path` applied to the previous chunk. An attacker supplies shard B's previous chunk header and its valid Merkle path. This passes. [5](#0-4) 

**Check 4 (receipt proofs):** Receipt proofs are validated against `shard_id` (shard A). These proofs are derived from on-chain data and are publicly computable by any peer. The chain-termination check at step 4g requires the receipt proof chain to end at `prev_chunk_header.height_included()`. When shard A and shard B have their previous chunks at the same block height — the common case in a healthy network — the attacker supplies shard B's previous chunk height, and the check passes. [6](#0-5) 

**Check 5 (`validate_state_root_node`):** Verifies `hash(state_root_node.data) == chunk.prev_state_root()`. The attacker supplies shard B's `state_root_node`, which is consistent with shard B's `prev_state_root`. This passes. [7](#0-6) 

**Storage:** The header is stored under key `StateHeaderKey(shard_id_A, sync_hash)` — but it contains shard B's chunk and shard B's `prev_state_root`. [8](#0-7) 

**`set_state_part` propagation:** `set_state_part` reads the stored header and extracts `state_root` from the chunk's inner: [9](#0-8) 

It then calls `validate_state_part(shard_id_A, &shard_B_state_root, part_id, part)`. Critically, `validate_state_part`'s implementation passes `shard_id` only to metrics — the actual trie validation uses only `state_root`: [10](#0-9) [11](#0-10) 

The attacker provides state parts from shard B's trie. These pass `Trie::validate_state_part` against shard B's `state_root` and are stored as shard A's state parts under `StatePartKey(sync_hash, shard_A, part_id)`.

When `apply_state_part` is later called for shard A, it writes shard B's trie nodes into shard A's `ShardUId` storage: [12](#0-11) 

---

### Impact Explanation

The exact corrupted value is `state_root` at `adapter.rs:543` — it is shard B's `prev_state_root` instead of shard A's. This value is used to validate and accept all state parts for shard A. The syncing node installs shard B's complete trie as shard A's state. After `set_state_finalize`, the node operates with a permanently corrupted shard A state: wrong account balances, wrong contract storage, wrong validator proposals — all silently accepted because every individual proof checked out against the substituted root.

---

### Likelihood Explanation

Any peer in the network can respond to `StateRequestHeader` messages. The attack requires only publicly available on-chain data: shard B's chunk, its Merkle path in the block, shard B's previous chunk, its Merkle path, shard B's `state_root_node`, and shard A's incoming receipt proofs. The precondition that shard A and shard B have their previous chunks at the same height is satisfied in every epoch where all shards produce chunks at every block — the normal operating condition. No validator or operator privilege is required.

---

### Recommendation

Add an explicit shard-ID binding check immediately after `validate_chunk_proofs` in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This single guard closes the substitution path. The same check should be applied to `prev_chunk_header.shard_id()` if present.

---

### Proof of Concept

A unit test in `chain/chain` can demonstrate this without a live network:

1. Build a two-shard test environment with blocks containing chunks for both shard 0 and shard 1.
2. Obtain shard 1's `ShardChunk`, its Merkle path in the block's `chunk_headers_root`, shard 1's previous chunk and its Merkle path, shard 1's `state_root_node`, and shard 0's incoming receipt proofs.
3. Construct a `ShardStateSyncResponseHeaderV2` with shard 1's chunk but addressed to shard 0.
4. Call `state_sync_adapter.set_state_header(shard_0_id, sync_hash, crafted_header)`.
5. Assert it returns `Ok(())` — demonstrating the cross-shard header is accepted.
6. Call `get_state_header(shard_0_id, sync_hash)` and assert the stored header's `chunk.shard_id()` is shard 1, not shard 0.
7. Provide shard 1's state parts and call `set_state_part` for shard 0 — assert they are accepted.

The missing guard is the absence of any `chunk.shard_id() != shard_id` check anywhere in `set_state_header`. [13](#0-12)

    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());

            let prev_block_hash = chunk.prev_block_hash();

        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
