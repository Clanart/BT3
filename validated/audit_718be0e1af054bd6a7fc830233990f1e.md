### Title
Gateway `skip_stateful_validations` Allows Invalid-Signature Invoke to Bypass `__validate__` via Deploy-Account-in-Mempool Whitelist - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway skips the `__validate__` entry-point execution for an invoke transaction with `nonce == 1` whenever the sender account already has any transaction in the mempool (or a recent committed block). Because the only check is `account_tx_in_pool_or_recent_block`, an attacker who first submits a valid `deploy_account` transaction can immediately submit a second invoke transaction with `nonce == 1` and a completely invalid signature. The gateway admits the invalid invoke to the mempool without ever calling `__validate__`, violating the invariant that every mempool transaction has a verified signature.

### Finding Description

`skip_stateful_validations` is the sequencer analog of the M-29 "whitelisted contract" bypass. In M-29, the UniswapHandler was whitelisted and allowed callers to route around the buy-block restriction. Here, the presence of a `deploy_account` transaction in the mempool acts as the "whitelist token": it causes `account_tx_in_pool_or_recent_block` to return `true`, which causes `skip_stateful_validations` to return `true`, which causes `run_validate_entry_point` to set `validate: false`, which causes the blockifier's `StatefulValidator::perform_validations` to return `Ok(())` without ever calling `__validate__`.

The exact code path:

1. `skip_stateful_validations` checks `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` and then calls `account_tx_in_pool_or_recent_block`. [1](#0-0) 

2. `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool — not specifically a `deploy_account`. [2](#0-1) 

3. When `skip_validate == true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`. [3](#0-2) 

4. `StatefulValidator::perform_validations` returns immediately without calling `__validate__` when `validate == false`. [4](#0-3) 

5. `validate_tx` in `AccountTransaction` also short-circuits on `!self.execution_flags.validate`. [5](#0-4) 

6. The mempool's own `validate_tx` only checks nonce range and duplicates — it does not verify the signature. [6](#0-5) 

7. During block execution, `AccountTransaction::new_for_sequencing` always sets `validate: true`, so the invalid-signature invoke will revert. [7](#0-6) 

### Impact Explanation

An attacker can inject an invoke transaction with an arbitrary (invalid) signature into the mempool. The transaction will be included in a block and revert, wasting block space. The mempool invariant — that every admitted transaction has a cryptographically valid signature — is broken. This maps to:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The reverted transaction still occupies a slot in the block and consumes bouncer weight, enabling a sustained block-space-exhaustion attack at the cost of only one valid `deploy_account` fee per attack cycle.

### Likelihood Explanation

The trigger is fully unprivileged. Any external user can:
1. Generate a fresh key pair and craft a valid `deploy_account` transaction.
2. Immediately submit an invoke with `nonce == 1` and a garbage signature.

No special role, no privileged access, and no race condition is required. The `deploy_account` transaction need not ever be executed; it only needs to be accepted into the mempool pool (`tx_pool.contains_account` returns `true` as soon as the tx is added).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `deploy_account` transaction (not just any transaction) is pending for the account. Alternatively, use the `max_nonce_for_validation_skip` config field that already exists in `StatefulTransactionValidatorConfig` but is currently unused in `skip_stateful_validations` — and additionally gate the skip on the presence of a `deploy_account` tx hash, mirroring the stricter logic in `native_blockifier/src/py_validator.rs` which requires `deploy_account_tx_hash.is_some()`. [8](#0-7) [9](#0-8) 

### Proof of Concept

```
1. Attacker generates key pair (sk, pk) and computes account_address A.

2. Attacker submits a valid deploy_account tx for A (nonce=0, valid __validate_deploy__ signature).
   → Gateway: account_nonce=0, tx_nonce=0, deploy_account path → full execution → admitted to mempool.
   → mempool.tx_pool.contains_account(A) == true.

3. Attacker submits invoke tx for A (nonce=1, signature=[0xDEAD, 0xBEEF] — garbage).
   → Gateway calls skip_stateful_validations:
       tx.nonce() == Nonce(1) ✓
       account_nonce == Nonce(0) ✓
       account_tx_in_pool_or_recent_block(A) == true ✓  (deploy_account is in pool)
   → skip_validate = true
   → run_validate_entry_point sets ExecutionFlags { validate: false }
   → StatefulValidator returns Ok(()) without calling __validate__
   → Invalid invoke admitted to mempool.

4. Batcher builds block:
   - deploy_account executes (nonce=0) → A is deployed.
   - invoke executes (nonce=1, validate=true) → __validate__ called → signature invalid → REVERT.
   - Reverted invoke occupies block space; attacker pays fee for the revert.

5. Repeat from step 3 with a new nonce=1 invoke (different tx_hash) to continuously
   inject reverted transactions into blocks.
```
