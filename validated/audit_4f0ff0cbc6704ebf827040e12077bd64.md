### Title
Partial `ft_transfer_call` Refund Not Handled ā€” Tokens Permanently Frozen in Bridge - (File: near/omni-bridge/src/lib.rs)

### Summary

The NEAR bridge's `is_refund_required` helper only detects a **full rejection** (`used_amount == 0`) from a recipient contract's `ft_on_transfer`. When a recipient accepts **some but not all** tokens (partial refund), `is_refund_required` returns `false`, the transfer is recorded as fully successful, and the refunded tokens are permanently stranded inside the bridge contract with no recovery path.

### Finding Description

`send_tokens` dispatches tokens to the recipient via `ft_transfer_call` whenever the caller supplies a non-empty `msg`: [1](#0-0) 

After the call settles, `fin_transfer_send_tokens_callback` (for normal finalisations) and `resolve_fast_transfer` (for fast-transfer finalisations) both delegate to `is_refund_required` to decide whether to revert state: [2](#0-1) 

The function reads the promise result, which is the `used_amount` returned by the token contract's `ft_resolve_transfer` (i.e., how many tokens the recipient actually consumed). It only triggers a revert when `used_amount == 0`: [3](#0-2) 

The NEAR FT standard allows `ft_on_transfer` to return any value between `0` and the full amount as the refund. If the recipient returns `R` tokens (`0 < R < amount`), then `used_amount = amount - R > 0`, so `is_refund_required` returns `false`. The bridge then:

1. Emits `FinTransferEvent` as if the full amount was delivered: [4](#0-3) 

2. Pays the fee recipient the full fee.
3. Leaves the `R` refunded tokens sitting in the bridge contract with no accounting entry and no recovery mechanism.

The same gap exists in `resolve_fast_transfer`: [5](#0-4) 

Here `burn_tokens_if_needed` is called for the full `amount` before the refund check. For a deployed (bridged) token with a partial refund, the burn succeeds (the bridge holds the original relayer tokens plus the partial refund), but the `R` refunded tokens remain in the bridge after the burn, permanently frozen.

### Impact Explanation

Any tokens partially rejected by a recipient contract are irrecoverably locked inside the bridge. The source-chain sender has already burned or locked their tokens; the destination-chain recipient received fewer tokens than the signed payload specified; and the bridge's `finalised_transfers` set marks the nonce as used, preventing any re-finalization. This is a **permanent, irrecoverable freeze of user funds** matching the Critical/High impact tier.

### Likelihood Explanation

The condition is reachable by any unprivileged user who:
1. Initiates a cross-chain transfer with a non-empty `msg` field (routing tokens into a DeFi contract on NEAR), and
2. Targets a recipient contract whose `ft_on_transfer` returns a non-zero partial refund (e.g., a vault with a deposit cap, an AMM that can only absorb part of the liquidity, or any contract that conditionally accepts tokens).

Both conditions are normal, documented NEAR FT-standard behaviour. No privileged access is required.

### Recommendation

After `send_tokens` resolves, compare the actual `used_amount` against the expected `amount`. If `used_amount < amount` (partial refund), treat it the same as a full rejection: burn the refunded tokens (for deployed tokens), revert lock-accounting changes, and emit `FailedFinTransferEvent`. Alternatively, record the actual delivered amount and emit a corrected event so the source chain can settle the difference.

Concretely, `is_refund_required` should be replaced with a function that returns the **actual refunded amount** and the callers should act on any non-zero refund, not only on a zero `used_amount`.

### Proof of Concept

1. Alice initiates a transfer of 1000 USDC from Ethereum to NEAR, with `msg = '{"action":"deposit"}'` targeting a NEAR vault contract `vault.near`.
2. A relayer calls `fin_transfer` with the Ethereum proof.
3. `process_fin_transfer_to_near` ā†’ `send_tokens` ā†’ `ft_transfer_call(vault.near, 1000, msg)`.
4. `vault.near`'s `ft_on_transfer` can only accept 600 tokens (deposit cap) and returns `U128(400)` as the refund.
5. The token contract's `ft_resolve_transfer` refunds 400 USDC to the bridge; `used_amount = 600`.
6. `fin_transfer_send_tokens_callback` is called; `is_refund_required` sees `amount.0 = 600 ā‰  0` ā†’ returns `false`.
7. Bridge emits `FinTransferEvent` for the full 1000 USDC, pays the relayer fee, and exits.
8. Alice's vault received 600 USDC; 400 USDC are permanently locked in the bridge contract with no recovery path. [6](#0-5) [7](#0-6)

        &mut self,
        token_id: &AccountId,
        fast_transfer_id: &FastTransferId,

    ) -> U128 {

        if Self::is_refund_required(is_ft_transfer_call) {
            self.remove_fast_transfer(fast_transfer_id);
            amount
        } else {
            U128(0)
        }
    }

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn send_tokens(
        &self,
        token: AccountId,
        recipient: AccountId,
        amount: U128,
        msg: &str,
    ) -> Promise {
        let ft_transfer_call_gas = env::prepaid_gas()
            .saturating_sub(env::used_gas())
            .saturating_sub(SEND_TOKENS_CALLBACK_GAS) // TODO: not all send_tokens callbacks has the same gas.
            .saturating_sub(MINT_TOKEN_GAS)
            .min(FT_TRANSFER_CALL_GAS);

        let is_deployed_token = self.is_deployed_token(&token);

        if token == self.wnear_account_id && msg.is_empty() {
            // Unwrap wNEAR and transfer NEAR tokens
            ext_wnear_token::ext(self.wnear_account_id.clone())
                .with_static_gas(WNEAR_WITHDRAW_GAS)
                .with_attached_deposit(ONE_YOCTO)
                .near_withdraw(amount)
                .then(
                    Self::ext(env::current_account_id())
                        .with_static_gas(NEAR_WITHDRAW_CALLBACK_GAS)
                        .near_withdraw_callback(recipient, NearToken::from_yoctonear(amount.0)),
                )
        } else if is_deployed_token {
            let deposit = if msg.is_empty() {
                NO_DEPOSIT
            } else {
                ONE_YOCTO
            };

            require!(
                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
                BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
            );

            ext_token::ext(token)
                .with_attached_deposit(deposit)
                .with_static_gas(MINT_TOKEN_GAS.saturating_add(ft_transfer_call_gas))
                .mint(
                    recipient,
                    amount,
                    (!msg.is_empty()).then(|| msg.to_string()),
                )
        } else if msg.is_empty() {
            ext_token::ext(token)
                .with_attached_deposit(ONE_YOCTO)
                .with_static_gas(FT_TRANSFER_GAS)
                .ft_transfer(recipient, amount, None)
        } else {
            require!(
                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
                BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
            );

            ext_token::ext(token)
                .with_attached_deposit(ONE_YOCTO)
                .with_static_gas(ft_transfer_call_gas)
                .ft_transfer_call(recipient, amount, None, msg.to_string())
        }
