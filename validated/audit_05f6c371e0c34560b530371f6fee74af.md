Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is an unrestricted `external` function that unconditionally transfers the router's entire ETH balance to `msg.sender`. When a user sends `msg.value > amountIn` to a payable swap entry point with `tokenIn == WETH`, `pay()` wraps only the required amount and leaves the surplus as raw ETH on the contract. A back-running attacker can immediately call `refundETH()` in a separate transaction to steal that surplus, causing direct loss of user ETH principal.

## Finding Description
`refundETH()` carries no access control:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

The `receive()` guard at L32-34 only rejects direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable entry points.

When `exactInputSingle` (or any WETH-input swap) is called with `msg.value > amountIn`, the `pay()` WETH branch at L73-78 wraps exactly `value` (= `amountIn`) and transfers it to the pool, leaving `msg.value - amountIn` as raw ETH on the router:

```solidity
// PeripheryPayments.sol L73-78
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    }
```

Neither `exactInputSingle` (L67-86) nor `multicall` (L39-44) performs any post-execution ETH refund. The surplus sits on the router until any external caller claims it via `refundETH()`.

## Impact Explanation
Direct, unconditional loss of user ETH principal. The stolen amount equals `msg.value − amountIn` for every victim transaction that sends excess ETH and omits the `refundETH()` cleanup step. This meets the Critical/High direct loss of user principal threshold under the allowed impact gate.

## Likelihood Explanation
`refundETH()` is a well-known optional cleanup step that users and integrators routinely omit, especially in single-function (non-multicall) calls. The attack is a simple mempool back-run: observe any `exactInputSingle` / `exactInput` call where `msg.value > amountIn` and no `refundETH()` is included, then submit `router.refundETH()` immediately after. No capital, no permissions, no token approvals, and no pool setup are required. The attack is repeatable against every such victim transaction.

## Recommendation
Restrict `refundETH()` to only be callable from within an active `multicall` delegatecall context by adding a `msg.sender == address(this)` guard, **or** automatically refund excess ETH at the end of every payable entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), **or** record per-caller deposited amounts in transient storage and only refund the recorded depositor.

## Proof of Concept
