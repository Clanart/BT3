### Title
Migration v6 `ToParams()` Omits `HeaderHashNum`, Causing BLOCKHASH Opcode to Return 0x0 for All Historical Blocks — (`x/evm/migrations/v4/types/params.go`)

---

### Summary

`V4Params.ToParams()` constructs a `currenttypes.Params` struct without setting `HeaderHashNum` or `HistoryServeWindow`. After the v6 migration stores this struct, `GetParams()` returns `HeaderHashNum = 0` for every subsequent block. This causes two compounding effects in `BeginBlock` and `GetHashFn` that together make the BLOCKHASH opcode return `0x0` for every historical block query, reachable through any public EVM execution path.

---

### Finding Description

**Root cause — `ToParams()` omits `HeaderHashNum`:** [1](#0-0) 

The returned `currenttypes.Params` has `HeaderHashNum = 0` and `HistoryServeWindow = 0` (Go zero values). No subsequent migration (v7, v8) corrects this: [2](#0-1) [3](#0-2) 

`Validate()` does not enforce a non-zero `HeaderHashNum`; it only checks for int64 overflow, so 0 passes: [4](#0-3) 

**Effect 1 — `BeginBlock` self-deletes every stored hash:**

With `headerHashNum = 0`, `i = ctx.BlockHeight() - 0 = N`. Since `N > 0`, `DeleteHeaderHash(ctx, N)` is called immediately after `SetHeaderHash(ctx)` stores the hash for block N: [5](#0-4) 

Every block's hash is stored and then immediately erased, so `GetHeaderHash` always returns an empty hash.

**Effect 2 — `GetHashFn` lower-bound excludes all historical blocks:**

With `headerHashNum = 0`, the lower bound calculation yields `lower = upper - 0 = upper`. Inside the `upper > num64` branch, the condition `num64 >= lower` becomes `num64 >= upper`, which is always false. The `GetHistoricalInfo` fallback is therefore unreachable: [6](#0-5) 

Combined: `GetHeaderHash` returns empty (Effect 1), and the staking fallback is gated out (Effect 2). The function falls through to `return common.Hash{}` for every historical block query.

**Execution path:**

`eth_call` / `eth_sendRawTransaction` → `ApplyTransaction` → `EVMConfig` → `GetParams` (returns `HeaderHashNum=0`) → `NewEVM` passes `cfg.Params.HeaderHashNum` to `GetHashFn`: [7](#0-6) 

---

### Impact Explanation

Any contract that reads `BLOCKHASH(n)` for `n < currentBlock` receives `0x0` instead of the real hash. Contracts using BLOCKHASH for commit-reveal schemes, randomness seeds, or cross-block verification are fully predictable/manipulable by any unprivileged caller. The incorrect data is fed into EVM execution through the public JSON-RPC and gRPC paths (`eth_call`, `eth_sendRawTransaction`, `EthCall` gRPC).

---

### Likelihood Explanation

Every chain that ran the v4→v5→v6 migration sequence without subsequently issuing a `MsgUpdateParams` governance proposal to explicitly set `HeaderHashNum` is affected. The `Validate()` guard does not catch `HeaderHashNum = 0`, so the migration succeeds silently. No privileged access is required to trigger the impact — any `eth_call` to a BLOCKHASH-dependent contract suffices.

---

### Recommendation

In `V4Params.ToParams()`, set the missing fields to their defaults:

```go
return currenttypes.Params{
    // ... existing fields ...
    HeaderHashNum:      currenttypes.DefaultHeaderHashNum,
    HistoryServeWindow: currenttypes.DefaultHistoryServeWindow,
}
```

Additionally, add a migration step (e.g., v9) that reads the stored params and, if `HeaderHashNum == 0`, sets it to `DefaultHeaderHashNum` before re-persisting. Also strengthen `Validate()` to reject `HeaderHashNum = 0`.

---

### Proof of Concept

1. Deploy a chain and run migrations v4→v5→v6 (without governance update to `HeaderHashNum`).
2. Query `GetParams()` — observe `HeaderHashNum = 0`.
3. Deploy a contract:
   ```solidity
   function getHash(uint256 n) external view returns (bytes32) {
       return blockhash(n);
   }
   ```
4. Call `eth_call` targeting `getHash(block.number - 1)`.
5. Observe return value is `0x0000...0000` instead of the real parent block hash.
6. Confirm `BeginBlock` deletes the hash: inspect KV store for `SetHeaderHash`/`DeleteHeaderHash` at the same height.
