### Title
`execute_meta_tx_v0` Allows Any Contract to Invoke `__execute__` on Arbitrary Account Contracts Without Authorization, Enabling Direct Fund Theft — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall in the StarkNet OS allows **any contract** to invoke the `__execute__` entry point of **any arbitrary target contract** without authorization. Because `meta_tx_v0` bypasses `__validate__` entirely, and because standard StarkNet account contracts only perform signature verification inside `__validate__` (not `__execute__`), an attacker can execute arbitrary calls on behalf of any victim account — including draining all its funds — in a single transaction.

---

### Finding Description

`execute_meta_tx_v0` (lines 286–400 of `syscall_impls.cairo`) is dispatched unconditionally from `execute_syscalls` for any contract that emits a syscall with `META_TX_V0_SELECTOR`:

```cairo
assert selector = META_TX_V0_SELECTOR;
execute_meta_tx_v0(block_context=block_context, caller_execution_context=execution_context);
``` [1](#0-0) 

Inside `execute_meta_tx_v0`, the target `contract_address` is taken directly from the attacker-controlled request with no authorization check:

```cairo
local contract_address = request.contract_address;
local selector = request.selector;
...
if (selector != EXECUTE_ENTRY_POINT_SELECTOR) {
    write_failure_response(...);
    return ();
}
``` [2](#0-1) 

A new `TxInfo` is constructed with `account_contract_address = contract_address` (the victim), `nonce = 0`, and an attacker-supplied `signature_start`/`signature_end`. The execution context is built with `caller_address = ORIGIN_ADDRESS` (0), which is identical to what the OS uses for legitimate top-level calls:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    account_contract_address=contract_address,
    ...
    signature_start=request.signature_start,
    signature_end=request.signature_end,
    transaction_hash=meta_tx_hash,
    nonce=0,
    ...
);
...
execution_info=new ExecutionInfo(
    ...
    caller_address=ORIGIN_ADDRESS,
    contract_address=contract_address,
    selector=selector,
),
``` [3](#0-2) 

`contract_call_helper` is then called, which invokes `select_execute_entry_point_func` — executing the victim's `__execute__` entry point **directly, without ever calling `__validate__`**:

```cairo
contract_call_helper(
    remaining_gas=remaining_gas,
    block_context=block_context,
    execution_context=execution_context,
);
``` [4](#0-3) 

In the standard StarkNet account abstraction model, `__validate__` is the sole location where signature verification occurs. `__execute__` simply runs the provided calls. Since `meta_tx_v0` bypasses `__validate__`, the attacker-supplied (or empty) signature is never verified by the OS, and standard account contracts do not re-verify it inside `__execute__`.

Additionally, `nonce = 0` is hardcoded in the new `TxInfo` and `check_and_increment_nonce` is never called for the target contract, meaning the same meta-transaction payload can be replayed across multiple outer transactions.

The analog to the external report is direct: just as `handleReallocation` accepted an arbitrary `userAddress` without verifying the transaction initiator, `execute_meta_tx_v0` accepts an arbitrary `contract_address` without verifying the caller is authorized to act on behalf of that contract.

---

### Impact Explanation

An attacker can steal all funds from any account contract on the network:

- The attacker constructs calldata encoding token-transfer calls from the victim's account to the attacker's address.
- `meta_tx_v0` causes the victim's `__execute__` to run those calls with no signature check.
- The victim's account contract cannot distinguish this from a legitimate OS-initiated call (both arrive with `caller_address = 0`).

**Impact: Critical — Direct loss of funds.**

---

### Likelihood Explanation

The attack requires only:
1. Deploying a malicious contract (permissionless on StarkNet).
2. Sending a single invoke transaction that calls `meta_tx_v0` targeting the victim.

No privileged role, leaked key, or special network condition is needed. Every account contract holding funds is a potential victim. Likelihood is **High**.

---

### Recommendation

The OS must enforce that a caller is authorized to invoke `__execute__` on a target contract via `meta_tx_v0`. Concrete options:

1. **Require explicit opt-in**: Only allow `meta_tx_v0` to target contracts that have registered themselves as meta-transaction-aware (e.g., via a storage flag or a dedicated entry point).
2. **Restrict the caller**: Only allow `meta_tx_v0` from contracts that hold a specific role (e.g., a registered paymaster address stored in the block context).
3. **Enforce signature verification at the OS level**: Before calling `contract_call_helper`, verify the provided signature against the computed `meta_tx_hash` using the target contract's public key, analogous to how `run_validate` is called for regular invoke transactions. [5](#0-4) 

---

### Proof of Concept

```
1. Attacker deploys contract `Attack` on StarkNet.

2. Attacker sends an invoke transaction calling Attack.exploit(victim_address, victim_token, amount).

3. Inside Attack.exploit:
   - Constructs calldata: [transfer(attacker_address, amount)] targeting victim_token.
   - Issues meta_tx_v0 syscall:
       contract_address  = victim_address
       selector          = EXECUTE_ENTRY_POINT_SELECTOR  (__execute__)
       calldata          = [transfer call]
       signature         = []  (empty — never checked)

4. OS executes execute_meta_tx_v0:
   - Computes meta_tx_hash over (victim_address, __execute__, calldata, chain_id).
   - Builds TxInfo with account_contract_address=victim_address, nonce=0, signature=[].
   - Calls victim.__execute__([transfer call]) with caller_address=0.

5. victim.__execute__ runs the transfer without any signature check.
   Funds move from victim to attacker.

6. Attacker repeats with nonce=1, nonce=2, ... until victim is drained,
   since the victim's nonce is never incremented by meta_tx_v0.
```
