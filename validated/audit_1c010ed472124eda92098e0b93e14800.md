### Title
Malicious or Non-ETH-Accepting Recipient Address Permanently Freezes Native ETH in `finTransfer()` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` delivers native ETH to `payload.recipient` via a raw `.call{value: ...}("")`. If the recipient is a contract that reverts on ETH receipt, the entire transaction reverts. Because the recipient address is cryptographically bound in the MPC-signed payload, no alternative delivery is possible, and the bridged ETH is permanently unclaimable.

---

### Finding Description

In `OmniBridge.finTransfer()`, when `payload.tokenAddress == address(0)` (native ETH bridge-out), the contract performs:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
``` [1](#0-0) 

The `payload.recipient` is an EVM address embedded in the Borsh-encoded message that the MPC network signs:

```solidity
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

The signature is verified against `nearBridgeDerivedAddress`:

```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

This means the recipient address is immutable once the MPC signature is issued. There is no WETH fallback, no alternative claim path, and no admin recovery function in the contract.

The `completedTransfers[payload.destinationNonce] = true` write at line 287 is set before the ETH transfer: [4](#0-3) 

When the ETH transfer reverts, the entire transaction reverts, so the nonce is also rolled back to `false`. The transfer can be retried indefinitely but will always fail because the recipient is fixed in the signed payload.

There is no WETH wrapping fallback anywhere in the contract: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

When a user bridges native ETH from NEAR to EVM and specifies a contract address as the EVM recipient that does not implement `receive()` or `fallback()` (or explicitly reverts), the following occurs:

1. The user's NEAR-side tokens are burned/locked on the NEAR bridge.
2. `finTransfer()` is called on EVM with the signed payload.
3. The raw ETH `.call` to the contract recipient reverts.
4. `FailedToSendEther` is thrown; the entire transaction reverts.
5. The nonce is not consumed (rolled back), but the recipient is immutably fixed in the MPC signature.
6. No subsequent call to `finTransfer()` with any different recipient can pass signature verification.
7. The ETH held in the bridge contract is permanently unclaimable by the user.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

**Medium-High.** This is a realistic scenario for any user who:

- Bridges ETH to a multisig wallet address (e.g., Gnosis Safe) that does not accept plain ETH transfers.
- Bridges ETH to a DeFi protocol contract (e.g., a vault, a router) that lacks a `receive()` function.
- Bridges ETH to a contract that was deployed after the transfer was initiated and does not accept ETH.
- Bridges ETH to a contract that was upgraded (via proxy) to reject ETH after the transfer was signed.

The bridge UI or relayer infrastructure does not validate whether the EVM recipient can accept ETH before the NEAR-side tokens are burned. The user has no recourse once the MPC signature is issued.

---

### Recommendation

Mirror the fix described in the external report: wrap the ETH into WETH before delivery, or attempt the raw ETH transfer and fall back to WETH deposit on failure. This ensures the transfer always finalizes regardless of the recipient's ETH-acceptance behavior.

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) {
        // Fallback: wrap as WETH and transfer ERC-20 to recipient
        IWETH(wethAddress).deposit{value: payload.amount}();
        IERC20(wethAddress).safeTransfer(payload.recipient, payload.amount);
    }
}
```

Alternatively, always deliver as WETH (as the external report recommends), eliminating the ETH-acceptance dependency entirely.

---

### Proof of Concept

1. **Setup:** Deploy a malicious recipient contract on EVM that reverts on ETH receive:
   ```solidity
   contract MaliciousRecipient {
       receive() external payable { revert("no ETH"); }
   }
   ```

2. **Initiate transfer on NEAR:** User calls `init_transfer` on the NEAR bridge specifying `MaliciousRecipient`'s address as the EVM recipient, bridging native ETH (wNEAR → ETH).

3. **MPC signs:** The MPC network issues a signature over the payload including `payload.recipient = address(MaliciousRecipient)`.

4. **Attempt finalization:** Anyone calls `OmniBridge.finTransfer(signatureData, payload)` on EVM.

5. **Revert:** Execution reaches line 319:
   ```solidity
   (bool success, ) = payload.recipient.call{value: payload.amount}("");
   ```
   `MaliciousRecipient.receive()` reverts → `success = false` → `revert FailedToSendEther()`.

6. **Permanent lock:** The nonce rolls back to `false`. No alternative payload with a different recipient can pass `ECDSA.recover(...) != nearBridgeDerivedAddress`. The ETH in the bridge is permanently locked; the user's NEAR-side tokens are already burned. [1](#0-0) [4](#0-3) [6](#0-5)
