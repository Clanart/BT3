Audit Report

## Title
Unguarded `refundETH()` / `sweepToken()` / `unwrapWETH9()` allow any caller to drain ETH/token residue left by `pay()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` deposits only the exact swap amount as WETH when `nativeBalance >= value`, silently leaving the surplus native ETH on the router with no per-depositor accounting. `refundETH()`, `sweepToken()`, and `unwrapWETH9()` carry no access control and no depositor attribution, so any external caller can immediately drain that residue. A user who calls `exactOutputSingle{value: amountInMaximum}(...)` directly (not via `multicall`) loses `amountInMaximum − amountIn` ETH to the first caller of `refundETH()`.

## Finding Description
**ETH residue creation — `pay()` (L73-84):**
When `token == WETH` and `address(this).balance >= value`, the function wraps and forwards exactly `value` wei as WETH to the pool. The remaining `nativeBalance − value` native ETH is left on the router with no record of which address deposited it.

**Unrestricted drain — `refundETH()` (L58-63):**
`refundETH()` is `external payable` with no modifier or ownership check. It reads `address(this).balance` and transfers the entire amount to `msg.sender` — any EOA or contract, not the original depositor.

**Unrestricted drain — `sweepToken()` / `unwrapWETH9()` (L37-55):**
Both functions are `public payable` with a caller-chosen `recipient` parameter and no access control, extending the same theft to ERC-20 and WETH residue.

**Trigger path — `exactOutputSingle` (L130-147):**
`exactOutputSingle` is `external payable`. The user sends `msg.value = amountInMaximum`; the pool determines `amountIn < amountInMaximum`; the callback invokes `pay(WETH, user, pool, amountIn)`, which wraps only `amountIn`. The surplus `msg.value − amountIn` remains on the router. The function returns without calling `refundETH()`. The `receive()` guard (L32-34) only blocks plain ETH transfers with no calldata; it does not affect ETH sent alongside a `payable` function call.

**Existing guard — `multicall` (L39-44):**
`multicall` allows batching `exactOutputSingle` + `refundETH()` in one transaction, which would protect the user. However, `exactOutputSingle` is independently callable as `external payable` with no enforcement that a refund step follows, so any direct caller is unprotected.

## Impact Explanation
Direct loss of user-owned ETH principal. For `exactOutputSingle` with a 2× `amountInMaximum` (common slippage buffer), up to 50 % of committed ETH is stolen. At current ETH prices this trivially exceeds the Critical threshold (>20 % loss, >$100). The same path applies to any ERC-20 or WETH residue via `sweepToken` / `unwrapWETH9`.

## Likelihood Explanation
No privileged access is required — a single `refundETH()` call from any EOA suffices. `exactOutputSingle` is the canonical "exact output" entry point; users routinely set generous `amountInMaximum` values to avoid slippage reverts. Any user who calls `exactOutputSingle` directly (rather than via a `multicall` that also encodes `refundETH`) is vulnerable. The attack is MEV-bot-friendly: the attacker can include `refundETH()` in the very next block after the victim's swap is confirmed, or sandwich it in the same block.

## Recommendation
1. **Auto-refund inside swap functions**: after `pay()` consumes the required WETH amount, call `_transferETH(msg.sender, address(this).balance)` at the end of `exactInputSingle` and `exactOutputSingle` before returning.
2. **Remove caller-chosen `recipient`** from `sweepToken` and `unwrapWETH9`, or restrict them so only the address that deposited in the same multicall context can invoke them.
3. **Track per-depositor balances**: record `msg.value` credited to `msg.sender` at each payable entry point and allow only that address to reclaim its share via `refundETH`.

## Proof of Concept
```solidity
// Victim transaction
router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
    pool:            address(wethPool),
    tokenIn:         WETH,
    tokenOut:        token1,
    zeroForOne:      true,
    amountOut:       1_000,
    amountInMaximum: 2 ether,
    recipient:       victim,
    deadline:        block.timestamp + 60,
    priceLimitX64:   0,
    extensionData:   ""
}));
// Pool charges amountIn = 1 ether.
// pay() at L75-77: deposits 1 ether as WETH → pool.
// 1 ether remains on router, unattributed.

// Attacker transaction (next block or same block via MEV)
// address(router).balance == 1 ether
router.refundETH();   // attacker receives 1 ether; victim lost 1 ether
```
