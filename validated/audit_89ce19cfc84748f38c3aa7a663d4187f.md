### Title
`deployToken` / `deploy_token` Signature Lacks Chain-ID Binding, Enabling Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`, `solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs`)

---

### Summary

The Borsh-encoded message signed by the NEAR MPC for `deployToken` / `deploy_token` does not include a destination chain identifier. Because the same ECDSA key (`nearBridgeDerivedAddress` / `omni_bridge_derived_address` / `derived_near_bridge_address`) is used across all chains, a signature produced for one chain is cryptographically valid on every other chain. An unprivileged attacker who observes a legitimate `deployToken` call on EVM can immediately replay the same `(signatureData, metadata)` tuple against the Starknet or Solana bridge, deploying the token on those chains without NEAR's explicit authorization.

---

### Finding Description

**`finTransfer` is protected; `deployToken` is not.**

The EVM `finTransfer` Borsh payload explicitly encodes `omniBridgeChainId` twice (as destination chain and recipient chain), binding the signature to a single chain:

```solidity
bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
Borsh.encodeUint64(payload.destinationNonce),
bytes1(payload.originChain),
Borsh.encodeUint64(payload.originNonce),
bytes1(omniBridgeChainId),          // ← destination chain bound
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),          // ← recipient chain bound
Borsh.encodeAddress(payload.recipient),
``` [1](#0-0) 

The EVM `deployToken` Borsh payload contains **no chain identifier**:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
    // ← no omniBridgeChainId
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

The Starknet `deploy_token` hashes the same chain-ID-free Borsh payload (confirmed by the test helper which encodes only `PayloadType::MetadataPayload`, `token`, `name`, `symbol`, `decimals`): [3](#0-2) 

And verifies it against the same derived address:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(), signature);
``` [4](#0-3) 

The Solana `deploy_token` follows the same pattern — `verify_signature` hashes `payload.serialize_for_near(())` (no chain ID) and checks against `derived_near_bridge_address`: [5](#0-4) [6](#0-5) 

Because all three chains verify the same byte sequence against the same NEAR MPC-derived key, a signature produced for any one chain is accepted by all others.

---

### Impact Explanation

An attacker who observes a legitimate `deployToken` transaction on EVM (signature is public calldata) can:

1. Extract `(signatureData, metadata)` from the EVM transaction.
2. Submit the identical pair to Starknet's `deploy_token` or Solana's `deploy_token`.
3. The token is deployed on the second chain **without NEAR explicitly authorizing deployment there**.

Concrete harms:
- **State desync**: NEAR's bridge contract tracks which tokens are deployed on which chains. A replay-deployed token is unknown to NEAR's state, so NEAR cannot route `fin_transfer` messages to that chain for that token through the normal flow.
- **Permanent deployment block**: When NEAR later tries to deploy the token legitimately on the replayed chain, it fails with `ERR_TOKEN_ALREADY_DEPLOYED` (Starknet) or an Anchor `init` constraint violation (Solana). There is no admin function on Starknet to remove a deployed token, making this permanent.
- **Unauthorized token deployment**: Fits the allowed impact "High — Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment." [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

- The attacker needs zero privileges — only the ability to read public on-chain calldata.
- Any `deployToken` call on EVM is immediately replayable on Starknet and Solana.
- The attack is deterministic and requires no timing window or race condition.
- Likelihood: **High**.

---

### Recommendation

Include the destination chain ID in the `deployToken` / `deploy_token` Borsh-encoded payload, exactly as `finTransfer` already does. For example, on EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ← add destination chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same change to the Starknet `MetadataPayload.to_borsh()` and the Solana `DeployTokenPayload::serialize_for_near()`. The NEAR MPC must include the target chain ID when signing metadata payloads, and each chain must verify that the chain ID in the payload matches its own `omniBridgeChainId`.

---

### Proof of Concept

1. NEAR MPC signs `keccak256(Borsh(PayloadType::Metadata | "usdc.near" | "USD Coin" | "USDC" | 6))` for EVM deployment.
2. Relayer submits `deployToken(sig, metadata)` on EVM — succeeds, USDC deployed at `0xABC`.
3. Attacker reads `sig` and `metadata` from EVM calldata (public).
4. Attacker calls Starknet `deploy_token(sig, metadata)`:
   - `_verify_borsh_signature` hashes the same Borsh bytes → same hash → same valid signature → passes.
   - `existing_token.is_zero()` → true (not yet deployed on Starknet) → passes.
   - USDC is deployed on Starknet without NEAR's authorization.
5. NEAR later attempts to deploy USDC on Starknet → `ERR_TOKEN_ALREADY_DEPLOYED` → permanently blocked. [9](#0-8) [10](#0-9)

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
