### Title
Non-Atomic Burn-and-Mint in `swap_migrated_token` Allows Unbacked New Token Minting - (File: near/omni-bridge/src/lib.rs)

### Summary

`swap_migrated_token` in `near/omni-bridge/src/lib.rs` uses `burn.and(mint)` with no callback to verify both sub-promises succeeded. In NEAR's promise model, `Promise::and()` schedules both receipts independently — if the `burn` receipt fails, the `mint` receipt still executes. Because `ft_on_transfer` returns `U128(0)` (consuming all transferred tokens) before the promises resolve, a burn failure leaves old tokens stranded in the bridge while new tokens are minted to the user, creating unbacked supply.

### Finding Description

The flow is triggered by any user calling `ft_transfer_call` on the old token contract with `msg: "SwapMigratedToken"`:

**`ft_on_transfer` dispatch** — `near/omni-bridge/src/lib.rs` lines 275–279: [1](#0-0) 

The function immediately returns `U128(0)`, meaning the bridge keeps all transferred tokens. The promise is detached with no callback.

**`swap_migrated_token`** — `near/omni-bridge/src/lib.rs` lines 2738–2753: [2](#0-1) 

`burn.and(mint)` creates two independent receipts. NEAR's `Promise::and()` does **not** provide atomicity: each sub-promise executes as a separate receipt regardless of the other's outcome. There is no `.then(callback)` to inspect results and roll back.

**`burn` implementation** — `near/omni-token/src/lib.rs` lines 146–151:


The burn calls `internal_withdraw` on the bridge's balance in the old token contract. This can fail if the old token contract is paused, has been upgraded with changed behavior, or if gas is exhausted for that receipt.

**`mint` implementation** — `near/omni-token/src/lib.rs` lines 127–143:


The mint deposits tokens directly to the user's account. It executes independently of the burn receipt's outcome.

The gas constants allocated are minimal: [3](#0-2) 

No explicit gas is attached to the `burn` or `mint` calls inside `swap_migrated_token`, making gas exhaustion for the burn receipt a realistic failure mode.

### Impact Explanation

**Critical — unauthorized mint of bridged assets / broken bridge collateralization.**

If the `burn` receipt fails while the `mint` receipt succeeds:
- The user receives `amount` of new tokens.
- The old tokens remain in the bridge contract's balance (they were transferred in via `ft_transfer_call` and the bridge returned `U128(0)`).
- The new token supply is now unbacked: `amount` new tokens exist without a corresponding destruction of old tokens.

This directly breaks bridge collateralization. The old tokens stranded in the bridge can be recovered by the DAO or re-used in further swaps, compounding the inflation.

If the `mint` receipt fails while the `burn` receipt succeeds:
- Old tokens are destroyed.
- New tokens are never minted.
- The user permanently loses their funds (irrecoverable lock).

### Likelihood Explanation

The trigger is reachable by any unprivileged user holding old (migrated) tokens — no special role is required. Failure scenarios for the burn receipt include:

1. The old token contract is paused for maintenance at the time of the swap.
2. The old token contract has been upgraded and its `burn` function behavior changed.
3. Gas exhaustion: `swap_migrated_token` does not set explicit gas on either sub-promise, and the remaining gas after `ft_on_transfer` dispatch may be insufficient for the burn receipt in edge cases.

Scenario 1 is operationally realistic during any maintenance window following a token migration.

### Recommendation

Replace the fire-and-forget `burn.and(mint)` with a sequenced, callback-verified pattern:

```rust
fn swap_migrated_token(
    &mut self,
    sender_id: AccountId,
    old_token: AccountId,
    amount: U128,
) -> Promise {
    let new_token = self
        .migrated_tokens
        .get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);

    ext_token::ext(old_token)
        .with_static_gas(BURN_TOKEN_GAS)
        .burn(amount)
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(MINT_TOKEN_GAS)
                .swap_migrated_token_callback(sender_id, new_token, amount),
        )
}

#[private]
pub fn swap_migrated_token_callback(
    &mut self,
    sender_id: AccountId,
    new_token: AccountId,
    amount: U128,
) -> Promise {
    require!(
        env::promise_result_count() == 1
            && matches!(env::promise_result(0), PromiseResult::Successful(_)),
        "Burn failed; aborting mint"
    );
    ext_token::ext(new_token)
        .with_static_gas(MINT_TOKEN_GAS)
        .mint(sender_id, amount, None)
}
```

This ensures the mint only executes after a confirmed successful burn, preserving the invariant that new token supply is always backed by destroyed old token supply.

### Proof of Concept

1. DAO calls `migrate_deployed_token(origin_chain, old_token, new_token)` — registers the migration. [4](#0-3) 

2. User calls `old_token.ft_transfer_call(bridge, amount, msg="SwapMigratedToken")`.

3. Bridge's `ft_on_transfer` returns `U128(0)` — bridge now holds `amount` of old tokens. [1](#0-0) 

4. `swap_migrated_token` fires `burn.and(mint)` detached — two independent receipts scheduled. [5](#0-4) 

5. Old token contract is paused (or burn receipt runs out of gas) → burn receipt fails.

6. Mint receipt executes independently → user receives `amount` of new tokens.

7. Old tokens remain in bridge. New token total supply is inflated by `amount` without backing.
