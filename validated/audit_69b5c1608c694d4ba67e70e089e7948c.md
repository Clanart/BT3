### Title
Integer Division in `NewDynamicFeeChecker` Floors `feeCap` to Zero, Enabling Fee-Free Cosmos Transactions When `NoBaseFee=true` — (File: `ante/evm/fee_checker.go`)

---

### Summary

In `NewDynamicFeeChecker`, the per-gas fee cap for a Cosmos tx is computed via integer division: `feeCap = fee / gas`. When `fee < gas`, this floors to zero. When London hardfork is enabled but `NoBaseFee = true`, `GetBaseFee` returns `big.NewInt(0)` (not `nil`), so the `nil`-guard fallback to `checkTxFeeWithValidatorMinGasPrices` is skipped. The guard `feeCap < baseFeeInt` then evaluates to `0 < 0 = false`, and `EffectiveGasPrice(0, 0, maxPriorityPrice)` returns `0`. The `DeductFeeDecorator` therefore deducts **zero** from the sender, even when the sender provided a non-zero fee that already passed the `MinGasPriceDecorator` check.

---

### Finding Description

**Step 1 — `GetBaseFee` returns `0` (not `nil`) when `NoBaseFee=true` and London is active.**

`x/evm/keeper/keeper.go` documents this explicitly:

```
// GetBaseFee returns current base fee, return values:
// - `nil`: london hardfork not enabled.
// - `0`: london hardfork enabled but feemarket is not enabled.
// - `n`: both london hardfork and feemarket are enabled.
``` [1](#0-0) 

The keeper test confirms: `{"enable london HF, not enable feemarket", true, false, big.NewInt(0)}`. [2](#0-1) 

**Step 2 — `NewDynamicFeeChecker` only falls back when `baseFee == nil`.**

```go
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
if baseFee == nil {
    // london hardfork is not enabled: fallback to min-gas-prices logic
    return checkTxFeeWithValidatorMinGasPrices(ctx, feeTx)
}
```

When `NoBaseFee=true` and London is active, `baseFee = 0` (not `nil`), so the fallback is **not** taken. [3](#0-2) 

**Step 3 — `feeCap` is computed with integer (truncating) division.**

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

`sdkmath.Int.Quo` is integer division. For any tx where `fee < gas` (e.g., `fee = 21`, `gas = 21000`), `feeCap = 0`. [4](#0-3) 

**Step 4 — The fee guard becomes `0 < 0 = false` and is bypassed.**

```go
if feeCap.LT(baseFeeInt) {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, ...)
}
```

With `feeCap = 0` and `baseFeeInt = 0`, the condition is always `false`. The check is silently skipped. [5](#0-4) 

**Step 5 — `EffectiveGasPrice` returns `0`, so `effectiveFee = 0`.**

```go
effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
effectiveFee := sdk.Coins{{Denom: denom, Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas))}}
```

`EffectiveGasPrice(0, 0, maxPriorityPrice) = min(0 + maxPriorityPrice, 0) = 0`. Therefore `effectiveFee = 0 * gas = 0`. The `DeductFeeDecorator` deducts this zero amount from the sender. [6](#0-5) 

**Step 6 — `MinGasPriceDecorator` passes but deduction is zero.**

`MinGasPriceDecorator` enforces `fee >= ceil(MinGasPrice * gas)`. A sender can satisfy this (e.g., `MinGasPrice = 0.001`, `gas = 21000`, `fee = 21`) while still having `fee < gas`, causing `feeCap = 0` and `effectiveFee = 0`. The sender passes the minimum-fee gate but pays nothing. [7](#0-6) 

---

### Impact Explanation

This is a **fee mis-accounting bug** in the ante handler path. Any Cosmos tx with `fee < gas` submitted to a chain configured with `NoBaseFee=true` (feemarket disabled) and London hardfork active will have its fee deduction silently zeroed out. If `MinGasPrice > 0`, the `MinGasPriceDecorator` still passes (the user provides the required minimum fee), but the `DeductFeeDecorator` deducts zero. The fee collector module receives nothing, and the sender retains their funds. This matches the allowed High impact: *"fee market or ante handler bug that permits valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

The configuration `NoBaseFee=true` with London hardfork enabled and `MinGasPrice > 0` is a realistic production deployment (a chain that wants a minimum fee floor but no dynamic EIP-1559 base fee). Any unprivileged user can trigger this by submitting a Cosmos tx with `fee < gas` (e.g., `fee = 21`, `gas = 21000`). No special permissions, validator access, or governance action is required.

---

### Recommendation

Replace the integer division for `feeCap` with decimal arithmetic to avoid flooring to zero:

```go
// Instead of:
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))

// Use ceiling division or LegacyDec:
feeCapDec := sdkmath.LegacyNewDecFromInt(fee).QuoInt64(int64(gas))
// then compare feeCapDec against baseFeeInt as a Dec
```

Alternatively, add an explicit guard: if `baseFeeInt.IsZero()` and `fee.IsPositive()`, use `fee` directly as the effective fee rather than computing `effectivePrice * gas` from a zeroed `feeCap`.

---

### Proof of Concept

```
Configuration:
  NoBaseFee = true
  London hardfork enabled (LondonBlock = 0)
  MinGasPrice = 0.001 (feemarket param)

Attacker submits Cosmos tx:
  fee  = 21 aevmos   (satisfies MinGasPriceDecorator: 0.001 * 21000 = 21)
  gas  = 21000

Execution trace:
  1. MinGasPriceDecorator: requiredFee = ceil(0.001 * 21000) = 21 ≤ 21 → PASS
  2. NewDynamicFeeChecker:
       baseFee = GetBaseFee(...) = 0  (NoBaseFee=true, London active)
       feeCap  = 21 / 21000 = 0      (integer division)
       0 < 0   → false               (fee guard bypassed)
       effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = 0
       effectiveFee   = 0 * 21000 = 0
  3. DeductFeeDecorator deducts 0 from sender.

Result: Sender retains 21 aevmos; fee collector receives 0.
```
