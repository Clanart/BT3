### Title
Self-Destructed Contract Account Not Deleted from Cosmos Store Due to `Finalise`/`Commit` Ordering Bug - (File: `x/evm/statedb/statedb.go`)

### Summary
When a contract executes `SELFDESTRUCT`, go-ethereum's EVM calls `StateDB.Finalise` which removes the self-destructed object from the in-memory `stateObjects` cache. Subsequently, `StateDB.Commit` skips the object because it is no longer present in `stateObjects`, and `keeper.DeleteAccount` is never called. The contract's code, storage, and auth module account remain permanently in the Cosmos store, leaving the contract callable after self-destruction — incorrect EVM state is committed to the chain.

### Finding Description

The vulnerability class is a **stateDB commit/revert bug**: a cleanup operation (account deletion) is silently skipped because the object it targets was already removed from the in-memory cache by a prior step, exactly analogous to M-17's `delete stack[position]` operating on a memory copy with no effect on storage.

**Step 1 — `Finalise` removes self-destructed objects from `stateObjects`:**

`StateDB.Finalise` (called by go-ethereum's `StateTransition.Execute` at the end of EVM execution) iterates over dirty addresses and deletes self-destructed objects from the in-memory `s.stateObjects` map:

```go
// x/evm/statedb/statedb.go:890-899
func (s *StateDB) Finalise(deleteEmptyObjects bool) {
    for addr := range s.journal.dirties {
        obj, exist := s.stateObjects[addr]
        if !exist {
            continue
        }
        if obj.selfDestructed || (deleteEmptyObjects && obj.empty()) {
            delete(s.stateObjects, obj.address)  // object removed from cache
        }
    }
}
``` [1](#0-0) 

**Step 2 — `Commit` silently skips objects not in `stateObjects`:**

`StateDB.Commit` (called by Ethermint's keeper after EVM execution) iterates over the same dirty addresses from the journal, but looks each one up in `s.stateObjects`. Because `Finalise` already deleted self-destructed objects from that map, the lookup fails and the object is skipped via `continue` — `keeper.DeleteAccount` is never reached:

```go
// x/evm/statedb/statedb.go:795-844
for _, addr := range s.journal.sortedDirties() {
    obj, exist := s.stateObjects[addr]
    if !exist {
        continue  // ← self-destructed objects silently skipped here
    }
    if obj.selfDestructed {
        // ... keeper.DeleteAccount(cacheCtx, obj.Address()) — NEVER CALLED
    }
}
``` [2](#0-1) 

**Step 3 — `DeleteAccount` is the only cleanup path:**

`keeper.DeleteAccount` is the sole function responsible for removing the contract's code, storage, and auth module account from the Cosmos KV store. It is only invoked inside the `obj.selfDestructed` branch of `Commit`. Since that branch is never reached, the Cosmos store retains the full contract state indefinitely. [3](#0-2) 

**Step 4 — `getStateObject` will find the stale account on subsequent calls:**

After the buggy commit, `getStateObject` falls through to `keeper.GetAccount` when the address is not in the live `stateObjects` cache (which is fresh for each transaction). Because `DeleteAccount` was never called, `keeper.GetAccount` returns the old account, the contract code is loaded, and the EVM executes the self-destructed contract as if it were alive:

```go
// x/evm/statedb/statedb.go:292-306
func (s *StateDB) getStateObject(addr common.Address) *stateObject {
    if obj := s.stateObjects[addr]; obj != nil {
        return obj
    }
    account := s.keeper.GetAccount(s.ctx, addr)  // ← finds stale account
    if account == nil {
        return nil
    }
    obj := newObject(s, addr, account)
    s.setStateObject(obj)
    return obj
}
``` [4](#0-3) 

The `HookedStateDB.Finalise` wrapper correctly delegates to `s.inner.Finalise`, so the bug is present in both execution paths: [5](#0-4) 

### Impact Explanation

This is a **High** EVM state transition bug. After a `SELFDESTRUCT` opcode executes:

- The contract's balance is correctly burned (via `SubBalance` → `ExecuteNativeAction`).
- The contract's code, storage, and auth module account are **not** deleted from the Cosmos KV store.
- In any subsequent transaction, `getStateObject` finds the stale account via `keeper.GetAccount`, loads the old code, and the EVM executes the contract as if it were never destroyed.
- This constitutes incorrect EVM state being committed to the chain: a self-destructed contract remains callable and its storage remains readable/writable, violating the EVM specification and enabling unexpected state mutations by any caller.

This fits the allowed impact: *"EVM state transition bug that permits invalid transactions to commit."*

### Likelihood Explanation

The trigger is any transaction that causes the `SELFDESTRUCT` (or `SELFDESTRUCT6780` for same-transaction-created contracts) opcode to execute. This is a standard EVM opcode reachable by any unprivileged user deploying or calling a contract. The execution path — go-ethereum EVM calls `Finalise`, then Ethermint's keeper calls `Commit` — is the normal, deterministic block-processing path for every EVM transaction.

### Recommendation

The root cause is that `Commit` relies on `s.stateObjects` to identify self-destructed accounts, but `Finalise` removes them from that map before `Commit` runs. Two viable fixes:

1. **Track self-destructed addresses separately**: Maintain a dedicated `selfDestructed map[common.Address]struct{}` that is populated in `SelfDestruct` and is not cleared by `Finalise`. In `Commit`, iterate this map to call `DeleteAccount` regardless of whether the object is still in `stateObjects`.

2. **Do not remove self-destructed objects in `Finalise`**: Remove the `delete(s.stateObjects, obj.address)` call for `obj.selfDestructed` in `Finalise`, preserving the object in the cache so `Commit` can find and process it. Only remove empty (non-self-destructed) objects in `Finalise`.

### Proof of Concept

1. Deploy a contract `Bomb` with a `kill()` function that calls `SELFDESTRUCT(msg.sender)`.
2. Call `kill()` in a transaction. During EVM execution:
   - `SelfDestruct` is called → `selfDestructed = true`, balance burned, `selfDestructChange` + `nativeChange` appended to journal.
   - go-ethereum calls `statedb.Finalise(true)` → `delete(s.stateObjects, bombAddr)`.
3. Ethermint's keeper calls `statedb.Commit()`:
   - `sortedDirties()` includes `bombAddr`.
   - `s.stateObjects[bombAddr]` → `exist = false` → `continue`.
   - `keeper.DeleteAccount` is never called.
4. In the next block, call `bombAddr` with any calldata:
   - `getStateObject(bombAddr)` → `keeper.GetAccount` returns the stale account.
   - The old bytecode is loaded and executed.
   - The contract responds as if it was never destroyed, with its original storage intact.
