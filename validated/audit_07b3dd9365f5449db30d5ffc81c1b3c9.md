### Title
Unauthenticated Bootstrap Path in `execute_declare_transaction` Allows Any Sender to Declare Classes Without Validation - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS contains a privileged "bootstrap" code path that is gated only on the felt-literal value `'BOOTSTRAP'` as the `sender_address`, with `nonce == 0`, `version == 3`, and `max_possible_fee == 0`. Because `'BOOTSTRAP'` is an ordinary felt value with no cryptographic binding to any privileged key or role, any unprivileged transaction sender can craft a v3 declare transaction that satisfies all four conditions and thereby declare an arbitrary class hash into the global class registry — completely bypassing `__validate_declare__` signature verification, nonce management, and fee payment.

---

### Finding Description

Inside `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the OS executes the following branch:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the OS skips:
- `check_and_increment_nonce` (line 779) — no replay protection
- `non_reverting_select_execute_entry_point_func` for `__validate_declare__` (line 804) — no signature check
- `charge_fee` (line 822) — no fee deducted [2](#0-1) 

The four conditions are all trivially satisfiable by any user:

1. **`sender_address == 'BOOTSTRAP'`** — `'BOOTSTRAP'` is a Cairo short-string felt literal. In StarkNet v3 transactions, `sender_address` is a user-supplied field included in the transaction hash. Any user can set their transaction's `sender_address` to the felt encoding of `'BOOTSTRAP'`.

2. **`tx_info.nonce == 0`** — The nonce field is user-supplied in the transaction. Setting it to `0` is trivial.

3. **`tx_info.version == 3`** — `get_account_tx_common_fields` hardcodes `version=3` for all v3 transactions; this is the standard version. [3](#0-2) 

4. **`max_possible_fee == 0`** — `compute_max_possible_fee` returns the sum of `max_amount * max_price_per_unit` for all three resource bounds. Setting all `max_amount` or `max_price_per_unit` fields to `0` yields a total of `0`. [4](#0-3) 

There is no cryptographic check, no on-chain role registry, and no sequencer-signed attestation that the sender is actually the privileged bootstrapper. The `'BOOTSTRAP'` string is not a protected address — it is just a number.

The `sender_address` field is part of the transaction hash computation via `hash_tx_common_fields`, so the transaction hash is consistent with `sender_address = 'BOOTSTRAP'`. The OS hint `%{ AssertTransactionHash %}` only verifies internal consistency, not that the sender is authorized. [5](#0-4) 

---

### Impact Explanation

**Direct loss of funds (Critical) and network-level fee accounting corruption (High):**

1. **Fee bypass / direct loss of funds for sequencer**: The attacker declares classes for free. The sequencer receives zero fee for processing these transactions. At scale, an attacker can flood the block with zero-cost class declarations, starving the sequencer of revenue and disrupting the economic model that sustains block production.

2. **Unauthorized class registration**: The attacker can register an arbitrary `(class_hash → compiled_class_hash)` mapping. If the attacker front-runs a legitimate class declaration, the legitimate declare will fail (the `prev_value=0` guard at line 770 enforces single-declaration). This can permanently block legitimate class deployments.

3. **Malicious class injection**: The attacker can declare a class hash pointing to a malicious `compiled_class_hash`. Any contract that subsequently calls `replace_class` to adopt this class hash, or any new contract deployed with this class hash, will execute the attacker's code — enabling direct theft of funds from those contracts.

4. **Signature verification bypass**: The entire `__validate_declare__` entry point is skipped, meaning the OS accepts a state-changing transaction with no cryptographic proof of authorization from the account owner.

---

### Likelihood Explanation

**High.** The conditions are trivially satisfiable by any user who can submit a StarkNet transaction. No special privileges, leaked keys, or trusted roles are required. The attacker only needs to:
- Set `sender_address` to the felt value of `'BOOTSTRAP'`
- Set `nonce = 0`
- Set all resource bounds to `0`
- Provide a valid Sierra class hash pre-image (verifiable off-chain)

The transaction hash will be valid and consistent. A sequencer that does not independently filter this address (which is not guaranteed by the OS protocol itself) will include it in a block, and the OS proof will accept it.

---

### Recommendation

Replace the felt-literal identity check with a cryptographically enforced mechanism. Options include:

1. **Remove the bootstrap path entirely** and require all class declarations to go through the standard `__validate_declare__` flow, even during bootstrapping (using a pre-deployed bootstrap account with a known key).

2. **Bind the bootstrap privilege to the sequencer's signing key**: require the bootstrap transaction to carry a valid signature from the sequencer's public key (already available in `os_global_context.public_keys`), and verify it inside the OS before taking the bypass path.

3. **Use a block-number-gated flag**: allow the bootstrap path only in block 0 or a configurable genesis block range, enforced by `block_context.block_info_for_execute.block_number == 0`, so the window closes permanently after genesis.

---

### Proof of Concept

An attacker constructs and submits the following v3 declare transaction:

```
sender_address  = felt_encoding('BOOTSTRAP')  // = 0x424f4f545354524150 (ASCII)
nonce           = 0
version         = 3
l1_gas          = ResourceBounds { max_amount: 0, max_price_per_unit: 0 }
l2_gas          = ResourceBounds { max_amount: 0, max_price_per_unit: 0 }
l1_data_gas     = ResourceBounds { max_amount: 0, max_price_per_unit: 0 }
tip             = 0
class_hash      = <any valid Sierra class hash>
compiled_class_hash = <attacker-chosen CASM hash>
```

The OS evaluates the branch at line 764:
- `sender_address == 'BOOTSTRAP'` → **true**
- `tx_info.nonce == 0` → **true**
- `tx_info.version == 3` → **true**
- `compute_max_possible_fee(tx_info) == 0` → **true** (all bounds are zero)

The OS executes `dict_update` to register `class_hash → compiled_class_hash` and returns immediately, skipping all validation and fee logic. The class is now permanently registered in the protocol state with no fee paid and no signature verified. [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
