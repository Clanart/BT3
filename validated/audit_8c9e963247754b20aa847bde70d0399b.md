### Title
Permanent Catchup Livelock: `block_catch_up_postprocess` Errors Unconditionally Re-queue the Same Block Forever — (`chain/chain/src/chain.rs`)

### Summary

In `catchup_blocks_step`, when `block_catch_up_postprocess` returns any error, the block is unconditionally pushed back onto `pending_blocks` and re-scheduled for the next catchup tick — forever. There is no retry limit, no error classification, and no escape hatch. If the error is deterministic (e.g., `StorageError::MissingTrieValue` from a corrupted or incomplete state-sync result, or `InvalidStateRoot` from a mismatched `ChunkExtra`), the block can never succeed, `BlocksCatchUpState::is_finished()` never returns `true`, `finish_catchup_blocks` is never called, and `DBCol::StateDlInfos` is never cleared. The node is permanently stuck: it cannot complete catchup for the epoch, cannot mark the epoch's blocks as done, and — because the `StateSyncInfo` entry persists — `run_catchup` re-enters the same loop on every 100 ms tick for the lifetime of the process.

### Finding Description

`catchup_blocks_step` in `chain/chain/src/chain.rs` drives the block-catchup phase of shard state sync. Its error-handling arm for `block_catch_up_postprocess` is:

```rust
Err(err) => {
    tracing::error!(target: "chain", ?err, "error processing block during catch up, retrying");
    blocks_catch_up_state.pending_blocks.push(queued_block);
}
```

The block is re-queued with no retry counter, no error-type check, and no circuit-breaker. On the next call to `catchup_blocks_step` (100 ms later, from `ClientActor::catchup`), the block is dequeued from `pending_blocks`, re-preprocessed via `apply_chunks_preprocessing`, and re-sent to `SyncJobsActor` as a new `BlockCatchUpRequest`. The response comes back, `block_catch_up_postprocess` is called again, fails again, and the cycle repeats indefinitely.

`block_catch_up_postprocess` can fail deterministically in several ways:

1. **`StorageError::MissingTrieValue`** — if the state-sync applied parts that reference trie nodes not present in the local DB (e.g., a partial or corrupted state-sync), every attempt to apply the chunk for the new shard will fail with the same missing-node error.
2. **`InvalidStateRoot`** — if the `ChunkExtra` written during state-sync finalization (`set_state_finalize`) contains a state root that does not match what the chunk header expects, `validate_chunk_with_chunk_extra_and_receipts_root` returns `Error::InvalidStateRoot` on every attempt.
3. **`ReshardingError`** — if `start_resharding` inside `block_catch_up_postprocess` fails (e.g., memtrie not loaded), the error is returned and the block is re-queued.

None of these errors are transient. The `is_bad_data` classification in `chain-primitives/src/error.rs` correctly marks `StorageError` as `false` (not bad data), but `catchup_blocks_step` does not consult `is_bad_data` at all — it treats every error identically.

The consequence is that `BlocksCatchUpState::is_finished()` — which requires `pending_blocks`, `scheduled_blocks`, and `processed_blocks` to all be empty — can never return `true`. Therefore `finish_catchup_blocks` is never called, `DBCol::StateDlInfos` retains the entry for the epoch, and `run_catchup` re-enters the same `CatchupState` on every tick.

### Impact Explanation

A validator node that enters this state:
- Cannot complete catchup for the new shard it is supposed to track in epoch T+1.
- Cannot produce or validate chunks for that shard in epoch T+1.
- Will be kicked out of the validator set for missing chunks, losing all staking rewards.
- The `StateSyncInfo` entry in `DBCol::StateDlInfos` is never removed, so the node burns CPU and I/O on every 100 ms tick re-applying the same failing block indefinitely.
- Recovery requires a manual node restart with a full data wipe or `undo-block`, as there is no in-process escape.

This is a **High** severity issue: a validator node is permanently disabled for the affected shard, with direct economic loss (missed rewards, potential kickout).

### Likelihood Explanation

The trigger is a deterministic `block_catch_up_postprocess` failure after state-sync completes. This can arise from:

- A corrupted or truncated state-sync (e.g., a peer serving incorrect parts that pass `validate_state_part` but produce a trie with missing interior nodes — `MissingTrieValue` is only caught at apply time, not at part-validation time).
- A `ChunkExtra` state-root mismatch written by `set_state_finalize` if the state-sync header's `chunk_height_included` points to a block whose chunk extra was already overwritten.
- A memtrie not being loaded in time for resharding inside `block_catch_up_postprocess`.

The CHANGELOG entry for 2.2.1 ("a bug in the integration between ethereum implicit accounts and the compiled contract cache which sometimes caused the nodes to get stuck... unable to sync") demonstrates that this class of stuck-node bug has occurred in production. The catchup retry loop is the same architectural pattern.

### Recommendation

1. **Classify errors before re-queuing.** Consult `err.is_bad_data()` (or a new `err.is_transient()` predicate). Deterministic errors (bad data, invalid state root, storage inconsistency) should not be retried; they should abort the catchup for this epoch and surface a fatal metric/log.

2. **Add a per-block retry limit.** Track retry counts in `BlocksCatchUpState`. After N failures (e.g., 3), mark the block as permanently failed and halt catchup with a clear error.

3. **Distinguish `StorageError` subtypes.** `MissingTrieValue` and `StorageInconsistentState` are not transient; `FlatStorageBlockNotSupported` may be. The retry path should only apply to the latter.
