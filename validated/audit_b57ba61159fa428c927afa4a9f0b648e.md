### Title
Wrapping Integer Overflow in `check_time_locks` Bypasses `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Timelocks — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` accepts a `nowrap: bool` parameter that switches between `wrapping_add` and `saturating_add` when computing the deadline for relative-time and relative-height conditions. When `nowrap = false`, a coin whose puzzle emits `ASSERT_SECONDS_RELATIVE 0xffff_ffff_ffff_ffff` (or `ASSERT_HEIGHT_RELATIVE 0xffff_ffff`) has its deadline computed via integer wrap-around, producing a value that is already in the past. The timelock check therefore passes immediately, allowing the coin to be spent without waiting the intended duration. The inverse overflow applies to `ASSERT_BEFORE_SECONDS_RELATIVE` / `ASSERT_BEFORE_HEIGHT_RELATIVE`, permanently locking coins that should be spendable.

---

**Arithmetic walkthrough (`ASSERT_SECONDS_RELATIVE u64::MAX`, `nowrap=false`):**

- `coin_confirmed_time = 1000`
- `seconds_relative = 0xffff_ffff_ffff_ffff`
- `1000_u64.wrapping_add(0xffff_ffff_ffff_ffff) = 999`
- Check: `timestamp (2000) < 999` → **false** → no error → spend accepted immediately

The intended semantics require waiting ~584 billion years; the wrapping path makes the condition trivially satisfied at any timestamp.

**Inverse overflow (`ASSERT_BEFORE_SECONDS_RELATIVE u64::MAX`, `nowrap=false`):**

- `1000_u64.wrapping_add(0xffff_ffff_ffff_ffff) = 999`
- Check: `timestamp (2000) >= 999` → **true** → error → coin permanently unspendable [4](#0-3) 

The Python binding exposes `nowrap` as a caller-supplied argument, meaning the value is not hardcoded in the Rust layer and depends entirely on what the full-node Python code passes: [5](#0-4) 

---

### Impact Explanation

**High — Timelock condition validation bypass enables unauthorized spend acceptance.**

A coin whose puzzle emits `ASSERT_SECONDS_RELATIVE 0xffff_ffff_ffff_ffff` is intended to be unspendable for an astronomically long period. Under `nowrap=false`, the wrapping addition reduces the required timestamp to a value already in the past, so any node calling `check_time_locks(…, nowrap=false)` accepts the spend immediately. This is a direct timelock bypass matching the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

Additionally, if nodes in the network disagree on the `nowrap` value for a given block (e.g., during a soft-fork transition window), they will reach opposite verdicts on the same spend bundle, producing **consensus divergence** — a Critical-class impact.

---

### Likelihood Explanation

**Medium.** The `nowrap` flag is a soft-fork transition parameter. During any window where `nowrap=false` is the active setting, any unprivileged user can craft a CLVM puzzle that emits `ASSERT_SECONDS_RELATIVE 0xffff_ffff_ffff_ffff` and spend the resulting coin immediately. No privileged access, key material, or network-level capability is required — only the ability to submit a valid spend bundle.

---

### Recommendation

1. **Remove the `wrapping_add` branch entirely.** Relative-time and relative-height deadline arithmetic should always use `saturating_add`. There is no valid consensus reason to allow wrap-around: a relative delay of `u64::MAX` seconds should always produce a deadline in the far future, never in the past.
2. **If backward compatibility requires the old path**, gate it on a consensus constant (block height) rather than a caller-supplied boolean, so all nodes deterministically agree on which arithmetic to apply for every block.
3. **Audit all callers** of `py_check_time_locks` in the Python full-node code to confirm `nowrap=True` is always passed for mainnet blocks.
