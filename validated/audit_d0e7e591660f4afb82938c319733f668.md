### Title
Ante Handler Nonce Pre-Increment Causes Deterministic CREATE Address Collisions in Batched EVM Transactions — (`x/evm/keeper/state_transition.go`, `ante/eth.go`)

---

### Summary

Ethermint's ante handler unconditionally pre-increments the account sequence for **every** `MsgEthereumTx` in a Cosmos batch — including contract-creation (CREATE) messages. The EVM state transition then resets the nonce back to `msg.Nonce` before calling `evm.Create()`. When a batch contains a CREATE transaction whose constructor performs a nested CREATE, the nested deployment consumes a nonce that a later batch message also targets, producing a deterministic address collision. The second CREATE fails with a VM error, the transaction is included in the block with `status=0`, and the sender's gas is fully consumed for a failure caused entirely by the protocol's own nonce management — not by any user error.

---

### Finding Description

**Root cause — ante handler (`ante/eth.go:321-326`):**

`CheckAndSetEthSenderNonce` iterates over every message in the batch and unconditionally increments the account sequence by 1 for each one, regardless of whether the message is a CALL or a CREATE:

```go
// increase sequence of sender
if err := acc.SetSequence(expectedNonce + 1); err != nil { ... }
ak.SetAccount(ctx, acc)
``` [1](#0-0) 

The function's own doc-comment claims "If the transaction is a contract creation, the nonce will be incremented during the transaction execution and not within this AnteHandler decorator," but the code contains no such guard — it increments for every message unconditionally. [2](#0-1) 

**Root cause — state transition (`x/evm/keeper/state_transition.go:468-482`):**

To compensate for the ante handler's pre-increment, `ApplyMessageWithConfig` saves the current (already-bumped) nonce, resets it to `msg.Nonce`, calls `evm.Create()`, then reconciles:

```go
oldNonce := stateDB.GetNonce(sender)          // post-ante nonce, e.g. N+batch_size
stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to msg.Nonce
ret, _, leftoverGas, vmErr = evm.Create(...)
afterCreateNonce := stateDB.GetNonce(sender)
nestedCreates := afterCreateNonce - msg.Nonce - 1
stateDB.SetNonce(sender, oldNonce+nestedCreates, ...)
``` [3](#0-2) 

**The collision scenario (documented in ADR-003):**

| Batch msg | `msg.Nonce` | Nonce reset to | Nonces consumed by `evm.Create()` | Issue |
|-----------|-------------|----------------|-----------------------------------|-------|
| tx0 | 0 | 0 | 0 (parent), 1 (nested child) | child deployed at `keccak(sender, 1)` |
| tx1 | 1 | 1 | 1 | tries to deploy at `keccak(sender, 1)` — **collision** |

tx1's `evm.Create()` hits `ErrContractAddressCollision` because the address derived from nonce 1 was already occupied by tx0's nested deployment. The EVM returns a VM error; the transaction is committed to the block with `status=0`; the sender's full gas limit is consumed. [4](#0-3) 

---

### Impact Explanation

This is an **EVM state transition bug** that causes a valid, ante-handler-accepted CREATE transaction to fail at execution time due to a protocol-internal nonce collision — not due to any user error. The sender:

1. Pays the full gas cost for a failed deployment.
2. Does not receive the deployed contract.
3. Cannot predict or avoid the failure using standard Ethereum tooling, because the collision is invisible at the individual-transaction level.

This matches the allowed High impact: *"EVM state transition … bug that permits … valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

- Any unprivileged user can submit a Cosmos batch (`sdk.Tx` with multiple `MsgEthereumTx`).
- Factory-pattern contracts (deployer constructors that spawn child contracts) are common in DeFi and NFT infrastructure.
- The collision is **deterministic**: given the same batch composition it will always occur, making it reliably triggerable and not dependent on timing or mempool ordering.
- The protocol explicitly still accepts batched EVM transactions; the ADR only "strongly discourages" them without enforcing rejection. [5](#0-4) 

---

### Recommendation

Enforce at the ante handler level that a Cosmos batch containing more than one `MsgEthereumTx` must not include any CREATE message (i.e., `msg.To == nil`). Reject such batches with `ErrInvalidRequest` before nonces are touched. This eliminates the collision surface without changing single-transaction behavior.

Alternatively, remove the nonce-reset workaround and instead prevent the ante handler from pre-incrementing nonces for CREATE messages, letting `evm.Create()` own the full nonce lifecycle as go-ethereum intends.

---

### Proof of Concept

```
Precondition: sender account nonce = 0.

Batch tx (single Cosmos Tx, two MsgEthereumTx):
  msg[0]: contractCreation=true, msg.Nonce=0
          constructor bytecode deploys one child contract (nested CREATE)
  msg[1]: contractCreation=true, msg.Nonce=1
          deploys an unrelated contract

Ante handler execution (CheckAndSetEthSenderNonce):
  msg[0]: sequence 0→1
  msg[1]: sequence 1→2
  Account nonce after ante = 2.

ApplyMessageWithConfig for msg[0]:
  oldNonce = 2
  SetNonce(sender, 0)          ← reset
  evm.Create() → parent at keccak(sender,0), child at keccak(sender,1)
  afterCreateNonce = 2, nestedCreates = 1
  SetNonce(sender, 2+1=3)

ApplyMessageWithConfig for msg[1]:
  oldNonce = 3
  SetNonce(sender, 1)          ← reset
  evm.Create() → tries keccak(sender,1) — ALREADY OCCUPIED
  vmErr = ErrContractAddressCollision
  tx included in block with status=0; gas fully consumed.
```

The sender loses the gas cost of `msg[1]` for a failure caused entirely by the protocol's nonce reset, not by any mistake in the submitted transaction. [3](#0-2) [6](#0-5)

```markdown
## Decision

We will align Ethermint's EVM transaction processing with go-ethereum's behavior and **strongly discourage the use of batched EVM transactions**. Specifically:

1. **Nonce Management Alignment**: The EVM state transition layer will implement nonce reset-and-reconcile logic that matches go-ethereum's behavior for single transactions.

2. **Documentation**: We will explicitly document that batched EVM transactions are not recommended and may produce unexpected results, particularly regarding contract address derivation.

3. **CREATE Transaction Handling**: For CREATE transactions, we will:
   - Reset the stateDB nonce to `msg.Nonce` before calling `evm.Create()`
   - Allow `evm.Create()` to increment the nonce internally (as go-ethereum does)

4. **CALL Transaction Handling**: For CALL transactions (including EIP-7702 delegated calls):
   - Process authorizations first (which may increment nonces)
   - Allow `evm.Call()` to increment the nonce internally (as go-ethereum does)

5. **No Batch Transaction Optimizations**: We will not add special handling or optimizations for batched EVM transactions, as such patterns do not exist in Ethereum and cannot be validated against the reference implementation.
```
