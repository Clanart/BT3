### Title
`eth_call` / `eth_estimateGas` Simulation Silently Bypasses `ErrStateConflict` Detection, Diverging from Actual On-Chain Execution — (`File: x/evm/keeper/state_transition.go`)

---

### Summary

`ApplyMessageWithConfig` is the single function used for both simulation (`eth_call`, `eth_estimateGas`) and actual block execution (`ApplyTransaction`). The two callers differ only in the `commit bool` parameter. When `commit=false` (simulation), `stateDB.Commit()` is never called, so the `ErrStateConflict` check — which detects that both the EVM and a nested native precompile action wrote to the same storage slot — is permanently skipped. When `commit=true` (actual execution), the same transaction is caught at commit time and returned as a `VmError` (status=0, gas consumed). The result is a systematic divergence: `eth_call` and `eth_estimateGas` report success and return a gas estimate for transactions that will deterministically fail on-chain.

---

### Finding Description

`ApplyMessageWithConfig` in `x/evm/keeper/state_transition.go` is called in three contexts:

1. **`ApplyTransaction`** — actual block execution, `commit=true`
2. **`EthCall` / `EstimateGas` gRPC handlers** — simulation, `commit=false`
3. **`prepareTrace` / debug tracing** — `commit=false` or `commit=true`

The `ErrStateConflict` detection lives exclusively inside the `if commit { stateDB.Commit() }` block:

```go
// state_transition.go lines 601-623
if commit {
    if err := stateDB.Commit(); err != nil {
        // Note: estimateGas and eth_call do not hit this path because commit is
        // false for simulations, so they will succeed even when a real execution
        // would produce a state conflict.
        if errors.Is(err, statedb.ErrStateConflict) {
            return &types.EVMResult{
                GasUsed:  gasUsed,
                VmError:  statedb.ErrStateConflict.Error(),
                ...
            }, nil
        }
        return nil, errorsmod.Wrap(err, "failed to commit stateDB")
    }
}
```

The comment itself acknowledges the divergence. `ErrStateConflict` is raised by `statedb.Commit()` when:

- A nested native precompile action (via `ExecuteNativeAction`) writes to a storage slot, **and**
- The outer EVM execution also writes to the same slot via `SSTORE`

```go
// statedb.go lines 766-787
for _, addr := range s.journal.sortedDirties() {
    ...
    for _, key := range obj.dirtyStorage.SortedKeys() {
        ...
        store := s.keeper.GetState(s.ctx, obj.Address(), key)
        if store != origin && store != dirty {
            return fmt.Errorf("%w: ...", ErrStateConflict, ...)
        }
    }
}
```

Because `stateDB.Commit()` is never called during simulation, the conflict check at lines 766–787 is never reached. The simulation EVM runs to completion, computes gas, and returns a clean `EVMResult` with no `VmError`. The `EstimateGas` binary search then converges on a gas value and returns it to the caller as a valid estimate.

The two code paths — simulation and execution — are structurally identical up to the `commit` gate, making this a direct analog to the Gnosis report: two paths that appear equivalent but diverge on a critical edge case.

---

### Impact Explanation

**High — Public JSON-RPC path feeds incorrect data that diverges from consensus execution.**

- `eth_call` returns `status=success` and return data for a transaction that will land on-chain as `status=0` (EVM failure, gas consumed).
- `eth_estimateGas` returns a gas estimate for a transaction that will fail on-chain. The user submits the transaction with the estimated gas; the transaction is included in the block, gas is fully consumed, and the receipt shows `status=0`.
- DApps and wallets that rely on `eth_call` to pre-validate transactions before submission will receive a false positive, causing them to submit transactions that fail on-chain and burn user funds (gas fees).
- The discrepancy is not observable from the RPC layer alone; the caller has no way to know the simulation result is wrong.

---

### Likelihood Explanation

The trigger condition requires a custom precompile that uses `ExecuteNativeAction` to write to a storage slot that the calling EVM contract also writes to via `SSTORE` in the same transaction. This is a realistic scenario for any Ethermint-based chain that ships custom precompiles (e.g., Cronos-style precompiles). The `ExecuteNativeAction` path is the standard mechanism for precompiles to interact with Cosmos state, and storage-slot overlap between a precompile and its caller contract is a natural consequence of precompile designs that mirror EVM storage. Any unprivileged user can trigger the discrepancy by calling `eth_call` or `eth_estimateGas` with a transaction targeting such a contract.

---

### Recommendation

**Short term:** In `ApplyMessageWithConfig`, when `commit=false`, perform a dry-run conflict check that mirrors the `Commit()` logic — iterate `journal.sortedDirties()` and check for native-vs-EVM storage overlap — and surface the result as a `VmError` in the returned `EVMResult`. This ensures simulation and execution agree on the outcome.

**Long term:** Refactor the conflict detection out of `stateDB.Commit()` into a standalone `stateDB.CheckConflicts()` method that can be called independently of the commit path, so both simulation and execution share the same validation logic without code duplication.

---

### Proof of Concept

1. Deploy a custom precompile `P` at address `0xPRE` that, when called, uses `ExecuteNativeAction` to write value `X` to storage slot `K` of contract `C`.
2. Deploy contract `C` with a function `f()` that:
   - Calls precompile `P` (triggering the native write to slot `K`)
   - Then executes `SSTORE K, Y` (EVM write to the same slot `K`)
3. Call `eth_call` targeting `C.f()`:
   - `ApplyMessageWithConfig` runs with `commit=false`
   - EVM executes, precompile writes `K=X` via `ExecuteNativeAction`, EVM writes `K=Y` via `SSTORE`
   - `stateDB.Commit()` is **never called**; no conflict is detected
   - `eth_call` returns `{"result": "0x...", "status": "0x1"}` — **success**
4. Submit the same transaction on-chain:
   - `ApplyMessageWithConfig` runs with `commit=true`
   - Same EVM execution occurs
   - `stateDB.Commit()` is called; conflict detected: `store(K) = X ≠ origin, dirty(K) = Y ≠ store(K)`
   - Returns `EVMResult{VmError: "state conflict"}` — **status=0, gas consumed**
5. The user's gas is burned for a transaction that `eth_call` predicted would succeed.

```go
	// Enforce the non-overlap invariant BEFORE flushing the native cache store.
	// A nested native action (via ExecuteNativeAction) commits its writes into s.cacheMS
	// (readable via s.ctx). If any EVM-dirty key was also written by such an action, the
	// store value visible through s.ctx will differ from originStorage. Detecting this
	// before flushing means we can abort cleanly — the parent context is never touched.
	//
	// Note: only EVM-dirty keys are checked; native-only writes have no EVM dirty bit
	// and are not in scope.
	for _, addr := range s.journal.sortedDirties() {
		obj, exist := s.stateObjects[addr]
		if !exist || obj.selfDestructed {
			continue
		}
		for _, key := range obj.dirtyStorage.SortedKeys() {
			origin := obj.originStorage[key]
			dirty := obj.dirtyStorage[key]
			if dirty == origin {
				continue
			}
			// A native action wrote this slot iff the store value differs from origin.
			// If it also differs from the EVM-dirty value, the two sides disagree — conflict.
			store := s.keeper.GetState(s.ctx, obj.Address(), key)
			if store != origin && store != dirty {
				return fmt.Errorf(
					"%w: address %s key %s modified by both EVM execution and native action (origin=%s, store=%s, dirty=%s)",
					ErrStateConflict,
					obj.Address().Hex(), key.Hex(), origin.Hex(), store.Hex(), dirty.Hex(),
				)
			}
		}
	}
```
