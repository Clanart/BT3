### Title
Child `ChunkExtra.proposed_split` Inherits Stale Parent Value After Resharding, Breaking Chunk Header Invariant - (File: chain/chain/src/resharding/manager.rs)

---

### Summary

When a shard is split during dynamic resharding, `process_memtrie_resharding_storage_update()` creates the child `ChunkExtra` by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field — which holds the `TrieSplit` that triggered the resharding — is never reset to `None`. Because the parent's final chunk (the one that proposed the split) has `proposed_split = Some(TrieSplit{...})`, the child `ChunkExtra` inherits that stale value. When the child shard's first chunk is produced in epoch N+2, the chunk producer freshly computes `proposed_split = None` (the resharding cooldown suppresses a new proposal) and embeds it in the chunk header. The validator then checks `prev_chunk_extra.proposed_split()` (= `Some(...)`) against `chunk_header.proposed_split()` (= `None`) in `validate_chunk_with_chunk_extra_and_receipts_root()`, producing `InvalidChunkHeaderShardSplit` and permanently blocking the child shard from producing its first chunk.

---

### Finding Description

**Root cause — `process_memtrie_resharding_storage_update()` in `manager.rs`:**

```rust
// chain/chain/src/resharding/manager.rs  lines 255-260
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
```

Only `state_root` and `congestion_info` are updated. Every other field — including `proposed_split`, `outcome_root`, `gas_used`, `balance_burnt`, `validator_proposals`, and `bandwidth_requests` — is silently inherited from the parent. [1](#0-0) 

The parent's final chunk (the one that proposed the split) has `proposed_split = Some(TrieSplit{boundary_account, left_memory, right_memory})` stored in its `ChunkExtra`. This is the value that triggered the resharding. After the clone, the child `ChunkExtra` carries the same `Some(TrieSplit{...})`.

**Broken invariant — `validate_chunk_with_chunk_extra_and_receipts_root()` in `validate.rs`:**

```rust
// chain/chain/src/validate.rs  lines 176-185
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(format!(
        "header has {:?}, expected {:?} ...",
        chunk_header.proposed_split(),
        prev_chunk_extra.proposed_split(),
    )));
}
``` [2](#0-1) 

When the child shard's first chunk is produced in epoch N+2:

1. The chunk producer reads `prev_chunk_extra` = child `ChunkExtra` (with `proposed_split = Some(...)`).
2. The chunk producer calls `compute_proposed_split()`, which returns `None` because `can_reshard()` is false during the cooldown period.
3. The chunk producer embeds `proposed_split = None` in the new chunk header (via `chunk_extra.proposed_split().cloned()` after applying the chunk).
4. Validation compares `prev_chunk_extra.proposed_split()` = `Some(TrieSplit{...})` against `chunk_header.proposed_split()` = `None` → **mismatch → `InvalidChunkHeaderShardSplit`**.

The same stale inheritance occurs in the implicit-transition path inside `validate_chunk_state_witness_impl()`, where the resharding branch also only updates `state_root` and `congestion_info`:

```rust
// chain/chain/src/stateless_validation/chunk_validation.rs  lines 736-737
*chunk_extra.state_root_mut() = new_state_root;
*chunk_extra.congestion_info_mut() = new_congestion_info;
``` [3](#0-2) 

The `proposed_split` carried forward through the implicit transition is the parent's stale value, so the state-witness validator also sees the mismatch.

The codebase's own documentation acknowledges the root cause:

> `chain/chain/src/resharding/manager.rs:249` — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field). [4](#0-3) 

**Exact corrupted value:** `child_chunk_extra.proposed_split` = `Some(TrieSplit { boundary_account: <parent's boundary>, left_memory: X, right_memory: Y })` instead of `None`.

---

### Impact Explanation

The child shard cannot produce or have validated its first chunk. Because the chunk producer always computes `proposed_split = None` (cooldown) while `prev_chunk_extra.proposed_split()` is `Some(...)`, every attempt to produce the child shard's first chunk fails with `InvalidChunkHeaderShardSplit`. The child shard is permanently halted from the moment it is created. All accounts and contracts assigned to the child shard become inaccessible. **Impact: High/Critical** — a shard-level liveness failure triggered automatically by the protocol's own resharding mechanism.

---

### Likelihood Explanation

The bug fires on every dynamic resharding event where the parent shard's final chunk had `proposed_split = Some(...)` (which is always the case, since that proposal is what triggered the resharding). The `DynamicResharding` protocol feature gates the code path; once enabled in production, every shard split will trigger this failure. **Likelihood: High** once the feature is active.

---

### Recommendation

In `process_memtrie_resharding_storage_update()`, explicitly reset `proposed_split` (and any other fields that are meaningless for a freshly created child shard) after cloning the parent `ChunkExtra`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Reset fields that are specific to the parent shard's last chunk
// and have no meaning for the newly created child shard.
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

Apply the same reset in the `ImplicitTransitionParams::Resharding` branch of `validate_chunk_state_witness_impl()` so that state-witness validation is consistent. [5](#0-4)
