### Title
Unbounded `proof_facts_size` in Invoke Transactions Causes Unbounded Poseidon Builtin Consumption During OS Hash Computation — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_invoke_transaction_hash` function in the StarkNet OS iterates over the user-controlled `proof_facts` array via `poseidon_hash_update_with_nested_hash` without any upper-bound check on `proof_facts_size`. Unlike every other user-controlled array in the same OS flow — `signature_len` and `calldata_size` are both explicitly capped at `SIERRA_ARRAY_LEN_BOUND - 1` — `proof_facts_size` carries no analogous guard. A malicious transaction sender can submit an invoke transaction with an arbitrarily large `proof_facts` payload. The OS must hash the entire array to compute the transaction hash, consuming O(`proof_facts_size`) Poseidon builtin instances. Because the L2 gas charged to the user does not account for this OS-level hashing work, the cost to *prove* the block can exceed the cost to *execute* the transaction, and can exceed the OS step/builtin limit, making the block unprovable and halting the network.

---

### Finding Description

**Root cause — missing bound on `proof_facts_size`**

In `execute_invoke_function_transaction` the proof-facts fields are loaded from the transaction hint with no size validation:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
``` [1](#0-0) 

They are then forwarded directly into `compute_invoke_transaction_hash`:

```cairo
let transaction_hash = compute_invoke_transaction_hash(
    common_fields=common_tx_fields,
    execution_context=tx_execution_context,
    account_deployment_data_size=account_deployment_data_size,
    account_deployment_data=account_deployment_data,
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
);
``` [2](#0-1) 

Inside `compute_invoke_transaction_hash`, the OS unconditionally iterates over every element of `proof_facts`:

```cairo
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
``` [3](#0-2) 

`poseidon_hash_update_with_nested_hash` computes a Poseidon hash over all `proof_facts_size` field elements, consuming one Poseidon builtin instance per element.

**Contrast with bounded fields**

Every other variable-length user field in the same code path carries an explicit `assert_nn_le(..., SIERRA_ARRAY_LEN_BOUND - 1)` guard:

- `signature_len` is bounded in `fill_account_tx_info`:

```cairo
assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1);
``` [4](#0-3) 

- `calldata_size` is bounded in `get_invoke_tx_execution_context`:

```cairo
assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
``` [5](#0-4) 

`proof_facts_size` has no equivalent guard anywhere in the OS code path.

**`check_proof_facts` does not bound the size**

The only validation of `proof_facts` in the OS is `check_proof_facts`, which asserts a *lower* bound (the array must be at least as large as the fixed headers) but imposes no upper bound:

```cairo
assert_le(ProofHeader.SIZE + VirtualOsOutputHeader.SIZE, proof_facts_size);
``` [6](#0-5) 

**Gas accounting mismatch**

The L2 gas charged to the user is derived from the Sierra execution gas (`L2_GAS_INDEX` resource bound). The OS-level Poseidon hashing of `proof_facts` is performed *after* execution, during transaction hash recomputation, and is not reflected in any per-transaction gas charge. This creates the same asymmetry as the external report: the cost to *include* the transaction (execution) is bounded by user-paid gas, but the cost to *prove* the block (OS hash computation) is unbounded.

---

### Impact Explanation

If a malicious user submits an invoke transaction with a `proof_facts` array of sufficient size, the OS will consume more Poseidon builtin instances than the block's builtin budget allows. The STARK prover cannot generate a valid proof for a block that exceeds its builtin limits. Because StarkNet blocks are sequential, an unprovable block prevents all subsequent blocks from being finalized, resulting in a **total network shutdown** — no new transactions can be confirmed until the block is discarded and the sequencer recovers.

**Allowed impact matched**: *High — Network not being able to confirm new transactions (total network shutdown).*

---

### Likelihood Explanation

- `proof_facts` is a user-submitted field in invoke transactions (v3 format); it is part of the transaction hash and exposed via `TxInfo.proof_facts_start/end`.
- The sequencer's gateway-level validation enforces L2 gas limits on *execution*, but the OS-level Poseidon cost of hashing `proof_facts` is not part of the L2 gas model and is therefore not checked at ingestion time.
- The attacker only needs to submit a single well-crafted invoke transaction; no privileged access, leaked keys, or operator collusion is required.
- The `proof_facts` feature is relatively new, making it plausible that sequencer-side size validation for this field is absent or incomplete.

---

### Recommendation

Add an explicit upper-bound assertion on `proof_facts_size` in the OS, consistent with the pattern used for `signature_len` and `calldata_size`:

```cairo
// In execute_invoke_function_transaction, after loading proof_facts_size:
assert_nn_le(proof_facts_size, SIERRA_ARRAY_LEN_BOUND - 1);
```

Additionally, the L2 gas model should be updated to charge gas proportional to `proof_facts_size` for the OS hashing work, so that the cost to prove a block is always bounded by the user-paid gas.

---

### Proof of Concept

1. Craft an invoke transaction (v3) with `proof_facts` set to an array of `N` arbitrary field elements, where `N` is chosen to exceed the Poseidon builtin budget of a single OS proof (e.g., `N = 10_000_000`).
2. Submit the transaction to the sequencer. The sequencer accepts it because the L2 execution gas is within the declared `resource_bounds[L2_GAS_INDEX].max_amount`.
3. The sequencer includes the transaction in a block and calls the prover.
4. The OS executes `compute_invoke_transaction_hash`, which calls `poseidon_hash_update_with_nested_hash(data_ptr=proof_facts, data_length=N)`, consuming `N` Poseidon builtin instances.
5. The prover fails: the Poseidon builtin usage exceeds the block's builtin limit.
6. The block cannot be proven. The network halts — no subsequent block can be finalized until the sequencer discards the block and the problematic transaction.
