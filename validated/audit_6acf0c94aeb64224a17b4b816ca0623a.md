### Title
Missing Minimum Normalized-Amount Validation in `init_transfer` Causes Permanent Fund Lock — (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR `omni-bridge` contract accepts a transfer in `init_transfer` (locking/burning user tokens) without verifying that the net transfer amount (after fee) will survive the decimal-normalization step applied later in `sign_transfer`. When `normalize_amount(amount_without_fee)` rounds down to zero due to floor division, `sign_transfer` permanently reverts with `InvalidAmountToTransfer`, and there is no cancel path to recover the already-locked tokens.

### Finding Description

**Step 1 — `init_transfer` accepts the transfer with only a weak fee guard.** [1](#0-0) 

The only validation is `fee < amount`. There is no check that `amount - fee` is at least as large as the normalization factor for the destination chain.

**Step 2 — Tokens are immediately locked/burned inside `init_transfer_internal`.** [2](#0-1) 

After this point the user's tokens are gone from their account and held by the bridge.

**Step 3 — `sign_transfer` applies floor-division normalization and hard-reverts if the result is zero.** [3](#0-2) 

`normalize_amount` is pure floor division: [4](#0-3) 

If `amount_without_fee < 10^(origin_decimals − decimals)`, the result is `0` and the call panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`. This panic is deterministic — every future call to `sign_transfer` for this transfer ID will produce the same result.

**Step 4 — No cancellation path exists.**

`remove_transfer_message` is only invoked inside `claim_fee_callback` (which requires a finalized destination-chain proof that can never exist) and inside `sign_transfer_callback` (which is only reached on MPC success, which never happens). There is no user-callable cancel or refund function.

**Concrete example** (NEAR → Ethereum, token with `origin_decimals = 24`, `decimals = 18`, normalization factor = 10^6):

| Field | Value |
|---|---|
| `amount` | 999 999 (< 10^6) |
| `fee.fee` | 0 |
| `amount_without_fee` | 999 999 |
| `normalize_amount(999 999)` | **0** |
| `sign_transfer` result | **permanent panic** |
| User recovery | **none** |

The same outcome is reachable for any token pair where `origin_decimals > decimals` and the user (or a fee-update) leaves `amount_without_fee` below the normalization threshold.

### Impact Explanation

User tokens are irrecoverably locked inside the bridge contract. The bridge's locked-token accounting records the full `amount`, but the destination chain never receives a mint/unlock because `sign_transfer` can never succeed. This breaks bridge collateralization (locked supply on NEAR has no corresponding supply on the destination chain) and constitutes a permanent, unrecoverable fund freeze — matching the **Critical** impact class: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

Any unprivileged user who calls `ft_transfer_call` with an `amount` (or `amount − fee`) below the normalization factor triggers this path. The normalization factor is `10^(origin_decimals − decimals)`. For tokens with a large decimal gap (e.g., a NEAR-native token with 24 decimals bridged to a chain with 6 decimals, factor = 10^18), even moderate token amounts can fall below the threshold. Additionally, a user who later calls `update_transfer_fee` to raise the fee — which is explicitly permitted — can push an existing transfer into this state, since `update_transfer_fee` only enforces `fee < amount`, not `normalize_amount(amount − fee) > 0`. [5](#0-4) 

### Recommendation

Add the normalization check at the point of transfer acceptance, before tokens are locked, so the transaction reverts cleanly and the user retains their funds:

```rust
// In init_transfer, after building transfer_message and looking up decimals:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let net = transfer_message.amount_without_fee()
            .near_expect(BridgeError::InvalidFee);
        require!(
            Self::normalize_amount(net, decimals) > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );
    }
}
```

Alternatively, mirror the existing `sign_transfer` guard into `init_transfer` so both entry points enforce the same minimum-amount invariant.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (normalization factor = 10^6).
2. Call `ft_transfer_call` with `amount = 500_000` and `fee = 0`. The `init_transfer` check `fee < amount` passes; tokens are locked.
3. Call `sign_transfer` for the resulting `transfer_id`. `normalize_amount(500_000, {24, 18}) = 500_000 / 10^6 = 0`. The contract panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
4. Repeat step 3 indefinitely — the result is always the same panic.
5. Observe that `remove_transfer_message` is never called; the transfer remains in `pending_transfers` and the tokens remain locked forever. [6](#0-5) [7](#0-6)

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
