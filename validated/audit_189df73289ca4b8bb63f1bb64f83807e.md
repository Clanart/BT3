### Title
Stake-Weight Ignored in Consensus Quorum Calculation Allows Minority-Stake Validator Coalition to Force Wrong Block Commitments — (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary

The `SingleHeightConsensus` state machine computes quorum thresholds using a raw validator count (`n_validators`) instead of the sum of actual staking weights, and hardcodes weight `1` for every vote regardless of the voter's on-chain stake. A coalition of validators controlling more than two-thirds of committee slots by count — but potentially far less than two-thirds of total stake — can reach a false consensus and force an arbitrary block commitment, corrupting the state root, state diff, and block hash that are written to storage and propagated to provers and sync.

### Finding Description

In `SingleHeightConsensus::new`, the value passed to `StateMachine` as `total_weight` is derived from the length of the validator list, not from the sum of their `StakingWeight` fields:

```rust
// TODO(matan): Use actual weights, not just `len`.
let n_validators =
    u64::try_from(validators.len()).expect("Should have way less than u64::MAX validators");
let state_machine = StateMachine::new(height, id, n_validators, ...);
``` [1](#0-0) 

Inside `StateMachine`, every incoming prevote and precommit is stored with a hardcoded weight of `1`, regardless of the actual stake of the voter:

```rust
// handle_prevote
let inserted = self.prevotes.insert((round, voter), (vote, 1)).is_none();
// handle_precommit
let inserted = self.precommits.insert((round, voter), (vote, 1)).is_none();
``` [2](#0-1) [3](#0-2) 

The quorum check in `votes_meet_threshold` then compares the sum of these unit weights against `self.total_weight` (the validator count):

```rust
let weight_sum = votes.iter()
    .filter_map(|(&(r, _voter), (v, w))| { ... .then_some(u64::from(*w)) })
    .sum();
threshold.is_met(weight_sum, self.total_weight)
``` [4](#0-3) 

This makes the quorum purely count-based. The `StakingManager` does compute a correct stake-weighted `total_weight` for proposer selection, but that value is never forwarded to the consensus state machine. [5](#0-4) 

When consensus reaches a decision, `decision_reached` in the batcher commits the winning proposal's `PartialBlockHashComponents` — which encodes the `state_diff_commitment`, `transaction_commitment`, `event_commitment`, and `receipt_commitment` — to storage and feeds them into `calculate_block_hash`: [6](#0-5) [7](#0-6) 

### Impact Explanation

A coalition controlling more than ⌈2N/3⌉ committee slots by count — where N is the committee size — can vote for any proposal and have it accepted as the consensus decision. Because the actual staking weights are ignored, this threshold can be reached with far less than two-thirds of total stake. The accepted proposal's `CommitmentStateDiff` is converted to a `ThinStateDiff`, its hash is written as the authoritative `state_diff_commitment`, and the resulting `BlockHash` is stored as the canonical block hash. Every downstream consumer — the committer (Patricia trie / global state root), the proof manager, state sync, and RPC — treats this value as ground truth. The result is **wrong state root, wrong state diff, wrong block hash** committed for the height, matching the "Critical. Wrong state … from blockifier/syscall/execution logic" impact category. [8](#0-7) [9](#0-8) 

### Likelihood Explanation

The committee is selected as the top-N stakers by weight, so entering the committee requires meaningful stake. However, once inside, all votes are equal. An attacker who can distribute stake across enough addresses to occupy more than two-thirds of committee slots by count — while each individual address holds only a small fraction of total stake — achieves a count-majority without a stake-majority. The TODO comment confirms this is a known, unmitigated gap, not a future design choice that is already enforced elsewhere. [10](#0-9) 

### Recommendation

1. Replace `n_validators` with the sum of actual `StakingWeight` values from `committee.members()` when constructing the `StateMachine`.
2. In `handle_prevote` and `handle_precommit` inside `StateMachine`, look up the voter's weight from the committee and store it instead of the hardcoded `1`.
3. Propagate the stake-weighted `total_weight` already computed in `StakingManager::fetch_and_build_committee` through `CommitteeTrait` so that `SingleHeightConsensus` can consume it.

### Proof of Concept

Assume a committee of 100 validators. The top 10 hold 90 % of total stake; the remaining 90 each hold 0.11 % of total stake.

1. An attacker acquires enough stake to place 67 addresses in the bottom 90 committee slots (each with minimal stake).
2. Each attacker-controlled validator connects and participates normally.
3. For a target height, all 67 attacker validators broadcast prevotes and precommits for a crafted proposal containing a malicious `CommitmentStateDiff` (e.g., altered storage values, wrong nonces, or a fabricated class hash).
4. `votes_meet_threshold` evaluates `67 > 100 * 2/3 = 66.67` → quorum met.
5. `SMRequest::DecisionReached` is emitted; the batcher's `decision_reached` commits the malicious `ThinStateDiff` and its `state_diff_commitment` to storage.
6. `calculate_block_hash` chains the corrupted `state_diff_commitment` into the block hash; the committer writes the wrong global state root to the Patricia trie.
7. All subsequent blocks use this corrupted state root as their parent, and provers receive wrong SNOS inputs derived from it.

            .chain(&block_hash_version.clone().into())

            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
