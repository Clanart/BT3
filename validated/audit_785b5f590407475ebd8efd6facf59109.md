### Title
`personal_sendTransaction` Password Parameter Silently Ignored — Unauthenticated Node-Key Transaction Signing Over HTTP JSON-RPC - (File: `rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

The `personal` JSON-RPC namespace exposes `personal_sendTransaction` and `personal_sign` over the HTTP JSON-RPC server. Both methods accept a `password` parameter that is supposed to authenticate the caller before the node's keyring key is used to sign. In Ethermint, both parameters are silently discarded (`_ string`). Additionally, the `personal` namespace is registered as `Public: false` — a go-ethereum convention meaning IPC-only — but Ethermint's HTTP server ignores this flag entirely and exposes the namespace over HTTP. Any remote caller who can reach the JSON-RPC port and knows an address held in the node's keyring can sign and broadcast arbitrary transactions from that address with no authentication.

---

### Finding Description

**1. Password parameter is unconditionally ignored in `personal_sendTransaction` and `personal_sign`**

`PrivateAccountAPI.SendTransaction` in `rpc/namespaces/ethereum/personal/api.go` declares the password as a blank identifier:

```go
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)
}
```

The `_ string` is the password. It is never forwarded to the backend. `backend.SendTransaction` in `rpc/backend/sign_tx.go` proceeds directly to keyring lookup and signing:

```go
func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
    _, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
    ...
    if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil { ... }
    ...
    rsp, err := b.broadcastTx(txBytes)
```

No password is ever checked. The same applies to `personal_sign`:

```go
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    return api.backend.Sign(addr, data)
}
```

**2. `Public: false` flag is never enforced — `personal` is exposed over HTTP**

In go-ethereum's node infrastructure, `Public: false` on an `rpc.API` restricts the namespace to IPC only. Ethermint does not use go-ethereum's node infrastructure. In `rpc/apis.go`, the `personal` namespace is registered with `Public: false`:

```go
PersonalNamespace: func(...) []rpc.API {
    ...
    return []rpc.API{{
        Namespace: PersonalNamespace,
        Service:   personal.NewAPI(ctx.Logger, evmBackend),
        Public:    false,   // ← supposed to mean IPC-only
    }}
},
```

But in `server/json_rpc.go`, all APIs are registered unconditionally via `rpcServer.RegisterName` without checking the `Public` field:

```go
for _, api := range apis {
    if err := rpcServer.RegisterName(api.Namespace, api.Service); err != nil {
        ...
    }
}
```

The `Public` field is never read. The `personal` namespace is therefore fully accessible over the HTTP JSON-RPC server whenever it is listed in the `api` config field.

**3. `personal` is listed as an available namespace and enabled in test/network setups**

`GetAPINamespaces()` in `server/config/config.go` returns `["web3", "eth", "personal", "net", "txpool", "debug", "miner"]`. The `testutil/network/network.go` calls `config.GetAPINamespaces()` to enable all namespaces including `personal` for the first validator. Operators who follow this pattern or who explicitly add `personal` to their `api` config expose the unauthenticated signing endpoint over HTTP.

---

### Impact Explanation

When the `personal` namespace is enabled, any remote caller who can reach the JSON-RPC HTTP port (default `127.0.0.1:8545`, but commonly reconfigured to `0.0.0.0:8545` in production deployments) and knows an Ethereum address held in the node's keyring can:

1. Call `personal_sendTransaction` with `from` set to that address, `to` set to an attacker-controlled address, and an arbitrary `value`.
2. The node will sign the transaction with its keyring key and broadcast it to the network.
3. Funds are transferred from the node's account to the attacker with no password or authentication check.

This is an unauthorized fund transfer from EVM-denom accounts held in the node's keyring, matching the **High** impact category: *Ethereum transaction signer verification bypass enabling unauthorized account/code mutation*.

---

### Likelihood Explanation

- The `personal` namespace is not in the default three-namespace set (`eth`, `net`, `web3`), but it is listed in `GetAPINamespaces()` as a first-class available namespace.
- The `testutil/network` code enables all namespaces including `personal`, making it easy for operators to copy this pattern into production.
- The `Public: false` flag creates a false sense of security — operators familiar with geth may believe `personal` is IPC-only and safe to enable, not realizing Ethermint ignores this flag.
- Nodes that hold validator or operator keys in their keyring (a common pattern for `eth_sendTransaction`-based workflows) are directly at risk.

---

### Recommendation

1. **Enforce the `Public` flag**: In `server/json_rpc.go`, skip registration of any `rpc.API` with `Public: false` over HTTP. Only register `Public: false` APIs over IPC (if IPC is supported), matching go-ethereum's intended semantics.
2. **Remove or gate `personal_sendTransaction`**: If password-based authentication cannot be implemented (due to the Cosmos keyring not supporting per-call decryption), the method should return an explicit error rather than silently ignoring the password and proceeding.
3. **Warn in documentation and config**: Add a prominent warning that enabling the `personal` namespace over HTTP exposes unauthenticated key-signing operations.

---

### Proof of Concept

Assuming a node with address `0xABCD...` in its keyring and `personal` namespace enabled:

```bash
curl -X POST http://<node-ip>:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"personal_sendTransaction",
    "params":[{
      "from":"0xABCD...",
      "to":"0xAttacker...",
      "value":"0xDE0B6B3A7640000"
    }, "any_password_or_empty"],
    "id":1
  }'
```

The node will sign and broadcast the transaction regardless of the password value, transferring 1 ETH-equivalent from the node's keyring account to the attacker. The password field is discarded at: [1](#0-0)

```go
func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}
```
