### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay Leading to Direct Loss of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes the meta-transaction hash without including any nonce or replay-prevention field. This is a direct structural analog to the reported vulnerability: just as the game contract omitted `msg.sender` from its ID hash, the OS omits a nonce from the meta-tx-v0 hash. Any attacker who observes a valid `meta_tx_v0` syscall on-chain can extract the signature and replay it against the same target contract indefinitely, because the OS-computed hash — which the target contract's `__execute__` validates against — is identical on every replay.

---

### Finding Description

`compute_meta_tx_v0_hash` is defined as:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The hash inputs are: `(INVOKE_HASH_PREFIX, 0, contract_address, selector, calldata_hash, 0, chain_id)`. No nonce, no replay counter, no caller identity.

In `execute_meta_tx_v0`, the OS then constructs a `TxInfo` with `nonce=0` hardcoded and `transaction_hash=meta_tx_hash`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    account_contract_address=contract_address,
    max_fee=0,
    signature_start=request.signature_start,
    signature_end=request.signature_end,
    transaction_hash=meta_tx_hash,
    chain_id=old_tx_info.chain_id,
    nonce=0,                    // ← always zero
    ...
);
``` [2](#0-1) 

The target contract's `__execute__` entry point receives this `TxInfo` and validates the caller-supplied `signature` against `transaction_hash`. Because the hash is purely a function of `(contract_address, selector, calldata, chain_id)` — all of which are attacker-controllable and observable on-chain — the hash is identical on every replay. The OS performs no nonce check for `meta_tx_v0`, unlike regular account transactions which go through `check_and_increment_nonce`. [3](#0-2) 

For comparison, regular invoke transactions include `sender_address` and `nonce` in `hash_tx_common_fields`: [4](#0-3) 

The `meta_tx_v0` path has neither.

---

### Impact Explanation

An attacker who replays a `meta_tx_v0` call causes the target account contract's `__execute__` to run again with the same signature and the same OS-computed hash. If the original call authorized a token transfer (e.g., ERC-20 `transfer` or `transferFrom`), the replay executes the same transfer again. Because the OS never increments a nonce for `meta_tx_v0` and the hash never changes, the attacker can drain the target account in a loop across multiple blocks.

**Impact: Critical — Direct loss of funds.**

---

### Likelihood Explanation

The attacker entry path requires only:
1. Deploying a Cairo contract (permissionless on StarkNet).
2. Observing any on-chain `meta_tx_v0` syscall to extract `(contract_address, selector, calldata, signature)` from the transaction trace.
3. Calling their own contract, which issues a `meta_tx_v0` syscall with the extracted parameters.

No privileged access, no leaked keys, no operator cooperation is required. The attacker is a standard unprivileged transaction sender. The only precondition is that at least one `meta_tx_v0` call has been made by a legitimate user.

---

### Recommendation

Include a nonce (or a monotonically increasing counter stored in the target contract's state) in the `compute_meta_tx_v0_hash` inputs, mirroring how regular account transactions include `nonce` in `additional_data`. Concretely, pass `additional_data_size=1` and `additional_data=&nonce` where `nonce` is fetched from the target contract's on-chain state and incremented by the OS after each successful `meta_tx_v0` execution — exactly as `check_and_increment_nonce` does for regular transactions.

---

### Proof of Concept

```
1. Victim's account contract (addr=V) is called via meta_tx_v0 with calldata=[transfer, attacker, 1000_STRK].
   Signature S is provided by the initiating contract and visible in the transaction trace.

2. meta_tx_hash = H(INVOKE_PREFIX, 0, V, EXECUTE_SELECTOR, calldata_hash, 0, chain_id)
   This hash is deterministic and does not change between calls.

3. Attacker deploys MaliciousContract.
   MaliciousContract.attack() issues:
       syscall meta_tx_v0(
           contract_address = V,
           selector         = EXECUTE_ENTRY_POINT_SELECTOR,
           calldata         = [transfer, attacker, 1000_STRK],
           signature        = S,   // extracted from step 1
       )

4. The OS computes the same meta_tx_hash (no nonce → same inputs → same hash).
   V.__execute__ receives tx_info.transaction_hash = meta_tx_hash, tx_info.signature = S.
   V validates S against meta_tx_hash → valid → transfer executes again.

5. Attacker repeats step 3 until V's balance is zero.
   Each iteration costs only the gas of one transaction; no additional secrets are needed.
```

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
