### Title
Relayer Fee Permanently Lost When `ft_transfer` Fails After `remove_transfer_message` in `claim_fee_callback` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `claim_fee_callback`, the transfer record is permanently deleted from `pending_transfers` via `remove_transfer_message` and `locked_tokens` is decremented via `unlock_tokens_if_needed` — both irreversible state changes — before the fee `ft_transfer` is dispatched. Because no error-handling callback is attached to the `ft_transfer` promise, a failure silently discards the relayer's fee with no recovery path.

---

### Finding Description

`claim_fee_callback` executes the following sequence:

1. **Line 1094** — `remove_transfer_message(fin_transfer.transfer_id)` permanently removes the transfer from `pending_transfers`. After this point, calling `claim_fee` again for the same transfer ID will panic with `BridgeError::TransferNotExist`.

2. **Line 2684** (inside `send_fee_internal`) — `unlock_tokens_if_needed(...)` decrements `locked_tokens` for the destination chain/token pair. This is a committed storage write.

3. **Lines 2693–2697** (inside `send_fee_internal`) — For non-deployed (native) tokens, the fee is dispatched via:
   ```rust
   ext_token::ext(token)
       .with_static_gas(FT_TRANSFER_GAS)
       .with_attached_deposit(ONE_YOCTO)
       .ft_transfer(fee_recipient, U128(token_fee), None)
       .into()
   ```
   This promise is returned as `PromiseOrValue::Promise` with **no `.then()` callback** to handle failure.

In NEAR Protocol, state writes are committed when the function returns — they are not rolled back if a subsequent promise fails. If `ft_transfer` fails (e.g., the fee recipient has no storage registration for the token), the NEAR runtime silently drops the failure. The result:

- The transfer message is gone — `claim_fee` cannot be retried.
- `locked_tokens` is already decremented — accounting is desynchronised.
- The fee tokens remain in the bridge contract but are unclaimable by anyone.

This is structurally identical to the reported pattern: a terminal state change occurs before an external call that triggers asset movement, and the state check that would allow recovery is no longer satisfiable after the call.

---

### Impact Explanation

The relayer's earned fee is permanently frozen inside the bridge contract. The `locked_tokens` counter for the destination chain is decremented even though the tokens were never transferred, creating a persistent accounting discrepancy. The fee cannot be recovered: the transfer ID no longer exists in `pending_transfers`, so `claim_fee` will always revert with `TransferNotExist` for that transfer. This constitutes an irrecoverable lock of relayer (user) funds in the bridge flow.

---

### Likelihood Explanation

`ft_transfer` on a NEP-141 token fails when the recipient account has no storage deposit registered with that token contract. This is a realistic condition:

- A relayer operating across many token types may not have pre-registered storage for every native token they relay fees for.
- A relayer account that was recently created or whose storage was withdrawn will fail `ft_transfer` for any native token.
- The relayer has no way to pre-check and guarantee success before `claim_fee` commits the state changes.

The vulnerability is reachable by any trusted relayer calling `claim_fee` for a non-deployed (native) token where the fee recipient lacks storage registration.

---

### Recommendation

Attach an error-handling callback to the `ft_transfer` promise in `send_fee_internal`, or move `remove_transfer_message` and `unlock_tokens_if_needed` into a callback that only executes after confirming the `ft_transfer` succeeded. Alternatively, perform a storage-balance check for the fee recipient before committing the state changes, mirroring the `check_or_pay_ft_storage` pattern already used in `process_fin_transfer_to_near`.

---

### Proof of Concept

1. A transfer from Ethereum to NEAR is initiated and signed. The transfer message is stored in `pending_transfers` with a non-zero token fee.
2. A trusted relayer calls `claim_fee` with a valid `FinTransferMessage` proof. The fee recipient is the relayer's own account, which has **not** registered storage for the native token (e.g., USDC).
3. `claim_fee_callback` executes:
   - `remove_transfer_message` removes the transfer from `pending_transfers`. [1](#0-0) 
   - `send_fee_internal` calls `unlock_tokens_if_needed`, decrementing `locked_tokens`. [2](#0-1) 
   - `ft_transfer` is dispatched with no error callback. [3](#0-2) 
4. `ft_transfer` fails because the relayer has no storage registration for the token.
5. The NEAR runtime silently drops the failure. The fee tokens remain in the bridge. `locked_tokens` is permanently understated. The relayer's fee is irrecoverable.
