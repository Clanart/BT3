### Title
Missing State Diff Commitment Verification in Central Sync `store_state_diff` Allows Feeder-Controlled State Corruption — (`crates/apollo_central_sync/src/lib.rs`)

### Summary
The `store_state_diff` function in `GenericStateSync` writes a feeder-supplied `StateDiff` to storage without verifying that `calculate_state_diff_hash(thin_state_diff) == header.state_diff_commitment`. The block header (with its correct `state_diff_commitment`) is already stored before the state diff arrives, so the node possesses the ground-truth commitment but never uses it to validate the incoming data.

### Finding Description
The central sync pipeline fetches state diffs from the feeder gateway and stores them via:

```
stream_new_state_diffs
  → SyncEvent::StateDiffAvailable { block_number, block_hash, state_diff, … }
  → process_sync_event
  → store_state_diff          ← TODO comment: "verify state diff against stored header"
  → append_state_diff         ← writes ThinStateDiff to mmap unconditionally
```

`store_state_diff` contains an explicit developer TODO acknowledging the missing check. [7](#0-6)  The `SyncEvent::StateDiffAvailable` variant does not carry the feeder's stated commitment, and `append_state_diff` performs no cross-table lookup against the stored header's `state_diff_commitment`. [8](#0-7) 

The `verify_blocks` configuration flag does not gate this path — it only controls sequencer public-key tracking. [3](#0-2) 

### Impact Explanation
A feeder gateway that serves a `StateDiff` whose content differs from the `state_diff_commitment` recorded in the block header will have that wrong diff accepted and written to `state_diffs_table[block_number]` in the mmap file. Downstream consumers that read `get_state_diff(block_number)` — the committer (global root derivation), RPC state queries, and execution — will operate on the corrupted value. This satisfies the Critical impact criterion: *wrong state, storage value, or class hash from execution/syscall logic*.

### Likelihood Explanation
Requires the ability to control or intercept the feeder gateway connection (MITM, DNS hijacking, or direct feeder compromise). This is a network-level attack, not an operator-privilege attack. The missing check is acknowledged in the source code, confirming it is a known gap.

### Recommendation
In `store_state_diff`, after converting to `ThinStateDiff`, read the stored block header and assert:

```rust
let header = txn.get_block_header(block_number)?.ok_or(...)?;
let computed = calculate_state_diff_hash(&thin_state_diff);
if computed != header.state_diff_commitment {
    return Err(StateSyncError::StateDiffCommitmentMismatch { … });
}
```

This resolves the existing TODO and closes the integrity gap regardless of feeder trustworthiness.

### Proof of Concept
```rust
// 1. Write a block header with a known state_diff_commitment.
let expected_diff = ThinStateDiff { /* some non-empty diff */ };
let commitment = calculate_state_diff_hash(&expected_diff);
writer.begin_rw_txn()?
    .append_header(BlockNumber(0), &header_with_commitment(commitment))?
    .commit()?;

// 2. Call store_state_diff with a DIFFERENT StateDiff (no error is returned).
sync.store_state_diff(
    BlockNumber(0),
    block_hash,
    different_state_diff,   // attacker-controlled
    IndexMap::new(),
).await.unwrap();           // succeeds — no commitment check

// 3. Read back and verify the invariant is broken.
let stored = reader.begin_ro_txn()?.get_state_diff(BlockNumber(0))?.unwrap();
let stored_commitment = calculate_state_diff_hash(&stored);
assert_ne!(stored_commitment, commitment); // PASSES — invariant violated
```
