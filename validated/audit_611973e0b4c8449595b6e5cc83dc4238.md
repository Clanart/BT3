Audit Report

## Title
Native ETH sent to non-WETH swap functions is silently stranded and immediately stealable via `refundETH` - (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol / metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
All four swap entry points in `MetricOmmSimpleRouter` are declared `payable`, allowing the EVM to accept `msg.value` unconditionally before execution. When `tokenIn != WETH`, the internal `pay()` helper never touches `address(this).balance`, leaving any sent ETH stranded on the router. The public, attribution-free `refundETH()` function then allows any caller to drain the entire ETH balance, enabling theft of a victim's stranded ETH.

## Finding Description
The four swap functions â€” `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` â€” are all declared `payable`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The `receive()` guard that reverts non-WETH senders only applies to bare ETH transfers (no calldata); it does not block `msg.value` sent alongside a function call. [5](#0-4) 

Inside `pay()`, native ETH is only consumed when `token == WETH`. When `token != WETH` and `payer != address(this)`, execution falls through to the `else` branch and calls `safeTransferFrom`, never reading `address(this).balance`. Any ETH deposited via `msg.value` remains on the router after the swap completes. [6](#0-5) 

`refundETH()` is `external`, carries no access control, and sends the router's **entire** ETH balance to `msg.sender` with no record of who originally deposited it. [7](#0-6) 

## Impact Explanation
Any user who sends ETH alongside a non-WETH swap suffers a direct, unbounded loss of principal. The stranded ETH is immediately extractable by any third party (including MEV bots) calling `refundETH()` in the same or a subsequent block. This is a direct loss of user principal with no lower bound, satisfying Critical/High severity under Sherlock thresholds.

## Likelihood Explanation
The pattern is common in Uniswap v3-style routers where users are expected to pair swaps with `refundETH()` inside a `multicall`. Any user or integrator who calls a swap function directly with `msg.value > 0` and `tokenIn != WETH`, or builds a multicall that sends ETH for a WETH hop but routes through a non-WETH pool first, will strand ETH. MEV bots routinely monitor well-known router addresses for stranded balances and call `refundETH` immediately. No special privileges are required.

## Recommendation
1. **Reject non-zero `msg.value` when `tokenIn != WETH`:** Add a guard at the top of each swap entry point (or inside `pay()`) that reverts if `msg.value > 0 && token != WETH`.
2. **Alternatively, add attribution to `refundETH`:** Track per-sender ETH deposits in transient storage and only refund the caller's own contribution.
3. **Auto-refund on leftover ETH:** After the swap completes, if `address(this).balance > 0` and `tokenIn != WETH`, revert or auto-refund `msg.sender`.
