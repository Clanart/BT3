### Title
`compute_max_possible_fee` Overestimates Chargeable Fee for Backward-Compatible Accounts by Including Excluded L1 Data Gas Bounds — (File: `execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` always computes the maximum chargeable fee using the full raw resource bounds (including L1 data gas), while `execute_get_execution_info` presents a modified view of resource bounds to backward-compatible accounts — specifically excluding L1 data gas. This creates an accounting split: the account's `__validate__` approves a transaction based on bounds *without* L1 data gas, but the OS enforces a ceiling that *includes* L1 data gas. A malicious sequencer can exploit this gap to charge users more than they authorized.

---

### Finding Description

In `execute_get_execution_info` (`syscall_impls.cairo`, lines 833–870), for accounts where `should_exclude_l1_data_gas` returns true, the resource bounds presented to the account contract are modified to exclude L1 data gas:

```cairo
if (tx_info.version == 3 and exclude_l1_data_gas != FALSE) {
    let (excluded_resource_bounds_end) = exclude_data_gas_of_resource_bounds(
        resource_bounds_start=tx_info.resource_bounds_start,
        resource_bounds_end=tx_info.resource_bounds_end,
    );
    assert [response_tx_info] = TxInfo(
        ...
        resource_bounds_end=excluded_resource_bounds_end,  // L1 data gas stripped
        ...
    );
``` [1](#0-0) 

The account's `__validate__` entry point runs with this modified `TxInfo` and authorizes a fee based on resource bounds that do **not** include L1 data gas.

However, `compute_max_possible_fee` in `transaction_impls.cairo` (lines 87–101) always uses the **raw** `tx_info` — the unmodified one with all three resource bounds:

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;  // L1 data gas included
}
``` [2](#0-1) 

`charge_fee` then uses this overestimated `max_fee` as the only on-chain upper bound on what the sequencer can charge:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
...
local low_actual_fee;
%{ LoadActualFee %}   // sequencer-controlled hint
...
assert_nn_le(calldata.amount.low, max_fee);  // only guard
``` [3](#0-2) 

The `LoadActualFee` hint is provided by the sequencer (prover). The only Cairo-enforced constraint is `actual_fee ≤ max_fee`. Because `max_fee` is inflated by the L1 data gas component that the account never authorized, the sequencer can set `low_actual_fee` to a value higher than what the account approved.

This is a direct structural analog to the ERC20 bug: the internal accounting mechanism (`compute_max_possible_fee`) does not use the adjusted value that is presented to the authorizing party (`execute_get_execution_info`), leading to overestimation.

---

### Impact Explanation

**Critical — Direct loss of funds.**

For any user whose account contract has a class hash that triggers `should_exclude_l1_data_gas`, a malicious sequencer can charge up to `l1_data_gas_bounds.max_amount × l1_data_gas_bounds.max_price_per_unit` more than the account authorized. This excess is transferred to the sequencer address via the ERC20 `transfer` call inside `charge_fee`. The proof generated by the sequencer is valid from the OS's perspective (all Cairo assertions pass), so the L1 verifier accepts it, and the overcharge is finalized on-chain.

---

### Likelihood Explanation

In the ZK rollup security model, the sequencer is the prover and is explicitly **not** a trusted party — the OS proof is the mechanism that is supposed to prevent a malicious sequencer from stealing funds. The `LoadActualFee` hint is entirely sequencer-controlled; the only Cairo-level guard is `assert_nn_le(actual_fee, max_fee)`. Because `max_fee` is overestimated for backward-compatible accounts, this guard is insufficient. Any sequencer operator who becomes malicious or is compromised can exploit this for every affected account in every block, with no additional preconditions.

---

### Recommendation

`compute_max_possible_fee` should apply the same resource-bounds adjustment that `execute_get_execution_info` applies to the account's view. Specifically, for accounts where `should_exclude_l1_data_gas` returns true, the L1 data gas component must be excluded from the max fee computation. The simplest fix is to call `exclude_data_gas_of_resource_bounds` (or an equivalent) before computing the fee ceiling, mirroring the logic already present in `execute_get_execution_info`.

---

### Proof of Concept

1. User deploys an account whose class hash causes `should_exclude_l1_data_gas` to return `TRUE`.
2. User submits a v3 `INVOKE` transaction with resource bounds: `l1_gas=(A, P1)`, `l2_gas=(B, P2)`, `l1_data_gas=(C, P3)` where `C * P3 > 0`.
3. During `__validate__`, `execute_get_execution_info` is called. The account receives `TxInfo` with `resource_bounds_end` pointing past only `l1_gas` and `l2_gas` (L1 data gas stripped). The account approves, authorizing a fee of at most `A*P1 + B*(P2+tip)`.
4. `charge_fee` calls `compute_max_possible_fee` with the raw `tx_info`. It returns `A*P1 + B*(P2+tip) + C*P3`.
5. The malicious sequencer sets `LoadActualFee` hint to `A*P1 + B*(P2+tip) + C*P3`.
6. `assert_nn_le(actual_fee, max_fee)` passes.
7. The ERC20 `transfer` of `A*P1 + B*(P2+tip) + C*P3` tokens from the user to the sequencer is executed and proven.
8. The proof is accepted on L1. The user loses `C*P3` tokens beyond what they authorized.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
```
