### Title
`get_state_response_part` Reads `prev_state_root` from Wrong Chunk Index at Resharding Epoch Boundary — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part` derives `shard_index` from the **new epoch's** shard layout and uses it to index into `prev_block.chunks()`, which is ordered by the **old epoch's** shard layout. At a resharding boundary where a non-last shard splits (ShardLayoutV3), unchanged shards shift their positional index, so the function reads `prev_state_root` from the wrong chunk. The sibling function `compute_state_response_header` performs the same lookup correctly by explicitly fetching the prev epoch's layout first.

---

### Finding Description

`compute_state_response_header` correctly resolves the shard index against the **previous epoch's** layout before indexing into `sync_prev_block.chunks()`: [1](#0-0) 

```rust
let shard_layout = self.epoch_manager.get_shard_layout(sync_block_epoch_id)?;
let prev_epoch_id = sync_prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let prev_shard_index = prev_shard_layout.get_shard_index(shard_id)?;
// ...
let chunk_header = chunks.get(prev_shard_index)...
```

`get_state_response_part` does not. It fetches the **new epoch's** layout and uses that index directly against `prev_block.chunks()`: [2](#0-1) 

```rust
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;   // new epoch
// ...
let prev_block = self.chain_store.get_block(header.prev_hash())?;    // old epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;           // index in NEW layout
let state_root = prev_block
    .chunks()
    .get(shard_index)          // but prev_block is ordered by OLD layout!
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

In `ShardLayoutV3` (dynamic resharding), shard IDs and shard indices are **not** the same: [3](#0-2) 

When a non-last shard splits, the two child shards are appended at the end of the `shard_ids` list and the split parent is removed. Every unchanged shard that appeared **after** the split parent in the old list shifts one position to the left. Concretely:

- Old layout: shards `[0, 1, 2, 3]`, indices `[0, 1, 2, 3]`
- Shard 1 splits → new layout: shards `[0, 2, 3, 4, 5]`, indices `[0, 1, 2, 3, 4]`
- Shard 2: old index = 2, **new index = 1**
- Shard 3: old index = 3, **new index = 2**

`get_state_response_part` for shard 2 computes `shard_index = 1` (from the new layout) and reads `prev_block.chunks().get(1)`, which is shard 1's chunk header (the split parent) in the old layout. The `prev_state_root` extracted is therefore shard 1's state root, not shard 2's.

This wrong `state_root` is then passed to `obtain_state_part`: [4](#0-3) 

The state part is generated from the wrong trie root. When the receiving node calls `set_state_part`, it validates the part against the state root from the header (which was computed correctly by `compute_state_response_header`): [5](#0-4) 

The validation fails because the part's trie root does not match the header's `prev_state_root`. State sync for the affected shard is permanently broken for that epoch boundary.

---

### Impact Explanation

Any node attempting state sync at a resharding epoch boundary where a non-last shard split occurred will receive state parts generated from the wrong trie root. `set_state_part` rejects every part with a validation error. The node cannot complete state sync and cannot join the network. This affects all syncing nodes simultaneously at the same resharding boundary.

**Severity: High**

---

### Likelihood Explanation

Dynamic resharding (`ShardLayoutV3`) is an active production feature. The split shard is chosen by `pick_shard_to_split` based on memory usage, which can select any shard — not necessarily the last one. Any resharding event where the selected shard is not the last in the `shard_ids` list triggers the index shift. State sync is a standard recovery path for new and restarting nodes.

---

### Recommendation

In `get_state_response_part`, mirror the pattern used in `compute_state_response_header`: fetch the **previous epoch's** shard layout from `prev_block.header().epoch_id()` and use `prev_shard_layout.get_shard_index(shard_id)` to compute the chunk index before reading `prev_state_root`.

```rust
// Correct fix (mirrors compute_state_response_header lines 94-96):
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
```

---

### Proof of Concept

1. Configure a network with `ShardLayoutV3` dynamic resharding and 4 shards `[0, 1, 2, 3]`.
2. Force-split shard 1 (a non-last shard) via `force_split_shards`.
3. After resharding activates, the new layout has shards `[0, 2, 3, 4, 5]`; shard 2 now has index 1 and shard 3 has index 2.
4. Trigger state sync for shard 2 at the resharding epoch boundary (`sync_hash` = first block of new epoch).
5. `get_state_response_part` computes `shard_index = 1` (from new layout) and reads `prev_block.chunks().get(1).prev_state_root()` — this is shard 1's (the split parent's) state root.
6. `obtain_state_part` generates a part rooted at shard 1's trie.
7. `set_state_part` validates the part against shard 2's correct state root (from the header) and returns `set_state_part failed: validate_state_part failed`.
8. State sync for shard 2 is permanently broken at this epoch boundary.

    pub fn get_shard_index(&self, shard_id: ShardId) -> Result<ShardIndex, ShardLayoutError> {
        match self {
            // In V0 & V1 the shard id and shard index are the same.
            Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
            // In V2 & V3 the shard id and shard index are **not** the same.
            Self::V2(v2) => v2.get_shard_index(shard_id),
            Self::V3(v3) => v3.get_shard_index(shard_id),
        }
