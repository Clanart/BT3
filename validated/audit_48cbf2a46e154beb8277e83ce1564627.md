### Title
Validator Silently Discards Oracle-Computed WEI Gas Prices, Allowing Proposer to Commit Wrong Block Hash and Inflate ETH Fees — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates the FRI-denominated L1 gas prices against the oracle but explicitly discards the oracle-computed WEI prices (`_l1_gas_prices_wei`) without comparing them to the proposer-supplied values. A malicious proposer can embed arbitrary `l1_gas_price_wei` / `l1_data_gas_price_wei` in `ProposalInit`, pass all validator checks, and have those values baked into `PartialBlockHashComponents` → `calculate_block_hash`. The committed block hash is therefore wrong, and V1/V2 ETH-paying transactions in that block pay inflated fees.

---

### Finding Description

In `is_block_info_valid` the validator calls `get_l1_prices_in_fri_and_wei` and receives both FRI and WEI oracle prices, but the WEI result is bound to a discarded binding:

```rust
// validate_proposal.rs:286-292
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
```

Only the FRI prices are then checked against the proposer's values:

```rust
// validate_proposal.rs:302-318
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
```

`init_proposed.l1_gas_price_wei` and `init_proposed.l1_data_gas_price_wei` are never compared to the oracle. The only guard on WEI prices is a non-fatal `warn!` for zero values inside `convert_to_sn_api_block_info`.

The unvalidated WEI values then propagate through the full commitment pipeline:

1. `convert_to_sn_api_block_info(init)` → `BlockInfo.gas_prices.eth_gas_prices.l1_gas_price` / `l1_data_gas_price` [1](#0-0) 

2. `PartialBlockHashComponents::new(&block_info, header_commitments)` captures `block_info.gas_prices.l1_gas_price_per_token()` which is `GasPricePerToken { price_in_wei, price_in_fri }` [2](#0-1) 

3. `calculate_block_hash` chains `gas_prices_to_hash(l1_gas_price, l1_data_gas_price, l2_gas_price, ...)` into the Poseidon hash — both wei and fri fields are included [3](#0-2) 

4. The resulting `PartialBlockHashComponents` is stored in `BlockExecutionArtifacts` and later written to storage via `commit_proposal_and_block` [4](#0-3) 

5. `finalize_commitment_output` reads those stored components and calls `calculate_block_hash` a second time to produce the final `BlockHash` that is persisted [5](#0-4) 

A proposer that sets `l1_gas_price_wei = K × actual_wei_price` (any K ≠ 1) while keeping `l1_gas_price_fri` within the allowed margin will have its proposal accepted by every validator, yet the committed block hash will differ from what any honest node would compute from the real L1 prices.

---

### Impact Explanation

**Wrong block hash committed to storage (Critical — wrong storage value).**
The block hash contract stores the result of `calculate_block_hash` which includes the manipulated WEI price. Every future call to the `get_block_hash` syscall for that block number returns the wrong value. [6](#0-5) 

**Proof-facts validation broken (Critical — wrong proof input/output).**
`validate_proof_block_hash` compares the proof-supplied block hash against the value stored in the block hash contract. Because the stored hash was computed with the manipulated WEI price, any client-side proof that was generated against the correct block hash will be rejected, while a proof generated against the wrong hash will be accepted. [7](#0-6) 

**Inflated ETH fees for V1/V2 transactions (Critical — economic impact).**
`eth_gas_prices.l1_gas_price` (the WEI price) is used directly in fee calculation for ETH-paying transactions. An inflated WEI price causes those transactions to pay more than the user expected — the direct sequencer analog of the ERC721Sale front-running scenario.

---

### Likelihood Explanation

Any consensus participant that wins a proposer slot in a given round can trigger this. No special admin key or out-of-band access is required; the proposer simply crafts a `ProposalInit` with an inflated `l1_gas_price_wei`. The BFT protocol does not prevent a single malicious proposer from committing one block before being detected. The attack is silent: the proposal passes all existing validator checks and produces a valid `ProposalCommitment`.

---

### Recommendation

In `is_block_info_valid`, stop discarding the oracle-computed WEI prices and validate them against the proposer-supplied values with the same `within_margin` check:

```rust
// validate_proposal.rs
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

if !(within_margin(l1_gas_price_fri_proposed,  l1_gas_prices_fri.l1_gas_price,      margin)
  && within_margin(l1_data_gas_price_fri_proposed, l1_gas_prices_fri.l1_data_gas_price, margin)
  && within_margin(init_proposed.l1_gas_price_wei,  l1_gas_prices_wei.l1_gas_price,     margin)
  && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

This mirrors the fix applied to `ERC721Sale.mint()`: add an explicit bound on the parameter that was previously unchecked so that the transaction reverts if the proposer deviates from the expected value.

---

### Proof of Concept

1. Malicious node wins a proposer slot for height H.
2. It queries its own oracle: `get_l1_prices_in_fri_and_wei(timestamp, ...)` → `(fri_prices, wei_prices)`.
3. It constructs `ProposalInit` with:
   - `l1_gas_price_fri` = `fri_prices.l1_gas_price` (within margin — passes check)
   - `l1_gas_price_wei` = `1000 * wei_prices.l1_gas_price` (arbitrary — never checked)
4. Proposal is broadcast; every validator runs `is_block_info_valid`. [8](#0-7) 
5. Only FRI prices are compared → proposal accepted.
6. `convert_to_sn_api_block_info(init)` builds `BlockInfo` with `eth_gas_prices.l1_gas_price = 1000 × actual`. [9](#0-8) 
7. Batcher executes block; `PartialBlockHashComponents::new` captures the inflated WEI price. [10](#0-9) 
8. `calculate_block_hash` produces `H_wrong ≠ H_correct`. [11](#0-10) 
9. `H_wrong` is committed to the block hash contract and returned by all future `get_block_hash(H)` syscalls.
10. Any V1/V2 transaction in block H pays `1000×` the expected ETH fee.
11. Client-side proofs referencing block H are rejected because `stored_block_hash ≠ proof_block_hash`. [12](#0-11)

pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}

    pub fn get_block_hash(&mut self, requested_block_number: BlockNumber) -> SyscallResult<Felt> {
        // Note: we take the actual block number (and not the rounded one for validate)
        // in any case; it is consistent with the OS implementation and safe (see `Validate` arm).
        let current_block_number = self.context.tx_context.block_context.block_info.block_number;

        if !block_number_in_range(requested_block_number, current_block_number) {
            // Requested block is too recent.
            match self.context.execution_mode {
                ExecutionMode::Execute => {
                    // Revert the syscall.
                    return Err(SyscallExecutionError::Revert {
                        error_data: vec![BLOCK_NUMBER_OUT_OF_RANGE_ERROR_FELT],
                    });
                }
                ExecutionMode::Validate => {
                    // In this case, the transaction must be **rejected** to avoid the following
                    // attack:
                    //   * query a given block in validate,
                    //   * if reverted - ignore, if succeeded - panic.
                    //   * in the gateway, the queried block is (actual_latest - 9),
                    //   * while in the sequencer, the queried block can be further than that.
                    self.reject_syscall_in_validate_mode("get_block_hash on recent blocks")?;
                }
            }
        }

        self.storage_access_tracker.accessed_blocks.insert(requested_block_number);
        let key = StorageKey::try_from(Felt::from(requested_block_number.0))?;
        let block_hash_contract_address = self
            .context
            .tx_context
            .block_context
            .versioned_constants
            .os_constants
            .os_contract_addresses
            .block_hash_contract_address();
        let block_hash = self.state.get_storage_at(block_hash_contract_address, key)?;
        self.storage_access_tracker.read_block_hash_values.push(BlockHash(block_hash));
        Ok(block_hash)

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
