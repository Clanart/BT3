### Title
`deprecated_declared_classes` Silently Dropped in `CommitmentStateDiff → ThinStateDiff` Conversion, Producing Wrong State Diff Commitment and Block Hash — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion, which is the sole path used by the sequencer's block-building pipeline to produce the `ThinStateDiff` fed into `calculate_block_commitments`, unconditionally sets `deprecated_declared_classes: Vec::new()`. Because `calculate_state_diff_hash` chains `deprecated_declared_classes` into the Poseidon hash, and because `ThinStateDiff::len()` counts them toward `state_diff_length` / `concatenated_counts`, any block that contains a deprecated (Cairo 0) class declaration will produce a wrong `state_diff_commitment`, a wrong `concatenated_counts` field, and therefore a wrong block hash. This is the direct sequencer analog of the Bunker/Compound bug where a distribution-tracking update was silently omitted for one class of participants while the hash function still expected it.

---

### Finding Description

**Root cause — the omission:** [1](#0-0) 

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` has no `deprecated_declared_classes` field, so the conversion cannot populate it and hard-codes an empty `Vec`. The TODO comment acknowledges this is a known structural gap.

**Where the wrong `ThinStateDiff` is consumed:**

`BlockExecutionArtifacts::new()` calls `ThinStateDiff::from(commitment_state_diff.clone())` and passes the result directly to `calculate_block_commitments`: [2](#0-1) 

`calculate_block_commitments` calls both `calculate_state_diff_hash` (which chains `deprecated_declared_classes` into the Poseidon hash) and `concat_counts` (which uses `state_diff.len()`, which counts `deprecated_declared_classes`): [3](#0-2) 

**The hash function that expects the field:** [4](#0-3) 

`chain_deprecated_declared_classes` chains `[count, class_hash_0, …]` into the Poseidon hash. When the count is always 0 (because the field is always empty), the hash diverges from the correct value whenever a deprecated class is declared in the block.

**The wrong commitment propagates into the block hash:**

The `state_diff_commitment` and `concatenated_counts` (both wrong) are stored in `BlockHeaderCommitments`, which is wrapped into `PartialBlockHashComponents`, which feeds `calculate_block_hash`: [5](#0-4) 

The same wrong `ThinStateDiff` is also returned via `BlockExecutionArtifacts::thin_state_diff()` and stored in the batcher's storage, so the persisted state diff is also missing the deprecated class hashes. [6](#0-5) 

**The correct path (not used in block building) does include the field:**

`ThinStateDiff::from_state_diff` (used by the central sync path) correctly chains `migrated_compiled_classes` and `deprecated_declared_classes`: [7](#0-6) 

This asymmetry — the sync path is correct, the execution/commitment path is not — is the direct analog of the Bunker bug where borrower distributions were tracked but supplier distributions were not.

---

### Impact Explanation

For every block that contains a `DeclareV1` (deprecated Cairo 0 class declaration) transaction:

1. **Wrong `state_diff_commitment`** — the Poseidon hash omits the deprecated class hashes, producing a value that differs from the canonical Starknet state diff commitment.
2. **Wrong `concatenated_counts`** — `state_diff.len()` under-counts by the number of deprecated declared classes, corrupting the concatenated-counts field that is also chained into the block hash.
3. **Wrong block hash** — both corrupted values feed `calculate_block_hash`, so the final block hash is wrong.
4. **Wrong stored state diff** — the `ThinStateDiff` written to batcher storage via `thin_state_diff()` is missing the deprecated class hashes, so any downstream consumer (RPC `starknet_getStateUpdate`, proof manager, SNOS input) receives an authoritative-looking but incorrect state diff.
5. **Proof / SNOS failure** — the SNOS verifies the state diff commitment; a wrong commitment will cause proof rejection or, if the prover trusts the sequencer's value, produce an invalid proof accepted by the sequencer.

This matches the allowed impact categories:
- **Critical**: wrong state/commitment value from blockifier/execution logic for accepted input.
- **High**: RPC returns authoritative-looking wrong state diff.

---

### Likelihood Explanation

`DeclareV1` (deprecated class declaration) transactions are still part of the Starknet transaction type set and are handled throughout the codebase. Any unprivileged user who submits a valid `DeclareV1` transaction that passes gateway validation and is included in a block will trigger the wrong commitment. No special privilege or coordination is required. The TODO comment confirms the developers are aware the field is missing but have not yet resolved it.

---

### Recommendation

1. **Add `deprecated_declared_classes` to `CommitmentStateDiff`** so the blockifier's execution engine can populate it when a `DeclareV1` transaction is executed.
2. **Update `From<CommitmentStateDiff> for ThinStateDiff`** to copy the field instead of hard-coding `Vec::new()`, and remove the TODO comment once done.
3. **Add a regression test** that executes a `DeclareV1` transaction, finalizes the block, and asserts that the resulting `state_diff_commitment` matches the value computed by `calculate_state_diff_hash` on a `ThinStateDiff` that includes the deprecated class hash.

---

### Proof of Concept

1. Submit a `DeclareV1` transaction declaring a Cairo 0 class `C` with hash `H`.
2. The batcher includes it; the blockifier executes it and produces a `CommitmentStateDiff` with no `deprecated_declared_classes` entry for `H`.
3. `BlockExecutionArtifacts::new()` calls `ThinStateDiff::from(commitment_state_diff)`, yielding `deprecated_declared_classes = []`.
4. `calculate_state_diff_hash` chains `[0]` (count = 0) for deprecated classes instead of `[1, H]`.
5. The resulting `state_diff_commitment` differs from the correct value by the contribution of `H` to the Poseidon hash.
6. `concat_counts` uses `state_diff.len()` which is `N` instead of `N+1`, corrupting `concatenated_counts`.
7. `calculate_block_hash` ingests both wrong values; the block hash diverges from the canonical value.
8. `starknet_getStateUpdate` for that block returns a `ThinStateDiff` with `deprecated_declared_classes = []`, omitting `H` — an authoritative-looking wrong value.
