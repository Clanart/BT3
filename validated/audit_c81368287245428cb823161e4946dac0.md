Let me look at the exact `get_class` implementation and the `read_pending_data` function side by side, and also check how the sync loop manages the two locks atomically.

### Title
Stale Pending Class Served via `get_class(Pending, H)` Without Parent-Hash Staleness Guard — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

`get_class` reads `self.pending_classes` directly without calling `read_pending_data`, bypassing the `parent_block_hash == latest_header.block_hash` staleness check that every other pending-data RPC path enforces. During the polling gap between a new committed block and the sync loop's next detection cycle, `pending_classes` still holds classes from the previous pending epoch while the committed chain has already advanced. An unprivileged RPC client calling `starknet_getClass(block_id=Pending, class_hash=H)` receives the stale class body, while a concurrent `starknet_getStorageAt(Pending, …)` call correctly returns an empty fallback — a concrete, observable inconsistency.

### Finding Description

**`get_class` — no staleness check on `pending_classes`** [1](#0-0) 

When `block_id = Pending`, the handler immediately reads `self.pending_classes.read().await.get_class(class_hash)` and returns on a hit. It never opens a storage transaction, never calls `read_pending_data`, and never compares `pending_data.block.parent_block_hash()` against the latest committed block hash.

**`read_pending_data` — the guard that `get_class` skips** [2](#0-1) 

Every other pending-data RPC method (`get_state_update`, `get_nonce`, `get_block_transaction_count`, `get_transaction_by_block_id_and_index`, etc.) calls `read_pending_data`, which checks `pending_data.block.parent_block_hash() == latest_header.block_hash` and returns an empty fallback block when the hashes diverge. `get_class` is the sole exception.

**Sync loop — non-atomic, sleep-gated clear** [3](#0-2) 

The sync loop sleeps for `sleep_duration` between polls. When it detects a new block it clears `pending_classes` (line 180) and then updates `pending_data` (line 182) in two separate lock acquisitions. During the entire sleep interval — which can be seconds — after a new block N+1 is committed to storage:

- `pending_data.block.parent_block_hash()` is still N (not yet updated).
- `pending_classes` still contains every class declared in the pending epoch on top of N.
- `latest_header.block_hash` is already N+1.

In this window `read_pending_data` returns an empty fallback (correct), but `get_class` returns the stale class body from `pending_classes` (incorrect).

**`PendingClasses.get_class` — no epoch binding** [4](#0-3) 

`PendingClasses` is a plain `HashMap`; `get_class` performs a key lookup with no epoch or parent-hash metadata attached to entries.

### Impact Explanation

An unprivileged RPC client calling `starknet_getClass(block_id=Pending, class_hash=H)` receives the body of a class that was declared in a now-superseded pending epoch. If class H was not included in the newly committed block N+1, the returned class body does not correspond to any class in the current pending block or committed state — an authoritative-looking wrong value. This matches the **High** impact criterion: *RPC pending view returns an authoritative-looking wrong value*.

### Likelihood Explanation

The race window is bounded by the sync loop's `sleep_duration` (configurable, typically seconds). Any client that queries `starknet_getClass(Pending, H)` immediately after a block is committed — before the sync loop wakes — will hit the stale path. No special privileges are required; the endpoint is publicly accessible.

### Recommendation

Apply the same staleness guard used by every other pending-data RPC method. Before reading `pending_classes`, open a storage transaction, call `read_pending_data`, and only consult `pending_classes` when the returned `PendingData` is not the empty fallback (i.e., when `parent_block_hash` matches). Alternatively, attach the parent-block-hash epoch to each `PendingClasses` entry and reject entries whose epoch does not match the current latest block hash at query time.

### Proof of Concept

```
1. Node has committed block N (block_hash = H_N).
2. Pending sync has populated pending_classes with class C declared in the
   pending block on top of N.
3. Block N+1 is committed to storage (block_hash = H_N1).
   The sync loop is sleeping; pending_data.block.parent_block_hash() is still H_N.
4. Unprivileged client calls starknet_getClass(block_id=Pending, class_hash=C).
   → get_class reads pending_classes → finds C → returns stale class body.
5. Same client calls starknet_getStorageAt(Pending, addr, key).
   → read_pending_data checks H_N == H_N1 → FALSE → returns empty fallback.
6. Inconsistency: get_class says C exists in Pending; get_storage_at sees an
   empty pending state. If C was not included in N+1, C does not exist in any
   current authoritative state, yet the RPC returned its full class body.
```
