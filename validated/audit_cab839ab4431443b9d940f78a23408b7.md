### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by the caller corresponds to a previously declared contract class. This is directly analogous to the external report's "insufficient validation of referral address" class: an unvalidated user-controlled input is accepted and written into protocol state without verification. A malicious contract can invoke `replace_class` with an arbitrary, undeclared class hash, permanently rendering itself uncallable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without checking whether that hash corresponds to a declared class: [1](#0-0) 

The code itself contains an explicit acknowledgment of the missing check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The same missing validation exists in the deprecated syscall path: [2](#0-1) 

When a subsequent transaction attempts to call the contract whose class was replaced with an invalid hash, `execute_entry_point` performs: [3](#0-2) 

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
```

If `execution_context.class_hash` is an undeclared value, `dict_read` returns the default value `0`. `find_element` then searches for a compiled class with hash `0`. Since no such class exists in the bundle, `find_element` fails with an assertion error, making the contract permanently uncallable. The sequencer cannot include any transaction targeting this contract without causing OS proof failure, so the contract's funds are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any funds (ETH, STRK, or arbitrary ERC-20 tokens) held in a contract's storage become permanently inaccessible once `replace_class` is called with an undeclared class hash. There is no recovery path: the contract cannot call `replace_class` again to restore a valid class because the contract itself is uncallable. The state entry persists with the invalid class hash across all future blocks.

---

### Likelihood Explanation

The attack requires only that a contract invoke the `replace_class` syscall with an arbitrary felt value as the class hash. This is reachable by:

1. **Malicious contract operator (rug-pull):** A contract deployed as a legitimate service (e.g., a vault or DEX) accumulates user funds, then the operator calls an internal function that invokes `replace_class(0)` or any other undeclared hash. All deposited funds are frozen.
2. **Exploitable contract bug:** Any contract with a logic flaw that allows an unprivileged caller to trigger `replace_class` with attacker-controlled input is vulnerable.

No privileged role, leaked key, or external dependency is required. The OS Cairo code is the necessary vulnerable step — it is the only enforcement layer for this check, and it explicitly omits it.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that it corresponds to a declared class by checking `contract_class_changes`:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This check should be added in both `execute_replace_class` in `syscall_impls.cairo` and `execute_replace_class` in `deprecated_execute_syscalls.cairo`, resolving the acknowledged TODO.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` (class hash `A`) with:
   - A `deposit()` entry point that accepts STRK from users.
   - A `freeze()` entry point that calls `replace_class(0xDEAD)` where `0xDEAD` is not a declared class.
2. Users call `deposit()`, accumulating funds in the contract's storage.
3. Attacker calls `freeze()`. The OS processes the `replace_class` syscall:
   - `request.class_hash = 0xDEAD` is written directly to `contract_state_changes` with no validation.
   - The state entry for `MaliciousVault`'s address now has `class_hash = 0xDEAD`.
4. In any subsequent block, a user attempts to call `withdraw()` on the contract:
   - `execute_entry_point` reads `class_hash = 0xDEAD` from state.
   - `dict_read` on `contract_class_changes` returns `0` (undeclared).
   - `find_element` searches for compiled class with hash `0` — not found — OS proof fails.
   - The sequencer cannot include this transaction; it is permanently rejected.
5. All user funds in the contract are permanently frozen with no recovery mechanism.
