### Title
Integer Overflow in `check_time_locks` Bypasses Relative Timelock Conditions via `wrapping_add` When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

The `check_time_locks` function accepts a `nowrap: bool` parameter. When `nowrap=false`, relative timelock arithmetic uses `wrapping_add` instead of `saturating_add`. A spend bundle containing `ASSERT_SECONDS_RELATIVE(u64::MAX)` or `ASSERT_HEIGHT_RELATIVE(u32::MAX)` will wrap the addition to a value smaller than the current timestamp/height, causing the timelock guard to evaluate as satisfied and the coin to be spendable immediately — bypassing the intended lock entirely.

---

### Finding Description

In `check_time_locks`, the `seconds_relative` and `height_relative` checks branch on `nowrap`:

```rust
if let Some(seconds_relative) = spend.seconds_relative {
    if nowrap {
        if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
            return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
        }
    } else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
        return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
    }
}
```

When `nowrap=false` and `seconds_relative = u64::MAX`:

- `coin_timestamp = 1000`
- `1000_u64.wrapping_add(u64::MAX) = 999`
- Guard: `timestamp < 999` → `2000 < 999` → **false** → no error → coin is spendable

The condition `ASSERT_SECONDS_RELATIVE(u64::MAX)` is intended to lock a coin for ~584 billion years, but with `nowrap=false` it is satisfied at any timestamp. The same overflow applies to `height_relative` using `u32::wrapping_add`.

The test suite explicitly documents this divergence:

```rust
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),
)]
```

The `nowrap` flag is exposed directly to Python via `py_check_time_locks`, meaning the Python full node selects which arithmetic path is used per block. The `nowrap=false` path is the pre-softfork legacy path; if it is still active for any block height range where new transactions are accepted, the bypass is reachable.

The analog to the Chainlink report is exact: just as the oracle returns a clamped value that the code accepts without checking whether it is within the valid range, `check_time_locks` accepts a `seconds_relative` value without checking whether `coin_timestamp + seconds_relative` overflows `u64`, and the unchecked overflow produces a wrong comparison result that passes the guard.

---

### Impact Explanation

A coin locked with `ASSERT_SECONDS_RELATIVE(u64::MAX)` or `ASSERT_HEIGHT_RELATIVE(u32::MAX)` can be spent at any time when `nowrap=false` is in effect. This is a **timelock condition validation bypass enabling unauthorized spend acceptance**, matching the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance or replay."*

The inverse overflow also occurs for `ASSERT_BEFORE_HEIGHT_RELATIVE(u32::MAX)` and `ASSERT_BEFORE_SECONDS_RELATIVE(u64::MAX)` with `nowrap=false`: the wrapped sum falls below the current height/timestamp, causing the before-guard to fire and permanently blocking a coin that should be spendable — a consensus-diverging false rejection.

---

### Likelihood Explanation

The `nowrap` parameter is passed from the Python full node. If the softfork activating `nowrap=true` has not yet been applied to the height range being validated, `nowrap=false` is used and the bypass is live. An attacker needs only to submit a spend bundle containing a coin whose puzzle emits `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a value of `u64::MAX` / `u32::MAX` respectively — a trivially constructable, unprivileged CLVM program.

---

### Recommendation

Remove the `wrapping_add` branch entirely. Use `saturating_add` unconditionally for all relative timelock arithmetic. A relative lock of `u64::MAX` seconds should be treated as "never spendable within the lifetime of the chain," which `saturating_add` correctly models (the sum saturates to `u64::MAX`, which no real timestamp can exceed). The `nowrap=false` legacy path provides no correct semantic that cannot be replicated with saturation.

---

### Proof of Concept

The test at lines 301–306 of `check_time_locks.rs` is the direct proof:

```
coin confirmed at timestamp 1000, checked at timestamp 2000
seconds_relative = 0xffff_ffff_ffff_ffff (u64::MAX)

nowrap=true:  1000.saturating_add(u64::MAX) = u64::MAX
              2000 < u64::MAX → true → Err(AssertSecondsRelativeFailed)  ✓ correct

nowrap=false: 1000_u64.wrapping_add(u64::MAX) = 999
              2000 < 999 → false → Ok(())  ✗ timelock bypassed
```

A spend bundle whose puzzle returns `(ASSERT_SECONDS_RELATIVE . 0xffffffffffffffff)` will be accepted by any node running `check_time_locks` with `nowrap=false`, regardless of how recently the coin was confirmed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,

        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
