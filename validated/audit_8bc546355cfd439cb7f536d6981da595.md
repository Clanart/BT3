### Title
`l1_gas_price_margin_percent = 10` (10%) Is Too Permissive for L1 Gas Price Validation, Enabling a Proposer to Commit a Manipulated Gas Price to the Block Hash — (`crates/apollo_consensus_orchestrator/resources/orchestrator_versioned_constants_0_14_2.json` / `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

All three versioned-constants files hard-code `l1_gas_price_margin_percent: 10`. The `is_block_info_valid` function in `validate_proposal.rs` uses this value inside `within_margin` to decide whether a proposed block's L1 gas price (in FRI) is close enough to the locally computed price. A 10 % window is unjustifiably wide: a consensus proposer can embed an L1 gas price up to ~11 % above the market price, every validator will accept the proposal, and the inflated price is committed to the block hash and used for every fee calculation in that block.

---

### Finding Description

**Root cause — versioned constants**

Every active versioned-constants file sets the same value:

```json
// orchestrator_versioned_constants_0_14_0/1/2.json
"l1_gas_price_margin_percent": 10
``` [1](#0-0) 

The field is documented as "the margin for the eth-to-fri rate disagreement, expressed as a percentage (parts per hundred)": [2](#0-1) 

**Validation path**

`is_block_info_valid` reads the constant and calls `within_margin` for both `l1_gas_price_fri` and `l1_data_gas_price_fri`: [3](#0-2) 

`within_margin` is defined as:

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN { return true; }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
``` [4](#0-3) 

`number1` is the **proposed** price. Because the margin is computed from the proposed value, the check is asymmetric: a proposer who inflates the price by `p%` needs `p < margin_percent * (1 + p/100)`, which for 10 % allows the proposed price to be up to **~11.1 % above** the computed price before rejection.

**Downstream commitment path**

Once `is_block_info_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which constructs the `BlockInfo` (including `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`) directly from the proposer-supplied `ProposalInit`: [5](#0-4) 

This `BlockInfo` is forwarded to the batcher via `ValidateBlockInput`. The batcher executes all transactions in the block using these gas prices for fee pre-validation and post-execution checks. The same prices are stored in the block header and hashed into the block commitment.

---

### Impact Explanation

| Step | Effect |
|---|---|
| Proposer sets `l1_gas_price_fri = computed × 1.10` | Passes `within_margin` (diff = 10 %, margin = 11 %) |
| All validators re-run `is_block_info_valid` | All accept — same 10 % window |
| Batcher executes block with inflated price | Every transaction's L1 gas fee is ~10 % higher |
| Block header stores inflated price | Block hash commits to wrong L1 gas price |
| State sync / RPC serve the committed header | Downstream consumers (provers, explorers, bridges) see wrong authoritative price |

This matches **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact**.

---

### Likelihood Explanation

Any consensus validator is eligible to be the proposer for a given round. The manipulation requires no special privilege beyond normal validator participation. The attack is deterministic (no randomness, no race condition) and repeatable every time the malicious validator is selected as proposer. The 10 % window is large enough to be exploitable without triggering any other guard.

---

### Recommendation

1. **Reduce `l1_gas_price_margin_percent`** from `10` to `2`–`3` across all versioned-constants files. A 2–3 % window is sufficient to absorb legitimate timing skew between validators while eliminating the manipulation surface.
2. **Compute the margin from the locally-computed price** (`number2`), not the proposed price (`number1`), to eliminate the asymmetry that currently favors upward manipulation:
   ```rust
   let margin = (number2.0 * margin_percent) / 100;
   ```
3. Consider adding a hard cap: if the L1 gas price deviates by more than a fixed absolute threshold (e.g., 5 gwei), reject regardless of percentage.

---

### Proof of Concept

```
Computed L1 gas price (validator's local view): 100 gwei
Proposer sets proposed L1 gas price:           110 gwei  (+10%)

within_margin(110, 100, 10):
  abs_diff = |110 - 100| = 10
  margin   = (110 * 10) / 100 = 11
  10 <= 11  →  true  (proposal accepted)

Block committed with l1_gas_price = 110 gwei.
Every transaction in the block is charged ~10% more L1 gas fee.
Block hash commits to the inflated price.
```

Concretely, in `validate_proposal.rs` line 302, replace `l1_gas_price_fri_proposed` with `l1_gas_price_fri` as the first argument to `within_margin` and lower `l1_gas_price_margin_percent` to 2 in all three JSON files to close both the asymmetry and the excessive window. fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}

pub(crate) fn convert_to_sn_api_block_info(
    init: &ProposalInit,
) -> Result<starknet_api::block::BlockInfo, StarknetApiError> {
    if init.l1_gas_price_fri.0 == 0
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);
    }

    let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?;
    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;
    let previous_block_info = PreviousBlockInfo::from(init);
    let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;

    let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)

            )
        })?;
    Ok(starknet_api::block::BlockInfo {
        block_number: init.height,
        block_timestamp: BlockTimestamp(init.timestamp),
        sequencer_address: init.builder,
        gas_prices: GasPrices {
            strk_gas_prices: GasPriceVector {

                l1_data_gas_price: l1_data_gas_price_fri,

                l1_gas_price: l1_gas_price_wei,
                l1_data_gas_price: l1_data_gas_price_wei,
                l2_gas_price: l2_gas_price_wei,
            },
        },
        use_kzg_da: init.l1_da_mode.is_use_kzg_da(),
        starknet_version: init.starknet_version,
    })
}
