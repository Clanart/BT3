### Title
`nativeFee` ETH Permanently Locked in `OmniBridge` — Missing `claimNativeFee` Implementation Despite Defined `PayloadType.ClaimNativeFee` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts a `nativeFee` ETH amount from every bridging user and retains it in the contract. `BridgeTypes.PayloadType.ClaimNativeFee` (enum value `2`) exists, proving the protocol intended a signed-claim mechanism for relayers to withdraw this ETH. However, no `claimNativeFee` function is implemented anywhere in the contract. Combined with an open `receive() external payable {}` and no sweep/recovery function, all `nativeFee` ETH sent by users is irrecoverably locked.

---

### Finding Description

**Step 1 — ETH enters and stays in the contract.**

In `OmniBridge.initTransfer`, for ERC20 transfers:

```solidity
extensionValue = msg.value - nativeFee;   // line 393
```

For native ETH transfers:

```solidity
extensionValue = msg.value - amount - nativeFee;  // line 391
```

In both cases `extensionValue` (not `nativeFee`) is forwarded to `initTransferExtension`. In `OmniBridgeWormhole.initTransferExtension`, only `value` (= `extensionValue`) is forwarded to Wormhole:

```solidity
_wormhole.publishMessage{value: value}(...)   // line 143
```

The `nativeFee` portion of `msg.value` is never forwarded anywhere. It accumulates in the contract's ETH balance.

**Step 2 — The protocol intended a claim path that was never built.**

`BridgeTypes.PayloadType` defines three payload types:

```solidity
enum PayloadType {
    TransferMessage,   // 0
    Metadata,          // 1
    ClaimNativeFee     // 2  ← defined but never consumed
}
```

`ClaimNativeFee` is the only payload type with no corresponding handler function in `OmniBridge.sol` or `OmniBridgeWormhole.sol`. The enum value exists solely to be signed by the MPC/NEAR bridge and submitted by a relayer to claim their earned fee — but the function that would verify such a signature and pay out ETH was never implemented.

**Step 3 — Additional ETH entry with no exit.**

```solidity
receive() external payable {}   // line 574
```

This allows arbitrary ETH to enter (accidental sends, contract self-destructs, airdrops). There is no `sweep()`, `rescueETH()`, `rescueERC20()`, or any admin recovery function anywhere in the contract.

---

### Impact Explanation

Every user who calls `initTransfer` with `nativeFee > 0` permanently loses that ETH. The ETH is locked in the bridge contract with no on-chain path to release it — not to the relayer it was intended for, not to the user, and not to the admin. This constitutes an irrecoverable lock of user funds in a bridge flow, matching the Critical impact class. The locked ETH also breaks the protocol's fee-incentive model: relayers who deliver messages on NEAR cannot claim their EVM-side fee, undermining liveness of the bridge.

---

### Likelihood Explanation

`nativeFee` is a first-class parameter of the public `initTransfer` and `initTransfer1155` entry points. Any user who follows the protocol's intended flow of paying a native fee to incentivize relayers will trigger this loss. The `ClaimNativeFee` payload type being defined in the shipped `BridgeTypes.sol` confirms this is an expected user-facing feature, not an edge case. Likelihood is high for any deployment where relayer incentivization via `nativeFee` is used.

---

### Recommendation

1. Implement a `claimNativeFee` function that verifies a MPC-signed `PayloadType.ClaimNativeFee` payload (analogous to `finTransfer`'s signature check) and pays out the specified ETH amount to the designated fee recipient.
2. Add a `sweep(address token, uint256 amount, address to)` admin-only function to recover accidentally locked ETH and ERC20 tokens, consistent with the fix applied in the referenced PR 39.
3. Consider whether `receive()` should be restricted or removed if direct ETH sends are not a protocol requirement.

---

### Proof of Concept

1. Relayer deploys and the bridge goes live with `OmniBridgeWormhole`.
2. User calls:
   ```solidity
   omniBridge.initTransfer{value: 0.01 ether + wormholeFee}(
       tokenAddress,   // ERC20
       amount,
       fee,
       0.01 ether,    // nativeFee — intended relayer incentive
       "alice.near",
       ""
   );
   ```
3. Inside `initTransfer`: `extensionValue = (0.01 ether + wormholeFee) - 0.01 ether = wormholeFee`.
4. `initTransferExtension` forwards only `wormholeFee` to Wormhole. The `0.01 ether` nativeFee remains in the contract.
5. The relayer delivers the message on NEAR and expects to call `claimNativeFee` on EVM to collect `0.01 ether`. No such function exists.
6. The `0.01 ether` is permanently locked. Repeat for every `initTransfer` call with `nativeFee > 0`.
7. Additionally, any ETH sent directly to the contract via `receive()` is also permanently locked with no admin recovery path.
