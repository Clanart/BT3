### Title
Stranded `msg.value` ETH on router consumed by any subsequent WETH-input caller via unattributed `address(this).balance` read in `pay` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay` helper reads `address(this).balance` to decide how much native ETH to wrap and forward as WETH. Because this balance is never attributed to the caller who sent it, any ETH left on the router from a prior call (excess `msg.value` not reclaimed via `refundETH`) is silently consumed by the next WETH-input swap, transferring the first user's ETH to the second user's pool payment at zero cost to the attacker.

---

### Finding Description

`PeripheryPayments.pay` contains the following branch for WETH payments:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← reads total router ETH, not caller's ETH
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`nativeBalance` is the **total** ETH held by the router at that moment, not the ETH contributed by the current `msg.sender`. There is no per-caller accounting.

Every public swap entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) is marked `external payable`: [2](#0-1) [3](#0-2) 

A user who calls any of these directly (not via `multicall`) with `msg.value > amountIn` will have the excess ETH stranded on the router after `pay` consumes only `amountIn` worth. The `receive()` guard blocks direct ETH donations from non-WETH addresses, but it does **not** block excess `msg.value` arriving through a legitimate swap call: [4](#0-3) 

The stranded ETH persists across transactions. Any subsequent caller whose WETH-input swap triggers `pay` will have that ETH used to fund their pool payment, paying nothing from their own wallet.

---

### Impact Explanation

**Direct loss of user ETH principal.** User A's excess ETH is silently redirected to fund User B's swap. User A receives no tokens for the stolen portion; User B receives their full swap output without spending any ETH or WETH. The loss is bounded only by how much excess ETH User A sent.

---

### Likelihood Explanation

**Medium.** The trigger requires a user to call a swap function directly (not via `multicall`) with `msg.value > amountIn`, or to send ETH to a swap that ultimately uses less than the full value. This is a realistic mistake: the mixed-native-and-WETH flow is explicitly tested and documented as a supported pattern (`test_mixedNativeAndWeth_exactInputSingle_wethForToken`), and users accustomed to that pattern may overshoot `msg.value` without including `refundETH`. Once ETH is stranded, any observer of the mempool or chain state can exploit it immediately with a zero-cost follow-up call. [5](#0-4) 

---

### Recommendation

Track the ETH balance **before** the swap call begins and use only the delta (ETH actually contributed in the current call) inside `pay`. Concretely, snapshot `address(this).balance - msg.value` at entry and subtract it from `address(this).balance` inside `pay` to obtain the caller-attributed ETH. Alternatively, enforce that `address(this).balance == msg.value` at the start of each non-multicall entry point, reverting if prior ETH is present.

---

### Proof of Concept

```
// Step 1 – User A strands ETH
router.exactInputSingle{value: 2_000}(ExactInputSingleParams{
    tokenIn:  WETH,
    amountIn: 1_000,   // pay() uses 1_000 ETH, 1_000 ETH left on router
    ...
});

// Step 2 – Attacker (User B) pays nothing
router.exactInputSingle{value: 0}(ExactInputSingleParams{
    tokenIn:  WETH,
    amountIn: 1_000,   // pay() reads address(this).balance == 1_000 (User A's!)
    ...                // deposits and forwards User A's ETH; pulls 0 from attacker
});
// Result: User A loses 1_000 ETH; attacker's swap is fully funded at zero cost.
```

The root cause is `uint256 nativeBalance = address(this).balance` at line 74 of `PeripheryPayments.sol` — a balance snapshot with no caller attribution, the direct analog of the TREC-3 TransferReceiver balance inflation bug. [6](#0-5)
