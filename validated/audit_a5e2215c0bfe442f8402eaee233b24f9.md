### Title
Unchecked Felt Arithmetic in `compute_max_possible_fee` Causes OS Panic During Fee Charging — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the fee cap using raw felt multiplication of `max_amount` and `max_price_per_unit` without bounding the result to `[0, 2^128)`. The result is then passed as the second argument to `assert_nn_le`, which internally calls `assert_nn` (a range-check builtin that only accepts values in `[0, 2^128)`). When the computed `max_fee` exceeds `2^128 - 1`, the range-check fails, the OS panics, and no proof can be generated for the block — halting the network.

---

### Finding Description

In `compute_max_possible_fee` (lines 87–102 of `transaction_impls.cairo`):

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
```

The individual field bounds enforced during transaction hash computation (in `pack_resource_bounds`, `transaction_hash.cairo` lines 103–108) are:

- `max_amount ≤ 2^64 - 1` (via `assert_nn_le`)
- `max_price_per_unit ∈ [0, 2^128 - 1]` (via `assert_nn`, which uses the range-check builtin)
- `tip ≤ 2^64 - 1` (via `assert_nn_le`)

The product `max_amount * max_price_per_unit` can therefore reach up to `(2^64 - 1) × (2^128 - 1) ≈ 2^192`, which is a valid felt element (below the Cairo prime P ≈ 2^251) but far exceeds `2^128 - 1`. The sum of three such products can reach `≈ 3 × 2^192`.

This felt value is then used in `charge_fee` (line 135):

```cairo
assert_nn_le(calldata.amount.low, max_fee);
```

`assert_nn_le(a, b)` expands to `assert_nn(a)` then `assert_le(a, b)`, where `assert_le(a, b)` calls `assert_nn(b - a)`. The range-check builtin enforces `b - a ∈ [0, 2^128)`. When `max_fee > 2^128 - 1` and `calldata.amount.low` is small, `max_fee - calldata.amount.low ≈ max_fee > 2^128 - 1`, causing the range-check to fail and the OS to panic.

---

### Impact Explanation

When the OS panics during `charge_fee`, the Cairo execution trace is invalid and no STARK proof can be generated for the block containing that transaction. The block cannot be finalized, and the network cannot confirm new transactions — a **total network shutdown** (High impact).

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a V3 transaction with:
- `max_amount = 2` (valid: ≤ 2^64 - 1)
- `max_price_per_unit = 2^127` (valid: passes `assert_nn`, which only checks ≥ 0 and ≤ 2^128 - 1)

For all three resource types, the computed `max_fee = 3 × 2 × 2^127 = 3 × 2^128`, which exceeds `2^128 - 1`. The transaction hash is valid (all field checks in `hash_fee_fields` / `pack_resource_bounds` pass). If the sequencer includes this transaction in a block, the OS fails. The sequencer has no OS-enforced guard against this condition.

---

### Recommendation

Add an explicit upper-bound check on `max_fee` before using it in `assert_nn_le`. After computing `max_fee`, assert it fits within `[0, 2^128 - 1]`:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
assert_nn_le(max_fee, MAX_FEE_BOUND);  // e.g., MAX_FEE_BOUND = 2^128 - 1
```

Alternatively, add per-field upper-bound checks on `max_price_per_unit` in `pack_resource_bounds` (e.g., `assert_nn_le(resource_bounds.max_price_per_unit, MAX_PRICE_PER_UNIT)`) so that the product of any two bounds cannot exceed `2^128 - 1`.

---

### Proof of Concept

1. Craft a V3 invoke transaction with:
   - `l1_gas_bounds.max_amount = 2`, `l1_gas_bounds.max_price_per_unit = 2^127`
   - `l2_gas_bounds.max_amount = 2`, `l2_gas_bounds.max_price_per_unit = 2^127`
   - `l1_data_gas_bounds.max_amount = 2`, `l1_data_gas_bounds.max_price_per_unit = 2^127`
   - `tip = 0`

2. All checks in `pack_resource_bounds` pass:
   - `assert_nn_le(2, 2^64 - 1)` ✓
   - `assert_nn(2^127)` ✓ (2^127 < 2^128)

3. Transaction hash is computed and accepted.

4. Sequencer includes the transaction in a block.

5. OS calls `compute_max_possible_fee`:
   - Result = `2 × 2^127 + 2 × 2^127 + 2 × 2^127 = 3 × 2^128`

6. OS calls `assert_nn_le(low_actual_fee, 3 × 2^128)`:
   - Internally: `assert_nn(3 × 2^128 - low_actual_fee)` where `3 × 2^128 > 2^128 - 1`
   - Range-check builtin rejects the value → OS panics

7. No proof can be generated for the block → network halted.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
