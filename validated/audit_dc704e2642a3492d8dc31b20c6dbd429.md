### Title
EIP-7702 Durable Authorization Committed to Parent Context Even When EVM Execution Fails via Post-Hook Revert ‚Äî (`File: x/evm/keeper/state_transition.go`)

### Summary

In `ApplyTransaction`, when a transaction carries EIP-7702 `SetCodeAuthorizations` and the EVM execution succeeds but the post-processing hook (`PostTxProcessing`) subsequently fails, the durable authorization context is still committed to the parent store. This means the EIP-7702 delegation (nonce bump + code pointer) is permanently written to chain state even though the overall transaction is treated as reverted (VmError set, logs cleared). An attacker can exploit this to install an arbitrary code delegation on any EOA they control without the EVM call itself succeeding, bypassing the atomicity guarantee that authorization effects should only persist when the transaction commits.

### Finding Description

In `ApplyTransaction` (`x/evm/keeper/state_transition.go`), the flow is:

1. A `tmpCtx` cache context is created (line 181) to isolate EVM execution.
2. A separate `durableAuthorizationCtx` cache context is created (line 188) to hold EIP-7702 authorization effects.
3. `ApplyMessageWithConfig` is called on `tmpCtx` with `commit=true`. Inside, `applyAuthorization` writes nonce+code to the main `stateDB` (which targets `tmpCtx`), and the durable replay writes the same effects to `durableAuthorizationCtx` and commits it (lines 502‚Äì512).
4. Back in `ApplyTransaction`, if `res.Failed()` is false (EVM succeeded), `PostTxProcessing` is called (line 232). If the hook returns an error, `tmpCtx` is **not** committed (`commit()` is never called, `tmpCtxCommitted` stays `false`).
5. At line 266‚Äì268, the code checks `if commitDurableAuthorization != nil && !tmpCtxCommitted` and calls `commitDurableAuthorization()`.

The critical flaw: `commitDurableAuthorization()` is called whenever `tmpCtxCommitted == false` ‚Äî which includes the post-hook failure path. The intent of the durable context is to survive an EVM-level failure (e.g., out-of-gas revert), but it should **not** survive a post-hook failure that explicitly reverts the entire transaction. In the post-hook failure case, the EVM execution itself succeeded, so the durable authorization was already written and committed to `durableAuthorizationCtx`. When `commitDurableAuthorization()` is then called, the authorization effects (nonce increment + delegation code) are flushed into the parent `ctx` store permanently, while all other EVM state changes (balance transfers, storage writes, logs) are discarded.

### Impact Explanation

An attacker who controls an EOA can craft an EIP-7702 transaction (`type=4`) that:
1. Carries a valid `SetCodeAuthorization` for their own address pointing to an arbitrary contract.
2. Includes a call that will succeed at the EVM level but trigger a post-processing hook failure (e.g., by interacting with a hook-registered contract that reverts under specific conditions).

Result: The delegation (`0xef0100<contract_address>`) is permanently installed on the attacker's EOA and the nonce is bumped, but no other EVM state changes (value transfers, storage writes) are committed. The attacker has effectively installed arbitrary delegated code on their account without paying the full cost of a successful transaction, and without the EVM call's side-effects being committed. In chains where post-hooks enforce invariants (e.g., token accounting, access control), this breaks the atomicity of EIP-7702 authorization and can be used to install delegations that bypass subsequent EOA-code checks or to corrupt the nonce/code state of an account in a way inconsistent with the transaction's committed outcome.

This maps to: **High ‚Äî EVM state transition / ante handler / EIP-7702 authorization bug that permits invalid transactions to commit (partial state: authorization effects) while the overall transaction is treated as reverted.**

### Likelihood Explanation

- Requires: (a) the chain has at least one registered post-tx hook (`k.hooks != nil`), and (b) the attacker can craft a transaction whose EVM execution succeeds but whose post-hook fails.
- Post-hooks are a standard extension point used by chains built on Ethermint (e.g., for ERC-20 precompile accounting, IBC callbacks, staking hooks). Any hook that can be made to revert by attacker-controlled call data or state is sufficient.
- The attacker needs no special privileges ‚Äî only a funded EOA and knowledge of a hook that can be triggered to fail.
- The condition `k.hooks != nil` is the only guard; without hooks the `tmpCtx`/`commit` path is not taken and the bug does not manifest.

### Recommendation

Change the `commitDurableAuthorization` guard to only fire when the transaction was reverted due to an **EVM-level** failure (i.e., `res.Failed()` is true), not when it was reverted due to a post-hook failure. Introduce a separate boolean to distinguish the two revert causes:

```go
postHookFailed := false
if !res.Failed() {
    if err = k.PostTxProcessing(tmpCtx, msg, receipt); err != nil {
        postHookFailed = true
        res.VmError = types.ErrPostTxProcessing.Error()
        res.Logs = nil
    } else if commit != nil {
        commit()
        tmpCtxCommitted = true
    }
}

// Only commit durable authorization when the EVM itself failed (not post-hook)
if commitDurableAuthorization != nil && !tmpCtxCommitted && !postHookFailed {
    commitDurableAuthorization()
}
```

This preserves the intended behavior (authorization survives EVM revert) while restoring atomicity when the post-hook explicitly reverts the transaction.

### Proof of Concept

1. Deploy a contract `MaliciousTarget` on a chain with a registered post-tx hook that reverts when called with a specific selector.
2. Craft an EIP-7702 type-4 transaction from EOA `victim`:
   - `authorizationList`: `[{chainId, address: MaliciousTarget, nonce: victim_nonce+1, sig: victim_sig}]`
   - `to`: hook-triggering contract address
   - `data`: selector that causes the post-hook to revert
3. Submit the transaction. The EVM call succeeds (returns normally), but `PostTxProcessing` returns an error.
4. Observe: `tmpCtx` is not committed (EVM state reverted), but `commitDurableAuthorization()` is called at line 266.
5. Query `eth_getCode(victim.address)` ‚Äî it returns `0xef0100<MaliciousTarget>` (delegation installed).
6. Query `eth_getTransactionReceipt` ‚Äî the transaction shows `status=0` (failed/reverted).

The delegation is now permanently installed on `victim`'s EOA despite the transaction being recorded as failed, with no corresponding committed EVM state changes. [4](#0-3) [5](#0-4)

func (k *Keeper) ApplyTransaction(ctx sdk.Context, msgEth *types.MsgEthereumTx) (*types.EVMResult, error) {
	ethTx := msgEth.AsTransaction()
	cfg, err := k.EVMConfig(ctx, k.eip155ChainID, ethTx.Hash())
	if err != nil {
		return nil, errorsmod.Wrap(err, "failed to load evm config")
	}

	msg := msgEth.AsMessage(cfg.BaseFee)
	// snapshot to contain the tx processing and post processing in same scope
	var commit func()
	var commitDurableAuthorization func()
	tmpCtxCommitted := false
	tmpCtx := ctx
	if k.hooks != nil {
		// Create a cache context to revert state when tx hooks fails,
		// the cache context is only committed when both tx and hooks executed successfully.
		// Didn't use `Snapshot` because the context stack has exponential complexity on certain operations,
		// thus restricted to be used only inside `ApplyMessage`.
		tmpCtx, commit = ctx.CacheContext()

		// Keep the EIP-7702 authorization effects in a separate cache so they
		// survive a later EVM/post-hook failure that discards tmpCtx. Only needed
		// for txs that actually carry authorizations.
		if len(msg.SetCodeAuthorizations) > 0 {
			var durableAuthorizationCtx sdk.Context
			durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()
			cfg.DurableSetCodeAuthorizationCtx = &durableAuthorizationCtx
		}
	}

	// pass true to commit the StateDB
	res, err := k.ApplyMessageWithConfig(tmpCtx, msg, cfg, true)
	if err != nil {
		return nil, errorsmod.Wrap(err, "failed to apply ethereum core message")
	}

	logs := types.LogsToEthereum(res.Logs)

	// Compute block bloom filter
	if len(logs) > 0 {
		bloom := ethtypes.Bloom{}
		for _, log := range logs {
			bloom.Add(log.Address.Bytes())
			for _, topic := range log.Topics {
				bloom.Add(topic[:])
			}
		}
		k.SetTxBloom(tmpCtx, bloom.Big())
	}

	var contractAddr common.Address
	if msg.To == nil {
		contractAddr = crypto.CreateAddress(msg.From, msg.Nonce)
	}

	receipt := &ethtypes.Receipt{
		Type:            ethTx.Type(),
		PostState:       nil, // TODO: intermediate state root
		Logs:            logs,
		TxHash:          cfg.TxConfig.TxHash,
		ContractAddress: contractAddr,
		GasUsed:         res.GasUsed,
		BlockHash:       cfg.TxConfig.BlockHash,
		BlockNumber:     cfg.BlockNumber,
	}

	if !res.Failed() {
		receipt.Status = ethtypes.ReceiptStatusSuccessful
		// Only call hooks if tx executed successfully.
		if err = k.PostTxProcessing(tmpCtx, msg, receipt); err != nil {
			// If hooks return error, revert the whole tx.
			res.VmError = types.ErrPostTxProcessing.Error()
			k.Logger(ctx).Error("tx post processing failed", "error", err)

			// If the tx failed in post processing hooks, we should clear the logs
			res.Logs = nil
		} else if commit != nil {
			// PostTxProcessing is successful, commit the tmpCtx
			commit()
			tmpCtxCommitted = true
			// Since the post-processing can alter the log, we need to update the result
			res.Logs = types.NewLogsFromEth(receipt.Logs)
		}
	}

	// Get the tracer and add OnGasChange hook for gas refund
	leftoverGas := msg.GasLimit - res.GasUsed

	// refund gas in order to match the Ethereum gas consumption instead of the default SDK one.
	if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
		return nil, errorsmod.Wrapf(err, "failed to refund leftover gas to sender %s", msg.From)
	}

	tracer := cfg.GetTracer()
	if tracer != nil && tracer.OnGasChange != nil {
		tracer.OnGasChange(leftoverGas, 0, tracing.GasChangeTxLeftOverReturned)
	}

	totalGasUsed, err := k.AddTransientGasUsed(ctx, res.GasUsed)
	if err != nil {
		return nil, errorsmod.Wrap(err, "failed to add transient gas used")
	}

	if commitDurableAuthorization != nil && !tmpCtxCommitted {
		commitDurableAuthorization()
	}

	// reset the gas meter for current cosmos transaction
	k.ResetGasMeterAndConsumeGas(ctx, totalGasUsed)

	return res, nil
}

// applyAuthorization validates and applies an EIP-7702 code delegation to the
// state, returning the recovered authority so callers can reuse it without a
// second (expensive) signature recovery.
func (k *Keeper) applyAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (common.Address, error) {
	authority, err := k.validateAuthorization(auth, stateDB)
	if err != nil {
		return authority, err
	}

	// If the account already exists in state, refund the new account cost
	// charged in the intrinsic calculation.
	if stateDB.Exist(authority) {
		stateDB.AddRefund(params.CallNewAccountGas - params.TxAuthTupleGas)
	}

	k.setAuthorizationDelegation(auth, authority, stateDB)
	return authority, nil
}

// setAuthorizationDelegation writes the nonce bump and delegation code for a
// validated authorization.
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
