### Title
Unchecked Deploy Syscall Failure: `failure_flag` Hardcoded to 0 Regardless of Constructor Revert, Enabling Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in `syscall_impls.cairo` unconditionally writes `failure_flag=0` into the syscall response header, regardless of whether the constructor execution reverted. This is the direct Cairo analog of the Solidity `transfer()` return-value omission: a critical sub-operation's failure status is never propagated to the caller. Any contract that calls the `deploy` syscall and then acts on the assumption of success (e.g., by sending funds to the newly "deployed" address) will permanently freeze those funds when the constructor silently reverts.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–555), after calling `deploy_contract`, the response header is written with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The TODO comment at line 538 explicitly acknowledges that failure handling is not implemented. The `deploy_contract` call returns `(retdata_size, retdata)` — its return data is used in the response body — but the failure status is never inspected and `failure_flag` is never set to 1.

The revert mechanism in `execute_entry_point` confirms that `deploy_contract` *can* return normally when the constructor reverts: when `is_reverted != FALSE`, a fresh inner revert log is created, `handle_revert` undoes the constructor's state changes, the original outer revert log is restored, and execution continues normally:

```cairo
if (is_reverted != FALSE) {
    let revert_log = init_revert_log();
} else {
    assert outputs = orig_outputs;
    tempvar revert_log = orig_revert_log;
}
``` [2](#0-1) 

After `handle_revert` completes, all state changes from the constructor (including the class_hash assignment to the new address) are rolled back. The address is left with `class_hash=0` (no code). Yet `execute_deploy` writes `failure_flag=0` and returns the computed `contract_address` to the caller as if deployment succeeded:

```cairo
assert [response] = DeployResponse(
    contract_address=contract_address,
    constructor_retdata_start=retdata,
    constructor_retdata_end=retdata + retdata_size,
);
``` [3](#0-2) 

The deprecated path (`execute_deploy_syscall` in `deprecated_execute_syscalls.cairo`) has the same structural flaw: the `DeployResponse` is pre-filled before the constructor runs, and no failure status is ever checked:

```cairo
assert syscall_ptr.response = DeployResponse(
    contract_address=contract_address,
    constructor_retdata_size=0,
    constructor_retdata=cast(0, felt*),
);
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a caller contract (e.g., a factory or vault) calls the `deploy` syscall:

1. The constructor reverts; the OS rolls back all state changes via `handle_revert`. The target address has `class_hash=0` (no code, no storage).
2. The OS writes `failure_flag=0` and returns the computed `contract_address` to the caller.
3. The caller, believing deployment succeeded, proceeds to transfer funds (via the fee token ERC-20) to `contract_address`.
4. The fee token records the balance at `contract_address`, but since the address has no code, there is no entry point to retrieve those funds.
5. The funds are permanently frozen.

In StarkNet, token balances are stored in the fee token contract's storage keyed by address. Sending tokens to a codeless address is valid at the ERC-20 level but irrecoverable, because no `__execute__` or custom entry point exists at the recipient address to initiate a withdrawal.

---

### Likelihood Explanation

- **Unprivileged entry path**: Any deployed contract can invoke the `deploy` syscall. No privileged role is required.
- **Common pattern**: Factory contracts that deploy sub-contracts and immediately fund them (e.g., liquidity pools, escrow vaults, proxy deployers) are standard DeFi primitives on StarkNet.
- **Attacker control**: An attacker who can influence the `class_hash` argument passed to a victim factory's deploy call (e.g., through a public `deploy_pool(class_hash, ...)` interface) can supply a class whose constructor always reverts. The factory receives `failure_flag=0`, funds the address, and the funds are frozen.
- **No constructor revert detection**: Because `failure_flag` is always 0, no on-chain guard in the caller can distinguish success from failure using the syscall response alone.

---

### Recommendation

Implement the acknowledged TODO. Propagate the constructor's revert status from `deploy_contract` and set `failure_flag` accordingly:

```cairo
let (is_reverted, retdata_size, retdata) = deploy_contract(...);

assert [response_header] = ResponseHeader(
    gas=remaining_gas,
    failure_flag=is_reverted,   // propagate actual result
);
```

If the constructor reverts, append `ERROR_ENTRY_POINT_FAILED` to `retdata` (mirroring `contract_call_helper`) and write the failure response. This ensures callers can branch on `failure_flag` and avoid acting on a failed deployment.

The same fix must be applied to `execute_deploy_syscall` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. **Attacker** declares a Sierra class `RevertingClass` whose constructor body always panics (e.g., `assert 0 = 1`).
2. **Victim factory contract** exposes a public function `create_pool(class_hash, salt)` that:
   - Calls `deploy(class_hash, salt, ...)` syscall.
   - On return (always `failure_flag=0`), calls `fee_token.transfer(deployed_address, initial_liquidity)`.
3. **Attacker** calls `victim.create_pool(RevertingClass_hash, salt)`.
4. OS executes the constructor → reverts → `handle_revert` rolls back state → `contract_address` has `class_hash=0`.
5. OS writes `failure_flag=0` to the syscall response.
6. Victim factory reads `failure_flag=0`, transfers `initial_liquidity` tokens to `contract_address`.
7. Fee token records the balance. `contract_address` has no code. Funds are permanently frozen.
