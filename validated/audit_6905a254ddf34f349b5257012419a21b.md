### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared contract class. This is an explicitly acknowledged missing check (marked with a TODO). Any unprivileged contract deployer can call `replace_class` with an arbitrary, undeclared class hash, permanently bricking the contract and freezing all funds held within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall: [1](#0-0) 

The critical section is:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [2](#0-1) 

The `class_hash` value is taken directly from the syscall request (i.e., from the calling contract's calldata) and written into `contract_state_changes` with **no validation** that the hash exists in `contract_class_changes` (the declared class registry). The TODO comment at line 898 explicitly acknowledges this missing check.

This is the direct analog to the original report's vulnerability class: a critical mutable state field (the contract's class hash) can be set to an arbitrary value without the necessary guard that the new value is a valid, declared class.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is updated to an undeclared value via `replace_class`, the contract becomes permanently non-functional:

1. The OS stores the new (invalid) class hash in `contract_state_changes`.
2. Any subsequent call to the contract will attempt to look up and execute the class with that hash.
3. Since the class was never declared, execution fails unconditionally.
4. There is no recovery path: the contract cannot call `replace_class` again because its class is already invalid and no entry point can be invoked.
5. All funds (tokens, NFTs, or any assets) held by the contract are permanently frozen.

This matches the allowed impact: **Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

**Medium.**

The entry path is reachable by any unprivileged contract deployer:

- A user deploys a contract (allowed entry path: contract deployer).
- The contract calls the `replace_class` syscall with an arbitrary, undeclared class hash (e.g., `0xdeadbeef` or any felt value not in the declared class set).
- The OS accepts the call without validation and commits the invalid class hash to state.

This can occur:
- **Intentionally** (attacker deploys a contract holding user funds, then bricks it).
- **Accidentally** (a buggy upgradeable contract passes a wrong hash to `replace_class`).

No privileged access, leaked keys, or operator cooperation is required.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, the OS must verify that the supplied `class_hash` exists in `contract_class_changes` (i.e., has a non-zero compiled class hash entry). Specifically, in `execute_replace_class`, add a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned value is non-zero before proceeding with the `dict_update` on `contract_state_changes`. [3](#0-2) 

---

### Proof of Concept

1. Attacker deploys contract `A` that holds user funds (e.g., users deposit tokens into it).
2. Contract `A` contains a function that calls the `replace_class` syscall with `class_hash = 0x1` (or any value not present in `contract_class_changes`).
3. Attacker invokes that function via an `invoke` transaction.
4. The OS executes `execute_replace_class`:
   - `request.class_hash = 0x1` (undeclared).
   - No check is performed against `contract_class_changes`.
   - `dict_update` writes `StateEntry(class_hash=0x1, ...)` for contract `A`.
5. The state transition is committed. Contract `A`'s class hash is now `0x1`.
6. Any future call to contract `A` fails: the OS cannot find a compiled class for hash `0x1`.
7. All funds in contract `A` are permanently frozen.

The root cause is entirely within the scoped file: [4](#0-3)
