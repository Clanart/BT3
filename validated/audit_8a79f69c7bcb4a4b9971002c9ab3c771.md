### Title
Fee Not Locked on Destination Chain in Fast Transfer to Non-NEAR Chain Causes Permanent Freezing of User Funds — (`near/omni-bridge/src/lib.rs`)

### Summary

In `fast_fin_transfer_to_other_chain`, only `amount_without_fee` is locked on the destination chain. However, `claim_fee_callback` later unlocks `fee_amount` from that same destination chain. This leaves `locked_tokens[destination] = amount_without_fee − fee_amount`, which is less than the `amount_without_fee` tokens actually in circulation on the destination chain. When a user subsequently tries to bridge those tokens back to NEAR, the `unlock_tokens` check panics with `ERR_INSUFFICIENT_LOCKED_TOKENS`, permanently freezing their funds.

### Finding Description

**Root cause — `fast_fin_transfer_to_other_chain` (lib.rs ~914–972):**

```rust
let amount_without_fee = fast_transfer.amount_without_fee()...;
self.burn_tokens_if_needed(fast_transfer.token_id.clone(), amount_without_fee.into());
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,          // ← only amount_without_fee locked, NOT full amount
);
``` [1](#0-0) 

Compare with the **non-fast** path in `process_fin_transfer_to_other_chain` (lib.rs ~1980–2025), which correctly locks the full amount:

```rust
self.lock_tokens_if_needed(destination_chain, &token, transfer_message.fee.fee.into());
// ...
self.lock_tokens_if_needed(destination_chain, &token, transfer_message.amount_without_fee()...);
// total = fee + amount_without_fee = full amount ✓
``` [2](#0-1) 

**Fee claim — `claim_fee_callback` (lib.rs ~1066–1133):**

```rust
let fee = transfer_message.amount.0 - denormalized_amount;   // ≈ fee_amount
self.send_fee_internal(&transfer_message, fee_recipient, fee)
```

Inside `send_fee_internal`:

```rust
self.unlock_tokens_if_needed(transfer_message.get_destination_chain(), &token, token_fee);
``` [3](#0-2) 

This unlocks `fee_amount` from the destination chain. After the fee claim:

```
locked[destination] = amount_without_fee − fee_amount
```

**Bridge-back — `process_fin_transfer_to_near` (lib.rs ~1867–1977):**

When the user bridges their `amount_without_fee` tokens back from the destination chain:

```rust
let lock_actions = vec![self.unlock_tokens_if_needed(
    transfer_message.get_origin_chain(),   // = destination chain of original fast transfer
    &token,
    transfer_message.amount.0,             // = amount_without_fee
)];
``` [4](#0-3) 

`unlock_tokens` enforces:

```rust
require!(available >= amount, TokenLockError::InsufficientLockedTokens.as_ref());
``` [5](#0-4) 

Since `amount_without_fee − fee_amount < amount_without_fee`, the require panics and the bridge-back is permanently blocked.

**Accounting comparison:**

| Path | Locked on destination after fee claim | Bridge-back unlock needed | Result |
|---|---|---|---|
| Normal `fin_transfer` | `full_amount − fee = amount_without_fee` | `amount_without_fee` | ✓ succeeds |
| Fast transfer to other chain | `amount_without_fee − fee` | `amount_without_fee` | ✗ panics |

### Impact Explanation

Any user whose tokens were delivered via a fast transfer to a non-NEAR destination chain (e.g., Solana, EVM) with a non-zero fee will have their tokens permanently frozen on that chain. The `ERR_INSUFFICIENT_LOCKED_TOKENS` panic in `unlock_tokens` is unconditional — there is no fallback, no admin override path, and no way for the user to recover their funds. The deficit accumulates: each such fast transfer understates `locked_tokens` by `fee_amount`, so the problem compounds across multiple transfers.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows (Critical)**.

### Likelihood Explanation

The fast-transfer-to-other-chain path is a standard, documented protocol feature. Any trusted relayer executing a fast transfer to a non-NEAR chain with a non-zero fee (the normal case) triggers the bug. No malicious actor is required — the relayer follows the protocol correctly. The fee claim step is also a normal relayer action. The only precondition is `fee > 0`, which is the expected production configuration.

### Recommendation

In `fast_fin_transfer_to_other_chain`, lock the **full amount** (including fee) on the destination chain, mirroring the non-fast path:

```rust
// Lock fee portion separately so claim_fee_callback can unlock it correctly
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    fast_transfer.fee.fee.0,
);
// Lock amount_without_fee for the recipient
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,
);
```

This ensures `locked[destination] = fee + amount_without_fee = full_amount`, so after `claim_fee_callback` unlocks `fee_amount`, the remaining `amount_without_fee` correctly backs the tokens in circulation on the destination chain.

### Proof of Concept

1. Token `T` is a bridged token (origin chain = Eth). `locked_tokens[(Sol, T)] = 0` initially.
2. Relayer calls `ft_on_transfer` with `FastFinTransferMsg { amount: 1000, fee: { fee: 100 }, recipient: Sol(...) }` on NEAR, sending 900 tokens (`amount_without_fee`).
3. `fast_fin_transfer_to_other_chain` executes: `lock_tokens_if_needed(Sol, T, 900)` → `locked[(Sol, T)] = 900`.
4. Relayer calls `sign_transfer` → MPC signs the transfer for Sol. 1000 tokens (normalized) are signed for Sol, 900 go to recipient, 100 as fee.
5. Relayer calls `claim_fee` with proof of Sol finalization (`fin_transfer.amount = 900`).
6. `claim_fee_callback`: `fee = 1000 − 900 = 100`. `unlock_tokens_if_needed(Sol, T, 100)` → `locked[(Sol, T)] = 800`.
7. User on Sol initiates bridge-back of their 900 tokens.
8. `fin_transfer_callback` → `process_fin_transfer_to_near` → `unlock_tokens_if_needed(Sol, T, 900)`.
9. `unlock_tokens`: `available = 800 < 900` → **panics with `ERR_INSUFFICIENT_LOCKED_TOKENS`**.
10. User's 900 tokens are permanently frozen on Solana with no recovery path.
