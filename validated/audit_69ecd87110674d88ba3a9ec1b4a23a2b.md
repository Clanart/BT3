### Title
`burn_tokens_if_needed` Uses `.detach()` — Silent Burn Failure Allows Unbacked Cross-Chain Mint — (File: near/omni-bridge/src/lib.rs)

---

### Summary

In `init_transfer_internal`, the bridge commits the transfer message to state and emits `InitTransferEvent` regardless of whether the cross-contract token burn succeeds. The burn is fired with `.detach()`, meaning its result is never observed. If the burn fails silently, the destination chain will mint tokens that are not backed by a successful burn on the source chain, creating unbacked supply — a direct analog to the external report's "state updated even if transfer fails" class.

---

### Finding Description

`burn_tokens_if_needed` is called during `init_transfer_internal` to destroy deployed (bridged) tokens on NEAR before the cross-chain transfer is finalized on the destination chain. The function fires a cross-contract `burn` call using `.detach()`: [1](#0-0) 

`.detach()` in NEAR SDK means the promise is scheduled but its success or failure is **never checked**. The calling function `init_transfer_internal` has already committed the transfer message to storage at line 1835 and proceeds unconditionally to call `lock_tokens_if_needed` and emit `InitTransferEvent`: [2](#0-1) 

Specifically, the sequence is:
1. `add_transfer_message` — transfer committed to state (line 1835)
2. `burn_tokens_if_needed` — burn fired and **forgotten** (line 1851)
3. `lock_tokens_if_needed` — locked balance incremented (line 1853)
4. `InitTransferEvent` emitted — cross-chain relay triggered (line 1863)

If the burn at step 2 fails (e.g., `BURN_TOKEN_GAS` is insufficient, or the token contract is in an unexpected state), steps 3 and 4 still execute. The MPC/relayer layer observes the `InitTransferEvent` and proceeds to finalize the transfer on the destination chain, minting new tokens there — while the source tokens were never actually destroyed.

---

### Impact Explanation

**Critical — unauthorized mint of bridged assets / bridge collateralization broken.**

For a deployed (bridged) token, the invariant is: burn on source chain ↔ mint on destination chain. If the burn silently fails:

- Source tokens remain in the bridge contract's balance (not destroyed).
- The destination chain mints an equivalent amount of tokens based on the recorded transfer.
- Net result: total supply of the bridged asset increases without backing, breaking collateralization.
- The unburned source tokens remain accessible; a subsequent bridge-back from destination to source would mint again on NEAR, doubling supply.

This matches the "balance/accounting corruption that breaks bridge collateralization" and "unauthorized mint of bridged assets" impact categories.

---

### Likelihood Explanation

The burn is dispatched with a fixed `BURN_TOKEN_GAS` gas budget. In NEAR, cross-contract calls can fail due to:

- Insufficient prepaid gas propagated to the burn sub-call (gas exhaustion in a complex call chain).
- The deployed token contract being paused, upgraded, or having a non-standard burn path that consumes more gas than `BURN_TOKEN_GAS` allocates.

An unprivileged user initiates this path by calling `ft_transfer_call` on a deployed bridge token with the bridge as recipient, which triggers `ft_on_transfer` → `init_transfer_internal` → `burn_tokens_if_needed`. No privileged role is required to reach the vulnerable code path.

---

### Recommendation

Replace the fire-and-forget `.detach()` pattern with a chained callback that checks the burn result. If the burn fails, the transfer message should be removed and the tokens refunded to the sender — mirroring the existing `remove_transfer_message_without_refund` pattern already used in `init_transfer_internal` for other failure cases:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) -> Option<Promise> {
    if self.is_deployed_token(&token) {
        Some(
            ext_token::ext(token)
                .with_static_gas(BURN_TOKEN_GAS)
                .burn(amount)
        )
    } else {
        None
    }
}
```

Then chain a callback in `init_transfer_internal` that reverts the transfer message and locked balance if the burn promise fails, analogous to how `fin_transfer_send_tokens_callback` handles `send_tokens` failures.

### Proof of Concept

1. User holds 1000 units of a deployed bridge token (e.g., `usdc.bridge.near`).
2. User calls `ft_transfer_call` on `usdc.bridge.near`, transferring 1000 tokens to the bridge with a valid `TransferMessage` targeting an EVM destination.
3. Bridge's `ft_on_transfer` is invoked → `init_transfer_internal` is called.
4. `add_transfer_message` records the transfer (state committed).
5. `burn_tokens_if_needed` fires `ext_token::burn(1000).detach()`. Suppose `BURN_TOKEN_GAS` is insufficient for the token's burn logic — the burn sub-call runs out of gas and fails. Because of `.detach()`, this failure is invisible to the bridge.
6. `lock_tokens_if_needed` increments the locked balance.
7. `InitTransferEvent` is emitted with the full transfer details.
8. The MPC relayer observes the event and produces a valid signature for the destination chain.
9. On the EVM destination, `finTransfer` is called — 1000 tokens are minted to the recipient.
10. Result: 1000 source tokens remain unburned in the bridge; 1000 new tokens exist on the destination chain. Total supply inflated by 1000 units with no backing. [1](#0-0) [3](#0-2)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
