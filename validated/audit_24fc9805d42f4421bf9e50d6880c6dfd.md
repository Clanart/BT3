### Title
Unbounded `max_price_per_unit` Causes Felt Arithmetic Overflow in `compute_max_possible_fee`, Enabling Fee Cap Bypass and Network Halt - (File: `execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the fee cap using unchecked felt-field multiplication of `max_amount` (bounded to `[0, 2^64-1]`) and `max_price_per_unit` (only checked non-negative, no upper bound). Because Cairo arithmetic is modular over the field prime `P ≈ 2^251`, the product wraps to an arbitrary felt. The resulting `max_fee` is then used as the upper bound in `assert_nn_le(actual_fee, max_fee)`. If `max_fee` lands in `[P/2, P-1]` (the "negative" half of the field), that assertion always fails, causing the OS to panic and the block proof to be invalid — a network halt.

---

### Finding Description

**Step 1 — No upper bound on `max_price_per_unit`.**

In `pack_resource_bounds`, `max_amount` is correctly bounded to `[0, 2^64-1]`, but `max_price_per_unit` is only checked to be non-negative:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);   // ← no upper bound
    ...
}
``` [1](#0-0) 

This means `max_price_per_unit` can be any value in `[0, P-1]`.

**Step 2 — Unchecked felt multiplication in `compute_max_possible_fee`.**

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [2](#0-1) 

With `max_amount` up to `2^64-1` and `max_price_per_unit` up to `P-1`, the product `max_amount * max_price_per_unit` can reach `≈ 2^315`, wrapping modulo `P` an arbitrary number of times. The resulting `max_fee` can be any felt in `[0, P-1]`.

**Step 3 — `assert_nn_le` fails when `max_fee ∈ [P/2, P-1]`.**

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

Cairo's `assert_nn_le(a, b)` internally checks that `b - a` is in `[0, P/2)` (signed-felt non-negative). If `max_fee ∈ [P/2, P-1]`, then `max_fee - actual_fee ≈ P - actual_fee`, which is in `[P/2, P-1]` — the "negative" half — so the assertion fails unconditionally, halting the Cairo program.

**Concrete overflow example:**

Set `l1_gas_bounds.max_amount = 2` and `l1_gas_bounds.max_price_per_unit = (P+1)/2`. Then:

```
max_amount * max_price_per_unit = 2 * (P+1)/2 = P+1 ≡ 1 (mod P)
```

`max_fee = 1`. The sequencer's actual fee (e.g., `10^15` wei) exceeds 1, so `assert_nn_le` fails → OS panic.

Alternatively, set `max_price_per_unit = P - 1`:

```
1 * (P-1) = P-1  ∈ [P/2, P-1]
```

`max_fee = P-1`. `assert_nn_le(actual_fee, P-1)` checks `(P-1) - actual_fee ∈ [0, P/2)`, but `P-1-actual_fee ≈ P` is in `[P/2, P-1]` → assertion fails → OS panic.

---

### Impact Explanation

When the OS panics (Cairo assertion failure), the STARK proof for the block cannot be generated. The block is invalid and cannot be committed to L1. No further blocks can be built on top of it until the sequencer discards the offending transaction and re-processes the block. If the sequencer's off-chain fee validator uses integer arithmetic (not felt arithmetic), it will not detect the overflow and will include the transaction, triggering the OS panic.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a V3 transaction with a specific `max_price_per_unit` value that causes the overflow. The OS itself does not reject such transactions during `__validate__` (the overflow only manifests at `charge_fee`, after execution). The sequencer's off-chain simulation may use a different arithmetic model (e.g., Rust `u128`) and not reproduce the felt overflow, causing it to include the transaction. The attack requires no privileged access, no leaked keys, and no third-party compromise.

---

### Recommendation

Add an explicit upper bound check on `max_price_per_unit` in `pack_resource_bounds`, consistent with the bound already applied to `max_amount`:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // add this
    ...
}
``` [1](#0-0) 

This ensures that `max_amount * max_price_per_unit ≤ (2^64-1)(2^128-1) < 2^192 ≪ P`, preventing felt overflow in `compute_max_possible_fee`. [4](#0-3) 

---

### Proof of Concept

1. Construct a V3 invoke transaction with:
   - `l1_gas_bounds.max_amount = 1`
   - `l1_gas_bounds.max_price_per_unit = P - 1` (maximum felt value)
   - All other resource bounds set to 0
2. Submit to the sequencer. The transaction hash is computed correctly (the hash function also uses `assert_nn`, not `assert_nn_le`, so it accepts the value).
3. The sequencer's off-chain validator computes `max_fee = 1 * (P-1) = P-1` in felt, but if using integer arithmetic it computes a large integer and considers the fee cap "very high" — it includes the transaction.
4. The OS executes the transaction. `compute_max_possible_fee` returns `P-1 ∈ [P/2, P-1]`.
5. `assert_nn_le(actual_fee, P-1)` checks `(P-1) - actual_fee`. For any `actual_fee > 0`, this value is in `[P/2, P-1]` (negative felt), so the assertion fails.
6. The Cairo OS program halts. The block proof cannot be generated. The network cannot confirm new transactions. [5](#0-4)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
