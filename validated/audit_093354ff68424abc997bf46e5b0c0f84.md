### Title
Unconstrained `is_reverted` Flag Allows Malicious Prover to Arbitrarily Skip or Force Transaction Execution — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a no-op that performs zero validation. The `is_reverted` flag, loaded from an unverified hint inside `execute_invoke_function_transaction`, is never constrained by any Cairo assertion. A malicious sequencer/prover can freely set this flag to any value, arbitrarily skipping the execute step of valid transactions (while still charging fees) or forcing execution of transactions that should have reverted.

---

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`, lines 338–341), the revert status of a transaction is loaded from a hint and then passed to `check_is_reverted`:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
``` [1](#0-0) 

The `check_is_reverted` function in `execution_constraints.cairo` is defined as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This function contains **no Cairo assertions, no range checks, and no constraints of any kind**. It is a pure no-op.

In Cairo's proving model, hint blocks (`%{ ... %}`) are executed by the prover but are **not part of the proof constraints** — only Cairo instructions (assignments, `assert`, arithmetic) create verifiable constraints. Because `check_is_reverted` adds no constraints, the prover can set `is_reverted` to `0` or `1` freely, and the resulting proof will be accepted by the verifier regardless of the actual execution outcome.

The unconstrained flag then directly controls whether the execute step runs:

```cairo
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    non_reverting_select_execute_entry_point_func(
        block_context=block_context, execution_context=updated_tx_execution_context
    );
}
``` [3](#0-2) 

The same pattern applies to `execute_l1_handler_transaction`, where `is_reverted` is used directly from a hint with **no call to `check_is_reverted` at all**:

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [4](#0-3) 

The analog to M-09 is exact: just as `tokenURI` was supposed to enforce a spec-required check but delegated to `nftRenderer` without it, `execute_invoke_function_transaction` is supposed to enforce that `is_reverted` reflects the true execution outcome but delegates to `check_is_reverted`, which enforces nothing.

---

### Impact Explanation

A malicious sequencer/prover can exploit this in two directions:

1. **Set `is_reverted = TRUE` for a valid transaction**: The execute step is skipped entirely. Fees are still charged (via `charge_fee`, which runs unconditionally after the branch). Any state changes that the execute step would have produced — including token transfers, storage writes, or cross-contract calls — are permanently suppressed. This constitutes **permanent freezing of user funds**.

2. **Set `is_reverted = FALSE` for a transaction whose execution reverted**: The execute step runs and its state changes are committed to the state tree, even though the execution failed. This can result in **direct loss of funds** by applying state transitions that the protocol should have rolled back.

Both outcomes are accepted by the L1 verifier because no Cairo constraint is violated — the proof is mathematically valid even with the manipulated flag.

---

### Likelihood Explanation

The sequencer controls the prover and therefore controls all hint values. The Cairo OS is the mechanism that is supposed to constrain the sequencer's behavior even when the sequencer is adversarial — that is the entire purpose of the ZK proof. Because `check_is_reverted` is a no-op, this constraint is absent. Any sequencer that controls the proving pipeline can exploit this on any invoke or L1-handler transaction in any block, with no special preconditions.

---

### Recommendation

`check_is_reverted` must be implemented to actually constrain `is_reverted` against the true execution result. The correct approach is to derive `is_reverted` from the return value of the entry point execution itself (i.e., from `entry_point_return_values.failure_flag` as already done inside `execute_entry_point`), rather than loading it from an unconstrained hint. The hint-loaded value should be eliminated or replaced with a Cairo-level assertion that ties `is_reverted` to the verifiable output of the execution.

```cairo
// Instead of:
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);  // no-op

// Should be derived from the actual execution return value and asserted:
let is_reverted = actual_execution_result.failure_flag;
assert is_reverted = expected_is_reverted;  // constrained
```

---

### Proof of Concept

1. Sequencer includes a valid user invoke transaction (e.g., an ERC-20 `transfer` call).
2. Sequencer sets `is_reverted = 1` (TRUE) in the `IsReverted` hint.
3. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
4. The `if (is_reverted == FALSE)` branch is not taken; the execute step is skipped.
5. `charge_fee` runs unconditionally — the user is charged the full fee.
6. The token transfer is never executed; the recipient receives nothing.
7. The Cairo proof is generated and is mathematically valid (no constraint was violated).
8. The L1 verifier accepts the proof and the block is finalized.
9. The user's funds are permanently frozen with no recourse.
