### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by the caller is an actually-declared class. Any contract can invoke `replace_class` with an arbitrary, undeclared class hash, permanently rendering itself unexecutable and freezing all funds it holds. The code itself contains a past-deadline TODO acknowledging this missing check.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `class_hash` directly from the user-supplied syscall request and writes it into `contract_state_changes` with no cross-check against the declared-class registry (`contract_class_changes`):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

Once the contract's class hash is set to an undeclared value, the ZK prover cannot supply a valid compiled class for it. The sequencer cannot include any transaction targeting that contract in a provable block. The contract becomes permanently unexecutable, and any funds it holds are permanently frozen with no recovery path.

The TODO comment is dated `1/1/2026`; today is `7/3/2026` — the deadline has passed and the check remains absent.

The `replace_class` syscall is dispatched from `execute_syscalls.cairo` and is callable by any executing contract on itself, with no OS-level caller restriction beyond gas. [2](#0-1) 

---

### Impact Explanation

Any funds held by a contract whose class hash has been replaced with an undeclared felt are permanently frozen. Recovery is impossible: `replace_class` can only be called by the contract itself (the `contract_address` implicit argument is the currently-executing contract), and if the class is invalid the contract can never be executed again. This matches **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The attack requires only that an attacker:
1. Deploy a contract (a standard, permissionless operation).
2. Attract user deposits.
3. Call `replace_class` with any undeclared felt (e.g., `1`).

No privileged role, leaked key, or external dependency is needed. The syscall is available to every contract with sufficient gas. The acknowledged TODO confirms the missing check is a known gap, not an edge case.

---

### Recommendation

Before writing the new class hash to state, verify that `class_hash` is present in `contract_class_changes` (declared in the current or a prior block). If the class hash is not declared, write a failure response and do not update the state — analogous to how other syscalls handle invalid arguments:

```cairo
// Pseudocode addition inside execute_replace_class, after reading class_hash:
let (declared_compiled_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
if (declared_compiled_hash == 0) {
    write_failure_response(remaining_gas=remaining_gas, failure_felt=ERROR_INVALID_ARGUMENT);
    return ();
}
```

---

### Proof of Concept

1. Attacker declares and deploys `MaliciousVault` (a valid Sierra class) that:
   - Accepts STRK deposits from users.
   - Exposes an owner-only function that issues the `replace_class` syscall with `class_hash = 0x1` (an undeclared felt).
2. Users deposit funds into `MaliciousVault`.
3. Attacker calls the owner function; the OS executes `execute_replace_class`:
   - `request.class_hash = 0x1` is read.
   - The TODO-guarded validation is absent.
   - `dict_update` writes `class_hash=0x1` into `contract_state_changes` for `MaliciousVault`'s address. [3](#0-2) 
4. In all subsequent blocks, any transaction targeting `MaliciousVault` requires the prover to supply a compiled class for hash `0x1`. No such class exists in `compiled_class_facts`; the block cannot be proven.
5. All deposited funds are permanently frozen. [4](#0-3)
