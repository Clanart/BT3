### Title
Nil-Pointer Dereference Panic in `ToTransaction` SetCodeTx Branch Causes Node Crash — (`x/evm/types/tx_args.go`)

### Summary

`ToTransaction` unconditionally dereferences `args.To` at line 93 in the `AuthorizationList != nil` branch. `SetTxDefaults` does not set `args.To` for SetCodeTx transactions and only rejects a nil `To` when there is no input data. An attacker who supplies both `authorizationList` and non-empty `data` while omitting `to` bypasses the only guard, causing a nil-pointer dereference panic that crashes the node.

### Finding Description

In `ToTransaction`, the `AuthorizationList != nil` branch directly dereferences `args.To` without a nil guard:

```go
// x/evm/types/tx_args.go lines 92-103
data = &types.SetCodeTx{
    To:         *args.To,                                        // ← panics if nil
    ChainID:    uint256.MustFromBig(args.ChainID.ToInt()),
    Nonce:      uint64(*args.Nonce),
    Gas:        uint64(*args.Gas),
    GasFeeCap:  uint256.MustFromBig((*big.Int)(args.MaxFeePerGas)),
    GasTipCap:  uint256.MustFromBig((*big.Int)(args.MaxPriorityFeePerGas)),
    Value:      uint256.MustFromBig((*big.Int)(args.Value)),
    ...
}
``` [1](#0-0) 

`SetTxDefaults` does populate `Value`, `Nonce`, `Gas`, `ChainID`, `MaxFeePerGas`, and `MaxPriorityFeePerGas` before `ToTransaction` is called. However, its only handling of `args.To == nil` is the contract-creation guard:

```go
// rpc/backend/call_tx.go lines 294-305
if args.To == nil {
    // Contract creation
    ...
    if len(input) == 0 {
        return args, errors.New("contract creation without any data provided")
    }
}
``` [2](#0-1) 

If `args.To == nil` **and** `args.Data` (or `args.Input`) is non-empty, `SetTxDefaults` returns without error and without setting `args.To`. The call then proceeds to `args.ToTransaction()` at line 55 of `SendTransaction`:

```go
// rpc/backend/sign_tx.go lines 50-55
args, err = b.SetTxDefaults(args)
if err != nil {
    return common.Hash{}, err
}
msg := args.ToTransaction()   // ← panics here
``` [3](#0-2) 

Note that `ToEthTransaction` (the simulator path) correctly guards this with an explicit nil check, but `ToTransaction` does not:

```go
// x/evm/types/tx_args.go lines 393-396 — ToEthTransaction has the guard
to := common.Address{}
if args.To != nil {
    to = *args.To
}
``` [4](#0-3) 

The other pointer dereferences in the same branch (`*args.Nonce`, `*args.Gas`, `args.ChainID.ToInt()`) are safe after `SetTxDefaults` because those fields are always populated. `uint256.MustFromBig(nil)` also does not panic (the library returns zero for a nil `*big.Int`). The sole exploitable panic is `*args.To`.

### Impact Explanation

A Go nil-pointer dereference is an unrecovered panic. Because Ethermint does not wrap `ToTransaction` in a recover, the panic propagates up and crashes the node process. A crashed validator node stops producing or voting on blocks, which — if enough validators are affected — halts the chain. This matches the **Critical** impact category: *valid unprivileged transaction or RPC submission path can halt the chain*.

### Likelihood Explanation

`eth_sendTransaction` requires the `from` address to be present in the node's keyring (checked at `sign_tx.go:40-44`). The attacker must therefore know a keyring account address, which is trivially discoverable via the public `eth_accounts` RPC method. No key material or signature is needed — the panic fires at `ToTransaction` (line 55), well before the `msg.Sign` call (line 76). Any node that exposes `eth_sendTransaction` over a public or semi-public RPC endpoint is directly vulnerable.

### Recommendation

Add a nil guard for `args.To` in the `AuthorizationList != nil` branch of `ToTransaction`, mirroring the pattern already used in `ToEthTransaction`:

```go
case args.AuthorizationList != nil:
    to := common.Address{}
    if args.To != nil {
        to = *args.To
    }
    data = &types.SetCodeTx{
        To: to,
        ...
    }
```

Additionally, `SetTxDefaults` should explicitly reject a nil `To` when `AuthorizationList` is non-nil, since EIP-7702 mandates a `to` field:

```go
if args.AuthorizationList != nil && args.To == nil {
    return args, errors.New("SetCode (EIP-7702) transaction requires a 'to' address")
}
```

### Proof of Concept

```go
// Reproduces the panic without any network or keyring dependency.
func TestToTransactionSetCodeNilToPanic(t *testing.T) {
    authList := []ethtypes.SetCodeAuthorization{{}}
    chainID  := hexutil.Big(*big.NewInt(9001))
    nonce    := hexutil.Uint64(1)
    gas      := hexutil.Uint64(21000)
    val      := hexutil.Big(*big.NewInt(0))
    maxFee   := hexutil.Big(*big.NewInt(2e9))
    maxTip   := hexutil.Big(*big.NewInt(1e8))

    args := evmtypes.TransactionArgs{
        // To is intentionally nil
        ChainID:              &chainID,
        Nonce:                &nonce,
        Gas:                  &gas,
        Value:                &val,
        MaxFeePerGas:         &maxFee,
        MaxPriorityFeePerGas: &maxTip,
        AuthorizationList:    authList,
    }
    // Must not panic:
    require.NotPanics(t, func() { args.ToTransaction() })
}
```

The RPC-level trigger is:
```json
{
  "method": "eth_sendTransaction",
  "params": [{
    "from":              "<known keyring address>",
    "authorizationList": [{}],
    "data":              "0xdeadbeef"
  }]
}
```

`SetTxDefaults` fills `Gas`, `ChainID`, `Nonce`, `Value`, `MaxFeePerGas`, and `MaxPriorityFeePerGas`, but leaves `To` as nil because `data` is non-empty (bypassing the contract-creation guard). `ToTransaction` then panics at `*args.To`.
