### Title
Unguarded `initSubscriptions()` in `RPCStream` Allows Concurrent Callers to Race on Shared Stream Pointers and Spawn Duplicate CometBFT Subscriptions - (File: `rpc/stream/rpc.go`)

---

### Summary

`RPCStream.initSubscriptions()` performs a nil-check on `s.headerStream` and then writes to `s.headerStream`, `s.logStream`, and spawns a `start()` goroutine — all without any mutex or `sync.Once`. Because `HeaderStream()` and `LogStream()` are called directly from concurrent JSON-RPC request handlers, two simultaneous unprivileged callers can both pass the nil-check, both write the stream pointers, both subscribe to CometBFT events under the same subscriber name, and both launch a `start()` goroutine. This is a data race on shared mutable state that is directly analogous to the external report's `validateBlobRequest` / `quorumCount` pattern.

---

### Finding Description

`initSubscriptions()` is the lazy initializer for the two shared stream objects:

```go
// rpc/stream/rpc.go
func (s *RPCStream) initSubscriptions() {
    if s.headerStream != nil {   // ← unsynchronized read
        return
    }
    s.headerStream = NewStream[RPCHeader](...)          // ← unsynchronized write
    s.logStream    = NewStream[*ethtypes.Log](...)      // ← unsynchronized write

    chBlocks, err := s.evtClient.Subscribe(ctx, streamSubscriberName, blockEvents, ...)
    if err != nil {
        panic(err)   // ← reachable panic
    }
    chLogs, err := s.evtClient.Subscribe(ctx, streamSubscriberName, evmEvents, ...)
    if err != nil {
        ...
        panic(err)   // ← reachable panic
    }
    go s.start(&s.wg, chBlocks, chLogs)   // ← goroutine spawned without guard
}
``` [1](#0-0) 

It is called without any lock from two public accessors:

```go
func (s *RPCStream) HeaderStream() *Stream[RPCHeader] {
    s.initSubscriptions()
    return s.headerStream
}
func (s *RPCStream) LogStream() *Stream[*ethtypes.Log] {
    s.initSubscriptions()
    return s.logStream
}
``` [2](#0-1) 

Those accessors are invoked directly from the public filter API on every `eth_newBlockFilter`, `eth_newFilter`, and `eth_subscribe` (newHeads/logs) call:

- `NewBlockFilter()` → `api.events.HeaderStream()` [3](#0-2) 
- `NewFilter()` → `api.events.LogStream()` [4](#0-3) 

The JSON-RPC server handles each request in its own goroutine. There is no serialization between concurrent callers of `HeaderStream()` / `LogStream()`.

**Race sequence (two concurrent callers):**

| Step | Goroutine A | Goroutine B |
|------|-------------|-------------|
| 1 | reads `s.headerStream == nil` → true | |
| 2 | | reads `s.headerStream == nil` → true |
| 3 | writes `s.headerStream = NewStream[...]` | |
| 4 | | writes `s.headerStream = NewStream[...]` (overwrites A) |
| 5 | subscribes CometBFT `streamSubscriberName/blockEvents` | |
| 6 | | subscribes CometBFT `streamSubscriberName/blockEvents` (duplicate) |
| 7 | `go s.start(chBlocks_A, chLogs_A)` | |
| 8 | | `go s.start(chBlocks_B, chLogs_B)` | [1](#0-0) 

---

### Impact Explanation

Three concrete consequences arise:

1. **Data race / undefined behavior on pointer writes.** Concurrent unsynchronized writes to `s.headerStream` and `s.logStream` are a Go memory-model violation. Without the race detector, this can produce torn pointer values, nil dereferences, or silent memory corruption in the stream objects returned to callers.

2. **Panic crashing the node process.** CometBFT's subscription system rejects a second `Subscribe()` call with the same `(subscriber, query)` pair. When goroutine B calls `s.evtClient.Subscribe(ctx, streamSubscriberName, blockEvents, ...)` and CometBFT returns an error, `initSubscriptions()` unconditionally calls `panic(err)`. [5](#0-4)  If the go-ethereum RPC server's panic recovery does not catch this (it runs in the request goroutine, not a recovered handler), the entire node process terminates, halting the validator.

3. **Duplicate event delivery.** If both goroutines succeed in subscribing, two `start()` goroutines run concurrently, both writing to the same `s.headerStream` / `s.logStream`. Every new block or EVM log is written twice, causing `eth_subscribe` (newHeads/logs) and `eth_getFilterChanges` to deliver duplicate headers and logs to all subscribers. This feeds incorrect data to any client relying on these streams for transaction monitoring or block tracking. [6](#0-5) 

---

### Likelihood Explanation

The race window is small but reliably triggerable: an attacker sends two concurrent `eth_newBlockFilter` (or `eth_subscribe newHeads`) requests to the JSON-RPC endpoint before the server has processed any filter request. This is a one-time race at server startup or after a restart. No authentication, no privileged key, and no special transaction is required — a plain HTTP POST to the public JSON-RPC port suffices. The `PublicFilterAPI` is enabled by default in Ethermint's JSON-RPC configuration.

---

### Recommendation

Replace the manual nil-check pattern with `sync.Once`:

```go
type RPCStream struct {
    ...
    initOnce sync.Once
}

func (s *RPCStream) initSubscriptions() {
    s.initOnce.Do(func() {
        s.headerStream = NewStream[RPCHeader](...)
        s.logStream    = NewStream[*ethtypes.Log](...)
        // subscribe and start goroutine
    })
}
```

`sync.Once` guarantees exactly-once execution with full memory-model safety, eliminating the data race, the duplicate-subscription panic, and the duplicate-event delivery.

### Proof of Concept

```
# Terminal 1 – send eth_newBlockFilter
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_newBlockFilter","params":[],"id":1}' &

# Terminal 2 – send eth_newFilter concurrently (races initSubscriptions)
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_newFilter","params":[{"fromBlock":"latest","toBlock":"latest"}],"id":2}' &

wait
```

Both requests reach `initSubscriptions()` simultaneously before `s.headerStream` is set. The race produces one of: (a) a panic from the duplicate CometBFT subscription, crashing the node; (b) two `start()` goroutines delivering duplicate events to all `eth_subscribe` / `eth_getFilterChanges` consumers; or (c) a torn pointer write causing a nil-dereference in a subsequent stream read. 	wg *sync.WaitGroup,
	chBlocks <-chan coretypes.ResultEvent,
	chLogs <-chan coretypes.ResultEvent,
) {
	wg.Add(1)
	defer func() {
		wg.Done()
		if err := s.evtClient.UnsubscribeAll(context.Background(), streamSubscriberName); err != nil {
			s.logger.Error("failed to unsubscribe", "err", err)
		}
	}()

	for {
		select {
		case ev, ok := <-chBlocks:
			if !ok {
				chBlocks = nil
				break
			}

			data, ok := ev.Data.(tmtypes.EventDataNewBlock)
			if !ok {
				s.logger.Error("event data type mismatch", "type", fmt.Sprintf("%T", ev.Data))
				continue
			}

			baseFee := types.BaseFeeFromEvents(data.ResultFinalizeBlock.Events)
			res, err := s.validatorAccount(
				types.ContextWithHeight(data.Block.Height),
				&evmtypes.QueryValidatorAccountRequest{
					ConsAddress: sdk.ConsAddress(data.Block.Header.ProposerAddress).String(),
				},
			)
			if err != nil {
				s.logger.Error("failed to get validator account", "err", err)
				continue
			}
			validator, err := sdk.AccAddressFromBech32(res.AccountAddress)
			if err != nil {
				s.logger.Error("failed to convert validator account", "err", err)

// NewFilter creates a new filter and returns the filter id. It can be
// used to retrieve logs when the state changes. This method cannot be
// used to fetch logs that are already stored in the state.
//
// Default criteria for the from and to block are "latest".
// Using "latest" as block number will return logs for mined blocks.
// Using "pending" as block number returns logs for not yet mined (pending) blocks.
// In case logs are removed (chain reorg) previously returned logs are returned
// again but with the removed property set to true.
//
// In case "fromBlock" > "toBlock" an error is returned.
//
// https://github.com/ethereum/wiki/wiki/JSON-RPC#eth_newfilter
func (api *PublicFilterAPI) NewFilter(criteria filters.FilterCriteria) (rpc.ID, error) {
	api.filtersMu.Lock()
	defer api.filtersMu.Unlock()

	if len(api.filters) >= int(api.backend.RPCFilterCap()) {
		return rpc.ID(""), fmt.Errorf("error creating filter: max limit reached")
	}

	if criteria.FromBlock != nil && criteria.ToBlock != nil &&
		criteria.FromBlock.Int64() >= 0 && criteria.ToBlock.Int64() >= 0 &&
		criteria.FromBlock.Int64() > criteria.ToBlock.Int64() {
		return rpc.ID(""), &types.InvalidParamsError{Message: "invalid block range params"}
	}

	id := rpc.NewID()
	_, offset := api.events.LogStream().ReadNonBlocking(-1)
	api.filters[id] = &filter{
		typ:      filters.LogsSubscription,
		deadline: time.NewTimer(deadline),
		crit:     criteria,
		offset:   offset,
	}

	return id, nil
}
