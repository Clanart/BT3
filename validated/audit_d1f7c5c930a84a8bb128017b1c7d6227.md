### Title
Stale `feemarketParams` Snapshot in `NewDynamicFeeChecker` Allows Cosmos SDK Transactions to Bypass Current Base Fee â€” (File: `ante/evm/fee_checker.go`, `evmd/ante/handler_options.go`)

### Summary
`newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712` read `evmParams` and `feemarketParams` once from the keeper at ante-handler **construction time** and pass them as captured pointers into `NewDynamicFeeChecker`. Because the fee-market base fee is updated every block, the closure always validates Cosmos SDK transaction fees against a stale (genesis-era) base fee rather than the live on-chain value. The EVM ante handler path (`newEthAnteHandler`) does not share this flaw â€” it fetches fresh params per-transaction via `EVMBlockConfig`. This is the direct Ethermint analog of the "user-settable decimals" class: a value that must be read from a canonical on-chain source is instead captured once as a caller-supplied snapshot.

---

### Finding Description

**Root cause â€” `newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712` snapshot params at construction time:**

```go
// evmd/ante/handler_options.go  (newCosmosAnteHandler, lines 179-187)
evmParams      := options.EvmKeeper.GetParams(ctx)          // read ONCE
feemarketParams := options.FeeMarketKeeper.GetParams(ctx)   // read ONCE
evmDenom       := evmParams.EvmDenom
...
txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)
``` [1](#0-0) 

The same pattern appears in `newLegacyCosmosAnteHandlerEip712`: [2](#0-1) 

**`NewDynamicFeeChecker` uses the captured (stale) pointer for every subsequent transaction:**

```go
// ante/evm/fee_checker.go  lines 54-56
denom   := evmParams.EvmDenom                                    // stale
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams) // stale BaseFee
``` [3](#0-2) 

`feemarketParams.BaseFee` is the live base fee stored in the fee-market module's params store; it is updated every block by the fee-market `EndBlock` handler. Because the closure holds a pointer to the local variable captured at construction time, it never sees subsequent updates.

**Contrast with the EVM ante handler, which fetches fresh params per-transaction:**

```go
// evmd/ante/handler_options.go  newEthAnteHandler, lines 88-96
blockCfg, err := options.EvmKeeper.EVMBlockConfig(ctx, options.EvmKeeper.ChainID())
evmParams      := &blockCfg.Params           // fresh every call
feemarketParams := &blockCfg.FeeMarketParams // fresh every call
baseFee        := blockCfg.BaseFee           // fresh every call
``` [4](#0-3) 

The inconsistency is structural: the EVM path reads from state on every invocation; the Cosmos path reads from a frozen snapshot.

---

### Impact Explanation

`NewDynamicFeeChecker` is wired into `authante.NewDeductFeeDecorator` for all Cosmos SDK (non-EVM) transactions. [5](#0-4) 

The checker computes `feeCap = feeCoins.AmountOf(denom) / gas` and rejects the transaction only if `feeCap < baseFeeInt`. [6](#0-5) 

Because `baseFeeInt` is derived from the stale `feemarketParams.BaseFee`, any Cosmos SDK transaction whose fee satisfies the **genesis** base fee (but not the **current**, higher base fee) passes the check and has its fees deducted at the stale rate. Validators and the fee collector receive less than the protocol-mandated amount â€” a direct fee mis-accounting impact matching the allowed High impact category: *"fee market, ante handler bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

Additionally, `evmDenom` is also frozen at construction time and passed to `MinGasPriceDecorator`: [7](#0-6) 

If governance updates `evm_denom` via `MsgUpdateParams`, the Cosmos ante handler continues enforcing the old denomination, causing all legitimate post-upgrade Cosmos SDK transactions to fail (denial of service) or to have their fees silently ignored (`AmountOf` returns 0 for an unknown denom, making `feeCap = 0 < baseFeeInt` and rejecting them).

---

### Likelihood Explanation

The fee-market base fee adjusts every block. Any deployment where block utilization causes the base fee to drift from its genesis value â€” which is the normal operating condition â€” immediately creates a gap between the stale value used by the Cosmos ante handler and the live value used by the EVM ante handler. No privileged action is required; any unprivileged user submitting a Cosmos SDK transaction (e.g., staking, governance, IBC) with a fee calibrated to the stale base fee triggers the mis-accounting path.

---

### Recommendation

Move the params read inside the returned closure so it executes per-transaction, mirroring `newEthAnteHandler`:

```go
// newCosmosAnteHandler â€” inside the returned sdk.AnteHandler closure
blockCfg, err := options.EvmKeeper.EVMBlockConfig(ctx, options.EvmKeeper.ChainID())
if err != nil { ... }
txFeeChecker = evm.NewDynamicFeeChecker(
    blockCfg.ChainConfig,
    &blockCfg.Params,
    &blockCfg.FeeMarketParams,
)
```

Alternatively, refactor `NewDynamicFeeChecker` to accept keeper interfaces and read params from state on each invocation, eliminating the snapshot entirely â€” the same fix recommended for the "user-settable decimals" class: retrieve the authoritative value from its canonical source rather than accepting it as a caller-supplied parameter.

---

### Proof of Concept

1. Chain starts with genesis `feemarketParams.BaseFee = 100`.
2. `newCosmosAnteHandler` is constructed at app init; the closure captures `feemarketParams.BaseFee = 100`.
3. After sustained high block utilization, the live base fee rises to `10_000` (stored in the fee-market params store and used correctly by `newEthAnteHandler`).
4. An attacker submits a Cosmos SDK `MsgDelegate` with `fee = 100 * gasLimit` (satisfying the stale base fee).
5. `NewDynamicFeeChecker` computes `baseFee = GetBaseFee(..., feemarketParams)` â†’ `100` (stale), accepts the transaction, and deducts only `100 * gasLimit` from the sender.
6. The correct fee should have been `10_000 * gasLimit`; the fee collector is under-credited by `9_900 * gasLimit` per transaction.
7. The same transaction would be rejected by the EVM ante handler path, which reads the live `10_000` base fee â€” demonstrating the inconsistency is exploitable specifically through the Cosmos SDK transaction path. [8](#0-7) [9](#0-8)

func NewDynamicFeeChecker(ethCfg *params.ChainConfig, evmParams *types.Params, feemarketParams *feemarkettypes.Params) authante.TxFeeChecker {
	return func(ctx sdk.Context, tx sdk.Tx) (sdk.Coins, int64, error) {
		feeTx, ok := tx.(sdk.FeeTx)
		if !ok {
			return nil, 0, fmt.Errorf("tx must be a FeeTx")
		}

		if ctx.BlockHeight() == 0 {
			// genesis transactions: fallback to min-gas-price logic
			return checkTxFeeWithValidatorMinGasPrices(ctx, feeTx)
		}

		denom := evmParams.EvmDenom

		baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
		if baseFee == nil {
			// london hardfork is not enabled: fallback to min-gas-prices logic
			return checkTxFeeWithValidatorMinGasPrices(ctx, feeTx)
		}

		// default to `MaxInt64` when there's no extension option.
		maxPriorityPrice := sdkmath.NewInt(math.MaxInt64)

		// get the priority tip cap from the extension option.
		if hasExtOptsTx, ok := tx.(authante.HasExtensionOptionsTx); ok {
			for _, opt := range hasExtOptsTx.GetExtensionOptions() {
				if extOpt, ok := opt.GetCachedValue().(*ethermint.ExtensionOptionDynamicFeeTx); ok {
					maxPriorityPrice = extOpt.MaxPriorityPrice
					break
				}
			}
		}

		if maxPriorityPrice.Sign() == -1 {
			return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, "priority fee is negative")
		}

		gas := feeTx.GetGas()
		feeCoins := feeTx.GetFee()
		fee := feeCoins.AmountOf(denom)

		feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
		baseFeeInt := sdkmath.NewIntFromBigInt(baseFee)

		if feeCap.LT(baseFeeInt) {
			return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, "insufficient gas prices; got: %s required: %s", feeCap, baseFeeInt)
		}

		// calculate the effective gas price using the EIP-1559 logic.
		effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))

		// NOTE: create a new coins slice without having to validate the denom
		effectiveFee := sdk.Coins{
			{
				Denom:  denom,
				Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)),
			},
		}

		bigPriority := effectivePrice.Sub(baseFeeInt).Quo(types.DefaultPriorityReduction)
		priority := int64(math.MaxInt64)

		if bigPriority.IsInt64() {
			priority = bigPriority.Int64()
		}

		return effectiveFee, priority, nil
	}
}
