### Title
Missing Zero-Value Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the supplied `class_hash` is non-zero. A contract can invoke `replace_class(0)`, writing `UNINITIALIZED_CLASS_HASH` (= `0`) into its own state entry. Once committed, every subsequent transaction targeting that contract fails at the OS execution layer because no class exists for hash `0`. Any funds held by the contract are permanently frozen with no recovery path.

---

### Finding Description

In `execute_replace_class` the OS reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` without any existence or non-zero check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The constant `UNINITIALIZED_CLASS_HASH = 0` is the sentinel value the OS uses to represent an undeployed contract: [2](#0-1) 

When a subsequent invoke transaction targets the poisoned contract, `get_invoke_tx_execution_context` reads `class_hash = 0` from state and builds an `ExecutionContext` with it: [3](#0-2) 

`run_validate` then calls `non_reverting_select_execute_entry_point_func` with `class_hash = 0`. Because no Sierra class is registered under hash `0`, the execution fails. Since `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0`, any attempt by the sequencer to include such a transaction in a provable block is impossible: [4](#0-3) 

The sequencer is therefore forced to permanently exclude all transactions targeting the poisoned contract, making its stored funds irrecoverable.

The parallel to the original report is exact:

| Original (Solidity) | StarkNet OS (Cairo) |
|---|---|
| `deposit(0)` passes `_getPositionData` with no zero-liquidity check | `replace_class(0)` passes `execute_replace_class` with no zero-hash check |
| Zero-liquidity position stored in `positionAccounts` | `class_hash = 0` stored in `contract_state_changes` |
| `withdraw` loops linearly, hits zero-liquidity entry, reverts permanently | OS tries to execute contract with `class_hash = 0`, fails; sequencer cannot include the tx |
| `firstAvailablePositionAccountIndex` can never advance past the bad entry | No mechanism exists to un-poison the contract's class hash |
| All withdrawals frozen | All funds in the contract frozen |

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to `0` in committed state, every future transaction targeting it is unprovable. The contract's ERC-20 balances, NFT ownership records, or any other stored value become permanently inaccessible. There is no OS-level escape hatch: `replace_class` itself requires the contract to execute, which is impossible when `class_hash = 0`.

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

1. **Malicious contract path**: Attacker deploys a contract whose `__execute__` function conditionally calls `replace_class(0)` (e.g., after a deposit threshold is reached). Victims deposit funds. Attacker triggers the condition. Funds are frozen.
2. **Exploit-a-bug path**: A legitimate DeFi contract contains a bug allowing an arbitrary caller to invoke `replace_class` with a caller-controlled hash. Attacker passes `0`. Contract funds are frozen.

No privileged role, leaked key, or network-level attack is required. The `replace_class` syscall is a standard, publicly accessible StarkNet feature.

---

### Recommendation

Add an explicit non-zero check immediately after reading `class_hash` in `execute_replace_class`:

```cairo
let class_hash = request.class_hash;

// Reject zero (UNINITIALIZED_CLASS_HASH) and undeclared class hashes.
with_attr error_message("Invalid class hash: must be non-zero and declared.") {
    assert_not_zero(class_hash);
    // Additionally verify class_hash exists in contract_class_changes / declared classes.
}
```

This directly addresses the acknowledged TODO at line 898 and closes the analogous gap identified in the external report.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract whose `__execute__` entry point calls `replace_class(0)` after `total_deposits >= THRESHOLD`.
2. Victims deposit 500,000 STRK into `MaliciousVault` (threshold reached).
3. Attacker sends one more deposit, triggering the `replace_class(0)` syscall.
4. The OS executes `execute_replace_class` with `class_hash = 0`; no validation fires; `contract_state_changes` is updated with `class_hash = 0` for `MaliciousVault`.
5. State is committed to the Patricia tree.
6. Any subsequent `invoke` targeting `MaliciousVault` causes the OS to build an `ExecutionContext` with `class_hash = 0`, fail during `run_validate` (no class found), and the sequencer cannot produce a valid proof for that transaction.
7. 500,000 STRK is permanently frozen.
