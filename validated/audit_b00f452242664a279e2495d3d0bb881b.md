Audit Report

## Title
Unspent ETH Stranded in Router After Partial Fill in `exactInputSingle` With Price Limit — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary

`exactInputSingle` is `payable` and `PeripheryPayments.pay` wraps native ETH into WETH during the swap callback, consuming only the actual pool-consumed amount. When a price limit causes a partial fill, the difference between `msg.value` and the consumed amount remains as raw ETH in the router with no automatic refund. Because `refundETH()` is permissionless, any third party can immediately drain the stranded ETH, causing direct, permanent loss of user principal.

## Finding Description

**Step 1 — Native ETH is accepted and held by the router.**

`exactInputSingle` is `payable` and the `receive()` fallback only rejects ETH from non-WETH senders, so `msg.value` sent with the function call is held in `address(this).balance`. [1](#0-0) [2](#0-1) 

**Step 2 — `pay()` wraps only the actual consumed amount.**

When `token == WETH` and `payer != address(this)`, `pay()` checks `address(this).balance` and wraps exactly `value` (the pool-demanded amount) into WETH, forwarding it to the pool. If the pool partially filled (price limit hit), `value < msg.value`, and the remainder stays as raw ETH in the router. [3](#0-2) [4](#0-3) 

**Step 3 — `exactInputSingle` has no post-swap ETH refund.**

After the swap, the function only enforces `amountOutMinimum` and clears transient context. There is no `refundETH()` call and no guard that reverts on partial fill (unlike `exactInput`, which reverts with `InvalidInputAmountAtHop` when `amountInActual < amount`). [5](#0-4) [6](#0-5) 

**Step 4 — `refundETH()` is permissionless.**

Any address can call `refundETH()` and receive the entire ETH balance of the router, including the victim's unspent ETH. [7](#0-6) 

## Impact Explanation

A user who sends `N` ETH with `tokenIn = WETH` and a binding `priceLimitX64` will have only the partially consumed portion wrapped and forwarded to the pool. The remaining `(N - consumed)` ETH is left in the router and can be drained by any caller of `refundETH()` in the same or next block. This is a direct, permanent loss of user principal with no recovery path, meeting the High/Critical threshold for direct loss of funds.

## Likelihood Explanation

Setting a non-trivial `priceLimitX64` is a standard, documented feature of the router interface. Sending native ETH instead of pre-wrapped WETH is the natural UX path. MEV bots routinely monitor for stranded ETH in router contracts and call `refundETH()` atomically in the same block. No privileged access or malicious setup is required; any ordinary user transaction with a binding price limit triggers the loss.

## Recommendation

Add an automatic ETH refund at the end of `exactInputSingle` (and `exactOutputSingle`) after clearing the callback context:

```solidity
_clearExpectedCallbackPool();
if (address(this).balance > 0) {
    _transferETH(msg.sender, address(this).balance);
}
```

Alternatively, add a partial-fill guard analogous to `exactInput`'s `InvalidInputAmountAtHop` check, reverting if the pool consumed less than `params.amountIn`.

## Proof of Concept

```
1. Alice calls exactInputSingle({
       pool: WETH/T pool,
       tokenIn: WETH,
       zeroForOne: true,
       amountIn: 1e18,
       amountOutMinimum: 0,
       priceLimitX64: P   // binding price limit
   }, { value: 1e18 });
2. Pool fills only 0.6e18 before hitting price P.
3. metricOmmSwapCallback fires; pay() wraps and forwards 0.6e18 ETH to the pool.
4. exactInputSingle returns; 0.4e18 ETH remains in the router.
5. Alice receives tokens worth 0.6 ETH but spent 1 ETH.
6. Bob (MEV bot) calls router.refundETH() and receives 0.4e18 ETH.
7. Alice has permanently lost 0.4 ETH with no recourse.
```
