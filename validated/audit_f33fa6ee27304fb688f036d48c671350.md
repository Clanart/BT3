### Title
Silently Ignored Burn Promise Result in `burn_tokens_if_needed` Enables Unbacked Supply Creation — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `burn_tokens_if_needed` function in the NEAR bridge fires the `burn` cross-contract call with `.detach()`, meaning the success or failure of the burn is never observed. If the burn promise fails (e.g., due to gas exhaustion), the bridge has already emitted `InitTransferEvent` and returned `U128(0)` to the token contract, locking the tokens in the bridge. The destination chain will finalize the transfer and mint new tokens, creating unbacked supply without the source tokens being destroyed.

---

### Finding Description

In `init_transfer_internal`, after storing the transfer message and updating storage balances, the bridge calls `burn_tokens_if_needed`: [1](#0-0) 

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result is never checked
    }
}
```

The `.detach()` call fires the burn promise and immediately discards the result. There is no callback, no `#[callback_result]`, and no error path. Execution continues unconditionally to emit `InitTransferEvent` and return `U128(0)`: [2](#0-1) 

This is the exact analog of the original report: a function whose return code (success/failure of the burn) is ignored, and the caller continues processing — emitting the cross-chain event and releasing the tokens to the destination chain — as if the burn had succeeded.

The project's own security checklist acknowledges this risk: [3](#0-2) 

> "Check .detach() usage: Detached promises should only be used for non-critical operations"

The burn of a bridged token is not a non-critical operation — it is the mechanism that prevents double-supply.

---

### Impact Explanation

If `burn` fails silently:

1. The bridge holds `amount` tokens (transferred in via `ft_transfer_call`, never burned).
2. `InitTransferEvent` is already emitted on-chain.
3. The bridge returned `U128(0)` to the NEP-141 token contract, so no refund occurs — tokens are permanently locked in the bridge.
4. The destination chain (EVM, Solana, StarkNet) observes the event and finalizes the transfer, minting `amount` new tokens.

Result: source-chain tokens are locked (not burned) while destination-chain tokens are minted — **unbacked supply is created**, breaking bridge collateralization. This matches the allowed impact: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

The burn fails if `BURN_TOKEN_GAS` is insufficient for the token contract's `burn` execution. NEAR gas costs for storage-touching operations can vary; if the token contract's `burn` touches more storage slots than anticipated (e.g., after a token upgrade or storage layout change), the allocated gas may be exhausted. Because the promise is detached, this failure is silent and leaves no on-chain trace. An unprivileged user initiating a bridge transfer via `ft_transfer_call` is the direct entry path — no privileged role is required.

---

### Recommendation

Replace the detached burn with an awaited promise and a callback that verifies success before emitting `InitTransferEvent`. If the burn fails, the callback should refund the tokens to the sender and remove the stored transfer message, mirroring the existing error-handling pattern used for storage balance failures: [4](#0-3) 

The burn result must be treated as a critical gate — `InitTransferEvent` should only be emitted after the burn is confirmed successful.

---

### Proof of Concept

1. User calls `ft_transfer_call(bridge_account, 1000, transfer_msg)` on a bridge-deployed token.
2. Token contract transfers 1000 tokens to bridge and calls `ft_on_transfer`.
3. Bridge calls `init_transfer_internal` → `burn_tokens_if_needed` fires burn with `.detach()`.
4. Due to gas exhaustion, the burn promise fails — silently, with no callback.
5. Bridge emits `InitTransferEvent { amount: 1000, ... }` and returns `U128(0)`.
6. Token contract does not refund (bridge returned 0); 1000 tokens remain locked in bridge.
7. Relayer on destination chain observes `InitTransferEvent` and calls `finTransfer`.
8. Destination bridge mints 1000 new tokens to recipient.
9. Net result: 1000 tokens locked on NEAR (not burned) + 1000 tokens minted on destination = unbacked supply of 1000 tokens.
