### Title
Missing `shard_id` Binding Check in `set_state_header` Allows Wrong-Shard State Root to Be Committed - (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`set_state_header` accepts a network-supplied `ShardStateSyncResponseHeader` and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the chunk embedded in the header actually belongs to the requested `shard_id`. A malicious peer can supply a header containing a different shard's chunk (with a valid Merkle proof for that chunk), causing the syncing node to commit the wrong shard's state root and subsequently corrupt the target shard's trie.

---

### Finding Description

`set_state_header` receives three parameters: a caller-supplied `shard_id`, a `sync_hash`, and a network-supplied `shard_state_header`. It performs five validation steps:

1. `validate_chunk_proofs` — checks internal consistency of the chunk (encoded Merkle root, etc.)
2. `verify_path` on `sync_prev_block_header.chunk_headers_root()` — checks that the chunk hash is present in the block's chunk Merkle tree
3. Prev-chunk Merkle proof check
4. Incoming receipts proof chain
5. `validate_state_root_node` — checks the state root node against `chunk_inner.prev_state_root()`

**None of these checks verify that `chunk.shard_id() == shard_id`.**

The `verify_path` check at step 2 only verifies that the chunk hash appears somewhere in the block's `chunk_headers_root` Merkle tree. The block's Merkle tree contains chunks for **all** shards. A malicious peer can supply shard X's chunk (which is a real, valid chunk in the block) together with shard X's valid Merkle proof, while the caller requested shard Y's state. `verify_path` returns `true` because shard X's chunk IS in the tree. The header is then stored under `StateHeaderKey(shard_id=Y, sync_hash)` with shard X's `prev_state_root`.

Downstream, `set_state_part` reads the state root from this stored header and validates parts against shard X's state root. `apply_state_part` then writes shard X's trie nodes into shard Y's `ShardUId` namespace, corrupting shard Y's state. [1](#0-0) 

The missing check is between lines 376 and 380 — after `cloned_chunk()` is called, there is no assertion that `chunk.shard_id() == shard_id`. [2](#0-1) 

`set_state_part` trusts the state root extracted from the stored header without re-verifying the shard binding. [3](#0-2) 

`apply_state_part` writes trie nodes and flat-state entries under the `shard_uid` derived from the caller-supplied `shard_id`, not from the chunk inside the header.

---

### Impact Explanation

A syncing node that accepts a crafted `ShardStateSyncResponseHeader` will:

1. Store shard X's `prev_state_root` as the authoritative state root for shard Y.
2. Download and apply state parts that are valid for shard X's trie but are written into shard Y's `ShardUId` namespace.
3. Emerge from state sync with a corrupted shard Y trie whose root does not match the canonical chain.
4. Produce chunks with wrong state roots, fail endorsement quorum, and be unable to participate in consensus for shard Y.

This is a **Critical** impact: an unprivileged network peer can silently corrupt the state of any shard being synced on a victim node, rendering it unable to produce valid chunks.

---

### Likelihood Explanation

State sync is performed by every new validator joining the network and by any validator reassigned to a new shard. The download source is an arbitrary network peer (`StateSyncDownloadSource`). No authentication or trust level is required to serve state sync data. The exploit requires only that the attacker be reachable as a state sync peer, which is a normal network condition. [4](#0-3) 

The downloader downloads the header from a peer and sends it directly to `set_state_header` for validation. There is no shard binding check before or after this call.

---

### Recommendation

Add an explicit shard binding check immediately after extracting the chunk from the header in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This check should be placed at line 377, immediately after `cloned_chunk()` is called and before any other validation. [5](#0-4) 

---

### Proof of Concept

**Setup**: A two-shard network. Victim node is performing state sync for shard 0. Attacker controls a peer serving state sync data.

**Steps**:

1. Attacker obtains the legitimate `ShardStateSyncResponseHeader` for shard 1 (shard 1's chunk, shard 1's Merkle proof, shard 1's state root node, shard 1's incoming receipts).
2. Attacker serves this header in response to the victim's state sync header request for `shard_id=0`.
3. Victim calls `set_state_header(shard_id=0, sync_hash, attacker_header)`.
4. `validate_chunk_proofs` passes — shard 1's chunk is internally valid.
5. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_1_proof, ChunkHashHeight(shard_1_chunk_hash, ...))` passes — shard 1's chunk IS in the block's Merkle tree.
6. `validate_state_root_node` passes — shard 1's state root node is valid for shard 1's state root.
7. Header is stored under `StateHeaderKey(shard_id=0, sync_hash)` with shard 1's `prev_state_root`.
8. Victim calls `set_state_part` / `apply_state_part` for shard 0, applying shard 1's trie nodes into shard 0's `ShardUId` namespace.
9. Victim's shard 0 trie is now corrupted with shard 1's state. The node produces chunks with wrong state roots and is excluded from consensus. [6](#0-5) 

The corrupted header is committed to `DBCol::StateHeaders` under the wrong shard key, and all subsequent state part application uses the wrong state root without any further shard binding check.

    pub fn ensure_shard_header(
        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
        cancel: CancellationToken,
    ) -> BoxFuture<'_, Result<ShardStateSyncResponseHeader, near_chain::Error>> {
        let store = self.store.clone();
        let validation_sender = self.header_validation_sender.clone();
        let source = self.source.clone();
        let task_tracker = self.task_tracker.clone();
        let clock = self.clock.clone();
        let retry_backoff = self.retry_backoff;
        async move {
            let handle = task_tracker.get_handle(&format!("shard {} header", shard_id)).await;
            handle.set_status("Reading existing header");
            let existing_header =
                get_state_header_if_exists_in_storage(&store, sync_hash, shard_id)?;
            if let Some(header) = existing_header {
                return Ok(header);
            }

            let attempt = || {
                async {
                    let header = source
                        .download_shard_header(shard_id, sync_hash, handle.clone(), cancel.clone())
                        .await?;

                        )
                        .await
                        .map_err(|_| {
                            near_chain::Error::Other(
                                "Validation request could not be handled".to_owned(),
                            )
                        })??;
                    Ok::<ShardStateSyncResponseHeader, near_chain::Error>(header)
                }
            };
