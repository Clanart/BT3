### Title
Meta-Transaction v0 Signature Replay Due to Missing Nonce in Hash and No Nonce Increment — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall handler constructs a transaction hash that omits any nonce, and the OS never checks or increments the target contract's nonce for version-0 meta-transactions. This allows any observer of a valid meta-tx v0 execution to replay the identical call — with the same signature — an unlimited number of times, enabling direct theft of funds from the target account.

---

### Finding Description

**Root cause 1 — Nonce absent from the meta-tx hash.**

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce committed
    );
``` [1](#0-0) 

The hash is therefore a pure function of `(contract_address, selector, calldata, chain_id)`. It is identical across every block for the same call.

**Root cause 2 — Target contract nonce is never checked or incremented.**

In `execute_meta_tx_v0`, the synthesised `new_tx_info` hard-codes `nonce=0` and `version=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
``` [2](#0-1) 

`check_and_increment_nonce` is never called for the meta-tx target. Even if it were, the function explicitly skips all nonce logic for version 0:

```cairo
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

The target contract's on-chain nonce is therefore never advanced after a meta-tx v0 execution.

**Combined effect — exact analog of the external report.**

| External report (LiquidityManager) | This codebase (meta_tx_v0) |
|---|---|
| Nonce not part of the signature | Nonce absent from `compute_meta_tx_v0_hash` |
| `amountsPerNonce` overridden / unreliable | Target contract nonce never incremented |
| `isArbitrageNonceUsed` never validated | `check_and_increment_nonce` skipped for v0 |
| Replay of deposit/release operations | Replay of any `__execute__` meta-call |

---

### Impact Explanation

**Critical — Direct loss of funds.**

A standard StarkNet account contract's `__execute__` entry point verifies the caller's signature against `tx_info.transaction_hash`. Because the meta-tx hash is nonce-free and the target's nonce is never incremented, the hash is identical on every replay. Any attacker who observes a successful meta-tx v0 (e.g., an ERC-20 `transfer` or `approve`) can re-submit the same `(contract_address, selector, calldata, signature)` tuple via their own outer transaction, causing the target account to execute the same call again. Repeated replays drain the target account's token balance to zero.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is reachable by any account contract that can submit an outer invoke transaction — no privileged role is required. The attacker only needs to observe a single successful meta-tx v0 on-chain (all data is public) and then submit their own outer transaction invoking the same syscall with the captured parameters. The vulnerability is deterministic and requires no brute-force or probabilistic attack.

---

### Recommendation

1. **Include the target contract's current nonce in the meta-tx hash.** Pass the nonce as `additional_data` to `deprecated_get_transaction_hash` inside `compute_meta_tx_v0_hash`, mirroring how `compute_l1_handler_transaction_hash` commits the L1-handler nonce: [4](#0-3) 

2. **Read, validate, and increment the target contract's nonce** inside `execute_meta_tx_v0` before dispatching the call, regardless of the synthetic `version=0` field. The nonce check must operate on the target `contract_address` state entry, not on the outer transaction's account.

3. **Set `nonce` in `new_tx_info`** to the actual current nonce of the target contract so that the target's `__validate__` / `__execute__` logic can observe and enforce it.

---

### Proof of Concept

1. Alice (target account) signs a meta-tx v0 authorising `transfer(bob, 100_STRK)`. The outer transaction is submitted; `execute_meta_tx_v0` runs, computes `H = hash(alice_addr, __execute__, calldata, chain_id)`, verifies Alice's signature against `H`, and executes the transfer. Alice's on-chain nonce is **not** incremented.

2. Attacker Eve observes the outer transaction on-chain, extracting `(alice_addr, calldata, alice_signature)`.

3. Eve submits a new outer invoke transaction from her own account. Inside her `__execute__`, she calls the `meta_tx_v0` syscall with the same `contract_address=alice_addr`, same `calldata`, and same `alice_signature`.

4. `execute_meta_tx_v0` recomputes `H` — identical to step 1 because no nonce is committed. Alice's signature still verifies. The transfer executes again. [5](#0-4) 

5. Eve repeats step 3–4 until Alice's balance is zero. Alice's nonce never changes, so no replay protection exists at the OS level.
