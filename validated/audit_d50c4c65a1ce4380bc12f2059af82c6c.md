### Title
No-op `check_is_reverted` Leaves Hint-Provided Revert Status Unconstrained, Enabling Direct Loss of User Funds — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is an empty no-op. In `execute_invoke_function_transaction`, the `is_reverted` flag is loaded from a prover-supplied hint and passed to this function, but since the function performs no assertions, the value is completely unconstrained in the Cairo proof. A malicious prover (sequencer) can freely set `is_reverted = TRUE` for any valid transaction, causing the `__execute__` entry point to be skipped while the fee is still charged — resulting in direct, permanent loss of funds for users. This is the direct analog of the commented-out whitelist `require` in the external report: the call exists, but the enforcement is absent.

---

### Finding Description

In `execution_constraints.cairo` lines 20–22, `check_is_reverted` is defined as a structural no-op:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

It accepts `is_reverted` as a parameter but performs zero assertions, constraints, or validation on it. [1](#0-0) 

In `transaction_impls.cairo`, `execute_invoke_function_transaction` loads `is_reverted` from a prover-supplied hint and calls this function:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(
            block_context=block_context, execution_context=updated_tx_execution_context
        );
    }
} else {
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

Because `check_is_reverted` is a no-op, there is **no Cairo constraint** on `is_reverted`. In the Cairo proof model, hints are prover-supplied non-deterministic values; only explicit assertions (`assert`, `assert_nn_le`, etc.) create proof constraints. Since none exist here, the prover can set `is_reverted` to any value and the resulting proof remains cryptographically valid.

Critically, `charge_fee` is called **unconditionally** outside the `if/else` block, meaning fees are deducted from the user regardless of whether `is_reverted` is `TRUE` or `FALSE`. [3](#0-2) 

The parallel to the external report is exact:

| External Report | This Codebase |
|---|---|
| `require(whitelistedStableTokens[token])` commented out | `check_is_reverted(is_reverted)` calls a no-op function |
| Token validation call exists, enforcement absent | Revert validation call exists, enforcement absent |
| Non-whitelisted tokens processed, funds frozen | Invalid `is_reverted` accepted, funds lost |

The same file contains `check_proof_facts`, which has full validation logic — confirming that `check_*` functions in this file are intended to enforce constraints, and that `check_is_reverted` being empty is anomalous. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious prover sets `is_reverted = TRUE` for a valid user transaction (e.g., an ERC-20 transfer or DeFi interaction). The `__execute__` entry point is skipped entirely — no state changes occur, no transfer happens. However, `charge_fee` executes unconditionally and deducts the fee from the user's account. The resulting OS proof is cryptographically valid (no Cairo constraint is violated), so the L1 verifier accepts it and the state is finalized. The user permanently loses their fee with no corresponding transaction execution.

The reverse direction is equally severe: setting `is_reverted = FALSE` for a transaction that should revert causes `__execute__` to run when it should not, potentially producing invalid state changes (e.g., transferring funds that should not have been transferred, bypassing balance checks).

---

### Likelihood Explanation

The StarkNet OS exists specifically to constrain the sequencer/prover via cryptographic proofs, preventing it from producing valid proofs of invalid state transitions. This is the core security guarantee of the system. The missing constraint in `check_is_reverted` directly undermines this guarantee for every invoke transaction. Any sequencer that is compromised or acts maliciously can exploit this without producing a detectable invalid proof. The exploit requires no special cryptographic capability — only the ability to set a hint value, which is the sequencer's normal operational role.

---

### Recommendation

Implement actual validation logic in `check_is_reverted`. At minimum, constrain `is_reverted` to be a boolean (`0` or `1`). Ideally, derive `is_reverted` from a provable execution result (e.g., the `failure_flag` returned by `execute_entry_point`) rather than accepting it as a free hint, so the value is determined by the proof itself rather than by prover discretion.

---

### Proof of Concept

1. User submits a valid invoke transaction (e.g., an ERC-20 `transfer`).
2. Malicious sequencer sets `is_reverted = 1` (TRUE) in the `%{ IsReverted %}` hint.
3. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` (the `__execute__` call) is skipped.
5. `charge_fee` executes unconditionally, deducting the full fee from the user's account.
6. The OS generates a valid Cairo proof — no constraint was violated.
7. The L1 verifier accepts the proof; state is finalized with the user's fee deducted and no transaction executed.
8. The user permanently loses their fee. The intended ERC-20 transfer never occurred. [1](#0-0) [2](#0-1)
