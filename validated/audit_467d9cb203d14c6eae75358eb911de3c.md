### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS does not validate that the new class hash provided via the `replace_class` syscall corresponds to a previously declared contract class. This is a direct analog to the ERC1155 receiver-check bypass: just as ERC1155 mandates verifying that a receiving contract accepts a transfer before completing it, the StarkNet protocol requires that `replace_class` only accept declared class hashes before committing the state update. The missing check allows any contract to permanently corrupt its own class hash, freezing all funds held within it.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the OS reads the requested `class_hash` from the syscall request and immediately writes it into `contract_state_changes` without consulting `contract_class_changes` to confirm the hash was ever declared:

```cairo
// Replaces the class.
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment at line 898 is an explicit, in-code acknowledgment that this validation is absent. The function signature does not even receive `contract_class_changes` as an implicit argument, making it structurally impossible to perform the check in its current form.

**Vulnerability class:** State-transition bypass — an invalid state entry (undeclared class hash) is committed to the contract state dictionary without the required existence check.

**Analog mapping:**
| ERC1155 (Hats) | StarkNet OS |
|---|---|
| `mintHat`/`transferHat` transfer tokens without checking if receiver implements `ERC1155TokenReceiver` | `replace_class` updates contract class hash without checking if the hash is in `contract_class_changes` |
| Result: tokens permanently locked in non-receiver contract | Result: funds permanently frozen in contract with invalid class hash |

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value, every subsequent call to that contract — whether `call_contract`, `invoke`, or `l1_handler` — will attempt to look up the class in `contract_class_changes` and fail to find it. The contract becomes permanently uncallable. Any ERC20 balances, ETH, or other assets held in the contract's storage are irrecoverably frozen. There is no upgrade path because the upgrade mechanism itself (`replace_class`) is what caused the corruption, and the contract can no longer execute any entry point to self-correct.

**Impact category: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract from within its own execution context — no privileged role is required. A malicious contract deployer can:

1. Deploy a contract that accepts user deposits.
2. After accumulating funds, call an internal function that invokes `replace_class` with an arbitrary, undeclared felt value as the class hash.
3. The OS accepts the syscall unconditionally (lines 896–910).
4. The contract is permanently bricked; all deposited funds are frozen.

Additionally, legitimate contracts with bugs in their upgrade logic (e.g., passing a wrong hash) can accidentally trigger this condition with no recovery path. The attack requires no special privilege, no leaked key, and no network-level capability — only the ability to deploy and interact with a contract.

---

### Recommendation

Before committing the state update in `execute_replace_class`, add a read from `contract_class_changes` to verify the requested class hash has a non-zero compiled class hash entry (i.e., it was previously declared). The function must be extended to receive `contract_class_changes` as an implicit argument, and the check should replace the existing TODO comment:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the pattern already used in `execute_declare_transaction` (`transaction_impls.cairo`, line 816–818), which enforces `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`.

---

### Proof of Concept

1. **Deploy** a contract `VaultWithBadUpgrade` that:
   - Accepts ETH/token deposits from users.
   - Exposes a function `corrupt_class()` that calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is never declared.

2. **Users deposit funds** into `VaultWithBadUpgrade`.

3. **Attacker (or buggy logic) calls** `corrupt_class()`.

4. **OS execution path:**
   - `execute_syscalls` dispatches to `execute_replace_class` (`syscall_impls.cairo` line 195–203).
   - `execute_replace_class` reads `request.class_hash = 0xdeadbeef` (line 896).
   - The TODO at line 898 is skipped — no check against `contract_class_changes`.
   - `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` to `contract_state_changes` (lines 906–910).

5. **All subsequent calls** to `VaultWithBadUpgrade` attempt to resolve class `0xdeadbeef`, find no entry in `contract_class_changes`, and fail.

6. **All user funds are permanently frozen** with no recovery mechanism. [1](#0-0) [2](#0-1) [3](#0-2)
