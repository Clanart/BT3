### Title
Non-UTF-8 `message` ByteArray in StarkNet `InitTransfer` causes permanent fund lock — (`near/omni-types/src/starknet/events.rs`)

### Summary

Cairo's `ByteArray` is an arbitrary byte array with no UTF-8 constraint. The NEAR-side parser enforces UTF-8 validity when decoding it into a Rust `String`. Any user who calls `init_transfer` on StarkNet with a non-UTF-8 `message` will have their tokens burned/locked on StarkNet, but the resulting event will be permanently unparseable by the NEAR prover, locking the funds forever.

### Finding Description

The StarkNet `init_transfer` entry point accepts `message: ByteArray` with no UTF-8 validation: [1](#0-0) 

Tokens are burned or transferred into the contract **before** the event is emitted: [2](#0-1) 

The emitted `InitTransfer` event carries `message: ByteArray` verbatim: [3](#0-2) 

On the NEAR side, `read_byte_array` collects the raw bytes from the felt stream and then calls `String::from_utf8`, returning `Err` on any invalid UTF-8 sequence: [4](#0-3) 

`parse_init_transfer` propagates that error via `?`: [5](#0-4) 

Because the proof submission path calls `parse_init_transfer` and returns the error to the caller, the proof is permanently rejected: [6](#0-5) 

### Impact Explanation

Any user who passes non-UTF-8 bytes (e.g., `[0xFF, 0xFE]`) as the `message` argument to the StarkNet `init_transfer` will:

1. Have their tokens burned/locked on StarkNet (irreversible on-chain state change).
2. Produce an event that the NEAR prover can never successfully parse.
3. Have no recovery path — the transfer ID is consumed, the nonce is incremented, and there is no refund mechanism.

This is a **permanent, irrecoverable fund lock** matching the Critical/High impact category.

### Likelihood Explanation

- Cairo's `ByteArray` is explicitly a raw byte container, not a UTF-8 string. Any programmatic caller (SDK, script, contract-to-contract call) can trivially pass arbitrary bytes.
- There is no on-chain guard in the Cairo contract rejecting non-UTF-8 bytes.
- The mismatch is a silent protocol-level invariant violation: the Cairo side accepts what the NEAR side rejects.

### Recommendation

Replace `String::from_utf8` with `String::from_utf8_lossy` (replacing invalid bytes with `U+FFFD`) in `read_byte_array`, or store the `msg` field as raw `Vec<u8>` / `Base64` throughout the pipeline. Alternatively, add an explicit UTF-8 validation step in the Cairo contract before emitting the event, but that is harder to enforce cross-chain.

    fn read_byte_array(&mut self) -> Result<String, String> {
        let num_full_words = self.read_u64()? as usize;
        let mut bytes = Vec::with_capacity(num_full_words * 31 + 31);

        for _ in 0..num_full_words {
            let word = self.read_felt()?;
            bytes.extend_from_slice(&word[1..32]);
        }

        let pending_word = self.read_felt()?;
        let pending_len = self.read_u64()? as usize;

        if pending_len > 31 {
            return Err(format!(
                "ByteArray: pending_word_len {pending_len} exceeds 31"
            ));
        }

        if pending_len > 0 {
            let start = 32 - pending_len;
            bytes.extend_from_slice(&pending_word[start..32]);
        }

        String::from_utf8(bytes).map_err(|e| format!("ByteArray: invalid UTF-8: {e}"))
    }
