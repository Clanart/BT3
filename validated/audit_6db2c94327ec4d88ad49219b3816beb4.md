### Title
Legacy EIP-712 `timeout_height` Omitted from Signed Type Schema Allows Relayer to Extend Transaction Validity Window — (File: `ethereum/eip712/eip712_legacy.go`)

---

### Summary

The legacy EIP-712 type schema in Ethermint explicitly omits `timeout_height` from the `Tx` type definition. Because `timeout_height` is not committed to in the EIP-712 signature, a relayer can silently modify or remove the field after the user signs, extending the transaction's validity window beyond the user's intended expiry block.

---

### Finding Description

In `ethereum/eip712/eip712_legacy.go`, the `Tx` EIP-712 type definition is constructed as:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
``` [1](#0-0) 

`timeout_height` is deliberately excluded from the EIP-712 type schema. The EIP-712 signing commitment is computed only over the fields present in the type definition; any field absent from the schema is invisible to the signature. A user who sets `timeout_height = N` in their Cosmos SDK transaction body signs a digest that does not include `N`. A relayer holding the signed transaction can freely rewrite `timeout_height` to any higher value (or zero, meaning no expiry) without producing an invalid signature.

The legacy EIP-712 ante handler path remains active in `evmd/ante/ante.go`:

```go
case "/ethermint.types.v1.ExtensionOptionsWeb3Tx":
    // Deprecated: Handle as normal Cosmos SDK tx, except signature is checked for Legacy EIP712 representation
    anteHandler = newLegacyCosmosAnteHandlerEip712(ctx, options, options.ExtraDecorators...)
``` [2](#0-1) 

The legacy ante handler chain includes `authante.NewTxTimeoutHeightDecorator()`, which enforces `timeout_height` at the consensus level: [3](#0-2) 

Because `TxTimeoutHeightDecorator` reads `timeout_height` from the transaction body (not from the signed digest), a relayer-modified `timeout_height` passes both signature verification and the timeout height check.

By contrast, the current (non-legacy) Protobuf EIP-712 path in `encoding.go` explicitly rejects any transaction where `body.TimeoutHeight != 0`:

```go
if body.TimeoutHeight != 0 ||
    body.GetTimeoutTimestamp() != nil || ...
    return apitypes.TypedData{}, errors.New(
        "body contains unsupported fields: TimeoutHeight, TimeoutTimestamp, ...")
``` [4](#0-3) 

This means the only EIP-712 path that allows `timeout_height` to be set at all is the legacy Amino path — and that path does not commit to it in the signature.

---

### Impact Explanation

A user who submits a legacy EIP-712 (`ExtensionOptionsWeb3Tx`) transaction with `timeout_height = N` intends the transaction to be invalid after block `N`. Because `timeout_height` is absent from the EIP-712 type schema, a relayer can:

1. Receive the signed transaction before block `N`.
2. Rewrite `timeout_height` to `N + Δ` (or `0` for indefinite validity).
3. Withhold the transaction until after block `N`.
4. Submit it at block `N + k` (where `k ≤ Δ`).

The `TxTimeoutHeightDecorator` accepts the modified value; `LegacyEip712SigVerificationDecorator` accepts the original signature because `timeout_height` was never part of the signed digest. The transaction executes at a time the user explicitly intended to be past its expiry.

For transactions that transfer funds, perform swaps, or authorize state changes, execution after the intended expiry can result in execution at an unfavorable price or after the user has taken conflicting actions — matching the allowed High impact: **EIP-712 authorization bypass enabling forged execution / unauthorized account mutation**.

---

### Likelihood Explanation

- The legacy EIP-712 path is still reachable by any unprivileged user submitting a transaction with the `ExtensionOptionsWeb3Tx` extension option.
- The attacker is a relayer or any party who receives the signed transaction before it is included in a block (standard mempool exposure).
- No special privileges, key compromise, or governance action is required.
- The modification is trivially performed by editing the serialized `TxBody` bytes before broadcast.

---

### Recommendation

Include `timeout_height` in the legacy EIP-712 `Tx` type schema so it is committed to in the signed digest:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    {Name: "timeout_height", Type: "string"},  // must be included
},
```

Any transaction whose signed `timeout_height` does not match the value in the transaction body must be rejected during signature verification. Alternatively, deprecate and disable the legacy EIP-712 path entirely, since the current Protobuf path already rejects `timeout_height` outright.

---

### Proof of Concept

1. User constructs a legacy EIP-712 Cosmos SDK transaction with `timeout_height = 100` and signs it with their Ethereum key. The EIP-712 digest is computed over the `Tx` type schema, which does **not** include `timeout_height`.

2. The signed transaction is broadcast to the mempool. A relayer intercepts it before block 100.

3. The relayer deserializes the `TxBody`, sets `timeout_height = 0` (no expiry), re-serializes, and rebroadcasts at block 150.

4. `TxTimeoutHeightDecorator` sees `timeout_height = 0` → no expiry constraint → passes.

5. `LegacyEip712SigVerificationDecorator` recomputes the EIP-712 digest using `LegacyWrapTxToTypedData` → `timeout_height` is absent from the type schema → digest is identical to the one the user signed → signature verification passes.

6. The transaction executes at block 150, 50 blocks after the user's intended expiry. [1](#0-0) [2](#0-1) [3](#0-2)

func newLegacyCosmosAnteHandlerEip712(ctx sdk.Context, options HandlerOptions, extra ...sdk.AnteDecorator) sdk.AnteHandler {

	feemarketParams := options.FeeMarketKeeper.GetParams(ctx)
	evmDenom := evmParams.EvmDenom
	chainID := options.EvmKeeper.ChainID()
	chainCfg := evmParams.GetChainConfig()
	ethCfg := chainCfg.EthereumConfig(chainID)
	var txFeeChecker authante.TxFeeChecker

		txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)

	decorators := make([]sdk.AnteDecorator, 0, 15+len(extra))
	decorators = append(decorators,

		cosmos.NewAuthzLimiterDecorator(options.DisabledAuthzMsgs),
		authante.NewSetUpContextDecorator(),
		authante.NewValidateBasicDecorator(),
		authante.NewTxTimeoutHeightDecorator(),
		cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams),
		authante.NewValidateMemoDecorator(options.AccountKeeper),
		authante.NewConsumeGasForTxSizeDecorator(options.AccountKeeper),
		authante.NewDeductFeeDecorator(options.AccountKeeper, options.BankKeeper, options.FeegrantKeeper, txFeeChecker),

		authante.NewSetPubKeyDecorator(options.AccountKeeper),

		authante.NewSigGasConsumeDecorator(options.AccountKeeper, options.SigGasConsumer),
		// Note: signature verification uses EIP instead of the cosmos signature validator
		cosmos.NewLegacyEip712SigVerificationDecorator(options.AccountKeeper, options.SignModeHandler),
		authante.NewIncrementSequenceDecorator(options.AccountKeeper),
		ibcante.NewRedundantRelayDecorator(options.IBCKeeper),
	)
	decorators = append(decorators, extra...)
	return sdk.ChainAnteDecorators(decorators...)
}
