### Title
Missing Mint Account Validation in `FinalizeTransfer` Allows Token Substitution and Unauthorized Vault Drainage — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

The `FinalizeTransfer` Solana instruction does not constrain the caller-supplied `mint` account against the token address embedded in the signed `FinalizeTransferPayload`. An unprivileged actor who possesses any valid MPC-signed payload can substitute an arbitrary mint, causing the program to release tokens from the wrong vault or to mint unauthorized bridged tokens, directly draining bridge collateral.

---

### Finding Description

In `finalize_transfer.rs`, the `FinalizeTransfer` account struct accepts a `mint` account with only a token-program ownership check:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [1](#0-0) 

The `vault` PDA is then derived exclusively from the caller-supplied `mint`:

```rust
seeds = [VAULT_SEED, mint.key().as_ref()],
``` [2](#0-1) 

The instruction is decorated with `#[instruction(data: SignedPayload<FinalizeTransferPayload>)]`, making the signed payload available for use in account constraints, but **no constraint ties `mint.key()` to any field in `data.payload`**. The Anchor framework only enforces what is explicitly declared in the `#[account(...)]` attribute; the absence of a constraint such as `address = data.payload.token_address` means the runtime never checks whether the supplied mint matches the token the MPC signed for. [3](#0-2) 

The `process` function then unconditionally acts on whichever mint was passed:

- **Native path** (`vault` is `Some`): transfers `data.amount` tokens out of the vault for the attacker-chosen mint.
- **Bridged path** (`vault` is `None`): mints `data.amount` tokens of the attacker-chosen mint, provided its `mint_authority` equals the bridge's authority PDA. [4](#0-3) 

The `FinalizeTransferResponse` posted back to NEAR uses `self.mint.key()` — the attacker-chosen mint — not the token from the payload:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),
    amount: data.amount,
    ...
}
``` [5](#0-4) 

This is the direct analog of the Serum audit finding: just as `base_vault` and `quote_vault` were not checked to be owned by `spl_token::ID`, here the `mint` (and therefore the vault derived from it) is not checked to correspond to the token the MPC authorized.

---

### Impact Explanation

**Critical — Unauthorized release of native assets from bridge vaults / unauthorized minting of bridged tokens.**

**Native-token vault drain:** If the bridge holds vaults for multiple registered native tokens (e.g., USDC and USDT), an attacker who holds a valid MPC-signed payload for token A can pass token B's mint. The program transfers `data.amount` units of token B from token B's vault to the recipient. Token B's vault is drained without any corresponding unlock on NEAR, permanently breaking bridge collateralization for token B holders.

**Bridged-token over-mint:** If the attacker passes a bridged mint (one whose `mint_authority` is the bridge's authority PDA), the program mints `data.amount` tokens of that bridged asset. Because the amount comes from the signed payload (not from the bridged token's actual backing), this creates unbacked supply — a direct theft of value from all holders of that bridged token.

---

### Likelihood Explanation

**Medium.** The `finalize_transfer` instruction is fully permissionless — any account can call it. A valid `SignedPayload<FinalizeTransferPayload>` is obtained by initiating a legitimate transfer from NEAR to Solana (even for a worthless or low-value token). The NEAR bridge's `init_transfer` path (via `ft_on_transfer`) does not gate on token registration at deposit time:
