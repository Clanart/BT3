### Title
`execute_meta_tx_v0` Omits Nonce from Transaction Hash, Enabling Unlimited Same-Chain Replay of Signed Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a version-0 transaction hash that does not commit to any nonce, and hardcodes `nonce=0` in the resulting `TxInfo`. Because the OS also explicitly skips nonce checking for version-0 transactions, any attacker who observes a valid `meta_tx_v0` signature can replay it an unlimited number of times within the same chain, leading to direct loss of funds from the victim's account.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` is a syscall that any executing contract can invoke. It:

1. Reads `contract_address`, `selector`, `calldata`, and `signature` from the syscall request (all attacker-controllable).
2. Calls `compute_meta_tx_v0_hash` to produce the hash that the target account's `__execute__` will verify.
3. Constructs a new `TxInfo` with `version=0` and `nonce=0` (hardcoded).
4. Calls the target contract's `__execute__` entry point under this synthetic context.

**Root cause — hash omits nonce:**

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0`:

```cairo
// transaction_hash.cairo lines 302–313
let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    tx_hash_prefix=INVOKE_HASH_PREFIX,
    version=0,
    contract_address=contract_address,
    entry_point_selector=entry_point_selector,
    calldata_size=calldata_size,
    calldata=calldata,
    max_fee=0,
    chain_id=chain_id,
    additional_data_size=0,          // ← no nonce committed
    additional_data=cast(0, felt*),
);
```

The hash is therefore a pure function of `(contract_address, selector, calldata, chain_id)`. No nonce, no block number, no expiry — nothing that changes between invocations.

**Root cause — nonce check is skipped for version 0:**

```cairo
// execute_transaction_utils.cairo lines 63–67
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
```

**Root cause — `__validate__` is skipped for version 0:**

```cairo
// execute_transaction_utils.cairo lines 127–130
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
```

The combination means: once a user has signed a meta_tx_v0 message, the OS provides zero replay protection. The same `(contract_address, selector, calldata, signature)` tuple produces the same hash on every invocation, and the OS will accept it every time.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `meta_tx_v0` mechanism is designed for gasless/relayer patterns: a user signs a v0-style message authorising a specific action (e.g., a token transfer encoded in `calldata`), and a relayer submits it. Because the hash does not include a nonce:

- An attacker who observes a single valid `(contract_address, calldata, signature)` tuple — from a public relayer network, mempool, or on-chain event — can replay it indefinitely.
- Each replay executes the same `__execute__` call on the victim's account with a valid signature, draining funds one transfer at a time (or in a single block if the attacker submits many replays).
- The victim's account nonce is never incremented (version-0 path skips it), so there is no on-chain state that prevents the next replay.

---

### Likelihood Explanation

**High.**

- `meta_tx_v0` is a publicly callable syscall; any deployed contract can invoke it.
- An attacker needs only to: (a) observe one legitimate meta_tx_v0 invocation, and (b) deploy a contract that calls `meta_tx_v0` with the same parameters. Both steps are trivially achievable by an unprivileged actor.
- No privileged access, leaked keys, or operator cooperation is required.
- The `meta_tx_v0` feature is explicitly designed for relayer/gasless flows, which are high-value targets and where signed messages are routinely broadcast publicly.

---

### Recommendation

Include the nonce in the `compute_meta_tx_v0_hash` computation and enforce nonce checking for meta-transactions. Concretely:

1. In `compute_meta_tx_v0_hash`, pass `additional_data_size=1` and `additional_data=&nonce`, where `nonce` is the current on-chain nonce of `contract_address` at the time of the call.
2. In `execute_meta_tx_v0`, read the current nonce from `contract_state_changes` for `contract_address`, include it in the hash, and increment it after successful execution (analogous to `check_and_increment_nonce` for v1/v3 transactions).
3. Alternatively, enforce that the caller provides an explicit nonce in the `MetaTxV0Request` struct and validate it against on-chain state before executing.

---

### Proof of Concept

**Step 1 — Victim signs a meta_tx_v0 message.**
User A signs a message authorising a transfer of 100 tokens from their account (`contract_address=A`) with specific `calldata`. The resulting signature `sig` is broadcast to a public relayer.

**Step 2 — Legitimate relayer submits the first execution.**
Relayer submits a v3 transaction. Its contract calls:
```
meta_tx_v0(contract_address=A, selector=__execute__, calldata=[transfer 100 tokens to B], signature=sig)
```
The OS computes `hash = H(invoke, 0, A, __execute__, calldata, 0, chain_id)` — identical every time. The account's `__execute__` verifies `sig` against `hash` and executes the transfer. ✓

**Step 3 — Attacker replays.**
Attacker deploys contract `Evil` and submits a v3 transaction calling:
```
meta_tx_v0(contract_address=A, selector=__execute__, calldata=[transfer 100 tokens to B], signature=sig)
```
The OS computes the **same hash** (nonce is not in it). The account's `__execute__` verifies `sig` against the same hash — it passes. Another 100 tokens are transferred. The victim's on-chain nonce is still not incremented (version-0 path). The attacker repeats until the account is drained.
