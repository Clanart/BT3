### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Allows Unbacked Cross-Chain Minting - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `initTransfer` function uses the caller-supplied `amount` parameter directly in the `InitTransfer` event and in `initTransferExtension` without verifying the actual token balance received after `safeTransferFrom`. For fee-on-transfer ERC20 tokens, the bridge receives fewer tokens than `amount`, but the event records the full `amount` as locked. The NEAR bridge processes this event and releases/mints the full `amount` on the destination chain, creating unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native ERC20 token locking as follows:

```solidity
// lines 406-411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
```

Immediately after, the same caller-supplied `amount` is emitted in the event and forwarded to the extension:

```solidity
// lines 415-436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
```

No balance-before/balance-after check is performed. For a fee-on-transfer ERC20 token, the actual amount credited to `address(this)` is `amount - transferFee`, but the `InitTransfer` event records `amount`. The NEAR bridge observes this event and authorizes release or minting of the full `amount` on the destination chain.

The same pattern exists in `initTransfer1155` (lines 458–464 and 480–489), where `IERC1155.safeTransferFrom` is called with `amount` and the same `amount` is emitted without verification.

---

### Impact Explanation

This is a **High** severity finding under "Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."

An attacker using a fee-on-transfer token registered with the bridge can repeatedly call `initTransfer` with `amount = X`. The bridge locks only `X - fee` but emits an event for `X`. The NEAR side mints/releases `X` tokens. Over repeated calls, the EVM-side collateral becomes progressively undercollateralized relative to the circulating supply on NEAR, breaking the 1:1 backing invariant of the bridge. Funds redeemed on NEAR cannot all be honored on EVM, resulting in permanent loss for later redeemers.

---

### Likelihood Explanation

Likelihood is **Medium**. Fee-on-transfer tokens exist in production (e.g., tokens with reflection mechanics or protocol fees). Any such token that is registered with the bridge (via `addCustomToken` or `deployToken`) and falls into the native ERC20 path (not `isBridgeToken`, not `customMinters`) is exploitable by any unprivileged user who holds that token and calls `initTransfer`. No privileged access is required beyond holding the token.

---

### Recommendation

Measure the actual balance change around the `safeTransferFrom` call and use the measured delta as the canonical locked amount for event emission and cross-chain messaging:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "Zero received");
// Use actualReceived (cast to uint128) in place of amount below
```

The `actualReceived` value should replace `amount` in both `initTransferExtension` and the `InitTransfer` event emission.

---

### Proof of Concept

1. A fee-on-transfer ERC20 token `T` with a 1% transfer fee is registered with the bridge (native path — not `isBridgeToken`, not `customMinters`).
2. Attacker calls `initTransfer(T, 1_000_000, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` executes: bridge receives `990_000` tokens (1% fee deducted by token contract).
4. `InitTransfer` event is emitted with `amount = 1_000_000`.
5. NEAR bridge relayer observes the event and finalizes a transfer of `1_000_000` tokens to `nearRecipient`.
6. Bridge is now undercollateralized by `10_000` tokens per call.
7. Repeating this 100 times creates a `1_000_000` token deficit — the bridge cannot honor all redemptions back to EVM. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
