### Title
Missing `shard_id` Binding Check in `set_state_header` Allows Malicious Peer to Corrupt Syncing Node's State — (File: `chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from a network peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the chunk embedded in the header actually belongs to the requested `shard_id`. A malicious peer can substitute a valid chunk from a different shard (e.g., shard 1) while responding to a request for shard 0, pass all existing checks, and cause the syncing node to reconstruct and apply the wrong shard's state.

### Finding Description

`set_state_header` performs two structural checks before persisting the header:

1. **`validate_chunk_proofs`** — verifies the chunk's internal hash/tx-root/receipts-root consistency. This check is shard-agnostic; any well-formed chunk from any shard passes it.

2. **`verify_path`** — verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the Merkle tree rooted at `sync_prev_block_header.chunk_headers_root()`. The `chunk_headers_root` covers **all** shards' chunks in the block. A Merkle proof for shard 1's chunk at index 1 is perfectly valid against this root; `verify_path` does not check which index the proof corresponds to.

Neither check compares `chunk.shard_id()` against the caller-supplied `shard_id` parameter. After both checks pass, the header is committed:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
store_update.commit();
```

The stored header now carries shard 1's `prev_state_root` under the key for shard 0.

Downstream, `set_state_part` reads this header to obtain the `state_root` it validates parts against:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// validates parts against shard 1's root, for shard_id = 0
self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part)
```

And `set_state_finalize` applies the chunk's transactions and receipts using `shard_uid` derived from the requested `shard_id` (0), but with the chunk body belonging to shard 1:

```rust
let shard_uid = shard_id_to_uid(self.epoch_manager.as_ref(), shard_id, block_header.epoch_id())?;
// shard_uid = shard 0, but chunk_header.prev_state_root() is shard 1's root
let apply_result = self.runtime_adapter.apply_chunk(
    RuntimeStorageConfig::new(chunk_header.prev_state_root(), true),
    ...
    ApplyChunkShardContext { shard_uid, ... },
    ...
    transactions,  // shard 1's transactions applied to shard 0's slot
)?;
```

### Impact Explanation

The syncing node installs shard 1's trie state into shard 0's slot. Every subsequent block application for shard 0 starts from the wrong state root, producing incorrect `ChunkExtra` entries. If the node is a chunk producer or chunk validator for shard 0, it will sign or endorse chunks built on a corrupted state root, which honest validators will reject. A validator node in this state cannot participate correctly in consensus and may be subject to kickout or slashing. The corruption is silent — no error is returned — and persists until the node re-syncs.

**Severity: High**

### Likelihood Explanation

State sync downloads headers from network peers. Any peer the syncing node connects to during state sync can serve a crafted `ShardStateSyncResponseHeader`. No special privilege is required: the attacker only needs to be reachable as a peer (e.g., via a boot node or a peer discovered through the DHT). The crafted header uses a legitimately signed chunk from a different shard, so no cryptographic forgery is needed. The attack is silent and requires no coordination beyond a single malicious response.

### Recommendation

Add an explicit shard binding check immediately after extracting the chunk in `set_state_header`, before any other validation:

```rust
// Binding check: the chunk inside the header must belong to the requested shard.
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} does not match requested shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

An analogous check should be added for `prev_chunk_header` when present:

```rust
if let Some(ref prev) = prev_chunk_header {
    if prev.shard_id() != shard_id {
        byzantine_assert!(false);
        return Err(Error::Other("set_shard_state failed: prev_chunk shard_id mismatch".into()));
    }
}
```

### Proof of Concept

1. Syncing node requests `StateRequestHeader { shard_id: 0, sync_hash }` from a malicious peer.
2. Malicious peer constructs a `ShardStateSyncResponseHeader` containing:
   - `chunk`: a valid `ShardChunk` for shard 1 (obtained from the same block)
   - `chunk_proof`: the legitimate Merkle path for shard 1's chunk at index 1 in `sync_prev_block.chunk_headers_root`
   - `state_root_node`: shard 1's state root node
3. Syncing node calls `set_state_header(shard_id=0, sync_hash, crafted_header)`.
4. `validate_chunk_proofs` passes — shard 1's chunk is internally consistent.
5. `verify_path(chunk_headers_root, shard1_proof, ChunkHashHeight(shard1_hash, ...))` passes — the proof is valid for shard 1's position in the tree.
6. Header is stored under `StateHeaderKey(shard_id=0, sync_hash)` with shard 1's `prev_state_root`.
7. Malicious peer serves shard 1's state parts; `set_state_part` validates them against shard 1's root — they pass.
8. `set_state_finalize(shard_id=0, sync_hash)` applies shard 1's chunk body under shard 0's `shard_uid`, corrupting shard 0's state permanently.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
