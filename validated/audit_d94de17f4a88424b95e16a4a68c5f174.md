### Title
Missing Zero-Value Validation for `class_hash` in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the caller-supplied `class_hash` is non-zero or corresponds to a declared contract class. This is directly analogous to the RedStone oracle's missing zero-value validation for `basePrice`/`tokenPrice`. An unprivileged contract can call `replace_class(0)`, causing the OS to commit an invalid class hash into the state. Any subsequent call to that contract will fail irrecoverably at the OS level, permanently freezing all funds held within it.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 877–916), the OS reads `class_hash` directly from the syscall request and writes it into the contract state without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The embedded TODO comment explicitly acknowledges the missing check. There is no `assert_not_zero(class_hash)` and no verification that `class_hash` exists in `contract_class_changes`.

When any future transaction attempts to call this contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // = 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // = 0, not declared
);
``` [2](#0-1) 

Since no compiled class with hash `0` is ever declared, `find_element` fails with an assertion error. The contract becomes permanently non-executable at the OS/proof level.

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

Any contract — including account contracts and vaults holding user deposits — that calls `replace_class(0)` (or any undeclared hash) becomes permanently non-executable. No future transaction can successfully call into it. All funds held in the contract are irrecoverably frozen. The OS, as the provable enforcer of protocol rules, commits this invalid state into the global state root without rejection.

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer or transaction sender:

1. A malicious contract is deployed that accepts user deposits and contains a hidden or exploitable code path that calls `replace_class(0)`.
2. Users deposit funds.
3. The attacker triggers the `replace_class(0)` call.
4. The OS accepts the syscall with no validation (confirmed by the TODO comment).
5. The contract's `class_hash` is committed as `0` in the proven state.
6. All deposited funds are permanently frozen.

Additionally, any contract with a reentrancy or access-control bug that allows an external attacker to trigger `replace_class` with an arbitrary argument is vulnerable. The OS provides zero protection at the proof level.

---

### Recommendation

Add an explicit zero-value check and a declared-class membership check inside `execute_replace_class`, before the `dict_update`:

```cairo
// Reject class_hash = 0.
assert_not_zero(class_hash);

// Verify class_hash is a declared class (exists in contract_class_changes).
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This resolves the acknowledged TODO and mirrors the validation already present for `compiled_class_hash` in `execute_declare_transaction`: [3](#0-2)

        MaliciousVault exposes a `deposit()` entry point and a hidden `freeze()` entry point

Step 2: Users call deposit() and send funds to MaliciousVault.

Step 3: Attacker calls freeze() on MaliciousVault.
        → execute_replace_class is invoked with class_hash = 0.
        → OS performs no zero-check (TODO comment, line 898).

        → Block is proven; state root now encodes class_hash=0 for MaliciousVault.

Step 4: Any user attempts to withdraw funds by calling MaliciousVault.
        → execute_entry_point reads class_hash = 0 from state.
        → dict_read on contract_class_changes[0] returns 0 (no declared class).
        → find_element(key=0) fails: assertion error, no compiled class found.
        → Transaction cannot be included in any provable block.
        → Funds are permanently frozen.
``` [4](#0-3) [5](#0-4)

    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }
