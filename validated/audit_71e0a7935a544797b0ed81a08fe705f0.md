Audit Report

## Title
Router `pay()` WETH Branch Consumes Stranded Native ETH Without Per-Caller Accounting, Enabling Free Swaps at Prior Users' Expense — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` resolves WETH payments by checking `address(this).balance` and wrapping the router's own ETH before touching the designated `payer`. Because no per-call ETH credit is tracked, any ETH left on the router from a prior transaction is silently consumed by the next caller's WETH-input swap. The prior user loses their ETH; the attacker receives full swap output at zero cost.

## Finding Description
`pay()` (L73–84) handles the WETH branch as follows:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
} else if (nativeBalance > 0) { ... }
```

When `nativeBalance >= value`, the `payer` argument is completely ignored. `exactInputSingle` (L67) is `payable`, so a user can send `msg.value > amountIn`; the surplus remains on the router after the callback settles. The `receive()` guard (L32–34) only blocks plain ETH transfers — it does not prevent ETH from accumulating via `msg.value` on payable entry points. No transient storage records how much ETH the current top-level call deposited, so the router cannot distinguish "my caller's ETH" from "a prior caller's ETH." `_justPayCallback` (L192–199) calls `pay()` directly with the payer set to `msg.sender` of the original swap, but that payer is never charged when the native balance suffices.

Exploit path:
1. Alice calls `exactInputSingle{value: 1000}(amountIn=500, tokenIn=WETH, ...)`. Callback wraps 500 wei; 500 wei remains. Alice omits `refundETH()`.
2. Bob calls `exactInputSingle{value: 0}(amountIn=500, tokenIn=WETH, ...)`. Callback sees `nativeBalance=500 >= 500`, wraps Alice's 500 wei, sends WETH to pool. Bob receives full token output; his WETH allowance is never touched.

## Impact Explanation
Direct loss of user principal. Alice's 500 wei is permanently transferred to the pool as payment for Bob's trade. Bob receives the full swap output without spending any ETH or WETH. This satisfies the "Critical/High/Medium direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The precondition — ETH stranded on the router — arises naturally when users send `msg.value` larger than `amountIn` (common defensive practice) or use `multicall` without a trailing `refundETH`. Once ETH is stranded, any observer can exploit it in the next block with a zero-cost `exactInputSingle{value: 0}` call specifying WETH as `tokenIn`.

## Recommendation
**Short term:** In the `nativeBalance >= value` branch, verify that the ETH credited to the current call (tracked in transient storage alongside the existing payer/token context) covers at least `value`, and deduct from that per-call credit rather than from the raw contract balance.

**Long term:** Enforce that `address(this).balance == 0` at the start of every non-payable entry point, or revert in `pay()` when `nativeBalance > 0` and the current call's `msg.value` is zero.
