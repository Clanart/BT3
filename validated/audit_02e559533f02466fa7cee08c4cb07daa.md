### Title
Unregistered Token Accepted in `initTransfer` Without Enforcing the Registered Token List — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::initTransfer` accepts any ERC20 token in its native-lock path without verifying the token is registered in the `ethToNearToken` mapping. The registered-token mapping (`ethToNearToken` / `nearToEthToken`) is the protocol's approved-token list — the direct analog of `selectedApplications` in the reference report — but it is never consulted before locking funds. When an unregistered token is locked, the NEAR-side finalization panics on a missing `token_decimals` entry, leaving the ERC20 permanently trapped in the bridge with no on-chain recovery path.

---

### Finding Description

The EVM bridge maintains two bidirectional mappings that constitute its approved-token registry:

```solidity
mapping(address => string) public ethToNearToken;   // EVM addr → NEAR token ID
mapping(string => address) public nearToEthToken;   // NEAR token ID → EVM addr
```

These are populated only through `deployToken` (MPC-signed) or `addCustomToken` (admin-only). They are the protocol's authoritative list of tokens eligible for bridging — the structural equivalent of `selectedApplications`.

`initTransfer` has three token-handling branches:

```solidity
if (customMinters[tokenAddress] != address(0)) {
    // custom-minter burn path
} else if (isBridgeToken[tokenAddress]) {
    // bridge-token burn path
} else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
```

The `else` branch — the native-lock path — executes for **any** ERC20 address that is neither a custom-minter nor a bridge token. There is no check that `ethToNearToken[tokenAddress]` is non-empty (i.e., that the token is registered). The nonce is incremented and `InitTransfer` is emitted unconditionally:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce, amount, fee, nativeFee, recipient, message
);
```

On the NEAR side, `fin_transfer_callback` attempts to resolve the token:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

If the token was never registered via `bind_token`, `token_decimals` has no entry and the callback panics. The NEAR finalization fails permanently. The ERC20 tokens remain locked in the EVM bridge contract with no withdrawal or rescue function available.

The same structural gap exists in `starknet/src/omni_bridge.cairo::init_transfer`, where `starknet_to_near_token` is the approved-token registry but is only consulted to choose burn-vs-lock, not to gate entry:

```cairo
if self.is_bridge_token(token_address) {
    IBridgeTokenDispatcher { contract_address: token_address }.burn(caller, amount.into());
} else {
    let success = IERC20Dispatcher { contract_address: token_address }
        .transfer_from(caller, get_contract_address(), amount.into());
    assert(success, 'ERR_TRANSFER_FROM_FAILED');
}
```

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds.**

Any ERC20 token locked via the unregistered path is stranded: the EVM bridge has no `rescue`, `withdraw`, or emergency-drain function for locked ERC20 balances. The only theoretical recovery is a proxy upgrade by the admin, which is not guaranteed and imposes governance overhead. The NEAR-side finalization will always revert for an unregistered token, so the transfer can never complete. Funds are permanently frozen in the bridge vault.

---

### Likelihood Explanation

**Moderate.** The native-lock path is the intended route for tokens that have been registered via `logMetadata` + `bind_token`. A user who calls `initTransfer` before completing registration (a plausible ordering mistake given the multi-step flow) will permanently lose their tokens. Additionally, a malicious actor could craft a token that passes `safeTransferFrom` but has no NEAR-side registration, then socially engineer victims into calling `initTransfer` with it.

---

### Recommendation

Add an explicit registration check in the native-lock branch of `initTransfer`:

```solidity
} else {
    require(
        bytes(ethToNearToken[tokenAddress]).length > 0,
        "ERR_TOKEN_NOT_REGISTERED"
    );
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
```

Apply the equivalent guard in `starknet/src/omni_bridge.cairo::init_transfer` for the non-bridge-token path, requiring that `starknet_to_near_token.read(token_address).len() > 0` before accepting the transfer.

---

### Proof of Concept

1. `tokenX` is a standard ERC20 deployed on Ethereum. `logMetadata` has never been called for it; `ethToNearToken[tokenX] == ""`.
2. User approves the bridge for `1000 tokenX` and calls:
   ```solidity
   omniBridge.initTransfer(tokenX, 1000, 0, 0, "alice.near", "");
   ```
3. `customMinters[tokenX] == address(0)` and `isBridgeToken[tokenX] == false`, so the `else` branch executes: `safeTransferFrom` moves 1000 `tokenX` into the bridge. `currentOriginNonce` increments. `InitTransfer` is emitted.
4. A relayer submits the proof to NEAR `fin_transfer`. `fin_transfer_callback` executes:
   ```rust
   let decimals = self.token_decimals.get(&init_transfer.token)
       .near_expect(BridgeError::TokenDecimalsNotFound); // PANICS
   ```
5. The NEAR transaction reverts. The EVM `InitTransfer` event is already emitted and the nonce consumed. No retry is possible with a different token. The 1000 `tokenX` are permanently locked in `OmniBridge`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
