### Title
Unvalidated `l1_gas_price_wei` / `l1_data_gas_price_wei` in `ProposalInit` Allows Proposer to Inject Arbitrary ETH-Denominated Gas Prices into Block Execution — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates only the FRI-denominated L1 gas prices from a proposer's `ProposalInit` against the validator's own oracle-derived values. The WEI-denominated counterparts (`l1_gas_price_wei`, `l1_data_gas_price_wei`) are computed by the validator but silently discarded. Those unvalidated WEI fields are then consumed verbatim by `convert_to_sn_api_block_info` to build the `BlockInfo` that drives block execution, and are also used to derive the `eth_to_fri_rate` that determines `l2_gas_price_wei`. A malicious proposer can therefore inject arbitrary WEI prices that pass consensus validation while corrupting every ETH-denominated fee calculation in the block.

---

### Finding Description

**Step 1 — Validation only checks FRI prices.**

In `is_block_info_valid`, the validator calls `get_l1_prices_in_fri_and_wei` (which applies `apply_fee_transformations`, including min/max clamping) and receives both a FRI and a WEI result. The WEI result is bound to `_l1_gas_prices_wei` — the leading underscore signals it is intentionally unused. Only the FRI prices are compared against the proposer's values: [1](#0-0) 

The WEI prices from the proposer's `init` are never compared to the validator's own computed WEI prices.

**Step 2 — Unvalidated WEI prices flow directly into execution.**

After `is_block_info_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)` with the proposer's original `init`: [2](#0-1) 

Inside `convert_to_sn_api_block_info`, the proposer-supplied WEI values are taken verbatim: [3](#0-2) 

**Step 3 — Manipulated WEI prices corrupt `eth_to_fri_rate` and `l2_gas_price_wei`.**

`convert_to_sn_api_block_info` derives `eth_to_fri_rate` from `calculate_eth_to_fri_rate`, which computes `l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei` using the proposer's values: [4](#0-3) 

`l2_gas_price_wei` is then derived as `l2_gas_price_fri / eth_to_fri_rate`. If `l1_gas_price_wei` is set to 1 (minimum nonzero), `eth_to_fri_rate` is inflated by ~10^18, collapsing `l2_gas_price_wei` to near zero.

**Step 4 — The clamping in `apply_fee_transformations` is the direct analog.**

The proposer's own `apply_fee_transformations` clamps WEI prices to `[min_l1_gas_price_wei, max_l1_gas_price_wei]`. A legitimate proposer respects this. A malicious proposer bypasses it by writing any value into `ProposalInit.l1_gas_price_wei` directly, since the validator never checks it: [5](#0-4) 

---

### Impact Explanation

The corrupted `BlockInfo` is passed to the batcher's `validate_block`, which executes all transactions in the block under wrong ETH-denominated gas prices. Consequences:

- **Wrong transaction receipts**: ETH fee amounts (`eth_gas_prices`) are computed from the manipulated WEI prices, producing incorrect `overall_fee` values in every receipt.
- **Wrong `l2_gas_price_wei`**: Derived from the inflated/deflated `eth_to_fri_rate`, this affects ETH-denominated L2 gas fee enforcement for all transactions.
- **Wrong block header gas prices**: The `BlockHeaderWithoutHash` stored in `apollo_storage` records the manipulated WEI prices, corrupting the authoritative on-chain state.
- **Block hash commitment over wrong values**: The block hash chains over gas prices; a wrong WEI price produces a wrong block hash that validators accept because the FRI check passed.

This maps to: **Critical — Wrong state, receipt, and revert result from blockifier/execution logic for accepted input.**

---

### Likelihood Explanation

Requires a node to win a consensus round as proposer. In a decentralized Tendermint setting any validator can become proposer. The attack requires no special privilege beyond being selected as proposer for a round, and no external oracle manipulation — the proposer simply writes an arbitrary value into `ProposalInit.l1_gas_price_wei` before broadcasting. The FRI price check passes as long as the FRI field is within the configured margin.

---

### Recommendation

In `is_block_info_valid`, validate the WEI prices from the proposer's `init` against the validator's own computed WEI prices using the same `within_margin` check already applied to FRI prices. The computed `_l1_gas_prices_wei` should not be discarded:

```rust
// In is_block_info_valid, after computing l1_gas_prices_fri and l1_gas_prices_wei:
let l1_gas_price_wei = l1_gas_prices_wei.l1_gas_price;
let l1_data_gas_price_wei = l1_gas_prices_wei.l1_data_gas_price;
let l1_gas_price_wei_proposed = init_proposed.l1_gas_price_wei;
let l1_data_gas_price_wei_proposed = init_proposed.l1_data_gas_price_wei;

if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, margin)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, margin)
    && within_margin(l1_gas_price_wei_proposed, l1_gas_price_wei, margin)       // ADD
    && within_margin(l1_data_gas_price_wei_proposed, l1_data_gas_price_wei, margin)) // ADD
```

This mirrors the Chainlink fix pattern: check the value is within an acceptable range, not merely nonzero.

---

### Proof of Concept

1. A node wins a consensus round as proposer.
2. It calls `get_l1_prices_in_fri_and_wei` normally, obtaining valid FRI prices (e.g., `l1_gas_price_fri = 1000 fri`).
3. It constructs `ProposalInit` with `l1_gas_price_fri = 1000` (valid, within margin) but `l1_gas_price_wei = 1` (1 wei, instead of the correct ~500 gwei).
4. Validators receive the `ProposalInit`. `is_block_info_valid` checks only FRI prices — the check passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`:
   - `eth_to_fri_rate = 1000 * 10^18 / 1 = 10^21` (inflated by ~2000×)
   - `l2_gas_price_wei = l2_gas_price_fri / 10^21 ≈ 0` → `NonzeroGasPrice::new` fails or rounds to 1
6. All transactions in the block execute with `eth_gas_prices.l1_gas_price = 1 wei` and near-zero `l2_gas_price_wei`.
7. Transaction receipts record wrong ETH fees. The block header stores `l1_gas_price_wei = 1`. The block hash is computed over these wrong values and accepted by all validators. [6](#0-5) [7](#0-6)

pub(crate) fn apply_fee_transformations(
    price_info: &mut PriceInfo,
    gas_price_params: &GasPriceParams,
) {
    price_info.base_fee_per_gas = price_info
        .base_fee_per_gas
        .saturating_add(gas_price_params.l1_gas_tip_wei)
        .clamp(gas_price_params.min_l1_gas_price_wei, gas_price_params.max_l1_gas_price_wei);

    price_info.blob_fee = GasPrice(
        (gas_price_params.l1_data_gas_price_multiplier * price_info.blob_fee.0).to_integer(),
    )
    .clamp(gas_price_params.min_l1_data_gas_price_wei, gas_price_params.max_l1_data_gas_price_wei);
}

fn calculate_eth_to_fri_rate(block_info: &PreviousBlockInfo) -> Result<u128, StarknetApiError> {
    let eth_to_fri_rate = block_info
        .l1_prices_fri
        .l1_gas_price
        .0
        .checked_mul(WEI_PER_ETH)
        .ok_or_else(|| {

                 block info: {:?}",
                block_info
            ))
        })?

            "Eth to fri rate is zero. Previous block info: {:?}",
