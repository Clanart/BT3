### Title
Missing Zero Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

Both the new and deprecated `execute_replace_class` syscall implementations in the StarkNet OS accept a caller-supplied `class_hash` of zero without any validation. Because `UNINITIALIZED_CLASS_HASH = 0` is the sentinel value that marks a contract as non-existent in the state tree, setting a live contract's class hash to zero permanently strips it of any executable class, making all funds it holds irrecoverable.

---

### Finding Description

`UNINITIALIZED_CLASS_HASH` is defined as the literal constant `0`: [1](#0-0) 

The new-syscall implementation of `execute_replace_class` reads the caller-supplied `class_hash` directly from the request and writes it into the contract's `StateEntry` with no zero-check and with an explicit TODO acknowledging the missing validation: [2](#0-1) 

The deprecated-syscall counterpart has the identical omission: [3](#0-2) 

The state-commitment logic treats a `StateEntry` with `class_hash == 0`, empty storage, and `nonce == 0` as a hash of `0`, i.e., indistinguishable from a never-deployed contract: [4](#0-3) 

Once the class hash is committed as `0`, no entry point can be dispatched on the contract. Because `replace_class` itself requires dispatching an entry point, the state is irreversible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 balance, NFT, or other asset stored in the contract's storage becomes permanently inaccessible. There is no recovery path: the contract cannot execute `replace_class` again (no class to dispatch), cannot call `transfer`, and cannot self-destruct. The funds are frozen in-protocol forever.

---

### Likelihood Explanation

The attack requires only that a contract reachable by an unprivileged transaction sender issues a `replace_class` syscall with `class_hash = 0`. Concrete realistic paths:

1. **Malicious vault/DEX deployer**: An attacker deploys a contract that appears to be a legitimate vault. After users deposit funds, the attacker triggers an internal function that calls `replace_class(0)` and returns successfully. The transaction commits, the class hash is zeroed, and all deposited funds are frozen.
2. **Buggy shared contract**: A multisig or DAO contract with a flawed access-control path allows an unauthorized caller to invoke `replace_class(0)`, freezing all treasury funds.

No privileged role, leaked key, or network-level attack is required. The OS is the necessary vulnerable step because it is the only layer that could enforce `class_hash != 0` before committing the state change.

---

### Recommendation

Add an `assert_not_zero(class_hash)` guard in both `execute_replace_class` implementations immediately after reading `class_hash` from the request, before any state mutation. This directly mirrors the fix recommended in the reference report (validate input before initializing/updating state). The existing TODO comment in `syscall_impls.cairo` already acknowledges that a class-existence check is needed; a zero-check is the minimal necessary subset of that validation. [5](#0-4) 

---

### Proof of Concept

1. Deploy a contract `Vault` that:
   - Accepts ERC-20 deposits from arbitrary users.
   - Exposes a function `freeze()` that executes `replace_class(class_hash=0)` and returns.

2. Users call `deposit()` on `Vault`, transferring tokens into its storage.

3. Attacker (or anyone who can call `freeze()`) submits a transaction invoking `freeze()`.

4. The OS processes the `replace_class` syscall via `execute_replace_class`. Because there is no `assert_not_zero(class_hash)` check, the syscall succeeds and `contract_state_changes` is updated with `StateEntry(class_hash=0, ...)`.

5. The transaction is included in a block. The state commitment now records `Vault`'s class hash as `UNINITIALIZED_CLASS_HASH`.

6. Any subsequent call to `Vault` (e.g., `withdraw()`) fails at entry-point dispatch because the class hash is `0` — no class exists. All deposited tokens are permanently frozen with no recovery mechanism available at the protocol level.
