Audit Report

## Title
Stranded ETH in Router Silently Consumed by Subsequent WETH Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` reads `address(this).balance` unconditionally when `token == WETH` and treats the entire native balance as available subsidy for the current payer. ETH left in the router by a prior user — via any `payable` entry point without a trailing `refundETH()` — is silently wrapped and forwarded to the pool on behalf of the next WETH-paying caller, permanently consuming the prior user's ETH and reducing the attacker's WETH cost by the same amount.

## Finding Description
The root cause is in `pay` at [1](#0-0)  — `nativeBalance = address(this).balance` captures the router's entire ETH balance with no attribution to the current caller. When `0 < nativeBalance < value`, the hybrid branch wraps and transfers `nativeBalance` to the pool first, then pulls only `value - nativeBalance` from `payer` via `safeTransferFrom`. There is no check that `nativeBalance` was contributed by the current `payer`.

ETH enters the router legitimately via any `payable` entry point: `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput` (all `payable` at [2](#0-1) ), and `multicall` at [3](#0-2) .

The `receive()` guard at [4](#0-3)  only blocks *direct* ETH pushes from non-WETH addresses. It does not prevent ETH from being forwarded as `msg.value` to any of the above `payable` functions. If user A calls `multicall{value: 1 ETH}([exactInputSingle(WETH, 0.5 ETH)])` without appending `refundETH()`, 0.5 ETH remains in the router. User B then calls `exactInputSingle(WETH, 1 ETH)` with `msg.value = 0`; inside `pay(WETH, userB, pool, 1e18)`, `nativeBalance = 0.5e18` (user A's residual), the router wraps and sends user A's 0.5 ETH to the pool, and pulls only 0.5 ETH worth of WETH from user B. User A's 0.5 ETH is permanently consumed; user B pays half price.

## Impact Explanation
Direct loss of user principal. The stranded ETH is irreversibly wrapped and delivered to the pool as part of a different user's payment. The loss equals the full stranded ETH amount, which is unbounded. This meets the High threshold for direct loss of user funds under the allowed impact gate.

## Likelihood Explanation
Medium. The precondition — ETH stranded in the router — arises naturally whenever a user sends excess ETH without `refundETH()`, a common mistake in the Uniswap v3 multicall pattern. An attacker can monitor the chain for a non-zero `address(router).balance` and immediately issue a WETH swap to drain it. No privileged access is required; any unprivileged caller can trigger this.

## Recommendation
Track only the ETH contributed by the current top-level call. Replace `address(this).balance` with `msg.value` (passed down through the call stack or stored in transient storage at entry), decremented as it is consumed. Alternatively, snapshot the router's ETH balance at the start of each top-level call and use only the delta (`address(this).balance - snapshotBalance`) as the available native subsidy, refusing to consume any pre-existing balance.

## Proof of Concept
```solidity
// 1. User A sends excess ETH via multicall, forgets refundETH
router.multicall{value: 1 ether}([
    abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        tokenIn: WETH, amountIn: 0.5 ether, ...
    })))
    // no refundETH call
]);
// router.balance == 0.5 ether (user A's ETH stranded)

// 2. User B (attacker) calls a WETH swap with msg.value = 0
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: WETH, amountIn: 1 ether, ...
}));
// pay(WETH, userB, pool, 1e18):
//   nativeBalance = 0.5e18 > 0  (user A's residual)
//   wraps 0.5 ETH (user A's) → sends to pool
//   safeTransferFrom(userB, pool, 0.5e18)  ← only half pulled from B
// Result: user A lost 0.5 ETH; user B paid 0.5 ETH less WETH
assert(address(router).balance == 0);
assert(userA_loss == 0.5 ether);
assert(userB_savings == 0.5 ether);
```
