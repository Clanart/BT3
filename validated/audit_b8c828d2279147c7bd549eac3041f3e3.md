### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter directly in the cross-chain message and emitted event after calling `safeTransferFrom`, without verifying the actual tokens received. For fee-on-transfer tokens (e.g., USDT with fee enabled), the bridge vault receives fewer tokens than `amount`, yet the cross-chain message instructs the destination chain to release the full `amount`, breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer`, when a native (non-bridge, non-custom-minter) ERC20 token is bridged, the code path is:

```solidity
// OmniBridge.sol lines 407–412
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← user-supplied, not verified post-transfer
);
```

Immediately after, the same unverified `amount` is forwarded to `initTransferExtension` and the `InitTransfer` event:

```solidity
// OmniBridge.sol lines 415–436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
```

In `OmniBridgeWormhole.initTransferExtension`, this `amount` is encoded directly into the Wormhole cross-chain payload:

```solidity
// OmniBridgeWormhole.sol lines 129–141
bytes memory payload = bytes.concat(
    ...
    Borsh.encodeUint128(amount),   // ← inflated amount in message
    ...
);
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

The NEAR bridge processes this message and mints/releases `amount` tokens to the recipient. No balance-before/after check is performed to determine the actual tokens received by the bridge contract.

The `customMinters` path (lines 394–402) has a related issue: `safeTransferFrom` sends `amount` to the custom minter, then `ICustomMinter.burn(tokenAddress, amount)` is called with the full `amount`. If the minter received less due to fees, the burn call operates on an incorrect quantity.

---

### Impact Explanation

For any fee-on-transfer ERC20 token bridged via `initTransfer`:

- The bridge vault receives `amount - fee_deducted` tokens.
- The cross-chain message records `amount`.
- The destination chain (NEAR) mints or releases `amount` tokens to the recipient.
- The bridge is undercollateralized by `fee_deducted` per transfer.

Repeated transfers drain the bridge's reserves, eventually causing legitimate withdrawals to fail (permanent fund lock) or enabling the attacker to extract more value than deposited. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

USDT's transfer fee is currently 0 but is a live, owner-controllable parameter. Other deployed fee-on-transfer tokens exist (e.g., STA, PAXG in certain configurations). Any unprivileged user can call `initTransfer` with such a token. No special access or key compromise is required. The entry path is fully permissionless.

---

### Recommendation

Record the bridge contract's token balance before and after `safeTransferFrom`, and use the delta as the canonical bridged amount for all downstream calculations and cross-chain messages:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived (cast to uint128) in place of amount going forward
```

Apply the same pattern to the `customMinters` path, passing `actualReceived` to `ICustomMinter.burn`.

---

### Proof of Concept

1. Deploy or use a fee-on-transfer ERC20 token `FeeToken` with a 1% transfer fee.
2. User approves `OmniBridge` for `1000 FeeToken`.
3. User calls `initTransfer(FeeToken, 1000, 0, nativeFee, "recipient.near", "")`.
4. Inside `initTransfer`, `safeTransferFrom(user, bridge, 1000)` executes; bridge receives `990` tokens (1% fee taken).
5. `initTransferExtension` is called with `amount = 1000`; Wormhole message is published encoding `amount = 1000`.
6. `InitTransfer` event emits `amount = 1000`.
7. NEAR bridge processes the Wormhole VAA and releases `1000 FeeToken` (or equivalent wrapped tokens) to the recipient.
8. Bridge vault holds only `990` tokens but has issued a claim for `1000`, creating a `10`-token deficit per transfer.
9. After sufficient transfers, the bridge cannot honor legitimate `finTransfer` redemptions, permanently locking user funds.
