### Title
`fin_transfer_send_tokens_callback` Ignores `ft_transfer` / `mint` Failure, Permanently Freezing Inbound User Funds — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fin_transfer_send_tokens_callback` only checks for a failed token delivery when `is_ft_transfer_call` is `true`. When the token transfer path is a plain `ft_transfer` or a `mint` with no message (both set `is_ft_transfer_call = false`), a failure of the underlying cross-contract call is silently ignored. Because the transfer is already recorded in `finalised_transfers` before `send_tokens` is dispatched, the nonce is consumed and the user's inbound funds are permanently frozen with no recovery path inside the protocol.

---

### Finding Description

**Step 1 – Transfer is finalized before token delivery.**

Inside `process_fin_transfer_to_near`, the very first action is:

```rust
let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
```

This inserts the transfer ID into `finalised_transfers`. Any subsequent attempt to re-submit the same proof will panic with `TransferAlreadyFinalised`.

**Step 2 – Token delivery is attempted asynchronously.**

`send_tokens` is called and chained to `fin_transfer_send_tokens_callback`:

```rust
self.send_tokens(token.clone(), recipient, U128(...), &msg)
    .then(Self::ext(...).fin_transfer_send_tokens_callback(
        transfer_message, &fee_recipient, !msg.is_empty(), ...
    ))
```

The `is_ft_transfer_call` argument is `!msg.is_empty()`.

**Step 3 – `send_tokens` uses `ft_transfer` or `mint` when `msg` is empty.**

For non-deployed tokens with an empty message:
```rust
} else if msg.is_empty() {
    ext_token::ext(token)
        .with_attached_deposit(ONE_YOCTO)
        .with_static_gas(FT_TRANSFER_GAS)
        .ft_transfer(recipient, amount, None)   // ← plain ft_transfer
```

For deployed tokens with an empty message, `mint()` is called with no message. In both cases `is_ft_transfer_call = false`.

**Step 4 – Callback never inspects the promise result for these paths.**

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) { ... }
    } else {
        // Not ft_transfer_call: don't refund
        false   // ← promise result is NEVER read
    }
}
```

If `ft_transfer` panics (token paused, recipient blacklisted, etc.), the NEAR runtime marks the promise as `Failed`. The callback is still invoked, but `is_refund_required` returns `false` without inspecting `env::promise_result_checked`. The callback therefore falls into the `else` branch, emits `FinTransferEvent`, and optionally mints fees — all as if the delivery succeeded.

**Step 5 – No recovery path exists.**

`finalised_transfers` already contains the transfer ID. Any re-submission of the proof will be rejected. There is no protocol-level function that lets a user reclaim funds from a finalized-but-undelivered transfer.

---

### Impact Explanation

A user bridging a non-deployed token (e.g., USDC, USDT) or any token with an empty `msg` to a NEAR recipient loses their funds permanently if the token contract is paused or the recipient is blacklisted at the moment of finalization. The transfer nonce is consumed, the source-chain tokens are already locked/burned, and the NEAR-side delivery silently fails. The only recovery is manual DAO intervention via `transfer_token_as_dao`, which is not guaranteed and requires the DAO to detect the silent failure from off-chain logs.

This matches the allowed impact: **Critical — Permanent freezing / irrecoverable lock of user funds in the bridge flow.**

---

### Likelihood Explanation

USDC and USDT on NEAR both implement pause and blacklist mechanisms controlled by their issuers. A regulatory freeze, a contract upgrade window, or a recipient address that becomes blacklisted between transfer initiation and finalization are all realistic triggers. The finalization window can span hours or days (relayer latency, proof generation), making the race condition non-trivial to avoid. Any unprivileged user who initiates a transfer is exposed.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, always inspect the promise result regardless of `is_ft_transfer_call`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => serde_json::from_slice::<U128>(&value)
                .map(|a| a.0 == 0)
                .unwrap_or(false),
            Err(_) => true,   // promise failed → refund/revert
        }
    } else {
        // ft_transfer or mint: treat a failed promise as a refund trigger
        env::promise_result_checked(0, 0).is_err()
    }
}
```

When a refund is triggered for a finalized transfer, `remove_fin_transfer` should be called to remove the ID from `finalised_transfers`, allowing the relayer to re-submit the proof after the external condition is resolved.

---

### Proof of Concept

1. Alice initiates a transfer of 10,000 USDC from Ethereum to NEAR.
2. The relayer calls `fin_transfer` with a valid proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` records the transfer ID in `finalised_transfers`. [1](#0-0) 
   - Storage checks pass.
   - `send_tokens` dispatches `ft_transfer(alice, 10000, None)` to the USDC contract. [2](#0-1) 
4. USDC contract is paused; `ft_transfer` panics → promise result is `Failed`.
5. `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false`. [3](#0-2) 
6. `is_refund_required(false)` returns `false` without reading the promise result. [4](#0-3) 
7. Callback emits `FinTransferEvent` as if delivery succeeded. [5](#0-4) 
8. Alice's 10,000 USDC remain in the bridge contract. The transfer ID is in `finalised_transfers`. Re-submission of the proof panics. Alice's funds are permanently frozen.
