### Title
Compiled-class deduplication key mismatch in `sync_pending_data` causes missing CASM for second class sharing a `compiled_class_hash`, breaking pending-block RPC execution — (`crates/apollo_central_sync/src/pending_sync.rs`)

### Summary

`sync_pending_data` deduplicates CASM fetch tasks by `compiled_class_hash`, but `get_pending_compiled_class` stores the result under `class_hash`. When two `DeclaredClassHashEntry` values share the same `compiled_class_hash` but have distinct `class_hash` values, only the first class gets its CASM stored in `pending_classes.compiled_classes`. The second class's CASM is permanently absent for the lifetime of the pending block, causing all pending-block RPC execution calls for that class to return `UndeclaredClassHash`.

### Finding Description

In `sync_pending_data` (`crates/apollo_central_sync/src/pending_sync.rs`), the set `processed_compiled_classes` is keyed on `compiled_class_hash`: [1](#0-0) 

When `compiled_class_hash = C` is seen for the first time (entry H1), `get_pending_compiled_class(H1, ...)` is queued. When the same `C` appears for H2, `processed_compiled_classes.insert(C)` returns `false` and no task is queued for H2.

`get_pending_compiled_class` stores the result under `class_hash`, not `compiled_class_hash`: [2](#0-1) 

`PendingClasses.compiled_classes` is a `HashMap<ClassHash, Arc<CasmContractClass>>`: [3](#0-2) 

So after processing, `compiled_classes` contains `H1 → CASM` but has no entry for H2.

### Impact Explanation

During pending-block RPC execution (`estimate_fee`, `call`, `simulate_transactions`), `get_compiled_class` in `ExecutionStateReader` first checks `pending_classes`: [4](#0-3) 

For H2: `get_class(H2)` returns `Some(ContractClass(sierra))` (Sierra was fetched correctly), but `get_compiled_class(H2)` returns `None`. The code falls through to `class_manager_handle.get_executable(H2)`, which returns `None` because H2 is a pending class not yet committed to the class manager, producing `StateError::UndeclaredClassHash(H2)`: [5](#0-4) 

The RPC layer surfaces this as an authoritative-looking error for a class that is legitimately declared in the pending block.

### Likelihood Explanation

Two Sierra classes with different `class_hash` but the same `compiled_class_hash` arise when the classes differ only in ABI or metadata fields that do not affect CASM output. Both declare transactions pass sequencer validation independently (each provides a valid `compiled_class_hash` matching its CASM). This requires deliberate crafting but no privileged access — any user can submit declare transactions.

### Recommendation

Replace the deduplication key with `class_hash` instead of `compiled_class_hash`, or remove the `processed_compiled_classes` set entirely and deduplicate only on `class_hash` (which is already done for the Sierra class fetch). The correct invariant is: every new `class_hash` in `declared_classes` needs its own CASM fetch, regardless of whether another class shares the same `compiled_class_hash`.

```rust

// With:
if !processed_classes.contains(&class_hash) {  // already inserted above
```

Or more simply, remove `processed_compiled_classes` entirely and always queue `get_pending_compiled_class` whenever `processed_classes.insert(class_hash)` returns `true` (i.e., merge the two `if` blocks into one).
