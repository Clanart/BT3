### Title
Native-ETH `finTransfer` Permanently Locks User Funds When Recipient Contract Reverts — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.finTransfer` marks the destination nonce as used before attempting to deliver native ETH to the recipient. If the recipient is a contract that reverts on receiving ETH, the entire transaction reverts (including the nonce mark), so the nonce is never consumed and the call can be retried — but it will always fail. Because the user's tokens were already burned or locked on NEAR when `initTransfer` was called, and no on-chain cancellation path exists, the funds are permanently irrecoverable.

### Finding Description

In `OmniBridge.finTransfer`, when `payload.tokenAddress == address(0)`, the contract delivers native ETH via a low-level call:

```solidity
completedTransfers[payload.destinationNonce] = true;
// … signature verification …
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

Because the revert unwinds the entire transaction, `completedTransfers` is also rolled back, leaving the nonce unconsumed. A relayer can retry indefinitely, but every attempt will revert if the recipient contract always rejects ETH (no `receive()` / `fallback()`, or one that conditionally reverts). There is no alternative delivery path, no recipient-change mechanism, and no refund path back to NEAR.

The NEAR-side `ft_on_transfer` / `init_transfer` flow burns or locks the user's tokens at initiation time. Once that cross-chain message is signed by MPC and submitted to EVM, the only way to release those NEAR-side tokens is a successful `finTransfer` on EVM. If `finTransfer` can never succeed, the NEAR-side tokens are permanently frozen.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who bridges native ETH to a smart-contract wallet (e.g., a Safe multisig, an account-abstraction wallet, or any contract without a payable fallback) will have their NEAR-side tokens permanently locked. The bridge has no escape hatch: the nonce is never consumed, so the transfer is neither finalized nor cancellable.

### Likelihood Explanation

Smart-contract wallets are common recipients for bridge transfers. Many Safe multisig deployments, ERC-4337 account-abstraction wallets, and DeFi protocol treasuries do not implement a payable `receive()` function. A user who copies their contract wallet address as the ETH recipient — a natural action — will trigger this condition. No privileged role or special knowledge is required; the user only needs to initiate a NEAR → ETH native-ETH transfer to a non-payable contract address.

### Recommendation

1. **Wrap-and-transfer pattern**: Instead of sending raw ETH, wrap it to WETH and transfer the ERC-20 token. This eliminates the dependency on the recipient's payability.
2. **Pull-over-push**: Record the claimable ETH balance for the recipient and let them call a `claimETH()` function, removing the push-delivery failure mode entirely.
3. **Recipient validation**: At minimum, add a pre-flight check (e.g., a zero-value call) to detect non-payable recipients before the NEAR-side tokens are locked, or enforce that native-ETH recipients must be EOAs (`payload.recipient.code.length == 0`).

### Proof of Concept

1. Alice holds 1 ETH worth of a NEAR-native token and calls `initTransfer` on NEAR, specifying her Safe multisig address (which has no `receive()`) as the EVM recipient. NEAR burns/locks her tokens.
2. The MPC network signs the `TransferMessagePayload` with `tokenAddress = address(0)` and `recipient = <Safe address>`.
3. A relayer calls `OmniBridge.finTransfer` with the signed payload. The contract sets `completedTransfers[nonce] = true`, verifies the signature, then executes:
   ```solidity
   (bool success, ) = payload.recipient.call{value: payload.amount}("");
   if (!success) revert FailedToSendEther();
   ```
   The Safe has no payable fallback → `success == false` → `revert FailedToSendEther()`.
4. The entire transaction reverts; `completedTransfers[nonce]` is rolled back.
5. Every subsequent retry by any relayer produces the same revert.
6. Alice's NEAR-side tokens remain burned/locked with no cancellation function available. Funds are permanently lost. [2](#0-1)
