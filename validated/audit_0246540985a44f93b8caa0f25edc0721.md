### Title
Permanently Locked ETH in `OmniBridge` Contract Due to `receive()` with No Withdrawal Mechanism — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary
`OmniBridge.sol` declares a bare `receive() external payable {}` function and several `payable` public entry points, but contains no ETH withdrawal, rescue, or sweep function. Any ETH sent directly to the contract — or any `nativeFee` ETH that is not forwarded to Wormhole — is permanently irrecoverable.

---

### Finding Description

`OmniBridge` exposes a bare `receive()` hook: [1](#0-0) 

This allows any caller to push ETH into the contract at any time. A full search of the contract confirms there is no `withdraw`, `rescue`, `sweep`, or `recoverEth` function anywhere in the contract or its Wormhole extension.

Additionally, `initTransfer` is `payable` and splits `msg.value` into `nativeFee` and `extensionValue`: [2](#0-1) 

Only `extensionValue` is forwarded to Wormhole in `OmniBridgeWormhole.initTransferExtension`: [3](#0-2) 

The `nativeFee` portion of `msg.value` is silently retained by the contract. There is no on-chain mechanism for the fee recipient or any party to claim it.

The same pattern applies to `logMetadata` and `logMetadata1155`, both of which are `external payable` with no access control: [4](#0-3) [5](#0-4) 

In the base `OmniBridge` (non-Wormhole deployment), `logMetadataExtension` is a no-op virtual, so any ETH attached to these calls is locked with no path out.

---

### Impact Explanation

ETH sent via `receive()` or retained as `nativeFee` in `initTransfer` accumulates in the contract with no withdrawal path. While ETH held for native-ETH bridging (`tokenAddress == address(0)`) can exit via `finTransfer`, ETH from direct sends or `nativeFee` payments is mixed into the same balance and is irrecoverable by the original payer. Over time, `nativeFee` accumulation across many user transfers results in a growing pool of permanently locked user funds. This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

- Any user who calls `initTransfer` with `nativeFee > 0` loses that ETH permanently — this is a normal, unprivileged, public entry point.
- Any user or contract that sends ETH directly to the bridge address (e.g., via a wallet mistake or a contract that calls `address(bridge).transfer(...)`) loses those funds.
- No privileged role is required; the entry paths are fully reachable by any unprivileged bridge user.

---

### Recommendation

1. **Remove `receive() external payable {}`** unless the contract explicitly needs to accept direct ETH sends. If it must remain (e.g., for Wormhole refunds), add an admin-controlled ETH rescue function.
2. **Add a `claimNativeFee` or `withdrawNativeFee` function** that allows the designated fee recipient to pull accumulated `nativeFee` ETH, or forward `nativeFee` directly to the fee recipient inside `initTransfer` rather than retaining it.
3. For `logMetadata` / `logMetadata1155`, either remove `payable` or revert if `msg.value > 0` when no ETH forwarding is performed.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` (or base `OmniBridge`).
2. Call `initTransfer(erc20Token, amount, 0, 1 ether, "recipient", "")` with `msg.value = 1 ether + wormholeFee`. The `extensionValue = wormholeFee` is forwarded to Wormhole; `1 ether` (`nativeFee`) stays in the contract.
3. Observe: contract ETH balance increases by `1 ether`. No function exists to withdraw it.
4. Alternatively, call `address(bridge).transfer(1 ether)` — ETH is accepted by `receive()` and is unrecoverable. [6](#0-5) [7](#0-6)
