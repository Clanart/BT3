### Title
Missing `parent_hash` Chain Verification in P2P Header Sync Allows Malicious Peer to Corrupt Stored Block Header — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts and stores a `SignedBlockHeader` from any p2p peer without verifying that `parent_hash` equals the stored `block_hash` of block N-1. The missing check is explicitly acknowledged by a TODO comment in production code. The corrupted `parent_hash` is persisted verbatim into `StorageBlockHeader` and subsequently served via RPC.

---

### Finding Description

`parse_data_for_block` for the header stream performs exactly two checks before accepting a `SignedBlockHeader`:

1. `block_number == expected_block_number`
2. `signatures.len() == ALLOWED_SIGNATURES_LENGTH` [1](#0-0) 

The explicit TODO at line 102 confirms the `parent_hash` check is absent:

```
// TODO(shahak): Check that parent_hash is the same as the previous block's hash
// and handle reverts.
```

After passing these two checks, the `SignedBlockHeader` is returned as `Ok(Some(signed_block_header))` and forwarded to `write_to_storage`, which calls `append_header` unconditionally: [2](#0-1) 

`append_header` constructs a `StorageBlockHeader` by copying `block_header.block_header_without_hash.parent_hash` directly from the peer-supplied value, with no cross-check against the stored `block_hash` of block N-1: [3](#0-2) 

The `StorageBlockHeader.parent_hash` field is then persisted to the headers table and later reconstructed verbatim by `get_block_header`: [4](#0-3) 

By contrast, the central sync path (`apollo_central_sync`) does perform this check via `verify_parent_block_hash`, which reads the stored `block_hash` of block N-1 and returns `Err(StateSyncError::ParentBlockHashMismatch)` on mismatch: [5](#0-4) 

The p2p sync path has no equivalent guard.

---

### Impact Explanation

The concrete corrupted value is `StorageBlockHeader.parent_hash` for block N, set to an attacker-controlled felt instead of the real `block_hash` of block N-1. Any RPC call that reads and returns the block header (e.g., `starknet_getBlockWithTxHashes`) will serve this wrong `parent_hash` as an authoritative value.

Additionally, `calculate_block_hash` chains `previous_block_hash` as the final input to the Poseidon hash: [6](#0-5) 

Any downstream consumer that recomputes the block hash using the stored `parent_hash` (e.g., a re-execution tool, a proof verifier, or a block hash CLI invocation) will produce a hash that diverges from the on-chain value.

**Impact: High** — RPC returns an authoritative-looking wrong `parent_hash` for the block header. Note: in the p2p sync path the `block_hash` field is taken directly from the peer (not recomputed via `calculate_block_hash`), so the "block hash chain broken" framing in the question overstates the immediate sequencer-side impact; the concrete, reachable harm is wrong header data served via RPC.

---

### Likelihood Explanation

Any peer the node queries for headers can trigger this. The node sends a `Query` and accepts whatever `SignedBlockHeader` the peer returns, subject only to the block-number and signature-count checks. No operator privilege is required.

---

### Recommendation

Before returning `Ok(Some(signed_block_header))` in `parse_data_for_block`, read the stored header for `block_number.prev()` from `storage_reader` and assert:

```rust
if let Some(prev_block_number) = block_number.prev() {
    let prev_hash = storage_reader
        .begin_ro_txn()?
        .get_block_header(prev_block_number)?
        .ok_or(/* fatal */)?
        .block_hash;
    if prev_hash != signed_block_header.block_header.block_header_without_hash.parent_hash {
        return Err(ParseDataError::BadPeer(BadPeerError::ParentHashMismatch { ... }));
    }
}
```

This mirrors the existing `verify_parent_block_hash` logic already present in `apollo_central_sync`. [7](#0-6) 

---

### Proof of Concept

1. Node has block N-1 stored with `block_hash = H_{N-1}`.
2. Malicious peer responds to the header query for block N with a `SignedBlockHeader` where `parent_hash = 0xdeadbeef` (≠ `H_{N-1}`) and `block_number = N`, `signatures.len() == 1`.
3. `parse_data_for_block` passes both checks and returns `Ok(Some(...))`.
4. `write_to_storage` → `append_header` stores `StorageBlockHeader { parent_hash: 0xdeadbeef, ... }` for block N.
5. `get_block_header(N)` returns a `BlockHeader` with `parent_hash = 0xdeadbeef`.
6. Calling `calculate_block_hash` with this stored `parent_hash` produces a hash that differs from the real on-chain hash of block N. [8](#0-7)

    // Compares the block's parent hash to the stored block.
    fn verify_parent_block_hash(
        &self,
        block_number: BlockNumber,
        block: &Block,
    ) -> StateSyncResult {
        let prev_block_number = match block_number.prev() {
            None => return Ok(()),
            Some(bn) => bn,
        };
        let prev_hash = self
            .reader
            .begin_ro_txn()?
            .get_block_header(prev_block_number)?
            .ok_or(StorageError::DBInconsistency {
                msg: format!(
                    "Missing block {prev_block_number} in the storage (for verifying block \
                     {block_number}).",
                ),
            })?
            .block_hash;

        if prev_hash != block.header.block_header_without_hash.parent_hash {
            // Block parent hash mismatch, log and restart sync loop.
            warn!(
                "Detected reorg in central. block number {block_number} had hash {prev_hash} but \
                 the next block's parent hash is {}.",
                block.header.block_header_without_hash.parent_hash
            );
            CENTRAL_SYNC_FORKS_FROM_FEEDER.increment(1);
            return Err(StateSyncError::ParentBlockHashMismatch {
                block_number,
                expected_parent_block_hash: block.header.block_header_without_hash.parent_hash,
                stored_parent_block_hash: prev_hash,
            });
        }

pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
