### Title
Integer Division Truncation in `NewDynamicFeeChecker` Allows Zero-Fee Cosmos Transactions When `NoBaseFee = true` - (File: `ante/evm/fee_checker.go`)

---

### Summary

`NewDynamicFeeChecker` computes `feeCap` via integer division `fee / gas`. When `fee < gas`, Go integer truncation floors `feeCap` to `0`. If the feemarket parameter `NoBaseFee = true` (which forces `baseFee = 0`), the `feeCap < baseFee` guard passes trivially (`0 < 0` is false), and the downstream `EffectiveGasPrice` call returns `0`, producing an `effectiveFee` of zero. The `DeductFeeDecorator` then skips deduction entirely. Any unprivileged user can submit a Cosmos-SDK transaction with a dust fee and a large gas limit and have it committed with zero fees charged.

---

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` computes the per-gas fee cap as:

```go
// ante/evm/fee_checker.go line 83
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

`sdkmath.Int.Quo` is integer (truncating) division. If a user submits a Cosmos tx with `fee = 1` (1 unit of EVM denom) and `gas = 1_000_000`, then `feeCap = 1 / 1_000_000 = 0`.

The next guard is:

```go
// ante/evm/fee_checker.go line 86-88
if feeCap.LT(baseFeeInt) {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, ...)
}
```

`baseFeeInt` is derived from `types.GetBaseFee` (`x/evm/types/utils.go` line 244). That function calls `feemarketParams.GetBaseFee()` (`x/feemarket/types/params.go` line 139). When `NoBaseFee = true`, `GetBaseFee()` returns `nil`, and `types.GetBaseFee` converts `nil` to `new(big.Int)` (i.e., `0`). So `baseFeeInt = 0`.

The guard evaluates `0 < 0 = false` — it passes.

Then:

```go
// ante/evm/fee_checker.go line 91
effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
```

`EffectiveGasPrice` (`x/evm/types/utils.go` line 232) computes `min(tipCap + baseFee, feeCap) = min(MaxInt64 + 0, 0) = 0`.

```go
// ante/evm/fee_checker.go line 94-99
effectiveFee := sdk.Coins{{
    Denom:  denom,
    Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)), // 0 * gas = 0
}}
```

`effectiveFee = 0`. This is returned to `DeductFeeDecorator.checkDeductFee` (`ante/evm/nativefee.go` line 110):

```go
if !fee.IsZero() {
    err := evmkeeper.DeductFees(...)
}
```

Since `fee.IsZero()` is true, no fees are deducted. The transaction is committed with zero fees paid.

The `NoBaseFee = true` condition is reached when:
- The feemarket param `NoBaseFee` is set to `true` (explicitly documented as "needed for 0 price calls"), **or**
- London hardfork is enabled but `NoBaseFee = true` is set via governance.

The `types.GetBaseFee` path that returns `0` (not `nil`) is critical: it prevents the fallback to `checkTxFeeWithValidatorMinGasPrices`, so the truncation bug is fully exposed.

---

### Impact Explanation

Any unprivileged user can submit Cosmos-SDK transactions (bank sends, governance votes, IBC messages, staking operations, etc.) with a dust fee (e.g., `1 aevmos`) and an arbitrarily large gas limit, paying **zero** effective fees. This is a direct fee mis-accounting bug in the ante handler: the fee collector module receives nothing, validators are not compensated, and the economic spam-prevention mechanism is completely bypassed for all Cosmos-native message types.

This matches the allowed High impact: *"ante handler bug that permits valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

`NoBaseFee = true` is an explicitly supported and documented configuration ("needed for 0 price calls"). Chains that deploy Ethermint with `NoBaseFee = true` — including testnets, chains that want to allow zero-fee transactions, or chains that have not yet activated the fee market — are fully exposed. The attack requires only crafting a standard Cosmos tx with `fee = 1` and a large `gas` value, which is trivially achievable by any wallet or CLI user. No special privileges, keys, or network position are required.

---

### Recommendation

Replace the truncating integer division with a ceiling division (round up) when computing `feeCap`, so that a user who provides any non-zero fee is never credited with a zero per-gas price:

```go
// Instead of:
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))

// Use ceiling division:
gasInt := sdkmath.NewIntFromUint64(gas)
feeCap := fee.Add(gasInt).Sub(sdkmath.OneInt()).Quo(gasInt)
// or equivalently: feeCap = (fee + gas - 1) / gas
```

Alternatively, add an explicit guard: if `fee > 0` but `feeCap == 0`, reject the transaction with `ErrInsufficientFee`. This mirrors the fix pattern recommended in the referenced Endaoment report.

---

### Proof of Concept

**Precondition:** Chain has London hardfork enabled and `NoBaseFee = true` in feemarket params.

1. Attacker constructs a Cosmos tx (e.g., `MsgSend`) with:
   - `fee = sdk.NewCoins(sdk.NewCoin("aevmos", sdkmath.NewInt(1)))`
   - `gas = 1_000_000`

2. Tx enters `DeductFeeDecorator.AnteHandle` → calls `NewDynamicFeeChecker`.

3. `types.GetBaseFee(height, ethCfg, feemarketParams)`:
   - London enabled → does not return `nil`.
   - `feemarketParams.GetBaseFee()` returns `nil` (NoBaseFee=true).
   - Returns `new(big.Int)` = `0`. [1](#0-0) 

4. `feeCap = 1 / 1_000_000 = 0` (integer truncation). [2](#0-1) 

5. Guard `feeCap.LT(baseFeeInt)` = `0 < 0` = false → passes. [3](#0-2) 

6. `effectivePrice = EffectiveGasPrice(0, 0, MaxInt64) = min(MaxInt64, 0) = 0`. [4](#0-3) 

7. `effectiveFee = 0 * 1_000_000 = 0`. [5](#0-4) 

8. `DeductFeeDecorator.checkDeductFee`: `fee.IsZero()` = true → no deduction. [6](#0-5) 

9. Tx is committed. Attacker paid 0 fees. Fee collector receives nothing.

The `NoBaseFee = true` → `baseFee = 0` path is confirmed by `feemarket/types/params.go`: [7](#0-6)
