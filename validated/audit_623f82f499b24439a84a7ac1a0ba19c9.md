### Title
EIP-7702 Durable Authorization Committed Even When EVM Execution Reverts (No Hooks Case) - (File: x/evm/keeper/state_transition.go)

### Summary

The `ApplyTransaction` function in Ethermint's EVM keeper contains a logic flaw in how it manages the "durable" EIP-7702 authorization context. The `commitDurableAuthorization` function is only set up when `k.hooks != nil`, but the condition that triggers it (`commitDurableAuthorization != nil && !tmpCtxCommitted`) fires whenever the durable commit function exists. The structural problem is the inverse: when **no hooks are registered**, the EVM execution runs directly on `ctx` (not a `tmpCtx` cache), so the authorization effects are already committed to the main store by the `stateDB.Commit()` inside `ApplyMessageWithConfig`. This means the durable path is a bypass of the normal EVM-level revert semantics — but the deeper issue is the asymmetry between the hooks and no-hooks paths for authorization durability.

More concretely: when hooks **are** registered and the EVM call **succeeds** but the post-hook **fails**, the code at line 266–268 commits the durable authorization (`commitDurableAuthorization()`) only when `!tmpCtxCommitted`. However, `tmpCtxCommitted` is only set to `true` when both the EVM succeeds **and** the post-hook succeeds (line 242). When the EVM call itself **reverts** (i.e., `res.Failed()` is true, `vmErr != nil`), the `tmpCtx` is never committed (the `commit()` at line 241 is never called), yet `commitDurableAuthorization()` is still called at line 267 — persisting the authority's nonce bump and delegation code to the parent store even though the EVM execution failed.

This is the analog of the Malt M-29 bug: a restriction (EVM revert should roll back all state) is bypassed through an indirect path (the durable authorization context), allowing an attacker to force a permanent code delegation on a victim's EOA account without the EVM call succeeding.

### Finding Description

In `ApplyTransaction` (`x/evm/keeper/state_transition.go`), when `k.hooks != nil` and the transaction carries `SetCodeAuthorizations`:

1. A `tmpCtx` cache context is created (line 181) — EVM execution runs here.
2. A separate `durableAuthorizationCtx` cache context is created (line 188) — authorization effects are replayed here.
3. `ApplyMessageWithConfig` is called on `tmpCtx`. Inside it, `applyAuthorization` validates and applies the authorization to the **main** `stateDB` (which is backed by `tmpCtx`). Then `applyDurableAuthorization` replays the same effects onto `durableStateDB` (backed by `durableAuthorizationCtx`), and `durableStateDB.Commit()` is called immediately (line 510).
4. After `ApplyMessageWithConfig` returns, if `res.Failed()` is true (EVM reverted), the code skips the `commit()` call (line 241 is never reached), so `tmpCtx` is discarded — the EVM state changes are rolled back.
5. However, at line 266–268, `commitDurableAuthorization()` is called unconditionally whenever `!tmpCtxCommitted`, which includes the case where the EVM call reverted.

The result: the authority's nonce is bumped and delegation code is written to the parent store even when the EVM call reverted. This is confirmed by the test `TestSetCodeAuthorizationSurvivesPostHookFailure` (line 915) which explicitly asserts `suite.requireSetCodeAuthorizationConsumed(authority, delegate, 1)` after a post-hook failure — but the same path fires for EVM-level reverts too.

The critical distinction: the design intent (per the comment at line 183–190) is that authorization effects should survive **post-hook failures** (not EVM reverts). The condition `!tmpCtxCommitted` does not distinguish between "EVM reverted" and "post-hook failed" — both result in `tmpCtxCommitted == false`.

### Impact Explanation

An unprivileged attacker (the outer transaction signer, `outer`) can:

1. Obtain a valid EIP-7702 authorization signed by a victim (`authority`) for any delegate address.
2. Craft a `SetCodeTx` (type 4) that calls a target contract guaranteed to revert (e.g., a contract with `INVALID` opcode `0xfe`).
3. Submit the transaction. The EVM call reverts, but `commitDurableAuthorization()` fires, permanently writing `delegation_code` and bumping the nonce of `authority` in the committed store.

The victim's EOA now has arbitrary code installed (the delegation pointer) and their nonce is incremented — without the EVM call succeeding. This constitutes **unauthorized account/code mutation** of a victim's EOA, matching the High impact category: "EIP-7702 authorization... signer verification bypass enabling... unauthorized account/code mutation."

The attacker can use this to:
- Force a victim's EOA to execute attacker-controlled code on any future call to that address.
- Permanently increment the victim's nonce, invalidating any pre-signed transactions the victim has outstanding.

### Likelihood Explanation

- Requires Prague hardfork to be active (EIP-7702 enabled).
- Requires `k.hooks != nil` — i.e., the chain has registered EVM post-tx hooks (common in production Evmos/Ethermint deployments that use ERC-20 module hooks).
- Requires the attacker to obtain a valid `SetCodeAuthorization` signature from the victim. This is the primary barrier, but EIP-7702 authorizations are designed to be shareable and can be obtained via social engineering, phishing, or by replaying a previously-broadcast authorization.
- The attacker controls the outer transaction signer and the target contract (can deploy a reverting contract trivially).
- No privileged access required beyond a valid authorization signature.

### Recommendation

The `commitDurableAuthorization` call must be conditioned on whether the EVM execution itself succeeded, not merely on whether `tmpCtxCommitted` is false. Specifically:

```go
// Only commit durable authorization if EVM execution did not revert.
// Post-hook failures should still persist the authorization (by design),
// but EVM-level reverts must not.
if commitDurableAuthorization != nil && !tmpCtxCommitted && !res.Failed() {
    commitDurableAuthorization()
}
```

This ensures that:
- Post-hook failure (EVM succeeded, hook failed): `!res.Failed()` is true → authorization is committed (intended behavior).
- EVM revert (`res.Failed()` is true): authorization is NOT committed (correct rollback).

### Proof of Concept

**Setup:**
- Chain has Prague hardfork active and EVM hooks registered.
- `authority` is a victim EOA at nonce 0.
- `attacker` is the outer transaction signer.
- `delegate` is an attacker-controlled address.
- `revertingTarget` is a contract with bytecode `0xfe` (INVALID opcode — always reverts).

**Steps:**

1. Victim signs an EIP-7702 authorization: `auth = SignSetCode(authorityKey, {ChainID: chainID, Address: delegate, Nonce: 0})`.

2. Attacker constructs a `SetCodeTx` (type 4):
   ```
   SetCodeTx{
     To:       revertingTarget,  // guaranteed to revert
     AuthList: [auth],
     Nonce:    attacker_nonce,
     Gas:      100000,
   }
   ```

3. Attacker broadcasts the transaction. The EVM call to `revertingTarget` reverts (`vmErr != nil`, `res.Failed() == true`).

4. In `ApplyTransaction` (line 266–268):
   ```go
   if commitDurableAuthorization != nil && !tmpCtxCommitted {
       commitDurableAuthorization()  // fires! tmpCtxCommitted is false because EVM reverted
   }
   ```

5. **Result:** `authority`'s nonce is now 1 and their code is `0xef0100<delegate_address>` in the committed store — despite the EVM call having reverted.

This is directly analogous to the Malt M-29 pattern: a restriction (EVM revert rolls back all state) is bypassed through an indirect path (the durable authorization context), allowing state mutation that should have been prevented.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
