### Title
Blocklisted EVM Recipient Permanently Locks User Funds Bridged from NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

When a user bridges a blocklist-capable token (e.g., USDC, USDT) from NEAR to EVM, their tokens are burned/locked on NEAR before EVM finalization occurs. If the EVM recipient address is subsequently blocklisted by the token contract, every call to `finTransfer` on EVM will revert. Because the NEAR bridge has no cancel-transfer or user-refund path, the user's funds are permanently frozen on NEAR with no recovery mechanism.

---

### Finding Description

**EVM side — `OmniBridge.sol` `finTransfer`:**

The function marks the destination nonce as used at line 287, then verifies the MPC signature, and finally transfers tokens to `payload.recipient`:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC20(payload.tokenAddress).safeTransfer(             // line 351
    payload.recipient,
    payload.amount
);
```

For native EVM tokens (the `else` branch at line 350–354), `SafeERC20.safeTransfer` is used. USDC and USDT implement contract-level admin-controlled blocklists; a transfer to a blocklisted address reverts. Because Solidity reverts roll back all state changes, the nonce marking at line 287 is also rolled back, so the nonce is not permanently consumed. However, the relayer can retry indefinitely and will always fail.

**NEAR side — permanent lock:**

When the user initiates the transfer via `ft_on_transfer` → `init_transfer` → `init_transfer_internal`, the tokens are immediately burned or locked:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

The pending transfer is stored in `pending_transfers`. The only code paths that remove a pending transfer are:

- `sign_transfer_callback` — only removes it when `fee.is_zero()` (line 657).
- `claim_fee_callback` — removes it after a successful EVM `finTransfer` is proven (line 1094).

There is no user-callable cancel, timeout, or refund function. The MPC signature covers the recipient address (it is part of the Borsh-encoded payload that is signed), so the recipient cannot be changed without a new MPC signing round — which the protocol does not support for an already-pending transfer.

The result: if `finTransfer` on EVM always reverts because the recipient is blocklisted, the pending transfer on NEAR is never removed, the burned/locked tokens are never restored, and the user's funds are permanently frozen.

---

### Impact Explanation

**Critical — permanent freezing of user funds.**

The user's tokens are burned on NEAR (for deployed/bridge tokens) or locked (for native NEAR tokens). There is no on-chain path to recover them once the EVM recipient is blocklisted. This matches the allowed impact class: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Low-to-Medium.**

USDC and USDT are the primary tokens with admin-controlled blocklists and are among the most commonly bridged assets. A user's address can be blocklisted:

- Due to regulatory action after the transfer is initiated but before finalization.
- If the user's address is associated with a sanctioned entity.
- Maliciously, if a token deployer registers a custom ERC-20 with a blocklist via `addCustomToken` (admin-only, so this path requires a compromised admin — out of scope), but the USDC/USDT path requires no privileged attacker.

The window between `initTransfer` on NEAR and `finTransfer` on EVM is non-zero (relayer latency, MPC signing time), creating a realistic race condition.

---

### Recommendation

1. **Pull-based delivery:** Instead of pushing tokens directly to `payload.recipient` in `finTransfer`, hold them in the bridge contract and allow the recipient to claim them via a separate `claimTransfer` call. This decouples delivery failure from fund loss.

2. **Cancel/refund path on NEAR:** Add a user-callable `cancel_transfer` function on the NEAR bridge that, after a configurable timeout with no successful finalization proof, burns the pending transfer record and re-mints/unlocks the tokens to the original sender.

3. **Recipient validation:** For known blocklist-capable tokens, validate on-chain (or off-chain via relayer) that the recipient is not blocklisted before signing and submitting `finTransfer`.

---

### Proof of Concept

1. Alice holds USDC on NEAR and calls `ft_transfer_call` to the NEAR bridge with `recipient = "0xAlice"` (her EVM address). `init_transfer_internal` burns her USDC on NEAR and stores the `TransferMessage` in `pending_transfers`. [1](#0-0) 

2. The relayer calls `sign_transfer`; the MPC signs a payload that includes `recipient = 0xAlice`. [2](#0-1) 

3. Before the relayer submits `finTransfer` on EVM, USDC's admin blocklists `0xAlice` (e.g., regulatory freeze).

4. The relayer calls `finTransfer` on EVM. The nonce is marked used at line 287, signature is verified, then `safeTransfer(0xAlice, amount)` reverts because `0xAlice` is blocklisted. The entire transaction reverts, including the nonce marking. [3](#0-2) 

5. Every subsequent retry of `finTransfer` also reverts. The nonce is never permanently consumed, but the transfer can never succeed.

6. On NEAR, `claim_fee_callback` is never triggered (no successful EVM finalization proof exists). The only removal paths for the pending transfer are `sign_transfer_callback` (fee-zero only) and `claim_fee_callback`. [4](#0-3) [5](#0-4) 

7. Alice's USDC is permanently burned on NEAR with no recovery path.
