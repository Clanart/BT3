### Title
`migrate_deployed_token` Does Not Migrate `locked_tokens` Accounting, Permanently Disabling Collateralization Checks for the New Token - (File: near/omni-bridge/src/lib.rs)

### Summary
`migrate_deployed_token` in the NEAR omni-bridge contract updates all token identity mappings (`deployed_tokens`, `deployed_tokens_v2`, `token_id_to_address`, `token_address_to_id`, `migrated_tokens`) when replacing `old_token` with `new_token`, but it never migrates the `locked_tokens` entry from `(origin_chain, old_token)` to `(origin_chain, new_token)`. After migration, the new token has no `locked_tokens` entry, causing every subsequent `lock_tokens` and `unlock_tokens` call for that token to silently return `LockAction::Unchanged` â€” permanently disabling the bridge's collateralization accounting for the new token.

### Finding Description

`migrate_deployed_token` (lib.rs lines 1604â€“1664) performs the following updates:

```rust
self.deployed_tokens.remove(&old_token);
self.deployed_tokens.insert(&new_token);
self.deployed_tokens_v2.remove(&old_token);
self.deployed_tokens_v2.insert(&new_token, &origin_chain);
self.token_id_to_address.remove(&(origin_chain, old_token.clone()));
self.token_id_to_address.insert(&(origin_chain, new_token.clone()), &origin_address);
self.token_address_to_id.insert(&origin_address, &new_token);
self.migrated_tokens.insert(&old_token, &new_token);
```

It does **not** do:
```rust
// MISSING:
let old_locked = self.locked_tokens.remove(&(origin_chain, old_token));
if let Some(amount) = old_locked {
    self.locked_tokens.insert(&(origin_chain, new_token), &amount);
}
```

The `locked_tokens` map is keyed by `(ChainKind, AccountId)`. After migration, `locked_tokens[(origin_chain, old_token)]` is orphaned with its old value, and `locked_tokens[(origin_chain, new_token)]` does not exist.

The `lock_tokens` and `unlock_tokens` internal functions in `token_lock.rs` both use an early-return pattern when no entry is found:

```rust
// lock_tokens (token_lock.rs:55-56)
let Some(current_amount) = self.locked_tokens.get(&key) else {
    return LockAction::Unchanged;
};

// unlock_tokens (token_lock.rs:78-80)
let Some(available) = self.locked_tokens.get(&key) else {
    return LockAction::Unchanged;
};
require!(available >= amount, TokenLockError::InsufficientLockedTokens.as_ref());
```

The `require!(available >= amount, ...)` collateralization guard is only reached when an entry exists. With no entry for `new_token`, the guard is silently skipped on every call.

This affects all three call sites that use the new token after migration:

1. **`process_fin_transfer_to_near`** (lib.rs line 1881â€“1885): calls `unlock_tokens_if_needed(origin_chain, &token, amount)` â€” silently skips the unlock and collateralization check.
2. **`process_fin_transfer_to_other_chain`** (lib.rs lines 1997â€“2005): calls `unlock_tokens_if_needed` and `lock_tokens_if_needed` â€” both silently skip.
3. **`init_transfer_internal`** (lib.rs lines 1853â€“1857): calls `lock_tokens_if_needed(destination_chain, &token_id, amount)` â€” silently skips.

### Impact Explanation

The `locked_tokens` map is the bridge's on-chain collateralization ledger. It tracks how many tokens are locked on each source chain, enforcing that the bridge cannot release more tokens than are backed by locked collateral. After `migrate_deployed_token` is called, this entire accounting layer is permanently disabled for the new token:

- Every `fin_transfer` for `new_token` bypasses the `require!(available >= amount)` guard, allowing the bridge to release tokens without decrementing any locked balance.
- Every `init_transfer` for `new_token` bypasses the lock increment, so the bridge never records new collateral being locked.
- The orphaned `locked_tokens[(origin_chain, old_token)]` entry is never decremented, creating a phantom locked balance that inflates the apparent collateral.

This constitutes **accounting corruption that breaks bridge collateralization**: the bridge's internal ledger permanently diverges from the actual locked state on the source chain for the migrated token.

### Likelihood Explanation

`migrate_deployed_token` is a DAO-only function intended for legitimate operational use (e.g., upgrading a token contract to a new account). It is a documented, expected operation. Once called, the broken state is permanent â€” there is no automatic recovery path. Any user who subsequently submits a valid `fin_transfer` proof for the new token will trigger the bypassed accounting path. The likelihood of the DAO calling this function is high (it exists precisely to be used), and the broken state is triggered by any normal bridge user after migration.

### Recommendation

In `migrate_deployed_token`, migrate the `locked_tokens` entry from the old token to the new token for all chain kinds that have an entry. At minimum, migrate the `origin_chain` entry:

```rust
// After updating token mappings, migrate locked_tokens:
if let Some(locked_amount) = self.locked_tokens.remove(&(origin_chain, old_token.clone())) {
    self.locked_tokens.insert(&(origin_chain, new_token.clone()), &locked_amount);
}
```

If the token may have `locked_tokens` entries for multiple chain kinds (e.g., `Sol`, `Base`, etc.), all such entries should be migrated. Alternatively, iterate over all known chain kinds and migrate any existing entries.

**Correction â€” the scenario applies to cross-chain transfers where destination â‰  origin:**

- `unlock_tokens_if_needed(Sol, new_token, amount)` is called
- `get_token_origin_chain(new_token) == Eth â‰  Sol`, so it proceeds to `unlock_tokens(Sol, new_token, amount)`

This means after migration, any user can bridge `new_token` between non-origin chains (e.g., NEAR â†” Solana, NEAR â†” Base) without the bridge ever tracking or enforcing the locked collateral, breaking the bridge's collateralization invariant for the migrated token.
