### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash corresponds to a previously declared contract class. An unprivileged contract deployer can replace a contract's class with an arbitrary, undeclared class hash, permanently bricking the contract and freezing any funds it holds or controls.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function (lines 878–916) updates a contract's class hash in the state without validating that the new class hash has been declared on-chain. The code itself contains an explicit acknowledgment of this missing check:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The function reads the current state entry, constructs a new `StateEntry` with the attacker-supplied `class_hash`, and writes it directly to `contract_state_changes` — with zero validation that `class_hash` exists in the compiled class facts or was ever declared via a `declare` transaction. ---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, the OS has no compiled class to dispatch execution to. Every subsequent call to that contract — including calls to withdraw, transfer, or otherwise recover assets — will fail at the entry point dispatch stage. Any ERC-20 balances, vault deposits, or other assets held by or exclusively accessible through that contract are permanently inaccessible. The state corruption is committed to the proven block output and cannot be reversed without a protocol-level intervention.

---

### Likelihood Explanation

**Medium-High.** The `replace_class` syscall is a standard, permissionless operation available to any contract. An attacker needs only to:
1. Deploy a contract (or control an existing one that holds or gates funds), and
2. Have that contract invoke `replace_class` with an arbitrary undeclared felt value as the class hash.

No privileged role, leaked key, or external dependency is required. The attack is a single transaction.

---

### Recommendation

Before writing the new `StateEntry`, validate that `class_hash` is present in the set of declared classes. Concretely, perform a lookup in `contract_class_changes` (or the global compiled class facts bundle) and assert a non-zero compiled class hash exists for the given `class_hash`. This is exactly what the existing TODO comment describes and what the analogous `dict_update` call in `execute_declare_transaction` enforces via `prev_value=0` / `assert_not_zero(compiled_class_hash)`. [2](#0-1) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` which holds user funds and exposes a `replace_class` call path.
2. Attacker submits an invoke transaction that causes `VaultContract` to call the `replace_class` syscall with `class_hash = 0x1337` (a felt that has never been declared).
3. `execute_replace_class` in `syscall_impls.cairo` (lines 896–913) accepts the call: gas is deducted, the response header is written with `failure_flag=0`, and `contract_state_changes` is updated with the invalid class hash.
4. The block is proven and committed. `VaultContract`'s on-chain class hash is now `0x1337`.
5. Any subsequent call to `VaultContract` (withdraw, transfer, etc.) fails because the OS cannot locate a compiled class for `0x1337` in the compiled class facts bundle.
6. All funds held by `VaultContract` are permanently frozen. [3](#0-2)

        // Not enough gas to execute the syscall.
