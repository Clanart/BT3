### Title
`LegacyEip712SigVerificationDecorator` Silently Skips All EIP-712 Signature Verification When `simulate=true` — (`File: ante/cosmos/eip712.go`)

### Summary

The `LegacyEip712SigVerificationDecorator.AnteHandle` function in `ante/cosmos/eip712.go` contains an unconditional early-return at line 157–159 that bypasses the entire EIP-712 signature verification step whenever `simulate=true`. Unlike the standard Cosmos SDK `SigVerificationDecorator` (which also skips verification in simulate mode but only because the signature bytes are genuinely absent), the legacy EIP-712 path carries the real `FeePayerSig` in the `ExtensionOptionsWeb3Tx` extension option and performs a full cryptographic check — yet the simulate guard silently skips it. This is the direct analog of the "phantom permit" pattern: a function that is supposed to perform a critical security check (signature verification) silently becomes a no-op under a reachable, attacker-controlled condition, providing no error and no indication that verification was skipped.

### Finding Description

**Root cause — `ante/cosmos/eip712.go`, lines 157–159:**

```go
if simulate {
    return next(ctx, tx, simulate)
}
```

This guard appears after all the sequence/account-number checks but before `VerifySignature` is called. When `simulate=true`, the function returns `next(ctx, tx, simulate)` without ever calling `VerifySignature`, meaning:

1. The `FeePayerSig` inside `ExtensionOptionsWeb3Tx` is never recovered or verified.
2. The `pubKey.Equals(pk)` check (line 282) is never reached.
3. The `recoveredFeePayerAcc.Equals(feePayer)` check (line 288) is never reached.
4. The final `ethcrypto.VerifySignature(pubKey.Bytes(), sigHash, feePayerSig[:len(feePayerSig)-1])` (line 294) is never reached.

The `simulate` path is reachable by any unprivileged user via the standard Cosmos SDK `SimulateTx` gRPC endpoint (`/cosmos.tx.v1beta1.Service/Simulate`) and via the `eth_estimateGas` / `eth_call` JSON-RPC paths that internally invoke the ante handler with `simulate=true`.

**Contrast with the standard Cosmos path:** In `newCosmosAnteHandler` (line 206 of `evmd/ante/handler_options.go`), the standard `authante.NewSigVerificationDecorator` is used. The Cosmos SDK's own implementation also skips verification in simulate mode, but that is safe because the Cosmos SDK simulate path sends zero-length signature bytes by convention — there is no real signature to verify. In the legacy EIP-712 path, however, the `FeePayerSig` is a real 65-byte ECDSA signature embedded in the extension option. The simulate guard was copied from the Cosmos SDK pattern without accounting for the fact that the EIP-712 decorator performs its own independent cryptographic check on a field that is always present.

**The phantom-permit analogy:** Just as `STETHVault.mintWithPermit` would silently succeed (or fail with a misleading error) because the underlying asset has no `permit`, the `LegacyEip712SigVerificationDecorator` silently succeeds (returns `nil` error) when `simulate=true` even though the underlying EIP-712 signature check is never performed. The function is callable, appears to succeed, and produces no error — a phantom verification.

### Impact Explanation

**High — EIP-712 authorization bypass enabling forged execution.**

An attacker who can submit a `simulate=true` request (via `SimulateTx` gRPC or any JSON-RPC path that calls the ante handler in simulate mode) with a crafted legacy EIP-712 Cosmos transaction can:

1. Forge the `FeePayer` field to any victim address.
2. Supply a garbage or zero `FeePayerSig`.
3. Have the ante handler accept the transaction as valid in simulate mode.

The direct consequence in simulate mode is that the simulation returns a successful gas estimate for a transaction that would fail in `DeliverTx`/`FinalizeBlock`. This is a **High** impact because:
- It enables an attacker to obtain a valid gas estimate for a transaction they cannot actually authorize, which can be used to craft replay or front-running attacks.
- More critically, if any downstream component (e.g., a relayer, a fee-grant mechanism, or a mempool pre-check) trusts the simulate result as proof of validity, the attacker can bypass the EIP-712 authorization check entirely for those components.
- The simulate path in Cosmos SDK does execute state-mutating logic in a cached context; any module that does not separately re-verify authorization in its `MsgServer` handler is vulnerable to unauthorized state mutation via the simulate path.

### Likelihood Explanation

**High.** The `SimulateTx` gRPC endpoint is publicly accessible on any Ethermint node. No privileged access is required. The attacker only needs to:
1. Construct a valid Cosmos transaction body with `ExtensionOptionsWeb3Tx` extension option.
2. Set `simulate=true` (which is the default for the simulate endpoint).
3. Provide any 65-byte value as `FeePayerSig` — it will never be checked.

The legacy EIP-712 path is explicitly still wired in `newLegacyCosmosAnteHandlerEip712` (line 56 of `evmd/ante/evm_handler.go`) and is reachable via the `/ethermint.types.v1.ExtensionOptionsWeb3Tx` extension option type URL (line 80–82 of `evmd/ante/ante.go`).

### Recommendation

Remove the unconditional `simulate` early-return from `LegacyEip712SigVerificationDecorator.AnteHandle`, or, if simulate-mode skipping is intentional for gas estimation, add an explicit check that the `FeePayerSig` is non-empty and structurally valid before skipping. The safest fix is to always run `VerifySignature` regardless of `simulate`, since the EIP-712 decorator's verification is independent of the Cosmos SDK signature bytes convention:

```go
// Remove or guard this block:
// if simulate {
//     return next(ctx, tx, simulate)
// }
```

If simulate-mode skipping must be preserved for compatibility, at minimum add a structural check:

```go
if simulate {
    // Still validate that FeePayerSig is present and well-formed,
    // even if full cryptographic verification is skipped.
    extOpt := ... // extract as in VerifySignature
    if len(extOpt.FeePayerSig) != ethcrypto.SignatureLength {
        return ctx, errorsmod.Wrap(errortypes.ErrorInvalidSigner, "invalid fee payer sig length in simulate mode")
    }
    return next(ctx, tx, simulate)
}
```

### Proof of Concept

1. Construct a Cosmos SDK transaction with:
   - Body containing any valid `sdk.Msg` (e.g., `MsgSend`) with `FeePayer = victim_address`.
   - Extension option of type `/ethermint.types.v1.ExtensionOptionsWeb3Tx` with `TypedDataChainID = <chain_id>`, `FeePayer = victim_address`, `FeePayerSig = [0x00 * 65]` (65 zero bytes).
   - `SignatureV2` with `SignMode = SIGN_MODE_LEGACY_AMINO_JSON` and empty `Signature` bytes.

2. Submit via `POST /cosmos/tx/v1beta1/simulate` (gRPC-gateway) or `grpc.Invoke("/cosmos.tx.v1beta1.Service/Simulate", ...)`.

3. Observe: the ante handler returns no error and provides a gas estimate, despite the `FeePayerSig` being entirely invalid. The `VerifySignature` function at line 161 of `ante/cosmos/eip712.go` is never reached because the `simulate` guard at line 157 exits early.

4. Contrast: submitting the same transaction via `BroadcastTx` (non-simulate) returns `ErrorInvalidSigner` because `VerifySignature` is called and the zero-byte signature fails `ethcrypto.Ecrecover`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
