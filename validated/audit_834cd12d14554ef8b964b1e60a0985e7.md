### Title
Proof-Facts Credential Not Bound to Transaction Sender — Same `SnosProofFacts`+`Proof` Reusable Across Unrelated Invoke V3 Transactions - (File: `crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

The `validate_proof_facts` function in the blockifier, and the corresponding `check_proof_facts` function in the Cairo OS, validate that a `SnosProofFacts` credential contains a globally-valid block hash and config hash, but neither check binds the credential to the specific transaction's `sender_address`, `nonce`, or `calldata`. Because the proof manager's `run_proof_verification` skips re-verification for any `proof_facts` already stored, the same `(proof_facts, proof)` pair can be freely reused across unrelated Invoke V3 transactions from different senders. This is the direct sequencer analog of the "Permission Reuse in Orders" bug: the proof credential is the "permission account," and the Invoke V3 transaction is the "order."

---

### Finding Description

**The `SnosProofFacts` struct contains no transaction-specific fields.** [1](#0-0) 

The struct holds only `proof_version`, `program_hash`, `block_number`, `block_hash`, and `config_hash`. There is no `sender_address`, `nonce`, `calldata`, or `transaction_hash`.

**`validate_proof_facts` validates only global chain state, not the submitting account.** [2](#0-1) 

The four checks performed are: (1) `program_hash` is in the allowed list, (2) `block_number` is not too recent, (3) `block_hash` matches the stored block hash, (4) `config_hash` matches the virtual OS config hash. None of these checks bind the credential to the `sender_address`, `nonce`, or `calldata` of the transaction that carries it.

**The Cairo OS `check_proof_facts` has the same gap.** [3](#0-2) 

It validates `base_block_hash` against storage and `starknet_os_config_hash` against the global context, but never asserts any relationship between the proof facts and the `sender_address` or `transaction_hash` computed just above it in `execute_invoke_function_transaction`. [4](#0-3) 

**`proof_facts` are included in the transaction hash, but the proof is not bound to the transaction hash.** [5](#0-4) 

The transaction hash commits to `sender_address`, `nonce`, `calldata`, and `proof_facts`. Two transactions from different senders carrying the same `proof_facts` will therefore have different transaction hashes. However, the STARK `proof` is a proof of the `proof_facts` public inputs only — it does not commit to the transaction hash, `sender_address`, or `nonce`. The proof is therefore reusable for any transaction that carries the same `proof_facts`.

**`run_proof_verification` skips re-verification for already-stored proof facts.** [6](#0-5) 

Once a `(proof_facts, proof)` pair is stored in the proof manager, any subsequent transaction carrying the same `proof_facts` passes the converter's verification gate unconditionally, regardless of who submitted it or what calldata it carries.

---

### Impact Explanation

An account contract that uses `proof_facts` to skip its `__validate__` signature check — the primary intended use case for client-side proving — can be exploited as follows. An attacker observes a valid `(proof_facts P, proof Q)` pair from Alice's transaction. The attacker submits their own Invoke V3 transaction with the same `P` and `Q` but a different `sender_address`, `calldata`, and `nonce`. The sequencer finds `P` already stored and skips re-verification. `validate_proof_facts` passes because `P` is globally valid. The attacker's account contract receives `P` via `tx_info.proof_facts` and, if it trusts valid proof facts to bypass signature verification, accepts the transaction without a valid signature. The result is an unauthorized transaction accepted through account validation logic.

This matches the Critical impact scope: *"Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce, chain id, fee/resource bound, paymaster, or account-deployment logic."*

---

### Likelihood Explanation

Medium. The attack requires an account contract that uses `proof_facts` to skip signature verification. The client-side proving feature is explicitly designed to enable exactly this pattern. Any account contract that implements "if proof_facts is valid, skip `__validate__`" is directly vulnerable. The attacker needs only to observe a valid `(proof_facts, proof)` pair from the public mempool or finalized blocks — no privileged access is required.

---

### Recommendation

Include the transaction hash (or at minimum `sender_address` + `nonce`) as a validated field inside `SnosProofFacts`. The `validate_proof_facts` function should assert that the `sender_address` and `nonce` embedded in the proof facts match those of the transaction being validated. The Cairo `check_proof_facts` function should enforce the same binding. This ensures a proof credential is cryptographically tied to exactly one transaction, preventing cross-transaction reuse.

---

### Proof of Concept

1. Alice submits Invoke V3 `tx_A`:
   - `sender_address = Alice`, `nonce = N`, `calldata = C_A`
   - `proof_facts = P` (valid: correct `block_hash`, `config_hash`, allowed `program_hash`)
   - `proof = Q` (valid STARK proof of `P`)
2. The sequencer's `run_proof_verification` verifies `Q` against `P` and stores the pair. [7](#0-6) 
3. Bob constructs Invoke V3 `tx_B`:
   - `sender_address = Bob`, `nonce = M`, `calldata = C_B` (arbitrary, different from Alice's)
   - `proof_facts = P` (copied from Alice's transaction)
   - `proof = Q` (copied from Alice's transaction)
4. The sequencer calls `contains_proof(P)` → `true` → skips verification entirely. [6](#0-5) 
5. `validate_proof_facts` checks `block_hash`, `config_hash`, `program_hash` for `P` → all pass. [8](#0-7) 
6. `tx_B` is admitted to the mempool and executed. Bob's account contract receives `proof_facts = P` via `tx_info.proof_facts`. If Bob's account uses `P` to skip `__validate__`, `tx_B` executes `C_B` without a valid signature from Bob.

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }

func check_proof_facts{range_check_ptr, contract_state_changes: DictAccess*}(
    proof_facts_size: felt,
    proof_facts: felt*,
    current_block_number: felt,
    virtual_os_config_hash: felt,
) {
    if (proof_facts_size == 0) {
        return ();
    }

    assert_le(ProofHeader.SIZE + VirtualOsOutputHeader.SIZE, proof_facts_size);

    // Validate the proof header.
    let proof_header = cast(proof_facts, ProofHeader*);
    assert is_program_hash_allowed(proof_header.program_hash) = TRUE;
    // Proof version and variant are for future compatibility.
    assert [proof_header] = ProofHeader(
        proof_version=PROOF_VERSION,
        proof_variant=VIRTUAL_SNOS,
        program_hash=proof_header.program_hash,
    );

    // Validate the virtual OS output header.
    let os_output_header = cast(&proof_facts[ProofHeader.SIZE], VirtualOsOutputHeader*);

    with_attr error_message("Virtual OS output version is not supported") {
        assert os_output_header.output_version = VIRTUAL_OS_OUTPUT_VERSION;
    }

    // Validate that the proof facts block number is not too recent.
    // (This is a sanity check - the following non-zero check ensures that the block hash is
    // not trivial).
    assert_nn_le(
        os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
    );
    // Not all block hashes are stored in the contract; Make sure the requested one is not trivial.
    assert_not_zero(os_output_header.base_block_hash);

    // validate that the proof facts block hash is the true hash of the proof facts block number.
    read_block_hash_from_storage(
        block_number=os_output_header.base_block_number,
        expected_block_hash=os_output_header.base_block_hash,
    );

    // validate that the proof facts config hash is the true hash of the OS config.
    assert os_output_header.starknet_os_config_hash = virtual_os_config_hash;

    return ();
}
