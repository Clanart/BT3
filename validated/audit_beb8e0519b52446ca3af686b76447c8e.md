### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS updates a contract's class hash in the state without verifying that the new class hash is actually declared in `contract_class_changes`. This is a direct analog to the reported state-transition bypass: a critical state change (contract class replacement) is committed to the proven state without verifying the prerequisite action (class declaration) actually occurred. A malicious contract can replace its class with an arbitrary undeclared hash, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 878–916) processes the `replace_class` syscall. It reads the new `class_hash` from the syscall request and immediately writes it into `contract_state_changes` via `dict_update`, with no check that the new class hash exists in `contract_class_changes` (the declared classes dictionary).

The code itself contains an explicit acknowledgment of this missing check:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `contract_state_changes` dictionary tracks per-contract state (class hash, storage, nonce), while `contract_class_changes` tracks declared class hashes to their compiled class hashes. These two dictionaries are independent. `execute_replace_class` updates only `contract_state_changes` and never cross-references `contract_class_changes` to confirm the target class hash was ever declared. [2](#0-1) 

By contrast, `execute_declare_transaction` in `transaction_impls.cairo` enforces `prev_value=0` and `assert_not_zero(compiled_class_hash)` when writing to `contract_class_changes`, ensuring a class can only be declared once and must have a non-zero compiled hash. No equivalent guard exists in `execute_replace_class`. [3](#0-2) 

---

### Impact Explanation

When a contract's class hash is set to an undeclared value, every subsequent call to that contract will fail at the entry-point dispatch stage because the OS cannot resolve the class. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — there is no recovery path once the proven state is committed to L1. This satisfies the **Critical: Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract from within its own execution context — no privileged role is required. A realistic attack scenario is:

1. Attacker deploys a contract (e.g., a yield vault or token bridge) that accepts user deposits.
2. Users deposit funds.
3. Attacker's contract calls `replace_class` with an arbitrary felt value (e.g., `0x1`) that has never been declared.
4. The OS writes the invalid class hash into `contract_state_changes` without validation.
5. The proof is generated and accepted on L1.
6. The contract is permanently non-functional; all deposited funds are frozen.

The attack requires only the ability to deploy a contract and call a standard syscall — both are available to any unprivileged L2 user.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, verify that it exists in `contract_class_changes` by performing a `dict_read` on `contract_class_changes` with the new class hash as the key and asserting the result is non-zero. This mirrors the validation already enforced during `execute_declare_transaction`.

---

### Proof of Concept

```
1. Attacker deploys ContractA (a vault) — class hash C_valid is declared.
2. Users call ContractA.deposit(), storing funds in ContractA's storage.
3. ContractA internally calls syscall replace_class(class_hash=0xDEAD).
   - execute_replace_class reads class_hash = 0xDEAD from the request.
   - No lookup in contract_class_changes is performed.
   - dict_update writes StateEntry(class_hash=0xDEAD, ...) into contract_state_changes.
4. Block is proven; L1 accepts the state root containing ContractA → class_hash=0xDEAD.
5. Any subsequent invoke targeting ContractA fails: the OS cannot find class 0xDEAD
   in contract_class_changes, so no entry point can be dispatched.
6. All user funds in ContractA's storage are permanently frozen.
``` [4](#0-3)

        // Not enough gas to execute the syscall.
