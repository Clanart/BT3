### Title
Hardcoded `failure_flag=0` in `execute_deploy` Ignores Constructor Revert Result — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

In `execute_deploy` inside `syscall_impls.cairo`, the `ResponseHeader` written back to the calling contract always sets `failure_flag=0` (success), regardless of whether the deployed contract's constructor actually succeeded or reverted. This is the direct Cairo/StarkNet analog of the "ignored return value" bug class from the reference report: a critical sub-operation's result is silently discarded, and the caller is given a false success signal.

---

### Finding Description

In `execute_deploy`, after calling `deploy_contract` to run the constructor, the OS writes the syscall response with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` function accepts a `revert_log` implicit argument (as seen in `execute_deploy_account_transaction`) and internally calls `execute_entry_point`, which fully supports constructor reverts via `handle_revert`. When the constructor reverts, `execute_entry_point` undoes the constructor's storage changes but the contract's class hash remains set in `contract_state_changes`. The return value of `deploy_contract` — which carries the revert status — is bound only to `(retdata_size, retdata)` with no `is_reverted` field surfaced. The `failure_flag` is then unconditionally written as `0`. [2](#0-1) 

The same pattern appears in `execute_deploy_account_transaction` in `transaction_impls.cairo`, where the return value of `deploy_contract` is entirely discarded (no binding at all):

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A calling contract that uses the `deploy` syscall to deploy a vault, wallet, or any fund-holding contract relies on `failure_flag` in the `DeployResponse` to determine whether the constructor succeeded. Because `failure_flag` is always `0`, the calling contract cannot distinguish a successful constructor from a reverted one.

Scenario:
1. Contract A calls `deploy` to deploy Contract B (a vault), whose constructor sets an owner/access-control storage slot.
2. The constructor of B reverts (e.g., out-of-gas, assertion failure, or attacker-induced condition).
3. The OS writes `failure_flag=0` — Contract A sees a successful deploy.
4. Contract A sends funds to Contract B's address (which is deployed but uninitialized — no owner set).
5. The funds are permanently frozen: Contract B exists on-chain with no valid owner or access-control state, and no one can withdraw.

This is directly reachable by any unprivileged transaction sender who deploys a contract using the `deploy` syscall.

---

### Likelihood Explanation

**Medium.** Any contract that:
- Uses the `deploy` syscall to deploy a fund-holding contract, AND
- Relies on `failure_flag` to gate subsequent fund transfers

is vulnerable. This is a standard and expected pattern for factory contracts, wallet deployers, and vault deployers on StarkNet. The constructor revert can be triggered by an attacker who controls the constructor calldata or by natural out-of-gas conditions.

---

### Recommendation

Surface the constructor revert status from `deploy_contract` and write it into the `ResponseHeader.failure_flag` field. The existing `TODO` comment at line 538 acknowledges this gap. The fix requires:

1. `deploy_contract` to return `is_reverted` alongside `retdata_size` and `retdata`.
2. `execute_deploy` to write `ResponseHeader(gas=remaining_gas, failure_flag=is_reverted)`.
3. Analogously fix `execute_deploy_account_transaction` to handle a reverted constructor (currently the return value is fully discarded). [4](#0-3) 

---

### Proof of Concept

1. Deploy a factory contract on StarkNet that:
   - Calls the `deploy` syscall with a class whose constructor unconditionally reverts.
   - Checks `response.failure_flag` — observes `0` (success).
   - Transfers funds to the returned `contract_address`.

2. The OS proof is valid (the OS accepted the block), yet the deployed contract has no initialized state. Funds sent to it are permanently frozen.

The root cause is at:

- `syscall_impls.cairo` line 539: `failure_flag=0` hardcoded.
- `transaction_impls.cairo` lines 642–646: `deploy_contract` return value fully discarded. [5](#0-4) [3](#0-2)
