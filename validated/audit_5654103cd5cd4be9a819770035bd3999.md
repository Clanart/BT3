### Title
`getAccountNonce` Breaks Inner Message Loop Early on Non-EVM Message, Causing `eth_getTransactionCount` to Return Incorrect Pending Nonce - (File: `rpc/backend/utils.go`)

---

### Summary

The `getAccountNonce` function in `rpc/backend/utils.go` uses `break` instead of `continue` when it encounters a non-`MsgEthereumTx` message while iterating over the messages of a pending transaction. This exits the inner message loop prematurely, silently skipping any `MsgEthereumTx` messages that appear after the first non-EVM message in the same Cosmos transaction. As a result, `eth_getTransactionCount` (with `pending=true`) returns a nonce that is too low, feeding incorrect data into transaction execution.

---

### Finding Description

In `rpc/backend/utils.go`, the `getAccountNonce` function iterates over pending transactions and their messages to compute the pending nonce for an Ethereum address:

```go
for _, tx := range pendingTxs {
    for _, msg := range (*tx).GetMsgs() {
        ethMsg, ok := msg.(*evmtypes.MsgEthereumTx)
        if !ok {
            // not ethereum tx
            break          // ← BUG: exits inner loop, not outer
        }

        sender, err := ethMsg.GetSenderLegacy(ethtypes.LatestSignerForChainID(b.chainID))
        if err != nil {
            continue
        }
        if sender == accAddr {
            nonce++
        }
    }
}
``` [1](#0-0) 

When a Cosmos SDK transaction in the mempool contains messages in the order `[non-EVM msg, MsgEthereumTx, ...]`, the `break` on line 106 exits the inner `for _, msg := range` loop entirely. All subsequent `MsgEthereumTx` messages in that same transaction are never examined. The outer loop then moves on to the next pending transaction, having silently skipped valid EVM messages.

The correct fix is `continue` (skip to the next message) rather than `break` (exit the message loop for this transaction). This is the direct analog of the M-16 `break`/`continue` bug: a loop exits early when it should skip only the current element and proceed to the next.

---

### Impact Explanation

`getAccountNonce` is the backend for `eth_getTransactionCount` when called with `pending` block tag. Wallets and dApps rely on this RPC call to determine the next valid nonce before submitting a transaction.

If the returned nonce is too low (because pending EVM messages were skipped), a user will be told to use a nonce that is already occupied by a pending transaction. Submitting with that nonce will either:
- Cause the new transaction to be rejected (nonce already used / nonce too low), or
- Unexpectedly replace the pending transaction (if the new tx has a higher gas price), causing the original pending tx to be dropped without the user's knowledge.

This matches the allowed High impact: **"Public JSON-RPC... path feeds incorrect consensus-critical data into transaction execution."** [2](#0-1) 

---

### Likelihood Explanation

Any Cosmos SDK transaction in the mempool whose message list begins with a non-EVM message (e.g., `MsgSend`, `MsgDelegate`, `MsgVote`) followed by one or more `MsgEthereumTx` messages triggers the bug. While Ethermint's ante handler typically enforces that a tx is either purely EVM or purely Cosmos, the mempool can contain transactions from other modules or from chains that allow mixed-message batches. An attacker can also deliberately submit such a transaction to grief a target user's nonce accounting. The entry path is an unprivileged JSON-RPC call (`eth_getTransactionCount` with `pending`) combined with a crafted or naturally occurring mixed-message pending transaction.

---

### Recommendation

Replace `break` with `continue` on the non-EVM message check:

```diff
- if !ok {
-     // not ethereum tx
-     break
- }
+ if !ok {
+     // not ethereum tx, skip to next message
+     continue
+ }
``` [3](#0-2) 

---

### Proof of Concept

1. A pending Cosmos transaction in the mempool contains messages: `[MsgSend, MsgEthereumTx{from: victim}]`.
2. A client calls `eth_getTransactionCount(victim, "pending")`.
3. `getAccountNonce` iterates over pending txs. For the above tx, it processes `MsgSend` first — `ok` is `false`, so `break` fires and exits the inner loop.
4. `MsgEthereumTx{from: victim}` is never reached; `nonce` is not incremented for it.
5. The RPC returns `nonce = N` instead of the correct `nonce = N+1`.
6. The victim's wallet uses nonce `N` for a new transaction, which conflicts with the already-pending EVM tx at nonce `N`, causing the new tx to fail or unexpectedly replace the pending one.

func (b *Backend) getAccountNonce(accAddr common.Address, pending bool, height int64, logger log.Logger) (uint64, error) {
	queryClient := authtypes.NewQueryClient(b.clientCtx)
	adr := sdk.AccAddress(accAddr.Bytes()).String()
	ctx := types.ContextWithHeight(height)
	res, err := queryClient.Account(ctx, &authtypes.QueryAccountRequest{Address: adr})
	if err != nil {
		st, ok := status.FromError(err)
		// treat as account doesn't exist yet
		if ok && st.Code() == codes.NotFound {
			return 0, nil
		}
		return 0, err
	}
	var acc sdk.AccountI
	if err := b.clientCtx.InterfaceRegistry.UnpackAny(res.Account, &acc); err != nil {
		return 0, err
	}

	nonce := acc.GetSequence()

	if !pending {
		return nonce, nil
	}

	// the account retriever doesn't include the uncommitted transactions on the nonce so we need to
	// to manually add them.
	pendingTxs, err := b.PendingTransactions()
	if err != nil {
		logger.Error("failed to fetch pending transactions", "error", err.Error())
		return nonce, nil
	}

	// add the uncommitted txs to the nonce counter
	// only supports `MsgEthereumTx` style tx
	for _, tx := range pendingTxs {
		for _, msg := range (*tx).GetMsgs() {
			ethMsg, ok := msg.(*evmtypes.MsgEthereumTx)
			if !ok {
				// not ethereum tx
				break
			}

			sender, err := ethMsg.GetSenderLegacy(ethtypes.LatestSignerForChainID(b.chainID))
			if err != nil {
				continue
			}
			if sender == accAddr {
				nonce++
			}
		}
	}

	return nonce, nil
```
