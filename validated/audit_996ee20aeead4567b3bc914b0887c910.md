### Title
Uint64 Underflow in CREATE Nonce Reconciliation Allows Nonce Replay After Reverted Deployment - (`x/evm/keeper/state_transition.go`)

### Summary

The nonce reconciliation formula at line 479 of `state_transition.go` contains a uint64 underflow when `evm.Create()` reverts. Because go-ethereum reverts the caller's nonce increment as part of snapshot rollback, `afterCreateNonce` equals `msg.Nonce` on revert, making `nestedCreates = afterCreateNonce - msg.Nonce - 1` wrap to `2^64 - 1`. The subsequent `SetNonce(sender, oldNonce + nestedCreates)` then wraps back to `msg.Nonce`, setting the sender's nonce to the **pre-transaction value** instead of the post-ante-handler value `msg.Nonce + 1`. The stateDB is committed unconditionally even on `vmErr`, so this incorrect nonce is persisted, allowing the sender to replay the same nonce.

---

### Finding Description

The relevant block in `ApplyMessageWithConfig`: [1](#0-0) 

The logic assumes `evm.Create()` **always** leaves `afterCreateNonce = msg.Nonce + 1 + nestedCreates`. The comment at line 476 makes this explicit: [2](#0-1) 

However, in go-ethereum v1.16.9 (the version used here): [3](#0-2) 

…the `create()` function in `evm.go` increments the **caller's** nonce **after** taking the snapshot, inside the snapshot scope. When the constructor reverts (out-of-gas or explicit revert), `RevertToSnapshot` undoes that increment. Therefore `afterCreateNonce == msg.Nonce` on any reverted CREATE.

**Arithmetic trace** (all uint64):

| Variable | Value |
|---|---|
| `oldNonce` (ante-handler pre-incremented) | `N + 1` |
| `stateDB.SetNonce(sender, msg.Nonce)` | resets to `N` |
| `evm.Create()` reverts → `afterCreateNonce` | `N` |
| `nestedCreates = N - N - 1` | `2^64 - 1` (underflow) |
| `SetNonce(sender, (N+1) + (2^64-1))` | `N + 2^64` → wraps to `N` |

The stateDB is committed unconditionally regardless of `vmErr`: [4](#0-3) 

So the committed nonce is `N` instead of `N + 1`.

---

### Impact Explanation

After a reverted CREATE, the sender's on-chain nonce is `N` (the pre-transaction value) rather than `N + 1`. This means:

1. **Nonce replay**: The sender can immediately submit another transaction with nonce `N`. The ante handler's sequence check will accept it because the stored sequence is `N`.
2. **Repeated deployment attempts at the same address**: Since the contract address is `keccak256(rlp([sender, N]))`, the sender can retry deploying to the same address with different bytecode or more gas, violating the uniqueness invariant.
3. **Nonce desynchronization**: Any pre-signed transactions with nonces `N+1, N+2, …` become invalid until the sender re-signs them with shifted nonces.

The question's claim of "astronomically large nonce permanently blocking the account" is **incorrect** — the uint64 double-wrap resolves to `msg.Nonce`, not `2^64 - 1`. The actual impact is nonce replay, not account blocking.

---

### Likelihood Explanation

- Triggerable by any unprivileged sender via a standard `MsgEthereumTx` CREATE with a constructor that reverts (e.g., `revert()` or insufficient gas).
- No special privileges, governance, or validator cooperation required.
- The ante handler path (`CheckAndSetEthSenderNonce`) pre-increments the nonce unconditionally for contract creation: [5](#0-4) [6](#0-5) 

- The stateDB commit at line 602–603 is unconditional on `vmErr`, so the corrupted nonce is always persisted.

---

### Recommendation

Guard the reconciliation against the revert case. The fix is to check whether `afterCreateNonce < msg.Nonce + 1` (i.e., the nonce was reverted) and handle it explicitly:

```go
afterCreateNonce := stateDB.GetNonce(sender)
// Guard: evm.Create() may revert and undo its own nonce increment.
// In that case afterCreateNonce == msg.Nonce; treat nestedCreates as 0.
var nestedCreates uint64
if afterCreateNonce > msg.Nonce {
    nestedCreates = afterCreateNonce - msg.Nonce - 1
}
stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
```

This ensures that on a reverted CREATE, `nestedCreates = 0` and the final nonce is `oldNonce` (= `msg.Nonce + 1`), matching the expected post-transaction value.

---

### Proof of Concept

1. Fund account `A` with nonce `N = 0`.
2. Submit a `MsgEthereumTx` CREATE with bytecode that immediately reverts (`PUSH1 0x00 REVERT`).
3. Observe that after block inclusion, `eth_getTransactionCount(A, "latest")` returns `0` (not `1`).
4. Submit a second transaction with nonce `0` — it is accepted by the ante handler and included in the next block.
5. The sender has effectively replayed nonce `0`, violating the nonce-uniqueness invariant.
