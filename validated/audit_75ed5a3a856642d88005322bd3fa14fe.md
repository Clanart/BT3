### Title
Incorrect Internal Balance Bookkeeping for Deflationary/Fee-on-Transfer Tokens in `init_transfer` - (File: starknet/src/omni_bridge.cairo)

---

### Summary

The StarkNet `OmniBridge.init_transfer` function accepts an arbitrary ERC20 token via `transfer_from` and emits a cross-chain `InitTransfer` event using the caller-supplied `amount` parameter — not the actual amount received by the contract. For deflationary tokens or tokens with a transfer fee, the contract receives fewer tokens than `amount`, yet the destination chain mints or releases the full `amount`, creating unbacked bridged supply.

---

### Finding Description

In `starknet/src/omni_bridge.cairo`, the `init_transfer` function handles non-bridge (native) ERC20 tokens as follows:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
```

After the transfer, the function emits:

```cairo
self.emit(Event::InitTransfer(InitTransfer {
    ...
    amount,       // <-- caller-supplied input, NOT actual received amount
    fee,
    ...
}))
``` [1](#0-0) [2](#0-1) 

There is no balance-before / balance-after check. The `amount` field in the emitted event is the raw user input, not the actual token delta credited to the bridge vault. The destination chain (NEAR, EVM, Solana) reads this event and mints or releases exactly `amount` tokens to the recipient.

The same structural pattern exists in the EVM `OmniBridge.sol`, where `safeTransferFrom` is called with the user-supplied `amount` and that same `amount` is emitted in `BridgeTypes.InitTransfer` without verifying the actual balance change. [3](#0-2) 

---

### Impact Explanation

For any ERC20 token with a transfer fee or deflationary mechanism (e.g., a 1% fee-on-transfer token):

- User calls `init_transfer` with `amount = 1000`.
- Bridge vault receives only `990` tokens (after 1% fee).
- The emitted event records `amount = 1000`.
- The destination chain mints or releases `1000` tokens to the recipient.
- The bridge is now undercollateralized by `10` tokens per transfer.

Repeated over many transfers, the total bridged supply on the destination chain exceeds the actual locked collateral on StarkNet, breaking bridge collateralization. An attacker can deliberately use a fee-on-transfer token they control to extract unbacked value from the destination chain.

**Impact class:** High — Balance/accounting corruption that breaks bridge collateralization, enabling unauthorized mint of unbacked bridged assets.

---

### Likelihood Explanation

- The entry point is fully unprivileged: any user can call `init_transfer` with any `token_address`.
- The bridge does not whitelist tokens or assert standard ERC20 behavior.
- Fee-on-transfer tokens (e.g., STA, PAXG, tokens with configurable fees) are widely deployed on StarkNet-compatible chains.
- No special access or key compromise is required; a single transaction is sufficient to trigger the discrepancy. [4](#0-3) 

---

### Recommendation

Replace the fixed-`amount` accounting with a balance-delta check:

```cairo
let balance_before = IERC20Dispatcher { contract_address: token_address }
    .balance_of(get_contract_address());

let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');

let balance_after = IERC20Dispatcher { contract_address: token_address }
    .balance_of(get_contract_address());

let actual_amount = balance_after - balance_before;
assert(actual_amount == amount.into(), 'ERR_FEE_ON_TRANSFER_TOKEN');
```

Either reject fee-on-transfer tokens outright (simplest), or use `actual_amount` in the emitted event and downstream accounting. Apply the same fix to the EVM `OmniBridge.sol` ERC20 deposit path.

---

### Proof of Concept

1. Deploy a StarkNet ERC20 token `FeeToken` that deducts a 10% fee on every `transfer_from`, crediting the fee to a treasury address.
2. Approve the `OmniBridge` contract to spend `1000 FeeToken` from attacker's wallet.
3. Call `OmniBridge.init_transfer(FeeToken, 1000, 0, 0, "near:attacker.near", "")`.
4. `transfer_from` executes: bridge vault receives `900`, treasury receives `100`.
5. `InitTransfer` event is emitted with `amount = 1000`.
6. NEAR bridge relayer picks up the event and calls `fin_transfer`, minting `1000` bridged `FeeToken` to `attacker.near`.
7. Attacker holds `1000` bridged tokens backed by only `900` real tokens — a `100`-token unbacked surplus per iteration.
8. Repeating this drains the bridge's collateral ratio until other users cannot redeem their bridged tokens. [5](#0-4) [6](#0-5)
