### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Substitution - (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` containing a chunk for shard A while the caller-supplied `shard_id` parameter is shard B. The Merkle proof check (`verify_path`) only confirms the chunk hash exists *somewhere* in `chunk_headers_root`; it does not verify the chunk occupies the leaf position that corresponds to `shard_id`. The header is then stored under `StateHeaderKey(shard_B, sync_hash)` carrying shard A's `prev_state_root`. Subsequent `set_state_part` / `apply_state_part` calls for shard B consume that wrong root and install shard A's trie data into shard B's `shard_uid`.

---

### Finding Description

**Production entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`, reached from `ClientActor::handle(StateHeaderValidationRequest)` after a peer delivers a `ShardStateSyncResponseHeader`.

The function performs five numbered checks before writing to `DBCol::StateHeaders`:

1. `validate_chunk_proofs` — validates the chunk's internal Reed-Solomon / encoded-merkle-root consistency. Does **not** compare `chunk.shard_id()` to the caller's `shard_id`.
2. `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — proves the chunk hash is a leaf in the block's `chunk_headers_root` Merkle tree.
3. `verify_path` for `prev_chunk_header` — same style of proof for the previous chunk.
4. Receipt-proof chain — verifies incoming receipts using the caller-supplied `shard_id` to hash `ReceiptList(shard_id, receipts)`.
5. `validate_state_root_node` — validates the `state_root_node` against `chunk_inner.prev_state_root()` (shard A's root).

**The gap**: `verify_path` is a plain Merkle-path reconstruction:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
```

It does **not** verify the leaf index. Compare with `verify_path_with_index`, which calls `verify_path_matches_index` first — that function is used for chunk-part proofs but is absent here.

The `chunk_headers_root` is built by merklizing `ChunkHashHeight` values ordered by shard index:

```rust
let (chunk_headers_root, chunk_proofs) = merklize(
    &sync_prev_block.chunks().iter()
        .map(|c| ChunkHashHeight(c.chunk_hash().clone(), c.height_included()))
        .collect::<Vec<_>>(),
);
```

A valid Merkle path for shard A's chunk (at leaf index A) will reconstruct the same `chunk_headers_root` regardless of what `shard_id` the caller passes. The function never asserts `chunk.shard_id() == shard_id`.

After all checks pass, the header is persisted:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

`shard_id` here is the attacker-controlled parameter (shard B), while `shard_state_header` contains shard A's chunk and shard A's `prev_state_root`.

**Downstream propagation**:

`set_state_part` reads the stored header by `StateHeaderKey(shard_B, sync_hash)`, extracts `prev_state_root` from the chunk inner (shard A's root), and validates/stores state parts against that root:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// validates part against shard A's state_root, stores under StatePartKey(sync_hash, shard_B, ...)
```

`set_state_finalize` then applies those parts to shard B's `shard_uid`, installing shard A's trie into shard B.

---

### Impact Explanation

A syncing node that accepts state from a malicious peer ends up with shard B's `shard_uid` populated with shard A's trie data. The node's `chunk_extra` for shard B will reference shard A's `state_root`, causing every subsequent chunk application for shard B to read the wrong state. This is a silent, persistent state corruption: the node will produce incorrect execution results for shard B and diverge from the canonical chain without any immediate crash or error.

---

### Likelihood Explanation

State sync peers are not required to be validators. Any peer that advertises possession of state data can serve as a provider. The attack requires:

1. Acting as (or MITM-ing) a state-sync provider for the victim node.
2. Constructing a `ShardStateSyncResponseHeader` with shard A's chunk, shard A's `chunk_proof` (publicly derivable from the block), valid receipt proofs for shard B's incoming receipts (all on-chain public data), and shard A's `state_root_node`.
3. Calling `set_state_header` with `shard_id = B`.

All required data is publicly available. No validator key or privileged access is needed. The attack is most straightforward when shard A and shard B have the same `chunk.height_included()` (the common case when no chunks are missing), so the receipt-proof range check at step 4g also passes.

---

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: reject cross-shard substitution
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, consider using `verify_path_with_index` (which calls `verify_path_matches_index`) instead of bare `verify_path` for the chunk-in-block proof, so the leaf position is cryptographically bound to the shard index.
