### Title
Permanent Fund Lock via USDC-Blacklisted Recipient in `finTransfer` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
When a user bridges USDC (or any token with a blacklist) from NEAR to an EVM chain, and the designated recipient address is or becomes USDC-blacklisted, the `finTransfer` call on the EVM side will always revert. Because the nonce is set and then reverted atomically with the failed transfer, the destination nonce is never consumed. The MPC-signed payload is bound to the blacklisted recipient and cannot be redirected. With no cancel or refund path on the NEAR side, the user's funds are permanently locked.

---

### Finding Description

In `OmniBridge.sol`, `finTransfer` marks the nonce as used and then performs the token transfer to the recipient in the same transaction:

```solidity
// Line 287 — set before transfer
completedTransfers[payload.destinationNonce] = true;

// ... signature verification ...

// Line 351-354 — transfer to recipient
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [1](#0-0) 

If `safeTransfer` reverts (e.g., USDC blacklist), the entire transaction reverts, including the `completedTransfers[payload.destinationNonce] = true` write. The nonce is therefore never consumed. Every subsequent attempt to finalize the same transfer will also revert for the same reason.

The MPC signature produced by `sign_transfer` on NEAR encodes the recipient address directly in the signed `TransferMessagePayload`. There is no mechanism to re-sign with a different recipient. [2](#0-1) 

On the NEAR side, when `fee.is_zero()`, the transfer is removed from `pending_transfers` immediately after signing:

```rust
if fee.is_zero() {
    self.remove_transfer_message(message_payload.transfer_id);
}
``` [3](#0-2) 

The user's tokens are already burned (for bridge tokens) or locked (for native tokens) at `init_transfer` time. When fee is non-zero, the transfer remains in `pending_transfers` indefinitely because `claim_fee` requires a proof of EVM finalization that can never be produced. Either way, the funds are irrecoverable. [4](#0-3) 

---

### Impact Explanation

A user's bridged USDC (or any blacklist-enforcing token) is permanently frozen. The NEAR-side funds are burned or locked at `initTransfer` time. The EVM-side `finTransfer` can never succeed for a blacklisted recipient. No cancel, redirect, or refund function exists in the NEAR contract to recover the locked funds. This satisfies the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

USDC blacklisting is an active, real-world operational mechanism used by Circle. A user's address can be blacklisted after a transfer is initiated but before it is finalized (the cross-chain finalization window can be minutes to hours). Additionally, a user could unknowingly specify a recipient address that is already blacklisted. The bridge handles USDC as a first-class asset (it is explicitly handled in the `else` branch of `finTransfer` via `safeTransfer`), making this a realistic scenario.

---

### Recommendation

1. **Pull-based withdrawal pattern**: Instead of pushing tokens directly to `payload.recipient` in `finTransfer`, credit the amount to an internal claimable balance mapping. The recipient then calls a separate `claim()` function to pull their tokens. A failed claim does not affect other transfers.

2. **Try/catch with escrow fallback**: Wrap the `safeTransfer` in a try/catch. On failure, escrow the funds in the contract under the recipient's address and emit an event, allowing the recipient to claim later or an admin to redirect after governance approval.

3. **Recipient override mechanism**: Allow the original sender (proven via MPC signature) to designate an alternative recipient if the primary recipient is provably unable to receive the token.

---

### Proof of Concept

1. Alice holds USDC on NEAR and calls `ft_on_transfer` with an `InitTransferMsg` specifying her EVM address as recipient. Her USDC is burned on NEAR and a `TransferMessage` is stored in `pending_transfers`.
2. The relayer calls `sign_transfer` on NEAR; the MPC signs a `TransferMessagePayload` binding Alice's EVM address as `recipient`.
3. Between step 1 and step 3, Circle blacklists Alice's EVM address (e.g., due to a compliance action).
4. The relayer calls `finTransfer` on the EVM `OmniBridge` with the signed payload.
5. Execution reaches `IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount)` at line 351. USDC's `transfer` reverts because `payload.recipient` is blacklisted.
6. The entire transaction reverts. `completedTransfers[payload.destinationNonce]` is rolled back to `false`.
7. Every retry of step 4 produces the same revert. The nonce is never consumed.
8. Alice's USDC on NEAR is permanently burned. No cancel or refund function exists. Funds are irrecoverably lost. [5](#0-4)

    function finTransfer(
        bytes calldata signatureData,
        BridgeTypes.TransferMessagePayload calldata payload
    ) external payable whenNotPaused(PAUSED_FIN_TRANSFER) {
        if (completedTransfers[payload.destinationNonce]) {
            revert NonceAlreadyUsed(payload.destinationNonce);
        }

        completedTransfers[payload.destinationNonce] = true;

        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
            Borsh.encodeUint64(payload.destinationNonce),
            bytes1(payload.originChain),
            Borsh.encodeUint64(payload.originNonce),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.tokenAddress),
            Borsh.encodeUint128(payload.amount),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.recipient),
            bytes(payload.feeRecipient).length == 0 // None or Some(String) in rust
                ? bytes("\x00")
                : bytes.concat(
                    bytes("\x01"),
                    Borsh.encodeString(payload.feeRecipient)
                ),
            bytes(payload.message).length == 0
                ? bytes("")
                : Borsh.encodeBytes(payload.message)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        MultiTokenInfo memory multiToken = multiTokens[payload.tokenAddress];

        if (payload.tokenAddress == address(0)) {
            // slither-disable-next-line arbitrary-send-eth
            (bool success, ) = payload.recipient.call{value: payload.amount}(
                ""
            );
            if (!success) revert FailedToSendEther();
        } else if (multiToken.tokenAddress != address(0)) {
            IERC1155(multiToken.tokenAddress).safeTransferFrom(
                address(this),
                payload.recipient,
                multiToken.tokenId,
                payload.amount,
                ""
            );
        } else if (customMinters[payload.tokenAddress] != address(0)) {
            ICustomMinter(customMinters[payload.tokenAddress]).mint(
                payload.tokenAddress,
                payload.recipient,
                payload.amount
            );
        } else if (isBridgeToken[payload.tokenAddress]) {
            if (payload.message.length == 0) {
                IBridgeToken(payload.tokenAddress).mint(
                    payload.recipient,
                    payload.amount
                );
            } else {
                IBridgeToken(payload.tokenAddress).mint(
                    payload.recipient,
                    payload.amount,
                    payload.message
                );
            }
        } else {
            IERC20(payload.tokenAddress).safeTransfer(
                payload.recipient,
                payload.amount
            );
        }

        finTransferExtension(payload);

        emit BridgeTypes.FinTransfer(
            payload.originChain,
            payload.originNonce,
            payload.tokenAddress,
            payload.amount,
            payload.recipient,
            payload.feeRecipient
        );
    }
