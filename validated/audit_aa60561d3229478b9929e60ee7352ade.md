### Title
Missing Expiry on MPC-Signed `MetadataPayload` Enables Stale Token Deployment with Wrong Decimals — (`evm/src/omni-bridge/contracts/BridgeTypes.sol`, `starknet/src/bridge_types.cairo`, `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The `MetadataPayload` struct signed by the MPC signer during `log_metadata` contains no expiry timestamp or deadline. Any signed metadata payload remains permanently valid and can be submitted to `deployToken` on EVM, StarkNet, or Solana at any future point. If a NEAR token's decimals change between two `log_metadata` calls, an attacker can submit the older signed payload first, deploying the bridged token with stale (wrong) decimals. This corrupts the bridge's decimal normalization accounting for all subsequent transfers of that token.

---

### Finding Description

The `MetadataPayload` struct used across all destination chains contains only static token metadata fields — no `valid_until`, `deadline`, or timestamp field:

**EVM (`BridgeTypes.sol`):**
```solidity
struct MetadataPayload {
    string token;
    string name;
    string symbol;
    uint8 decimals;   // no expiry
}
```

**StarkNet (`bridge_types.cairo`):**
```cairo
pub struct MetadataPayload {
    pub token: ByteArray,
    pub name: ByteArray,
    pub symbol: ByteArray,
    pub decimals: u8,   // no expiry
}
```

The `deployToken` function on EVM verifies the MPC signature over this payload and deploys the bridged token:

```solidity
function deployToken(
    bytes calldata signatureData,
    BridgeTypes.MetadataPayload calldata metadata
) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
    bytes memory borshEncoded = bytes.concat(
        bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
        Borsh.encodeString(metadata.token),
        Borsh.encodeString(metadata.name),
        Borsh.encodeString(metadata.symbol),
        bytes1(metadata.decimals)
    );
    bytes32 hashed = keccak256(borshEncoded);
    if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
        revert InvalidSignature();
    }
    // ... deploys token with metadata.decimals
```

The only guard against re-deployment is `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")`. There is no check that the signed payload is recent or that it matches the current on-chain state of the NEAR token.

The same pattern exists in StarkNet's `deploy_token` and Solana's `deploy_token` instruction.

---

### Impact Explanation

If a NEAR token's decimals change (e.g., from 18 to 6 via a token upgrade), the following sequence breaks bridge accounting:

1. `log_metadata` is called for token A (decimals=18) → MPC signs `MetadataPayload{token="A", decimals=18}`. The signed payload is emitted as a public event.
2. Token A's decimals are changed to 6 on NEAR.
3. `log_metadata` is called again → MPC signs `MetadataPayload{token="A", decimals=6}`.
4. An attacker submits the **old** signed payload (decimals=18) to `deployToken` on EVM before the correct one.
5. The EVM bridged token is deployed with `decimals=18` instead of 6.
6. The correct payload (decimals=6) now fails with `ERR_TOKEN_EXIST`.

The NEAR bridge's `token_decimals` mapping stores the correct decimals (6) from the proof, but the EVM token contract has 18 decimals. All subsequent cross-chain transfers of this token will be normalized using the wrong decimal factor (off by 10^12), causing users to receive drastically wrong amounts. This constitutes decimal/accounting corruption that breaks bridge collateralization.

---

### Likelihood Explanation

- Signed `MetadataPayload` events are permanently stored on-chain and publicly accessible to any observer.
- Token decimal changes, while uncommon, do occur during token upgrades or migrations.
- The attack requires no special privileges — any unprivileged user can call `deployToken` with a valid (but stale) signature.
- The window of vulnerability is permanent: once a signed payload exists, it never expires.

---

### Recommendation

Add a `valid_until` (Unix timestamp) field to `MetadataPayload` and enforce it in `deployToken`:

```solidity
struct MetadataPayload {
    string token;
    string name;
    string symbol;
    uint8 decimals;
    uint64 validUntil;  // ADD: expiry timestamp
}
```

```solidity
function deployToken(...) external payable {
    require(block.timestamp <= metadata.validUntil, "ERR_SIGNATURE_EXPIRED");
    // ... existing logic
}
```

Apply the same fix to `MetadataPayload` in `starknet/src/bridge_types.cairo` and `DeployTokenPayload` in `solana/programs/bridge_token_factory/src`.

---

### Proof of Concept

1. Token `foo.near` exists with `decimals=18`. `log_metadata("foo.near")` is called → MPC signs `MetadataPayload{token="foo.near", decimals=18}`. The `SignTransferEvent` (or equivalent) is emitted and recorded on-chain.
2. `foo.near` is upgraded; its decimals change to 6.
3. `log_metadata("foo.near")` is called again → MPC signs `MetadataPayload{token="foo.near", decimals=6}`.
4. Attacker reads the old signed payload from step 1 (public blockchain history) and calls `OmniBridge.deployToken(oldSignature, {token:"foo.near", decimals:18})`.
5. EVM token is deployed with `decimals=18`. The `isBridgeToken` and `nearToEthToken` mappings are set.
6. The correct call `deployToken(newSignature, {token:"foo.near", decimals:6})` reverts with `ERR_TOKEN_EXIST`.
7. All bridge transfers for `foo.near` now use the wrong decimal factor, misdirecting value for every user who bridges this token.
