### Title
Recipient Contract Can Permanently Freeze Sender's Bridged ERC1155 (or Native ETH) Funds by Reverting the Delivery Callback — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` delivers ERC1155 tokens via `IERC1155.safeTransferFrom`, which invokes `onERC1155Received` on the recipient contract. If the recipient contract reverts that callback, the entire `finTransfer` transaction reverts. Because the recipient address is cryptographically bound inside the MPC-signed payload and cannot be substituted, and because the NEAR bridge has no public cancel/refund path for the originating sender, the sender's tokens are permanently locked in the NEAR bridge. The same root cause applies to the native-ETH delivery path, where a reverting `receive()` function triggers `FailedToSendEther` and the same permanent lock.

---

### Finding Description

`finTransfer` in `OmniBridge.sol` marks the destination nonce as used, verifies the MPC signature, then dispatches the asset to `payload.recipient`. For ERC1155 tokens the dispatch is:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
```

`safeTransferFrom` calls `onERC1155Received` on `payload.recipient` when it is a contract. If that hook reverts, the whole transaction reverts — including the `completedTransfers[payload.destinationNonce] = true` write at line 287, so the nonce is not permanently consumed.

However, `payload.recipient` is part of the Borsh-encoded message that was signed by the MPC key:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
```

No alternative recipient can be substituted without invalidating the signature. There is therefore no way to re-route the delivery.

On the NEAR side, `init_transfer_internal` locks or burns the sender's tokens at the moment of initiation:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

A search of the entire NEAR bridge source (`near/omni-bridge/src/*.rs`) finds no public `cancel_transfer`, `refund_transfer`, or equivalent function that would allow the sender to reclaim tokens from a pending NEAR→EVM transfer whose EVM-side finalization is permanently blocked. The internal `remove_transfer_message` helper is only invoked by the bridge's own callbacks, never by an unprivileged user.

The identical structural problem exists for the native-ETH path:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
```

A recipient contract whose `receive()` or fallback reverts will cause `finTransfer` to revert on every attempt, with the same permanent-lock outcome.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds in the bridge.**

Once `init_transfer` is confirmed on NEAR the sender's tokens are gone from their account. If the EVM-side `finTransfer` can never succeed — because the recipient contract always reverts the ERC1155 callback or the ETH receive hook — those tokens remain locked in the NEAR bridge forever with no protocol-level escape hatch for the sender.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** The ERC1155 bridge path is a live, production feature (`initTransfer1155` / `logMetadata1155` / `multiTokens` mapping). Realistic trigger conditions:

1. **Recipient contract upgrade**: A DAO, multisig, or DeFi vault that initially accepted ERC1155 tokens is upgraded or paused after the NEAR-side transfer is initiated, causing `onERC1155Received` to revert.
2. **Deliberate griefing**: A recipient who wants to prevent the sender from completing the transfer (e.g., to keep the sender's capital locked) deploys or upgrades a contract to revert the callback.
3. **Accidental revert**: A recipient contract with a bug in `onERC1155Received` that triggers under certain conditions.

The native-ETH variant is equally reachable for any contract recipient whose `receive()` reverts.

---

### Recommendation

1. **Pull-payment pattern for ERC1155 and native ETH**: Instead of pushing assets to the recipient inside `finTransfer`, store them in a claimable mapping and let the recipient pull them in a separate transaction. This decouples delivery failure from finalization.

2. **Add a sender-accessible cancel/refund path on NEAR**: Introduce a time-locked `cancel_transfer` function that allows the original sender to reclaim locked/burned tokens after a configurable timeout if the destination-chain finalization has not been confirmed.

3. **Use `transferFrom` instead of `safeTransferFrom` for ERC1155**: Removes the `onERC1155Received` callback entirely, though this trades one risk for another (recipient may not be aware of the receipt).

---

### Proof of Concept

**Setup**: ERC1155 token `T` is registered in the bridge via `logMetadata1155`; `multiTokens[deterministicAddr]` is populated.

**Step 1 — Initiation on NEAR**:
Alice calls `ft_transfer_call` → `init_transfer` on the NEAR bridge specifying Bob's EVM contract address as recipient. The NEAR bridge locks Alice's tokens via `burn_tokens_if_needed` / `lock_tokens_if_needed`. [1](#0-0) 

**Step 2 — Bob upgrades his contract**:
Bob upgrades his EVM contract so that `onERC1155Received` always reverts (e.g., `revert("not accepting")`).

**Step 3 — Relayer attempts `finTransfer` on EVM**:
```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,   // Bob's contract
    multiToken.tokenId,
    payload.amount,
    ""
);
// → onERC1155Received reverts → entire tx reverts
``` [2](#0-1) 

**Step 4 — Nonce not consumed, but recipient is fixed**:
`completedTransfers[nonce]` reverts with the transaction. However, the MPC signature covers `payload.recipient`: [3](#0-2) 

No alternative recipient can be used without a new MPC signature, and the MPC will not re-sign for a different recipient for the same transfer.

**Step 5 — No recovery on NEAR**:
There is no public `cancel_transfer` or sender-accessible refund function in the NEAR bridge. Alice's tokens remain permanently locked. [4](#0-3) 

**Result**: Alice's ERC1155 tokens are irrecoverably locked in the NEAR bridge. Bob paid nothing and lost nothing; Alice loses her entire bridged amount.

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
