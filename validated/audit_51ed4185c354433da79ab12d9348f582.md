### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Corruption — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from an untrusted peer and stores it under `StateHeaderKey(shard_id, sync_hash)` in `DBCol::StateHeaders` without ever asserting that the embedded chunk belongs to `shard_id`. An attacker-controlled peer can supply a header whose `chunk` field belongs to shard X while the call is for shard Y, causing shard X's `prev_state_root` to be persisted as shard Y's state root. All subsequent `set_state_part` and `set_state_finalize` calls then operate on the wrong trie root for shard Y.

### Finding Description

`set_state_header` performs five validation steps:

1. **`validate_chunk_proofs`** — verifies the chunk's internal hash/tx/receipt consistency. It never reads `chunk.shard_id()` against the `shard_id` parameter. [1](#0-0) 

2. **`verify_path` for `chunk`** — verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in `sync_prev_block_header.chunk_headers_root()`. The merkle path encodes the leaf *position* (shard index), but the code never checks that this position corresponds to `shard_id`. A proof for shard 0's chunk at position 0 passes even when `shard_id=1` is requested. [2](#0-1) 

3. **`verify_path` for `prev_chunk_header`** — same structural gap; no shard-id binding. [3](#0-2) 

4. **Receipt proof loop** — hashes receipts as `ReceiptList(shard_id, receipts)` using the *parameter* `shard_id`, so an attacker can supply valid on-chain receipts for shard Y (the target) while the `chunk` field belongs to shard X. These are independent fields in `ShardStateSyncResponseHeaderV2`. [4](#0-3) 

5. **`validate_state_root_node`** — validates the node against `chunk_inner.prev_state_root()`, which is shard X's root. This passes trivially. [5](#0-4) 

After all checks pass, the header is stored under `StateHeaderKey(shard_id=Y, sync_hash)`: [6](#0-5) 

The concrete attack payload is:
- `chunk`: shard X's chunk (valid `validate_chunk_proofs`, valid merkle proof at position X)
- `prev_chunk_header`: shard Y's prev chunk header (valid merkle proof at position Y — bypasses the receipt-range height check at step 4g)
- `incoming_receipts_proofs`: valid on-chain receipts for shard Y
- `state_root_node`: shard X's state root node

No check in the function ever asserts `chunk.shard_id() == shard_id`.

### Impact Explanation

After the corrupted header is stored, `set_state_part` reads it back and validates parts against shard X's `prev_state_root` for shard Y's `StatePartKey`: [7](#0-6) 

`set_state_finalize` then applies shard X's chunk body under shard Y's `ShardUId`, writing shard X's trie nodes into shard Y's persisted state and flat storage: [8](#0-7) 

The exact corrupted value is: shard X's `prev_state_root` installed as shard Y's state root in `DBCol::StateHeaders` under `StateHeaderKey(Y, sync_hash)`, propagating into the trie and flat storage for shard Y.

### Likelihood Explanation

The call path is: peer network → `StateSyncDownloader::ensure_shard_header` → `StateHeaderValidationRequest` → `ClientActor::handle` → `set_state_header`. Any peer can serve state sync data; no validator or operator privilege is required. [9](#0-8) [10](#0-9) 

The attack is straightforward in normal network operation where all shards produce chunks at every height (the common case), making the height-range constraint trivially satisfiable.

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD THIS:
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        format!("set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
                chunk.shard_id(), shard_id).into()
    ));
}
```

A parallel check should be added for `prev_chunk_header.shard_id()`.

### Proof of Concept

A cargo integration test would:
1. Build a two-shard chain with `TestEnv`.
2. Call `get_state_response_header(shard_id=0, sync_hash)` on client 0 to obtain a valid header for shard 0.
3. Call `set_state_header(shard_id=1, sync_hash, header_for_shard_0)` on client 1.
4. Assert the call returns `Err(...)`, OR assert that `get_state_header(1, sync_hash).chunk.shard_id() == 1` (which currently fails — it returns 0).

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
