### Title
Accumulated `nativeFee` ETH Irrecoverably Locked in `OmniBridge` — Missing `ClaimNativeFee` Implementation - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

Every call to `initTransfer` with a non-zero `nativeFee` deposits ETH into the `OmniBridge` contract that can never be withdrawn. A `ClaimNativeFee` payload type is declared in `BridgeTypes.sol`, confirming the feature was planned, but no corresponding claim function was ever implemented. The ETH accumulates permanently with no admin-rescue path.

---

### Finding Description

`OmniBridge.initTransfer` is `payable` and requires callers to include `nativeFee` in `msg.value`:

```solidity
// For ERC20 tokens
extensionValue = msg.value - nativeFee;
// For native ETH
extensionValue = msg.value - amount - nativeFee;
``` [1](#0-0) 

Only `extensionValue` is forwarded to the Wormhole relay fee; `nativeFee` is silently retained by the contract:

```solidity
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [2](#0-1) 

The contract also has an unconditional `receive()` that accepts arbitrary ETH:

```solidity
receive() external payable {}
``` [3](#0-2) 

The only path for ETH to leave the contract is `finTransfer` when `payload.tokenAddress == address(0)`, which requires a valid MPC signature and is reserved for bridging native ETH — not for fee recovery:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [4](#0-3) 

Critically, `BridgeTypes.sol` declares a `ClaimNativeFee` payload type — proving the fee-claim flow was designed but never implemented:

```solidity
enum PayloadType {
    TransferMessage,
    Metadata,
    ClaimNativeFee   // declared but no handler exists
}
``` [5](#0-4) 

No function in `OmniBridge.sol` or `OmniBridgeWormhole.sol` handles `ClaimNativeFee`, and there is no `adminTransferETH`, `withdrawFee`, or equivalent rescue function anywhere in the contract. [6](#0-5) 

---

### Impact Explanation

Every `initTransfer` call with `nativeFee > 0` permanently locks that ETH in the contract. Over the bridge's lifetime this accumulates into a material sum of user-paid protocol fees. The funds are irrecoverable without a contract upgrade. This matches the **Critical** impact class: *permanent freezing / irrecoverable lock of user or protocol funds in bridge flows*, and also the **High** class: *fee accounting corruption that misdirects value* (fees intended for relayers are misdirected to the contract and locked).

---

### Likelihood Explanation

Any unprivileged user calling `initTransfer` with `nativeFee > 0` triggers the issue. The NEAR-side bridge code shows `native_token_fee` is a standard field in `InitTransferMsg` and is actively used in production flows. [7](#0-6) 

The `nativeFee` parameter is also emitted in the `InitTransfer` event, confirming it is a live, user-facing parameter — not a placeholder. [8](#0-7) 

---

### Recommendation

Implement the missing `claimNativeFee` function that the `ClaimNativeFee` payload type already anticipates. At minimum, add an admin-only ETH rescue function analogous to the original report's recommendation:

```solidity
function adminTransferETH(address receiver, uint256 amount)
    external
    onlyRole(DEFAULT_ADMIN_ROLE)
{
    (bool success, ) = receiver.call{value: amount}("");
    if (!success) revert FailedToSendEther();
}
```

A more complete fix would implement the full `ClaimNativeFee` signed-payload flow so that accumulated fees can be claimed by the designated fee recipient in a trustless, auditable way.

---

### Proof of Concept

1. Alice calls `initTransfer(tokenAddress=USDC, amount=1000e6, fee=0, nativeFee=0.01 ether, recipient="near:alice.near", message="")` with `msg.value = 0.01 ether` (the Wormhole fee is 0, so `extensionValue = 0`).
2. Inside `initTransfer`, `extensionValue = msg.value - nativeFee = 0`. `OmniBridgeWormhole.initTransferExtension` calls `_wormhole.publishMessage{value: 0}(...)`.
3. The `0.01 ether` `nativeFee` remains in the contract balance.
4. After 10,000 such transfers, `0.01 ether × 10,000 = 100 ETH` is locked.
5. No function exists to withdraw it. The `ClaimNativeFee` payload type is defined but unhandled. The funds are irrecoverable without a UUPS upgrade. [9](#0-8) [10](#0-9)
