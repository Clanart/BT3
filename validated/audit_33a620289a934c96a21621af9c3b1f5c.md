### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions via Overly Broad Pre-Deployment Phase Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (the only place where the transaction signature is cryptographically verified) for invoke transactions with `nonce=1` when the account has `nonce=0` in state and `account_tx_in_pool_or_recent_block` returns `true`. The mempool check is not specific to `deploy_account` transactions — it returns `true` for **any** transaction from the account in the pool. An unprivileged attacker can exploit this by submitting an invoke with an invalid signature for any account that has a pending `deploy_account` in the mempool, causing the gateway to admit the transaction without signature verification.

---

### Finding Description

The UX optimization in `skip_stateful_validations` is the direct Sequencer analog of the RocketStorage `tx.origin` initialization-phase bypass:

| RocketStorage | Sequencer |
|---|---|
| `storageInit == false` (initialization phase) | `account_nonce == 0` (pre-deployment phase) |
| `tx.origin == guardian` trusted | `account_tx_in_pool_or_recent_block(sender)` trusted |
| Guardian calls malicious contract | Legitimate user submits `deploy_account` for address A |
| Malicious contract writes to storage | Attacker submits invoke with invalid signature for A |
| `storageInit = true` set later | Account deployed later; invalid tx already in mempool |

The skip condition in `skip_stateful_validations`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second branch is circular reasoning. More critically, the check is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type from the account — not exclusively `deploy_account`. There is no type filter.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

And `validate_tx` in the blockifier immediately returns `Ok(None)` when `validate == false`, skipping the `__validate__` call entirely:

```rust
fn validate_tx(...) -> TransactionExecutionResult<Option<CallInfo>> {
    if !self.execution_flags.validate {
        return Ok(None);
    }
``` [4](#0-3) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and duplicate detection — it performs no cryptographic signature verification:

```rust
fn validate_incoming_tx(...) -> MempoolResult<()> {
    if self.tx_pool.get_by_tx_hash(tx_reference.tx_hash).is_ok() {
        return Err(MempoolError::DuplicateTransaction { ... });
    }
    self.state.validate_incoming_tx(tx_reference, incoming_account_nonce)
}
``` [5](#0-4) 

---

### Impact Explanation

An attacker can submit an invoke transaction with an **arbitrary invalid signature** for any account address that has a pending `deploy_account` in the mempool. The gateway admits this transaction without running `__validate__`, meaning the signature is never checked at the admission boundary. The transaction enters the mempool and is forwarded to the batcher.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The batcher's `ExecutionFlags::default()` has `validate: true`, so the batcher would re-run `__validate__` and the transaction would fail at execution time. However, the gateway/mempool admission invariant is broken: an invalid (unsigned) transaction for an account the attacker does not control is accepted into the sequencer's mempool.

---

### Likelihood Explanation

Medium. The attack window is the period between a legitimate user submitting a `deploy_account` and that transaction being committed to a block. During this window, any observer of the mempool can submit an invoke with `nonce=1` and an invalid signature for the target address. No privileged access is required. The attacker only needs to monitor the public mempool for pending `deploy_account` transactions.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. The mempool should expose a dedicated query such as `deploy_account_in_pool(address)` that only returns `true` when a `deploy_account` transaction (not any transaction) is pending for the given address. This mirrors the intent described in the comment and closes the gap between the documented assumption and the actual implementation.

---

### Proof of Concept

1. Alice submits `deploy_account` for address `A` with valid signature → enters mempool.
2. `account_tx_in_pool_or_recent_block(A)` now returns `true` (Alice's tx is in `tx_pool`).
3. Eve submits `Invoke { sender_address: A, nonce: 1, calldata: <arbitrary>, signature: <invalid> }`.
4. Gateway calls `extract_state_nonce_and_run_validations`:
   - `account_nonce = get_nonce_from_state(A)` → `Nonce(0)` (A not deployed yet).
   - `validate_nonce`: `0 <= 1 <= max_allowed_nonce` → passes.
   - `validate_by_mempool`: nonce ordering check passes, no signature check.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
5. `run_validate_entry_point` called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` not executed.
6. Eve's invoke with invalid signature is admitted to the mempool. [6](#0-5) [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
