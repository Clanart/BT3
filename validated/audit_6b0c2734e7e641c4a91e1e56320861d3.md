### Title
Missing Token Address Binding in `FinalizeTransfer` Payload Enables Unauthorized Minting of Bridged Assets — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

The `FinalizeTransferPayload` struct does not include the token address (mint), and the `mint` account in the `FinalizeTransfer` instruction context carries no constraint tying it to the signed payload. An attacker who holds a valid MPC-signed payload for a low-value bridged token can substitute any other bridged token's mint account and cause the program to mint an arbitrary amount of that high-value token to themselves.

---

### Finding Description

`FinalizeTransferPayload` is defined as:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
``` [1](#0-0) 

There is no `token` / `mint` field. The MPC signature therefore covers only `(destination_nonce, transfer_id, amount, fee_recipient)` — it does not commit to which token is being released.

The `FinalizeTransfer` accounts struct constrains the `mint` account only as:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [2](#0-1) 

There is no `constraint = ...` clause linking `mint.key()` to anything in `data.payload`. The `recipient` account is similarly unconstrained (`UncheckedAccount`). [3](#0-2) 

Inside `process()`, the only guard for the bridged-token path is:

```rust
require!(
    self.mint.mint_authority.contains(self.authority.key),
    ErrorCode::InvalidBridgedToken
);
mint_to(..., data.amount ...)?;
``` [4](#0-3) 

Every bridged token deployed by the bridge shares the same `authority` PDA as its mint authority, so this check passes for **any** bridge-deployed token — not just the one the payload was signed for.

The `FinalizeTransferResponse` posted back to NEAR records `token: self.mint.key()` (the substituted token), but NEAR's `claim_fee_callback` resolves the fee using the token stored in `pending_transfers` keyed by `transfer_id`, not the token reported in the response. NEAR therefore removes the original (low-value) transfer record and considers it settled, while the attacker has already received the high-value tokens on Solana. [5](#0-4) 

---

### Impact Explanation

An attacker can mint an unbounded quantity of any bridge-deployed SPL token on Solana by replaying a legitimately signed payload against a different mint. This constitutes **direct unauthorized minting of bridged assets** — a Critical impact under the allowed scope.

---

### Likelihood Explanation

Any unprivileged user can:
1. Initiate a small transfer of a low-value bridged token from NEAR to Solana via `ft_on_transfer → InitTransfer`.
2. Receive the MPC-signed `FinalizeTransferPayload` (publicly observable on-chain / via relayer).
3. Call `finalize_transfer` on Solana substituting a high-value bridged token's mint.

No privileged access, leaked keys, or colluding MPC signers are required. The signed payload is a normal protocol artifact.

---

### Recommendation

Add an explicit token-address field to `FinalizeTransferPayload` and enforce it in the account constraint:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub token: Pubkey,          // ← add this
    pub recipient: Pubkey,      // ← add this
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
```

Then constrain the accounts:

```rust
#[account(
    mut,
    constraint = mint.key() == data.payload.token @ ErrorCode::InvalidMint,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,

/// CHECK: validated against signed payload
#[account(constraint = recipient.key() == data.payload.recipient @ ErrorCode::InvalidRecipient)]
pub recipient: UncheckedAccount<'info>,
```

This ensures the MPC signature cryptographically binds to the specific token and recipient, eliminating the substitution attack.

---

### Proof of Concept

1. Attacker calls NEAR `ft_on_transfer` with `InitTransfer` for 1 unit of `low_value_token` destined for Solana.
2. NEAR MPC signs `FinalizeTransferPayload { destination_nonce: N, transfer_id: T, amount: 1, fee_recipient: None }`.
3. Attacker calls Solana `finalize_transfer` passing:
   - `data` = the valid signed payload above
   - `mint` = `high_value_token` mint (any bridge-deployed token with `authority` as mint authority)
   - `vault` = `None` (bridged path)
   - `recipient` = attacker's token account
4. `UsedNonces::use_nonce(N, ...)` succeeds (nonce N not yet used).
5. `require!(self.mint.mint_authority.contains(self.authority.key))` passes because `high_value_token` is bridge-deployed.
6. `mint_to(..., 1)` mints 1 unit of `high_value_token` to the attacker.
7. `FinalizeTransferResponse { token: high_value_token, amount: 1, transfer_id: T }` is posted to Wormhole.
8. NEAR's `claim_fee_callback` removes transfer `T` (for `low_value_token`) from `pending_transfers` — the bridge considers the transfer settled.

Net result: attacker receives `high_value_token` on Solana; NEAR's accounting is corrupted. Repeatable with fresh nonces from new low-value transfers.
