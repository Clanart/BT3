### Summary

The Borsh-encoded message that the NEAR MPC signs for `deployToken` / `deploy_token` / `deploy_token` (EVM / Solana / StarkNet) contains only token metadata â€” name, symbol, decimals, and NEAR token ID. It omits both the destination chain ID and the contract/program address. By contrast, `finTransfer` / `finalize_transfer` on every chain explicitly encodes `omniBridgeChainId` / `SOLANA_OMNI_BRIDGE_CHAIN_ID` / `chain_id` into the signed payload. Because the same NEAR MPC key (`nearBridgeDerivedAddress`) is shared across all EVM deployments, a valid `deployToken` signature obtained on one chain (e.g., Ethereum) can be submitted verbatim on any other chain (e.g., Arbitrum, Base) to deploy the same token without NEAR-side authorization for that chain.

### Impact Explanation

**Classification:** High â€” MPC signature verification bypass enabling unauthorized token deployment.

Because the signed message is chain-agnostic, a single MPC signature authorizing `deployToken` for token `foo.near` on Ethereum is equally valid on Arbitrum, Base, or any other EVM chain that shares the same `nearBridgeDerivedAddress`. An attacker who observes a legitimate `deployToken` transaction on chain A can immediately replay it on chain B:

1. A new `BridgeToken` proxy is deployed on chain B and registered in `nearToEthToken["foo.near"]` without the NEAR bridge having authorized this deployment for chain B.
2. The `DeployToken` event emitted on chain B is indistinguishable from a legitimate one; NEAR-side indexers or relayers that watch for this event may register the token and begin signing `finTransfer` messages for it, enabling minting on chain B.
3. The legitimate deployment of `foo.near` on chain B is permanently blocked (`ERR_TOKEN_EXIST`), forcing an admin intervention or contract upgrade to recover.

The same replay is possible within Solana if multiple program deployments share the same `derived_near_bridge_address`. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The attack requires no special privilege. Any unprivileged user can:
- Monitor the mempool or on-chain history of any supported EVM chain for a `deployToken` call.
- Extract the `signatureData` and `metadata` arguments.
- Submit the identical call to any other EVM chain where the same `nearBridgeDerivedAddress` is configured.

The NEAR bridge is deployed on multiple EVM chains (Ethereum, Arbitrum, Base, etc.) with the same MPC-derived signer, making the precondition trivially satisfied for any token that has been deployed on at least one chain.

---

### Recommendation

**Short term:** Include `omniBridgeChainId` (and optionally `address(this)`) in the Borsh-encoded payload that the MPC signs for `deployToken` on EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),   // ADD: bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `DeployTokenPayload::serialize_for_near` on Solana (prepend `SOLANA_OMNI_BRIDGE_CHAIN_ID`) and to `MetadataPayload::to_borsh` on StarkNet (prepend `omni_bridge_chain_id`). The NEAR MPC signing logic must be updated in lockstep to include the chain ID when producing `deployToken` signatures.

**Long term:** Adopt a uniform domain-separation scheme (analogous to EIP-712) for all MPC-signed messages that encodes chain ID, contract/program address, and a message type tag, ensuring no signed payload is valid on more than one deployment.
