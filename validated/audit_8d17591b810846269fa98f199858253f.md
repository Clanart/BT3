### Title
`eth_call` / `estimateGas` Silently Succeeds Where Real Execution Fails with `ErrStateConflict` — (`x/evm/keeper/state_transition.go`, `x/evm/statedb/statedb.go`)

---

### Summary

`eth_call` and `eth_estimateGas` both call `ApplyMessageWithConfig` with `commit=false`, which skips the `StateDB.Commit()` step entirely. The `ErrStateConflict` check — which fires when an EVM-dirty storage slot was also written by a nested native action (`ExecuteNativeAction`) in the same transaction — lives exclusively inside `Commit()`. As a result, any transaction that would fail on-chain with `ErrStateConflict` (status=0, gas consumed) is reported as successful by both simulation endpoints. Users who rely on `eth_call` or `eth_estimateGas` to pre-validate such transactions will submit them, pay gas, and receive a failed receipt.

---

### Finding Description

**The conflict-detection gate is only reachable when `commit=true`.**

In `ApplyMessageWithConfig`:

```go
if commit {
    if err := stateDB.Commit(); err != nil {
        // Note: estimateGas and eth_call do not hit this path because commit is
        // false for simulations, so they will succeed even when a real execution
        // would produce a state conflict.
        if errors.Is(err, statedb.ErrStateConflict) {
            return &types.EVMResult{
                GasUsed: gasUsed,
                VmError: statedb.ErrStateConflict.Error(),
                ...
            }, nil
        }
        ...
    }
}
``` [1](#0-0) 

`EthCall` and `EstimateGas` both pass `commit=false`:

```go
// EthCall
res, err := k.ApplyMessageWithConfig(ctx, msg, cfg, false)

// EstimateGas (executable closure)
rsp, err = k.ApplyMessageWithConfig(ctx, msg, cfg, false)
``` [2](#0-1) [3](#0-2) 

**The conflict is detected in `Commit()` only:**

`StateDB.Commit()` iterates every EVM-dirty storage slot and checks whether a nested native action (committed into `s.cacheMS` via `ExecuteNativeAction`) wrote a different value to the same slot:

```go
store := s.keeper.GetState(s.ctx, obj.Address(), key)
if store != origin && store != dirty {
    return fmt.Errorf("%w: address %s key %s ...", ErrStateConflict, ...)
}
``` [4](#0-3) 

**How `ExecuteNativeAction` creates the conflict:**

Every balance transfer (`Transfer`, `AddBalance`, `SubBalance`) and every stateful precompile write goes through `ExecuteNativeAction`, which commits native Cosmos state into a nested `cachemulti.Store` layer that is invisible to the EVM's in-memory `dirtyStorage`. If a precompile also writes to the same EVM contract storage slot that the outer EVM execution dirtied, the two writes diverge and `Commit()` returns `ErrStateConflict`. [5](#0-4) 

**Concrete trigger path (stateful precompile pattern):**

The official precompile guide documents the exact pattern that creates this divergence: a precompile calls `ExecuteNativeAction` to write Cosmos state, and that action creates an inner `StateDB` that writes to the same EVM contract storage slot the outer EVM call also dirtied. [6](#0-5) 

The `TestReentryAttack` and `TestNestedStateDBConflictDetection` tests confirm the conflict is real and that `Commit()` correctly rejects it on the consensus path: [7](#0-6) 

---

### Impact Explanation

When a transaction involves both an EVM `SSTORE` to a slot and a native precompile write to the same slot (via `ExecuteNativeAction`):

1. `eth_call` returns a successful result (no VmError, correct return data).
2. `eth_estimateGas` returns a valid gas estimate.
3. The user submits the transaction with the estimated gas.
4. On-chain, `ApplyMessageWithConfig` is called with `commit=true`; `Commit()` detects `ErrStateConflict` and returns an `EVMResult` with `VmError = "state conflict"` and `status=0`.
5. The transaction is included in the block, gas is fully consumed, and no state changes are applied.

This matches the allowed High impact: **"Public JSON-RPC, gRPC, simulation, tracing, receipt/log, or indexer path feeds incorrect consensus-critical data into transaction execution."** The simulation endpoints systematically misreport the outcome of an entire class of transactions, causing users to lose gas fees on transactions that will always fail.

---

### Likelihood Explanation

The trigger requires a chain that:
- Deploys a stateful precompile using `ExecuteNativeAction` (the documented and recommended pattern).
- Has a contract that both calls that precompile and writes to the same storage slot in the same transaction.

This is not a contrived edge case. The official `docs/precompile_creation_guide.md` explicitly documents the bank precompile pattern (Cronos) where a precompile writes to contract storage via a nested `StateDB`. Any contract that calls such a precompile and also performs its own `SSTORE` to the same slot will hit this divergence. The conflict is structurally guaranteed for any such interaction, making it reliably reproducible by any unprivileged user who calls the affected contract.

---

### Recommendation

1. **Detect `ErrStateConflict` in the simulation path.** In `ApplyMessageWithConfig`, when `commit=false`, perform the conflict-detection scan from `Commit()` (the loop over `dirtyStorage` vs `keeper.GetState`) without flushing, and surface the result as a `VmError` in the returned `EVMResult`. This makes `eth_call` and `estimateGas` faithfully reflect what on-chain execution will do.

2. **Alternatively**, document the limitation explicitly in the `EthCall` and `EstimateGas` gRPC handlers and add a warning to the precompile guide that precompiles writing to EVM contract storage slots that the EVM also writes will cause simulation/execution divergence.

---

### Proof of Concept

**Step 1 – Deploy a contract with a storage slot `slot0`.**

**Step 2 – Register a stateful precompile that, when called, creates an inner `StateDB` and writes `slot0 = 999` via `ExecuteNativeAction`.**

**Step 3 – Write a contract function that:**
- Calls the precompile (triggers inner `ExecuteNativeAction` → `slot0 = 999` in `cacheMS`).
- Also executes `SSTORE slot0 = 42` (marks `slot0` dirty in outer EVM with `dirty=42`, `origin=0`).

**Step 4 – Call `eth_call` / `eth_estimateGas` on this function.**

Result: Both return success. `eth_estimateGas` returns a valid gas figure. No VmError.

**Step 5 – Submit the transaction on-chain.**

Result: `ApplyMessageWithConfig` calls `stateDB.Commit()`. The conflict check fires:
- `origin = 0` (slot was 0 at tx start)
- `dirty = 42` (EVM wrote 42)
- `store = 999` (native action wrote 999 into `cacheMS`)
- `store != origin` (999 ≠ 0) AND `store != dirty` (999 ≠ 42) → `ErrStateConflict`

The transaction is included in the block with `VmError = "state conflict"`, `status=0`, and full gas consumed. The user paid gas for a transaction that `eth_call` and `estimateGas` both predicted would succeed. [8](#0-7) [9](#0-8)

func (s *StateDB) ExecuteNativeAction(contract common.Address, converter EventConverter, action func(ctx sdk.Context) error) error {
	prevStore := s.cacheMS
	prevLayerCount := len(s.cacheLayers)

	nextStore, ok := s.cacheMS.CacheMultiStore().(cachemulti.Store)
	if !ok {
		panic("expect nested CacheMultiStore result to be cachemulti.Store")
	}

	eventManager := sdk.NewEventManager()
	actionCtx := s.ctx.WithMultiStore(nextStore).WithEventManager(eventManager)

	if err := action(actionCtx); err != nil {
		return err
	}

	s.cacheMS = nextStore
	s.cacheLayers = append(s.cacheLayers, nextStore)
	s.ctx = s.ctx.WithMultiStore(nextStore)

	events := eventManager.Events()
	s.emitNativeEvents(contract, converter, events)
	s.nativeEvents = s.nativeEvents.AppendEvents(events)
	s.journal.append(nativeChange{
		previousStore:      prevStore,
		previousLayerCount: prevLayerCount,
		events:             len(events),
	})
	return nil

```go
	// Enforce the non-overlap invariant BEFORE flushing the native cache store.
	// A nested native action (via ExecuteNativeAction) commits its writes into s.cacheMS
	// (readable via s.ctx). If any EVM-dirty key was also written by such an action, the
	// store value visible through s.ctx will differ from originStorage. Detecting this
	// before flushing means we can abort cleanly — the parent context is never touched.
	//
	// Note: only EVM-dirty keys are checked; native-only writes have no EVM dirty bit
	// and are not in scope.
	for _, addr := range s.journal.sortedDirties() {
		obj, exist := s.stateObjects[addr]
		if !exist || obj.selfDestructed {
			continue
		}
		for _, key := range obj.dirtyStorage.SortedKeys() {
			origin := obj.originStorage[key]
			dirty := obj.dirtyStorage[key]
			if dirty == origin {
				continue
			}
			// A native action wrote this slot iff the store value differs from origin.
			// If it also differs from the EVM-dirty value, the two sides disagree — conflict.
			store := s.keeper.GetState(s.ctx, obj.Address(), key)
			if store != origin && store != dirty {
				return fmt.Errorf(
					"%w: address %s key %s modified by both EVM execution and native action (origin=%s, store=%s, dirty=%s)",
					ErrStateConflict,
					obj.Address().Hex(), key.Hex(), origin.Hex(), store.Hex(), dirty.Hex(),
				)
			}
		}
	}
```

// TestNestedStateDBConflictDetection verifies that Commit() returns an error when
// both the outer EVM dirty storage and an inner native action (via ExecuteNativeAction)
// modify the same storage key, enforcing the non-overlap invariant.
func (suite *StateDBTestSuite) TestNestedStateDBConflictDetection() {
	_, ctx, keeper := setupTestEnv(suite.T())

	contract := common.BigToAddress(big.NewInt(999))
	storageKey := common.BigToHash(big.NewInt(1))
	initVal := common.BigToHash(big.NewInt(1000))

	// Setup: write initial value and commit it
	db := statedb.New(ctx, keeper, emptyTxConfig)
	db.CreateAccount(contract)
	db.SetState(contract, storageKey, initVal)
	suite.Require().NoError(db.Commit())

	// Outer EVM: read the key (populates originStorage), then dirty it
	db = statedb.New(ctx, keeper, emptyTxConfig)
	state := db.GetState(contract, storageKey)
	suite.Require().Equal(initVal, state, "outer should read initial value")
	db.SetState(contract, storageKey, common.BigToHash(big.NewInt(10)))

	// Inner native action modifies the same storage key via keeper directly
	db.ExecuteNativeAction(contract, nil, func(innerCtx sdk.Context) error {
		innerDB := statedb.NewWithParams(innerCtx, keeper, emptyTxConfig, "uphoton")
		innerDB.SetState(contract, storageKey, common.BigToHash(big.NewInt(1100)))
		return innerDB.Commit()
	})

	// Commit should fail: outer dirty and inner native action both wrote to storageKey
	err := db.Commit()
	suite.Require().Error(err)
	suite.Require().Contains(err.Error(), "state conflict")
}
