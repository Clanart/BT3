### Title
Zero-Value Storage Slots Never Deleted from KV Store Due to `value.Bytes()` Mismatch — (`x/evm/statedb/statedb.go` + `x/evm/keeper/statedb.go`)

---

### Summary

When an EVM contract clears a storage slot (SSTORE to zero), `StateDB.Commit()` calls `keeper.SetState` with `value.Bytes()` of a zero `common.Hash`. Because `common.Hash{}.Bytes()` returns a 32-byte zero slice (not an empty slice), the keeper's delete-branch (`len(value) == 0`) is never triggered. The zero-value slot is **stored** in the KV store instead of **deleted**. Over time, a contract's storage prefix accumulates unbounded zero-value entries. Every subsequent `ForEachStorage` call — used by both `DeleteAccount` (selfdestruct) and `GetStorageRoot` — must iterate all of them, including the dead zero-value entries, mirroring the `shareBalance` EnumerableMap bloat in the Phi report.

---

### Finding Description

**Root cause — `x/evm/statedb/statedb.go` `Commit()`, lines 836–841:**

```go
for _, key := range obj.dirtyStorage.SortedKeys() {
    value := obj.dirtyStorage[key]
    if value == obj.originStorage[key] {
        continue
    }
    s.keeper.SetState(s.origCtx, obj.Address(), key, value.Bytes())
}
```

When a slot is cleared, `value` is `common.Hash{}`. `value.Bytes()` returns `[]byte{0x00, 0x00, …, 0x00}` — a 32-byte non-empty slice. [1](#0-0) 

**Delete guard — `x/evm/keeper/statedb.go` `SetState()`, lines 155–163:**

```go
func (k *Keeper) SetState(ctx sdk.Context, addr common.Address, key common.Hash, value []byte) {
    store := prefix.NewStore(ctx.KVStore(k.storeKey), types.AddressStoragePrefix(addr))
    if len(value) == 0 {          // ← only fires for nil / empty slice
        store.Delete(key.Bytes())
    } else {
        store.Set(key.Bytes(), value) // ← 32 zero bytes land here
    }
}
``` [2](#0-1) 

The guard `len(value) == 0` is **never satisfied** by a zero `common.Hash`, so every cleared slot is written as a 32-byte zero value and persists in the KV store forever.

**Unbounded iteration — `x/evm/keeper/statedb.go` `DeleteAccount()`, lines 209–212:**

```go
k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
    k.SetState(ctx, addr, key, nil)
    return true
})
``` [3](#0-2) 

`ForEachStorage` uses a KV prefix iterator over the entire storage prefix of the address: [4](#0-3) 

Every zero-value slot accumulated over the contract's lifetime is iterated here. With N cleared slots, this is O(N) KV reads + O(N) KV deletes, all charged against the Cosmos block gas meter.

**Second iteration site — `x/evm/statedb/statedb.go` `GetStorageRoot()`, lines 532–541:**

```go

Zero-value slots are fed into the trie, producing a storage root that diverges from the canonical Ethereum result (which excludes zero-value slots from the trie entirely).

---

### Impact Explanation

1. **Selfdestruct DoS / fund lock (High):** A contract that has accumulated K cleared storage slots requires O(K) KV store operations during `DeleteAccount`. An attacker can pre-bloat a contract's storage by repeatedly writing and clearing slots (each clear costs one SSTORE refund but leaves a zero-value KV entry). When the contract selfdestructs, `DeleteAccount` iterates all K dead entries. If K is large enough, the Cosmos block gas meter is exhausted, the transaction reverts, and the contract's ETH balance is permanently locked — the selfdestruct can never succeed.

2. **Incorrect `GetStorageRoot` / consensus divergence (High):** Zero-value slots are included in the `StackTrie` used to compute the storage root. This produces a hash that differs from what a standard Ethereum node would compute, corrupting any consensus or proof logic that relies on the storage root.

---

### Likelihood Explanation

Any unprivileged EVM transaction can trigger this. A contract needs only to:
- Write a non-zero value to a storage slot (SSTORE, costs ~20 000 gas)
- Clear it back to zero (SSTORE, costs ~5 000 gas with EIP-2200 refund)

Each such round-trip leaves one permanent zero-value KV entry. A single transaction can clear hundreds of slots. After enough transactions the contract's storage prefix is bloated. No privileged access, governance, or validator cooperation is required.

---

### Recommendation

In `x/evm/statedb/statedb.go` `Commit()`, pass `nil` (not `value.Bytes()`) when the dirty value is the zero hash, so the keeper's delete branch fires correctly:

```go
for _, key := range obj.dirtyStorage.SortedKeys() {
    value := obj.dirtyStorage[key]
    if value == obj.originStorage[key] {
        continue
    }
    var raw []byte
    if value != (common.Hash{}) {
        raw = value.Bytes()
    }
    s.keeper.SetState(s.origCtx, obj.Address(), key, raw)
}
```

Alternatively, extend `keeper.SetState` to treat a 32-byte all-zero slice as a delete signal:

```go
if len(value) == 0 || value == make([]byte, 32) {
    store.Delete(key.Bytes())
}
```

The first approach is preferred because it keeps the semantic contract of `SetState` simple.

---

### Proof of Concept

1. Deploy a contract with a loop that writes 5 000 distinct storage slots then clears each one to zero within the same or subsequent transactions.
2. After all clears are committed, call `eth_getStorageAt` on any of those slots — the RPC returns `0x0000…0000` (correct), but the underlying KV store still holds 5 000 entries (verifiable via a Cosmos KV store dump).
3. Trigger `SELFDESTRUCT` on the contract. `DeleteAccount` must iterate all 5 000 zero-value KV entries. Measure Cosmos gas consumed — it scales linearly with the number of cleared slots and can exceed the block gas limit, causing the selfdestruct to revert and permanently locking the contract's ETH balance.
4. Call `GetStorageRoot` on the bloated contract — the returned hash is non-zero (includes zero-value slots in the trie) instead of the empty-storage root `0x56e81f171…`, demonstrating the consensus-diverging incorrect state root.

func (k *Keeper) ForEachStorage(ctx sdk.Context, addr common.Address, cb func(key, value common.Hash) bool) {
	store := ctx.KVStore(k.storeKey)
	prefix := types.AddressStoragePrefix(addr)

	iterator := storetypes.KVStorePrefixIterator(store, prefix)
	defer iterator.Close()

	for ; iterator.Valid(); iterator.Next() {
		key := common.BytesToHash(iterator.Key())
		value := common.BytesToHash(iterator.Value())

		// check if iteration stops
		if !cb(key, value) {
			return
		}
	}
