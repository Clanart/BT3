Audit Report

## Title
Unrestricted `refundETH`, `sweepToken`, and `unwrapWETH9` allow any caller to drain ETH and tokens stranded on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.refundETH()`, `sweepToken()`, and `unwrapWETH9()` are all `public`/`external` with no caller restriction or ownership check. Any address can invoke them to redirect the router's entire ETH, ERC-20, or WETH balance to an arbitrary recipient. ETH residue is structurally created by `exactOutputSingle` when `msg.value > actualAmountIn`, and that residue is immediately stealable by any third party calling `refundETH()`.

## Finding Description

All three settlement helpers in `PeripheryPayments` operate on the router's **total** balance with no attribution or caller guard:

- `refundETH()` (L58–63): `external payable`, sends `address(this).balance` to `msg.sender` unconditionally. [1](#0-0) 
- `sweepToken()` (L48–55): `public payable`, sends the router's full ERC-20 balance to a caller-supplied `recipient`. [2](#0-1) 
- `unwrapWETH9()` (L37–45): `public payable`, withdraws and sends the router's full WETH balance to a caller-supplied `recipient`. [3](#0-2) 

ETH residue is structurally produced by the `pay()` helper: when `token == WETH` and `nativeBalance >= value`, exactly `value` is wrapped and transferred, leaving `nativeBalance - value` as raw ETH on the router. [4](#0-3) 

`exactOutputSingle` is `external payable` and callable directly without `multicall`. It does not call `refundETH()` internally; after the swap completes, any `msg.value - actualAmountIn` ETH remains on the router with no automatic return. [5](#0-4) 

`multicall` uses `delegatecall` and preserves `msg.sender`, but there is no enforcement that `refundETH()` must be included in the same batch — a user who omits it leaves ETH exposed. [6](#0-5) 

A search across all `.sol` files confirms no `onlyInitiator`, transient-storage initiator slot, or any equivalent caller guard exists anywhere in the periphery contracts for these three functions.

## Impact Explanation

Direct loss of user principal. An attacker who monitors the mempool can call `refundETH()` immediately after a victim's `exactOutputSingle{value: X}` confirms, stealing `X − actualAmountIn` ETH. For `sweepToken`, the attacker can redirect the router's entire token balance — potentially the full swap output — to an arbitrary address. No privileged role is required; any EOA can execute the attack. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation

`exactOutputSingle` with native ETH input is a standard user flow. Users routinely supply a round `msg.value` when the exact cost is unknown. Omitting `refundETH()` from the multicall, or calling `exactOutputSingle` directly, is a realistic and common mistake. The attack requires only a single public call with no setup, making it trivially repeatable by any mempool observer.

## Recommendation

Add a caller-attribution guard to all three helpers. Record `msg.sender` in transient storage at the start of each top-level router entry point and require the same address calls the settlement helpers:

```solidity
modifier onlyInitiator() {
    require(msg.sender == _tloadAddress(T_SLOT_INITIATOR), "not initiator");
    _;
}

function refundETH() external payable onlyInitiator { ... }
function sweepToken(...) public payable onlyInitiator { ... }
function unwrapWETH9(...) public payable onlyInitiator { ... }
```

Alternatively, remove the `recipient` parameter from `sweepToken` and `unwrapWETH9` and always send to `msg.sender`, and apply the initiator guard to `refundETH`.

## Proof of Concept

```
Setup:
  - Router deployed with WETH address.
  - Pool: WETH / Token1, seeded with liquidity.
  - Victim has 2 ETH and wants exactly 1000 Token1.

Step 1 — Victim calls exactOutputSingle directly (no multicall):
  router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
      pool: pool,
      tokenIn: WETH,
      tokenOut: Token1,
      zeroForOne: true,
      amountOut: 1000e18,
      amountInMaximum: 2 ether,
      recipient: victim,
      deadline: block.timestamp + 60,
      priceLimitX64: 0,
      extensionData: ""
  }));
  // Actual cost = 0.8 ETH. Remaining 1.2 ETH sits on the router.

Step 2 — Attacker (next tx or same block) calls:
  router.refundETH();
  // Attacker receives 1.2 ETH. Victim's excess is permanently lost.

Alternative (token theft):
  // Victim calls exactInputSingle with recipient = address(router).
  // Attacker calls sweepToken(Token1, 0, attacker) and receives all Token1.
```
