### Title
Unhandled External Call Revert in `finTransfer` Permanently Locks User Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.finTransfer`, when delivering native ETH or ERC1155 tokens to a recipient contract, the external calls have no error handling. If the recipient contract rejects ETH (missing `receive` function or reverts on receipt) or rejects ERC1155 tokens (missing `IERC1155Receiver` implementation), the entire `finTransfer` transaction reverts. Because the nonce mark at line 287 also reverts with the transaction, the transfer can be retried indefinitely but will always fail. With no cancel or refund path on the NEAR side, the originating user's funds are permanently locked in the NEAR bridge contract.

---

### Finding Description

`finTransfer` marks the destination nonce as consumed and then performs the actual asset delivery: [1](#0-0) 

```solidity
if (completedTransfers[payload.destinationNonce]) {
    revert NonceAlreadyUsed(payload.destinationNonce);
}
completedTransfers[payload.destinationNonce] = true;
```

After signature verification, the function branches on token type. For native ETH: [2](#0-1) 

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
```

For ERC1155 tokens: [3](#0-2) 

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
```

`IERC1155.safeTransferFrom` internally calls `onERC1155Received` on the recipient contract. If the recipient is a contract without a `receive` function (ETH case) or without a conforming `IERC1155Receiver` implementation (ERC1155 case), the call reverts. Because Solidity reverts roll back all state changes, `completedTransfers[payload.destinationNonce]` is also rolled back. The nonce is never consumed, so the relayer can retry — but every retry will produce the same revert. The transfer is permanently unfinalisable.

On the NEAR side, the originating transfer is stored in `pending_transfers` after `init_transfer` / `fin_transfer_callback`: [4](#0-3) 

There is no `cancel_transfer`, `refund_transfer`, or equivalent function in the NEAR contract that would allow the user to recover locked tokens when the EVM finalisation is permanently blocked. The `remove_transfer_message` path is only reachable through `sign_transfer_callback` (fee-zero path) or `claim_fee_callback`, neither of which applies here. [5](#0-4) 

---

### Impact Explanation

A user who initiates a NEAR→EVM bridge transfer specifying a contract recipient that rejects ETH (e.g., a multisig or smart-contract wallet without a `receive` function) or rejects ERC1155 tokens (e.g., a contract that does not implement `IERC1155Receiver`) will have their funds permanently locked in the NEAR bridge contract. There is no on-chain recovery path. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

Smart-contract wallets, multisigs, and protocol treasuries are common bridge recipients. Many do not implement a `receive` function or `IERC1155Receiver`. A user who bridges ETH or ERC1155 tokens to such an address — a realistic and common action — triggers the permanent lock. No privileged role or attacker cooperation is required; the user's own choice of recipient is sufficient.

---

### Recommendation

Wrap the ETH delivery and ERC1155 delivery in error-handling logic analogous to the Connext fix. For ETH, the existing `success` boolean already captures failure; instead of reverting, store the undeliverable amount for a separate claim. For ERC1155, use a try/catch:

```solidity
// ETH
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) {
    // store claimable balance for payload.recipient instead of reverting
    pendingEthClaims[payload.recipient] += payload.amount;
}

// ERC1155
try IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
) {} catch {
    // store claimable balance for payload.recipient
}
```

This ensures the nonce is consumed and the transfer is finalised even when delivery to the recipient fails, preventing permanent fund lock.

---

### Proof of Concept

1. User on NEAR calls `ft_transfer_call` → bridge stores transfer in `pending_transfers`, emits `InitTransferEvent` with `recipient = 0xDeadContract` (a contract with no `receive` function).
2. MPC signs the payload; relayer calls `OmniBridge.finTransfer` with the signed payload.
3. `completedTransfers[nonce] = true` executes at line 287.
4. `payload.recipient.call{value: payload.amount}("")` returns `success = false` at line 319–320.
5. `revert FailedToSendEther()` fires at line 322; entire transaction reverts including the nonce mark.
6. Relayer retries — same revert every time.
7. No cancel/refund function exists on NEAR; user's tokens are permanently locked. [6](#0-5) [7](#0-6)
