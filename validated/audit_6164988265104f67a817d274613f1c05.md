### Title
Unconstrained `is_reverted` Hint Allows Prover to Bypass Invoke Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`, `execution/transaction_impls.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a complete no-op. The `is_reverted` local variable in `execute_invoke_function_transaction` is loaded from an unconstrained hint and never cryptographically bound to the actual execution outcome. A malicious prover can set `is_reverted = 1` for any invoke transaction, causing the execute step to be silently skipped while fees are still charged — producing a valid proof that the L1 verifier accepts.

---

### Finding Description

In `execute_invoke_function_transaction`, the execution flow is:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    non_reverting_select_execute_entry_point_func(...);
} else {
    // Skip execution entirely.
    ...
}
// Fee is ALWAYS charged regardless of is_reverted.
charge_fee(...);
``` [1](#0-0) 

The function `check_is_reverted` that is supposed to constrain this value is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This is a pure no-op. In Cairo's proving model, a hint-loaded `local` variable is **unconstrained** unless subsequent Cairo assertions bind it. `check_is_reverted` performs zero assertions. The `if (is_reverted == FALSE)` branch only constrains which code path is taken — it does not link `is_reverted` to the actual execution result of the entry point.

The only constraint that exists is in `non_reverting_select_execute_entry_point_func`, which asserts `is_reverted = 0` after calling the entry point:

```cairo
assert is_reverted = 0;
``` [3](#0-2) 

This constraint only applies when the prover chooses `is_reverted == FALSE` (i.e., the execute branch). If the prover sets `is_reverted = 1`, the execute branch is never entered, so this assertion is never reached. No other assertion constrains `is_reverted` to match the actual execution outcome.

The same unconstrained pattern exists for L1 handler transactions, where there is no `check_is_reverted` call at all:

```cairo
local is_reverted;
%{ IsReverted %}
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [4](#0-3) 

For L1 handlers, if `is_reverted != FALSE`, the function returns before `consume_l1_to_l2_message` is called, meaning the L1→L2 message is never written to the OS output and the bridged funds are permanently frozen. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds / Permanent freezing of funds.**

For invoke transactions: a malicious prover sets `is_reverted = 1` for any invoke transaction. The execute step is skipped. `charge_fee` is still called and deducts fees from the user's account. The user pays fees but receives no execution. The resulting proof is valid — no Cairo assertion fails — and the L1 verifier accepts it. This is direct loss of funds.

For L1 handler transactions: a malicious prover sets `is_reverted = 1` for an L1 handler. The handler is skipped and `consume_l1_to_l2_message` is never called. The L1 message is not included in the OS output. The ETH/tokens locked on L1 are never released on L2. This is permanent freezing of funds.

---

### Likelihood Explanation

The exploit requires a malicious prover. In StarkNet's current architecture the sequencer acts as prover, which is a trusted role. However, the StarkNet OS is architecturally designed to be a **trustless** proof system — the entire security model rests on the guarantee that even a malicious prover cannot produce a valid proof that misrepresents execution. The empty `check_is_reverted` breaks this foundational guarantee. Any future decentralization of proving, or any scenario where the sequencer is compromised, makes this immediately exploitable with no additional preconditions.

---

### Recommendation

Implement `check_is_reverted` to cryptographically bind `is_reverted` to the actual execution outcome. The function must assert that `is_reverted` equals the `failure_flag` returned by the entry point execution. One approach: restructure the flow so that the entry point is always called and its `failure_flag` return value directly drives the branch, eliminating the hint-loaded `is_reverted` entirely. Alternatively, if the pre-execution revert check is intentional (e.g., for pre-validated reverts), the hint value must be constrained against a verifiable on-chain source before the branch is taken.

---

### Proof of Concept

1. User submits an invoke transaction (e.g., ERC-20 `transfer`) that would succeed.
2. Malicious prover populates the `IsReverted` hint with value `1` for this transaction.
3. `check_is_reverted(1)` executes and returns immediately — no assertion fires.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
5. `charge_fee` executes normally, deducting fees from the user's fee token balance.
6. The token transfer never occurs; the recipient receives nothing.
7. The Cairo program completes with no failed assertions.
8. A valid STARK proof is generated and submitted to L1.
9. The L1 verifier accepts the proof; the state root is updated to reflect fee deduction but no transfer.
10. The user has permanently lost their fee with no corresponding execution. [2](#0-1) [6](#0-5)
