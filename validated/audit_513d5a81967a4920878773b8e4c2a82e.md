### Title
Failed `fin_transfer` with Non-Empty `msg` Burns Returned Bridge Tokens Instead of Refunding User — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a cross-chain transfer to NEAR uses a non-empty `msg` field (triggering `ft_transfer_call` on the destination token), and the recipient contract rejects the tokens, the NEAR bridge's callback incorrectly **burns** the returned bridge tokens instead of minting them back to the original sender. Because the destination nonce is already marked as finalized and the transfer record is removed, there is no recovery path. The user's source-chain tokens (burned or locked during `init_transfer`) are permanently destroyed.

---

### Finding Description

**Vulnerability class:** Asset-accounting bug / callback-state desync — wrong recovery action (burn instead of mint-to-sender) after a failed destination-side `ft_transfer_call`.

**Root cause — `fin_transfer_send_tokens_callback`:** [1](#0-0) 

When `is_refund_required` returns `true` (the `ft_transfer_call` result is `U128(0)`, meaning the recipient returned all tokens), the bridge calls `burn_tokens_if_needed` on the token that was just returned to the bridge contract, then removes the fin-transfer record:

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    self.burn_tokens_if_needed(
        token.clone(),
        U128(transfer_message.amount_without_fee()...),
    );
    self.revert_lock_actions(&lock_actions);
    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
    // FailedFinTransferEvent emitted — no refund to user
}
```

`burn_tokens_if_needed` unconditionally burns the token if it is a deployed (bridge) token: [2](#0-1) 

The destination nonce was already marked as used inside `add_fin_transfer` (called at the top of `process_fin_transfer_to_near`): [3](#0-2) 

After `remove_fin_transfer` the transfer ID is gone and the nonce cannot be replayed.

**How the `msg` field triggers `ft_transfer_call`:**

`process_fin_transfer_to_near` passes `is_ft_transfer_call = !msg.is_empty()` to the callback: [4](#0-3) 

The `msg` field originates from the user-supplied `message` parameter in `initTransfer` on EVM: [5](#0-4) 

And from `init_transfer` on StarkNet: [6](#0-5) 

**Token lifecycle that causes permanent loss:**

1. Source chain (`initTransfer` EVM / `init_transfer` StarkNet): bridge token is **burned** from the user. [7](#0-6) [8](#0-7) 

2. NEAR `fin_transfer`: `send_tokens` mints the bridge token to the bridge contract itself, then calls `ft_transfer_call` to the recipient with the user's `msg`. [9](#0-8) 

3. Recipient contract rejects (`ft_on_transfer` returns the full amount). `ft_resolve_transfer` refunds tokens back to the bridge contract and returns `U128(0)`.

4. `fin_transfer_send_tokens_callback` sees `U128(0)` → `is_refund_required = true` → **burns** the returned tokens instead of minting them to the original sender. [10](#0-9) 

5. Source-chain tokens are already burned. Destination tokens are now burned. No recovery path exists.

---

### Impact Explanation

**Critical — Permanent, irrecoverable loss of user bridge tokens.**

For any bridge token (deployed via `deployToken`), a user who supplies a non-empty `message` and whose recipient contract rejects the `ft_transfer_call` will permanently lose their entire transferred amount. The nonce is consumed, the transfer record is deleted, and the tokens are burned on both sides of the bridge.

---

### Likelihood Explanation

**Medium.** The `message` field is a standard, documented feature of the bridge (used for DeFi composability). Any user who:
- bridges to a NEAR smart contract recipient, and
- the recipient's `ft_on_transfer` panics, returns a non-zero refund amount, or is not registered for the token

will trigger this path. This is a realistic, non-adversarial scenario (e.g., a contract upgrade that changes `ft_on_transfer` behavior, or a misconfigured recipient).

---

### Recommendation

In `fin_transfer_send_tokens_callback`, when `is_refund_required` is true and the token is a deployed bridge token, **mint the tokens back to the original sender** (`transfer_message.sender`) instead of burning them:

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    // Instead of burn_tokens_if_needed:
    if self.is_deployed_token(&token) {
        let sender_account = self.get_near_account_from_omni(&transfer_message.sender);
        ext_token::ext(token)
            .with_static_gas(MINT_TOKEN_GAS)
            .mint(sender_account, amount_without_fee, None)
            .detach();
    }
    // ... rest of cleanup
}
```

This mirrors the correct fix applied in the referenced MorphL2 report: use `mint` (not `transfer` or `burn`) when returning tokens to a user after a failed cross-chain delivery.

---

### Proof of Concept

1. Deploy a NEAR contract `reject_receiver` whose `ft_on_transfer` always returns the full amount.
2. Call `initTransfer` on EVM with `tokenAddress = <bridge_token>`, `amount = 1000`, `message = "trigger"`, `recipient = "reject_receiver.near"`.
3. EVM bridge burns 1000 bridge tokens from the user. `InitTransfer` event emitted.
4. Relayer submits proof to NEAR `fin_transfer`.
5. NEAR bridge mints 1000 tokens to itself, calls `ft_transfer_call("reject_receiver.near", 1000, "trigger")`.
6. `reject_receiver.ft_on_transfer` returns `U128(1000)` (full refund).
7. `ft_resolve_transfer` refunds 1000 tokens to bridge, returns `U128(0)`.
8. `fin_transfer_send_tokens_callback` receives `U128(0)` → `is_refund_required = true`.
9. `burn_tokens_if_needed` burns the 1000 tokens from the bridge.
10. `remove_fin_transfer` deletes the transfer record.
11. User has lost 1000 tokens permanently. No replay possible (nonce consumed).
