### Title
`batch_insert` Bypasses Duplicate-Key Validation, Causing `block_status_cache`/Blob State Inconsistency and Corrupted Merkle Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces. When a caller supplies duplicate `KeyId` or `Hash` values, `batch_insert` silently writes multiple leaf nodes into the raw blob while the `block_status_cache` (the authoritative `key→index` and `hash→index` maps) is silently overwritten to point only to the last-inserted copy. The earlier copies become permanently orphaned in the blob with `Parent(None)`, the tree structure is invalid, and the computed Merkle root no longer reflects the actual stored data. This allows forged inclusion/exclusion proofs against a corrupted root.

---

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (lines 570–657) takes a completely different path for the bulk of its work. When the tree already has ≥ 2 leaves (`leaf_count > 1`), **every** item in the batch is written directly via `insert_entry_to_blob` with no duplicate check. When the tree starts with ≤ 1 leaf, only the last two items (popped from the end of the vector) go through `insert()` with duplicate checking; all remaining items (iterated from the front) again bypass the check:

```rust
for ((key, value), hash) in keys_values_hashes {   // no duplicate check
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode {
        parent: Parent(None), hash, key, value,
    })};
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` — a silent overwrite:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silently overwrites
}
```

Consequences of inserting a duplicate key `K`:
1. **Blob**: two `LeafNode` blocks with `key = K` exist at distinct indexes. The first has `Parent(None)` and is never wired into the tree.
2. **Cache**: `key_to_index[K]` now points only to the second (last) index; the first is unreachable via the cache.
3. **Tree structure**: the orphaned first node is never linked to any parent, so `check_integrity` would detect `IntegrityKeyToIndexCacheLength` or `IntegrityUnmatchedChildParentRelationships` — but only if called.
4. **Root hash**: computed from the connected subtree only; the orphaned leaf's data is excluded, so the root does not commit to all inserted data.

The Python binding `py_batch_insert` (lines 1503–1519) exposes this path directly to callers:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

The only pre-call check is that `keys_values.len() == hashes.len()`; no deduplication is performed.

This is structurally identical to the reference report: a registration function that writes to one data structure (the blob) but fails to keep a second data structure (the cache / mapping) consistent, leaving the system in a silently broken state.

---

### Impact Explanation

A caller supplying duplicate `KeyId` or `Hash` values to `batch_insert` (directly in Rust or via the Python/WASM binding) causes:

- **Corrupted Merkle root**: the root hash does not commit to all inserted leaves, because orphaned nodes are excluded from the connected tree.
- **Forged inclusion proofs**: a key that was "orphaned" cannot be proven present (the cache no longer tracks it), even though its data was accepted.
- **Forged exclusion proofs**: a key that was inserted twice appears absent from the tree (only the last copy is reachable), allowing a proof of non-inclusion for data that was submitted.
- **Persistent state corruption**: the blob is written to disk (`to_path`); once persisted, the corruption is permanent and survives restarts.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion API for the DataLayer Merkle tree and is exposed via Python bindings used by the Chia full node's DataLayer subsystem. Any code path that feeds externally-sourced key-value pairs into `batch_insert` without pre-deduplication is vulnerable. The function's own contract (mirroring `insert`) implies duplicate rejection, but the implementation silently accepts duplicates. No existing test covers `batch_insert` with duplicate keys.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards inside `batch_insert` before calling `insert_entry_to_blob`, mirroring what `insert` does:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

Additionally, add a test analogous to `test_double_insert_fails` that calls `batch_insert` with a repeated key and asserts `KeyAlreadyPresent` is returned.

### Proof of Concept

```rust
#[test]
fn test_batch_insert_duplicate_key_corrupts_tree() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so leaf_count > 1 and ALL batch items
    // bypass self.insert() (the duplicate-checking path).
    let k0 = KeyId(0); let v0 = ValueId(0);
    let k1 = KeyId(1); let v1 = ValueId(1);
    blob.insert(k0, v0, &sha256_num(&0i64), InsertLocation::Auto {}).unwrap();
    blob.insert(k1, v1, &sha256_num(&1i64), InsertLocation::Auto {}).unwrap();

    // batch with a duplicate key (k0 appears twice)
    let batch = vec![
        ((k0, ValueId(99)), sha256_num(&99i64)),   // duplicate key
        ((KeyId(2), ValueId(2)), sha256_num(&2i64)),
    ];

    // Should return Err(KeyAlreadyPresent) but currently returns Ok(())
    blob.batch_insert(batch).unwrap();

    // Cache says leaf_count == 3, but blob has 4 leaf blocks.
    // Root hash does not commit to the orphaned first k0 node.
    // check_integrity() will detect the inconsistency:
    blob.check_integrity().expect_err("tree is corrupted");
}
```
