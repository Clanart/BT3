### Title
Unconstrained `is_reverted` Hint Allows Sequencer to Skip `__execute__` While Charging Full Fee - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted` in `execution_constraints.cairo` is an empty no-op function. The `is_reverted` flag that controls whether an invoke transaction's `__execute__` entry point runs is loaded from an unconstrained prover hint. Because no Cairo constraint enforces that `is_reverted` reflects the actual execution outcome, the sequencer/prover can set `is_reverted = TRUE` for any invoke transaction, skip its execution entirely, and still charge the user the full fee — producing a proof that the L1 verifier accepts as valid.

---

### Finding Description

In `execute_invoke_function_transaction`, the revert decision is made as follows:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // Stack alignment only — no execution.
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}

// Charge fee — always runs regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The value of `is_reverted` is provided entirely by the prover hint `%{ IsReverted %}`. The only validation call is `check_is_reverted(is_reverted)`, whose full implementation is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This function imposes **zero Cairo constraints** on `is_reverted`. There is no `assert`, no range check, no comparison — nothing. The prover can assign any non-zero felt to `is_reverted` and the OS will accept it, skip the `__execute__` entry point, and proceed to charge the fee.

The fee itself is also hint-driven (`%{ LoadActualFee %}`), with the only constraint being `assert_nn_le(calldata.amount.low, max_fee)` — meaning the sequencer can charge up to the user's declared maximum. [3](#0-2) 

The nonce is incremented unconditionally before the revert check, so the user cannot replay the transaction. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A sequencer/prover can:
1. Accept a user's signed invoke transaction (valid signature, valid nonce).
2. Set `is_reverted = TRUE` via the `%{ IsReverted %}` hint.
3. Skip the `__execute__` entry point entirely — no user-intended state changes occur.
4. Charge the user up to their declared `max_fee` via `charge_fee`.
5. Increment the user's nonce, preventing replay.
6. Produce a ZK proof that the L1 verifier accepts as fully valid, because `check_is_reverted` imposes no constraint.

The user loses their fee with no execution of their intended transaction. The sequencer retains the fee. This is a provably valid proof of an invalid state transition.

---

### Likelihood Explanation

The sequencer is the entity that generates the OS proof and provides all hints. The OS is the protocol's trust anchor — its Cairo constraints are the only mechanism preventing a sequencer from lying about execution outcomes. Because `check_is_reverted` is empty, this constraint is entirely absent. Any sequencer running this OS version can exploit this on any invoke transaction at will, with no cryptographic barrier.

---

### Recommendation

`check_is_reverted` must enforce that `is_reverted` is a boolean (0 or 1):

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (1 - is_reverted) = 0;
    return ();
}
```

Additionally, the OS should enforce that if `is_reverted == FALSE`, the execution entry point was actually called and returned a non-failure flag, rather than relying on the prover's pre-execution claim. The `is_reverted` value should be derived from the actual `failure_flag` returned by `execute_entry_point`, not from a pre-execution hint. [5](#0-4) 

---

### Proof of Concept

**Setup:** User submits an invoke transaction with `max_l2_gas_amount = 1_000_000`, `max_price_per_unit = 1` (max fee = 1,000,000 STRK).

**Attack steps:**

1. Sequencer receives the transaction. Signature is valid, nonce matches.
2. Sequencer generates the OS proof. At `%{ IsReverted %}` in `execute_invoke_function_transaction`, the sequencer sets `is_reverted = 1` (any non-zero felt).
3. `check_is_reverted(1)` is called — returns immediately with no assertion.
4. The `if (is_reverted == FALSE)` branch is not taken. `non_reverting_select_execute_entry_point_func` is never called. The user's `__execute__` entry point never runs.
5. `charge_fee` is called. At `%{ LoadActualFee %}`, the sequencer sets `low_actual_fee = 1_000_000`. The constraint `assert_nn_le(1_000_000, 1_000_000)` passes.
6. An ERC-20 transfer of 1,000,000 STRK from the user to the sequencer is executed.
7. The user's nonce is incremented.
8. The proof is submitted to L1 and verified successfully.

**Result:** User loses 1,000,000 STRK. Their intended transaction never executed. The proof is cryptographically valid. The user cannot replay the transaction (nonce consumed). [2](#0-1) [6](#0-5)
