Audit Report

## Title
Stranded Router ETH Silently Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` reads `address(this).balance` — the router's entire native ETH balance — when partially covering a WETH payment. ETH left on the router by a prior user (e.g., via a `payable` `multicall` that omitted `refundETH`) is silently wrapped and forwarded to the pool as part of the next user's WETH payment. The prior user loses their ETH permanently; the subsequent user has their WETH pull reduced by the stolen amount.

## Finding Description
The hybrid WETH branch in `pay` reads the router's global native balance with no per-call accounting: [1](#0-0) 

When `0 < address(this).balance < value`, the partial-ETH branch wraps all available ETH (regardless of who deposited it) and transfers it to the pool, then pulls only the remainder from `payer` via `safeTransferFrom`.

ETH can be stranded on the router because `multicall` is `payable`: [2](#0-1) 

The `receive()` guard only blocks bare ETH transfers (no calldata) from non-WETH senders: [3](#0-2) 

It does not prevent ETH from landing on the router via a `payable` function call such as `multicall{value: X}(...)`. If the user omits `refundETH` from the batch, that ETH remains on the router indefinitely.

The exploit path that triggers `pay` with `token == WETH` and `payer == msg.sender`: [4](#0-3) [5](#0-4) 

The existing `receive()` guard is insufficient because it only applies to bare ETH sends, not to ETH attached to `payable` function calls.

## Impact Explanation
Direct, permanent loss of user principal (HIGH). User A's stranded ETH is wrapped and forwarded to the pool with no compensation to User A. User B receives an unearned discount on their WETH payment funded entirely by User A's ETH. The pool is made whole, so no pool insolvency occurs, but User A suffers a full loss of whatever ETH was stranded. The loss is proportional to the stranded balance and requires no special permissions.

## Likelihood Explanation
Medium. The trigger condition — ETH stranded on a `payable` multicall router — is a well-documented class of user error. Users routinely send ETH for a WETH swap and omit `refundETH`, send excess ETH expecting automatic refund, or have a swap partially fill leaving residual ETH. Any subsequent WETH swap by any user will consume the stranded ETH. No attacker coordination is required; the consuming user need not even be aware of the stranded balance.

## Recommendation
Replace `address(this).balance` with the `msg.value` that belongs to the current top-level call:

1. Thread `msg.value` from `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` into the transient callback context alongside the existing payer/token fields.
2. In `pay`, use the stored per-call `msg.value` as the cap for native ETH consumption instead of `address(this).balance`.
3. As a defense-in-depth measure, add a post-call check in `multicall` (or in each swap entry point) that asserts `address(this).balance == 0` after execution, reverting if ETH was not fully consumed or refunded.

## Proof of Concept
```
1. User A calls multicall{value: 0.5 ether}([exactInputSingle(tokenA→tokenB, ...)])
   — omits refundETH; 0.5 ETH is stranded on the router.

2. User B calls exactInputSingle{value: 0}(WETH→tokenC, amountIn=1e18)
   — User B has approved the router for 1e18 WETH.

3. Pool calls metricOmmSwapCallback → _justPayCallback →
   pay(WETH, userB, pool, 1e18)

4. Inside pay (lines 73-84):
   nativeBalance = address(this).balance = 0.5e18  (User A's ETH)
   0 < nativeBalance < value → partial-ETH branch fires:
     deposit(0.5e18)          ← User A's ETH wrapped
     safeTransfer(pool, 0.5e18)
     safeTransferFrom(userB, pool, 0.5e18)  ← only 0.5 WETH pulled from B

5. Result:
   - User A: lost 0.5 ETH, received nothing.
   - User B: paid 0.5 WETH instead of 1 WETH (discount funded by User A).
   - Pool: received 1e18 WETH-equivalent (correct), unaware of the theft.

assert router.balance == 0          // User A's ETH is gone
assert userA_eth_loss == 0.5 ether  // confirmed theft
assert userB_weth_spent == 0.5e18   // B underpaid by 0.5e18
```
