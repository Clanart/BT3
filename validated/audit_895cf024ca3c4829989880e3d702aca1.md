### Title
ERC1155 `safeTransferFrom` Callback Rejection in `finTransfer` Permanently Locks Bridged Tokens — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.finTransfer` uses `IERC1155.safeTransferFrom` to deliver ERC1155 tokens to `payload.recipient`. ERC1155's safe-transfer standard mandates that if the recipient is a contract, it must implement `IERC1155Receiver.onERC1155Received` and return the correct magic selector. If the recipient contract reverts or lacks this interface, the entire `finTransfer` call reverts. Because the recipient address is fixed inside the MPC-signed payload and there is no pull-based fallback or admin rescue path for ERC1155 tokens, the bridged tokens become permanently irrecoverable.

### Finding Description

In `OmniBridge.finTransfer`, the nonce guard and the token delivery are both inside the same transaction:

```solidity
// OmniBridge.sol line 283-287
if (completedTransfers[payload.destinationNonce]) {
    revert NonceAlreadyUsed(payload.destinationNonce);
}
completedTransfers[payload.destinationNonce] = true;
```

Then, for ERC1155 tokens:

```solidity
// OmniBridge.sol lines 323-330
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this),
        payload.recipient,
        multiToken.tokenId,
        payload.amount,
        ""
    );
}
```

`IERC1155.safeTransferFrom` unconditionally calls `onERC1155Received` on the recipient if it is a contract (EIP-1155 §4.1). If that callback reverts — because the recipient is a multisig, a DeFi vault, a smart-contract wallet, or any contract that does not implement `IERC1155Receiver` — the entire `finTransfer` transaction reverts. The Solidity revert unwinds the `completedTransfers` write, so the nonce is not consumed. However, the signed payload encodes the recipient address as part of the MPC-signed message:

```solidity
// OmniBridge.sol lines 289-313 (Borsh encoding + ECDSA.recover)
bytes memory borshEncoded = bytes.concat(
    ...
    Borsh.encodeAddress(payload.recipient),
    ...
);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

No alternative payload with a different recipient can be finalized for the same origin transfer without a new MPC signature. The contract has no admin rescue function for ERC1155 tokens and no pull-based claim mechanism. Every retry of `finTransfer` with the same signed payload will always revert, leaving the ERC1155 tokens locked in the bridge contract forever.

On the source chain (NEAR), the tokens were already locked or burned at `initTransfer` time, so the user has no recourse on either side.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who bridges ERC1155 tokens from NEAR to an EVM contract address that does not implement `IERC1155Receiver` (a common situation: Gnosis Safe multisigs, Uniswap pools, lending protocols, etc.) will have their tokens permanently locked inside the bridge contract. The source-chain tokens are already consumed; the destination-chain delivery can never succeed; and there is no admin escape hatch.

### Likelihood Explanation

**Medium-High.** ERC1155 bridging is an explicitly supported protocol feature (`initTransfer1155`, `logMetadata1155`, `multiTokens` mapping). Many legitimate contract addresses — multisigs, protocol treasuries, smart-contract wallets — do not implement `IERC1155Receiver`. A user who specifies any such address as the recipient will trigger this condition. No attacker cooperation is required; the user's own honest action is sufficient to permanently lock funds.

### Recommendation

Replace the push-based `safeTransferFrom` delivery with a pull-based pattern for ERC1155 tokens:

1. Record the pending ERC1155 claim in a mapping (recipient → tokenAddress → tokenId → amount) instead of calling `safeTransferFrom` directly.
2. Expose a separate `claimERC1155(...)` function that the recipient calls at their convenience, using `safeTransferFrom` only then (so any callback revert harms only the claimer, not the bridge state).
3. Alternatively, use the non-safe `transferFrom` variant (if the ERC1155 implementation supports it) or wrap the `safeTransferFrom` in a try/catch and store unclaimed amounts for later retrieval.

### Proof of Concept

1. User on NEAR calls `initTransfer1155` bridging 100 units of an ERC1155 token, specifying a Gnosis Safe multisig address on EVM as `recipient`. NEAR locks the tokens.
2. MPC signs a `TransferMessagePayload` with `recipient = <GnosisSafe>`, `tokenAddress = <deterministicERC1155>`, `amount = 100`.
3. Relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
4. `completedTransfers[nonce]` is set to `true` (line 287).
5. `IERC1155.safeTransferFrom(address(this), <GnosisSafe>, tokenId, 100, "")` is called (line 324).
6. The ERC1155 contract calls `GnosisSafe.onERC1155Received(...)`. Gnosis Safe does not implement this interface → reverts.
7. Entire `finTransfer` transaction reverts. `completedTransfers[nonce]` is reset to `false`.
8. No alternative signed payload exists for a different recipient. Every retry reverts identically.
9. The 100 ERC1155 tokens remain locked in `OmniBridge` with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2)
