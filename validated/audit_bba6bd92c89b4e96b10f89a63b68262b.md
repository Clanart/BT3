### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Locks More Value Cross-Chain Than Actually Held ŌĆö (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` calls `safeTransferFrom(msg.sender, address(this), amount)` for native ERC-20 tokens, then unconditionally passes the caller-supplied `amount` into the cross-chain message and event. For fee-on-transfer (deflationary/rebasing) tokens, the contract receives fewer tokens than `amount`, but the bridge commits to releasing the full `amount` on the destination chain, permanently undercollateralizing the vault.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer()` function handles native ERC-20 tokens (those that are neither bridge tokens nor custom-minter tokens) in the `else` branch:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ŌåÉ caller-supplied, not balance-checked
    );
}
```

After the transfer, the original `amount` parameter is forwarded without any balance-delta check:

```solidity
initTransferExtension(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ŌåÉ still the pre-transfer caller value
    fee,
    ...
);

emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ŌåÉ same unchecked value
    ...
);
```

In `OmniBridgeWormhole.sol`, `initTransferExtension` encodes this `amount` directly into the Wormhole cross-chain payload:

```solidity
Borsh.encodeUint128(amount),   // line 136 ŌĆö the unchecked caller value
```

For a fee-on-transfer token with a 1% transfer fee, a user calling `initTransfer` with `amount = 1000` causes the bridge to:
- Receive only `990` tokens into its vault
- Broadcast a cross-chain message claiming `1000` tokens were locked
- The destination chain finalizes and releases `1000` tokens to the recipient

Each such transfer drains 10 tokens of real collateral from the bridge vault. Repeated transfers progressively undercollateralize the bridge until later users cannot withdraw their tokens.

---

### Impact Explanation

**High ŌĆö Balance/accounting corruption that breaks bridge collateralization.**

The bridge vault holds fewer tokens than the sum of all outstanding cross-chain transfer commitments. Once the deficit accumulates, `finTransfer` calls for legitimate users will fail because the bridge cannot cover the `safeTransfer(recipient, amount)` on the EVM side, permanently freezing those users' funds. The attacker does not need to steal tokens directly; they profit by receiving the full `amount` on the destination chain while the bridge absorbs the fee deficit.

---

### Likelihood Explanation

**Medium.** Any unprivileged user can call `initTransfer` with any ERC-20 token address that is not in `isBridgeToken` or `customMinters`. Fee-on-transfer tokens exist on mainnet (e.g., STA, REFLECT-style tokens). No special role or privileged access is required. The attacker only needs to initiate a standard bridge transfer with such a token.

---

### Recommendation

Measure the actual received amount using a before/after balance check, and use that measured value in the cross-chain message instead of the caller-supplied `amount`:

```solidity
} else {
    uint256 beforeBalance = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
    uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - beforeBalance;
    amount = uint128(actualReceived); // use actual received amount downstream
}
```

Then pass the adjusted `amount` to `initTransferExtension` and the `InitTransfer` event. This mirrors the fix applied in the referenced Beanstalk `LibTransfer.receiveToken()` remediation.

---

### Proof of Concept

1. A fee-on-transfer ERC-20 token `FOT` (2% fee) is deployed on EVM. It is not a bridge token and has no custom minter, so it falls into the `else` branch of `initTransfer`.
2. Attacker calls `OmniBridge.initTransfer(FOT, 1000, 0, 0, "near:recipient", "")`.
3. `safeTransferFrom(attacker, address(this), 1000)` executes; the bridge receives `980` tokens (2% fee taken by the token).
4. `initTransferExtension` is called with `amount = 1000` and the Wormhole message encodes `1000`. [1](#0-0) 
5. `emit InitTransfer(..., amount=1000, ...)` is emitted. [2](#0-1) 
6. The NEAR bridge processes the Wormhole message and signs a `finTransfer` releasing `1000` FOT-equivalent tokens to the recipient on NEAR. [3](#0-2) 
7. The bridge vault now holds `980` but has committed to `1000`. The 20-token deficit is absorbed by the protocol.
8. Repeating this 50 times drains the entire vault, and all subsequent legitimate `finTransfer` calls on EVM revert because `safeTransfer(recipient, amount)` cannot be satisfied. [4](#0-3)
