### Title
Unconstrained `recipient` Account in Solana `FinalizeTransfer` Enables Theft of Bridged Assets - (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary
The Solana `FinalizeTransfer` instruction accepts a caller-supplied `recipient` account that is never validated against the signed payload. Any party who possesses a valid `SignedPayload<FinalizeTransferPayload>` can substitute their own address as `recipient`, causing the bridge to mint or release tokens to the attacker instead of the intended beneficiary.

### Finding Description
In `finalize_transfer.rs`, the Anchor account struct declares `recipient` as an `UncheckedAccount` with the comment `/// CHECK: this can be any type of account` and no account constraint tying it to the instruction payload: [1](#0-0) 

The `token_account` — the actual destination of the minted or released tokens — is derived solely from this unchecked `recipient`: [2](#0-1) 

The `process()` function then transfers or mints tokens directly into `self.token_account` (which is the ATA of `self.recipient`) without ever asserting that `self.recipient.key()` equals any recipient field in the verified payload: [3](#0-2) 

The `SignedPayload<FinalizeTransferPayload>` is signature-verified (covering `destination_nonce`, `amount`, `fee_recipient`, `transfer_id`), but the `recipient` account passed in the instruction accounts list is entirely caller-controlled and is never cross-checked against the payload. The `#[instruction(data: SignedPayload<FinalizeTransferPayload>)]` attribute makes the payload available for account constraints, yet no constraint of the form `recipient.key() == data.payload.recipient` is present. [4](#0-3) 

Compare this with the EVM `finTransfer`, which embeds `payload.recipient` inside the Borsh-encoded, ECDSA-signed blob, so the recipient is cryptographically bound and cannot be substituted: [5](#0-4) 

The Solana path has no equivalent binding.

### Impact Explanation
**Critical — Direct theft of bridged assets.**

An attacker who obtains any valid `SignedPayload<FinalizeTransferPayload>` (e.g., by observing a pending transaction in the mempool, replaying a broadcast, or front-running a relayer submission) can call `finalize_transfer` with the authentic signed payload but substitute their own public key as the `recipient` account. The signature check passes because the payload bytes are unchanged. The nonce is consumed, permanently preventing the legitimate recipient from ever claiming the transfer. Tokens are minted (bridged token path) or released from the vault (native token path) directly into the attacker's associated token account.

### Likelihood Explanation
**High.** Solana transactions are publicly observable before finalization. Any relayer, MEV searcher, or passive observer can extract a valid signed payload from a pending or recently broadcast transaction and re-submit it with a substituted `recipient`. No privileged access, leaked key, or colluding party is required — only the ability to submit a Solana transaction.

### Recommendation
Add an explicit account constraint that binds the `recipient` account to the recipient field in the verified payload. For example, if `FinalizeTransferPayload` carries a `recipient: Pubkey` field:

```rust
#[account(
    constraint = recipient.key() == data.payload.recipient @ ErrorCode::InvalidRecipient
)]
/// CHECK: validated against signed payload recipient field
pub recipient: UncheckedAccount<'info>,
```

If the payload does not currently carry a recipient field, add one and include it in the signed data so the MPC/NEAR signature cryptographically commits to the intended Solana recipient, mirroring the EVM implementation.

### Proof of Concept

1. Victim initiates a cross-chain transfer from NEAR/EVM to Solana. The NEAR bridge signs a `FinalizeTransferPayload` for the victim's Solana public key.
2. Attacker observes the relayer's pending `finalize_transfer` transaction on Solana (or the signed payload in any off-chain channel).
3. Attacker constructs a new `finalize_transfer` instruction using the **identical** `SignedPayload<FinalizeTransferPayload>` (signature + payload unchanged) but replaces the `recipient` account with their own public key and derives the corresponding `token_account` (their own ATA).
4. Attacker submits the transaction. The nonce check passes (not yet used), the signature check passes (payload bytes unchanged), and tokens are minted/transferred to the attacker's ATA.
5. The legitimate victim's transfer nonce is now consumed; the victim receives nothing and has no recourse. [6](#0-5)

impl FinalizeTransfer<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        UsedNonces::use_nonce(
            data.destination_nonce,
            &self.used_nonces,
            &mut self.config,
            self.authority.to_account_info(),
            self.common.payer.to_account_info(),
            &Rent::get()?,
            self.system_program.to_account_info(),
        )?;

        if let Some(vault) = &self.vault {
            // Native version. We have a proof of token registration by vault existence
            transfer_checked(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    TransferChecked {
                        from: vault.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                        mint: self.mint.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
                self.mint.decimals,
            )?;
        } else {
            // Bridged version. May be a fake token with our authority set but it will be ignored on the near side
            require!(
                self.mint.mint_authority.contains(self.authority.key),
                ErrorCode::InvalidBridgedToken
            );

            mint_to(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    MintTo {
                        mint: self.mint.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
            )?;
        }

        let payload = FinalizeTransferResponse {
            token: self.mint.key(),
            amount: data.amount,
            fee_recipient: data.fee_recipient.unwrap_or_default(),
            transfer_id: data.transfer_id,
        }
        .serialize_for_near(())?;

        self.common.post_message(payload)?;

        Ok(())
    }
