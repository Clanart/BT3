### Title
Unvalidated L1 WEI Gas Prices in `ProposalInit` Allow Wrong Block Hash Commitment and Incorrect ETH Fee Calculations — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates the proposed L1 gas prices only in FRI units, silently discarding the WEI prices returned by the provider. Yet both WEI and FRI prices are committed into the block hash via `gas_prices_to_hash`, and the WEI prices are used to derive `l2_gas_price_wei` for ETH-denominated fee calculations. A proposer can therefore supply arbitrary WEI prices that pass all validation checks, causing every validator to accept a block whose hash commitment encodes wrong WEI prices and whose ETH fee accounting is manipulated.

---

### Finding Description

**Root cause — discarded WEI prices in `is_block_info_valid`:**

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  lines 286-292
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
```

`_l1_gas_prices_wei` is intentionally discarded. Only the FRI prices are compared against the proposer's values:

```rust
// lines 302-319
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

`l1_gas_price_wei` and `l1_data_gas_price_wei` from `ProposalInit` are never compared against the provider's WEI prices. The only constraint is that they are non-zero (enforced by `NonzeroGasPrice::new` in `convert_to_sn_api_block_info`).

**WEI prices feed the block hash commitment:**

`PartialBlockHashComponents::new` copies both WEI and FRI prices from `BlockInfo` into the commitment structure:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  lines 224-235
pub fn new(block_info: &BlockInfo, header_commitments: BlockHeaderCommitments) -> Self {
    Self {
        l1_gas_price: block_info.gas_prices.l1_gas_price_per_token(),   // wei + fri
        l1_data_gas_price: block_info.gas_prices.l1_data_gas_price_per_token(),
        l2_gas_price: block_info.gas_prices.l2_gas_price_per_token(),
        ...
    }
}
```

`gas_prices_to_hash` (for Starknet ≥ 0.13.4) chains all six values — including the three WEI prices — into the Poseidon block hash:

```rust
// lines 423-434
vec![
    HashChain::new()
        .chain(&STARKNET_GAS_PRICES0)
        .chain(&l1_gas_price.price_in_wei.0.into())
        .chain(&l1_gas_price.price_in_fri.0.into())
        .chain(&l1_data_gas_price.price_in_wei.0.into())
        .chain(&l1_data_gas_price.price_in_fri.0.into())
        .chain(&l2_gas_price.price_in_wei.0.into())
        .chain(&l2_gas_price.price_in_fri.0.into())
        .get_poseidon_hash(),
]
```

**WEI prices also determine `l2_gas_price_wei` via the ETH-to-FRI rate:**

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs  lines 304-314
let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
// eth_to_fri_rate = l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei

let l2_gas_price_wei = NonzeroGasPrice::new(
    init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?
)?;
```

If `l1_gas_price_wei` is set to a small value (e.g., 1), `eth_to_fri_rate` becomes `l1_gas_price_fri × 10^18`, making `l2_gas_price_wei` collapse to `l2_gas_price_fri / l1_gas_price_fri` — potentially near zero — which distorts all ETH-denominated L2 gas fees for every transaction in the block.

**Attack path:**

1. Proposer broadcasts `ProposalInit` with valid FRI prices (passes `within_margin`) but `l1_gas_price_wei = 1`, `l1_data_gas_price_wei = 1`.
2. `is_block_info_valid` passes — only FRI prices are checked.
3. `convert_to_sn_api_block_info` builds `BlockInfo` with the manipulated WEI prices and a derived `l2_gas_price_wei` that is orders of magnitude too small.
4. The batcher executes the block with this `BlockInfo`; `PartialBlockHashComponents::new` records the wrong WEI prices.
5. The proposer's `ProposalCommitment` (partial block hash) is computed from the same wrong `BlockInfo`, so it matches the batcher's output.
6. Every validator runs the same batcher path with the same `BlockInfo` → same commitment → proposal accepted.
7. The committed block hash encodes wrong WEI prices; ETH fee receipts for all transactions in the block are wrong.

---

### Impact Explanation

**Wrong block hash commitment (High/Critical):** The Poseidon block hash chains the WEI gas prices. A proposer-controlled WEI price produces a block hash that diverges from what an honest sequencer would produce. Any downstream consumer that reconstructs or verifies the block hash (e.g., SNOS proof facts validation via `validate_proof_block_hash`, P2P sync header verification, re-execution) will either fail or accept a hash that does not correspond to the true L1 gas prices.

**Incorrect ETH fee accounting (Critical):** `l2_gas_price_wei` is derived from the manipulated `eth_to_fri_rate`. All ETH-denominated fee checks (`FeeCheckReport`, `PostValidationReport`, `PostExecutionReport`) and fee transfers use this wrong price. A proposer can make ETH fees arbitrarily cheap for colluding users, constituting a direct economic impact.

---

### Likelihood Explanation

The proposer role rotates among consensus validators. Any validator that becomes proposer can exploit this without any external dependency. The only constraint is that FRI prices must be within the configured margin — WEI prices are entirely free. The attack is silent: no error is logged, no metric is incremented, and the proposal succeeds normally.

---

### Recommendation

In `is_block_info_valid`, validate the proposed WEI prices against the provider's WEI prices using the same `within_margin` logic already applied to FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

if !(within_margin(l1_gas_price_fri_proposed,  l1_gas_prices_fri.l1_gas_price,  margin)
  && within_margin(l1_data_gas_price_fri_proposed, l1_gas_prices_fri.l1_data_gas_price, margin)
  && within_margin(init_proposed.l1_gas_price_wei, l1_gas_prices_wei.l1_gas_price, margin)
  && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

Alternatively, derive the WEI prices deterministically from the validated FRI prices and the validated ETH-to-FRI rate, removing the proposer's freedom to choose them independently.

### Proof of Concept

```
Precondition:
  - Honest ETH-to-FRI rate: 800 × 10^18  (1 ETH = 800 STRK)
  - Honest l1_gas_price_wei: 10^9 (1 Gwei)
  - Honest l1_gas_price_fri: 800 × 10^9 (800 Gwei in FRI)
  - l2_gas_price_fri: 10^9

Malicious ProposalInit:
  l1_gas_price_fri     = 800 × 10^9   ← within margin, passes validation
  l1_gas_price_wei     = 1             ← not validated, accepted
  l1_data_gas_price_fri = <valid>
  l1_data_gas_price_wei = 1            ← not validated, accepted

Derived eth_to_fri_rate (in convert_to_sn_api_block_info):
  = 800×10^9 × 10^18 / 1 = 800×10^27

Derived l2_gas_price_wei:
  = 10^9 × 10^18 / (800×10^27) = 10^27 / (800×10^27) ≈ 0
  → NonzeroGasPrice::new(0) returns Err → proposal rejected in this extreme case

With l1_gas_price_wei = 10^6 (still 1000× too small):
  eth_to_fri_rate = 800×10^9 × 10^18 / 10^6 = 800×10^21
  l2_gas_price_wei = 10^9 × 10^18 / (800×10^21) = 10^27 / (800×10^21) ≈ 1250
  → NonzeroGasPrice::new(1250) succeeds
  → ETH L2 gas price is 1250 wei instead of honest 1.25×10^9 wei (10^6× cheaper)
  → All ETH-denominated fees in the block are 10^6× underpriced
  → Block hash encodes price_in_wei = 10^6 instead of 10^9
