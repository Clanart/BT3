### Title
Empty `check_is_reverted` Allows Prover to Arbitrarily Skip Invoke Transaction Execution - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a complete no-op stub. It is called in `execute_invoke_function_transaction` to validate the hint-supplied `is_reverted` flag before deciding whether to run the `__execute__` entry point. Because the function contains no logic, the `is_reverted` value is entirely unconstrained. A malicious prover can freely set `is_reverted = TRUE` for any invoke transaction, causing the OS to skip the `__execute__` entry point while still charging fees, producing a valid proof for an incorrect state transition.

---

### Finding Description

In `execution_constraints.cairo`, the function `check_is_reverted` is defined as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

It accepts the `is_reverted` argument and immediately returns without performing any assertion, range check, or constraint. The file is explicitly named "Execution constraints for transaction execution," and the sibling function `check_proof_facts` in the same file is fully implemented with multiple assertions. The empty body of `check_is_reverted` is therefore a missing implementation, not an intentional design.

This function is called inside `execute_invoke_function_transaction` in `transaction_impls.cairo`:

    ...
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}

// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

The value of `is_reverted` is loaded from the hint `%{ IsReverted %}`. In Cairo, hints are prover-supplied and are not constrained by the VM itself — constraints must come from subsequent assertions. Because `check_is_reverted` adds zero constraints, the prover can supply any value for `is_reverted` and the OS will accept it as valid. Specifically, the prover can set `is_reverted = TRUE` (any non-zero felt) for a transaction that was never actually reverted, causing the OS to:

1. Skip the `__execute__` entry point entirely.
2. Still call `charge_fee`, deducting fees from the user.
3. Commit a state transition that omits all storage writes, token transfers, and other effects that `__execute__` would have produced.

The resulting proof is cryptographically valid from the verifier's perspective, yet it attests to an incorrect state root.

---

### Impact Explanation

**Critical — Permanent freezing of funds and direct loss of funds.**

- Any invoke transaction whose `__execute__` entry point performs a token transfer or state mutation can be silently skipped. The intended recipient never receives the funds, and the sender's balance is not restored (fees are still deducted). Funds are permanently frozen in the sender's account with no recourse.
- The fee is charged regardless of whether execution occurred, constituting a direct loss of funds for the transaction sender.
- Because the OS proof is accepted as valid, the incorrect state root is finalized on L1, making the damage irreversible.

---

### Likelihood Explanation

The prover in the StarkNet architecture is the entity that runs the OS Cairo program and generates the STARK proof. The entire purpose of the OS is to be the cryptographic trust anchor that prevents even a malicious sequencer/prover from committing an invalid state transition. A bug that leaves a critical branch condition unconstrained directly undermines this guarantee. Any prover operating the OS — including a future decentralized prover — can exploit this without any special key or privilege beyond the ability to run the OS program and supply hints.

---

### Recommendation

Implement `check_is_reverted` with a concrete constraint. At minimum, assert that `is_reverted` is a boolean (0 or 1):

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (1 - is_reverted) = 0;
    return ();
}
```

A stronger implementation should verify that `is_reverted` is consistent with the actual execution outcome recorded in the execution helper, analogous to how `check_proof_facts` validates proof-fact consistency. [3](#0-2) 

---

### Proof of Concept

1. A user submits an invoke transaction (e.g., an ERC-20 `transfer` call).
2. The malicious prover, when running the OS, supplies `is_reverted = 1` via the `%{ IsReverted %}` hint in `execute_invoke_function_transaction`.
3. `check_is_reverted(1)` is called and returns immediately — no assertion fires.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
5. `charge_fee` executes normally, deducting the full fee from the user.
6. The OS produces a valid STARK proof attesting to a state root that reflects the fee deduction but none of the `__execute__` effects (e.g., the token transfer never happened).
7. The proof is verified and accepted on L1. The recipient's balance is never credited; the sender's funds are permanently frozen. [2](#0-1) [1](#0-0)
