### Title
Integer Division Truncation in `calculate_next_base_gas_price` Causes Gas Price to Permanently Freeze at Low Values — (`File: crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`)

---

### Summary

In `calculate_next_base_gas_price`, when the current L2 gas price is below the configured minimum, the recovery increment is computed as `price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR` using integer division. When `price.0 < MIN_GAS_PRICE_INCREASE_DENOMINATOR` (i.e., `price.0 < 333`), this division truncates to zero, so `adjusted = price.0 + 0 = price.0`, and the gas price never increases. The frozen price is then written into the `ProposalFin`'s `L2GasInfo.next_l2_gas_price_fri` field, which is committed to the block header and propagated to all validators and the storage layer as the authoritative next-block gas price.

---

### Finding Description

In `crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`, the below-minimum recovery branch computes:

```rust
let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;  // line 104
let adjusted = price.0 + max_increase;                             // line 105
```

`MIN_GAS_PRICE_INCREASE_DENOMINATOR` is `333`. For any `price.0` in the range `[1, 332]`, integer division yields `max_increase = 0`, so `adjusted == price.0`. The price is then returned unchanged as `GasPrice(adjusted_price)` at line 112.

This frozen price is passed directly into `L2GasInfo { next_l2_gas_price_fri: next_l2_gas_price, ... }` inside `ProposalFin` (line 325 of `build_proposal.rs`), which is broadcast to all consensus peers and written into the committed block header field `next_l2_gas_price` (protobuf field 19 of `SignedBlockHeader`). Every subsequent block inherits this frozen price as its starting point, repeating the same truncation indefinitely.

The analogous correct path in the EIP-1559 branch (lines 122–124) explicitly avoids this by multiplying before dividing using U256 arithmetic, with a comment noting the fix. The below-minimum branch has no such protection.

---

### Impact Explanation

The `next_l2_gas_price` value written into the committed block header is the authoritative L2 gas price for the next block. It is:

1. Included in `ProposalFin.fin_payload.l2_gas_info.next_l2_gas_price_fri`, which is part of the proposal commitment agreed upon by BFT consensus.
2. Stored in the block header's `next_l2_gas_price` field, serialized to storage and propagated via P2P sync.
3. Used by the gateway's stateful validator to compute the minimum acceptable L2 gas price threshold for incoming transactions.

A permanently frozen gas price at a sub-minimum value means:
- The fee market's minimum price enforcement is silently bypassed for all subsequent blocks.
- Transactions paying the frozen (sub-minimum) price are admitted by the gateway and sequenced, producing wrong fee receipts and wrong state diffs.
- The block header commitment encodes a wrong `next_l2_gas_price`, causing divergence between nodes that recompute the price and nodes that trust the committed value.

This matches: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact** and **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

The condition `price.0 < 333` is reachable whenever:
- A node bootstraps from a synced block whose `next_l2_gas_price` was set to a very small value (e.g., 1 fri) by a misconfigured or adversarial proposer.
- The versioned-constants `min_gas_price` is raised via a dynamic config update while the current price is still very low.
- A test or devnet environment initializes with a small gas price.

Once the price enters the frozen range, it cannot self-recover through the normal fee-market mechanism, making the condition persistent.

---

### Recommendation

Replace the integer division with a saturating minimum increment of 1 when the computed increase is zero:

```rust
let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
let max_increase = max_increase.max(1); // ensure at least 1 fri increase per block
let adjusted = price.0 + max_increase;
```

Alternatively, mirror the EIP-1559 branch and use rational arithmetic or U256 to avoid truncation, or enforce a protocol-level minimum of `MIN_GAS_PRICE_INCREASE_DENOMINATOR` fri for any gas price that enters the below-minimum recovery path.

---

price = GasPrice(100)   // 100 fri, below any realistic min_gas_price

adjusted     = 100 + 0  = 100
adjusted_price = min(100, min_gas_price.0) = 100

// Returned: GasPrice(100) — identical to input.
// Next block starts with price = 100 again → infinite loop.
