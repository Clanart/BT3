### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Validation for All But the Last Two Entries, Corrupting the Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` only routes the final two items through the guarded `insert()` path (which enforces `KeyAlreadyPresent` / `HashAlreadyPresent`). Every other item in the batch is written directly to the blob via `insert_entry_to_blob` with no duplicate check. Supplying a batch that contains repeated `KeyId` values silently inserts multiple leaf nodes sharing the same key, corrupting the `block_status_cache` index and producing a Merkle root that commits to phantom state. The function is exposed as a Python binding (`py_batch_insert`) and is called with caller-supplied data in the DataLayer protocol.

---

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (lines 578–603) only calls `self.insert()` for the last two items when the tree has ≤ 1 leaf. All remaining items bypass this check and are written directly:

```rust
for ((key, value), hash) in keys_values_hashes {   // no duplicate check
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` (line 1025) calls `block_status_cache.add_leaf()`, which does a plain `HashMap::insert` — silently overwriting the previous index for a duplicate key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.key_to_index.insert(leaf.key, index);          // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

The result is that the blob contains two (or more) leaf nodes with the same `KeyId`, while `block_status_cache.key_to_index` only tracks the last-written one. The Merkle root is then computed over all physical leaf nodes — including the phantom duplicates — producing a root that does not correspond to any valid key-value mapping.

The Python binding `py_batch_insert` (line 1503) passes the caller-supplied list directly to `batch_insert` with no pre-deduplication:

```rust
self.batch_insert(zip(keys_values, hashes).collect())?;
```

---

### Impact Explanation

After a `batch_insert` with duplicate keys:

1. **Corrupted root hash**: `calculate_lazy_hashes()` propagates hashes over a tree containing phantom leaf nodes. The committed root does not represent any valid key-value set.
2. **Forged inclusion proofs**: `get_proof_of_inclusion` resolves a key through `block_status_cache`, which points to only one of the duplicate leaves. The proof path is built over a tree whose root was computed with the other duplicate included, so the proof verifies against a root that encodes state the tree does not actually hold.
3. **Exclusion proof bypass**: A key that was "deleted" by being overwritten in the cache still physically exists in the blob and contributes to the root, so exclusion proofs for that key are invalid.
4. **`check_integrity` failure**: The integrity check at line 862 compares `leaf_count` (physical traversal) against `key_to_index_cache_length` (HashMap size). With duplicates, `leaf_count > cache_length`, triggering `IntegrityKeyToIndexCacheLength` — but only if `check_integrity` is called. Normal DataLayer operation does not call it on every mutation.

This matches: **High — DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is the primary bulk-load path for the DataLayer and is called directly from Python (e.g., `tests/test_datalayer.py` lines 334, 520–521). Any caller that does not pre-deduplicate its key list — including code that aggregates updates from multiple sources — can trigger this silently. No error is returned; the function succeeds and returns `Ok(())`.

---

### Recommendation

Add a duplicate-key check inside the fast path of `batch_insert`, before writing to the blob. The minimal fix is to check `block_status_cache.contains_key(key)` (and `contains_leaf_hash(hash)`) for every item in the loop, mirroring the guard already present in `insert()`. Alternatively, collect all keys into a temporary `HashSet` before the loop and reject the batch if any duplicate is detected, consistent with the report's recommendation to enforce uniqueness in the input collection.

import hashlib

def h(n: int) -> bytes32:
    return bytes32(hashlib.sha256(n.to_bytes(8, "big")).digest())

blob = MerkleBlob(blob=bytearray())

# Batch with a duplicate KeyId(1) — third item onward bypasses the check

blob.batch_insert(keys_values, hashes)   # succeeds, no error
blob.calculate_lazy_hashes()

assert len(kv) == 3          # cache sees 3 unique keys

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
