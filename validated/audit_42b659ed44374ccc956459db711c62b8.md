### Title
Missing Length Check Between `pks` and `msgs` in `BlsCache::py_aggregate_verify` Enables Silent Signature Verification Bypass — (`File: crates/chia-bls/src/bls_cache.rs`)

### Summary
`BlsCache::py_aggregate_verify` (the Python binding for `BLSCache.aggregate_verify`) accepts two separate lists — `pks` and `msgs` — and zips them without checking that their lengths are equal. Rust's `Iterator::zip` silently truncates to the shorter iterator. When `pks` is longer than `msgs`, the trailing public keys are never paired with any message and are silently dropped from the aggregate verification. This can cause the function to return `true` for a signature that does not cover all intended `(pk, msg)` pairs, constituting a signature validation bypass at the Python binding boundary. The same flaw exists in `py_evict`, where it causes silent incomplete cache eviction.

### Finding Description

`py_aggregate_verify` in `crates/chia-bls/src/bls_cache.rs` collects `pks` and `msgs` into two independent `Vec`s and then zips them:

```rust
Ok(self.aggregate_verify(pks.into_iter().zip(msgs), sig))
``` [1](#0-0) 

There is no guard that `pks.len() == msgs.len()`. Rust's `zip` stops at the shorter iterator, silently discarding the excess elements of the longer one. If a caller passes `N` public keys but only `M < N` messages, only `M` `(pk, msg)` pairs are verified. The `N - M` trailing public keys are never checked.

The identical pattern appears in `py_evict`:

```rust
self.evict(pks.into_iter().zip(msgs));
``` [2](#0-1) 

This is directly inconsistent with `AugSchemeMPL::aggregate_verify` in `wheel/src/api.rs`, which **does** perform the length check before zipping:

```rust
if pks.len() != msgs.len() {
    return Err(PyRuntimeError::new_err(
        "aggregate_verify expects the same number of public keys as messages",
    ));
}
``` [3](#0-2) 

The two Python-exposed aggregate-verify entry points behave differently for the same class of input, creating a cross-language disagreement in a consensus-critical operation.

### Impact Explanation

**Signature validation bypass.** If `pks` has more entries than `msgs`, the extra public keys are silently dropped. The aggregate signature is then verified against only the truncated set of `(pk, msg)` pairs. A signature that is valid for only a subset of the required signers can be accepted as valid for the full set. In the context of Chia spend bundle validation, this means a spend bundle whose aggregate signature does not cover all required AGG_SIG conditions could pass mempool pre-validation when `BLSCache.aggregate_verify` is used with mismatched list lengths.

For `py_evict`: if `pks` is longer than `msgs`, the trailing cache entries are not evicted. Stale GT pairings remain in the cache and will be reused in future `aggregate_verify` calls, potentially causing a cached-but-invalid pairing to contribute to a future verification result.

### Likelihood Explanation

The Python full node constructs `pks` and `msgs` lists from spend bundle conditions. Under normal operation these lists are the same length. However:

1. The API contract is not enforced — any caller that passes mismatched lists gets a silently wrong result with no error.
2. The inconsistency with `AugSchemeMPL.aggregate_verify` (which does enforce the check) means developers may not expect this behavior from `BLSCache.aggregate_verify`.
3. Any future refactor or bug in the Python caller that produces mismatched lists will silently bypass signature verification rather than raising an error.

The attacker-controlled entry path is: submit a crafted spend bundle to the mempool. If the Python code that extracts `(pk, msg)` pairs produces lists of unequal length (e.g., due to a secondary parsing bug), `BLSCache.aggregate_verify` will silently verify only the shorter prefix, potentially accepting an invalid aggregate signature.

### Recommendation

Add a length equality check in both `py_aggregate_verify` and `py_evict` before zipping, matching the pattern already used in `AugSchemeMPL::aggregate_verify`:

```rust
if pks.len() != msgs.len() {
    return Err(PyValueError::new_err(
        "aggregate_verify requires pks and msgs to have the same length",
    ));
}
``` [1](#0-0) [2](#0-1) 

### Proof of Concept

```python
from chia_rs import BLSCache, G1Element, G2Element, AugSchemeMPL, PrivateKey

seed = b"\x01" * 32
sk1 = AugSchemeMPL.key_gen(seed)
sk2 = AugSchemeMPL.key_gen(b"\x02" * 32)
pk1 = sk1.get_g1()
pk2 = sk2.get_g1()

msg1 = b"hello"
# Sign only msg1 with sk1 — sk2 never signs anything
sig = AugSchemeMPL.sign(sk1, msg1)

cache = BLSCache()

# Pass 2 public keys but only 1 message.
# zip truncates: only (pk1, msg1) is verified.
# pk2 is silently dropped. The signature is valid for (pk1, msg1) only,
# but the caller intended to verify both pk1 and pk2.
result = cache.aggregate_verify([pk1, pk2], [msg1], sig)
print(result)  # True — pk2's required signature was never checked

# Contrast: AugSchemeMPL.aggregate_verify raises an error for the same input
try:
    AugSchemeMPL.aggregate_verify([pk1, pk2], [msg1], sig)
except RuntimeError as e:
    print(e)  # "aggregate_verify expects the same number of public keys as messages"
