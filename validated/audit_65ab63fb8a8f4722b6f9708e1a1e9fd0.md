### Title
`MerkleBlob::batch_insert` Skips Duplicate Key/Hash Validation, Corrupting DataLayer Tree Root - (File: crates/chia-datalayer/src/merkle/blob.rs)

---

### Summary

`MerkleBlob::batch_insert` performs duplicate-key and duplicate-hash validation only for the last two items in the input batch (which are routed through the guarded `insert()` path). All remaining items bypass both checks and are written directly into the blob, allowing duplicate `KeyId` or `Hash` values to be silently inserted. This corrupts the tree structure, produces an incorrect Merkle root, and invalidates all subsequent proofs of inclusion.

---

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (lines 570–657) takes a different code path for all but the last two items. When `leaf_count <= 1`, it pops the last two entries from the vector and routes them through `insert()`. Every remaining entry is then processed in the fast-path loop (lines 587–603) that calls `get_new_index()` and `insert_entry_to_blob()` directly — with no duplicate check at all:

```rust
for ((key, value), hash) in keys_values_hashes {   // no guard here
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which silently overwrites the existing `key_to_index` and `leaf_hash_to_index` entries via `HashMap::insert` (lines 191–192):

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.key_to_index.insert(leaf.key, index);       // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index); // silent overwrite
}
```

The result is a blob that contains two distinct leaf blocks sharing the same `KeyId` (or `Hash`), while the cache tracks only the later one. The tree's internal nodes are then built over this corrupted leaf set, producing a root hash that does not correspond to any valid Merkle tree state.

---

### Impact Explanation

- The DataLayer Merkle root stored on-chain is computed from a tree whose leaf set contains duplicate keys. Any proof of inclusion generated from this root is structurally invalid.
- A verifier that recomputes the root from the proof will obtain a different hash than the stored root, causing inclusion proofs to fail for legitimately inserted keys.
- Conversely, a proof constructed over the duplicate leaf can be presented as valid against the corrupted root, allowing an untrusted party to prove inclusion of a key-value pair that was never legitimately committed.
- `check_integrity` will detect the inconsistency (`leaf_count != key_to_index_cache_length`) and panic/error, but only after the corruption has already been written.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion API exposed to Python callers (`py_batch_insert`, line 1503) and used in delta processing. Any DataLayer client that submits a batch containing a repeated `KeyId` — whether by accident or by a malicious peer supplying crafted delta data — triggers the corruption. The attacker-controlled entry path is the `batch_insert` public API, reachable without any privileged role.

---

### Recommendation

Add duplicate-key and duplicate-hash guards at the top of the fast-path loop in `batch_insert`, mirroring the checks in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path code
}
```

Additionally, validate uniqueness within the batch itself before any insertion begins, to catch intra-batch duplicates before any partial state is written.

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }
