### Title
Missing Pre-Transfer Normalization Check Causes Permanent Fund Lock - (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge contract burns/locks user tokens in `init_transfer_internal` before validating that the decimal-normalized transfer amount is non-zero. When a user initiates a transfer where `amount - fee < 10^(origin_decimals - decimals)`, the floor-division in `normalize_amount` produces zero, causing `sign_transfer` to permanently revert. With no cancellation path, the locked/burned tokens are irrecoverable.

### Finding Description

`normalize_amount` uses integer floor division to scale a NEAR-native token amount down to the destination chain's decimal precision: [1](#0-0) 

When `origin_decimals > decimals` (e.g., a NEAR token with 24 decimals bridging to an EVM chain with 18 decimals, giving `diff_decimals = 6`), any `amount - fee` smaller than `10^6` (i.e., `1_000_000`) normalizes to zero.

The only validation in `init_transfer` before tokens are committed is: [2](#0-1) 

This only enforces `fee < amount`, not that `normalize_amount(amount - fee) > 0`. Immediately after, `init_transfer_internal` burns or locks the **full** `amount` (including fee): [3](#0-2) 

Later, when a relayer calls `sign_transfer`, the normalization is computed and a hard guard fires: [4](#0-3) 

Because `amount_to_transfer == 0`, `sign_transfer` always panics. No MPC signature is ever produced, so no `FinTransfer` event is emitted on the destination chain, so `claim_fee` (which requires a `FinTransfer` proof) can never be called. The only function that removes a pending transfer is `remove_transfer_message`, called exclusively inside `claim_fee_callback`: [5](#0-4) 

There is no admin escape hatch, no cancel function, and no DAO-accessible removal path for stuck pending transfers. The tokens are permanently frozen.

### Impact Explanation

A user who transfers `amount = 1_000_001` with `fee = 1_000_000` to an EVM chain (24â†’18 decimal token) has `amount_without_fee = 1`. `normalize_amount(1, {origin:24, dest:18}) = 1 / 10^6 = 0`. The full `1_000_001` units are burned/locked on NEAR at `init_transfer_internal`, but `sign_transfer` will revert forever with `ERR_INVALID_AMOUNT_TO_TRANSFER`. The transfer sits in `pending_transfers` indefinitely with no recovery mechanism.

This matches: **Critical â€” Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows.**

### Likelihood Explanation

Any NEAR token registered with `origin_decimals > decimals` (the standard case for NEAR-native tokens bridging to EVM, e.g., 24 â†’ 18) is affected. A user only needs to submit `amount - fee < 10^(origin_decimals - decimals)`. This is a realistic mistake for users unfamiliar with decimal normalization, and the protocol provides no warning or rejection at deposit time. The entry point is the fully public `ft_transfer_call` â†’ `ft_on_transfer` path, requiring no special role.

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) **before** burning or locking tokens. Specifically, after constructing `transfer_message`, compute `normalize_amount(amount_without_fee, decimals)` and require it to be greater than zero. This mirrors the guard already present in `sign_transfer` but must be enforced at deposit time so tokens are never committed to an unserviceable transfer:

```rust
// In init_transfer, after fee validation and before init_transfer_internal:
let token_address = self.get_token_address(destination_chain, token_id);
if let Some(decimals) = self.token_decimals.get(&token_address) {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee().expect("fee < amount"),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

Alternatively, enforce a minimum deposit amount per token based on its registered decimal difference.

### Proof of Concept

Assume a NEAR-native token registered with `origin_decimals = 24`, `decimals = 18` (diff = 6, divisor = 1_000_000).

1. User calls `ft_transfer_call` with `amount = 1_000_001`, `fee = 1_000_000`, recipient on EVM.
2. `init_transfer` passes the only guard: `fee (1_000_000) < amount (1_000_001)` âś“.
3. `init_transfer_internal` burns `1_000_001` tokens and emits `InitTransferEvent`.
4. Relayer calls `sign_transfer`:
   - `amount_without_fee() = 1_000_001 - 1_000_000 = 1`
   - `normalize_amount(1, {24,18}) = 1 / 1_000_000 = 0`
   - `require!(0 > 0, ...)` â†’ **panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`**
5. No MPC signature is produced. No `FinTransfer` on EVM. `claim_fee` cannot be called.
6. `pending_transfers` retains the entry forever. The `1_000_001` tokens are permanently lost.

        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);

    }
