### Title
Missing Declared-Class Validation in `replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract is actually declared in `contract_class_changes`. Any contract can replace its own class hash with an arbitrary, undeclared value. Because the OS cannot resolve an undeclared class hash at execution time, all future calls to that contract will fail, permanently freezing any funds held within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The developer-acknowledged TODO at line 898 confirms the missing check: `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.` [2](#0-1) 

The OS maintains two separate dictionaries:
- `contract_state_changes`: maps contract address → `StateEntry` (which includes `class_hash`)
- `contract_class_changes`: maps class hash → compiled class hash (the class registry) [3](#0-2) 

`execute_replace_class` updates `contract_state_changes` with the new class hash but never checks that the new class hash exists in `contract_class_changes`. This creates an accounting inconsistency: the contract state records a class hash that has no corresponding compiled class entry, directly analogous to the reported bug where the token balance becomes inconsistent with the LMCV debt record.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract's class hash is set to an undeclared value, the OS cannot resolve the compiled class for any future call to that contract. Every subsequent invocation will fail at class resolution, making the contract permanently uncallable. Any ERC-20 tokens, ETH, STRK, or other assets held in the contract's storage become permanently inaccessible — no withdrawal, transfer, or recovery function can be executed.

The state inconsistency is committed to the proven state update output: [4](#0-3) 

Once the block is proven and finalized on L1, the corrupted class hash is part of the canonical state and cannot be reversed.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is callable by any contract from within its own execution context. An attacker can:

1. Deploy a contract that accepts user deposits (e.g., a vault, multi-sig, or staking contract).
2. Include a function that calls `replace_class` with an arbitrary felt value not present in the class registry.
3. After accumulating user funds, invoke that function.
4. The OS accepts the syscall without validation, updating the contract's class hash to the invalid value.
5. All future calls to the contract revert at class resolution; funds are permanently frozen.

No privileged role, leaked key, or sequencer cooperation is required beyond normal transaction submission. The attacker is an ordinary contract deployer — a public protocol input user.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, add a Cairo-level assertion that the class hash exists in `contract_class_changes`. Specifically, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the returned compiled class hash is non-zero. This mirrors the existing pattern used in `execute_deploy` and `deploy_contract`, which look up the class before constructing the execution context.

The TODO comment at line 898 already identifies this gap — it must be resolved before the OS is considered production-safe.

---

### Proof of Concept

1. Attacker deploys contract `VaultTrap` with two entry points:
   - `deposit()`: accepts STRK/ETH from users, stores balances in contract storage.
   - `brick()`: calls `replace_class(0xdeadbeef)` — an arbitrary felt not declared in any block.

2. Users deposit funds; `VaultTrap` accumulates assets.

3. Attacker calls `brick()`. The OS executes `execute_replace_class`:
   - `request.class_hash = 0xdeadbeef`
   - No check against `contract_class_changes`
   - `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`

4. Block is proven. The state commitment now records `VaultTrap.class_hash = 0xdeadbeef`.

5. Any user calls `withdraw()` on `VaultTrap`. The OS looks up class `0xdeadbeef` in the compiled class registry — it does not exist. Execution fails. Funds are permanently frozen. [5](#0-4)

func execute_replace_class{

    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

    // Reduce gas.

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
