### Title
`process_fin_transfer_to_other_chain` Does Not Unlock `locked_tokens` for Destination Chain After Fast-Transfer Relayer Reimbursement — (File: near/omni-bridge/src/lib.rs)

---

### Summary

When a fast transfer to a non-NEAR chain is finalized, the NEAR bridge reimburses the relayer on NEAR but never decrements `locked_tokens` for the destination chain by `amount_without_fee`. This permanently inflates the destination chain's locked-token accounting after every such fast-transfer cycle, breaking bridge collateralization.

---

### Finding Description

**Step 1 — Relayer executes `fast_fin_transfer_to_other_chain`:**

The relayer pre-pays the user on the destination chain from their own funds. The bridge records this obligation by calling:

```rust
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,   // ← locked_tokens[dest][token] += amount_without_fee
);
``` [1](#0-0) 

**Step 2 — Original transfer finalized via `process_fin_transfer_to_other_chain`:**

The bridge:
1. Unlocks `amount` from the origin chain — correct.
2. Locks only `fee` on the destination chain — correct.
3. Sends `amount_without_fee` to the relayer **on NEAR** — correct.
4. **Never calls `unlock_tokens_if_needed(destination_chain, token, amount_without_fee)`** — missing. [2](#0-1) 

After the full cycle the net effect on `locked_tokens` is:

| Entry | Change | Expected |
|---|---|---|
| `locked_tokens[origin][token]` | `−amount` | correct |
| `locked_tokens[dest][token]` | `+amount_without_fee + fee` | should be `+fee` only |

The `amount_without_fee` increment from Step 1 is never reversed, so `locked_tokens[dest][token]` is permanently inflated by `amount_without_fee` per fast-transfer cycle.

The `unlock_tokens` guard that enforces `available >= amount` is the mechanism that prevents the bridge from releasing more tokens than are locked: [3](#0-2) 

With an inflated counter, this guard is weaker than intended: future `unlock_tokens_if_needed(dest, token, X)` calls will succeed for values of `X` that exceed the tokens actually locked on the destination chain.

---

### Impact Explanation

**High — accounting corruption that breaks bridge collateralization.**

After N fast-transfer-to-other-chain cycles for a given `(dest_chain, token)` pair, `locked_tokens[dest][token]` exceeds the true locked supply by `N × amount_without_fee`. The bridge will then permit finalization of inbound transfers from the destination chain whose cumulative unlock amount exceeds what is actually locked there. For non-deployed (custodial) tokens this means the NEAR bridge's accounting no longer reflects the actual collateral held in the destination-chain vault, breaking the 1:1 backing invariant. For deployed (mint/burn) tokens the bridge's supply tracking diverges from the actual circulating supply on the destination chain.

---

### Likelihood Explanation

**Medium-High.** The fast-transfer-to-other-chain path is a normal, incentivized protocol operation. Any relayer executing a fast transfer to a non-NEAR chain triggers the bug. The inflation compounds with every such transfer, so the accounting drift grows monotonically over time with ordinary protocol usage.

---

### Recommendation

In `process_fin_transfer_to_other_chain`, when a fast transfer is detected and the relayer is reimbursed on NEAR, also call:

```rust
self.unlock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token,
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
);
```

before or after `send_tokens` to the relayer, so that the destination chain's locked-token counter is correctly decremented to reflect that the bridge's obligation there has been settled.

---

### Proof of Concept

Assume token `T` originates from Ethereum; `amount = 1000`, `fee = 10`, `amount_without_fee = 990`.

1. User bridges 1000 T from Ethereum → NEAR: `locked_tokens[Eth][T] = 1000`.
2. User initiates transfer of 1000 T from NEAR → Solana.
3. Relayer executes fast transfer (`fast_fin_transfer_to_other_chain`):
   - Burns 990 T on NEAR.
   - `lock_tokens_if_needed(Sol, T, 990)` → `locked_tokens[Sol][T] = 990`.
4. Original transfer finalized (`process_fin_transfer_to_other_chain`):
   - `unlock_tokens_if_needed(Near, T, 1000)` → `locked_tokens[Near][T] -= 1000`.
   - `lock_tokens_if_needed(Sol, T, 10)` → `locked_tokens[Sol][T] = 1000`.
   - Relayer receives 990 T on NEAR.
   - **`unlock_tokens_if_needed(Sol, T, 990)` is never called.**
5. Final state: `locked_tokens[Sol][T] = 1000`, but only 10 T (the fee) are actually locked/owed on Solana. The counter is inflated by 990.
6. The bridge now permits up to 1000 T to be unlocked from Solana in future inbound transfers, even though only 10 T are actually there, breaking the collateralization invariant.

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
