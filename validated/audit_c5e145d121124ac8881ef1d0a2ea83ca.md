### Title
Native Token Fee Permanently Locked in Bridge Contract When Transfer Is Never Finalized — (`near/omni-bridge/src/lib.rs`)

### Summary
When a NEAR-side user initiates a bridge transfer with a non-zero `native_token_fee`, the fee is immediately deducted from their storage balance inside the bridge contract. If the transfer is never finalized on the destination chain (and `claim_fee` is never called), there is no mechanism for the user to recover the locked NEAR. The `storage_unregister` path only refunds the `available` balance, which no longer includes the consumed native fee.

### Finding Description

In `init_transfer_internal`, the native fee is folded into the required storage deduction: [1](#0-0) 

```rust
let required_storage_balance = self
    .add_transfer_message(transfer_message.clone(), storage_owner.clone())
    .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));
```

`try_update_storage_balance` then deducts this entire amount — including the native fee — from the user's `available` balance in `accounts_balances`. [2](#0-1) 

The native fee is only ever paid out when a relayer calls `claim_fee`, which triggers `send_fee_internal`: [3](#0-2) 

```rust
} else if origin_chain == ChainKind::Near {
    Promise::new(fee_recipient.clone())
        .transfer(NearToken::from_yoctonear(transfer_message.fee.native_fee.0))
        .detach();
```

If `claim_fee` is never called (relayer abandons the transfer, destination chain is permanently paused, or the bridge is paused indefinitely), the native fee remains consumed from the user's storage balance with no recovery path.

`storage_unregister` only refunds `required_balance_for_account() + storage.available`: [4](#0-3) 

```rust
let refund = self
    .required_balance_for_account()
    .saturating_add(storage.available);
Promise::new(account_id).transfer(refund).detach();
```

Because the native fee was already subtracted from `available` at transfer creation time, it is not included in this refund — even when `force = true` is passed. There is no `cancel_transfer` or equivalent function in the production code that would restore the native fee to the user's available balance.

### Impact Explanation

A user who initiates a NEAR-to-EVM/Solana/StarkNet transfer with `native_token_fee > 0` and whose transfer is never finalized loses the native fee permanently. The NEAR tokens are held by the bridge contract and cannot be reclaimed by the user through any on-chain path in the current implementation. This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation

The scenario requires a transfer to remain in `pending_transfers` indefinitely. Realistic triggers include:
- A relayer going permanently offline after the user's transfer is signed but before `claim_fee` is submitted.
- The destination chain being paused or decommissioned after `init_transfer` is emitted.
- The bridge being paused (via the `pause` modifier) for an extended period, preventing `claim_fee` from executing.

Any ordinary bridge user who sets a non-zero `native_token_fee` is exposed. The fee can be substantial (tests show values of `NearToken::from_near(1)` or more).

### Recommendation

Add a user-callable `cancel_transfer` (or `withdraw_native_fee`) function that:
1. Verifies the caller is the transfer owner.
2. Removes the pending transfer from `pending_transfers`.
3. Restores the native fee to the user's available storage balance (or transfers it directly back to the user).
4. Reverts any token lock/burn actions associated with the transfer.

Alternatively, restore the native fee to `available` inside `remove_transfer_message_without_refund` so that any code path that removes a pending transfer also returns the native fee to the user.

### Proof of Concept

1. User calls `ft_transfer_call` on a NEAR token contract with `msg` containing `InitTransferMsg { native_token_fee: U128(1_000_000_000_000_000_000_000_000), ... }` (1 NEAR).
2. `init_transfer_internal` deducts `storage_cost + 1 NEAR` from the user's `accounts_balances.available`.
3. The transfer is recorded in `pending_transfers`.
4. The relayer goes offline; `claim_fee` is never called.
5. User calls `storage_unregister(force: Some(true))`.
6. Refund = `required_balance_for_account() + storage.available` — the 1 NEAR native fee is **not** included.
7. The 1 NEAR remains in the bridge contract with no on-chain recovery path. [5](#0-4) [6](#0-5) [7](#0-6)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    #[payable]
    pub fn storage_unregister(&mut self, force: Option<bool>) -> bool {
        assert_one_yocto();
        let account_id = env::predecessor_account_id();
        let Some(storage) = self.storage_balance_of(&account_id) else {
            return false;
        };

        if !force.unwrap_or_default() {
            require!(
                storage.total.saturating_sub(storage.available)
                    == self.required_balance_for_account(),
                BridgeError::StoragePendingTransfers.as_ref()
            );
        }

        self.accounts_balances.remove(&account_id);

        let refund = self
            .required_balance_for_account()
            .saturating_add(storage.available);
        Promise::new(account_id).transfer(refund).detach();
        true
    }
