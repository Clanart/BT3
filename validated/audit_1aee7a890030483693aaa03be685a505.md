### Title
`deployToken` MPC Signature Lacks Chain ID Binding, Enabling Cross-Chain Replay to Force Unauthorized Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deployToken` / `deploy_token` operation on every supported destination chain (EVM, StarkNet, Solana) hashes the metadata payload **without including a chain identifier**. Because `finTransfer` correctly binds its hash to `omniBridgeChainId`, the omission in `deployToken` is a deliberate asymmetry that creates a cross-chain replay surface: a single valid MPC signature authorizing token deployment on one chain is cryptographically valid on every other chain that shares the same `nearBridgeDerivedAddress` key format.

---

### Finding Description

**`finTransfer` — chain-bound (correct)**

`OmniBridge.sol` lines 289–309 encode `omniBridgeChainId` twice (for the token address chain and the recipient chain) before hashing:

```solidity
bytes1(omniBridgeChainId),          // line 294
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),          // line 297
Borsh.encodeAddress(payload.recipient),
``` [1](#0-0) 

**`deployToken` — NOT chain-bound (vulnerable)**

`OmniBridge.sol` lines 142–149 hash only `PayloadType.Metadata + token + name + symbol + decimals` — no chain ID:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

The identical chain-ID-free Borsh layout is reproduced on StarkNet in `MetadataPayloadTrait::to_borsh()`:

```cairo
borsh_bytes.append_byte(PayloadType::Metadata.into()); // 0x01
borsh_bytes.append(@borsh::encode_byte_array(self.token));
borsh_bytes.append(@borsh::encode_byte_array(self.name));
borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
borsh_bytes.append_byte(*self.decimals);
``` [3](#0-2) 

And on Solana in `DeployTokenPayload::serialize_for_near()`:

```rust
IncomingMessageType::Metadata.serialize(&mut writer)?;
self.serialize(&mut writer)?; // token, name, symbol, decimals — no chain ID
``` [4](#0-3) 

**Key shared-key observation.** All EVM chains and StarkNet verify the signature against the same Ethereum-address-format key (`nearBridgeDerivedAddress` / `omni_bridge_derived_address`). StarkNet uses `verify_eth_signature` with an `EthAddress`: [5](#0-4) 

Because the Borsh encoding is byte-for-byte identical across EVM and StarkNet, and the verifying key is the same Ethereum address on both, a signature produced by the MPC network for `deployToken` on Ethereum is a valid signature for `deployToken` on Arbitrum, Base, Polygon, BNB, and StarkNet — and vice versa.

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on any one chain (e.g., Ethereum mainnet) can immediately replay the same `(signatureData, MetadataPayload)` call on every other EVM chain and StarkNet where the token has not yet been deployed. Concrete consequences:

1. **Unauthorized token deployment on unready chains.** The bridge operator may intentionally stage rollouts (deploy on Ethereum first, then Arbitrum weeks later after liquidity is arranged). The attacker collapses this to a single block, deploying the bridge token on all chains simultaneously without operator consent.

2. **Permanent blocking of legitimate deployment.** Once the replayed deployment succeeds, `nearToEthToken[metadata.token]` is set and `isBridgeToken[...]` is `true`. The `ERR_TOKEN_EXIST` guard prevents any future legitimate deployment attempt on that chain. The operator cannot re-deploy with a corrected configuration (e.g., different `tokenImplementationAddress` or custom minter).

3. **Misdirected bridge flows.** If the operator later registers a custom minter or multi-token mapping for the same NEAR token ID on that chain (via `addCustomToken`), the replayed bridge token already occupies the `nearToEthToken` slot, making `addCustomToken` silently inconsistent or blocked.

This matches the allowed High impact: **signature/MPC verification bypass enabling unauthorized token deployment**.

---

### Likelihood Explanation

- All `deployToken` transactions are public on-chain. Any observer can extract `signatureData` and `MetadataPayload` from a confirmed transaction on one chain and submit them to another chain's `OmniBridge` contract.
- No privileged access, leaked keys, or colluding MPC signers are required — the attacker only needs a funded wallet on the target chain.
- The attack is executable in a single transaction with no preconditions beyond the existence of a valid `deployToken` on any one chain.

---

### Recommendation

Include `omniBridgeChainId` in the Borsh-encoded metadata hash for `deployToken`, mirroring the pattern already used in `finTransfer`. On EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same fix to `MetadataPayloadTrait::to_borsh()` in `starknet/src/bridge_types.cairo` and `DeployTokenPayload::serialize_for_near()` in `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`. Ensure the NEAR MPC signing side includes the destination chain ID in the metadata payload it signs.

---

### Proof of Concept

1. Observe a confirmed `deployToken` call on Ethereum for token `"usdc.near"` (name `"USD Coin"`, symbol `"USDC"`, decimals `6`) with signature `sig_eth`.
2. Confirm `nearToEthToken["usdc.near"] == address(0)` on Arbitrum (token not yet deployed there).
3. Call `OmniBridge.deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on the Arbitrum `OmniBridge` contract.
4. `ECDSA.recover(keccak256(borshEncoded), sig_eth)` returns `nearBridgeDerivedAddress` — the same address on both chains — so the signature check passes.
5. A new `BridgeToken` proxy is deployed on Arbitrum and `nearToEthToken["usdc.near"]` is set.
6. The bridge operator's subsequent attempt to deploy `"usdc.near"` on Arbitrum with a custom minter reverts with `ERR_TOKEN_EXIST`.
