### Title
`FakeProver` Placeholder Bypasses All Proof Verification in Unpaused `ENearProxy.finaliseNearToEthTransfer`, Enabling Unauthorized eNear Minting — (`File: evm/src/eNear/contracts/ENearProxy.sol`, `evm/src/eNear/contracts/FakeProver.sol`)

### Summary

`ENearProxy` is deployed with `FakeProver` as its prover, a contract that unconditionally returns `true` for any proof input. The `finaliseNearToEthTransfer` function on `ENearProxy` is guarded only by `whenNotPaused(PAUSED_LEGACY_FIN_TRANSFER)`, but `initialize()` never sets that pause flag — `__Pausable_init()` explicitly initializes `_pausedFlags = 0`. Any unprivileged caller can therefore invoke `ENearProxy.finaliseNearToEthTransfer` with attacker-crafted `proofData`, bypass proof verification entirely via `FakeProver`, and cause the eNear contract to mint an arbitrary amount of eNear tokens to any address.

### Finding Description

**Root cause — `FakeProver` is a proof-verification placeholder:**

`FakeProver.proveOutcome()` accepts any input and always returns `true`:

```solidity
// evm/src/eNear/contracts/FakeProver.sol
contract FakeProver is INearProver {
    function proveOutcome(bytes calldata, uint64) external pure returns (bool) {
        return true;
    }
}
``` [1](#0-0) 

This is the intended prover for the deployed `ENearProxy`, as confirmed by the deployment script and e2e pipeline:

```typescript
// evm/src/eNear/scripts.ts — deploy-e-near-proxy task
const eNearProxy = await upgrades.deployProxy(
  eNearProxyContract,
  [taskArgs.enear, taskArgs.prover, nearConnector, 0, taskArgs.admin],
  ...
)
``` [2](#0-1) 

The e2e snakefile deploys `FakeProver` first and passes its address as `--prover` to `deploy-e-near-proxy`: [3](#0-2) 

**Root cause — `PAUSED_LEGACY_FIN_TRANSFER` is never set in `initialize()`:**

`SelectivePausableUpgradable.__Pausable_init_unchained()` sets `_pausedFlags = 0`: [4](#0-3) 

`ENearProxy.initialize()` calls `__Pausable_init()` but never calls `_pause(PAUSED_LEGACY_FIN_TRANSFER)`: [5](#0-4) 

**The vulnerable function:**

```solidity
function finaliseNearToEthTransfer(
    bytes memory proofData,
    uint64 proofBlockHeight
) external whenNotPaused(PAUSED_LEGACY_FIN_TRANSFER) {
    require(
        prover.proveOutcome(proofData, proofBlockHeight),
        "Proof should be valid"
    );
    eNear.finaliseNearToEthTransfer(proofData, proofBlockHeight);
}
``` [6](#0-5) 

- `external` — no role restriction, callable by anyone
- `whenNotPaused(PAUSED_LEGACY_FIN_TRANSFER)` — passes because flag is `0` by default
- `prover.proveOutcome(...)` — `FakeProver` returns `true` for any input
- `eNear.finaliseNearToEthTransfer(proofData, ...)` — eNear's own prover is also replaced with `FakeProver` per the README, so it too accepts any proof

**Proof data format is publicly documented in `ENearProxy.mint()`:**

The `mint()` function (restricted to `MINTER_ROLE`) shows exactly how to craft valid-looking `proofData`: [7](#0-6) 

An attacker can replicate this encoding with their own `to` address and arbitrary `amount`.

### Impact Explanation

**Critical — Direct unauthorized mint of bridged eNear tokens on Ethereum.**

Any unprivileged user can mint an unbounded quantity of eNear (the bridged NEAR token on Ethereum, deployed at `0x85F17Cf997934a597031b2E18a9aB6ebD4B9f6a4` on mainnet) to any address. This directly inflates eNear supply without any corresponding NEAR being locked on the NEAR side, breaking bridge collateralization and enabling theft of real NEAR value from legitimate eNear holders via market sell pressure or redemption.

### Likelihood Explanation

**High.** The conditions are met by the intended production deployment:
1. `FakeProver` is the designated prover — it is deployed and passed to `ENearProxy` by the official deployment scripts.
2. `PAUSED_LEGACY_FIN_TRANSFER` is `0` after `initialize()` — no additional admin action is required to leave the function open; it is open by default.
3. The eNear contract's own prover is also replaced with `FakeProver` per the README, removing the second layer of defense.
4. The attack requires no privileged access, no leaked keys, and no special conditions — a single transaction suffices.

The README states the intent to pause `finaliseNearToEthTransfer` on the eNear contract via `adminPause`, but this is a separate operational step not enforced by `initialize()`, creating a permanent code-level gap on the proxy side and a deployment-window gap on the eNear side.

### Recommendation

1. **Pause by default in `initialize()`**: Add `_pause(PAUSED_LEGACY_FIN_TRANSFER)` to `ENearProxy.initialize()` so the legacy path is disabled from the moment of deployment, with no reliance on a post-deployment admin call.
2. **Remove or restrict `finaliseNearToEthTransfer` on the proxy**: Since the README states this function should only be callable by `eNearProxy` as admin, the proxy-level `finaliseNearToEthTransfer` should either be removed entirely or restricted to `onlyRole(DEFAULT_ADMIN_ROLE)`.
3. **Replace `FakeProver` with a real prover or a strict revert**: If the legacy proof path is permanently disabled, the prover field is irrelevant; but if it is ever re-enabled, `FakeProver` must not be the active prover.
