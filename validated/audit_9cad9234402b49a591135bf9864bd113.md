### Title
EIP-712 Legacy `Tx` Type Schema Omits `timeout_height`, Enabling Replay of Expired Transactions - (File: `ethereum/eip712/eip712_legacy.go`)

### Summary

The legacy EIP-712 `Tx` type schema deliberately omits the `timeout_height` field. Because EIP-712 only hashes fields declared in the type schema, the signed commitment does not bind to `timeout_height`. An attacker who observes a signed legacy EIP-712 transaction whose `timeout_height` has already passed can strip that field and resubmit the transaction with the original signature intact, bypassing the user's intended expiry constraint and forcing execution of a transaction the user meant to cancel.

### Finding Description

In `ethereum/eip712/eip712_legacy.go`, the `extractMsgTypes` function defines the `Tx` EIP-712 type schema. The `timeout_height` field is explicitly commented out:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id", Type: "string"},
    {Name: "fee", Type: "Fee"},
    {Name: "memo", Type: "string"},
    {Name: "msgs", Type: "Msg[]"},
    {Name: "sequence", Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
```

The same omission exists in `ethereum/eip712/types.go` for the non-legacy path's `createEIP712Types`.

The server-side `VerifySignature` function in `ante/cosmos/eip712.go` reconstructs the Amino sign bytes from the **live transaction**, including whatever `timeout_height` the submitter placed on it:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // reads from the submitted tx
    legacytx.StdFee{Amount: tx.GetFee(), Gas: tx.GetGas()},
    msgs, tx.GetMemo(),
)
```

These bytes are then passed to `eip712.LegacyWrapTxToTypedData`, which parses the JSON and builds the EIP-712 typed data. Because `timeout_height` is absent from the `Tx` type schema, EIP-712's `hashStruct` silently ignores it. The resulting hash is identical regardless of whether `timeout_height` is `0` or any other value.

Critically, the `LegacyEip712SigVerificationDecorator` path (reached via `ExtensionOptionsWeb3Tx`) does **not** call `LegacyGetEIP712BytesForMsg` or `legacyDecodeProtobufSignDoc`, which do reject `TimeoutHeight != 0`. It calls `LegacyWrapTxToTypedData` directly, bypassing those guards entirely.

The `newLegacyCosmosAnteHandlerEip712` ante chain does include `authante.NewTxTimeoutHeightDecorator()`, which enforces the expiry — but only against the `timeout_height` value present in the submitted transaction, not the one the user signed over.

### Impact Explanation

An attacker who observes a signed legacy EIP-712 transaction with `timeout_height = N` in the mempool (or on-chain before it is included) can:

1. Wait until block height > N so the original transaction is rejected by `TxTimeoutHeightDecorator`.
2. Reconstruct the Cosmos SDK transaction bytes with `timeout_height = 0` (removing the expiry).
3. Attach the original `FeePayerSig` from `ExtensionOptionsWeb3Tx` unchanged.
4. Broadcast the modified transaction.
5. `TxTimeoutHeightDecorator` passes (no expiry).
6. `LegacyEip712SigVerificationDecorator` passes (EIP-712 hash is identical).
7. The transaction executes — e.g., a `MsgSend` transfers the user's funds to the attacker-chosen recipient — after the user intended it to have expired.

Because the nonce was never consumed (the original tx expired without execution), the account sequence is still valid, and the replay succeeds. This constitutes unauthorized execution of a fund transfer the user explicitly intended to cancel via `timeout_height`.

### Likelihood Explanation

The legacy EIP-712 path (`ExtensionOptionsWeb3Tx`) is deprecated but remains fully active and reachable via any Cosmos SDK transaction carrying that extension option. Transactions are broadcast publicly; any observer can extract the signed bytes from the mempool. The modification requires only changing one field in the Cosmos SDK tx body — no cryptographic material needs to be forged. The attack is therefore low-effort for any party monitoring the mempool.

### Recommendation

Add `timeout_height` to the `Tx` type schema in both `ethereum/eip712/eip712_legacy.go` (`extractMsgTypes`) and `ethereum/eip712/types.go` (`createEIP712Types`), so the EIP-712 hash commits to it. Alternatively — and more conservatively — reject any legacy EIP-712 transaction whose `timeout_height != 0` inside `VerifySignature` in `ante/cosmos/eip712.go`, mirroring the guard already present in `legacyDecodeProtobufSignDoc`.

### Proof of Concept

1. User constructs a `MsgSend` wrapped in a legacy EIP-712 (`ExtensionOptionsWeb3Tx`) transaction with `timeout_height = 500` and broadcasts it.
2. The transaction sits in the mempool; block 500 passes without inclusion; `TxTimeoutHeightDecorator` rejects it.
3. Attacker copies the transaction, sets `timeout_height = 0` in the `TxBody`, keeps `FeePayerSig` unchanged.
4. Attacker broadcasts the modified transaction at block 501.
5. `TxTimeoutHeightDecorator`: passes (no expiry).
6. `LegacyEip712SigVerificationDecorator` calls `legacytx.StdSignBytes(..., 0, ...)` → `LegacyWrapTxToTypedData` → EIP-712 hash identical to the original → `Ecrecover` recovers the correct fee payer → signature accepted.
7. `MsgSend` executes; user's funds are transferred.

func newLegacyCosmosAnteHandlerEip712(ctx sdk.Context, options HandlerOptions, extra ...sdk.AnteDecorator) sdk.AnteHandler {
	evmParams := options.EvmKeeper.GetParams(ctx)
	feemarketParams := options.FeeMarketKeeper.GetParams(ctx)
	evmDenom := evmParams.EvmDenom
	chainID := options.EvmKeeper.ChainID()
	chainCfg := evmParams.GetChainConfig()
	ethCfg := chainCfg.EthereumConfig(chainID)
	var txFeeChecker authante.TxFeeChecker
	if options.DynamicFeeChecker {
		txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)
	}
	decorators := make([]sdk.AnteDecorator, 0, 15+len(extra))
	decorators = append(decorators,
		cosmos.RejectMessagesDecorator{}, // reject MsgEthereumTxs
		// disable the Msg types that cannot be included on an authz.MsgExec msgs field
		cosmos.NewAuthzLimiterDecorator(options.DisabledAuthzMsgs),
		authante.NewSetUpContextDecorator(),
		authante.NewValidateBasicDecorator(),
		authante.NewTxTimeoutHeightDecorator(),
		cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams),
		authante.NewValidateMemoDecorator(options.AccountKeeper),
		authante.NewConsumeGasForTxSizeDecorator(options.AccountKeeper),
		authante.NewDeductFeeDecorator(options.AccountKeeper, options.BankKeeper, options.FeegrantKeeper, txFeeChecker),
		// SetPubKeyDecorator must be called before all signature verification decorators
		authante.NewSetPubKeyDecorator(options.AccountKeeper),
		authante.NewValidateSigCountDecorator(options.AccountKeeper),
		authante.NewSigGasConsumeDecorator(options.AccountKeeper, options.SigGasConsumer),
		// Note: signature verification uses EIP instead of the cosmos signature validator
		cosmos.NewLegacyEip712SigVerificationDecorator(options.AccountKeeper, options.SignModeHandler),
		authante.NewIncrementSequenceDecorator(options.AccountKeeper),
		ibcante.NewRedundantRelayDecorator(options.IBCKeeper),
	)
	decorators = append(decorators, extra...)
	return sdk.ChainAnteDecorators(decorators...)
