### Title
Native ETH `finTransfer` Always Reverts in `OmniBridgeWormhole` Due to Incorrect `msg.value` Forwarding - (File: `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole.finTransferExtension` forwards the full `msg.value` to `_wormhole.publishMessage` even after the base `finTransfer` logic has already spent `payload.amount` ETH sending native ETH to the recipient. This makes every native-ETH finalization on Wormhole-connected chains (Base, Arbitrum, etc.) revert unconditionally, permanently locking user funds.

---

### Finding Description

`OmniBridge.finTransfer` is `payable`. When `payload.tokenAddress == address(0)` (native ETH), it first sends `payload.amount` ETH to the recipient:

```solidity
// OmniBridge.sol lines 317-322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

After that call, the contract's remaining ETH balance is `msg.value − payload.amount`. Control then passes to `finTransferExtension`, which in `OmniBridgeWormhole` does:

```solidity
// OmniBridgeWormhole.sol lines 109-113
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    messagePayload,
    _consistencyLevel
);
```

`msg.value` here is the **original** call value, not the remaining balance. The contract no longer holds `msg.value` worth of ETH (it already sent `payload.amount` to the recipient), so the EVM will revert with an insufficient-balance error on every native-ETH finalization.

The `initTransferExtension` counterpart correctly avoids this by computing a separate `extensionValue = msg.value − amount − nativeFee` and forwarding only that remainder to Wormhole:

```solidity
// OmniBridgeWormhole.sol line 143
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

`finTransferExtension` has no equivalent adjustment.

---

### Impact Explanation

Any user who bridges native ETH from NEAR to a Wormhole-connected EVM chain (Base, Arbitrum, BNB, Polygon) via `OmniBridgeWormhole` will have their funds permanently locked. The NEAR-side transfer is already signed and the destination nonce is marked used on the first successful call attempt; because the revert happens inside the ETH-send/Wormhole-publish sequence (after `completedTransfers[payload.destinationNonce] = true` is set in storage), a retry with a corrected `msg.value` is impossible — the nonce is consumed on success, but the whole transaction reverts on failure, so the nonce is never actually consumed. However, the relayer cannot fix the underlying arithmetic: no matter what `msg.value` the relayer provides, the contract will always try to forward the full original `msg.value` to Wormhole after having already spent `payload.amount` of it. The transfer is therefore permanently unclaimable.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

Native ETH is explicitly supported in `initTransfer` (`tokenAddress == address(0)` branch) and in `finTransfer`. `OmniBridgeWormhole` is the deployed variant for Base, Arbitrum, BNB, and Polygon — chains where native ETH (or native gas token) transfers are expected. Any user who initiates such a transfer triggers the broken path. No special privileges are required; any bridge user can reach this code path.

---

### Recommendation

In `OmniBridgeWormhole.finTransferExtension`, do not use `msg.value` directly. Instead, pass only the Wormhole message fee (queried via `_wormhole.messageFee()`) or accept an explicit `remainingValue` parameter from `finTransfer` that accounts for any ETH already disbursed to the recipient:

```solidity
// Option A: query the fee directly
_wormhole.publishMessage{value: _wormhole.messageFee()}(
    wormholeNonce, messagePayload, _consistencyLevel
);

// Option B: mirror the initTransfer pattern — compute extensionValue
// before the ETH send and thread it through finTransferExtension
```

The `initTransferExtension` signature already demonstrates the correct pattern: compute the value available for the external call *before* any ETH is disbursed, and pass it explicitly.

---

### Proof of Concept

1. User calls `OmniBridge.initTransfer{value: 1 ether + wormhole_fee}(address(0), 1 ether, 0, 0, recipient, "")` on Base (Wormhole variant). `extensionValue = msg.value − 1 ether − 0 = wormhole_fee`. Wormhole message published correctly. ✓

2. NEAR signs the corresponding `TransferMessagePayload` with `tokenAddress = address(0)`, `amount = 1 ether`.

3. Relayer calls `OmniBridgeWormhole.finTransfer{value: 1 ether + wormhole_fee}(sig, payload)`.
   - Contract balance: `1 ether + wormhole_fee`.
   - `payload.recipient.call{value: 1 ether}("")` succeeds. Contract balance: `wormhole_fee`.
   - `_wormhole.publishMessage{value: 1 ether + wormhole_fee}(...)` — contract only holds `wormhole_fee`. **EVM reverts.**

4. Transaction reverts. Nonce not consumed. Relayer retries — same arithmetic, same revert. User's NEAR-side funds are locked; ETH is never released. [1](#0-0) [2](#0-1) [3](#0-2)
