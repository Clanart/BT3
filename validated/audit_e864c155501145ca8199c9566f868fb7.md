### Title
Unbounded `max_price_per_unit` Causes `assert_nn_le` Constraint Failure in `charge_fee`, Making Blocks Unprovable — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`pack_resource_bounds` only validates `max_price_per_unit >= 0` via `assert_nn`, leaving it unbounded up to `2^128 - 1`. When `compute_max_possible_fee` multiplies this by `max_amount` (up to `2^64 - 1`), the resulting `max_fee` felt value can exceed `2^128`. The subsequent `assert_nn_le(actual_fee, max_fee)` in `charge_fee` uses a range-check-backed comparison that requires both operands and their difference to fit in `[0, 2^128)`. When `max_fee >= 2^128`, no valid `actual_fee` can satisfy this constraint, making the block unprovable and halting the network.

---

### Finding Description

**Step 1 — Missing upper bound on `max_price_per_unit`.**

In `pack_resource_bounds`:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // bounded to 64 bits ✓
    assert_nn(resource_bounds.max_price_per_unit);            // only checks >= 0, allows up to 2^128 - 1 ✗
    ...
}
```

`assert_nn_le` is used to cap `max_amount` at `2^64 - 1`, but `max_price_per_unit` only receives `assert_nn`, which in Cairo's range-check system constrains the value to `[0, 2^128 - 1]` — no tighter upper bound is enforced. [1](#0-0) 

**Step 2 — Unchecked felt multiplication in `compute_max_possible_fee`.**

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

With `max_amount ∈ [0, 2^64 - 1]` and `max_price_per_unit ∈ [0, 2^128 - 1]`, each product term can reach `(2^64 - 1) × (2^128 - 1) ≈ 2^192`. The sum of three such terms can be up to `~2^193.6`, well above `2^128`. Cairo felt arithmetic is modular (mod P ≈ 2^251), so no field overflow occurs, but the returned `max_fee` felt value is a large integer exceeding `2^128`. [2](#0-1) 

**Step 3 — `assert_nn_le` constraint failure in `charge_fee`.**

```cairo
local low_actual_fee;
%{ LoadActualFee %}
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    amount=Uint256(low=low_actual_fee, high=0),
);
// Verify that the charged amount is not larger than the transaction's max_fee field.
assert_nn_le(calldata.amount.low, max_fee);
```

`assert_nn_le(a, b)` is implemented via two range checks:
1. `a ∈ [0, 2^128)` — checks `actual_fee` is non-negative and fits in 128 bits.
2. `b - a ∈ [0, 2^128)` — checks `max_fee - actual_fee` fits in 128 bits.

When `max_fee >= 2^128`, for any realistic `actual_fee < 2^128`, the difference `max_fee - actual_fee >= max_fee - (2^128 - 1)`. If `max_fee >= 2^129`, then `max_fee - actual_fee >= 2^128`, violating constraint (2). No valid `actual_fee` in `[0, 2^128)` can satisfy both constraints simultaneously, making the Cairo proof unsatisfiable. [3](#0-2) 

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

If a block contains a transaction with `max_fee >= 2^128`, the StarkNet OS cannot generate a valid STARK proof for that block. The prover will fail to satisfy the `assert_nn_le` range-check constraint. Since the block is unprovable, it cannot be finalized on L1, halting the network's ability to confirm new transactions.

---

### Likelihood Explanation

A V3 transaction sender directly controls `max_price_per_unit` in their resource bounds. The OS code imposes no upper bound tighter than `2^128 - 1` on this field. Setting `max_amount = 2` and `max_price_per_unit = 2^128 - 1` for any single resource type yields `max_fee = 2^129 - 2 > 2^128`, which is sufficient to trigger the failure. The sequencer has no OS-enforced reason to reject such a transaction — a high `max_price_per_unit` looks like a user willing to pay a premium. If the sequencer lacks an independent off-chain bound check for this field, a single crafted transaction can render the entire block unprovable.

---

### Recommendation

Add an explicit upper bound on `max_price_per_unit` in `pack_resource_bounds`, analogous to the existing bound on `max_amount`:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 64 - 1);  // add this
    ...
}
```

Alternatively, add a post-computation check in `compute_max_possible_fee` asserting the result fits in `[0, 2^128 - 1]`, or restructure `charge_fee` to use a Uint256-aware comparison instead of `assert_nn_le`.

---

### Proof of Concept

1. Attacker constructs a V3 invoke transaction with:
   - `max_amount = 2` for L1 gas
   - `max_price_per_unit = 2^128 - 1` for L1 gas (passes `assert_nn` check at line 105 of `transaction_hash.cairo`)
   - All other resource bounds set to 0

2. `pack_resource_bounds` accepts the transaction — `assert_nn(2^128 - 1)` passes since `2^128 - 1 < 2^128`.

3. Sequencer includes the transaction (it appears to be a high-fee transaction).

4. OS calls `compute_max_possible_fee`:
   ```
   max_fee = 2 * (2^128 - 1) = 2^129 - 2
   ```

5. OS calls `charge_fee`. Sequencer hint `LoadActualFee` sets `low_actual_fee` to the real cost, e.g., `1000`.

6. `assert_nn_le(1000, 2^129 - 2)` evaluates:
   - Check 1: `1000 ∈ [0, 2^128)` → passes.
   - Check 2: `(2^129 - 2) - 1000 = 2^129 - 1002 ∈ [0, 2^128)`? → **FAILS** since `2^129 - 1002 > 2^128`.

7. The range-check constraint is unsatisfiable. The STARK proof for the block cannot be generated. The block is permanently stuck, halting the network.
