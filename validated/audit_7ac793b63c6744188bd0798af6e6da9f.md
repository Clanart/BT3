### Title
Unprivileged Caller Can Deploy Arbitrary NEAR Tokens via Unguarded `logMetadata` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.logMetadata` is a public, payable function with no access control. It reads `name`, `symbol`, and `decimals` directly from any caller-supplied address via `IERC20Metadata`, then publishes a Wormhole message through the legitimate bridge contract. Because the Wormhole VAA's `emitter_address` is the bridge itself (not the token), NEAR's only factory guard passes, and `deploy_token_callback` deploys a real NEAR token for the attacker-controlled EVM address with attacker-chosen metadata.

---

### Finding Description

**Step 1 — Unguarded EVM entrypoint**

`logMetadata` in `OmniBridge.sol` carries no `onlyRole` or any other access modifier:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name    = IERC20Metadata(tokenAddress).name();
    string memory symbol  = IERC20Metadata(tokenAddress).symbol();
    uint8  decimals       = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

There is no check that `tokenAddress` is a registered bridge token, a known ERC-20, or anything else.

**Step 2 — Wormhole message published by the legitimate bridge**

`OmniBridgeWormhole.logMetadataExtension` publishes the message via `_wormhole.publishMessage`. The Wormhole VAA's `emitter` field is set by the Wormhole core contract to `msg.sender` — i.e., the `OmniBridgeWormhole` contract address, not the attacker-supplied `tokenAddress`:

```solidity
function logMetadataExtension(
    address tokenAddress, string memory name,
    string memory symbol, uint8 decimals
) internal override {
    bytes memory payload = bytes.concat(
        bytes1(uint8(MessageType.LogMetadata)),
        bytes1(omniBridgeChainId),
        Borsh.encodeAddress(tokenAddress),
        Borsh.encodeString(name),
        Borsh.encodeString(symbol),
        bytes1(decimals)
    );
    _wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
    wormholeNonce++;
}
``` [2](#0-1) 

The Wormhole guardians sign the VAA without inspecting payload content; they only attest that the registered emitter published it.

**Step 3 — NEAR factory check passes**

`deploy_token_callback` on NEAR extracts the `emitter_address` from the prover result and checks it against the registered factory map:

```rust
let chain = metadata.emitter_address.get_chain();
require!(
    self.factories.get(&chain) == Some(metadata.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
``` [3](#0-2) 

Because the VAA's emitter is the legitimate `OmniBridgeWormhole` contract (the registered factory), this check passes unconditionally regardless of what `token_address` or metadata the payload contains.

**Step 4 — NEAR deploys a token for the attacker-controlled address**

`deploy_token_internal` is then called with the attacker-supplied `token_address` and metadata:

```rust
self.deploy_token_internal(
    chain,
    &metadata.token_address,
    BasicMetadata { name: metadata.name, symbol: metadata.symbol, decimals: metadata.decimals },
    attached_deposit,
)
``` [4](#0-3) 

This registers the attacker's EVM address in `token_address_to_id`, `token_id_to_address`, `token_decimals`, and `deployed_tokens`, and calls the token deployer to create a live NEP-141 token on NEAR.

The `emitter_address` field in the parsed `LogMetadataMessage` is populated from `event.address` (the bridge contract), not from the token address:

```rust
emitter_address: OmniAddress::new_from_evm_address(
    chain_kind,
    H160(event.address.into()),
)?,
``` [5](#0-4) 

There is no secondary check anywhere in the path that validates `token_address` against a whitelist or the bridge's own `isBridgeToken` / `ethToNearToken` mappings.

---

### Impact Explanation

An unprivileged attacker can:

1. **Deploy arbitrary NEAR tokens** for any EVM address they control, with any name/symbol/decimals (e.g., `name="USDC"`, `symbol="USDC"`, `decimals=6`). The resulting NEAR token is fully registered in the bridge's token registry.
2. **Pollute the token registry** with attacker-controlled EVM↔NEAR mappings, enabling future manipulation of transfer flows that rely on `token_address_to_id` / `token_id_to_address` lookups.
3. **Impersonate legitimate tokens** by name/symbol, misleading users and off-chain tooling into treating the fake token as a real bridge-wrapped asset.

This matches the allowed High impact: *"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized transfer finalization, token deployment, or message execution."* The Wormhole verification is not bypassed cryptographically, but the bridge's own invariant — that only legitimate tokens may be registered — is bypassed because the emitter check is insufficient to enforce it.

---

### Likelihood Explanation

- No privileged role, leaked key, or compromised guardian is required.
- The attacker only needs to deploy a minimal ERC-20 contract and call one public function.
- The Wormhole fee (`msg.value`) is the only cost.
- Fully reproducible on a local fork or testnet.

---

### Recommendation

Add an access control guard to `logMetadata` (e.g., `onlyRole(DEFAULT_ADMIN_ROLE)` or a dedicated `METADATA_MANAGER_ROLE`), or add an explicit check that `tokenAddress` is a known, registered token before publishing the Wormhole message:

```solidity
function logMetadata(address tokenAddress) external payable onlyRole(DEFAULT_ADMIN_ROLE) {
    ...
}
```

Alternatively, validate on the NEAR side that `metadata.token_address` is already present in `token_address_to_id` before allowing `deploy_token` to proceed for a second registration, and reject any `LogMetadata` VAA whose token address is not already known to the bridge.

---

### Proof of Concept

```solidity
// 1. Deploy a fake ERC-20
contract FakeUSDC {
    function name()     external pure returns (string memory) { return "USD Coin"; }
    function symbol()   external pure returns (string memory) { return "USDC"; }
    function decimals() external pure returns (uint8)         { return 6; }
}

// 2. Call logMetadata on the live OmniBridgeWormhole with the Wormhole fee
FakeUSDC fake = new FakeUSDC();
bridge.logMetadata{value: wormhole.messageFee()}(address(fake));

// 3. Capture the resulting Wormhole VAA.
//    Its payload is byte-for-byte identical to a legitimate USDC logMetadata VAA
//    except for the token address field.
//    The emitter_address in the VAA is the legitimate bridge contract.

// 4. Submit the VAA to NEAR's deploy_token.
//    NEAR's factory check passes (emitter == registered factory).
//    NEAR deploys a NEP-141 token named "USD Coin" / "USDC" mapped to address(fake).
```

The resulting NEAR token is indistinguishable by name/symbol from a legitimate USDC bridge token and is fully registered in the bridge's token registry.
