### Title
Dust-Amount Transfer Permanently Locks/Burns User Tokens With No Recovery - (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR→EVM bridge transfer with an amount smaller than the decimal normalization factor (`10^(origin_decimals - decimals)`), `init_transfer_internal` burns or locks the tokens successfully, but the subsequent `sign_transfer` call always panics with `InvalidAmountToTransfer` because `normalize_amount` floors to zero. There is no cancel or refund path, so the user's tokens are permanently lost.

### Finding Description

`init_transfer` (called via `ft_on_transfer`) accepts any amount satisfying `fee < amount` and immediately burns or locks the tokens in `init_transfer_internal`: [1](#0-0) [2](#0-1) 

No check is made that `amount - fee` is large enough to survive decimal normalization. Later, when a trusted relayer calls `sign_transfer`, the amount is normalized: [3](#0-2) [4](#0-3) 

`normalize_amount` performs integer floor division: `amount / 10^(origin_decimals - decimals)`. For a token registered with `origin_decimals = 24` and `decimals = 18` (a common NEAR→EVM pairing), the divisor is `10^6`. Any `amount_without_fee < 1_000_000` normalizes to `0`, causing `sign_transfer` to panic and roll back — but the tokens were already burned/locked in the prior `ft_transfer_call` transaction.

The transfer message remains in `pending_transfers` indefinitely. There is no `cancel_transfer`, `refund_transfer`, or any other recovery path visible in the contract. The tokens are irrecoverably lost. [5](#0-4) 

The code comment on `normalize_amount` acknowledges dust loss only for sub-unit remainders, not for the case where the entire amount normalizes to zero: [6](#0-5) 

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

For any token with `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridging to an EVM chain where it is registered with 18 decimals):

- Bridged tokens (`is_deployed_token = true`): burned via `burn_tokens_if_needed` — permanently destroyed.
- Native tokens: locked in the bridge contract via `lock_tokens_if_needed` — permanently frozen with no unlock path.

The transfer message occupies storage in `pending_transfers` forever, and `sign_transfer` will always revert for this transfer ID, making the lock/burn unclaimable.

### Likelihood Explanation

**Medium.** The condition requires a token registered with `origin_decimals > decimals` (e.g., 24 vs 18, giving a factor of 10^6). A user sending fewer than 10^6 base units — a plausible mistake for a token whose human-readable unit is large — triggers the loss. No privileged access is required; any user calling `ft_transfer_call` with a small amount is affected. The `fee < amount` guard does not prevent this.

### Recommendation

Add a minimum-amount check in `init_transfer` (before burning/locking) that verifies `normalize_amount(amount - fee, decimals) > 0`. Alternatively, enforce this check at the start of `init_transfer_internal` and return the full `amount` to the caller (triggering a NEP-141 refund) if the normalized amount is zero, mirroring the existing storage-balance failure path at lines 1838–1848. [7](#0-6) 

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (normalization factor = `10^6`).
2. User calls `ft_transfer_call` on the token contract with `amount = 500_000`, `fee = 0`, targeting the bridge contract with an `InitTransfer` message to an EVM recipient.
3. `init_transfer` passes the `fee < amount` check (`0 < 500_000`).
4. `init_transfer_internal` burns (or locks) `500_000` base units of the token. Transaction succeeds; `InitTransferEvent` is emitted.
5. Trusted relayer calls `sign_transfer` for this transfer ID.
6. `normalize_amount(500_000, {origin_decimals:24, decimals:18}) = 500_000 / 1_000_000 = 0`.
7. `require!(amount_to_transfer > 0, ...)` panics → `sign_transfer` reverts.
8. The transfer message remains in `pending_transfers`. The 500,000 base units are permanently burned/locked. The user has no recovery path.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
