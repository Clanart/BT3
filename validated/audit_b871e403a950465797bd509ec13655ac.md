### Title
`removeCustomToken()` Deletes Token Mappings Without Checking In-Flight Transfers, Permanently Locking User Funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.removeCustomToken()` unconditionally deletes all token routing state (`isBridgeToken`, `customMinters`, `ethToNearToken`, `nearToEthToken`) for a custom token without verifying whether any cross-chain transfers are currently in-flight. After deletion, any pending NEAR→EVM `finTransfer()` call for that token falls through to a `safeTransfer()` branch that always reverts because the bridge contract holds no token balance for burn-and-mint custom tokens. The user's funds on the NEAR side are already burned/locked and cannot be recovered, resulting in permanent loss.

### Finding Description

`removeCustomToken()` is an admin function that removes a custom token registration:

```solidity
function removeCustomToken(
    address tokenAddress
) external onlyRole(DEFAULT_ADMIN_ROLE) {
    delete isBridgeToken[tokenAddress];
    delete nearToEthToken[ethToNearToken[tokenAddress]];
    delete ethToNearToken[tokenAddress];
    delete customMinters[tokenAddress];
}
``` [1](#0-0) 

There is no check for pending or in-flight transfers before deletion.

When a custom token is registered via `addCustomToken()`, the bridge uses a burn-and-mint model: on `initTransfer()`, tokens are transferred to `customMinters[tokenAddress]` and burned there; on `finTransfer()`, the custom minter mints tokens to the recipient. [2](#0-1) 

The `finTransfer()` dispatch logic checks `customMinters` and `isBridgeToken` in order, and falls through to a plain `safeTransfer` if both are cleared:

```solidity
} else if (customMinters[payload.tokenAddress] != address(0)) {
    ICustomMinter(customMinters[payload.tokenAddress]).mint(...);
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(...);
} else {
    IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
}
``` [3](#0-2) 

After `removeCustomToken()` clears both `customMinters[tokenAddress]` and `isBridgeToken[tokenAddress]`, every `finTransfer()` call for that token hits the `safeTransfer` branch. Because the bridge contract never held these tokens (they were burned via the custom minter), the `safeTransfer` reverts. The entire transaction reverts, so the nonce is not consumed, but the call will revert on every retry — the transfer can never be finalized.

### Impact Explanation

**Permanent freezing of user funds.** A user who initiated a NEAR→EVM transfer of a custom token has their NEAR-side tokens burned (or locked in `locked_tokens`) before `finTransfer()` is called on EVM. If `removeCustomToken()` is called between the NEAR-side burn and the EVM-side finalization, the EVM `finTransfer()` permanently reverts. The user loses their tokens with no recovery path: the NEAR side has already committed the burn, and the EVM side can never release the corresponding amount.

This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows.**

### Likelihood Explanation

The admin may legitimately call `removeCustomToken()` to replace a faulty custom minter, deprecate a token, or respond to an incident — without being aware of in-flight transfers. Cross-chain transfers have multi-block latency; the window between a NEAR-side burn and EVM-side finalization is non-trivial. Any admin action during this window causes permanent loss. No malicious intent is required.

### Recommendation

Add a guard in `removeCustomToken()` that prevents removal while transfers may be in-flight. The simplest approach mirrors the Tapioca recommendation: track a per-token in-flight counter (incremented on `initTransfer`, decremented on `finTransfer`) and require it to be zero before deletion. Alternatively, introduce a two-phase deprecation: first pause new transfers for the token, wait for all pending transfers to settle, then allow removal.

```solidity
function removeCustomToken(
    address tokenAddress
) external onlyRole(DEFAULT_ADMIN_ROLE) {
+   require(pendingTransferCount[tokenAddress] == 0, "in-flight transfers exist");
    delete isBridgeToken[tokenAddress];
    delete nearToEthToken[ethToNearToken[tokenAddress]];
    delete ethToNearToken[tokenAddress];
    delete customMinters[tokenAddress];
}
```

### Proof of Concept

1. Admin calls `addCustomToken("near-token", tokenAddr, customMinterAddr, 18)`.
2. User on NEAR initiates a NEAR→EVM transfer of `near-token`. NEAR side burns the user's tokens and emits a signed `TransferMessage`.
3. Before the relayer calls `finTransfer()` on EVM, admin calls `removeCustomToken(tokenAddr)`. This sets `customMinters[tokenAddr] = address(0)` and `isBridgeToken[tokenAddr] = false`.
4. Relayer calls `finTransfer()` with the signed payload. The dispatch logic skips the custom minter branch (cleared) and the bridge token branch (cleared), and falls to `IERC20(tokenAddr).safeTransfer(recipient, amount)`.
5. The bridge contract holds zero balance of `tokenAddr` → `safeTransfer` reverts.
6. The relayer retries indefinitely; every attempt reverts. The user's NEAR-side tokens are permanently lost. [1](#0-0) [4](#0-3) [5](#0-4)
