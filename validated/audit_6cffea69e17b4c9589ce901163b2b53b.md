### Title
Missing Backward-Compatibility Edge Case in `calculate_global_state_root` Produces Wrong State Commitment - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/state/commitment.cairo`)

### Summary

`calculate_global_state_root` in `commitment.cairo` is documented to handle three distinct cases but only implements two. The missing case — when the contract class tree is empty (`contract_class_root == 0`) but the contract state tree is non-empty (`contract_state_root != 0`) — causes the OS to commit a wrong global state root to its output. This is the direct structural analog of the TWAV bug: a condition check uses the wrong field (or an incomplete set of fields) to determine state, causing the function to return an incorrect value instead of the correct one.

### Finding Description

The function comment at lines 31–37 explicitly documents three cases:

```
// If both the contract class and contract state trees are empty, the global root is set to 0.
// If the contract class tree is empty, the global state root is equal to the
// contract state root (for backward compatibility);
// Otherwise, the global root is obtained by:
//     global_root = H(GLOBAL_STATE_VERSION, contract_state_root, contract_class_root).
```

The implementation, however, only handles two:

```cairo
func calculate_global_state_root{poseidon_ptr: PoseidonBuiltin*, range_check_ptr}(
    contract_state_root: felt, contract_class_root: felt
) -> (global_root: felt) {
    if (contract_state_root == 0 and contract_class_root == 0) {
        // The state is empty.
        return (global_root=0);
    }

    tempvar elements: felt* = new (GLOBAL_STATE_VERSION, contract_state_root, contract_class_root);
    let (global_root) = poseidon_hash_many(n=3, elements=elements);
    return (global_root=global_root);
}
``` [1](#0-0) 

When `contract_class_root == 0` and `contract_state_root != 0`, the code falls through to the `poseidon_hash_many` branch and returns `H(GLOBAL_STATE_VERSION, contract_state_root, 0)` instead of the documented backward-compatible value `contract_state_root`. The condition `contract_state_root == 0 and contract_class_root == 0` is the wrong guard — it is analogous to checking `timestamp == 0` in the TWAV bug: the field used (`contract_class_root`) can legitimately be zero even when valid state exists, causing the function to return the wrong value.

This wrong root propagates directly into `state_update` in `state.cairo`:

```cairo
let (local final_global_root) = calculate_global_state_root(
    contract_state_root=contract_state_tree_update_output.final_root,
    contract_class_root=contract_class_tree_update_output.final_root,
);
tempvar state_update_output = new CommitmentUpdate(
    initial_root=initial_global_root, final_root=final_global_root
);
``` [2](#0-1) 

And is then serialized to the OS output via `serialize_output_header`:

```cairo
serialize_word(os_output_header.state_update_output.initial_root);
serialize_word(os_output_header.state_update_output.final_root);
``` [3](#0-2) 

### Impact Explanation

The global state root is a core protocol commitment verified by L1. The comment explicitly marks the backward-compatibility case as a protocol requirement. When `contract_class_root == 0` and `contract_state_root != 0`, the OS emits `H(GLOBAL_STATE_VERSION, contract_state_root, 0)` as the state root instead of `contract_state_root`. Any L1 verifier or downstream consumer that correctly implements the documented three-case logic will reject this proof, because the committed root does not match the expected backward-compatible value. This causes every block produced under this condition to be unprovable/unverifiable on L1, resulting in a **total network halt** — no new transactions can be confirmed.

### Likelihood Explanation

The triggering condition (`contract_class_root == 0`, `contract_state_root != 0`) occurs in the early life of any new StarkNet deployment or chain fork before any class declaration has been processed, while contract state changes (e.g., pre-deployed system contracts, L1 handler messages, or any invoke transaction) have already been applied. Any unprivileged user submitting a transaction — an invoke, deploy, or L1→L2 message — in such a block triggers the wrong root computation. No special privilege is required; the sequencer processes the block normally and the OS produces the wrong commitment automatically.

### Recommendation

Add the missing backward-compatibility branch, matching the documented specification:

```cairo
func calculate_global_state_root{poseidon_ptr: PoseidonBuiltin*, range_check_ptr}(
    contract_state_root: felt, contract_class_root: felt
) -> (global_root: felt) {
    if (contract_state_root == 0 and contract_class_root == 0) {
        // The state is empty.
        return (global_root=0);
    }

    // Backward compatibility: if the class tree is empty, return the contract state root directly.
    if (contract_class_root == 0) {
        return (global_root=contract_state_root);
    }

    tempvar elements: felt* = new (GLOBAL_STATE_VERSION, contract_state_root, contract_class_root);
    let (global_root) = poseidon_hash_many(n=3, elements=elements);
    return (global_root=global_root);
}
```

This aligns the implementation with the three-case specification in the function's own comment.

### Proof of Concept

1. Start a new StarkNet network (or any deployment where no class declarations have been processed yet). The contract class Patricia tree is empty: `contract_class_root = 0`.
2. An unprivileged user submits any transaction that modifies contract state (e.g., an L1→L2 message to a pre-deployed contract, or a deploy-account transaction). After execution, `contract_state_root != 0`.
3. `state_update` in `state.cairo` calls `calculate_global_state_root(contract_state_root=X, contract_class_root=0)` where `X != 0`.
4. The condition `contract_state_root == 0 and contract_class_root == 0` is false, so the code falls through to `poseidon_hash_many(n=3, [GLOBAL_STATE_VERSION, X, 0])`.
5. The OS output contains `final_root = H(GLOBAL_STATE_VERSION, X, 0)` instead of the correct backward-compatible value `X`.
6. The L1 verifier, which implements the documented three-case logic, expects `final_root = X` and rejects the proof.
7. No block can be confirmed on L1 → total network halt. [4](#0-3) [5](#0-4) [6](#0-5)
