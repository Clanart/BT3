### Title
`ProofOfInclusion::valid()` Never Validates Against an External Trusted Root — Forged Inclusion Proofs Always Pass — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only verifies internal hash-chain consistency. It never compares the derived root against any externally trusted root hash. Because `root_hash()` is derived entirely from the proof's own fields, the final equality check inside `valid()` is a tautology — it is always `true` when the loop completes. An attacker can craft an arbitrary, internally consistent `ProofOfInclusion` for any `node_hash` they choose, and `valid()` will return `true`, regardless of whether the proof corresponds to any real DataLayer tree.

### Finding Description

`ProofOfInclusion::valid()` performs two-stage validation analogous to the RaptorCast pattern:

**Stage 1 — Internal hash chain consistency (implemented):** [1](#0-0) 

For each layer, it computes `calculate_internal_hash(existing_hash, other_hash_side, other_hash)` and checks it equals `layer.combined_hash`. If all layers pass, it sets `existing_hash = calculated_hash`.

**Stage 2 — Comparison against an externally trusted root (missing):**

The final check is:
```rust
existing_hash == self.root_hash()
```

But `root_hash()` is: [2](#0-1) 

`root_hash()` returns `last.combined_hash` — a field from the proof itself. After the loop, `existing_hash` is exactly `last.combined_hash` (because the loop's invariant guarantees `existing_hash = calculated_hash == layer.combined_hash` after each iteration). Therefore `existing_hash == self.root_hash()` is always `true` when the loop passes. The final check is a tautology and provides zero security.

**Contrast with the correct pattern** used in the consensus Merkle set: [3](#0-2) 

`validate_merkle_proof` takes an external `root: &[u8; 32]` parameter and explicitly checks `tree.get_root() != *root`, returning an error if they differ. `ProofOfInclusion::valid()` has no such parameter.

**Attack construction:** An attacker picks any target `node_hash` value (e.g., a key-value pair they want to falsely prove is in the DataLayer). They construct a `ProofOfInclusion` with:
- `node_hash` = attacker-chosen leaf hash
- `layers` = any sequence where each `combined_hash = calculate_internal_hash(prev, side, other_hash)` (trivially satisfiable by choosing arbitrary `other_hash` values and computing `combined_hash` forward)

The resulting struct passes `valid()` unconditionally.

The `ProofOfInclusion` type is fully serializable and exposed via Python bindings: [4](#0-3) [5](#0-4) 

It can be constructed directly in Python or deserialized from bytes (`from_bytes`, `from_bytes_unchecked`, `parse_rust`), making it reachable from untrusted network input.

### Impact Explanation

Any DataLayer consumer that receives a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` as its sole verification step will accept forged inclusion proofs. The attacker can prove that any arbitrary key-value pair is included in any DataLayer store, without possessing the actual tree or its root. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `valid()` method name strongly implies complete validation. There is no documentation warning that callers must separately compare `proof.root_hash()` against a trusted external root. Any DataLayer peer-to-peer protocol that exchanges `ProofOfInclusion` objects and relies on `valid()` alone is exploitable by any network peer with zero cost.

### Recommendation

`valid()` must accept an external trusted root parameter and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    // existing internal chain check ...
    // final check:
    existing_hash == *trusted_root  // NOT self.root_hash()
}
```

Alternatively, rename the current method to `is_internally_consistent()` and add a separate `valid_for_root(trusted_root: &Hash) -> bool` that performs the complete check. The Python binding should expose only the root-anchored variant as the primary API.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from chia_rs.sized_bytes import bytes32
from chia_sha2 import Sha256  # or any sha256

# Attacker wants to forge proof that arbitrary_leaf is in some tree.
arbitrary_leaf = bytes32(b'\xde\xad' * 16)

# Build a single-layer proof. Choose any other_hash.
other_hash = bytes32(b'\xaa' * 32)

# Compute combined_hash = calculate_internal_hash(arbitrary_leaf, side=0, other_hash)
# (using the same hash function as chia-datalayer)
h = Sha256()
h.update(b'\x01')  # MIDDLE prefix
h.update(arbitrary_leaf)
h.update(other_hash)
combined = bytes32(h.finalize())

layer = ProofOfInclusionLayer(
    other_hash_side=0,
    other_hash=other_hash,
    combined_hash=combined,
)
forged_proof = ProofOfInclusion(node_hash=arbitrary_leaf, layers=[layer])

assert forged_proof.valid()  # Returns True — forged proof accepted
# forged_proof.root_hash() == combined (attacker-controlled, not any real tree root)
```

The `valid()` call returns `True` for a completely fabricated proof that has no relationship to any real DataLayer tree.

    pub fn valid(&self) -> bool {
        let mut existing_hash = self.node_hash;

        for layer in &self.layers {
            let calculated_hash = crate::calculate_internal_hash(
                &existing_hash,
                layer.other_hash_side,
                &layer.other_hash,
            );

            if calculated_hash != layer.combined_hash {
                return false;
            }

            existing_hash = calculated_hash;
        }

        existing_hash == self.root_hash()
    }
