### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts a user-controlled `class_hash` parameter and updates the contract's state without verifying that the provided hash corresponds to a declared contract class. This is directly analogous to the reported SVFHook vulnerability, where a user-controlled parameter (pool key) is accepted without validation against an expected set of valid values. In the StarkNet OS, the missing check allows any contract to replace its own class with an arbitrary, undeclared hash, rendering the contract permanently uncallable and freezing any funds held within it.

---

### Finding Description

In `execute_replace_class`, the OS reads `class_hash` directly from the user-controlled `ReplaceClassRequest` struct and writes it into `contract_state_changes` without any validation that the hash corresponds to a declared Sierra class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The missing check is explicitly acknowledged by the TODO comment at line 898. The `class_hash` field comes directly from the syscall request, which is fully controlled by the calling contract (and thus by the contract deployer/user).

Once the state is updated with an undeclared class hash, any subsequent call to that contract reaches `execute_entry_point`, which performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [2](#0-1) 

Since the class hash was never declared, `dict_read` returns the default value `0`. The subsequent `find_element` call then searches for a compiled class fact with hash `0`:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

If no compiled class with hash `0` exists (the normal case), `find_element` fails with an assertion error. The sequencer's blockifier will detect this failure during simulation and refuse to include any call to the affected contract in future blocks. The contract is therefore permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash becomes permanently uncallable. All funds (tokens, ETH, or other assets) held in that contract's storage are permanently frozen with no recovery path. This is a direct, irreversible loss of user funds.

A realistic scenario: a vault or multisig contract holds user deposits. A malicious or buggy upgrade path calls `replace_class` with an arbitrary hash. The OS accepts the update without validation. The contract is permanently bricked, and all deposited funds are frozen forever.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any deployed contract — no privileged role is required. The entry path is fully reachable by an unprivileged contract deployer. The missing validation is acknowledged in the codebase itself (the TODO comment), confirming it is a known gap. Any contract whose upgrade logic does not independently validate the class hash (or any contract with a bug in its access control for `replace_class`) is vulnerable. The likelihood is **medium**: it requires a contract to call `replace_class` with an invalid hash, but the OS provides no safety net to prevent this.

---

### Recommendation

In `execute_replace_class`, before updating `contract_state_changes`, validate that `request.class_hash` is present in `contract_class_changes` (i.e., it has been declared). Specifically, perform a `dict_read` on `contract_class_changes` with `request.class_hash` as the key and assert that the returned compiled class hash is non-zero. This mirrors the validation already performed implicitly in `execute_entry_point` and closes the gap identified by the TODO comment.

---

### Proof of Concept

1. An unprivileged user deploys a contract `C` that holds user funds and exposes a function that calls the `replace_class` syscall.
2. The user (or an attacker exploiting a bug in `C`'s access control) calls `replace_class` with `class_hash = <any undeclared felt value>`.
3. The OS processes `execute_replace_class`: gas is deducted, and `contract_state_changes` is updated with the new (undeclared) class hash. **No validation occurs.**
4. In any subsequent block, when any party attempts to call contract `C`, `execute_entry_point` performs `dict_read` on `contract_class_changes` for the undeclared hash, receives `0`, and `find_element` fails to locate a compiled class fact.
5. The sequencer's blockifier detects the failure and excludes all calls to `C` from future blocks.
6. Contract `C` is permanently uncallable. All funds in its storage are permanently frozen.
