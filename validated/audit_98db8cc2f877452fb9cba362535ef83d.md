### Title
User Funds Permanently Locked Due to Missing Pre-Validation of Dust-Normalized Transfer Amount in `init_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The `init_transfer` function in the NEAR bridge contract accepts and locks/burns user tokens without validating that the net transfer amount (after fee deduction) will survive decimal normalization to a non-zero value. When `sign_transfer` is later called, it applies `normalize_amount()` and panics with `ERR_INVALID_AMOUNT_TO_TRANSFER` if the result is zero. Since there is no cancel or refund path for a stored transfer message, the user's tokens are permanently irrecoverable.

### Finding Description

The NEAR bridge normalizes token amounts when crossing chains with different decimal precisions. `normalize_amount` uses floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

For a token with `origin_decimals = 24` (NEAR) and `decimals = 18` (EVM), `diff_decimals = 6`, meaning any `amount_without_fee < 10^6` normalizes to zero.

`init_transfer` only validates that `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

It does **not** validate that `normalize_amount(amount - fee, decimals) > 0`. After this check passes, `init_transfer_internal` immediately burns or locks the user's tokens: [3](#0-2) 

Later, when a relayer calls `sign_transfer`, the normalization is applied and the zero-check fires:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

This panic happens before the MPC signing call, so `sign_transfer_callback` is never reached and the transfer message is never removed. There is no `cancel_transfer` or admin-rescue function in the contract. The transfer message remains in storage indefinitely, and the locked/burned tokens are unrecoverable.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any user who initiates a transfer where `(amount - fee) < 10^(origin_decimals - dest_decimals)` will have their tokens permanently locked in the bridge (for native tokens) or permanently burned (for deployed/bridged tokens) with no recovery path. The transfer message persists in contract storage but can never be finalized or cancelled.

### Likelihood Explanation

**Medium.** The condition is reachable by any unprivileged user calling `ft_transfer_call` → `init_transfer`. It can occur:
- **Accidentally**: A user unfamiliar with decimal normalization sends a small amount (e.g., 500,000 yoctoNEAR-equivalent units of a 24-decimal token to an 18-decimal EVM chain, where the minimum transferable unit is 10^6).
- **Intentionally (self-harm)**: A user sets `amount = 10^6` and `fee = 10^6 - 1`, passing the `fee < amount` check while making `amount_without_fee = 1`, which normalizes to 0.

Tokens with large decimal differences (e.g., 24 vs 6, diff = 18) make this trivially reachable for any amount below 10^18 units.

### Recommendation

Add a pre-validation in `init_transfer` (or in `init_transfer_internal`) that checks the normalized net amount is non-zero before accepting the transfer and locking/burning tokens:

```rust
// After constructing transfer_message, before locking tokens:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the fix pattern from the external report: validate the dust-adjusted (normalized) amount at deposit/initiation time, not at signing time.

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (diff = 6, so minimum transferable unit = 10^6).
2. Attacker (or naive user) calls `ft_transfer_call` with `amount = 10^6` and `InitTransferMsg { fee: U128(10^6 - 1), ... }`.
3. `init_transfer` check: `fee (10^6 - 1) < amount (10^6)` — passes.
4. `init_transfer_internal` burns/locks `10^6` token units from the user.
5. Transfer message is stored with `amount = 10^6`, `fee.fee = 10^6 - 1`.
6. Relayer calls `sign_transfer` for this transfer.
7. `amount_without_fee() = 10^6 - (10^6 - 1) = 1`.
8. `normalize_amount(1, {origin: 24, dest: 18}) = 1 / 10^6 = 0`.
9. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
10. The transfer message remains in storage permanently. The user's `10^6` token units are irrecoverably burned/locked. [5](#0-4) [4](#0-3) [6](#0-5)
