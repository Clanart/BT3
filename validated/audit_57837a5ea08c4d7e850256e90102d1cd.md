### Title
Non-EVM-native Cosmos Bank Tokens Permanently Orphaned on SELFDESTRUCT — (`File: x/evm/statedb/statedb.go`)

### Summary

`StateDB.Commit()` burns only the EVM-denom balance of a self-destructed contract before calling `DeleteAccount`. Any non-EVM-native Cosmos bank tokens (IBC coins, staking tokens, or any other denom) held at the contract's Cosmos address are never drained. After `DeleteAccount` removes the auth account, those tokens remain in the bank module at the dead address with no account to authorize spending them — permanently frozen. The code itself acknowledges this gap in a comment but leaves it unresolved.

---

### Finding Description

When a contract self-destructs, `StateDB.Commit()` processes the account deletion in `x/evm/statedb/statedb.go`:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
writeCache()
``` [1](#0-0) 

The `keeper.GetBalance` call is parameterized on `s.evmDenom` only — a single denom. The `BankKeeper` interface exposed to the EVM keeper does not include `GetAllBalances`, so there is no mechanism to enumerate and drain all non-EVM-native coins before the account is deleted. [2](#0-1) 

`DeleteAccount` removes the auth account and EVM storage, but explicitly does not touch bank balances:

```go
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
    ...
    k.accountKeeper.RemoveAccount(ctx, acct)
    ...
}
``` [3](#0-2) 

After `writeCache()` commits, the cosmos address has no auth account but retains any non-EVM-native bank balances. The `SubBalance` path (`SendCoinsFromAccountToModule`) requires an existing auth account, so those tokens can never be burned or transferred through normal means. [4](#0-3) 

The `BankKeeper` interface used by the EVM module does not include `GetAllBalances`: [5](#0-4)

### Impact Explanation

Any non-EVM-native Cosmos bank tokens held at a contract's Cosmos address at the time of SELFDESTRUCT are permanently frozen. The bank module's total supply is not reduced (no burn), but the tokens are inaccessible — no auth account exists to authorize a spend or burn. This is a permanent mis-accounting of Cosmos bank funds triggered by a standard EVM state transition.

In chains that deploy a bank precompile (as documented in Ethermint's own precompile guide), a secondary theft path exists: an attacker can redeploy a contract at the same CREATE2 address after SELFDESTRUCT, creating a fresh auth account at the same Cosmos address, and then use the bank precompile's `transfer` method to drain the orphaned tokens to an attacker-controlled address. [6](#0-5) 

This matches the allowed High impact: **EVM state transition bug that permits valid user funds to be permanently mis-accounted.**

---

### Likelihood Explanation

The entry path requires no special privileges:

1. Any Cosmos user can send non-EVM-native tokens (IBC coins, staking rewards, etc.) to a contract's Cosmos address via a standard `MsgSend` bank transaction. Ethermint's Cosmos address space is shared with the EVM address space — every EVM contract address has a corresponding Cosmos bech32 address that can receive bank sends.
2. The contract self-destructs via the SELFDESTRUCT opcode (EIP-6780: only effective when created in the same tx, or on pre-Cancun chains unconditionally).
3. `StateDB.Commit()` deletes the auth account but leaves non-EVM-native bank balances.

This is realistic in any Ethermint-based chain that supports IBC (which is the primary use case for Ethermint deployments). The scenario is not hypothetical — IBC tokens are routinely sent to contract addresses in Cosmos EVM ecosystems.

---

### Recommendation

In `StateDB.Commit()`, before calling `DeleteAccount`, enumerate and burn **all** bank balances at the contract's Cosmos address, not just the EVM denom. This requires adding `GetAllBalances(ctx, addr)` to the `BankKeeper` interface and iterating over the result:

```go
// Drain all bank denoms, not just evmDenom
allCoins := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allCoins {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-native balance")
    }
}
```

Alternatively, if burning non-EVM-native tokens is undesirable (e.g., IBC tokens cannot be burned by the EVM module), they should be transferred to a designated recovery address or the chain's community pool before account deletion, rather than left as permanently orphaned balances.

### Proof of Concept

**Step 1**: Deploy a contract via CREATE2 from a factory. The contract's EVM address maps to a deterministic Cosmos bech32 address.

**Step 2**: Send IBC tokens (e.g., `ibc/...` denom) to the contract's Cosmos address via a standard Cosmos `MsgSend`:
```
gaiad tx bank send alice cosmos1<contract_cosmos_addr> 1000000ibc/XXXX
```

**Step 3**: Trigger SELFDESTRUCT on the contract in the same transaction as its creation (EIP-6780 path):
```solidity
function attackInOneTx(bytes32 salt) external payable returns (address childAddr) {
    childAddr = create2(0, initCode, salt);
    SelfDestructTarget(childAddr).destroy(); // SELFDESTRUCT in same tx
}
```

**Step 4**: After the transaction commits, verify:
- `eth_getCode(childAddr)` returns `0x` (account deleted)
- Cosmos bank query `bank balances <cosmos_addr>` still shows the IBC tokens
- Any attempt to spend those tokens fails (no auth account)

**Step 5** (theft variant, chains with bank precompile): Redeploy at the same CREATE2 address, then call the bank precompile's `transfer` to drain the orphaned IBC tokens to an attacker address.

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
```
