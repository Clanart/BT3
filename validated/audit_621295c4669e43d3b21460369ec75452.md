### Title
Queued Future-Round Proposals Bypass Timestamp Staleness Check, Committing Stale L1 Gas Prices to Block Header - (File: crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs)

### Summary

When `SequencerConsensusContext::validate_proposal` receives a `ProposalInit` for a round greater than the current round, it stores the proposal verbatim in `queued_proposals` without any timestamp validation. When the round eventually advances and the proposal is dequeued, `is_block_info_valid` only rejects timestamps that are **too far in the future** (`> now + block_timestamp_window_seconds`). There is no lower-bound staleness check beyond `>= last_block_timestamp`. A proposal whose `init.timestamp` was valid when sent but is now arbitrarily old passes all checks, causing the block to be committed with a stale `block_timestamp` and stale L1 gas prices derived from that old timestamp.

### Finding Description

**Root cause — `validate_proposal` in `sequencer_consensus_context.rs`:**

When a proposal arrives for a future round, it is queued without any timestamp check:

```rust
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
``` [1](#0-0) 

Later, in `set_height_and_round`, the queued proposal is dequeued and forwarded to `validate_current_round_proposal` with the original `init` (including its original `init.timestamp`): [2](#0-1) 

`validate_current_round_proposal` calls `validate_proposal`, which calls `is_block_info_valid`. The timestamp checks in `is_block_info_valid` are:

```rust
if init_proposed.timestamp < last_block_timestamp { ... }   // lower bound: only previous block
if init_proposed.timestamp > now + block_timestamp_window_seconds { ... }  // upper bound: future only
``` [3](#0-2) 

There is **no check** of the form `init_proposed.timestamp >= now - staleness_threshold`. The `block_timestamp_window_seconds` is configured as **1 second** in production: [4](#0-3) 

**Gas price computation uses the stale timestamp:**

After the timestamp check passes, `is_block_info_valid` computes expected L1 gas prices using `init_proposed.timestamp` (the stale value):

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // <-- stale timestamp
    ...
).await;
``` [5](#0-4) 

The proposer also computed L1 gas prices at `init_proposed.timestamp`, so the margin check passes. The block is then committed with `block_timestamp = T_stale` and `l1_gas_price = price_at(T_stale)`.

`convert_to_sn_api_block_info` embeds the stale timestamp and stale gas prices directly into the `BlockInfo` used for execution: [6](#0-5) 

### Impact Explanation

A block is committed with:
- A `block_timestamp` that is arbitrarily stale (bounded only by the previous block's timestamp, not by wall-clock time)
- `l1_gas_price` and `l1_data_gas_price` computed at the stale timestamp, which may differ significantly from current L1 market prices

All transactions in that block are executed with wrong gas prices. Fee calculations, resource bounds checks, and L1 handler fee accounting all use the stale prices. This matches: **Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

### Likelihood Explanation

**Low-to-Medium.** Requires:
1. A proposer (legitimate or adversarial) to send a `ProposalInit` for a future round (e.g., round N+k) while the network is at round N. This is a normal part of Tendermint-style consensus — proposals for future rounds are expected and cached.
2. Multiple rounds to elapse before round N+k is reached, allowing the timestamp to become stale.
3. The stale timestamp to still be ≥ the previous block's timestamp (trivially satisfied if the previous block was committed before the proposal was sent).

A malicious proposer who knows they will be the proposer for a future round (deterministic in committee-based BFT) can deliberately send a proposal with a timestamp from the past, knowing it will be queued and later accepted.

### Recommendation

Add a staleness lower-bound check in `is_block_info_valid` (or at the dequeue site in `set_height_and_round`) that rejects proposals whose `init.timestamp` is more than `block_timestamp_window_seconds` (or a configurable `max_proposal_timestamp_age_seconds`) behind the current wall-clock time:

```rust
if now > init_proposed.timestamp + block_info_validation.block_timestamp_window_seconds {
    return Err(ValidateProposalError::InvalidBlockInfo(
        init_proposed.clone(),
        block_info_validation.clone(),
        format!("Timestamp is too stale: now={}, proposed={}", now, init_proposed.timestamp),
    ));
}
```

Alternatively, record the wall-clock time when a proposal is queued and reject it at dequeue time if the elapsed time exceeds a threshold.

### Proof of Concept

1. Network is at height H, round 0. The committee deterministically assigns Alice as proposer for round 5.
2. Alice builds a `ProposalInit` with `timestamp = T` (current time) and `l1_gas_price = price_at(T)`, and broadcasts it for round 5.
3. Validators receive the proposal. In `validate_proposal`, `init.round (5) > current_round (0)`, so the proposal is inserted into `queued_proposals` with no timestamp check.
4. Rounds 1–4 elapse (e.g., 5 minutes of slow consensus). L1 gas prices change significantly.
5. `set_height_and_round` advances to round 5. The queued proposal is dequeued and passed to `validate_current_round_proposal` with the original `init.timestamp = T`.
6. `is_block_info_valid` is called. `now = T + 5min`. Checks: `T >= last_block_timestamp` ✓; `T <= (T + 5min) + 1` ✓. Both pass.
7. `get_l1_prices_in_fri_and_wei(provider, T, ...)` returns `price_at(T)` (5-minute-old prices). The proposer's claimed prices also equal `price_at(T)`. Margin check passes.
8. `convert_to_sn_api_block_info` sets `block_timestamp = T` and `l1_gas_price = price_at(T)`.
9. The batcher executes all transactions in the block using 5-minute-old gas prices. The block is committed with a stale timestamp and stale L1 gas prices, producing wrong fee accounting for every transaction in the block.

    init: &ProposalInit,
) -> Result<starknet_api::block::BlockInfo, StarknetApiError> {
    if init.l1_gas_price_fri.0 == 0
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);
    }

    let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?;
    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;
    let previous_block_info = PreviousBlockInfo::from(init);
    let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;

    let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)
        .inspect_err(|_| {
            warn!(
                "L2 gas price in wei is zero! Conversion rate: {eth_to_fri_rate}, L2 gas price in \
                 FRI: {}",
                init.l2_gas_price_fri
            )
        })?;
    Ok(starknet_api::block::BlockInfo {
        block_number: init.height,
        block_timestamp: BlockTimestamp(init.timestamp),
        sequencer_address: init.builder,
        gas_prices: GasPrices {
            strk_gas_prices: GasPriceVector {
                l1_gas_price: l1_gas_price_fri,
                l1_data_gas_price: l1_data_gas_price_fri,
                l2_gas_price: l2_gas_price_fri,
            },
            eth_gas_prices: GasPriceVector {
                l1_gas_price: l1_gas_price_wei,
                l1_data_gas_price: l1_data_gas_price_wei,
                l2_gas_price: l2_gas_price_wei,
            },
        },
        use_kzg_da: init.l1_da_mode.is_use_kzg_da(),
        starknet_version: init.starknet_version,
    })
}
