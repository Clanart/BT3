### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of User Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. A malicious contract deployer can exploit this to replace a contract's class with an arbitrary, undeclared hash, permanently bricking the contract and freezing any user funds held within it. The OS itself acknowledges this missing check via an explicit TODO comment.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 877–916) reads the requested class hash directly from the syscall request and writes it into the contract's state entry without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing invariant enforcement. No check is performed against `contract_class_changes` or `compiled_class_facts` to confirm the new class hash is declared.

When a subsequent call is made to the bricked contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the class hash is undeclared, the lookup into `compiled_class_facts` fails. The contract becomes permanently uncallable — there is no recovery path in the OS.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 tokens, bridged assets, or staked funds held in a contract whose class hash has been replaced with an undeclared value are permanently inaccessible. The contract cannot execute any entry point (including withdrawal functions), and the OS provides no administrative override or recovery mechanism. The state transition is committed to the proven block output and is irreversible.

---

### Likelihood Explanation

The attack path is straightforward and requires only the ability to deploy a contract — an unprivileged, permissionless action on StarkNet:

1. Attacker deploys a contract that appears legitimate (e.g., a yield vault, staking pool, or bridge adapter).
2. Users deposit funds, trusting the contract's published source code.
3. The attacker calls a function (or a backdoored upgrade path) that issues `replace_class(invalid_hash)` where `invalid_hash` is any felt value not corresponding to a declared class.
4. The OS accepts the syscall without validation.
5. All subsequent calls to the contract fail at the OS level.
6. User funds are permanently frozen.

This is a realistic, low-cost attack. No leaked keys, no trusted-role compromise, and no network-level attack are required. The attacker only needs to be a contract deployer.

---

### Recommendation

In `execute_replace_class`, before updating the contract state, verify that `class_hash` exists as a key in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the returned compiled class hash is non-zero. This enforces the protocol invariant that `replace_class` can only target a declared class, consistent with the intent expressed in the existing TODO comment.

---

### Proof of Concept

1. Deploy contract `MaliciousVault` with:
   - A `deposit()` function that accepts user tokens.
   - A `rug()` function that calls `replace_class(0xdeadbeef)` — an undeclared hash.
2. Users call `deposit()`, transferring tokens into `MaliciousVault`.
3. Attacker calls `rug()`. The OS executes `execute_replace_class` and writes `class_hash=0xdeadbeef` into the contract's `StateEntry` with no validation. [3](#0-2) 
4. Any subsequent call to `MaliciousVault` (e.g., `withdraw()`) reaches `execute_entry_point`, which calls `dict_read` on `contract_class_changes` for `class_hash=0xdeadbeef`, gets compiled hash `0`, then `find_element` fails to locate a compiled class with hash `0`. [4](#0-3) 
5. All calls revert permanently. User funds are frozen with no recovery path.
