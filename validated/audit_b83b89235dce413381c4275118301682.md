### Title
`meta_tx_v0` Signature Replay Attack Due to Nonce Exclusion from Hash — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a version-0 transaction context with a hardcoded `nonce=0` and computes a transaction hash that contains no nonce or block-specific data. The OS nonce-increment function explicitly skips all version-0 transactions. As a result, any valid `meta_tx_v0` signature is permanently replayable across an unlimited number of outer transactions by any unprivileged caller, enabling repeated unauthorized execution of the signed action — including fund transfers.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` is a syscall that allows a contract to call another contract under a synthetic version-0 transaction context, supplying a custom signature. The OS computes the hash for this inner call via `compute_meta_tx_v0_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
``` [1](#0-0) 

The hash is a pure function of `(contract_address, selector, calldata, chain_id)`. No nonce, no block number, no timestamp is included.

Inside `execute_meta_tx_v0`, the synthetic `TxInfo` is assembled with `nonce=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // ← hardcoded zero
    ...
);
``` [2](#0-1) 

The OS nonce-enforcement function `check_and_increment_nonce` explicitly exempts version-0 transactions from any nonce check or increment:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

Because `execute_meta_tx_v0` never calls `check_and_increment_nonce` and the hash is nonce-free, the OS provides zero replay protection for `meta_tx_v0` calls. The same `(contract_address, selector, calldata, signature)` tuple produces an identical hash on every invocation, and the signature validates every time.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A contract that uses `meta_tx_v0` to authorize a fund transfer (e.g., a session-key contract, a paymaster, or any account abstraction pattern that delegates execution via this syscall) is permanently vulnerable to replay. An attacker who observes a valid `meta_tx_v0` invocation in any confirmed block can replay the identical inner call inside a new outer transaction (which carries its own fresh nonce, so the outer replay protection does not help). The target contract's `__execute__` entry point receives the same hash and the same signature, validates successfully, and executes the transfer again. This can be repeated indefinitely until the signer's account is drained.

---

### Likelihood Explanation

`meta_tx_v0` is a first-class OS syscall exposed to all contracts. Any account abstraction pattern, session-key scheme, or paymaster that relies on it for authorization — without independently implementing nonce tracking in contract storage — is immediately exploitable by any unprivileged transaction sender who can observe the L2 mempool or confirmed blocks. No privileged access, leaked key, or external dependency is required. The attacker's only requirement is to have seen one valid `meta_tx_v0` call.

---

### Recommendation

Include a nonce in `compute_meta_tx_v0_hash` and enforce it at the OS level, analogous to how `check_and_increment_nonce` works for version-1/3 transactions. Concretely:

1. Add a `nonce` parameter to `compute_meta_tx_v0_hash` and pass it as `additional_data` (size=1) to `deprecated_get_transaction_hash`.
2. In `execute_meta_tx_v0`, read the current nonce from `contract_state_changes` for `contract_address`, include it in the hash, and increment it after a successful call — mirroring the pattern in `check_and_increment_nonce`.
3. Remove the blanket version-0 nonce bypass in `check_and_increment_nonce`, or introduce a dedicated nonce-increment path for `meta_tx_v0`.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Victim account `A` signs a `meta_tx_v0` payload authorizing transfer of 100 tokens to address `B`. The hash is `H = hash(INVOKE_PREFIX, 0, A, __execute__, calldata, 0, chain_id)`.
2. A session-key contract `S` submits outer tx (nonce=N) calling `meta_tx_v0(A, __execute__, calldata, sig)`. The OS computes `H`, validates `sig`, and executes the transfer. 100 tokens leave `A`.
3. The attacker submits a new outer tx (nonce=N+1, different sender) calling `S` again with the identical `meta_tx_v0` parameters.
4. The OS recomputes `H` — identical, because no nonce is included. `sig` validates again. The transfer executes again. Another 100 tokens leave `A`.
5. Steps 3–4 repeat until `A` is drained.

The outer transaction nonces differ (preventing outer-level replay), but the inner `meta_tx_v0` hash is identical across all repetitions, providing no protection. [4](#0-3) [5](#0-4) [3](#0-2)

        return ();
