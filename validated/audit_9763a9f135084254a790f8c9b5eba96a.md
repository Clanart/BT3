### Title
Push-Only Token Delivery in `finTransfer` Permanently Freezes User Funds When Recipient Is Blacklisted - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.finTransfer` uses a push-only delivery model: it attempts to transfer tokens directly to `payload.recipient` in a single atomic call. If the token transfer reverts (e.g., because the recipient is blacklisted in USDC/USDT), the entire transaction reverts. Because there is no pull-pattern fallback, no alternative-recipient mechanism, and no origin-chain refund path, the user's funds locked on the origin chain become permanently irrecoverable.

---

### Finding Description

In `OmniBridge.finTransfer`, the nonce is marked used and then the token is pushed directly to the recipient:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC20(payload.tokenAddress).safeTransfer(             // line 351
    payload.recipient,
    payload.amount
);
```

If `safeTransfer` reverts (e.g., USDC/USDT blacklist), the whole transaction reverts and the nonce is not consumed. However, **every future attempt to call `finTransfer` for this transfer will also revert** for the same reason. There is no mechanism to:

1. Redirect the tokens to an alternative address.
2. Store the tokens in the contract for later pull-claim by the user.
3. Prove the destination failure back to the origin chain to unlock/refund the locked tokens.

The same pattern exists in `starknet/src/omni_bridge.cairo` `fin_transfer`:

```cairo
_set_transfer_finalised(ref self, payload.destination_nonce);  // line 250
// ...
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');                        // line 262
```

On the origin chain (NEAR), once `init_transfer` is processed, the tokens are locked or burned. The NEAR bridge has no "cancel transfer" or "refund on destination failure" function. `fin_transfer_send_tokens_callback` only handles NEAR-side `ft_transfer_call` refunds, not EVM-side delivery failures.

---

### Impact Explanation

A user who initiates a bridge transfer from NEAR (or another chain) to an EVM/StarkNet address that is blacklisted in the destination token (USDC, USDT, or any token with transfer restrictions) will have their funds permanently locked:

- Origin-chain tokens are burned/locked and cannot be recovered.
- Destination-chain `finTransfer` always reverts; the transfer can never be finalized.
- No admin or user action can recover the funds without a contract upgrade.

This matches: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

USDC and USDT both implement address blacklisting (OFAC compliance). A user's address can be blacklisted at any time after they initiate a bridge transfer but before `finTransfer` is executed. This is a realistic, non-hypothetical scenario that has occurred on other bridges. The user does not need to be malicious; they may simply be caught by a regulatory action.

---

### Recommendation

Implement a pull pattern for token delivery on the EVM and StarkNet sides:

1. If the direct token transfer to `payload.recipient` fails, store the claimable amount in a mapping: `claimable[payload.recipient][payload.tokenAddress] += payload.amount`.
2. Expose a `claim(address token)` function that lets the recipient (or a designated alternative address) withdraw their tokens later.
3. Alternatively, allow the MPC/NEAR bridge to sign a "redirect" message that changes the recipient before finalization, enabling recovery without a contract upgrade.

---

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR and initiates a bridge transfer to her EVM address `0xAlice` via `init_transfer`. NEAR bridge locks 10,000 USDC.
2. Before the relayer calls `finTransfer` on EVM, Circle blacklists `0xAlice` (e.g., OFAC action).
3. Relayer calls `OmniBridge.finTransfer(sig, payload)` where `payload.recipient = 0xAlice` and `payload.tokenAddress = USDC`.
4. `IERC20(USDC).safeTransfer(0xAlice, 10000e6)` reverts due to blacklist.
5. Entire transaction reverts. `completedTransfers[nonce]` is not set.
6. Every subsequent `finTransfer` call for this nonce reverts identically.
7. Alice's 10,000 USDC is permanently locked on NEAR with no refund path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn fin_transfer_send_tokens_callback(
        &mut self,
        #[serializer(borsh)] transfer_message: TransferMessage,
        #[serializer(borsh)] fee_recipient: &AccountId,
        #[serializer(borsh)] is_ft_transfer_call: bool,
        #[serializer(borsh)] storage_owner: &AccountId,
        #[serializer(borsh)] lock_actions: Vec<LockAction>,
    ) {
        let token = self.get_token_id(&transfer_message.token);

        if Self::is_refund_required(is_ft_transfer_call) {
            self.burn_tokens_if_needed(
                token.clone(),
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
            );

            self.revert_lock_actions(&lock_actions);

            self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);

            env::log_str(
                &OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string(),
            );
