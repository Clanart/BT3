### Title
Unchecked Constructor Failure in `execute_deploy` Syscall Always Reports Success — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` (success) into the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or failed. This is the direct Cairo/OS analog of the external report's unchecked-return-value class: a critical operation's outcome is silently discarded, and the caller is always told it succeeded.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function calls `deploy_contract` and then writes the response header with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns only `(retdata_size, retdata)` — there is no `is_reverted` return value, and the TODO comment explicitly acknowledges that failure propagation is not implemented. [2](#0-1) 

Contrast this with `contract_call_helper`, which handles `call_contract` correctly:

```cairo
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(...)
...
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The `call_contract` path properly propagates `is_reverted` into `failure_flag`, but `execute_deploy` does not. This is the same structural gap as the external report: one code path uses the safe/checked pattern, another does not.

---

### Impact Explanation

When a constructor reverts, the OS rolls back the constructor's storage writes via the revert log, but the contract address entry (with its class hash) remains registered in `contract_state_changes`. The calling contract, however, receives `failure_flag=0` and has no way to distinguish a successful deploy from a failed one.

A calling contract that:
1. Deploys a token or vault contract via the `deploy` syscall,
2. Receives the always-success response,
3. Immediately transfers funds to the deployed address,

will send funds to a contract whose constructor never completed — meaning access controls, owner fields, or initialization guards were never set. Depending on the deployed class, those funds may be permanently unrecoverable (**permanent freezing of funds**) or claimable by an arbitrary caller (**direct loss of funds**).

---

### Likelihood Explanation

The entry path is fully unprivileged: any account contract can issue a `deploy` syscall. A constructor can fail for many legitimate reasons (assertion failure, out-of-gas, inner call revert). Because the OS always reports success, any contract that defensively checks the deploy result before transferring funds is given a false guarantee. The TODO comment confirms the team is aware the failure path is unimplemented, meaning this is a live gap in the current production OS code, not a theoretical edge case.

---

### Recommendation

Mirror the pattern used in `contract_call_helper`: have `deploy_contract` return an `is_reverted` flag, and propagate it into the `ResponseHeader.failure_flag` field instead of hardcoding `0`. Until then, any contract relying on the deploy syscall's success/failure signal for fund-safety decisions is operating on incorrect data.

```cairo
// Correct pattern (analogous to contract_call_helper):
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

---

### Proof of Concept

1. Deploy a factory contract `F` on StarkNet.
2. `F.__execute__` calls the `deploy` syscall targeting class `C`, whose constructor contains `assert 1 = 0` (always reverts).
3. The OS executes the constructor, which reverts; the revert log rolls back constructor storage writes, but the contract address is still registered with class hash `C`.
4. The OS writes `ResponseHeader(gas=..., failure_flag=0)` — success — back into the syscall segment.
5. `F` reads `failure_flag == 0`, concludes the deploy succeeded, and calls `transfer(deployed_address, amount)` on the fee token.
6. Funds arrive at the deployed address. Because the constructor never ran, no withdrawal function is initialized; funds are permanently frozen.

The root cause is exclusively in the OS production file at the cited lines — no privileged role, no external dependency, no test-only path. [4](#0-3)
