### Title
Fast-Transfer Relayer Reimbursement Marked Finalised Before Token Transfer Confirmed - (File: near/omni-bridge/src/lib.rs)

### Summary

In `process_fin_transfer_to_other_chain`, when a fast-transfer relayer is to be reimbursed upon proof arrival, the bridge calls `send_tokens(...).detach()` (fire-and-forget) and then immediately calls `mark_fast_transfer_as_finalised(...)` in the same synchronous frame. The state transition to `finalised = true` is committed regardless of whether the token transfer to the relayer actually succeeds. If `send_tokens` fails, the fast transfer is permanently locked in the finalised state with no retry path, and the relayer's pre-funded tokens are irrecoverably frozen inside the bridge contract.

### Finding Description

In `process_fin_transfer_to_other_chain`, when a matching fast-transfer entry exists, the code path is:

```rust
if let Some(relayer) = recipient {
    self.send_tokens(
        token,
        relayer,
        U128(transfer_message.amount_without_fee()...),
        "",
    )
    .detach();                                          // ← result never checked
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // ← always runs
}
``` [1](#0-0) 

`mark_fast_transfer_as_finalised` sets `status.finalised = true` in persistent storage: [2](#0-1) 

Because `.detach()` schedules the promise without awaiting its result, the finalised flag is written before the NEAR runtime executes the token transfer. If `send_tokens` fails (e.g., the relayer's account lacks storage registration for the token, or the token contract panics), the fast transfer entry remains permanently finalised with no mechanism to retry or reverse the reimbursement.

The same pattern appears in `utxo_fin_transfer_fast` for the non-NEAR destination branch:

```rust
self.mark_fast_transfer_as_finalised(&fast_transfer.id());
// ...
self.send_tokens(fast_transfer.token_id.clone(), fast_transfer_status.relayer, amount, "")
    .detach();
``` [3](#0-2) 

Contrast this with `process_fin_transfer_to_near`, which correctly awaits the token transfer result via a callback chain before removing the fast transfer entry: [4](#0-3) 

### Impact Explanation

If `send_tokens` fails after `mark_fast_transfer_as_finalised` has committed:

- The fast transfer is permanently in `finalised = true` state; no code path can re-enter it.
- The relayer pre-funded the recipient (sent tokens out of their own balance during the fast-transfer leg) but receives no reimbursement.
- The equivalent token amount that arrived via the original proof remains locked inside the bridge contract with no withdrawal path.

This constitutes **permanent freezing / irrecoverable lock of relayer and protocol funds** in the bridge flow.

### Likelihood Explanation

`send_tokens` dispatches an `ft_transfer` (or `mint`) cross-contract call. Failure conditions include: the relayer account lacking storage registration for the specific token, the token contract reverting, or the bridge holding insufficient balance. While uncommon in normal operation, the relayer's storage registration for a given token is not enforced at fast-transfer initiation time, making the failure scenario reachable without any privileged action. Any proof submitter (trusted relayer) who triggers `fin_transfer` for a transfer that had a prior fast-transfer leg can land in this state if the relayer's token storage is absent.

### Recommendation

Replace the detached call + immediate finalisation pattern with a callback that only marks the fast transfer as finalised after confirming the token transfer succeeded, mirroring the `process_fin_transfer_to_near` approach:

```rust
self.send_tokens(token, relayer, amount, "")
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(RESOLVE_FAST_TRANSFER_GAS)
            .resolve_fast_transfer_other_chain(&fast_transfer.id()),
    );
// mark_fast_transfer_as_finalised moved into the callback
```

Apply the same fix to `utxo_fin_transfer_fast`.

### Proof of Concept

1. Relayer R executes a fast transfer for origin transfer T (EVM → Solana). R sends tokens to the recipient on NEAR and the bridge records `FastTransferStatus { finalised: false, relayer: R, ... }`.
2. R's NEAR account does not have storage registered for the bridged token (e.g., a newly deployed token where R never called `storage_deposit`).
3. Any proof submitter calls `fin_transfer` with the valid EVM proof for T. `fin_transfer_callback` → `process_fin_transfer_to_other_chain` is reached.
4. The bridge finds R's fast-transfer status, calls `send_tokens(token, R, amount, "").detach()`, then immediately calls `mark_fast_transfer_as_finalised(R's fast transfer id)`.
5. The `ft_transfer` to R fails (no storage). The finalised flag is already `true` in storage.
6. R's pre-funded tokens are permanently unrecoverable; the bridge holds the equivalent amount with no withdrawal path.
