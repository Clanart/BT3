### Title
Unvalidated `message` Field Causes Permanent Fund Loss When NEAR Recipient Lacks `ft_on_transfer` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a cross-chain transfer to a NEAR recipient with a non-empty `message` field, the bridge dispatches `ft_transfer_call` (or `mint` with message) to the recipient. If the recipient contract does not implement `ft_on_transfer`, the call panics, tokens are returned to the bridge, and the bridge then **burns them** (for deployed tokens) or **permanently locks them** (for native tokens). Because the `message` is immutably embedded in the on-chain proof, re-submission always reproduces the same failure, making the loss irrecoverable.

---

### Finding Description

The `send_tokens` function in `near/omni-bridge/src/lib.rs` dispatches tokens to the NEAR recipient in one of two ways depending on whether a `msg` is present:

- **Deployed tokens + non-empty `msg`**: calls `mint(recipient, amount, Some(msg))` on the token contract, which internally calls `ft_transfer_call` → `ft_on_transfer` on the recipient.
- **Non-deployed tokens + non-empty `msg`**: calls `ft_transfer_call(recipient, amount, None, msg)` directly on the token contract, which calls `ft_on_transfer` on the recipient. [1](#0-0) 

If the recipient does not implement `ft_on_transfer`, the call panics. NEAR's NEP-141 `ft_resolve_transfer` then refunds the tokens back to the bridge and returns `0` as the used amount.

The callback `fin_transfer_send_tokens_callback` checks `is_refund_required`: [2](#0-1) 

When `amount.0 == 0` (all tokens refunded, i.e., `ft_on_transfer` panicked), `is_refund_required` returns `true`. The callback then:

1. Calls `burn_tokens_if_needed` — **burns the tokens from the bridge's balance** for deployed tokens.
2. Calls `remove_fin_transfer` — reverts the NEAR-side finalization record. [3](#0-2) 

`remove_fin_transfer` allows the same proof to be re-submitted. However, the `message` is immutably embedded in the source-chain `InitTransfer` event (EVM, Solana, etc.) and is part of the signed/proven payload. Re-submitting the same proof always carries the same `message`, causing the same panic, the same burn, and the same revert — an infinite failure loop.

The `message` field is user-supplied with no validation that the recipient implements `ft_on_transfer`: [4](#0-3) 

The `process_fin_transfer_to_near` function passes `transfer_message.msg` directly to `send_tokens` without any check: [5](#0-4) 

---

### Impact Explanation

**For deployed bridge tokens (e.g., wETH minted on NEAR):**
- Source-chain tokens (ETH) are burned/locked on EVM.
- NEAR mints tokens to recipient, `ft_on_transfer` panics, tokens return to bridge, bridge burns them.
- Re-submission repeats the mint-and-burn cycle.
- Source-chain tokens are **permanently locked** with no recovery path for the user.

**For native/locked tokens (e.g., USDC locked on EVM, held in bridge on NEAR):**
- USDC is locked on EVM.
- NEAR `ft_transfer_call` fails, USDC returns to bridge's NEAR balance.
- Re-submission always fails; USDC is **permanently locked** in the NEAR bridge contract.

In both cases the user's funds are irrecoverably lost without DAO intervention. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Any unprivileged bridge user can trigger this by:
1. Initiating a transfer to NEAR with a non-empty `message` field (e.g., to trigger a DeFi hook on arrival).
2. Specifying a recipient that does not implement `ft_on_transfer` (e.g., a plain wallet, a contract without the NEP-141 receiver interface, or a contract that panics in `ft_on_transfer`).

The `message` field is documented as an optional "destination-chain hook payload" and is explicitly supported by the protocol. Users integrating with DeFi protocols on NEAR are likely to use it. There is no on-chain validation that the recipient supports `ft_on_transfer` before the source-chain tokens are committed.

---

### Recommendation

1. **Validate recipient capability before committing source-chain funds**: On the source chain (EVM `initTransfer`, NEAR `ft_on_transfer`), if a `message` is non-empty, require that the recipient is a whitelisted or known-compatible contract address.
2. **Alternatively, on NEAR finalization**: Before calling `ft_transfer_call`, verify that the recipient has registered storage (a proxy for NEP-141 receiver support) and emit a clear error without burning tokens if the check fails, preserving the ability to recover.
3. **Recovery path**: If `ft_on_transfer` fails, do not burn the tokens. Instead, hold them in escrow and allow the user to claim them back (with a different recipient or no message) via a separate recovery function, rather than burning or permanently locking them.
4. **Documentation**: Clearly document that specifying a `message` requires the NEAR recipient to implement `ft_on_transfer`, and that failure to do so results in permanent fund loss.

---

### Proof of Concept

**Attacker/user path (EVM → NEAR with message):**

1. User calls `OmniBridge.initTransfer(tokenAddress, amount, fee, nativeFee, "near:bad-contract.near", "{\"action\":\"swap\"}")` on EVM. Tokens are burned/locked. `InitTransfer` event is emitted with `message = "{\"action\":\"swap\"}"`. [6](#0-5) 

2. Relayer submits proof to NEAR `fin_transfer`. `fin_transfer_callback` reconstructs the `TransferMessage` with `msg = "{\"action\":\"swap\"}"`. [7](#0-6) 

3. `process_fin_transfer_to_near` calls `send_tokens(token, "bad-contract.near", amount, "{\"action\":\"swap\"}")`. [5](#0-4) 

4. `send_tokens` calls `ft_transfer_call("bad-contract.near", amount, None, "{\"action\":\"swap\"}")`. `bad-contract.near` does not implement `ft_on_transfer` → panics. NEP-141 `ft_resolve_transfer` returns `0` (used amount). [8](#0-7) 

5. `fin_transfer_send_tokens_callback` is called. `is_refund_required` returns `true` (used amount == 0). `burn_tokens_if_needed` burns the tokens (deployed token case). `remove_fin_transfer` reverts NEAR finalization. [9](#0-8) 

6. User attempts to re-submit the same proof → same `message` → same panic → same burn. Source-chain tokens are permanently lost.

            extensionValue = msg.value - nativeFee;

                    amount

                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }

            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
