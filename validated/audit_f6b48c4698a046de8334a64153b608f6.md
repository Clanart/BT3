### Title
Deploy Syscall Unconditionally Reports Success Regardless of Constructor Failure, Enabling Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall implementation in the StarkNet OS always writes `failure_flag=0` (success) in the response header, even when the deployed contract's constructor execution fails and its state changes are reverted. A calling contract has no protocol-level way to detect the failure and may proceed to transfer tokens or other assets to the address of a contract that was never successfully deployed, permanently locking those funds.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 452–556), after invoking `deploy_contract`, the OS unconditionally writes a success response:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The function signature includes `revert_log: RevertLogEntry*` as an implicit argument, confirming that constructor execution can fail and that state changes can be rolled back via the revert log. Despite this, the failure is never communicated to the calling contract — `failure_flag` is hardcoded to `0` regardless of outcome.

The `DeployResponse` written immediately after also unconditionally includes the computed `contract_address`:

```cairo
assert [response] = DeployResponse(
    contract_address=contract_address,
    constructor_retdata_start=retdata,
    constructor_retdata_end=retdata + retdata_size,
);
```

When the constructor fails and the revert log unwinds the state changes, the contract at `contract_address` ends up with `class_hash=0` (no deployed code). The calling contract, however, receives `failure_flag=0` and a valid-looking `contract_address`, and has no mechanism to distinguish this from a successful deployment.

The explicit developer TODO — `// TODO(Yoni, 1/1/2026): support failures.` — confirms this is a known incomplete implementation that has been left in a broken state in production code.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A calling contract that uses the `deploy` syscall and subsequently transfers tokens or assets to the returned `contract_address` will send those assets to an address with no deployed class. Because there is no contract code at that address, no withdrawal or recovery function can ever be invoked. The assets are permanently locked.

This is a direct protocol-level analog to the Rubicon H-02 finding: in that case, a router contract became the owner of an offer with no cancel path; here, the OS deploy syscall becomes the authority that reports success with no failure path, causing callers to act on false information and lock funds irrecoverably.

---

### Likelihood Explanation

The `deploy` syscall is a standard, publicly accessible syscall available to any Cairo 1 contract. Any contract that:
1. Calls `deploy` with a constructor that can fail (e.g., due to an assertion, out-of-gas, or any revert condition), and
2. Subsequently transfers tokens or assets to the returned address (a common pattern in factory contracts, vault deployers, escrow systems, etc.)

is vulnerable. No privileged access, leaked key, or malicious operator is required. The failure can be triggered accidentally by any user who causes the constructor to revert (e.g., by supplying edge-case calldata).

---

### Recommendation

Implement proper failure propagation in `execute_deploy`. Specifically:

1. `deploy_contract` should return an `is_reverted` flag indicating whether the constructor succeeded.
2. If `is_reverted != 0`, write `ResponseHeader(gas=remaining_gas, failure_flag=1)` and a `FailureReason` object instead of the `DeployResponse`.
3. The calling contract can then read `failure_flag` and handle the failure (e.g., by reverting its own execution or taking a recovery path).

This mirrors how `contract_call_helper` already handles reverts for `call_contract` and `library_call` syscalls.

---

### Proof of Concept

**Step-by-step attack path (no privileged access required):**

1. A factory contract `F` is deployed on StarkNet. Its logic is: call `deploy(class_hash_of_V, calldata)`, then call `token.transfer(deployed_address, user_amount)`.
2. A user calls `F` with `calldata` that causes the constructor of vault contract `V` to revert (e.g., an assertion fails, or gas runs out).
3. The OS executes `execute_deploy`:
   - `deploy_contract` is called; the constructor reverts; the revert log unwinds state changes; `contract_address` has `class_hash=0`.
   - The OS writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` — **always success**.
   - `F` reads `failure_flag=0` and the valid-looking `contract_address`.
4. `F` proceeds to call `token.transfer(contract_address, user_amount)`.
5. The token contract updates its storage: `balances[contract_address] += user_amount`. This state change is committed.
6. The outer transaction completes successfully and is included in the block.
7. `user_amount` tokens now reside at `contract_address`, which has `class_hash=0` and no executable code.
8. **The tokens are permanently frozen.** No withdrawal function exists; no recovery is possible.
