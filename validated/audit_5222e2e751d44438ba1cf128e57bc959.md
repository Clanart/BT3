### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Enables Complete Fee Bypass - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in the StarkNet OS performs unchecked felt-field arithmetic on user-supplied resource bound values. Because Cairo felt arithmetic is implicitly modulo the Stark prime `P ≈ 2^251 + 17·2^192 + 1`, an unprivileged transaction sender can craft `ResourceBounds` fields whose products sum to exactly `P ≡ 0 (mod P)`. The `charge_fee` function then observes `max_fee == 0` and returns immediately, charging no fee at all. The resulting OS execution is provably valid, the proof is accepted by the L1 verifier, and the sequencer receives nothing for processing the transaction.

---

### Finding Description

`compute_max_possible_fee` computes the upper bound on what can be charged for a v3 transaction:

```cairo
// transaction_impls.cairo lines 87-102
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
         + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
         + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
}
```

All three terms are plain felt multiplications and additions — no `assert_nn_le` or range-check guards are applied to the individual `max_amount` or `max_price_per_unit` fields before or after the arithmetic. Felt values are elements of `[0, P)`, so any product or sum that reaches or exceeds `P` silently wraps.

`charge_fee` then uses the result as the sole gate:

```cairo
// transaction_impls.cairo lines 121-125
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
```

If `max_fee` wraps to `0`, the function returns immediately — no ERC-20 transfer is executed, no fee is deducted from the sender, and no payment reaches the sequencer.

**Concrete overflow construction:**

Choose resource bounds as follows (all values are valid felts in `[0, P)`):

| Field | Value |
|---|---|
| `l1_gas.max_amount` | `P − 1` |
| `l1_gas.max_price_per_unit` | `1` |
| `l2_gas.max_amount` | `1` |
| `l2_gas.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas.max_amount` | `0` |
| `l1_data_gas.max_price_per_unit` | `0` |

Felt computation:

```
(P−1)·1  +  1·(1+0)  +  0·0
= (P−1) + 1
= P
≡ 0  (mod P)
```

`compute_max_possible_fee` returns `0`. `charge_fee` exits at line 123 without charging anything.

The attacker signs this transaction normally (the transaction hash commits to these resource bound values), submits it, and the sequencer — whose off-chain Rust code computes the product in 256-bit integers and sees a large but non-zero `max_fee` — may include it. The OS then proves execution with zero fee charged, and the L1 verifier accepts the proof.

There are no upstream guards in `get_account_tx_common_fields`, `fill_account_tx_info`, or `assert_deprecated_tx_fields_consistency` that bound `max_amount` or `max_price_per_unit` to values that prevent overflow in their product. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

**Direct loss of funds (Critical):** The sequencer processes real computation (validate + execute entry points, state updates, proof generation) but receives zero fee. At scale, an attacker can flood the network with crafted zero-fee transactions, draining sequencer resources without compensation.

**Network shutdown (High):** Because the OS enforces no minimum fee and the proof is valid, the sequencer has no on-chain mechanism to recoup costs. Sustained zero-fee spam can exhaust sequencer capacity, preventing legitimate transactions from being confirmed — a total network shutdown.

---

### Likelihood Explanation

- The attacker is an ordinary transaction sender; no privileged role is required.
- The crafted values (`P−1`, `1`, `1`, `1`, `0`, `0`) are all valid felt field elements and pass every existing structural check in the OS.
- The transaction hash correctly commits to these values, so signature verification succeeds.
- The sequencer's off-chain Rust arithmetic computes a large (non-zero) `max_fee` and may include the transaction without suspicion.
- The discrepancy between the sequencer's 256-bit arithmetic and the OS's felt arithmetic is the root cause and is not currently detected at any layer.

---

### Recommendation

1. **Add overflow guards on resource bound fields.** Before computing the fee, assert that each `max_amount` and `max_price_per_unit` fits within a safe range (e.g., 64-bit or 128-bit), using `assert_nn_le`:

```cairo
assert_nn_le(l1_gas_bounds.max_amount, MAX_RESOURCE_BOUND);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_PRICE_PER_UNIT);
// ... repeat for l2 and l1_data bounds
```

2. **Assert `max_fee > 0` for non-trivial transactions.** If the transaction has non-zero resource bounds, the computed `max_fee` must be non-zero; treat a zero result as an invalid transaction rather than a fee-free one.

3. **Align sequencer and OS arithmetic.** The sequencer's off-chain validation should replicate felt-modular arithmetic exactly so that any transaction the sequencer accepts is guaranteed to produce a non-zero `max_fee` in the OS.

### Proof of Concept

An attacker submits a v3 Invoke transaction with the following resource bounds (expressed as Cairo struct values):

```
resource_bounds[L1_GAS_INDEX]      = ResourceBounds { max_amount: P-1, max_price_per_unit: 1 }
resource_bounds[L2_GAS_INDEX]      = ResourceBounds { max_amount: 1,   max_price_per_unit: 1 }
resource_bounds[L1_DATA_GAS_INDEX] = ResourceBounds { max_amount: 0,   max_price_per_unit: 0 }
tip = 0
```

Where `P = 0x800000000000011000000000000000000000000000000000000000000000001`.

**OS execution trace:**

1. `execute_invoke_function_transaction` calls `get_account_tx_common_fields` — resource bounds loaded from hint, no range checks applied. [3](#0-2) 

2. Transaction hash is computed and signature verified — passes, because the hash correctly covers the crafted values.

3. `charge_fee` is called. `compute_max_possible_fee` evaluates:
   `(P−1)·1 + 1·(1+0) + 0·0 = P ≡ 0 (mod P)`. [4](#0-3) 

4. `max_fee == 0` → `charge_fee` returns immediately. No ERC-20 transfer is executed. [5](#0-4) 

5. The OS produces a valid proof. The L1 verifier accepts the state update. The attacker's transaction executed for free.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
