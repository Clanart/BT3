### Title
Missing Zero-Address Recipient Validation in `init_transfer` Causes Permanent Irrecoverable Lock of User Funds — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR bridge's `init_transfer` function accepts any `OmniAddress` as a recipient as long as its chain kind is not `Near`. It never calls `OmniAddress::is_zero()`, which exists in the type library. A user who supplies `OmniAddress::Eth(H160::ZERO)` (or any other chain's zero address) as the recipient will have their tokens permanently locked on NEAR with no recovery path, because the resulting MPC-signed `finTransfer` call on EVM will always revert for ERC20 and bridge tokens.

---

### Finding Description

`near/omni-bridge/src/lib.rs::init_transfer` performs exactly one recipient validation:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

`OmniAddress::is_zero()` is fully implemented in the type library and correctly identifies zero addresses across all chain variants:

```rust
pub fn is_zero(&self) -> bool {
    match self {
        Self::Eth(address) | Self::Arb(address) | ... => address.is_zero(),
        Self::Near(address) => *address == ZERO_ACCOUNT_ID,
        Self::Sol(address) | Self::Fogo(address) => address.is_zero(),
        ...
    }
}
``` [2](#0-1) 

It is never called in `init_transfer`. The zero address for each chain is also constructible via `OmniAddress::new_zero()`: [3](#0-2) 

The transfer message is stored with the zero-address recipient and the tokens are locked: [4](#0-3) 

On the EVM side, `finTransfer` marks the destination nonce as used **before** the token transfer (CEI pattern), then dispatches to the appropriate token handler: [5](#0-4) 

For bridge tokens, it calls `IBridgeToken.mint(payload.recipient, payload.amount)`: [6](#0-5) 

For regular ERC20 tokens, it calls `safeTransfer(payload.recipient, payload.amount)`: [7](#0-6) 

Both OpenZeppelin `_mint` and `safeTransfer` revert when `to == address(0)`. Because the entire transaction reverts, the nonce marking at line 287 also reverts — the nonce is never consumed. However, the MPC signature is bound to `recipient = address(0)` (it is part of the Borsh-encoded payload that was signed), so no valid `finTransfer` call can ever succeed for this transfer. The NEAR tokens remain locked forever.

There is no `cancel_transfer`, `refund_transfer`, or timeout-based recovery function anywhere in `near/omni-bridge/src/lib.rs`.

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

When a user initiates a NEAR → EVM transfer with a zero EVM recipient:

- NEAR tokens are locked in the bridge contract.
- The MPC signs a payload committing `recipient = address(0)`.
- Every subsequent `finTransfer` call on EVM reverts (for ERC20 and bridge tokens), so the nonce is never consumed and the transfer is never finalized.
- No cancel or refund path exists on NEAR.
- The locked tokens are permanently unclaimable.

For native ETH transfers (`tokenAddress == address(0)`), the ETH `.call` to `address(0)` succeeds (ETH is burned), the nonce is consumed, and the NEAR tokens remain locked — a different but equally permanent loss.

---

### Likelihood Explanation

**Low-to-medium.** `OmniAddress::Eth(H160::ZERO)` is a syntactically valid, parseable address that passes all current validation. It can be submitted by:

- A dApp with an uninitialized or default-value address field.
- A user copy-pasting a placeholder zero address.
- A programmatic caller that constructs `InitTransferMsg` without validating the recipient.

The `OmniAddress::new_zero()` constructor makes it trivially constructible. No special privilege or chain access is required — any user calling `ft_transfer_call` on NEAR can trigger this.

---

### Recommendation

Add a zero-address guard in `init_transfer` immediately after the chain-kind check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
require!(
    !init_transfer_msg.recipient.is_zero(),
    BridgeError::InvalidRecipient.as_ref()  // add this error variant
);
``` [1](#0-0) 

Apply the same guard in the EVM `initTransfer` for the `recipient` string (reject empty string or known zero-address string representations), and add a `require(payload.recipient != address(0))` guard in `finTransfer` as defense-in-depth: [8](#0-7) [9](#0-8)

// 1. Calls ft_transfer_call on a NEP-141 token with:

// 2. NEAR bridge accepts it — no is_zero() check.

//    completedTransfers[nonce] = true  ← set

// 5. finTransfer can never succeed (signature binds recipient = address(0)).
//    NEAR tokens locked forever. No cancel path exists.
