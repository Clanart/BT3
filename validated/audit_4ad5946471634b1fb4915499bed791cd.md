### Title
Unvalidated L1 WEI Gas Prices in `ProposalInit` Allow Malicious Proposer to Commit Blocks with Incorrect ETH Gas Prices - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `is_block_info_valid` function validates the FRI gas prices from a proposer's `ProposalInit` against the local oracle but silently discards the computed WEI gas prices, leaving `l1_gas_price_wei` and `l1_data_gas_price_wei` entirely unvalidated. A malicious proposer can set arbitrary WEI prices while keeping FRI prices within the accepted margin. All validators will accept the proposal, execute transactions with the manipulated ETH gas prices, and commit the block — with the `ProposalFinMismatch` check passing because both sides use the proposer-supplied WEI values.

---

### Finding Description

In `is_block_info_valid`, the validator checks `l1_gas_price_fri` and `l1_data_gas_price_fri` against its local oracle using `within_margin`. The call to `get_l1_prices_in_fri_and_wei` returns both FRI and WEI prices, but the WEI result is explicitly discarded via the `_l1_gas_prices_wei` binding:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
``` [1](#0-0) 

Only the FRI prices are then checked:

```rust
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
``` [2](#0-1) 

The proposer-supplied `l1_gas_price_wei` and `l1_data_gas_price_wei` are never compared against the validator's locally computed WEI prices.

After `is_block_info_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info`, which directly uses the unvalidated WEI values to populate `eth_gas_prices` in the `BlockInfo` sent to the batcher:

```rust
let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
``` [3](#0-2) 

The batcher then executes all transactions — including L1 handler transactions — using these manipulated ETH prices.

The `ProposalFinMismatch` check at the end of `validate_proposal` compares the batcher's computed `partial_block_hash` against the proposer's `received_fin.proposal_commitment`:

```rust
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
``` [4](#0-3) 

Because the block hash includes WEI gas prices (via `gas_prices_to_hash` over `GasPricePerToken` which carries both `price_in_fri` and `price_in_wei`):

```rust
l1_gas_price: block_info.gas_prices.l1_gas_price_per_token(),
l1_data_gas_price: block_info.gas_prices.l1_data_gas_price_per_token(),
``` [5](#0-4) 

both the proposer and every validator compute the block hash using the same proposer-supplied WEI values. The hashes match, the mismatch check passes, and the block is committed with the incorrect ETH gas prices.

---

### Impact Explanation

The `eth_gas_prices` field in `BlockInfo` (populated from the unvalidated WEI values) is used by the blockifier for:

1. **L1 handler transaction fee validation** — the fee paid on L1 (in ETH) is compared against the required fee computed using `eth_gas_prices.l1_gas_price`. Setting `l1_gas_price_wei = 1` makes every L1 handler transaction appear to require only 1 wei per gas unit, allowing L1 handler transactions that paid an insufficient fee on L1 to pass fee validation and be included in the block.
2. **ETH-denominated fee accounting** — all resource-bound and fee calculations that reference the ETH price path are corrupted.

This matches the allowed impact: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

Any validator can be elected proposer in the Tendermint-style consensus used by the sequencer. A single malicious validator, on any round where it is chosen as proposer, can craft a `ProposalInit` with valid FRI prices (passing the margin check) and arbitrary WEI prices. No special access beyond validator status is required. The manipulation is invisible to other validators because the WEI validation gap is structural, not conditional.

---

### Recommendation

In `is_block_info_valid`, use the locally computed WEI prices to validate the proposer's WEI prices, analogously to the existing FRI validation:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

// existing FRI checks ...

let l1_gas_price_wei = l1_gas_prices_wei.l1_gas_price;
let l1_data_gas_price_wei = l1_gas_prices_wei.l1_data_gas_price;
if !(within_margin(init_proposed.l1_gas_price_wei, l1_gas_price_wei, margin)
    && within_margin(init_proposed.l1_data_gas_price_wei, l1_data_gas_price_wei, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

### Proof of Concept

1. Malicious proposer constructs `ProposalInit` with:
   - `l1_gas_price_fri = X` — valid, within the oracle margin
   - `l1_data_gas_price_fri = Y` — valid, within the oracle margin
   - `l1_gas_price_wei = 1` — arbitrary, far below the true ETH price
   - `l1_data_gas_price_wei = 1` — arbitrary

2. Proposer computes `partial_block_hash` using these values (including `l1_gas_price_wei = 1`) and broadcasts `ProposalFin` with this commitment.

3. Each validator calls `validate_proposal` → `is_block_info_valid`:
   - FRI margin check passes ✓
   - WEI prices are **not checked** — `_l1_gas_prices_wei` is discarded ✓ (bug)

4. `initiate_validation` → `convert_to_sn_api_block_info` sets `eth_gas_prices.l1_gas_price = 1 wei`.

5. Batcher executes L1 handler transactions. Each transaction's required ETH fee is computed as `gas_used × 1 wei`, which is negligible. L1 handler transactions that paid only a dust fee on L1 pass fee validation and are included.

6. Batcher computes `partial_block_hash` using `l1_gas_price_wei = 1` — identical to the proposer's computation.

7. `built_block == received_fin.proposal_commitment` → `ProposalFinMismatch` check passes.

8. `decision_reached` commits the block. The chain now contains a block where all L1 handler transactions were executed with `l1_gas_price_wei = 1`, bypassing the ETH fee requirement enforced on L1.
