### Title
Hardcoded `failure_flag=0` in `execute_deploy` Syscall Silently Swallows Constructor Failures — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS always writes `failure_flag=0` in the response header, regardless of whether the deployed contract's constructor actually succeeded or failed. A calling contract that relies on `failure_flag` to gate subsequent fund transfers will be systematically misled, potentially sending funds to an uninitialized or partially-initialized contract from which they cannot be recovered.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function handles the `deploy` syscall. After calling `deploy_contract`, it unconditionally writes a success response:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `failure_flag` field is the standard StarkNet protocol mechanism by which a calling contract determines whether a syscall succeeded. For `call_contract`, the OS correctly propagates `failure_flag=is_reverted`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

But for `deploy`, the flag is hardcoded to `0`. The `deploy_contract` call does carry `revert_log` as an implicit argument (indicating it is designed to handle constructor reverts gracefully and return normally), yet the return value carries no `is_reverted` signal, and the response header never reflects a failure:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [3](#0-2) 

The `revert_log` implicit argument is present in `execute_deploy`'s signature, confirming the OS is designed to handle constructor reverts without aborting the whole transaction: [4](#0-3) 

The analogous path in `transaction_impls.cairo` (deploy-account) also uses `revert_log`, confirming `deploy_contract` is expected to return normally on constructor failure: [5](#0-4) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

Any on-chain factory or proxy-deployment pattern that:
1. Calls the `deploy` syscall,
2. Reads `failure_flag` from the response to decide whether to proceed, and
3. Transfers tokens or ETH to the freshly-deployed address on success,

will be systematically deceived. Because `failure_flag` is always `0`, the calling contract cannot distinguish a successful deployment from a failed one. If the constructor reverted (e.g., due to an invalid argument, an access-control check, or an out-of-gas condition), the calling contract proceeds as if deployment succeeded and sends funds to an address whose contract state was rolled back by the revert log. Those funds are permanently locked in an uninitialized contract with no recovery path.

---

### Likelihood Explanation

Factory-contract patterns (deploy-then-fund) are ubiquitous in DeFi (token vaults, account abstraction wallets, AMM pair creation). Any such contract deployed on StarkNet that checks `failure_flag` after a `deploy` syscall is affected. Constructor failures can be triggered by:
- Supplying invalid constructor arguments (attacker-controlled input to a factory),
- Gas exhaustion during a complex constructor,
- Any assertion inside the constructor that evaluates to false.

An unprivileged user can trigger this by interacting with any factory contract that accepts user-supplied class hashes or constructor arguments, causing the constructor to fail while the factory still transfers funds.

---

### Recommendation

Replace the hardcoded `failure_flag=0` with the actual constructor execution result. `deploy_contract` should be refactored to return an `is_reverted` flag (analogous to `execute_entry_point`), and the response header should be written as:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This mirrors the correct pattern already used in `contract_call_helper` for `call_contract` responses. [6](#0-5) 

---

### Proof of Concept

1. Deploy a factory contract `F` on StarkNet. `F` implements:
   ```
   func create_and_fund(class_hash, salt, calldata):
       let (addr) = deploy(class_hash, salt, calldata, deploy_from_zero=FALSE)
       // failure_flag is always 0 — no revert check possible
       transfer(token, addr, amount)   // funds sent unconditionally
   ```
2. Provide a `class_hash` whose constructor always reverts (e.g., `assert 1 = 0`).
3. Call `F.create_and_fund(...)` with that class hash.
4. The OS processes the `deploy` syscall: the constructor reverts, the revert log rolls back constructor state changes, but `failure_flag=0` is written to the response.
5. `F` reads `failure_flag=0`, concludes deployment succeeded, and calls `transfer(token, addr, amount)`.
6. `addr` holds the class hash but has no initialized storage. The transferred tokens are permanently locked. [7](#0-6)
