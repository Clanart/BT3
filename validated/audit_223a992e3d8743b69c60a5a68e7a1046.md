### Title
Unchecked ERC20 Fee Transfer Return Value in `charge_fee` Allows Silent Fee Payment Failure — (`execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in `transaction_impls.cairo` invokes the ERC20 `transfer` entry point via `non_reverting_select_execute_entry_point_func` but **completely discards the return value**. The ERC20 `transfer` function returns a boolean success flag in its `retdata`. If the fee token's `transfer` returns `false` without reverting, the OS proceeds as if the fee was successfully charged, allowing transactions to be processed without the sequencer receiving payment.

---

### Finding Description

In `charge_fee` (`transaction_impls.cairo`, lines 161–163):

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
```

The function `non_reverting_select_execute_entry_point_func` is declared to return `(retdata_size: felt, retdata: felt*, is_deprecated: felt)` (`execute_transaction_utils.cairo`, lines 188–189). The `retdata` contains the ERC20 `transfer` function's boolean return value. However, in `charge_fee`, the return tuple is **not captured at all** — the call is a bare statement with no `let (...)` binding.

Contrast this with every other call site of the same function in the same file, where the return value **is** captured and validated:

- `run_validate` (`execute_transaction_utils.cairo`, lines 149–155): `let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(...)` → `assert retdata[0] = VALIDATED`
- `execute_deploy_account_transaction` (`transaction_impls.cairo`, lines 677–683): same pattern, asserts `retdata[0] = VALIDATED`
- `execute_declare_transaction` (`transaction_impls.cairo`, lines 804–811): same pattern, asserts `retdata[0] = VALIDATED`

The `non_reverting_select_execute_entry_point_func` wrapper (`execute_transaction_utils.cairo`, lines 192–195) only asserts `is_reverted = 0`, meaning it catches hard reverts. It does **not** inspect `retdata`. A fee token that returns `false` (ERC20-compliant failure signaling) without reverting passes the `is_reverted = 0` assertion silently.

The fee token address is read from `block_context.os_global_context.starknet_os_config.fee_token_address` (`transaction_impls.cairo`, line 138). The `TRANSFER_ENTRY_POINT_SELECTOR` (`constants.cairo`, lines 49–52) is used as the selector. The calldata is a `TransferCallData` struct (`transaction_impls.cairo`, lines 81–84) containing the sequencer as recipient and `low_actual_fee` as the amount.

---

### Impact Explanation

If the fee token's `transfer` returns `false` without reverting:

1. The sequencer receives **zero fee tokens** despite the OS recording the transaction as fee-charged.
2. The transaction's state changes are committed normally.
3. The sequencer suffers a **direct loss of funds** equal to the actual fee that should have been collected.
4. This applies to all transaction types: invoke (`execute_invoke_function_transaction`, line 361), deploy-account (`execute_deploy_account_transaction`, line 687), and declare (`execute_declare_transaction`, line 822–824).

This matches the **Critical: Direct loss of funds** impact category.

---

### Likelihood Explanation

The current production fee tokens (STRK, ETH-wrapped) revert on transfer failure, so the `is_reverted = 0` assertion catches failures for those tokens. However:

- The fee token address is a **configurable protocol parameter** (`starknet_os_config.fee_token_address`), not hardcoded.
- Any future fee token that follows the ERC20 standard of returning `false` on failure (rather than reverting) would trigger this silently.
- The inconsistency with every other `non_reverting_select_execute_entry_point_func` call site — all of which check `retdata` — indicates this is an unintentional omission rather than a deliberate design choice.

Likelihood is **medium**: not immediately exploitable with current fee tokens, but structurally present and triggered by any ERC20-compliant fee token that signals failure via return value.

---

### Recommendation

Capture and validate the return value of the fee transfer call in `charge_fee`, consistent with all other call sites:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
// Verify the ERC20 transfer succeeded (returned true / non-zero).
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = 1;  // ERC20 transfer must return true.
}
```

### Proof of Concept

**Root cause location:** [1](#0-0) 

The return value is silently dropped here. Compare with the validated pattern used in `run_validate`: [2](#0-1) 

The `non_reverting_select_execute_entry_point_func` wrapper only asserts `is_reverted = 0` and does not inspect `retdata`: [3](#0-2) 

The ERC20 transfer is constructed with `TRANSFER_ENTRY_POINT_SELECTOR` targeting the fee token: [4](#0-3) 

**Attack path:**

1. A fee token contract is deployed/configured whose `transfer` function returns `felt 0` (false) when the sender has insufficient balance, instead of reverting.
2. An unprivileged user submits any V3 transaction (invoke, declare, or deploy-account) with `max_fee > 0` but with a balance below `actual_fee`.
3. The OS calls `charge_fee`, which constructs the `TransferCallData` and calls `non_reverting_select_execute_entry_point_func`.
4. The fee token's `transfer` executes, returns `[0]` in `retdata`, and does **not** set `failure_flag = 1`.
5. `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` — passes.
6. `charge_fee` returns without checking `retdata`.
7. The transaction is finalized. The sequencer's balance is unchanged. The user paid nothing.
