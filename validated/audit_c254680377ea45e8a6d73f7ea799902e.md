### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Breaks Bridge Collateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` trusts the caller-supplied `amount` parameter for non-bridge ERC20 tokens without verifying the actual amount received by the contract. For fee-on-transfer tokens, the bridge receives fewer tokens than `amount` but emits an `InitTransfer` event crediting the full `amount`. The NEAR side then mints or releases the full `amount` to the recipient, permanently undercollateralizing the EVM bridge.

---

### Finding Description

In `OmniBridge.initTransfer`, when `tokenAddress` is a non-bridge, non-custom-minter ERC20 token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // user-supplied
);
``` [1](#0-0) 

Immediately after, the same user-supplied `amount` is emitted in the `InitTransfer` event and forwarded to `initTransferExtension`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

There is no before/after balance snapshot to determine the actual amount received. For fee-on-transfer tokens (tokens that deduct a percentage during `transferFrom`), the bridge receives `amount - transfer_fee` but the event records `amount`. The NEAR bridge reads this event, signs a `TransferMessage` for the full `amount`, and the recipient claims the full `amount` on NEAR — while the EVM vault is short by `transfer_fee`.

The `logMetadata` function is fully permissionless:

```solidity
function logMetadata(address tokenAddress) external payable {
``` [3](#0-2) 

This means any caller can register an arbitrary token (including a custom fee-on-transfer token) for bridging. Once the NEAR relayer processes the `LogMetadata` event and deploys the corresponding bridge token on NEAR, the attacker can call `initTransfer` with that token.

The `initTransfer` function itself has no check that the token is registered: [4](#0-3) 

---

### Impact Explanation

Every `initTransfer` call using a fee-on-transfer token creates a permanent deficit in the EVM vault. The NEAR side mints or releases `amount` tokens backed by only `amount - transfer_fee` tokens locked on EVM. Repeated exploitation drains the EVM collateral pool, eventually making it impossible for NEAR-to-EVM redemptions to be fulfilled — a direct case of **bridge collateralization corruption** and potential **permanent freezing of user funds** when the vault is depleted.

This matches the allowed impact: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

- `logMetadata` is permissionless; any user can register a fee-on-transfer token.
- `initTransfer` accepts any ERC20 address without a registry check.
- Fee-on-transfer tokens exist in production (e.g., tokens with reflection mechanics, some stablecoins with configurable fees).
- An attacker can deploy their own fee-on-transfer token, register it via `logMetadata`, wait for NEAR-side deployment, and then exploit `initTransfer` repeatedly.
- No privileged role or leaked key is required.

---

### Recommendation

Replace the direct `safeTransferFrom` call with a before/after balance check to determine the actual received amount, and use that value in the event and downstream logic:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "ERR_FEE_ON_TRANSFER_TOKEN");
```

Alternatively, explicitly disallow fee-on-transfer tokens by requiring `actualReceived == amount` and documenting this as a protocol invariant. The `amount` used in the `InitTransfer` event and passed to `initTransferExtension` must reflect the actual received amount, not the caller-supplied parameter.

---

### Proof of Concept

1. **Deploy** a malicious ERC20 token `M` that deducts 10% on every `transferFrom` (i.e., transferring `X` results in the recipient receiving `0.9X`).
2. **Register** `M` by calling `OmniBridge.logMetadata(address(M))`. This emits a `LogMetadata` event.
3. **Wait** for the NEAR relayer to process the event and deploy the corresponding bridge token `M_NEAR` on NEAR.
4. **Approve** the bridge to spend `1000` units of `M`.
5. **Call** `OmniBridge.initTransfer(address(M), 1000, 0, 0, nearRecipient, "")`.
   - The bridge calls `M.transferFrom(attacker, bridge, 1000)`.
   - Due to the 10% fee, the bridge receives only `900` units of `M`.
   - The `InitTransfer` event is emitted with `amount = 1000`.
6. **NEAR bridge** processes the signed message for `amount = 1000` and mints `1000` units of `M_NEAR` to `nearRecipient`.
7. **Result**: The EVM bridge holds `900` tokens but has issued `1000` tokens on NEAR. The bridge is undercollateralized by `100` tokens per transaction.
8. **Repeat** to drain the EVM vault, eventually preventing legitimate EVM redemptions. [5](#0-4) [6](#0-5)
