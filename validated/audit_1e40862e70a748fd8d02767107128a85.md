### Title
`eth_estimateGas`/`eth_call` Simulation Divergence from On-Chain Execution via Skipped `stateDB.Commit()` — (File: `x/evm/keeper/state_transition.go`)

---

### Summary

`ApplyMessageWithConfig` is called with `commit=false` for all simulation paths (`eth_estimateGas`, `eth_call`, `EstimateGas` gRPC). Because `stateDB.Commit()` is only invoked when `commit=true`, the `ErrStateConflict` error path — which surfaces only at commit time — is structurally unreachable during simulation. The code itself documents this divergence. A transaction that passes simulation with a valid gas estimate can fail on-chain with `VmError = "state conflict"`, consuming the user's gas fees for a transaction that was predicted to succeed.

---

### Finding Description

In `x/evm/keeper/state_transition.go`, the commit block is guarded by the `commit` boolean:

```go
if commit {
    if err := stateDB.Commit(); err != nil {
        if errors.Is(err, statedb.ErrStateConflict) {
            return &types.EVMResult{
                GasUsed:          gasUsed,
                VmError:          statedb.ErrStateConflict.Error(),
                ...
            }, nil
        }
        return nil, errorsmod.Wrap(err, "failed to commit stateDB")
    }
}
```

The code comment immediately above this block explicitly acknowledges the divergence:

> *"Note: estimateGas and eth_call do not hit this path because commit is false for simulations, so they will succeed even when a real execution would produce a state conflict."*

The `EstimateGas` gRPC handler calls `ApplyMessageWithConfig` with `commit=false`:

```go
rsp, err = k.ApplyMessageWithConfig(ctx, msg, cfg, false)
```

The `EthCall` handler does the same. The `SimulateV1` path (`eth_simulateV1`) uses a separate `Simulator` struct that also never calls `stateDB.Commit()` on the shared state.

`ErrStateConflict` is triggered when a nested native (precompile) action modifies Cosmos-native state that conflicts with the EVM's in-flight state view. Because the commit is skipped in simulation, this conflict is never detected. The simulation returns `status=0x1` and a gas estimate. When the user submits the transaction on-chain, `ApplyTransaction` calls `ApplyMessageWithConfig` with `commit=true`, the commit runs, the conflict is detected, and the transaction is included in the block with `VmError = "state conflict"` (status=0). The user's gas fees are fully consumed.

---

### Impact Explanation

**Allowed impact matched**: *"Public JSON-RPC, gRPC, simulation, tracing, receipt/log, or indexer path feeds incorrect consensus-critical data into transaction execution or exposes a reachable route to the impacts above"* and *"EVM state transition … bug that permits … valid user funds/fees to be mis-accounted."*

The `eth_estimateGas` RPC returns a gas estimate and an implicit success signal. Wallets and DApps use this result to decide whether to submit the transaction and what gas limit to set. When the simulation incorrectly reports success, the user submits the transaction, it is included in a block, fails with VmError, and the gas fees are burned/distributed — a concrete mis-accounting of user funds. A malicious DApp or precompile author can craft interactions that reliably pass simulation but fail on-chain, causing targeted users to waste gas on every interaction.

---

### Likelihood Explanation

**Medium-Low.** The trigger requires a transaction that interacts with a custom precompile that writes to native Cosmos state in a way that conflicts with EVM state at commit time. This is not reachable via pure EVM-to-EVM calls. However, Ethermint explicitly supports custom precompiles (`customContractFns`), and any chain deploying such precompiles (e.g., staking, governance, IBC precompiles common in Cosmos EVM chains) exposes this path. An unprivileged user can craft a transaction targeting such a precompile; no privileged role is required. The attacker-controlled entry point is a standard `eth_estimateGas` call followed by a signed `MsgEthereumTx` submission.

---

### Recommendation

1. **Surface `ErrStateConflict` during simulation**: In `ApplyMessageWithConfig`, when `commit=false`, perform a dry-run commit on a snapshot (or check for pending conflicts before skipping commit) so that `ErrStateConflict` is detectable in simulation mode.
2. **Alternatively**, document and enforce that any precompile writing to native state must implement conflict-free semantics, and add a simulation-time conflict pre-check in the precompile framework.
3. **At minimum**, update `eth_estimateGas` and `eth_call` responses to include a warning or distinct error code when the simulation result may diverge from on-chain execution due to native state interactions.

### Proof of Concept

**Root cause location**: [1](#0-0) 

The self-documenting comment at lines 608–611 confirms the divergence is known and structural.

**Simulation entry point** (`eth_estimateGas` → gRPC `EstimateGas`): [2](#0-1) 

`commit=false` is hardcoded for every simulation call.

**On-chain execution entry point** (`EthereumTx` → `ApplyTransaction`): [3](#0-2) 

`ApplyTransaction` calls `ApplyMessageWithConfig` with `commit=true`, reaching the guarded commit block.

**Attack path**:
1. Attacker deploys or identifies a DApp that calls a custom precompile writing to native Cosmos state.
2. Victim calls `eth_estimateGas` on the transaction → simulation runs with `commit=false` → `ErrStateConflict` is never triggered → returns `{gas: N, status: success}`.
3. Victim submits the transaction with gas limit `N`.
4. On-chain: `ApplyTransaction` → `ApplyMessageWithConfig(commit=true)` → `stateDB.Commit()` → `ErrStateConflict` → `EVMResult{VmError: "state conflict"}`.
5. Transaction is included in block with status=0; victim's gas fees are consumed.
