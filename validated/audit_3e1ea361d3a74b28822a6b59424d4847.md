### Title
No Cancel Function for Stuck Pending Transfers Causes Permanent Irrecoverable Fund Loss - (File: near/omni-bridge/src/lib.rs)

### Summary

The NEAR OmniBridge contract stores outbound cross-chain transfers in `pending_transfers` after burning or locking user tokens, but exposes no function — for the DAO, the transfer owner, or any other role — to cancel a pending transfer and return the user's assets. This is the direct analog of the external report: just as the Governor contract became admin of Timelock but had no path to call `cancelTransaction`, the DAO here is the privileged administrator of the bridge but has no path to remove a stuck entry from `pending_transfers` and restore the underlying tokens.

### Finding Description

When a user bridges tokens from NEAR to a non-UTXO destination chain (EVM, Solana, StarkNet), the flow through `init_transfer_internal` burns or locks the user's tokens and inserts the transfer into `pending_transfers`: [1](#0-0) 

The only code paths that ever remove an entry from `pending_transfers` are:

1. **`sign_transfer_callback`** — removes the entry only when `fee.is_zero()` AND the MPC signing call succeeds: [2](#0-1) 

2. **`claim_fee_callback`** — removes the entry only after a relayer submits valid on-chain proof that the transfer was finalized on the destination chain: [3](#0-2) 

3. **`submit_transfer_to_utxo_chain_connector`** — only applicable to BTC/Zcash UTXO chains: [4](#0-3) 

4. **`remove_transfer_message_without_refund`** — a private helper called only during failed storage checks at insertion time, never after a transfer is successfully committed: [5](#0-4) 

There is no public `cancel_transfer`, `refund_transfer`, or equivalent function anywhere in the contract. The DAO role, despite being the privileged administrator, has no mechanism to remove a transfer from `pending_transfers` and restore the burned or locked tokens to the user.

The `pending_transfers` map is a core piece of contract state: [6](#0-5) 

The `set_locked_tokens` DAO function can adjust accounting but does not remove the pending transfer entry or return tokens: [7](#0-6) 

`storage_unregister` with `force: true` returns only the NEAR storage deposit; the pending transfer entry and the burned/locked bridged tokens are unaffected.

### Impact Explanation

If a transfer enters `pending_transfers` and cannot be processed to completion, the user's tokens are permanently lost:

- **Burned tokens** (deployed/bridged tokens): `burn_tokens_if_needed` destroys them at insertion time. Without a mint-back path triggered by cancellation, they are gone forever.
- **Locked tokens** (native NEAR tokens): `lock_tokens_if_needed` increments `locked_tokens`. Without a corresponding unlock triggered by cancellation, the tokens are irrecoverably locked in the bridge contract.

This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock of user or protocol funds in bridge or vault flows.**

### Likelihood Explanation

Multiple realistic, non-colluding scenarios cause a transfer to become permanently stuck:

1. **MPC signing failure**: `sign_transfer_callback` silently does nothing on error — the transfer stays in `pending_transfers` indefinitely. If the MPC signer contract is upgraded, migrated, or experiences a key rotation that invalidates in-flight requests, all pending transfers are frozen. [8](#0-7) 

2. **Fee-bearing transfer with unfinalizeable destination**: When `fee` is non-zero, the transfer is intentionally kept in `pending_transfers` after signing, waiting for `claim_fee` proof. If the destination chain bridge is paused, the token is delisted, or the factory address is rotated, no valid proof can ever be produced and the transfer is stuck forever.

3. **DAO removes token mapping or factory**: If the DAO removes the factory entry for a destination chain or the token decimals mapping, `sign_transfer` panics on the next call, and the transfer can never be signed or claimed. The DAO has no cancel path to recover the user's funds.

4. **Destination chain nonce collision or replay protection**: If the destination chain has already consumed the `destination_nonce` assigned to this transfer (e.g., due to a bridge upgrade or state migration), finalization is impossible and `claim_fee` can never succeed.

### Recommendation

**Short term**: Add a privileged `cancel_transfer` function callable by the DAO (and optionally by the transfer owner after a timeout) that:
- Removes the entry from `pending_transfers` via `remove_transfer_message`
- For burned deployed tokens: mints the original amount back to the transfer owner
- For locked native tokens: calls `unlock_tokens` to restore the locked balance and transfers the tokens back to the owner

**Long term**: Implement a time-based expiry mechanism for pending transfers. After a configurable timeout (e.g., 7 days), allow the transfer owner to self-cancel and reclaim their tokens without requiring DAO intervention, reducing the trust surface.

### Proof of Concept

1. Alice calls `ft_transfer_call` on a deployed bridged token contract, sending 1000 tokens to the NEAR bridge with an `InitTransfer` message targeting an EVM recipient with a non-zero fee.
2. `ft_on_transfer` → `init_transfer` → `init_transfer_internal` burns Alice's 1000 tokens via `burn_tokens_if_needed` and inserts the transfer into `pending_transfers`.
3. A trusted relayer calls `sign_transfer`. The MPC signer contract is temporarily unavailable; `sign_transfer_callback` receives `Err(PromiseError::Failed)` and returns without removing the transfer.
4. The MPC signer is subsequently upgraded and the old signing key is rotated. All future `sign_transfer` calls for this transfer produce signatures that the destination chain rejects.
5. No `claim_fee` proof can ever be produced. No `cancel_transfer` function exists.
6. Alice's 1000 tokens are permanently burned. The `pending_transfers` entry is irremovable. Alice has no recourse.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    fn remove_transfer_message_without_refund(
        &mut self,
        transfer_id: TransferId,
    ) -> TransferMessage {
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        transfer.message
    }
