Audit Report

## Title
Residual ETH Stranded in Router via Payable Swap Calls Is Silently Consumed by Subsequent WETH-Input Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

Any ETH sent as `msg.value` to a `payable` swap function where `tokenIn != WETH` is silently retained in the router because `pay()` takes the non-WETH branch and never touches the ETH. A subsequent caller whose `tokenIn == WETH` triggers `pay()` to read `address(this).balance`, which includes the stranded ETH, and deposits it as WETH on behalf of the second caller — permanently destroying the first caller's principal.

## Finding Description

`receive()` rejects plain ETH transfers from non-WETH addresses: [1](#0-0) 

However, `receive()` is **not** invoked when ETH is attached as `msg.value` to a named `payable` function. All four swap entry-points are declared `external payable`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

When `tokenIn != WETH`, `pay()` falls through to the `else` branch and calls `safeTransferFrom` — the ETH is never touched and remains in the router: [6](#0-5) 

When the next caller swaps with `tokenIn == WETH`, the swap callback invokes `pay(WETH, payer, pool, value)`. The function reads `address(this).balance` unconditionally before checking the payer's allowance: [7](#0-6) 

If `nativeBalance >= value`, the entire payment is sourced from the router's ETH balance — the second caller's WETH allowance is never touched. If `0 < nativeBalance < value`, the stranded ETH partially covers the payment. In both cases the original depositor's ETH is permanently consumed with no accounting or revert.

There is no per-transaction tracking of `msg.value`; `address(this).balance` aggregates ETH from all prior calls. The `refundETH()` function exists but is not enforced by the router, and omitting it is a realistic user/UI error.

## Impact Explanation

Direct, permanent loss of user principal. User A's ETH (sent inadvertently with a non-WETH swap) is consumed to subsidize User B's WETH-input swap. User A receives no output tokens for the lost ETH and has no on-chain recourse. This meets the Critical/High threshold for direct loss of user principal with no privileged access required.

## Likelihood Explanation

All swap functions are `payable`, so any wallet or front-end can silently attach ETH to a non-WETH swap. `multicall{value: X}(...)` without a trailing `refundETH()` is a standard and documented pattern. An on-chain attacker can monitor the mempool for transactions that strand ETH in the router and immediately follow with a WETH-input swap of the exact stranded amount to drain it. No privileged role, malicious pool, or non-standard token is required.

## Recommendation

Remove the `nativeBalance` shortcut from `pay()` entirely, or cap consumable native balance to the `msg.value` of the current transaction stored in transient storage at each swap entry-point. The simplest safe fix:

```solidity
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else {
        IERC20(token).safeTransferFrom(payer, recipient, value);
    }
}
```

Callers who want to pay with native ETH should use `multicall([wrapETH, swap, refundETH])` explicitly. Alternatively, store `msg.value` in transient storage at each entry-point and allow `pay()` to consume only up to that amount, preventing cross-transaction ETH leakage.

## Proof of Concept

```solidity
// 1. User A accidentally sends ETH with a non-WETH swap
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    tokenIn: address(token1),   // NOT WETH — pay() takes else branch, ETH stranded
    tokenOut: address(token2),
    amountIn: 1000,
    ...
}));
// router.balance == 1 ether

// 2. User B exploits the stranded ETH
uint256 wethBefore = weth.balanceOf(userB);
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: address(weth),
    amountIn: 1 ether,
    ...
}));
// pay() reads address(this).balance == 1 ether >= value == 1 ether
// deposits User A's ETH as WETH, transfers to pool
// weth.balanceOf(userB) == wethBefore (unchanged — User B paid nothing)
// User A's 1 ETH is permanently gone
```
