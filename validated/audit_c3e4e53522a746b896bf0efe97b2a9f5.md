### Title
Missing `state_diff_commitment` Hash Verification in P2P State Diff Sync Allows Malicious Peer to Store Corrupted ThinStateDiff - (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`StateDiffStreamBuilder::parse_data_for_block` validates only the **count** (`state_diff_length`) of received `StateDiffChunk`s against the stored header, but never verifies the assembled `ThinStateDiff` against the **Poseidon hash** (`state_diff_commitment`) also stored in that header. A malicious p2p peer can send chunks whose total `len()` matches the header's `state_diff_length` but whose storage diffs, nonces, or class hashes are entirely fabricated. The assembled wrong `ThinStateDiff` passes all checks and is written unconditionally to storage via `append_state_diff`, corrupting the values subsequently served by `get_storage_at` and `get_nonce_at`.

---

### Finding Description

The header stored during the header-sync phase contains two independent fields:

- `state_diff_length` — an integer count of state diff entries
- `state_diff_commitment` — a `StateDiffCommitment(PoseidonHash(...))` over the full content of the diff

`parse_data_for_block` reads only `state_diff_length` from the stored header: [1](#0-0) 

It accumulates chunks until the count matches, then performs a final count equality check at line 99: [2](#0-1) 

After that, only `validate_deprecated_declared_classes_non_conflicting` is called (a duplicate-class-hash check), and the assembled `ThinStateDiff` is returned with no hash verification: [3](#0-2) 

`write_to_storage` then calls `append_state_diff` directly with no commitment check: [4](#0-3) 

`append_state_diff` writes the diff's storage diffs, nonces, deployed contracts, and class hashes into the storage tables with no verification against `state_diff_commitment`: [5](#0-4) 

The `state_diff_commitment` field is present in the stored header and is the correct guard — `calculate_state_diff_hash` computes a Poseidon hash over all fields of `ThinStateDiff` — but it is never read in this path: [6](#0-5) 

By contrast, the committer path does have an optional `verify_state_diff_hash` guard, but that is a separate code path used only during block production, not during p2p sync: [7](#0-6) 

---

### Impact Explanation

The corrupted `ThinStateDiff` is stored into the `contract_storage`, `nonces`, `deployed_contracts`, and `compiled_class_hash` tables. Any subsequent RPC call to `starknet_getStorageAt`, `starknet_getNonce`, or fee estimation / simulation that reads state at the affected block number will receive the attacker-controlled wrong value. This matches the **High** impact: "RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."

---

### Likelihood Explanation

Any node reachable on the p2p network can serve state diff responses. The attacker only needs to be selected as the peer for the state diff sub-protocol query for a given block range. The header and state diff sub-protocols are independent sessions, so a node that served a legitimate header cannot be forced to also serve the state diff — a different peer can answer the state diff query. No operator privilege is required.

---

### Recommendation

After assembling the full `ThinStateDiff` (after line 106), compute `calculate_state_diff_hash(&result)` and compare it against `block_header.state_diff_commitment`. If they differ, return `ParseDataError::BadPeer(...)` and report the peer. The `state_diff_commitment` field is already present in the stored header and the hash function already exists in `starknet_api::block_hash::state_diff_hash`.

---

### Proof of Concept

1. Sync a header for block N that has `state_diff_length = 2` and `state_diff_commitment = H(real_diff)`.
2. As the state diff peer, send two `StateDiffChunk::ContractDiff` messages, each with one storage diff entry (total `len() == 2`), but with a fabricated contract address and storage key/value.
3. `parse_data_for_block` accepts the chunks: `current_state_diff_len (2) == target_state_diff_len (2)`.
4. `write_to_storage` calls `append_state_diff(block_N, fabricated_diff)`.
5. A subsequent `get_storage_at(block_N, real_contract, real_key)` returns `0x0` (missing) instead of the real value, and `get_storage_at(block_N, fake_contract, fake_key)` returns the attacker-chosen value.

    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
