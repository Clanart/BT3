### Title
Unguarded `refundETH()` allows any caller to steal excess ETH stranded on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`refundETH()` is a public, zero-access-control function that transfers the router's entire native ETH balance to `msg.sender`. Because the `pay()` helper consumes only the exact swap amount from the router's native balance and leaves any overpayment behind, a user who sends excess ETH without including `refundETH` in the same multicall permanently strands that ETH on the router. Any subsequent caller can drain it in a separate transaction.

### Finding Description

`refundETH()` contains no ownership or origin check:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← sends to whoever calls, not the depositor
    }
}
``` [1](#0-0) 

The `pay()` helper, when `tokenIn == WETH`, consumes exactly `value` wei from the router's native balance and leaves the rest untouched:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
    // excess nativeBalance - value stays on the router
}
``` [2](#0-1) 

`multicall` is `payable` and forwards `msg.value` into the execution context, so a user can legitimately send more ETH than the swap needs: [3](#0-2) 

The protocol's own documentation and tests confirm the intended pattern is `multicall{value}([swap, refundETH])`. If the user omits `refundETH`, the excess ETH is permanently accessible to any caller. [4](#0-3) 

### Impact Explanation
Direct, irreversible loss of the victim's excess ETH. The attacker calls one public function with no preconditions other than timing. No privileged role, malicious pool, or non-standard token is required.

### Likelihood Explanation
The ETH-input multicall pattern is explicitly documented and tested. Users who overpay (common with exact-input swaps where the exact WETH cost is unknown off-chain) and omit `refundETH` — whether by mistake, wallet UX error, or a failed second call — leave funds claimable by any observer. MEV bots routinely monitor for exactly this class of stranded balance.

### Recommendation
Add a reentrancy-safe per-transaction depositor registry, or restrict `refundETH` to only return ETH to the address that sent `msg.value` in the current (or most recent) multicall. The simplest fix is to record `msg.sender` in transient storage at the start of `multicall` and require `msg.sender == _transientDepositor()` inside `refundETH`.

### Proof of Concept

```
1. User calls router.multicall{value: 1 ether}([
       exactInputSingle(tokenIn=WETH, amountIn=0.5 ether, ...)
       // refundETH omitted
   ])
2. pay() deposits 0.5 ETH as WETH → pool; 0.5 ETH remains on router.
3. Attacker calls router.refundETH() in the next transaction.
4. _transferETH(attacker, 0.5 ether) executes; attacker receives victim's 0.5 ETH.
5. assert attacker.balance increased by 0.5 ETH; victim cannot recover funds.
```

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH`) does not protect against this — it only blocks direct ETH pushes, not `msg.value` attached to `payable` function calls. [5](#0-4)
