### Title
Integer Overflow in Relative Timelock Arithmetic Enables Timelock Bypass — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary

In `check_time_locks`, when the `nowrap` flag is `false` (the pre-soft-fork legacy path), relative timelock checks use `wrapping_add` for both `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE`. An attacker can craft a CLVM puzzle with a large enough relative timelock value that `coin_timestamp + seconds_relative` overflows `u64`, wrapping to a small number. The comparison `current_timestamp < small_wrapped_value` evaluates to `false`, so the timelock check passes immediately — the coin is spendable at any time, bypassing the intended lock.

### Finding Description

In `check_time_locks`, the `nowrap=false` branch uses `wrapping_add` for relative timelock arithmetic:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

If `unspent.timestamp + seconds_relative` overflows `u64`, the result wraps to a small value (e.g., `1000 + u64::MAX` wraps to `999`). The check `current_timestamp < 999` is `false`, so no error is returned and the spend is accepted — even though the intended timelock is `u64::MAX` seconds (~584 billion years).

The same overflow applies to `ASSERT_HEIGHT_RELATIVE` via `wrapping_add` on `u32`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
``` [2](#0-1) 

The `nowrap=true` branch correctly uses `saturating_add`, which clamps to `u64::MAX` and always fails the check for overflow values:

```rust
if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [3](#0-2) 

The divergence between the two paths is explicitly documented in the test suite:

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),
)]
``` [4](#0-3) 

The `nowrap` parameter is supplied externally (from the Python full node) and is `false` for blocks below the relevant soft-fork activation height. The Python binding exposes this directly: [5](#0-4) 

### Impact Explanation

Any CLVM puzzle that uses `ASSERT_SECONDS_RELATIVE N` where `N > u64::MAX - coin_timestamp` (currently ~`u64::MAX - 1.7×10⁹`) will have its timelock silently bypassed on nodes running with `nowrap=false`. The coin is spendable immediately at any block, regardless of the intended lock duration. This is a **timelock validation bypass enabling unauthorized spend acceptance** — matching the High allowed impact.

Additionally, when the soft fork activates and nodes transition to `nowrap=true`, nodes on opposite sides of the fork height will disagree on the validity of spends involving overflow-causing relative timelocks, producing **consensus divergence**.

### Likelihood Explanation

The attacker-controlled entry path is a standard CLVM spend bundle submitted to the mempool. Crafting a puzzle with a large `ASSERT_SECONDS_RELATIVE` value is trivial for any CLVM programmer. The coin must first be confirmed on-chain (the attacker creates it), then the attacker immediately spends it. The overflow threshold is computable from the coin's on-chain timestamp. No privileged access, leaked keys, or network-level attack is required.

### Recommendation

1. **Remove the `nowrap=false` wrapping path entirely** or gate it behind a block height that has already passed on mainnet, so all active nodes use `saturating_add`.
2. **Reject `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` values that would cause overflow** at condition-parse time in `conditions.rs`, analogous to how `PositiveOverflow` is already handled for these conditions.
3. **Accelerate the soft-fork activation** (`SOFT_FORK9_HEIGHT`) to eliminate the window during which `nowrap=false` is in effect.

### Proof of Concept

Let `coin_timestamp = 1_700_000_000` (approximate current mainnet epoch).

Craft a CLVM puzzle output containing:
```
(ASSERT_SECONDS_RELATIVE 18446744072009551616)
; 18446744072009551616 = u64::MAX - 1_700_000_000 + 1
; coin_timestamp + seconds_relative = 1_700_000_000 + 18446744072009551616
;   = 18446744073709551616 mod 2^64 = 0
```

On a node with `nowrap=false`:
- Check: `current_timestamp < 0` → `false` → **spend accepted immediately**

On a node with `nowrap=true`:
- Check: `current_timestamp < saturating_add(1_700_000_000, 18446744072009551616)` = `current_timestamp < u64::MAX` → `true` → **spend rejected**

The spend bundle is accepted by pre-soft-fork nodes and rejected by post-soft-fork nodes, constituting both a timelock bypass and a consensus split. [6](#0-5) [7](#0-6)
