### Title
`batch_insert` Bypasses Duplicate-Key and Duplicate-Hash Checks via Direct `insert_entry_to_blob` Call, Corrupting Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` inserts the bulk of its leaf nodes by calling the internal `insert_entry_to_blob` directly, bypassing the duplicate-key and duplicate-hash guards that the public `insert` function enforces. An attacker who controls the input batch (3 or more entries) can inject duplicate `KeyId` or duplicate `Hash` values into the Merkle blob, silently corrupting the tree structure and its root hash, which enables forged DataLayer inclusion/exclusion proofs.

---

### Finding Description

`MerkleBlob::insert` (the public entry point) enforces two invariants before writing any leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` is the performance-optimised bulk path. It pops the **last two** items from the input vector and routes them through the guarded `insert`:

```rust
// lines 578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { … };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // ← guarded
    }
}
```

All **remaining** items (positions `0 .. len-2`) are written with the bare internal primitive:

```rust
// lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← NO duplicate check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` writes raw bytes into the blob and does **not** update `block_status_cache`. Consequently:

- Duplicate `KeyId` values land in the blob; the cache is unaware of the second copy.
- Duplicate `Hash` values land in the blob; the cache is unaware of the second copy.
- The tree-building loop then computes `internal_hash` over the corrupted leaf set, producing a wrong root hash.
- `check_integrity` will later detect the mismatch between `leaf_count` and `key_to_index_cache_length`, but only if called explicitly — it is not called inside `batch_insert`.

---

### Impact Explanation

The corrupted root hash is the authoritative commitment used by `get_proof_of_inclusion`. A tree with a duplicate key or hash can produce a proof path that verifies against the stored (wrong) root but does not correspond to any legitimately inserted key-value pair, or can suppress a valid key from being provable. This satisfies the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

`batch_insert` is exposed to Python callers via the `datalayer.pyi` stub and is used inside `deltas.rs` during delta application, making it reachable from untrusted serialized DataLayer input.

---

### Likelihood Explanation

**Low-to-Medium.** The caller must supply a batch of ≥ 3 entries where at least two of the first `len-2` entries share a `KeyId` or `Hash`. This is trivially constructable by any caller of the Python binding or any code that feeds attacker-controlled delta data into `batch_insert`. No privileged role or key material is required.

---

### Recommendation

Move the duplicate-key and duplicate-hash checks out of `insert` and into a shared helper, then call that helper at the top of `batch_insert` before any writes:

```rust
pub fn batch_insert(
    &mut self,
    keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Validate all entries before touching the blob
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // … existing bulk-insert logic …
}
```

Additionally, `insert_entry_to_blob` should update `block_status_cache` for leaf nodes, or its use should be restricted to internal nodes only, to prevent future bypasses of the same kind.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId, Hash
import hashlib

blob = MerkleBlob(b"")

def h(n: int) -> Hash:
    return Hash(hashlib.sha256(n.to_bytes(8, "big")).digest())

# Items at positions 0..len-2 bypass duplicate checks.
# Positions 0 and 1 share the same KeyId(1) — only position 2 (popped last) goes through insert().
batch = [
    ((KeyId(1), ValueId(10)), h(10)),   # position 0 — bypasses check
    ((KeyId(1), ValueId(20)), h(20)),   # position 1 — bypasses check (duplicate key!)
    ((KeyId(2), ValueId(30)), h(30)),   # position 2 — popped, goes through insert()
]
blob.batch_insert(batch)   # succeeds; duplicate KeyId(1) is now in the blob

# Root hash is now computed over a tree containing two leaves with KeyId(1).
# Proof-of-inclusion for KeyId(1) is ambiguous / forged.
print("leaf count in cache:", blob.leaf_count)   # may show 2, not 3
