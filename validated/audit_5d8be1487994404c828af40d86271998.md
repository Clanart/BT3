### Title
Panic in `ProofOfSpace::update_digest()` via `.expect()` on Invalid v2 Proof Bytes Causes Node Process Abort — (File: `crates/chia-protocol/src/proof_of_space.rs`)

---

### Summary

The `Streamable::update_digest()` implementation for `ProofOfSpace` unconditionally calls `.expect()` on the result of `quality_string()` for v2 (version=1) proofs. Because `quality_string()` returns `Option<Bytes32>` and can return `None` for a structurally valid but cryptographically invalid proof, any call to `update_digest()` on such a proof panics. A v2 `ProofOfSpace` with a valid wire structure but invalid proof bytes is accepted by the `parse()` deserializer, making this panic reachable from untrusted network input. In the Python/pyo3 deployment context, a Rust `panic!` aborts the entire process rather than raising a Python exception.

---

### Finding Description

**Root cause — unconditional `.expect()` in `update_digest()`:** [1](#0-0) 

```rust
// line 249-256 (update_digest, version == 1 branch)
self.quality_string()
    .expect("internal error. Can't compute hash of invalid ProofOfSpace")
    .update_digest(digest);
```

`quality_string()` delegates to `chia_pos2::quality_string_from_proof`, which returns `None` when the proof bytes do not satisfy the proof-of-space constraints: [2](#0-1) 

**Deserializer accepts structurally valid v2 proofs with arbitrary proof bytes:**

The `parse()` implementation for `version == 1` validates only that exactly one of `pool_public_key` / `pool_contract_puzzle_hash` is set; it performs no cryptographic check on the `proof` field itself: [3](#0-2) 

A peer can therefore transmit a `ProofOfSpace` with `version=1`, one of the two pool fields set, and `proof` filled with arbitrary bytes. `parse::<false>()` returns `Ok(...)`.

**`update_digest()` is called during block-header hashing:**

`update_digest` is the `Streamable` trait method used to compute the SHA-256 hash of any streamable object. `ProofOfSpace` is embedded in block headers; computing a block-header hash requires calling `update_digest` on the embedded `ProofOfSpace`. Because the `Streamable` trait signature is `fn update_digest(&self, digest: &mut Sha256)` (no `Result` return), the only way the current code can signal failure is to panic: [4](#0-3) 

**Secondary panic — `compute_plot_id()` on v0 proof with neither pool key nor contract:**

`compute_plot_id_v1()` also panics unconditionally when both optional fields are `None`: [5](#0-4) 

The `parse()` path for `version == 0` imposes no such constraint (the round-trip test `v0_neither_lenient` explicitly confirms this is accepted), so a v0 proof with neither field set is a valid wire object that triggers this panic when `compute_plot_id()` is called on it: [6](#0-5) 

---

### Impact Explanation

`update_digest` is invoked whenever a block header containing a `ProofOfSpace` is hashed — an operation that occurs early in block validation (before cryptographic proof verification). An attacker who propagates a block carrying a v2 `ProofOfSpace` with valid wire structure but invalid proof bytes causes every receiving full node to call `update_digest`, hit the `.expect()` panic, and abort the process. In the pyo3 Python binding context, a Rust panic is not catchable as a Python exception; it terminates the chia-blockchain daemon. If the malformed block is relayed widely, this constitutes a deterministic, network-wide chain halt triggered by a single unprivileged serialized network object.

This matches the allowed Critical impact: *"Valid unprivileged … serialized network object can trigger deterministic consensus divergence, chain halt, or committed state corruption."*

---

### Likelihood Explanation

- No authentication, key material, or privileged role is required.
- The attacker only needs to craft a `ProofOfSpace` with `version=1`, one pool field set, and random proof bytes — all fields are freely chosen.
- The malformed block passes `parse::<false>()` without error.
- The panic fires on the first call to `update_digest` (i.e., the first attempt to hash the block header), before any expensive cryptographic check.
- Likelihood: **High** — trivially constructable payload, no preconditions.

---

### Recommendation

1. Change `update_digest` to propagate errors, or pre-validate the proof before hashing. At minimum, replace the `.expect()` with a fallback that writes a sentinel hash value and logs an error, so the caller can detect and reject the block without aborting.
2. Add a validity check in `parse()` for `version == 1` that verifies `quality_string()` is `Some` before returning `Ok`, so an invalid proof is rejected at deserialization time rather than at hashing time.
3. For `compute_plot_id_v1()` and `compute_plot_group_id_v2()`, replace `panic!` with `Result`-returning signatures so callers can handle the neither-set case gracefully.

impl Streamable for ProofOfSpace {
    fn update_digest(&self, digest: &mut Sha256) {
        self.challenge.update_digest(digest);
        self.pool_public_key.update_digest(digest);

        if self.version == 0 {
            self.pool_contract_puzzle_hash.update_digest(digest);
            self.plot_public_key.update_digest(digest);
            self.size.update_digest(digest);
            self.proof.update_digest(digest);
        } else if self.version == 1 {
            if let Some(pool_contract) = self.pool_contract_puzzle_hash {
                0b11_u8.update_digest(digest);
                pool_contract.update_digest(digest);
            } else {
                0b10_u8.update_digest(digest);
            }

            self.plot_public_key.update_digest(digest);
            self.plot_index.update_digest(digest);
            self.meta_group.update_digest(digest);
            self.strength.update_digest(digest);

            // for v2 proofs, we don't hash the full proof directly. The full
            // proof is the witness to this quality string commitment.
            self.quality_string()
                .expect("internal error. Can't compute hash of invalid ProofOfSpace")
                .update_digest(digest);
        } else {
            panic!("version field must be 0 or 1, but it's {}", self.version);
        }
    }
